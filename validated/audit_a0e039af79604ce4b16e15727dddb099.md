### Title
Duplicate Transaction Hashes in Proposal Content Cause Node-Crashing Panic in `finalize_decision` After Consensus — (`crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

### Summary

`handle_proposal_part` in the validator path accepts `ProposalPart::Transactions` batches without checking for duplicate transaction hashes. A malicious proposer can stream the same transaction hash twice. Because the blockifier executes the first copy successfully and rejects the second (nonce already incremented), both proposer and validator compute identical `execution_infos_and_signatures` (one entry) and identical block commitments, so the proposal passes the `ProposalFin` comparison and consensus is reached. When `decision_reached` subsequently calls `finalize_decision`, it iterates over the stored `content` — which still contains both copies — and panics unconditionally on the duplicate hash, crashing the node after the block has already been committed to consensus.

### Finding Description

**Step 1 — No deduplication in `handle_proposal_part`.**

When a `ProposalPart::Transactions` batch arrives, the validator converts each transaction and appends the entire batch to `content` without any uniqueness check on transaction hashes:

```rust
content.push(txs.clone());
let input = SendTxsForProposalInput { proposal_id, txs };
```

A proposer can therefore stream `[tx_A, tx_A]` across one or more batches.

**Step 2 — Commitments match despite the duplicate.**

The blockifier executes `tx_A` successfully (nonce 0→1) and then rejects the second copy (nonce mismatch). Both proposer and validator end up with `execution_infos_and_signatures = {tx_A: info}` (one entry) and `rejected_tx_hashes = {tx_A}` (one entry). `prepare_txs_hashing_data` iterates only `execution_infos_and_signatures`, so `calculate_block_commitments` sees exactly one transaction on both sides. The `concatenated_counts` field encodes `transaction_count = 1` on both sides. The `ProposalFin` comparison passes.

**Step 3 — `executed_transaction_count = 2` is accepted.**

The proposer sets `executed_transaction_count = 2` (both copies were processed). The validator's guard only checks `executed_txs_count > n_received_txs`; since 2 ≤ 2, it passes. `truncate_to_executed_txs` keeps both copies in `content`.

**Step 4 — `finalize_decision` panics.**

After consensus is reached, `finalize_decision` is called with `transactions = [[tx_A, tx_A]]`. It builds a `HashMap` by iterating the flattened content:

```rust
for tx in transactions.into_iter().flatten() {
    let key = tx.tx_hash();
    if transactions_hash_map.insert(key, tx).is_some() {
        // TODO(Dafna): Handle this error gracefully.
        panic!("Duplicate transactions found with the same tx_hash: {key:?}");
    }
}
```

The second insertion of `tx_A` returns `Some(...)` and the unconditional `panic!` fires.

### Impact Explanation

The panic propagates out of `finalize_decision` → `decision_reached` → the consensus manager task, crashing the node process. The block has already been committed to consensus at this point, but `update_state_sync_with_new_block` is never called. The node's local state diverges from the committed chain state. Every validator that voted for this block and calls `decision_reached` is affected simultaneously, potentially halting the chain until all nodes are manually restarted and re-synced.

This matches the allowed impact: **Wrong state/receipt/event/storage value for accepted input** — the block is accepted by consensus but the resulting state is never written, and the node crashes.

### Likelihood Explanation

The attacker must be the legitimate proposer for a round, which requires being a validator. In a permissionless network this is achievable by anyone willing to stake. The proposer is selected deterministically, so the attacker can wait for their turn. No special tooling is needed beyond the ability to craft a proposal with a repeated transaction in the `TransactionBatch` stream.

### Recommendation

1. **In `handle_proposal_part`**: maintain a `HashSet<TransactionHash>` across all received batches and return `HandledProposalPart::Invalid` if a duplicate hash is detected.
2. **In `finalize_decision`**: replace the `panic!` with a graceful error path (the existing `TODO(Dafna)` acknowledges this). The function should return a `Result` and propagate the error so the node can log and recover rather than crash.

### Proof of Concept

```
Proposer streams:
  ProposalPart::Init(...)
  ProposalPart::Transactions([tx_A, tx_A])   // same tx hash twice
  ProposalPart::Fin { executed_transaction_count: 2, proposal_commitment: C }

Validator receives both copies, executes:
  tx_A → success  → execution_infos_and_signatures = {tx_A: info}
  tx_A → rejected → rejected_tx_hashes = {tx_A}

Both sides compute identical commitments (transaction_count=1).
ProposalFin comparison: C == C  ✓  → proposal accepted.

consensus reaches decision → decision_reached() → finalize_decision()
  transactions = [[tx_A, tx_A]]
  iteration 1: insert(tx_A) → None  (ok)
  iteration 2: insert(tx_A) → Some  → panic!("Duplicate transactions found…")
  → node crashes
```

async fn collect_execution_results_and_stream_txs(
    tx_chunk: &[InternalConsensusTransaction],
    results: Vec<TransactionExecutorResult<TransactionExecutionOutput>>,
    execution_data: &mut BlockTransactionExecutionData,
    pre_confirmed_tx_sender: &mut Option<PreconfirmedTxSender>,
) -> BlockBuilderResult<()> {
    assert!(
        results.len() == tx_chunk.len(),
        "The number of results match the number of transactions."
    );

    for (input_tx, result) in tx_chunk.iter().zip(results) {
        let optional_l1_handler_tx =
            if let InternalConsensusTransaction::L1Handler(l1_handler_tx) = input_tx {
                Some(l1_handler_tx.tx.clone())
            } else {
                None
            };
        let tx_hash = input_tx.tx_hash();

        // Insert the tx_hash into the appropriate collection if it's an L1_Handler transaction.
        if let InternalConsensusTransaction::L1Handler(_) = input_tx {
            let is_new_entry = execution_data.consumed_l1_handler_tx_hashes.insert(tx_hash);
            // Even though this doesn't get past the set insertion, this indicates a major, possibly
            // reorg-producing bug, either in some batcher cache or the l1 provider.
            assert!(is_new_entry, "Duplicate L1 handler transaction hash: {tx_hash}.");
        }

        match result {
            Ok((tx_execution_info, state_maps)) => {
                if let Some(ref revert_error) = tx_execution_info.revert_error {
                    warn!(
                        "Transaction {} is reverted during execution while still accepted. Revert \
                         Error: {}",
                        input_tx.tx_hash(),
                        revert_error,
                    );
                }
                let (tx_index, duplicate_tx_hash) =
                    execution_data.execution_infos_and_signatures.insert_full(
                        tx_hash,
                        (tx_execution_info, input_tx.tx_signature_for_commitment()),
                    );
                assert_eq!(duplicate_tx_hash, None, "Duplicate transaction: {tx_hash}.");

                if let Some(block_number) = proof_facts_block_number(input_tx) {
                    execution_data.proof_facts_block_numbers.insert(tx_hash, block_number);
                }

                // Skip sending the pre confirmed executed transactions, receipts and state diffs
                // during validation flow or if the channel was closed. In validate flow
                // pre_confirmed_tx_sender is None.
                if let Some(pre_confirmed_sender) = pre_confirmed_tx_sender {
                    let tx_receipt = StarknetClientTransactionReceipt::from((
                        tx_hash,
                        TransactionOffsetInBlock(tx_index),
                        // TODO(noamsp): Consider using tx_execution_info and moving the line that
                        // consumes it below this (if it doesn't change functionality).
                        &execution_data.execution_infos_and_signatures[&tx_hash].0,
                        optional_l1_handler_tx,
                    ));

                    let tx_state_diff = StarknetClientStateDiff::from(state_maps).0;

                    let result = pre_confirmed_sender.try_send((
                        input_tx.clone(),
                        tx_receipt,
                        tx_state_diff,
                    ));

                    match result {
                        Ok(_) => {}
                        Err(TrySendError::Closed(_)) => {
                            warn!(
                                "Preconfirmed block writer channel was closed. Skipping to send \
                                 further preconfirmed transactions."
                            );
                            *pre_confirmed_tx_sender = None;
                        }
                        Err(err) => {
                            warn!("Sending data to preconfirmed block writer failed: {:?}", err);
                        }
                    }
                }
            }
            Err(err) => {
                info!(
                    "Transaction {} failed to execute with error: {}.",
                    tx_hash,
                    err.log_compatible_to_string()
                );
                let is_new_entry = execution_data.rejected_tx_hashes.insert(tx_hash);
                assert!(is_new_entry, "Duplicate rejected transaction hash: {tx_hash}.");
            }
        }
    }

    Ok(())
}
