### Title
`parameters_votes` Not Cleared on Resharing Cancellation, Allowing Non-Participant to Force Resharing - (File: crates/contract/src/state/resharing.rs)

---

### Summary

When resharing is cancelled via `vote_cancel_resharing`, the contract reverts to `previous_running_state`. However, if a re-proposal (`vote_new_parameters`) was cast during the resharing phase, the accumulated votes are stored in `previous_running_state.parameters_votes` and are **not cleared** on cancellation. A non-current participant who voted for that re-proposal can exploit these stale votes to immediately re-trigger resharing with a single call, bypassing the normal full-quorum voting requirement in the Running state.

---

### Finding Description

`ResharingContractState` stores a snapshot of the pre-resharing running state in `previous_running_state`. [1](#0-0) 

When the contract transitions from Running → Resharing, `previous_running_state` is constructed via `RunningContractState::new(...)`, which initialises `parameters_votes` to the empty default. [2](#0-1) 

During resharing, participants may submit a re-proposal via `vote_new_parameters`. This delegates to `self.previous_running_state.process_new_parameters_proposal(proposal)`, which accumulates votes directly into `previous_running_state.parameters_votes`. [3](#0-2) [4](#0-3) 

When resharing is cancelled, `vote_cancel_resharing` clones `previous_running_state` and returns it as the new running state — **without resetting `parameters_votes`**: [5](#0-4) 

The stale votes from the re-proposal phase therefore survive into the restored Running state. A non-current participant D whose vote is already recorded in `parameters_votes` can then call `vote_new_parameters` with the same proposal. The admission gate for new participants checks only whether `threshold` current-epoch participants have already voted for that proposal: [6](#0-5) 

Because A and B's stale votes satisfy the threshold check, D is admitted, D's vote is re-inserted (replacing the existing entry), the total reaches `new_num_participants`, and resharing is triggered immediately — with no fresh votes from A, B, or C required in the Running state.

---

### Impact Explanation

Forcing the contract into Resharing state sets `accept_requests = false`, halting all new signature and CKD requests. D can repeat this cycle (trigger → participants cancel → trigger again) as long as D can cast a re-proposal vote during each resharing phase, creating a persistent request-lifecycle disruption. This breaks the production safety invariant that resharing requires a fresh, unanimous vote from all proposed participants in the Running state.

---

### Likelihood Explanation

Low. The preconditions are: (1) a resharing was initiated; (2) a re-proposal was cast during that resharing; (3) threshold current participants voted for the re-proposal, admitting D; (4) the resharing was subsequently cancelled. All four conditions can arise naturally in adversarial operational scenarios (e.g., a proposed participant deliberately stalls resharing to accumulate stale votes, then forces cancellation).

---

### Recommendation

In `vote_cancel_resharing`, explicitly reset `parameters_votes` on the restored running state before returning it:

```rust
let running_state = if threshold_cancellation_votes_reached {
    let mut previous_running_state = self.previous_running_state.clone();
    previous_running_state.previously_cancelled_resharing_epoch_id =
        Some(self.prospective_epoch_id());
    // Fix: clear stale re-proposal votes so they cannot be replayed
    previous_running_state.parameters_votes = ThresholdParametersVotes::default();
    Some(previous_running_state)
} else {
    None
};
``` [5](#0-4)

### Proof of Concept

```
State: Running — participants {A, B, C}, threshold 2

Step 1: A, B, C all vote for proposal P (adding D, threshold 3)
        → contract enters Resharing state
        → previous_running_state.parameters_votes = {} (empty, from RunningContractState::new)

Step 2: During resharing, A and B vote for re-proposal P′
        → previous_running_state.parameters_votes = {A→P′, B→P′}

Step 3: D votes for P′
        → admission check: n_votes(P′, {A,B,C}) = 2 ≥ threshold 2 → D admitted
        → previous_running_state.parameters_votes = {A→P′, B→P′, D→P′}
        → total = 3 < new_num_participants(P′) = 3? No, 3 == 3 → re-proposal accepted
          (if C has not yet voted; otherwise re-proposal succeeds and votes are cleared)

        [Assume C has not voted for P′, so total = 3 but new_num_participants = 3 only if
         P′ proposes {A, B, D}. With {A,B,D} as proposed set, 3 == 3 → re-proposal triggers
         a new ResharingContractState with fresh previous_running_state. To keep the stale
         votes, assume P′ proposes {A, B, C, D} (4 participants) so 3 < 4 and votes persist.]

Step 4: A and B vote to cancel resharing (2 ≥ threshold 2) → resharing cancelled
        → restored running state has parameters_votes = {A→P′, B→P′, D→P′}

Step 5: D calls vote_new_parameters(prospective_epoch_id = cancelled+1, proposal = P′)
        → D is not a current participant
        → n_votes(P′, current participants {A,B,C}) = 2 ≥ threshold 2 → D admitted
        → D's vote re-inserted → total = 3 (A, B, D)
        → if new_num_participants(P′) == 3 → resharing triggered immediately
        → accept_requests = false, signing halted
``` [7](#0-6) [8](#0-7) [2](#0-1)

    pub fn vote_cancel_resharing(&mut self) -> Result<Option<RunningContractState>, Error> {
        let previous_running_participants = self.previous_running_state.parameters.participants();
        let authenticated_candidate = AuthenticatedAccountId::new(previous_running_participants)?;
        self.cancellation_requests.insert(authenticated_candidate);

        let cancellation_votes_count = self.cancellation_requests.len() as u64;
        let previous_running_threshold = self.previous_running_state.parameters.threshold();

        let threshold_cancellation_votes_reached: bool =
            cancellation_votes_count >= previous_running_threshold.value();

        let running_state = if threshold_cancellation_votes_reached {
            let mut previous_running_state = self.previous_running_state.clone();
            let prospective_epoch_id = self.prospective_epoch_id();
            previous_running_state.previously_cancelled_resharing_epoch_id =
                Some(prospective_epoch_id);

            Some(previous_running_state)
        } else {
            None
        };

        Ok(running_state)
