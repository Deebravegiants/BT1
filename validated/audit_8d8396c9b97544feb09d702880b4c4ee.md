### Title
Fee Quality Check Is Advisory-Only: Malicious Harvester Can Unconditionally Redirect Farmer Reward — (`chia/farmer/farmer_api.py`)

### Summary
A harvester peer that sends `NewProofOfSpace` with `farmer_reward_address_override` set and `fee_info.applied_fee_threshold` deliberately set to a failing value will still have its override applied to `DeclareProofOfSpace`. The fee quality check in `notify_farmer_reward_taken_by_harvester_as_fee` only emits a log warning; it neither returns a blocking sentinel nor sets any state that `_process_respond_signatures` consults before accepting the override.

### Finding Description
In `FarmerAPI.new_proof_of_space`:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
```

The return value is unused. `notify_farmer_reward_taken_by_harvester_as_fee` logs either `"Fee threshold passed"` or `"Invalid fee threshold"` and returns `None` in both cases. No flag is set, no early return is triggered.

Later, in `_process_respond_signatures`, which runs when the harvester's `RespondSignatures` arrives:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

There is no check of whether the fee quality check passed. The override is applied unconditionally, and `DeclareProofOfSpace` is broadcast to the full node with the attacker-controlled address as `farmer_reward_address`.

### Impact Explanation
Every block won by the farmer while a malicious harvester is connected can have its farmer reward (currently 0.25 XCH per block) redirected to an address controlled by the harvester operator. The farmer receives no XCH for those blocks. This is unauthorized payout redirection — a listed High impact.

### Likelihood Explanation
Any harvester operator who connects to a farmer can trigger this. No key compromise, no cryptographic break, and no admin access is required. The harvester only needs a valid proof of space (which it legitimately has) and the ability to set two fields in protocol messages it already controls.

### Recommendation
`new_proof_of_space` must treat a failed fee quality check as a hard block, not a log event. The simplest fix:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    if not self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space):
        return None   # abort; do not send RequestSignatures
```

`notify_farmer_reward_taken_by_harvester_as_fee` must be changed to return `True` on pass and `False` on fail. Additionally, `_process_respond_signatures` should verify that the override address matches what was recorded during `new_proof_of_space` (i.e., store the approved override address per quality string and compare).
