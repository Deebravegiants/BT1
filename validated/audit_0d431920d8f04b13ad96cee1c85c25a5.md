### Title
Missing On-Chain Verification for `AppPublicKey` CKD Responses Allows Single Participant to Forge Confidential Key Derivation Output - (File: crates/contract/src/lib.rs)

### Summary
The `respond_ckd` function in the MPC contract performs cryptographic on-chain verification of CKD responses only for the `AppPublicKeyPV` variant. For the `AppPublicKey` (legacy) variant, no verification is performed. A single attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse`, bypassing the threshold requirement and delivering an attacker-controlled confidential key derivation output to the requesting user.

### Finding Description
In `crates/contract/src/lib.rs`, the `respond_ckd` function handles two variants of CKD app public keys:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV` requests, the contract calls `ckd_output_check` against the MPC root public key, ensuring the response is cryptographically consistent with the threshold-computed output. For `AppPublicKey` requests, the arm is a no-op: any `CKDResponse` with arbitrary `big_y` and `big_c` values is accepted unconditionally and forwarded to the user via `pending_requests::resolve_yields_for`. [2](#0-1) 

This is structurally analogous to the OmniPortal report's finding: a code path that should enforce a security invariant (threshold-consistent output) silently skips it for a specific input variant, allowing a single actor to produce an outcome that should require threshold cooperation.

By contrast, the `respond` function for signatures always verifies the signature cryptographically against the derived public key before resolving the yield: [3](#0-2) 

The asymmetry is clear: signature responses are always verified; `AppPublicKey` CKD responses are never verified.

### Impact Explanation
A single Byzantine attested participant (below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request in the contract state.
2. Call `respond_ckd` with a forged `CKDResponse` containing attacker-chosen `big_y` and `big_c` BLS12-381 G1 points.
3. The contract accepts the forged response and resolves the user's yield with the attacker-controlled values.
4. The user receives a confidential key derivation output that was not produced by the MPC threshold protocol — it was unilaterally chosen by one participant.

Because the attacker chose the forged key material, they know the resulting derived key. Any data the user subsequently encrypts under that key is readable by the attacker. This constitutes **unauthorized confidential key derivation output without the required participant authorization**, matching the Critical impact tier.

### Likelihood Explanation
Medium. The attacker must be an attested participant in the MPC network (cleared the TEE attestation check via `assert_caller_is_attested_participant_and_protocol_active`). [4](#0-3) 

However, no collusion with other participants is required — a single rogue node suffices. The `AppPublicKey` variant is labeled "legacy" in the README but remains fully accepted by the contract, and users who do not independently verify the response off-chain (which requires knowing the MPC protocol internals) will not detect the substitution.

### Recommendation
- **Preferred:** Reject `AppPublicKey` CKD requests at the `request_app_private_key` entry point and require callers to use `AppPublicKeyPV`, which supports on-chain verification.
- **Alternative:** If `AppPublicKey` must remain supported for backwards compatibility, document explicitly that the contract provides no on-chain integrity guarantee for responses to such requests, and add a prominent warning in the SDK and contract ABI that users must perform off-chain verification before trusting the derived key.
- Do not silently accept the response without any check; at minimum, emit a log warning so monitoring infrastructure can flag unverified CKD resolutions.

### Proof of Concept
```rust
// 1. Legitimate user submits an AppPublicKey CKD request
let app_public_key = CKDAppPublicKey::AppPublicKey(some_bls_g1_point);
contract.request_app_private_key(CKDRequestArgs {
    derivation_path: "my/path".to_string(),
    app_public_key: app_public_key.clone(),
    domain_id: DomainId::default(),
});

// 2. Single malicious attested participant forges the response
// (no threshold cooperation needed)
let forged_response = CKDResponse {
    big_y: dtos::Bls12381G1PublicKey([0xAB; 48]), // attacker-chosen
    big_c: dtos::Bls12381G1PublicKey([0xCD; 48]), // attacker-chosen
};

// Switch context to malicious participant
with_active_participant_and_attested_context(&contract);

// 3. Contract accepts the forged response — no ckd_output_check is called
contract
    .respond_ckd(ckd_request, forged_response)
    .expect("succeeds with no verification");

// 4. User's yield is resolved with attacker-controlled key material.
// Attacker knows the forged key and can decrypt any data the user encrypts.
```
