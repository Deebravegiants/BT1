### Title
Old Participants Blocked from Calling `respond*` During Resharing — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond`, `respond_ckd`, and `respond_verify_foreign_tx` functions explicitly permit execution during the `Resharing` protocol state via `is_running_or_resharing()`, yet their shared access-control guard `assert_caller_is_attested_participant_and_protocol_active()` resolves the participant set through `active_participants()`, which during resharing returns only the **new (proposed)** participants. Old participants who are being removed from the set hold the existing key shares, compute valid signatures off-chain, and are the only nodes capable of calling `respond*` — but the contract incorrectly blocks them, causing all in-flight requests to time out.

---

### Finding Description

`assert_caller_is_attested_participant_and_protocol_active` is the single gate used by all three `respond*` entry points:

```rust
fn assert_caller_is_attested_participant_and_protocol_active(&self) {
    let participants = self.protocol_state.active_participants(); // ← key line
    Self::assert_caller_is_signer();
    let attestation_check = self.tee_state.is_caller_an_attested_participant(participants);
    assert_matches::assert_matches!(attestation_check, Ok(()), "Caller must be an attested participant");
}
```

`active_participants()` resolves as follows:

```rust
pub fn active_participants(&self) -> &Participants {
    match self {
        ProtocolContractState::Running(state)   => state.parameters.participants(),
        ProtocolContractState::Resharing(state) => state.resharing_key.proposed_parameters().participants(), // NEW set
        ProtocolContractState::Initializing(state) => state.generating_key.proposed_parameters().participants(),
        ...
    }
}
```

During `Resharing`, this returns the **proposed new participant set**. An old participant not included in the new set will fail `is_caller_an_attested_participant` at the very first lookup:

```rust
let info = participants.info(&signer_id)
    .ok_or(AttestationCheckError::CallerNotParticipant)?; // ← fails for removed nodes
```

The contract panics with `"Caller must be an attested participant"` before the signature can be recorded.

Contrast this with `authenticate_update_vote`, which is used by governance functions (`voter_or_panic`) and correctly uses the **old** participant set during resharing:

```rust
ProtocolContractState::Resharing(state) => {
    AuthenticatedParticipantId::new(
        state.previous_running_state.parameters.participants(), // OLD set — correct
    )?;
}
```

The two authorization paths are inconsistent. `respond*` uses the new set; governance uses the old set. For signing, the old set is the correct one because only old participants hold key shares.

### Impact Explanation

During resharing, the contract explicitly allows `respond*` calls (`is_running_or_resharing()` returns `true`). Users can submit new `sign`, `request_app_private_key`, and `verify_foreign_transaction` requests throughout resharing. Old participants compute valid threshold signatures off-chain using their existing key shares. When a removed participant attempts to submit the result via `respond*`, the call panics. The yield-resume promise is never resolved; the request eventually times out via `fail_on_timeout`. All in-flight requests whose signing was led by a node being removed are permanently lost for that resharing window.

This breaks the request-lifecycle invariant: the contract accepts requests during resharing but cannot complete them if the responding node is being removed.

**Impact class**: Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

This condition is triggered by any resharing that removes at least one participant — a routine operational event (node rotation, TEE upgrade, participant set reduction). No adversarial action is required. The e2e test suite explicitly exercises shrinking resharings (e.g., `test_key_resharing` drops node 0 in resharing 2). [7](#0-6) 

---

### Recommendation

During `Resharing`, `respond*` functions should authenticate the caller against the **old** participant set (the set that holds the existing key shares), not the new proposed set. One approach: introduce a separate `respond_participants()` helper that returns `previous_running_state.parameters.participants()` during resharing and `parameters.participants()` during running, mirroring the pattern already used by `authenticate_update_vote`.

---

### Proof of Concept

```
1. Contract is Running with participants {A, B, C}, threshold 2.
2. Participants vote to reshard to {B, C} (removing A).
3. Contract enters Resharing state.
   - active_participants() → {B, C}  (new set)
4. User calls sign(). Contract enqueues a yield-resume promise.
5. Nodes A, B, C run the threshold signing protocol off-chain.
   Node A (being removed) aggregates the signature and calls respond().
6. assert_caller_is_attested_participant_and_protocol_active() fires:
   - active_participants() returns {B, C}
   - participants.info("A") → None → CallerNotParticipant
   - Contract panics: "Caller must be an attested participant"
7. The yield is never resolved. The sign request times out.
   User's cross-chain operation fails permanently for this request.
```
