After thorough investigation of the codebase, I identified a concrete analog to the reported resource-limit bypass vulnerability.

---

### Title
Missing `validateTooManyCollateralInputs` Enforcement in `utxoTransition` Allows Collateral Input Count Limit Bypass — (`File: eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs`)

---

### Summary

The `validateTooManyCollateralInputs` predicate is defined and exported in the Alonzo UTXO rule module, and the formal specification explicitly requires `‖collateral tx‖ ≤ maxCollateralInputs pp`. A comment in `utxoTransition` marks exactly where this check should be enforced, but the corresponding `runTest` call is absent. An unprivileged transaction submitter can therefore include arbitrarily many collateral inputs — bounded only by `maxTxSize` — bypassing the `maxCollateralInputs` protocol parameter entirely.

---

### Finding Description

The function `validateTooManyCollateralInputs` is defined at: [1](#0-0) 

```haskell
validateTooManyCollateralInputs pp txBody =
  failureUnless (numColl <= maxColl) $
    TooManyCollateralInputs Mismatch {mismatchSupplied = numColl, mismatchExpected = maxColl}
  where
    maxColl, numColl :: Word16
    maxColl = pp ^. ppMaxCollateralInputsL
    numColl = fromIntegral . Set.size $ txBody ^. collateralInputsTxBodyL
```

It is also exported from the module: [2](#0-1) 

However, inside `utxoTransition` — the transition rule used for both Alonzo and Babbage eras — the comment marking where the check belongs is present, but no `runTest` call follows it: [3](#0-2) 

```haskell
  {-   totExunits tx ≤ maxTxExUnits pp    -}
  runTest $ validateExUnitsTooBigUTxO pp tx

  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  -- ← NO runTest call here

  updatedGovState <-
    trans @(EraRule "UTXOS" era) $
```

The formal specification for both Alonzo and Babbage explicitly requires this check side-by-side with the ExUnits check: [4](#0-3) 

```
\fun{totExunits}~{tx} \leq \fun{maxTxExUnits}~{pp} &  \| \fun{collateral}~{tx} \| \leq \fun{maxCollateralInputs}~{pp}
``` [5](#0-4) 

The `feesOK` predicate (also called in `utxoTransition`) does not check the collateral count — it only checks that collateral inputs are VKey-locked, Ada-only, non-empty, and sufficient to cover the fee percentage: [6](#0-5) 

The `maxCollateralInputs` protocol parameter is defined and accessible: [7](#0-6) 

---

### Impact Explanation

Each collateral input requires a UTxO lookup and a VKey signature verification. The `maxCollateralInputs` parameter (currently 3 on mainnet) is designed to bound this per-transaction cost. Without enforcement, an attacker can include collateral inputs up to the transaction size limit (`maxTxSize = 16384` bytes). A `TxIn` is 36 bytes and a `WitVKey` signature is ~96 bytes; a maximally-packed transaction could carry ~100+ collateral inputs — roughly 33× the intended limit — forcing every validating node to perform proportionally more signature checks and UTxO lookups per transaction. This constitutes attacker-controlled transactions exceeding intended validation limits, matching the **Medium** allowed impact: *"Attacker-controlled transactions, blocks, certificates, votes, proposals, scripts, witnesses, or serialized inputs exceed intended validation limits."*

---

### Likelihood Explanation

Any unprivileged transaction submitter can craft such a transaction. No special role, key, or governance access is required. The attacker only needs to own enough UTxOs to use as collateral inputs and pay the transaction fee. The cost to the attacker is minimal (a single transaction fee), while the amplification of validation work on every node is significant.

---

### Recommendation

Add the missing enforcement call immediately after the existing comment in `utxoTransition`:

```haskell
  {-   totExunits tx ≤ maxTxExUnits pp    -}
  runTest $ validateExUnitsTooBigUTxO pp tx

  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  runTest $ validateTooManyCollateralInputs pp txBody   -- ADD THIS LINE
``` [3](#0-2) 

---

### Proof of Concept

1. Construct a transaction with a Plutus script input requiring collateral.
2. Populate `collateralInputsTxBodyL` with `N > maxCollateralInputs` valid VKey-locked, Ada-only UTxOs (e.g., N = 100 when `maxCollateralInputs = 3`).
3. Provide valid VKey witnesses for all N collateral inputs.
4. Keep `totExUnits` within `maxTxExUnits` and total transaction size within `maxTxSize`.
5. Submit the transaction.

Because `validateTooManyCollateralInputs` is never called in `utxoTransition`, the transaction passes the UTXO rule. Every validating node must verify all N signatures for the collateral inputs, performing 33× the intended signature verification work per such transaction. [1](#0-0) [8](#0-7)
