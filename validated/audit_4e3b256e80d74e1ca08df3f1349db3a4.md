### Title
`accept_requests` Guard in `respond` / `respond_ckd` / `respond_verify_foreign_tx` Permanently Freezes In-Flight Signature Requests When TEE Validation Fails - (File: `crates/contract/src/lib.rs`)

---

### Summary

When `verify_tee()` sets `accept_requests = false` due to a TEE validation failure, the same flag that is correctly used to block new incoming `sign()` requests also blocks `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()`. These three functions deliver already-computed MPC signatures back to users whose yield-resume promises are already parked and waiting. Because the node-side response path is gated on `accept_requests`, every in-flight request that was accepted before the flag was cleared will time out and fail â€” even though the MPC network has already computed a valid signature for it.

---

### Finding Description

`MpcContract` stores a boolean field `accept_requests` (line 162) that is set to `false` by `verify_tee()` when fewer than `threshold` participants have a valid TEE attestation (lines 1737â€“1738). This is the correct mechanism to stop accepting new user requests.

However, the same flag is also checked inside the three node-response methods:

**`respond()`** (lines 579â€“581):
```rust
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
```

**`respond_ckd()`** (lines 662â€“664):
```rust
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
```

**`respond_verify_foreign_tx()`** (lines 711â€“713):
```rust
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
```

These three functions are not "accepting new requests" â€” they are *delivering responses* to requests that were already accepted and are currently parked as yield-resume promises in `pending_signature_requests`, `pending_ckd_requests`, and `pending_verify_foreign_tx_requests`. Blocking them does not protect the system; it only prevents the MPC nodes from resolving the already-queued yields.

The lifecycle is:

1. User calls `sign()` â†’ `check_request_preconditions` passes (at this point `accept_requests = true`) â†’ yield-resume promise is created and stored in `pending_signature_requests`.
2. MPC nodes run the threshold signing protocol and compute a valid signature.
3. `verify_tee()` is called (by any participant) â†’ TEE validation fails â†’ `accept_requests = false`.
4. MPC nodes call `respond()` to deliver the signature â†’ blocked by `if !self.accept_requests` â†’ returns `TeeValidationFailed`.
5. The tx-sender retries (see `crates/node/src/indexer/tx_sender.rs` lines 196â€“206), but every retry also fails.
6. After ~200 blocks the NEAR runtime fires the yield-callback `return_signature_and_clean_state_on_success` with `Err(PromiseError::Failed)` â†’ `fail_on_timeout` is called â†’ the user's transaction fails permanently.

The user paid the deposit, the MPC network completed the cryptographic work, but the result can never be delivered.

---

### Impact Explanation

This matches the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

- Every signature request that was in-flight at the moment `accept_requests` transitions to `false` is permanently lost. The user's cross-chain operation fails; they must resubmit after the TEE situation is resolved.
- The MPC nodes' computed signatures are discarded. The nodes' work (triple/presignature consumption) is wasted.
- If `accept_requests` remains `false` for an extended period (e.g., manual intervention is required, as the contract itself logs: *"This requires manual intervention"*), all requests submitted during that window are frozen until timeout.
- The deposit (1 yoctoNEAR) is not the primary concern; the concern is that the user's cross-chain transaction â€” which may represent a time-sensitive bridge operation â€” fails silently from their perspective.

---

### Likelihood Explanation

`verify_tee()` is callable by any current participant at any time. TEE attestation expiry is a routine operational event (attestations have a configurable expiry). During a node upgrade cycle or when a participant's attestation lapses, `verify_tee()` will set `accept_requests = false`. This is not a rare edge case â€” it is an expected operational state. Any signature requests in flight at that moment are affected.

---

### Recommendation

Remove the `accept_requests` guard from `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()`. These functions deliver responses to already-accepted requests; they are not "accepting new requests." The `accept_requests` flag should only gate the user-facing entry points (`sign()`, `request_app_private_key()`, `verify_foreign_transaction()`), which it already does via `check_request_preconditions()` (line 300â€“302).

```rust
// In respond(), respond_ckd(), respond_verify_foreign_tx():
// REMOVE these blocks:
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
```

### Proof of Concept

1. Contract is in `Running` state with `accept_requests = true`.
2. User calls `sign(request)` â†’ yield-resume promise created, stored in `pending_signature_requests`. User's NEAR transaction is parked.
3. MPC nodes complete threshold signing and prepare to call `respond(request, signature)`.
4. Before step 3 lands on-chain, a participant calls `verify_tee()`. One participant's attestation has expired. The surviving set is below the threshold relation â†’ `accept_requests = false` (line 1737).
5. MPC node submits `respond(request, signature)` â†’ hits line 579: `if !self.accept_requests { return Err(TeeError::TeeValidationFailed.into()); }` â†’ transaction fails.
