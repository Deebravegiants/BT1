Looking at the `init()` function and comparing it to `init_running()`, and then examining the deployment scripts to understand the initialization flow.

### Title
Unprotected `init()` Enables Front-Running to Seize Full MPC Network Control — (`crates/contract/src/lib.rs`)

---

### Summary

`MpcContract::init()` carries `#[init]` (a one-time guard) but **no `#[private]` attribute**, meaning any NEAR account can call it on a freshly deployed but uninitialized contract. An attacker who observes the deployment transaction can front-run the legitimate deployer's `init()` call, installing themselves as the sole participant with `threshold = 1`, thereby seizing full governance and signing authority over the MPC network before any legitimate participant can act.

---

### Finding Description

**Root cause — missing `#[private]` on `init()`:**

`init()` is declared as:

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

Compare with `init_running()`, which is correctly restricted:

```rust
#[private]
#[init]
#[handle_result]
pub fn init_running(...)
``` [2](#0-1) 

In NEAR SDK, `#[private]` enforces `predecessor_account_id == current_account_id`, blocking all external callers. `init()` has no such guard — any NEAR account can call it.

**The deployment window:**

The deployment scripts and documentation show that contract deployment and `init()` are **separate transactions**:

```shell
near contract call-function as-transaction mpc-contract.test.near init \
  file-args ${init_args} ...
``` [3](#0-2) 

This creates a window between deployment and initialization during which any account can call `init()`.

**Mocked attestations bypass TEE hardware requirement:**

`init()` calls `TeeState::with_mocked_participant_attestations(initial_participants)`, which installs mock TEE attestations for all initial participants: [4](#0-3) 

This means the attacker does not need real TEE hardware to set up a functioning MPC node after seizing the contract — the mock attestation path is already used for initial participants.

**Exploit steps:**

1. Attacker monitors the NEAR blockchain for a new deployment of the MPC contract binary.
2. Attacker calls `init({ parameters: { threshold: 1, participants: [["attacker.near", 0, {...}]] } })` before the legitimate deployer does.
3. Deployer's `init()` call panics: `#[init]` detects state already exists and aborts.
4. Attacker is now the sole participant with `threshold = 1`.
5. Attacker runs their own MPC node (open-source software, mock attestations accepted).
6. Attacker calls `vote_add_domains(...)` to add a signing domain.
7. Attacker's node calls `start_keygen_instance(...)` and `vote_pk(...)` to register a key.
8. Attacker calls `respond(request, signature)` to issue signatures for arbitrary foreign-chain transactions.

The sandbox test confirms `#[init]` prevents re-initialization but does **not** restrict who calls it first: [5](#0-4) 

---

### Impact Explanation

**Critical — Unauthorized threshold signature issuance.** The attacker gains sole control of the MPC network's signing authority. They can issue threshold signatures for arbitrary foreign-chain transactions (ECDSA via `respond`, EdDSA via `respond`, CKD via `respond_ckd`) without any legitimate participant authorization. All governance functions (`vote_add_domains`, `vote_new_parameters`, `propose_update`, `vote_update`) are also under attacker control. Legitimate participants are permanently locked out of the contract.

This matches the allowed impact: *"Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization."*

---

### Likelihood Explanation

The NEAR blockchain is public; any observer can detect a contract deployment. The deployment scripts explicitly separate deployment from initialization, creating a reliable window. The attack requires no special hardware, no privileged access, no collusion, and no leaked keys — only the ability to submit a NEAR transaction. The MPC node software is open-source and the mock attestation path removes the TEE hardware barrier for the initial setup.

---

### Recommendation

Combine contract deployment and `init()` into a single **batch transaction** (the standard NEAR pattern for preventing initialization front-running). Alternatively, add an explicit caller check inside `init()`:

```rust
assert_eq!(
    env::predecessor_account_id(),
    env::current_account_id(),
    "init must be called by the contract account itself"
);
```

This mirrors the protection already applied to `init_running()` via `#[private]`.

### Proof of Concept

```bash
# Step 1: Attacker observes deployment of mpc-contract.near on-chain.

# Step 2: Attacker front-runs init() with themselves as sole participant, threshold=1.
near contract call-function as-transaction mpc-contract.near init \
  json-args '{
    "parameters": {
      "threshold": 1,
      "participants": {
        "next_id": 1,
        "participants": [["attacker.near", 0, {"url": "https://attacker-node.example", "tls_public_key": "<attacker_ed25519_pk>"}]]
      }
    }
  }' \
  prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
  sign-as attacker.near network-config mainnet sign-with-keychain send

# Step 3: Legitimate deployer's init() call fails — state already exists.

# Step 4: Attacker's MPC node (running open-source software) calls vote_add_domains,
#         start_keygen_instance, vote_pk to register a key under attacker control.

# Step 5: Attacker issues signatures for arbitrary foreign-chain transactions via respond().
```
