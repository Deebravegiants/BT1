**`blob_to_keyset` validation (the gap):** [1](#0-0) 

After `from_bin` for both `Q` and `K`, there is no `curve.check()` call and no `is_in_subgroup()` call.

**Contrast with `blob_to_key` in the sibling `eddsa_2p` API:** [2](#0-1) 

`blob_to_key` explicitly calls `curve.check(key.Q)` which internally calls `is_in_subgroup()`.

**`ecurve_t::check()` does the subgroup guard:** [3](#0-2) 

**`from_bin` does NOT check subgroup membership — confirmed by test:** [4](#0-3) 

A torsion point (e.g., the order-2 point) passes `from_bin` with `SUCCESS` but fails `curve.check()`.

**How `derive_keys` uses the torsion Q:** [5](#0-4) 

`Q = key.root.Q` is used directly. `Q_derived = Q + delta_G` and `derived_keys[i].Q = Q_derived + non_hard_delta_G`. If Q is a torsion point T, then `derived_keys[i].Q = T + (delta + non_hard_delta)*G`, which is **not** in the prime-order subgroup.

**For K specifically — the ZK proof acts as a natural guard:**

The `zk_dh1.verify(P, other_K_share, Z1, sid, 1)` call verifies that `Z1 = k_share * P` against `other_K_share = K - K_share`. If K is a torsion point, `other_K_share` would not equal the actual `k1_share * G`, so the ZK proof would fail. **A torsion K is caught by the protocol; a torsion Q is not.**

**The derived key blobs are then serialized without any subgroup check:** [6](#0-5) 

`serialize_eddsa2pc_key_blob` serializes the torsion-contaminated `derived_keys[i].Q` into the output blobs, and `derive_eddsa_2p_keys` returns `SUCCESS`.

**When those derived blobs are later used for signing, `blob_to_key` rejects them:** [2](#0-1) 

So the concrete outcome is: `derive_eddsa_2p_keys` returns `SUCCESS` with derived key blobs that contain torsion-contaminated public keys, but those blobs fail deserialization at the signing stage.

### Title
Missing subgroup check on `root.Q` in `blob_to_keyset` allows torsion-contaminated derived key blobs to be silently accepted — (`src/cbmpc/api/hd_keyset_eddsa_2p.cpp`)

### Summary
`blob_to_keyset` calls `from_bin` on `root_Q_compressed` and `root_K_compressed` but omits the `curve.check()` call that the sibling `blob_to_key` function performs. A crafted keyset blob with a torsion `root_Q_compressed` passes deserialization with `SUCCESS`, propagates into `derive_keys`, and causes all output `eddsa_2p` key blobs to carry torsion-contaminated public keys. The API returns `SUCCESS` for the derivation, but the resulting blobs are unusable for signing.

### Finding Description
In `src/cbmpc/api/hd_keyset_eddsa_2p.cpp`, `blob_to_keyset` (lines 63–81) deserializes `root_Q_compressed` and `root_K_compressed` via `from_bin` but never calls `curve.check()` on either point. `ecurve_ed_t::from_bin` only verifies that the encoding is a valid curve point; it does not check subgroup membership. The order-2 torsion point (y = p−1, x = 0) is a valid on-curve encoding that passes `from_bin` with `SUCCESS` but fails `curve.check()` / `is_in_subgroup()`.

The sibling function `blob_to_key` in `src/cbmpc/api/eddsa2pc.cpp` (lines 40–42) correctly calls `curve.check(key.Q)` after `from_bin`. The keyset deserialization path lacks this guard entirely.

In `derive_keys` (`src/cbmpc/protocol/hd_keyset_eddsa_2p.cpp`, lines 109, 149, 159), `Q = key.root.Q` is used directly to compute `Q_derived = Q + delta*G` and `derived_keys[i].Q = Q_derived + non_hard_delta*G`. No ZK proof or subgroup check is applied to Q at any point in the derivation. The torsion component propagates additively into every derived public key.

For `K`, the VRF ZK proof (`zk_dh1.verify` / `zk_dh2.verify`) implicitly catches a torsion K because `other_K_share = K − K_share` would not match the actual `k_share * G`, causing proof verification to fail. So only the Q path is exploitable.

### Impact Explanation
An attacker who can supply a crafted keyset blob (e.g., via a compromised persistence layer, a migration API, or any path where the caller does not generate the blob via `dkg`) can cause `derive_eddsa_2p_keys` to return `SUCCESS` while producing derived key blobs whose embedded public keys are not in the prime-order subgroup. These blobs appear structurally valid but fail at the signing stage when `deserialize_key_blob` → `blob_to_key` → `curve.check()` rejects the torsion-contaminated Q. This constitutes unsafe state acceptance and honest-party divergence: the derivation API signals success, but the output is cryptographically invalid and unusable.

### Likelihood Explanation
Requires the attacker to control the `keyset_blob` argument passed to `derive_eddsa_2p_keys` or `refresh`. In a typical deployment the blob is generated by `dkg` and stored locally, limiting exposure. However, any deployment that accepts externally-supplied keyset blobs (e.g., blob migration, cross-device sync, or testing harnesses) is directly reachable.

### Recommendation
Add `curve.check()` calls in `blob_to_keyset` immediately after each `from_bin`, mirroring the pattern in `blob_to_key`:

```cpp
error_t rv = keyset.root.Q.from_bin(keyset.curve, blob.root_Q_compressed);
if (rv) return rv;
if (keyset.curve.check(keyset.root.Q)) return coinbase::error(E_FORMAT, "invalid keyset blob: Q not in subgroup");

rv = keyset.root.K.from_bin(keyset.curve, blob.root_K_compressed);
if (rv) return rv;
if (keyset.curve.check(keyset.root.K)) return coinbase::error(E_FORMAT, "invalid keyset blob: K not in subgroup");
```

### Proof of Concept
1. Encode the Ed25519 order-2 torsion point (y = 2²⁵⁵ − 20, x = 0) as a 32-byte compressed point (bytes: `0xec, 0xff×30, 0x7f`).
2. Construct a well-formed `keyset_blob_v1_t` with `root_Q_compressed` set to this torsion encoding, valid `root_K_compressed` (a legitimate subgroup point), and valid scalar shares.
3. Call `coinbase::api::hd_keyset_eddsa_2p::derive_eddsa_2p_keys(job, crafted_blob, hardened_path, non_hardened_paths, sid, out)`.
4. Assert the return value is `SUCCESS`.
5. Deserialize each element of `out` via `coinbase::api::eddsa_2p::deserialize_key_blob` (or inspect the embedded `Q_compressed` field directly) and verify that the public key is not in the prime-order subgroup — confirming torsion contamination propagated through the derivation.
