Audit Report

## Title
Uninitialized `tokenFeeBps` for Newly Added Tokens Allows Fee-Free Deposits - (File: `contracts/pools/RSETHPool.sol`)

## Summary
`addSupportedToken` sets `supportedTokenOracle[token]` and `tokenBridge[token]` but never initializes `tokenFeeBps[token]`, which Solidity defaults to `0`. Any depositor can call `deposit` during the window between `addSupportedToken` and a subsequent `setTokenFeeBps` call and receive rsETH with zero fee deducted, permanently depriving the protocol of fee revenue on those deposits.

## Finding Description
`addSupportedToken` (L637–656) pushes the token into `supportedTokenList`, sets `supportedTokenOracle[token]` and `tokenBridge[token]`, but leaves `tokenFeeBps[token]` at its Solidity default of `0`:

```solidity
// L651-653
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] is never written
```

The overloaded `viewSwapRsETHAmountAndFee(uint256, address)` (L335–336) reads this uninitialized value directly:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];   // == 0
fee = amount * feeBpsForToken / 10_000;         // == 0
```

`deposit` (L298–300) calls this function and accumulates the zero fee:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;   // += 0
IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
```

`setTokenFeeBps` (L583–594) is a completely independent admin call with no coupling to `addSupportedToken`. The `onlySupportedToken` modifier on `deposit` (L292) only checks `supportedTokenOracle[token] != address(0)`, which is satisfied immediately after `addSupportedToken` completes. There is no guard that prevents deposits while `tokenFeeBps[token]` is still zero.

The structural gap is unavoidable: `addSupportedToken` requires `TIMELOCK_ROLE` and `setTokenFeeBps` requires `DEFAULT_ADMIN_ROLE`; these are separate roles that may require separate governance transactions, making the zero-fee window span multiple blocks.

## Impact Explanation
**High — Theft of unclaimed yield.** Every deposit made during the zero-fee window pays `0` in protocol fees. The `feeEarnedInToken[token]` accumulator records `0` for these deposits, making the loss permanent and unrecoverable. An attacker who deposits a large amount during this window receives the full rsETH equivalent of their tokens with no fee deducted, extracting value that should have accrued to the protocol as fee yield.

## Likelihood Explanation
**Medium.** Adding a new supported token is a routine protocol operation. The two-step pattern is structurally required by the current code, making the zero-fee window unavoidable on every new token addition. Any depositor — not only a sophisticated front-runner — who deposits during the multi-block window between `addSupportedToken` and `setTokenFeeBps` benefits. A depositor monitoring on-chain `AddSupportedToken` events can exploit this immediately and repeatably for every new token addition.

## Recommendation
Add a `feeBps` parameter directly to `addSupportedToken` and initialize `tokenFeeBps[token]` atomically in the same call:

```solidity
function addSupportedToken(
    address token,
    address oracle,
    address bridge,
    uint256 feeBps
) external onlyRole(TIMELOCK_ROLE) {
    ...
    if (feeBps > 10_000) revert InvalidFeeAmount();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    tokenFeeBps[token] = feeBps;   // initialize atomically
    emit AddSupportedToken(token, oracle, bridge);
}
```

This eliminates the zero-fee window entirely by ensuring `tokenFeeBps[token]` is set in the same transaction that makes the token depositable.

## Proof of Concept
1. `TIMELOCK_ROLE` calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` is `0`, `supportedTokenOracle[wstETH]` is set.
2. Attacker observes the `AddSupportedToken` event and calls `deposit(wstETH, 1_000e18, "")`.
3. `viewSwapRsETHAmountAndFee(1_000e18, wstETH)` computes `fee = 1_000e18 * 0 / 10_000 = 0`; attacker receives the full rsETH equivalent of 1,000 wstETH with no fee deducted.
4. `feeEarnedInToken[wstETH] += 0` — loss is recorded as zero and is permanent.
5. `DEFAULT_ADMIN_ROLE` later calls `setTokenFeeBps(wstETH, 50)`, but the attacker's deposit has already settled at 0% fee.
6. This is repeatable for every new token addition.

**Foundry test sketch:**
```solidity
function test_zeroFeeWindowOnAddToken() public {
    vm.prank(timelockRole);
    pool.addSupportedToken(address(wstETH), oracle, bridge);
    // tokenFeeBps[wstETH] == 0 here
    uint256 amount = 1_000e18;
    deal(address(wstETH), attacker, amount);
    vm.startPrank(attacker);
    wstETH.approve(address(pool), amount);
    pool.deposit(address(wstETH), amount, "");
    vm.stopPrank();
    assertEq(pool.feeEarnedInToken(address(wstETH)), 0); // fee permanently lost
}
```
