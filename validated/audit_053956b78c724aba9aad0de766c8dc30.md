Audit Report

## Title
`LRTWithdrawalManager.instantWithdrawal()` Lacks Minimum Output Guard, Exposing Users to Oracle Slippage With No Recourse - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`instantWithdrawal()` burns the caller's rsETH at line 229 using a live oracle rate with no `minAssetAmountOut` guard. If the oracle rate moves unfavorably between transaction submission and inclusion, the rsETH is permanently destroyed and the user receives fewer assets than expected with no on-chain protection. This contrasts directly with `LRTDepositPool.depositETH` and `depositAsset`, which both enforce a `minRSETHAmountExpected` parameter, demonstrating the protocol is aware of the pattern but omitted it on the withdrawal side.

## Finding Description

`instantWithdrawal` (lines 212–253, `contracts/LRTWithdrawalManager.sol`) accepts only `asset`, `rsETHUnstaked`, and `referralId`. The payout is computed at line 228 via `getExpectedAssetAmount`, which divides live oracle prices:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
// LRTWithdrawalManager.sol line 593
```

The rsETH burn executes unconditionally at line 229 before any availability check:

```solidity
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked); // line 229
...
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) { // line 231
    revert CantInstantWithdrawMoreThanAvailable();
}
```

There is no check of the form `if (assetAmountUnlocked < minAssetAmountOut) revert`. Once line 229 executes, the rsETH is gone regardless of how small `assetAmountUnlocked` turns out to be relative to the user's expectation at submission time.

By contrast, `LRTDepositPool.depositETH` (line 87) and `depositAsset` (line 111) both call `_beforeDeposit(asset, amount, minRSETHAmountExpected)`, which enforces the minimum output before any state change. The same gap exists in `RSETHPool.deposit()` (line 265–278) and `RSETHPoolNoWrapper.deposit()` (lines 231–244), where `viewSwapRsETHAmountAndFee` computes rsETH output from a live oracle rate and transfers it with no minimum-output guard.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user calling `instantWithdrawal` receives the oracle-determined fair value at execution time, but that value may be materially less than what the oracle quoted at submission time. Because the rsETH burn is irreversible and precedes the payout, the user has no on-chain recourse. The same oracle-rate exposure applies to depositors in the RSETHPool family, who may receive fewer rsETH tokens than the rate quoted at submission time.

## Likelihood Explanation

**Medium.** Both `rsETHPrice` (derived from total protocol assets and total rsETH supply) and LST asset prices (Chainlink feeds) update regularly. On a congested network, a transaction can sit in the mempool for multiple blocks during which oracle prices shift. No attacker action is required; normal market volatility is sufficient. The instant-withdrawal path is specifically designed for time-sensitive users, making them more likely to transact during volatile periods.

## Recommendation

1. Add a `minAssetAmountOut` parameter to `instantWithdrawal` and revert before the burn if `assetAmountUnlocked < minAssetAmountOut`, mirroring the `minRSETHAmountExpected` guard in `LRTDepositPool`.
2. Add a `minRsETHAmountOut` parameter to every `deposit()` overload in the RSETHPool family and revert if the computed `rsETHAmount` falls below it.

## Proof of Concept

1. User observes `rsETHPrice = 1.05e18` and `getAssetPrice(stETH) = 1.00e18`, expecting `1.05 stETH` for `1 rsETH`.
2. User submits `instantWithdrawal(stETH, 1e18, "")`.
3. Before inclusion, an oracle update sets `rsETHPrice = 1.00e18` (e.g., a slashing event reduces total assets).
4. `getExpectedAssetAmount` returns `1.00e18` (1 stETH) instead of `1.05e18`.
5. Line 229 burns the user's `1e18` rsETH unconditionally.
6. User receives `1.00 stETH` minus the instant-withdrawal fee — `0.05 stETH` less than expected — with no on-chain protection and no ability to cancel.

**Foundry fork test plan:** Fork mainnet, mock `lrtOracle.rsETHPrice()` to return `1.05e18` at call-setup time, then update the mock to `1.00e18` before `instantWithdrawal` executes (simulating an oracle update in the same block). Assert that `userAmount` received equals `1.00e18 * (1 - fee/10000)` rather than `1.05e18 * (1 - fee/10000)`, and that no revert occurred despite the shortfall.
