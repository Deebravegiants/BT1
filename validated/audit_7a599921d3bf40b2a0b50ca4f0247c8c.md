Audit Report

## Title
`collectFees` Over-Allocates Pool Tokens When Both Spread and Notional Fees Are Active, Causing Pool Insolvency - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
When both spread fees and notional fees are non-zero, `collectFees` computes the spread fee payout from `surplus0Scaled`/`surplus1Scaled` (which already excludes `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled`), and then separately pays out the full notional fee accumulators. The total tokens transferred can exceed the actual surplus held by the pool, draining tokens that belong to LPs and causing pool insolvency. This is confirmed by the protocol's own test `test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive`.

## Finding Description

In `MetricOmmPool.collectFees` (lines 385–430):

```solidity
uint256 surplus0Scaled =
  balance0() * TOKEN_0_SCALE_MULTIPLIER - uint256(binTotals.scaledToken0) - notionalFee0AmountScaled;
uint256 surplus1Scaled =
  balance1() * TOKEN_1_SCALE_MULTIPLIER - uint256(binTotals.scaledToken1) - notionalFee1AmountScaled;
```

`surplus0Scaled` is the balance minus LP bins minus the notional accumulator — i.e., it represents only the **spread fee surplus**. The code then computes spread fee payouts as a fraction of this surplus (correct), but also pays out the full notional accumulator (`notionalFee0AmountScaled`) on top:

```solidity
uint256 notionalFee0ToProtocolScaled = notionalFee0AmountScaled - notionalFee0ToAdminScaled;
...
uint256 totalFee0ToProtocolScaled = spreadFee0ToProtocolScaled + notionalFee0ToProtocolScaled;
```

The combined transfer is:
- `spreadFee0ToAdmin + spreadFee0ToProtocol` ≈ `surplus0Scaled` (the full spread surplus, since `adminSpreadFeeE6 + protocolSpreadFeeE6 = spreadSumE6`)
- `+ notionalFee0ToAdmin + notionalFee0ToProtocol` = `notionalFee0AmountScaled`

Total attempted payout = `surplus0Scaled + notionalFee0AmountScaled`, but the actual available balance above LP bins is only `surplus0Scaled + notionalFee0AmountScaled` (the raw surplus before subtracting notional). However, `surplus0Scaled` was already computed by subtracting `notionalFee0AmountScaled`, so the spread fee portion consumes the entire raw surplus, and then the notional portion tries to take an additional `notionalFee0AmountScaled` from LP principal.

The protocol's own test confirms this:
```solidity
assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
```

The `unchecked` block at line 390 suppresses overflow/underflow detection, so the ERC-20 `transfer` calls will attempt to send more tokens than the pool holds above LP principal, either reverting (bricking fee collection) or — if the pool has any extra balance — draining LP funds.

**Call path:**
1. Any address calls `MetricOmmPoolFactory.collectPoolFees(pool)` (permissionless)
2. Factory calls `pool.collectFees(protocolSpreadFeeE6, adminSpreadFeeE6, protocolNotionalFeeE8, adminNotionalFeeE8, adminFeeDestination)`
3. Both `spreadSumE6 > 0` and `notionalSumE8 > 0`
4. `surplus0Scaled` is computed excluding notional accumulator
5. Spread fees consume the full spread surplus
6. Notional fees attempt to additionally drain `notionalFeeToken0Scaled` from LP principal
7. Pool transfers more tokens than it should, or reverts, bricking fee collection permanently

## Impact Explanation
Direct loss of LP principal: when both fee types are active, `collectFees` attempts to transfer `notionalFeeToken0Scaled + notionalFeeToken1Scaled` beyond the available surplus, pulling from LP-owned token balances. This constitutes pool insolvency — LP claims are no longer fully backed. The permissionless `collectPoolFees` entry point means any address can trigger this at any time. Severity: **High** (direct loss of LP principal / pool insolvency).

## Likelihood Explanation
The condition requires both `spreadSumE6 > 0` and `notionalSumE8 > 0` simultaneously, which is a normal and documented operational configuration. `collectPoolFees` is explicitly permissionless (any keeper, bot, or attacker can call it). After any swaps accumulate notional fees, a single call to `collectPoolFees` triggers the over-allocation. Repeatability is limited only by the notional accumulator being reset to zero after each call, but the damage to LP principal is permanent per call.

## Recommendation
The root cause is that `surplus0Scaled` is computed by subtracting the notional accumulator, but then the notional accumulator is paid out *in addition to* the full spread surplus. Fix by computing the spread fee payout from the raw surplus (before subtracting notional), or by capping the total payout to the actual available balance:

```solidity
// Raw surplus = balance - binTotals (includes both spread and notional fees)
uint256 rawSurplus0Scaled = balance0() * TOKEN_0_SCALE_MULTIPLIER - uint256(binTotals.scaledToken0);
// Spread surplus = raw surplus minus notional accumulator
uint256 spreadSurplus0Scaled = rawSurplus0Scaled - notionalFee0AmountScaled;

// Spread fees are a fraction of spreadSurplus only
uint256 spreadFee0ToAdmin = (spreadSurplus0Scaled * adminSpreadFeeE6_) / spreadSumE6;
uint256 spreadFee0ToProtocol = (spreadSurplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
// Notional fees are paid from the notional accumulator (already separated)
```

This ensures `spreadFee0ToAdmin + spreadFee0ToProtocol + notionalFee0ToAdmin + notionalFee0ToProtocol <= rawSurplus0Scaled`.

## Proof of Concept

The protocol's own test at `metric-core/test/MetricOmmPool.notionalFee.t.sol:211-266` already demonstrates the over-allocation:

```solidity
function test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive() public {
    // Setup pool with both spread and notional fees active
    pool.setPoolFees(PROTOCOL_FEE + ADMIN_FEE, FEE_1_PCT_E8);
    _addLiquidity(1, -5, 4, 100_000, 0);
    // Run swaps to accumulate both fee types
    for (uint256 i = 0; i < 8; i++) {
        _swap(0, users[0], false, int128(50_000), type(uint128).max);
        _swap(0, users[0], true, int128(10_000), 0);
    }
    // Demonstrates total attempted payout > available surplus
    assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
    assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
}
```
