### Title
Missing State Diff Commitment Verification in P2P Sync Allows Unauthenticated Peer to Corrupt `compiled_class_hash` Mapping — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

The `parse_data_for_block` function in `StateDiffStreamBuilder` assembles a `ThinStateDiff` from peer-supplied `StateDiffChunk` messages but **never verifies the assembled state diff against the `state_diff_commitment` stored in the block header**. An unauthenticated p2p peer can send a `StateDiffChunk::DeclaredClass` with a correct `class_hash` but an arbitrary `compiled_class_hash`. The wrong mapping is inserted into `class_hash_to_compiled_class_hash` and written to storage unconditionally.

### Finding Description

**Entrypoint**: Any unauthenticated p2p peer that responds to a `StateDiff` query from a syncing node.

**Step 1 — Header accepted without signature verification.**
`HeaderStreamBuilder::parse_data_for_block` only checks block number ordering and that exactly one signature is present. It does **not** verify the `BlockSignature` against the sequencer's public key, and does not verify the `block_hash`. [1](#0-0) 

This means an attacker can also forge the header (including `state_diff_length` and `state_diff_commitment`), but the attack works even against a legitimately-sourced header.

**Step 2 — State diff assembled without commitment check.**
`StateDiffStreamBuilder::parse_data_for_block` reads `state_diff_length` from the stored header to bound the loop, but reads `state_diff_commitment` from the header **nowhere**. After the loop, only structural checks are performed: total length matches, no duplicate keys, no empty chunks. [2](#0-1) 

**Step 3 — Wrong `compiled_class_hash` inserted.**
`unite_state_diffs` inserts `(declared_class.class_hash, declared_class.compiled_class_hash)` into `state_diff.class_hash_to_compiled_class_hash` with no content validation. [3](#0-2) 

**Step 4 — Corrupted state diff written to storage.**
`write_to_storage` calls `append_state_diff` directly, with no hash verification. [4](#0-3) 

**The commitment exists but is never used here.**
`calculate_state_diff_hash` chains `class_hash_to_compiled_class_hash` into the Poseidon hash, so a wrong `compiled_class_hash` would produce a different commitment — but this check is never performed in the p2p sync path. [5](#0-4) 

The `Committer`'s `verify_state_diff_hash` flag exists in `apollo_committer` and is used in the batcher's block-building path, but the p2p sync client bypasses the committer entirely and writes directly to storage. [6](#0-5) 

### Impact Explanation

The corrupted `class_hash_to_compiled_class_hash` entry in `ThinStateDiff` is the authoritative mapping used by the node to resolve which compiled (CASM/native) artifact to execute for a declared Sierra class. Any subsequent transaction that invokes the affected class will look up the wrong compiled class hash, causing wrong execution results, wrong fee accounting, and wrong block production if the node is a sequencer. This matches the **Critical** impact category: "Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution."

### Likelihood Explanation

The attack requires only that the attacker be a p2p peer that the syncing node queries for state diffs. The p2p sync protocol is open to any peer. The attacker needs to know the correct `class_hash` for a class declared in the target block (publicly observable on-chain) and supply a wrong `compiled_class_hash` of the same byte length. No privileged access is required.

### Recommendation

After assembling the full `ThinStateDiff`, compute `calculate_state_diff_hash(&result)` and compare it against `header.state_diff_commitment`. If they differ, treat the peer as bad (`BadPeerError`) and discard the state diff. This is the same pattern already implemented in `Committer::commit_or_load` and should be applied symmetrically in `StateDiffStreamBuilder::parse_data_for_block`. [7](#0-6) 

### Proof of Concept

In a unit test over `StateDiffStreamBuilder::parse_data_for_block`:

1. Store a block header with `state_diff_length = 1` and `state_diff_commitment = calculate_state_diff_hash(correct_state_diff)`.
2. Send a single `StateDiffChunk::DeclaredClass { class_hash: C, compiled_class_hash: WRONG }` where `WRONG ≠ correct_compiled_class_hash`.
3. The length check passes (1 == 1). No commitment check is performed.
4. `parse_data_for_block` returns `Ok(Some((result, block_number)))`.
5. `write_to_storage` calls `append_state_diff`, persisting `class_hash_to_compiled_class_hash = { C => WRONG }`.
6. Assert `storage.get_state_diff(block_number).class_hash_to_compiled_class_hash[C] == WRONG` — the corruption is confirmed.
7. Any subsequent class lookup for `C` returns `WRONG`, causing wrong compiled class selection for execution.
