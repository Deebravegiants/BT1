Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison for ETH deposits: the ETH branch tests only whether the existing total already exceeds the limit, without adding the incoming `amount`. The ERC20 branch correctly includes `amount`. Any depositor can push ETH holdings above the configured deposit cap in a single transaction.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682), the ETH branch at line 679 evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)`, omitting `amount`. The ERC20 branch at line 681 correctly evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. [1](#0-0) 

`_beforeDeposit` (lines 648–670) calls this function and reverts with `MaximumDepositLimitReached` only when it returns `true`. [2](#0-1) 

`depositETH` (lines 76–93) is a public payable function that passes `msg.value` as `depositAmount` into `_beforeDeposit`. [3](#0-2) 

Because the ETH check ignores `amount`, as long as `totalAssetDeposits ≤ depositLimit`, the function returns `false` regardless of how large `msg.value` is, and the deposit proceeds without reverting.

## Impact Explanation
The deposit limit is a protocol-level risk control governing ETH exposure in EigenLayer strategies. When bypassed, the protocol accumulates more ETH than governance has approved, causing the contract to fail to enforce its promised deposit cap. Deposited funds are not lost or stolen.

**Impact: Low** — Contract fails to deliver promised returns (deposit cap enforcement), but deposited funds are not lost.

## Likelihood Explanation
`depositETH` is the primary public entry point for ETH deposits. No special role or precondition is required beyond the contract being unpaused and ETH being a supported asset. The bypass condition (`totalAssetDeposits ≤ limit`) is the normal operating state of the protocol, making this reachable by any depositor at any time.

**Likelihood: Medium** — Reachable by any unprivileged depositor during normal protocol operation.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 path:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets ETH deposit limit to 100,000 ETH via `LRTConfig.updateAssetDepositLimit`.
2. Current `getTotalAssetDeposits(ETH_TOKEN)` = 99,999 ETH.
3. Attacker calls `depositETH{value: 2000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `99,999 > 100,000` → `false`; no revert.
5. 2,000 ETH is accepted; total becomes 101,999 ETH — 1,999 ETH above the configured cap.
6. The deposit limit is silently bypassed in a single transaction by any depositor.

**Foundry test plan:** Deploy `LRTDepositPool` with ETH limit set to `100_000 ether`. Seed `getTotalAssetDeposits` to `99_999 ether`. Call `depositETH{value: 2000 ether}` from an unprivileged address. Assert the call succeeds and `getTotalAssetDeposits(ETH_TOKEN)` exceeds `100_000 ether`.

    function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
