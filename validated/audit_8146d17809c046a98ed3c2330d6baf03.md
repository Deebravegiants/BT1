### Title
Rounding Down to Zero in `viewSwapRsETHAmountAndFee` Allows Depositing ETH/Tokens Without Receiving wrsETH - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
All L2 pool contracts share a `viewSwapRsETHAmountAndFee` function that computes the wrsETH/rsETH output via integer division. For any deposit of 1 wei of ETH or a supported LST token, the division truncates to zero because `rsETHToETHrate > 1e18` (rsETH accrues yield over time). No zero-value guard exists on the result before the mint/transfer executes, so the depositor's ETH or tokens are permanently absorbed into the pool while they receive nothing.

### Finding Description
In every pool variant (`RSETHPoolV3`, `RSETHPoolNoWrapper`, `RSETHPool`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV3ExternalBridge`) the ETH swap path computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // RSETHPoolV3.sol L307
```

and the token swap path computes:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;  // RSETHPoolV3.sol L334
```

Both are plain integer divisions with no rounding-up and no post-computation zero check. Because rsETH is a yield-bearing token, `rsETHToETHrate` is always strictly greater than `1e18` after genesis. Therefore, for `amountAfterFee = 1` wei:

- ETH path: `1 * 1e18 / 1.05e18 = 0` (truncated)
- stETH path: `1 * 1.0e18 / 1.05e18 = 0` (truncated)

The deposit functions only guard against `amount == 0`:

```solidity
if (amount == 0) revert InvalidAmount();          // RSETHPoolV3.sol L256, L282
```

After that guard, the token transfer executes unconditionally, then `wrsETH.mint(msg.sender, 0)` (or `rsETH.safeTransfer(msg.sender, 0)`) is called with the zero result. The deposited value is silently absorbed into the pool's bridgeable balance.

The same pattern is present identically across all five pool contracts confirmed by grep: `RSETHPool.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`.

### Impact Explanation
Any depositor who sends a dust amount (e.g., 1 wei ETH or 1 wei of a supported LST) receives zero wrsETH/rsETH in return. The deposited value is permanently added to the pool's non-fee balance and will be swept to L1 by the next `bridgeAssets` / `moveAssetsForBridging` call, effectively transferring the user's funds to the protocol without issuing any receipt token. The contract fails to deliver its promised return for these inputs.

**Impact: Low** — Contract fails to deliver promised returns. Individual loss per call is 1 wei; no single call causes material loss, but the invariant "non-zero deposit ⟹ non-zero wrsETH" is broken.

### Likelihood Explanation
The condition is reachable by any unprivileged caller on any supported L2 chain with a single public `deposit()` call. No special setup is required. The rsETH rate exceeds 1e18 on every live deployment (rsETH has been accruing yield since launch), so the rounding-to-zero condition is permanently active for 1-wei inputs. Likelihood is **Low** because rational users do not intentionally deposit 1 wei, but the path is always open and requires no attacker capability beyond calling a public function.

### Recommendation
Add a zero-value check on `rsETHAmount` immediately after `viewSwapRsETHAmountAndFee` returns, in every `deposit` function across all pool contracts:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, enforce a protocol-level minimum deposit that guarantees `rsETHAmount >= 1` given the current rate, similar to the `minAmountToDeposit` guard used in `LRTDepositPool._beforeDeposit`.

### Proof of Concept
1. Assume `rsETHToETHrate = 1.05e18` (realistic post-genesis value).
2. Caller sends `msg.value = 1` wei to `RSETHPoolV3.deposit(string)`.
3. `amount == 0` check passes (1 ≠ 0).
4. `viewSwapRsETHAmountAndFee(1)` executes: `fee = 1 * feeBps / 10_000 = 0`; `amountAfterFee = 1`; `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`.
5. `feeEarnedInETH += 0` — no fee recorded.
6. `wrsETH.mint(msg.sender, 0)` — caller receives 0 wrsETH.
7. The 1 wei ETH is now part of the pool's bridgeable balance and will be forwarded to L1 on the next `bridgeAssets` call.
8. Caller has permanently lost 1 wei with no receipt token issued.
