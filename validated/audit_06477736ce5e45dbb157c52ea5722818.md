Audit Report

## Title
Floor Division in `removeLiquidity` Burns LP Shares and Reduces Bin Balances While Returning Zero Native Tokens - (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`LiquidityLib.removeLiquidity` performs two sequential floor divisions: share-to-scaled at lines 205–206, then scaled-to-native at line 275. For tokens with a large `token0ScaleMultiplier` (e.g., USDC with 6 decimals → multiplier = 10¹²), the scaled amount removed from the bin can be non-zero while the native transfer rounds to zero. The LP's shares are burned and `binTotals.scaledToken0` is permanently decremented, but no tokens are transferred. The orphaned scaled units surface as surplus in `collectFees` and are redistributed to protocol/admin fee collectors.

## Finding Description
In `removeLiquidity`, the share burn and bin accounting update are unconditional: [1](#0-0) 

`binState.token0BalanceScaled`, `binTotalShares`, and `positionBinShares` are all decremented before any transfer check. Then `binTotals.scaledToken0` is decremented: [2](#0-1) 

The native transfer amount is computed via floor division in `_deltasScaledToExternal`: [3](#0-2) 

If `totalToken0ToRemoveScaled < token0ScaleMultiplier`, `amount0Removed = 0` and the `safeTransfer` at line 242 is skipped: [4](#0-3) 

The orphaned scaled units (still held as ERC-20 balance but no longer tracked in `binTotals.scaledToken0`) appear as surplus in `collectFees`: [5](#0-4) 

The asymmetry is confirmed: `addLiquidity` uses `Math.Rounding.Ceil` for the scaled-to-external conversion (line 142), while `removeLiquidity` uses `Math.Rounding.Floor` (line 240): [6](#0-5) [7](#0-6) 

## Impact Explanation
An LP burns valid shares and has their bin balance permanently reduced, but receives zero native tokens. The lost value is silently redirected to fee collectors via the surplus calculation in `collectFees`. This is a direct, unrecoverable loss of LP principal with no revert or warning. The `TOKEN_0_SCALE_MULTIPLIER` immutable confirms the scaling design: [8](#0-7) 

For a USDC pool (`token0ScaleMultiplier = 10¹²`), any removal where `totalToken0ToRemoveScaled ∈ [1, 10¹² − 1]` produces `amount0Removed = 0`. The maximum loss per call is just under 1 USDC. Because `addLiquidity` uses `ceilDiv`, an LP who adds a small position pays at least 1 USDC but recovers 0 USDC on removal — a 100% loss of their deposit.

## Likelihood Explanation
No privileged access is required. Any LP interacting with a pool whose token has fewer than 18 decimals (USDC, USDT, WBTC) is exposed. The `MINIMAL_MINTABLE_LIQUIDITY` guard at line 200 only prevents leaving a dust remainder in the position; it does not prevent removing a quantity of shares whose scaled value is below `token0ScaleMultiplier`. The asymmetric rounding guarantees the LP always pays at least 1 native unit when adding but may receive 0 when removing. [9](#0-8) 

## Recommendation
1. **Revert on zero-output removal**: After computing `amount0Removed` and `amount1Removed`, revert if either is zero while the corresponding scaled amount is non-zero:
   ```solidity
   if (totalToken0ToRemoveScaled > 0 && amount0Removed == 0) revert ZeroTokenOutput();
   if (totalToken1ToRemoveScaled > 0 && amount1Removed == 0) revert ZeroTokenOutput();
   ```
2. **Alternatively**, expose `minAmount0Out` / `minAmount1Out` parameters so callers can enforce a minimum.
3. **For the share-to-scaled step**, consider using `Math.ceilDiv` symmetrically with `addLiquidity` so the LP always recovers at least 1 scaled unit per share removed.

## Proof of Concept
**Setup**: USDC/ETH pool, `token0 = USDC` (6 decimals), `token0ScaleMultiplier = 10¹²`, `initialAmount0PerShareE18 = 1e6`, `MINIMAL_MINTABLE_LIQUIDITY = 1000`.

**Add liquidity** (1000 shares, empty bin):
```
amount0Scaled = ceilDiv(1e18 × 1000, 1e18) = 1000   // scaled units
amount0Added  = ceilDiv(1000, 1e12)         = 1       // 1 USDC paid by LP
```

**Remove liquidity** (all 1000 shares):
```
amount0Scaled   = 1000 × 1000 / 1000 = 1000          // bin balance decremented
amount0Removed  = 1000 / 1e12        = 0              // 0 USDC transferred to LP
```

LP paid 1 USDC, received 0 USDC. The 1000 scaled units are orphaned in the pool's ERC-20 balance and swept to fee collectors on the next `collectFees` call.
