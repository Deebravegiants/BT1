### Title
Expired DRep Reactivated via `updateVotingDRepExpiries` Without Expiry Check — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

`updateVotingDRepExpiries` resets a DRep's stored expiry epoch whenever it casts a vote, without first checking whether the DRep is already expired. An expired DRep can therefore reactivate itself by submitting a vote, causing its vote to count in governance ratification as if it were an active DRep. This is the direct structural analog of the Phala `LocalCache#set_expire` bug: an expiry-update path that mutates state without guarding against the already-expired case.

---

### Finding Description

**Root cause — `updateVotingDRepExpiries`** in `Certs.hs`:

```haskell
updateVotingDRepExpiries tx currentEpoch drepActivity certState =
  let numDormantEpochs = certState ^. certVStateL . vsNumDormantEpochsL
      updateVSDReps vsDReps =
        Map.foldlWithKey'
          ( \dreps voter _ -> case voter of
              DRepVoter cred ->
                Map.adjust
                  (drepExpiryL .~ computeDRepExpiry drepActivity currentEpoch numDormantEpochs)
                  cred
                  dreps
              _ -> dreps
          )
          vsDReps
          (unVotingProcedures $ tx ^. bodyTxL . votingProceduresTxBodyL)
   in certState & certVStateL . vsDRepsL %~ updateVSDReps
``` [1](#0-0) 

For every `DRepVoter` in the transaction's voting procedures, `Map.adjust` unconditionally overwrites `drepExpiryL` with `computeDRepExpiry drepActivity currentEpoch numDormantEpochs` — a future epoch — regardless of whether the DRep's actual expiry (`drepExpiry + numDormantEpochs`) has already passed.

**Contrast with `updateDormantDRepExpiry`**, which correctly guards against the expired case:

```haskell
updateExpiry =
  drepExpiryL
    %~ \currentExpiry ->
      let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
       in if actualExpiry < currentEpoch
            then currentExpiry   -- skip if already expired
            else actualExpiry
``` [2](#0-1) 

`updateDormantDRepExpiry` explicitly skips DReps whose actual expiry is already past. `updateVotingDRepExpiries` has no such guard.

**The GOV rule does not reject votes from expired DReps.** The `VotersDoNotExist` check only verifies that the DRep credential exists in `knownDReps = vsDReps certVState` (i.e., is registered), not that it is active:

```haskell
DRepVoter cred -> DRepVoter <$> internMap cred knownDReps
``` [3](#0-2) 

`checkVotersAreValid` for `DRepVoter` only calls `isDRepVotingAllowed (gasAction gas)`, which checks the action type, not the DRep's expiry status: [4](#0-3) 

**Ratification uses the stored `drepExpiry` directly.** After `updateVotingDRepExpiries` resets the stored value to a future epoch, `dRepAcceptedRatio` in `Ratify.hs` sees the DRep as active:

```haskell
Just drepState
  | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, skip
  | otherwise -> ...  -- vote counts
``` [5](#0-4) 

**The LEDGER rule applies `updateVotingDRepExpiries` before the GOV and CERTS sub-rules**, so the expiry is already reset by the time the vote is recorded and before the epoch-boundary pulser snapshot is taken: [6](#0-5) 

**The formal spec disagrees with this behavior.** The test `"expiry is not updated for inactive DReps"` is explicitly disabled with `disableInConformanceIt` because the implementation diverges from the spec — the spec requires that expired DReps do NOT receive expiry updates: [7](#0-6) 

---

### Impact Explanation

An expired DRep that controls or influences a governance vote can reactivate itself by submitting a vote on any live governance action. Once reactivated, its delegated stake counts toward the DRep acceptance ratio in ratification. If the DRep holds sufficient delegated stake, it can tip a governance action — parameter change, treasury withdrawal, hard-fork initiation, committee update, or constitution change — over the ratification threshold that it would not have reached with only legitimately active DReps. This constitutes an unauthorized governance action being enacted, matching the **Critical** impact category: *"Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted."*

---

### Likelihood Explanation

The attacker must control a DRep credential with non-trivial stake delegated to it and allow (or wait for) that DRep to expire. Expiry is a normal lifecycle event — any DRep that stops voting for `ppDRepActivity` epochs expires automatically. The attacker then submits a single transaction containing a vote on any live governance action. No privileged access, key compromise, or consensus majority is required. The entry point is an ordinary, unprivileged transaction.

---

### Recommendation

Add an expiry guard to `updateVotingDRepExpiries` mirroring the one already present in `updateDormantDRepExpiry`. Before resetting a DRep's expiry, check whether its actual expiry (`drepExpiry + numDormantEpochs`) has already passed the current epoch; if so, leave the stored value unchanged:

```haskell
updateVSDReps vsDReps =
  Map.foldlWithKey'
    ( \dreps voter _ -> case voter of
        DRepVoter cred ->
          Map.adjust
            ( \drepState ->
                let actualExpiry = binOpEpochNo (+) numDormantEpochs (drepState ^. drepExpiryL)
                 in if actualExpiry < currentEpoch
                      then drepState  -- already expired; do not reactivate
                      else drepState & drepExpiryL .~ computeDRepExpiry drepActivity currentEpoch numDormantEpochs
            )
            cred
            dreps
        _ -> dreps
    )
    vsDReps
    (unVotingProcedures $ tx ^. bodyTxL . votingProceduresTxBodyL)
```

This aligns the implementation with the formal specification and with the existing guard in `updateDormantDRepExpiry`.

### Proof of Concept

1. Set `ppDRepActivityL = EpochInterval 2` and `ppGovActionLifetimeL = EpochInterval 10`.
2. Register a DRep with 1 000 000 lovelace delegated to it.
3. Submit a governance proposal (e.g., `ParameterChange`) that requires DRep approval.
4. Pass 3 epochs without the DRep voting. The DRep is now expired (`isDRepExpired drep == True`).
5. Submit a transaction in which the expired DRep votes `VoteYes` on the proposal.
6. Observe that `updateVotingDRepExpiries` resets `drepExpiry` to `currentEpoch + drepActivity - numDormantEpochs` (a future epoch).
7. Pass to the epoch boundary. The pulser is initialized with the updated DRep state (future expiry).
8. Observe that `dRepAcceptedRatio` counts the DRep's vote, and the proposal is ratified — even though the DRep was expired at the time of voting.

The existing disabled test `"expiry is not updated for inactive DReps"` in `EpochSpec.hs` documents exactly this discrepancy between the implementation and the formal specification. [8](#0-7)
