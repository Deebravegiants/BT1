### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Arbitrary Derived Key to User - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract delivers a Confidential Key Derivation (CKD) response to users without any cryptographic verification when the legacy `AppPublicKey` variant is used. A single Byzantine MPC participant can call `respond_ckd` with a crafted `CKDResponse` containing arbitrary `big_y`/`big_c` values, and the contract will accept and deliver it to the user. This is the direct analog of the external report's "borrow without repayment check" pattern: the contract hands out a cryptographic output without verifying the obligation (correctness of the derived key) was fulfilled.

---

### Finding Description

In `respond_ckd`, the contract performs a cryptographic output check only for the `AppPublicKeyPV` variant. For the `AppPublicKey` (legacy, privately verifiable) variant, the match arm is explicitly empty — no verification is performed at all:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

After this match, the contract unconditionally calls `resolve_yields_for`, which resumes all queued yield promises with the unverified response bytes: [2](#0-1) 

The `AppPublicKeyPV` variant exists precisely because the `AppPublicKey` variant cannot be verified on-chain (the contract does not know the user's private scalar `a`). However, the `AppPublicKeyPV` variant proves the response is correct via a BLS12-381 pairing check: [3](#0-2) 

For `AppPublicKey`, no equivalent check exists. Any single attested participant can call `respond_ckd` with arbitrary `big_y` and `big_c` values, and the contract will deliver them to the user as if they were the legitimate MPC output.

The `respond_ckd` entry point only requires the caller to be an attested participant — a single Byzantine node below the signing threshold satisfies this: [4](#0-3) 

Because `resolve_yields_for` is first-come-first-served (it drains the entire queue on the first successful call), a Byzantine participant can race honest nodes to respond first: [5](#0-4) 

---

### Impact Explanation

The CKD protocol derives an application-specific private key for the user. The user receives `(big_y, big_c)` and computes `derived_key = big_c − a·big_y` using their private scalar `a`. If a Byzantine participant supplies arbitrary `big_y`/`big_c`, the user decrypts a garbage value that no party knows. Any funds sent to the address corresponding to this wrong derived key are permanently locked — no one holds the correct private key. This constitutes **permanent freezing of funds** controlled by the MPC network's key derivation flow.

---

### Likelihood Explanation

The attack requires only one of the MPC participants to be Byzantine. The participant must win a race to call `respond_ckd` before honest nodes. In a network with `n` participants, a single malicious one has a `1/n` chance of being the first to submit the transaction in any given block, and can retry across multiple requests. The `AppPublicKey` variant is still accepted by the contract (it is not deprecated or gated), so real users can trigger this path today.

---

### Recommendation

1. **Deprecate `AppPublicKey` in `respond_ckd`**: Reject `respond_ckd` calls that use the `AppPublicKey` variant, forcing all new CKD requests to use `AppPublicKeyPV`, which has an on-chain pairing check.
2. **Or add an equivalent check**: Require the node to also supply a proof (e.g., a BLS signature or commitment) that allows the contract to verify the `AppPublicKey` response without knowing the user's private scalar.
3. **Emit a warning/deprecation log** for `AppPublicKey` responses in the interim.

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk1 = a·G1)` and a derivation path.
2. Byzantine participant (one of the `n` MPC nodes) calls `respond_ckd` with:
   ```
   request = <the pending CKD request>
   response = CKDResponse { big_y: G1, big_c: G1 }  // arbitrary garbage
   ```
3. The contract executes the `AppPublicKey(_) => {}` branch — no check — and calls `resolve_yields_for`, delivering the garbage response to the user.
4. User computes `derived_key = G1 − a·G1 = (1−a)·G1`, which is a deterministic but wrong key that no party knows.
5. User sends funds to the address derived from this key; funds are permanently locked.

The attack is executable by any single attested participant without threshold collusion, matching the "Byzantine participant strictly below the signing threshold" entry path.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
