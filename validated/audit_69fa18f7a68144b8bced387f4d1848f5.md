Audit Report

## Title
LP Principal Permanently Locked When Position Owner Is Blacklisted by USDC/USDT — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`LiquidityLib.removeLiquidity` unconditionally transfers redeemed tokens to the `owner` address with no alternative `recipient` parameter and no admin rescue path. If `owner` is blacklisted by USDC or USDT (both explicitly in scope), every withdrawal attempt reverts at the `safeTransfer` step, permanently locking the LP's principal in the pool with no recovery mechanism.

## Finding Description

In `LiquidityLib.removeLiquidity`, after all bin accounting is updated, the function executes:

```solidity
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);
}
if (amount1Removed > 0) {
    IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
}
``` [1](#0-0) 

The `owner` address is the sole hardcoded recipient. There is no `recipient` parameter and no fallback address.

The caller-side enforcement in `MetricOmmPool.removeLiquidity` compounds the issue:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [2](#0-1) 

Because `msg.sender` must equal `owner`, no third party — including the pool admin or factory — can call `removeLiquidity` on the owner's behalf to redirect tokens to a clean address.

The `MinimalLiquidity` guard further tightens the trap: a partial withdrawal that would leave residual shares below `MINIMAL_MINTABLE_LIQUIDITY` also reverts, preventing even a partial drain of the position. [3](#0-2) 

The `whenNotPaused` modifier applies only to `swap`, not to `removeLiquidity`, so pausing the pool creates no rescue path. [4](#0-3) 

No emergency withdrawal or factory-level rescue function exists anywhere in the pool contract. [5](#0-4) 

## Impact Explanation

An LP whose address is blacklisted by USDC or USDT will find that every call to `removeLiquidity` reverts at the `safeTransfer` step. Because Solidity reverts roll back all state changes, the LP's shares are never burned — they remain on-chain indefinitely. The LP retains a share balance that correctly entitles them to underlying tokens, but they can never execute the transfer. The tokens are permanently locked in the pool with no recovery mechanism. This constitutes a direct, permanent loss of user principal, satisfying the "Critical/High/Medium direct loss of user principal" and "Broken core pool functionality causing loss of funds or unusable withdraw/liquidity flows" impact gates.

## Likelihood Explanation

USDC and USDT both maintain on-chain blacklists enforced at the token level and are explicitly in scope per contest rules. Blacklisting occurs for regulatory compliance, sanctions enforcement, or fraud response. Any LP who deposits into a USDC/USDT pool and is subsequently blacklisted faces permanent loss. No attacker action is required — the condition is triggered by a standard, documented feature of the in-scope tokens. The pool has no mechanism to detect or handle this condition.

## Recommendation

1. **Add an optional `recipient` parameter** to `removeLiquidity` so the owner can redirect proceeds to a non-blacklisted address. Replace `safeTransfer(owner, ...)` with `safeTransfer(recipient, ...)` in `LiquidityLib.removeLiquidity`, where `recipient` defaults to `owner` if `address(0)` is passed. [6](#0-5) 

2. **Add an admin emergency rescue function** (behind a timelock/multisig) that can transfer any token balance in excess of `binTotals` accounting to a specified address, providing a last-resort recovery path.

## Proof of Concept

1. Pool is deployed with `token0 = USDC`, `token1 = WETH`.
2. Alice calls `addLiquidity` via a router callback, depositing USDC into bin `+4`. Her position is recorded: `positionBinShares[keccak256(alice, salt, 4)] = 10_000`.
3. USDC's issuer blacklists Alice's address for a compliance reason.
4. Alice calls `removeLiquidity(alice, salt, deltas, "")`.
5. `MetricOmmPool` passes the `msg.sender == alice == owner` check at line 206.
6. `LiquidityLib.removeLiquidity` computes `amount0Removed > 0`, then executes `IERC20(USDC).safeTransfer(alice, amount0Removed)` at line 243.
7. USDC's `transfer` reverts because Alice is blacklisted. The entire transaction reverts, including all bin accounting updates.
8. Alice's shares remain at `10_000`. She retries — same revert every time.
9. No other address can call `removeLiquidity` for Alice (`NotPositionOwner` revert at line 206).
10. Alice's USDC principal is permanently locked in the pool.
