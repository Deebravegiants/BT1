### Title
Missing Deposit Refund in `submit_participant_info` Re-Submission Path - (File: crates/contract/src/lib.rs)

### Summary
`submit_participant_info` is marked `#[payable]` and accepts any attached deposit, but when an **existing participant** re-submits their attestation (not a new entry), the entire deposit-handling block — including the refund — is skipped. Any NEAR tokens attached to such a call are permanently locked in the contract with no recovery path.

### Finding Description
The function `submit_participant_info` computes a boolean gate to decide whether to charge and refund the caller:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... cost check and refund logic ...
}
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — i.e., the caller **is** an existing participant (`voter_account()` succeeds) **and** the insertion result is **not** `NewlyInsertedParticipant` (a re-submission/renewal) — the `if` block is never entered. The `env::attached_deposit()` value is accepted by the `#[payable]` function but never read, checked, or refunded. [2](#0-1) 

The re-submission path is the normal production lifecycle: nodes resubmit their attestation on a 1-hour cadence and on attestation-removal events. [3](#0-2) 

Compare this to every other payable entry point in the contract — `sign`, `request_app_private_key`, `verify_foreign_transaction`, and `propose_update` — all of which call `require_deposit` or an explicit `checked_sub` refund unconditionally: [4](#0-3) 

### Impact Explanation
Any NEAR tokens attached to a re-submission call by an existing participant are permanently transferred to the contract balance with no mechanism to recover them. This breaks the balance/accounting invariant that `#[payable]` functions must either consume the deposit for a defined purpose or refund the excess. The contract's own README states: "Any excess deposit is automatically refunded" — a guarantee that is violated on this code path. [5](#0-4) 

Impact: **Medium** — balance accounting invariant broken; funds are permanently locked in the contract without any authorized purpose.

### Likelihood Explanation
The `mpc-node` software currently attaches `0` deposit for re-submissions (the docs note "the node attaches `0`"), so the default automated path does not trigger the loss. However:

1. The function is publicly callable by any existing participant with any deposit amount.
2. Operators who manually invoke `submit_participant_info` (e.g., for attestation renewal, debugging, or following documentation examples that show `attached-deposit '100 yoctoNEAR'`) will silently lose whatever they attach.
3. No warning, error, or log is emitted when a non-zero deposit is silently consumed on this path.

Likelihood: **Low-Medium** — requires a manual call with a non-zero deposit, but the production re-submission cadence and operator tooling make accidental overpayment realistic.

### Recommendation
Add an unconditional refund of any excess deposit outside the `if attestation_storage_must_be_paid_by_caller` block, mirroring the pattern already used in `require_deposit` and `propose_update`:

```rust
// After the if block:
let attached = env::attached_deposit();
if attached > NearToken::from_yoctonear(0) {
    Promise::new(account_id).transfer(attached).detach();
}
```

Alternatively, restructure the deposit handling so that `require_deposit(NearToken::from_yoctonear(0), &account_id)` is always called, which will refund any non-zero deposit on the free re-submission path.

### Proof of Concept

1. Deploy the contract and register participant `alice.near` with a valid attestation (new insertion — storage deposit is charged and refunded correctly).
2. `alice.near` is now an existing participant (`voter_account()` succeeds).
3. Call `submit_participant_info` again from `alice.near` with `attached_deposit = 1_000_000 yoctoNEAR` (simulating a manual renewal with an accidental deposit).
4. The call succeeds (`Ok(())`).
5. `alice.near`'s balance decreases by `1_000_000 yoctoNEAR`; the contract balance increases by the same amount.
6. No refund promise is scheduled; the funds are permanently locked.

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),

                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
