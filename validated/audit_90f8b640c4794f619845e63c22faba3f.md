### Title
Deposits from non-executed `propose_update` proposals are permanently locked when `do_update` clears all entries — (`File: crates/contract/src/update.rs`)

---

### Summary

`propose_update` requires a storage-staking deposit proportional to the WASM binary size (up to ~17 NEAR). When any proposal reaches threshold and `do_update` executes, it bulk-clears **all** pending proposals without refunding the deposits paid by the proposers of the non-executed entries. Those NEAR tokens are permanently locked in the contract with no rescue path.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` collects a deposit from the proposer: [1](#0-0) 

The `required` amount (covering storage for the WASM blob plus a 128-account-vote overhead) is retained in the contract; only the excess above `required` is refunded immediately. [2](#0-1) 

When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry and then bulk-clears every other pending proposal and every vote: [3](#0-2) 

`self.entries.clear()` frees the on-chain storage for all non-executed proposals, but **no refund is issued to any of those proposers**. The `UpdateEntry` struct does not record the proposer's account ID or the deposit amount, so there is no information available to perform a refund even if the code tried to: [4](#0-3) 

There is no `remove_proposal` endpoint, no `withdraw_deposit` endpoint, and no other path that returns these funds. `remove_update_vote` only removes the vote record, not the proposal or its deposit: [5](#0-4) 

---

### Impact Explanation

Every participant whose proposal is cleared by a competing proposal permanently loses their storage deposit. For a full WASM binary the deposit is approximately 17 NEAR (confirmed by the test constant `CURRENT_CONTRACT_DEPLOY_DEPOSIT = NearToken::from_millinear(17000)`): [6](#0-5) 

With N competing proposals, N-1 proposers each lose up to 17 NEAR with no recovery path. This breaks the production accounting invariant that storage-staking deposits are returned when the corresponding storage is freed.

This maps to the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

Multiple participants independently proposing different contract upgrades is a realistic and documented scenario (the test `test_propose_update_contract_many` and `test_propose_update_config` both exercise it). The loss is triggered automatically and silently every time any proposal reaches threshold while competing proposals exist. [7](#0-6) 

---

### Recommendation

1. Add `proposer: AccountId` and `deposit: NearToken` fields to `UpdateEntry`.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule `Promise::new(entry.proposer).transfer(entry.deposit)` for each.
3. Alternatively, expose a `withdraw_proposal(id: UpdateId)` endpoint that lets a proposer retract their own proposal and receive a refund, and call it implicitly during `do_update` cleanup.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB WASM blob, attaching ~17 NEAR. The deposit is retained in the contract.
2. Participant B calls `propose_update` with a different WASM blob, attaching ~17 NEAR. The deposit is retained.
3. Threshold participants vote for proposal A. `vote_update` calls `do_update(&id_A, gas)`.
4. Inside `do_update`: `self.entries.remove(&id_A)` removes A's entry; `self.entries.clear()` removes B's entry. No refund is issued to B.
5. B's ~17 NEAR is permanently locked in the contract. There is no function to recover it. [8](#0-7)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
