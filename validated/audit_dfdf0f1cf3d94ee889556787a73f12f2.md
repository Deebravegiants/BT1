Audit Report

## Title
`_updateRsETHPrice()` Reverts for Public Callers on Legitimate Price Increases, Leaving `rsETHPrice` Stale and Enabling Over-Minting of rsETH — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle._updateRsETHPrice()` reverts with `PriceAboveDailyThreshold` for any non-manager caller when the computed new rsETH price exceeds `highestRsethPrice` by more than `pricePercentageLimit`. Because `updateRSETHPrice()` is a public function callable by anyone, this revert freezes `rsETHPrice` at a stale, below-actual value. During the stale window, `LRTDepositPool.getRsETHAmountToMint()` mints excess rsETH to new depositors, extracting value from existing holders' accrued yield.

## Finding Description

`updateRSETHPrice()` is declared `public` with no role restriction: [1](#0-0) 

Inside `_updateRsETHPrice()`, when `newRsETHPrice > highestRsethPrice` and the difference exceeds `pricePercentageLimit`, non-manager callers are reverted: [2](#0-1) 

This revert causes execution to exit before `rsETHPrice` and `highestRsethPrice` are written: [3](#0-2) 

The stale `rsETHPrice` is used directly as the denominator in deposit minting: [4](#0-3) 

A stale-low denominator causes `rsethAmountToMint` to be inflated. The excess rsETH minted represents a claim on TVL that already belonged to existing holders (their accrued staking yield). The withdrawal path is also affected (stale-low numerator yields fewer assets per rsETH), but the deposit path is the primary extraction vector.

The only recovery path is `updateRSETHPriceAsManager()`, which requires a manager to act manually: [5](#0-4) 

## Impact Explanation

**High — Theft of unclaimed yield.** The price increase above `pricePercentageLimit` represents staking rewards that have accrued to existing rsETH holders. When `rsETHPrice` is frozen below the actual value, new depositors receive more rsETH than their contribution warrants. Each over-minted rsETH dilutes existing holders' proportional claim on TVL. For a 1% price gap and a 1,000 ETH deposit, approximately 14.6 rsETH is over-minted (~15 ETH of value extracted from existing holders at the actual price). The gap can persist for hours or days and is exploitable by multiple depositors simultaneously.

## Likelihood Explanation

No attacker action is required to create the stale condition — it arises naturally whenever `newRsETHPrice` jumps more than `pricePercentageLimit` above `highestRsethPrice` in a single call. Realistic triggers include: accumulated staking rewards after a period of oracle inactivity, a sudden LST rate increase (e.g., post-Lido reward distribution), or a small `pricePercentageLimit` (e.g., 1e16 = 1%). An attacker only needs to observe that `updateRSETHPrice()` reverts and then deposit before the manager intervenes. The exploit is repeatable across multiple depositors within the same stale window.

## Recommendation

1. **Clamp instead of revert**: when `isPriceIncreaseOffLimit` is true for a public caller, update `rsETHPrice` to `highestRsethPrice + pricePercentageLimit.mulWad(highestRsethPrice)` (the band ceiling) rather than reverting. This ensures the price always advances to at least the permitted boundary.
2. **Separate the guard from the price write**: allow the price to be written up to the capped value unconditionally; only gate the full uncapped jump behind the manager role.
3. **Keeper automation**: deploy an on-chain keeper or Gelato task that calls `updateRSETHPriceAsManager()` promptly whenever the public call reverts, minimizing the stale window as a defense-in-depth measure.

## Proof of Concept

1. Deploy with `highestRsethPrice = 1.04e18`, `rsETHPrice = 1.04e18`, `pricePercentageLimit = 1e16` (1%).
2. Allow staking rewards to accumulate such that on-chain TVL implies `newRsETHPrice = 1.056e18` (1.54% above `highestRsethPrice`).
3. Any unprivileged account calls `updateRSETHPrice()`. Inside `_updateRsETHPrice()`: `isPriceIncreaseOffLimit = true`, caller lacks `MANAGER` role → `revert PriceAboveDailyThreshold()`.
4. `rsETHPrice` remains `1.04e18`; `highestRsethPrice` remains `1.04e18`.
5. Attacker deposits 1,000 ETH of stETH via `LRTDepositPool.depositAsset()`. `getRsETHAmountToMint` computes: `1000e18 * 1e18 / 1.04e18 ≈ 961.5 rsETH`. Correct amount at actual price: `1000e18 * 1e18 / 1.056e18 ≈ 946.9 rsETH`. Over-mint: **~14.6 rsETH**.
6. Manager calls `updateRSETHPriceAsManager()`; price updates to `1.056e18`. Attacker's 961.5 rsETH is now worth `961.5 * 1.056 ≈ 1,015.3 ETH` — a ~15 ETH profit extracted from existing holders' accrued yield.

**Foundry fork test plan**: fork mainnet, set `pricePercentageLimit = 1e16`, manipulate TVL to produce a >1% price jump, call `updateRSETHPrice()` as a non-manager (assert revert), call `depositAsset()` as attacker, assert `rsethAmountToMint > correctAmount`, call `updateRSETHPriceAsManager()` as manager, assert attacker's rsETH value exceeds deposit value.
