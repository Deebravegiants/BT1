Audit Report

## Title
Permanent Freezing of User ETH Principal and Protocol Yield via ETH-Rejecting Recipient — (`contracts/LRTWithdrawalManager.sol`)

## Summary
A contract without a `receive()` function can initiate an ETH withdrawal, have it unlocked by an operator, and then permanently trap both its own ETH principal and all future ETH yield in `LRTWithdrawalManager`. Every attempt to complete the withdrawal reverts atomically (including `completeWithdrawalForUser`, which always targets the original `user` address), `unlockedWithdrawalsCount[ETH]` can never reach zero, and no admin or emergency path exists to cancel, redirect, or recover the stuck funds.

## Finding Description

**Step 1 — Initiation:** Any contract (e.g., a multisig, vault, or proxy) without a `receive()` function calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. This is fully permissionless; rsETH is transferred from the caller and a `WithdrawalRequest` is stored. [1](#0-0) 

**Step 2 — Unlock:** An operator calls `unlockQueue(ETH_TOKEN, ...)`. Inside `_unlockWithdrawalRequests`, the request is processed and `unlockedWithdrawalsCount[ETH]++` is executed. The rsETH is burned and the ETH principal is redeemed from the unstaking vault into the `LRTWithdrawalManager` contract. [2](#0-1) 

**Step 3 — Completion always reverts:** `_processWithdrawalCompletion` calls `_transferAsset(asset, user, request.expectedAssetAmount)`. For ETH, `_transferAsset` does a low-level call to `user` and reverts with `EthTransferFailed()` if the call fails. [3](#0-2) 

Because the revert is atomic, all state changes in `_processWithdrawalCompletion` — including `popFront()`, `delete withdrawalRequests[requestId]`, and `unlockedWithdrawalsCount[asset]--` — are rolled back on every call. [4](#0-3) 

**Step 4 — No redirect path:** `completeWithdrawalForUser` passes the original `user` address directly to `_processWithdrawalCompletion`, so it also always reverts for an ETH-rejecting contract. There is no parameter to redirect the ETH to a different address. [5](#0-4) 

**Step 5 — `sweepRemainingAssets` permanently blocked:** The sweep function is gated by `hasUnlockedWithdrawals`, which checks `unlockedWithdrawalsCount[asset] > 0`. Since the counter can never be decremented, this check always reverts with `PendingWithdrawalsExist()`. [6](#0-5) [7](#0-6) 

There is no admin, operator, or emergency function anywhere in the contract that can cancel a withdrawal, force-decrement `unlockedWithdrawalsCount`, or redirect a stuck withdrawal to a different address.

## Impact Explanation

**Critical — Permanent freezing of funds.** The ETH principal redeemed from the unstaking vault during `unlockQueue` sits in `LRTWithdrawalManager` with no recovery path. The user cannot receive it (ETH rejected), no admin can redirect it, and `sweepRemainingAssets` is permanently blocked. The ETH is irrecoverably frozen in the contract.

**Medium — Permanent freezing of unclaimed yield** (secondary). `sweepRemainingAssets` being permanently blocked also prevents any surplus ETH yield from ever being swept to the treasury.

The dominant impact is Critical: permanent freezing of user funds.

## Likelihood Explanation

The trigger requires only that a contract without a `receive()` function holds rsETH and calls `initiateWithdrawal`. This is realistic: any protocol integrating rsETH (multisigs, vaults, proxies, ERC-4626 wrappers) may lack a `receive()` function. The attacker/victim loses their rsETH (burned) and their ETH (frozen), making deliberate exploitation costly but accidental occurrence plausible. A single such event permanently blocks `sweepRemainingAssets` for ETH.

## Recommendation

1. **Pull-payment pattern (preferred):** In `_processWithdrawalCompletion`, instead of pushing ETH to `user`, credit a `claimable[user][asset]` balance and decrement `unlockedWithdrawalsCount` immediately. Add a separate `claim()` function where users pull their ETH. Transfer failure no longer blocks state updates.
2. **Cancellation/redirect mechanism:** Add an operator-callable function that, given a stuck `requestId`, decrements `unlockedWithdrawalsCount[asset]`, deletes the request, and either returns the ETH to a specified safe address or re-deposits it into the unstaking vault.
3. **Wrap `_transferAsset` in a try/catch equivalent:** Use a low-level call that does not revert on failure; instead, credit the user's claimable balance if the push fails.

contract ETHRejecter {
