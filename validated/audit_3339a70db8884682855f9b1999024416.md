### Title
Proposer Deposits Permanently Locked When Competing `propose_update` Entries Are Cleared by `do_update` - (File: crates/contract/src/update.rs)

### Summary

When `propose_update` is called, each proposer attaches a NEAR deposit sized to cover storage for their contract code or config payload. When any one proposal reaches threshold and `do_update` executes, it calls `self.entries.clear()` to wipe all competing proposals from storage. The freed storage bytes are returned to the contract's own balance, but the original proposers of the cleared entries receive no refund. Because `UpdateEntry` stores neither the proposer's `AccountId` nor the deposit amount, there is no recovery path: those NEAR tokens are permanently locked inside the contract.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires each caller to attach a deposit equal to `ProposedUpdates::required_deposit(&update)`, which is computed as `env::storage_byte_cost() * bytes_used(update)`. For a full contract binary this can be tens to hundreds of NEAR. Only the excess above the required amount is refunded immediately; the required portion is retained by the contract to cover storage. [1](#0-0) 

The `UpdateEntry` struct that is stored per proposal contains only the update payload and a `bytes_used` field — no proposer account ID and no deposit amount: [2](#0-1) 

When `do_update` is triggered by a threshold vote, it removes the winning entry and then unconditionally clears every other pending entry and all votes: [3](#0-2) 

NEAR's storage model returns freed bytes to the contract's own balance, so the contract's balance grows by the storage cost of every cleared proposal. However, because neither the proposer identity nor the deposited amount is recorded anywhere, the contract has no information with which to issue refunds. There is no `withdraw_proposal` or equivalent method in the public API. [4](#0-3) 

### Impact Explanation

Every participant who proposed a competing update loses their storage deposit permanently when any other proposal reaches threshold. For a 1 MB contract binary the required deposit is on the order of 100 NEAR per proposer. In a governance scenario where multiple participants independently propose different upgrades (e.g., different code versions or config values), all but the winning proposer forfeit their deposits with no recourse. The locked NEAR accumulates in the contract balance and cannot be withdrawn by any participant or governance action. This breaks the accounting invariant that every deposit paid for a proposal is either consumed by that proposal's execution or returned to the depositor.

### Likelihood Explanation

The scenario is realistic in any contested governance round: participants may disagree on which contract version to adopt and submit competing proposals. The `propose_update` flow is the only mechanism for contract upgrades, so this path is exercised in every upgrade cycle. No adversarial intent is required — the loss occurs as a side effect of normal threshold governance.

### Recommendation

Store the proposer's `AccountId` and the deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. Alternatively, expose a `withdraw_proposal(id: UpdateId)` method that lets a proposer reclaim their deposit and remove their own entry at any time before execution.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~100 NEAR. The deposit is accepted and `UpdateEntry { update, bytes_used }` is stored — no proposer or deposit field.
2. Participant B calls `propose_update` with a different binary, also attaching ~100 NEAR.
3. Participants vote for B's proposal until threshold is reached; `vote_update` calls `do_update`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry is deleted. NEAR's storage accounting credits the freed bytes back to the contract balance.
5. A's 100 NEAR is now part of the contract's balance. There is no method to retrieve it: `UpdateEntry` never recorded A's identity or deposit, and no `withdraw_proposal` method exists. [5](#0-4) [6](#0-5) [7](#0-6)
