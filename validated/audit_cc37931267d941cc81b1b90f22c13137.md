### Title
rsETH Inflation Attack via Unrestricted ETH Donation and Floor-Division Share Minting Yields Zero rsETH to Victim - (File: contracts/LRTDepositPool.sol)

---

### Summary

`LRTDepositPool` exposes a bare `receive()` payable fallback that lets anyone donate ETH directly to the pool without minting rsETH. Because `LRTOracle.updateRSETHPrice()` is public and `pricePercentageLimit` defaults to `0` (never initialized), an attacker can inflate `rsETHPrice` to an arbitrary value in a single call. `getRsETHAmountToMint` uses plain integer floor division, so a victim's deposit against the inflated price truncates to zero. `_beforeDeposit` does not revert when `rsethAmountToMint == 0` as long as `minRSETHAmountExpected == 0`, which is the common default. The victim's ETH enters the pool and is captured by the attacker's pre-existing rsETH position.

---

### Finding Description

**Vulnerable entry points**

`LRTDepositPool.receive()` accepts arbitrary ETH with no accounting: [1](#0-0) 

`LRTOracle.updateRSETHPrice()` is unrestricted and callable by any address: [2](#0-1) 

The price-increase guard is disabled when `pricePercentageLimit == 0`, which is the storage default because `initialize()` never sets it: [3](#0-2) 

The new rsETH price is computed as `totalETHInProtocol / rsethSupply`, so a donation that inflates `totalETHInProtocol` while `rsethSupply` stays at 1 wei produces an astronomically large price: [4](#0-3) 

`getRsETHAmountToMint` uses plain `/` (floor division). When `rsETHPrice` is inflated, the result truncates to zero for any realistic deposit: [5](#0-4) 

`_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`. When `minRSETHAmountExpected == 0` (the common default), a zero-mint deposit is silently accepted: [6](#0-5) 

---

### Impact Explanation

**Critical — Direct theft of user funds.**

The victim deposits ETH and receives 0 rsETH. Their ETH is permanently absorbed into the pool and is redeemable only by existing rsETH holders (the attacker). The victim has no token representing their claim and cannot recover the deposit.

---

### Likelihood Explanation

**Medium.**

Two conditions must hold simultaneously:

1. `pricePercentageLimit == 0` — this is the **storage default** because `LRTOracle.initialize()` never assigns it. Unless the admin explicitly calls `setPricePercentageLimit` post-deployment, the guard is absent.
2. The victim passes `minRSETHAmountExpected == 0` — this is the conventional default in scripts, integrations, and many front-ends that omit slippage protection.

Both conditions are realistic in a live deployment. The attacker requires capital equal to the victim's deposit but recovers it plus the victim's deposit upon redemption.

---

### Recommendation

1. **Remove or restrict the bare `receive()` fallback** in `LRTDepositPool`. Unaccounted ETH donations are the root enabler. If ETH must be receivable, gate it to known callers (e.g., `receiveFromNodeDelegator`, `receiveFromLRTConverter`).
2. **Require `rsethAmountToMint > 0`** inside `_beforeDeposit`, independent of `minRSETHAmountExpected`. A deposit that mints zero shares should always revert.
3. **Set a non-zero `pricePercentageLimit` in `initialize()`** so that a single-block price manipulation is bounded even if the above fixes are not applied.
4. Consider adding a virtual offset (analogous to OZ ERC-4626's `_decimalsOffset`) to the rsETH price calculation so that a small initial supply does not make the exchange rate trivially manipulable.

---

### Proof of Concept

```
Initial state: rsethSupply = 0, rsETHPrice = 1e18 (hardcoded when supply is zero),
               pricePercentageLimit = 0 (default), minAmountToDeposit = 0 (default)

Step 1 — Attacker deposits 1 wei ETH via depositETH(minRSETHAmountExpected=0):
  rsethAmountToMint = (1 * 1e18) / 1e18 = 1 wei rsETH
  Pool ETH: 1 wei | rsETH supply: 1 wei

Step 2 — Attacker sends V wei ETH directly to LRTDepositPool (receive() fallback):
  Pool ETH: 1 + V wei | rsETH supply: 1 wei (unchanged)

Step 3 — Attacker calls LRTOracle.updateRSETHPrice():
  totalETHInProtocol = 1 + V
  newRsETHPrice = (1 + V) * 1e18 / 1 = (1 + V) * 1e18
  pricePercentageLimit == 0 → isPriceIncreaseOffLimit = false → no revert
  rsETHPrice stored = (1 + V) * 1e18

Step 4 — Victim calls depositETH{value: V}(minRSETHAmountExpected=0):
  rsethAmountToMint = (V * 1e18) / ((1 + V) * 1e18)
                    = V / (1 + V) = 0  ← floor division
  minRSETHAmountExpected = 0, so 0 < 0 is false → no revert
  Victim's V wei ETH enters pool; victim receives 0 rsETH

Step 5 — Pool state: ETH = 1 + V + V = 1 + 2V | rsETH supply = 1 wei

Step 6 — Attacker redeems 1 wei rsETH (via withdrawal mechanism):
  Attacker recovers ≈ 1 + 2V wei ETH

Attacker net P&L:
  Spent:  1 wei (deposit) + V wei (donation) ≈ V
  Gained: 1 + 2V wei
  Profit: ≈ V wei  (the victim's entire deposit)
```
