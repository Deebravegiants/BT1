Audit Report

## Title
Stale `rsETHPrice` Denominator in `getRsETHAmountToMint` Allows Depositors to Capture Accrued Yield at Existing Holders' Expense - (File: `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint()` divides by the stored `rsETHPrice` from `LRTOracle`, which is only updated on keeper-driven or public calls to `updateRSETHPrice()`. Between updates, accrued staking rewards cause the true price (`totalETHInProtocol / rsethSupply`) to exceed the stored price. Any depositor can exploit this lag by depositing at the stale low price, receiving more rsETH than fair value, and thereby diluting existing holders' proportional claim on accrued yield.

## Finding Description

`LRTOracle.updateRSETHPrice()` is unrestricted public:

```solidity
// contracts/LRTOracle.sol:87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`LRTDepositPool.getRsETHAmountToMint()` uses the stored (potentially stale) `rsETHPrice` as the denominator:

```solidity
// contracts/LRTDepositPool.sol:520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`_updateRsETHPrice()` recomputes the price as `(totalETHInProtocol - protocolFeeInETH) / rsethSupply` where `totalETHInProtocol` reflects all on-chain accrued rewards. Between keeper updates, `rsETHPrice` is stale-low relative to the true value.

The `pricePercentageLimit` guard at lines 252–266 only blocks a non-manager from calling `updateRSETHPrice()` when the price increase exceeds the configured threshold. It does **not** prevent depositing at the stale price. The attacker's profit is realized at deposit time, not at price-update time:

1. Rewards accrue → `totalETHInProtocol > rsethSupply × rsETHPrice_stored`
2. Attacker calls `depositETH()` or `depositAsset()` — `getRsETHAmountToMint()` divides by the stale low `rsETHPrice`, minting excess rsETH
3. Price update occurs (by keeper or attacker if within limit) — attacker's rsETH is now worth more than deposited
4. Existing holders' rsETH represents a smaller fraction of TVL than before the deposit

The `_beforeDeposit` path at line 665 calls `getRsETHAmountToMint()` with no price freshness check, and `_mintRsETH` at line 689 mints unconditionally.

## Impact Explanation

Every unit of excess rsETH minted to the attacker represents accrued staking yield owed pro-rata to pre-existing rsETH holders. The attacker captures a quantifiable portion of unclaimed yield per deposit. This maps directly to the allowed impact: **High — Theft of unclaimed yield**. The per-attack magnitude is bounded by the `pricePercentageLimit` (when set), but the attack is repeatable every reward accrual cycle with no cooldown enforced on depositors.

## Likelihood Explanation

No privileged access is required. Any EOA or contract can call `depositETH()` or `depositAsset()`. The staleness condition is trivially observable on-chain by comparing `LRTDepositPool.getTotalAssetDeposits()` × asset prices against `rsETHPrice × rsethSupply`. The attack requires only a standard deposit transaction timed before the keeper's price update. Likelihood is **Medium-High**.

## Recommendation

1. **Atomically refresh the price before minting**: call `_updateRsETHPrice()` (or an equivalent view computation) inside `_beforeDeposit` so `getRsETHAmountToMint()` always uses the current true price.
2. **Alternatively, compute the mint amount on-the-fly** using `_getTotalEthInProtocol() / rsethSupply` rather than the stored `rsETHPrice`, eliminating the staleness window entirely.
3. If keeper-driven updates are preferred, restrict `updateRSETHPrice()` to `onlyLRTManager` (matching `updateRSETHPriceAsManager()`) and enforce an atomic price refresh inside deposit flows.

## Proof of Concept

```
State:
  rsETHPrice (stored) = 1.00e18
  totalETHInProtocol  = 1.01e18 × S  (rewards accrued, S = current rsETH supply)
  True price          = 1.01e18

Step 1 — Attacker calls depositETH(100 ether):
  getRsETHAmountToMint uses rsETHPrice = 1.00e18 (stale)
  rsethAmountToMint = 100e18 * 1e18 / 1.00e18 = 100 rsETH
  Fair amount       = 100e18 * 1e18 / 1.01e18 ≈ 99.01 rsETH
  Excess minted     ≈ 0.99 rsETH

Step 2 — Keeper (or attacker) calls updateRSETHPrice():
  New price = (1.01*S*1e18 + 100e18) / (S + 100) ≈ 1.01e18 for large S

Result:
  Attacker deposited 100 ETH, holds rsETH worth ≈ 101 ETH at new price.
  Existing holders' rsETH is diluted by ≈ 0.99 rsETH worth of TVL.
  Attack is repeatable every reward cycle.
```

Foundry fork test outline:
1. Fork mainnet at a block where rewards have accrued since last `updateRSETHPrice()`.
2. Record `rsETHPrice` and compute `_getTotalEthInProtocol() / rsethSupply` to confirm staleness.
3. Deposit a fixed ETH amount as attacker; record rsETH received.
4. Call `updateRSETHPrice()` (or wait for keeper tx); record new `rsETHPrice`.
5. Assert `attacker_rsETH × new_price > deposited_ETH` and that existing holder rsETH value decreased proportionally.
