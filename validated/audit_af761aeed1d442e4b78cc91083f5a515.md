### Title
Validator Accepts Arbitrary `fee_proposal_fri` During Startup Window, Enabling Persistent L2 Gas Price Manipulation — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` intentionally skips the `fee_proposal_fri` bounds check when `fee_actual` is `None` (the startup window: the first `fee_proposal_window_size` blocks, or any window containing pre-V0_14_3 blocks). The proposer is documented to freeze at `l2_gas_price` during this window, but the validator enforces no such constraint. A malicious proposer can therefore publish any `fee_proposal_fri` value (e.g., `u128::MAX`) during the startup window, have it accepted by all validators, committed to storage, and seeded into the `fee_proposals_window`. Once the window fills, `fee_actual` is the median of those manipulated values, permanently corrupting the L2 gas price for all subsequent blocks.

### Finding Description

**Proposer side** (`sequencer_consensus_context.rs`, `compute_proposer_fee_proposal`): when `fee_actual` is `None`, the proposer freezes at `l2_gas_price`:

```rust
let Some(fee_actual) = fee_actual else {
    warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
    return self.l2_gas_price;
};
``` [1](#0-0) 

**Validator side** (`validate_proposal.rs`, `is_proposal_init_valid`): the bounds check is silently skipped when `fee_actual` is `None`:

```rust
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    // ... bounds check only runs here
}
``` [2](#0-1) 

Neither the validator nor any downstream path enforces that `fee_proposal_fri == l2_gas_price` during the startup window. The `fee_proposal_fri` value is then:

1. Bound into the `ProposalCommitment` via `proposal_commitment_from`:
   ```rust
   ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
   ``` [3](#0-2) 

2. Stored in `BlockHeaderWithoutHash.fee_proposal_fri` and forwarded to state sync: [4](#0-3) 

3. Recorded into `fee_proposals_window` and used to compute `fee_actual` for future blocks: [5](#0-4) 

4. `fee_actual` drives `calculate_next_l2_gas_price_for_fin`, which sets the L2 gas price for every subsequent block.

The `ProposalInitValidation.fee_actual` field is explicitly documented as `None` until the window accumulates `fee_proposal_window_size` entries: [6](#0-5) 

### Impact Explanation

A malicious proposer who controls a majority of the first `fee_proposal_window_size` blocks (at genesis, or during the V0_14_3 protocol upgrade transition when pre-V0_14_3 blocks still occupy the window) can:

- Publish `fee_proposal_fri = u128::MAX` in every `ProposalInit`.
- Validators accept each proposal (bounds check skipped).
- Each block is committed with `fee_proposal_fri = u128::MAX` in storage.
- After `window_size` such blocks, `compute_fee_actual` returns `u128::MAX` as the median.
