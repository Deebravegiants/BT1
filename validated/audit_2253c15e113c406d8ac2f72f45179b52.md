### Title
Participant-Set State Mismatch in `respond_ckd` During Resharing Enables Fake CKD Response Injection — (File: `crates/contract/src/lib.rs`)

---

### Summary

During a resharing, `respond_ckd` authorizes callers against the **new (proposed)** participant set, while the CKD request was submitted under the **old** participant set. A new participant being added via resharing — who has a valid TEE attestation but does not possess the BLS key material — can call `respond_ckd` with an arbitrary fake response for any pending `AppPublicKey` (legacy) CKD request. Because `AppPublicKey` responses carry **no on-chain cryptographic verification**, the contract accepts the fake response and delivers it to the waiting user, allowing the attacker to know the user's derived secret.

---

### Finding Description

**Root cause 1 — `active_participants()` switches to the new set during resharing**

`ProtocolContractState::active_participants()` returns different participant sets depending on the protocol phase:

```rust
// crates/contract/src/state.rs  lines 255-270
pub fn active_participants(&self) -> &Participants {
    match self {
        ProtocolContractState::Running(state) => state.parameters.participants(),
        ProtocolContractState::Resharing(state) => {
            state.resharing_key.proposed_parameters().participants()  // NEW set
        }
        ...
    }
}
``` [1](#0-0) 

**Root cause 2 — `respond_ckd` is callable during resharing and uses `active_participants()`**

`respond_ckd` permits calls while `is_running_or_resharing()` is true, then delegates authorization to `assert_caller_is_attested_participant_and_protocol_active()`, which internally calls `active_participants()`:

```rust
// crates/contract/src/lib.rs  lines 653-666
pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
    ...
    if !self.protocol_state.is_running_or_resharing() {
        return Err(InvalidState::ProtocolStateNotRunning.into());
    }
    ...
    self.assert_caller_is_attested_participant_and_protocol_active(); // uses NEW set
    ...
``` [2](#0-1) 

During resharing, a new participant (who does not hold the old BLS key material) passes this check because they are in the proposed new set.

**Root cause 3 — `AppPublicKey` responses have zero on-chain cryptographic verification**

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [3](#0-2) 

The `AppPublicKeyPV` variant is protected by a BLS pairing check (`ckd_output_check`), but the legacy `AppPublicKey` variant accepts any `(big_y, big_c)` pair without verification. [4](#0-3) 

**Timing / state-mismatch analogy (direct analog to the external report)**

| External report | NEAR MPC analog |
|---|---|
| Bribes calculated at `EPOCH_END - 1` using old voting power | CKD request submitted under old participant set |
| Distribution happens at `EPOCH_END + X` | `respond_ckd` called during resharing window |
| Voter switches vote between the two points | New participant (no key material) is now authorized to respond |
| Gauge A paid bribes but emissions go to Gauge B | User receives a fake key chosen by the attacker |

### Impact Explanation

The attacker controls `big_y` and `big_c`. By choosing, for example, `big_y = identity_G1` and `big_c = k·G1` for a known scalar `k`, the user computes:

```
secret = big_c − big_y · a = k·G1 − 0 = k·G1
```

The attacker knows `k`, so they know the user's derived secret. If the user uses this secret as a wallet key or to protect funds, the attacker can steal those funds. The pending request is consumed by the fake response; legitimate nodes' subsequent `respond_ckd` calls find no pending entry and silently fail, so the real key is never delivered.

**Impact class:** Critical — direct theft of funds controlled by the MPC network's CKD output.

---

### Likelihood Explanation

- A resharing is a normal, recurring governance operation (adding/removing participants).
- A new participant must have a valid TEE attestation to pass `assert_caller_is_attested_participant_and_protocol_active()`. This is a meaningful barrier in production.
- However, a **single Byzantine participant strictly below the signing threshold** — explicitly in scope — who has obtained a valid attestation can exploit this during any resharing window.
- The attacker only needs to submit their `respond_ckd` transaction before the legitimate nodes do; on NEAR, transaction ordering within a block is deterministic and can be influenced by gas price.
- The `AppPublicKey` (legacy) variant is still accepted by the contract and used in production tooling. [5](#0-4) 

---

### Recommendation

**Option A (preferred):** During resharing, `respond_ckd` (and `respond`) should validate callers against the **old** participant set — the set that actually holds the key material — not the proposed new set. The `authenticate_update_vote` function already demonstrates this pattern for governance votes:

```rust
// crates/contract/src/state.rs  lines 232-235
ProtocolContractState::Resharing(state) => {
    AuthenticatedParticipantId::new(
        state.previous_running_state.parameters.participants(), // OLD set
    )?;
}
``` [6](#0-5) 

Apply the same logic in `assert_caller_is_attested_participant_and_protocol_active()` for the `respond*` family of functions.

**Option B:** Require cryptographic verification for **all** CKD response variants. Remove the no-op branch for `AppPublicKey` and require at minimum a BLS signature over `(app_id, big_c, big_y)` verifiable against the stored BLS public key.

### Proof of Concept

```
1. Contract is Running with participants {A, B, C}, BLS domain active.
2. User U submits request_app_private_key with AppPublicKey variant (legacy).
   → pending_ckd_requests[req] = [yield_id]
3. Governance votes trigger resharing; proposed new set = {A, B, C, M}.
   → contract.protocol_state = Resharing(new_set = {A,B,C,M})
4. M submits a valid TEE attestation via submit_participant_info.
   → M is now in active_participants() (the new set)
5. M calls respond_ckd(req, {big_y: identity_G1, big_c: k·G1}) before A/B/C.
   → assert_caller_is_attested_participant_and_protocol_active() passes (M ∈ new set)
   → AppPublicKey branch: no verification
   → resolve_yields_for delivers fake response to U
6. U computes secret = k·G1 − identity·a = k·G1. M knows k → M knows secret.
7. A/B/C attempt respond_ckd → pending entry is gone → silently ignored.
   Real key is never delivered.
```

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
