### Title
Governance Proposal Deposit Permanently Lost When Return Address Is Unregistered During Bootstrap Phase — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

---

### Summary

During the Conway bootstrap phase, the `GOV` rule skips the check that a proposal's `pProcReturnAddr` must correspond to a registered stake account. However, the `EPOCH` rule's `returnProposalDeposits` function unconditionally requires the credential to be registered in order to credit the refund. This creates a layer mismatch: a transaction submitting a governance proposal with a fresh (never-registered) return address succeeds with no predicate failure, but when the proposal later expires or is enacted the deposit is silently diverted to the treasury rather than refunded. Because a proposal can only be submitted once per `GovActionId`, the depositor has no mechanism to recover the funds.

---

### Finding Description

**Layer 1 — GOV rule (proposal submission, `Gov.hs` lines 504–520):**

```haskell
unless (hardforkConwayBootstrapPhase $ pp ^. ppProtocolVersionL) $ do
  let refundAddress = proposal ^. pProcReturnAddrL
      govAction     = proposal ^. pProcGovActionL
  isAccountRegistered
      (refundAddress ^. accountAddressCredentialL)
      (certDState ^. accountsL)
    ?! (injectFailure . ProposalReturnAccountDoesNotExist) refundAddress
  ...
```

The `unless hardforkConwayBootstrapPhase` guard completely disables the `ProposalReturnAccountDoesNotExist` predicate during bootstrap. A transaction that supplies a fresh, never-registered credential as `pProcReturnAddr` passes the `GOV` rule without error and the proposal is accepted into the governance proposals map.

**Layer 2 — EPOCH rule (deposit refund, `Epoch.hs` lines 187–193):**

```haskell
processProposal gas (!accounts, !unclaimed)
  | (Just _accountState, newAccounts) <-
        updateLookupAccountState addRefund cred accounts =
      (newAccounts, unclaimed)
  | otherwise =
      (accounts, Map.insert (gasId gas) (gasDeposit gas) unclaimed)
  where
    cred = gasReturnAddr gas ^. accountAddressCredentialL
```

`returnProposalDeposits` is called unconditionally for every removed proposal (expired, enacted, or invalidated). If `cred` is not present in the accounts map the deposit is placed in `unclaimed` — there is no error, no event that signals the loss to the submitter.

**Layer 3 — unclaimed deposits are swept to the treasury (`Epoch.hs` lines 347–350):**

```haskell
chainAccountState3 =
  chainAccountState2
    & casTreasuryL <>~ (utxoState0 ^. utxosDonationL <> fold unclaimed)
```

The full `unclaimed` map (including the lost deposit) is added to the treasury. The original depositor receives nothing and has no recourse.

**The mismatch:** The `GOV` rule (submission time) and the `returnProposalDeposits` helper (epoch boundary) apply inconsistent requirements to the return address. The submission layer is intentionally relaxed during bootstrap; the refund layer is not. No error is surfaced at submission time, so the depositor has no warning.

---

### Impact Explanation

Any ADA locked as a governance proposal deposit (`ppGovActionDeposit`, which can be set to an arbitrarily large value by governance) is permanently transferred to the treasury when the proposal's `pProcReturnAddr` credential is unregistered at the epoch boundary. The depositor cannot re-submit the same proposal (each `GovActionId` is unique), cannot withdraw the funds from the treasury without a separate governance action (which itself requires a new deposit), and cannot force a refund without a hard fork. This constitutes **permanent loss of ADA deposits** matching the **High** impact tier: *Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork.*

---

### Likelihood Explanation

The bootstrap phase is active at the start of the Conway era (protocol version 9.x). During this window any unprivileged transaction sender can submit a governance proposal. A depositor who:

- uses a freshly generated key pair as the return address (common when automating proposal submission), or
- uses a key that was registered in a prior era but whose stake registration was not carried forward,

will silently lose their deposit. No special privilege, key compromise, or majority is required. The attacker-controlled entry path is a single ordinary transaction containing a `ProposalProcedure` with a fresh `pProcReturnAddr`.

---

### Recommendation

1. **Remove the bootstrap exemption for the return-address check.** The `unless hardforkConwayBootstrapPhase` guard in `Gov.hs` should not suppress `ProposalReturnAccountDoesNotExist`. Proposals submitted during bootstrap should be held to the same return-address requirement as post-bootstrap proposals.

2. **Alternatively**, if relaxed submission is intentional during bootstrap, `returnProposalDeposits` should auto-create the account (or send the refund to a fallback address) rather than silently diverting the deposit to the treasury, so that the submitter is not penalised for a condition the ledger itself permitted.

---

### Proof of Concept

1. The network is in the Conway bootstrap phase (`hardforkConwayBootstrapPhase pv = True`).
2. Alice generates a fresh key pair `(vk, sk)` and derives `returnAddr` from `vk`. She does **not** register the corresponding stake credential.
3. Alice submits a transaction containing a `ProposalProcedure` (e.g. `ParameterChange` or `InfoAction`) with `pProcReturnAddr = returnAddr` and `pProcDeposit = ppGovActionDeposit`.
4. The `GOV` rule processes the proposal. Because `hardforkConwayBootstrapPhase` is `True`, the `isAccountRegistered` check is skipped. The transaction is accepted; the deposit is deducted from Alice's UTxO.
5. The proposal sits in the governance proposals map. Alice votes or waits; eventually the proposal expires after `ppGovActionLifetime` epochs.
6. At the epoch boundary, `epochTransition` calls `returnProposalDeposits`. The credential for `returnAddr` is absent from the accounts map; the deposit is placed in `unclaimed`.
7. `chainAccountState3` adds `unclaimed` to the treasury. Alice's deposit is gone. She has no mechanism to recover it. [1](#0-0) [2](#0-1) [3](#0-2)
