### Title
`verify_foreign_transaction` Gates on Deprecated Intersection-Based Chain Set Instead of Threshold-Based Available Set — (File: `crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` uses `get_supported_foreign_chains()` (the deprecated strict-intersection rule over `node_foreign_chain_support`) to gate foreign-chain verification requests, instead of `get_available_foreign_chains()` (the threshold-based rule over `foreign_chains_configs`). This is a direct analog to the external report's wrong-data-source accounting bug: the wrong set is consulted, causing both over-rejection (valid requests refused) and under-rejection (stale-data requests accepted but unserviceable).

### Finding Description

In `crates/contract/src/lib.rs` at line 534, `verify_foreign_transaction` calls:

```rust
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(...)
}
```

`get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains from the legacy `node_foreign_chain_support` storage (keyed by account ID). The design documentation explicitly states that `verify_foreign_transaction` must gate on `get_available_foreign_chains()` instead — the threshold-based cache stored in `foreign_chains_configs` (keyed by TLS key), computed by `update_available_chains_config_cache` and exposed via `recompute_available_foreign_chains`.

The two data sources diverge in two critical ways:

1. **Threshold vs. intersection**: `get_available_foreign_chains()` marks a chain available when ≥ `signing_threshold` active participants cover it. `get_supported_foreign_chains()` requires **all** participants to have registered it. A single participant that has not yet registered (e.g., a new participant added via resharing) drops every chain from the supported set, causing all `verify_foreign_transaction` calls to be rejected.

2. **Stale data after resharing**: `node_foreign_chain_support` retains departed participants' registrations until `clean_foreign_chain_data` is explicitly called. During the window between resharing completion and cleanup, the intersection may still include departed participants' entries, making a chain appear "supported" even when fewer than `signing_threshold` active nodes can serve it. Requests for such chains are accepted by the gate but cannot reach the signing threshold and time out.

The design documentation in `docs/design/calculating-supported-foreign-chains.md` and `docs/foreign-chain-transactions.md` explicitly calls out both problems and mandates the switch to `get_available_foreign_chains()`, but the production call site at line 534 was never updated.

### Impact Explanation

**Over-rejection (broken request lifecycle):** Any resharing event that adds a new participant who has not yet called `register_foreign_chains_config` causes `get_supported_foreign_chains()` to return an empty intersection. Every subsequent `verify_foreign_transaction` call panics with `ForeignChainNotSupported`, regardless of how many active participants actually cover the chain. This breaks the entire foreign-chain verification flow until all participants register — a production safety invariant violation.

**Under-rejection (stale-data acceptance):** In the window after resharing removes participants but before `clean_foreign_chain_data` runs, the intersection may include stale entries from departed nodes. A chain can appear "supported" when only 1–2 active participants cover it (below threshold). Requests are accepted and deposited, but the MPC network cannot produce a signature; they time out and the deposit is consumed without service. This breaks the request-lifecycle accounting invariant that an accepted request must be serviceable.

Both directions match the Medium allowed impact: **request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants**.

### Likelihood Explanation

- **Over-rejection** is triggered by any resharing that adds a new participant — a routine governance operation. The new participant need not be malicious; the bug fires automatically until they register.
- **Under-rejection** fires in the window between every resharing completion and the `clean_foreign_chain_data` callback. This window is non-zero in production.
- No privileged access, TEE attack, or network-level DoS is required. Any unprivileged user calling `verify_foreign_transaction` during these windows is affected.

### Recommendation

Replace the `get_supported_foreign_chains()` call in `verify_foreign_transaction` with `get_available_foreign_chains()`:

```rust
// Before (wrong data source):
let supported_chains = self.get_supported_foreign_chains();

// After (correct threshold-based source):
let supported_chains = self.get_available_foreign_chains();
```

This aligns the production gate with the documented invariant and with the already-correct `recompute_available_foreign_chains` / `update_available_chains_config_cache` logic.

### Proof of Concept

**Over-rejection path:**

1. Contract is in Running state with 4 participants (threshold = 3). Bitcoin is whitelisted. Participants 1–3 register Bitcoin via `register_foreign_chains_config` → `get_available_foreign_chains()` returns `{Bitcoin}`.
2. Resharing adds Participant 4. Participant 4 has not yet called `register_foreign_chains_config`.
3. User calls `verify_foreign_transaction` with a Bitcoin request.
4. Line 534: `self.get_supported_foreign_chains()` computes the intersection of all 4 participants' `node_foreign_chain_support` entries. Participant 4 has no entry → intersection is empty.
5. `supported_chains.contains(&Bitcoin)` is `false` → `env::panic_str("ForeignChainNotSupported")`.
6. Result: request rejected despite 3 of 4 active participants covering Bitcoin (threshold met). `get_available_foreign_chains()` would have returned `{Bitcoin}` and accepted the request.

**Under-rejection path:**

1. 4 participants all register Bitcoin. `get_supported_foreign_chains()` returns `{Bitcoin}`.
2. Resharing removes Participants 3 and 4. New Running state has only Participants 1 and 2 (below threshold = 3). `clean_foreign_chain_data` has not yet been called.
3. `node_foreign_chain_support` still contains entries for Participants 3 and 4. Intersection still includes Bitcoin.
4. User calls `verify_foreign_transaction` with Bitcoin. Line 534: `get_supported_foreign_chains()` returns `{Bitcoin}` → request accepted, deposit taken, yield enqueued.
5. MPC nodes attempt to sign but only 2 active participants can contribute (below threshold). Request times out; deposit is consumed without producing a signature.
6. `get_available_foreign_chains()` would have returned `{}` (only 2 active participants cover Bitcoin, below threshold = 3) and rejected the request immediately. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```rust
    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
```
