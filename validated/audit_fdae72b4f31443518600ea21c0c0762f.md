Audit Report

## Title
Unconditional `metricOmmSwapCallback` invocation with zero deltas causes `MetricOmmSimpleRouter` to always revert on no-op swaps - (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.swap()` unconditionally invokes `metricOmmSwapCallback` on `msg.sender` after every swap, including when `_executeSwap` returns `(0, 0, 0)` due to an immediately-satisfied price limit. `MetricOmmSimpleRouter.metricOmmSwapCallback()` reverts with `InvalidSwapDeltas` whenever `amount0Delta <= 0 && amount1Delta <= 0`, which is true for `(0, 0)`. This makes any swap through the router that produces zero deltas permanently revert, breaking the core swap flow for a documented, reachable input class.

## Finding Description
In `MetricOmmPool.swap()`, after `_executeSwap` returns `(amount0Delta, amount1Delta, protocolFeeAmount)`, the callback is fired unconditionally in both the `zeroForOne` and `!zeroForOne` branches with no guard on whether either delta is non-zero:

```solidity
// zeroForOne branch (L257-258)
uint256 balance0Before = balance0();
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);

// !zeroForOne branch (L271-272)
uint256 balance1Before = balance1();
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
```

The internal swap helpers (e.g., `_swapToken0ForToken1SpecifiedInput`) return `(0, 0, 0)` when the price limit is immediately satisfied:

```solidity
if (params.priceLimitX64 >= initialPriceX64) {
    return (0, 0, 0);
}
```

`MetricOmmSimpleRouter.metricOmmSwapCallback()` contains a guard that reverts on this exact case:

```solidity
if (amount0Delta <= 0 && amount1Delta <= 0) revert InvalidSwapDeltas();
```

When the pool calls the callback with `(0, 0, callbackData)`, the condition `0 <= 0 && 0 <= 0` is `true`, so `InvalidSwapDeltas` is thrown and propagates back through `swap()`, reverting the entire transaction.

The contrast with `LiquidityLib.addLiquidity` is direct: that code guards its callback with `if (amount0Added > 0 || amount1Added > 0)` before invoking `metricOmmModifyLiquidityCallback`, establishing the correct pattern that `swap()` does not follow.

## Impact Explanation
The core swap flow through `MetricOmmSimpleRouter` — the primary user-facing router — is broken for any input that produces zero deltas. This is a documented, reachable state (price limit immediately satisfied), not an edge case. Affected entry points include `exactInputSingle`, `exactOutputSingle`, `exactInput`, and `exactOutput`. No user funds are directly drained, but the swap functionality is rendered unusable for this class of inputs, constituting broken core pool functionality under the allowed impact gate.

## Likelihood Explanation
The zero-delta case is reachable by any unprivileged caller via a standard `exactInputSingle` call. It occurs whenever the caller supplies a `priceLimitX64` that is at or beyond the current oracle mid-price (e.g., a conservative slippage guard that the oracle has already moved past), or when oracle price moves between transaction submission and execution. Both scenarios are normal in live trading conditions.

## Recommendation
**Option A (preferred — fix in pool):** Skip the callback entirely when both deltas are zero, mirroring the guard already used in `LiquidityLib.addLiquidity`:

```solidity
if (amount0Delta != 0 || amount1Delta != 0) {
    IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
}
```

**Option B (fix in router):** Change the router guard to distinguish zero from negative, and return early on zero:

```solidity
if (amount0Delta < 0 && amount1Delta < 0) revert InvalidSwapDeltas();
if (amount0Delta == 0 && amount1Delta == 0) return;
```

Option A is preferred because it fixes the root cause at the pool level and protects all current and future callback implementors.

## Proof of Concept
1. Deploy pool with token0/token1 and add liquidity.
2. Deploy `MetricOmmSimpleRouter`.
3. Call `router.exactInputSingle(...)` with `zeroForOne = false` and `priceLimitX64 = 1` (immediately satisfied: `1 < initialPriceX64` for the `!zeroForOne` direction).
4. Pool calls `_swapToken1ForToken0SpecifiedInput` → `params.priceLimitX64 >= initialPriceX64` is false for this direction, but the symmetric case (`_swapToken0ForToken1SpecifiedInput` with `priceLimitX64` set to a value ≥ `initialPriceX64`) returns `(0, 0, 0)`.
5. Pool unconditionally calls `metricOmmSwapCallback(0, 0, "")`.
6. Router guard fires: `revert InvalidSwapDeltas()`.
7. Entire transaction reverts.

A Foundry test can reproduce this by setting `priceLimitX64` to `type(uint128).max` for a `zeroForOne = true` swap (ensuring `priceLimitX64 >= initialPriceX64`), confirming the revert with `InvalidSwapDeltas`.
