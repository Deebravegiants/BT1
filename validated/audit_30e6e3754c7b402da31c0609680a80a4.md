### Title
Zero-Confirmation Bitcoin Requests Bypass Finality Verification in `verify_foreign_transaction` — (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

An unprivileged caller can submit a `verify_foreign_transaction` request for a Bitcoin transaction with `confirmations: 0`. Because `BlockConfirmations` is an unconstrained `u64` and the node-side finality check is `threshold <= actual`, a zero threshold is trivially satisfied for any transaction that has even one confirmation. The MPC network then issues a valid threshold signature over a payload that encodes zero required confirmations, allowing a bridge contract that does not re-inspect the confirmation field to release funds against an insufficiently-final Bitcoin transaction, enabling a double-spend via chain reorganization.

---

### Finding Description

`BlockConfirmations` is defined as a plain `u64` wrapper with no enforced minimum: [1](#0-0) 

The ABI schema confirms `"minimum": 0.0`, so `confirmations: 0` is a valid on-chain argument: [2](#0-1) 

The contract's `verify_foreign_transaction` entry point performs no validation of the `confirmations` field — it only checks domain purpose, gas, deposit, and chain support: [3](#0-2) 

On the node side, `BitcoinInspector::extract` enforces finality with:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { … });
}
``` [4](#0-3) 

When `block_confirmations_threshold = 0`, the inequality `0 <= actual_confirmations` is always true for any confirmed transaction, so the guard is bypassed unconditionally. The inspector then proceeds to sign a `ForeignTxSignPayload::V1` whose embedded `request` encodes `confirmations: 0` — meaning the signed attestation carries no meaningful finality guarantee.

The existing test suite explicitly documents that the inspector succeeds with a zero-extractor list (a related degenerate case), confirming that no lower-bound guard exists for degenerate request parameters: [5](#0-4) 

---

### Impact Explanation

The MPC network produces a valid threshold signature over `ForeignTxSignPayload::V1 { request: { tx_id, confirmations: 0, … }, values: […] }`. Any bridge contract that:

1. Trusts the MPC signature as the sole finality oracle, and
2. Does not independently re-check the `confirmations` field embedded in the signed payload,

will release NEAR-side funds against a Bitcoin transaction that has as few as one confirmation. A one-confirmation Bitcoin transaction can be reorganized with non-negligible probability, allowing the attacker to double-spend: collect NEAR-side funds via the bridge, then replace the Bitcoin transaction on-chain.

This maps to the **High** allowed impact: *forged foreign-chain verification / light-client-style verification bypass that causes invalid bridge execution or double-spend conditions.*

---

### Likelihood Explanation

- The attack requires only a 1-yoctonear deposit and a valid `ForeignTx` domain — no privileged access.
- `confirmations: 0` is accepted by the ABI schema and passes all contract-level checks.
- Bridge contracts that delegate finality entirely to the MPC signature (the intended trust model for this feature) are the primary consumers of `verify_foreign_transaction`; they are the realistic vulnerable target.
- Bitcoin 1-block reorganizations occur in practice; deeper reorgs are rarer but not impossible.

---

### Recommendation

Enforce a protocol-level minimum on `BlockConfirmations` at two layers:

1. **Contract layer** (`verify_foreign_transaction`): reject requests where `confirmations == 0` with an `InvalidParameters` error, analogous to how the contract rejects unsupported chains.
2. **Node layer** (`BitcoinInspector::extract`): assert `block_confirmations_threshold >= 1` before executing the RPC call, so a misconfigured or future caller cannot bypass the guard even if the contract check is absent.

Similarly, consider rejecting `extractors: []` at the contract layer, since a signature over an empty extracted-value set provides no content attestation and is analogous to the zero-amount swap case in the reference report.

---

### Proof of Concept

```json
// Attacker calls verify_foreign_transaction with:
{
  "request": {
    "domain_id": 0,
    "payload_version": "V1",
    "request": {
      "Bitcoin": {
        "tx_id": "<hex of target tx with 1 confirmation>",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    }
  }
}
// Deposit: 1 yoctonear  (satisfies MINIMUM_SIGN_REQUEST_DEPOSIT)
```

Node-side execution path:

1. `verify_foreign_transaction` passes all contract checks (domain, gas, deposit, chain in supported set). [6](#0-5) 
2. MPC nodes index the request and call `BitcoinInspector::extract(tx_id, BlockConfirmations(0), extractors)`.
3. `0 <= 1` → `enough_block_confirmations = true` → guard skipped. [7](#0-6) 
4. Canonical-block check passes (transaction is confirmed in a canonical block).
5. MPC threshold signature is produced over `ForeignTxSignPayload::V1 { request: { confirmations: 0, … }, values: [BlockHash(…)] }`.
6. Attacker receives a valid MPC signature, submits it to a bridge contract, collects NEAR-side funds.
7. Attacker broadcasts a conflicting Bitcoin transaction; with only 1 confirmation the original is reorganized out, completing the double-spend.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
