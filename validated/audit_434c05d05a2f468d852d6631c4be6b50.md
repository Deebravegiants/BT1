### Title
`DirectDeposits` Not Included in Preservation-of-Value Check Allows ADA Creation Out of Thin Air — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `DirectDeposits` field in transaction bodies that allows a transaction to directly credit ADA into registered stake accounts. However, the `DirectDeposits` amount is applied to stake account balances in the `ENTITIES` rule but is **not included in the `produced` side of the preservation-of-value (PoV) check** in the UTxO rule. This breaks the global ADA conservation invariant: a transaction with non-zero `DirectDeposits` passes the PoV check while creating ADA out of thin air.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — missing value flow in the preservation-of-value equation.

**Root cause in `getProducedDijkstraValue`:**

`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` defines the produced-value calculation for Dijkstra:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)
```

`conwayProducedValue` expands to:

```haskell
conwayProducedValue pp isRegPoolId txBody =
  getProducedMaryValue pp isRegPoolId txBody
    <+> inject (txBody ^. treasuryDonationTxBodyL)
```

which ultimately is: `outputs + fee + cert_deposits + treasury_donation`. **`DirectDeposits` (`dtbrDirectDeposits`) is absent.** [1](#0-0) 

**Root cause in `getConsumedDijkstraValue`:**

The consumed-value calculation also does not include `DirectDeposits`:

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels txBody
    (\topTxBody -> txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody)
    txBodyConsumedValue
  where
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
```

`getConsumedMaryValue` computes: `UTxO_inputs + refunds + withdrawals + minted_assets`. **`DirectDeposits` is absent.** [2](#0-1) 

**Where `DirectDeposits` are actually applied:**

In `dijkstraEntitiesTransition`, after certificate processing, `applyDirectDeposits` is called unconditionally on the certified state:

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [3](#0-2) 

`applyDirectDeposits` adds the deposit amounts to stake account balances:

```haskell
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [4](#0-3) 

**The accounting gap:**

The PoV check enforced by `validateValueNotConservedUTxO` verifies:

```
consumed == produced
⟺
UTxO_inputs + refunds + withdrawals = outputs + fee + cert_deposits + treasury_donation
```

But the actual ledger state change is:

| Pot | Change |
|-----|--------|
| UTxO | `outputs − inputs` |
| Stake accounts | `direct_deposits − withdrawals` |
| Fee pot | `+fee` |
| Deposit pot | `+cert_deposits − refunds` |
| Treasury | `+treasury_donation` |

For total ADA to be conserved, the PoV equation must be:

```
inputs + refunds + withdrawals = outputs + fee + cert_deposits + treasury_donation + direct_deposits
```

Because `direct_deposits` is absent from `produced`, the check passes for any `direct_deposits ≥ 0`, while the ledger's total ADA silently increases by exactly `direct_deposits`.

This is structurally identical to the wfCash analog: a balance check measures the wrong "form" of value (WETH vs. native ETH there; UTxO-only vs. UTxO+accounts here), causing a residual value flow to go unaccounted.

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

A transaction with `DirectDeposits = X` ADA passes the PoV check while adding X ADA to stake accounts without deducting X from the UTxO. The attacker can then withdraw those X ADA via a standard reward withdrawal in a subsequent transaction. Repeated exploitation inflates the total ADA supply, violating the 45-billion-ADA hard cap and breaking the economic security of the protocol.

---

### Likelihood Explanation

Any unprivileged transaction sender in the Dijkstra era can include a non-zero `DirectDeposits` field in a top-level or sub-transaction body. The only prerequisite is that the target stake credential is registered (enforced by `directDepositsMissingAccounts`), which the attacker controls. No special keys, governance majority, or privileged access is required. [5](#0-4) 

---

### Recommendation

Include `DirectDeposits` in the `produced` side of the preservation-of-value equation, analogously to how `treasuryDonationTxBodyL` is already included in `conwayProducedValue`. Concretely, `getProducedDijkstraValue` (or `dijkstraProducedValue`) should add:

```haskell
<+> inject (fold . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
```

This ensures that the UTxO inputs must cover the direct-deposit amounts, preserving the global ADA conservation invariant. [6](#0-5) 

---

### Proof of Concept

1. Register stake credential `C` (pays `keyDeposit`).
2. Construct a Dijkstra-era transaction `tx` with:
   - `inputs`: one UTxO entry worth `fee` ADA (just enough to pay the fee).
   - `outputs`: empty (or a change output of 0).
   - `fee`: `fee` ADA.
   - `DirectDeposits`: `{ AccountAddress(C) → 1_000_000_000 }` (1,000 ADA).
   - All other fields: default/empty.
3. Submit `tx`. The PoV check evaluates:
   - `consumed = fee` (UTxO input)
   - `produced = fee` (fee pot)
   - `consumed == produced` ✓ — transaction accepted.
4. The ENTITIES rule calls `applyDirectDeposits`, crediting 1,000 ADA to `C`'s stake account.
5. Submit a withdrawal transaction draining `C`'s account: attacker receives 1,000 ADA that was never in the UTxO.
6. Net result: 1,000 ADA created from nothing. [7](#0-6) [8](#0-7)
