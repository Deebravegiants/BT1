### Title
Byzantine Participant Can Permanently Inflate `IndexerHeightTracker` to `u64::MAX`, Emptying `eligible_leaders` and Freezing All Signing Request Processing — (`crates/node/src/network/indexer_heights.rs`, `crates/node/src/requests/queue.rs`)

---

### Summary

A single authenticated Byzantine participant can send one `Packet::IndexerHeight(IndexerHeightMessage { height: u64::MAX })` message. Because `set_height` is monotonically increasing and never decreases, the stored height for that participant is permanently set to `u64::MAX`. While the Byzantine participant remains connected (alive), `eligible_leaders_and_maximum_height` computes `maximum_height = u64::MAX`, causing the staleness filter to exclude every participant — including the attacker — from `eligible_leaders`, producing an empty set and completely halting signing request processing.

---

### Finding Description

**Step 1 — Entrypoint: Authenticated TLS message delivery**

Any participant with a valid TLS certificate (i.e., a registered MPC participant) can send arbitrary `Packet` variants. In `incoming_connection_handler`, a received `Packet::IndexerHeight` is forwarded without any value validation: [1](#0-0) 

**Step 2 — Monotonically increasing, never-decreasing `set_height`**

`set_height` stores the height only if it is strictly greater than the current value. Once `u64::MAX` is stored, no future legitimate update can ever decrease it: [2](#0-1) 

**Step 3 — `maximum_height` is derived from alive participants only**

`eligible_leaders_and_maximum_height` computes `maximum_height` as the max height across all currently alive participants: [3](#0-2) 

If the Byzantine participant is alive and their stored height is `u64::MAX`, then `maximum_height = u64::MAX`.

**Step 4 — Staleness filter excludes every participant**

The eligibility filter checks:

```
honest_height + STALE_PARTICIPANT_THRESHOLD >= maximum_height
``` [4](#0-3) 

With `maximum_height = u64::MAX`:
- For any honest participant with realistic height `h` and any finite threshold `T`, `h + T < u64::MAX` → excluded.
- For the Byzantine participant themselves: `u64::MAX + T` wraps (Rust release mode) to `T - 1`, which is also `< u64::MAX` → also excluded.

Result: `eligible_leaders` is the **empty set**.

**Step 5 — Persistence**

The stored `u64::MAX` height in `IndexerHeightTracker` is permanent for the lifetime of the process. Even if the Byzantine participant disconnects (removing them from `alive_participants` and temporarily restoring `maximum_height`), reconnecting immediately re-inflates `maximum_height` to `u64::MAX` because the stored height was never reset.

---

### Impact Explanation

With `eligible_leaders` empty, no participant can be selected as a leader for any signing request. All pending and incoming signing requests are frozen — they cannot be processed, assigned, or completed. This constitutes a complete halt of the MPC signing service, matching the Medium impact category: **request-lifecycle manipulation that breaks production safety/accounting invariants** without relying on network-level DoS or operator misconfiguration. A single below-threshold Byzantine participant is sufficient.

---

### Likelihood Explanation

- Requires only one compromised or malicious registered participant (below signing threshold).
- Requires sending a single, trivially crafted message over an already-established authenticated TLS connection.
- No special timing, no collusion, no network-level attack.
- The effect persists as long as the attacker maintains their connection; the stored height persists across reconnections until node restart.

---

### Recommendation

1. **Cap incoming heights**: In `set_height` or at the message-receive boundary, reject or clamp heights that exceed a reasonable upper bound (e.g., current local indexer height + some tolerance).
2. **Use saturating addition in the filter**: Replace `honest_height + STALE_PARTICIPANT_THRESHOLD >= maximum_height` with `honest_height.saturating_add(STALE_PARTICIPANT_THRESHOLD) >= maximum_height`. This prevents overflow wrapping but does not fix the root cause of an inflated `maximum_height`.
3. **Exclude outlier heights from `maximum_height`**: Compute `maximum_height` using a median or trimmed-mean approach rather than the raw maximum, so a single outlier cannot dominate.
4. **Allow height to decrease on reconnect**: Remove the monotonic-only constraint in `set_height`, or reset the stored height when a participant reconnects.

### Proof of Concept

```rust
// Unit test sketch (no network required)
use crate::network::indexer_heights::IndexerHeightTracker;
use crate::primitives::ParticipantId;

let participants = vec![ParticipantId(0), ParticipantId(1), ParticipantId(2)];
let tracker = IndexerHeightTracker::new(&participants);

// Honest participants report realistic heights
tracker.set_height(ParticipantId(0), 1_000_000);
tracker.set_height(ParticipantId(1), 1_000_001);

// Byzantine participant (ParticipantId(2)) sends u64::MAX
tracker.set_height(ParticipantId(2), u64::MAX);

let heights = tracker.get_heights();
// maximum_height = u64::MAX (from participant 2, if alive)
let maximum_height = heights.values().copied().max().unwrap_or(0);
assert_eq!(maximum_height, u64::MAX);

// Staleness check for honest participant 0:
// 1_000_000 + STALE_PARTICIPANT_THRESHOLD < u64::MAX → excluded
let threshold: u64 = 10; // whatever STALE_PARTICIPANT_THRESHOLD is
assert!(heights[&ParticipantId(0)] + threshold < maximum_height);
assert!(heights[&ParticipantId(1)] + threshold < maximum_height);
// eligible_leaders is empty → signing is frozen
```
