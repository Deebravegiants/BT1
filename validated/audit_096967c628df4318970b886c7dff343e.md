Audit Report

## Title
Swap and `addLiquidity` Permanently Broken When USDT Transfer Fee Is Non-Zero — (File: `metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`MetricOmmPool.swap()` and `LiquidityLib.addLiquidity()` verify callback settlement by asserting `balanceBefore + expectedAmount <= balanceAfter`. When USDT (explicitly in scope) has a non-zero transfer fee, the callback delivers `amount - fee` to the pool, making the condition always true and causing every swap and liquidity addition to revert. `removeLiquidity()` silently delivers `amount - fee` to LPs, causing unrecoverable principal loss.

## Finding Description
In `MetricOmmPool.swap()`, the `zeroForOne` branch snapshots `balance0Before`, invokes the callback instructing it to pay `amount0Delta`, then checks:

```solidity
if (amount0Delta > 0 && balance0Before + uint256(amount0Delta) > balance0()) {
    revert IncorrectDelta();
}
```

`balance0()` is a plain `IERC20(TOKEN0).balanceOf(address(this))` call. When token0 is USDT with fee `f > 0`, the callback's `transferFrom(payer, pool, amount0Delta)` causes the pool to receive `amount0Delta - f`. The condition evaluates to `balance0Before + amount0Delta > balance0Before + amount0Delta - f`, which is always `true`, so `IncorrectDelta` always fires.

The identical pattern exists in `LiquidityLib.addLiquidity()`:

```solidity
uint256 balance0Before = IERC20(ctx.token0).balanceOf(address(this));
IMetricOmmModifyLiquidityCallback(msg.sender)
    .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
if (amount0Added > 0 && balance0Before + amount0Added > IERC20(ctx.token0).balanceOf(address(this))) {
    revert IMetricOmmPoolActions.InsufficientTokenBalance();
}
```

For `removeLiquidity()`, `safeTransfer(owner, amount0Removed)` is called with the pre-fee computed amount; USDT deducts its fee in transit, so the LP receives `amount0Removed - fee` while the pool's internal accounting debits the full `amount0Removed`, causing a permanent balance shortfall.

## Impact Explanation
Any pool whose token0 or token1 is USDT with a non-zero fee becomes completely non-functional: `swap()` reverts with `IncorrectDelta` on every call; `addLiquidity()` reverts with `InsufficientTokenBalance`. This constitutes broken core pool functionality. Additionally, `removeLiquidity()` silently delivers less than the computed owed amount to LPs, constituting direct loss of LP principal — both within the allowed impact gate.

## Likelihood Explanation
USDT's `basisPointsRate` is currently 0 but the setter is live and callable by Tether's owner at any time without any on-chain precondition from the protocol's side. Pools pairing USDT are a natural and intended deployment target (confirmed by the contest's explicit USDT-in-scope declaration and the numerous USDT feed configurations in the deployment scripts). Once the fee is enabled, breakage is immediate and requires no attacker action — any ordinary swap or liquidity call triggers it.

## Recommendation
Replace expected-amount checks with actual-received-delta checks in both locations:

```solidity
// swap, zeroForOne branch
uint256 balance0Before = balance0();
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
uint256 actualReceived = balance0() - balance0Before;
if (amount0Delta > 0 && actualReceived < uint256(amount0Delta)) {
    revert IncorrectDelta();
}
```

Apply the same pattern in `LiquidityLib.addLiquidity()`. For `removeLiquidity()`, either accept fee-reduced delivery or explicitly gate pool creation to reject fee-on-transfer tokens.

## Proof of Concept
1. Deploy a pool with USDT as `token0` and any standard ERC20 as `token1`.
2. Enable USDT's fee: `usdt.setParams(10, 100)` (10 bps, max 100 USDT).
3. Seed the pool with liquidity via direct token transfer (bypassing `addLiquidity`).
4. Call `pool.swap(recipient, true, 10_000e6, 0, callbackData, "")`.
5. The router callback executes `usdt.transferFrom(payer, pool, amount0Delta)`; USDT deducts its fee; pool receives `amount0Delta - fee`.
6. `balance0Before + amount0Delta > balance0()` evaluates `true` → `IncorrectDelta` revert.
7. Repeat for `addLiquidity` — same revert path via `InsufficientTokenBalance`.
8. Call `removeLiquidity` — succeeds, but LP wallet receives `amount0Removed - fee`; the shortfall is unrecoverable. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
