### Title
Stale rsETH Price Propagated Cross-Chain When `PriceAboveDailyThreshold` Guard Blocks Non-Manager Updates - (`contracts/cross-chain/MultiChainRateProvider.sol`)

---

### Summary

`updateRate()` in `MultiChainRateProvider` has no access control and reads the stored `rsETHPrice` state variable directly from `LRTOracle`. When the true rsETH price rises beyond `pricePercentageLimit`, `_updateRsETHPrice()` reverts for non-managers, leaving `rsETHPrice` at the old lower value. Any caller can then invoke `updateRate()` to broadcast this stale, underpriced rate to all destination chains via LayerZero.

---

### Finding Description

**Call chain:**

`updateRate()` (no access control) → `getLatestRate()` → `ILRTOracle(rsETHPriceOracle).rsETHPrice()`

`RSETHMultiChainRateProvider.getLatestRate()` simply reads the stored `rsETHPrice` state variable: [1](#0-0) 

`rsETHPrice` is only updated at the very end of `_updateRsETHPrice()`: [2](#0-1) 

When the new computed price exceeds `highestRsethPrice` by more than `pricePercentageLimit`, non-managers hit this guard: [3](#0-2) 

The function reverts before reaching line 313, so `rsETHPrice` stays at the pre-increase value. Meanwhile, `updateRate()` carries no role check: [4](#0-3) 

Any unprivileged caller can immediately invoke `updateRate()`, which reads the stale `rsETHPrice` and broadcasts it to every registered `RSETHRateReceiver` via LayerZero.

---

### Impact Explanation

Destination chain pools that consume the received rate to price rsETH/ETH deposits will see an artificially low rate (old price < true price). Users depositing ETH on destination chains receive more rsETH per ETH than the true exchange rate warrants, diluting existing rsETH holders. This matches the scoped impact: **Low — contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

The precondition — a legitimate TVL/reward accrual event that pushes the price above `pricePercentageLimit` — is a normal operational occurrence (e.g., a large staking reward epoch). The window between the revert and the manager's `updateRSETHPriceAsManager()` call is the attack window. `updateRate()` is permissionless and costs only gas + LZ fees, so any actor (including a bot) can exploit it immediately after the revert.

---

### Recommendation

Add a freshness/consistency check in `updateRate()` (or in `getLatestRate()`) that calls `updateRSETHPrice()` first and only proceeds if it succeeds, or alternatively gate `updateRate()` behind the same manager role that is required when the price is above the threshold. A simpler mitigation is to have `RSETHMultiChainRateProvider.getLatestRate()` call a view function that recomputes the price on-the-fly rather than reading the potentially stale stored `rsETHPrice`.

---

```solidity

// 2. Simulate TVL increase of 2% (mock asset oracle returns higher price)
// 3. Call lrtOracle.updateRSETHPrice() as non-manager → expect revert PriceAboveDailyThreshold
// 4. Assert lrtOracle.rsETHPrice() == preIncreasePrice (stale)
// 5. Call rsETHMultiChainRateProvider.updateRate{value: fee}()
// 6. Assert emitted RateUpdated event carries preIncreasePrice, not the true higher price

```

The stale rate is provably the pre-increase value because `rsETHPrice` is never written when `_updateRsETHPrice()` reverts, and `getLatestRate()` reads only that stored variable. [1](#0-0) [5](#0-4)
