### Title
Single Byzantine Participant Can Deliver Forged CKD Response for `AppPublicKey` Requests, Bypassing Threshold Requirement - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_ckd` in the MPC contract performs no cryptographic verification of the `CKDResponse` when the request uses the `AppPublicKey` variant. Any single attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary fabricated response, which the contract unconditionally accepts and delivers to the user. Because `resolve_yields_for` removes the pending request on the first successful call, honest participants cannot subsequently deliver the correct response.

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 653–689), the response verification is gated on the variant of `app_public_key`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check` to verify the response against the BLS12-381 master public key. For `AppPublicKey` (the "privately verifiable" / legacy variant, still accepted per the README), the arm is a no-op: any `CKDResponse` with arbitrary `big_y` and `big_c` values passes straight through to `resolve_yields_for`, which drains the entire pending queue and delivers the fabricated output to every waiting caller. [2](#0-1) 

`resolve_yields_for` removes the map entry on the first successful call:

```rust
let resumed = requests
    .remove(request)          // ← entry gone after first call
    .unwrap_or_default()
    ...
``` [3](#0-2) 

Once the malicious participant's `respond_ckd` transaction lands, the pending entry is gone. Any subsequent honest `respond_ckd` call returns `Err(RequestNotFound)` and cannot correct the damage.

The access guard (`assert_caller_is_attested_participant_and_protocol_active`) requires only that the caller is a single attested participant in the current epoch — it does not require threshold agreement on the response value. [4](#0-3) 

### Impact Explanation

**Critical — Unauthorized confidential key derivation output without the required participant authorization.**

A user calling `request_app_private_key` with `AppPublicKey` expects to receive a key share derived from the MPC master BLS12-381 key via the threshold protocol. Instead, a single Byzantine participant can:

1. Observe the pending `CKDRequest` on-chain.
2. Construct an arbitrary `CKDResponse` (`big_y`, `big_c` of their choosing).
3. Submit `respond_ckd` before the honest leader, consuming the pending slot.
4. The user receives a fake derived key — one the attacker chose and therefore knows.

Because `AppPublicKey` is "privately verifiable," the user can only check internal consistency (`big_c` is correctly related to `big_y` under their own app private key), not that `big_y` is the correct MPC-derived point. The attack is undetectable by the user at response time. The attacker can deliver a key they fully control, enabling them to decrypt any data the user subsequently encrypts under that key.

### Likelihood Explanation

The attacker must be an attested participant (requires a valid TEE attestation), but needs only one such participant — strictly below the signing threshold. The attack is a simple race: submit `respond_ckd` with a crafted response before the honest leader. On NEAR, transaction ordering within a block is determined by the validator; a malicious participant can submit with maximum gas to maximize priority. The `AppPublicKey` variant is still accepted as a production path (described as "legacy" in the README, not deprecated or gated off). [5](#0-4) 

### Recommendation

Add cryptographic verification for `AppPublicKey` responses in `respond_ckd`, analogous to the `AppPublicKeyPV` path. If on-chain verification is not feasible for the privately-verifiable variant, either:

- Deprecate `AppPublicKey` and require all new requests to use `AppPublicKeyPV` (which has on-chain verification via `ckd_output_check`), or
- Require the response to include a zero-knowledge proof or commitment that binds `big_y` to the BLS12-381 master public key and the request's derivation parameters, verifiable without the user's app private key.

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(g1_point)` and `domain_id` pointing to a BLS12-381 domain. A yield is enqueued in `pending_ckd_requests`.
2. Malicious attested participant observes the pending `CKDRequest` on-chain.
3. Malicious participant calls `respond_ckd(ckd_request, CKDResponse { big_y: attacker_point, big_c: attacker_point })` — any BLS12-381 G1 points.
4. `respond_ckd` passes the `AppPublicKey(_) => {}` arm with no check, then calls `resolve_yields_for`, which removes the entry and resumes the user's yield with the fabricated response.
5. The user's `request_app_private_key` promise resolves with the attacker-chosen `CKDResponse`. Honest participants' subsequent `respond_ckd` calls return `Err(RequestNotFound)`.
6. The user decrypts `big_c` with their app private key and obtains a key the attacker chose — not the MPC-derived key. [6](#0-5) [7](#0-6)

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
