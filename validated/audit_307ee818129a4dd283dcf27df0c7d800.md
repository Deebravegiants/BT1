### Title
`danglingVRFKeyHashes` uses `Map.withoutKeys` (full deletion) instead of count-decrement, silently erasing shared VRF1 from `psVRFKeyHashes` and breaking the post-v11 VRF uniqueness invariant — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

After the v11 hardfork, `psVRFKeyHashes` tracks VRF key usage counts to enforce uniqueness. When a pool re-registers with a new VRF key, `poolReapTransition` computes `danglingVRFKeyHashes` (the old VRF keys being replaced) and removes them with `Map.withoutKeys`. This is a **full deletion** regardless of the count. If the old VRF key has count > 1 (because multiple pools shared it pre-v11, which was allowed), the entire entry is deleted even though other pools still hold that key. The `psVRFKeyHashes` map then becomes inconsistent with `psStakePools`, and the uniqueness guard in `poolTransition` can be bypassed to register a new pool with a VRF key already in active use.

### Finding Description

**Root cause — `PoolReap.hs` lines 138–148 and 224–227:**

`danglingVRFKeyHashes` is the set of old VRF keys being replaced by re-registration:

```haskell
danglingVRFKeyHashes =
  Set.fromList $ Map.elems $
    Map.merge
      Map.dropMissing Map.dropMissing
      (Map.zipWithMaybeMatched $ \_ sps sppF ->
          if sps ^. spsVrfL /= sppF ^. sppVrfL
            then Just (sps ^. spsVrfL) else Nothing)
      (ps0 ^. psStakePoolsL)
      (ps0 ^. psFutureStakePoolParamsL)
``` [1](#0-0) 

These are then removed from `psVRFKeyHashes` via:

```haskell
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )
``` [2](#0-1) 

`Map.withoutKeys` deletes the entire map entry. For retired pools, the code correctly uses `removeVRFKeyHashOccurrences` which decrements the count by 1 and only removes the entry when it reaches 0:

```haskell
removeVRFKeyHashOccurrence =
  Map.update (mapNonZero (\n -> n - 1))
``` [3](#0-2) 

No equivalent decrement is applied to `danglingVRFKeyHashes`. The asymmetry is the bug.

**Exact state trace:**

| Step | Action | `psStakePools` | `psFutureStakePoolParams` | `psVRFKeyHashes` |
|------|--------|----------------|--------------------------|-----------------|
| 1 | Register A(VRF1), B(VRF1) pre-v11 | {A:VRF1, B:VRF1} | {} | {} (not yet populated) |
| 2 | Hardfork v11 → `populateVRFKeyHashes` | {A:VRF1, B:VRF1} | {} | {VRF1:2} |
| 3 | Re-register A with VRF2 (post-v11) | {A:VRF1, B:VRF1} | {A:VRF2} | {VRF1:2, VRF2:1} |
| 4 | Epoch boundary (POOLREAP) | {A:VRF2, B:VRF1} | {} | **{VRF2:1}** ← VRF1 gone! |
| 5 | Register C with VRF1 | {A:VRF2, B:VRF1, C:VRF1} | {} | {VRF1:1, VRF2:1} |

**Step 3 detail — re-registration check passes correctly:**

The re-registration guard in `poolTransition` checks:
```haskell
sppVrf == stakePoolState ^. spsVrfL   -- VRF2 == VRF1 → False
  || Map.notMember sppVrf psVRFKeyHashes  -- VRF2 ∉ {VRF1:2} → True
``` [4](#0-3) 

VRF2 is fresh, so this is correct. The `updateFutureVRFKeyHash` branch inserts VRF2 with count 1 (no prior future params for A): [5](#0-4) 

**Step 4 detail — POOLREAP destroys VRF1:**

`danglingVRFKeyHashes` merges `psStakePools` and `psFutureStakePoolParams` with `Map.dropMissing` on both sides — only pool A appears in both. Pool A's current VRF1 ≠ future VRF2, so `danglingVRFKeyHashes = {VRF1}`. `Map.withoutKeys {VRF1}` then removes VRF1 entirely from `{VRF1:2, VRF2:1}`, yielding `{VRF2:1}`. Pool B still holds VRF1 in `psStakePools` but VRF1 is now absent from the tracking map.

**Step 5 detail — uniqueness guard bypassed:**

New-pool registration check:
```haskell
Map.notMember sppVrf psVRFKeyHashes
  ?! injectFailure (VRFKeyHashAlreadyRegistered sppId sppVrf)
``` [6](#0-5) 

`Map.notMember VRF1 {VRF2:1}` = `True` → guard passes → pool C registered with VRF1. Both pool B and pool C now appear in `psStakePools` with VRF1, violating the post-v11 invariant.

**Secondary corruption:** After step 5, `psVRFKeyHashes = {VRF1:1, VRF2:1}`. But the true count for VRF1 is 2 (pools B and C). Any subsequent retirement of pool B will decrement VRF1 to 0 and remove it, leaving pool C's VRF1 untracked — the invariant remains broken indefinitely.

### Impact Explanation

The VRF key is used for slot-leader election. With two active pools (B and C) sharing VRF1 in `psStakePools` and the tracking map inconsistent, the ledger is in an invalid state. If the attacker controls the private VRF key corresponding to VRF1 (which they must, to register pool C with it), they can produce blocks for slots where pool B is elected, redirecting block rewards from pool B's delegators to pool C's delegators. This constitutes direct ADA reward misdirection through an invalid ledger state transition. Additionally, the `psVRFKeyHashes` count for VRF1 will be permanently wrong, causing further corruption on every subsequent retirement or re-registration touching VRF1.

---

### Likelihood Explanation

The precondition — two pools sharing a VRF key pre-v11 — was explicitly allowed and is a realistic historical state. The hardfork to v11 is a normal protocol upgrade. Re-registering a pool with a new VRF key is a routine operation. All five steps are reachable by unprivileged pool operators through standard pool certificate transactions. No governance majority, leaked key, or privileged access is required beyond controlling the private VRF key that was used to register pool C (which the attacker chose themselves).

### Recommendation

Replace `Map.withoutKeys danglingVRFKeyHashes` with `removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)` — the same count-decrement function already used for retired pools. This ensures that when a VRF key has count > 1, only one reference is removed per re-registration, preserving the correct count for pools that still hold that key.

```haskell
-- Current (buggy):
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )

-- Fixed:
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
     )
```

### Proof of Concept

Construct the following ledger sequence (all via standard pool certificates):

1. At protocol version 10: submit `RegPool A VRF1` and `RegPool B VRF1`. Both succeed (pre-v11 allows shared VRF keys).
2. Enact hardfork to v11 via governance. `populateVRFKeyHashes` runs: `psVRFKeyHashes = {VRF1:2}`. [7](#0-6) 
3. Submit `RegPool A VRF2` (re-registration). Guard passes (VRF2 is fresh). `psVRFKeyHashes = {VRF1:2, VRF2:1}`, `psFutureStakePoolParams = {A:VRF2}`.
4. `passEpoch`. POOLREAP computes `danglingVRFKeyHashes = {VRF1}` and calls `Map.withoutKeys {VRF1}`. `psVRFKeyHashes = {VRF2:1}`. Pool B still has VRF1 in `psStakePools`.
5. Submit `RegPool C VRF1`. Guard: `Map.notMember VRF1 {VRF2:1}` = True → **succeeds**.
6. Assert `psStakePools` contains both B and C with VRF1 → invariant broken.
7. Assert `psVRFKeyHashes = {VRF1:1, VRF2:1}` → count is wrong (should be 2 for VRF1).

The existing test `"Retiring a stake pool with a duplicate VRF Keyhash after v11 HardFork"` in `HardForkSpec.hs` covers retirement of duplicate-VRF pools but does **not** cover the re-registration path that triggers `danglingVRFKeyHashes` with a count-2 entry, leaving this path untested. [8](#0-7)
