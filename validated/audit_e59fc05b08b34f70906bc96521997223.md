### Title
`respond_verify_foreign_tx` Accepts Arbitrary `payload_hash` Without Binding It to the Original Request - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the MPC root public key. It never checks that `payload_hash` was actually derived from a `ForeignTxSignPayload` that encodes the original `request.request`. A single Byzantine attested participant (strictly below the signing threshold) can resolve a pending foreign-tx request with a legitimately-produced MPC signature that was computed for a *different* request, delivering a forged verification outcome to the caller.

### Finding Description

In `respond_verify_foreign_tx` the contract performs two checks:

1. The caller is an attested participant.
2. `verify_ecdsa_signature(signature, response.payload_hash, root_pk)` succeeds. [1](#0-0) 

It then resolves the yield keyed on `request`: [2](#0-1) 

The missing check is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: <actual extracted values> }))`. Because the extracted values are never stored on-chain, the contract has no way to recompute the expected hash — but it also makes no attempt to bind the hash to the original request at all.

Contrast this with the regular `respond` function, which verifies the signature directly against `request.payload` (the payload stored in the pending request), making cross-request substitution impossible: [3](#0-2) 

The `ForeignTxSignPayload` structure that nodes sign includes both the original `ForeignChainRpcRequest` and the `ExtractedValue` list: [4](#0-3) 

The SDK-level verifier (`ForeignChainSignatureVerifier`) does perform this binding check, but the on-chain contract does not: [5](#0-4) 

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) can:

1. Participate honestly in signing Request B (tx\_id = 0xBB), obtaining the threshold signature and `payload_hash_B`.
2. Call `respond_verify_foreign_tx(request = A, response = {payload_hash_B, sig_B})` — substituting Request A's key with Request B's signed payload.
3. The contract accepts the call (signature is valid over `payload_hash_B`, and Request A exists in the pending map).
4. Request A's yield is resolved and the caller receives `VerifyForeignTransactionResponse { payload_hash: payload_hash_B, signature: sig_B }`.

The caller of Request A now holds a valid MPC signature that commits to a completely different foreign-chain transaction (tx\_id = 0xBB). Any downstream smart contract that does not independently recompute and compare the expected `payload_hash` (as the SDK recommends but does not enforce) will accept this as proof that tx\_id = 0xAA was verified, enabling forged foreign-chain verification and potential invalid bridge execution.

### Likelihood Explanation

- Requires only **one** Byzantine attested participant — strictly below the signing threshold.
- Requires two concurrent pending foreign-tx requests, which is a normal production condition.
- The Byzantine node already possesses the threshold signature for Request B as a natural byproduct of participating in the signing round.
- No special tooling or privileged access beyond being an attested participant is needed.

### Recommendation

The contract must bind `response.payload_hash` to the original request before accepting the response. Since the extracted values are not stored on-chain, the simplest fix is to restructure the response so that the MPC nodes include the full `ForeignTxSignPayload` (or at minimum the `ForeignChainRpcRequest` portion) in the response, allowing the contract to verify:

```
response.payload_hash == SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: response.values }))
```

Alternatively, store a commitment to the request inside the pending-request map entry and verify that `payload_hash` is a hash that could only have been produced from that specific request (e.g., by checking that the first N bytes of the pre-image match the Borsh encoding of `request.request`).

### Proof of Concept

```
// Setup: two concurrent pending requests
verify_foreign_transaction(request_A)  // tx_id = 0xAA
verify_foreign_transaction(request_B)  // tx_id = 0xBB

// MPC network signs payload_B = SHA-256(borsh(ForeignTxSignPayload { request_B, values_B }))
// Byzantine node obtains (payload_hash_B, sig_B)

// Byzantine node calls respond with request_A but response for B:
respond_verify_foreign_tx(
    request = request_A,
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_B,   // hash of B's payload
        signature: sig_B,               // valid MPC sig over payload_hash_B
    }
)

// Contract checks:
// 1. caller is attested participant ✓
// 2. verify_ecdsa_signature(sig_B, payload_hash_B, root_pk) ✓  (legitimate sig)
// 3. request_A exists in pending map ✓
// → resolves request_A's yield with response for B

// Caller of request_A receives a valid MPC signature over tx_id=0xBB's data,
// presented as verification of tx_id=0xAA.
```

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
