Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Cap Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool` omits the incoming `amount` from the ETH branch of the deposit cap check, while correctly including it for ERC20 assets. As a result, any single ETH deposit can push total ETH deposits arbitrarily above the configured `depositLimitByAsset` cap without reverting.

## Finding Description
In `contracts/LRTDepositPool.sol`, `_checkIfDepositAmountExceedesCurrentLimit` has an asymmetric implementation: [1](#0-0) 

The ETH branch at line 679 evaluates `totalAssetDeposits > depositLimit`, ignoring the incoming `amount`. The ERC20 branch at line 681 correctly evaluates `totalAssetDeposits + amount > depositLimit`. This means the ETH check only detects a limit breach if the pre-existing total has *already* exceeded the cap — the new deposit's contribution is never considered.

The call path is: `depositETH` (L76–93) → `_beforeDeposit` (L648–670) → `_checkIfDepositAmountExceedesCurrentLimit` (L676–682). The `_beforeDeposit` function passes `msg.value` as `depositAmount` and reverts on `true`, but the ETH branch never returns `true` for a deposit that would push the total over the limit. [2](#0-1) 

No other guard in `depositETH` or `_beforeDeposit` enforces the deposit cap for ETH.

## Impact Explanation
**Low — Contract fails to deliver promised returns.**

The `depositLimitByAsset[ETH_TOKEN]` cap is a protocol invariant intended to bound total ETH exposure. Because the ETH branch ignores `amount`, a depositor can mint rsETH for an arbitrarily large ETH deposit in a single transaction, violating the cap. No funds are directly stolen and no user loses value, but the protocol mints more rsETH than the cap was designed to allow, breaking the stated invariant.

## Likelihood Explanation
Any unprivileged user calling `depositETH` with `msg.value` large enough to exceed the remaining cap will succeed without revert, provided `getTotalAssetDeposits(ETH_TOKEN)` has not already crossed the limit before their call. No special role, front-running, or multi-step setup is required — a single transaction suffices.

## Recommendation
Mirror the ERC20 branch in the ETH branch by adding `amount`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [3](#0-2) 

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether`.
2. Normal usage accumulates `getTotalAssetDeposits(ETH_TOKEN) = 99 ether`.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit`: evaluates `99e18 > 100e18` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 10 000 ETH; total ETH in protocol becomes 10 099 ETH, ~100× the intended cap.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    vm.prank(admin);
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, 100 ether);
    // seed 99 ether via normal deposits ...
    vm.deal(attacker, 10_000 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 10_000 ether}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 100 ether);
}
```

    function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
        }
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
