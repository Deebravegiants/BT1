Audit Report

## Title
LP Shares Burned With Zero Token Return Due to Floor Division Truncation in `removeLiquidity` - (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

In `LiquidityLib.removeLiquidity`, per-bin token amounts owed to a withdrawing LP are computed via plain floor division with no zero-output guard. When a bin's scaled token balance is small relative to total shares ŌĆö a state reachable through normal swap activity ŌĆö the division truncates to zero. The LP's shares are permanently burned while the pool retains the tokens, silently redistributing the LP's principal to remaining LPs.

## Finding Description

At lines 205ŌĆō206 of `LiquidityLib.sol`, the token amounts owed to the withdrawing LP are computed as:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [1](#0-0) 

Both are plain Solidity floor divisions. When `binState.token0BalanceScaled * sharesToRemove < binTotalSharesVal`, `amount0Scaled` evaluates to `0`. The code then unconditionally proceeds to:

1. Subtract `0` from `binState.token0BalanceScaled` (no change to pool balance) ŌĆö line 210
2. Reduce `binTotalShares[binIdx]` by `sharesToRemove` ŌĆö line 213
3. Reduce `positionBinShares[posKey]` by `sharesToRemove` ŌĆö line 214
4. Transfer `0` tokens to the LP owner (transfer is gated on `amount0Removed > 0`) ŌĆö lines 242ŌĆō244 [2](#0-1) [3](#0-2) 

The only existing guard is the `minimalMintableLiquidity` check at lines 200ŌĆō202:

```solidity
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
``` [4](#0-3) 

This guard only prevents leaving a dust *position* (non-zero `newUserShares < minimalMintableLiquidity`). It does not prevent removing shares that yield zero tokens. An LP removing all their shares (`newUserShares = 0`) bypasses this check entirely.

**How the vulnerable state arises:** Swaps consume `token0BalanceScaled` without touching `binTotalShares`. After heavy trading through a bin, `token0BalanceScaled` can be reduced to near zero while `binTotalShares` remains large. If a second LP subsequently adds shares to the depleted bin (increasing `binTotalShares` further via `addLiquidity`'s `ceilDiv` path at line 109), the ratio `token0BalanceScaled / binTotalShares` becomes negligible. [5](#0-4) 

## Impact Explanation

Direct loss of LP principal. The LP burns valid shares and receives zero tokens in return. The forfeited token balance remains in the bin and is redistributed to remaining LPs proportionally. This is a silent, irreversible transfer of value from the withdrawing LP to other LPs, triggered by normal pool operation (swaps depleting a bin's balance). No privileged access is required. The impact meets the Sherlock threshold for direct loss of user principal.

## Likelihood Explanation

Any bin that has been the active trading bin and has had its token balance significantly depleted by swaps is vulnerable. No malicious setup, privileged access, or non-standard tokens are required. The condition arises organically in any pool with active trading volume. An LP who adds liquidity early, waits for swaps to deplete the bin, and then removes liquidity will silently lose their remaining token claim. A second LP adding shares to the depleted bin amplifies the effect by increasing `binTotalShares` while `token0BalanceScaled` remains near zero.

## Recommendation

Add a zero-output guard in `removeLiquidity`. After computing `amount0Scaled` and `amount1Scaled`, check:

```solidity
if (sharesToRemove > 0) {
    if (amount0Scaled == 0 && binState.token0BalanceScaled > 0) {
        revert ZeroTokenReturn();
    }
    if (amount1Scaled == 0 && binState.token1BalanceScaled > 0) {
        revert ZeroTokenReturn();
    }
}
```

This forces the LP to either remove a larger share batch (so the proportional amount rounds to at least 1 scaled unit) or wait for the bin to be replenished. This is safer than using `ceilDiv` for the LP's benefit, as `ceilDiv` could allow the LP to extract more than their proportional share from a nearly-empty bin.

## Proof of Concept

```
Setup:
  binTotalShares[0]            = 10_000   (LP1 initial deposit)
  binState.token0BalanceScaled = 10_000   (1:1 ratio after initial deposit)

After swaps consume all but 1 unit of token0:
  binTotalShares[0]            = 10_000   (unchanged by swaps)
  binState.token0BalanceScaled = 1

LP2 adds 90_000 shares to the depleted bin (addLiquidity uses ceilDiv, so LP2 pays ceil(1 * 90_000 / 10_000) = 9 scaled units):
  binTotalShares[0]            = 100_000
  binState.token0BalanceScaled = 10        (1 + 9)

LP1 removes all 10_000 shares:
  amount0Scaled = (10 * 10_000) / 100_000 = 1   ŌåÉ LP1 gets 1 scaled unit

But if token0BalanceScaled = 1 (before LP2 joins):
  amount0Scaled = (1 * 10_000) / 100_000 = 0   ŌåÉ LP1 burns 10_000 shares, receives 0 tokens

Foundry test plan:
1. Deploy pool, LP1 adds 10_000 shares to bin 0.
2. Simulate swaps that reduce token0BalanceScaled to 1 (direct storage manipulation or swap calls).
3. LP2 adds 90_000 shares to bin 0.
4. LP1 calls removeLiquidity for all 10_000 shares.
5. Assert amount0Removed == 0 and LP1's token balance is unchanged, confirming share burn with zero return.
```
