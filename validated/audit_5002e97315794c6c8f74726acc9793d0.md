### Title
Stale Measurement Votes from Removed Participants Count Toward New-Epoch Threshold in Race Window After Resharing — (`crates/contract/src/lib.rs`, `crates/contract/src/tee/measurements.rs`)

---

### Summary

`vote_add_os_measurement` uses an unfiltered `count_votes` that tallies every entry in `MeasurementVotes::vote_by_account` regardless of whether the voter is still a current participant. A detached `clean_tee_status` promise cleans stale votes only in the *next* block. In the one-block window between resharing completion and that cleanup, a removed participant's vote cast during the Resharing phase combines with new-participant votes, reducing the effective honest-vote requirement by the number of stale entries.

---

### Finding Description

**Entrypoint and guard chain in `vote_add_os_measurement`:**

`voter_or_panic()` → `voter_account()` → `authenticate_update_vote()` checks the signer against the *state-appropriate* participant set: [1](#0-0) 

During **Resharing**, this checks `previous_running_state.parameters.participants()` — the *old* set — so a removed participant P_old can vote. After resharing completes (**Running**), it checks the new set, so P_new can vote.

`threshold_parameters_or_panic()` follows the same pattern: [2](#0-1) 

During Resharing it returns the old parameters; after resharing it returns the new parameters.

**The unfiltered vote count:** [3](#0-2) 

`count_votes` iterates every value in `vote_by_account` with no participant-membership filter. It counts P_old's stale entry alongside P_new's fresh entry.

**The race window:**

`vote_reshared` transitions `protocol_state` to `Running` and then spawns `clean_tee_status` as a *detached* promise: [4](#0-3) 

Detached promises execute in the **next block**. During that one-block gap, `measurement_votes` still contains votes from removed participants.

`clean_tee_status` eventually calls: [5](#0-4) 

which filters `measurement_votes` to current participants — but only after the window has passed.

**Contrast with `vote_update`:**

`vote_update` explicitly defends against exactly this pattern with an inline filter: [6](#0-5) 

The comment reads: *"This ensures correctness even if the cleanup promise in `MpcContract::vote_reshared()` fails."* No equivalent guard exists in `vote_add_os_measurement`.

**Concrete attack sequence (N_old = N_new = 5, T_new = 3; P5 removed, P6 added):**

1. **Resharing state**: P5 (Byzantine, to-be-removed) calls `vote_add_os_measurement(M_malicious)`. `authenticate_update_vote` passes (P5 is in `previous_running_state`). Vote stored in `measurement_votes`.
2. **`vote_reshared` completes**: state transitions to Running `{P1,P2,P3,P4,P6}`. `clean_tee_status` detached.
3. **Race window (same or next block)**: P6 (Byzantine, new) calls `vote_add_os_measurement(M_malicious)`. `authenticate_update_vote` passes (P6 is in new Running set). `count_votes` = P5's stale vote + P6's vote = **2**.
4. P4 (Byzantine, new) calls `vote_add_os_measurement(M_malicious)`. `count_votes` = **3** = T_new → `add_measurement` fires. M_malicious is now an allowed OS measurement.
5. `clean_tee_status` runs in the next block — too late.

Without the stale vote, 3 Byzantine *new* participants would be required. With it, only 2 are needed.

---

### Impact Explanation

Adding a malicious OS measurement (`allowed_measurements`) allows a node running in a non-genuine TEE environment to pass attestation verification (`reverify_participants`), become an attested participant, and participate in key-generation or resharing rounds. Because the TEE is the confidentiality boundary for key-share material, an attested node running in a compromised environment can exfiltrate shares. The direct on-chain effect is a governance invariant break: the threshold for approving a new measurement is effectively T_new − (number of stale votes), not T_new.

---

### Likelihood Explanation

The race window is exactly one NEAR block (~1 s). Byzantine participants monitoring the chain can pre-sign and broadcast `vote_add_os_measurement` transactions immediately after observing the `vote_reshared` receipt. The attack requires one Byzantine removed participant (who voted during Resharing) plus T_new − 1 Byzantine new participants — strictly below the signing threshold. No operator misconfiguration or network-level attack is needed.

---

### Recommendation

Apply the same defensive inline filter used in `vote_update` to `vote_add_os_measurement` (and symmetrically to `vote_remove_os_measurement`, `vote_code_hash`, and `vote_add_launcher_hash`):

```rust
// After recording the vote, count only votes from current participants:
let current_participants = self.protocol_state.threshold_parameters_or_panic().participants();
let valid_votes = self.tee_state
    .measurement_votes
    .get_remaining_votes(current_participants)
    .count_votes_for(&action);   // expose count_votes as pub or add a helper

if valid_votes >= self.threshold()?.value() { … }
```

This mirrors the pattern already present in `vote_update` and makes correctness independent of whether `clean_tee_status` has executed.

```rust
fn stale_measurement_vote_counts_in_race_window() {
    // Setup: 5 participants, threshold 3; P4 will be removed, P5 added.
    let mut all_5 = gen_participants(5);
    let new_4_plus_p5 = /* subset {P0,P1,P2,P3} + new P5 */;

    let mut tee_state = TeeState::default();

    // P4 votes during Resharing (old set).
    set_signer(P4);
    let auth_old = AuthenticatedParticipantId::new(&all_5).unwrap();
    tee_state.vote_measurement(MeasurementVoteAction::Add(M_malicious), &auth_old);

    // Resharing completes → new set active; clean_tee_status NOT yet run.

    // P5 (new) votes.
    set_signer(P5);
    let auth_new = AuthenticatedParticipantId::new(&new_4_plus_p5).unwrap();
    let v1 = tee_state.vote_measurement(MeasurementVoteAction::Add(M_malicious), &auth_new);
    // v1 == 2 (P4 stale + P5 fresh), not 1.

    // P3 (new) votes → threshold 3 reached with only 2 Byzantine new participants.
    set_signer(P3);
    let auth_p3 = AuthenticatedParticipantId::new(&new_4_plus_p5).unwrap();
    let v2 = tee_state.vote_measurement(MeasurementVoteAction::Add(M_malicious), &auth_p3);
    assert_eq!(v2, 3); // threshold reached — M_malicious would be added

    // After clean_tee_status runs, stale vote is gone.
    tee_state.clean_non_participant_votes(&new_4_plus_p5);
    // Now only 2 votes remain — threshold not met.
}
```

The test asserts that `count_votes` returns 3 (threshold met) before cleanup, confirming the invariant break.
