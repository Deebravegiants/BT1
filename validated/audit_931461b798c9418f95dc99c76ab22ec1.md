### Title
Stale `ethValueInWithdrawal` in `LRTConverter` Understates Protocol TVL, Enabling Yield Theft from rsETH Holders - (File: contracts/LRTConverter.sol)

---

### Summary

`LRTConverter.ethValueInWithdrawal` records the ETH-denominated value of LST assets at the moment they are transferred into the converter, using the oracle price at that instant. This stored value is never refreshed. Because rebasing LSTs (e.g., stETH) continuously accrue yield, the stored value becomes stale and understates the actual ETH value of assets sitting in the converter. `LRTOracle._getTotalEthInProtocol()` consumes this stale figure directly, causing `rsETHPrice` to be understated for the entire duration assets remain in the converter. Any depositor who mints rsETH during this window receives more rsETH than they are entitled to, diluting existing holders and stealing their accrued yield.

---

### Finding Description

When an operator calls `LRTConverter.transferAssetFromDepositPool()`, the ETH value of the transferred LST is computed once and added to `ethValueInWithdrawal`:

```solidity
// LRTConverter.sol line 140
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
``` [1](#0-0) 

This value is never updated again until the ETH is eventually claimed and sent back to the deposit pool via `_sendEthToDepositPool`. During the entire unstaking window (which for Lido can span days to weeks), `ethValueInWithdrawal` remains frozen at the price captured at transfer time.

`LRTDepositPool.getETHDistributionData()` feeds this stale figure directly into the protocol's total asset accounting:

```solidity
// LRTDepositPool.sol line 499
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
``` [2](#0-1) 

Critically, `getAssetDistributionData()` for any LST explicitly zeroes out the converter contribution for that asset:

```solidity
// LRTDepositPool.sol line 460
assetLyingInConverter = 0; // assets in converter are accounted in their eth value => getETHDistributionData()
``` [3](#0-2) 

This means the LST tokens in the converter are **exclusively** accounted for through `ethValueInWithdrawal`. There is no fallback that reads the live token balance or applies the current oracle price.

`LRTOracle._getTotalEthInProtocol()` then sums all asset values — including the stale converter figure — to compute `rsETHPrice`:

```solidity
// LRTOracle.sol lines 341-343
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [4](#0-3) 

```solidity
// LRTOracle.sol line 250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [5](#0-4) 

Because `totalETHInProtocol` is understated by the yield that accrued on the LSTs while in the converter, `rsETHPrice` is understated by the same amount.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When `rsETHPrice` is understated, `getRsETHAmountToMint()` in `LRTDepositPool` mints more rsETH per unit of deposited asset than is warranted:

```solidity
// LRTDepositPool.sol line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [6](#0-5) 

A lower `rsETHPrice` denominator means more rsETH is minted per deposit. This dilutes all existing rsETH holders. The yield that stETH accrued while sitting in the converter — which rightfully belongs to existing rsETH holders — is instead captured by new depositors who receive inflated rsETH amounts. The magnitude scales with: (a) the amount of LSTs in the converter, (b) the yield rate of the LST, and (c) the duration of the unstaking window. For Lido stETH, the unstaking queue can last weeks, and stETH accrues ~4% APR, making the mis-accounting material.

---

### Likelihood Explanation

**Medium-High.** The converter is a core operational component used regularly to convert LSTs to ETH for the withdrawal queue. Every time `transferAssetFromDepositPool` is called and assets sit in the converter for any non-trivial duration, the vulnerability is active. No special conditions or attacker setup are required — any user calling `depositETH()` or `depositAsset()` during the window exploits the underpriced rsETH passively.

---

### Recommendation

Instead of storing a fixed ETH value at transfer time, `ethValueInWithdrawal` should be derived dynamically from the live token balance and current oracle price. Replace the stored scalar with a mapping of `(asset → amount)` and compute the ETH value on-the-fly:

```solidity
// Instead of:
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;

// Store the raw token amount:
converterAssetBalance[_asset] += _amount;

// And compute ETH value dynamically in the view function:
function ethValueInWithdrawal() external view returns (uint256 total) {
    for each asset in converterAssets:
        total += converterAssetBalance[asset] * lrtOracle.getAssetPrice(asset) / 1e18;
}
```

This ensures the converter's contribution to `totalETHInProtocol` always reflects the current market value of the assets held, including any yield accrued since transfer.

---

### Proof of Concept

1. Protocol holds 1000 stETH in `LRTConverter` after `transferAssetFromDepositPool` is called. At transfer time, stETH price = 1.05 ETH, so `ethValueInWithdrawal = 1050 ETH`.
2. 30 days pass. stETH rebases; price is now 1.054 ETH. The actual ETH value of the 1000 stETH is 1054 ETH. But `ethValueInWithdrawal` is still 1050 ETH — 4 ETH understated.
3. `updateRSETHPrice()` is called. `_getTotalEthInProtocol()` returns a value 4 ETH lower than reality. `rsETHPrice` is understated proportionally.
4. An attacker calls `depositETH(1 ETH, ...)`. `getRsETHAmountToMint` divides by the understated `rsETHPrice`, minting more rsETH than the depositor is entitled to.
5. The attacker now holds rsETH representing more than 1 ETH of protocol value, at the expense of existing holders whose share was diluted by the unaccounted 4 ETH of stETH yield. [7](#0-6) [8](#0-7)
