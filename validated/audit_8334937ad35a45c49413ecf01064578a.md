### Title
ETH Deposit Limit Check Missing `amount` Operand Allows Deposits to Exceed Protocol Cap - (File: contracts/LRTDepositPool.sol)

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison for ETH vs. ERC20 assets. The ETH branch omits the incoming `amount` from the limit check, making the guard weaker than intended and allowing any depositor to push the ETH TVL past the configured cap in a single call.

### Finding Description

The internal helper `_checkIfDepositAmountExceedesCurrentLimit` is called by `_beforeDeposit` for every deposit. For ERC20 assets the check correctly includes the new deposit amount:

```solidity
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

But for ETH the `amount` term is absent:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount missing
}
``` [1](#0-0) 

Because the check only asks "has the limit already been exceeded?" rather than "would this deposit exceed the limit?", a depositor can always succeed with one additional deposit when `totalAssetDeposits == depositLimit`, pushing the total to `depositLimit + amount`. [2](#0-1) 

The entry path is the public `depositETH()` function, callable by any user with no role requirement. [3](#0-2) 

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The deposit cap for ETH is a risk-management parameter (e.g., matching EigenLayer strategy capacity or concentration limits). Bypassing it causes the protocol to accept more ETH than intended, minting excess rsETH. Excess ETH that cannot be deployed to EigenLayer strategies sits idle, diluting yield for all rsETH holders. No funds are directly stolen, but the protocol operates outside its intended risk envelope and rsETH holders receive lower-than-promised returns.

### Likelihood Explanation

High. Any unprivileged ETH depositor can trigger this in a single transaction whenever `totalAssetDeposits` is at or near the configured limit. No special setup, flash loan, or oracle manipulation is required.

### Recommendation

Add `amount` to the ETH branch, mirroring the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
-   if (asset == LRTConstants.ETH_TOKEN) {
-       return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
-   }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept

1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 100 ether`.
2. Users deposit until `getTotalAssetDeposits(ETH_TOKEN) == 100 ether`.
3. Attacker calls `depositETH{value: 50 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `100 ether > 100 ether` → `false` → no revert.
5. 50 ETH is accepted; `getTotalAssetDeposits(ETH_TOKEN)` becomes 150 ETH, 50% above the cap.
6. Excess rsETH is minted; ETH that cannot be deployed to EigenLayer sits idle, reducing yield for all holders. [1](#0-0)
