### Title
Missing Share Verification in Non-PV CKD Coordinator Allows Single Byzantine Participant to Silently Corrupt Derived Key Output — (`crates/threshold-signatures/src/confidential_key_derivation/protocol.rs`)

---

### Summary

The non-public-verifiable CKD protocol (`protocol.rs`) aggregates participant share contributions with no cryptographic verification. A single Byzantine participant can send arbitrary `CKDOutput` values — including identity-point pairs — and the coordinator will silently incorporate them, producing a derived key that differs from the correct `msk · H(pk ∥ app_id)`. The client has no mechanism to detect the corruption. The public-verifiable variant (`protocol_pv.rs`) explicitly fixes this with a pairing check, confirming the non-PV path is a known gap that was not closed.

---

### Finding Description

**Root cause — `protocol.rs` `do_ckd_coordinator`:**

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

There is no zero-knowledge proof, no pairing check, and no subgroup membership check on the received `(big_y, big_c)` pairs. Any value is accepted and added.

**Contrast with `protocol_pv.rs`**, which performs an explicit pairing check after aggregation:

```rust
if !aggregated_output_check(&ckd_output, app_pk, &key_pair.public_key, &hash_point) {
    return Err(ProtocolError::AssertionFailed(
        "CKD output failed to verify".to_string(),
    ));
}
``` [2](#0-1) 

The pairing check verifies `e(C, g₂) = e(Y, app_pk₂) · e(H(pk ∥ app_id), msk·g₂)`, which is unsatisfiable if any participant contributed a wrong share.

**Production dispatch** in `sign.rs` routes both variants:

```rust
dtos::CKDAppPublicKey::AppPublicKey(pk) => {
    let protocol = ckd(...)?;          // protocol.rs — NO verification
    run_protocol("ckd", channel, protocol).await?
}
dtos::CKDAppPublicKey::AppPublicKeyPV(pv) => {
    let protocol = ckd_pv(...)?;       // protocol_pv.rs — pairing check
    run_protocol("ckd_pv", channel, protocol).await?
}
``` [3](#0-2) 

**Contract-level `respond_ckd`** mirrors this gap — the `AppPublicKey` arm is an empty no-op:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(...) { env::panic_str(...); }
    }
}
``` [4](#0-3) 

So for the `AppPublicKey` variant, neither the node-level protocol nor the contract performs any integrity check on the aggregated output.

---

### Impact Explanation

The `unmask` function computes `C − a·Y` to recover `msk · H(pk ∥ app_id)`: [5](#0-4) 

If a Byzantine participant injects `(big_y = 0, big_c = 0)` (identity), the aggregated `C` and `Y` are each missing that participant's Lagrange-weighted contribution. The result of `unmask` is a group element that is not `msk · H(pk ∥ app_id)`. The client receives this silently wrong value with no error, no rejection, and no way to distinguish it from a correct derivation.

A more targeted attacker can inject `(big_y = −Σ_honest Y_i, big_c = T)` for an arbitrary target `T`, steering the output to any desired point — provided they can observe the honest participants' contributions before the coordinator finalizes (which is possible if the attacker controls the coordinator role, since the coordinator aggregates all inputs before producing output).

---

### Likelihood Explanation

The `AppPublicKey` (non-PV) path is a live production code path reachable by any client that submits a CKD request with a plain `AppPublicKey`. A single Byzantine MPC node — one that has been compromised or is acting maliciously — participates in every CKD session it is selected for. Because the CKD protocol selects exactly `threshold` participants and requires all of them to contribute, a single Byzantine node among the selected set is sufficient to corrupt the output. No threshold collusion is required. [6](#0-5) 

---

### Recommendation

Apply the same `aggregated_output_check` pairing verification that `protocol_pv.rs` already uses to `protocol.rs` after aggregation. Alternatively, deprecate the non-PV `AppPublicKey` path and require all callers to use `AppPublicKeyPV`, which already has the fix. [7](#0-6)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`
fn aggregated_output_check(
    output: &CKDOutput,
    app_pk: &PublicVerificationKey,
    public_key: &VerifyingKey,
    hash_point: &ElementG1,
) -> bool {
    if !check_valid_point_g1(output.big_c.into()) || !check_valid_point_g1(output.big_y.into()) {
        return false;
    }
    multi_miller_loop(&[
        (output.big_c, -ElementG2::generator()),
        (output.big_y, app_pk.pk2),
        (*hash_point, public_key.to_element()),
    ])
}
