### Title
Zero-Royalty NFT Offer Settlement via Integer Division Truncation in `compute_royalty_amount` - (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary

The `compute_royalty_amount` function uses integer floor division that silently truncates to zero for small offered amounts or low royalty percentages. When this occurs, `make_nft1_offer` skips the royalty spend entirely and passes an empty `trade_prices_list` to the NFT ownership layer puzzle, bypassing on-chain royalty enforcement and allowing NFT transfers without paying the creator-mandated royalty.

### Finding Description

`compute_royalty_amount` in `chia/wallet/nft_wallet/nft_wallet.py` computes the royalty using two sequential integer floor divisions:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

`MAX_ROYALTY_BASIS_POINTS` is 10000. For any combination where `abs(offered_amount) // royalty_split * percentage < 10000`, the result is 0. The only guard present checks `royalty >= abs(offered_amount)`, which is never triggered when `royalty == 0`. So a zero royalty is silently returned. [2](#0-1) 

In `make_nft1_offer`, this zero-royalty amount is used in two critical places:

**1. Royalty spend is skipped when total royalty is zero:**

```python
if sum(p.amount for _, p in payments) == 0:
    continue
``` [3](#0-2) 

**2. The `trade_prices_list` passed to the NFT ownership layer puzzle is filtered to exclude any price where the royalty would be zero:**

```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
``` [4](#0-3) 

The `trade_prices_list` is the mechanism by which the NFT ownership layer CLVM puzzle enforces royalty payments on transfer. When this list is empty, the puzzle does not generate any royalty payment conditions, and the NFT is transferred with zero royalty paid on-chain.

The royalty payment construction in `make_nft1_offer` confirms the full path: [5](#0-4) 

The static `royalty_calculation` method used by the `nft_calculate_royalties` RPC endpoint has the identical truncation with no guard at all: [6](#0-5) 

### Impact Explanation

An unprivileged attacker (offer taker) can craft an offer requesting an NFT while offering a fungible amount small enough that `compute_royalty_amount` truncates to zero. The resulting spend bundle:

- Contains no royalty payment coin
- Passes an empty `trade_prices_list` to the NFT ownership layer puzzle
- Satisfies all CLVM validation (no royalty conditions are asserted)
- Transfers the NFT to the attacker with zero mojos paid to the royalty recipient

The royalty recipient — typically the NFT creator — receives nothing. This is an unauthorized accounting change affecting NFT offer settlement: the creator-mandated royalty embedded in the singleton puzzle is bypassed entirely without any consensus-level rejection.

### Likelihood Explanation

The truncation condition is reachable with realistic parameters. Any offer where `abs(offered_amount) * royalty_percentage < 10000` triggers it. Examples:

- 9999 mojos offered, 1 basis point royalty: `9999 * 1 // 10000 = 0`
- 99 mojos offered, 100 basis point (1%) royalty: `99 * 100 // 10000 = 0`
- 1999 mojos offered, 5 basis point royalty: `1999 * 5 // 10000 = 0`

For CAT-denominated offers (where 1 CAT = 1000 mojos), the threshold is even easier to reach. The NFT owner must accept the offer, but the wallet UI does not warn that royalties are zero, and the `nft_calculate_royalties` RPC (which uses the same truncating `royalty_calculation`) would also display 0, giving the owner no signal that royalty enforcement has been bypassed.

### Recommendation

1. In `compute_royalty_amount`, add an explicit check: if `percentage > 0` and the computed `royalty == 0`, raise a `ValueError` (analogous to the CGDA fix asserting `swapAmountIn > 0`).
2. In `make_nft1_offer`, before constructing the offer, validate that for any NFT with a non-zero royalty percentage, the computed royalty amount is also non-zero; reject the offer construction otherwise.
3. In `royalty_calculation` (used by the RPC), apply the same guard so the display layer accurately reflects that the offer cannot be settled with royalties.

### Proof of Concept

**Setup:** NFT minted with `royalty_percentage = 1` (1 basis point, 0.01%). Attacker constructs an offer requesting this NFT and offering `9999` mojos of XCH.

**Execution path:**

1. `make_nft1_offer` is called with `fungible_asset_dict = {None: 9999}` (offering 9999 mojos) and `required_royalty_info = [(launcher_id, address, 1)]`.
2. `compute_royalty_amount(amount=-9999, royalty_split=1, percentage=1)` computes `9999 // 1 * 1 // 10000 = 0`. No `ValueError` is raised because `0 < 9999`.
3. `payment_list = [(launcher_id, CreateCoin(address, 0, [address]))]`; `royalty_payments[None] = payment_list`.
4. At line 911: `sum(p.amount for _, p in payments) == 0` → `True` → `continue`. No royalty coin spend is created.
5. `trade_prices` contains `(uint64(9999 // 1), OFFER_MOD_HASH)` = `(9999, ...)`. The filter at line 896: `9999 * 1 // 10000 = 0 → False` → price excluded. `trade_prices_list = []`.
6. The NFT ownership layer puzzle is solved with an empty `trade_prices_list`. No royalty payment conditions are asserted. The spend bundle is valid and accepted by the network.
7. The NFT is transferred to the attacker. The royalty recipient receives 0 mojos. [1](#0-0) [5](#0-4) [7](#0-6)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
