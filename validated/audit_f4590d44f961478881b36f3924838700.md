### Title
Reward Accumulation Permanently Frozen via Integer Truncation in `rewardPerToken()` When `dt` Is Kept Small - (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool.rewardPerToken()` computes the per-token reward increment as `rewardRate * dt * DECIMAL_PRECISION / totalKernelStaked`. Because `updatedAt` is overwritten to `lastTimeRewardApplicable()` on every `updateReward` invocation, an attacker who calls `stake(1)` every block keeps `dt` equal to one block interval (~12 s). When `rewardRate * 12 * 1e18 < totalKernelStaked`, Solidity integer division truncates the increment to zero, `rewardPerTokenStored` never advances, and all stakers permanently earn zero rewards for the entire distribution window.

---

### Finding Description

`rewardPerToken()` accumulates the global reward index:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L408-L413
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
``` [1](#0-0) 

The `updateReward` modifier, applied to every state-changing user function (`stake`, `stakeFor`, `initiateWithdrawal`, `getReward`), snapshots `updatedAt` to the current applicable timestamp before executing the function body:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L232-L242
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();   // ← resets the clock every call
    ...
}
``` [2](#0-1) 

`rewardRate` is set by integer division in `notifyRewardAmount`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L579-L584
if (block.timestamp >= finishAt) {
    rewardRate = receivedAmount / duration;
} else {
    uint256 remaining = (finishAt - block.timestamp) * rewardRate;
    rewardRate = (receivedAmount + remaining) / duration;
}
``` [3](#0-2) 

The truncation condition is:

```
rewardRate * dt * 1e18 < totalKernelStaked
```

where `dt = lastTimeRewardApplicable() - updatedAt`. If an attacker calls `stake(1)` every Ethereum block, `dt` is pinned to ~12 seconds. The condition becomes:

```
rewardRate * 12 * 1e18 < totalKernelStaked
```

**Concrete example with a 6-decimal reward token (e.g., USDC):**

| Parameter | Value |
|---|---|
| `totalKernelStaked` | 1,000,000 KERNEL = 1e24 wei |
| `duration` | 7 days = 604,800 s |
| `receivedAmount` | 40,000 USDC = 4e10 wei |
| `rewardRate` | 4e10 / 604800 ≈ 66,137 |
| Truncation check | 66,137 × 12 × 1e18 = 7.94e23 < 1e24 ✓ |

Every call to `stake(1)` sets `updatedAt = block.timestamp`, so the 12-second window is never allowed to accumulate. `rewardPerTokenStored` stays at zero for the entire reward period. The 40,000 USDC deposited by the admin remains locked in the contract with no recovery path.

---

### Impact Explanation

`rewardPerTokenStored` never increases → `earned()` returns zero for every staker → `getReward()` transfers nothing. The reward tokens deposited via `notifyRewardAmount` are permanently frozen in the contract because there is no `sweep` or token-recovery function. The next call to `notifyRewardAmount` does not account for the stranded balance (it uses `rewardRate`, not the actual token balance), so the frozen tokens are never redistributed.

**Impact: Medium — Permanent freezing of unclaimed yield.**

---

### Likelihood Explanation

The attack is reachable by any unprivileged address via the public `stake(uint256)` function with `_amount = 1`. No special role is required. The gas cost of calling `stake(1)` every 12 seconds is the only economic deterrent; a validator/block proposer pays no gas. The truncation condition is realistic whenever the reward token has low decimals (USDC, USDT) and the reward amount is moderate relative to a large staked supply — a common operational configuration for a staking pool.

---

### Recommendation

1. **Accumulate remainder**: Track the fractional reward dust that is truncated each call and add it back to the next period's `rewardRate` calculation, similar to how Synthetix handles dust in `notifyRewardAmount`.
2. **Minimum `dt` guard**: Skip the `rewardPerTokenStored` update if `dt` is below a threshold (e.g., 1 hour), preventing sub-threshold calls from resetting `updatedAt` without contributing meaningful reward.
3. **Higher precision**: Scale `rewardRate` by `DECIMAL_PRECISION` at storage time (i.e., store `rewardRate * 1e18`) and divide only at read time, eliminating the per-second truncation.
4. **Add a token recovery function**: Allow the admin to recover stranded reward tokens so that any frozen balance can be redistributed.

---

### Proof of Concept

**Setup:**
- `duration = 604800` (7 days)
- `rewardToken` = USDC (6 decimals)
- Admin calls `notifyRewardAmount(40_000e6)` → `rewardRate = 40_000e6 / 604800 = 66137`
- `totalKernelStaked = 1_000_000e18` (1M KERNEL)

**Attack loop (every Ethereum block, ~12 s):**
```
attacker.stake(1)   // triggers updateReward
```

**Each iteration:**
```
dt = block.timestamp - updatedAt = 12
increment = 66137 * 12 * 1e18 / 1e24
           = 793644 * 1e18 / 1e24
           = 793644 / 1e6
           = 0  (integer truncation)
rewardPerTokenStored += 0  // no change
updatedAt = block.timestamp  // clock reset
```

After 604,800 seconds (the full reward window), `rewardPerTokenStored` remains 0. Every staker calls `getReward()` and receives 0. The 40,000 USDC is permanently locked in `KernelDepositPool`. [4](#0-3) [1](#0-0) [5](#0-4)
