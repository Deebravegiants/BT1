Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice` Performs No Staleness Validation on Chainlink Price Data - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but discards all staleness-related return values (`updatedAt`, `roundId`, `answeredInRound`), silently accepting any stale price. When a Chainlink feed goes stale with an inflated price, any depositor calling `LRTDepositPool.depositAsset` receives more rsETH than their deposit is worth, diluting the proportional ETH claim of all existing rsETH holders.

## Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice` captures only the `price` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

There is no `block.timestamp - updatedAt <= heartbeat` check, no `require(price > 0)`, and no `answeredInRound >= roundId` guard. The raw price is returned unconditionally.

`LRTOracle.getAssetPrice` (line 157) delegates directly to this oracle:

```solidity
return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
```

This price feeds two critical paths:

1. **Deposit minting** — `LRTDepositPool.getRsETHAmountToMint` (line 520) computes:
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```
   A stale inflated asset price inflates `rsethAmountToMint` directly.

2. **rsETH price update** — `LRTOracle._getTotalEthInProtocol` (lines 336–343) multiplies each asset balance by `getAssetPrice(asset)` to compute total protocol ETH, which then sets `rsETHPrice`. A stale inflated price here inflates `rsETHPrice`, which partially offsets the deposit minting inflation — but only after `updateRSETHPrice` is called. Between a stale price event and the next price update, depositors exploit the stale numerator while the denominator (`rsETHPrice`) still reflects the last valid price.

The `pricePercentageLimit` guard in `_updateRsETHPrice` only triggers on the `updateRSETHPrice` call path, not on `depositAsset`. It provides no protection during the deposit flow.

`ChainlinkOracleForRSETHPoolCollateral` does attempt two checks (`answeredInRound < roundID` and `timestamp == 0`), but neither is a heartbeat check. The deprecated `answeredInRound < roundID` pattern is unreliable on modern Chainlink feeds, and `timestamp == 0` only catches incomplete rounds, not stale ones. This contract is not the oracle backing `LRTOracle.getAssetPrice` — `ChainlinkPriceOracle` is — so these checks are irrelevant to the primary attack surface.

## Impact Explanation

**High — Theft of unclaimed yield.**

When a Chainlink LST/ETH feed goes stale with an inflated last price, every depositor calling `depositAsset` during the stale window receives more rsETH than the fair value of their deposit. This over-minting dilutes the ETH-per-rsETH ratio for all existing holders, constituting theft of their proportional share of accrued protocol yield. The attacker requires no special role, no front-running, and no governance access — only a normal deposit transaction during the stale window.

## Likelihood Explanation

Chainlink LST/ETH feeds (e.g., stETH/ETH, cbETH/ETH) have heartbeat windows of 1–24 hours. During low-volatility periods, feeds routinely approach their full heartbeat duration without updating. Network congestion or a sequencer outage can extend staleness further. Any depositor transacting during such a window — without any special knowledge or privilege — triggers the issue. The condition is periodic and repeatable.

## Recommendation

Add a per-asset heartbeat mapping and enforce it in `ChainlinkPriceOracle.getAssetPrice`:

```solidity
mapping(address asset => uint256 heartbeat) public assetHeartbeat;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    require(price > 0, "Invalid price");
    require(block.timestamp - updatedAt <= assetHeartbeat[asset], "Stale price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Require `assetHeartbeat[asset]` to be set alongside `updatePriceFeedFor`. Replace the `answeredInRound < roundID` and `timestamp == 0` checks in `ChainlinkOracleForRSETHPoolCollateral` with a proper heartbeat check as well.

## Proof of Concept

1. Deploy a fork of mainnet. Advance `block.timestamp` by 2 hours without triggering a Chainlink stETH/ETH feed update (warp time, do not poke the feed).
2. Confirm `latestRoundData()` on the stETH/ETH feed returns `updatedAt` that is 2 hours old, with `price = 1.05e18` (last reported value, true market: `1.00e18`).
3. Call `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")` as an unprivileged EOA.
4. Observe `getRsETHAmountToMint` computes `100e18 * 1.05e18 / rsETHPrice`, minting ~5% more rsETH than fair value.
5. Verify existing rsETH holders' ETH-per-rsETH ratio has decreased by the dilution amount.
6. No admin action, no special role, no front-running required.

Foundry fork test outline:
```solidity
function testStaleOracleMinting() public {
    vm.warp(block.timestamp + 2 hours); // stale feed, no update
    uint256 rsethBefore = rseth.balanceOf(attacker);
    vm.prank(attacker);
    depositPool.depositAsset(stETH, 100e18, 0, "");
    uint256 rsethMinted = rseth.balanceOf(attacker) - rsethBefore;
    uint256 fairMint = (100e18 * 1e18) / lrtOracle.rsETHPrice(); // at true 1:1 price
    assertGt(rsethMinted, fairMint); // attacker received excess rsETH
}
