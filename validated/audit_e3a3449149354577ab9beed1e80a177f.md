### Title
`respond_ckd` Accepts Unvalidated CKD Output for `AppPublicKey` Variant — Byzantine Participant Below Threshold Can Corrupt Confidential Key Derivation Responses - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs cryptographic output validation **only** for the `AppPublicKeyPV` variant. For the legacy `AppPublicKey` variant the validation branch is an empty no-op. A single Byzantine participant (below signing threshold) can race the honest MPC response, deliver arbitrary garbage `big_y`/`big_c` bytes, permanently resolve the pending CKD yield with invalid output, and remove the request from contract state — making a retry impossible.

---

### Finding Description

In `respond_ckd`, after the caller-authentication checks, the response content is validated conditionally:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` performs a BLS12-381 pairing check — `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` — that cryptographically binds the output to the master key and the user's ephemeral key. [2](#0-1) 

For `AppPublicKey` (the legacy, "privately verifiable" path), the arm is empty. Any `big_y` and `big_c` byte arrays — including all-zero, identity-point encodings, or random garbage — pass through unchecked. Immediately after, `resolve_yields_for` is called unconditionally:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [3](#0-2) 

This call (a) serialises the garbage response into the NEAR yield slot — delivering it to the waiting user — and (b) removes the entry from `pending_ckd_requests`, as confirmed by the existing test assertion `assert!(contract.get_pending_ckd_request(&ckd_request).is_none())`. [4](#0-3) 

The existing unit test for the `AppPublicKey` path passes `big_y: [1u8; 48]` and `big_c: [2u8; 48]` — byte strings that are not valid compressed BLS12-381 G1 points — and the contract accepts them without complaint, confirming the absence of any point-validity or pairing check on this path. [5](#0-4) 

The `respond_ckd` entry point is gated only by `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a current participant with a valid TEE attestation — a single Byzantine node below threshold satisfies this. [6](#0-5) 

---

### Impact Explanation

A single Byzantine participant (below signing threshold) can:

1. Observe a pending `CKDRequest` with the `AppPublicKey` variant on-chain.
2. Call `respond_ckd` with `CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }` (or any garbage).
3. The contract accepts the call, resolves the NEAR yield with the garbage payload, and removes the request from `pending_ckd_requests`.
4. The user's callback (`return_ck_and_clean_state_on_success`) fires and returns the garbage derived key to the user.
5. Honest MPC nodes' subsequent `respond_ckd` calls fail because the request no longer exists in state — no retry is possible.

The user receives an invalid confidential key derivation output. Any application secret encrypted to that garbage key is permanently unrecoverable. This breaks the production safety invariant that `respond_ckd` delivers a cryptographically correct CKD output, and permanently corrupts the request lifecycle for the affected user.

This maps to the allowed Medium impact: **"request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."**

---

### Likelihood Explanation

- Requires only **one** Byzantine participant with a valid TEE attestation — strictly below the signing threshold.
- The `AppPublicKey` (legacy) variant is still accepted by the contract ABI and is the default path for callers that do not supply `AppPublicKeyPV`.
- The attack is a simple, single NEAR transaction with no special tooling.
- The attacker can monitor the NEAR chain for pending CKD requests and race the honest response; NEAR's deterministic block ordering makes this straightforward.

---

### Recommendation

Add point-validity and pairing-based output validation for the `AppPublicKey` variant in `respond_ckd`, mirroring the `AppPublicKeyPV` path. At minimum:

1. Validate that `big_y` and `big_c` are valid, non-identity compressed BLS12-381 G1 points using `env::bls12381_p1_decompress`.
2. Consider deprecating the unverifiable `AppPublicKey` variant entirely and requiring all new requests to use `AppPublicKeyPV`, which provides on-chain verifiability.
3. If the legacy path must remain, document explicitly that it provides no on-chain output integrity guarantee and gate it behind a feature flag or config option.

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
