Audit Report

## Title
Uninitialized `minRsEthAmountToWithdraw` Enables Dust Withdrawal Queue Flooding, Causing Temporary Fund Freeze - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTWithdrawalManager.initiateWithdrawal()` enforces a minimum rsETH amount via `minRsEthAmountToWithdraw[asset]`, but this mapping is never set in `initialize()` and defaults to `0` for every asset. The guard therefore only blocks a zero-value call, allowing any unprivileged user to flood the withdrawal queue with dust entries at minimal cost. Because `_unlockWithdrawalRequests` processes the queue strictly in FIFO order, dust entries inserted ahead of legitimate requests must be drained first, temporarily freezing the withdrawals of honest users who have already burned their rsETH.

## Finding Description

**Root cause — uninitialized minimum:**

`minRsEthAmountToWithdraw` is declared as a per-asset mapping: [1](#0-0) 

`initialize()` sets only `withdrawalDelayBlocks` and `lrtConfig`; the mapping is never touched: [2](#0-1) 

**Broken guard in `initiateWithdrawal`:**

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [3](#0-2) 

When `minRsEthAmountToWithdraw[asset] == 0`, the second condition `rsETHUnstaked < 0` is always false for `uint256`. Any call with `rsETHUnstaked >= 1` passes, appends a `WithdrawalRequest`, and increments `nextUnusedNonce[asset]`. [4](#0-3) 

**FIFO queue forces dust traversal before legitimate requests:**

`_unlockWithdrawalRequests` iterates strictly from `nextLockedNonce` upward: [5](#0-4) 

The two early-exit conditions (`withdrawalDelayBlocks` check and `availableAssetAmount < payoutAmount`) do **not** protect against dust entries: once the delay has elapsed, dust entries with a ~1-wei `payoutAmount` will be processed continuously as long as any assets are available, consuming a storage write per entry. The operator controls `firstExcludedIndex` and can batch calls, but must still drain every dust entry before `nextLockedNonce` advances past them to reach legitimate requests.

**Enabling dust rsETH acquisition:**

`minAmountToDeposit` in `LRTDepositPool` is similarly uninitialized: [6](#0-5) [7](#0-6) 

The deposit guard `depositAmount == 0 || depositAmount < minAmountToDeposit` only blocks zero-value deposits: [8](#0-7) 

This lets an attacker acquire dust rsETH cheaply to fuel the queue-flooding attack.

**Admin setter exists but is never called at deployment:** [9](#0-8) 

The window is open from the first block after deployment.

## Impact Explanation

**Medium — Temporary freezing of funds.**

Legitimate users call `initiateWithdrawal`, which immediately transfers their rsETH to the contract. They cannot recover it until the operator calls `unlockQueue` and advances `nextLockedNonce` past their nonce. If N dust entries occupy nonces 0…N-1 and a legitimate request is at nonce N, the operator must process all N dust entries first. With N large enough (e.g., 50,000), the operator must issue many batched `unlockQueue` calls before any honest withdrawal can be unlocked or completed. During this period, honest users' rsETH is held by the contract with no recourse — a concrete temporary freeze of funds matching the allowed Medium impact.

Additionally, the operator bears unbounded cumulative gas cost to drain the dust queue, matching the allowed **Medium — Unbounded gas consumption** impact.

## Likelihood Explanation

**High.** The attack requires only:
1. A small amount of ETH (enough to mint ≥ 1 wei rsETH via `depositETH`) — no role or permission required.
2. Repeated calls to `initiateWithdrawal` with the dust rsETH amount.

No admin compromise, oracle manipulation, front-running, or governance capture is needed. The vulnerability is exploitable by any external account from the first block after deployment, and is repeatable at will until an admin calls `setMinRsEthAmountToWithdraw` with a non-zero value.

## Recommendation

1. **Initialize `minRsEthAmountToWithdraw` in `initialize()`** for each supported asset to a protocol-meaningful floor (e.g., equivalent to 0.001 ETH in rsETH terms), mirroring how `withdrawalDelayBlocks` is set.
2. **Initialize `minAmountToDeposit` in `LRTDepositPool.initialize()`** to a non-zero value to prevent dust rsETH minting that fuels the attack.
3. **Enforce a non-zero minimum** in `setMinRsEthAmountToWithdraw` (revert if `_minAmount == 0`), consistent with the pattern used in `KernelVaultETH.setMinDeposit`.
4. **Add a per-user cap** on pending withdrawal requests to bound queue growth per address.

## Proof of Concept

```solidity
// Preconditions: minRsEthAmountToWithdraw[ETH_TOKEN] == 0 (default)
//               minAmountToDeposit == 0 (default)

for (uint i = 0; i < 50_000; i++) {
    // Acquire dust rsETH (minAmountToDeposit == 0 allows tiny deposits)
    lrtDepositPool.depositETH{value: smallAmount}(0, "");

    // Queue a dust withdrawal (minRsEthAmountToWithdraw == 0 allows 1-wei entries)
    rsETH.approve(address(lrtWithdrawalManager), 1);
    lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");
}
// nextUnusedNonce[ETH_TOKEN] == 50_000
// Operator must drain all 50_000 dust entries via batched unlockQueue calls
// before nextLockedNonce reaches any legitimate request nonce
// → honest users' rsETH is frozen in the contract until the backlog is cleared
```

**Foundry test plan:** Deploy `LRTWithdrawalManager` and `LRTDepositPool` on a mainnet fork. Confirm `minRsEthAmountToWithdraw[ETH_TOKEN] == 0`. Submit a legitimate withdrawal at nonce N. Then submit N dust withdrawals at nonces 0…N-1. Call `unlockQueue` with `firstExcludedIndex = N+1` and measure gas; confirm the legitimate request at nonce N cannot be completed until all dust entries are processed. Assert `nextLockedNonce[ETH_TOKEN] < N` after a single bounded `unlockQueue` call that stops short of N.
