### Title
Unvalidated Peer-Supplied Solution in `fetch_coin_spend` Enables Forged Singleton Lineage in PoolWallet — (`chia/wallet/util/wallet_sync_utils.py`)

---

### Summary

`fetch_coin_spend` validates the puzzle hash of a peer-returned `RespondPuzzleSolution` but performs **zero validation of the solution**. An unprivileged peer can return the correct puzzle with a crafted solution that, when executed by `compute_additions`, produces a different `CREATE_COIN` output than the real on-chain spend. This forged `CoinSpend` is stored in `WalletPoolStore` and used to derive the next singleton coin, permanently corrupting the pool wallet's tracked lineage.

---

### Finding Description

**Step 1 — Validation gap in `fetch_coin_spend`** [1](#0-0) 

The function checks:
- `puzzle.get_tree_hash() == coin.puzzle_hash` — puzzle identity is verified
- `coin_name == coin_id` — coin identity is verified
- **The `solution` field is accepted verbatim with no validation whatsoever.**

The returned `CoinSpend` is constructed with `make_spend(coin, response.puzzle, response.solution)` — the attacker's solution is embedded directly.

**Step 2 — `compute_additions` executes the CLVM with the attacker's solution** [2](#0-1) 

`run_with_cost(cs.puzzle_reveal, max_cost, cs.solution)` actually **runs the CLVM** with the attacker-supplied solution. Critically, all `AGG_SIG_*` conditions (lines 35–46) are **skipped** — they are only counted for cost, never cryptographically verified. This means the pool waiting room puzzle's `AGG_SIG_ME` requirement does not prevent the CLVM from executing successfully with a forged solution.

**Step 3 — Forged coin derived from CLVM output** [3](#0-2) 

`get_most_recent_singleton_coin_from_coin_spend` calls `compute_additions` and returns the first odd-amount coin. With a forged solution that encodes a different target pool state, the pool waiting room puzzle outputs a `CREATE_COIN` with a different puzzle hash (the hash of the forged destination inner puzzle). The returned `Coin` has `parent_coin_info = real_coin.name()` but a forged `puzzle_hash` and thus a different `name()`.

**Step 4 — Forged spend stored in persistent DB** [4](#0-3) 

The loop:
1. Calls `fetch_coin_spend_for_coin_state` → gets forged spend
2. Calls `apply_state_transition(cs, ...)` → passes the `tip_coin.name() == new_state.coin.name()` check (both are the real on-chain coin) and calls `pool_store.add_spend`
3. Calls `get_most_recent_singleton_coin_from_coin_spend(cs)` → returns the forged next coin
4. Subscribes to the forged coin via `add_interested_coin_ids` [5](#0-4) 

`apply_state_transition` only checks that `tip_coin.name() == new_state.coin.name()`. Since `new_state.coin` is the real on-chain coin (from `curr_coin_state`), this passes. The forged spend is then written to `WalletPoolStore`.

**Step 5 — `WalletPoolStore.add_spend` does not reject the forged spend** [6](#0-5) 

The only lineage check is `spend.coin.parent_coin_info != prev.coin.name()`. This checks the **coin's** parent, not the solution's lineage proof. Since `spend.coin` is the real on-chain singleton coin, its `parent_coin_info` correctly matches the previous spend's coin name. The forged spend is persisted to the SQLite database.

### Impact Explanation

After the attack:
- `WalletPoolStore` contains a forged spend with a fabricated next singleton coin
- The wallet subscribes to this forged coin (which does not exist on-chain)
- The malicious peer returns a `CoinState` for the forged coin with `spent_height=None`, ending the loop
- The wallet's tip is now the forged coin; all future pool operations (`claim_pool_rewards`, `generate_travel_transactions`, `new_peak`) use this forged tip
- Pool reward claims fail because `create_absorb_spend` uses the wrong singleton lineage
- Pool state transitions fail because the wallet tries to spend a non-existent coin
- The forged spend is in the persistent SQLite DB; reconnecting to an honest peer does not fix it without manual DB intervention

This is a permanent, targeted denial of pool wallet functionality matching the **High** impact category: long-lived inability to process pool actions under normal network assumptions.

---

### Likelihood Explanation

The attacker only needs to operate a reachable full node peer. The wallet connects to untrusted peers during sync. The attacker must know the real singleton puzzle (public, derivable from the launcher ID) and the real lineage proof (public, from the parent spend). Crafting a syntactically valid pool waiting room solution with a different target state is straightforward — the pool waiting room inner solution format is `[spend_type, key_value_list, destination_inner_puzzle_hash]`, all of which the attacker controls. No private keys are required because `compute_additions` skips all AGG_SIG validation.

---

### Recommendation

After `fetch_coin_spend` returns, validate the computed additions against the blockchain before trusting them for lineage tracking:

1. **Cross-check additions against block data**: After computing `new_singleton_coin = get_most_recent_singleton_coin_from_coin_spend(cs)`, call `request_and_validate_additions` to verify that `new_singleton_coin` is actually present in the block at `curr_coin_state.spent_height`. This uses the existing Merkle proof infrastructure already present in `wallet_sync_utils.py`.

2. **Alternatively**, validate the solution against the removals root: use `request_and_validate_removals` to confirm the coin was actually spent in the claimed block, and then verify the computed child coin appears in the additions Merkle set.

The puzzle hash check in `fetch_coin_spend` is necessary but not sufficient — the solution determines what coins are created, and it must be anchored to on-chain block data.

async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:
    solution_response = await peer.call_api(

    return make_spend(
        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )
