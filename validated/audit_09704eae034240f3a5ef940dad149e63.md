### Title
Untracked EigenPod ETH Balance Allows NDC Removal With Residual Funds, Understating rsETH Price - (File: contracts/LRTDepositPool.sol)

### Summary
`_checkResidueEthBalance` in `LRTDepositPool.sol` does not verify the native ETH balance sitting directly inside the `eigenPod` contract. When beacon-chain withdrawals land in the eigenPod but no checkpoint has been run yet, that ETH is invisible to all three checks performed before removal. An admin can therefore remove a `NodeDelegator` that still holds real protocol ETH, causing the TVL used to price rsETH to be permanently understated until the NDC is re-added.

### Finding Description
`_removeNodeDelegatorContractFromQueue` calls `_checkResidueEthBalance` before removing an NDC from the queue. [1](#0-0) 

The three conditions checked are:

1. `INodeDelegator(nodeDelegatorAddress).getEffectivePodShares() != 0`
2. `address(nodeDelegatorAddress).balance > maxNegligibleAmount`
3. `INodeDelegator(nodeDelegatorAddress).getAssetUnstaking(LRTConstants.ETH_TOKEN) > 0`

`getEffectivePodShares()` returns `stakedButUnverifiedNativeETH + withdrawableShare`, where `withdrawableShare` is the EigenLayer DelegationManager's view of the NDC's beacon-chain ETH strategy shares. [2](#0-1) 

EigenLayer only awards shares after a checkpoint is finalized. ETH that has arrived at the eigenPod from beacon-chain withdrawals (partial or full) but has **not yet been checkpointed** is held in `address(eigenPod).balance`. It is:

- **Not** in `stakedButUnverifiedNativeETH` (that counter is decremented when `verifyWithdrawalCredentials` is called, not when ETH arrives at the pod).
- **Not** in `withdrawableShare` (no checkpoint â†’ no EigenLayer shares).
- **Not** in `address(nodeDelegatorAddress).balance` (ETH is in the eigenPod, not the NDC).
- **Not** in `getAssetUnstaking()` (no queued withdrawal).

All three guards therefore return zero, and the NDC is removed from `nodeDelegatorQueue` while the eigenPod still holds real ETH.

After removal, `getETHDistributionData()` no longer iterates over this NDC: [3](#0-2) 

The orphaned eigenPod ETH is excluded from `_getTotalEthInProtocol()` in `LRTOracle`, which drives the rsETH price: [4](#0-3) 

### Impact Explanation
The rsETH price (`rsETHPrice`) is computed as `totalETHInProtocol / rsethSupply`. With the eigenPod ETH excluded from `totalETHInProtocol`, the price is understated. Every subsequent deposit mints more rsETH than warranted, diluting all existing rsETH holders. The ETH is not permanently destroyed â€” an admin can re-add the NDC, run a checkpoint, and remove it again â€” but until that remediation occurs the protocol misprices rsETH and existing holders suffer dilution. This maps to **Low: Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
The scenario is operationally realistic: validators exit on the beacon chain, their 32 ETH lands in the eigenPod, and the operator has not yet run `startCheckpoint`. If the admin then decides to retire the NDC (e.g., to rebalance operators), the removal succeeds silently. No malicious actor is required; a well-intentioned admin following normal operational procedures can trigger this.

### Recommendation
Add a check for the eigenPod's raw ETH balance inside `_checkResidueEthBalance`:

```solidity
IEigenPod pod = INodeDelegator(nodeDelegatorAddress).eigenPod();
if (address(pod) != address(0) && address(pod).balance > maxNegligibleAmount) {
    revert NodeDelegatorHasETH();
}
```

This ensures that any ETH awaiting a checkpoint in the eigenPod blocks NDC removal until it has been properly checkpointed and accounted for.

### Proof of Concept
1. NDC has one validator; `verifyWithdrawalCredentials` was called â†’ `stakedButUnverifiedNativeETH = 0`, EigenLayer holds 32 ETH of shares.
2. Validator exits on the beacon chain; 32 ETH is swept to `address(eigenPod)`.
3. No `startCheckpoint` / `verifyCheckpointProofs` has been called â†’ EigenLayer shares remain at 32 ETH (not yet decremented), but `withdrawableShare` from `getWithdrawableShares` reflects the current EigenLayer state.
4. Admin initiates `initiateUnstaking` for the 32 ETH shares â†’ `getEffectivePodShares()` returns 0 after the withdrawal is queued, and `getAssetUnstaking` returns 32 ETH.
5. Admin completes unstaking â†’ ETH moves to the unstaking vault; `getAssetUnstaking` returns 0.
6. Meanwhile, the beacon-chain sweep also deposited an additional partial-withdrawal reward (e.g., 0.5 ETH) directly to the eigenPod. This 0.5 ETH is in `address(eigenPod).balance`, not tracked anywhere.
7. Admin calls `removeNodeDelegatorContractFromQueue`. All three guards in `_checkResidueEthBalance` return zero. NDC is removed.
8. `getETHDistributionData()` no longer includes the 0.5 ETH. `_getTotalEthInProtocol()` is understated. `updateRSETHPrice()` sets a lower `rsETHPrice`, diluting existing holders. [5](#0-4) [6](#0-5)

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
