### Title
Unprotected `init` Function Allows Front-Running Contract Initialization With Attacker-Controlled Participants - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `init` function in `MpcContract` is decorated with `#[init]` but **not** with `#[private]`, meaning any external NEAR account can call it before the legitimate deployer. An attacker who front-runs this call can install an arbitrary participant set and threshold, gaining full control over the MPC network's signing capability.

---

### Finding Description

In NEAR SDK, `#[init]` only prevents re-initialization (panics if state already exists). It does **not** restrict who may call the function. The `#[private]` attribute is what enforces `predecessor_account_id == current_account_id`.

The `init` function at line 1924 carries only `#[init]`:

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

By contrast, `init_running` — the migration-path initializer — correctly carries **both** `#[private]` and `#[init]`:

```rust
#[private]
#[init]
#[handle_result]
pub fn init_running(
``` [2](#0-1) 

The `init` function accepts a fully attacker-controlled `ThresholdParameters` (participant set + threshold) and `InitConfig`, and immediately transitions the contract into `ProtocolContractState::Running` with those parameters: [3](#0-2) 

There is no check that `env::predecessor_account_id() == env::current_account_id()` or any other caller restriction. Any NEAR account can submit this transaction.

---

### Impact Explanation

**Critical.** An attacker who wins the race against the legitimate deployer initializes the contract with a participant set they fully control (e.g., a single participant at threshold 1, or a colluding majority). From that point:

- The attacker-controlled participants run DKG and generate the MPC key shares.
- They can issue threshold signatures for arbitrary foreign-chain transactions without any legitimate participant's involvement.
- The legitimate deployer cannot re-initialize (NEAR's `#[init]` guard prevents a second call), so the contract is permanently captured.

This satisfies: *"Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization."*

---

### Likelihood Explanation

**High.** Contract deployment and initialization are two separate NEAR transactions. The window between `DeployContract` and the `init` call is observable on-chain by any party monitoring the mempool or block stream. A bot watching for newly deployed contracts with a known code hash can immediately submit a competing `init` call. No privileged access, leaked key, or collusion is required — only a NEAR account with enough gas.

---

### Recommendation

Add `#[private]` to `init`, mirroring the protection already applied to `init_running`:

```rust
#[handle_result]
#[init]
#[private]          // ← add this
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

`#[private]` causes the NEAR runtime to panic if `predecessor_account_id != current_account_id`, so only the contract account itself (i.e., the deployer calling via a batch transaction or a self-call) can initialize it. [4](#0-3) 

---

### Proof of Concept

1. Attacker monitors NEAR for a `DeployContract` action targeting the MPC contract account.
2. Immediately after (or in the same block), attacker submits:
   ```
   near contract call-function as-transaction <mpc-contract-account> init \
     json-args '{"parameters": {"participants": {"next_id": 1, "participants": [["attacker.near", 0, {...}]]}, "threshold": 1}}' \
     prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
     sign-as attacker.near ...
   ```
3. If this lands before the legitimate `init` call, the contract is initialized with `attacker.near` as the sole participant at threshold 1.
4. Attacker's node calls `start_keygen` / `vote_pk` to complete DKG, producing a key share only they hold.
5. Attacker can now issue signatures for any foreign-chain transaction via `sign()` / `respond()` without any legitimate participant. [5](#0-4)
