### Title
Excess NEAR Deposit Permanently Frozen in `submit_participant_info` When Called by Existing Participant - (File: `crates/contract/src/lib.rs`)

### Summary

`submit_participant_info` is `#[payable]` and is used for both first-time participant joins (which require a storage deposit) and re-submissions by existing participants (which do not). When an existing participant re-submits, the deposit-handling block is skipped entirely. Any NEAR tokens attached to such a call are permanently frozen in the contract with no refund.

### Finding Description

`submit_participant_info` conditionally handles the attached deposit based on whether storage must be paid by the caller:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... deposit sufficiency check and excess refund
}
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — i.e., when an **existing participant** re-submits their attestation (updating TLS key or refreshing attestation data) — the entire deposit-handling block is bypassed. The function is marked `#[payable]`, so the NEAR runtime accepts any attached deposit without reverting. No refund is issued and no zero-deposit assertion is enforced, leaving any attached NEAR permanently frozen in the contract. [2](#0-1) 

The analogous pattern in the original report is: `takeOrders()` only checks `msg.value >= totalPrice` when `currency == address(0)`, but never asserts `msg.value == 0` for ERC20 orders. Here, `submit_participant_info` only checks/refunds the deposit when `attestation_storage_must_be_paid_by_caller == true`, but never asserts `attached_deposit == 0` when it is `false`.

### Impact Explanation

Any NEAR tokens attached to a re-submission call by an existing participant are permanently frozen inside the MPC contract. There is no withdrawal mechanism, no refund path, and no revert. This constitutes a permanent, irreversible loss of user funds.

This matches the allowed impact: **Medium — balance or contract execution-flow manipulation that breaks production safety/accounting invariants.**

### Likelihood Explanation

The operational documentation explicitly instructs participants to call `submit_participant_info` manually with `--deposit` for first-time joins:

> "first-time joiners must attach enough yoctoNEAR for storage; the node attaches `0`, so call `submit_participant_info` manually with `--deposit` once."



The same function is used for both first-time joins (deposit required) and re-submissions (deposit not required). A participant who has previously joined and needs to re-submit (e.g., after a TLS key rotation or attestation refresh) may reasonably attach a deposit again, following the same manual procedure they used initially. The probability is non-negligible because the same endpoint serves both deposit-required and deposit-not-required flows, with no on-chain signal to the caller that a deposit is unnecessary.

### Recommendation

Add an explicit zero-deposit assertion in the `else` branch (when `attestation_storage_must_be_paid_by_caller` is `false`):

```rust
if attestation_storage_must_be_paid_by_caller {
    // existing deposit check and refund logic
} else {
    // Existing participant re-submission: no storage cost, reject any attached deposit
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        return Err(InvalidParameters::InsufficientDeposit {
            attached: attached.as_yoctonear(),
            required: 0,
        }.into());
        // or: Promise::new(account_id).transfer(attached).detach();
    }
}
```

This mirrors the recommended fix from M-05: explicitly assert the "unused payment path" carries zero value, rather than silently accepting and freezing it.

### Proof of Concept

1. Participant `alice.near` joins the MPC network for the first time. She calls `submit_participant_info` manually with `--deposit 1000000000000000000000` yoctoNEAR to cover storage. `is_new_attestation = true`, so the deposit is consumed correctly.

2. Alice's node rotates its TLS key. Alice must re-submit her attestation. Following the same manual procedure, she again attaches `1000000000000000000000` yoctoNEAR.

3. On re-submission: `is_new_attestation = false` (existing entry updated, not newly inserted) and `caller_is_not_participant = false` (Alice is a current participant). Therefore `attestation_storage_must_be_paid_by_caller = false`.

4. The deposit-handling block at lines 826–849 is skipped entirely. [3](#0-2) 

5. Alice's `1000000000000000000000` yoctoNEAR is permanently frozen in the contract. No refund is issued. The call succeeds with `Ok(())`.
