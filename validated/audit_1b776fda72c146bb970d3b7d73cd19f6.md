### Title
PVE-AC Key Share Substitution via Unverified Ciphertext in `combine_ac` — (`src/cbmpc/api/pve_batch_ac.cpp`)

### Summary

The public API function `combine_ac` (and its C ABI counterpart `cbmpc_pve_ac_combine`) recovers PVE-AC key shares from an attacker-supplied ciphertext blob without ever running the PVE ciphertext verification. An adversary who can substitute the ciphertext input — e.g., by compromising ciphertext storage — can cause the recovery to silently return attacker-chosen scalars instead of the original encrypted key shares. The only post-recovery guard (`x[j] * G != Q[j]`) is trivially satisfied because both `x[j]` and `Q[j]` are read from the same attacker-controlled blob.

### Finding Description

`combine_ac` hard-codes `skip_verify=true` when it calls the internal `aggregate_to_restore_row`:

```cpp
rv = pve_ct.aggregate_to_restore_row(bridge, ac_internal, attempt_index, label, quorum_bn, xs_bn,
                                     /*skip_verify=*/true, pk_ptrs);
``` [1](#0-0) 

`aggregate_to_restore_row` only calls `verify()` when `skip_verify` is `false`:

```cpp
if (!skip_verify) {
    if (all_ac_pks.empty()) { ... }
    if (rv = verify(base_pke, ac, all_ac_pks, Q, label)) return rv;
}
``` [2](#0-1) 

After skipping verification, the function reconstructs `K` from the quorum shares, AES-GCM-decrypts `row.c`, and then checks:

```cpp
if (x[j] * G != Q[j]) return coinbase::error(E_CRYPTO);
``` [3](#0-2) 

Both `x[j]` (derived from `row.c` and `row.x_bin`) and `Q[j]` are read from the ciphertext blob itself. An attacker who controls the blob can set `Q[j] = x_attacker * G` and craft `row.c` to decrypt to `x_attacker` under a chosen `K`, making the check pass unconditionally.

The public C++ header documents this omission but frames it only as "designed to not leak secret information," not as "may return wrong key material":

```
// This function intentionally does not verify `ciphertext` before reconstruction.
// Invalid ciphertexts may cause reconstruction to fail, but are designed to not
// leak secret information.
// If you need ciphertext validation, call `verify_ac(...)` first.
``` [4](#0-3) 

The same note appears in the C ABI header: [5](#0-4) 

The `party_decrypt_row` function that produces the quorum shares also performs no cross-check against the public keys — it simply decrypts and returns the raw plaintext:

```cpp
if (rv = base_pke.decrypt(prv_key, L, c->ct_ser, plain)) return rv;
out_share = bn_t::from_bin(plain);
return SUCCESS;
``` [6](#0-5) 

So quorum members unwittingly decrypt the attacker's `K`-shares from the crafted leaf ciphertexts, and the aggregator reconstructs the attacker's `K`, completing the substitution without any party detecting the fraud.

### Impact Explanation

An adversary who can replace the ciphertext blob passed to `combine_ac` / `cbmpc_pve_ac_combine` can cause every honest quorum member to participate in recovering an attacker-chosen scalar `x_attacker` in place of the original backed-up key share `x`. The honest parties receive no error; the `x[j] * G == Q[j]` check passes because the attacker controls both sides. The recovered value is then used as the restored private key share, meaning the attacker has substituted the key material that the PVE backup was designed to protect.

This maps directly to the allowed Critical scope: *"A shipped API or protocol-peer path lets an attacker … substitute key shares … or equivalent secret material."*

### Likelihood Explanation

The attack requires the adversary to control the serialized ciphertext blob that is passed to `combine_ac`. This is realistic whenever:
- The ciphertext is stored on a server (the common deployment model for key backup) and that server is compromised.
- The ciphertext is fetched over a channel that is not integrity-protected end-to-end.

No quorum private key needs to be compromised. The adversary only needs the leaf public keys (which are public) to craft valid leaf ciphertexts for the attacker's `K`-shares.

### Recommendation

Remove the hard-coded `skip_verify=true` in `combine_ac` and require callers to supply the leaf public keys so that `aggregate_to_restore_row` can run `verify()` before reconstruction. Alternatively, split the function into a verified and an explicitly-unsafe variant, and make the unsafe variant unavailable through the stable public API. The current documentation warning is insufficient because it only mentions information leakage, not key-material substitution.

### Proof of Concept

**Setup**: access structure `THRESHOLD(2-of-3)` over leaves `{p1, p2, p3}`; curve secp256k1; original secret `x_orig`.

**Attack**:
1. Choose `x_attacker` (any scalar the adversary wants the recovery to return).
2. Compute `Q_fake = x_attacker * G`.
3. Choose a random `K_fake`; compute `k_aes, iv = RO(K_fake, L)`.
4. AES-GCM-encrypt `x_attacker` (as `x_bin`) under `k_aes` to produce `row.c_fake`.
5. Secret-share `K_fake` under the 2-of-3 structure; encrypt each share under the corresponding leaf public key to produce `quorum_c_fake`.
6. Assemble a well-formed `ec_pve_ac_t` blob with `Q = [Q_fake]`, `row.c = row.c_fake`, `row.quorum_c = quorum_c_fake`.
7. Submit this blob as the `ciphertext` argument to `cbmpc_pve_ac_partial_decrypt_attempt` for two quorum members → they decrypt their `K_fake` shares without error.
8. Call `cbmpc_pve_ac_combine` with the crafted ciphertext and the two shares.
9. Internally, `aggregate_to_restore_row` reconstructs `K_fake`, decrypts `row.c_fake` to `x_attacker`, checks `x_attacker * G == Q_fake` ✓, and returns `x_attacker`.

`combine_ac` returns `x_attacker` with `SUCCESS`; no party detects the substitution.
