### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Pending Request Content — (`crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash`, but never checks that `payload_hash` actually commits to the content of the `request` being resolved. A single Byzantine attested participant (below threshold) can replay a valid signature produced for a previous legitimate request to resolve a completely different pending `verify_foreign_transaction` request, delivering a fabricated attestation to the caller.

### Finding Description

In `respond_verify_foreign_tx`, the contract performs two independent checks:

1. The caller is an attested participant.
2. The signature in `response` is valid over `response.payload_hash` using the domain's root public key.

```rust
// crates/contract/src/lib.rs lines 718-734
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

What is **never** checked is whether `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request.request, observed_values }))`. The `payload_hash` field is a free parameter supplied by the responding node. The contract trusts it without binding it to the `request` argument that identifies which pending yield to resolve.

Compare this to the regular `respond` function, which derives the expected public key from `request.tweak` and verifies the signature against the **derived** key, tightly binding the response to the specific request:

```rust
// crates/contract/src/lib.rs lines 597-608
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
.is_ok()
``` [2](#0-1) 

For `respond_verify_foreign_tx`, no such binding exists. The `ForeignTxSignPayload` structure that nodes are supposed to sign includes the request content:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [3](#0-2) 

But the contract never recomputes or verifies this hash against the `request` it is resolving.

### Impact Explanation

A single Byzantine attested participant can:

1. Observe (or participate in) a legitimate `respond_verify_foreign_tx` call for request A, obtaining a valid MPC-produced signature `sig_A` over `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload{tx_A, values_A}))`. This signature is publicly visible on-chain.
2. Monitor the NEAR blockchain for a new pending `verify_foreign_transaction` request B (for a different foreign transaction).
3. Submit `respond_verify_foreign_tx(request=B, response={payload_hash=payload_hash_A, signature=sig_A})`.
4. The contract checks: (a) caller is attested participant ✓, (b) `sig_A` is valid over `payload_hash_A` under the root key ✓. It then resolves request B's pending yield with the fabricated response.

The legitimate response for request B (when it arrives) finds no pending entry and fails silently. The caller of request B receives `{payload_hash: payload_hash_A, signature: sig_A}` — data attesting to a completely different foreign transaction.

If the caller uses `ForeignChainSignatureVerifier::verify_signature` from the SDK, it will detect the mismatch between `response.payload_hash` and the locally recomputed hash:

```rust
// crates/near-mpc-sdk/src/foreign_chain.rs lines 57-63
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [4](#0-3) 

However, the pending request is already consumed and cannot be retried without a new deposit. For callers that do not perform this verification (or implement it incorrectly), the fabricated attestation is accepted as proof that a foreign transaction finalized — enabling invalid bridge execution (e.g., releasing Omnibridge funds for a transaction that was never verified).

### Likelihood Explanation

The attacker is a single attested MPC participant (below threshold). No threshold collusion is required: the signature used in the replay was legitimately produced by the full MPC network for a prior request and is publicly available on-chain. The attacker only needs to:

- Be an attested participant (a role that exists in production).
- Monitor the NEAR blockchain for pending `verify_foreign_transaction` requests.
- Submit the replay transaction before the legitimate response arrives.

Because MPC nodes already monitor the chain for events, a Byzantine node has the infrastructure to race the legitimate response.

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, also verify that `response.payload_hash` is consistent with the `request` being resolved. The most robust fix is to require the responding node to submit the full `ForeignTxSignPayload` (or at minimum the `ForeignChainRpcRequest` portion), recompute the hash on-chain, and assert equality:

```rust
// Pseudocode
let expected_prefix_hash = sha256(borsh(request.request));
assert!(payload_hash_commits_to_request(&response.payload_hash, &expected_prefix_hash),
    "payload_hash does not commit to the resolved request");
```

If submitting the full payload exceeds NEAR's promise-data limits, an alternative is to include the `VerifyForeignTransactionRequest` hash as a domain-separation prefix in the signed payload, so the contract can verify the binding using only the `request` argument it already holds.

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(Bitcoin tx_id=[0x01;32]) → pending request A
2. MPC nodes sign payload_hash_A = SHA-256(borsh(ForeignTxSignPayload{Bitcoin{[0x01;32]}, [BlockHash([0xaa;32])]}))
3. Legitimate respond_verify_foreign_tx(request=A, response={payload_hash_A, sig_A}) is submitted → request A resolved, sig_A is now on-chain
4. Bob submits verify_foreign_transaction(Bitcoin tx_id=[0x02;32]) → pending request B
5. Byzantine attested participant submits respond_verify_foreign_tx(request=B, response={payload_hash=payload_hash_A, signature=sig_A})
6. Contract checks: sig_A valid over payload_hash_A under root key ✓ → resolves request B with fabricated response
7. Bob's contract receives {payload_hash_A, sig_A} — an attestation for tx [0x01;32], not [0x02;32]
8. Legitimate response for request B arrives → no pending entry found → silently dropped
```
