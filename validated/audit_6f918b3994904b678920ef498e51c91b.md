### Title
Missing EVM Chain ID in `ForeignTxSignPayloadV1` Enables Cross-Chain Replay of Foreign Transaction Attestations — (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `ForeignTxSignPayloadV1` struct, whose Borsh-serialized SHA-256 hash is what MPC nodes collectively sign during `verify_foreign_transaction`, does not include the EVM numeric chain ID. Chain identity is encoded only as a Borsh enum-variant discriminant (`ForeignChainRpcRequest::Bnb`, `::Base`, `::Arbitrum`, etc.). If any supported EVM chain forks after deployment, the same transaction ID and block hash exist on both the original chain and the fork, producing an identical `msg_hash`. The threshold signature produced for the original chain is therefore cryptographically valid for the fork chain, enabling cross-chain replay of bridge attestations and double-spend conditions.

---

### Finding Description

`ForeignTxSignPayloadV1` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

The 32-byte message hash that MPC nodes sign is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [1](#0-0) 

The `ForeignChainRpcRequest` enum variant discriminant (e.g., `Bnb`, `Base`, `Arbitrum`) is included in the Borsh encoding, which prevents replay *between* currently registered chains. However, the `EvmRpcRequest` inner struct — shared by all EVM-compatible chains — contains no EVM chain ID:

```rust
pub struct EvmRpcRequest {
    pub tx_id: EvmTxId,
    pub extractors: Vec<EvmExtractor>,
    pub finality: EvmFinality,
}
``` [2](#0-1) 

All currently active EVM-family chains — `Abstract`, `Bnb`, `Base`, `Arbitrum`, `Polygon`, `HyperEvm` — route through this same struct: [3](#0-2) 

In `build_signature_request`, the `tweak` is hardcoded to all-zeros, and the payload hash is derived purely from `ForeignTxSignPayload`: [4](#0-3) 

No EVM chain ID is injected anywhere in the signing pipeline between `execute_foreign_chain_request` and the final `make_signature_leader_given_parameters` call: [5](#0-4) 

The design document mentions a startup chain-identity probe (`eth_chainId` RPC call), but this is a one-time startup check, not a per-request binding. It does not embed the chain ID into the signed payload and cannot prevent replay once a signature has been produced.

---

### Impact Explanation

**High — Cross-chain replay / double-spend.**

If a supported EVM chain (e.g., BNB Smart Chain) undergoes a hard fork after the MPC network is deployed:

1. The same `tx_id` and `BlockHash` exist on both the original chain and the fork at the fork point.
2. The MPC network signs `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: ForeignChainRpcRequest::Bnb(EvmRpcRequest { tx_id, ... }), values: [BlockHash(...)] })`.
3. Because no EVM chain ID is included, the resulting `msg_hash` and threshold signature are identical for both chains.
4. A bridge contract on NEAR that calls `verify_foreign_transaction` to attest a deposit on BNB mainnet receives a `VerifyForeignTransactionResponse` that is equally valid as an attestation for the fork chain.
5. An attacker replays the same response to claim a second payout on the fork, causing a double-spend.

This directly matches the allowed impact: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

**Medium.** EVM chain forks are historically common (ETH/ETC, BSC/BNB governance disputes, L2 sequencer upgrades that create temporary divergence). The MPC network already supports multiple EVM chains (`Bnb`, `Base`, `Arbitrum`, `Polygon`, `HyperEvm`, `Abstract`). Any fork of any of these chains after the MPC network goes live creates the replay window. The attacker only needs to submit a standard `verify_foreign_transaction` call — no privileged access is required.

---

### Recommendation

**Short term:** Add the EVM numeric chain ID to `EvmRpcRequest` and include it in the Borsh-serialized `ForeignTxSignPayloadV1`. This binds each MPC attestation to a specific chain ID, making the signature invalid on any fork with a different chain ID.

```rust
pub struct EvmRpcRequest {
    pub tx_id: EvmTxId,
    pub extractors: Vec<EvmExtractor>,
    pub finality: EvmFinality,
    pub chain_id: u64,  // e.g., 56 for BNB, 8453 for Base, 42161 for Arbitrum
}
```

**Long term:** Extend the chain-identity probe (currently a startup check) to also verify the chain ID of each RPC response at request time, and document the fork-risk policy for each supported chain.

---

### Proof of Concept

1. BNB Smart Chain (chain ID 56) is a whitelisted chain. Alice deposits 10,000 USDT into a NEAR bridge contract that calls `verify_foreign_transaction` to attest BNB deposits.

2. Alice's deposit transaction `tx_id = 0xDEAD...` is confirmed. The MPC network signs:
   ```
   msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
       request: ForeignChainRpcRequest::Bnb(EvmRpcRequest {
           tx_id: 0xDEAD...,
           extractors: [BlockHash],
           finality: Finalized,
       }),
       values: [BlockHash(0xBEEF...)],
   })))
   ```
   No chain ID is included. The bridge contract releases 10,000 USDT on NEAR.

3. A governance dispute causes BNB to fork. The fork chain retains chain ID 56 and the same transaction history up to the fork block, so `tx_id = 0xDEAD...` with `BlockHash = 0xBEEF...` exists on the fork too.

4. The attacker replays the identical `VerifyForeignTransactionResponse` (same `payload_hash`, same threshold signature) to the bridge contract, claiming a second deposit on the fork chain.

5. Because the `msg_hash` is identical (no chain ID in the payload), the bridge contract's signature verification passes, and the attacker receives a second 10,000 USDT payout — a double-spend. [6](#0-5) [7](#0-6)

        let mut hasher = sha2::Sha256::new();
