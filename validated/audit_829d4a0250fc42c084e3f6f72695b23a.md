Audit Report

## Title
`getExpectedAssetAmount` Returns Gross Amount Excluding `instantWithdrawalFee`, Overstating Net Asset Received - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`getExpectedAssetAmount` is documented as returning "Amount of underlying to receive" but computes only the oracle-derived gross amount with no awareness of `instantWithdrawalFee`. `instantWithdrawal` uses this same gross value as its base and then silently deducts the fee before transferring to the user, creating a structural mismatch between the view function's implied semantics and the actual execution result. Any caller that uses `getExpectedAssetAmount` as a quote for `instantWithdrawal` will consistently receive fewer assets than quoted.

## Finding Description
`getExpectedAssetAmount` (L580–594) computes `underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)` with no fee deduction. Its NatSpec return tag is `@return underlyingToReceive Amount of underlying to receive`, implying this is the user's net receipt.

Inside `instantWithdrawal` (L228–250), the same function is called to obtain `assetAmountUnlocked`, which is then used as the gross base:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked); // gross
...
uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
uint256 userAmount = assetAmountUnlocked - fee;                             // net
_transferAsset(asset, msg.sender, userAmount);
```

`instantWithdrawalFee` is configurable up to 1000 bps (L372–374). Whenever it is non-zero, `userAmount < assetAmountUnlocked`, so the user receives strictly less than `getExpectedAssetAmount` returned. An on-chain integrator that enforces `received >= getExpectedAssetAmount(asset, rsETHAmount)` will revert on every call while the fee is active, permanently blocking that integration path. No existing guard in `instantWithdrawal` warns the caller of this discrepancy.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** The view function's documented return value (`Amount of underlying to receive`) overstates the actual transfer by exactly `fee` basis points. No funds are lost from the protocol; the fee is correctly routed to the fee recipient. However, any caller relying on `getExpectedAssetAmount` as a minimum-output quote for `instantWithdrawal` will either receive less than expected or have their transaction revert if a slippage guard is applied against the quoted value.

## Likelihood Explanation
The condition is triggered whenever `instantWithdrawalFee > 0`, which is a normal, intended protocol state (the fee exists precisely to be used). Any user or integrator following the natural "query view → execute" pattern is affected. No special privileges, flash loans, or external dependencies are required — a standard EOA or smart contract wallet calling `getExpectedAssetAmount` then `instantWithdrawal` is sufficient to observe the discrepancy.

## Recommendation
Add a dedicated view function that returns the post-fee net amount for `instantWithdrawal` callers:

```solidity
function getExpectedAssetAmountAfterFee(address asset, uint256 rsETHAmount)
    public view returns (uint256 netAmount)
{
    uint256 gross = getExpectedAssetAmount(asset, rsETHAmount);
    uint256 fee = (gross * instantWithdrawalFee) / 10_000;
    netAmount = gross - fee;
}
```

At minimum, update the NatSpec of `getExpectedAssetAmount` to explicitly state it returns the gross pre-fee amount and is not suitable as a quote for `instantWithdrawal`.

## Proof of Concept
1. `instantWithdrawalFee` is set to 100 bps (1 %) via `setInstantWithdrawalFee(100)`.
2. Integrator calls `getExpectedAssetAmount(ETH, 1000e18)` → returns `X` (gross).
3. Integrator calls `instantWithdrawal(ETH, 1000e18, "")` expecting to receive `X`.
4. Contract computes `fee = X * 100 / 10_000 = 0.01 * X`; transfers `userAmount = 0.99 * X`.
5. Integrator receives 1 % less than quoted. If the integrator enforces `received >= X`, the call reverts on every execution while the fee is non-zero. [1](#0-0) [2](#0-1) [3](#0-2)
