Audit Report

## Title
Missing initialization of `maxNumberOfWithdrawalsPerUser` causes DoS on `initiateWithdrawal()` before admin configuration - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never assigns a value to `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal()` guards with `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, the condition evaluates to `0 >= 0 = true` for every caller, causing every withdrawal attempt to revert with `WithdrawalLimitReached` until the admin explicitly calls `setMaxNumberOfWithdrawalsPerUser`. Any user who stakes before that admin call has their tokens temporarily frozen with no withdrawal path.

## Finding Description
`initialize()` sets only `kernelToken` and `rewardsToken`; neither `withdrawalDelay` nor `maxNumberOfWithdrawalsPerUser` receives a default value:

```solidity
// L259-271
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
    // maxNumberOfWithdrawalsPerUser remains 0
}
```

`initiateWithdrawal()` at L323 checks:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```

With `maxNumberOfWithdrawalsPerUser == 0`, any user whose pending withdrawal list is empty (`length == 0`) satisfies `0 >= 0`, causing an immediate revert. The only path to unblock this is the admin calling `setMaxNumberOfWithdrawalsPerUser`, which itself validates that the argument is non-zero (L614), confirming `0` is never a valid operational value. Meanwhile, `stake()` (L281-289) has no such dependency and accepts deposits freely, creating an asymmetry where funds can enter but not exit.

## Impact Explanation
**Medium — Temporary freezing of funds.** Any user who stakes KERNEL tokens before the admin completes post-`initialize()` configuration cannot initiate a withdrawal. Their staked balance is locked in the contract for an indefinite period. The freeze is not permanent (admin can unblock), but the user has no self-service remedy and receives no on-chain warning that withdrawals are disabled at the time of staking.

## Likelihood Explanation
The deployment sequence is: deploy proxy → `initialize()` → (optionally) `setWithdrawalDelay` + `setMaxNumberOfWithdrawalsPerUser` → open to users. There is no on-chain enforcement that the two setter calls precede any user interaction. A user who stakes immediately after `initialize()` — before the admin completes configuration — will find their tokens frozen. This is a realistic race condition in any deployment where staking is not gated behind a separate activation step.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to the existing constant `MAX_WITHDRAWALS_PER_USER` (value: `100`) inside `initialize()`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // add this
}
```

Similarly, `withdrawalDelay` should be initialized to a non-zero default (e.g., `1 days`) to prevent zero-delay withdrawals if `setWithdrawalDelay` is also not called.

## Proof of Concept
1. Deploy `KernelDepositPool` proxy and call `initialize(admin, kernelToken, rewardToken)`.
2. Admin does **not** call `setMaxNumberOfWithdrawalsPerUser` (incomplete deployment script).
3. User calls `stake(100e18)` → **succeeds**. `balanceOf[user] = 100e18`.
4. User calls `initiateWithdrawal(100e18)` → **reverts** with `WithdrawalLimitReached` because `userWithdrawalIds[user].length (0) >= maxNumberOfWithdrawalsPerUser (0)` is `true`.
5. User's `100e18` KERNEL tokens are locked with no withdrawal path until admin calls `setMaxNumberOfWithdrawalsPerUser(N)`.

**Foundry test sketch:**
```solidity
function test_withdrawalDosBeforeAdminConfig() public {
    // deploy and initialize without calling setMaxNumberOfWithdrawalsPerUser
    pool.initialize(admin, address(kernelToken), address(rewardToken));
    kernelToken.mint(user, 100e18);
    vm.startPrank(user);
    kernelToken.approve(address(pool), 100e18);
    pool.stake(100e18); // succeeds
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(100e18); // reverts
    vm.stopPrank();
}
```
