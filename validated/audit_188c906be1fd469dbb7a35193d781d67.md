### Title
Cross-Chain Replay of `deployToken` Metadata Signature Bypasses Domain Separation — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.deployToken` verifies a NEAR-MPC signature over token metadata but omits `omniBridgeChainId` (or any EVM chain identifier) from the signed payload. Because the same MPC-derived key (`nearBridgeDerivedAddress`) signs for all EVM deployments, a signature that was legitimately issued and publicly submitted on Chain A can be replayed verbatim on Chain B by any unprivileged observer, causing `deployTokenExtension` to execute on the wrong chain without authorization.

---

### Finding Description

`deployToken` constructs its signed digest as:

```
PayloadType.Metadata | token | name | symbol | decimals
``` [1](#0-0) 

There is no `omniBridgeChainId`, no EVM `block.chainid`, and no contract address in this hash. The contract stores `omniBridgeChainId` as a state variable and uses it correctly in `finTransfer`: [2](#0-1) 

The Aptos bridge explicitly documents the design intent — chain ID is "mixed into transfer payload hashes to prevent cross-chain replay": [3](#0-2) 

`deployToken` is the only public entry point that reaches `deployTokenExtension`: [4](#0-3) 

(`addCustomToken` is `onlyRole(DEFAULT_ADMIN_ROLE)` and is out of the unprivileged attacker path.)

In `OmniBridgeWormhole`, `deployTokenExtension` publishes a Wormhole message registering the token on the target chain: [5](#0-4) 

---

### Impact Explanation

An attacker who observes a legitimate `deployToken(signatureData, metadata)` call on Chain A (all calldata is public) can submit the identical call on Chain B:

1. Token is deployed on Chain B at a new address without NEAR-side authorization for that chain.
2. `nearToEthToken[metadata.token]` is permanently set on Chain B — any subsequent legitimate deployment attempt reverts with `ERR_TOKEN_EXIST`, permanently blocking the correct deployment.
3. In the Wormhole variant, a Wormhole VAA is emitted claiming the token is live on Chain B, which the NEAR bridge may process and register as a valid mapping.

This fits the allowed impact: **acceptance of insufficiently-bound signatures that bypass execution gates**, and **irreversible state corruption that freezes the legitimate deployment path**.

---

### Likelihood Explanation

- The NEAR MPC bridge uses a single derived key across all supported EVM chains (same `nearBridgeDerivedAddress` configured on each deployment).
- Every `deployToken` call exposes `signatureData` as public calldata.
- No privileged access, leaked key, or special tooling is required — only the ability to submit a transaction on a second chain.
- The attack is permanent: once `nearToEthToken[token]` is set, it cannot be cleared by an unprivileged party.

---

### Recommendation

Include `omniBridgeChainId` (and optionally `address(this)`) in the signed payload for `deployToken`, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR MPC signer must include the destination chain ID when producing the signature.

---

### Proof of Concept

```
Setup:
  - Deploy OmniBridgeWormhole on Chain A (chainId=1) and Chain B (chainId=56),
    both configured with the same nearBridgeDerivedAddress (standard MPC setup).
  - NEAR bridge issues signature S over (Metadata, "token.near", "Token", "TKN", 18).

Step 1 (Chain A — legitimate):
  deployToken(S, {token:"token.near", name:"Token", symbol:"TKN", decimals:18})
  → Token deployed at address X on Chain A. Wormhole VAA emitted for chain 1.

Step 2 (Chain B — replay, unprivileged attacker):
  deployToken(S, {token:"token.near", name:"Token", symbol:"TKN", decimals:18})
  → ECDSA.recover(keccak256(same bytes), S) == nearBridgeDerivedAddress ✓
  → Token deployed at address Y on Chain B. Wormhole VAA emitted for chain 56.
  → nearToEthToken["token.near"] = Y permanently set on Chain B.

Step 3 (verify divergence):
  - Chain B now has an unauthorized token registration.
  - Any admin attempt to deploy "token.near" on Chain B reverts: ERR_TOKEN_EXIST.
  - Balances and finalization state diverge from the invariant: a token is
    registered on Chain B without a corresponding NEAR-side authorization for chain 56.
```

The two route-equivalent messages (Chain A deployment, Chain B replay) produce divergent finalised state — the invariant that domain separation across chain kind and bridge address holds is violated.
