### Title
Missing `builder` Field Validation in `ProposalInit` Allows Malicious Proposer to Redirect Sequencer Fees - (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

### Summary

`is_proposal_init_valid` validates many fields of `ProposalInit` but never checks the `builder` field (the sequencer/fee-recipient address). A malicious proposer can set `builder` to an arbitrary address. The validator blindly forwards this address to the batcher as the sequencer address in `BlockInfo`, causing all transaction fees in the block to be credited to the attacker's address. The final `ProposalCommitment` comparison does not catch this because both sides execute with the same (attacker-supplied) builder address.

### Finding Description

`ProposalInit` carries a `builder` field — the address of the entity that sequences the block and receives transaction fees. [1](#0-0) 

When a validator processes an incoming proposal, `is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices, `starknet_version`, `version_constant_commitment`, `timestamp`, and `fee_proposal_fri`. It does **not** check `builder`. [2](#0-1) 

`ProposalInitValidation` — the struct that carries the validator's reference values — has no `builder` field at all, so there is no locally-trusted value to compare against. [3](#0-2) 

After `is_proposal_init_valid` passes, `initiate_validation` calls `convert_to_sn_api_block_info(init)` and forwards the result to the batcher as `ValidateBlockInput.block_info`. The batcher executes all transactions in the block using this `BlockInfo`, which includes the proposer-supplied `builder` as the sequencer address. [4](#0-3) 

The proposer sets its own `builder` from a locally-configured `builder_address`: [5](#0-4) 

Because the validator uses the received `init.builder` (not its own configured address) to drive execution, both sides compute the same `partial_block_hash` and the final commitment comparison passes: [6](#0-5) 

### Impact Explanation

The `builder` address is the Starknet sequencer address. All transaction fees collected during block execution are credited to this address. A malicious proposer that substitutes `builder = attacker_address` causes every validator to execute the block with `attacker_address` as the sequencer, so:

- All fees for that block are credited to the attacker's account in the resulting state diff.
- `get_sequencer_address` syscall returns the attacker's address to any contract that queries it.
- The committed block and its state root reflect the wrong fee recipient.

This matches the impact category: **Incorrect fee, gas, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

### Likelihood Explanation

The attacker must be a registered validator and be selected as the proposer for at least one round. In a Tendermint-style committee, proposer selection is deterministic and rotates among validators, so any validator node can reach this code path without any additional privilege beyond committee membership.

### Recommendation

Add `builder` to `ProposalInitValidation` and check it in `is_proposal_init_valid`:

```rust
pub(crate) struct ProposalInitValidation {
    pub height: BlockNumber,
    pub builder: ContractAddress,   // add this
    // ... existing fields
}

// in is_proposal_init_valid:
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

Populate `proposal_init_validation.builder` from the node's configured `builder_address` in both the current-round path and the queued-proposal path inside `set_height_and_round`. [7](#0-6) [8](#0-7) 

### Proof of Concept

1. Attacker operates a validator node and is selected as proposer for height H, round R.
2. Attacker constructs `ProposalInit` with all fields matching the expected values **except** `builder = attacker_address`.
3. Attacker streams the proposal to peer validators.
4. Each validator calls `is_proposal_init_valid` — all checks pass because `builder` is never compared.
5. `initiate_validation` forwards `attacker_address` as the sequencer address to the batcher.
6. The batcher executes all transactions; fees are credited to `attacker_address` in the state diff.
7. `finish_proposal` returns a `partial_block_hash` computed with `attacker_address` as sequencer.
8. `proposal_commitment_from(partial_block_hash, fee_proposal_fri)` matches the proposer's `ProposalFin.proposal_commitment` — the final check passes.
9. Consensus commits the block; the state root encodes `attacker_address` as the fee recipient for all transactions in block H.
