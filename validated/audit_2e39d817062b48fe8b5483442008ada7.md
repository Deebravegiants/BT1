### Title
`GasKeyFunctionCall` Permission Restrictions Not Enforced in Gas Key Transaction Validation - (`File: runtime/runtime/src/verifier.rs`)

### Summary

The `verify_and_charge_gas_key_tx_ephemeral` function, which validates gas key transactions (those carrying a `nonce_index`), does not enforce the `FunctionCallPermission` constraints (receiver account and allowed method names) present on `GasKeyFunctionCall` access keys. A relayer holding such a restricted key can bypass the restrictions and call any method on any contract.

### Finding Description

`AccessKeyPermission` has four variants:
- `FunctionCall(FunctionCallPermission)` — regular function-call key
- `FullAccess` — full access key
- `GasKeyFunctionCall(GasKeyInfo, FunctionCallPermission)` — gas key restricted to specific receiver/methods
- `GasKeyFullAccess(GasKeyInfo)` — gas key with full access [1](#0-0) 

For regular access key transactions, `verify_and_charge_tx_ephemeral` explicitly calls `verify_function_call_permission` to enforce the `FunctionCallPermission` constraints: [2](#0-1) 

`verify_function_call_permission` enforces receiver and method name restrictions: [3](#0-2) 

However, for gas key transactions (those with a `nonce_index`), the code dispatches to `verify_and_charge_gas_key_tx_ephemeral`. This function validates the gas key info, nonce index, nonce, and balance — but **never calls `verify_function_call_permission`**: [4](#0-3) 

The function proceeds directly to balance checks and returns `TxVerdict::Success` without any check on the `FunctionCallPermission` receiver or method name restrictions embedded in a `GasKeyFunctionCall` key. [5](#0-4) 

The `validate_action_with_mode` dispatch in `action_validation.rs` only validates that the `GasKeys` protocol feature is enabled for `TransferToGasKey`/`WithdrawFromGasKey` actions; it does not re-check function call permission constraints at the receipt level either: [6](#0-5) 

### Impact Explanation

A `GasKeyFunctionCall` key is designed to allow a third-party relayer to pay gas on behalf of a user, but only for a specific receiver account and a specific set of method names. Because `verify_and_charge_gas_key_tx_ephemeral` skips the `verify_function_call_permission` check, a relayer holding such a key can:

1. Call any method on any contract (not just the permitted receiver/methods).
2. Effectively treat a `GasKeyFunctionCall` key as a `GasKeyFullAccess` key.

The corrupted protocol value is the `FunctionCallPermission` restriction stored in the access key state: the runtime accepts and executes transactions that violate the on-chain permission, corrupting the intended authorization invariant. This can lead to unauthorized contract calls (e.g., unauthorized token transfers, unauthorized state mutations on arbitrary contracts) chargeable to the gas key's balance.

### Likelihood Explanation

The `GasKeys` feature is gated behind `ProtocolFeature::GasKeys`. Once enabled, any user who creates a `GasKeyFunctionCall` key and delegates it to a relayer is exposed. The relayer is an unprivileged external party who controls the signed transaction fields (receiver, method name, `nonce_index`) and can trivially construct a transaction that violates the permission. No special privileges are required beyond possession of the gas key's private key.

### Recommendation

Add a call to `verify_function_call_permission` inside `verify_and_charge_gas_key_tx_ephemeral` after the nonce check, mirroring the check already present in `verify_and_charge_tx_ephemeral`:

```rust
// After nonce validation, before balance checks:
if let Some(function_call_permission) = access_key.permission.function_call_permission()
    && let Err(e) = verify_function_call_permission(function_call_permission, tx)
{
    return TxVerdict::Failed(e);
}
``` [2](#0-1) 

### Proof of Concept

1. Alice creates a `GasKeyFunctionCall` key for relayer Bob, restricted to `receiver_id = "dex.near"` and `method_names = ["swap"]`.
2. Bob constructs a `TransactionV1` with `nonce_index = 0` (gas key nonce), `receiver_id = "token.near"`, and action `FunctionCall { method_name: "transfer", ... }`.
3. Bob signs with the gas key private key and submits.
4. `verify_and_charge_gas_key_tx_ephemeral` validates the gas key info, nonce, and balance — but never checks that `receiver_id == "dex.near"` or `method_name == "swap"`.
5. The transaction is accepted and executed, calling `token.near::transfer` despite the key being restricted to `dex.near::swap`. [7](#0-6)

/// Validates FunctionCall permission constraints:
/// - Transaction must have exactly one action
/// - Action must be FunctionCall with zero deposit
/// - Receiver must match permission's receiver
/// - Method name must be in allowed list (if list is non-empty)
fn verify_function_call_permission(
    function_call_permission: &FunctionCallPermission,
    tx: &Transaction,
) -> Result<(), InvalidTxError> {
    if tx.actions().len() != 1 {
        return Err(InvalidTxError::InvalidAccessKeyError(
            InvalidAccessKeyError::RequiresFullAccess,
        ));
    }
    let Some(Action::FunctionCall(function_call)) = tx.actions().get(0) else {
        return Err(InvalidTxError::InvalidAccessKeyError(
            InvalidAccessKeyError::RequiresFullAccess,
        ));
    };
    if function_call.deposit > Balance::ZERO {
        return Err(InvalidTxError::InvalidAccessKeyError(
            InvalidAccessKeyError::DepositWithFunctionCall,
        ));
    }
    let tx_receiver = tx.receiver_id();
    let ak_receiver = &function_call_permission.receiver_id;
    if tx_receiver != ak_receiver {
        return Err(InvalidTxError::InvalidAccessKeyError(
            InvalidAccessKeyError::ReceiverMismatch {
                tx_receiver: tx_receiver.clone(),
                ak_receiver: ak_receiver.clone(),
            },
        ));
    }
    if !function_call_permission.method_names.is_empty()
        && function_call_permission
            .method_names
            .iter()
            .all(|method_name| &function_call.method_name != method_name)
    {
        return Err(InvalidTxError::InvalidAccessKeyError(
            InvalidAccessKeyError::MethodNameMismatch {
                method_name: function_call.method_name.clone(),
            },
        ));
    }
    Ok(())

/// Validates a single given action.
/// The `mode` only affects nested validation of `Action::Delegate` payloads
fn validate_action_with_mode(

    action: &Action,

    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,

    match action {

        Action::DeleteAccount(a) => validate_delete_action(a),
