Audit Report

## Title
ETH Deposit Limit Check Excludes Deposit Amount, Allowing Bypass of `depositLimitByAsset` - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ETH it evaluates `totalAssetDeposits > limit` (omitting `amount`), while for LST assets it correctly evaluates `totalAssetDeposits + amount > limit`. Any depositor can therefore push total ETH deposits arbitrarily above the configured cap in a single `depositETH` call, bypassing the protocol's primary deposit-cap control.

## Finding Description
In `contracts/LRTDepositPool.sol` lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
```

`_beforeDeposit` (lines 648–670) calls this function at line 661 and reverts with `MaximumDepositLimitReached` only when it returns `true`. Because the ETH branch never adds `amount` to `totalAssetDeposits`, the check only fires when the cap has **already** been exceeded before the deposit. A deposit that would push the total from just-below-limit to far-above-limit passes without error. The LST path correctly blocks such a deposit. Existing guards (`whenNotPaused`, minimum deposit amount) do not address this asymmetry.

## Impact Explanation
The `depositLimitByAsset` cap is the protocol's primary risk-management control over ETH exposure accepted into EigenLayer restaking. Bypassing it allows unbounded ETH accumulation beyond the admin-configured ceiling, violating the invariant the contract is supposed to enforce. No funds are directly stolen or frozen. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`depositETH` is `public payable` with no access control beyond `whenNotPaused`. The condition is exploitable whenever `totalAssetDeposits(ETH_TOKEN) ≤ depositLimitByAsset(ETH_TOKEN)`, which is the normal operating state. Any depositor can trigger this in a single transaction with no special role, front-running, or external dependency. **Likelihood: High**.

## Recommendation
Include `amount` in the ETH branch to match the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Attacker calls `depositETH{value: 500 ether}(minRSETH, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999e18 > 1000e18` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the full 500 ETH; total ETH deposits become `1499 ether`.
6. The same call with an LST would evaluate `999e18 + 500e18 > 1000e18` → `true` → `MaximumDepositLimitReached` revert.

Foundry test plan: deploy with a mock `lrtConfig` returning `depositLimitByAsset = 1000 ether`, seed `999 ether` of prior deposits, call `depositETH{value: 500 ether}`, assert the call succeeds and `getTotalAssetDeposits` returns `1499 ether` — confirming the cap is not enforced.
