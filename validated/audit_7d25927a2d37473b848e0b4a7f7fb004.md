### Title
`KernelTop100MerkleDistributor.sol`: `withdrawTokens` function does not prevent KERNEL distribution tokens from being transferred - (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

### Summary

`KernelTop100MerkleDistributor` holds KERNEL tokens that are meant to be distributed to users via a vesting/merkle claim schedule. The `withdrawTokens` function allows the owner to withdraw any token — including the KERNEL token itself — with no guard protecting tokens owed to claimants. If the owner drains the KERNEL balance, every subsequent `claim()` or `claimAndStake()` call reverts due to insufficient balance, permanently freezing unclaimed yield.

### Finding Description

The `withdrawTokens` function performs no check on whether the token being withdrawn is the distribution token (`kernel`) and no check on how much of the balance is already committed to pending user claims: [1](#0-0) 

```solidity
function withdrawTokens(address _token, uint256 _amount, address _recipient) external onlyOwner {
    UtilLib.checkNonZeroAddress(_token);
    UtilLib.checkNonZeroAddress(_recipient);
    if (_amount == 0) {
        revert ZeroValueProvided();
    }
    IERC20(_token).safeTransfer(_recipient, _amount);
    emit TokensWithdrawn(_token, _amount, _recipient);
}
```

The contract tracks per-user vested amounts via `userClaims` and a merkle root, but maintains no global counter of total KERNEL tokens still owed to all eligible claimants. [2](#0-1) 

When a user calls `claim()`, the contract transfers KERNEL directly from its own balance: [3](#0-2) 

If the owner has already drained the KERNEL balance via `withdrawTokens`, this transfer reverts.

The same revert path exists in `claimAndStake()`: [4](#0-3) 

### Impact Explanation

**High — Theft of unclaimed yield / Permanent freezing of unclaimed yield.**

All users who have valid merkle proofs and unvested KERNEL tokens lose access to their entitled distribution. The vesting schedule and merkle proof remain valid on-chain, but the underlying token balance is gone. There is no recovery path for affected users within the contract itself.

### Likelihood Explanation

**Low-Medium.** The function is gated to `onlyOwner` and is intended for recovering accidentally sent tokens. The risk materialises either through a malicious owner or — more plausibly — an operator mistake where the owner treats the KERNEL balance as "excess" tokens and sweeps it, not realising it is the distribution reserve. The absence of any guard (unlike the analogous `lpToken`/`muteToken` checks in MuteAmplifier) makes an accidental drain realistic.

### Recommendation

Add a guard inside `withdrawTokens` that prevents withdrawing more KERNEL than the surplus above the total remaining claimable amount. Because the contract does not track a global "total owed" counter, the simplest fix is to either:

1. **Reject KERNEL withdrawals entirely** (the distribution reserve should never need rescuing), or
2. **Track total tokens committed** to the merkle distribution and enforce:

```solidity
if (_token == address(kernel)) {
    uint256 committed = totalDistributableAmount - totalAlreadyClaimed;
    require(
        IERC20(kernel).balanceOf(address(this)) - _amount >= committed,
        "KernelTop100MerkleDistributor: cannot withdraw committed KERNEL"
    );
}
```

### Proof of Concept

1. The contract is funded with 1,000,000 KERNEL for distribution to 100 eligible users.
2. Users begin claiming; 200,000 KERNEL has been claimed so far.
3. Owner calls `withdrawTokens(address(kernel), 800,000e18, ownerAddress)`.
4. The call succeeds — no revert — and the contract KERNEL balance drops to 0.
5. Any remaining eligible user calls `claim()`. The `kernel.safeTransfer(user, amountToSend)` at line 335 reverts with an ERC-20 insufficient-balance error.
6. All 800,000 KERNEL of unclaimed yield is permanently inaccessible to its rightful recipients. [5](#0-4) [1](#0-0)
