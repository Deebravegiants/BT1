### Title
Non-Atomic Deposit Accounting in Dijkstra LEDGER Rule Enables Permanent Deposit Pot Inflation via Subtransaction Pool Registration — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`)

---

### Summary

The Dijkstra era's `dijkstraLedgerTransition` passes the **original pre-subtransaction `certState`** to UTXOW/UTXO for deposit accounting, while ENTITIES uses the **post-subtransaction `certState`**. An unprivileged transaction author can craft a batch transaction where a subtransaction registers a stake pool and the top-level transaction re-registers the same pool. Because UTXO sees the pool as "not yet registered" (using the stale certState), it counts a full pool deposit again, permanently inflating `utxosDeposited` by one pool deposit amount (`ppPoolDepositL`) while the actual obligation tracked in `certState` does not increase.

---

### Finding Description

In `dijkstraLedgerTransition`:

**Step 1 — Original certState is captured:** [1](#0-0) 

**Step 2 — SUBLEDGERS processes all subtransactions, producing `certStateAfterSubLedgers` (which may include newly registered pools):** [2](#0-1) 

**Step 3 — ENTITIES processes top-level certificates using `certStateAfterSubLedgers`:** [3](#0-2) 

**Step 4 — UTXOW is called with the ORIGINAL `certState` (`lsCertState ledgerState`), not `certStateFinal`:** [4](#0-3) 

Inside `dijkstraUtxoTransition`, this original `certState` is used for both value conservation and deposit accounting: [5](#0-4) [6](#0-5) 

The deposit calculation path is `Babbage.updateUTxOStateByTxValidity` → `updateUTxOState` → `certsTotalDepositsTxBody pp certState txBody`: [7](#0-6) 

For Dijkstra, `certsTotalDepositsTxBody = conwayCertsTotalDepositsTxBody`, which checks `psStakePools conwayCertPState` to determine whether a pool is already registered: [8](#0-7) [9](#0-8) 

The pool-registration deposit counter in `shelleyTotalDepositsTxCerts` only skips a pool if `isRegPoolRegistered sppId` returns `True`: [10](#0-9) 

With the original certState, a pool registered in a subtransaction is absent from `psStakePools`, so `isRegPoolRegistered` returns `False` and the deposit `D` is counted again.

### Impact Explanation

**Concrete execution trace:**

| Step | `utxosDeposited` | `obligations` (certState) |
|---|---|---|
| Initial | X | X |
| After subtx registers pool P | X + D | X + D |
| After ENTITIES: top-level `RegPool P` is a re-registration (pool P already in `certStateAfterSubLedgers`) | X + D | X + D (unchanged) |
| After UTXOW/UTXO: top-level `RegPool P` counted as new registration (pool P absent from original certState) | **X + 2D** | X + D |

The `potEqualsObligation` invariant is violated by `D`: [11](#0-10) 

The PostCondition assertion on the Dijkstra LEDGER rule checks this invariant: [12](#0-11) [13](#0-12) 

If this PostCondition assertion is **disabled** in production (the typical configuration for performance), the transaction is accepted and the excess `D` ADA is permanently locked in the deposit pot. It cannot be recovered without a hard fork that corrects `utxosDeposited`. This matches:

> **High. Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork.**

If assertions are **enabled**, the transition fails and the transaction is rejected — no harm done, but the root cause remains.

---

### Likelihood Explanation

- **Attacker-controlled entry path:** An unprivileged transaction author submits a Dijkstra-era batch transaction. No privileged access is required.
- **Preconditions:** (1) Dijkstra era is deployed on-chain; (2) assertions are disabled in the production node (standard for performance); (3) the attacker has `D` ADA available to fund the top-level transaction's value conservation requirement.
- **Cost to attacker:** The attacker must spend `D` ADA (the pool deposit) to lock `D` ADA permanently. This is a griefing/protocol-harm attack rather than a profit attack.
- **Likelihood:** **Medium** — the Dijkstra era is currently experimental, but the code is in-scope and the attack path is straightforward once the era is active.

---

### Recommendation

In `dijkstraLedgerTransition`, pass `certStateFinal` (post-ENTITIES) to UTXOW for deposit accounting, while retaining the original certState only for refund lookups (deregistration refunds require the pre-CERTS certState to look up the original deposit amount). Concretely:

1. Split the `DijkstraUtxoEnv` to carry two certState snapshots: one for deposit calculation (post-subtransaction) and one for refund lookup (pre-CERTS).
2. Alternatively, compute the deposit delta for subtransactions separately and subtract it from the top-level deposit calculation to avoid double-counting.
3. Add an explicit invariant check (not just a PostCondition assertion) that rejects the transaction if `utxosDeposited` would exceed `obligations` after the transition.
