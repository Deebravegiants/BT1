Audit Report

## Title
`instantWithdrawal()` Uses Unsynced `queuedWithdrawalsBuffer` Instead of `assetsCommitted`, Enabling Drainage of Assets Reserved for Queued Withdrawal Users - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.instantWithdrawal()` guards against over-withdrawal using `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal()`, which subtracts a manually-set `queuedWithdrawalsBuffer` from the vault balance. This buffer defaults to zero and is never automatically synchronized with `assetsCommitted` — the accounting variable that tracks assets already promised to queued withdrawal users. When the buffer is zero (the default), any rsETH holder can drain the entire `LRTUnstakingVault`, consuming assets that are already committed to queued withdrawal users and leaving those users unable to complete their withdrawals.

## Finding Description
The protocol has two withdrawal paths. In the queued path, `initiateWithdrawal()` increments `assetsCommitted[asset]` at [1](#0-0)  and the corresponding assets sit in `LRTUnstakingVault` until the operator calls `unlockQueue()`, which pulls them via `unstakingVault.redeem()` at [2](#0-1) .

In the instant path, the availability guard is: [3](#0-2) 

`getAssetsAvailableForInstantWithdrawal()` computes: [4](#0-3) 

`queuedWithdrawalsBuffer` is a mapping that defaults to zero for every asset: [5](#0-4) 

It is only updated when the operator explicitly calls `setQueuedWithdrawalsBuffer()`: [6](#0-5) 

`assetsCommitted` and `queuedWithdrawalsBuffer` are entirely independent — there is no code path that synchronizes them. When `queuedWithdrawalsBuffer` is zero, `getAssetsAvailableForInstantWithdrawal()` returns the full vault balance, even if that entire balance is already committed to queued withdrawal users. The queued path correctly guards against over-commitment via `getAvailableAssetAmount()` at [7](#0-6) , but the instant path does not use `assetsCommitted` at all.

Additionally, `instantWithdrawal()` burns the caller's rsETH **before** performing the availability check: [8](#0-7) 

## Impact Explanation
When `queuedWithdrawalsBuffer` is zero (the default for all assets), an attacker holding rsETH can call `instantWithdrawal()` to drain the entire `LRTUnstakingVault` balance for a given asset. Assets already promised to queued withdrawal users (tracked by `assetsCommitted`) are consumed. When the operator subsequently calls `unlockQueue()`, `_createUnlockParams()` reads `unstakingVault.balanceOf(asset)` as zero at [9](#0-8) , causing the revert at [10](#0-9) . Queued withdrawal users' rsETH is already held in `LRTWithdrawalManager` (transferred during `initiateWithdrawal()`) and cannot be recovered until the vault is replenished.

**Impact: Medium — Temporary (potentially permanent) freezing of funds for queued withdrawal users.**

## Likelihood Explanation
- `queuedWithdrawalsBuffer` defaults to zero for every asset. No operator action is required to create the vulnerable state; it is the initial state.
- The operator must have enabled instant withdrawals via `setInstantWithdrawalEnabled()` and assets must be present in the vault (having been unstaked from EigenLayer), both of which are normal operational conditions.
- Any rsETH holder can call `instantWithdrawal()` permissionlessly once `isInstantWithdrawalEnabled[asset]` is true.
- The attack window is open for the entire period between assets arriving in the vault and `unlockQueue()` being called — no precise front-running is required.

**Likelihood: Medium.**

## Recommendation
Replace the `queuedWithdrawalsBuffer`-based check in `instantWithdrawal()` with a check that accounts for `assetsCommitted`. Expose `assetsCommitted[asset]` from `LRTWithdrawalManager` and use it inside `getAssetsAvailableForInstantWithdrawal()`, or perform the check directly in `instantWithdrawal()`:

```solidity
uint256 vaultBalance = unstakingVault.balanceOf(asset);
uint256 committed = assetsCommitted[asset];
uint256 available = committed >= vaultBalance ? 0 : vaultBalance - committed;
if (assetAmountUnlocked > available) revert CantInstantWithdrawMoreThanAvailable();
```

This mirrors the pattern used in `initiateWithdrawal()` and ensures assets committed to queued withdrawal users are never consumed by instant withdrawals.

## Proof of Concept
1. Alice calls `initiateWithdrawal(stETH, rsETHAmount, "")`. Her rsETH is transferred to `LRTWithdrawalManager`. `assetsCommitted[stETH] = 10e18`. The corresponding stETH is in `LRTUnstakingVault` (having been unstaked from EigenLayer by the operator).
2. `queuedWithdrawalsBuffer[stETH]` is 0 (default, never set by operator). `isInstantWithdrawalEnabled[stETH]` is true.
3. Bob calls `instantWithdrawal(stETH, rsETHEquivalentOf10stETH, "")`.
   - `getAssetsAvailableForInstantWithdrawal(stETH)` returns `10e18 - 0 = 10e18`. Check passes.
   - `unstakingVault.redeem(stETH, 10e18)` drains the vault.
   - Bob receives 10 stETH (minus fee).
4. Operator calls `unlockQueue(stETH, ...)`. `params.totalAvailableAssets = unstakingVault.balanceOf(stETH) = 0`. Reverts at `if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero()`.
5. Alice calls `completeWithdrawal(stETH, "")`. Her request is still locked (never unlocked). Her rsETH remains locked in `LRTWithdrawalManager` indefinitely.

**Foundry test plan:** Deploy contracts with stETH as a supported asset. Enable instant withdrawals. Seed `LRTUnstakingVault` with 10e18 stETH. Have Alice call `initiateWithdrawal` (asserting `assetsCommitted[stETH] == 10e18`). Have Bob call `instantWithdrawal` with equivalent rsETH. Assert vault balance is 0. Assert `unlockQueue` reverts. Assert Alice cannot complete her withdrawal.
