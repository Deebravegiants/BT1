Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch comparison, checking only whether the pre-existing total already exceeds the cap. Every LST asset correctly includes `+ amount`. As a result, any unprivileged caller can deposit an arbitrary amount of ETH past the configured cap in a single transaction, rendering the ETH deposit ceiling completely unenforceable.

## Finding Description
The guard function at L676ŌĆō682 contains an asymmetric comparison:

```solidity
// contracts/LRTDepositPool.sol L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ŌåÉ amount NOT added
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ŌåÉ correct for LSTs
}
``` [1](#0-0) 

For ETH, the function returns `true` (triggering revert) only when `totalAssetDeposits` **already** exceeds the limit before the new deposit is counted. The incoming `amount` is never factored in. For every LST the analogous line correctly uses `totalAssetDeposits + amount`.

This check is the sole cap enforcement called from `_beforeDeposit`: [2](#0-1) 

Which is invoked by the public, permissionless `depositETH` entry point: [3](#0-2) 

**Exploit scenario:**
- `depositLimitByAsset(ETH)` = 100 ETH
- `totalAssetDeposits(ETH)` before tx = 99 ETH
- Attacker calls `depositETH{value: 10_000 ether}(...)`
- Check evaluates: `99 > 100` ŌåÆ `false` ŌåÆ no revert ŌåÆ 10,000 ETH accepted and rsETH minted
- Post-deposit total: 10,099 ETH ŌĆö 100├Ś the configured cap

The same 10,000 ETH deposit in any LST would correctly revert at the same check.

## Impact Explanation
The ETH deposit cap is the protocol's primary TVL risk-management control for ETH. Bypassing it allows unbounded rsETH minting beyond the intended ceiling. Excess ETH cannot be immediately deployed to EigenLayer strategies, sitting idle in the pool and diluting yield for existing rsETH holders. The protocol fails to deliver its promised return characteristics (yield backed by restaked ETH) for the excess deposits. This matches the allowed impact: **Low ŌĆö Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The entry path (`depositETH`) is public, requires no special role, and has no preconditions beyond having sufficient ETH. The bug is deterministic and reproducible on every call where `totalAssetDeposits Ōēż depositLimit`. No front-running, governance capture, or external dependency is required. Likelihood: **High**.

## Recommendation
Add `+ amount` to the ETH branch to match the LST branch:

```diff
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
-       return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+       return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Foundry test sequence:
1. Deploy/fork with `depositLimitByAsset(ETH_TOKEN)` set to 100 ether.
2. Seed the pool so `getTotalAssetDeposits(ETH_TOKEN)` returns 99 ether.
3. Call `depositETH{value: 10_000 ether}(0, "")` from an unprivileged EOA.
4. Assert the call does **not** revert (confirming the bypass).
5. Assert `getTotalAssetDeposits(ETH_TOKEN)` now returns 10,099 ether (> cap).
6. Repeat step 3 with a supported LST via `depositAsset` to confirm it correctly reverts with `MaximumDepositLimitReached`, demonstrating the asymmetry.
