### Title
`ForeignChain::Ton` (and `Solana`, `Ethereum`) Registered On-Chain but Node Execution Path Permanently Bails — (`File: crates/node/src/providers/verify_foreign_tx/sign.rs`)

### Summary

The NEAR MPC contract's `ForeignChain` enum and `ForeignChainRpcRequest` type include `Ton`, `Solana`, and `Ethereum` as first-class, fully-typed chain variants. The contract's `verify_foreign_transaction` entry point accepts requests for any chain that has been registered in the on-chain supported-chains set — including these three. However, the node's `execute_foreign_chain_request` function explicitly `bail!`s for all three, and no inspector implementation exists for any of them. Any `verify_foreign_transaction` request targeting these chains will be accepted and queued on-chain, but will **never** be fulfilled by the MPC network, permanently breaking the request lifecycle for those chains.

---

### Finding Description

**Spec-intent gap — missing node-side execution path for registered chain types.**

The `ForeignChain` enum defines twelve chains, including `Ton` (variant 10), `Solana` (variant 0), and `Ethereum` (variant 2): [1](#0-0) 

All twelve variants are also present in `ForeignChainRpcRequest`, each with a fully-typed request struct (`TonRpcRequest`, `SolanaRpcRequest`, `EvmRpcRequest`): [2](#0-1) 

The on-chain `verify_foreign_transaction` function accepts a request for any chain that has been added to the contract's supported-chains set. It performs no node-capability check: [3](#0-2) 

On the node side, `execute_foreign_chain_request` is the single dispatch point that maps a `ForeignChainRpcRequest` to an inspector call. For `Ton`, `Solana`, and `Ethereum` it unconditionally bails: [4](#0-3) [5](#0-4) 

The `ForeignChainInspectors` struct has no `ethereum`, `solana`, or `ton` field, and the `foreign-chain-inspector` crate has no module for any of these chains: [6](#0-5) [7](#0-6) 

The node-side `mpc_client.rs` loop picks up every queued `verify_foreign_tx` attempt and calls `make_verify_foreign_tx_leader`, which calls `execute_foreign_chain_request`. For Ton/Solana/Ethereum this always returns an error, so the leader task fails, no `respond_verify_foreign_tx` transaction is ever submitted, and the on-chain yield never resumes: [8](#0-7) 

---

### Impact Explanation

Any `verify_foreign_transaction` request targeting `Ton`, `Solana`, or `Ethereum` is permanently stuck. The on-chain yield is queued, the deposit is locked, and the MPC network will never deliver a response. The request lifecycle invariant — "a valid, accepted request will eventually receive a response" — is broken for these three chains. This is a **Medium** impact: request-lifecycle and contract execution-flow invariant violation that does not require network-level DoS or privileged access.

---

### Likelihood Explanation

`Ton` and `Solana` are prominent chains with active bridge use cases. The contract's ABI explicitly exposes them as valid `ForeignChain` values. An operator following the contract's ABI to register Ton or Solana as a supported chain (a normal governance action) would unknowingly enable a path that permanently fails at the node level. Any user who then submits a `verify_foreign_transaction` request for those chains — a call the contract accepts without error — will have their request permanently unserviced. The gap is invisible at the contract layer and only manifests at runtime in the node.

---

### Recommendation

1. **Short-term:** Add a contract-level guard in `verify_foreign_transaction` that rejects requests for chains that have no node-side inspector implementation (e.g., a separate `node_supported_chains` set, or an explicit allowlist checked before queuing the yield).
2. **Long-term:** Implement Ton, Solana, and Ethereum inspectors in `ForeignChainInspectors` and `execute_foreign_chain_request`, or remove these variants from `ForeignChain` / `ForeignChainRpcRequest` until they are implemented, to eliminate the spec-intent gap.

---

### Proof of Concept

1. Operator calls `register_foreign_chain_configuration` with `ForeignChain::Ton` and a valid RPC provider entry (accepted by the contract with no error).
2. User calls `verify_foreign_transaction` with a `ForeignChainRpcRequest::Ton(...)` request targeting that domain. The contract passes the `supported_chains.contains(&requested_chain)` check and enqueues a yield.
3. Every MPC node's leader loop picks up the request, calls `execute_foreign_chain_request`, hits `bail!("ForeignChainRpcRequest::Ton is unsupported")` at `sign.rs:309`, and the task fails.
4. No `respond_verify_foreign_tx` is ever submitted. The yield never resumes. The request is permanently stuck until the NEAR yield timeout expires, at which point the user receives no response and the deposit is consumed.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
