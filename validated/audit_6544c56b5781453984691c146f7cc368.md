### Title
Attached NEAR Deposit Not Refunded to Existing Participants on Re-submission in `submit_participant_info` - (File: crates/contract/src/lib.rs)

### Summary

The `submit_participant_info` function is marked `#[payable]` and accepts NEAR deposits, but contains a conditional refund path that is entirely skipped when an existing participant (voter) re-submits their attestation. Any NEAR attached in that code path is silently absorbed by the contract with no refund.

### Finding Description

`submit_participant_info` computes a boolean flag to decide whether to charge the caller for storage:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;
```

The entire deposit-accounting block — including the refund — is gated behind this flag:

```rust
if attestation_storage_must_be_paid_by_caller {
    // ... cost calculation ...
    // Refund the difference if the proposer attached more than required
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
```

When `attestation_storage_must_be_paid_by_caller` is `false` — i.e., the caller **is** an existing participant **and** the submission is an update, not a new insertion — the `if` block is skipped entirely. `env::attached_deposit()` is never read, and no refund promise is ever scheduled. Any NEAR the caller attached is permanently absorbed by the contract.

The first time a participant submits, they must attach NEAR to cover storage costs. On subsequent re-submissions (e.g., to refresh an expiring attestation or update a TLS key), the function is still `#[payable]`, giving no indication that a deposit is now unnecessary. A participant following the same invocation pattern as their initial submission will silently lose whatever NEAR they attach.

This is structurally identical to the reference report: in both cases a function accepts funds, performs an operation, and fails to return the residual — in the reference report because the wrong token type is measured; here because the refund branch is entirely absent for a reachable code path.

### Impact Explanation

Any NEAR attached by an existing participant during a re-submission is permanently locked in the MPC contract. There is no recovery mechanism for the participant. This breaks the accounting invariant that a `#[payable]` function must either consume a deposit for a legitimate purpose or refund it.

**Impact: Medium** — balance/accounting invariant violation causing direct, permanent loss of NEAR tokens for MPC participants, matching the allowed scope: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

MPC nodes re-submit their attestation periodically (e.g., before expiry, after a TLS key rotation, or after a TEE upgrade). The function signature is `#[payable]` with no documentation distinguishing first-submission from re-submission deposit requirements. A node operator following the same CLI invocation or SDK call used for the initial submission will attach NEAR and lose it silently. No privileged access beyond being an existing participant is required.

### Recommendation

Add an unconditional refund of any excess deposit at the end of `submit_participant_info`, mirroring the pattern used in `require_deposit` and `propose_update`:

```rust
// After the conditional storage-charge block:
let attached = env::attached_deposit();
if attached > NearToken::from_yoctonear(0) {
    Promise::new(account_id).transfer(attached).detach();
}
```

Alternatively, restructure the function so that `env::attached_deposit()` is always read and any amount not consumed for storage is always refunded, regardless of whether `attestation_storage_must_be_paid_by_caller` is true or false.

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` for the first time, attaching 1 mNEAR to cover storage. `is_new_attestation = true`, so `attestation_storage_must_be_paid_by_caller = true`. The storage cost is deducted and the remainder is refunded correctly.

2. Alice's attestation approaches expiry. She calls `submit_participant_info` again with the same 1 mNEAR attached (same invocation pattern). Now `is_new_attestation = false` and `caller_is_not_participant = false`, so `attestation_storage_must_be_paid_by_caller = false`.

3. The entire `if attestation_storage_must_be_paid_by_caller { ... }` block is skipped. `env::attached_deposit()` is never called inside it. No refund promise is scheduled.

4. The function returns `Ok(())`. Alice's 1 mNEAR is permanently absorbed by the contract with no log, no error, and no refund. [1](#0-0) [2](#0-1) [3](#0-2)
