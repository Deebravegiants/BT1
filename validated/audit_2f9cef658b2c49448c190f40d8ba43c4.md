Audit Report

## Title
Zero-Rate Broadcast via Unguarded `updateRate()` Before First Oracle Initialization — (`contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary

`MultiChainRateProvider.updateRate()` has no access control and no zero-rate guard. `LRTOracle.rsETHPrice` is a plain `uint256` that defaults to `0` and is not set during `initialize()`. Any caller can invoke `updateRate()` before the first `updateRSETHPrice()` call, broadcasting a zero rate via LayerZero to all registered `CrossChainRateReceiver` contracts, which causes division-by-zero reverts in `RSETHPoolV2.deposit()` and `RSETHPoolV3.deposit()` on destination chains.

## Finding Description

**No access control on `updateRate()`:**

`MultiChainRateProvider.updateRate()` is `external payable nonReentrant` with no role check, allowing any caller to trigger a cross-chain rate broadcast at any time. [1](#0-0) 

**Rate source defaults to zero:**

`RSETHMultiChainRateProvider.getLatestRate()` reads directly from `ILRTOracle(rsETHPriceOracle).rsETHPrice()`. [2](#0-1) 

`rsETHPrice` is a plain `uint256` storage variable in `LRTOracle`, initialized to `0` by EVM default. [3](#0-2) 

`LRTOracle.initialize()` does **not** call `_updateRsETHPrice()`, so `rsETHPrice` remains `0` until an explicit `updateRSETHPrice()` call. [4](#0-3) 

**No zero-rate guard before broadcast:**

`updateRate()` encodes and sends whatever `getLatestRate()` returns — including `0` — with no validation before the LayerZero send. [5](#0-4) 

**Receiver stores zero without guard:**

`CrossChainRateReceiver.lzReceive()` stores the decoded rate unconditionally, including `0`. [6](#0-5) 

**Downstream division by zero:**

`RSETHPoolV2.viewSwapRsETHAmountAndFee()` divides by `rsETHToETHrate` with no zero guard, called directly by `deposit()`. [7](#0-6) 

`RSETHPoolV3.viewSwapRsETHAmountAndFee()` has the identical unguarded pattern. [8](#0-7) 

Note: `RSETHPoolV3.viewSwapAssetToPremintedRsETH()` does include a zero-rate guard (`if (rsETHToETHrate == 0) revert UnsupportedOracle()`), but the primary `deposit()` path through `viewSwapRsETHAmountAndFee()` does not. [9](#0-8) 

**Exploit path:**
1. `LRTOracle` and `RSETHMultiChainRateProvider` are deployed; `rsETHPrice == 0`.
2. Owner adds one or more rate receivers via `addRateReceiver()`.
3. Before `updateRSETHPrice()` is called, an unprivileged attacker calls `updateRate{value: fee}()`.
4. `getLatestRate()` returns `0`; the zero rate is encoded and sent via LayerZero to all receivers.
5. Each `CrossChainRateReceiver` stores `rate = 0`.
6. Any user calling `deposit()` on `RSETHPoolV2` or `RSETHPoolV3` triggers `viewSwapRsETHAmountAndFee()`, which executes `amountAfterFee * 1e18 / 0`, reverting with a division-by-zero panic.

## Impact Explanation

All `deposit()` calls on destination-chain `RSETHPoolV2`/`RSETHPoolV3` contracts revert via division-by-zero panic for as long as the zero rate persists. No user funds are lost; the state is recoverable by calling `updateRSETHPrice()` then `updateRate()` with a valid rate. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation

The vulnerable window is the deployment phase: after `RSETHMultiChainRateProvider` is deployed and receivers are added, but before the first `updateRSETHPrice()` call. The attack is permissionless, requires only ETH for LayerZero fees, and can be executed by monitoring the mempool for the deployment transaction or the `RateReceiverAdded` event. The attack is also repeatable during any future redeployment or migration where the provider is reconfigured before the oracle is primed.

## Recommendation

Add a zero-rate guard at the top of `updateRate()`:

```solidity
function updateRate() external payable nonReentrant {
    uint256 latestRate = getLatestRate();
    require(latestRate != 0, "MultiChainRateProvider: zero rate");
    rate = latestRate;
    // ... rest of function
}
```

Alternatively, restrict `updateRate()` to an authorized keeper role, or ensure `updateRSETHPrice()` is called atomically during deployment before any receivers are registered.

## Proof of Concept

```solidity
function testZeroRateBroadcast() public {
    // 1. Deploy mock LRTOracle whose rsETHPrice() returns 0 (default state)
    MockLRTOracle oracle = new MockLRTOracle(); // rsETHPrice = 0

    // 2. Deploy RSETHMultiChainRateProvider pointing to mock oracle
    RSETHMultiChainRateProvider provider = new RSETHMultiChainRateProvider(
        address(oracle), address(mockLZEndpoint)
    );

    // 3. Owner adds one receiver (onlyOwner gate passed)
    vm.prank(owner);
    provider.addRateReceiver(101, address(mockReceiver));

    // 4. Unprivileged attacker calls updateRate() — no access control blocks this
    vm.prank(attacker);
    provider.updateRate{value: 0.01 ether}();

    // 5. Provider stored rate is 0
    assertEq(provider.rate(), 0);

    // 6. LZ payload decodes to 0
    bytes memory payload = mockLZEndpoint.lastPayload();
    assertEq(abi.decode(payload, (uint256)), 0);

    // 7. Simulate receiver storing rate = 0 (as lzReceive would)
    mockReceiver.setRate(0);

    // 8. Any deposit on RSETHPoolV2/V3 reverts — division by zero
    vm.expectRevert(); // Panic: division by zero
    pool.deposit{value: 1 ether}("ref");
}
```
