Audit Report

## Title
Permissionless `sendFunds()` With No Caller Incentive Enables Yield Dilution of Existing rsETH Holders — (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` is a permissionless function with no caller reward. MEV and execution-layer rewards accumulate in `FeeReceiver` without being reflected in the protocol's TVL, because `LRTDepositPool.getETHDistributionData()` explicitly excludes the `FeeReceiver` balance. Any deposit made while rewards sit uncollected mints excess rsETH at an understated price, permanently diluting the yield owed to existing rsETH holders.

## Finding Description

`FeeReceiver.sendFunds()` has no access control and no caller incentive:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

`LRTDepositPool.getETHDistributionData()` explicitly excludes `FeeReceiver` from TVL accounting — it only sums `address(this).balance`, NDC balances, EigenLayer stakes, the converter, and the unstaking vault:

```solidity
// contracts/LRTDepositPool.sol L464-466
/// @dev rewards are not accounted here
/// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
``` [2](#0-1) 

`LRTOracle._getTotalEthInProtocol()` computes the rsETH price by calling `getTotalAssetDeposits()` for each supported asset, which routes through `getETHDistributionData()` for ETH: [3](#0-2) 

`LRTDepositPool.getRsETHAmountToMint()` uses the cached `rsETHPrice` to determine how much rsETH to mint:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

**Exploit path:**
1. FeeReceiver passively accumulates MEV rewards (e.g., 10 ETH on a 1,000 ETH TVL base).
2. `updateRSETHPrice()` is called (public, no access control) — price is computed without the 10 ETH, so it is understated at 1.000 ETH/rsETH instead of the fair 1.010 ETH/rsETH.
3. Attacker calls `depositETH{value: 100 ether}(0, "")` — receives `100 / 1.000 = 100 rsETH` instead of the fair `100 / 1.010 ≈ 99.01 rsETH`.
4. `sendFunds()` is called — 10 ETH enters TVL.
5. `updateRSETHPrice()` is called — TVL = 1,110 ETH, supply = 1,100 rsETH, price = 1.009 ETH/rsETH.

Had `sendFunds()` been called before step 3, the fair price would be 1.010 ETH/rsETH and the final price after deposit would remain 1.010 ETH/rsETH. Existing holders lose ≈ 0.001 ETH per rsETH (≈ 0.1% of their accrued yield) to the late depositor. The excess rsETH minted in step 3 is permanent and irreversible.

No existing guard prevents this: `_beforeDeposit()` only checks minimum deposit amount and deposit limits; `minRSETHAmountExpected` protects the depositor, not existing holders; and the `pricePercentageLimit` circuit-breaker only triggers on price increases above a threshold, not on the dilution itself. [5](#0-4) 

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders permanently lose a portion of MEV/execution-layer yield to depositors who enter while rewards are stranded in `FeeReceiver`. The dilution is irreversible once the excess rsETH is minted. The magnitude scales with the ratio of uncollected rewards to TVL and the volume of deposits during the delay window.

## Likelihood Explanation
**Low.** The protocol team is expected to call `sendFunds()` periodically off-chain. However, there is no on-chain enforcement, no keeper incentive, and no minimum call frequency. During periods of high MEV (e.g., around major on-chain events), rewards can accumulate rapidly. A sophisticated depositor who monitors `FeeReceiver.balance` can time a large deposit to exploit the window before `sendFunds()` is called. The attack requires no special privileges and is repeatable every accumulation cycle.

## Recommendation
1. **Auto-flush on deposit:** Call `sendFunds()` (or an equivalent internal flush) at the start of `LRTDepositPool.depositETH()` / `depositAsset()` so rewards are always counted before new rsETH is minted.
2. **Caller incentive:** Reward the `sendFunds()` caller with a small fixed percentage (e.g., 0.1–0.5%) of the forwarded balance, analogous to liquidation bonuses in lending protocols, to create on-chain incentive for timely calls.
3. **Keeper automation:** Deploy a Gelato/Chainlink Automation keeper that calls `sendFunds()` whenever `FeeReceiver.balance` exceeds a configurable threshold, funded by a portion of the rewards.

## Proof of Concept
Foundry fork test outline:

```solidity
function test_yieldDilution() public {
    // 1. Simulate 10 ETH MEV rewards arriving in FeeReceiver
    vm.deal(address(feeReceiver), 10 ether);

    // 2. Update rsETH price (excludes FeeReceiver balance)
    lrtOracle.updateRSETHPrice();
    uint256 understatedPrice = lrtOracle.rsETHPrice();
    // understatedPrice == 1.000e18 (10 ETH not counted)

    // 3. Attacker deposits 100 ETH at understated price
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 100 ether}(0, "");
    uint256 attackerRsETH = rsETH.balanceOf(attacker);
    // attackerRsETH == 100e18 (should be ~99.01e18)

    // 4. sendFunds() moves 10 ETH into TVL
    feeReceiver.sendFunds();

    // 5. Update price — now reflects 1110 ETH / 1100 rsETH
    lrtOracle.updateRSETHPrice();
    uint256 finalPrice = lrtOracle.rsETHPrice();
    // finalPrice ≈ 1.009e18 instead of fair 1.010e18

    // 6. Assert existing holders are diluted
    uint256 existingHolderValue = existingHolderRsETH * finalPrice / 1e18;
    assertLt(existingHolderValue, existingHolderRsETH * 1.010e18 / 1e18);
}
```
