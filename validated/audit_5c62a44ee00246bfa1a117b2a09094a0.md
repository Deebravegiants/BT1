### Title
Single Byzantine Participant Can Permanently Block All Foreign-Chain Verification Requests via `register_foreign_chain_support` - (File: crates/contract/src/lib.rs)

### Summary
`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which computes the **strict intersection** of every active participant's registered chain sets. A single Byzantine participant below the signing threshold can call `register_foreign_chain_support` with an empty set, collapsing the intersection to the empty set and permanently rejecting every subsequent `verify_foreign_transaction` call until the participant re-registers or is reshared out.

### Finding Description
`verify_foreign_transaction` checks whether the requested chain is in the supported set before enqueuing the yield-resume request: [1](#0-0) 

`get_supported_foreign_chains()` computes the intersection of all active participants' registered chain sets: [2](#0-1) 

The intersection requires **every** active participant to have registered support for a chain. Any participant can call `register_foreign_chain_support` (gated only by `voter_or_panic()`, i.e., being a current participant) and supply an empty `SupportedForeignChains`: [3](#0-2) 

Submitting an empty set causes `get_supported_foreign_chains()` to return an empty set, because no chain can be a superset of all active participants when one participant contributes nothing. The project's own design document explicitly acknowledges this root cause:

> "Today, `get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains, and `verify_foreign_transaction` rejects any request whose target chain is not in it. A single node that registers an empty list (or hasn't registered yet) drops **every** chain ŌĆö one operator can take the whole feature down."



The new `get_available_foreign_chains()` / threshold-based path exists in the contract but `verify_foreign_transaction` has **not been migrated** to use it; it still calls the legacy intersection function: [4](#0-3) 

### Impact Explanation
Every `verify_foreign_transaction` call for every chain is rejected with `ForeignChainNotSupported` for as long as the malicious participant's empty registration persists. The entire cross-chain bridge verification feature is rendered non-functional. Users who depend on the bridge to move assets across chains cannot complete their operations. This is a contract execution-flow manipulation that breaks a production safety/accounting invariant (the availability of the foreign-chain verification path) without requiring network-level DoS or operator misconfiguration. Impact: **Medium** ŌĆö request-lifecycle and contract execution-flow manipulation that breaks production safety invariants.

### Likelihood Explanation
Any single active participant can execute this attack with one on-chain call. No collusion, no leaked key, and no threshold of participants is required ŌĆö one participant acting alone is sufficient. The call is permissioned only to current participants, but the MPC system is explicitly designed to tolerate Byzantine participants below the signing threshold for signing operations; this path has no equivalent threshold protection. Likelihood: **High** ŌĆö trivially executable by any single compromised or malicious participant node.

### Recommendation
Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which requires only `signing_threshold` participants to cover a chain rather than unanimous registration. This is the fix already designed and tracked in the project (issue #3434 / `docs/design/calculating-supported-foreign-chains.md`). The migration path is additive and does not require a flag-day upgrade.

### Proof of Concept
1. Network is Running with participants `[P1, P2, P3]`, threshold = 2. All three have registered Bitcoin support via `register_foreign_chain_support`.
2. `get_supported_foreign_chains()` returns `{Bitcoin}`. `verify_foreign_transaction(Bitcoin, ...)` succeeds.
3. `P1` (a single Byzantine participant, below threshold) calls:
   ```
   register_foreign_chain_support({ foreign_chain_support: [] })
   ```
   This is accepted because `P1` is a valid participant (`voter_or_panic()` passes).
4. `node_foreign_chain_support` now maps `P1 ŌåÆ {}`, `P2 ŌåÆ {Bitcoin}`, `P3 ŌåÆ {Bitcoin}`.
5. `get_supported_foreign_chains()` computes: for Bitcoin, supporters = `{P2, P3}`. Is `{P2, P3}` a superset of all active participants `{P1, P2, P3}`? No. Bitcoin is dropped.
6. Every subsequent `verify_foreign_transaction(Bitcoin, ...)` call panics with `ForeignChainNotSupported`.
7. The bridge is fully blocked until `P1` re-registers or a resharing removes `P1` ŌĆö a process that itself requires threshold votes and a full resharing ceremony. [5](#0-4) [1](#0-0)

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }
