Audit Report

## Title
Stale `rsETHPrice` Silently Propagated as Fresh by `RSETHPriceFeed` When Protocol Is Paused — (File: `contracts/oracles/RSETHPriceFeed.sol`)

## Summary
`RSETHPriceFeed.latestRoundData()` computes rsETH/USD by multiplying a live Chainlink ETH/USD answer by `LRTOracle.rsETHPrice()`, but returns only the ETH/USD `updatedAt` timestamp. When `LRTOracle._updateRsETHPrice()` detects a price drop exceeding `pricePercentageLimit`, it calls `_pause()` and returns early without writing the new (lower) price to `rsETHPrice`. The public `updateRSETHPrice()` is then blocked by `whenNotPaused`. During the pause window, `RSETHPriceFeed` serves an inflated pre-drop rsETH/USD price paired with a fresh ETH/USD `updatedAt`, enabling over-borrowing against rsETH collateral in any integrated lending protocol.

## Finding Description

**Root cause 1 — `RSETHPriceFeed` never tracks rsETH price staleness.**

`contracts/oracles/RSETHPriceFeed.sol` lines 63–70:
```solidity
function latestRoundData() external view returns (...) {
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
}
```
`updatedAt` is taken verbatim from the ETH/USD Chainlink feed, which updates every few minutes. The rsETH/ETH component — `RS_ETH_ORACLE.rsETHPrice()` — is a stored state variable whose own last-write timestamp is never recorded or surfaced. Any consumer relying on `updatedAt` for staleness detection receives no signal that the rsETH component is stale.

**Root cause 2 — downside protection freezes `rsETHPrice` at the pre-drop value.**

`contracts/LRTOracle.sol` lines 270–282:
```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;   // rsETHPrice is NOT updated
}
```
`newRsETHPrice` (the lower, accurate value) is computed but never written to `rsETHPrice` before the early `return`. The assignment `rsETHPrice = newRsETHPrice` at line 313 is never reached.

**Root cause 3 — public price update is blocked while paused.**

`contracts/LRTOracle.sol` line 87:
```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```
Any unprivileged caller attempting to refresh the price after the pause is reverted. The privileged escape hatch `updateRSETHPriceAsManager()` (line 94) is not gated by `whenNotPaused`, but it requires the `MANAGER` role and depends on timely admin intervention — it does not prevent the exploitation window.

**Exploit flow:**
1. A price drop exceeding `pricePercentageLimit` is detected. `_pause()` fires; `rsETHPrice` stays at the pre-drop (inflated) value.
2. `updateRSETHPrice()` reverts for all unprivileged callers.
3. `RSETHPriceFeed.latestRoundData()` continues to return `answer = inflated_rsETH_price × fresh_ETH_USD` and `updatedAt = ETH_USD_updatedAt` (recent).
4. A lending protocol's staleness check passes (ETH/USD `updatedAt` is fresh).
5. The protocol values rsETH at the inflated price and allows over-borrowing.
6. Attacker deposits rsETH, borrows the maximum, and defaults; the lending protocol is left with under-collateralized debt.

## Impact Explanation
**Critical — Direct theft of user funds from integrated lending protocols.** An attacker can borrow assets worth more than the true value of their rsETH collateral. The difference between the inflated price and the true price is an immediate, unrecoverable loss to lenders. The impact is not hypothetical: `RSETHPriceFeed` is purpose-built as a Chainlink-compatible feed for exactly this integration pattern.

## Likelihood Explanation
**Medium-High.** The trigger condition — a price drop exceeding `pricePercentageLimit` — is an explicitly designed circuit-breaker expected to fire during adverse market events (EigenLayer slashing, LST depeg). If `pricePercentageLimit` is set conservatively, even routine volatility can trigger it. The exploitation window lasts until an admin calls `updateRSETHPriceAsManager()` or unpauses, which may take hours. The attacker needs only to call `RSETHPriceFeed.latestRoundData()` indirectly through a lending protocol's borrow function — no special privileges required.

## Recommendation

1. **Write the new (lower) price before pausing.** In `_updateRsETHPrice()`, move `rsETHPrice = newRsETHPrice` to before the `_pause()` + `return` block so the stored value reflects reality even while the protocol is paused.

2. **Track and surface rsETH price staleness.** Store a `rsETHPriceUpdatedAt` timestamp in `LRTOracle` whenever `rsETHPrice` is written. In `RSETHPriceFeed.latestRoundData()`, return `min(ethUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt`.

3. **Add a staleness guard in `RSETHPriceFeed`.** Revert if `rsETHPriceUpdatedAt` is older than a configurable threshold (e.g., 24 hours), so consumers cannot silently consume a stale composite price.

## Proof of Concept

1. Deploy `LRTOracle` with `pricePercentageLimit = 5e16` (5%).
2. Call `updateRSETHPrice()` to set `rsETHPrice = 1.05e18` and `highestRsethPrice = 1.05e18`.
