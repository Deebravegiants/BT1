### Title
Irreversible Eviction Side-Effect in `try_make_space` Allows Attacker to Permanently Expel Legitimate Transactions from the Mempool at Zero Cost — (`crates/apollo_mempool/src/mempool.rs`)

---

### Summary

When the mempool is at capacity and an incoming transaction triggers `try_make_space`, gap-account transactions are permanently evicted from the pool even if the incoming transaction is ultimately rejected because it is still too large after eviction. The eviction is never rolled back. An unprivileged attacker can exploit this to permanently remove legitimate transactions from the mempool at zero cost (their own transaction is rejected, so no gas is ever charged), satisfying the same "capacity-limit weaponization" invariant as the external M-16 report.

---

### Finding Description

The mempool enforces a byte-capacity limit via `MempoolStaticConfig::capacity_in_bytes` (default 1 GB in production). When `add_tx` is called and the pool is full, `add_tx_validations` calls `handle_capacity_overflow`, which in turn calls `try_make_space` to evict transactions belonging to "gap accounts" (accounts whose lowest pooled nonce exceeds their on-chain account nonce). [1](#0-0) 

`try_make_space` removes transactions from the pool and queue permanently: [2](#0-1) 

If the freed space is still insufficient for the incoming transaction, `handle_capacity_overflow` returns `MempoolError::MempoolFull`: [3](#0-2) 

The evictions performed by `try_make_space` are **never rolled back**. This is explicitly acknowledged in the test suite: [4](#0-3) 

The `MempoolFull` error propagates through the gateway as `StarknetErrorCode::TransactionLimitExceeded`: [5](#0-4) 

**Attack path:**

1. Attacker observes that the mempool is nearly full and that gap-account transactions exist (accounts with nonce gaps are tracked in `accounts_with_gap`).
2. Attacker submits a transaction with a maximally large signature (up to `max_signature_length` felts, validated stateless-only before reaching the mempool).
3. The transaction passes stateless and stateful gateway validation (no gas is charged at this stage).
4. Inside `add_tx_validations`, `exceeds_capacity` returns `true`.
5. `handle_capacity_overflow` determines `creating_gap = false` (attacker's tx nonce equals their account nonce), so `try_make_space` is called.
6. `try_make_space` evicts one or more gap-account transactions, permanently removing them from both `tx_pool` and `tx_queue`.
7. If the attacker's transaction is still too large after eviction, `try_make_space` returns `false` and `handle_capacity_overflow` returns `MempoolError::MempoolFull`.
8. The attacker's transaction is rejected. **No gas is charged.** The evicted legitimate transactions are gone permanently.

The attacker can repeat this indefinitely. Each attempt costs nothing (rejected before execution) but permanently removes victim transactions from the mempool.

---

### Impact Explanation

Valid transactions that were already admitted to the mempool are permanently removed without being sequenced. Their owners must resubmit them (paying submission overhead again) and may miss time-sensitive sequencing windows. This matches the allowed impact: **"Mempool/gateway/RPC admission … rejects valid transactions before sequencing."**

The evicted transactions belong to gap accounts, meaning they are stuck pending a lower-nonce fill. However, they are valid, fee-paying transactions that the user legitimately submitted. Their permanent removal without sequencing is an admission-layer denial of service with no cost to the attacker.

---

### Likelihood Explanation

- The mempool is a shared, bounded resource (1 GB in production). Under normal load it will approach capacity.
- Gap-account transactions are common (any account that submits nonce N+1 before N is committed creates a gap).
- The attacker needs only one rejected transaction per eviction cycle. No privileged access, no miner collusion, and no flash loan is required.
- The stateless validator caps signature length at `max_signature_length` (4 000 felts = ~128 KB per transaction), which is large enough to trigger multi-transaction evictions when the pool is near capacity. [6](#0-5) 

---

### Recommendation

Roll back evictions when the incoming transaction is ultimately rejected. The simplest fix is to collect the evicted transactions in `try_make_space`, and if `handle_capacity_overflow` returns `Err(MempoolError::MempoolFull)`, re-insert them into the pool and queue before propagating the error. Alternatively, perform a dry-run capacity check (without mutating state) before committing any evictions, and only evict if the incoming transaction is guaranteed to fit after eviction.

---

### Proof of Concept

The existing test `rejects_or_accepts_tx_based_on_freed_space` in `crates/apollo_mempool/src/fee_mempool_test.rs` already demonstrates the exact behavior: [7](#0-6) 

The `#[case::insufficient_eviction_space(20, false)]` case shows that when the attacker's signature is 20 felts (too large to fit after eviction), `tx1` and `tx2` are still permanently evicted even though `large_tx` is rejected. The comment on line 1936–1937 explicitly confirms the irreversibility is intentional but unguarded against adversarial use.

    pub fn try_make_space(&mut self, required_space: u64) -> bool {
        let mut total_space_freed = 0;
        let mut evicted_txs = Vec::new();

        while total_space_freed < required_space {
            let Some(address) = self.get_evictable_account() else {
                break;
            };

            let txs: Vec<_> = self.tx_pool.account_txs_sorted_by_nonce(address).copied().collect();
            for tx_ref in txs.iter().rev() {
                let tx = self
                    .tx_pool
                    .remove(tx_ref.tx_hash)
                    .expect("Transaction must exist in the pool.");
                total_space_freed += tx.total_bytes();
                evicted_txs.push(*tx_ref);
                metric_count_evicted_txs(1);
                self.decrement_stuck_txs_if_gap_account(address, 1);
                if total_space_freed >= required_space {
                    break;
                }
            }

            // Clean up if account is now empty.
            if !self.tx_pool.contains_account(address) {
                self.accounts_with_gap.swap_remove(&address);
            }
        }

        // Keep the queue consistent with the pool: the evicted txs were removed from the pool, so
        // drop their (now-orphaned) queue references too. In fee mode gap accounts are never
        // queued, so this is a no-op there; in Echonet/FIFO mode they are, and skipping this leaves
        // a dangling reference that panics the next `get_txs`.
        self.tx_queue.remove_txs(&evicted_txs);

        total_space_freed >= required_space

#[case::insufficient_eviction_space(20, false)]
#[case::sufficient_eviction_space(8, true)]
fn rejects_or_accepts_tx_based_on_freed_space(
    #[case] signature_size: usize,
    #[case] expect_success: bool,
) {
    let tx1 = add_tx_input!(tx_hash: 1, address: "0x1", tx_nonce: 1, account_nonce: 0);
    let tx2 = add_tx_input!(tx_hash: 2, address: "0x1", tx_nonce: 2, account_nonce: 0);

    let large_signature = vec![felt!("0x0"); signature_size];
    let large_tx = AddTransactionArgs {
        tx: internal_invoke_tx(invoke_tx_args!(
            tx_hash: tx_hash!(3),
            signature: TransactionSignature(large_signature.into())
        )),
        account_state: AccountState { address: contract_address!("0x0"), nonce: nonce!(0) },
    };

    let capacity = tx1.tx.total_bytes() + tx2.tx.total_bytes();
    let mut mempool = Mempool::new(
        MempoolConfig {
            static_config: MempoolStaticConfig {
                capacity_in_bytes: capacity,
                ..Default::default()
            },
            ..Default::default()
        },
        Arc::new(FakeClock::default()),
    );

    for tx in [&tx1, &tx2] {
        add_tx(&mut mempool, tx);
    }

    if expect_success {
        add_tx(&mut mempool, &large_tx);

    }

    // Transactions tx1 and tx2 are evicted regardless of whether large_tx is accepted or rejected.
    // We do not revert the eviction attempt even if adding large_tx ultimately fails.
    assert!(!mempool.tx_pool.contains_account(contract_address!("0x1")));
