### Title
Script Credential Withdrawal Bypass of DRep Delegation Requirement — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

The `validateWithdrawalsDelegated` function in the Conway `LEDGER` rule applies the DRep delegation check exclusively to `KeyHashObj` (key-hash) stake credentials. `ScriptHashObj` (script) credentials are silently excluded via the `credKeyHash` filter. An unprivileged transaction author who controls a script-based stake credential can accumulate rewards and withdraw them without ever delegating to a DRep, bypassing the post-bootstrap governance participation requirement that applies to all other withdrawal addresses.

---

### Finding Description

The vulnerability class from the external report is **"wrong entity being checked"**: a limit or condition is enforced on one entity (the receiver) while the actual actor (the depositor) is never checked. The analog here is that the DRep delegation condition is enforced on `KeyHashObj` withdrawal addresses but never on `ScriptHashObj` withdrawal addresses — the wrong subset of entities is checked.

The root cause is in `validateWithdrawalsDelegated`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs, lines 478–488
validateWithdrawalsDelegated accounts tx =
  let wdrls = unWithdrawals $ tx ^. bodyTxL . withdrawalsTxBodyL
      wdrlsKeyHashes =
        [ kh | (ra, _) <- Map.toList wdrls
             , Just kh <- [credKeyHash $ ra ^. accountAddressCredentialL]
        ]                                    -- ❌ ScriptHashObj yields Nothing → silently dropped
      isNotDRepDelegated keyHash = isNothing $ do
        accountState <- lookupAccountState (KeyHashObj keyHash) accounts
        accountState ^. dRepDelegationAccountStateL
      nonExistentDelegations = filter isNotDRepDelegated wdrlsKeyHashes
   in failureOnNonEmpty nonExistentDelegations ConwayWdrlNotDelegatedToDRep
```

`credKeyHash` returns `Nothing` for any `ScriptHashObj` credential. The list comprehension therefore produces an empty list for script-based withdrawal addresses, so `nonExistentDelegations` is always empty for them, and `ConwayWdrlNotDelegatedToDRep` is never raised.

The same function is re-used verbatim in the Dijkstra era's `ENTITIES` and `SUBENTITIES` rules:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs, line 197
runTest $ Conway.validateWithdrawalsDelegated accounts tx
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubEntities.hs, line 163
runTest $ Conway.validateWithdrawalsDelegated accounts tx
```

The bypass therefore propagates to every era that inherits this rule.

Critically, script credentials are **capable** of holding a DRep delegation — the `dRepDelegationAccountStateL` field exists in every `AccountState` regardless of credential type. The omission is therefore not a structural impossibility; it is a missing check.

---

### Impact Explanation

**Medium.** The Conway era's post-bootstrap design requires that any key-hash withdrawal address be delegated to a DRep before rewards can be withdrawn (`ConwayWdrlNotDelegatedToDRep`). This is a protocol-level parameter governing when withdrawals are permitted. A script-based stake credential can withdraw its full accumulated reward balance without satisfying this condition, executing withdrawals outside the intended design parameters. The ledger state transition is accepted by honest nodes even though the governance participation precondition is unmet, constituting a deterministic rule evaluation discrepancy between the intended and implemented behaviour.

---

### Likelihood Explanation

Any unprivileged transaction author can exploit this:

1. Deploy any script (e.g., `alwaysSucceeds`) and derive a `ScriptHashObj` stake credential from its hash.
2. Register the credential with a `RegDepositTxCert`.
3. Accumulate rewards (e.g., via governance action expiry, as shown in `submitAndExpireProposalToMakeReward`).
4. Submit a withdrawal transaction. The script witness satisfies the UTXOW spending check; `validateWithdrawalsDelegated` never inspects the credential because `credKeyHash` returns `Nothing`.

No governance majority, privileged key, or Sybil attack is required. The entry path is a standard, attacker-authored transaction.

---

### Recommendation

Extend the DRep delegation check to cover `ScriptHashObj` credentials. Replace the `credKeyHash` filter with a check over all credential types:

```haskell
validateWithdrawalsDelegated accounts tx =
  let wdrls = unWithdrawals $ tx ^. bodyTxL . withdrawalsTxBodyL
      wdrlsCreds = [ cred | (ra, _) <- Map.toList wdrls
                          , let cred = ra ^. accountAddressCredentialL ]
      isNotDRepDelegated cred = isNothing $ do
        accountState <- lookupAccountState cred accounts
        accountState ^. dRepDelegationAccountStateL
      nonExistentDelegations = filter isNotDRepDelegated wdrlsCreds
   in failureOnNonEmpty nonExistentDelegations ConwayWdrlNotDelegatedToDRep
```

The same fix must be applied wherever `Conway.validateWithdrawalsDelegated` is called in the Dijkstra `ENTITIES` and `SUBENTITIES` rules.

5. passNEpochs (lifetime + 2)                        -- reward credited to cred's account
6. Submit withdrawal tx:

   → ConwayWdrlNotDelegatedToDRep is never raised
