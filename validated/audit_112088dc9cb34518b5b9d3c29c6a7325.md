Audit Report

## Title
Missing Chainlink Staleness Check Enables Over-Minting of rsETH via Stale High Asset Price — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all freshness fields, using only the raw `price` value. When a Chainlink feed goes stale at a price above the true market value, `LRTDepositPool.getRsETHAmountToMint()` mints rsETH proportional to the inflated stale price while `rsETHPrice` still reflects the last correctly-computed stored value. The attacker receives more rsETH than their collateral is worth, diluting all existing rsETH holders and reducing the protocol's collateral ratio below 1.0.

## Finding Description

**Root cause — `ChainlinkPriceOracle.getAssetPrice()`:** [1](#0-0) 

All five return values of `latestRoundData()` are available in the interface definition (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`), but only `answer` is captured. There is no check on `updatedAt` (heartbeat staleness) or `answeredInRound < roundId` (incomplete round). [2](#0-1) 

**Contrast with sibling oracle — `ChainlinkOracleForRSETHPoolCollateral.getRate()`:** [3](#0-2) 

The same codebase already implements `answeredInRound < roundID` and `timestamp == 0` guards in this sibling contract, confirming the protocol is aware of the staleness risk and has chosen to handle it selectively — but not in the primary deposit oracle path.

**Minting formula — `LRTDepositPool.getRsETHAmountToMint()`:** [4](#0-3) 

`lrtOracle.getAssetPrice(asset)` is a **live Chainlink call** (no cache). `lrtOracle.rsETHPrice()` is a **stored value** updated only when `updateRSETHPrice()` is called. This temporal mismatch is the exploit surface: a stale high price in the numerator against a stale-but-correct denominator yields excess rsETH.

**`rsETHPrice` update path — `LRTOracle._getTotalEthInProtocol()`:** [5](#0-4) 

Both the numerator (`getAssetPrice`) and the denominator (`rsETHPrice`) ultimately depend on the same Chainlink feed, but they are evaluated at **different times** — the numerator at deposit time, the denominator at the last `updateRSETHPrice()` call.

**Downside guard is insufficient:** [6](#0-5) 

The `pricePercentageLimit` guard only triggers on `updateRSETHPrice()` calls after the price corrects downward. It does not prevent the initial over-minting, and its default value is `0` (disabled unless explicitly configured by admin).

## Impact Explanation

An attacker who deposits during the staleness window receives rsETH representing a claim on more ETH than they deposited. When `rsETHPrice` is later recomputed at the corrected (lower) price, the attacker's excess rsETH dilutes all existing holders' redemption value. This constitutes **direct theft of user funds** (Critical) and, at scale, **protocol insolvency** (Critical) — the total rsETH supply represents more ETH than the protocol holds.

## Likelihood Explanation

Chainlink feeds go stale during Ethereum network congestion (high gas prices prevent keeper updates) and during oracle infrastructure incidents — both are documented historical events. The stETH/ETH feed has a 24-hour heartbeat, meaning a stale price can persist for up to 24 hours. The deposit path (`depositAsset`) is permissionless and open to any user. No privileged role, front-running, or brute-force is required. The attacker only needs the stale price to be higher than the true market price at the time of deposit, which is a realistic condition (e.g., stETH depegs downward while the feed is frozen at a pre-depeg value).

## Recommendation

Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optional: per-feed heartbeat check
    // if (block.timestamp - updatedAt > MAX_HEARTBEAT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept

**Minimal call sequence (no fork required):**

1. Deploy `MockStaleFeed` returning `(roundId=2, answer=1.05e18, startedAt=0, updatedAt=block.timestamp - 2 days, answeredInRound=1)` — both `answeredInRound < roundId` and `updatedAt` stale.
2. Configure `ChainlinkPriceOracle` with `assetPriceFeed[stETH] = MockStaleFeed`.
3. Set `rsETHPrice = 1e18` (last correct update, before staleness).
4. Call `LRTDepositPool.depositAsset(stETH, 1e18, 0, "")`.
   - `getRsETHAmountToMint` → `1e18 * 1.05e18 / 1e18 = 1.05e18` rsETH minted.
5. True market price is 0.99e18. Call `updateRSETHPrice()` with corrected feed.
   - `totalETHInProtocol = 1e18 * 0.99e18 / 1e18 = 0.99e18`
   - `rsethSupply = 1.05e18`
   - `newRsETHPrice = 0.99e18 / 1.05e18 ≈ 0.9428e18`
6. All pre-existing rsETH holders now redeem at `0.9428e18` instead of `1.00e18` — their ETH value was stolen.

**Invariant that breaks:**
```
assert(rsETHPrice * totalRsETHSupply / 1e18 <= totalETHInProtocol);  // FAILS
```

`ChainlinkOracleForRSETHPoolCollateral.getRate()` would correctly revert at step 4 with `StalePrice()`, while `ChainlinkPriceOracle.getAssetPrice()` silently accepts the stale answer. [7](#0-6) [8](#0-7)
