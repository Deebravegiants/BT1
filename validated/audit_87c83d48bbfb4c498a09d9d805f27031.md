### Title
Unbounded Gas Consumption in `LRTOracle.updateRSETHPrice()` via Multiplicative Nested Loops Over Supported Assets, NDCs, and EigenLayer Queued Withdrawals — (File: contracts/LRTOracle.sol)

---

### Summary

The publicly callable `updateRSETHPrice()` in `LRTOracle` triggers a chain of nested loops whose total gas cost is the product of: (1) the number of supported assets — which has **no explicit size cap** in `LRTConfig` — (2) the number of NodeDelegator contracts, and (3) the number of EigenLayer queued withdrawals per NDC. As the protocol grows normally, this function can become prohibitively expensive or exceed the block gas limit, permanently bricking oracle updates and causing stale pricing across all deposit and withdrawal flows.

---

### Finding Description

The call chain is:

```
LRTOracle.updateRSETHPrice()          [public, whenNotPaused — callable by anyone]
  └─ _updateRsETHPrice()
       └─ _getTotalEthInProtocol()
            └─ for each asset in supportedAssetList:          ← LOOP 1 (no size cap)
                 ILRTDepositPool.getTotalAssetDeposits(asset)
                   └─ getAssetDistributionData(asset)
                        └─ for each NDC in nodeDelegatorQueue: ← LOOP 2
                             INodeDelegator.getAssetUnstaking(asset)
                               └─ DelegationManager.getQueuedWithdrawals()
                                    └─ for each withdrawal:    ← LOOP 3
                                         for each strategy:    ← LOOP 4
```

**Loop 1 — `supportedAssetList` has no size cap.**
`LRTConfig.addNewSupportedAsset()` pushes to `supportedAssetList` with no upper-bound check:

```solidity
// contracts/LRTConfig.sol:106-118
function _addNewSupportedAsset(address asset, uint256 depositLimit) private {
    ...
    isSupportedAsset[asset] = true;
    supportedAssetList.push(asset);   // ← no cap enforced
    ...
}
```

**Loop 2 — NDC queue iterated per asset.**
`getAssetDistributionData()` iterates every NDC for every asset:

```solidity
// contracts/LRTDepositPool.sol:447-456
for (uint256 i; i < ndcsCount;) {
    assetLyingInNDCs += IERC20(asset).balanceOf(nodeDelegatorQueue[i]);
    assetStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetBalance(asset);
    assetUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset);
    ...
}
```

**Loops 3 & 4 — `getAssetUnstaking()` iterates all EigenLayer queued withdrawals.**
Each NDC call fetches and iterates the full queued-withdrawal list from EigenLayer:

```solidity
// contracts/NodeDelegator.sol:406-427
(IDelegationManager.Withdrawal[] memory queuedWithdrawals, ...) =
    _getDelegationManager().getQueuedWithdrawals(address(this));

for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
    ...
    for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
        ...
    }
}
```

Total gas is **O(assets × NDCs × queuedWithdrawals × strategies)**. The same nested traversal is triggered by every user-facing call that reaches `getTotalAssetDeposits()`:

- `LRTDepositPool.depositETH()` → `_beforeDeposit()` → `_checkIfDepositAmountExceedesCurrentLimit()` → `getTotalAssetDeposits()`
- `LRTDepositPool.depositAsset()` — same path
- `LRTWithdrawalManager.initiateWithdrawal()` → `getAvailableAssetAmount()` → `lrtDepositPool.getTotalAssetDeposits()`

---

### Impact Explanation

**Medium — Unbounded gas consumption.**

If the cumulative gas cost of `updateRSETHPrice()` exceeds the block gas limit (~30 M gas on mainnet), the function reverts permanently. The stored `rsETHPrice` becomes stale. Downstream effects:

- `depositETH()` / `depositAsset()` use `lrtOracle.rsETHPrice()` (the stale cached value) to mint rsETH, causing users to receive incorrect amounts.
- `initiateWithdrawal()` uses `lrtOracle.rsETHPrice()` to compute expected asset amounts, causing incorrect withdrawal commitments.
- Even the privileged `updateRSETHPriceAsManager()` path is blocked if the gas cost exceeds the block limit.

---

### Likelihood Explanation

The protocol is designed to support multiple LSTs and multiple NDCs. As it scales:

- Each new supported asset (TIME_LOCK_ROLE, normal governance) multiplies the cost of every oracle update and every deposit/withdrawal.
- Each new NDC (admin, bounded by `maxNodeDelegatorLimit`) multiplies the inner loop.
- Each queued EigenLayer withdrawal (operator, bounded by `maxUncompletedWithdrawalCount`) multiplies the innermost loop.

No attacker action is required; normal protocol growth drives the cost upward. With, e.g., 10 assets × 10 NDCs × 50 queued withdrawals × 3 strategies = 15,000 storage-heavy iterations per `updateRSETHPrice()` call, gas exhaustion is realistic.

---

### Recommendation

1. **Cap `supportedAssetList`**: Add an explicit maximum (e.g., `MAX_SUPPORTED_ASSETS = 20`) enforced in `_addNewSupportedAsset()`.
2. **Cache `totalETHInProtocol`**: Store the result and update it lazily or via a separate keeper, rather than recomputing it on every oracle update and every user deposit/withdrawal.
3. **Decouple oracle update from deposit path**: `depositETH()` / `depositAsset()` should not recompute the full TVL on every call; use the cached `rsETHPrice` and only recompute on explicit oracle updates.
4. **Bound `getAssetUnstaking()` iterations**: Consider storing the unstaking amount as a running counter updated on `initiateUnstaking()` / `completeUnstaking()` rather than iterating EigenLayer's full withdrawal queue on every read.

---

### Proof of Concept

1. Protocol adds 15 supported LSTs via `LRTConfig.addNewSupportedAsset()` (TIME_LOCK_ROLE — normal governance).
2. Protocol deploys 10 NDCs, `maxNodeDelegatorLimit` raised to 10 (admin — normal operation).
3. Operators call `initiateUnstaking()` up to `maxUncompletedWithdrawalCount` times across all NDCs.
4. Any external caller invokes `LRTOracle.updateRSETHPrice()`.
5. The function executes 15 × 10 × (queued withdrawals per NDC) × (strategies per withdrawal) iterations of storage reads and external calls, consuming gas proportional to this product.
6. At sufficient scale, the transaction reverts out-of-gas; `rsETHPrice` is never updated; all subsequent `depositETH()`, `depositAsset()`, and `initiateWithdrawal()` calls operate on a stale exchange rate. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```text
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
```

```text
    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
```
