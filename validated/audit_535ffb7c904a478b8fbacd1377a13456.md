### Title
Fee-on-Transfer Token Accounting Divergence Creates Unbacked Wrapped Supply — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` records and emits the caller-supplied `amount` parameter in the `InitTransfer` event without measuring the actual tokens received. For fee-on-transfer ERC20 tokens, the contract receives `amount − fee` but the event broadcasts `amount`. NEAR's `fin_transfer_callback` trusts the event value unconditionally and mints/releases the full `amount` of wrapped tokens, permanently over-crediting the destination and creating a shortfall in the EVM vault that grows with every deposit.

---

### Finding Description

`initTransfer` in `OmniBridge.sol` handles native ERC20 lock-and-bridge with:

```solidity
// OmniBridge.sol lines 407–411
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-controlled
);
```

No balance snapshot is taken before or after the transfer. Immediately after, the function emits:

```solidity
// OmniBridge.sol lines 427–436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // same caller-supplied value, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [1](#0-0) [2](#0-1) 

On the NEAR side, `fin_transfer_callback` decodes the prover result (which is derived from the EVM event) and uses the event's `amount` field verbatim to credit the recipient:

```rust
// near/omni-bridge/src/lib.rs lines 709, 729
let Ok(ProverResult::InitTransfer(init_transfer)) = Self::decode_prover_result(0) ...
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [3](#0-2) 

There is no independent balance verification at any point in the pipeline. The token registration path (`logMetadata`) is fully permissionless — any caller can register any ERC20 — so no admin gate prevents a fee-on-transfer token from entering the bridge:

```solidity
// OmniBridge.sol lines 224–231
function logMetadata(address tokenAddress) external payable {
    ...
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [4](#0-3)

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer token deposits `amount − δ` into the EVM vault while crediting `amount` of wrapped tokens on NEAR. The cumulative shortfall `Σδ` grows monotonically. When users attempt to bridge back, `finTransfer` on EVM calls:

```solidity
IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
``` [5](#0-4) 

Once the vault balance falls below the outstanding wrapped supply, later redeemers receive a hard revert. Their wrapped tokens are permanently unclaimable — an irreversible fund lock. This is a direct backing-guarantee violation: wrapped tokens on NEAR are not 1:1 backed by locked tokens on EVM.

---

### Likelihood Explanation

**Medium.** Fee-on-transfer tokens exist on mainnet (e.g., STA, PAXG, tokens with configurable transfer taxes). Because `logMetadata` is permissionless and `initTransfer` imposes no token whitelist, any user can introduce such a token into the bridge without operator involvement. The shortfall is silent — no revert, no warning — and accumulates across all depositors of that token. A griefing attacker can deliberately amplify the shortfall by depositing a large amount and immediately redeeming, leaving honest depositors holding unredeemable wrapped tokens.

---

### Recommendation

1. **Measure actual received amount** using a balance snapshot in `initTransfer`:
   ```solidity
   uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
   IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
   uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
   // use `received` in the event, not `amount`
   ```
2. Emit `InitTransfer` with the measured `received` value so NEAR credits only what the vault actually holds.
3. Optionally maintain a per-token vault accounting map and enforce that `finTransfer` cannot release more than the recorded balance.

---

### Proof of Concept

1. Attacker calls `logMetadata(FOT_TOKEN)` — permissionless, registers a 1%-fee-on-transfer token.
2. NEAR relayer processes the metadata event; token is now bridgeable.
3. Alice calls `initTransfer(FOT_TOKEN, 1000, ...)`. EVM vault receives **990** FOT; event emits `amount = 1000`. NEAR mints **1000** wrapped FOT to Alice.
4. Attacker calls `initTransfer(FOT_TOKEN, 10000, ...)`. EVM vault receives **9900** FOT (total vault: **10890**); NEAR mints **10000** wrapped FOT to attacker.
5. Attacker immediately bridges back 10000 wrapped FOT. NEAR burns 10000; EVM calls `safeTransfer(attacker, 10000)`. Vault balance: **10890 − 10000 = 890** FOT.
6. Alice bridges back 1000 wrapped FOT. EVM calls `safeTransfer(Alice, 1000)`. Vault only holds **890** FOT → **revert**. Alice's 1000 wrapped FOT are permanently unclaimable.

The attacker spent ~100 FOT in fees to permanently lock Alice's 1000 FOT redemption path — an irreversible fund lock matching the Critical/High impact scope.
