The critical evidence is at `client.rs:1888-1908`: the transaction pool (`sharded_tx_pool`) is explicitly resharded when the shard layout changes, but `pending_transaction_queue` has no equivalent migration.

### Title
ShardedPendingTransactionQueue Not Migrated on Resharding Silently Bypasses P_MAX and Deploy-Exclusivity — (`chain/client/src/pending_transaction_queue.rs`, `chain/client/src/client.rs`)

### Summary

When a resharding epoch boundary is crossed, `sharded_tx_pool` is correctly migrated from old to new `ShardUId` keys, but `ShardedPendingTransactionQueue` is never migrated. Transactions from epoch N's uncertified blocks remain indexed under the old `ShardUId` (old layout version). Chunk production for epoch N+1 creates `PendingTxSession` with the new child `ShardUId`; the lookup `guard.get(&self.shard_uid)` returns `None`, and `PendingStateSnapshot::default()` (all zeros) is used. This silently zeroes `access_key_tx_count`, `deploy_tx_count`, and `paid_from_balance`, bypassing P_MAX and deploy-exclusivity enforcement for all transactions admitted into the first chunks of the resharded epoch.

### Finding Description

**Step 1 — Transactions stored under old ShardUId.**

In `update_pending_transaction_queue_for_block`, the `shard_uid` is derived from the block's own `epoch_id`: [1](#0-0) 

For the last block of epoch N, this produces `ShardUId { version: V, shard_id: S }` (old layout).

**Step 2 — Transaction pool is resharded; PTQ is not.**

At the epoch boundary, `sharded_tx_pool` is explicitly migrated: [2](#0-1) 

There is no equivalent call for `pending_transaction_queue`. `ShardedPendingTransactionQueue` has no `reshard` method and is never migrated. [3](#0-2) 

**Step 3 — Chunk production uses new child ShardUId.**

`produce_chunks` derives `epoch_id` for the new epoch, and `produce_chunk_internal` calls `shard_id_to_uid` with that new epoch's shard layout: [4](#0-3) 

`PendingTxSession` is then created with the new child `ShardUId`: [5](#0-4) 

**Step 4 — Lookup returns `None`; default zeros bypass enforcement.** [6](#0-5) 

Because the PTQ only has entries under the old parent `ShardUId`, `guard.get(&self.shard_uid)` returns `None`. `PendingStateSnapshot::default()` gives `access_key_tx_count = 0`, `deploy_tx_count = 0`, `paid_from_balance = 0`.

**Step 5 — P_MAX and deploy-exclusivity checks pass trivially.** [7](#0-6) 

With all counts at zero, neither the deploy-exclusivity guard nor the P_MAX guard fires, and `PendingTxCheckResult::Admit` is returned for every transaction regardless of how many are already pending from epoch N.

### Impact Explanation

An unprivileged user who submits P_MAX access-key transactions from a contract account in epoch N (which remain uncertified at the epoch boundary) can then submit a P_MAX+1th (or more) transaction that is admitted into epoch N+1's first chunks. The NEP-611 invariant — at most P_MAX pending access-key transactions per contract account across uncertified blocks — is violated. Deploy-exclusivity is similarly bypassed: a deploy transaction can be admitted alongside other pending access-key transactions, or vice versa, violating the single-deploy-per-window invariant. This is a concrete mempool admission bypass.

### Likelihood Explanation

Requires SPICE protocol enabled, `spice_pending_transaction_queue_enabled = true`, and a resharding epoch boundary. The attacker only needs to submit valid signed transactions via the public RPC — no validator or operator privileges are required. The timing (transactions in uncertified blocks at the epoch boundary) is a normal occurrence in the SPICE protocol, not an adversarial precondition.

### Recommendation

Add a `reshard` method to `ShardedPendingTransactionQueue` that, given old and new `ShardLayout`, migrates each `PendingTransactionQueue` from old `ShardUId` keys to the appropriate new child `ShardUId` keys (splitting the per-account and per-nonce aggregates proportionally, or conservatively attributing all counts to both children). Call this method alongside `sharded_tx_pool.reshard(...)` at the epoch boundary in `client.rs:1899-1902`.

### Proof of Concept

1. Configure a test network with two protocol versions and `epoch_length = 5`, where the second version introduces a new shard layout (one shard splits into two).
2. From a contract account, submit exactly P_MAX access-key transactions in epoch N. Confirm they are included in blocks but remain uncertified (delay endorsements or use a test harness that controls certification).
3. At the epoch N → N+1 boundary, submit a P_MAX+1th access-key transaction.
4. Observe that `PendingTxSession::check_pending` returns `PendingTxCheckResult::Admit` for the P_MAX+1th transaction (because `guard.get(&child_shard_uid)` returns `None` and the default snapshot has `access_key_tx_count = 0`).
5. Assert the transaction is included in the first chunk of epoch N+1, violating the P_MAX invariant.

pub struct ShardedPendingTransactionQueue {
    queues: HashMap<ShardUId, PendingTransactionQueue>,
}

impl ShardedPendingTransactionQueue {
    pub fn new() -> Self {
        Self { queues: HashMap::new() }
    }

    pub fn get_or_create(&mut self, shard_uid: ShardUId) -> &mut PendingTransactionQueue {
        self.queues.entry(shard_uid).or_insert_with(PendingTransactionQueue::new)
    }

    pub fn get(&self, shard_uid: &ShardUId) -> Option<&PendingTransactionQueue> {
        self.queues.get(shard_uid)
    }

    pub fn get_mut(&mut self, shard_uid: &ShardUId) -> Option<&mut PendingTransactionQueue> {
        self.queues.get_mut(shard_uid)
    }

    pub fn remove_certified_block(&mut self, block_hash: &CryptoHash) {
        for queue in self.queues.values_mut() {
            queue.remove_certified_chunk_by_block_hash(block_hash);
        }
    }

    pub fn clear(&mut self) {
        self.queues.clear();
    }
}
