### Title
`vote_add_domains` Uses Hardcoded 100% Quorum Instead of Configurable Governance Threshold — (File: `crates/contract/src/state/running.rs`)

---

### Summary

`RunningContractState::vote_add_domains` hardcodes the quorum for domain-addition proposals as **100% of all participants** (`participants.len()`), completely ignoring the configurable governance threshold stored in `self.parameters.threshold()`. Because the governance threshold can be updated via `vote_new_parameters` to any value between 60% and 100%, any update to that threshold has **zero effect** on domain-addition votes. A single participant strictly below the governance threshold can permanently block new domain additions, breaking the protocol's own governance invariant.

---

### Finding Description

In `RunningContractState::vote_add_domains` (`crates/contract/src/state/running.rs`, line 237):

```rust
let n_votes = self.add_domains_votes.vote(domains.clone(), &participant);
if self.parameters.participants().len() as u64 == n_votes {
    // transition to InitializingContractState
```

The quorum check compares `n_votes` against the **total participant count** (100%), not against `self.parameters.threshold()` — the configurable governance threshold that is explicitly stored in `ThresholdParameters` and validated to lie in `[ceil(0.6·n), n]`. [1](#0-0) 

The governance threshold is a first-class, updatable parameter: [2](#0-1) 

Its lower bound is enforced at 60% of participants: [3](#0-2) 

Other governance operations — TEE image voting, OS-measurement voting, and foreign-chain RPC whitelisting — all gate on `self.threshold()?.value()` (confirmed in `docs/foreign-chain-transactions.md`):

> "Voting uses the protocol's existing signing threshold (`self.threshold()?.value()`), the same gate as `verify_tee` and `vote_add_os_measurement`." [4](#0-3) 

`vote_add_domains` is the sole governance action that deviates from this rule, silently requiring unanimous consent regardless of what the governance threshold is set to.

The analog to FrankenDAO is exact:

| FrankenDAO | NEAR MPC |
|---|---|
| `stakedTimeBonus` uses hardcoded `/2` (50%) | `vote_add_domains` uses hardcoded `participants.len()` (100%) |
| `monsterMultiplier` is the configurable state variable | `self.parameters.threshold()` is the configurable state variable |
| Governance call to `setMonsterMultiplier()` is ignored | Governance call to `vote_new_parameters` (threshold update) is ignored |

---

### Impact Explanation

**Medium.** Any single participant can unilaterally veto domain additions by simply not calling `vote_add_domains`, even after the governance threshold of votes has been reached. Because the governance threshold can be as low as 60% of participants (4 out of 7, for example), a single non-voting participant below that threshold can permanently prevent the MPC network from adding new signing domains (e.g., new curves or protocols). This breaks the production safety invariant that governance decisions require only `threshold` votes, not unanimous consent, and can freeze the protocol's ability to expand its key set without any threshold-level collusion.

---

### Likelihood Explanation

**Medium.** The governance threshold is actively configurable via `vote_new_parameters` and is commonly set below 100% (the minimum is 60%). Any participant who wishes to block a domain addition — for competitive, economic, or adversarial reasons — can do so by simply abstaining. No special capability, key material, or collusion is required; a single participant acting alone is sufficient.

---

### Recommendation

Replace the hardcoded participant-count comparison in `vote_add_domains` with the stored governance threshold:

```rust
// Before (hardcoded 100%):
if self.parameters.participants().len() as u64 == n_votes {

// After (uses configurable governance threshold):
if self.parameters.threshold().value() <= n_votes {
```

This aligns `vote_add_domains` with every other governance action in the contract, which already gates on `threshold()`. [1](#0-0) 

---

### Proof of Concept

1. Network initializes with **10 participants** and governance threshold set to **7** (70%, the minimum for n=10).
2. Governance later calls `vote_new_parameters` to lower the threshold to **6** (60%). This succeeds and is stored in `self.parameters.threshold()`.
3. A proposal to add a new Ed25519 domain is submitted. **9 out of 10 participants** call `vote_add_domains` with the same `Vec<DomainConfig>`.
4. `add_domains_votes.vote(...)` returns `n_votes = 9`.
5. The check `self.parameters.participants().len() as u64 == n_votes` evaluates `10 == 9` → **false**. The proposal does not pass.
6. The one non-voting participant has effectively vetoed the proposal, despite the governance threshold of 6 having been exceeded by 3 votes.
7. The governance threshold update from step 2 had **zero effect** on domain-addition quorum. [5](#0-4) [6](#0-5)

    pub fn vote_add_domains(
        &mut self,
        domains: Vec<DomainConfig>,
    ) -> Result<Option<InitializingContractState>, Error> {
        if domains.is_empty() {
            return Err(DomainError::AddDomainsMustAddAtLeastOneDomain.into());
        }
        let num_participants = u64::try_from(self.parameters.participants().len())
            .expect("participant count fits in u64");
        for domain in &domains {
            validate_domain_purpose(domain)?;
            validate_domain_threshold(domain, num_participants)?;
        }
        // Keep trust assumptions consistent: a domain must never require more shares to
        // reconstruct than the GovernanceThreshold demands to govern. Route through the
        // canonical helper so the cross-domain invariant has a single source of truth.
        ThresholdParameters::validate_governance_against_reconstruction(
            num_participants,
            self.parameters.threshold(),
            max_reconstruction_threshold(&domains),
        )?;
        let participant = AuthenticatedParticipantId::new(self.parameters.participants())?;
        let n_votes = self.add_domains_votes.vote(domains.clone(), &participant);
        if self.parameters.participants().len() as u64 == n_votes {
            let new_domains = self.domains.add_domains(domains.clone())?;
            Ok(Some(InitializingContractState {
                generated_keys: self.keyset.domains.clone(),
                domains: new_domains,
                epoch_id: self.keyset.epoch_id,
                generating_key: KeyEvent::new(
                    self.keyset.epoch_id,
                    domains[0].clone(),
                    self.parameters.clone(),
                ),
                cancel_votes: BTreeSet::new(),
            }))
        } else {
            Ok(None)
        }
    }

    fn validate_threshold(n_shares: u64, k: Threshold) -> Result<(), Error> {
        if k.value() > n_shares {
            return Err(InvalidThreshold::MaxRequirementFailed {
                max: n_shares,
                found: k.value(),
            }
            .into());
        }
        if k.value() < MIN_THRESHOLD_ABSOLUTE {
            return Err(InvalidThreshold::MinAbsRequirementFailed.into());
        }
        let lower_relative_bound = governance_threshold_lower_relative_bound(n_shares);
        if k.value() < lower_relative_bound {
            return Err(InvalidThreshold::MinRelRequirementFailed {
                required: lower_relative_bound,
                found: k.value(),
            }
            .into());
        }
        let upper_relative_bound = governance_threshold_upper_relative_bound(n_shares);
        if k.value() > upper_relative_bound {
            return Err(InvalidThreshold::MaxRelRequirementFailed {
                max: upper_relative_bound,
                found: k.value(),
            }
            .into());
        }
        Ok(())
    }

```markdown
#### Why protocol signing threshold (not unanimous, not a separate per-chain knob)

Voting uses the protocol's existing signing threshold (`self.threshold()?.value()`), the same gate as `verify_tee` and `vote_add_os_measurement`. An earlier design proposed a separate per-chain *voting* threshold so mainnet and testnet could be voted in under different agreement requirements; that was dropped because (a) there's no setter that could safely populate it without itself being voted in, leaving a hardcoded default that's strictly weaker than the protocol threshold, and (b) the per-chain numeric on `ChainVote.quorum` already covers the *runtime* security knob — how many of N whitelisted providers must agree for a node to accept a response — which is what operators actually need to tune per chain.
```
