### Title
Deterministic Account ID Uses 160-bit Truncated keccak256, Enabling Birthday-Attack Collision to Deploy Malicious Code at a Committed Address - (`core/primitives/src/utils.rs`)

### Summary

`derive_near_deterministic_account_id` derives a `0s`-prefixed account ID by taking only the last 20 bytes (`[12..32]`) of a 256-bit keccak256 digest of the borsh-serialised `DeterministicAccountStateInit`. This 160-bit truncation means 2^96 distinct `state_init` blobs map to every single account ID. A birthday attack requiring ~2^80 keccak256 evaluations can find two different `state_init` values that collide on the same account ID. An attacker who finds such a pair can deploy the malicious variant first, permanently occupying the address with wrong code before the legitimate deployer arrives.

### Finding Description

`derive_near_deterministic_account_id` in `core/primitives/src/utils.rs` computes:

```
account_id = "0s" + hex( keccak256( borsh(state_init) )[12..32] )
``` [1](#0-0) 

The only on-chain guard is `validate_deterministic_state_init`, which checks `derived_id == receiver_id`: [2](#0-1) 

Because both the `state_init` and the `receiver_id` are supplied by the transaction sender, the check only enforces internal consistency — it does **not** prevent a second `state_init` that hashes to the same 20-byte suffix from passing validation.

`action_deterministic_state_init` then deploys the code from whichever `state_init` arrives first and silently ignores subsequent ones: [3](#0-2) 

The `DeterministicAccountStateInit` struct accepts an arbitrary `data: BTreeMap<Vec<u8>, Vec<u8>>` and a `GlobalContractIdentifier` (either a code hash or an account ID), giving an attacker unbounded freedom to vary inputs: [4](#0-3) 

**Birthday-attack path (unprivileged):**

1. Attacker deploys a malicious global contract under an account they control.
2. Attacker generates ~2^80 `state_init_M` blobs (varying `data` entries) referencing the malicious contract and records their 20-byte account IDs.
3. Attacker simultaneously generates ~2^80 `state_init_L` blobs (varying `data` entries) referencing a legitimate-looking global contract and records their account IDs.
4. With ~2^80 evaluations a collision `account_id(state_init_M) == account_id(state_init_L)` is found (birthday bound on 160-bit space).
5. Attacker publishes `state_init_L` as the "canonical" initialiser for a protocol (e.g., in documentation or a factory contract).
6. Attacker submits a `DeterministicStateInitAction` with `state_init = state_init_M` and `receiver_id = colliding_account_id` — validation passes.
7. The account is now permanently initialised with malicious code.
8. Any subsequent `DeterministicStateInitAction` with `state_init_L` is a no-op (`account.contract().is_none()` is false).

No validator, relayer, or admin privilege is required; any NEAR account can submit a `DeterministicStateInitAction`.

### Impact Explanation

The deterministic account ID is intended to be a cryptographic commitment: users should be able to trust that an account at address `0sXXX…` was initialised from the unique `state_init` that hashes to `XXX…`. The 160-bit truncation breaks this commitment. An attacker who finds a collision can:

- Permanently occupy a deterministic address with arbitrary (malicious) contract code.
- Prevent the legitimate `state_init` from ever being deployed at that address.
- Deceive users and integrators who rely on the account ID as a commitment to the code and initial state.

The corrupted DB entry is the contract code stored in the trie for the deterministic account, and the corrupted protocol invariant is the uniqueness of `state_init → account_id`.

### Likelihood Explanation

Per the analysis in the referenced Napier/KyberSwap discussions, a birthday attack on 160-bit space requires ~2^80 keccak256 evaluations and ~2^80 entries of storage (~160 TB with the naive algorithm, reducible with the algorithm from the cited paper). keccak256 is far cheaper to evaluate than elliptic-curve operations, so the cost is lower than the Ethereum address-collision case. The attack is not cheap today but is within reach of a well-funded adversary and becomes progressively cheaper as compute costs fall. Deterministic accounts are designed to be long-lived and immutable once deployed, so the window of exposure is permanent.

### Recommendation

Use the full 256-bit keccak256 output (or SHA-256) as the account ID instead of truncating to 160 bits. NEAR account IDs support up to 64 characters; a 32-byte hash hex-encoded is 64 characters, which fits exactly. Alternatively, adopt the `0u` universal account ID scheme already present in the codebase (`core/primitives-core/src/universal_account_id.rs`), which encodes the full 256-bit hash: [5](#0-4) 

Using the full hash eliminates the birthday-attack surface entirely.

### Proof of Concept

```
state_init_M = DeterministicAccountStateInit::V1 {
    code: GlobalContractIdentifier::AccountId("attacker.near"),
    data: { b"x" => [i as u8, ...] }   // vary i over 0..2^80
}

state_init_L = DeterministicAccountStateInit::V1 {
    code: GlobalContractIdentifier::AccountId("legitimate.near"),
    data: { b"y" => [j as u8, ...] }   // vary j over 0..2^80
}

// After ~2^80 evaluations, find i*, j* such that:
// keccak256(borsh(state_init_M(i*)))[12..32]
//   == keccak256(borsh(state_init_L(j*)))[12..32]

// Attacker submits first:
SignedTransaction::deterministic_state_init(
    nonce, attacker, colliding_account_id,
    state_init_M(i*), deposit
)
// Passes validate_deterministic_state_init: derived_id == colliding_account_id ✓
// Account deployed with attacker's malicious code.

// Legitimate deployer submits later:
SignedTransaction::deterministic_state_init(
    nonce, user, colliding_account_id,
    state_init_L(j*), deposit
)
// Passes validate_deterministic_state_init: derived_id == colliding_account_id ✓
// action_deterministic_state_init: account.contract().is_none() == false → no-op.
// Account permanently retains malicious code.

pub fn derive_near_deterministic_account_id(
    state_init: &DeterministicAccountStateInit,
) -> AccountId {
    use sha3::Digest;
    let data = borsh::to_vec(&state_init).expect("borsh must not fail");
    let hash = sha3::Keccak256::digest(&data);
    format!("0s{}", hex::encode(&hash[12..32])).parse().unwrap()
}

fn validate_deterministic_state_init(
    limit_config: &LimitConfig,
    action: &DeterministicStateInitAction,
    receiver_id: &AccountId,
) -> Result<(), ActionsValidationError> {
    validate_global_contract_identifier(action.state_init.code())?;

    let derived_id = derive_near_deterministic_account_id(&action.state_init);

    if derived_id != *receiver_id {
        return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver {
            derived_id,
            receiver_id: receiver_id.clone(),
        });

//! A UAID encodes a 32-byte hash as
//! `0u` + 52 Crockford-base32 symbols of the hash + a 6-symbol Bech32m BCH checksum
//! = 60 characters, all lowercase `[0-9a-z]`, which is a valid NEAR account id.
//!
//! This is a pure codec: it turns a hash into an address and back and validates
//! the checksum. Hashing a `StateInit` into the 32-byte input lives with the
//! account-id derivation, not here.
//!
//! The base32 and checksum are implemented in this module rather than taken from
//! an external crate. No existing crate fits the whole codec: the `bech32` crates
//! implement full Bech32/Bech32m framing and their own alphabet, which we don't
//! use, and a base32 crate would cover only the encoding half while the checksum

//! glyph-to-value pass, since they share one 5-bit symbol pipeline. The
//! implementation was cross-checked against the `data-encoding` crate over 40M
//! random cases with zero correctness divergence and on-par performance.
