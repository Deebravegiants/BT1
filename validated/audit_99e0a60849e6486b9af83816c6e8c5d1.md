Let me check the peer manager and whether block signature verification is called anywhere in the sync path.

### Title
P2P Sync Stores Peer-Supplied Transaction Hashes Without Verification, Corrupting RPC Transaction Lookups — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts `transaction_hash` directly from an unauthenticated peer and writes it to persistent storage without recomputing or verifying it. The number of transactions consumed is controlled by `n_transactions` from a `SignedBlockHeader` whose cryptographic signature is never verified in the p2p sync path. A malicious peer can therefore poison the node's `transaction_hash_to_idx` table, causing `get_transaction_by_hash` to return wrong transactions or fail to find canonical ones.

---

### Finding Description

**Step 1 — Header accepted without signature verification.**

`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` from a peer with only two checks: the block number matches the expected value, and the `signatures` vector has exactly one element. [1](#0-0) 

`verify_block_signature` exists in the API crate but is never called in this path. [2](#0-1) 

The peer-supplied `block_hash`, `n_transactions`, and `transaction_commitment` are stored verbatim. [3](#0-2) 

**Step 2 — `n_transactions` from the stored header drives transaction consumption.**

`parse_data_for_block` for transactions reads `n_transactions` from the already-stored (peer-controlled) header and uses it as the loop bound: [4](#0-3) 

**Step 3 — Peer-supplied `transaction_hash` stored without recomputation.**

Inside the loop, `transaction_hash` from the peer's `FullTransaction` is pushed directly into `block_body.transaction_hashes`. The developer TODO explicitly acknowledges the missing validation: [5](#0-4) 

**Step 4 — Hashes written into the `transaction_hash_to_idx` index.**

`append_body` → `write_transactions` inserts the peer-supplied hash as the key in `transaction_hash_to_idx_table` and stores it in `TransactionMetadata`: [6](#0-5) 

**Step 5 — RPC `get_transaction_by_hash` trusts the index.**

The RPC handler looks up the hash in exactly this table and returns the transaction at the stored index: [7](#0-6) 

The `transaction_commitment` stored in the header is never used to verify the actual hashes received from the peer. [8](#0-7) 

---

### Impact Explanation

A syncing node that connects to a malicious peer will store attacker-chosen transaction hashes in its persistent storage. Consequences:

- `get_transaction_by_hash(canonical_hash)` returns `TRANSACTION_HASH_NOT_FOUND` for every transaction in the poisoned block.
- `get_transaction_by_hash(attacker_hash)` returns the wrong transaction body.
- Receipt and event lookups keyed by transaction hash are similarly corrupted.
- The node propagates this corrupted view to any downstream clients or other nodes that sync from it.

This matches the allowed impact: **"High. RPC execution... returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The attacker only needs to be a peer that the victim node connects to — no credentials, no stake, no sequencer role. The p2p sync protocol is unauthenticated at the data-content level. The only structural barrier is peer scoring/reputation, but the malicious peer sends structurally valid messages (correct block number, one signature element) so it is never reported as a bad peer and the data is committed to storage.

---

### Recommendation

1. **Recompute and verify transaction hashes.** After receiving each `FullTransaction`, recompute the hash from the transaction body using the canonical hash function and reject (report peer) if it does not match the peer-supplied value.
2. **Verify the `transaction_commitment` in the header against the actual transaction hashes.** After collecting all transactions for a block, compute `calculate_transaction_commitment` over the received hashes and compare it to the `transaction_commitment` stored in the header. Reject if they differ.
3. **Cryptographically verify the block signature** in `HeaderStreamBuilder::parse_data_for_block` using `verify_block_signature` against the known sequencer public key before storing the header.

---

### Proof of Concept

```
1. Attacker node connects to victim via the p2p sync protocol.
2. Victim requests headers. Attacker responds with:
     SignedBlockHeader {
       block_number: N,
       n_transactions: 2,
       transaction_commitment: <any value>,
       block_hash: <any value>,
       signatures: [<any single signature>],   // length check passes
     }
3. Header stored; n_transactions = 2.
4. Victim requests transactions for block N. Attacker responds with:
     FullTransaction { transaction: <real tx body>, transaction_hash: 0xDEAD }
     FullTransaction { transaction: <real tx body>, transaction_hash: 0xBEEF }
5. Both hashes stored in transaction_hash_to_idx_table.
6. Victim RPC: get_transaction_by_hash(0xCANONICAL) → TRANSACTION_HASH_NOT_FOUND
              get_transaction_by_hash(0xDEAD)      → returns wrong transaction
```
