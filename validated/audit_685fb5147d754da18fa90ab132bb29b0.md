### Title
Unauthenticated Mirror Launcher-ID Injection via Crafted Parent Spend in `DataLayerWallet.coin_added` — (`chia/data_layer/data_layer_wallet.py`)

### Summary

`DataLayerWallet.coin_added` trusts the `launcher_id` returned by `get_mirror_info` without verifying it corresponds to the coin being processed. Because `fetch_coin_spend` validates the puzzle hash but **not the solution**, a malicious peer can return a crafted solution for any parent coin, causing an arbitrary attacker-controlled coin to be registered as a mirror for a legitimately tracked DL singleton. A second, purely on-chain variant requires no malicious peer at all.

---

### Finding Description

**Entry point — any coin with the mirror puzzle hash triggers `coin_added`:**

In `wallet_state_manager.py`, the DL wallet is assigned as the handler for *any* coin whose puzzle hash equals `MIRROR_PUZZLE_HASH`: [1](#0-0) 

This means an attacker who creates any coin with the mirror puzzle hash will cause `coin_added` to be invoked on the DL wallet.

**`coin_added` fetches the parent spend from the peer without solution validation:** [2](#0-1) 

`fetch_coin_spend` validates only the puzzle hash and coin name — the solution is accepted verbatim from the peer: [3](#0-2) 

**`get_mirror_info` extracts the launcher_id from the first matching CREATE_COIN condition, with no binding to the specific coin being processed:** [4](#0-3) 

It finds the **first** `CREATE_COIN` condition whose puzzle hash equals the mirror puzzle hash and reads `memos[0]` as the `launcher_id`. It does **not** verify that this condition creates the specific `coin` passed to `coin_added` (matching amount, parent, or coin ID).

**If the extracted `launcher_id` is tracked, the coin is unconditionally registered as a mirror:** [5](#0-4) 

---

### Attack Vectors

**Vector 1 — Malicious peer (crafted solution):**
1. Attacker creates any coin with the mirror puzzle hash (e.g., from a standard p2 parent they control).
2. Wallet sees the coin (via the `MIRROR_PUZZLE_HASH` catch-all at line 1837 of `wallet_state_manager.py`) and calls `coin_added`.
3. `coin_added` calls `fetch_coin_spend` on the peer for the parent coin. The peer returns the correct puzzle (passes hash check) but a **crafted solution** — e.g., a delegated puzzle that emits `CREATE_COIN(mirror_puzzle_hash, amount, [tracked_launcher_id, url])`.
4. `get_mirror_info` runs the puzzle with the crafted solution and returns `tracked_launcher_id`.
5. `is_launcher_tracked(tracked_launcher_id)` returns `True`.
6. The attacker's coin is written to the mirror DB as a mirror for `tracked_launcher_id`.

**Vector 2 — Pure on-chain, no malicious peer:**
1. Attacker creates a parent coin whose actual on-chain solution produces two `CREATE_COIN` conditions with the mirror puzzle hash:
   - Condition 1: `memos = [tracked_launcher_id, url1]` → creates coin A
   - Condition 2: `memos = [attacker_launcher_id, url2]` → creates coin B
2. When coin B is processed, `get_mirror_info` returns `tracked_launcher_id` (from condition 1, the first match).
3. Coin B — whose actual memos encode `attacker_launcher_id` — is registered as a mirror for `tracked_launcher_id`.

---

### Impact Explanation

The mirror DB is the authoritative source for `dl_get_mirrors`, which the Data Layer uses to build its URL subscription list via `update_subscriptions_from_wallet`. Injecting attacker-controlled entries:
- Pollutes the mirror list for a legitimate singleton with attacker-chosen URLs.
- Causes `dl_get_mirrors` to return false mirrors to the Data Layer, corrupting sync subscriptions.
- Persists until the wallet is re-synced; the bogus entry cannot be removed via `delete_mirror` (ownership check fails), leaving the state permanently corrupted until a full resync.

This satisfies the **High** impact criterion: *Corruption of wallet sync state / Data Layer root/store state with direct security impact.*

---

### Likelihood Explanation

- Vector 1 requires the attacker to be a peer the wallet connects to (any unprivileged node).
- Vector 2 requires only the ability to submit a transaction to the network (no special access).
- Tracked launcher IDs are public on-chain.
- No key material or privileged access is needed.

---

### Recommendation

In `coin_added`, after calling `get_mirror_info`, verify that the `CREATE_COIN` condition found by `get_mirror_info` actually creates the coin being processed. Specifically, confirm that the condition's puzzle hash, amount, and derived coin ID match `coin`:

```python
# After get_mirror_info returns (launcher_id, urls):
# Re-scan conditions to find the specific CREATE_COIN that matches `coin`
expected_coin_id = coin.name()
found_matching = False
for condition in conditions:
    if (condition.first() == CREATE_COIN_OPCODE
        and condition.at("rf") == mirror_puzzle_hash
        and Coin(parent_state.coin.name(), mirror_puzzle_hash,
                 int.from_bytes(condition.at("rrf"), "big")).name() == expected_coin_id):
        # verify memos match this specific condition
        found_matching = True
        break
if not found_matching:
    return
```

Additionally, consider validating the parent spend solution against the block's removals root (as done in `validate_received_state_from_peer`) rather than trusting the peer's solution verbatim.

```python
async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:
    solution_response = await peer.call_api(
        FullNodeAPI.request_puzzle_solution, RequestPuzzleSolution(coin.name(), height)
    )
    if solution_response is None or not isinstance(solution_response, RespondPuzzleSolution):
        raise PeerRequestException(f"Was not able to obtain solution {solution_response}")
    coin_id = coin.name()
    if solution_response.response.puzzle.get_tree_hash() != coin.puzzle_hash:
        raise PeerRequestException(f"Peer returned wrong puzzle hash for coin {coin_id}")
    if solution_response.response.coin_name != coin_id:
        raise PeerRequestException(f"Peer returned wrong coin name in puzzle solution for coin {coin_id}")

    return make_spend(
        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )
```

```python
def get_mirror_info(
    parent_puzzle: Program | SerializedProgram, parent_solution: Program | SerializedProgram
) -> tuple[bytes32, list[bytes]]:
    assert type(parent_puzzle) is type(parent_solution)
    _, conditions = run_with_cost(parent_puzzle, INFINITE_COST, parent_solution)
    for condition in conditions.as_iter():
        if (
            condition.first().as_python() == ConditionOpcode.CREATE_COIN
            and condition.at("rf").as_python() == create_mirror_puzzle().get_tree_hash()
        ):
            memos: list[bytes] = condition.at("rrrf").as_python()
            launcher_id = bytes32(memos[0])
            return launcher_id, [url for url in memos[1:]]
    raise ValueError("The provided puzzle and solution do not create a mirror coin")
```
