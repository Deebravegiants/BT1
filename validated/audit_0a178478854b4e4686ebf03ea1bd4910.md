### Title
Dijkstra Era Minimum Fee Underestimates Reference Script Size Across Sub-Transactions - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The `getMinFeeTxUtxo` implementation for the Dijkstra era reuses `getConwayMinFeeTxUtxo`, which only measures reference-script bytes from the **top-level** transaction's inputs. Sub-transactions introduced in the Dijkstra era carry their own inputs and reference inputs, whose reference scripts are deserialized by every validating node but are **not counted** in the minimum-fee check. A purpose-built helper `batchNonDistinctRefScriptsSize` that correctly aggregates reference-script bytes across all sub-transactions exists in the same file but is never wired into the fee-validation path.

---

### Finding Description

The Conway era introduced a tiered reference-script fee (`tierRefScriptFee`) to prevent a DDoS vector where large scripts could be deserialized cheaply. The fee is enforced through `getMinFeeTxUtxo`, which computes the total reference-script byte count and passes it to `getMinFeeTx`.

For Conway, `getConwayMinFeeTxUtxo` computes this count via `txNonDistinctRefScriptsSize`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs : 174-175
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs : 183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

This only inspects the top-level transaction's `inputsTxBodyL` and `referenceInputsTxBodyL`.

The Dijkstra era inherits this unchanged:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs : 141
  getMinFeeTxUtxo = getConwayMinFeeTxUtxo
```

Yet the Dijkstra era adds **sub-transactions** (`subTransactionsTxBodyL`), each of which has its own inputs and reference inputs that can point to UTxO entries carrying reference scripts. The same file defines a helper that correctly aggregates across all sub-transactions:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs : 264-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

`batchNonDistinctRefScriptsSize` is never called from `getMinFeeTxUtxo`. The minimum-fee check therefore ignores every reference script that appears exclusively in sub-transaction inputs, allowing a submitter to include arbitrarily large reference scripts in sub-transactions while paying only the fee for the top-level transaction's reference scripts.

The produced-value calculation for Dijkstra sub-transactions (`dijkstraSubTxProducedValue`) does correctly include deposits and treasury donations for sub-transactions, confirming that sub-transaction fields are otherwise accounted for — making the omission in the fee path a clear inconsistency:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs : 258-261
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody <> txBody ^. treasuryDonationTxBodyL)
    <> burnedMultiAssets txBody
```

---

### Impact Explanation

**Medium — Attacker-controlled transactions modify fees outside design parameters.**

An unprivileged transaction submitter can craft a Dijkstra top-level transaction whose sub-transactions reference UTxO entries containing large Plutus scripts. Every validating node must deserialize those scripts (the exact overhead the tiered reference-script fee was designed to price), yet the minimum-fee check passes with a fee computed only from the top-level transaction's reference scripts. The submitter pays less than the protocol-intended cost, replicating the class of resource-exhaustion attack documented in ADR-009 (the June 2024 DDoS event) but through the sub-transaction surface. ---

### Likelihood Explanation

Any unprivileged user who can submit a Dijkstra transaction can exploit this. No special role, key, or governance majority is required. The attacker only needs to:

1. Create UTxO entries containing large Plutus scripts (standard transactions).
2. Construct a Dijkstra top-level transaction with sub-transactions whose `inputsTxBodyL` or `referenceInputsTxBodyL` point to those UTxO entries.
3. Set the transaction fee to the minimum computed by `getConwayMinFeeTxUtxo` (which ignores sub-transaction reference scripts).

The ledger's `validateFeeTooSmallUTxO` check will pass because it calls `getMinFeeTxUtxo`, which is `getConwayMinFeeTxUtxo`, which does not include sub-transaction reference-script bytes.

---

### Recommendation

Replace the `getMinFeeTxUtxo` binding in the `EraUTxO DijkstraEra` instance with an implementation that delegates to `batchNonDistinctRefScriptsSize`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
getDijkstraMinFeeTxUtxo ::
  ( EraTx era
  , DijkstraEraTxBody era
  ) =>
  PParams era -> Tx TopTx era -> UTxO era -> Coin
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx

instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo
```

This mirrors the pattern used in Conway (`getConwayMinFeeTxUtxo` → `txNonDistinctRefScriptsSize`) but extends it to the full batch of sub-transactions, consistent with how `dijkstraProducedValue` already aggregates over sub-transactions.

Additionally, verify that the per-transaction and per-block reference-script size caps (`ppMaxRefScriptSizePerTxG`, `ppMaxRefScriptSizePerBlockG`) are enforced against the **batch** total, not only the top-level transaction total.

---

### Proof of Concept

1. **Setup**: Deploy two UTxO entries each containing a ~25 KiB Plutus script (near the first pricing tier boundary).

2. **Craft transaction**: Build a Dijkstra top-level transaction with:
   - Top-level inputs: one ordinary UTxO (no reference script) — contributes 0 bytes to `txNonDistinctRefScriptsSize`.
   - Sub-transaction 1: `referenceInputsTxBodyL` pointing to the first 25 KiB script UTxO.
   - Sub-transaction 2: `referenceInputsTxBodyL` pointing to the second 25 KiB script UTxO.

3. **Fee calculation by node**: `getConwayMinFeeTxUtxo` calls `txNonDistinctRefScriptsSize` on the top-level transaction only → 0 reference-script bytes → `tierRefScriptFee` contributes 0 lovelace.

4. **Actual node work**: Both 25 KiB scripts are deserialized for sub-transaction validation → ~50 KiB of script deserialization work, which at `minFeeRefScriptCostPerByte = 15` and the 1.2× multiplier would cost approximately `tierRefScriptFee(50_000) ≈ 862_500` lovelace under the correct calculation.

5. **Result**: Transaction is accepted with a fee that is ~862_500 lovelace lower than the protocol-intended minimum, demonstrating the underestimation. [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
