### Title
Single Attested Participant Can Forge CKD Response for `AppPublicKey` Requests, Bypassing Threshold Authorization ŌĆö (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract applies output verification only for `CKDAppPublicKey::AppPublicKeyPV` requests, leaving `CKDAppPublicKey::AppPublicKey` requests with **no cryptographic check on the response content**. Any single attested participant ŌĆö a Byzantine node strictly below the signing threshold ŌĆö can call `respond_ckd` with an arbitrary `CKDResponse` for any pending `AppPublicKey` CKD request. The contract accepts and resolves the request unconditionally, delivering a forged confidential key to the user and permanently blocking the legitimate threshold response.

---

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 653ŌĆō689), after verifying the caller is an attested participant and the protocol is active, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ŌåÉ NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which uses BLS12-381 pairings (requiring both a G1 and G2 point) to verify the encrypted output is correctly derived from the threshold key. For `AppPublicKey` (a single G1 point), no equivalent check exists ŌĆö the response is serialized and forwarded to the user verbatim.

The `resolve_yields_for` call then drains **all** queued yield slots for that request key, meaning once a forged response is accepted, the pending request is permanently removed from contract state. Legitimate nodes can no longer respond.

This is structurally analogous to the reported DFM vulnerability: a conditional branch (the `AppPublicKey` arm) silently skips the invariant check (output correctness), while the downstream state mutation (`resolve_yields_for`) proceeds unconditionally ŌĆö exactly the pattern of "check uses original value, effect uses transformed/unchecked value." ### Impact Explanation

**Critical ŌĆö Unauthorized confidential key derivation output without required participant authorization.**

A single Byzantine attested participant can:

1. Observe any pending `AppPublicKey` CKD request in contract state.
2. Construct an arbitrary `CKDResponse` (e.g., a response encrypting a key they control, or a random blob).
3. Call `respond_ckd` with the real `CKDRequest` and the forged `CKDResponse`.
4. The contract accepts it, resolves all queued yields, and delivers the forged key to the user.

The user receives a confidential key that was **not** derived from the MPC network's threshold BLS key. Because `AppPublicKey` is "privately verifiable," the user must decrypt the response with their app secret key to detect the forgery ŌĆö the contract provides no on-chain guarantee. The legitimate threshold-derived response is permanently suppressed (the request is gone from state). Any application relying on the derived key for cryptographic operations (e.g., decryption, authentication) will silently use an attacker-controlled value. [2](#0-1) 

---

### Likelihood Explanation

The attacker must be an **attested participant** ŌĆö a node that has passed TEE attestation and is registered in the contract's participant set. This is the explicit Byzantine-below-threshold threat model called out in scope. No threshold collusion is required; a single compromised or malicious node suffices. The attack requires only:

- Monitoring the NEAR chain for pending `request_app_private_key` calls using `AppPublicKey`.
- Calling `respond_ckd` before the legitimate leader does.

Because `respond_ckd` has no leader-only restriction ŌĆö any attested participant may call it ŌĆö a racing Byzantine node can reliably win by front-running the legitimate response. [3](#0-2) [4](#0-3) 

---

### Recommendation

Apply an equivalent output check for `AppPublicKey` requests. Since `AppPublicKey` is a single G1 point and the BLS pairing check requires a G2 point, on-chain verification is not directly possible with the current type. Two mitigations:

1. **Require `AppPublicKeyPV` for all new requests** and deprecate `AppPublicKey`. The `AppPublicKeyPV` variant already has a sound on-chain check via `ckd_output_check`.
2. **If `AppPublicKey` must be retained**, require a quorum (threshold-of-n) of attested participants to submit identical `CKDResponse` values before resolving the request, so a single Byzantine node cannot unilaterally determine the output. [4](#0-3)

### Proof of Concept

**Setup**: MPC network running with `n` participants, threshold `t`. Attacker controls one attested participant node `P_evil`.

**Steps**:

1. User `alice` calls `request_app_private_key` with:
   ```json
   {
     "app_public_key": { "AppPublicKey": "<alice_bls_g1_pubkey>" },
     "domain_id": 0,
     "derivation_path": "my-app"
   }
   ```
   The request is stored in `pending_ckd_requests`.

2. `P_evil` observes the pending `CKDRequest` on-chain (it is public state).

3. `P_evil` constructs a forged `CKDResponse` ŌĆö e.g., a BLS12-381 G1 point encrypting an attacker-chosen value ŌĆö and calls:
   ```
   respond_ckd(request=<alice's CKDRequest>, response=<forged CKDResponse>)
   ```

4. The contract executes `respond_ckd`:
   - `assert_caller_is_attested_participant_and_protocol_active()` ŌåÆ passes (`P_evil` is attested).
   - `match &request.app_public_key { AppPublicKey(_) => {} ... }` ŌåÆ **no check performed**.
   - `resolve_yields_for(...)` ŌåÆ alice's yield is resolved with the forged response.

5. Alice's NEAR transaction completes and returns the forged `CKDResponse`.

6. Legitimate nodes attempt `respond_ckd` for the same request ŌåÆ `resolve_yields_for` finds no pending yields (already drained) and returns `RequestNotFound`.

**Expected outcome**: Alice receives a confidential key not derived from the MPC threshold key. The attack requires one Byzantine attested participant and zero threshold collusion. [2](#0-1) [5](#0-4)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
