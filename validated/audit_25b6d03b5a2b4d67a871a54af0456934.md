### Title
Bare `assert` Statements in Weight Proof Segment Validation Crash Node Sync on Crafted Peer Input — (File: `chia/full_node/weight_proof.py`)

### Summary
The weight proof validation path in `chia/full_node/weight_proof.py` contains bare Python `assert` statements inside functions that process untrusted peer-supplied `WeightProof` data. A malicious peer can craft a `WeightProof` whose segment data triggers an `AssertionError` in `__validate_pospace` or `__get_cc_sub_slot`, crashing the validation coroutine and permanently blocking the node's ability to sync via weight proofs.

### Finding Description

The codebase's own test suite documents this vulnerability class under the label **SEC-614** and explicitly states:

> "A malicious peer could send a crafted proof where `first_idx == 0`; the old `assert first_idx` rejected this with an `AssertionError` instead of cleanly failing validation." [1](#0-0) 

The partial fix applied to `__get_rc_sub_slot` replaced `assert first_idx` with a guarded `return None`. However, two additional bare `assert` statements remain in the same validation call chain:

**1. `__get_cc_sub_slot` — `assert sub_slot is not None` (line 1539)**

```python
def __get_cc_sub_slot(sub_slots, idx, ses):
    sub_slot: SubSlotData | None = None
    for i in reversed(range(idx)):          # empty when idx == 0
        sub_slot = sub_slots[i]
        if sub_slot.cc_slot_end_info is not None:
            break
    assert sub_slot is not None             # AssertionError when idx == 0
    assert sub_slot.cc_slot_end_info is not None
``` [2](#0-1) 

When `idx == 0` and `sub_epoch_n > 0`, the guard in `__validate_pospace` does **not** short-circuit to `GENESIS_CHALLENGE`:

```python
if first_in_sub_epoch and segment.sub_epoch_n == 0 and idx == 0:
    cc_sub_slot_hash = constants.GENESIS_CHALLENGE
else:
    cc_sub_slot_hash = __get_cc_sub_slot(segment.sub_slots, idx, ses).get_hash()
``` [3](#0-2) 

**2. `__validate_pospace` — `assert sub_slot_data.proof_of_space is not None` (line 1424)**

```python
# validate proof of space
assert sub_slot_data.proof_of_space is not None
``` [4](#0-3) 

A crafted segment can supply a `SubSlotData` entry with `signage_point_index` set (making it appear to be a challenge block) but `proof_of_space = None`. This reaches line 1424 regardless of the `first_idx` fix.

**3. `__validate_pospace` — `assert curr_slot.cc_slot_end_info` (line 1413)**

```python
curr_slot = segment.sub_slots[idx - 1]
assert curr_slot.cc_slot_end_info
``` [5](#0-4) 

A crafted overflow-block entry whose predecessor sub-slot has `cc_slot_end_info = None` triggers this assert.

None of these `AssertionError` exceptions are caught in `_validate_sub_epoch_segments` or `validate_weight_proof_single_proc`:

```python
if _validate_sub_epoch_segments(self.constants, rng, wp_segment_bytes, summary_bytes, peak_height) is None:
    return False, uint32(0)
``` [6](#0-5) 

### Impact Explanation

Weight proofs are the mechanism by which nodes sync to the chain tip without replaying every block. A malicious peer that sends a crafted `WeightProof` triggering any of the above asserts causes an unhandled `AssertionError` to propagate out of the validation coroutine. The syncing node cannot complete weight proof validation and cannot advance its chain tip, resulting in a permanent sync stall for as long as the attacker continues to serve the crafted proof. This satisfies the **High** impact criterion: *"Permanent or long-lived inability for honest nodes… to process valid blocks, spend bundles, sync updates… under normal network assumptions."*

### Likelihood Explanation

Any unprivileged peer on the Chia network can respond to a `RequestProofOfWeight` message with an arbitrary `WeightProof` blob. The `WeightProof` type is deserialized from raw bytes via `Streamable`, which performs no semantic validation. Constructing a segment with `proof_of_space = None` and a non-None `signage_point_index` is trivial. The attacker does not need keys, admin access, or a cryptographic break.

### Recommendation

Replace all bare `assert` statements inside weight proof validation functions that process peer-supplied data with explicit guarded returns:

- In `__get_cc_sub_slot`: replace `assert sub_slot is not None` and `assert sub_slot.cc_slot_end_info is not None` with `if sub_slot is None or sub_slot.cc_slot_end_info is None: return None` (and update the return type to `ChallengeChainSubSlot | None`, propagating the `None` check in callers).
- In `__validate_pospace`: replace `assert curr_slot.cc_slot_end_info` with `if not curr_slot.cc_slot_end_info: return None` and replace `assert sub_slot_data.proof_of_space is not None` with `if sub_slot_data.proof_of_space is None: return None`.

Apply the same pattern used to fix `__get_rc_sub_slot` (lines 1462–1464 and 1491–1493). [7](#0-6) 

### Proof of Concept

1. Connect to a syncing Chia full node as a peer.
2. When the node sends `RequestProofOfWeight`, respond with a crafted `WeightProof` whose `sub_epoch_segments` contains at least one `SubEpochChallengeSegment` where a `SubSlotData` entry has `signage_point_index` set to a non-overflow value and `proof_of_space = None`.
3. Ensure the crafted segment is selected as the sampled segment (predictable from `summaries[-2].get_hash()` seeding the RNG).
4. The node's `_validate_sub_epoch_segments` call reaches `__validate_pospace` → line 1424 → `AssertionError` → unhandled exception → sync coroutine terminates → node cannot advance its chain tip.
