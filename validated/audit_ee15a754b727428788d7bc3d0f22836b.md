### Title
Consensus Validator Path Bypasses Stateless Transaction Validation, Allowing OS-Unprovable Transactions Into Committed Blocks — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The gateway path enforces a full suite of stateless checks on every incoming `RpcTransaction` before it enters the mempool. The consensus validator path — used when a validator node receives a proposal from a proposer — converts the same `RpcTransaction` type through `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal`, which performs **no stateless validation**. A malicious proposer can therefore include transactions that carry OS-forbidden fields (non-empty `paymaster_data`, non-empty `account_deployment_data`, or `nonce_data_availability_mode`/`fee_data_availability_mode` ≠ 0). The blockifier executes them successfully, the block is committed, and the Starknet OS then hard-asserts those fields are zero during proving, making the block permanently unprovable.

### Finding Description

**Gateway path (correct)**

`GenericGateway::add_tx_inner` calls `StatelessTransactionValidator::validate` before any conversion:

```
validate_empty_paymaster_data()
validate_empty_account_deployment_data()
validate_resource_bounds()
validate_nonce_data_availability_mode()   // enforces L1 (== 0)
validate_fee_data_availability_mode()     // enforces L1 (== 0)
validate_tx_size()
``` [1](#0-0) [2](#0-1) 

**Consensus validator path (missing checks)**

When a validator node receives `ProposalPart::Transactions`, `handle_proposal_part` calls:

```rust
transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
``` [3](#0-2) 

This resolves to `convert_rpc_tx_to_internal`, which for an `Invoke` transaction does nothing but strip the proof and convert the struct — **no stateless validation is invoked**:

```rust
RpcTransaction::Invoke(RpcInvokeTransaction::V3(tx)) => {
    let proof_data = if tx.proof_facts.is_empty() { None }
                     else { Some((tx.proof_facts.clone(), tx.proof.clone())) };
    (InternalRpcTransactionWithoutTxHash::Invoke(tx.into()), proof_data)
}
// ...
let tx_hash = tx_without_hash.calculate_transaction_hash(&self.chain_id)?;
``` [4](#0-3) 

The converted transaction is forwarded to the batcher via `send_txs_for_proposal` and executed by the blockifier. The blockifier's `perform_pre_validation_stage` checks only nonce, fee bounds, and proof facts — it does **not** check `paymaster_data`, `account_deployment_data`, or DA modes: [5](#0-4) 

**OS hard-asserts the missing checks**

The Starknet OS Cairo program contains unconditional `assert` statements (not soft reverts) for exactly the fields the stateless validator guards:

```cairo
assert common_fields.paymaster_data_length = 0;
assert common_fields.nonce_data_availability_mode = 0;
assert common_fields.fee_data_availability_mode = 0;
``` [6](#0-5) 

```cairo
assert account_deployment_data_size = 0;
``` [7](#0-6) 

A Cairo `assert` is a hard failure: it aborts the entire OS execution, not just the individual transaction. The block becomes permanently unprovable.

### Impact Explanation

A malicious proposer (any validator node selected for a round) crafts an `RpcInvokeTransactionV3` with `paymaster_data = [0x1]` (or non-zero `nonce_data_availability_mode`, etc.), signs it from their own account whose `__validate__` ignores `paymaster_data`, and includes it in a `ProposalPart::Transactions` batch. The validator node:

1. Calls `convert_consensus_tx_to_internal_consensus_tx` — no stateless check fires.
2. Sends the transaction to the batcher via `send_txs_for_proposal`.
3. The blockifier executes it successfully (blockifier does not assert `paymaster_data_length == 0`).
4. The block is committed; the state root is written to storage.
5. The OS hard-asserts `paymaster_data_length = 0` during proving and aborts.
6. The block is permanently unprovable; the sequencer is stuck at that height.

This is a **Critical** impact: wrong committed state that cannot be verified, matching "Wrong state … from blockifier/syscall/execution logic for accepted input" and "Invalid … transaction accepted through … paymaster … logic."

### Likelihood Explanation

Any validator node that wins a proposer slot can trigger this. No special privilege beyond normal consensus participation is required. The attack requires crafting a single transaction with a non-empty `paymaster_data` field, which is trivial. The validator node has no defense because the stateless check is entirely absent from the consensus path.

### Recommendation

Apply the same stateless validation in the consensus validator path that the gateway applies. The cleanest fix is to call `StatelessTransactionValidator::validate` (or an equivalent function) inside `convert_rpc_tx_to_internal` (or in `handle_proposal_part` before forwarding to the batcher), so that any `RpcTransaction` arriving via consensus is subject to the same field-level invariants as one arriving via the HTTP gateway.

Alternatively, add the individual field checks directly inside `convert_rpc_tx_to_internal`:
- Assert `paymaster_data.is_empty()`
- Assert `account_deployment_data.is_empty()`
- Assert `nonce_data_availability_mode == L1`
- Assert `fee_data_availability_mode == L1`

and return a `TransactionConverterError` on violation, causing `handle_proposal_part` to return `HandledProposalPart::Failed` and abort the proposal.

### Proof of Concept

1. Validator node A is selected as proposer for height H.
2. A constructs `RpcInvokeTransactionV3 { paymaster_data: vec![Felt::ONE], ... }` signed by A's own account (whose `__validate__` does not inspect `paymaster_data`).
3. A includes this transaction in `ProposalPart::Transactions` and broadcasts the proposal.
4. Validator node B receives the proposal. `handle_proposal_part` calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal`. No stateless check fires. The transaction is forwarded to the batcher.
5. The blockifier executes the transaction. `perform_pre_validation_stage` passes (nonce and fee bounds are valid). The `__validate__` entry point succeeds. The transaction is committed.
6. `ProposalFin` is received; `finish_proposal` returns the `ProposalCommitment`. Consensus reaches decision. `decision_reached` commits the block.
7. The prover runs the Starknet OS on the block. `hash_tx_common_fields` executes `assert common_fields.paymaster_data_length = 0` — this fails because `paymaster_data_length = 1`. The OS aborts. The block is unprovable. The sequencer is stuck at height H.
