### Title
Missing Persistent Deduplication for `verify_foreign_transaction` Allows Cross-Chain Replay of Already-Processed Foreign Transactions - (File: `crates/contract/src/lib.rs`)

### Summary

The `pending_verify_foreign_tx_requests` map uses `VerifyForeignTransactionRequest` — a struct containing no caller identity, no nonce, and no epoch — as its deduplication key. When a request is resolved, `resolve_yields_for` removes the map entry entirely with no persistent record of the processed foreign tx. Any unprivileged caller can immediately re-submit the identical `verify_foreign_transaction` request, causing the MPC network to re-verify and re-sign the same foreign chain transaction, enabling cross-chain replay and double-spend in bridge contracts that rely on the MPC contract as the source of truth for foreign tx finality.

### Finding Description

**Root cause — caller-agnostic, ephemeral key with no "used" set:**

`verify_foreign_transaction` in `crates/contract/src/lib.rs` converts the user-supplied args via `args_into_verify_foreign_tx_request`:

```rust
// crates/contract/src/dto_mapping.rs  lines 840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

The resulting `VerifyForeignTransactionRequest` struct — the map key — contains only `(request, domain_id, payload_version)`:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs  lines 124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

No caller account ID, no nonce, no epoch, no monotonic counter. This is structurally analogous to the reported nonce-reset bug: the identifier that should prevent replay carries no state that advances after each use.

**Resolution removes the only guard:**

When `respond_verify_foreign_tx` succeeds, `resolve_yields_for` removes the map entry:

```rust
// crates/contract/src/pending_requests.rs  lines 74-81
let resumed = requests
    .remove(request)          // ← entry deleted; no "used" set written
    .unwrap_or_default()
    ...
```

After removal, the contract holds zero memory that this `(tx_id, domain_id, payload_version)` tuple was ever processed. The same tuple can be re-submitted immediately.

**Re-submission is unrestricted:**

`verify_foreign_transaction` performs no historical lookup before enqueuing:

```rust
// crates/contract/src/lib.rs  lines 519-557
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(...);   // gas + deposit only
    // ← no check: "was this tx_id already processed?"
    let request = args_into_verify_foreign_tx_request(request);
    self.enqueue_yield_request(..., move |this, id| this.add_verify_foreign_tx_request(request, id));
}
```

The MPC nodes' local `VerifyForeignTransactionRequestStorage` deduplicates by receipt ID (a per-NEAR-transaction hash), not by foreign tx ID. A new NEAR transaction carrying the same `ForeignChainRpcRequest` produces a new receipt ID and is accepted as a fresh request by both the contract and the nodes.

**Signed payload is deterministic:**

`ForeignTxSignPayloadV1` contains only `(request, extracted_values)` — no timestamp, no block height, no epoch. Re-verification of the same finalized foreign tx produces the same `payload_hash` and a fresh valid MPC signature over it.

### Impact Explanation

**High — Cross-chain replay enabling double-spend.**

A bridge contract that calls `verify_foreign_transaction` to gate an inbound mint (e.g., "mint NEAR-BTC only after MPC attests this Bitcoin deposit is final") and that does not maintain its own processed-tx ledger can be drained:

1. Attacker deposits 1 BTC on Bitcoin → bridge calls `verify_foreign_transaction(tx_id=T)`.
2. MPC responds → bridge mints 1 NEAR-BTC → map entry for `T` is deleted.
3. Attacker re-calls `verify_foreign_transaction(tx_id=T)` (new NEAR tx, new receipt, same args).
4. MPC re-verifies the same finalized Bitcoin tx → produces another valid signature over the same `payload_hash`.
5. Bridge receives a second valid attestation → mints another 1 NEAR-BTC.
6. Repeat until bridge is drained.

The MPC contract's stated purpose is to let "NEAR contracts react to external chain events without a trusted relayer." Bridge contracts written against this API have a reasonable expectation that the MPC layer prevents re-attestation of the same foreign tx. The absence of a persistent used-set violates that contract.

### Likelihood Explanation

**Low.** Exploitation requires a bridge contract that does not implement its own tx-ID deduplication. However, the MPC contract's API design actively encourages bridge authors to omit that guard: the fan-out behavior (multiple callers receiving the same response for the same request) is explicitly documented and tested, signalling that the contract manages deduplication. A bridge author following the documented pattern would naturally omit a redundant check, leaving the double-spend window open.

### Recommendation

Maintain a persistent `LookupSet<ForeignTxId>` (or equivalent) in `MpcContract` state. Before enqueuing a `verify_foreign_transaction` request, check whether the foreign tx ID has already been resolved. Reject re-submissions with a typed error (e.g., `RequestError::ForeignTxAlreadyProcessed`). The set entry should be written inside `return_verify_foreign_tx_and_clean_state_on_success` on the success path, not inside `respond_verify_foreign_tx`, to ensure atomicity with the yield-resume resolution.

### Proof of Concept

```
// Step 1 – initial submission (any caller)
verify_foreign_transaction({ request: Bitcoin { tx_id: T, ... }, domain_id: D, payload_version: V1 })
  → pending_verify_foreign_tx_requests[K] = [yield_0]

// Step 2 – MPC responds
respond_verify_foreign_tx(K, response_0)
  → resolve_yields_for removes K from map
  → bridge contract receives response_0, mints 1 NEAR-BTC

// Step 3 – attacker re-submits (new NEAR tx, same args)
verify_foreign_transaction({ request: Bitcoin { tx_id: T, ... }, domain_id: D, payload_version: V1 })
  → pending_verify_foreign_tx_requests[K] = [yield_1]   // K is identical; no guard fires

// Step 4 – MPC re-verifies same finalized Bitcoin tx, responds again
respond_verify_foreign_tx(K, response_1)   // same payload_hash, fresh valid signature
  → bridge contract receives response_1, mints another 1 NEAR-BTC
```

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
