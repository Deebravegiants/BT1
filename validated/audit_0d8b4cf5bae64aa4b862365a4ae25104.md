### Title
`respond_verify_foreign_tx` Does Not Verify `payload_hash` Corresponds to the Submitted `request` — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx()` verifies that `response.signature` is a valid MPC signature over `response.payload_hash`, but never checks that `response.payload_hash` is actually the hash of a `ForeignTxSignPayload` derived from the accompanying `request`. A single Byzantine attested participant can reuse a valid MPC signature obtained from any prior signing session to resolve an unrelated pending `verify_foreign_transaction` request with a fabricated payload hash, delivering a forged foreign-chain attestation to the original caller.

### Finding Description

When a user calls `verify_foreign_transaction()`, the contract queues a pending yield keyed on the full `VerifyForeignTransactionRequest` (containing `domain_id`, `payload_version`, and `request`). [1](#0-0) 

MPC nodes are expected to independently query the foreign chain, compute `payload_hash = SHA-256(borsh(ForeignTxSignPayload{request, values}))`, obtain a threshold signature over it, and call `respond_verify_foreign_tx()` with the matching `request`, the computed `payload_hash`, and the signature.

The contract's verification in `respond_verify_foreign_tx()` only checks that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key: [2](#0-1) 

It never recomputes or constrains what `response.payload_hash` must be relative to `request`. The `payload_hash` is taken entirely from the caller-supplied `response` struct: [3](#0-2) 

Compare this to the regular `respond()` function, which takes the payload directly from the stored `request` (not from the caller-supplied response), so the signed message is always bound to the original request: [4](#0-3) 

The `ForeignTxSignPayloadV1` struct does include the `request` field in the signed data, so the hash is request-bound at the cryptographic level: [5](#0-4) 

But the contract never enforces that the `payload_hash` in the response is the hash of a payload containing the correct `request`. Any valid MPC signature over any 32-byte value satisfies the check.

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) who has observed the final signature output from any prior legitimate signing session for request Y holds a valid MPC signature over `H_Y`. It can call `respond_verify_foreign_tx` with:

- `request` = any currently-pending request X (observed from the blockchain)
- `response.payload_hash` = `H_Y` (hash from a completely different foreign-chain payload)
- `response.signature` = the valid MPC signature over `H_Y`

The contract accepts this call (signature is valid over `H_Y` under the domain key), resolves all pending yields for request X, and delivers `{payload_hash: H_Y, signature: sig_Y}` to the original caller. The caller receives what appears to be a valid MPC-attested foreign-chain verification response, but the `payload_hash` corresponds to a different transaction's data entirely.

Any bridge or application contract that does not independently recompute the expected `payload_hash` from the request and extracted values (i.e., does not use the SDK's `ForeignChainSignatureVerifier::verify_signature`) will accept this forged attestation as genuine, enabling invalid bridge execution or double-spend conditions. [6](#0-5) 

### Likelihood Explanation

- The attacker is a single Byzantine node that is an attested participant — no threshold collusion required.
- Every participant in a signing session observes the final signature output, so any participant can accumulate valid signatures over arbitrary payload hashes from prior sessions.
- The attacker only needs to race the legitimate response submission. Since the attacker can monitor the NEAR blockchain for newly queued `verify_foreign_transaction` requests and submit immediately, this race is winnable.
- The `resolve_yields_for` call drains all queued yields for the request key, so a single fabricated response poisons every caller waiting on that request. [7](#0-6) 

### Recommendation

The contract should verify that `response.payload_hash` is consistent with `request.request`. Since the contract does not know the extracted values, it cannot recompute the full hash. Two mitigations are possible:

1. **Bind the request to the signed payload at the contract level**: Require that the `payload_hash` is the hash of a `ForeignTxSignPayload` whose embedded `request` field matches `request.request`. This requires the contract to partially deserialize or re-hash the payload, which is expensive in WASM.

2. **Mirror the `respond()` pattern**: Store the expected `payload_hash` in the pending request map at submission time (once the MPC nodes agree on it via a commit-reveal or quorum step), and verify `response.payload_hash == stored_hash` before accepting the response.

3. **Minimum fix**: Document that callers **must** use `ForeignChainSignatureVerifier::verify_signature` to validate the response, and add an on-chain assertion that `response.payload_hash` is non-zero and matches a hash prefix derived from the request (e.g., include `domain_id` and `request` hash in the signed payload so the contract can at least check the domain binding).

### Proof of Concept

1. User calls `verify_foreign_transaction` for Bitcoin tx X → pending request X is queued.
2. The MPC network legitimately processes request Y (Bitcoin tx Y), producing signature `sig_Y` over `H_Y = SHA-256(borsh(ForeignTxSignPayload{request: Bitcoin(tx_Y), values: [BlockHash(hash_Y)]}))`.
3. Byzantine node (single attested participant) observes `sig_Y` from the signing session output.
4. Byzantine node calls:
   ```
   respond_verify_foreign_tx(
     request = <pending request X>,
     response = { payload_hash: H_Y, signature: sig_Y }
   )
   ```
5. Contract checks: `verify_ecdsa_signature(sig_Y, H_Y, domain_pubkey)` → **passes** (signature is valid).
6. Contract resolves all pending yields for request X with `{payload_hash: H_Y, signature: sig_Y}`.
7. Original caller of `verify_foreign_transaction` for Bitcoin tx X receives a response attesting to Bitcoin tx Y's data.
8. Any bridge contract that does not call `ForeignChainSignatureVerifier::verify_signature` accepts this as a valid attestation of Bitcoin tx X. [8](#0-7) [9](#0-8)
