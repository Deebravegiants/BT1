Audit Report

## Title
`addSupportedToken` Does Not Initialize `tokenFeeBps`, Allowing Fee-Free Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains a per-token fee mapping `tokenFeeBps` that defaults to `0` for any newly added token. `addSupportedToken` never initializes this mapping, and `deposit(token, amount, referralId)` is immediately callable by any unprivileged user after the token is added. Every deposit made before the admin separately calls `setTokenFeeBps` charges zero fee, permanently losing that protocol fee revenue.

## Finding Description
`tokenFeeBps` is declared at line 88 as a plain mapping with no default initialization:

```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
```

`addSupportedToken` (lines 637–656) only writes `supportedTokenOracle[token]` and `tokenBridge[token]`; it never touches `tokenFeeBps[token]`, leaving it at the Solidity default of `0`.

`viewSwapRsETHAmountAndFee` (lines 335–336) reads this mapping directly:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```

With `feeBpsForToken == 0`, `fee` is always `0`, and `feeEarnedInToken[token]` accumulates nothing (line 300). The depositor receives rsETH computed on the full `amount` with no deduction.

Setting the fee requires a completely separate call to `setTokenFeeBps` (lines 583–594), which is gated by `DEFAULT_ADMIN_ROLE` — a different role from `TIMELOCK_ROLE` used by `addSupportedToken`. There is no enforcement that `setTokenFeeBps` must be called before or atomically with `addSupportedToken`, and no guard in `deposit` that checks whether a fee has been explicitly configured.

The `onlySupportedToken` modifier (lines 100–103) only checks that `supportedTokenOracle[token] != address(0)`, which is satisfied immediately after `addSupportedToken` executes. No other check blocks deposits.

## Impact Explanation
Every token deposit made while `tokenFeeBps[token] == 0` charges zero fee. The depositor receives rsETH for the full deposited amount, and `feeEarnedInToken[token]` remains at zero. Protocol fee revenue for all deposits in this window is permanently and irrecoverably lost. This constitutes **High — Theft of unclaimed yield**: the protocol's expected fee yield is diverted to depositors who receive excess rsETH.

## Likelihood Explanation
`addSupportedToken` is gated by `TIMELOCK_ROLE`, meaning it goes through a timelock delay and is publicly visible on-chain before execution. Any observer can front-run the execution or simply deposit immediately after the timelock fires and before the admin submits a separate `setTokenFeeBps` transaction. Even without deliberate front-running, the two-step configuration creates a guaranteed fee-free window between two separate transactions. No special privileges or unusual conditions are required for the depositor.

## Recommendation
Pass the initial `feeBps` as a parameter to `addSupportedToken` and set it atomically:

```solidity
function addSupportedToken(address token, address oracle, address bridge, uint256 _feeBps)
    external onlyRole(TIMELOCK_ROLE)
{
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    // ... existing checks ...
    tokenFeeBps[token] = _feeBps;
    // ... rest of function ...
}
```

Alternatively, add a guard in `deposit(token, ...)` that reverts if `tokenFeeBps[token]` has not been explicitly set (e.g., using a separate `isTokenFeeConfigured` flag or a sentinel value distinct from `0`).

## Proof of Concept
1. Admin queues `addSupportedToken(wstETH, oracle, bridge)` through the timelock — publicly visible on-chain.
2. Timelock executes: `tokenFeeBps[wstETH]` is `0`, `supportedTokenOracle[wstETH]` is set.
3. Depositor (or front-runner) calls `deposit(wstETH, 100 ether, "ref")` immediately.
4. `viewSwapRsETHAmountAndFee(100 ether, wstETH)` computes `fee = 100 ether * 0 / 10_000 = 0`.
5. Depositor receives rsETH for the full `100 ether`; `feeEarnedInToken[wstETH]` remains `0`.
6. Admin later calls `setTokenFeeBps(wstETH, 30)` — fee revenue from step 3–5 is permanently lost.

Foundry test plan: deploy `RSETHPool` with a mock oracle and mock token; call `addSupportedToken`; immediately call `deposit` from an unprivileged address; assert `feeEarnedInToken[token] == 0` and that the depositor received rsETH equal to the full amount conversion with no fee deducted.
