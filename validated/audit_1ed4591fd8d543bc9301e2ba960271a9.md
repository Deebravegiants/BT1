Audit Report

## Title
Incomplete Pause Enforcement: `_transfer` Lacks `whenNotPaused`, Allowing ERC20 Transfers While Paused - (File: contracts/RSETH.sol)

## Summary

`RSETH` applies `whenNotPaused` to `mint` and `burnFrom` but omits it from the overridden `_transfer` function. Because all ERC20 `transfer` and `transferFrom` calls route through `_transfer`, any rsETH holder can freely move tokens while the contract is paused. The pause mechanism fails to deliver its stated guarantee of halting all token activity.

## Finding Description

`RSETH` inherits `PausableUpgradeable` and exposes `pause()` / `unpause()`: [1](#0-0) 

`mint` is guarded: [2](#0-1) 

`burnFrom` is guarded: [3](#0-2) 

The `_transfer` override, which is the sole code path for every `transfer` and `transferFrom` call, has no `whenNotPaused` check: [4](#0-3) 

The root cause is a straightforward omission: the developer guarded the privileged entry points (`mint`, `burnFrom`) but did not extend the same guard to the public transfer path. The existing `_enforceNotBlocked` checks inside `_transfer` address a different concern (per-address blocks) and do not substitute for a pause check.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The contract's pause mechanism is advertised as a full halt of token activity (evidenced by `whenNotPaused` on both `mint` and `burnFrom`). When paused, `transfer` and `transferFrom` continue to succeed, meaning the contract does not deliver the security guarantee it promises. No user funds are directly stolen and no yield is lost; the failure is that the pause does not achieve its intended effect for the transfer path. The submitted claim of "Medium — Temporary freezing of funds" is incorrect: that impact class describes funds being frozen when they should not be, which is the opposite of what occurs here. The correct classification is Low.

## Likelihood Explanation

Any rsETH holder can call `transfer` or `transferFrom` at any time, including while the contract is paused. No special privilege, setup, or front-running is required. The condition is simply that the contract is paused, which is a normal operational state triggered by `PAUSER_ROLE`.

## Recommendation

Add `whenNotPaused` to the `_transfer` override:

```solidity
function _transfer(address from, address to, uint256 amount)
    internal
    override
    whenNotPaused
{
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
``` [4](#0-3) 

## Proof of Concept

1. Deploy `RSETH` and mint tokens to `alice`.
2. Call `RSETH.pause()` as `PAUSER_ROLE`.
3. Confirm `mint` reverts with `EnforcedPause`.
4. Call `rsETH.transfer(bob, amount)` from `alice`.
5. Observe the transfer succeeds — `_transfer` at L287–291 has no pause check, so execution reaches `super._transfer` and completes normally.
6. Confirm `bob` now holds the transferred balance despite the contract being paused.

A minimal Foundry test:
```solidity
function test_transferWhilePaused() public {
    vm.prank(pauser);
    rsETH.pause();

    vm.prank(alice);
    rsETH.transfer(bob, 1 ether); // should revert, but succeeds
    assertEq(rsETH.balanceOf(bob), 1 ether);
}
``` [4](#0-3)
