### Title
Dijkstra Sub-Transaction Reference Script Fee Bypass via Inconsistent Size Computation — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`)

---

### Summary

In the Dijkstra era, the per-transaction reference script size limit check uses `batchNonDistinctRefScriptsSize` (which aggregates reference scripts across the top-level transaction **and all its sub-transactions**), while the minimum fee calculation uses `txNonDistinctRefScriptsSize` (which only counts reference scripts in the **top-level transaction**). This inconsistency allows an attacker to include large reference scripts exclusively in sub-transactions, bypassing the exponential tiered ref-script fee that was specifically designed to deter DDoS attacks, while still passing the size limit check.

---

### Finding Description

The Dijkstra era introduces nested transactions (sub-transactions embedded in a top-level transaction body). Two separate computations over reference script sizes are performed during validation:

**1. Size limit check** — `validateAllRefScriptSize` in `dijkstraLedgerTransition`:

```haskell
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) ...
```

`batchNonDistinctRefScriptsSize` is defined as:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

This sums ref script sizes from the top-level transaction **and** all sub-transactions.

**2. Fee calculation** — `getConwayMinFeeTxUtxo` (reused from Conway, imported into Dijkstra):

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

`txNonDistinctRefScriptsSize` only inspects `referenceInputsTxBodyL` and `inputsTxBodyL` of the **top-level** transaction body — sub-transactions are never consulted:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

The result is a structural split:

| Check | Function used | Scope |
|---|---|---|
| Size limit (`ppMaxRefScriptSizePerTxG`) | `batchNonDistinctRefScriptsSize` | Top-level + all sub-txs |
| Ref-script fee (`tierRefScriptFee`) | `txNonDistinctRefScriptsSize` | Top-level only |

An attacker who places all reference scripts inside sub-transactions pays **zero** exponential ref-script fee, while the node must still deserialize and process those scripts. The size limit check still fires if the combined total exceeds `ppMaxRefScriptSizePerTxG` (200 KiB), but within that cap the attacker can force up to 200 KiB of ref-script deserialization work at the cost of only the linear byte-size fee component.

This is the direct analog of the external report's pattern: a value (ref-script size) is computed once for a limit/accounting check and computed again — using a different, narrower function — for the actual fee charged, with the two computations yielding different results for the same transaction.

---

### Impact Explanation

**Medium — Attacker-controlled transactions modify fees outside design parameters.**

The exponential `tierRefScriptFee` was introduced specifically after a real DDoS attack on June 25, 2024 (documented in `docs/adr/2024-08-14_009-refscripts-fee-change.md`) to make large-ref-script transactions prohibitively expensive. By routing all reference scripts through sub-transactions, an attacker can submit a transaction with up to 200 KiB of reference scripts while paying the ref-script fee for 0 bytes. For 200 KiB at current parameters, the avoided fee is substantial (the tiered formula grows exponentially past 25 KiB increments). This restores the pre-fix DDoS attack surface within the Dijkstra era.

---

### Likelihood Explanation

Any unprivileged transaction submitter can exploit this. No special keys, governance roles, or majority are required. The attacker only needs to:
1. Possess UTxOs containing large reference scripts (or create them).
2. Construct a top-level transaction with no reference inputs, embedding sub-transactions that reference those UTxOs.

The attack is deterministic and reproducible.

---

### Recommendation

Replace the fee-calculation call in `getConwayMinFeeTxUtxo` (or provide a Dijkstra-specific override of `getMinFeeTxUtxo`) so that it uses `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize` when computing the ref-script size passed to `getMinFeeTx`. This ensures the fee and the size-limit check operate on the same scope:

```haskell
-- Dijkstra-specific override
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx
```

The `EraUTxO DijkstraEra` instance should bind `getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo` rather than inheriting `getConwayMinFeeTxUtxo`.

---

### Proof of Concept

1. Deploy UTxOs containing large Plutus reference scripts (e.g., 50 KiB each × 4 = 200 KiB total).
2. Construct a top-level Dijkstra transaction with:
   - `inputsTxBodyL` = one ordinary UTxO to spend (no reference scripts)
   - `referenceInputsTxBodyL` = empty
   - `subTransactionsTxBodyL` = four sub-transactions, each with `referenceInputsTxBodyL` pointing to one 50 KiB script UTxO
3. Compute the fee: `getConwayMinFeeTxUtxo pp tx utxo` → `txNonDistinctRefScriptsSize = 0` → ref-script fee = `tierRefScriptFee(..., 0)` = **0 Lovelace**.
4. Compute the size check: `batchNonDistinctRefScriptsSize utxo tx` = 0 + 200 KiB = 200 KiB ≤ `ppMaxRefScriptSizePerTxG` → **passes**.
5. The transaction is accepted. The node deserializes 200 KiB of reference scripts at a cost equivalent to a transaction that would have paid the full exponential fee (≈ several ADA at current parameters) had those scripts been in the top-level transaction.
