Audit Report

## Title
`getCurrentDay()` Arithmetic Underflow Causes `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` to Revert Before `startTimestamp` - (File: contracts/pools/RSETHPoolV2.sol, contracts/pools/RSETHPoolV3.sol)

## Summary
`getCurrentDay()` performs an unchecked `block.timestamp - startTimestamp` subtraction in Solidity 0.8.27, which reverts with an arithmetic panic whenever `block.timestamp < startTimestamp`. Because `reinitialize` enforces that `startTimestamp` must be at or after the current block, there is always a window during which the public view functions `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` — both of which delegate unconditionally to `getCurrentDay()` — revert instead of returning usable data.

## Finding Description
In both `RSETHPoolV2` and `RSETHPoolV3`, `getCurrentDay()` is implemented as:

```solidity
function getCurrentDay() public view returns (uint256) {
    return (block.timestamp - startTimestamp) / 1 days;
}
``` [1](#0-0) [2](#0-1) 

`reinitialize` enforces that `_startTimestamp` cannot be in the past:

```solidity
if (block.timestamp > _startTimestamp) {
    revert InvalidStartTimestamp();
}
``` [3](#0-2) [4](#0-3) 

This guarantees a window — from the moment `reinitialize` is called until `block.timestamp >= startTimestamp` — during which `block.timestamp - startTimestamp` underflows and reverts under Solidity 0.8.27 checked arithmetic.

Both `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` call `getCurrentDay()` unconditionally: [5](#0-4) [6](#0-5) 

By contrast, the `limitDailyMint` modifier — the only state-changing caller of `getCurrentDay()` — is correctly guarded before invoking it: [7](#0-6) [8](#0-7) 

The view functions have no equivalent guard, so any public caller triggers the revert during the pre-`startTimestamp` window.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` are public informational view functions intended to be queried by off-chain clients, integrators, and front-ends. During the pre-`startTimestamp` window they revert unconditionally, returning no usable data. No funds are at risk and no state-changing path is broken, but the contract fails to deliver its promised informational interface.

## Likelihood Explanation
**Medium.** Every deployment that calls `reinitialize` with a future `startTimestamp` — which is the only valid input — creates this window. The window length equals `startTimestamp - block.timestamp` at the time of `reinitialize`, which can be hours or days. No special attacker action is required; any public caller triggers the revert. The same pattern is present in both `RSETHPoolV2` and `RSETHPoolV3`.

## Recommendation
Add a pre-condition guard in `getCurrentDay()` that returns a safe sentinel value when `block.timestamp < startTimestamp`, mirroring the guard already present in `limitDailyMint`:

```solidity
function getCurrentDay() public view returns (uint256) {
    if (block.timestamp < startTimestamp) return 0;
    return (block.timestamp - startTimestamp) / 1 days;
}
```

Alternatively, `remainingDailyMintLimit()` can explicitly return `dailyMintLimit` (full limit, no minting has occurred yet) when `block.timestamp < startTimestamp`, and `getNextDailyLimitResetTimestamp()` can return `startTimestamp` directly in that case.

## Proof of Concept
1. Admin calls `reinitialize(dailyMintLimit, block.timestamp + 1 days)` on `RSETHPoolV2` or `RSETHPoolV3`.
2. `startTimestamp` is set to `block.timestamp + 1 days`.
3. Any external caller immediately calls `remainingDailyMintLimit()`.
4. Internally, `getCurrentDay()` computes `block.timestamp - startTimestamp` = `block.timestamp - (block.timestamp + 1 days)`, a negative value, triggering an arithmetic underflow revert in Solidity 0.8.27.
5. `remainingDailyMintLimit()` reverts with a panic instead of returning the expected `dailyMintLimit`.
6. Same revert occurs for `getNextDailyLimitResetTimestamp()`.

**Foundry test sketch:**
```solidity
function test_remainingDailyMintLimit_revertsBeforeStart() public {
    uint256 futureStart = block.timestamp + 1 days;
    vm.prank(admin);
    pool.reinitialize(1000 ether, futureStart);
    // Should revert with arithmetic panic
    vm.expectRevert(stdError.arithmeticError);
    pool.remainingDailyMintLimit();
}
```
