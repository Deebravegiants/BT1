### Title
Receipt `max_receipt_size` invariant bypassed via post-validation `output_data_receivers` mutation in `promise_return` path — (`runtime/runtime/src/lib.rs`)

### Summary
A contract can craft a receipt whose serialized size is exactly at `max_receipt_size`, pass the `NewReceipt` size check, and then have the runtime append `output_data_receivers` to it **after** validation via the `promise_return` code path. The resulting oversized receipt is forwarded cross-shard without re-validation, violating the `max_receipt_size` protocol invariant. The nearcore team tracks this as issue #12606 and the test suite explicitly documents it as an unresolved bug.

### Finding Description

**Vulnerability class**: Unbounded user-controlled data bypassing a size-limit check because the check occurs before a post-validation mutation that increases the payload size. This is the direct nearcore analog of the OFTCore `_toAddress` bug: a user-controlled input passes a size gate, is then enlarged by the protocol itself, and the enlarged payload is forwarded to a downstream consumer without re-checking the limit.

**Root cause — two-phase mutation without re-validation**

In `runtime/runtime/src/lib.rs`, after each action in a receipt is applied, newly generated receipts are validated: [1](#0-0) 

`validate_receipt` with `ValidateReceiptMode::NewReceipt` checks the Borsh-serialized size against `max_receipt_size`: [2](#0-1) 

However, **after** this validation loop completes, the runtime enters the `promise_return` path. If the contract returned `ReturnData::ReceiptIndex`, the runtime appends the current action receipt's `output_data_receivers` directly onto the validated receipt: [3](#0-2) 

No size re-check occurs after this mutation. The receipt's Borsh size now exceeds `max_receipt_size` by the size of the appended `DataReceiver` entries, but it is forwarded as a normal outgoing receipt.

**Receiving-shard processing skips the size check**

When the oversized receipt arrives at the destination shard, it is processed with `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size gate: [4](#0-3) 

The `ExistingReceipt` mode was added precisely because this bug already allows oversized receipts to exist in the state.

**Congestion-control workaround acknowledges the bug**

The congestion-control layer contains an explicit workaround that clamps the reported size of any receipt exceeding `max_receipt_size` to avoid it getting stuck in the outgoing buffer: [5](#0-4) 

The comment directly references issue #12606.

**Test suite confirms the bug is reachable and unresolved**

`test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` constructs a receipt of exactly `max_receipt_size`, triggers `promise_return`, and asserts that an oversized receipt is observed in the chain — confirming the bug is live: [6](#0-5) [7](#0-6) 

A second path exists via `value_return`: returning a value of size `max_receipt_size` produces a `DataReceipt` whose total Borsh size exceeds `max_receipt_size`, because `validate_data_receipt` only checks `max_length_returned_data`, not the full receipt size: [8](#0-7) [9](#0-8) 

### Impact Explanation

The concrete corrupted protocol value is the **receipt size field in the outgoing receipts Merkle tree**: a receipt whose Borsh-serialized size exceeds `max_receipt_size` (4,194,304 bytes) is committed to the outgoing receipts root and forwarded cross-shard. This violates the `max_receipt_size` protocol invariant that every node is supposed to enforce. Downstream consequences include:

1. **State witness size**: The oversized receipt contributes to the state witness of the receiving shard. If the receipt is large enough, it can push the witness past `main_storage_proof_size_soft_limit` (4 MB), causing chunk validators to reject the witness.
2. **Congestion-control accounting error**: The clamping workaround means the outgoing-size accounting underestimates the true byte cost of the receipt, potentially allowing more data to be forwarded than the bandwidth scheduler intends.
3. **Protocol invariant violation**: Any node that independently recomputes the outgoing receipts root will include the oversized receipt; nodes that enforce `max_receipt_size` strictly would diverge.

### Likelihood Explanation

The attacker path requires only unprivileged user actions:

1. Deploy a contract (any account can do this).
2. Call the contract with a `then` callback — this causes the outer receipt to have non-empty `output_data_receivers`.
3. Inside the contract, use `promise_create` with args sized to bring the inner receipt to exactly `max_receipt_size - overhead`, then call `promise_return`.
4. The runtime appends the outer receipt's `output_data_receivers` to the inner receipt post-validation, producing a receipt above `max_receipt_size`.

No validator, node-operator, or privileged access is required. The `max_arguments_length` limit (4 MB) and `max_receipt_size` (4 MB) are close enough that a carefully sized `args` payload can reach the target size. The test contract `max_receipt_size_promise_return_method1` in `runtime/near-test-contracts/test-contract-rs/src/lib.rs` demonstrates the exact construction.

### Recommendation

Re-validate the receipt size **after** appending `output_data_receivers` in the `promise_return` path (`runtime/runtime/src/lib.rs` lines 1028–1035). Specifically, after the `extend_from_slice` calls, call `validate_receipt(..., ValidateReceiptMode::NewReceipt)` again and propagate any `ReceiptSizeExceeded` error as a `NewReceiptValidationError`, consistent with how other post-action receipt validation errors are handled at lines 856–865.

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete proof of concept: [10](#0-9) 

The test deploys a contract, calls `max_receipt_size_promise_return_method1` with `args_size = max_receipt_size - base_receipt_size`, and then asserts via `assert_oversized_receipt_occurred` that a receipt with `receipt_size > max_receipt_size` was observed in the chain — confirming the invariant is violated on a live test network. [11](#0-10)

fn validate_data_receipt(
    limit_config: &LimitConfig,
    receipt: &DataReceipt,
) -> Result<(), ReceiptValidationError> {
    let data_len = receipt.data.as_ref().map(|data| data.len()).unwrap_or(0);
    if data_len as u64 > limit_config.max_length_returned_data {
        return Err(ReceiptValidationError::ReturnedValueLengthExceeded {
            length: data_len as u64,
            limit: limit_config.max_length_returned_data,
        });
    }
    Ok(())

/// Assert that there was an incoming receipt with size above max_receipt_size
fn assert_oversized_receipt_occurred(node: &TestLoopNode<'_>) {
    let client = node.client();
    let chain = &client.chain;
    let epoch_manager = &*client.epoch_manager;

    let tip = chain.head().unwrap();
    let epoch_id = epoch_manager.get_epoch_id(&tip.last_block_hash).unwrap();
    let protocol_version = epoch_manager.get_epoch_protocol_version(&epoch_id).unwrap();
    let runtime_config = client.runtime_adapter.get_runtime_config(protocol_version);
    let max_receipt_size = runtime_config.wasm_config.limit_config.max_receipt_size;

    let mut block = chain.get_block(&tip.last_block_hash).unwrap();

    // Go over all blocks down to genesis looking for a receipt above max_receipt_size.
    loop {
        if block.header().is_genesis() {
            panic!("Didn't find receipt with size above max_receipt_size!");
        }
        let prev_block = chain.get_block(block.header().prev_hash()).unwrap();

        let shard_layout = epoch_manager
            .get_shard_layout(&epoch_manager.get_epoch_id(block.hash()).unwrap())
            .unwrap();

        let oversized = if ProtocolFeature::Spice.enabled(protocol_version) {
            // With spice chunks are executed asynchronously and their produced receipts are
            // persisted as receipt proofs keyed by the block in which the chunk was applied,
            // rather than as incoming receipts on the following block.
            shard_layout.shard_ids().any(|shard_id| {
                chain
                    .chain_store()
                    .iter_receipt_proofs_for_shard(block.hash(), shard_id)
                    .iter()
                    .flat_map(|proof| proof.0.iter())
                    .any(|receipt| receipt_is_oversized(receipt, max_receipt_size))
            })
        } else {
            block.chunks().iter_new().any(|new_chunk| {
                let shard_id = new_chunk.shard_id();
                let prev_shard_index = epoch_manager
                    .get_prev_shard_id_from_prev_hash(block.header().prev_hash(), shard_id)
                    .unwrap()
                    .2;
                let prev_height_included =
                    prev_block.chunks().get(prev_shard_index).unwrap().height_included();
                let incoming_receipts_proofs = get_incoming_receipts_for_shard(
                    &chain.chain_store,
                    epoch_manager,
                    shard_id,
                    &shard_layout,
                    *block.hash(),
                    prev_height_included,
                    ReceiptFilter::TargetShard,
                )
                .unwrap();
                incoming_receipts_proofs
                    .iter()
                    .flat_map(|response| response.1.iter())
                    .flat_map(|proof| proof.0.iter())
                    .any(|receipt| receipt_is_oversized(receipt, max_receipt_size))
            })
        };

        if oversized {
            return;
        }

        block = prev_block;
    }
}

fn receipt_is_oversized(receipt: &Receipt, max_receipt_size: u64) -> bool {
    let receipt_size: u64 = borsh::object_length(receipt).unwrap().try_into().unwrap();
    if receipt_size > max_receipt_size {
        tracing::info!(%receipt_size, %max_receipt_size, "found receipt above max size");
        return true;
    }
    false
}
