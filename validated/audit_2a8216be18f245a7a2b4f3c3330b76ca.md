Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Comparison — (`File: contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric limit check: for ERC20 assets it correctly tests `totalAssetDeposits + amount > limit`, but for ETH it tests only `totalAssetDeposits > limit`, omitting the incoming `amount`. This allows any depositor to push total ETH deposits above the configured cap whenever `totalAssetDeposits == depositLimit`, because the check evaluates `limit > limit` → `false` and permits the deposit.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
```

The ETH branch uses strict `>` without adding `amount`, so the check only rejects deposits when the cap is **already exceeded**. A deposit that brings the total from exactly `limit` to `limit + amount` evaluates as `limit > limit` → `false` → not exceeded → accepted.

This function is called unconditionally at line 661 inside `_beforeDeposit`, which is the sole pre-flight guard invoked by both `depositETH` (line 87) and `depositAsset` (line 111). No other ETH deposit cap enforcement exists. The `nonReentrant` and `whenNotPaused` modifiers on `depositETH` do not constrain the limit logic.

## Impact Explanation
**Low — Contract fails to deliver promised returns.**
The ETH deposit cap is a protocol-level risk parameter. Because the incoming `amount` is excluded from the ETH branch comparison, the cap is silently unenforced: depositors can mint rsETH and deposit ETH beyond the configured ceiling. No direct fund theft or permanent freeze results, but the protocol accepts more ETH exposure than it is configured to handle, violating its own deposit limit invariant. This matches the allowed Low impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
**Medium.** The precondition `totalAssetDeposits == depositLimit` is a normal operational state (cap fully reached). Any unprivileged depositor monitoring on-chain state can call `depositETH` at that moment with no special privileges, front-running, or brute-force required. The condition is repeatable: each over-limit deposit increases `totalAssetDeposits` above `limit`, but subsequent deposits still pass because `totalAssetDeposits > limit` remains `false` only until the very first over-limit deposit; after that, `totalAssetDeposits > limit` becomes `true` and blocks further deposits. However, the single bypass at the boundary is sufficient to violate the invariant, and the window is predictable and observable.

## Recommendation
Add `amount` to the ETH branch to mirror the ERC20 branch, eliminating the asymmetry:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. Protocol accumulates `getTotalAssetDeposits(ETH_TOKEN) == 1000 ether` (cap exactly reached).
3. Attacker calls `depositETH{value: 100 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → limit not exceeded → no revert.
5. `_mintRsETH` mints rsETH for the attacker; total ETH deposits become `1100 ether`, 10% above the configured cap.
