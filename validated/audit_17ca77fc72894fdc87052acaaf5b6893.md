### Title
Gateway Admits Transactions That Blockifier Rejects Due to Stale L2 Gas Price Threshold — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` uses the **previous committed block's** `l2_gas_price` as the admission threshold, while the blockifier executes the next block using the **EIP-1559-adjusted** `next_l2_gas_price`. When gas usage is high and `next_l2_gas_price > previous_block_l2_gas_price`, transactions admitted by the gateway with `max_price_per_unit ∈ [previous_block_l2_gas_price, next_l2_gas_price)` are rejected by the blockifier with `ResourceBoundsError::MaxGasPriceTooLow`, corrupting the admitted transaction set.

### Finding Description

**Gateway admission path:**

`validate_resource_bounds` reads the threshold from the latest committed block's `BlockInfo`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` populates this from `block_header.l2_gas_price.price_in_fri` — the **current block's** l2 gas price, not the next block's: [2](#0-1) 

The admission check is: `tx.max_price_per_unit >= (min_gas_price_percentage% × P_N)`. With the default `min_gas_price_percentage = 100`, the threshold is exactly `P_N`. [3](#0-2) 

**Blockifier execution path:**

The batcher builds block N+1 using `block_metadata.block_info` provided by the consensus orchestrator. The consensus orchestrator's `l2_gas_price` field holds `P_{N+1}` — already computed via `calculate_next_l2_gas_price_for_fin` from block N's gas usage: [4](#0-3) 

The `BlockBuilderFactory` creates the `BlockContext` directly from `block_metadata.block_info`, so the blockifier sees `l2_gas_price = P_{N+1}`: [5](#0-4) 

The blockifier's pre-validation then checks `resource_bounds.max_price_per_unit < actual_gas_price.get()` where `actual_gas_price` is `P_{N+1}`: [6](#0-5) 

**EIP-1559 price increase:**

`calculate_next_base_gas_price` can raise the price by up to `price / gas_price_max_change_denominator` per block when `gas_used > gas_target`. Under sustained high load, `P_{N+1}` can be meaningfully above `P_N`: [7](#0-6) 

**The gap:** Gateway threshold = `P_N`. Blockifier threshold = `P_{N+1}`. Any transaction with `max_price_per_unit ∈ [P_N, P_{N+1})` passes gateway admission but fails blockifier pre-validation.

### Impact Explanation

- **High**: Gateway/mempool admission accepts transactions that blockifier rejects. The admitted transaction set is corrupted — transactions that passed all gateway checks are silently dropped during block building with `ResourceBoundsError::MaxGasPriceTooLow`.
- Users who set `max_price_per_unit` exactly at the gateway-advertised threshold (the previous block's price) will have their transactions admitted but never included in a block.
- Under sustained high load (the exact condition that triggers EIP-1559 price increases), this discrepancy is maximized and persistent across every block.

### Likelihood Explanation

- Requires no special privileges — any user submitting a V3 (`AllResources`) transaction is affected.
- Triggered automatically whenever `l2_gas_used > gas_target` for the previous block, which is a normal operating condition under load.
- The TODO comment in the code explicitly acknowledges this is the wrong value: `// TODO(Arni): getnext_l2_gas_price from the block header.` [8](#0-7) 

### Recommendation

Replace the `get_block_info()` call with a read of `BlockHeaderWithoutHash::next_l2_gas_price` from the latest committed block header. The `GatewayFixedBlockStateReader` trait (or a new companion trait) should expose `next_l2_gas_price` directly from the stored `BlockHeaderWithoutHash`, which already carries this field: [9](#0-8) 

The `GatewayFixedBlockSyncStateClient` already fetches the full `block_header_without_hash`; it just needs to surface `next_l2_gas_price` instead of (or in addition to) `l2_gas_price`. [10](#0-9) 

### Proof of Concept

1. Let block N have `l2_gas_price = P_N = 1000` and `l2_gas_used >> gas_target`, so `next_l2_gas_price = P_{N+1} = 1125` (12.5% increase).
2. User submits a V3 invoke tx with `l2_gas.max_price_per_unit = 1000`.
3. Gateway calls `validate_resource_bounds`: threshold = `100% × 1000 = 1000`. Since `1000 >= 1000`, the transaction is **admitted**.
4. Batcher builds block N+1 with `block_info.gas_prices.strk_gas_prices.l2_gas_price = 1125`.
5. Blockifier pre-validation: `1000 < 1125` → `ResourceBoundsError::MaxGasPriceTooLow` → transaction **rejected**.
6. The transaction was admitted to the mempool but will never be included in a block.
