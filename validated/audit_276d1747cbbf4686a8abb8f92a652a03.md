### Title
`respond_ckd` Skips Cryptographic Output Verification for `AppPublicKey` Variant, Allowing a Single Attested Participant to Forge Confidential Key Derivation Output — (`crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` contract method applies a cryptographic pairing check (`ckd_output_check`) only when the CKD request used the `AppPublicKeyPV` variant. When the legacy `AppPublicKey` variant is used, the response is accepted with **zero verification** — any attested participant can submit arbitrary `(big_y, big_c)` values and the contract will resolve the user's yield with those forged values. This is a direct analog to the LES bug: one code path (the `AppPublicKey` branch) is missing a critical step that the parallel code path (`AppPublicKeyPV`) correctly performs.

---

### Finding Description

In `respond_ckd`, the verification logic is gated on the variant of `app_public_key` stored in the request:

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk, app_id), mpc_pk)`, which cryptographically binds the response to the MPC master secret key and the user's ephemeral key pair. [2](#0-1) 

For `AppPublicKey`, the empty arm `{}` means **any** `(big_y, big_c)` pair is accepted unconditionally. The existing test confirms this: it passes completely arbitrary byte arrays (`[1u8; 48]`, `[2u8; 48]`) as the response and the contract accepts them without error. [3](#0-2) 

The `respond` method for ECDSA/EdDSA always verifies the signature against the derived public key before resolving the yield — there is no analogous unverified branch. [4](#0-3) 

**Attack path:**

1. A user calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(A)` (the legacy/privately-verifiable variant), paying the required deposit. The request is queued as a pending yield. [5](#0-4) 
2. A single attested MPC participant (Byzantine, below threshold) calls `respond_ckd` with the matching `CKDRequest` and a forged `CKDResponse { big_y: identity_point, big_c: attacker_chosen_point }`.
3. The contract passes the `assert_caller_is_attested_participant_and_protocol_active` check, enters the `AppPublicKey` arm, performs no verification, and calls `resolve_yields_for`, resolving the user's yield with the forged values. [6](#0-5) 
4. The user's app receives `(big_y, big_c)` and computes `sig = big_c + (−a) · big_y`. With `big_y = identity`, this simplifies to `sig = big_c` — a value the attacker chose and already knows.
5. The user computes `s = HKDF(sig)`, deriving a key that the attacker can independently compute, fully compromising the confidentiality guarantee of CKD.

The `AppPublicKey` variant is documented as "privately verifiable" — meaning only the app holding `a` can verify the result. This is precisely why the on-chain check was omitted. However, the omission means the threshold property is silently broken: instead of requiring t-of-n nodes to contribute honest shares, a single Byzantine participant can substitute the entire output.

---

### Impact Explanation

**Critical.** A single attested MPC participant (Byzantine, strictly below the signing threshold) can deliver a CKD output they fully control to any user who submitted a `request_app_private_key` call using the `AppPublicKey` variant. The attacker can make the derived key `s` be any value they precompute, completely defeating the confidentiality and integrity guarantee of the CKD protocol. The user's app receives no error and has no on-chain mechanism to detect the forgery.

---

### Likelihood Explanation

**Medium.** The `AppPublicKey` variant is the legacy/default format and is actively used (documented in README examples, localnet fixtures, and e2e tests). Any single attested participant — a realistic Byzantine adversary below threshold — can race to call `respond_ckd` before the honest leader does. The attacker only needs to observe the pending request on-chain and submit first.

---

### Recommendation

Apply the same cryptographic output verification to the `AppPublicKey` branch. Since the contract cannot perform a pairing check without `pk2`, the fix is to require callers to use `AppPublicKeyPV` for all new requests, or to add an off-chain proof (e.g., a BLS signature over the response committed to the request) that the contract can verify for the `AppPublicKey` variant. At minimum, the `AppPublicKey` branch should be deprecated and gated behind a config flag, with a migration path to `AppPublicKeyPV`.

---

### Proof of Concept

The existing unit test at `crates/contract/src/lib.rs:3404` is itself the proof of concept. It submits `CKDResponse { big_y: [1u8; 48], big_c: [2u8; 48] }` — values that are not even valid BLS12-381 curve points — and `respond_ckd` accepts them without error, resolving the user's yield with the forged data. [7](#0-6) 

A concrete exploit using the identity-point trick:

```
big_y = G1::identity()   // compressed: 0xc0 || 0x00 * 47
big_c = r * G1           // attacker picks scalar r, computes r·G1

// User computes:
sig = big_c + (−a) · big_y = r·G1 + (−a)·0 = r·G1

// Attacker already knows r, so they know sig = r·G1
// Attacker computes s = HKDF(r·G1) — the user's "secret" key
```

The contract accepts this response for any pending `AppPublicKey` CKD request because the `AppPublicKey` arm in `respond_ckd` is an empty no-op. [8](#0-7)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
