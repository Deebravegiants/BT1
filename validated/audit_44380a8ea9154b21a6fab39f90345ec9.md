### Title
Missing CKD Output Validation for Legacy `AppPublicKey` Variant Allows Single Byzantine Participant to Hijack Pending Requests - (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_ckd` function in the MPC contract performs no cryptographic validation of the CKD response when the request uses the legacy `CKDAppPublicKey::AppPublicKey` variant. A single attested-but-malicious MPC participant (below the signing threshold) can call `respond_ckd` with an arbitrary, unvalidated response — including invalid BLS12-381 G1 curve points or fabricated values — and the contract will accept and deliver it, permanently resolving the pending request and preventing any legitimate response from ever being returned.

---

### Finding Description

In `respond_ckd` at `crates/contract/src/lib.rs` lines 675–682, the contract branches on the `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no validation whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` validates that the response points (`big_c`, `big_y`) satisfy the correct pairing equation and are in the prime-order subgroup, delegating to `bls12381_pairing_check`: [2](#0-1) 

For the legacy `AppPublicKey` variant, the `=> {}` arm performs **zero** validation. The `CKDResponse` contains BLS12-381 G1 points (`big_c`, `big_y`) that are accepted as raw bytes with no curve-membership, subgroup, or pairing check. This is the direct analog of the gnark report: cryptographic output values (curve points) are accepted without any field-element or subgroup validation.

The `K256Signature` / `K256AffinePoint` / `K256Scalar` types used in `respond` are similarly raw byte arrays at the DTO layer: [3](#0-2) 

However, for ECDSA `respond`, the `verify_ecdsa_signature` call delegates to `ecrecover` with `malleability_flag=true`, which implicitly validates `r < n` and `s < n/2`: [4](#0-3) 

No equivalent implicit validation exists for the `AppPublicKey` CKD path.

**Attacker entry path:**
1. A user submits `request_app_private_key` with `app_public_key: AppPublicKey(...)` (the legacy single-G1-point variant).
2. The request is queued in `pending_ckd_requests`.
3. A single malicious attested participant calls `respond_ckd` with a crafted `CKDResponse` containing arbitrary bytes for `big_c` and `big_y` — e.g., all-`0xFF` bytes that are not a valid BLS12-381 G1 point.
4. The contract passes the `AppPublicKey` branch with no validation, then calls `resolve_yields_for`, which permanently resolves the pending yield with the fraudulent response.
5. The user's promise resolves with the fraudulent CKD output; no legitimate response can ever be delivered for that request. [5](#0-4) 

---

### Impact Explanation

**Medium.** A single Byzantine MPC participant (below the signing threshold) can permanently hijack any pending legacy `AppPublicKey` CKD request by delivering a fraudulent, cryptographically invalid response. The contract resolves the yield immediately, making it impossible for the legitimate threshold-computed response to be returned. This breaks the request-lifecycle safety invariant: the user receives garbage key material instead of the threshold-derived confidential key, and the request is consumed. If the user does not independently verify the response (which is the expected behavior for the "privately verifiable" legacy variant), they may attempt to use invalid key material, causing downstream failures in their application.

---

### Likelihood Explanation

**Medium.** The attacker must be a single attested MPC participant — a realistic Byzantine threat model explicitly within scope. No threshold collusion is required. The `AppPublicKey` variant is documented as the legacy path still in active use. The attack requires only a single malicious `respond_ckd` call with crafted bytes, which is trivially constructable.

---

### Recommendation

Add the same pairing-based output validation for the `AppPublicKey` variant that already exists for `AppPublicKeyPV`. At minimum, validate that `big_c` and `big_y` in the response are valid, prime-order BLS12-381 G1 points before resolving the yield. If full pairing verification is not possible without the second key component, reject `AppPublicKey` requests at the contract level or require nodes to include a proof of correct computation.

---

### Proof of Concept

A malicious attested participant constructs:

```rust
let fraudulent_response = CKDResponse {
    big_c: Bls12381G1PublicKey([0xFF; 48]),  // not a valid G1 point
    big_y: Bls12381G1PublicKey([0xFF; 48]),
};
contract.respond_ckd(request_with_app_public_key_variant, fraudulent_response);
// → passes the AppPublicKey branch with no validation
// → resolve_yields_for delivers fraudulent bytes to the user
// → pending request is permanently consumed
```

The `AppPublicKey` arm at line 676 is a no-op `{}`, so no panic or error is raised. The `resolve_yields_for` call at line 684 then permanently resolves the yield with the fraudulent payload. [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(app_pk1, g2) = e(g1, app_pk2)`.
///
/// Point validation is fully delegated to the host: the decompression
/// functions abort execution on malformed or off-curve encodings, and
/// `bls12381_pairing_check` returns `false` when a point is outside its
/// prime-order subgroup.
pub(crate) fn app_public_key_check(app_public_key: &dtos::CKDAppPublicKeyPV) -> bool {
    let pk1 = env::bls12381_p1_decompress(&app_public_key.pk1);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);

    let pairing_input = [
        pk1.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        G1_GENERATOR_UNCOMPRESSED.as_slice(),
        pk2.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
