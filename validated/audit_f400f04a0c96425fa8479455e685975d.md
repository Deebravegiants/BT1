### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Enables Targeted Queue-Saturation DoS - (File: `crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

`verify_foreign_transaction` uses a request key (`VerifyForeignTransactionRequest`) that contains no caller identity. Because the fan-out queue is keyed solely on the foreign-chain transaction content, any unprivileged account can saturate the 128-slot queue for a specific foreign transaction, permanently blocking legitimate users from submitting the same verification request until the attacker stops refilling.

---

### Finding Description

`SignatureRequest` binds the caller's identity into the map key via a `tweak` derived from `predecessor_id` and `path`:

```rust
// crates/near-mpc-crypto-types/src/sign.rs:118-125
pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
    let tweak = crate::kdf::derive_tweak(predecessor_id, path);
    SignatureRequest { domain_id: domain, tweak, payload }
}
```

`VerifyForeignTransactionRequest`, however, carries **no caller field**:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

The contract itself logs the predecessor but never folds it into the stored key:

```rust
// crates/contract/src/lib.rs:519-556
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    // predecessor is logged but never used in the key
    let request = args_into_verify_foreign_tx_request(request);  // no predecessor passed
    self.enqueue_yield_request(..., move |this, id| this.add_verify_foreign_tx_request(request, id));
}
```

The fan-out queue enforces a hard cap of 128 yields per key:

```rust
// crates/contract/src/pending_requests.rs:37,51-57
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
}
```

The codebase itself documents and tests that different callers share one queue entry:

```rust
// crates/contract/src/lib.rs:3242-3255
// "And: caller bob submits the identical request — a different account would today
//  be blocked from receiving a response by alice's submission."
// "Then: both yields are queued under the single (caller-agnostic) request key."
```

An attacker submits 128 `verify_foreign_transaction` calls for a target `(chain, tx_id, domain_id)` tuple. The queue is now full. Every subsequent call from any legitimate user for that same foreign transaction panics with `PendingRequestQueueFull`. The attacker can continuously refill the queue after each drain cycle (MPC response or timeout) at a cost of 128 × 1 yoctoNEAR per cycle.

---

### Impact Explanation

A legitimate bridge user whose funds depend on verifying a specific foreign-chain transaction (e.g., a Bitcoin tx_id that triggers fund release) is permanently denied service for that transaction as long as the attacker keeps the queue saturated. This breaks the request-lifecycle safety invariant of the `verify_foreign_transaction` flow without requiring any privileged access, threshold collusion, or network-level interference. The attacker pays essentially zero (128 yoctoNEAR per cycle) to sustain the attack indefinitely.

This maps to: **Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

---

### Likelihood Explanation

- Any unprivileged NEAR account can call `verify_foreign_transaction` with 1 yoctoNEAR deposit.
- The target `(chain, tx_id, domain_id)` tuple is observable on-chain before the victim's request lands.
- Filling 128 slots costs 128 yoctoNEAR and fits in a single batch of cross-contract calls (the test harness itself demonstrates this pattern via `make_duplicate_sign_calls`).
- The attack is repeatable every drain cycle with no additional setup.

---

### Recommendation

Include `env::predecessor_account_id()` in the `VerifyForeignTransactionRequest` key, mirroring the `tweak`-based isolation used by `SignatureRequest`. This ensures each caller's queue slot is independent and an attacker can only saturate their own slots, not those of other users.

Alternatively, enforce a per-account rate limit or per-account slot cap inside `push_pending_yield` for the foreign-tx map.

---

### Proof of Concept

1. Victim observes a Bitcoin transaction `tx_id = [0xAB; 32]` they need to verify to release bridged funds.
2. Attacker front-runs by submitting 128 calls to `verify_foreign_transaction` with `{ request: Bitcoin { tx_id: [0xAB;32], ... }, domain_id: 0, payload_version: V1 }`, each with 1 yoctoNEAR deposit.
3. `push_pending_yield` fills the queue to `MAX_PENDING_REQUEST_FAN_OUT = 128`.
4. Victim submits the same request → contract panics: `"Pending-request queue is full (limit: 128)"`.
5. MPC nodes respond; all 128 attacker yields drain. Attacker immediately resubmits 128 requests.
6. Victim's retry is again rejected. The cycle repeats indefinitely at ~128 yoctoNEAR per cycle.

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```
