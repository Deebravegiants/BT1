### Title
User-Controlled Finality Parameter Allows MPC Signing for Unfinalized Foreign Transactions, Enabling Forged Foreign-Chain Verification via Reorg - (`crates/foreign-chain-inspector/src/evm/inspector.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint accepts a fully user-controlled `finality` field (including `EvmFinality::Latest` for EVM chains and an arbitrarily low `BlockConfirmations` for Bitcoin). The MPC contract enforces no minimum finality level. MPC nodes faithfully sign payloads for transactions that are only in the "latest" (unfinalized) block. If the foreign chain undergoes a reorg after signing, the MPC network has issued a valid threshold signature attesting to a transaction that never finalized, enabling forged foreign-chain verification and double-spend in bridge protocols — a direct analog to the Uniswap V2 spot-price oracle manipulation described in the reference report.

---

### Finding Description

**Attacker-controlled entry point**: Any unprivileged NEAR account can call `verify_foreign_transaction` and freely set the `finality` field inside `EvmRpcRequest` (or `confirmations` inside `BitcoinRpcRequest`). [1](#0-0) 

The `ForeignTxSignPayload` that MPC nodes sign is `SHA-256(borsh(ForeignTxSignPayload))`, which includes the full `ForeignChainRpcRequest` (including the user-chosen `finality`/`confirmations`) and the extracted values. [2](#0-1) 

**No minimum finality enforcement in the contract**: `verify_foreign_transaction` checks only that the chain is in the supported-chains list and that gas/deposit preconditions are met. It does not validate or restrict the finality level. [3](#0-2) 

The on-chain `ChainEntry` configuration stores a `quorum` field but no minimum finality requirement. [4](#0-3) 

**`EthereumFinality::Latest` is accepted and processed**: The `EthereumFinality` enum explicitly includes `Latest`. [5](#0-4) 

When `Latest` is used, `verify_finality_level` fetches the latest block and checks only that `head.number >= receipt_block_number`. A transaction in the very latest block passes this check immediately. [6](#0-5) 

**Bitcoin: user-controlled `confirmations` with no floor**: The `BitcoinRpcRequest.confirmations` field is a raw `u64`. Setting it to `1` (or even `0`) causes the inspector to accept a transaction with a single confirmation — far below the 6-confirmation standard for Bitcoin finality. [7](#0-6) 

**`FanOut` tolerates `NotFinalized` as transient**: If one RPC provider returns success and another returns `NotFinalized`, `FanOut` classifies `NotFinalized` as transient and returns the single success verdict. This means a single provider's "latest block" view is sufficient to proceed with signing even when other providers disagree. [8](#0-7) [9](#0-8) 

**MPC nodes use the user-specified finality directly**: In `execute_foreign_chain_request`, the finality value from the user's request is passed directly to the inspector without any override or floor enforcement. [10](#0-9) 

---

### Impact Explanation

Once the MPC network produces a threshold signature over `ForeignTxSignPayload{request, values}` where `request.finality == Latest`, that signature is cryptographically valid and irrevocable. If the foreign chain reorgs the block containing the attested transaction, the signature still passes `respond_verify_foreign_tx`'s verification: [11](#0-10) 

A bridge contract that does not independently validate the finality level embedded in the signed payload will accept this signature and release funds on NEAR for a foreign-chain transaction that was reorganized away — a direct double-spend. This matches the **High** allowed impact: *forged foreign-chain verification that causes invalid bridge execution or double-spend conditions*.

---

### Likelihood Explanation

- **EVM chains (BNB, Polygon, Abstract, HyperEVM)**: 1–2 block reorgs occur naturally with non-negligible frequency. An attacker submits a transaction, immediately calls `verify_foreign_transaction` with `finality: Latest`, receives the MPC signature (seconds to low minutes), and then benefits from a natural reorg. No active reorg-forcing is required.
- **Bitcoin with `confirmations: 1`**: Single-confirmation Bitcoin transactions are reorganized occasionally; the attacker sets `confirmations: 1` to minimize the finality bar.
- **Ethereum mainnet post-merge**: Reorgs are rare, making this attack harder there, but the vulnerability is chain-agnostic and the contract/node code applies identically to all configured EVM chains.

---

### Recommendation

1. **Enforce a minimum finality level on-chain**: Add a `min_finality` field to `ChainEntry` and reject `verify_foreign_transaction` requests whose `finality` is weaker than the on-chain minimum.
2. **Enforce a minimum `BlockConfirmations` floor for Bitcoin**: Reject requests with `confirmations < N` (e.g., `N = 6`) at the contract level.
3. **Remove `EthereumFinality::Latest` from the accepted set** for the `verify_foreign_transaction` path, or gate it behind an explicit governance opt-in per chain.
4. **Classify `NotFinalized` as non-transient in `FanOut`**: A provider reporting "not finalized" is giving a substantive verdict, not a transient network failure; treating it as transient allows a single optimistic provider to override a majority that correctly reports the transaction is not yet final.

---

### Proof of Concept

1. Attacker submits a transaction `T` on BNB chain (3-second blocks, occasional 1-block reorgs).
2. Attacker calls `verify_foreign_transaction` with `finality: EvmFinality::Latest` immediately after `T` lands in block `N`.
3. MPC nodes call `eth_getTransactionReceipt` → receipt found in block `N`; call `eth_getBlockByNumber("latest")` → head ≥ `N`; call `eth_getBlockByNumber(N)` → canonical hash matches. All checks pass. Nodes sign `ForeignTxSignPayload{request(finality=Latest), values=[blockHash(N)]}`.
4. Block `N` is reorganized; `T` is evicted from the canonical chain.
5. Attacker submits the valid MPC signature to the bridge contract on NEAR. The bridge contract calls `respond_verify_foreign_tx`; the contract verifies the ECDSA signature against the payload hash — it is valid. Funds are released.
6. `T` never finalized on BNB; the attacker has received NEAR-side bridge funds for a transaction that does not exist on the canonical BNB chain.

        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }
