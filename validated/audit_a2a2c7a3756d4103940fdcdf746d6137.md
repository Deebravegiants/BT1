### Title
`ethValueInWithdrawal` in `LRTConverter` Fails to Scale by the Original Asset Price When Assets Are Returned — (`contracts/LRTConverter.sol`)

---

### Summary

`LRTConverter` maintains `ethValueInWithdrawal` to represent the ETH-denominated value of assets currently being unstaked. When assets are moved from the deposit pool to the converter, the value is recorded at the **current** oracle price. When those same assets are returned to the deposit pool, the deduction also uses the **current** oracle price — which may have changed. This stale-price mismatch causes `ethValueInWithdrawal` to be permanently over- or under-stated, corrupting the rsETH price calculation for all users.

---

### Finding Description

`transferAssetFromDepositPool` records the ETH value of assets at price **P1** (time of transfer): [1](#0-0) 

```solidity
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
```

`transferAssetToDepositPool` later deducts using price **P2** (time of return): [2](#0-1) 

```solidity
uint256 assetValue = (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
ethValueInWithdrawal = ethValueInWithdrawal > assetValue ? ethValueInWithdrawal - assetValue : 0;
```

Because the same `_amount` of the same asset is added at P1 and subtracted at P2, any price drift between the two operations leaves `ethValueInWithdrawal` permanently incorrect:

- **P1 > P2** (price fell): `ethValueInWithdrawal` retains phantom ETH value → rsETH price is inflated.
- **P1 < P2** (price rose): `ethValueInWithdrawal` is over-subtracted and clamped to 0 → rsETH price is deflated, losing real accounting.

A second path exists via `_sendEthToDepositPool` (called by `claimStEth`/`claimSwEth`): it subtracts the raw ETH amount received from Lido, not the oracle-valued amount that was originally added, producing the same residual mismatch. [3](#0-2) 

---

### Impact Explanation

`ethValueInWithdrawal` is consumed directly by `LRTDepositPool.getETHDistributionData()`: [4](#0-3) 

```solidity
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
```

This feeds `getTotalAssetDeposits(ETH)` → `_getTotalEthInProtocol()` → `rsETHPrice` in `LRTOracle`: [5](#0-4) 

Any phantom value in `ethValueInWithdrawal` directly inflates `totalETHInProtocol`, which inflates the rsETH price. New depositors receive fewer rsETH tokens than they are entitled to (value transferred to existing holders). Conversely, when `ethValueInWithdrawal` is zeroed out prematurely, the rsETH price is deflated and existing holders receive less ETH on withdrawal.

**Impact: Low** — Contract fails to deliver promised returns; no direct fund theft, but systematic mis-accounting of rsETH price affects every depositor and withdrawer.

---

### Likelihood Explanation

LST oracle prices (stETH, ETHx, etc.) drift continuously as staking rewards accrue. Any gap between `transferAssetFromDepositPool` and `transferAssetToDepositPool` (or `claimStEth`) — which in practice spans days to weeks due to Lido's withdrawal queue — guarantees a non-zero price delta. This is a normal operational flow, not an edge case.

**Likelihood: Medium** — Triggered by routine operator operations with no malicious intent required; price drift is inevitable over the withdrawal window.

---

### Recommendation

Store the ETH value at the time of addition and use that stored value for the deduction, rather than re-querying the oracle at return time:

```solidity
// In transferAssetFromDepositPool:
uint256 addedValue = (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
ethValueInWithdrawal += addedValue;
// Store addedValue keyed by (asset, amount, nonce) for later deduction.

// In transferAssetToDepositPool:
ethValueInWithdrawal -= storedValue; // use the originally recorded value
```

Alternatively, track `ethValueInWithdrawal` per-asset in token units and convert to ETH only at read time using the current price, so the accounting is always consistent.

---

### Proof of Concept

1. Operator calls `transferAssetFromDepositPool(stETH, 100e18)` when stETH oracle price = **1.10 ETH** → `ethValueInWithdrawal = 110e18`.
2. stETH price drifts down to **1.05 ETH** (normal market movement over days).
3. Operator calls `transferAssetToDepositPool(stETH, 100e18)` → deducts `100 × 1.05 = 105e18` → `ethValueInWithdrawal = 5e18` (phantom).
4. `_getTotalEthInProtocol()` now counts 5 ETH that does not exist in the protocol.
5. rsETH price is inflated; the next depositor who sends 100 ETH receives fewer rsETH than they should, with the difference silently accruing to existing holders.
