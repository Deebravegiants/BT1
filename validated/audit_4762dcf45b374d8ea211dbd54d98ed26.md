### Title
VMLogic Register Count Check Blocks Updates at Limit, Reducing Effective Register Capacity - (File: runtime/near-vm-runner/src/logic/vmstate.rs)

### Summary
The `check_set_register` function in the NEAR VM runtime enforces the `max_number_registers` cap using `>=` before resolving whether the write targets an existing register or a new slot. This causes any write — including updates to already-occupied registers — to return `MemoryAccessViolation` once the register map is full, making the effective usable register capacity lower than the protocol-declared limit for read-modify-write patterns.

### Finding Description
In `Registers::check_set_register`, the count guard fires unconditionally before the `entry()` call that would reveal whether the target `register_id` is already occupied:

```rust
// Fun fact: if we are at the limit and we replace a register, we'll
// fail even though we should be succeeding.  This bug is now part of
// the protocol so we can't change it.
if self.registers.len() as u64 >= config.max_number_registers {
    return Err(HostError::MemoryAccessViolation.into());
}

let entry = self.registers.entry(register_id);   // ← never reached when full
```

Because `self.registers.len()` is evaluated before `entry()`, the code cannot distinguish between:
- **Adding** a new register (which should be blocked when full), and
- **Replacing** an existing register (which should always succeed regardless of count).

When `registers.len() == max_number_registers` (100 on mainnet), every subsequent `set` call — even to a register that already exists — returns `MemoryAccessViolation`. The protocol documentation in `LimitConfig` explicitly acknowledges this:

> "Note that due to an implementation quirk [read: a bug] in VMLogic, if we have this number of registers, no subsequent writes to the registers will succeed even if they replace an existing register."

The unit test `registers_max_number_limit` confirms the behaviour: after filling 2 registers with `max_number_registers = 2`, attempting to overwrite register 42 with a new value fails.

### Impact Explanation
Any smart contract that:
1. Writes to all 100 registers (filling the map), then
2. Attempts to update the value of any existing register

will receive `HostError::MemoryAccessViolation` and the receipt will fail. The receipt outcome (failure vs. success) is part of the **outcome root** committed in the block header. Contracts relying on a read-modify-write pattern across the full register set will produce a failed receipt where a correct implementation would produce a successful one, corrupting the expected execution outcome for that receipt.

### Likelihood Explanation
Moderate. The trigger requires a contract to saturate all 100 registers and then attempt an in-place update. Contracts performing iterative cryptographic operations (e.g., accumulating BLS12-381 partial results across many host-function calls into the same register) or contracts that cache many intermediate values are realistic candidates. An attacker can craft and deploy such a contract via a normal signed transaction, requiring no special privileges.

### Recommendation
Move the count guard to apply only when the target entry is `Vacant`:

```rust
let entry = self.registers.entry(register_id);
if matches!(entry, Entry::Vacant(_))
    && self.registers.len() as u64 >= config.max_number_registers
{
    return Err(HostError::MemoryAccessViolation.into());
}
```

Because the current behaviour is documented as "part of the protocol," this fix requires a protocol version gate to avoid consensus divergence between old and new nodes.

### Proof of Concept
The existing test `registers_max_number_limit` in `runtime/near-vm-runner/src/logic/vmstate.rs` already demonstrates the bug:

```rust
ctx.cfg.max_number_registers = 2;
ctx.assert_set_success(42, "foo");   // register 42 written
ctx.assert_set_success(24, "bar");   // register 24 written  (count == 2 == limit)
ctx.assert_set_failure(12, "baz");   // new register blocked  ← correct
ctx.assert_set_failure(42, "O_o");   // UPDATE to existing register 42 blocked ← bug
ctx.assert_set_failure(24, "O_o");   // UPDATE to existing register 24 blocked ← bug
```

A deployed WASM contract that writes 100 distinct registers and then calls any host function that writes back to one of those registers (e.g., `storage_get` into an already-used `register_id`) will fail with `MemoryAccessViolation`, producing a failed receipt outcome instead of the expected successful one.
