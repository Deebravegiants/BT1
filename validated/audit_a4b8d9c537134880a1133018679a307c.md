### Title
Missing `x_share * G == root_Q` Binding Check Allows Attacker-Controlled Derived Public Key in HD EdDSA-2P Keyset - (`src/cbmpc/api/hd_keyset_eddsa_2p.cpp`)

### Summary

`blob_to_keyset` accepts a `keyset_blob` where `root_Q` is any valid curve point independent of `x_share`. When p1 sets `root_Q = x_share_p1 * G` (their own share's public key), the derived key's `Q` equals `derived_x_share_p1 * G`, giving p1 the full private key for the derived key and allowing unilateral signing without p2.

### Finding Description

In `blob_to_keyset` (`src/cbmpc/api/hd_keyset_eddsa_2p.cpp`, lines 63–81), the deserialization path:
- Checks `q.is_in_range(blob.x_share)` ✓
- Calls `keyset.root.Q.from_bin(keyset.curve, blob.root_Q_compressed)` ✓
- **Never verifies `x_share * G == root_Q`** ✗

In `key_share_eddsa_hdmpc_2p_t::derive_keys` (`src/cbmpc/protocol/hd_keyset_eddsa_2p.cpp`):
- `Q = key.root.Q` is taken directly from the blob (line 109)
- `Q_derived = Q + delta * G` (line 149) — uses attacker's `Q`
- `non_hard_delta = non_hard_derive(Q_derived, ...)` (line 150) — derived from attacker's `Q_derived`
- `derived_keys[i].Q = Q_derived + non_hard_delta_G` (line 159)
- For p1: `derived_keys[i].x_share = x_share + delta + non_hard_delta[i]` (line 161)

The `Q` field is **never exchanged with or verified by the peer**. The ZK-DH proof only binds `k_share` to `K_share`; it says nothing about `x_share` vs `root_Q`.

**Attack construction:**

Let p1 craft a blob with `x_share = s` (any valid scalar) and `root_Q = s * G` (their own share's public key, which they can compute). After running `derive_eddsa_2p_keys` with honest p2:

- p1 computes: `Q_derived_p1 = s*G + delta*G`
- p1 computes: `non_hard_delta_p1 = non_hard_derive(Q_derived_p1, chain_code, paths)`
- p1's derived key: `derived_x_share_p1 = s + delta + non_hard_delta_p1[i]`
- p1's derived key: `derived_Q_p1 = (s + delta + non_hard_delta_p1[i]) * G = derived_x_share_p1 * G`

P1 now holds the **complete private key** `derived_x_share_p1` for the derived public key `derived_Q_p1`. They can produce valid standard EdDSA signatures under `derived_Q_p1` without any participation from p2.

### Impact Explanation

A malicious p1 can:
1. Craft a keyset blob with `root_Q = x_share_p1 * G`
2. Participate honestly in `derive_eddsa_2p_keys` (the ZK-DH proof on `k_share` passes)
3. Obtain a derived key blob where `Q = derived_x_share_p1 * G`
4. Sign arbitrary messages unilaterally using `derived_x_share_p1` via standard EdDSA — bypassing the 2-party signing requirement entirely

Any application that uses `get_public_key_compressed` on p1's derived blob to obtain the "authoritative" public key will accept these unilateral signatures as valid. This breaks the core security property of the 2-party scheme.

### Likelihood Explanation

The attack requires p1 to control their own keyset blob (which they always do — it is their own persisted state) and to participate in one `derive_eddsa_2p_keys` session with an honest p2. No threshold collusion, no leaked keys, no compromised host memory is needed. The manipulation is a single field substitution in a blob that the attacker owns.

### Recommendation

Add a consistency check in `blob_to_keyset` after decoding `root_Q`:

```cpp
// After decoding root_Q from blob:
ecc_point_t expected_Q_share = keyset.curve.mul_to_generator(keyset.root.x_share);
// Cannot check full Q = x_share_p1*G + x_share_p2*G from a single party's blob,
// but at minimum reject if root_Q == x_share*G (trivial single-party key).
// Better: require both parties to cross-check Q during derive_keys by exchanging
// their x_share public points and verifying Q == Q_p1 + Q_p2.
```

The proper fix is to have `derive_keys` exchange and verify each party's `x_share * G` contribution and confirm they sum to `root_Q`, analogous to how `dkg` uses `eckey::dkg_2p_t` with ZK proofs of knowledge of the discrete log. [5](#0-4) 

### Proof of Concept

```
1. p1 picks x_share = 1, computes root_Q = 1*G (Ed25519 base point).
2. p1 crafts a keyset_blob_v1_t with x_share=1, root_Q=G, k_share=<any valid scalar>, root_K_compressed=<matching K>.
3. p1 and p2 call derive_eddsa_2p_keys with this blob (p2 uses their honest blob).
4. Protocol completes: delta is derived from Z = Z1+Z2 (both ZK-DH proofs pass).
5. p1's derived key: Q = G + delta*G + non_hard_delta*G = (1 + delta + non_hard_delta)*G.
6. p1's derived x_share = 1 + delta + non_hard_delta.
7. Assert: derived_x_share * G == derived_Q  ← holds, p1 has the full private key.
8. p1 signs a message using standard Ed25519 with scalar (1 + delta + non_hard_delta).
9. Verify signature against derived_Q from p1's blob → SUCCESS, without p2's participation.
```
