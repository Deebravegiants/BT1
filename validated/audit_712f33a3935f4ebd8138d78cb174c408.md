### Title
Multiple Sub-Transactions Can Double-Claim Stake Deposit Refunds in Dijkstra Batch Transactions - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, a batch transaction containing multiple sub-transactions can include `UnRegDepositTxCert` certificates for the **same** stake credential in each sub-transaction. The `dijkstraUtxoTransition` function validates batch-level reward withdrawals via `validateBatchWithdrawals` (which aggregates across all sub-transactions and checks against the original account balance), but has **no equivalent batch-level check for deposit refunds**. The disabled test `xit "Multiple subtransactions cannot get the same refund"` in `CertSpec.hs` explicitly acknowledges this gap with `error "TODO: predicate failure not yet implemented"`, confirming the protection is absent.

---

### Finding Description

The `validateBatchWithdrawals` function correctly prevents double-withdrawal of rewards by summing all withdrawals across the top-level transaction and all sub-transactions, then comparing against the original account balance: [1](#0-0) 

However, no equivalent batch-level check exists for deposit refunds from `UnRegDepositTxCert` certificates. The value conservation check is applied to the top-level `txBody` using the **original** `certState`: [2](#0-1) 

This means each sub-transaction's deposit refund accounting is evaluated independently against the pre-batch state, not cumulatively. An attacker can craft a batch transaction where:

- `subTx1` contains `UnRegDepositTxCert stakingCred keyDeposit`
- `subTx2` contains `UnRegDepositTxCert stakingCred keyDeposit`

Both sub-transactions claim the same deposit refund. The disabled test at lines 53–75 of `CertSpec.hs` explicitly names this scenario and marks it as unimplemented: [3](#0-2) 

The `xit` prefix disables the test entirely, and `error "TODO: predicate failure not yet implemented"` confirms no rejection predicate has been defined for this case. The contrast with `validateBatchWithdrawals` — which has a fully implemented analogous check for withdrawals — makes the gap explicit.

The `SUBENTITIES` rule processes sub-transaction certificates sequentially through `SUBCERTS`: [4](#0-3) 

However, the top-level value conservation check (`validateValueNotConservedUTxO`) uses the original `certState` and the top-level `txBody` only — it does not aggregate the net deposit refund effect of all sub-transactions before checking balance. Each sub-transaction's refund is accounted for independently, allowing the same deposit to be claimed multiple times within a single batch.

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

If two sub-transactions each successfully claim the same stake deposit refund of `D` ADA, the deposit pot decreases by `D` once (when the credential is deregistered), but the UTxO set gains `D` ADA twice — once per sub-transaction output. This creates `D` ADA from nothing, violating the preservation-of-value invariant and constituting a direct invalid ledger state transition.

---

### Likelihood Explanation

**Medium.** The Dijkstra era introduces sub-transactions as a new feature. Any unprivileged user who owns a registered stake credential can craft a batch transaction with multiple sub-transactions. No privileged access, governance majority, or leaked key is required. The attacker only needs to submit a single crafted transaction.

---

### Recommendation

Implement a batch-level check analogous to `validateBatchWithdrawals` that aggregates all `UnRegDepositTxCert` certificates across the top-level transaction and all sub-transactions, verifying that no stake credential is deregistered more than once within the batch. Define the corresponding predicate failure and re-enable the `xit` test in `CertSpec.hs` with the correct expected failure. [5](#0-4) 

---

### Proof of Concept

1. Register stake credential `C` paying deposit `D` ADA.
2. Craft batch transaction `TX` with:
   - `subTx1` body: `UnRegDepositTxCert C D` (claims refund of `D` ADA into `subTx1` outputs)
   - `subTx2` body: `UnRegDepositTxCert C D` (claims refund of `D` ADA into `subTx2` outputs)
3. Submit `TX`. If accepted, the UTxO set gains `2D` ADA while the deposit pot loses only `D` ADA, creating `D` ADA from nothing.

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL

dijkstraSubEntitiesTransition = do
  TRC (subCertsEnv, certState, certificates) <- judgmentContext
  let tx = certsTx subCertsEnv
      pp = certsPParams subCertsEnv
      curEpoch = certsCurrentEpoch subCertsEnv
      withdrawals = tx ^. bodyTxL . withdrawalsTxBodyL
      accounts = certState ^. certDStateL . accountsL

  runTest $ Conway.validateWithdrawalsDelegated accounts tx

  network <- liftSTS $ asks networkId
  let (missingWithdrawals, exceededWithdrawals) =
        case withdrawalsThatExceedAccountBalance withdrawals network accounts of
          Nothing -> (Map.empty, Map.empty)
          Just (missing, exceeded) -> (unWithdrawals missing, exceeded)
  failOnNonEmptyMap missingWithdrawals $
    injectFailure . SubWithdrawalsMissingAccounts . Withdrawals . NEM.toMap
  failOnNonEmptyMap exceededWithdrawals $ injectFailure . SubWithdrawalAmountsExceedAccountBalances

  let certStateBeforeSubCerts =
        certState
          & Conway.updateDormantDRepExpiries tx curEpoch
          & Conway.updateVotingDRepExpiries tx curEpoch (pp ^. ppDRepActivityL)
          & certDStateL . accountsL %~ applyWithdrawals withdrawals
  certStateAfterSubCerts <-
    trans @(EraRule "SUBCERTS" era) $ TRC (subCertsEnv, certStateBeforeSubCerts, certificates)
