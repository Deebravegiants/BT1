Audit Report

## Title
Stale `rsETHPrice` Snapshot Allows Depositors to Mint Excess rsETH at Expense of Existing Holders — (`contracts/LRTOracle.sol`)

## Summary
`LRTOracle` stores `rsETHPrice` as a state variable updated only when `updateRSETHPrice()` is explicitly called. `LRTDepositPool.getRsETHAmountToMint()` divides by this stored snapshot without first refreshing it. Any depositor who acts after yield has accrued but before the next price update receives more rsETH than their deposit warrants, permanently diluting the redemption value of all pre-existing rsETH holders.

## Finding Description
`rsETHPrice` is a public state variable in `LRTOracle` updated only inside `_updateRsETHPrice()`, which is called by the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`. Neither is called anywhere in the deposit path. [1](#0-0) 

`LRTDepositPool.getRsETHAmountToMint()` computes the mint amount as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

The deposit path is `depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()`, with no call to `updateRSETHPrice()` at any point. [3](#0-2) 

When EigenLayer staking rewards cause `totalETHInProtocol` to grow, the actual fair price of rsETH rises above the stored `rsETHPrice`. A depositor during this window receives `amount / rsETHPrice` rsETH, where `rsETHPrice` is lower than fair value, yielding more shares than warranted.

When `_updateRsETHPrice()` is eventually called, it computes `previousTVL = rsethSupply.mulWad(rsETHPrice)` using the same stale price against the now-inflated supply (which includes the attacker's excess shares). The yield is then spread across the larger supply base, permanently reducing the per-share redemption value for all pre-existing holders. [4](#0-3) 

No existing check prevents this. The `minRSETHAmountExpected` slippage parameter protects the depositor from receiving *too few* shares, not existing holders from dilution. There is no price-freshness guard anywhere in the deposit flow.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders lose a portion of their accrued staking yield proportional to the size of the late deposit and the magnitude of yield earned since the last price update. The attacker's gain is exactly the existing holders' loss: the attacker receives rsETH shares representing more ETH than deposited, permanently reducing the redemption value of all pre-existing rsETH. This maps directly to the allowed impact "Theft of unclaimed yield."

## Likelihood Explanation
**Medium.** EigenLayer staking rewards accrue continuously, so the staleness window always exists between price updates. `updateRSETHPrice()` is a public function called off-chain (e.g., by a keeper bot), not on every deposit. Any unprivileged depositor can monitor `LRTDepositPool.getTotalAssetDeposits()` versus the stored `rsETHPrice` to identify a profitable gap and deposit before the next update. No special access is required; the attack is repeatable every update cycle.

## Recommendation
Call `updateRSETHPrice()` (or an equivalent internal `_updateRsETHPrice()`) at the start of `LRTDepositPool.depositAsset()` and `depositETH()`, before `_beforeDeposit()` computes `getRsETHAmountToMint()`. This ensures the exchange rate used for minting always reflects the current TVL, eliminating the staleness window. Alternatively, make `_updateRsETHPrice()` accessible from `LRTDepositPool` and invoke it atomically within `_beforeDeposit()`.

## Proof of Concept
1. **Initial state**: 1,000 ETH in protocol, 1,000 rsETH supply, `rsETHPrice = 1.000 ETH`.
2. **Yield accrues**: EigenLayer rewards add 10 ETH → `totalETHInProtocol = 1,010 ETH`. `rsETHPrice` is not yet updated (still `1.000`). Fair price = `1,010 / 1,000 = 1.010 ETH/rsETH`.
3. **Attacker calls `depositAsset()`** with 100 ETH. `getRsETHAmountToMint()` computes `100 / 1.000 = 100 rsETH`. Fair amount = `100 / 1.010 ≈ 99.01 rsETH`. Attacker receives **~0.99 rsETH excess**.
4. **State after deposit**: `totalETHInProtocol = 1,110 ETH`, `rsethSupply = 1,100 rsETH`.
5. **Price update fires**:
   - `previousTVL = 1,100 × 1.000 = 1,100 ETH`
   - `rewardAmount = 1,110 − 1,100 = 10 ETH` (fee correctly charged on yield only)
   - `newRsETHPrice = (1,110 − fee) / 1,100 ≈ 1.00909 ETH/rsETH`
6. **Without attacker**: `newRsETHPrice = (1,010 − fee) / 1,000 ≈ 1.010 ETH/rsETH`.
7. **Result**: Existing holders' rsETH is worth `~0.001 ETH/rsETH` less than it should be. The attacker's 100 rsETH redeems for `~100.9 ETH` — a `~0.9 ETH` profit extracted from existing holders' unclaimed yield. The attack scales linearly with deposit size and the yield gap since the last update.

**Foundry fork test plan**: Fork mainnet, set `rsETHPrice` to a value below `_getTotalEthInProtocol() / rsethSupply`, call `depositAsset()` as an unprivileged address, then call `updateRSETHPrice()`, and assert that `rsETHPrice` is lower than it would have been without the interleaved deposit. Assert that the attacker's rsETH balance redeems for more ETH than deposited.
