### Title
Double-Finalization via `remove_fin_transfer` Erasure After `ft_transfer_call` Rejection — (`near/omni-bridge/src/lib.rs`)

### Summary

When a `fin_transfer` to a NEAR recipient uses `ft_transfer_call` (non-empty `msg`) and the recipient's `ft_on_transfer` returns the full amount as refund (used = 0), `fin_transfer_send_tokens_callback` calls `remove_fin_transfer`, which **deletes** the `transfer_id` from `finalised_transfers`. Because all provers are stateless (no used-proof tracking), the identical proof can be re-submitted via `fin_transfer` a second time, passing `add_fin_transfer`'s uniqueness check and minting or unlocking the same tokens again.

---

### Finding Description

**Step 1 — Entry point: `fin_transfer`**

`fin_transfer` is gated by `#[trusted_relayer]`, which is permissionless: any account can call `apply_for_trusted_relayer` with a sufficient NEAR stake and, after the waiting period, auto-promote to trusted status without DAO approval. [1](#0-0) [2](#0-1) 

**Step 2 — `add_fin_transfer` marks the transfer as finalised**

`process_fin_transfer_to_near` immediately inserts `transfer_id` into `finalised_transfers` before any async token transfer occurs. [3](#0-2) [4](#0-3) 

**Step 3 — `send_tokens` dispatches `ft_transfer_call` when `msg` is non-empty**

The `msg` field originates from the `InitTransfer` event on the source chain, which the attacker controls as the sender. A non-empty `msg` causes `send_tokens` to call `ft_transfer_call` (native token path) or `mint` with a msg (deployed token path), both of which invoke `ft_on_transfer` on the recipient. [5](#0-4) [6](#0-5) 

**Step 4 — Attacker's recipient returns full refund; `is_refund_required` fires**

In NEP-141, `ft_resolve_transfer` returns the *used* amount. If the attacker's `ft_on_transfer` returns the full amount as refund, used = 0, so the promise result is `U128(0)`. `is_refund_required` treats this as a failure. [7](#0-6) 

**Step 5 — `remove_fin_transfer` deletes the uniqueness record**

Inside `fin_transfer_send_tokens_callback`, the refund branch calls `remove_fin_transfer`, which unconditionally removes `transfer_id` from `finalised_transfers`, then emits `FailedFinTransferEvent`. [8](#0-7) [9](#0-8) 

**Step 6 — Provers are stateless; same proof re-verifies cleanly**

None of the three prover contracts (EVM, Wormhole, MPC) maintain a set of consumed proofs. They only verify cryptographic validity. [10](#0-9) [11](#0-10) [12](#0-11) 

**Step 7 — Re-submission succeeds; `add_fin_transfer` passes again**

With `transfer_id` absent from `finalised_transfers`, the second `fin_transfer` call passes the `require!` in `add_fin_transfer` and proceeds to mint or unlock tokens a second time for the same origin event. [4](#0-3) 

---

### Impact Explanation

- **Deployed (bridged) tokens**: First attempt mints tokens to attacker, which are refunded and burned. Second attempt mints again with no corresponding locked supply on the source chain → unbacked token supply inflation.
- **Native (locked) tokens**: First attempt transfers tokens from bridge to attacker, which are refunded. `revert_lock_actions` restores the locked accounting. Second attempt transfers again and decrements locked accounting a second time → double-unlock, draining bridge reserves.

Both paths violate the invariant that each `(origin_chain, origin_nonce)` pair settles at most once.

---

### Likelihood Explanation

The attacker needs to:
1. Stake NEAR and wait the activation period to become a trusted relayer (permissionless, no DAO approval).
2. Emit an `InitTransfer` on the source chain with a non-empty `msg` and their own NEAR contract as recipient.
3. Submit the proof, have their contract reject the tokens, then re-submit the same proof.

All steps are fully attacker-controlled and require no privileged cooperation. The only cost is the relayer stake, which is recoverable via `resign_trusted_relayer`. [13](#0-12) 

---

### Recommendation

**Do not remove the finalization record on failure.** The `finalised_transfers` set is the sole replay-prevention mechanism for cross-chain proofs. Once a `transfer_id` is inserted, it must never be removed, regardless of whether the downstream token transfer succeeded or failed.

For the refund/retry use-case, introduce a separate `failed_transfers` set (or a status enum on the existing record) that allows the recipient to retry delivery without re-verifying the proof. Alternatively, keep the `transfer_id` in `finalised_transfers` permanently and expose a separate `retry_fin_transfer` entry point that checks the failed-transfer record instead of re-running proof verification.

---

8. Attacker swaps recipient to an accepting contract, re-submits identical proof via fin_transfer.
9. Bridge: add_fin_transfer inserts (Eth,42) again (no panic); tokens minted/unlocked a second time.

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }

    pub fn verify_callback(
        &self,
        #[serializer(borsh)] proof_kind: ProofKind,
        #[serializer(borsh)] sign_payload_bytes: Vec<u8>,
        #[serializer(borsh)] chain_kind: ChainKind,
        #[callback_result] call_result: Result<
            VerifyForeignTransactionResponse,
            near_sdk::PromiseError,
        >,
    ) -> Result<ProverResult, String> {
        let mpc_response = call_result.map_err(|_| ProverError::InvalidProof.to_string())?;

        let sign_payload = ForeignTxSignPayload::try_from_slice(&sign_payload_bytes)
            .map_err(|_| ProverError::ParseArgs.to_string())?;

        let expected_hash = sign_payload
            .compute_msg_hash()
            .map_err(|_| ProverError::InvalidPayloadHash.to_string())?;

        if expected_hash != mpc_response.payload_hash {
            return Err(ProverError::InvalidPayloadHash.to_string());
        }

        let ForeignTxSignPayload::V1(ref payload_v1) = sign_payload;

        match chain_kind {
            ChainKind::Strk => Self::parse_starknet_result(proof_kind, chain_kind, payload_v1),
            ChainKind::Aptos => Self::parse_aptos_result(proof_kind, payload_v1),
            _ => {
                let log_entry_data = Self::extract_evm_log(payload_v1)?;
                parse_evm_proof(proof_kind, chain_kind, log_entry_data)
            }
        }
