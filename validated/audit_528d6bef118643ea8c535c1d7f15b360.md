### Title
Missing Deposit Amount in ETH Deposit Limit Check Allows Limit Bypass â€” (`contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric limit check: for ERC-20 assets it correctly adds the incoming deposit amount to the running total before comparing against the cap, but for ETH it omits the incoming amount entirely. Any unprivileged depositor can therefore push the protocol's ETH holdings above the configured deposit limit in a single call to `depositETH`.

---

### Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` (lines 676â€“682) contains a branch for ETH that evaluates:

```solidity
// ETH branch â€” amount is NOT included
return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
```

while the ERC-20 branch correctly evaluates:

```solidity
// ERC-20 branch â€” amount IS included
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

The ETH branch is missing the `+ amount` term. The correct condition should be:

```solidity
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

This is structurally identical to the BigNumber `innerAdd` carry bug: a term that must be added is absent from the conditional expression, making the guard systematically weaker than intended. The check only blocks a deposit when the limit is already exceeded *before* the deposit lands; it never blocks a deposit that would *cause* the limit to be exceeded.

`_beforeDeposit` (lines 648â€“670) calls this function and reverts with `MaximumDepositLimitReached` only when it returns `true`. Because the ETH branch never returns `true` for a deposit that would push the total over the limit (only for one arriving after the limit is already breached), the revert is never triggered for the boundary-crossing deposit itself.

---

### Impact Explanation

An unprivileged depositor can call `depositETH` and receive rsETH even when the resulting total ETH in the protocol exceeds `depositLimitByAsset(ETH_TOKEN)`. The deposit limit is the protocol's primary mechanism for capping ETH exposure to EigenLayer strategies. Bypassing it allows the protocol to hold more ETH than the admin intended, violating the promised cap and potentially over-exposing the protocol to EigenLayer slashing or liquidity risk.

**Impact: Low** â€” the contract fails to enforce its promised deposit cap, but no funds are directly stolen or frozen.

---

### Likelihood Explanation

The entry path requires only a standard `depositETH` call with `msg.value > 0`. No special role, flash loan, or oracle manipulation is needed. Any depositor who monitors the current total deposits and submits a deposit that would cross the limit will trigger the bug. Likelihood is **high** given the permissionless nature of the entry point.

---

### Recommendation

Add the incoming `amount` to `totalAssetDeposits` in the ETH branch, matching the ERC-20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    // Unified check: include the incoming amount for both ETH and ERC-20
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept

Assume `depositLimitByAsset(ETH_TOKEN) = 1000 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.

1. Attacker calls `depositETH{value: 10 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)`.
3. Inside the function: `totalAssetDeposits = 999 ether`; ETH branch evaluates `999 ether > 1000 ether` â†’ `false`.
4. No revert; the deposit proceeds and the protocol now holds `1009 ether` â€” 9 ETH above the configured limit.
5. The attacker receives rsETH for the full 10 ETH deposit.
