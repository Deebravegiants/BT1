Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` + Stale `rsETHPrice` Enables Front-Running Yield Theft from Existing rsETH Holders — (`contracts/FeeReceiver.sol`)

## Summary
`FeeReceiver.sendFunds()` has no access control, allowing any caller to flush accumulated MEV/execution-layer rewards into the deposit pool at will. Because `LRTDepositPool.getRsETHAmountToMint()` uses the cached `rsETHPrice` storage variable (which does not reflect FeeReceiver's balance), an attacker can deposit ETH at the artificially low stale price, then atomically flush the FeeReceiver balance and trigger a price update — capturing a disproportionate share of yield that belongs to pre-existing rsETH holders.

## Finding Description

**Root cause 1 — `FeeReceiver.sendFunds()` has no access control.**

`sendFunds()` is declared `external` with no role modifier: [1](#0-0) 

Any EOA or contract can call it at any time.

**Root cause 2 — FeeReceiver balance is intentionally excluded from TVL.**

`getETHDistributionData()` only reads from the deposit pool, NDCs, unstaking vault, and converter. The FeeReceiver address is never queried: [2](#0-1) 

The comment at lines 465–466 confirms this is by design: rewards are excluded until `sendFunds()` moves them.

**Root cause 3 — Deposits use the stale cached `rsETHPrice`.**

`getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, which is a storage variable last written by a prior `updateRSETHPrice()` call — not the live TVL: [3](#0-2) 

**Root cause 4 — `updateRSETHPrice()` is permissionless.**

Only a `whenNotPaused` guard is present; no role restriction: [4](#0-3) 

**Exploit flow (single transaction or two sequential txs):**

1. FeeReceiver accumulates N ETH of MEV rewards; `rsETHPrice` is stale (low).
2. Attacker calls `depositETH{value: X}` → receives rsETH minted at the stale price.
3. Attacker calls `FeeReceiver.sendFunds()` → N ETH enters the deposit pool; TVL jumps.
4. Attacker calls `LRTOracle.updateRSETHPrice()` → price recalculated over the larger TVL.
5. Attacker's rsETH is now worth more than deposited; pre-existing holders receive less yield.

**Why `pricePercentageLimit` does not prevent the attack.**

The threshold check at lines 252–266 only gates the `updateRSETHPrice()` call: [5](#0-4) 

- If `pricePercentageLimit == 0`, there is no protection at all.
- If the FeeReceiver balance is small enough relative to TVL to stay within the threshold, the attack succeeds silently.
- Even when the threshold would revert the attacker's `updateRSETHPrice()` call, the attacker has already deposited at the stale price. When the manager eventually calls `updateRSETHPriceAsManager()`, the price updates and the attacker's position appreciates — the threshold only delays, not prevents, the profit.

## Impact Explanation
Pre-existing rsETH holders earned the MEV/execution-layer rewards sitting in FeeReceiver. By depositing at the stale (understated) price before flushing those rewards into TVL, an attacker dilutes their yield share. The profit is directly proportional to the FeeReceiver balance and the attacker's deposit size. This is a concrete, quantifiable theft of unclaimed yield matching the **High — Theft of unclaimed yield** impact class.

## Likelihood Explanation
Both `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` are callable by any unprivileged external account. FeeReceiver accumulates MEV rewards continuously between `sendFunds()` calls, so the exploitable window is always open. No special privileges, oracle manipulation, or governance capture is required. The attack is repeatable every time rewards accumulate.

## Recommendation
1. **Restrict `sendFunds()`** to a privileged role (e.g., `MANAGER` or `LRT_OPERATOR`) so only authorized parties can flush rewards into the deposit pool.
2. **Alternatively, include FeeReceiver balance in TVL** inside `getETHDistributionData()` so the rsETH price already reflects pending rewards and there is no exploitable gap.
3. **Atomically update the price** inside `receiveFromRewardReceiver()` so the price is always current when rewards enter the pool.

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "../contracts/FeeReceiver.sol";
import "../contracts/LRTOracle.sol";
import "../contracts/LRTDepositPool.sol";
import "../contracts/interfaces/IRSETH.sol";

contract YieldTheftPoC is Test {
    LRTDepositPool depositPool;
    LRTOracle      oracle;
    FeeReceiver    feeReceiver;
    IRSETH         rsETH;

    address attacker = address(0xBEEF);

    function setUp() public {
        // Deploy/configure protocol with 1000 ETH TVL, 1000 rsETH supply
        // Fund FeeReceiver with 100 ETH simulating accumulated MEV rewards
    }

    function testYieldTheft() public {
        uint256 priceBefore = oracle.rsETHPrice();
        // priceBefore ≈ 1.000e18 (FeeReceiver 100 ETH excluded from TVL)

        vm.deal(attacker, 10 ether);
        vm.prank(attacker);
        depositPool.depositETH{value: 10 ether}(0, "");
        uint256 attackerRsETH = rsETH.balanceOf(attacker);
        // attackerRsETH ≈ 10e18 (minted at stale price 1.000)

        vm.prank(attacker);
        feeReceiver.sendFunds(); // permissionless — no revert

        vm.prank(attacker);
        oracle.updateRSETHPrice(); // permissionless — no revert

        uint256 priceAfter = oracle.rsETHPrice();
        // priceAfter ≈ 1.099e18

        assertGt(priceAfter, priceBefore);
        uint256 attackerETHValue = attackerRsETH * priceAfter / 1e18;
        assertGt(attackerETHValue, 10 ether, "attacker profited from yield theft");
    }
}
```
