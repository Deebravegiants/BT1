### Title
Cross-Request `payload_hash` Substitution in `respond_verify_foreign_tx` Allows Single Malicious Node to Forge Foreign-Chain Verification Results - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function verifies that a submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but never verifies that `payload_hash` is the canonical hash of the foreign-chain data for the *specific pending request* being resolved. A single attested MPC participant can replay a legitimately-produced signature from one foreign-chain request to close out a completely different pending request, delivering a forged attestation to the waiting caller.

---

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs two checks before resolving a pending yield:

1. The ECDSA signature in `response` is valid over `response.payload_hash` under the domain's root public key.
2. The `request` key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

There is **no check** that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the specific request being resolved. The `payload_hash` field is completely unconstrained by the contract.

The canonical hash is defined as:

```rust
pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
    let mut hasher = sha2::Sha256::new();
    borsh::BorshSerialize::serialize(self, &mut hasher)?;
    Ok(Hash256(hasher.finalize().into()))
}
``` [2](#0-1) 

The signed payload encodes the specific request and its extracted values:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [3](#0-2) 

On the node side, `build_signature_request` computes the hash and uses a zero tweak (root key):

```rust
tweak: Tweak::new([0u8; 32]),
``` [4](#0-3) 

Because the contract only checks signature validity over the *caller-supplied* `payload_hash`, any attested participant holding a legitimately-produced `(payload_hash_B, signature_B)` pair can submit it as the response to a *different* pending request `request_A`. The signature check passes (it is a valid signature over `payload_hash_B`), the request lookup passes (request_A exists), and the contract resolves request_A with the wrong attestation.

This is the direct analog of the Meson hash-collision bug: in Meson, the same signature is valid for both a *request* and a *release* message because the digest format is shared. Here, the same signature is valid as the response to *any* pending `verify_foreign_transaction` request because the contract never binds `payload_hash` to the specific request being resolved.

---

### Impact Explanation

A single malicious MPC node (attested participant, strictly below the signing threshold) can:

1. Observe a legitimately-produced signature `(payload_hash_B, signature_B)` for `request_B` (e.g., a Bitcoin block-hash attestation for tx `0xBB…`).
2. Call `respond_verify_foreign_tx(request_A, { payload_hash_B, signature_B })` targeting a different pending request `request_A` (e.g., a different Bitcoin tx or a completely different chain).
3. The contract accepts: signature valid ✓, request_A pending ✓.
4. The caller waiting on `request_A` receives `{ payload_hash_B, signature_B }` — an attestation that encodes `request_B`'s foreign-chain data, not `request_A`'s.

A bridge contract consuming this result would see a valid MPC signature and, if it does not independently reconstruct and compare the expected `payload_hash`, would treat the wrong foreign-chain event as verified. This enables forged foreign-chain verification and can cause invalid bridge execution or double-spend conditions.

**Impact class**: High — forged foreign-chain verification / participant authorization bypass causing invalid bridge execution.

---

### Likelihood Explanation

- Requires **one** malicious attested participant — no threshold collusion needed.
- The attacker does not forge any cryptographic material; they replay a legitimately-produced signature.
- Every MPC node participates in signing and therefore observes all produced signatures.
- The attack is executable whenever two or more `verify_foreign_transaction` requests are pending simultaneously, which is the normal operating condition for any active bridge.

---

### Recommendation

The contract must bind `response.payload_hash` to the specific pending request before accepting the response. Two complementary approaches:

**Short term**: Have MPC nodes return the full `ForeignTxSignPayload` alongside the signature. The contract recomputes `expected_hash = SHA-256(borsh(payload))`, asserts `expected_hash == response.payload_hash`, and also asserts `payload.request == pending_request.request`. This is the direct analog of the Meson recommendation to include all message fields in the top-level digest.

**Long term**: Embed the `VerifyForeignTransactionRequest` (or its hash) inside the signed payload so the contract can verify the binding without receiving the full extracted values. For example:

```rust
pub struct ForeignTxSignPayloadV2 {
    pub request: ForeignChainRpcRequest,   // already present
    pub domain_id: DomainId,              // add
    pub values: Vec<ExtractedValue>,
}
```

Then on-chain: assert `payload.request == pending_request.request && payload.domain_id == pending_request.domain_id` after recomputing the hash.

---

### Proof of Concept

```
1. Alice calls verify_foreign_transaction(request_A = Bitcoin tx 0xAA…)
   → pending_verify_foreign_tx_requests[request_A] = [alice_yield_id]

2. Bob calls verify_foreign_transaction(request_B = Bitcoin tx 0xBB…)
   → MPC network honestly processes request_B, produces:
       payload_hash_B = SHA-256(borsh(ForeignTxSignPayload{request_B, values_B}))
       signature_B    = ECDSA_sign(root_key, payload_hash_B)

3. Malicious node calls:
   respond_verify_foreign_tx(
       request  = request_A,          // Alice's pending request
       response = { payload_hash_B,   // hash of Bob's data
                    signature_B }     // valid signature over payload_hash_B
   )

4. Contract checks:
   - signature_B valid over payload_hash_B under root public key  ✓
   - request_A exists in pending map                              ✓
   → resolves alice_yield_id with { payload_hash_B, signature_B }

5. Alice's bridge contract receives { payload_hash_B, signature_B }.
   It verifies the signature (valid ✓) and, without independently
   reconstructing the expected hash from request_A, treats tx 0xBB…'s
   block-hash as proof that tx 0xAA… was finalized → invalid bridge action.
```

The root cause — `respond_verify_foreign_tx` accepting any `payload_hash` that carries a valid root-key signature without binding it to the pending request — is a direct analog of the Meson untyped-signature hash-collision vulnerability.
