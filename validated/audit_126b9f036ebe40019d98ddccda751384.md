### Title
PlutusV4 Guard Script Receives Top-Level Transaction Context Instead of Sub-Transaction Context, Bypassing Sub-Transaction Validation - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs)

---

### Summary

In the Dijkstra era's `EraPlutusTxInfo 'PlutusV4 DijkstraEra` instance, the `toPlutusTxInfo` function builds sub-transaction `TxInfo` values for the `GuardingPurpose` case but then **discards them entirely**, passing only the top-level transaction's `TxInfo` to the guard script. A guard script (PlutusV4 Plutus script with `GuardingPurpose`) is the mechanism by which sub-transactions are validated beyond basic UTxO rules; receiving the wrong context means it cannot inspect sub-transaction properties at all. This is the direct Cardano analog of the firewall report's root cause: the raw/outer context is used instead of the overridden/inner context in a system that supports nested transactions.

---

### Finding Description

The `EraPlutusTxInfo 'PlutusV4 DijkstraEra` instance in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs` implements `toPlutusTxInfo` as follows for the top-level transaction path:

```haskell
mkTopTxInfo tx = PlutusTxInfoResult $ do
  txInfo <- mkAnyLevelTxInfo tx
  let topTxInfo = txInfo {PV3.txInfoFee = transCoinToLovelace (tx ^. bodyTxL . feeTxBodyL)}
  Right $ \case
    purpose@(GuardingPurpose AsPurpose) -> do
      _subTxInfosForGuards <-
        forM (OMap.elems (tx ^. bodyTxL . subTransactionsTxBodyL)) $ \subTx -> do
          ...
          left (SubTxContextError txId) $ mkTxInfo purpose
      -- TODO: Include _subTxInfosForGuards
      Right topTxInfo          -- ← sub-tx TxInfos discarded; top-level TxInfo returned
    _ -> Right topTxInfo
``` [1](#0-0) 

The `forM` loop at lines 509–523 correctly computes each sub-transaction's `PlutusTxInfo` and binds the results to `_subTxInfosForGuards`. The leading underscore and the `-- TODO: Include _subTxInfosForGuards` comment at line 524 confirm that the computed values are intentionally (but incompletely) discarded. The function then returns `topTxInfo` — the top-level transaction's `TxInfo` — for **every** `GuardingPurpose` invocation. [2](#0-1) 

The `PV3.ScriptContext` that is ultimately assembled in `toPlutusV4Args` places this `txInfo` value directly into `scriptContextTxInfo`:

```haskell
pure $ PlutusV4Args $ PV3.ScriptContext
  { PV3.scriptContextTxInfo = txInfo   -- ← always topTxInfo for GuardingPurpose
  , ...
  }
``` [3](#0-2) 

A compounding issue exists on the same instance: `toPlutusScriptPurpose` is a hard `error` stub:

```haskell
toPlutusScriptPurpose _ = error "stub: PlutusV4 not yet implemented"
``` [4](#0-3) 

This stub is called unconditionally inside `toPlutusV4Args` before the `ScriptContext` is assembled, meaning any attempt to execute a PlutusV4 script today would panic the node rather than reach the context-mismatch path. Both defects must be fixed together.

The guard mechanism itself is well-defined: `getDijkstraScriptsNeeded` collects `GuardingPurpose` script hashes from `guardsTxBodyL`, and the UTXOW rule enforces that every guard required by a sub-transaction is present in the top-level transaction's guard set. [5](#0-4) [6](#0-5) 

The structural intent is clear: a guard script is supposed to receive the sub-transaction TxInfos so it can inspect and approve or reject sub-transaction behavior (inputs, outputs, minting, certificates, governance actions). The context mismatch makes that inspection impossible.

---

### Impact Explanation

Guard scripts are the primary programmable constraint on sub-transaction behavior in the Dijkstra era. A guard script author writes logic such as "sub-transaction must not mint policy X" or "sub-transaction outputs must satisfy condition Y." Because the script receives `topTxInfo` instead of the sub-transaction TxInfos, none of those checks can be expressed or enforced. A sub-transaction author can freely include minting, certificate, withdrawal, or governance actions that the guard script was designed to block, and the guard script will pass because it only observes the top-level transaction's fields.

Concretely, this enables:
- Unauthorized minting or burning of native assets within sub-transactions, bypassing guard-script-enforced policy constraints → maps to **Critical: direct creation or destruction of native assets through an invalid ledger state transition**.
- Unauthorized governance votes, proposals, or treasury donations within sub-transactions that a guard script was intended to gate → maps to **Critical: unauthorized governance action enacted**.

---

### Likelihood Explanation

The Dijkstra era is currently experimental and not deployed on mainnet. The `error "stub: PlutusV4 not yet implemented"` at line 498 prevents any PlutusV4 script from executing today; a node receiving such a transaction would panic rather than accept it. However:

1. The stub is a temporary placeholder that will be replaced when PlutusV4 is completed.
2. The context-mismatch bug is independent of the stub and will persist unless explicitly fixed.
3. Once Dijkstra is deployed and the stub is removed, any transaction author who places a PlutusV4 guard script in a top-level transaction can exploit the mismatch by crafting a sub-transaction whose behavior the guard was designed to block.

Likelihood is **medium** for the post-deployment window: the attacker entry path requires only the ability to submit a valid Dijkstra-era transaction (unprivileged), and the exploit is deterministic.

---

### Recommendation

1. **Fix the context mismatch**: Replace the discarded `_subTxInfosForGuards` binding with a proper inclusion of sub-transaction TxInfos in the `PV3.TxInfo` passed to the guard script. The `PV3.TxInfo` type (or a Dijkstra-specific extension) must carry the list of sub-transaction TxInfos so the guard script can inspect them.

2. **Remove the stub**: Implement `toPlutusScriptPurpose` for `PlutusV4` before enabling Dijkstra on any network. The stub `error` must not reach production.

3. **Add a regression test**: Add an imp-test (analogous to those in `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/UtxowSpec.hs`) that submits a top-level transaction with a PlutusV4 guard script that asserts a sub-transaction property, verifies the guard fails when the sub-transaction violates it, and verifies the guard passes when it does not.

### Proof of Concept

**Setup**: Dijkstra era node with PlutusV4 enabled (stub removed).

1. Deploy a PlutusV4 guard script `G` whose logic is: "the sub-transaction must not mint any tokens" (i.e., `txInfoMint` of the sub-transaction TxInfo must be empty).

2. Construct a top-level transaction `T_top` that:
   - Lists `G` in `guardsTxBodyL`.
   - Contains a sub-transaction `T_sub` in `subTransactionsTxBodyL` that mints 1000 units of some policy.
   - `T_sub` lists `G` in its `requiredTopLevelGuardsL`.

3. Submit `T_top`.

**Expected (correct) behavior**: `G` receives `T_sub`'s TxInfo, observes `txInfoMint ≠ ∅`, and fails → transaction rejected.

**Actual (buggy) behavior**: `G` receives `T_top`'s TxInfo. If `T_top` itself does not mint, `G` observes `txInfoMint = ∅` and succeeds → transaction accepted, 1000 tokens minted in violation of the guard's policy.
