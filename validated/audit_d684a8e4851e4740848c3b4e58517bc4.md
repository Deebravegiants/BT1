### Title
Dijkstra Era Minimum Fee Calculation Omits Sub-Transaction Reference Script Costs — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces nested ("sub") transactions embedded inside a top-level transaction. The minimum fee validation for a Dijkstra top-level transaction uses `getConwayMinFeeTxUtxo`, which only measures reference-script bytes belonging to the **top-level** transaction. A separate helper, `batchNonDistinctRefScriptsSize`, correctly aggregates reference-script bytes across the top-level transaction **and all sub-transactions**, and is already used for the per-transaction size-limit check. Because the fee check and the size-limit check use different size functions, an attacker can pack up to `maxRefScriptSizePerTx` bytes of reference scripts exclusively into sub-transactions and pay zero reference-script fee surcharge, while the node still deserializes and validates every byte.

---

### Finding Description

**Root cause — mismatched size functions**

`EraUTxO DijkstraEra` binds `getMinFeeTxUtxo` to the Conway implementation:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs  line 141
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` passes the reference-script size to `getMinFeeTx` using `txNonDistinctRefScriptsSize`, which only inspects the **top-level** transaction's inputs:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs  lines 174-175
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
``` [2](#0-1) 

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs  lines 183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

Sub-transactions are never visited. By contrast, the per-transaction size-limit check in the LEDGER rule uses `batchNonDistinctRefScriptsSize`, which **does** include sub-transactions:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs  lines 271-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [4](#0-3) 

The LEDGER rule's size-limit check correctly calls `batchNonDistinctRefScriptsSize`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs  lines 321-329
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $
        DijkstraTxRefScriptsSizeTooBig ...
``` [5](#0-4) 

But the UTXO rule's fee check calls `validateFeeTooSmallUTxO`, which internally calls `getMinFeeTxUtxo` (i.e., `getConwayMinFeeTxUtxo` → `txNonDistinctRefScriptsSize`):

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs  lines 372-373
{- minfee pp txTop utxo₀ ≤ txfee txb -}
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
``` [6](#0-5) 

The result: the fee enforced by the ledger is computed over top-level reference scripts only, while the node must deserialize and validate reference scripts from every sub-transaction as well.

---

### Impact Explanation

The Dijkstra era's `getConwayMinFeeTx` applies a tiered reference-script surcharge (`tierRefScriptFee`) proportional to the total reference-script bytes:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs  lines 103-112
getConwayMinFeeTx pp tx refScriptsSize =
  alonzoMinFeeTx pp tx <+> refScriptsFee
  where
    refScriptsFee =
      tierRefScriptFee
        (unboundRational $ pp ^. ppRefScriptCostMultiplierG)
        ...
        refScriptsSize
``` [7](#0-6) 

When `refScriptsSize` is computed as `txNonDistinctRefScriptsSize` (top-level only) instead of `batchNonDistinctRefScriptsSize` (top-level + sub-transactions), the surcharge for sub-transaction reference scripts is zero. An attacker can place up to `maxRefScriptSizePerTx` bytes of reference scripts entirely inside sub-transactions, paying only the base byte-size fee for the top-level transaction body, while the node incurs the full deserialization cost. This is the same class of underpricing that enabled the June 25 2024 DDoS attack documented in ADR-009, now re-opened through the Dijkstra sub-transaction surface. [8](#0-7) 

**Allowed impact matched:** *Medium — attacker-controlled transactions exceed intended validation limits or modify fees outside design parameters.*

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a `DijkstraTx` with one or more `DijkstraSubTx` entries whose `referenceInputsTxBodyL` point to UTxOs containing large Plutus scripts. No special privilege, key, or governance action is required. The attacker controls the sub-transaction bodies entirely. The only bound is `maxRefScriptSizePerTx` (currently `200 KiB` in Conway, promoted to a protocol parameter in Dijkstra), which is large enough to impose meaningful deserialization cost at near-zero fee.

---

### Recommendation

Override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance to use `batchNonDistinctRefScriptsSize` instead of inheriting `getConwayMinFeeTxUtxo`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo pp tx utxo =
    getMinFeeTx pp tx (batchNonDistinctRefScriptsSize utxo tx)
```

This mirrors the pattern already used by `validateAllRefScriptSize` and ensures the fee surcharge covers the full deserialization cost of reference scripts across the entire batch (top-level + all sub-transactions).

---

### Proof of Concept

1. Deploy a large Plutus V3 script (~100 KiB) as a reference script in a UTxO output `refOut`.
2. Construct a `DijkstraTx` whose top-level body has **no** reference inputs and **no** spending inputs that carry reference scripts (so `txNonDistinctRefScriptsSize` returns 0).
3. Embed a `DijkstraSubTx` whose `referenceInputsTxBodyL` contains `refOut`.
4. Compute the minimum fee using `getMinFeeTxUtxo` — it returns the base fee with zero reference-script surcharge.
5. Submit the transaction. The ledger accepts it (fee check passes, size-limit check passes since 100 KiB < `maxRefScriptSizePerTx`).
6. The node deserializes the 100 KiB script from `refOut` during sub-transaction validation, incurring the full deserialization cost at near-zero fee.
7. Repeat with many such transactions in a block to exhaust node resources at attacker-controlled cost.

The discrepancy is directly observable by comparing:
- `getMinFeeTxUtxo pp tx utxo` (returns base fee, zero surcharge)
- `getMinFeeTx pp tx (batchNonDistinctRefScriptsSize utxo tx)` (returns base fee + surcharge for 100 KiB) [9](#0-8) [10](#0-9)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
