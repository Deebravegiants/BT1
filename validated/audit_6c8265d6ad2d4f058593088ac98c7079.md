### Title
Stale `CodeHashesVotes` from Removed Participants Counted Without Filtering in `vote_code_hash` After Resharing — (`crates/contract/src/tee/proposal.rs`, `crates/contract/src/lib.rs`)

---

### Summary

After a resharing concludes, `vote_reshared` spawns `clean_tee_status` as a **detached** (fire-and-forget) promise to purge stale `CodeHashesVotes` entries from removed participants. If that promise fails (e.g., insufficient `clean_tee_status_tera_gas`) or has not yet executed, stale votes remain in `CodeHashesVotes.proposal_by_account`. A subsequent call to `vote_code_hash` by a current participant invokes `count_votes`, which counts **all** entries in `proposal_by_account` without filtering by the current participant set. This allows stale votes from removed participants to inflate the count and cause the threshold to be reached with fewer current-participant votes than required, whitelisting an unauthorized MPC Docker image hash.

---

### Finding Description

**Root cause — `count_votes` has no participant-set filter:**

`CodeHashesVotes::count_votes` in `crates/contract/src/tee/proposal.rs` counts every entry in `proposal_by_account` that matches the proposal hash:

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
``` [1](#0-0) 

There is no check that the voter's `AuthenticatedParticipantId` key still belongs to the current participant set. Compare this to `ForeignChainRpcWhitelist`, which explicitly filters stale votes at count time.

**`vote_code_hash` relies entirely on the detached cleanup:**

```rust
pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
    self.voter_or_panic();
    let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();
    let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
    let votes = self.tee_state.vote(code_hash, &participant);
    // ...
    if votes >= self.threshold()?.value() {
        self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
    }
    Ok(())
}
``` [2](#0-1) 

`AuthenticatedParticipantId::new` only validates that the **caller** is in the current participant set; it does not purge or filter pre-existing stale entries from removed participants before counting.

**The cleanup is detached and can fail silently:**

```rust
// Spawn a promise to drop votes cast by non-participants.
// Note: MpcContract::vote_update uses filtering to ensure correctness even if this cleanup fails.
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        vec![],
        NearToken::from_yoctonear(0),
        Gas::from_tgas(self.config.clean_tee_status_tera_gas),
    )
    .detach();
``` [3](#0-2) 

The inline comment explicitly acknowledges that `vote_update` uses filtering to remain correct even if this cleanup fails — but `vote_code_hash` has **no such filtering**. `clean_tee_status` is `#[private]` (only callable by the contract itself as a promise), so if the detached promise fails, there is no external recovery path until the next resharing. [4](#0-3) 

**`AuthenticatedParticipantId` is a numeric `ParticipantId` (u32), not an account ID:**

```rust
pub struct AuthenticatedParticipantId(ParticipantId);
// ...
pub fn new(participants: &Participants) -> Result<Self, Error> {
    let signer = env::signer_account_id();
    participants.participants().iter()
        .find(|(a_id, _, _)| *a_id == signer)
        .map(|(_, p_id, _)| AuthenticatedParticipantId(*p_id))
        .ok_or_else(|| InvalidState::NotParticipant { account_id: signer }.into())
}
``` [5](#0-4) 

Stale entries from removed participants (e.g., `ParticipantId(0)`, `ParticipantId(1)`) remain in `proposal_by_account` as distinct keys from new participants' entries. `count_votes` sums them all.

**The codebase's own test documents the exact attack scenario:**

```rust
/// Stale CodeHashesVotes entries from removed participants must not count toward
/// quorum after resharing.
/// Scenario (N=5, T=3):
/// 1. P1 and P2 vote for malicious hash before resharing.
/// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
/// 3. clean_non_participant_votes removes stale votes.
/// 4. P3 votes for the same hash — only 1 vote, not 3.
``` [6](#0-5) 

This test only verifies the **happy path** (cleanup runs). It does not test the failure path where `clean_tee_status` does not run.

---

### Impact Explanation

If `clean_tee_status` fails (gas misconfiguration of `clean_tee_status_tera_gas`) or has not yet executed when a current participant calls `vote_code_hash`:

1. Stale votes from removed participants persist in `CodeHashesVotes.proposal_by_account`.
2. `count_votes` counts them alongside fresh votes from current participants.
3. The threshold can be reached with fewer current-participant votes than required.
4. `whitelist_tee_proposal` is called, adding the malicious Docker image hash to `allowed_docker_image_hashes`.
5. Nodes running the malicious image can now pass `add_participant` / `reverify_participants` attestation checks.
6. This constitutes **participant/attestation authorization bypass**: future `submit_participant_info` calls from nodes running the unauthorized image are accepted, enabling invalid bridge execution. [7](#0-6) 

---

### Likelihood Explanation

**Primary trigger — gas misconfiguration:** `clean_tee_status_tera_gas` is a configurable value. If set too low (operator misconfiguration or a future config update), the detached promise fails silently. Since `clean_tee_status` is `#[private]`, there is no external recovery path; stale votes persist until the next resharing. A removed participant who pre-voted for a malicious hash before resharing benefits from this indefinitely.

**Secondary trigger — timing window:** Even with correct gas, there is a block-level window between `vote_reshared` completing and the `clean_tee_status` receipt being processed. Within this window, a current participant can call `vote_code_hash` and stale votes are counted. This window is narrow but real on NEAR.

The attack requires a participant who is subsequently removed to pre-vote for a malicious hash — a realistic Byzantine behavior strictly below the signing threshold.

---

### Recommendation

Apply the same defense used by `vote_update` and `ForeignChainRpcWhitelist`: filter stale votes at count time inside `vote_code_hash`, independent of whether `clean_tee_status` has run.

**Option A — Filter at count time in `vote_code_hash`:**
```rust
let current_participants = threshold_parameters.participants();
let votes = self.tee_state.votes
    .proposal_by_account
    .iter()
    .filter(|(pid, hash)| {
        **hash == code_hash
            && current_participants.is_participant_given_participant_id(&pid.get())
    })
    .count() as u64;
```

**Option B — Filter in `CodeHashesVotes::count_votes`** by passing the current `Participants` and filtering keys, mirroring `get_remaining_votes`.

**Option C — Eagerly clean before counting** by calling `get_remaining_votes` at the start of `vote_code_hash` and replacing `self.tee_state.votes` before the count, so the cleanup is synchronous and not dependent on the detached promise.

// threshold reached → whitelist_tee_proposal fires

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
    }
