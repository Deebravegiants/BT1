### Title
Byzantine node can store a `K256Signature` with invalid `big_r` prefix that passes `verify_ecdsa_signature` but fails `TryFrom<&K256AffinePoint>` — (`crates/near-mpc-crypto-types/src/conversions/near.rs`)

### Summary
`to_ecrecover_bytes` discards `big_r.affine_point[0]`, so `verify_ecdsa_signature` accepts any prefix byte. A single Byzantine attested participant can call `respond` with `big_r.affine_point[0] = 0x04`, causing the contract to store and return a `K256Signature` whose `big_r` cannot be decompressed via the standard `TryFrom<&K256AffinePoint> for k256::AffinePoint` conversion.

### Finding Description
`K256Signature::to_ecrecover_bytes` extracts only `affine_point[1..33]` (the x-coordinate) and ignores `affine_point[0]` (the SEC1 compressed-point prefix). `verify_ecdsa_signature` calls `to_ecrecover_bytes` and passes the result to `near_sdk::env::ecrecover`, which succeeds for any prefix value as long as `(r, s, recovery_id)` are valid. The contract's `respond` method accepts the signature and serializes the full `K256Signature` — including the raw `affine_point[0]` byte — into contract storage. Any downstream caller that invokes `TryFrom<&K256AffinePoint> for k256::AffinePoint` on the returned signature receives `Err(InvalidPoint)` because `k256::AffinePoint::from_bytes` requires a `0x02`/`0x03` prefix.

### Impact Explanation
The contract's acceptance of a `K256Signature` carries an implicit invariant: `big_r` is a valid SEC1 compressed point decompressible via the standard conversion. A Byzantine node breaks this invariant for any signing request it processes. Callers that rely on the full R point — for example, to perform non-ecrecover-based ECDSA verification or to reconstruct the R point for protocol-level checks — receive an unrecoverable error from the standard SDK conversion path. The signature's `(r, s)` values remain mathematically valid, but the `big_r` field is permanently corrupted in the stored response.

### Likelihood Explanation
Requires a single attested participant node acting maliciously. The node participates normally in the threshold signing protocol, obtains valid `(r, s)` output, then flips one byte before calling `respond`. No threshold collusion, key reconstruction, or network-level capability is needed. Any attested participant can trigger this on any signing request assigned to it.

### Recommendation
Validate `big_r.affine_point[0]` in `verify_ecdsa_signature` (or in `respond` before calling it) to reject any prefix outside `{0x02, 0x03}`. Alternatively, add a prefix check inside `to_ecrecover_bytes` or introduce a dedicated `K256Signature::validate_big_r_prefix` guard called at the contract boundary.
