Audit Report

## Title
ETH Push Transfer to Contract Recipient Without `receive` Permanently Freezes Funds After rsETH Burn - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._transferAsset` delivers ETH via a low-level `.call{value}` that reverts if the recipient contract has no `receive` or `fallback` function. Because rsETH is irreversibly burned in a prior, separate `unlockQueue` transaction, any subsequent failure of `completeWithdrawal` leaves the corresponding ETH permanently locked inside `LRTWithdrawalManager` with no admin recovery path.

## Finding Description
`_transferAsset` (L876-883) is the sole ETH disbursement primitive:

```solidity
function _transferAsset(address asset, address to, uint256 amount) internal {
    if (asset == LRTConstants.ETH_TOKEN) {
        (bool sent,) = payable(to).call{ value: amount }("");
        if (!sent) revert EthTransferFailed();
    } else {
        IERC20(asset).safeTransfer(to, amount);
    }
}
```

It is called at the end of `_processWithdrawalCompletion` (L734) after all state mutations (nonce pop at L705, request deletion at L712, `unlockedWithdrawalsCount` decrement at L717) have already executed in the same transaction. If `_transferAsset` reverts, the EVM rolls back all those mutations, leaving the request permanently in the queue.

The critical separation of concerns is in `unlockQueue` (L305):

```solidity
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);
```

This is a committed, separate transaction. Once rsETH is burned and ETH is redeemed into `LRTWithdrawalManager`, there is no rollback. Every subsequent call to `completeWithdrawal` by a contract recipient without `receive` will revert identically and indefinitely.

`sweepRemainingAssets` (L403) is gated by `hasUnlockedWithdrawals(asset)`, which returns `true` while `unlockedWithdrawalsCount[asset] > 0`. Because the decrement at L717 is always reverted alongside the failed transfer, this counter never reaches zero for the stuck asset, permanently blocking the sweep escape hatch as well. No admin function exists to redirect a withdrawal to an alternative address.

The developer comment on `completeWithdrawalForUser` (L191) — *"Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH"* — confirms awareness of ETH-delivery edge cases but does not address the permanent-freeze scenario arising from the two-transaction lifecycle.

## Impact Explanation
**Critical — Permanent freezing of funds.** The user's rsETH is irreversibly destroyed in `unlockQueue`. The corresponding ETH allocation sits in `LRTWithdrawalManager` and can never be delivered or recovered: `completeWithdrawal` always reverts for the affected user, and `sweepRemainingAssets` is permanently blocked by the non-zero `unlockedWithdrawalsCount`. This matches the allowed Critical impact class exactly.

## Likelihood Explanation
**Medium.** Smart-contract wallets (e.g., Safe multisigs), protocol treasuries, EigenLayer operator contracts, and DeFi vaults routinely hold and interact with LRT tokens. Any such contract that calls `initiateWithdrawal` for ETH without implementing `receive() external payable` will trigger this freeze. The pattern requires no attacker — it is self-triggerable by any unprivileged contract user following the normal withdrawal workflow.

## Recommendation
Replace the push-ETH pattern with a pull pattern: record the owed ETH amount per user in a mapping and expose a separate `claimETH()` function that the user calls to withdraw. Alternatively, wrap the ETH into WETH before transferring (`IWETH(WETH_ADDRESS).deposit{value: amount}(); IERC20(WETH_ADDRESS).safeTransfer(to, amount);`), which always succeeds regardless of the recipient's fallback implementation and eliminates the delivery failure vector entirely.

## Proof of Concept
1. Deploy `VaultUser` — a contract with no `receive` function that holds rsETH.
2. `VaultUser` calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`; a `WithdrawalRequest` is recorded.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. At L305, `burnFrom` destroys the rsETH; `unstakingVault.redeem` moves ETH into `LRTWithdrawalManager`. This transaction commits and cannot be reversed.
4. `VaultUser` calls `completeWithdrawal(ETH_TOKEN, "")`. Inside `_processWithdrawalCompletion`, `_transferAsset` executes `payable(VaultUser).call{value: amount}("")`. Because `VaultUser` has no `receive`, the call returns `false`; `EthTransferFailed` is thrown; the entire transaction reverts, restoring the queue entry.
5. Step 4 can be repeated indefinitely — it always reverts. `VaultUser`'s rsETH is permanently burned and the ETH is permanently locked in `LRTWithdrawalManager`. `sweepRemainingAssets` cannot be called because `unlockedWithdrawalsCount[ETH_TOKEN] > 0` remains true forever.

Foundry test outline:
```solidity
function test_permanentFreeze() public {
    VaultUser vault = new VaultUser(); // no receive()
    rsETH.transfer(address(vault), rsETHAmount);
    vm.prank(address(vault));
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
    // operator unlocks — rsETH burned, ETH in withdrawalManager
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, ...);
    // complete always reverts
    vm.prank(address(vault));
    vm.expectRevert(LRTWithdrawalManager.EthTransferFailed.selector);
    withdrawalManager.completeWithdrawal(ETH_TOKEN, "");
    // ETH locked, sweep blocked
    assertTrue(withdrawalManager.hasUnlockedWithdrawals(ETH_TOKEN));
    assertGt(address(withdrawalManager).balance, 0);
}
```
