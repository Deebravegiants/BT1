Audit Report

## Title
Stale `rsETHPrice` Allows Depositor to Capture Unclaimed Yield Before Price Update — (`contracts/LRTDepositPool.sol` / `contracts/LRTOracle.sol` / `contracts/FeeReceiver.sol`)

## Summary

`depositETH` mints rsETH using a stored, potentially stale `rsETHPrice` state variable with no freshness check. Because `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` are both permissionless, an unprivileged EOA can flush accumulated MEV/EL rewards into the deposit pool and immediately deposit ETH at the pre-reward price, capturing a proportional share of yield that belonged to existing rsETH holders. The price update — whenever it occurs — permanently dilutes existing holders' unclaimed yield in favor of the attacker.

## Finding Description

**Root cause — stale stored price used for minting:**

`getRsETHAmountToMint` reads `lrtOracle.rsETHPrice()`, a stored state variable, not a freshly computed value:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`_beforeDeposit` calls `getRsETHAmountToMint` with no price refresh: [2](#0-1) 

**`updateRSETHPrice()` has no access control:**

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [3](#0-2) 

**`FeeReceiver.sendFunds()` has no access control:**

```solidity
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
``` [4](#0-3) 

**`getETHDistributionData()` uses live `address(this).balance`**, so rewards pushed by `sendFunds()` immediately inflate TVL used by `_updateRsETHPrice()`: [5](#0-4) 

**Exploit flow:**

1. Rewards accumulate in `FeeReceiver` (MEV, EL rewards).
2. Attacker calls `FeeReceiver.sendFunds()` — rewards are pushed into `LRTDepositPool`, increasing `address(this).balance` and thus TVL. `rsETHPrice` remains stale.
3. Attacker calls `depositETH{value: X}()` — mints rsETH at the stale (pre-reward) price, receiving more rsETH than the current TVL warrants.
4. `updateRSETHPrice()` is called (by anyone, including the attacker when `pricePercentageLimit == 0`). The new price reflects the full TVL including rewards, but the attacker's rsETH was minted at the old ratio — the attacker's share now includes a portion of the rewards that belonged to existing holders.

**`pricePercentageLimit` is not a sufficient guard:** It only blocks the price update call when the increase exceeds the threshold AND the caller is not a manager. It does not prevent the stale-price deposit itself. The attacker's profit is locked in at step 3; the price update at step 4 can be performed by the protocol's normal keeper or by the attacker when the limit is unset (`pricePercentageLimit == 0`). [6](#0-5) 

**ETH deposit-limit check omits the incoming amount** (separate bug, but confirms the deposit is not blocked for large amounts): [7](#0-6) 

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders accumulate yield as ETH rewards raise TVL without increasing rsETH supply. That yield is "unclaimed" until `updateRSETHPrice()` is called. An attacker who deposits at the stale price and waits for the update dilutes existing holders' yield proportionally to their deposit size.

Numerical example:

| State | TVL | rsETH supply | rsETHPrice |
|---|---|---|---|
| Before rewards | 100 ETH | 100 | 1.000 |
| After `sendFunds()` (5 ETH) | 105 ETH | 100 | 1.000 (stale) |
| Attacker deposits 100 ETH | 205 ETH | 200 | 1.000 (stale) |
| After `updateRSETHPrice()` | 205 ETH | 200 | 1.025 |

- Attacker: deposited 100 ETH, holds 100 rsETH × 1.025 = **102.5 ETH (+2.5 ETH)**
- Original holders: 100 rsETH × 1.025 = 102.5 ETH instead of the 105 ETH owed (**−2.5 ETH stolen yield**)

This is a concrete, quantifiable theft of unclaimed yield, matching the **High** impact class exactly.

## Likelihood Explanation

- No privileged role required for any step.
- No front-running required — all three steps are callable by the same EOA in consecutive transactions within the same block.
- Rewards accumulate continuously; the attack window is always open.
- Profit scales linearly with deposit size and accumulated reward amount.
- The attack is repeatable every reward accumulation cycle.

## Recommendation

1. **Call `_updateRsETHPrice()` inside `_beforeDeposit`** (or at minimum read a freshly computed price) before computing `rsethAmountToMint`, so deposits always use a price that reflects current TVL.
2. Alternatively, **restrict `FeeReceiver.sendFunds()` to a privileged role** (e.g., `MANAGER`) so reward flushing cannot be weaponized by an unprivileged caller.
3. As defense-in-depth, **fix the ETH deposit-limit check** to include the incoming amount: `totalAssetDeposits + amount > depositLimit`, matching the ERC-20 path on line 681.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "../contracts/LRTDepositPool.sol";
import "../contracts/LRTOracle.sol";
import "../contracts/FeeReceiver.sol";
import "../contracts/RSETH.sol";

contract YieldDiversionTest is Test {
    LRTDepositPool depositPool;
    LRTOracle oracle;
    FeeReceiver feeReceiver;
    RSETH rseth;
    address existingHolder = address(0xBEEF);
    address attacker = address(0xBAD);

    // Setup: deploy contracts, mint 100 rsETH to existingHolder at 1:1 (100 ETH TVL)

    function test_stalePrice_yieldTheft() public {
        // 1. Simulate 5 ETH of MEV rewards sitting in FeeReceiver
        vm.deal(address(feeReceiver), 5 ether);

        uint256 holderBalanceBefore = rseth.balanceOf(existingHolder); // 100 rsETH

        // 2. Attacker flushes rewards into deposit pool (permissionless)
        vm.prank(attacker);
        feeReceiver.sendFunds();
        // TVL = 105 ETH, rsETHPrice still 1.0 (stale)

        // 3. Attacker deposits at stale price
        vm.deal(attacker, 100 ether);
        vm.prank(attacker);
        depositPool.depositETH{value: 100 ether}(0, "");
        // Attacker receives 100 rsETH (should receive ~95.24 at fair price)

        // 4. Price update (permissionless when pricePercentageLimit == 0)
        vm.prank(attacker);
        oracle.updateRSETHPrice();
        // newRsETHPrice = 205 ETH / 200 rsETH = 1.025

        // 5. Attacker captured yield
        uint256 attackerValue = rseth.balanceOf(attacker) * oracle.rsETHPrice() / 1e18;
        assertGt(attackerValue, 100 ether); // 102.5 ETH > 100 ETH deposited

        // 6. Existing holders lost yield
        uint256 holderValue = holderBalanceBefore * oracle.rsETHPrice() / 1e18;
        assertLt(holderValue, 105 ether); // 102.5 ETH < 105 ETH owed
    }
}
```
