Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Configured Cap to Be Exceeded - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` contains an ETH-specific branch that evaluates `totalAssetDeposits > depositLimit` without adding the incoming `amount`, while the ERC20 branch correctly evaluates `totalAssetDeposits + amount > depositLimit`. Because the ETH branch returns early, the configured ETH deposit cap is never properly enforced, allowing any depositor to push total ETH deposits beyond the admin-set limit.

## Finding Description
In `contracts/LRTDepositPool.sol`, the function `_checkIfDepositAmountExceedesCurrentLimit` (L676–682) branches on `asset == LRTConstants.ETH_TOKEN` and returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` — omitting `amount` entirely. The ERC20 path on the next line correctly returns `totalAssetDeposits + amount > depositLimit`. [1](#0-0) 

This means two bypass scenarios exist:
1. When `totalAssetDeposits == depositLimit` (limit exactly reached), the check returns `false`, so the deposit is not blocked.
2. When `totalAssetDeposits < depositLimit` but `totalAssetDeposits + msg.value > depositLimit`, the check also returns `false`, allowing a deposit that overshoots the cap.

`depositETH` passes `msg.value` as `depositAmount` to `_beforeDeposit`: [2](#0-1) 

`_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit` and reverts only if it returns `true`: [3](#0-2) 

Because the ETH branch never returns `true` for the overshoot case, `_beforeDeposit` does not revert, and `_mintRsETH` proceeds, minting rsETH for the excess ETH.

## Impact Explanation
The ETH deposit limit set by the admin via `lrtConfig.depositLimitByAsset(ETH_TOKEN)` is not enforced for ETH deposits. Any depositor can push total ETH deposits beyond the configured cap, and rsETH is minted for the excess. This matches the allowed impact: **Low — Contract fails to deliver promised returns** (the deposit-cap guarantee is a promised protocol constraint that is not upheld).

## Likelihood Explanation
High. Any unprivileged external caller can trigger this by calling `depositETH{value: ...}(...)` when total deposits are at or near the limit. No special role, front-running, or external dependency is required. The only preconditions are that the protocol is unpaused and a deposit limit has been configured.

## Recommendation
Change the ETH branch to include the incoming deposit amount, consistent with the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
1. Admin sets ETH deposit limit: `lrtConfig.updateAssetDepositLimit(ETH_TOKEN, 1000 ether)`.
2. Deposits accumulate until `getTotalAssetDeposits(ETH_TOKEN) == 1000 ether`.
3. Attacker calls `depositETH{value: 100 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 100 ether)` evaluates `1000 ether > 1000 ether` → `false`.
5. `_beforeDeposit` does not revert; `_mintRsETH` executes.
6. Total ETH deposits become 1100 ether — 10% above the configured cap — with rsETH minted for the excess.

Foundry test plan: deploy `LRTDepositPool` with a mock `LRTConfig` returning `depositLimitByAsset = 1000 ether`; mock `getTotalAssetDeposits` to return `1000 ether`; call `depositETH{value: 1 ether}`; assert the call succeeds (does not revert with `MaximumDepositLimitReached`) and that rsETH is minted, confirming the bypass.

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
