### Title
Publicly Callable `updateRSETHPrice()` Enables Sandwich Attack to Steal Yield from rsETH Holders - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle.updateRSETHPrice()` is a permissionless public function. An attacker can deposit ETH at a stale (lower) rsETH price, immediately call `updateRSETHPrice()` themselves to push the price to its actual higher value, then withdraw at the updated price — extracting the yield that accrued to existing rsETH holders during the staleness window.

### Finding Description

`LRTOracle.updateRSETHPrice()` is declared `public` with no access restriction:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The rsETH price is computed from the actual ETH held across the protocol (EigenLayer, NDCs, unstaking vault, etc.) divided by total rsETH supply. Between calls, the stored `rsETHPrice` is stale. Because anyone can call `updateRSETHPrice()`, an attacker can control exactly when the price transitions from stale to current.

**Deposit pricing** in `LRTDepositPool.getRsETHAmountToMint()`:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
A lower (stale) `rsETHPrice` yields more rsETH per ETH deposited.

**Instant withdrawal pricing** in `LRTWithdrawalManager.instantWithdrawal()`:
```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
// getExpectedAssetAmount: amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)
```
A higher (updated) `rsETHPrice` yields more ETH per rsETH burned.

**Queued withdrawal pricing** in `LRTWithdrawalManager.initiateWithdrawal()`:
```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
assetsCommitted[asset] += expectedAssetAmount;
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
```
The `expectedAssetAmount` is locked in at the time of `initiateWithdrawal()`. `_calculatePayoutAmount()` returns `min(expectedAssetAmount, currentReturn)`, so if the price stays flat or rises after initiation, the attacker receives the full locked-in amount calculated at the updated price.

### Impact Explanation

The attacker captures staking yield that belongs to existing rsETH holders. By depositing at the stale price they receive more rsETH than the protocol's actual NAV justifies, diluting all existing holders. Withdrawing at the corrected price converts that excess rsETH back to ETH at the expense of the pool. This is **theft of unclaimed yield** (High impact).

The `pricePercentageLimit` guard in `_updateRsETHPrice()` limits the maximum exploitable price jump per update to the configured threshold (e.g. 1%), but does not prevent the attack for price increases within that bound.

### Likelihood Explanation

**Low.** The attack requires:
1. A meaningful gap between the stale stored `rsETHPrice` and the actual on-chain ETH value (normal between oracle updates).
2. For the instant-withdrawal path: `isInstantWithdrawalEnabled[asset]` must be `true` (manager-controlled flag).
3. For the queued-withdrawal path: the attacker must wait `withdrawalDelayBlocks` (~8 days) and bear price-decrease risk during that window.
4. Sufficient liquidity in `LRTUnstakingVault` to cover the withdrawal.

The price staleness window is a routine operational condition, making the preconditions realistic whenever instant withdrawals are enabled.

### Recommendation

1. **Restrict `updateRSETHPrice()`** to a permissioned role (e.g. `MANAGER` or `OPERATOR_ROLE`) so attackers cannot trigger price updates at will.
2. **Add a same-block deposit-withdrawal guard**: record the block number of each user's last deposit and revert `initiateWithdrawal` / `instantWithdrawal` if called in the same block.
3. **Snapshot the rsETH price at deposit time** and use `min(depositPrice, currentPrice)` when computing the rsETH mint amount, or alternatively use `max(depositPrice, currentPrice)` when computing the withdrawal amount, so depositing at a stale price cannot be immediately monetised.

### Proof of Concept

**Setup**: rsETH price stored in `LRTOracle` is stale at 1.00 ETH/rsETH. Actual on-chain ETH value supports 1.01 ETH/rsETH (0.5% below the `pricePercentageLimit`).

**Step 1 — Deposit at stale price**
```
LRTDepositPool.depositETH{value: 100 ETH}(0, "");
// rsethAmountToMint = 100e18 * 1e18 / 1.00e18 = 100 rsETH
// Correct amount at true price would be: 100e18 * 1e18 / 1.01e18 ≈ 99.01 rsETH
// Attacker receives ~0.99 rsETH excess
```

**Step 2 — Update price (permissionless)**
```
LRTOracle.updateRSETHPrice();
// rsETHPrice now = 1.01e18
```

**Step 3a — Instant withdrawal (if enabled)**
```
LRTWithdrawalManager.instantWithdrawal(ETH_TOKEN, 100e18, "");
// assetAmountUnlocked = 100e18 * 1.01e18 / 1e18 = 101 ETH
// fee deducted, but net > 100 ETH deposited
```

**Step 3b — Queued withdrawal (always available)**
```
LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 100e18, "");
// expectedAssetAmount = 100 * 1.01 = 101 ETH  ← locked in at updated price
// ... wait withdrawalDelayBlocks (~8 days) ...
LRTWithdrawalManager.completeWithdrawal(ETH_TOKEN, "");
// receives 101 ETH
```
