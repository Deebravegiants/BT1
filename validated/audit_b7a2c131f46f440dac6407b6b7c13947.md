### Title
Unbounded Sub-Transaction Count in `SUBLEDGERS` Rule Allows Disproportionate Validation Work — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs`)

---

### Summary

In the Dijkstra era, the `SUBLEDGERS` STS rule processes all sub-transactions embedded in a top-level transaction via an unconstrained `foldM` loop. No protocol parameter limits the number of sub-transactions. Because each sub-transaction triggers a full `SUBLEDGER` evaluation chain (`SUBUTXOW` → `SUBENTITIES` → `SUBGOV`), an attacker can pack a transaction with the maximum number of minimal sub-transactions (bounded only by `maxTxSize`) and cause validation work that is disproportionate to the fees paid, exceeding the intended per-transaction validation budget.

---

### Finding Description

`dijkstraSubLedgersTransition` in `SubLedgers.hs` iterates over the sub-transaction list with no count guard:

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
``` [1](#0-0) 

The `subTxs` list is derived from `subTransactionsStAnnTx stAnnTx` in `dijkstraLedgerTransition`, which reads the `dtbrSubTransactions` field of the top-level transaction body — an `OMap TxId (Tx SubTx era)` with no upper-bound constraint: [2](#0-1) 

The `DijkstraPParams` record introduces several new Dijkstra-specific protocol parameters (`dppMaxRefScriptSizePerTx`, `dppMaxRefScriptSizePerBlock`, etc.) but contains **no `maxSubTransactions` field**: [3](#0-2) 

The only size constraint applied to the batch is the overall `maxTxSize` check on the serialized top-level transaction in `dijkstraUtxoTransition`: [4](#0-3) 

Each sub-transaction processed by `SUBLEDGER` triggers:

1. **`SUBUTXOW`** — two UTxO membership lookups (`validateBadInputsUTxO` against both `originalUtxo` and the evolving `utxoState`), validity-interval check, forecast check, output-size checks, network-ID checks, and witness/script validation.
2. **`SUBENTITIES`** — full certificate processing (stake registration/deregistration, pool registration, DRep registration) via `SUBCERTS`.
3. **`SUBGOV`** — governance proposal and vote processing. [5](#0-4) [6](#0-5) 

The Cardano fee formula charges `txFeeFixed + txFeePerByte × txSize`. A minimal valid sub-transaction body (one input, one output, no certs, no scripts) serializes to roughly 70–100 bytes, yet still incurs the full fixed per-sub-transaction rule-evaluation overhead listed above. With `maxTxSize = 16 384` bytes, an attacker can embed approximately 160 minimal sub-transactions in a single transaction, causing validation work that scales as `O(N × fixedOverhead)` while fees scale as `O(N × minSubTxBytes × feePerByte)` — the fee formula does not compensate for the fixed per-sub-transaction overhead.

The CDDL schema confirms there is no upper-bound annotation on the `sub_transactions` field:

```
sub_transactions = nonempty_oset<sub_transaction>
``` [7](#0-6) 

Additionally, `dijkstraSubUtxoTransition` contains **no** `validateExUnitsTooBigUTxO` check, meaning sub-transaction Plutus execution units are not individually bounded by `maxTxExUnits`: [6](#0-5) 

(Compare with the top-level check at `dijkstraUtxoTransition` line 415.)

---

### Impact Explanation

**Medium — Attacker-controlled transactions exceed intended validation limits.**

An unprivileged transaction author can submit a single transaction containing the maximum number of minimal sub-transactions. Every validating node and block producer must execute the full `SUBLEDGER` rule chain for each sub-transaction. The fees collected do not cover the disproportionate validation work, allowing an attacker to impose unbounded (within `maxTxSize`) computational cost on the network at below-intended cost. This matches the allowed impact: *"Attacker-controlled transactions … exceed intended validation limits."*

---

### Likelihood Explanation

Any transaction author can craft such a transaction with no special privileges, keys, or governance access. The attack requires only the ability to submit a valid Dijkstra-era transaction, which is available to any network participant. The cost to the attacker is only the transaction fee based on serialized size.

---

### Recommendation

1. Add a `maxSubTransactions :: Word16` (or similar) field to `DijkstraPParams`.
2. Enforce the limit in `dijkstraSubLedgersTransition` (or in `dijkstraUtxoTransition` before the `SUBLEDGERS` call) by checking `length subTxs ≤ ppMaxSubTransactions pp` and failing with a new predicate failure if exceeded.
3. Extend the fee formula or introduce a per-sub-transaction fee component so that the fee charged is proportional to the actual validation work, analogous to how `maxTxExUnits` and execution-unit prices bound Plutus script costs.

---

### Proof of Concept

1. Construct a Dijkstra-era top-level transaction with `floor(maxTxSize / minSubTxSerializedSize)` sub-transactions, each containing exactly one UTxO input and one output (minimum valid sub-transaction body, ~80 bytes serialized).
2. Sign and submit the transaction.
3. `dijkstraLedgerTransition` calls `trans @(EraRule "SUBLEDGERS" era)` with the full list of sub-transactions.
4. `dijkstraSubLedgersTransition` iterates via `foldM` with **no count check**, invoking `SUBLEDGER` for each sub-transaction.
5. Each `SUBLEDGER` invocation runs `SUBUTXOW` (two UTxO lookups + witness validation) + `SUBENTITIES` + `SUBGOV`.
6. Total validation work ≈ 160 × full-subledger-evaluation, while fees ≈ 160 × 80 bytes × `feePerByte` — the fixed per-sub-transaction overhead is not reflected in the fee. [8](#0-7) [9](#0-8)

dijkstraSubUtxoTransition = do
  TRC (SubUtxoEnv slot pp certState originalUtxo (IsValid isValid), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG

  let txBody = tx ^. bodyTxL

  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo
  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  let allSizedOutputs = txBody ^. allSizedOutputsTxBodyF
  let allOutputs = fmap sizedValue allSizedOutputs
  runTest $ Alonzo.validateOutputTooBigUTxO pp allOutputs

  runTest $ Shelley.validateInputSetEmptyUTxO txBody

  let inputs = txBody ^. inputsTxBodyL
  let refInputs = txBody ^. referenceInputsTxBodyL
  runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxoState) inputs

  runTestOnSignal $ Shelley.validateOutputBootAddrAttrsTooBig allOutputs

  runTestOnSignal $ Babbage.validateOutputTooSmallUTxO pp allSizedOutputs

  netId <- liftSTS $ asks networkId
  runTestOnSignal $ Shelley.validateWrongNetwork netId allOutputs
  runTestOnSignal $ Shelley.validateWrongNetworkWithdrawal netId txBody
  runTestOnSignal $ validateWrongNetworkInDirectDeposit netId txBody
  runTestOnSignal $ Alonzo.validateWrongNetworkInTxBody netId txBody

  if isValid
    then do
      newState <-
        Shelley.updateUTxOStateNoFees
          pp
          utxoState
          txBody
          certState
          (utxosGovState utxoState)
          (tellEvent . TotalDeposits (hashAnnotated txBody))
          (\a b -> tellEvent $ TxUTxODiff a b)
      pure $ newState & utxosDonationL <>~ txBody ^. treasuryDonationTxBodyL
    else
      pure utxoState
