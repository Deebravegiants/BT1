### Title
ETH Withdrawal Permanently Stuck for Contract Recipients Without `receive()` - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager` always sends ETH withdrawals to the original depositor address with no mechanism to redirect to an alternative recipient. If the depositor is a smart contract without a `receive()` or `fallback()` function, `completeWithdrawal` will permanently revert after the user's rsETH has already been burned in `unlockQueue`, permanently freezing the ETH in the contract.

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` is split across two operator/user steps:

**Step 1 — `unlockQueue` (operator-called):** rsETH is burned and ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager`. [1](#0-0) 

**Step 2 — `completeWithdrawal` / `completeWithdrawalForUser` (user or operator-called):** ETH is forwarded to `user` via `_transferAsset`. [2](#0-1) 

The ETH transfer in `_transferAsset` uses `payable(to).call{ value: amount }("")` and reverts with `EthTransferFailed` on failure. If `to` is a contract without a `receive()` function, this call will always revert. Since the entire `_processWithdrawalCompletion` transaction reverts, the withdrawal request record is preserved — but the rsETH was already irreversibly burned in Step 1. [3](#0-2) 

There is no function to redirect the withdrawal to an alternative address. `completeWithdrawalForUser` still sends to `user`, not to a caller-specified recipient: [4](#0-3) 

Additionally, `sweepRemainingAssets` cannot rescue the ETH because it requires `hasUnlockedWithdrawals(asset) == false`, but the stuck withdrawal keeps `unlockedWithdrawalsCount[asset] > 0` indefinitely: [5](#0-4) 

### Impact Explanation

**Medium — Permanent freezing of funds.**

A contract depositor whose address cannot receive ETH (no `receive()`/`fallback()`) will have:
- rsETH permanently burned (Step 1 is irreversible)
- ETH permanently locked in `LRTWithdrawalManager` (Step 2 always reverts, sweep blocked)

The ETH cannot be recovered by the user or by the protocol without an upgrade.

### Likelihood Explanation

**Medium.** Smart contract wallets (multisigs, DAOs, protocol treasuries) are common depositors in LRT protocols. A contract that calls `depositETH` or `depositAsset` to obtain rsETH may not implement `receive()`. There is no on-chain check preventing such a contract from initiating a withdrawal. The failure only manifests at Step 2, after rsETH is already burned.

### Recommendation

Add an optional `recipient` parameter to `completeWithdrawal` (defaulting to `msg.sender`) so users can specify an EOA or a contract that can receive ETH:

```solidity
function completeWithdrawal(
    address asset,
    address recipient,   // <-- new
    string calldata referralId
) external nonReentrant whenNotPaused {
    _processWithdrawalCompletion(asset, msg.sender, recipient, referralId);
}
```

Alternatively, allow users to register a withdrawal recipient address before the operator calls `unlockQueue`, analogous to how the original Fractional report recommended an alternate send-to address.

### Proof of Concept

1. A Gnosis Safe (or any contract without `receive()`) calls `depositETH` and receives rsETH.
2. It calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned; ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager`. [1](#0-0) 
4. The Safe calls `completeWithdrawal(ETH_TOKEN, "")`. `_transferAsset` executes `payable(safeAddress).call{value: amount}("")`. The Safe has no `receive()`, so the call returns `success = false`. The function reverts with `EthTransferFailed`. [6](#0-5) 
5. The withdrawal request is preserved (revert undoes the delete), but rsETH remains burned. Step 4 will revert on every future attempt. ETH is permanently locked.

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
