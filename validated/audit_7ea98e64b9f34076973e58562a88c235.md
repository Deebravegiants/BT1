### Title
Fee Bypass via Integer Truncation in Deposit Fee Calculation - (File: contracts/pools/RSETHPool.sol)

### Summary
All RSETHPool and AGETHPool contracts compute swap fees using integer division without rounding up. When a depositor passes a sufficiently small `amount`, the expression `amount * feeBps / 10_000` truncates to zero, allowing the depositor to receive rsETH/agETH with no fee paid to the protocol.

### Finding Description
Every pool contract in the repository computes the protocol fee identically:

```solidity
fee = amount * feeBps / 10_000;
```

Because Solidity integer division truncates toward zero, any `amount` satisfying `amount * feeBps < 10_000` produces `fee = 0`. For example, with a typical `feeBps = 5` (0.05%), any deposit of fewer than **2 000 wei** incurs zero fee. With `feeBps = 100` (1%), the threshold is fewer than **100 wei**.

An attacker can split a larger intended deposit into many sub-threshold transactions, each paying zero fee, and receive the full rsETH/agETH output as if no fee existed. On L2 networks (Arbitrum, Base, etc.) where these pools are deployed, per-transaction gas costs are low enough to make this economically rational.

The pattern is present in every public `deposit()` entry point across the codebase:

### Impact Explanation
The protocol collects fees on every deposit to fund operations and yield. By making many micro-deposits below the truncation threshold, an attacker avoids all fee payments while still receiving the full rsETH/agETH output. This constitutes **theft of unclaimed yield** (protocol fee revenue) — classified as **High** impact under the allowed scope.

### Likelihood Explanation
The attack requires no special privileges; any depositor can call `deposit()`. On L2 chains (Arbitrum, Base, Optimism, etc.) where these contracts are deployed, gas costs per transaction are on the order of cents, making it economically viable to split deposits into many sub-threshold calls. The threshold is small (e.g., < 2 000 wei for a 0.05% fee), so even modest deposits can be split into thousands of fee-free transactions.

### Recommendation
Round up the fee calculation so that any non-zero deposit always incurs at least 1 wei of fee:

```solidity
// Before (truncates to zero for small amounts):
fee = amount * feeBps / 10_000;

// After (rounds up):
fee = (amount * feeBps + 9_999) / 10_000;
// or equivalently using a math library:
fee = Math.ceilDiv(amount * feeBps, 10_000);
```

Apply this fix to every `viewSwapRsETHAmountAndFee` / `viewSwapAgETHAmountAndFee` function across all pool contracts listed above.

### Proof of Concept
Assume `feeBps = 5` (0.05%) and rsETH/ETH rate = 1e18 (1:1 for simplicity).

1. Attacker wants to deposit 1 ETH (1e18 wei) without paying fees.
2. Instead of one call, attacker makes 500 000 calls each depositing 1 999 wei.
3. For each call: `fee = 1999 * 5 / 10_000 = 9995 / 10_000 = 0` (truncated).
4. Each call receives `rsETHAmount = 1999 * 1e18 / 1e18 = 1999` rsETH wei with zero fee.
5. Total deposited: `500_000 * 1999 = 999_500_000` wei ≈ 1 ETH, total fee paid: **0**.
6. Expected fee at 0.05%: `1e18 * 5 / 10_000 = 5e13` wei ≈ 0.00005 ETH — entirely stolen from the protocol.

On Arbitrum, 500 000 transactions at ~$0.001 each costs ~$500 in gas to steal ~$0.15 in fees on a 1 ETH deposit. However, scaled to large depositors (e.g., 10 000 ETH), the stolen fees (~$1 500) far exceed the gas cost, making the attack profitable.
