### Title
Validation-Skip Mechanism Admits Invoke Transactions with Invalid Signatures for Undeployed Accounts — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the account's `__validate__` entrypoint (signature verification) for any invoke transaction with `nonce == 1` targeting an address whose `account_nonce == 0` and which has **any** transaction present in the mempool or a recent block. Because the check does not verify that the submitter of the invoke is the same party who submitted the deploy_account, an unprivileged attacker can submit an invoke with an arbitrary/invalid signature for a victim's undeployed account, have it admitted to the mempool without signature verification, and displace the victim's legitimate first invoke via fee escalation.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` (lines 429–461) implements the following logic:

```
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
```

When this returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false` (line 312), which causes `StatefulValidator::perform_validations` to return `Ok(())` immediately without calling the account's `__validate__` entrypoint (lines 79–81 of `crates/blockifier/src/blockifier/stateful_validator.rs`).

`account_tx_in_pool_or_recent_block` (line 697 of `crates/apollo_mempool/src/mempool.rs`) returns `true` if the address appears in `self.state` (staged/committed) or `self.tx_pool`:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

The code comment asserts: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is incorrect: the check does not verify that the entity submitting the invoke is the same entity that submitted the deploy_account. Any third party can observe a deploy_account in the mempool and immediately submit an invoke with `nonce=1` for that address, triggering the skip.

The `validate_by_mempool` call that precedes the skip check (line 406) only validates nonce ordering and fee escalation — it does not verify the signature.

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the mempool can submit `Invoke(sender=victim_address, nonce=1, calldata=arbitrary, signature=attacker_key)`. The gateway:

1. Passes `validate_state_preconditions` (nonce 1 ≥ 0, within `max_allowed_nonce_gap`).
2. Passes `validate_by_mempool` (no duplicate hash, nonce not too old).
3. Evaluates `skip_stateful_validations` → `account_tx_in_pool_or_recent_block` returns `true` → `skip_validate = true`.
4. Calls `run_validate_entry_point` with `execution_flags.validate = false` → `__validate__` is **never called** → transaction admitted to mempool.

If the victim also submitted a legitimate `invoke(nonce=1)`, the attacker's higher-fee version replaces it via fee escalation. At block execution time, the batcher creates `AccountTransaction` with `validate=true` (sequencing default), so `__validate__` is called and the attacker's transaction reverts. The nonce is still incremented to 2, making the victim's original `invoke(nonce=1)` permanently stale. The victim must resubmit with `nonce=2`.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The attack window is the interval between a `deploy_account` being admitted to the mempool and the block being sealed. An attacker monitoring the mempool's public `account_tx_in_pool_or_recent_block` endpoint (exposed as a gRPC/IPC call) can detect this condition and submit the malicious invoke within the same block window. No privileged access is required. The attacker pays fees for the reverted transaction, making this a griefing attack with a small but non-zero cost.

### Recommendation

The skip criterion must be tightened so that only the **owner** of the deploy_account can benefit from the skip. Concretely:

1. Store the `sender_address` (or transaction hash) of the pending `deploy_account` in the mempool alongside the account entry.
2. In `skip_stateful_validations`, additionally verify that the invoke's `sender_address` matches the address that submitted the deploy_account, or that the deploy_account transaction hash was explicitly provided by the submitter (as is done in the legacy `PyValidator::should_run_stateful_validations` via the `deploy_account_tx_hash` parameter).
3. Alternatively, restrict the skip to cases where the deploy_account and invoke arrive in the **same gateway request** (atomic submission), preventing third-party exploitation.

### Proof of Concept

```
1. Victim submits:
     deploy_account(class_hash=C, salt=S, calldata=D)  → address V, nonce 0
     invoke(sender=V, nonce=1, calldata=victim_call, sig=victim_sig)

2. Attacker observes deploy_account for V in mempool.

3. Attacker submits:
     invoke(sender=V, nonce=1, calldata=attacker_call, sig=attacker_sig, tip=HIGH)

4. Gateway evaluation for attacker's invoke:
     validate_state_preconditions: account_nonce=0, tx_nonce=1 → OK
     validate_by_mempool: no dup hash, nonce 1 >= 0 → OK
     skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0 → true
       account_tx_in_pool_or_recent_block(V) → true (victim's deploy_account is there)
       → skip_validate = true
     run_validate_entry_point: execution_flags.validate=false → __validate__ NOT called
     → attacker's invoke admitted to mempool

5. Fee escalation: attacker's higher-tip invoke replaces victim's invoke(nonce=1).

6. Block execution:
     deploy_account(V, nonce=0) → success, V deployed, nonce→1
     attacker's invoke(V, nonce=1) → __validate__ called → sig check fails → REVERT, nonce→2

7. Victim's original invoke(nonce=1) is stale; victim must resubmit with nonce=2.
```

**Corrupted value:** The mempool contains `invoke(sender=V, nonce=1, sig=attacker_sig)` — a transaction that will fail account validation at execution time — admitted without signature verification due to the unconstrained skip criterion in `skip_stateful_validations`. async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
