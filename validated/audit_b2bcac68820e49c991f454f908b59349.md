### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Single Byzantine Participant to Forge Foreign-Chain Verification Results - (File: crates/contract/src/lib.rs)

### Summary

The `respond_verify_foreign_tx` function accepts a caller-supplied `payload_hash` in the response without verifying that it corresponds to the `request` being resolved. A single Byzantine attested participant (below threshold) can use a valid threshold signature obtained for one foreign-tx request to forge the verification result for any other pending foreign-tx request, delivering a cryptographically valid but semantically incorrect attestation to the caller.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` verifies only that the signature is valid over the caller-supplied `response.payload_hash`:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract does **not** verify that `response.payload_hash` is the correct hash for the `request` being resolved. The `payload_hash` is entirely caller-supplied and is never bound to the pending request.

Contrast this with the regular `respond()` function, where the payload hash is derived from the `request` itself (stored on-chain at submission time), making it impossible to substitute:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
.is_ok()
``` [2](#0-1) 

For `verify_foreign_transaction`, the correct `payload_hash` is `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))`, where `extracted_values` are obtained by querying the foreign chain at signing time. [3](#0-2) 

Because the contract does not know the extracted values at request time, it cannot recompute the expected hash — and currently makes no attempt to bind `payload_hash` to `request`. The only checks before resolving the yield are participant attestation and signature validity: [4](#0-3) 

The attack path for a single Byzantine leader node:

1. Node is elected leader for **Request A** (`tx_id=0x1234`).
2. Node participates in the threshold protocol for Request A, obtaining a valid threshold signature `sig_A` over `payload_hash_A = SHA-256(borsh({request_A, [block_hash_A]}))`.
3. Instead of calling `respond_verify_foreign_tx(request=A, response={payload_hash_A, sig_A})`, the Byzantine node calls `respond_verify_foreign_tx(request=B, response={payload_hash_A, sig_A})` for a different pending **Request B** (`tx_id=0x5678`).
4. The contract accepts: `sig_A` is a valid MPC signature, and Request B exists in `pending_verify_foreign_tx_requests`.
5. Request B is resolved and removed from pending state; its caller receives `{payload_hash_A, sig_A}`.

The node-side signing code confirms that `payload_hash` is computed from the actual foreign chain query result and is not transmitted to the contract in a verifiable way: [5](#0-4) 

The SDK's `ForeignChainSignatureVerifier::verify_signature` does check that `expected_payload_hash == response.payload_hash`, but this is a client-side check in bridge contracts — not enforced by the MPC contract itself: [6](#0-5) 

### Impact Explanation

A single Byzantine attested participant (below threshold) can forge the verification result for any pending `verify_foreign_transaction` request. The forged `VerifyForeignTransactionResponse` contains a cryptographically valid MPC threshold signature, making it indistinguishable from a legitimate response at the signature-verification level. Bridge contracts that do not independently recompute and compare the `payload_hash` (e.g., those not using the SDK's `ForeignChainSignatureVerifier`) will accept the forged attestation and may release funds for a foreign-chain transaction that was never actually verified. Even bridge contracts that do use the SDK suffer a DoS: Request B is permanently resolved with the forged response, preventing any legitimate response from being submitted for it.

This directly enables **forged foreign-chain verification** and **invalid bridge execution** — the primary use case of this feature is the Omnibridge inbound flow where funds are released on NEAR upon verified foreign-chain deposit.

### Likelihood Explanation

The attacker must be a single attested MPC participant elected as leader for any foreign-tx request. Leader election is deterministic and rotates across participants, so any Byzantine participant will eventually be elected leader. No threshold collusion is required — one compromised or malicious node suffices. The attack is silent (the forged call is structurally identical to a legitimate `respond_verify_foreign_tx` call) and requires no special tooling beyond submitting a NEAR transaction.

### Recommendation

Bind `response.payload_hash` to the `request` in the contract. Two options:

1. **Include extracted values in the response**: Have nodes submit the full `ForeignTxSignPayload` (including `values`), and have the contract recompute `SHA-256(borsh(payload))` and verify it matches `response.payload_hash`. This increases response size but closes the gap entirely.
2. **Domain-bind the signature**: Modify the signing protocol so nodes sign `SHA-256(request_id || payload_hash)` instead of `payload_hash` alone. The contract can then verify the signature over `SHA-256(request_id || response.payload_hash)`, binding the signature to the specific request without needing the extracted values.

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(tx_id=0x1234, extractors=[BlockHash])
   → pending_verify_foreign_tx_requests[request_A] = [yield_A]

2. Bob submits verify_foreign_transaction(tx_id=0x5678, extractors=[BlockHash])
   → pending_verify_foreign_tx_requests[request_B] = [yield_B]

3. Byzantine node Eve is elected leader for request_A.
   Eve queries the foreign chain, extracts block_hash_A, computes:
     payload_hash_A = SHA-256(borsh(ForeignTxSignPayload{request_A, [block_hash_A]}))
   Eve participates in threshold protocol → obtains (sig_A, pk)

4. Eve calls:
   respond_verify_foreign_tx(
     request = request_B,          // ← Bob's request
     response = {
       payload_hash: payload_hash_A, // ← hash of Alice's data
       signature: sig_A              // ← valid threshold sig over payload_hash_A
     }
   )

5. Contract checks:
   - Eve is attested participant ✓
   - verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) ✓
   - request_B exists in pending_verify_foreign_tx_requests ✓
   → resolve_yields_for(request_B, {payload_hash_A, sig_A}) ← accepted

6. Bob's bridge contract receives {payload_hash_A, sig_A}.
   If it does not recompute and compare payload_hash for tx_id=0x5678,
   it accepts the forged attestation and releases funds.
   Request_B is permanently consumed; no legitimate response can be submitted.
```
