### Title
Post-Batch Reference Script Size Check Allows Excessive Computation Before Limit Enforcement in Dijkstra LEDGER Rule - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`)

---

### Summary

In the Dijkstra era's `dijkstraLedgerTransition`, all sub-transactions are fully processed by the `SUBLEDGERS` rule — including script deserialization and execution — before the batch reference script size limit (`ppMaxRefScriptSizePerTxG`) is enforced via `validateAllRefScriptSize`. This is a direct structural analog to the JSON-RPC post-serialization size check: expensive work is fully materialized before the resource cap is applied, allowing an unprivileged transaction sender to force validators to perform computation that exceeds the intended per-batch resource budget before the transaction is rejected.

---

### Finding Description

In `dijkstraLedgerTransition`, the execution order is:

1. **Line 370–383**: `trans @(EraRule "SUBLEDGERS" era)` — all sub-transactions are processed sequentially via `foldM` in `dijkstraSubLedgersTransition`, each going through `SUBLEDGER → SUBUTXOW → SUBUTXO`. This includes full script deserialization and phase-1/phase-2 validation for every sub-transaction.

2. **Line 392**: `runTest $ validateAllRefScriptSize pp originalUtxo tx` — only after all sub-transactions are processed does the ledger check whether the total reference script size across the batch (top-level + all sub-transactions) exceeds `ppMaxRefScriptSizePerTxG`. [1](#0-0) 

The `validateAllRefScriptSize` function uses `batchNonDistinctRefScriptsSize`, which sums reference script sizes across the top-level transaction and all sub-transactions: [2](#0-1) 

Critically, the `SUBLEDGER` rule (and its sub-rules `SUBUTXOW`/`SUBUTXO`) does **not** apply any per-sub-transaction reference script size check. The only reference script size enforcement for sub-transactions is the batch check in `LEDGER`, which fires after all sub-transactions have been processed. [3](#0-2) 

In Conway, `ppMaxRefScriptSizePerTxG` is hardcoded to `200 * 1024` bytes (200 KiB). In Dijkstra it becomes a proper protocol parameter, but the ordering flaw is present in both: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

The `ppMaxRefScriptSizePerTxG` limit exists specifically to bound the deserialization cost of reference scripts during block validation (as documented in the Conway ADR on reference script fee changes). By processing all sub-transactions before this check, the limit is bypassed for the computation phase: a validator node fully deserializes and validates all reference scripts across all sub-transactions before discovering the batch exceeds the limit and rejecting it.

An attacker who pre-populates the UTxO with large reference scripts can craft a top-level transaction with N sub-transactions, each referencing scripts whose combined size per sub-transaction stays below any individual check, but whose batch total far exceeds `ppMaxRefScriptSizePerTxG`. The node performs all deserialization and script validation work before the rejection. This allows attacker-controlled transactions to exceed the intended per-batch validation resource budget, matching the **Medium** impact: *attacker-controlled transactions exceed intended validation limits*.

---

### Likelihood Explanation

The attack path requires only an unprivileged transaction sender. The attacker must first create UTxO entries containing large reference scripts (bounded by fee costs), then submit a top-level transaction with multiple sub-transactions referencing those scripts. No privileged access, governance majority, or key compromise is required. The Dijkstra era's sub-transaction feature is the direct enabler: it allows a single top-level transaction to carry an unbounded number of sub-transactions (the CDDL only requires `nonempty_oset`, with no count cap enforced in the ledger rules), each of which is processed before the batch size check fires. [6](#0-5) 

---

### Recommendation

Move `validateAllRefScriptSize` (or an equivalent per-sub-transaction check) to execute **before** `trans @(EraRule "SUBLEDGERS" era)` in `dijkstraLedgerTransition`. Specifically:

- Apply a per-sub-transaction reference script size check inside `SUBLEDGER`/`SUBUTXO` (analogous to how `validateRefScriptSize` is applied per-transaction in Conway's `LEDGER` rule).
- Alternatively, enforce the batch size check at the start of `dijkstraLedgerTransition`, before any sub-transaction processing, by computing `batchNonDistinctRefScriptsSize` eagerly and rejecting early.

This mirrors the "Option B" mitigation from the external report: pre-reject oversized batches before materializing results.

---

### Proof of Concept

1. Create UTxO entries containing large Plutus reference scripts (e.g., 50 KiB each), paying the required fees.
2. Craft a top-level Dijkstra transaction with 5 sub-transactions, each referencing 4 of those scripts (200 KiB per sub-transaction × 5 = 1 MiB batch total, far exceeding the 200 KiB `ppMaxRefScriptSizePerTxG` limit in Conway / configurable in Dijkstra).
3. Submit the transaction to a block producer.
4. The node executes `dijkstraSubLedgersTransition`, processing all 5 sub-transactions (deserializing all 20 reference scripts, running phase-1 validation) before reaching `validateAllRefScriptSize` at line 392.
5. `validateAllRefScriptSize` fires, computes `batchNonDistinctRefScriptsSize = 1 MiB > ppMaxRefScriptSizePerTxG`, and rejects the transaction — but only after the full deserialization and validation cost has been paid by the validator. [7](#0-6)

  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $

          Mismatch
            { mismatchSupplied = totalRefScriptSize

batchNonDistinctRefScriptsSize ::
  ( EraTx era
  , DijkstraEraTxBody era
  ) =>
  UTxO era ->
  Tx TopTx era ->
  Int
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
