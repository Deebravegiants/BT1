### Title
Missing Lower-Bound Validation on `BlockConfirmations` Allows Zero-Confirmation Bitcoin Transaction Signing - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The Bitcoin inspector's finality check only validates that the RPC-returned confirmation count meets the caller-supplied threshold (`threshold <= actual`), but never enforces that the threshold itself is `>= 1`. An unprivileged caller can submit `BlockConfirmations(0)` in a `verify_foreign_transaction` request, causing every MPC node to unconditionally pass the finality gate and produce a threshold signature over an unconfirmed (or minimally confirmed) Bitcoin transaction. This is a direct structural analog to the zkevm-rom BYTE bug: both perform a one-sided comparison that ignores the other component of the range, allowing an out-of-range input to silently pass.

---

### Finding Description

`BlockConfirmations` is a plain `u64` newtype with no minimum enforcement at the type level or at the contract's `verify_foreign_transaction` entry point. [1](#0-0) 

The Bitcoin inspector's finality gate is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { … });
}
``` [2](#0-1) 

When `block_confirmations_threshold = 0`, the predicate `0 <= x` is trivially true for every `u64` value `x`, including `x = 0` (mempool transaction). The guard is never triggered. Execution continues to `verify_block_is_canonical` and then to value extraction and signing.

The contract's `verify_foreign_transaction` performs no validation of the `confirmations` field before enqueuing the yield request: [3](#0-2) 

The signed `ForeignTxSignPayload` embeds the full `BitcoinRpcRequest` (including `confirmations: BlockConfirmations(0)`), so the payload hash commits to the zero threshold — but the MPC network itself never refuses to sign it. [4](#0-3) 

**Structural analog to the BYTE bug:**

| | zkevm-rom BYTE bug | NEAR MPC analog |
|---|---|---|
| Input | 256-bit value B split into limbs B0–B7 | `BlockConfirmations` threshold (u64) |
| Check performed | `31 - B0 >= 0` (only low limb) | `threshold <= actual` (only upper direction) |
| Missing check | B1–B7 == 0 | `threshold >= 1` |
| Effect | Out-of-range B passes; wrong byte extracted | Zero threshold passes; unconfirmed tx signed |

---

### Impact Explanation

A valid MPC threshold signature is produced over a `ForeignTxSignPayload` that attests to a Bitcoin transaction with zero required confirmations. In a bridge context (the primary stated use-case of `verify_foreign_transaction`), a downstream NEAR contract that does not independently re-validate the `BlockConfirmations` field in the signed payload will release funds against an unconfirmed transaction. The attacker can then broadcast a conflicting double-spend transaction on Bitcoin, causing the bridge to release funds for a transaction that is subsequently invalidated. This matches the **High** impact category: forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.

---

### Likelihood Explanation

The entry point `verify_foreign_transaction` is a public, payable contract method callable by any NEAR account with a 1 yoctoNEAR deposit. No privileged role is required. The `BlockConfirmations` field is a plain JSON integer in the request body; submitting `"confirmations": 0` requires no special tooling. The `near-mpc-sdk`'s `ForeignChainSignatureVerifier` does not validate the `BlockConfirmations` value either, so downstream contracts built with the SDK inherit the gap. [5](#0-4) 

---

### Recommendation

1. **Enforce a minimum at the type boundary or at the contract entry point.** The simplest fix is to reject `BlockConfirmations(0)` in `verify_foreign_transaction` before the request is enqueued:

```rust
if request.confirmations.0 == 0 {
    env::panic_str("BlockConfirmations must be >= 1");
}
```

2. **Enforce the minimum in the Bitcoin inspector** as a defence-in-depth measure, mirroring the pattern used for `quorum` in `ChainEntry`:

```rust
if block_confirmations_threshold.0 == 0 {
    return Err(ForeignChainInspectionError::InvalidConfirmationsThreshold);
}
``` [6](#0-5) 

3. **Document the minimum in `BlockConfirmations`** so SDK users know to validate it in downstream contracts.

### Proof of Concept

```json
// POST to verify_foreign_transaction on the MPC contract
{
  "request": {
    "request": {
      "Bitcoin": {
        "tx_id": "<hex of a mempool or 1-conf tx>",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    },
    "domain_id": <foreign_tx_domain_id>,
    "payload_version": 1
  }
}
```

1. Attacker broadcasts a Bitcoin transaction `T` to the mempool.
2. Attacker calls `verify_foreign_transaction` with `confirmations: 0` and `tx_id = T`.
3. Each MPC node calls `getrawtransaction(T, verbose=true)`. Bitcoin returns `confirmations: 0` (mempool) or `confirmations: 1` (just mined).
4. The check `0 <= 0` (or `0 <= 1`) passes in every node's inspector.
5. Nodes proceed to extract `BlockHash` and sign `SHA256(borsh(ForeignTxSignPayload { request, values }))`.
6. The contract receives `respond_verify_foreign_tx` with a valid threshold signature.
7. The downstream NEAR bridge contract, if it does not re-check `confirmations`, releases funds.
8. Attacker broadcasts a double-spend of `T` on Bitcoin; the original transaction is replaced. [7](#0-6) [1](#0-0)
