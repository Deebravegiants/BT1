Audit Report

## Title
Permanent Freezing of Unclaimed Yield When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary

`KernelDepositPool` uses a Synthetix-style reward accounting model where `rewardPerToken()` returns the frozen `rewardPerTokenStored` value when `totalKernelStaked == 0`, causing all rewards emitted during that period to be permanently untrackable. Because `initiateWithdrawal` immediately decrements `totalKernelStaked` before the withdrawal delay elapses (up to 30 days), any last staker — intentionally or not — can create a multi-day gap during which all emitted rewards are irrecoverably locked in the contract.

## Finding Description

In `rewardPerToken()` (L408–414), when `totalKernelStaked == 0`, the function returns `rewardPerTokenStored` unchanged:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;  // frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

`initiateWithdrawal` (L320–338) immediately decrements `totalKernelStaked` at call time, before the withdrawal delay elapses:

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
```

The `updateReward` modifier (L232–242) runs before the function body (`_;` is last), so it snapshots `rewardPerTokenStored` and sets `updatedAt = lastTimeRewardApplicable()` at the moment of the call — correctly accounting for rewards up to that point. After this, `totalKernelStaked` becomes 0 and the reward clock freezes.

When the next staker calls `stake()`, `updateReward` runs again while `totalKernelStaked` is still 0 (the increment happens after `_;`), so `rewardPerToken()` again returns the frozen value, and `updatedAt` is reset to the current time. The entire gap interval is silently skipped — `rewardRate * gap_duration` worth of reward tokens are permanently locked in the contract with no mechanism to recover or redistribute them.

The contract's own NatSpec (L18–22) acknowledges this risk but relies entirely on an operational mitigation (admin ensures tokens are always staked), with no on-chain enforcement. `notifyRewardAmount` (L566–570) guards against starting a reward period with zero staked tokens, but provides no protection against `totalKernelStaked` dropping to zero mid-period.

## Impact Explanation

Any rewards emitted by `rewardRate` during the period when `totalKernelStaked == 0` are permanently locked in the contract. No staker — present or future — can ever claim them. `rewardsToken.balanceOf(address(KernelDepositPool))` will permanently exceed the sum of all claimable rewards by `rewardRate * gap_duration`. With `MAX_WITHDRAWAL_DELAY = 30 days` (L35), the locked amount can be `rewardRate * 30 days` worth of reward tokens. This matches **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation

The trigger is a call to the public, permissionless `initiateWithdrawal` function by the last remaining staker. This requires no special role. It can occur naturally (all stakers exit simultaneously during a market downturn or protocol migration) or deliberately (a sole staker calls `initiateWithdrawal` to zero out `totalKernelStaked`, waits out the delay, then re-stakes, permanently destroying gap rewards). The `withdrawalDelay` is admin-configurable up to 30 days, making the potential loss window large. The scenario is realistic and repeatable.

## Recommendation

When `totalKernelStaked` drops to zero, the reward clock must be paused in a way that is resumed correctly when staking resumes. Two concrete options:

1. **Extend `finishAt`**: Track the cumulative duration during which `totalKernelStaked == 0` and extend `finishAt` by that amount when staking resumes, so no rewards are lost.
2. **Revert on last-staker withdrawal during active period**: In `initiateWithdrawal`, revert if the withdrawal would reduce `totalKernelStaked` to zero while `block.timestamp < finishAt`, forcing the last staker to wait until the reward period ends before fully withdrawing.

The `updateReward` modifier already correctly snapshots `updatedAt = lastTimeRewardApplicable()` at the moment of the call, so the fix only needs to handle the resumption of the clock when the next staker arrives.

## Proof of Concept

1. Admin calls `notifyRewardAmount` with `rewardRate = R` and `duration = D`. `finishAt = now + D`. `totalKernelStaked > 0` (required by L570).
2. Alice is the only staker with `totalKernelStaked = X`.
3. Alice calls `initiateWithdrawal(X)`. `updateReward(alice)` snapshots her earned rewards correctly up to this point. `totalKernelStaked` becomes 0 immediately.
4. For the next `withdrawalDelay` seconds (up to 30 days), `rewardPerToken()` returns the frozen `rewardPerTokenStored`. All `R * withdrawalDelay` reward tokens emitted during this window are untracked.
5. Alice calls `claimWithdrawal` and gets her KERNEL back. She re-stakes. `updateReward` runs with `totalKernelStaked == 0`, returns frozen value, resets `updatedAt` to now. Future rewards resume normally, but gap rewards are gone.
6. `rewardsToken.balanceOf(address(KernelDepositPool))` permanently exceeds the sum of all claimable rewards by `R * withdrawalDelay`.

Foundry test plan: deploy contract, stake, `notifyRewardAmount`, `initiateWithdrawal` (last staker), `warp` by `withdrawalDelay`, `claimWithdrawal`, re-stake, warp to `finishAt`, call `getReward` for all users, assert `rewardsToken.balanceOf(contract) > 0` (stranded rewards).
