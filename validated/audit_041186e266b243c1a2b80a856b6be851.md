### Title
Blacklisted ERC-20 Recipient Permanently Locks Bridged Funds in `finTransfer` - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.finTransfer` marks the destination nonce as consumed **before** attempting the token transfer to the recipient. When the token is a blacklist-enforcing ERC-20 (e.g., USDC, USDT), a blacklisted recipient causes `safeTransfer` to revert, rolling back the entire transaction including the nonce mark. Because the recipient address is cryptographically bound in the MPC-signed payload and no cancel/redirect mechanism exists, the user's funds are permanently unclaimable: burned or locked on the origin chain, undeliverable on EVM.

### Finding Description

In `OmniBridge.finTransfer`, the execution order is:

1. Check and set `completedTransfers[payload.destinationNonce] = true`
2. Verify MPC signature over a payload that includes `payload.recipient`
3. Dispatch the token to `payload.recipient` [1](#0-0) 

For the native-token (locked) path, the dispatch is: [2](#0-1) 

`safeTransfer` (OpenZeppelin) reverts on failure. When the recipient is USDC/USDT-blacklisted, the revert unwinds the entire transaction, including the nonce mark at line 287. The nonce is therefore **not** consumed, but the signed payload is permanently bound to the blacklisted address — the MPC signature covers `payload.recipient` directly: [3](#0-2) 

A new MPC signature with a different recipient would require a new `sign_transfer` call on NEAR, which re-signs the existing stored `TransferMessage`. That message's `recipient` field was fixed at `init_transfer` time and cannot be changed. There is no cancel-and-refund path for the origin-chain tokens once `init_transfer` has burned or locked them.

The same pattern exists in the Starknet bridge: the nonce is set before the transfer, and a failed `transfer` return value causes an `assert` panic that reverts the nonce mark, leaving the signed payload permanently unexecutable: [4](#0-3) 

By contrast, the NEAR-side `fin_transfer_send_tokens_callback` **does** handle delivery failure gracefully by reverting lock actions and removing the transfer record: [5](#0-4) 

No equivalent recovery exists on EVM or Starknet.

### Impact Explanation

A user who bridges a blacklist-enforcing token (USDC, USDT) from any origin chain to EVM or Starknet, and whose destination address is subsequently blacklisted by the token issuer, will have their funds permanently locked:

- Origin-chain tokens are burned (deployed/wrapped token) or locked (native token) at `init_transfer` time.
- Every `finTransfer` attempt reverts at `safeTransfer`, leaving the nonce unconsumed but the payload unexecutable.
- No cancel, redirect, or refund path exists.

This satisfies: **Critical — Irreversible fund lock, frozen redemption path, or permanently unclaimable user value in bridge flows.**

### Likelihood Explanation

USDC and USDT are the most commonly bridged stablecoins. Both enforce on-chain blacklists. Circle has blacklisted thousands of addresses. A user whose address is blacklisted after initiating a bridge transfer — or who unknowingly uses a blacklisted address as recipient — triggers this condition through normal protocol use with no privileged access required.

### Recommendation

1. **Wrap the token dispatch in a try/catch (EVM):** Use a low-level call or `try` pattern so that a failed transfer does not revert the nonce mark. Instead, credit the amount to a claimable balance mapping keyed by `(nonce, recipient)`, allowing the user to withdraw via a separate `claim` function.
2. **Add a recipient-redirect or cancel path on NEAR:** Allow the sender to update the recipient address in a stored `TransferMessage` before the MPC signs it, or provide a cancel function that refunds origin-chain tokens when a transfer has been pending beyond a timeout.
3. **Starknet:** Apply the same try/credit pattern instead of asserting on the `transfer` return value.

### Proof of Concept

1. Alice holds 10,000 USDC on NEAR (bridged). She calls `init_transfer` targeting her EVM address `0xAlice`. Tokens are burned on NEAR; a `TransferMessage` with `recipient = 0xAlice` is stored.
2. The MPC signs a `TransferMessagePayload` binding `recipient = 0xAlice`.
3. Circle blacklists `0xAlice` (e.g., due to a sanctions designation).
4. A relayer calls `OmniBridge.finTransfer(signature, payload)`.
5. `completedTransfers[nonce] = true` executes.
6. `IERC20(USDC).safeTransfer(0xAlice, 10000e6)` reverts — USDC blacklist check fails.
7. The entire transaction reverts; `completedTransfers[nonce]` is rolled back to `false`.
8. Every subsequent `finTransfer` attempt with the same signed payload repeats steps 4–7.
9. Alice's 10,000 USDC is permanently burned on NEAR and undeliverable on EVM. [6](#0-5)
