### Title
`ClassHash::default()` in `BroadcastedDeclareV1Transaction` Conversion Causes Wrong Transaction Hash and State Diff in `starknet_estimateFee` / `starknet_simulateTransactions` — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

When an unprivileged RPC client submits a `BroadcastedDeclareV1Transaction` to `starknet_estimateFee` or `starknet_simulateTransactions`, the conversion to `ExecutableTransactionInput::DeclareV1` hard-codes `class_hash: ClassHash::default()` (i.e., `0`). This zero value propagates into two downstream computations: (1) the transaction hash fed to the blockifier (and thus to the `get_tx_info()` syscall seen by `__validate__`), and (2) the `deprecated_declared_classes` field of the returned simulation state diff. Both values are wrong relative to the real class hash derived from the submitted contract class.

---

### Finding Description

**Step 1 — Conversion sets `class_hash = 0`**

In `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput`, the V1 arm constructs a `DeclareTransactionV0V1` with a hard-coded zero class hash:

```rust
// crates/apollo_rpc/src/v0_8/api/mod.rs, lines 494–508
Ok(Self::DeclareV1(
    starknet_api::transaction::DeclareTransactionV0V1 {
        max_fee,
        signature,
        nonce,
        // The blockifier doesn't need the class hash, but it uses the SN_API
        // DeclareTransactionV0V1 which requires it.
        class_hash: ClassHash::default(),   // ← always 0
        sender_address,
    },
    sn_api_contract_class,
    abi_length,
    false,
))
``` [1](#0-0) 

**Step 2 — Hash computation uses the zero class hash**

`calc_tx_hash` calls `get_transaction_hash` → `get_declare_transaction_v1_hash`, which chains `transaction.class_hash.0` directly into the Pedersen hash:

```rust
// crates/starknet_api/src/transaction_hash.rs, lines 550–561
HashChain::new()
    .chain(&DECLARE)
    .chain(&transaction_version.0)
    .chain(transaction.sender_address.0.key())
    .chain(&Felt::ZERO)
    .chain(&HashChain::new().chain(&transaction.class_hash.0).get_pedersen_hash())  // ← 0
    .chain(&transaction.max_fee.0.into())
    .chain(&Felt::try_from(chain_id)?)
    .chain(&transaction.nonce.0)
    .get_pedersen_hash()
``` [2](#0-1) 

The resulting `tx_hash` (computed with `class_hash=0`) is passed to `to_blockifier_tx` and then to `BlockifierTransaction::from_api`, making it the value returned by the `get_tx_info()` syscall inside `__validate__`. [3](#0-2) 

**Step 3 — State diff records `ClassHash(0)` as the declared class**

Before calling `to_blockifier_tx`, `execute_transactions` reads `deprecated_declared_class_hash` from the same `DeclareTransactionV0V1.class_hash` field:

```rust
// crates/apollo_rpc_execution/src/lib.rs, lines 740–745
ExecutableTransactionInput::DeclareV1(
    DeclareTransactionV0V1 { class_hash, .. }, _, _, _,
) => Some(*class_hash),   // ← ClassHash(0)
``` [4](#0-3) 

This is passed to `induced_state_diff`, which places it in `ThinStateDiff.deprecated_declared_classes`:

```rust
// crates/apollo_rpc_execution/src/execution_utils.rs, lines 141–142
deprecated_declared_classes: deprecated_declared_class_hash
    .map_or_else(Vec::new, |class_hash| vec![class_hash]),
``` [5](#0-4) 

The simulation response therefore reports `deprecated_declared_classes: [0x0]` instead of the real class hash.

---

### Impact Explanation

Two authoritative-looking wrong values are returned to the caller:

1. **Wrong `transaction_hash` in `__validate__` syscall**: The account's `__validate__` entrypoint receives a hash computed with `class_hash=0`. Any account that verifies the ECDSA signature over the transaction hash (standard OpenZeppelin-style accounts) will see a signature mismatch and revert. This causes `starknet_estimateFee` to report a reverted/failed result for a transaction that is actually valid on-chain — a false negative that misleads clients about whether their transaction will succeed.

2. **Wrong `deprecated_declared_classes` in simulation state diff**: The `SimulatedTransaction` returned by `starknet_simulateTransactions` includes `deprecated_declared_classes: [ClassHash(0)]` instead of the real class hash. A client using this to predict post-state or to verify what class was declared receives a completely wrong value.

Both fit: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

- Reachable by any unauthenticated RPC client with a single JSON-RPC call.
- Affects every `BroadcastedDeclareV1Transaction` submitted to `starknet_estimateFee` or `starknet_simulateTransactions` — 100% reproduction rate.
- The code path is unconditional; there is no guard that computes the real class hash before setting it to zero.

---

### Recommendation

Before constructing the `DeclareTransactionV0V1`, compute the real class hash from the submitted `sn_api_contract_class` (e.g., via `ClassHash::from(&sn_api_contract_class)` or the equivalent hash function used by the gateway), and use that value instead of `ClassHash::default()`. The comment "The blockifier doesn't need the class hash" is incorrect with respect to transaction hash computation and state diff reporting.

---

### Proof of Concept

Concrete execution trace (no test code needed — the static path is deterministic):

1. Client calls `starknet_estimateFee` with a `BroadcastedDeclareV1Transaction` containing a known contract class whose real class hash is `H_real`.
2. `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput` sets `class_hash = ClassHash::default() = 0x0`.
3. `calc_tx_hash` → `get_declare_transaction_v1_hash` computes `H_wrong = pedersen(... pedersen(0x0) ...)` instead of `pedersen(... pedersen(H_real) ...)`.
4. The blockifier executes `__validate__` with `get_tx_info().transaction_hash = H_wrong`.
5. The account verifies `ecdsa_verify(pubkey, H_wrong, sig)` — this fails because the client signed `H_real`.
6. `starknet_estimateFee` returns a reverted-transaction error for a valid transaction.
7. Separately, `starknet_simulateTransactions` (with `SkipValidate`) returns `SimulatedTransaction { induced_state_diff: { deprecated_declared_classes: [0x0] }, ... }` — reporting the wrong declared class hash.
