### Title
`accept_requests` Guard Blocks `respond*` Completions for In-Flight Requests — (`crates/contract/src/lib.rs`)

### Summary

The `accept_requests` flag is intended to stop the contract from accepting **new** user-facing requests when TEE validation fails. However, the same flag is also checked inside every node-facing `respond*` method, which means that once `accept_requests` is set to `false`, MPC nodes can no longer deliver computed signatures for requests that were already accepted and are actively pending. This is the direct analog of M-07: a guard meant to block entry accidentally also blocks exit.

---

### Finding Description

`check_request_preconditions` — the shared precondition helper called by `sign`, `request_app_private_key`, and `verify_foreign_transaction` — enforces the `accept_requests` flag at line 300: [1](#0-0) 

This is the correct and intended location for the guard: refuse new work when the TEE posture is unhealthy.

The same flag is then re-checked inside all three node-facing response methods:

- `respond` (signature delivery), lines 579–581: [2](#0-1) 

- `respond_ckd` (CKD delivery), lines 662–664: [3](#0-2) 

- `respond_verify_foreign_tx` (foreign-tx delivery), lines 711–713: [4](#0-3) 

These three methods are not "accepting new requests" — they are **completing requests that were already accepted and enqueued** via NEAR's yield-resume mechanism. Their job is to call `pending_requests::resolve_yields_for`, which resumes the parked user promise: [5](#0-4) 

When `accept_requests` flips to `false`, every `respond*` call from every MPC node is rejected with `TeeError::TeeValidationFailed`. The pending yield entries remain in `pending_signature_requests`, `pending_ckd_requests`, and `pending_verify_foreign_tx_requests` with no way to be resolved by the nodes. [6](#0-5) 

---

### Impact Explanation

Every in-flight request submitted before `accept_requests` became `false` is now frozen:

1. MPC nodes have computed (or will compute) a valid threshold signature, but every `respond*` call returns an error — the signature cannot be delivered on-chain.
2. The NEAR yield-resume runtime will eventually fire the timeout callback (`return_signature_and_clean_state_on_success` / `return_ck_and_clean_state_on_success`) with `Err(PromiseError::Failed)` after ~200 blocks, causing the user's call to fail.
3. The user's attached deposit (required by `MINIMUM_SIGN_REQUEST_DEPOSIT`) is consumed or lost depending on the timeout-callback refund path, and the signature is never delivered.
4. If `accept_requests` remains `false` for an extended period (e.g., during a contested TEE upgrade), every request submitted in that window is permanently unresolvable — the MPC network holds a valid signature it cannot post, and the user cannot retrieve their result.

This breaks the request-lifecycle safety invariant: a request that was accepted and paid for must either be answered or cleanly refunded; the current code leaves it in a stuck state until the runtime timeout fires.

---

### Likelihood Explanation

The `accept_requests` flag is a live production field initialized to `true` at contract deployment: [7](#0-6) 

It is toggled by the TEE validation subsystem. Any scenario in which TEE re-validation fails — a node software upgrade, an expired attestation, a verifier outage — can set it to `false`. Because the contract is designed to operate continuously with pending requests in flight, the window during which `accept_requests = false` overlaps with live pending requests is realistic and not contrived.

---

### Recommendation

Remove the `accept_requests` guard from all three `respond*` methods. The guard belongs exclusively in `check_request_preconditions`, which is the correct chokepoint for refusing **new** work. Responses to already-accepted requests must always be allowed to proceed so that the yield-resume lifecycle can complete cleanly:

```diff
 pub fn respond(&mut self, request: SignatureRequest, response: dtos::SignatureResponse) -> Result<(), Error> {
     let signer = Self::assert_caller_is_signer();
     self.assert_caller_is_attested_participant_and_protocol_active();
     if !self.protocol_state.is_running_or_resharing() {
         return Err(InvalidState::ProtocolStateNotRunning.into());
     }
-    if !self.accept_requests {
-        return Err(TeeError::TeeValidationFailed.into());
-    }
     ...
 }
```

Apply the same removal to `respond_ckd` and `respond_verify_foreign_tx`.

---

### Proof of Concept

1. Contract is `Running`, `accept_requests = true`.
2. User calls `sign(...)` — request is enqueued, deposit attached, yield parked.
3. TEE validation fails; `accept_requests` is set to `false`.
4. MPC nodes finish computing the threshold signature and call `respond(request, response)`.
5. `respond` hits the `!self.accept_requests` guard at line 579 and returns `Err(TeeError::TeeValidationFailed)`.
6. The pending yield entry remains in `pending_signature_requests`; the user's promise stays parked.
7. After ~200 blocks the NEAR runtime fires the timeout callback with `Err(PromiseError::Failed)`.
8. The user receives a failure, their deposit is not returned as a valid signature, and the MPC network's computed signature is discarded.

```rust
/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
