### Title
Missing Witness-Count Parity Check Allows Unauthorized UTxO Spending - (File: `eras/byron/ledger/impl/src/Cardano/Chain/UTxO/Validation.hs`)

---

### Summary

In the Byron-era concrete ledger implementation, `updateUTxOTxWitness` uses Haskell's `zip` to pair transaction inputs with their witnesses before calling `validateWitness`. Because `zip` silently truncates to the shorter list, a transaction that supplies fewer witnesses than inputs causes the surplus inputs to be spent with **no witness validation at all**. The formal executable specification explicitly guards against this with a length-equality check that the concrete implementation omits.

---

### Finding Description

`updateUTxOTxWitness` in `eras/byron/ledger/impl/src/Cardano/Chain/UTxO/Validation.hs` performs per-input witness validation as follows:

```haskell
mapM_
  (uncurry $ validateWitness pmi sigData)
  (zip addresses (V.toList witness))
``` [1](#0-0) 

`addresses` is derived from `NE.toList $ txInputs tx` — one entry per transaction input. `witness` is the `TxWitness` (a `Vector TxInWitness`) attached to the `ATxAux`. Haskell's `zip` stops at the end of the **shorter** list without error. If `length witness < length addresses`, the tail inputs are silently dropped from the validation loop and are spent without any cryptographic check.

The Byron executable specification (`eras/byron/ledger/executable-spec/src/Byron/Spec/Ledger/STS/UTXOW.hs`) explicitly requires length equality before zipping:

```haskell
witnessed (Tx tx wits) utxo =
  length wits == length ins && all (isWitness tx utxo) (zip ins wits)
``` [2](#0-1) 

The concrete implementation diverges from the spec by omitting the `length wits == length ins` guard entirely.

`validateTxAux`, called immediately after the witness loop, checks only transaction size, minimum fee, and input/output balance — it does not verify that the witness vector length matches the input count. [3](#0-2) 

The `ATxAux` CBOR decoder likewise imposes no such constraint; it decodes the `Tx` and `TxWitness` fields independently. [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of ADA through an invalid ledger state transition.**

An attacker who constructs a Byron transaction spending N inputs (some belonging to victims) but supplies only M < N witnesses will have only the first M inputs witness-validated. The remaining N−M inputs are consumed and removed from the UTxO without any authorization check, transferring the associated ADA to attacker-controlled outputs. The ledger state transition is accepted as valid, constituting a permanent, irreversible theft.

---

### Likelihood Explanation

The attack requires only the ability to submit a well-formed Byron transaction with a deliberately undersized witness vector. No privileged access, key compromise, or governance majority is needed. The `TxWitness` type is a plain `Vector TxInWitness` with no enforced minimum length relative to inputs, so crafting such a transaction is straightforward for any participant who can broadcast to the network during the Byron era. [5](#0-4) 

---

### Recommendation

Replace the bare `zip` with an explicit length check that mirrors the formal specification:

```haskell
let witList = V.toList witness
    addrList = addresses
unless (length witList == length addrList) $
  throwError $ UTxOValidationTxValidationError TxValidationWitnessCountMismatch
mapM_
  (uncurry $ validateWitness pmi sigData)
  (zip addrList witList)
```

A new `TxValidationWitnessCountMismatch` constructor (or reuse of an existing error) should be added to `TxValidationError`. This restores conformance with the `witnessed` predicate in the executable specification. [6](#0-5) 

---

### Proof of Concept

1. Identify two UTxO entries in the Byron UTxO set: `(txIn_A, txOut_A)` owned by the attacker and `(txIn_B, txOut_B)` owned by a victim.
2. Construct a `Tx` with `txInputs = [txIn_A, txIn_B]` and `txOutputs` routing both values to an attacker address.
3. Produce a `TxWitness` vector of length **1** containing only a valid `VKWitness` for `txIn_A`.
4. Wrap in an `ATxAux` and submit.
5. `updateUTxOTxWitness` computes `addresses = [addr_A, addr_B]` and `V.toList witness = [wit_A]`.
6. `zip addresses (V.toList witness)` yields `[(addr_A, wit_A)]`; only `txIn_A`'s witness is validated.
7. `validateTxAux` passes (size, fee, and balance checks are satisfied).
8. `updateUTxOTx` removes both `txIn_A` and `txIn_B` from the UTxO and adds the attacker's output — victim's ADA is stolen. [7](#0-6)

validateTxAux env utxo (ATxAux (Annotated tx _) _ txBytes) = do
  -- Check that the size of the transaction is less than the maximum
  txSize
    <= maxTxSize
    `orThrowError` TxValidationTxTooLarge txSize maxTxSize

  -- Calculate the minimum fee from the 'TxFeePolicy'
  minFee <-
    if isRedeemUTxO inputUTxO
      then pure $ mkKnownLovelace @0
      else calculateMinimumFee feePolicy

  -- Calculate the balance of the output 'UTxO'
  balanceOut <-
    balance (txOutputUTxO tx)
      `wrapError` TxValidationLovelaceError "Output Balance"

  -- Calculate the balance of the restricted input 'UTxO'
  balanceIn <-
    balance inputUTxO
      `wrapError` TxValidationLovelaceError "Input Balance"

  -- Calculate the 'fee' as the difference of the balances
  fee <-
    subLovelace balanceIn balanceOut
      `wrapError` TxValidationLovelaceError "Fee"

  -- Check that the fee is greater than the minimum
  (minFee <= fee) `orThrowError` TxValidationFeeTooSmall tx minFee fee
  where

        calculateTxSizeLinear txSizeLinear txSize

updateUTxOTxWitness env utxo ta = do
  whenTxValidation $ do
    -- Get the signing addresses for each transaction input from the 'UTxO'
    addresses <-
      mapM (`UTxO.lookupAddress` utxo) (NE.toList $ txInputs tx)
        `wrapError` UTxOValidationUTxOError

    mapM_
      (uncurry $ validateWitness pmi sigData)
      (zip addresses (V.toList witness))
      `wrapError` UTxOValidationTxValidationError

    validateTxAux env utxo ta
