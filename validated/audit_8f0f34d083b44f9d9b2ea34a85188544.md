### Title
`LineaMessenger` dust-bridge guard allows near-zero L1 delivery — (`contracts/bridges/LineaMessenger.sol`)

### Summary
`LineaMessenger.sendETHToL1ViaBridge` guards against `value <= minimumFee` but permits `value = minimumFee + 1 wei`, which causes the Linea bridge to deliver only 1 wei to the L1 vault while consuming nearly all ETH as the bridge fee. The pool's `bridgeAssetsViaNativeBridge()` emits a `BridgedETHToL1ViaNativeBridge` event recording the full `ethBalanceMinusFees` as bridged, creating a misleading accounting record.

### Finding Description

`LineaMessenger.sendETHToL1ViaBridge` reads `minimumFeeInWei()` from the Linea bridge and enforces:

```solidity
if (value <= minimumFee) revert InsufficientAmountForBridge();
ILineaMessageService(l2bridge).sendMessage{ value: value }(target, minimumFee, bytes(""));
``` [1](#0-0) 

The guard is a strict `<=` check, so `value = minimumFee + 1` passes. The Linea `sendMessage` call receives `value` ETH and the `_fee` parameter is `minimumFee`, meaning the net amount delivered to `target` on L1 is `value - minimumFee = 1 wei`.

The pool's `bridgeAssetsViaNativeBridge()` passes the entire `ethBalanceMinusFees` as `value`:

```solidity
uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
    l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees
);
emit BridgedETHToL1ViaNativeBridge(l1VaultETHForL2Chain, ethBalanceMinusFees);
``` [2](#0-1) [3](#0-2) 

If `ethBalanceMinusFees` is just above `minimumFeeInWei()` (e.g., `minimumFee + 1`), the pool event records the full amount as bridged, but the L1 vault receives only 1 wei. The same pattern exists in `RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge()`. [4](#0-3) 

### Impact Explanation
The L1 vault (`l1VaultETHForL2Chain`) receives a negligible ETH amount (as low as 1 wei) despite a successful bridge call. The pool's event logs the full `ethBalanceMinusFees` as bridged, misrepresenting the actual L1 delivery. No ETH is lost (it is consumed as the Linea bridge fee), but the promised return — meaningful ETH delivery to L1 — is not fulfilled. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
This requires `ethBalanceMinusFees` to be in the range `(minimumFeeInWei, minimumFeeInWei + smallDelta]` when the `BRIDGER_ROLE` calls `bridgeAssetsViaNativeBridge()`. This is a realistic operational condition: after most ETH has been bridged via LayerZero, a small dust remainder may accumulate. The `BRIDGER_ROLE` operator calling the native bridge path on that remainder would trigger this silently. No adversarial action is needed — normal operation suffices.

### Recommendation
Add a minimum net-amount check in `LineaMessenger.sendETHToL1ViaBridge`:

```solidity
uint256 minimumFee = ILineaMessageService(l2bridge).minimumFeeInWei();
uint256 netAmount = value - minimumFee; // safe: value > minimumFee already checked
if (value <= minimumFee || netAmount < MIN_BRIDGE_AMOUNT) revert InsufficientAmountForBridge();
```

Alternatively, the pool's `bridgeAssetsViaNativeBridge()` should check that `ethBalanceMinusFees - minimumFee` exceeds a configurable dust threshold before proceeding.

### Proof of Concept

```solidity
// Precondition: pool.ethBalanceMinusFees() == minimumFee + 1
// minimumFee = lineaBridge.minimumFeeInWei() (e.g., 0.001 ETH on Linea)

// BRIDGER_ROLE calls:
pool.bridgeAssetsViaNativeBridge();
// → LineaMessenger.sendETHToL1ViaBridge(l2Bridge, l1Vault, minimumFee + 1)
// → guard: (minimumFee + 1) <= minimumFee → false → passes
// → lineaBridge.sendMessage{value: minimumFee + 1}(l1Vault, minimumFee, "")
// → L1 vault receives: (minimumFee + 1) - minimumFee = 1 wei
// → pool emits: BridgedETHToL1ViaNativeBridge(l1Vault, minimumFee + 1)
// → LineaMessenger emits: ETHSentViaLineaBridge(l2Bridge, l1Vault, minimumFee + 1, minimumFee)

// Assert: mock bridge recorded delivery == 1 wei
// Assert: pool event recorded bridged == minimumFee + 1
// Invariant broken: bridged amount is not economically meaningful relative to fee paid
```
