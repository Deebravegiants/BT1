### Title
Uninitialized `tokenFeeBps` Mapping Allows Zero-Fee Token Deposits After `addSupportedToken()` - (File: contracts/pools/RSETHPool.sol)

### Summary

`RSETHPool.sol` maintains a per-token fee mapping `tokenFeeBps[token]` that is never initialized when a new token is added via `addSupportedToken()`. Because Solidity mappings default to zero, any depositor who calls `deposit(token, amount, referralId)` before the admin separately calls `setTokenFeeBps(token, feeBps)` pays a 0% fee and receives more rsETH than intended, stealing protocol fee revenue.

### Finding Description

`RSETHPool.sol` declares a per-token fee mapping:

```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
``` [1](#0-0) 

When a new token is added via `addSupportedToken()`, only the oracle and bridge are stored. `tokenFeeBps[token]` is never set:

```solidity
function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
    // ...
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    // tokenFeeBps[token] is NOT initialized here
    emit AddSupportedToken(token, oracle, bridge);
}
``` [2](#0-1) 

Setting the per-token fee is a completely separate privileged call:

```solidity
function setTokenFeeBps(address token, uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) onlySupportedToken(token) {
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    tokenFeeBps[token] = _feeBps;
    emit TokenFeeBpsSet(token, _feeBps);
}
``` [3](#0-2) 

The token deposit path reads `tokenFeeBps[token]` directly in the fee calculation:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount, address token) public view onlySupportedToken(token) returns (uint256 rsETHAmount, uint256 fee) {
    uint256 feeBpsForToken = tokenFeeBps[token];
    fee = amount * feeBpsForToken / 10_000;
    uint256 amountAfterFee = amount - fee;
    // ...
    rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
}
``` [4](#0-3) 

The `deposit(address token, ...)` function calls this view function and credits `feeEarnedInToken[token]` with the result. If `tokenFeeBps[token]` is 0, `fee` is 0, `feeEarnedInToken[token]` receives nothing, and the depositor receives the full token value converted to rsETH with no fee deducted. [5](#0-4) 

### Impact Explanation

Every token deposit made in the window between `addSupportedToken()` and `setTokenFeeBps()` pays zero protocol fee. The depositor receives more rsETH than they would at the intended fee rate. The protocol's `feeEarnedInToken[token]` accumulates nothing for those deposits, permanently losing that fee revenue. This is a direct theft of unclaimed protocol yield.

**Impact: High** — Theft of unclaimed yield (protocol fee revenue).

### Likelihood Explanation

The window exists every time a new token is listed. The two calls (`addSupportedToken` and `setTokenFeeBps`) are separate transactions by separate roles (`TIMELOCK_ROLE` vs `DEFAULT_ADMIN_ROLE`), making an atomic initialization impossible without a deployment script that batches them. An attacker monitoring the mempool can front-run or immediately follow `addSupportedToken()` with a large deposit before `setTokenFeeBps()` is confirmed. Even without front-running, any deposit in the gap (which could span multiple blocks or even hours if the admin forgets) is affected.

**Likelihood: Medium** — Requires a timing window, but the window is structurally guaranteed by the two-step design.

### Recommendation

Initialize `tokenFeeBps[token]` inside `addSupportedToken()` by accepting a `_feeBps` parameter:

```diff
function addSupportedToken(
    address token,
    address oracle,
-   address bridge
+   address bridge,
+   uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    // ...
+   if (_feeBps > 10_000) revert InvalidFeeAmount();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
+   tokenFeeBps[token] = _feeBps;
+   emit TokenFeeBpsSet(token, _feeBps);
    emit AddSupportedToken(token, oracle, bridge);
}
```

This eliminates the uninitialized window entirely, mirroring how `feeBps` is set atomically in `initialize()` for ETH deposits.

### Proof of Concept

**Prerequisites:**
- `RSETHPool` is live on Arbitrum with wstETH already supported.
- Admin decides to add a new LST token (e.g., `rETH`) via `addSupportedToken(rETH, rETHOracle, rETHBridge)`.

**Steps:**

1. **Block N**: Admin submits `addSupportedToken(rETH, rETHOracle, rETHBridge)`. `tokenFeeBps[rETH]` is now 0 (default).

2. **Block N (same or next)**: Attacker calls `deposit(rETH, 100e18, "")`.

   Inside `viewSwapRsETHAmountAndFee(100e18, rETH)`:
   ```
   feeBpsForToken = tokenFeeBps[rETH] = 0
   fee = 100e18 * 0 / 10_000 = 0
   amountAfterFee = 100e18
   rsETHAmount = 100e18 * rETHToETHRate / rsETHToETHrate  // full value, no fee
   ```
   `feeEarnedInToken[rETH] += 0` — protocol earns nothing.

3. **Block N+1**: Admin calls `setTokenFeeBps(rETH, 50)` (0.5% fee). Too late — the attacker's deposit already settled at 0%.

**Result**: The attacker received rsETH equivalent to the full 100 rETH value. At 0.5% intended fee on 100 rETH (~$300 at current prices), the protocol lost ~$1.50 per 100 rETH deposited. Scaled to the pool's typical deposit volumes, this is material protocol fee theft.
