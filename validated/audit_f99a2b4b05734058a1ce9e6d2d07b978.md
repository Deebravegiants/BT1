### Title
Attacker Can Front-Run Victim's First Claim by Seeding a Stale Cumulative Baseline — (`contracts/utils/MerkleDistributor/MerkleBlastPointsDistributor.sol`)

---

### Summary

`claim()` has no `msg.sender == account` restriction and accepts any `index` in `[1, currentIndex]`. When the active Merkle tree encodes leaves for multiple historical indices, an attacker can call `claim` on behalf of a victim at a stale lower index, permanently setting the victim's cumulative baseline to a lower historical value. The victim's subsequent claim at `currentIndex` then yields only the delta instead of the full cumulative entitlement.

---

### Finding Description

**Root cause 1 — no caller restriction.**
`claim()` accepts an arbitrary `account` parameter with no check that `msg.sender == account`. [1](#0-0) 

**Root cause 2 — index range allows historical indices.**
The only index guard is `index == 0 || index > currentIndex`. Any value in `[1, currentIndex]` is accepted, so a claim at `oldIndex < currentIndex` is structurally valid. [2](#0-1) 

**Root cause 3 — `isClaimed` is a `>=` comparison, not equality.**
`isClaimed` returns `true` only when `lastClaimedIndex >= index`. For a first-time claimant (`lastClaimedIndex == 0`), any index from 1 to `currentIndex` passes. [3](#0-2) 

**Root cause 4 — baseline is overwritten with the claimed cumulative.**
After a successful claim, `cumulativeBlastPointAmount` and `cumulativeBlastGoldAmount` are set to the values from the leaf. A future claim at a higher index subtracts this baseline. [4](#0-3) 

**Attack sequence (precondition: current root encodes leaves for indices 1 and 2):**

| Step | Actor | Action | State after |
|------|-------|--------|-------------|
| 1 | Attacker | `claim(1, victim, 100, 10, proof1)` | `victim.lastClaimedIndex=1`, baselines=(100,10) |
| 2 | Victim | `claim(2, victim, 200, 20, proof2)` | receives `200-100=100` pts, `20-10=10` gold |
| — | Expected | victim claims at index 2 first | would receive `200-0=200` pts, `20-0=20` gold |

The attacker loses nothing (Blast Points are off-chain; the attacker's own claim state is unaffected). The victim permanently loses half their entitlement.

---

### Impact Explanation

The victim's `Claimed` event records a reduced `claimableBlastPoints`/`claimableBlastGold` amount. The off-chain Blast Points operator distributes points based on these events. The victim receives fewer Blast Points and Blast Gold than their full cumulative entitlement — a direct, partial theft of unclaimed yield. The loss is proportional to the historical baseline injected by the attacker (up to `currentCumulative - historicalCumulative`).

**Impact: High — Theft of unclaimed yield.**

---

### Likelihood Explanation

- No special role or privilege is required; `claim()` is a public, permissionless function.
- The attacker only needs a valid Merkle proof for `(oldIndex, victim, lowCumulative, lowGold)` against the current root. This is realistic whenever the operator publishes a tree that includes leaves for multiple historical indices (a common design to let late claimers catch up).
- The attack can be executed as a simple front-run or even without front-running (the attacker can act any time before the victim's first claim).
- No gas cost beyond a single transaction; no funds at risk for the attacker.

---

### Recommendation

1. **Restrict `claim` to the account itself:** Add `require(msg.sender == account, "caller != account")`. This eliminates the ability for a third party to set another user's baseline.
2. **Alternatively, only allow claiming at `currentIndex`:** Change the index check to `require(index == currentIndex)`, so stale-index leaves are never usable.
3. **Or track claimed state per index:** Use a `mapping(address => mapping(uint256 => bool))` so a claim at index 1 does not affect the baseline for index 2.
