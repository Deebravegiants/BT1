### Title
Permanently Locked Funds When `normalize_amount` Reduces `amount_without_fee` to Zero тАФ (`near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates a NEAR-side transfer with an amount that is smaller than the minimum representable unit on the destination chain (i.e., `amount_without_fee < 10^(origin_decimals - decimals)`), the bridge accepts and locks the tokens at `init_transfer` time, but `sign_transfer` permanently panics with `InvalidAmountToTransfer` because `normalize_amount` floor-divides the amount to zero. No cancel or refund path exists, so the funds are irreversibly locked.

### Finding Description

**Step 1 тАФ `init_transfer` accepts the transfer.**

`init_transfer` in `near/omni-bridge/src/lib.rs` only validates that `fee < amount`: [1](#0-0) 

There is no check that `amount_without_fee` is representable on the destination chain. The tokens are immediately locked/burned and the `TransferMessage` is stored in `pending_transfers`.

**Step 2 тАФ `normalize_amount` uses floor division.** [2](#0-1) 

For a token with `origin_decimals = 24` and `decimals = 18` (diff = 6), any `amount_without_fee < 1_000_000` normalizes to `0`.

**Step 3 тАФ `sign_transfer` permanently panics.** [3](#0-2) 

Every call to `sign_transfer` for this transfer will panic. The MPC signer is never reached, so `sign_transfer_callback` never runs, and the transfer message is never removed from `pending_transfers`.

**Step 4 тАФ No recovery path exists.**

`sign_transfer_callback` only removes the message when fee is zero AND signing succeeds: [4](#0-3) 

`claim_fee` requires a proof of a `FinTransfer` event from the destination chain, which can never exist because the transfer was never signed. `update_transfer_fee` cannot help: the fee must remain strictly less than `amount`, so `amount_without_fee` can never be raised to `>= 10^diff_decimals` through fee manipulation. [5](#0-4) 

There is no cancel or refund function anywhere in the contract.

### Impact Explanation

Any user who sends a token amount below the minimum representable unit on the destination chain (with zero or near-zero fee) will have their tokens permanently locked in the bridge with no recovery path. This satisfies the **Critical тАФ Irreversible fund lock** criterion.

The code comment acknowledges dust loss when `fee = 0`, but only for the sub-unit *remainder* after normalization: [2](#0-1) 

The unaddressed case is when the *entire* `amount_without_fee` normalizes to zero тАФ the whole transfer amount is lost, not just dust.

### Likelihood Explanation

This is reachable by any unprivileged user via the public `ft_transfer_call` тЖТ `ft_on_transfer` тЖТ `init_transfer` path. It is most likely to occur with tokens that have a large decimal difference between chains (e.g., 24-decimal NEAR tokens bridged to 6-decimal EVM tokens, diff = 18, minimum transferable unit = 10^18). A user sending fewer than 10^18 base units with fee = 0 triggers the lock. The bridge UI or integrators may not enforce a minimum amount, making accidental triggering plausible.

### Recommendation

Add a minimum-amount check at `init_transfer` time, before tokens are locked:

```rust
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

Alternatively, add a permissionless `cancel_transfer` function that refunds the sender when a transfer has been pending for longer than a timeout period.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 6` (diff = 18).
2. Call `ft_transfer_call` with `amount = 999_999_999_999_999_999` (< 10^18), `fee = 0`, valid EVM recipient.
3. `init_transfer` succeeds; tokens are locked; `TransferMessage` is stored.
4. Trusted relayer calls `sign_transfer` for this transfer ID.
5. `normalize_amount(999_999_999_999_999_999, {decimals:6, origin_decimals:24})` = `999_999_999_999_999_999 / 10^18` = `0`.
6. `require!(0 > 0, ...)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. Every subsequent `sign_transfer` call panics identically. Tokens are permanently locked.

    /// Uses floor division тАФ any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
