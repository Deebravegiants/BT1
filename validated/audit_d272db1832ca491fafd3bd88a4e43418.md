### Title
Missing CKD Response Validation for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function validates the CKD response only for the `AppPublicKeyPV` variant. For the `AppPublicKey` variant the match arm is empty — no cryptographic output check is performed. A single malicious attested participant can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will accept and deliver it to the requesting user without any verification.

---

### Finding Description

In `crates/contract/src/lib.rs`, `respond_ckd` handles two variants of `CKDAppPublicKey`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

Compare this with `respond` (ECDSA) and `respond_verify_foreign_tx`, both of which cryptographically verify the response before resolving pending yields:

- `respond` calls `near_mpc_signature_verifier::verify_ecdsa_signature(...)` and returns `Err(RespondError::InvalidSignature)` on failure. [2](#0-1) 
- `respond_verify_foreign_tx` similarly verifies the signature against the root public key before resolving. [3](#0-2) 

For `respond_ckd` with `AppPublicKey`, no such check exists. After the empty match arm, the function unconditionally calls `pending_requests::resolve_yields_for`, delivering whatever `CKDResponse` the caller supplied to every queued yield under that request key. [4](#0-3) 

The only gate is `assert_caller_is_attested_participant_and_protocol_active()` — a single-participant check, not a threshold check. [5](#0-4) 

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can forge the confidential key derivation output for any pending `AppPublicKey` CKD request. The user receives a fake derived key that was never produced by the threshold MPC protocol. This is unauthorized confidential key derivation output without the required participant authorization — matching the critical allowed impact scope. The attacker can supply any `CKDResponse` bytes they choose; the contract performs zero cryptographic binding between the response and the request or the network's BLS12-381 root key.

---

### Likelihood Explanation

Any single attested MPC participant can exploit this. The attacker must:
1. Hold a valid TEE attestation (be a registered participant).
2. Observe a pending `AppPublicKey` CKD request on-chain.
3. Race-call `respond_ckd` with a forged `CKDResponse` before the honest threshold response arrives.

Step 3 is straightforward: the attacker monitors `pending_ckd_requests` and submits immediately upon seeing a new entry. Because the contract resolves and drains the entire yield queue on the first accepted `respond_ckd` call, the honest response arriving later finds no pending yield to satisfy.

---

### Recommendation

Add output validation for the `AppPublicKey` variant in `respond_ckd`, equivalent to the `AppPublicKeyPV` path. If the `AppPublicKey` variant cannot be cryptographically verified against the network's root public key (e.g., because it carries no proof of validity), the contract should either:
- Require a verifiable proof before accepting any `CKDResponse`, or
- Remove the `AppPublicKey` variant from the callable `respond_ckd` path and require callers to use `AppPublicKeyPV`.

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(some_pk)`.
2. The request is queued in `pending_ckd_requests`.
3. Malicious attested participant constructs an arbitrary `CKDResponse` (e.g., a derived key they control).
4. Malicious participant calls `respond_ckd(request, forged_response)`.
5. Contract enters the `AppPublicKey(_) => {}` branch — no check performed.
6. `resolve_yields_for` drains the entire pending queue, delivering `forged_response` to the user.
7. User receives a fake derived key; the honest threshold response (if it arrives later) finds no pending yield and is silently discarded. [6](#0-5)
