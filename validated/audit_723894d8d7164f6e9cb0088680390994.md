Audit Report

## Title
Cross-Token Withdrawal Drains Bridger Collateral and Permanently Freezes Depositor Assets - (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary

`_withdraw` enforces only that the requested asset is in `allowedTokens`, with no record of which token each wrsETH unit was minted against. When two tokens are simultaneously allowed, any wrsETH holder can burn wrsETH minted from tokenA to withdraw tokenB, draining the bridger's collateral and leaving tokenA permanently irrecoverable because `totalSupply()` reaches zero with non-zero tokenA still held by the contract.

## Finding Description

`_withdraw` at line 121 performs exactly one asset-level check:

```solidity
if (!allowedTokens[_asset]) revert TokenNotAllowed();
```

`_deposit` at lines 134–141 mints wrsETH 1:1 against any allowed token without recording which token backed each unit. wrsETH is a single fungible token whose `totalSupply` aggregates deposits of all allowed tokens. `depositBridgerAssets` at lines 162–170 allows the bridger to post collateral for a specific asset; its cap is computed per-asset at line 109:

```solidity
return wrsETHSupply - balanceOfAssetInWrapper;
```

This means the bridger can legitimately deposit N tokenB as long as `totalSupply() - tokenB.balanceOf(wrapper) >= N`, regardless of how much tokenA is already held. Once a second token is added via `addAllowedToken` (TIMELOCK_ROLE) or `reinitialize` (DEFAULT_ADMIN_ROLE), any wrsETH holder can call `withdraw(tokenB, amount)` to redeem tokenB using wrsETH that was originally minted from tokenA. There is no per-user or per-token share accounting anywhere in the contract, and no admin rescue path exists.

## Impact Explanation

Two Critical impacts are simultaneously triggered:

1. **Direct theft of user funds**: The bridger's tokenB collateral is transferred to the attacker via a normal `withdraw` call.
2. **Permanent freezing of funds**: After the attacker burns all wrsETH, `totalSupply() == 0`. No wrsETH remains to redeem the tokenA still held by the contract. There is no emergency withdrawal, sweep, or admin recovery function in the contract.

Both match allowed Critical impacts: *Direct theft of any user funds* and *Permanent freezing of funds*.

## Likelihood Explanation

The precondition — two tokens in `allowedTokens` — is the explicit purpose of `addAllowedToken` and `reinitialize`. The bridger depositing collateral for a second token is the documented normal operational flow (`depositBridgerAssets` is described as the mechanism to collateralize already-minted wrsETH). No privileged role, leaked key, oracle failure, or victim mistake is required. Any wrsETH holder executes step 3 unilaterally via the public `withdraw` function.

## Recommendation

Introduce per-token share accounting. The minimal fix is a `mapping(address token => uint256 shares)` tracking how much of each token backs the current `totalSupply`, updated on every `_deposit`, `_withdraw`, `mint`, and `depositBridgerAssets` call. `_withdraw` must then verify that `tokenShares[_asset] >= _amount` before burning. Alternatively, maintain a `mapping(address user => mapping(address token => uint256 balance))` and enforce that withdrawals only redeem the token originally deposited.

## Proof of Concept

```
Precondition: tokenA and tokenB both in allowedTokens; totalSupply = 0.

Step 1 — user calls deposit(tokenA, 100):
  tokenA.balanceOf(wrapper) = 100
  totalSupply()             = 100

Step 2 — bridger calls depositBridgerAssets(tokenB, 100):
  maxAmountToDepositBridgerAsset(tokenB)
    = totalSupply() - tokenB.balanceOf(wrapper)
    = 100 - 0 = 100  ✓ (no revert)
  tokenB.balanceOf(wrapper) = 100
  totalSupply()             = 100  (unchanged, no mint on bridger deposit)

Step 3 — user calls withdraw(tokenB, 100):
  allowedTokens[tokenB] == true  ✓ (only guard, passes)
  burns 100 wrsETH from user
  transfers 100 tokenB to user

Final state:
  tokenA.balanceOf(wrapper) = 100  ← permanently frozen, zero wrsETH to redeem it
  tokenB.balanceOf(wrapper) = 0    ← bridger collateral stolen
  totalSupply()             = 0    ← no wrsETH left to ever redeem tokenA
```

**Foundry invariant test plan**: Deploy `RsETHTokenWrapper` with two mock ERC20 tokens. Add both via `addAllowedToken`. Fuzz `deposit(tokenA, x)`, `depositBridgerAssets(tokenB, y)`, `withdraw(tokenB, z)` sequences. Assert invariant: `sum over all allowedTokens t of t.balanceOf(wrapper) <= totalSupply()`.     function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
        if (!allowedTokens[_asset]) return 0;

        // get totalSupply of wrsETH minted
        uint256 wrsETHSupply = totalSupply();
        // balance of _asset with the contract
        uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));

        if (balanceOfAssetInWrapper > wrsETHSupply) return 0;

        return wrsETHSupply - balanceOfAssetInWrapper;
    }
