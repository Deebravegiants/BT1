### Title
Queued Future-Round Proposals Bypass `block_timestamp_window_seconds` Validation, Allowing a Byzantine Proposer to Commit a Block with a Future Timestamp - (File: crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs)

### Summary

When a `ProposalInit` arrives for a future round, it is stored in `queued_proposals` without any timestamp validation. When the round eventually advances and the proposal is dequeued, `is_proposal_init_valid` is called with `clock.unix_now()` at dequeue time. Because wall-clock time has advanced by the inter-round delay, a timestamp that would have been rejected at receipt time (it exceeded `now_at_receipt + block_timestamp_window_seconds`) passes the check at dequeue time. The production default for `block_timestamp_window_seconds` is **1 second**, so any inter-round delay of even a few seconds widens the effective window by that many seconds.

### Finding Description

**Queue path — no validation:** [1](#0-0) 

A proposal for a future round is inserted into `queued_proposals` with its raw `init` (including `timestamp`) and the original `timeout`. No call to `is_proposal_init_valid` is made here.

**Dequeue path — validation uses current wall-clock time:** [2](#0-1) 

When `set_height_and_round` advances the round, the queued proposal is extracted and passed to `validate_current_round_proposal`, which spawns `validate_proposal`. Inside `validate_proposal`, `is_proposal_init_valid` is called: [3](#0-2) 

The timestamp check inside `is_proposal_init_valid` reads `clock.unix_now()` at the moment of dequeue: [4](#0-3) 

The check is:
```
init_proposed.timestamp > now_at_dequeue + block_timestamp_window_seconds
```

If the proposal was queued for `T` seconds, `now_at_dequeue = now_at_receipt + T`, so the effective allowed future window at receipt time becomes `T + block_timestamp_window_seconds` instead of just `block_timestamp_window_seconds`.

**Production window is 1 second:** [5](#0-4) 

### Impact Explanation

A Byzantine proposer selected for round `R` can craft a `ProposalInit` with `timestamp = now_at_send + 1s + ε` (just beyond the 1-second window). Validators receive it during round `R-1`, queue it without validation, and then accept it when round `R` begins after the inter-round timeout elapses. The committed block carries a future timestamp.

The future timestamp is fed directly into:

1. **L1 gas price lookup** — `get_l1_prices_in_fri_and_wei` is called with `init_proposed.timestamp` as the lookup key. A future timestamp causes the oracle to return prices for a time that has not yet occurred, producing incorrect `l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, and `l1_data_gas_price_wei` values that are embedded in the committed block header. [6](#0-5) 

2. **Block header commitment** — the timestamp is part of `PartialBlockHashComponents` and therefore part of the `ProposalCommitment` that consensus signs over. A wrong timestamp produces a wrong block hash that is permanently committed on-chain.

3. **Fee and gas accounting** — incorrect L1 gas prices propagate into transaction fee calculations for every transaction in the block, satisfying the "Incorrect fee, gas … with economic impact" critical impact criterion.

### Likelihood Explanation

Requires a Byzantine proposer — a validator that has been legitimately selected as proposer for a specific round. This is the standard BFT threat model (up to `f < n/3` Byzantine validators). The attack is passive from the network's perspective: the malicious node simply sends a well-formed proposal with a slightly inflated timestamp. No special network position or privileged access beyond being the elected proposer is needed. The inter-round timeout (typically several seconds) reliably provides the necessary time gap.

### Recommendation

Perform the timestamp check at **queue time**, before inserting into `queued_proposals`. Concretely, in the `std::cmp::Ordering::Greater` branch of `validate_proposal` in `sequencer_consensus_context.rs`, call `is_proposal_init_valid` (or at minimum the timestamp sub-check) before queuing. Alternatively, record `received_at = clock.unix_now()` alongside the queued proposal and pass it as the reference time when the proposal is later dequeued, so the effective window is anchored to receipt time rather than dequeue time.

### Proof of Concept

1. Byzantine proposer for round `R` sends `ProposalInit { timestamp: now_at_send + 2, ... }` (2 seconds in the future; 1 second beyond the 1-second window).
2. Validators are in round `R-1`. The `Greater` branch fires:
   ```rust
   self.queued_proposals.insert(init.round, ((init, timeout, content_receiver), fin_sender));
   ```
   No timestamp check. The proposal is stored.
3. Round `R-1` times out after ≥ 1 second. `set_height_and_round(height, R)` is called.
4. The proposal is dequeued. `now_at_dequeue ≥ now_at_send + 1`.
5. `is_proposal_init_valid` checks: `(now_at_send + 2) > (now_at_send + 1) + 1` → `now_at_send + 2 > now_at_send + 2` → **false** → check passes.
6. `initiate_validation` is called with `block_info` containing the future timestamp. The batcher executes the block with that timestamp.
7. `finish_proposal` returns a `ProposalCommitment` derived from a `PartialBlockHash` that encodes the future timestamp. All validators agree on this commitment and the block is finalized with a future timestamp and the L1 gas prices that correspond to that future time. [1](#0-0) [7](#0-6) [4](#0-3)
