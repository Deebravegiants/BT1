### Title
P2P Sync Client Accepts Peer-Supplied `transaction_hash` Without Validation, Corrupting Stored Transaction Hash Index — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`TransactionStreamFactory::parse_data_for_block` blindly trusts the `transaction_hash` field of every `FullTransaction` received from a p2p peer. No check is performed that the field matches the cryptographic hash of the embedded `Transaction`. The wrong hash is then written into `block_body.transaction_hashes` and flushed to the node's persistent storage, corrupting the `transaction_hash_to_idx` lookup table and `TransactionMetadata` for every affected transaction.

---

### Finding Description

In `parse_data_for_block`, after destructuring the peer-supplied `FullTransaction`, the code pushes `transaction_hash` directly into `block_body.transaction_hashes` with an explicit TODO acknowledging the missing check: [1](#0-0) 

`write_to_storage` then calls `append_body` unconditionally: [2](#0-1) 

Inside `write_transactions`, the attacker-controlled hash is inserted into two persistent structures:

- `transaction_hash_to_idx_table` (the primary lookup index for `starknet_getTransactionByHash`): [3](#0-2) 

- `TransactionMetadata` (stores `tx_hash` alongside the file offsets): [4](#0-3) 

There is no post-write integrity check anywhere in the pipeline.

---

### Impact Explanation

A node running in p2p sync mode (configured with `p2p_sync_client_config`) will store the attacker-chosen hash as the canonical hash for each injected transaction. Consequences:

1. **RPC lookups broken**: `starknet_getTransactionByHash(correct_hash)` returns `TRANSACTION_HASH_NOT_FOUND`; `starknet_getTransactionByHash(attacker_hash)` returns the real transaction body — the node serves authoritative-looking wrong data.
2. **`get_block_transaction_hashes` returns wrong values**: Any downstream consumer (e.g., another p2p sync server re-serving this block) propagates the corrupted hashes further.
3. **Inconsistency with header commitment**: The `transaction_commitment` stored in the block header (received via the separate header sync) is computed over the real hashes; the stored body hashes no longer match it, making the node's storage internally inconsistent.

The impact is **High** under the allowed scope: *"RPC execution … returns an authoritative-looking wrong value"* and *"Transaction conversion or signature/hash logic binds the wrong … hash … or executable payload."*

The claim of Critical impact on the sequencer's own consensus is **not supported**: the sequencer node's block production uses the `convert_sync_block_to_block_data` / internal-blocks-receiver path, not the p2p peer path. [5](#0-4) 

---

### Likelihood Explanation

Any node on the p2p network that responds to a `TransactionQuery` can inject this. No special privilege is required. The attacker only needs to be a reachable peer; the victim node will query it as part of normal sync. The TODO comment confirms the developers are aware the source is untrusted and the check is simply not yet implemented.

---

### Recommendation

After destructuring the `FullTransaction`, recompute the hash from the `transaction` body using `get_transaction_hash` (already available in `crates/starknet_api/src/transaction_hash.rs`) and compare it against the peer-supplied `transaction_hash`. On mismatch, return `ParseDataError::BadPeer` (consistent with how other invalid peer data is handled in the same file) so the peer is reported and the session is aborted.

---

### Proof of Concept

1. Stand up a p2p sync client against a mock peer (using the existing `mock_register_sqmr_protocol_client` / `run_test` harness in `transaction_test.rs`).
2. Send a valid `SignedBlockHeader` with `n_transactions = 1`.
3. Send a `FullTransaction` where `transaction` is a valid `InvokeTransactionV1` but `transaction_hash` is `TransactionHash(Felt::from(0xdeadbeef_u64))`.
4. After the body marker advances, call `get_block_transaction_hashes(BlockNumber(0))`.
5. Assert the stored hash equals `0xdeadbeef` (not the real hash of the invoke transaction).
6. Assert `get_transaction_by_hash(real_hash)` returns `None` while `get_transaction_by_hash(0xdeadbeef)` returns the transaction body.
