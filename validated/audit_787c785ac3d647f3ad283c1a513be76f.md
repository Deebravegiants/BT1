Audit Report

## Title
Permissionless `addLiquidity` Owner Injection Enables Griefing DoS on `removeLiquidity` Full-Exit — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address with no `msg.sender == owner` guard, allowing any attacker to inject shares into an arbitrary victim's position. Because `LiquidityLib.removeLiquidity` reverts with `MinimalLiquidity` whenever the post-burn share count is non-zero but below `MINIMAL_MINTABLE_LIQUIDITY`, an attacker can permanently block a victim's full exit by frontrunning every withdrawal with a 1-share donation, leaving a dust remainder that trips the guard. The victim's principal remains locked in the pool indefinitely.

## Finding Description

**Root cause — no ownership check in `addLiquidity`:**

`MetricOmmPool.addLiquidity` passes the caller-supplied `owner` directly into `LiquidityLib.addLiquidity` without verifying `msg.sender == owner`: [1](#0-0) 

By contrast, `removeLiquidity` does enforce `msg.sender == owner`: [2](#0-1) 

**Guard in `addLiquidity` — passes for dust injection:**

Inside `LiquidityLib.addLiquidity`, the only guard on the resulting share count is: [3](#0-2) 

If the victim already holds `N >= MINIMAL_MINTABLE_LIQUIDITY` shares, adding 1 share satisfies `N + 1 >= MINIMAL_MINTABLE_LIQUIDITY`, so the injection succeeds.

**Guard in `removeLiquidity` — reverts on dust remainder:** [4](#0-3) 

**Attack flow:**

1. Victim holds `N` shares (`N >= MINIMAL_MINTABLE_LIQUIDITY`) and submits `removeLiquidity(shares=[N])` to fully exit.
2. Attacker (a contract implementing `IMetricOmmModifyLiquidityCallback`) frontruns with `addLiquidity(owner=victim, shares=[1])`.
3. Victim's share count becomes `N + 1`.
4. Victim's `removeLiquidity(N)` executes: `newUserShares = 1`, satisfying `1 > 0 && 1 < MINIMAL_MINTABLE_LIQUIDITY` → **reverts**.
5. Attacker repeats on every retry.

**Attacker cost per frontrun:**

The token cost for 1 share is computed as: [5](#0-4) 

`Math.ceilDiv(token0BalanceScaled * 1, binTotalSharesVal)` rounds up to at minimum 1 scaled unit. After descaling via `_deltasScaledToExternal` with `Math.Rounding.Ceil`, this is at most 1 native token unit (1 wei for an 18-decimal token). The donated tokens enter the victim's position and are not recoverable by the attacker, making the attack economically viable indefinitely.

## Impact Explanation

A victim LP cannot fully exit any bin position as long as the attacker keeps frontrunning. The victim's principal (token0 and token1 locked in the bin) is inaccessible for full withdrawal. Partial withdrawals that leave `>= MINIMAL_MINTABLE_LIQUIDITY` shares succeed, but the victim can never reach zero and reclaim the final tranche. This constitutes broken core pool withdraw functionality with direct, persistent loss of access to user principal — matching the allowed impact gate for "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows."

## Likelihood Explanation

- `addLiquidity` is a public, permissionless function with no `msg.sender == owner` guard — confirmed in production code.
- The attacker needs only a contract implementing `IMetricOmmModifyLiquidityCallback` (trivial to deploy) and a negligible token balance (~1 wei per frontrun).
- No privileged role, special setup, or non-standard ERC20 behavior is required.
- The attack is repeatable at negligible cost and cannot be mitigated by the victim alone without a protocol-level fix.

## Recommendation

**Option A (preferred):** In `removeLiquidity`, add a "remove all" path that caps `sharesToRemove` at `userShares` when the caller requests more than they own, or unconditionally sets `sharesToRemove = userShares` when the caller passes `type(uint256).max` as a sentinel. This lets a victim always fully exit regardless of injected dust:

```solidity
// LiquidityLib.sol removeLiquidity — inside the per-bin loop
uint256 sharesToRemove = deltas.shares[i];
if (sharesToRemove >= userShares) sharesToRemove = userShares; // full-exit sentinel
uint256 newUserShares = userShares - sharesToRemove;
// newUserShares == 0 → passes the guard unconditionally
```

**Option B:** Restrict `addLiquidity` so that `msg.sender == owner` (or require explicit approval from `owner`), preventing third-party share injection entirely.

## Proof of Concept

```
Constants: MINIMAL_MINTABLE_LIQUIDITY = 1_000

State before attack:
  victim.shares[bin=4] = 10_000

Step 1 — victim submits:
  removeLiquidity(owner=victim, salt=S, binIdxs=[4], shares=[10_000])

Step 2 — attacker (contract) frontruns (~1 wei of token0):
  addLiquidity(owner=victim, salt=S, binIdxs=[4], shares=[1])
  → victim.shares[bin=4] = 10_001

Step 3 — victim's tx executes:
  newUserShares = 10_001 - 10_000 = 1
  1 > 0 && 1 < 1_000  →  revert MinimalLiquidity(1, 1_000)

Step 4 — victim retries with shares=[10_001]:
  attacker frontruns again with shares=[1]
  → victim.shares = 10_002, victim's tx reverts again.

Victim's principal is permanently locked.
```

Foundry test plan: deploy a mock callback contract, seed a victim position with `N = MINIMAL_MINTABLE_LIQUIDITY` shares, call `addLiquidity(owner=victim, shares=[1])` from the attacker contract, then assert that `removeLiquidity(owner=victim, shares=[N])` reverts with `MinimalLiquidity(1, MINIMAL_MINTABLE_LIQUIDITY)`.
