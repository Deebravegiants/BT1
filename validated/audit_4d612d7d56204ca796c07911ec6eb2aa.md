### Title
FunctionCall Access Key Allowance Race Condition Enables N+M Over-Spend - (File: `runtime/runtime/src/verifier.rs`)

### Summary
When an account owner (Alice) attempts to change a function-call access key's allowance from N to M by submitting a `DeleteKey`+`AddKey` transaction, the key holder (Bob) can front-run that transaction and spend N before the change is applied, then spend M after the new key is created — for a total of N+M instead of the intended M.

### Finding Description

NEAR's `FunctionCallPermission` carries an `allowance: Option<Balance>` field that caps how much NEAR a delegated key may spend on gas and transaction fees. [1](#0-0) 

The protocol's own documentation acknowledges that to change or increase the allowance, the old key must be deleted and a new key created: [2](#0-1) 

The standard pattern is therefore a single transaction containing `[DeleteKeyAction, AddKeyAction]` with the same public key and the new allowance M. The runtime enforces the allowance at transaction-verification time via `check_and_compute_new_allowance`: [3](#0-2) 

This check reads the allowance from the **current trie state** at the moment the transaction is processed. If Bob's spend transaction is ordered before Alice's delete+add transaction in the same or an earlier chunk, the runtime sees allowance=N, accepts Bob's transaction, and decrements the allowance. Alice's transaction then executes: the old key (now with allowance N−spent) is deleted, and a new key with the **same public key** and allowance M is written to state. Because Bob still holds the private key, he can immediately submit further transactions against the new key and spend up to M more.

The state write path that persists the new allowance is: [4](#0-3) 

And the `VerificationResult::apply` that stamps the decremented allowance into the in-memory key before the write: [5](#0-4) 

There is no atomic "compare-and-set" or "require current allowance == 0" guard anywhere in the allowance-change path.

### Impact Explanation

Bob's effective spending limit is N+M instead of the M Alice intended. The corrupted trie entry is the `FunctionCallPermission.allowance` field of the access key stored under `TrieKey::AccessKey { account_id, key_handle }`. The financial loss is bounded by N (the original allowance), which can be set to any value up to the account balance. Because the allowance covers gas and fees only (not direct transfers), the maximum loss per key is the original allowance amount.

### Likelihood Explanation

Bob must observe Alice's pending transaction before it is included in a chunk and submit his own transaction in time to be ordered first. NEAR has no gas-price auction, so ordering is not deterministic from Bob's perspective, but Bob can:
- Run his own node and submit directly to the chunk producer's mempool
- Broadcast to multiple nodes simultaneously to maximize inclusion probability

The attack is realistic whenever Bob is an active participant who monitors the mempool, which is a reasonable assumption for any party that has been granted a function-call key.

### Recommendation

Apply the same mitigation recommended for ERC-20: require the current allowance to be zero before setting a non-zero value, or introduce `increaseAllowance`/`decreaseAllowance` semantics. Concretely, `check_and_compute_new_allowance` (or the `action_add_key` path in `access_keys.rs`) should reject an `AddKeyAction` that sets a non-zero allowance on a key whose predecessor had a non-zero allowance unless the predecessor's allowance was first zeroed in the same receipt. Alternatively, expose a dedicated `UpdateAllowance` action that atomically replaces the allowance only if the current value matches an expected value supplied by the caller. [6](#0-5) 

### Proof of Concept

1. Alice creates a function-call access key for Bob's public key `PK` with `allowance = N`.
2. Alice decides to reduce the allowance to M < N and submits a transaction (signed with her full-access key):
   `[DeleteKeyAction { public_key: PK }, AddKeyAction { public_key: PK, access_key: FunctionCall { allowance: M, … } }]`
3. Bob observes Alice's pending transaction in the mempool before it is included in a chunk.
4. Bob immediately submits a transaction signed with `PK` that spends up to N (e.g., a function call with maximum prepaid gas).
5. The chunk producer includes Bob's transaction before Alice's. `check_and_compute_new_allowance` sees allowance=N, accepts the transaction, and decrements the allowance. `set_tx_state_changes` writes the updated key to the trie.
6. Alice's transaction executes next: `action_delete_key` removes the key; `action_add_key` writes a fresh key with `allowance = M` and the same public key `PK`.
7. Bob submits a second transaction signed with `PK`. The runtime now sees allowance=M and accepts it.
8. **Net result**: Bob spent up to N+M of Alice's NEAR on gas, while Alice intended a cap of M.

fn check_and_compute_new_allowance(
    access_key: &AccessKey,
    account_id: &AccountId,
    public_key: &PublicKey,
    total_cost: Balance,
) -> Result<Option<Balance>, InvalidTxError> {
    let Some(fc) = access_key.permission.function_call_permission() else {
        return Ok(None);
    };
    let Some(allowance) = fc.allowance else {
        return Ok(None);
    };
    let new_allowance = allowance.checked_sub(total_cost).ok_or_else(|| {
        InvalidTxError::InvalidAccessKeyError(InvalidAccessKeyError::NotEnoughAllowance {
            account_id: account_id.clone(),
            public_key: public_key.clone().into(),
            allowance,
            cost: total_cost,
        })
    })?;
    Ok(Some(new_allowance))
}

pub(crate) fn action_add_key(
    apply_state: &ApplyState,
    state_update: &mut TrieUpdate,
    account: &mut Account,
    result: &mut ActionResult,
    account_id: &AccountId,
    add_key: &AddKeyAction,
) -> Result<(), StorageError> {
    if get_access_key(state_update, account_id, &add_key.public_key)?.is_some() {
        result.result = Err(ActionErrorKind::AddKeyAlreadyExists {
            account_id: account_id.to_owned(),
            public_key: add_key.public_key.clone().into(),
        }
        .into());
        return Ok(());
    }

    let fee_config = &apply_state.config.fees;

    if let Some(gas_key_info) = add_key.access_key.gas_key_info() {
        add_gas_key(
            fee_config,
            state_update,
            account,
            account_id,
            &add_key.public_key,
            &add_key.access_key,
            gas_key_info,
            apply_state.block_height,
        )?;
    } else {
        add_regular_key(
            fee_config,
            state_update,
            account,
            account_id,
            &add_key.public_key,
            &add_key.access_key,
            apply_state.block_height,
        )?;
    }

    Ok(())
}
