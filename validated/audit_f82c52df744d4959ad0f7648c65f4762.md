Audit Report

## Title
Rebasing stETH Token Amounts Stored as Fixed Values in Withdrawal Queue Lead to Permanent Fund Freeze - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`LRTWithdrawalManager` transfers stETH from `LRTUnstakingVault` into itself during `unlockQueue()` and stores fixed token amounts in `request.expectedAssetAmount`. Because stETH is a rebasing token, a negative rebase between `unlockQueue()` and `completeWithdrawal()` reduces the contract's actual stETH balance below the sum of all committed amounts. Early claimers drain the available balance; late claimers' `completeWithdrawal()` calls revert permanently, while their rsETH has already been burned and is unrecoverable.

## Finding Description

**Stage 1 – `initiateWithdrawal()`**: A fixed `expectedAssetAmount` is computed from oracle prices and stored in `withdrawalRequests`, and `assetsCommitted[asset]` is incremented by the same fixed token amount.

```solidity
// L168-173
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

**Stage 2 – `unlockQueue()`**: The operator calls this, which reads the vault's stETH balance via `unstakingVault.balanceOf(asset)` (which calls `IERC20(asset).balanceOf(address(this))` — a rebasing amount), recalculates each request's payout, overwrites `request.expectedAssetAmount = payoutAmount`, burns rsETH, and calls `unstakingVault.redeem(asset, assetAmountUnlocked)` to transfer the exact stETH amount into `LRTWithdrawalManager`.

```solidity
// L301-307
(rsETHBurned, assetAmountUnlocked) = _unlockWithdrawalRequests(...);
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);
```

`LRTUnstakingVault.redeem()` performs a plain `IERC20(asset).safeTransfer(msg.sender, amount)`, transferring a fixed token amount of stETH to `LRTWithdrawalManager`.

**Stage 3 – `completeWithdrawal()`**: The contract transfers exactly `request.expectedAssetAmount` stETH to the user via `IERC20(asset).safeTransfer`.

```solidity
// L734
_transferAsset(asset, user, request.expectedAssetAmount);
```

**Root cause**: After Stage 2, `LRTWithdrawalManager` holds a fixed stETH token balance equal to the sum of all unlocked `expectedAssetAmount` values. stETH is a rebasing token — its `balanceOf()` changes autonomously without any transfer. A negative rebase (e.g., validator slashing) between Stage 2 and Stage 3 reduces the contract's actual stETH balance below the sum of committed amounts. The contract has no mechanism to detect or handle this discrepancy. No share-based accounting is used anywhere in the withdrawal lifecycle.

The `_transferAsset` function uses a plain `IERC20.safeTransfer` with no balance check or share conversion:

```solidity
// L876-882
function _transferAsset(address asset, address to, uint256 amount) internal {
    if (asset == LRTConstants.ETH_TOKEN) { ... }
    else { IERC20(asset).safeTransfer(to, amount); }
}
```

**Secondary impact (Stage 1 → Stage 2 window)**: `assetsCommitted[asset]` tracks fixed token amounts. A negative rebase between `initiateWithdrawal()` and `unlockQueue()` reduces `getTotalAssetDeposits(asset)` while `assetsCommitted[asset]` remains unchanged, causing `getAvailableAssetAmount()` to return 0 and blocking all new withdrawal initiations:

```solidity
// L599-603
function getAvailableAssetAmount(address asset) public view override returns (uint256 availableAssetAmount) {
    uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
    availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
}
```

**Why existing checks fail**: There is no balance check before `_transferAsset` for non-ETH assets (the Aave balance check at L720-731 is ETH-only). The `_processWithdrawalCompletion` function deletes the request and decrements `unlockedWithdrawalsCount` before calling `_transferAsset`, but since the entire transaction reverts on transfer failure, the request remains in the queue — however, the user's rsETH is already burned at `unlockQueue()` time and cannot be recovered regardless of whether the transfer succeeds or fails.

## Impact Explanation

**Permanent freezing of funds (Critical)**: After `unlockQueue()` burns rsETH and transfers stETH to `LRTWithdrawalManager`, a negative rebase reduces the contract's stETH balance. Users who call `completeWithdrawal()` first drain the available balance. Subsequent users' calls revert because `safeTransfer` fails when the contract lacks sufficient stETH. Their rsETH has already been burned at `unlockQueue()` time and is unrecoverable. Their withdrawal requests remain in the queue but can never be fulfilled — there is no mechanism to add replacement stETH (rsETH is already burned), no cancel-and-refund path, and no proportional redistribution. The affected users' funds are permanently frozen.

**Protocol insolvency**: The burned rsETH supply no longer corresponds to recoverable assets, leaving rsETH undercollateralized.

## Likelihood Explanation

stETH is an explicitly supported and actively used asset: `initialize2()` seeds `unlockedWithdrawalsCount` for stETH, `LRTConverter` contains dedicated stETH unstaking logic via `UnstakeStETH`, and `LRTConstants.ST_ETH_TOKEN` is a named constant. The withdrawal delay is 8 days, creating a substantial window during which stETH sits in `LRTUnstakingVault` (Stage 1→2) and then in `LRTWithdrawalManager` (Stage 2→3) with no share-based protection. Ethereum validator slashing events are infrequent but historically documented and realistic. Any negative rebase during either window triggers the vulnerability. No attacker action is required — the trigger is an external protocol event (Lido rebase) that any user can observe and exploit by front-running `completeWithdrawal()` calls.

## Recommendation

Store and transfer stETH in **shares** rather than token amounts throughout the withdrawal lifecycle:

1. In `initiateWithdrawal()`, convert `expectedAssetAmount` to stETH shares using `IStETH(stETH).getSharesByPooledEth(expectedAssetAmount)` and store the shares value.
2. In `_unlockWithdrawalRequests()`, compute `payoutAmount` in shares and store shares in `request.expectedAssetAmount`.
3. In `_processWithdrawalCompletion()`, use `IStETH(stETH).transferShares(user, sharesAmount)` instead of `IERC20.safeTransfer`.
4. Update `assetsCommitted[stETH]` to track shares, and convert to token amounts only when computing `getAvailableAssetAmount()` using `IStETH.getPooledEthByShares()`.

This ensures all accounting is rebase-invariant and each user receives their proportional share of the stETH balance regardless of rebasing events.

## Proof of Concept

1. Alice and Bob each call `initiateWithdrawal(stETH, ...)`. Their `expectedAssetAmount` values (90 stETH and 10 stETH) are stored as fixed amounts; `assetsCommitted[stETH] = 100e18`.
2. The operator calls `unlockQueue(stETH, ...)` after the 8-day delay. The vault's 100 stETH is transferred to `LRTWithdrawalManager`. Both requests are unlocked with `expectedAssetAmount` = 90 and 10 stETH respectively. rsETH is burned for both via `IRSETH.burnFrom`.
3. A validator slashing event causes a 10% negative rebase. `LRTWithdrawalManager.balanceOf(stETH)` is now 90 stETH (was 100).
4. Alice calls `completeWithdrawal(stETH, ...)` and receives 90 stETH — her full amount, draining the contract.
5. Bob calls `completeWithdrawal(stETH, ...)`. The `safeTransfer` of 10 stETH reverts because the contract holds 0 stETH. Bob's rsETH is already burned and unrecoverable. His withdrawal request remains in the queue but can never be fulfilled.
6. The protocol has burned 100 rsETH but only delivered 90 stETH worth of value, leaving rsETH undercollateralized.

**Foundry fork test plan**: Fork mainnet, deploy/configure the protocol with stETH support, simulate two users initiating withdrawals, call `unlockQueue`, manipulate the stETH oracle/rebase by directly calling `Lido.handleOracleReport` with a slashing scenario on a fork, then demonstrate that Alice's `completeWithdrawal` succeeds while Bob's reverts with an ERC20 insufficient balance error, and verify Bob's rsETH balance remains 0.
