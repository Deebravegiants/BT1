Audit Report

## Title
Blacklisted LP Cannot Withdraw Principal — Permanent Fund Lock in `removeLiquidity` - (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
`LiquidityLib.removeLiquidity` unconditionally transfers withdrawn tokens to `owner` via `safeTransfer`. `MetricOmmPool.removeLiquidity` enforces `msg.sender == owner` with no alternative recipient parameter, so an LP whose address is blacklisted by a pool token (USDC/USDT) after depositing can never recover their principal. Every withdrawal attempt reverts atomically, permanently locking the LP's funds.

## Finding Description
`MetricOmmPool.removeLiquidity` enforces caller identity at line 206:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [1](#0-0) 

It then delegates to `LiquidityLib.removeLiquidity`, which burns shares (lines 210–214) and transfers tokens directly to `owner`:

```solidity
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);
}
if (amount1Removed > 0) {
    IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
}
``` [2](#0-1) 

The share-burn state updates at lines 210–214 occur before the transfer within the same transaction: [3](#0-2) 

Because EVM execution is atomic, a revert at the `safeTransfer` step rolls back all state changes, leaving `positionBinShares` and `binTotalShares` intact. There is no `removeLiquidityTo(address recipient)` overload, no admin rescue path for LP principal, and no try-catch around the transfer confirmed by code search.

## Impact Explanation
An LP who provided liquidity to a USDC or USDT pool and is subsequently blacklisted by the token issuer permanently loses their entire deposited principal. The funds remain accounted for in `binState.token0BalanceScaled`/`token1BalanceScaled` but can never be extracted. This is a direct, permanent loss of user principal with no recovery path — matching the "Critical/High direct loss of user principal" allowed impact gate.

## Likelihood Explanation
USDC and USDT both implement address blacklisting, and the contest README explicitly includes blocklists of ERC20 tokens as in-scope behavior. Any LP using a USDC or USDT pool who is later blacklisted — whether due to regulatory action, a compromised key, or mistaken flagging — triggers this condition. No attacker is required; the LP themselves is the victim. The condition is permanent and repeatable on every retry.

## Recommendation
Add a `recipient` parameter to `MetricOmmPool.removeLiquidity` and thread it through to `LiquidityLib.removeLiquidity`, replacing `safeTransfer(owner, ...)` with `safeTransfer(recipient, ...)`. The `msg.sender == owner` check continues to prevent unauthorized withdrawals; only the destination of the tokens changes, allowing a blacklisted owner to direct funds to a non-blacklisted address.

## Proof of Concept
1. Pool is deployed with USDC as `token1`.
2. Alice (`owner = alice`) calls `addLiquidity` and deposits 10,000 USDC into bin 0.
3. USDC issuer blacklists `alice`.
4. Alice calls `removeLiquidity(alice, salt, deltas, "")`.
5. `MetricOmmPool` passes the call to `LiquidityLib.removeLiquidity`.
6. Share accounting is computed; `amount1Removed = 10,000 USDC`.
7. `IERC20(usdc).safeTransfer(alice, 10_000e6)` reverts because `alice` is blacklisted.
8. The entire transaction reverts; Alice's shares are not burned, but the transfer fails.
9. Alice retries — same revert every time. Her 10,000 USDC is permanently locked with no rescue path available in the contract.
