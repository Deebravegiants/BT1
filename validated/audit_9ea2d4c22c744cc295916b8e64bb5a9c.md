Audit Report

## Title
Daily Fee Mint Limit Unconditionally Blocks All Price Updates — (`contracts/LRTOracle.sol`)

## Summary

`_checkAndUpdateDailyFeeMintLimit` is called on every execution of `_updateRsETHPrice`, with no bypass for any role. Once accumulated fee minting within a 24-hour period reaches `maxFeeMintAmountPerDay`, every subsequent call to either `updateRSETHPrice()` or `updateRSETHPriceAsManager()` reverts with `DailyFeeMintLimitExceeded`, leaving `rsETHPrice` stale for up to 24 hours. No privileged escape path exists in the contract.

## Finding Description

Both public entry points unconditionally delegate to `_updateRsETHPrice`: [1](#0-0) 

Inside `_updateRsETHPrice`, `_checkAndUpdateDailyFeeMintLimit` is called on every code path — with the computed fee amount when `protocolFeeInETH > 0`, and with `0` otherwise: [2](#0-1) 

The revert fires at line 206, before `rsETHPrice = newRsETHPrice` at line 313 is ever reached: [3](#0-2) [4](#0-3) 

The manager-role bypass that exists for the `PriceAboveDailyThreshold` check (line 263) is absent for the fee-limit check — `_checkAndUpdateDailyFeeMintLimit` is called identically regardless of `msg.sender`: [5](#0-4) 

The lockout persists until `block.timestamp >= feePeriodStartTime + 1 days`, at which point the counter resets: [6](#0-5) 

**Trigger sequence:**
1. Over a 24-hour window, multiple successful price updates accumulate `currentPeriodMintedFeeAmount` close to `maxFeeMintAmountPerDay`.
2. A TVL increase (e.g., EigenLayer staking rewards) produces a `rsethAmountToMintAsProtocolFee` that pushes `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay`.
3. Both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` revert for the remainder of the period.
4. `rsETHPrice` is frozen at its last value for up to 24 hours.

## Impact Explanation

**Medium — Temporary freezing of funds.**

`rsETHPrice` is the exchange rate consumed by `LRTWithdrawalManager`. `unlockQueue` reads `lrtOracle.rsETHPrice()` directly to compute payout amounts and validate price bounds: [7](#0-6) [8](#0-7) 

A stale price during a period of TVL growth means the stored rate is lower than the true rate. Operators enforcing tight price bounds (the safe operational posture) will find `unlockQueue` reverting, blocking withdrawal queue processing for up to 24 hours. Users with pending withdrawal requests cannot have their positions settled at the correct rate during the lockout window.

## Likelihood Explanation

Likelihood is **medium**. The conditions are:
- `maxFeeMintAmountPerDay` set to a finite value — this is the intended production configuration, as evidenced by `setMaxFeeMintAmountPerDay` being a named manager function.
- Sufficient TVL growth within a single 24-hour window to exhaust the daily fee cap — normal for a live protocol with significant EigenLayer restaking rewards.

No attacker action is required. Organic protocol operation is sufficient to trigger the lockout. The lockout is self-resolving after 24 hours but is repeatable every period in which TVL growth is large enough.

## Recommendation

Decouple the fee-minting limit from the price-update path. The safest fix is to skip the fee mint (not the price update) when the limit is reached: replace the `revert` in `_checkAndUpdateDailyFeeMintLimit` with a return value indicating whether minting should proceed, and in `_updateRsETHPrice`, skip the `IRSETH.mint` call and emit a `FeeMintSkipped` event when the cap is hit. `rsETHPrice = newRsETHPrice` must always execute regardless of fee-minting outcome.

Alternatively, add a manager bypass: check `IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)` before calling `_checkAndUpdateDailyFeeMintLimit`, mirroring the existing bypass at line 263 for `PriceAboveDailyThreshold`.

## Proof of Concept

```solidity
function test_dailyFeeLimitBlocksPriceUpdate() public {
    // Configure a finite daily cap
    vm.prank(manager);
    oracle.setMaxFeeMintAmountPerDay(1 ether);

    // Drive currentPeriodMintedFeeAmount to 0.99 ether via repeated
    // updateRSETHPrice() calls with small TVL increments

    // Simulate a TVL jump that produces feeAmount = 0.02 ether
    // (total would be 1.01 ether > maxFeeMintAmountPerDay)
    mockTVLIncrease(largeAmount);

    // Both entry points revert
    vm.expectRevert(LRTOracle.DailyFeeMintLimitExceeded.selector);
    oracle.updateRSETHPrice();

    vm.expectRevert(LRTOracle.DailyFeeMintLimitExceeded.selector);
    vm.prank(manager);
    oracle.updateRSETHPriceAsManager();

    // rsETHPrice is stale — unlockQueue with tight price bounds reverts
    // Warp past period boundary to confirm self-resolution
    vm.warp(block.timestamp + 1 days);
    oracle.updateRSETHPrice(); // succeeds, resets counter
}
```

The test confirms no privileged role can bypass the lockout, and the only resolution is waiting for the 24-hour period to expire.
