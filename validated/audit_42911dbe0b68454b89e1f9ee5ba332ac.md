### Title
Single Attested Node Can Deliver Arbitrary CKD Response for `AppPublicKey` Requests, Bypassing Threshold Requirement — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in `MpcContract` performs no cryptographic output verification for the `CKDAppPublicKey::AppPublicKey` variant. A single attested MPC participant (strictly below the signing threshold) can call `respond_ckd` with a completely fabricated `CKDResponse`, resolve the pending yield, and deliver arbitrary key material to the caller — bypassing the threshold-signature requirement for confidential key derivation.

---

### Finding Description

The `respond_ckd` function handles two variants of `CKDAppPublicKey`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which verifies a pairing equation ensuring the response is cryptographically consistent with the network's master key and the request's `app_id`. For `AppPublicKey`, the branch is a no-op — the contract immediately proceeds to resolve the yield with whatever `CKDResponse` the caller supplied:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The `CKDResponse` struct contains two BLS12-381 G1 points, `big_y` and `big_c`, with no domain-level constraints enforced by the contract: [3](#0-2) 

The CKD protocol for `AppPublicKey` works as follows: the MPC network computes `big_s = hash_app_id_with_pk(network_pk, app_id) * msk`, picks a random scalar `y`, and returns `big_y = G1 * y` and `big_c = big_s + app_pk1 * y`. The caller, knowing the discrete log `a` of `app_pk1` (i.e., `app_pk1 = G1 * a`), recovers the shared secret as `big_c − a * big_y = big_s`.

An attacker who delivers `big_y = G1_identity` (the identity element, a valid G1 point) and `big_c = G1 * s'` for any chosen scalar `s'` causes the caller to compute:

```
big_c − a * big_y = G1 * s' − a * G1_identity = G1 * s'
```

The attacker knows `s'`, so they know the caller's derived app private key. The contract has no guard against this because the `AppPublicKey` branch performs zero verification.

The access control check (`assert_caller_is_attested_participant_and_protocol_active`) only requires the caller to be a single attested participant — well below the signing threshold: [4](#0-3) 

---

### Impact Explanation

A single attested MPC node can unilaterally determine the confidential key derivation output for any pending `AppPublicKey` CKD request. By choosing `big_y = G1_identity` and `big_c = G1 * s'` for a known scalar `s'`, the attacker learns the caller's derived app private key. This is a direct bypass of the threshold-signature requirement: the entire purpose of the MPC network is that no single participant can determine the output, yet for `AppPublicKey` CKD requests the contract enforces no such guarantee. The pending yield is consumed and the legitimate threshold-computed response can never be delivered.

This matches the allowed impact: **Critical — bypass of threshold-signature requirements and unauthorized access to confidential key derivation output without required participant authorization.**

---

### Likelihood Explanation

The attack requires exactly one attested MPC participant to act maliciously. The participant set is small (on the order of tens of nodes), each of which is individually capable of executing this attack against any `AppPublicKey` CKD request at any time. No collusion, no leaked key, and no privileged operator access beyond normal participant status is required. The `AppPublicKey` variant is the "legacy" mode and is actively used in production (confirmed by sandbox and e2e tests).

---

### Recommendation

Apply the same cryptographic output verification to the `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. For `AppPublicKey`, the contract can verify the response by checking the pairing equation or an equivalent algebraic identity that ties `(big_y, big_c)` to the network's public key and the request's `app_id`, without requiring the caller's private key. Alternatively, reject `AppPublicKey` requests at the contract level and require all callers to migrate to `AppPublicKeyPV`, which already has the output check.

---

### Proof of Concept

1. User Alice calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(app_pk1)` where `app_pk1 = G1 * a` for Alice's private scalar `a`. The contract enqueues a yield and stores the pending request.

2. Malicious attested node Eve observes the pending `CKDRequest` on-chain.

3. Eve calls `respond_ckd(ckd_request, CKDResponse { big_y: G1_identity, big_c: G1 * s' })` for any scalar `s'` she chooses.

4. The contract executes:
   - `assert_caller_is_attested_participant_and_protocol_active()` — passes (Eve is an attested participant).
   - `match &request.app_public_key { AppPublicKey(_) => {} ... }` — no check performed.
   - `pending_requests::resolve_yields_for(...)` — yield resolved with Eve's fabricated response.

5. Alice's transaction resumes and she receives `{ big_y: G1_identity, big_c: G1 * s' }`.

6. Alice computes her derived key: `big_c − a * big_y = G1 * s' − a * G1_identity = G1 * s'`.

7. Eve knows `s'`, so she knows Alice's derived app private key `G1 * s'`.

The legitimate threshold-computed response can never be delivered because the yield has already been consumed. [5](#0-4)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
