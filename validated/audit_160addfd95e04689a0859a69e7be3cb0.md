### Title
Attacker Can Exhaust `verify_foreign_transaction` Per-Request Queue, Blocking Legitimate Bridge Users and Obtaining Bulk Threshold Signatures - (File: `crates/contract/src/pending_requests.rs`)

---

### Summary

`VerifyForeignTransactionRequest` — the map key used to queue pending foreign-chain verification yields — contains no caller identity field. Any unprivileged account can therefore fill the shared per-request queue (capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`) for **any** foreign transaction, blocking legitimate callers from submitting the same request and collecting up to 128 copies of the resulting MPC threshold signature at near-zero cost.

---

### Finding Description

The contract stores pending foreign-transaction verification yields in:

```rust
pending_verify_foreign_tx_requests: LookupMap<VerifyForeignTransactionRequest, Vec<YieldIndex>>,
``` [1](#0-0) 

The map key type is:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [2](#0-1) 

Crucially, **no caller identity (predecessor account ID) is included**. The conversion from user-supplied args to the stored key strips the caller entirely:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [3](#0-2) 

This is the opposite of the `sign` flow, where `SignatureRequest` is keyed on a tweak derived from `(predecessor_id, path)`, making each caller's queue entry unique:

```rust
impl SignatureRequest {
    pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
        let tweak = crate::kdf::derive_tweak(predecessor_id, path);
``` [4](#0-3) 

The shared queue is bounded by `MAX_PENDING_REQUEST_FAN_OUT = 128`. Once full, `push_pending_yield` panics:

```rust
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string(),
    );
}
``` [5](#0-4) 

When MPC nodes call `respond_verify_foreign_tx`, `resolve_yields_for` drains **all** 128 queued yields in one pass, delivering the same `VerifyForeignTransactionResponse` to every queued caller:

```rust
let resumed = requests
    .remove(request)
    .unwrap_or_default()
    .into_iter()
    .map(|YieldIndex { data_id }| {
        env::promise_yield_resume(&data_id, response_bytes.clone());
    })
    .count();
``` [6](#0-5) 

The sandbox test explicitly confirms that callers from **different accounts** share the same queue slot and receive the same fanned-out response: [7](#0-6) 

---

### Impact Explanation

**Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants.**

Two concrete harms arise:

1. **Queue exhaustion / bridge denial**: An attacker who submits 128 identical `verify_foreign_transaction` calls for a target `(domain_id, tx_id, payload_version)` fills the queue before a legitimate user can enqueue. Every subsequent legitimate call for that foreign transaction panics with `PendingRequestQueueFull`. Because each batch of 128 attacker yields expires after `REQUEST_EXPIRATION_BLOCKS = 200` blocks (~4 minutes), the attacker can continuously refill the queue at a cost of 128 × 1 yoctoNEAR + gas per cycle, indefinitely blocking a victim's bridge withdrawal or cross-chain settlement.

2. **Bulk threshold-signature acquisition**: When MPC nodes do respond, the attacker receives 128 copies of the same `VerifyForeignTransactionResponse` — a root-key MPC threshold signature over the foreign transaction — for 128 × 1 yoctoNEAR. Unlike the `sign` path (where the signature is over a caller-specific derived key), the foreign-tx signature is issued under the **root MPC key** with no tweak. Possessing 128 authenticated copies of this proof enables the attacker to replay the same verification credential against any NEAR contract that accepts `VerifyForeignTransactionResponse` without tracking per-response consumption, enabling double-spend or duplicate-claim conditions across bridge integrations.

---

### Likelihood Explanation

**High.** The attack requires no privileged access, no threshold collusion, and no key material. The only cost is 128 × 1 yoctoNEAR + gas per 200-block window. The target request parameters (`domain_id`, `tx_id`, `payload_version`) are fully public on-chain. The attack can be scripted to monitor the NEAR mempool and front-run any `verify_foreign_transaction` call, or applied proactively to any known pending foreign transaction.

---

### Recommendation

Include the caller's account ID in the `VerifyForeignTransactionRequest` key, mirroring the `sign` path. Concretely, add a `predecessor_id` (or a hash thereof) field to `VerifyForeignTransactionRequest` and populate it from `env::predecessor_account_id()` inside `verify_foreign_transaction`, so that each caller occupies a distinct queue slot that no other account can pre-fill.

If the fan-out-to-multiple-callers behaviour is intentional (e.g., to let many bridge users share one MPC signing round for the same tx), the queue should at minimum be partitioned by caller so that one account cannot exhaust the slots available to another.

---

### Proof of Concept

```
// Setup: Alice wants to bridge Bitcoin tx_id = 0xABCD...
// Attacker (Bob) observes the pending tx in the mempool.

// Step 1: Bob submits 128 identical verify_foreign_transaction calls
for i in 0..128 {
    bob.call(mpc_contract, "verify_foreign_transaction")
       .args_json({"request": {"domain_id": 0, "payload_version": 1,
                               "request": {"Bitcoin": {"tx_id": "0xABCD...",
                                                       "confirmations": 1,
                                                       "extractors": []}}}})
       .deposit(1)  // 1 yoctoNEAR
       .max_gas()
       .transact_async();
}

// Step 2: Alice submits the same request — PANICS with PendingRequestQueueFull
alice.call(mpc_contract, "verify_foreign_transaction")
     .args_json({"request": { /* identical */ }})
     .deposit(1)
     .transact();
// => Error: PendingRequestQueueFull { limit: 128 }

// Step 3: After 200 blocks Bob's yields expire; Bob repeats Step 1.
// Alice is blocked indefinitely at ~128 yoctoNEAR + gas per 200-block cycle.

// Step 4 (signature multiplication): When MPC nodes eventually respond to Bob's
// 128 requests, Bob receives 128 copies of VerifyForeignTransactionResponse
// containing a root-key MPC signature over tx_id 0xABCD..., usable as replay
// credentials against any bridge contract that does not deduplicate responses.
```
