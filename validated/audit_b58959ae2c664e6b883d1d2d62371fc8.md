### Title
P2P Peer-Controlled `n_transactions` in `SignedBlockHeader` Corrupts Block Body Storage and RPC Views — (`crates/apollo_p2p_sync/src/client/transaction.rs`, `crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

`HeaderStreamFactory::parse_data_for_block` accepts a `SignedBlockHeader` from any p2p peer with no cryptographic signature verification and no validation of `n_transactions`. The stored value is then used verbatim by `TransactionStreamFactory::parse_data_for_block` as the loop bound for consuming transactions. A malicious peer can set `n_transactions = N'` for a block whose canonical body has `N_actual` transactions, causing the node to write a `BlockBody` with N' arbitrary transactions to storage, permanently corrupting every downstream RPC view for that block.

---

### Finding Description

**Step 1 — Header accepted without signature or field verification.**

`HeaderStreamFactory::parse_data_for_block` performs only two checks before accepting a `SignedBlockHeader`:

- Block number ordering (line 104–113)
- Signatures vector length == 1 (line 115–119) [1](#0-0) 

There is no cryptographic verification of the signature content, no verification that `block_hash` commits to the header fields, and no validation of `n_transactions` against `transaction_commitment` or `concatenated_counts`. The developers explicitly acknowledge this gap with a TODO: [2](#0-1) 

**Step 2 — Forged header written to storage as-is.**

`SignedBlockHeader::write_to_storage` calls `append_header` with the full peer-supplied `BlockHeader`, including the attacker-controlled `n_transactions` value. [3](#0-2) 

**Step 3 — Transaction sync uses stored `n_transactions` as the loop bound.**

`TransactionStreamFactory::parse_data_for_block` reads `target_transaction_len` directly from the stored header: [4](#0-3) 

The loop then consumes exactly N' transactions from the peer stream: [5](#0-4) 

A second TODO confirms that transaction hashes are also not validated: [6](#0-5) 

**Step 4 — Corrupted `BlockBody` committed to storage.**

`(BlockBody, BlockNumber)::write_to_storage` calls `append_body` with the N'-transaction body, advancing `body_marker` and making the corruption permanent. [7](#0-6) 

---

### Impact Explanation

Any RPC call that reads the block body from storage — `starknet_getBlockWithTxs`, `starknet_getBlockTransactionCount`, `starknet_getTransactionByBlockIdAndIndex` — will return the attacker-supplied N' transactions instead of the canonical N_actual transactions. The `body_marker` advances past the corrupted block, so the node treats the corruption as finalized. This is a **High** impact: an authoritative-looking wrong value is served by RPC for every query against the affected block.

---

### Likelihood Explanation

Any unauthenticated p2p peer that can respond to a header query can execute this attack. The only barriers are the block number check and the signatures-vector-length check, both of which are trivially satisfied. No validator key, operator privilege, or special network position is required.

---

### Recommendation

1. **Verify the block hash cryptographically** in `HeaderStreamFactory::parse_data_for_block` before storing the header. The `block_hash` field must commit to all header fields including `n_transactions`; verify the signature over `block_hash` using the known sequencer public key.
2. **Validate `n_transactions` against `concatenated_counts`** (resolve the existing TODO at `header.rs:140`).
3. **Validate transaction hashes** in `TransactionStreamFactory::parse_data_for_block` against `transaction_commitment` from the stored header (resolve the existing TODO at `transaction.rs:88`).

---

### Proof of Concept

```
1. Attacker peer responds to a header query for block B (canonical n_transactions = 3).
2. Attacker sends SignedBlockHeader { n_transactions: 5, signatures: [any_single_sig], block_number: B, ... }.
3. parse_data_for_block (header) passes: block_number matches, signatures.len() == 1.
4. Header stored with n_transactions = 5.
5. Attacker responds to the subsequent transaction query with 5 arbitrary FullTransactions.
6. parse_data_for_block (transaction) loop runs 5 times (0..5 < 5), writes BlockBody with 5 txs.
7. append_body commits; body_marker advances to B+1.
8. starknet_getBlockTransactionCount(B) returns 5 instead of 3.
9. starknet_getBlockWithTxs(B) returns 5 attacker-chosen transactions.
```
