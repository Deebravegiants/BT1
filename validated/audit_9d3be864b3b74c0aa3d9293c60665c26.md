### Title
Single Byzantine Participant Can Freeze All `verify_foreign_transaction` Requests via Empty Chain Registration — (`File: crates/contract/src/lib.rs`)

---

### Summary

`get_supported_foreign_chains()` computes the set of serviceable foreign chains as the **strict intersection** of every active participant's registered chain list. A single Byzantine participant (well below the signing threshold) can call `register_foreign_chain_support` with an empty set, instantly collapsing the intersection to the empty set. Because `verify_foreign_transaction` gates on this computed set, every subsequent foreign-chain verification request is rejected at the contract level — no signing ever begins.

---

### Finding Description

`get_supported_foreign_chains()` iterates over every active participant's registered chain list and returns only chains that appear in **all** of them: [1](#0-0) 

The filter predicate is `nodes_supporting_chain.is_superset(&active_participant_account_ids)` — a chain is included only when every active participant has registered it. If any participant's entry is absent or empty, the intersection is empty.

`register_foreign_chain_support` lets any authenticated participant overwrite their entry with an arbitrary `SupportedForeignChains` value, including an empty set, with no lower-bound validation: [2](#0-1) 

`verify_foreign_transaction` then gates every incoming request on this computed set: [3](#0-2) 

The project's own design document explicitly acknowledges this property:

> "A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down." [4](#0-3) 

The proposed fix (threshold-based availability instead of strict intersection) is documented but **not yet implemented** in the production contract code. [5](#0-4) 

---

### Impact Explanation

**Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants.**

A single participant's empty registration immediately and silently removes every foreign chain from the supported set. All calls to `verify_foreign_transaction` for any chain are rejected with `ForeignChainNotSupported`. Users who have submitted requests lose their 1 yoctonear deposit; more critically, time-sensitive cross-chain operations (e.g., a bridge unlock that must be signed within a block window) cannot be completed, potentially causing loss of funds on the foreign chain. The disruption persists until the attacker re-registers their chains or is removed via a resharing vote — a slow governance process.

This maps directly to M-07's pattern: a single actor manipulates a shared state value (oracle rate / supported-chain set) through a legitimate on-chain action, causing disproportionate harm to all other users' pending positions/requests.

---

### Likelihood Explanation

The attacker must be an active MPC participant — a Byzantine participant strictly below the signing threshold. The attack requires only one such participant and a single on-chain transaction (`register_foreign_chain_support` with `foreign_chain_support: []`). No capital, no coordination, no cryptographic capability is needed. The call is authenticated but not rate-limited or bounded below. The attack is repeatable: even if the honest network detects and responds, the attacker can re-register an empty list after every honest re-registration.

---

### Recommendation

Implement the threshold-based availability rule already proposed in `docs/design/calculating-supported-foreign-chains.md`:

- Replace the strict-intersection rule in `get_supported_foreign_chains()` with a threshold-quorum rule: a chain is available if ≥ `signing_threshold` active participants have registered it.
- Gate `verify_foreign_transaction` on this threshold-based set rather than the all-or-nothing intersection.
- Add a non-empty validation or minimum-chain-count check in `register_foreign_chain_support` to prevent trivially empty registrations from being accepted without review. [6](#0-5) 

---

### Proof of Concept

**Setup:** MPC network with `n = 5` participants, signing threshold `t = 3`. All five participants have previously registered `{Bitcoin, Ethereum}` via `register_foreign_chain_support`.

**Attack:**

1. Byzantine participant `P_evil` (one of the five) calls:
   ```json
   register_foreign_chain_support({ "foreign_chain_support": [] })
   ```
   This is a valid, authenticated call — `voter_or_panic()` passes because `P_evil` is a participant.

2. `node_foreign_chain_support.foreign_chain_support_by_node` now contains:
   ```
   P1 → {Bitcoin, Ethereum}
   P2 → {Bitcoin, Ethereum}
   P3 → {Bitcoin, Ethereum}
   P4 → {Bitcoin, Ethereum}
   P_evil → {}
   ```

3. `get_supported_foreign_chains()` computes the intersection:
   - For Bitcoin: `{P1,P2,P3,P4}` is **not** a superset of `{P1,P2,P3,P4,P_evil}` → excluded.
   - For Ethereum: same → excluded.
   - Result: `{}` (empty set).

4. Any user calling `verify_foreign_transaction` for Bitcoin or Ethereum now receives:
   ```
   InvalidParameters::ForeignChainNotSupported { requested: Bitcoin }
   ```
   The request is rejected before any signing begins. The user's 1 yoctonear deposit is consumed.

5. The attack persists until `P_evil` re-registers or is removed via a resharing governance vote — which requires threshold-level coordination and multiple blocks. [7](#0-6) [2](#0-1) [3](#0-2)

```rust
    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
```
