### Title
Solana `FinalizeTransferPayload` Omits `message` Field from Signature Verification, Permanently Locking NEAR→Solana Transfers with Non-Empty Messages — (File: `solana/programs/bridge_token_factory/src/state/message/finalize_transfer.rs`)

---

### Summary

The Solana `FinalizeTransferPayload.serialize_for_near()` never includes the `message` field in the serialized payload used for ECDSA signature verification. The NEAR side's `TransferMessagePayload.encode_hashable()` conditionally includes `message` bytes when non-empty. This structural inconsistency means any NEAR→Solana transfer initiated with a `DestHexMsg` message produces a NEAR-signed hash that Solana can never reproduce, permanently locking the user's funds.

---

### Finding Description

**NEAR signing side** (`near/omni-types/src/lib.rs`):

`TransferMessagePayload` carries an optional `message: Vec<u8>` field: [1](#0-0) 

`encode_hashable()` branches on whether `message` is empty:
- Empty → serializes as `TransferMessagePayloadV1` (no `message` field)
- Non-empty → serializes as `TransferMessagePayload` (includes `message` bytes) [2](#0-1) 

In `sign_transfer`, the `message` field is populated from `DestinationChainMsg::DestHexMsg`: [3](#0-2) 

**Solana verification side** (`solana/programs/bridge_token_factory/src/state/message/finalize_transfer.rs`):

`FinalizeTransferPayload` has no `message` field at all: [4](#0-3) 

`serialize_for_near()` ends at `fee_recipient` — no message bytes are ever appended: [5](#0-4) 

Signature verification hashes the output of `serialize_for_near()`: [6](#0-5) 

**Contrast with other chains**: EVM, Starknet, and Aptos all correctly include the `message` bytes in their verification payloads when non-empty: [7](#0-6) [8](#0-7) [9](#0-8) 

Only Solana omits it.

---

### Impact Explanation

**Critical — Irreversible fund lock.**

When a user initiates a NEAR→Solana transfer with a non-empty `DestHexMsg` message:
1. NEAR signs `keccak256(borsh(prefix | nonce | transfer_id | token | amount | recipient | fee_recipient | message_bytes))`
2. Solana verifies against `keccak256(borsh(prefix | nonce | transfer_id | token | amount | recipient | fee_recipient))` — no message
3. The hashes differ → `SignatureVerificationFailed` → `finalize_transfer` always reverts
4. The user's tokens are permanently locked in the NEAR bridge contract with no recovery path

---

### Likelihood Explanation

**Medium.** The `msg` field in `InitTransferMsg` is a standard user-facing parameter. Any user who passes a `{"DestHexMsg": "0x..."}` JSON value in `msg` for a Solana-destined transfer triggers this. The `DestHexMsg` variant is the documented mechanism for forwarding destination-chain hook payloads. No privileged access is required — a regular unprivileged user calling `ft_on_transfer` with a Solana recipient and a non-empty `DestHexMsg` is sufficient.

---

### Recommendation

Add a `message: Option<Vec<u8>>` field to `FinalizeTransferPayload` in `solana/programs/bridge_token_factory/src/state/message/finalize_transfer.rs` and append it in `serialize_for_near()` to match the NEAR-side `encode_hashable()` behavior:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
    pub message: Option<Vec<u8>>,  // add this
}

// In serialize_for_near, after fee_recipient:
if let Some(msg) = &self.message {
    msg.serialize(&mut writer)?;
}
```

This mirrors the conditional inclusion in `encode_hashable()` and matches the EVM/Starknet/Aptos implementations.

---

### Proof of Concept

1. User calls `ft_on_transfer` on NEAR with:
   - `recipient = "sol:4Nd1mBQtrMJVYVfKf2PX98YbKGi1jdng17mpNkHkwznB"`
   - `msg = "{\"DestHexMsg\":\"deadbeef\"}"`
2. NEAR stores the transfer; relayer calls `sign_transfer`.
3. NEAR extracts `message = [0xde, 0xad, 0xbe, 0xef]` and signs:
   `keccak256(borsh(TransferMessagePayload { ..., message: [0xde,0xad,0xbe,0xef] }))`
4. Relayer submits `finalize_transfer` on Solana with the signed payload.
5. Solana calls `FinalizeTransferPayload::serialize_for_near()` → produces bytes ending at `fee_recipient`, no message.
6. `keccak::hash(serialized)` differs from the NEAR-signed hash.
7. `secp256k1_recover` returns a different address → `require!(signer.0 == *derived_near_bridge_address)` fails.
8. Transaction reverts with `SignatureVerificationFailed`. The transfer is permanently unclaimable; user funds are locked forever.

```rust
impl<P: Payload> SignedPayload<P> {
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
```
