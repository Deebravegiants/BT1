### Title
Missing `payload_hash` Binding to `request` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that a caller-supplied `response.payload_hash` is signed by the MPC root key, but never checks that `payload_hash` was actually derived from the `request` being answered. A single Byzantine attested participant can replay any previously published threshold signature (from a prior, legitimately completed request) against a new pending request, causing the contract to resolve that request with a forged verification response.

### Finding Description

In `respond_verify_foreign_tx` the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,          // root public key — no tweak, no request binding
)
.is_ok()
``` [1](#0-0) 

`payload_hash` is taken verbatim from the caller's `response` argument. The contract only asserts that the ECDSA signature is valid for *that hash* under the root key. It does **not** assert that `payload_hash == hash(request_content || extracted_values)`.

By contrast, the regular `respond` path derives the expected key from the request's tweak before verifying:

```rust
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
``` [2](#0-1) 

No equivalent binding exists for `respond_verify_foreign_tx`. After the signature check passes, `resolve_yields_for` drains the queue for `request` and delivers the attacker-chosen `response` bytes to every waiting yield:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [3](#0-2) 

The `resolve_yields_for` helper itself only checks whether the queue is non-empty; it has no knowledge of whether the response content is semantically correct for the request:

```rust
let resumed = requests
    .remove(request)
    .unwrap_or_default()
    .into_iter()
    .map(|YieldIndex { data_id }| {
        env::promise_yield_resume(&data_id, response_bytes.clone());
    })
    .count();
``` [4](#0-3) 

### Impact Explanation

Every `respond_verify_foreign_tx` call that succeeds is permanently recorded on-chain with its `(payload_hash, signature)` pair. A single attested participant can take any such historical pair `(H₁, σ₁)` — produced by the MPC network for a completely different foreign transaction — and submit it as the response to a new, unrelated pending request R₂. The contract accepts the call because σ₁ is a valid root-key signature over H₁, and R₂ is present in `pending_verify_foreign_tx_requests`. Every caller waiting on R₂ receives a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to a different transaction entirely. Any downstream contract that does not independently re-verify the hash (e.g., by calling `ForeignChainSignatureVerifier::verify_signature` from the SDK) will treat the forged response as proof that R₂'s foreign transaction was confirmed, enabling invalid bridge execution or double-spend conditions.

The client-side verifier in the SDK does catch the mismatch:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [5](#0-4) 

However, this check is entirely client-side and optional; the contract itself provides no enforcement. Any integrator that trusts the contract's yield-resume value directly is vulnerable.

### Likelihood Explanation

The attack requires only a single Byzantine attested participant — explicitly within the stated threat model ("Byzantine participant strictly below the signing threshold"). No new cryptographic capability is needed: the attacker simply reads a previously published `(payload_hash, signature)` pair from chain history and resubmits it. The `respond_verify_foreign_tx` entry point imposes no rate limit or cooldown, so the attack can be executed immediately after any new `verify_foreign_transaction` request appears in the queue.

### Recommendation

The contract must bind `payload_hash` to the `request` before accepting the response. The minimal fix is to reconstruct the expected payload prefix from the known `request` fields and assert that `response.payload_hash` is consistent with it. A stronger fix is to have MPC nodes sign `hash(request_canonical_bytes || payload_hash)` instead of `payload_hash` alone, so the contract can verify the binding without knowing the extracted values. At minimum, add a check analogous to:

```rust
// Verify payload_hash commits to the request being answered.
// (Exact implementation depends on ForeignTxSignPayload serialization.)
require!(
    payload_hash_commits_to_request(&response.payload_hash, &request),
    "LOP-analog: payload_hash does not match request"
);
```

### Proof of Concept

1. Alice submits `verify_foreign_transaction(R1)` for Bitcoin tx A. MPC nodes compute `payload_hash_1 = hash(R1 || extracted_values_A)`, produce threshold signature `σ₁`, and a legitimate participant calls `respond_verify_foreign_tx(R1, {payload_hash_1, σ₁})`. The call succeeds; `(payload_hash_1, σ₁)` is now on-chain.
2. Bob submits `verify_foreign_transaction(R2)` for Bitcoin tx B. R2 enters `pending_verify_foreign_tx_requests`.
3. Malicious attested participant M calls `respond_verify_foreign_tx(R2, {payload_hash_1, σ₁})`.
4. The contract checks: σ₁ valid for `payload_hash_1` under root key → **yes**; R2 in pending map → **yes**. `resolve_yields_for` drains R2's queue and resumes Bob's yield with `{payload_hash_1, σ₁}`.
5. Bob's contract receives a `VerifyForeignTransactionResponse` asserting that Bitcoin tx B was verified, but the embedded `payload_hash` and signature actually cover Bitcoin tx A. If Bob's contract does not re-verify the hash, it proceeds as if tx B was confirmed — enabling invalid bridge execution.
