### Title
Missing `curve.check()` on `root.Q` in `blob_to_keyset` Allows Torsion-Point Root Public Key to Be Accepted — (`src/cbmpc/api/hd_keyset_eddsa_2p.cpp`)

---

### Summary

`blob_to_keyset` deserializes `root_Q_compressed` using only `from_bin`, which validates that the bytes encode a point on the Ed25519 curve but does **not** check subgroup membership. The analogous function `blob_to_key` in `src/cbmpc/api/eddsa2pc.cpp` explicitly calls `curve.check()` after `from_bin`, which rejects torsion points. That guard is absent in `blob_to_keyset`, so any caller of `extract_root_public_key_compressed`, `refresh`, or `derive_eddsa_2p_keys` that supplies a crafted blob with a torsion-point `root_Q_compressed` will receive `SUCCESS` and a non-subgroup point as the authoritative root public key.

---

### Finding Description

**Vulnerable path:**

```
extract_root_public_key_compressed(keyset_blob, out_Q_compressed)
  → deserialize_keyset_blob(keyset_blob, keyset)
    → blob_to_keyset(blob, keyset)
        keyset.root.Q.from_bin(keyset.curve, blob.root_Q_compressed)  // succeeds for torsion point
        // ← curve.check(keyset.root.Q) is MISSING here
  → out_Q_compressed = keyset.root.Q.to_compressed_bin()   // returns torsion point bytes
  → return SUCCESS
```

**Guard present in the analogous function (`blob_to_key`, `src/cbmpc/api/eddsa2pc.cpp:40-42`):**

```cpp
error_t rv = key.Q.from_bin(key.curve, blob.Q_compressed);
if (rv) return coinbase::error(rv, "invalid key blob");
if (key.curve.check(key.Q)) return coinbase::error(E_FORMAT, "invalid key blob");  // ← present
``` [1](#0-0) 

**Guard absent in `blob_to_keyset` (`src/cbmpc/api/hd_keyset_eddsa_2p.cpp:78-80`):**

```cpp
error_t rv = keyset.root.Q.from_bin(keyset.curve, blob.root_Q_compressed);
if (rv) return rv;
return keyset.root.K.from_bin(keyset.curve, blob.root_K_compressed);
// ← no curve.check(keyset.root.Q)
``` [2](#0-1) 

**Why `from_bin` alone is insufficient:**

`ecurve_ed_t::from_bin` only solves the curve equation to recover a point; it does not check subgroup membership. The test suite explicitly confirms this: the order-2 torsion point (y = p−1) passes `from_bin` with `SUCCESS`, `is_on_curve()` returns `true`, but `is_in_subgroup()` returns `false` and `curve.check()` returns non-`SUCCESS`. [3](#0-2) [4](#0-3) 

**`ecurve_t::check` performs the required subgroup check:**

```cpp
error_t ecurve_t::check(const ecc_point_t& point) const {
  if (!point.valid()) return crypto::error("EC-point invalid");
  if (point.get_curve() != *this) return crypto::error("EC-point of wrong curve");
  if (!point.is_in_subgroup()) return crypto::error("EC-point is not on curve");
  if (!thread_local_store_allow_ecc_infinity)
    if (point.is_infinity()) return crypto::error("EC-point is infinity");
  return SUCCESS;
}
``` [5](#0-4) 

The same missing guard affects `refresh` and `derive_eddsa_2p_keys`, both of which call `deserialize_keyset_blob` → `blob_to_keyset` without any subsequent subgroup check on `keyset.root.Q`. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

An attacker who can supply a `keyset_blob` to `extract_root_public_key_compressed` receives `SUCCESS` and a torsion-point encoding as the "authoritative root public key." Callers that rely on this API to validate the blob and obtain a trustworthy root public key are misled: the returned bytes decode to a point that fails `is_in_subgroup()`. The same torsion-contaminated `root.Q` propagates into `derive_keys` where it is used as the base for `Q_derived = Q + delta_G`, producing derived key public keys that are torsion-contaminated. This is a public-API-reachable validation bypass that causes accepted invalid cryptographic output, matching the "High" impact tier.

---

### Likelihood Explanation

The attacker only needs to craft a 32-byte `root_Q_compressed` field encoding a known Ed25519 torsion point (e.g., the order-2 point with y = p−1, whose encoding is publicly documented) and embed it in a well-formed `keyset_blob_v1_t`. No protocol interaction, threshold collusion, or privileged access is required. The API is a single-party call with no peer authentication.

---

### Recommendation

Add the same `curve.check()` guard to `blob_to_keyset` immediately after `from_bin` for both `root.Q` and `root.K`:

```cpp
error_t rv = keyset.root.Q.from_bin(keyset.curve, blob.root_Q_compressed);
if (rv) return rv;
if (keyset.curve.check(keyset.root.Q)) return coinbase::error(E_FORMAT, "invalid keyset blob");

rv = keyset.root.K.from_bin(keyset.curve, blob.root_K_compressed);
if (rv) return rv;
if (keyset.curve.check(keyset.root.K)) return coinbase::error(E_FORMAT, "invalid keyset blob");
return SUCCESS;
```

This mirrors the pattern already used in `blob_to_key` (`src/cbmpc/api/eddsa2pc.cpp:42`) and in `attach_private_scalar` (`src/cbmpc/api/eddsa2pc.cpp:204,216`).

---

### Proof of Concept

1. Construct a valid `keyset_blob_v1_t` with `version=1`, `role=0`, `curve=ed25519`, valid `x_share` and `k_share` in range, and `root_K_compressed` encoding any valid subgroup point.
2. Set `root_Q_compressed` to the 32-byte encoding of the Ed25519 order-2 torsion point:
   - `bytes[0] = 0xec`, `bytes[1..30] = 0xff`, `bytes[31] = 0x7f`
3. Serialize and call `extract_root_public_key_compressed(keyset_blob, out)`.
4. Assert the return value is `SUCCESS`.
5. Decode `out` with `from_bin`; assert `is_on_curve() == true` and `is_in_subgroup() == false`.

The test at `tests/unit/crypto/test_eddsa.cpp:199-204` already confirms that this exact torsion-point encoding passes `from_bin` and fails `is_in_subgroup()` / `curve.check()`, establishing the primitive needed for the exploit. [8](#0-7)
