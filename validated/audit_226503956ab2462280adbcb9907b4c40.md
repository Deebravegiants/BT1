### Title
`respond_verify_foreign_tx` Accepts Replayed Signatures Without Binding `payload_hash` to the Pending Request — (File: crates/contract/src/lib.rs)

### Summary
The `respond_verify_foreign_tx` function verifies that the submitted signature is valid over the caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` is cryptographically bound to the specific `request` being resolved. A single Byzantine attested participant (below the signing threshold) can replay a valid MPC threshold signature obtained from a prior legitimate foreign-transaction verification to resolve a completely different pending request, delivering a forged attestation to the waiting bridge contract.

### Finding Description
In `respond_verify_foreign_tx` (lines 718–743 of `crates/contract/src/lib.rs`), the contract performs two checks before resolving the pending request:

1. It verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the MPC root public key.
2. It calls `pending_requests::resolve_yields_for` keyed on `request`, delivering the full `response` (including `payload_hash` and `signature`) to every queued yield. [1](#0-0) 

The critical missing check: the contract never verifies that `response.payload_hash` equals `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: <actual extracted values> }).compute_msg_hash()`. The `payload_hash` field is entirely caller-supplied and is only checked for signature validity, not for content binding to the pending request.

Compare this with the regular `respond` function, which correctly takes the payload directly from `request.payload` (the value committed to at submission time) and verifies the signature against it — the caller cannot substitute a different hash: [2](#0-1) 

In `respond_verify_foreign_tx`, the payload is not stored in the pending request because the contract cannot know the foreign-chain extracted values at submission time. This design gap means the contract has no anchor to validate what `payload_hash` actually commits to.

The `ForeignTxSignPayload` type that nodes are supposed to sign binds both the original `request` (including `tx_id`) and the `values` extracted from the foreign chain: [3](#0-2) 

The SDK-side `ForeignChainSignatureVerifier.verify_signature()` does perform this binding check (comparing `expected_payload_hash == response.payload_hash`), but this check lives off-chain in the SDK and is not enforced by the contract: [4](#0-3) 

### Impact Explanation
A Byzantine attested participant (a single rogue MPC node operator) can execute a cross-chain replay attack:

1. Attacker calls `verify_foreign_transaction({ tx_id: B, ... })` for a non-existent or adversarially chosen transaction, creating a pending request keyed on `{tx_id: B}`.
2. A legitimate flow processes `verify_foreign_transaction({ tx_id: A, ... })`. An honest node calls `respond_verify_foreign_tx(request={tx_id: A}, response={payload_hash: H_A, signature: sig_A})`. This is public on-chain.
3. Attacker calls `respond_verify_foreign_tx(request={tx_id: B}, response={payload_hash: H_A, signature: sig_A})`.
4. The contract checks: is `sig_A` valid over `H_A` under the root public key? **Yes.** Does `{tx_id: B}` have a pending entry? **Yes.** Both pass.
5. The contract resolves the pending request for `tx_id B` with `{payload_hash: H_A, signature: sig_A}`.

The bridge contract waiting on `verify_foreign_transaction(tx_id=B)` receives a response whose `payload_hash` encodes `tx_id A`'s data. Any bridge contract that does not independently call `ForeignChainSignatureVerifier.verify_signature()` (the off-chain SDK check) will accept this as a valid attestation that `tx_id B` was verified, enabling invalid bridge execution or double-spend conditions. [5](#0-4) 

### Likelihood Explanation
The attack requires only a single Byzantine attested participant — no threshold collusion is needed. The attacker does not forge a signature; they reuse a legitimately produced MPC threshold signature that is already public on-chain. The only prerequisites are: (a) holding a valid TEE attestation (i.e., being an active MPC node operator), and (b) observing any prior `respond_verify_foreign_tx` call on-chain to obtain a reusable `(payload_hash, signature)` pair. Both conditions are routinely satisfied in a live network.

### Recommendation
- **Short term:** In `respond_verify_foreign_tx`, require the caller to also submit the `values: Vec<ExtractedValue>` that were extracted from the foreign chain. The contract can then recompute `expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request.clone(), values }).compute_msg_hash()` and assert `expected_hash == response.payload_hash` before accepting the response. This mirrors how `respond` binds the signature to `request.payload`.
- **Long term:** Ensure that all bridge contracts consuming `verify_foreign_transaction` responses are required (or strongly guided) to call `ForeignChainSignatureVerifier.verify_signature()` from the SDK, which already performs the binding check. Consider making the `values` part of the on-chain response type so the binding is verifiable by any on-chain consumer without off-chain SDK logic.

### Proof of Concept
```
// Step 1 — attacker creates a pending request for a bogus/adversarial tx
verify_foreign_transaction({ tx_id: B, extractors: [...], domain_id: X })
  → pending_verify_foreign_tx_requests[{tx_id: B, ...}] = [yield_B]

// Step 2 — legitimate flow produces a valid (payload_hash_A, sig_A) on-chain
verify_foreign_transaction({ tx_id: A, ... })
respond_verify_foreign_tx(
    request = { tx_id: A, ... },
    response = { payload_hash: H_A, signature: sig_A }
)
  → H_A = SHA256(borsh(ForeignTxSignPayload::V1({ request: {tx_id: A}, values: [block_hash_A] })))
  → sig_A is a valid MPC threshold signature over H_A  (now public on-chain)

// Step 3 — attacker replays sig_A against the pending request for tx_id B
respond_verify_foreign_tx(
    request = { tx_id: B, ... },          // matches pending entry
    response = { payload_hash: H_A,        // encodes tx_id A's data, NOT tx_id B
                 signature: sig_A }        // valid over H_A ✓
)
  → verify_ecdsa_signature(sig_A, H_A, root_pk) → OK  ✓
  → pending_verify_foreign_tx_requests[{tx_id: B}] exists  ✓
  → resolve_yields_for({tx_id: B}, serialize({H_A, sig_A}))

// Step 4 — bridge contract receives forged attestation for tx_id B
// payload_hash in response = H_A (commits to tx_id A, not tx_id B)
// Bridge contract that skips ForeignChainSignatureVerifier.verify_signature()
// accepts this as proof that tx_id B was verified → invalid bridge execution
```

        Ok(Hash256(hasher.finalize().into()))
