### Title
Receipt Size Limit Bypassed via Post-Validation `output_data_receivers` Mutation in `promise_return` Path - (File: `runtime/runtime/src/lib.rs`)

### Summary
The NEAR runtime validates newly-created receipts for size compliance before they are finalized, but then mutates those already-validated receipts by appending `output_data_receivers` entries after the size check. This allows an unprivileged user to craft a contract that produces receipts exceeding `max_receipt_size`, which are then committed to the outgoing receipts root and forwarded cross-shard, bypassing the protocol's receipt size invariant.

### Finding Description

The vulnerability is a two-phase inconsistency analogous to the reported `_mint`/`_safeMint` mismatch:

**Phase 1 — Strict validation (NewReceipt mode):**
In `apply_action_receipt`, after each action is applied, the runtime validates every newly-created receipt using `ValidateReceiptMode::NewReceipt`, which enforces the `max_receipt_size` limit:

```rust
// runtime/runtime/src/lib.rs lines 856-865
if let Err(e) = new_result.new_receipts.iter().try_for_each(|receipt| {
    validate_receipt(
        &apply_state.config.wasm_config.limit_config,
        receipt,
        apply_state.current_protocol_version,
        ValidateReceiptMode::NewReceipt,  // ← checks size
    )
}) {
    new_result.result = Err(ActionErrorKind::NewReceiptValidationError(e).into());
}
``` [1](#0-0) 

**Phase 2 — Post-validation mutation (bypasses size check):**
After validation, if the action receipt has `output_data_receivers` and the result is `ReturnData::ReceiptIndex` (i.e., `promise_return` was called), the runtime **mutates** the already-validated receipt by appending `output_data_receivers` to it:

```rust
// runtime/runtime/src/lib.rs lines 1019-1037
if !action_receipt.output_data_receivers().is_empty() {

        match result.new_receipts.get_mut(receipt_index as usize)...receipt_mut() {
            ReceiptEnum::Action(new_action_receipt) | ... => new_action_receipt
                .output_data_receivers
                .extend_from_slice(&action_receipt.output_data_receivers()),
            ...
        }
    }
}
``` [2](#0-1)

### Impact Explanation

The concrete corrupted protocol value is the **outgoing receipts root** committed in the chunk header. It includes receipts that violate the `max_receipt_size` invariant. Downstream effects include:

1. **Receipt size invariant violation**: Receipts exceeding `max_receipt_size` (4,194,304 bytes) are committed to the outgoing receipts root and forwarded cross-shard, violating the protocol's stated size guarantee.
2. **Congestion control miscalculation**: The bandwidth scheduler and `try_forward` logic receive receipts whose true size exceeds `max_receipt_size`, requiring ad-hoc clamping workarounds that were not part of the original protocol design.
3. **Storage proof size limit interaction**: The `per_receipt_storage_proof_size_limit` (4,000,000 bytes) is smaller than `max_receipt_size` (4,194,304 bytes). An oversized receipt could exceed this limit, potentially causing stateless validation failures for chunk validators processing the receipt.

### Likelihood Explanation

Any unprivileged user can trigger this by:
1. Deploying a contract via a standard RPC `broadcast_tx` call.
2. Calling a method that creates a receipt sized exactly at `max_receipt_size` (without `output_data_receivers`) and calls `promise_return` on it.
3. The parent receipt's `output_data_receivers` are appended post-validation, pushing the child receipt over the limit.

The test contract `max_receipt_size_promise_return_method1` in the test suite demonstrates this is straightforwardly reproducible. [10](#0-9) 

### Recommendation

Move the `output_data_receivers` mutation to occur **before** the `validate_receipt` call, or re-validate the receipt after mutation. Specifically, in `apply_action_receipt`, the size check in `ValidateReceiptMode::NewReceipt` must be applied to the **final** form of the receipt (after `output_data_receivers` have been appended), not the intermediate form. This mirrors the fix suggested in the external report: apply the stricter check at the point where the object reaches its final state.

### Proof of Concept

1. Deploy a contract with a method that:
   - Creates promise A targeting self with args sized to make the receipt exactly `max_receipt_size` bytes (4,194,304 bytes).
   - Creates promise B targeting self.
   - Calls `promise_then(A, B)` and `promise_return(A)`.
2. Call the method. The runtime:
   - Validates receipt A at size = `max_receipt_size` → passes.
   - Appends B's `data_id`/`receiver_id` to A's `output_data_receivers` → size > `max_receipt_size`.
   - Commits the oversized receipt A to the outgoing receipts root.
3. The receiving shard processes receipt A with `ExistingReceipt` mode (no size check) → receipt executes successfully despite exceeding the protocol limit.

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` confirms this path produces an oversized receipt that is accepted and processed. [11](#0-10)

pub enum ValidateReceiptMode {
    /// Used for validating new receipts that were just created.
    /// More strict than `OldReceipt` mode, which has to handle older receipts.
    NewReceipt,
    /// Used for validating older receipts that were saved in the state/received. Less strict than

    ///    to handle them gracefully until the receipt size limit bug is fixed.

    ExistingReceipt,
}

#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
