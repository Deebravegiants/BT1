### Title
Missing CKD Response Integrity Check for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Attacker-Controlled Derived Key - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC contract performs no cryptographic output validation when the pending request uses the `AppPublicKey` (legacy) variant of `CKDAppPublicKey`. Any single attested participant can call `respond_ckd` with an arbitrary `CKDResponse` — one whose `big_y`/`big_c` BLS12-381 points encode a key the attacker controls — and the contract will accept it, resolve all queued yields, and deliver the attacker-chosen key material to the requesting user. The `AppPublicKeyPV` variant is protected by an on-chain `ckd_output_check`; the `AppPublicKey` branch is entirely unguarded.

---

### Finding Description

`respond_ckd` (lines 653–689) enforces that the caller is an attested participant, then branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

For `AppPublicKeyPV`, `ckd_output_check` verifies that `big_y` and `big_c` are correctly derived from the BLS root key and the user's `app_id`. For `AppPublicKey`, the arm is an empty block — the response is serialised and delivered to the user without any verification that it was honestly computed.

The `CKDRequest` key is publicly readable via `get_pending_ckd_request`. A malicious attested participant can observe any pending `AppPublicKey`-mode request, construct a `CKDResponse` whose `big_y`/`big_c` encode a key they control, and race `respond_ckd` before honest nodes. Because `resolve_yields_for` drains the entire queue on the first successful call, the attacker's response is delivered to every queued caller of that request. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

**Critical — Confidential key derivation output without required participant authorization.**

The user's application decrypts the received `(big_y, big_c)` pair to obtain the derived BLS key. If the attacker supplied those points, the decrypted key is one the attacker already knows. The attacker can then:

- Sign arbitrary transactions on any foreign chain using the victim's derived key path.
- Silently exfiltrate the derived secret without the victim detecting the substitution (the response is structurally valid; only the key material is wrong).

Every user who calls `request_app_private_key` with `AppPublicKey` mode is affected. The `AppPublicKey` variant is explicitly documented as the supported legacy path, so it is reachable in production. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

A single Byzantine attested participant (strictly below the signing threshold) is sufficient. The attacker does not need to collude with other nodes, break TEE isolation, or know any secret material beyond what a legitimate participant already possesses. The pending request map is public state; the attacker can poll `get_pending_ckd_request` to identify targets. The race window is the block interval between the user's `request_app_private_key` call and the first honest `respond_ckd` call — a window the attacker can exploit by submitting their transaction in the same or next block. [5](#0-4) 

---

### Recommendation

Apply the same `ckd_output_check` guard to the `AppPublicKey` branch. If the `AppPublicKey` variant cannot provide a publicly-verifiable commitment (by design), the contract should either:

1. **Reject `AppPublicKey` requests at the contract level** and require all callers to use `AppPublicKeyPV`, or
2. **Add an equivalent integrity check** for the `AppPublicKey` case — for example, verifying that `big_y` is a valid encryption of the expected derived key under the user's supplied public key using the BLS root key.

The fix for `AppPublicKeyPV` is the reference implementation:

```rust
dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
    if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
        env::panic_str("CKD output check failed");
    }
}
```

The same pattern must be applied (or an equivalent) to the `AppPublicKey` arm. [3](#0-2) [6](#0-5)

### Proof of Concept

1. **Setup**: Contract is in `Running` state with ≥1 attested participant. Attacker controls one participant account (Byzantine, below threshold).

2. **Observe target**: Victim calls `request_app_private_key` with `CKDRequestArgs { app_public_key: AppPublicKey(victim_pk), derivation_path, domain_id }`. The resulting `CKDRequest` (keyed by `app_id = derive_app_id(victim_account, derivation_path)`) is now visible in `pending_ckd_requests`.

3. **Craft fake response**: Attacker generates a BLS12-381 key pair `(attacker_sk, attacker_pk)` and constructs `CKDResponse { big_y: encrypt(attacker_sk, victim_pk), big_c: attacker_pk }` — a response that decrypts to a key the attacker knows.

4. **Race honest nodes**: Attacker calls `respond_ckd(target_request, fake_response)` from their attested participant account. Because `AppPublicKey` arm has no check, the call succeeds.

5. **Outcome**: `resolve_yields_for` drains the queue and delivers `fake_response` to the victim via `promise_yield_resume`. The victim's application decrypts a key the attacker already knows. The attacker can now sign foreign-chain transactions on the victim's behalf. [7](#0-6) [8](#0-7)

```rust
    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
```

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
