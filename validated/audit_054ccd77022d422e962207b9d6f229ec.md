Audit Report

## Title
Unrestricted `addLiquidity` Caller Griefs Any LP Position via `minimalMintableLiquidity` Trap — (File: `metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`addLiquidity` accepts a caller-supplied `owner` address with no `msg.sender == owner` guard, allowing any unprivileged caller to add dust shares to any existing LP position. By adding a share amount `delta` such that `1 ≤ delta < minimalMintableLiquidity`, an attacker forces the victim's original partial-removal amount to leave a remainder below `minimalMintableLiquidity`, causing `removeLiquidity` to revert. The attack is repeatable at near-zero cost and permanently breaks partial withdrawal for smart contract LPs that specify fixed share amounts.

## Finding Description
`MetricOmmPool.addLiquidity` (L182–196) passes the caller-supplied `owner` directly to `LiquidityLib.addLiquidity` with no access restriction. The NatSpec at L147 documents this as an intentional "operator pattern," but no consent mechanism exists — any EOA or contract can add shares to any position.

Inside `LiquidityLib.addLiquidity` (L72–121), shares are credited to `positionBinShares[_positionBinKey(owner, salt, bin)]` regardless of who the caller is. The only guard is:

```solidity
uint256 newUserShares = userShares + sharesToAdd;
if (newUserShares < ctx.minimalMintableLiquidity) revert MinimalLiquidity(...);
```

Since the victim already holds `X >= minimalMintableLiquidity`, adding `delta = 1` satisfies `X + 1 >= minimalMintableLiquidity` and passes.

In `removeLiquidity` (L199–202), the check is:

```solidity
uint256 newUserShares = userShares - sharesToRemove; // = (X+1) - X = 1
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

The victim's partial withdrawal of `X` shares reverts. `removeLiquidity` enforces `msg.sender == owner` (L206), so only the victim can remove — but the attacker has already corrupted the share count. The `DepositAllowlistExtension` does not mitigate this: its `beforeAddLiquidity` hook checks `allowedDepositor[pool][owner]`, so any owner who already has a position (and is thus allowlisted) remains vulnerable.

The callback `metricOmmModifyLiquidityCallback` is invoked on `msg.sender` (the attacker), so the attacker pays the token cost for the dust shares — near-zero for bins with large `binTotalShares`.

## Impact Explanation
The direct impact is broken partial withdrawal for any targeted LP position. Smart contract LPs (vaults, routers, liquidity managers) that compute and store a fixed share amount to remove will have their `removeLiquidity` calls permanently revert until upgraded or the removal amount is adjusted. The attack is repeatable: after the victim adjusts their removal amount, the attacker adds another dust amount. The victim's tokens are not stolen — they receive the attacker's deposited tokens on full withdrawal — but the core partial-withdrawal flow is rendered unusable for the targeted position. This meets the "broken core pool functionality causing unusable withdraw/liquidity flows" threshold.

## Likelihood Explanation
No privilege is required. Any EOA or contract can call `addLiquidity` with any `owner` address. The position key `keccak256(abi.encode(owner, salt, bin))` is fully predictable from on-chain data. Attack cost is approximately 1 wei per token for bins with high total shares. The attack is repeatable after each victim adjustment, sustaining the grief indefinitely.

## Recommendation
Add an access control check in `MetricOmmPool.addLiquidity` requiring `msg.sender == owner`, mirroring the guard already present in `removeLiquidity`. Alternatively, implement an operator-approval mechanism (similar to ERC-721 `setApprovalForAll`) so that only the owner or an approved operator can add liquidity on behalf of an owner:

```solidity
// In MetricOmmPool.addLiquidity, before calling LiquidityLib:
if (msg.sender != owner) revert NotPositionOwner();
```

## Proof of Concept
1. Victim holds `X = minimalMintableLiquidity` shares in bin `0`, salt `0`.
2. Attacker (a contract implementing `IMetricOmmModifyLiquidityCallback`) calls `pool.addLiquidity(owner=victim, salt=0, deltas={binIdxs:[0], shares:[1]}, ...)`.
   - `newUserShares = X + 1 >= minimalMintableLiquidity` → passes the add-side guard.
   - Attacker pays ≈ 1 wei of each token via `metricOmmModifyLiquidityCallback`.
3. Victim calls `pool.removeLiquidity(owner=victim, salt=0, deltas={binIdxs:[0], shares:[X]}, ...)`.
   - `newUserShares = (X+1) - X = 1`.
   - `1 > 0 && 1 < minimalMintableLiquidity` → **REVERTS** with `MinimalLiquidity`.
4. Victim is forced to remove all `X + 1` shares or none.
5. Attacker repeats step 2 after victim adjusts, sustaining the grief indefinitely.
