### Title
Unsupported `ForeignChainRpcRequest::Ton` in Node Causes Permanent Request Timeout and Presignature Drain - (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

The MPC node's `execute_foreign_chain_request` function unconditionally bails on `ForeignChainRpcRequest::Ton` with a hard-coded error, while the on-chain contract fully recognizes `ForeignChain::Ton` as a valid whitelistable chain and the contract-level sandbox test suite explicitly expects TON verification to succeed. Once TON is legitimately whitelisted through the threshold-vote governance mechanism, every TON `verify_foreign_transaction` request accepted by the contract will permanently time out at the node layer, consuming a presignature from the shared `ForeignTx` domain on each attempt.

---

### Finding Description

**Root cause — node code:**

In `execute_foreign_chain_request`, the `Ton` arm is a hard bail with no inspector path:

```rust
dtos::ForeignChainRpcRequest::Ton(_request) => {
    bail!("ForeignChainRpcRequest::Ton is unsupported")
}
``` [1](#0-0) 

**Contract side — TON is a fully recognized chain variant:**

`ForeignChain::Ton` is a first-class enum member, serializable and deserializable over the wire: [2](#0-1) 

The contract's `verify_foreign_transaction` entry point accepts any request whose chain is in the on-chain supported set — it has no knowledge of whether the node can actually process it: [3](#0-2) 

**Governance path — TON can be whitelisted:**

Any threshold of participants can vote TON into the `ForeignChainRpcWhitelist` via `vote_update_foreign_chain_providers`. Once threshold votes agree, TON is stored in `AllowedProviders` and becomes part of the available set: [4](#0-3) 

**Presignature consumed before the bail:**

In `make_verify_foreign_tx_leader`, a presignature is irrevocably taken from the domain store at line 63 (`take_owned()`), *before* `execute_foreign_chain_request` is called at line 73–78. When the bail propagates via `?`, the presignature is already gone: [5](#0-4) 

**Timeout path — deposit consumed, request fails:**

When no node ever calls `respond_verify_foreign_tx`, the yield expires. `return_verify_foreign_tx_and_clean_state_on_success` receives an error and chains a call to `fail_on_timeout`, which panics with `RequestError::Timeout`. The user's transaction fails and the deposit is not refunded: [6](#0-5) 

**Sandbox test confirms the contract expects TON to succeed:**

The contract-level test suite includes TON in the `verify_foreign_transaction__should_succeed` parameterized test, confirming the contract interface was designed to accept TON requests: [7](#0-6) 

This test passes because it manually constructs the node response — it does not exercise the node's `execute_foreign_chain_request`. The discrepancy between the passing contract test and the node's hard bail is the core inconsistency.

---

### Impact Explanation

Once TON is whitelisted through governance:

1. **Permanent request lifecycle failure**: Every TON `verify_foreign_transaction` request accepted by the contract will time out. The contract's invariant — "a chain in the available set is serviceable" — is permanently violated for TON.
2. **Presignature drain**: Each TON request consumes one presignature from the shared `ForeignTx` domain before the bail. Presignatures are a finite, computationally expensive resource shared across all foreign-chain verification. Sustained TON submissions drain the pool and degrade liveness for all other chains (Bitcoin, Starknet, etc.).
3. **User funds lost**: Gas fees for each timed-out transaction are unrecoverable. The 1 yoctoNEAR deposit is also consumed.

This breaks the production safety invariant that accepted requests are fulfillable, and the accounting invariant that presignatures are only consumed when they can produce a signature.

---

### Likelihood Explanation

- `ForeignChain::Ton` is a first-class enum variant in the public ABI, and the sandbox test suite explicitly includes it in the "should succeed" test. This strongly signals that governance participants will vote TON into the whitelist, believing it is supported end-to-end.
- Once TON is whitelisted and threshold participants register it as covered, **any unprivileged user** can trigger the failure by submitting a `verify_foreign_transaction` with a TON request — no special access is required beyond the 1 yoctoNEAR deposit.
- The failure is silent from the governance perspective: the contract accepts the request, the node logs an internal error, and the user sees only a timeout. There is no on-chain signal that TON is broken at the node layer.

---

### Recommendation

1. **Implement TON support** in `execute_foreign_chain_request` with a real inspector, or
2. **Remove `ForeignChain::Ton` from the public ABI** and the sandbox test until node support is complete, or
3. **Add a contract-level guard** in `vote_update_foreign_chain_providers` that rejects whitelisting of chains not yet supported by the node binary (requires a node-reported capability set), or
4. **At minimum**, add a node-startup validation that panics if a whitelisted chain has no handler in `execute_foreign_chain_request`, making the misconfiguration visible before requests are accepted.

The same issue applies to `ForeignChainRpcRequest::Ethereum` and `ForeignChainRpcRequest::Solana`, which also unconditionally bail. [8](#0-7) 

---

### Proof of Concept

1. Threshold participants call `vote_update_foreign_chain_providers` with `ForeignChain::Ton` and a valid `ChainEntry`. After threshold votes agree, TON is stored in `AllowedProviders`.
2. Threshold participants call `register_foreign_chains_config` including `ForeignChain::Ton`. TON enters the available set.
3. An unprivileged user calls `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "domain_id": <foreign_tx_domain>,
       "payload_version": "V1",
       "request": { "Ton": { "tx_id": "...", "extractors": [...] } }
     }
   }
   ```
   with 1 yoctoNEAR attached.
4. The contract passes the `get_supported_foreign_chains()` check and enqueues a yield, consuming the deposit.
5. Each MPC node's leader calls `make_verify_foreign_tx_leader`:
   - Line 63: `presignature_store.take_owned()` — presignature is consumed.
   - Line 73–78: `execute_foreign_chain_request` hits the `Ton` arm and returns `Err("ForeignChainRpcRequest::Ton is unsupported")`.
   - The `?` propagates the error; the presignature is permanently lost.
6. No node calls `respond_verify_foreign_tx`. The yield expires.
7. `return_verify_foreign_tx_and_clean_state_on_success` receives `Err(_)`, calls `fail_on_timeout`.
8. The user's transaction fails with `RequestError::Timeout`. Deposit is not refunded.
9. Repeat from step 3 to drain the presignature pool.

    pub fn return_verify_foreign_tx_and_clean_state_on_success(
        &mut self,
        request: VerifyForeignTransactionRequest,
        #[callback_result] response: Result<VerifyForeignTransactionResponse, PromiseError>,
    ) -> PromiseOrValue<VerifyForeignTransactionResponse> {
        match response {
            Ok(response) => PromiseOrValue::Value(response),
            Err(_) => {
                pending_requests::pop_oldest_pending_yield(
                    &mut self.pending_verify_foreign_tx_requests,
                    &request,
                );
                let fail_on_timeout_gas = Gas::from_tgas(self.config.fail_on_timeout_tera_gas);
                let promise = Promise::new(env::current_account_id()).function_call(
                    method_names::FAIL_ON_TIMEOUT.to_string(),
                    vec![],
                    NearToken::from_near(0),
                    fail_on_timeout_gas,
                );
                near_sdk::PromiseOrValue::Promise(promise.as_return())
            }
        }
    }

    #[private]
    pub fn fail_on_timeout() {
