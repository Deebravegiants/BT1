### Title
Clamped `admission_gas` Used for Admission Check but Original `gas` Used for Budget Deduction in `try_forward` - (File: `runtime/runtime/src/congestion_control.rs`)

### Summary
In `ReceiptSinkV2::try_forward`, when `ProtocolFeature::ClampOutgoingGasAdmission` is enabled (protocol version 85), the code correctly computes a clamped `admission_gas = gas.min(allowed_shard_outgoing_gas)` for the admission gate check, but then deducts the original uncapped `gas` from `forward_limit.gas`. This is the exact same "computed-but-unused capped value" bug class as the VaultManager report: the capped value governs the gate but the uncapped value is applied to the budget, over-consuming it.

### Finding Description

In `runtime/runtime/src/congestion_control.rs`, `ReceiptSinkV2::try_forward`:

```rust
let admission_gas = if ProtocolFeature::ClampOutgoingGasAdmission
    .enabled(apply_state.current_protocol_version)
{
    gas.min(apply_state.config.congestion_control_config.allowed_shard_outgoing_gas)
} else {
    gas
};

if forward_limit.gas >= admission_gas && forward_limit.size >= size {
    outgoing_receipts.push(receipt);
    forward_limit.gas = forward_limit.gas.saturating_sub(gas);  // BUG: uses `gas`, not `admission_gas`
    forward_limit.size -= size;
    ...
}
``` [1](#0-0) 

The intent of `ClampOutgoingGasAdmission` is to allow the "allowed shard" to forward receipts with high attached gas to a fully-congested shard by clamping the gas used for the admission check to `allowed_shard_outgoing_gas` (1 PGAS). However, once the receipt passes the gate, the full unclamped `gas` is subtracted from `forward_limit.gas` instead of `admission_gas`.

**Concrete scenario:**
- `forward_limit.gas = 2 PGAS` (e.g., near-congested shard, small remaining budget)
- `allowed_shard_outgoing_gas = 1 PGAS`
- Receipt with `gas = 5 PGAS` (user-controlled via `promise_batch_action_function_call`)

Without the bug: `admission_gas = 5 PGAS`, check `2 PGAS >= 5 PGAS` → fails, receipt stays buffered.

With the bug: `admission_gas = 1 PGAS`, check `2 PGAS >= 1 PGAS` → passes, receipt forwarded, then `forward_limit.gas = saturating_sub(2 PGAS, 5 PGAS) = 0`. The remaining 1 PGAS of budget that should have been available for subsequent receipts is silently consumed.

### Impact Explanation

The corrupted value is `forward_limit.gas` — the per-chunk outgoing gas budget for a target shard. After one high-gas receipt is forwarded, the budget is zeroed out instead of being reduced by only `admission_gas`. All subsequent receipts in the same chunk that could have been forwarded within the remaining budget are instead left in the outgoing buffer. This causes:

1. **Incorrect congestion state**: `own_congestion_info.buffered_receipts_gas` remains higher than it should be after the chunk, because fewer receipts were drained from the buffer.
2. **Elevated congestion level reported to other shards**: The next block's `CongestionInfo` reflects a higher buffered gas than the protocol intended, causing other shards to further reduce their forwarding to this shard.
3. **Reduced receipt throughput**: Receipts are delayed in the outgoing buffer by additional chunks, degrading liveness of cross-shard transactions. [2](#0-1) 

### Likelihood Explanation

`ClampOutgoingGasAdmission` is assigned to protocol version 85, which is the current stable version (`STABLE_PROTOCOL_VERSION = 86` means version 85 features are active). [3](#0-2) 

Any unprivileged user can create a receipt with high attached gas (up to `max_total_prepaid_gas = 300 PGAS`) via a `FunctionCallAction`. When such a receipt is routed to a shard with a small-but-nonzero outgoing gas budget (i.e., `allowed_shard_outgoing_gas <= forward_limit.gas < gas`), the bug triggers on every chunk that drains the outgoing buffer. This is a normal operating condition during congestion events. [4](#0-3) 

### Recommendation

Replace the deduction on line 454 to use `admission_gas` instead of `gas`:

```rust
forward_limit.gas = forward_limit.gas.saturating_sub(admission_gas);
```

This ensures the budget is consumed by the same clamped amount that was used for the admission check, matching the intent of `ClampOutgoingGasAdmission`.

### Proof of Concept

1. Deploy a contract on shard A that creates a cross-shard receipt to shard B with `gas = max_total_prepaid_gas` (300 PGAS).
2. Induce partial congestion on shard B such that `forward_limit.gas` for shard A → shard B is between `allowed_shard_outgoing_gas` (1 PGAS) and `gas` (300 PGAS), e.g., 2 PGAS.
3. The receipt passes the admission check (`2 PGAS >= 1 PGAS`), is forwarded, but `forward_limit.gas` is set to 0 via `saturating_sub(300 PGAS)` instead of 1 PGAS.
4. All other receipts in the outgoing buffer for shard B in the same chunk fail the admission check (`0 >= 1 PGAS` is false) and remain buffered.
5. Observe that `own_congestion_info.buffered_receipts_gas` is higher than expected, and the congestion level reported in the next block is elevated beyond what the protocol intends. [5](#0-4)
