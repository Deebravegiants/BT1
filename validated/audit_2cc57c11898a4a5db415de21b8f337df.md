Audit Report

## Title
FIFO Withdrawal Queue Blocking Causes Temporary Freeze of User rsETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager` admits withdrawal requests based on `getTotalAssetDeposits` (protocol-wide, including EigenLayer-deployed assets) but unlocks them using only `unstakingVault.balanceOf` (physically returned assets). The unlock loop breaks on the first head-of-queue request it cannot cover, and there is no `cancelWithdrawal` function, leaving all subsequent users' rsETH frozen in the contract with no user-accessible exit path.

## Finding Description
**Admission check (L170):** `getAvailableAssetAmount` computes `totalAssets - assetsCommitted` where `totalAssets = lrtDepositPool.getTotalAssetDeposits(asset)` — the full protocol balance including assets still deployed in EigenLayer strategies. [1](#0-0) 

This allows a user to queue a withdrawal for an amount far exceeding what is physically present in `LRTUnstakingVault`.

**Unlock loop (L790–815):** `_unlockWithdrawalRequests` iterates from `nextLockedNonce[asset]` in strict FIFO order. When the head-of-queue request's `payoutAmount` exceeds `availableAssetAmount` (sourced from the vault balance), the loop `break`s immediately and `nextLockedNonce` is not advanced. [2](#0-1) 

**Completion check (L707):** `_processWithdrawalCompletion` reverts with `WithdrawalLocked` for any nonce ≥ `nextLockedNonce[asset]`. Because `nextLockedNonce` is stuck at the blocking request's nonce, every subsequent user's nonce is ≥ that value and is permanently blocked from completing. [3](#0-2) 

**No cancel path:** A grep across all contracts confirms there is no `cancelWithdrawal` function. The user's rsETH was transferred into the contract at `initiateWithdrawal` (L166) and cannot be reclaimed. [4](#0-3) 

**Concrete exploit path:**
1. Protocol holds 100 ETH total: 80 in EigenLayer, 20 in vault.
2. User A calls `initiateWithdrawal` for rsETH ≈ 80 ETH. Admission: `100 - 0 = 100 ≥ 80` → passes. `assetsCommitted = 80`. Nonce = 0.
3. User B calls `initiateWithdrawal` for rsETH ≈ 10 ETH. Admission: `100 - 80 = 20 ≥ 10` → passes. `assetsCommitted = 90`. Nonce = 1.
4. Operator calls `unlockQueue`; vault has 20 ETH. Loop: nonce 0 → `payoutAmount = 80 > 20` → `break`. `nextLockedNonce` stays at 0.
5. User B calls `completeWithdrawal`: `popFront()` → nonce 1; `1 >= 0` → `WithdrawalLocked` revert. User B's rsETH remains locked.

## Impact Explanation
User B's rsETH is locked inside `LRTWithdrawalManager` with no user-accessible exit. The freeze duration is unbounded from User B's perspective — it depends entirely on when the operator unstakes sufficient assets from EigenLayer to cover User A's larger request. This is **temporary freezing of funds** (Medium severity per the allowed impact scope). The funds are not permanently lost but are inaccessible to the user for an indeterminate period outside their control.

## Likelihood Explanation
Medium. The gap between `getTotalAssetDeposits` (admission) and `unstakingVault.balanceOf` (unlock) is the normal operational state of the protocol — the majority of assets are routinely deployed in EigenLayer. Any large withdrawal request queued ahead of smaller ones triggers this condition. No adversarial intent is required; ordinary concurrent usage by independent users is sufficient.

## Recommendation
1. **Add a `cancelWithdrawal` function** that allows a user to remove their own still-locked (nonce ≥ `nextLockedNonce`) withdrawal request and receive their rsETH back. This requires decrementing `assetsCommitted[asset]` by `request.expectedAssetAmount` and returning the rsETH via `safeTransfer`.
2. **Alternatively**, modify `_unlockWithdrawalRequests` to `continue` (skip) rather than `break` when a request cannot be covered, so smaller subsequent requests are not blocked by a single large head-of-queue entry. Note this changes FIFO semantics and may require additional accounting.

function test_FIFOQueueBlocking() public {
    // Setup: 100 ETH total assets, 20 ETH in unstaking vault, 80 ETH in EigenLayer

    vm.prank(operator);

    // nextLockedNonce[ETH] == 0 (unchanged)

    vm.prank(userB);
    vm.expectRevert(ILRTWithdrawalManager.WithdrawalLocked.selector);
    withdrawalManager.completeWithdrawal(ETH, "");
    // nonce 1 >= nextLockedNonce 0 → WithdrawalLocked
    // User B's rsETH remains locked; no cancel function exists
}
