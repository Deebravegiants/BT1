### Title
Proposal Deposits Permanently Lost When `do_update` Clears All Pending Proposals - (`File: crates/contract/src/update.rs`)

### Summary

When `vote_update` reaches threshold and `do_update` executes, it unconditionally clears **all** pending update proposals and their associated votes. However, the storage deposits attached by proposers of the non-executed proposals are never refunded. Because `UpdateEntry` does not record the depositor's account ID or the deposit amount, there is no mechanism to return these funds. Proposers permanently lose their deposits whenever any other update reaches threshold.

### Finding Description

`propose_update` requires a deposit proportional to the storage consumed by the proposal:

```rust
// crates/contract/src/lib.rs
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// ...
let id = self.proposed_updates.propose(update);
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

The deposit is consumed by the contract but **not stored** in `UpdateEntry`:

```rust
// crates/contract/src/update.rs
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,   // ← no depositor AccountId, no deposit amount
}
```

When any update reaches threshold, `do_update` clears every entry:

```rust
// crates/contract/src/update.rs
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ... no refund loop
}
```

Because neither the depositor's `AccountId` nor the deposit amount is stored in `UpdateEntry`, there is no way to issue refunds to the proposers of the cleared entries. Their NEAR is permanently locked in the contract.

The deposit required for a 1 MB contract binary is approximately:

```
bytes_used ≈ sizeof(UpdateEntry) + 128 × sizeof(AccountId) + code.len()
           ≈ 40 + 8192 + 1_000_000 ≈ 1_008_232 bytes
deposit    ≈ 1_008_232 × 10^19 yoctoNEAR ≈ 10 NEAR
```

### Impact Explanation

This is a **Medium** impact finding: balance/accounting invariant violation that breaks production safety invariants. Any participant who proposes an update while another proposal is pending will permanently lose their deposit (potentially ~10 NEAR for a full contract binary) when the other proposal is executed. The invariant "if your proposal is not executed, your deposit is returned" is broken. The funds are not stolen by an attacker — they are permanently locked in the contract with no recovery path.

### Likelihood Explanation

Realistic in production. Multiple participants can independently propose updates (e.g., one proposes a code upgrade, another proposes a config change). When the first proposal to reach threshold is executed, all other proposals are cleared and their deposits are lost. No adversarial action is required — this is the normal operation of the contract under concurrent proposals.

### Recommendation

Store the depositor's `AccountId` and the deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) depositor: AccountId,   // add
    pub(super) deposit: NearToken,     // add
}
```

Then, in `do_update`, iterate over the entries being cleared and issue refund promises before clearing:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Refund deposits for all non-executed proposals
    for (_, cleared_entry) in self.entries.iter() {
        Promise::new(cleared_entry.depositor.clone())
            .transfer(cleared_entry.deposit)
            .detach();
    }
    self.entries.clear();
    self.vote_by_participant.clear();
    // ...
}
```

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR deposit. Proposal ID = 0 is stored.
2. Participant B calls `propose_update` with a different contract binary, attaching ~10 NEAR deposit. Proposal ID = 1 is stored.
3. Participants vote for proposal 1 until threshold is reached.
4. `vote_update(id=1)` triggers `do_update(id=1)`, which:
   - Removes entry 1 (executed)
   - Calls `self.entries.clear()` — removes entry 0 (Participant A's proposal)
   - Calls `self.vote_by_participant.clear()`
   - Issues no refund to Participant A
5. Participant A's ~10 NEAR deposit is permanently locked in the contract with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
