### Title
Missing Minimum Output Amount (Slippage) Check in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPoolV3.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol, RSETHPoolV2ExternalBridge.sol)

### Summary

All L2 pool `deposit()` functions lack a caller-supplied minimum rsETH output parameter. The oracle-derived exchange rate used to compute the rsETH amount can shift between transaction submission and execution, causing depositors to receive fewer rsETH tokens than they anticipated with no on-chain protection.

### Finding Description

Every L2 pool contract exposes two public `deposit()` entry points — one for native ETH and one for supported ERC-20 tokens. Both compute the rsETH amount solely from the live oracle rate at execution time and immediately transfer/mint that amount to the caller, with no `minRsETHAmount` guard:

```solidity
// RSETHPoolV3ExternalBridge.sol – ETH deposit (line 366)
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);          // ← no minimum check
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}

// RSETHPoolV3ExternalBridge.sol – token deposit (line 390)
function deposit(address token, uint256 amount, string memory referralId)
    external nonReentrant whenNotPaused
    onlySupportedToken(token) limitDailyMint(amount, token)
{
    if (amount == 0) revert InvalidAmount();
    IERC20(token).safeTransferFrom(msg.sender, address(this), amount);

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
    feeEarnedInToken[token] += fee;
    wrsETH.mint(msg.sender, rsETHAmount);          // ← no minimum check
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId, token);
}
```

The same pattern is present in `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPoolV3.sol`, `RSETHPool.sol`, `RSETHPoolNoWrapper.sol`, and `RSETHPoolV2ExternalBridge.sol`.

By contrast, the L1 `LRTDepositPool` explicitly enforces a caller-supplied minimum:

```solidity
// LRTDepositPool.sol – _beforeDeposit (line 667)
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

The L1 contract accepts `minRSETHAmountExpected` in both `depositETH` and `depositAsset`, making the protection intentional and well-understood by the protocol designers. The L2 pool contracts simply omit it.

### Impact Explanation

**Low — Contract fails to deliver promised returns, but does not lose value.**

A depositor who previews the rate off-chain (e.g., via `viewSwapRsETHAmountAndFee`) and then submits a transaction may receive materially fewer rsETH tokens if the oracle rate is updated upward (rsETH appreciates) before the transaction is mined. The depositor's ETH/LST is accepted by the pool at the new, less-favorable rate. The user cannot enforce the rate they observed; there is no revert path. The ETH value is not stolen — it remains in the pool — but the user receives fewer rsETH than the protocol implicitly promised at preview time.

### Likelihood Explanation

The rsETH oracle rate is updated periodically. On L2 networks with variable block times and mempool congestion, a transaction can sit pending long enough for one or more oracle updates to occur. Any depositor who previews the rate and submits a transaction is exposed. No special privileges or external compromise are required; the path is fully reachable by any unprivileged depositor.

### Recommendation

Add a `minRsETHAmount` parameter to both `deposit()` overloads in every L2 pool contract, mirroring the pattern already used in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert MinimumAmountToReceiveNotMet();
    ...
}
```

### Proof of Concept

1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and sees she will receive `X` rsETH.
2. Alice submits `deposit{value: 1 ether}("ref")` to `RSETHPoolV3ExternalBridge`.
3. Before Alice's transaction is mined, the rsETH oracle rate increases (rsETH is now worth more ETH per unit).
4. Alice's transaction executes; `viewSwapRsETHAmountAndFee` now returns `X - delta` rsETH.
5. `wrsETH.mint(msg.sender, rsETHAmount)` mints the reduced amount with no revert.
6. Alice receives fewer rsETH than she expected and has no recourse.
