Audit Report

## Title
Donation Attack via Direct ETH/LST Transfer Inflates `getTotalAssetDeposits()`, Triggering Deposit Limit DoS - (File: contracts/LRTDepositPool.sol)

## Summary
An unprivileged attacker can send ETH directly to `LRTDepositPool` via its open `receive()` function, or transfer LST tokens directly to the contract address, inflating the raw on-chain balances that `getTotalAssetDeposits()` reads. If the inflated total exceeds `depositLimitByAsset`, every subsequent call to `depositETH` or `depositAsset` reverts with `MaximumDepositLimitReached`, blocking all new deposits until an admin raises the cap.

## Finding Description
`LRTDepositPool` exposes an unconditional `receive()` at line 58:

```solidity
receive() external payable { }
```

`getETHDistributionData()` reads the raw contract balance at line 480:

```solidity
ethLyingInDepositPool = address(this).balance;
```

`getAssetDistributionData()` reads the raw ERC-20 balance at line 444:

```solidity
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
```

Both values flow into `getTotalAssetDeposits()` (lines 385–397), which is consumed by `_checkIfDepositAmountExceedesCurrentLimit()` (lines 676–682):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

`_beforeDeposit()` (lines 661–663) reverts if this returns true:

```solidity
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
```

For ETH specifically, when `depositETH` is called the `msg.value` is already included in `address(this).balance` before `_beforeDeposit` runs, so the check is `totalAssetDeposits > limit` (not `+ amount`). A donation of just `(limit - currentTVL + 1)` wei is sufficient to make this condition permanently true for all subsequent depositors, with no minimum deposit size required to trigger the revert.

There are no guards preventing direct ETH sends or ERC-20 transfers to the contract address. The donated assets are not rejected or quarantined; they are silently counted as protocol TVL.

## Impact Explanation
All new ETH or LST deposits are blocked for as long as the donated balance remains in the contract and the admin has not raised `depositLimitByAsset`. This is a **Medium – Temporary freezing of funds**: new depositors cannot enter the protocol, and resolution requires privileged admin action. Existing depositors are unaffected, and the donated assets inflate rsETH's price, benefiting current holders.

## Likelihood Explanation
The attack requires no special permissions and is executable by any externally-owned account. The only cost is the gap between current TVL and the deposit cap, which can be negligible when the protocol is near its limit. The attack is repeatable: if the admin raises the cap, the attacker can donate again. No victim mistake or external dependency is required.

## Recommendation
Decouple deposit-limit accounting from raw on-chain balances. Introduce an explicit `depositedAmount[asset]` storage variable incremented only through `depositETH`/`depositAsset` and decremented through the withdrawal path. Use this variable instead of `address(this).balance` and `IERC20(asset).balanceOf(address(this))` inside `_checkIfDepositAmountExceedesCurrentLimit`. Alternatively, add a permissioned `sweepDonations()` function that forwards unsolicited balances to a NodeDelegator so they are properly accounted for and cannot be used to grief the limit.

## Proof of Concept
```
Initial state:
  depositLimitByAsset[ETH] = 1000 ETH
  address(lrtDepositPool).balance = 990 ETH  (10 ETH headroom)

Step 1 – Attacker donates:
  (bool ok,) = address(lrtDepositPool).call{value: 11 ether}("");
  // receive() accepts unconditionally; balance is now 1001 ETH

Step 2 – Verify inflation:
  getETHDistributionData().ethLyingInDepositPool == 1001 ETH
  getTotalAssetDeposits(ETH_TOKEN) == 1001 ETH

Step 3 – Victim attempts deposit:
  lrtDepositPool.depositETH{value: 1 ether}(0, "");
  // _beforeDeposit → _checkIfDepositAmountExceedesCurrentLimit
  // totalAssetDeposits (1001+1=1002) > 1000 → true
  // → revert MaximumDepositLimitReached()

Foundry test sketch:
  function testDonationDoS() public {
      // fund attacker and victim
      vm.deal(attacker, 11 ether);
      vm.deal(victim, 1 ether);
      // set limit at 1000 ETH, seed 990 ETH via legitimate deposits
      // attacker donates
      vm.prank(attacker);
      (bool s,) = address(pool).call{value: 11 ether}("");
      assertTrue(s);
      // victim deposit reverts
      vm.prank(victim);
      vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
      pool.depositETH{value: 1 ether}(0, "");
  }
