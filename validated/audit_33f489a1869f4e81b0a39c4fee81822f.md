Looking at the `setFeeCaps` function in `MetricOmmPoolFactory.sol` and how it interacts with per-pool admin fee state, there is a direct analog to the external bug.

---

### Title
`setFeeCaps` Does Not Clamp Existing Per-Pool Admin Fees, Allowing Admin Fees to Exceed Newly Lowered Caps — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

When the factory owner calls `setFeeCaps` to lower `maxAdminSpreadFeeE6` or `maxAdminNotionalFeeE8`, the function clamps the global default protocol fees but silently leaves all existing per-pool `poolFeeConfig[pool].adminSpreadFeeE6` / `adminNotionalFeeE8` values above the new cap. Every subsequent swap on those pools charges users an admin fee that exceeds the governance-intended ceiling.

---

### Finding Description

`setFeeCaps` updates the four cap variables and then clamps only the two **default** protocol-fee globals: [1](#0-0) 

```solidity
maxProtocolSpreadFeeE6  = newMaxProtocolSpreadFeeE6;
maxAdminSpreadFeeE6     = newMaxAdminSpreadFeeE6;      // cap lowered
maxProtocolNotionalFeeE8 = newMaxProtocolNotionalFeeE8;
maxAdminNotionalFeeE8   = newMaxAdminNotionalFeeE8;    // cap lowered

if (spreadProtocolFeeE6 > newMaxProtocolSpreadFeeE6) { // ← clamped
    spreadProtocolFeeE6 = newMaxProtocolSpreadFeeE6;
}
if (protocolNotionalFeeE8 > newMaxProtocolNotionalFeeE8) { // ← clamped
    protocolNotionalFeeE8 = newMaxProtocolNotionalFeeE8;
}
// poolFeeConfig[pool].adminSpreadFeeE6 / adminNotionalFeeE8 — NOT touched
```

Per-pool admin fees stored in `poolFeeConfig` are **never clamped** here. The only place they are clamped is inside `setPoolProtocolFee`, which the factory owner must call individually for every pool: [2](#0-1) 

Until that per-pool call is made, the pool continues to operate with the stale above-cap admin fee because `setPoolFees` on the pool was never updated: [3](#0-2) 

`setPoolAdminFees` correctly blocks the pool admin from *raising* the fee above the new cap: [4](#0-3) 

But the existing above-cap fee is never removed, so the invariant `poolFeeConfig[pool].adminSpreadFeeE6 ≤ maxAdminSpreadFeeE6` is broken for every pool that had a higher admin fee before the cap was lowered.

---

### Impact Explanation

Every swap executed on an affected pool deducts an admin fee above the governance-intended ceiling. The excess fee is transferred to `poolAdminFeeDestination`, constituting a direct, ongoing loss of swap output for traders. The magnitude equals `(old_adminFee − new_cap) × notional_per_swap` on every trade until the factory owner manually calls `setPoolProtocolFee` for each affected pool.

---

### Likelihood Explanation

The factory owner is expected to use `setFeeCaps` as a recurring governance lever (no one-time guard exists). Any time the owner tightens the admin fee cap — a normal protective action — all pools whose admin fees were set above the new cap silently continue overcharging. With multiple pools deployed, the owner must remember to follow up with a per-pool `setPoolProtocolFee` call for each one; missing even one pool leaves it overcharging indefinitely.

---

### Recommendation

Inside `setFeeCaps`, after updating the cap variables, add enforcement analogous to what `setPoolProtocolFee` already does — either:

1. **Emit a bounded-iteration clamp** over all registered pools (feasible if `nextPoolIdx` is small), calling `setPoolFees` on each pool whose admin fee exceeds the new cap; or
2. **Enforce the cap at collection time**: in `collectFees` / `collectPoolFees`, clamp the admin fee component to `min(stored, maxAdminCap)` before distributing, so no above-cap amount ever leaves the pool regardless of when `setFeeCaps` was called.

The minimal guard analogous to the external report's recommendation:

```solidity
// inside setFeeCaps, after updating caps:
require(
    poolFeeConfig[pool].adminSpreadFeeE6 <= newMaxAdminSpreadFeeE6,
    "existing admin fee exceeds new cap"
);
```

(applied per-pool, or replaced by the lazy-clamp approach above).

---

### Proof of Concept

1. Pool admin calls `setPoolAdminFees(pool, 150_000, 0)` — valid, cap is 200_000. [5](#0-4) 

2. Factory owner calls `setFeeCaps(200_000, 100_000, 1_000_000, 1_000_000)` — lowers admin spread cap to 100_000. [6](#0-5) 

3. `poolFeeConfig[pool].adminSpreadFeeE6` remains **150_000**; `maxAdminSpreadFeeE6` is now **100_000**. Invariant broken.

4. Every swap on the pool charges a 150_000 (15%) admin spread fee. The governance-intended ceiling is 100_000 (10%). The extra 5% of every swap's notional is extracted to the pool admin's fee destination — a direct, per-swap loss for traders above the cap the factory owner intended to enforce.

5. `setPoolAdminFees` now blocks the admin from *changing* the fee (it would revert at line 414), but the existing 150_000 fee is never removed until the factory owner separately calls `setPoolProtocolFee(pool, ...)`. [7](#0-6)

  function setPoolAdminFees(address pool, uint24 newAdminSpreadFeeE6, uint24 newAdminNotionalFeeE8)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();

    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool)
      .collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
      );

    c.adminSpreadFeeE6 = newAdminSpreadFeeE6;
    c.adminNotionalFeeE8 = newAdminNotionalFeeE8;
    poolFeeConfig[pool] = c;

    IMetricOmmPoolFactoryActions(pool)
      .setPoolFees(c.protocolSpreadFeeE6 + c.adminSpreadFeeE6, c.protocolNotionalFeeE8 + c.adminNotionalFeeE8);
    emit PoolAdminSpreadFeeUpdated(pool, newAdminSpreadFeeE6);
    emit PoolAdminNotionalFeeUpdated(pool, newAdminNotionalFeeE8);
