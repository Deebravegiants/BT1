### Title
`nativeFee` ETH and directly-deposited ETH permanently locked in `OmniBridge` — no withdrawal path exists - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.sol` collects a `nativeFee` in ETH from every `initTransfer` caller and also exposes a bare `receive() external payable {}` fallback. Neither the accumulated `nativeFee` ETH nor any ETH sent directly via `receive()` can ever leave the contract: there is no `withdraw`, `rescue`, or equivalent function anywhere in the EVM contract suite.

### Finding Description
Two independent ETH-lock paths exist in `OmniBridge.sol`:

**Path 1 — `nativeFee` accumulation in `initTransfer`**

Every call to `initTransfer` requires the caller to include `nativeFee` wei in `msg.value`. The function computes `extensionValue` by subtracting `nativeFee` from `msg.value`:

- ETH transfer path: `extensionValue = msg.value - amount - nativeFee` [1](#0-0) 
- ERC-20 transfer path: `extensionValue = msg.value - nativeFee` [2](#0-1) 

Only `extensionValue` is forwarded downstream. In `OmniBridgeWormhole.initTransferExtension`, the Wormhole message fee is paid with `value` (= `extensionValue`): [3](#0-2) 

The `nativeFee` portion is never forwarded anywhere — it silently stays in the contract's ETH balance with no accounting entry and no withdrawal path.

**Path 2 — bare `receive()` fallback**

The contract exposes an unconditional payable fallback: [4](#0-3) 

Any ETH sent directly to the contract address (e.g., by a Wormhole refund, a relayer mistake, or any external caller) is accepted and permanently locked.

**No withdrawal function exists.** A full grep across all EVM contracts for `withdraw`, `rescue`, `sweep`, `recoverEth`, and `recoverNative` returns zero matches. 

### Impact Explanation
Every `initTransfer` call that includes a non-zero `nativeFee` permanently locks that ETH in the contract. Over the protocol's lifetime this accumulates into an irrecoverable pool of protocol fee value. Additionally, any ETH routed to the contract via `receive()` — including Wormhole message-fee refunds — is also permanently locked. This matches the allowed impact: **Critical — permanently unclaimable protocol value in fee flows**.

### Likelihood Explanation
`initTransfer` is the primary public bridge entry point. Relayers and users are expected to supply a non-zero `nativeFee` to incentivize relaying. Every such call contributes to the locked pool. The `receive()` path is also reachable by any external account. No privileged access is required; the loss is automatic and cumulative.

### Recommendation
1. Track accumulated `nativeFee` in a dedicated storage variable (e.g., `uint256 public accruedNativeFees`), incremented inside `initTransfer`.
2. Add an admin-gated withdrawal function:
   ```solidity
   function withdrawNativeFees(address payable to)
       external onlyRole(DEFAULT_ADMIN_ROLE) {
       uint256 amount = accruedNativeFees;
       accruedNativeFees = 0;
       (bool ok,) = to.call{value: amount}("");
       require(ok, "transfer failed");
   }
   ```
3. Either remove the bare `receive()` fallback or add a separate rescue function for ETH sent outside of normal protocol flows.

### Proof of Concept
1. Alice calls `initTransfer(tokenAddress, 1e18, 0, 0.01 ether, "alice.near", "")` sending `msg.value = 1e18 + 0.01 ether`.
2. `extensionValue = msg.value - 1e18 - 0.01 ether = 0`. Wormhole receives `0` wei. The `0.01 ether` `nativeFee` stays in `OmniBridge`.
3. After 10,000 such transfers at `0.01 ether` each, `100 ETH` of protocol fees are locked in the contract.
4. The protocol team attempts to withdraw — no function exists. The ETH is permanently unrecoverable without a contract upgrade.
