### Title
Missing `payload_hash`-to-Request Binding in `respond_verify_foreign_tx` Allows Forged Foreign-Chain Verification Attestation - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` is actually derived from the pending `request`'s chain data. A single Byzantine attested participant (below the signing threshold) can resolve any pending foreign-tx verification request with a `payload_hash` harvested from a completely different, previously-processed request, producing a forged on-chain attestation.

---

### Finding Description

The `respond_verify_foreign_tx` entry point accepts two caller-controlled arguments: a `VerifyForeignTransactionRequest` (used as the map key to locate the pending yield) and a `VerifyForeignTransactionResponse` containing a free-form `payload_hash` and a `signature`. [1](#0-0) 

The only cryptographic check performed is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
``` [2](#0-1) 

This confirms only that `signature` is a valid ECDSA signature over `payload_hash` under the domain's root public key. It does **not** confirm that `payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, observed_values }))` for the specific `request` being resolved.

Contrast this with the regular `respond` path, where the payload is embedded in the `SignatureRequest` itself and the contract verifies the signature against `request.payload` — a value the contract already holds and controls: [3](#0-2) 

For foreign-tx responses the `payload_hash` is not stored anywhere in the pending-request map; only the `ForeignChainRpcRequest` key is stored. The contract therefore has no reference value to compare against.

The design intent — that `payload_hash` must commit to both the original request and the MPC-observed extracted values — is documented in the SDK's `ForeignChainSignatureVerifier`: [4](#0-3) 

That check lives entirely off-chain in the SDK and is never enforced by the contract itself.

The `ForeignTxSignPayload` structure that should be the source of truth: [5](#0-4) 

---

### Impact Explanation

**High — Forged foreign-chain verification causing invalid bridge execution.**

A Byzantine attested participant executes the following steps without any threshold collusion:

1. Submit (or observe) any `verify_foreign_transaction(request_X)` that the honest MPC network processes, yielding a valid threshold signature `sig_X` over `hash_X = SHA-256(borsh(ForeignTxSignPayload { request_X, values_X }))`.
2. Wait for a victim to submit `verify_foreign_transaction(request_Y)` (e.g., a bridge deposit attestation for a different transaction).
3. Call `respond_verify_foreign_tx(request_Y, { payload_hash: hash_X, signature: sig_X })`.
4. The contract verifies `sig_X` is valid over `hash_X` — it is — and resolves the yield for `request_Y` with the fabricated response.
5. The victim's bridge contract receives a `VerifyForeignTransactionResponse` whose `payload_hash` commits to `request_X`'s data, not `request_Y`'s data.

Because the victim's contract cannot independently reconstruct the expected `payload_hash` (it does not know the MPC-observed `values`), it cannot detect the substitution. A bridge contract that trusts the MPC attestation will process the wrong transaction data — enabling double-spend or value inflation/deflation attacks.

---

### Likelihood Explanation

**Medium.** The attacker must be an attested participant (requires passing TEE attestation), which is a meaningful barrier. However, no threshold collusion is required — a single Byzantine node below the signing threshold is sufficient. The attacker only needs one legitimately-produced threshold signature (from any prior request on the same domain) to execute the substitution. In a live bridge deployment with continuous traffic, such signatures are routinely available.

---

### Recommendation

The contract must bind `response.payload_hash` to the pending `request` before accepting the response. Since the contract does not store the observed `values`, the binding must be enforced structurally. Two options:

1. **Recompute and compare on-chain**: Store the expected `payload_hash` alongside the pending yield at submission time (requires nodes to commit to a hash before signing — a protocol change).
2. **Embed the request in the signed payload and verify inclusion**: Require that `payload_hash` is a hash that provably includes the `ForeignChainRpcRequest` fields, and verify this on-chain by re-hashing the request portion and checking it is a prefix/component of `payload_hash`.

The minimal fix is to add a check analogous to what `ForeignChainSignatureVerifier::verify_signature` does in the SDK — recompute `expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request.clone(), values: ??? }))` — but this requires the contract to know `values`. The cleanest solution is to store a commitment to the expected hash at request-submission time, derived from the request alone (e.g., `SHA-256(borsh(request))`), and require that `payload_hash` starts with or is derived from that commitment.

    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {

// The contract only checks: is prior_sig valid over prior_payload_hash? YES.
// It does NOT check: does prior_payload_hash correspond to victim_request? NO CHECK.
contract.respond_verify_foreign_tx(
    victim_request,
    VerifyForeignTransactionResponse {
        payload_hash: prior_payload_hash, // hash of a DIFFERENT request
        signature: prior_sig,             // valid sig over that different hash

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
