### Title
`respond_verify_foreign_tx` Accepts Mismatched `payload_hash` Without Binding It to the Request - (File: `crates/contract/src/lib.rs`)

### Summary
The `respond_verify_foreign_tx` contract method verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but never verifies that `response.payload_hash` is actually the canonical hash of the submitted `request`. A single Byzantine attested participant (strictly below the signing threshold) who acts as the signing leader for one pending request can redirect the legitimately produced MPC signature to a *different* pending request, causing the contract to deliver a forged foreign-chain verification attestation to an unrelated user.

### Finding Description

The `respond_verify_foreign_tx` function performs three checks:

1. Caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`)
2. Protocol is in running/resharing state
3. The ECDSA signature in `response` is valid over `response.payload_hash` under the domain's root public key [1](#0-0) 

What it does **not** check is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayloadV1 { request: <the submitted request>, values: <observed values> }))`. The canonical payload structure is: [2](#0-1) 

Because the contract never binds the hash to the request, any valid `(payload_hash, signature)` pair produced by the MPC network for *any* request on the same domain can be submitted as the response to a *different* pending request. The contract will accept it, resolve the pending yield, and return the mismatched response to the original caller.

This is directly analogous to the external report's root cause: the `FullRangeHook` enforces a narrower tick range than `TickMath` actually allows, so values that fall in the gap pass the system's validity check but violate the hook's invariant. Here, the contract enforces only "signature is valid over this hash" but not "this hash corresponds to this request" — the gap between those two constraints is the attack surface.

The SDK-level verifier (`ForeignChainSignatureVerifier::verify_signature`) does perform the binding check: [3](#0-2) 

But this check lives in an optional off-chain SDK, not in the on-chain contract. The contract itself delivers the forged response unconditionally.

### Impact Explanation

A Byzantine leader node for request Y can:

1. Obtain the legitimately produced MPC signature `sig_Y` over `hash_Y = SHA-256(borsh({request_Y, values_Y}))` (it is the leader, so it assembles the final signature).
2. Call `respond_verify_foreign_tx(request = request_X, response = {payload_hash: hash_Y, signature: sig_Y})` where `request_X` is a *different* pending request.
3. The contract verifies `sig_Y` is valid over `hash_Y` under the domain key — this passes.
4. The contract resolves the pending yield for `request_X` and returns `{payload_hash: hash_Y, sig_Y}` to the user who submitted `request_X`.

The user of `request_X` receives what appears to be a valid MPC attestation, but it attests to a completely different foreign-chain transaction (tx Y, not tx X). Any bridge contract that does not independently recompute the expected `payload_hash` from the original request and compare it to `response.payload_hash` will accept this as proof that tx X was verified — enabling invalid bridge execution or double-spend conditions (e.g., crediting an inbound bridge transfer that was never actually finalized on the foreign chain, or crediting it twice by reusing the same signature for two different pending requests).

This matches the allowed High impact: *"forged foreign-chain verification… that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

- Requires a **single** Byzantine attested participant acting as the signing leader for one request — strictly below the signing threshold.
- Requires two concurrent pending `verify_foreign_transaction` requests on the same domain (common in any active bridge deployment).
- In production the TEE model is intended to prevent Byzantine nodes, but the vulnerability exists at the contract layer independently of TEE; a software-level TEE compromise or a transition/grace-period window suffices.
- Bridge contracts that use `ForeignChainSignatureVerifier::verify_signature` from the NEAR MPC SDK are protected; those that perform only raw signature verification are fully exposed.

### Recommendation

The contract cannot recompute the full `payload_hash` on-chain because it does not receive the extracted values. Two mitigations are possible:

**Option A (preferred):** Require the responding node to also submit the `Vec<ExtractedValue>` alongside the response. The contract recomputes `expected_hash = SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))` and asserts `expected_hash == response.payload_hash` before accepting the response.

**Option B (partial):** Commit to the request inside the signed payload in a way the contract can verify without the values — for example, prepend a domain-separated hash of the request to the payload so the contract can at least verify the request component of the hash.

Either option closes the gap between what the contract validates (signature over an arbitrary hash) and what it should validate (signature over the hash of *this specific request*).

### Proof of Concept

```
// Setup: two concurrent pending requests on the same ForeignTx domain
User A → verify_foreign_transaction(request_X)   // pending in contract
User B → verify_foreign_transaction(request_Y)   // pending in contract

// MPC nodes legitimately process request_Y
// Leader node obtains: sig_Y = MPC_sign(hash_Y) where hash_Y = SHA256(borsh({request_Y, [block_hash_Y]}))

// Byzantine leader for request_Y calls:
respond_verify_foreign_tx(
    request  = request_X,                          // ← different request
    response = {
        payload_hash: hash_Y,                      // ← hash for request_Y
        signature:    sig_Y,                       // ← valid sig over hash_Y
    }
)

// Contract checks:
// ✓ caller is attested participant
// ✓ sig_Y is valid over hash_Y under domain public key
// ✗ hash_Y == SHA256(borsh({request_X, ...}))  ← NOT CHECKED

// Result: User A's yield resolves with {payload_hash: hash_Y, sig_Y}
// User A's bridge contract receives a valid-looking attestation for tx Y
// presented as the response to their request for tx X.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
