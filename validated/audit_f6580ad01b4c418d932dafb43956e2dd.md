### Title
`respond_verify_foreign_tx()` Does Not Bind `payload_hash` to the Submitted `request` - (`File: crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx()` verifies that the submitted `response.signature` is a valid ECDSA signature over `response.payload_hash`, but it never checks that `response.payload_hash` is actually derived from the `request` argument. A single Byzantine MPC participant (below the signing threshold) can replay any previously produced, on-chain-visible signature over an unrelated `payload_hash` to resolve a live pending foreign-tx verification request with fabricated attestation data.

### Finding Description

`respond_verify_foreign_tx` performs two independent checks and then resolves the pending request:

1. **Signature validity**: `verify_ecdsa_signature(signature_response, &payload_hash, &secp_pk)` — confirms the signature is valid over the caller-supplied `payload_hash`.
2. **Request existence**: `resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &request, ...)` — confirms a pending yield exists for the caller-supplied `request`. [1](#0-0) 

What is **never checked** is the binding between the two: that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))` for the specific `request` being resolved.

The intended payload construction is: [2](#0-1) 

The `ForeignTxSignPayload` embeds the full `ForeignChainRpcRequest` (including `tx_id`), so signatures for different requests produce different hashes. However, the contract never enforces this derivation relationship on-chain.

### Impact Explanation

A Byzantine MPC participant (below the signing threshold) can:

1. Observe any previously produced, legitimate `respond_verify_foreign_tx` call on-chain. The `payload_hash` and `signature` are public.
2. Wait for a high-value pending `verify_foreign_transaction` request (`request_B`, e.g., a large bridge deposit for `tx_id_B`).
3. Call `respond_verify_foreign_tx(request_B, response { payload_hash: payload_hash_A, signature: sig_A })` — where `payload_hash_A` and `sig_A` were produced for a completely different transaction `tx_id_A`.
4. The contract accepts the call: `sig_A` is a valid signature over `payload_hash_A`, and `request_B` exists in `pending_verify_foreign_tx_requests`.
5. The pending yield for `request_B` is resolved and the caller receives `VerifyForeignTransactionResponse { payload_hash: payload_hash_A, signature: sig_A }`.

The caller's bridge contract receives an attestation that does not correspond to `tx_id_B`. If the bridge contract does not independently recompute and compare the expected `payload_hash` (as the SDK helper `ForeignChainSignatureVerifier::verify_signature` does), it will process a fraudulent bridge event — a forged foreign-chain verification leading to invalid bridge execution or double-spend conditions. [3](#0-2) 

The SDK-level check is a caller-side defence, not a contract-level invariant. The contract is the authoritative source of truth for what was verified; it must enforce the binding itself.

### Likelihood Explanation

- Any single attested MPC participant (below the signing threshold) can execute this attack — no collusion required.
- All previously produced `payload_hash` / `signature` pairs are permanently visible on-chain.
- The attacker only needs to wait for a pending request they wish to forge a response for.
- The `assert_caller_is_attested_participant_and_protocol_active` guard is the only access control, and a Byzantine participant satisfies it by definition. [4](#0-3) 

### Recommendation

The contract must verify that `response.payload_hash` is consistent with `request`. The cleanest fix is to require the responder to also supply the `extracted_values`, compute `expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, extracted_values }))` on-chain, and assert `response.payload_hash == expected_hash` before accepting the response. This mirrors the SDK's `verify_signature` logic but enforces it at the contract layer where it cannot be bypassed.

If including extracted values in the response is infeasible due to NEAR promise-data size limits, an alternative is to have the contract store a commitment to the expected payload structure at request submission time and verify against it on response.

### Proof of Concept

```
// Setup: a legitimate response for request_A was previously submitted on-chain.
// payload_hash_A and sig_A are now public.

// Step 1: Attacker (Byzantine participant) observes payload_hash_A, sig_A from chain history.

// Step 2: A new pending request for request_B (high-value bridge deposit) exists.
// contract.pending_verify_foreign_tx_requests contains request_B.

// Step 3: Attacker calls:
contract.respond_verify_foreign_tx(
    request_B,                                          // legitimate pending request
    VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,                   // hash from a DIFFERENT transaction
        signature: sig_A,                               // valid signature over payload_hash_A
    }
);

// Step 4: Contract checks:
//   verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) -> Ok  ✓ (sig is valid over hash)
//   resolve_yields_for(request_B, ...)                -> Ok  ✓ (request_B is pending)
// Contract resolves request_B with forged response.

// Step 5: Caller of verify_foreign_transaction(request_B) receives
//   VerifyForeignTransactionResponse { payload_hash: payload_hash_A, sig: sig_A }
// which attests to tx_id_A's data, not tx_id_B's data.
// Bridge contract that skips payload_hash recomputation processes a fraudulent event.
```

The root cause is identical to the reference report: the post-condition check (signature validity over `payload_hash`) can be satisfied by pre-existing state (a previously produced signature) without performing the intended action (verifying the specific foreign transaction identified by `request`).
