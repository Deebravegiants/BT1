### Title
`LRTOracle.updateRSETHPrice()` Reverts for Non-Managers When Price Increase Exceeds `pricePercentageLimit`, Leaving `rsETHPrice` Stale and Enabling Over-Minting of rsETH — (`contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle._updateRsETHPrice()` contains a guard that reverts for any non-manager caller when the computed new rsETH price exceeds `highestRsethPrice` by more than `pricePercentageLimit`. Because `updateRSETHPrice()` is a public function intended to be called by anyone (e.g., keepers, users), this guard silently freezes the stored `rsETHPrice` at a stale, below-actual value whenever a legitimate price jump occurs. During the stale window, every deposit mints more rsETH than the depositor is entitled to, diluting all existing rsETH holders and constituting a theft of their accrued yield.

---

### Finding Description

`LRTOracle._updateRsETHPrice()` computes `newRsETHPrice` from on-chain TVL and rsETH supply, then applies two band checks against `highestRsethPrice` (the all-time-high stored price):

**Price-increase gate** (lines 252–266):

```solidity
if (newRsETHPrice > highestRsethPrice) {
    uint256 priceDifference = newRsETHPrice - highestRsethPrice;
    bool isPriceIncreaseOffLimit =
        pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);

    if (isPriceIncreaseOffLimit) {
        if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
            revert PriceAboveDailyThreshold();   // ← blocks public callers
        }
    }
}
```

When this revert fires, `rsETHPrice` (line 313) and `highestRsethPrice` (line 295) are **never updated**. The stored price remains at its previous, below-actual value until a manager manually calls `updateRSETHPriceAsManager()`.

**Price-decrease gate** (lines 270–281) — the symmetric downside path — pauses the deposit pool and withdrawal manager and returns early, also leaving `rsETHPrice` un-updated.

The stale `rsETHPrice` is consumed directly in two critical user-facing paths:

`LRTDepositPool.getRsETHAmountToMint()` (line 520):
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`LRTWithdrawalManager.getExpectedAssetAmount()` (line 593):
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

When `rsETHPrice` is stale-low:
- The deposit denominator is smaller → depositors receive **more rsETH** than their deposit is worth.
- The withdrawal numerator is smaller → withdrawers receive **fewer assets** than their rsETH is worth.

The net effect is that new depositors extract value from existing rsETH holders: each rsETH minted at the stale price represents a claim on more protocol TVL than the depositor contributed, diluting every existing holder's share.

---

### Impact Explanation

**Theft of unclaimed yield (High).** Every deposit made while `rsETHPrice` is stale-low mints excess rsETH. The excess is backed by TVL that already belonged to existing holders. The magnitude equals:

```
excess_rsETH_per_deposit = deposit_ETH_value * (1/stale_price - 1/actual_price)
```

For a 1 % price gap and a 1 000 ETH deposit, approximately 9.5 rsETH is over-minted, representing ~10 ETH of value extracted from existing holders at the actual price. The gap can persist for hours or days if the manager is slow to respond, and multiple depositors can exploit it simultaneously.

---

### Likelihood Explanation

The trigger condition — `newRsETHPrice` jumping more than `pricePercentageLimit` above `highestRsethPrice` in a single call — is realistic:

1. **Reward accrual after a long oracle silence**: if `updateRSETHPrice()` is not called for an extended period (e.g., during low-activity periods), accumulated staking rewards can push the computed price above the threshold in one step.
2. **Underlying LST price jump**: a sudden increase in the stETH/ETH or ETHx/ETH rate (e.g., after a large Lido reward distribution) directly raises `totalETHInProtocol` and thus `newRsETHPrice`.
3. **Small `pricePercentageLimit`**: if set to 1 % (1e16), even a modest reward accumulation over a few days can exceed the cap.

No attacker action is required to create the condition; a depositor only needs to observe that `updateRSETHPrice()` reverts and then deposit before the manager intervenes.

---

### Recommendation

1. **Clamp instead of revert**: when `isPriceIncreaseOffLimit` is true for a public caller, update `rsETHPrice` to `highestRsethPrice * (1 + pricePercentageLimit)` (the band boundary) rather than reverting. This mirrors the GTE Perps recommendation of executing at the band boundary.
2. **Separate the guard from the price write**: allow the price to be written up to the capped value unconditionally, and only gate the full jump behind the manager role.
3. **Keeper incentive / automation**: ensure an on-chain keeper or Gelato task calls `updateRSETHPriceAsManager()` promptly whenever the public call reverts, minimising the stale window.

---

### Proof of Concept

1. Protocol is live; `highestRsethPrice = 1.04e18`, `rsETHPrice = 1.04e18`, `pricePercentageLimit = 1e16` (1 %).
2. Staking rewards accumulate over several days without a price update. On-chain TVL now implies `newRsETHPrice = 1.056e18` (a 1.54 % jump above `highestRsethPrice`).
3. Any user calls `updateRSETHPrice()`. Inside `_updateRsETHPrice()`, `isPriceIncreaseOffLimit = true` and `msg.sender` lacks `MANAGER` role → `revert PriceAboveDailyThreshold()`.
4. `rsETHPrice` remains `1.04e18`; `highestRsethPrice` remains `1.04e18`.
5. Attacker deposits 1 000 ETH of stETH via `LRTDepositPool.depositAsset()`. `getRsETHAmountToMint` computes: `1000e18 * 1e18 / 1.04e18 ≈ 961.5 rsETH`. Correct amount at actual price: `1000e18 * 1e18 / 1.056e18 ≈ 946.9 rsETH`. Over-mint: **~14.6 rsETH** (~15.4 ETH of value at actual price).
6. Manager eventually calls `updateRSETHPriceAsManager()`, price jumps to `1.056e18`. The attacker's 961.5 rsETH is now worth `961.5 * 1.056 ≈ 1015.3 ETH` — a ~15 ETH profit extracted from existing holders. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
