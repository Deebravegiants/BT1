### Title
`RequireGuard` Native Script Authorization Bypass via Attacker-Controlled `guards` Field - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`)

### Summary

In the Dijkstra era, the new `RequireGuard` native script type evaluates by checking whether a credential is present in the transaction body's `guards` field. For `KeyHashObj` credentials, the ledger never verifies that the corresponding key has actually signed the transaction. Because the `guards` field is entirely attacker-controlled (it is part of the transaction body authored by the submitter), any attacker can satisfy `RequireGuard (KeyHashObj kh)` for an arbitrary key hash `kh` by simply inserting `KeyHashObj kh` into their own transaction's `guards` field — no signature from `kh` is required. This allows unauthorized spending of any UTxO locked by a `RequireGuard (KeyHashObj kh)` script.

### Finding Description

**Root cause — `evalDijkstraNativeScript` trusts the attacker-supplied `guards` set:**

In `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`, the `RequireGuard` branch of `evalDijkstraNativeScript` is:

```haskell
RequireGuard cred -> cred `OSet.member` guards
```

The `guards` value is taken directly from the transaction body via `tx ^. bodyTxL . guardsTxBodyL`. [1](#0-0) 

`validateDijkstraNativeScript` passes the `guards` field from the transaction body straight into `evalDijkstraNativeScript`:

```haskell
validateDijkstraNativeScript tx =
  evalDijkstraNativeScript vhks (tx ^. bodyTxL . vldtTxBodyL) (tx ^. bodyTxL . guardsTxBodyL)
  where
    vhks = Set.map witVKeyHash (tx ^. witsTxL . addrTxWitsL)
``` [2](#0-1) 

**No witness requirement for key-hash guards:**

The Dijkstra UTXOW transition rule calls `Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody`, which uses `getWitsVKeyNeeded = getConwayWitsVKeyNeeded`. [3](#0-2) 

`getConwayWitsVKeyNeeded` is inherited unchanged and does not inspect the `guards` field. [4](#0-3) 

`getDijkstraScriptsNeeded` only adds `ScriptHashObj` credentials from `guards` to `scriptsNeeded` (via `credScriptHash`), and completely ignores `KeyHashObj` credentials: [5](#0-4) 

The result is that `KeyHashObj kh` entries in `guards` are never checked against the transaction's VKey witness set. The ledger only verifies that the credential appears in the `guards` field — a field the attacker writes themselves.

**Confirmed by the test suite:**

The test "Spending inputs locked by script requiring a keyhash guard" explicitly demonstrates that submitting `tx & bodyTxL . guardsTxBodyL .~ [guardKeyHash]` succeeds with no signature for `guardKeyHash`. The comment in `impDijkstraSatisfyNativeScript` even acknowledges the gap: `-- TODO: actual satisfy the native scripts by updating the transaction's guards`. [6](#0-5) [7](#0-6) 

**Structural analogy to the QuickAccManager bug:**

The QuickAccManager bug arose because a singleton authorization manager trusted an attacker-supplied `id` embedded in the signature for its privilege check. Here, the Dijkstra ledger trusts the attacker-supplied `guards` field for the `RequireGuard` authorization check. In both cases, the authorization reference is opaque data authored by the attacker, and the system performs no independent verification that the referenced entity actually consented.

**Exploit path:**

1. Alice locks a UTxO with the native script `RequireGuard (KeyHashObj aliceKH)`, intending that only Alice (holder of `aliceKH`) can spend it.
2. Attacker constructs a transaction spending Alice's UTxO, setting `guardsTxBodyL .~ [KeyHashObj aliceKH]` in the transaction body. The attacker does not possess Alice's signing key and provides no witness for `aliceKH`.
3. UTXOW rule calls `validateDijkstraNativeScript`, which calls `evalDijkstraNativeScript` with `guards = {KeyHashObj aliceKH}`. The check `KeyHashObj aliceKH `OSet.member` guards` returns `True`.
4. `validateNeededWitnesses` does not require a witness for `aliceKH` because `getConwayWitsVKeyNeeded` never inspects `guards`.
5. The transaction is accepted. Alice's funds are stolen.

### Impact Explanation

**Critical — Direct loss of ADA or native assets through an invalid ledger state transition.**

Any UTxO locked by `RequireGuard (KeyHashObj kh)` for any key hash `kh` can be unconditionally spent by any unprivileged attacker. The attacker needs only to know `kh` (which is public, derivable from the script hash on-chain) and to include it in their transaction's `guards` field. No private key material is required. All ADA and native assets held in such outputs are at risk of immediate theft.

### Likelihood Explanation

High. The attacker requires no privileged access, no leaked keys, and no governance majority. The exploit requires only the ability to submit a transaction — the baseline capability of any Cardano user. The `guards` field is a standard transaction body field freely settable by any transaction author. The key hash of the guard credential is derivable from the on-chain script hash. The attack is deterministic and requires no brute force.

### Recommendation

Key-hash credentials appearing in the `guards` field must be required as VKey witnesses, analogous to how `RequireSignature` works. The fix is to extend `getWitsVKeyNeeded` (or introduce a Dijkstra-specific override) to collect all `KeyHashObj` credentials from `txBody ^. guardsTxBodyL` and add their key hashes to the required witness set. This mirrors the existing `RequireSignature` pattern:

```haskell
-- In the Dijkstra-specific witsVKeyNeeded:
guardKeyHashes = Set.fromList
  [ kh | KeyHashObj kh <- OSet.toList (txBody ^. guardsTxBodyL) ]
```

These hashes must then be included in the set checked by `validateNeededWitnesses`, so that `validateVerifiedWits` enforces a valid signature from each guard key hash.

### Proof of Concept

```
1. Deploy UTxO locked by: RequireGuard (KeyHashObj aliceKH)
   (Alice's key hash, e.g. derived from her payment key)

2. Attacker constructs Tx:
   - inputs:  [aliceUTxO]
   - outputs: [attackerAddress, full_value]
   - guards:  [KeyHashObj aliceKH]   ← attacker writes this freely
   - witnesses: [attackerSignature]  ← no signature from aliceKH

3. Submit Tx to node.

4. UTXOW rule:
   - validateDijkstraNativeScript:
       evalDijkstraNativeScript ... {KeyHashObj aliceKH} (RequireGuard (KeyHashObj aliceKH))
       → KeyHashObj aliceKH `OSet.member` {KeyHashObj aliceKH}  → True  ✓
   - validateNeededWitnesses:
       getConwayWitsVKeyNeeded does not include aliceKH  → no witness required  ✓

5. Tx accepted. Alice's funds transferred to attacker.
```

evalDijkstraNativeScript ::
  (DijkstraEraScript era, NativeScript era ~ DijkstraNativeScript era) =>
  Set.Set (KeyHash Witness) ->
  ValidityInterval ->
  OSet (Credential Guard) ->
  NativeScript era ->
  Bool
evalDijkstraNativeScript keyHashes (ValidityInterval txStart txExp) guards = go
  where
    -- the evaluation will stop as soon as it reaches the required number of valid scripts
    isValidMOf n SSeq.Empty = n <= 0
    isValidMOf n (ts SSeq.:<| tss) =
      n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
    go = \case
      RequireTimeStart lockStart -> lockStart `lteNegInfty` txStart
      RequireTimeExpire lockExp -> txExp `ltePosInfty` lockExp
      RequireSignature hash -> hash `Set.member` keyHashes
      RequireAllOf xs -> all go xs
      RequireAnyOf xs -> any go xs
      RequireMOf m xs -> isValidMOf m xs
      RequireGuard cred -> cred `OSet.member` guards
      _ -> error "Impossible: All NativeScripts should have been accounted for"
```
