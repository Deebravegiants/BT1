Audit Report

## Title
Stale Cross-Chain Rate in `CrossChainRateReceiver.getRate()` Allows Depositors to Mint Excess agETH, Stealing Unclaimed Yield — (`contracts/cross-chain/CrossChainRateReceiver.sol` / `contracts/agETH/AGETHPoolV3.sol`)

## Summary
`CrossChainRateReceiver` records `lastUpdated` on every `lzReceive` call but never enforces a maximum staleness window in `getRate()`. `AGETHPoolV3.deposit()` blindly consumes whatever rate is stored. When the L2 rate lags behind the true L1 agETH/ETH rate — which monotonically increases as staking yield accrues — any depositor can mint more agETH than their ETH warrants, extracting yield that belongs to existing holders.

## Finding Description
`CrossChainRateReceiver.getRate()` (lines 103–105) returns `rate` unconditionally. `lastUpdated` is written in `lzReceive` (line 97) but is never read back in any validation path. `AGETHPoolV3.viewSwapAgETHAmountAndFee` (lines 160–169) calls `getRate()` and computes:

```solidity
agETHAmount = amountAfterFee * 1e18 / agETHToETHrate;
```

Because the agETH/ETH rate rises over time as staking rewards accrue, a stale (lower) `agETHToETHrate` produces a larger `agETHAmount` than the depositor is entitled to. `deposit()` (lines 115–128) mints this inflated amount with no intervening staleness guard. There is no `require(block.timestamp - lastUpdated <= MAX_STALENESS)` anywhere in the call chain. The `lastUpdated` field's existence signals the intent to enforce freshness, but the enforcement was never implemented.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing agETH holders have accrued staking yield on L1, reflected in the rising agETH/ETH rate. A depositor who mints at a stale (lower) rate receives a proportionally larger share of total agETH supply than their ETH contribution justifies. On redemption at the true rate on L1, they recover more ETH than they deposited. The surplus is extracted directly from yield that had already accrued to prior holders but had not yet been reflected in the L2 oracle. The magnitude scales linearly with staleness duration and deposit size.

## Likelihood Explanation
LayerZero rate updates are not guaranteed on any fixed schedule; the keeper/relayer can be delayed or inactive for hours or days. `lastUpdated` is a public storage slot, so an attacker can trivially monitor on-chain when the rate was last refreshed and time a large deposit to coincide with maximum staleness. No privileged access is required — `deposit()` is fully permissionless. The attack is repeatable every time the oracle falls behind.

## Recommendation
Add a staleness check in `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours; // configurable by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

This causes `deposit()` to revert when the oracle is stale, preventing exploitation while the rate is being refreshed. Alternatively, enforce the check inside `AGETHPoolV3.viewSwapAgETHAmountAndFee` directly.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Fork test on L2 deployment
// 1. Deploy AGETHPoolV3 pointing to a MockAGETHRateReceiver
// 2. Mock returns STALE_RATE = 1.04e18 (simulating ~1 day of staleness vs TRUE_RATE = 1.05e18)
// 3. Attacker deposits 100 ETH
// 4. Assert minted agETH * trueRate > 100 ETH (attacker profits at existing holders' expense)

contract StaleRatePoC {
    AGETHPoolV3 pool;
    uint256 constant TRUE_RATE  = 1.05e18;
    uint256 constant STALE_RATE = 1.04e18; // ~1% stale

    function testStaleRateExploit() public {
        uint256 deposit = 100 ether;
        uint256 agETHMinted = pool.deposit{value: deposit}("");
        // agETHMinted ≈ 100e18 * 1e18 / 1.04e18 ≈ 96.15 agETH
        // Redeemable at true rate: 96.15 * 1.05e18 / 1e18 ≈ 100.96 ETH
        // Profit ≈ 0.96 ETH extracted from existing holders per 100 ETH deposited
        uint256 redeemableETH = agETHMinted * TRUE_RATE / 1e18;
        assert(redeemableETH > deposit);
    }
}
