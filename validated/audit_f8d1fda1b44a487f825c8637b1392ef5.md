### Title
Missing Early Return in `validate_delegate_action_key` Allows `DepositWithFunctionCall` Error to Be Overwritten, Producing Wrong Execution Outcome - (File: runtime/runtime/src/actions.rs)

### Summary

In `validate_delegate_action_key`, when a `DelegateAction` uses a function-call access key and the inner `FunctionCall` has a non-zero deposit, the code sets `result.result = Err(DepositWithFunctionCall)` but — before protocol version 85 — does **not** return early. Execution falls through to the `receiver_id` and `method_name` checks. If the receiver or method name also mismatches, the error is silently overwritten with `ReceiverMismatch`, producing a wrong error code in the execution outcome. This is the direct analog of the `VirtualPool` bug where a security check is set but falls through to subsequent checks that overwrite it.

### Finding Description

In `runtime/runtime/src/actions.rs`, `validate_delegate_action_key` validates the access key used to sign a `DelegateAction` (meta-transaction). When the access key has `FunctionCall` permission, the code checks whether the inner action has a non-zero deposit:

```rust
if function_call.deposit > Balance::ZERO {
    result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
        InvalidAccessKeyError::DepositWithFunctionCall,
    ).into());
    // Before this fix, the missing early return allowed execution
    // to fall through to the receiver_id and method_name checks,
    // which could overwrite this error with a different one.
    if ProtocolFeature::FixDelegateActionDepositWithFunctionCallError
        .enabled(apply_state.current_protocol_version)
    {
        return Ok(());
    }
}
if delegate_action.receiver_id() != &function_call_permission.receiver_id {
    result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
        InvalidAccessKeyError::ReceiverMismatch { ... }
    ).into());
    return Ok(());
}
```

Before `ProtocolFeature::FixDelegateActionDepositWithFunctionCallError` (protocol version 85), the missing `return Ok()` after setting `DepositWithFunctionCall` allows execution to fall through. If the `receiver_id` also mismatches, `ReceiverMismatch` overwrites `DepositWithFunctionCall` in `result.result`. The execution outcome stored on-chain and returned via RPC carries the wrong error variant.

Additionally, in the case where `deposit > 0` AND `receiver_id` matches AND `method_name` matches, execution falls all the way through to the nonce update block (lines 685–699), advancing the sender's access key nonce even though the action is ultimately rejected — a state mutation that should not occur for an invalid action. ### Impact Explanation

The corrupted protocol value is the **error code in the execution outcome** (`TxExecutionError::ActionError::ActionErrorKind`), which is committed to the outcome root and returned by RPC. An unprivileged user submitting a `DelegateAction` with `deposit > 0` and a mismatching `receiver_id` receives `ReceiverMismatch` instead of `DepositWithFunctionCall`. Applications and wallets that branch on the specific error variant will misdiagnose the failure. In the receiver-matches case, the access key nonce DB entry is advanced without a corresponding successful action, a state inconsistency reachable by any relayer submitting a signed meta-transaction. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

Any unprivileged user can submit a `DelegateAction` via the public RPC. The trigger requires a function-call access key with a non-zero deposit in the inner `FunctionCall` action — a realistic mistake by a sender or a deliberate submission by a relayer. The bug is active on all nodes running protocol version < 85 and is preserved in the production codebase as a backward-compatibility path. [6](#0-5) [7](#0-6) 

### Recommendation

The fix — `return Ok()` immediately after setting `DepositWithFunctionCall` — is already gated behind `ProtocolFeature::FixDelegateActionDepositWithFunctionCallError` at protocol version 85. Ensure all production nodes have upgraded past version 85 so the legacy fall-through path is never exercised. The legacy branch should be removed once the minimum supported protocol version exceeds 84. [8](#0-7) 

### Proof of Concept

1. Create an account `sender.near` with a function-call access key permitting calls to `receiver.near::method_foo`.
2. Sign a `DelegateAction` with `receiver_id = other.near` (mismatch), `actions = [FunctionCall { method_name: "method_foo", deposit: 1 yoctoNEAR }]`.
3. Submit via a relayer on a node running protocol version < 85.
4. Observe the execution outcome error is `ReceiverMismatch { tx_receiver: other.near, ak_receiver: receiver.near }` instead of the correct `DepositWithFunctionCall`.
5. The test `test_delegate_deposit_with_function_call_reports_receiver_mismatch_before_fix` in `runtime/runtime/src/actions.rs` confirms this exact behavior. [9](#0-8)
