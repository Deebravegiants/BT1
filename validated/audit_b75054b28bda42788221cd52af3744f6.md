### Title
`ForeignTxSignPayloadV1` Lacks Caller Binding and Expiry — Enables Cross-Chain Signature Replay for Double-Spend - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The MPC network's `verify_foreign_transaction` flow signs a payload (`ForeignTxSignPayloadV1`) that contains only the foreign-chain request and extracted values. It includes no NEAR caller address, no NEAR block height, and no expiry timestamp. Because the signed payload is deterministic and permanently valid, any party who observes a completed `VerifyForeignTransactionResponse` — or who re-submits the same foreign-chain tx_id — can obtain an identical, valid MPC signature and replay it against any NEAR bridge contract that uses `ForeignChainSignatureVerifier::verify_signature()` without its own deduplication registry.

---

### Finding Description

`ForeignTxSignPayloadV1` is the canonical payload the MPC network signs for foreign-chain transaction verification:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [1](#0-0) 

The 32-byte `msg_hash` is `SHA-256(borsh(ForeignTxSignPayload))` — a pure function of the foreign-chain data with no NEAR-side context. [2](#0-1) 

`args_into_verify_foreign_tx_request` converts the user-supplied args into the on-chain request key, explicitly dropping `env::predecessor_account_id()`:

```rust
pub fn args_into_verify_foreign_tx_request(

    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [3](#0-2) 

`VerifyForeignTransactionRequest` — the pending-request map key — contains no caller field:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [4](#0-3) 

The contract's own test explicitly documents this as "caller-agnostic": both Alice and Bob's submissions for the same foreign tx map to the same pending-request queue entry. [5](#0-4) 

`respond_verify_foreign_tx` verifies the signature against the root public key and resolves the yield, but maintains no "already-signed" registry for completed requests. [6](#0-5) 

The SDK's `ForeignChainSignatureVerifier::verify_signature()` checks only that the signature is cryptographically valid for the `(request, extracted_values)` pair — no expiry, no caller binding:

```rust

// ...
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response, &payload_hash, &secp_pk,
).is_ok()
``` [7](#0-6) 

---

### Impact Explanation

A valid `VerifyForeignTransactionResponse` for a given foreign-chain tx_id is:

1. **Permanently valid** — no NEAR block height or timestamp is included in the signed payload, so the signature never expires.
2. **Caller-agnostic** — the signed payload contains no NEAR predecessor account, so the signature is not bound to the original requester.
3. **Deterministic** — the same foreign-chain tx_id always produces the same `msg_hash` and therefore the same MPC signature.

An attacker who observes a completed `VerifyForeignTransactionResponse` on-chain (all NEAR transactions are public) can replay it to any bridge contract that calls `ForeignChainSignatureVerifier::verify_signature()` without maintaining its own processed-tx registry. This enables double-spend: the same foreign-chain deposit event can be used to mint or unlock funds on NEAR multiple times.

**Impact class:** High — cross-chain replay / forged foreign-chain verification bypass causing double-spend conditions.

---

### Likelihood Explanation

- All NEAR transactions and their return values are publicly observable; a completed `VerifyForeignTransactionResponse` is trivially extractable from chain history.
- Re-submitting the same foreign-chain tx_id to `verify_foreign_transaction` is permissionless (requires only 1 yoctoNEAR deposit).
- The MPC network will re-sign the same payload deterministically, producing an identical valid signature.
- Bridge contracts built on the `near-mpc-sdk` `ForeignChainSignatureVerifier` that do not implement their own tx_id deduplication are directly exploitable.
- No privileged access, threshold collusion, or TEE bypass is required.

---

### Recommendation

1. **Add a NEAR caller binding to the signed payload.** Include `predecessor_account_id` (or a hash of it) in `ForeignTxSignPayloadV1` so the signature is only valid for the specific NEAR account that requested it. This is the direct analog of the external report's fix.

2. **Add a NEAR block height or timestamp to the signed payload.** Include the NEAR block height at request time so the signature expires after a configurable window, preventing indefinite replay.

3. **Maintain a completed-request registry in the MPC contract.** After `respond_verify_foreign_tx` resolves a request, record the `(request, domain_id)` tuple in a bounded set and reject re-submissions of already-completed requests.

4. **Update `ForeignChainSignatureVerifier::verify_signature()` in the SDK** to enforce expiry and caller-binding checks once those fields are added to the payload.

### Proof of Concept

**Step 1 — Legitimate bridge call:**
```
bridge.deposit_from_eth(eth_tx_id=0xABC)
  → mpc.verify_foreign_transaction({ request: Bitcoin(tx_id=0xABC), domain_id: 1, payload_version: V1 })
  → MPC nodes verify tx, sign SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))
  → bridge callback receives VerifyForeignTransactionResponse { payload_hash, signature }
  → bridge mints 100 USDC to Alice
```

**Step 2 — Attacker replay (no privileged access needed):**
```
attacker calls mpc.verify_foreign_transaction({ request: Bitcoin(tx_id=0xABC), domain_id: 1, payload_version: V1 })
  → MPC nodes verify the same tx again (it's still on-chain), produce identical signature
  → attacker's callback receives the same VerifyForeignTransactionResponse
  → attacker presents it to bridge.deposit_from_eth(eth_tx_id=0xABC)
  → bridge calls ForeignChainSignatureVerifier::verify_signature() — passes (signature is valid)
  → bridge mints another 100 USDC (double-spend)
```

    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
