### Title
MPC Key Share Material and Encryption Keys Exposed via CLI Positional Arguments — (`File: crates/node/src/cli.rs`)

### Summary

The `ImportKeyshareCmd` and `ExportKeyshareCmd` subcommands of `mpc-node` require sensitive cryptographic material — including the raw MPC private key share (`keyshare_json`) and the local AES storage encryption key (`local_encryption_key_hex`) — to be supplied as **positional command-line arguments** with no environment-variable alternative. This causes the key share material to be permanently recorded in shell history and transiently visible in `/proc/<pid>/cmdline` and `ps` output, directly analogous to the eth2deposit password-leakage finding.

---

### Finding Description

`ImportKeyshareCmd` is defined with two sensitive positional arguments:

```rust
#[derive(Args, Debug)]
pub struct ImportKeyshareCmd {
    #[arg(long, env("MPC_HOME_DIR"))]
    pub home_dir: String,

    #[arg(
        help = "JSON string with the keyshare in format: {\"epoch\":1,\"private_share\":\"...\",\"public_key\":\"...\"}"
    )]
    pub keyshare_json: String,          // ← positional, no env() alternative

    #[arg(help = "Hex-encoded 16 byte AES key for local storage encryption")]
    pub local_encryption_key_hex: String,  // ← positional, no env() alternative
}
``` [1](#0-0) 

Neither `keyshare_json` nor `local_encryption_key_hex` has an `env(...)` annotation, so clap provides no environment-variable path. The operator **must** pass them on the command line:

```bash
mpc-node import-keyshare \
  --home-dir /data \
  '{"epoch":1,"private_share":"<RAW_SECRET_SCALAR>","public_key":"..."}' \
  <32-hex-AES-key>
```

The same pattern applies to `ExportKeyshareCmd`, which also takes `local_encryption_key_hex` as a positional argument: [2](#0-1) 

Additionally, `backup-cli`'s `GetKeysharesArgs` and `PutKeysharesArgs` expose `backup_encryption_key_hex` as a named CLI flag (no env-only path enforced): [3](#0-2) 

By contrast, `SecretsStartConfig` in the node-config crate correctly implements a custom `Debug` that redacts both keys: [4](#0-3) 

This shows the codebase is aware of the sensitivity of these values — but the protection is absent at the CLI entry point where the values are first supplied.

---

### Impact Explanation

`keyshare_json` contains the `private_share` field, which is the node's individual MPC key share (a secret scalar). This is the most sensitive value in the entire system. When passed on the command line:

1. **Shell history** (`~/.bash_history`, `~/.zsh_history`) permanently records the full JSON including `private_share`.
2. **`/proc/<pid>/cmdline`** exposes the arguments to any local user while the process runs.
3. **`ps aux`** shows the full argument list to any user with process-listing permission.

An attacker who reads the shell history of `t` or more operator machines (where `t` is the signing threshold) can reconstruct the full MPC private key and forge arbitrary threshold signatures for any foreign chain — Bitcoin, Ethereum, Solana — controlled by the MPC network. This constitutes **unauthorized access to MPC key shares and secret material that materially enables forgery or secret recovery**.

`local_encryption_key_hex` leakage is a secondary impact: combined with access to the encrypted keyshare file on disk, it allows offline decryption of the stored key share without needing the import command at all.

---

### Likelihood Explanation

`import-keyshare` is an **operational procedure** used during node migration and recovery. The migration guide explicitly instructs operators to run this command: [5](#0-4) 

Shell history is enabled by default on all common shells. Any attacker who gains unprivileged shell access to an operator's workstation or server — through credential theft, SSH key compromise, or a shared-user environment — can read `~/.bash_history` and recover the key share. The `/proc/<pid>/cmdline` vector is exploitable by any co-tenant process during the brief window the import runs.

---

### Recommendation

**Short term:** Remove `keyshare_json` and `local_encryption_key_hex` as positional arguments. Instead, read the keyshare JSON from a file path argument (e.g., `--keyshare-file`) and supply the AES key via an environment variable (e.g., `MPC_LOCAL_ENCRYPTION_KEY`). This matches the pattern already used by `StartCmd.secret_store_key_hex`: [6](#0-5) 

**Long term:** Adopt a secrets-manager pattern (e.g., read from a file descriptor, a named pipe, or a secrets vault) for all key material ingestion, consistent with the TEE-based design goal of never exposing secrets outside the enclave.

---

### Proof of Concept

An operator migrating a node runs:

```bash
mpc-node import-keyshare \
  --home-dir /data \
  '{"epoch":3,"private_share":"deadbeef...","public_key":"..."}' \
  aabbccddeeff00112233445566778899
```

Immediately after, any local user can recover both secrets:

```bash
# From shell history
cat ~/.bash_history | grep import-keyshare

# From process listing (while running)
ps aux | grep import-keyshare

# From procfs (while running)
cat /proc/$(pgrep mpc-node)/cmdline | tr '\0' ' '
```

An attacker who collects `private_share` values from `t` operator machines (the threshold) can reconstruct the full MPC signing key and issue unauthorized threshold signatures for any foreign-chain transaction, bypassing the MPC network's security model entirely. [1](#0-0)

## Step 3: Generate and Set Encryption Key

For additional security, the backup and restore process encrypts keyshares during transport using AES encryption. You need to generate a shared encryption key and set it as an environment variable for both your old and new nodes, as well as for the backup-cli commands.

**Important:** The `MPC_BACKUP_ENCRYPTION_KEY_HEX` must be the same between the backup-cli and the node it is currently communicating with (e.g., the old node when running `get-keyshares`, and the new node when running `put-keyshares`).


### Retrieve a key from an existing node.

**Note:** If your node has been running without the `MPC_BACKUP_ENCRYPTION_KEY_HEX` environment variable set, the node automatically generates an encryption key and stores it in a file called `backup_encryption_key.hex` in your `$MPC_HOME_DIR` directory. You can retrieve it with:

```bash
export BACKUP_ENCRYPTION_KEY=$(cat $MPC_HOME_DIR/backup_encryption_key.hex)
```

Copy this key and set it as the `BACKUP_ENCRYPTION_KEY` environment variable for the backup-cli when running `get-keyshares`.

**TEE (TDX/dstack) nodes:** `$MPC_HOME_DIR` (`/data`) is inside the CVM's encrypted disk, so you cannot read the auto-generated `backup_encryption_key.hex`. Provide the key yourself instead: set it in the `[mpc_node_config.secrets]` block of the node's `user-config.toml` (see [Prepare MPC Node Configuration](https://github.com/near/mpc/blob/main/docs/running-an-mpc-node-in-tdx-external-guide.md#prepare-mpc-node-configuration) in the operator guide) and keep a copy outside the CVM:

```toml
[mpc_node_config.secrets]
backup_encryption_key_hex = "<your 32-byte hex key>"
```

This is the key you pass to the backup-cli — if the node is already deployed, it is the value you set in `backup_encryption_key_hex` at deploy time. The node reads it from the config on every start, so you can add or change it on a running node via `update-user-config` + restart.



**Note on key differences:**
- `BACKUP_ENCRYPTION_KEY` (this key) is used to encrypt keyshares during transport between nodes and the backup-cli
- `local_storage_aes_key` (from Step 1) is used to encrypt keyshares stored on disk in the backup home directory
- These are two different keys serving different purposes


**TEE Migration Note:** This guide covers the Soft Launch migration process where the encryption key can be accessed from the file system. For TEE-to-TEE migrations in the Hard Launch phase, the backup service will run autonomously within a TEE and handle encryption keys securely without file system access. Refer to [migration-service.md](./migration-service.md) for Hard Launch details.
