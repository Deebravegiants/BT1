### Title
Stale `out0Scaled` after fallback `totalIn1Scaled` assignment causes trader to overpay in exact-input token1→token0 swaps — (`metric-core/contracts/libraries/SwapMath.sol`)

### Summary
In `buyToken0InBinSpecifiedIn`, when two successive approximation steps both underestimate the required input for a given bin position, a fallback at line 621 forces `totalIn1Scaled = state.amountSpecifiedRemainingScaled` without recomputing `out0Scaled`. The pool then consumes the full remaining token1 input while releasing only the token0 computed for the underestimated position, breaking the swap conservation invariant and causing direct trader principal loss.

### Finding Description
The function uses a two-step iterative refinement to find the bin position that exactly consumes the trader's specified token1 input:

1. **Analytical quadratic** (`computeAnalyticalTargetPosForBuyToken0`) gives an initial `targetPos`.
2. **Proportional scale-up** (lines 598-618) corrects underestimates by scaling `delta` proportionally and recomputing `out0Scaled`/`totalIn1Scaled`.
3. **Fallback** (lines 620-622): if the scaled estimate is still below `remaining` and `targetPos < maxFinalBinPos`, the code sets `totalIn1Scaled = remaining` but leaves `out0Scaled` at the value computed for `scaledTarget`.

After the fallback, `totalIn1Scaled == remaining`, so the scale-down branch (lines 624-634) — which correctly rescales `out0Scaled` — is skipped. Settlement then charges the full `remaining` input and credits the stale, smaller `out0Scaled`.

The condition for the fallback to fire: `scaledTarget <= maxFinalBinPos` (the proportional scale-up did not saturate the price limit) AND the recomputed `totalIn1Scaled` for `scaledTarget` is still below `remaining`. This occurs when the price curve is sufficiently concave within the bin (large `deltaPriceX64 / lowerPriceX64` ratio), causing the linear proportional correction to underestimate the true quadratic relationship.

The identical structural bug exists in `buyToken1InBinSpecifiedIn` (lines 758-760), affecting token0→token1 exact-input swaps symmetrically.

### Impact Explanation
The trader pays `remaining` token1 but receives `out0(scaledTarget)` token0 instead of `out0(true_target)`. The shortfall `out0(true_target) - out0(scaledTarget)` remains in the bin and accrues to LPs. This is a direct, uncompensated loss of trader principal on every swap that triggers the fallback. The magnitude scales with bin spread and token0 balance; for wide bins it can reach several percent of the swap output.

### Likelihood Explanation
Any exact-input token1→token0 swap through a bin with a sufficiently large price spread (`upperPriceX64 / lowerPriceX64` ratio) and an input amount that causes both the analytical and proportional refinement to underestimate will trigger this path. No privileged access, malicious setup, or non-standard token behavior is required — the path is reachable through the normal public `swap` entrypoint.

### Recommendation
Replace the fallback at lines 620-622 with a correct resolution: set `targetPos = maxFinalBinPos` and recompute `out0Scaled` for `maxFinalBinPos`, consistent with the semantics of "the remaining input is sufficient to reach the price limit." Alternatively, add a third proportional refinement step mirroring lines 598-618. The scale-down branch (lines 624-634) already shows the correct pattern for adjusting `out0Scaled` proportionally when `totalIn1Scaled` and `remaining` diverge — the fallback should follow the same pattern.
