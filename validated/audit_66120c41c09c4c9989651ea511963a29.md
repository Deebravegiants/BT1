### Title
Single Attested Participant Can Forge CKD Response for `AppPublicKey` Requests Without Threshold Authorization - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs **no on-chain cryptographic verification** of the CKD response when the request uses the `AppPublicKey` variant. Any single attested participant can call `respond_ckd` with a completely arbitrary `CKDResponse`, and the contract will silently accept it, resolve all pending yields, and deliver forged key material to the user — bypassing the threshold requirement entirely.

---

### Finding Description

The `respond_ckd` function contains a match on the `CKDAppPublicKey` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

For `AppPublicKeyPV`, the contract performs a BLS12-381 pairing check (`ckd_output_check`) that cryptographically proves the response is consistent with the MPC network's public key and the user's app public key. For `AppPublicKey`, the arm is an empty block — the contract performs **zero validation** of `big_c` or `big_y` before resolving all queued yields with the supplied bytes.

This is directly analogous to the external report: just as `recreateBlockheaders` skipped the loop when `_blockheaders` was empty and still stored state and emitted an event, `respond_ckd` skips all verification for `AppPublicKey` and still resolves the pending request with whatever bytes the caller supplies.

The `AppPublicKey` variant is the legacy (non-publicly-verifiable) variant and is actively used in production flows. The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` demonstrates this: it passes `CKDResponse { big_y: [1u8; 48], big_c: [2u8; 48] }` — clearly invalid BLS points — and the contract accepts them without complaint.

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request on-chain.
2. Call `respond_ckd` with an arbitrary `CKDResponse` (e.g., all-zero or all-one byte arrays).
3. The contract resolves all queued yields for that request with the forged bytes.
4. The user's callback receives forged key material as if it were the legitimate threshold-derived confidential key.

If the user uses the derived key to encrypt a wallet, protect a secret, or derive a signing key, they will compute a wrong key and permanently lose access to any assets or secrets protected by the correct key. This constitutes **confidential key derivation output without the required participant authorization** — a single participant bypasses the threshold requirement entirely.

**Impact: Critical** — unauthorized CKD output without threshold authorization; potential permanent loss of user funds/secrets.

---

### Likelihood Explanation

- The `AppPublicKey` variant is the legacy variant and is the default in many SDK examples and integration tests.
- Any single attested participant (one of potentially 4–30 nodes) can trigger this. No collusion is required.
- The attack is silent: the contract logs the call but emits no error or anomaly signal.
- The attacker only needs to race the honest nodes to submit `respond_ckd` first for a target request.

**Likelihood: High.**

---

### Recommendation

1. **Require on-chain verification for `AppPublicKey` responses**: If the user's `app_pk` (G1 point) is stored in the request, the contract can verify `e(big_c, G2) = e(big_y, G2)^? · e(H(app_id), PK)` using the existing BLS host functions — or at minimum verify that `big_c` and `big_y` are valid G1 points.
2. **Alternatively, deprecate `AppPublicKey`** in favor of `AppPublicKeyPV`, which already has a correct pairing check, and reject new `AppPublicKey` requests.
3. **At minimum, add a guard** requiring that `respond_ckd` for `AppPublicKey` can only be called once per request (first-writer-wins), so a racing honest node's correct response cannot be overwritten by a later forged one — though this does not prevent a fast attacker from winning the race.

---

### Proof of Concept

```
1. Alice calls request_app_private_key({
       app_public_key: AppPublicKey(<valid G1 point>),
       derivation_path: "my-wallet",
       domain_id: <CKD domain>
   }) with 1 NEAR deposit.
   → Contract stores pending CKD request, queues yield.

2. Malicious participant Bob (one of N nodes, no collusion needed) calls:
   respond_ckd(
       request = <Alice's CKDRequest>,
       response = CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }
   )
   → Contract enters AppPublicKey arm: empty block, no check.
   → resolve_yields_for drains Alice's pending queue, resumes her yield
     with the forged bytes.

3. Alice's callback receives CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }.
   She computes derived_key = big_c - app_sk * big_y = 0 - 0 = 0 (wrong key).
   Any funds or secrets she protects with this key are permanently inaccessible.
```

The existing unit test at [1](#0-0)  already demonstrates that `[1u8; 48]` / `[2u8; 48]` (invalid BLS points) are accepted without error for the `AppPublicKey` variant.

The unguarded arm is at [2](#0-1) , immediately followed by unconditional state resolution at [3](#0-2) .

The contrasting `AppPublicKeyPV` path that correctly performs the pairing check is implemented in [4](#0-3) .

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
