### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Causes Undercollateralized Bridge — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer` records and emits the caller-supplied `amount` in the `InitTransfer` event without verifying the actual tokens received. For fee-on-transfer (FoT) tokens, the bridge custodies fewer tokens than the event declares, causing NEAR to mint or release more wrapped/locked value than is backed, permanently undercollateralizing the bridge for that token.

### Finding Description

In `OmniBridge.sol`, the non-bridge-token path of `initTransfer` performs a `safeTransferFrom` for the caller-supplied `amount` and then immediately emits `InitTransfer` with that same `amount`:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol, lines 406-412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← caller-controlled
    );
}
``` [1](#0-0) 

Immediately after, the event is emitted with the unverified `amount`:

```solidity
// lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← same caller-supplied value, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

There is no before/after balance check anywhere in `initTransfer` or its extension hook `initTransferExtension`. [3](#0-2) 

The NEAR bridge contract reads the `InitTransfer` event amount and uses it verbatim to mint wrapped tokens or credit the recipient. The `init_transfer_internal` function records `transfer_message.amount` directly into `pending_transfers` and calls `lock_tokens_if_needed` with that same value: [4](#0-3) 

The same pattern exists in the Starknet bridge — `init_transfer` calls `transfer_from` for `amount` and emits `InitTransfer` with the same `amount` without a balance delta check: [5](#0-4) 

### Impact Explanation

For any ERC20 token that deducts a fee on `transferFrom` (e.g., tokens with configurable transfer taxes), the bridge vault receives `amount - fee_taken` while the cross-chain message records `amount`. NEAR mints or releases the full `amount` of wrapped/locked tokens. The bridge is now undercollateralized by `fee_taken` per transfer. Repeated transfers drain the backing reserve. When users attempt to redeem wrapped tokens back to EVM, the bridge cannot satisfy the last redeemers — their funds are permanently locked. This matches the **Critical** impact category: irreversible fund lock and broken backing guarantees.

### Likelihood Explanation

The bridge accepts any ERC20 token permissionlessly via `logMetadata` + `deployToken`. An attacker can deploy a custom ERC20 with a configurable transfer fee, register it through the bridge, and repeatedly call `initTransfer` to inflate the NEAR-side supply beyond the EVM-side custody. No privileged access is required. The attacker entry point is the public `initTransfer` function callable by any EOA.

### Recommendation

Replace the fixed-`amount` transfer with a before/after balance check to derive the actual received amount, and use that value in the emitted event and any downstream accounting:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived instead of amount in the event and extension call
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported by adding a check that `actualReceived == amount` and reverting otherwise. Apply the same fix to the Starknet `init_transfer` in `starknet/src/omni_bridge.cairo`.

### Proof of Concept

1. Attacker deploys `FoTToken` — an ERC20 that deducts 10% on every `transferFrom`.
2. Attacker calls `logMetadata(FoTToken)` to register it, then waits for NEAR to deploy the wrapped token.
3. Attacker calls `initTransfer(FoTToken, 1000, 0, 0, "near:attacker.near", "")`.
4. `safeTransferFrom` moves 1000 tokens from attacker; bridge vault receives 900 (10% fee taken by token).
5. `InitTransfer` event emits `amount = 1000`.
6. NEAR relayer processes the event; NEAR bridge mints 1000 wrapped `FoTToken` to `attacker.near`.
7. Attacker now holds 1000 wrapped tokens backed by only 900 real tokens.
8. Attacker redeems 1000 wrapped tokens on NEAR; NEAR signs a `finTransfer` for 1000.
9. EVM `finTransfer` attempts `safeTransfer(attacker, 1000)` — succeeds (vault had 900 + any other deposits).
10. Repeated iterations drain the vault; subsequent legitimate users cannot redeem — funds permanently locked. [6](#0-5)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
