### Title
Wrong SafeCast width (`toUint128` instead of `toUint104`) on `uint104` bin balance in `addLiquidity` silently truncates LP deposits — (File: metric-core/contracts/libraries/LiquidityLib.sol)

---

### Summary

`LiquidityLib.addLiquidity` updates the per-bin `token0BalanceScaled` and `token1BalanceScaled` fields (both typed `uint104`) using `.toUint128()` from OpenZeppelin SafeCast. SafeCast's `toUint128` only reverts when the value exceeds `type(uint128).max` (2¹²⁸ − 1). It does **not** revert when the value exceeds `type(uint104).max` (2¹⁰⁴ − 1). Any value in the range `(type(uint104).max, type(uint128).max]` passes the SafeCast check but is silently truncated when stored into the `uint104` storage field, corrupting the bin's recorded balance. The swap path in `SwapMath.sol` correctly uses `.toUint104()` for the same fields, making this an isolated regression in the liquidity path.

---

### Finding Description

`BinState` declares both balance fields as `uint104`:

```solidity

  uint104 token0BalanceScaled;   // max ≈ 2.03 × 10³¹
