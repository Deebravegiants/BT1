### Title
Missing `builder` Address Validation in `ProposalInit` Allows Malicious Proposer to Redirect Block Fees and Corrupt `sequencer_address` State - (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates many fields of a received `ProposalInit` (height, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices, `starknet_version`, `version_constant_commitment`, timestamp bounds, `fee_proposal_fri`) but never validates the `builder` field. The `builder` field is passed verbatim as `sequencer_address` into `BlockInfo` for both proposal execution and block commitment. A malicious proposer can set `builder` to any attacker-controlled address; the validator will execute the block with that address, produce the same `partial_block_hash` as the proposer, accept the commitment, and commit a block whose `sequencer_address` is the attacker's address — redirecting all transaction fees and corrupting the on-chain sequencer address.

---

### Finding Description

`ProposalInit` carries a `builder` field (the block builder / sequencer address): [1](#0-0) 

`convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address` in the `BlockInfo` that is handed to the batcher for execution: [2](#0-1) 

Both the proposer path (`initiate_build`) and the validator path (`initiate_validation`) call `convert_to_sn_api_block_info` and pass the resulting `BlockInfo` to the batcher: [3](#0-2) [4](#0-3) 

`is_proposal_init_valid` checks height, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices, `starknet_version`, `version_constant_commitment`, timestamp, and `fee_proposal_fri` — but **never checks `builder`**: [5](#0-4) 

`ProposalInitValidation` — the struct that carries the validator's expected values — has no `builder` field at all: [6](#0-5) 

The proposer sets `builder` from its own local configuration: [7](#0-6) 

Because the validator never checks `builder`, a malicious proposer can substitute any address. Both the proposer and the validator will execute the block with the same (attacker-supplied) `sequencer_address`, produce the same `partial_block_hash`, and the commitment check will pass: [8](#0-7) 

---

### Impact Explanation

`sequencer_address` is the address to which transaction fees are transferred during execution. It is also returned by the `get_sequencer_address` syscall and stored in the committed block header. By injecting an arbitrary `builder`:

1. **Fee redirection (Critical)**: All transaction fees in the block are collected to the attacker's address instead of the legitimate sequencer's address. This is a direct economic impact.
2. **Wrong state (Critical)**: The committed block header stores the attacker's address as `sequencer_address`, corrupting the on-chain state visible to contracts and RPC clients.
3. **Wrong syscall result (Critical)**: Any contract calling `get_sequencer_address` during that block receives the attacker's address, which can be exploited by contracts that gate logic on the sequencer identity.

---

### Likelihood Explanation

Any consensus participant who wins the proposer slot for a round can exploit this. No special privilege beyond being a scheduled proposer is required. The attack is silent — the commitment check passes because both sides execute with the same (attacker-supplied) `builder`. There is no existing guard that catches this.

---

### Recommendation

Add `builder` to `ProposalInitValidation` and enforce it in `is_proposal_init_valid`:

```rust
// In ProposalInitValidation:
pub builder: ContractAddress,

// In is_proposal_init_valid, alongside the existing combined check:
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

Populate `proposal_init_validation.builder` from the node's configured `builder_address` when constructing `ProposalInitValidation` in `set_height_and_round` / `validate_proposal` call sites.

---

### Proof of Concept

1. Attacker controls a consensus validator that wins the proposer slot for height H.
2. Attacker calls `build_proposal` normally but overrides `builder` to `attacker_address` in the constructed `ProposalInit` before streaming it.
3. The batcher executes the block with `sequencer_address = attacker_address`; fees are collected to `attacker_address`; the resulting `partial_block_hash` encodes `attacker_address` in the block header.
4. The attacker sends `ProposalFin { proposal_commitment: Poseidon(partial_block_hash, fee_proposal_fri) }`.
5. Each validator receives `ProposalInit { builder: attacker_address, ... }`.
6. `is_proposal_init_valid` passes — `builder` is never checked. [5](#0-4) 
7. `initiate_validation` calls `convert_to_sn_api_block_info(init)` → `sequencer_address = attacker_address` → passed to batcher. [9](#0-8) 
8. The batcher executes with `attacker_address`, produces the same `partial_block_hash` as the proposer.
9. The validator computes `batcher_block_commitment == received_fin.proposal_commitment` → match → proposal accepted. [10](#0-9) 
10. `decision_reached` commits the block. `finalize_decision` calls `convert_to_sn_api_block_info(init)` again with `attacker_address` and passes it to state sync and cende. [11](#0-10) 
11. The committed block has `sequencer_address = attacker_address`; all fees for height H are in the attacker's account.
