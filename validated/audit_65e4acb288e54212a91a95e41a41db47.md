### Title
Missing `N_tag != old_N` Check in ECDSA-2PC Refresh Allows Malicious P1 to Reuse Paillier Key — (`src/cbmpc/protocol/ecdsa_2p.cpp`)

### Summary
The `ecdsa2pc::refresh` function (called by `key_share_ecdsa_hdmpc_2p_t::refresh`) does not verify that the new Paillier modulus `N_tag` sent by P1 differs from the existing modulus `N`. A malicious P1 can send the old `N` as `N_tag`, causing P2 to accept a "refreshed" keyset whose Paillier key is identical to the pre-refresh key, defeating the forward-secrecy guarantee of the refresh operation.

### Finding Description

In `src/cbmpc/protocol/ecdsa_2p.cpp`, P2's validation block after receiving P1's first message checks only size bounds on `N_tag`: [1](#0-0) 

The checks are:
- `N_tag > 0`
- bit-length ≥ `3·|q| + 3·SEC_P_STAT + SEC_P_COM + 1`
- bit-length == `paillier_t::bit_size` (2048)
- `create_pub(N_tag)` succeeds (valid composite)
- `verify_cipher(c_key_tag)` succeeds

There is **no check** `N_tag != key.paillier.get_N()`. A malicious P1 can simply set `new_key.paillier = key.paillier` and transmit the old `N` as `N_tag`.

The `two_paillier_equal_interactive_t` ZK proof does not enforce `N0 ≠ N1` either. Its `verify` function checks only size bounds, commitment opening, and the Sigma-protocol equations: [2](#0-1) 

When P0 == P1 (same key), the proof statement "same plaintext encrypted under P0 and P1" is trivially satisfied and the proof verifies. The `valid_paillier_interactive_t` proof also passes because the reused key is already a valid RSA modulus.

The `prover_msg1` function in the interactive proof also has no such check: [3](#0-2) 

The attack path through the public API is:
`coinbase::api::hd_keyset_ecdsa_2p::refresh` → `coinbase::mpc::key_share_ecdsa_hdmpc_2p_t::refresh` → `ecdsa2pc::refresh` [4](#0-3) [5](#0-4) 

### Impact Explanation

After a malicious P1 executes this attack:
- P2 stores a `new_keyset_blob` where `paillier` is byte-for-byte identical to the pre-refresh Paillier key.
- The `x_share` and `c_key` are updated (rho is applied), but the Paillier private key `(p, q)` is unchanged.
- Any party that previously obtained P1's Paillier private key (e.g., from a leaked backup) can still decrypt `c_key` in the new keyset to recover P1's updated `x_share`.
- The forward-secrecy guarantee of the refresh protocol — that compromise of old Paillier key material does not affect the new keyset — is entirely defeated.
- P2 has no way to detect this: the accepted `new_keyset_blob` is structurally valid and all ZK proofs verify.

This is a public-API-reachable validation bypass in the refresh protocol that causes P2 to accept cryptographically invalid output (a keyset whose Paillier component is not actually refreshed).

### Likelihood Explanation

The attack requires P1 to be a malicious protocol participant, which is an explicit part of the threat model for a 2-party MPC refresh. P1 controls the Paillier key generation step entirely and can trivially substitute the old key. No special cryptographic capability is needed — it is a straightforward omission of a single equality check.

### Recommendation

Add an explicit check in P2's validation block that `N_tag != key.paillier.get_N()`:

```cpp
// In ecdsa2pc::refresh, after the existing N_tag size checks:
if (N_tag == bn_t(key.paillier.get_N()))
    return rv = job.mpc_abort(E_CRYPTO, "N' must differ from old N");
```

The same check should be added in the `hd_keyset_ecdsa_2p` refresh path. Optionally, the `two_paillier_equal_interactive_t::verify` function could also assert `N0 != N1` as a defense-in-depth measure.

### Proof of Concept

A deterministic 2-party simulation test demonstrating the issue:

1. Run honest DKG to produce `(key_p1, key_p2)`.
2. Intercept P1's transport layer; override `new_key.paillier = key_p1.paillier` and `N_tag = key_p1.paillier.get_N()` before sending.
3. Run the refresh protocol to completion — both parties return `SUCCESS`.
4. Assert `new_key_p1.paillier.get_N() == key_p1.paillier.get_N()` — this assertion holds, confirming the Paillier key was not refreshed.
5. Assert `key_p1.paillier.decrypt(new_key_p2.c_key) == new_key_p1.x_share` — the old Paillier private key still decrypts the new ciphertext.

The existing unit test `check_hd_key_pairs_diff` in `tests/unit/protocol/test_hdmpc_ecdsa_2p.cpp` already asserts `EXPECT_NE(k1.paillier.get_N(), new_k1.paillier.get_N())` for the honest execution, confirming that key rotation is an intended invariant — but this invariant is not enforced by the protocol itself. ```cpp
error_t key_share_ecdsa_hdmpc_2p_t::refresh(job_2p_t& job, key_share_ecdsa_hdmpc_2p_t& key,
                                            key_share_ecdsa_hdmpc_2p_t& new_key) {
  // NOTE: this is not an optimized version of the refresh function and has twice as many rounds as needed since
  //       the refresh operations are performed sequentially. It can be made more optimized by copy-pasting the
  //       two refresh code and interleaving the operations.
  //       We are using this less optimized version for simplicity.
  error_t rv = UNINITIALIZED_ERROR;
  ecdsa2pc::key_t ecdsa_key =
      ecdsa2pc::key_t{party_t(key.party_index), key.curve, key.root.Q, key.root.x_share, key.c_key, key.paillier};
  eckey::key_share_2p_t root_key =
      eckey::key_share_2p_t{party_t(key.party_index), key.curve, key.root.K, key.root.k_share};

  ecdsa2pc::key_t new_ecdsa_key;
  eckey::key_share_2p_t new_root_key;

  if (rv = ecdsa2pc::refresh(job, ecdsa_key, new_ecdsa_key)) return rv;
  if (rv = root_key.refresh(job, root_key, new_root_key)) return rv;

  new_key.party_index = key.party_index;
  new_key.curve = key.curve;

  new_key.root.Q = new_ecdsa_key.Q;
  new_key.root.x_share = new_ecdsa_key.x_share;
  new_key.c_key = new_ecdsa_key.c_key;
  new_key.paillier = new_ecdsa_key.paillier;

  new_key.root.k_share = new_root_key.x_share;
  new_key.root.K = new_root_key.Q;

  return SUCCESS;
}
