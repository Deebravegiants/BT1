### Title
Single Failing NodeDelegator External Call in `getAssetDistributionData` Loop Permanently Bricks All Deposits - (File: `contracts/LRTDepositPool.sol`)

---

### Summary

Every call to `depositAsset` or `depositETH` in `LRTDepositPool` passes through `_checkIfDepositAmountExceedesCurrentLimit`, which calls `getTotalAssetDeposits`, which calls `getAssetDistributionData`. That function loops over every registered NodeDelegator and makes two external calls per NDC — `getAssetBalance` and `getAssetUnstaking` — both of which ultimately call into EigenLayer's live contracts. If any single NDC's external call reverts (e.g., due to an EigenLayer upgrade that changes the interface, a deprecated strategy, or a bug in the DelegationManager), the entire deposit path reverts on every invocation, permanently bricking all deposits for that asset.

---

### Finding Description

The deposit entry points `depositETH` and `depositAsset` both call `_beforeDeposit`, which calls `_checkIfDepositAmountExceedesCurrentLimit`, which calls `getTotalAssetDeposits`, which calls `getAssetDistributionData`:

```solidity
// LRTDepositPool.sol – _beforeDeposit
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
```

```solidity
// LRTDepositPool.sol – _checkIfDepositAmountExceedesCurrentLimit
uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
```

```solidity
// LRTDepositPool.sol – getAssetDistributionData
uint256 ndcsCount = nodeDelegatorQueue.length;
for (uint256 i; i < ndcsCount;) {
    assetLyingInNDCs += IERC20(asset).balanceOf(nodeDelegatorQueue[i]);
    assetStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetBalance(asset);       // external call
    assetUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset); // external call
    unchecked { ++i; }
}
```

`getAssetBalance` on each `NodeDelegator` calls into an EigenLayer `IStrategy.userUnderlyingView`, and `getAssetUnstaking` calls `IDelegationManager.getQueuedWithdrawals`. Both are live external calls to EigenLayer contracts with no `try/catch` or error isolation. There is no mechanism to skip a failing NDC or remove it from the loop path without admin intervention.

The identical pattern applies to ETH deposits via `getETHDistributionData`, which loops through NDCs calling `getEffectivePodShares()` and `getAssetUnstaking(ETH_TOKEN)`.

---

### Impact Explanation

If any single NDC's call to EigenLayer reverts — whether due to an EigenLayer contract upgrade that changes the `getQueuedWithdrawals` or `userUnderlyingView` interface, a deprecated strategy, or a transient bug — every subsequent call to `depositAsset` or `depositETH` for that asset will revert at the same point. There is no fallback path. All user deposits are frozen until an admin removes the offending NDC from the queue (which itself requires the NDC to have zero asset balance, per `_verifyAndRemoveNodeDelegator`). If the NDC still holds assets, it cannot be removed, creating a permanent deadlock. User funds already in the protocol are not directly at risk, but no new deposits can be accepted, and the deposit pool is effectively bricked.

**Impact**: Medium — Temporary (potentially permanent) freezing of deposits.

---

### Likelihood Explanation

EigenLayer is an actively upgraded protocol. The `DelegationManager` and `IStrategy` interfaces have already changed across major versions. Any future upgrade that alters the return type or reverts behavior of `getQueuedWithdrawals` or `userUnderlyingView` would trigger this condition. Additionally, if a strategy is deprecated by EigenLayer and its contract is deactivated, calls to `userUnderlyingView` on that strategy would revert. The protocol currently has no circuit-breaker for individual NDC failures in the deposit-limit check loop.

---

### Recommendation

1. Wrap each NDC's external calls inside a `try/catch` block in `getAssetDistributionData` and `getETHDistributionData`, treating a revert as a zero balance contribution rather than a fatal error.
2. Alternatively, decouple the deposit-limit check from the live EigenLayer balance query by caching the total deposited amount in storage and updating it lazily, so that a failing NDC does not block deposits.
3. Add an emergency function allowing an operator to temporarily exclude a specific NDC from the balance aggregation loop without requiring it to have zero balance.

---

### Proof of Concept

Call path that bricks all deposits for a given asset:

1. EigenLayer upgrades `DelegationManager`; `getQueuedWithdrawals` now reverts for NDC at index `k`.
2. User calls `LRTDepositPool.depositAsset(asset, amount, minRSETH, "")`.
3. `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit` → `getTotalAssetDeposits` → `getAssetDistributionData`.
4. Loop reaches index `k`: `INodeDelegator(nodeDelegatorQueue[k]).getAssetUnstaking(asset)` → `_getDelegationManager().getQueuedWithdrawals(address(this))` → **reverts**.
5. The entire `depositAsset` call reverts. Every subsequent deposit attempt for this asset follows the same path and reverts identically.
6. The NDC at index `k` cannot be removed via `removeNodeDelegatorContractFromQueue` because it still holds EigenLayer shares, so `_verifyAndRemoveNodeDelegator` reverts with `NodeDelegatorHasAssetBalance`.
7. Deposits remain permanently frozen until a governance upgrade. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
