Audit Report

## Title
`_withdrawFromAave` Principal Cap Causes `completeWithdrawal` to Revert When Aave Interest Exceeds Withdrawal Shortfall — (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._withdrawFromAave` caps every Aave redemption at `totalETHDepositedToAave` (the tracked principal), ignoring accrued interest. When a user's withdrawal shortfall exceeds the remaining principal but is within the full aWETH balance (principal + interest), `_withdrawFromAave` silently withdraws less than requested, causing the post-withdrawal balance check in `_processWithdrawalCompletion` to revert with `InsufficientLiquidityForWithdrawal`. The user's rsETH has already been burned; their ETH is frozen until operator intervention.

## Finding Description
In `_withdrawFromAave` (L905–921), the cap is applied as:

```solidity
uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave
    ? aaveBalance
    : totalETHDepositedToAave;
withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
```

Once Aave interest accrues, `aaveBalance > totalETHDepositedToAave`, so `withdrawablePrincipal = totalETHDepositedToAave`. If `amount > totalETHDepositedToAave`, `withdrawnAmount` is capped below `amount`.

Back in `_processWithdrawalCompletion` (L719–731):

```solidity
uint256 balanceAfter = address(this).balance;
if (balanceAfter < request.expectedAssetAmount) {
    revert InsufficientLiquidityForWithdrawal();
}
```

Because `withdrawnAmount < amountNeeded`, `balanceAfter` remains below `request.expectedAssetAmount` and the revert fires unconditionally. `_collectInterestToTreasury` does not fix this — it only reduces `aaveBalance` back to the principal, leaving the user in the same blocked state.

## Impact Explanation
**Medium — Temporary freezing of funds.** The user's rsETH is burned at `unlockQueue` time; they cannot re-enter the queue. Their ETH is inaccessible until an operator calls `emergencyWithdrawFromAave` or deposits additional ETH to raise `totalETHDepositedToAave`. This matches the allowed impact class "Temporary freezing of funds."

## Likelihood Explanation
Aave interest accrues continuously and automatically from the moment the integration is active. No attacker action is required; the condition arises in normal operation. Any user whose withdrawal shortfall falls in the range `(totalETHDepositedToAave, aaveAWETH.balanceOf(address(this))]` is affected. The longer the integration runs, the wider this range becomes, increasing the number of affected users over time.

## Recommendation
Remove the principal cap when servicing user withdrawals. Allow the full `aaveBalance` to be used:

```solidity
withdrawnAmount = amount > aaveBalance ? aaveBalance : amount;
```

Update `totalETHDepositedToAave` to not underflow when interest-inclusive withdrawals exceed the tracked principal (e.g., clamp to zero). Interest accounting for the treasury should be handled separately via `_collectInterestToTreasury`, which already operates independently of `_withdrawFromAave`.

## Proof of Concept
1. Aave integration enabled. 100 ETH deposited → `totalETHDepositedToAave = 100 ETH`.
2. Aave accrues 5 ETH interest → `aaveAWETH.balanceOf(address(this)) = 105 ETH`.
3. `address(this).balance = 0`.
4. User has an unlocked withdrawal request for 101 ETH (rsETH already burned).
5. User calls `completeWithdrawal(ETH_TOKEN, ...)`:
   - `amountNeeded = 101 - 0 = 101 ETH`
   - `_withdrawFromAave(101)`: `withdrawablePrincipal = min(105, 100) = 100`, `withdrawnAmount = min(101, 100) = 100`
   - `balanceAfter = 100 ETH < 101 ETH` → `revert InsufficientLiquidityForWithdrawal`
6. 105 ETH was available in Aave (4 ETH more than needed). The user's funds are frozen.

A Foundry fork test against Aave v3 mainnet can reproduce this by: deploying the contract, depositing 100 ETH to Aave, fast-forwarding blocks to accrue interest, queuing and unlocking a 101 ETH withdrawal, then asserting that `completeWithdrawal` reverts despite sufficient Aave balance.
