### Title
Caller-Controlled `EvmFinality::Latest` Allows MPC Threshold Signature Over Reorg-Susceptible Foreign Chain State — (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

### Summary
Any unprivileged caller can submit a `verify_foreign_transaction` request specifying `EvmFinality::Latest`. The EVM inspector verifies the transaction against the current "latest" (unfinalized) block and, if the block is canonical at that instant, the MPC network issues a valid threshold signature attesting to the transaction. Because `Latest` blocks carry no finality guarantee, a reorg occurring after the signature is issued leaves the network holding a valid, irrevocable threshold signature for a transaction that no longer exists on the canonical chain — enabling invalid bridge execution.

### Finding Description

The `EvmFinality` enum exposed in the public contract API includes three variants:

```rust
pub enum EvmFinality {
    Latest,   // ← no finality guarantee
    Safe,
    Finalized,
}
``` [1](#0-0) 

The `verify_foreign_transaction` contract entry point accepts any `EvmFinality` value from any caller without restriction and enqueues the request for MPC processing. [2](#0-1) 

Inside the EVM inspector, `verify_finality_level` maps `EvmFinality::Latest` directly to `FinalityTag::Latest` and fetches the current latest block head. The only check performed is `head.number >= receipt_block_number`:

```rust
let finality_tag = match finality {
    EthereumFinality::Finalized => FinalityTag::Finalized,
    EthereumFinality::Safe      => FinalityTag::Safe,
    EthereumFinality::Latest    => FinalityTag::Latest,   // ← no reorg protection
};
// ...
if head.number < receipt_block_number {
    return Err(ForeignChainInspectionError::NotFinalized);
}
Ok(())
``` [3](#0-2) 

The subsequent `verify_block_is_canonical` call confirms the block is canonical **at the moment of the RPC call**, but this is a point-in-time snapshot — it provides no guarantee the block will remain canonical after the signature is issued. [4](#0-3) 

The signed payload (`ForeignTxSignPayloadV1`) commits to the request (including `finality: Latest`) and the extracted values (e.g., `BlockHash`). Once the threshold signature is produced, it is irrevocable — there is no on-chain mechanism to invalidate it if the underlying block is later reorged. [5](#0-4) 

The same structural weakness applies to `SolanaFinality::Processed` (leader-processed, not cluster-confirmed) and `SolanaFinality::Confirmed` (supermajority-confirmed but not rooted/finalized), which are accepted by the contract ABI. [6](#0-5) 

### Impact Explanation

The primary use case documented for `verify_foreign_transaction` is the **Omnibridge inbound flow** (foreign chain → NEAR), where the MPC signature serves as the authoritative attestation that a foreign deposit transaction finalized. [7](#0-6) 

If an attacker submits a bridge deposit on an EVM chain and immediately calls `verify_foreign_transaction` with `EvmFinality::Latest`, the MPC network signs an attestation for the deposit. If the deposit block is subsequently reorged (naturally or via a targeted reorg attack on chains with weaker consensus such as BNB, Polygon, or Abstract), the attacker holds a valid threshold signature for a transaction that no longer exists. They can redeem bridge funds on NEAR while recovering their original deposit on the foreign chain — a double-spend.

This is **forged foreign-chain verification causing invalid bridge execution**, matching the High impact tier.

### Likelihood Explanation

- On Ethereum post-merge, 1–2 block reorgs occur rarely but are not zero. The window between MPC signing (which happens quickly after the request is enqueued) and a reorg is narrow but real.
- On the other supported EVM chains (BNB, Polygon, Abstract, HyperEVM, Arbitrum, Base), consensus is weaker or block times shorter, making short reorgs more frequent.
- The attacker does not need any privileged access — `verify_foreign_transaction` is a public payable method.
- The attacker controls the `finality` field in the request; no on-chain validation rejects `Latest`.

Likelihood: **Medium** (requires a reorg after signing, which is chain-dependent but structurally possible on all supported EVM chains).

### Recommendation

1. **Reject `EvmFinality::Latest` at the contract level** for `ForeignTx`-purpose domains. Only `Safe` and `Finalized` provide meaningful reorg resistance on EVM chains. Add a validation step in `verify_foreign_transaction` (or in the node's request handler) that rejects requests whose finality level falls below a configurable minimum for each chain.

2. **Reject `SolanaFinality::Processed` and `SolanaFinality::Confirmed`** for bridge-critical domains. Only `Finalized` (rooted) provides the irreversibility guarantee needed for bridge attestations.

3. If `Latest` must remain available for non-bridge use cases, enforce **domain-level finality policy**: add a `minimum_finality` field to `DomainConfig` so governance can mandate `Finalized` for `ForeignTx` domains used by the Omnibridge while allowing weaker finality for other applications.

### Proof of Concept

1. Attacker deposits 10 ETH into the Omnibridge contract on an EVM chain (e.g., BNB).
2. Attacker immediately calls `verify_foreign_transaction` on the NEAR MPC contract with:
   ```json
   {
     "request": {
       "Bnb": {
         "tx_id": "<deposit_tx_hash>",
         "finality": "Latest",
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": <foreign_tx_domain>,
     "payload_version": "V1"
   }
   ```
3. MPC nodes query the BNB RPC, find the transaction in the latest block, pass `verify_finality_level` (latest block ≥ receipt block) and `verify_block_is_canonical` (block is canonical at query time), and produce a threshold signature.
4. A 1-block reorg on BNB removes the deposit transaction from the canonical chain. The attacker's 10 BNB is returned to their wallet.
5. Attacker submits the MPC signature to the NEAR Omnibridge contract, which accepts it as proof of deposit and releases 10 ETH-equivalent tokens on NEAR.
6. Net result: attacker holds both the original 10 BNB and the bridged 10 ETH-equivalent — a double-spend enabled by the MPC network signing over unfinalized foreign chain state.

    /// Checks that the receipt's block has reached the requested finality level — i.e. that the
    /// head of the chain at `finality` is at or past `receipt_block_number`.
    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
    }
