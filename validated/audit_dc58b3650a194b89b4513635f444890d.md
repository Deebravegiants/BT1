### Title
`get_nonce(Pending, addr)` Returns `CONTRACT_NOT_FOUND` Instead of `Nonce(0)` for Pending-Deployed Contracts with No Nonce Entry — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

`execution_utils::get_nonce_at` only consults the `nonces` map from the pending state diff. It does not check `deployed_contracts`. When a contract is deployed in the pending block but has no explicit nonce entry (nonce is 0, the default, and is omitted from the diff), the function falls through to the accepted-block state reader, which returns `None`. The RPC then converts `None` to `CONTRACT_NOT_FOUND`, which is incorrect — the contract exists in the pending state with nonce 0.

---

### Finding Description

**Call chain:**

`get_nonce` (api_impl.rs:681) reads only the `nonces` map from pending data:

``` [1](#0-0) 

It passes this to `execution_utils::get_nonce_at`, which checks only the nonces map — if the address is absent, it falls through to the accepted-block storage reader:

```rust

    if let Some(nonce) = pending_nonces.get(&contract_address) {
        return Ok(Some(*nonce));
    }
}
txn.get_state_reader()?.get_nonce_at(state_number, &contract_address)
``` [2](#0-1) 

For a contract only in the pending `deployed_contracts` list (not yet in accepted state), the storage reader returns `None`. Back in `get_nonce`, `None` is mapped to `CONTRACT_NOT_FOUND`: [3](#0-2) 

**Contrast with `get_class_hash_at`**, which correctly walks `deployed_contracts` and `replaced_classes` in the pending diff before falling through: [4](#0-3) 

`get_nonce_at` has no equivalent check for `deployed_contracts`.

**The existing test** (labeled "Ask for nonce in pending block where the contract is deployed in the pending block") does not exercise this gap — it inserts an explicit nonce entry for the new address before calling the RPC, so the nonces-map lookup succeeds and the bug is never triggered: [5](#0-4) 

---

### Impact Explanation

Any unprivileged RPC caller can trigger this by querying `starknet_getNonce` with `block_id = pending` for a contract address that appears in `pending_data.state_update.state_diff.deployed_contracts` but has no entry in `pending_data.state_update.state_diff.nonces`. This is the normal case for any freshly deployed contract (nonce 0 is the default and is not written to the diff). The RPC returns `CONTRACT_NOT_FOUND` instead of `Nonce(0)`, which is an incorrect authoritative pending-view response. Wallets and clients relying on this endpoint to determine the nonce for a just-deployed account contract will receive an error and be unable to construct or submit the first transaction from that account during the pending window.

The claim that this directly corrupts mempool admission or blockifier account validation is **overstated** — those paths use internal state readers, not the RPC endpoint. The confirmed impact is: **RPC pending view returns an authoritative-looking wrong value (High per scope rules)**.

---

### Likelihood Explanation

High. Every `DEPLOY_ACCOUNT` transaction in the pending block produces exactly this state: the new contract address appears in `deployed_contracts` with nonce 0, and no nonce entry is written to the diff. Any caller querying the nonce of such an address against the pending block hits the bug deterministically.

---

### Recommendation

In `execution_utils::get_nonce_at`, add a check analogous to what `get_class_hash_at` does: accept an optional `pending_deployed_contracts` slice alongside `pending_nonces`. If the contract address is found in `deployed_contracts` (and not in `nonces`), return `Ok(Some(Nonce::default()))` instead of falling through to the accepted-block reader.

Alternatively, in `get_nonce` (api_impl.rs), pass both `nonces` and `deployed_contracts` from the pending diff, and after `get_nonce_at` returns `None`, check whether the address is in `deployed_contracts` before emitting `CONTRACT_NOT_FOUND`.

Add a unit test: deploy a contract in the pending diff with no nonce entry, call `get_nonce(Pending, addr)`, and assert `Nonce(0)` is returned, not `CONTRACT_NOT_FOUND`.

---
