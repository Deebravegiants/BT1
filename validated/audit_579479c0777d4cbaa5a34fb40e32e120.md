### Title
`psVRFKeyHashes` Dangling VRF Key Removal Uses `Map.withoutKeys` Instead of Decrement, Allowing Duplicate VRF Registration After v11 Hard Fork - (`File: eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

In `poolReapTransition`, the set of "dangling" VRF key hashes (old VRF keys being replaced by pool re-registration) is removed from `psVRFKeyHashes` using `Map.withoutKeys`, which **completely deletes** the entry regardless of its reference count. This is inconsistent with how retired pool VRF keys are handled (via `removeVRFKeyHashOccurrences`, which decrements the count). When multiple pools share a VRF key (permitted before the v11 hard fork), and one pool re-registers with a new VRF, the shared VRF key is completely erased from `psVRFKeyHashes` at the epoch boundary, even though other pools still hold it. A subsequent new pool registration can then reuse that VRF key, bypassing the `VRFKeyHashAlreadyRegistered` guard and violating the uniqueness invariant introduced in v11.

---

### Finding Description

**Vulnerability class:** Wrong key/index used to access/modify a map — the map is updated with the wrong operation (complete deletion instead of reference-count decrement), causing the map to diverge from the actual ledger state.

**Root cause — `poolReapTransition` in `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`:**

At the epoch boundary, `poolReapTransition` computes `danglingVRFKeyHashes` — the set of VRF keys that belong to pools whose current VRF differs from their staged future VRF (i.e., pools that re-registered with a new VRF key during the epoch):

```haskell
danglingVRFKeyHashes =
  Set.fromList $
    Map.elems $
      Map.merge
        Map.dropMissing
        Map.dropMissing
        ( Map.zipWithMaybeMatched $ \_ sps sppF ->
            if sps ^. spsVrfL /= sppF ^. sppVrfL then Just (sps ^. spsVrfL) else Nothing
        )
        (ps0 ^. psStakePoolsL)
        (ps0 ^. psFutureStakePoolParamsL)
```

These dangling keys are then removed from `psVRFKeyHashes` using `Map.withoutKeys`:

```haskell
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )
```

`Map.withoutKeys` unconditionally deletes the entire entry from the map, regardless of the stored count (`NonZero Word64`). This is the **wrong operation** for a reference-counted map.

By contrast, retired pool VRF keys are handled correctly via `removeVRFKeyHashOccurrences`, which decrements the count and only removes the entry when it reaches zero:

```haskell
removeVRFKeyHashOccurrence =
  -- Removes the key from the map if the value drops to 0
  Map.update (mapNonZero (\n -> n - 1))
```

**How the map gets a count > 1:** The v11 hard fork calls `populateVRFKeyHashes`, which counts all VRF keys across both `psStakePools` and `psFutureStakePoolParams`:

```haskell
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
```

Before v11, multiple pools could register with the same VRF key (the uniqueness check did not exist). After the hard fork, `psVRFKeyHashes[vrf]` is set to the number of pools sharing that VRF. The test in `HardForkSpec.hs` explicitly confirms this: three pools registered with the same VRF before v11 yield `expectVRFs [(vrf, 3)]` after the fork.

**Attack path:**

1. Before v11: Pool A (`kh1`) and Pool B (`kh2`) both register with `vrf1` (permitted).
2. v11 hard fork: `populateVRFKeyHashes` sets `psVRFKeyHashes[vrf1] = 2`.
3. Pool A re-registers with `vrf2` (allowed; `vrf2` is fresh). `psVRFKeyHashes[vrf2] = 1`. `psFutureStakePoolParams[kh1].sppVrf = vrf2`.
4. At POOLREAP: `danglingVRFKeyHashes = {vrf1}` (kh1's current VRF is `vrf1`, future is `vrf2`). `Map.withoutKeys {vrf1}` **completely removes** `vrf1` from `psVRFKeyHashes` (count was 2).
5. Pool B (`kh2`) still uses `vrf1`, but `psVRFKeyHashes` no longer contains `vrf1`.
6. Pool C (`kh3`) submits a `RegPool` certificate with `vrf1`. The check `Map.notMember vrf1 psVRFKeyHashes` is now `True`, so the transaction is accepted.
7. Pool B and Pool C now both hold `vrf1`, violating the uniqueness invariant.

The attacker controls Pool A (to trigger the dangling removal) and Pool C (to exploit the gap). These can be the same operator.

---

### Impact Explanation

**Impact: Medium.** An attacker-controlled pool registration transaction bypasses the `VRFKeyHashAlreadyRegistered` validation guard, exceeding the intended validation limits introduced in v11. Two pools end up sharing the same VRF key. Since VRF keys are used for slot leader election in Ouroboros Praos, both pools would produce identical VRF outputs for every slot, making them simultaneously eligible (or ineligible) for the same slots. This can result in two valid competing blocks for the same slot from pools controlled by the same operator, enabling a persistent fork or double-block production that honest nodes must resolve — a direct violation of the VRF uniqueness invariant the v11 hard fork was designed to enforce.

---

### Likelihood Explanation

**Likelihood: Low-Medium.** The precondition — two pools sharing a VRF key before v11 — was valid on-chain behavior and may exist in mainnet state. After the v11 hard fork activates, any pool operator who (a) shared a VRF key with another pool before v11 and (b) re-registers with a new VRF key triggers the bug at the next epoch boundary. A coordinated attacker controlling both Pool A and Pool C can execute this deterministically. The window is narrow (requires the specific pre-v11 shared-VRF state), but the steps are fully attacker-controlled once that precondition is met.

---

### Recommendation

Replace `Map.withoutKeys danglingVRFKeyHashes` with `removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)` in `poolReapTransition`. This ensures that when a pool re-registers with a new VRF, the old VRF key's reference count is decremented rather than the entry being unconditionally deleted, preserving the invariant that `psVRFKeyHashes[vrf]` equals the number of currently registered pools using `vrf`.

The corrected update should be:

```haskell
& certPStateL . psVRFKeyHashesL
  %~ removeVRFKeyHashOccurrences (retiredVRFKeyHashes ++ Set.toList danglingVRFKeyHashes)
```
