### Title
ETH Withdrawal Permanently Frozen When Recipient Contract Cannot Receive ETH - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary
`LRTWithdrawalManager._transferAsset` uses a low-level `call{value: amount}("")` to push ETH to the withdrawal recipient. If the recipient is a smart-contract wallet (e.g., Gnosis Safe) that is reconfigured after withdrawal initiation to reject ETH, every attempt to complete the withdrawal reverts and the user's ETH is permanently frozen. No admin rescue path exists.

---

### Finding Description
`_processWithdrawalCompletion` is the single code path that finalises every ETH withdrawal — both the user-facing `completeWithdrawal` and the operator-facing `completeWithdrawalForUser` route through it: [1](#0-0) 

At the end of that function the asset is pushed to the user: [2](#0-1) 

For ETH, this is a raw `call`. If `to` is a contract whose `receive()` reverts (or has none), `sent` is `false` and the function reverts with `EthTransferFailed`. Because the state mutations (`popFront`, `delete withdrawalRequests`, `unlockedWithdrawalsCount--`) all occur in the same transaction, they are also reverted, leaving the request permanently in the unlocked queue.

There is no function that lets the user redirect the payout to a different address, no `cancelWithdrawal` to reclaim rsETH, and no admin function to forcibly complete or cancel a stuck request. `sweepRemainingAssets` is also blocked because `hasUnlockedWithdrawals(asset)` remains `true`: [3](#0-2) 

The `completeWithdrawalForUser` operator path is explicitly acknowledged to be problematic for ETH but dismissed as "non-impactful": [4](#0-3) 

---

### Impact Explanation
A user whose withdrawal address is a smart-contract wallet (Gnosis Safe, upgradeable proxy, etc.) that is reconfigured — or whose `receive()` is removed — after `initiateWithdrawal` is called will have their ETH permanently frozen inside `LRTWithdrawalManager`. The rsETH was already transferred to the contract at initiation time: [5](#0-4) 

There is no way to recover either the rsETH or the underlying ETH. This matches the **Permanent freezing of funds** impact class.

---

### Likelihood Explanation
Smart-contract wallets (Gnosis Safe, account-abstraction wallets, upgradeable proxies) are common in DeFi. A user can legitimately reconfigure their Safe (change threshold, remove fallback handler, upgrade implementation) between the time they queue a withdrawal and the time it is unlocked (~8 days delay). The attacker-controlled entry path is:

1. Deploy or use an existing smart-contract wallet that accepts ETH.
2. Call `initiateWithdrawal(ETH_TOKEN, rsETHAmount, ...)` — rsETH is pulled from the wallet.
3. Reconfigure the wallet to reject ETH (remove fallback handler, upgrade to a version without `receive`).
4. After the unlock delay, every call to `completeWithdrawal` or `completeWithdrawalForUser` reverts permanently.

Likelihood is **Low** (requires a smart-contract wallet and a deliberate or accidental reconfiguration), but the impact is critical (permanent fund loss with no recovery path).

---

### Recommendation
1. **Pull-over-push**: Store the owed ETH amount in a per-user mapping and let the user call a separate `claimETH()` function. This decouples the accounting state update from the ETH delivery.
2. **Redirect / cancel**: Add an admin function that can redirect a stuck withdrawal to a new address, or cancel it and return the rsETH to the user.
3. **Wrap ETH as WETH**: Transfer WETH instead of raw ETH; ERC-20 transfers do not depend on the recipient having a `receive()` function.

---

### Proof of Concept
```
1. Alice deploys a Gnosis Safe (accepts ETH via fallback handler).
2. Alice calls initiateWithdrawal(ETH_TOKEN, 1e18, "").
   → rsETH transferred to LRTWithdrawalManager; withdrawal request created.
3. Alice removes the fallback handler from her Safe (Safe now rejects ETH).
4. Operator calls unlockQueue(ETH_TOKEN, ...) after the 8-day delay.
   → Request is unlocked; unlockedWithdrawalsCount[ETH]++.
5. Alice calls completeWithdrawal(ETH_TOKEN, "").
   → _transferAsset calls payable(alice).call{value: 1e18}("").
   → Safe has no receive(); call returns false.
   → EthTransferFailed revert; all state changes rolled back.
6. Operator calls completeWithdrawalForUser(ETH_TOKEN, alice, "").
   → Same revert.
7. No other function exists to rescue the ETH.
   → Alice's ETH is permanently frozen in LRTWithdrawalManager.
``` [2](#0-1) [1](#0-0)

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }

        if (asset == LRTConstants.ETH_TOKEN) {
