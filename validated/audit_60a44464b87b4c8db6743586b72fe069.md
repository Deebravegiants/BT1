### Title
State Mutation Before Deposit Check in `submit_participant_info` Allows Free Attestation Registration - (File: `crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` mutates `tee_state.stored_attestations` before verifying the caller's deposit is sufficient. In NEAR SDK, `#[handle_result]` functions that return `Err` **commit state changes** (only `panic!` rolls back state). An attacker with a valid TEE attestation can register their node for free, breaking the storage-cost accounting invariant and draining the contract's NEAR balance.

---

### Finding Description

In `submit_participant_info` (`crates/contract/src/lib.rs`, lines 756–852), the execution order is:

1. **Line 780**: `initial_storage = env::storage_usage()` — snapshot taken.
2. **Lines 796–815**: `self.tee_state.add_participant(...)` — **state is mutated**; the attestation is inserted into `stored_attestations` (`tee_state.rs` line 191–197).
3. **Lines 826–841**: Deposit sufficiency check — if `attached < cost`, the function returns `Err(InsufficientDeposit)`.

Because the function is annotated `#[handle_result]` (line 759), returning `Err` does **not** roll back the state change made in step 2. The NEAR runtime only rolls back state on `panic!`; a graceful `Err` return commits all prior mutations. The deposit check is therefore ineffective: the attestation is permanently stored regardless of whether the caller paid. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Every byte stored in NEAR contract state is backed by the contract account's NEAR balance (storage staking). When `add_participant` inserts a `NodeAttestation` entry without the caller paying, the contract's own NEAR balance is silently reduced to cover the storage cost. Repeated free registrations (each ~hundreds of bytes) cumulatively drain the contract's balance, eventually threatening its ability to accept new storage and breaking the invariant that callers pay for their own state footprint.

Additionally, a successfully registered attestation in `tee_state.stored_attestations` is a prerequisite for passing `assert_caller_is_attested_participant_and_protocol_active`, which gates `respond`, `respond_ckd`, `vote_pk`, `vote_reshared`, and `start_keygen_instance`. While participant-set membership (a separate governance vote) is also required, free attestation registration removes one of the two barriers. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

`submit_participant_info` is a public, unprivileged endpoint callable by any NEAR account. The only prerequisite is a valid TEE attestation (a real Intel TDX quote in production). Any node operator running TEE hardware — including a Byzantine participant below the signing threshold — can exploit this on every registration or re-registration cycle. The call pattern is indistinguishable from a legitimate submission, so it is undetectable without off-chain monitoring of deposit amounts. [6](#0-5) 

---

### Recommendation

Move the deposit check **before** the state-mutating `add_participant` call, or replace the `return Err(...)` path with `env::panic_str(...)` so that NEAR's runtime rolls back the storage insertion on failure. The pattern used in `require_deposit` (which panics on insufficient deposit) is the correct model:

```rust
// 1. Check deposit FIRST (panics on failure → state rollback guaranteed)
let initial_storage = env::storage_usage();
// ... validate inputs ...

// 2. Only then mutate state
let attestation_insertion_result = self.tee_state.add_participant(...)?;

// 3. Compute and verify cost
let storage_used = env::storage_usage().saturating_sub(initial_storage);
let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
let attached = env::attached_deposit();
if attached < cost {
    env::panic_str(&InvalidParameters::InsufficientDeposit { ... }.to_string());
}
``` [7](#0-6) 

---

### Proof of Concept

1. Attacker holds a valid TEE attestation (Intel TDX quote) and calls:
   ```
   submit_participant_info(valid_attestation, tls_public_key)
   // attached deposit = 0 yoctoNEAR
   ```
2. Execution reaches `add_participant` (line 796); the attestation is inserted into `stored_attestations`. `env::storage_usage()` increases by ~N bytes.
3. `storage_used = N`, `cost = N × storage_byte_cost` (e.g., ~10^19 yoctoNEAR/byte × 500 bytes ≈ 5×10^21 yoctoNEAR = 0.005 NEAR).
4. `attached (0) < cost` → function returns `Err(InsufficientDeposit)`.
5. **The attestation entry persists in `stored_attestations`**; the contract's balance is reduced by the storage cost with no payment from the attacker.
6. Repeating this with distinct TLS keys accumulates unbounded free storage at the contract's expense. [8](#0-7) [9](#0-8)

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}

    /// Adds a participant attestation for the given node iff the attestation succeeds verification.
    pub(crate) fn add_participant(
        &mut self,
        node_id: NodeId,
        attestation: Attestation,
        tee_upgrade_deadline_duration: Duration,
    ) -> Result<ParticipantInsertion, AttestationSubmissionError> {
        let expected_report_data: ReportData = ReportDataV1::new(
            *node_id.tls_public_key.as_bytes(),
            *node_id.account_public_key.as_bytes(),
        )
        .into();

        let accepted_measurements = self.get_accepted_measurements();
        // TODO(#3264): run DCAP in the verifier contract (Promise + callback) and
        // do the post-DCAP checks here, instead of verifying locally in-WASM.
        let AcceptedAttestation {
            attestation: verified_attestation,
            advisory_ids,
        } = attestation.verify_locally(
            expected_report_data.into(),
            Self::current_time_seconds(),
            &self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration),
            &self.get_allowed_launcher_compose_hashes(),
            &accepted_measurements,
        )?;

        log_informational_advisory_ids(&advisory_ids);

        let tls_pk = node_id.tls_public_key.clone();

        // Authorization: a TLS key registered to one account must not be
        // overwritten by a submission from a different account. Without this,
        // any caller could replace any participant's stored attestation, since
        // the entry is keyed only by `tls_public_key`.
        if let Some(existing) = self.stored_attestations.get(&tls_pk)
            && existing.node_id.account_id != node_id.account_id
        {
            return Err(AttestationSubmissionError::TlsKeyOwnedByOtherAccount);
        }

        let insertion = self.stored_attestations.insert(
            tls_pk,
            NodeAttestation {
                node_id,
                verified_attestation,
            },
        );

        Ok(match insertion {
            Some(_previous_attestation) => ParticipantInsertion::UpdatedExistingParticipant,
            None => ParticipantInsertion::NewlyInsertedParticipant,
        })
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
