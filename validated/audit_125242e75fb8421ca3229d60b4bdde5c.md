Audit Report

## Title
Missing Chainlink Staleness Check Allows Stale Price to Drive Incorrect rsETH Minting — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but discards `updatedAt` and `answeredInRound`, performing no freshness validation. A stale Chainlink price propagates directly into `LRTDepositPool.getRsETHAmountToMint`, causing depositors to receive an rsETH amount computed from an outdated asset/ETH rate rather than the current fair rate.

## Finding Description

In `ChainlinkPriceOracle.getAssetPrice` (line 52), `latestRoundData()` is called with all staleness fields silently discarded:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

`updatedAt` and `answeredInRound` are never checked. If the Chainlink feed stops updating, the oracle continues returning the last known price indefinitely.

This stale price flows directly into the minting formula in `LRTDepositPool.getRsETHAmountToMint`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

`getAssetPrice(asset)` is fetched live from the (potentially stale) Chainlink feed, while `rsETHPrice` is a stored value last computed when `updateRSETHPrice()` was called. These two values can diverge when:

1. `updateRSETHPrice()` is called at time T₀ using asset price P₀ → `rsETHPrice` is set.
2. The Chainlink feed updates to P₁ (P₁ ≠ P₀) at time T₁ > T₀.
3. The feed goes stale at P₁ (no further updates).
4. A depositor calls `depositAsset` at time T₂ > T₁: `getAssetPrice()` returns P₁, but `rsETHPrice` was computed with P₀.

Result: `rsethAmountToMint = amount × P₁ / rsETHPrice(P₀)`. If P₁ > P₀, the depositor receives more rsETH than fair value (diluting existing holders). If P₁ < P₀, the depositor receives fewer rsETH than fair value.

The full call chain is confirmed:

- `depositAsset` → `_beforeDeposit` [3](#0-2) 
- → `getRsETHAmountToMint` [4](#0-3) 
- → `LRTOracle.getAssetPrice` [5](#0-4) 
- → `ChainlinkPriceOracle.getAssetPrice` with no staleness check [1](#0-0) 

The `minRSETHAmountExpected` slippage guard in `_beforeDeposit` only protects the depositor from receiving fewer rsETH than their own minimum — it does not prevent over-minting that dilutes existing holders, and it does not enforce any price freshness invariant.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A depositor calling `depositAsset` when the Chainlink feed is stale receives an rsETH amount computed from an outdated price rather than the current fair rate. The deposited LST tokens are not lost, but the rsETH minted does not accurately reflect the current asset/ETH exchange rate, violating the protocol's core minting invariant. In the over-minting direction, existing rsETH holders are diluted.

## Likelihood Explanation

No attacker action is required. The condition arises passively from Chainlink infrastructure failure (node outage, heartbeat miss, feed deprecation). Chainlink feeds for major LSTs (stETH, rETH, cbETH) have heartbeats of 1–24 hours; a >24-hour staleness is a realistic failure mode. Any depositor transacting during the stale window is affected. The divergence between the stale `getAssetPrice` return and the stored `rsETHPrice` (computed at a different time) is the necessary and sufficient condition.

## Recommendation

Add a staleness check in `ChainlinkPriceOracle.getAssetPrice` using the `updatedAt` return value:

```solidity
(, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
if (block.timestamp - updatedAt > STALENESS_THRESHOLD) {
    revert StalePrice();
}
```

A configurable `STALENESS_THRESHOLD` per asset (matching each feed's documented heartbeat) is recommended. Optionally, also check `answeredInRound >= roundId` to guard against incomplete rounds.

## Proof of Concept

Foundry fork test outline:

1. Fork mainnet with deployed `LRTDepositPool`, `LRTOracle`, `ChainlinkPriceOracle`, and a supported LST (e.g., stETH).
2. Call `updateRSETHPrice()` to snapshot `rsETHPrice` at the current Chainlink price P₀.
3. Use `vm.mockCall` to make `latestRoundData()` return a different price P₁ with an `updatedAt` timestamp 48 hours in the past (simulating a stale feed that last updated at P₁ ≠ P₀).
4. Call `pool.getRsETHAmountToMint(stETH, 1e18)` — it returns a value computed from P₁ / rsETHPrice(P₀), not the fair rate.
5. Call `pool.depositAsset(stETH, 1e18, 0, "")` — it succeeds with no staleness revert, minting the incorrect amount.
6. Assert no revert occurred and that the minted amount differs from the fair-rate amount, confirming the missing guard at `ChainlinkPriceOracle.sol` line 52.
