Audit Report

## Title
Aave V3 Pool Pause Permanently Blocks ETH Withdrawal Completion With No Protocol-Level Recovery Path - (File: contracts/LRTWithdrawalManager.sol)

## Summary
When `isAaveIntegrationEnabled` is `true` and the Aave V3 pool is paused by its guardian, every call to `completeWithdrawal()` for ETH reverts because `_withdrawFromAave()` unconditionally calls `aaveWETHGateway.withdrawETH()`, which propagates Aave's `RESERVE_PAUSED` revert. All three administrative recovery paths — `emergencyWithdrawFromAave()`, `setAaveIntegrationEnabled(false)`, and `configureAaveIntegration()` — also invoke `_withdrawFromAave()` before the integration flag is cleared, so none can succeed while Aave is paused. Users whose rsETH has already been burned via `unlockQueue()` cannot receive their ETH for the duration of the pause.

## Finding Description

`_withdrawFromAave()` at line 917 unconditionally calls `aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this))` with no guard against a paused Aave pool:

```solidity
// L905-921
function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
    ...
    aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
    ...
}
```

`_processWithdrawalCompletion()` at lines 720–731 calls `_withdrawFromAave()` whenever the contract's idle ETH balance is insufficient, which is the normal state when all ETH has been deposited to Aave:

```solidity
// L720-731
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
    uint256 contractBalance = address(this).balance;
    if (contractBalance < request.expectedAssetAmount) {
        uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
        _withdrawFromAave(amountNeeded);
        ...
    }
}
```

The critical design flaw is that `setAaveIntegrationEnabled(false)` sets `isAaveIntegrationEnabled = false` at line 503 **only after** `_withdrawFromAave()` succeeds at line 495. If the withdrawal reverts (Aave paused), the flag remains `true` and the function reverts before reaching line 503. There is no code path that sets `isAaveIntegrationEnabled = false` without first successfully withdrawing from Aave.

`emergencyWithdrawFromAave()` at line 560 also calls `_withdrawFromAave(amount)` directly and does not modify `isAaveIntegrationEnabled`, providing no escape hatch.

## Impact Explanation

This constitutes **temporary freezing of funds** (Medium severity). Users who have initiated ETH withdrawals and whose requests have been unlocked — meaning their rsETH has already been burned in `unlockQueue()` — cannot call `completeWithdrawal()` for the entire duration of the Aave pause. The freeze is bounded by the Aave pause duration but is total: every in-flight ETH withdrawal is blocked with no protocol-level workaround.

## Likelihood Explanation

Aave V3 has a guardian role that has exercised its pause capability on mainnet in response to security incidents. The `WETH_ADDRESS` constant is hardcoded to the mainnet WETH address, confirming this integration targets mainnet. No attacker action is required; the condition is triggered by Aave's own guardian acting within its designed role. Any Aave pause event, however brief, activates the freeze for all concurrent ETH withdrawers.

## Recommendation

1. In `setAaveIntegrationEnabled(false)`, move `isAaveIntegrationEnabled = false` to **before** the `_withdrawFromAave()` call. This allows the flag to be cleared even when Aave is paused, unblocking future `completeWithdrawal()` calls.
2. Add a dedicated emergency flag-reset function callable by `PAUSER_ROLE` that sets `isAaveIntegrationEnabled = false` without any Aave interaction, for use when Aave is paused.
3. In `_processWithdrawalCompletion()`, check `IPool(aavePool).paused()` before calling `_withdrawFromAave()` and skip the Aave path (or revert with a descriptive error) when the pool is paused.
4. Optionally wrap the `_withdrawFromAave()` call in `_processWithdrawalCompletion()` in a `try/catch` to allow serving from idle ETH balance when available.

## Proof of Concept

1. Manager calls `setAaveIntegrationEnabled(true)` — `isAaveIntegrationEnabled` is `true`.
2. Manager calls `unlockQueue(ETH_TOKEN, ...)` — ETH is pulled from `LRTUnstakingVault` and auto-deposited to Aave; contract idle ETH balance is 0, `totalETHDepositedToAave > 0`.
3. User calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, ...)` — rsETH transferred to contract, request queued.
4. Aave V3 guardian pauses the pool.
5. User calls `completeWithdrawal(ETH_TOKEN, ...)`:
   - `_processWithdrawalCompletion()` entered; `address(this).balance < request.expectedAssetAmount` → true.
   - `_withdrawFromAave(amountNeeded)` called → `aaveWETHGateway.withdrawETH(aavePool, ...)` reverts with `RESERVE_PAUSED`.
   - Transaction reverts. User cannot receive ETH.
6. `PAUSER_ROLE` calls `emergencyWithdrawFromAave(type(uint256).max)` → same revert at line 560.
7. Manager calls `setAaveIntegrationEnabled(false)` → `_withdrawFromAave()` called at line 495 before line 503 → same revert; `isAaveIntegrationEnabled` remains `true`.
8. All ETH withdrawal completions remain blocked for the duration of the Aave pause. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
