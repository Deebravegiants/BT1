Audit Report

## Title
Stale Cached `rsETHPrice` Allows Depositors to Mint Excess rsETH, Stealing Accrued Yield from Existing Holders - (File: contracts/LRTDepositPool.sol)

## Summary

`LRTOracle` stores `rsETHPrice` as a persistent state variable that is only updated when `updateRSETHPrice()` is explicitly called. `LRTDepositPool.getRsETHAmountToMint()` divides by this cached value to compute mint amounts, and neither `depositETH()` nor `depositAsset()` refreshes the price before minting. When staking rewards cause the true rsETH/ETH rate to exceed the stored value, any depositor can mint more rsETH than warranted, permanently diluting existing holders' accrued yield.

## Finding Description

`LRTOracle` holds `rsETHPrice` as a storage variable (`contracts/LRTOracle.sol` L28) updated only via `_updateRsETHPrice()` (`contracts/LRTOracle.sol` L214–313), which is triggered by the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`.

`LRTDepositPool.getRsETHAmountToMint()` (`contracts/LRTDepositPool.sol` L506–521) computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`depositETH()` (`contracts/LRTDepositPool.sol` L76–93) and `depositAsset()` (`contracts/LRTDepositPool.sol` L99–118) both call `_beforeDeposit` → `getRsETHAmountToMint` without ever calling `updateRSETHPrice()` first.

When the true backing per rsETH rises above `rsETHPrice` (due to accrued staking rewards), the stale denominator is too small, minting excess rsETH. After the price is eventually updated, `newRsETHPrice = totalETHInProtocol / rsethSupply` is lower than it would have been without the dilutive deposit, permanently reducing the value of every existing holder's position.

The `pricePercentageLimit` guard (`contracts/LRTOracle.sol` L252–266) creates a guaranteed multi-block staleness window: when the true price increase exceeds the configured threshold, `updateRSETHPrice()` reverts with `PriceAboveDailyThreshold` for any non-manager caller. During this window the price is frozen at the stale value and only a manager can advance it, while deposits continue to execute at the stale price.

## Impact Explanation

**High — Theft of unclaimed yield.** Existing rsETH holders have accrued yield reflected in the true rsETH/ETH rate exceeding the stored `rsETHPrice`. A depositor who deposits before `updateRSETHPrice()` is called receives more rsETH than the true rate warrants. When the price is eventually updated, the inflated supply permanently dilutes existing holders. The attacker can redeem or sell the excess rsETH at the true (higher) price, capturing yield that belonged to existing holders. This matches the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation

Staking rewards accrue continuously, so any gap between `updateRSETHPrice()` calls creates an exploitable window. The `pricePercentageLimit` scenario makes this a guaranteed multi-block window: when the true price increase exceeds the threshold, non-managers cannot call `updateRSETHPrice()` (it reverts), yet `depositETH`/`depositAsset` remain open. MEV bots have a direct financial incentive to deposit during this window rather than trigger the price update. The exploit requires no special privileges — only a standard ETH or LST deposit.

## Recommendation

Call `updateRSETHPrice()` at the start of `depositETH()` and `depositAsset()` before computing `rsethAmountToMint`:

```solidity
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
+   ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    emit ETHDeposit(msg.sender, msg.value, rsethAmountToMint, referralId);
}
```

Apply the same fix to `depositAsset()`. This ensures the rsETH price always reflects current accrued yield before new shares are minted. Note that when `pricePercentageLimit` is active and the price increase exceeds the threshold, `updateRSETHPrice()` will revert for non-managers, which would also block deposits — this is the correct safe behavior (deposits should not proceed at a stale price).

## Proof of Concept

1. Staking rewards accrue; true rsETH price rises from `P_stale = 1.00e18` to `P_true = 1.05e18`. `updateRSETHPrice()` has not been called (or `pricePercentageLimit` prevents non-manager calls).
2. Attacker calls `depositETH{value: 100 ether}(0, "")`.
3. `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.00e18 = 100e18` rsETH instead of the correct `100e18 * 1e18 / 1.05e18 ≈ 95.24e18` rsETH.
4. Attacker receives `≈ 4.76 rsETH` excess.
5. Manager calls `updateRSETHPriceAsManager()`; new price = `(totalETH + 100) / (supply + 100)` which is lower than `P_true`, permanently diluting existing holders.
6. Attacker redeems or sells the excess `≈ 4.76 rsETH` at `P_true`, capturing `≈ 5 ETH` of yield belonging to existing holders.

**Foundry fork test plan:** Fork mainnet, snapshot state with accrued rewards (so true price > stored `rsETHPrice`), record existing holder's ETH-equivalent balance, execute attacker deposit, call `updateRSETHPriceAsManager`, assert existing holder's ETH-equivalent balance decreased and attacker's rsETH balance exceeds fair-value mint amount.
