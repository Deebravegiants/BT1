Audit Report

## Title
Missing Zero-Output Guard Allows Silent Loss of Dust Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary
Every L2 pool contract accepts ERC-20 token deposits and computes the rsETH output via integer division in `viewSwapRsETHAmountAndFee`. When the deposited amount is small enough that `amountAfterFee * tokenToETHRate / rsETHToETHrate` truncates to zero, the contract silently accepts the user's tokens and transfers or mints 0 rsETH in return. No minimum-output guard exists anywhere in the deposit path, unlike `LRTDepositPool.depositAsset` which enforces a `minRSETHAmountExpected` parameter.

## Finding Description
In `RSETHPool.deposit` (lines 284–305), the only input validation is `if (amount == 0) revert InvalidAmount()`. After transferring tokens from the user, `viewSwapRsETHAmountAndFee(amount, token)` is called. Inside that function (lines 326–347), the output is computed as:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

Solidity integer division truncates toward zero. When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, `rsETHAmount` evaluates to `0`. The deposit function does not check for a zero output; it proceeds to call `IERC20(address(wrsETH)).safeTransfer(msg.sender, 0)` (line 302), which succeeds silently. The user's tokens are permanently held by the pool with no rsETH issued.

The identical pattern is confirmed in:
- `RSETHPoolNoWrapper.sol` lines 250–271 and 292–312
- `RSETHPoolV3.sol` lines 271–293 and 315–335
- `RSETHPoolV3ExternalBridge.sol` lines 390–412 and 433–452
- `RSETHPoolV3WithNativeChainBridge.sol` lines 307–329 and 351–371

The existing `amount == 0` guard is insufficient because it only blocks a zero-input deposit, not a zero-output one. `LRTDepositPool.depositAsset` (lines 99–118) demonstrates the correct pattern via `minRSETHAmountExpected`, but this guard was never ported to the L2 pool contracts.

## Impact Explanation
A depositor who sends a dust amount of a supported ERC-20 token will have their tokens transferred into the pool and receive 0 rsETH in return. The tokens are not refunded; they accumulate in the pool balance and are eventually bridged to L1 as protocol revenue. The user suffers a complete loss of the deposited dust amount. For ETH-pegged LSTs where `tokenToETHRate ≈ rsETHToETHrate ≈ 1e18`, the maximum losable amount per call is approximately 1 wei. The contract fails to deliver its promised exchange, matching the allowed impact: **Low — Contract fails to deliver promised returns**.

## Likelihood Explanation
Any unprivileged depositor can trigger this by calling `deposit(token, amount, referralId)` with a sufficiently small `amount`. No special role, no front-running, and no second error is required. The path is fully reachable with zero barriers. It can occur accidentally via UI rounding errors, test transactions, or contract integrations computing residual amounts. Likelihood is low given that normal users deposit meaningful amounts, but the path is unconditionally reachable.

## Recommendation
Add a zero-output guard immediately after computing `rsETHAmount` in every `deposit` function across all five pool contracts:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount(); // or a dedicated ZeroRsETHOutput() error
```

Alternatively, add a `minRsETHAmountExpected` parameter to each `deposit` function, mirroring the pattern already used in `LRTDepositPool.depositAsset`.

## Proof of Concept
Assume `rsETHToETHrate = 1.05e18`, `tokenToETHRate = 1e18`, `feeBps = 0`.

Call on any pool variant (e.g., `RSETHPoolNoWrapper`):
```
deposit(wstETH, 1, "")
```

Execution trace:
1. `amount = 1` passes `if (amount == 0)` check. [1](#0-0) 
2. `safeTransferFrom(msg.sender, address(this), 1)` — 1 wei of wstETH leaves the user. [2](#0-1) 
3. `fee = 1 * 0 / 10_000 = 0`; `amountAfterFee = 1`.
4. `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation). [3](#0-2) 
5. `rsETH.safeTransfer(msg.sender, 0)` — user receives nothing. [4](#0-3) 
6. Transaction succeeds; user has permanently lost 1 wei of wstETH.

A Foundry fuzz test parameterizing `amount` over `[1, rsETHToETHrate/tokenToETHRate - 1]` would confirm the zero-output condition across all five pool contracts. [5](#0-4)

    function viewSwapRsETHAmountAndFee(
        uint256 amount,
        address token
    )
        public
        view
        onlySupportedToken(token)
        returns (uint256 rsETHAmount, uint256 fee)
    {
        uint256 feeBpsForToken = tokenFeeBps[token];
        fee = amount * feeBpsForToken / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // rate of token in ETH
        uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
    }
