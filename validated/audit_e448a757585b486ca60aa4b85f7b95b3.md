### Title
Gateway Stateless Validator Rejects Valid Client-Side Proving Transactions Due to `ZeroResourceBounds` Check — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` contains a check that rejects any transaction whose `max_possible_fee` equals zero. Client-side proving (SNIP-35) Invoke V3 transactions are a documented, production-supported transaction type that **requires** all `max_price_per_unit` fields to be zero (no fees are charged because proving is client-side). With the default production configuration (`validate_resource_bounds: true`, `min_gas_price: 8_000_000_000`), these valid transactions are unconditionally rejected at the gateway admission layer before they can reach the mempool or sequencer.

---

### Finding Description

`StatelessTransactionValidator::validate` calls `validate_resource_bounds` before `validate_client_side_proving_allowed`:

```
validate_contract_address
validate_empty_account_deployment_data
validate_empty_paymaster_data
validate_resource_bounds          ← fires first, rejects here
...
validate_client_side_proving_allowed  ← never reached
```

Inside `validate_resource_bounds`:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs:66-68
if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
{
    return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
}
```

`max_possible_fee` is computed as:

```
l1_gas.amount × l1_gas.price + l2_gas.amount × (l2_gas.price + tip) + l1_data_gas.amount × l1_data_gas.price
```

A valid client-side proving transaction (as documented in the README and present in `block_post_0_14_2.json`) carries:

| Field | Value |
|---|---|
| `l1_gas.max_price_per_unit` | `0x0` |
| `l2_gas.max_price_per_unit` | `0x0` |
| `l1_data_gas.max_price_per_unit` | `0x0` |
| `l2_gas.max_amount` | `0x5f5e100` (non-zero, OS gas limit) |

This yields `max_possible_fee = 0x5f5e100 × 0 = 0`, triggering `ZeroResourceBounds`.

Even if that check were removed, the immediately following check:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs:71-75
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
}
```

would also reject, because `0 < 8_000_000_000` (the production default).

The gateway's own test suite reveals the workaround: the `client_side_proving` positive-flow test case is forced to use `validate_resource_bounds: false` to avoid the rejection:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator_test.rs:148-151
#[case::client_side_proving(
    DEFAULT_VALIDATOR_CONFIG_FOR_TESTING.clone(),  // validate_resource_bounds: false
    RpcTransactionArgs { proof_facts: create_valid_proof_facts_for_testing(), proof: Proof::proof_for_testing(), ..Default::default()}
)]
```

The production default is `validate_resource_bounds: true`. There is no code path that skips the resource bounds check for client-side proving transactions.

---

### Impact Explanation

**High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

Any user submitting a valid client-side proving Invoke V3 transaction (zero prices, non-zero `l2_gas.max_amount`, non-empty `proof_facts`/`proof`) through the gateway with the default production configuration receives a `ZeroResourceBounds` or `GAS_PRICE_TOO_LOW` rejection. The transaction never reaches the mempool or sequencer. This is the exact analog of the external bug: a validation check that rejects a valid operation because a sub-component value is legitimately zero.

---

### Likelihood Explanation

**High.** The default production configuration has `validate_resource_bounds: true` and `min_gas_price: 8_000_000_000`. Client-side proving is enabled by default (`allow_client_side_proving: true`). Any user following the documented transaction format (zero prices, non-zero `l2_gas.max_amount`) will be rejected on every submission attempt.

---

### Recommendation

Add a guard in `validate_resource_bounds` to skip the zero-fee and minimum-price checks when the transaction carries client-side proof data (non-empty `proof_facts` or `proof`), mirroring the pattern used in `validate_client_side_proving_allowed`:

```rust
fn validate_resource_bounds(&self, tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
    if !self.config.validate_resource_bounds {
        return Ok(());
    }

    // Client-side proving transactions legitimately carry zero prices.
    if let RpcTransaction::Invoke(RpcInvokeTransaction::V3(invoke_tx)) = tx {
        if !invoke_tx.proof_facts.is_empty() || !invoke_tx.proof.is_empty() {
            return Ok(());
        }
    }

    let resource_bounds = *tx.resource_bounds();
    if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
        return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
    }
    if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
        return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
    }
    Ok(())
}
```

### Proof of Concept

Submit the following Invoke V3 transaction to the gateway with the default production configuration (`validate_resource_bounds: true`, `min_gas_price: 8_000_000_000`, `allow_client_side_proving: true`):

```json
{
  "type": "INVOKE",
  "version": "0x3",
  "resource_bounds": {
    "l1_gas":      {"max_amount": "0x0",       "max_price_per_unit": "0x0"},
    "l2_gas":      {"max_amount": "0x5f5e100", "max_price_per_unit": "0x0"},
    "l1_data_gas": {"max_amount": "0x0",       "max_price_per_unit": "0x0"}
  },
  "tip": "0x0",
  "proof_facts": ["0x50524f4f4630", "0x5649525455414c5f534e4f53", ...],
  "proof": "<base64-encoded proof>",
  ...
}
```

**Expected (correct) result:** Transaction accepted and forwarded to mempool.

**Actual result:** Gateway rejects with `ZeroResourceBounds` — "At least one resource bound (L1, L2, or L1 Data) must be non-zero."

The `max_possible_fee` computation in `max_possible_fee()` returns `0x5f5e100 × 0 = 0`, triggering the rejection before `validate_client_side_proving_allowed` is ever reached.

        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)

                gas_amount: resource_bounds.l2_gas.max_amount,

    pub fn max_possible_fee(&self, tip: Tip) -> Fee {
        match self {
            ValidResourceBounds::L1Gas(l1_bounds) => {
                l1_bounds.max_amount.saturating_mul(l1_bounds.max_price_per_unit)
            }
            ValidResourceBounds::AllResources(AllResourceBounds {
                l1_gas,
                l2_gas,
                l1_data_gas,
            }) => l1_gas
                .max_amount
                .saturating_mul(l1_gas.max_price_per_unit)
                .saturating_add(
                    l2_gas
                        .max_amount
                        .saturating_mul(l2_gas.max_price_per_unit.saturating_add(tip.into())),
                )
                .saturating_add(
                    l1_data_gas.max_amount.saturating_mul(l1_data_gas.max_price_per_unit),
                ),
        }
    }
