### Title
Missing Attestation Validity Check in `is_caller_an_attested_participant` Allows Nodes with Expired or Revoked TEE Attestations to Submit Signatures and Participate in Key Events - (File: crates/contract/src/tee/tee_state.rs)

---

### Summary

`is_caller_an_attested_participant` in `crates/contract/src/tee/tee_state.rs` only verifies that a stored attestation *exists* and *belongs to the caller* (ownership check), but never calls `re_verify` to confirm the attestation is still valid — i.e., not expired, image hash still in the allowlist, launcher compose hash still allowed, and measurements still accepted. This is the direct analog of the referrer bug: the referrer check confirms the referrer account exists and the authority matches, but never confirms the referrer has actually completed the required prerequisite. Here, the attestation check confirms the attestation exists and the key matches, but never confirms the attestation is still valid. As a result, a node whose TEE attestation has expired or whose image hash has been revoked can still call `respond`, `respond_ckd`, `respond_verify_foreign_tx`, `vote_pk`, `vote_reshared`, `vote_abort_key_event_instance`, `start_keygen_instance`, and `start_reshare_instance`.

---

### Finding Description

**Root cause — `is_caller_an_attested_participant` performs only an existence/ownership check:** [1](#0-0) 

The function retrieves the stored attestation for the caller's TLS key, checks that `attestation.node_id.account_id == signer_id`, and checks that `attestation.node_id.account_public_key == signer_ed25519`. It returns `Ok(())` without ever calling `re_verify` or `reverify_participants`.

**Contrast with `reverify_participants`, which performs full validity checks:** [2](#0-1) 

`reverify_participants` calls `verified_attestation.re_verify(timestamp, allowed_mpc_docker_image_hashes, allowed_launcher_compose_hashes, allowed_measurements)`, which checks expiry, image hash allowlist membership, launcher compose hash, and measurements.

**`re_verify` checks that are skipped by `is_caller_an_attested_participant`:** [3](#0-2) 

**`assert_caller_is_attested_participant_and_protocol_active` calls only the ownership check:** [4](#0-3) 

**All of the following production methods are gated exclusively by this insufficient check:**

- `respond` (line 573): [5](#0-4) 
- `respond_ckd` (line 666): [6](#0-5) 
- `respond_verify_foreign_tx` (line 705): [7](#0-6) 
- `vote_pk`, `vote_reshared`, `vote_abort_key_event_instance`, `start_keygen_instance`, `start_reshare_instance` (per the comment at line 2420–2422): [8](#0-7) 

**`reverify_and_cleanup_participants` (the correct validity check) is only called in `vote_new_parameters` and `verify_tee`:** [9](#0-8) [10](#0-9) 

**Attack path:**

1. A node is a valid participant with a stored attestation. Its attestation has `expiry_timestamp_seconds` set (e.g., 7 days from submission, per the documented resubmit cadence).
2. The attestation expires, OR the node's `mpc_image_hash` is removed from the allowlist via a governance vote (e.g., because a security vulnerability was found in that image).
3. `verify_tee` has not yet been called (it requires an explicit call from a participant; there is no automatic trigger).
4. `accept_requests` remains `true` because the overall network still has ≥ threshold valid participants — only a global flag, not per-node.
5. The node calls `respond` with a valid cryptographic signature (it still holds the key material from its TEE). `is_caller_an_attested_participant` passes because the attestation entry still exists in `stored_attestations` and the ownership fields match. The signature is accepted and the pending yield is resolved.
6. The same node can call `vote_pk` / `vote_reshared` to influence key-event outcomes, or `start_keygen_instance` / `start_reshare_instance` if it is the elected leader.

---

### Impact Explanation

The TEE attestation system's core invariant is: **only nodes running currently-authorized code inside a genuine TEE may participate in threshold signing and key events**. `is_caller_an_attested_participant` breaks this invariant by accepting any node that once had a valid attestation, regardless of whether that attestation is still valid today.

Concretely:
- A node running a revoked image (e.g., one with a known signing-path vulnerability) can still call `respond` and have its signature contribution accepted by the contract, bypassing the revocation.
- A node with an expired attestation can still vote on `vote_pk` / `vote_reshared`, influencing which public key is committed on-chain after a key generation or resharing event.
- The `accept_requests` flag does not mitigate this: it is a global gate that only trips when fewer than threshold participants have valid attestations. A single expired/revoked node is invisible to it.

This breaks the production safety invariant that participant authorization requires a currently-valid TEE attestation, without requiring any network-level DoS or operator misconfiguration.

**Impact class: Medium** — Participant-state and contract execution-flow manipulation that breaks production safety/accounting invariants.

---

### Likelihood Explanation

- TEE attestations expire on a fixed schedule (every ~7 days per the documented resubmit cadence). The window between expiry and the next `verify_tee` call is a routine operational gap, not a rare edge case.
- Image hash revocations are a planned governance action (e.g., in response to a discovered vulnerability). The gap between the governance vote removing the hash and the resharing completing is the exploitation window.
- No privileged access is required: the attacker is the node itself, calling public contract methods with its own NEAR account key.
- The node already holds the key material needed to produce valid signatures, so no additional capability is needed.

---

### Recommendation

Add a call to `reverify_participants` (or an equivalent inline `re_verify` call) inside `is_caller_an_attested_participant`, so that the function rejects callers whose stored attestation fails current-time validity checks:

```rust
pub(crate) fn is_caller_an_attested_participant(
    &self,
    participants: &Participants,
    tee_upgrade_deadline_duration: Duration,
) -> Result<(), AttestationCheckError> {
    // ... existing ownership checks ...

    // NEW: re-verify the attestation is still valid right now
    let tee_status = self.reverify_participants(&attestation.node_id, tee_upgrade_deadline_duration);
    if !matches!(tee_status, TeeQuoteStatus::Valid) {
        return Err(AttestationCheckError::AttestationExpiredOrRevoked);
    }

    Ok(())
}
```

All callers of `assert_caller_is_attested_participant_and_protocol_active` would then automatically enforce current-time validity, closing the gap between `verify_tee` invocations.

### Proof of Concept

1. Deploy the contract in `Running` state with 3 participants (threshold = 2). All three submit valid attestations with `expiry_timestamp_seconds = T`.
2. Advance block time past `T` for participant P3 only (P1 and P2 remain valid). Do **not** call `verify_tee`.
3. `accept_requests` is still `true` (P1 + P2 ≥ threshold = 2).
4. As P3, call `respond` with a valid cryptographic signature for a pending request. `is_caller_an_attested_participant` passes (P3's entry is still in `stored_attestations`, ownership fields match). The signature is accepted and the yield is resolved — despite P3's attestation being expired.
5. Confirm: calling `reverify_and_cleanup_participants` at this point would classify P3 as `TeeQuoteStatus::Invalid` and exclude it. The gap between step 2 and a future `verify_tee` call is the exploitation window. [1](#0-0) [11](#0-10) [12](#0-11)

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }

        allowed_mpc_docker_image_hashes: &[NodeImageHash],
        allowed_launcher_docker_compose_hashes: &[LauncherDockerComposeHash],
        allowed_measurements: &[ExpectedMeasurements],
    ) -> Result<(), VerificationError> {
        match self {
            Self::Dstack(ValidatedDstackAttestation {
                mpc_image_hash,
                launcher_compose_hash,
                expiry_timestamp_seconds: expiration_timestamp_seconds,
                measurements,
            }) => {
                let attestation_has_expired = *expiration_timestamp_seconds < timestamp_seconds;

                if attestation_has_expired {
                    return Err(VerificationError::Custom(format!(
                        "The attestation expired at t = {:?}, time_now = {:?}",
                        expiration_timestamp_seconds, timestamp_seconds
                    )));
                }

                let () = verify_mpc_hash(mpc_image_hash, allowed_mpc_docker_image_hashes)?;
                let () = verify_launcher_compose_hash(
                    launcher_compose_hash,
                    allowed_launcher_docker_compose_hashes,
                )?;

                verify_measurements(measurements, allowed_measurements)?;

                Ok(())
            }

                timestamp_seconds,
                allowed_mpc_docker_image_hashes,
                allowed_launcher_docker_compose_hashes,
                allowed_measurements,
            ),
        }
