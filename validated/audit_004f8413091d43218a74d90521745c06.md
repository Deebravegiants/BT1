### Title
VRF Key Reference Count Underflow in `poolReapTransition` Allows Duplicate VRF Key Registration After v11 Hard Fork - (File: `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

The `POOLREAP` rule unconditionally removes "dangling" VRF key hashes from the reference-counted `psVRFKeyHashes` map using `Map.withoutKeys`, without decrementing the reference count. When multiple pools share the same VRF key (permitted before protocol version 11), a single pool re-registration causes the shared VRF key's entry to be deleted entirely from the map, even though other pools still hold that key. This allows a subsequent pool registration to reuse the "freed" VRF key, bypassing the uniqueness invariant introduced in v11 and enabling an attacker to gain disproportionate block-production opportunities.

---

### Finding Description

`PState` tracks registered VRF key hashes in a reference-counted map:

```haskell
psVRFKeyHashes :: !(Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64))
``` [1](#0-0) 

At the v11 hard fork, `populateVRFKeyHashes` correctly initialises this map by counting every VRF key appearing in both `psStakePools` and `psFutureStakePoolParams`, so two pools sharing VRF key `V` produce a count of 2. [2](#0-1) 

In `poolReapTransition`, when a pool re-registers with a new VRF key, the old key is identified as "dangling":

```haskell
danglingVRFKeyHashes =
  Set.fromList $ Map.elems $
    Map.merge Map.dropMissing Map.dropMissing
      ( Map.zipWithMaybeMatched $ \_ sps sppF ->
          if sps ^. spsVrfL /= sppF ^. sppVrfL then Just (sps ^. spsVrfL) else Nothing
      )
      (ps0 ^. psStakePoolsL)
      (ps0 ^. psFutureStakePoolParamsL)
``` [3](#0-2) 

These dangling keys are then **completely deleted** from the map:

```haskell
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )
``` [4](#0-3) 

`Map.withoutKeys` removes the entry entirely, regardless of its reference count. By contrast, the retirement path correctly uses `removeVRFKeyHashOccurrence`, which decrements the count by 1 and only removes the entry when it reaches zero:

```haskell
removeVRFKeyHashOccurrence =
  Map.update (mapNonZero (\n -> n - 1))
``` [5](#0-4) 

The v11 pool-registration check that enforces uniqueness is:

```haskell
Map.notMember sppVrf psVRFKeyHashes
  ?! injectFailure (VRFKeyHashAlreadyRegistered sppId sppVrf)
``` [6](#0-5) 

Once `V` is deleted from `psVRFKeyHashes` by the dangling-key removal, `Map.notMember V psVRFKeyHashes` returns `True`, and a new pool registration with `V` is accepted.

---

### Impact Explanation

An attacker who controls two pools (`kh1`, `kh3`) can:

1. **Before v11**: Register `kh1` with VRF key `V` — the same key already used by an honest pool `kh2` (duplicate VRF keys were permitted before v11).
2. **After v11**: Re-register `kh1` with a fresh VRF key `V2`. `psVRFKeyHashes` becomes `{V: 2, V2: 1}`.
3. **Epoch boundary**: `danglingVRFKeyHashes = {V}`. `Map.withoutKeys {V}` deletes `V` entirely. `psVRFKeyHashes` becomes `{V2: 1}`, even though `kh2` still uses `V`.
4. **New registration**: Register `kh3` with `V`. The uniqueness check passes (`Map.notMember V psVRFKeyHashes = True`). Now `kh2` and `kh3` both hold VRF key `V`.

Both `kh2` and `kh3` will be elected slot leaders for every slot where `VRF(V, nonce, slot) < threshold`, giving the attacker double the block-production rate for those slots and correspondingly inflated rewards — outside the design parameters of the protocol.

**Allowed impact matched**: *Medium — Attacker-controlled transactions modify rewards outside design parameters.*

---

### Likelihood Explanation

- Requires the attacker to have registered a pool with a duplicate VRF key before the v11 hard fork. This was explicitly permitted before v11 and is a realistic historical state.
- The attacker controls only their own pool operator keys; no privileged access, governance majority, or key compromise is required.
- The re-registration and new registration are ordinary, unprivileged transactions.
- The test suite in `HardForkSpec.hs` already exercises the duplicate-VRF retirement path but does not test the re-registration + duplicate-VRF scenario described here. [7](#0-6) 

---

### Recommendation

Replace the unconditional `Map.withoutKeys danglingVRFKeyHashes` with the same reference-count decrement used for retired pools:

```haskell
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
     )
```

This mirrors the correct pattern already used for pool retirement and ensures that a VRF key is only removed from `psVRFKeyHashes` when its reference count reaches zero — i.e., when no remaining pool holds that key.

### Proof of Concept

```
Epoch N (before v11):
  RegPool kh1 {vrf = V}   -- attacker's pool 1
  RegPool kh2 {vrf = V}   -- honest pool (or attacker's pool 2)

Hard fork to v11:
  populateVRFKeyHashes → psVRFKeyHashes = {V: 2}

Epoch N+1 (after v11):
  RegPool kh1 {vrf = V2}  -- re-register with fresh VRF
  → psVRFKeyHashes = {V: 2, V2: 1}

Epoch boundary (POOLREAP):
  danglingVRFKeyHashes = {V}
  Map.withoutKeys {V} → psVRFKeyHashes = {V2: 1}   ← BUG: V deleted despite count=2

Epoch N+2:
  RegPool kh3 {vrf = V}   -- attacker's new pool
  Map.notMember V psVRFKeyHashes = True → check passes!
  → psVRFKeyHashes = {V: 1, V2: 1}
  → kh2 and kh3 both hold VRF key V; attacker earns double slot-leader elections
```

populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
```
