### Title
Missing `payload_hash` Validation in `respond_verify_foreign_tx` Enables Cross-Request Replay of Foreign-Chain Verification Signatures - (File: crates/contract/src/lib.rs)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over `response.payload_hash`, but never validates that `response.payload_hash` was actually derived from the `request` parameter. A single Byzantine attested participant can replay any previously observed `(payload_hash, signature)` pair — produced by the MPC network for a different foreign-tx request — to resolve an unrelated pending request, delivering a forged verification response to the caller.

---

### Finding Description

In `respond_verify_foreign_tx` (lines 691–754 of `crates/contract/src/lib.rs`), the contract performs the following checks:

1. Caller is an attested participant.
2. Protocol state is running or resharing.
3. `accept_requests` is true.
4. The ECDSA signature in `response.signature` is valid over `response.payload_hash` under the domain's root public key.
5. `resolve_yields_for` is called with `request` as the map key to drain the pending yield queue. [1](#0-0) 

What is **never checked**: that `response.payload_hash` is a valid hash of a `ForeignTxSignPayload` whose embedded `request` field matches the `request` parameter. The contract only verifies the signature is valid over *some* hash — it does not verify *which* payload that hash represents.

Compare this to the `respond` function for regular signing, where the payload is taken directly from `request.payload` (part of the pending-request key), making it implicitly validated: [2](#0-1) 

The SDK's `ForeignChainSignatureVerifier::verify_signature` does perform the missing check on the caller side: [3](#0-2) 

But the **contract itself** does not enforce it, leaving the invariant entirely to callers.

The canonical hash construction is: [4](#0-3) 

A correct response for request R must satisfy `payload_hash = SHA256(borsh(ForeignTxSignPayload { request: R.request, values: <observed_values> }))`. The contract never asserts this.

---

### Impact Explanation

**High — Cross-chain replay / forged foreign-chain verification causing invalid bridge execution.**

A Byzantine attested participant executes the following replay:

1. A legitimate `verify_foreign_transaction` for Bitcoin tx A completes. The MPC network produces `(H_A, S_A)` where `H_A = SHA256(borsh(ForeignTxSignPayload { request: Bitcoin_tx_A, values: [block_hash_A] }))`. This response is submitted on-chain via `respond_verify_foreign_tx` and is publicly observable.

2. A bridge user submits `verify_foreign_transaction` for Bitcoin tx B (a different transaction, e.g., a deposit they want to claim). The pending request is queued under key `VerifyForeignTransactionRequest { request: Bitcoin_tx_B, domain_id, payload_version }`.

3. The Byzantine participant calls `respond_verify_foreign_tx` with:
   - `request` = Bitcoin tx B's `VerifyForeignTransactionRequest` (matches the pending key)
   - `response.payload_hash` = `H_A` (from Bitcoin tx A)
   - `response.signature` = `S_A` (from Bitcoin tx A)

4. The contract checks: is `S_A` a valid ECDSA signature over `H_A` under the ForeignTx domain's root key? **Yes** — it was produced by the MPC network for this exact domain.

5. `resolve_yields_for` finds the pending request for Bitcoin tx B and resumes the caller's yield with `(H_A, S_A)`.

6. The bridge contract receives `(H_A, S_A)`. If it does not independently verify that `H_A` encodes the expected values for Bitcoin tx B (e.g., it only checks the signature is valid over the returned hash), it accepts the fabricated response and releases funds based on a different transaction's attestation.

Even if the bridge contract does verify `payload_hash`, the pending request for Bitcoin tx B has been consumed. The caller must re-submit, and the attacker can repeat the replay indefinitely, permanently blocking the request from being fulfilled — a targeted denial of service against specific bridge deposits.

---

### Likelihood Explanation

- **Attacker capability**: Any single attested MPC participant. No threshold collusion required. The attacker only needs their own signing key to call `respond_verify_foreign_tx`.
- **Observation cost**: Zero. All `respond_verify_foreign_tx` calls are on-chain NEAR transactions. The `(payload_hash, signature)` pair from any prior response is permanently public.
- **Targeting**: The attacker can choose which pending request to poison and which prior response to replay. They can wait for a high-value bridge deposit request and replay any prior response against it.
- **Precondition**: At least one prior `verify_foreign_transaction` must have been successfully completed (to obtain a replayable `(H, S)` pair). In a live bridge, this is trivially satisfied.

---

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, recompute the expected payload hash from the `request` and assert it matches `response.payload_hash`:

```rust
// After signature verification, before resolve_yields_for:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    // The contract cannot know the extracted values, but it can verify
    // that the hash is at minimum consistent with the request's chain/tx_id
    // by requiring nodes to include the request in the signed payload.
    // Alternatively, store the expected payload_hash at enqueue time.
    ..
});
```

The cleanest fix is to store the expected `payload_hash` range (or at minimum the `request` field) alongside the pending yield at enqueue time in `verify_foreign_transaction`, then assert `response.payload_hash` encodes a payload whose `request` field matches the stored request. This mirrors how `respond` implicitly validates the payload via the request key.

Alternatively, require nodes to submit the full `ForeignTxSignPayload` (not just the hash) so the contract can recompute and verify the hash itself — though this has on-chain data-size implications.
