### Title
`OmniToken.set_withdraw_relayer_address()` Is Permanently Unreachable from the Bridge Controller, Causing Irreversible Fund Lock for Legacy WITHDRAW_TO Transfers - (File: near/omni-token/src/lib.rs)

---

### Summary

`OmniToken` exposes `set_withdraw_relayer_address()`, which is gated by `assert_controller()` (only the bridge contract may call it). The `omni-bridge` contract — the designated controller of every deployed `OmniToken` — has no function that calls `set_withdraw_relayer_address()` on any token. Because the relayer address can never be configured, any user who invokes the legacy "WITHDRAW_TO:" memo flow has their tokens silently routed to the token contract itself, where they are permanently locked with no recovery path.

---

### Finding Description

`OmniToken.set_withdraw_relayer_address()` stores a relayer `AccountId` that is consumed by the overridden `ft_transfer` implementation:

```rust
// near/omni-token/src/lib.rs  lines 113-117
pub fn set_withdraw_relayer_address(&mut self, relayer: &AccountId) {
    self.assert_controller();
    env::storage_write(WITHDRAW_RELAYER_ADDRESS, &borsh::to_vec(relayer).unwrap());
}
``` [1](#0-0) 

`assert_controller()` enforces that only `self.controller` (set at deployment to `env::predecessor_account_id()` — the `omni-bridge` contract) may call this function:

```rust
// near/omni-token/src/lib.rs  lines 98-104
fn assert_controller(&self) {
    let caller = env::predecessor_account_id();
    require!(
        caller == self.controller,
        TokenError::MissingPermission.as_ref()
    );
}
``` [2](#0-1) 

The `ft_transfer` override uses the stored relayer address for the legacy Near Intents withdrawal path:

```rust
// near/omni-token/src/lib.rs  lines 194-207
fn ft_transfer(&mut self, receiver_id: AccountId, amount: U128, memo: Option<String>) {
    if receiver_id == env::current_account_id()
        && memo.as_ref().is_some_and(|m| m.starts_with(WITHDRAW_MEMO_PREFIX))
    {
        if let Some(withdraw_relayer) = Self::read_withdraw_relayer_address() {
            return self.token.ft_transfer(withdraw_relayer, amount, memo);
        }
    }
    self.token.ft_transfer(receiver_id, amount, memo);
}
``` [3](#0-2) 

When no relayer is stored (the `None` branch), execution falls through to `self.token.ft_transfer(receiver_id, amount, memo)` where `receiver_id` is the token contract itself. The tokens are deposited into the token contract's own balance.

The `omni-bridge` contract — the sole entity permitted to call `set_withdraw_relayer_address()` — contains **no function that invokes it**. A search for `set_withdraw_relayer` across all of `near/omni-bridge/src/` returns zero matches. The bridge does call `set_metadata` and `set_controller_for_tokens` on tokens, but there is no analogous wrapper for `set_withdraw_relayer_address`. [4](#0-3) 

The `TokenDeployer.deploy_token()` sets the controller to `env::predecessor_account_id()` (the bridge), confirming the bridge is the only authorized caller: [5](#0-4) 

---

### Impact Explanation

Tokens transferred to the token contract via the legacy WITHDRAW_TO memo path are permanently locked. The bridge's `burn()` function calls `internal_withdraw(&env::predecessor_account_id(), ...)`, which withdraws from the bridge's own balance — not from the token contract's balance — so there is no recovery path for tokens stranded in the token contract.

This is an **irreversible fund lock** matching the Critical impact criterion: permanently unclaimable user value in a bridge token flow.

---

### Likelihood Explanation

The WITHDRAW_TO memo path is explicitly described in the code as a "Legacy bridging flow used by Near Intents." Users migrating from Near Intents or following legacy documentation may invoke `ft_transfer(token_contract_id, amount, "WITHDRAW_TO:...")`. Because the relayer can never be configured (the bridge has no entrypoint), every such call silently locks funds. The user receives no error — the transfer succeeds from the NEP-141 perspective, but the tokens are irrecoverably deposited into the token contract.

---

### Recommendation

Add a privileged function in `near/omni-bridge/src/lib.rs` that calls `set_withdraw_relayer_address` on a given token, restricted to `Role::DAO` or an equivalent governance role, mirroring the existing `set_token_metadata` pattern:

```rust
#[access_control_any(roles(Role::DAO))]
pub fn set_token_withdraw_relayer(
    &self,
    token_account_id: AccountId,
    relayer: AccountId,
) -> Promise {
    ext_token::ext(token_account_id)
        .with_static_gas(/* appropriate gas */)
        .set_withdraw_relayer_address(&relayer)
}
```

This mirrors the existing `set_token_metadata` wrapper pattern already present in the bridge. [4](#0-3) 

---

### Proof of Concept

1. Bridge deploys an `OmniToken` via `TokenDeployer.deploy_token()`; the token's `controller` is set to the bridge contract address. [5](#0-4) 
2. No call to `set_withdraw_relayer_address` is ever possible because the bridge exposes no such entrypoint. `WITHDRAW_RELAYER_ADDRESS` storage slot remains empty. [6](#0-5) 
3. A user calls `ft_transfer(token_contract_id, 1000, "WITHDRAW_TO:some_address")` on the token.
4. `ft_transfer` enters the legacy branch (`receiver_id == env::current_account_id()` and memo matches prefix), reads `None` from storage, and falls through to `self.token.ft_transfer(token_contract_id, 1000, memo)`. [3](#0-2) 
5. 1000 tokens are credited to the token contract's own account. The bridge's `burn()` cannot withdraw them (it only withdraws from `env::predecessor_account_id()`). Funds are permanently locked. [7](#0-6)
