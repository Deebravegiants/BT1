### Title
Liquidity Deposited for an Uncontrollable `owner` Is Permanently Locked — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no validation that the address is capable of calling `removeLiquidity`. Because `removeLiquidity` strictly enforces `msg.sender == owner`, any position minted for a contract that cannot invoke the pool (or for `address(0)` when called directly) results in permanently irrecoverable LP tokens and the underlying deposited principal.

### Finding Description

`addLiquidity` implements an explicit operator pattern: `msg.sender` pays tokens via the `IMetricOmmModifyLiquidityCallback` callback, while the position is recorded under the caller-supplied `owner` address. [1](#0-0) 

The pool performs no validation on `owner` — no zero-address guard, no contract-capability check. Shares are written to `_positionBinShares[keccak256(owner, salt, binIdx)]` and tokens are pulled from `msg.sender`.

`removeLiquidity` then enforces a hard ownership gate: [2](#0-1) 

Only the `owner` address itself can ever burn those shares. If `owner` is:
- `address(0)` (reachable by calling the pool directly, bypassing the periphery),
- a contract without a `removeLiquidity` call path (e.g., a multisig, a vault, or any contract that does not implement pool interaction),

then `msg.sender == owner` can never be satisfied and the position is permanently locked.

The periphery `MetricOmmPoolLiquidityAdder` adds only a zero-address guard: [3](#0-2) 

It does not verify that `owner` is capable of calling `removeLiquidity`. Any non-zero contract address that cannot interact with the pool passes this check.

The interface documentation acknowledges the operator pattern but provides no safety net: [4](#0-3) 

### Impact Explanation

Tokens transferred from `msg.sender` during the `metricOmmModifyLiquidityCallback` settlement are permanently locked in the pool under an inaccessible position. The pool's `binTotals` and `_binStates` correctly account for the tokens, but no address can ever call `removeLiquidity` to recover them. This is a direct, irreversible loss of user principal with no recovery path. [5](#0-4) 

### Likelihood Explanation

The trigger is unprivileged — any caller can invoke `addLiquidity` directly on the pool with `owner = address(0)`, or invoke the periphery `addLiquidityExactShares(pool, contractOwner, ...)` with a non-zero contract that cannot call `removeLiquidity`. Protocol integrators using the operator pattern (e.g., yield optimizers, vaults, or airdrop distributors that add liquidity on behalf of users) are the most realistic trigger: they may supply a contract address as `owner` without verifying that the contract can call `removeLiquidity`.

### Recommendation

1. Add a zero-address guard on `owner` inside `MetricOmmPool.addLiquidity` itself (not only in the periphery):
   ```solidity
   require(owner != address(0), InvalidOwner());
   ```
2. Consider splitting `addLiquidity` into `addLiquidity(...)` (self-owned, `owner = msg.sender`) and `addLiquidityFor(address owner, ...)` (operator path), mirroring the fix applied to the OGV staking contract.
3. Document clearly that callers using the operator path are responsible for ensuring `owner` can call `removeLiquidity`, and consider an off-chain or on-chain ERC-165 check for contract receivers.

### Proof of Concept

```solidity
// Attacker (or mistaken integrator) calls pool directly — bypasses LiquidityAdder's zero-address check
pool.addLiquidity(
    address(0),   // owner: address(0) — no one can ever be msg.sender == address(0)
    salt,
    deltas,
    callbackData, // msg.sender must implement IMetricOmmModifyLiquidityCallback
    ""
);
// Tokens are pulled from msg.sender via callback and locked forever.
// removeLiquidity(address(0), ...) always reverts with NotPositionOwner.
```

For the non-zero contract case via the periphery:

```solidity
// owner = address of a contract that has no removeLiquidity call path
liquidityAdder.addLiquidityExactShares(
    pool,
    address(vaultWithNoPoolInteraction), // passes _validateOwner (non-zero)
    salt,
    deltas,
    maxAmount0,
    maxAmount1,
    ""
);
// Tokens pulled from msg.sender. Position locked under vaultWithNoPoolInteraction.
// vaultWithNoPoolInteraction cannot call pool.removeLiquidity — funds are irrecoverable.
``` [6](#0-5) [7](#0-6)

  function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
    external
    nonReentrant(PoolActions.REMOVE_LIQUIDITY)
    returns (uint256 amount0Removed, uint256 amount1Removed)
  {
    if (deltas.binIdxs.length == 0) return (0, 0);
    if (deltas.binIdxs.length != deltas.shares.length) revert LiquidityDeltaLengthMismatch();

    );
    _afterRemoveLiquidity(msg.sender, owner, salt, deltas, amount0Removed, amount1Removed, extensionData);
