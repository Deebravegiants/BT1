### Title
`valid_proposals` cache poisoned with wrong commitment before `ProposalFinMismatch` check causes panic on `decision_reached` - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

In `validate_proposal`, the proposal is unconditionally inserted into the `valid_proposals` cache with the **batcher-computed commitment** before the `ProposalFinMismatch` guard is evaluated. When the guard fires and the function returns an error, the poisoned entry is never removed. If consensus subsequently decides on the same `(height, round)` with the **network commitment** (the value from `ProposalFin`), `decision_reached` calls `get_proposal`, which `assert_eq!`-panics on the stored-vs-requested commitment mismatch, crashing the consensus task.

---

### Finding Description

`validate_proposal` in `validate_proposal.rs` follows this sequence:

```
// 1. Batcher finishes; batcher_block_commitment is computed from finished_info
HandledProposalPart::Finished(batcher_block_commitment, received_fin, finished_info)

// 2. Proposal inserted into valid_proposals with batcher_block_commitment as key
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);
//   â†‘ stored commitment = batcher_block_commitment

// 3. Commitment comparison
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
    // â†‘ returns error, but the entry in valid_proposals is NOT removed
}
``` [1](#0-0) 

The comment at line 238 acknowledges the ordering is intentional ("to avoid a race condition with `repropose`"), but there is no compensating removal of the entry on the error path.

`valid_proposals` is a `BTreeMap<BlockNumber, BTreeMap<Round, (ProposalCommitment, ProposalContent)>>` â€” one entry per `(height, round)`. [2](#0-1) 

`insert_proposal` computes the stored key as `proposal_commitment_from(finished_info.partial_block_hash, init.fee_proposal_fri)` â€” i.e., the **batcher's** commitment. [3](#0-2) 

When `decision_reached` is later called with the **network's** commitment (the value that 2/3+ validators agreed on), it calls `get_proposal(&height, &round, &network_commitment)`: [4](#0-3) 

`get_proposal` performs a hard `assert_eq!` between the stored commitment and the requested commitment: [5](#0-4) 

Because `stored_commitment == batcher_commitment â‰  network_commitment`, the assertion panics, crashing the consensus task.

---

### Impact Explanation

A validator that experiences `ProposalFinMismatch` â€” batcher computes commitment **X**, proposer's `ProposalFin` carries commitment **Y** â€” silently leaves commitment **X** in `valid_proposals`. The validator does not vote for the proposal. If 2/3+ of the remaining validators accept the proposal with commitment **Y** and consensus decides on **Y**, `decision_reached` is invoked on this node with **Y**. The `assert_eq!(X, Y)` inside `get_proposal` panics, terminating the consensus task and making the node unable to commit the decided block or participate in subsequent heights.

This maps directly to the M-9 pattern: a terminal event (`ProposalFinMismatch`) does not clean up the shared state (`valid_proposals`), leaving it in a stale/wrong condition that causes a hard failure when the downstream consumer (`decision_reached`) relies on that state being consistent.

---

### Likelihood Explanation

The `CONSENSUS_PROPOSAL_FIN_MISMATCH` metric counter is explicitly defined and initialized, indicating the developers expect this path to fire in production. [6](#0-5) 

A `ProposalFinMismatch` can arise from:
- Execution non-determinism between the proposer's batcher and this validator's batcher (e.g., different L1 gas price snapshots, different `fee_proposal_fri` folding, or any divergence in `PartialBlockHashComponents`)
- A bug in `proposal_commitment_from` or `PartialBlockHash::from_partial_block_hash_components` that affects only some nodes

Neither trigger requires a privileged or malicious actor. The `TODO(matan): Switch to signature validation` comment confirms this check is a temporary measure expected to be exercised. [7](#0-6) 

---

### Recommendation

On the `ProposalFinMismatch` error path, remove the entry that was just inserted before returning the error:

```rust
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init.clone(), content, &args.proposal_id, finished_info);

if built_block != received_fin.proposal_commitment {
    // Remove the poisoned entry so decision_reached cannot panic on it.
    valid_proposals.remove_proposal(args.init.height, args.init.round);
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}
```

Alternatively, defer `insert_proposal` until after the commitment check succeeds, and handle the `repropose` race condition by a separate mechanism (e.g., a pending-proposal flag or by checking whether the entry already exists before inserting in `repropose`).

### Proof of Concept

1. Validator V receives a proposal for `(height=H, round=R)`.
2. Batcher executes the transactions and returns `partial_block_hash = X`; `proposal_commitment_from(X, fee_proposal)` = commitment **A**.
3. The proposer's `ProposalFin` carries `proposal_commitment` = **B** (â‰  **A**), e.g., due to a divergent gas-price snapshot.
4. `validate_proposal` calls `valid_proposals.insert_proposal(...)` â†’ entry `(H, R) â†’ (A, content)` is written.
5. `built_block (A) != received_fin.proposal_commitment (B)` â†’ `Err(ProposalFinMismatch)` is returned; entry is **not** removed.
6. V does not send a precommit for this proposal.
7. 2/3+ other validators accept the proposal with commitment **B**; consensus decides on **B**.
8. `decision_reached(H, R, B)` is called on V.
9. `get_proposal(&H, &R, &B)` retrieves the entry `(A, content)` and executes `assert_eq!(A, B)` â†’ **panic**.
10. The consensus task on V crashes; V cannot commit block H or advance to height H+1. [8](#0-7) [9](#0-8)
