Audit Report

## Title
Uninitialized `tokenFeeBps` Mapping Allows Complete Fee Bypass via Token Deposit Path - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains two separate fee-rate variables: `feeBps` for ETH deposits and `tokenFeeBps[token]` for LST/ERC-20 deposits. The `tokenFeeBps` mapping is never initialized in `addSupportedToken`, silently defaulting to `0` for every added token. Because depositing an LST worth `X ETH` yields the same rsETH output as depositing `X ETH` directly, any depositor can route through the token path to pay zero protocol fee, depriving the protocol of all fee revenue on token deposits until an admin explicitly calls `setTokenFeeBps`.

## Finding Description
`RSETHPool.sol` declares two distinct fee-rate variables:
- `feeBps` (L43) — applied to ETH deposits in `viewSwapRsETHAmountAndFee(uint256)` at L312: `fee = amount * feeBps / 10_000`
- `tokenFeeBps[token]` (L88) — applied to token deposits in `viewSwapRsETHAmountAndFee(uint256, address)` at L335–336: `uint256 feeBpsForToken = tokenFeeBps[token]; fee = amount * feeBpsForToken / 10_000`

`addSupportedToken` (L637–656) registers a token's oracle and bridge but never writes to `tokenFeeBps[token]`, leaving it at the Solidity default of `0`. The only write path is `setTokenFeeBps` (L583–594), which requires `DEFAULT_ADMIN_ROLE` and must be called as a separate transaction after token addition.

The public `deposit(address token, uint256 amount, string referralId)` function (L284–305) is callable by any address with no role restriction. It invokes `viewSwapRsETHAmountAndFee(amount, token)`, which uses `tokenFeeBps[token] = 0`, resulting in `fee = 0` and `feeEarnedInToken[token] += 0`. The depositor receives the full rsETH equivalent of their token's ETH value with no fee deducted, while the ETH deposit path would have charged `feeBps` basis points on the same economic value.

No existing check in `deposit(address, uint256, string)` or `viewSwapRsETHAmountAndFee(uint256, address)` validates that `tokenFeeBps[token]` has been explicitly set. The `onlySupportedToken` modifier only checks that `supportedTokenOracle[token] != address(0)`, which is orthogonal to fee initialization.

## Impact Explanation
**High. Theft of unclaimed yield.** The protocol's fee revenue is the yield it is entitled to collect on every deposit. With `tokenFeeBps[token] = 0`, the protocol collects zero fee on all token deposits. Every unit of rsETH minted through the token path represents fee revenue the protocol should have earned but does not. This is a direct, ongoing, unbounded loss of protocol fee revenue with no cap on the amount that can be extracted, repeatable by any depositor on every transaction.

## Likelihood Explanation
The condition is trivially met from the moment any token is added via `addSupportedToken`. No special role, flash loan, oracle manipulation, or governance action is required. The attacker only needs to hold or acquire any supported LST (wstETH, ETHx, stETH, etc.), all of which are freely tradeable on Arbitrum. The entry point `deposit(address, uint256, string)` is fully public. The asymmetry is live immediately after token addition and persists until the admin separately calls `setTokenFeeBps`, a step that is not enforced or atomically coupled to `addSupportedToken`.

## Recommendation
Require an explicit `_feeBps` parameter in `addSupportedToken` and set `tokenFeeBps[token] = _feeBps` atomically during token registration. Alternatively, default `tokenFeeBps[token]` to the global `feeBps` value inside `addSupportedToken`. At minimum, add a guard in `deposit(address, uint256, string)` or `viewSwapRsETHAmountAndFee(uint256, address)` that reverts if `tokenFeeBps[token]` has not been explicitly initialized (e.g., via a separate `tokenFeeBpsSet[token]` boolean flag).

## Proof of Concept
Assume `feeBps = 100` (1%) and wstETH/ETH oracle rate = 1.2e18. wstETH is added via `addSupportedToken` with no subsequent `setTokenFeeBps` call.

**ETH path** — `deposit{value: 1.2e18}(referralId)`:
- `fee = 1.2e18 * 100 / 10_000 = 0.012e18`
- `feeEarnedInETH += 0.012e18`
- `rsETHAmount = (1.2e18 - 0.012e18) * 1e18 / rsETHToETHrate`

**wstETH path** — `deposit(wstETH, 1e18, referralId)`:
- `feeBpsForToken = tokenFeeBps[wstETH] = 0`
- `fee = 0`, `feeEarnedInToken[wstETH] += 0`
- `rsETHAmount = 1e18 * 1.2e18 / rsETHToETHrate` (identical output)

The depositor receives the same rsETH in both cases but pays 0.012 ETH less via the wstETH path. A Foundry fork test can confirm this by: (1) deploying/forking the contract with `feeBps = 100`, (2) adding wstETH as a supported token without calling `setTokenFeeBps`, (3) calling both deposit variants with economically equivalent inputs, and (4) asserting `feeEarnedInToken[wstETH] == 0` while `feeEarnedInETH > 0`.
