Audit Report

## Title
Block Stuffing Exploits 24-Hour Hard Expiry Gap to Allow Blocked Address to Transfer rsETH — (`contracts/RSETH.sol`)

## Summary
`blockUserTransfers` sets a hard 24-hour expiry with no on-chain renewal mechanism. If an attacker sustains block stuffing long enough to delay the manager's refresh transaction past `blockedUntil`, `_enforceNotBlocked` silently clears the mapping and allows the blocked address to freely transfer rsETH. `recoverFrozenFunds` also becomes unavailable during this gap because it requires an active block.

## Finding Description
`blockUserTransfers` writes `transfersBlockedUntil[account] = block.timestamp + 1 days` ( [1](#0-0) ) with no on-chain auto-renewal. `_enforceNotBlocked` only reverts while `block.timestamp < blockedUntil`; once the timestamp reaches or exceeds `blockedUntil`, it deletes the mapping entry and returns without reverting ( [2](#0-1) ). `_transfer` calls `_enforceNotBlocked` for both `from` and `to` ( [3](#0-2) ), so once the block expires the blocked address can execute transfers freely. Simultaneously, `recoverFrozenFunds` checks `blockedUntil == 0 || block.timestamp >= blockedUntil` and reverts with `NoActiveTransferBlock` ( [4](#0-3) ), removing the admin's recovery path during the same gap. The exploit path is: (1) manager blocks target; (2) attacker fills blocks with max-gas transactions to exclude the manager's refresh call; (3) once `block.timestamp >= blockedUntil`, `_enforceNotBlocked` no longer reverts; (4) blocked address transfers its entire balance; (5) manager's refresh lands too late.

## Impact Explanation
**Low — Block stuffing.** The impact is concrete: a blocked address can move its entire rsETH balance during the gap window, and `recoverFrozenFunds` is simultaneously disabled. Even a single-block gap (~12 s on mainnet) is sufficient for the blocked address to drain its balance. This matches the allowed impact class "Low. Block stuffing."

## Likelihood Explanation
The attack is economically rational only when the rsETH balance of the blocked address exceeds the cost of sustained block stuffing (tens of ETH per minute on mainnet). This limits the realistic attacker population to large holders, but the design provides no on-chain fallback: any delay of the manager's refresh — whether from block stuffing, network congestion, or operational error — opens the gap. The manager has no keeper or auto-renewal mechanism in the contract.

## Recommendation
1. **Auto-renew on attempted transfer**: in `_enforceNotBlocked`, when `block.timestamp < blockedUntil`, reset `transfersBlockedUntil[account] = block.timestamp + 1 days` so every attempted transfer by the blocked address extends the window, making expiry impossible to weaponise.
2. **Grace period for `recoverFrozenFunds`**: allow the admin to call `recoverFrozenFunds` for a short window (e.g., 1 hour) after `blockedUntil` has passed, providing a fallback even if the refresh is delayed.
3. **Off-chain monitoring with lead time**: emit a warning event well before expiry (e.g., at `blockedUntil - 1 hour`) so monitoring infrastructure can submit the refresh with sufficient margin to survive normal congestion.

## Proof of Concept
The following Foundry test reproduces the gap without any block stuffing infrastructure; `vm.warp` simulates the effect of the manager's refresh being excluded from blocks until expiry:

```solidity
function testGapWindowExploitable() public {
    // 1. Manager blocks target
    vm.prank(manager);
    address[] memory accounts = new address[](1);
    accounts[0] = target;
    rseth.blockUserTransfers(accounts);

    uint256 blockedUntil = rseth.transfersBlockedUntil(target);

    // 2. Warp to exactly the expiry timestamp (simulates block stuffing delay)
    vm.warp(blockedUntil);

    // 3. Transfer succeeds — _enforceNotBlocked clears the mapping and returns
    vm.prank(target);
    rseth.transfer(other, rseth.balanceOf(target));

    // 4. Manager refresh lands one second later — funds already moved
    vm.warp(blockedUntil + 1);
    vm.prank(manager);
    rseth.blockUserTransfers(accounts);

    assertEq(rseth.balanceOf(target), 0);
    assertGt(rseth.balanceOf(other), 0);
}
```

At `vm.warp(blockedUntil)`, `_enforceNotBlocked` hits the `block.timestamp < blockedUntil` branch as false, deletes the mapping, and returns — allowing the transfer to proceed. [5](#0-4)
