## Analysis

The core claim is an off-by-one in `valid_entries` that causes the last expired entry to be included in the returned slice. Let me trace the exact code.

**The function** (`crates/contract/src/tee/proposal.rs`, lines 170–193):

```rust
fn valid_entries(&self, tee_upgrade_deadline_duration: Duration) -> Vec<AllowedMpcDockerImage> {
    let current_time = Timestamp::now();
    let cutoff_index = self
        .allowed_tee_proposals
        .iter()
        .rposition(|allowed_docker_image| {
            let Some(grace_period_deadline) = allowed_docker_image
                .added
                .checked_add(tee_upgrade_deadline_duration)
            else {
                return true;
            };
            grace_period_deadline < current_time   // predicate: "is this entry expired?"
        })
        .unwrap_or(0);

    self.allowed_tee_proposals
        .get(cutoff_index..)   // ← inclusive of cutoff_index
        .unwrap_or(&[])
        .to_vec()
}
```

`rposition` returns the index of the **last** element satisfying the predicate (i.e., the last expired entry). `get(cutoff_index..)` is **inclusive**, so the expired entry at `cutoff_index` is always returned.

**Concrete trace** — entries `[hash_v1(expired), hash_v2(valid), hash_v3(valid)]`:
- `rposition` scans right-to-left: hash_v3 → false, hash_v2 → false, hash_v1 → true → returns `Some(0)`
- `get(0..)` = `[hash_v1, hash_v2, hash_v3]` — hash_v1 is incorrectly included

**The existing test** (`test_clean_expired`) does NOT catch this because it sets `current_time` to `second_entry_time + 10_days + 1ns`, which expires **both** entries simultaneously. When all entries are expired, `rposition` returns the last index and `get(last..)` returns just that one entry — the intentional "always keep at least one" behavior. The mixed case (some expired, some valid) is never tested.

**Timestamp manipulation is not required.** The bug manifests in normal operation: once hash_v1's grace period expires while hash_v2 is still valid, `valid_entries` incorrectly includes hash_v1. No block-time manipulation is needed.

**Impact path:**
- `valid_entries` feeds `get_image_hashes` [1](#0-0) 
- `get_image_hashes` is used in `TeeState` to build the allowed list passed to attestation verification [2](#0-1) 
- `submit_participant_info` and `verify_tee` both check submitted image hashes against this list [3](#0-2) 
- `verify_mpc_hash` does a simple `allowed_hashes.contains(image_hash)` check [4](#0-3) 

A node running an expired image hash (hash_v1) will pass `verify_mpc_hash` because hash_v1 is still in the list returned by `valid_entries`. The `verify_tee` kickout mechanism will also fail to evict it.

**However**, the question's framing about an "unprivileged attacker advancing block time" is incorrect. NEAR block timestamps are set by validators, not by user transactions. The bug does not require any timestamp manipulation — it is a pure logic error triggered by normal time passage.

**Scope assessment:**

The impact is attestation authorization bypass: nodes running rotated-out image hashes remain attested. The allowed High impact category includes "participant/attestation authorization bypass." However, the full High impact statement requires this to "cause invalid bridge execution or double-spend conditions." The bypass alone does not directly cause double-spends — it only allows nodes running old images to remain in the participant set. Whether this enables further exploitation depends on whether the rotated image had a security vulnerability, which is speculative.

The bug is real, local-testable, and in production code. The off-by-one is at: [5](#0-4)

### Title
Off-by-one in `AllowedDockerImageHashes::valid_entries` causes expired image hashes to remain in the allowed list — (`crates/contract/src/tee/proposal.rs`)

### Summary
`valid_entries` uses `rposition` to find the last expired entry, then returns `get(cutoff_index..)` which is **inclusive** of that expired entry. In any scenario where at least one entry has expired and at least one newer entry is still valid, the last expired entry is incorrectly included in the returned allowed list.

### Finding Description
`rposition` returns the index of the rightmost element satisfying `grace_period_deadline < current_time`. The subsequent slice `get(cutoff_index..)` includes the element at `cutoff_index`, meaning the expired entry is always returned. The correct slice would be `get(cutoff_index + 1..)` (with special handling for the all-expired case to preserve the "always keep at least one" invariant).

The existing unit test `test_clean_expired` does not expose this because it sets the clock so that all entries expire simultaneously — the all-expired path is exercised, but the mixed expired/valid path is not. [6](#0-5) 

### Impact Explanation
Nodes running an image hash whose grace period has expired but which is still the last expired entry in the list will pass `verify_mpc_hash` and will not be evicted by `verify_tee`. This is an attestation authorization bypass: the contract's image-hash rotation enforcement is silently defeated for the most recently expired hash whenever a newer valid hash exists. [4](#0-3) 

### Likelihood Explanation
This triggers in normal operation — no attacker action required. Any time governance votes in a new image hash and the old one's grace period expires, the old hash remains accepted. The timestamp manipulation framing in the question is a red herring; the bug is structural.

### Recommendation
Change `get(cutoff_index..)` to `get(cutoff_index + 1..)`, with a fallback to `get(cutoff_index..)` when `cutoff_index` is the last element (to preserve the "always keep at least one" invariant). Alternatively, restructure the logic to find the first valid entry rather than the last expired one.

fn test_expired_entry_not_included_when_newer_valid_entry_exists() {

    let deadline = Duration::from_secs(10 * 24 * 60 * 60); // 10 days

    // t=0: add hash_v1
    testing_env!(VMContextBuilder::new().block_timestamp(0).build());
    allowed.insert(dummy_code_hash(1), deadline);

    // t=1s: add hash_v2
    testing_env!(VMContextBuilder::new().block_timestamp(1_000_000_000).build());
    allowed.insert(dummy_code_hash(2), deadline);

    // t = 10 days + 2ns: hash_v1 expired, hash_v2 still valid
    let t = deadline.as_nanos() as u64 + 2;
    testing_env!(VMContextBuilder::new().block_timestamp(t).build());

    let hashes = allowed.get_image_hashes(deadline);
    // BUG: hash_v1 is still in hashes due to get(cutoff_index..) being inclusive
    assert!(!hashes.contains(&dummy_code_hash(1)), "expired hash_v1 must not be returned");
    assert!(hashes.contains(&dummy_code_hash(2)), "valid hash_v2 must be returned");
}

    pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact — the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
                }

                // here, we set it to true, because at this point, we have at least `threshold`
                // number of participants with an accepted Tee status.
                self.accept_requests = true;

                // do we want to adjust the threshold?
                //let n_participants_new = new_participants.len();
                //let new_threshold = (3 * n_participants_new + 4) / 5; // minimum 60%
                //let new_threshold = new_threshold.max(2); // but also minimum 2
                let new_threshold = usize::try_from(current_params.threshold().value())
                    .expect("threshold value fits in usize");

                let threshold_parameters = ThresholdParameters::new(
                    participants_with_valid_attestation,
                    Threshold::new(new_threshold as u64),
                )
                .expect("Require valid threshold parameters"); // this should never happen.
                current_params.validate_incoming_proposal(&threshold_parameters)?;
                // This resharing only changes the participant set, so the
                // per-domain reconstruction-threshold updates map is empty.
                let proposed_parameters =
                    ProposedThresholdParameters::new(threshold_parameters, BTreeMap::new());
                let res = running_state.transition_to_resharing_no_checks(&proposed_parameters);
                if let Some(resharing) = res {
                    self.protocol_state = ProtocolContractState::Resharing(resharing);
                }

                Ok(true)
            }
        }
    }

fn verify_mpc_hash(
    image_hash: &NodeImageHash,
    allowed_hashes: &[NodeImageHash],
) -> Result<(), VerificationError> {
    if allowed_hashes.is_empty() {
        return Err(VerificationError::Custom(
            "the allowed mpc image hashes list is empty".to_string(),
        ));
    }

    let image_hash_is_allowed = allowed_hashes.contains(image_hash);
    if !image_hash_is_allowed {
        return Err(VerificationError::Custom(format!(
            "MPC image hash {:?} is not in the allowed hashes list",
            image_hash
        )));
    }

    Ok(())
}
