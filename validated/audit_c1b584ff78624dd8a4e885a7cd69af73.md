### Title
Dijkstra Sub-Transaction Deposit Refund Double-Claiming Enables ADA Drain from Deposit Pot - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs)

---

### Summary

In the Dijkstra era, `dijkstraTotalRefundsTxCerts` uses user-supplied deposit values directly from deregistration certificates (`UnRegDepositTxCert`, `UnRegDRepTxCert`) for the preservation-of-value accounting, discarding the ledger-state lookup functions entirely. The comment in the code explicitly states that "ledger rules will validate that they are accurate," but the sub-transaction path lacks the predicate check that prevents multiple sub-transactions within a single top-level transaction from each claiming the same deposit refund. This allows an attacker to drain ADA from the deposit pot by submitting a top-level transaction whose sub-transactions each independently claim the same credential's deposit.

---

### Finding Description

**Root cause — `dijkstraTotalRefundsTxCerts` ignores stored-deposit lookups:**

In Conway, `conwayDRepRefundsTxCerts` and `shelleyTotalRefundsTxCerts` compute refunds by calling `lookupDRepDeposit` / `lookupDeposit` — functions that read the **stored** deposit from ledger state. The user-supplied value in the certificate is validated separately in the DELEG/GovCert rule and must equal the stored value.

In Dijkstra, the refund calculation is redesigned:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs, lines 228-238
-- | Unlike previous eras, we no longer need to lookup refunds from the ledger state,
-- since all of the certificates specify the actual refund and ledger rules will
-- validate that they are accurate.
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit   -- user-supplied value used directly
  UnRegDRepTxCert _ deposit -> deposit      -- user-supplied value used directly
  _ -> zero
```

The `getTotalRefundsTxCerts` instance for Dijkstra discards the lookup parameters entirely:

```haskell
-- line 285
getTotalRefundsTxCerts _ _ _ = dijkstraTotalRefundsTxCerts
```

The design contract is: "ledger rules will validate that they are accurate." For top-level transactions this is satisfied by the Conway DELEG/GovCert rules (which check the supplied refund against the stored deposit). However, for **sub-transactions** the analogous `SubDeleg` rule does not yet implement this predicate check.

**Missing sub-transaction validation — confirmed by skipped tests with explicit TODO:**

The Dijkstra test library contains two tests that are explicitly skipped (`xit`) with `TODO: predicate failure not yet implemented`:

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs, lines 53-75
xit "Multiple subtransactions cannot get the same refund" $ do
  stakingCred <- KeyHashObj <$> freshKeyHash
  _ <- registerStakeCredential stakingCred
  keyDeposit <- getsPParams ppKeyDepositL
  ...
  let
    subTx1 = mkBasicTx mkBasicTxBody
      & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = mkBasicTx mkBasicTxBody
      & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = mkBasicTx mkBasicTxBody
      & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

The `error "TODO: predicate failure not yet implemented"` is not a test assertion — it is a placeholder acknowledging that the ledger rule that should reject this transaction **does not exist yet**. The test is skipped precisely because the production code does not yet enforce the invariant.

**How the accounting discrepancy arises:**

Each sub-transaction is evaluated with its own preservation-of-value check using `dijkstraTotalRefundsTxCerts`. Because this function reads the user-supplied deposit value from the certificate rather than the stored deposit, and because the `SubDeleg` rule does not yet validate that the credential is still registered and that the refund matches the stored deposit, both `subTx1` and `subTx2` can independently pass their individual balance checks while each counting `keyDeposit` as consumed. The deposit pot (`utxosDeposited`) is debited twice for a single registered credential.

---

### Impact Explanation

**Critical — Direct loss of ADA through an invalid ledger state transition.**

An attacker who has registered a stake credential (paying deposit `D`) can submit a single top-level Dijkstra transaction containing `N` sub-transactions each bearing `UnRegDepositTxCert stakingCred D`. If the `SubDeleg` rule does not enforce that the credential is still registered at the time each sub-transaction is processed, each sub-transaction independently satisfies its preservation-of-value check by counting `D` as a refund. The deposit pot is debited `N × D` while only `D` was ever deposited, creating ADA out of thin air at the expense of the deposit pot (which is backed by honest users' deposits and ultimately the reserves).

---

### Likelihood Explanation

**Medium.** The Dijkstra era introduces nested/sub-transactions as a new primitive. The vulnerability is reachable by any unprivileged transaction author who can submit a valid Dijkstra top-level transaction. No privileged role, key compromise, or consensus majority is required. The attacker only needs to know the credential they registered and the deposit amount (both public). The barrier is that the Dijkstra era must be active on the network; it is currently marked experimental but the production code path is present and the missing predicate is explicitly acknowledged in the codebase.

---

### Recommendation

1. **Implement the missing `SubDeleg` predicate**: The sub-transaction DELEG rule must validate that (a) the credential is still registered at the time the sub-transaction is processed, and (b) the user-supplied refund in `UnRegDepositTxCert` / `UnRegDRepTxCert` exactly matches the stored deposit. This mirrors the `checkInvalidRefund` logic already present in `conwayDelegTransition`.

2. **Enforce sequential state sharing across sub-transactions**: Sub-transactions within a single top-level transaction must observe each other's state mutations. A credential deregistered by `subTx1` must appear as unregistered when `subTx2` is evaluated, so that the second attempt fails with `StakeKeyNotRegisteredDELEG`.

3. **Remove the `xit` placeholder and implement the predicate failure**: The skipped test `"Multiple subtransactions cannot get the same refund"` should be activated once the rule is implemented, replacing `error "TODO: predicate failure not yet implemented"` with the correct `ConwayDelegPredFailure` or equivalent Dijkstra failure constructor.

### Proof of Concept

```
1. Attacker registers stake credential `cred` in a prior transaction,
   paying deposit D = ppKeyDeposit.

2. Attacker constructs a top-level Dijkstra transaction T containing
   two sub-transactions:

     subTx1 = { certs: [UnRegDepositTxCert cred D],
                outputs: [attacker_addr ← D] }

     subTx2 = { certs: [UnRegDepositTxCert cred D],
                outputs: [attacker_addr ← D] }

     T = { subTransactions: [subTx1, subTx2] }

3. dijkstraTotalRefundsTxCerts counts D as consumed for each sub-tx.
   Because SubDeleg does not yet check that `cred` is still registered
   when subTx2 is processed, both sub-transactions pass their individual
   preservation-of-value checks.

4. Result: utxosDeposited is decremented by 2D; attacker receives 2D
   in outputs; only D was ever deposited. Net gain for attacker: D ADA.
   The deposit pot (and ultimately reserves) absorbs the loss.
```
