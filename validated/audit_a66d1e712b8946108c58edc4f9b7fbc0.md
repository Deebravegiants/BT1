### Title
Missing Address Type Validation in `Farmer.set_reward_targets` Allows DID Address as Pool Target — (`chia/farmer/farmer.py`)

### Summary

`Farmer.set_reward_targets` calls `decode_puzzle_hash` on caller-supplied addresses without validating that they are `AddressType.XCH`. A valid DID address (`did:chia:...`) passes `decode_puzzle_hash` successfully, and the resulting puzzle hash is persisted to `config.yaml`, permanently redirecting pool rewards to a DID singleton puzzle hash.

### Finding Description

`FarmerRpcApi.set_reward_targets` passes the raw caller-supplied strings directly to `Farmer.set_reward_targets` with no pre-validation: [1](#0-0) 

`Farmer.set_reward_targets` then calls `decode_puzzle_hash` on both targets and writes them to config: [2](#0-1) 

There is **no call** to `ensure_valid_address` or any `AddressType` check anywhere in `farmer.py`: [3](#0-2) 

`ensure_valid_address` enforces HRP prefix matching against an `allowed_types` set (e.g., `{AddressType.XCH}`), which would reject `did:chia:` prefixed addresses. `decode_puzzle_hash` performs no such check — it decodes any valid bech32m payload regardless of HRP. Since DID addresses are valid 32-byte bech32m strings with HRP `did:chia:`, they decode without error. [4](#0-3) 

### Impact Explanation

After a successful call, `config["pool"]["xch_target_address"]` is set to the DID address string and `self.pool_target` is set to the DID singleton's puzzle hash. All subsequent pool block rewards are sent to that puzzle hash. A DID singleton puzzle hash is not a standard pay-to-puzzle-hash address; coins sent there are either unspendable or controlled by whoever owns the DID singleton. This constitutes permanent pool reward diversion.

### Likelihood Explanation

The farmer RPC has no authentication beyond network-level access control. Any process or user with access to the farmer RPC port (default `localhost:8559`) can call `/set_reward_targets`. This includes any local process on the farmer machine. The precondition (RPC accessible) is the normal operating state of a running farmer.

### Recommendation

In `Farmer.set_reward_targets`, validate both addresses against `{AddressType.XCH}` using `ensure_valid_address` before calling `decode_puzzle_hash`:

```python
from chia.wallet.util.address_type import AddressType, ensure_valid_address

def set_reward_targets(self, farmer_target_encoded, pool_target_encoded):
    if farmer_target_encoded is not None:
        ensure_valid_address(farmer_target_encoded, allowed_types={AddressType.XCH}, config=self.config)
    if pool_target_encoded is not None:
        ensure_valid_address(pool_target_encoded, allowed_types={AddressType.XCH}, config=self.config)
    # ... rest of function
```

### Proof of Concept

1. Generate a valid DID address (any `did:chia:` bech32m string encoding 32 bytes).
2. POST to `/set_reward_targets` with `{"pool_target": "did:chia:<valid_payload>"}`.
3. Observe no error is returned.
4. Read `config.yaml` — `pool.xch_target_address` now contains the DID address.
5. Verify `self.pool_target` equals the DID singleton's puzzle hash, not an XCH puzzle hash.

def ensure_valid_address(address: str, *, allowed_types: set[AddressType], config: dict[str, Any]) -> str:
    hrp, b32data = bech32_decode(address)
    if not b32data or not hrp:
        raise ValueError(f"Invalid address: {address}")
    # Match by prefix (hrp) and return the corresponding address type
    address_type = next(
        (addr_type for (addr_type, addr_hrp) in ((a, a.hrp(config)) for a in allowed_types) if addr_hrp == hrp),
        None,
    )
    if address_type is None:
        raise ValueError(
            f"Invalid address: {address}. "
            f"Expected an address with one of the following prefixes: {[t.hrp(config) for t in allowed_types]}"
        )
