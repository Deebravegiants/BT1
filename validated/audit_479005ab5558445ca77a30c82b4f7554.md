### Title
`LRTConfig.addNewSupportedAsset` Creates Exploitable Intermediate State Before Oracle Is Set, Freezing rsETH Price Updates and Protocol Fee Minting - (File: contracts/LRTConfig.sol / contracts/LRTOracle.sol)

---

### Summary

`LRTConfig.addNewSupportedAsset()` immediately appends a new asset to `supportedAssetList`, but the corresponding price oracle in `LRTOracle` is configured via a separate, subsequent call to `updatePriceOracleFor()`. Between these two transactions, `LRTOracle.updateRSETHPrice()` reverts for every caller because `_getTotalEthInProtocol()` iterates over all entries in `supportedAssetList` and calls `getAssetPrice()` for each — which reverts with `AssetOracleNotSupported` for the newly added asset whose oracle is not yet set. This freezes the rsETH price and blocks protocol fee minting for the entire duration of the intermediate state.

---

### Finding Description

**Step 1 — Asset added, oracle not yet set:**

`LRTConfig.addNewSupportedAsset()` (gated by `TIME_LOCK_ROLE`) calls `_addNewSupportedAsset()`, which sets `isSupportedAsset[asset] = true` and pushes the asset into `supportedAssetList`:

```solidity
isSupportedAsset[asset] = true;
supportedAssetList.push(asset);
depositLimitByAsset[asset] = depositLimit;
```

At this point `assetPriceOracle[newAsset]` in `LRTOracle` is still `address(0)`.

**Step 2 — `updateRSETHPrice()` is now broken for all callers:**

`LRTOracle._getTotalEthInProtocol()` iterates over the full `supportedAssetList` returned by `lrtConfig.getSupportedAssetList()` and calls `getAssetPrice(asset)` for every entry:

```solidity
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);   // ← reverts for newAsset
    ...
}
```

`getAssetPrice()` carries the `onlySupportedOracle` modifier:

```solidity
modifier onlySupportedOracle(address asset) {
    if (assetPriceOracle[asset] == address(0)) {
        revert AssetOracleNotSupported();
    }
    _;
}
```

Because `assetPriceOracle[newAsset] == address(0)`, the revert propagates through `_getTotalEthInProtocol()` → `_updateRsETHPrice()` → the public `updateRSETHPrice()`. The function is callable by anyone but reverts for everyone until `updatePriceOracleFor(newAsset, oracle)` is executed by the admin.

**Step 3 — Consequences of the frozen price:**

`_updateRsETHPrice()` is the only place where:
- `rsETHPrice` is updated
- Protocol fees are minted to the treasury (`IRSETH.mint(treasury, rsethAmountToMintAsProtocolFee)`)

While the price is frozen, all staking rewards that accrue in EigenLayer strategies are not reflected in `rsETHPrice`. Any user who deposits existing supported assets (stETH, ETHx) during this window uses the stale (understated) `rsETHPrice`, receiving more `rsETH` than the fair share — diluting existing holders. Protocol fee minting is also completely blocked.

**Step 4 — The two-role separation widens the window:**

`addNewSupportedAsset` requires `TIME_LOCK_ROLE` (typically a timelock contract), while `updatePriceOracleFor` requires `DEFAULT_ADMIN_ROLE`. These are separate actors and separate transactions. In practice the window can span multiple blocks or longer.

---

### Impact Explanation

- **Protocol fee minting is frozen** for the entire intermediate window. Rewards that accrue during this period are never captured as fees — this is a temporary freezing of unclaimed yield.
- **rsETH price is stale.** Depositors of existing assets (stETH, ETHx) receive rsETH calculated against the last stored `rsETHPrice`. If rewards have accrued since the last update, the price is understated and depositors are overminted rsETH at the expense of existing holders.
- **Impact classification: Medium** — temporary freezing of unclaimed yield; share/asset mis-accounting.

---

### Likelihood Explanation

- `addNewSupportedAsset` and `updatePriceOracleFor` are separate privileged calls with different role requirements, making a single-transaction atomic setup impossible without a multicall wrapper.
- The window is observable on-chain; a depositor watching for `AddedNewSupportedAsset` events can immediately deposit at the stale price before the oracle is set.
- The admin may not set the oracle in the same block, especially if the timelock executes `addNewSupportedAsset` automatically and the admin must react manually.
- **Likelihood: Low** — requires the admin to not atomically set the oracle, but the structural separation of roles makes this a realistic operational pattern.

---

### Recommendation

1. **Atomic asset onboarding:** Extend `addNewSupportedAsset` (or add a new function) to accept the oracle address as a parameter and call `LRTOracle.updatePriceOracleFor` atomically in the same transaction, so the asset is never in `supportedAssetList` without a valid oracle.
2. **Guard `_getTotalEthInProtocol`:** Skip assets whose oracle is not set (or revert with a clear error naming the missing asset) rather than reverting the entire price update.
3. **Rename / document:** If `addNewSupportedAsset` is intended to be called before the oracle is set, document this clearly and add a `whenOracleSet` guard on `updateRSETHPrice` to prevent silent staleness.

---

### Proof of Concept

```
Block N:   Admin (via timelock) calls LRTConfig.addNewSupportedAsset(newAsset, limit)
           → newAsset appended to supportedAssetList
           → assetPriceOracle[newAsset] == address(0)  (oracle NOT yet set)

Block N+1: Anyone calls LRTOracle.updateRSETHPrice()
           → _updateRsETHPrice()
           → _getTotalEthInProtocol()
           → getAssetPrice(newAsset)
           → onlySupportedOracle: assetPriceOracle[newAsset] == address(0) → REVERT
           ← updateRSETHPrice() reverts; rsETHPrice is stale; fee minting blocked

Block N+1: Depositor calls LRTDepositPool.depositAsset(stETH, amount, minRSETH, "")
           → getRsETHAmountToMint uses stale rsETHPrice (lower than fair value)
           → depositor receives excess rsETH, diluting existing holders

Block N+k: Admin calls LRTOracle.updatePriceOracleFor(newAsset, oracle)
           → assetPriceOracle[newAsset] set
           → updateRSETHPrice() works again; accrued fees during window are lost
```

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
