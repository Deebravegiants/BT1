### Title
Integer Division Truncation in `compute_royalty_amount` Enables Complete NFT Royalty Bypass During Offer Settlement — (`chia/wallet/nft_wallet/nft_wallet.py`)

### Summary
`compute_royalty_amount` uses chained integer floor-division to compute the royalty owed on an NFT offer. When the offered fungible amount is small enough relative to the royalty percentage, the result truncates to exactly 0. The wallet then silently omits the royalty `CreateCoin`, strips the `AssertPuzzleAnnouncement` that would have bound the NFT spend to a royalty payment, and passes an empty `trade_prices_list` to the NFT transfer-program puzzle. The on-chain puzzle therefore enforces no royalty at all, and the NFT creator's royalty is permanently lost on that trade.

### Finding Description

`compute_royalty_amount` computes the royalty with two sequential integer divisions:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
```

`MAX_ROYALTY_BASIS_POINTS` is 10 000. Whenever `abs(offered_amount) // royalty_split * percentage < 10 000`, the result is 0. The existing test `test_small_amount_truncates_to_zero` explicitly documents this: offering 50 mojos at 1 % (100 bp) yields royalty = 0. [1](#0-0) 

The zero royalty then propagates through three independent gates in `make_nft1_offer`, each of which silently drops the royalty rather than rejecting the offer:

**Gate 1 — `AssertPuzzleAnnouncement` filter (line 853):** Only payments with `p.amount > 0` generate an announcement. A zero-royalty `CreateCoin` produces no announcement, so the fungible spend is not bound to any royalty output. [2](#0-1) 

**Gate 2 — royalty spend skip (line 911):** If the sum of all royalty payments is 0, the entire royalty offer-mod spend is skipped with `continue`. [3](#0-2) 

**Gate 3 — `trade_prices_list` filter (line 896):** The NFT singleton spend receives only trade prices where `price[0] * royalty_percentage // 10000 != 0`. A zero trade price (from `amount // offer_side_royalty_split` truncating to 0 at line 768) is excluded, so the NFT transfer-program puzzle receives an empty list and enforces no royalty. [4](#0-3) 

The same truncation pattern also appears in the display-only `royalty_calculation` static method: [5](#0-4) 

### Impact Explanation

An unprivileged buyer can create a valid, on-chain-accepted offer for any royalty-enabled NFT while paying zero royalties to the creator. Because the NFT transfer-program puzzle receives an empty `trade_prices_list`, it does not assert any royalty payment condition. The resulting spend bundle is consensus-valid and settles on-chain. The NFT creator's royalty — a protected accounting right embedded in the singleton — is permanently diverted to 0 on that trade. This is an unauthorized accounting change affecting NFT singleton-controlled assets and constitutes unauthorized offer settlement.

### Likelihood Explanation

The threshold is low and attacker-controlled. For a 1 % royalty (100 bp), any offer below 100 mojos triggers the bypass. For a 0.01 % royalty (1 bp), any offer below 10 000 mojos (~0.00001 XCH) triggers it. The attacker simply crafts the fungible amount to be one unit below the truncation threshold. No special privilege, leaked key, or cryptographic break is required — only the ability to call `create_offer_for_ids` via the standard wallet RPC. The test suite already documents the truncation-to-zero behavior as an accepted outcome: [6](#0-5) 

### Recommendation

1. **Reject offers where the computed royalty is 0 but the NFT has a non-zero royalty percentage.** Add a guard in `compute_royalty_amount` or in `make_nft1_offer` that raises `ValueError` when `percentage > 0` and the computed `amount == 0`, forcing the user to offer a larger fungible amount.

2. **Enforce a minimum fungible amount per royalty-enabled NFT.** Before building the offer, validate that `abs(offered_amount) // royalty_split >= MAX_ROYALTY_BASIS_POINTS // percentage` (i.e., that at least 1 mojo of royalty will result).

3. **Apply the same guard to the `trade_prices` computation** at line 768 (`amount // offer_side_royalty_split`) to prevent a zero trade price from silently emptying the `trade_prices_list` passed to the NFT puzzle.

### Proof of Concept

```python
# NFT minted with royalty_percentage = 100 (1 %)
# Attacker offers 99 mojos (XCH) for the NFT

from chia.wallet.nft_wallet.nft_wallet import compute_royalty_amount

royalty = compute_royalty_amount(offered_amount=-99, royalty_split=1, percentage=100)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
