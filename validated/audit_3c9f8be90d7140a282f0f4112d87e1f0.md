## Verdict

### Title
Direct Deposits Absent from UTxO Value Conservation — ADA Created from Nothing — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

### Summary
The `directDepositsTxBodyL` field in Dijkstra-era transactions credits account balances via `applyDirectDeposits`, but the deposited amount D appears on **neither** the `consumed` nor the `produced` side of `validateValueNotConservedUTxO`. An unprivileged sender can include an arbitrary D in `directDepositsTxBodyL`, pass the conservation check without sourcing D from UTxO inputs, and have D credited to an account balance — creating ADA from nothing.

The ordering sub-question is also confirmed: because `directDepositsMissingAccounts` checks `accountsAfterCerts` (post-CERTS state), a credential C registered in the same transaction via `RegDepositTxCert` is already present when the check runs, so the attacker can register C and immediately direct-deposit to C in one transaction.

### Finding Description

`dijkstraEntitiesTransition` applies `applyDirectDeposits` to the final `CertState` after CERTS completes:

```haskell
-- Entities.hs lines 211-216
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

The UTxO rule runs `validateValueNotConservedUTxO` using `conwayConsumed` → `getConsumedDijkstraValue` → `getConsumedMaryValue`:

```haskell
-- Mary/UTxO.hs lines 82-86
consumedValue =
  sumUTxO (txInsFilter utxo (txBody ^. inputsTxBodyL))
    <> inject (refunds <> withdrawals)
```

And `dijkstraProducedValue` → `conwayProducedValue` → `getProducedMaryValue`:

```haskell
-- Conway/UTxO.hs lines 129-131
conwayProducedValue pp isStakePool txBody =
  getProducedMaryValue pp isStakePool txBody
    <+> inject (txBody ^. treasuryDonationTxBodyL)
```

`directDepositsTxBodyL` is absent from both sides. The conservation equation is:

```
UTxO_inputs + refunds + withdrawals = outputs + fee + cert_deposits + treasury_donation
```

D is not in this equation. The attacker constructs a balanced transaction (inputs = outputs + fee + cert_deposit for RegDepositTxCert) and appends `directDepositsTxBodyL = {C: D}` for any D. The conservation check passes; D is credited to C's account balance from nowhere.

### Impact Explanation

**Critical.** An unprivileged transaction sender can set `directDepositsTxBodyL` to an arbitrarily large coin value. After the transaction is accepted, that coin exists in the account balance but was never deducted from any UTxO, deposit pot, treasury, or fee pot. This is direct ADA creation from nothing, violating the fundamental ledger invariant.

### Likelihood Explanation

The attack requires only a valid Dijkstra-era transaction with a non-empty `directDepositsTxBodyL` targeting any registered account (or one registered in the same transaction). No privileged access, governance majority, or leaked key is needed. The attacker must pay the normal transaction fee and the staking deposit for `RegDepositTxCert` (if registering a new credential), but can recover the staking deposit later via `UnRegDepositTxCert`. The net cost is only the transaction fee; the gain is unbounded.

### Recommendation

Include the total direct-deposit amount on the **produced** side of the UTxO value conservation check, analogously to how `treasuryDonationTxBodyL` is included. Concretely, `dijkstraProducedValue` (and `dijkstraSubTxProducedValue` for sub-transactions) should add `inject (fold . unDirectDeposits $ txBody ^. directDepositsTxBodyL)` to the produced value. This ensures that every lovelace credited to an account via a direct deposit is sourced from the transaction's UTxO inputs.

### Proof of Concept

1. Pick any registered credential C (or register it in the same transaction with `RegDepositTxCert`).
2. Construct a Dijkstra-era transaction where:
   - `inputsTxBodyL` = {utxo_in} (value V)
   - `outputsTxBodyL` = {addr: V - fee - staking_deposit}
   - `certsTxBodyL` = {RegDepositTxCert C}
   - `directDepositsTxBodyL` = {C: 1_000_000_000_000} (1M ADA)
   - `feeTxBodyL` = fee
3. Submit the transaction.
4. `validateValueNotConservedUTxO` computes:
   - consumed = V
   - produced = (V - fee - staking_deposit) + fee + staking_deposit = V ✓
5. `dijkstraEntitiesTransition` runs CERTS (registers C), then `applyDirectDeposits` adds 1M ADA to C's balance.
6. Query C's account balance: 1M ADA exists, sourced from no UTxO input.
