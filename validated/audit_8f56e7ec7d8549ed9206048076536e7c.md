Audit Report

## Title
Permissionless `updateRate()` Enables Profitable Deposits Against Stale Cross-Chain Rate — (`contracts/cross-chain/CrossChainRateProvider.sol`, `contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary
`CrossChainRateProvider.updateRate()` and `MultiChainRateProvider.updateRate()` carry no access control, allowing any caller to trigger a rate broadcast from L1 to L2 at will. `RSETHPoolV3` on L2 mints new `wrsETH` using the stale cached rate stored in `CrossChainRateReceiver`, so a depositor who acts while the L2 rate lags the L1 price receives more `wrsETH` than their ETH is worth at the true rate. The surplus represents accrued yield belonging to existing `wrsETH` holders that is instead captured by the attacker.

## Finding Description

**Permissionless rate broadcast.** `CrossChainRateProvider.updateRate()` is declared `external payable nonReentrant` with no `onlyOwner` or role guard, meaning any EOA or contract can call it after paying the LayerZero fee. [1](#0-0) 
The same is true for `MultiChainRateProvider.updateRate()`. [2](#0-1) 

**L1 rate is public.** `RSETHRateProvider.getLatestRate()` reads `ILRTOracle(rsETHPriceOracle).rsETHPrice()`, a public view function, so any observer can compute the current L1 price without any privilege. [3](#0-2) 

**L2 receiver caches a stale rate.** `CrossChainRateReceiver` stores the last received rate in `rate` storage and only updates it inside `lzReceive`. Between LayerZero deliveries, `getRate()` returns the stale value. [4](#0-3) 

**`RSETHPoolV3` mints new supply at the stale rate.** `viewSwapRsETHAmountAndFee` divides the deposit amount by `getRate()`, which resolves to the stale `CrossChainRateReceiver.rate`. The result is passed directly to `wrsETH.mint`. [5](#0-4) [6](#0-5) 

**Exploit flow:**
1. Attacker reads `LRTOracle.rsETHPrice()` on L1 → `1.05e18`.
2. Attacker reads `CrossChainRateReceiver.rate` on L2 → `1.04e18` (stale).
3. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("")` on L2. The pool computes `rsETHAmount = 100e18 * 1e18 / 1.04e18 ≈ 96.15 wrsETH` and mints it. At the true rate the correct amount is `≈ 95.24 wrsETH`; the excess is `≈ 0.91 wrsETH`.
4. Attacker calls `RSETHRateProvider.updateRate{value: lzFee}()` on L1. LayerZero delivers the message; `CrossChainRateReceiver.rate` updates to `1.05e18`.
5. The attacker's 96.15 wrsETH is now worth `96.15 * 1.05e18 / 1e18 ≈ 100.96 ETH` — a net gain of `≈ 0.96 ETH` on a 100 ETH deposit, minus gas and LayerZero fees.

**Existing guards are insufficient.** The `dailyMintLimit` in `RSETHPoolV3` caps total daily minting but does not prevent the attack; it only limits the scale per day. The `pricePercentageLimit` guard in `LRTOracle._updateRsETHPrice()` applies only to L1 oracle updates and has no equivalent in `CrossChainRateReceiver.lzReceive()`. [7](#0-6) [8](#0-7) 

**`RSETHPoolNoWrapper` is a lesser but related case.** Its `deposit()` transfers pre-existing rsETH from the pool's balance rather than minting, so the impact is pool reserve depletion rather than supply dilution. The primary minting-based impact is in `RSETHPoolV3`. [9](#0-8) 

## Impact Explanation
**High — Theft of unclaimed yield.** The yield that accrued to existing rsETH/wrsETH holders (reflected in the L1 price rising from the stale L2 rate to the true rate) is partially captured by the attacker through the stale-rate mint. Each deposit at the stale rate mints more wrsETH than the deposited ETH warrants, diluting all existing wrsETH holders by exactly the accrued-but-not-yet-propagated yield. The profit scales linearly with deposit size and the magnitude of the rate lag, and is repeatable after every reward accrual event on L1.

## Likelihood Explanation
Both the L1 oracle price and the L2 cached rate are public on-chain state readable by any EOA. No mempool monitoring or privileged access is required. Because `updateRate()` is permissionless, the attacker controls the exact moment the rate is pushed to L2, ensuring they can always deposit before the update arrives. The attack is deterministic and repeatable; the only cost is the LayerZero fee for the rate update call.

## Recommendation
1. **Restrict `updateRate()`** in both `CrossChainRateProvider` and `MultiChainRateProvider` to a trusted role (e.g., `onlyOwner` or a dedicated `RATE_UPDATER_ROLE`), removing the attacker's ability to control rate-update timing.
2. **Add a rate-delta bound in `CrossChainRateReceiver.lzReceive()`** — reject or cap updates that increase the rate by more than a configurable percentage per period, mirroring the `pricePercentageLimit` guard already present in `LRTOracle._updateRsETHPrice()` on L1.
3. **Consider a minimum deposit lock-up** on `RSETHPoolV3` so that freshly minted wrsETH cannot be immediately redeemed or sold against the updated rate in the same or adjacent block.

## Proof of Concept
**Foundry fork test outline (Optimism/Arbitrum fork + Ethereum mainnet fork):**

```solidity
// 1. Fork L2 at a block where CrossChainRateReceiver.rate = 1.04e18
// 2. Simulate L1 rsETHPrice() returning 1.05e18 (or use a mock)
// 3. Record attacker wrsETH balance before deposit
uint256 balBefore = wrsETH.balanceOf(attacker);

// 4. Attacker deposits 100 ETH
vm.prank(attacker);
pool.deposit{value: 100 ether}("");

uint256 balAfter = wrsETH.balanceOf(attacker);
// Assert attacker received ~96.15e18 wrsETH (stale rate)
assertApproxEqRel(balAfter - balBefore, 96.15e18, 1e15);

// 5. Simulate rate update delivery (mock lzReceive with new rate 1.05e18)
vm.prank(layerZeroEndpoint);
receiver.lzReceive(srcChainId, abi.encodePacked(rateProvider, receiver), 0,
    abi.encode(1.05e18));

// 6. Compute ETH value of attacker's wrsETH at new rate
uint256 ethValue = (balAfter - balBefore) * 1.05e18 / 1e18;
// Assert ethValue > 100 ether (profit extracted)
assertGt(ethValue, 100 ether);
```

The test demonstrates that the attacker receives wrsETH worth more ETH than deposited, with the surplus sourced from existing holders' accrued yield.

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {

        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH

        // Calculate the final rsETH amount
