Audit Report

## Title
Stranded ETH on `MetricOmmSimpleRouter` / `MetricOmmPoolLiquidityAdder` Consumed by Subsequent WETH-Input Callers — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` reads `address(this).balance` when settling WETH-input payments, consuming the contract's entire native ETH balance rather than only the ETH sent in the current call. Any ETH left on the router from a prior user who omitted `refundETH()` is silently spent on behalf of the next WETH-input caller, resulting in a direct, unrecoverable loss of the original depositor's ETH.

## Finding Description
In `PeripheryPayments.pay()` (lines 73–84), when `token == WETH`, the function reads `uint256 nativeBalance = address(this).balance` and uses it to wrap and forward ETH to the pool:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // entire contract balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } ...
}
``` [1](#0-0) 

The `receive()` guard at lines 32–34 only blocks plain ETH transfers from non-WETH addresses; it does not apply to ETH sent via `msg.value` through any `payable` function. [2](#0-1) 

`exactInputSingle` (line 67) is `payable` and stores `msg.sender` as the payer in transient storage. The pool callback fires `_justPayCallback` (lines 192–199), which calls `pay()` with the original `msg.sender` as `payer`. If the user sent more ETH than consumed and did not include `refundETH()` in the same multicall, the surplus ETH persists on the contract across transaction boundaries. [3](#0-2) [4](#0-3) 

The next caller who invokes any WETH-input swap with `msg.value = 0` will have their payment fully or partially covered by the prior user's stranded ETH, because `pay()` reads the aggregate balance. The same `pay()` is shared by `MetricOmmPoolLiquidityAdder.metricOmmModifyLiquidityCallback` (lines 172–177), making liquidity additions equally affected. [5](#0-4) 

Additionally, `refundETH()` (lines 58–63) sends `address(this).balance` to `msg.sender` with no access control, so a racing attacker can also drain stranded ETH directly without performing a swap. [6](#0-5) 

No per-call ETH tracking exists anywhere in `MetricOmmSimpleRouter` or `PeripheryPayments`. The transient storage used by the router tracks callback context (pool, payer, token, mode) but not `msg.value`. [7](#0-6) 

## Impact Explanation
Direct loss of user principal (ETH). Alice's stranded ETH is irreversibly transferred to the pool on Bob's behalf; Alice receives nothing in return and has no protocol-level recovery path. This constitutes a Critical/High direct loss of user principal meeting Sherlock thresholds.

## Likelihood Explanation
The ETH-input swap pattern is explicitly supported and tested by the protocol (`test_multicall_ethInput_exactInputSingle_refundsUnusedEth`). The `refundETH()` step is optional and must be manually included in a multicall; any direct call to `exactInputSingle` with excess ETH, or any multicall that omits `refundETH()`, leaves ETH stranded. MEV bots can detect such transactions in the mempool and immediately follow up with a zero-cost WETH swap or a direct `refundETH()` call. No special privileges are required; any unprivileged caller can exploit this.

## Recommendation
Track per-call ETH using transient storage. At the entry of each `payable` function, record `msg.value` in a transient slot. In `pay()`, read only that recorded amount rather than `address(this).balance`, and decrement it as ETH is consumed. This ensures only ETH sent in the current transaction can subsidise the current caller's payment, eliminating cross-user ETH leakage.

## Proof of Concept
```
1. Alice calls exactInputSingle{value: 2 ether}(amountIn: 1 ether, tokenIn: WETH, ...)
   → pay() reads nativeBalance = 2 ether >= 1 ether
   → wraps 1 ether, sends WETH to pool
   → 1 ether remains on the router (Alice omitted refundETH)

2. Bob calls exactInputSingle{value: 0}(amountIn: 1 ether, tokenIn: WETH, ...)
   → pay() reads nativeBalance = 1 ether >= 1 ether
   → wraps Alice's 1 ether, sends WETH to pool on Bob's behalf
   → Bob receives swap output; Alice's 1 ether is permanently lost

Alternatively:
2b. Bob calls refundETH() directly
    → receives Alice's 1 ether with no swap required
```
