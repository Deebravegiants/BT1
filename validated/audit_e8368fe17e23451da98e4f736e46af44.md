### Title
Double-Counting of Deposit Refunds Across Sub-Transactions Allows ADA Creation from Nothing - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces nested sub-transactions. The `getConsumedDijkstraValue` function aggregates consumed values (including deposit refunds) from all sub-transactions using the **same pre-transaction ledger state snapshot**. This allows multiple sub-transactions to each claim the same deposit refund via `UnRegDepositTxCert` for the same staking credential, inflating the `consumed` side of the balance check and enabling an attacker to produce more ADA in outputs than they supplied in inputs — creating ADA from nothing. The developers have explicitly acknowledged this gap in the test suite with a disabled test and a `TODO: predicate failure not yet implemented` placeholder.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — double-counting of deposit refunds across sub-transactions.

**Root cause — `getConsumedDijkstraValue`:** [1](#0-0) 

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
```

The `lookupStakingDeposit` closure is derived once from the **pre-transaction** `CertState` and passed unchanged to every sub-transaction's consumed-value calculation. There is no state update between sub-transactions. If sub-transaction 1 and sub-transaction 2 both contain `UnRegDepositTxCert stakingCred D`, both calls to `getConsumedValue` will find the same deposit `D` in the pre-transaction state and add it to the total consumed value, yielding `2*D` in consumed refunds.

**Root cause — `dijkstraTotalRefundsTxCerts`:** [2](#0-1) 

```haskell
-- | Unlike previous eras, we no longer need to lookup refunds from the ledger state,
-- since all of the certificates specify the actual refund and ledger rules will validate
-- that they are accurate.
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert _ deposit -> deposit
  _ -> zero
```

This function reads the deposit amount directly from the certificate without cross-checking against the ledger state or tracking whether the same credential has already been claimed. It is called independently for each sub-transaction body, so N sub-transactions each carrying `UnRegDepositTxCert stakingCred D` contribute `N*D` to the total consumed value.

**Deposit pot update is not symmetric:** The deposit pot reduction uses `conwayCertsTotalRefundsTxBody` (reused by Dijkstra via `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/State/CertState.hs`), which only inspects the **top-level** transaction body's certificates, not sub-transaction certificates. [3](#0-2) 

So the deposit pot is reduced by at most `1*D` (from a top-level cert), while the balance check allows `N*D` on the consumed side.

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

The balance invariant `consumed == produced` is the core preservation-of-value check. With N sub-transactions each carrying `UnRegDepositTxCert stakingCred D`:

- `consumed = UTxO_inputs + N*D + withdrawals`
- `produced = outputs + fees + new_deposits`

For the equation to hold, `outputs = UTxO_inputs + N*D - fees - new_deposits`. The attacker places `N*D` more ADA in outputs than they supplied in inputs. Only `1*D` (or `0*D` if the unregistration is in a sub-transaction only) is removed from the deposit pot. The difference `(N-1)*D` is created from nothing, violating the total ADA supply invariant.

---

### Likelihood Explanation

**High.** The attack path is straightforward: register a staking credential, then submit a top-level Dijkstra transaction containing N sub-transactions each with `UnRegDepositTxCert` for the same credential. No privileged access, governance majority, or external dependency is required. The only prerequisite is that the Dijkstra era is active on-chain. The developers have already identified the scenario and noted the missing guard.

---

### Recommendation

1. **Track claimed credentials across sub-transactions.** Before computing the consumed value, collect all `UnRegDepositTxCert` and `UnRegDRepTxCert` credentials across all sub-transactions and enforce uniqueness. Reject the top-level transaction if any credential appears more than once.

2. **Thread state through sub-transaction processing.** Update `lookupStakingDeposit` / `lookupDRepDeposit` after each sub-transaction is processed, so subsequent sub-transactions see the post-unregistration state (deposit = `Nothing`).

3. **Enable and complete the disabled test.** Replace `error "TODO: predicate failure not yet implemented"` with the actual `ConwayDelegPredFailure` or equivalent, and un-skip the test with `it`.

---

### Proof of Concept

```
1. Register staking credential C, paying deposit D.
   Ledger state: accounts[C].deposit = D

2. Construct top-level Dijkstra transaction T:
     subTx1.certs = [UnRegDepositTxCert C D]
     subTx2.certs = [UnRegDepositTxCert C D]
     T.outputs    = [addr -> (UTxO_inputs + 2*D - fees)]

3. getConsumedDijkstraValue evaluates:
     consumed = UTxO_inputs
              + getConsumedMaryValue(subTx1) = D   ← lookupStakingDeposit C = Just D
              + getConsumedMaryValue(subTx2) = D   ← same pre-tx state, still Just D
              = UTxO_inputs + 2*D

4. produced = (UTxO_inputs + 2*D - fees) + fees = UTxO_inputs + 2*D  ✓ balance holds

5. Deposit pot is reduced by D (first unregistration only).
   Net ADA created: D.

6. Repeat with N sub-transactions to extract N*D ADA per top-level transaction.
```
