### Title
Dijkstra Batch Transaction Minimum Fee Excludes Sub-Transaction Reference Script Costs — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the minimum fee check for batch transactions uses `getConwayMinFeeTxUtxo`, which computes the reference-script fee component using only the top-level transaction's inputs. Sub-transactions in the same batch can carry their own reference inputs pointing to large Plutus scripts, but those scripts are never counted in the minimum fee. The function `batchNonDistinctRefScriptsSize` was written specifically to aggregate reference-script sizes across the entire batch, yet it is never wired into `getMinFeeTxUtxo`. An attacker can therefore submit a batch whose sub-transactions reference arbitrarily large scripts while paying only the base fee, bypassing the tiered `minFeeRefScriptCostPerByte` deterrent introduced after the June 2024 DDoS attack.

---

### Finding Description

`EraUTxO DijkstraEra` sets:

```haskell
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
```

`getConwayMinFeeTxUtxo` is:

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

`txNonDistinctRefScriptsSize` inspects only the top-level transaction's spend inputs and reference inputs. It has no knowledge of sub-transactions.

The Dijkstra era introduces nested sub-transactions (`dtbrSubTransactions :: OMap TxId (Tx SubTx era)`). Each sub-transaction body carries its own reference-input set (`dstbrReferenceInputs :: Set TxIn`), which can point to UTxO entries containing large Plutus scripts.

The module itself defines `batchNonDistinctRefScriptsSize` to handle exactly this case:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

This function is exported but never called from `getMinFeeTxUtxo`.

The minimum fee check in `dijkstraUtxoTransition` is:

```haskell
{- minfee pp txTop utxo₀ ≤ txfee txb -}
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
```

`validateFeeTooSmallUTxO` calls `getMinFeeTxUtxo`, which resolves to `getConwayMinFeeTxUtxo`, which calls `txNonDistinctRefScriptsSize` — the top-level-only variant. Sub-transaction reference scripts are therefore invisible to the fee check.

The sub-transaction produced-value function confirms sub-transactions have no fee field of their own:

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody
               <> txBody ^. treasuryDonationTxBodyL)
    <> burnedMultiAssets txBody
```

No `feeTxBodyL` term appears; the entire batch pays a single fee declared in the top-level body. That fee is validated only against the top-level reference-script cost.

---

### Impact Explanation

The `minFeeRefScriptCostPerByte` / `tierRefScriptFee` mechanism was introduced in Conway specifically to make large-reference-script transactions expensive and deter DDoS attacks. In the Dijkstra era, an attacker can place all large reference scripts inside sub-transaction reference inputs, keeping the top-level transaction's reference-script size at zero. The minimum fee check passes with only the base `a + b·size` fee, while the node must still deserialize every sub-transaction reference script to validate the batch. The attacker pays a fee proportional to the top-level transaction size alone, not the full computational cost of the batch.

**Impact category**: Medium — attacker-controlled transactions modify fees outside design parameters (paying below the intended minimum for the actual validation work performed).

---

### Likelihood Explanation

The Dijkstra era is experimental but is production Haskell code present in the repository. Once deployed, any unprivileged transaction sender can craft such a batch. No special keys, governance majority, or privileged access is required. The construction is straightforward: place large reference scripts in UTxO entries, reference them only from sub-transaction `referenceInputs`, and set the top-level fee to the Conway-era minimum. The existence of `batchNonDistinctRefScriptsSize` as an exported but unused function strongly suggests the omission is unintentional.

---

### Recommendation

Override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance with a Dijkstra-specific implementation that uses `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx
```

Then set:

```haskell
getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo
```

This ensures the tiered reference-script fee covers all scripts deserialized during batch validation, not only those referenced by the top-level transaction.

---

### Proof of Concept

1. Deploy a large Plutus script (e.g., 50 KiB) as a reference script on a UTxO output `U_ref`.
2. Construct a Dijkstra batch transaction `T` where:
   - The top-level body has no reference inputs and no spend inputs that carry reference scripts.
   - One sub-transaction body includes `U_ref` in its `referenceInputs`.
3. Compute `getConwayMinFeeTxUtxo pp T utxo`. Because `txNonDistinctRefScriptsSize` sees zero reference-script bytes at the top level, the result is the base `a + b·|T|` fee with no `tierRefScriptFee` surcharge.
4. Set `txfee` to this base minimum and submit `T`.
5. The `validateFeeTooSmallUTxO` check passes. The node must nonetheless deserialize the 50 KiB script from `U_ref` to validate the sub-transaction, incurring the full deserialization cost that `minFeeRefScriptCostPerByte` was designed to price in — but the attacker paid nothing for it.
