Audit Report

## Title
Missing `priceLimitX64 != 0` Guard in Initial Price-Limit Check Silently Blocks All `!zeroForOne` Swaps with No Price Limit — (File: `metric-core/contracts/MetricOmmPool.sol`)

## Summary
Both `_swapToken1ForToken0SpecifiedInput` and `_swapToken1ForToken0SpecifiedOutput` perform an early-exit check `if (params.priceLimitX64 <= initialPriceX64)` without first confirming `priceLimitX64 != 0`. When a caller passes `priceLimitX64 == 0` (the conventional "no upper price limit" sentinel), the condition `0 <= initialPriceX64` is unconditionally true because `initialPriceX64` is always positive, causing both functions to immediately return zero amounts. The swap transaction completes successfully but transfers nothing to the recipient and debits nothing from the caller.

## Finding Description

The initial guard in `_swapToken1ForToken0SpecifiedInput`:

```solidity
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}
``` [1](#0-0) 

And in `_swapToken1ForToken0SpecifiedOutput`:

```solidity
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0, 0);
}
``` [2](#0-1) 

Both lack the `priceLimitX64 != 0` guard that the inner loop of `_swapToken1ForToken0SpecifiedInput` explicitly includes:

```solidity
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= upperPriceX64) {
    break;
}
``` [3](#0-2) 

`initialPriceX64` is computed via `SwapMath.calculatePriceAtBinPosition(lowerPriceX64, upperPriceX64, ...)` where both price bounds are derived from `distanceE6ToPriceX64(...)` applied to `midPriceX64`. [4](#0-3) 

`midPriceX64` is derived from `bidPriceX64` and `askPriceX64`, which the oracle enforces to be strictly positive (`bid == 0` reverts with `BidIsZero()`). [5](#0-4) 

Therefore `initialPriceX64 > 0` always holds, and `0 <= initialPriceX64` is always true, causing the early return to fire unconditionally when `priceLimitX64 == 0`.

The `swap` entry point imposes no non-zero requirement on `priceLimitX64`: [6](#0-5) 

After the early return, `amount0Delta = 0` and `amount1Delta = 0`. In the `!zeroForOne` branch, `transferToken0` is skipped (condition `amount0Delta < 0` is false), the callback is invoked with `(0, 0)`, and the `IncorrectDelta` guard evaluates false (`amount1Delta > 0` is false), so the transaction succeeds silently. [7](#0-6) 

Note: the symmetric `zeroForOne` functions (`_swapToken0ForToken1SpecifiedInput`, `_swapToken0ForToken1SpecifiedOutput`) use `priceLimitX64 >= initialPriceX64` for their initial check. When `priceLimitX64 == 0`, this evaluates `0 >= initialPriceX64` which is false (since `initialPriceX64 > 0`), so the `zeroForOne` direction is unaffected. The bug is exclusive to the `!zeroForOne` path. [8](#0-7) 

## Impact Explanation
The entire `!zeroForOne` swap path (token1 → token0) is completely unusable when `priceLimitX64 == 0`. Any router or integrator following the Uniswap V3 convention of passing `0` for "no price limit" — including `MetricOmmSimpleRouter` — will silently receive zero output on every such swap. This constitutes broken core pool swap functionality: the recipient receives zero token0, the caller pays nothing, and no error is raised. This meets the "broken core pool functionality causing loss of funds or unusable swap flows" impact gate.

## Likelihood Explanation
Medium-to-High. The `swap` function accepts `priceLimitX64` as a raw `uint128` with no non-zero enforcement. The inner loop's explicit `!= 0` guard is strong evidence the protocol intended `0` to mean "no limit." Any router built on this convention will silently fail on all `!zeroForOne` swaps. The condition is deterministic and reproducible with no special preconditions beyond calling `swap` with `zeroForOne = false` and `priceLimitX64 = 0`.

## Recommendation
Add the `priceLimitX64 != 0` guard to the initial check in both functions, matching the inner loop pattern:

```solidity
// _swapToken1ForToken0SpecifiedInput
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}

// _swapToken1ForToken0SpecifiedOutput
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0, 0);
}
```

## Proof of Concept
1. Deploy a pool with token0/token1 and add liquidity across bins.
2. Call `pool.swap(recipient, false, 1e18, 0, callbackData, "")` — `zeroForOne = false`, `priceLimitX64 = 0`.
3. `_executeSwap` routes to `_swapToken1ForToken0SpecifiedInput` with `params.priceLimitX64 = 0`.
4. `_getInitialStateForSwap(false, false, params, amountInScaled)` computes `initialPriceX64 > 0` from the current bin position.
5. `0 <= initialPriceX64` → `true` → function returns `(0, 0, 0)` immediately.
6. Back in `_executeSwap`: `amount0DeltaScaled = 0`, `amount1DeltaScaled = 0`.
7. `deltasScaledToExternal(0, 0)` → `amount0Delta = 0`, `amount1Delta = 0`.
8. In `swap`: `amount0Delta < 0` → false, no token0 sent to recipient.
9. Callback invoked with `(0, 0)` — caller pays nothing.
10. `amount1Delta > 0` → false, `IncorrectDelta` not triggered.
11. `swap` returns `(0, 0)`. Recipient receives 0 token0. Swap silently fails.
