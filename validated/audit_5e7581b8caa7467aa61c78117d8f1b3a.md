### Title
Unbounded Governance Proposal Accumulation Causes O(n) Stack Growth in `ratifyTransition` at Epoch Boundary - (File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs)

### Summary

The Conway governance `RATIFY` rule processes every accumulated governance proposal at each epoch boundary via a recursive Haskell function (`ratifyTransition`). Because there is no protocol-enforced cap on the total number of live proposals in the ledger state, and because the non-ratified branch of `ratifyTransition` is **not** tail-recursive, an attacker who accumulates a sufficient number of proposals can cause a stack overflow during epoch-boundary processing, halting all honest nodes and permanently freezing all funds and deposits until a hard fork is deployed.

---

### Finding Description

**Root cause — no limit on live proposals.**

The `GOV` rule accepts any number of `ProposalProcedure` entries from a transaction and appends each one to the global `Proposals` OMap without checking the total count:

```haskell
proposals <-
  foldlM' processProposal st $
    indexedGovProps (SSeq.fromStrict (OSet.toStrictSeq gsProposalProcedures))
``` [1](#0-0) 

The only economic barrier is `ppGovActionDepositL`, a governance parameter that can be set to any value and that is refunded on expiry, making the attack capital-efficient over multiple epochs.

**Root cause — non-tail-recursive `ratifyTransition`.**

At every epoch boundary, `finishDRepPulser` calls `runConwayRatify`, which calls `applySTS @(RATIFY era)` with the full snapshot of all live proposals as the signal:

```haskell
!ratifySig = RatifySignal dpProposals
...
!ratifyState' = runConwayRatify dpGlobals ratifyEnv ratifyState ratifySig
``` [2](#0-1) 

`ratifyTransition` processes the proposal sequence by recursively calling `trans @(RATIFY era)`. The **else branch** (proposal not yet ratified) is **not** tail-recursive — it binds the result of the recursive call and then performs additional work:

```haskell
else do
  -- This action hasn't been ratified yet. Process the remaining actions.
  st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
  -- Finally, filter out actions that have expired.
  if gasExpiresAfter < reCurrentEpoch
    then pure $ st' & rsExpiredL %~ Set.insert gasId
    else pure st'
``` [3](#0-2) 

Because the vast majority of proposals in any realistic scenario will **not** be ratified (they require supermajority votes from DReps, SPOs, and the committee), the `else` branch is taken for nearly every proposal, causing the Haskell call stack to grow **O(n)** in the number of live proposals. With a sufficiently large `n`, this produces a stack overflow inside every honest node's epoch-boundary processing.

The `then` branch (proposal ratified) is tail-recursive and does not contribute to stack growth:

```haskell
then do
  newEnactState <- trans @(EraRule "ENACT" era) $ TRC (...)
  let st' = ...
  trans @(RATIFY era) $ TRC (env, st', RatifySignal sigs)
``` [4](#0-3) 

**Epoch-boundary trigger.**

The `EPOCH` rule calls `extractDRepPulsingState`, which forces `finishDRepPulser` and therefore `runConwayRatify` over all proposals:

```haskell
ratifyState@RatifyState {rsEnactState, rsEnacted, rsExpired} =
  extractDRepPulsingState pulsingState
``` [5](#0-4) 

This is an unconditional step at every epoch boundary; there is no way to skip it.

**Proposal accumulation across epochs.**

Proposals live for `ppGovActionLifetime` epochs (an `EpochInterval`). An attacker can submit proposals across many consecutive epochs, building up a backlog that persists until expiry. Because each proposal's deposit is refunded on expiry, the net cost is only the opportunity cost of locked ADA, not a permanent burn.

---

### Impact Explanation

When the Haskell runtime exhausts its stack during `ratifyTransition`, every honest node crashes at the same epoch boundary. On restart, each node attempts the same epoch transition and crashes again. The chain halts. All ADA locked in UTxOs, staking rewards, and governance proposal deposits becomes inaccessible. Recovery requires a hard fork to either convert `ratifyTransition` to an iterative algorithm, raise the runtime stack limit, or surgically remove the excess proposals from the ledger state.

This matches: **High — Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork.**

---

### Likelihood Explanation

- Any unprivileged user holding sufficient ADA can submit governance proposals.
- Proposals accumulate across epochs; the attacker does not need to act in a single block.
- The deposit is refunded on expiry, so the attack is capital-efficient.
- No governance majority or privileged role is required to submit proposals.
- The `ppGovActionDeposit` parameter is currently 100,000 ADA on mainnet, which is a meaningful but not prohibitive barrier for a well-funded attacker; if governance ever lowers this value, the barrier drops proportionally.

---

### Recommendation

1. **Convert `ratifyTransition` to an iterative fold.** Replace the recursive STS self-call with a strict left fold over the proposal sequence, eliminating O(n) stack growth entirely.

2. **Enforce a protocol-level cap on live proposals.** Add a `ppMaxGovActions` parameter (or derive a hard cap from existing parameters) and reject new proposals in the `GOV` rule when the cap is reached, analogous to how `ppMaxTxSize` bounds transaction size.

3. **Raise the minimum deposit.** As a short-term mitigation, ensure `ppGovActionDeposit` is set high enough that accumulating thousands of proposals is economically infeasible.

---

### Proof of Concept

1. Set `ppGovActionLifetime = EpochInterval N` (e.g., 6 epochs, the current mainnet value).
2. Each epoch, submit the maximum number of `InfoAction` proposals affordable from a funded wallet, paying `ppGovActionDepositL` per proposal.
3. After `N` epochs, the live proposal count equals `N × proposals_per_epoch`.
4. At the next epoch boundary, `finishDRepPulser` calls `runConwayRatify` with all live proposals as the `RatifySignal`.
5. `ratifyTransition` recurses once per proposal via the non-tail-recursive `else` branch (since `InfoAction` proposals are never ratified — `votingDRepThreshold` returns `NoVotingThreshold` for `InfoAction`, so `dRepAccepted` returns `False`). [6](#0-5) 

6. With a stack depth proportional to the number of live proposals, the Haskell runtime raises a stack overflow exception, crashing the node before the epoch transition completes.
7. Every honest node repeats this crash on restart, halting the chain.
