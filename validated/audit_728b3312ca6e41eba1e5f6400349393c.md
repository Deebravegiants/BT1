### Title
Batch Offer Cancellation Griefing via Announcement Ring Dependency — (`File: chia/wallet/trade_manager.py`)

### Summary
`cancel_pending_offers` in `chia/wallet/trade_manager.py` chains all cancellation coin spends into a single atomic `SpendBundle` using a ring of `CREATE_COIN_ANNOUNCEMENT` / `ASSERT_COIN_ANNOUNCEMENT` conditions. Any taker who front-runs the batch by accepting one of the included offers causes the entire batch cancellation to fail on-chain, leaving all other offers permanently open and the maker's assets exposed.

### Finding Description

When a user calls the `cancel_offers` RPC endpoint, `WalletRpcApi.cancel_offers` pages through pending offers and calls `cancel_pending_offers` with a list of `trade_ids`.

Inside `cancel_pending_offers`, a shared `announcement_nonce` is derived from all trade IDs:

```python
announcement_nonce: bytes32 = std_hash(b"".join(trade_ids))
```

For every cancellation coin across every offer, a `CreateCoinAnnouncement` and its corresponding `AssertCoinAnnouncement` are appended to two deques. Then the assertions are rotated by one position to form a ring:

```python
announcement_assertions.rotate(1)
```

This means coin N asserts the announcement from coin N-1, and coin 0 asserts the announcement from the last coin. Every coin in the batch is therefore mutually dependent on every other coin.

All individual spend bundles are then aggregated into a single atomic `final_spend_bundle`:

```python
final_spend_bundle = WalletSpendBundle.aggregate(
    [tx.spend_bundle for tx in all_txs if tx.spend_bundle is not None]
)
```

Because the announcement ring makes every spend depend on every other spend, if any single cancellation coin is already spent when the bundle reaches the mempool, the entire bundle fails with `ASSERT_ANNOUNCE_CONSUMED_FAILED`. There is no NoThrow / partial-success variant.

### Impact Explanation

An unprivileged attacker who is the taker of any one offer included in the batch can:

1. Observe the pending batch cancellation bundle in the mempool.
2. Front-run it by submitting an acceptance of their offer, spending that offer's coin.
3. The batch cancellation bundle becomes invalid because the announcement ring is broken.
4. All other offers in the batch remain in `PENDING_ACCEPT` state on-chain, keeping the maker's XCH, CATs, or NFTs locked and exposed.

The maker's local wallet marks all offers as `PENDING_CANCEL` (line 384), but the on-chain state is unchanged — a direct corruption of offer/trade settlement state. The attacker can repeat this indefinitely against every retry, permanently preventing batch cancellation and forcing the maker to cancel each offer individually while their assets remain at risk.

### Likelihood Explanation

The attack requires only that the attacker be the taker of one of the offers in the batch. Any open offer is publicly visible and can be accepted by anyone. The `cancel_offers` RPC is the standard path for bulk cancellation (e.g., cancelling all offers for a given asset). The announcement ring is always constructed when `len(trade_records) > 1` or `len(cancellation_coins) > 1`, which is the common case for batch cancellation.

### Recommendation

- **Short term:** Implement a NoThrow / best-effort variant of batch cancellation that submits each offer's cancellation as an independent spend bundle, without cross-offer announcement dependencies. The announcement ring is only needed to prevent partial cancellation of a single multi-coin offer, not across distinct offers.
- **Long term:** When building multi-offer batch cancellations, validate that each offer's cancellation coin is still unspent before including it in the ring. Remove any offer whose coin is already spent from the batch rather than failing the entire batch.

### Proof of Concept

1. Maker has two open offers: offer A (XCH→CAT) and offer B (XCH→NFT).
2. Maker calls `cancel_offers` with `cancel_all=True`. `cancel_pending_offers` is called with `[trade_id_A, trade_id_B]`.
3. `cancel_pending_offers` builds a two-coin announcement ring: coin_A asserts coin_B's announcement, coin_B asserts coin_A's announcement. Both are aggregated into one `final_spend_bundle`.
4. Attacker (taker of offer B) observes the pending cancellation bundle and front-runs it by submitting `offer_B.to_valid_spend()`, spending offer B's coin.
5. The batch cancellation bundle reaches the mempool. Offer B's coin is already spent, so `ASSERT_ANNOUNCE_CONSUMED_FAILED` is raised and the entire bundle is rejected.
6. Offer A remains open on-chain. Maker's XCH is still locked in offer A and can be taken by anyone.
7. Attacker repeats step 4 on every retry, indefinitely blocking batch cancellation.

    @tx_endpoint(push=True, merge_spends=False)
    @marshal
    async def cancel_offers(
        self,
        request: CancelOffers,
        action_scope: WalletActionScope,
        extra_conditions: tuple[Condition, ...] = tuple(),
    ) -> CancelOffersResponse:
        trade_mgr = self.service.wallet_state_manager.trade_manager
        log.info(f"Start cancelling offers for  {'all' if request.cancel_all else 'asset_id: ' + request.asset_id} ...")
        # Traverse offers page by page
        for start in count(0, request.batch_size):
            records = {
                record.trade_id: record
                for record in await trade_mgr.trade_store.get_trades_between(
                    start,
                    start + request.batch_size,
                    reverse=True,
                    exclude_my_offers=False,
                    exclude_taken_offers=True,
                    include_completed=False,
                )
                if request.cancel_all
                or (record.offer != b"" and request.query_key in Offer.from_bytes(record.offer).arbitrage())
            }

            if records == {}:
                break

            async with self.service.wallet_state_manager.lock:
                await trade_mgr.cancel_pending_offers(
                    list(records.keys()),
                    action_scope,
                    request.batch_fee,
                    request.secure,
                    records,
                    extra_conditions=extra_conditions,
                )

            log.info(f"Created offer cancellations for {start} to {start + request.batch_size} ...")

        # tx_endpoint will fill in default values here
        return CancelOffersResponse(unsigned_transactions=[], transactions=[])
