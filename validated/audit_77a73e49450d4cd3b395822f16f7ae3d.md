Audit Report

## Title
Public `updateRSETHPrice()` Enables Deposit/Price-Update/Instant-Withdrawal Sandwich to Steal Yield — (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is callable by any address with no access control beyond `whenNotPaused`. Because `rsETHPrice` is a cached state variable that grows stale as underlying LSTs accrue staking rewards, an attacker can deposit at the stale (lower) price, trigger the price update to the current (higher) value, and immediately call `instantWithdrawal()` to redeem at the updated price — extracting yield that belongs to existing rsETH holders.

## Finding Description

**Root cause:** `rsETHPrice` is a cached value updated only on explicit calls to `updateRSETHPrice()`. Both deposit minting and instant withdrawal payout use this same cached value, but an attacker can atomically control when the cache is refreshed.

**Step 1 — Deposit at stale price.**
`updateRSETHPrice()` has no access control: [1](#0-0) 

`getRsETHAmountToMint()` divides by the stored `rsETHPrice`: [2](#0-1) 

When `rsETHPrice` is stale-low (e.g., 1.000 while actual is 1.005), the attacker receives more rsETH than their deposit is worth.

**Step 2 — Trigger price update.**
The attacker calls `updateRSETHPrice()` directly. `_updateRsETHPrice()` recomputes the price from live oracle data and updates the stored `rsETHPrice` to the current higher value: [3](#0-2) 

**Step 3 — Instant withdrawal at updated price.**
`instantWithdrawal()` calls `getExpectedAssetAmount()`, which reads the now-updated `rsETHPrice`: [4](#0-3) [5](#0-4) 

The attacker burns their inflated rsETH balance and receives assets computed at the higher price. There is no withdrawal delay for `instantWithdrawal()` — unlike `initiateWithdrawal()`, it settles atomically.

**Why the `pricePercentageLimit` guard fails.**
The guard only reverts a non-manager caller when `pricePercentageLimit > 0` AND the price increase exceeds the configured threshold: [6](#0-5) 

When `pricePercentageLimit == 0` (no limit set), the guard is entirely bypassed. Even when set, normal LST staking yield accrual (typically 3–5% APR, or ~0.01% per day) will fall within any reasonable daily limit, making the attack viable every update cycle.

**Instant withdrawal liquidity check.**
`instantWithdrawal()` checks that the unstaking vault has sufficient assets: [7](#0-6) 

This is a precondition, not a mitigation — the vault must hold assets for instant withdrawals to be enabled at all, so this check is satisfied in normal operation.

## Impact Explanation

This is **theft of unclaimed yield** (High severity per the allowed impact scope). The profit per attack equals approximately `(updatedPrice − stalePrice) × depositAmount / updatedPrice`. This yield is directly extracted from existing rsETH holders who have not yet had their price updated. The attack is repeatable every time the price becomes stale.

## Likelihood Explanation

- No special privilege is required; any EOA or contract can call `updateRSETHPrice()` and `instantWithdrawal()`.
- `rsETHPrice` naturally becomes stale between keeper calls; the longer the gap, the larger the exploitable discrepancy.
- The only preconditions are: (a) `isInstantWithdrawalEnabled[asset] == true` — the normal operating state when the feature is deployed — and (b) sufficient assets in the unstaking vault, which is required for the feature to function at all.
- The three steps can be executed atomically in a single transaction via a flash-loan-funded attack contract.

Likelihood: **Medium** (feature must be enabled; vault must have liquidity).

## Recommendation

1. **Restrict `updateRSETHPrice()`** to a trusted keeper role (e.g., `onlyLRTManager`) so unprivileged callers cannot trigger price updates.
2. **Alternatively**, compute the rsETH price live (without caching) inside both `getRsETHAmountToMint()` and `getExpectedAssetAmount()` so deposit and withdrawal always use a consistent price within a single transaction.
3. **Ensure `pricePercentageLimit` is always non-zero** to bound the maximum exploitable price jump per update cycle.

## Proof of Concept

```
// Assume rsETHPrice stored = 1.000e18; actual live value = 1.005e18
// isInstantWithdrawalEnabled[stETH] == true
// LRTUnstakingVault holds >= 1005 stETH available for instant withdrawal

// All three calls in a single transaction via attacker contract:

1. LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")
   // rsethAmountToMint = 1000e18 * 1e18 / 1.000e18 = 1000e18 rsETH
   // (attacker should have received ~995e18 rsETH at the correct price)

2. LRTOracle.updateRSETHPrice()
   // rsETHPrice updated to ~1.005e18

3. LRTWithdrawalManager.instantWithdrawal(stETH, 1000e18, "")
   // assetAmountUnlocked = 1000e18 * 1.005e18 / 1e18 = 1005e18 stETH
   // attacker receives 1005 stETH having deposited 1000 stETH
   // profit: ~5 stETH stolen from existing rsETH holders
```

**Foundry fork test plan:** Fork mainnet, deploy attacker contract, seed the unstaking vault with stETH, advance time to allow LST yield to accrue (making `rsETHPrice` stale), execute the three-step sequence atomically, and assert `attackerStETHBalance > initialDeposit`.
