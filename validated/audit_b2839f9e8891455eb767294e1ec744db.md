### Title
Attestation Expiry Window Shorter Than Node Resubmission Cadence Creates Liveness-Breaking Race — (File: crates/mpc-attestation/src/attestation.rs)

### Summary

The `DEFAULT_EXPIRATION_DURATION_SECONDS` constant is set to **1 day (86,400 seconds)**, while the `periodic_attestation_submission` task resubmits every **1 hour** (`ATTESTATION_RESUBMISSION_INTERVAL`). However, the design documentation explicitly states the intended resubmission cadence for the TEE Context is **every 7 days**, and the `tee-lifecycle.md` design doc shows "every 7 days" as the periodic attestation interval. This creates a structural mismatch: if the TEE Context design (7-day resubmission) is deployed against the current 1-day expiry window, honest nodes will have their attestations expire before they resubmit, causing `verify_tee()` to kick them out and halt signing.

More critically, the `expiry_timestamp_seconds` stored in `ValidatedDstackAttestation` is **stamped off-chain** by the node at verification time using `current_timestamp_seconds + DEFAULT_EXPIRATION_DURATION_SECONDS`, with an acknowledged `TODO(#1639)` noting the timestamp cannot come from the contract but should be extracted from the certificate itself. This means the expiry window is derived from a heuristic that can lag real certificate expiry, and the contract enforces it without independently computing it.

### Finding Description

**Root cause — expiry stamped from block timestamp, not certificate:**

In `crates/mpc-attestation/src/attestation.rs`, `AcceptedAttestation::dstack()` computes the expiry as:

```rust
let expiration_timestamp_seconds =
    current_timestamp_seconds + DEFAULT_EXPIRATION_DURATION_SECONDS;
```

where `DEFAULT_EXPIRATION_DURATION_SECONDS = 60 * 60 * 24` (1 day). [1](#0-0) [2](#0-1) 

The `current_timestamp_seconds` passed in is `TeeState::current_time_seconds()` — the NEAR block timestamp at the moment of `submit_participant_info` execution: [3](#0-2) 

The stored `ValidatedDstackAttestation.expiry_timestamp_seconds` is then used by `re_verify` to gate all subsequent signing operations: [4](#0-3) 

**Mismatch between expiry window and resubmission cadence:**

The `mpc-node` production code resubmits every 1 hour: [5](#0-4) 

But the TEE Context design document specifies a **7-day** periodic attestation interval: [6](#0-5) [7](#0-6) 

The `DEFAULT_EXPIRATION_DURATION_SECONDS` is 1 day. If the TEE Context's 7-day cadence is adopted (as the design intends), honest nodes will fail to resubmit within the 1-day window, causing `verify_tee()` to classify them as having expired attestations and trigger resharing that removes them from the participant set.

**The `verify_tee()` consequence:**

When `reverify_and_cleanup_participants` finds expired attestations, it triggers a resharing that removes those participants: [8](#0-7) 

If enough participants are removed to drop below threshold, `accept_requests` is set to `false`, halting all signing: [9](#0-8) 

**The `TODO(#1639)` acknowledged design gap:**

The code itself acknowledges the expiry timestamp cannot be correctly derived from the block timestamp and should come from the certificate: [10](#0-9) 

The design doc confirms: *"the expiration duration, today a hardcoded `DEFAULT_EXPIRATION_DURATION_SECONDS` of 7 days, stamped off-chain at attestation time and carried through the DTO into the stored entry (the contract enforces it, it doesn't compute it)."* — note the doc says 7 days but the code says 1 day, a direct inconsistency. [11](#0-10) 

### Impact Explanation

This is a **Medium** impact finding under the allowed scope: **balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants**.

Concretely:
- If the TEE Context 7-day resubmission cadence is deployed while `DEFAULT_EXPIRATION_DURATION_SECONDS` remains at 1 day, all honest nodes will have their attestations expire within 1 day of their last submission.
- `verify_tee()` (callable by any participant) will then classify all nodes as having expired attestations, trigger resharing, and if the surviving set drops below threshold, permanently halt signing (`accept_requests = false`).
- This breaks the production signing invariant without any attacker needing to compromise keys or collude above threshold.

### Likelihood Explanation

**Medium.** The mismatch between the 1-day code constant and the 7-day design-doc cadence is a documented, acknowledged inconsistency (`TODO(#1639)`). The current `mpc-node` production code uses a 1-hour resubmission interval, which keeps nodes within the 1-day window. However:
1. The TEE Context design (the intended future architecture) specifies 7-day resubmission.
2. Any deployment that adopts the TEE Context's 7-day cadence without simultaneously raising `DEFAULT_EXPIRATION_DURATION_SECONDS` will trigger the failure.
3. Any adversarial participant can call `verify_tee()` at any time to force the expiry check.

### Recommendation

1. **Align the expiry window with the resubmission cadence.** `DEFAULT_EXPIRATION_DURATION_SECONDS` must be strictly greater than the maximum resubmission interval. If the TEE Context adopts a 7-day cadence, the expiry must be at least 8–14 days (the design doc's `auto-remove-launcher-hashes` proposal uses 14 days as the TTL for exactly this reason).
2. **Resolve `TODO(#1639)`.** Extract the actual certificate expiry from the TDX quote's certificate chain rather than computing it as `block_timestamp + constant`. This removes the heuristic entirely and makes the expiry authoritative.
3. **Add a contract-side invariant check** that rejects any submitted attestation whose computed expiry is less than `min_resubmission_interval` seconds in the future, so a misconfigured constant cannot silently create a liveness trap.

### Proof of Concept

1. Deploy the contract with `DEFAULT_EXPIRATION_DURATION_SECONDS = 86400` (current value).
2. Configure nodes to use the TEE Context 7-day resubmission cadence (`Duration::from_secs(7 * 24 * 3600)`).
3. Nodes submit attestations at T=0. Expiry is stamped at T+86400.
4. At T=86401, any participant calls `verify_tee()`.
5. `reverify_and_cleanup_participants` calls `re_verify` on all stored attestations; all fail with `"The attestation expired"`.
6. If the surviving set (0 nodes with valid attestations) is below threshold, `accept_requests = false` is set and all `sign()` calls are rejected.
7. The network is halted with no path to recovery except manual operator intervention to resubmit attestations — which itself requires the nodes to be running and able to reach the contract.
