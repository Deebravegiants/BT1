### Title
Excess Deposit Silently Retained by Contract in `submit_participant_info` When Existing Participant Re-submits Attestation - (File: crates/contract/src/lib.rs)

### Summary
`MpcContract::submit_participant_info` is marked `#[payable]` and accepts any NEAR deposit, but its deposit-handling block is gated behind a condition that evaluates to `false` for existing participants performing an attestation update. When that condition is false, any attached deposit is never checked and never refunded — it is silently absorbed by the contract. This is the direct analog of the external report's pattern: a payable entry point that accepts excess payment but fails to return it to the caller under a specific, reachable execution branch.

### Finding Description
`submit_participant_info` computes a boolean guard:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... storage cost check + excess refund
}
``` [1](#0-0) 

`is_new_attestation` is `true` only when `add_participant` returns `ParticipantInsertion::NewlyInsertedParticipant` — i.e., the TLS key was not previously in `stored_attestations`. [2](#0-1) 

`caller_is_not_participant` is `true` only when `self.voter_account().is_err()` — i.e., the caller is not a current governance participant. [3](#0-2) 

`add_participant` in `tee_state.rs` returns `UpdatedExistingParticipant` whenever the TLS key already exists in `stored_attestations`, which is the normal case for a periodic attestation refresh by an active node: [4](#0-3) 

When both conditions are false — the caller is an existing governance participant AND the TLS key is already registered — `attestation_storage_must_be_paid_by_caller` is `false`, the entire `if` block is skipped, and `env::attached_deposit()` is never read, checked, or refunded. The function returns `Ok(())` and the contract retains whatever NEAR was attached.

Contrast this with the correct pattern used in `require_deposit` (called by `sign`, `request_app_private_key`, `verify_foreign_transaction`) and in `propose_update`, both of which unconditionally refund any excess above the required minimum: [5](#0-4) [6](#0-5) 

### Impact Explanation
Any NEAR attached to a re-submission call by an existing participant is permanently locked in the contract with no recovery path. The contract's own README states "Any excess deposit is automatically refunded," but this invariant is violated for the update branch of `submit_participant_info`. This breaks the production accounting invariant that callers receive back any deposit above the required cost. [7](#0-6) 

### Likelihood Explanation
The operational guide explicitly instructs operators to call `submit_participant_info` manually with `--deposit` for first-time joins, and nodes re-submit periodically to refresh expiring attestations. A human operator following the documented pattern for a re-submission, or a node configured with a non-zero deposit for safety, will silently lose the attached NEAR on every update call. The re-submission path is confirmed reachable by the existing unit test `updating_existing_participant_returns_existing_participant`, which shows `UpdatedExistingParticipant` is the normal return value for a refresh. [8](#0-7) 

### Recommendation
Move the excess-refund logic outside the `if attestation_storage_must_be_paid_by_caller` guard, or add an unconditional refund of any deposit above zero when `attestation_storage_must_be_paid_by_caller` is false. The simplest fix mirrors the pattern already used in `require_deposit`:

```rust
// After the if block:
if !attestation_storage_must_be_paid_by_caller {
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, restructure so that the refund of any excess is always performed regardless of which branch is taken.

### Proof of Concept
1. Deploy the contract with two participants (threshold 2-of-2).
2. Participant A calls `submit_participant_info` for the first time — `NewlyInsertedParticipant` is returned, deposit is charged and excess refunded correctly.
3. Participant A calls `submit_participant_info` again (attestation refresh) and attaches 1 NEAR.
   - `is_new_attestation = false` (`UpdatedExistingParticipant` returned by `add_participant`)
   - `caller_is_not_participant = false` (A is a current voter)
   - `attestation_storage_must_be_paid_by_caller = false`
   - The `if` block is skipped; `env::attached_deposit()` is never read.
4. The call returns `Ok(())`. Participant A's account is debited 1 NEAR; the contract balance increases by 1 NEAR. No refund promise is scheduled. [9](#0-8)
