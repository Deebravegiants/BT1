### Title
Missing Payload-Hash Binding in `respond_verify_foreign_tx` Enables Single-Leader Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over `response.payload_hash`, but never checks that `response.payload_hash` was actually derived from the `request` field supplied in the same call. A single malicious leader node can take a legitimately-produced threshold signature for pending request A and use it to resolve a different pending request B, delivering a forged attestation to the caller of request B.

### Finding Description

In `respond` (the regular signing path), the payload that is verified comes from the on-chain pending request itself:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,          // ← derived from the stored request, not from the response
    &expected_public_key,
)
``` [1](#0-0) 

In `respond_verify_foreign_tx`, the payload hash is taken directly from the node-supplied response instead:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← fully attacker-controlled

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [2](#0-1) 

The contract never reconstructs `SHA-256(borsh(ForeignTxSignPayload { request, values }))` from the `request` argument and compares it to `response.payload_hash`. The SDK helper `ForeignChainSignatureVerifier::verify_signature` does perform this check, but it lives in the off-chain SDK, not in the on-chain contract:

```rust
let expected_payload_hash = expected_payload.compute_msg_hash()...;
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [3](#0-2) 

The node-side signing path uses a zero tweak for foreign-tx requests, meaning the threshold signing is performed under the root key:

```rust
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),   // zero tweak → root key
    ...
})
``` [4](#0-3) 

With a zero tweak, `derive_key_secp256k1` returns the root public key (GENERATOR × 0 + PK = PK), which is exactly what `respond_verify_foreign_tx` verifies against. Any valid threshold signature produced for any foreign-tx request is therefore a valid signature under the same root key and will pass the contract's check regardless of which `request` is named in the call. [5](#0-4) 

### Impact Explanation

A single malicious leader node can execute the following cross-request replay:

1. Two foreign-tx verification requests are pending simultaneously: request A (e.g., "Bitcoin tx X confirmed") and request B (e.g., "Bitcoin tx Y confirmed").
2. The malicious leader legitimately drives the threshold signing protocol for request A, obtaining a valid root-key signature `sig_A` over `hash_A = SHA-256(borsh(ForeignTxSignPayload{request_A, values_A}))`.
3. The leader calls `respond_verify_foreign_tx(request_B, { payload_hash: hash_A, signature: sig_A })`.
4. The contract checks: is `sig_A` a valid ECDSA signature over `hash_A` under the root key? Yes — and it accepts.
5. The pending yield for request B is resolved with `{ payload_hash: hash_A, signature: sig_A }`.
6. The bridge contract waiting on request B receives an attestation that claims a different transaction (or different extracted values) was verified.

Any bridge contract that does not independently recompute the expected `payload_hash` from the transaction it requested will accept this forged attestation, enabling false bridge execution or double-spend conditions. This matches the **High** impact category: forged foreign-chain verification causing invalid bridge execution. [6](#0-5) 

### Likelihood Explanation

- Requires only **one** malicious attested participant acting as the signing leader for a request. Leader election is deterministic (lowest participant ID for a given request), so a malicious participant can wait for or engineer a situation where they lead a request.
- Requires two foreign-tx verification requests to be pending simultaneously — a normal operational condition for any active bridge.
- The followers participate in the threshold signing for request A without knowing the leader will replay the result against request B; they sign a legitimate hash and are not complicit.
- No threshold collusion is required.

### Recommendation

Inside `respond_verify_foreign_tx`, reconstruct the expected payload hash from the `request` argument and compare it to `response.payload_hash` before accepting the response. Because the contract does not know the extracted values, the simplest fix is to store the expected `payload_hash` in the pending request at submission time (computed by the node that initiates the signing) and verify it on respond, or alternatively require the response to include the full `ForeignTxSignPayload` so the contract can hash it itself. At minimum, mirror the check already present in `ForeignChainSignatureVerifier::verify_signature` in the SDK. [7](#0-6) 

### Proof of Concept

```
1. Alice calls verify_foreign_transaction({ request: Bitcoin{tx_id: X, ...}, domain_id: D })
   → pending_verify_foreign_tx_requests[request_A] = [yield_A]

2. Bob calls verify_foreign_transaction({ request: Bitcoin{tx_id: Y, ...}, domain_id: D })
   → pending_verify_foreign_tx_requests[request_B] = [yield_B]

3. Malicious leader drives threshold signing for request_A:
   payload_A = ForeignTxSignPayload::V1 { request: request_A, values: [BlockHash(real_hash_X)] }
   hash_A    = SHA-256(borsh(payload_A))
   sig_A     = threshold_sign(hash_A, tweak=0)   // valid root-key signature

4. Malicious leader calls:
   respond_verify_foreign_tx(
       request  = request_B,                          // Bob's request
       response = { payload_hash: hash_A, sig: sig_A } // Alice's hash/sig
   )

5. Contract check:
   verify_ecdsa(sig_A, hash_A, root_pk) → OK   ✓
   resolve_yields_for(request_B, response)      ✓  (no hash-to-request binding check)

6. Bob's bridge contract receives { payload_hash: hash_A, sig: sig_A }
   → attests that Bitcoin tx Y was verified, but the signed payload actually describes tx X.
   → If the bridge contract does not recompute the expected hash, it accepts the forged attestation.
``` [8](#0-7) [9](#0-8)

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }

pub fn derive_key_secp256k1(
    public_key: &k256_types::PublicKey,
    tweak: &Tweak,
) -> Result<dtos::Secp256k1PublicKey, TweakNotOnCurve> {
    let tweak = k256::Scalar::from_repr(tweak.as_bytes().into())
        .into_option()
        .ok_or(TweakNotOnCurve)?;

    let derived = (<Secp256k1 as CurveArithmetic>::ProjectivePoint::GENERATOR * tweak + public_key)
        .to_affine();
    let pk = k256::PublicKey::try_from(derived).map_err(|_| TweakNotOnCurve)?;
    Ok(dtos::Secp256k1PublicKey::from(&pk))
}
