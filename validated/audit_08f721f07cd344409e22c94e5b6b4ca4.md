### Title
`PubkeyRegistry.addPubkeys` can permanently block `stake32Eth` for any pre-registered pubkey â€” (`contracts/PubkeyRegistry.sol`)

### Summary
The `LRT_MANAGER`-only `addPubkeys` function registers pubkeys in the `pubkeyRegistry` mapping without verifying that those pubkeys correspond to validators already staked on the beacon chain. Because `stake32Eth` reverts on any pubkey already present in the registry, and because there is no removal function, any pubkey pre-registered via `addPubkeys` is permanently barred from the staking path.

### Finding Description
`PubkeyRegistry.addPubkeys` writes `pubkeyRegistry[keccak256(pubkeys[i])] = true` for every supplied key. [1](#0-0) 

`NodeDelegator.stake32Eth` reads that same mapping and reverts immediately if the key is already set, before any ETH is moved or `stakedButUnverifiedNativeETH` is incremented. [2](#0-1) 

There is no `removePubkey`, `resetPubkey`, or equivalent function anywhere in `PubkeyRegistry`, making the block permanent. [3](#0-2) 

### Impact Explanation
ETH already transferred to the NDC (via `sendETHFromDepositPoolToNDC`) for the purpose of staking those validators cannot be staked into EigenLayer. It sits idle in the NDC, earning no restaking yield. The funds are not lost (they can be transferred back), but the protocol's core promise â€” that ETH held in an NDC will be restaked â€” is broken for those pubkeys. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
`addPubkeys` has a plausible legitimate use (bulk migration of pre-existing validators). A manager running a migration script that accidentally includes a pubkey not yet deposited to the beacon chain, or that pre-registers pubkeys for validators planned but not yet staked, triggers the block. No malicious intent is required; operational error is sufficient.

### Recommendation
1. Add a guard in `addPubkeys` that verifies each pubkey already has a corresponding deposit on the beacon chain (e.g., via the `IETHPOSDeposit.get_deposit_count` or an off-chain proof), **or**
2. Add a privileged `removePubkey` / `resetPubkey` function so the manager can correct accidental registrations, **or**
3. Remove `addPubkeys` entirely if its only purpose is migration, and handle migration through a one-time initializer with stricter guards.
