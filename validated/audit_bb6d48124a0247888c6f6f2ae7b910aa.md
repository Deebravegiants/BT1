Audit Report

## Title
Missing Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Inflated rsETH Minting During LST Depeg Events - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp and `answeredInRound` value, performing no staleness or round-completeness check. When a supported LST (stETH, rETH, ETHx, swETH) depegs on the market while the Chainlink feed lags, any unprivileged depositor can call `LRTDepositPool.depositAsset()` and receive rsETH calculated at the stale inflated oracle price, depositing collateral worth less than the rsETH minted. This dilutes all existing rsETH holders and drives the protocol toward insolvency.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at L49–55 reads the Chainlink feed as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The `updatedAt`, `answeredInRound`, and `roundId` return values are entirely discarded. No heartbeat check, no round-completeness check, and no non-negative price check are performed. [1](#0-0) 

This oracle is consumed by `LRTOracle.getAssetPrice()` (L156–158), which delegates directly to the registered `IPriceFetcher`: [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` (L506–521) uses this price to compute the rsETH amount to mint:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

`rsETHPrice` is a cached storage value updated only when `updateRSETHPrice()` is called. The minting ratio is therefore entirely dependent on the Chainlink feed being current.

`depositAsset()` is fully permissionless — guarded only by `whenNotPaused` and `onlySupportedERC20Token`: [4](#0-3) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L26–37) — used for the pool collateral path — does perform the round-completeness and incomplete-round checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [5](#0-4) 

This demonstrates the protocol is aware of the staleness-check pattern but failed to apply it to the primary deposit path.

The stale price also propagates into `_getTotalEthInProtocol()` (L331–349), which calls `getAssetPrice()` for every supported asset when computing the rsETH price update, meaning the inflated price contaminates the TVL calculation as well: [6](#0-5) 

## Impact Explanation

**Protocol insolvency (Critical).**

When stETH trades at 0.95 ETH on the market while the Chainlink feed still reports 1.00 ETH:

1. An attacker deposits 10,000 stETH (market value: 9,500 ETH) via `depositAsset()`.
2. `getRsETHAmountToMint()` prices stETH at 1.00 ETH per the stale feed; with `rsETHPrice ≈ 1.02e18`, the attacker receives ≈ 9,804 rsETH representing a ≈ 10,000 ETH claim.
3. True ETH value deposited: 9,500 ETH. The 500 ETH shortfall is borne by all existing rsETH holders.
4. When `updateRSETHPrice()` is eventually called with the correct stETH price, `_getTotalEthInProtocol()` computes a lower TVL, `rsETHPrice` drops, and all existing holders are diluted.
5. The downside-protection pause in `_updateRsETHPrice()` (L270–282) may trigger if the price drop exceeds `pricePercentageLimit`, but this does not undo the already-minted rsETH or recover the shortfall. [7](#0-6) 

Repeated at scale, the protocol's backing collapses below 1:1, making rsETH insolvent.

## Likelihood Explanation

LST depeg events are not hypothetical — stETH traded at a significant discount to ETH during the 2022 Celsius/3AC crisis. Chainlink feeds for LSTs have heartbeats of up to 24 hours (or update only on 0.5% deviation). During a rapid depeg, the feed can lag the market by minutes to hours. Any MEV bot or informed trader monitoring the spread between the Chainlink feed and DEX spot price can profitably exploit this window. The entry path (`depositAsset`) is fully permissionless with no access control, no deposit fee, and no slippage protection beyond the caller-supplied `minRSETHAmountExpected`.

## Recommendation

Add staleness, round-completeness, and non-negative price checks in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt != 0, "Incomplete round");
    require(block.timestamp - updatedAt <= maxStaleness[asset], "Price too old");
    require(price > 0, "Invalid price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`maxStaleness` should be configured per-asset based on the Chainlink feed's documented heartbeat (e.g., 3,600 seconds for a 1-hour heartbeat feed, 86,400 seconds for a 24-hour heartbeat feed).

## Proof of Concept

**Foundry fork test outline:**

```solidity
// Fork mainnet at a block where stETH/ETH Chainlink feed is near its heartbeat boundary
// Warp block.timestamp forward past the feed's heartbeat without a new round
// Confirm priceFeed.latestRoundData() returns an updatedAt > MAX_STALENESS seconds ago
// Attacker calls LRTDepositPool.depositAsset(stETH, 10_000e18, 0, "")
// Assert rsETH minted > (9_500e18 * 1e18) / rsETHPrice  (i.e., minted at inflated price)
// Call LRTOracle.updateRSETHPrice() with the now-stale feed still active
// Assert rsETHPrice has decreased, confirming dilution of existing holders
```

Minimal reproducible steps (local call sequence):

1. Deploy or fork with `ChainlinkPriceOracle` pointing to a mock Chainlink feed returning `updatedAt = block.timestamp - 25 hours` (past 24-hour heartbeat).
2. Call `LRTDepositPool.depositAsset(stETH, 10_000e18, 0, "")` as an unprivileged attacker holding stETH.
3. Observe rsETH minted is computed at the stale 1.00 ETH/stETH price rather than the market 0.95 ETH/stETH price.
4. Call `LRTOracle.updateRSETHPrice()`.
5. Observe `rsETHPrice` decreases, confirming existing holders are diluted by the shortfall.
