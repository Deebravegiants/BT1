### Title
Tautological Lower-Bound Check in `canUpdate` Allows Script Version Downgrade via Unsigned Arithmetic Wrap - (File: `eras/byron/ledger/impl/src/Cardano/Chain/Update/Validation/Registration.hs`)

---

### Summary

In `canUpdate`, the script-version constraint `(0 <= scriptVersionDiff && scriptVersionDiff <= 1)` contains a tautological sub-expression. `scriptVersionDiff` is computed by subtracting two `Word16` values. Because `Word16` is an unsigned type in Haskell, the expression `0 <= scriptVersionDiff` is always `True` regardless of the operands. This is the direct analog of the reported `config.version >= config.version` pattern.

---

### Finding Description

`ppScriptVersion` is declared as `!Word16` in `ProtocolParameters`: [1](#0-0) 

In `canUpdate`, the version-change guard is:

```haskell
(0 <= scriptVersionDiff && scriptVersionDiff <= 1)
  `orThrowError` InvalidScriptVersion
    adoptedScriptVersion
    newScriptVersion
where
  adoptedScriptVersion = ppScriptVersion adoptedPP   -- Word16
  newScriptVersion     = ppScriptVersion proposedPP  -- Word16
  scriptVersionDiff    = newScriptVersion - adoptedScriptVersion
``` [2](#0-1) 

Because `Word16` is unsigned, `scriptVersionDiff` is always `>= 0`; the left half of the conjunction is a tautology and contributes nothing to the guard. The intent — to reject proposals that lower the script version — is only partially enforced by the right half (`scriptVersionDiff <= 1`).

The right half catches most decreases because they wrap to large values (e.g., `adoptedScriptVersion = 5`, `newScriptVersion = 4` → `scriptVersionDiff = 65535`, which fails `<= 1`). However, when `adoptedScriptVersion = 65535` and `newScriptVersion = 0`, the subtraction wraps to `1` (`0 - 65535 mod 2^16 = 1`), which satisfies `scriptVersionDiff <= 1`, and the check passes — allowing a script-version downgrade from `65535` to `0`.

The executable-spec counterpart in `Byron.Spec.Ledger.Update` uses `Integer` arithmetic and correctly enforces `0 <= scriptVersionDiff`: [3](#0-2) 

The implementation diverges from the spec at this point.

---

### Impact Explanation

A proposer who controls a node with `adoptedScriptVersion = 65535` could submit a protocol update proposal setting `newScriptVersion = 0`. The `canUpdate` check would pass (wrap-around yields `scriptVersionDiff = 1`), and the proposal could be voted through and enacted. This constitutes an invalid ledger state transition: the enacted protocol parameters would violate the invariant that the script version never decreases, potentially breaking script-validation assumptions for any scripts that depend on the minimum script version.

Mapped impact: **Medium** — attacker-controlled proposal modifies a protocol parameter (script version) outside design parameters.

---

### Likelihood Explanation

**Low.** The exploit requires `ppScriptVersion` to reach `65535` (the `Word16` maximum). Script versions increment by at most 1 per protocol update, so reaching `65535` would require 65,535 successful protocol updates — far beyond any realistic deployment timeline. The Byron era is also a legacy component no longer active on mainnet. The tautological check is a real code defect matching the reported vulnerability class, but practical exploitation is negligible.

---

### Recommendation

Replace the unsigned subtraction with a signed comparison to match the executable spec:

```haskell
-- Instead of:
scriptVersionDiff = newScriptVersion - adoptedScriptVersion  -- Word16 underflow
(0 <= scriptVersionDiff && scriptVersionDiff <= 1) ...

-- Use:
let sv_diff = (fromIntegral newScriptVersion :: Int)
            - (fromIntegral adoptedScriptVersion :: Int)
(0 <= sv_diff && sv_diff <= 1) ...
```

This matches the executable-spec definition at `Byron.Spec.Ledger.Update.checkUpdateConstraints` which uses `(<=?)` over `Natural`/`Integer` arithmetic.

---

### Proof of Concept

1. Assume current `adoptedScriptVersion = 65535` (max `Word16`).
2. Craft a `UProp` with `newScriptVersion = 0`.
3. `scriptVersionDiff = (0 :: Word16) - (65535 :: Word16) = 1` (modular wrap).
4. `(0 <= 1 && 1 <= 1)` evaluates to `True`.
5. `canUpdate` does **not** throw `InvalidScriptVersion`.
6. The proposal is registered via `registerProtocolUpdate` and, upon sufficient endorsement, enacted — setting `ppScriptVersion` to `0` despite the current version being `65535`. [4](#0-3)
