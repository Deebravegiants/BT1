Audit Report

## Title
Unpermissioned `sendFunds()` in `FeeReceiver` Allows Anyone to Inject ETH as Protocol Rewards, Triggering Unauthorized Fee Minting and rsETH Holder Dilution - (File: `contracts/FeeReceiver.sol`)

## Summary

`FeeReceiver.sendFunds()` has no access control modifier, allowing any caller to flush arbitrary ETH into `LRTDepositPool` as if it were legitimate MEV/execution-layer rewards. Because `LRTDepositPool.getETHDistributionData()` counts `address(this).balance` verbatim, the injected ETH inflates the protocol's reported TVL. When `LRTOracle.updateRSETHPrice()` is subsequently called, the oracle treats the TVL increase as organic yield, computes a protocol fee on it, and mints that fee as rsETH to the treasury — diluting every existing rsETH holder's share of the underlying ETH.

## Finding Description

**Root cause — missing access control on `sendFunds()`:**

`contracts/FeeReceiver.sol` lines 49–58 confirm the vulnerability exactly as described:

```solidity
receive() external payable { }

function sendFunds() external {          // ← no onlyRole guard
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Every other fund-movement function in the contract (e.g., `setDepositPool`) carries `onlyRole(LRTConstants.MANAGER)`, but `sendFunds()` does not.

**Step 1 — ETH lands in `LRTDepositPool`:**

`receiveFromRewardReceiver()` in `contracts/LRTDepositPool.sol` line 61 is a plain `external payable` with no access control, so the ETH is accepted unconditionally. `getETHDistributionData()` at line 480 then counts it:

```solidity
ethLyingInDepositPool = address(this).balance;
```

**Step 2 — Oracle treats injected ETH as organic yield:**

`_updateRsETHPrice()` in `contracts/LRTOracle.sol` lines 234 and 244–246 computes `previousTVL` from the stored price and supply, then treats any excess as a reward:

```solidity
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);
...
if (!protocolPaused && totalETHInProtocol > previousTVL) {
    uint256 rewardAmount = totalETHInProtocol - previousTVL;
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}
```

The injected ETH is indistinguishable from real staking yield.

**Step 3 — Fee is minted as rsETH to treasury:**

Lines 299–307 of `contracts/LRTOracle.sol`:

```solidity
uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
if (rsethAmountToMintAsProtocolFee > 0) {
    IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee);
}
```

**Step 4 — Secondary: oracle freeze if `pricePercentageLimit` is set:**

Lines 252–265 of `contracts/LRTOracle.sol` revert with `PriceAboveDailyThreshold` for any non-manager caller if the injected ETH pushes `newRsETHPrice` above `highestRsethPrice * (1 + pricePercentageLimit)`, blocking all public oracle updates until a manager intervenes.

## Impact Explanation

**Primary — High: Theft of unclaimed yield.**
Existing rsETH holders are entitled to the full benefit of ETH entering the protocol. By routing ETH through `FeeReceiver.sendFunds()`, the attacker forces the fee mechanism to treat the donation as organic yield and divert `protocolFeeInBPS / 10_000` of it to the treasury as freshly minted rsETH. This dilutes every existing holder: at 10% fee, 100 ETH injected causes 10 ETH worth of rsETH to be minted to the treasury at holders' expense, reducing the ETH-per-rsETH ratio they would otherwise have received.

**Secondary — Medium: Temporary freezing of funds.**
If `pricePercentageLimit` is configured, a sufficiently large injection causes `updateRSETHPrice()` to revert for all non-manager callers, freezing oracle updates and blocking deposit/withdrawal price calculations until a manager calls `updateRSETHPriceAsManager()`.

## Likelihood Explanation

The attack requires only ETH and two public function calls — no special role, no governance capture, no front-running dependency. The attacker's ETH is permanently locked in the protocol (it becomes part of TVL), but the damage to holders scales with `protocolFeeInBPS`. The attack is economically rational for a competitor or griever willing to spend ETH to harm the protocol's reputation, exhaust the daily fee mint cap, or repeatedly dilute holders. It is repeatable as often as the attacker is willing to fund it.

## Recommendation

Add `onlyRole(LRTConstants.MANAGER)` to `sendFunds()`, consistent with the pattern used by `setDepositPool()` and other privileged functions in the same contract:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Optionally, restrict `receive()` to accept ETH only from known node operators or the EigenLayer withdrawal path.

## Proof of Concept

```
Setup: protocol has 1000 ETH TVL, 1000 rsETH supply, price = 1.0 ETH/rsETH,
       protocolFeeInBPS = 1000 (10%), pricePercentageLimit = 0.

1. Attacker sends 100 ETH to FeeReceiver (via receive()).
2. Attacker calls FeeReceiver.sendFunds() — no role check, succeeds.
   → LRTDepositPool.receiveFromRewardReceiver{value: 100 ether}() called.
   → LRTDepositPool.balance = 1100 ETH.
3. Anyone calls LRTOracle.updateRSETHPrice().
   → totalETHInProtocol = 1100 ETH.
   → previousTVL = 1000 rsETH * 1.0 = 1000 ETH.
   → rewardAmount = 100 ETH.
   → protocolFeeInETH = 100 * 1000 / 10_000 = 10 ETH.
   → newRsETHPrice = (1100 - 10) / 1000 = 1.09 ETH/rsETH.
   → rsethAmountToMintAsProtocolFee = 10 / 1.09 ≈ 9.17 rsETH minted to treasury.
4. Existing holders now hold rsETH worth 1.09 ETH each instead of 1.10 ETH each.
   The 0.01 ETH/rsETH difference (≈ 10 ETH total) was diverted to the treasury.

Foundry fork test: deploy FeeReceiver + LRTDepositPool + LRTOracle on a mainnet
fork, call sendFunds() from address(0xdead), then updateRSETHPrice(), and assert
that treasury rsETH balance increased and existing holder ETH-per-rsETH is lower
than it would be without the fee path.
