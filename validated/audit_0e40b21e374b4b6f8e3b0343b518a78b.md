Audit Report

## Title
Mid-Epoch `totalKernelStaked` Drop to Zero Permanently Strands Reward Tokens — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool` uses the Synthetix staking pattern where `rewardPerToken()` returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`, but the `updateReward` modifier still advances `updatedAt`. If all stakers call `initiateWithdrawal` mid-epoch — a normal, unprivileged action — `totalKernelStaked` immediately drops to zero, and `rewardRate × elapsed_zero_staked_seconds` worth of reward tokens become permanently unclaimable with no on-chain recovery path.

## Finding Description

**Root cause — `rewardPerToken()` silently skips accrual when `totalKernelStaked == 0`:** [1](#0-0) 

When `totalKernelStaked == 0`, the function returns `rewardPerTokenStored` unchanged. However, the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()`: [2](#0-1) 

The clock advances, consuming the elapsed seconds, but the corresponding `rewardRate × Δt` is never credited to `rewardPerTokenStored`. Those rewards are permanently stranded.

**`initiateWithdrawal` immediately decrements `totalKernelStaked` at initiation, not at claim time:** [3](#0-2) 

If all current stakers call `initiateWithdrawal` during an active reward period, `totalKernelStaked` drops to zero immediately, triggering the silent-skip path above for all subsequent time until a new staker arrives.

**The `notifyRewardAmount` guard is insufficient — it only prevents starting an epoch with zero stakers, not reaching zero mid-epoch:** [4](#0-3) 

**No on-chain recovery path exists.** The last admin function is `setMaxNumberOfWithdrawalsPerUser` at line 610; there is no `recoverERC20` or equivalent. [5](#0-4) 

**The contract's own NatSpec acknowledges this limitation but only as an off-chain operational commitment:** [6](#0-5) 

The comment states the team will "ensure there are always some tokens staked... for the entire duration of the reward period," but this is not enforced on-chain.

## Impact Explanation

Rewards for the zero-staked window — `rewardRate × N_zero_seconds` — are permanently locked in the contract. No staker can ever claim them, and no admin function can recover them. The contract fails to deliver the promised reward distribution for that window. Reward tokens are not lost from the system (the contract holds them), but they are permanently unclaimable.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation

Any combination of normal user actions that drives `totalKernelStaked` to zero mid-epoch triggers this. No malicious intent is required — all current stakers independently deciding to exit during a reward period is sufficient. The `withdrawalDelay` does not mitigate this because `totalKernelStaked` is decremented at `initiateWithdrawal` time, not at `claimWithdrawal` time. The scenario is repeatable across every reward epoch.

## Recommendation

1. **Add a recovery function** so the admin can reclaim stranded reward tokens after an epoch ends:
   ```solidity
   function recoverStrandedRewards() external onlyRole(DEFAULT_ADMIN_ROLE) {
       require(block.timestamp > finishAt, "Epoch not finished");
       uint256 stranded = rewardsToken.balanceOf(address(this)) - totalAccruedButUnclaimed();
       rewardsToken.safeTransfer(msg.sender, stranded);
   }
   ```
2. **Or enforce the invariant on-chain** by reverting `initiateWithdrawal` if it would drop `totalKernelStaked` to zero during an active reward period (`block.timestamp < finishAt`).
3. **Or carry forward** unallocated rewards into the next epoch inside `notifyRewardAmount` by comparing `rewardsToken.balanceOf(address(this))` against the expected distributed amount.

## Proof of Concept

```solidity
// 1. Admin calls notifyRewardAmount(1000e18) with Alice staking 100 KERNEL
//    rewardRate = 1000e18 / duration

// 2. Alice calls initiateWithdrawal(100) → totalKernelStaked = 0
//    updateReward runs: rewardPerToken() returns rewardPerTokenStored (no change)
//    updatedAt advances to block.timestamp

// 3. Warp N seconds (totalKernelStaked == 0 the whole time)
//    Any call to rewardPerToken() returns rewardPerTokenStored unchanged
//    updatedAt advances on each updateReward call, consuming N seconds

// 4. Bob stakes 100 KERNEL → totalKernelStaked = 100
//    updateReward runs: rewardPerToken() still returns rewardPerTokenStored
//    (updatedAt was already advanced past the zero window)

// 5. Warp to finishAt

// 6. Bob calls getReward() → receives rewardRate * (duration - N) / 100 * 100
//    Alice calls getReward() → receives 0 (balance was 0 after step 2)

// Total claimed = rewardRate * (duration - N)
// Total deposited = rewardRate * duration
// Stranded = rewardRate * N  ← permanently locked, no recovery path
```

Foundry invariant test: assert that `rewardsToken.balanceOf(address(pool))` equals the sum of all `rewards[user]` for all users at any point after `finishAt`. The invariant will break whenever a zero-staked window occurs mid-epoch.
