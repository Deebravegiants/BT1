Audit Report

## Title
Missing Slippage Protection in `removeLiquidity` with No Wrapper Escape Hatch - (File: metric-core/contracts/MetricOmmPool.sol, metric-core/contracts/libraries/LiquidityLib.sol)

## Summary

`MetricOmmPool.removeLiquidity` accepts no minimum-output parameters and computes token amounts purely from live bin state at execution time. Because `msg.sender` must equal `owner`, LPs cannot route through any wrapper contract to add slippage protection. A front-running swap that shifts the bin cursor into the LP's bin before the removal settles causes the LP to receive a materially different token composition with no on-chain recourse.

## Finding Description

`LiquidityLib.removeLiquidity` computes output amounts directly from live `binState` at execution time with no floor check:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [1](#0-0) 

These amounts are transferred directly to `owner` with no minimum-output guard: [2](#0-1) 

The pool-level entry point accepts no slippage parameters: [3](#0-2) 

Critically, `msg.sender != owner` reverts with `NotPositionOwner`, which structurally prevents any periphery wrapper from adding slippage protection — the LP must call the pool directly: [4](#0-3) 

`MetricOmmPoolLiquidityAdder` exclusively wraps `addLiquidity` with `maxAmountToken0`/`maxAmountToken1` caps and provides no `removeLiquidity` path: [5](#0-4) 

By contrast, `MetricOmmSimpleRouter` enforces `amountOutMinimum` on every swap variant: [6](#0-5) 

The extension system (`IMetricOmmExtensions.afterRemoveLiquidity`) receives `amount0Removed`/`amount1Removed` but is a pool-deployment-time construct controlled by the pool creator, not a per-call user-supplied guard: [7](#0-6) 

**Exploit path:**
1. LP holds shares in bin 1 (above cursor → 100% token0). LP submits `removeLiquidity`.
2. Attacker front-runs with `MetricOmmSimpleRouter.exactInputSingle` (token1→token0), driving the cursor into bin 1. Bin 1 now holds mixed token0/token1.
3. LP's removal executes against the modified bin state, receiving a token1 position they did not intend to hold.
4. No revert occurs; no minimum-output check exists anywhere in the call path.

## Impact Explanation

The LP receives a different token composition than submitted for. Because the `msg.sender == owner` constraint eliminates any wrapper-based mitigation, the LP has no on-chain mechanism to revert if output falls below an acceptable threshold. When the oracle price is stale relative to market at execution time, the token1 received is worth less than the displaced token0, producing direct principal loss. Even without oracle staleness, the LP is forced into an additional swap (with fees and spread) to rebalance, constituting a loss of funds above Sherlock Medium thresholds. The attack requires no special privilege — any address can call `swap`.

## Likelihood Explanation

MEV front-running of pending liquidity-removal transactions is a well-established attack on public mempools. Any LP removing liquidity from a bin near the active cursor is a target. The attacker needs only to submit a swap that moves the cursor into the victim's bin before the removal settles. No special privilege is required; `swap` is callable by any unprivileged address. The attack is repeatable and profitable whenever the oracle spread or staleness creates a favorable arbitrage.

## Recommendation

Add `uint256 minAmount0Out` and `uint256 minAmount1Out` parameters to `removeLiquidity` and revert if computed outputs fall below those floors:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Out,
    uint256 minAmount1Out,
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) returns (uint256 amount0Removed, uint256 amount1Removed) {
    // ... existing logic ...
    if (amount0Removed < minAmount0Out || amount1Removed < minAmount1Out) {
        revert InsufficientOutput(amount0Removed, amount1Removed, minAmount0Out, minAmount1Out);
    }
}
```

Alternatively, relax the `msg.sender == owner` constraint to allow a trusted periphery wrapper (mirroring the `addLiquidity` operator pattern) and add a `removeLiquidity` path to `MetricOmmPoolLiquidityAdder` that enforces minimum output amounts after the pool call returns.

## Proof of Concept

**Setup**: Pool with bins −1, 0, 1. Oracle mid-price places cursor at bin 0. LP holds 10,000 shares in bin 1 (above cursor → 100% token0, 0 token1).

**Step 1**: LP calls `pool.removeLiquidity(owner, salt, deltas, "")` expecting ~X token0 and 0 token1.

**Step 2**: Attacker front-runs with `MetricOmmSimpleRouter.exactInputSingle` (token1→token0, large amount). Swap exhausts bin 0 and partially fills bin 1, moving cursor into bin 1. Bin 1 now holds ~60% token0 and ~40% token1.

**Step 3**: LP's removal executes:
```solidity
amount0Scaled = binState.token0BalanceScaled * 10_000 / binTotalSharesVal  // ~60% of original
amount1Scaled = binState.token1BalanceScaled * 10_000 / binTotalSharesVal  // ~40% of original
```

LP receives 60% of expected token0 plus an unwanted token1 position. No revert occurs. The LP must execute an additional swap to rebalance, paying fees and spread, with direct principal loss if the oracle is stale.
