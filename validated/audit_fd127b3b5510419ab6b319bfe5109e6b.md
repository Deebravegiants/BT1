### Title
Unvalidated `ProposalInit.builder` Field Allows Malicious Proposer to Redirect All Block Fees to an Arbitrary Address — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`ProposalInit.builder` is a proposer-supplied field that is passed verbatim into `BlockInfo.sequencer_address` and used as the fee-transfer recipient for every transaction in the block. `is_proposal_init_valid` never checks this field. A malicious proposer can set `builder` to any address, causing all transaction fees for the block to be credited to an attacker-controlled account, and causing the committed block hash and state-sync record to embed the wrong sequencer address.

---

### Finding Description

`ProposalInit` carries a `builder` field described as "Address of the one who builds/sequences the block." [1](#0-0) 

During proposal building, the proposer sets this field from its own local configuration: [2](#0-1) 

During proposal validation, `is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, all four L1 gas-price fields, and `fee_proposal_fri`. It never checks `builder`: [3](#0-2) 

The `proposer` field is separately validated in the consensus manager against the committee: [4](#0-3) 

But `builder` has no equivalent guard anywhere in the codebase.

`convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address` in the `BlockInfo` that is handed to the batcher for both proposal building and validation: [5](#0-4) 

The batcher uses this `BlockInfo` to construct a `BlockContext`. Inside `execute_fee_transfer`, the sequencer address is the **recipient** of every fee transfer: [6](#0-5) 

The same `builder` value is also embedded in `PartialBlockHashComponents.sequencer`, which feeds `calculate_block_hash`: [7](#0-6) [8](#0-7) 

And it is stored in the state-sync record as the block's sequencer: [9](#0-8) 

---

### Impact Explanation

Because the validator re-executes the block using the same `builder` value it received from the proposer, both sides compute identical state diffs and an identical `PartialBlockHash`. The `ProposalFinMismatch` check therefore passes: [10](#0-9) 

The proposal is accepted by consensus. The committed block contains:

1. **Wrong fee recipient** — every transaction fee is transferred to the attacker's address instead of the legitimate sequencer.
2. **Wrong sequencer address in the block hash** — the block hash encodes the attacker's address as the sequencer, corrupting the canonical chain record.
3. **Wrong sequencer address in state sync** — downstream components (RPC, state sync) see the attacker's address as the block's sequencer.

This matches the allowed impact: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

---

### Likelihood Explanation

Any validator node that is legitimately selected as proposer for a round can exploit this. The proposer role rotates among all validators; it is not a permanently privileged position. No special capability beyond being selected as proposer is required. The attack is silent — no error is logged, no mismatch is detected, and the block is committed normally.

---

### Recommendation

Add a check inside `is_proposal_init_valid` (or in `handle_proposal` alongside the `proposer` check) that enforces `init.builder` equals the expected sequencer address for this node. The simplest fix is to require `builder == proposer` (since in the current architecture the proposer is the builder), or to compare `builder` against a locally-configured `builder_address` that the validator knows independently:

```rust
// In is_proposal_init_valid or handle_proposal:
if init_proposed.builder != expected_builder_address {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!("builder mismatch: expected {:?}, got {:?}",
            expected_builder_address, init_proposed.builder),
    ));
}
```

The `expected_builder_address` should be derived from the same source the proposer uses (`args.builder_address`), not from the received `ProposalInit`.

---

### Proof of Concept

A malicious validator node, when selected as proposer, modifies `initiate_build` (or intercepts the `ProposalInit` before it is sent) to set `builder` to an attacker-controlled address `ATTACKER`:

```
ProposalInit {
    height: N,
    proposer: <legitimate committee address>,  // passes committee check
    builder: ATTACKER,                          // not validated — passes all checks
    l2_gas_price_fri: <valid>,
    ...
}
```

1. The proposer sends this `ProposalInit` to all validators.
2. Each validator calls `is_proposal_init_valid` — passes (no `builder` check).
3. Each validator calls `convert_to_sn_api_block_info` → `sequencer_address = ATTACKER`.
4. The batcher executes all transactions; every `execute_fee_transfer` sends fees to `ATTACKER`.
5. `PartialBlockHashComponents.sequencer = ATTACKER`; both proposer and validator compute the same `PartialBlockHash`.
6. `built_block == received_fin.proposal_commitment` → no `ProposalFinMismatch`.
7. The block is committed. `ATTACKER` has received all transaction fees. The block hash and state-sync record embed `ATTACKER` as the sequencer.
