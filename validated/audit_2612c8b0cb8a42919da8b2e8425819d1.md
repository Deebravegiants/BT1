### Title
stETH Rebasing Causes Stale `ethValueInWithdrawal` Accounting, Enabling rsETH Price Manipulation and Yield Theft — (`contracts/LRTConverter.sol`)

---

### Summary

`LRTConverter.ethValueInWithdrawal` records a point-in-time ETH value snapshot of stETH transferred from the deposit pool. Because stETH is a rebasing token whose balance changes outside of transfers, this snapshot diverges from the actual ETH value held in the converter. The stale value is consumed directly by `LRTDepositPool.getETHDistributionData()` and ultimately by `LRTOracle._getTotalEthInProtocol()`, causing the rsETH price to be incorrect for the entire duration stETH sits in the converter.

---

### Finding Description

When `transferAssetFromDepositPool` is called with stETH, the converter records the ETH value of the transferred stETH as a snapshot:

```solidity
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
``` [1](#0-0) 

This snapshot is never updated to reflect subsequent stETH rebases. Meanwhile, `getAssetDistributionData` explicitly excludes stETH held in the converter from the per-asset accounting:

```solidity
assetLyingInConverter = 0; // assets in converter are accounted in their eth value => getETHDistributionData()
``` [2](#0-1) 

Instead, the converter's stETH is represented solely through `ethValueInWithdrawal` in `getETHDistributionData`:

```solidity
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
``` [3](#0-2) 

This value feeds directly into `_getTotalEthInProtocol()`, which drives rsETH price:

```solidity
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [4](#0-3) 

**Positive rebase scenario:** stETH balance in the converter grows (e.g., daily Lido rebase ~3–4% APY). `ethValueInWithdrawal` understates the actual ETH value. `_getTotalEthInProtocol()` returns less than the true TVL. rsETH price is understated. New depositors receive more rsETH than they are entitled to, permanently diluting existing holders of the yield that accrued during the rebase period.

**Negative rebase scenario (slashing):** stETH balance in the converter shrinks. `ethValueInWithdrawal` overstates the actual ETH value. rsETH price is inflated. After `claimStEth` is called, `_sendEthToDepositPool` reduces `ethValueInWithdrawal` only by the actual ETH received:

```solidity
if (ethValueInWithdrawal > _amount) {
    ethValueInWithdrawal -= _amount;
} else {
    ethValueInWithdrawal = 0;
}
``` [5](#0-4) 

Because `_amount < ethValueInWithdrawal` after a negative rebase, a permanent phantom residual remains in `ethValueInWithdrawal`. This phantom value permanently inflates `_getTotalEthInProtocol()` and rsETH price, causing new depositors to receive fewer rsETH than they are entitled to indefinitely.

---

### Impact Explanation

**Positive rebase (likely, daily):** Existing rsETH holders lose yield. During the window when stETH is in the converter and has rebased upward, rsETH is underpriced. Any user who deposits ETH or stETH during this window receives more rsETH than their proportional share of the TVL, permanently diluting existing holders. This is a continuous, low-magnitude theft of unclaimed yield from existing rsETH holders.

**Negative rebase (rare, slashing):** After a negative rebase and subsequent `claimStEth`, a permanent phantom ETH value remains in `ethValueInWithdrawal`. This permanently overstates rsETH price. All future depositors receive fewer rsETH than they are entitled to. The protocol's accounting is permanently broken until an admin manually corrects it (no such correction mechanism exists).

---

### Likelihood Explanation

stETH positive rebases occur daily and are predictable (Lido distributes rewards approximately every 24 hours). Any time stETH is held in the converter between `transferAssetFromDepositPool` and `claimStEth`, the accounting is stale. This window can span days or weeks depending on operator cadence. An unprivileged depositor can observe the on-chain state of `ethValueInWithdrawal` and the actual stETH balance of the converter to identify when rsETH is mispriced and exploit it by depositing. Negative rebases are rare but have occurred historically during validator slashing events.

---

### Recommendation

Replace the static ETH value snapshot with a live balance-based calculation. Instead of storing `ethValueInWithdrawal` as a fixed ETH amount, store the raw stETH token amount held in the converter and compute its current ETH value dynamically using the live oracle price at query time:

```solidity
// Store token amount, not ETH value
uint256 public stETHInConverter;

// In getETHDistributionData, compute live:
ethLyingInConverter = (stETHInConverter * lrtOracle.getAssetPrice(stETH)) / 1e18;
```

This ensures that any rebase (positive or negative) is automatically reflected in `_getTotalEthInProtocol()` without requiring any manual update.

---

### Proof of Concept

1. Operator calls `transferAssetFromDepositPool(stETH, 1000e18)`. Assume stETH price = 1 ETH. `ethValueInWithdrawal = 1000 ETH`.
2. Lido daily rebase occurs: stETH balance in converter grows to 1003e18 (0.3% daily rebase).
3. `_getTotalEthInProtocol()` returns `1000 ETH` for the converter portion instead of `1003 ETH`. rsETH price is understated by ~0.3% relative to true TVL.
4. Attacker deposits 10 ETH into `LRTDepositPool.depositETH`. They receive `10 / rsETHPrice` rsETH — approximately 0.3% more rsETH than their proportional share of the true TVL.
5. Operator calls `unstakeStEth(1003e18)` and later `claimStEth`. ETH received = 1003 ETH. `_sendEthToDepositPool(1003)` sets `ethValueInWithdrawal = 0` (since `1000 < 1003`). The 1003 ETH is now in the deposit pool and correctly counted.
6. The attacker's excess rsETH from step 4 is permanent — they captured yield that belonged to existing holders.

For the negative rebase case: replace step 2 with a 1% slashing event (stETH balance drops to 990e18). After `claimStEth` receives 990 ETH: `ethValueInWithdrawal = 1000 - 990 = 10 ETH` phantom value remains permanently, inflating all future rsETH price calculations.
