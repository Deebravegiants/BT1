Audit Report

## Title
Missing Chainlink Staleness Check Enables wrsETH Over-Issuance During Feed Freeze — (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

## Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs no `block.timestamp - updatedAt` staleness check, meaning a Chainlink feed that has stopped updating will still return its last price indefinitely. `RSETHPoolV2NBA.deposit()` consumes this rate directly in the mint formula with no daily mint cap, so any period of feed inactivity allows depositors to receive more wrsETH than the pool's rsETH collateral can back, creating an unbounded insolvency gap.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` (L26–37) calls `latestRoundData()` and applies only two guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();   // round completeness only
if (timestamp == 0) revert IncompleteRound();          // non-zero timestamp only
```

Neither check bounds how old `updatedAt` may be relative to `block.timestamp`. A feed frozen for 48 hours passes both guards as long as `answeredInRound >= roundID` and `timestamp != 0`.

`RSETHPoolV2NBA.deposit()` (L106–118) calls `viewSwapRsETHAmountAndFee()` (L124–133), which calls `getRate()` and uses the result directly:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Then immediately mints with no supply ceiling:

```solidity
wrsETH.mint(msg.sender, rsETHAmount);
```

Because rsETH accrues staking rewards, its ETH-denominated rate rises monotonically. A stale oracle frozen at a lower historical rate (e.g., 1.00 ETH/rsETH when the true rate is 1.05 ETH/rsETH) causes every depositor to receive `1/1.00 = 1.0 wrsETH` per ETH instead of the correct `1/1.05 ≈ 0.952 wrsETH`. The surplus wrsETH is unbacked by collateral.

Critically, `RSETHPoolV2NBA` has no `dailyMintLimit` or `limitDailyMint` modifier — unlike `RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`, which all implement a `dailyMintLimit` state variable and a `limitDailyMint` modifier as a defense-in-depth measure. `RSETHPoolV2NBA` has none of these, so the over-issuance is entirely unbounded.

## Impact Explanation
**Critical — Protocol insolvency.**

Over-issued wrsETH tokens represent claims on rsETH that the pool cannot satisfy upon redemption. The insolvency gap scales linearly with deposit volume during the stale window and with the magnitude of rate drift. Because `wrsETH.mint` is called with no supply ceiling in `RSETHPoolV2NBA`, the gap is unbounded. This directly matches the allowed impact: *Protocol insolvency*.

## Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for ETH/USD on mainnet, 24 hours on some L2s). Network congestion, keeper failures, or L2 sequencer downtime can cause feeds to go stale for hours without any oracle operator compromise or adversarial action. This is a well-known, non-adversarial operational failure mode. Any unprivileged depositor who calls `deposit()` during a stale window automatically receives excess wrsETH — no special knowledge or coordination is required. The condition is repeatable across any stale period.

## Recommendation
Add a configurable `MAX_STALENESS` constant and enforce it in `getRate()` in `ChainlinkOracleForRSETHPoolCollateral.sol`:

```solidity
uint256 public constant MAX_STALENESS = 3600; // adjust per feed heartbeat

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 updatedAt, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
    if (ethPrice <= 0) revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

Additionally, add a `dailyMintLimit` and `limitDailyMint` modifier to `RSETHPoolV2NBA` consistent with the pattern already implemented in `RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`.

## Proof of Concept

```solidity
// Fork the target L2 at a block where rsETH rate = 1.00 ETH (oracle updatedAt = block.timestamp)
// Warp forward 48 hours without triggering a Chainlink update
vm.warp(block.timestamp + 48 hours);

// True rsETH rate is now ~1.04 ETH (2 days of ~2% APY accrual)
// Oracle still returns 1.00 ETH — passes answeredInRound and timestamp checks

uint256 deposit = 100 ether;
pool.deposit{value: deposit}("");

// Minted:  100e18 * 1e18 / 1.00e18 = 100 wrsETH
// Correct: 100e18 * 1e18 / 1.04e18 ≈ 96.15 wrsETH
// Insolvency gap: ~3.85 wrsETH per 100 ETH deposited, unbounded across all depositors
uint256 minted = wrsETH.balanceOf(address(this));
assertGt(minted, 96.15 ether); // over-issued, unbacked
```

The test is reproducible as a Foundry fork test against any L2 deployment of `RSETHPoolV2NBA` using the `ChainlinkOracleForRSETHPoolCollateral` oracle.
