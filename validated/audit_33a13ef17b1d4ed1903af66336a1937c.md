Audit Report

## Title
Missing Staleness Check in `ChainlinkPriceOracle.getAssetPrice` Enables Over-Redemption via `instantWithdrawal` — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice` silently discards the `updatedAt` return value from `latestRoundData`, performing no staleness check. `instantWithdrawal` uses this potentially stale price as the denominator in `getExpectedAssetAmount` with no price-bounds guard, unlike `unlockQueue` which enforces caller-supplied min/max bounds via `_validatePrices`. A stale, deflated asset price inflates `assetAmountUnlocked`, allowing an attacker to burn a small amount of rsETH and redeem a disproportionately large amount of yield-bearing assets from `LRTUnstakingVault`.

## Finding Description

**Root cause 1 — no staleness check in `ChainlinkPriceOracle.getAssetPrice`:**

At `contracts/oracles/ChainlinkPriceOracle.sol` line 52, only `answer` is consumed from `latestRoundData`; `updatedAt` and `answeredInRound` are silently dropped:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

There is no check of the form `block.timestamp - updatedAt <= STALENESS_THRESHOLD`.

**Root cause 2 — `instantWithdrawal` has no price-bounds guard:**

`instantWithdrawal` calls `getExpectedAssetAmount` directly at line 228 and uses the result without any sanity bound:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
```

`getExpectedAssetAmount` (line 593) computes:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

`rsETHPrice` is a stored value updated by `updateRSETHPrice()`. `getAssetPrice(asset)` is read live from Chainlink at call time. A stale, deflated asset price inflates the result.

**Contrast with `unlockQueue`:**

`unlockQueue` (lines 288–295) calls `_validatePrices` with caller-supplied `minimumAssetPrice`/`maximumAssetPrice`/`minimumRsEthPrice`/`maximumRsEthPrice` before proceeding. `instantWithdrawal` has no equivalent guard, demonstrating the protocol is aware of price-manipulation risk but failed to apply the same protection to the instant path.

**Exploit flow:**

1. Attacker monitors on-chain `updatedAt` for a supported LST Chainlink feed (e.g., stETH/ETH).
2. Feed goes stale (heartbeat elapsed, network congestion, no deviation trigger) — `updatedAt` is old but the contract never checks it.
3. Attacker calls `instantWithdrawal(stETH, rsETHUnstaked, "")`.
4. `getExpectedAssetAmount` divides by the stale (deflated) price, yielding `assetAmountUnlocked` larger than the fair ETH equivalent of the rsETH burned.
5. The check `assetAmountUnlocked > getAssetsAvailableForInstantWithdrawal(asset)` only caps the total available, not the per-unit fairness.
6. rsETH is burned, and the inflated asset amount is redeemed from `LRTUnstakingVault` and transferred to the attacker (minus `instantWithdrawalFee`).

## Impact Explanation

**High — Theft of unclaimed yield.**

The attacker burns rsETH worth `X` ETH at fair prices but redeems assets worth `X * (fairPrice / stalePrice)` ETH. The surplus is extracted directly from `LRTUnstakingVault`, draining yield-bearing assets (stETH, ETHx, etc.) belonging to other protocol participants. The `instantWithdrawalFee` (0–10%) reduces but does not eliminate profit when price deviation is large enough (e.g., a 30% stale deflation yields ~43% over-redemption before fees).

## Likelihood Explanation

**Medium.** Chainlink LST/ETH feeds have heartbeat intervals of 1–24 hours and deviation thresholds of 0.5–1%. A feed can go stale without oracle operator action during network congestion or a period where price has not moved enough to trigger a deviation update. No privileged access, oracle operator compromise, or front-running of admin transactions is required. An attacker passively monitoring `updatedAt` on-chain can detect staleness and act within the same block.

## Recommendation

1. **Add a staleness check in `ChainlinkPriceOracle.getAssetPrice`** (`contracts/oracles/ChainlinkPriceOracle.sol`, line 52):

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= STALENESS_THRESHOLD, "Stale price");
require(price > 0, "Non-positive price");
```

2. **Add price-bounds validation to `instantWithdrawal`** (`contracts/LRTWithdrawalManager.sol`), mirroring the `_validatePrices` pattern already used in `unlockQueue`. Accept caller-supplied or governance-configured min/max bounds for both `rsETHPrice` and `assetPrice` and revert if either is outside the window.

## Proof of Concept

```solidity
// Foundry fork test (mainnet fork)

// 1. Deploy a MockAggregator returning a price 30% below real stETH/ETH rate
//    with updatedAt = block.timestamp - 25 hours (past any reasonable heartbeat).
MockAggregator staleFeed = new MockAggregator(
    realPrice * 70 / 100,
    block.timestamp - 25 hours
);

// 2. As LRTManager, point the stETH price feed to the mock.
//    (updatePriceFeedFor is onlyLRTManager — used here only to simulate a stale feed;
//     the same effect occurs naturally when a real feed goes stale.)
chainlinkOracle.updatePriceFeedFor(stETH, address(staleFeed));

// 3. Attacker holds rsETHUnstaked worth fairValue ETH at the real price.
uint256 rsETHUnstaked = 1 ether;
uint256 fairAssetAmount = rsETHUnstaked * rsETHPrice / realAssetPrice;

// 4. Call instantWithdrawal — no staleness revert, no price-bounds revert.
vm.prank(attacker);
withdrawalManager.instantWithdrawal(stETH, rsETHUnstaked, "");

// 5. Assert attacker received more than fair value.
uint256 received = stETH.balanceOf(attacker);
// received ≈ rsETHUnstaked * rsETHPrice / (realPrice * 0.70)
//           = fairAssetAmount / 0.70 ≈ fairAssetAmount * 1.43
assertGt(received, fairAssetAmount);
// Profit ≈ 43% of fairAssetAmount minus instantWithdrawalFee.
```

The assertion passes on unmodified production code because `ChainlinkPriceOracle.getAssetPrice` never checks `updatedAt` and `instantWithdrawal` never validates the price against any bounds.
