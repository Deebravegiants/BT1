### Title
Detached NEAR Token Transfer Promises Silently Discard Refunds on Failure — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `require_deposit` helper and `propose_update` method both schedule NEAR token refund transfers using `Promise::new(...).transfer(diff).detach()`. The `.detach()` call is the NEAR analog of ignoring a Solidity `transfer()` return value: if the promise fails at execution time, the failure is completely silent — no error is raised, no state is rolled back, and the contract proceeds as though the refund succeeded. Excess deposits attached by callers are permanently lost.

---

### Finding Description

Two distinct sites in `crates/contract/src/lib.rs` exhibit the pattern:

**Site 1 — `require_deposit` (line 137)**

```rust
fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        ...
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach(); // ← unchecked
            }
        }
    }
}
``` [1](#0-0) 

`require_deposit` is called from every user-facing entry point: `sign`, `request_app_private_key`, and `verify_foreign_transaction`. Any caller who attaches more than the 1-yoctoNEAR minimum triggers this refund path.

**Site 2 — `propose_update` (line 1330)**

```rust
if let Some(diff) = attached.checked_sub(required)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(proposer).transfer(diff).detach(); // ← unchecked
}
``` [2](#0-1) 

Here the required deposit can be substantial (it must cover storage staking for a new WASM binary). A voter who over-attaches loses the difference if the transfer promise fails.

In NEAR, `.detach()` is semantically equivalent to ignoring a Solidity `transfer()` return value: the promise is scheduled but its outcome is never observed. If the runtime fails to execute the transfer receipt (e.g., the recipient account was deleted between scheduling and execution, or the account balance falls below the minimum-balance threshold after the transfer is scheduled), the yoctoNEAR is consumed by the contract and never returned. The contract's own state does not record the outstanding liability, so there is no recovery path.

---

### Impact Explanation

**Medium.** The contract's deposit-accounting invariant — "every yoctoNEAR attached above the minimum is returned to the caller" — is silently violated whenever a detached transfer promise fails. Affected callers permanently lose the excess deposit. Because the contract holds no record of the failed refund, there is no on-chain mechanism to detect or remediate the loss. This breaks a production safety/accounting invariant without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

Low-to-medium. NEAR `Promise::transfer()` to an account that exists at scheduling time almost always succeeds. However, the failure surface is non-zero:

- A predecessor account can be deleted (via `DeleteAccount`) between the time the calling receipt executes (scheduling the transfer) and the time the transfer receipt executes (a later block).
- A contract upgrade or storage-staking change could push the recipient's balance below the minimum, causing the runtime to reject the incoming transfer.
- Any future NEAR protocol change that introduces new transfer failure modes would silently affect all detached refund paths simultaneously.

The `sign` and `verify_foreign_transaction` entry points are called by arbitrary unprivileged callers, making the `require_deposit` path reachable without any special role.

---

### Recommendation

Replace every `.detach()` on a refund transfer with a chained callback that checks the promise result and re-tries or logs a recoverable error. The pattern already used elsewhere in the contract (e.g., `return_signature_and_clean_state_on_success` using `#[callback_result]`) should be applied here:

```rust
// Instead of:
Promise::new(predecessor.clone()).transfer(diff).detach();

// Use a checked callback:
Promise::new(predecessor.clone())
    .transfer(diff)
    .then(
        Self::ext(env::current_account_id())
            .with_static_gas(Gas::from_tgas(5))
            .on_refund_complete(predecessor.clone(), diff),
    );
```

The `on_refund_complete` callback should inspect `#[callback_result]`, log a warning on failure, and — if the contract is expected to hold no residual balance — panic to surface the accounting error.

---

### Proof of Concept

1. Deploy the `mpc-contract` to a NEAR sandbox.
2. Create account `alice.near` and fund it.
3. Call `sign(...)` from `alice.near` attaching `1_000_000` yoctoNEAR (well above the 1-yoctoNEAR minimum).
4. Before the refund transfer receipt executes (i.e., in the same block or the next), submit a `DeleteAccount` action for `alice.near` targeting a beneficiary.
5. Observe: the `sign` call succeeds, the refund transfer receipt fails silently (no error surfaces to the caller or to contract logs beyond the initial `log!` line), and the `999_999` yoctoNEAR excess is permanently retained by the contract with no on-chain record of the liability.

The same sequence applies to `propose_update` with a voter account that over-attaches and is subsequently deleted.
