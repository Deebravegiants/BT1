Audit Report

## Title
Daily Mint Limit Permanently Consumed Without Decrement Enables Deposit DoS - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The `limitDailyMint` modifier across all L2 pool variants increments `dailyMintAmount` on every `deposit()` call but no code path ever decrements it. An attacker can deposit ETH to exhaust the daily mint limit, immediately sell the received wrsETH on a secondary market to recover capital, and block all other users from depositing for up to 24 hours at a cost of only the protocol fee and DEX slippage. The attack is repeatable every day indefinitely.

## Finding Description
In `RSETHPoolV3.sol`, `RSETHPoolV2ExternalBridge.sol`, `RSETHPoolV3ExternalBridge.sol`, and `RSETHPoolV3WithNativeChainBridge.sol`, the `limitDailyMint` modifier resets `dailyMintAmount` only when a new day begins:

```solidity
// RSETHPoolV3.sol L113-116
if (currentDay > lastMintDay) {
    lastMintDay = currentDay;
    dailyMintAmount = 0;
}
```

It then increments unconditionally on every deposit:

```solidity
// RSETHPoolV3.sol L119-124
if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
    revert DailyMintLimitExceeded();
}
dailyMintAmount += rsETHAmount;
```

There is no decrement path anywhere in any of the four contracts. The only reverse-flow function, `swapAssetToPremintedRsETH()`, does not touch `dailyMintAmount` and is restricted to `OPERATOR_ROLE` (or `OPERATOR_ROLE`/`WHITELISTED_USER_ROLE` in `RSETHPoolV2ExternalBridge`), making it inaccessible to ordinary users.

Because wrsETH is a freely transferable ERC-20 that trades on secondary markets, an attacker can:
1. Call `deposit{value: X}("")` sizing `X` so that `viewSwapRsETHAmountAndFee(X).rsETHAmount == dailyMintLimit - dailyMintAmount`, exhausting the limit in one transaction.
2. Receive wrsETH and immediately sell it on a DEX, recovering capital minus fee and slippage.
3. Every subsequent `deposit()` call by any user reverts with `DailyMintLimitExceeded` until midnight UTC.
4. Repeat the next day.

The same root cause applies identically across all four pool contracts.

## Impact Explanation
Once `dailyMintAmount == dailyMintLimit`, every call to either `deposit()` overload reverts for every user for the remainder of the 24-hour window. All L2 deposit paths — ETH and supported LST tokens — are gated by `limitDailyMint`. This constitutes a **temporary freezing of funds**: users' ability to deposit and receive wrsETH is completely blocked for up to 24 hours per attack cycle, repeatable indefinitely.

**Impact: Medium — Temporary freezing of funds.**

## Likelihood Explanation
- wrsETH is a liquid token traded on secondary markets; the attacker recovers capital immediately after depositing.
- The recurring attack cost is bounded by `feeBps × dailyMintLimit / 10_000` plus DEX slippage — potentially a fraction of a percent of the limit.
- No privileged role is required; any EOA or contract can call `deposit()`.
- The attack can be repeated every day indefinitely.
- The only deterrent is the upfront capital requirement (equal to the daily mint limit in ETH value), but since the attacker recovers it via the DEX, the recurring cost is only fees and slippage.

**Likelihood: Medium.**

## Recommendation
Decrement `dailyMintAmount` whenever wrsETH is effectively returned to the system. Inside `swapAssetToPremintedRsETH()`, after computing the `rsETHAmount` returned, add:

```solidity
if (getCurrentDay() == lastMintDay && dailyMintAmount >= rsETHAmount) {
    dailyMintAmount -= rsETHAmount;
}
```

Additionally, consider enforcing a non-zero minimum `feeBps` to raise the cost of cycling attacks, and monitor `dailyMintAmount` approaching `dailyMintLimit` as an on-chain signal.

## Proof of Concept
1. Attacker calls `deposit{value: X}("")` on `RSETHPoolV3` where `X` is sized so that `viewSwapRsETHAmountAndFee(X).rsETHAmount == dailyMintLimit - dailyMintAmount` (exhausting the limit in one transaction).
2. `limitDailyMint` sets `dailyMintAmount = dailyMintLimit`. [1](#0-0) 
3. Attacker receives wrsETH tokens and immediately sells them on a DEX for ETH, recovering capital minus fee and slippage.
4. Any subsequent call to `deposit()` by any user reverts: `dailyMintAmount + rsETHAmount > dailyMintLimit`. [2](#0-1) 
5. `dailyMintAmount` is never decremented — `swapAssetToPremintedRsETH()` does not touch it and is operator-gated. [3](#0-2) 
6. The block persists until `getCurrentDay() > lastMintDay` resets `dailyMintAmount = 0`. [4](#0-3) 
7. The same root cause and exploit path apply identically to `RSETHPoolV2ExternalBridge` (L104-126), `RSETHPoolV3ExternalBridge` (L130-159), and `RSETHPoolV3WithNativeChainBridge` (L108-137). [5](#0-4) [6](#0-5) [7](#0-6) 

**Foundry test plan:** Deploy `RSETHPoolV3` with a mock oracle and wrsETH, set `dailyMintLimit = 100 ether`. Call `deposit{value: X}` to exhaust the limit. Assert `dailyMintAmount == dailyMintLimit`. Call `deposit{value: 1 wei}` from a second address and assert it reverts with `DailyMintLimitExceeded`. Warp time by less than 1 day and assert the revert persists. Warp past 1 day and assert the deposit succeeds.
