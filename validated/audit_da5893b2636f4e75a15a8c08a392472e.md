Audit Report

## Title
LP Fee Sandwich Attack via Instant Liquidity Add/Remove — (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
LP fees from swaps are credited directly to per-bin token balances (`binState.token0BalanceScaled` / `binState.token1BalanceScaled`) with no per-share fee-growth checkpoint. Because `addLiquidity` and `removeLiquidity` impose no holding period or cooldown, an attacker can sandwich any large swap: add a dominant share position before the swap, let the swap deposit LP fees into the bin, then immediately remove the position and extract a proportional share of those fees, stealing revenue from existing LPs.

## Finding Description
**Fee accumulation root cause**

In `SwapMath.buyToken0InBinSpecifiedIn`, the LP fee (`token1FeeScaled - protocolFeeAmountScaled`) is added directly to `binState.token1BalanceScaled`:

```solidity
// SwapMath.sol L636-641
uint256 token1FeeScaled = lpFeeScaledFromGrossInput(totalIn1Scaled, currBinBuyFeeX64, onePlusBuyFeeX64);
uint256 protocolFeeAmountScaled = (token1FeeScaled * spreadFeeE6) / 1e6;
binState.token0BalanceScaled -= out0Scaled.toUint104();
binState.token1BalanceScaled =
  uint256((binState.token1BalanceScaled) + totalIn1Scaled - protocolFeeAmountScaled).toUint104();
```

After the swap, `binState.token1BalanceScaled` is permanently higher by `lpFee = token1FeeScaled - protocolFeeAmountScaled`. There is no separate fee accumulator; the fee is indistinguishable from principal.

**Share-proportional withdrawal with no checkpoint**

`LiquidityLib.removeLiquidity` pays out strictly proportional to the current bin balance:

```solidity
// LiquidityLib.sol L205-206
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

There is no `feeGrowthInside` snapshot or equivalent. Any shares minted before a swap immediately entitle the holder to a proportional share of fees deposited by that swap.

**No time-lock or holding period**

`addLiquidity` has no minimum holding period — only a `nonReentrant` guard and an optional extension hook (`_beforeAddLiquidity`) that is not enforced by default:

```solidity
// MetricOmmPool.sol L182-196
function addLiquidity(...) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
  ...
  _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
  (amount0Added, amount1Added) = LiquidityLib.addLiquidity(...);
  _afterAddLiquidity(...);
}
```

`removeLiquidity` only checks `msg.sender == owner`:

```solidity
// MetricOmmPool.sol L206
if (msg.sender != owner) revert NotPositionOwner();
```

**Attack flow**

Let bin have `B1` token1, total shares `S`, incoming swap LP fee `F`:

1. **Front-run**: attacker calls `addLiquidity` with `s` shares. Cost (ceiling): `B1 * s / S` token1. New state: `B1*(S+s)/S` token1, total shares `S+s`.
2. **Swap executes**: LP fee `F` added. Bin holds `B1*(S+s)/S + F` token1.
3. **Back-run**: attacker calls `removeLiquidity` with `s` shares. Receives: `(B1*(S+s)/S + F) * s/(S+s)` = `B1*s/S + F*s/(S+s)`.
4. **Attacker profit** ≈ `F * s/(S+s)`. With `s >> S`, attacker captures ~100% of `F`.

Existing LPs receive only `F * S/(S+s)` of the fees they earned.

## Impact Explanation
Direct, quantifiable loss of LP fee revenue for all existing liquidity providers in the sandwiched bin. The attacker extracts real token balances (`token1BalanceScaled` or `token0BalanceScaled`) credited to the pool by a legitimate swap. Pool solvency is not broken, but the fee portion owed to long-term LPs is diluted and captured by the attacker. This constitutes a direct loss of owed LP assets above Sherlock thresholds, qualifying as Critical/High severity.

## Likelihood Explanation
High. Any large swap broadcast to a public mempool is exploitable. The attacker requires only capital to add liquidity (no special role or permission), and a standard Flashbots bundle or priority-fee front/back-run. The `MINIMAL_MINTABLE_LIQUIDITY` floor is a dust guard, not a meaningful barrier. No default extension enforces a holding period. The attack is repeatable on every large swap.

## Recommendation
1. **Per-share fee-growth checkpoint** (preferred): Track a `feeGrowthInside` accumulator per bin (analogous to Uniswap v3). Snapshot the accumulator at mint time; only fees accumulated *after* a position is opened are claimable by that position. This eliminates the sandwich entirely.
2. **Minimum holding period**: Enforce a block-number or timestamp lock on newly minted positions before `removeLiquidity` is callable. This raises attack cost but does not fully eliminate it.
3. **Private mempool guidance**: Advise integrators to route large swaps through private relays (Flashbots Protect, MEV Blocker) to reduce front-running visibility.

## Proof of Concept
```
Setup:
  Bin 0: token1Balance = 1_000_000 (scaled), total shares S = 1_000_000
  Existing LP holds all S shares.
  Incoming swap: LP fee F ≈ 1_500 token1 (scaled) stays in bin.

Step 1 — Attacker front-runs addLiquidity(bin=0, shares=9_000_000):
  Cost (ceiling): 1_000_000 * 9_000_000 / 1_000_000 = 9_000_000 token1
  New bin token1: 10_000_000, total shares: 10_000_000

Step 2 — Swap executes:
  Bin token1 becomes: 10_000_000 + 1_500 = 10_001_500

Step 3 — Attacker back-runs removeLiquidity(bin=0, shares=9_000_000):
  Receives: 10_001_500 * 9_000_000 / 10_000_000 = 9_001_350 token1
  Profit: 9_001_350 − 9_000_000 = 1_350 token1
  (= F * s/(S+s) = 1_500 * 9/10 = 1_350 ✓)

Existing LP receives:
  10_001_500 * 1_000_000 / 10_000_000 = 1_000_150 token1
  Fee captured: 150 instead of 1_500 — 90% stolen.
```

Foundry test plan: deploy pool, seed bin with known balances and shares, execute `addLiquidity` → `swap` → `removeLiquidity` in sequence, assert attacker profit equals `F * s/(S+s)` and existing LP fee receipt equals `F * S/(S+s)`.
