### Title
Pool Wallet Singleton Permanently Stuck in `LEAVING_POOL` After Wallet Restart Due to Non-Persisted `target_state` - (File: `chia/pools/pool_wallet.py`)

---

### Summary

`PoolWallet.target_state` is a pure in-memory Python attribute that is never written to the database. When the wallet restarts while a pool-exit is in progress (singleton already on-chain in `LEAVING_POOL`), `target_state` is silently reset to `None`. The automatic second-phase transition logic in `new_peak()` guards on `target_state is None` and returns immediately, so the singleton is permanently stranded in `LEAVING_POOL` with no automatic recovery. The codebase itself labels this condition `INVALID_STATE`.

---

### Finding Description

**Root cause — `target_state` is ephemeral:**

`PoolWallet` is declared as a plain dataclass field with a default of `None`: [1](#0-0) 

`create_from_db()`, called on every wallet startup, constructs a fresh `PoolWallet` object without restoring `target_state`: [2](#0-1) 

`WalletStateManager.create()` calls `create_from_db()` for every `POOLING_WALLET` entry: [3](#0-2) 

**The broken transition — `new_peak()` returns early when `target_state is None`:**

The only code path that automatically submits the second singleton spend (the one that moves the singleton from `LEAVING_POOL` to `SELF_POOLING` or `FARMING_TO_POOL`) is gated entirely on `self.target_state`: [4](#0-3) 

If `target_state` is `None` the function returns immediately, and the second transaction is never generated regardless of how many blocks pass: [5](#0-4) 

**The codebase acknowledges the broken state:**

`pprint_pool_wallet_state` explicitly detects `LEAVING_POOL` with `target is None` and prints `INVALID_STATE`: [6](#0-5) 

**Attack / trigger path (unprivileged, normal operation):**

1. User calls `pw_self_pool()` or `pw_join_pool()` → `target_state` is set in memory, first singleton spend (→ `LEAVING_POOL`) is submitted and confirmed on-chain.
2. Wallet process restarts (crash, upgrade, normal shutdown) before `new_peak()` fires the second spend.
3. `create_from_db()` rebuilds `PoolWallet` with `target_state = None`.
4. `new_peak()` returns at line 814 on every subsequent block.
5. Singleton is permanently stranded in `LEAVING_POOL`; no second spend is ever submitted automatically.

---

### Impact Explanation

While the user can manually recover by re-calling `pw_self_pool()` after the `relative_lock_height` elapses, the automatic state machine is broken under a normal operational event (wallet restart). Until the user notices and manually intervenes:

- The singleton remains in `LEAVING_POOL`, meaning the pool's puzzle is still active and block rewards sent to `p2_singleton_puzzle_hash` are claimable by the pool, not the farmer.
- `new_peak()` silently does nothing on every block, giving no indication that action is required.
- The pool membership state is corrupted relative to the user's intent, matching the allowed High impact: *"Corruption of… pool membership state… with direct security impact"* and *"long-lived inability for… farmers… to process… pool actions… under normal network assumptions."*

---

### Likelihood Explanation

Any wallet restart (crash, upgrade, OS reboot) that occurs after the first singleton spend is confirmed but before `new_peak()` submits the second spend triggers this condition. The window is `relative_lock_height` blocks wide (minimum 5, commonly 32–100 blocks), making accidental triggering plausible for any user who restarts their wallet during normal operation.

---

### Recommendation

Persist `target_state`, `next_transaction_fee`, and `next_tx_config` to the `pool_state_transitions` database table (or a dedicated key-value store entry) so that `create_from_db()` can restore them. On startup, if the on-chain state is `LEAVING_POOL` and a persisted `target_state` exists, restore it so `new_peak()` can resume the transition automatically.

---

### Proof of Concept

```
1. Create pool wallet in FARMING_TO_POOL state.
2. Call pw_self_pool() → first spend confirmed, singleton now LEAVING_POOL on-chain.
   At this point: pool_wallet.target_state == <SELF_POOLING PoolState>
3. Kill/restart the wallet process.
4. On restart: create_from_db() → pool_wallet.target_state == None
5. Farm relative_lock_height + 10 blocks.
6. Observe: new_peak() is called each block but returns at line 814.
   Singleton remains in LEAVING_POOL indefinitely.
7. pw_status shows: "Current state: INVALID_STATE. Please leave/join again after block height X"
   (pprint_pool_wallet_state line 154)
8. Pool continues to receive farming rewards that should belong to the user.
```
