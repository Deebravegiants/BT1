### Title
`ForeignTxSignPayload` Hash Excludes Contract Identity and NEAR Chain ID, Enabling Cross-Instance Signature Replay - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `verify_foreign_transaction` flow produces a threshold signature over a hash that contains only the foreign-chain request and extracted values. It omits the NEAR contract account ID, the NEAR chain/network ID, and the `domain_id`. A `VerifyForeignTransactionResponse` (payload hash + signature) obtained from one contract deployment can be replayed verbatim to any other deployment that shares the same MPC key material, causing the second contract to accept a forged foreign-chain attestation without the MPC nodes ever processing the request on that instance.

---

### Finding Description

`ForeignTxSignPayload::compute_msg_hash` computes the 32-byte message that the MPC threshold signs as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload))
```

`ForeignTxSignPayload::V1` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

No NEAR contract account ID, no NEAR chain/network ID, and no `domain_id` is mixed into this hash. [1](#0-0) 

In `build_signature_request` (the node-side function that constructs the signing job), the key-derivation tweak is hardcoded to the zero scalar `[0u8; 32]`, so the root domain key is used directly with no per-request binding:

```rust
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),
    ...
})
``` [2](#0-1) 

On the contract side, `respond_verify_foreign_tx` verifies the submitted signature only against `response.payload_hash` and the domain's public key. It does not verify that `payload_hash` was computed over a payload that includes the current contract's account ID or the NEAR chain ID:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
``` [3](#0-2) 

Because the signed message is `SHA-256(borsh({request, values}))` with no instance-specific context, any valid `(payload_hash, signature)` pair produced for one contract deployment is cryptographically valid for every other deployment that holds the same public key.

---

### Impact Explanation

The NEAR MPC network is known to serve at least two live deployments simultaneously: `v1.signer` (mainnet) and `v1.signer-prod.testnet` (testnet), as documented in the contract README. [4](#0-3) 

Both deployments share the same MPC participant set and key material. A `VerifyForeignTransactionResponse` legitimately produced for a foreign-chain transaction on testnet can be submitted to `respond_verify_foreign_tx` on mainnet (or vice versa). The mainnet contract will accept it because the signature is valid over the same payload hash under the same public key.

In the primary use case — an omnibridge inbound flow where a NEAR contract credits funds after verifying a foreign-chain deposit — this replay allows an attacker to credit the same deposit on a second contract instance without the MPC nodes ever independently verifying the transaction on that instance. This constitutes a **double-spend / forged foreign-chain attestation**.

The same attack applies across contract upgrades/migrations that preserve key material, or across any two contract accounts that share a domain key.

---

### Likelihood Explanation

- The attack requires no threshold collusion, no key leakage, and no privileged access. Any party who observes a completed `verify_foreign_transaction` response on-chain (which is public) obtains the `payload_hash` and `signature` needed for the replay.
- The attacker only needs to call `respond_verify_foreign_tx` on the target contract with a matching pending request (which they can create themselves by calling `verify_foreign_transaction` with the same foreign-chain tx ID on the target contract first).
- The two live deployments (mainnet and testnet) share the same MPC key material, making the cross-deployment replay immediately exploitable without any special setup.

---

### Recommendation

**Short term:** Include the NEAR contract account ID (`env::current_account_id()`) and the NEAR chain ID in the `ForeignTxSignPayload` before hashing, so that signatures are bound to a specific contract instance and network:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub contract_id: AccountId,   // env::current_account_id() at request time
    pub near_chain_id: u64,       // env::block_height() or a stable chain-id field
    pub domain_id: DomainId,
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

**Long term:** Adopt a structured domain-separation scheme (analogous to EIP-712) that canonically encodes the contract identity, chain ID, domain ID, and payload version into every signed message, and document the scheme so that verifiers can reconstruct and check it independently.

---

### Proof of Concept

1. **Setup:** Two contract deployments share the same MPC key — e.g., `v1.signer` (mainnet) and `v1.signer-prod.testnet` (testnet). Both have a `ForeignTx` domain with the same public key.

2. **Step 1 — Obtain a valid response on testnet:** Submit `verify_foreign_transaction` for Bitcoin tx `0xABCD` on testnet. The MPC nodes verify the transaction and call `respond_verify_foreign_tx`, producing:
   ```
   payload_hash = SHA-256(borsh(V1 { request: Bitcoin(0xABCD, ...), values: [BlockHash(0x1234)] }))
   signature    = ECDSA_sign(root_key, payload_hash)
   ```
   The response is returned on-chain and is publicly observable.

3. **Step 2 — Plant a pending request on mainnet:** Call `verify_foreign_transaction` on mainnet with the identical `request` (same Bitcoin tx ID `0xABCD`, same extractors). This creates a pending yield on mainnet without the MPC nodes needing to respond.

4. **Step 3 — Replay the testnet response to mainnet:** Call `respond_verify_foreign_tx` on mainnet with the testnet `request` and `response`. Because `payload_hash` is identical (no contract ID or chain ID in the hash) and the public key is the same, `verify_ecdsa_signature` succeeds. [5](#0-4) 

5. **Result:** The mainnet contract resolves the pending yield and delivers the forged `VerifyForeignTransactionResponse` to the caller, attesting that Bitcoin tx `0xABCD` was independently verified on mainnet — without the MPC nodes ever querying a mainnet-configured RPC provider for that transaction. A bridge contract consuming this attestation would credit the deposit a second time.
