Audit Report

## Title
Instant Withdrawal Can Drain `LRTUnstakingVault` and DoS `unlockQueue()` — (File: contracts/LRTWithdrawalManager.sol)

## Summary
When instant withdrawal is enabled for an asset and `queuedWithdrawalsBuffer` is zero (the default), any rsETH holder can drain the `LRTUnstakingVault` via `instantWithdrawal()`. This causes every subsequent `unlockQueue()` call by the operator to revert unconditionally at the `AmountMustBeGreaterThanZero` guard, temporarily freezing all queued withdrawal requests until the vault is replenished — at which point the attacker can drain it again.

## Finding Description
`unlockQueue()` is restricted to `onlyAssetTransferOrOperatorRole` and reads the vault's live balance via `_createUnlockParams`:

```solidity
// LRTWithdrawalManager.sol L846-850
return UnlockParams({
    ...
    totalAvailableAssets: unstakingVault.balanceOf(asset)
});
```

It then reverts unconditionally if that balance is zero:

```solidity
// LRTWithdrawalManager.sol L297
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
```

`instantWithdrawal()`, callable by any rsETH holder when the feature is enabled, pulls assets directly from the vault:

```solidity
// LRTWithdrawalManager.sol L235
unstakingVault.redeem(asset, assetAmountUnlocked);
```

The amount available for instant withdrawal is bounded by `getAssetsAvailableForInstantWithdrawal`:

```solidity
// LRTUnstakingVault.sol L235-237
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
```

`queuedWithdrawalsBuffer` is a `mapping(address => uint256)` whose default value is `0`:

```solidity
// LRTUnstakingVault.sol L43
mapping(address asset => uint256 buffer) public queuedWithdrawalsBuffer;
```

With a zero buffer, `availableAmount == vaultBalance`, so the entire vault can be drained in one or more `instantWithdrawal` calls. Once empty, `unlockQueue()` reverts for every call regardless of how many pending withdrawal requests exist. The attacker can repeat the drain each time the vault is refilled, indefinitely extending the freeze without any privileged access.

## Impact Explanation
**Medium — Temporary freezing of funds.** Users who have already submitted withdrawal requests via `initiateWithdrawal()` (burning rsETH and committing expected asset amounts) cannot complete their withdrawals because `unlockQueue()` — the sole mechanism to advance the queue — reverts. The freeze persists until the vault is replenished and the operator either sets a non-zero buffer or disables instant withdrawal, both of which require privileged actions. The attacker can repeat the drain to extend the freeze window indefinitely.

## Likelihood Explanation
Two conditions must hold simultaneously:
1. `isInstantWithdrawalEnabled[asset] == true` — a deliberate manager action enabling a designed protocol feature.
2. `queuedWithdrawalsBuffer[asset] == 0` — the **default** state; the operator must explicitly call `setQueuedWithdrawalsBuffer` to protect the vault.

Once instant withdrawal is enabled (a normal operational state), any rsETH holder with sufficient balance can execute the attack without frontrunning, privileged access, or coordination. The attack is repeatable and cheap relative to the damage caused to queued users.

## Recommendation
1. **Enforce a non-zero buffer when enabling instant withdrawal.** In `setInstantWithdrawalEnabled`, require `queuedWithdrawalsBuffer[asset] > 0` before allowing `enabled = true`.
2. **Alternatively**, make `unlockQueue()` degrade gracefully when `totalAvailableAssets == 0` (e.g., return `(0, 0)`) rather than reverting, so the operator's call does not fail outright when the vault is temporarily empty.

## Proof of Concept
1. Manager calls `setInstantWithdrawalEnabled(ETH, true)`.
2. `queuedWithdrawalsBuffer[ETH]` remains `0` (default — operator never called `setQueuedWithdrawalsBuffer`).
3. Multiple users call `initiateWithdrawal(ETH, ...)`, locking their rsETH and committing expected ETH amounts.
4. Protocol refills the vault (e.g., via `completeUnstaking` → `NodeDelegator` → vault).
5. Attacker calls `instantWithdrawal(ETH, largeAmount, ...)`, draining the vault to zero (permitted because `getAssetsAvailableForInstantWithdrawal` returns the full balance when buffer is 0).
6. Operator calls `unlockQueue(ETH, firstExcludedIndex, ...)` → reverts with `AmountMustBeGreaterThanZero` at `LRTWithdrawalManager.sol` L297.
7. Queued users cannot complete their withdrawals. Attacker repeats step 5 each time the vault is refilled.

**Foundry test plan:** Deploy `LRTWithdrawalManager` and `LRTUnstakingVault` on a local fork. Enable instant withdrawal for ETH with zero buffer. Fund the vault. Call `instantWithdrawal` as an unprivileged rsETH holder to drain it. Assert that `unlockQueue` reverts with `AmountMustBeGreaterThanZero`. Confirm that setting a non-zero buffer via `setQueuedWithdrawalsBuffer` before enabling instant withdrawal prevents the drain.
