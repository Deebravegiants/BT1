Audit Report

## Title
Stale Cross-Chain Rate Used for rsETH Minting Without Freshness Check — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness validation against `lastUpdated`. `RSETHPoolV2.deposit()` is a permissionless function that uses this rate to compute rsETH minted per deposited ETH. Because rsETH is a yield-bearing token whose L1 rate monotonically increases, a stale rate causes the destination chain to over-issue rsETH, diluting existing holders' unclaimed yield.

## Finding Description

`CrossChainRateReceiver.getRate()` unconditionally returns `rate` with no check on `lastUpdated`: [1](#0-0) 

`lastUpdated` is set on every `lzReceive` call but is never compared against any maximum age anywhere in the receiver or in any consuming contract: [2](#0-1) 

`RSETHPoolV2.getRate()` delegates directly to this oracle: [3](#0-2) 

`viewSwapRsETHAmountAndFee` uses the rate to compute `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`: [4](#0-3) 

`deposit()` is fully permissionless and calls this path: [5](#0-4) 

`CrossChainRateProvider.updateRate()` is permissionless but requires the caller to pay LayerZero cross-chain message fees with no economic incentive to do so: [6](#0-5) 

There is no on-chain enforcement of update frequency. The contrast with `ChainlinkOracleForRSETHPoolCollateral`, which implements staleness checks (`answeredInRound < roundID`, `timestamp == 0`), confirms this protection was not applied to the cross-chain receiver: [7](#0-6) 

**Exploit path:**
1. Rate updates go uncalled for an extended period (e.g., 7 days) due to elevated gas costs or operational lapse.
2. True L1 rate grows from `R_old` to `R_new > R_old` via staking rewards.
3. Attacker calls `deposit()` with ETH; `viewSwapRsETHAmountAndFee` computes `rsETHAmount = amountAfterFee * 1e18 / R_old`, which is greater than the fair amount `amountAfterFee * 1e18 / R_new`.
4. Attacker receives excess rsETH that is unbacked at the current L1 rate.
5. Attacker bridges excess rsETH to L1 and redeems for more ETH than deposited, drawing from collateral belonging to other rsETH holders.

The `dailyMintLimit` bounds per-day damage but does not eliminate it: [8](#0-7) 

## Impact Explanation

**High — Theft of unclaimed yield.** At ~5% APY, a 7-day staleness window produces ~0.096% rate drift. Every depositor during the stale window receives slightly more rsETH than their ETH backs at the current L1 rate. When redeemed on L1, the excess draws from staking yield accrued to existing rsETH holders. The `dailyMintLimit` caps the absolute daily damage but does not prevent the dilution. Protocol insolvency is not a realistic outcome given the bounded drift and mint cap, making "High — Theft of unclaimed yield" the accurate classification.

## Likelihood Explanation

**Medium.** `updateRate()` is permissionless but imposes a real, uncompensated cost (LayerZero fees) on the caller. During elevated gas periods or any operational lapse, the rate can go stale for days. There is no on-chain circuit-breaker that pauses minting when `lastUpdated` is too old. The `PAUSER_ROLE` can pause manually, but this requires off-chain monitoring and human action, which is not guaranteed.

## Recommendation

Add a configurable `maxRateAge` to `CrossChainRateReceiver` and revert in `getRate()` if `block.timestamp - lastUpdated > maxRateAge`:

```solidity
uint256 public maxRateAge; // e.g., 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxRateAge, "Rate is stale");
    return rate;
}
```

This mirrors the staleness protection already present in `ChainlinkOracleForRSETHPoolCollateral`. Additionally, consider incentivizing `updateRate()` callers or running a protocol-owned keeper on a fixed schedule.
