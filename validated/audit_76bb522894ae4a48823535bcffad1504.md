### Title
`withdrawalDelayBlocks` Applied at Completion Time Retroactively Freezes Funds of Users Who Already Waited the Original Delay - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager` stores the withdrawal delay as a mutable parameter (`withdrawalDelayBlocks`) that is read at **completion time**, not at initiation time. When the manager increases this value, users who already initiated a withdrawal and waited the full original delay period find their funds re-locked, unable to complete their withdrawal until the new (longer) delay has elapsed from their original start block.

### Finding Description

When a user calls `initiateWithdrawal`, their rsETH is immediately transferred into the contract and a `WithdrawalRequest` is stored with `withdrawalStartBlock = block.number`. [1](#0-0) 

The completion check in `_processWithdrawalCompletion` reads the **current** `withdrawalDelayBlocks` state variable, not the value that was in effect at initiation: [2](#0-1) 

The manager can increase `withdrawalDelayBlocks` at any time, up to the 16-day ceiling: [3](#0-2) 

The initial value is set to `8 days / 12 seconds` at initialization: [4](#0-3) 

**Concrete scenario:**
1. User calls `initiateWithdrawal` at block `N`; rsETH is pulled from them and locked in the contract. `withdrawalDelayBlocks = 57,600` (~8 days).
2. User waits 57,600 blocks. They are now past the delay and expect to call `completeWithdrawal`.
3. Manager calls `setWithdrawalDelayBlocks(115,200)` (16 days) for a legitimate security reason.
4. User calls `completeWithdrawal` — it reverts: `block.number < N + 115,200`.
5. User's rsETH remains locked in the contract for an additional ~8 days beyond what they agreed to.

This is directly analogous to the GMX M-4 pattern: a mutable threshold is applied retroactively to users who were already in a compliant state under the old threshold, causing their funds to be frozen.

### Impact Explanation

The user's rsETH is held in the `LRTWithdrawalManager` contract from the moment `initiateWithdrawal` is called. After the delay increase, the user cannot complete their withdrawal and cannot reclaim their rsETH. The freeze lasts until the new delay period elapses from the original `withdrawalStartBlock`. The maximum additional freeze is bounded by the 16-day ceiling, making this a **temporary** freeze.

**Impact: Medium — Temporary freezing of funds.**

### Likelihood Explanation

The `setWithdrawalDelayBlocks` function is callable by any address holding `LRTManager` role, which is a protocol-operational role (not a timelock). The manager may legitimately increase the delay in response to a security incident or operational concern, inadvertently re-locking withdrawals that were already past the original threshold. No malicious intent is required; the bug manifests from normal parameter governance.

**Likelihood: Low-Medium** — requires a manager-initiated delay increase while users have pending withdrawals that have already passed the old delay.

### Recommendation

Snapshot `withdrawalDelayBlocks` into each `WithdrawalRequest` at initiation time and use that stored value for the completion check:

```solidity
// In WithdrawalRequest struct, add:
uint256 withdrawalDelayBlocksAtInitiation;

// In _addUserWithdrawalRequest:
withdrawalRequests[requestId] = WithdrawalRequest({
    rsETHUnstaked: rsETHUnstaked,
    expectedAssetAmount: expectedAssetAmount,
    withdrawalStartBlock: block.number,
    withdrawalDelayBlocksAtInitiation: withdrawalDelayBlocks  // snapshot
});

// In _processWithdrawalCompletion, replace:
if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();
// with:
if (block.number < request.withdrawalStartBlock + request.withdrawalDelayBlocksAtInitiation) revert WithdrawalDelayNotPassed();
```

This ensures users are only ever bound by the delay that was in effect when they committed their rsETH.

### Proof of Concept

```
State: withdrawalDelayBlocks = 57,600 (8 days)

1. Block 1,000,000: Alice calls initiateWithdrawal(ETH, 1e18 rsETH)
   - Alice's 1e18 rsETH transferred to LRTWithdrawalManager
   - withdrawalRequests[id] = { rsETHUnstaked: 1e18, withdrawalStartBlock: 1_000_000, ... }

2. Block 1,057,600: Alice is now past the 8-day delay.
   - She prepares to call completeWithdrawal.

3. Block 1,057,599: Manager calls setWithdrawalDelayBlocks(115_200) (16 days).
   - withdrawalDelayBlocks is now 115,200.

4. Block 1,057,600: Alice calls completeWithdrawal.
   - Check: block.number (1,057,600) < 1,000,000 + 115,200 (1,115,200) → TRUE → revert WithdrawalDelayNotPassed()

5. Alice's rsETH remains locked until block 1,115,200 (~57,600 additional blocks / ~8 more days).

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
