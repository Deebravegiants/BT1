Audit Report

## Title
Stranded ETH in Payable Router/Liquidity Contracts Is Capturable by Any Subsequent WETH Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
All swap and liquidity entry points in `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` are marked `payable` but do not enforce `msg.value == 0` when the input token is not WETH. ETH sent alongside a non-WETH call is silently retained by the contract. The `pay()` helper in `PeripheryPayments` unconditionally consumes the contract's entire native-ETH balance for any WETH-input call, allowing any subsequent attacker who performs a WETH swap or WETH-leg liquidity add to steal the stranded ETH at zero cost.

## Finding Description
`PeripheryPayments` has a `receive()` guard that reverts with `NotWETH()` if `msg.sender != WETH`:

```solidity
// PeripheryPayments.sol L32-34
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
```

This guard only applies to plain ETH transfers (no calldata). It does **not** apply when ETH is sent alongside a `payable` function call — in that case the function itself receives `msg.value` directly, bypassing `receive()`.

All four swap entry points in `MetricOmmSimpleRouter` are `payable` with no WETH check:
- `exactInputSingle` (L67), `exactInput` (L92), `exactOutputSingle` (L130), `exactOutput` (L154)

All four liquidity entry points in `MetricOmmPoolLiquidityAdder` are `payable` with no WETH check:
- `addLiquidityExactShares` (L56, L71), `addLiquidityWeighted` (L88, L123)

When a victim calls `exactInputSingle{value: 1 ether}(tokenIn=USDC, ...)`, the swap executes normally via `safeTransferFrom(USDC)`, but 1 ETH remains stranded in the router.

The `pay()` WETH branch then enables theft:

```solidity
// PeripheryPayments.sol L73-78
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    }
```

When an attacker calls `exactInputSingle{value: 0}(tokenIn=WETH, amountIn=1e18, ...)`, the swap callback triggers `_justPayCallback` → `pay(WETH, attacker, pool, 1e18)`. Since `address(this).balance == 1e18 >= 1e18`, the contract wraps the victim's ETH and forwards it to the pool as the attacker's input. The `payer` (attacker) is completely bypassed — no `transferFrom` is issued against them. The attacker receives the full swap output at zero cost.

`refundETH()` exists but is a separate optional call and provides no protection against front-running between the victim's transaction and the attacker's.

## Impact Explanation
Direct loss of user ETH principal. A victim who accidentally sends ETH with a non-WETH call loses that ETH to any attacker who subsequently performs a WETH-input swap or WETH-leg liquidity add. The attacker receives the full swap/liquidity output at zero cost. This is a High-severity direct loss of user funds meeting Sherlock thresholds.

## Likelihood Explanation
Low. The attack requires a user error: sending ETH alongside a non-WETH call. However, all entry points are `payable` and the WETH-via-ETH pattern is explicitly supported and documented, making this a realistic mistake. Once ETH is stranded on-chain, front-running it is trivial and requires no special privileges — any unprivileged caller can execute the theft in the same or next block.

## Recommendation
1. **Primary fix**: Add `if (msg.value > 0 && tokenIn != WETH) revert ETHNotAccepted();` at the top of each swap and liquidity function that does not use native ETH.
2. **Alternative**: Remove `payable` from functions that have no WETH leg. For functions that may or may not use ETH, enforce `msg.value == 0` when the relevant token is not WETH.
3. **Defense-in-depth**: In `pay()`, assert that `address(this).balance` consumed belongs to the current `payer` (e.g., by tracking deposited ETH per-call in transient storage), or revert if `address(this).balance > msg.value` when entering a WETH payment path.
