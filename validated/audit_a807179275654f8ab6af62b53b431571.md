Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks `initiateWithdrawal` Until Admin Intervenes - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at its Solidity default of `0`. The guard in `initiateWithdrawal` checks `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, which evaluates to `0 >= 0 = true` for every caller from the moment of deployment, causing unconditional revert with `WithdrawalLimitReached()`. All staked KERNEL tokens are frozen until an admin manually calls `setMaxNumberOfWithdrawalsPerUser`.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared as a plain storage variable with no default:

```solidity
// L108
uint256 public maxNumberOfWithdrawalsPerUser;
```

The `initialize` function sets roles and token addresses but never touches this variable:

```solidity
// L259-271
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... roles and token setup only ...
}
```

Inside `initiateWithdrawal`, the guard fires before any state change:

```solidity
// L323
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```

With `maxNumberOfWithdrawalsPerUser == 0`, a fresh user's `userWithdrawalIds[msg.sender].length` is also `0`, so `0 >= 0` is unconditionally `true`. The function reverts before decrementing `balanceOf` or creating any withdrawal record, making `claimWithdrawal` unreachable. The setter `setMaxNumberOfWithdrawalsPerUser` (L610–620) exists but is never invoked during initialization, and no deployment script calls it.

## Impact Explanation
Every user who stakes KERNEL tokens post-deployment is immediately unable to initiate a withdrawal. Their tokens remain locked in the contract for the entire window between deployment and admin remediation. This constitutes **Medium – Temporary freezing of funds**, matching the allowed impact scope. The freeze is not permanent only because an admin can eventually call `setMaxNumberOfWithdrawalsPerUser`; however, if the admin key is delayed, lost, or compromised, the freeze extends indefinitely.

## Likelihood Explanation
No attacker action is required. The broken state is the automatic post-deployment default. Any user who stakes and then calls `initiateWithdrawal` will be blocked immediately. The condition is deterministic and affects 100% of users from block 0 of deployment.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero value inside `initialize`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... existing setup ...
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER;
}
```

`MAX_WITHDRAWALS_PER_USER` is already defined as the constant `100` (L38), making it the natural safe default.

## Proof of Concept
1. Deploy `KernelDepositPool` behind a proxy and call `initialize(admin, kernelToken, rewardToken)`.
2. Verify: `maxNumberOfWithdrawalsPerUser()` returns `0`.
3. As any user, approve and call `stake(1e18)` — succeeds; `balanceOf[user] == 1e18`.
4. Call `initiateWithdrawal(1e18)` — reverts with `WithdrawalLimitReached` because `0 >= 0`.
5. Confirm `balanceOf[user]` is unchanged and no withdrawal record exists; `claimWithdrawal` is unreachable.
6. Admin calls `setMaxNumberOfWithdrawalsPerUser(10)` — step 4 now succeeds, confirming the root cause.
