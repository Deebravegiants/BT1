Audit Report

## Title
Stale Cross-Chain Oracle Rate Allows Depositors to Receive Excess wrsETH from Pool Reserves - (`contracts/pools/RSETHPool.sol`)

## Summary
`RSETHPool.deposit()` computes wrsETH output using `rsETHToETHrate` from `rsETHOracle.getRate()` with no staleness validation. The cross-chain oracle `CrossChainRateReceiver` stores `lastUpdated` on every `lzReceive()` call but `getRate()` returns the stored `rate` unconditionally. Because rsETH is a monotonically appreciating yield-bearing token, any period of oracle lag produces a stale rate that is lower than the true rate, causing the pool to dispense more wrsETH than the deposited ETH is worth — directly draining the pool's wrsETH reserves.

## Finding Description

`CrossChainRateReceiver.lzReceive()` records both `rate` and `lastUpdated` when a LayerZero message arrives: [1](#0-0) 

However, `getRate()` returns the stored rate with no check against `lastUpdated` or `block.timestamp`: [2](#0-1) 

`RSETHPool.getRate()` delegates directly to this oracle: [3](#0-2) 

`viewSwapRsETHAmountAndFee()` uses the returned rate as the denominator: [4](#0-3) 

`deposit()` then transfers the computed `rsETHAmount` of wrsETH to the caller: [5](#0-4) 

Because `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`, a stale (lower) rate produces a larger `rsETHAmount`. The excess wrsETH is taken directly from the pool's wrsETH balance. No existing check in `deposit()`, `viewSwapRsETHAmountAndFee()`, or `getRate()` validates freshness.

## Impact Explanation

**Critical — Direct theft of funds at-rest in the pool.** The wrsETH balance held by `RSETHPool` is the pool's reserve asset. A depositor who exploits a stale oracle window receives more wrsETH than their ETH contribution is worth at the true rate. The excess wrsETH can be immediately sold on secondary markets (Uniswap, etc.) at the true market rate. The loss is borne by the pool's wrsETH reserves, constituting direct theft of funds at-rest.

## Likelihood Explanation

Cross-chain oracle lag is a routine operational condition for LayerZero-based systems — message delivery can be delayed by minutes to hours under normal network conditions. No special privileges, front-running, or external protocol compromise is required. An attacker only needs to: (1) read `CrossChainRateReceiver.lastUpdated` on-chain, (2) compare the stored `rate` against the publicly readable L1 rate, and (3) call `deposit()` during any staleness window. The attack is repeatable on every staleness window and requires no victim interaction.

## Recommendation

1. **Add a staleness threshold in `CrossChainRateReceiver.getRate()`:**
   ```solidity
   uint256 public constant MAX_STALENESS = 24 hours;
   function getRate() external view returns (uint256) {
       require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate stale");
       return rate;
   }
   ```
2. **Add a minimum rate floor in `viewSwapRsETHAmountAndFee()`** to revert if the oracle rate deviates significantly from an expected floor.
3. **Add a `minRsETHOut` parameter to `deposit()`** so depositors can specify slippage protection against rate manipulation.

## Proof of Concept

```solidity
// Arbitrum fork test
function testStaleOracleExploit() public {
    // 1. True L1 rate has increased to 1.155e18 but oracle still reports 1.05e18
    vm.mockCall(
        address(rsETHOracle),
        abi.encodeWithSelector(IOracle.getRate.selector),
        abi.encode(1.05e18) // stale, ~10% below true rate
    );

    // 2. Attacker deposits 1 ETH
    vm.deal(attacker, 1 ether);
    vm.prank(attacker);
    pool.deposit{value: 1 ether}("ref");

    // 3. Attacker received ~0.952e18 wrsETH; fair value at true rate is ~0.866e18
    uint256 fairAmount = 1e18 * 1e18 / 1.155e18; // ~0.866e18
    uint256 received = wrsETH.balanceOf(attacker);
    assertGt(received, fairAmount); // excess ~0.086e18 wrsETH drained from pool
}
```
