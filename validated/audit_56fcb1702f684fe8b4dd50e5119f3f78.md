### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

### Summary

In `respond_ckd`, when a CKD request uses the `AppPublicKey` (non-publicly-verifiable, legacy) variant, the contract performs **zero verification** of the CKD response before delivering it to the user. A single malicious attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged CKD output, bypassing the threshold requirement entirely. The pending request is consumed and all queued yields are resumed with the forged key material.

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` handles responses to confidential key derivation requests. For the `AppPublicKeyPV` variant it calls `ckd_output_check` to verify the response on-chain. For the `AppPublicKey` variant the match arm is **empty** — no check is performed:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

The function then unconditionally calls `pending_requests::resolve_yields_for`, which removes the request from `pending_ckd_requests` and resumes every queued yield with the unverified response bytes. The request is permanently consumed and the forged key material is delivered to all waiting callers. [1](#0-0) 

The `AppPublicKey` variant is still accepted by `request_app_private_key` and is described as the "legacy" single-G1-point format. It remains a live code path. [2](#0-1) 

By contrast, `respond` (for signatures) always verifies the ECDSA or EdDSA signature cryptographically before resolving yields, and `respond_verify_foreign_tx` likewise verifies the signature over the payload hash. Only `respond_ckd` with `AppPublicKey` skips all output verification. [3](#0-2) 

### Impact Explanation

**Critical — Confidential key derivation output without the required participant authorization.**

The threshold cryptography model guarantees that no coalition smaller than the reconstruction threshold can forge outputs. The missing check breaks this guarantee for `AppPublicKey` CKD requests: a **single** malicious attested participant can:

1. Monitor `pending_ckd_requests` for entries using the `AppPublicKey` variant.
2. Construct an arbitrary `CKDResponse` — e.g., one that encrypts a known plaintext to the user's ephemeral public key.
3. Call `respond_ckd` with the forged response before honest nodes respond.
4. The contract removes the request from state and fans the forged key out to every queued caller via `resolve_yields_for`. [4](#0-3) 

The attacker knows the plaintext of the fake derived key they injected. Any data the victim subsequently encrypts to that key is readable by the attacker. The victim's CKD request is permanently consumed — re-submitting the same request will produce a fresh yield, but the attacker can race again. The `near-mpc-sdk` client-side verifier (`ForeignChainSignatureVerifier`) has no equivalent for CKD, so there is no off-chain safety net.

### Likelihood Explanation

**High.** The `AppPublicKey` variant is the legacy path and is still accepted. Any single node that passes TEE attestation (`assert_caller_is_attested_participant_and_protocol_active`) can submit a forged response. On NEAR, transaction ordering within a block is not guaranteed, so the attacker can attempt to front-run honest nodes. The attack requires no threshold cooperation, no key material, and no special tooling beyond a standard NEAR function call. [5](#0-4) 

### Recommendation

Add on-chain output verification for the `AppPublicKey` variant in `respond_ckd`, or reject `AppPublicKey` CKD requests at the `respond_ckd` entry point and require callers to migrate to `AppPublicKeyPV`. The `AppPublicKeyPV` path already has the correct guard via `ckd_output_check`; the `AppPublicKey` arm must not be a silent no-op. [6](#0-5) 

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey` variant; the request is stored in `pending_ckd_requests`.
2. Malicious attested participant constructs a `CKDResponse` containing a known-plaintext encrypted key.
3. Malicious participant calls `respond_ckd(request, forged_response)`.
4. `assert_caller_is_attested_participant_and_protocol_active` passes (participant is legitimately attested).
5. The `AppPublicKey` match arm executes with no body — no check runs.
6. `resolve_yields_for` removes the request and resumes all queued yields with the forged bytes.
7. The user's promise resolves with the forged derived key; the attacker knows its plaintext.
8. Honest nodes' subsequent `respond_ckd` calls return `RequestNotFound` — the request is already gone. [5](#0-4) [7](#0-6)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
