### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Submitted Request, Enabling Cross-Request Replay of Foreign-Chain Attestations — (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_verify_foreign_tx` function in the MPC contract verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never checks that `response.payload_hash` was actually derived from the `request` argument supplied in the same call. A single malicious attested participant (strictly below the signing threshold) can replay a legitimately-produced response from a prior foreign-chain verification against any currently-pending request for a *different* foreign transaction, causing the victim caller to receive a forged attestation.

---

### Finding Description

`respond_verify_foreign_tx` accepts two independent arguments:

```rust
pub fn respond_verify_foreign_tx(
    &mut self,
    request: VerifyForeignTransactionRequest,   // identifies which pending yield to drain
    response: VerifyForeignTransactionResponse, // contains payload_hash + signature
) -> Result<(), Error>
```

The on-chain `VerifyForeignTransactionRequest` struct contains only `(request: ForeignChainRpcRequest, domain_id, payload_version)` — no caller account ID, no nonce, no timestamp. [1](#0-0) 

The `VerifyForeignTransactionResponse` struct contains only `(payload_hash: Hash256, signature: SignatureResponse)`. [2](#0-1) 

The contract's validation logic is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // taken from the RESPONSE

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```
