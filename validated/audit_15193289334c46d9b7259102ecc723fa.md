Audit Report

## Title
`CrossChainRateReceiver.getRate()` Returns Stale Rate Without Staleness Check, Enabling Over-Minting of wrsETH on L2 Pools - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver` stores `lastUpdated` on every `lzReceive` call but never consults it in `getRate()`, which unconditionally returns the stored `rate`. When `RSETHRateReceiver` (which inherits this base) is set as `rsETHOracle` in `RSETHPoolV3`, a stale rate causes the pool to mint more `wrsETH` than deposited ETH is worth, allowing an attacker to extract unclaimed yield from the pool at the expense of existing holders.

## Finding Description
`CrossChainRateReceiver.lzReceive()` sets `lastUpdated = block.timestamp` on every rate update, but `getRate()` ignores it entirely:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol
uint256 public lastUpdated;   // line 16 — stored but never validated

function getRate() external view returns (uint256) {
    return rate;              // lines 103-105 — no staleness check
}
```

`RSETHRateReceiver` inherits `CrossChainRateReceiver` without adding any staleness guard. `RSETHPoolV3` sets this receiver as `rsETHOracle` and delegates its own `getRate()` directly to it:

```solidity
// contracts/pools/RSETHPoolV3.sol line 235-237
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
```

The rate is then used as the denominator when computing how many `wrsETH` tokens to mint:

```solidity
// contracts/pools/RSETHPoolV3.sol lines 299-308
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because rsETH is a yield-bearing token, its ETH price increases monotonically. If `updateRate()` on `CrossChainRateProvider` is not called (it is permissionless but requires an ETH fee payment for LayerZero), the stored rate on the receiver falls below the true current rate. A lower denominator produces a larger `rsETHAmount`, so the depositor receives more `wrsETH` than their ETH is worth at the true rate.

The protocol already applies staleness validation in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
```

This confirms the protocol is aware of the staleness risk class but omitted the equivalent guard in `CrossChainRateReceiver`.

The `dailyMintLimit` modifier in `RSETHPoolV3` caps the total wrsETH minted per day, which limits the scale of a single day's exploitation but does not prevent the attack — an attacker can repeat it each day the rate remains stale.

## Impact Explanation
An attacker who deposits ETH while the cross-chain rate is stale receives more `wrsETH` than their ETH is worth at the true current rate. When they later redeem or bridge that `wrsETH` at the correct (higher) rate, they extract more ETH than they deposited. The excess comes from yield that should have accrued to existing `wrsETH` holders. This is **theft of unclaimed yield**, matching the High severity impact class in the allowed scope.

## Likelihood Explanation
`CrossChainRateProvider.updateRate()` is permissionless but requires the caller to supply ETH to cover LayerZero message fees. There is no on-chain keeper or enforced update cadence. LayerZero message delivery is also not instantaneous and can be delayed by network congestion or bridge downtime. Any of these conditions — absent keeper, fee cost, or bridge delay — can leave the receiver's rate stale for hours or days. The attack requires no special privileges: `RSETHPoolV3.deposit()` is callable by any address with ETH.

## Recommendation
Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the stored rate is too old:

```solidity
uint256 public maxStaleness; // e.g., set to 24 hours by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Stale rate");
    return rate;
}
```

The owner should be able to update `maxStaleness` via an `onlyOwner` setter. An initial value of 0 should be rejected to prevent accidentally deploying with no staleness protection.

## Proof of Concept
1. At `t=0`, `CrossChainRateReceiver.rate = 1.05e18` and `lastUpdated = t0`. `updateRate()` is not called again (no keeper, fee cost, or bridge delay).
2. By `t=48h`, the true rsETH/ETH rate has grown to `1.06e18`, but the receiver still holds `1.05e18`.
3. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("")`.
4. Pool computes `rsETHAmount = 100e18 * 1e18 / 1.05e18 ≈ 95.238 wrsETH`. At the true rate the correct amount is `100e18 * 1e18 / 1.06e18 ≈ 94.340 wrsETH`. The attacker receives `≈ 0.898 wrsETH` excess.
5. Attacker redeems or bridges the excess `wrsETH` at the true rate of `1.06e18`, recovering `≈ 0.952 ETH` more than deposited, extracted from the pool's liquidity (other users' unclaimed yield).
6. The attack is bounded per day by `dailyMintLimit` but repeatable each day the rate remains stale.

**Foundry fork test outline:**
- Fork an L2 where `RSETHRateReceiver` is deployed as `rsETHOracle`.
- `vm.warp(block.timestamp + 2 days)` to simulate staleness without a new `lzReceive` call.
- Call `RSETHPoolV3.deposit{value: 100 ether}("")` from an attacker address.
- Assert `wrsETH.balanceOf(attacker) > expectedAmountAtTrueRate`.
