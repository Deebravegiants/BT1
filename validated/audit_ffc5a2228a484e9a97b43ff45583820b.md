### Title
Permissionless `FeeReceiver.sendFunds()` Allows Atomic Yield Front-Running via Stale rsETH Price — (`contracts/FeeReceiver.sol` / `contracts/LRTDepositPool.sol`)

---

### Summary

`FeeReceiver.sendFunds()` carries no access control and can be called by any external account. It pushes accumulated MEV/execution-layer ETH rewards into `LRTDepositPool`, immediately increasing the protocol's TVL. Because `LRTDepositPool.depositETH()` / `depositAsset()` price rsETH using the **stale, stored** `rsETHPrice` (never refreshed during the deposit flow), an attacker can atomically call `sendFunds()` and then deposit in the same transaction, minting rsETH at the pre-reward price and capturing yield that belongs to existing rsETH holders.

---

### Finding Description

`FeeReceiver.sendFunds()` is declared with no role modifier:

```solidity
// contracts/FeeReceiver.sol:53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

This call increases `address(depositPool).balance`, which is counted in `getETHDistributionData()` and therefore in `_getTotalEthInProtocol()` inside `LRTOracle`. [2](#0-1) 

However, `depositETH()` / `depositAsset()` compute the rsETH to mint using the **last stored** `rsETHPrice`, not a freshly computed one:

```solidity
// contracts/LRTDepositPool.sol:519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

`rsETHPrice` is only updated when `updateRSETHPrice()` is explicitly called — it is never invoked inside the deposit path. [4](#0-3) 

An attacker can therefore deploy a proxy contract that, in a single transaction:
1. Calls `FeeReceiver.sendFunds()` — moves accumulated rewards into the deposit pool, raising TVL.
2. Immediately calls `LRTDepositPool.depositETH()` — mints rsETH at the now-stale (pre-reward) lower price.

When `updateRSETHPrice()` is eventually called, the oracle computes `previousTVL = rsethSupply × rsETHPrice`. Because the attacker's deposit is already included in `rsethSupply`, the detected `rewardAmount` is still the full reward `R`, and the protocol fee is still taken. But the attacker's rsETH was minted at the old price, so their position is immediately worth more than they paid — the surplus comes directly from the yield that should have accrued to pre-existing rsETH holders. [5](#0-4) 

---

### Impact Explanation

**Theft of unclaimed yield (High).**

Concrete example:
- TVL = 1 000 ETH, supply = 1 000 rsETH, stored price = 1 ETH/rsETH.
- 10 ETH of MEV rewards sit in `FeeReceiver`.
- Attacker calls `sendFunds()` then `depositETH(100 ETH)` atomically.
  - Attacker receives `100 / 1 = 100 rsETH` at the stale price.
- `updateRSETHPrice()` is called later:
  - `previousTVL = 1 100 × 1 = 1 100 ETH`
  - `rewardAmount = 1 110 − 1 100 = 10 ETH`
  - `protocolFeeInETH = 1 ETH` (10 % example)
  - `newRsETHPrice = 1 109 / 1 100 ≈ 1.00818 ETH/rsETH`
- Attacker's 100 rsETH is now worth ≈ 100.82 ETH — a profit of ≈ 0.82 ETH.
- Pre-existing holders receive ≈ 8.18 ETH of yield instead of the full 9 ETH they were entitled to.

The attacker scales profit linearly with deposit size and with the size of accumulated rewards.

---

### Likelihood Explanation

Any unprivileged external account can call `sendFunds()` — no role, no whitelist, no time-lock. The attack requires only a proxy contract and a deposit, both of which are standard on-chain operations. MEV rewards accumulate continuously, so the opportunity recurs every time rewards build up before `updateRSETHPrice()` is called.

---

### Recommendation

Apply one or more of the following mitigations:

1. **Restrict `sendFunds()`** to a trusted role (e.g., `MANAGER`) so that only the protocol can decide when to flush rewards into the deposit pool.
2. **Refresh the price before minting**: call `updateRSETHPrice()` at the start of `depositETH()` / `depositAsset()` so the deposit always uses the current TVL.
3. **Track the last `sendFunds()` timestamp** and disallow deposits (or disallow the price benefit) within the same block, analogous to the Tigris Trade mitigation of tracking the last limit-order update timestamp.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IFeeReceiver { function sendFunds() external; }
interface IDepositPool { function depositETH(uint256 minRSETH, string calldata ref) external payable; }

contract YieldFrontRunner {
    IFeeReceiver  public feeReceiver;
    IDepositPool  public depositPool;

    constructor(address _feeReceiver, address _depositPool) {
        feeReceiver = IFeeReceiver(_feeReceiver);
        depositPool = IDepositPool(_depositPool);
    }

    /// @notice Atomically flush rewards then deposit at the stale (pre-reward) price.
    function attack() external payable {
        // Step 1: push accumulated MEV rewards into the deposit pool → TVL rises, price stays stale
        feeReceiver.sendFunds();

        // Step 2: deposit at the now-stale lower rsETH price → receive more rsETH than fair value
        depositPool.depositETH{value: msg.value}(0, "");
    }

    receive() external payable {}
}
```

The attacker deploys `YieldFrontRunner`, funds it with ETH, and calls `attack()`. The rsETH minted in step 2 is immediately worth more than the deposited ETH once `updateRSETHPrice()` is called, with the surplus extracted from existing rsETH holders' accrued yield.
