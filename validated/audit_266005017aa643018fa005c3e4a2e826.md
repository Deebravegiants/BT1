### Title
Non-Deterministic `create` Deployment in Price Provider Factories Enables Reorg-Based Pool Disruption — (`smart-contracts-poc/contracts/PriceProviderFactoryL2.sol`, `PriceProviderFactory.sol`, `AnchoredProviderFactory.sol`)

### Summary

All three price provider factories deploy their provider contracts using the plain `new` opcode (CREATE), which derives the deployed address from `keccak256(deployer, nonce)` and is therefore non-deterministic. On L2 chains — which this protocol explicitly targets via `PriceProviderFactoryL2` — a reorg can cause the deployed provider to land at a different address than the one recorded in the factory's state and passed to `MetricOmmPoolFactory.createPool()`. Any pool configured with the pre-reorg provider address then references a stale or empty contract, permanently breaking swap execution if the price provider was set as immutable.

### Finding Description

**Root cause — three factories, same pattern:**

`PriceProviderFactory.createPriceProvider()` (L1): [1](#0-0) 

`PriceProviderFactoryL2.createPriceProvider()` (L2 — highest-risk surface): [2](#0-1) 

`AnchoredProviderFactory.createAnchoredProvider()`: [3](#0-2) 

None of these calls passes a `salt`; all three use the plain CREATE opcode. The factory immediately records the returned address as authoritative: [4](#0-3) 

**Contrast with pool deployment — which is already safe:**

`MetricOmmPoolDeployer.deploy()` correctly uses CREATE2: [5](#0-4) 

So the pool address is stable across reorgs, but the price provider address it was configured with is not.

**End-to-end exploit path (L2 reorg):**

1. Attacker or honest user calls `PriceProviderFactoryL2.createPriceProvider()` → provider deployed at address **X** (factory nonce **N**).
2. Caller calls `MetricOmmPoolFactory.createPool()` with `priceProvider = X`. The factory validates `IPriceProvider(X).token0()/token1()` and stores X in the pool's immutable slot. [6](#0-5) 
3. A reorg occurs on the L2. The `createPriceProvider` transaction is replayed at a different nonce **N'** → provider now lives at address **Y ≠ X**.
4. The pool (deterministic CREATE2 address, unchanged) still holds `priceProvider = X`. Address X is now either empty or occupied by a different contract.
5. Every subsequent `swap()` call on the pool invokes the price provider at X. If X is empty the call reverts; if X is now a different contract, it returns an arbitrary price.

**Immutable price provider makes the damage permanent:**

When `priceProviderTimelock == type(uint256).max`, the pool admin cannot ever update the price provider: [7](#0-6) [8](#0-7) 

The pool is permanently bricked for swaps with no recovery path.

### Impact Explanation

- **Unusable swap flows** — every `swap()` call reverts because the price provider at the stale address is empty. This satisfies the "broken core pool functionality / unusable swap flows" gate.
- **Bad-price execution** — if another contract is coincidentally deployed at the old address post-reorg (e.g., a different provider from the same factory at the same nonce), the pool executes swaps against an unvalidated price source, violating the "bad-price execution" gate.
- **Permanent pool insolvency risk** — LPs cannot exit via swaps; if the pool is immutable-provider, the only exit is `removeLiquidity`, but the pool's token balances remain locked against a broken swap surface.
- Severity: **Medium** (matches the external report's rating; requires a reorg event, but L2 reorgs are a documented, non-negligible risk).

### Likelihood Explanation

- `PriceProviderFactoryL2` is explicitly named and deployed for L2 chains (Optimism, Arbitrum, Base, etc.), which have documented sequencer-driven reorg windows.
- The attack requires no privileged access — `createPriceProvider` is permissionless on all three factories.
- The window is narrow (one reorg event between provider deployment and pool creation), but L2 reorgs during sequencer downtime or forced-inclusion periods are realistic.
- The `AnchoredProviderFactory` is also permissionless and suffers the same issue. [9](#0-8) 

### Recommendation

Replace plain `new` with CREATE2 in all three factories by passing a deterministic salt derived from caller-controlled and feed-specific inputs, e.g.:

```solidity
// PriceProviderFactoryL2 example
bytes32 salt = keccak256(abi.encode(msg.sender, _oracle, _feedId, _baseToken, _quoteToken));
PriceProviderL2 p = new PriceProviderL2{salt: salt}(
    address(this), _oracle, _feedId, _marginStep,
    _maxTimeDelta, _futureTolerance, _baseToken, _quoteToken
);
```

Including `msg.sender` in the salt prevents front-running of the deterministic address while keeping it stable across reorgs. Apply the same pattern to `PriceProviderFactory.createPriceProvider()` and `AnchoredProviderFactory.createAnchoredProvider()`.

### Proof of Concept

```
Setup (L2 fork, e.g. Optimism):
  1. Deploy PriceProviderFactoryL2 (factory nonce = 0).
  2. Call factory.createPriceProvider(oracle, feedId, ...) 
     → PriceProviderL2 deployed at address X = CREATE(factory, nonce=1).
  3. Call MetricOmmPoolFactory.createPool({priceProvider: X, priceProviderTimelock: type(uint256).max, ...})
     → Pool deployed at deterministic CREATE2 address P, immutable priceProvider = X.

Reorg simulation:
  4. Roll back the chain to before step 2.
  5. Insert a different transaction that increments the factory nonce (e.g., another createPriceProvider call).
  6. Replay step 2 → PriceProviderL2 now deployed at address Y = CREATE(factory, nonce=2) ≠ X.
  7. Replay step 3 → Pool P still created with priceProvider = X (address X is now empty).

Exploit:
  8. Call pool.swap(...) → reverts because IPriceProvider(X) has no code.
  9. Call factory.proposePoolPriceProvider(P, newProvider) → reverts with PriceProviderImmutable().
  10. Pool P is permanently bricked; all LP capital is stranded in a swap-unusable pool.
```
