Audit Report

## Title
LP Principal Permanently Locked When Position Owner Is Blacklisted by Pool Token â€” (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`LiquidityLib.removeLiquidity` unconditionally transfers redeemed tokens to the `owner` address with no alternative recipient and no admin rescue path. If `owner` is blacklisted by USDC or USDT (both explicitly in scope), every withdrawal attempt reverts at the `safeTransfer` step, permanently locking the LP's principal in the pool with no recovery mechanism.

## Finding Description
In `LiquidityLib.removeLiquidity` (lines 242â€“247), after all bin accounting is updated and shares are burned, the function executes:

```solidity
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);
}
if (amount1Removed > 0) {
    IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
}
```

`owner` is the sole hardcoded recipient. There is no `recipient` parameter and no fallback address.

In `MetricOmmPool.removeLiquidity` (line 206), the caller-side check `if (msg.sender != owner) revert NotPositionOwner()` ensures no third party â€” including the pool admin or factory â€” can call `removeLiquidity` on the owner's behalf to redirect tokens to a clean address.

The `MinimalLiquidity` guard (lines 200â€“202) prevents partial withdrawals that would leave residual shares below `MINIMAL_MINTABLE_LIQUIDITY`, closing the partial-drain escape hatch.

A search of the entire `metric-core/contracts/` tree confirms there is no emergency withdrawal, token rescue, or admin sweep function anywhere in the pool or factory. The `whenNotPaused` modifier applies only to `swap`, not to `removeLiquidity`, so pausing the pool creates no rescue path.

## Impact Explanation
An LP whose address is blacklisted by USDC or USDT will find that every call to `removeLiquidity` reverts at the `safeTransfer` step. Because Solidity reverts roll back all state changes, the LP's shares are never burned and remain on-chain indefinitely. The LP retains a share balance that correctly entitles them to underlying tokens, but can never execute the transfer. This constitutes a direct, permanent loss of user principal â€” matching the contest's Critical/High allowed impact gate for "direct loss of user principal" and "broken core pool functionality causing loss of funds or unusable withdraw flow."

## Likelihood Explanation
USDC and USDT both maintain on-chain blacklists enforced at the token level, and both are explicitly in scope per contest rules. Blacklisting occurs for regulatory compliance, sanctions enforcement, or fraud response. Any LP who deposits into a USDC/USDT pool and is subsequently blacklisted faces permanent loss. No attacker is required; the condition is triggered by a standard, documented feature of the in-scope tokens. The condition is irreversible once triggered.

## Recommendation
1. Add an optional `recipient` parameter to `removeLiquidity` so the owner can redirect proceeds to a non-blacklisted address. Replace `safeTransfer(owner, ...)` with `safeTransfer(recipient, ...)` inside `LiquidityLib.removeLiquidity`.
2. Add an admin emergency rescue function (ideally behind a timelock/multisig) that can transfer any token balance in excess of `binTotals` accounting to a specified address.

## Proof of Concept
1. Pool is deployed with `token0 = USDC`, `token1 = WETH`.
2. Alice calls `addLiquidity` via a router callback, depositing USDC into bin `+4`. Her position is recorded: `positionBinShares[keccak256(alice, salt, 4)] = 10_000`.
3. USDC's issuer blacklists Alice's address.
4. Alice calls `removeLiquidity(alice, salt, deltas, "")`.
5. `MetricOmmPool` passes `msg.sender == alice == owner` check (line 206).
6. `LiquidityLib.removeLiquidity` computes `amount0Removed > 0`, then executes `IERC20(USDC).safeTransfer(alice, amount0Removed)` (line 243).
7. USDC's `transfer` reverts because Alice is blacklisted. The entire transaction reverts, including all share-burn state changes.
8. Alice's shares remain at `10_000`. Every retry produces the same revert.
9. No other address can call `removeLiquidity` for Alice (`NotPositionOwner` revert at line 206).
10. Alice's USDC principal is permanently locked in the pool.
