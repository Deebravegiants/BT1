### Title
Missing On-Chain CKD Response Validation for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Forged Confidential Key - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` performs a cryptographic pairing check (`ckd_output_check`) on the `CKDResponse` only when the request used the `AppPublicKeyPV` variant. For the `AppPublicKey` (legacy) variant, the branch is an explicit no-op. A single attested participant — strictly below the signing threshold — can therefore call `respond_ckd` with completely arbitrary `big_y` / `big_c` values, resolve the pending yield, and deliver a forged confidential key to the requesting user without any threshold cooperation.

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` guards the response with a match on the request's `app_public_key` field:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no validation
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically proves the response is the correct output of the threshold CKD protocol. [2](#0-1) 

For `AppPublicKey`, the arm is empty. Any `CKDResponse { big_y, big_c }` — including all-zero or all-one byte arrays — is accepted and forwarded to the user via `resolve_yields_for`. [3](#0-2) 

The existing unit test for the `AppPublicKey` path explicitly uses `[1u8; 48]` / `[2u8; 48]` as the response and expects success, confirming the absence of any check: [4](#0-3) 

By contrast, `respond` (ECDSA/EdDSA) always verifies the signature against the derived public key before resolving: [5](#0-4) 

### Impact Explanation

The `AppPublicKey` variant is the legacy format and is actively used in production — it is the default in node tests, e2e tests, and the contract README: [6](#0-5) 

A single Byzantine attested participant can:
1. Observe any pending `AppPublicKey` CKD request in `pending_ckd_requests`.
2. Call `respond_ckd` with a chosen `CKDResponse` (e.g., a key pair the attacker controls).
3. The contract resolves the yield and delivers the forged key to the user.

The user receives a `CKDResponse` whose `big_c` encodes a key the attacker knows. Any secret derived from it is compromised. This is **unauthorized confidential key derivation output without the required participant authorization** — a single participant below threshold bypasses the threshold CKD protocol entirely.

### Likelihood Explanation

- `AppPublicKey` is the legacy/default variant and is widely used.
- The attacker only needs to be a single attested participant in the active set — no threshold cooperation is required.
- The attack is a single on-chain transaction (`respond_ckd`) with no observable precondition beyond having a pending request.
- The contract provides no mechanism to detect or reject the forged response.

### Recommendation

For the `AppPublicKey` variant, on-chain verification is not possible without the G2 counterpart (`pk2`), which is why `AppPublicKeyPV` was introduced. The recommended mitigations are:

1. **Deprecate and remove `AppPublicKey`** in favour of `AppPublicKeyPV`, which supports the full on-chain pairing check.
2. If `AppPublicKey` must be retained for backwards compatibility, **require callers to upgrade to `AppPublicKeyPV`** for new requests, and document that existing `AppPublicKey` responses carry no on-chain integrity guarantee.
3. Alternatively, add a **threshold-signature-over-the-response** scheme so that the contract can verify that at least `t` participants agreed on the `(big_y, big_c)` values before accepting the response.

### Proof of Concept

```
1. Alice submits request_app_private_key({ app_public_key: AppPublicKey(pk1), ... })
   → contract stores pending CKD request, yields.

2. Byzantine participant Bob (single node, below threshold) calls:
   respond_ckd(
     request = <Alice's CKDRequest>,
     response = CKDResponse {
       big_y: Bls12381G1PublicKey([0xAA; 48]),  // attacker-chosen
       big_c: Bls12381G1PublicKey([0xBB; 48]),  // attacker-chosen
     }
   )

3. Contract executes AppPublicKey branch → no-op, no validation.
   resolve_yields_for resolves Alice's yield with the forged response.

4. Alice receives CKDResponse { big_y: [0xAA;48], big_c: [0xBB;48] }.
   Bob knows the discrete log of this key; Alice's derived secret is compromised.
```

The test at `crates/contract/src/lib.rs:3403–3441` already demonstrates this path succeeds with arbitrary byte arrays, confirming the root cause. [7](#0-6)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
