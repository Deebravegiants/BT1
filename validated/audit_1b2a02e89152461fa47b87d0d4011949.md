### Summary

In the Dijkstra era, the minimum fee calculation for a top-level transaction uses `txNonDistinctRefScriptsSize`, which only sums reference script bytes from the top-level transaction's inputs. However, the per-transaction validation guard (`validateAllRefScriptSize`) correctly uses `batchNonDistinctRefScriptsSize`, which sums reference script bytes across the top-level transaction **and all its subtransactions**. The fee charged to the submitter therefore does not account for the reference scripts loaded by subtransactions, allowing an attacker to submit transactions whose actual validation cost (deserialization of large reference scripts) far exceeds what the tiered fee formula charges.

---

### Finding Description

The Dijkstra era introduces nested ("sub") transactions. Two separate functions compute reference-script byte totals:

**`txNonDistinctRefScriptsSize`** (Conway, top-level only): [1](#0-0) 

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

**`batchNonDistinctRefScriptsSize`** (Dijkstra, top-level + subtransactions): [2](#0-1) 

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum (foldMap' (Sum . txNonDistinctRefScriptsSize utxo)
                       (tx ^. bodyTxL . subTransactionsTxBodyL))
```

The Dijkstra `EraUTxO` instance binds `getMinFeeTxUtxo` to the Conway implementation, which calls only `txNonDistinctRefScriptsSize`: [3](#0-2) [4](#0-3) 

```haskell
-- Dijkstra instance:
getMinFeeTxUtxo = getConwayMinFeeTxUtxo

-- Conway implementation:
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

The validation guard, however, correctly uses `batchNonDistinctRefScriptsSize`: [5](#0-4) 

```haskell
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $ ...
```

The result is a structural mismatch: the **size limit** is enforced on the full batch (top-level + subtransactions), but the **fee** is computed only on the top-level transaction. An attacker can place all large reference scripts exclusively in subtransactions, paying a fee as if no reference scripts were present while still loading up to `maxRefScriptSizePerTx` bytes of scripts for the node to deserialize.

---

### Impact Explanation

The tiered reference-script fee (`tierRefScriptFee`) was introduced specifically to deter the DDoS attack that occurred on June 25, 2024, where cheap transactions forced expensive script deserialization on every validating node. [6](#0-5) 

By routing all reference scripts through subtransactions, an attacker bypasses the tiered pricing entirely. The fee paid equals the base linear fee (no reference-script surcharge), while the node must still deserialize up to `maxRefScriptSizePerTx` bytes of scripts per transaction. This allows attacker-controlled transactions to modify the effective fee outside design parameters, matching the **Medium** impact: *"Attacker-controlled transactions... modify fees... outside design parameters."*

---

### Likelihood Explanation

The Dijkstra era is experimental but is present in the production repository and is the declared next era after Conway. Any node operator running a Dijkstra-era chain is immediately exposed. The attack requires only the ability to submit a valid transaction with subtransactions â€” no privileged access, no key compromise, no governance majority. The construction is straightforward: place large reference scripts in subtransaction inputs, leave the top-level transaction's input/reference-input sets empty of reference scripts.

---

### Recommendation

Replace the `getMinFeeTxUtxo` binding in the Dijkstra `EraUTxO` instance with a new implementation that calls `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
-- In Cardano.Ledger.Dijkstra.UTxO
getDijkstraMinFeeTxUtxo :: ... => PParams era -> Tx TopTx era -> UTxO era -> Coin
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx

instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo
```

This ensures the fee charged to the submitter reflects the full reference-script deserialization cost imposed on the network, consistent with the invariant already enforced by `validateAllRefScriptSize`.

### Proof of Concept

1. Obtain a UTxO containing outputs with large Plutus reference scripts (e.g., near `maxRefScriptSizePerTx` bytes total).
2. Construct a Dijkstra top-level transaction whose `subTransactionsTxBodyL` contains subtransactions that reference those UTxO outputs as inputs or reference inputs.
3. Leave the top-level transaction's own `inputsTxBodyL` and `referenceInputsTxBodyL` pointing only to outputs with no reference scripts.
4. Compute `getMinFeeTxUtxo pp tx utxo` â€” it returns the base fee with zero reference-script surcharge, because `txNonDistinctRefScriptsSize` sees no reference scripts in the top-level inputs.
5. Submit the transaction with that fee. `validateAllRefScriptSize` passes (total batch size â‰¤ `maxRefScriptSizePerTx`), and `feesOK` passes (fee â‰Ą computed minimum).
6. The node must deserialize all subtransaction reference scripts at validation time, incurring the full deserialization cost, while the submitter paid only the base fee â€” the reference-script overhead exceeds what the fee formula intended to charge. [7](#0-6) [8](#0-7) [9](#0-8)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
