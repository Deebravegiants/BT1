### Title
Unguarded `fee_proposal_fri` During Startup Window Allows Arbitrary L2 Gas Price Seeding — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

During the first `fee_proposal_window_size` blocks (currently 10 blocks across all deployed versioned-constants files), the validator explicitly skips the `fee_proposal_fri` bounds check because `fee_actual` is `None`. A malicious proposer who controls the first `window_size` blocks — or who proposes during the startup/restart window — can inject arbitrarily large or small `fee_proposal_fri` values that are accepted without any range enforcement. Those values are then committed to storage, recorded into `fee_proposals_window`, and used as the sole input to `compute_fee_actual` (a median) for all subsequent blocks. Once the window fills, `fee_actual` is permanently seeded at the attacker's chosen value, and every future `fee_proposal` is bounded relative to that poisoned baseline.

---

### Finding Description

**The bypass:**

In `is_proposal_init_valid` (`validate_proposal.rs`, lines 396–416), the fee-proposal bounds check is wrapped in a conditional that fires only when both `fee_actual` and `fee_proposal_fri` are `Some`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    ...
}
```

`fee_actual` is `None` whenever `compute_fee_actual` returns `None`, which happens when `height < window_size` (10) or when any entry in the window is missing. [1](#0-0) 

`compute_fee_actual` returns `None` for `height < window_size`:

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ({window_size})");
    return None;
};
``` [2](#0-1) 

**The poisoning path:**

After consensus decides on a block, `finalize_decision` unconditionally records the proposer's `fee_proposal_fri` from the `ProposalInit` into the in-memory `fee_proposals_window`:

```rust
self.update_l2_gas_price(height, l2_gas_used);
self.record_fee_proposal(height, init.fee_proposal_fri);
``` [3](#0-2) 

`record_fee_proposal` simply inserts the value without any validation:

```rust
fn record_fee_proposal(&mut self, height: BlockNumber, fee_proposal_fri: Option<GasPrice>) {
    self.fee_proposals_window.insert(height, fee_proposal_fri);
}
``` [4](#0-3) 

The value is also persisted to `BlockHeaderWithoutHash.fee_proposal_fri` in storage and re-read by `initialize_fee_proposals_window` on restart: [5](#0-4) [6](#0-5) 

**The window size is 10 blocks across all deployed versions:** [7](#0-6) 

**The margin is only 2 parts-per-thousand (0.2%) per block:** [8](#0-7) 

Once the window is filled with poisoned values, the attacker's chosen `fee_actual` becomes the anchor for all future proposals. With a 0.2% per-block margin, moving the fee away from the poisoned value takes ~800–1400 blocks (as confirmed by the convergence test). [9](#0-8) 

---

### Impact Explanation

This is a **High** impact finding under the "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact" category.

- An attacker who controls the proposer role for the first 10 blocks (genesis, network restart, or after a reorg that resets the window) can set `fee_proposal_fri` to any value — e.g., `u128::MAX` or `1` — in each of those blocks.
- After 10 such blocks, `compute_fee_actual` returns the median of those poisoned values as `fee_actual`.
- All subsequent honest proposers are bounded to within ±0.2% of that poisoned `fee_actual` per block.
- The L2 gas price charged to all users for hundreds of subsequent blocks is wrong: either artificially inflated (extracting excess fees from users) or artificially deflated (making the network economically unviable / enabling spam).
- The `fee_proposal_fri` is also forwarded to the centralized cende pipeline and stored in `BlockHeaderWithoutHash`, so the poisoned value propagates to the L1 anchoring and proving pipeline. [10](#0-9) 

---

### Likelihood Explanation

- The window is only 10 blocks. At Starknet's target block time of 1–2 seconds, this is 10–20 seconds of exposure at every network genesis or restart.
- The proposer for each height is determined by the committee. If the attacker controls even one validator seat, they can propose during their assigned heights in the startup window.
- The attack is also reachable after a node restart: `initialize_fee_proposals_window` re-reads `fee_proposal_fri` from storage, but if the stored values were already poisoned (from a prior attack or from a chain that had no bounds enforcement), the window is re-seeded with the poisoned values.
- No special privilege beyond being a valid proposer is required.

---

### Recommendation

1. **Enforce a hard cap on `fee_proposal_fri` during the startup window.** When `fee_actual` is `None`, validate that `fee_proposal_fri` is within a reasonable absolute range (e.g., `[min_gas_price, some_max_startup_price]`) rather than skipping all validation.

2. **Alternatively, use the proposer's own `l2_gas_price` as the reference during startup.** The proposer already computes `fee_proposal` as `self.l2_gas_price` when `fee_actual` is `None` (see `compute_proposer_fee_proposal`). The validator should enforce the same: `fee_proposal_fri` must equal the locally-computed `l2_gas_price` (within a small tolerance) during the startup window. [11](#0-10) 

3. **Bound the startup window values against `min_gas_price`.** At minimum, reject any `fee_proposal_fri` below `VersionedConstants::latest_constants().min_gas_price` even when `fee_actual` is `None`.

---

### Proof of Concept

**Setup:** Network at genesis (height 0–9). Attacker controls the proposer slot for heights 0–9 (or any subset).

**Step 1:** Attacker sends `ProposalInit` with `fee_proposal_fri = Some(GasPrice(u128::MAX))` for heights 0–9.

**Step 2:** `is_proposal_init_valid` is called. At height 0–9, `compute_fee_actual` returns `None` because `height < window_size (10)`. The bounds check at lines 398–416 is skipped entirely. The proposal passes validation. [12](#0-11) 

**Step 3:** Consensus decides on the block. `finalize_decision` calls `record_fee_proposal(height, Some(GasPrice(u128::MAX)))`. The poisoned value is stored in `fee_proposals_window` and in `BlockHeaderWithoutHash.fee_proposal_fri`. [3](#0-2) 

**Step 4:** At height 10, `compute_fee_actual` now has a full window of 10 entries, all `u128::MAX`. It returns `Some(GasPrice(u128::MAX))` as `fee_actual`. [13](#0-12) 

**Step 5:** All future proposals must have `fee_proposal_fri` within 0.2% of `u128::MAX`. The L2 gas price for all users is permanently set near `u128::MAX` until ~800–1400 blocks of honest proposals gradually move it back — during which time every transaction is charged an astronomically wrong fee.
