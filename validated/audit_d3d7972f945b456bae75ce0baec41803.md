Audit Report

## Title
Stale `rsETHPrice` Cache in Deposit Minting Dilutes Existing Holders' Yield - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides a live Chainlink asset price by `lrtOracle.rsETHPrice()`, a state variable that is only written when `_updateRsETHPrice()` is explicitly called. As staking rewards accrue and the cached price falls behind the true protocol value, any depositor receives excess rsETH, diluting the yield of all existing holders once the price is eventually refreshed.

## Finding Description
`rsETHPrice` is declared as a plain storage variable in `LRTOracle` and is written only at the end of `_updateRsETHPrice()`:

```solidity
rsETHPrice = newRsETHPrice;  // LRTOracle.sol L313
```

The mint formula in `getRsETHAmountToMint` (LRTDepositPool.sol L520) uses this cached value as the denominator:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`getAssetPrice` (LRTOracle.sol L156-158) reads a live price from `IPriceFetcher` on every call, while `rsETHPrice` reflects the last explicit update. The deposit path (`depositETH` → `_beforeDeposit` → `getRsETHAmountToMint`) is entirely `view`/`nonReentrant` and never triggers a price refresh.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (LRTOracle.sol L260-265) reverts permissionless callers when the price increase exceeds the configured threshold, meaning that during high-reward periods the only entity that can update the price is a MANAGER. This extends the staleness window precisely when the discrepancy is largest.

## Impact Explanation
When `rsETHPrice` is stale and lower than the true current value, the denominator is artificially small, so the depositor receives more rsETH than their deposit is worth. When `updateRSETHPrice()` is eventually called, the new price is computed as `totalETHInProtocol / rsethSupply`; the inflated supply lowers the per-token price, directly reducing the redemption value of every existing holder's position. This is **theft of unclaimed yield** (High severity) from existing rsETH holders.

## Likelihood Explanation
- `updateRSETHPrice()` is a separate call with no on-chain enforcement that it must precede deposits.
- Staking rewards and LST appreciation accrue continuously, so the stale window opens naturally between keeper calls.
- The `pricePercentageLimit` guard (LRTOracle.sol L256-265) can block permissionless updates during high-reward periods, widening the window.
- Any unprivileged depositor can compare the on-chain `rsETHPrice` against a locally simulated `_getTotalEthInProtocol() / rsethSupply` to confirm and time the attack.
- No special privileges, flash loans, or external protocol compromise are required.

## Recommendation
Force a price refresh before computing the mint amount. Either:
1. Call `lrtOracle.updateRSETHPrice()` at the start of `depositETH` / `depositAsset` (requires making `_beforeDeposit` non-`view`), or
2. Expose a pure/view helper in `LRTOracle` that computes `_getTotalEthInProtocol() / rsethSupply` without writing state, and use that value instead of `rsETHPrice` in `getRsETHAmountToMint`.

## Proof of Concept
1. Staking rewards accrue; `_getTotalEthInProtocol()` grows so the true rsETH price is 1.001 ETH, but `rsETHPrice` is still stored as 1.000 ETH.
2. `updateRSETHPrice()` has not been called (or is blocked by `pricePercentageLimit` for non-managers).
3. Attacker calls `depositETH(minRSETHAmountExpected, "")` with 100 ETH.
4. `getRsETHAmountToMint` computes: `(100e18 * 1e18) / 1.000e18 = 100 rsETH` instead of the correct `≈ 99.9 rsETH`.
5. Attacker receives ~0.1 rsETH excess extracted from existing holders' accrued yield.
6. When `updateRSETHPrice()` is eventually called, the inflated supply lowers the price for all holders.

**Foundry fork test plan**: Fork mainnet, advance time to accumulate rewards, assert `_getTotalEthInProtocol() / rsethSupply > rsETHPrice`, deposit as an unprivileged address, assert minted rsETH exceeds `deposit * livePrice / liveRsETHPrice`, then call `updateRSETHPrice()` and assert the per-token price is lower than it would have been without the stale-price deposit.
