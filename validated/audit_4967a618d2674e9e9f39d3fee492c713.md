### Title
Epoch-layout mismatch in `spice_shard_congestion_info` silently drops congestion check across resharding boundary, admitting transactions to congested child shards — (`chain/client/src/rpc_handler.rs`)

### Summary

In `process_tx_internal`, when Spice is active and the last certified block header is in epoch N while the current head is in epoch N+1 (after resharding), the `shard_layout` from epoch N+1 is passed to `spice_shard_congestion_info` together with the epoch-N certified header. Inside that function, the `ShardUId` is constructed from the epoch-N+1 layout version, but the `ChunkExtra` lookup targets an epoch-N block hash. Both the primary lookup (wrong `ShardUId` version) and the fallback lookup (child `shard_id` that never existed in epoch N) return nothing. The function returns `None`. `validate_tx` treats `None` as "no congestion data available" and skips the congestion gate entirely, admitting the transaction unconditionally.

### Finding Description

**Step 1 — epoch mismatch is set up in `process_tx_internal`:** [1](#0-0) 

`epoch_id` and `shard_layout` are derived from `head.last_block_hash` (epoch N+1). `receiver_shard` is a child shard ID that only exists in epoch N+1's layout. `certified_header` is obtained from `get_last_certified_block_header`, which can legitimately return an epoch-N header when execution lags behind the resharding boundary. [2](#0-1) 

The certified header is the predecessor of the oldest uncertified block; under normal execution delay this is routinely in the previous epoch immediately after a resharding event, as confirmed by the existing regression test: [3](#0-2) 

**Step 2 — `spice_shard_congestion_info` receives mismatched arguments:** [4](#0-3) 

- Line 273: `ShardUId::from_shard_id_and_layout(child_shard_id, epoch_N+1_layout)` — produces a `ShardUId` with the epoch-N+1 version tag.
- Line 276: `get_chunk_extra(epoch_N_block_hash, &shard_uid_N+1)` — fails because the `ChunkExtra` stored for the epoch-N block was keyed with the epoch-N `ShardUId` version.
- Line 279: `get_execution_result_from_store(chain_store, epoch_N_block_hash, child_shard_id)` — also fails because the child shard did not exist in epoch N; no execution result was ever written for `(epoch_N_block_hash, child_shard_id)`.
- The function returns `None`.

**Contrast with the correct sibling function** `spice_block_congestion_info`, which always derives the layout from the block header's own epoch: [5](#0-4) 

`process_tx_internal` bypasses this epoch-aware derivation by passing the current-epoch layout directly.

**Step 3 — `validate_tx` skips the congestion gate when `receiver_congestion_info` is `None`:** [6](#0-5) 

The entire `CongestionControl::shard_accepts_transactions()` check is inside the `if let Some(...)` arm. A `None` result causes the function to fall through to `validate_transaction`, which performs only signature/nonce/balance checks — no congestion enforcement.

### Impact Explanation

During the execution-lag window that follows every resharding boundary (epoch N → N+1), any unprivileged user can submit transactions whose receiver maps to a child shard. The RPC node will admit those transactions regardless of how congested the child shard is, because `receiver_congestion_info` is always `None` in this window. Congestion control — the mechanism that prevents flooding of overloaded shards — is completely bypassed for the duration of the lag. An attacker can exploit this window to inject a burst of transactions into a congested child shard, worsening congestion and delaying legitimate traffic.

### Likelihood Explanation

The preconditions are all normal protocol behavior:
- Spice is a planned production feature.
- Resharding is a planned production event.
- Execution lag (certified header trailing the head by several blocks) is the expected steady-state in Spice, not an edge case.

The existing test `test_spice_certified_results_across_resharding` explicitly confirms that the first blocks of a resharded epoch routinely have their certified header in the previous epoch. The attacker needs no special privileges — only the ability to submit a transaction via the public RPC at the right moment.

### Recommendation

In `process_tx_internal`, derive the shard layout for `spice_shard_congestion_info` from the certified header's epoch, not from the current head's epoch:

```rust
// Instead of:
spice_shard_congestion_info(&self.chain_store, &shard_layout, certified_header.as_ref(), receiver_shard)

// Use:
let certified_epoch_id = certified_header.epoch_id();
let certified_shard_layout = self.epoch_manager.get_shard_layout(certified_epoch_id)?;
let certified_receiver_shard = certified_shard_layout.account_id_to_shard_id(receiver_id);
spice_shard_congestion_info(&self.chain_store, &certified_shard_layout, certified_header.as_ref(), certified_receiver_shard)
```

This mirrors the pattern already used correctly in `spice_block_congestion_info`.

### Proof of Concept

1. Configure a Spice-enabled network with a resharding boundary (epoch N → N+1, e.g. `ShardLayout::multi_shard(1, v)` → two shards).
2. Introduce an endorsement/execution delay of ≥2 blocks (as in the existing resharding test).
3. At the first block of epoch N+1, assert that `get_last_certified_block_header` returns an epoch-N header.
4. Congest a child shard (e.g. by filling its delayed-receipt queue above the `reject_tx_congestion_threshold`).
5. Submit a transaction via RPC whose receiver maps to that child shard under epoch N+1's layout.
6. Assert `ProcessTxResponse::ValidTx` is returned instead of `ProcessTxResponse::InvalidTx(ShardCongested{..})`.
7. Verify that `spice_shard_congestion_info` returned `None` by tracing the `ShardUId` version mismatch and the absent child-shard execution result in the epoch-N store.

/// Returns the header of the last fully certified block relative to the given block.
/// All chunks in blocks at or below this height have been certified.
pub fn get_last_certified_block_header(
    chain_store: &ChainStoreAdapter,
    block_hash: &CryptoHash,
) -> Result<Arc<BlockHeader>, Error> {
    let uncertified_chunks = get_uncertified_chunks(chain_store, block_hash)?;
    let oldest_uncertified = find_oldest_uncertified_block_header(chain_store, uncertified_chunks)?;
    if let Some(header) = oldest_uncertified {
        Ok(chain_store.get_block_header(header.prev_hash())?)
    } else {
        // No uncertified-chunks tracking means the block has nothing to
        // certify: genesis, or a pre-spice block at the activation
        // boundary. Both are fully certified by definition.
        let header = chain_store.get_block_header(block_hash)?;
        debug_assert!(
            header.is_genesis() || !header.is_spice(),
            "post-genesis spice blocks should always have uncertified chunks"
        );
        Ok(header)
    }
}

    let mut result = BTreeMap::new();

        }

pub fn spice_shard_congestion_info(
    chain_store: &ChainStoreAdapter,
    shard_layout: &ShardLayout,
    block_header: &BlockHeader,
    shard_id: ShardId,
) -> Option<ExtendedCongestionInfo> {
    let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, shard_layout);
    let chunk_store = chain_store.chunk_store();
    let congestion_info =
        if let Ok(chunk_extra) = chunk_store.get_chunk_extra(block_header.hash(), &shard_uid) {
            chunk_extra.congestion_info()
        } else {
            get_execution_result_from_store(chain_store, block_header.hash(), shard_id)?
                .chunk_extra
                .congestion_info()
        };
    Some(ExtendedCongestionInfo::new(congestion_info, 0))
}
