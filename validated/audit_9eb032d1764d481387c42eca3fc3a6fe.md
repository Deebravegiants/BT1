Audit Report

## Title
Missing Caller Authorization in `MerkleDistributor.claim()` Allows Forced Claims with Fee Deduction - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` accepts an arbitrary `account` parameter but never verifies `msg.sender == account`. Any caller with a valid Merkle proof for a target user can force that user's claim, permanently consuming their claimable index while deducting up to 10% as a protocol fee. The sibling contract `KernelMerkleDistributor` already enforces this check, confirming the pattern is known but was not applied consistently.

## Finding Description
In `MerkleDistributor.claim()`, there is no authorization check on `msg.sender` relative to `account`. The function verifies the Merkle proof against `(index, account, cumulativeAmount)`, updates `userClaims[account]` permanently, transfers `claimableAmount - fee` to `account`, and transfers `fee` to `protocolTreasury` — all callable by any external address. [1](#0-0) 

The fee is owner-configurable up to `MAX_FEE_IN_BPS = 1000` (10%) and is applied unconditionally at claim time: [2](#0-1) 

The claim state is permanently updated before the transfer, making re-claiming impossible: [3](#0-2) 

By contrast, `KernelMerkleDistributor._processClaim()` explicitly enforces the caller check at lines 311–313: [4](#0-3) 

Merkle proof data is published off-chain and publicly accessible, so no privileged access is required to obtain a valid proof for any target address.

## Impact Explanation
**High — Theft of unclaimed yield.** A user who is waiting for `feeInBPS` to be reduced (e.g., from 1000 to 0) before claiming can have their claim forced by an attacker while the fee is still at maximum. The user permanently loses up to 10% of their entire allocation — the exact amount they could have received fee-free — because `userClaims[account]` is updated irreversibly and `AlreadyClaimed` will revert any subsequent attempt. [5](#0-4) 

## Likelihood Explanation
- Merkle proof data is public; no capital, permissions, or privileged access is required.
- `feeInBPS` is non-zero in normal protocol operation and can be set up to 10% by the owner at any time via `setFeeInBPS()`. [6](#0-5) 
- The scenario where a user delays claiming in anticipation of a fee reduction is realistic and incentivized.
- The attack is repeatable against any unclaimed address with a known proof, requires a single transaction, and costs only gas.

## Recommendation
Add a caller authorization check matching the pattern already implemented in `KernelMerkleDistributor`:

```diff
+ error Unauthorized();

 function claim(
     uint256 index,
     address account,
     uint256 cumulativeAmount,
     bytes32[] calldata merkleProof
 ) external override whenNotPaused {
+    if (msg.sender != account) revert Unauthorized();
     // ... rest of function unchanged
 }
```

## Proof of Concept
```solidity
function test_forceClaim_stealsYieldViaFeeReduction() public {
    address alice = address(0xA11CE);
    address attacker = address(0xBAD);

    // Setup: feeInBPS = 1000 (10%), alice has 100e18 claimable, waiting for fee to drop

    uint256 aliceBalanceBefore = token.balanceOf(alice);

    // Attacker obtains alice's public merkle proof and forces her claim
    vm.prank(attacker);
    merkleDistributor.claim(index, alice, 100e18, aliceProof);

    // Alice receives only 90e18; 10e18 goes to protocolTreasury
    assertEq(token.balanceOf(alice), aliceBalanceBefore + 90e18);

    // Owner later reduces fee to 0%
    vm.prank(owner);
    merkleDistributor.setFeeInBPS(0);

    // Alice tries to claim — reverts because state was already updated
    vm.expectRevert(IMerkleDistributor.AlreadyClaimed.selector);
    vm.prank(alice);
    merkleDistributor.claim(index, alice, 100e18, aliceProof);

    // Alice permanently lost 10e18 (10% fee) she could have received fee-free
}
```
