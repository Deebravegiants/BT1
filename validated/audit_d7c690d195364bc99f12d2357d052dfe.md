### Title
Fan-out Queue Saturation via Caller-Agnostic `verify_foreign_transaction` Key Enables Theft of MPC-Signed Bridge Response — (`crates/contract/src/pending_requests.rs`)

---

### Summary

An unprivileged attacker can pre-saturate the `pending_verify_foreign_tx_requests` fan-out queue for any specific foreign-chain transaction by submitting 128 identical `verify_foreign_transaction` calls at negligible cost. This causes the legitimate caller's subsequent submission to panic and fail, while the attacker receives all copies of the MPC-signed verification response. Because the response signature is issued against the MPC root key with no caller-specific tweak, the attacker can present it to any downstream bridge contract to claim the victim's funds.

---

### Finding Description

**Root cause — caller-agnostic request key**

`verify_foreign_transaction` converts its arguments into a `VerifyForeignTransactionRequest` that contains only `domain_id`, `payload_version`, and the raw `ForeignChainRpcRequest` (e.g., Bitcoin `tx_id`). The caller's account ID is never mixed into the key. [1](#0-0) 

Compare this with `sign`, where `SignatureRequest::new` derives a `tweak` from `(predecessor_id, path)`, making every caller's request key unique: [2](#0-1) 

The `verify_foreign_transaction` path has no equivalent binding. The test itself documents this property: [3](#0-2) 

**Root cause — hard queue cap with panic on overflow**

`push_pending_yield` enforces `MAX_PENDING_REQUEST_FAN_OUT = 128`. When the queue is full it calls `env::panic_str`, which reverts the entire transaction: [4](#0-3) 

**Root cause — response signature carries no caller binding**

`respond_verify_foreign_tx` verifies the MPC signature against the **root** public key with no per-caller tweak: [5](#0-4) 

This is structurally different from `respond`, which verifies against a key derived from the original requester's tweak. The foreign-tx signed response is therefore redeemable by anyone who holds it.

**Attack flow**

1. Attacker monitors the foreign chain (e.g., Bitcoin) for a victim's inbound bridge deposit transaction `tx_id = X`.
2. Before the victim calls `verify_foreign_transaction`, the attacker submits 128 identical calls with the same `(domain_id, payload_version, tx_id = X)`.
3. Each call costs `MINIMUM_SIGN_REQUEST_DEPOSIT` (1 yoctoNEAR in tests), so the total cost is ~128 yoctoNEAR ≈ $0.
4. The victim's call arrives; `push_pending_yield` finds `queue.len() == 128` and panics — the victim's transaction is reverted.
5. MPC nodes call `respond_verify_foreign_tx`; `resolve_yields_for` drains all 128 attacker yields, delivering the signed response to each: [6](#0-5) 

6. The attacker presents the signed `VerifyForeignTransactionResponse` to the downstream bridge contract to claim the victim's funds.

---

### Impact Explanation

The `VerifyForeignTransactionResponse` is an ECDSA signature over a `ForeignTxSignPayload` derived solely from the foreign-chain transaction data. It carries no caller identity. Any party holding this signature can present it to a bridge contract to prove the MPC network verified the transaction and unlock the corresponding funds. The victim, whose call was reverted, holds nothing and cannot retry until the attacker's 128 yields time out (~200 blocks). This constitutes direct theft of funds controlled by the MPC-verified bridge flow, matching the **High** allowed impact: *"Cross-chain replay, forged foreign-chain verification… that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- The minimum deposit is 1 yoctoNEAR per call; 128 calls cost ~128 yoctoNEAR (effectively free).
- The attacker does not need to observe the victim's NEAR transaction; they only need to observe the foreign-chain deposit (e.g., a Bitcoin UTXO sent to the bridge address), which is public.
- Submitting 128 calls can be batched into a small number of NEAR transactions, all of which land before the victim's single call.
- No privileged access, key material, or threshold collusion is required.

---

### Recommendation

**Primary fix**: Include the caller's account ID in the `VerifyForeignTransactionRequest` key, exactly as `SignatureRequest` does via `tweak`. Derive a per-caller tweak from `(predecessor_id, derivation_path)` and store it in `VerifyForeignTransactionRequest`. This makes every caller's queue slot independent, eliminating the shared-key collision surface.

**Secondary fix**: Verify the response signature against the caller-derived key (as `respond` does), so a stolen response cannot be redeemed by a different account.

**Tertiary mitigation**: Require a meaningful economic deposit (e.g., proportional to the bridged value) to make queue-saturation attacks economically irrational.

                .pending_signature_requests
                .get(&signature_request)
                .map(|q| q.len()),
            Some(usize::from(MAX_PENDING_REQUEST_FAN_OUT)),

        // When: one more append is attempted.
        let result = panic::catch_unwind(panic::AssertUnwindSafe(|| {
            contract.add_signature_request(signature_request.clone(), [0xff; 32]);

        let err = result.expect_err("appending past the cap should panic");
        let msg = err
            .downcast_ref::<String>()
            .map(String::as_str)
            .or_else(|| err.downcast_ref::<&str>().copied())

        assert!(
            msg.contains("Pending-request queue is full"),
            "unexpected panic message: {msg}",

        assert_eq!(
            contract
                .pending_signature_requests
                .get(&signature_request)
                .map(|q| q.len()),
            Some(usize::from(MAX_PENDING_REQUEST_FAN_OUT)),
            "queue should not have grown past the cap",
        );

pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
