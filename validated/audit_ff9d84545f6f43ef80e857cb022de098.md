Audit Report

## Title
Reward Tokens Permanently Frozen When `totalKernelStaked` Drops to Zero Mid-Period — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`initiateWithdrawal` contains no guard preventing `totalKernelStaked` from reaching zero during an active reward period. When this occurs, `rewardPerToken()` freezes at `rewardPerTokenStored`, and all reward tokens accruing for the remainder of the period (`rewardRate × (finishAt − T_zero)`) become permanently locked. Recovery via `notifyRewardAmount` is impossible because it measures only the transfer-in delta, ignoring pre-existing stuck balances.

## Finding Description

**Root cause — `rewardPerToken()` short-circuits on zero supply:** [1](#0-0) 

When `totalKernelStaked == 0`, the function returns the frozen `rewardPerTokenStored` regardless of elapsed time, meaning `rewardRate × elapsed` tokens accumulate in the contract with no accounting entry.

**`initiateWithdrawal` decrements `totalKernelStaked` with no active-period guard:** [2](#0-1) 

Any staker can call this at any time during an active reward period.

**`updateReward` snapshots state *before* the balance change:** [3](#0-2) 

When the last staker calls `initiateWithdrawal`:
1. `updateReward` runs: `rewardPerTokenStored` is updated to the current value, `updatedAt = block.timestamp`.
2. `totalKernelStaked -= amount` → `totalKernelStaked = 0`.
3. For all subsequent blocks until `finishAt`, `rewardPerToken()` returns the frozen `rewardPerTokenStored`.
4. When a new staker eventually calls `stake`, `updateReward` runs again with `totalKernelStaked` still 0 at modifier time (the increment happens after `_;`), so `rewardPerTokenStored` stays frozen and `updatedAt` silently advances past the entire gap period.

**Recovery via `notifyRewardAmount` is impossible — delta-only accounting:** [4](#0-3) 

The function computes `receivedAmount = balanceAfter - balanceBefore`. Tokens already stuck in the contract are not counted as new rewards and remain permanently locked with no withdrawal path.

**The contract's own NatDoc acknowledges this failure mode but provides no on-chain enforcement:** [5](#0-4) 

The `notifyRewardAmount` guard (`NoStakedTokens` revert at L570) only prevents *starting* a period with zero staked; it does not prevent stakers from withdrawing to zero *during* an active period. [6](#0-5) 

## Impact Explanation
Reward tokens equal to `rewardRate × (finishAt − T_zero)` are permanently locked in the contract. They cannot be claimed by any staker and cannot be recaptured by `notifyRewardAmount`. This matches **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
- No special role or privilege is required; any staker can call `initiateWithdrawal` at any time.
- A single large staker holding the majority of `totalKernelStaked` can trigger this unilaterally.
- Multiple stakers withdrawing independently for legitimate reasons (market conditions, risk management) can trigger it without coordination.
- The `withdrawalDelay` only delays the token transfer, not the `totalKernelStaked` decrement, so the freeze occurs at `initiateWithdrawal` time.

## Recommendation
Add an on-chain guard in `initiateWithdrawal` that prevents `totalKernelStaked` from reaching zero while a reward period is active:

```solidity
// After decrementing totalKernelStaked:
if (totalKernelStaked == 0 && block.timestamp < finishAt) {
    revert CannotLeaveZeroStakeDuringRewardPeriod();
}
```

Alternatively, add an admin-callable `recoverStuckRewards` function that sweeps the unaccounted balance after a period ends, or track `undistributedRewards` explicitly and roll them into the next period via `notifyRewardAmount`.

## Proof of Concept

```solidity
// 1. Admin sets duration; Alice stakes; admin notifies rewards
pool.setRewardsDuration(7 days);
kernelToken.approve(address(pool), 1000e18);
pool.stake(1000e18);                         // Alice stakes; totalKernelStaked = 1000e18
pool.notifyRewardAmount(700e18);             // rewardRate = 700e18 / 7 days

// 2. Mid-period: Alice initiates withdrawal (no guard prevents this)
vm.warp(block.timestamp + 1 days);          // 1 day into the 7-day period
pool.initiateWithdrawal(1000e18);           // totalKernelStaked → 0; freeze begins

// 3. Warp to end of period
vm.warp(pool.finishAt());

// 4. New staker stakes 1 wei — updateReward runs with totalKernelStaked==0,
//    rewardPerTokenStored stays frozen, updatedAt advances past the gap
kernelToken.approve(address(pool), 1);
pool.stake(1);

// 5. Assert: contract holds far more reward tokens than all earned() sums
uint256 contractBalance = rewardsToken.balanceOf(address(pool));
uint256 aliceEarned = pool.earned(alice);
// contractBalance ≈ 700e18 (all rewards), aliceEarned ≈ 100e18 (1 day only)
// ~600e18 permanently frozen, unclaimable by anyone
assert(contractBalance > aliceEarned);
```
