### Title
Unchecked Caller Ownership in `claim()` Allows Forced Fee Extraction on Any User's Allocation — (`File: contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

---

### Summary

`MerkleDistributor.claim()` accepts an arbitrary `account` parameter but never verifies that `msg.sender == account`. Any unprivileged caller can trigger a claim on behalf of any address, forcing the protocol fee (up to 10%) to be permanently extracted from the victim's claimable allocation. The sister contract `KernelMerkleDistributor` explicitly guards against this with an `account != msg.sender` revert; `MerkleDistributor` does not.

---

### Finding Description

`MerkleDistributor.claim()` is a public, permissionless function that accepts `account` as a caller-supplied parameter:

```solidity
function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
``` [1](#0-0) 

The function verifies the merkle proof against `account` and then transfers tokens:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);
``` [2](#0-1) 

There is **no check** that `msg.sender == account`. The entire function body from line 107 to 147 contains zero caller-identity validation. [3](#0-2) 

By contrast, `KernelMerkleDistributor._processClaim()` — the analogous function for KERNEL token distribution — explicitly enforces this invariant:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
``` [4](#0-3) 

The fee ceiling is 10% (`MAX_FEE_IN_BPS = 1000`): [5](#0-4) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When `feeInBPS > 0`, an attacker can call `claim(index, victimAddress, cumulativeAmount, proof)` for any address that has a valid merkle allocation. This forces the fee (up to 10% of the victim's claimable amount) to be permanently deducted and sent to `protocolTreasury` before the victim has chosen to claim. The victim's `userClaims` state is updated and marked as claimed at the current index, so the fee cannot be recovered.

The concrete theft scenario: the owner announces a fee reduction (e.g., from 10% to 0%). An attacker front-runs the `setFeeInBPS` transaction by calling `claim` for every eligible address at the current high fee rate. All victims permanently lose the fee difference — yield they were entitled to keep had they waited one block.

Even absent a fee change, the attacker permanently forces the fee extraction at a time of the attacker's choosing, not the victim's. The victim's unclaimed yield is reduced by up to 10% with no recourse.

---

### Likelihood Explanation

**High.** The entry path is fully permissionless — no role, no signature, no prior state required. Any address can call `claim` with any valid `(index, account, cumulativeAmount, merkleProof)` tuple. All inputs are public on-chain (merkle tree data is published off-chain but readable). The attack is a single transaction and costs only gas. Fee-reduction governance actions are observable in the mempool, making front-running straightforward.

---

### Recommendation

Add a caller check identical to the one already present in `KernelMerkleDistributor`:

```solidity
function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
+   if (account != msg.sender) revert Unauthorized();
    ...
}
``` [1](#0-0) 

---

### Proof of Concept

1. Alice has a valid merkle allocation of 1000 tokens at `index = 5`, `cumulativeAmount = 1000`. Current `feeInBPS = 1000` (10%).
2. Owner submits `setFeeInBPS(0)` to reduce the fee to zero.
3. Attacker observes the pending transaction in the mempool.
4. Attacker front-runs with: `MerkleDistributor.claim(5, alice, 1000, aliceProof)`.
5. The contract transfers 900 tokens to Alice and 100 tokens to `protocolTreasury`. Alice's `userClaims[alice].lastClaimedIndex` is set to 5.
6. Owner's `setFeeInBPS(0)` confirms. Alice attempts to claim — reverts with `AlreadyClaimed`.
7. Alice permanently loses 100 tokens (10% of her allocation) that she would have received in full had she claimed after the fee reduction. [6](#0-5)
