### Title
Missing CKD Output Verification for `AppPublicKey` Requests Enables Single-Participant Response Forgery - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` function in the MPC contract unconditionally skips on-chain verification of the Confidential Key Derivation (CKD) output when the pending request uses the legacy `AppPublicKey` format. A single attested MPC participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary `CKDResponse` for any pending `AppPublicKey` request and the contract will accept it, bypassing the threshold-signature requirement entirely.

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV` the contract calls `ckd_output_check`, which cryptographically verifies that the returned ciphertext is a correctly-formed encryption of the MPC-derived secret under the caller's ephemeral public key. For `AppPublicKey` (the legacy, privately-verifiable format) the arm is an empty no-op: the contract performs zero verification and immediately proceeds to `resolve_yields_for`, delivering whatever bytes the caller supplied to every waiting yield. [2](#0-1) 

The `AppPublicKey` format is still accepted by `request_app_private_key` and is documented as the supported legacy path: [3](#0-2) 

The only guard before the branch is `assert_caller_is_attested_participant_and_protocol_active()`, which requires the caller to be a single attested participant — not a threshold of them.

**Exploit path:**

1. Any user submits `request_app_private_key` with `AppPublicKey` (a single BLS12-381 G1 point). The request is stored in `pending_ckd_requests` under its key.
2. A single malicious attested MPC participant (the leader role, or any participant who races to call first) calls `respond_ckd` with an arbitrary `CKDResponse` — e.g., all-zero bytes, or a ciphertext they constructed themselves.
3. The contract's `AppPublicKey` arm executes the empty branch, skips all verification, and calls `resolve_yields_for`, which drains the entire fan-out queue and delivers the forged bytes to every waiting caller.
4. The pending request is removed from state; no legitimate MPC response can now be delivered. [4](#0-3) 

The analog to the external report is direct: just as `getRewardPerToken()` lacked a `block.timestamp >= periodFinish` guard and kept accumulating rewards beyond the intended boundary, `respond_ckd` lacks a CKD-output validity guard for the `AppPublicKey` branch and accepts any response beyond the intended threshold boundary.

### Impact Explanation

The threshold-signature requirement for CKD is completely bypassed for `AppPublicKey` requests. A single attested participant — one of potentially many, and strictly below the reconstruction threshold — can unilaterally deliver an unauthorized CKD output to the caller without any cooperation from the remaining participants. This satisfies the Critical impact criterion: **unauthorized confidential key derivation output without the required participant authorization**.

Concretely:
- The caller's application receives a ciphertext it cannot decrypt (denial of the legitimate derived key).
- If the attacker also controls the app's BLS private key (e.g., a compromised app server that is also an MPC participant), they can craft a well-formed ciphertext encrypting an attacker-chosen secret, causing the caller to silently use a key the attacker knows — enabling impersonation or decryption of data the caller subsequently encrypts with that key.
- The legitimate MPC-produced response can never be delivered after the forged one drains the queue, making the loss permanent for that request.

### Likelihood Explanation

`AppPublicKey` is the documented legacy format and remains fully supported. Any attested MPC participant who acts as the signing leader for a CKD request can trivially exploit this: instead of running the threshold protocol, they submit a forged response immediately. No collusion with other participants is required. The attack is reachable from a single Byzantine participant strictly below the threshold.

### Recommendation

Add CKD output verification for the `AppPublicKey` branch. If on-chain verification is impossible for the privately-verifiable format (because the app's private key is not known to the contract), the branch should at minimum enforce that the response has the correct ciphertext length and structure, or — preferably — deprecate `AppPublicKey` in favour of the publicly-verifiable `AppPublicKeyPV` format for which `ckd_output_check` already provides a full cryptographic guarantee.

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {
        // Enforce structural validity at minimum, or reject entirely
        // and require AppPublicKeyPV for new requests.
        env::panic_str("AppPublicKey format is deprecated; use AppPublicKeyPV");
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

### Submitting a confidential key derivation (ckd) request

Users can submit a ckd request to the MPC network via the
`request_app_private_key` endpoint of this contract. A **deposit of 1
yoctonear is required** (see [Deposit requirement](#deposit-requirement)).

The ckd request takes the following arguments:

- `derivation_path` (String): the derivation path (used to derive different keys from the same account).
- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
