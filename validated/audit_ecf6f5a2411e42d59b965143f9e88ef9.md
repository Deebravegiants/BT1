Audit Report

## Title
Asymmetric `priceLimitX64 = 0` Handling Silently Blocks All `zeroForOne = false` Swaps — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary

Passing `priceLimitX64 = 0` to `swap()` with `zeroForOne = false` causes both `_swapToken1ForToken0SpecifiedInput` and `_swapToken1ForToken0SpecifiedOutput` to return `(0, 0, 0)` immediately due to an initial guard that evaluates `0 <= initialPriceX64` as always true. The same sentinel `0` works correctly as "no lower bound" for `zeroForOne = true` swaps, creating a confirmed asymmetric silent failure. The transaction does not revert; it emits a `Swap` event with zero deltas and transfers nothing.

## Finding Description

**Root cause — `_swapToken1ForToken0SpecifiedInput`:** [1](#0-0) 

With `priceLimitX64 = 0` and any positive `initialPriceX64` (which is always the case for a live pool), `0 <= initialPriceX64` is unconditionally true, so the function returns `(0, 0, 0)` before touching any bin.

**Root cause — `_swapToken1ForToken0SpecifiedOutput`:** [2](#0-1) 

Identical guard, same unconditional early exit.

**Contrast — `_swapToken0ForToken1SpecifiedInput` (`zeroForOne = true`):** [3](#0-2) 

With `priceLimitX64 = 0`, `0 >= initialPriceX64` is false for any positive price, so the swap proceeds. The inner loop guard at line 1157 further reinforces this with an explicit `!= 0` check: [4](#0-3) 

The `zeroForOne = false` inner loop has the same `!= 0` guard at line 979, but it is never reached because the initial guard at line 970 already returned.

**Silent failure in `swap()`:**

When `_executeSwap` returns `(0, 0, 0)`, `amount0Delta` and `amount1Delta` are both zero. The `zeroForOne = false` branch skips the `transferToken0` call (since `amount0Delta < 0` is false), calls the callback with `(0, 0, callbackData)`, and the `IncorrectDelta` check passes trivially. A `Swap` event is emitted with zero deltas and the function returns `(0, 0)`. [5](#0-4) [6](#0-5) 

## Impact Explanation

Every `zeroForOne = false` swap with `priceLimitX64 = 0` silently executes as a no-op. The caller receives zero token0 output and pays zero token1 input. Any router or aggregator that symmetrically uses `0` as "no price limit" for both directions will have all `zeroForOne = false` swaps silently fail. This constitutes broken core swap functionality — a complete, unconditional denial of one of the two swap directions for a natural and undocumented-as-invalid input value.

## Likelihood Explanation

`priceLimitX64 = 0` is the natural "unlimited" sentinel for `zeroForOne = true` and is accepted without revert by `swap()` for either direction. [7](#0-6) 

No validation rejects `priceLimitX64 = 0` at the entry point. Any unprivileged caller — router, aggregator, or direct integrator — that mirrors the `zeroForOne = true` convention will trigger this on every `zeroForOne = false` swap. The condition is deterministic and 100% reproducible.

## Recommendation

Add a direction-aware sentinel at the `swap()` entry point or inside `_executeSwap`: when `zeroForOne = false` and `priceLimitX64 == 0`, substitute `type(uint128).max` (meaning "no upper bound"), mirroring how `0` is treated as "no lower bound" for `zeroForOne = true`. Alternatively, add a `require(priceLimitX64 != 0 || zeroForOne, InvalidPriceLimit())` guard at the top of `swap()` to revert explicitly rather than silently no-op.

## Proof of Concept

1. Deploy a pool with token0/token1; add liquidity to at least one bin so `initialPriceX64 > 0`.
2. Call `pool.swap(recipient, false, 1e18, 0, callbackData, "")` — `zeroForOne = false`, `priceLimitX64 = 0`.
3. Trace: `swap()` → `_executeSwap` → `_swapToken1ForToken0SpecifiedInput` → hits `if (0 <= initialPriceX64)` at line 970 → returns `(0, 0, 0)`.
4. `swap()` calls callback with `(0, 0, callbackData)`, emits `Swap(..., 0, 0, ...)`, returns `(0, 0)`. No tokens move.
5. Repeat with `zeroForOne = true, priceLimitX64 = 0`: `_swapToken0ForToken1SpecifiedInput` hits `if (0 >= initialPriceX64)` at line 1148 → false → swap executes normally across all bins, confirming the asymmetry.
