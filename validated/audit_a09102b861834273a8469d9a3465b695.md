### Title
Missing Validation of Zero rsETH Minted in `_beforeDeposit` Allows Depositor to Lose Assets - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._beforeDeposit` computes `rsethAmountToMint` via integer division but never validates that the result is non-zero before proceeding. When a user passes `minRSETHAmountExpected = 0` and the deposit amount is small enough that the division truncates to zero, the deposit succeeds, the user's assets are transferred into the protocol, and 0 rsETH is minted — leaving the user with no shares and no way to reclaim their funds.

### Finding Description
`_beforeDeposit` computes the rsETH amount to mint as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The only guard against a zero result is:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [1](#0-0) 

When `minRSETHAmountExpected = 0` (which the caller controls), the condition `0 < 0` is always false, so the revert never fires. The function returns `rsethAmountToMint = 0`, and `_mintRsETH(0)` is called, minting nothing to the depositor. [2](#0-1) 

The integer division truncates to zero whenever:

```
amount * lrtOracle.getAssetPrice(asset) < lrtOracle.rsETHPrice()
```

For ETH, `getAssetPrice(ETH) = 1e18`, so any deposit of `amount < rsETHPrice / 1e18` wei truncates to zero. Once `rsETHPrice` exceeds `1e18` (i.e., after any yield accrual), a 1-wei ETH deposit produces `rsethAmountToMint = 0`. [3](#0-2) 

The `minAmountToDeposit` guard only prevents zero-amount deposits; it defaults to `0` and does not prevent dust deposits that round down to zero rsETH. [4](#0-3) 

### Impact Explanation
A depositor who calls `depositETH(0, "")` or `depositAsset(asset, dustAmount, 0, "")` with a sufficiently small amount and `minRSETHAmountExpected = 0` will have their assets transferred into the protocol's TVL while receiving 0 rsETH. With no rsETH balance, the user has no mechanism to reclaim their deposited assets — the funds are effectively frozen from the user's perspective. The assets silently inflate the TVL, benefiting all existing rsETH holders at the depositor's expense.

Impact: **Medium — Temporary freezing of funds** (user's deposited assets are irrecoverable to them).

### Likelihood Explanation
- `minAmountToDeposit` defaults to `0`, so no admin action is required to enable the vulnerable path.
- Any unprivileged depositor can trigger this by passing `minRSETHAmountExpected = 0` (the parameter is fully caller-controlled) with a dust deposit.
- The condition becomes reachable as soon as `rsETHPrice > 1e18`, which occurs after any yield accrual — a normal, expected protocol state.
- No front-running, governance capture, or external dependency failure is required.

### Recommendation
Add an explicit non-zero check on `rsethAmountToMint` inside `_beforeDeposit`:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint == 0) {
    revert ZeroRsETHMinted();
}

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

This mirrors the recommendation in the reference report: always validate that shares (here, rsETH) were actually received before proceeding.

### Proof of Concept
1. Protocol has been running; `rsETHPrice` is now `1.05e18` (5% yield accrued).
2. Attacker (or naive user) calls:
   ```solidity
   lrtDepositPool.depositETH{value: 1}(0, "");
   ```
3. Inside `_beforeDeposit`:
   - `depositAmount = 1 wei`, passes `depositAmount == 0` check.
   - `rsethAmountToMint = (1 * 1e18) / 1.05e18 = 0` (integer division truncates).
   - `0 < 0` → false → no revert.
4. `_mintRsETH(0)` is called; 0 rsETH is minted to the caller.
5. The 1 wei ETH is now part of the protocol's TVL. The caller holds no rsETH and cannot withdraw. [5](#0-4) [6](#0-5)

            revert MaximumDepositLimitReached();
