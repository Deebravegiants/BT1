Audit Report

## Title
Stale rsETH Price Broadcast Cross-Chain When `PriceAboveDailyThreshold` Guard Blocks Non-Manager Price Update - (`contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary

`RSETHMultiChainRateProvider.getLatestRate()` reads the stored `rsETHPrice` state variable directly from `LRTOracle`. When the true rsETH price rises beyond `pricePercentageLimit` above `highestRsethPrice`, `_updateRsETHPrice()` reverts for non-manager callers before writing the new price, leaving `rsETHPrice` at the old lower value. Because `updateRate()` carries no access control, any caller can immediately broadcast this stale, underpriced rate to all registered destination chains via LayerZero.

## Finding Description

**Root cause:** `RSETHMultiChainRateProvider.getLatestRate()` reads the stored `rsETHPrice` state variable rather than computing the current price on-the-fly. [1](#0-0) 

`rsETHPrice` is only written at the very end of `_updateRsETHPrice()`, after all guards have passed: [2](#0-1) 

When the newly computed price exceeds `highestRsethPrice` by more than `pricePercentageLimit`, non-manager callers hit this guard and revert before line 313 is reached: [3](#0-2) 

`updateRate()` carries no role check — only `nonReentrant`: [4](#0-3) 

**Exploit flow:**
1. A legitimate TVL/reward event causes the computed rsETH price to exceed `highestRsethPrice * (1 + pricePercentageLimit)`.
2. Any caller invokes `LRTOracle.updateRSETHPrice()` → `_updateRsETHPrice()` reverts with `PriceAboveDailyThreshold`; `rsETHPrice` remains at the pre-increase value.
3. Before the manager calls `updateRSETHPriceAsManager()`, an attacker calls `RSETHMultiChainRateProvider.updateRate{value: lzFee}()`.
4. `getLatestRate()` returns the stale (lower) `rsETHPrice`; this value is encoded and sent via LayerZero to every registered `RSETHRateReceiver`.

**Why existing checks are insufficient:** The `PriceAboveDailyThreshold` guard protects `_updateRsETHPrice()` but has no corresponding guard in `updateRate()` or `getLatestRate()`. The two functions are entirely decoupled — the rate provider can broadcast whatever value happens to be stored, regardless of whether it reflects the current true price.

## Impact Explanation

Destination chain pools that consume the received rate to price rsETH/ETH deposits will see an artificially low rate. Users depositing ETH on destination chains receive more rsETH per ETH than the true exchange rate warrants, diluting existing rsETH holders' share of the protocol's TVL. This matches **Low — contract fails to deliver promised returns, but doesn't lose value**, as the protocol's cross-chain rate oracle fails to deliver the correct current rate during a normal operational condition.

## Likelihood Explanation

The precondition — a price increase above `pricePercentageLimit` — is a routine operational event (e.g., a large staking reward epoch or a significant asset price appreciation). The attack window opens the moment `updateRSETHPrice()` reverts and closes when the manager calls `updateRSETHPriceAsManager()`. `updateRate()` is permissionless and costs only gas plus LayerZero fees, making it trivially executable by any EOA or bot monitoring the mempool for the revert.

## Recommendation

Add a consistency check in `updateRate()` or `getLatestRate()` that attempts to call `updateRSETHPrice()` first and only proceeds if it succeeds (or if the caller is a manager). Alternatively, have `RSETHMultiChainRateProvider.getLatestRate()` call a view function that recomputes the current price on-the-fly (mirroring the logic in `_getTotalEthInProtocol`) rather than reading the potentially stale stored `rsETHPrice`. A minimal mitigation is to gate `updateRate()` behind the same `MANAGER` role required when the price is above the threshold.

// 4. Call lrtOracle.updateRSETHPrice() as an unprivileged address → expect revert PriceAboveDailyThreshold.

// 6. Call rsETHMultiChainRateProvider.updateRate{value: fee}() as the same unprivileged address.
