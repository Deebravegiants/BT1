### Title
`respond_verify_foreign_tx` Accepts Node-Supplied `payload_hash` Without Binding It to the Pending Request — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies the MPC signature against a `payload_hash` that is entirely supplied by the responding node, without checking that the hash was derived from the pending `request`. A single Byzantine participant (below the signing threshold) can replay a legitimately-produced threshold signature — originally created for request B — as the response to a different pending request A, causing the contract to resolve request A with a forged payload hash.

### Finding Description

In `respond`, the payload that is signed is taken directly from the stored `request.payload`:

```rust
// crates/contract/src/lib.rs ~line 600
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,          // ← comes from the request, not the response
    &expected_public_key,
)
``` [1](#0-0) 

In `respond_verify_foreign_tx`, however, the `payload_hash` is taken from the node-supplied `response`, not recomputed from the pending request:

```rust
// crates/contract/src/lib.rs ~line 726-734
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← node-controlled

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,              // root key, no tweak
)
.is_ok()
``` [2](#0-1) 

The contract cannot independently compute the expected hash because the `ForeignTxSignPayload` includes `extracted_values` that are only known to the MPC nodes after querying the foreign chain. The canonical hash is `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))`. [3](#0-2) 

Because the contract only checks `signature_is_valid` (i.e., "is this signature a valid ECDSA signature over this hash under the root key?") and never checks "does this hash correspond to the pending request?", a single attested participant can:

1. Observe a legitimately-produced threshold signature `(payload_hash_B, sig_B)` that was generated for pending request B.
2. Call `respond_verify_foreign_tx(request_A, VerifyForeignTransactionResponse { payload_hash: payload_hash_B, signature: sig_B })` for a different pending request A.
3. The contract finds request A in `pending_verify_foreign_tx_requests`, verifies `sig_B` against `payload_hash_B` (valid), and resolves request A with the wrong response. [4](#0-3) 

### Impact Explanation

The user who submitted request A receives `VerifyForeignTransactionResponse { payload_hash: payload_hash_B, signature: sig_B }`. The `payload_hash` attests to the state of foreign-chain transaction B, not transaction A. Any downstream smart contract or bridge that trusts the MPC contract's resolution and does not independently recompute and compare the expected hash will accept this as a valid verification of transaction A. This enables forged foreign-chain verification and potential double-spend or invalid bridge execution.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass … that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

The attack requires exactly one Byzantine attested participant (strictly below the signing threshold). No threshold collusion is needed — the attacker only needs to observe a legitimately-produced signature for any other pending request (a normal protocol output) and submit it against a different pending request. The `assert_caller_is_attested_participant_and_protocol_active` guard is satisfied by any single attested node. [5](#0-4) 

### Recommendation

The contract must bind the `payload_hash` to the pending request. Two complementary mitigations:

1. **Include `extracted_values` in the response** so the contract can recompute `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))` and compare it to `response.payload_hash` before accepting the response.
2. **Require a threshold of independent `respond_verify_foreign_tx` calls** (vote-style, analogous to `vote_pk`) so that a single Byzantine node cannot unilaterally resolve a pending request.

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(request_A)  → pending_verify_foreign_tx_requests[request_A] = [yield_A]

2. Bob  submits verify_foreign_transaction(request_B)  → pending_verify_foreign_tx_requests[request_B] = [yield_B]

3. Threshold nodes honestly process request_B:
   payload_hash_B = SHA-256(borsh(ForeignTxSignPayload{request_B, extracted_values_B}))
   sig_B = threshold_sign(payload_hash_B)

4. Malicious node calls:
   respond_verify_foreign_tx(
       request  = request_A,          // ← Alice's pending request
       response = { payload_hash: payload_hash_B, signature: sig_B }  // ← Bob's response
   )

5. Contract checks:
   - request_A exists in map  ✓
   - verify_ecdsa(sig_B, payload_hash_B, root_pk) == Ok  ✓  (never checks hash ↔ request_A)
   - resolves yield_A with response_B

6. Alice's caller receives payload_hash_B + sig_B, believing it attests to request_A.
   Any downstream contract that trusts the MPC resolution accepts this as valid
   verification of transaction A.
```

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
