Audit Report

## Title
rsETH Price Understated Due to Unaccounted FeeReceiver Rewards, Enabling Sandwich Attack on Minting - (File: `contracts/LRTOracle.sol`, `contracts/FeeReceiver.sol`)

## Summary
`LRTOracle._getTotalEthInProtocol()` excludes ETH sitting in `FeeReceiver`, causing `rsETHPrice` to be understated relative to true protocol TVL. Because `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` are both permissionless, an attacker can deposit at the understated price before the reward flush, then hold rsETH worth more than what was paid — extracting a portion of existing holders' accumulated MEV/execution-layer yield.

## Finding Description
`_getTotalEthInProtocol()` iterates over supported assets and calls `ILRTDepositPool.getTotalAssetDeposits(asset)` for each. For ETH, this delegates to `getETHDistributionData()`, which explicitly documents that `FeeReceiver` rewards are excluded and only accounted once moved to the deposit pool. `FeeReceiver` accumulates MEV and execution-layer rewards passively via its `receive()` fallback. Its `sendFunds()` function carries no access-control modifier and is callable by any address. Similarly, `updateRSETHPrice()` is `public whenNotPaused` with no role restriction. The minting formula in `getRsETHAmountToMint()` divides by the stored `rsETHPrice`, so deposits made while `FeeReceiver` holds unflused rewards receive more rsETH than they are entitled to.

**Exploit path:**
1. `FeeReceiver` holds `F` ETH in accumulated rewards. Tracked TVL = `T`, supply = `S`, stored price = `T/S` (true price = `(T+F)/S`).
2. Attacker calls `depositETH(X)` — minted rsETH = `X*S/T`, more than the fair `X*S/(T+F)`.
3. Attacker calls `FeeReceiver.sendFunds()` — `F` ETH moves to deposit pool.
4. Attacker calls (or waits for) `updateRSETHPrice()` — price updates to `(T+X+F)/(S+X*S/T)`.
5. Attacker's rsETH is worth more than `X` ETH; surplus `≈ F*X/(T+X)` is taken from existing holders' yield.

The `pricePercentageLimit` guard is insufficient: it is bypassed when `pricePercentageLimit == 0`, when accumulated rewards are small relative to TVL (price jump within threshold), or when the attacker simply waits for a manager or keeper bot to call `updateRSETHPrice()` rather than calling it themselves.

## Impact Explanation
Existing rsETH holders are entitled to MEV and execution-layer rewards accumulated in `FeeReceiver`. An attacker who deposits just before the reward flush captures a share of those rewards proportional to their deposit size relative to total TVL. This is a direct, quantifiable theft of unclaimed yield from existing holders, matching the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation
Both trigger functions are permissionless and callable atomically in a single transaction. `FeeReceiver` accumulates rewards continuously as a normal operational condition. The attack requires no special privileges, no flash loans, and no oracle manipulation. The `pricePercentageLimit` guard provides only partial protection and is easily circumvented. **Likelihood: Medium** — requires rewards to have accumulated and the price jump to fall within the configured threshold (or the limit to be unset).

## Recommendation
Include the `FeeReceiver` balance in `getETHDistributionData()` (or `_getTotalEthInProtocol()`) so that `rsETHPrice` always reflects true TVL including unsettled rewards:

```solidity
// In getETHDistributionData():
address feeReceiver = lrtConfig.getContract(LRTConstants.LRT_FEE_RECEIVER);
uint256 ethLyingInFeeReceiver = feeReceiver.balance;
// add ethLyingInFeeReceiver to the returned total
```

Alternatively, add an access-control modifier to `FeeReceiver.sendFunds()` so reward settlement cannot be triggered atomically by an unprivileged attacker.

## Proof of Concept
```
State before attack:
  FeeReceiver balance (F) = 10 ETH
  Deposit pool tracked TVL (T) = 1000 ETH
  rsETH supply (S) = 1000 rsETH
  Stored rsETHPrice = 1.0 ETH/rsETH  (true = 1.01)

Step 1 — depositETH(100 ETH):
  rsETH minted = 100 / 1.0 = 100 rsETH  (fair ≈ 99.01 rsETH)

Step 2 — FeeReceiver.sendFunds():
  Deposit pool ETH = 1000 + 100 + 10 = 1110 ETH

Step 3 — updateRSETHPrice():
  New supply = 1100 rsETH
  New price = 1110 / 1100 ≈ 1.00909 ETH/rsETH

Attacker's 100 rsETH = 100.909 ETH → profit ≈ 0.909 ETH stolen from existing holders
```

Foundry fork test: deploy against a mainnet fork with non-zero `FeeReceiver` balance, execute the three-call sequence in one transaction, assert attacker rsETH value exceeds deposited ETH, and assert existing holder rsETH value decreased relative to pre-attack baseline.
