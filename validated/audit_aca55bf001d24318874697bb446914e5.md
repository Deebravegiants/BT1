Audit Report

## Title
Stale Cross-Chain Oracle Rate Allows Depositors to Mint Excess rsETH at Existing Holders' Expense - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All L2 pool contracts use a cross-chain oracle rate stored in `CrossChainRateReceiver` that is only updated when `updateRate()` is called on the L1 `CrossChainRateProvider`. Because rsETH is a yield-bearing token whose price monotonically increases, the stored L2 rate is structurally stale between updates. Any depositor can exploit this lag to receive more rsETH than their deposit is worth at the true L1 rate, then bridge and redeem on L1 at the true rate, extracting yield from existing rsETH holders.

## Finding Description
`CrossChainRateReceiver.getRate()` simply returns the last stored `rate` with no staleness check:

```solidity
// CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

This `rate` is only updated when a LayerZero message arrives via `lzReceive`, which is triggered by calling `updateRate()` on the L1 `CrossChainRateProvider`. Between calls, the rate is frozen at its last value.

`RSETHPoolV3.getRate()` delegates directly to this receiver:

```solidity
// RSETHPoolV3.sol L235-237
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
```

Both `viewSwapRsETHAmountAndFee` overloads use this stale rate as the denominator:

```solidity
// RSETHPoolV3.sol L304-307 (ETH path)
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;

// RSETHPoolV3.sol L328-334 (token path)
uint256 rsETHToETHrate = getRate();
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

When `rsETHToETHrate` (stale) < true L1 rate, the division yields a larger rsETH amount than the deposited value warrants. For the token path, if the Chainlink `tokenToETHRate` is simultaneously at the top of its deviation band, both errors compound: the numerator is inflated and the denominator is deflated.

`feeBps` has no enforced minimum — `setFeeBps` accepts 0 — so the fee cannot be relied upon as a structural guard. The daily mint limit does not prevent the attack; it only caps the per-day magnitude.

The same flaw exists identically in `RSETHPoolV3ExternalBridge` (L433-452), `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper` (L292-311).

## Impact Explanation
**High — Theft of unclaimed yield.** The excess rsETH minted is a claim on L1 backing assets computed at the true (higher) rsETH price. When the attacker bridges to L1 and redeems via `LRTWithdrawalManager`, they receive more ETH than they deposited. The shortfall is absorbed by all existing rsETH holders through dilution of the backing ratio, constituting a direct, permissionless drain of accrued staking yield.

## Likelihood Explanation
The L2 oracle is structurally stale between every pair of `updateRate()` calls. rsETH accrues staking yield continuously (~4–5% APY), so any gap between updates creates a non-zero exploitable discrepancy. No privileged access, flash loans, or front-running are required. The attacker only needs to observe the on-chain `lastUpdated` timestamp on the receiver and the current L1 rsETH price (publicly available), then deposit when the lag is profitable relative to the fee. With `feeBps = 0`, any positive lag is immediately profitable. With `feeBps = 10` (10 bps), a lag of roughly 7 days at 5% APY (~10 bps of drift) suffices. The Chainlink deviation band for collateral tokens (e.g. 0.5% for wstETH/ETH) is a well-known, publicly observable parameter that compounds the discrepancy on the token deposit path.

## Recommendation
1. **Enforce a minimum `feeBps`** that exceeds the maximum expected oracle lag plus the Chainlink deviation threshold for all supported tokens. Remove the ability to set `feeBps = 0`.
2. **Add a staleness check** in `getRate()` or in `viewSwapRsETHAmountAndFee`: revert if `block.timestamp - lastUpdated > heartbeatWindow` for a configurable `heartbeatWindow`.
3. **Add a withdrawal delay or fee** on the L1 side for rsETH redeemed shortly after an L2 deposit, to break the round-trip arbitrage.
4. **Consider a TWAP or rate cap** on the L2 oracle to limit the maximum rate change per update, smoothing sudden discrepancies.

## Proof of Concept
**Setup**: `RSETHPoolV3` on Arbitrum. True rsETH/ETH rate on L1 = 1.060 ETH/rsETH. Last propagated L2 oracle rate = 1.050 ETH/rsETH (stale, e.g. 7+ days since last `updateRate()` call). Chainlink wstETH/ETH oracle at top of 0.5% deviation band: reports 1.005 ETH/wstETH. `feeBps = 0`.

**Step 1**: Attacker calls `deposit(wstETH, 1000e18, "")` on `RSETHPoolV3`.

`viewSwapRsETHAmountAndFee(1000e18, wstETH)` computes:
- `rsETHToETHrate = 1.050e18` (stale oracle, `CrossChainRateReceiver.rate`)
- `tokenToETHRate = 1.005e18` (Chainlink at top of band)
- `rsETHAmount = 1000e18 * 1.005e18 / 1.050e18 = 957.14e18` wrsETH minted

True fair rsETH amount = `1000 * 1.000 / 1.060 = 943.40` rsETH.

Attacker receives **957.14 rsETH** instead of **943.40 rsETH** — excess of **13.74 rsETH**.

**Step 2**: Attacker unwraps wrsETH → rsETH, bridges to L1 via LayerZero OFT.

**Step 3**: Attacker calls `LRTWithdrawalManager.initiateWithdrawal` then `completeWithdrawal`. At the true rate of 1.060 ETH/rsETH, 957.14 rsETH redeems for **1014.57 ETH**.

**Profit**: 14.57 ETH extracted from existing rsETH holders per 1000 wstETH deposited.

**Foundry fork test plan**: Fork Arbitrum mainnet, deploy or point to the live `RSETHPoolV3` and `RSETHRateReceiver`. Warp time forward by 7+ days without calling `updateRate()`. Fetch the true L1 rsETH price via `ILRTOracle.rsETHPrice()`. Assert that `viewSwapRsETHAmountAndFee` returns an rsETH amount greater than `deposit_value_in_ETH * 1e18 / true_L1_rate`.
