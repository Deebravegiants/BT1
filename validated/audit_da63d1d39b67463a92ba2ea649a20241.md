**Gateway state reader setup** (`sync_state_reader.rs:525-549`): `SyncStateReaderFactory::get_blockifier_state_reader_and_gateway_fixed_block_from_latest_block` creates a `SyncStateReader::from_number(... latest_block_number ...)` — the state reader is pinned at `latest_block_number = L`.

The block context uses `L + 1` as `current_block_number`.

**`validate_proof_block_number`** (`account_transaction.rs:244-250`):

With `current_block_number = L + 1`, `max_allowed = L + 1 - STORED_BLOCK_HASH_BUFFER`.

**`pre_process_block`** (`block.rs:26-30`): When processing block `N`, it writes `block_hash[N - STORED_BLOCK_HASH_BUFFER]`. So the state committed at block `L` contains hashes only up to `L - STORED_BLOCK_HASH_BUFFER`.

This is not a flush/race condition as the claim frames it — it is a **structural off-by-one**: the gateway always advances `block_number` by 1 for the block context but never advances the state reader to match.

### Title
Gateway `validate_proof_block_hash` Rejects Valid Proof Facts at the Off-by-One Block Boundary — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`, `crates/blockifier/src/transaction/account_transaction.rs`)

### Summary
The gateway constructs a block context with `block_number = latest_synced_block + 1` but reads storage from a state pinned at `latest_synced_block`. `validate_proof_block_number` therefore permits `proof_block_number = latest_synced_block + 1 - STORED_BLOCK_HASH_BUFFER`, but that storage slot is absent from the pinned state, causing `validate_proof_block_hash` to return "Block hash mismatch" and reject a structurally valid transaction.

### Finding Description
In `run_validate_entry_point`: [1](#0-0) 

`block_info.block_number` is incremented to `L + 1`, but the underlying `CachedState` wraps a `SyncStateReader` pinned at `L`: [2](#0-1) 

`pre_process_block` writes `block_hash[N - STORED_BLOCK_HASH_BUFFER]` only when block `N` is executed: [3](#0-2) 

So the state at block `L` contains hashes only up to `L - STORED_BLOCK_HASH_BUFFER`. Yet `validate_proof_block_number` computes `max_allowed = (L+1) - STORED_BLOCK_HASH_BUFFER`: [4](#0-3) 

A transaction with `proof_block_number = L + 1 - STORED_BLOCK_HASH_BUFFER` passes the number check but `get_storage_at` returns `Felt::ZERO` for that slot, causing: [5](#0-4) 

to reject the transaction with "Block hash mismatch".

### Impact Explanation
Any unprivileged user submitting a valid Invoke V3 transaction whose `proof_facts` reference the most recently allowed block number (`latest_synced_block + 1 - STORED_BLOCK_HASH_BUFFER`) will have their transaction incorrectly rejected at the gateway. This is a **High** impact: valid transactions are rejected before sequencing.

### Likelihood Explanation
The off-by-one is permanent and deterministic — it is not a transient race. Every transaction referencing the boundary block number is rejected. Users who generate proofs against the most recent eligible block (a natural choice) will always hit this.

### Recommendation
In `run_validate_entry_point`, either:
1. Do **not** increment `block_info.block_number` before constructing the block context used for `validate_proof_block_number` / `validate_proof_block_hash`, so the permitted range matches the available state; or
2. Clamp `max_allowed` in `validate_proof_block_number` to `current_block_number - STORED_BLOCK_HASH_BUFFER - 1` when called from the gateway path (i.e., when the state is one block behind the block context).

### Proof of Concept
1. Let `L = latest_synced_block`, `B = STORED_BLOCK_HASH_BUFFER`.
2. Construct a valid Invoke V3 transaction with `proof_block_number = L + 1 - B` and the correct `proof_block_hash` for that block.
3. Submit to the gateway.
4. `validate_proof_block_number` passes (`L + 1 - B <= (L+1) - B`).
5. `state.get_storage_at(block_hash_contract_address, StorageKey::from(L + 1 - B))` returns `Felt::ZERO` (the hash was written during block `L+1` processing, which has not happened yet).
6. `validate_proof_block_hash` returns `Err(InvalidProofFacts("Block hash mismatch ..."))`.
7. The gateway rejects the valid transaction.
