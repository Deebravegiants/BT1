### Title
`Map.withoutKeys` in POOLREAP Removes Dangling VRF Keys Without Decrementing Reference Count, Breaking VRF Uniqueness — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

`poolReapTransition` uses `Map.withoutKeys danglingVRFKeyHashes` to remove "overwritten" VRF keys from `psVRFKeyHashes`. This operation deletes the key entirely regardless of its reference count. When a second pool Q shares the same VRF key X (registered before the v11 hard fork), retiring pool P (which re-registered with a new VRF Y) causes X to be fully evicted from `psVRFKeyHashes` even though Q still holds it. After the epoch, a new pool can register with X, breaking the VRF uniqueness invariant.

---

### Finding Description

**Root cause — `PoolReap.hs` lines 224–227:**

```haskell
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )
```

`removeVRFKeyHashOccurrences` (line 235–237) correctly decrements the reference count and only removes the key when it reaches zero:

```haskell
removeVRFKeyHashOccurrence =
  Map.update (mapNonZero (\n -> n - 1))
```

But `Map.withoutKeys danglingVRFKeyHashes` (line 226) unconditionally deletes every key in the set, ignoring the count entirely. [1](#0-0) [2](#0-1) 

**`danglingVRFKeyHashes` computation (lines 138–148):**

Finds pools present in both `psStakePools` and `psFutureStakePoolParams` whose VRF key changed. For pool P (current VRF X, future VRF Y), X is added to the dangling set. [3](#0-2) 

**Re-registration clears retirement (Pool.hs line 305):**

```haskell
& psRetiringL %~ Map.delete sppId
```

Re-registering P with VRF Y clears any pending retirement. A subsequent `RetirePool` cert in the same epoch re-adds it. Both operations are valid and can appear in the same block. [4](#0-3) 

**Duplicate VRF keys are possible before v11:**

`hardforkConwayDisallowDuplicatedVRFKeys` only activates at protocol version > 10. Before that, pools P and Q could both register with VRF X. At the v11 hard fork, `populateVRFKeyHashes` correctly sets `psVRFKeyHashes = {X: 2}`. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

After the buggy POOLREAP:

- `psVRFKeyHashes` no longer contains X, even though pool Q is still active with VRF X.
- The POOL rule's duplicate-VRF guard (`Map.notMember sppVrf psVRFKeyHashes`) now passes for X.
- A new pool R can register with VRF X while Q still uses it.
- Two active pools share the same VRF key, breaking the uniqueness invariant that the v11 hard fork was designed to enforce.
- Both Q and R are elected for the same slots (slot leadership is VRF-derived), enabling double-block-production and potential chain forks.

This matches: **High — Deterministic disagreement between honest nodes from ledger rule evaluation.** [7](#0-6) 

---

### Likelihood Explanation

**Preconditions:**
1. Pools P and Q must share VRF X — only possible if both were registered before the v11 hard fork. On mainnet, if any such pairs exist in the ledger state, the attack is immediately available.
2. The attacker controls pool P (a legitimate pool operator — no privileged access required).
3. The attacker submits two certificates in the same epoch: `RegPool P (VRF=Y)` then `RetirePool P`.

All three steps are within normal pool operator capabilities and require no governance majority, leaked keys, or Sybil attack.

---

### Recommendation

Replace `Map.withoutKeys danglingVRFKeyHashes` with `removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)` so that dangling VRF keys are decremented by one occurrence rather than unconditionally deleted. This mirrors the correct reference-counting logic already used for `retiredVRFKeyHashes`. [8](#0-7) 

---

--   -> psFutureStakePoolParams[P].vrf = Y
--   -> psVRFKeyHashes = {X: 2, Y: 1}
--   -> psRetiring[P] deleted
-- Tx2: retire P at epoch M+1
--   -> psRetiring[P] = M+1

-- Epoch M+1: POOLREAP runs
--   danglingVRFKeyHashes = {X}  (P: current=X, future=Y)
--   activate future: P.vrf = Y
--   retired = {P}
--   retiredVRFKeyHashes = [Y]
--   removeVRFKeyHashOccurrences [Y]: Y count 1->0, removed
--   Map.withoutKeys {X}: X removed entirely (BUG: count was 2, Q still uses X)
--   psVRFKeyHashes = {}   <-- Q is active but X is gone

-- Post-attack:
-- Register new pool R with VRF X -> succeeds (Map.notMember X {} == True)
-- psVRFKeyHashes = {X: 1}
-- Both Q and R are active with VRF X -- uniqueness invariant broken

assert (Map.member vrf_X (psVRFKeyHashes pState))  -- FAILS

```haskell
populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
```
