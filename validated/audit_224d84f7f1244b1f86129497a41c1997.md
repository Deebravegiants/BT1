### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Single Byzantine Participant to Inject Fabricated Key Derivation Output — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs a cryptographic pairing check (`ckd_output_check`) only for the `AppPublicKeyPV` variant. For the `AppPublicKey` variant, the match arm is an empty no-op. A single Byzantine attested participant can call `respond_ckd` with completely arbitrary `big_y` and `big_c` values, and the contract will accept and resolve the pending yield with the fabricated output — bypassing the threshold requirement entirely.

---

### Finding Description

In `respond_ckd`, the guard at lines 675–682 is asymmetric: [1](#0-0) 

The `AppPublicKeyPV` arm calls `ckd_output_check`, which verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`: [2](#0-1) 

This pairing check cryptographically binds the response to the MPC network's secret key `msk`, ensuring the output could only have been produced by threshold-many honest participants. The `AppPublicKey` arm has no equivalent binding — it is a bare `{}`.

After the match, regardless of variant, the response is unconditionally serialized and passed to `resolve_yields_for`: [3](#0-2) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active` at line 666, which requires the caller to be a single attested participant in the current epoch — not a threshold of them: [4](#0-3) 

---

### Impact Explanation

The `AppPublicKey` CKD protocol is designed so the caller holds `app_sk` and the MPC network contributes `big_s = hash(app_id) * msk`. The caller recovers `derived_key = big_c − big_y * app_sk = big_s`. The security guarantee is that `big_s` requires threshold-many honest participants to compute.

A Byzantine leader can craft a response that breaks this guarantee while remaining computable without any secret knowledge:

- Set `big_y = g1 * r` for attacker-chosen scalar `r`
- Set `big_c = app_pk * r + big_s_fake` for attacker-chosen `big_s_fake`
- Since `app_pk = g1 * app_sk` is public (it is in the request), the attacker can compute `app_pk * r`
- The caller then recovers `big_c − big_y * app_sk = app_pk * r + big_s_fake − g1 * r * app_sk = big_s_fake`

The attacker knows `big_s_fake`, so they know the caller's derived key. This enables impersonation of the derived key or prediction of any downstream cryptographic material derived from it — unauthorized confidential key derivation output produced by a single Byzantine participant, below the signing threshold.

The existing test at lines 3424–3427 already demonstrates the contract accepts `big_y=[1u8;48], big_c=[2u8;48]` for `AppPublicKey` without any rejection: [5](#0-4) 

---

### Likelihood Explanation

The attacker must be a single attested participant — a node that has passed TEE attestation and is registered in the current epoch. This is a Byzantine participant below the signing threshold. The exploit requires no collusion, no key leakage, and no network-level attack. The attacker simply calls `respond_ckd` with crafted values after a user submits a `request_app_private_key` with `AppPublicKey`.

---

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` variant. For `AppPublicKey`, the check degenerates to verifying `e(big_c, g2) = e(big_y, g2) * e(hash_point, public_key)` (since `app_pk2 = g2` when the app key is a plain G1 point with no separate G2 component). Alternatively, require `AppPublicKey` requests to be upgraded to `AppPublicKeyPV` so the caller always provides a verifiable key pair, and remove the unverified variant from the production interface.

---

### Proof of Concept

In a contract unit test with `AppPublicKey` variant:

1. Call `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(app_pk)`.
2. Switch context to an attested participant.
3. Call `respond_ckd` with `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (arbitrary, not derived from `msk`).
4. Assert `Ok(())` is returned and the pending yield is resolved.
5. Compare the resolved output against the honest CKD computation from `create_response_ckd` in `crates/contract/tests/sandbox/utils/sign_utils.rs` — they will differ, confirming the fabricated output was accepted. [6](#0-5)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)

pub fn create_response_ckd(
    account_id: &AccountId,
    app_public_key: &dtos::Bls12381G1PublicKey,
    domain_id: &DomainId,
    key_package: &KeygenOutput<BLS12381SHA256>,
    derivation_path: &str,
) -> (CKDRequest, CKDResponse) {
    let request = CKDRequest::new(
        CKDAppPublicKey::AppPublicKey(app_public_key.clone()),
        *domain_id,
        account_id,
        derivation_path,
    );

    let app_id = derive_app_id(account_id, derivation_path);
    let app_pk: ckd::ElementG1 = app_public_key
        .try_into()
        .expect("invalid BLS12-381 G1 point");
    let msk = key_package.private_share.to_scalar();

    let big_s = hash_app_id_with_pk(&key_package.public_key, app_id.as_ref()) * msk;
    let y = ckd::Scalar::random(OsRng);
    let big_y = ckd::ElementG1::generator() * y;
    let big_c = big_s + app_pk * y;

    let response = CKDResponse {
        big_y: (&big_y).into(),
        big_c: (&big_c).into(),
    };
