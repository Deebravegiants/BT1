### Title
Division by Zero in `get_current_authentication_token` Crashes Farmer Pool Loop When Malicious Pool Returns Zero `authentication_token_timeout` — (`File: chia/protocols/pool_protocol.py`)

### Summary
`get_current_authentication_token` in `chia/protocols/pool_protocol.py` performs integer division by the caller-supplied `timeout` parameter without guarding against zero. A malicious pool operator can return `authentication_token_timeout = 0` in its `/pool_info` HTTP response, causing an unhandled `ZeroDivisionError` that propagates through the farmer's pool-state update loop before any exception handler is reached, permanently stalling pool participation for that farmer.

### Finding Description

**Root cause — unguarded division:**

`get_current_authentication_token` divides by `timeout` with no zero-check:

```python
# chia/protocols/pool_protocol.py:169-170
def get_current_authentication_token(timeout: uint8) -> uint64:
    return uint64(int(time.time() / 60) / timeout)   # ZeroDivisionError when timeout == 0
``` [1](#0-0) 

`timeout` is typed `uint8`, which accepts the value `0`. No caller validates it before passing it in.

**Source of the zero — pool-controlled field:**

The farmer fetches `/pool_info` from the pool's HTTP server and stores the response field directly:

```python
# chia/farmer/farmer.py:612
pool_state["authentication_token_timeout"] = pool_info.authentication_token_timeout
``` [2](#0-1) 

`GetPoolInfoResponse.authentication_token_timeout` is declared as `uint8` with no minimum-value constraint: [3](#0-2) 

**Crash path — exception escapes before any handler:**

In `update_pool_state`, the only guard is a `None` check, not a zero check:

```python
# chia/farmer/farmer.py:652-653
if authentication_token_timeout is not None:
    farmer_info, error_code = await update_pool_farmer_info()
``` [4](#0-3) 

`update_pool_farmer_info()` calls `_pool_get_farmer`, which calls `get_current_authentication_token` **before** its own `try` block:

```python
# chia/farmer/farmer.py:398  (before the try: at ~line 410)
authentication_token = get_current_authentication_token(authentication_token_timeout)
``` [5](#0-4) 

The same pre-`try` pattern exists in `_pool_post_farmer` (line 449) and `_pool_put_farmer` (line 497), and in `farmer_api.py` during partial submission: [6](#0-5) 

Because the division happens before any `except` clause, the `ZeroDivisionError` propagates up through `update_pool_farmer_info` → `update_pool_state`, crashing the periodic pool-state update task.

### Impact Explanation

**High — long-lived inability for farmers to process pool actions.**

Once the farmer stores a zero `authentication_token_timeout` from a malicious pool, every subsequent pool-state update cycle raises `ZeroDivisionError` before reaching any recovery path. The farmer can no longer submit partials, register with the pool, or update farmer info for that pool entry. Pool rewards (7/8 of block reward directed to the pool's `p2_singleton` address) stop being credited. Recovery requires manual removal of the pool from the farmer's config and a restart — an action most automated farming setups will not perform automatically.

### Likelihood Explanation

Any operator of a pool HTTP server reachable by a farmer can trigger this with a single crafted `/pool_info` response containing `"authentication_token_timeout": 0`. No cryptographic material, no privileged Chia keys, and no network position beyond running an HTTP endpoint are required. The farmer connects to pool URLs it was configured with; the pool's response is fully attacker-controlled.

### Recommendation

1. **Enforce a minimum value in `get_current_authentication_token`:** raise `ValueError` (or clamp to 1) if `timeout == 0`.
2. **Validate `authentication_token_timeout` immediately after parsing `GetPoolInfoResponse`:** reject or log-and-skip any pool response where this field is zero before storing it in `pool_state`.
3. **Move the `get_current_authentication_token` call inside the existing `try/except` blocks** in `_pool_get_farmer`, `_pool_post_farmer`, and `_pool_put_farmer` so that a bad value degrades gracefully rather than crashing the update loop.

### Proof of Concept

1. Stand up an HTTP server that responds to `GET /pool_info` with:
   ```json
   {"name":"evil","logo_url":"","minimum_difficulty":1,"relative_lock_height":100,
    "protocol_version":1,"fee":"0","description":"","target_puzzle_hash":"<valid 32-byte hex>",
    "authentication_token_timeout":0}
   ```
2. Configure a Chia farmer to join this pool URL.
3. Wait for the farmer's `update_pool_state` periodic task to fire (default interval).
4. Observe `ZeroDivisionError` propagating from `get_current_authentication_token(0)` at `chia/protocols/pool_protocol.py:170`, crashing the pool-state update task and halting all partial submissions to that pool indefinitely.
