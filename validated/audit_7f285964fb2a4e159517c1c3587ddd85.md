### Title
Block-Producer-Predictable Entropy Used as Sole Randomness Source for ECDSA Presignature Rerandomization - (File: crates/node/src/types.rs)

### Summary
The `entropy` field injected into every `SignatureRequest` (and `CKDRequest`, `VerifyForeignTxRequest`) is taken verbatim from `block.header.random_value` — NEAR's VRF output, which the block producer knows before publishing the block. This entropy is then used as the **sole unpredictable input** to `RerandomizationArguments::derive_randomness()`, which computes the rerandomization scalar `delta` protecting all ECDSA presignatures. The GS21 scheme's security proof explicitly requires this entropy to be "fresh, unpredictable, and public." A malicious NEAR block producer violates the unpredictability requirement, enabling selective-inclusion manipulation of `delta` and potentially a Wagner-style nonce-relation attack to recover the MPC private key.

---

**Step 3 — This entropy is the sole unpredictable input to `RerandomizationArguments`.**

In both the OT-based and Robust ECDSA `SignComputation::compute()`, the block entropy is passed directly as the `entropy` argument:

```rust
let rerand_args = RerandomizationArguments::new(
    self.keygen_out.public_key.to_element().to_affine(),
    tweak,
    self.msg_hash,
    self.presign_out.big_r,
    participants,
    self.entropy,   // <-- block.header.random_value
);
``` [6](#0-5) [7](#0-6) 

**Step 4 — `derive_randomness()` uses this entropy as the HKDF key material.**

```rust
let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);
``` [8](#0-7) 

The code's own warning states:

> `*** Warning *** Following [GS21], the entropy should be public, freshly generated, and unpredictable.` [9](#0-8) 

**Step 5 — NEAR's `random_value` is the block producer's VRF output, known to them before block publication.**

In NEAR Protocol, `block.header.random_value` is the VRF output of the current block producer. The block producer computes it deterministically from their VRF secret key and the previous block hash, and therefore **knows it before they publish the block**. This means:

- The block producer can compute `delta = HKDF(random_value, pk, msg_hash, big_r, participants)` for any pending sign request before deciding to include it.
- By selectively including or withholding a sign request across consecutive blocks, the block producer can choose from multiple possible `delta` values (one per candidate block).
- All MPC nodes derive the same `entropy` from the same block, so the block producer's choice of which block to include the request in determines `delta` for all participants.

---

### Impact Explanation

The rerandomization scalar `delta` is the security-critical value that prevents Wagner-style attacks on presignatures. The GS21 scheme's security proof requires `delta` to be unpredictable to any adversary who knows the presignature `big_r`. A malicious block producer:

1. Knows `random_value` (and therefore `delta`) before publishing the block.
2. Can selectively include/exclude the sign request to choose among several `delta` values.
3. Across multiple signing sessions (each using a different presignature), can bias the distribution of rerandomized nonces `big_r' = big_r * delta`.

If the block producer can arrange for two signing sessions to use algebraically related `delta` values, the resulting ECDSA nonces are related, enabling a standard nonce-reuse/Wagner attack to recover the MPC private key. Key recovery leads to **Critical** impact: unauthorized signing of arbitrary transactions on all supported foreign chains (Ethereum, Bitcoin, Solana, etc.) and permanent loss of all funds controlled by the MPC network.

---

### Likelihood Explanation

NEAR validators are economically incentivized participants. A validator who is also a sophisticated attacker (or who colludes with one) can mount this attack without any threshold-level collusion among MPC participants — the block producer role is entirely separate from the MPC participant set. The attack requires:

- Being a NEAR block producer (validator) — achievable by staking.
- Observing pending sign requests in the mempool — trivially possible.
- Performing selective inclusion across multiple signing sessions — standard validator capability.

No TEE bypass, no key leakage, and no MPC participant collusion is required. Likelihood is **Medium** (requires being a validator, but validators are a known and realistic threat actor in any PoS network).

---

### Recommendation

Replace the sole reliance on `block.header.random_value` as the entropy source. Options:

1. **Mix in a node-local secret**: Each MPC node contributes its own locally-generated random bytes to the entropy, combined with the block entropy via a hash (e.g., `H(block_entropy || node_local_random)`). The block producer cannot predict the node-local contributions.
2. **Use a commit-reveal scheme**: Nodes commit to random values before the block is finalized, then reveal them; the combined entropy is unpredictable to any single party.
3. **Derive entropy from the receipt ID and a node secret**: The receipt ID is determined before the block producer knows which block it will land in, and mixing it with a node-local secret provides unpredictability.

---

### Proof of Concept

1. Attacker stakes NEAR tokens to become a block producer.
2. Attacker observes a pending `sign()` call in the mempool targeting the MPC contract.
3. For the current candidate block, attacker computes `random_value_1` (their VRF output) and derives `delta_1 = HKDF(random_value_1, pk, msg_hash, big_r_1, participants)`.
4. Attacker also computes `random_value_2` for the next block they will produce and derives `delta_2`.
5. Attacker selects the block that yields a `delta` satisfying a desired algebraic relation (e.g., `delta_2 / delta_1 = k` for a chosen `k`).
6. Across two signing sessions with presignatures `(big_r_1, k_1, sigma_1)` and `(big_r_2, k_2, sigma_2)`, the rerandomized nonces satisfy `k_1' / k_2' = (k_1 * inv_delta_1) / (k_2 * inv_delta_2)`.
7. With two ECDSA signatures sharing a known nonce relation, the attacker applies standard nonce-reuse algebra to recover the MPC private key share aggregate, enabling unauthorized signing of arbitrary foreign-chain transactions.
