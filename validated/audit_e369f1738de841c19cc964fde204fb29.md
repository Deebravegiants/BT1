The code is confirmed exactly as described. The vulnerability is valid.

Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Amount in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for LST tokens it evaluates `totalAssetDeposits + amount > limit`, but for ETH it evaluates only `totalAssetDeposits > limit`, silently dropping the incoming `amount`. Any unprivileged caller can therefore deposit an arbitrarily large ETH amount in a single transaction and push total ETH holdings far above the configured cap.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682 of `contracts/LRTDepositPool.sol`), the ETH branch at line 679 returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without adding `amount`, while the LST branch at line 681 correctly returns `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. `_beforeDeposit` (lines 648–670) calls this function and reverts only when it returns `true`; because the ETH branch never returns `true` until the running total already exceeds the limit from a prior deposit, the cap is never enforced prospectively. The public entry point `depositETH` (lines 76–93) passes `msg.value` as `amount` through `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit`, but that value is discarded in the ETH branch. No access control, role, or other guard prevents an arbitrary caller from supplying any `msg.value`.

## Impact Explanation
The ETH deposit limit is the protocol's primary risk-management cap on native ETH exposure. Bypassing it allows any depositor to push total ETH holdings arbitrarily above the configured ceiling in a single call, causing the contract to accept and mint rsETH for deposits it was explicitly configured to reject. This constitutes **Low — Contract fails to deliver promised returns**: the protocol advertises and enforces a deposit cap for all assets, but silently fails to enforce it for ETH.

## Likelihood Explanation
`depositETH` is public, payable, and requires no special role. Any depositor who observes that `getTotalAssetDeposits(ETH_TOKEN)` is at or near the limit can immediately send an oversized deposit. No flash loan, governance action, or multi-step setup is required. The condition is trivially observable on-chain and repeatable at any time.

## Recommendation
Add `amount` to the ETH branch to mirror the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Assume `depositLimitByAsset(ETH_TOKEN) = 1000 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.

1. Attacker calls `depositETH{value: 5000 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 5000 ether)`.
3. ETH branch (line 679) evaluates `999 ether > 1000 ether` → `false` → no revert.
4. `_mintRsETH` mints rsETH for 5000 ETH.
5. `getTotalAssetDeposits(ETH_TOKEN)` is now `6999 ether`, nearly 7× the cap.

For LST tokens the same scenario evaluates `999 ether + 5000 ether > 1000 ether` → `true` → `MaximumDepositLimitReached` revert. A Foundry fuzz test parameterising `amount` over `[1, type(uint128).max]` with `totalAssetDeposits` initialised to `limit - 1` would confirm the ETH path never reverts while the LST path always does.
