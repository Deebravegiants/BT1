### Title
Cross-Chain Replay of MPC-Signed MetadataPayload in `deployToken` Due to Missing Chain-ID Binding — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The Borsh-encoded blob that `deployToken` passes to `keccak256` for MPC-signature verification contains no destination-chain identifier (`omniBridgeChainId`). A valid MPC signature obtained for chain A can be replayed verbatim on any other OmniBridge deployment that shares the same `nearBridgeDerivedAddress`, deploying an unbacked bridge token and permanently blocking the legitimate token deployment on that chain.

---

### Finding Description

In `deployToken`, the signed message is constructed as:

```
PayloadType.Metadata | token | name | symbol | decimals
``` [1](#0-0) 

There is no `omniBridgeChainId` field in this blob. Compare with `finTransfer`, which correctly embeds `omniBridgeChainId` twice (for `tokenAddress` and `recipient` chain scoping): [2](#0-1) 

The contract stores `omniBridgeChainId` as a state variable set at initialization, but it is never read inside `deployToken`: [3](#0-2) 

The `MetadataPayload` struct itself carries no chain field: [4](#0-3) 

Because the NEAR bridge contract is a single contract serving all EVM deployments, every OmniBridge instance on every EVM chain is initialized with the **same** `nearBridgeDerivedAddress` (the MPC-derived Ethereum address of the NEAR bridge account). Therefore the ECDSA check:

```solidity
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress)
    revert InvalidSignature();
``` [5](#0-4) 

passes identically on every chain for the same `(signatureData, metadata)` tuple.

---

### Impact Explanation

1. **Spurious canonical token registered on chain B**: `isBridgeToken[spuriousProxy] = true` and `nearToEthToken[metadata.token] = spuriousProxy` are written on chain B with no locked collateral backing. [6](#0-5) 

2. **Legitimate deployment permanently blocked**: The guard `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` will revert every future legitimate `deployToken` call for that NEAR token on chain B. [7](#0-6) 

3. **Subsequent `finTransfer` mints on the spurious token**: Once the NEAR bridge registers the spurious token address for chain B (which it may do by reading on-chain state), MPC-signed `finTransfer` messages will call `IBridgeToken(spuriousProxy).mint(...)`, issuing supply backed by nothing. [8](#0-7) 

---

### Likelihood Explanation

- The precondition (shared `nearBridgeDerivedAddress` across chains) is the **standard deployment topology** — one NEAR bridge contract, many EVM chains.
- Any relayer who legitimately observes a `deployToken` call on chain A (or intercepts the MPC signature off-chain) can replay it on chain B with zero additional privilege.
- The attack is a single public transaction with no timing constraint.

---

### Recommendation

Include `omniBridgeChainId` in the Borsh-encoded blob inside `deployToken`, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // domain separator
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR MPC signer must correspondingly include the destination chain ID when producing MetadataPayload signatures, matching the convention already established for `TransferMessage` payloads.

---

### Proof of Concept

```solidity
// 1. Deploy two OmniBridge instances with the same nearBridgeDerivedAddress
//    but different omniBridgeChainId values (e.g., 1 for Ethereum, 56 for BSC).
OmniBridge bridgeA = new OmniBridge(); bridgeA.initialize(impl, MPC_ADDR, 1);
OmniBridge bridgeB = new OmniBridge(); bridgeB.initialize(impl, MPC_ADDR, 56);

// 2. Obtain a legitimate deployToken signature for chain A (e.g., from a
//    real MPC signing event observed on-chain or via the relayer network).
bytes memory sig = <valid_MPC_signature_for_chainA>;
BridgeTypes.MetadataPayload memory meta = BridgeTypes.MetadataPayload({
    token: "token.near", name: "Token", symbol: "TKN", decimals: 18
});

// 3. Call deployToken on chain A — succeeds legitimately.
address tokenA = bridgeA.deployToken(sig, meta);

// 4. Replay the IDENTICAL (sig, meta) on chain B — also succeeds.
address tokenB = bridgeB.deployToken(sig, meta);

// 5. Assert spurious state on chain B.
assert(bridgeB.isBridgeToken(tokenB));          // true — unbacked token registered
assert(bridgeB.nearToEthToken("token.near") == tokenB);
// Legitimate future deployToken on chain B for "token.near" will revert with ERR_TOKEN_EXIST.
```
