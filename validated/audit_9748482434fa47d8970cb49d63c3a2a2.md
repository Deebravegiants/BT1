### Title
`max_receipt_size` Limit Not Enforced After `output_data_receivers` Appended via `promise_return` — (`runtime/runtime/src/receipt_manager.rs`)

### Summary

The `LimitConfig::max_receipt_size` field is intended as a hard cap on the serialized size of any receipt. However, a receipt that passes the size check at creation time can later have `output_data_receivers` appended to it (via `promise_return` / `promise_then` DAG rewiring) without any re-validation, causing the receipt to exceed `max_receipt_size` and be committed to chain state. The codebase explicitly acknowledges this as an open bug (issue #12606) and added a lenient `ExistingReceipt` validation mode specifically to avoid crashing when processing these oversized receipts.

### Finding Description

`validate_receipt` enforces `max_receipt_size` only when `mode == ValidateReceiptMode::NewReceipt`:

```rust
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 = borsh::object_length(receipt)...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [1](#0-0) 

The `ExistingReceipt` mode skips this check entirely, and the code comment explains why:

> "There is a bug which allows to create receipts that are above the size limit. Runtime has to handle them gracefully until the receipt size limit bug is fixed. See https://github.com/near/nearcore/issues/12606 for details." [2](#0-1) 

The root cause is in `ReceiptManager::create_action_receipt`. When a new receipt is created that depends on an existing receipt (via `promise_then` or `promise_return`), the function appends a `DataReceiver` entry to the **existing** receipt's `output_data_receivers` field:

```rust
self.action_receipts
    .get_mut(receipt_index as usize)
    .ok_or(HostError::InvalidReceiptIndex { receipt_index })?
    .output_data_receivers
    .push(DataReceiver { data_id: *data_id, receiver_id: receiver_id.clone() });
``` [3](#0-2) 

No size re-check is performed after this mutation. A contract can therefore craft a receipt whose serialized size is exactly `max_receipt_size` (passing the `NewReceipt` check), then call `promise_return` to cause `output_data_receivers` to be appended, pushing the receipt over the limit. The oversized receipt is then committed to state and propagated to other shards.

A second variant exists: returning a value as large as `max_receipt_size` from a function call wraps it in a `DataReceipt` whose total serialized size exceeds the limit, and that data receipt is also accepted.

Both variants are confirmed by integration tests that explicitly assert oversized receipts appear in chain state: [4](#0-3) [5](#0-4) 

### Impact Explanation

The corrupted protocol invariant is `max_receipt_size`: receipts exceeding this limit are accepted into chain state and forwarded across shards. This means:

- The `max_receipt_size` cap — a consensus-level parameter — is not enforced, directly analogous to the DAO `maxMembers` cap that was stored but never checked.
- Oversized receipts are processed by the runtime under the lenient `ExistingReceipt` mode, meaning they execute normally and produce state changes, gas charges, and further receipts.
- Any node that applies the chunk containing the oversized receipt must also accept it (since `ExistingReceipt` mode is used for receipts already in state), so the violation is consensus-consistent but violates the stated protocol limit.
- The corrupted value is the receipt stored in the DB / state trie, whose borsh-serialized length exceeds `limit_config.max_receipt_size`.

### Likelihood Explanation

Any unprivileged user who can deploy a contract and call it can trigger this. The attack requires:
1. Deploy a contract that uses `promise_create` + `promise_then` (or `promise_return`) with carefully sized arguments.
2. Size the inner receipt's arguments so that the receipt is exactly at `max_receipt_size` before `output_data_receivers` are appended.
3. The `promise_return` / `promise_then` call appends a `DataReceiver`, pushing the receipt over the limit with no further validation.

This is a straightforward, low-cost attack requiring only a normal signed transaction. The nearcore test suite already contains a working proof-of-concept (`test_max_receipt_size_promise_return`).

### Recommendation

Re-validate the size of every receipt in `ReceiptManager` after any mutation to `output_data_receivers`. Specifically, after the `push` in `create_action_receipt`, compute the new serialized size of the modified receipt and return an error if it exceeds `max_receipt_size`. Alternatively, enforce the limit at the point where new receipts are finalized in `function_call.rs` before they are added to `result.new_receipts`.

### Proof of Concept

The existing integration test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete proof of concept. It:

1. Deploys a contract that creates a promise DAG `[A -then-> B]`.
2. When `A` executes, it creates receipt `C` sized exactly at `max_receipt_size` and calls `promise_return(C)`.
3. The runtime appends `output_data_receivers` to `C`, making it exceed `max_receipt_size`.
4. `assert_oversized_receipt_occurred` confirms the oversized receipt appears in chain state. [6](#0-5)

#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
