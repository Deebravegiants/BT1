### Title
`accept_requests` Guard on `respond*` Functions Causes In-Flight Signature Requests to Permanently Fail on TEE Validation Failure — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond`, `respond_ckd`, and `respond_verify_foreign_tx` functions in `MpcContract` check `self.accept_requests` before processing MPC node responses. When `verify_tee()` sets `accept_requests = false` due to TEE attestation failure, all three response functions return `TeeValidationFailed`, preventing MPC nodes from delivering valid signatures for requests that were already accepted and queued. These in-flight requests then time out (~200 NEAR blocks) with no recovery mechanism.

---

### Finding Description

The `accept_requests` boolean flag is a circuit breaker: when `verify_tee()` determines that fewer than `threshold` participants have valid TEE attestations, it sets `accept_requests = false`. [1](#0-0) 

This correctly blocks new `sign()`, `request_app_private_key()`, and `verify_foreign_transaction()` calls via `check_request_preconditions`. [2](#0-1) 

However, the same flag also gates the **response path** in all three `respond*` functions: [3](#0-2) [4](#0-3) [5](#0-4) 

The problematic scenario:

1. User calls `sign()` while `accept_requests = true` → yield is created, request is queued in `pending_signature_requests`.
2. MPC nodes begin the threshold signing protocol.
3. A participant calls `verify_tee()` → one or more attestations have expired → `accept_requests = false`.
4. MPC nodes complete the signing protocol and call `respond()` with a valid signature.
5. `respond()` returns `Err(TeeError::TeeValidationFailed)` — the response is rejected.
6. The node retries, but every retry fails for the same reason.
7. After ~200 NEAR blocks, the yield-resume promise times out.
8. `return_signature_and_clean_state_on_success` fires with `Err(PromiseError::Failed)`, pops the yield, and calls `fail_on_timeout`. [6](#0-5) [7](#0-6) 

9. The user's `sign()` call fails with `RequestError::Timeout`. There is no mechanism to retry or recover the failed request.

The `verify_tee()` function is callable by any participant (not just operators), and TEE attestation expiry is a normal operational event governed by `tee_upgrade_deadline_duration_seconds`. [8](#0-7) 

---

### Impact Explanation

In-flight signature requests that were accepted while the contract was healthy cannot be completed even when MPC nodes have computed a valid signature. This breaks the request-lifecycle invariant: a request accepted by the contract (yield created, deposit taken) should be completable if the MPC nodes produce a valid response. For `verify_foreign_transaction` requests, a foreign-chain transaction that was submitted expecting MPC signing cannot be completed, and the user must restart the entire cross-chain flow from scratch.

This matches the **Medium** allowed impact: *"request-lifecycle... manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

TEE attestation expiry is a routine operational event. Any participant can call `verify_tee()` at any time. The window for in-flight requests to be affected is ~200 NEAR blocks (~200 seconds). During high-traffic periods or when attestation expiry coincides with active signing sessions, multiple requests can be simultaneously stranded. The `accept_requests = false` state can persist until all participants renew their attestations and `verify_tee()` is called again successfully — a process that may take hours.

---

### Recommendation

Remove the `accept_requests` check from `respond`, `respond_ckd`, and `respond_verify_foreign_tx`. The guard is appropriate for new request submission but must not block delivery of responses to already-accepted requests. The `accept_requests` flag should only gate the user-facing submission functions (`sign`, `request_app_private_key`, `verify_foreign_transaction`).

---

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
