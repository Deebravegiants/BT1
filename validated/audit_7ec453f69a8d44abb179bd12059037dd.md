### Title
Unconstrained `fee_proposal_fri` During V0_14_3 Transition Window Bypasses Bounds Enforcement, Corrupting Future L2 Gas Price — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `is_proposal_init_valid` function skips `fee_proposal_fri` bounds enforcement whenever `fee_actual` is `None`. The code comment documents this as a near-genesis startup exception, but `compute_fee_actual` also returns `None` during the V0_14_3 protocol upgrade transition — the `fee_proposal_window_size` (10) blocks immediately after the upgrade — because any pre-V0_14_3 block in the sliding window carries `fee_proposal_fri = None`, causing the entire window computation to short-circuit to `None`. During those 10 transition blocks, a proposer can publish any arbitrary `fee_proposal_fri` value; validators accept it without bounds enforcement, store it in `fee_proposals_window`, and commit it to the block header. Once the window fills with V0_14_3 blocks, `fee_actual` is computed from these unconstrained values, permanently inflating the `next_l2_gas_price` floor for all subsequent blocks.

---

### Finding Description

**The check path** — `is_proposal_init_valid` in `validate_proposal.rs`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(...);
    }
}
``` [1](#0-0) 

The guard is an `if let (Some(...), Some(...))` pattern. When `fee_actual` is `None`, the entire block is silently skipped — no error, no fallback bound.

**Why `fee_actual` is `None` during the transition** — `compute_fee_actual` in `dynamic_gas_price/mod.rs`:

```rust
for source_height in (start..height.0).map(BlockNumber) {
    match fee_proposals_window.get(&source_height) {
        Some(Some(price)) => window.push(*price),
        Some(None) | None => {
            warn!(...);
            return None;   // ← any None entry aborts the whole computation
        }
    }
}
``` [2](#0-1) 

Pre-V0_14_3 blocks are stored with `fee_proposal_fri = None`. For the 10 blocks immediately after the upgrade, the window `[height - 10, height - 1]` contains at least one such block, so `compute_fee_actual` returns `None` for every one of those heights.

**The execution path still uses the accepted value** — `finalize_decision` in `sequencer_consensus_context.rs`:

```rust
self.record_fee_proposal(height, init.fee_proposal_fri);
``` [3](#0-2) 

And `update_state_sync_with_new_block`:

```rust
fee_proposal_fri: init.fee_proposal_fri,
``` [4](#0-3) 

The accepted (unconstrained) `fee_proposal_fri` is stored both in the in-memory `fee_proposals_window` and in the on-chain block header. Once the window fills with V0_14_3 blocks, `compute_fee_actual` computes the median of these stored values — including any extreme values injected during the transition.

**The corrupted `fee_actual` then becomes the floor for `next_l2_gas_price`** — `calculate_next_l2_gas_price_for_fin` in `fee_market/mod.rs`:

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),
    None => config_min,
};
calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
``` [5](#0-4) 

An extreme `fee_actual` forces `effective_min` to be extreme, which drives `next_l2_gas_price` upward every block via the EIP-1559 adjustment loop.

**The mismatch is structural, mirroring the external bug:**

| | External Bug (M-4) | Sequencer Analog |
|---|---|---|
| **Check** | Uses average price → no swap needed → no incentive | Uses `fee_actual = None` → no bounds → any `fee_proposal_fri` accepted |
