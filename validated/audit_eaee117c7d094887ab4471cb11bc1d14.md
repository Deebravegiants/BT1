Looking at the external report's vulnerability class — **funds permanently frozen when a position is fully closed due to incomplete state cleanup** — I need to find an analog in Cardano Ledger where a credential/position identifier is destroyed but associated funds are not fully returned.

## Investigation

The external report's root cause: when `totalShortAmount == 0`, the collateral check passes trivially (`minCollateral == 0`), allowing a user-supplied under-value `collateralAmount` to succeed. The NFT is then burned, permanently locking the remaining collateral.

The Cardano analog would require: (1) a user specifying a smaller refund than their actual deposit, (2) a check that passes anyway, (3) the credential being removed from ledger state, and (4) the remaining deposit permanently locked.

### DRep Unregistration (`ConwayUnRegDRep`)

The `conwayDRepRefundsTxCerts` function in `eras/conway/impl/src/Cardano/Ledger/Conway/TxCert.hs` uses a wildcard `_` for the refund amount in `UnRegDRepTxCert`: [1](#0-0) 

The UTXO balance consumed-value calculation uses `lookupDRepDeposit cred` (the actual stored deposit), **not** the certificate's ref
