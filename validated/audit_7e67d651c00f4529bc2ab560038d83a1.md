### Title
Conflicting Boundary Check in `setSlippageTolerance` and `getMinAmount` Permanently Breaks `bridgeTokenToL1` When Slippage Is Set to 100% - (File: contracts/bridges/TACWETHBridge.sol)

### Summary
`TACWETHBridge.setSlippageTolerance` accepts `slippageTolerance == BASIS_POINTS_DIVISOR` (10,000) as a valid value because its guard uses strict `>`. However, `getMinAmount`, which is called internally by `bridgeTokenToL1`, computes `minAmount = 0` at that boundary and immediately reverts with `InvalidMinAmount`. This is a direct structural analog to the reported off-by-one conflict between two `require` statements: one function accepts the boundary value, the other rejects it.

### Finding Description

In `TACWETHBridge.sol`, `setSlippageTolerance` validates the new value with:

```solidity
if (newSlippageTolerance > BASIS_POINTS_DIVISOR) {
    revert InvalidSlippageTolerance();
}
```

This allows `newSlippageTolerance = 10_000` (equal to `BASIS_POINTS_DIVISOR`) to be stored successfully. [1](#0-0) 

`getMinAmount`, called internally by `bridgeTokenToL1` via `getNativeFee`, computes:

```solidity
uint256 minAmount = amount * (BASIS_POINTS_DIVISOR - slippageTolerance) / BASIS_POINTS_DIVISOR;
if (minAmount == 0) {
    revert InvalidMinAmount();
}
```

When `slippageTolerance == 10_000`, the subtraction yields zero, so `minAmount = 0` for any `amount`, and the function always reverts. [2](#0-1) 

`bridgeTokenToL1` calls `getNativeFee(amount, recipient)` as its first step, which in turn calls `getMinAmount(amount)`: [3](#0-2) 

Because `getMinAmount` is called before any token transfer, the revert occurs before any state change, but the entire bridge function is rendered permanently non-functional until the admin resets the slippage.

### Impact Explanation

When `slippageTolerance` is set to `BASIS_POINTS_DIVISOR`, every call to `bridgeTokenToL1` reverts unconditionally. The `TACWETHBridge` is the sole bridge path for WETH from TAC to ETH mainnet. No WETH can be bridged to L1 until the admin intervenes. This matches the **Low** impact class: "Contract fails to deliver promised returns, but doesn't lose value" — tokens are not stuck (the revert happens before transfer), but the bridge's core function is completely inoperable.

### Likelihood Explanation

The admin can legitimately call `setSlippageTolerance(10_000)` — the contract accepts it without error. An admin intending to set maximum slippage tolerance (e.g., to avoid bridge failures during high volatility) would naturally try `10_000` as the "100%" value, since the setter's own guard explicitly permits it. The conflict is not obvious from the setter alone.

### Recommendation

Change the guard in `setSlippageTolerance` (and the constructor) from strict `>` to `>=`:

```solidity
if (newSlippageTolerance >= BASIS_POINTS_DIVISOR) {
    revert InvalidSlippageTolerance();
}
```

This ensures `slippageTolerance` can never reach `BASIS_POINTS_DIVISOR`, making `getMinAmount` always return a non-zero value for any non-zero `amount`. [4](#0-3) [5](#0-4) 

### Proof of Concept

1. Admin calls `setSlippageTolerance(10_000)`. The check `10_000 > 10_000` is `false`, so the call succeeds and `slippageTolerance = 10_000` is stored. [1](#0-0) 

2. Any caller (bridge user or pool contract) calls `bridgeTokenToL1(recipient, 1 ether)`.

3. Internally, `getNativeFee(1 ether, recipient)` is called, which calls `getMinAmount(1 ether)`:
   - `minAmount = 1e18 * (10_000 - 10_000) / 10_000 = 0`
   - `if (minAmount == 0) revert InvalidMinAmount();` → **reverts** [6](#0-5) 

4. `bridgeTokenToL1` is permanently broken for all callers until admin resets `slippageTolerance` to a value below `10_000`. [3](#0-2)
