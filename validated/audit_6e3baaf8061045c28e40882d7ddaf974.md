### Title
Missing Minimum Confirmation Threshold Enforcement for Bitcoin Transactions — (`crates/contract/src/lib.rs`, `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` contract entry point accepts a caller-supplied `BitcoinRpcRequest.confirmations` field of type `BlockConfirmations(u64)` with no minimum value check. A value of `0` passes all contract-level guards and is forwarded verbatim to the MPC node, where the confirmation check `0 <= actual_confirmations` is trivially true for every possible `u64`. The MPC network therefore issues a threshold signature over a Bitcoin transaction that may have zero on-chain confirmations, enabling a bridge double-spend.

---

### Finding Description

**Root cause — no contract-level guard on `confirmations`:**

`BitcoinRpcRequest` is defined as:

```rust
pub struct BitcoinRpcRequest {
    pub tx_id: BitcoinTxId,
    pub confirmations: BlockConfirmations,  // plain u64 wrapper, no minimum
    pub extractors: Vec<BitcoinExtractor>,
}
``` [1](#0-0) 

`BlockConfirmations` is a transparent `u64` newtype with no invariant:

```rust
pub struct BlockConfirmations(u64);
``` [2](#0-1) 

`verify_foreign_transaction` checks only gas attachment, deposit, and chain support — it never inspects the `confirmations` value:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(...);
    if !supported_chains.contains(&requested_chain) { env::panic_str(...); }
    // ← no confirmation threshold check
    let request = args_into_verify_foreign_tx_request(request);
    self.enqueue_yield_request(...);
}
``` [3](#0-2) 

`args_into_verify_foreign_tx_request` is a pure pass-through with no validation:

```rust
pub fn args_into_verify_foreign_tx_request(args: ...) -> ... {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,       // confirmations=0 forwarded unchanged
        payload_version: args.payload_version,
    }
}
``` [4](#0-3) 

**Node-level check is trivially bypassed with `confirmations=0`:**

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
// With threshold=0: 0 <= any_u64 is always true
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
``` [5](#0-4) 

The node then proceeds to `verify_block_is_canonical` and extracts the block hash, after which the MPC network signs the payload. [6](#0-5) 

**`NotEnoughBlockConfirmations` is classified as transient** (meaning nodes retry on it), but with `threshold=0` this error is never emitted — the check always passes: [7](#0-6) 

---

### Impact Explanation

An attacker can obtain a valid MPC threshold signature attesting that a Bitcoin transaction is in a canonical block with zero required confirmations. Bitcoin transactions with 0–1 confirmations are susceptible to reorganization. This enables a bridge double-spend:

1. Attacker broadcasts a Bitcoin deposit transaction.
2. Attacker immediately calls `verify_foreign_transaction` with `confirmations=0`.
3. The MPC network signs the payload (the transaction is in a canonical block at that moment).
4. Attacker uses the signature to claim bridged assets on the NEAR side.
5. Attacker (or a colluding miner) reorganizes the Bitcoin chain, invalidating the original deposit transaction.
6. The MPC signature remains valid; the attacker retains the bridged assets while the deposit is gone.

This is a **High** impact finding: forged foreign-chain verification / light-client-style verification bypass causing invalid bridge execution and double-spend conditions.

---

### Likelihood Explanation

- Exploitable by any unprivileged caller with no special access.
- Requires only a standard `verify_foreign_transaction` call with `confirmations=0`.
- Bitcoin 0-confirmation or 1-confirmation reorganizations are realistic (selfish mining, fee-sniping, or simply natural short reorgs on mainnet).
- No threshold collusion, TEE compromise, or operator access is needed.

---

### Recommendation

Enforce a minimum confirmation threshold at the **contract level**, before the request is enqueued. For example:

```rust
const BITCOIN_MIN_CONFIRMATIONS: u64 = 6;

if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    if *btc_req.confirmations < BITCOIN_MIN_CONFIRMATIONS {
        env::panic_str(&InvalidParameters::InsufficientConfirmations { ... }.to_string());
    }
}
```

This must be done in `verify_foreign_transaction` in `crates/contract/src/lib.rs` before `enqueue_yield_request`, so that the invariant is enforced on-chain and cannot be bypassed by a node.

---

### Proof of Concept

Write a contract unit/sandbox test:

```rust
let request = VerifyForeignTransactionRequestArgs {
    domain_id: bitcoin_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: some_valid_tx_id,
        confirmations: BlockConfirmations(0),  // zero confirmations
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};
// If the contract accepts this call without panicking, the invariant is broken.
let result = contract.verify_foreign_transaction(request);
assert!(result.is_err(), "Contract must reject confirmations=0");
```

If the contract accepts the call (no `InvalidParameters` panic), the invariant is broken and the MPC network will sign for unfinalized Bitcoin transactions.
