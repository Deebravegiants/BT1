### Title
Missing ETH Value Forwarding in `coreReceiveWithData` Breaks HyperCore→Bridge Transfer Path — (`File: evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`HlBridgeToken.coreReceiveWithData()` is not `payable` and calls `IOmniBridgeInitTransfer(owner()).initTransfer(...)` with zero ETH and a hardcoded `nativeFee = 0`. When the bridge owner is `OmniBridgeWormhole`, this causes `_wormhole.publishMessage{value: 0}(...)` to revert on every `ACTION_INIT_TRANSFER` invocation, permanently breaking the HyperCore-to-bridge transfer path.

---

### Finding Description

`HlBridgeToken.coreReceiveWithData()` handles the `ACTION_INIT_TRANSFER` case by calling `initTransfer` on the owning bridge contract:

```solidity
IOmniBridgeInitTransfer(owner()).initTransfer(
    address(this),
    amount128,
    fee,
    0,          // nativeFee hardcoded to zero
    recipient,
    message
);
``` [1](#0-0) 

The call carries no `{value: ...}` and `nativeFee` is hardcoded to `0`. `coreReceiveWithData` itself is not declared `payable`, so it can never receive ETH from the HyperLiquid system address.

Inside `OmniBridge.initTransfer`, `extensionValue` is computed as:

```solidity
extensionValue = msg.value - nativeFee;   // 0 - 0 = 0
``` [2](#0-1) 

This `extensionValue = 0` is passed to `initTransferExtension`. In `OmniBridgeWormhole`, that extension calls:

```solidity
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
``` [3](#0-2) 

Wormhole's `publishMessage` requires `msg.value >= messageFee()`. With `value = 0` and any non-zero Wormhole fee, the call reverts unconditionally. Because the revert unwinds the entire transaction, the preceding `_update(_systemAddress, address(this), amount)` is also rolled back, so no tokens are permanently locked — but the entire `ACTION_INIT_TRANSFER` path is rendered permanently non-functional. [4](#0-3) 

---

### Impact Explanation

The `ACTION_INIT_TRANSFER` branch of `coreReceiveWithData` is the only on-chain path for a HyperCore user to bridge tokens out to NEAR or other chains via `OmniBridgeWormhole`. Because `coreReceiveWithData` is not `payable` and the HyperLiquid system address does not attach ETH to the callback, there is no mechanism to supply the Wormhole fee. Every attempt reverts. This constitutes a **permanently frozen redemption path** for all HyperCore-originated bridge transfers — fitting the Critical impact category of "frozen redemption path or permanently unclaimable user or protocol value."

---

### Likelihood Explanation

Any HyperCore user who calls `sendToEvmWithData` with `ACTION_INIT_TRANSFER` data triggers this path. No privilege is required. The failure is deterministic whenever `IWormhole.messageFee() > 0`, which is the normal production state of Wormhole. Likelihood is **High**.

---

### Recommendation

1. Make `coreReceiveWithData` `payable` so it can receive ETH from the system address (if the HyperLiquid system supports attaching value to the callback).
2. Forward the received ETH to `initTransfer`:
   ```solidity
   IOmniBridgeInitTransfer(owner()).initTransfer{value: msg.value}(
       address(this),
       amount128,
       fee,
       uint128(msg.value),   // nativeFee = forwarded ETH
       recipient,
       message
   );
   ```
3. If the HyperLiquid system address cannot attach ETH to the callback, implement an alternative fee-funding mechanism (e.g., a pre-funded fee reserve held by `HlBridgeToken` that is drawn from on each `ACTION_INIT_TRANSFER`).

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` with a live Wormhole endpoint where `messageFee() > 0`.
2. Deploy `HlBridgeToken` owned by the `OmniBridgeWormhole` instance.
3. From HyperCore, call `sendToEvmWithData` targeting `HlBridgeToken` with `data = abi.encodePacked(uint8(1), abi.encode(fee, recipient, message))` (`ACTION_INIT_TRANSFER = 1`).
4. The HyperLiquid system address invokes `coreReceiveWithData(from, ..., amount, data)` with no ETH.
5. Execution reaches `_wormhole.publishMessage{value: 0}(...)`.
6. Wormhole reverts with insufficient fee; the entire transaction reverts.
7. The bridge transfer never completes; the path is permanently broken for all users. [5](#0-4) [6](#0-5)

    function coreReceiveWithData(
        address from,
        bytes32 /*destinationRecipient*/,
        uint32 /*destinationChainId*/,
        uint256 amount,
        uint64 /*coreNonce*/,
        bytes calldata data
    ) external override {
        if (msg.sender != _systemAddress) revert NotSystemAddress();
        if (data.length == 0) revert EmptyActionData();

        uint8 action = uint8(data[0]);
        bytes calldata tail = data[1:];

        if (action == ACTION_TRANSFER) {
            address recipient = abi.decode(tail, (address));
            _update(_systemAddress, recipient, amount);
        } else if (action == ACTION_INIT_TRANSFER) {
            (uint128 fee, string memory recipient, string memory message) = abi
                .decode(tail, (uint128, string, string));
            uint128 amount128 = amount.toUint128();
            _update(_systemAddress, address(this), amount);
            IOmniBridgeInitTransfer(owner()).initTransfer(
                address(this),
                amount128,
                fee,
                0,
                recipient,
                message
            );
        } else {
            revert UnknownAction(action);
        }

        emit CoreReceived(from, action, amount, data);
    }
