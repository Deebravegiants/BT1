### Title
Unchecked `.unwrap()` on Empty Domain Registry Panics and Blocks `vote_new_parameters` During Resharing - (File: `crates/contract/src/state/resharing.rs`)

---

### Summary

`ResharingContractState::vote_new_parameters` unconditionally calls `.get_domain_by_index(0).unwrap()` on the previous running state's domain registry. If that registry is empty (zero domains), the call returns `None` and the `.unwrap()` panics, permanently blocking any re-proposal vote during an active resharing. This is the direct structural analog to GoGoPool M-21, where `allotment / enabledCount` panicked when `enabledCount == 0`.

---

### Finding Description

In `crates/contract/src/state/resharing.rs`, the `vote_new_parameters` method on `ResharingContractState` constructs a new `KeyEvent` for the re-proposal:

```rust
resharing_key: KeyEvent::new(
    self.prospective_epoch_id().next(),
    self.previous_running_state
        .domains
        .get_domain_by_index(0)   // returns Option<&DomainConfig>
        .unwrap()                  // panics if domains is empty
        .clone(),
    proposal.parameters().clone(),
),
``` [1](#0-0) 

`get_domain_by_index(0)` is defined to return `Option<&DomainConfig>`:

```rust
pub fn get_domain_by_index(&self, index: usize) -> Option<&DomainConfig> {
    self.domains.get(index)
}
``` [2](#0-1) 

The contract's `init()` function initializes the protocol state with `DomainRegistry::default()` — an empty domain list — meaning a contract can legally be in a `Running` state with zero domains:

```rust
protocol_state: ProtocolContractState::Running(RunningContractState::new(
    DomainRegistry::default(),   // zero domains
    Keyset::new(EpochId::new(0), Vec::new()),
    parameters,
    AddDomainsVotes::default(),
)),
``` [3](#0-2) 

A parallel unchecked call also exists in `RunningContractState` (one match in `running.rs`), meaning the same panic can be triggered from the running state's own `vote_new_parameters` path before resharing even begins. [4](#0-3) 

---

### Impact Explanation

When `previous_running_state.domains` is empty and a participant calls `vote_new_parameters` during an active resharing (a re-proposal), the NEAR runtime receives a panic and the transaction reverts. Because the resharing state is not mutated before the panic, the contract is stuck: no re-proposal can ever succeed, and the resharing cannot be cancelled through the normal `vote_cancel_resharing` path either (which requires threshold votes from the *previous* running participants — a set that may itself be the reason resharing was initiated). This permanently freezes the participant-set management lifecycle, matching the "contract execution-flow manipulation that breaks production safety/accounting invariants" impact tier.

---

### Likelihood Explanation

The contract is explicitly designed to start with zero domains (`init()` uses `DomainRegistry::default()`). Domains are added later via `vote_add_domains`. If a governance action to change participants (`vote_new_parameters`) reaches quorum before any domain has been added — a realistic race condition during initial network setup — the resharing state is entered with an empty domain registry. Any subsequent re-proposal vote then triggers the panic. This requires no privileged access: any participant can cast the vote that crosses the quorum threshold.

---

### Recommendation

Guard the `get_domain_by_index(0)` call with an explicit check and return a graceful error instead of panicking:

```rust
let first_domain = self
    .previous_running_state
    .domains
    .get_domain_by_index(0)
    .ok_or(DomainError::NoDomainsDefined)?
    .clone();
```

Apply the same fix to the analogous call in `RunningContractState::vote_new_parameters`. Additionally, consider whether `vote_new_parameters` should be rejected outright when the domain registry is empty, since resharing without any domain to reshare is semantically undefined.

---

### Proof of Concept

1. Deploy the contract via `init()` with N participants and governance threshold T. Domain registry is empty (`DomainRegistry::default()`).
2. T participants call `vote_new_parameters` with a new participant set before any `vote_add_domains` succeeds. The running state transitions to `ResharingContractState` with `previous_running_state.domains` empty.
3. Any participant now calls `vote_new_parameters` on the resharing state (a re-proposal).
4. Execution reaches `self.previous_running_state.domains.get_domain_by_index(0).unwrap()`.
5. `get_domain_by_index(0)` returns `None`; `.unwrap()` panics.
6. The transaction reverts. The resharing state is unchanged. No re-proposal can ever succeed. The participant-set update lifecycle is permanently blocked. [5](#0-4) [6](#0-5)
