Audit Report

## Title
Stale Oracle Rate Causes Premature Daily Mint Limit Exhaustion via Block Stuffing — (`contracts/pools/oracle/InterimRSETHOracle.sol`, `contracts/pools/RSETHPoolV2.sol`)

## Summary

`InterimRSETHOracle` stores the rsETH/ETH rate with no on-chain staleness enforcement. On zkSync, an attacker can stuff blocks to delay the `MANAGER_ROLE` holder's `setRate()` call. While the rate is stale-low, every `RSETHPoolV2.deposit()` inflates the computed `rsETHAmount` and accumulates it against `dailyMintAmount`, exhausting `dailyMintLimit` prematurely and reverting all subsequent deposits with `DailyMintLimitExceeded` for the remainder of the day.

## Finding Description

`InterimRSETHOracle` holds a single `uint256 public rate` with no timestamp or heartbeat. `getRate()` returns whatever was last written by `setRate()`, regardless of age. [1](#0-0) [2](#0-1) 

`RSETHPoolV2.viewSwapRsETHAmountAndFee` divides the ETH amount by this rate to compute `rsETHAmount`: [3](#0-2) 

The `limitDailyMint` modifier accumulates `rsETHAmount` into `dailyMintAmount` and reverts if it exceeds `dailyMintLimit`: [4](#0-3) 

When `rsETHToETHrate` is stale-low (e.g., `1e18` instead of the correct `1.1e18`), the division yields a ~10% larger `rsETHAmount` per ETH. Each deposit therefore consumes more of `dailyMintLimit` than it should.

**Attack path on zkSync:**
1. rsETH appreciates; the manager prepares a `setRate(1.1e18)` transaction.
2. The attacker stuffs consecutive blocks with high-gas filler transactions, exploiting zkSync's lower per-block gas limit, preventing the manager's `setRate` call from being included.
3. The rate remains at `1e18` (stale-low).
4. Legitimate users deposit ETH; each deposit mints ~10% more rsETH than correct, accumulating `dailyMintAmount` ~10% faster.
5. `dailyMintLimit` is hit after only ~91% of the intended ETH capacity is deposited.
6. All subsequent `deposit()` calls revert with `DailyMintLimitExceeded` until the next day resets `dailyMintAmount`. [5](#0-4) 

## Impact Explanation

Legitimate depositors are denied access to `RSETHPoolV2.deposit()` for the remainder of the day. No funds are lost, but the protocol fails to deliver its promised daily throughput. This matches **Low — Block stuffing** in the allowed impact scope.

## Likelihood Explanation

- zkSync Era's block gas limit is materially lower than Ethereum mainnet, reducing the cost of sustained block stuffing.
- The attack window is bounded: it only needs to last long enough to keep the rate stale while the daily limit is consumed by organic depositors.
- `InterimRSETHOracle` is explicitly an interim manual oracle, making periodic rate updates a routine, predictable operation an attacker can anticipate.
- No privileged access is required on the attacker's side; only ETH to fund filler transactions.

## Recommendation

1. **Add a staleness deadline**: record `lastRateUpdate` in `InterimRSETHOracle` and revert in `viewSwapRsETHAmountAndFee` or in `limitDailyMint` if `block.timestamp - lastRateUpdate > MAX_RATE_AGE`.
2. **Denominate `dailyMintLimit` in ETH, not rsETH**: accumulate `amountAfterFee` rather than `rsETHAmount` in `dailyMintAmount`. This makes the limit rate-independent and eliminates the inflation vector entirely.
3. **Use a push-based oracle with on-chain freshness** (e.g., Chainlink with a heartbeat check) rather than a purely manual setter.

## Proof of Concept

```solidity
// Foundry test (local or zkSync fork)
// oracle.setRate(1e18);          // stale rate (manager tx blocked)
// pool.setDailyMintLimit(100e18);

// At stale rate: 1 ETH → 1e18 rsETH
// At correct rate (1.1e18): 1 ETH → ~0.909e18 rsETH

// Deposit 100 ETH in 100 × 1 ETH calls at stale rate:
for (uint i = 0; i < 100; i++) {
    pool.deposit{value: 1 ether}("ref");
}
// dailyMintAmount == 100e18 == dailyMintLimit → limit exhausted

// One more deposit reverts, even though at correct rate only ~90.9e18 rsETH
// would have been consumed (leaving ~9.1e18 rsETH / ~10 ETH of headroom):
vm.expectRevert(RSETHPoolV2.DailyMintLimitExceeded.selector);
pool.deposit{value: 1 ether}("ref");
```

This test runs on unmodified production code. The only precondition is that the oracle rate has not been updated (simulating a successful block-stuffing window), which is directly triggerable by an unprivileged attacker on zkSync.

```text
    modifier limitDailyMint(uint256 amount) {
        if (block.timestamp < startTimestamp) {
            revert MintBeforeStartTimestamp();
        }

        // Calculate the amount of rsETH that will be minted
        (uint256 rsETHAmount,) = viewSwapRsETHAmountAndFee(amount);
        uint256 currentDay = getCurrentDay();

        // If the current day is greater than the last mint day, reset the daily mint amount
        if (currentDay > lastMintDay) {
            lastMintDay = currentDay;
            dailyMintAmount = 0;
        }

        // Check if the daily mint amount plus the amount to mint is greater than the daily mint limit
        if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
            revert DailyMintLimitExceeded();
        }

        dailyMintAmount += rsETHAmount;
        _;
    }
```

```text
    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
```
