### Title
Missing Minimum `BlockConfirmations` Validation Allows MPC Signature Over Unconfirmed Bitcoin Transactions - (File: crates/contract/src/lib.rs, crates/foreign-chain-inspector/src/bitcoin/inspector.rs)

### Summary
The `verify_foreign_transaction` contract entry point accepts a `BitcoinRpcRequest` with `confirmations: 0` without any on-chain validation. Because the node-side inspector check is `block_confirmations_threshold <= actual_confirmations`, a threshold of 0 always passes for any transaction — including unconfirmed mempool transactions. An unprivileged caller can therefore obtain a valid MPC threshold signature attesting to a Bitcoin transaction that has never been confirmed, enabling double-spend attacks against any bridge that relies on this attestation.

### Finding Description
`BlockConfirmations` is defined as a plain `u64` newtype with no enforced minimum:

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI schema exposes `"minimum": 0.0`, meaning 0 is a valid on-wire value. The `verify_foreign_transaction` function enqueues the request without inspecting the `confirmations` field:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(...);          // checks domain, gas, deposit
    // chain whitelist check only
    if !supported_chains.contains(&requested_chain) { ... }
    // no confirmations validation
    let request = args_into_verify_foreign_tx_request(request);
    self.enqueue_yield_request(...);
}
```

On the node side, `execute_foreign_chain_request` passes the caller-supplied value directly to the inspector:

```rust
let block_confirmations = request.confirmations.0.into();
let extracted_values = inspector
    .extract(transaction_id, block_confirmations, extractors)
    ...
```

Inside `BitcoinInspector::extract`, the finality guard is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

With `block_confirmations_threshold = 0`, the condition `0 <= N` is trivially true for every `N ≥ 0`, including 0 (mempool / unconfirmed). Bitcoin's `getrawtransaction` returns mempool transactions with `confirmations: 0`, so the inspector succeeds, extracts the block hash, and the leader node signs the `ForeignTxSignPayload` containing the full request (including `confirmations: 0`). The contract's `respond_verify_foreign_tx` validates only the ECDSA signature against the root public key — it does not re-examine the `confirmations` field — and resolves the yield, returning the signature to the caller.

### Impact Explanation
The MPC network issues a valid threshold signature over a payload that includes `confirmations: 0`. Any bridge contract (e.g., Omnibridge inbound flow) that consumes this signature to release NEAR-side assets without independently enforcing a minimum confirmation count will process the release for a Bitcoin transaction that has not yet been mined into a block. The attacker can then broadcast a conflicting transaction (double-spend) to reclaim the Bitcoin, while keeping the NEAR-side assets. This constitutes direct theft of funds controlled by the bridge flow that the MPC network is supposed to protect.

**Allowed impact matched:** *High — forged foreign-chain verification / light-client-style verification bypass that causes invalid bridge execution or double-spend conditions.*

### Likelihood Explanation
- Any unprivileged NEAR account can call `verify_foreign_transaction` with `confirmations: 0` and a valid `tx_id` of a mempool transaction they just broadcast.
- The contract accepts the request (only 1 yoctoNEAR deposit required).
- All MPC nodes will independently query the Bitcoin RPC, find the mempool transaction with 0 confirmations, pass the `0 <= 0` check, and produce signature shares.
- No collusion, privileged access, or TEE bypass is required.

### Recommendation
Enforce a protocol-level minimum confirmation count on-chain inside `verify_foreign_transaction`, before the request is enqueued:

```rust
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    const MINIMUM_BITCOIN_CONFIRMATIONS: u64 = 1; // or a configurable floor
    if btc_req.confirmations.0 < MINIMUM_BITCOIN_CONFIRMATIONS {
        env::panic_str("Bitcoin confirmations must be at least 1");
    }
}
```

Alternatively, store a per-chain minimum confirmation floor in the on-chain `ChainEntry` (voted in alongside the RPC provider whitelist) and validate against it at request-submission time, so the network can raise the floor without a contract upgrade.

### Proof of Concept
1. Attacker broadcasts a Bitcoin transaction `TX` sending funds to a bridge deposit address. `TX` is in the mempool with 0 confirmations.
2. Attacker calls `verify_foreign_transaction` on the MPC contract with:
   ```json
   {
     "request": {
       "Bitcoin": {
         "tx_id": "<TX hash>",
         "confirmations": 0,
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": <foreign_tx_domain>,
     "payload_version": 1
   }
   ```
   with 1 yoctoNEAR attached. The contract enqueues the request.
3. MPC nodes call `getrawtransaction` on Bitcoin RPC. The mempool transaction is returned with `confirmations: 0`. The inspector check `0 <= 0` passes. Nodes sign `ForeignTxSignPayload { request: { confirmations: 0, ... }, values: [BlockHash(...)] }`.
4. `respond_verify_foreign_tx` is called; the contract validates the ECDSA signature (valid) and resolves the yield, returning the signature to the attacker.
5. Attacker submits the signature to the bridge contract to claim NEAR-side assets.
6. Attacker broadcasts a double-spend of `TX` on Bitcoin, reclaiming the Bitcoin.

        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,

            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(

        let callback_args = serde_json::to_vec(&(&request,)).unwrap();

            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
