Audit Report

## Title
Stale Cross-Chain Rate in `getRate()` Enables Excess rsETH Distribution to Depositors — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` variable unconditionally, with no check against `lastUpdated`. All L2 deposit pools use this rate to compute how many rsETH tokens to transfer per deposited ETH. Because rsETH is a yield-bearing token whose rate monotonically increases, a stale (lower-than-current) rate causes pools to transfer more rsETH than the deposited ETH is worth at the true current rate, diluting existing holders' accrued yield in favor of the depositor.

## Finding Description

`CrossChainRateReceiver` stores two fields: `rate` (the last received rsETH/ETH exchange rate) and `lastUpdated` (the timestamp of the last LayerZero delivery). [1](#0-0) 

The `getRate()` function returns `rate` with no staleness check: [2](#0-1) 

`rate` is only refreshed when a LayerZero message arrives via `lzReceive()`, which is triggered by an off-chain call to the permissionless but manually-initiated, fee-paying `MultiChainRateProvider.updateRate()`: [3](#0-2) [4](#0-3) 

There is a second layer of caching: `RSETHMultiChainRateProvider.getLatestRate()` reads `ILRTOracle(rsETHPriceOracle).rsETHPrice()`, which is itself a stored value only updated when `LRTOracle.updateRSETHPrice()` is called on L1: [5](#0-4) [6](#0-5) 

Every L2 deposit pool calls `getRate()` to compute the rsETH amount to transfer. In `RSETHPoolNoWrapper`: [7](#0-6) [8](#0-7) 

The minting formula `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` means a stale-low denominator produces a larger rsETH output than the deposited ETH warrants at the true current rate.

The same pattern is present in `RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge`: [9](#0-8) [10](#0-9) 

The contrast with `ChainlinkOracleForRSETHPoolCollateral.getRate()` is instructive — that oracle reverts on stale data, but `CrossChainRateReceiver.getRate()` has no equivalent guard: [11](#0-10) 

## Impact Explanation

rsETH accrues staking yield continuously; its price monotonically increases. When the L2 rate is stale-low, a depositor receives more rsETH tokens than their ETH is worth at the true current rate. Each rsETH token is a proportional claim on the underlying ETH held in the L1 protocol. The excess rsETH transferred to the depositor represents a claim on ETH that belongs to existing rsETH holders (their accrued, unredeemed yield). When the depositor later redeems or sells, they extract value that should have accrued to prior holders.

**Impact: High — Theft of unclaimed yield from existing rsETH/wrsETH holders.**

## Likelihood Explanation

- `updateRate()` is permissionless but requires paying LayerZero fees and manually initiating the call; there is no on-chain enforcement of update frequency.
- During LayerZero congestion, fee spikes, or simply infrequent keeper activity, the L2 rate can lag by hours or days.
- The L2 rate can additionally lag the true L1 rate by the L1 oracle update delay (`updateRSETHPrice()` must also be called first).
- Any unprivileged user can compare L1 `rsETHPrice` against L2 `CrossChainRateReceiver.rate` and deposit when the gap is largest — no special access or front-running required.
- The attack is repeatable across every stale window and scales linearly with deposit size.

**Likelihood: Medium** — stale windows occur regularly given the manual, fee-paying update mechanism.

## Recommendation

Add a configurable staleness threshold to `CrossChainRateReceiver.getRate()`, reverting if `block.timestamp - lastUpdated` exceeds the maximum allowed staleness:

```solidity
uint256 public maxStaleness; // e.g., 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This mirrors the staleness-guard pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`. The `maxStaleness` value should be set conservatively (e.g., 24 hours) and be owner-configurable. Additionally, consider automating `updateRate()` calls via a keeper or Gelato task to minimize the stale window.

## Proof of Concept

1. L1 `LRTOracle.rsETHPrice` is updated to `1.05e18` via `updateRSETHPrice()`.
2. No one calls `RSETHMultiChainRateProvider.updateRate()` to propagate the update (LayerZero fee not paid, or congestion).
3. L2 `CrossChainRateReceiver.rate` still holds the old value `1.04e18`.
4. Attacker calls `RSETHPoolNoWrapper.deposit{value: 100 ether}("")`.
5. Pool computes: `rsETHAmount = 100e18 * 1e18 / 1.04e18 ≈ 96.15 rsETH`.
6. At the true rate of `1.05e18`, the correct amount is: `100e18 * 1e18 / 1.05e18 ≈ 95.24 rsETH`.
7. Attacker receives `≈ 0.91 rsETH` excess — worth `≈ 0.955 ETH` at the true rate — extracted from existing holders' accrued yield.
8. Attacker repeats for every stale window, scaling with deposit size.

**Foundry fork test plan**: Fork the target L2 chain. Deploy or mock `CrossChainRateReceiver` with `rate = 1.04e18` and `lastUpdated` set to a timestamp 25+ hours in the past. Deploy `RSETHPoolNoWrapper` pointing to this oracle. Call `deposit{value: 100 ether}`. Assert that `rsETHAmount` received exceeds `100e18 * 1e18 / 1.05e18` (the correct amount at the true L1 rate), confirming the excess transfer.

        rate = latestRate;

        for (uint256 i; i < rateReceiversLength;) {

            bytes memory remoteAndLocalAddresses = abi.encodePacked(rateReceivers[i]._contract, address(this));

                .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

            );

            unchecked {
                ++i;
            }
        }

        emit RateUpdated(rate);
