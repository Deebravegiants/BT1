### Title
`directDeposits` Omitted from Produced-Value Accounting Enables ADA Creation from Thin Air — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the `directDeposits` field of a transaction body is applied to staking-account balances by the `ENTITIES` / `SUBENTITIES` rules, but the corresponding coin amounts are absent from both `getConsumedDijkstraValue` and `dijkstraProducedValue` / `dijkstraSubTxProducedValue`. The value-conservation check (`consumed == produced`) therefore passes without accounting for the deposited coin, allowing an unprivileged sender to mint arbitrary ADA into registered staking accounts.

---

### Finding Description

**Vulnerability class:** funds/accounting bug — a component of the "produced" side of the preservation-of-value equation is missing, analogous to the `shareProfitAmts` omission in the reference report.

**Background.** The Dijkstra era introduces `directDeposits :: DirectDeposits` as a new field in both `DijkstraTxBodyRaw TopTx` and `DijkstraSubTxBodyRaw`. When a transaction is processed, the `ENTITIES` rule calls `applyDirectDeposits`, which **adds** coin to the targeted staking accounts:

```haskell
-- libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [1](#0-0) 

This is semantically identical to a treasury donation: coin leaves the transaction and enters a ledger pot. Treasury donations are correctly included in the produced value:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs
conwayProducedValue pp isStakePool txBody =
  getProducedMaryValue pp isStakePool txBody
    <+> inject (txBody ^. treasuryDonationTxBodyL)
``` [2](#0-1) 

**The gap.** `dijkstraProducedValue` (top-level) delegates to `conwayProducedValue` and then folds over sub-transactions, but never adds `directDeposits`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)
``` [3](#0-2) 

`dijkstraSubTxProducedValue` (sub-transaction) includes outputs, deposits, treasury donation, and burned multi-assets — but again omits `directDeposits`:

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody <> txBody ^. treasuryDonationTxBodyL)
    <> burnedMultiAssets txBody
``` [4](#0-3) 

`getConsumedDijkstraValue` also does not include direct deposits on the consumed side:

```haskell
txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
``` [5](#0-4) 

**Consequence.** The value-conservation predicate enforced by `validateValueNotConservedUTxO` reduces to:

```
inputs + withdrawals + refunds  ==  outputs + fees + deposits + treasury_donation
``` [6](#0-5) 

`directDeposits` appear on neither side. A transaction that satisfies the standard balance equation passes the check regardless of how large its `directDeposits` are. The `ENTITIES` rule then unconditionally adds those amounts to the targeted accounts:

```haskell
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [7](#0-6) 

The coin credited to accounts is never deducted from the UTxO or any other pot, violating the preservation-of-value invariant.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker can mint an unbounded amount of ADA into any set of registered staking accounts in a single Dijkstra transaction. Because the ledger's preservation-of-value check does not cover `directDeposits`, every honest node accepts the transaction and permanently records the inflated account balances. This constitutes a direct, irreversible creation of ADA outside the protocol's monetary policy.

---

### Likelihood Explanation

**High.** The attack requires only:
1. A valid Dijkstra-era transaction (no special role or key).
2. At least one registered staking account as the deposit target (the attacker can register their own).
3. A transaction body that balances under the standard equation (trivially achievable).

No governance majority, privileged key, or external dependency is needed. The attack is deterministic and repeatable.

---

### Recommendation

Add the sum of `directDeposits` amounts to the produced value, mirroring the treatment of `treasuryDonationTxBodyL`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs

dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> inject (foldMap id . unDirectDeposits $ txBody ^. directDepositsTxBodyL)  -- ADD
    <> foldMap'
         (getProducedValue pp isRegPoolId . view bodyTxL)
         (txBody ^. subTransactionsTxBodyL)

dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject ( getTotalDepositsTxBody pp isRegPoolId txBody
             <> txBody ^. treasuryDonationTxBodyL
             <> foldMap id (unDirectDeposits $ txBody ^. directDepositsTxBodyL) )  -- ADD
    <> burnedMultiAssets txBody
```

The same correction must be applied to any future sub-transaction produced-value function that handles `directDeposits`.

### Proof of Concept

```
1. Register staking account A (credential cred_A).

2. Build a Dijkstra TopTx:
     inputs:          { utxo_in  (value = 10 ADA) }
     outputs:         { utxo_out (value = 9 ADA)  }
     fee:             1 ADA
     directDeposits:  { AccountAddress(cred_A) → 1_000_000 ADA }

3. Standard balance check:
     consumed = 10 ADA
     produced = 9 ADA + 1 ADA = 10 ADA   ✓  (directDeposits not counted)

4. ENTITIES rule executes applyDirectDeposits:
     balance(cred_A) += 1_000_000 ADA

5. Result: 1_000_000 ADA created from thin air; total ADA supply increased by
   1_000_000 ADA with no corresponding deduction from any ledger pot.
```

validateValueNotConservedUTxO ::

  PParams era ->
  UTxO era ->
  CertState era ->
  TxBody TopTx era ->
  Test (ShelleyUtxoPredFailure era)
validateValueNotConservedUTxO pp utxo certState txBody =
  failureUnless (consumedValue == producedValue) $
    ValueNotConservedUTxO Mismatch {mismatchSupplied = consumedValue, mismatchExpected = producedValue}
  where
    consumedValue = consumed pp certState utxo txBody
    producedValue = produced pp certState txBody
