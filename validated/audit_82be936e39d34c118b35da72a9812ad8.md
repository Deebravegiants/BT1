### Title
Missing Time-Based Staleness Check in `ChainlinkOracleForRSETHPoolCollateral.getRate()` Enables Excess rsETH Minting and Daily Limit Exhaustion — (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

---

### Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` only validates `answeredInRound < roundID` but performs no time-based freshness check (`block.timestamp - updatedAt > heartbeat`). When the collateral oracle returns a stale, inflated price, any depositor can receive excess rsETH proportional to the price inflation, and the `limitDailyMint` modifier simultaneously exhausts the daily cap using the same inflated figure.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` fetches the Chainlink price and applies only the round-completeness guard:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol
if (answeredInRound < roundID) revert StalePrice();   // line 30
if (timestamp == 0)            revert IncompleteRound(); // line 31
if (ethPrice <= 0)             revert InvalidPrice();    // line 32
``` [1](#0-0) 

There is no check of the form `block.timestamp - timestamp > MAX_DELAY`. A price that was answered in the same round it was started (so `answeredInRound == roundID`) but was last updated many hours ago passes all three guards and is returned as current.

This oracle is registered as the collateral-token oracle in pool contracts via `supportedTokenOracle[token]`. When a token deposit is processed, `viewSwapRsETHAmountAndFee(amount, token)` is called:

```solidity
// contracts/pools/RSETHPoolV3.sol  lines 328-334
uint256 rsETHToETHrate  = getRate();                                          // rsETH oracle (current)
uint256 tokenToETHRate  = IOracle(supportedTokenOracle[token]).getRate();     // collateral oracle (potentially stale)
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

The `limitDailyMint` modifier calls this same function to compute the rsETH amount it will count against the daily cap:

```solidity
// contracts/pools/RSETHPoolV3.sol  lines 101-123
(rsETHAmount,) = viewSwapRsETHAmountAndFee(amount, token);
...
if (dailyMintAmount + rsETHAmount > dailyMintLimit) revert DailyMintLimitExceeded();
dailyMintAmount += rsETHAmount;
``` [3](#0-2) 

After the modifier, the deposit body calls `viewSwapRsETHAmountAndFee` a second time and mints the same inflated `rsETHAmount` to the caller:

```solidity
// contracts/pools/RSETHPoolV3.sol  lines 286-290
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
...
wrsETH.mint(msg.sender, rsETHAmount);
``` [4](#0-3) 

Because both the modifier and the body read the same stale oracle, the inflated amount is both minted to the attacker and charged against the daily limit.

The same pattern is present in `RSETHPoolV3ExternalBridge` and `RSETHPoolV3WithNativeChainBridge`. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Critical — Direct theft of user funds.**

The attacker deposits real collateral tokens (e.g., wstETH) and receives wrsETH (wrapped rsETH) in excess of the true collateral value. wrsETH is redeemable 1:1 for rsETH on L1, which is backed by the protocol's L1 collateral pool. The attacker can bridge the excess wrsETH to L1 and redeem it for more collateral than they deposited, extracting value from other protocol participants.

**Secondary — Medium: Temporary freezing of funds.**

The inflated rsETHAmount is charged against `dailyMintAmount`. A single deposit with a 2× stale price exhausts the daily cap at half the intended deposit volume, blocking all other users from depositing for the remainder of the day.

---

### Likelihood Explanation

Chainlink price feeds can become stale due to network congestion, sequencer downtime (relevant on L2), or feed-specific heartbeat windows. The `answeredInRound == roundID` condition is satisfied for any price that was answered in its own round, regardless of age. No attacker action is required to cause staleness; the attacker only needs to observe the oracle state and submit a deposit while the stale high price is active. This is a passive, permissionless exploit path.

---

### Recommendation

Add a configurable maximum age check in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
uint256 public constant MAX_PRICE_AGE = 3600; // e.g. 1 hour; tune per feed heartbeat

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID)              revert StalePrice();
    if (timestamp == 0)                         revert IncompleteRound();
    if (ethPrice <= 0)                          revert InvalidPrice();
    if (block.timestamp - timestamp > MAX_PRICE_AGE) revert StalePrice(); // ADD THIS

    uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
    return normalizedPrice;
}
```

The `MAX_PRICE_AGE` value should be set per-feed based on the Chainlink heartbeat for that price pair (e.g., 3600 s for a 1-hour heartbeat feed).

### Proof of Concept

```
Setup:
  dailyMintLimit = 1000e18 rsETH
  true wstETH/ETH rate = 1.1e18
  stale wstETH/ETH rate returned by oracle = 2.2e18  (2× true, answeredInRound == roundID, timestamp = 12 hours ago)
  rsETH/ETH rate (current) = 1.05e18
  feeBps = 0

Attacker deposits 500e18 wstETH:
  tokenToETHRate  = 2.2e18   (stale)
  rsETHToETHrate  = 1.05e18  (current)
  rsETHAmount     = 500e18 * 2.2e18 / 1.05e18 ≈ 1047e18

limitDailyMint check:
  dailyMintAmount + 1047e18 > 1000e18  → revert DailyMintLimitExceeded

Attacker adjusts to 450e18 wstETH:
  rsETHAmount = 450e18 * 2.2e18 / 1.05e18 ≈ 942e18
  dailyMintAmount becomes 942e18  (94.2% of cap consumed)

True rsETH owed for 450e18 wstETH at correct rate:
  = 450e18 * 1.1e18 / 1.05e18 ≈ 471e18

Attacker profit: 942e18 − 471e18 ≈ 471e18 rsETH stolen
Remaining daily cap: 1000e18 − 942e18 = 58e18 rsETH
All other users are effectively locked out for the day.
```

The `answeredInRound < roundID` guard at line 30 of `ChainlinkOracleForRSETHPoolCollateral.sol` passes because the stale round was self-answered. No admin action, oracle compromise, or governance capture is required. [7](#0-6)
