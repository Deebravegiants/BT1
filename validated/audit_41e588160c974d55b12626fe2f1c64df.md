### Title
Stale Chainlink Price Accepted in `getAssetPrice` Enables Deposit-at-Inflated-Price / Withdraw-at-True-Price Arbitrage — (`contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but discards every safety field (`updatedAt`, `answeredInRound`, `roundId`). When a Chainlink feed is stale and shows a price higher than reality, an attacker can deposit an LST at the inflated price (minting excess rsETH), wait for the feed to self-correct, call the permissionless `updateRSETHPrice()` to push `rsETHPrice` downward, then initiate a withdrawal that redeems the excess rsETH for more underlying than was deposited. The profit is extracted from honest depositors.

---

### Finding Description

**Root cause — `ChainlinkPriceOracle.getAssetPrice`** [1](#0-0) 

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`updatedAt`, `answeredInRound`, and `roundId` are all silently discarded. No heartbeat check, no `answeredInRound >= roundId` guard, no zero-price guard.

Compare with the sibling contract in the same repo that **does** implement these checks: [2](#0-1) 

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The inconsistency confirms the omission in `ChainlinkPriceOracle` is unintentional.

**How the stale price propagates into minting**

`LRTDepositPool.getRsETHAmountToMint` uses the live oracle price: [3](#0-2) 

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If `getAssetPrice(asset)` returns a stale-high value, the attacker receives more rsETH than the true value of their deposit warrants.

**How `updateRSETHPrice` locks in the loss for other depositors**

`updateRSETHPrice` is permissionless: [4](#0-3) 

`_getTotalEthInProtocol` re-prices all assets at the now-corrected oracle price: [5](#0-4) 

Because the attacker's deposit added real tokens but was valued at the stale-high price, the corrected TVL is lower per rsETH share, so `rsETHPrice` drops.

**How the withdrawal converts the excess rsETH into profit**

`getExpectedAssetAmount` in `LRTWithdrawalManager`: [6](#0-5) 

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

At this point `rsETHPrice` is the corrected (lower) value and `getAssetPrice(asset)` is the true (lower) price. The ratio `rsETHPrice / assetPrice` is still inflated relative to the attacker's true deposit value, so `underlyingToReceive > deposit_amount`.

**The `_calculatePayoutAmount` min() does not help** [7](#0-6) 

The min is taken between `expectedAssetAmount` (already inflated, set at initiation) and `currentReturn`. If prices are stable between initiation and unlock, both values are equal and the attacker receives the full inflated amount.

**The `pricePercentageLimit` downside-pause does not reliably prevent the attack** [8](#0-7) 

The pause only fires when the price drop exceeds `pricePercentageLimit`. If the limit is 0 (unset) or the stale deviation is within the limit (e.g., a 0.5 % stale price with a 1 % limit), the price update goes through and the attack succeeds.

---

### Impact Explanation

**Protocol insolvency (Critical).** The attacker extracts real LST tokens from the pool in excess of what they deposited. The shortfall is borne by all other depositors whose rsETH is now backed by less collateral per share. Repeated attacks compound the deficit.

**Algebraic proof of profit:**

Let:
- `T` = total ETH in protocol before attack, `S` = rsETH supply, `R = T/S` = rsETH price
- Attacker deposits `N` tokens; stale price `P_s > P_t` (true price)
- rsETH minted = `N·P_s / R`
- After feed corrects, new rsETH price: `R_new = (T + N·P_t) / (S + N·P_s/R)`
- Attacker's withdrawal: `(N·P_s/R) · R_new / P_t`

Simplifying the profit condition:

```
P_s · (T + N·P_t) > R · P_t · (S + N·P_s/R)
P_s · T > P_t · R · S          (N·P_s·P_t terms cancel)
P_s / P_t > R·S / T = 1        (since R = T/S)
```

The condition `P_s > P_t` is exactly the definition of a stale-high price, so **the attacker always profits whenever the feed is stale**.

---

### Likelihood Explanation

Chainlink feeds go stale during network congestion, sequencer downtime (L2), or node operator issues. LST/ETH feeds have heartbeats of 24 h with 0.5 % deviation triggers; a feed can lag by hours without triggering a heartbeat update. The attack requires no privileged role, no flash loan, and no MEV infrastructure — only the ability to observe a stale feed and submit two transactions.

---

### Recommendation

1. **Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice`**, mirroring `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();
```

2. Set `STALENESS_THRESHOLD` per feed based on its documented heartbeat (e.g., 25 h for a 24 h heartbeat feed).

3. Consider making `updateRSETHPrice` callable only by a trusted keeper or adding a per-call price-deviation guard so a single call cannot move `rsETHPrice` by more than the permitted percentage.

### Proof of Concept

**Invariant fuzz test (Foundry):**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "forge-std/Test.sol";

contract StalePriceArbitrageTest is Test {
    // Assume fork setup with:
    //   - LRTDepositPool, LRTOracle, ChainlinkPriceOracle, LRTWithdrawalManager
    //   - A mock Chainlink feed for stETH/ETH

    MockChainlinkFeed feed;
    // ... (deploy protocol in setUp)

    function testStalePriceArbitrage() public {
        uint256 depositAmount = 10 ether;

        // 1. Set stale-high price (1.1 ETH per stETH)
        feed.setPrice(1.1e18);

        // 2. Attacker deposits 10 stETH → receives 11 rsETH
        vm.prank(attacker);
        depositPool.depositAsset(stETH, depositAmount, 0, "");
        uint256 rsETHReceived = rsETH.balanceOf(attacker);
        assertGt(rsETHReceived, depositAmount); // excess rsETH minted

        // 3. Feed corrects to true price (1.0 ETH per stETH)
        feed.setPrice(1.0e18);

        // 4. Anyone calls updateRSETHPrice — rsETHPrice drops
        lrtOracle.updateRSETHPrice();
        uint256 newRsETHPrice = lrtOracle.rsETHPrice();

        // 5. Attacker initiates withdrawal
        vm.prank(attacker);
        withdrawalManager.initiateWithdrawal(stETH, rsETHReceived, "");

        // 6. Fast-forward past delay, operator unlocks, attacker completes
        vm.roll(block.number + withdrawalDelayBlocks + 1);
        withdrawalManager.unlockQueue(stETH, type(uint256).max, 0, 0, type(uint256).max, type(uint256).max);
        vm.prank(attacker);
        withdrawalManager.completeWithdrawal(stETH, "");

        uint256 stETHReceived = IERC20(stETH).balanceOf(attacker);

        // INVARIANT VIOLATION: attacker redeems more stETH than deposited
        assertGt(stETHReceived, depositAmount,
            "Attacker extracted more stETH than deposited — protocol insolvency");
    }
}
```

The test will pass (invariant violated) on unmodified code whenever `P_stale > P_true`, confirming the vulnerability.
