### Title
Sub-Transaction Plutus ExUnits Not Validated Against `maxTxExUnits` Limit - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs)

### Summary

In the Dijkstra era, the `validateExUnitsTooBigUTxO` check in `dijkstraUtxoTransition` only sums ExUnits from the top-level transaction's redeemers. Sub-transactions embedded in the batch each carry their own redeemers with independent ExUnits declarations that are never validated against `ppMaxTxExUnitsL`. An unprivileged submitter can craft a top-level transaction with zero top-level ExUnits and arbitrarily many sub-transactions each declaring ExUnits just below `maxTxExUnits`, causing the Plutus interpreter to execute a total budget that is a multiple of the intended per-transaction cap.

### Finding Description

The Dijkstra era introduces nested ("batch") transactions: a top-level `Tx TopTx era` may embed an ordered map of `Tx SubTx era` sub-transactions via `dtbrSubTransactions`.

**The per-transaction ExUnits check** in `dijkstraUtxoTransition` is:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [1](#0-0) 

`validateExUnitsTooBigUTxO` computes `totalExUnits = totExUnits tx`, where:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
``` [2](#0-1) 

This folds only over the **top-level** transaction's redeemer map. Sub-transactions are stored in `tx ^. bodyTxL . subTransactionsTxBodyL`; each sub-transaction has its own `witsTxL . rdmrsTxWitsL` carrying independent ExUnits declarations that are never touched by this fold.

**The `SUBUTXO` rule** (`dijkstraSubUtxoTransition`) processes each sub-transaction but contains no call to `validateExUnitsTooBigUTxO` and no call to the `UTXOS` sub-rule: [3](#0-2) 

**The block-level check** in `alonzoBbodyTransition` also uses `foldMap totExUnits txs`, which again only sums top-level redeemers across all transactions in the block: [4](#0-3) 

Sub-transaction Plutus scripts are collected and executed at the UTXOW level via `plutusScriptsWithContextDijkstraStAnnTx`, which aggregates scripts from both the top-level and all sub-transactions: [5](#0-4) 

The declared ExUnits in sub-transaction redeemers serve as the Plutus interpreter's execution budget. Because they are never checked against `ppMaxTxExUnitsL`, an attacker can declare arbitrarily large budgets in sub-transaction redeemers.

### Impact Explanation

An unprivileged transaction submitter can force every validating node to execute Plutus scripts totalling `N × maxTxExUnits` of computation per submitted transaction, where `N` is the number of sub-transactions. This directly exceeds the intended per-transaction and per-block resource limits (`ppMaxTxExUnitsL`, `ppMaxBlockExUnitsL`), which exist to bound the CPU and memory cost of block validation. Sustained exploitation can degrade or halt block production on honest nodes. This matches the allowed impact: **"Attacker-controlled transactions exceed intended validation limits."**

### Likelihood Explanation

The Dijkstra era is the only era with sub-transactions. Any user who can submit a valid Dijkstra-era transaction can exploit this. No privileged access, key compromise, or governance majority is required. The attacker only needs to construct a top-level transaction with an empty redeemer set (passing the `maxTxExUnits` check) and embed sub-transactions whose redeemers declare large ExUnits.

### Recommendation

In `dijkstraUtxoTransition`, replace the single-level `totExUnits tx` with a batch-aware sum that includes sub-transaction redeemers:

```haskell
let batchExUnits =
      totExUnits tx
        <> foldMap totExUnits (OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL)
runTest $ failureUnless (pointWiseExUnits (<=) batchExUnits (pp ^. ppMaxTxExUnitsL)) $
  ExUnitsTooBigUTxO Mismatch { mismatchSupplied = batchExUnits
                              , mismatchExpected = pp ^. ppMaxTxExUnitsL }
```

Similarly, update the block-level `validateExUnits` in the Dijkstra BBODY rule to sum sub-transaction ExUnits across all transactions in the block.

### Proof of Concept

1. Construct a `DijkstraEra` top-level transaction with:
   - A single spend input (to satisfy `validateInputSetEmptyUTxO`)
   - Zero redeemers in the top-level witness set → `totExUnits tx = ExUnits 0 0` → passes `validateExUnitsTooBigUTxO`
   - `K` sub-transactions, each with one Plutus V3 script input and a redeemer declaring `ExUnits (maxMem / K + 1) (maxSteps / K + 1)` — individually below `maxTxExUnits` but collectively `K × maxTxExUnits`

2. Submit the transaction. The ledger accepts it because:
   - `totExUnits tx = ExUnits 0 0 ≤ maxTxExUnits` ✓ (only top-level checked)
   - `SUBUTXO` for each sub-transaction runs no ExUnits check ✓

3. During UTXOW/UTXOS execution, the Plutus interpreter runs all `K` sub-transaction scripts, each consuming up to `maxTxExUnits / K` budget, for a total of `K × maxTxExUnits` — far exceeding the protocol's intended cap. [1](#0-0) [6](#0-5) [3](#0-2)

validateExUnitsTooBigUTxO pp tx =

    -- This sums up the ExUnits for all embedded Plutus Scripts anywhere in the transaction:
    totalExUnits = totExUnits tx
