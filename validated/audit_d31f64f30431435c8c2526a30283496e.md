Audit Report

## Title
`depositAsset()` / `depositETH()` Mint rsETH Against Stale Cached `rsETHPrice`, Enabling Theft of Unclaimed Yield — (File: contracts/LRTDepositPool.sol)

## Summary

`LRTOracle` stores `rsETHPrice` as a state variable updated only by explicit calls to `updateRSETHPrice()` or `updateRSETHPriceAsManager()`. Neither `LRTDepositPool.depositAsset()` nor `depositETH()` triggers a price refresh before computing the rsETH mint amount. When protocol TVL has grown since the last price update (e.g., stETH rebase, EigenLayer yield accrual), the stale lower `rsETHPrice` causes depositors to receive more rsETH than their deposit is worth at the true current exchange rate, diluting existing holders and transferring their accrued yield to the depositor.

## Finding Description

`LRTOracle.rsETHPrice` is a plain storage variable declared at line 28 of `contracts/LRTOracle.sol`:

```solidity
uint256 public override rsETHPrice;
```

It is written only inside `_updateRsETHPrice()`, which is invoked by the public `updateRSETHPrice()` (lines 87–89) or the manager-only `updateRSETHPriceAsManager()` (lines 94–96). Neither function is called from the deposit path.

The deposit flow is:

```
depositAsset() / depositETH()
  └─ _beforeDeposit()          (LRTDepositPool.sol L648–670)
       └─ getRsETHAmountToMint()  (LRTDepositPool.sol L506–521)
            └─ lrtOracle.rsETHPrice()   ← reads cached storage value
```

`getRsETHAmountToMint()` at line 520 computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` (line 156–158) calls a live `IPriceFetcher` and returns a fresh market price. `lrtOracle.rsETHPrice()` returns the stale cached value. When protocol TVL has grown since the last `_updateRsETHPrice()` call — for example because stETH has rebased and `getTotalAssetDeposits(stETH)` now returns a higher balance — the true rsETH/ETH rate is higher than the stored `rsETHPrice`. The denominator in the mint formula is therefore smaller than it should be, and the depositor receives more rsETH than their deposit is worth at the true current rate.

Additionally, `_updateRsETHPrice()` mints protocol fees (lines 299–308) and can auto-pause the protocol on a price drop (lines 270–282). Both side effects are skipped during a deposit, meaning fee dilution is also avoided by the attacker and safety pauses may be bypassed.

No existing check in `_beforeDeposit()` or `getRsETHAmountToMint()` guards against a stale `rsETHPrice`. The `minRSETHAmountExpected` slippage parameter protects the depositor from receiving *too little* rsETH, not from receiving *too much*.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders have accrued yield embedded in the protocol TVL (stETH rebase increases the stETH balance held by the protocol; EigenLayer rewards increase ETH-denominated TVL). This yield is reflected in the true rsETH/ETH rate but not yet in the stored `rsETHPrice`. A depositor who mints against the stale lower price receives extra rsETH. When `updateRSETHPrice()` is eventually called, the new price is computed over the enlarged supply (including the attacker's extra rsETH), so the price rise that existing holders earned is partially captured by the attacker. This is a direct, quantifiable transfer of yield from existing holders to the depositor, matching the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation

**Medium.** `updateRSETHPrice()` is permissionless but is not called atomically within the deposit flow. stETH rebases occur approximately every 24 hours; EigenLayer rewards accrue continuously. Any window between a TVL-increasing event and the next keeper call is exploitable. A sophisticated depositor can compare on-chain TVL (computable from `getTotalAssetDeposits` across all assets) against the stored `rsETHPrice` to identify and time the maximum-spread window. No special privileges, flash loans, or oracle manipulation are required — only a standard `depositAsset()` or `depositETH()` call. The attack is repeatable every rebase cycle.

## Recommendation

At the start of `depositAsset()` and `depositETH()`, before `_beforeDeposit()` is called, invoke `ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice()`. This ensures the price used for minting reflects the current true protocol state, including accrued rewards, fee minting, and any required safety pauses. Because `updateRSETHPrice()` is already `public` and `whenNotPaused`, this call is safe to make from the deposit path; if the oracle is paused, the deposit will revert, which is the correct behavior.

## Proof of Concept

**Setup (fork or local state):**

1. Deploy protocol with 1000 stETH deposited across NDCs. Call `updateRSETHPrice()`. `rsETHPrice = 1.00e18`, rsETH supply = 1000, TVL = 1000 ETH.

2. Simulate a stETH rebase: increase the stETH balance held by the NDC by 1% (10 stETH). Protocol TVL is now 1010 ETH. True rsETH price = 1010/1000 = 1.01e18. `rsETHPrice` in storage is still `1.00e18`.

3. Attacker calls `depositAsset(stETH, 100e18, 0, "")`. `getRsETHAmountToMint` computes:
   ```
   rsethAmountToMint = (100e18 × 1e18) / 1.00e18 = 100 rsETH
   ```
   At the true price of 1.01e18, the correct amount is:
   ```
   100e18 / 1.01e18 ≈ 99.0099 rsETH
   ```
   Attacker receives **≈ 0.99 rsETH extra** (≈ 1 ETH of value stolen from existing holders).

4. Call `updateRSETHPrice()`. New TVL = 1110 ETH, new supply = 1100 rsETH. New price = 1110/1100 ≈ 1.009e18. The price increase that existing holders earned from the rebase is now diluted by the attacker's extra rsETH.

**Foundry test plan:** Fork mainnet, mock stETH rebase by directly increasing the NDC's stETH balance via `deal`, assert that `getRsETHAmountToMint` returns a value greater than `depositAmount * 1e18 / truePrice` where `truePrice` is computed by calling `_getTotalEthInProtocol` (or replicating its logic) divided by current rsETH supply.
