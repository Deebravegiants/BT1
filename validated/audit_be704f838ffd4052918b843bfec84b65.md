### Title
Permissionless `logMetadata` Allows Attacker to Deploy NEAR Tokens Mapped to Arbitrary EVM Addresses - (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `near/omni-bridge/src/lib.rs`)

### Summary
`OmniBridge.logMetadata` is a fully permissionless function that accepts any ERC20 address and emits a `LogMetadata` event from the registered bridge contract. The NEAR-side `deploy_token_callback` only validates that the event emitter is the registered factory — it does not validate that `token_address` is a legitimate token. An attacker can therefore deploy a NEAR bridge token with arbitrary name/symbol/decimals backed by a worthless attacker-controlled ERC20.

### Finding Description

**Step 1 — Permissionless EVM entry point.**

`OmniBridge.logMetadata` accepts any `tokenAddress` with no access control:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name = IERC20Metadata(tokenAddress).name();
    string memory symbol = IERC20Metadata(tokenAddress).symbol();
    uint8 decimals = IERC20Metadata(tokenAddress).decimals();
    logMetadataExtension(tokenAddress, name, symbol, decimals);
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [1](#0-0) 

The emitter of this event is the registered `OmniBridge` contract address itself. The `tokenAddress` in the event payload is attacker-controlled.

**Step 2 — EVM prover extracts emitter from `event.address`, not from event data.**

`parse_evm_proof` with `ProofKind::LogMetadata` calls `parse_evm_event`, which constructs a `LogMetadataMessage`. The `emitter_address` is taken from `event.address` (the contract that emitted the log — the registered bridge), while `token_address` is taken from `event.data.tokenAddress` (the attacker's ERC20):

```rust
impl TryFromLog<Log<LogMetadata>> for LogMetadataMessage {
    fn try_from_log(chain_kind: ChainKind, event: Log<LogMetadata>) -> Result<Self, Self::Error> {
        Ok(Self {
            token_address: OmniAddress::new_from_evm_address(chain_kind, H160(event.data.tokenAddress.into()))?,
            ...
            emitter_address: OmniAddress::new_from_evm_address(chain_kind, H160(event.address.into()))?,
        })
    }
}
``` [2](#0-1) 

**Step 3 — `deploy_token_callback` only checks `emitter_address`.**

The factory check passes because the emitter is the registered bridge contract. There is no check on `metadata.token_address`:

```rust
let chain = metadata.emitter_address.get_chain();
require!(

self.deploy_token_internal(

    attached_deposit,

``` [3](#0-2) 

**Step 4 — `deploy_token_internal` has no allowlist check.**

The function derives a NEAR token ID from the EVM address prefix and registers the mapping without any validation that the EVM address is a legitimate token:

```rust
fn deploy_token_internal(&mut self, chain_kind: ChainKind, token_address: &OmniAddress, metadata: BasicMetadata, ...) -> Promise {
    let prefix = token_address.get_token_prefix();
    let token_id: AccountId = format!("{prefix}.{deployer}").parse()...;
    ...
    self.add_token(&token_id, token_address, metadata.decimals, metadata.decimals);
    require!(self.deployed_tokens.insert(&token_id), BridgeError::TokenExists.as_ref());
    ...
}
``` [4](#0-3) 

The only uniqueness guard is `deployed_tokens.insert(&token_id)`, which is keyed by the EVM address-derived prefix — not by name/symbol. Multiple NEAR tokens with identical name/symbol can be deployed for different EVM addresses.

### Impact Explanation

An attacker deploys a NEAR bridge token with name=`USDC`, symbol=`USDC`, decimals=`6` backed by a worthless attacker-controlled ERC20. This token appears indistinguishable from the legitimate USDC bridge token in wallets and explorers. Users who receive this token (via DEX liquidity, airdrops, or social engineering) and attempt to bridge it back to EVM receive the attacker's worthless ERC20 instead of real USDC. This breaks the protocol's backing guarantee: every NEAR bridge token must be redeemable for the legitimate EVM asset it claims to represent.

This maps to: **Asset-identity divergence that breaks backing guarantees or sends value to the wrong party.**

### Likelihood Explanation

The attack requires only:
1. Deploying a malicious ERC20 (trivial, no cost beyond gas)
2. Calling the permissionless `logMetadata` on the registered bridge (no role required)
3. Obtaining a valid Merkle proof for the emitted event (standard relayer tooling)
4. Calling `deploy_token` on NEAR with sufficient storage deposit

No privileged access, no key compromise, no colluding validators. Fully executable by any unprivileged actor.

### Recommendation

Add an allowlist or registration check in `logMetadata` on the EVM side, restricting it to pre-approved token addresses (e.g., `onlyRole(DEFAULT_ADMIN_ROLE)`), or add a corresponding check in `deploy_token_callback`/`deploy_token_internal` on the NEAR side that validates `token_address` against a protocol-maintained allowlist before deploying a NEAR token.

### Proof of Concept

```
1. Deploy MaliciousERC20 with name="USDC", symbol="USDC", decimals=6 at address 0xATTACKER
2. Call OmniBridge.logMetadata(0xATTACKER)
   → emits LogMetadata(tokenAddress=0xATTACKER, name="USDC", symbol="USDC", decimals=6)
   → emitter = registered OmniBridge address (passes factory check)
3. Obtain Merkle proof for this log entry from the finalized block
4. Call omni-bridge.deploy_token(proof_kind=LogMetadata, proof=<above proof>)
   → deploy_token_callback: factories.get(Eth) == emitter_address ✓
   → deploy_token_internal: creates NEAR token "attacker....deployer.near"
     with name="USDC", symbol="USDC", decimals=6, mapped to 0xATTACKER
5. Assert: token_address_to_id[Eth:0xATTACKER] == "attacker....deployer.near"
           ft_metadata("attacker....deployer.near").symbol == "USDC"
   Users bridging this NEAR token back to EVM receive 0xATTACKER tokens (worthless).
```
