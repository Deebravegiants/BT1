### Title
Consensus Vote Signature Verification Skipped — Any Peer Can Forge Validator Votes to Force False Quorum and Divergent Block Commitment - (File: `crates/apollo_consensus/src/single_height_consensus.rs`)

---

### Summary

`handle_vote` in `single_height_consensus.rs` carries an explicit `// TODO(Asmaa): verify the signature` comment and performs **zero** cryptographic verification of incoming consensus votes. Any network peer can craft a `Vote` message with an arbitrary `voter` field set to any committee member's address and any `proposal_commitment`, and the message will be accepted and counted toward quorum. A compounding issue exists in `build_precommit_vote_message_digest` in `signature_manager.rs`: even if verification were wired in, the signed payload covers only `block_hash` — omitting `height`, `round`, and `chain_id` — making valid signatures replayable across heights and chains.

---

### Finding Description

**Root cause 1 — Signature verification is entirely absent:**

`handle_vote` in `crates/apollo_consensus/src/single_height_consensus.rs` performs three checks before accepting a vote:

1. `vote.height == current_height`
2. `vote.voter` is a member of the committee
3. The vote is not a duplicate or conflict

The signature field is read off the wire and deserialized but never verified:

```rust
pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
    // TODO(Asmaa): verify the signature   ← explicitly skipped
    ...
    if !self.committee.members().iter().any(|s| s.address == vote.voter) {
        debug!("Ignoring vote from non validator: vote={:?}", vote);
        return VecDeque::new();
    }
    ...
    self.state_machine.handle_event(sm_vote)
}
```

Because `vote.voter` is attacker-controlled and the only membership check is an address equality test, an attacker who knows the committee composition (public information broadcast in every proposal) can set `voter` to any committee member's `ContractAddress` and the vote will pass all guards.

The self-vote path in `state_machine.rs` confirms signatures are also not produced yet:

```rust
let vote = Vote {
    ...
    // TODO(Asmaa): sign the vote
    signature: RawSignature::default(),
};
```

**Root cause 2 — Precommit message digest omits height, round, and chain_id:**

`build_precommit_vote_message_digest` in `crates/apollo_signature_manager/src/signature_manager.rs` constructs the signed payload as:

```rust
fn build_precommit_vote_message_digest(block_hash: BlockHash) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let mut message = Vec::with_capacity(PRECOMMIT_VOTE.len() + block_hash.len());
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash);
    MessageDigest(blake2s_to_felt(&message))
}
```

The payload is `"PRECOMMIT_VOTE" || block_hash`. It does not bind `height`, `round`, `chain_id`, or `voter`. Even after root cause 1 is fixed, a valid precommit signature for block hash `X` at height `H` is cryptographically identical to a precommit for `X` at any other height, any other round, or on any other chain sharing the same validator key — a direct replay-attack analog to the external report's nonce-less pause/unpause messages.

---

### Impact Explanation

**Forged-quorum path (root cause 1):**

The `upon_decision` function in `state_machine.rs` fires when:
- A proposal for round `R` exists in `self.proposals` (populated after the local node validates a proposal received from the network), and
- 2/3+ weighted precommit votes for that proposal's commitment exist, and
- The virtual proposer is among the precommit voters.

All three conditions are satisfiable by an attacker:
- The proposal commitment is observable from the broadcast proposal stream.
- The attacker forges precommit votes for each committee member (including the virtual proposer) by setting `voter` to their known `ContractAddress` and `proposal_commitment` to the observed value.
- No signature check blocks acceptance.

Once `upon_decision` fires, `SMRequest::DecisionReached` is returned, `decision_reached` is called on the context, `batcher.decision_reached` commits the block, and `finalize_decision` writes the state diff and block header to storage. The node has committed a block based on fabricated consensus.

If the attacker targets a proposal that only a subset of nodes have validated (e.g., due to network partitioning or a proposal that arrived late), the attacked node commits a block the rest of the network has not agreed on, producing a **fork and wrong committed state root**.

**Replay path (root cause 2):**

Once root cause 1 is fixed, a validator's precommit signature for block hash `X` at height `H` can be replayed at any future height `H'` where the same partial block hash appears, or on a different chain. This is the exact sequencer analog of the external report's replay attack: the signed message lacks a nonce/height/chain binding.

---

### Likelihood Explanation

- The gossip network is open to any peer; no authentication is required to inject `Vote` messages.
- The committee composition (validator addresses) is public — it is embedded in every `ProposalInit` and derivable from the staking contract.
- The attack requires only knowledge of the current committee and the ability to send gossip messages; no privileged access, no key material, and no computational work beyond constructing a valid protobuf `Vote` message.
- The `// TODO(Asmaa): verify the signature` comment confirms this is a known gap, not an accidental omission.

---

### Recommendation

1. **Immediately implement signature verification in `handle_vote`** (`crates/apollo_consensus/src/single_height_consensus.rs`, line 242). Reject any vote whose `signature` does not verify against the `voter`'s registered public key.

2. **Fix the message digest** in `build_precommit_vote_message_digest` (`crates/apollo_signature_manager/src/signature_manager.rs`, line 138) to include `height`, `round`, `chain_id`, and `voter` address in the signed payload, preventing cross-height and cross-chain replay:
   ```
   digest = blake2s("PRECOMMIT_VOTE" || chain_id || height || round || voter || block_hash)
   ```

3. **Implement vote signing** in `make_self_vote` (`crates/apollo_consensus/src/state_machine.rs`, line 254) using the `SignatureManager` before broadcasting.
