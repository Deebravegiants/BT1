### Title
ERC1155 tokens permanently locked in `OmniBridge` when recipient contract does not implement `IERC1155Receiver` - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
In `OmniBridge.finTransfer`, ERC1155 tokens are delivered to the recipient via `safeTransferFrom`. If the recipient is a contract that does not implement `IERC1155Receiver`, the call reverts unconditionally. Because the recipient address is cryptographically bound inside the signed payload, no alternative delivery path exists and the escrowed ERC1155 tokens are permanently locked inside the bridge contract.

### Finding Description
`finTransfer` handles the final leg of a cross-chain transfer. When the token being delivered is a registered ERC1155 multi-token, the function executes:

```solidity
// OmniBridge.sol line 323-330
} else if (multiToken.tokenAddress != address(0)) {
    IERC1155(multiToken.tokenAddress).safeTransferFrom(
        address(this),
        payload.recipient,
        multiToken.tokenId,
        payload.amount,
        ""
    );
}
``` [1](#0-0) 

Per EIP-1155, `safeTransferFrom` calls `onERC1155Received` on the recipient if it is a contract, and reverts if the selector is not returned correctly. Any contract recipient that does not implement `IERC1155Receiver` (e.g., a multisig, DAO treasury, or generic vault) will cause this call to revert every time.

The nonce guard is set to `true` before the transfer attempt:

```solidity
// OmniBridge.sol line 283-287
if (completedTransfers[payload.destinationNonce]) {
    revert NonceAlreadyUsed(payload.destinationNonce);
}
completedTransfers[payload.destinationNonce] = true;
``` [2](#0-1) 

Because the entire transaction reverts when `safeTransferFrom` fails, the nonce is not permanently consumed — the call can be retried. However, `payload.recipient` is part of the Borsh-encoded message that is verified against the `nearBridgeDerivedAddress` ECDSA signature:

```solidity
// OmniBridge.sol line 309-312
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [3](#0-2) 

The recipient cannot be changed without a new valid signature from the bridge authority. No re-routing or rescue mechanism exists in the contract. The ERC1155 tokens that were locked into the bridge during `initTransfer1155` are therefore permanently stranded.

The bridge contract itself implements `IERC1155Receiver` only to accept its own outbound-then-returned transfers, and explicitly rejects direct sends from other operators:

```solidity
// OmniBridge.sol line 522-534
function onERC1155Received(...) external view override returns (bytes4) {
    if (operator != address(this)) {
        revert ERC1155DirectSendNotAllowed();
    }
    return this.onERC1155Received.selector;
}
``` [4](#0-3) 

This confirms the bridge itself cannot be used as a relay to rescue stuck tokens.

### Impact Explanation
ERC1155 tokens bridged from EVM to another chain and back — where the EVM recipient is a contract without `IERC1155Receiver` — are permanently locked inside `OmniBridge`. There is no admin rescue function, no alternative delivery path, and no way to re-sign the payload with a different recipient. This constitutes an irreversible fund lock of bridged ERC1155 assets, matching the "Critical — Irreversible fund lock" impact category.

### Likelihood Explanation
Many widely-used contract wallets, DAO treasuries, and DeFi vaults do not implement `IERC1155Receiver`. A user initiating a cross-chain transfer of ERC1155 tokens and specifying such a contract as the EVM recipient will trigger this lock. The path is fully reachable through the public `finTransfer` entrypoint with no privileged role required.

### Recommendation
Replace `safeTransferFrom` with `transferFrom` (non-safe variant) when delivering ERC1155 tokens to the recipient in `finTransfer`. This removes the `onERC1155Received` callback requirement and allows delivery to any address, consistent with how the ERC20 path uses `safeTransfer` (which has no receiver-interface requirement):

```solidity
// Replace:
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this), payload.recipient, multiToken.tokenId, payload.amount, ""
);

// With:
IERC1155(multiToken.tokenAddress).transferFrom(
    address(this), payload.recipient, multiToken.tokenId, payload.amount, ""
);
```

Alternatively, implement a pull-based rescue: if `safeTransferFrom` reverts, record the pending claim and allow the recipient to pull tokens via a separate `claimTokens` function, preventing permanent lock regardless of recipient interface support.

### Proof of Concept
1. User holds ERC1155 token (e.g., `tokenAddress = 0xABC`, `tokenId = 1`) on EVM.
2. User calls `initTransfer1155(0xABC, 1, 100, 0, 0, "near_recipient", "")` — tokens are transferred into `OmniBridge` via `safeTransferFrom` at line 458. [5](#0-4) 
3. On NEAR, the user receives wrapped tokens and initiates a return transfer specifying `recipient = 0xVAULT`, a contract that does not implement `IERC1155Receiver`.
4. The bridge signs a `TransferMessagePayload` with `recipient = 0xVAULT`.
5. Relayer calls `finTransfer(sig, payload)` on EVM.
6. Execution reaches line 324: `IERC1155(multiToken.tokenAddress).safeTransferFrom(address(this), 0xVAULT, 1, 100, "")`.
7. `0xVAULT` does not implement `onERC1155Received` → call reverts.
8. The entire transaction reverts; nonce is not consumed.
9. Every subsequent `finTransfer` attempt with the same signed payload reverts identically.
10. The 100 ERC1155 tokens remain locked in `OmniBridge` with no recovery path.
