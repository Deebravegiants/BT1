### Title
Signature Replay in `respond_verify_foreign_tx` Enables Stale Foreign-Chain Attestation Delivery — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` does not track which `(request, payload_hash)` pairs have already been consumed. Because the contract also imposes no barrier against re-submitting the same foreign-chain transaction request after a prior one has been resolved, a Byzantine MPC participant (strictly below the signing threshold) can replay a previously observed, legitimately produced `VerifyForeignTransactionResponse` against a freshly queued pending request for the identical foreign tx. The contract accepts the replay because the MPC signature over the caller-supplied `payload_hash` is still cryptographically valid, and the pending-request map entry now exists again. The caller receives a stale attestation — one whose extracted values (e.g., block hash) may reflect a chain state that has since been reorganized away — enabling invalid bridge execution or double-spend conditions.

---

### Finding Description

**Root cause — no used-response tracking and no re-submission guard.**

`respond_verify_foreign_tx` performs two checks before resolving queued yields:

1. The caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over the caller-supplied `response.payload_hash` under the MPC root public key. [1](#0-0) 

Neither check prevents the same `(request, response)` tuple from being submitted a second time. The contract does not record which `payload_hash` values have been delivered, and it does not verify that `response.payload_hash` is the hash the MPC network would compute *today* for the given `request`. The hash is entirely caller-supplied and is accepted as long as the signature over it is valid.

**Re-submission is always accepted.**

`verify_foreign_transaction` places no guard against re-submitting a foreign tx whose request key has already been resolved. After `resolve_yields_for` removes the map entry on a successful respond, the identical `VerifyForeignTransactionRequest` can be enqueued again by any caller paying 1 yoctoNEAR. [2](#0-1) 

The `VerifyForeignTransactionRequest` key contains only `(ForeignChainRpcRequest, domain_id, payload_version)` — no caller identity, no block height, no timestamp — so the same foreign tx can be re-queued indefinitely. [3](#0-2) 

**`resolve_yields_for` removes the entry on first respond, then accepts a new one.** [4](#0-3) 

Once the entry is gone, a re-submission creates a fresh entry. A Byzantine node that observed the original `respond_verify_foreign_tx` call on-chain can immediately replay the same arguments in a new NEAR transaction, resolving the fresh entry before honest nodes respond.

---

### Impact Explanation

The `verify_foreign_transaction` / `respond_verify_foreign_tx` flow is the MPC network's mechanism for attesting that a foreign-chain transaction finalized. Bridge contracts (e.g., Omnibridge inbound flow) consume the returned `VerifyForeignTransactionResponse` — specifically its `payload_hash` and `signature` — to authorize asset releases on NEAR.

If a Byzantine node replays a stale response after a foreign-chain reorganization:

* The `payload_hash` encodes extracted values (e.g., `BlockHash`) from the pre-reorg state.
* The MPC signature over that hash is still valid (the key has not changed).
* The contract delivers the stale attestation to the caller.
* A bridge contract that trusts the MPC attestation without independently verifying the block hash against the current chain tip will process a transfer for a transaction that is no longer finalized on the foreign chain.

This is a **cross-chain replay / forged foreign-chain verification** scenario causing invalid bridge execution or double-spend conditions — matching the High impact tier.

---

### Likelihood Explanation

**Attacker capability required:**

* A single Byzantine MPC participant (below the signing threshold) who is an attested participant — sufficient to call `respond_verify_foreign_tx`.
* Knowledge of a prior valid `(request, response)` pair, which is public on-chain.
* The ability to submit a new `verify_foreign_transaction` for the same tx (costs 1 yoctoNEAR) and then immediately replay the old response before honest nodes respond.

**Triggering condition:** A foreign-chain reorganization after the original attestation was produced. This is realistic for chains with probabilistic finality (Bitcoin, Ethereum pre-finality checkpoints, etc.) and is precisely the scenario the `confirmations` field in `BitcoinRpcRequest` is meant to guard against — but the replay bypasses that guard entirely.

No threshold collusion, no key leakage, and no privileged operator access are required. A single Byzantine node suffices.

---

### Recommendation

1. **Track consumed `(request, payload_hash)` pairs.** Maintain a `LookupSet<(VerifyForeignTransactionRequest, PayloadHash)>` of digests that have been responded to. Reject any `respond_verify_foreign_tx` call whose `(request, payload_hash)` is already in the set.

2. **Bind `payload_hash` to the request on-chain.** When `verify_foreign_transaction` is enqueued, record the expected `payload_hash` range or at minimum a monotonic sequence number so that a replayed response for a stale block cannot satisfy a fresh request.

3. **Prevent re-submission of already-resolved requests.** Maintain a separate `LookupSet<VerifyForeignTransactionRequest>` of requests that have been successfully resolved, and reject re-submissions.

### Proof of Concept

```
1. Alice calls verify_foreign_transaction(Bitcoin, tx_id=X, confirmations=1).
   → Contract queues request R under pending_verify_foreign_tx_requests[R].

2. MPC nodes query Bitcoin RPC: tx X is in block B, block_hash = H.
   → payload = ForeignTxSignPayload::V1 { request: R.request, values: [BlockHash(H)] }
   → payload_hash = hash(payload)  [call it PH1]
   → MPC signs PH1 → signature S1

3. Honest node calls respond_verify_foreign_tx(R, { payload_hash: PH1, sig: S1 }).
   → Contract verifies sig S1 over PH1 ✓
   → resolve_yields_for removes pending_verify_foreign_tx_requests[R]
   → Alice receives { payload_hash: PH1, sig: S1 }

4. Bitcoin reorganization: tx X is now in block B', block_hash = H' ≠ H.

5. Bob calls verify_foreign_transaction(Bitcoin, tx_id=X, confirmations=1).
   → Same request R. Contract queues it: pending_verify_foreign_tx_requests[R] = [Bob's yield].

6. Byzantine MPC node (single, below threshold) replays:
   respond_verify_foreign_tx(R, { payload_hash: PH1, sig: S1 })
   → Contract checks: is caller attested participant? YES.
   → Contract checks: verify_ecdsa_signature(S1, PH1, root_pk)? YES (key unchanged).
   → pending_verify_foreign_tx_requests[R] exists? YES (Bob's yield).
   → resolve_yields_for drains Bob's yield with stale response.

7. Bob receives { payload_hash: PH1, sig: S1 } — attesting block_hash = H (pre-reorg).
   Bob's bridge contract releases funds on NEAR for a Bitcoin tx that is no longer
   finalized, enabling a double-spend.
```

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
