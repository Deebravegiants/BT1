### Title
Unconstrained `fee_proposal_fri` During V0_14_3 Transition Bypasses Fee-Proposal Bounds, Permanently Corrupting `fee_actual` and `l2_gas_price` — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The SNIP-35 fee-proposal mechanism enforces that a proposer's `fee_proposal_fri` must lie within a geometric margin of `fee_actual` (the median of the last `window_size` proposals). However, the bounds check in `is_proposal_init_valid` is guarded by an `if let (Some(fee_actual), Some(fee_proposal))` pattern: when `fee_actual` is `None`, the entire check is silently skipped. `fee_actual` is `None` for every block during the V0_14_2 → V0_14_3 transition window (and at genesis), because `compute_fee_actual` returns `None` whenever any entry in the window is `None`. A malicious proposer selected during this unconstrained window can publish an arbitrary `fee_proposal_fri` (e.g., `u128::MAX`). That value is accepted by all validators, committed to consensus, and recorded in `fee_proposals_window`. Once the window fills with V0_14_3 blocks, `fee_actual` is computed as the median of the window — which now includes the injected extreme values — and is used as a hard floor for `l2_gas_price` in every subsequent block.

---

### Finding Description

**Invariant:** `fee_proposal_fri` in `ProposalInit` must be within `[fee_actual / (1 + margin), fee_actual * (1 + margin)]` so that no single proposer can move the gas price more than `margin_ppt` parts-per-thousand per block.

**Root cause — `validate_proposal.rs` lines 396–416:**

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(...);
    }
}
```

When `proposal_init_validation.fee_actual` is `None`, the entire block is skipped — no bounds are enforced on `fee_proposal_fri`.

**When is `fee_actual` `None`?** `compute_fee_actual` returns `None` whenever any entry in `fee_proposals_window[height - window_size .. height - 1]` is `None` or absent:

```rust
Some(None) | None => {
    warn!("Cannot compute fee_actual for height {height}: ...");
    return None;
}
```

Pre-V0_14_3 blocks are stored with `fee_proposal_fri = None`. Therefore, for the entire first `window_size` blocks after V0_14_3 activates, `fee_actual` is `None` and the bounds check is skipped.

**Attack path:**

1. V0_14_3 activates at height H. The window `[H - W, H - 1]` is all `None` (pre-V0_14_3).
2. For blocks `H` through `H + W - 1`, `compute_fee_actual` returns `None` because the window still contains `None` entries.
3. A malicious proposer selected for any of these blocks sets `fee_proposal_fri = u128::MAX`. Validators call `is_proposal_init_valid`, reach the bounds check, find `fee_actual = None`, and skip it. The proposal is accepted.
4. `finalize_decision` calls `record_fee_proposal(height, init.fee_proposal_fri)`, inserting `u128::MAX` into `fee_proposals_window`.
5. Once the malicious proposer has been selected for `⌈W/2⌉` of the `W` transition blocks, the median of the window becomes `u128::MAX`.
6. At height `H + W`, `compute_fee_actual` returns `Some(u128::MAX)`.
7. `calculate_next_l2_gas_price_for_fin` sets `effective_min = max(config_min, u128::MAX) = u128::MAX`, so `l2_gas_price = u128::MAX` for all subsequent blocks.

**Downstream propagation:** `l2_gas_price` is stored in `SequencerConsensusContext.l2_gas_price`, emitted in every future `ProposalInit.l2_gas_price_fri`, and used by the blockifier to price every transaction. Setting it to `u128::MAX` makes every transaction prohibitively expensive.

---

### Impact Explanation

The corrupted value is `l2_gas_price` (and `ProposalInit.l2_gas_price_fri` for all future blocks). This is a permanent, consensus-committed corruption: once the manipulated `fee_proposal` values are in the window and `fee_actual` is computed from them, every honest validator independently derives the same extreme `l2_gas_price` and embeds it in their own proposals. The fee market is irreversibly broken until a manual override or a chain upgrade is applied.

Impact category: **Critical — Incorrect fee/gas accounting with economic impact** (all L2 transactions become unexecutable at `l2_gas_price = u128::MAX`).

---

### Likelihood Explanation

The attack window is exactly `window_size` blocks (a protocol constant). The attacker needs to be selected as proposer for `⌈window_size / 2⌉` of those blocks. In a stake-weighted proposer-selection scheme, a validator with ≥50% stake can guarantee this. A validator with less stake has a proportionally lower probability but can still succeed with some probability. The transition from V0_14_2 to V0_14_3 is a one-time, predictable event, giving an attacker advance notice to position themselves.

---

### Recommendation

Remove the silent skip. When `fee_actual` is `None` during the transition window, either:

1. **Reject any `fee_proposal_fri` that deviates from `l2_gas_price` by more than `margin_ppt`** — the same fallback the honest proposer uses (`compute_proposer_fee_proposal` freezes at `l2_gas_price` when `fee_actual` is `None`). The validator should enforce the same constraint:

```rust
let reference = proposal_init_validation.fee_actual
    .unwrap_or(proposal_init_validation.l2_gas_price_fri);
let (lower_bound, upper_bound) = fee_proposal_bounds(
    reference,
    VersionedConstants::latest_constants().fee_proposal_margin_ppt,
);
if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
    return Err(...);
}
```

2. **Or**, record the transition-period `fee_proposal` values as `None` in the window (treating them as pre-V0_14_3) so they do not contribute to `fee_actual` once the window fills.

```
window_size = W (e.g., 10)
V0_14_3 activates at height H

For blocks H, H+1, ..., H+W-1:
  - compute_fee_actual returns None (window contains None entries)

  - fee_proposals_window[H..H+W-1] = [u128::MAX, ..., u128::MAX]  (if attacker controlled ≥ W/2 slots)
  - compute_fee_actual returns Some(u128::MAX)
  - calculate_next_l2_gas_price_for_fin:
      effective_min = max(config_min, u128::MAX) = u128::MAX
      l2_gas_price = u128::MAX

All future ProposalInit.l2_gas_price_fri = u128::MAX
All transactions fail fee checks or are priced at u128::MAX FRI per gas unit
```
