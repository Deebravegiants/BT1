Audit Report

## Title
ETH Permanently Frozen When Withdrawal Recipient Contract Cannot Accept ETH - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager._transferAsset` sends native ETH via `.call{value:}` and reverts on failure. Because rsETH is irreversibly burned and ETH is pulled into the withdrawal manager during `unlockQueue` (a separate, earlier transaction), any subsequent `completeWithdrawal` call targeting a non-ETH-accepting smart contract will revert indefinitely. The ETH is permanently locked in the contract with no admin rescue path, and the user's rsETH is already gone.

## Finding Description
The execution flow spans two separate transactions:

**Transaction 1 — `unlockQueue`:** rsETH is burned and ETH is redeemed from the unstaking vault into `LRTWithdrawalManager` at L305–307. These state changes are final and cannot be undone.

**Transaction 2 — `completeWithdrawal` / `completeWithdrawalForUser`:** `_processWithdrawalCompletion` (L699–738) pops the nonce, deletes the request, decrements `unlockedWithdrawalsCount`, then calls `_transferAsset(asset, user, request.expectedAssetAmount)` at L734. `_transferAsset` (L876–883) executes `payable(to).call{value: amount}("")` and reverts with `EthTransferFailed` if `sent == false`. Because this revert unwinds all state changes within Transaction 2, the withdrawal request is fully restored — but the rsETH burn from Transaction 1 is not affected.

Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` targets the same `user` address (L202 passes `user` directly to `_processWithdrawalCompletion`), so both paths revert identically. The `sweepRemainingAssets` escape hatch at L403 requires `hasUnlockedWithdrawals(asset) == false`, which is permanently blocked because `unlockedWithdrawalsCount[asset]` remains > 0 as long as the stuck request exists. There is no admin function to redirect the ETH to an alternative address or credit it to a claimable mapping.

The NatDoc on `completeWithdrawalForUser` (L191) explicitly states "Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH," confirming the protocol is aware of ETH-specific issues with this path but provides no alternative delivery mechanism.

## Impact Explanation
**Critical — Permanent freezing of funds.** The user's rsETH is burned irreversibly in `unlockQueue`. The corresponding ETH sits inside `LRTWithdrawalManager` with no reachable code path to deliver it elsewhere or return it. The funds are permanently inaccessible to both the user and the protocol.

## Likelihood Explanation
**Medium.** Smart-contract callers are common in DeFi: DAO vaults, yield aggregators, and custom wrapper contracts routinely hold rsETH and initiate withdrawals. Contracts that explicitly reject ETH (no `receive()`, or a reverting `receive()`) are a realistic and non-negligible subset. A single such caller permanently bricks their own withdrawal and, via the `sweepRemainingAssets` guard, blocks treasury sweeps for the entire ETH asset queue.

## Recommendation
1. **Pull-over-push:** On ETH transfer failure, do not revert — instead credit the amount to a `claimable[user]` mapping and expose a separate `claimETH()` function.
2. **Configurable recipient:** Allow the user to specify an alternative ETH recipient at `initiateWithdrawal` or `completeWithdrawal` time.
3. **Deferred burn:** Do not burn rsETH in `unlockQueue` until ETH has been successfully delivered, or implement a refund path that re-mints rsETH if delivery permanently fails.

## Proof of Concept
1. Smart-contract vault `VaultA` (no `receive()`) calls `initiateWithdrawal(ETH_TOKEN, 1e18, "ref")`. rsETH is transferred to `LRTWithdrawalManager` (L166).
2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. At L305–307, rsETH is burned and ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. These changes are final.
3. `VaultA` calls `completeWithdrawal(ETH_TOKEN, "ref")`. At L734, `_transferAsset` executes `payable(VaultA).call{value: amount}("")`. `VaultA` has no `receive()`, so `sent == false`. L879 reverts with `EthTransferFailed`. All Transaction 2 state changes are undone; rsETH burn is not.
4. Operator calls `completeWithdrawalForUser(ETH_TOKEN, VaultA, "ref")` (L192–203). Same ETH destination, same revert.
5. `unlockedWithdrawalsCount[ETH_TOKEN] > 0` permanently → `sweepRemainingAssets` reverts at L403 for all callers.

**Foundry test sketch:**
```solidity
function test_permanentFreeze() public {
    // Deploy VaultA with no receive()
    NoReceiveVault vault = new NoReceiveVault();
    // Fund vault with rsETH, initiate withdrawal
    vm.prank(address(vault));
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1e18, "ref");
    // Operator unlocks queue (burns rsETH, pulls ETH)
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH_TOKEN, ...);
    // Attempt completion — must revert
    vm.prank(address(vault));
    vm.expectRevert(LRTWithdrawalManager.EthTransferFailed.selector);
    withdrawalManager.completeWithdrawal(ETH_TOKEN, "ref");
    // sweepRemainingAssets also blocked
    vm.prank(manager);
    vm.expectRevert(LRTWithdrawalManager.PendingWithdrawalsExist.selector);
    withdrawalManager.sweepRemainingAssets(ETH_TOKEN);
}
```
