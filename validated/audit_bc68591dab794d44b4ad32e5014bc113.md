### Title
Unbounded Iteration Over All Registered DReps During Per-Transaction Validation Triggered by Governance Proposals — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

When a Conway-era transaction containing at least one governance proposal procedure is processed, the ledger unconditionally iterates over **every registered DRep** in the global `VState` to update dormant-epoch expiry counters. This O(N-DReps) traversal is performed inside the per-transaction `CERTS` / `LEDGER` rule, is not bounded by any protocol parameter, and is not reflected in the transaction fee. An unprivileged actor can amplify the per-transaction validation cost by registering a large number of DReps (deposits are refundable) and then submitting a single cheap governance-proposal transaction, causing every validating node to perform unbounded work proportional to the total DRep population rather than to the transaction size or declared execution units.

### Impact Explanation

Every honest node that validates a qualifying transaction must execute `Map.map updateExpiry` over the entire DRep registry. Because:

1. The number of registered DReps is unbounded by protocol parameters.
2. The work is performed inside the per-transaction LEDGER rule, not amortised across an epoch.
3. The transaction fee is proportional to byte-size, not to `|vsDReps|`.

An attacker can submit a minimum-size governance-proposal transaction (paying only the `govActionDeposit` and a small byte-fee) that forces every validating node to perform O(N) map operations, where N can be made arbitrarily large by pre-registering DReps. Because DRep deposits are refundable upon unregistration, the net cost to the attacker approaches zero after reclaiming deposits, while the per-node validation cost scales with N.

This matches the allowed Medium impact: *"Attacker-controlled transactions … exceed intended validation limits … outside design parameters."* The intended limit is `maxTxSize`; the actual validation work is decoupled from it.

---

### Likelihood Explanation

- **Organic growth path**: As the Cardano governance ecosystem matures, the DRep count grows naturally. Any quiet period (no proposals for ≥1 epoch) followed by a proposal submission triggers the full traversal over all legitimately registered DReps — no attacker action required.
- **Active exploitation path**: An attacker registers N DReps (each paying `drepDeposit`, currently 500 ADA on mainnet), waits for `numDormantEpochs > 0`, submits one governance-proposal transaction, then unregisters all DReps to recover deposits. The net cost is only transaction fees; the capital is temporarily locked but fully recoverable.
- **No privileged access required**: DRep registration and governance-proposal submission are both permissionless operations available to any ADA holder.

Likelihood is **Medium** for the organic path (inevitable as governance activity grows) and **Low-Medium** for the active exploitation path (requires temporary capital).

---

### Recommendation

1. **Bound the per-transaction DRep traversal**: Replace the unconditional `Map.map` with a lazy or incremental update. For example, store a single `pendingDormantBump :: EpochNo` field and apply it lazily when a DRep's expiry is actually read (e.g., during ratification), rather than eagerly updating every DRep on every proposal transaction.

2. **Add a protocol parameter cap on DRep count** or charge a per-DRep fee component in the minimum-fee formula so that the fee paid reflects the actual validation work.

3. **Alternatively, move the dormant-epoch expiry bump entirely to the epoch boundary** (`EPOCH` rule), where it is already amortised and expected to be O(N).

### Proof of Concept

**Setup**

```
1. Register K DRep credentials (K × 500 ADA deposit, refundable).
2. Allow ≥1 epoch to pass with zero governance proposals
   so that vsNumDormantEpochs increments to ≥1.
```

**Attack transaction**

```
3. Submit a minimal Conway transaction containing exactly one
   ProposalProcedure (e.g., an InfoAction).
   Transaction byte-size ≈ 300–500 bytes → fee ≈ 0.2 ADA.
```

**Effect on every validating node**

```
4. conwayLedgerTransition calls updateDormantDRepExpiries.
   hasProposals = True  →  updateDormantDRepExpiry is invoked.
   Map.map updateExpiry iterates all K DRep entries.
   Work = O(K), fee paid = O(1).
```

**Cleanup**

```
5. Unregister all K DReps → recover K × 500 ADA.
   Net attacker cost ≈ registration + unregistration tx fees only.
```

updateDormantDRepExpiry ::
  -- | Current Epoch
  EpochNo ->
  VState era ->
  VState era
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry
  where
    numDormantEpochs = vState ^. vsNumDormantEpochsL
    updateExpiry =
      drepExpiryL
        %~ \currentExpiry ->
          let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
           in if actualExpiry < currentEpoch
                then currentExpiry
                else actualExpiry
