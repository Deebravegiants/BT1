### Title
Missing `recompute_available_foreign_chains` Call in `vote_cancel_resharing` Leaves Foreign-Chain Availability Cache Stale - (File: `crates/contract/src/lib.rs`)

---

### Summary

`vote_cancel_resharing` transitions the contract back to `Running` state but omits the `recompute_available_foreign_chains()` call that `vote_reshared` correctly performs on every other `Resharing → Running` transition. This leaves the `available_foreign_chains` cache reflecting the participant set of the cancelled resharing epoch rather than the restored previous running state, breaking the production invariant that the cache accurately represents which foreign chains the current participant set can service.

---

### Finding Description

Two contract entry points both transition the protocol from `Resharing` to `Running` state:

**`vote_reshared`** (successful resharing completion): [1](#0-0) 

```rust
if let Some(new_state) = self.protocol_state.vote_reshared(key_event_id)? {
    self.protocol_state = new_state;
    self.recompute_available_foreign_chains();   // ← cache updated
    // ... cleanup promises spawned
}
```

**`vote_cancel_resharing`** (resharing cancellation): [2](#0-1) 

```rust
pub fn vote_cancel_resharing(&mut self) -> Result<(), Error> {
    Self::assert_caller_is_signer();
    if let Some(new_state) = self.protocol_state.vote_cancel_resharing()? {
        self.protocol_state = new_state;   // ← recompute_available_foreign_chains() MISSING
    }
    Ok(())
}
```

`recompute_available_foreign_chains` rebuilds the `available_foreign_chains` cache by iterating the **current** participant set's TLS keys and the ForeignTx domain reconstruction threshold: [3](#0-2) 

During resharing, `register_foreign_chains_config` triggers incremental cache updates using the resharing state's participant set (the proposed new participants). When resharing is cancelled, `vote_cancel_resharing` restores `previous_running_state` — a different participant set — but the cache is never recomputed against that restored set. [4](#0-3) 

The test `recompute_available_foreign_chains__should_update_cache_during_resharing` explicitly confirms that the cache is mutated during resharing based on the resharing participant set, making the post-cancellation stale state reachable in production: [5](#0-4) 

Additionally, `vote_cancel_resharing` omits all six cleanup promises that `vote_reshared` spawns on the same transition (including `REMOVE_NON_PARTICIPANT_UPDATE_VOTES`, `CLEAN_FOREIGN_CHAIN_DATA`, `REMOVE_NON_PARTICIPANT_TEE_VERIFIER_VOTES`, etc.): [6](#0-5) 

---

### Impact Explanation

The `available_foreign_chains` cache is the authoritative on-chain signal that MPC nodes and external callers query via `get_available_foreign_chains()` to determine which foreign chains the network can currently service: [7](#0-6) 

After a resharing cancellation the cache can be wrong in two directions:

1. **Cache shows a chain as available when it is not** — if the proposed (cancelled) participant set covered a chain that the restored previous participant set does not. MPC nodes will attempt to process `verify_foreign_transaction` requests for that chain, but the restored participant set cannot produce a valid threshold signature, causing every such request to time out and permanently fail, freezing any funds or bridge operations that depend on that chain's verification.

2. **Cache shows a chain as unavailable when it is** — if the restored previous participant set covers a chain that the cancelled proposed set did not. Legitimate `verify_foreign_transaction` requests for that chain are silently blocked at the availability gate, breaking the bridge execution flow.

Both outcomes break the production safety invariant that the available-chain set correctly reflects the current signing capability of the participant set, matching the **Medium** allowed impact: *"contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

Resharing cancellation is a normal, designed protocol operation. Any threshold of previous-running-state participants can trigger it by calling `vote_cancel_resharing`. No privileged access, key leakage, or collusion above threshold is required. The stale cache condition is triggered deterministically whenever:

1. At least one participant calls `register_foreign_chains_config` during the resharing window (updating the cache to reflect the proposed participant set), and
2. The resharing is subsequently cancelled via `vote_cancel_resharing`.

Both steps are routine operations in the protocol lifecycle (e.g., a new node pre-registers its foreign-chain support before resharing completes, then resharing is cancelled because a proposed participant goes offline). [5](#0-4) 

---

### Recommendation

Add `self.recompute_available_foreign_chains()` immediately after the state transition in `vote_cancel_resharing`, mirroring the pattern in `vote_reshared`:

```rust
pub fn vote_cancel_resharing(&mut self) -> Result<(), Error> {
    Self::assert_caller_is_signer();
    if let Some(new_state) = self.protocol_state.vote_cancel_resharing()? {
        self.protocol_state = new_state;
        self.recompute_available_foreign_chains(); // ← add this
    }
    Ok(())
}
```

Additionally, consider spawning the same set of cleanup promises (`REMOVE_NON_PARTICIPANT_UPDATE_VOTES`, `CLEAN_FOREIGN_CHAIN_DATA`, `REMOVE_NON_PARTICIPANT_TEE_VERIFIER_VOTES`, etc.) that `vote_reshared` spawns, since the participant set changes on cancellation just as it does on successful resharing.

---

### Proof of Concept

1. Deploy the contract in `Running` state with participants `{A, B, C}`, threshold 2. All three register Bitcoin support → Bitcoin is available.
2. Start resharing to `{A, B, D}` (removing C, adding D). D registers Bitcoin support during resharing → `recompute_available_foreign_chains` is called with the proposed set `{A, B, D}`.
3. Now C (old participant) deregisters Bitcoin support (or D registers with no chains). The cache is recomputed: only A and B cover Bitcoin → Bitcoin still available (2 ≥ threshold 2).
4. Cancel resharing: `vote_cancel_resharing` is called by A and B (threshold met). State reverts to `{A, B, C}` with threshold 2. **`recompute_available_foreign_chains` is NOT called.**
5. Query `get_available_foreign_chains()` → returns the stale cache value from step 3, which does not account for C's current registration state in the restored participant set.
6. The contract's reported available chains diverge from the actual signing capability of the restored `{A, B, C}` participant set, breaking the bridge execution flow for any chain affected by the discrepancy. [2](#0-1) [1](#0-0)

    fn recompute_available_foreign_chains__should_update_cache_during_resharing() {
        // Given: Running contract with Bitcoin whitelisted; threshold 3, only 2 participants
        // registered → Bitcoin not yet available.
        let (_context, mut contract, _) =
            basic_setup_with_protocol(Protocol::CaitSith, DomainPurpose::ForeignTx, &mut OsRng);
        let participants = participant_account_ids(&contract);
        whitelist_chain(&mut contract, dtos::ForeignChain::Bitcoin);
        for account_id in participants.iter().take(2) {
            register_foreign_chain_config(&mut contract, account_id, [dtos::ForeignChain::Bitcoin]);
        }
        assert!(contract.get_available_foreign_chains().is_empty());

        // Transition to Resharing. Use the same participant set so mocked attestations remain valid.
        let resharing = {
            let ProtocolContractState::Running(ref mut state) = contract.protocol_state else {
                panic!("expected Running state");
            };
            let proposal =
                ProposedThresholdParameters::new(state.parameters.clone(), BTreeMap::new());
            state
                .transition_to_resharing_no_checks(&proposal)
                .expect("contract has at least one domain")
        };
        contract.protocol_state = ProtocolContractState::Resharing(resharing);

        // When: the 3rd participant (from the old running set) registers during Resharing.
        register_foreign_chain_config(
            &mut contract,
            &participants[2],
            [dtos::ForeignChain::Bitcoin],
        );

        // Then: cache updated using the embedded previous running-state — Bitcoin now available.
        assert!(
            contract
                .get_available_foreign_chains()
                .contains(&dtos::ForeignChain::Bitcoin)
        );
    }
