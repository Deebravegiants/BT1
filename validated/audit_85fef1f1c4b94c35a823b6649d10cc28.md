### Title
Caller-Controlled `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay to Forge Foreign-Chain Verification Attestations - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx()` verifies that `response.signature` is a valid MPC signature over `response.payload_hash`, but never checks that `response.payload_hash` is the canonical hash derived from the submitted `request`. A single Byzantine attested participant (below threshold) can recycle a valid MPC signature from any prior legitimate signing operation and use it to resolve a different pending `verify_foreign_transaction` request, delivering a forged attestation to the caller.

---

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` accepts a `VerifyForeignTransactionResponse` from an attested participant and resolves the pending yield for the matching `request`. The signature check performed is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;  // taken from caller-supplied response

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // caller-controlled
    &secp_pk,
)
.is_ok()
```

The contract only verifies: *"is `response.signature` a valid signature over `response.payload_hash`?"*

It does **not** verify: *"does `response.payload_hash` equal `SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: ... }))`?"*

The `payload_hash` field is entirely caller-controlled. The contract never derives the expected hash from the stored `request` and compares it against the submitted `response.payload_hash`.

By contrast, the regular `respond()` function for sign requests derives the expected public key from the stored request's tweak and verifies the signature against the payload taken from the stored request — not from the response. The foreign-tx path lacks this binding.

The `ForeignTxSignPayload::compute_msg_hash()` function exists and is used correctly by honest nodes and by the SDK's `ForeignChainSignatureVerifier::verify_signature()`, but the contract itself never calls it during response validation.

---

### Impact Explanation

A single Byzantine attested participant (below threshold) who possesses any previously obtained valid MPC threshold signature over some hash `H` (e.g., from a prior legitimate `verify_foreign_transaction` for tx_id=A) can:

1. Observe a new pending `verify_foreign_transaction` request for tx_id=B.
2. Call `respond_verify_foreign_tx(request_B, response={ payload_hash: H, signature: sig_A })`.
3. The contract accepts: `sig_A` is a valid threshold signature over `H`, so `signature_is_valid = true`.
4. `resolve_yields_for` resolves the pending yield for `request_B` with the forged response.
5. The caller of `verify_foreign_transaction(tx_id=B)` receives a `VerifyForeignTransactionResponse` containing a valid MPC signature, but the signature attests to tx_id=A's payload, not tx_id=B.

Any on-chain bridge contract (e.g., Omnibridge inbound flow) that trusts the returned `VerifyForeignTransactionResponse` without independently re-deriving the expected `payload_hash` from the original request will process a transaction that was never actually verified on the foreign chain. This enables invalid bridge execution or double-spend conditions.

The SDK's `ForeignChainSignatureVerifier::verify_signature()` does perform the payload_hash binding check, but this is an off-chain SDK helper — the on-chain contract does not enforce it, and bridge contracts that call `verify_foreign_transaction` directly and trust the on-chain response are vulnerable.

---

### Likelihood Explanation

- Requires a single Byzantine attested participant (below threshold) — within the allowed attacker model.
- The attacker needs a previously obtained valid MPC threshold signature over any hash. Such signatures are produced routinely by the network for every completed `verify_foreign_transaction` or `sign` request and are publicly visible on-chain.
- No threshold collusion is required: the attacker only needs to submit a transaction calling `respond_verify_foreign_tx` with a recycled `(payload_hash, signature)` pair.
- The attack is replayable across any pending request whose `request` key is in `pending_verify_foreign_tx_requests`.

---

### Recommendation

In `respond_verify_foreign_tx`, derive the expected `payload_hash` from the `request` fields and the `response.values` (or require the node to include the full `ForeignTxSignPayload` in the response so the contract can recompute and compare). At minimum, the contract must verify:

```rust
// Recompute expected hash from the request and the submitted values
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(), // nodes must submit extracted values
});
let expected_hash = expected_payload.compute_msg_hash()?;
assert_eq!(response.payload_hash, expected_hash, "payload_hash mismatch");
```

This mirrors the binding already enforced by `ForeignChainSignatureVerifier::verify_signature()` in the SDK and closes the replay path.

---

### Proof of Concept

1. Legitimate flow: user submits `verify_foreign_transaction(Bitcoin, tx_id=A)`. Honest nodes process it, produce threshold signature `sig_A` over `hash_A = SHA-256(borsh(ForeignTxSignPayload{request: Bitcoin/tx_id=A, values: [block_hash_A]}))`. Response is stored on-chain.

2. New request: user submits `verify_foreign_transaction(Bitcoin, tx_id=B)`. This creates a pending yield under key `request_B`.

3. Byzantine participant (single, attested) calls:
   ```
   respond_verify_foreign_tx(
     request = request_B,                    // matches pending entry
     response = {
       payload_hash: hash_A,                 // hash from tx_id=A's response
       signature: sig_A,                     // valid threshold sig over hash_A
     }
   )
   ```

4. Contract check at line 729: `verify_ecdsa_signature(sig_A, hash_A, root_pk)` → **passes** (sig_A is a valid threshold signature over hash_A).

5. `resolve_yields_for` resolves `request_B`'s yield with the forged response.

6. The caller of `verify_foreign_transaction(tx_id=B)` receives `{ payload_hash: hash_A, signature: sig_A }` — a valid-looking MPC attestation that actually attests to tx_id=A, not tx_id=B.
