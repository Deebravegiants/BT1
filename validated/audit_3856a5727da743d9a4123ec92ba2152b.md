### Title
ZK-Valid-Paillier Proof Accepts Prime N via Fermat's Little Theorem — (`src/cbmpc/zk/zk_paillier.cpp`)

### Summary

Both `valid_paillier_t::verify` and `valid_paillier_interactive_t::verify` are supposed to prove that N is a product of two large primes (a valid RSA/Paillier modulus). The core check is `sigma[i]^N mod N == rho[i]`. For a prime N, Fermat's Little Theorem gives `a^N ≡ a (mod N)` for all a, so the check reduces to `sigma[i] mod N == rho[i]`. An attacker who sets `sigma[i] = rho[i]` (which they can compute since the DRBG seed is public) trivially satisfies every check, causing `paillier_valid_key = zk_flag::verified` to be set for a prime N. The small-factors check does not detect primality. This is reachable in the ECDSA 2PC DKG flow.

---

### Finding Description

**The broken check:** [1](#0-0) 

```cpp
if (sigma[i].pow_mod(N, N) != rho) return coinbase::error(E_CRYPTO);
```

For a prime N, Fermat's Little Theorem states `a^N ≡ a (mod N)`. So `sigma[i]^N mod N = sigma[i] mod N`. If the attacker sets `sigma[i] = rho[i]`, the check becomes `rho[i] == rho[i]` — always true.

**The attacker can compute rho[i]:** The DRBG seed is `k = hash(N, session_id, aux)`. The attacker controls N (their own Paillier modulus) and knows `session_id` and `aux` as a protocol participant. They can reproduce the exact same DRBG sequence and set `sigma[i] = rho[i]` for all i. [2](#0-1) 

**The small-factors check does not detect primality:** [3](#0-2) 

`check_integer_with_small_primes` only verifies that N has no prime factor ≤ alpha = 2^13 = 8192. A 2048-bit prime N has no small factors at all — it passes this check trivially. [4](#0-3) 

**The coprimality check also passes:** `rho_prod` is a product of values in [1, N-1] mod N. Since N is prime, every element of [1, N-1] is coprime to N, so `mod_t::coprime(rho_prod, N)` returns true. [5](#0-4) 

**`create_pub` does not check for primality either:** [6](#0-5) 

`is_valid_modulus` only checks `m > 1`, bit-length ≤ MAX_MODULUS_BITS, and odd. A 2048-bit prime passes all three.

**The same flaw exists in the interactive variant:** [7](#0-6)

### Impact Explanation

In the ECDSA 2PC DKG, P1 (the malicious party) generates the Paillier key and sends N to P2. P2 calls `step4_p2_output`, which calls `valid.verify(...)` on the interactive proof: [8](#0-7) 

If P1 uses a prime N and supplies `sigma[i] = rho[i]`, `valid.verify` returns SUCCESS and sets `paillier_valid_key = zk_flag::verified`. P2 then accepts the prime N as a valid Paillier key and proceeds with the protocol.

With prime N, P1 knows `phi(N) = N - 1` and can compute `N^{-1} mod phi(N)`. This means P1 can decrypt any Paillier ciphertext under N. During the signing protocol, P2 performs homomorphic operations on P1's Paillier ciphertext and sends back encrypted intermediate values. P1 can decrypt these to recover P2's secret key share `x2`, obtaining the full ECDSA private key without P2's consent.

**Impact: Critical** — a single malicious P1 below threshold can recover P2's secret key share from the signing protocol, forging signatures unilaterally.

---

### Likelihood Explanation

The attack requires:
1. Generating a 2048-bit prime N — computationally feasible (standard primality generation).
2. Computing `rho[i]` from the public DRBG seed — trivial, same code as the honest prover.
3. Setting `sigma[i] = rho[i]` — trivial.

No threshold collusion is required. A single Byzantine P1 can execute this attack.

---

### Recommendation

The fix is to add a primality check (or compositeness check) on N inside `valid_paillier_t::verify` and `valid_paillier_interactive_t::verify`. Specifically, after the small-factors check, verify that N is **not** prime (e.g., using a Miller-Rabin test). Alternatively, the proof protocol itself should be redesigned so that the N-th root extraction check is not trivially satisfiable for prime N — for example, by requiring the prover to demonstrate knowledge of the factorization, or by using a different soundness argument that explicitly excludes prime moduli.

---

### Proof of Concept

```
1. Attacker (P1) generates a 2048-bit prime N.
2. Attacker computes k = hash(N, session_id, aux).bitlen128() and runs
   drbg_aes_ctr_t drbg(k) to obtain rho[i] = drbg.gen_bn(N) for i in [0, t).
3. Attacker sets sigma[i] = rho[i] for all i.
4. Attacker sends sigma[] to P2 as the valid_paillier proof.
5. P2 calls valid.verify():
   - N > 0: PASS (prime N is positive)
   - check_integer_with_small_primes(N, 8192): PASS (prime N has no small factors)
   - sigma[i] > 0: PASS (rho[i] > 0 with overwhelming probability)
   - sigma[i].pow_mod(N, N) == rho[i]: PASS (Fermat: rho[i]^N mod N = rho[i])
   - coprime(rho_prod, N): PASS (all rho[i] in [1,N-1] are coprime to prime N)
   - paillier_valid_key = zk_flag::verified  ← INCORRECTLY SET
6. P2 accepts prime N as a valid Paillier key and proceeds with DKG/signing.
7. During signing, P2 sends Paillier-encrypted intermediate values under N.
8. P1 decrypts them using phi(N) = N-1, recovering P2's secret key share x2.
```

static error_t check_integer_with_small_primes(const bn_t& prime, int alpha) {
  for (int i = 0; i < small_primes_count; i++) {
    int small_prime = small_primes[i];
    if (small_prime > alpha) break;
    if (mod_t::mod(prime, small_prime) == 0) return coinbase::error(E_CRYPTO);
