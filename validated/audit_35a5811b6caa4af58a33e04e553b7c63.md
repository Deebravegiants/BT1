### Title
ML-DSA-65 Wire-Length vs. Trie-Length Mismatch in Gas-Key Host Function Fee Calculation Corrupts Exec Gas - (File: `runtime/near-vm-runner/src/logic/logic.rs`)

### Summary
The host functions `promise_batch_action_add_gas_key_with_full_access`, `promise_batch_action_add_gas_key_with_function_call`, and `promise_batch_action_transfer_to_gas_key` compute gas fees using `public_key_len as usize` — the raw borsh wire length of the key (1953 bytes for ML-DSA-65) — instead of the decoded key's `trie_id_len()` (33 bytes for ML-DSA-65). The transaction-level fee path correctly uses `trie_id_len()`. This is a direct analog to the EIP-712 type hash mismatch: one side of the system defines/uses a different encoding length than the other, causing the gas value charged to be wrong by a factor of ~59×.

### Finding Description

The `gas_key_add_key_exec_fee` function in `core/parameters/src/cost.rs` computes the nonce trie key length as:

```rust
let nonce_key_len =
    access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
```

The **transaction-level** path in `runtime/runtime/src/config.rs` correctly passes `public_key.trie_id_len()` (33 bytes for ML-DSA-65):

```rust
let nonce_fee = gas_key_add_key_exec_fee(
    fees,
    account_id.len(),
    public_key.trie_id_len(),   // ← 33 bytes for ML-DSA-65
    gas_key_info.num_nonces,
);
```

But the **host function** path in `runtime/near-vm-runner/src/logic/logic.rs` passes `public_key_len as usize` — the raw borsh wire length supplied by the contract (1953 bytes for ML-DSA-65):

```rust
let exec_fee = gas_key_add_key_exec_fee(
    &self.fees_config,
    receiver_id.len(),
    public_key_len as usize,    // ← 1953 bytes for ML-DSA-65 (WRONG)
    num_nonces,
);
```

The identical mismatch exists in `runtime/near-vm-runner/src/wasmtime_runner/logic.rs` at the same call sites, and in `promise_batch_action_transfer_to_gas_key` for both `gas_key_transfer_send_fee` and `gas_key_transfer_exec_fee`.

The actual nonce trie key for an ML-DSA-65 access key uses the 33-byte SHA3-256 hash form (`[tag=3] || sha3_256(domain || raw_pubkey)`), not the 1953-byte borsh wire form. The fee calculation therefore uses a length that is 1920 bytes larger than the actual trie key written.

### Impact Explanation

For each nonce written, the overcharge is:

- Extra bytes: 1953 − 33 = **1920 bytes**
- `gas_key_byte` exec fee: 101,435,400 gas/byte
- Extra exec gas per nonce: 1920 × 101,435,400 ≈ **194 Ggas**

For `promise_batch_action_transfer_to_gas_key`, the send fee is also overcharged:
- `gas_key_byte` send fee: 59,357,464 gas/byte
- Extra send gas: 1920 × 59,357,464 ≈ **113 Ggas**

Concrete consequences:
1. **Gas value corrupted**: The exec gas charged for ML-DSA-65 gas-key host function calls is ~59× higher than the actual trie work performed and ~59× higher than the equivalent transaction-level `AddKey` action.
2. **Feature broken at scale**: For ~1,548 nonces, the overcharge alone (~300 Tgas) hits the per-receipt gas limit, causing an out-of-gas failure. The maximum `num_nonces` is `u16::MAX` = 65,535, which would produce ~12.7 Pgas of overcharge — far beyond any gas limit.
3. **Inconsistency between paths**: A transaction-level `AddKey` with an ML-DSA-65 gas key and 1,000 nonces charges the correct exec gas; the identical operation via host function charges ~194 Tgas more. This breaks the invariant that the two paths are equivalent.

### Likelihood Explanation

The `PostQuantumSignatures` protocol feature gate must be active for ML-DSA-65 keys to be accepted by action validation. However, borsh deserialization of `PublicKey::MLDSA65` is explicitly **not** gated, so a contract can pass an ML-DSA-65 key to the host function and trigger the overcharge even before the feature activates (the action will be rejected later, but the gas is already charged). Once `PostQuantumSignatures` is active, any deployed contract calling these host functions with an ML-DSA-65 key will trigger the mismatch.

### Recommendation

Replace `public_key_len as usize` with the decoded key's `trie_id_len()` in all three host function fee calculations:

- `runtime/near-vm-runner/src/logic/logic.rs` lines 3158, 3229, 3091–3095
- `runtime/near-vm-runner/src/wasmtime_runner/logic.rs` lines 3398, 3502, 3323–3325

The decoded `PublicKey` is already available at those call sites (via `public_key.decode()?`), so `public_key.decode()?.trie_id_len()` can be used directly, matching the transaction-level path in `runtime/runtime/src/config.rs`.

### Proof of Concept

1. Deploy a contract that calls `promise_batch_action_add_gas_key_with_full_access` targeting itself, passing a borsh-encoded ML-DSA-65 public key (1953 bytes) and `num_nonces = 1000`.
2. The host function computes `nonce_key_len = access_key_key_len(10, 1953) + 2 = 1967` and charges `1967 × 101,435,400 × 1000 ≈ 199 Tgas` exec gas.
3. The equivalent transaction-level `AddKey` action charges `access_key_key_len(10, 33) + 2 = 47` → `47 × 101,435,400 × 1000 ≈ 4.8 Tgas` exec gas.
4. The host function overcharges by ~41× the correct amount for this case, and the overcharge grows linearly with `num_nonces`, making the host function unusable for any significant number of nonces. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

pub fn gas_key_add_key_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
    num_nonces: NonceIndex,
) -> GasKeyAddFee {
    let num_nonces = num_nonces as u64;
    let base =
        cfg.fee(ActionCosts::gas_key_nonce_write_base).exec_fee().checked_mul(num_nonces).unwrap();
    let nonce_key_len =
        access_key_key_len(account_id_len, public_key_len) + std::mem::size_of::<NonceIndex>();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((nonce_key_len + AccessKey::NONCE_VALUE_LEN) as u64)
        .unwrap()
        .checked_mul(num_nonces)
        .unwrap();
    GasKeyAddFee { base, per_byte }
