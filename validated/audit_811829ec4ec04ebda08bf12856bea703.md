### Title
NFT Royalty Bypass via Integer Division Rounding to Zero in Offer Settlement - (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary
The `compute_royalty_amount` function and the `trade_prices_list` filter in `make_nft1_offer` both use integer floor division that rounds to zero for small offered amounts. When the royalty rounds to zero, the `trade_prices_list` embedded in the NFT CLVM spend is empty, causing the on-chain transfer program to enforce no royalty payments. An NFT seller can exploit this by creating an offer requesting a tiny amount (e.g., 1 mojo), bypassing royalties entirely while settling the actual payment off-chain.

### Finding Description

**Root cause 1 — `compute_royalty_amount` truncates to zero:** [1](#0-0) 

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
```

For any `offered_amount` where `abs(offered_amount) * percentage < MAX_ROYALTY_BASIS_POINTS` (10000), the result is 0. The test suite explicitly acknowledges this: [2](#0-1) 

A 1% royalty (100 basis points) on any amount ≤ 99 mojos rounds to zero. A 5% royalty (500 basis points) on any amount ≤ 19 mojos rounds to zero. Even a 99% royalty (9900 basis points) on 1 mojo rounds to zero.

**Root cause 2 — `make_nft1_offer` filters out zero-royalty prices from `trade_prices_list`:** [3](#0-2) 

```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
```

When the requested amount per NFT is small enough that `price[0] * royalty_percentage // 10000 == 0`, the price is excluded. With a requested amount of 1 mojo and any royalty percentage < 10000 basis points (i.e., < 100%), all prices are excluded, producing an **empty `trade_prices_list`**.

**Root cause 3 — The CLVM transfer program only enforces royalties present in `trade_prices_list`:** [4](#0-3) 

The `trade_prices` list is built from requested fungible amounts divided by `offer_side_royalty_split`. This per-NFT price is what gets passed to the CLVM transfer program. An empty list means the transfer program creates no royalty payment conditions on-chain.

**The full offer construction path:** [5](#0-4) 

`compute_royalty_amount` is called here to build `royalty_payments`. If the result is 0 for all NFTs, `royalty_payments` contains only zero-amount `CreateCoin` entries, and the royalty coin spend is skipped entirely: [6](#0-5) 

### Impact Explanation

An NFT seller can bypass royalty payments owed to NFT creators by crafting an offer with a tiny requested amount. The NFT CLVM spend is created with an empty `trade_prices_list`, so the on-chain transfer program enforces no royalty conditions. The NFT creator receives 0 royalty instead of their entitled percentage. This constitutes unauthorized accounting change and offer settlement bypass affecting NFT-controlled singleton assets.

### Likelihood Explanation

The attack requires the NFT seller to coordinate with the buyer to settle the actual payment off-chain (the on-chain offer requests only 1 mojo). This coordination is trivially achievable — seller and buyer simply agree to the real price privately. No special privileges, leaked keys, or cryptographic breaks are required. Any NFT owner can create such an offer. The attack is repeatable across all NFTs with any non-zero royalty percentage.

### Recommendation

1. In `compute_royalty_amount`, reject offers where a non-zero royalty percentage produces a zero royalty amount:

```python
if percentage > 0 and amount == 0:
    raise ValueError("Royalty rounds to zero; offered amount is too small to enforce royalties")
```

2. In `make_nft1_offer`, instead of silently filtering out zero-royalty prices, raise an error:

```python
for price in trade_prices:
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS == 0:
        raise ValueError("Offered amount too small to enforce on-chain royalties for this NFT")
```

### Proof of Concept

1. Mint an NFT with 5% royalty (500 basis points) to creator Bob.
2. Alice (NFT owner) calls `make_nft1_offer` with `offer_dict = {nft_asset_id: -1, None: 1}` (offer NFT, request 1 mojo XCH).
3. Inside `make_nft1_offer`:
   - `trade_prices` = `[(uint64(1), OFFER_MOD_HASH)]` (1 mojo per NFT)
   - Filter check: `1 * 500 // 10000 = 0` → price is excluded
   - `trade_prices_list` = `[]` (empty)
4. NFT spend is created with empty `trade_prices_list` via `generate_signed_transaction`.
5. Carol accepts the offer, paying 1 mojo on-chain.
6. The CLVM transfer program runs with an empty trade prices list → no `CREATE_COIN` royalty conditions are asserted.
7. Bob receives 0 royalty. Alice and Carol settle the real price (e.g., 1 XCH) off-chain. [7](#0-6) [8](#0-7) [9](#0-8)

```python
MAX_ROYALTY_BASIS_POINTS = 10000


def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
