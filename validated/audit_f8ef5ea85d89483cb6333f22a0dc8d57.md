### Title
EVM `initTransfer` Accepts `uint128` Amounts Exceeding Solana's `u64::MAX`, Causing Permanent Fund Lock on Cross-Chain EVM→Solana Transfers — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The EVM `OmniBridge.initTransfer` accepts `uint128 amount` with no upper-bound check against `u64::MAX`. When the destination chain is Solana, the Solana `finalize_transfer` instruction must cast the `u128` amount from the signed NEAR payload to `u64` via `try_into()`. Any amount exceeding `u64::MAX` (18,446,744,073,709,551,615) causes this cast to fail with `AmountOverflow`, permanently blocking finalization. Because the EVM tokens are already burned or locked and no cross-chain refund path exists, the funds are irreversibly locked.

---

### Finding Description

**Step 1 — EVM accepts `uint128` with no Solana-aware bound check.**

`OmniBridge.initTransfer` is declared:

```solidity
function initTransfer(
    address tokenAddress,
    uint128 amount,
    uint128 fee,
    uint128 nativeFee,
    string calldata recipient,
    string calldata message
) external payable whenNotPaused(PAUSED_INIT_TRANSFER)
``` [1](#0-0) 

The only fee validation is `fee >= amount` → revert. There is no check that `amount <= type(uint64).max` even when `recipient` encodes a Solana destination. Tokens are burned or locked unconditionally: [2](#0-1) 

**Step 2 — Solana `finalize_transfer` hard-casts `u128 → u64`.**

The Solana `FinalizeTransferPayload` stores `amount: u128`: [3](#0-2) 

Both the native-token path and the bridged-token path cast this to `u64` via `try_into()`: [4](#0-3) [5](#0-4) 

The SOL path has the same cast: [6](#0-5) 

**Step 3 — Nonce rollback does not help.**

Because Solana transactions are atomic, the `UsedNonces::use_nonce` call is rolled back on failure. The nonce is never permanently consumed, so the relayer can retry indefinitely — but every retry will fail with `AmountOverflow` because the amount in the MPC-signed payload is fixed and cannot be changed. [7](#0-6) 

**Step 4 — No cross-chain refund path.**

There is no mechanism in the EVM bridge or the NEAR bridge to issue a refund when a Solana finalization permanently fails. The EVM tokens remain burned/locked with no recovery path.

---

### Impact Explanation

This matches the allowed impact: **Irreversible fund lock — permanently unclaimable user value in bridge flows.**

Any EVM→Solana transfer where `amount > u64::MAX` results in:
- EVM tokens burned or locked (irreversible).
- NEAR MPC signs a payload with the oversized amount.
- Every Solana `finalize_transfer` call reverts with `AmountOverflow`.
- No refund or recovery mechanism exists.

---

### Likelihood Explanation

`u64::MAX` in raw token units is `18,446,744,073,709,551,615`. For tokens with high supply or low per-unit value (e.g., 18-decimal meme tokens where `u64::MAX` ≈ 18.4 billion whole tokens), this threshold is reachable. The EVM interface accepts `uint128`, so a user can supply any value up to `2^128 - 1` without any on-chain rejection. The likelihood is **low-to-medium** depending on token economics, but the impact is total and irreversible when triggered.

---

### Recommendation

Add a destination-chain-aware upper-bound check in `OmniBridge.initTransfer` (and `initTransfer1155`) when the recipient encodes a Solana destination, or add a global guard:

```solidity
// In initTransfer, before burning/locking:
require(amount <= type(uint64).max, "Amount exceeds Solana u64 max");
```

Alternatively, enforce the bound in the NEAR bridge before signing the Solana-destined payload, so the MPC never produces a signature for an amount that Solana cannot finalize.

---

### Proof of Concept

1. Deploy EVM `OmniBridge` pointing to a Solana-destined recipient string (e.g., `"sol:<base58_address>"`).
2. Call `initTransfer(tokenAddress, 2**64, 0, 0, "sol:<addr>", "")` — this succeeds; tokens are burned.
3. NEAR MPC signs a `FinalizeTransferPayload` with `amount = 2**64`.
4. Relayer calls Solana `finalize_transfer` with the signed payload.
5. `data.amount.try_into::<u64>()` returns `Err(_)` → `error!(ErrorCode::AmountOverflow)` → transaction reverts.
6. Nonce is rolled back; relayer retries → same failure forever.
7. EVM tokens are permanently lost with no refund path.
