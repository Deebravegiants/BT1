Audit Report

## Title
Missing Zero-Output Guard in ERC20 Token Deposit Functions Allows Dust-Amount Token Loss - (File: contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPool.sol)

## Summary
The `deposit(address token, uint256 amount, string memory referralId)` functions in `RSETHPoolNoWrapper` and `RSETHPool` perform no check that the computed `rsETHAmount` is non-zero before transferring the user's input tokens into the pool. For tokens with fewer than 18 decimals (e.g., USDC at 6 decimals), a sufficiently small `amount` causes integer division to truncate `rsETHAmount` to zero. The user's tokens are permanently absorbed into the pool while the user receives nothing in return.

## Finding Description
In `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee(uint256 amount, address token)` (lines 292–312) and the equivalent in `RSETHPool` (lines 326–347), the output is computed as:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`rsETHToETHrate` is enforced to be `>= 1e18` by `InterimRSETHOracle._setRate` (which reverts if `newRate < 1e18`). `tokenToETHRate` for a 6-decimal token such as USDC is approximately `price_in_ETH * 1e18 / 1e6 * 1e6 = price_in_ETH * 1e18` — but the oracle normalizes to 1e18 precision regardless of token decimals. For USDC at ~$1 with ETH at ~$3000, `tokenToETHRate ≈ 3.33e14`. The truncation condition is:

```
amountAfterFee * 3.33e14 < 1e18  →  amountAfterFee < ~3000 wei of USDC
```

Any deposit of fewer than ~3000 wei of USDC (≈ $0.003) produces `rsETHAmount = 0`. The deposit function then executes:

```solidity
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);  // tokens leave user
rsETH.safeTransfer(msg.sender, 0);                                   // silent no-op
```

The `safeTransfer(msg.sender, 0)` call succeeds silently under ERC-20 semantics. The deposited tokens are credited to the pool's balance with no user-accessible recovery path. The only existing guard is `if (amount == 0) revert InvalidAmount()`, which does not protect against a non-zero `amount` that produces a zero output.

The same pattern exists in `RSETHPool.deposit(address token, ...)` at lines 284–305, where `IERC20(address(wrsETH)).safeTransfer(msg.sender, 0)` is the silent no-op.

## Impact Explanation
**Low — Contract fails to deliver promised returns.**

A depositor who sends a dust-level amount of a supported low-decimal ERC20 token (e.g., USDC, USDT) receives zero rsETH while their tokens are permanently absorbed into the pool. The protocol does not lose value; the depositor loses their dust input. This matches the allowed Low impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
Triggerable by any unprivileged external caller with no special preconditions beyond holding a supported low-decimal token. The threshold is approximately $0.003 per transaction for USDC at current ETH prices. While the per-transaction loss is negligible, the condition is deterministic and repeatable. A user unfamiliar with the dust threshold, or a UI that does not pre-validate the output quote, could trigger this inadvertently.

## Recommendation
Add a zero-output revert immediately after computing `rsETHAmount` in both `deposit(address token, ...)` functions:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, expose `viewSwapRsETHAmountAndFee` as a pre-check in the UI and revert on-chain when the output is zero. A caller-supplied `minRsETHAmount` parameter (slippage guard) would also address this and the broader rounding risk.

## Proof of Concept
Foundry fuzz/unit test plan:

```solidity
function test_dustDepositZeroOutput() public {
    // Setup: deploy RSETHPoolNoWrapper with USDC as supported token,
    // rsETHToETHrate = 1.05e18 (realistic post-yield rate),
    // tokenToETHRate for USDC = 3.33e14 (USDC/ETH at ETH=$3000)

    address usdc = address(mockUSDC);  // 6-decimal mock
    uint256 dustAmount = 2999;         // wei of USDC, below truncation threshold

    // Pre-condition: pool holds rsETH to transfer
    rsETH.mint(address(pool), 1e18);

    uint256 userUSDCBefore = mockUSDC.balanceOf(user);
    uint256 userRsETHBefore = rsETH.balanceOf(user);

    vm.prank(user);
    mockUSDC.approve(address(pool), dustAmount);
    vm.prank(user);
    pool.deposit(usdc, dustAmount, "ref");  // should revert but doesn't

    // Assert: user lost USDC, received 0 rsETH
    assertEq(mockUSDC.balanceOf(user), userUSDCBefore - dustAmount);
    assertEq(rsETH.balanceOf(user), userRsETHBefore);  // unchanged
    // Pool absorbed the dust with no revert
}
```
