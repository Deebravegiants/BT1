### Title
Unchecked Detached Refund Promise Silently Discards Excess Deposits — (`File: crates/contract/src/lib.rs`)

### Summary
The `require_deposit` helper in the MPC contract refunds excess user deposits via `Promise::new(predecessor.clone()).transfer(diff).detach()`. The `.detach()` call schedules the transfer as a fire-and-forget promise whose outcome is never observed. If the transfer receipt fails, the excess deposit is permanently locked inside the contract with no recovery path. This is the NEAR analog of Solidity's `address.transfer()` pattern: both are "push" refund mechanisms that silently swallow failures.

### Finding Description
In `crates/contract/src/lib.rs`, the `require_deposit` function is the single refund gate called by every user-facing entry point (`sign`, `request_app_private_key`, `verify_foreign_transaction`):

```rust
fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => { /* panic */ }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach(); // ← detached, result ignored
            }
        }
    }
}
``` [1](#0-0) 

`.detach()` in NEAR SDK means the promise is enqueued but the current execution context does not await it and does not propagate its failure. If the transfer receipt is rejected by the runtime (e.g., the predecessor account no longer exists at receipt execution time, or a future NEAR protocol change raises the base cost of a transfer action above the gas budget implicitly allocated to a detached promise), the refund silently disappears. The contract's own balance is reduced by `diff` (the transfer is deducted optimistically), yet the predecessor never receives the funds, and there is no on-chain event or error that would allow recovery.

The same `.detach()` pattern appears in the `tee` module's deposit-refund paths, making the surface broader than a single call site. [2](#0-1) 

### Impact Explanation
Every call to `sign`, `request_app_private_key`, or `verify_foreign_transaction` that attaches more than the 1-yoctoNEAR minimum deposit goes through `require_deposit`. A failed detached transfer permanently locks the excess in the contract. Because the contract exposes no administrative withdrawal method for such stranded funds, the loss is irreversible. Over time, or under a NEAR protocol gas-repricing event, this accumulates into a permanent, unrecoverable balance discrepancy — a production safety/accounting invariant violation matching the **Medium** allowed impact tier.

### Likelihood Explanation
Under current NEAR mainnet conditions `Promise::transfer` to an existing account succeeds reliably, so the probability of a single failure is low. However:
- NEAR protocol upgrades have historically repriced host-function gas costs; a future repricing could make the implicit gas budget of a detached promise insufficient.
- A predecessor contract account that self-destructs (via `DeleteAccount`) between transaction submission and receipt execution would cause the transfer to fail silently.
- The pattern is present on every user-facing signing entry point, so the cumulative exposure grows with protocol usage.

The external report's reference scenario (EVM SLOAD repricing from 200 → 800 gas breaking `transfer()`) is directly analogous to a NEAR host-function repricing breaking a detached transfer.

### Recommendation
Replace the detached transfer with an awaited promise and propagate the result, or use NEAR's `#[handle_result]` pattern to surface refund failures to the caller:

```rust
// Instead of:
Promise::new(predecessor.clone()).transfer(diff).detach();

// Use an awaited return so failure is observable:
return PromiseOrValue::Promise(Promise::new(predecessor.clone()).transfer(diff));
```

If a fire-and-forget refund is architecturally required, at minimum log a structured error in a `.then` callback so that stranded funds can be detected and recovered via governance.

### Proof of Concept
1. Deploy the MPC contract on a NEAR sandbox.
2. Call `sign(...)` from a contract account `caller.near`, attaching 2 yoctoNEAR (1 yocto above the minimum).
3. Before the refund receipt executes, delete `caller.near` (possible in sandbox via `DeleteAccount`).
4. Observe: the main `sign` receipt succeeds, the refund receipt fails silently, and the 1-yocto excess remains locked in the MPC contract with no recovery path.
5. Repeat across many requests to demonstrate accumulation. [3](#0-2) [4](#0-3)

/// Checks that the caller attached at least `minimum_deposit` and refunds any excess.
///
/// A non-zero deposit is required so that the transaction must be signed by a
/// full-access key (or a function-call access key whose `deposit` allowance is
/// explicitly set). This prevents a **malicious frontend** from silently
/// submitting signature requests on behalf of a user via a restricted
/// function-call access key, because such keys cannot attach deposits by
/// default. In other words, requiring a deposit ensures the user (or their
/// full-access key) explicitly authorised the call.
///
/// See the "Deposit requirement" section in the contract README for more
/// details.
fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}
