Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Deposit Cap Bypass — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison: for LSTs it evaluates `totalAssetDeposits + amount > limit`, but for ETH it evaluates only `totalAssetDeposits > limit`, omitting the incoming `amount`. When `totalAssetDeposits == limit`, the ETH check returns `false` and the deposit proceeds, pushing total ETH deposits above the configured cap. Any unprivileged user can exploit this to breach the deposit limit invariant.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without adding `amount`, while the LST branch correctly returns `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. When `totalAssetDeposits` equals the configured limit exactly, the strict `>` comparison evaluates to `false`, so `_beforeDeposit` (L661–663) does not revert with `MaximumDepositLimitReached`. Control returns to `depositETH` (L76–93), which is `external payable` and callable by any user, and `_mintRsETH` is called, minting rsETH for the over-limit deposit. After the call, `getTotalAssetDeposits(ETH_TOKEN) > depositLimitByAsset(ETH_TOKEN)`, breaking the invariant that is correctly enforced for every LST. The existing `getAssetCurrentLimit` (L402–409) uses `>` to detect an already-exceeded state but does not prevent the breach from occurring in the first place.

## Impact Explanation
The deposit limit is the protocol's primary mechanism for bounding ETH exposure to EigenLayer strategies and NodeDelegators. Bypassing it means the protocol accepts and restakes more ETH than governance intended. No funds are directly stolen, but the protocol fails to deliver the promised deposit-cap guarantee for ETH while enforcing it correctly for LSTs. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
Any unprivileged external caller can reach `depositETH`. The required precondition (`totalAssetDeposits == depositLimit`) is fully attacker-controllable: an attacker reads on-chain state, deposits exactly `depositLimit − totalAssetDeposits` ETH to bring the total to the boundary, then immediately deposits an arbitrary additional amount. No privileged role, victim mistake, or external dependency is required. The attack is repeatable as long as the limit is not raised.

## Recommendation
Add the incoming `amount` to the ETH branch to match the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // fix: add amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Read `D = lrtConfig.depositLimitByAsset(ETH_TOKEN)` and `T = getTotalAssetDeposits(ETH_TOKEN)`.
2. Call `depositETH{value: D − T}(0, "")` to bring `totalAssetDeposits` exactly to `D`.
3. Call `depositETH{value: X}(0, "")` for any `X > 0`.
   - `_checkIfDepositAmountExceedesCurrentLimit`: `D > D` → `false` → no revert.
   - `_mintRsETH` mints rsETH for `X`.
4. Assert `getTotalAssetDeposits(ETH_TOKEN) == D + X > D`, confirming the cap is breached while the caller holds freshly minted rsETH.

Foundry invariant test: set `depositLimitByAsset(ETH_TOKEN) = D`, fuzz `depositETH` calls, and assert `getTotalAssetDeposits(ETH_TOKEN) <= D` after every call. The invariant will fail at step 3 above. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
