### Title
Cross-Network Replay of `SignedDelegateAction` Due to Missing Chain Identifier in Signed Payload - (File: core/primitives/src/action/delegate.rs)

### Summary

`DelegateAction` (meta-transaction) signatures in nearcore do not include any chain-specific identifier (chain ID, genesis hash, or network ID) in the signed payload. A malicious relayer who receives a `SignedDelegateAction` intended for one NEAR network (e.g., testnet) can replay it verbatim on a different NEAR network (e.g., mainnet), causing the user's actions to execute on the wrong network without their consent.

### Finding Description

The `DelegateAction` struct is the user-signed payload for NEAR meta-transactions (NEP-366). The user signs it independently and hands it to an untrusted relayer, who wraps it in an outer `SignedTransaction` and submits it. The security model explicitly assumes the relayer is untrusted.

The signed hash is computed in `get_nep461_hash()`:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
``` [1](#0-0) 

`SignableMessage` prepends only a fixed NEP discriminant (`2^30 + 366`) to the borsh-serialized `DelegateAction`:

```rust

    pub discriminant: MessageDiscriminant,
    pub msg: &'a T,
}
``` [2](#0-1) 

The discriminant is a compile-time constant with no chain-specific component:

```rust

// ...
MessageDiscriminant::new_on_chain(NEP_366_META_TRANSACTIONS).unwrap()

The `DelegateAction` struct itself contains no chain identifier:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
}
``` [4](#0-3) 

The outer `SignedTransaction` wrapping the delegate action does include a `block_hash` that ties it to a specific chain, but that hash is signed by the **relayer**, not the user. The user's `SignedDelegateAction` is signed independently and is the only thing protecting the user's intent. Since the relayer constructs the outer transaction, a malicious relayer can create a fresh outer transaction on any NEAR network and embed the user's unchanged `SignedDelegateAction` inside it.

Verification in `apply_delegate_action` calls `signed.verify()` which only checks the signature against the chain-agnostic `get_nep461_hash()`:

```rust
impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;
        self.signature.verify(hash.as_ref(), public_key)
    }
}
``` [5](#0-4) 

### Impact Explanation

A malicious relayer who receives a user's `SignedDelegateAction` intended for testnet can submit it on mainnet (or any other NEAR network) by constructing a new outer transaction with a valid mainnet `block_hash`. If the user has the same account and key on mainnet (common for developers and power users), and the nonce encoded in the `DelegateAction` is greater than the user's current mainnet access-key nonce, the runtime will accept and execute the action. This can result in:

- Unauthorized token transfers on mainnet
- Unauthorized contract calls (e.g., FT transfers, NFT operations)
- Unauthorized key additions or deletions

The corrupted protocol value is the **account balance and state** on the target network — funds are moved or keys are modified without the user's authorization for that network.

### Likelihood Explanation

The conditions for exploitation are:
1. The user has the same account ID and signing key on both testnet and mainnet (very common for developers and wallets that derive keys from the same seed).
2. The user's mainnet access-key nonce is lower than the nonce in the `DelegateAction` (likely if the user is more active on testnet).
3. The `max_block_height` in the `DelegateAction` is still in the future on mainnet (block heights on testnet and mainnet are comparable in magnitude).

All three conditions are routinely satisfied. The relayer is explicitly an untrusted party in the meta-transaction model, making this an unprivileged attacker-controlled entry path.

### Recommendation

Include a chain-specific identifier in the `DelegateAction` signed payload. The most robust approach is to add the genesis block hash (or a dedicated chain ID) to `DelegateAction` and incorporate it into `get_nep461_hash()`. This mirrors the protection that `block_hash` provides for outer `SignedTransaction`s. Alternatively, `SignableMessage` could be extended with a network-scoped domain separator, analogous to EIP-712's `chainId` field.

### Proof of Concept

1. User on testnet creates and signs a `DelegateAction` (e.g., FT transfer of 100 NEAR-denominated tokens) with `nonce = 5`, `max_block_height = 200_000_000`.
2. User sends the `SignedDelegateAction` to a relayer, expecting it to be submitted on testnet.
3. Malicious relayer instead constructs a mainnet outer `SignedTransaction` with a valid mainnet `block_hash`, embedding the user's unchanged `SignedDelegateAction`.
4. The mainnet runtime calls `signed_delegate_action.verify()`, which recomputes `get_nep461_hash()` — identical on both networks — and the signature verifies successfully.
5. The runtime checks `nonce (5) > current_mainnet_ak_nonce (e.g., 3)` — passes.
6. The runtime checks `current_mainnet_height < max_block_height (200_000_000)` — passes.
7. The FT transfer executes on mainnet, draining the user's mainnet balance without their consent. [6](#0-5) [7](#0-6)

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
