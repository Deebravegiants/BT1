### Title
Uninitialized `minRsEthAmountToWithdraw` Allows Dust Withdrawal Queue Flooding, Causing Unbounded Gas and Temporary Fund Freeze - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal()` enforces a minimum rsETH amount via `minRsEthAmountToWithdraw[asset]`, but this mapping is never initialized in `initialize()` and defaults to `0` for every asset. The guard therefore only blocks a zero-value call, allowing any unprivileged user to flood the withdrawal queue with 1-wei dust entries. The operator-called `unlockQueue` iterates the entire pending queue in an unbounded `while` loop, making it run out of gas when the queue is large, which temporarily freezes all legitimate withdrawals.

---

### Finding Description

`LRTWithdrawalManager` declares `minRsEthAmountToWithdraw` as a per-asset mapping: [1](#0-0) 

The `initialize()` function sets only `withdrawalDelayBlocks` and never touches this mapping: [2](#0-1) 

Consequently every asset's minimum is `0`. The guard in `initiateWithdrawal` is: [3](#0-2) 

When `minRsEthAmountToWithdraw[asset] == 0`, the condition `rsETHUnstaked < 0` is always false, so any call with `rsETHUnstaked >= 1` passes. Each accepted call appends a `WithdrawalRequest` to the queue and increments `nextUnusedNonce[asset]`: [4](#0-3) 

The operator-facing `unlockQueue` processes the queue via `_unlockWithdrawalRequests`, which iterates in an unbounded `while` loop from `nextLockedNonce` up to `firstExcludedIndex`: [5](#0-4) 

With tens of thousands of dust entries ahead of legitimate requests, this loop exhausts the block gas limit, making `unlockQueue` uncallable and freezing all pending withdrawals.

The same uninitialized-minimum pattern exists in `LRTDepositPool` for `minAmountToDeposit`: [6](#0-5) [7](#0-6) 

This lets the attacker cheaply acquire dust rsETH (1 wei ETH → dust rsETH) to fuel the queue-flooding attack: [8](#0-7) 

The admin setter `setMinRsEthAmountToWithdraw` exists but is never called during deployment, leaving the window open from the moment the contract is live: [9](#0-8) 

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

An attacker who floods the queue with N dust entries forces the operator's `unlockQueue` `while` loop to iterate through all N entries before reaching any legitimate request. Once N is large enough to exceed the block gas limit, `unlockQueue` reverts on every call, and no withdrawal request — including those of honest users who have already burned their rsETH — can be unlocked or completed. Funds are temporarily frozen until the admin sets a non-zero minimum and the operator processes the backlog in smaller batches (if the contract supports it).

Additionally, the `_unlockWithdrawalRequests` loop itself constitutes **unbounded gas consumption** (Medium), as the operator must pay for iterating every dust entry.

---

### Likelihood Explanation

**High.** The attack requires only:
1. A small amount of ETH (1 wei per iteration) to deposit via `LRTDepositPool.depositETH()` and receive dust rsETH — possible because `minAmountToDeposit == 0`.
2. Repeated calls to `initiateWithdrawal` with 1-wei rsETH — no role or permission required.

No admin compromise, oracle manipulation, or front-running is needed. The vulnerability is exploitable from the first block after deployment.

---

### Recommendation

1. **Initialize `minRsEthAmountToWithdraw` in `initialize()`** for each supported asset to a protocol-meaningful floor (e.g., equivalent to 0.001 ETH in rsETH terms), mirroring how `withdrawalDelayBlocks` is set.
2. **Initialize `minAmountToDeposit` in `LRTDepositPool.initialize()`** to a non-zero value to prevent dust rsETH minting that fuels the attack.
3. **Add a per-user cap** on the number of pending withdrawal requests (analogous to `maxNumberOfWithdrawalsPerUser` used in `KernelDepositPool`) to bound queue growth per address.
4. **Enforce a non-zero minimum** in `setMinRsEthAmountToWithdraw` (revert if `_minAmount == 0`), consistent with the pattern used in `KernelVaultETH.setMinDeposit`. [10](#0-9)

### Proof of Concept

```solidity
// Attacker script (pseudo-code)
// Precondition: minRsEthAmountToWithdraw[ETH_TOKEN] == 0 (default)
//               minAmountToDeposit == 0 (default)

for (uint i = 0; i < 50_000; i++) {
    // Step 1: get 1 wei of rsETH via dust deposit (minAmountToDeposit == 0)
    lrtDepositPool.depositETH{value: 1}(0, "");

    // Step 2: queue a 1-wei withdrawal (minRsEthAmountToWithdraw == 0)
    rsETH.approve(address(lrtWithdrawalManager), 1);
    lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");
}

// Result: nextUnusedNonce[ETH_TOKEN] == 50_000
// unlockQueue now iterates 50_000 entries → OOG → all legitimate withdrawals frozen
```

The `AssetWithdrawalQueued` event is emitted for every iteration: [11](#0-10) 

and `_unlockWithdrawalRequests` must traverse every one of them before reaching any legitimate request, exhausting the block gas limit.
