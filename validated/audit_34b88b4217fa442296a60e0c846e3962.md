### Title
Stale-Attestation Authorization Bypass in `is_caller_an_attested_participant` — (`crates/contract/src/tee/tee_state.rs`)

### Summary

`is_caller_an_attested_participant` performs only an existence + identity check on the stored attestation. It never calls `re_verify`, so a node whose image hash has been rotated out of the whitelist (or whose attestation has expired) continues to pass the guard indefinitely — until `clean_invalid_attestations` is explicitly invoked. Any function gated solely by `assert_caller_is_attested_participant_and_protocol_active` (including `respond_verify_foreign_tx`) is reachable by a node running a revoked TEE image.

### Finding Description

`is_caller_an_attested_participant` (lines 469–498) performs three checks:

1. The caller's account ID is in the active `Participants` set.
2. A `NodeAttestation` keyed by the caller's TLS public key exists in `stored_attestations`.
3. The stored `account_id` and `account_public_key` match the transaction signer. [1](#0-0) 

It does **not** call `re_verify`. By contrast, `reverify_participants` (lines 206–232) calls `VerifiedAttestation::re_verify`, which checks:

- The stored `mpc_image_hash` against the current `allowed_docker_image_hashes` whitelist.
- The stored `launcher_compose_hash` against the current launcher whitelist.
- The `expiry_timestamp_seconds` against the current block timestamp.
- The stored `measurements` against the current allowed measurements. [2](#0-1) 

For a real `Dstack` attestation, `re_verify` enforces all four constraints: [3](#0-2) 

The cleanup path `clean_invalid_attestations` removes stale entries, but it is not called automatically on every transaction — it must be explicitly invoked with a `max_scan` budget: [4](#0-3) 

The gap: after `image_hash_A`'s grace period expires, `reverify_participants` returns `TeeQuoteStatus::Invalid` for a node whose stored `ValidatedDstackAttestation.mpc_image_hash == hash_A`, but `is_caller_an_attested_participant` returns `Ok(())` for the same node, because it never consults the whitelist.

### Impact Explanation

Any contract entry point gated by `assert_caller_is_attested_participant_and_protocol_active` — including `respond_verify_foreign_tx` — is reachable by a node running a revoked TEE image. That node can submit foreign-chain verification responses that the contract accepts as coming from a legitimately attested participant, enabling forged foreign-chain verification and potentially invalid bridge execution. This matches the **High** scope: *participant/attestation authorization bypass that causes invalid bridge execution*.

### Likelihood Explanation

The window is bounded by:
- The grace period duration (configurable, can be hours to days).
- The 24-hour Dstack attestation expiry (`DEFAULT_EXPIRATION_DURATION_SECONDS`).
- The cadence at which `clean_invalid_attestations` is called. [5](#0-4) 

A node operator who deliberately continues running deprecated software after the grace period, and whose attestation has not yet been cleaned up, can exploit this window. The node must still be in the active `Participants` set (no resharing has removed them), which is a realistic condition immediately after an image rotation before a resharing completes.

### Recommendation

`is_caller_an_attested_participant` should call `reverify_participants` (or inline an equivalent `re_verify` call) before returning `Ok(())`. The check should fail if the stored attestation no longer passes re-verification against the current whitelist and timestamp. Alternatively, `clean_invalid_attestations` should be called atomically as part of any state-mutating entry point that relies on attestation authorization, rather than as a separate, manually-triggered maintenance operation.

### Proof of Concept

A deterministic unit test demonstrating the divergence:

1. Insert a `MockAttestation::WithConstraints { mpc_docker_image_hash: Some(hash_A), ... }` for a participant node.
2. Call `whitelist_tee_proposal(hash_B, grace_duration)` so `hash_A` enters its grace period.
3. Advance `block_timestamp` past the grace period so `hash_A` is no longer returned by `get_allowed_mpc_docker_image_hashes`.
4. Set the VM signer to the participant's account/key and call `is_caller_an_attested_participant` — assert it returns `Ok(())`.
5. Call `reverify_participants` for the same node — assert it returns `TeeQuoteStatus::Invalid`.

Steps 4 and 5 operate on identical contract state and demonstrate that the two authorization paths disagree: the guard used by `respond_verify_foreign_tx` accepts the caller while the re-verification path rejects them. [1](#0-0) [2](#0-1) 

**Note:** I was unable to read the full body of `assert_caller_is_attested_participant_and_protocol_active` and `respond_verify_foreign_tx` in `crates/contract/src/lib.rs` due to tool iteration limits. If `assert_caller_is_attested_participant_and_protocol_active` internally calls `reverify_participants` before delegating to `is_caller_an_attested_participant`, the gap is closed and this finding does not apply. The analysis above is valid only if the wrapper does not add a `re_verify` step, which is consistent with the naming and the structure of `is_caller_an_attested_participant` as the sole attestation check.

    pub(crate) fn reverify_participants(
        &self,
        node_id: &NodeId,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeQuoteStatus {
        let allowed_mpc_docker_image_hashes =
            self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
        let allowed_launcher_compose_hashes = self.get_allowed_launcher_compose_hashes();
        let allowed_measurements = self.get_accepted_measurements();

        let participant_attestation = self.stored_attestations.get(&node_id.tls_public_key);
        let Some(participant_attestation) = participant_attestation else {
            return TeeQuoteStatus::Invalid("participant has no attestation".to_string());
        };

        // Verify the attestation quote
        let time_stamp_seconds = Self::current_time_seconds();
        match participant_attestation.verified_attestation.re_verify(
            time_stamp_seconds,
            &allowed_mpc_docker_image_hashes,
            &allowed_launcher_compose_hashes,
            &allowed_measurements,
        ) {
            Ok(()) => TeeQuoteStatus::Valid,
            Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
        }
    }

    pub fn clean_invalid_attestations(
        &mut self,
        tee_upgrade_deadline_duration: Duration,
        max_scan: usize,
    ) -> u32 {
        let has_invalid_attestation = |node_id: &NodeId| {
            !matches!(
                self.reverify_participants(node_id, tee_upgrade_deadline_duration),
                TeeQuoteStatus::Valid
            )
        };

        // Materialize candidates before any mutation to avoid iterator invalidation.
        let invalid_tls_keys: Vec<Ed25519PublicKey> = self
            .stored_attestations
            .iter()
            .take(max_scan)
            .filter(|(_, node_attestation)| has_invalid_attestation(&node_attestation.node_id))
            .map(|(tls_pk, _)| tls_pk.clone())
            .collect();

        let removed = u32::try_from(invalid_tls_keys.len())
            .expect("u32 should always be convertible from usize on wasm32");

        for tls_pk in invalid_tls_keys {
            self.stored_attestations.remove(&tls_pk);
        }
        removed
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
