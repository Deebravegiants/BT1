Audit Report

## Title
rsETHPrice Reset to 1e18 on Zero Supply Ignores Residual sfrxETH Holdings, Enabling First-Minter Yield Theft — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle._updateRsETHPrice` unconditionally resets `rsETHPrice` to `1e18` when `rsethSupply == 0` without inspecting actual protocol holdings. Because `LRTWithdrawalManager._calculatePayoutAmount` pays the minimum of `expectedAssetAmount` and `currentReturn`, sfrxETH yield that accrues between withdrawal initiation and unlock is structurally left as a residual in `LRTUnstakingVault`. After a complete wind-down, this residual is still counted by `getTotalAssetDeposits` but the zero-supply branch ignores it, allowing the first subsequent depositor to mint rsETH at `1e18` and immediately claim all residual yield.

## Finding Description

**Root cause — `LRTOracle._updateRsETHPrice` (lines 218–222):**

```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```

The branch returns without calling `_getTotalEthInProtocol()`, so any sfrxETH still in the vault is silently ignored. [1](#0-0) 

**How residual sfrxETH accumulates:**

`initiateWithdrawal` locks `expectedAssetAmount` using the live `rsETHPrice` and live `assetPrice` (sfrxETH `pricePerShare`) at request time: [2](#0-1) 

`_calculatePayoutAmount` in `unlockQueue` pays `min(expectedAssetAmount, currentReturn)`:

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
``` [3](#0-2) 

Because `SfrxETHPriceOracle.getAssetPrice` always returns the live `pricePerShare`, if sfrxETH yield accrues between initiation and unlock, `assetPrice_unlock > assetPrice_initiation`, making `currentReturn < expectedAssetAmount`. The vault redeems only `currentReturn`, but `assetsCommitted` is reduced by the original `expectedAssetAmount`: [4](#0-3) 

The difference `expectedAssetAmount − currentReturn` remains in `LRTUnstakingVault` and is not tracked by `assetsCommitted`. `getTotalAssetDeposits` still counts it via:

```solidity
assetLyingUnstakingVault = IERC20(asset).balanceOf(lrtUnstakingVault);
``` [5](#0-4) 

After all rsETH is burned through `unlockQueue`, `rsethSupply == 0` while the vault holds residual sfrxETH.

**Exploit flow:**

1. Call `updateRSETHPrice()` (public, no access control) → zero-supply branch fires, `rsETHPrice = 1e18`.
2. Deposit `D` sfrxETH via `depositAsset`. `getRsETHAmountToMint` computes:
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```
   With `rsETHPrice = 1e18` and `pricePerShare = P`, attacker receives `D × P / 1e18` rsETH. [6](#0-5) 
3. Call `updateRSETHPrice()` again → `_getTotalEthInProtocol()` now includes `(D + R) × P` (attacker's deposit plus residual `R`), divided by the rsETH minted in step 2. `rsETHPrice` rises to reflect all holdings.
4. Attacker initiates withdrawal for all their rsETH, receiving `D + R` sfrxETH — extracting `R` sfrxETH they did not deposit.

The existing `sweepRemainingAssets` function only sweeps from `LRTWithdrawalManager`'s own balance, not from `LRTUnstakingVault`, so it does not mitigate this. [7](#0-6) 

## Impact Explanation

**High — Theft of unclaimed yield.** The residual sfrxETH in the vault represents yield that accrued to sfrxETH between withdrawal initiation and unlock. Withdrawing users received `currentReturn` (fewer sfrxETH tokens at the higher `pricePerShare`), leaving the yield difference in the vault. The first depositor after a zero-supply reset can claim this entire residual, constituting direct theft of unclaimed yield from the protocol.

## Likelihood Explanation

**Low.** The precondition requires `rsethSupply` to reach exactly zero while sfrxETH remains in `LRTUnstakingVault`. This requires a complete protocol wind-down. However, the residual accumulation mechanism is structural and automatic: sfrxETH yield accrues continuously, `rsETHPrice` is only updated on explicit calls, and the `min`-payout logic in `_calculatePayoutAmount` always produces a residual whenever `pricePerShare` rises between initiation and unlock. A complete wind-down (migration, emergency shutdown) is a realistic lifecycle event. Once the state is reached, the exploit is trivially executable by any unprivileged caller.

## Recommendation

Replace the unconditional early return with a check that also inspects actual protocol holdings:

```solidity
if (rsethSupply == 0) {
    uint256 totalETH = _getTotalEthInProtocol();
    if (totalETH == 0) {
        rsETHPrice = 1 ether;
        highestRsethPrice = 1 ether;
    }
    // If totalETH > 0 with zero supply, leave rsETHPrice unchanged
    // to block new mints until the residual is swept or redistributed.
    return;
}
```

Additionally, add a guard in `_beforeDeposit` / `depositETH` that reverts when `rsethSupply == 0` but `_getTotalEthInProtocol() > 0`, preventing any mint until governance explicitly resolves the residual. Consider also adding a sweep function on `LRTUnstakingVault` callable by the manager to redirect residual assets to the treasury.

## Proof of Concept

```
// Fork/unit test setup
// sfrxETH.pricePerShare = 1.05e18 at initiation time
// User initiates withdrawal of 100e18 rsETH for sfrxETH
//   expectedAssetAmount = 100e18 * rsETHPrice / 1.05e18
//   assetsCommitted[sfrxETH] += expectedAssetAmount
//
// sfrxETH.pricePerShare rises to 1.10e18 before unlockQueue is called
//
// unlockQueue is called:
//   currentReturn = 100e18 * rsETHPrice / 1.10e18  (< expectedAssetAmount)
//   payoutAmount = currentReturn
//   assetsCommitted[sfrxETH] -= expectedAssetAmount  (original)
//   vault.redeem(sfrxETH, currentReturn)
//   residual R = expectedAssetAmount - currentReturn remains in vault
//
// After all rsETH burned: rsETH.totalSupply() == 0, vault.balanceOf(sfrxETH) == R > 0

// Step 1: attacker calls LRTOracle.updateRSETHPrice()
//   → rsETHPrice = 1e18 (zero-supply branch)

// Step 2: attacker calls LRTDepositPool.depositAsset(sfrxETH, 1e18, 0, "")
//   → rsethAmountToMint = 1e18 * 1.10e18 / 1e18 = 1.1e18 rsETH minted
//   → vault now holds 1 sfrxETH (attacker) + R sfrxETH (residual)

// Step 3: attacker calls LRTOracle.updateRSETHPrice()
//   → totalETH = (1 + R) * 1.10e18
//   → rsETHPrice = (1 + R) * 1.10e18 / 1.1e18 = (1 + R) * 1e18

// Step 4: attacker initiates withdrawal for 1.1e18 rsETH
//   → expectedAssetAmount = 1.1e18 * (1+R)*1e18 / 1.10e18 = (1+R) sfrxETH

// Assert: attacker deposited 1 sfrxETH, withdraws (1 + R) sfrxETH
// R sfrxETH of residual yield stolen
```
