Audit Report

## Title
Unbounded Gas Loop in `_unlockWithdrawalRequests` Enables Temporary Freezing of Legitimate Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

## Summary
Any rsETH holder can call `initiateWithdrawal()` with the minimum non-zero amount (as low as 1 wei when `minRsEthAmountToWithdraw[asset]` is unset and defaults to 0) to flood the global FIFO withdrawal queue. The `_unlockWithdrawalRequests` while-loop iterates O(N) over all queued entries with no per-call iteration cap. With enough spam entries, the loop exhausts the block gas limit, and legitimate users behind the attacker's entries cannot call `completeWithdrawal()` until the operator drains all spam entries via repeated batched calls.

## Finding Description

`minRsEthAmountToWithdraw` is declared as a plain mapping with no default initialization in `initialize()`, so it starts at 0 for every asset. [1](#0-0) 

The only guard in `initiateWithdrawal()` is `rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]`. With `minRsEthAmountToWithdraw[asset] == 0`, any amount ≥ 1 passes. [2](#0-1) 

Each call pushes one entry into the global sequential queue and increments `nextUnusedNonce[asset]`. [3](#0-2) 

`_unlockWithdrawalRequests` iterates from `nextLockedNonce` to `firstExcludedIndex` (capped at `nextUnusedNonce[asset]`) with only two early-exit conditions: (1) withdrawal delay not yet elapsed, (2) available assets exhausted. [4](#0-3) 

Because each spam entry commits only 1 wei of `expectedAssetAmount`, condition (2) never fires early — `availableAssetAmount` is not meaningfully depleted by tiny requests. The loop must visit every attacker-inserted entry before reaching legitimate users' nonces, consuming ~3 SSTOREs + 1 SLOAD per iteration. At ~30 M block gas, roughly 500 iterations fit per call. Tens of thousands of spam entries make a single-call full-queue unlock impossible.

`completeWithdrawal()` checks `usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]` and reverts if the request has not been unlocked. Until `nextLockedNonce` advances past all spam entries, legitimate users cannot complete their withdrawals. [5](#0-4) 

## Impact Explanation

Two concrete allowed impacts are triggered:

- **Temporary freezing of funds (Medium)**: Legitimate users whose nonces are behind the attacker's entries cannot call `completeWithdrawal()` successfully. Their funds remain locked in the contract until the operator manually drains all spam entries via many batched `unlockQueue()` calls.
- **Unbounded gas consumption (Medium)**: If the operator passes a large `firstExcludedIndex` (e.g., `type(uint256).max` or `nextUnusedNonce[asset]`), the while-loop runs out of gas and reverts, blocking the unlock path entirely until the operator discovers the need for smaller batches.

## Likelihood Explanation

- `initiateWithdrawal()` is permissionless for any rsETH holder.
- `minRsEthAmountToWithdraw[asset]` defaults to 0; spam entries cost only gas fees.
- The attacker's rsETH is returned as the underlying asset after the delay, so the net financial cost is gas only.
- The attack is repeatable across multiple supported assets and can be sustained continuously to keep the queue growing faster than the operator can drain it.
- No special role, oracle access, or governance capture is required.

## Recommendation

1. **Enforce a meaningful minimum withdrawal amount** for every supported asset via `setMinRsEthAmountToWithdraw` at deployment/upgrade time, making spam economically prohibitive.
2. **Add a per-call iteration cap** inside `_unlockWithdrawalRequests` (e.g., `maxIterations` parameter) so the operator always processes in bounded batches and the function cannot be made to revert by queue size alone.
3. **Cap the number of pending requests per user** to prevent a single address from monopolizing queue slots.
4. **Document and enforce a safe upper bound for `firstExcludedIndex`** in `unlockQueue()` so operators always pass a bounded range.

## Proof of Concept

1. Attacker holds or acquires rsETH (e.g., via `LRTDepositPool.depositETH()`).
2. Attacker calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "")` 50,000 times. Each call succeeds because `minRsEthAmountToWithdraw[ETH_TOKEN] == 0` and 1 > 0. Each call increments `nextUnusedNonce[ETH_TOKEN]` and commits 1 wei of `assetsCommitted`.
3. Legitimate users submit their own withdrawal requests (nonces 50,000+).
4. After the 8-day delay (`withdrawalDelayBlocks = 8 days / 12 seconds`), the operator calls `unlockQueue(ETH_TOKEN, type(uint256).max, ...)`.
5. `firstExcludedIndex` is capped at `nextUnusedNonce[ETH_TOKEN]` (50,000+). The while-loop attempts to iterate all 50,000 attacker entries; each iteration reads/writes storage. The transaction runs out of gas and reverts.
6. `nextLockedNonce[ETH_TOKEN]` has not advanced. Legitimate users call `completeWithdrawal(ETH_TOKEN, "")` and receive `WithdrawalLocked` revert.
7. The operator must discover the need for batched calls and submit many transactions to drain the spam entries before legitimate users can withdraw.

Foundry fuzz/invariant test plan: fuzz `N` spam entries (1 ≤ N ≤ 100,000), assert that a single `unlockQueue` call with `firstExcludedIndex = nextUnusedNonce` reverts with out-of-gas when N exceeds ~500, and assert that `completeWithdrawal` for a legitimate user at nonce N reverts with `WithdrawalLocked` until all N spam entries are drained.
