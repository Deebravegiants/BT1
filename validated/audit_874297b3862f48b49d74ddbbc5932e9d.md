### Title
Zero-Confirmation Bitcoin Attestation via Unvalidated `BlockConfirmations` — (`crates/foreign-chain-inspector/src/bitcoin/inspector.rs` / `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary

Neither the NEAR contract nor the MPC node's Bitcoin inspector enforces a minimum of 1 confirmation. An unprivileged caller can submit a `BitcoinRpcRequest` with `confirmations=0`, and the node will sign a payload for a Bitcoin transaction that has zero confirmations (just mined into the tip block, not yet built upon), enabling premature bridge execution over a transaction that may be reorganized out.

### Finding Description

**`BlockConfirmations` is an unconstrained `u64` wrapper.** [1](#0-0) 

The value `0` is structurally valid and accepted by Borsh/JSON deserialization with no invariant enforcement.

**The contract's `verify_foreign_transaction` performs no confirmation-count validation.** [2](#0-1) 

The only checks are: domain purpose is `ForeignTx`, the chain is in `get_supported_foreign_chains()`, and gas/deposit requirements are met. There is no guard on `request.confirmations > 0`.

**The node's `BitcoinInspector::extract` uses a `<=` comparison with the caller-supplied threshold.** [3](#0-2) 

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
```

When `block_confirmations_threshold = BlockConfirmations(0)`, the condition `0 <= N` is always true for any `u64 N`, including `N = 0`. A transaction with 0 confirmations (mined into the tip block but not yet built upon) passes this check unconditionally.

**The `GetRawTransactionVerboseResponse` struct requires `blockhash` as a non-optional field.** [4](#0-3) 

This means a true mempool transaction (not yet in any block) would fail deserialization and be rejected. The exploitable case is a **0-confirmation mined transaction** — one that is in the current tip block but has not yet been built upon. Bitcoin's `getrawtransaction` returns `blockhash` and `confirmations: 0` for such transactions, which passes deserialization and the `0 <= 0` check.

After the confirmation check passes, `verify_block_is_canonical` is called and may succeed (the block is canonical at that instant), resulting in the node signing the payload. [5](#0-4) 

### Impact Explanation

A bridge contract receives a valid MPC attestation (`ForeignTxSignPayloadV1` signed by the MPC network) over a Bitcoin transaction with 0 confirmations. That transaction can be reorganized out of the chain (Bitcoin reorgs at depth 0 are common). The bridge contract, trusting the MPC attestation, releases funds or executes a cross-chain action before the Bitcoin transaction is irreversible. This enables double-spend conditions on the bridge.

### Likelihood Explanation

The attack requires only that Bitcoin is in `get_supported_foreign_chains` and that the attacker submits a `BitcoinRpcRequest` with `confirmations=0`. No privileged access, threshold collusion, or network-level attack is needed. The path is fully reachable by any unprivileged NEAR account.

### Recommendation

1. **Contract-level**: In `verify_foreign_transaction`, reject `BitcoinRpcRequest` with `confirmations == 0`:
   ```rust
   if let ForeignChainRpcRequest::Bitcoin(ref btc) = request.request {
       require!(btc.confirmations.0 >= 1, "Bitcoin requests require at least 1 confirmation");
   }
   ```
2. **Node-level (defense-in-depth)**: In `BitcoinInspector::extract`, add an explicit guard before the threshold comparison:
   ```rust
   if block_confirmations_threshold.0 == 0 {
       return Err(ForeignChainInspectionError::ZeroConfirmationsNotAllowed);
   }
   ```

### Proof of Concept

Unit test sketch:

```rust
// In BitcoinInspector tests:
let tx_response = GetRawTransactionVerboseResponse {
    blockhash: some_canonical_hash,
    confirmations: 0u64,  // 0-confirmation mined tx
};
let threshold = BlockConfirmations(0);  // attacker-supplied

// block_confirmations_threshold <= transaction_block_confirmation
// 0 <= 0 → true → passes, proceeds to sign
let result = inspector.extract(tx_id, threshold, vec![BitcoinExtractor::BlockHash]).await;
assert!(result.is_ok()); // succeeds — MPC signs a 0-confirmation tx
```

At the contract level, `verify_foreign_transaction` with `confirmations: 0.into()` is accepted without panic, as confirmed by the absence of any confirmation-count guard in `crates/contract/src/lib.rs`. [6](#0-5) [7](#0-6) [8](#0-7)
