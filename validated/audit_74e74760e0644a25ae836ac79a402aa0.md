### Title
Decimal Mismatch in `viewSwapRsETHAmountAndFee` for Non-18-Decimal Tokens Causes Severe Under-Minting of wrsETH - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolNoWrapper.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPool.sol)

---

### Summary

Every pool variant's token-deposit path computes the wrsETH amount to mint with the formula:

```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
```

`tokenToETHRate` is always normalised to 18 decimals by `ChainlinkOracleForRSETHPoolCollateral.getRate()`, but `amountAfterFee` is the raw token amount in the token's **native** decimal precision. For 18-decimal LSTs the formula is correct; for any token with fewer decimals (e.g. 6 for USDC/USDT) the result is `10^(18 − tokenDecimals)` times smaller than the correct value, causing the depositor to receive a negligible amount of wrsETH while their full token balance is transferred into the pool.

---

### Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` always returns a price normalised to 1e18:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L34
uint256 normalizedPrice = uint256(ethPrice) * 1e18
    / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

This value represents **"ETH per 1 whole token"** scaled to 1e18. For example, if 1 USDC = 0.0004 ETH, `getRate()` returns `4e14`.

The pool then computes:

```solidity
// contracts/pools/RSETHPoolV3.sol L334
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`amountAfterFee` is the raw ERC-20 transfer amount — for USDC it is in 6-decimal units. The formula implicitly assumes `amountAfterFee` is in 18-decimal units. The missing normalisation factor is `1e18 / 10^tokenDecimals`.

**Correct formula for a token with `d` decimals:**
```
rsETHAmount = amountAfterFee * tokenToETHRate * 1e18
              / (rsETHToETHrate * 10^d)
```

The same broken formula is copy-pasted across all pool variants:
- `RSETHPoolV3.sol` L334
- `RSETHPoolV3ExternalBridge.sol` L452
- `RSETHPoolNoWrapper.sol` L311
- `RSETHPoolV3WithNativeChainBridge.sol` L370
- `RSETHPool.sol` L346

---

### Impact Explanation

For a 6-decimal token the result is `1e12` times smaller than the correct value.

**Concrete example — 1 USDC deposit:**
| Variable | Value |
|---|---|
| `amountAfterFee` | `1e6` (1 USDC, 6 dec) |
| `tokenToETHRate` | `4e14` (0.0004 ETH/USDC, 18 dec) |
| `rsETHToETHrate` | `1.05e18` (1.05 ETH/rsETH, 18 dec) |
| **Actual result** | `1e6 × 4e14 / 1.05e18 ≈ 380 wei wrsETH` |
| **Correct result** | `4e14 / 1.05e18 × 1e18 ≈ 3.81e14 wei wrsETH` |

The user's USDC is transferred to the pool in full, but they receive `~380` wei of wrsETH instead of `~3.81e14` wei — a loss of `>99.9999%` of deposited value. The tokens are effectively frozen in the pool because the user cannot redeem them with the dust wrsETH they received.

Impact classification: **Medium — Temporary (effectively permanent) freezing of user funds / contract fails to deliver promised returns.**

---

### Likelihood Explanation

The pool's `supportedTokenOracle` mapping is set by an admin role and is not restricted to 18-decimal tokens by any on-chain check. The protocol already supports multiple token types across chains. Any future addition of a non-18-decimal collateral token (USDC, USDT, or similar) by an admin acting in good faith would silently activate this bug for every depositor of that token. The likelihood is **Low-to-Medium**: the current LST-only token set is 18-decimal, but the code is generic and the constraint is not enforced.

---

### Recommendation

Normalise `amountAfterFee` to 18 decimals before applying the rate formula, or divide the result by `10^tokenDecimals`:

```solidity
// Fetch token decimals once (or cache per token)
uint8 tokenDecimals = IERC20Metadata(token).decimals();

rsETHAmount = amountAfterFee
    * tokenToETHRate
    * 1e18
    / rsETHToETHrate
    / (10 ** tokenDecimals);
```

Alternatively, enforce at token-registration time that only 18-decimal tokens may be added.

---

### Proof of Concept

1. Admin adds USDC (6 decimals) as a supported token with a valid Chainlink USDC/ETH oracle.
2. User calls `deposit(usdcAddress, 1_000_000, "ref")` (1 USDC).
3. Pool transfers 1 USDC from user.
4. `viewSwapRsETHAmountAndFee(1_000_000, usdcAddress)` executes:
   - `fee = 1_000_000 * feeBps / 10_000` (small)
   - `amountAfterFee ≈ 1_000_000`
   - `tokenToETHRate = ChainlinkOracle.getRate()` → `~4e14`
   - `rsETHToETHrate = getRate()` → `~1.05e18`
   - `rsETHAmount = 1_000_000 * 4e14 / 1.05e18 ≈ 380`
5. Pool mints `380` wei wrsETH to user instead of `~3.81e14` wei.
6. User's 1 USDC is locked in the pool; the `380` wei wrsETH is insufficient to redeem any meaningful amount. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    function getRate() public view returns (uint256) {
        (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
            AggregatorV3Interface(oracle).latestRoundData();

        if (answeredInRound < roundID) revert StalePrice();
        if (timestamp == 0) revert IncompleteRound();
        if (ethPrice <= 0) revert InvalidPrice();

        uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());

        return normalizedPrice;
    }
