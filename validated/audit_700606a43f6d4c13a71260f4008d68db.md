Audit Report

## Title
Reward Sandwich Attack via Front-Running `notifyRewardAmount` with No Warmup Period — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` implements a Synthetix-style staking rewards model with no warmup period. An unprivileged attacker can front-run the admin's `notifyRewardAmount` call by staking immediately before it, causing their `userRewardPerTokenPaid` to be snapshotted at the pre-period `rewardPerTokenStored`. From the first second of the new reward period, the attacker earns rewards proportional to their diluted share of `totalKernelStaked`, then claims them via `getReward()` — stealing yield from legitimate long-term stakers.

## Finding Description
The `updateReward` modifier (L232–242) snapshots `userRewardPerTokenPaid[_account] = rewardPerTokenStored` on every user action, including `stake()`. When an attacker calls `stake()` immediately before `notifyRewardAmount` confirms, their paid checkpoint is set to the current (pre-new-period) `rewardPerTokenStored`.

`notifyRewardAmount` (L566–591) then sets `rewardRate`, `finishAt = block.timestamp + duration`, and `updatedAt = block.timestamp`. Because the attacker's `userRewardPerTokenPaid` equals `rewardPerTokenStored` at the start of the new period, `earned(attacker)` (L421–423) accumulates from the very first second of the new period, proportional to `attackerStake / totalKernelStaked`.

The attacker then calls `getReward()` (L382–390) to claim the accumulated yield at any time — no withdrawal is required to steal rewards.

Regarding the `withdrawalDelay == 0` amplifier: `initialize()` (L259–271) never sets `withdrawalDelay`, so it defaults to `0`. `initiateWithdrawal` computes `unlockTime = block.timestamp + 0` (L330), making `claimWithdrawal`'s guard `block.timestamp < withdrawal.unlockTime` (L355) always false. However, `initialize()` also never sets `maxNumberOfWithdrawalsPerUser`, which defaults to `0`. The check at L323 — `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser` — evaluates `0 >= 0 = true` and reverts with `WithdrawalLimitReached`. This means the attacker **cannot** exit principal until the admin calls `setMaxNumberOfWithdrawalsPerUser`. The PoC's claim of same-block exit is therefore incorrect. Nevertheless, the yield theft via `getReward()` is fully independent of withdrawal and remains exploitable regardless.

The `notifyRewardAmount` guard `if (totalKernelStaked == 0) revert NoStakedTokens()` (L570) does not prevent the attack — it only requires that some stake exists before the call, which the attacker's front-run satisfies.

## Impact Explanation
**High — Theft of unclaimed yield.** Legitimate stakers who have been staked for the entire reward period receive a diluted share of rewards. The attacker earns `attackerStake / (attackerStake + legitimateStake)` of all rewards accrued from the start of the new period, claimable at any time via `getReward()`. The attacker's principal is temporarily locked (until `maxNumberOfWithdrawalsPerUser` is set), but the yield theft itself requires no principal recovery and is immediately executable.

## Likelihood Explanation
**Medium.** The attacker must monitor the mempool for a pending `notifyRewardAmount` transaction (trivially observable) and hold sufficient KERNEL to make dilution profitable. No privileged access is required. `stake()` is fully permissionless. The attack is repeatable on every new reward period.

## Recommendation
1. **Implement a warmup/cooldown period**: newly staked tokens should not accrue rewards until at least one full reward epoch has elapsed since staking, or snapshot eligible stakers at the time `notifyRewardAmount` is called.
2. **Initialize `withdrawalDelay` to a non-zero value inside `initialize()`** (e.g., 7 days).
3. **Initialize `maxNumberOfWithdrawalsPerUser` to a non-zero value inside `initialize()`** to prevent the contract from being deployed in a state where withdrawals are permanently blocked.

## Proof of Concept
**Setup**: Alice has staked `100e18` KERNEL. Admin has set `duration`. Admin is about to call `notifyRewardAmount(1_000_000e18)`. `withdrawalDelay == 0`, `maxNumberOfWithdrawalsPerUser == 0` (never initialized).

**Attack**:
```
Block N:
  Attacker observes pending notifyRewardAmount(1_000_000e18) in mempool.
  Attacker calls stake(9_900e18).
  → totalKernelStaked = 10_000e18 (attacker: 99%, Alice: 1%)
  → updateReward snapshots userRewardPerTokenPaid[attacker] = rewardPerTokenStored (current)

Block N (same block, higher gas):
  notifyRewardAmount(1_000_000e18) confirms.
  → rewardRate = 1_000_000e18 / duration
  → finishAt = block.timestamp + duration, updatedAt = block.timestamp

Block N+K (some blocks later):
  Attacker calls getReward()
  → earned(attacker) ≈ 0.99 × rewardRate × elapsed_time
  → Attacker receives ~99% of all rewards accrued so far.
  (No initiateWithdrawal needed to steal yield)
```

**Foundry test plan**:
1. Deploy `KernelDepositPool`, initialize with admin/kernelToken/rewardToken.
2. Call `setRewardsDuration(30 days)` as admin.
3. Have Alice stake `100e18`.
4. Attacker stakes `9_900e18` (simulating front-run).
5. Admin calls `notifyRewardAmount(1_000_000e18)`.
6. `vm.warp(block.timestamp + 1 days)`.
7. Attacker calls `getReward()` — assert attacker receives ~99% of 1-day rewards.
8. Alice calls `getReward()` — assert Alice receives ~1% of 1-day rewards.
