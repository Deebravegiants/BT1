### Title
Stale Cross-Chain Oracle Rate Enables Over-Minting of wrsETH at Depositors' Expense - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

### Summary
`CrossChainRateReceiver` stores a `lastUpdated` timestamp but never enforces a staleness check in `getRate()`. Because rsETH is an appreciating asset whose ETH-denominated rate grows over time, a stale (lower-than-current) rate causes every pool variant to mint more wrsETH per deposited ETH than the depositor is entitled to, diluting existing wrsETH holders.

### Finding Description
All L2 pool contracts (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolNoWrapper`) compute the wrsETH mint amount using the formula:

```
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate
``` [1](#0-0) 

The `rsETHToETHrate` is fetched from the oracle via `getRate()`: [2](#0-1) 

When the oracle is `RSETHRateReceiver` (which extends `CrossChainRateReceiver`), `getRate()` simply returns the stored `rate` field with no staleness enforcement: [3](#0-2) 

The contract does record `lastUpdated` on every LayerZero message receipt: [4](#0-3) 

But `lastUpdated` is never read inside `getRate()`. There is no `require(block.timestamp - lastUpdated <= MAX_STALENESS)` guard anywhere in the call path. The same structural gap exists in `InterimRSETHOracle`, where a MANAGER sets the rate manually with no expiry: [5](#0-4) 

rsETH accrues EigenLayer restaking rewards continuously, so its ETH-denominated rate (`rsETHToETHrate`) increases monotonically over time. Any period during which the oracle is not refreshed (LayerZero congestion, rate-provider downtime, or manual update lag) leaves the stored rate below the true current rate. Because the mint amount is inversely proportional to `rsETHToETHrate`, a stale low rate inflates the wrsETH minted per unit of ETH deposited.

### Impact Explanation
**Medium — Theft of unclaimed yield.**

Existing wrsETH holders' proportional claim on the underlying rsETH pool is diluted. The attacker deposits ETH while the oracle is stale, receives excess wrsETH, and after the oracle updates to the correct (higher) rate, redeems or holds wrsETH that is worth more ETH than was deposited. The profit comes directly from the yield that had accrued to prior holders. The daily mint limit (`dailyMintLimit`) caps per-day exposure but does not prevent the attack across multiple days or at the limit boundary. [6](#0-5) 

### Likelihood Explanation
**Medium.** Cross-chain LayerZero message delivery is subject to real-world delays (congestion, relayer downtime). rsETH's rate grows continuously, so even a few hours of staleness creates a measurable gap. No attacker capability beyond a standard `deposit()` call is required; the attacker only needs to observe that `lastUpdated` is old and act before the oracle is refreshed. The `InterimRSETHOracle` path is additionally susceptible whenever the MANAGER delays a manual update.

### Recommendation
Add a configurable maximum staleness period to `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public constant MAX_STALENESS = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate is stale");
    return rate;
}
```

Apply the same guard to `InterimRSETHOracle.getRate()`. This causes deposits to revert when the oracle is stale rather than silently minting at an incorrect rate, giving operators time to refresh the rate before users can exploit the gap.

### Proof of Concept

**Setup:**
- True rsETH/ETH rate on L1: `1.10e18`
- Stale rate stored in `CrossChainRateReceiver`: `1.05e18` (last updated 30 hours ago)
- LayerZero relayer is congested; no update has arrived on L2

**Attack steps:**

1. Attacker observes `CrossChainRateReceiver.lastUpdated` is 30 hours old and `rate = 1.05e18`.
2. Attacker calls `RSETHPoolV3.deposit{value: 10 ether}("")`.
3. Pool computes: `rsETHAmount = 10e18 * 1e18 / 1.05e18 ≈ 9.524e18` wrsETH minted.
4. Fair amount at true rate: `10e18 * 1e18 / 1.10e18 ≈ 9.091e18` wrsETH.
5. Attacker receives `≈ 0.433e18` excess wrsETH (≈ 4.3% over-issuance on 10 ETH).
6. LayerZero delivers the updated rate (`1.10e18`); oracle refreshes.
7. Attacker's `9.524` wrsETH now represents `9.524 * 1.10 ≈ 10.476 ETH` worth of rsETH.
8. Net profit: `≈ 0.476 ETH` extracted from existing wrsETH holders' accrued yield.

The attack is repeatable each day up to `dailyMintLimit`, and across all deployed pool variants sharing the same stale oracle. [7](#0-6) [8](#0-7) [9](#0-8)
