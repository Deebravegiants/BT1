Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice` Accepts Stale Chainlink Prices, Enabling Over-Minting of rsETH - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but silently discards the `updatedAt` timestamp and `answeredInRound` fields, accepting any price regardless of age or round completeness. This stale price flows directly into `LRTDepositPool.getRsETHAmountToMint`, allowing a depositor to receive more rsETH than the actual ETH-equivalent value of their deposit when a Chainlink LST/ETH feed is stale at an inflated price, diluting all existing rsETH holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice` (L49–55) destructures only the `price` field from `latestRoundData()`, discarding `updatedAt` and `answeredInRound`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
```

The five return values are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. No check is made that `updatedAt != 0` (incomplete round), `answeredInRound >= roundId` (stale round), or that `block.timestamp - updatedAt` is within an acceptable staleness window.

The protocol's own `ChainlinkOracleForRSETHPoolCollateral` (L27–32) demonstrates the correct pattern with all three guards (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`), confirming the protocol is aware of the requirement.

The stale price propagates through `LRTOracle.getAssetPrice` → `LRTDepositPool.getRsETHAmountToMint` (L519–521):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`getRsETHAmountToMint` is called by `_beforeDeposit` (L665), which is invoked by both `depositETH` (L87) and `depositAsset` (L111) — the primary public deposit entry points. No privileged role is required to trigger this path.

## Impact Explanation
When a Chainlink LST/ETH feed is stale and its last reported price is higher than the current market price (e.g., stETH depegs but the oracle has not yet updated within its heartbeat window), the numerator `amount * lrtOracle.getAssetPrice(asset)` is inflated. The attacker receives more rsETH than the ETH-equivalent value of their deposit. Because rsETH price is computed as `totalETHInProtocol / totalRsETHSupply`, this over-minting reduces the rsETH/ETH exchange rate for all existing holders, constituting **theft of unclaimed yield** (High severity).

## Likelihood Explanation
Chainlink LST/ETH feeds (e.g., stETH/ETH) have 24-hour heartbeat intervals and 0.5% deviation thresholds. During network congestion, keeper failures, or rapid LST depeg events that fall within the deviation threshold, the feed can remain stale for minutes to hours while the market price diverges. An attacker needs only to monitor on-chain `updatedAt` values, identify a stale window with an inflated price, and call the permissionless `depositAsset`. No privileged access, victim interaction, or governance action is required. The attack is repeatable for as long as the feed remains stale.

## Recommendation
Add staleness and round-completeness checks to `ChainlinkPriceOracle.getAssetPrice`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept
1. Chainlink stETH/ETH feed last reported `1.0 ETH` per stETH at time `T`. Heartbeat is 24 hours; no update has occurred since.
2. At time `T + 12h`, stETH depegs to `0.97 ETH` on the market, but the Chainlink feed still returns `1.0 ETH` (stale, within heartbeat, deviation < 0.5% threshold so no update triggered).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice` returns `1.0e18` (stale).
5. `rsethAmountToMint = (100e18 * 1.0e18) / rsETHPrice` — attacker receives ~3% more rsETH than the actual ETH value of their deposit warrants.
6. All existing rsETH holders are diluted by this over-minting, losing a proportional share of their accrued yield.

**Foundry fork test plan:** Fork mainnet, mock the stETH/ETH Chainlink aggregator to return a stale `updatedAt` (e.g., `block.timestamp - 25 hours`) with price `1.0e18`, call `depositAsset` with 100 stETH, assert that `rsethAmountToMint` exceeds the fair value computed using the current market price, and assert that `lrtOracle.rsETHPrice()` decreases post-deposit relative to a baseline with no deposit.
