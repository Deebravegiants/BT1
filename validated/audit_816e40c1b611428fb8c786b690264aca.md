Audit Report

## Title
Third-party `addLiquidity` on behalf of victim creates dust remainder that permanently blocks `removeLiquidity` — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`MetricOmmPool.addLiquidity` allows any `msg.sender` to credit shares to an arbitrary `owner` address (the operator pattern). `LiquidityLib.removeLiquidity` enforces a `minimalMintableLiquidity` dust floor on the *remaining* shares after a burn. An attacker can front-run a victim's full-withdrawal transaction by adding exactly 1 share to the victim's position, causing the victim's transaction to leave a 1-share remainder below the dust floor, reverting indefinitely. The victim's principal is permanently locked as long as the attacker repeats the front-run.

## Finding Description

`MetricOmmPool.addLiquidity` accepts an `owner` parameter that is fully independent of `msg.sender`: [1](#0-0) 

The interface NatSpec explicitly documents this as the "operator pattern" — `msg.sender` pays but need not equal `owner`: [2](#0-1) 

The callback is invoked on `msg.sender`, so the attacker pays the token cost, but shares are credited to the victim's `(owner, salt, binIdx)` key: [3](#0-2) 

The add-side dust check computes `newUserShares = userShares + sharesToAdd` and only reverts if the *total* is below the floor. If the victim already holds `X >= MINIMAL_MINTABLE_LIQUIDITY` shares, adding 1 share yields `X + 1 >= MINIMAL_MINTABLE_LIQUIDITY`, which passes: [4](#0-3) 

`removeLiquidity` enforces `msg.sender == owner`, so only the victim can call it: [5](#0-4) 

`removeLiquidity` enforces that any non-zero remainder must be at or above `minimalMintableLiquidity`: [6](#0-5) 

**Attack sequence:**
1. Victim holds `X` shares in bin `B` and submits `removeLiquidity(victim, salt, {X shares}, "")`.
2. Attacker front-runs with `addLiquidity(victim, salt, {1 share in bin B}, callbackData, "")`. Victim now holds `X + 1` shares.
3. Victim's transaction executes: `newUserShares = (X+1) - X = 1`. The check `1 > 0 && 1 < MINIMAL_MINTABLE_LIQUIDITY` is true → reverts with `MinimalLiquidity(1, 1000)`.
4. Attacker repeats on every retry.

The position key `keccak256(abi.encode(owner, salt, bin))` is fully reconstructible from on-chain `LiquidityAdded` events, so the attacker can always target the correct position: [7](#0-6) 

The `DepositAllowlistExtension` would block this only if the pool has it configured and the attacker is not on the allowlist. Pools without this optional extension are fully vulnerable: [8](#0-7) 

The periphery `addLiquidityExactShares` overload explicitly supports adding on behalf of another owner, confirmed by a passing test: [9](#0-8) 

## Impact Explanation

An EOA victim cannot withdraw their liquidity as long as the attacker keeps front-running. The victim's principal is locked in the pool indefinitely. This constitutes broken core pool functionality causing loss of funds and an unusable withdraw/liquidity flow — a direct match to the allowed impact gate. The victim's only escape would be a contract-based atomic read-and-remove, which the protocol's periphery does not provide.

## Likelihood Explanation

The attack requires MEV front-running capability (available on most EVM chains) and knowledge of the victim's `(owner, salt, binIdx)` tuple, which is fully public from on-chain `LiquidityAdded` events. The per-attack token cost is the value of 1 share (minimal), making sustained griefing economically viable. Any pool without a `DepositAllowlistExtension` configured is unconditionally vulnerable to any unprivileged caller.

## Recommendation

1. **Restrict third-party adds**: In `addLiquidity`, require `msg.sender == owner` unless the owner has explicitly granted operator approval via an on-chain allowance mapping or signed permit.
2. **Alternatively**, add a `removeAllLiquidity` variant that burns the caller's entire share balance in a bin without checking the dust floor on the remainder (since the remainder is 0).
3. **Alternatively**, remove the `MinimalLiquidity` check from `removeLiquidity` entirely (apply it only on `addLiquidity`), so users can always exit to zero regardless of intermediate share counts.

## Proof of Concept

```solidity
// Attacker contract
contract Attacker is IMetricOmmModifyLiquidityCallback {
    function grief(address pool, address victim, uint80 salt, int256 bin) external {
        int256[] memory bins = new int256[](1);
        bins[0] = bin;
        uint256[] memory shares = new uint256[](1);
        shares[0] = 1; // passes add-side check because victim already has >= 1000
        LiquidityDelta memory d = LiquidityDelta({binIdxs: bins, shares: shares});
        IMetricOmmPool(pool).addLiquidity(victim, salt, d, "", "");
    }
    function metricOmmModifyLiquidityCallback(uint256 a0, uint256 a1, bytes calldata) external {
        if (a0 > 0) IERC20(token0).transfer(msg.sender, a0);
        if (a1 > 0) IERC20(token1).transfer(msg.sender, a1);
    }
}

// Scenario:
// 1. victim has 10_000 shares in bin 4
// 2. victim submits removeLiquidity(victim, salt, {10_000 shares}, "")
// 3. attacker.grief(pool, victim, salt, 4)  ← front-runs; victim now has 10_001 shares
// 4. victim's tx: newUserShares = 10_001 - 10_000 = 1 < 1000 → MinimalLiquidity revert
// 5. repeat indefinitely
```
