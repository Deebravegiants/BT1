### Title
Aave WETH Pool at 100% Utilization Temporarily Bricks All ETH Withdrawal Paths With No Admin Bypass - (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

When Aave integration is enabled and the WETH reserve reaches 100% utilization, `completeWithdrawal`, `emergencyWithdrawFromAave`, and `setAaveIntegrationEnabled(false)` all revert. Because `setAaveIntegrationEnabled(false)` atomically requires a successful Aave withdrawal before it can flip the flag, there is no admin path to disable the integration while Aave is illiquid. All pending ETH withdrawals are frozen until external borrowers repay enough to restore liquidity.

---

### Finding Description

`_processWithdrawalCompletion` calls `_withdrawFromAave` with no try/catch when the contract's ETH balance is insufficient: [1](#0-0) 

`_withdrawFromAave` calls `aaveWETHGateway.withdrawETH` with no error handling: [2](#0-1) 

This is in direct contrast to the deposit path in `unlockQueue`, which wraps `depositToAaveExternal` in a try/catch and silently continues on failure: [3](#0-2) 

The developers clearly anticipated Aave deposit failures but did not apply the same defensive pattern to withdrawals.

`setAaveIntegrationEnabled(false)` forces a full Aave withdrawal before setting the flag. If `_collectInterestToTreasury` or `_withdrawFromAave` reverts (both call `withdrawETH`), the flag is never flipped: [4](#0-3) 

`emergencyWithdrawFromAave` has the same structure — it calls `_collectInterestToTreasury` then `_withdrawFromAave`, both of which revert under 100% utilization: [5](#0-4) 

`_collectInterestToTreasury` itself calls `withdrawETH` and will revert if there is any accrued interest and the pool is illiquid: [6](#0-5) 

---

### Impact Explanation

**Medium. Temporary freezing of funds.**

When Aave WETH utilization is 100%:
- `completeWithdrawal` reverts for all ETH requests where the contract's idle balance is insufficient.
- `emergencyWithdrawFromAave` reverts.
- `setAaveIntegrationEnabled(false)` reverts — there is no admin path to bypass the withdrawal and force-flip the flag.

The freeze is **temporary, not permanent**, for two reasons:
1. Aave's interest rate model spikes borrow rates sharply at high utilization, creating strong economic incentive for borrowers to repay, making sustained 100% utilization self-correcting.
2. The contract is deployed behind an upgradeable proxy, so an admin could deploy a patched implementation that separates the flag-flip from the withdrawal.

The "permanent freezing" and "protocol insolvency" framing in the question is overstated. The correct scoped impact is **Medium — temporary freezing of funds**.

---

### Likelihood Explanation

Low-to-medium. Aave WETH at 100% utilization is rare under normal conditions but has occurred historically during acute market stress (e.g., rapid ETH price movements, large coordinated borrows). The window is typically short due to rate mechanics, but it is long enough to block time-sensitive user withdrawals.

---

### Recommendation

1. **Decouple the flag from the withdrawal in `setAaveIntegrationEnabled(false)`**: Allow the admin to set `isAaveIntegrationEnabled = false` immediately, and handle the Aave withdrawal separately (e.g., via a follow-up `emergencyWithdrawFromAave` call once liquidity is restored).

2. **Add a try/catch in `_processWithdrawalCompletion`**: Mirror the pattern used in `unlockQueue`. If `_withdrawFromAave` fails, fall back to serving the withdrawal from idle contract balance if sufficient, or revert with a more informative error.

3. **Add a liquidity pre-check**: Before calling `_withdrawFromAave`, check available liquidity (e.g., `IERC20(WETH_ADDRESS).balanceOf(address(aaveAWETH))`) and revert early with a clear error rather than letting the Aave call revert opaquely.

---

### Proof of Concept

```solidity
// Fork test on Aave V3 mainnet
// 1. Deploy/configure LRTWithdrawalManager with Aave integration enabled
// 2. Simulate 100% WETH utilization: borrow all available WETH from Aave pool
// 3. Call completeWithdrawal(ETH_TOKEN, ...) → reverts with InsufficientLiquidityForWithdrawal
// 4. Call emergencyWithdrawFromAave(type(uint256).max) → reverts
// 5. Call setAaveIntegrationEnabled(false) → reverts
// 6. Assert: no ETH recovery path is available until external borrowers repay
```

The key assertion is step 5: `setAaveIntegrationEnabled(false)` reverts because it unconditionally calls `_withdrawFromAave` before setting the flag, with no bypass for the illiquid case. [7](#0-6)

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }

    function _collectInterestToTreasury() internal returns (uint256 interestAmount) {
        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        uint256 principal = totalETHDepositedToAave;

        // Return 0 if no interest or balance is less than principal (accounting for rounding)
        if (aaveBalance <= principal) return 0;

        interestAmount = aaveBalance - principal;

        aaveWETHGateway.withdrawETH(aavePool, interestAmount, address(this));

        address treasury = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        (bool sent,) = payable(treasury).call{ value: interestAmount }("");
        if (!sent) revert TreasuryTransferFailed();

        emit InterestCollectedToTreasury(interestAmount, treasury);
    }
