### Title
Cross-Chain Replay of `deployToken` Signature Due to Missing Chain-ID Binding — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The Borsh-encoded payload that the NEAR MPC signer signs for `deployToken` / `deploy_token` contains no destination chain identifier. Because every deployed bridge instance shares the same `nearBridgeDerivedAddress` (derived from the same NEAR MPC key), a signature produced for one chain is cryptographically valid on every other chain. An unprivileged attacker can harvest a legitimate `deployToken` signature from NEAR events and replay it against any other chain's bridge contract to deploy the same bridge token there without NEAR-side authorization.

---

### Finding Description

**EVM — `OmniBridge.deployToken`**

The signed payload is assembled at: [1](#0-0) 

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

The payload is `[type_byte | token | name | symbol | decimals]`. **No `omniBridgeChainId` is present.**

Contrast this with `finTransfer`, which encodes `omniBridgeChainId` twice (once for the token address and once for the recipient address): [2](#0-1) 

The chain-ID bytes at lines 294 and 297 are what make `finTransfer` signatures chain-specific. `deployToken` has no equivalent binding.

**Starknet — `OmniBridge.deploy_token`**

`MetadataPayload.to_borsh()` produces the identical chain-ID-free layout: [3](#0-2) 

`fin_transfer` passes `chain_id` into `to_borsh(chain_id)` and encodes it at two positions: [4](#0-3) 

`deploy_token` calls `payload.to_borsh()` with no chain-ID argument: [5](#0-4) 

**Solana — `deploy_token`**

`DeployTokenPayload::serialize_for_near` writes `[IncomingMessageType::Metadata | token | name | symbol | decimals]` with no `SOLANA_OMNI_BRIDGE_CHAIN_ID` byte: [6](#0-5) 

Compare `FinalizeTransferPayload::serialize_for_near`, which writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` at positions 3 and 5: [7](#0-6) 

**Encoding identity across chains**

All three chains use the same Borsh wire format for the metadata payload: a `u8` type discriminant (`0x01`), then each string as a little-endian `u32` length prefix followed by UTF-8 bytes, then a `u8` decimals byte. Because the bytes are identical, a single NEAR MPC signature is valid on all three chains simultaneously.

---

### Impact Explanation

An attacker who replays a `deployToken` signature on an unintended chain:

1. Deploys a bridge token contract on that chain without NEAR-side authorization for that specific chain.
2. Sets `isBridgeToken[proxy] = true` and populates `nearToEthToken` / `ethToNearToken` mappings on the target chain.
3. Causes the NEAR bridge to observe a `DeployToken` event from a factory it may have registered, potentially registering the token for that chain in the NEAR-side token registry before the NEAR side has set up the correct decimals mapping (`token_decimals`).
4. If decimals are absent, subsequent `fin_transfer_callback` calls for that token on that chain panic with `BridgeError::TokenDecimalsNotFound`, permanently locking any in-flight transfer that targeted the prematurely-deployed token on that chain.

This fits the allowed impact: **acceptance of insufficiently-bound signatures that bypass execution gates** (the `deployToken` authorization gate is bypassed on the target chain), and can produce **irreversible fund lock** for transfers routed to the prematurely-deployed token before the NEAR side completes its own registration.

---

### Likelihood Explanation

- `deployToken` signatures are published as NEAR events (`OmniBridgeEvent::LogMetadataEvent`) and are fully public.
- The attacker needs only to call `deployToken` on any other chain's bridge contract with the harvested `(signatureData, metadata)` tuple.
- No privileged access, leaked key, or colluding party is required.
- All production bridge deployments share the same MPC-derived address, so the same signature is valid everywhere.

---

### Recommendation

Include the destination chain ID in the `deployToken` / `deploy_token` signed payload, mirroring the pattern already used in `finTransfer`. For EVM:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // add chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent change to `MetadataPayload.to_borsh()` in Starknet and `DeployTokenPayload::serialize_for_near` in Solana, and update the NEAR MPC signing logic in `log_metadata_callback` to include the target chain ID in `MetadataPayload`.

---

### Proof of Concept

1. NEAR side calls `log_metadata` for token `"usdc.near"` targeting Ethereum (chain ID `1`). The MPC signer produces signature `S` over `keccak256([0x01 | "usdc.near" | "USD Coin" | "USDC" | 0x06])`.
2. The `SignTransferEvent` (or equivalent log metadata event) is emitted on NEAR and is publicly observable.
3. Attacker submits `OmniBridge.deployToken(S, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` to the Arbitrum bridge contract (chain ID `2`).
4. `ECDSA.recover(keccak256([0x01|...]), S) == nearBridgeDerivedAddress` — passes, because the payload bytes are identical.
5. `isBridgeToken[nearToEthToken["usdc.near"]]` is `false` on Arbitrum — the `ERR_TOKEN_EXIST` guard does not fire.
6. A new `BridgeToken` proxy is deployed on Arbitrum; `isBridgeToken`, `ethToNearToken`, and `nearToEthToken` are populated.
7. The NEAR side observes the `DeployToken` event from the Arbitrum factory and registers the token, but `token_decimals` for the Arbitrum address is not yet set.
8. Any user who subsequently initiates a transfer of `usdc.near` to Arbitrum triggers `fin_transfer_callback`, which panics at `self.token_decimals.get(&token_address).near_expect(BridgeError::TokenDecimalsNotFound)`, permanently locking those funds. [8](#0-7) [3](#0-2) [6](#0-5)

        bytes calldata signatureData,
        BridgeTypes.MetadataPayload calldata metadata
    ) external payable whenNotPaused(PAUSED_DEPLOY_TOKEN) returns (address) {
        if (tokenImplementationAddress == address(0)) {
            revert TokenImplementationNotSet();
        }
        bytes memory borshEncoded = bytes.concat(
            bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
            Borsh.encodeString(metadata.token),
            Borsh.encodeString(metadata.name),
            Borsh.encodeString(metadata.symbol),
            bytes1(metadata.decimals)
        );
        bytes32 hashed = keccak256(borshEncoded);

        if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
            revert InvalidSignature();
        }

        require(
            !isBridgeToken[nearToEthToken[metadata.token]],
            "ERR_TOKEN_EXIST"
        );
        uint8 decimals = _normalizeDecimals(metadata.decimals);

                    metadata.name,
                    metadata.symbol,
                    decimals
                )
            )

            metadata.symbol,

            metadata.decimals

        ethToNearToken[address(bridgeTokenProxy)] = metadata.token;

    fn serialize_for_near(&self, params: Self::AdditionalParams) -> Result<Vec<u8>> {
        let mut writer = BufWriter::new(Vec::with_capacity(DEFAULT_SERIALIZER_CAPACITY));
        // 0. prefix
        IncomingMessageType::InitTransfer.serialize(&mut writer)?;
        // 1. destination_nonce
        self.destination_nonce.serialize(&mut writer)?;
        // 2. transfer_id
        writer.write_all(&[self.transfer_id.origin_chain])?;
        self.transfer_id.origin_nonce.serialize(&mut writer)?;
        // 3. token
        writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;
        params.0.serialize(&mut writer)?;
        // 4. amount
        self.amount.serialize(&mut writer)?;

        params.1.serialize(&mut writer)?;
        // 6. fee_recipient
        self.fee_recipient.serialize(&mut writer)?;
