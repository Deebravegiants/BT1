### Title
stETH Rebase Yield During Lido Withdrawal Period Is Invisible to `updateRSETHPrice`, Understating rsETH Price and Stealing Yield from Exiting Holders — (`contracts/LRTConverter.sol`)

---

### Summary

`ethValueInWithdrawal` is set once as a fixed ETH snapshot when stETH is moved to the converter. It is never updated to reflect stETH rebase rewards that accrue while the withdrawal request sits in the Lido queue. Because `updateRSETHPrice` uses this stale value as the sole accounting for the converter's assets, the rsETH price is systematically understated during the entire unstaking window. rsETH holders who exit during this window receive less ETH than their fair share; the accrued rebase yield is silently transferred to holders who remain until after `claimStEth` is called.

---

### Finding Description

**Step 1 — Snapshot is taken at transfer time.** [1](#0-0) 

`ethValueInWithdrawal` is incremented by `(_amount * oracle.getAssetPrice(stETH)) / 1e18` — a one-time snapshot of the stETH value at the moment of transfer. No mechanism exists to update this value while the withdrawal is pending.

**Step 2 — stETH is burned into the Lido queue; converter balance drops to zero.** [2](#0-1) 

After `_unstakeStEth`, the converter holds no stETH. The only accounting for the pending ETH is `ethValueInWithdrawal`.

**Step 3 — `getAssetDistributionData` explicitly zeroes the converter's stETH contribution.** [3](#0-2) 

stETH held in the converter is intentionally excluded from the per-asset total and redirected to the ETH accounting path.

**Step 4 — The ETH accounting path reads the stale snapshot.** [4](#0-3) 

`getETHDistributionData` returns `ethValueInWithdrawal` as `ethLyingInConverter`. This is the only place the pending withdrawal value enters the price calculation.

**Step 5 — `_getTotalEthInProtocol` sums all assets including ETH, using the stale snapshot.** [5](#0-4) 

`totalETHInProtocol` is computed from `getTotalAssetDeposits` for each supported asset. For ETH, this resolves to `getETHDistributionData()`, which includes the stale `ethValueInWithdrawal`. Any rebase yield that accrued in the Lido queue since the snapshot is invisible here.

**Step 6 — At claim time, the surplus ETH is silently absorbed.** [6](#0-5) 

When `ETH received > ethValueInWithdrawal`, `_sendEthToDepositPool` zeros `ethValueInWithdrawal` and forwards the full balance. The surplus (rebase yield) enters the deposit pool and inflates the rsETH price for remaining holders — at the expense of those who already exited.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield.**

rsETH holders who burn their tokens during the Lido unstaking window receive an rsETH price that excludes the rebase yield accrued in the withdrawal queue. That yield is later captured by holders who wait until after `claimStEth`. The magnitude scales with: (stETH in queue) × (stETH APY) × (days waiting). At 3.5% APY and a 7-day queue, ~0.067% of the queued value is misattributed per cycle. For large queues (e.g., 10 000 stETH ≈ $35 M), this is ~$23 500 per cycle, recurring with every unstake-claim round trip.

---

### Likelihood Explanation

**Likelihood: High.**

- Lido rebases every ~24 hours; yield accrual is continuous and certain.
- Lido v2 withdrawal finalization typically takes 1–7 days; the window is always open.
- `updateRSETHPrice` is called regularly (by bots or users); price updates during the window are expected.
- No attacker action is required — the mispricing occurs automatically as part of normal protocol operation. Any user who happens to withdraw rsETH during the window is silently harmed.

---

### Recommendation

Replace the fixed snapshot with a live valuation. Two options:

1. **Dynamic recomputation**: In `getETHDistributionData`, instead of returning `ethValueInWithdrawal`, compute the current ETH value of all pending Lido withdrawal NFTs by querying the Lido withdrawal queue for the finalized ETH amount (or the current share-rate equivalent). This requires iterating tracked request IDs.

2. **Periodic refresh**: Add an operator-callable `refreshEthValueInWithdrawal()` that re-prices the pending stETH using the current oracle price before each `updateRSETHPrice` call. This is simpler but still requires timely operator action.

Either way, `ethValueInWithdrawal` must reflect the current ETH value of the pending withdrawal, not the value at the time of the original transfer.

// Fork test — run against mainnet fork at a recent block
// Assumes: stETH, LRTConverter, LRTOracle, LRTDepositPool are deployed and configured

import "forge-std/Test.sol";

interface ILRTConverter {
    function transferAssetFromDepositPool(address asset, uint256 amount) external;
    function unstakeStEth(uint256 amount) external;
    function claimStEth(uint256 requestId, uint256 hint) external;
    function ethValueInWithdrawal() external view returns (uint256);
}

interface ILRTOracle {
    function updateRSETHPrice() external;
    function rsETHPrice() external view returns (uint256);
}

interface IStETH {
    function balanceOf(address) external view returns (uint256);
    function submit(address) external payable returns (uint256);
}

contract RebaseYieldPoC is Test {
    ILRTConverter converter = ILRTConverter(CONVERTER_ADDR);
    ILRTOracle oracle = ILRTOracle(ORACLE_ADDR);
    IStETH stETH = IStETH(STETH_ADDR);

    function testRebaseYieldInvisibleDuringWithdrawal() public {
        // 1. Operator transfers stETH to converter and initiates unstake
        vm.prank(ASSET_TRANSFER_ROLE);
        converter.transferAssetFromDepositPool(address(stETH), 1000 ether);

        uint256 ethValueBefore = converter.ethValueInWithdrawal();

        vm.prank(OPERATOR);
        converter.unstakeStEth(1000 ether);

        // 2. Advance 7 days — stETH rebases occur daily
        vm.warp(block.timestamp + 7 days);
        // Simulate Lido oracle rebase (on a real fork this happens automatically)

        // 3. updateRSETHPrice during the waiting window
        oracle.updateRSETHPrice();
        uint256 priceAfterRebase = oracle.rsETHPrice();

        // 4. ethValueInWithdrawal has NOT changed despite 7 days of rebase
        uint256 ethValueAfterRebase = converter.ethValueInWithdrawal();
        assertEq(ethValueAfterRebase, ethValueBefore,
            "ethValueInWithdrawal must equal snapshot — rebase yield invisible");

        // 5. Claim — ETH received > ethValueInWithdrawal
        uint256 balanceBefore = address(DEPOSIT_POOL).balance;
        vm.prank(OPERATOR);
        converter.claimStEth(REQUEST_ID, HINT);
        uint256 balanceAfter = address(DEPOSIT_POOL).balance;

        uint256 ethReceived = balanceAfter - balanceBefore;
        assertGt(ethReceived, ethValueBefore,
            "ETH received must exceed snapshot — surplus is rebase yield");

        // 6. ethValueInWithdrawal is now 0; surplus silently absorbed by remaining holders
        assertEq(converter.ethValueInWithdrawal(), 0);

        // Conclusion: rsETH price during step 3 was understated by (ethReceived - ethValueBefore).
        // Any rsETH burned between steps 2 and 5 received less ETH than entitled.
    }
}
