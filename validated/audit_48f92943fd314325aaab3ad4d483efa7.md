### Title
Gateway Admission Accepts Declare Transactions with Unbounded `l2_gas.max_amount` — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `validate_resource_bounds` function in `StatelessTransactionValidator` enforces a `max_l2_gas_amount` ceiling on the `l2_gas.max_amount` field for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. Any unprivileged user can submit a Declare transaction with `l2_gas.max_amount` set to an arbitrarily large value (up to `u64::MAX`) and the gateway will admit it, violating the admission invariant that `max_l2_gas_amount` is meant to enforce uniformly.

### Finding Description

In `StatelessTransactionValidator::validate_resource_bounds`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { .. });
}
``` [1](#0-0) 

The `if let RpcTransaction::Declare(_) = tx { }` branch is an intentional no-op — the `max_l2_gas_amount` guard is never reached for Declare transactions. The default configured limit is `1_210_000_000` gas units: [2](#0-1) 

The test `valid_l2_gas_amount_on_declare` explicitly documents and asserts this bypass — a Declare transaction with `max_amount: GasAmount(200)` against a limit of `100` passes validation: [3](#0-2) 

The corresponding negative test `test_invalid_max_l2_gas_amount` only covers `DeployAccount` and `Invoke` transaction types, confirming Declare is excluded: [4](#0-3) 

### Impact Explanation

This is a direct analog to the `availableCap`/`maxMint` bug class: the gateway's admission function returns `Ok(())` for Declare transactions without accounting for the `max_l2_gas_amount` constraint, exactly as `maxMint` returned `uint256.max` without accounting for `availableCap`.

A Declare transaction admitted with `l2_gas.max_amount` far exceeding `max_l2_gas_amount` (e.g., equal to the full block `sierra_gas` budget of `5_000_000_000`) enters the mempool and reaches the batcher. Inside `EntryPointExecutionContext::max_steps`, the step limit is derived directly from `max_amount`:

```rust
ValidResourceBounds::AllResources(AllResourceBounds {
    l2_gas: ResourceBounds { max_amount, .. }, ..
}) => max_amount.0.saturating_div(l2_gas_per_step)
``` [5](#0-4) 

This gives the Declare transaction a step budget proportional to the attacker-controlled `max_amount`, capped only by the block-level `invoke_tx_max_n_steps`. The gateway's per-transaction admission cap — the only place where `max_l2_gas_amount` is enforced — is bypassed entirely for Declare transactions.

**Impact category:** High — Mempool/gateway admission accepts invalid transactions before sequencing. A Declare transaction with `l2_gas.max_amount > max_l2_gas_amount` is admitted when it should be rejected.

### Likelihood Explanation

Exploitability is straightforward: any user can craft a valid Declare transaction (with a real contract class) and set `l2_gas.max_amount` to any value above `1_210_000_000`. No privileged access is required. The gateway's stateless validator is the first and only enforcement point for this bound; the stateful validator only checks the L2 gas *price* threshold, not the amount: [6](#0-5) 

### Recommendation

Remove the Declare-specific exemption and apply the `max_l2_gas_amount` check uniformly to all transaction types. The TODO comment acknowledges this gap:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
```

Change to:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { .. });
}
```

If Declare transactions legitimately require a higher gas ceiling (e.g., due to class compilation costs), introduce a separate `max_l2_gas_amount_declare` config parameter rather than removing the check entirely.

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` with:
   - A real Sierra contract class (any minimal contract).
   - `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` (or any value `> 1_210_000_000`).
   - `resource_bounds.l2_gas.max_price_per_unit = GasPrice(min_gas_price)` (e.g., `8_000_000_000`).
2. Submit to the gateway's `add_transaction` endpoint.
3. Observe: the gateway's `StatelessTransactionValidator::validate` returns `Ok(())` — the transaction is admitted to the mempool despite `max_amount` exceeding `max_l2_gas_amount` by orders of magnitude.
4. For Invoke with the same bounds, the gateway correctly returns `MaxGasAmountTooHigh`.

The existing test at line 191–201 already serves as a proof of concept, asserting `Ok(())` for a Declare transaction with `max_amount` double the configured limit. [7](#0-6)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
