Audit Report

## Title
`maxNumberOfWithdrawalsPerUser` Uninitialized in `initialize()` Blocks All Withdrawal Initiations — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal()` checks `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, which evaluates to `0 >= 0 = true` for every caller, causing every withdrawal initiation to revert with `WithdrawalLimitReached`. All staked KERNEL tokens are inaccessible until an admin separately calls `setMaxNumberOfWithdrawalsPerUser()`.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared as a plain storage `uint256` at line 108, defaulting to `0`: [1](#0-0) 

The `initialize()` function sets `kernelToken`, `rewardsToken`, and the admin role, but never assigns this variable: [2](#0-1) 

The guard in `initiateWithdrawal()` at line 323 uses `>=`: [3](#0-2) 

Since `userWithdrawalIds[msg.sender].length` is `0` for a fresh user, and `maxNumberOfWithdrawalsPerUser` is `0`, the condition `0 >= 0` is always `true`, causing an immediate revert. A constant `MAX_WITHDRAWALS_PER_USER = 100` exists at line 38 but is never used to seed the state variable: [4](#0-3) 

A setter `setMaxNumberOfWithdrawalsPerUser()` exists at lines 610–620 but is not called in `initialize()` or any deployment script (no `.s.sol` files exist in the repository): [5](#0-4) 

## Impact Explanation
**Medium — Temporary freezing of funds.** Any user who has staked KERNEL tokens via `stake()` or `stakeFor()` cannot call `initiateWithdrawal()` to begin the unstaking process. Their balance is recorded in `balanceOf[user]` and `totalKernelStaked`, but the only exit path (`initiateWithdrawal` → `claimWithdrawal`) is permanently gated behind the broken limit check until an admin calls `setMaxNumberOfWithdrawalsPerUser()`. Funds are not permanently lost, but they are inaccessible from deployment until that admin action is taken. [6](#0-5) 

## Likelihood Explanation
**High.** The condition triggers for every user on every call from the moment the contract is deployed. No special attacker capability is required — any ordinary staker calling `initiateWithdrawal()` will hit the revert. The only mitigation is an out-of-band admin action that is not part of the deployment flow and is not enforced anywhere in the codebase. [2](#0-1) 

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to the intended cap inside `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
```

Alternatively, add a non-zero invariant check at the end of `initialize()` to ensure the value is always set before the contract becomes operational. [2](#0-1) 

## Proof of Concept
1. Deploy `KernelDepositPool` via its proxy and call `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` is now `0`.
2. As any user, approve and call `stake(1e18)`. Succeeds; `balanceOf[user] = 1e18`.
3. Call `initiateWithdrawal(1e18)`.
4. The check at line 323: `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser` → `0 >= 0` → `true`.
5. Transaction reverts with `WithdrawalLimitReached`.
6. User's KERNEL tokens remain locked. Repeat for any amount or any user — the result is identical.

Foundry test sketch:
```solidity
function test_initiateWithdrawal_blockedByZeroLimit() public {
    // deploy proxy, initialize
    pool.initialize(admin, address(kernelToken), address(rewardToken));
    // stake
    vm.startPrank(user);
    kernelToken.approve(address(pool), 1e18);
    pool.stake(1e18);
    // attempt withdrawal — must revert
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(1e18);
    vm.stopPrank();
}
``` [1](#0-0)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
```

    function setMaxNumberOfWithdrawalsPerUser(uint256 _maxNumberOfWithdrawalsPerUser)
        external
        onlyRole(DEFAULT_ADMIN_ROLE)
    {
        if (_maxNumberOfWithdrawalsPerUser == 0 || _maxNumberOfWithdrawalsPerUser > MAX_WITHDRAWALS_PER_USER) {
            revert InvalidMaxNumberOfWithdrawalsPerUser();
        }

        maxNumberOfWithdrawalsPerUser = _maxNumberOfWithdrawalsPerUser;
        emit MaxNumberOfWithdrawalsPerUserUpdated(_maxNumberOfWithdrawalsPerUser);
    }
