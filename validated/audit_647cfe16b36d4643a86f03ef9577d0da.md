### Title
Missing N0 ≠ N1 Guard in `two_paillier_equal_interactive_t::verify` Allows Malicious P1 to Reuse Old Paillier Modulus Through Refresh — (`src/cbmpc/zk/zk_paillier.cpp`, `src/cbmpc/protocol/ecdsa_2p.cpp`)

### Summary

Neither `two_paillier_equal_interactive_t::verify` nor the `ecdsa2pc::refresh` protocol checks that the new Paillier modulus `N_tag` differs from the old one. A malicious P1 can send `N_tag = N_old`, construct a valid `two_paillier_equal` proof (trivially, since they hold the private key for both "keys"), pass all other checks, and cause P2 to accept a "refreshed" key whose Paillier component is identical to the pre-refresh key. The refresh's proactive-security guarantee for the Paillier layer is completely nullified.

### Finding Description

**Entrypoint:** `cbmpc_ecdsa_2p_refresh` (C API) → `coinbase::api::ecdsa_2p::refresh` → `coinbase::mpc::ecdsa2pc::refresh`.

**Step 1 — P1 reuses the old key.**
In `ecdsa2pc::refresh`, P1 is supposed to call `new_key.paillier.generate()` to produce a fresh RSA modulus. A malicious P1 instead sets `new_key.paillier = key.paillier` (or simply skips `generate()` and copies the old key), so `N_tag = key.paillier.get_N() = N_old`. [1](#0-0) 

**Step 2 — P2 receives N_tag and performs no equality check.**
P2 validates the bit-length of `N_tag`, calls `create_pub(N_tag)`, and calls `verify_cipher(c_key_tag)`. None of these checks compare `N_tag` against `key.paillier.get_N()`. [2](#0-1) 

**Step 3 — `two_paillier_equal_interactive_t::verify` has no N0 ≠ N1 guard.**
The verifier extracts `N0` and `N1` from P0 and P1 respectively, checks positivity and bit-length, then proceeds directly to the ZK proof equations. There is no `if (N0 == N1) return error(...)` guard anywhere in the function. [3](#0-2) 

The non-interactive variant has the same omission: [4](#0-3) 

**Step 4 — Proof is trivially satisfiable with N0 = N1.**
When P0 = P1 (same key), P1 encrypts `x_share` under the old key to produce `c_key_tag`, then constructs the `two_paillier_equal` proof with both slots using the same private key. The proof equations are satisfied because the same `x` is encrypted under the same modulus in both positions. The `valid_paillier_interactive` proof on `new_key.paillier` also succeeds because P1 already holds `phi(N_old)`. [5](#0-4) 

**Step 5 — P2 accepts and stores the "refreshed" key.**
`zk_two_paillier_equal.verify` returns `SUCCESS`, `com.open` succeeds, and `new_key.paillier` is stored with `N = N_old`. The refreshed key blob serialized and returned to the caller contains the unchanged Paillier modulus. [6](#0-5) 

### Impact Explanation

The refresh protocol's stated purpose is proactive security: after a successful refresh, a prior compromise of P1's Paillier private key (`phi(N_old)`) should be useless. If P1 reuses `N_old`, any adversary who previously obtained `phi(N_old)` retains the ability to decrypt `c_key_new = Enc_{N_old}(x1_new)` and recover P1's new share `x1_new`. Combined with a future compromise of P2's share `x2_new`, the full ECDSA private key `x = x1_new + x2_new` is recoverable — exactly the scenario refresh is designed to prevent. The proactive-security guarantee of the Paillier layer is entirely voided by a single malicious refresh invocation.

This is a **High**-severity validation bypass in a shipped refresh API path that causes P2 to accept cryptographically invalid output (a "refreshed" key with an unchanged Paillier modulus).

### Likelihood Explanation

P1 is a protocol participant who controls their own key material. Deliberately reusing the old Paillier key requires only that P1 skip or override the `generate()` call and copy the old `paillier_t` object — a trivial code-level modification for a malicious implementation. No cryptographic break or side-channel is required. The attack is silent: P2 receives a well-formed, proof-verified key blob and has no way to detect the reuse.

### Recommendation

1. **In `ecdsa2pc::refresh`:** After P2 receives `N_tag`, add an explicit check:
   ```cpp
   if (bn_t(N_tag) == bn_t(key.paillier.get_N()))
     return rv = job.mpc_abort(E_CRYPTO, "N' must differ from old N");
   ``` [7](#0-6) 

2. **In `two_paillier_equal_t::verify` and `two_paillier_equal_interactive_t::verify`:** Add a guard at the top of each verify function:
   ```cpp
   if (bn_t(N0) == bn_t(N1)) return coinbase::error(E_CRYPTO);
   ``` [8](#0-7) [9](#0-8)
