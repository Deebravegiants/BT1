### Title
Single Proof-Manager Task Failure DOS Entire Proposal Validation — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

During proposal validation, when `ProposalFin` is received, all `verify_and_store_proof_tasks` are awaited **sequentially** in a `for` loop. A single task failure — due to proof-manager unavailability, filesystem error, or proof-verification failure — causes the **entire** proposal validation to abort. This is the direct sequencer analog of Notional M-7: just as a single external money-market revert DOS'd Notional's rebalancing loop, a single proof-manager failure DOS's the validator's `ProposalFin` processing loop.

---

### Finding Description

**Root cause — sequential task loop in `handle_proposal_part` (`ProposalFin` arm)**

When `ProposalFin` is received, the validator drains and awaits every `VerifyAndStoreProofTask` one at a time:

```rust

---

### Impact Explanation

A validator node rejects a **valid** proposal because one proof-manager or class-manager call failed transiently. The `ProposalCommitment` is never sent to consensus. If a quorum of validators is affected simultaneously (e.g., shared proof-manager service outage, filesystem pressure, or a network partition between validators and the proof-manager), consensus cannot finalize the block. Block production stalls for the duration of the outage — a direct liveness DOS on the sequencer, matching the "rejects valid transactions before sequencing" impact category.

---

### Likelihood Explanation

Medium. The proof manager uses filesystem-backed storage (`FsProofStorage`) and is accessed via an RPC client in distributed deployments. Transient I/O errors, network blips, or a brief service restart are realistic failure modes. The probability scales with the number of transactions carrying `proof_facts` in a proposal: a single such transaction is sufficient to trigger the failure. [8](#0-7) [9](#0-8) 

---

### Recommendation

1. **Decouple task failure from proposal failure.** Treat a proof-manager communication error as a transient condition: log it, skip the failed task, and continue awaiting the remaining tasks. Only fail the proposal if the proof is cryptographically invalid (i.e., `VerifyProofError::Verification`), not on infrastructure errors.

2. **Parallelize and isolate.** Await all tasks with `futures::future::join_all` and collect results, then inspect each result individually — mirroring the pattern already used for transaction conversion — so one failure does not short-circuit the rest.

3. **Retry on transient errors.** For `ProofManagerClientError::ClientError` (communication failure), retry with bounded back-off before treating the task as failed.

---

### Proof of Concept

1. A proposer builds a proposal containing at least one `Invoke` transaction with non-empty `proof_facts` (client-side proving enabled).
2. The proof manager service becomes transiently unavailable on validator nodes (e.g., brief restart, filesystem full, network partition).
3. `spawn_verify_and_store_proof` spawns a task; the task calls `proof_manager_client.contains_proof()` and receives `ProofManagerClientError::ClientError(CommunicationFailure(...))`.
4. The task returns `Err(TransactionConverterError::ProofManagerClientError(...))`.
5. `await_verify_and_store_proof_task` maps this to `Err(String)`.
6. The `for task in verify_and_store_proof_tasks.drain(..)` loop returns `HandledProposalPart::Failed(e)` on the **first** failing task; remaining tasks are abandoned.
7. `validate_proposal` calls `batcher_abort_proposal` and returns `Err(ValidateProposalError::ProposalPartFailed(...))`.
8. The validator sends no commitment to consensus; the proposal round fails.
9. If ≥ f+1 validators share the same proof-manager outage, no quorum is reached and block production halts for the entire outage window.

    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,

    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

            return Ok(false);

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)

        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }

#[async_trait::async_trait]
impl ProofStorage for FsProofStorage {
    type Error = FsProofStorageError;

    async fn set_proof(&self, facts_hash: Felt, proof: Proof) -> Result<(), Self::Error> {
        self.write_proof_atomically(facts_hash, proof).await
    }

    async fn get_proof(&self, facts_hash: Felt) -> Result<Option<Proof>, Self::Error> {
        if !self.contains_proof(facts_hash).await? {
            return Ok(None);
        }

        match self.read_proof_from_file(facts_hash).await {
            Ok(proof) => Ok(Some(proof)),
            Err(FsProofStorageError::IoError(e)) if e.kind() == std::io::ErrorKind::NotFound => {
                Ok(None)
            }
            Err(e) => Err(e),
        }
    }

    async fn contains_proof(&self, facts_hash: Felt) -> Result<bool, Self::Error> {
        Ok(tokio::fs::try_exists(self.get_persistent_dir(facts_hash)).await?)
    }
