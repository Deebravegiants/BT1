Audit Report

## Title
Permissionless `updateRSETHPrice()` Auto-Pause Freezes Post-Burn Withdrawal Assets - (File: contracts/LRTOracle.sol, contracts/LRTWithdrawalManager.sol)

## Summary

`LRTOracle.updateRSETHPrice()` is callable by any address and, when the rsETH price has fallen more than `pricePercentageLimit` below `highestRsethPrice`, unconditionally pauses `LRTWithdrawalManager`. Because `completeWithdrawal()` is gated by `whenNotPaused`, users whose rsETH has already been irreversibly burned by `unlockQueue()` cannot retrieve their owed ETH/LST until an admin manually unpauses the contract, leaving them holding neither rsETH nor the underlying asset.

## Finding Description

**Step 1 — Permissionless trigger.** `updateRSETHPrice()` is declared `public` with only the oracle's own `whenNotPaused` guard; there is no role check. [1](#0-0) 

**Step 2 — Auto-pause on price drop.** Inside `_updateRsETHPrice()`, when `newRsETHPrice` is below `highestRsethPrice` by more than `pricePercentageLimit`, the function calls `withdrawalManager.pause()`, pauses the oracle itself, and returns without updating the stored price. [2](#0-1) 

**Step 3 — Irreversible rsETH burn in `unlockQueue()`.** The operator's `unlockQueue()` call burns all rsETH held for the batch and redeems the corresponding assets from `LRTUnstakingVault` into `LRTWithdrawalManager` in a single transaction. After this point the user's rsETH is permanently destroyed. [3](#0-2) 

**Step 4 — `completeWithdrawal()` blocked by pause.** Both user-facing completion functions carry `whenNotPaused`, so they revert unconditionally while the contract is paused. [4](#0-3) [5](#0-4) 

**Exploit path:**
1. Alice calls `initiateWithdrawal()` — rsETH moves into `LRTWithdrawalManager`.
2. Operator calls `unlockQueue()` — Alice's rsETH is burned; ETH/LST sits in the contract.
3. An EigenLayer slashing event reduces `totalETHInProtocol`, pushing `newRsETHPrice` below `highestRsethPrice * (1 - pricePercentageLimit)`.
4. Any address calls `updateRSETHPrice()` — `LRTWithdrawalManager` is paused.
5. Alice calls `completeWithdrawal()` — reverts with `Pausable: paused`. Alice holds no rsETH and no underlying asset until an admin calls `unpause()`.

**Why existing checks are insufficient.** The `whenNotPaused` guard on `updateRSETHPrice()` only checks the oracle's own pause state, not the caller's identity. Once the price condition is satisfied on-chain, the pause is atomically triggered by any EOA or contract in a single call. There is no on-chain mechanism to bypass the pause for requests that have already been unlocked, and no automatic unpause path exists.

## Impact Explanation

**Temporary freezing of funds (Medium).** After `unlockQueue()` burns a user's rsETH, the user is in an irrecoverable intermediate state: rsETH is gone and the owed ETH/LST is stranded inside `LRTWithdrawalManager`. The freeze persists until an admin calls `LRTWithdrawalManager.unpause()`. This matches the "Temporary freezing of funds" impact class in the allowed scope.

## Likelihood Explanation

**Medium.** A genuine rsETH price decline (e.g., from EigenLayer validator slashing reducing `totalETHInProtocol`) is required to satisfy the threshold. However, once the on-chain price condition is met, the trigger is fully permissionless — any address can call `updateRSETHPrice()` to atomically pause the withdrawal manager. The window between `unlockQueue()` (rsETH burned) and `completeWithdrawal()` (assets transferred) spans at least one separate transaction and typically multiple blocks, making the race condition realistic in practice.

## Recommendation

- **Short term:** Remove `whenNotPaused` from `completeWithdrawal()` and `completeWithdrawalForUser()`. Requests that have already been unlocked (rsETH burned, assets redeemed) represent a firm, irrevocable obligation; blocking their completion during a pause provides no protocol protection and only harms users who have already surrendered their rsETH.
- **Long term:** Introduce a two-tier pause: one modifier that blocks new `initiateWithdrawal()` entries and a separate modifier that blocks `unlockQueue()`, while leaving `completeWithdrawal()` always accessible. This allows the protocol to halt new exposure while still honouring existing obligations.

## Proof of Concept

```solidity
// 1. Alice initiates withdrawal
lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1 ether, "ref");

// 2. Operator unlocks queue — Alice's rsETH is burned, ETH redeemed into WithdrawalManager
lrtWithdrawalManager.unlockQueue(ETH_TOKEN, nextNonce, minAP, minRP, maxAP, maxRP);
// State: Alice.rsETH == 0, WithdrawalManager.balance += expectedETH

// 3. EigenLayer slashing reduces totalETHInProtocol so that:
//    newRsETHPrice < highestRsethPrice * (1 - pricePercentageLimit)

// 4. Attacker (any address) triggers auto-pause
lrtOracle.updateRSETHPrice();
// _updateRsETHPrice() executes lines 278-281:
//   withdrawalManager.pause();
//   _pause(); return;

// 5. Alice attempts to collect her ETH
lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "ref");
// ↳ reverts: "Pausable: paused"
// Alice's rsETH is gone; her ETH is frozen until admin calls withdrawalManager.unpause()
```

A Foundry fork test can reproduce this by: (a) forking mainnet, (b) manipulating `highestRsethPrice` storage to a value above the current price by more than `pricePercentageLimit`, (c) calling `updateRSETHPrice()` from an unprivileged address, and (d) asserting that `completeWithdrawal()` reverts while `unlockedWithdrawalsCount` is non-zero.
