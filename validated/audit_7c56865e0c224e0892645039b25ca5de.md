### Title
Merkle Leaf Lacks Contract/Chain Binding in `KernelTop100MerkleDistributor`, Enabling Cross-Deployment Proof Replay — (File: `contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

---

### Summary

`KernelTop100MerkleDistributor._verifyClaimProof` constructs the merkle leaf as `keccak256(abi.encodePacked(user, amount))` — the most minimal possible encoding, containing no contract address, no chain ID, and no distributor-epoch identifier. Because LRT-rsETH is a cross-chain protocol that naturally deploys the same reward distribution contracts on multiple networks, any user included in the merkle tree can replay their single valid proof across every chain where the same root is live, draining KERNEL tokens from each deployment.

---

### Finding Description

In `_verifyClaimProof`, the leaf is constructed as:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
``` [1](#0-0) 

The leaf encodes only `(user, amount)`. It does not include:
- `address(this)` — the contract address
- `block.chainid` — the chain identifier
- Any distributor-epoch or round identifier

The merkle root is fixed at initialization and never rotated:

```solidity
merkleRoot = _merkleRoot;
``` [2](#0-1) 

There is no `setMerkleRoot` function in this contract. The root is immutable after deployment. For a "Top 100" reward campaign, the natural operational pattern is to compute one merkle tree off-chain and deploy the same root to every chain where KERNEL is distributed. Because the leaf contains no chain-binding data, a proof that is valid on Chain A is cryptographically identical and valid on Chain B, Chain C, and any future deployment sharing the same root.

The `claim` and `claimAndStake` functions both delegate to `_verifyClaimProof` and then track claims only in the local `userClaims` mapping:

```solidity
userClaims[user].lastClaimTimestamp = block.timestamp;
userClaims[user].amountClaimed += claimableAmount;
``` [3](#0-2) 

This state is per-chain and per-contract. A claim recorded on Chain A has no effect on the `userClaims` mapping on Chain B.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Every user in the merkle tree can submit the same `(amount, merkleProof)` tuple to every chain deployment that shares the root. Each call succeeds independently because each contract's `userClaims` mapping is empty at the start. The attacker receives `N × claimableAmount` KERNEL tokens (minus fees) where `N` is the number of chains with the same root, instead of the intended single allocation. The KERNEL tokens held in each chain's distributor contract are fully drained by the set of eligible claimants.

---

### Likelihood Explanation

**Low-to-Medium.**

The LRT-rsETH protocol is explicitly cross-chain (L1 + multiple L2s). The `KernelTop100MerkleDistributor` is a named "Top 100" campaign — a fixed, one-time allocation. The operational default for such a campaign is to compute one merkle tree and deploy it uniformly across all supported chains. No admin error is required; the vulnerability is triggered by the standard deployment pattern. The only mitigating factor is that the protocol must actually deploy this contract on more than one chain with the same root, which is not guaranteed but is the natural path.

---

### Recommendation

Bind the leaf to the deployment context by including `address(this)` and `block.chainid` in the hash:

```solidity
// Before (vulnerable)
bytes32 leaf = keccak256(abi.encodePacked(user, amount));

// After (fixed)
bytes32 leaf = keccak256(abi.encodePacked(address(this), block.chainid, user, amount));
```

The off-chain merkle tree generator must be updated to include the same fields when computing leaves. This costs one additional `keccak256` input slot and zero extra storage.

---

### Proof of Concept

1. Protocol deploys `KernelTop100MerkleDistributor` on Ethereum (Chain A) and Arbitrum (Chain B), both initialized with the same `_merkleRoot` covering the same top-100 user set.
2. Alice is in the tree with `(user=Alice, amount=1000e18)`. She obtains her merkle proof off-chain.
3. Alice calls `claim(1000e18, proof)` on Chain A. `_verifyClaimProof` computes `leaf = keccak256(abi.encodePacked(Alice, 1000e18))`, verifies against `merkleRoot`, succeeds. Alice receives `~1000e18` KERNEL on Chain A. `userClaims[Alice].amountClaimed = 1000e18` on Chain A.
4. Alice calls `claim(1000e18, proof)` on Chain B with the identical calldata. `_verifyClaimProof` computes the same `leaf` (same `user`, same `amount`, no chain binding), verifies against the same `merkleRoot`, succeeds. `userClaims[Alice].amountClaimed` on Chain B is still `0`, so `_getUnclaimedVestedAmount` returns a non-zero amount. Alice receives another `~1000e18` KERNEL on Chain B.
5. Alice repeats for every additional chain. Total received: `N × ~1000e18` KERNEL instead of `~1000e18`.

The root cause is identical to the referenced report: the merkle leaf does not encode the context (here: chain + contract) that should bind it to a single valid claim site. [4](#0-3)

    function _verifyClaimProof(address user, uint256 amount, bytes32[] calldata merkleProof) internal view {
        UtilLib.checkNonZeroAddress(user);

        if (merkleRoot == bytes32(0)) {
            revert ZeroValueProvided();
        }

        if (amount == 0) {
            revert ZeroValueProvided();
        }

        // Verify the merkle proof
        bytes32 leaf = keccak256(abi.encodePacked(user, amount));
        bool isValid = MerkleProofUpgradeable.verify(merkleProof, merkleRoot, leaf);

        if (!isValid) {
            revert InvalidMerkleProof();
        }
```
