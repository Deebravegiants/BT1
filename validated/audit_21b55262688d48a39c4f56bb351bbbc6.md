Audit Report

## Title
Missing Zero-Amount Guard on Fee Transfer Enables Temporary Freeze of All Unclaimed Yield When Treasury Is Blacklisted — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary

`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` at line 144 with no `if (fee > 0)` guard. When the distributed token is a blacklistable ERC20 (e.g., USDC) and `protocolTreasury` is added to that token's blacklist, every user's `claim()` invocation reverts — freezing all unclaimed yield for the entire user base until the owner updates the treasury address. The sibling contract `KernelTop100MerkleDistributor` applies the correct guard, confirming the fix is known but was not applied here.

## Finding Description

In `MerkleDistributor.claim()`, after state updates at lines 134–135, the function computes the fee and unconditionally transfers it:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;   // line 138
uint256 amountToSend = claimableAmount - fee;           // line 139

IERC20(token).safeTransfer(account, amountToSend);      // line 141
IERC20(token).safeTransfer(protocolTreasury, fee);      // line 144 — no guard
```

There is no `if (fee > 0)` check before line 144. For blacklistable tokens such as USDC, the `notBlacklisted(to)` modifier fires on every `transfer()` call regardless of amount — including zero-value transfers. Two distinct failure modes exist:

- **`feeInBPS > 0`**: `fee > 0` is transferred to `protocolTreasury`; if `protocolTreasury` is blacklisted, the call reverts.
- **`feeInBPS = 0`**: `fee = 0` is still passed to `safeTransfer(protocolTreasury, 0)`; USDC's blacklist check fires and reverts.

Because the state update (`userClaims[account].lastClaimedIndex` and `userClaims[account].cumulativeAmount`) precedes both transfers at lines 134–135, a revert on the second transfer rolls back the entire transaction, leaving user state unchanged. Every subsequent `claim()` call by any user hits the same revert path, freezing all unclaimed yield until the owner calls `setProtocolTreasury()`.

By contrast, `KernelTop100MerkleDistributor.claim()` correctly guards the fee transfer at lines 332–334:

```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
```

This demonstrates the fix is known within the codebase but was not applied to `MerkleDistributor`.

## Impact Explanation

**Medium — Temporary freezing of unclaimed yield.** When `protocolTreasury` is blacklisted by the distributed token, no user can successfully call `claim()`. All unclaimed yield is frozen for the entire user base until the contract owner calls `setProtocolTreasury()` to point to a non-blacklisted address. The freeze is temporary (owner-remediable) but affects all users simultaneously, not just the attacker.

## Likelihood Explanation

**Low.** Requires two conditions: (1) the `MerkleDistributor` instance is configured with a blacklistable token such as USDC — a realistic configuration given the contract is explicitly generic and supports arbitrary ERC20 tokens; and (2) the `protocolTreasury` address is added to that token's blacklist by the token issuer. USDC blacklisting is a documented, exercised feature (Circle has blacklisted addresses in the past, including Tornado Cash-related addresses). The protocol cannot prevent Circle from blacklisting any address. The combination is low-probability but non-negligible given the contract's generic design.

## Recommendation

Add a zero-amount guard before the fee transfer, consistent with `KernelTop100MerkleDistributor`:

```solidity
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

Optionally, consider a pull-payment pattern: accumulate fees in a storage variable and expose a separate `withdrawFees()` function callable by the treasury, fully decoupling fee collection from user claims.

## Proof of Concept

1. Deploy `MerkleDistributor` with USDC as `token`, `feeInBPS = 0`, and `protocolTreasury = 0xTREASURY`.
2. Owner calls `setMerkleRoot(root)` with a valid Merkle root covering at least one user.
3. USDC's blacklist operator adds `0xTREASURY` to the USDC blacklist.
4. A legitimate user calls `claim(index, account, cumulativeAmount, proof)` with a valid Merkle proof.
5. Execution reaches `IERC20(token).safeTransfer(protocolTreasury, 0)` at line 144.
6. USDC's `transfer()` applies `notBlacklisted(0xTREASURY)`, which reverts.
7. The entire transaction reverts; `userClaims[account]` is unchanged.
8. Every subsequent `claim()` call by any user hits the same revert — all unclaimed yield is frozen until the owner calls `setProtocolTreasury(newAddress)`.
