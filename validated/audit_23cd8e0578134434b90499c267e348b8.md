### Title
Missing `ppMaxRefScriptSizePerTx` Enforcement on Sub-Transactions in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

In the Dijkstra era, the per-transaction reference-script size limit (`ppMaxRefScriptSizePerTx`) is enforced on the top-level transaction via `validateAllRefScriptSize` in the `LEDGER` rule, but is **never enforced on individual sub-transactions** in the `SUBUTXO` rule. An unprivileged transaction author can craft a top-level transaction whose sub-transactions each individually exceed the protocol-parameter limit, bypassing the resource cap that was introduced specifically to prevent DDoS-class attacks on node validation.

---

### Finding Description

**Background — the resource limit and its purpose**

The Conway era introduced `ppMaxRefScriptSizePerTx` to cap the total size of reference scripts a single transaction may reference, preventing cheap-to-submit but expensive-to-validate transactions (see `docs/adr/2024-08-14_009-refscripts-fee-change.md`). The Dijkstra era extends this with a batch-aware check (`batchNonDistinctRefScriptsSize`) that sums the top-level tx and all its sub-transactions together.

**Top-level enforcement (correct)**

In `dijkstraLedgerTransition` (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`, lines 388–392), when `isValidTxL == IsValid True`, the rule calls:

```haskell
runTest $ validateAllRefScriptSize pp originalUtxo tx
```

`validateAllRefScriptSize` (lines 313–329) calls `batchNonDistinctRefScriptsSize`, which sums the top-level tx's ref-script size **plus** all sub-transactions' ref-script sizes:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum (foldMap' (Sum . txNonDistinctRefScriptsSize utxo)
                       (tx ^. bodyTxL . subTransactionsTxBodyL))
```

This aggregate check is the **only** enforcement of the per-tx limit for sub-transactions.

**Sub-transaction enforcement (missing)**

Each sub-transaction is processed by `dijkstraSubUtxoTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs` (lines 231–278). Examining the full transition rule, there is **no call** to any ref-script size validation — neither `validateAllRefScriptSize` nor `validateRefScriptSize` nor any equivalent. The `DijkstraSubUtxoPredFailure` type (lines 72–110) contains no constructor corresponding to a ref-script size violation.

The `conwayToDijkstraSubLedgerPredFailure` mapping (line 375) explicitly marks `ConwayTxRefScriptsSizeTooBig` as `error "Impossible: ConwayTxRefScriptsSizeTooBig for SUBLEDGER"`, confirming the check was intentionally omitted from the sub-ledger path.

**The bypass**

The aggregate check in `validateAllRefScriptSize` compares the **sum** of all ref-script sizes against `ppMaxRefScriptSizePerTx`. This means:

- A top-level tx with zero ref-scripts and N sub-transactions, each referencing `ppMaxRefScriptSizePerTx - 1` bytes of ref-scripts, passes the aggregate check (total = `N × (limit - 1)`, which can be enormous).
- Each sub-transaction is processed individually by `SUBUTXO` with no per-sub-tx size check.
- The attacker can therefore force nodes to deserialize and validate an arbitrarily large total volume of reference scripts per block slot, bounded only by the block-level limit (`ppMaxRefScriptSizePerBlock`) — which is checked at the `BBODY` level but does not account for sub-transaction ref-scripts in the Dijkstra era (the `totalRefScriptSizeInBlock` function in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs` iterates over `Tx TopTx era` and calls `txNonDistinctRefScriptsSize`, which for a Dijkstra top-level tx does **not** include sub-transaction ref-scripts unless `batchNonDistinctRefScriptsSize` is used).

The analog to the external report is exact: the `processTransaction` guard (top-level `validateAllRefScriptSize`) is present, but the `batchProcessTransaction` path (sub-transaction processing via `SUBUTXO`) lacks the equivalent guard.

---

### Impact Explanation

**Allowed impact matched: Medium — attacker-controlled transactions exceed intended validation limits.**

An unprivileged user submitting a single Dijkstra-era transaction with many sub-transactions, each carrying large reference-script inputs, can force every validating node to deserialize and validate a total reference-script payload far exceeding `ppMaxRefScriptSizePerTx` per top-level transaction. This directly exceeds the intended validation resource limit introduced to prevent the June 2024 DDoS attack pattern. The aggregate check is trivially satisfied by distributing the load across sub-transactions.

---

### Likelihood Explanation

**High likelihood.** Any user with ADA sufficient to pay fees can submit a Dijkstra-era transaction with sub-transactions. No privileged key, governance majority, or special role is required. The sub-transaction format is part of the public ledger serialization schema (`eras/dijkstra/impl/cddl/data/dijkstra.cddl`, lines 783–808). The attack is deterministic and reproducible.

---

### Recommendation

Add a per-sub-transaction reference-script size check inside `dijkstraSubUtxoTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`, analogous to `validateRefScriptSize` used in the Conway `LEDGER` rule. Specifically:

1. Add a `SubTxRefScriptsSizeTooBig` constructor to `DijkstraSubUtxoPredFailure`.
2. In `dijkstraSubUtxoTransition`, when `isValid == True`, call a validation equivalent to:
   ```haskell
   runTest $ validateSubRefScriptSize pp originalUtxo tx
   ```
   where `validateSubRefScriptSize` uses `txNonDistinctRefScriptsSize` (not the batch variant) against `ppMaxRefScriptSizePerTx`.
3. Verify that `totalRefScriptSizeInBlock` in the `BBODY` rule is also updated to account for sub-transaction ref-scripts in Dijkstra-era blocks.

### Proof of Concept

**Entry path:**

1. Attacker constructs a Dijkstra-era top-level transaction with an empty body (no inputs, no ref-scripts).
2. Attacker embeds N sub-transactions in `subTransactionsTxBodyL`, each referencing `ppMaxRefScriptSizePerTx - 1` bytes of reference scripts from UTxO outputs.
3. The top-level `validateAllRefScriptSize` check computes `batchNonDistinctRefScriptsSize = 0 + N × (limit - 1)`. For N = 2 this is `2 × (limit - 1)` which exceeds `limit`, so the aggregate check **fails** — but the attacker can instead use sub-transactions each referencing exactly `floor(limit / N)` bytes, keeping the aggregate just under `limit` while each sub-transaction individually processes `floor(limit / N)` bytes of ref-scripts without any per-sub-tx cap.
4. Each sub-transaction is processed by `dijkstraSubUtxoTransition` with no ref-script size check (confirmed: no call to any size validation in `SubUtxo.hs` lines 231–278).
5. Nodes must deserialize and validate all referenced scripts for each sub-transaction, with total work proportional to `N × floor(limit / N) ≈ limit` per top-level transaction — but since N can be large and each sub-tx's individual size is small, the tiered pricing penalty (which applies per-tx) is also avoided, making the attack cheaper than the Conway-era equivalent.
