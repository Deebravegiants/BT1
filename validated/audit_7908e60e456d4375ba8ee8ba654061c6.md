### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Pending Request — Cross-Request Signature Replay by a Single Byzantine Leader - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` contract method verifies only that the submitted `signature` is cryptographically valid over the caller-supplied `payload_hash` under the MPC domain public key. It does not verify that `payload_hash` is derived from the pending `request` that is being resolved. A single Byzantine MPC node acting as signing leader can replay a legitimately-produced threshold signature from a prior signing round (for a different foreign-chain request) to resolve an unrelated pending request, causing the contract to return a forged `VerifyForeignTransactionResponse` to the original caller.

### Finding Description

In `respond_verify_foreign_tx` (lines 691–754 of `crates/contract/src/lib.rs`), the contract:

1. Accepts a `request: VerifyForeignTransactionRequest` and a `response: VerifyForeignTransactionResponse` from any attested MPC participant.
2. Verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's MPC public key.
3. Resolves all pending yields queued under `request` by writing `response` (including the unchecked `payload_hash`) into the yield data.


The `payload_hash` field is a caller-supplied value. The contract never reconstructs or bounds-checks it against the `request` that is being resolved. This is the direct analog of the Sandclock `receiveFlashLoan` bug: a parameter provided by an external caller (`feeAmounts` / `payload_hash`) is silently accepted without validation.

The node-side signing flow shows that the leader independently computes `payload_hash` from the foreign chain query result and submits it to the contract:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs:34-47
fn build_signature_request(request: &VerifyForeignTxRequest,
                            foreign_tx_payload: &dtos::ForeignTxSignPayload)
    -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    ...
}
```

After the threshold protocol completes, the leader holds the full threshold signature over `payload_hash`. Nothing prevents the leader from submitting that signature paired with a *different* `payload_hash` value (from a prior signing round) when calling `respond_verify_foreign_tx` for a different pending request.

### Impact Explanation

A single Byzantine MPC node that has previously acted as leader for any completed signing round possesses a valid threshold signature `sig_B` over some hash `H_B = SHA256(borsh(ForeignTxSignPayload{request_B, values_B}))`. It can call:

```
respond_verify_foreign_tx(request = request_A, response = {payload_hash = H_B, signature = sig_B})
```

The contract's signature check passes (`sig_B` is valid over `H_B` under the MPC key). The contract resolves all yields queued under `request_A` with `{payload_hash = H_B, signature = sig_B}`. Every caller who submitted `request_A` receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes the extracted values of a completely different foreign-chain transaction (`request_B`).

A downstream bridge contract that verifies only the ECDSA signature (which is valid) but does not re-derive and compare the `payload_hash` against its own expected values will accept this as a legitimate attestation of `request_A`. This enables forged foreign-chain verification and invalid bridge execution (e.g., releasing bridged assets for a transaction that was never confirmed, or for a transaction with fewer confirmations than required).

The SDK-level verifier (`ForeignChainSignatureVerifier::verify_signature` in `crates/near-mpc-sdk/src/foreign_chain.rs`) does perform the payload_hash binding check, but it is a client-side library — the contract itself provides no on-chain enforcement.

### Likelihood Explanation

- Requires exactly one Byzantine MPC node to have been the signing leader for any prior completed request. This is a realistic condition in production: any node that has ever led a signing round holds a reusable threshold signature.
- No threshold collusion is required. The threshold signature for `request_B` was legitimately produced; the Byzantine node is simply replaying it.
- The attack is silent: the contract emits no error, and the victim caller receives a structurally valid `VerifyForeignTransactionResponse`.
- Bridge contracts that rely on the MPC attestation without re-verifying `payload_hash` content are directly exploitable.

### Recommendation

**Short term:** In `respond_verify_foreign_tx`, require the responding node to also supply the `Vec<ExtractedValue>` that were observed. Recompute `expected_hash = SHA256(borsh(ForeignTxSignPayload{request, values}))` on-chain and assert `response.payload_hash == expected_hash` before accepting the response. If NEAR promise-data size limits prevent including extracted values in the response, store a commitment to the expected hash at request submission time (once the node reports it) and verify against that commitment.

**Long term:** Document and enforce the invariant that `payload_hash` in `VerifyForeignTransactionResponse` must be derived from the specific `request` being resolved. Add a contract-level test that attempts to resolve request A with a valid signature produced for request B and asserts rejection.

### Proof of Concept

1. Honest user Alice submits `verify_foreign_transaction` for Bitcoin tx `0xAA` with 6 confirmations. This creates a pending yield under `request_A`.
2. In a prior round, the MPC network legitimately signed `request_B` (Bitcoin tx `0xBB`, 1 confirmation). The leader node holds `(H_B, sig_B)` where `sig_B` is a valid threshold signature over `H_B`.
3. The Byzantine leader calls:
   ```
   respond_verify_foreign_tx(
     request = request_A,   // Alice's pending request
     response = { payload_hash = H_B, signature = sig_B }
   )
   ```
4. The contract at line 729 verifies `verify_ecdsa_signature(sig_B, H_B, mpc_pk)` → passes.
5. Line 749–753 resolves Alice's yield with `{payload_hash = H_B, signature = sig_B}`.
6. Alice's bridge contract receives the response, verifies the ECDSA signature (valid), and interprets it as MPC attestation that tx `0xAA` was confirmed with 6 confirmations — but the signed payload actually encodes tx `0xBB` with 1 confirmation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```rust
impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
```
