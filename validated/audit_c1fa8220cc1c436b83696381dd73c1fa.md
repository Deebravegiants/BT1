### Title
Missing Cryptographic Validation of CKD Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract validates the CKD response cryptographically only for the `AppPublicKeyPV` (publicly verifiable) variant. For the `AppPublicKey` (privately verifiable / legacy) variant, **no cryptographic check is performed on the response**. Because `resolve_yields_for` delivers the first accepted response to the user, a single Byzantine attested participant — strictly below the signing threshold — can race to submit an arbitrary forged `CKDResponse` and have it accepted and delivered to the requesting user.

---

### Finding Description

In `respond_ckd`, after verifying the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO validation
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, cryptographically binding the response to the user's public key and the network's master key. For `AppPublicKey`, the branch is an empty no-op — any `big_y` and `big_c` values pass through unconditionally.

Immediately after, `resolve_yields_for` drains the pending queue and delivers the response to the user:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

`resolve_yields_for` is a first-writer-wins mechanism: the first call that finds the pending entry removes it and resumes all queued yields with the supplied bytes. There is no threshold quorum on `respond_ckd` — a single attested participant suffices.

By contrast, `respond` (for ECDSA/EdDSA signatures) always verifies the signature cryptographically against the derived public key before calling `resolve_yields_for`, so the same race is not exploitable there.

The test at `lib.rs:4513–4521` inadvertently confirms the gap: it passes `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (invalid BLS12-381 points) for an `AppPublicKey` request and the contract accepts the call without error.

---

### Impact Explanation

A single Byzantine MPC participant (below the signing threshold) can submit a forged `CKDResponse` for any pending `AppPublicKey` CKD request. The contract delivers the forged `(big_y, big_c)` to the user. Because the `AppPublicKey` variant is privately verifiable — only the holder of the corresponding BLS scalar can decrypt and check the output — the user cannot detect the forgery on-chain. The user's derived confidential key is therefore attacker-controlled, constituting unauthorized confidential key derivation output without the required threshold participant authorization.

This maps to the **Critical** allowed impact: *Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization.*

---

### Likelihood Explanation

The `AppPublicKey` variant is the legacy and default CKD path (documented as the primary example in `crates/contract/README.md`). Any single attested MPC participant can exploit this by monitoring the NEAR chain for pending CKD requests and racing to call `respond_ckd` with arbitrary values before honest nodes respond. No threshold collusion, key leakage, or TEE bypass is required — only membership in the attested participant set.

---

### Recommendation

Apply the same cryptographic output check to the `AppPublicKey` variant. The `ckd_output_check` function already exists and handles the pairing verification; a parallel check for the `AppPublicKey` case should verify that `e(big_c, g2) = e(big_y, G2) · e(hash_point, public_key)` (substituting the generator for `app_pk2` since `AppPublicKey` uses `G1` only). Alternatively, deprecate the `AppPublicKey` variant entirely and require all new requests to use `AppPublicKeyPV`.

---

### Proof of Concept

1. User submits a CKD request with `AppPublicKey` variant (the legacy/default path).
2. Byzantine participant (one of the attested MPC nodes) observes the pending request on-chain.
3. Byzantine participant calls `respond_ckd` with:
   - `request`: the exact `CKDRequest` from the pending map (publicly visible on-chain).
   - `response`: `CKDResponse { big_y: <attacker-chosen G1 point>, big_c: <attacker-chosen G1 point> }`.
4. `respond_ckd` passes `assert_caller_is_attested_participant_and_protocol_active()` (attacker is a legitimate participant).
5. The `AppPublicKey(_) => {}` branch performs no validation.
6. `resolve_yields_for` removes the pending entry and resumes the user's yield with the forged bytes.
7. The user receives `(big_y, big_c)` chosen by the attacker. When the user decrypts with their private scalar, they obtain a key entirely controlled by the attacker rather than the MPC network's honest derivation.

The existing unit test at `crates/contract/src/lib.rs:4513–4521` demonstrates that `respond_ckd` accepts `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (not valid BLS12-381 curve points) for an `AppPublicKey` request without any rejection. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
