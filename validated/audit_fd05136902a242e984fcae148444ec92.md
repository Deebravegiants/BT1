### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Byzantine Participant to Inject Arbitrary Key Material — (`crates/contract/src/lib.rs`)

### Summary
`respond_ckd` performs no cryptographic check on the `(big_y, big_c)` response when the request uses the `AppPublicKey` variant. Any single attested participant can race to resolve a pending CKD request with attacker-chosen values, causing the victim application to derive a confidential key the attacker controls.

### Finding Description
The `respond_ckd` function matches on `request.app_public_key` to decide whether to verify the CKD output. For `AppPublicKeyPV`, it calls `ckd_output_check`, which enforces the pairing equation `e(C, g2) = e(Y, pk2) · e(H(pk, app_id), msk_pk)`. For `AppPublicKey`, the match arm is empty — the response `(big_y, big_c)` is accepted without any verification and immediately passed to `resolve_yields_for`, which resolves the NEAR yield and delivers the response to the waiting caller.

The design rationale is that `AppPublicKey` lacks a G2 component (`pk2`), making the pairing check impossible on-chain. The intent is that security relies on the threshold node protocol. However, the contract does not enforce that the submitted response is the output of the threshold protocol — it only checks that the caller is an attested participant. Any single attested participant can call `respond_ckd` with arbitrary `big_y` and `big_c` before honest nodes respond.

### Impact Explanation
The victim application receives `C - a·Y` as its confidential key, where `C` and `Y` are attacker-chosen. The attacker can select values for which they know the discrete log relationship (e.g., `big_y = r·G`, `big_c = r·pk1 + t·G` for known `r, t`), giving them full knowledge of the derived key. Any assets, secrets, or capabilities the application protects with that key are compromised. This constitutes unauthorized confidential key derivation output — the threshold requirement is bypassed by a single Byzantine participant.

### Likelihood Explanation
The attacker must be an attested participant (passed TEE attestation). This is Byzantine behavior strictly below the signing threshold. The race window is the time between a CKD request being submitted and honest nodes responding. In practice, honest nodes respond within seconds, but a Byzantine participant monitoring the chain can submit immediately upon seeing the request.

### Recommendation
Either:
1. Require `AppPublicKey` requests to also provide a G2 component (effectively requiring `AppPublicKeyPV`), enabling on-chain verification for all CKD requests, or
2. Implement a threshold-response aggregation scheme at the contract level for `AppPublicKey` requests, requiring `t-of-n` participants to submit matching `(big_y, big_c)` values before the response is accepted.

### Proof of Concept
Submit a CKD request with `AppPublicKey(pk)`. Before honest nodes respond, call:
```
respond_ckd(
  request = CKDRequest { app_public_key: AppPublicKey(pk), app_id: victim_app_id, domain_id },
  response = CKDResponse { big_y: G1_generator, big_c: G1_generator }
)
```
The call succeeds. The victim receives key material `G1 - a·G1 = (1-a)·G1`, which is not `msk·H(pk, app_id)`. The attacker knows the value `(1-a)` if they know `a` (the app's secret scalar), or can choose `big_c = big_y = identity` to force the derived key to the identity point, effectively nullifying it.
