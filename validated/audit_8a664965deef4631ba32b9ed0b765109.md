### Title
Hardcoded Reference Script Fee and Size Parameters in Conway Era Prevent Governance Adjustment - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/PParams.hs`)

---

### Summary

In the Conway era, four critical reference-script resource-limit parameters are hardcoded as compile-time constants in the `ConwayEraPParams` typeclass instance rather than being stored as adjustable protocol parameters. Because governance cannot change these values without a hard fork, if the constants prove insufficient to deter resource-exhaustion attacks, an unprivileged transaction sender can submit transactions whose reference-script processing cost exceeds what the fee formula charges, effectively paying below the true validation cost and exceeding the intended fee design parameters.

---

### Finding Description

The `ConwayEraPParams` instance for `ConwayPParams` implements four getter functions as pure constants:

```haskell
ppMaxRefScriptSizePerTxG    = L.to . const $ 200 * 1024          -- 204 800 bytes
ppMaxRefScriptSizePerBlockG = L.to . const $ 1024 * 1024         -- 1 048 576 bytes
ppRefScriptCostMultiplierG  = L.to . const . fromJust $ boundRational 1.2
ppRefScriptCostStrideG      = L.to . const $ knownNonZeroBounded @25_600
``` [1](#0-0) 

These four getters are the sole source of truth for:

1. **Per-transaction reference-script size enforcement** (`validateRefScriptSize`), which reads `ppMaxRefScriptSizePerTxG`: [2](#0-1) 

2. **Per-block reference-script size enforcement** (`validateBodyRefScriptsSizeTooBig`), which reads `ppMaxRefScriptSizePerBlockG`: [3](#0-2) 

3. **Minimum-fee calculation** (`getConwayMinFeeTx` → `tierRefScriptFee`), which reads both `ppRefScriptCostMultiplierG` and `ppRefScriptCostStrideG`: [4](#0-3) 

The tiered fee formula itself is: [5](#0-4) 

The ADR explicitly acknowledges the intentional hardcoding: *"we had to hard code some values, which will be turned into proper protocol parameters in the next era"* and lists the hard caps as *"currently hard coded, but will be turned into actual protocol parameters in the next era after Conway."* [6](#0-5) [7](#0-6) 

In the Dijkstra era these four values are properly promoted to updatable protocol parameters (`dppMaxRefScriptSizePerBlock`, `dppMaxRefScriptSizePerTx`, `dppRefScriptCostStride`, `dppRefScriptCostMultiplier`): [8](#0-7)

### Impact Explanation

**Medium — Attacker-controlled transactions exceed intended validation limits / modify fees outside design parameters.**

Because `ppRefScriptCostMultiplierG` (1.2) and `ppRefScriptCostStrideG` (25 600) are compile-time constants, the exponential tier pricing cannot be tightened by a governance `ParameterChange` action in the Conway era. If the actual CPU/memory cost of deserializing reference scripts rises (e.g., due to a node software change or a newly discovered script pattern), the fee charged by `tierRefScriptFee` will remain anchored to the hardcoded curve. An unprivileged sender can then submit transactions with reference scripts up to the hardcoded 200 KiB cap while paying a fee that is below the true validation cost, allowing sustained resource exhaustion at sub-economic cost — the same class of attack that occurred on 25 June 2024. Governance has no on-chain lever to raise the multiplier or lower the cap without a hard fork.

---

### Likelihood Explanation

**Medium.** The June 25 2024 DDoS attack on Cardano mainnet (referenced in the ADR) demonstrates that reference-script resource exhaustion is a live, non-theoretical threat. The hardcoded values were chosen as an emergency fix under time pressure; the ADR itself acknowledges they may not be optimal. Any change to node deserialization performance or discovery of a new script pattern that is cheap to execute but expensive to deserialize would reactivate the attack surface with no governance remedy available in Conway.

---

### Recommendation

1. **Short term (Conway):** Introduce an intra-era hard-fork guard (similar to existing `hardforkConway*` flags) that allows the four constants to be overridden via a genesis-file extension, giving operators an emergency lever without waiting for the Dijkstra upgrade.
2. **Long term (already done in Dijkstra):** The Dijkstra era correctly promotes all four values to proper protocol parameters (`dppMaxRefScriptSizePerBlock`, `dppMaxRefScriptSizePerTx`, `dppRefScriptCostStride`, `dppRefScriptCostMultiplier`), making them adjustable by governance `ParameterChange` actions. [9](#0-8)

### Proof of Concept

1. Obtain the current Conway protocol parameters; note `minFeeRefScriptCostPerByte` (the only adjustable base price).
2. Construct a transaction that includes reference inputs whose combined script size approaches the hardcoded 200 KiB cap (`ppMaxRefScriptSizePerTxG = 200 * 1024`).
3. Compute the required fee using `tierRefScriptFee 1.2 25600 baseFee totalSize` — this is the exact formula enforced by the ledger.
4. Submit the transaction. It will be accepted as long as the declared fee ≥ the computed minimum.
5. Repeat at high frequency. Because the multiplier (1.2) and stride (25 600) are fixed, governance cannot raise the fee curve in response; the only remedy is a hard fork. The per-block cap (`ppMaxRefScriptSizePerBlockG = 1 MiB`) bounds the damage per block but does not eliminate the sustained low-cost attack surface.

tierRefScriptFee multiplier sizeIncrement
  | multiplier <= 0 || sizeIncrement <= 0 = error "Size increment and multiplier must be positive"
  | otherwise = go 0
  where
    go !acc !curTierPrice !n
      | n < sizeIncrement =
          Coin $ floor (acc + toRational n * curTierPrice)
      | otherwise =
          go (acc + sizeIncrementRational * curTierPrice) (multiplier * curTierPrice) (n - sizeIncrement)
    sizeIncrementRational = toRational sizeIncrement
```
