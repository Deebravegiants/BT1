### Title
Peer-Controlled `highest_height` Triggers Unconditional DB Wipe via `StaleSyncHash` — (`chain/client/src/sync/state/mod.rs`)

### Summary

A connected peer can send a `PeerMessage::Block` with an arbitrarily large height. The height is recorded in `ConnectedPeerState.block_info` without cryptographic verification before the block is validated. On the next sync tick, `StateSync::run` reads this peer-supplied height as `highest_height` and compares it against the stale-sync-hash threshold. If the fake height exceeds `sync_hash_height + epoch_length + STALE_SYNC_HASH_THRESHOLD`, the node unconditionally wipes its data directory and restarts. The attack is repeatable across restarts, permanently preventing state sync completion.

### Finding Description

**Step 1 — Peer height is recorded without block validation.**

In `chain/network/src/peer_manager/network_state/mod.rs`, when a `PeerMessage::Block` arrives, `update_block_info` is called immediately before the block is forwarded to the client for validation: [1](#0-0) 

`update_block_info` in `connected_peers.rs` applies only a monotonicity check (new height ≥ stored height) — no signature or chain validity check: [2](#0-1) 

**Step 2 — The fake height propagates to `highest_height_peers`.**

`highest_height_peers()` in `peer_manager_actor.rs` builds `HighestHeightPeerInfo` directly from `ConnectedPeerState.block_info`, which now contains the attacker-supplied height: [3](#0-2) 

The function returns all peers within `highest_peer_horizon` of the maximum height. A single peer claiming a very large height dominates the list: [4](#0-3) 

**Step 3 — The only filter (`is_block_invalid`) does not protect against unprocessed blocks.**

`syncing_info()` in `client_actor.rs` filters peers whose `highest_block_hash` is already known to be invalid: [5](#0-4) 

A freshly received block whose hash is not yet in the invalid-block set passes this filter. Block processing is asynchronous — the invalid-block mark is written only after the client processes the block, which can happen after the sync tick fires. The attacker can also rotate hashes (send a new block with a different hash at the same fake height) to stay ahead of the invalid-block filter indefinitely.

**Step 4 — `StateSync::run` fires the DB-wipe path.**

`highest_height` (the attacker-controlled value) is passed directly into `StateSync::run`, which compares it against the stale-sync-hash threshold with no further verification: [6](#0-5) 

`STALE_SYNC_HASH_THRESHOLD` is 100 in production — the attacker only needs to claim a height of `sync_hash_height + epoch_length + 101`: [7](#0-6) 

**Step 5 — `StaleSyncHash` unconditionally wipes the DB.**

`StaleSyncHash` propagates to `SyncHandlerRequest::EpochSyncDataReset`: [8](#0-7) 

Which triggers the shutdown signal: [9](#0-8) 

On restart, `check_epoch_sync_data_reset_marker` deletes the entire data directory: [10](#0-9) 

### Impact Explanation

Any peer connected to a node that is in `StateSync` can force an unconditional data-directory wipe and restart. After the restart, the node begins state sync from scratch. The attacker reconnects and repeats the attack. The victim node can never complete state sync and never recovers canonical state. The impact is permanent prevention of state sync recovery, matching the scoped "fail to recover canonical state" criterion.

### Likelihood Explanation

The attack requires only a standard P2P connection to the victim node — no validator keys, no stake, no special privileges. The attacker sends a single `PeerMessage::Block` with a fabricated height. The race window (between `update_block_info` and the block being marked invalid) is wide because block processing is async and the sync tick fires every ~2 seconds. Even if the race is lost, the attacker can rotate block hashes to stay ahead of the invalid-block filter. The attack is trivially repeatable across restarts.

### Recommendation

The `highest_height` value used in the stale-sync-hash check must not be taken directly from unverified peer-reported block heights. Mitigations include:

1. **Verify before recording:** Only update `block_info` after the block passes at least header-level signature validation, not before.
2. **Require quorum:** Require a majority (or supermajority) of connected peers to agree on a height before treating it as the network tip for the stale-sync-hash check.
3. **Decouple the check from peer heights:** Use the locally verified header chain tip (from `chain.header_head()`) rather than peer-reported heights for the stale-sync-hash comparison. The node already has headers synced before entering state sync, so the local header head is a trustworthy upper bound.
4. **Rate-limit or delay the DB-wipe action:** Require the stale condition to persist for multiple consecutive sync ticks before triggering the irreversible DB wipe.

### Proof of Concept

A test-loop test demonstrating the issue:

1. Start a node, advance it past the far-horizon threshold so it enters `StateSync`.
2. Block state part downloads (as in the existing `test_far_horizon_stale_sync_hash_detection` test).
3. Instead of advancing the real chain, inject a single `PeerMessage::Block` from a connected peer with `height = sync_hash_height + epoch_length + STALE_SYNC_HASH_THRESHOLD + 1` and an arbitrary hash not yet in the invalid-block set.
4. Run one sync tick.
5. Assert the node is denylisted (DB wipe triggered) even though the real network tip has not advanced past the epoch.

The existing test at `test-loop-tests/src/tests/sync/far_horizon.rs` lines 777–860 already demonstrates the `StaleSyncHash` path fires correctly when `highest_height` exceeds the threshold — the only difference here is that `highest_height` is peer-fabricated rather than reflecting real chain progress. /// Checks for the epoch sync data reset marker and deletes the data directory if found.
/// Archival nodes skip deletion to prevent accidental data loss.
fn check_epoch_sync_data_reset_marker(hot_store_path: &Path, is_archival: bool) {
    let marker_path = hot_store_path.join(EPOCH_SYNC_DATA_RESET_MARKER_FILE_NAME);
    if !marker_path.exists() {
        return;
    }
    if is_archival {
        tracing::warn!(target: "neard", "epoch sync data reset marker found but node is archival, ignoring");
    } else {
        tracing::info!(target: "neard", ?hot_store_path, "epoch sync data reset marker found, deleting data directory");
        std::fs::remove_dir_all(hot_store_path)
            .expect("failed to delete data directory for epoch sync reset");
    }
}
