### Title
Unvalidated `ProposalInit.builder` Allows Malicious Proposer to Redirect Sequencer Fees and Corrupt Block Hash — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`ProposalInit.builder` is a proposer-supplied field that flows directly into `sequencer_address` in `BlockInfo` (via `convert_to_sn_api_block_info`). The `sequencer_address` is both the fee recipient during block execution and a direct input to the block hash computation. The `is_proposal_init_valid()` function validates many `ProposalInit` fields but never checks `builder`. Any legitimate proposer can set `builder` to an arbitrary address, causing all validators to execute the block with the wrong `sequencer_address`, permanently committing a wrong block hash and crediting fees to an attacker-controlled address.

### Finding Description

**Step 1 — Proposer sets `builder` freely.**

In `initiate_build()`, `ProposalInit.builder` is set from `args.builder_address`, which comes from `self.config.static_config.builder_address` — a value the proposer node controls entirely: [1](#0-0) 

**Step 2 — `builder` becomes `sequencer_address` in `BlockInfo`.**

`convert_to_sn_api_block_info()` maps `init.builder` directly to `sequencer_address`: [2](#0-1) 

**Step 3 — `sequencer_address` enters the block hash.**

`PartialBlockHashComponents::new()` copies `block_info.sequencer_address` into the `sequencer` field, which is then chained into the Poseidon block hash: [3](#0-2) [4](#0-3) 

**Step 4 — `is_proposal_init_valid()` never checks `builder`.**

The validator checks `timestamp`, `starknet_version`, `version_constant_commitment`, `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices, and `fee_proposal_fri` — but `builder` is absent from every check: [5](#0-4) 

**Step 5 — `proposer` is validated; `builder` is not.**

`single_height_consensus.rs` verifies `init.proposer` against the committee-derived expected proposer, but there is no analogous check for `init.builder`: [6](#0-5) 

**Step 6 — The wrong `sequencer_address` propagates into `ProposalCommitment`.**

`PartialBlockHash::from_partial_block_hash_components()` is the value that becomes `ProposalCommitment`. Because `sequencer` (from `builder`) is an input, a tampered `builder` produces a tampered `ProposalCommitment` that all validators agree on: [7](#0-6) 

### Impact Explanation

A malicious proposer sets `ProposalInit.builder` to an attacker-controlled address. All validators accept the proposal (no guard rejects a wrong `builder`), execute the block with `sequencer_address = attacker_address`, and compute the same `ProposalCommitment`. Consensus reaches agreement and commits the block. Two permanent effects follow:

1. **Wrong block hash committed.** The Starknet block hash includes `sequencer_address` as a direct Poseidon input. The committed hash is permanently wrong relative to the legitimate sequencer address, corrupting the chain's canonical hash chain.

2. **Sequencer fees stolen.** The blockifier credits all sequencer fees (L1 gas, L2 gas, data gas) to `sequencer_address`. With `builder` pointing to the attacker's address, every fee paid in the block is transferred to the attacker instead of the legitimate sequencer.

Both effects are irreversible once the block is finalized. This matches the Critical impact categories: *wrong block hash committed* and *incorrect fee/balance effect with economic impact*.

### Likelihood Explanation

Any validator that is legitimately scheduled as the proposer for a round can trigger this. In a BFT committee of `n` validators, a single malicious member suffices. The attack requires no special tooling: the proposer simply configures `builder_address` in their node config to an attacker-controlled address before their proposer turn. No other validator can detect or prevent it because `builder` is never validated.

### Recommendation

Add a `builder` check inside `is_proposal_init_valid()`. The expected `builder` address must be derived from the committee (analogous to how `proposer` is derived via `committee.get_proposer()`), not accepted verbatim from the network. Until the committee provides a canonical builder address, the validator should at minimum reject proposals whose `builder` differs from the locally configured `builder_address`:

```rust
// In ProposalInitValidation, add:
pub expected_builder: ContractAddress,

// In is_proposal_init_valid(), add:
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.expected_builder, init_proposed.builder
        ),
    ));
}
```

The `expected_builder` should be populated from `self.config.static_config.builder_address` in `SequencerConsensusContext::validate_proposal()`, mirroring how `l2_gas_price_fri` and `l1_da_mode` are sourced from local config.

### Proof of Concept

1. Attacker controls validator node `V_attacker` that is scheduled as proposer for block `N`, round `R`.
2. `V_attacker` sets `static_config.builder_address = ATTACKER_WALLET` in its node configuration.
3. `V_attacker` calls `build_proposal()`. `initiate_build()` constructs `ProposalInit { builder: ATTACKER_WALLET, ... }` and broadcasts it.
4. All peer validators receive the `ProposalInit`. `is_proposal_init_valid()` checks timestamp, gas prices, starknet_version, etc. — none of these checks involve `builder`. Validation passes.
5. `initiate_validation()` calls `convert_to_sn_api_block_info(init)`, producing `BlockInfo { sequencer_address: ATTACKER_WALLET, ... }`. The batcher executes all transactions with this `BlockInfo`.
6. Every fee-paying transaction credits `ATTACKER_WALLET` as the sequencer fee recipient.
7. `PartialBlockHashComponents::new(&block_info, ...)` sets `sequencer = ATTACKER_WALLET`. `PartialBlockHash::from_partial_block_hash_components()` hashes this into `ProposalCommitment`.
8. All validators compute the same `ProposalCommitment` (with `ATTACKER_WALLET` baked in), reach BFT agreement, and commit block `N` with `sequencer_address = ATTACKER_WALLET` and a permanently wrong block hash.

async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is in the future: now={}, block_timestamp_window_seconds={}, \
                 proposed={}",
                now,
                proposal_init_validation.block_timestamp_window_seconds,
                init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.starknet_version != proposal_init_validation.starknet_version {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "starknet_version mismatch: expected={:?}, proposed={:?}",
                proposal_init_validation.starknet_version, init_proposed.starknet_version
            ),
        ));
    }
    // `version_constant_commitment` is proposer-supplied (network-derived). It is not yet a real
    // commitment (see `expected_version_constant_commitment`): the only valid value is the
    // sentinel, so reject anything else. Enforcing the same value the proposer emits keeps the two
    // sides in lockstep, so a real value cannot ship on one side without the other.
    let expected_commitment = expected_version_constant_commitment();
    if init_proposed.version_constant_commitment != expected_commitment {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "version_constant_commitment mismatch: expected={expected_commitment:?}, \
                 proposed={:?}",
                init_proposed.version_constant_commitment
            ),
        ));
    }
    if !(init_proposed.height == proposal_init_validation.height
        && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
        && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)

            init_proposed.clone(),
            proposal_init_validation.clone(),
