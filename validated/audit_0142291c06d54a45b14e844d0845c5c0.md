Audit Report

## Title
Missing Withdrawal Cancellation Mechanism Permanently Freezes User rsETH When Vault Is Insolvent - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`LRTWithdrawalManager.initiateWithdrawal()` transfers rsETH from the user into the contract and increments `assetsCommitted[asset]`, with no corresponding `cancelWithdrawal()` path. The only mechanism to decrement `assetsCommitted` and release the locked rsETH is the operator-gated `unlockQueue()` → `_unlockWithdrawalRequests()` sequence. When `LRTUnstakingVault` balance is insufficient to cover the front-of-queue request — a direct consequence of EigenLayer slashing — the sequential unlock loop breaks without decrementing `assetsCommitted`, leaving affected users' rsETH permanently locked in the contract with no on-chain remediation path.

## Finding Description

`initiateWithdrawal()` performs three irreversible state mutations:
1. Transfers rsETH from the caller into the contract ( [1](#0-0) ).
2. Computes `expectedAssetAmount` and adds it to `assetsCommitted[asset]` ( [2](#0-1) ).
3. Pushes a `WithdrawalRequest` struct into the per-user `userAssociatedNonces` queue ( [3](#0-2) ).

`assetsCommitted[asset]` is **only ever decremented** inside `_unlockWithdrawalRequests()` at the point where a request is successfully processed ( [4](#0-3) ).

The unlock loop is strictly sequential and breaks immediately when the vault's real-time balance is insufficient to cover the front-of-queue request:

```solidity
if (availableAssetAmount < payoutAmount) break;
``` [5](#0-4) 

`totalAvailableAssets` is sourced from `unstakingVault.balanceOf(asset)` — the real-time vault balance, not a protocol-wide accounting figure — meaning any shortfall from slashing directly causes the break condition ( [6](#0-5) ).

`_calculatePayoutAmount` takes `min(expectedAssetAmount, currentReturn)`, so a price drop from slashing does reduce `payoutAmount`, but if the vault balance is zero or near-zero (assets were slashed before reaching the vault), `availableAssetAmount < payoutAmount` holds regardless ( [7](#0-6) ).

There is no `cancelWithdrawal()`, no `rescueTokens()`, and no admin sweep for rsETH. `sweepRemainingAssets()` only handles non-rsETH asset surpluses (LST/ETH balances) and is additionally gated on `hasUnlockedWithdrawals(asset) == false` ( [8](#0-7) ). The rsETH held in the contract is only burned inside `unlockQueue()` for successfully unlocked requests ( [9](#0-8) ); requests that never unlock are never burned and never returned.

`completeWithdrawal()` reverts with `WithdrawalLocked` for any nonce at or above `nextLockedNonce[asset]` ( [10](#0-9) ), so users behind a blocked nonce cannot complete their withdrawal either.

## Impact Explanation

When EigenLayer slashing reduces the assets flowing back into `LRTUnstakingVault` below the sum of `expectedAssetAmount` values for queued requests, the sequential unlock loop breaks at `nextLockedNonce` without decrementing `assetsCommitted`. Because `getAvailableAssetAmount` returns `totalAssets - assetsCommitted` ( [11](#0-10) ), and `assetsCommitted` is now stuck at an elevated value, new `initiateWithdrawal()` calls also revert with `ExceedAmountToWithdraw`. The contract holds rsETH that is neither burned nor returnable. This constitutes **Temporary freezing of funds** (Medium) when the vault can eventually be replenished, escalating to **Permanent freezing of funds** (Critical) when slashing is severe enough that the vault can never be replenished to cover the committed amounts.

## Likelihood Explanation

EigenLayer slashing is an explicitly accepted protocol risk — the entire purpose of `NodeDelegator` is to restake assets in EigenLayer strategies and native EigenPods. The `LRTUnstakingVault` is the sole source of assets for `unlockQueue`. Any slashing event that reduces completed-withdrawal proceeds below the sum of `expectedAssetAmount` values for pending requests triggers the freeze. This is not a theoretical edge case; it is the exact scenario the protocol's withdrawal system must handle gracefully. No attacker capability is required — the freeze is triggered by a normal user calling `initiateWithdrawal()` followed by an external slashing event.

## Recommendation

1. **Add a `cancelWithdrawal(address asset)` function** that allows a user to pop their oldest pending (locked) request, decrement `assetsCommitted[asset]` by `request.expectedAssetAmount`, and return the held rsETH to the user.
2. Alternatively, expose an **admin-callable `forceUnlockRequest(address asset, uint256 nonce, uint256 reducedPayout)`** that can unlock a specific nonce with a slashing-adjusted payout, unblocking the sequential queue without requiring full asset coverage.
3. Consider adding a **`rescueRsETH()` emergency function** (gated to `PAUSER_ROLE`) that can return rsETH to users whose requests have been waiting beyond a configurable timeout with no unlock progress.

## Proof of Concept

```
1. Alice calls initiateWithdrawal(stETH, 10e18 rsETH).
   → 10e18 rsETH transferred to LRTWithdrawalManager (L166).
   → assetsCommitted[stETH] += 10.5 stETH (L173).
   → WithdrawalRequest stored at nonce N; nextLockedNonce[stETH] = N.

2. EigenLayer slashing event: NodeDelegator's stETH strategy shares are slashed 20%.
   → completeUnstaking() delivers only 8.4 stETH to LRTUnstakingVault instead of 10.5.

3. Operator calls unlockQueue(stETH, N+1, ...).
   → _createUnlockParams: totalAvailableAssets = unstakingVault.balanceOf(stETH) = 8.4 stETH (L849).
   → _unlockWithdrawalRequests: payoutAmount for nonce N = min(10.5, currentReturn).
   → availableAssetAmount (8.4) < payoutAmount → break (L800).
   → nextLockedNonce[stETH] unchanged at N.
   → assetsCommitted[stETH] NOT decremented (L802 never reached).

4. Alice calls completeWithdrawal(stETH) → reverts: WithdrawalLocked (L707).

5. Alice has no cancelWithdrawal() to call.
   Her 10e18 rsETH is stuck in the contract.
   getAvailableAssetAmount(stETH) = totalAssets - assetsCommitted ≈ 0 (L602).
   New withdrawal requests by other users also revert: ExceedAmountToWithdraw (L170).
```

Foundry fork test plan: fork mainnet, deploy/configure the protocol, have a test address call `initiateWithdrawal`, simulate slashing by directly reducing `LRTUnstakingVault` balance (via `deal`), call `unlockQueue`, assert `nextLockedNonce` is unchanged, assert `completeWithdrawal` reverts with `WithdrawalLocked`, assert no function exists to recover the rsETH held by the contract.
