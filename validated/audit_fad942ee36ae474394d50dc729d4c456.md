### Title
Sub-Transaction Execution Units Excluded from Per-Transaction and Per-Block ExUnits Limits — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

### Summary

In the Dijkstra era, a top-level transaction may embed an attacker-controlled number of sub-transactions, each carrying its own Plutus redeemers with declared `ExUnits`. The `validateExUnitsTooBigUTxO` check and the BBODY-level `validateExUnits` check both call `totExUnits`, which only sums redeemers from the **top-level** transaction's witness set. Sub-transaction redeemers are never aggregated. An unprivileged transaction author can therefore submit a single transaction whose total declared execution budget across all sub-transactions is an unbounded multiple of `maxTxExUnits`, while the per-transaction and per-block ExUnits guards pass without triggering. The minimum-fee formula uses the same `totExUnits`, so the attacker also pays a fee that does not cover the actual script execution cost.

### Finding Description

**Root cause — `totExUnits` is not sub-transaction-aware**

`totExUnits` is defined in `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It reads only `tx ^. witsTxL` — the top-level transaction's witness set. [1](#0-0) 

Sub-transactions are stored in `dtbrSubTransactions :: OMap TxId (Tx SubTx era)` inside the top-level body. Each `Tx SubTx era` carries its own `witsTxL` with its own redeemers and ExUnits. [2](#0-1) 

`totExUnits` is never overridden for the Dijkstra era and no Dijkstra-specific aggregation function exists (confirmed by the absence of any `batchExUnits` or equivalent in the codebase).

**Per-transaction ExUnits check — UTXO rule**

`dijkstraUtxoTransition` enforces:

```
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [3](#0-2) 

`validateExUnitsTooBigUTxO` computes `totalExUnits = totExUnits tx`, which is the top-level ExUnits only. [4](#0-3) 

Sub-transaction ExUnits are silently excluded.

**Per-block ExUnits check — BBODY rule**

`dijkstraBbodyTransition` calls:

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
``` [5](#0-4) 

`validateExUnits` computes `txTotal = foldMap totExUnits txs`, again summing only top-level redeemers across all transactions in the block. [6](#0-5) 

Sub-transaction ExUnits from every transaction in the block are excluded from the block-level cap.

**Fee underpayment**

The Alonzo `minfee` formula includes `txscriptfee (prices pp) (totExUnits tx)`. Because `totExUnits` omits sub-transaction ExUnits, the minimum fee charged to the attacker does not cover the actual Plutus execution cost of sub-transaction scripts. [7](#0-6) 

**Sub-transaction processing loop**

`SUBLEDGERS` processes all sub-transactions sequentially via `foldM`, executing each sub-transaction's Plutus scripts through `SUBUTXOW`:

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM (\ls subTx -> trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx))
        ledgerState subTxs
``` [8](#0-7) 

Each sub-transaction's scripts are executed with their declared ExUnits, none of which are checked against any limit.

**Contrast with the ref-script size check**

The Dijkstra era correctly aggregates reference-script sizes across the entire batch:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum (foldMap' (Sum . txNonDistinctRefScriptsSize utxo)
                       (tx ^. bodyTxL . subTransactionsTxBodyL))
``` [9](#0-8) 

The analogous aggregation is absent for ExUnits.

**Exploit path**

1. Attacker constructs a top-level Dijkstra transaction with zero top-level redeemers.
2. Attacker embeds N sub-transactions, each containing a Plutus script with declared ExUnits equal to `maxTxExUnits`.
3. The top-level transaction's serialized size must fit within `maxTxSize`, which bounds N but still allows multiple sub-transactions.
4. `validateExUnitsTooBigUTxO` sees `totExUnits tx = ExUnits 0 0`, passes.
5. `validateExUnits` at the block level also sees zero contribution from this transaction.
6. All N sub-transaction scripts are executed by every validating node, consuming up to N × `maxTxExUnits` actual resources.
7. The fee paid covers only the top-level script cost (zero), not the sub-transaction execution cost.

### Impact Explanation

This matches the allowed Medium impact: **attacker-controlled transactions exceed intended validation limits and modify fees outside design parameters.**

- The `maxTxExUnits` and `maxBlockExUnits` protocol parameters are designed to bound the CPU and memory cost of validating any single transaction and any single block. Both bounds are bypassed for sub-transaction Plutus scripts.
- The minimum-fee formula is also bypassed: nodes execute sub-transaction scripts without receiving the corresponding fee compensation, breaking the economic model that funds node operation.
- Nodes must execute all declared sub-transaction ExUnits before they can reject or accept the transaction; there is no early-exit once the top-level limit is satisfied.

### Likelihood Explanation

- No privileged access is required. Any Dijkstra-era transaction author can embed sub-transactions.
- The attacker only needs to construct a valid transaction body with sub-transactions containing Plutus scripts and high declared ExUnits.
- The transaction size limit (`maxTxSize`) provides a partial natural bound, but within that budget an attacker can still declare sub-transaction ExUnits far exceeding `maxTxExUnits`.
- The Dijkstra era is new and this pattern has not been exercised in prior eras, making it likely to be overlooked.

### Recommendation

1. **Aggregate sub-transaction ExUnits in `totExUnits` or introduce a Dijkstra-specific batch variant.** Define a `batchTotExUnits` function analogous to `batchNonDistinctRefScriptsSize` that sums ExUnits across the top-level transaction and all sub-transactions, and use it in both `validateExUnitsTooBigUTxO` (UTXO rule) and `validateExUnits` (BBODY rule) for the Dijkstra era.

2. **Include sub-transaction ExUnits in the minimum-fee calculation.** The `minfee` formula must account for sub-transaction script fees so that the attacker bears the full execution cost.

3. **Consider a per-batch ExUnits protocol parameter** (`maxBatchExUnits`) distinct from `maxTxExUnits`, to allow the protocol to set a tighter bound on the total execution cost of a top-level transaction plus all its sub-transactions.

### Proof of Concept

**Deterministic reasoning (no runtime required):**

Given:
- `maxTxExUnits = ExUnits { exUnitsMem = M, exUnitsSteps = S }`
- `maxTxSize = T` bytes

Construct a top-level Dijkstra transaction with:
- Empty top-level inputs/outputs/redeemers (minimal size).
- N sub-transactions, each containing one Plutus script with declared `ExUnits { exUnitsMem = M, exUnitsSteps = S }`.
- N is chosen so the total serialized size ≤ T.

Validation trace:
1. `dijkstraUtxoTransition` calls `Alonzo.validateExUnitsTooBigUTxO pp tx`.
2. `validateExUnitsTooBigUTxO` computes `totalExUnits = totExUnits tx = foldMap snd (tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL) = ExUnits 0 0`.
3. `pointWiseExUnits (<=) (ExUnits 0 0) maxTxExUnits` → `True`. Check passes.
4. `SUBLEDGERS` calls `foldM` over N sub-transactions; each sub-transaction's Plutus script is executed with `ExUnits M S`.
5. Total actual execution: `ExUnits (N*M) (N*S)`, which can be a multiple of `maxTxExUnits`.
6. `dijkstraBbodyTransition` calls `Alonzo.validateExUnits txs (pp ^. ppMaxBlockExUnitsL)`, which computes `foldMap totExUnits txs = ExUnits 0 0` for this transaction. Block-level check also passes.
7. Fee paid = `minfee pp tx` = base fee + `txscriptfee prices (ExUnits 0 0)` = base fee only, with no compensation for sub-transaction script execution.
