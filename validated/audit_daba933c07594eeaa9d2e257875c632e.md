Guard Scripts Receive No Sub-Transaction TxInfo During Guarding Operations — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs`)

---

### Summary

In the Dijkstra era, the `GuardingPurpose` Plutus script mechanism is the direct analog to Bunni's Hooklet system. Guard scripts are designed to authorize sub-transactions embedded in a top-level transaction. The `toPlutusTxInfo` implementation for `PlutusV4` computes the TxInfo for every sub-transaction (`_subTxInfosForGuards`) but then discards it entirely, passing only the top-level `topTxInfo` to the guard script. A `-- TODO: Include _subTxInfosForGuards` comment in the source code confirms this is a known incomplete provisioning.

---

### Finding Description

In `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs`, the `EraPlutusTxInfo 'PlutusV4 DijkstraEra` instance implements `toPlutusTxInfo` with a `mkTopTxInfo` branch that handles the `GuardingPurpose`:

```haskell
Right $ \case
  purpose@(GuardingPurpose AsPurpose) -> do
    _subTxInfosForGuards <-
      forM (OMap.elems (tx ^. bodyTxL . subTransactionsTxBodyL)) $ \subTx -> do
        ...
        left (SubTxContextError txId) $ mkTxInfo purpose
    -- TODO: Include _subTxInfosForGuards
    Right topTxInfo          -- ← sub-tx data silently dropped
  _ -> Right topTxInfo
```

The code iterates over all sub-transactions, translates each one into a `PlutusTxInfo`, and binds the results to `_subTxInfosForGuards`. The `_` prefix is a Haskell convention that suppresses the "unused variable" warning, confirming the binding is intentionally (but incompletely) left out of the returned `topTxInfo`. The guard script therefore receives only the top-level transaction's `PV3.TxInfo` — with no visibility into sub-transaction inputs, outputs, minting, withdrawals, certificates, or any other sub-transaction fields.

The design intent is clear from the surrounding infrastructure:

- Sub-transactions carry a `requiredTopLevelGuardsL` field (`Map (Credential Guard) (StrictMaybe (Data era))`), which explicitly declares which top-level guard scripts must authorize them, along with optional datums for those scripts. [1](#0-0) 
- The UTXOW rule enforces that every guard credential declared by sub-transactions is present in the top-level `guardsTxBodyL`. [2](#0-1) 
- `getDijkstraScriptsNeeded` registers each script-hash guard as a `GuardingPurpose` that must be satisfied (i.e., the Plutus script must execute and succeed). [3](#0-2) 

The entire purpose of a guard script is to inspect the sub-transactions it is authorizing and apply custom logic. Without sub-transaction TxInfo, a guard script can only see the top-level transaction body — it is blind to what the sub-transactions actually do.

### Impact Explanation

**Classification: Medium — Attacker-controlled transactions exceed intended validation limits.**

Guard scripts are the sole Plutus-level mechanism for enforcing custom constraints on sub-transactions. Because they receive no sub-transaction TxInfo, they cannot inspect or restrict:

- Which UTxOs a sub-transaction spends
- What tokens a sub-transaction mints or burns
- What withdrawals a sub-transaction performs
- What certificates a sub-transaction submits
- What governance votes or proposals a sub-transaction includes

A sub-transaction author (an unprivileged transaction sender) can craft sub-transactions that perform operations the guard script was designed to prohibit — unauthorized fund movements, unauthorized minting, unauthorized withdrawals — and the guard script will succeed regardless, because it has no data on which to base a rejection. The guard script's redeemer and datum are provided, but the `scriptContextTxInfo` it receives contains only the top-level transaction, making any sub-transaction-aware logic impossible to implement.

This directly maps to: *attacker-controlled transactions exceed intended validation limits or modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters.*

### Likelihood Explanation

The vulnerability becomes reachable when the Dijkstra era is deployed with PlutusV4 guard script support. The current `toPlutusScriptPurpose _ = error "stub: PlutusV4 not yet implemented"` stub prevents PlutusV4 guard scripts from executing today. [6](#0-5) 

However, the `toPlutusTxInfo` implementation — including the `_subTxInfosForGuards` TODO — is already committed to the production source tree. When `toPlutusScriptPurpose` is implemented to complete PlutusV4 support, the TODO will remain unless explicitly addressed. Any guard script author who writes logic that inspects sub-transaction data will find their script always receives an empty/top-level-only context, silently bypassing their intended constraints. The likelihood is **high** once Dijkstra is deployed, because the omission is invisible to script authors: the script compiles and executes, but the sub-transaction fields it queries will always be absent.

---

### Recommendation

Before Dijkstra era deployment, the `PV3.TxInfo` type (or a new `PV4`-specific TxInfo type) must be extended with a field carrying the list of sub-transaction TxInfos, and `mkTopTxInfo` must populate that field from `_subTxInfosForGuards`. The `-- TODO: Include _subTxInfosForGuards` marker at line 524 of `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs` must be resolved before `toPlutusScriptPurpose` is implemented for PlutusV4. [7](#0-6)

### Proof of Concept

1. Deploy a Dijkstra top-level transaction containing:
   - A sub-transaction `subTx` that withdraws from an account the guard script is supposed to restrict.
   - `subTx.requiredTopLevelGuards = { ScriptHashObj guardScriptHash → SJust datum }` declaring the guard.
   - The top-level `guardsTxBodyL` containing `ScriptHashObj guardScriptHash`.
   - A PlutusV4 guard script that, given sub-transaction TxInfo, would check `txInfoWdrl` and fail if the withdrawal target is not approved.

2. When the guard script executes, `scriptContextTxInfo` is `topTxInfo` — the top-level transaction's TxInfo. `txInfoWdrl` in that context reflects only top-level withdrawals, not `subTx`'s withdrawals.

3. The guard script's withdrawal check passes vacuously (the withdrawal it is supposed to block does not appear in the context it receives), and the sub-transaction's unauthorized withdrawal is accepted by the ledger.

The code path is:
- `getDijkstraScriptsNeeded` → registers `GuardingPurpose (AsIxItem idx guardScriptHash)` [3](#0-2) 
- `toPlutusTxInfo` → `mkTopTxInfo` → `GuardingPurpose` branch → computes `_subTxInfosForGuards`, discards it, returns `Right topTxInfo` [4](#0-3) 
- Guard script receives `topTxInfo` with no sub-transaction data → cannot enforce sub-transaction constraints → succeeds unconditionally on sub-transaction content.
