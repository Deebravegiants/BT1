**Where the question's framing is wrong:**

The "ledger divergence between nodes that construct scripts via JSON vs CBOR" claim is **not accurate**. All nodes run the same `evalMultiSig` / `isValidMOf` function regardless of how the script was originally constructed. There is no divergence — every honest node evaluates `RequireMOf (-1) []` identically as `True`. The formal spec (`multi-sig.tex` line 753) even notes "n.b., should also check that this is >= 0" in the Plutus reference implementation, but the native evaluator never received that guard.

**What the real vulnerability is:**

The missing non-negativity check on `m` in both the JSON parser and the CBOR decoder means any party can craft a `RequireMOf (negative) []` script. That script's hash produces a valid script address. Any funds sent to that address — whether by the attacker themselves or by a victim who was shown a script JSON file with `"required": -1` disguised as a legitimate multisig — can be spent by anyone with zero signatures. The UTXOW rule accepts the transaction because `validateNativeScript` returns `True`.

**Verdict:**

### Title
Negative `required` field in `RequireMOf` bypasses all signature requirements — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Scripts.hs`)

### Summary
Both `shelleyBasedEraNativeScriptJSONParser` and the CBOR decoder for `MultiSigRaw` accept a negative integer for the `m` field of `RequireMOf` without any bounds check. The evaluator `isValidMOf` treats any `m ≤ 0` as immediately satisfied, so a `RequireMOf (-1) []` script always validates with zero witnesses. Funds locked at the corresponding script address can be spent by anyone.

### Finding Description
- `shelleyBasedEraNativeScriptJSONParser` parses `"required"` as `Int` with no lower-bound guard.
- The CBOR decoder (`decCBOR` for `MultiSigRaw`, tag 3) also accepts any `Int` for `m`.
- `mkRequireMOf` stores the value verbatim in `MemoBytes`.
- `isValidMOf n Empty = n <= 0` returns `True` for any negative `n`.
- `validateNativeScript = validateMultiSig` is the function called by the UTXOW STS rule; it calls `evalMultiSig`, which calls `isValidMOf`.
- No other guard in the UTXOW rule checks the structural validity of `m`.

### Impact Explanation
Any funds locked at a `RequireMOf (negative) []` script address can be spent by submitting a transaction with the script as witness and no key signatures. The UTXOW rule accepts it. This is a direct, unauthorized transfer of ADA or native assets through an invalid ledger state transition.

### Likelihood Explanation
The attacker must get funds locked at the malicious address. The realistic vector is presenting a script JSON file with `"required": -1` to a victim (e.g., as a "2-of-3 multisig" where the JSON looks plausible but the field is negative). The victim computes the script hash and sends funds. Alternatively, the attacker can craft the CBOR directly without going through JSON at all. No privileged access is required.

### Recommendation
Add a non-negativity check in both the JSON parser and the CBOR decoder:
- In `shelleyBasedEraNativeScriptJSONParser`, after parsing `required`, fail the parser if the value is negative.
- In `DecCBOR (Annotator (MultiSigRaw era))`, after decoding `m`, reject the record if `m < 0`.
- Optionally add the same check in `mkRequireMOf` as a defensive invariant.
- The same fix must be applied to `TimelockMOf` in `Allegra/Scripts.hs` and all downstream eras, since `allegraBasedEraNativeScriptJSONParser` delegates to `shelleyBasedEraNativeScriptJSONParser`.

### Proof of Concept
```
1. Parse JSON: {"kind": "MOf", "required": -1, "scripts": []}
   → MultiSigMOf (-1) Empty
2. Compute scriptHash of the resulting MultiSig.
3. Fund a UTxO at (ScriptAddr scriptHash).
4. Submit a Tx spending that UTxO with:
     txwitsScript = {scriptHash ↦ script}
     txwitsVKey   = {}   -- no signatures
5. UTXOW rule: validateNativeScript tx script
             = evalMultiSig {} (RequireMOf (-1) [])
             = isValidMOf (-1) Empty
             = (-1) <= 0
             = True   ← script passes
6. Transaction accepted; funds transferred without authorization.
```
