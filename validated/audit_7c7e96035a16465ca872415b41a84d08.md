### Title
Fee-on-Transfer Token Deposit Causes rsETH Over-Minting and Protocol Insolvency - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.depositAsset` calculates the rsETH mint amount from the caller-supplied `depositAmount` before the `safeTransferFrom` call, then mints that pre-computed amount unconditionally. If any supported ERC-20 asset charges a transfer fee, fewer tokens arrive in the contract than `depositAmount` represents, yet the full rsETH quantity is still minted. Over time this inflates rsETH supply beyond the real backing, producing protocol insolvency.

### Finding Description
In `LRTDepositPool.depositAsset`:

```solidity
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);

IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
```

`_beforeDeposit` calls `getRsETHAmountToMint(asset, depositAmount)`, which computes:

```
rsethAmountToMint = (depositAmount √ó assetPrice) / rsETHPrice
```

using the full `depositAmount` as the asset quantity. [1](#0-0) 

The mint amount is therefore fixed before the transfer executes. If the token deducts a fee during transfer, the contract receives `depositAmount ‚àí fee` tokens, but `rsethAmountToMint` ‚Äî computed on the full `depositAmount` ‚Äî is minted anyway. [2](#0-1) 

The same pattern exists in `RSETHPoolV3.deposit(address token, ‚Ä¶)`, where `viewSwapRsETHAmountAndFee(amount, token)` is called with the input `amount` and `wrsETH.mint` is called with the result, again without verifying actual receipt: [3](#0-2) 

Contrast this with `KernelDepositPool.notifyRewardAmount`, which explicitly uses a balance-before/after pattern to determine the real received amount ‚Äî demonstrating the developers are aware of the correct approach but did not apply it to the deposit path: [4](#0-3) 

### Impact Explanation
Every deposit with a fee-on-transfer asset mints rsETH in excess of the real collateral received. `getTotalAssetDeposits` reads `IERC20(asset).balanceOf(address(this))` for the deposit pool's share, so the on-chain balance correctly reflects the lower amount, but the already-minted rsETH supply is permanently inflated. [5](#0-4)  Each subsequent depositor's rsETH is worth less than promised; at scale this constitutes **protocol insolvency** (Critical).

### Likelihood Explanation
The protocol's `lrtConfig` governance can add any ERC-20 as a supported asset. A token that introduces a transfer tax post-listing, or a new LST variant with built-in fees, would silently trigger this path for every depositor without any on-chain signal. The entry point (`depositAsset`) is fully public and requires no special role. Likelihood is **Low** in isolation but the consequence is irreversible once it occurs.

### Recommendation
Adopt the balance-before/after pattern already used in `KernelDepositPool.notifyRewardAmount` for every inbound ERC-20 transfer, and use the measured received amount for all downstream calculations:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
_mintRsETH(rsethAmountToMint);
```

Apply the same fix to `RSETHPoolV3.deposit(address,uint256,string)` and any other inbound transfer site that uses the caller-supplied amount for minting or accounting.

### Proof of Concept
1. Admin adds a token `FeeToken` (2 % transfer fee) as a supported asset in `lrtConfig`.
2. Alice calls `depositAsset(FeeToken, 100e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` based on `100e18`.
4. `safeTransferFrom` moves `100e18` from Alice; `FeeToken` deducts 2 %, so the contract receives `98e18`.
5. `_mintRsETH(rsethAmountToMint)` mints rsETH equivalent to `100e18` of `FeeToken`.
6. The protocol now holds `98e18` tokens but has issued rsETH backed by `100e18` ‚Äî a 2 % insolvency gap per deposit, compounding with every subsequent depositor. [6](#0-5)

    function _beforeDeposit(
        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
