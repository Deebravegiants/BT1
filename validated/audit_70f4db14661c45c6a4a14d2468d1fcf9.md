### Title
Proposer Deposits Permanently Destroyed When `do_update` Bulk-Clears All Pending Update Entries - (File: `crates/contract/src/update.rs`)

### Summary
`propose_update` collects a storage-proportional NEAR deposit from each proposer and stores the `bytes_used` amount in `UpdateEntry`. When any update reaches threshold and `do_update` is called, it bulk-clears **all** pending entries via `self.entries.clear()` without ever reading `bytes_used` to refund the proposers. Every participant who proposed a competing update loses their deposit permanently — an exact structural analog to the original `transferStake` overwrite bug, where a credit operation destroys pre-existing value instead of preserving it.

### Finding Description
`propose_update` in `crates/contract/src/lib.rs` enforces a deposit equal to `ProposedUpdates::required_deposit(&update)`, which is `env::storage_byte_cost() × bytes_used`. Only the excess above the required amount is refunded at proposal time; the required portion is retained by the contract. [1](#0-0) 

`bytes_used` is persisted inside `UpdateEntry` alongside the update payload: [2](#0-1) 

When `vote_update` accumulates enough votes it calls `do_update`, which removes the winning entry and then calls `self.entries.clear()` on all remaining entries: [3](#0-2) 

`do_update` never reads `entry.bytes_used` (the winning entry) nor the `bytes_used` of any cleared entry to issue refund promises. The proposer's `AccountId` is also never stored in `UpdateEntry`, so the contract has no recipient to refund even if it tried. [4](#0-3) 

The `vote_update` call site confirms no refund logic exists around `do_update`: [5](#0-4) 

### Impact Explanation
Every participant who proposed an update that was not the one executed loses their full storage deposit permanently. For a contract-code update the deposit is proportional to the wasm binary size (typically hundreds of KB to several MB). At NEAR's storage cost of ~10^19 yoctoNEAR/byte, a 500 KB binary requires ~5 NEAR; a 1 MB binary ~10 NEAR. Because `bytes_used` is stored but never consumed for refunding, the contract silently absorbs these funds on every governance upgrade cycle. This breaks the production accounting invariant that storage deposits must be returned when the corresponding storage is freed — a direct balance-destruction analog to the original `transferStake` overwrite.

**Allowed impact matched:** *Medium — Balance/request-lifecycle/contract execution-flow manipulation that breaks production safety/accounting invariants.*

### Likelihood Explanation
The MPC contract is upgraded periodically in production (mainnet: `v1.signer`, testnet: `v1.signer-prod.testnet`). Each upgrade cycle where more than one participant proposes a competing update triggers the loss. Because participants are incentivised to propose their preferred version, competing proposals are a normal governance scenario, not an edge case.

### Recommendation
- **Short term:** Store the proposer's `AccountId` in `UpdateEntry` alongside `bytes_used`. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(proposer).transfer(required_deposit(entry.bytes_used))` for each. Also refund the winning entry's proposer.
- **Long term:** Add an invariant test asserting that the sum of all outstanding `bytes_used × storage_byte_cost` values never exceeds the contract's balance minus operational reserves, and that this sum reaches zero after every `do_update` execution.

### Proof of Concept

1. Participant Alice calls `propose_update` with a 1 MB contract binary, paying ~10 NEAR deposit. `UpdateEntry { update: ..., bytes_used: 1_056_784 }` is stored under `UpdateId(0)`.
2. Participant Bob calls `propose_update` with a different binary, paying ~10 NEAR deposit. `UpdateEntry { ..., bytes_used: ... }` stored under `UpdateId(1)`.
3. Bob's proposal `UpdateId(1)` reaches threshold votes. `vote_update(UpdateId(1))` is called.
4. `do_update` executes:
   - `self.entries.remove(&UpdateId(1))` — removes Bob's entry (deposit not refunded).
   - `self.entries.clear()` — removes Alice's entry (deposit not refunded).
   - `self.vote_by_participant.clear()`.
5. Alice's ~10 NEAR is permanently locked in the contract with no refund path. Bob's ~10 NEAR is also lost.

The `bytes_used` field on both entries is discarded without ever being used to compute or issue a refund, mirroring the original `transferStake` bug where `staked` was read and then assigned to the destination while the destination's pre-existing balance was silently destroyed. [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
