### Title
Missing Unique Context in `ForeignTxSignPayload` Enables Cross-Request Signature Replay — (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

`ForeignTxSignPayloadV1` — the structure whose hash the MPC network signs for every `verify_foreign_transaction` request — contains only the raw `ForeignChainRpcRequest` and the extracted `values`. It includes no nonce, no NEAR contract address, and no block-height timestamp. Because the signed hash is therefore identical for any two requests that share the same `(tx_id, extractors, domain_id, extracted_values)` tuple, a previously produced MPC signature can be replayed verbatim to satisfy a later pending request for the same foreign transaction, delivering stale or attacker-chosen extracted values to the waiting caller.

---

### Finding Description

**Root cause — `ForeignTxSignPayloadV1` has no unique context**

`ForeignTxSignPayloadV1` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

and its hash is computed as:

```rust
pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
    let mut hasher = sha2::Sha256::new();
    borsh::BorshSerialize::serialize(self, &mut hasher)?;
    Ok(Hash256(hasher.finalize().into()))
}
``` [1](#0-0) 

No nonce, no NEAR contract address, no block height, and no caller identity is mixed into the hash. The hash is therefore a pure deterministic function of the foreign-chain request parameters and the extracted values.

**Contrast with `sign()` and `request_app_private_key()`**

For ordinary signature requests, `SignatureRequest` stores a `tweak` derived from `(predecessor_id, path)`:

```rust
pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
    let tweak = crate::kdf::derive_tweak(predecessor_id, path);
    ...
}
``` [2](#0-1) 

For CKD requests, `CKDRequest::new` similarly binds the request to `(predecessor, derivation_path)`. Neither binding exists for `verify_foreign_transaction`.

**`VerifyForeignTransactionRequest` also carries no caller identity**

`args_into_verify_foreign_tx_request` converts the user-supplied args into the stored request, discarding the `predecessor_account_id` that `check_request_preconditions` returns:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [3](#0-2) 

The stored key in `pending_verify_foreign_tx_requests` is therefore `(ForeignChainRpcRequest, DomainId, ForeignTxPayloadVersion)` — identical for any two callers submitting the same foreign-chain request.

**`respond_verify_foreign_tx` does not re-derive or re-verify the payload hash**

The contract verifies only that the supplied `response.payload_hash` is signed by the root key; it does not recompute the hash from the stored request or check that the hash corresponds to any particular set of extracted values:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
``` [4](#0-3) 

A valid `(payload_hash, signature)` pair produced in a previous signing session is therefore unconditionally accepted for any new pending request that matches the same `VerifyForeignTransactionRequest` key.

**MPC nodes use a zero tweak for foreign-tx signing**

`build_signature_request` in the node hardcodes `tweak: Tweak::new([0u8; 32])`, confirming the root key is used directly and the signature is not bound to any per-request derivation: [5](#0-4) 

---

### Impact Explanation

**High — Cross-chain replay / forged foreign-chain verification enabling invalid bridge execution**

A Byzantine attested participant (one node, well below the signing threshold) can observe a valid `(request, response)` pair from a previous `respond_verify_foreign_tx` call (all NEAR transactions are public). When the same foreign transaction is queried again — for example after a Bitcoin reorg places the transaction in a different block — the Byzantine participant can immediately replay the old response to `respond_verify_foreign_tx` before honest nodes complete a new signing session.

The waiting caller (e.g., a bridge contract) receives the old `VerifyForeignTransactionResponse` containing the stale `payload_hash` and signature. The `ForeignChainSignatureVerifier` in the SDK will confirm the signature is valid (it is — it was produced by the MPC network), but the `extracted_values` embedded in the hash reflect the old chain state. A bridge contract that uses the extracted block hash to gate a cross-chain transfer would accept an invalid or reorganized state, enabling double-spend or invalid bridge execution.

---

### Likelihood Explanation

**Medium-High.** The preconditions are:

1. A foreign transaction is queried via `verify_foreign_transaction` at least twice (common for bridge retry logic or after a reorg).
2. One attested MPC participant is Byzantine (below the signing threshold — a single compromised node suffices).
3. The Byzantine participant replays the old on-chain response before honest nodes complete a new signing round.

All three conditions are realistic in a production bridge deployment. Bitcoin reorgs are a known operational event; bridge contracts routinely re-query after reorgs. A single Byzantine node is the minimum adversary model for threshold systems.

---

### Recommendation

Include a unique, request-scoped context in `ForeignTxSignPayloadV1` so that each signing session produces a non-replayable hash. Concretely:

1. **Add the NEAR contract address** (`env::current_account_id()`) to `ForeignTxSignPayloadV1` so signatures are bound to a specific contract deployment.
2. **Add a per-request nonce** (e.g., the NEAR yield `data_id` / receipt ID assigned when the request is enqueued) so that even two requests for the same foreign transaction produce distinct hashes.
3. Alternatively, include the NEAR block height at which the request was accepted, providing a time-bound that prevents indefinite replay.

The analogous fix in the `sign()` path is the `tweak` derived from `(predecessor_id, path)` — a similar binding should be introduced for the foreign-tx path.

---

### Proof of Concept

**Setup**: MPC network running with ≥1 Byzantine attested participant. Bitcoin is supported as a foreign chain.

**Step 1 — First request (honest)**
```
Alice calls verify_foreign_transaction({
    request: BitcoinRpcRequest { tx_id: TX_X, confirmations: 6, extractors: [BlockHash] },
    domain_id: FOREIGN_TX_DOMAIN,
    payload_version: V1,
})
```
MPC nodes query Bitcoin RPC, observe `block_hash = H1`, sign:
```
payload = ForeignTxSignPayloadV1 { request: TX_X_request, values: [BlockHash(H1)] }
hash_1  = SHA-256(borsh(payload))
sig_1   = ECDSA_sign(root_key, hash_1)
```
`respond_verify_foreign_tx(request, { payload_hash: hash_1, signature: sig_1 })` is called on-chain. Alice receives `(hash_1, sig_1)`. The pending request is removed from state.

**Step 2 — Bitcoin reorg**
TX_X is reorganized into block `B2` with `block_hash = H2 ≠ H1`.

**Step 3 — Second request (replay)**
Bob calls `verify_foreign_transaction` with the identical `BitcoinRpcRequest { tx_id: TX_X, ... }`. The contract stores a new pending entry under the same key `VerifyForeignTransactionRequest { request: TX_X_request, domain_id, payload_version }`.

**Step 4 — Byzantine participant replays old response**
The Byzantine attested participant calls:
```
respond_verify_foreign_tx(
    request = VerifyForeignTransactionRequest { request: TX_X_request, ... },
    response = { payload_hash: hash_1, signature: sig_1 }   // ← old response
)
```

**Step 5 — Contract accepts**
- `assert_caller_is_attested_participant_and_protocol_active()` passes (Byzantine node is attested).
- `verify_ecdsa_signature(sig_1, hash_1, root_pk)` passes (signature is cryptographically valid).
- `resolve_yields_for(pending_verify_foreign_tx_requests, request, response)` resolves Bob's yield.

Bob receives `{ payload_hash: hash_1, signature: sig_1 }` — a response claiming TX_X is in block `H1`, which is no longer canonical. Bob's bridge contract calls `ForeignChainSignatureVerifier::verify_signature` with `expected_extracted_values = [BlockHash(H1)]`, which passes, and proceeds to execute a bridge action based on the stale/invalid chain state.     #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
