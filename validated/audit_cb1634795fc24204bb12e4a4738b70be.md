### Title
ECDSA-2PC `detach_private_scalar` Leaks Paillier Private Key in "Public" Blob, Enabling Private Scalar Recovery - (File: `src/cbmpc/api/ecdsa2pc.cpp`)

### Summary
`coinbase::api::ecdsa_2p::detach_private_scalar` is a public API intended to separate P1's private scalar share from its key blob for backup/restore flows. For P1 (role==0), the produced `out_public_key_blob` retains both the Paillier private key (`p`, `q`) and `c_key` (the Paillier encryption of `x_share`). Any party that obtains this "public" blob can decrypt `c_key` with the embedded Paillier private key to recover the private scalar share `x_share mod q`, fully breaking the MPC trust model.

### Finding Description

The ECDSA-2PC key blob structure (`key_blob_v1_t`) contains four secret fields for P1:

- `x_share` — the private scalar share
- `c_key` — Paillier encryption of `x_share mod N`
- `paillier` — the Paillier key, which for P1 includes the private factors `p` and `q` [1](#0-0) 

`detach_private_scalar` is supposed to strip the private scalar and return a "public" blob. It sets `pub.x_share = N` (out-of-range sentinel) to prevent signing, but it copies both `c_key` and the full `paillier` object (including `p`, `q`) verbatim into the output blob: [2](#0-1) 

The `paillier_t::convert` serializer writes `p` and `q` whenever `has_private == true`: [3](#0-2) 

The invariant enforced at deserialization time confirms that `c_key` is an encryption of `x_share mod N` under the Paillier key: [4](#0-3) 

So the "public" blob for P1 contains everything needed to recover `x_share`: the Paillier private key to decrypt, and `c_key` as the ciphertext.

### Impact Explanation

An attacker who obtains the `out_public_key_blob` produced by `detach_private_scalar` for P1 can:

1. Deserialize the blob and extract `paillier` (with `p`, `q`) and `c_key`.
2. Call `paillier.decrypt(c_key)` to recover `x_share mod N`.
3. Reduce modulo the curve order `q` to obtain the private scalar share `x_share mod q`.

This is a complete recovery of P1's private key share from a blob that the API names `out_public_key_blob` and that callers are expected to treat as non-secret (e.g., store in less-protected storage, pass to a backup service). The MPC security model is broken: a single-party compromise of the "public" blob suffices to recover the private scalar without any interaction with P2.

This fits the Critical impact tier: a shipped public API path lets an attacker recover a private scalar share.

### Likelihood Explanation

The `detach_private_scalar` / `attach_private_scalar` API pair is explicitly documented in the README as the mechanism for "application-managed backup and restore flows, including verifiable backup schemes such as publicly verifiable encryption (PVE)": [5](#0-4) 

The API header names the output `out_public_key_blob` and describes it as "a key blob with its private scalar removed": [6](#0-5) 

There is no warning in the header or in `SECURE_USAGE.md` that the ECDSA-2PC "public" blob still contains the Paillier private key. Callers following the documented backup pattern will naturally store this blob with reduced protection (e.g., in a database, alongside the PVE-encrypted scalar), making exploitation straightforward for any party that can read that storage.

### Recommendation

In `detach_private_scalar` for ECDSA-2PC, strip the Paillier private key from the output blob before serializing it. Replace the private Paillier key with a public-only copy (using `paillier_t::create_pub(N)`) so that `c_key` cannot be decrypted from the "public" blob:

```cpp
// Replace private Paillier key with public-only copy.
coinbase::crypto::paillier_t pub_paillier;
pub_paillier.create_pub(key.paillier.get_N().value());
pub.paillier = pub_paillier;
```

Additionally, add an explicit warning in the `ecdsa_2p.h` header and `SECURE_USAGE.md` that the ECDSA-2PC "public" blob differs from the ECDSA-MP case: until this fix is applied, it must still be treated as secret material.

error_t detach_private_scalar(mem_t key_blob, buf_t& out_public_key_blob, buf_t& out_private_scalar) {
  if (const error_t rv = coinbase::api::detail::validate_mem_arg_max_size(key_blob, "key_blob",
                                                                          coinbase::api::detail::MAX_OPAQUE_BLOB_SIZE))
    return rv;
  coinbase::mpc::ecdsa2pc::key_t key;
  error_t rv = deserialize_key_blob(key_blob, key);
  if (rv) return rv;

  curve_id cid;
  if (!from_internal_curve(key.curve, cid)) return coinbase::error(E_BADARG, "unsupported curve");
  if (cid == curve_id::ed25519) return coinbase::error(E_BADARG, "unsupported curve");

  // Variable-length big-endian encoding (may grow after refresh).
  out_private_scalar = key.x_share.to_bin();

  // Produce a v1-format blob with an invalid (out-of-range) scalar share so it is
  // rejected by sign/refresh APIs.
  key_blob_v1_t pub;
  pub.role = static_cast<uint32_t>(key.role);
  pub.curve = static_cast<uint32_t>(cid);
  pub.Q_compressed = key.Q.to_compressed_bin();
  pub.x_share = key.paillier.get_N().value();  // x_share == N is out of range
  pub.c_key = key.c_key;
  pub.paillier = key.paillier;
  out_public_key_blob = coinbase::convert(pub);
  return SUCCESS;
}
