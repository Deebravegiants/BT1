### Title
Unauthenticated P2P Peer Can Inject Arbitrary Transaction Hashes Into Storage, Corrupting `starknet_getTransactionByHash` RPC Responses — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary
`TransactionStreamFactory::parse_data_for_block` accepts the `transaction_hash` field from an unauthenticated P2P peer's `FullTransaction` message and stores it verbatim into the `transaction_hash_to_idx` table and `TransactionMetadata`. No recomputation or validation against the transaction body is performed. The RPC handler for `starknet_getTransactionByHash` queries this table directly, so a malicious peer can make the RPC return a transaction for a hash that does not match the transaction body, and make the real hash return `TRANSACTION_HASH_NOT_FOUND`.

### Finding Description

**Step 1 — Peer-supplied hash accepted without validation.**

In `parse_data_for_block`, the `transaction_hash` field from the protobuf `FullTransaction` is pushed directly into `block_body.transaction_hashes`: [1](#0-0) 

The explicit `TODO` at line 88 acknowledges this is unvalidated.

**Step 2 — Fake hash written to the `transaction_hash_to_idx` table.**

`write_to_storage` calls `append_body`, which calls `write_transactions`. This function uses the peer-supplied hash as the key in the lookup table and in `TransactionMetadata`: [2](#0-1) 

**Step 3 — RPC `starknet_getTransactionByHash` queries the corrupted table.**

The RPC handler resolves a transaction by looking up the caller-supplied hash in `transaction_hash_to_idx`: [3](#0-2) 

**Step 4 — No cross-validation against the header's `transaction_commitment`.**

The block header stores a `transaction_commitment` field (a Merkle/Poseidon commitment over the real transaction hashes): [4](#0-3) 

However, the header sync path (`HeaderStreamBuilder::parse_data_for_block`) only checks that the block number matches and that exactly one signature is present — it does **not** cryptographically verify the signature, and there is no post-sync step that verifies the stored transaction hashes against the stored `transaction_commitment`: [5](#0-4) 

The two sync streams (headers and transactions) are independent and merged; no reconciliation step exists.

### Impact Explanation

A malicious P2P peer sends a `FullTransaction` with `transaction_hash = H_fake` (any arbitrary `Felt`) while the real hash of the transaction body is `H_real`. After `append_body` commits:

- `starknet_getTransactionByHash(H_fake)` returns the transaction body — an authoritative-looking wrong answer (wrong hash → correct body).
- `starknet_getTransactionByHash(H_real)` returns `TRANSACTION_HASH_NOT_FOUND` — a real transaction is invisible.
- `starknet_getTransactionReceipt` and `starknet_getTransactionStatus` are equally affected since they use the same `get_transaction_idx_by_hash` lookup. [6](#0-5) 

This satisfies the allowed impact: **"High. RPC execution … returns an authoritative-looking wrong value."**

### Likelihood Explanation

Any node running P2P sync (`p2p_sync_client_config` enabled) is exposed. The attacker only needs to be a connectable P2P peer — no authentication, no stake, no operator privilege. The `Protocol::Transaction` SQMR channel is open to any peer the network manager connects to. [7](#0-6) 

### Recommendation

In `parse_data_for_block`, after receiving each `FullTransaction`, recompute the transaction hash from the transaction body using the chain-id-aware hash function and compare it to the peer-supplied `transaction_hash`. If they differ, treat the peer as bad (`ParseDataError::BadPeer`) and report it. The `transaction_commitment` in the stored block header should also be verified against the set of recomputed hashes before committing the body.

### Proof of Concept

1. Connect a custom P2P peer to a node running with `p2p_sync_client_config` enabled.
2. When the node queries `Protocol::Transaction` for block N, respond with a valid `FullTransaction` but set `transaction_hash` to an arbitrary `Felt` (e.g., `0xdeadbeef`).
3. After the body marker advances past block N, call `starknet_getTransactionByHash(0xdeadbeef)` on the node's RPC — it returns the transaction body.
4. Recompute the real hash from the stored transaction body; assert it differs from `0xdeadbeef`.
5. Call `starknet_getTransactionByHash(real_hash)` — it returns `TRANSACTION_HASH_NOT_FOUND`. [8](#0-7)
