### Title
Deposit Silently Consumed Without Refund on Participant Re-Attestation - (File: `crates/contract/src/lib.rs`)

### Summary

`submit_participant_info` is marked `#[payable]` and accepts any attached deposit, but when an existing participant re-submits their attestation (a routine hourly operation), the entire deposit-accounting block is skipped. Any NEAR tokens attached to such a call are permanently locked in the contract with no refund path.

### Finding Description

In `submit_participant_info`, after `add_participant` returns, the contract computes a guard:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check minimum deposit, compute cost, refund excess
}
```

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — i.e., the caller is already a current MPC participant **and** the insertion result is `UpdatedExistingParticipant` (not `NewlyInsertedParticipant`) — the entire deposit-handling block is bypassed. Because the function is `#[payable]`, the NEAR runtime has already accepted and credited any attached deposit to the contract account. With no refund scheduled and no panic to roll back the call, those tokens are permanently absorbed by the contract.

The analogous flaw in the reference report is `if (_dontMint) return;` — a flag-controlled early exit that skips the `creditsOf` update, silently consuming the user's payment. Here the mechanism is a conditional block that is simply not entered, with the same accounting consequence: value in, no credit out.

The re-submission path is the **normal operational path** for existing participants. The MPC node's `submit_remote_attestation` task fires on a 1-hour cadence and on every attestation-removal event. Operators are also explicitly instructed to call `submit_participant_info` manually when the automated node submission fails (e.g., after a TEE upgrade). The operator guide notes that first-time joiners must attach a deposit; an operator who follows the same pattern on a re-submission will silently lose those tokens.

### Impact Explanation

Any NEAR tokens attached to a `submit_participant_info` call by an existing participant are permanently locked in the contract. The contract's accounting invariant — that `#[payable]` functions refund excess deposits — is broken for this code path. The loss is bounded by the deposit the caller attaches, but the function imposes no upper bound on the deposit, and the NEAR runtime will accept arbitrarily large values.

This maps to the allowed Medium impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

The automated node path (`submit_remote_attestation`) does not attach a deposit, so the routine hourly re-submission is safe in practice. However:

1. The operator guide explicitly instructs operators to call `submit_participant_info` manually with `--deposit` when the node fails to submit. An operator who applies this instruction to a re-submission (not just a first-time join) will lose the deposit.
2. The function is `#[payable]` with no documentation warning that deposits are non-refundable on re-submissions, making the trap invisible to callers.
3. Any tooling or script that always attaches a deposit "to be safe" will silently drain funds on every re-attestation cycle.

### Recommendation

Add an unconditional excess-refund at the end of `submit_participant_info`, mirroring the pattern used in `propose_update` and `require_deposit`:

```rust
// After the attestation_storage_must_be_paid_by_caller block:
let attached = env::attached_deposit();
let cost = if attestation_storage_must_be_paid_by_caller {
    env::storage_byte_cost()
        .saturating_mul(env::storage_usage().saturating_sub(initial_storage) as u128)
} else {
    NearToken::from_yoctonear(0)
};
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
```

Alternatively, restructure the guard so the refund path is always reached regardless of whether storage cost must be paid by the caller.

### Proof of Concept

```rust
// Existing participant (already in tee_state) re-submits attestation with a deposit.
// Precondition: participant has previously called submit_participant_info successfully.

let ctx = VMContextBuilder::new()
    .signer_account_id(existing_participant_id.clone())
    .predecessor_account_id(existing_participant_id.clone())
    .attached_deposit(NearToken::from_near(1))   // 1 NEAR attached
    .build();
testing_env!(ctx);

let balance_before = env::account_balance(); // contract balance

// Re-submit: is_new_attestation = false, caller_is_not_participant = false
// => attestation_storage_must_be_paid_by_caller = false
// => deposit block skipped, no refund scheduled
contract.submit_participant_info(valid_attestation, tls_public_key).unwrap();

let balance_after = env::account_balance();
// balance_after == balance_before + 1 NEAR
// The participant's 1 NEAR is permanently locked in the contract.
assert!(balance_after > balance_before, "deposit silently consumed");
```

            // rather than underflow. Intentional asymmetry: we do not refund freed bytes
            // either — the caller already paid for the larger entry, and we'd rather
            // accept that asymmetry than open a refund path for payload-shrinking games.
            let storage_used = env::storage_usage().saturating_sub(initial_storage);
            let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
            let attached = env::attached_deposit();

                return Err(InvalidParameters::InsufficientDeposit {
                    attached: attached.as_yoctonear(),
                    required: cost.as_yoctonear(),
