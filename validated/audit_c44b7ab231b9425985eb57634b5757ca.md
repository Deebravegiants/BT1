### Title
Attacker Can Permanently Block `DeterministicStateInit` by Pre-Setting Account Contract via `UseGlobalContract` - (File: `runtime/runtime/src/deterministic_account_id.rs`)

### Summary

An unprivileged attacker can permanently prevent a deterministic account from being correctly initialized by front-running the `DeterministicStateInit` action. The attacker first creates the target deterministic account via a `Transfer`, then uses a malicious contract to send a `UseGlobalContract` receipt to it. This sets the account's `contract` field to a non-`None` value, causing `action_deterministic_state_init` to silently skip the entire `deploy_deterministic_account` call. All subsequent `DeterministicStateInit` calls will also do nothing, permanently leaving the account with the wrong contract and no initial data.

### Finding Description

`action_deterministic_state_init` in `runtime/runtime/src/deterministic_account_id.rs` guards the initialization path with a single check:

```rust
if account.contract().is_none() {
    // `uninit` -> `active` account state transition
    deploy_deterministic_account(
        state_update, account, account_id, &action.state_init, result, storage_usage_config,
    )?;
}
``` [1](#0-0) 

The intent is that `account.contract().is_none()` is `true` only for accounts in the `uninit` state (created via a bare `Transfer`). However, `UseGlobalContract` can be sent to any existing account by any contract, with no restriction on account type:

```rust
Action::UseGlobalContract(_) => {
    if account.is_none() {
        return Err(ActionErrorKind::AccountDoesNotExist { ... });
    }
}
``` [2](#0-1) 

`use_global_contract` sets `account.contract` to `AccountContract::Global(...)` or `AccountContract::GlobalByAccount(...)`:

```rust
account.set_contract(contract);
``` [3](#0-2) 

After this, `account.contract().is_none()` returns `false`, so `deploy_deterministic_account` is permanently skipped for all future `DeterministicStateInit` calls.

The `deploy_deterministic_account` function itself documents this pre-condition:

```
/// Pre-condition: The account must not have any contract data stored when this
/// is called. Otherwise, the storage usage calculations would fail to take
/// overwritten existing values into account.
``` [4](#0-3) 

**Attack path (unprivileged, public RPC):**

1. Attacker computes the deterministic account ID for a known `DeterministicAccountStateInit` (the ID is deterministic and public).
2. Attacker sends a `Transfer` to the deterministic account ID. Because it is a `NearDeterministicAccount`, `action_implicit_account_creation_transfer` creates it with `AccountContract::None`:

```rust
AccountType::NearDeterministicAccount => {
    *account = Some(create_deterministic_account(
        deposit,
        &apply_state.config.fees.storage_usage_config,
    ));
}
``` [5](#0-4) 

3. Attacker's deployed contract creates a promise targeting the deterministic account and appends a `UseGlobalContract` action (referencing any attacker-deployed global contract). This is a standard cross-contract call available to any NEAR contract.
4. The receipt executes: `use_global_contract` sets the account's `contract` field to the attacker's global contract.
5. The legitimate `DeterministicStateInit` action is now permanently blocked: `account.contract().is_none()` is `false`, so `deploy_deterministic_account` is never called. The account is stuck with the wrong contract and no initial data. Repeated `DeterministicStateInit` calls silently do nothing and refund the deposit.

### Impact Explanation

The corrupted protocol value is the deterministic account's `contract` field and `storage_usage` in the trie state. The account is permanently stuck with an attacker-chosen global contract and zero initial data, instead of the intended contract and data from `DeterministicAccountStateInit`. Because the idempotency guard (`account.contract().is_none()`) is now permanently `false`, no future `DeterministicStateInit` can recover the account. Any application relying on the deterministic account being correctly initialized (e.g., sharded contracts per NEP-616) is permanently broken for that account ID.

**Impact: Medium** — permanent corruption of a specific account's runtime state; does not affect consensus or other accounts.

### Likelihood Explanation

The attacker needs only: (a) knowledge of the target `DeterministicAccountStateInit` (public/computable), (b) a small NEAR balance for the Transfer and gas, and (c) a deployed contract and global contract (both permissionless). No validator or privileged access is required. The attack can be executed entirely via public RPC transactions.

**Likelihood: Medium** — requires modest setup but is straightforward for any motivated unprivileged user who knows the target state init.

### Recommendation

Replace the `account.contract().is_none()` guard with a dedicated, tamper-proof marker that is set only by `DeterministicStateInit` itself. For example, store a special trie key (e.g., `TrieKey::DeterministicStateInitialized { account_id }`) that is written atomically during `deploy_deterministic_account` and checked instead of `account.contract().is_none()`. This ensures that `UseGlobalContract` (or any other action that sets the contract field) cannot spoof the "already initialized" state.

### Proof of Concept

1. Compute `det_account = derive_near_deterministic_account_id(&state_init)` for a known `state_init`.
2. Submit `Transfer(1 yoctoNEAR)` to `det_account` — creates it with `AccountContract::None`.
3. Deploy a global contract `G` and a malicious contract `M` (both permissionless).
4. Call `M.attack(det_account)` which executes:
   ```
   promise_batch_create(det_account)
   promise_batch_action_use_global_contract(promise_idx, G)
   ```
   This sends a `UseGlobalContract(G)` receipt to `det_account`, setting its `contract` to `AccountContract::GlobalByAccount(G)`.
5. Now submit the legitimate `DeterministicStateInit` action targeting `det_account`.
6. In `action_deterministic_state_init`: `account.contract().is_none()` → `false` → `deploy_deterministic_account` is skipped.
7. The account is permanently stuck with contract `G` and no initial data. All future `DeterministicStateInit` calls also silently do nothing.
