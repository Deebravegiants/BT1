### Title
NFT Royalty Payment Silently Bypassed via Integer Division Truncation — (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary
`compute_royalty_amount()` uses sequential integer floor-division that truncates to zero when the offered fungible amount is below a threshold determined by the royalty percentage and split count. When the computed royalty is zero, `make_nft1_offer()` silently omits the royalty announcement assertion and skips the royalty coin spend entirely, allowing an NFT to transfer without paying the creator's on-chain royalty.

### Finding Description
`compute_royalty_amount()` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

The two sequential `//` operations cause truncation to zero whenever:

```
abs(offered_amount) // royalty_split * percentage < 10000
```

The project's own test suite explicitly documents this:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

When `compute_royalty_amount` returns 0, `make_nft1_offer()` stores a `CreateCoin` with `amount=0` in `royalty_payments`. Two downstream guards then silently drop the royalty entirely:

1. **Announcement guard** — only asserts the royalty puzzle announcement when `p.amount > 0`, so no on-chain enforcement of the royalty payment is added to the spend bundle: [3](#0-2) 

2. **Spend guard** — skips the royalty coin spend entirely when the total royalty sum is zero: [4](#0-3) 

The same truncation formula is also present in `royalty_calculation()`, the static method used by the `nft_calculate_royalties` RPC and CLI display, meaning the user-facing royalty summary also shows `0` and does not alert the seller: [5](#0-4) 

### Impact Explanation
An unprivileged buyer can craft an offer whose fungible amount is small enough that the royalty truncates to zero. The resulting spend bundle contains no royalty coin, no royalty announcement assertion, and no puzzle-level enforcement of the creator's royalty. The NFT singleton transfers to the buyer, the seller receives the offered mojos, and the royalty recipient (the NFT creator) receives nothing — a silent bypass of the offer/trade settlement royalty obligation.

This matches the allowed High impact: *"Corruption of … offer/trade settlement state … with direct security impact"* and *"payout redirection"* (royalty diverted from creator to zero).

### Likelihood Explanation
The threshold below which royalty truncates to zero is `10000 / percentage` mojos. For a 1 % royalty NFT (100 basis points) the threshold is 100 mojos; for 0.1 % (10 basis points) it is 1000 mojos. These are tiny amounts in XCH terms, but:

- CAT tokens with low per-unit value can have economically meaningful amounts below the threshold.
- The `royalty_split` divisor (number of NFTs requested in one offer) amplifies truncation: requesting N NFTs divides the per-NFT royalty by N before the percentage multiplication, pushing the truncation threshold N× higher.
- Automated takers or bots that accept any matching offer are susceptible without human review.

### Recommendation
Add a guard in `make_nft1_offer()` (or inside `compute_royalty_amount()`) that rejects the offer construction when a non-zero royalty percentage would produce a zero royalty amount, analogous to the existing guard that rejects royalties ≥ 100 %:

```python
if percentage > 0 and royalty == 0:
    raise ValueError(
        f"Offered amount too small: royalty truncates to 0 mojos "
        f"(offered {abs(offered_amount)} mojos, {percentage} bps, split {royalty_split})"
    )
``` [1](#0-0) 

### Proof of Concept
1. Mint an NFT with `royalty_percentage = 100` (1 %).
2. As an unprivileged buyer, call `create_offer_for_ids` offering **50 mojos** of XCH for the NFT.
3. `compute_royalty_amount(-50, 1, 100)` → `50 // 1 * 100 // 10000` → `5000 // 10000` → **0**.
4. `make_nft1_offer()` skips the royalty announcement assertion (line 853 filter `if p.amount > 0`) and skips the royalty coin spend (line 911 `continue`).
5. The resulting spend bundle is valid on-chain: the NFT transfers to the buyer, the seller receives 50 mojos, and the NFT creator receives **0 mojos** in royalty despite the 1 % on-chain royalty setting. [6](#0-5) [4](#0-3)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
