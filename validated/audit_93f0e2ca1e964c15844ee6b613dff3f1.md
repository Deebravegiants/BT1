### Title
Unconstrained `fee_proposal_fri` During Bootstrap/Transition Period Allows Malicious Proposer to Permanently Distort L2 Gas Price - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `is_proposal_init_valid` function in `validate_proposal.rs` intentionally skips the `fee_proposal_fri` bounds check when `fee_actual` is `None`. This `None` state occurs during the first `window_size` (10) blocks after genesis **and** during the entire `window_size`-block window following a V0_14_2→V0_14_3 network upgrade (because pre-V0_14_3 blocks carry `fee_proposal_fri = None`, which causes `compute_fee_actual` to return `None`). During either window, a malicious block proposer can set `fee_proposal_fri` to any arbitrary value (e.g., `u128::MAX`). Validators accept the block unconditionally. Once the window fills with extreme values, `fee_actual` becomes extreme, `calculate_next_l2_gas_price_for_fin` uses it as a floor, and `l2_gas_price` is permanently driven to `u128::MAX`, making every future transaction prohibitively expensive.

---

### Finding Description

**Root cause — skipped bounds check when `fee_actual` is `None`:** [1](#0-0) 

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
```

The `if let` pattern silently drops the check when `fee_actual` is `None`. No rejection, no warning, no fallback bound.

**When `fee_actual` is `None`:**

`compute_fee_actual` returns `None` whenever any of the `window_size` prior heights is absent from the window **or** carries `fee_proposal_fri = None` (pre-V0_14_3 blocks): [2](#0-1) 

```rust
for source_height in (start..height.0).map(BlockNumber) {
    match fee_proposals_window.get(&source_height) {
        Some(Some(price)) => window.push(*price),
        Some(None) | None => {
            warn!("...");
            return None;   // ← any None entry collapses the whole result
        }
    }
}
```

`window_size = 10` (from versioned constants): [3](#0-2) 

This means the `None` state persists for exactly 10 blocks after the V0_14_2→V0_14_3 upgrade, because the window still contains pre-upgrade blocks with `fee_proposal_fri = None`.

**Honest proposer fallback vs. malicious deviation:**

An honest proposer freezes at `l2_gas_price` when `fee_actual` is `None`: [4](#0-3) 

```rust
let Some(fee_actual) = fee_actual else {
    warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
    return self.l2_gas_price;
};
```

A malicious proposer ignores this and sets `fee_proposal_fri = u128::MAX`. The validator's `is_proposal_init_valid` does not reject it.

**Extreme value recorded permanently:**

On decision, `finalize_decision` calls `record_fee_proposal(height, init.fee_proposal_fri)`: [5](#0-4) 

The extreme value is now in the `fee_proposals_window`. After 10 such blocks, `compute_fee_actual` returns `u128::MAX` as the median.

**`fee_actual` used as a floor for `l2_gas_price`:** [6](#0-5) 

> `fee_market` for EIP-1559-style base-fee adjustment, which receives `fee_actual` as a floor. [7](#0-6) 

`calculate_next_l2_gas_price_for_fin` receives `fee_actual` as a floor parameter. With `fee_actual = u128::MAX`, `l2_gas_price` is forced to `u128::MAX`.

**`fee_proposal_fri` is bound to `ProposalCommitment`:** [8](#0-7) 

```rust
pub(crate) fn proposal_commitment_from(
    partial: PartialBlockHash,
    fee_proposal: Option<GasPrice>,
) -> ProposalCommitment {
    let Some(fee_proposal) = fee_proposal else {
        return ProposalCommitment(partial.0);
    };
    ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
}
```

Validators compute `batcher_block_commitment = proposal_commitment_from(partial_block_hash, fee_proposal)` using the `fee_proposal_fri` from the received `ProposalInit`: [9](#0-8) 

Since validators use the proposer's `fee_proposal_fri` to compute the commitment, they will agree with the malicious proposer's commitment and vote for the block. The extreme value is permanently committed to the chain.

**Recovery is practically impossible:**

The per-block margin is `fee_proposal_margin_ppt = 2` (0.2%): [10](#0-9) 

```rust
let lower = u128::try_from(fee_actual_u256 * denom / scaled).unwrap_or(0);
```

With `fee_actual = u128::MAX`, `lower ≈ u128::MAX × 0.998`. The gas price can only decrease by 0.2% per block. Recovering from `u128::MAX` to a normal value would require an astronomical number of blocks.

---

### Impact Explanation

Once the attack succeeds, `l2_gas_price` is permanently set to `u128::MAX`. Every subsequent `ProposalInit` must carry `l2_gas_price_fri = u128::MAX` (enforced by the exact-match check at line 312–321 of `validate_proposal.rs`). All transaction fee computations use this price, making every transaction prohibitively expensive and effectively halting the network. This is a **Critical** impact: incorrect fee/gas with severe economic impact.

---

### Likelihood Explanation

The attack window is exactly 10 blocks:
- **Bootstrap (genesis):** The first 10 blocks have `fee_actual = None` unconditionally.
- **V0_14_2→V0_14_3 upgrade:** The first 10 post-upgrade blocks have `fee_actual = None` because the window still contains pre-upgrade `None` entries.

To set the median to `u128::MAX`, the attacker needs to control at least 6 of the 10 blocks (for even `window_size`, the median is the average of the 5th and 6th sorted values). In a decentralized consensus with rotating proposers, this requires controlling ≥60% of the validator set during the transition window — a significant but not impossible threshold for a well-resourced attacker, especially at genesis when the validator set is small.

---

### Recommendation

When `fee_actual` is `None`, fall back to `l2_gas_price_fri` (the current gas price, which is already validated by the exact-match check) as the reference for bounds enforcement. This is consistent with the honest proposer's behavior:

```rust
// Use l2_gas_price as fallback when fee_actual is None (bootstrap / transition).
let effective_fee_actual = proposal_init_validation
    .fee_actual
    .unwrap_or(proposal_init_validation.l2_gas_price_fri);

if let Some(fee_proposal) = init_proposed.fee_proposal_fri {
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        effective_fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
```

This ensures that even during the bootstrap/transition period, `fee_proposal_fri` cannot deviate more than 0.2% from the current `l2_gas_price`, preventing extreme value injection.

```rust
pub(crate) fn fee_proposal_bounds(fee_actual: GasPrice, margin_ppt: u128) -> (u128, u128) {
    let denom = U256::from(PPT_DENOMINATOR);
    let scaled = denom + U256::from(margin_ppt);
    let fee_actual_u256 = U256::from(fee_actual.0);
    let upper = u128::try_from(fee_actual_u256 * scaled / denom).unwrap_or(u128::MAX);
    let lower = u128::try_from(fee_actual_u256 * denom / scaled).unwrap_or(0);
    (lower, upper)
}
```
