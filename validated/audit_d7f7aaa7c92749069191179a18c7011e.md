### Title
Missing Post-Aggregation Integrity Check in Non-PV CKD Coordinator Allows Single Byzantine Participant to Silently Corrupt Derived Key — (`crates/threshold-signatures/src/confidential_key_derivation/protocol.rs`)

### Summary

The `CKDAppPublicKey::AppPublicKey` production path dispatches to `protocol::ckd` (from `protocol.rs`), whose coordinator aggregates participant contributions with **no post-aggregation validation**. A single Byzantine participant can send `CKDOutput{big_y: G1::identity(), big_c: G1::identity()}`, causing the coordinator to produce a `(big_y, big_c)` pair that does not satisfy `big_c − app_sk·big_y = msk·H(pk‖app_id)`. The contract cannot detect this for the non-PV variant (no G2 key available for pairing checks), so the client silently receives a wrong confidential key.

The `AppPublicKeyPV` path dispatches to `protocol_pv::ckd_pv`, which **does** call `aggregated_output_check` and would catch this. The non-PV path has no equivalent guard.

**Why the contract cannot compensate for the non-PV path:**

For `CKDAppPublicKey::AppPublicKey`, only a G1 point is available. Pairing-based verification requires a G2 component. The contract has no mechanism to verify the non-PV `(big_y, big_c)` output and returns it directly to the caller.

### Impact Explanation

A Byzantine participant sends `CKDOutput::new(G1Projective::identity(), G1Projective::identity())`. Adding the identity element to the running sums is a no-op, so the aggregated output is missing that participant's Lagrange-weighted share `(λ_i·Y_i, λ_i·C_i)`. The resulting `big_c − app_sk·big_y` equals `Σ_{j≠i} λ_j·x_j·H(pk‖app_id)`, which is not `msk·H(pk‖app_id)`. The client receives a silently wrong confidential key with no error signal. This constitutes unauthorized CKD output — the correct key is denied and a forged key is delivered.

---

### Likelihood Explanation

The node selects exactly `threshold` participants per CKD session: [6](#0-5) 

A single Byzantine participant among `t` selected nodes is strictly below the signing threshold. Any MPC node that is compromised (e.g., via a TEE exploit, key exfiltration, or software supply-chain attack) can trigger this on every `AppPublicKey`-variant CKD request it participates in. The `AppPublicKey` variant is a live, non-default-feature production path.

---

### Recommendation

Apply the same `aggregated_output_check` pattern from `protocol_pv.rs` to `protocol.rs`. Since the non-PV variant only has a G1 app key, a pairing check is not directly available. The fix options are:

1. **Require callers to always use the PV variant** (`AppPublicKeyPV`) and deprecate `AppPublicKey`.
2. **Add a Schnorr/DLEQ proof** from each participant proving that their submitted `(norm_big_y, norm_big_c)` is consistent with their committed key share, verified by the coordinator before aggregation.
3. **Add a coordinator-side re-derivation check**: after aggregation, verify the output against the known public key using an alternative consistency equation that does not require G2.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`
fn aggregated_output_check(
    output: &CKDOutput,
    app_pk: &PublicVerificationKey,
    public_key: &VerifyingKey,
    hash_point: &ElementG1,

    if !check_valid_point_g1(output.big_c.into()) || !check_valid_point_g1(output.big_y.into()) {
        return false;
    }
    multi_miller_loop(&[
        (output.big_c, -ElementG2::generator()),
        (output.big_y, app_pk.pk2),
        (*hash_point, public_key.to_element()),
    ])
}
