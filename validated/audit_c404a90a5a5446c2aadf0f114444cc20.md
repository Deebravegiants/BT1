Audit Report

## Title
Reward tokens permanently locked in `KernelDepositPool` when `totalKernelStaked` drops to zero during an active reward window - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary

`KernelDepositPool` implements a Synthetix-style staking rewards model. When `totalKernelStaked` reaches zero during an active reward distribution window, `rewardPerToken()` stops accumulating rewards to avoid division by zero. All reward tokens that would have been distributed from that point until `finishAt` are permanently locked in the contract with no on-chain recovery mechanism. The contract's own NatSpec explicitly acknowledges this risk, confirming it is a known operational scenario with zero on-chain enforcement.

## Finding Description

`KernelDepositPool` distributes `rewardsToken` to KERNEL stakers over a `duration`-length window set by `notifyRewardAmount`. The `updateReward` modifier snapshots `rewardPerToken()` on every state-changing call:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L232-242
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

`rewardPerToken()` guards against division by zero:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // rewards stop accruing
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

When the last staker calls `initiateWithdrawal`, `totalKernelStaked` is decremented to zero and `updatedAt` is set to `block.timestamp`. From that moment until `finishAt`, `rewardRate × (finishAt − updatedAt)` worth of reward tokens accumulate in the contract but are never credited to any account:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L325-326
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
```

The guard in `notifyRewardAmount` (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new reward period with no stakers — it does not prevent `totalKernelStaked` from dropping to zero after a reward period has already started. There is no `rescue`, `sweep`, or admin withdrawal function anywhere in the contract. The contract's own NatSpec at lines 18–22 explicitly acknowledges this:

> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract. In this deployment, we're avoiding this issue by ensuring there are always some tokens staked..."*

The stated mitigation is purely operational with zero on-chain enforcement.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Any reward tokens allocated to a distribution window but not yet emitted at the moment `totalKernelStaked` hits zero are permanently locked in `KernelDepositPool`. The magnitude equals `rewardRate × (finishAt − t_zero)` where `t_zero` is the timestamp at which the last staker unstaked. For a 30-day window with a large reward allocation, this can represent a significant fraction of the total reward budget. This maps directly to the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation

**Medium.** The scenario requires all stakers to withdraw during an active reward window. This is realistic because:
- `withdrawalDelay` is capped at `MAX_WITHDRAWAL_DELAY = 30 days`, the same order of magnitude as a typical reward `duration`. A coordinated or organic mass exit is plausible.
- There is no minimum-stake floor, no "dead shares" equivalent, and no on-chain guard preventing `totalKernelStaked` from reaching zero mid-window.
- The contract itself documents the risk at lines 18–22, confirming the developers are aware it is a real operational scenario.
- Any single staker who is the last remaining staker can unilaterally trigger this by calling `initiateWithdrawal` — no coordination required if only one staker remains.

## Recommendation

1. **On-chain guard in `initiateWithdrawal`**: revert (or redirect remaining rewards) if the withdrawal would reduce `totalKernelStaked` to zero while `block.timestamp < finishAt`.
2. **Admin rescue function**: add a permissioned function that, when `totalKernelStaked == 0 && block.timestamp < finishAt`, resets `finishAt = block.timestamp` and transfers the unallocated balance (`rewardRate × (old_finishAt − block.timestamp)`) to a treasury address.
3. **Minimum permanent stake**: require the admin to maintain a non-withdrawable "seed" stake so `totalKernelStaked` can never reach zero.

## Proof of Concept

```solidity
// 1. Admin calls notifyRewardAmount for a 30-day window of 30_000e18 rewardsToken
//    rewardRate = 30_000e18 / 30 days ≈ 11.57e18 per second
//    finishAt   = block.timestamp + 30 days

// 2. Alice stakes 1_000e18 KERNEL (only staker)

// 3. After 15 days (50% of window elapsed), Alice calls initiateWithdrawal(1_000e18)
//    → updateReward snapshots rewardPerToken correctly for Alice's 15-day share
//    → totalKernelStaked becomes 0
//    → updatedAt = block.timestamp (day 15)
//    → Alice earned ≈ 15_000e18 rewardsToken (correct)

// 4. From day 15 to day 30, rewardPerToken() always returns rewardPerTokenStored
//    because totalKernelStaked == 0.

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
