### Title
Silent Transaction Drop in FIFO Mempool Mode Returns False Success to Caller - (`crates/apollo_mempool/src/communication.rs`)

### Summary

In FIFO/Echonet mode, when `fetch_and_update_tx_block_metadata` fails for a transaction, the mempool's `add_tx` handler silently drops the transaction and returns `Ok(())`. This propagates back through the gateway as a success response, so the user receives a valid transaction hash — but the transaction is never added to the mempool and will never be sequenced.

### Finding Description

The `MempoolCommunicationWrapper::add_tx` function in `crates/apollo_mempool/src/communication.rs` contains the following logic:

```rust
pub(crate) async fn add_tx(
    &mut self,
    args_wrapper: AddTransactionArgsWrapper,
) -> MempoolResult<()> {
    if self.mempool.is_fifo() {
        let tx_hash = args_wrapper.args.tx.tx_hash();
        if !self.fetch_and_update_tx_block_metadata(tx_hash).await {
            warn!("Failed to fetch tx block metadata for tx {}, skipping transaction", tx_hash);
            return Ok(());   // ← silent drop, no error returned
        }
    }
    self.mempool.add_tx(args_wrapper.args.clone())?;
    ...
``` [1](#0-0) 

When `fetch_and_update_tx_block_metadata` returns `false` (a failure), the function returns `Ok(())` without calling `self.mempool.add_tx(...)`. The transaction is silently discarded.

The gateway's `add_tx_inner` treats any `Ok(())` from the mempool client as a successful submission:

```rust
let mempool_client_result = self.mempool_client.add_tx(add_tx_args).await;
match mempool_client_result_to_deprecated_gw_result(&tx_signature, mempool_client_result) {
    Ok(()) => {}   // ← no error, gateway returns success
    Err(e) => { ... return Err(e); }
};
// ...
Ok(gateway_output)   // ← transaction hash returned to user
``` [2](#0-1) 

The full admission path is:

1. User submits via `apollo_http_server` → `add_tx` / `add_rpc_tx`
2. HTTP server calls `gateway_client.add_tx(gateway_input)` [3](#0-2) 
3. Gateway performs stateless + stateful validation, then calls `mempool_client.add_tx(add_tx_args)` [4](#0-3) 
4. Mempool communication wrapper silently drops the tx and returns `Ok(())` [5](#0-4) 
5. Gateway returns `Ok(gateway_output)` with the transaction hash to the user

The transaction never enters the mempool pool or queue, so it is never picked up by the batcher and never sequenced.

### Impact Explanation

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

A valid, fully-validated transaction is silently rejected at the mempool boundary. The user receives a transaction hash indicating success, but the transaction will never be included in any block. This is a direct analog to the LES bug: an alternative code path (FIFO mode) omits the forwarding step that the primary path performs, and the caller receives a misleading success response.

### Likelihood Explanation

The condition is triggered whenever:
1. The node is configured in FIFO/Echonet mode (`BehaviorMode::Echonet`), and
2. `fetch_and_update_tx_block_metadata` fails for any reason (e.g., the metadata service is unavailable, a network partition, or a transient error).

Any user submitting a transaction to a node in Echonet mode during a metadata-fetch failure will be silently affected. The `warn!` log is the only observable signal; no error is surfaced to the caller.

### Recommendation

Replace the silent `return Ok(())` with a propagated error so the caller is informed that the transaction was not admitted:

```rust
if !self.fetch_and_update_tx_block_metadata(tx_hash).await {
    warn!("Failed to fetch tx block metadata for tx {}, rejecting transaction", tx_hash);
    return Err(MempoolError::BlockMetadataFetchFailed { tx_hash });
}
```

This ensures the gateway propagates the failure back to the user rather than returning a false success, matching the behavior of the non-FIFO path.

### Proof of Concept

1. Deploy the sequencer node with `BehaviorMode::Echonet` (FIFO mode).
2. Make `fetch_and_update_tx_block_metadata` return `false` (e.g., by making the metadata service unreachable).
3. Submit a valid invoke transaction via `POST /gateway/add_transaction` or `starknet_addInvokeTransaction`.
4. Observe: the HTTP response returns HTTP 200 with a valid `transaction_hash`.
5. Query the transaction status — it will never appear in any block or pending state, because it was never inserted into the mempool. [1](#0-0) [2](#0-1) [6](#0-5)

async fn add_tx_inner(
    app_state: AppState,
    headers: HeaderMap,
    tx: RpcTransaction,
) -> HttpServerResult<Json<GatewayOutput>> {
    let gateway_input: GatewayInput = GatewayInput { rpc_tx: tx, message_metadata: None };
    // Wrap the gateway client interaction with a tokio::spawn as it is NOT cancel-safe.
    // Even if the current task is cancelled, e.g., when a request is dropped while still being
    // processed, the inner task will continue to run.
    let region = headers
        .get(CLIENT_REGION_HEADER)
        .and_then(|region| region.to_str().ok())
        .unwrap_or("N/A")
        .to_string();
    let add_tx_result = tokio::spawn(
        async move {
            let add_tx_result = app_state.gateway_client.add_tx(gateway_input).await.map_err(|e| {
                debug!("Error while adding transaction: {}", e);
                HttpServerError::from(Box::new(e))
            });
            record_added_transactions(&add_tx_result, &region);
            add_tx_result
        }
        .instrument(tracing::Span::current()),
    )
    .await
    .expect("Should be able to get add_tx result");

    Ok(Json(add_tx_result?))
