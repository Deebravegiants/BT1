### Title
Reward Tokens Permanently Locked in KernelDepositPool When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool` distributes `rewardsToken` to KERNEL stakers using a Synthetix-style accumulator. When `totalKernelStaked` drops to zero mid-period, `rewardPerToken()` freezes its return value while `updatedAt` still advances, causing all reward tokens that should have been distributed during the zero-staking window to be permanently unattributable and stuck in the contract. There is no admin or emergency function to recover them.

### Finding Description
`rewardPerToken()` short-circuits to `rewardPerTokenStored` whenever `totalKernelStaked == 0`:

```solidity
// KernelDepositPool.sol#L408-L413
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;  // frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

The `updateReward` modifier always advances `updatedAt` regardless:

```solidity
// KernelDepositPool.sol#L232-L241
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();   // advances even when totalKernelStaked == 0
    ...
}
```

Any staker can call `initiateWithdrawal()`, which immediately decrements `totalKernelStaked`:

```solidity
// KernelDepositPool.sol#L325-L326
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
```

If `totalKernelStaked` reaches zero during an active reward period, the following happens:
- `rewardPerTokenStored` stays frozen at its current value.
- `updatedAt` advances to `lastTimeRewardApplicable()` on every subsequent interaction.
- When staking resumes, the new staker's `userRewardPerTokenPaid` is set to the frozen `rewardPerTokenStored`.
- All `rewardRate * elapsed_zero_staking_time` tokens that accumulated in the contract during the gap are never attributed to anyone and cannot be recovered.

The contract has no sweep, rescue, or recovery function for `rewardsToken`. The only admin functions are `setRewardsDuration`, `notifyRewardAmount`, `setWithdrawalDelay`, and `setMaxNumberOfWithdrawalsPerUser` — none of which can retrieve stranded reward tokens.

The protocol's own NatSpec acknowledges this risk:

```
// KernelDepositPool.sol#L18-L22
* @dev If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards
*      for that period will stay locked in the contract.
```

The guard in `notifyRewardAmount` only prevents starting a new period with zero stakers; it does not prevent stakers from withdrawing to zero after a period has already started.

### Impact Explanation
Every `rewardRate * T` tokens (where `T` is the duration of the zero-staking window) are permanently frozen inside `KernelDepositPool`. They cannot be claimed by any user and cannot be recovered by any admin. This is a permanent freezing of unclaimed yield.

### Likelihood Explanation
The trigger is a normal, permissionless user action: calling `initiateWithdrawal()`. A single staker who holds 100% of `totalKernelStaked` can reduce it to zero unilaterally. In a low-participation scenario (e.g., early in a reward epoch, or after a large coordinated exit), this is realistic. The protocol's mitigation is purely operational ("ensure tokens are always staked"), not enforced by code.

### Recommendation
Add a token-rescue function restricted to `DEFAULT_ADMIN_ROLE` that can transfer any surplus `rewardsToken` balance (i.e., `rewardsToken.balanceOf(address(this))` minus the sum of all `rewards[user]` entries) to a treasury address. Alternatively, track the "unallocated" reward amount explicitly and allow it to be rolled into the next reward period via `notifyRewardAmount`.

### Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` while `totalKernelStaked > 0`. `rewardRate` is set, `finishAt = block.timestamp + duration`.
2. The sole staker calls `initiateWithdrawal(totalKernelStaked)`. `totalKernelStaked` becomes 0. The `updateReward` modifier fires: `rewardPerTokenStored` stays frozen, `updatedAt` advances to `block.timestamp`.
3. Time passes. `rewardRate * elapsed` tokens accumulate in the contract but `rewardPerToken()` keeps returning the frozen value.
4. A new staker calls `stake(1)`. `updateReward` fires again: `rewardPerTokenStored` is still frozen, `updatedAt` advances to now. The new staker's `userRewardPerTokenPaid` equals `rewardPerTokenStored`.
5. All future rewards are calculated from the current frozen baseline. The tokens that accrued during step 3 are permanently unattributable.
6. `rewardsToken.balanceOf(address(this))` exceeds the sum of all claimable `rewards[user]` by exactly `rewardRate * T`. No function exists to recover the difference. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
