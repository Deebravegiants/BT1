### Title
MPC Node Indexer Silently Drops All Signing/CKD/ForeignTx Requests Contained in Multi-Action Receipts - (File: `crates/node/src/indexer/handler.rs`, `crates/chain-gateway/src/event_subscriber/streamer/block_processor.rs`)

---

### Summary

Both the legacy MPC node indexer and the new chain-gateway indexer contain a `try_extract_function_call_args` function that unconditionally returns `None` for any NEAR receipt that carries more than one action. Because NEAR allows a transaction or cross-contract call to bundle multiple actions into a single receipt, any `sign`, `request_app_private_key`, or `verify_foreign_transaction` call that arrives inside a multi-action receipt is silently invisible to every MPC node. The contract accepts and enqueues the request (creating a yield), but no node ever picks it up, so the request is guaranteed to time out and fail — permanently breaking the request lifecycle for that call pattern.

---

### Finding Description

`try_extract_function_call_args` in both indexer implementations enforces a hard `actions.len() != 1` guard:

```rust
// crates/node/src/indexer/handler.rs  lines 281-300
fn try_extract_function_call_args(receipt: &ReceiptView) -> Option<(&FunctionArgs, &String)> {
    let ReceiptEnumView::Action { ref actions, .. } = receipt.receipt else {
        return None;
    };
    if actions.len() != 1 {   // ← silently drops any multi-action receipt
        return None;
    }
    ...
}
```

The identical guard exists in the chain-gateway:

```rust
// crates/chain-gateway/src/event_subscriber/streamer/block_processor.rs  lines 191-210
fn try_extract_function_call_args(receipt: &ReceiptView) -> Option<(&FunctionArgs, &String)> {
    ...
    if actions.len() != 1 {   // ← same silent drop
        return None;
    }
    ...
}
```

This function is the sole entry point for extracting `sign`, `request_app_private_key`, and `verify_foreign_transaction` calls from the NEAR block stream. When it returns `None`, the outer loop in `handle_block` (lines 173–222 of `handler.rs`) skips the receipt entirely — no `SignatureRequestFromChain`, `CKDRequestFromChain`, or `VerifyForeignTxRequestFromChain` is ever emitted for that block.

In NEAR, a receipt can legitimately contain multiple actions. A smart contract (e.g., a bridge or aggregator) can construct a receipt that pairs a `Transfer` with a `FunctionCall(sign)`, or that batches two `FunctionCall` actions targeting the MPC contract. The NEAR runtime executes all actions in the receipt and the MPC contract successfully enqueues the yield — but every MPC node's indexer discards the receipt without logging a warning, so no node ever calls `respond`. The yield then expires after ~200 blocks, `return_signature_and_clean_state_on_success` fires with `Err(PromiseError::Failed)`, and `fail_on_timeout` panics the caller's transaction.

The same silent-drop path also applies to the **completion** side: line 225 of `handler.rs` calls `try_extract_function_call_args` again to detect `return_signature_and_clean_state_on_success` receipts. If that callback receipt somehow arrives in a multi-action context, completed signatures would also be missed, causing nodes to redundantly re-sign already-completed requests.

---

### Impact Explanation

**Medium.** The impact is request-lifecycle corruption: a class of valid, contract-accepted signing requests is permanently unserviceable by the MPC network. The contract's on-chain state diverges from what the nodes observe — the contract holds a live yield that no node will ever resume. Every such request is guaranteed to time out, breaking the safety invariant that "a successfully submitted sign request will eventually be answered or explicitly rejected by the network." Bridge contracts or aggregators that batch a `Transfer` + `FunctionCall(sign)` into one receipt, or that issue two `sign` calls in a single receipt, will find their signing requests silently dropped with no diagnostic output from any node.

The deposit (1 yoctonear minimum) is refunded on timeout, so direct fund loss is negligible. However, the **signed payload is never produced**, which in a bridge context means the inbound cross-chain transfer is permanently stuck — the bridge cannot release funds on the destination chain. This constitutes a production safety/accounting invariant break without requiring any operator misconfiguration.

---

### Likelihood Explanation

**Medium.** Any smart contract that calls `sign` as part of a multi-action receipt triggers this path. Concrete realistic cases:

1. A bridge contract that attaches a small `Transfer` (for storage staking) alongside a `FunctionCall(sign)` in the same receipt.
2. An aggregator contract that issues two `sign` calls in a single receipt to batch signing work.
3. A contract that prepends a `DeleteKey` or `AddKey` action before a `FunctionCall(sign)`.

None of these patterns require attacker privilege — any unprivileged contract caller can construct them. The NEAR protocol explicitly supports multi-action receipts and does not document any restriction against combining `FunctionCall` with other actions when targeting the MPC contract.

---

### Recommendation

Replace the hard `actions.len() != 1` guard with logic that iterates over all actions in the receipt and extracts the first (or all) `FunctionCall` action(s):

```rust
fn try_extract_function_call_args(receipt: &ReceiptView) -> Option<(&FunctionArgs, &String)> {
    let ReceiptEnumView::Action { ref actions, .. } = receipt.receipt else {
        return None;
    };
    // Find the first FunctionCall action instead of requiring exactly one action.
    actions.iter().find_map(|action| {
        if let ActionView::FunctionCall { ref method_name, ref args, .. } = action {
            Some((args, method_name))
        } else {
            None
        }
    })
}
```

If multiple `FunctionCall` actions in a single receipt must all be processed, the function signature should be changed to return an iterator or `Vec` and the call sites updated accordingly. Additionally, a `tracing::warn!` should be emitted when a multi-action receipt targeting the MPC contract is encountered, to make the current silent-drop visible in production logs.

The fix must be applied identically in both:
- `crates/node/src/indexer/handler.rs`
- `crates/chain-gateway/src/event_subscriber/streamer/block_processor.rs`

### Proof of Concept

Deploy a NEAR smart contract with the following logic:

```rust
pub fn batch_sign_with_transfer(&self, mpc_contract: AccountId, payload: Vec<u8>) -> Promise {
    // Creates a receipt with TWO actions: Transfer + FunctionCall(sign)
    Promise::new(mpc_contract.clone())
        .transfer(NearToken::from_yoctonear(1))   // action 0
        .function_call(                            // action 1
            "sign".to_string(),
            serde_json::to_vec(&sign_args(payload)).unwrap(),
            NearToken::from_yoctonear(1),
            Gas::from_tgas(15),
        )
}
```

Call `batch_sign_with_transfer`. Observe:
1. The MPC contract's `sign` method executes successfully (the yield is enqueued, `pending_signature_requests` grows by one).
2. No MPC node emits a `MPC_NUM_SIGN_REQUESTS_INDEXED` metric increment.
3. After ~200 blocks, `return_signature_and_clean_state_on_success` fires with `Err(PromiseError::Failed)` and the caller receives a timeout error — despite the network being fully operational.
