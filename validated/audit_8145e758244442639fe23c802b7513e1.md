### Title
Fee-on-Transfer Token Deposit Mints Excess wrsETH Against Actual Received Amount - (File: contracts/pools/RSETHPoolV3.sol)

### Summary
All L2 pool contracts (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPool`, `RSETHPoolNoWrapper`) compute the wrsETH/rsETH mint amount using the caller-supplied `amount` parameter rather than the actual tokens received after a `safeTransferFrom`. When a Fee-on-Transfer (FoT) token is used as a supported deposit token, the pool receives fewer tokens than `amount` but mints wrsETH as if it received the full `amount`, creating unbacked wrsETH and leading to protocol insolvency.

### Finding Description
Every token-deposit path in the L2 pool contracts follows this three-step pattern:

```solidity
// Step 1: transfer ‚Äî FoT token delivers (amount - fee) to the pool
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);

// Step 2: compute mint amount using the nominal `amount`, not actual received
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);

// Step 3: mint wrsETH based on the inflated figure
wrsETH.mint(msg.sender, rsETHAmount);
```

`viewSwapRsETHAmountAndFee` converts `amount` (not the actual balance delta) to an rsETH quantity via the oracle rate. Because the pool holds fewer tokens than `amount`, every FoT deposit creates a shortfall: the pool's token balance grows by `amount √ó (1 ‚àí feeRate)` while the wrsETH supply grows as if it received the full `amount`.

The same flaw exists identically in:
- `RSETHPoolV3.deposit(address,uint256,string)` ‚Äî lines 284‚Äì290
- `RSETHPoolV3ExternalBridge.deposit(address,uint256,string)` ‚Äî lines 403‚Äì409
- `RSETHPoolV3WithNativeChainBridge.deposit(address,uint256,string)` ‚Äî lines 320‚Äì326
- `RSETHPool.deposit(address,uint256,string)` ‚Äî lines 296‚Äì302
- `RSETHPoolNoWrapper.deposit(address,uint256,string)` ‚Äî lines 262‚Äì268

By contrast, `KernelDepositPool.notifyRewardAmount` correctly measures the actual received amount via a before/after balance check (lines 573‚Äì577).

### Impact Explanation
**Critical ‚Äî Protocol insolvency.** Each FoT deposit inflates the wrsETH supply relative to the pool's real token holdings. When the bridger calls `bridgeAssets` it transfers only the actual token balance to L1, which is less than what the outstanding wrsETH represents. Over many deposits the cumulative shortfall grows. Eventually the L1 vault holds fewer assets than the total wrsETH supply implies, meaning some wrsETH holders cannot redeem at the promised rate ‚Äî a direct, permanent loss of user funds.

### Likelihood Explanation
The pool's `addSupportedToken` function imposes no restriction against FoT tokens; any token with a valid oracle can be whitelisted. Several real LST-adjacent tokens (e.g., rebasing tokens with transfer taxes) exist on the chains where these pools are deployed. A single depositor using a FoT token triggers the shortfall immediately and irreversibly.

### Recommendation
Measure the actual received amount using a before/after balance check, mirroring the pattern already used in `KernelDepositPool`:

```solidity
uint256 balanceBefore = IERC20(token).balanceOf(address(this));
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
uint256 received = IERC20(token).balanceOf(address(this)) - balanceBefore;

(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(received, token);
```

Apply this fix to all five pool contracts listed above.

### Proof of Concept
1. Admin adds a FoT token (1% transfer fee) as a supported token in `RSETHPoolV3` with a valid oracle.
2. Attacker calls `deposit(fotToken, 10_000e18, "")`.
3. `safeTransferFrom` delivers `9_900e18` to the pool (1% fee taken).
4. `viewSwapRsETHAmountAndFee(10_000e18, fotToken)` computes rsETH based on `10_000e18`.
5. Pool mints wrsETH worth `10_000e18` of the token but only holds `9_900e18`.
6. Repeated across many depositors: pool's token balance is perpetually `1%` below what the outstanding wrsETH represents.
7. When `bridgeAssets` is called, only `9_900e18 √ó N` tokens reach L1 while wrsETH worth `10_000e18 √ó N` has been minted ‚Äî the last redeemers cannot be made whole. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
