Audit Report

## Title
ETH Deposit Limit Check Ignores Incoming Deposit Amount, Allowing Unlimited Over-Deposit Beyond Protocol Cap - (File: contracts/LRTDepositPool.sol)

## Summary

`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric guard: for ERC20 assets it correctly tests `totalAssetDeposits + amount > limit`, but for ETH it discards `amount` entirely and only tests whether the pre-deposit total already exceeds the cap. Any unprivileged caller can therefore push ETH holdings arbitrarily above the configured limit in a single `depositETH` call, causing unbounded rsETH over-minting and diluting all existing holders.

## Finding Description

The root cause is the ETH branch at [1](#0-0)  which returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` — the incoming `amount` is never added. The ERC20 branch at [2](#0-1)  correctly includes `amount` in the comparison.

The full call path is:

1. `depositETH` (public, payable, no role required) calls `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, ...)`. [3](#0-2) 
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` and reverts only if it returns `true`. [4](#0-3) 
3. Because the ETH branch ignores `amount`, the function returns `false` whenever `totalAssetDeposits <= depositLimit`, regardless of how large `msg.value` is.
4. Execution continues to `_mintRsETH(rsethAmountToMint)`, minting rsETH for the full `msg.value`. [5](#0-4) 

No existing guard compensates for this: `getAssetCurrentLimit` is a view function that cannot block the deposit, and `_beforeDeposit` has no secondary ETH-specific cap check. [6](#0-5) 

## Impact Explanation

**Critical — Protocol insolvency.** The deposit limit exists to cap EigenLayer exposure. When bypassed, rsETH is minted proportional to the full over-deposit at the current oracle rate. `getTotalAssetDeposits` will reflect the inflated balance across the deposit pool, NDCs, EigenLayer strategies, converter, and unstaking vault. [7](#0-6)  The already-minted rsETH cannot be recalled without governance intervention. Under any EigenLayer slashing event at the over-exposed scale, the protocol cannot make all rsETH holders whole, constituting protocol insolvency — a concrete allowed Critical impact.

## Likelihood Explanation

**High.** `depositETH` is public, payable, and requires no special role. [8](#0-7)  The precondition (`totalAssetDeposits <= depositLimit`) is the normal operating state of the protocol. The required information (`getTotalAssetDeposits`) is readable on-chain by anyone. No front-running, oracle manipulation, or governance capture is needed — a single transaction suffices.

## Recommendation

Apply the same prospective check used for ERC20 tokens to ETH, eliminating the special-case branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept

```
depositLimit               = 1_000 ETH
totalAssetDeposits (before) = 999 ETH

Attacker calls: depositETH{value: 5_000 ETH}(0, "")

_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 5_000 ETH):
  totalAssetDeposits = 999 ETH
  ETH branch: return (999 > 1_000) → false   ← deposit NOT blocked

_mintRsETH mints rsETH for 5_000 ETH at current oracle rate.

totalAssetDeposits (after) = 5_999 ETH  (~6× the intended cap)
```

**Foundry test plan:** Deploy `LRTDepositPool` with `depositLimitByAsset(ETH) = 1_000 ether`. Seed the pool with 999 ETH. Call `depositETH{value: 5_000 ether}(0, "")` from an unprivileged address. Assert the call succeeds (no revert), assert `getTotalAssetDeposits(ETH_TOKEN) == 5_999 ether`, and assert the attacker's rsETH balance is non-zero — all three assertions will pass, confirming the bypass.
