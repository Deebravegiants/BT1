### Title
Governance Proposal Deposit Permanently Redirected to Treasury When Return-Address Credential Is Deregistered After Submission — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Epoch.hs`)

---

### Summary

The `returnProposalDeposits` function in the Conway `EPOCH` rule silently redirects a governance proposal's deposit to the treasury whenever the stake credential embedded in the proposal's `gasReturnAddr` is no longer registered at the moment the proposal is removed (expired, enacted, or pruned). Because stake-credential deregistration only requires a witness for the credential itself, an unprivileged attacker who can satisfy a script credential used as a return address can deregister that credential between proposal submission and proposal removal, permanently diverting the proposer's deposit to the treasury with no recovery path.

---

### Finding Description

Every governance proposal carries a `pProcReturnAddr :: AccountAddress` and a `pProcDeposit :: Coin`. At proposal submission the `GOV` rule validates that the return address is currently registered:

```
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs ~line 511
let nonRegisteredAccounts = ...
failOnNonEmpty (Map.keys nonRegisteredAccounts)
               (injectFailure . TreasuryWithdrawalReturnAccountsDoNotExist)
``` [1](#0-0) 

However, no invariant prevents the same credential from being deregistered while the proposal is live. At every epoch boundary the `EPOCH` rule calls `returnProposalDeposits` over all removed proposals:

```haskell
returnProposalDeposits removedProposals oldAccounts =
  foldr' processProposal (oldAccounts, mempty) removedProposals
  where
    processProposal gas (!accounts, !unclaimed)
      | (Just _accountState, newAccounts) <-
            updateLookupAccountState addRefund cred accounts =
          (newAccounts, unclaimed)
      | otherwise = (accounts, Map.insert (gasId gas) (gasDeposit gas) unclaimed)
      where
        addRefund = balanceAccountStateL <>~ compactCoinOrError (gasDeposit gas)
        cred = gasReturnAddr gas ^. accountAddressCredentialL
``` [2](#0-1) 

If `cred` is not registered the deposit is placed in `unclaimed` and immediately swept into the treasury:

```haskell
chainAccountState3 =
  chainAccountState2
    & casTreasuryL <>~ (utxoState0 ^. utxosDonationL <> fold unclaimed)
``` [3](#0-2) 

Stake-credential deregistration (`ConwayUnRegCert`) only requires (a) a valid witness for the credential and (b) a zero reward-account balance:

```haskell
checkStakeKeyHasZeroRewardBalance = do
  accountState <- mAccountState
  let balanceCompact = accountState ^. balanceAccountStateL
  guard (balanceCompact /= mempty)
  Just $ fromCompact balanceCompact
failOnJust checkStakeKeyHasZeroRewardBalance
           (injectFailure . StakeKeyHasNonZeroAccountBalanceDELEG)
``` [4](#0-3) 

The proposal deposit is tracked in `utxosDeposited`, **not** in the reward-account balance, so the balance check does not protect against deregistration of a credential that has a pending proposal deposit. For a script credential whose script is satisfiable by the attacker (e.g., `RequireAllOf []`, a multi-sig with a threshold the attacker controls, or any native script whose signatories the attacker possesses), the attacker can submit a valid deregistration certificate and redirect the proposer's deposit to the treasury.

The same silent-skip pattern applies to enacted `TreasuryWithdrawals` in `applyEnactedWithdrawals`:

```haskell
successfulWithdrawls =
  Map.mapMaybeWithKey
    (\cred w -> compactCoinOrError w <$ guard (isAccountRegistered cred accounts))
    enactedWithdrawals
``` [5](#0-4) 

If a withdrawal recipient's credential is deregistered between ratification and enactment, the treasury funds earmarked for that recipient are silently retained in the treasury.

---

### Impact Explanation

The governance action deposit (`ppGovActionDepositL`, currently 100,000,000 lovelace on mainnet) is permanently transferred to the treasury. The proposer has no mechanism to reclaim it: there is no sweep function, no re-registration path that restores the deposit, and no governance action that can redirect treasury funds back to an individual. Recovery would require a hard fork. This satisfies **High — Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork**.

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender who can satisfy the script of a script-credential return address. The attacker needs only to:

1. Observe an active proposal whose `gasReturnAddr` embeds a script credential they can witness.
2. Confirm the reward-account balance of that credential is zero (publicly visible on-chain).
3. Submit a deregistration certificate with the satisfying script witness before the proposal expires.

No privileged access, governance majority, or key compromise is required. The constraint is that the return address must use a script credential the attacker can satisfy; key-credential return addresses are not exploitable by third parties.

---

### Recommendation

1. **Block deregistration when a pending proposal deposit exists.** Before processing `ConwayUnRegCert`, check whether the credential appears as `gasReturnAddr` in any live `GovActionState`. If so, reject the deregistration with a new predicate failure (e.g., `StakeKeyUsedAsProposalReturnAddrDELEG`).

2. **Alternatively, store the deposit against the credential, not the return address.** Record the deposit amount in the account state at proposal submission so it survives deregistration and can be claimed on re-registration.

3. **At minimum, document the invariant** that a return address must remain registered for the full lifetime of the proposal, and surface a warning or failure if a deregistration would violate it.

---

### Proof of Concept

```
1. Attacker identifies a live governance proposal P with:
     gasReturnAddr = AccountAddress Testnet (AccountId (ScriptHashObj sh))
   where sh is the hash of a native script S that the attacker can satisfy
   (e.g., RequireSignature attackerKey, or RequireAllOf []).

2. Attacker checks: reward balance of sh = 0  (no rewards accumulated yet).
   The proposal deposit D is in utxosDeposited, NOT in the reward balance.

3. Attacker submits a transaction containing:
     ConwayUnRegCert (ScriptHashObj sh) SNothing
   with the script S as witness.
   The DELEG rule accepts it because balanceAccountStateL = 0.

4. The credential ScriptHashObj sh is removed from the Accounts map.

5. After ppGovActionLifetimeL epochs, proposal P expires.
   EPOCH rule calls:
     proposalsApplyEnactment rsEnacted rsExpired proposals
   P is in expiredActions.

6. returnProposalDeposits is called with P in removedProposals.
   updateLookupAccountState returns Nothing (credential not registered).
   D is inserted into unclaimed.

7. chainAccountState3 = chainAccountState2 & casTreasuryL <>~ fold unclaimed
   D is permanently added to the treasury.

8. The original proposer has permanently lost D with no recovery path.
