### Title
User-Controlled Finality/Confirmation Parameters Allow Signing of Unfinalized Foreign-Chain Transactions - (File: crates/contract/src/lib.rs)

### Summary

The `verify_foreign_transaction` contract method accepts user-supplied finality parameters (`EvmFinality::Latest` for EVM chains, `BlockConfirmations(0)` for Bitcoin) without enforcing any minimum security threshold. The MPC network will sign an attestation for a transaction that has not reached cryptographic finality, enabling a double-spend attack against any bridge (e.g., Omnibridge) that relies on these attestations.

### Finding Description

`verify_foreign_transaction` queues a signing request without validating the caller-supplied `finality` or `confirmations` fields:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    // ...
    self.check_request_preconditions(...)  // only checks gas/deposit/domain
    // NO validation of request.request.finality or request.request.confirmations
    self.enqueue_yield_request(...)
}
``` [1](#0-0) 

The `EvmFinality` enum exposes `Latest` as a fully valid variant with no contract-side rejection: [2](#0-1) 

`BlockConfirmations` is a plain `u64` with `minimum: 0.0` in the ABI schema — zero is accepted: [3](#0-2) 

The node-side inspector passes the user-supplied value directly to the RPC check:

```rust
let block_confirmations = request.confirmations.0.into();
let extracted_values = inspector
    .extract(transaction_id, block_confirmations, extractors)
    ...
``` [4](#0-3) 

In the Bitcoin inspector, the threshold check is `block_confirmations_threshold <= transaction_block_confirmation`. With threshold = 0, this is always true: [5](#0-4) 

For EVM, `finality: Latest` maps to `FinalityTag::Latest`, meaning the check only requires the latest block number ≥ receipt block number — trivially satisfied for any included transaction: [6](#0-5) 

The signed payload includes the full `ForeignChainRpcRequest` (including the user-supplied `finality`/`confirmations`), so the signature is over a payload that explicitly encodes the weak finality level: [7](#0-6) 

### Impact Explanation

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow: a user deposits on a foreign chain, obtains an MPC-signed attestation, and claims funds on NEAR. With `finality: Latest` (EVM) or `confirmations: 0`/`1` (Bitcoin):

1. Attacker deposits funds on Ethereum/Bitcoin.
2. Attacker immediately calls `verify_foreign_transaction` with `finality: Latest` or `confirmations: 1`.
3. MPC network signs an attestation for the unfinalized transaction.
4. Attacker claims funds on NEAR using the valid MPC signature.
5. Attacker double-spends the original deposit (EVM reorg within the ~13-minute finalization window, or Bitcoin RBF/reorg with 1 confirmation).
6. Bridge loses funds; the MPC-signed attestation remains valid and cannot be revoked.

This is a **High** impact finding: forged foreign-chain verification that causes invalid bridge execution and double-spend conditions.

### Likelihood Explanation

- Any unprivileged caller can submit `verify_foreign_transaction` with a 1-yoctoNEAR deposit.
- No special access, key material, or threshold collusion is required.
- EVM reorgs within the pre-finality window are a known, realistic event (Ethereum finality takes ~13 minutes; `Latest` blocks reorg regularly on chains like Polygon).
- Bitcoin 1-confirmation reorgs are rare but economically motivated when bridge funds are large.

### Recommendation

Enforce a minimum finality level in `verify_foreign_transaction` before queuing the request. Options:

1. **Contract-side rejection**: Reject `EvmFinality::Latest` and `BlockConfirmations` below a per-chain minimum (e.g., 6 for Bitcoin, `Finalized` for Ethereum) directly in `verify_foreign_transaction`.
2. **Per-chain minimum in `ChainEntry`**: Add a `min_confirmations`/`min_finality` field to the on-chain `ChainEntry` (voted in by participants alongside the provider list), and validate the user-supplied value against it at request time.

The second approach mirrors the existing `ChainEntry.quorum` pattern and allows per-chain tuning without a contract upgrade. [8](#0-7) 

### Proof of Concept

```
1. Deploy/use an existing Omnibridge contract that calls verify_foreign_transaction.
2. Send 10 ETH to the bridge deposit address on Ethereum (tx_id = 0xABC...).
3. Wait for the transaction to appear in the latest block (1 confirmation).
4. Call verify_foreign_transaction({
       request: ForeignChainRpcRequest::Ethereum(EvmRpcRequest {
           tx_id: 0xABC...,
           finality: EvmFinality::Latest,   // <-- attacker-controlled
           extractors: [BlockHash],
       }),
       domain_id: <foreign_tx_domain>,
       payload_version: V1,
   }) with 1 yoctoNEAR attached.
5. MPC nodes query eth_getTransactionReceipt (succeeds, tx is in latest block).
   verify_finality_level: head.number >= receipt_block_number → passes.
6. MPC signs ForeignTxSignPayloadV1 { request, values: [BlockHash(0x...)] }.
7. Attacker receives valid MPC signature on NEAR; claims 10 ETH worth of bridged tokens.
8. Attacker submits a higher-fee replacement transaction (EIP-1559 RBF) on Ethereum,
   reversing the original deposit before Ethereum finality (~13 min).
9. Bridge on NEAR has paid out; original deposit is gone. Net loss = 10 ETH.
```

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
