### Title
Missing Zero-Amount Guard in `bridgeAssetsViaNativeBridge` Allows Zero-Value Bridge Dispatch — (`contracts/pools/RSETHPool.sol`)

---

### Summary

`RSETHPool.bridgeAssetsViaNativeBridge` computes `ethBalanceMinusFees = address(this).balance - feeEarnedInETH` and passes it directly to `BaseMessenger.sendETHToL1ViaBridge` with no zero-value guard. When that difference is zero, a zero-value bridge message is dispatched to the Base bridge, consuming gas and a bridge nonce while L1Vault receives nothing.

---

### Finding Description

`bridgeAssetsViaNativeBridge` computes the bridgeable amount and immediately forwards it: [1](#0-0) 

```solidity
function bridgeAssetsViaNativeBridge() external nonReentrant onlyRole(BRIDGER_ROLE) {
    ...
    uint256 ethBalanceMinusFees = getETHBalanceMinusFees();   // can be 0

    IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
        l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees   // value: 0, amount: 0
    );
    ...
}
```

`getETHBalanceMinusFees()` returns `address(this).balance - feeEarnedInETH`: [2](#0-1) 

This is zero whenever the pool holds no non-fee ETH — e.g., the pool has no deposits yet, all ETH was already bridged in a prior call, or `feeEarnedInETH` equals the full balance (possible at `feeBps == 10_000`).

`BaseMessenger.sendETHToL1ViaBridge` has no zero-value guard either: [3](#0-2) 

```solidity
function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value) external payable nonReentrant {
    if (msg.value != value) revert MismatchedMsgValue();   // 0 == 0 → passes
    IBaseMessenger(l2bridge).bridgeETHTo{ value: value }(target, DEFAULT_GAS_LIMIT, bytes(""));
}
```

When `value == 0`, the `msg.value != value` check passes (both are 0), and `bridgeETHTo{value:0}` is called on the L2 bridge — a zero-value cross-chain message.

**Contrast with `bridgeTokens`**, which correctly guards against this: [4](#0-3) 

```solidity
if (balance == 0) {
    revert ZeroBridgeAmount();
}
```

`bridgeAssetsViaNativeBridge` has no equivalent check, making the inconsistency a clear oversight.

---

### Impact Explanation

A zero-value bridge call is dispatched to the Base L2 bridge. Gas is consumed, a bridge nonce is used, and L1Vault receives 0 ETH. No funds are lost, but the function fails to deliver its promised behavior (bridging ETH to L1Vault). This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

The `BRIDGER_ROLE` is a trusted off-chain operator, but the contract provides no on-chain enforcement to prevent this call when `ethBalanceMinusFees == 0`. It can be triggered:
- By calling `bridgeAssetsViaNativeBridge` twice in a row (second call has zero balance)
- Before any ETH deposits have occurred
- After fees have grown to equal the full pool balance

No attacker control is needed; the BRIDGER_ROLE itself can trigger this inadvertently.

---

### Recommendation

Add a zero-amount guard in `bridgeAssetsViaNativeBridge`, consistent with `bridgeTokens`:

```solidity
uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
if (ethBalanceMinusFees == 0) revert ZeroBridgeAmount();
```

Optionally, add the same guard in `BaseMessenger.sendETHToL1ViaBridge` as a defense-in-depth measure.

---

### Proof of Concept

```solidity
// 1. Deploy RSETHPool with BRIDGER_ROLE set, l2Bridge/messenger/l1VaultETHForL2Chain configured

// 4. Observe: ethBalanceMinusFees = 0 - 0 = 0
//             BaseMessenger receives {value:0}, passes msg.value check (0==0)

```
