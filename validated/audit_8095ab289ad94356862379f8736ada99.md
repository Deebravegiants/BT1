### Title
Unvalidated `builder` Address in `ProposalInit` Allows Sequencer Fee Redirection - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` validates many fields of `ProposalInit` (height, l1_da_mode, l2_gas_price_fri, starknet_version, L1 gas prices, fee_proposal_fri, version_constant_commitment, timestamp) but never validates the `builder` field. The `builder` field is passed directly as `sequencer_address` into `BlockInfo` for block execution. A malicious proposer can set `builder` to any arbitrary address; the validator executes the block with that address as the sequencer, redirecting all transaction fees in the block to the attacker-controlled address. The `ProposalFinMismatch` check does not catch this because both proposer and validator derive the commitment from the same proposer-supplied `builder` value.

### Finding Description

`ProposalInit.builder` is the block-builder/sequencer address. It is accepted verbatim from the network peer and flows into execution without any equality check against a locally-trusted value.

**Validation gap in `is_proposal_init_valid`:**

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs L312-321
if !(init_proposed.height == proposal_init_validation.height
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
{
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
// builder is never checked
```

`ProposalInitValidation` does not even carry an expected `builder` field:

```rust
// L75-85
pub(crate) struct ProposalInitValidation {
    pub height: BlockNumber,
    pub block_timestamp_window_seconds: u64,
    pub previous_proposal_init: Option<PreviousProposalInitInfo>,
    pub l1_da_mode: L1DataAvailabilityMode,
    pub l2_gas_price_fri: GasPrice,
    pub starknet_version: StarknetVersion,
    pub fee_actual: Option<GasPrice>,
    // no builder field
}
```

The unchecked `builder` is then used as `sequencer_address` in block execution:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs L329-332
Ok(starknet_api::block::BlockInfo {
    block_number: init.height,
    block_timestamp: BlockTimestamp(init.timestamp),
    sequencer_address: init.builder,   // ← attacker-controlled
    ...
})
```

And stored in the committed block header:

```rust
// crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs L397
let sequencer = SequencerContractAddress(init.builder);
```

**Why `ProposalFinMismatch` does not save us:**

The `proposal_commitment` is derived from `PartialBlockHash`, which is computed by the batcher using the `BlockInfo` it received — including the attacker-supplied `sequencer_address`. The proposer pre-computes the same commitment with the same `builder` value. Both sides agree, so the check at line 244 passes:

```rust
// L244-247
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}
```

The block is committed with the wrong `sequencer_address` and the wrong block hash.

### Impact Explanation

In Starknet, `sequencer_address` (= `builder`) is the address that receives transaction fees during block execution. By substituting their own address, a malicious proposer diverts all fees from every transaction in the block to themselves. Additionally, the committed block header and block hash permanently encode the wrong sequencer address, corrupting the authoritative on-chain state. This matches: **"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."**

### Likelihood Explanation

The attacker must be selected as a proposer by the consensus algorithm. In the current production code there is no enforced validator set (the comment `// TODO(Matan): Remove this once we have a proper validator set` at `DEFAULT_VALIDATOR_ID` confirms this). Any participating node can be a proposer. Likelihood is **Medium** — one malicious proposer turn per height is sufficient.

### Recommendation

Add `builder` to `ProposalInitValidation` and check it in `is_proposal_init_valid`:

```rust
// In ProposalInitValidation
pub expected_builder: ContractAddress,

// In is_proposal_init_valid, alongside the existing checks:
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.expected_builder, init_proposed.builder
        ),
    ));
}
```

`expected_builder` should be populated from the node's own `static_config.builder_address` (the same source used when building proposals).

### Proof of Concept

1. Attacker node participates in consensus and is selected as proposer for block N.
2. Attacker constructs `ProposalInit` with `builder = attacker_address` (any address they control).
3. Attacker computes `proposal_commitment` using `attacker_address` as `sequencer_address` — this is the value placed in `ProposalFin`.
4. Validator node receives the proposal, calls `is_proposal_init_valid` — `builder` is never checked, validation passes.
5. `initiate_validation` calls `convert_to_sn_api_block_info(init)` → `sequencer_address = attacker_address` is passed to the batcher.
6. Batcher executes all transactions with `sequencer_address = attacker_address`; all fees are credited to `attacker_address`.
7. Batcher returns `PartialBlockHash` computed with `attacker_address`; `proposal_commitment_from(partial_block_hash, fee_proposal_fri)` matches the proposer's `ProposalFin.proposal_commitment`.
8. `ProposalFinMismatch` check passes; block is committed.
9. `update_state_sync_with_new_block` stores `SequencerContractAddress(attacker_address)` in the block header.
10. All fees from block N are permanently in `attacker_address`; the legitimate sequencer receives nothing.

    init: &ProposalInit,
) -> Result<starknet_api::block::BlockInfo, StarknetApiError> {
    if init.l1_gas_price_fri.0 == 0
        || init.l1_gas_price_wei.0 == 0
        || init.l1_data_gas_price_fri.0 == 0
        || init.l1_data_gas_price_wei.0 == 0
        || init.l2_gas_price_fri.0 == 0
    {
        warn!("Zero gas price detected in block info: {:?}", init);
    }

    let l1_gas_price_fri = NonzeroGasPrice::new(init.l1_gas_price_fri)?;
    let l1_data_gas_price_fri = NonzeroGasPrice::new(init.l1_data_gas_price_fri)?;
    let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
    let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
    let l2_gas_price_fri = NonzeroGasPrice::new(init.l2_gas_price_fri)?;
    let proposal_init_info = PreviousProposalInitInfo::from(init);
    let eth_to_fri_rate = calculate_eth_to_fri_rate(&proposal_init_info)?;

    let l2_gas_price_wei = NonzeroGasPrice::new(init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?)

                "L2 gas price in wei is zero! Conversion rate: {eth_to_fri_rate}, L2 gas price in \

                init.l2_gas_price_fri
            )
        })?;
    Ok(starknet_api::block::BlockInfo {
        block_number: init.height,
        block_timestamp: BlockTimestamp(init.timestamp),
        sequencer_address: init.builder,
        gas_prices: GasPrices {
            strk_gas_prices: GasPriceVector {
                l1_gas_price: l1_gas_price_fri,
                l1_data_gas_price: l1_data_gas_price_fri,
                l2_gas_price: l2_gas_price_fri,
            },
            eth_gas_prices: GasPriceVector {
                l1_gas_price: l1_gas_price_wei,
                l1_data_gas_price: l1_data_gas_price_wei,
                l2_gas_price: l2_gas_price_wei,
            },
        },
        use_kzg_da: init.l1_da_mode.is_use_kzg_da(),
        starknet_version: init.starknet_version,
    })
