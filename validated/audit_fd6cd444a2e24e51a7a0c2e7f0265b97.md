Audit Report

## Title
Router's Stranded Native ETH Consumed for Attacker's WETH-Input Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` settles WETH-input swaps by wrapping the router's entire `address(this).balance` without verifying the ETH belongs to the current caller's `msg.value`. If a prior user sent excess ETH and omitted `refundETH`, an attacker can call `exactInputSingle{value: 0}` with `tokenIn=WETH` and have the router wrap the victim's stranded ETH to pay for the attacker's swap, giving the attacker free output tokens at the victim's expense.

## Finding Description

In `PeripheryPayments.pay()` (lines 73–84), when `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` as the available native ETH:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
```

`nativeBalance` is the router's total ETH balance across all callers and transactions, not the current call's `msg.value`. The router accumulates ETH from `msg.value` in payable entry points (`exactInputSingle`, `exactOutputSingle`, `multicall`). The `receive()` guard (lines 32–34) only blocks arbitrary ETH pushes from non-WETH addresses; it does not prevent exploitation of ETH already sitting in the router.

The exploit path:
1. Victim calls `exactInputSingle{value: 10_000}(tokenIn=WETH, amountIn=1_000)`. The swap consumes 1,000 ETH worth of WETH; 9,000 ETH remains in the router. Victim omits `refundETH`.
2. Attacker calls `exactInputSingle{value: 0}(tokenIn=WETH, amountIn=9_000)`. The pool calls `metricOmmSwapCallback` on the router. The router calls `_justPayCallback` → `pay(WETH, attacker, pool, 9_000)`. Since `nativeBalance (9_000) >= value (9_000)`, the router wraps its own 9,000 ETH and transfers WETH to the pool.
3. The pool's `IncorrectDelta` check (lines 261–263 of `MetricOmmPool.sol`) only verifies `balance0Before + amount0Delta <= balance0()` — it does not care whether the WETH was wrapped from the attacker's ETH or the victim's stranded ETH. The check passes.
4. Attacker receives pool output tokens having paid nothing.

The payer stored in transient storage via `_setNextCallbackContext` is `msg.sender` (the attacker), but `pay()` only falls back to `safeTransferFrom(payer, ...)` when `nativeBalance == 0`. With stranded ETH present, the `safeTransferFrom` branch is never reached.

## Impact Explanation

Direct, permanent loss of the victim's ETH. The victim's stranded ETH is consumed to settle the attacker's swap. The attacker receives real pool output tokens (token0 or token1) without transferring any asset. Loss magnitude equals `min(router.balance, attacker's amountIn)` and is bounded only by how much ETH the victim left in the router. This is a direct loss of user principal, meeting Critical/High severity thresholds.

## Likelihood Explanation

The `receive()` guard limits ETH accumulation to `msg.value` from payable calls, so the precondition (stranded ETH) requires a user to over-send ETH and omit `refundETH`. This is a realistic user error, especially for users calling `exactInputSingle` directly rather than via multicall — the test suite itself documents the multicall+`refundETH` pattern as the correct approach (lines 106–133 of `MetricOmmSimpleRouter.native.t.sol`), implying direct callers may not follow it. An attacker can monitor the router's ETH balance on-chain and exploit it atomically in the next block. The attack requires no special privileges and is repeatable.

## Recommendation

Track the ETH attributable to the current call context. Inside `pay()`, when `token == WETH`, only consume native ETH up to the `msg.value` of the originating entry-point call (passed as a parameter or stored in transient storage at entry alongside the payer). Any shortfall must be pulled from the payer via `safeTransferFrom`. Alternatively, enforce `address(this).balance == 0` at the start of every non-multicall payable entry point, or document and enforce via a modifier that WETH-input swaps must always be paired with `refundETH` in a multicall.

## Proof of Concept

```
Setup:
- Router deployed with WETH address.
- Pool with token0=WETH, token1=TokenA exists and has liquidity.

Step 1 (Victim):
  victim.call{value: 10_000}(
    router.exactInputSingle(tokenIn=WETH, amountIn=1_000, ...)
  )
  // Swap uses 1_000 ETH worth of WETH; 9_000 ETH remains in router.
  // Victim does NOT call refundETH.
  assert(address(router).balance == 9_000)

Step 2 (Attacker):
  attacker.call{value: 0}(
    router.exactInputSingle(tokenIn=WETH, amountIn=9_000, ...)
  )
  // Pool calls metricOmmSwapCallback on router.
  // Router: pay(WETH, attacker, pool, 9_000)
  // nativeBalance = 9_000 >= value = 9_000 → wraps router's ETH, transfers WETH to pool.
  // Pool IncorrectDelta check passes (pool received correct WETH).
  // Attacker receives TokenA output.

Result:
  assert(address(router).balance == 0)   // victim's ETH gone
  assert(attacker paid 0 ETH)
  assert(attacker received TokenA)        // free trade
```

Foundry test: set up two actors, have victim call `exactInputSingle{value: 10_000}` with `amountIn=1_000` and skip `refundETH`, then have attacker call `exactInputSingle{value: 0}` with `amountIn=9_000` and assert attacker received output tokens while `address(router).balance == 0`.
