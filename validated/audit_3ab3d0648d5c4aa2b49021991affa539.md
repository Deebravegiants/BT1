Audit Report

## Title
Stale `tokenToETHRate` via Block Stuffing Prematurely Exhausts `dailyMintLimit`, Temporarily Blocking Token Deposits - (`contracts/pools/RSETHPoolV3.sol`)

## Summary
`RSETHPoolV3.deposit(address,uint256,string)` applies the `limitDailyMint` modifier, which computes `rsETHAmount` using `IOracle(supportedTokenOracle[token]).getRate()` with no time-based staleness check. Both oracle implementations (`CrossChainRateReceiver` and `ChainlinkOracleForRSETHPoolCollateral`) are push-based and lack heartbeat validation, so an attacker who stuffs blocks to delay oracle updates can keep an inflated `tokenToETHRate` in place. Each deposit then over-counts its rsETH equivalent, exhausting `dailyMintLimit` prematurely and reverting all further token deposits with `DailyMintLimitExceeded` for up to 24 hours.

## Finding Description
**Entrypoint:** `deposit(address token, uint256 amount, string referralId)` at `RSETHPoolV3.sol:271` is public and permissionless.

**`limitDailyMint` modifier (`RSETHPoolV3.sol:96-125`):** Calls `viewSwapRsETHAmountAndFee(amount, token)` to compute `rsETHAmount`, adds it to `dailyMintAmount`, and reverts with `DailyMintLimitExceeded` if `dailyMintAmount + rsETHAmount > dailyMintLimit`. No staleness guard is applied to the oracle rate used in this computation.

**`viewSwapRsETHAmountAndFee` (`RSETHPoolV3.sol:315-334`):**
```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```
The rate is consumed directly with no age check.

**Oracle 1 — `CrossChainRateReceiver` (`CrossChainRateReceiver.sol:102-105`):** `getRate()` returns the stored `rate` unconditionally. `lastUpdated` is written in `lzReceive` but never read in `getRate()`. `lzReceive` is a regular on-chain transaction that can be excluded from blocks via stuffing.

**Oracle 2 — `ChainlinkOracleForRSETHPoolCollateral` (`ChainlinkOracleForRSETHPoolCollateral.sol:26-37`):** The only staleness guard is `answeredInRound < roundID`, which detects an incomplete round but not elapsed time. If block stuffing prevents Chainlink keepers from submitting a new round, the last completed round's data passes this check indefinitely.

**Attack flow:**
1. A supported token begins depegging; its true ETH value is falling.
2. The oracle update transaction (LayerZero `lzReceive` or Chainlink keeper) would lower `tokenToETHRate`.
3. Attacker stuffs blocks with high-gas transactions to exclude the oracle update, keeping the pre-depeg (inflated) `tokenToETHRate` in place.
4. Each `deposit(token, amount, ...)` computes an inflated `rsETHAmount = amountAfterFee * inflatedRate / rsETHToETHrate`, consuming `dailyMintAmount` faster than the true rate would.
5. `dailyMintAmount + rsETHAmount > dailyMintLimit` triggers `DailyMintLimitExceeded`, blocking all further token deposits until `getCurrentDay() > lastMintDay` resets `dailyMintAmount` to 0 (up to 24 hours).

## Impact Explanation
Legitimate users cannot deposit the affected supported token for up to 24 hours. No funds already in the contract are at risk, but the deposit service is temporarily frozen for the affected token. This matches the allowed impact: **Low. Block stuffing.**

## Likelihood Explanation
Block stuffing is economically viable on L2 chains (where `RSETHPoolV3` is deployed) due to lower block gas limits and cheaper gas prices than Ethereum mainnet. The oracle update cadence is predictable for both Chainlink heartbeat intervals and periodic LayerZero pushes. No privileged access is required; the attack is executable by any external caller. The attacker bears the cost of block stuffing with no direct financial return, which limits real-world motivation, but the technical path is fully concrete.

## Recommendation
Add a time-based staleness check in `viewSwapRsETHAmountAndFee` (or in the oracle wrappers themselves) that reverts if `block.timestamp - lastUpdated > maxStaleness`. For `CrossChainRateReceiver`, expose `lastUpdated` and enforce a maximum age in `getRate()`. For `ChainlinkOracleForRSETHPoolCollateral`, add a `block.timestamp - timestamp > heartbeat` check alongside the existing `answeredInRound` check. This ensures a stale rate causes a revert rather than silently inflating `rsETHAmount` and consuming `dailyMintLimit`.

## Proof of Concept
Deploy `RSETHPoolV3` with `dailyMintLimit = 100e18` rsETH and `rsETHToETHrate = 1e18`. Add a supported token with a mock oracle returning `tokenToETHRate = 1.1e18` (inflated, simulating block-stuffed staleness). Call `deposit(token, 10e18, "")` in a loop:
- With stale rate `1.1e18`: each deposit counts as `11 rsETH`; limit exhausted after ~9 deposits.
- With correct rate `0.9e18`: each deposit counts as `9 rsETH`; limit exhausted after ~11 deposits.

After ~9 deposits with the stale rate, all subsequent calls revert with `DailyMintLimitExceeded`. The Foundry test in the submission directly demonstrates this: set `tokenOracle.rate = 1.1e18`, loop `deposit` calls, assert `depositsBeforeLimit < 10`.
