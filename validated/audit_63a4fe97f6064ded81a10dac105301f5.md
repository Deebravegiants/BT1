Audit Report

## Title
Precision Truncation in Token-to-rsETH Swap Yields Zero rsETHAmount, Permanently Stranding User Tokens - (`contracts/pools/RSETHPool.sol`, `RSETHPoolV3.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`)

## Summary

All five L2 pool contracts compute `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing the token amount to 18 decimals. For low-decimal tokens (e.g., USDC at 6 decimals), the numerator can be less than the denominator, causing integer truncation to zero. Because no zero-output guard exists in any deposit function, the user's tokens are transferred to the pool and they receive 0 rsETH in return, with no accounting entry for the stranded principal.

## Finding Description

**Root cause:** The ETH deposit path correctly scales by `1e18` to compensate for ETH's 18-decimal precision:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The token deposit path omits this normalization:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes all Chainlink prices to 1e18 precision regardless of the token's own decimal count:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

For USDC (6 decimals, $1 when ETH = $3,000): `tokenToETHRate ≈ 3.33e14`. For `rsETHToETHrate ≈ 1.05e18`, truncation to zero occurs when `amountAfterFee * 3.33e14 < 1.05e18`, i.e., `amountAfterFee < 3,154` USDC wei (~0.003 USDC).

**Exploit flow:**
1. User calls `deposit(USDC, 3_000, "")` on any affected pool.
2. `if (amount == 0) revert InvalidAmount()` passes (amount = 3,000 ≠ 0).
3. `safeTransferFrom` moves 3,000 USDC wei from user to pool.
4. `viewSwapRsETHAmountAndFee(3_000, USDC)` returns `(0, 0)` due to truncation.
5. `feeEarnedInToken[token] += 0` — no accounting entry for the principal.
6. `safeTransfer(user, 0)` or `wrsETH.mint(user, 0)` succeeds silently.
7. User holds 0 rsETH; pool holds 3,000 USDC wei with no recovery path.

**Existing checks are insufficient:** The only guard is `if (amount == 0) revert InvalidAmount()`, which does not protect against a non-zero input producing a zero output. There is no `if (rsETHAmount == 0) revert` anywhere in the deposit path across all five contracts.

## Impact Explanation

**Low — Contract fails to deliver promised returns.** A depositor who sends a sub-threshold amount of a supported low-decimal token receives 0 rsETH while their tokens are permanently retained by the pool. The `feeEarnedInToken` mapping is not incremented for the lost principal, so the funds are stranded with no accounting entry and no user-accessible recovery mechanism. The monetary loss per transaction is sub-cent for USDC, but the contract silently violates its core deposit guarantee.

## Likelihood Explanation

Low-to-medium. The threshold is deterministic and small (~0.003 USDC for current ETH prices). It can be triggered accidentally by UI rounding, test transactions, or intentionally by any unprivileged user. No special privileges, oracle manipulation, or external conditions are required — only a supported low-decimal token and a deposit below the truncation threshold.

## Recommendation

1. Add a zero-output guard in every token deposit function:
   ```solidity
   if (rsETHAmount == 0) revert InvalidAmount();
   ```
2. Normalize the token amount to 18 decimals before applying the rate, mirroring the ETH path:
   ```solidity
   uint256 normalizedAmount = amountAfterFee * 10**(18 - tokenDecimals);
   rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
   ```
   This requires storing or querying `IERC20Metadata(token).decimals()` for each supported token.

## Proof of Concept

**Parameters:** USDC (6 decimals), `tokenToETHRate = 3.33e14`, `rsETHToETHrate = 1.05e18`, `feeBps = 0`.

```
amountAfterFee = 3_000   // 0.003 USDC
rsETHAmount    = 3_000 * 3.33e14 / 1.05e18
               = 9.99e17 / 1.05e18
               = 0        // integer truncation
```

**Foundry fuzz test plan:**
```solidity
function testFuzz_tokenDepositZeroOutput(uint256 amount) public {
    // Bound amount to [1, threshold - 1]
    amount = bound(amount, 1, 3_153);
    deal(address(usdc), user, amount);
    vm.startPrank(user);
    usdc.approve(address(pool), amount);
    pool.deposit(address(usdc), amount, "");
    vm.stopPrank();
    // User received 0 rsETH but lost their tokens
    assertEq(wrsETH.balanceOf(user), 0);
    assertEq(usdc.balanceOf(user), 0);
    assertEq(usdc.balanceOf(address(pool)), amount); // tokens stranded
}
```

This test will pass (demonstrating the bug) against the current code in all five affected contracts.
