### Title
Proposer Storage Deposits Permanently Trapped in Contract When `do_update` Clears All Entries — (`crates/contract/src/update.rs`)

### Summary

When `propose_update` is called, each proposer must attach a storage deposit proportional to the size of the update payload. When `do_update` executes (triggered by `vote_update` reaching threshold), it clears **all** pending entries and votes — freeing the storage — but never refunds any proposer's deposit. The NEAR tokens remain locked in the contract permanently, with no recovery path.

### Finding Description

`propose_update` collects a deposit from the caller to cover storage costs: [1](#0-0) 

The required deposit is computed by `ProposedUpdates::required_deposit`, which deliberately over-estimates storage (assuming 128 participant votes per entry): [2](#0-1) 

Only the **excess** above the required amount is refunded to the proposer. The required portion stays in the contract.

When `vote_update` reaches threshold, it calls `do_update`, which removes the executed entry and then calls `entries.clear()` and `vote_by_participant.clear()`, freeing all storage for all proposals: [3](#0-2) 

At this point the storage is freed (NEAR's storage-staking lock is released back to the contract's liquid balance), but **no deposit is returned to any proposer**. The `UpdateEntry` struct does not even record the proposer's account ID, making a refund structurally impossible without a redesign: [4](#0-3) 

### Impact Explanation

Every contract upgrade execution permanently traps all proposers' storage deposits in the contract. For a realistic contract binary (~100 KB), the deposit is approximately 10 NEAR per proposal (at NEAR's ~1 NEAR / 10 KB storage cost), plus the over-estimated vote overhead. If multiple participants each propose a competing update (a normal governance pattern), all of their deposits are lost simultaneously when any one update is executed. The funds are not stolen by an external party but are permanently inaccessible to the rightful owners (the proposers), breaking the accounting invariant that storage deposits are returned when storage is freed. This maps to the **Medium** allowed impact: balance/accounting invariant violation in the contract execution flow.

### Likelihood Explanation

This is triggered by the normal, intended operation of the contract. Every successful contract upgrade or config update execution causes the loss. Given that upgrades are a routine governance action, the probability of this occurring in production is certain rather than probabilistic.

### Recommendation

1. Add a `proposer: AccountId` field to `UpdateEntry` to record who paid the deposit.
2. Add a `deposit_paid: NearToken` field to record the exact amount held.
3. In `do_update`, before clearing entries, iterate over all entries and issue `Promise::new(entry.proposer).transfer(entry.deposit_paid)` for each one.
4. Alternatively, maintain a separate `BTreeMap<UpdateId, (AccountId, NearToken)>` for deposit tracking so that refunds survive the `entries.clear()` call.

### Proof of Concept

1. Participant A calls `propose_update` with a 100 KB contract binary, attaching ~10 NEAR deposit. Only excess above `required_deposit` is refunded; ~10 NEAR stays in the contract.
2. Participant B calls `propose_update` with a different binary, also attaching ~10 NEAR.
3. Threshold participants call `vote_update` for A's proposal. `do_update` is triggered.
4. `entries.clear()` removes both A's and B's entries, freeing ~20 NEAR of storage staking back to the contract's liquid balance.
5. Neither A nor B receives any refund. The ~20 NEAR is permanently absorbed into the contract's balance with no withdrawal mechanism. [5](#0-4) [6](#0-5)

        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {

                    NearToken::from_near(0),

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
