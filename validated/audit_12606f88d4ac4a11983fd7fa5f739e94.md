### Title
ETH Withdrawal Permanently Freezes Funds for Contract Recipients Unable to Receive ETH - (File: contracts/LRTWithdrawalManager.sol)

### Summary
The two-step ETH withdrawal process in `LRTWithdrawalManager` permanently freezes user funds when the withdrawing address is a smart contract without a `receive()` function. Because rsETH is burned in a prior `unlockQueue()` transaction and ETH is moved to the withdrawal manager, a subsequent `completeWithdrawal()` that always reverts (due to the recipient being unable to receive ETH) leaves the user with permanently burned rsETH and inaccessible ETH.

### Finding Description
The withdrawal lifecycle in `LRTWithdrawalManager` is split across two separate transactions:

**Step 1 — `initiateWithdrawal()`**: The user's rsETH is transferred from `msg.sender` to the withdrawal manager. The user's address (`msg.sender`) is permanently recorded as the beneficiary. [1](#0-0) 

**Step 2 — `unlockQueue()`** (operator-only): rsETH held by the withdrawal manager is burned, and ETH is pulled from the unstaking vault into the withdrawal manager. This is irreversible. [2](#0-1) 

**Step 3 — `completeWithdrawal()`**: ETH is sent to the recorded user address via `_transferAsset()`. [3](#0-2) 

The ETH transfer uses a plain `call` that reverts on failure:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` is a contract without a `receive()` function (or one whose `receive()` reverts), every call to `completeWithdrawal()` will revert. Because all state changes inside `_processWithdrawalCompletion` are rolled back on revert, the withdrawal request remains perpetually unlocked but uncompletable. [4](#0-3) 

The rsETH burned in Step 2 is **not** rolled back — it was destroyed in a prior, already-finalized transaction. The ETH sits in the withdrawal manager with no recovery path:

- `sweepRemainingAssets()` requires `!hasUnlockedWithdrawals(asset)`, which is false as long as the stuck withdrawal exists.
- `completeWithdrawalForUser()` also calls `_processWithdrawalCompletion(asset, user, ...)` with the same non-receiving `user` address, so it also always reverts. [5](#0-4) [6](#0-5) 

### Impact Explanation
A user whose address is a smart contract without a `receive()` function (e.g., a DAO treasury, a smart contract wallet without ETH receive capability, or any contract that reverts on ETH receipt) will:

1. Have their rsETH permanently burned (Step 2 is irreversible).
2. Be unable to ever receive their ETH (Step 3 always reverts).
3. Have no protocol-level recovery path.

This constitutes **permanent freezing of user funds** — the ETH is locked in the withdrawal manager indefinitely and the rsETH is gone. Impact: **Critical**.

### Likelihood Explanation
Smart contracts regularly interact with DeFi protocols as first-class participants: DAO treasuries, protocol-owned liquidity managers, yield aggregators, and smart contract wallets. Not all of these implement a `receive()` function. Any such contract that calls `initiateWithdrawal()` for ETH will trigger this freeze once the operator runs `unlockQueue()`. The developer comment `/// @dev Not expected to be used for ETH` on `completeWithdrawalForUser` acknowledges awareness of ETH-transfer edge cases but does not prevent the scenario. Likelihood: **Low-Medium** (requires a contract depositor without ETH receive capability, which is uncommon but realistic).

### Recommendation
1. **Preferred fix**: In `_processWithdrawalCompletion`, if the ETH transfer to `user` fails, do not revert. Instead, record the owed amount in a claimable mapping and allow the user to pull their ETH via a separate `claimETH()` function (pull-over-push pattern). This ensures rsETH burn and ETH accounting are never left in an inconsistent state.

2. **Alternative**: Validate at `initiateWithdrawal()` time that if `asset == ETH_TOKEN` and `msg.sender` is a contract, it must be able to receive ETH (e.g., check `msg.sender.code.length == 0` or require a test transfer). This prevents the problematic state from ever being created.

3. **Minimum mitigation**: Add an admin-callable `rescueStuckWithdrawal(address asset, address user, address alternateRecipient)` function that can redirect a permanently stuck ETH withdrawal to an alternate address, bypassing the frozen state.

### Proof of Concept
1. Deploy a contract `NoReceive` with no `receive()` function that holds rsETH and calls `initiateWithdrawal(ETH_TOKEN, amount, "")`.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH is burned, ETH moves to withdrawal manager.
3. `NoReceive` calls `completeWithdrawal(ETH_TOKEN, "")` — reverts with `EthTransferFailed`.
4. Repeat step 3 indefinitely — always reverts.
5. Observe: rsETH balance of `NoReceive` is 0 (burned), ETH balance of withdrawal manager includes the stuck amount, `unlockedWithdrawalsCount[ETH_TOKEN] > 0` (preventing `sweepRemainingAssets`). Funds are permanently frozen.

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
