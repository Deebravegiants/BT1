### Title
Unverified `CKDResponse` for `AppPublicKey` Variant Allows Single Malicious Participant to Corrupt Key Derivation Output — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract applies a cryptographic validity check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`. For the `AppPublicKey` variant, **no verification of the response is performed**. A single malicious attested participant (below the signing threshold) can call `respond_ckd` with an arbitrary `CKDResponse`, causing the contract to mark the request as completed and deliver incorrect key material to the caller — permanently without recourse, because `resolve_yields_for` drains the entire fan-out queue in one pass.

---

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 653–689), the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

For `AppPublicKeyPV`, `ckd_output_check` cryptographically binds the response to the master BLS12-381 public key and the app's public key, making forgery infeasible for a sub-threshold actor. For `AppPublicKey`, the arm is an empty no-op. Any attested participant can supply any `CKDResponse` — including random bytes or a response crafted for a different request — and the contract will:

1. Remove the request from `pending_ckd_requests` (via `resolve_yields_for`).
2. Resume every queued yield with the attacker-supplied bytes.
3. Return the corrupted response to all callers who submitted the same request.

Because `resolve_yields_for` drains the entire queue atomically, subsequent honest-node calls to `respond_ckd` for the same request will fail with `RequestNotFound` — the window for correction is permanently closed.

The analog to the external report is direct: just as the Solidity `cast` function marks a spell "done" without checking whether `exec` actually succeeded, `respond_ckd` marks a CKD request "done" without checking whether the supplied key material is correct for the `AppPublicKey` variant.

---

### Impact Explanation

A caller who receives a corrupted `CKDResponse` will derive an incorrect private key. If that key is used to receive funds on a foreign chain (the primary use-case for CKD), those funds are permanently frozen: the address is derived from the wrong public key, and no one holds the corresponding private key. This constitutes **permanent freezing of funds** in the verified foreign-chain flow, matching the Critical impact tier.

Additionally, because `resolve_yields_for` fans the corrupted response out to every duplicate submission queued under the same request key (up to `MAX_PENDING_REQUEST_FAN_OUT = 128`), a single malicious call can simultaneously corrupt up to 128 independent callers' key derivation results.

---

### Likelihood Explanation

**Medium-High.** The attacker must be an attested MPC participant — a role the system explicitly models as potentially Byzantine below the signing threshold. No threshold cooperation is required; a single participant suffices. The attack surface is open on every `request_app_private_key` call that uses the `AppPublicKey` variant. The attacker only needs to submit their `respond_ckd` transaction before the honest node's transaction is included; NEAR's transaction ordering does not prevent this race.

---

### Recommendation

Apply the same cryptographic binding used for `AppPublicKeyPV` to the `AppPublicKey` variant. Concretely, verify that the response's `big_y` field is the correct BLS12-381 derivation of the master public key for the given `app_id`, using the master public key already stored in contract state. If a closed-form on-chain check is not feasible for this variant, require threshold-many identical responses before accepting (mirroring the `vote_pk` / `vote_reshared` pattern used for key-event governance).

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk)`.
2. Contract stores the request in `pending_ckd_requests` and parks the caller's yield.
3. Malicious attested participant reads the pending request from chain state.
4. Malicious participant calls `respond_ckd(request, CKDResponse { big_y: [0u8;48].into(), big_c: [0u8;48].into() })`.
5. Contract enters the `AppPublicKey` branch — empty arm, no check — then calls `resolve_yields_for`, which removes the request and resumes all queued yields with the attacker-supplied bytes.
6. Honest nodes' subsequent `respond_ckd` calls return `Err(RequestNotFound)`.
7. Caller's yield fires with the corrupted `CKDResponse`; caller derives an incorrect private key.
8. Caller uses the incorrect public key to receive funds on a foreign chain; funds are permanently frozen.

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
