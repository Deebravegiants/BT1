### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never verifies that `response.payload_hash` is the correct hash for the specific `request` being resolved. A single Byzantine attested participant (below the signing threshold) can replay any past legitimate MPC signature — whose `(payload_hash, signature)` pair is permanently visible on-chain — as a response to a completely different pending foreign-tx request, causing that request's yield queue to be drained with fabricated attestation data.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs exactly two security checks before resolving the pending yield queue:

1. **Signature validity**: `verify_ecdsa_signature(signature_response, &payload_hash, &secp_pk)` — confirms the MPC network signed *something* with the domain key.
2. **Queue membership**: `resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &request, …)` — confirms a pending yield exists for `request`. [1](#0-0) 

What is **absent** is any check that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: _ })))`. The `payload_hash` is a free caller-supplied field; the contract never reconstructs or bounds-checks it against the `ForeignChainRpcRequest` embedded in the pending `request`.

Compare this with the regular `respond` path for signature requests, where the contract verifies the signature directly against `request.payload` — a value stored on-chain at submission time, providing a hard binding between response and request: [2](#0-1) 

The correct `payload_hash` for a foreign-tx response is defined as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: ForeignChainRpcRequest,   // same as request.request
    values:  Vec<ExtractedValue>,      // NOT stored on-chain
})))
``` [3](#0-2) 

Because `values` are never stored on-chain, the contract cannot reconstruct the expected hash — but it also performs **no partial check** (e.g., verifying that the hash was computed from a payload whose `request` field matches `request.request`). The SDK's `ForeignChainSignatureVerifier` does perform this check client-side, but the contract itself enforces nothing: [4](#0-3) 

---

### Impact Explanation

A Byzantine attested participant (strictly below the signing threshold) can execute the following replay:

1. **Observe** a past legitimate `respond_verify_foreign_tx` call for request **T1** on-chain. The arguments `(payload_hash_T1, signature_T1)` are permanently public.
2. **Wait** for any user to submit `verify_foreign_transaction(T2)` — a different foreign-chain transaction — creating a pending yield for **T2**.
3. **Call** `respond_verify_foreign_tx(request=T2, response={payload_hash=payload_hash_T1, signature=signature_T1})`.
4. The contract accepts: `signature_T1` is a valid ECDSA signature over `payload_hash_T1` under the domain key ✓; T2 exists in the pending queue ✓.
5. T2's entire yield queue is drained and all waiting callers receive `{payload_hash=payload_hash_T1, signature=signature_T1}` — T1's attestation data, not T2's.

Any bridge contract that does not independently verify the `payload_hash` binding (i.e., does not call `ForeignChainSignatureVerifier::verify_signature` with its own expected extracted values) will accept T1's foreign-chain attestation as proof that T2 was verified. This enables **forged foreign-chain verification** — the bridge releases funds or triggers actions based on a transaction that was never actually inspected by the MPC network for that request. Even when the bridge does verify, T2's yield is permanently consumed with wrong data, permanently blocking T2's callers from receiving a correct response (irreversible DoS).

This matches: *"High. Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- **Attacker capability**: A single Byzantine attested participant below the signing threshold. No threshold collusion is required; the replayed signature was produced legitimately by the full network for a prior request.
- **Data availability**: `payload_hash` and `signature` from any past `respond_verify_foreign_tx` call are permanently on-chain and trivially observable.
- **Target availability**: Any pending `verify_foreign_transaction` request is a valid target.
- **Bridge contract behavior**: The `ForeignChainSignatureVerifier` SDK helper is optional; bridge contracts are not forced to use it. Omnibridge-style contracts that trust the MPC contract's output without re-verifying the `payload_hash` binding are directly exploitable.

---

### Recommendation

The contract must bind `response.payload_hash` to `request.request` before resolving yields. Two concrete options:

1. **Store the request content at submission time** and verify on response that `payload_hash` was computed from a `ForeignTxSignPayload` whose `request` field matches the stored value. This requires persisting `ForeignChainRpcRequest` alongside the yield index.

2. **Require the responder to supply `values`** alongside the response, reconstruct `expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values })))`, and assert `expected_hash == response.payload_hash` before accepting.

Option 2 is the stronger fix: it makes the on-chain verification self-contained and eliminates reliance on client-side `ForeignChainSignatureVerifier` for security.

### Proof of Concept

```
// Step 1 — attacker reads past on-chain call:
//   respond_verify_foreign_tx(request=T1, response={payload_hash_T1, sig_T1})

// Step 2 — victim submits a new request:
//   verify_foreign_transaction(T2)   →  T2 enters pending_verify_foreign_tx_requests

// Step 3 — attacker (Byzantine attested participant) calls:
respond_verify_foreign_tx(
    request = T2,                          // valid pending request key
    response = {
        payload_hash: payload_hash_T1,     // hash from T1's legitimate response
        signature:    sig_T1,              // MPC signature over payload_hash_T1
    }
)

// Contract path:
//   verify_ecdsa_signature(sig_T1, payload_hash_T1, domain_pk)  → OK  ✓
//   resolve_yields_for(pending_verify_foreign_tx_requests, T2, response_bytes)
//       → drains T2's yield queue, resumes all T2 callers with T1's data  ✓

// T2's bridge contract receives {payload_hash_T1, sig_T1}.
// If it skips ForeignChainSignatureVerifier:
//   → accepts T1's attestation as proof T2 was verified  → invalid bridge execution
// If it uses ForeignChainSignatureVerifier:
//   → rejects (hash mismatch), but T2's yield is permanently consumed → irreversible DoS
```

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
