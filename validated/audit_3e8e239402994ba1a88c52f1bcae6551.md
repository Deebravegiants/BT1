Audit Report

## Title
Single Depositor Can Exhaust Global Daily Mint Limit, DoS-ing All Subsequent Deposits for Up to 24 Hours - (File: contracts/RSETH.sol)

## Summary
`RSETH.sol` enforces a single shared `currentPeriodMintedAmount` counter with no per-user sub-limit. Any depositor who supplies enough ETH or LST to consume the remaining `maxMintAmountPerDay` quota in one transaction will cause every subsequent `RSETH.mint()` call to revert with `DailyMintLimitExceeded` for up to 24 hours. The attack is fully permissionless, repeatable each period, and requires no permanent capital loss.

## Finding Description
The `checkDailyMintLimit` modifier in `RSETH.sol` maintains a single global counter:

```solidity
modifier checkDailyMintLimit(uint256 amount) {
    if (block.timestamp >= periodStartTime + 1 days) {
        currentPeriodMintedAmount = 0;
        periodStartTime = getCurrentPeriodStartTime();
    }
    if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
        revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
    }
    currentPeriodMintedAmount += amount;
    _;
}
``` [1](#0-0) 

`RSETH.mint()` applies this modifier on every call: [2](#0-1) 

The public deposit entry points `depositETH` and `depositAsset` in `LRTDepositPool.sol` are callable by any address with no per-user quota: [3](#0-2) 

They both terminate in `_mintRsETH`, which calls `RSETH.mint` directly: [4](#0-3) 

`_beforeDeposit` only enforces a minimum deposit amount and a total protocol deposit cap — there is no per-user daily sub-limit: [5](#0-4) 

**Exploit path:**
1. Attacker calls `LRTDepositPool.depositETH{value: V}(0, "")` where `V` is sized so that `getRsETHAmountToMint` returns a value equal to the remaining `maxMintAmountPerDay - currentPeriodMintedAmount`.
2. `_mintRsETH` calls `RSETH.mint(attacker, rsethAmountToMint)`.
3. `checkDailyMintLimit` passes (quota exactly met) and sets `currentPeriodMintedAmount = maxMintAmountPerDay`.
4. Every subsequent `RSETH.mint()` call reverts with `DailyMintLimitExceeded` until `block.timestamp >= periodStartTime + 1 days`.
5. Attacker repeats at the start of each new period.

No existing guard prevents this: the deposit cap (`depositLimitByAsset`) is a lifetime total, not a per-period or per-user limit, and there is no per-address minting quota anywhere in the codebase.

## Impact Explanation
**Medium — Temporary freezing of funds.**

All honest depositors are completely blocked from depositing ETH or LSTs into the protocol for up to 24 hours per attack cycle. Both `depositETH` and `depositAsset` revert because they unconditionally call `RSETH.mint`, which is gated by the exhausted daily limit. This matches the allowed impact class "Temporary freezing of funds."

## Likelihood Explanation
**Medium.**

The attacker must supply capital equal to the remaining daily quota. They receive rsETH in return, so the only cost is opportunity cost plus the EigenLayer withdrawal queue lock-up period (7+ days). No privileged role is required — only a standard `depositETH` call. A well-capitalised actor (competing protocol, large holder, or griefing party) can execute this at the start of every 24-hour window indefinitely.

## Recommendation
1. **Per-user sub-limits**: Track `mintedPerPeriod[address]` and cap each address's contribution to a configurable fraction of `maxMintAmountPerDay` per period.
2. **Per-transaction cap**: Reject any single mint that would consume more than a configurable percentage (e.g., 10%) of `maxMintAmountPerDay` in one call.
3. **Minimum deposit cooldown**: Enforce a per-address cooldown between large deposits to prevent rapid quota exhaustion within a single period.

## Proof of Concept
```
Precondition:
  maxMintAmountPerDay = D (e.g., 1000 ETH worth of rsETH)
  currentPeriodMintedAmount = 0 (fresh period)
  minAmountToDeposit = 0 (or attacker deposits above it)

Step 1: Attacker calls LRTDepositPool.depositETH{value: V}(0, "")
        where V is chosen so getRsETHAmountToMint(ETH, V) == D.

Step 2: _beforeDeposit passes (amount > 0, below total deposit cap).
        _mintRsETH calls RSETH.mint(attacker, D).
        checkDailyMintLimit(D): 0 + D == maxMintAmountPerDay → passes.
        currentPeriodMintedAmount is now D.

Step 3: Honest user calls LRTDepositPool.depositETH{value: 1 ether}(0, "").
        RSETH.mint runs checkDailyMintLimit(rsethForUser):
        D + rsethForUser > maxMintAmountPerDay → REVERT DailyMintLimitExceeded.

Step 4: All deposits revert until block.timestamp >= periodStartTime + 1 days.
        Attacker repeats at the start of the next period.

Foundry test sketch:
  1. Deploy protocol with maxMintAmountPerDay = 1000e18.
  2. vm.prank(attacker); depositPool.depositETH{value: V}(0, "");
  3. vm.prank(victim);  depositPool.depositETH{value: 1 ether}(0, "");
     → expect revert DailyMintLimitExceeded.
  4. vm.warp(block.timestamp + 1 days + 1);
  5. vm.prank(victim);  depositPool.depositETH{value: 1 ether}(0, "");
     → succeeds, confirming the 24-hour DoS window.
```
