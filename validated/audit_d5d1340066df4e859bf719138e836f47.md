### Title
Expired TEE Attestations Not Re-Verified Before Node API Authorization in `is_caller_an_attested_participant()` — (File: `crates/contract/src/tee/tee_state.rs`)

---

### Summary

`is_caller_an_attested_participant()` — the gating check for every node-facing contract call (`respond`, `vote_reshared`, `start_reshare_instance`, `respond_ckd`, `respond_verify_foreign_tx`) — only confirms that an attestation *exists* in `stored_attestations` and that the stored account/key fields match the caller. It never re-verifies whether the stored attestation has since expired. The function that does perform expiry re-verification, `reverify_and_cleanup_participants()` / `clean_invalid_attestations()`, is never invoked before the authorization check. As a result, a node whose TEE attestation has expired can continue to call all node API endpoints — including submitting threshold signatures — for an unbounded window until `verify_tee()` or `clean_invalid_attestations()` is explicitly triggered by another participant.

---

### Finding Description

`assert_caller_is_attested_participant_and_protocol_active()` is the single authorization gate for all node-facing calls: [1](#0-0) 

It delegates to `is_caller_an_attested_participant()`: [2](#0-1) 

This function performs three checks: (1) the caller is in the active participant set, (2) an entry exists in `stored_attestations` keyed by the participant's TLS public key, and (3) the stored `account_id` and `account_public_key` match the caller. It does **not** inspect `verified_attestation` for expiry or re-run any verification logic.

The function that *does* check expiry is `reverify_and_cleanup_participants()`, called only inside `verify_tee()` and `vote_new_parameters()`: [3](#0-2) 

The separate `clean_invalid_attestations()` endpoint removes expired entries from `stored_attestations`: [4](#0-3) 

After a successful resharing, `clean_invalid_attestations` is spawned as a **detached** promise — it runs asynchronously and is not guaranteed to complete before the next node API call: [5](#0-4) 

Between the moment an attestation expires and the moment `verify_tee()` or `clean_invalid_attestations()` is explicitly invoked, the expired entry remains in `stored_attestations`. During this window, `is_caller_an_attested_participant()` returns `Ok(())` for the expired node, and all node API calls succeed — including `respond()`: [6](#0-5) 

The `accept_requests` flag (also checked in `respond()`) is set by `verify_tee()`, which is not called automatically. If `accept_requests` is still `true` from a prior `verify_tee()` call made before the attestation expired, both guards pass.

---

### Impact Explanation

A Byzantine MPC node whose TEE attestation has expired — meaning the hardware-level guarantee that the node is running unmodified code in a secure enclave has lapsed — can continue to:

- Submit threshold signatures via `respond()`, `respond_ckd()`, `respond_verify_foreign_tx()`
- Cast votes in resharing via `vote_reshared()` and `start_reshare_instance()`

This breaks the production safety invariant that only nodes with a currently valid TEE attestation may participate in protocol operations. The TEE model is the primary mechanism preventing a compromised node from participating in key-share usage after its hardware guarantee has lapsed. Bypassing it allows a node that may have leaked its key share to continue contributing to threshold operations, weakening the effective security margin below the declared threshold without any on-chain signal.

This maps to: **Medium — participant-state and contract execution-flow manipulation that breaks production safety/accounting invariants.**

---

### Likelihood Explanation

Attestations carry explicit expiry timestamps (as confirmed by `MockAttestation::WithConstraints { expiry_timestamp_seconds }` and the `clean_invalid_attestations__should_remove_expired_entries` test). In production, DCAP attestations expire on a schedule. `verify_tee()` is not called on a fixed schedule — it is triggered by participants. `clean_invalid_attestations()` is bounded by `max_scan` and runs as a detached promise. Any node whose attestation expires between explicit cleanup calls can exploit this window. The window length is operator-dependent but can be arbitrarily long if no participant calls `verify_tee()`.

---

### Recommendation

`is_caller_an_attested_participant()` should call `reverify_participants()` on the retrieved `NodeAttestation` before returning `Ok(())`, rejecting callers whose stored attestation fails re-verification (expired, stale image hash, etc.). Alternatively, `assert_caller_is_attested_participant_and_protocol_active()` should invoke `reverify_participants()` inline and return an error if the attestation is no longer valid, mirroring the pattern already used in `verify_tee()` and `vote_new_parameters()`.

---

### Proof of Concept

1. Node A submits a valid attestation with `expiry_timestamp_seconds = T` via `submit_participant_info()`. The entry is stored in `stored_attestations`.
2. Block timestamp advances past `T`. Node A's attestation is now expired.
3. Neither `verify_tee()` nor `clean_invalid_attestations()` has been called since expiry. `stored_attestations` still contains Node A's entry. `accept_requests` is still `true`.
4. Node A calls `respond(request, valid_signature)`.
5. `assert_caller_is_attested_participant_and_protocol_active()` → `is_caller_an_attested_participant()` finds Node A's entry in `stored_attestations`, confirms account/key match, returns `Ok(())`. No expiry check is performed.
6. `accept_requests` is `true` → no rejection.
7. The cryptographically valid signature is accepted and the pending yield is resolved.

Node A, whose TEE guarantee has lapsed, has successfully submitted a threshold signature response, bypassing the TEE authorization invariant. [2](#0-1) [1](#0-0) [6](#0-5)

        max_scan: usize,

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
