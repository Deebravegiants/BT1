Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Per-Deposit Amount Validation — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch comparison, checking only whether the running total already exceeds the limit rather than whether `totalAssetDeposits + amount` would exceed it. Any unprivileged caller can overshoot the configured `depositLimitByAsset[ETH]` by an arbitrary margin in a single `depositETH` call, violating the protocol's deposit cap invariant.

## Finding Description
At `contracts/LRTDepositPool.sol` lines 676–682, the ETH and ERC20 branches apply asymmetric logic:

```solidity
// L678-679: ETH — amount is silently ignored
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
// L681: ERC20 — correctly includes incoming amount
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

`depositETH` (lines 76–93) is `external payable` with no access control. It calls `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected)`, which internally calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)`. Because the ETH branch discards `amount`, the guard returns `false` (not exceeded) whenever `totalAssetDeposits ≤ limit`, regardless of how large `msg.value` is. The full `msg.value` is then accepted and rsETH is minted. Existing guards (`nonReentrant`, `whenNotPaused`, `onlySupportedAsset`) do not address the missing amount inclusion.

## Impact Explanation
The deposit cap is the protocol's primary mechanism for controlling ETH exposure. Bypassing it allows the protocol to accept more ETH than operators intended, breaking the invariant `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset[ETH]` that the rest of the system relies on. `getAssetCurrentLimit` will retroactively return `0` while deposits already succeeded beyond the cap. No funds are stolen or frozen; the contract fails to enforce its promised deposit ceiling. This matches the allowed impact: **Low — Contract fails to deliver promised returns**.

## Likelihood Explanation
`depositETH` is public and payable with no access control. The only precondition is that `totalAssetDeposits[ETH] ≤ depositLimitByAsset[ETH]`, which is the normal operating state. No flash loans, front-running, privileged keys, or victim mistakes are required. Any user can trigger this in a single transaction.

## Recommendation
Include the incoming deposit amount in the ETH branch, matching the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Set `depositLimitByAsset[ETH_TOKEN] = 100_000 ether`, `getTotalAssetDeposits(ETH_TOKEN) = 99_999 ether`.
2. Call `depositETH{value: 50_000 ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50_000 ether)` evaluates `99_999 ether > 100_000 ether` → `false` → deposit proceeds.
4. rsETH is minted for the full 50,000 ETH.
5. Post-call: `getTotalAssetDeposits(ETH_TOKEN) = 149_999 ether` (49,999 ETH above the cap); `getAssetCurrentLimit(ETH_TOKEN)` returns `0`.

**Foundry fuzz test plan**: Fuzz `depositETH` with `amount > getAssetCurrentLimit(ETH_TOKEN)` while `totalAssetDeposits ≤ limit`; assert that `getTotalAssetDeposits(ETH_TOKEN) ≤ depositLimitByAsset[ETH_TOKEN]` after each call. The invariant will fail, confirming the bug.
