Audit Report

## Title
Missing Chainlink `latestRoundData()` Return Value Validation Enables Stale Price Exploitation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `roundId`, `updatedAt`, and `answeredInRound`, accepting any price including stale values without validation. The same protocol already implements correct validation in `ChainlinkOracleForRSETHPoolCollateral`, making the omission in the core oracle inconsistent. A stale inflated price fed into `LRTDepositPool.getRsETHAmountToMint` allows an attacker to mint excess rsETH, extracting value from existing rsETH holders.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` silently discards all safety-critical fields from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` validates all three fields:

```solidity
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

`ChainlinkPriceOracle` is the oracle adapter registered in `LRTOracle` for LST assets. Its output flows directly into `getRsETHAmountToMint`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

And into `_getTotalEthInProtocol` via `LRTOracle._updateRsETHPrice`, which computes the rsETH price used as the denominator: [4](#0-3) 

The exploit path:
1. A Chainlink LST/ETH feed (e.g., stETH/ETH, 24-hour heartbeat) has not updated within its heartbeat window. `updatedAt` is stale but no revert condition exists in `ChainlinkPriceOracle`.
2. The stale price is above the true current value (e.g., 1.002 ETH/stETH vs. true 0.998 ETH/stETH).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, amount)` — a public, permissionless function.
4. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns the stale inflated price.
5. Attacker receives more rsETH than the deposited asset is worth.
6. Attacker initiates withdrawal, redeeming excess rsETH for ETH, extracting value from all existing rsETH holders.

No existing check in `_beforeDeposit` or `getRsETHAmountToMint` validates oracle freshness. [5](#0-4) 

## Impact Explanation
**High — Theft of unclaimed yield.**

When a Chainlink LST/ETH feed enters a stale state with a price above the true current value, an attacker can deposit LST at the inflated oracle price and receive more rsETH than the deposited asset is worth. Upon redemption, the attacker extracts value proportional to the price deviation and deposit size from all existing rsETH holders, constituting theft of unclaimed yield. The magnitude scales with deposit size and price deviation, both of which are attacker-controllable within protocol deposit limits.

## Likelihood Explanation
Chainlink LST/ETH feeds have documented heartbeat intervals (up to 24 hours for some LST feeds). During low-volatility periods, feeds may not update for the full heartbeat window. An attacker monitoring on-chain feed `updatedAt` timestamps can identify the stale window and act within it. No privileged access is required; `depositAsset` on `LRTDepositPool` is open to any user. The condition is passively observable on-chain and repeatable across multiple heartbeat windows.

## Recommendation
Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();
require(answeredInRound >= roundId, "Stale price");
require(updatedAt != 0, "Incomplete round");
require(price > 0, "Invalid price");
```

Additionally, add a configurable `stalePriceThreshold` (e.g., `block.timestamp - updatedAt <= maxStaleness`) to bound the acceptable age of a price, consistent with each feed's documented heartbeat interval.

## Proof of Concept
Foundry fork test plan:

1. Fork mainnet at a block where a Chainlink LST/ETH feed (e.g., stETH/ETH) has not updated for its full heartbeat window — `updatedAt` is stale but `answeredInRound == roundId`.
2. Deploy or reference the existing `ChainlinkPriceOracle` registered in `LRTOracle` for stETH.
3. Call `ChainlinkPriceOracle.getAssetPrice(stETH)` — confirm it returns the stale price without reverting.
4. As an unprivileged attacker, call `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
5. Record `rsethMinted` from the `AssetDeposit` event.
6. Compute fair rsETH using the true current stETH/ETH rate; assert `rsethMinted > fairAmount`.
7. Initiate withdrawal of the excess rsETH and confirm the attacker recovers more ETH value than deposited, at the expense of existing rsETH holders' share of protocol TVL.
