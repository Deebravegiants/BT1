### Title
Attached NEAR Deposit Permanently Frozen in `submit_participant_info` When Caller Is an Existing Participant — (File: `crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is marked `#[payable]` and therefore accepts an attached NEAR deposit. However, when the caller is an **existing participant** updating their attestation (not a new insertion), the function's deposit-handling branch is entirely skipped, and any attached NEAR is silently absorbed into the contract with no refund path.

---

### Finding Description

The function `submit_participant_info` computes a boolean gate to decide whether to charge and refund the caller:

```rust
// crates/contract/src/lib.rs  lines 817-848
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... charge for storage bytes used ...
    // Refund the difference if the proposer attached more than required
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
``` [1](#0-0) 

When both conditions are `false` — i.e., the caller **is** a participant (`caller_is_not_participant = false`) **and** the attestation is an **update** rather than a new insertion (`is_new_attestation = false`) — the entire `if` block is bypassed. No refund is issued. Because the function is `#[payable]`, any NEAR tokens attached to such a call are permanently locked in the contract. [2](#0-1) 

The analogous root cause to the original report is exact: a payable entry point accepts native tokens unconditionally, but the refund logic is gated on a condition that is `false` for a legitimate, common call path (existing participant re-attesting after a TEE upgrade), leaving the deposit with no recovery mechanism.

---

### Impact Explanation

Any NEAR tokens attached by an existing participant calling `submit_participant_info` to refresh their attestation are permanently frozen inside the MPC contract. There is no withdrawal function, no admin sweep, and no subsequent code path that returns the deposit. This breaks the production accounting invariant that `#[payable]` functions must either consume or refund every attached deposit.

**Impact: Medium** — balance/accounting invariant broken, funds permanently frozen, no reliance on network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

MPC nodes are automated software that periodically re-submit attestations (e.g., after TEE image upgrades). Node operators or the node software itself may attach a small NEAR deposit when calling `submit_participant_info` — either by convention, by copy-paste from the initial submission flow, or because the caller cannot know in advance whether the contract will classify the call as "new" or "update." Every such call by an already-registered participant silently discards the deposit. The scenario is reachable by any unprivileged existing participant without any collusion.

---

### Recommendation

Move the refund logic **outside** the `if attestation_storage_must_be_paid_by_caller` block so that any excess deposit is always returned to the caller, regardless of whether storage was charged:

```rust
if attestation_storage_must_be_paid_by_caller {
    let storage_used = env::storage_usage().saturating_sub(initial_storage);
    let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
    let attached = env::attached_deposit();
    if attached < cost {
        return Err(...);
    }
    // refund excess inside this block
    refund_excess(attached, cost, &account_id);
} else {
    // Always refund the full deposit when no storage is charged
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, restructure so a single unconditional refund-excess call covers all branches, mirroring the pattern already used in `require_deposit` and `propose_update`. [3](#0-2) [4](#0-3) 

---

### Proof of Concept

1. Participant `alice.near` is already registered (i.e., `self.voter_account()` returns `Ok`).
2. Alice's TEE image is upgraded; she calls `submit_participant_info` with a new attestation and attaches `1 NEAR` as a deposit (matching the pattern used during initial registration).
3. Inside the function:
   - `attestation_insertion_result` is `ParticipantInsertion::UpdatedExistingParticipant` → `is_new_attestation = false`.
   - `self.voter_account().is_err()` → `false` (Alice is a participant).
   - `attestation_storage_must_be_paid_by_caller = false || false = false`.
4. The `if` block at line 826 is skipped entirely.
5. The function returns `Ok(())`. Alice's `1 NEAR` is now held by the contract with no mechanism to recover it. [5](#0-4)
