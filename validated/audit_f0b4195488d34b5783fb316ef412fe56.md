## Assessment

The chain is real and fully traceable. However, the impact must be evaluated carefully against the allowed scope:

**What is actually wrong:** When the pending block is stale, `starknet_estimateFee` with `block_id=Tag::Pending` computes fees using `NonzeroGasPrice::MIN` (1) for `l1_data_gas_price` and `l2_gas_price` instead of the values from the latest accepted block header. The `l1_gas_price` is correctly copied from the latest header, so only the data-gas and L2-gas components of the fee estimate are wrong.

**What is NOT wrong:** The fallback is an intentional design choice for the stale-pending case. The code comments explicitly state "In older versions, data gas price was 0" — the `DeprecatedPendingBlock` type is used precisely because it represents a pre-0.13.1 block format that lacks these fields. The fallback is not a bug in the sense that it is a deliberate degraded-mode response. Furthermore, the latest header's `l1_data_gas_price` and `l2_gas_price` are available in `latest_header.block_header_without_hash` but are simply not copied into the fallback — this is a design gap, not an accidental omission of a field that exists in `DeprecatedPendingBlock`.

**Stale condition reachability:** The stale condition occurs naturally during normal operation (new block produced, pending sync lag). An attacker cannot force it on demand — they can only observe it opportunistically. There is no attacker-controlled trigger.

**Economic impact:** The fee estimate returned will understate the data-gas and L2-gas cost components. A caller relying on this estimate to set resource bounds may submit a transaction that gets rejected for insufficient fee. This is a wrong RPC value, but the economic harm flows to the caller themselves (their transaction may fail), not to the network or other parties. The sequencer's actual fee accounting is unaffected — this is purely an RPC estimation artifact.

**Fits "High. RPC execution, fee estimation... returns an authoritative-looking wrong value"?** Technically yes — the value is wrong. But the condition requires a naturally-occurring race (stale pending block), the wrong values affect only the caller's own fee planning, and the `l1_gas_price` (the dominant fee component for most transactions) is correctly set. The impact is limited to the data-gas and L2-gas components of the estimate.

Given the scope rules require a **concrete unprivileged path** with **exact corrupted value** and **economic impact** — the path exists, the corrupted values are `l1_data_gas_price = NonzeroGasPrice::MIN` and `l2_gas_price = NonzeroGasPrice::MIN` instead of the latest header values, and the impact is an authoritative-looking wrong fee estimate returned to any caller who queries `starknet_estimateFee` with `block_id=Tag::Pending` during a stale-pending window.

### Title
Stale-pending fallback in `read_pending_data` omits `l1_data_gas_price` and `l2_gas_price` from latest header, causing `starknet_estimateFee` to return wrong fee estimates using `NonzeroGasPrice::MIN` — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
When the cached pending block is stale, `read_pending_data` constructs a `DeprecatedPendingBlock` fallback that copies `l1_gas_price` from the latest accepted block header but leaves `l1_data_gas_price` and `l2_gas_price` at zero (the type has no such fields). Accessor methods on `PendingBlockOrDeprecated::Deprecated` return `GasPricePerToken::default()` for both. These zeros flow through `client_pending_data_to_execution_pending_data` into `create_block_context`, where `NonzeroGasPrice::new(0).unwrap_or(NonzeroGasPrice::MIN)` substitutes the minimum representable price. Any `starknet_estimateFee` or `starknet_simulateTransactions` call with `block_id=Tag::Pending` during a stale-pending window returns a fee estimate computed with `NonzeroGasPrice::MIN` for data gas and L2 gas instead of the actual latest-block values.

### Finding Description
`read_pending_data` at lines 1573–1593 of `api_impl.rs` constructs the fallback as `PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock { eth_l1_gas_price: ..., strk_l1_gas_price: ..., ..Default::default() })`. The `DeprecatedPendingBlock` struct has no `l1_data_gas_price` or `l2_gas_price` fields. The accessor `l1_data_gas_price()` on `PendingBlockOrDeprecated::Deprecated` unconditionally returns `GasPricePerToken::default()` (zero for both wei and fri). The same applies to `l2_gas_price()`. These zeros are passed through `client_pending_data_to_execution_pending_data` (lines 19–20 of `pending.rs`) into `ExecutionPendingData`, then into `create_block_context` (lines 384–395 of `lib.rs`), where `NonzeroGasPrice::new(0).unwrap_or(NonzeroGasPrice::MIN)` is used. The latest header's `l1_data_gas_price` and `l2_gas_price` fields exist and are non-zero on modern Starknet but are never read in the fallback path.

### Impact Explanation
`starknet_estimateFee` and `starknet_simulateTransactions` with `block_id=Tag::Pending` return fee estimates where the data-gas and L2-gas price components are `NonzeroGasPrice::MIN` (1) instead of the actual values from the latest accepted block. For v3 transactions (which use L2 gas), the overall fee estimate will be significantly understated. Callers using this estimate to set `resource_bounds` may submit transactions that are rejected for insufficient fee.

### Likelihood Explanation
The stale-pending condition occurs naturally during normal node operation whenever a new block is accepted and the pending sync has not yet updated. It is not attacker-controllable but is a routine operational state. Any caller querying `starknet_estimateFee` with `block_id=Tag::Pending` during this window receives the wrong estimate.

### Recommendation
In the stale-pending fallback in `read_pending_data`, copy `l1_data_gas_price` and `l2_gas_price` from `latest_header.block_header_without_hash` into the fallback pending data. Since `DeprecatedPendingBlock` lacks these fields, the fallback should either construct a `PendingBlockOrDeprecated::Current(PendingBlock { ... })` with all prices populated from the latest header, or the fallback `ExecutionPendingData` should be constructed directly (bypassing `DeprecatedPendingBlock`) with all gas price fields set from the latest header.

### Proof of Concept
1. Node has latest block N with non-zero `l1_data_gas_price` and `l2_gas_price` in its header.
2. The cached `pending_data` has `parent_block_hash` pointing to block N-1 (stale).
3. Caller sends `starknet_estimateFee([v3_tx], [], Tag::Pending)`.
4. `read_pending_data` detects mismatch, constructs `DeprecatedPendingBlock` fallback.
5. `l1_data_gas_price()` returns `GasPricePerToken { price_in_wei: 0, price_in_fri: 0 }`.
6. `create_block_context` sets `l1_data_gas_price = NonzeroGasPrice::MIN` for both eth and strk.
7. Returned `FeeEstimation` has `l1_data_gas_price = 1` (MIN) instead of the actual header value. [2](#0-1) [3](#0-2) [6](#0-5) [7](#0-6)
