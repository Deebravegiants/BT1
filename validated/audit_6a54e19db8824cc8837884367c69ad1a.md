Audit Report

## Title
Missing Minimum Output (Slippage) Protection in L2 RSETHPool `deposit()` Functions - (File: contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol)

## Summary
All L2 RSETHPool `deposit()` variants compute the rsETH/wrsETH output at execution time using a live oracle rate (`getRate()`) but accept no `minRsETHAmountExpected` parameter. If the oracle rate increases between transaction submission and execution, the user receives fewer rsETH/wrsETH than anticipated with no on-chain mechanism to revert. The L1 counterpart `LRTDepositPool.sol` explicitly enforces this guard, confirming the protocol intended the protection to exist.

## Finding Description
Every public `deposit()` function across the four L2 pool contracts delegates output calculation to `viewSwapRsETHAmountAndFee()`, which calls `getRate()` at execution time:

```solidity
// RSETHPool.sol L311-319
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

The result is immediately transferred/minted with no floor check:

- `RSETHPool.sol` L265–278 (ETH path) and L284–305 (token path): `IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount)`
- `RSETHPoolNoWrapper.sol` L231–244 and L250–271: `rsETH.safeTransfer(msg.sender, rsETHAmount)`
- `RSETHPoolV2ExternalBridge.sol` L289–301: `wrsETH.mint(msg.sender, rsETHAmount)`
- `RSETHPoolV3ExternalBridge.sol` L366–384 and L390–412: `wrsETH.mint(msg.sender, rsETHAmount)`

By contrast, `LRTDepositPool.sol` `_beforeDeposit()` (L648–669) enforces:
```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

Because rsETH is a yield-bearing token, its ETH rate is monotonically non-decreasing. Any oracle update that occurs between a user's transaction submission and its inclusion in a block will silently reduce the rsETH output. No adversarial action is required.

## Impact Explanation
A depositor receives fewer rsETH/wrsETH tokens than the rate observed at submission time, with no ability to revert or recover the difference. The deposited ETH/tokens remain in the pool at the updated (worse-for-user) rate. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The rsETH oracle rate is updated periodically by the protocol as staking rewards accrue. On L2 chains with short block times and frequent oracle updates, any pending deposit transaction can be executed after an oracle update in the same or a subsequent block. No attacker is required; the condition arises from normal protocol operation and affects every depositor on every supported L2 chain systematically.

## Recommendation
Add a `minRsETHAmountExpected` parameter to all `deposit()` overloads in every RSETHPool variant, mirroring the pattern in `LRTDepositPool.sol`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    ...
}
```

Apply the same pattern to all token-path overloads and to `RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`, and `RSETHPoolV3ExternalBridge`.

## Proof of Concept
1. User queries `viewSwapRsETHAmountAndFee(1 ether)` off-chain; oracle returns `rsETHToETHrate = 1.05e18`, expected output ≈ 0.952 wrsETH.
2. User submits `RSETHPool.deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, the oracle is updated to `rsETHToETHrate = 1.10e18`.
4. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 wrsETH` — ~4.5% less than expected.
5. User receives 0.909 wrsETH; the 1 ETH is retained by the pool at the new rate; the user cannot recover the shortfall.

**Foundry fork test outline:**
```solidity
function test_depositSlippage() public {
    uint256 depositAmount = 1 ether;
    uint256 expectedOut = pool.viewSwapRsETHAmountAndFee(depositAmount);  // snapshot rate
    // simulate oracle update (prank oracle updater, set higher rate)
    vm.prank(oracleUpdater);
    oracle.updateRate(newHigherRate);
    // deposit executes at new rate
    vm.prank(user);
    pool.deposit{value: depositAmount}("ref");
    uint256 actualOut = wrsETH.balanceOf(user);
    assertLt(actualOut, expectedOut);  // user received less, no revert possible
}
