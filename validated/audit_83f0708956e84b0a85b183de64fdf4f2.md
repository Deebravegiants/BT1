### Title
Stale Governance Votes from Removed Participants Persist After Resharing, Enabling Sub-Threshold Code-Hash Whitelisting — (`File: crates/contract/src/lib.rs`, `crates/contract/src/tee/tee_state.rs`)

### Summary

After a resharing completes, the contract spawns several detached cleanup promises. One of them — `clean_tee_status` — is responsible for removing stale governance votes (code-hash, launcher-hash, OS-measurement) cast by participants who are no longer in the new epoch. Unlike `vote_update`, which explicitly re-filters votes against the current participant set at counting time, the TEE governance vote functions (`vote_code_hash`, `vote_add_launcher_hash`, `vote_add_os_measurement`) rely entirely on this detached promise for correctness. If the promise fails silently, stale votes from removed participants persist and are counted toward the governance threshold, allowing a malicious image hash to be whitelisted with fewer than the required number of current participants' votes.

### Finding Description

When `vote_reshared` finalises a resharing, it transitions the contract to `Running` and then fires a set of detached cleanup promises: [1](#0-0) 

The `clean_tee_status` promise calls `TeeState::clean_non_participant_votes`, which replaces the three vote maps with filtered copies: [2](#0-1) 

Because the promise is detached (`.detach()`), any failure — insufficient gas, or the contract being in a non-`Running` state when the receipt executes (e.g., a concurrent `verify_tee` call triggers a second resharing before the cleanup receipt lands) — is silent and unrecoverable. `clean_tee_status` is `#[private]`, so no external caller can retry it. [3](#0-2) 

After the cleanup fails, stale votes remain in `votes`, `launcher_votes`, and `measurement_votes`. When a current participant subsequently calls `vote_code_hash`, the vote count returned by `tee_state.vote(...)` includes those stale entries: [4](#0-3) 

The threshold check at line 1425 fires on the raw count, not on a filtered count. The developers recognised this exact risk for `vote_update` and added an explicit re-filter there: [5](#0-4) 

The comment at line 1376 reads: *"This ensures correctness even if the cleanup promise in `MpcContract::vote_reshared()` fails."* No equivalent defence exists for `vote_code_hash`, `vote_add_launcher_hash`, or `vote_add_os_measurement`.

The test suite explicitly documents that without cleanup, stale votes are counted: [6](#0-5) 

### Impact Explanation

If a malicious code hash is whitelisted with fewer than threshold current-participant votes:

1. A node running the malicious image can submit a passing `submit_participant_info` attestation (the attestation verifier accepts the whitelisted hash).
2. That node can be voted into the participant set via `vote_new_parameters`.
3. Once a participant, the malicious node participates in threshold signing and key-resharing, with the ability to leak key shares or produce unauthorized signatures.

This breaks the production safety invariant: *"a TEE image hash may only be whitelisted if at least T current participants vote for it."*

**Allowed impact matched:** Medium — participant-state and contract execution-flow manipulation that breaks production safety/accounting invariants.

### Likelihood Explanation

The failure condition is a detached promise executing when the contract is not in `Running` state. A realistic trigger: `vote_reshared` transitions to `Running` and schedules the cleanup receipt; before that receipt is processed, an operator calls `verify_tee`, which detects an expired attestation and immediately transitions the contract back to `Resharing`. The `clean_tee_status` receipt then fails its state guard and returns an error that is silently discarded.

Pre-conditions required:
- At least one removed participant must have pre-voted for the target malicious hash before the resharing.
- The cleanup receipt must fail (achievable via the race above or via gas misconfiguration).
- One current participant must vote for the same hash after the resharing.

Below-threshold collusion (e.g., one or two malicious participants out of five) is sufficient.

### Recommendation

Apply the same defence-in-depth pattern used in `vote_update` to all three TEE governance vote functions. At counting time, filter the stored vote map against the current participant set before comparing to the threshold:

```rust
// In vote_code_hash (and analogously for launcher/measurement):
let valid_votes = threshold_parameters
    .participants()
    .participants()
    .iter()
    .filter(|(account_id, _, _)| {
        self.tee_state.votes.proposal_by_account
            .get(*account_id)
            .is_some_and(|h| *h == code_hash)
    })
    .count() as u64;

if valid_votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
```

This makes correctness independent of whether the cleanup promise succeeded, mirroring the explicit comment already present in `vote_update`.

### Proof of Concept

**Setup:** N=5, T=3. Participants: {P1, P2, P3, P4, P5}. P1 and P2 are adversarial.

1. P1 calls `vote_code_hash(H_evil)` → 1 vote stored.
2. P2 calls `vote_code_hash(H_evil)` → 2 votes stored.
3. Honest participants trigger a resharing that removes P1 and P2. New set: {P3, P4, P5}, T=3.
4. `vote_reshared` completes; the `clean_tee_status` detached promise fails (e.g., `verify_tee` races and puts the contract into `Resharing` before the receipt lands).
5. Stale votes for `H_evil` from P1 and P2 remain in `tee_state.votes`.
6. P3 (colluding, or independently choosing the same hash) calls `vote_code_hash(H_evil)`.
7. `tee_state.vote(H_evil, &p3_auth_id)` returns 3 (2 stale + 1 fresh) ≥ T=3.
8. `whitelist_tee_proposal(H_evil, ...)` executes — `H_evil` is now an allowed MPC image hash.
9. A node running the malicious image submits a valid attestation and is subsequently voted into the participant set, gaining access to threshold signing. [7](#0-6) [4](#0-3) [8](#0-7)

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
