### Title
Dijkstra Era Fee Validation Omits Sub-Transaction Reference Script Costs, Allowing Undercharged Fees - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era's `EraUTxO` instance binds `getMinFeeTxUtxo` to `getConwayMinFeeTxUtxo`, which computes the reference-script fee component using only the top-level transaction's inputs. Sub-transactions embedded in a Dijkstra top-level transaction can carry their own reference inputs pointing to large scripts, but those scripts are never counted in the minimum-fee check. A function that correctly aggregates reference-script sizes across the entire batch (`batchNonDistinctRefScriptsSize`) already exists in the same module but is not wired into the fee-validation path.

---

### Finding Description

**Root cause — wrong `getMinFeeTxUtxo` binding for `DijkstraEra`**

`DijkstraEra`'s `EraUTxO` instance delegates minimum-fee computation to the Conway helper:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs  line 141
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` passes the reference-script size to `getMinFeeTx` using `txNonDistinctRefScriptsSize`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs  lines 174-175
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
``` [2](#0-1) 

`txNonDistinctRefScriptsSize` only inspects the **top-level** transaction's `inputsTxBodyL` and `referenceInputsTxBodyL`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs  lines 183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

Sub-transactions are stored in `dtbrSubTransactions` of the top-level body and are never visited by this function.

**The correct Dijkstra-aware function already exists but is unused in fee validation**

`batchNonDistinctRefScriptsSize` sums reference-script sizes across the top-level transaction **and** every embedded sub-transaction:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs  lines 271-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [4](#0-3) 

**Fee check in `dijkstraUtxoTransition` uses the under-counting path**

The UTXO transition rule enforces the minimum fee via:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs  line 373
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
``` [5](#0-4) 

`validateFeeTooSmallUTxO` calls `getMinFeeTxUtxo`, which resolves to `getConwayMinFeeTxUtxo` for `DijkstraEra`. Because `getConwayMinFeeTxUtxo` calls `txNonDistinctRefScriptsSize` (top-level only), any reference scripts attached exclusively to sub-transaction inputs are invisible to the fee check.

**Sub-transaction body structure confirms the attack surface**

`DijkstraSubTxBodyRaw` carries its own `dstbrSpendInputs` and `dstbrReferenceInputs` fields: [6](#0-5) 

These inputs can point to UTxO entries that hold large reference scripts. Those scripts are deserialized and validated by nodes but are not counted in the minimum fee.

---

### Impact Explanation

The reference-script fee (`minFeeRefScriptCostPerByte` / `tierRefScriptFee`) was introduced specifically to price the node-side cost of deserializing scripts and to prevent DDoS via cheap-to-submit but expensive-to-validate transactions (see ADR-009). By omitting sub-transaction reference scripts from the minimum-fee calculation, an attacker can submit a Dijkstra top-level transaction whose sub-transactions collectively reference arbitrarily large scripts while paying only the fee for the top-level transaction's (possibly empty) reference-script set. The actual node validation cost is proportional to the total batch reference-script size, but the fee charged is proportional only to the top-level portion.

This directly modifies fees outside design parameters: the attacker pays less than the protocol-mandated minimum fee for the work imposed on validators.

**Allowed impact matched**: Medium — "Attacker-controlled transactions... modify fees... outside design parameters."

---

### Likelihood Explanation

The Dijkstra era is the current development tip. Any unprivileged user who can submit a valid Dijkstra transaction can exploit this: craft a top-level transaction with minimal or no reference inputs, embed one or more sub-transactions each referencing large-script UTxOs, and set the `fee` field to the (underestimated) Conway-style minimum. The ledger will accept it because `validateFeeTooSmallUTxO` computes the minimum fee without the sub-transaction reference-script component. No special privilege, key, or governance action is required.

---

### Recommendation

Override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance to use `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo pp tx utxo =
    getMinFeeTx pp tx (batchNonDistinctRefScriptsSize utxo tx)
```

This mirrors the pattern already used for `batchNonDistinctRefScriptsSize` in the Ledger rule and ensures the fee check accounts for the full batch of reference scripts, consistent with the node's actual validation cost.

---

### Proof of Concept

1. Deploy two UTxO entries each holding a large Plutus reference script (e.g., 50 KiB each, total 100 KiB).
2. Construct a Dijkstra top-level transaction with:
   - No reference inputs at the top level.
   - Two sub-transactions, each with one of the large-script UTxOs as a reference input.
3. Compute the minimum fee using `getMinFeeTxUtxo` (Conway path): it returns `getMinFeeTx pp tx 0` — zero reference-script bytes, so the reference-script fee component is `tierRefScriptFee ... 0 = Coin 0`.
4. Compute the correct minimum fee using `batchNonDistinctRefScriptsSize`: it returns 100 KiB, yielding a substantially higher fee via `tierRefScriptFee`.
5. Set `fee` to the (lower) Conway-computed value and submit. The ledger accepts the transaction because `validateFeeTooSmallUTxO` uses the Conway path and sees `minFee <= theFee`.
6. Nodes deserialize and validate 100 KiB of reference scripts while the submitter paid for 0 KiB of reference scripts.
