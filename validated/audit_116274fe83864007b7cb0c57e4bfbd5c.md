### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Allows Any Unprivileged Caller to Receive MPC-Signed Foreign-Chain Attestations - (File: crates/contract/src/lib.rs)

---

### Summary

`verify_foreign_transaction` stores pending requests under a key that does not include the caller's account ID. Any unprivileged NEAR account can submit an identical foreign-chain verification request and be queued alongside the legitimate caller. When MPC nodes respond, every queued caller â€” including the attacker â€” receives the same `VerifyForeignTransactionResponse` (payload hash + MPC signature). Because the signature is not bound to the original submitter, the attacker obtains a valid MPC attestation for a foreign-chain event they did not initiate, enabling unauthorized bridge execution.

---

### Finding Description

**`sign()` binds the request key to the caller.**
`SignatureRequest::new` hashes `(predecessor_id, path)` into a `Tweak` that becomes part of the map key:

```rust
// crates/near-mpc-crypto-types/src/sign.rs  line 118-125
impl SignatureRequest {
    pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
        let tweak = crate::kdf::derive_tweak(predecessor_id, path);
        SignatureRequest { domain_id: domain, tweak, payload }
    }
}
```

Two different callers with the same payload and path produce **different** `SignatureRequest` keys and therefore different map entries.

**`verify_foreign_transaction()` discards the caller entirely.**
`check_request_preconditions` returns the `predecessor` but `verify_foreign_transaction` never uses it when constructing the request:

```rust
// crates/contract/src/lib.rs  lines 526-556
self.check_request_preconditions(
    request.domain_id, DomainPurpose::ForeignTx, ...
);
// predecessor is returned but silently dropped
let request = args_into_verify_foreign_tx_request(request);
// ...
move |this, id| this.add_verify_foreign_tx_request(request, id)
```

`args_into_verify_foreign_tx_request` maps only `{domain_id, request, payload_version}` â€” no caller field:

```rust
// crates/contract/src/dto_mapping.rs  lines 840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

`VerifyForeignTransactionRequest` itself has no account-ID field:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs  lines 124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

The contract's own unit test explicitly confirms that two different callers share one map entry:

```
// crates/contract/src/lib.rs  line 3255
// "both yields are queued under the single (caller-agnostic) request key"
```

When `respond_verify_foreign_tx` is called, `resolve_yields_for` drains **all** queued yields with the same response bytes, so every caller â€” including the attacker â€” receives the identical `VerifyForeignTransactionResponse`.

---

### Impact Explanation

The `VerifyForeignTransactionResponse` carries a `payload_hash` and an MPC `signature` over `SHA-256(borsh(ForeignTxSignPayload))`. That payload encodes the foreign-chain transaction ID and the extracted values (e.g., Bitcoin block hash). The signature is **not bound to any caller identity**.

A bridge contract that accepts this response as proof of a foreign-chain event (the primary stated use-case: Omnibridge inbound flow) has no on-chain way to distinguish whether the response was obtained by the legitimate initiator or by an attacker who free-rode on the same request. An attacker who receives the response can:

- Submit it to a bridge contract to claim assets corresponding to a foreign-chain deposit they did not make.
- Race the legitimate caller to execute the bridge action first (front-running).
- Accumulate up to `MAX_PENDING_REQUEST_FAN_OUT = 128` copies of the same signed attestation by filling the queue, then replay them across multiple bridge invocations if the bridge contract does not track processed `payload_hash` values.

This breaks the production safety invariant that the MPC network's signed attestation of a foreign-chain event is delivered exclusively to the party that requested it, enabling invalid bridge execution.

---

### Likelihood Explanation

The attack requires no special privilege. The attacker only needs to observe a pending `verify_foreign_transaction` call (visible on-chain as a NEAR receipt) and submit an identical `{ForeignChainRpcRequest, domain_id, payload_version}` tuple with 1 yoctoNEAR deposit before the MPC nodes respond. The foreign transaction ID is public on the foreign chain. The domain ID is a small integer readable from contract state. No threshold collusion, key material, or operator access is needed.

---

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, mirroring the `sign()` design. Derive a `Tweak` from `(predecessor_id, derivation_path)` and store it in `VerifyForeignTransactionRequest`, so two different callers for the same foreign transaction produce distinct map entries and receive independent yield-resume promises. The `VerifyForeignTransactionResponse` should then be delivered only to the caller whose yield was resumed, preventing any third party from obtaining the MPC attestation without submitting their own request.

---

### Proof of Concept

1. Alice (a bridge contract) calls `verify_foreign_transaction` for Bitcoin tx `X` (a deposit to Alice's address). Her yield is queued under key `K = {Bitcoin(tx_id=X, ...), domain_id=2, V1}`.
2. Bob (attacker) observes Alice's receipt on-chain and immediately calls `verify_foreign_transaction` with the identical arguments. Bob's yield is appended to the same queue under key `K`.
3. MPC nodes verify tx `X`, compute `payload_hash = SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))`, and call `respond_verify_foreign_tx(K, response)`.
