## Full Trace

**Step 1 — Pre-condition (reachable on mainnet)**

Before protocol version 11, `hardforkConwayDisallowDuplicatedVRFKeys` is `False`, so no VRF uniqueness check exists. [1](#0-0) 

Two pools (A and B) can legitimately share VRF key V1 before v11. At the v11 hard fork, `populateVRFKeyHashes` is called and correctly sets `psVRFKeyHashes[V1] = 2` using `Map.insertWith combine` (counter-incrementing). [2](#0-1) 

**Step 2 — Pool A re-registers with V2 (within the same epoch, post-v11)**

In `poolTransition`, the re-registration check passes because `sppVrf (V2) == stakePoolState.spsVrf (V1)` is false but `Map.notMember V2 psVRFKeyHashes` is true. [3](#0-2) 

`updateFutureVRFKeyHash` fires the `Nothing` branch (no prior future params for A), inserting V2 with count 1 but **not decrementing V1** — correct, because V1 is still Pool A's active VRF. [4](#0-3) 

State after re-registration:
- `psStakePools[A].vrf = V1`, `psFutureStakePoolParams[A].vrf = V2`
- `psStakePools[B].vrf = V1`
- `psVRFKeyHashes[V1] = 2`, `psVRFKeyHashes[V2] = 1`

**Step 3 — POOLREAP epoch boundary**

`danglingVRFKeyHashes` merges `psStakePoolsL` and `psFutureStakePoolParamsL` with `Map.dropMissing` on **both** sides — only pools present in **both** maps are considered. Pool B is absent from `psFutureStakePoolParams`, so it is silently dropped. Only Pool A matches, and since `V1 ≠ V2`, V1 is emitted: [5](#0-4) 

Result: `danglingVRFKeyHashes = {V1}`.

Then the update is applied:

```haskell
& certPStateL . psVRFKeyHashesL
    %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
           . (`Map.withoutKeys` danglingVRFKeyHashes)
       )
``` [6](#0-5) 

`Map.withoutKeys {V1}` **unconditionally deletes** the V1 entry from `psVRFKeyHashes`, regardless of its counter value (2). After this call, `psVRFKeyHashes[V1]` is **absent**, even though Pool B still actively uses V1.

**Step 4 — Exploit: Pool C registers with V1**

Post-epoch, the new-pool registration check is:

```haskell
Map.notMember sppVrf psVRFKeyHashes
  ?! injectFailure (VRFKeyHashAlreadyRegistered sppId sppVrf)
``` [7](#0-6) 

Since V1 was deleted from `psVRFKeyHashes`, `Map.notMember V1 psVRFKeyHashes` is `True`. Pool C's registration with V1 succeeds. Now Pool B and Pool C both hold V1 as their active VRF key — the invariant the v11 hard fork was designed to enforce is broken.

## Why the Existing Tests Do Not Cover This

The existing test `"Retiring a stake pool with a duplicate VRF Keyhash after v11 HardFork"` covers **retirement** of duplicate-VRF pools (counter decremented via `removeVRFKeyHashOccurrences`), not **re-registration** with a new VRF while a sibling pool still holds the old one. [8](#0-7) 

The test `"re-register a retiring pool with a fresh VRF"` only tests the single-pool case and asserts `expectVRFs [vrfNew]` — it never checks that V1 is still blocked when a second pool holds it. [9](#0-8)

## Root Cause

`Map.withoutKeys danglingVRFKeyHashes` is a **set-delete** (removes the key entirely), but the correct operation is a **counter-decrement** (the same `removeVRFKeyHashOccurrence` already used for retired pools). When `psVRFKeyHashes[V1] = 2`, the correct result after Pool A switches to V2 is `psVRFKeyHashes[V1] = 1` (Pool B still holds it), not absent. [10](#0-9)

### Title
`psVRFKeyHashes` counter corrupted by `Map.withoutKeys` in `poolReapTransition` when a sibling pool shares the dangling VRF key — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

### Summary
`danglingVRFKeyHashes` correctly identifies VRF keys whose owning pool has switched to a new VRF, but the subsequent `Map.withoutKeys` call deletes those keys from `psVRFKeyHashes` unconditionally, ignoring the reference count. When a second pool holds the same VRF key (possible for pools registered before v11), the key is deleted while still in use, allowing a third pool to register with it and bypass the VRF-uniqueness invariant.

### Finding Description
In `poolReapTransition` (`PoolReap.hs` lines 138–227), `danglingVRFKeyHashes` is a `Set` of old VRF keys belonging to pools that re-registered with a new VRF. The map update at line 226 applies `Map.withoutKeys danglingVRFKeyHashes`, which is a hard delete. However, `psVRFKeyHashes` is a reference-counted map (`NonZero Word64` values), and the correct operation for "one pool no longer uses this VRF" is a decrement (as done for retired pools via `removeVRFKeyHashOccurrence`). If another pool shares the same VRF key (a state reachable via pre-v11 registration), the counter should drop from 2 to 1, not to absent.

### Impact Explanation
After the corrupted epoch boundary, `Map.notMember V1 psVRFKeyHashes` returns `True`, so any pool operator can submit a `RegPool` certificate with V1 and have it accepted. Two pools now share the same VRF key, meaning both are elected slot leaders for identical slots, both produce valid blocks, and the VRF-uniqueness guarantee introduced by the v11 hard fork is nullified. This constitutes a deterministic ledger-rule evaluation error: the ledger accepts a pool registration it must reject, producing an incorrect `psVRFKeyHashes` state that all honest nodes agree on (no divergence, but the invariant is permanently broken until the pool retires or re-registers).

### Likelihood Explanation
Requires: (1) two pools sharing a VRF key before v11 — permitted by the pre-v11 rules and plausible on mainnet; (2) one of them submitting a `RegPool` certificate with a new VRF key after v11 — a routine, unprivileged operation. No governance majority, leaked key, or Sybil attack is needed. The attacker can be the operator of Pool A (triggering the corruption) or Pool C (exploiting it).

### Recommendation
Replace `Map.withoutKeys danglingVRFKeyHashes` with `removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)` so that each dangling VRF key has its counter decremented by one rather than being unconditionally deleted. Add a regression test: register two pools with V1 before v11, enact v11, re-register one pool with V2, pass the epoch boundary, and assert `psVRFKeyHashes[V1] == 1` and that a new pool cannot register with V1.

populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
