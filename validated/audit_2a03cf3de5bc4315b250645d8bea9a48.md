### Title
Stale Post-Resharing Votes Inflate TEE Code-Hash Whitelist Threshold — (`crates/contract/src/tee/proposal.rs`, `crates/contract/src/lib.rs`)

---

### Summary

After a resharing completes, `vote_reshared` schedules `clean_tee_status` as a **detached** (fire-and-forget) promise to remove stale votes from removed participants. Until that cleanup receipt executes — a window of at least one block — `vote_code_hash`, `vote_add_launcher_hash`, and `vote_add_os_measurement` count **all** stored votes, including those cast by participants who no longer exist in the protocol. A single Byzantine participant in the new set can exploit this window to push a malicious code hash over the whitelisting threshold using the stale votes of removed participants, bypassing the true threshold requirement.

---

### Finding Description

`CodeHashesVotes::vote` in `crates/contract/src/tee/proposal.rs` counts every stored vote for a proposal without filtering by the current participant set:

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
```

`vote_code_hash` in `crates/contract/src/lib.rs` uses this unfiltered count directly against the threshold:

```rust
let votes = self.tee_state.vote(code_hash, &participant);
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
```

The cleanup that removes stale votes is spawned as a detached promise inside `vote_reshared`:

```rust
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        ...
    )
    .detach();
```

In NEAR, detached cross-contract calls execute in a **subsequent receipt**, meaning there is always at least one block between the resharing transition and the stale-vote removal. During this window, `vote_code_hash` sees the inflated count. If `clean_tee_status` runs out of gas or panics, the stale votes persist indefinitely.

The same pattern applies to `vote_add_launcher_hash` (via `LauncherHashVotes::vote`) and `vote_add_os_measurement` (via `MeasurementVotes`), both of which also count all stored votes without participant filtering.

The existing test at `crates/contract/src/tee/tee_state.rs:1496–1545` explicitly documents that stale votes **are** counted before cleanup runs, and that `clean_non_participant_votes` is required to correct this — but that call is not synchronous.

---

### Impact Explanation

If T−1 removed participants had voted for a malicious MPC node image hash H before resharing, a single Byzantine participant in the new set can call `vote_code_hash(H)` during the cleanup window. The unfiltered count reaches T (stale T−1 + new 1), crossing the threshold and whitelisting H. Nodes running the malicious image then pass `is_caller_an_attested_participant` checks, allowing them to call `respond`, `vote_pk`, `vote_reshared`, and other privileged methods. This enables unauthorized threshold signature issuance and participation in key generation/resharing, constituting a **Critical** impact: unauthorized transaction execution and potential key-share disclosure.

---

### Likelihood Explanation

The attack requires:
1. T−1 Byzantine participants to vote for a malicious hash before being removed via resharing.
2. At least one Byzantine participant in the new set.
3. A call to `vote_code_hash` in the window between `vote_reshared` and the `clean_tee_status` receipt.

The window is guaranteed to exist (at least one block). If `clean_tee_status` fails (insufficient gas allocation via `clean_tee_status_tera_gas`), the window is permanent. The scenario is realistic in adversarial resharing conditions where Byzantine nodes are being evicted — precisely the situation where this protection matters most.

---

### Recommendation

Replace the unfiltered `count_votes` call in `CodeHashesVotes::vote` (and the equivalent in `LauncherHashVotes` and `MeasurementVotes`) with a participant-filtered count, passing the current `ThresholdParameters` into the vote function — mirroring the correct pattern already used in `TeeVerifierVotes::vote`:

```rust
voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
```

Alternatively, make the stale-vote cleanup **synchronous** (inline, before the threshold check) rather than a detached promise, so the count is always accurate at the point of evaluation.

---

### Proof of Concept

Setup: N=5, T=3. Participants {P0, P1, P2, P3, P4}.

1. P0 and P1 (Byzantine) call `vote_code_hash(malicious_H)`. Stored votes: {P0→H, P1→H}. Count = 2 < 3. Hash not whitelisted.
2. Governance votes to reshare, removing P0 and P1. New set: {P2, P3, P4}, threshold 3. `vote_reshared` completes; `clean_tee_status` is scheduled as a detached promise but has not yet executed.
3. In the same or next block, P2 (Byzantine, in new set) calls `vote_code_hash(malicious_H)`. `CodeHashesVotes::vote` inserts P2's vote and calls `count_votes`, which iterates all stored entries: {P0→H (stale), P1→H (stale), P2→H}. Count = 3 ≥ threshold 3.
4. `whitelist_tee_proposal(malicious_H)` executes. The malicious image hash is now in `allowed_docker_image_hashes`.
5. Nodes running the malicious image submit attestations, pass `is_caller_an_attested_participant`, and gain full participant privileges — including calling `respond` to issue signatures and `vote_reshared` to influence future key state.

    pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
        log!(
            "vote_code_hash: signer={}, code_hash={:?}",
            env::signer_account_id(),
            code_hash,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let votes = self.tee_state.vote(code_hash, &participant);

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        // If the vote threshold is met and the new Docker hash is allowed by the TEE's RTMR3,
        // update the state
        if votes >= self.threshold()?.value() {
            self.tee_state
                .whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
        }

        Ok(())
    }

            ProtocolContractState::Running(state) => state.parameters.participants(),

        self.tee_state.clean_non_participant_votes(participants);
        Ok(())
    }

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
    }
