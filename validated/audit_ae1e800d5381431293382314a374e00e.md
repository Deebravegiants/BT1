### Title
Unbounded Per-Slot `SubMessageQueue` Growth via `PrivateChannel::recv` Continuation Loop Enables Memory Exhaustion on Honest MPC Nodes - (File: `crates/threshold-signatures/src/protocol/internal.rs`)

---

### Summary

The `MessageBuffer` in the MPC protocol framework enforces a `max_entries` cap on the number of distinct `MessageHeader` keys in its `HashMap`, but places **no limit** on the number of messages that can be queued per occupied slot. Each slot's `SubMessageQueue` is backed by a `futures::channel::mpsc::unbounded()` channel. The `PrivateChannel::recv` loop contains a "continuation" condition — analogous to the Bio.sol continuation-character skip — that discards messages from unexpected senders and loops back to wait for more. A Byzantine participant strictly below the signing threshold can exploit this by flooding a private-channel header slot with crafted messages, causing unbounded memory growth and eventual node crash on honest MPC nodes, breaking the signing request lifecycle.

---

### Finding Description

**Root cause — two-layer design flaw:**

**Layer 1: No per-slot message limit in `MessageBuffer::push`** [1](#0-0) 

When a `MessageHeader` key already exists in the `HashMap` (`Occupied` branch), the code calls `entry.get().send(from, message)` with **no capacity check**. Only the `Vacant` branch (new headers) checks `len < self.max_entries`. The `SubMessageQueue` uses `futures::channel::mpsc::unbounded()`: [2](#0-1) 

This channel is truly unbounded — `unbounded_send` never applies backpressure.

**Layer 2: The `PrivateChannel::recv` continuation loop** [3](#0-2) 

This is the direct analog to the Bio.sol continuation-character skip. When a message arrives from a sender that is not `self.to`, the loop:
1. Pops the message from the `SubMessageQueue` (consuming it)
2. Discards it (`from != self.to`)
3. Calls `yield_point()` and **`continue`s** — looping back to wait for the next message

The attacker can send new messages faster than the loop consumes them, keeping the `SubMessageQueue` perpetually full and growing.

**Attack construction:**

The `PrivateChannel` header is derived deterministically: [4](#0-3) 

Participant IDs are sorted before hashing, so `root_private(B, C) == root_private(C, B)`. A Byzantine participant A can compute the header for any `(B, C)` pair and craft messages with that header. The TLS layer authenticates the sender as A, but routing is based solely on the `MessageHeader` bytes extracted from the message payload: [5](#0-4) 

Messages from A with the `(B, C)` header are accepted into B's `SubMessageQueue` for that slot (occupied branch, no limit). B's `PrivateChannel::recv` loop pops each one, sees `from == A ≠ C`, discards it, and loops — while A keeps sending.

**The `pop` bypass compounds the issue:** [6](#0-5) 

`pop` uses `or_default()` to create a new `SubMessageQueue` entry without checking `max_entries`. The code comment acknowledges this explicitly: [7](#0-6) 

Once the honest protocol calls `pop` for a header (creating the slot), the attacker can flood that slot with unlimited messages via the `Occupied` branch.

---

### Impact Explanation

A Byzantine MPC participant strictly below the signing threshold can cause unbounded memory growth in the `SubMessageQueue` of any honest node participating in a two-party sub-protocol (OT-based ECDSA triple generation, MTA, correlated OT extension). Memory exhaustion crashes the honest node, preventing it from completing signing operations. Pending `sign()` requests on the NEAR contract cannot be fulfilled, effectively freezing funds that depend on MPC-signed transactions. This breaks the signing request lifecycle without requiring threshold-or-above collusion.

---

### Likelihood Explanation

Any single Byzantine MPC participant (below threshold) can execute this attack during any protocol run in which they participate. The `PrivateChannel` headers are deterministic and computable from public participant IDs. No privileged access, key material, or network-level capabilities are required beyond the ability to send crafted P2P messages — which is the normal capability of any MPC participant.

---

### Recommendation

1. **Add a per-slot message limit** to `SubMessageQueue`. Replace `futures::channel::mpsc::unbounded()` with a bounded channel (e.g., `futures::channel::mpsc::channel(MAX_MESSAGES_PER_SLOT)`). When the bounded channel is full, `push` should return `MessageError::BufferFull` for occupied slots as well.

2. **Add a per-sender rate limit** inside `PrivateChannel::recv`. Track how many messages have been discarded from unexpected senders and abort the protocol if the count exceeds a threshold (e.g., `2 * num_participants`).

3. **Enforce the `max_entries` invariant in `pop`**: instead of `or_default()`, check whether the entry count would exceed `max_entries` before inserting, and return an error if so.

---

### Proof of Concept

```rust
// Byzantine participant A (below threshold) attacks honest node B
// during a signing protocol run where B expects messages from C.

// Step 1: A computes the PrivateChannel header for (B, C)
// (deterministic, based on sorted participant IDs)
let header = ChannelTag::root_private(B_id, C_id).with_waitpoint(0);

// Step 2: A crafts messages with that header and sends them to B
// The TLS layer authenticates sender as A, but routing uses the header
loop {
    let mut msg = header.to_bytes().to_vec();
    msg.extend_from_slice(&crafted_payload);
    transport.send_to(B_id, msg); // authenticated as from A
}

// Step 3: B's PrivateChannel::recv loop runs:
// - pops message from SubMessageQueue (from A, not C)
// - discards it (from != self.to where self.to == C)
// - continues looping
// Meanwhile A keeps sending → SubMessageQueue grows without bound
// → memory exhaustion → node crash → signing requests frozen

    fn push(
        &self,
        header: MessageHeader,
        from: Participant,
        message: MessageData,
    ) -> Result<(), MessageError> {
        let mut messages_lock = self.messages.lock().expect("lock should not fail");
        let len = messages_lock.len();
        match messages_lock.entry(header) {
            std::collections::hash_map::Entry::Occupied(entry) => {
                entry.get().send(from, message);
                Ok(())
            }
            std::collections::hash_map::Entry::Vacant(entry) => {
                if len < self.max_entries {
                    entry.insert(SubMessageQueue::default()).send(from, message);
                    Ok(())
                } else {
                    Err(MessageError::BufferFull {
                        capacity: self.max_entries,
                    })
                }
            }
        }
    }
