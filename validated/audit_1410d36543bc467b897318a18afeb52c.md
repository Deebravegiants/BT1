### Title
Missing CKD Output Validation for `AppPublicKey` Variant Allows Single Malicious Participant to Submit Arbitrary Key Derivation Response - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract validates the CKD output cryptographically only when the `AppPublicKeyPV` variant is used. When the `AppPublicKey` variant is used, **no output validation is performed**. A single malicious attested participant (Byzantine, below the signing threshold) can race to call `respond_ckd` with an arbitrary `CKDResponse`, causing the user to receive an attacker-controlled derived key instead of the correct threshold-computed one.

---

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 654–689), the contract conditionally validates the CKD response:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VALIDATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKey` arm, the `response` is passed directly to `resolve_yields_for` without any cryptographic check that the returned key material is the correct output of the threshold CKD protocol. The contract simply accepts whatever the first eligible caller submits.

Contrast this with the `respond` function for threshold signatures, which **always** validates the signature cryptographically before resolving the yield:

```rust
let signature_is_valid = match (&response, public_key) {
    (dtos::SignatureResponse::Secp256k1(sig), PublicKeyExtended::Secp256k1 { .. }) => {
        near_mpc_signature_verifier::verify_ecdsa_signature(...).is_ok()
    }
    ...
};
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
``` [2](#0-1) 

The `AppPublicKeyPV` variant was introduced precisely to close this gap via `ckd_output_check`, but the `AppPublicKey` variant remains in production with no equivalent guard. [3](#0-2) 

---

### Impact Explanation

A single malicious attested participant can call `respond_ckd` with an arbitrary `CKDResponse` for any pending `AppPublicKey` CKD request. Because the contract performs no output validation for this variant, the attacker-supplied response is accepted and returned to the requesting user as the legitimate derived key.

If the user subsequently uses this derived key to control assets on a foreign chain (the primary use-case for CKD), the attacker — who knows the key they injected — can spend those assets. This constitutes **direct theft of funds** controlled by the MPC network's key derivation output.

This maps to the allowed Critical impact: *"Confidential key derivation output without the required participant authorization"* and *"Bypass of threshold-signature requirements … that materially enables forgery or secret recovery."*

---

### Likelihood Explanation

The attacker must be a single active, TEE-attested participant in the MPC network — a Byzantine participant strictly below the signing threshold. The access check `assert_caller_is_attested_participant_and_protocol_active` is the only gate: [4](#0-3) 

Once the user's `request_app_private_key` call is enqueued and visible on-chain, the malicious participant races to call `respond_ckd` before the honest coordinator. Because the malicious participant does not need to wait for the threshold protocol to complete (they fabricate the response), they have a timing advantage. On NEAR, where block times are ~1 second, this race is practically winnable.

---

### Recommendation

Apply `ckd_output_check` (or an equivalent cryptographic proof of correctness) unconditionally for **both** `AppPublicKey` and `AppPublicKeyPV` variants, or deprecate `AppPublicKey` entirely in favor of `AppPublicKeyPV`. The `respond` function's pattern — always verifying the output before resolving the yield — should be the model:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(app_pk) => {
        // Apply equivalent output check here
        if !ckd_output_check_legacy(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key: AppPublicKey(user_pk)`. The request is enqueued on-chain with a yield ID.
2. Malicious attested participant observes the pending request on-chain.
3. Before the honest coordinator completes the threshold CKD protocol, the malicious participant constructs an arbitrary `CKDResponse` containing an attacker-controlled derived key.
4. Malicious participant calls `respond_ckd(request, attacker_response)`. The contract checks only that the caller is an attested participant and that the domain is BLS12-381 — no output validation is performed for `AppPublicKey`.
5. `resolve_yields_for` resolves the user's yield with the attacker-controlled key material.
6. User receives the attacker-controlled derived key and uses it to generate a foreign-chain address.
7. Attacker, knowing the private key behind that address, drains any assets the user deposits there. [5](#0-4)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
