### Title
Multiple Committee Cold Keys Can Authorize the Same Hot Key, Allowing a Single Vote to Count Multiple Times - (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

The Conway era `GOVCERT` rule allows multiple committee cold keys to authorize the same hot key credential. Because `committeeAcceptedRatio` in `Ratify.hs` iterates over cold keys and resolves each one's hot key independently, a single hot-key vote is counted once per cold key that authorized it. An attacker controlling two committee cold keys can register the same hot key for both, then cast a single vote that is counted as two committee yes-votes, inflating the acceptance ratio and potentially ratifying governance actions that should not pass.

---

### Finding Description

`CommitteeState` stores a unidirectional mapping:

```haskell
csCommitteeCreds :: Map (Credential ColdCommitteeRole) CommitteeAuthorization
```

The code itself acknowledges the absence of a reverse mapping:

> "there is no unique mapping from Hot to Cold credential, therefore we produce a Set, instead of a Map." [1](#0-0) 

The `checkAndOverwriteCommitteeMemberState` helper in `GovCert.hs` validates a `ConwayAuthCommitteeHotKey` certificate by checking only two things: (1) the cold key has not previously resigned, and (2) the cold key is a current or proposed future committee member. It does **not** check whether the supplied hot key is already registered for a different cold key. [2](#0-1) 

The result is that two separate `ConwayAuthCommitteeHotKey` certificates — one for `coldA` and one for `coldB` — can both name the same `hotH`, producing:

```
csCommitteeCreds[coldA] = CommitteeHotCredential hotH
csCommitteeCreds[coldB] = CommitteeHotCredential hotH
```

At ratification time, `committeeAcceptedRatio` folds over every cold key in the elected committee and, for each one, looks up its hot key in the `gasCommitteeVotes` map:

```haskell
Just (CommitteeHotCredential hotKey) ->
  case Map.lookup hotKey votes of
    ...
    Just VoteYes -> (yes + 1, tot + 1)
``` [3](#0-2) 

Because `hotH` appears in `gasCommitteeVotes` once (the single vote submitted by the hot key), `Map.lookup hotH votes` returns `VoteYes` for **both** `coldA` and `coldB`. The single vote is therefore counted twice: `yes + 2`, `tot + 2`.

Votes are stored in `gasCommitteeVotes :: Map (Credential HotCommitteeRole) Vote`, keyed by hot credential. [4](#0-3) 

The `GOV` rule records a committee vote as `CommitteeVoter hotCred` and inserts it into that map; there is no deduplication against the cold-key mapping at vote-submission time. [5](#0-4) 

---

### Impact Explanation

**Critical — Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.**

By inflating the committee acceptance ratio, an attacker can cause governance actions to be ratified that would otherwise fail the committee threshold. Affected action types include treasury withdrawals, protocol parameter changes, hard-fork initiations, committee updates, and constitutional changes. A committee of size N with threshold t requires ⌈N·t⌉ yes-votes; if an attacker controls k cold keys all sharing one hot key, a single vote counts as k yes-votes, potentially satisfying the threshold alone.

---

### Likelihood Explanation

An attacker needs to control at least two committee cold keys simultaneously. This is achievable if:

1. A single actor is elected to the committee with two distinct cold keys (e.g., via two separate `UpdateCommittee` governance actions, or a single action listing both).
2. Two colluding committee members agree to share a hot key.

Both scenarios require the attacker to first be elected to the committee, which itself requires a governance vote. However, once elected, the exploit requires only two ordinary transactions (two `AuthCommitteeHotKeyTxCert` certificates) and one vote transaction — all submitted by an unprivileged transaction sender who holds the cold-key signing keys. No privileged access beyond committee membership is needed.

---

### Recommendation

In `checkAndOverwriteCommitteeMemberState` (GovCert.hs), before inserting `CommitteeHotCredential hotCred`, verify that `hotCred` is not already present as the value for any other cold key in `csCommitteeCreds`. Concretely, maintain a reverse mapping `Map (Credential HotCommitteeRole) (Credential ColdCommitteeRole)` alongside `csCommitteeCreds`, or scan the existing values before accepting the certificate:

```haskell
-- Proposed additional check inside checkAndOverwriteCommitteeMemberState:
let hotAlreadyUsed =
      any (\case CommitteeHotCredential hk -> hk == hotCred; _ -> False)
          (Map.elems csCommitteeCreds)
failWhen hotAlreadyUsed $ injectFailure (ConwayCommitteeHotKeyAlreadyAuthorized hotCred)
```

Alternatively, store the committee state as a bidirectional map so that authorizing a hot key for a new cold key automatically removes it from any previous cold key, analogous to the mitigation proposed in the external report.

### Proof of Concept

```
1. coldA and coldB are both elected committee members.
   Committee = { coldA → epoch 100, coldB → epoch 100 }, threshold = 2/3

2. Attacker submits:
     Tx1: AuthCommitteeHotKeyTxCert coldA hotH
     Tx2: AuthCommitteeHotKeyTxCert coldB hotH
   Both succeed — GovCert.hs has no uniqueness check on hotH.

   State: csCommitteeCreds = { coldA → CommitteeHotCredential hotH,
                                coldB → CommitteeHotCredential hotH }

3. Attacker submits a vote:
     CommitteeVoter hotH votes VoteYes on govActionId G

   gasCommitteeVotes G = { hotH → VoteYes }

4. At ratification, committeeAcceptedRatio iterates members:
     member = coldA → lookup hotH in votes → VoteYes → yes=1, tot=1
     member = coldB → lookup hotH in votes → VoteYes → yes=2, tot=2

   acceptedRatio = 2/2 = 1.0  ≥  threshold 2/3  → ACCEPTED

5. With only one actual yes-vote (from hotH), a governance action requiring
   2/3 committee approval is enacted.  If the action is a treasury withdrawal,
   protocol parameter change, or hard-fork initiation, it takes effect
   without the required committee consensus.
```
