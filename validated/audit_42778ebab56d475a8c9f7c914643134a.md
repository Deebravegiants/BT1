### Title
Dev TCB Measurements Compiled Into Production Contract Bypass Attestation Security Gate - (File: `crates/mpc-attestation/src/attestation.rs`)

### Summary

The production MPC contract binary permanently compiles in a second set of "dev" OS measurements (`tcb_info_dev.json`) alongside the production measurements (`tcb_info.json`). This is explicitly flagged as a security TODO that was never resolved. Any node presenting a Dstack attestation whose RTMR values match the dev image passes the attestation gate, even though the dev image uses a local key provider (`local-sgx`) rather than a production KMS, weakening the TEE security guarantee for that participant.

### Finding Description

`default_measurements()` in `crates/mpc-attestation/src/attestation.rs` returns a static array of two `ExpectedMeasurements` entries compiled in at build time:

```rust
static MEASUREMENTS: [ExpectedMeasurements; 2] = [
    include_measurements!("assets/tcb_info.json"),
    // TODO(#1433): Security - remove dev measurements from production builds after testing is complete
    include_measurements!("assets/tcb_info_dev.json"),
];
``` [1](#0-0) 

These measurements are seeded into the contract's `allowed_measurements` list at deployment/migration time and are used by `TeeState::add_participant` to verify every incoming Dstack attestation:

```rust
let AcceptedAttestation { .. } = attestation.verify_locally(
    expected_report_data.into(),
    Self::current_time_seconds(),
    &self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration),
    &self.get_allowed_launcher_compose_hashes(),
    &accepted_measurements,   // ← includes dev measurements
)?;
``` [2](#0-1) 

Comparing the two compiled-in JSON files, the RTMR values differ materially. The dev file's `key-provider` event payload decodes to `local-sgx` (a local key provider), whereas the production file uses a remote KMS. The dev `rtmr2` is `c812d42b...` while production is `9284cde2...`. [3](#0-2) [4](#0-3) 

The `verify_measurements` function accepts any attestation whose measurements appear in the allowed list — including the dev entry:

```rust
fn verify_measurements(
    measurements: &ExpectedMeasurements,
    allowed_measurements: &[ExpectedMeasurements],
) -> Result<(), VerificationError> {
    if !allowed_measurements.contains(measurements) {
        return Err(VerificationError::MeasurementsNotAllowed);
    }
    Ok(())
}
``` [5](#0-4) 

### Impact Explanation

A Byzantine participant (below the signing threshold) can run a node booted from the dev OS image. Because the dev measurements are compiled into the production contract and seeded into `allowed_measurements` at deployment, `submit_participant_info` accepts the attestation without error. The participant's node uses `local-sgx` as its key provider, meaning its key material is derived from a local SGX key rather than a production remote KMS. This breaks the production safety invariant that every accepted participant must be running a production-grade, remotely-attested TEE environment. A participant operating under `local-sgx` has weaker confidentiality guarantees for their key share, which degrades the overall security of the threshold signing network.

### Likelihood Explanation

The dev measurements are compiled into every production binary and seeded at every deployment/migration. Any participant who intentionally or accidentally runs the dev image will silently pass attestation. The TODO is explicitly labeled `Security` and has a tracking issue (`#1433`), confirming the team is aware but has not resolved it.

### Recommendation

Remove `include_measurements!("assets/tcb_info_dev.json")` from the `default_measurements()` array before any production deployment. Dev measurements should only be present in builds explicitly compiled with a dev feature flag, never in the default production binary. After removal, operators can still add dev measurements on-chain via `vote_add_os_measurement` for controlled testing environments. [6](#0-5) 

### Proof of Concept

1. Build the MPC contract from the current codebase. `default_measurements()` compiles in both `tcb_info.json` and `tcb_info_dev.json`.
2. Deploy the contract. Both measurement sets are seeded into `TeeState::allowed_measurements`.
3. Boot an MPC node from the dev OS image (RTMR values matching `tcb_info_dev.json`, `key-provider = local-sgx`).
4. Call `submit_participant_info` with a valid Dstack attestation from that dev node.
5. `TeeState::add_participant` calls `attestation.verify_locally(...)`, which calls `verify_measurements(...)`. The dev measurements match the second entry in `allowed_measurements`, so verification succeeds.
6. The dev node is now a registered, attested participant despite running with a local key provider rather than a production KMS, bypassing the intended production-only attestation gate. [7](#0-6) [1](#0-0)

    /// Adds a participant attestation for the given node iff the attestation succeeds verification.
    pub(crate) fn add_participant(
        &mut self,
        node_id: NodeId,
        attestation: Attestation,
        tee_upgrade_deadline_duration: Duration,
    ) -> Result<ParticipantInsertion, AttestationSubmissionError> {
        let expected_report_data: ReportData = ReportDataV1::new(
            *node_id.tls_public_key.as_bytes(),
            *node_id.account_public_key.as_bytes(),
        )
        .into();

        let accepted_measurements = self.get_accepted_measurements();
        // TODO(#3264): run DCAP in the verifier contract (Promise + callback) and
        // do the post-DCAP checks here, instead of verifying locally in-WASM.
        let AcceptedAttestation {
            attestation: verified_attestation,
            advisory_ids,
        } = attestation.verify_locally(
            expected_report_data.into(),
            Self::current_time_seconds(),
            &self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration),
            &self.get_allowed_launcher_compose_hashes(),
            &accepted_measurements,
        )?;

        log_informational_advisory_ids(&advisory_ids);

        let tls_pk = node_id.tls_public_key.clone();

        // Authorization: a TLS key registered to one account must not be
        // overwritten by a submission from a different account. Without this,
        // any caller could replace any participant's stored attestation, since
        // the entry is keyed only by `tls_public_key`.
        if let Some(existing) = self.stored_attestations.get(&tls_pk)
            && existing.node_id.account_id != node_id.account_id
        {
            return Err(AttestationSubmissionError::TlsKeyOwnedByOtherAccount);
        }

        let insertion = self.stored_attestations.insert(
            tls_pk,
            NodeAttestation {
                node_id,
                verified_attestation,
            },
        );

        Ok(match insertion {
            Some(_previous_attestation) => ParticipantInsertion::UpdatedExistingParticipant,
            None => ParticipantInsertion::NewlyInsertedParticipant,
        })
    }
