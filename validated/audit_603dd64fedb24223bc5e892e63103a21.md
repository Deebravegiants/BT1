### Title
Rebasing ERC20 Token Vault Balance Divergence Breaks Redemption Invariant - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.sol`'s `initTransfer` locks native ERC20 tokens using the caller-supplied `amount` parameter and emits that same value in the `InitTransfer` event. The NEAR side mints wrapped tokens equal to that emitted amount. When the user redeems, `finTransfer` releases exactly `payload.amount` (the originally-locked figure, now NEAR-signed). For rebasing tokens whose vault balance drifts between lock and unlock, this invariant breaks: a negative rebase makes `finTransfer` revert permanently (fund freeze), and a positive rebase leaves excess tokens permanently stranded in the contract.

### Finding Description

`initTransfer` for native (non-bridge) ERC20 tokens performs a plain `safeTransferFrom` using the caller-supplied `amount` and immediately emits that same value in the `InitTransfer` event:

```solidity
// OmniBridge.sol lines 406-412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
``` [1](#0-0) 

The event then broadcasts the user-supplied `amount` as the canonical locked value:

```solidity
// OmniBridge.sol lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← user-supplied, not actual-received
    ...
);
``` [2](#0-1) 

There is no balance-before / balance-after measurement to capture the actual amount received. The NEAR bridge reads this event and mints exactly `amount` wrapped tokens.

On the return path, `finTransfer` releases exactly `payload.amount` (the NEAR-signed figure, which equals the originally-emitted `amount`):

```solidity
// OmniBridge.sol lines 350-354
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount
    );
}
``` [3](#0-2) 

For a rebasing token, the vault balance at redemption time is not guaranteed to equal `payload.amount`. The same pattern exists in the Starknet bridge:

```cairo
// starknet/src/omni_bridge.cairo lines 260-262
let success = IERC20Dispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());
assert(success, 'ERR_TRANSFER_FAILED');
``` [4](#0-3) 

### Impact Explanation

**Negative rebase (fund freeze — Critical):** The vault holds fewer tokens than `payload.amount`. `safeTransfer` reverts. The NEAR-signed payload is already consumed (nonce marked used at line 287 before the transfer), so the call cannot be retried with a smaller amount. The user's wrapped tokens on NEAR are already burned. Result: user funds are permanently frozen in the bridge with no recovery path. [5](#0-4) 

**Positive rebase (permanent value leak — Critical):** The vault holds more tokens than `payload.amount`. The excess accumulates silently in the contract with no mechanism to claim or redistribute it. Over many rebase cycles and users, this becomes a growing pool of permanently unclaimable value.

### Likelihood Explanation

The bridge does not exclude rebasing tokens from the native-token lock path. Any ERC20 that is not registered as a `isBridgeToken` follows the `safeTransferFrom → lock → safeTransfer → unlock` path. Well-known rebasing tokens (e.g., AMPL, stETH in rebase mode, aTokens) are standard ERC20s that pass all interface checks. A user locking such a token triggers the vulnerability without any privileged action; the rebase itself is the autonomous trigger. No attacker action is required beyond the initial `initTransfer` call.

### Recommendation

1. **Measure actual received amount** in `initTransfer` using a balance-before / balance-after pattern and emit the delta, not the parameter:
   ```solidity
   uint256 before = IERC20(tokenAddress).balanceOf(address(this));
   IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
   uint128 actualReceived = uint128(IERC20(tokenAddress).balanceOf(address(this)) - before);
   // emit actualReceived, not amount
   ```
2. **Alternatively**, explicitly document and enforce that rebasing tokens are not supported (e.g., a token allowlist or a revert if `balanceOf` post-transfer differs from `amount`).
3. Apply the same fix to the Starknet `init_transfer` in `starknet/src/omni_bridge.cairo`.

### Proof of Concept

1. A rebasing ERC20 token `REBASE` is registered as a native token on EVM (not a bridge token).
2. Alice calls `initTransfer(REBASE, 1000, ...)`. The contract receives 1000 tokens. `InitTransfer` event emits `amount = 1000`.
3. NEAR processes the event and mints 1000 wrapped `REBASE` to Alice.
4. `REBASE` undergoes a negative rebase: the contract's balance drops from 1000 to 800.
5. Alice burns 1000 wrapped `REBASE` on NEAR. NEAR signs a `TransferMessagePayload` with `amount = 1000`.
6. Relayer calls `finTransfer` with `payload.amount = 1000`. The nonce is marked used at line 287.
7. `IERC20(REBASE).safeTransfer(alice, 1000)` reverts because the contract only holds 800.
8. The nonce is already consumed; the call cannot be retried. Alice's 1000 wrapped tokens are burned. Her 800 real tokens are permanently locked in the contract. [6](#0-5) [7](#0-6)
