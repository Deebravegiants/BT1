### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Rewards via Unverified `farmer_reward_address_override` - (File: chia/farmer/farmer_api.py)

### Summary
A harvester peer can set `farmer_reward_address_override` to any arbitrary puzzle hash in `NewProofOfSpace` and `RespondSignatures` messages. The farmer accepts this override without any cryptographic authorization check and without enforcing the CHIP-22 fee convention — it only logs a warning. The override is then used verbatim in `DeclareProofOfSpace` sent to the full node, permanently redirecting the 2 XCH farmer block reward to the attacker's address.

### Finding Description

CHIP-22 introduced `farmer_reward_address_override` to allow third-party harvesters to take a fee by redirecting the farmer reward. The fee convention is supposed to be enforced by comparing `fee_quality` (derived from the proof) against `fee_threshold` (reported by the harvester). However, the enforcement is purely advisory.

**Step 1 — `NewProofOfSpace` handler accepts the override without blocking:**

In `FarmerAPI.new_proof_of_space()`, when `farmer_reward_address_override is not None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [1](#0-0) 

`notify_farmer_reward_taken_by_harvester_as_fee()` only emits log warnings when the fee threshold is invalid — it never returns an error, never raises, and never blocks the override from proceeding: [2](#0-1) 

**Step 2 — `_process_respond_signatures()` unconditionally applies the override:**

After the farmer requests signatures from the harvester, the harvester's `RespondSignatures` message also carries `farmer_reward_address_override`. In `_process_respond_signatures()`, this field is applied without any check: [3](#0-2) 

The resulting `farmer_reward_address` (the attacker's puzzle hash) is then embedded in `DeclareProofOfSpace` and broadcast to the full node: [4](#0-3) 

**Step 3 — The `RespondSignatures` protocol message carries the override field:** [5](#0-4) 

There is no cryptographic binding between the harvester's signing key and the `farmer_reward_address_override` value. Any harvester can set it to any `bytes32` value.

### Impact Explanation

A malicious harvester that finds a valid proof of space can redirect the entire 2 XCH farmer block reward to an address it controls. The farmer's own `farmer_target` address is bypassed. This is an unauthorized diversion of XCH rewards — Critical impact under the allowed scope.

### Likelihood Explanation

The CHIP-22 third-party harvester model explicitly allows external harvesters to connect to a farmer. Any such harvester — including a compromised or malicious one — can exploit this. The attacker only needs to be a connected harvester peer and find a valid proof of space (which is their normal job). No special privileges, leaked keys, or cryptographic breaks are required.

### Recommendation

The farmer must cryptographically verify that the `farmer_reward_address_override` is authorized. Options:
1. Require the harvester to sign the override address with its plot key (the same key used for the proof), so the farmer can verify the signature before accepting the redirect.
2. Enforce the fee convention by **rejecting** (not just logging) any `farmer_reward_address_override` where `fee_quality > fee_threshold`, and reject overrides with no `fee_info` entirely.
3. Allow farmers to configure a whitelist of permitted harvester reward addresses or a maximum fee percentage, and reject overrides that exceed it.

### Proof of Concept

1. Attacker operates a harvester and connects it to a victim farmer.
2. Harvester finds a valid proof of space for a signage point.
3. Harvester sends `NewProofOfSpace` with `farmer_reward_address_override = attacker_puzzle_hash` and `fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=0xFFFFFFFF)` (maximum threshold, always passes the log check).
4. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()` — logs "Fee threshold passed" and continues.
5. Farmer sends `RequestSignatures` to the harvester.
6. Harvester responds with `RespondSignatures(farmer_reward_address_override=attacker_puzzle_hash, ...)`.
7. Farmer's `_process_respond_signatures()` sets `farmer_reward_address = attacker_puzzle_hash` unconditionally.
8. Farmer broadcasts `DeclareProofOfSpace` with `farmer_reward_address = attacker_puzzle_hash` to the full node.
9. Full node farms the block; the 2 XCH farmer reward is paid to the attacker's address.
