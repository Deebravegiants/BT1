Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Amount in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for LST tokens it evaluates `totalAssetDeposits + amount > limit`, but for ETH it evaluates only `totalAssetDeposits > limit`, silently dropping the incoming deposit amount. Any unprivileged caller can therefore push total ETH deposits arbitrarily above the configured cap in a single `depositETH` call.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682 of `contracts/LRTDepositPool.sol`):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The ETH branch (line 679) never includes `amount`, so the function returns `false` ("not exceeded") for any ETH deposit as long as the running total has not already crossed the limit from a prior deposit. `_beforeDeposit` (line 661) calls this function and reverts only on `true`; because the ETH branch cannot return `true` prospectively, the cap is never enforced at deposit time for ETH. The public entry point `depositETH` (lines 76–93) is payable, unrestricted, and routes directly through `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit`.

## Impact Explanation
The ETH deposit limit is the protocol's primary risk-management cap on native ETH exposure. Bypassing it allows any depositor to inflate total ETH holdings far above the configured ceiling in a single transaction, forcing the protocol to restake or manage more ETH than its operational parameters were designed for. This can result in temporary freezing of funds if the downstream EigenLayer delegation or unstaking queue cannot absorb the excess, and constitutes a contract failing to deliver its promised deposit-cap guarantee. Impact: **Medium — temporary freezing of funds / contract fails to deliver promised returns**.

## Likelihood Explanation
`depositETH` is public, payable, and requires no special role or setup. Any depositor who observes that `getTotalAssetDeposits(ETH_TOKEN)` is at or near the limit can immediately send an arbitrarily large ETH deposit in a single transaction. No flash loan, governance capture, or multi-step setup is required. The condition is trivially observable on-chain. Likelihood: **High**.

## Recommendation
Add `amount` to the ETH branch, mirroring the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Assume `depositLimitByAsset(ETH_TOKEN) = 1000 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.

1. Attacker calls `depositETH{value: 5000 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 5000 ether)`.
3. ETH branch evaluates `999 ether > 1000 ether` → `false` → no revert.
4. `_mintRsETH` mints rsETH for 5000 ETH.
5. `getTotalAssetDeposits(ETH_TOKEN)` is now `6999 ether`, nearly 7× the intended cap.

For LST tokens the same scenario evaluates `999 ether + 5000 ether > 1000 ether` → `true` → reverts with `MaximumDepositLimitReached`. The ETH path silently accepts the deposit.

**Foundry test plan:** Deploy `LRTDepositPool` with `depositLimitByAsset(ETH_TOKEN) = 1000 ether`, seed `999 ether` of prior deposits, call `depositETH{value: 5000 ether}`, assert the call does not revert and `getTotalAssetDeposits(ETH_TOKEN) > 1000 ether`.
