### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Submitted `request` — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the ECDSA signature in the response is valid over the caller-supplied `response.payload_hash` under the domain root key. It never checks that `response.payload_hash` was actually derived from the `request`'s foreign-chain data. A Byzantine attested participant (strictly below the signing threshold) can therefore substitute a valid MPC signature produced for one foreign transaction into the response slot of a completely different pending request, delivering a forged `VerifyForeignTransactionResponse` to the original caller.

---

### Finding Description

In `respond_verify_foreign_tx` the signature check is:

```rust
// crates/contract/src/lib.rs  lines 718-743
let signature_is_valid = match (&response.signature, public_key) {
    (dtos::SignatureResponse::Secp256k1(signature_response),
     PublicKeyExtended::Secp256k1 { near_public_key }) => {
        let secp_pk = ...;
        let payload_hash: [u8; 32] = response.payload_hash.0;   // ← taken verbatim from caller
        near_mpc_signature_verifier::verify_ecdsa_signature(
            signature_response,
            &payload_hash,          // ← never checked against `request`
            &secp_pk,
        ).is_ok()
    }
    ...
};
``` [1](#0-0) 

The contract then resolves all queued yields for `request` with the unchecked `response`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The correct payload hash is defined off-chain as `ForeignTxSignPayload::V1 { request, values }.compute_msg_hash()`, where `values` are the extracted foreign-chain values. [3](#0-2)  The contract never recomputes or constrains this hash; it accepts whatever `payload_hash` the caller provides, so long as the signature over it is valid under the root key.

The `VerifyForeignTransactionRequest` used as the map key contains only `{ request: ForeignChainRpcRequest, domain_id, payload_version }` — no caller identity, no unique nonce. [4](#0-3)  Any attested participant can call `respond_verify_foreign_tx` with any matching request key. [5](#0-4) 

---

### Impact Explanation

**High — forged foreign-chain verification enabling invalid bridge execution.**

Attack scenario:

1. Victim bridge service calls `verify_foreign_transaction` for `tx_id_A` (e.g., a deposit of 100 ETH). The contract queues a yield under the key `{request: tx_id_A, domain_id, payload_version}`.
2. Byzantine attested participant submits `verify_foreign_transaction` for `tx_id_B` (e.g., a deposit of 0.001 ETH or a reverted transaction). The MPC network honestly verifies `tx_id_B` and produces a valid threshold signature over `hash({request: tx_id_B, values_B})`.
3. Before honest nodes respond to `tx_id_A`, the attacker calls `respond_verify_foreign_tx` with `request = {tx_id_A, ...}` but `response = { payload_hash = hash(tx_id_B, values_B), signature = valid_sig_over_that_hash }`.
4. The contract passes the signature check (the signature is genuinely valid over the provided hash under the root key) and resolves the victim's yield with the fraudulent response.
5. The victim's bridge contract receives `VerifyForeignTransactionResponse { payload_hash = hash(tx_id_B, values_B), signature }` — a response that attests to the wrong transaction.

If the bridge contract does not call `ForeignChainSignatureVerifier::verify_signature` from the SDK (which re-derives the expected hash from the original request and expected extracted values and compares), it will accept the forged attestation and may release funds or authorize a cross-chain action based on a transaction that was never verified. [6](#0-5) 

This maps directly to the allowed impact: *"forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

**Medium.** The attacker must be an attested MPC participant (reachable via the TEE attestation path, explicitly in scope as a "Byzantine participant strictly below the signing threshold"). No threshold collusion is required — the attacker reuses a legitimately produced MPC signature for a different transaction. The attacker must front-run honest nodes for the target request, which is feasible given NEAR's public mempool. The residual risk depends on whether deployed bridge contracts use the SDK verifier; contracts that trust the contract's response without re-verifying the payload hash are directly exploitable.

---

### Recommendation

The contract should bind the response's `payload_hash` to the submitted `request`. Since the contract cannot independently recompute the extracted values, the recommended fix is to have the MPC nodes include the extracted values in the respond call and have the contract recompute and assert:

```rust
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),   // new field
});
let expected_hash = expected_payload.compute_msg_hash()?;
assert!(
    expected_hash == response.payload_hash,
    "payload_hash does not match the request and extracted values"
);
```

Alternatively, enforce at the SDK level that `ForeignChainSignatureVerifier::verify_signature` is mandatory for all bridge contracts consuming `VerifyForeignTransactionResponse`, and document this as a security requirement.

---

### Proof of Concept

```
1. Attacker (attested participant) calls:
     verify_foreign_transaction({ request: tx_id_B, domain_id, payload_version })
   MPC network signs hash(tx_id_B, values_B) → sig_B

2. Victim calls:
     verify_foreign_transaction({ request: tx_id_A, domain_id, payload_version })
   Contract queues yield under key {tx_id_A, domain_id, payload_version}.

3. Attacker calls (before honest nodes respond):
     respond_verify_foreign_tx(
       request  = { tx_id_A, domain_id, payload_version },   // victim's key
       response = { payload_hash = hash(tx_id_B, values_B),  // wrong hash
                    signature    = sig_B }                    // valid sig over wrong hash
     )

4. Contract checks: verify_ecdsa_signature(sig_B, hash(tx_id_B, values_B), root_pk) → OK
   Resolves victim's yield with fraudulent response.

5. Victim's bridge contract receives VerifyForeignTransactionResponse attesting to tx_id_B,
   not tx_id_A. If it skips ForeignChainSignatureVerifier::verify_signature, it proceeds
   with an invalid cross-chain action.
```

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
