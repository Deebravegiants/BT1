Audit Report

## Title
Missing Staleness and Validity Checks in `ChainlinkPriceOracle.getAssetPrice()` Allows Stale Price to Distort rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `answeredInRound`, `roundId`, and does not validate that `price > 0`. A stale or invalid Chainlink price propagates into the rsETH exchange rate computation. Because `updateRSETHPrice()` is public and permissionless, an attacker can commit a stale deflated price on-chain and then deposit assets to receive excess rsETH, diluting existing holders when the oracle corrects.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at line 52 discards all validity fields:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

In contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` — used for the pool collateral path — correctly validates all three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The stale price flows through the following chain:

1. `LRTOracle.getAssetPrice()` delegates to `ChainlinkPriceOracle.getAssetPrice()` for each supported LST. [3](#0-2) 
2. `_getTotalEthInProtocol()` accumulates `totalETHInProtocol` using the stale price for every supported asset. [4](#0-3) 
3. `_updateRsETHPrice()` computes `newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply)` from the corrupted total. [5](#0-4) 
4. `updateRSETHPrice()` is public with no access control beyond `whenNotPaused`, so any caller can commit the stale price on-chain. [6](#0-5) 

**Existing guards are insufficient.** The `pricePercentageLimit` downside protection at lines 270–281 only triggers a pause if the price decrease exceeds the configured limit. [7](#0-6)  This guard fails in two ways: (a) `pricePercentageLimit` defaults to `0`, disabling it entirely until an admin sets it; (b) even when set, a staleness within the limit (e.g., a 0.5% deflation on a feed with a 1% limit) passes through undetected.

## Impact Explanation
**High — Theft of unclaimed yield from existing rsETH holders.**

When a Chainlink feed for any supported LST goes stale at a price lower than the true current price (e.e., feed froze during a price rally), `totalETHInProtocol` is undervalued, deflating `rsETHPrice`. An attacker calls the public `updateRSETHPrice()` to commit this deflated price, then deposits assets via `LRTDepositPool` and receives more rsETH than the true exchange rate warrants. When the oracle corrects on the next honest update, the attacker's excess rsETH represents a proportional claim on the pool that was not paid for — extracted directly from the yield accrued by existing rsETH holders.

## Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (1 hour to 24 hours depending on the feed). During low-volatility periods, feeds may not update for the full heartbeat window. On L2 deployments, sequencer downtime can cause feeds to go stale for extended periods. The attack requires no privileged access — only the ability to observe a stale feed and call the public `updateRSETHPrice()` at the right moment. The condition (stale feed + public price update function) is realistic and repeatable.

## Recommendation
Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_STALENESS_PERIOD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS_PERIOD` should be configured per-feed based on the Chainlink heartbeat (e.g., 3600 seconds for a 1-hour heartbeat feed, with a small buffer).

## Proof of Concept
**Setup:** stETH/ETH Chainlink feed has a 1-hour heartbeat. The feed last updated 59 minutes ago at `1.001e18`. The true current price has risen to `1.010e18` (feed froze during a rally), but the feed has not yet updated.

**Attack sequence:**
1. Attacker observes the stale feed returning `1.001e18` while the true price is `1.010e18`.
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.001e18`.
4. `totalETHInProtocol` is undervalued by ~0.9% → `newRsETHPrice` is deflated below its true value.
5. The price decrease is within `pricePercentageLimit` (or the limit is 0), so no pause is triggered.
6. Attacker deposits stETH via `LRTDepositPool.depositAsset()` and receives rsETH minted at the deflated price — more rsETH than the true exchange rate warrants.
7. When the Chainlink feed updates to `1.010e18` and an honest caller triggers `updateRSETHPrice()`, `rsETHPrice` corrects upward.
8. The attacker's excess rsETH is now worth more than what was deposited, at the expense of existing rsETH holders whose proportional share of the pool was diluted.

**Foundry fork test plan:** Fork mainnet/L2 at a block where a supported LST Chainlink feed is near its heartbeat boundary. Mock `latestRoundData()` to return a price with `updatedAt = block.timestamp - heartbeat - 1`. Call `updateRSETHPrice()`, record `rsETHPrice`. Deposit assets as attacker. Advance time, mock the corrected price, call `updateRSETHPrice()` again. Assert that attacker's rsETH redemption value exceeds the deposited asset value, and that existing holder redemption value decreased.
