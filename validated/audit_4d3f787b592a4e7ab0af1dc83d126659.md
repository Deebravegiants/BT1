### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Reward via Unvalidated `farmer_reward_address_override` in `RespondSignatures` — (`File: chia/farmer/farmer_api.py`)

### Summary
In `chia/farmer/farmer_api.py`, the `_process_respond_signatures` function accepts `farmer_reward_address_override` from a harvester's `RespondSignatures` message and uses it as the farmer reward puzzle hash in `DeclareProofOfSpace` without any fee-quality validation. A connected harvester can set this field to any arbitrary address, unconditionally redirecting the farmer's XCH block reward (1/8 of block reward + transaction fees) to an attacker-controlled address.

### Finding Description

The CHIP-22 fee mechanism allows third-party harvesters to redirect the farmer reward to themselves as a fee, but only when the proof's fee quality meets the harvester's declared threshold. The farmer is supposed to enforce this constraint. Instead, the farmer's `_process_respond_signatures` blindly substitutes any non-`None` `farmer_reward_address_override` from `RespondSignatures` as the final `farmer_reward_address`:

```python
# chia/farmer/farmer_api.py lines 916-919
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [1](#0-0) 

This `farmer_reward_address` is then passed directly into `DeclareProofOfSpace` and broadcast to the full node:

```python
return farmer_protocol.DeclareProofOfSpace(
    ...
    farmer_reward_address,   # attacker-controlled
    ...
)
``` [2](#0-1) 

The only fee-quality check that exists is in `notify_farmer_reward_taken_by_harvester_as_fee`, which is called in `new_proof_of_space` when the override appears in `NewProofOfSpace`. It only **logs a warning** on failure — it does not reject the proof or block the override:

```python
self.log.warning(
    "Harvester illegitimately took a fee reward that "
    + "did not belong to it or it incorrectly applied the fee convention."
)
``` [3](#0-2) 

Critically, `_process_respond_signatures` never calls `notify_farmer_reward_taken_by_harvester_as_fee` at all. A harvester can set `farmer_reward_address_override=None` in `NewProofOfSpace` (bypassing even the log warning) and then set `farmer_reward_address_override=<attacker_address>` only in `RespondSignatures`, which is the field that actually determines the on-chain reward destination.

The `farmer_reward_address_override` field is part of the `RespondSignatures` protocol message:

```python
class RespondSignatures(Streamable):
    ...
    farmer_reward_address_override: bytes32 | None
``` [4](#0-3) 

The full node uses whatever `farmer_puzzle_hash` the farmer declares, with no independent validation of whether it matches the farmer's configured address:

```python
farmer_ph = request.farmer_puzzle_hash
``` [5](#0-4) 

### Impact Explanation

When a farmer wins a block, the farmer reward coin (1/8 of block reward + all transaction fees) is created at `farmer_reward_puzzle_hash`. A malicious harvester that sets `farmer_reward_address_override` to its own puzzle hash in `RespondSignatures` causes this entire reward to be paid to the attacker's address instead of the farmer's configured `xch_target_address`. The farmer's XCH is permanently diverted with no recourse — the block is valid on-chain and the farmer has no mechanism to detect or prevent this before the block is confirmed.

This matches the **High** impact category: unauthorized payout redirection of XCH via bypass of farmer reward authorization.

### Likelihood Explanation

CHIP-22 is explicitly designed for third-party harvester services. Farmers who use third-party harvesters (a documented and supported use case) are directly exposed. The attack requires only a network connection from the harvester to the farmer — no key compromise, no cryptographic break. The harvester already holds the plot private key half needed to produce valid signatures, so it can complete the full block-winning flow while substituting its own reward address. The farmer has no in-protocol defense.

### Recommendation

In `_process_respond_signatures`, before accepting `response.farmer_reward_address_override`, perform the same fee-quality check that `notify_farmer_reward_taken_by_harvester_as_fee` performs. If the fee quality does not satisfy the declared threshold (or if no `fee_info` was provided in the original `NewProofOfSpace`), reject the override and fall back to `self.farmer.farmer_target`. The farmer should store the `fee_info` and `farmer_reward_address_override` from `NewProofOfSpace` alongside the proof in `self.farmer.proofs_of_space` so that `_process_respond_signatures` can cross-validate the `RespondSignatures` override against the originally declared threshold.

### Proof of Concept

1. A third-party harvester connects to the farmer's server.
2. The harvester finds a valid proof of space for a signage point.
3. The harvester sends `NewProofOfSpace` with `farmer_reward_address_override=None` and `fee_info=None` — no warning is triggered on the farmer.
4. The farmer sends `RequestSignatures` back to the harvester.
5. The harvester responds with `RespondSignatures` where `farmer_reward_address_override=<attacker_puzzle_hash>`.
6. `_process_respond_signatures` (lines 916–918) substitutes `attacker_puzzle_hash` as `farmer_reward_address` with no validation.
7. The farmer broadcasts `DeclareProofOfSpace` with `farmer_puzzle_hash=attacker_puzzle_hash` to the full node.
8. The full node creates the block; the farmer reward coin is locked to `attacker_puzzle_hash`.
9. The attacker spends the farmer reward coin to their wallet. The farmer receives nothing. [6](#0-5) [7](#0-6)
