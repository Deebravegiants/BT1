### Title
Missing Peer Qi Consistency Check in `deserialize_ac_key_blob` Allows Byzantine Participant to Force Honest-Party Abort in `sign_ac` — (`src/cbmpc/api/schnorr_mp.cpp`)

---

### Summary

`sign_ac` calls the jobless overload of `deserialize_ac_key_blob`, which only enforces the self-binding invariant (`x_share * G == Qi_self`) and does **not** validate peer Qis against the public key `Q` or any DKG transcript. A Byzantine participant can craft a blob with valid `x_share`/`Qi_self` but wrong peer Qis, pass deserialization, and cause `sign_batch`'s `h_consistency` broadcast to diverge between parties, forcing all honest parties to abort with `E_CRYPTO`.

---

### Finding Description

**Deserialization gap.** `sign_ac` calls the jobless `deserialize_ac_key_blob(ac_key_blob, ac_key)` overload: [1](#0-0) 

This overload (lines 222–258) performs:
- Version/curve/party-name format checks
- `x_share` range check
- `Q` is a valid curve point
- All `Qis` are valid curve points
- **Self-binding only**: `blob.x_share * G == Qis[blob.party_name]` [2](#0-1) 

It explicitly **omits** the `SUM(Qis) == Q` check that the additive `deserialize_key_blob` enforces: [3](#0-2) 

The comment at line 207–208 says peer Qi validation is deferred to "use sites," but `sign_ac` never performs it either. [4](#0-3) 

**Propagation through `to_additive_share`.** After deserialization, `sign_ac` calls `ac_key.to_additive_share(ac, quorum_names, additive_key)`: [5](#0-4) 

`to_additive_share` calls `reconstruct_pub_additive_shares` for each quorum party, which reads `Qis.at(node->name)` directly from the blob's (potentially crafted) Qis and applies Lagrange interpolation: [6](#0-5) [7](#0-6) 

The resulting `additive_key.Q` is copied verbatim from the blob's `Q` field, and `additive_key.Qis` are derived from the crafted peer Qis: [8](#0-7) 

**`sign_batch` pre-checks pass for the Byzantine party.** Because Lagrange interpolation is linear and `x_share * G == Qi_self` is enforced at deserialization, the Byzantine party's `new_x_share * G == new_Qi_self` holds, so line 67 passes: [9](#0-8) 

The Byzantine party can craft peer Qis such that `SUM(Lagrange-interpolated Qis) == Q` (e.g., `Qi_A' = Qi_A + delta`, `Qi_C' = Qi_C - delta`), so line 68 also passes.

**`h_consistency` divergence.** `sign_batch` computes and broadcasts: [10](#0-9) 

The hash covers `key.Q` and `key.Qis`. The Byzantine party's `additive_key.Qis` contain the crafted (wrong) individual Qi values, while honest parties' `additive_key.Qis` contain the correct DKG-derived values. The hashes diverge. When parties compare: [11](#0-10) 

All honest parties return `E_CRYPTO` and abort.

---

### Impact Explanation

A single Byzantine participant who controls their own `ac_key_blob` can force all honest parties in a signing quorum to abort with `E_CRYPTO` on every signing attempt. The honest parties cannot distinguish this from a legitimate protocol failure. This is honest-party divergence (Medium scope per the audit criteria).

---

### Likelihood Explanation

The attack requires only that the Byzantine party modify their own locally-stored blob — a trivial operation. No threshold collusion is needed; a single party below threshold suffices. The crafted blob passes all deserialization checks. The attack is deterministic and repeatable.

---

### Recommendation

In `deserialize_ac_key_blob` (both overloads), add a cross-party Qi consistency check. For the AC case, the appropriate invariant is that the sum of all stored Qis equals the stored `Q`:

```cpp
// After deserializing all Qis:
ecc_point_t Q_sum = curve.infinity();
for (const auto& kv : Qis) Q_sum += kv.second;
if (Q != Q_sum) return coinbase::error(E_FORMAT, "invalid ac key blob: Q != SUM(Qis)");
```

This mirrors the check already present in `deserialize_key_blob` for additive blobs. [3](#0-2) 

Note: for threshold AC blobs the Qis are threshold shares and their sum is not necessarily Q in general. If that is the case, the appropriate check is to validate each peer Qi against the access structure and the public key using the ZK proofs from DKG — or at minimum to bind the blob to a DKG session ID so that peer Qis cannot be substituted post-hoc.

---

### Proof of Concept

In a 3-party simulation with access structure `THRESHOLD[2](p0, p1, p2)` and signing quorum `{p0, p1}`:

1. Run honest DKG; party `p1` (Byzantine) obtains a valid `ac_key_blob` with `Qi_p0`, `Qi_p1 = x_p1 * G`, `Qi_p2`, `Q = Qi_p0 + Qi_p1 + Qi_p2`.
2. `p1` crafts a modified blob: `Qi_p0' = Qi_p0 + delta*G`, `Qi_p2' = Qi_p2 - delta*G` (for any nonzero scalar `delta`), keeping `Qi_p1` and `Q` unchanged. `SUM(Qis') = Q` still holds, so `sign_batch` line 68 passes.
3. `p0` uses its honest blob; `p1` uses the crafted blob.
4. `p0` computes `h_consistency = SHA256(msg, Q, {Qi_p0, Qi_p1})` (correct Lagrange-interpolated Qis).
5. `p1` computes `h_consistency = SHA256(msg, Q, {Qi_p0', Qi_p1})` (wrong Lagrange-interpolated Qi for p0).
6. Both parties broadcast; the check at `schnorr_mp.cpp:89` fails for both.
7. Assert: both parties return `E_CRYPTO`; no signature is produced.
