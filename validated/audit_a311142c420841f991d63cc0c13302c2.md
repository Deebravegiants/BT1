### Title
Deposit Permanently Frozen When Sub-Transaction Registers a Credential Deregistered by the Top-Level Transaction — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`)

---

### Summary

In the Dijkstra era's `LEDGER` rule, the top-level `UTXOW` sub-rule is invoked with the **original** `certState` (before any sub-transaction processing), while the `ENTITIES` rule that validates top-level certificates uses the **post-sub-transaction** `certState`. When a sub-transaction registers a stake credential (paying deposit `D`) and the top-level transaction deregisters that same credential, the deposit-refund calculation inside `UTXOW`/`UTXOS` returns zero because the credential is absent from the original `certState`. The deposit `D` is therefore permanently frozen in the deposit pot with no mechanism for recovery short of a hard fork.

---

### Finding Description

The Dijkstra `LEDGER` transition rule (`dijkstraLedgerTransition`) processes sub-transactions first, then the top-level certificates, and finally the top-level UTxO witness rule:

```
SUBLEDGERS  →  ENTITIES  →  GOV  →  UTXOW
```

Sub-transactions are processed by `SUBLEDGERS`, which threads the `LedgerState` through each `SUBLEDGER` call. Each `SUBLEDGER` in turn runs `SUBENTITIES` (which applies `applyWithdrawals` and processes certificates, updating `certState`) and then `SUBUTXOW` (which updates the UTxO state and deposit pot using the **pre-SUBENTITIES** `certState` for that sub-transaction — intentionally correct for that sub-transaction's own refund calculations).

After all sub-transactions complete, the top-level `ENTITIES` rule runs against `certStateAfterSubLedgers` — the fully updated cert state. This means a credential `X` registered by a sub-transaction is visible to the top-level `ENTITIES` rule, so a top-level `ConwayUnRegCert X SNothing` certificate succeeds.

However, the top-level `UTXOW` is then called with the **original** `certState`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs, lines 436–442
utxoStateFinal <-
  trans @(EraRule "UTXOW" era) $
    TRC
      ( DijkstraUtxoEnv slot pp (lsCertState ledgerState) originalUtxo
      , utxoStateBeforeUtxow
      , stAnnTx
      )
```

Inside `UTXOW` → `UTXO` → `updateUTxOStateNoFees`, the deposit-change calculation is:

```haskell
-- eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Utxo.hs, lines 627–629
totalRefunds = certsTotalRefundsTxBody pp certState txBody
totalDeposits = certsTotalDepositsTxBody pp certState txBody
depositChange = totalDeposits <-> totalRefunds
```

Because `certState` here is `lsCertState ledgerState` (the original, pre-sub-transaction state), credential `X` — registered only by a sub-transaction — is **not present**. `certsTotalRefundsTxBody` therefore computes a refund of `0` for the top-level deregistration of `X`, and the deposit pot is not decremented.

The sub-transaction's `SUBUTXO` correctly incremented the deposit pot by `D` when it registered `X`. The top-level `UTXOW` never decrements it. `D` is permanently stranded in the deposit pot.

The `ENTITIES` rule does not directly adjust the deposit pot; that is exclusively the responsibility of `UTXOW`/`UTXOS`. The `ConwayUnRegCert X SNothing` form skips the refund-amount cross-check in `ENTITIES`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs, lines 242–259
checkInvalidRefund = do
  SJust suppliedRefund <- Just sMayRefund   -- SNothing short-circuits here
  ...
```

So the deregistration succeeds, `X` is removed from `certState`, and `D` is irrecoverable.

---

### Impact Explanation

**High — Permanent freezing of funds (deposit) requiring a hard fork to recover.**

Once the top-level transaction is accepted:
- Credential `X` is deregistered and removed from `certState`; it cannot be re-deregistered.
- The deposit `D` remains in `utxosDeposited` with no corresponding registered credential to claim it.
- ADA conservation holds (the lovelace is in the deposit pot), but `D` is permanently inaccessible to any user.
- The only remediation is a hard fork that explicitly redistributes the orphaned deposit.

---

### Likelihood Explanation

**Medium.** The Dijkstra era is the only era with sub-transactions, so the attack surface is new and not yet on mainnet. However, the pattern — registering a credential in a sub-transaction and deregistering it in the outer transaction — is a natural use case for atomic credential lifecycle management.
