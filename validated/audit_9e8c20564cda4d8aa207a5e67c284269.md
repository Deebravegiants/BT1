### Title
ERC20 Tokens Sent to `FeeReceiver` Are Permanently Frozen — (File: contracts/FeeReceiver.sol)

---

### Summary

The `FeeReceiver` contract receives MEV/execution-layer ETH rewards and can forward them to the deposit pool via `sendFunds()`. However, the contract has no mechanism to withdraw ERC20 tokens. Any ERC20 tokens (LSTs, protocol tokens, or other ERC20 rewards) that arrive at `FeeReceiver` are permanently locked with no recovery path.

---

### Finding Description

`FeeReceiver` exposes only two operational functions:

- `receive() external payable` — accepts ETH
- `sendFunds() external` — forwards the entire ETH balance to `depositPool` [1](#0-0) 

There is no `withdrawToken`, `recoverTokens`, or any ERC20 transfer function anywhere in the contract. [2](#0-1) 

The interface `IFeeReceiver` also defines no ERC20 recovery event or function. [3](#0-2) 

By contrast, every other fee-accumulating contract in the codebase (e.g., `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3`, `RSETHPoolV3WithNativeChainBridge`) implements paired `withdrawFees(address receiver)` and `withdrawFees(address receiver, address token)` overloads, and the utility contract `Recoverable` provides both `recoverTokens` and `recoverETH`. [4](#0-3) 

`FeeReceiver` inherits none of these patterns.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

If any ERC20 tokens reach `FeeReceiver` — for example, EigenLayer ERC20 reward tokens distributed to the contract, LST tokens mistakenly routed here by an operator, or any future protocol integration that sends token rewards to the registered reward receiver address — those tokens are irrecoverable. There is no upgrade path within the contract itself to add a rescue function (it is an upgradeable proxy, so an upgrade could theoretically fix it, but the contract as written provides no on-chain remedy). The frozen tokens represent unclaimed yield that the protocol and its depositors are entitled to.

---

### Likelihood Explanation

**Low.** The primary and intended use of `FeeReceiver` is to accumulate ETH MEV/execution-layer rewards, which are handled correctly. ERC20 tokens would arrive only through an operational mistake (e.g., an operator sending LST rewards to the wrong address) or through a future protocol change that routes ERC20 rewards to the registered reward receiver. Neither path is part of the current normal flow, but both are plausible over the lifetime of a live protocol.

---

### Recommendation

Add an ERC20 token recovery function with appropriate access control, analogous to `Recoverable.recoverTokens`:

```solidity
function recoverERC20(address token, address recipient, uint256 amount)
    external
    onlyRole(DEFAULT_ADMIN_ROLE)
{
    IERC20(token).safeTransfer(recipient, amount);
}
```

Alternatively, inherit from `Recoverable` as other contracts in the codebase do.

---

### Proof of Concept

1. An operator or external integration calls `IERC20(stETH).transfer(feeReceiver, amount)` — a plausible mistake when routing LST rewards.
2. The tokens arrive at `FeeReceiver`.
3. No function in `FeeReceiver` can move ERC20 tokens out.
4. `sendFunds()` only transfers `address(this).balance` (ETH), leaving ERC20 balances untouched.
5. The tokens are permanently frozen. [5](#0-4)

pragma solidity 0.8.27;

import { Initializable } from "@openzeppelin/contracts-upgradeable/proxy/utils/Initializable.sol";
import { AccessControlUpgradeable } from "@openzeppelin/contracts-upgradeable/access/AccessControlUpgradeable.sol";
import { LRTConstants } from "./utils/LRTConstants.sol";
import { ILRTDepositPool } from "./interfaces/ILRTDepositPool.sol";
import { IFeeReceiver } from "./interfaces/IFeeReceiver.sol";

/// @title FeeReceiver
/// @notice Receives Mev/Execution-layer rewards
/// @dev also known as RewardReceiver Contract in LRTConstants.sol
contract FeeReceiver is IFeeReceiver, Initializable, AccessControlUpgradeable {
    address public _legacyProtocolTreasury;
    address public depositPool;
    uint256 public _legacyProtocolFeePercentInBPS;

    /// @custom:oz-upgrades-unsafe-allow constructor
    constructor() {
        _disableInitializers();
    }

    function initialize(
        address _protocolTreasury,
        address _depositPool,
        uint256 _protocolFeePercentInBPS,
        address admin,
        address manager
    )
        external
        initializer
    {
        if (
            _protocolTreasury == address(0) || _depositPool == address(0) || _protocolFeePercentInBPS == 0
                || admin == address(0) || manager == address(0)
        ) {
            revert InvalidEmptyValue();
        }

        _legacyProtocolTreasury = _protocolTreasury;
        depositPool = _depositPool;
        _legacyProtocolFeePercentInBPS = _protocolFeePercentInBPS;

        __AccessControl_init();
        _setupRole(DEFAULT_ADMIN_ROLE, admin);
        _setupRole(LRTConstants.MANAGER, manager);
    }

    /// @dev fallback to receive funds
    receive() external payable { }

    /// @dev send all rewards to deposit pool
    function sendFunds() external {
        uint256 balance = address(this).balance;
        ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();

        emit MevRewardsAddedToTVL(balance);
    }

    /*//////////////////////////////////////////////////////////////
                            MANAGER FUNCTIONS
    //////////////////////////////////////////////////////////////*/

    /// @dev Set the deposit pool
    /// @param _depositPool Address of the deposit pool
    function setDepositPool(address _depositPool) external onlyRole(LRTConstants.MANAGER) {
        if (_depositPool == address(0)) revert InvalidEmptyValue();

        depositPool = _depositPool;

        emit DepositPoolSet(_depositPool);
    }
}
