### Title
`respond_verify_foreign_tx` Does Not Validate `payload_hash` Against the Submitted Request - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` contract method verifies only that the ECDSA signature in the response is valid over the caller-supplied `response.payload_hash`. It never verifies that `response.payload_hash` is actually derived from the `request` content that was originally submitted. A single Byzantine MPC node that has legitimately assembled a threshold signature for any foreign-tx request can reuse that signature to resolve a completely different pending request with a fabricated `payload_hash`, causing forged foreign-chain verification or permanent request lifecycle corruption.

---

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` (lines 691–754) accepts two caller-supplied arguments: a `VerifyForeignTransactionRequest` (used as the queue key) and a `VerifyForeignTransactionResponse` (containing `payload_hash` and `signature`).

The contract's only cryptographic check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract verifies that `response.signature` is a valid MPC root-key signature over `response.payload_hash`. It does **not** verify that `response.payload_hash` is the canonical hash `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` for the specific `request` being resolved.

The correct hash is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> { ... }
}
``` [2](#0-1) 

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) does perform this check — it recomputes the expected hash from `(request, expected_extracted_values)` and compares it to `response.payload_hash` — but this check lives entirely off-chain in the caller's contract, not in the MPC contract itself. [3](#0-2) 

The MPC node-side code correctly computes `payload_hash` from the actual inspection result before submitting:

```rust
let payload_hash = response.0.0.compute_msg_hash()?;
let response = contract_args::VerifyForeignTransactionRespondArgs::from_signature(
    verify_foreign_tx_attempt.request.clone(),
    payload_hash, ...
)?;
``` [4](#0-3) 

But the contract never enforces this invariant on-chain.

---

### Impact Explanation

**Forged foreign-chain verification / request lifecycle corruption.**

A Byzantine MPC node that was the leader for a legitimately completed foreign-tx request (request A) possesses the assembled threshold signature `sig_A` over `payload_hash_A`. It can then call `respond_verify_foreign_tx` for any other pending request B with:

- `request = B` (a real pending request, observable from contract state)
- `response.payload_hash = payload_hash_A` (the hash from request A)
- `response.signature = sig_A` (the valid threshold signature over `payload_hash_A`)

The contract will:
1. Confirm the caller is an attested participant ✓
2. Verify `sig_A` over `payload_hash_A` against the root public key ✓
3. Find request B in `pending_verify_foreign_tx_requests` ✓
4. Resolve request B's yield with the wrong response [5](#0-4) 

The caller of request B receives `VerifyForeignTransactionResponse { payload_hash: payload_hash_A, signature: sig_A }`. Two outcomes follow:

- **If the bridge uses the SDK verifier**: the `expected_payload_hash_B != payload_hash_A` check fails, the bridge transaction reverts, and the yield is permanently consumed — the request cannot be retried. This is irreversible request lifecycle corruption.
- **If the bridge is naive and only checks signature validity** (not the `payload_hash` binding): the bridge accepts the response as proof that transaction B was verified, when the MPC network actually verified transaction A. This enables forged foreign-chain verification and potential double-spend conditions.

---

### Likelihood Explanation

The attack requires only a single Byzantine node that has been the leader for any one legitimate foreign-tx request. In a live network with multiple concurrent requests, any participant node will eventually lead a request and obtain a complete threshold signature. The pending request queue is public on-chain, so the attacker can observe and target any pending request B. No threshold collusion is required — the attacker reuses a legitimately produced signature.

---

### Recommendation

The contract must verify that `response.payload_hash` is bound to the submitted `request`. Since the `values` (extracted data) are not stored on-chain, the contract cannot recompute the full `payload_hash`. The recommended fix is to change the signing payload structure so the contract can verify the binding:

**Option A**: Extend `ForeignTxSignPayloadV2` to include a separately verifiable `request_hash` field:
```rust
pub struct ForeignTxSignPayloadV2 {
    pub request_hash: Hash256,  // SHA-256(borsh(ForeignChainRpcRequest))
    pub values: Vec<ExtractedValue>,
}
```
The contract can then compute `SHA-256(borsh(request))` from the submitted `request` and verify it equals `response.request_hash` before accepting the response.

**Option B**: In `respond_verify_foreign_tx`, require the node to also submit the `values` (extracted data), recompute `payload_hash` on-chain, and verify the signature against the recomputed hash. This is more expensive but fully eliminates the trust assumption.

---

### Proof of Concept

1. User submits `verify_foreign_transaction` for Bitcoin tx B → pending request B is queued.
2. MPC network processes a separate request A (Bitcoin tx A). The Byzantine leader node assembles the threshold signature `sig_A` over `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload::V1 { request_A, values_A }))`.
3. Byzantine node calls `respond_verify_foreign_tx(request=B, response={payload_hash: payload_hash_A, signature: sig_A})`.
4. Contract verifies `sig_A` over `payload_hash_A` → valid. Resolves request B's yield with `{payload_hash: payload_hash_A, signature: sig_A}`.
5. The bridge contract for request B receives a response claiming Bitcoin tx B was verified, but the `payload_hash` actually encodes the extracted values for Bitcoin tx A.
6. A naive bridge that only checks `verify_ecdsa_signature(sig_A, payload_hash_A, mpc_root_key)` accepts this as proof of tx B's finality — forged verification achieved with a single Byzantine node.
