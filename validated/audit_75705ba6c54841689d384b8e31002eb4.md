### Title
Fabricated CREATE_COIN Memo Hints in DL Singleton Spend Corrupt Wallet's `SingletonRecord` Root and Inner Puzzle Hash — (`chia/data_layer/data_layer_wallet.py`)

### Summary

`singleton_removed()` in `DataLayerWallet` extracts `root` and `inner_puzzle_hash` directly from CREATE_COIN memo hints (`condition[3][1]`, `condition[3][2]`) without verifying they are consistent with the actual child coin's puzzle hash (`condition[1]`). Any DL singleton owner can craft a spend with fabricated memo hints, causing every wallet tracking that singleton to permanently store a corrupted `SingletonRecord` with arbitrary `root` and `inner_puzzle_hash`.

### Finding Description

In `singleton_removed()`, the wallet runs the puzzle against the solution to obtain conditions, then for the first odd-amount CREATE_COIN condition:

- `full_puzzle_hash = bytes32(condition[1])` — the actual child coin puzzle hash, consensus-enforced [1](#0-0) 
- `root = bytes32(condition[3][1])` — taken from the memo hint, **never validated** [2](#0-1) 
- `inner_puzzle_hash = bytes32(condition[3][2])` — taken from the memo hint, **never validated** [2](#0-1) 

These two unvalidated values are then written directly into the DB:

```python
await self.wallet_state_manager.dl_store.add_singleton_record(
    SingletonRecord(
        ...
        root=root,                   # fabricated memo value
        inner_puzzle_hash=inner_puzzle_hash,  # fabricated memo value
        ...
        lineage_proof=LineageProof(
            parent_name,
            create_host_layer_puzzle(inner_puzzle_hash, root)  # derived from fabricated values
                .get_tree_hash_precalc(inner_puzzle_hash),
            amount,
        ),
    )
)
``` [3](#0-2) 

The missing invariant check is:

```python
create_host_fullpuz(inner_puzzle_hash, root, launcher_id)
    .get_tree_hash_precalc(inner_puzzle_hash) == full_puzzle_hash
```

This exact check **is** performed for the genesis spend in `match_dl_launcher()`: [4](#0-3) 

It is also available from `match_dl_singleton()`, which decodes the actual curried `root` and `innerpuz` from the parent puzzle itself: [5](#0-4) 

But `singleton_removed()` calls `match_dl_singleton()` only to confirm the puzzle shape, discards its return values, and then blindly trusts the memo hints. [6](#0-5) 

`add_singleton_record()` performs no validation; it is a plain `INSERT OR REPLACE`: [7](#0-6) 

### Impact Explanation

The corrupted `SingletonRecord` propagates into every downstream operation that reads `record.root` or `record.inner_puzzle_hash`:

1. **`get_coins_to_offer()`** computes the coin's puzzle hash from the corrupted fields, returning a coin that does not exist on-chain, breaking offer construction. [8](#0-7) 

2. **`get_puzzle_info()`** embeds the corrupted root into the `PuzzleInfo` metadata used for offer settlement. [9](#0-8) 

3. **`create_update_state_spend()`** reconstructs the current full puzzle from `singleton_record.root`, so any subsequent spend attempt by the wallet owner will produce the wrong puzzle and fail. [10](#0-9) 

4. The `lineage_proof.inner_puzzle_hash` stored in the record is also derived from the fabricated values, corrupting the lineage chain for all future generations. [11](#0-10) 

### Likelihood Explanation

The attacker must own the DL singleton — the inner puzzle requires a valid BLS signature. However, owning a DL singleton requires no special privilege; any network participant can create one. The victim's wallet must be tracking the singleton (e.g., via `track_new_launcher_id` for DL offer settlement, which is a standard workflow). The crafted spend is a fully valid on-chain transaction; consensus does not validate memo hint contents. The corruption is permanent (confirmed, generation-incremented record) and survives wallet restarts.

### Recommendation

In `singleton_removed()`, after extracting `root` and `inner_puzzle_hash` from the memo hints, add the cross-check that `match_dl_launcher()` already performs for the genesis spend:

```python
expected_full_puzzle_hash = create_host_fullpuz(
    inner_puzzle_hash, root, singleton_record.launcher_id
).get_tree_hash_precalc(inner_puzzle_hash)

if full_puzzle_hash != expected_full_puzzle_hash:
    self.log.warning(
        f"Parent {parent_name}: memo hints do not match child puzzle hash; ignoring spend"
    )
    return
```

Alternatively, derive `root` and `inner_puzzle_hash` from the child puzzle reveal (if available) rather than from unverified memo hints.

```

The unit test proposed in the question (calling `singleton_removed()` with a crafted `CoinSpend` whose CREATE_COIN memo has `root=bytes32([0xff]*32)` and `inner_puzzle_hash=bytes32([0xaa]*32)`, then asserting the stored record matches the actual child puzzle hash) would fail against the current code, confirming the bug.

def match_dl_singleton(puzzle: Program | SerializedProgram) -> tuple[bool, Iterator[Program]]:
    """
    Given a puzzle test if it's a CAT and, if it is, return the curried arguments
    """
    mod, singleton_curried_args = uncurry(puzzle)
    if mod == SINGLETON_TOP_LAYER_MOD:
        mod, dl_curried_args = uncurry(singleton_curried_args.at("rf"))
        if mod == NFT_STATE_LAYER_MOD and dl_curried_args.at("rrf") == ACS_MU_PH:
            launcher_id = singleton_curried_args.at("frf")
            root = dl_curried_args.at("rff")
            innerpuz = dl_curried_args.at("rrrf")
            return True, iter((innerpuz, root, launcher_id))

    return False, iter(())
```
