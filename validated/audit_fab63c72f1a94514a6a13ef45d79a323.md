### Title
Missing CKD Response Validity Check for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Confidential Key Derivation Output — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` enforces a cryptographic validity check on the CKD response only for the `AppPublicKeyPV` variant. For the `AppPublicKey` (legacy) variant, no such check exists. This is directly analogous to the ERC20Boost inconsistency: one code path enforces the full invariant while another silently skips it. A single malicious attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept and deliver it to the user.

---

### Finding Description

In `respond_ckd`, the CKD output check is gated on the variant of `app_public_key`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract verifies that the CKD output is cryptographically consistent with the supplied app public key pair before resolving the yield. For `AppPublicKey`, the contract performs **no cryptographic verification whatsoever** and immediately calls `resolve_yields_for`. [2](#0-1) 

Compare this to `respond`, which always verifies the ECDSA or EdDSA signature against the derived public key before resolving: [3](#0-2) 

And `respond_verify_foreign_tx`, which always verifies the ECDSA signature against the root public key before resolving: [4](#0-3) 

The threshold-enforcement mechanism in `respond` and `respond_verify_foreign_tx` is the cryptographic signature check: only a valid threshold-produced signature can pass. For `respond_ckd` with `AppPublicKeyPV`, `ckd_output_check` plays the same role. For `respond_ckd` with `AppPublicKey`, **there is no equivalent gate**, so the threshold requirement is entirely absent for this variant.

The `AppPublicKey` variant is described as "privately verifiable, legacy" — meaning only the user holding the corresponding private key can verify the output. The contract cannot verify it on-chain. However, the contract still resolves the yield and delivers the response to the user without any on-chain assurance of correctness. This creates an exploitable gap: a single Byzantine participant can supply any `CKDResponse` value and the contract will accept it.

---

### Impact Explanation

**Critical.** A single malicious attested participant (Byzantine, strictly below the signing threshold) can call `respond_ckd` with a forged `CKDResponse` targeting any pending `AppPublicKey` CKD request. The contract accepts the forged response and delivers it to the user via the yield-resume mechanism. The user receives a confidential key derivation output that was never produced by the threshold MPC protocol — it is entirely attacker-controlled. This constitutes unauthorized confidential key derivation output without the required participant authorization, matching the Critical impact tier.

---

### Likelihood Explanation

**High.** The `AppPublicKey` variant is still a supported, live code path (not removed or deprecated at the contract level). Any single attested participant — one that has passed TEE attestation and is in the active participant set — can exploit this. No threshold collusion is required. The attacker only needs to observe a pending `AppPublicKey` CKD request on-chain (which is public) and race to call `respond_ckd` with a forged response before the honest nodes do.

---

### Recommendation

Add a cryptographic validity check for the `AppPublicKey` variant in `respond_ckd`, or reject `AppPublicKey` requests at the `respond_ckd` entry point if on-chain verification is impossible for that variant. The check should be structurally equivalent to the `AppPublicKeyPV` branch so that both variants enforce the same threshold-authorization invariant before resolving yields.

---

### Proof of Concept

1. A user calls `request_app_private_key` with `app_public_key: AppPublicKey(g1_point)`. The request is stored in `pending_ckd_requests`.
2. A single malicious attested participant observes the pending request on-chain.
3. The attacker calls `respond_ckd(request, forged_ckd_response)` where `forged_ckd_response` contains an attacker-chosen confidential key encrypted to the user's `g1_point`.
4. `respond_ckd` passes all checks: caller is attested participant, protocol is running, TEE validation passes.
5. The `AppPublicKey` branch executes `{}` — no check — and `resolve_yields_for` is called immediately.
6. The user's pending yield is resolved with the forged response. The user receives a confidential key that was never produced by the threshold MPC protocol and is fully controlled by the attacker.
7. The honest MPC nodes' legitimate response, when it eventually arrives, finds no pending yield to resolve (already drained) and is silently discarded.
