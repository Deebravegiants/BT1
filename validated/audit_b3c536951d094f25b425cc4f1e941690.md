Audit Report

## Title
Unlocked ETH Withdrawal Requests Cannot Be Completed When Aave Has Insufficient Liquidity, Temporarily Freezing User Funds - (File: contracts/LRTWithdrawalManager.sol)

## Summary
When Aave integration is enabled, `unlockQueue` irreversibly burns the user's rsETH and deposits the redeemed ETH into Aave v3. If the Aave WETH pool subsequently reaches full utilization (all WETH borrowed out), `completeWithdrawal` reverts with `InsufficientLiquidityForWithdrawal` and no user-callable cancel or fallback path exists. The user's rsETH is permanently burned and their ETH is inaccessible until Aave liquidity is restored or a privileged admin intervenes.

## Finding Description
The ETH withdrawal lifecycle proceeds in three stages:

**Stage 1 — `initiateWithdrawal` (L166):** User transfers rsETH to the contract; a `WithdrawalRequest` is stored.

**Stage 2 — `unlockQueue` (L305–317):** rsETH is burned unconditionally (`burnFrom` at L305), ETH is redeemed from the vault (L307), and then deposited into Aave v3 via a `try/catch` self-call (L311). The `try/catch` means the deposit silently succeeds or fails — if it succeeds, the ETH is now inside Aave.

**Stage 3 — `completeWithdrawal` → `_processWithdrawalCompletion` (L720–731):** If the contract's ETH balance is below `request.expectedAssetAmount`, `_withdrawFromAave` is called. Inside `_withdrawFromAave` (L917), `aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this))` is called. If the Aave WETH pool has zero available liquidity (all WETH is borrowed), this external call reverts, propagating the revert up through `_processWithdrawalCompletion`. The balance check at L728 also independently reverts with `InsufficientLiquidityForWithdrawal` if `withdrawETH` returns without reverting but delivers less than needed.

**Root cause:** There is no user-callable cancel or fallback path. The `WithdrawalRequest` struct (ILRTWithdrawalManager.sol L39–43) has no state field. No `cancelWithdrawal` function exists anywhere in the contract. The only remediation paths are privileged: `emergencyWithdrawFromAave` (L551, callable only by `PAUSER_ROLE`) or `setAaveIntegrationEnabled(false)` (L469, callable only by `onlyLRTManager`), both of which require trusted admin action. The user has no independent recourse.

The `popFront` at L705 is atomically reverted with the rest of the transaction on failure, so the nonce is not lost — but the user is stuck retrying indefinitely until Aave liquidity recovers or an admin acts.

## Impact Explanation
**Medium — Temporary freezing of funds.** The user's rsETH is permanently burned (Stage 2 is irreversible) and the corresponding ETH is locked inside Aave. The user cannot complete the withdrawal, cannot cancel it, and cannot recover any asset independently. Funds remain frozen until Aave's WETH pool regains sufficient liquidity or a privileged role intervenes. This directly matches the allowed impact: *Medium. Temporary freezing of funds.*

## Likelihood Explanation
**Medium.** Aave v3 WETH pool liquidity crunches are documented real events during periods of high borrowing demand or market stress. The Kelp protocol may hold substantial ETH in Aave on behalf of many users simultaneously, meaning a single high-utilization event would block all pending ETH withdrawals at once. The Aave integration is an active, production-intended code path controlled by the manager (`isAaveIntegrationEnabled`). No attacker action is required — normal market conditions can trigger this state. Any user with an unlocked ETH withdrawal request is affected.

## Recommendation
1. **Add a user-callable `cancelUnlockedWithdrawal`** that verifies the caller is the original withdrawer, withdraws the ETH from Aave (or uses idle contract balance), and sends the ETH equivalent directly to the user (since rsETH is already burned).
2. **Alternatively**, modify `_processWithdrawalCompletion` so that when `_withdrawFromAave` cannot satisfy the full amount, the request is re-queued (nonce pushed back to front) rather than reverting, allowing the user to retry without losing their position.
3. **At minimum**, ensure `emergencyWithdrawFromAave` can be called without `isAaveIntegrationEnabled` being true, so the PAUSER_ROLE can always unblock users even if the integration is in a degraded state.

## Proof of Concept
1. `isAaveIntegrationEnabled == true`; Aave addresses are configured.
2. User calls `initiateWithdrawal(ETH_TOKEN, 1 ether rsETH, "")` → rsETH transferred to contract.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`:
   - L305: `IRSETH(...).burnFrom(address(this), rsETHBurned)` — rsETH permanently burned.
   - L307: `unstakingVault.redeem(asset, assetAmountUnlocked)` — ETH received by contract.
   - L311: `this.depositToAaveExternal(assetAmountUnlocked)` succeeds — ETH is now in Aave.
4. Aave WETH pool utilization reaches 100% (all WETH borrowed out by third parties).
5. User calls `completeWithdrawal(ETH_TOKEN, "")`:
   - L721–722: `address(this).balance < request.expectedAssetAmount` → true (ETH is in Aave).
   - L724: `_withdrawFromAave(amountNeeded)` called.
   - L917: `aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this))` reverts (no liquidity), OR L728: `balanceAfter < request.expectedAssetAmount` → `revert InsufficientLiquidityForWithdrawal()`.
6. Transaction reverts. User's rsETH is gone. No cancel function exists. ETH is frozen in Aave until liquidity is restored or admin intervenes via `emergencyWithdrawFromAave` or `setAaveIntegrationEnabled(false)`.
