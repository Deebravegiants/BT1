Audit Report

## Title
Block Stuffing Delays `sendFunds()`, Causing `updateRSETHPrice()` to Underprice rsETH by Excluding Pending MEV Rewards — (`contracts/FeeReceiver.sol`)

## Summary

`FeeReceiver.sendFunds()` is permissionless and is the only mechanism to move accumulated MEV/execution-layer rewards into `LRTDepositPool` where they are counted in TVL. `LRTOracle._getTotalEthInProtocol()` exclusively reads balances from the deposit pool and its downstream contracts; `FeeReceiver.balance` is explicitly excluded by design until `sendFunds()` is called. An attacker can use block stuffing to prevent `sendFunds()` from landing before calling `updateRSETHPrice()`, causing the oracle to record a price lower than the true backing per rsETH, enabling the attacker to deposit at the artificially depressed price and receive more rsETH than entitled, diluting existing holders.

## Finding Description

`FeeReceiver.sendFunds()` has no access control — any caller can invoke it, but it can also be blocked from inclusion via block stuffing:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`LRTOracle.updateRSETHPrice()` is equally permissionless:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`_getTotalEthInProtocol()` iterates supported assets and calls `getTotalAssetDeposits()` for each, which for ETH routes through `getETHDistributionData()`. That function explicitly excludes `FeeReceiver.balance` — the code comment at `LRTDepositPool.sol` L464–466 confirms this is intentional:

```solidity
/// @dev rewards are not accounted here
/// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
function getETHDistributionData()
```

`getETHDistributionData()` only counts `address(this).balance` (deposit pool), NDC balances, EigenLayer pod shares, the unstaking vault, and the converter — never `FeeReceiver.balance`:

```solidity
// contracts/LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

The rsETH price is then computed as:

```solidity
// contracts/LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

**Exploit path:**
1. Attacker monitors `FeeReceiver.balance` accumulating MEV rewards.
2. Attacker submits high-gas-price filler transactions to occupy block space across N blocks, preventing any `sendFunds()` call from being included.
3. Within the stuffed window, attacker calls `updateRSETHPrice()`.
4. Oracle records `rsETHPrice` with `FeeReceiver.balance` excluded from `totalETHInProtocol`, setting price below true backing.
5. Attacker calls `depositETH()` at the depressed price, receiving more rsETH than the true backing warrants (`rsethAmountToMint = (amount * assetPrice) / rsETHPrice` — lower denominator yields more rsETH).
6. Block stuffing ends; `sendFunds()` is called; `updateRSETHPrice()` corrects the price upward.
7. Attacker holds rsETH minted at the artificially low price, permanently diluting existing holders.

The existing downside-protection check at `LRTOracle.sol` L270–282 does not prevent this: it only pauses the protocol if the price drop exceeds `pricePercentageLimit`, which requires `FeeReceiver.balance` to be a large fraction of total TVL. For smaller reward balances the price drop passes through silently and `rsETHPrice` is updated to the understated value.

## Impact Explanation

**Low — Contract fails to deliver promised returns; Low — Block stuffing.**

The recorded `rsETHPrice` is lower than the true backing by `FeeReceiver.balance / rsethSupply`. New depositors receive more rsETH than they are entitled to, permanently diluting existing holders' share of protocol assets. The mispricing persists until `sendFunds()` is called and the oracle is updated again. No funds are directly stolen from the deposit pool, but existing rsETH holders receive less ETH-per-token than the protocol's actual backing warrants during the attack window.

## Likelihood Explanation

Both `sendFunds()` and `updateRSETHPrice()` are permissionless, so the attacker controls both sides of the timing race without any privileged role. Block stuffing on Ethereum mainnet is expensive but has been executed in practice against time-sensitive oracle windows. MEV rewards accumulate continuously; a sufficiently large `FeeReceiver.balance` makes the attack economically rational (profit from dilution exceeds block-stuffing cost). The attack is repeatable each time rewards accumulate to a profitable threshold.

## Recommendation

Include `FeeReceiver.balance` directly in `getETHDistributionData()` by reading the reward receiver address from config and adding it to `ethLyingInDepositPool`:

```solidity
address rewardReceiver = lrtConfig.getContract(LRTConstants.LRT_REWARD_RECEIVER);
ethLyingInDepositPool = address(this).balance + rewardReceiver.balance;
```

Alternatively, call `sendFunds()` atomically inside `_updateRsETHPrice()` before computing TVL. Either approach eliminates the timing dependency and makes the oracle invariant hold regardless of when `sendFunds()` is called.

## Proof of Concept

```solidity
// Fork test against local Anvil fork
function test_blockStuffing_rsETHMispricing() public {
    // 1. Simulate MEV rewards accumulating in FeeReceiver
    uint256 mevRewards = 10 ether;
    vm.deal(address(feeReceiver), mevRewards);

    // 2. Record price WITHOUT calling sendFunds() first
    //    (simulates attacker stuffing blocks to delay sendFunds)
    lrtOracle.updateRSETHPrice();
    uint256 priceWithoutRewards = lrtOracle.rsETHPrice();

    // 3. Now call sendFunds() and update price again
    feeReceiver.sendFunds();
    lrtOracle.updateRSETHPrice();
    uint256 priceWithRewards = lrtOracle.rsETHPrice();

    // 4. Assert price difference equals mevRewards / totalSupply (1e18 precision)
    uint256 rsethSupply = rseth.totalSupply();
    uint256 expectedDelta = (mevRewards * 1e18) / rsethSupply;

    assertApproxEqAbs(
        priceWithRewards - priceWithoutRewards,
        expectedDelta,
        1e9,
        "rsETH price must reflect pending MEV rewards"
    );

    // 5. Demonstrate attacker profit: deposit at depressed price
    uint256 depositAmount = 1 ether;
    uint256 rsethAtDepressedPrice = (depositAmount * 1e18) / priceWithoutRewards;
    uint256 rsethAtTruePrice     = (depositAmount * 1e18) / priceWithRewards;
    assertGt(rsethAtDepressedPrice, rsethAtTruePrice, "attacker receives excess rsETH");
}
```
