### Title
`verify_foreign_transaction` gates on strict all-participant intersection (`get_supported_foreign_chains`) instead of threshold-based availability, allowing a single Byzantine participant to block all foreign-chain verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` rejects any request whose target chain is absent from `get_supported_foreign_chains()`, which computes the **strict intersection** of every active participant's registered chain sets. Because the intersection collapses to empty the moment any single participant registers an empty list, one Byzantine participant strictly below the signing threshold can permanently block every `verify_foreign_transaction` request. This is the direct analog of the `current < previous` check that blocked adapters on any single-wei deviation: both lack a tolerance margin, and both allow a sub-threshold actor to disable a critical flow.

---

### Finding Description

In `crates/contract/src/lib.rs` at lines 533–541, `verify_foreign_transaction` calls `self.get_supported_foreign_chains()` and panics if the requested chain is absent:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported {
            requested: requested_chain,
        }
        .to_string(),
    );
}
``` [1](#0-0) 

`get_supported_foreign_chains()` computes the **strict intersection** of all active participants' registered foreign-chain sets. The design document `docs/design/calculating-supported-foreign-chains.md` explicitly states:

> "Today, `get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains, and `verify_foreign_transaction` rejects any request whose target chain is not in it. A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down." [2](#0-1) 

The proposed fix — gating on `get_available_foreign_chains()` (threshold-based: ≥ `signing_threshold` participants cover the chain) — is tracked under issue #3434 and is still in **"Proposed"** status, meaning the strict intersection check remains in production code. [3](#0-2) 

The analog to the external report is exact:

| External Report | NEAR MPC |
|---|---|
| `current < previous` (any single wei) blocks adapter | strict intersection (any single empty registration) blocks all chains |
| Missing `ADAPTER_BREAKS_LOSS_POINT` tolerance | Missing threshold-based tolerance (`signing_threshold` participants must cover chain) |
| Fix: only block if loss exceeds margin | Fix: only block if fewer than `signing_threshold` participants cover chain |

The `node_foreign_chain_support` field (type `SupportedForeignChainsByNode`) stores per-participant registrations and feeds `get_supported_foreign_chains()`. Any participant can update their own entry via `register_available_foreign_chain_config` (or the deprecated `register_foreign_chains_config`). [4](#0-3) 

---

### Impact Explanation

A Byzantine participant strictly below the signing threshold calls `register_available_foreign_chain_config` with an empty chain set. This immediately causes `get_supported_foreign_chains()` to return an empty intersection, and every subsequent `verify_foreign_transaction` call panics with `ForeignChainNotSupported`. All pending and future bridge verification requests are permanently blocked until the participant re-registers — a request-lifecycle invariant break that the threshold security model is supposed to prevent.

This matches the allowed **Medium** impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."* The threshold security model guarantees that up to `n − threshold` Byzantine participants cannot break the system, but this check allows a **single** participant to break the entire foreign-chain verification flow.

---

### Likelihood Explanation

The attack requires only a single participant (any one of the n MPC nodes) to call `register_available_foreign_chain_config` with an empty chain set. This is a normal, permissioned contract call available to any active participant. No threshold collusion, no key material, no TEE bypass, and no privileged access is needed. The participant can be Byzantine (below the signing threshold) and still execute this attack unilaterally.

---

### Recommendation

Replace `self.get_supported_foreign_chains()` in `verify_foreign_transaction` with `self.get_available_foreign_chains()`, which requires ≥ `signing_threshold` participants to cover the chain before accepting requests. This is precisely the fix proposed in `docs/design/calculating-supported-foreign-chains.md` and tracked under issue #3434. It mirrors the external report's recommendation to reintroduce `ADAPTER_BREAKS_LOSS_POINT`: instead of requiring unanimous coverage (strict intersection), require only a threshold of participants to cover the chain. [5](#0-4)

### Proof of Concept

1. Deploy the contract with n=3 participants, signing threshold=2.
2. All 3 participants register support for Bitcoin via `register_available_foreign_chain_config({Bitcoin})
