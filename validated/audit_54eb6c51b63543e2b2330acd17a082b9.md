### Title
Missing `maxCollateralInputs` Enforcement in Alonzo UTXO Transition Rule - (`eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs`)

---

### Summary

The Alonzo era UTXO transition rule defines and exports `validateTooManyCollateralInputs` but never calls it. The formal specification explicitly requires `‖collateral tx‖ ≤ maxCollateralInputs pp`, yet the implementation leaves only a dead comment at the enforcement site. An unprivileged transaction author can submit a transaction carrying far more collateral inputs than the `maxCollateralInputs` protocol parameter permits, bypassing the intended resource limit entirely.

---

### Finding Description

The Alonzo formal specification mandates two resource-limit checks side-by-side:

```
totExunits tx ≤ maxTxExUnits pp    ‖collateral tx‖ ≤ maxCollateralInputs pp
``` [1](#0-0) 

The implementation correctly enforces the first check. For the second, `validateTooManyCollateralInputs` is defined and exported: [2](#0-1) 

However, inside `utxoTransition`, after the `validateExUnitsTooBigUTxO` call, the collateral-count check is present only as a comment with no `runTest` invocation:

```haskell
  {-   totExunits tx ≤ maxTxExUnits pp    -}
  runTest $ validateExUnitsTooBigUTxO pp tx

  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  -- ← no runTest call here
  updatedGovState <- ...
``` [3](#0-2) 

Every later era correctly calls the check. Babbage's `babbageUtxoValidation`: [4](#0-3) 

Dijkstra's `dijkstraUtxoTransition`: [5](#0-4) 

Conway inherits from `babbageUtxoValidation` and is therefore also protected. Only the Alonzo `utxoTransition` is missing the enforcement. [6](#0-5) 

---

### Impact Explanation

**Medium — Attacker-controlled transactions exceed intended validation limits.**

The `maxCollateralInputs` protocol parameter exists specifically to bound the number of additional VKey signatures that must be checked during collateral validation and to cap the work done when consuming collateral on script failure. Without the enforcement call, a transaction author in the Alonzo era can include an arbitrarily large collateral input set (limited only by `maxTxSize`). This:

1. Violates the protocol-parameter constraint that governance set via `ppMaxCollateralInputsL`.
2. Forces every validating node to perform unbounded collateral-input processing work on any such transaction.
3. Constitutes a deviation from the formal specification that honest nodes implementing the spec strictly would reject, while nodes running this code would accept — a potential source of deterministic disagreement. [7](#0-6) 

---

### Likelihood Explanation

The entry path requires only constructing a valid Alonzo-era transaction body with a collateral input set whose cardinality exceeds `maxCollateralInputs` (currently 3 on mainnet). No privileged access, key compromise, or governance majority is needed. The transaction must still satisfy `maxTxSize`, but packing many `TxIn` references (each 34–36 bytes) into a transaction body is straightforward. Any wallet or script author can trigger this.

---

### Recommendation

Add the missing `runTest` call immediately after `validateExUnitsTooBigUTxO` in `utxoTransition`, mirroring the pattern already used in Babbage and Dijkstra:

```haskell
  {-   totExunits tx ≤ maxTxExUnits pp    -}
  runTest $ validateExUnitsTooBigUTxO pp tx

  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  runTest $ validateTooManyCollateralInputs pp txBody
``` [8](#0-7) 

---

### Proof of Concept

1. Construct an Alonzo-era `TxBody` with redeemers (so collateral is required) and populate `collateralInputsTxBodyL` with `N > maxCollateralInputs pp` distinct `TxIn` values, keeping total serialized size under `maxTxSize`.
2. Submit the transaction to an Alonzo-era ledger node.
3. The `utxoTransition` rule runs all checks including `validateExUnitsTooBigUTxO`, reaches the dead comment at line 570, and proceeds without rejecting the transaction.
4. The transaction is accepted and included in a block, despite violating `‖collateral tx‖ ≤ maxCollateralInputs pp`.
5. A conformant implementation that enforces the formal spec would reject the same transaction, producing a `TooManyCollateralInputs` predicate failure — demonstrating the divergence. [9](#0-8) [10](#0-9)

-- | Ensure that number of collaterals does not exceed the allowed @maxCollInputs@ parameter.
--
-- > ‖collateral tx‖  ≤  maxCollInputs pp
validateTooManyCollateralInputs ::
  AlonzoEraTxBody era =>
  PParams era ->
  TxBody TopTx era ->
  Test (AlonzoUtxoPredFailure era)
validateTooManyCollateralInputs pp txBody =
  failureUnless (numColl <= maxColl) $
    TooManyCollateralInputs Mismatch {mismatchSupplied = numColl, mismatchExpected = maxColl}
  where
    maxColl, numColl :: Word16
    maxColl = pp ^. ppMaxCollateralInputsL
    numColl = fromIntegral . Set.size $ txBody ^. collateralInputsTxBodyL
```
