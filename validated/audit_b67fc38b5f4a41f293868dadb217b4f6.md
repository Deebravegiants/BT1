Looking at the `InitTransfer` account struct and `process` function in detail:

### Title
Missing `token::authority = user` Constraint on `from` Account Enables Delegate-Authorized Theft via `init_transfer` — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

The `InitTransfer` account struct imposes no ownership check on the `from` token account. Because SPL Token's `transfer_checked` accepts a **delegate** as a valid authority, an attacker who holds a delegate approval on a victim's token account can call `init_transfer`, lock the victim's tokens in the bridge vault, and receive the equivalent tokens on NEAR — with themselves credited as the sender.

---

### Finding Description

The `from` account in `InitTransfer` is constrained only by mint and token-program equality: [1](#0-0) 

There is no `token::authority = user` constraint, so Anchor never verifies that `user` is the **owner** of `from`.

In `process`, the native-token path calls `transfer_checked` with `authority: self.user`: [2](#0-1) 

SPL Token's `transfer_checked` accepts either the account owner **or** an approved delegate with sufficient `delegated_amount`. If the signer (`user`) is a delegate on the victim's token account, the CPI succeeds.

The Wormhole message then encodes `self.user.key()` — the attacker — as the `sender`: [3](#0-2) 

This is serialized as field `params.1` in `serialize_for_near`: [4](#0-3) 

The same missing check applies to the burn path (bridged tokens, no vault): [5](#0-4) 

---

### Impact Explanation

**Critical.** The attacker locks the victim's Solana tokens in the bridge vault and is credited as the sender on NEAR, receiving the equivalent wrapped/native tokens there. The victim's tokens are irreversibly transferred out of their account without their consent to bridge. This is an unauthorized custody escape of locked bridge assets.

---

### Likelihood Explanation

Delegate approvals are routine in Solana DeFi (DEX routers, yield aggregators, lending protocols). Any existing approval — even one granted for a completely different purpose — is sufficient. The attacker does not need to compromise any key; they only need to be listed as `delegate` on the victim's token account with `delegated_amount >= transfer_amount`. This is a realistic, locally testable condition.

---

### Recommendation

Add `token::authority = user` to the `from` account constraint:

```rust
#[account(
    mut,
    token::mint = mint,
    token::authority = user,   // <-- add this
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

In Anchor, `token::authority` checks the `owner` field of the SPL token account, not the delegate. This ensures only the actual owner of `from` can initiate a bridge transfer.

---

### Proof of Concept

1. Victim holds token account `V` for mint `M` with balance 1,000,000.
2. Victim previously approved attacker pubkey `A` as delegate on `V` with `delegated_amount = 1,000,000` (e.g., via a DEX approval).
3. Attacker constructs an `init_transfer` instruction:
   - `from = V` (victim's account — passes `token::mint = mint` check)
   - `user = A` (attacker, signs the transaction)
   - `vault = bridge vault for M` (native token path)
   - `payload.amount = 1,000,000`, `payload.recipient = attacker's NEAR address`
4. Anchor validates `from`: only checks `token::mint` and `token::token_program` — both pass.
5. `transfer_checked` CPI: `authority = A`, `from = V` — SPL Token checks delegate list, finds `A` with sufficient allowance, **succeeds**.
6. 1,000,000 tokens move from `V` into the bridge vault.
7. Wormhole message posted with `sender = A` (attacker's pubkey).
8. On NEAR, the bridge credits attacker's NEAR address with the bridged tokens.

Victim loses 1,000,000 tokens. Attacker gains equivalent value on NEAR. No privileged access required beyond a pre-existing delegate approval.
