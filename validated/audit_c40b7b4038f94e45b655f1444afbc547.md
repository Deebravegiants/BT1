### Title
Missing Minimum Confirmation Threshold Validation in `verify_foreign_transaction` Allows Signing of Unconfirmed Bitcoin Transactions - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` contract entry point accepts a user-controlled `BlockConfirmations` value for Bitcoin requests without enforcing any minimum. Because the MPC nodes pass this value directly as the finality threshold to the Bitcoin inspector, a caller can set `confirmations: 0` and obtain a valid threshold signature attesting to a Bitcoin transaction that has zero on-chain confirmations — i.e., a mempool transaction that can still be double-spent or reorged.

---

### Finding Description

`BlockConfirmations` is a plain `u64` newtype with no minimum constraint:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:1282
pub struct BlockConfirmations(pub u64);
```

The ABI schema confirms `"minimum": 0.0`, meaning `0` is a valid wire value.

The contract's `verify_foreign_transaction` entry point performs only two checks — domain purpose and chain whitelist — before enqueuing the request:

```rust
// crates/contract/src/lib.rs:526-556
self.check_request_preconditions(request.domain_id, DomainPurpose::ForeignTx, ...);
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) { env::panic_str(...); }
// ← no validation of confirmations value
let request = args_into_verify_foreign_tx_request(request);
self.enqueue_yield_request(...);
``` [1](#0-0) 

The MPC node then passes the user-supplied value directly as the threshold to the Bitcoin inspector:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs:138-146
let block_confirmations = request.confirmations.0.into();
let extracted_values = inspector
    .extract(transaction_id, block_confirmations, extractors)
    ...
``` [2](#0-1) 

The Bitcoin inspector's finality check is:

```rust
// crates/foreign-chain-inspector/src/bitcoin/inspector.rs:51-58
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
``` [3](#0-2) 

When `block_confirmations_threshold = 0`, the condition `0 <= transaction_block_confirmation` is trivially true for every transaction, including mempool transactions with zero confirmations. Bitcoin's `getrawtransaction` (verbose) returns unconfirmed transactions with `confirmations: 0`, so the check passes unconditionally.

The signed payload (`ForeignTxSignPayloadV1`) embeds the full `ForeignChainRpcRequest`, including the attacker-chosen `confirmations: 0`, so the resulting signature attests to a 0-confirmation transaction:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:1478-1479
pub enum ForeignTxSignPayload {
    V1(ForeignTxSignPayloadV1),
}
``` [4](#0-3) 

```rust
// docs/foreign-chain-transactions.md:176-179
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,  // ← includes confirmations: 0
    pub values: Vec<ExtractedValue>,
}

### Impact Explanation

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow: a bridge contract on NEAR releases funds only after the MPC network attests that a Bitcoin deposit transaction has finalized. With `confirmations: 0`, an attacker can:

1. Broadcast a Bitcoin transaction to the mempool (0 confirmations).
2. Call `verify_foreign_transaction` with `confirmations: 0` before the transaction is mined.
3. Receive a valid MPC threshold signature over the 0-confirmation transaction.
4. Submit that signature to the bridge contract to claim NEAR-side funds immediately.
5. Double-spend or RBF-replace the original Bitcoin transaction, keeping both the Bitcoin and the NEAR-side funds.

This is a **forged foreign-chain verification** enabling a **double-spend / invalid bridge execution**. The MPC network's role is precisely to be the trusted enforcer of finality; bypassing it here removes the only on-chain safety guarantee.

**Impact: High** — matches "forged foreign-chain verification, light-client-style verification bypass... that causes invalid bridge execution or double-spend conditions."

---

### Likelihood Explanation

- Any unprivileged NEAR account can call `verify_foreign_transaction` with a 1-yoctoNEAR deposit.
- No special role, key, or collusion is required.
- The attack requires only a live Bitcoin mempool transaction and knowledge of the API, both trivially available.
- The `confirmations: 0` value is explicitly permitted by the ABI schema (`"minimum": 0.0`).

Likelihood: **High**.

---

### Recommendation

Enforce a protocol-level minimum confirmation threshold in the contract before enqueuing the request. For Bitcoin, the standard safe minimum is 6 confirmations; the contract should reject any request below a configurable floor:

```rust
// In verify_foreign_transaction, before enqueue_yield_request:
if let ForeignChainRpcRequest::Bitcoin(ref btc) = request.request {
    let min = self.config.bitcoin_min_confirmations; // e.g., 6
    if btc.confirmations.0 < min {
        env::panic_str("confirmations below minimum");
    }
}
```

Alternatively, ignore the caller-supplied value entirely and use the operator-configured minimum from the on-chain foreign-chain configuration, so the caller cannot weaken the finality guarantee.

### Proof of Concept

```
1. Deploy the MPC contract with Bitcoin whitelisted.
2. Broadcast a Bitcoin transaction T to the mempool (0 confirmations).
3. Call verify_foreign_transaction({
       request: Bitcoin({ tx_id: T, confirmations: 0, extractors: [BlockHash] }),
       domain_id: <foreign_tx_domain>,
       payload_version: V1
   }) with 1 yoctoNEAR attached.
4. The contract enqueues the request (no minimum check).
5. MPC nodes call getrawtransaction(T, verbose=true) → returns confirmations=0.
6. Inspector check: 0 <= 0 → passes.
7. MPC nodes sign ForeignTxSignPayloadV1 { request: Bitcoin({..., confirmations:0}), values: [BlockHash(H)] }.
8. respond_verify_foreign_tx is called; contract verifies signature against root key → valid.
9. Bridge contract receives VerifyForeignTransactionResponse with valid signature.
10. Attacker double-spends T on Bitcoin; bridge has already released NEAR-side funds.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
