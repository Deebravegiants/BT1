### Title
Missing Cryptographic Validation of CKD Response for Legacy `AppPublicKey` Variant Allows Single Byzantine Participant to Corrupt Pending Requests - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` performs no on-chain cryptographic validation of the `CKDResponse` when the request uses the legacy `AppPublicKey` (privately verifiable) variant. A single Byzantine attested participant can call `respond_ckd` with an arbitrary fabricated `CKDResponse`, which the contract unconditionally accepts, resolves the pending yield, and delivers wrong key material to the user. The pending request is permanently consumed, preventing the legitimate MPC-computed response from ever being delivered.

### Finding Description

In `respond_ckd` at `crates/contract/src/lib.rs` lines 675–682, the contract branches on the `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no validation whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For the `AppPublicKeyPV` (publicly verifiable) variant, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, cryptographically binding the response to the MPC network's key and the user's ephemeral public key.

For the `AppPublicKey` (legacy, privately verifiable) variant, the arm is an empty block — **any** `CKDResponse` with arbitrary `big_c` and `big_y` values passes without any check. [1](#0-0) 

The `ckd_output_check` function that is skipped for the legacy variant: [2](#0-1) 

After the (absent) check, `resolve_yields_for` is called unconditionally, draining all queued yields for the request and delivering the fabricated response to every waiting caller: [3](#0-2) 

The caller of `respond_ckd` must pass `assert_caller_is_attested_participant_and_protocol_active`, which requires only that the caller is a single attested participant in the current epoch — well below the signing threshold: [4](#0-3) [5](#0-4) 

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request on-chain.
2. Call `respond_ckd` with a fabricated `CKDResponse` containing arbitrary `big_c` / `big_y` G1 points.
3. The contract accepts the response without any pairing check, resolves all queued yields, and delivers the fabricated key material to the user(s).
4. The pending request entry is permanently removed from `pending_ckd_requests`; the legitimate MPC-computed response can never be delivered.

The user receives encrypted key material that does not satisfy the CKD protocol equation. If the user does not perform the off-chain verification step (as in the CLI example), they silently derive a wrong application key. Even if they do verify and reject it, the request is consumed and they must re-submit (paying the deposit again), while the attacker can repeat the attack indefinitely.

This breaks the production safety invariant that every delivered CKD response must be a valid output of the threshold MPC protocol over the network's key and the user's ephemeral public key.

**Allowed impact match**: Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation

The `AppPublicKey` (legacy) variant is the default and most widely documented format in the contract README and the `ckd-example-cli`. Any single attested participant — one that has passed TEE attestation and is registered in the current epoch — can exploit this. No threshold collusion is required. The attacker only needs to monitor the chain for pending `AppPublicKey` CKD requests and race to call `respond_ckd` before the legitimate leader does. [6](#0-5) 

### Recommendation

Apply `ckd_output_check` unconditionally for both variants, or reject the legacy `AppPublicKey` variant in `respond_ckd` if no on-chain check is possible for it. The publicly verifiable path already demonstrates the correct pattern:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {
        // Either: reject legacy variant in respond_ckd (force callers to use AppPublicKeyPV)
        // Or: add an equivalent pairing check using only pk1 (G1 point) if feasible
        env::panic_str("Legacy AppPublicKey variant cannot be verified on-chain; use AppPublicKeyPV");
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

If the legacy variant must remain supported for backwards compatibility, document clearly that it provides no on-chain integrity guarantee and consider adding a rate-limit or requester-only-respond mechanism.

### Proof of Concept

1. Alice submits a `request_app_private_key` call using the legacy `AppPublicKey` variant with deposit ≥ 1 yoctoNEAR. The request enters `pending_ckd_requests`. [7](#0-6) 

2. A Byzantine attested participant (Mallory) observes the pending request on-chain and immediately calls `respond_ckd` with:
   - `request`: the exact `CKDRequest` from Alice's submission
   - `response`: `CKDResponse { big_y: [0u8; 48].into(), big_c: [1u8; 48].into() }` (arbitrary garbage G1 points)

3. `respond_ckd` passes `assert_caller_is_attested_participant_and_protocol_active` (Mallory is a valid attested participant), hits the `AppPublicKey(_) => {}` arm with no check, and calls `resolve_yields_for`, draining Alice's yield with the fabricated response. [3](#0-2) 

4. Alice's transaction resolves with `CKDResponse { big_y: [0u8;48], big_c: [1u8;48] }`. The legitimate MPC response, when it arrives, finds no pending entry and is silently discarded.

5. Alice computes `secret = big_c - big_y * ephemeral_private_key` and gets a wrong key. The pairing verification `e(secret, g2) == e(hash_point, public_key)` fails, confirming the corruption — but the request is already gone.

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }

  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
