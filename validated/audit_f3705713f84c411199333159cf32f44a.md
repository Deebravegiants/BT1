### Title
Fee-on-Transfer Token Support Causes wrsETH Over-Minting and Wrapper Undercollateralization - (File: contracts/L2/RsETHTokenWrapper.sol)

### Summary
`RsETHTokenWrapper._deposit()` mints wrsETH 1:1 based on the caller-supplied `_amount` parameter rather than the actual tokens received after transfer. If an allowed `altRsETH` token charges a fee on transfer, the wrapper receives fewer tokens than `_amount` but mints the full `_amount` of wrsETH, permanently undercollateralizing the wrapper and causing the last redeemers to be unable to withdraw their funds.

### Finding Description
In `RsETHTokenWrapper._deposit()`, the contract performs a `safeTransferFrom` for `_amount` tokens and then immediately mints exactly `_amount` wrsETH to the recipient:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);   // <-- uses _amount, not actual received balance
    emit Deposit(_asset, msg.sender, _to, _amount);
}
```

If `_asset` is a fee-on-transfer token, the wrapper receives `_amount - fee` but mints `_amount` wrsETH. The invariant that `totalSupply() == sum(balanceOf(asset))` is broken from the first such deposit. Every subsequent deposit compounds the deficit. When users call `withdraw()`, the contract burns `_amount` wrsETH and attempts to `safeTransfer(_to, _amount)` of the underlying asset, but the contract holds less than `totalSupply()` worth of the asset. The last withdrawers will find the contract insolvent.

The same pattern exists in `AGETHTokenWrapper._deposit()` and in the ERC20 token deposit paths of `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`, where `amount` (the input) is used to compute `rsETHAmount` to mint rather than the actual tokens received.

### Impact Explanation
**Temporary/Permanent Freezing of Funds (Medium → Critical depending on fee magnitude).**

The wrapper becomes undercollateralized by the cumulative transfer fees. Users who deposited last and hold wrsETH will be unable to redeem their full balance of altRsETH because the contract's altRsETH balance is less than `totalSupply()`. In the worst case (large fee or many deposits), the shortfall grows unboundedly and the last redeemers permanently lose funds — constituting a permanent freeze of funds.

### Likelihood Explanation
The `allowedTokens` mapping is managed by `TIMELOCK_ROLE` via `addAllowedToken`. The initial token is set at `initialize` time. While current altRsETH bridge tokens are not fee-on-transfer, the protocol explicitly supports adding new allowed tokens. Additionally, tokens like USDT have the ability to enable transfer fees retroactively. Any future addition of a fee-on-transfer token to the allowed list, or any existing token enabling fees, immediately triggers this vulnerability for all subsequent deposits. The entry path is fully unprivileged — any user calling `deposit()` or `depositTo()` triggers the mis-accounting.

### Recommendation
Measure the actual received amount by comparing the contract's balance before and after the transfer, and use that delta for minting:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    uint256 balanceBefore = ERC20Upgradeable(_asset).balanceOf(address(this));
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    uint256 actualReceived = ERC20Upgradeable(_asset).balanceOf(address(this)) - balanceBefore;
    _mint(_to, actualReceived);
    emit Deposit(_asset, msg.sender, _to, actualReceived);
}
```

Apply the same fix to `AGETHTokenWrapper._deposit()` and all pool `deposit(token, amount, ...)` functions.
