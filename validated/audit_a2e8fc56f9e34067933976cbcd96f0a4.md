Audit Report

## Title
ETH Deposit Cap Bypass Due to Missing `amount` in Limit Check — (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric bounds check: the ERC-20 branch correctly includes the incoming `amount` in the comparison (`totalAssetDeposits + amount > limit`), but the ETH branch omits it entirely (`totalAssetDeposits > limit`). Any unprivileged depositor can call `depositETH` and push the protocol's ETH holdings arbitrarily above the configured `depositLimitByAsset` cap, violating the invariant that total ETH deposits must not exceed the configured ceiling.

## Finding Description
The function `_checkIfDepositAmountExceedesCurrentLimit` at L676–682 is the sole enforcement gate for the per-asset deposit cap:

```solidity
// contracts/LRTDepositPool.sol L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
``` [1](#0-0) 

For ETH, the check only asks whether the current total already exceeds the limit — it never considers the incoming deposit. This means:
- A deposit of any size passes as long as `totalAssetDeposits ≤ limit` at the time of the call.
- The equality boundary is also missed: when `totalAssetDeposits == limit`, the check returns `false`, allowing one more deposit of arbitrary size.

This function is called unconditionally from `_beforeDeposit` at L661: [2](#0-1) 

Which is invoked by the public, permissionless `depositETH` entry point at L87: [3](#0-2) 

No other guard exists for the ETH cap. The `getAssetCurrentLimit` view at L402–409 will return 0 or underflow once the limit is exceeded, giving integrators incorrect data: [4](#0-3) 

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The `depositLimitByAsset` cap for ETH is the protocol's primary safety bound on how much native ETH it will accept. Bypassing it means the protocol mints rsETH and accepts ETH beyond the intended ceiling, violating the invariant `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`. No direct fund theft or freeze occurs; the excess ETH is restaked normally. The impact is that the protocol does not enforce the cap it promises to enforce for ETH, while correctly enforcing it for all LSTs.

## Likelihood Explanation
**High.** `depositETH` is a public, payable, permissionless function. No role, whitelist, or additional condition stands between a depositor and the flawed check. The attacker only needs to observe that `totalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)` and call `depositETH{value: largeAmount}`. No special timing, front-running, or privileged access is required, and the attack is repeatable.

## Recommendation
Apply the same combined check used for ERC-20 tokens to the ETH branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This unifies the logic and ensures both the boundary case (`totalAssetDeposits == limit`) and the over-limit case are caught for ETH, mirroring the correct ERC-20 behavior.

## Proof of Concept
Assume `depositLimitByAsset(ETH) = 100 ether` and `getTotalAssetDeposits(ETH) = 99 ether`.

1. Attacker calls `depositETH{value: 500 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500 ether)`:
   - `totalAssetDeposits = 99 ether`
   - ETH branch: `99 ether > 100 ether` → `false` → limit not exceeded.
3. `_beforeDeposit` does not revert; rsETH is minted for 500 ETH.
4. `getTotalAssetDeposits(ETH)` is now 599 ETH — nearly 6× the intended cap.

For ERC-20 (e.g., stETH), the same attempt with `depositAmount = 500 ether` evaluates `99 + 500 > 100` → `true` → `MaximumDepositLimitReached` revert, confirming the asymmetry.

**Foundry test plan:**
```solidity
function test_ethDepositBypassesCap() public {
    // set depositLimitByAsset(ETH) = 100 ether, seed pool with 99 ether
    vm.deal(attacker, 500 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 500 ether}(0, "");
    assertGt(lrtDepositPool.getTotalAssetDeposits(ETH_TOKEN), 100 ether);
}
```
