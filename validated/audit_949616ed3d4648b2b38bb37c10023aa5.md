### Title
Missing On-Chain Output Validation for Legacy `AppPublicKey` CKD Variant Allows Single Byzantine Participant to Deliver Fabricated Derived Key - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC smart contract performs no cryptographic verification of the `CKDResponse` when the request uses the legacy `CKDAppPublicKey::AppPublicKey` variant. A single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with an entirely fabricated `(big_y, big_c)` pair for any pending CKD request, and the contract will accept it and deliver the wrong derived key to the user. The publicly-verifiable `AppPublicKeyPV` variant is protected by `ckd_output_check`, but the legacy variant has no equivalent guard.

---

### Finding Description

The `respond_ckd` function in `crates/contract/src/lib.rs` contains a branch that explicitly skips all output validation for the `AppPublicKey` (privately-verifiable, legacy) variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

After this match, the function unconditionally resolves all queued yields with whatever `CKDResponse` was supplied:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

The `AppPublicKeyPV` path is protected because the caller supplies both `pk1 = a·G1` and `pk2 = a·G2`, enabling the contract to run a pairing-based check (`ckd_output_check`) that verifies the ciphertext `(big_y, big_c)` is a valid encryption of `H(pk || app_id) · msk` under the user's ephemeral key. The `AppPublicKey` path provides only `pk1 = a·G1` (no `pk2`), so the same pairing check cannot be applied — but the contract makes no attempt to apply any alternative check. The result is that any single attested participant can submit an arbitrary `(big_y, big_c)` and the contract will accept it.

The `respond` function (for ECDSA/EdDSA signatures) does not have this problem: it always verifies the signature cryptographically against the derived public key before resolving yields. The `respond_ckd` function is inconsistent with this design.

---

### Impact Explanation

A user whose TEE application calls `request_app_private_key` with the `AppPublicKey` variant receives a `CKDResponse` that it decrypts as:

```
s = (big_c − a · big_y) / H(pk || app_id)
```

If `big_y` and `big_c` are fabricated by a Byzantine participant, `s` is a garbage or attacker-influenced value rather than the legitimate deterministic secret. The TEE application has no way to detect this on-chain (the contract accepted the response as valid). Any security-critical operation the TEE app performs with `s` — key derivation, encryption, signing — is compromised. This constitutes **confidential key derivation output manipulation by a single Byzantine participant below the signing threshold**, matching the allowed critical impact: *"Unauthorized… confidential key derivation output without the required participant authorization."*

---

### Likelihood Explanation

The `AppPublicKey` variant is the legacy default and is still actively used (the `ckd-example-cli` README and sandbox tests both exercise it). Any single attested participant can observe pending `AppPublicKey` CKD requests on-chain (they are stored in `pending_ckd_requests`) and race to call `respond_ckd` with a fabricated response before the honest leader does. No threshold cooperation is required; no key material needs to be leaked. The attacker only needs to be an attested participant — a role explicitly listed as an allowed adversary in the scope ("Byzantine participant strictly below the signing threshold").

---

### Recommendation

Apply the same cryptographic output verification to the `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. One approach is to require all new CKD requests to use `AppPublicKeyPV` (deprecating `AppPublicKey` for new submissions) so that `ckd_output_check` is always enforced. Alternatively, derive a commitment to the expected output during request submission (e.g., store a hash of the expected `big_s = H(pk || app_id) · msk` contribution) and verify it on response. At minimum, add a comment and a tracking issue acknowledging that `AppPublicKey` responses are unverified on-chain and that the security guarantee for this variant relies entirely on honest node behavior.

---

### Proof of Concept

1. User submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)` for `derivation_path = "mykey"`.
2. The request is stored in `pending_ckd_requests` and is publicly visible on-chain.
3. Byzantine attested participant constructs a fabricated response:
   ```rust
   let response = CKDResponse {
       big_y: dtos::Bls12381G1PublicKey([0xde; 48]),  // arbitrary
       big_c: dtos::Bls12381G1PublicKey([0xad; 48]),  // arbitrary
   };
   ```
4. Byzantine participant calls `respond_ckd(ckd_request, response)`.
5. The contract executes lines 675–688 of `crates/contract/src/lib.rs`: the `AppPublicKey` arm is a no-op, so `resolve_yields_for` is called immediately with the fabricated response.
6. The user's TEE application receives `(big_y=[0xde;48], big_c=[0xad;48])` and decrypts a garbage key `s`, believing it to be the legitimate MPC-derived secret.
7. The honest MPC nodes' subsequent (correct) `respond_ckd` call finds no pending yield to resolve (it was already drained) and returns an error, so the correct response is silently discarded.

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` at line 3403 of `crates/contract/src/lib.rs` already demonstrates that `CKDResponse { big_y: [1u8;48], big_c: [2u8;48] }` — clearly not a valid BLS12-381 point — is accepted without error for the `AppPublicKey` variant, confirming the absence of any output check.
