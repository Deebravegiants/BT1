### Title
`ForeignTxSignPayloadV1` Lacks Contract-Identity Domain Separation, Enabling Cross-Deployment Replay of Foreign-Chain Verification Signatures — (`File: crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The canonical signed payload for `verify_foreign_transaction` — `ForeignTxSignPayloadV1` — is Borsh-serialized and SHA-256 hashed to produce the 32-byte `msg_hash` that MPC nodes sign. This payload contains only the raw `ForeignChainRpcRequest` and the `Vec<ExtractedValue>`. It omits the NEAR contract account ID, the NEAR network identifier, and the `domain_id`. Because the signature is verified against the **root** (un-tweaked) public key, a valid `respond_verify_foreign_tx` response produced for one contract deployment is cryptographically indistinguishable from a valid response for any other deployment sharing the same key material. An unprivileged attacker who observes a valid response on one deployment can replay it verbatim on a second deployment to obtain an accepted foreign-chain verification attestation without the MPC network ever independently verifying the transaction for that deployment.

---

### Finding Description

`ForeignTxSignPayloadV1` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

The `msg_hash` is computed as:

```rust
pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
    let mut hasher = sha2::Sha256::new();
    borsh::BorshSerialize::serialize(self, &mut hasher)?;
    Ok(Hash256(hasher.finalize().into()))
}
```

The signed preimage commits to nothing beyond the foreign-chain request data and extracted values. It does **not** include:

- `env::current_account_id()` — the NEAR contract address
- A NEAR network identifier (mainnet vs. testnet)
- `domain_id` — which key domain produced the signature
- `payload_version` — the versioning discriminant

In `respond_verify_foreign_tx`, the contract verifies the signature against the **root** public key for the given domain (no tweak is applied for `ForeignTx` domains, as confirmed by the comment `// simulate signature with the root key (no tweak for foreign tx)` in the test):

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

Because the signed payload is purely a function of `(ForeignChainRpcRequest, Vec<ExtractedValue>)` and the verification key is the root key, any two contract deployments sharing the same root key will accept the same `(payload_hash, signature)` pair for the same foreign-chain request.

The `verify_foreign_transaction` function does not bind the pending request to the caller or to the contract identity — the request key is `VerifyForeignTransactionRequest { request, domain_id, payload_version }`, which an attacker can trivially reproduce on a second deployment:

```rust
let request = args_into_verify_foreign_tx_request(request);
// ...
move |this, id| this.add_verify_foreign_tx_request(request, id),
```

### Impact Explanation

**Cross-deployment replay of foreign-chain verification attestations.**

The NEAR MPC contract is subject to migration and upgrade. The documented migration path (`crates/node-types`, `crates/contract-history`, `conclude_node_migration`) transfers key material to a new contract account. During or after such a migration — or in any scenario where two contract deployments share the same `ForeignTx` domain root key (e.g., testnet/mainnet with accidentally shared key material, or a parallel deployment for a second bridge application) — the following holds:

A `VerifyForeignTransactionResponse` that was legitimately produced by the MPC network for deployment A is **cryptographically valid** on deployment B. The attacker does not need to compromise any MPC node or key share. They only need to:

1. Observe a valid `respond_verify_foreign_tx` call on deployment A (public NEAR transaction).
2. Submit the same `verify_foreign_transaction` request on deployment B (anyone can do this).
3. Call `respond_verify_foreign_tx` on deployment B with the replayed `(request, response)`.

Deployment B's contract accepts the response, resolves the pending yield, and returns the `VerifyForeignTransactionResponse` to the caller — without the MPC network ever querying the foreign chain for deployment B's context. Any bridge or application built on top of `verify_foreign_transaction` that trusts the returned attestation will accept a foreign-chain event as verified when it was not independently verified for that deployment.

This matches the allowed impact: **High — Cross-chain replay, forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

The precondition is that two contract deployments share the same `ForeignTx` domain root key. This is not the default steady-state, but it is a realistic condition during:

- **Contract migration/upgrade**: The documented migration service (`BackupServiceInfo`, `DestinationNodeInfo`, `NodeMigrations`) transfers key shares to a new contract. During the transition window, both the old and new contract may hold the same key.
- **Parallel deployments**: A second instance of the MPC contract deployed for a different bridge application, initialized with the same key material (e.g., by mistake or by design for key reuse).
- **Testnet/mainnet key reuse**: If the same key material is used on both networks during testing.

Once the precondition is met, the attack requires only reading public NEAR transaction history and submitting two transactions. No privileged access, no threshold collusion, no TEE bypass is needed. Likelihood is **Medium**.

---

### Recommendation

Include the NEAR contract account ID and the `domain_id` in the signed preimage. The minimal safe change is to add a domain-binding prefix to `ForeignTxSignPayloadV1` or to wrap the existing payload hash with a domain tag before signing:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub contract_id: AccountId,   // env::current_account_id() at request time
    pub domain_id: DomainId,
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

Alternatively, compute the final `msg_hash` as:

```
msg_hash = SHA-256(borsh(contract_id) || borsh(domain_id) || borsh(ForeignTxSignPayload))
```

The `contract_id` must be captured at the time `verify_foreign_transaction` is called (stored in the pending request or passed to the node via the indexed event) so that nodes include it when computing the payload hash before signing. The `domain_id` is already available in `VerifyForeignTxRequest` and can be included at zero cost.

A versioned payload format (`ForeignTxSignPayloadV2`) should be introduced to avoid breaking existing verifiers, with `V1` deprecated.

### Proof of Concept

**Setup**: Two NEAR MPC contract deployments, `v1.mpc.near` and `v2.mpc.near`, both initialized with the same `ForeignTx` domain root key `K` (e.g., after a contract migration).

**Step 1**: On `v1.mpc.near`, a bridge service submits:
```
verify_foreign_transaction({
    domain_id: 3,
    payload_version: V1,
    request: Bitcoin { tx_id: T, confirmations: 6, extractors: [BlockHash] }
})
```

**Step 2**: MPC nodes query Bitcoin, extract `block_hash = H`, and call:
```
respond_verify_foreign_tx(
    request = { domain_id: 3, payload_version: V1, request: Bitcoin{T, 6, [BlockHash]} },
    response = {
        payload_hash: SHA-256(borsh(ForeignTxSignPayload::V1 { request: Bitcoin{T,6,[BlockHash]}, values: [BlockHash(H)] })),
        signature: ECDSA_K(payload_hash)
    }
)
```
This call is public on the NEAR blockchain. The attacker records `(request, response)`.

**Step 3**: On `v2.mpc.near`, the attacker submits the same `verify_foreign_transaction` request (1 yoctoNEAR deposit, anyone can do this). A pending yield is created.

**Step 4**: The attacker calls `respond_verify_foreign_tx` on `v2.mpc.near` with the exact same `(request, response)` from Step 2.

**Step 5**: `v2.mpc.near` executes:
```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// payload_hash == SHA-256(borsh(ForeignTxSignPayload::V1{Bitcoin{T,6,[BlockHash]}, [BlockHash(H)]}))
// This is identical to what was signed for v1.mpc.near — no contract ID in the preimage.
near_mpc_signature_verifier::verify_ecdsa_signature(signature_response, &payload_hash, &secp_pk)
// secp_pk is the root key K, same on both deployments.
// Verification PASSES.
```

The pending yield on `v2.mpc.near` resolves successfully. The bridge on `v2.mpc.near` receives a `VerifyForeignTransactionResponse` attesting that Bitcoin tx `T` was verified — but the MPC network never queried Bitcoin for `v2.mpc.near`'s request. The attacker has forged a foreign-chain verification attestation on the second deployment at the cost of two NEAR transactions. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config

        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
