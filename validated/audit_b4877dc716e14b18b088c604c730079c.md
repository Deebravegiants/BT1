### Title
Deposits of Non-Executed Proposals Permanently Lost When `do_update` Clears All Entries - (File: crates/contract/src/update.rs)

### Summary

When a contract-update proposal reaches threshold and `do_update` is executed, it unconditionally clears **all** pending proposals and votes. Because `UpdateEntry` does not record the proposer's account ID or the attached deposit amount, the NEAR storage-staking deposits paid by every other proposer are permanently locked in the contract with no refund path.

### Finding Description

`propose_update` requires a deposit proportional to the size of the proposed update (contract binary or config blob):

```rust
// crates/contract/src/lib.rs
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// excess is refunded; the required portion stays in the contract
``` [1](#0-0) 

The deposit is computed from `bytes_used`, which for a 1 MB contract binary is on the order of 100 NEAR at NEAR's storage-byte cost:

```rust
fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
``` [2](#0-1) 

However, `UpdateEntry` stores only the update payload and its byte count — **not** the proposer's account ID or the deposit amount:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
``` [3](#0-2) 

When `do_update` fires, it removes the winning entry and then calls `self.entries.clear()` on all remaining proposals, issuing **zero refunds**:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
``` [4](#0-3) 

Because neither the proposer identity nor the deposit amount is persisted anywhere, there is no mechanism — now or later — to return those funds.

### Impact Explanation

Every participant who proposed a competing update loses their full storage-staking deposit the moment any other proposal crosses threshold. For a 1 MB contract binary the required deposit is approximately 100 NEAR per proposer. With multiple concurrent proposals (the contract explicitly supports this — `test_propose_update_contract_many` demonstrates two live proposals simultaneously), the total locked amount scales linearly with the number of competing proposers. The funds are permanently irrecoverable: the storage is freed by `clear()` but the NEAR tokens remain in the contract account with no withdrawal path. [5](#0-4) 

This breaks the production accounting invariant that storage-staking deposits must be returned when the storage they cover is released.

### Likelihood Explanation

The scenario is realistic in normal governance operation. Participants routinely propose competing upgrades (different binary versions, different configs). The contract is explicitly designed to hold multiple concurrent proposals. Any time a threshold vote executes one proposal, all others are silently wiped with no refund. No adversarial coordination is required — the loss is a deterministic consequence of the normal `vote_update` flow.

### Recommendation

Store the proposer's `AccountId` and the exact deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,      // add
    pub(super) deposit: NearToken,       // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the refund pattern already used in `propose_update` for excess deposits.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~100 NEAR. `UpdateEntry { update: ..., bytes_used: 1_000_000+ }` is stored; the 100 NEAR stays in the contract.
2. Participant B calls `propose_update` with a different binary, attaching another ~100 NEAR.
3. Threshold participants vote for proposal A via `vote_update`.
4. `do_update` is triggered: proposal A executes, then `self.entries.clear()` silently drops proposal B's entry.
5. Participant B's ~100 NEAR is permanently locked in the contract. No refund is ever issued because `UpdateEntry` carries no record of B's identity or deposit. [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
