Audit Report

## Title
Reward Accumulation Silently Skipped When `totalKernelStaked` Is Zero, Permanently Freezing Unclaimed Yield — (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary

In `KernelDepositPool`, the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`, while `rewardPerToken()` returns the stored value unchanged in that case. Any reward tokens emitted during a zero-staked interval are permanently unclaimable, with no admin sweep or recovery path in the contract.

## Finding Description

`rewardPerToken()` short-circuits and returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`:

```solidity
// L408-411
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
```

The `updateReward` modifier unconditionally advances `updatedAt` regardless:

```solidity
// L232-242
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();   // always advances
    ...
}
```

`initiateWithdrawal` applies the modifier before reducing `totalKernelStaked`, so the last staker's exit correctly snapshots `rewardPerTokenStored` and advances `updatedAt`. After that, `totalKernelStaked == 0` and time continues to pass within the reward window.

When the next `stake()` call fires `updateReward`: `rewardPerToken()` returns `rewardPerTokenStored` unchanged (supply is still 0 at modifier execution time, before `totalKernelStaked += _amount` in the function body), but `updatedAt` is set to the current timestamp. The entire idle interval `[last_updatedAt, now]` is consumed with zero `rewardPerTokenStored` accumulation. `rewardRate * T` tokens are permanently locked in the contract.

`notifyRewardAmount` guards against starting a period with zero stakers (`if (totalKernelStaked == 0) revert NoStakedTokens()`), but provides no protection against stakers exiting after a period has begun. The contract's own NatSpec at L14–23 acknowledges the root cause but relies entirely on an off-chain operational assumption with no on-chain enforcement.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** Reward tokens transferred via `notifyRewardAmount` become permanently unclaimable for any period during which `totalKernelStaked == 0`. The `rewardsToken` balance of the contract will exceed the sum of all claimable `rewards[user]` values, with the surplus irrecoverable. There is no admin sweep, rescue function, or redistribution mechanism.

## Likelihood Explanation

Any combination of stakers calling `initiateWithdrawal` that drives `totalKernelStaked` to zero during an active reward window triggers the condition. No privileged access is required. A natural full exit — all stakers unstaking after a reward period is announced but before it ends — is sufficient. The condition is repeatable across multiple reward periods.

## Recommendation

In the `updateReward` modifier, only advance `updatedAt` when `totalKernelStaked > 0`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This freezes the timestamp during zero-staked intervals so that reward emission is preserved and distributed once staking resumes.

## Proof of Concept

1. Admin calls `notifyRewardAmount(1000e18)` with `totalKernelStaked > 0`. `finishAt = block.timestamp + duration`, `rewardRate = 1000e18 / duration`.
2. All stakers call `initiateWithdrawal` for their full balance. Each call triggers `updateReward` (correctly advancing `rewardPerTokenStored`). After the last call, `totalKernelStaked == 0`.
3. `vm.warp(block.timestamp + T)` where `T < finishAt - block.timestamp`. During this window, `rewardRate * T` tokens should accrue but do not.
4. A new staker calls `stake(1e18)`. `updateReward` fires: `rewardPerToken()` returns `rewardPerTokenStored` unchanged (supply is 0 at modifier time). `updatedAt` is set to `block.timestamp`.
5. From this point, `rewardPerToken()` calculates from the new `updatedAt`, skipping the idle window entirely. Assert: `rewardsToken.balanceOf(address(pool)) > sum(rewards[user] for all users)` — the surplus is irrecoverable.
