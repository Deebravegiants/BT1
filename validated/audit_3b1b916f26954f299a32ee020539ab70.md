### Title
Merkle Leaf Omits Domain Separator, Enabling Cross-Deployment Proof Replay to Drain KERNEL Rewards — (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

### Summary
`KernelTop100MerkleDistributor._verifyClaimProof` constructs the Merkle leaf as `keccak256(abi.encodePacked(user, amount))`, binding only the recipient address and token amount. No contract address, chain ID, or any other domain separator is included. This is the direct analog of the ZK transcript omission: a proof component (the deployment context) is excluded from the commitment, so a proof that is valid in one deployment context is unconditionally valid in every other deployment that shares the same Merkle root. For a multi-chain protocol that distributes the same KERNEL reward tree across several networks, any eligible user can replay their single proof on every chain instance and drain the full vested allocation from each one.

### Finding Description
The vulnerable leaf construction is at line 293 of `KernelTop100MerkleDistributor.sol`:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
bool isValid = MerkleProofUpgradeable.verify(merkleProof, merkleRoot, leaf);
```

The leaf encodes only `(user, amount)`. It does not encode:
- `address(this)` — the contract instance
- `block.chainid` — the network
- Any index or epoch identifier

Compare this with the sibling contracts `KernelMerkleDistributor` and `MerkleDistributor`, which both include an `index` field in the leaf:

```solidity
// KernelMerkleDistributor line 320
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));

// MerkleDistributor line 120
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
```

`KernelTop100MerkleDistributor` has no `setMerkleRoot` function; the root is fixed at initialization. If Kelp DAO deploys this contract on multiple EVM chains with the same Merkle root (a natural operational choice when the same "Top 100" snapshot applies across chains), the leaf `keccak256(abi.encodePacked(user, amount))` is identical on every chain. A proof that verifies on Chain A verifies identically on Chain B, C, and D.

The vesting state (`userClaims`) is per-contract storage, so it is independent on each chain. The `amountClaimed` guard on one chain does not protect any other chain.

Attack flow:
1. Kelp DAO deploys `KernelTop100MerkleDistributor` on Ethereum, Arbitrum, and Optimism with the same `merkleRoot` (same Top-100 snapshot).
2. Attacker (a legitimate Top-100 user) calls `claim(amount, merkleProof)` on Ethereum and receives their vested KERNEL allocation.
3. Attacker submits the identical `(amount, merkleProof)` to the Arbitrum instance. The leaf `keccak256(abi.encodePacked(attacker, amount))` is the same bytes; `MerkleProofUpgradeable.verify` returns `true`; `userClaims[attacker].amountClaimed` on Arbitrum is still 0, so the full vested amount is paid out again.
4. Repeat on Optimism and every other chain instance.

### Impact Explanation
Each chain instance holds its own KERNEL token balance. A single valid proof drains the full vested allocation from every chain instance that shares the root. All 100 eligible users can independently execute this replay, collectively draining the entire KERNEL reward pool across all chains except the first one they claim on. This is direct theft of unclaimed yield (KERNEL tokens) from the protocol's reward contracts.

**Severity: High — Theft of unclaimed yield.**

### Likelihood Explanation
Kelp DAO is an explicitly multi-chain protocol. Deploying the same "Top 100" reward snapshot across several EVM networks is the natural operational pattern (one snapshot, many chains). The attacker requires only a legitimate inclusion in the Merkle tree and knowledge of their own proof — both are available to any Top-100 user. No privileged access, no front-running, no brute force is required. The attack is a single transaction per chain.

### Recommendation
Include a domain separator in the leaf hash that binds the proof to a specific contract instance and chain:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(
    block.chainid,
    address(this),
    user,
    amount
));
```

The off-chain Merkle tree builder must include the same fields when constructing leaves. This ensures a proof generated for one deployment is cryptographically invalid on any other deployment, exactly as a complete Fiat-Shamir transcript binds a ZK proof to its specific statement.

### Proof of Concept
1. Deploy `KernelTop100MerkleDistributor` on Chain A and Chain B with identical `merkleRoot`.
2. Fund both contracts with KERNEL tokens.
3. As a Top-100 user, obtain the Merkle proof for `(user=attacker, amount=X)`.
4. Call `claim(X, proof)` on Chain A → succeeds, `userClaims[attacker].amountClaimed = vestedAmount` on Chain A.
5. Call `claim(X, proof)` on Chain B → `_verifyClaimProof` recomputes `keccak256(abi.encodePacked(attacker, X))`, which is identical to the Chain A leaf; `MerkleProofUpgradeable.verify` returns `true`; `userClaims[attacker].amountClaimed` on Chain B is still 0; full vested amount is transferred again.

            revert ZeroValueProvided();

            revert InvalidMerkleProof();
