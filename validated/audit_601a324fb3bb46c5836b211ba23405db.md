### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Byzantine Participant - (File: crates/contract/src/lib.rs)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted ECDSA signature is valid over `response.payload_hash` under the domain's root public key. It does **not** verify that `payload_hash` is cryptographically bound to the `request` argument. A Byzantine participant (below signing threshold) who has observed a legitimately produced MPC signature for one pending request can replay that signature against a different pending request, causing the contract to deliver a forged verification response to an unrelated caller.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract performs two independent checks:

1. **Signature validity**: Is `response.signature` a valid ECDSA signature over `response.payload_hash` under the domain root key?
2. **Request lookup**: Does `request` match a pending yield in `pending_verify_foreign_tx_requests`? [1](#0-0) 

Critically, there is **no check** that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: <actual_extracted_values> }))`. The `payload_hash` is completely unconstrained relative to the `request` key used to resolve the yield. [2](#0-1) 

The intended invariant is that nodes independently compute:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))
``` [3](#0-2) 

and sign that hash. The contract is supposed to be the on-chain enforcer of this binding, but it is not.

**Contrast with `respond`** for regular `sign()` requests: there, the signature is verified against the *derived* key (root key tweaked by `derive_tweak(predecessor, path)`), so the tweak — which is part of the `SignatureRequest` — implicitly binds the signature to the request. [4](#0-3) 

For `verify_foreign_transaction`, the tweak is hardcoded to `[0u8; 32]` (the root key is used directly), so no such implicit binding exists. [5](#0-4) 

---

### Impact Explanation

**Attack scenario** (Byzantine participant strictly below signing threshold):

1. User A submits `verify_foreign_transaction(request_A)` — e.g., Bitcoin tx `[0x01; 32]`.
2. User B submits `verify_foreign_transaction(request_B)` — e.g., Bitcoin tx `[0x02; 32]`, same `domain_id`.
3. The MPC network honestly processes `request_A` and produces `(payload_hash_A, signature_A)` where `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload { request_A.request, extracted_values_A }))`.
4. The Byzantine participant (an attested participant who observed the network output) calls:
   ```
   respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: signature_A })
   ```
5. The contract verifies: is `signature_A` valid over `payload_hash_A` under the root key? **Yes** — it is a legitimately produced threshold signature.
6. The contract resolves the yield for `request_B` with `{ payload_hash_A, signature_A }`.
7. User B's transaction receives a response that claims to verify Bitcoin tx `[0x02; 32]` but actually contains the hash of Bitcoin tx `[0x01; 32]`'s extracted values.

A bridge contract consuming this response that checks only signature validity (without verifying `payload_hash` is bound to the expected request and extracted values) would accept a forged attestation and could execute an invalid cross-chain action (e.g., minting tokens for a transaction that was never verified).

The SDK's `ForeignChainSignatureVerifier` does perform the binding check: [6](#0-5) 

But the contract itself — the trust anchor — does not enforce it, leaving callers who do not use the SDK correctly exposed.

---

### Likelihood Explanation

**Medium.** The attacker must be an attested MPC participant (a real network member, not an arbitrary external caller). They do not need to forge a threshold signature — they only need to observe a legitimately produced signature for any request on the same domain and find a concurrent pending request to target. Both conditions are routinely satisfied in a live network processing multiple bridge requests.

---

### Recommendation

Require the caller to submit the `extracted_values` alongside the response, and verify on-chain that:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: extracted_values,
}).compute_msg_hash()?;
require!(expected_hash == response.payload_hash, "payload_hash does not match request");
```

This binds the `payload_hash` to the specific `request` being resolved, closing the replay window. Alternatively, include the `request` hash as a domain-separation prefix in the signed payload so the contract can verify the binding without needing the `extracted_values`.

---

### Proof of Concept

```rust
// Setup: two pending requests on the same ForeignTx domain
contract.verify_foreign_transaction(request_args_A); // pending for request_A
contract.verify_foreign_transaction(request_args_B); // pending for request_B

// Honest MPC network produces a valid response for request_A
let payload_A = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request_A.request.clone(),
    values: vec![ExtractedValue::BitcoinExtractedValue(
        BitcoinExtractedValue::BlockHash([0xAA; 32].into()),
    )],
});
let payload_hash_A = payload_A.compute_msg_hash().unwrap();
let (signature_A, recovery_id) = root_signing_key.sign_prehash_recoverable(&payload_hash_A.0).unwrap();
let response_A = VerifyForeignTransactionResponse {
    payload_hash: payload_hash_A,
    signature: SignatureResponse::Secp256k1(K256Signature::from_ecdsa_recoverable(&signature_A, recovery_id)),
};

// Byzantine participant replays response_A against request_B
// The contract accepts it because signature_A is valid over payload_hash_A under the root key
contract
    .respond_verify_foreign_tx(request_B.clone(), response_A.clone())
    .expect("contract incorrectly accepts cross-request replay");

// request_B's yield is now resolved with payload_hash_A (hash of request_A's data)
// User B receives a forged verification response

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
