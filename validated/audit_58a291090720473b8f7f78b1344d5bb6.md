Audit Report

## Title
RSETH Daily Mint Limit Blocks `_updateRsETHPrice()` When Protocol Fee Minting Is Required, Causing Temporary Oracle DoS — (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle._updateRsETHPrice()` calls `IRSETH.mint()` to collect protocol fees, which is gated by `RSETH.checkDailyMintLimit`. This modifier tracks a shared `currentPeriodMintedAmount` counter that accumulates across all rsETH minting, including ordinary user deposits via `LRTDepositPool._mintRsETH()`. Once user deposits exhaust `maxMintAmountPerDay`, any call to `updateRSETHPrice()` that would generate a non-zero protocol fee reverts entirely because `rsETHPrice = newRsETHPrice` is placed unconditionally after the unguarded `IRSETH.mint()` call. The price oracle is frozen for up to 24 hours, permanently losing that period's protocol fee and enabling new depositors to receive excess rsETH at the expense of existing holders.

## Finding Description
`LRTOracle._updateRsETHPrice()` contains two independent daily-limit checks that operate on separate counters:

1. **Oracle-level fee limit** (`_checkAndUpdateDailyFeeMintLimit`): tracks `currentPeriodMintedFeeAmount` against `maxFeeMintAmountPerDay` in `LRTOracle`. This check passes regardless of how much rsETH users have minted that day. [1](#0-0) 

2. **RSETH-level total mint limit** (`checkDailyMintLimit`): tracks `currentPeriodMintedAmount` against `maxMintAmountPerDay` in `RSETH.sol`. This counter is incremented by every call to `RSETH.mint()`, including user deposits routed through `LRTDepositPool._mintRsETH()`. [2](#0-1) [3](#0-2) 

The oracle's fee-limit check at line 303 can pass while the RSETH-level limit at line 50 fails, because the RSETH guard counts user-deposit minting that the oracle guard ignores entirely.

The critical structural flaw is that `rsETHPrice = newRsETHPrice` at line 313 is placed **after** the unguarded `IRSETH.mint()` call at line 306, with no `try/catch`. If `IRSETH.mint()` reverts with `DailyMintLimitExceeded`, the entire transaction reverts and the price is never updated. [4](#0-3) 

The `RSETH.mint()` function applies `checkDailyMintLimit` unconditionally on every call from any minter role, including both the deposit pool and the oracle. [5](#0-4) 

Both `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` (manager-only) call the same `_updateRsETHPrice()` with no bypass path. [6](#0-5) 

## Impact Explanation
When `currentPeriodMintedAmount` reaches `maxMintAmountPerDay`:
- Every call to `updateRSETHPrice()` reverts whenever `protocolFeeInETH > 0`, freezing `rsETHPrice` at its last stored value for up to 24 hours (**Temporary freezing of funds — Medium**).
- The protocol fee for the entire blocked period is permanently unrecoverable (**Permanent freezing of unclaimed yield — Medium**).
- New depositors calling `depositETH`/`depositAsset` during the freeze receive rsETH calculated at the stale (lower) price, minting excess rsETH and diluting existing holders' accrued yield (**Theft of unclaimed yield — High**).

The highest concrete impact is **High: Theft of unclaimed yield**, as the stale price directly transfers value from existing rsETH holders to new depositors during the freeze window.

## Likelihood Explanation
- `remainingDailyMintLimit()` on `RSETH` is a public view function, allowing an attacker to precisely measure the remaining capacity before acting. [7](#0-6) 
- The attacker deposits assets into `LRTDepositPool` to fill `currentPeriodMintedAmount` to `maxMintAmountPerDay`. The attacker receives rsETH in return, so net capital at risk is only price-movement exposure and gas — not a pure burn.
- The attack is repeatable every 24 hours with the same capital (deposit → wait for period reset → repeat).
- No privileged access is required; any external caller can trigger this via the public `depositETH` and `updateRSETHPrice` functions.
- Likelihood: **Medium** — requires meaningful capital but is economically rational if the attacker profits from the stale price window.

## Recommendation
Decouple the price write from the fee mint. Apply a `try/catch` around `IRSETH.mint()` so that a failed fee mint does not block the price update:

```solidity
if (rsethAmountToMintAsProtocolFee > 0) {
    try IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee) {
        emit FeeMinted(treasury, rsethAmountToMintAsProtocolFee);
    } catch {
        // fee deferred; roll back oracle counter
        currentPeriodMintedFeeAmount -= rsethAmountToMintAsProtocolFee;
    }
}
rsETHPrice = newRsETHPrice;
```

Alternatively, move `rsETHPrice = newRsETHPrice` before the fee-minting block so the price is always committed regardless of fee-mint success. Additionally, consider whether the RSETH-level `maxMintAmountPerDay` should exclude oracle fee mints from its counter, or whether the two limits should be coordinated so the oracle fee always has guaranteed headroom.

## Proof of Concept
1. Read `RSETH.remainingDailyMintLimit()` → returns `R` (remaining capacity in rsETH).
2. Attacker calls `LRTDepositPool.depositETH{value: R_in_ETH}(0, "")`, minting exactly `R` rsETH. `currentPeriodMintedAmount` now equals `maxMintAmountPerDay`.
3. EigenLayer rewards accrue; `totalETHInProtocol > previousTVL`, so `protocolFeeInETH > 0`.
4. Anyone calls `LRTOracle.updateRSETHPrice()`.
5. Execution reaches `IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee)` at line 306.
6. `checkDailyMintLimit` fires: `currentPeriodMintedAmount + rsethAmountToMintAsProtocolFee > maxMintAmountPerDay` → `revert DailyMintLimitExceeded`.
7. The entire `updateRSETHPrice()` call reverts. `rsETHPrice` remains at its stale value.
8. New depositors call `depositETH`/`depositAsset` at the stale lower price, receiving excess rsETH, diluting existing holders.
9. After 24 hours, `currentPeriodMintedAmount` resets and the oracle recovers — but the dilution and missed fee are permanent.
