Audit Report

## Title
Missing Initialization of `maxNumberOfWithdrawalsPerUser` in `KernelDepositPool.initialize()` Causes Temporary Freezing of Staked KERNEL Funds - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` does not set `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal()` unconditionally reverts when `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, and `0 >= 0` is always true, no user can ever initiate a withdrawal until the admin separately calls `setMaxNumberOfWithdrawalsPerUser()`. Any KERNEL staked between deployment and that admin call is temporarily frozen.

## Finding Description
`initialize()` only sets `_admin`, `_kernelToken`, and `_rewardToken`: [1](#0-0) 

`maxNumberOfWithdrawalsPerUser` therefore remains `0`. In `initiateWithdrawal()`, the guard at L323 is: [2](#0-1) 

Since `userWithdrawalIds[msg.sender].length` is a `uint256` starting at `0`, the condition `0 >= 0` evaluates to `true` on the very first call, reverting with `WithdrawalLimitReached()` before any state is modified. The only fix is the admin calling `setMaxNumberOfWithdrawalsPerUser()`: [3](#0-2) 

This setter enforces a non-zero value, confirming `0` is an invalid operational state. Because it is never invoked from `initialize()`, the contract is deployed in a broken configuration. `stake()` and `stakeFor()` succeed normally, so users can deposit KERNEL without any indication that withdrawal is impossible.

## Impact Explanation
All staked KERNEL tokens are temporarily frozen from deployment until the admin calls `setMaxNumberOfWithdrawalsPerUser()` with a valid value. This matches **Medium — Temporary freezing of funds** from the allowed impact scope. The freeze is not permanent because the admin can resolve it, but the window is unbounded and users have no recourse during it.

## Likelihood Explanation
Deployment requires a separate post-deployment admin transaction to configure `maxNumberOfWithdrawalsPerUser`. If the deployment script omits this step, the contract is silently misconfigured. The broken state is not immediately visible because `stake()` succeeds; only withdrawal attempts reveal the issue. This is a realistic deployment error. Likelihood is **Medium**.

## Recommendation
Set `maxNumberOfWithdrawalsPerUser` (and `withdrawalDelay`) directly inside `initialize()`, either by accepting them as parameters with the same bounds checks used in their respective setters, or by hardcoding a safe default. For example:

```solidity
function initialize(
    address _admin,
    address _kernelToken,
    address _rewardToken,
    uint256 _maxNumberOfWithdrawalsPerUser,
    uint256 _withdrawalDelay
) external initializer {
    // existing checks ...
    if (_maxNumberOfWithdrawalsPerUser == 0 || _maxNumberOfWithdrawalsPerUser > MAX_WITHDRAWALS_PER_USER)
        revert InvalidMaxNumberOfWithdrawalsPerUser();
    if (_withdrawalDelay == 0 || _withdrawalDelay > MAX_WITHDRAWAL_DELAY)
        revert InvalidWithdrawalDelay();

    maxNumberOfWithdrawalsPerUser = _maxNumberOfWithdrawalsPerUser;
    withdrawalDelay = _withdrawalDelay;
    // ...
}
```

## Proof of Concept
1. Admin deploys `KernelDepositPool` proxy and calls `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` is `0`.
2. User approves and calls `stake(1e18)`. Succeeds. User's KERNEL is now held by the contract.
3. User calls `initiateWithdrawal(1e18)`.
4. L323 evaluates: `userWithdrawalIds[msg.sender].length` is `0`; `maxNumberOfWithdrawalsPerUser` is `0`; `0 >= 0` is `true`.
5. Transaction reverts with `WithdrawalLimitReached()`.
6. User's KERNEL is frozen. No user-accessible path exists to recover funds until admin calls `setMaxNumberOfWithdrawalsPerUser(N)` where `1 <= N <= 100`.
