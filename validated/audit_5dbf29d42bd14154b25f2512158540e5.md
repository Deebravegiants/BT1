Audit Report

## Title
Dual-Oracle Staleness Divergence Enables agETH Over-Minting, Diluting Existing Holder Backing — (`contracts/agETH/AGETHPoolV3.sol`)

## Summary

`AGETHPoolV3.viewSwapAgETHAmountAndFee(uint256,address)` computes the agETH mint amount using two independent `CrossChainRateReceiver` oracles — one for the deposited token's ETH rate and one for agETH's ETH rate — with no staleness validation on either. When the two rates diverge due to natural cross-chain latency (one stale-high, the other stale-low), an unprivileged depositor receives more agETH than the ETH-equivalent value of their deposit, diluting the backing ratio for all existing holders and constituting theft of unclaimed yield.

## Finding Description

`AGETHPoolV3.deposit(address,uint256,string)` at lines 134–154 calls `viewSwapAgETHAmountAndFee(amount, token)`, which computes:

```solidity
agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate;
``` [1](#0-0) 

Both rates are fetched from `CrossChainRateReceiver` contracts:
- `agETHToETHrate = IOracle(agETHOracle).getRate()` (line 188)
- `tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate()` (line 191) [2](#0-1) 

`CrossChainRateReceiver.getRate()` returns the last stored `rate` with no staleness check whatsoever — `lastUpdated` is recorded but never validated against `block.timestamp`:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
``` [3](#0-2) [4](#0-3) 

The two receivers are updated independently via separate LayerZero `lzReceive` calls. Their `lastUpdated` timestamps can diverge naturally due to cross-chain latency, message drops, or network congestion — no oracle operator compromise is required. [5](#0-4) 

**Exploit path:**
1. Attacker monitors on-chain `rate` and `lastUpdated` on both receivers (both are `public`).
2. Waits for a favorable divergence: `supportedTokenOracle[token]` stale-high (e.g. +5%), `agETHOracle` stale-low (e.g. −5%).
3. Calls `deposit(token, amount, "")` — no role or pause gate exists in this path.
4. Receives `agETHAmount ≈ amount * 1.05 / 0.95 ≈ 1.105 × amount` in agETH, while the correct amount is `1.00 / 1.00 = 1.00 × amount`.
5. The ~10.5% excess agETH is backed by no additional ETH-equivalent value, diluting the backing ratio for all existing holders.

No existing guard prevents this: `onlySupportedToken` only checks that the oracle address is non-zero; `nonReentrant` prevents reentrancy only; there is no divergence check, no staleness threshold, and no circuit breaker in the deposit path. [6](#0-5) 

## Impact Explanation

Every agETH token represents a pro-rata claim on the pool's ETH-equivalent backing. As underlying assets (wstETH, etc.) appreciate, the agETH/ETH rate rises, accruing yield to existing holders. Over-minting agETH without proportional backing reduces the redemption value per agETH for all existing holders — this is a direct dilution of accrued, unclaimed yield. The attacker immediately holds agETH worth more than what they deposited, at the expense of existing holders. This matches the **High — Theft of unclaimed yield** impact class.

The `CrossChainRateReceiver` contracts are first-party protocol contracts (in `contracts/cross-chain/`), not third-party oracles, so the SECURITY.md exclusion for "Incorrect data supplied by third-party oracles" does not apply. [7](#0-6) 

## Likelihood Explanation

- Both oracles are `CrossChainRateReceiver` contracts updated via LayerZero; independent update cadences make divergence a routine occurrence, not a rare edge case.
- No role, pause gate, or staleness guard exists in the deposit path.
- The attacker needs only to observe public on-chain state (`rate`, `lastUpdated`) and submit a standard ERC-20 `approve` + `deposit` transaction.
- The attack is repeatable every time divergence recurs.
- No oracle operator compromise, governance capture, or privileged access is required.

## Recommendation

1. **Add a staleness threshold** in `CrossChainRateReceiver.getRate()` or in `AGETHPoolV3` before consuming rates: revert if `block.timestamp - lastUpdated > MAX_STALENESS`.
2. **Enforce a maximum divergence check** between the two rates before minting: if `|tokenToETHRate / agETHToETHrate - 1e18| > threshold`, revert.
3. Consider using a single on-chain oracle (e.g. Chainlink) for the token/ETH rate on the deployment chain to eliminate cross-chain latency as an attack surface.
