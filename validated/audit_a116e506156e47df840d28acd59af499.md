### Title
`_withdrawFromAave` Silently Caps Withdrawal Amount, Causing Temporary Freezing of User ETH Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

When Aave has suffered a loss (`aaveBalance < totalETHDepositedToAave`), `_withdrawFromAave` silently caps the amount withdrawn to `withdrawablePrincipal` without reverting. This causes `_processWithdrawalCompletion` to revert with `InsufficientLiquidityForWithdrawal`, temporarily freezing all user ETH withdrawals until privileged admin intervention.

---

### Finding Description

`_withdrawFromAave` (lines 905–921) computes a cap:

```solidity
uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave
    ? aaveBalance
    : totalETHDepositedToAave;

withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
```

When `aaveBalance < totalETHDepositedToAave` (Aave has suffered a loss), `withdrawablePrincipal = aaveBalance`. If the caller requests `amount > aaveBalance`, the function silently withdraws only `aaveBalance` and **returns without reverting**, delivering less ETH than requested. [1](#0-0) 

The caller in `_processWithdrawalCompletion` (lines 719–731) then checks the post-withdrawal balance:

```solidity
uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
_withdrawFromAave(amountNeeded);

uint256 balanceAfter = address(this).balance;
if (balanceAfter < request.expectedAssetAmount) {
    revert InsufficientLiquidityForWithdrawal();
}
``` [2](#0-1) 

Because `_withdrawFromAave` silently returned less than `amountNeeded`, `balanceAfter < request.expectedAssetAmount` is true and the entire transaction reverts. Since the revert unwinds all state changes (including the `delete withdrawalRequests[requestId]` and `unlockedWithdrawalsCount[asset]--` that already executed), the withdrawal request remains in the queue but **cannot be completed** — every subsequent call by any ETH withdrawer hits the same revert path until an admin manually disables the Aave integration.

This is the direct analog to the BentoBox `_exit()` pattern: a silent partial withdrawal (there, via `try/catch`; here, via silent capping) leaves a deficit that the protocol cannot self-heal, requiring privileged intervention and creating a centralization risk during the freeze window. [3](#0-2) 

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

All users holding unlocked ETH withdrawal requests are unable to call `completeWithdrawal` successfully. The freeze persists until the `PAUSER_ROLE` calls `emergencyWithdrawFromAave` and the LRT Manager calls `setAaveIntegrationEnabled(false)`. During this window, admins have full discretionary control over the timing of recovery, introducing a centralization risk analogous to the one described in the reference report.

---

### Likelihood Explanation

**Realistic.** Aave v3 WETH markets can experience periods where the aToken balance falls below the tracked principal due to bad debt socialization or protocol-level loss events. The contract explicitly routes all unlocked ETH through Aave (`unlockQueue` deposits to Aave on every unlock cycle), so the entire ETH withdrawal path is exposed whenever Aave's reported balance diverges from `totalETHDepositedToAave`. No attacker action is required; normal Aave market conditions are sufficient. [4](#0-3) 

---

### Recommendation

In `_processWithdrawalCompletion`, replace the bare `_withdrawFromAave` call with a check that compares the returned `withdrawnAmount` against `amountNeeded` before the balance verification, and revert with a clear error if the Aave withdrawal was insufficient. Alternatively, consider wrapping the Aave withdrawal in a try/catch that automatically sets `isAaveIntegrationEnabled = false` on failure so that subsequent user withdrawals can proceed from the contract's native ETH balance without requiring admin intervention.

---

### Proof of Concept

1. Aave integration is enabled. Operator calls `unlockQueue` for ETH; 100 ETH is deposited to Aave. `totalETHDepositedToAave = 100`.
2. Aave suffers a loss event. `aaveAWETH.balanceOf(address(this))` drops to `5`.
3. Alice calls `completeWithdrawal(ETH_TOKEN, "ref")`. Her `request.expectedAssetAmount = 10 ETH`. `address(this).balance = 0`.
4. `amountNeeded = 10 ETH`. `_withdrawFromAave(10 ETH)` is called.
5. Inside `_withdrawFromAave`: `aaveBalance = 5`, `withdrawablePrincipal = min(5, 100) = 5`, `withdrawnAmount = min(10, 5) = 5`. `withdrawETH(pool, 5, this)` executes; contract receives 5 ETH.
6. Back in `_processWithdrawalCompletion`: `balanceAfter = 5 ETH < 10 ETH` → `revert InsufficientLiquidityForWithdrawal()`.
7. The entire transaction reverts. Alice's withdrawal request remains locked in the queue. Every other ETH withdrawer hits the same revert. All ETH withdrawals are frozen until admin intervention. [5](#0-4)

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
```
