### Title
Frontrunning of `SignedDelegateAction` Allows Attacker to Grief Relayer and Block Meta Transaction Execution - (File: `runtime/runtime/src/actions.rs`)

---

### Summary

NEAR's meta transaction system (NEP-366) allows a user (Alice) to sign a `DelegateAction` off-chain and hand it to a relayer, who wraps it in an outer `SignedTransaction` and submits it to the network. Because the `SignedDelegateAction` is fully public inside the outer transaction — which is broadcast to multiple validators via P2P `ForwardTx` messages — an unprivileged attacker running a node can extract the `SignedDelegateAction`, wrap it in their own outer transaction, and submit it first. When the attacker's transaction is included before the relayer's, Alice's access-key nonce is consumed, and the relayer's transaction fails with `DelegateActionInvalidNonce`. This is a direct protocol-level analog to the EIP-2612 permit frontrunning vulnerability.

---

### Finding Description

**The off-chain signed payload is fully reusable by any submitter.**

A `SignedDelegateAction` is a self-contained, cryptographically signed object: [1](#0-0) 

It contains Alice's signature over the `DelegateAction` hash. The outer `SignedTransaction` that the relayer submits is signed by the relayer, but the inner `SignedDelegateAction` is signed only by Alice. There is no binding between the outer transaction's signer and the inner `SignedDelegateAction`. Any party who possesses the `SignedDelegateAction` bytes can wrap them in their own outer transaction and submit it.

**The outer transaction is broadcast to the network before inclusion.**

When the relayer submits the transaction, it is forwarded to upcoming chunk producers via P2P `ForwardTx` messages: [2](#0-1) 

Any node connected to the network can observe these messages and extract the `SignedDelegateAction` payload.

**Nonce is unconditionally consumed on first valid execution.**

`apply_delegate_action` calls `validate_delegate_action_key`, which checks the nonce and — if valid — immediately updates it: [3](#0-2) [4](#0-3) [5](#0-4) 

Once the attacker's transaction is included and the nonce is advanced to `delegate_nonce.nonce()`, the relayer's transaction — carrying the identical `SignedDelegateAction` — fails because `delegate_nonce.nonce() <= current_nonce` is now true.

**There is no fallback or idempotency check.**

Unlike the EIP-2612 fix (check if allowance is already sufficient), `apply_delegate_action` has no mechanism to detect that the intended action was already executed successfully. The relayer's transaction simply fails and the relayer's gas is burned.

---

### Impact Explanation

- **Relayer loses gas (NEAR tokens)** for the failed outer transaction — the relayer is the signer and pays all fees.
- **Alice's intended action is not executed** by the relayer's transaction. Alice must sign a new `DelegateAction` with a higher nonce and coordinate with the relayer again.
- **The attacker also spends gas** (they pay for their own outer transaction), so there is no profit motive — this is a pure griefing/DoS attack.
- The corrupted protocol value is Alice's **access-key nonce** (stored in the trie under `sender_id` + `public_key`) and the **action receipt** that the relayer's transaction was supposed to generate — it is never created.
- If the relayer has no fallback path (e.g., it does not retry with a fresh nonce), the DoS is long-term for that specific `SignedDelegateAction`.

---

### Likelihood Explanation

- The attacker only needs a NEAR account with a small NEAR balance to pay gas for the outer transaction — no validator or operator privileges are required.
- The `SignedDelegateAction` is fully visible to any node observing P2P `ForwardTx` messages, which are sent to multiple upcoming chunk producers.
- The attack is easy to automate: watch for `ForwardTx` messages containing `Action::Delegate`, extract the payload, and immediately re-submit it in a self-signed outer transaction.
- The attack is especially effective against high-value or time-sensitive meta transactions (e.g., token transfers with a tight `max_block_height`).

---

### Recommendation

Apply the same "trustless" pattern recommended in the external report. In `apply_delegate_action`, after a nonce failure, check whether the nonce was already advanced by the identical `SignedDelegateAction` (by storing and comparing the hash of the last consumed `SignedDelegateAction` alongside the nonce). If the hash matches, treat the execution as a no-op success rather than a failure. Alternatively, at the relayer application layer, always re-check the current on-chain nonce before submitting and, on `DelegateActionInvalidNonce`, verify whether the action was already executed before giving up.

---

### Proof of Concept

**Attack flow:**

1. Alice signs a `DelegateAction` (e.g., `ft_transfer("john", 10)`) off-chain and sends the `SignedDelegateAction` to a relayer.
2. The relayer constructs and submits:
   ```
   SignedTransaction {
     signer_id: relayer,
     actions: [Action::Delegate(alice_signed_delegate_action)],
     ...
   }
   ```
3. The node forwards this transaction to upcoming chunk producers via `NetworkRequests::ForwardTx`.
4. An attacker's node observes the `ForwardTx` P2P message, extracts `alice_signed_delegate_action`.
5. The attacker submits their own transaction:
   ```
   SignedTransaction {
     signer_id: attacker,          // different outer signer
     actions: [Action::Delegate(alice_signed_delegate_action)],  // same inner payload
     ...
   }
   ```
6. The attacker's transaction is included first. `validate_delegate_action_key` advances Alice's nonce to `delegate_nonce.nonce()`.
7. The relayer's transaction is then processed. `validate_delegate_action_key` finds `delegate_nonce.nonce() <= current_nonce` and returns `DelegateActionInvalidNonce`.
8. The relayer's transaction fails. The relayer loses gas. Alice's action was executed by the attacker's transaction (the attacker paid for it), but the relayer is uncompensated and its transaction is wasted.

The test `test_validate_delegate_action_key_update_nonce` in `runtime/runtime/src/actions.rs` already confirms that submitting the same `SignedDelegateAction` twice produces `DelegateActionInvalidNonce` on the second attempt: [6](#0-5)

pub struct SignedDelegateAction {
    pub delegate_action: DelegateAction,
    pub signature: Signature,

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}
