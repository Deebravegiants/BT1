Audit Report

## Title
`LiquidityLib.addLiquidity` permanently reverts via `SafeCast.toUint104()` when bin's `token0BalanceScaled` or `token1BalanceScaled` reaches `type(uint104).max` — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`LiquidityLib.addLiquidity` computes a proportional token deposit for an existing bin and immediately casts the updated running total to `uint104` via `SafeCast.toUint104()`. Although the entire function body is wrapped in `unchecked {}`, `SafeCast.toUint104()` is a function call and is unaffected by `unchecked` — it reverts unconditionally when its argument exceeds `type(uint104).max`. Once a bin's `token0BalanceScaled` or `token1BalanceScaled` reaches `type(uint104).max`, every subsequent `addLiquidity` call for that bin reverts, permanently closing it to new deposits.

## Finding Description

The `unchecked {}` block opens at line 51 and wraps the entire function body: [1](#0-0) 

Inside the `binTotalSharesVal > 0` branch, the proportional deposit is computed and immediately cast: [2](#0-1) 

`_checkedMul` is a separate internal function with no `unchecked` body of its own — it performs checked multiplication: [3](#0-2) 

In Solidity 0.8+, `unchecked {}` suppresses native arithmetic overflow checks only within the block itself; it does **not** propagate into called functions. `SafeCast.toUint104()` is a function call and will revert unconditionally if its argument exceeds `type(uint104).max`, regardless of the surrounding `unchecked` context.

**How the near-max state is reached (first LP path):** When `binTotalSharesVal == 0` and `binIdx > curBinIdxCache`, the first LP's deposit is: [4](#0-3) 

With `initialScaledToken0PerShareE18 = 1e18` and `sharesToAdd = type(uint104).max`, `amount0Scaled = type(uint104).max`. The cast at line 114 then succeeds (`0 + type(uint104).max = type(uint104).max`), setting `token0BalanceScaled = type(uint104).max`.

**DoS trigger (second LP):** Any subsequent call with `sharesToAdd = 1`:
- `amount0Scaled = ceilDiv(type(uint104).max × 1, type(uint104).max) = 1`
- Sum = `type(uint104).max + 1 > type(uint104).max` → `.toUint104()` reverts

The same overflow applies to `token1BalanceScaled` at line 118: [5](#0-4) 

The swap path in `SwapMath.sol` contains the identical pattern and can organically push a bin balance to exactly `type(uint104).max` through normal trading, leaving the bin in the DoS state without any deliberate griefing: [6](#0-5) 

`BinState.token0BalanceScaled` and `token1BalanceScaled` are both `uint104`: [7](#0-6) 

## Impact Explanation

`addLiquidity` is permanently broken for any bin whose `token0BalanceScaled` or `token1BalanceScaled` reaches `type(uint104).max`. No new LP can deposit into that bin. Existing LPs can still call `removeLiquidity` (which uses floor division and decreases the balance), but the bin is closed to all new deposits. This constitutes broken core pool liquidity functionality meeting the Sherlock medium threshold.

## Likelihood Explanation

Two independent paths reach the DoS state:

1. **Deliberate griefing:** A first LP sets `sharesToAdd = type(uint104).max` with `initialScaledToken0PerShareE18 = 1e18`. The actual token cost is `type(uint104).max / token0ScaleMultiplier`. For tokens where the scale multiplier is 1 (tokens with the highest decimal count among the pair), this equals `type(uint104).max` raw units — economically costly but feasible for high-supply or high-decimal tokens.

2. **Organic swap accumulation:** Normal buy-token1 swaps accumulate `token0BalanceScaled` via the `toUint104()` cast in `SwapMath`. A swap that lands the balance at exactly `type(uint104).max` succeeds and leaves the bin in the DoS state with no attacker involvement. This path requires no capital beyond normal trading.

## Recommendation

Before writing the updated balance, validate the sum fits in `uint104`:

```solidity
uint256 newBalance0 = uint256(binState.token0BalanceScaled) + amount0Scaled;
if (newBalance0 > type(uint104).max) revert BinBalanceOverflow();
binState.token0BalanceScaled = uint104(newBalance0);
```

Apply the same fix to `token1BalanceScaled` at line 118 and to the analogous `toUint104()` casts in `SwapMath.sol` at lines 499–501.

// Foundry unit test sketch (18-decimal tokens, initialScaledToken0PerShareE18 = 1e18)
function test_addLiquidity_token0BalanceScaled_overflow() public {
    // Step 1: First LP sets token0BalanceScaled = type(uint104).max
    // sharesToAdd = type(uint104).max, binIdx > curBinIdx (token0-only bin)
    _doAddLiquidity(USER_A, 0, _createDeltaArray(binIdx, type(uint104).max));

    // Verify bin balance is at max
    (uint104 bal0,,,,) = _getBinState(int8(binIdx));
    assertEq(bal0, type(uint104).max);

    // Step 2: Any second LP adding even 1 share reverts via SafeCast.toUint104()
    // amount0Scaled = ceilDiv(type(uint104).max * 1, type(uint104).max) = 1
    // sum = type(uint104).max + 1 overflows uint104 → revert
    vm.expectRevert(); // SafeCast: value doesn't fit in 104 bits
    _doAddLiquidity(USER_B, 0, _createDeltaArray(binIdx, 1));
}
