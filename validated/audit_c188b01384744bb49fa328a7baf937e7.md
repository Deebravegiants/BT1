### Title
`DirectDeposits` Omitted from Value Conservation Check Allows Unbounded ADA Creation — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the new `DirectDeposits` field in transaction bodies allows a transaction to credit ADA directly into registered stake account balances. However, the `directDeposits` amount is never included in the **produced** side of the preservation-of-value equation. The value conservation check therefore passes even when the transaction's UTxO inputs do not cover the direct-deposit amounts, allowing an unprivileged transaction sender to create ADA from nothing.

---

### Finding Description

The Dijkstra era introduces `DirectDeposits` — a `Map AccountAddress Coin` field present in both `TopTx` and `SubTx` transaction bodies — that credits ADA directly into registered stake account balances via `applyDirectDeposits`.

The Cardano ledger enforces a strict preservation-of-value invariant: every valid transaction must satisfy `consumed == produced`, where:

- **consumed** = UTxO inputs + reward withdrawals + deposit refunds
- **produced** = UTxO outputs + fees + certificate deposits + treasury donation

The Dijkstra-era produced-value function `dijkstraProducedValue` is defined as:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody          -- top-level body
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)             -- sub-transactions
```

`conwayProducedValue` is a Conway-era function that predates `DirectDeposits` and does not account for it. `dijkstraProducedValue` adds only sub-transaction produced values on top, without adding `directDeposits` from the top-level body.

For sub-transactions, `dijkstraSubTxProducedValue` is:

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody <> txBody ^. treasuryDonationTxBodyL)
    <> burnedMultiAssets txBody
```

Again, `directDeposits` are absent.

Despite this, `applyDirectDeposits` is unconditionally called in both the `ENTITIES` and `SUBENTITIES` rules after the value conservation check has already passed:

```haskell
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

The `applyDirectDeposits` function adds the specified coin amounts to account balances with no further validation that the transaction provided those funds.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker can craft a transaction where:

- UTxO inputs = X ADA
- UTxO outputs = X − fee ADA
- fee = fee ADA
- `directDeposits` = Y ADA (arbitrarily large)

The value conservation check evaluates: `X = (X − fee) + fee` → passes.

`applyDirectDeposits` then adds Y ADA to the targeted account balance. The total ADA in the system increases by Y ADA, violating the global preservation-of-value invariant. The attacker can subsequently withdraw this ADA via a standard withdrawal transaction.

---

### Likelihood Explanation

**High.** The entry path requires only a valid Dijkstra-era transaction with a non-empty `directDeposits` field targeting a registered account. No privileged access, governance majority, or special role is required. Any transaction sender can exploit this.

---

### Recommendation

Include `directDeposits` in the produced-value calculation for both top-level and sub-transaction bodies. Specifically:

1. In `dijkstraProducedValue`, add the sum of all `directDeposits` from the top-level transaction body to the produced value.
2. In `dijkstraSubTxProducedValue`, add the sum of all `directDeposits` from the sub-transaction body to the produced value.

The fix mirrors how `treasuryDonationTxBodyL` is already handled — it is an ADA outflow that is included in the produced value via `inject`.

---

### Proof of Concept

**Step 1.** Register a stake credential and obtain an `AccountAddress` `acct`.

**Step 2.** Construct a Dijkstra top-level transaction:
- Spend a UTxO worth 2 ADA
- Produce one output of 1.8 ADA
- Set `fee = 0.2 ADA`
- Set `directDeposits = { acct: 1_000_000 ADA }`

**Step 3.** Submit the transaction. The value conservation check evaluates:
- consumed = 2 ADA
- produced = 1.8 + 0.2 = 2 ADA ✓ (`directDeposits` absent from produced)

**Step 4.** `applyDirectDeposits` credits 1,000,000 ADA to `acct`.

**Step 5.** Submit a withdrawal transaction draining `acct`. The attacker receives 1,000,000 ADA that was never in the UTxO set.
