### Title
Uninitialized MPC Contract Susceptible to Front-Run Takeover via Unrestricted `init()` - (File: crates/contract/src/lib.rs)

### Summary
The `MpcContract::init()` function lacks any caller restriction (`#[private]` is absent), and all official deployment procedures deploy the contract in a separate transaction from initialization (`without-init-call`). This creates a block-level window during which any unprivileged NEAR account can race the deployer and call `init()` with attacker-controlled participants, seizing full governance and signing authority over the MPC network before the legitimate deployer can initialize it.

### Finding Description
The `init()` function is the sole public entry point for bootstrapping the MPC contract from scratch. It is decorated with `#[init]` (which prevents re-initialization once state exists) but carries **no** `#[private]` guard and no caller identity check:

```rust
// crates/contract/src/lib.rs:1924-1926
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

By contrast, `init_running()` — the function used to transfer an already-running network to a new contract — is correctly protected:

```rust
// crates/contract/src/lib.rs:1976-1979
#[private]
#[init]
#[handle_result]
pub fn init_running(...)
``` [2](#0-1) 

Every official deployment procedure — localnet, testnet, and the TEE production cluster — explicitly deploys the contract **without** an init call, leaving the contract in an uninitialized state for at least one block:

- `docs/localnet/localnet.md` line 131: `near contract deploy mpc-contract.test.near use-file "$MPC_CONTRACT_PATH" without-init-call ...`
- `docs/testnet/setup-guide-for-testnet-with-tee-support.md` line 118: `near contract deploy $MPC_CONTRACT_ACCOUNT use-file "$MPC_CONTRACT_PATH" without-init-call ...`
- `localnet/tee/scripts/rust-launcher/deploy-tee-cluster.sh` line 1143: `near contract deploy "$MPC_CONTRACT_ACCOUNT" use-file "$MPC_CONTRACT_PATH" without-init-call ...` [3](#0-2) [4](#0-3) [5](#0-4) 

The `init()` call is then issued as a **separate, subsequent transaction** (e.g., `scripts/launch-localnet.sh` line 216, `localnet/tee/scripts/rust-launcher/deploy-tee-cluster.sh` line 1176-1180), creating a multi-block window between deployment and initialization. [6](#0-5) 

### Impact Explanation
An attacker who wins the race calls `init()` with a `ThresholdParameters` struct containing only their own account(s) and a threshold of 1. The contract transitions immediately to `ProtocolContractState::Running` with the attacker as the sole participant: [7](#0-6) 

From that point:
- The attacker is the only recognized participant; all legitimate nodes are excluded.
- The attacker controls all `vote_add_domains`, `vote_pk`, `vote_reshared`, and `respond*` calls.
- The attacker can issue threshold signatures for any foreign-chain payload submitted to the contract, enabling unauthorized transaction execution on Bitcoin, Ethereum, Solana, etc.
- The legitimate deployer's subsequent `init()` call will revert because state already exists (`#[init]` prevents re-initialization), leaving the attacker in permanent control.

This matches the **Critical** impact class: *Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization.*

### Likelihood Explanation
NEAR's mempool is public. An attacker monitoring the chain for a new deployment of the well-known MPC contract WASM can detect the deployment transaction in the same block it lands and submit their own `init()` transaction targeting the newly deployed account. Because the deployer's `init()` is a separate transaction (confirmed by all deployment scripts), the attacker has at least one full block — and in practice several blocks given the multi-step manual procedures — to win the race. No privileged access, leaked key, or TEE bypass is required; a standard NEAR account and a few hundred yoctoNEAR in gas suffice.

### Recommendation
**Option A (preferred):** Add `#[private]` to `init()`, matching the protection already applied to `init_running()`. Since all deployment scripts sign as the contract account itself (`sign-as mpc-contract.test.near` / `sign-as "$MPC_CONTRACT_ACCOUNT"`), the predecessor equals the current account and `#[private]` will still permit the legitimate call while blocking any external account.

```rust
#[private]   // add this
#[handle_result]
#[init]
pub fn init(parameters: dtos::ThresholdParameters, ...) -> Result<Self, Error> {
```

**Option B:** Deploy and initialize atomically in a single NEAR batch transaction (deploy action + function-call action in the same transaction), eliminating the uninitialized window entirely.

### Proof of Concept
1. Attacker monitors NEAR for a `DeployContract` action targeting the MPC contract account.
2. In the next block, attacker submits:
   ```
   near contract call-function as-transaction <mpc-contract-account> init \
     json-args '{"parameters": {"threshold": 1, "participants": {"next_id": 1, "participants": [["attacker.near", 0, {"tls_public_key": "...", "url": "..."}]]}}}' \
     prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
     sign-as attacker.near network-config mainnet sign-with-keychain send
   ```
3. If included before the deployer's `init()`, the contract is now in `Running` state with `attacker.near` as the sole participant at threshold 1.
4. The deployer's `init()` reverts with "Contract already initialized."
5. Attacker calls `vote_add_domains` to register signing domains, then responds to any user `sign()` request with an attacker-generated signature, issuing unauthorized cross-chain transactions. [8](#0-7)

    #[init]
    pub fn init(
        parameters: dtos::ThresholdParameters,
        init_config: Option<dtos::InitConfig>,
    ) -> Result<Self, Error> {
        let parameters: ThresholdParameters = parameters.try_into_contract_type()?;
        // Log participant count and hash - full parameters exceed NEAR's 16KB log limit at ~100 participants
        let params_hash = env::sha256_array(borsh::to_vec(&parameters).unwrap());
        log!(
            "init: signer={}, num_participants={}, parameters_hash={:?}, init_config={:?}",
            env::signer_account_id(),
            parameters.participants().len(),
            params_hash,
            init_config,
        );
        parameters.validate()?;

        // TODO(#1087): Every participant must have a valid attestation, otherwise we risk
        // participants being immediately kicked out once contract transitions into running.
        let initial_participants = parameters.participants();
        let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);

        Ok(Self {
            protocol_state: ProtocolContractState::Running(RunningContractState::new(
                DomainRegistry::default(),
                Keyset::new(EpochId::new(0), Vec::new()),
                parameters,
                AddDomainsVotes::default(),
            )),
