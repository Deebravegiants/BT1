### Title
Unvalidated `ProposalInit.builder` Field Allows Malicious Proposer to Redirect All Block Fees to Arbitrary Address - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` validates many `ProposalInit` fields but never checks `builder`. That field is forwarded verbatim as `sequencer_address` in `BlockInfo`, which governs fee collection, the `get_sequencer_address` syscall, and the sequencer-address component of the block hash. A malicious-but-legitimate proposer can set `builder` to any address, redirect all block fees to an attacker-controlled wallet, corrupt every `get_sequencer_address` result in that block, and produce a wrong-sequencer-address block hash that every validator agrees on.

---

### Finding Description

`ProposalInit` carries a `builder` field (the address that builds/sequences the block): [1](#0-0) 

`is_proposal_init_valid` validates `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices, `starknet_version`, `version_constant_commitment`, `timestamp`, and `fee_proposal_fri`. It does **not** validate `builder`: [2](#0-1) 

After validation passes, `convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address`: [3](#0-2) 

That `BlockInfo` is passed to the batcher via `validate_block` in `initiate_validation`: [4](#0-3) 

The batcher executes all transactions with `sequencer_address = init.builder`, so every fee transfer and every `get_sequencer_address` syscall uses the proposer-supplied value. The `PartialBlockHash` the batcher returns also encodes this address; `proposal_commitment_from` wraps it into the final `ProposalCommitment`: [5](#0-4) 

Because both proposer and validator derive the commitment from the same (unvalidated) `builder`, the commitment check at line 244 passes: [6](#0-5) 

The only identity check in the consensus manager is on `proposer`, not `builder`: [7](#0-6) 

---

### Impact Explanation

| Effect | Mechanism |
|---|---|
| **Fee theft** | All transaction fees in the block are transferred to `init.builder`; a malicious proposer sets this to an attacker wallet |
| **Wrong block hash / state commitment** | `sequencer_address` is a component of `PartialBlockHashComponents`; every validator commits to the wrong value |
| **Wrong `get_sequencer_address` result** | Any contract calling this syscall during execution receives the attacker address, potentially breaking protocol-level logic |

This matches the allowed impact: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact"* and *"Wrong state, receipt, event … storage value … from blockifier/syscall/execution logic"*.

---

### Likelihood Explanation

Any validator is eligible to be the proposer for a given height/round. No special privilege beyond normal validator participation is required. In a network of N validators, a single malicious validator can exploit this on every round it is selected as proposer (~1/N of all blocks). The attack requires no external setup and leaves no on-chain trace distinguishing it from a legitimate proposal.

---

### Recommendation

Add a `builder` field to `ProposalInitValidation` (populated from the node's own configured sequencer address) and enforce an exact-match check inside `is_proposal_init_valid`, analogous to the existing `height` / `l1_da_mode` / `l2_gas_price_fri` checks:

```rust

### Proof of Concept

1. Malicious validator `V` is selected as proposer for height `H`, round `R`.
2. `V` calls `initiate_build` and overrides `builder: ATTACKER_ADDR` in the constructed `ProposalInit` before broadcasting.
3. Honest validators receive the proposal and call `validate_proposal` → `is_proposal_init_valid`. All checked fields pass; `builder` is never inspected.
4. `initiate_validation` calls `convert_to_sn_api_block_info(init)`, producing `BlockInfo { sequencer_address: ATTACKER_ADDR, … }`, and forwards it to the batcher via `validate_block`.
5. The batcher executes every transaction with `sequencer_address = ATTACKER_ADDR`; all fee transfers go to `ATTACKER_ADDR`.
6. `finish_proposal` returns a `PartialBlockHash` that encodes `ATTACKER_ADDR`; `proposal_commitment_from` wraps it into `batcher_block_commitment`.
7. `V` also computed the same commitment (same `builder`), so `built_block == received_fin.proposal_commitment` passes.
8. The proposal is inserted into `valid_proposals` and consensus votes on it; the block is committed with wrong fees and wrong sequencer address in the block hash.

#[instrument(level = "warn", skip_all, fields(?proposal_init_validation, ?init_proposed))]
async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is in the future: now={}, block_timestamp_window_seconds={}, \
                 proposed={}",
                now,
                proposal_init_validation.block_timestamp_window_seconds,
                init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.starknet_version != proposal_init_validation.starknet_version {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "starknet_version mismatch: expected={:?}, proposed={:?}",
                proposal_init_validation.starknet_version, init_proposed.starknet_version
            ),
        ));
    }
    // `version_constant_commitment` is proposer-supplied (network-derived). It is not yet a real
    // commitment (see `expected_version_constant_commitment`): the only valid value is the
    // sentinel, so reject anything else. Enforcing the same value the proposer emits keeps the two
    // sides in lockstep, so a real value cannot ship on one side without the other.
    let expected_commitment = expected_version_constant_commitment();
    if init_proposed.version_constant_commitment != expected_commitment {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "version_constant_commitment mismatch: expected={expected_commitment:?}, \
                 proposed={:?}",
                init_proposed.version_constant_commitment
            ),
        ));
    }
    if !(init_proposed.height == proposal_init_validation.height
        && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
        && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
    {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            "ProposalInit validation failed".to_string(),
        ));
    }
