### Title
Quorum Check Uses Total Proposed-Participant Count Instead of Governance Threshold, Allowing a Single Byzantine Node to Permanently Block Resharing — (`File: crates/contract/src/state/running.rs`)

### Summary

`process_new_parameters_proposal` compares the raw count of **all proposed participants** (`new_num_participants`) against the vote tally to decide whether to trigger a resharing transition. The governance threshold (`self.parameters.threshold()`) — the intended quorum — is never used for this check. A single participant who is in the proposed set but refuses to cast a vote can therefore block every `vote_new_parameters`-driven resharing indefinitely, including participant-set changes and key-refresh proposals.

### Finding Description

In `crates/contract/src/state/running.rs`, the function `process_new_parameters_proposal` collects votes and decides whether to transition to resharing:

```rust
// finally, vote.
let n_votes = self.parameters_votes.vote(proposal, candidate);
Ok(new_num_participants == n_votes)   // line 207
```

`new_num_participants` is the count of participants in the **proposed** (incoming) set:

```rust
let new_num_participants =
    u64::try_from(proposal.participants().len()).expect("participant count fits in u64");
```

The transition fires only when `n_votes` equals `new_num_participants`, i.e., every single proposed participant has voted. The governance threshold — `self.parameters.threshold()` — is computed and validated earlier in the same function for other purposes but is **never used** as the quorum gate.

The public ABI documentation (the `vote_new_parameters` doc-comment captured in the ABI snapshot) explicitly states:

> "If a **threshold** number of votes are reached on the exact same proposal, this will transition the contract into the Resharing state."

The code contradicts this: it requires **unanimous** participation from the proposed set, not a threshold quorum. The same pattern appears in `vote_add_domains`:

```rust
if self.parameters.participants().len() as u64 == n_votes {
```

which requires every current participant to vote before a new domain can be added.

The mismatch is structurally identical to the Zap `zapOut` bug: one numeric unit (total participant count) is substituted for a different numeric unit (governance threshold) at the decision point, causing the condition to be far stricter than the protocol intends.

### Impact Explanation

**Medium — participant-state and contract execution-flow manipulation that breaks production safety/accounting invariants.**

Because the transition requires `n_votes == new_num_participants` (unanimity), a single participant who is included in the proposed set can permanently stall any `vote_new_parameters`-driven resharing by simply not calling the method. Concretely:

- **Participant-set changes are blocked.** Adding or removing a participant requires resharing via `vote_new_parameters`. If one participant in the proposed set withholds their vote, the transition never fires regardless of how many others have voted.
- **Key-refresh proposals are blocked.** A participant can prevent the network from rotating keys or updating per-domain reconstruction thresholds.
- **The governance threshold invariant is violated.** The protocol guarantees that `threshold` votes are sufficient to make governance decisions. The code silently raises that bar to unanimity, breaking the documented safety property.

`verify_tee` can trigger resharing via `transition_to_resharing_no_checks` and bypasses this vote path, so TEE-expiry-driven resharing is unaffected. However, all operator-initiated participant-set changes go exclusively through `vote_new_parameters` → `process_new_parameters_proposal`, and those are fully blocked.

### Likelihood Explanation

**Medium.** The attacker need only be a single participant included in the proposed set — a position strictly below the signing threshold. No collusion, no key material, and no privileged access are required. The attack is passive (simply not calling `vote_new_parameters`) and leaves no on-chain trace. Any participant who wishes to prevent their own removal from the set, or to block a competitor's admission, has a direct incentive to exploit this.

### Recommendation

Replace the unanimity check with the governance threshold:

```rust
// Before (wrong unit — total proposed participants):
Ok(new_num_participants == n_votes)

// After (correct unit — governance quorum):
Ok(self.parameters.threshold().value() <= n_votes)
```

`n_votes` should count only votes cast by **current** participants (filtered through `self.parameters.participants()`), consistent with how the `VoterPending` guard counts them at line 199. New (incoming) participants should be allowed to vote only after the threshold of current participants has been reached, which the existing `VoterPending` guard already enforces.

Apply the same fix to `vote_add_domains`:

```rust
// Before:
if self.parameters.participants().len() as u64 == n_votes {
// After:
if self.parameters.threshold().value() <= n_votes {
```

### Proof of Concept

**Setup:** 5 participants, governance threshold = 3. Participant E is in the proposed set but refuses to vote.

1. Participants A, B, C, D each call `vote_new_parameters` with the same proposal.
2. After each call, `n_votes` increments: 1 → 2 → 3 → 4.
3. `new_num_participants` = 5 (A, B, C, D, E).
4. The condition `5 == 4` is false; no resharing transition occurs.
5. E never calls `vote_new_parameters`.
6. The contract remains in `Running` state indefinitely; the participant-set change is permanently blocked despite 4 out of 5 (and 4 out of 3 threshold) votes having been cast.
