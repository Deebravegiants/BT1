### Title
Single Malicious Participant Can Deliver Forged CKD Response for `AppPublicKey` Variant, Bypassing On-Chain Output Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` enforces a cryptographic output check (`ckd_output_check`) only when the requester uses the `AppPublicKeyPV` variant. When the standard `AppPublicKey` variant is used, the check is silently skipped. A single malicious attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary fabricated `CKDResponse`, resolve the pending request, and deliver a garbage or attacker-chosen key to every queued requester before the honest coordinator submits the real response.

---

### Finding Description

In `crates/contract/src/lib.rs`, `respond_ckd` branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty: no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk, app_id), pk)`, which cryptographically proves the response is a valid encryption of the threshold-derived key. [2](#0-1) 

For `AppPublicKey`, the branch body is empty — no pairing check, no point validity check, no relationship between `big_y`/`big_c` and the MPC network's public key is enforced on-chain.

The only gate before `resolve_yields_for` is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a single attested participant — not a threshold quorum. [3](#0-2) 

`AppPublicKey` is the default, non-publicly-verifiable variant and is the one shown in all documentation and CLI examples. [4](#0-3) 

Once `resolve_yields_for` is called, all pending yield slots for that request are drained. A subsequent honest `respond_ckd` from the real coordinator finds no pending yields and is a no-op.

---

### Impact Explanation

**Impact: Medium — request-lifecycle and contract execution-flow manipulation breaking production safety invariants.**

A single malicious attested participant can:

1. Monitor the contract for `request_app_private_key` calls using `AppPublicKey`.
2. Immediately call `respond_ckd` with a fabricated `CKDResponse` (e.g., `big_y = identity`, `big_c = k·G1` for attacker-chosen `k`).
3. The contract accepts the call, resolves all queued yield slots, and delivers the garbage response to every requester.
4. The honest coordinator's subsequent `respond_ckd` finds no pending yields and is silently ignored.

Requesters who perform off-chain BLS verification will detect the forgery (the fabricated `sig = big_c − a·big_y` will not satisfy `e(H(pk, app_id), pk) = e(sig, G2)`). However:

- Smart contracts or integrations that do not perform off-chain verification will accept the garbage key and use it to control assets, resulting in **permanent loss of access** (funds frozen under an unknown key).
- Even for verifying requesters, the request slot is permanently consumed. The requester must submit a new `request_app_private_key` transaction and pay again, while the attacker can repeat the attack indefinitely.

This breaks the production invariant that a CKD response delivered by the contract is a valid threshold-derived key.

---

### Likelihood Explanation

- Requires only **one** malicious attested participant, well below the signing threshold.
- The attacker participates in the off-chain CKD protocol and knows exactly when the coordinator is about to submit. They can race the coordinator's `respond_ckd` submission.
- `AppPublicKey` is the standard variant; most production integrations use it.
- No special privileges, leaked keys, or network-level access are required — only a valid TEE attestation and participant status.

---

### Recommendation

Apply `ckd_output_check` unconditionally for both `AppPublicKey` and `AppPublicKeyPV` variants. For `AppPublicKey`, the check can be performed using only `pk1` (the G1 component), since the pairing equation `e(big_c, G2) = e(big_y, pk1) · e(H(pk, app_id), pk)` does not require the G2 component `pk2`. Alternatively, require `AppPublicKeyPV` for all on-chain requests so the contract can always verify the response before resolving yields.

---

### Proof of Concept

**Setup**: Contract in `Running` state with a BLS12381 CKD domain. Attacker is one of the `n` attested participants (below threshold `t`).

**Step 1 — Victim submits CKD request:**
```
request_app_private_key({
  derivation_path: "mykey",
  app_public_key: AppPublicKey(victim_pk1),   // standard variant
  domain_id: 2
})
```
Contract stores a pending yield for the victim.

**Step 2 — Attacker front-runs with fabricated response:**
```
respond_ckd(
  request = { app_public_key: AppPublicKey(victim_pk1), app_id: ..., domain_id: 2 },
  response = { big_y: identity_G1, big_c: k·G1 }   // attacker chooses k
)
```
The `AppPublicKey` branch is empty — no check fires. `resolve_yields_for` drains the victim's yield slot and delivers `{ big_y: 0, big_c: k·G1 }`.

**Step 3 — Victim receives garbage key:**
Victim computes `sig = k·G1 − a·0 = k·G1`. BLS verification fails: `e(H(pk, app_id), pk) ≠ e(k·G1, G2)` unless `k = msk` (unknown to attacker). Victim has no valid key.

**Step 4 — Honest coordinator's `respond_ckd` is a no-op:**
No pending yields remain; the real response is silently discarded. [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
