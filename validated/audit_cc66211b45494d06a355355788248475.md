### Title
Execute-Phase Sequencer Balance Write Silently Overwritten by `add_fee_to_sequencer_balance` in Concurrency Mode — (`crates/blockifier/src/concurrency/fee_utils.rs`)

---

### Summary

When a transaction's execute phase writes to the sequencer balance storage keys (e.g., by calling `transfer(sequencer_address, amount)` on the fee token contract), the concurrent commit path in `add_fee_to_sequencer_balance` unconditionally overwrites that write in `execution_output.state_diff` using a stale balance read from `tx_versioned_state`. The execute-phase transfer amount is silently dropped, producing a post-state root that diverges from sequential execution for the same accepted input.

---

### Finding Description

The concurrent fee transfer is split across two phases:

**Phase 1 — `concurrency_execute_fee_transfer`** (execute time): [1](#0-0) 

A nested `transfer_state` is created. The fee transfer runs inside it with a zeroed sequencer balance. Before committing, the sequencer balance keys are removed from `transfer_state.writes`:

```
storage_writes.remove(&(fee_address, sequencer_balance_key_low));
storage_writes.remove(&(fee_address, sequencer_balance_key_high));
transfer_state.commit();
```

This correctly prevents the fee transfer's write from polluting the outer state. However, if the user's **execute phase** already wrote to those same keys (e.g., via a direct ERC-20 `transfer` to the sequencer), that write survives in the outer `state.writes` and is reflected in `execution_output.state_diff`. The existing test confirms this: [2](#0-1) 

After `execute_raw`, `state_diff.storage[(fee_token_address, sequencer_balance_key_low)]` = `PREV_BALANCE + TRANSFER_AMOUNT`.

**Phase 2 — `complete_fee_transfer_flow` / `add_fee_to_sequencer_balance`** (commit time): [3](#0-2) 

`complete_fee_transfer_flow` reads the sequencer balance from `tx_versioned_state`: [4](#0-3) 

`tx_versioned_state` is pinned at `tx_index` and reflects only previously committed transactions — it does **not** include the current transaction's execute-phase write. So `sequencer_balance` = `PREV_BALANCE`.

`add_fee_to_sequencer_balance` then computes `new_value_low = PREV_BALANCE + FEE` and unconditionally inserts it into `state_diff`: [5](#0-4) 

This `HashMap::insert` **overwrites** the execute-phase write of `PREV_BALANCE + TRANSFER_AMOUNT` with `PREV_BALANCE + FEE`. The `TRANSFER_AMOUNT` is permanently lost from the committed state diff.

The comment at line 140–142 reveals the broken assumption: [6](#0-5) 

> "We assume that a non-sequencer sender cannot reduce the sequencer's balance—only increases are possible."

This assumption is false when the execute phase itself increases the sequencer balance.

---

### Impact Explanation

| Mode | Final sequencer balance |
|---|---|
| Sequential | `PREV_BALANCE + TRANSFER_AMOUNT + FEE` |
| Concurrent | `PREV_BALANCE + FEE` |

The `TRANSFER_AMOUNT` tokens are effectively burned: deducted from the sender's account (execute phase write is correct) but never credited to the sequencer (overwritten at commit). The committed state root diverges from sequential execution for the same accepted transaction, violating the correctness guarantee of the concurrency engine.

---

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a standard invoke transaction whose calldata calls `transfer(sequencer_address, N)` on the fee token contract. No special privileges, no proxy contract, no deployment required — a plain ERC-20 transfer to the sequencer address suffices. The bug activates whenever the sequencer runs in concurrency mode (the production path).

---

### Recommendation

In `add_fee_to_sequencer_balance`, before inserting into `state_diff`, check whether the execute phase already wrote to the sequencer balance key. If so, use that already-written value as the base for the fee addition rather than the stale `sequencer_balance` read from `tx_versioned_state`. Concretely:

```rust
// Instead of:
let base_low = sequencer_balance_low_as_u128;

// Use:
let base_low = state_diff.storage
    .get(&(fee_token_address, sequencer_balance_key_low))
    .and_then(|f| f.to_u128())
    .unwrap_or(sequencer_balance_low_as_u128);
```

Apply the same logic for the high word. This ensures the fee is added on top of any execute-phase sequencer balance update, matching sequential execution semantics.

---

### Proof of Concept

1. Fund account with sufficient STRK.
2. Submit an invoke transaction with calldata: `call fee_token_contract.transfer(sequencer_address, 1000)`.
3. Execute the block in concurrency mode.
4. Assert: `state.get_storage_at(fee_token_address, sequencer_balance_key_low)` == `PREV_BALANCE + FEE` (not `PREV_BALANCE + 1000 + FEE`).
5. Execute the same transaction sequentially.
6. Assert: `state.get_storage_at(fee_token_address, sequencer_balance_key_low)` == `PREV_BALANCE + 1000 + FEE`.
7. The two post-state roots differ for the same accepted input — confirmed vulnerability.
