### Title
Caller Deposit Permanently Locked in Wallet Contract on Intermediate Callback Failure - (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary
The `WalletContract` tracks an external caller's attached deposit via `CallerDeposit` and is designed to refund it on failure. However, two intermediate callbacks — `address_check_callback` and `nep_141_storage_balance_callback` — have multiple early-return failure paths that return an error response without refunding the `caller_deposit`. When these callbacks fail, the caller's NEAR tokens are permanently locked inside the wallet contract with no recovery path.

### Finding Description
`rlp_execute` accepts an attached deposit from an external caller and records it in a `CallerDeposit` struct, which is threaded through the promise chain so it can be refunded on failure. The terminal callback `rlp_execute_callback` correctly handles this:

```rust
// runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs, lines 296-305
match env::promise_result(0) {
    PromiseResult::Failed => {
        if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
            let refund_promise = env::promise_batch_create(&account_id);
            env::promise_batch_action_transfer(refund_promise, ...);
        }
        ...
    }
```

But the two intermediate callbacks that can short-circuit the chain before reaching `rlp_execute_callback` do not refund the deposit:

**`address_check_callback` (lines 134–192)** — three early-return paths with no refund:
- `PromiseResult::Failed` (lines 142–147): registrar call fails
- Unexpected JSON response (lines 151–157): registrar returns unparseable data
- Invalid target, non-self signer (lines 167–173): relayer sent wrong target

**`nep_141_storage_balance_callback` (lines 194–273)** — three early-return paths with no refund:
- `PromiseResult::Failed` (lines 204–209): `storage_balance_of` call fails
- Unexpected JSON response (lines 213–219): token contract returns unparseable data
- Unexpected action type (lines 245–253): action is not a `FunctionCall`

In all six paths, the function returns `PromiseOrValue::Value(ExecuteResponse { success: false, ... })` without issuing a refund transfer to `caller_deposit.account_id`. The deposit, already transferred from the caller to the wallet contract by the NEAR runtime when `rlp_execute` was called, remains in the wallet contract's balance permanently.

### Impact Explanation
The corrupted protocol value is the caller's account balance. When an external caller attaches a non-zero deposit to `rlp_execute` for an EOA base-token transfer (triggering the address-check path) or an ERC-20 transfer (triggering the NEP-141 storage-balance path), and the intermediate callback fails, the caller's balance is permanently reduced by `caller_deposit.yocto_near` with no on-chain recovery mechanism. The wallet contract's balance is correspondingly inflated. This is a direct, irreversible balance corruption for the caller.

### Likelihood Explanation
The address registrar contract or the NEP-141 token contract can fail for legitimate reasons: the contract panics, runs out of gas, or is temporarily unavailable. Any external account (relayer or direct caller) that attaches a deposit to `rlp_execute` for these transaction kinds is exposed. The `CallerDeposit` mechanism was explicitly designed to handle this case, confirming the refund was intended but omitted in the intermediate callbacks.

### Recommendation
In every early-return failure branch of `address_check_callback` and `nep_141_storage_balance_callback`, issue a refund transfer before returning, mirroring the pattern already used in `rlp_execute_callback`:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund_promise = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(
        refund_promise,
        NearToken::from_yoctonear(yocto_near.into()),
    );
}
```

This should be applied to all six early-return paths identified above.

### Proof of Concept

1. External caller calls `rlp_execute` on a wallet contract with `attached_deposit = 5 NEAR`, targeting an eth-implicit address (triggering the address-check path). `CallerDeposit { account_id: caller, yocto_near: 5e24 }` is created. [1](#0-0) 

2. `inner_rlp_execute` schedules `address_registrar.lookup(address).then(address_check_callback(..., caller_deposit))`. [2](#0-1) 

3. The address registrar call fails (e.g., contract panics). `address_check_callback` is invoked with `PromiseResult::Failed`. [3](#0-2) 

4. The callback returns early with an error response. `caller_deposit` is never used; no refund transfer is issued. The 5 NEAR remains in the wallet contract's balance.

5. The caller's balance is permanently reduced by 5 NEAR. The NEAR runtime's deposit-refund mechanism only fires when a receipt *fails* (panics); a contract that returns a value successfully (even an error response) does not trigger an automatic deposit refund. [4](#0-3) 

The same scenario applies to `nep_141_storage_balance_callback` when `storage_balance_of` fails on a token contract during an ERC-20 transfer. [5](#0-4)

```markdown
# Refunds

When execution of a receipt fails or there is some unused amount of prepaid gas left after a function call, the Runtime generates refund receipts.

The are 2 types of refunds:

- Refunds for the failed receipt for attached deposits. Let's call them **deposit refunds**.
- Refunds for the unused gas and fees. Let's call them **gas refunds**.

Refund receipts are identified by having `predecessor_id == "system"`. They are also special because they don't cost any gas to generate or execute. As a result, they also do not contribute to the block gas limit.

If the execution of a refund fails, the refund amount is burnt.
The refund receipt is an `ActionReceipt` that consists of a single action `Transfer` with the `deposit` amount of the refund.
```
