Audit Report

## Title
Missing Chainlink Staleness Validation Enables Deposit-at-Inflated-Price Share Dilution — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData` but silently discards `updatedAt`, `roundId`, and `answeredInRound`, accepting any price regardless of age. When a Chainlink LST feed is stale at a price above the true current value (e.g., after a slashing event the feed has not yet reflected), an attacker can deposit at the inflated stale price, receive excess rsETH, wait for the feed to correct and `updateRSETHPrice()` to be called, then withdraw — extracting ETH value from existing rsETH holders.

## Finding Description

**Root cause — `ChainlinkPriceOracle.getAssetPrice` (L52):**

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All staleness fields (`updatedAt`, `roundId`, `answeredInRound`) are discarded. No `block.timestamp - updatedAt > heartbeat` guard exists.

**Deposit minting uses this price directly (`LRTDepositPool.sol` L520):**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If `getAssetPrice` returns a stale-high value, the attacker receives more rsETH than their deposit is worth.

**`updateRSETHPrice()` recomputes TVL at the corrected price (`LRTOracle.sol` L339):**

```solidity
uint256 assetER = getAssetPrice(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```

Once the Chainlink feed updates, `updateRSETHPrice()` (public, callable by anyone) recomputes TVL at the true lower price, diluting `rsETHPrice` across all holders.

**Withdrawal records `expectedAssetAmount` at the post-correction price (`LRTWithdrawalManager.sol` L593):**

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

**`_calculatePayoutAmount` takes `min(expectedAssetAmount, currentReturn)` (`LRTWithdrawalManager.sol` L833–834):**

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

Since the attacker initiates withdrawal *after* `updateRSETHPrice()` has already set the lower `rsETHPrice`, both `expectedAssetAmount` and `currentReturn` reflect the same post-correction prices, so the minimum equals the inflated amount — the attacker receives more LST than they deposited.

**Existing guards are insufficient:**

The `pricePercentageLimit` downside-pause in `LRTOracle._updateRsETHPrice` (L270–282) only triggers if the price drop exceeds the configured threshold. A drop within the limit (e.g., 0.9% < 1% limit) allows the attack to proceed without triggering a pause. If `pricePercentageLimit` is 0, there is no protection at all.

## Impact Explanation

**Critical — Direct theft of existing user funds.**

Concrete example (10% stale premium, 100 LST TVL):

| Step | Detail |
|---|---|
| Attacker deposits 10 LST at stale price 1.10 | Receives 11 rsETH (fair share = 10) |
| Feed corrects; `updateRSETHPrice()` called | rsETHPrice drops from 1.00 → 0.9910 |
| Attacker withdraws 11 rsETH | Receives ≈10.90 LST |
| **Attacker profit** | **+0.90 LST (0.90 ETH)** |
| **Existing 100 rsETH holders** | **Lose 0.90 ETH collectively** |

This is direct, irreversible extraction of ETH value from existing rsETH holders through share dilution. The loss scales with the stale price premium and the attacker's deposit size.

## Likelihood Explanation

- Chainlink LST feeds (e.g., stETH/ETH) have heartbeat intervals up to 24 hours and deviation thresholds of 0.5–1%. During this window, the on-chain price can lag the true price.
- A stale-high price occurs when the LST loses value (slashing event, depeg) and the feed has not yet updated. Slashing events on major LST protocols have occurred historically.
- `updateRSETHPrice()` is public and permissionless — the attacker can call it themselves after the Chainlink feed updates, without relying on any third party.
- `unlockQueue()` is called by operators as part of normal protocol operations; the attacker only needs to wait for the routine unlock cycle.
- The attacker requires no privileged access, no flash loan, and no front-running — only the ability to observe that the feed is stale and deposit before it updates.
- The `pricePercentageLimit` guard only protects against drops exceeding the configured threshold; drops within the limit (common for small slashing events) are unprotected.

## Recommendation

Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
require(price > 0, "Non-positive price");
```

`MAX_STALENESS` should be configured per-feed based on the Chainlink heartbeat (e.g., 3600 s for 1-hour feeds, 86 400 s for 24-hour feeds). Consider storing this per-asset alongside `assetPriceFeed`.

## Proof of Concept

**Foundry fork test outline (fork mainnet near stETH/ETH feed heartbeat boundary):**

1. Deploy/fork with `stETH/ETH` Chainlink feed near its 24-hour heartbeat (price hasn't updated for ~23h, stETH has lost value on-chain).
2. Record `stalePriceReported = oracle.getAssetPrice(stETH)` — confirm it is above true market price.
3. Attacker calls `depositPool.depositAsset(stETH, 10e18, 0, "")` — receives `11e18` rsETH.
4. Warp time past the heartbeat; Chainlink feed updates to true lower price.
5. Call `oracle.updateRSETHPrice()` — `rsETHPrice` drops.
6. Attacker calls `withdrawalManager.initiateWithdrawal(stETH, 11e18, "")` — `expectedAssetAmount ≈ 10.9e18` recorded.
7. Warp `withdrawalDelayBlocks`; operator calls `unlockQueue(stETH, ...)`.
8. Attacker calls `completeWithdrawal(stETH, "")` — receives ≈10.9 stETH.

**Invariant assertion:**

```solidity
// ETH redeemed must not exceed ETH deposited at true price at deposit time
assertLe(
    ethRedeemed,
    depositAmountInLST * truePriceAtDepositTime / 1e18,
    "Invariant violated: attacker extracted more than deposited"
);
```

This invariant breaks on unmodified production code under the described conditions.
