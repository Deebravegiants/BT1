Audit Report

## Title
Uninitialized `tokenFeeBps` Defaults to Zero, Allowing Fee-Free Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary
`addSupportedToken` registers a new token but never writes to `tokenFeeBps[token]`, leaving it at the Solidity mapping default of `0`. Any depositor can call `deposit(token, amount, referralId)` immediately after a token is listed and receive the full wrsETH equivalent of their deposit with zero fee deducted, diverting fee revenue that should accrue to the protocol treasury.

## Finding Description
`addSupportedToken` (L637–656) sets `supportedTokenOracle[token]`, `tokenBridge[token]`, and pushes to `supportedTokenList`, but never assigns `tokenFeeBps[token]`:

```solidity
// L651-655
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] is never written — defaults to 0
emit AddSupportedToken(token, oracle, bridge);
```

`viewSwapRsETHAmountAndFee(uint256, address)` (L326–347) reads directly from that mapping:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token]; // 0 for any new token
fee = amount * feeBpsForToken / 10_000;       // always 0
uint256 amountAfterFee = amount - fee;        // equals full amount
```

`deposit(address, uint256, string)` (L284–305) then transfers the full wrsETH equivalent to the caller and accumulates `feeEarnedInToken[token] += 0`. The only remediation path is a separate `setTokenFeeBps` call (L583–594) gated behind `DEFAULT_ADMIN_ROLE`, which is a different role from `TIMELOCK_ROLE` used by `addSupportedToken`. There is no atomic mechanism enforcing fee initialization in the same transaction as token listing, and no guard in `deposit` that rejects a zero-fee configuration.

## Impact Explanation
The protocol's fee on token deposits is its sole compensation for providing the swap service. When `tokenFeeBps[token] == 0`, every deposit transfers the full wrsETH amount to the depositor instead of deducting the intended fee and retaining it in `feeEarnedInToken[token]`. The wrsETH that should have been held as protocol fee revenue is instead paid out to depositors. This constitutes **High — Theft of unclaimed yield**: protocol fee revenue is diverted to unprivileged depositors for the entire duration of the zero-fee window.

## Likelihood Explanation
Adding new supported tokens is a routine protocol operation. The `AddSupportedToken` event is publicly observable on-chain, and the `deposit` function requires no special privilege. A monitoring bot or any depositor can detect the event and immediately begin depositing large amounts of the newly listed token at zero cost. The window persists until the admin separately calls `setTokenFeeBps`, which requires a different role (`DEFAULT_ADMIN_ROLE` vs. `TIMELOCK_ROLE`), making the gap non-trivially short in any realistic deployment sequence. The attack is repeatable with no per-transaction cost beyond gas.

## Recommendation
Pass the fee basis points as a parameter to `addSupportedToken` and store it atomically:

```solidity
function addSupportedToken(
    address token,
    address oracle,
    address bridge,
    uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    // ... existing checks ...
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    tokenFeeBps[token] = _feeBps; // initialize atomically
    emit AddSupportedToken(token, oracle, bridge);
}
```

This eliminates the exploitable window entirely by ensuring no deposit can occur before a fee is set.

## Proof of Concept
1. Admin calls `addSupportedToken(tokenX, oracle, bridge)` — `tokenFeeBps[tokenX]` is `0`.
2. Attacker observes the `AddSupportedToken` event and calls `deposit(tokenX, 1_000_000e18, "")`.
3. `viewSwapRsETHAmountAndFee(1_000_000e18, tokenX)` returns `fee = 1_000_000e18 * 0 / 10_000 = 0`.
4. Attacker receives `rsETHAmount = 1_000_000e18 * tokenToETHRate / rsETHToETHrate` wrsETH with zero fee deducted.
5. `feeEarnedInToken[tokenX]` remains `0`; the protocol treasury receives nothing.
6. Steps 2–5 repeat until admin calls `setTokenFeeBps(tokenX, N)`.

Foundry fork test plan: deploy `RSETHPool`, call `addSupportedToken`, assert `tokenFeeBps[token] == 0`, call `deposit` as an unprivileged address, assert `feeEarnedInToken[token] == 0` and that the depositor received the full wrsETH amount (no fee deducted), confirming the protocol earned zero fee revenue.
