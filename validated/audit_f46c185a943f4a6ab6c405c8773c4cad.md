### Title
Double-Counted Deposit Refund Across Sub-Transactions Enables ADA Creation — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, `getConsumedDijkstraValue` computes the total consumed value of a top-level transaction by summing the consumed value of the top-level body **and** each sub-transaction body, using the **same** pre-transaction `lookupStakingDeposit` snapshot for every sub-transaction. If two or more sub-transactions each include an `UnRegDepositTxCert` for the **same** staking credential, the deposit refund is counted once per sub-transaction in the consumed total, while the deposit pot is only debited once. This inflates the consumed side of the `consumed == produced` balance check, allowing the attacker to place the phantom ADA into transaction outputs and create lovelace from nothing.

The developers have already identified this gap: a test titled `"Multiple subtransactions cannot get the same refund"` exists in the testlib but is disabled with `xit` and carries the comment `error "TODO: predicate failure not yet implemented"`, confirming that no guard currently rejects such a transaction.

---

### Finding Description

**Root cause — `getConsumedDijkstraValue`**

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
``` [1](#0-0) 

The closure `lookupStakingDeposit` is a **fixed snapshot** of the pre-transaction ledger state. It is passed unchanged into every call of `getConsumedValue` for every sub-transaction. `getConsumedMaryValue` (called inside) invokes `getTotalRefundsTxBody`, which calls `conwayTotalRefundsTxCerts`, which calls `lookupStakingDeposit` to resolve the deposit amount for each `UnRegDepositTxCert` certificate. [2](#0-1) 

Because the same snapshot is used for every sub-transaction, if sub-tx₁ and sub-tx₂ both contain `UnRegDepositTxCert stakingCred D`, both calls to `lookupStakingDeposit stakingCred` return `Just D`. The `foldMap'` accumulates `D + D = 2D` into the consumed total, even though the deposit pot only holds `D` for that credential.

**The balance check**

The top-level `validateValueNotConservedUTxO` enforces `consumed == produced`: [3](#0-2) 

With consumed inflated by `(N−1)·D` for N sub-transactions claiming the same refund, the attacker can place `(N−1)·D` extra lovelace into outputs and satisfy the check. The deposit pot is only debited once (by the first sub-transaction that actually executes the deregistration), so the extra lovelace in outputs is unbacked.

**Developer acknowledgement**

The testlib contains a disabled test that directly describes this scenario:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
    ...
    subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = mkBasicTx mkBasicTxBody & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
    submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [4](#0-3) 

The `xit` prefix disables the test because `submitFailingTx` would report the transaction as **succeeding** (no predicate failure is raised), which is the wrong outcome. The `error "TODO: predicate failure not yet implemented"` confirms no rejection rule exists yet.

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker registers a staking credential (paying deposit `D`), then submits a single Dijkstra top-level transaction containing N sub-transactions each bearing `UnRegDepositTxCert stakingCred D`. The consumed-value calculation counts `N·D` in refunds; the deposit pot only releases `D`. The attacker places `(N−1)·D` extra lovelace into outputs. After the transaction is accepted, `(N−1)·D` lovelace exists in the UTxO with no corresponding debit from any pot, violating ADA preservation.

---

### Likelihood Explanation

**High.** The Dijkstra era is the next planned era. The attack requires only:
1. Registering a staking credential (permissionless, costs only the deposit itself).
2. Constructing a valid top-level transaction with multiple sub-transactions — a core Dijkstra feature available to any transaction author.
3. No privileged access, no governance majority, no key compromise.

The disabled test and the `TODO` comment confirm the gap is known and unguarded in the current codebase.

---

### Recommendation

Before the Dijkstra era activates, `getConsumedDijkstraValue` must not reuse the pre-transaction `lookupStakingDeposit` snapshot across sub-transactions. Two complementary fixes:

1. **Stateful refund accumulation**: Thread the `CertState` through sub-transaction consumed-value computation, updating it after each sub-transaction's certificates are processed, so that a credential deregistered by sub-tx₁ returns `Nothing` for sub-tx₂.

2. **Cross-sub-transaction deduplication**: Before computing the aggregate consumed value, verify that no staking or DRep credential appears in `UnRegDepositTxCert`/`UnRegDRepTxCert` certificates across more than one sub-transaction (or across a sub-transaction and the top-level body). Raise a dedicated predicate failure (the `TODO` slot) if a duplicate is found.

The `conwayDRepRefundsTxCerts` function already demonstrates the correct intra-transaction pattern (tracking `drepRegsInTx` to avoid double-counting within a single cert list); the same discipline must be extended across sub-transaction boundaries. [5](#0-4)

### Proof of Concept

```
Setup:
  stakingCred  ← freshKeyHash
  D            ← ppKeyDepositL (e.g. 2_000_000 lovelace)
  Register stakingCred, paying deposit D.

Craft top-level tx:
  subTx1:
    inputs  = {utxo_input_1}          -- funds subTx1's outputs
    certs   = [UnRegDepositTxCert stakingCred D]
    outputs = [addr1 ← value1 + D]    -- claims the refund

  subTx2:
    inputs  = {utxo_input_2}          -- funds subTx2's outputs
    certs   = [UnRegDepositTxCert stakingCred D]
    outputs = [addr2 ← value2 + D]    -- claims the same refund again

  topTx:
    subTransactions = [subTx1, subTx2]
    outputs = []   (balance satisfied by sub-tx outputs)

Balance check (getConsumedDijkstraValue):
  consumed = UTxO(utxo_input_1) + UTxO(utxo_input_2)
           + D   (refund counted for subTx1, lookupStakingDeposit stakingCred = Just D)
           + D   (refund counted for subTx2, same snapshot → Just D again)
           = value1 + value2 + 2D

  produced = (value1 + D) + (value2 + D)   -- outputs of both sub-txs
           = value1 + value2 + 2D  ✓

  → consumed == produced, balance check passes.

Execution:
  subTx1 executes: stakingCred deregistered, D returned from deposit pot → addr1.
  subTx2 executes: stakingCred already deregistered.
    → If SUBDELEG/SUBCERTS does NOT re-check registration before processing
      UnRegDepositTxCert, subTx2 also "returns" D to addr2 from the deposit pot,
      overdrawing it by D.
    → If SUBDELEG/SUBCERTS does reject the second deregistration, the overall
      transaction fails — but the xit test evidence indicates it currently succeeds.

Net result (if transaction succeeds): addr2 receives D lovelace that was never
deposited, creating D lovelace from nothing. Repeating with N sub-transactions
yields (N−1)·D of unbacked lovelace per transaction.
```

conwayDRepRefundsTxCerts lookupDRepDeposit = snd . F.foldl' go (Map.empty, Coin 0)
  where
    go accum@(!drepRegsInTx, !totalRefund) = \case
      RegDRepTxCert cred deposit _ ->
        -- Track registrations
        (Map.insert cred deposit drepRegsInTx, totalRefund)
      UnRegDRepTxCert cred _
        -- DRep previously registered in the same tx.
        | Just deposit <- Map.lookup cred drepRegsInTx ->
            (Map.delete cred drepRegsInTx, totalRefund <+> deposit)
        -- DRep previously registered in some other tx.
