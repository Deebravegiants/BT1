### Title
Unvalidated `l2bridge` Parameter Enables Gas-Bomb Block Stuffing — (`contracts/bridges/UnichainMessenger.sol`)

---

### Summary

`UnichainMessenger.sendETHToL1ViaBridge` is a public, permissionless function that accepts a fully caller-controlled `l2bridge` address and forwards all available gas to it via an external call. An attacker can supply a gas-bomb contract as `l2bridge`, consuming the entire Unichain block gas limit per transaction and delaying legitimate `BRIDGER_ROLE` bridge operations.

---

### Finding Description

`UnichainMessenger.sendETHToL1ViaBridge` has no access control and no allowlist validation on the `l2bridge` parameter: [1](#0-0) 

The only guard is a `msg.value == value` check. The call on line 26 forwards all available gas to the arbitrary `l2bridge` address — `DEFAULT_GAS_LIMIT = 200_000` is passed as a *parameter value* to `bridgeETHTo` (the L1 minimum gas limit hint), not as a gas stipend on the EVM call itself: [2](#0-1) 

Under EIP-150, the call forwards `(63/64) * gasleft()` to the callee. A gas-bomb contract implementing `bridgeETHTo` that loops until out-of-gas will consume nearly the entire block gas budget in a single transaction.

The pool contracts that legitimately invoke this messenger (`RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge`, `RSETHPoolV2.bridgeAssets`, `RSETHPoolV2ExternalBridge.bridgeAssetsViaNativeBridge`) are all gated by `onlyRole(BRIDGER_ROLE)` and pass a stored `l2Bridge` state variable: [3](#0-2) 

But the `UnichainMessenger` itself imposes no such restriction — any EOA or contract can call it directly with an arbitrary `l2bridge`.

---

### Impact Explanation

An attacker can repeatedly call `UnichainMessenger.sendETHToL1ViaBridge{value: 1 wei}(gasBombAddress, anyTarget, 1)` to saturate Unichain blocks. On OP Stack chains, gas prices are extremely low, making sustained block stuffing economically viable. This delays or prevents the `BRIDGER_ROLE` operator from executing `bridgeAssetsViaNativeBridge`, temporarily freezing the ETH bridging flow from Unichain to L1.

**Impact: Low — Block stuffing.**

---

### Likelihood Explanation

The attack requires only:
1. Deploying a contract with a gas-consuming `bridgeETHTo` implementation (trivial).
2. Calling the public `sendETHToL1ViaBridge` with 1 wei and the gas-bomb address.

No role, no approval, no special state is required. The function is unconditionally callable by any address on Unichain.

---

### Recommendation

Add an allowlist for the `l2bridge` parameter, or restrict `sendETHToL1ViaBridge` to only be callable by authorized pool contracts (e.g., via an `onlyRole` check or an immutable trusted-caller address set at construction). The simplest fix is to make `l2bridge` an immutable set at deployment rather than a caller-supplied argument, mirroring how the pool contracts already store `l2Bridge` as a validated state variable.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import {UnichainMessenger} from "contracts/bridges/UnichainMessenger.sol";

contract GasBomb {
    // Implements IUnichainMessenger.bridgeETHTo — burns all forwarded gas
    function bridgeETHTo(address, uint32, bytes memory) external payable {
        uint256 i;
        while (true) { unchecked { i++; } } // burns all gas
    }
    receive() external payable {}
}

contract PoC {
    function exploit(address messenger) external payable {
        GasBomb bomb = new GasBomb();
        // Single call saturates the block; repeat to stuff consecutive blocks
        UnichainMessenger(messenger).sendETHToL1ViaBridge{value: 1}(
            address(bomb),
            address(0xdead),
            1
        );
    }
}
```

Deploy on a local Unichain fork, call `exploit` with `value = 1 wei`, and observe that the transaction consumes the full block gas limit. The `BRIDGER_ROLE` operator's subsequent `bridgeAssetsViaNativeBridge` call cannot be included until the attacker stops stuffing blocks.
