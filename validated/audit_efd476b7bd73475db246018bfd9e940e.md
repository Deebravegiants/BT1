### Title
Missing Initial Reduction in `SUM_MOD` Causes Incorrect Modular Sum When `v[0] >= q` — (`include-internal/cbmpc/internal/crypto/base_bn256.h`)

### Summary
`SUM_MOD` assigns `s = v[0]` without reducing it modulo `q`. The subsequent `add_mod` calls assume both inputs are in `[0, q)` and perform only a single conditional subtraction. If `v[0] >= q`, the running sum is permanently off, producing an unreduced result that is accepted as valid.

### Finding Description
`SUM_MOD` (declared in `include-internal/cbmpc/internal/crypto/base_bn256.h`, defined in `src/cbmpc/crypto/base_bn256.cpp:1396`) initializes its accumulator as `s = v[0]` with no modular reduction. It then iterates `add_mod(s, s, v[i], q)` for `i >= 1`.

`add_mod` (lines 870–891) computes `a + b` with carry, then subtracts `q` once and uses a conditional mask to select the result. This is only correct when both `a, b ∈ [0, q)`. If `a >= q`, then `a + b >= q + 0 = q`, and after one subtraction the result can still be `>= q`.

`bn256_t::from_bin` (line 955) and `bn256_t::convert` (line 981) perform no range check — they accept any 32-byte big-endian value, including values `>= q`. A Byzantine peer can therefore inject a `bn256_t` with value `>= q` into a protocol message.

`SUM_MOD`/`SUM` is called from production MPC protocol files: `src/cbmpc/protocol/ecdsa_mp.cpp` (17 call sites), `src/cbmpc/protocol/ec_dkg.cpp` (3 call sites), and `src/cbmpc/protocol/schnorr_mp.cpp` (1 call site), where the vectors are populated from deserialized peer messages.

### Impact Explanation
A Byzantine participant below threshold sends a share contribution with `bn256_t` value `= q + δ` (where `δ >= 0`). When honest parties call `SUM_MOD` to reconstruct the scalar sum (e.g., a combined nonce, key share, or ZK scalar), the result is wrong by a multiple of `q` that `add_mod`'s single subtraction cannot correct. Honest parties accept this wrong sum, producing an invalid reconstructed secret share or an incorrect signing scalar, causing protocol divergence or acceptance of a cryptographically invalid output.

### Likelihood Explanation
Any Byzantine participant below threshold can trigger this by sending a single out-of-range `bn256_t` field. No special privilege is required beyond participation in the protocol. The deserialization path (`convert` → `from_bin`) has no guard.

### Recommendation
Add an initial reduction in `SUM_MOD`:
```cpp
bn256_t SUM_MOD(const std::vector<bn256_t>& v, const mod_t& q) {
  bn256_t s;
  if (!v.empty()) {
    bn256_t::add_mod(s, v[0], bn256_t(0), q);  // reduces v[0] mod q
    for (int i = 1; i < int(v.size()); i++) bn256_t::add_mod(s, s, v[i], q);
  }
  return s;
}
```
Alternatively, add a range check in `bn256_t::convert` to reject values `>= q` when a modulus context is active, or document and enforce the precondition that all `bn256_t` values passed to `SUM_MOD` must already be reduced.

### Proof of Concept
```
q = secp256k1 order
v[0] = q + 1   (injected by Byzantine peer via convert/from_bin, no rejection)
v[1] = q - 1

SUM_MOD({v[0], v[1]}, q):
  s = q + 1                          // unreduced
  add_mod(s, s, v[1], q):
    sum_256 = (q+1) + (q-1) = 2q
    // 2q > 2^256 for secp256k1, so carry=1, 256-bit part = 2q - 2^256
    diff = (2q - 2^256) - q = q - 2^256  // underflows, borrow=1
    subx(carry=1, 0, borrow=1) → new borrow=0, mask=0
    result = q - 2^256 + 2^256 = q   // keeps subtracted (underflowed) value
  returns q

Expected: 0 mod q
Actual:   q  (≡ 0 mod q but stored unreduced, corrupting all downstream add_mod calls)
```

void bn256_t::add_mod(bn256_t& r, const bn256_t& a, const bn256_t& b, const mod_t& mod) {
  const auto m = mod.value().val.d;

  uint64_t carry = 0;
  uint64_t a0 = addx(a.w0, b.w0, carry);
  uint64_t a1 = addx(a.w1, b.w1, carry);
  uint64_t a2 = addx(a.w2, b.w2, carry);
  uint64_t a3 = addx(a.w3, b.w3, carry);

  uint64_t borrow = 0;
  uint64_t s0 = subx(a0, m[0], borrow);
  uint64_t s1 = subx(a1, m[1], borrow);
  uint64_t s2 = subx(a2, m[2], borrow);
  uint64_t s3 = subx(a3, m[3], borrow);

  subx(carry, 0, borrow);
  uint64_t mask = -borrow;
  r.w0 = s0 ^ ((a0 ^ s0) & mask);
  r.w1 = s1 ^ ((a1 ^ s1) & mask);
  r.w2 = s2 ^ ((a2 ^ s2) & mask);
  r.w3 = s3 ^ ((a3 ^ s3) & mask);
}
