Audit Report

## Title
Missing Minimum Output Parameter in L2 Pool Deposit Functions Exposes Depositors to Oracle-Rate Slippage - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPool.sol`)

## Summary
All four L2 pool `deposit()` functions compute the rsETH/wrsETH output solely from the live oracle rate at execution time and immediately mint or transfer that amount, with no caller-supplied minimum output floor. The L1 `LRTDepositPool` enforces a `minRSETHAmountExpected` parameter in both `depositETH` and `depositAsset`, but this protection is entirely absent from every L2 pool variant. A user who previews their expected output via `viewSwapRsETHAmountAndFee()` and then submits a deposit may silently receive fewer tokens than anticipated if the oracle rate is updated before their transaction executes, with no on-chain mechanism to revert.

## Finding Description
Every L2 pool ETH deposit follows this pattern (shown for `RSETHPoolV3.sol` L246–265, identical in the other three contracts):

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
// no minimum check
wrsETH.mint(msg.sender, rsETHAmount);
```

`viewSwapRsETHAmountAndFee` fetches the live oracle rate via `getRate()` at call time (`RSETHPoolV3.sol` L304). If the oracle rate increases between the user's off-chain preview and on-chain execution — a routine event as staking rewards accrue — `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` yields a smaller value, and the mint proceeds unconditionally.

By contrast, `LRTDepositPool.depositETH` (L76–93) and `depositAsset` (L99–118) both pass `minRSETHAmountExpected` to `_beforeDeposit`, which enforces:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();   // L667–669
}
```

No equivalent guard exists in any L2 pool. The same missing check applies to the token `deposit(address, uint256, string)` overloads in all four contracts.

## Impact Explanation
The user's deposited ETH or LST is consumed at a worse rate than they previewed, with no revert and no recourse. The deposited value is not lost — the user receives rsETH at the current oracle rate — but the contract fails to deliver the output amount the user observed before submitting. This maps directly to the allowed impact: **Low — Contract fails to deliver promised returns, but does not lose value**.

## Likelihood Explanation
The rsETH oracle rate increases monotonically over time as staking rewards accrue. Oracle rate updates are routine, scheduled protocol operations, not edge cases. Any deposit submitted in the same block as or shortly after an oracle update will silently receive fewer rsETH/wrsETH than the user previewed. No attacker action is required; the condition arises from normal protocol operation. Likelihood is **Medium**.

## Recommendation
Add a `minRsETHAmountExpected` parameter to all L2 pool `deposit()` functions, mirroring the L1 `LRTDepositPool` pattern:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    wrsETH.mint(msg.sender, rsETHAmount);
    ...
}
```

Apply the same pattern to the token `deposit(address, uint256, string)` overloads in `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolNoWrapper.sol`, and `RSETHPool.sol`.

## Proof of Concept
1. Call `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV3` — observe expected output `X` wrsETH at current oracle rate `R`.
2. Protocol executes a routine oracle rate update, increasing the rate to `R' > R` (rsETH now worth more ETH per token).
3. User's pending `deposit{value: 1 ether}("ref")` executes after the update.
4. `rsETHAmount = (1 ether - fee) * 1e18 / R'` yields `Y < X` wrsETH.
5. `wrsETH.mint(msg.sender, Y)` executes unconditionally — no revert, no warning.
6. User receives `Y < X` wrsETH; the shortfall is unrecoverable.

Foundry fork test: deploy against a mainnet fork, call `viewSwapRsETHAmountAndFee`, advance the oracle rate via the rate-setter role, then call `deposit` and assert `wrsETH.balanceOf(user) < expectedAmount` with no revert.
