### Title
Multiple Sub-Transactions Can Claim the Same Deposit Refund in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs`)

---

### Summary

The Dijkstra era introduces nested (sub-)transactions. A top-level transaction can embed multiple sub-transactions, each processed through the `SUBLEDGERS` rule. The protection that prevents two sub-transactions from both claiming the deposit refund for the same staking credential is explicitly acknowledged as **not yet implemented** in the production codebase. An attacker can craft a single top-level transaction containing two sub-transactions that each carry an `UnRegDepositTxCert` for the same credential, receiving the deposit refund twice while only having paid it once.

---

### Finding Description

The Dijkstra `LEDGER` rule processes all sub-transactions first via `SUBLEDGERS`, then processes the top-level transaction body: [1](#0-0) 

The `originalUtxo` captured before sub-transaction processing is passed into the `SubLedgerEnv` and used for all witness/validation lookups across sub-transactions. The critical question is whether the `CertState` (which tracks registered credentials and their deposits) is properly updated and threaded between sequential sub-transactions so that the second sub-transaction cannot re-claim a refund already claimed by the first.

The test suite for the Dijkstra era contains the following disabled test: [2](#0-1) 

The test is marked `xit` (disabled/pending) and the expected predicate failure is `error "TODO: predicate failure not yet implemented"`. This is direct evidence that the guard preventing multiple sub-transactions from claiming the same deposit refund **has not been implemented**. The test documents the intended invariant ("Multiple subtransactions cannot get the same refund") but the enforcement mechanism is absent.

The deposit/refund accounting in `updateUTxOStateNoFees` uses the `CertState` passed in the environment — which in the Conway/Dijkstra `LEDGER` rule is intentionally the **pre-CERTS** state to allow deregistration + refund in the same transaction: [3](#0-2) [4](#0-3) 

If the `CertState` visible to each sub-transaction's `UTXOW` is the state **before** any prior sub-transaction's certificates were applied, both sub-transactions compute a non-zero refund for the same credential. The `conwayDRepRefundsTxCerts` / `shelleyTotalRefundsTxCerts` lookup functions return the stored deposit for a registered credential: [5](#0-4) [6](#0-5) 

The `conwayCertsTotalRefundsTxBody` function used by `updateUTxOStateNoFees` reads from the `CertState` snapshot passed to it: [7](#0-6) 

If that snapshot is stale (pre-subTx1 certificates), both sub-transactions independently compute a full refund, and `utxosDeposited` is decremented twice for a single deposit — creating ADA.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker registers one staking credential paying deposit `D`. They then submit a top-level transaction with two sub-transactions, each carrying `UnRegDepositTxCert stakingCred D`. If the missing guard is absent, both sub-transactions pass balance-conservation checks (each consuming `D` from `utxosDeposited`) and both produce outputs containing `D`. The attacker recovers `2D` having paid only `D`, directly creating `D` lovelace from the deposit pot. Repeated at scale this drains `utxosDeposited` and breaks the ADA conservation invariant across the entire ledger.

---

### Likelihood Explanation

The Dijkstra era is experimental/pre-production, but the vulnerability is reachable by any unprivileged transaction author who can submit a valid top-level transaction with sub-transactions. No privileged access, governance majority, or key compromise is required. The attack requires only: (1) a registered staking credential, (2) two UTxO inputs to fund the two sub-transactions, and (3) knowledge of the missing guard. The test file explicitly encodes the attack scenario, confirming the developers are aware of the gap.

---

### Recommendation

1. **Implement the missing predicate failure** referenced by the `xit` test in `Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`. The `SUBLEDGERS` or `SUBLEDGER` rule must ensure that the `CertState` is fully committed (post-CERTS) before it is used as the environment for the next sub-transaction's `UTXOW` deposit/refund calculation.
2. **Thread the post-CERTS `CertState`** from each sub-transaction into the environment of the subsequent sub-transaction, so that a credential deregistered in `subTx1` is no longer visible as registered when `subTx2` computes its refunds.
3. **Re-enable and complete the test** `"Multiple subtransactions cannot get the same refund"` with the correct predicate failure once the fix is in place.
4. Audit all other deposit-bearing certificate types (`RegDRepTxCert`/`UnRegDRepTxCert`, pool registration) for the same double-claim pattern across sub-transactions.

---

### Proof of Concept

```
1. Register stakingCred, paying keyDeposit D.

2. Construct:
     subTx1 = { inputs: {input1}, certs: [UnRegDepositTxCert stakingCred D] }
     subTx2 = { inputs: {input2}, certs: [UnRegDepositTxCert stakingCred D] }
     topTx  = { subTransactions: [subTx1, subTx2] }

3. Submit topTx.

4. If the CertState snapshot used by subTx2's UTXOW is the pre-subTx1 state:
     - subTx1 balance check: consumed includes refund D  ✓
     - subTx2 balance check: consumed includes refund D  ✓  (same stale snapshot)
     - utxosDeposited decremented by D twice
     - Attacker's outputs contain 2D; deposit pot loses 2D for one D paid.

5. Net effect: D lovelace created from thin air.
```

This mirrors the StEthHyperdrive reentrancy exactly: the callback (sub-transaction) executes against the **pre-update state** (pre-CERTS `CertState`), allowing the same economic position to be exploited twice before the state is committed.

```haskell
conwayDRepRefundsTxCerts lookupDRepDeposit = snd . F.foldl' go (Map.empty, Coin 0)
  where
    go accum@(!drepRegsInTx, !totalRefund) = \case
      RegDRepTxCert cred deposit _ ->
        -- Track registrations
        (Map.insert cred deposit drepRegsInTx, totalRefund)
      UnRegDRepTxCert cred _
        -- DRep previously registered in the same tx.
        | Just deposit <- Map.lookup cred drepRegsInTx ->
            (Map.delete cred drepRegsInTx, totalRefund <+> deposit)
        -- DRep previously registered in some other tx.
        | Just deposit <- lookupDRepDeposit cred -> (drepRegsInTx, totalRefund <+> deposit)
      _ -> accum
```

```haskell
shelleyTotalRefundsTxCerts pp lookupDeposit = snd . F.foldl' accum (mempty, Coin 0)
  where
    keyDeposit = pp ^. ppKeyDepositL
    accum (!regCreds, !totalRefunds) cert =
      case lookupRegStakeTxCert cert of
        Just k ->
          -- Need to track new delegations in case that the same key is later deregistered in
          -- the same transaction.
          (Set.insert k regCreds, totalRefunds)
        Nothing ->
          case lookupUnRegStakeTxCert cert of
            Just cred
              -- We first check if there was already a registration certificate in this
              -- transaction.
              | Set.member cred regCreds -> (Set.delete cred regCreds, totalRefunds <+> keyDeposit)
              -- Check for the deposit left during registration in some previous
              -- transaction. This de-registration check will be matched first, despite being
              -- the last case to match, because registration is not possible without
              -- de-registration.
              | Just deposit <- lookupDeposit cred -> (regCreds, totalRefunds <+> deposit)
            _ -> (regCreds, totalRefunds)
```
