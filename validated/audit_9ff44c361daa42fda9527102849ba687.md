### Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Unlimited ETH Deposits Beyond Protocol Cap - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric deposit-limit check: for LST assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it tests only `totalAssetDeposits > depositLimit`, silently dropping the incoming `amount`. Any depositor can therefore push ETH deposits arbitrarily beyond the configured cap and receive a proportional rsETH mint for the full over-limit amount.

### Finding Description
The deposit-limit guard in `_checkIfDepositAmountExceedesCurrentLimit` branches on the asset type:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount missing
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for LSTs
}
```

For ETH the function returns `true` (i.e., "limit exceeded, revert") only when the *pre-deposit* total already exceeds the cap. The new `amount` is never added. As long as the pool has not yet crossed the limit, any deposit size passes the check.

The check is invoked unconditionally inside `_beforeDeposit`, which is called by the public `depositETH` entry point:

```solidity
// contracts/LRTDepositPool.sol  lines 86-92
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    ...
}
```

`_mintRsETH` then mints rsETH proportional to the full `msg.value` regardless of how far it exceeds the cap.

### Impact Explanation
The deposit limit is the protocol's primary risk-management lever for controlling ETH exposure. Bypassing it means:

1. The protocol accepts and mints rsETH for ETH volumes it was not designed to handle.
2. Excess ETH that cannot be deployed to EigenLayer strategies sits idle in the deposit pool, earning no yield, while rsETH holders expect restaking returns — the contract fails to deliver its promised returns.
3. If EigenLayer strategy capacity is exhausted, the idle ETH cannot be put to work, permanently degrading yield for all rsETH holders.

No direct theft of funds occurs because the depositor sends real ETH; however, the protocol's safety invariant (capped TVL) is broken by any unprivileged caller.

**Impact class**: Low — Contract fails to deliver promised returns, but doesn't lose value.

### Likelihood Explanation
The entry point `depositETH` is public and payable with no access restriction beyond `whenNotPaused`. Any depositor who observes that `totalAssetDeposits` is at or near the limit can immediately exploit the missing check by sending a large ETH deposit. No special role, flash loan, or multi-step setup is required.

### Recommendation
Add `amount` to the ETH branch of the limit check, mirroring the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
Assume `depositLimitByAsset(ETH_TOKEN) = 100 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 99 ether`.

1. Attacker calls `depositETH{value: 1000 ether}(minAmount, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1000 ether)`.
3. Check evaluates `99 ether > 100 ether` → `false` → limit not exceeded.
4. `getRsETHAmountToMint(ETH_TOKEN, 1000 ether)` returns rsETH for the full 1000 ETH.
5. `_mintRsETH` mints that rsETH to the attacker.
6. Protocol now holds 1099 ETH against a 100 ETH cap; the excess 999 ETH cannot be deployed to EigenLayer strategies, leaving rsETH holders with degraded yield. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
