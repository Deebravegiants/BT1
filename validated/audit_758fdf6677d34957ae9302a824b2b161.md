Audit Report

## Title
Stale Cross-Chain Rate in `CrossChainRateReceiver.getRate()` Causes `AGETHPoolV3.deposit()` to Under-Mint agETH — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally without consulting `lastUpdated`, which is written on every LayerZero delivery but never read by any on-chain logic. If a LayerZero outage or delay freezes the stored rate while the true agETH/ETH rate falls (e.g., due to a slashing event or rate correction), `AGETHPoolV3.deposit()` will compute a smaller agETH mint than the current backing warrants, leaving depositors with a position immediately worth less than what they paid.

## Finding Description

**`CrossChainRateReceiver` — missing staleness guard**

`lzReceive` writes both `rate` and `lastUpdated` on every delivery: [1](#0-0) 

`getRate()` returns `rate` with no age check; `lastUpdated` is never read by any on-chain logic: [2](#0-1) 

**`AGETHPoolV3` — deposit math divides by the oracle rate**

`deposit()` calls `viewSwapAgETHAmountAndFee`, which fetches the rate via `getRate()` and computes: [3](#0-2) 

`getRate()` in the pool delegates directly to `agETHOracle`, which is the `AGETHRateReceiver` instance: [4](#0-3) 

`agETHOracle` is set at initialization and can only be changed by the admin: [5](#0-4) 

**Exploit path:** A LayerZero outage freezes the stored rate at a value higher than the true current rate (possible during a slashing event or rate correction). Any unprivileged user who calls `deposit()` during this window receives `amountAfterFee * 1e18 / staleHighRate` agETH — fewer tokens than `amountAfterFee * 1e18 / trueRate` would yield. No existing check in `deposit()`, `viewSwapAgETHAmountAndFee()`, or `getRate()` consults `lastUpdated` or enforces a maximum rate age.

## Impact Explanation

Depositors receive fewer agETH than the current backing justifies. The deposited ETH is not lost — it remains in the pool — but the depositor's position is immediately worth less than what they paid. This matches the **Low** allowed impact: *contract fails to deliver promised returns, but doesn't lose value*.

## Likelihood Explanation

agETH's rate normally increases monotonically, so a stale-high scenario requires either a slashing event causing a rate decrease or a prolonged LayerZero outage that freezes the rate while the true rate falls. LayerZero message delays and outages are documented operational risks. The presence of `lastUpdated` as a written-but-never-read public field confirms the staleness guard was anticipated but never implemented, making this a latent gap rather than a purely theoretical one.

## Recommendation

Add a maximum staleness threshold in `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

Alternatively, expose `lastUpdated` to `AGETHPoolV3` and enforce the check before minting in `viewSwapAgETHAmountAndFee`.

## Proof of Concept

```solidity
function testStaleHighRateUnderMints() public {
    AGETHRateReceiver receiver = AGETHRateReceiver(<deployed_address>);

    uint256 trueRate = receiver.getRate();        // e.g. 1.05e18
    uint256 staleRate = trueRate * 105 / 100;     // 1.1025e18 — stale-high

    // Overwrite storage slot for `rate`; lastUpdated is NOT updated
    vm.store(address(receiver), bytes32(0), bytes32(staleRate));

    uint256 agETHBefore = agETH.balanceOf(alice);
    vm.prank(alice);
    pool.deposit{value: 1 ether}("");
    uint256 minted = agETH.balanceOf(alice) - agETHBefore;

    uint256 expected = 1 ether * 1e18 / trueRate;

    // minted < expected — depositor is underpaid
    assertLt(minted, expected);
}
```

The test passes on unmodified code because `getRate()` returns `rate` without checking `lastUpdated`. [2](#0-1)
