### Title
Token-2022 MetadataPointer Allows Mint-Metadata Dissociation in `log_metadata` — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`parse_metadata_account` in `log_metadata.rs` verifies that the supplied metadata account's address matches the mint's `MetadataPointer` and that the account is owned by the Metaplex program, but **never checks that `metadata.mint == self.mint.key()`**. An unprivileged attacker can create a Token-2022 mint with a `MetadataPointer` extension pointing to any existing Metaplex metadata account (e.g., USDC's), causing the bridge to emit a `LogMetadataPayload` that carries the attacker's mint pubkey paired with a legitimate token's name and symbol.

---

### Finding Description

The vulnerable path is the "Third-party metadata" branch in `process()`: [1](#0-0) 

This calls `parse_metadata_account` with the address stored in the mint's `MetadataPointer` extension. Inside that function: [2](#0-1) 

The two checks performed are:
1. `metadata.key() == address` — the passed account matches the pointer address.
2. `metadata.owner == &MetaplexID` — the account is owned by Metaplex.

There is **no** `require_keys_eq!(metadata.mint, self.mint.key(), ...)`. The Metaplex `MetadataAccount` struct carries a `mint: Pubkey` field that identifies which mint the metadata belongs to, but it is never consulted here.

The resulting payload encodes the attacker's mint pubkey alongside the stolen name/symbol: [3](#0-2) 

---

### Impact Explanation

The `LogMetadataPayload` Wormhole message is consumed by the NEAR side to register a Solana token. Because `token` is set to the attacker's mint `M` but `name`/`symbol` are taken from a high-value token `T` (e.g., USDC), NEAR registers `M` as a token whose display identity is indistinguishable from `T`. Any user or front-end that trusts the on-chain name/symbol to identify the asset will be deceived. This is a direct asset-identity divergence: the backing guarantee that "the name/symbol in `LogMetadataPayload` belongs to the mint being registered" is broken. [4](#0-3) 

---

### Likelihood Explanation

- The attacker needs only to: (a) create a Token-2022 mint with a `MetadataPointer` extension (permissionless, costs ~0.01 SOL), and (b) call `log_metadata` (permissionless, no privileged role required).
- The Metaplex metadata PDA for any token is deterministically derivable from public information (`[METADATA_SEED, MetaplexID, mint.key()]`), so the target address is always known in advance.
- No key compromise, colluding validators, or network-level attack is required.

---

### Recommendation

Add a mint-binding check immediately after deserializing the Metaplex account in `parse_metadata_account`:

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let metadata = MplMetadata::try_deserialize(&mut data.as_ref())?;
    // ADD THIS:
    require_keys_eq!(
        metadata.mint,
        self.mint.key(),
        ErrorCode::InvalidTokenMetadataAddress,
    );
    Ok((metadata.name.clone(), metadata.symbol.clone()))
}
``` [5](#0-4) 

---

### Proof of Concept

```
1. Let T = USDC mint pubkey (known).
2. Compute T_meta = find_program_address(
       [b"metadata", MetaplexID, T.to_bytes()], MetaplexID).0
   // T_meta is USDC's Metaplex metadata account, owned by MetaplexID,
   // with name="USD Coin", symbol="USDC", mint=T.

3. Create Token-2022 mint M with:
   - MetadataPointer extension: metadata_address = T_meta
   - mint_authority != bridge authority PDA  (passes the constraint check)

4. Call log_metadata(mint=M, metadata=T_meta, token_program=Token-2022).

5. Inside process():
   - metadata_pointer.metadata_address.0 == T_meta != M  → "Third-party" branch
   - parse_metadata_account(T_meta) is called
   - metadata.key() == T_meta  ✓
   - metadata.owner == MetaplexID  ✓
   - returns ("USD Coin", "USDC")   ← stolen from T

6. LogMetadataPayload { token: M, name: "USD Coin", symbol: "USDC", decimals: M.decimals }
   is posted as a Wormhole message.

7. NEAR registers M as a token with USDC's identity.
```

A unit test using the existing Mollusk harness can reproduce this by constructing a mint account with a `MetadataPointer` pointing to a separately-created Metaplex metadata account whose `mint` field is a different pubkey, then asserting that `log_metadata` either rejects the call or that the emitted name/symbol matches the actual mint — neither of which currently holds. [6](#0-5)

    fn parse_metadata_account(&self, address: Pubkey) -> Result<(String, String)> {
        let metadata = self

            address,
