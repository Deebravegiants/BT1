### Title
Overly Strict All-Participant Intersection Rule in `verify_foreign_transaction` Allows Single Byzantine Participant to Block All Foreign Chain Verification — (`crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` function gates request acceptance on `get_supported_foreign_chains()`, which requires **every** active participant to have registered a given chain. A single Byzantine participant below the signing threshold can register an empty foreign-chain configuration, collapsing the supported-chain set to empty and permanently blocking all `verify_foreign_transaction` requests — without any threshold-level collusion.

---

### Finding Description

`verify_foreign_transaction` (line 534) calls `get_supported_foreign_chains()` to validate the requested chain:

```rust
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(...)
}
```

`get_supported_foreign_chains()` (lines 2203–2217) computes the **strict intersection** of all active participants' registered chains:

```rust
foreign_chain_to_node_mapping
    .into_iter()
    .filter_map(|(foreign_chain, nodes_supporting_chain)| {
        let all_active_nodes_supports_chain =
            nodes_supporting_chain.is_superset(&active_participant_account_ids);
        if all_active_nodes_supports_chain { Some(foreign_chain) } else { None }
    })
```

The condition `is_superset(&active_participant_account_ids)` means a chain is only "supported" if **every** active participant has registered it. One participant registering an empty configuration drops every chain from the supported set.

The contract already implements the correct alternative — `get_available_foreign_chains()` — which counts per-chain supporters and requires only `signing_threshold` participants to cover a chain. The design document at `docs/design/calculating-supported-foreign-chains.md` explicitly identifies this as a known bug:

> *"A single node that registers an empty list (or hasn't registered yet) drops every chain — one operator can take the whole feature down."*

However, `verify_foreign_transaction` still uses the old strict rule. The fix (switching to `get_available_foreign_chains()`) is proposed but not yet applied.

---

### Impact Explanation

**Medium.** A single Byzantine participant (strictly below the signing threshold) can call `register_foreign_chain_config` with an empty set. This causes `get_supported_foreign_chains()` to return an empty set, making every subsequent `verify_foreign_transaction` call fail with `ForeignChainNotSupported`. The entire foreign-chain verification feature — including bridge inbound flows — is disabled until the participant re-registers. This breaks the request-lifecycle and contract execution-flow invariant that the network should tolerate up to `n − signing_threshold` non-covering nodes, without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

**Medium.** The `register_foreign_chain_config` call is permissioned to active participants only, so an external unprivileged caller cannot trigger this directly. However, only **one** Byzantine participant is required — well below the signing threshold — making this reachable by any single compromised or malicious node operator. The action is a single on-chain call with no special preconditions.

---

### Recommendation

Replace `get_supported_foreign_chains()` with `get_available_foreign_chains()` in `verify_foreign_transaction`. The available-chain set already enforces the correct invariant: a chain is serviceable iff ≥ `signing_threshold` active participants cover it, matching the actual cryptographic requirement. The design document (tracked in issue #3434) already specifies this migration path.

---

### Proof of Concept

1. Contract is running with `N = 4` participants, signing threshold `T = 3`.
2. Byzantine participant `P` calls `register_foreign_chain_config({})` (empty configuration).
3. `get_supported_foreign_chains()` evaluates `nodes_supporting_chain.is_superset(&active_participant_account_ids)` for every chain — since `P` is in `active_participant_account_ids` but not in any chain's supporter set, the superset check fails for all chains.
4. `get_supported_foreign_chains()` returns an empty set.
5. Any user calling `verify_foreign_transaction` for Bitcoin, Ethereum, or any other chain receives `ForeignChainNotSupported` and the request is rejected on-chain.
6. The bridge inbound flow is fully disabled. No threshold-level collusion was required — only one participant acted.

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
