### Title
Unenforceable `farmer_reward_address_override` in `RespondSignatures` Allows Malicious Harvester to Divert Farmer Block Rewards - (File: `chia/farmer/farmer_api.py`)

### Summary
A third-party harvester (CHIP-22) can set `farmer_reward_address_override` to any arbitrary puzzle hash in a `RespondSignatures` message. The farmer unconditionally substitutes this value as the `farmer_reward_address` in `DeclareProofOfSpace` with no enforcement of the fee quality threshold and no validation against the farmer's configured address. A malicious harvester can redirect 100% of the farmer's block rewards to an attacker-controlled address.

### Finding Description

In `_process_respond_signatures` in `chia/farmer/farmer_api.py`, when the harvester's `RespondSignatures` message contains a non-`None` `farmer_reward_address_override`, the farmer unconditionally replaces its own configured `farmer_target` with the harvester-supplied value:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

This value is then passed directly into `DeclareProofOfSpace` as `farmer_reward_address` and forwarded to the full node, which uses it verbatim to create the block's foliage with the farmer reward coin paying to that puzzle hash. [1](#0-0) 

The only guard in the codebase is `notify_farmer_reward_taken_by_harvester_as_fee`, called when `new_proof_of_space.farmer_reward_address_override is not None` (a **separate** field on `NewProofOfSpace`, not on `RespondSignatures`). This function only **logs** a warning when the fee quality does not meet the declared threshold — it does not reject the override, abort block creation, or prevent the `DeclareProofOfSpace` from being sent. [2](#0-1) [3](#0-2) 

Because `RespondSignatures.farmer_reward_address_override` and `NewProofOfSpace.farmer_reward_address_override` are **two independent fields**, a malicious harvester can:

1. Send `NewProofOfSpace` with `farmer_reward_address_override=None` — bypassing even the advisory fee-quality log entirely.
2. Later respond to `RequestSignatures` with `RespondSignatures` where `farmer_reward_address_override=<attacker_address>`.
3. The farmer applies the override with zero checks and sends `DeclareProofOfSpace` to the full node with the attacker's puzzle hash as the farmer reward destination.

The full node's `declare_proof_of_space` handler accepts `request.farmer_puzzle_hash` directly from the farmer without any independent validation of who that address belongs to: [4](#0-3) 

### Impact Explanation

Every block won by the farmer while connected to the malicious harvester will pay the 1/8 base farmer reward (plus any transaction fees) to the attacker's address instead of the farmer's configured `xch_target_address`. This is an on-chain, irreversible transfer of XCH to an attacker-controlled puzzle hash — a direct reward diversion matching the Critical impact criterion.

### Likelihood Explanation

The CHIP-22 third-party harvester model is a production feature explicitly supported by the codebase. Any harvester operator who connects to a farmer can exploit this. No key material, admin access, or cryptographic break is required — only the ability to respond to `RequestSignatures` with a crafted `RespondSignatures` message, which is normal harvester protocol behavior.

### Recommendation

The farmer should enforce the fee quality threshold as a hard gate, not merely a log. Specifically:

- In `_process_respond_signatures`, before accepting `response.farmer_reward_address_override`, verify that the corresponding proof's fee quality satisfies the declared `applied_fee_threshold`. If it does not, discard the override and use `self.farmer.farmer_target`.
- Alternatively, expose a farmer configuration option to disable `farmer_reward_address_override` entirely for operators who do not use third-party harvesters.
- Ensure the fee quality check covers `RespondSignatures.farmer_reward_address_override`, not only `NewProofOfSpace.farmer_reward_address_override`.

### Proof of Concept

1. Malicious harvester connects to a farmer running with plots.
2. When a valid proof of space is found, the harvester sends `NewProofOfSpace` with `farmer_reward_address_override=None` (no fee quality check triggered).
3. Farmer sends `RequestSignatures` back to the harvester.
4. Harvester responds with `RespondSignatures` where `farmer_reward_address_override = <attacker_bytes32_puzzle_hash>`.
5. Farmer executes lines 917–918 of `farmer_api.py`: `farmer_reward_address = response.farmer_reward_address_override`.
6. Farmer sends `DeclareProofOfSpace` to the full node with `farmer_puzzle_hash = <attacker_address>`.
7. Full node creates the block foliage with the attacker's puzzle hash as the farmer reward coin destination (line 1069 of `full_node_api.py`).
8. Block is accepted by the network; the farmer reward coin is created paying to the attacker's address. [5](#0-4) [6](#0-5)

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
