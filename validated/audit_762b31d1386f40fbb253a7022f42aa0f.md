### Title
Sub-Transaction ExUnits Bypass Allows Exceeding `maxTxExUnits` and `maxBlockExUnits` Limits Without Paying Script Fees - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, the per-transaction execution-unit limit (`maxTxExUnits`) and the minimum-fee script-cost calculation both rely on `totExUnits`, which only sums redeemers from the **top-level** transaction's witness set. Sub-transactions embedded in `subTransactionsTxBodyL` carry their own witness sets with their own redeemers and ExUnits budgets. Those sub-transaction ExUnits are never counted against `maxTxExUnits`, `maxBlockExUnits`, or the minimum fee. An unprivileged sender can therefore force validators to execute arbitrarily large amounts of Plutus computation while paying fees and staying within limits calculated only on the (near-zero) top-level ExUnits.

---

### Finding Description

`totExUnits` is defined as:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
``` [1](#0-0) 

It only traverses the top-level transaction's redeemer map. Sub-transactions are stored in the body field `subTransactionsTxBodyL` and each carries its own `witsTxL . rdmrsTxWitsL` with independent ExUnits entries. Those entries are invisible to `totExUnits`.

The Dijkstra UTXO transition enforces the per-transaction limit as:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [2](#0-1) 

`validateExUnitsTooBigUTxO` compares `totExUnits tx` (top-level only) against `ppMaxTxExUnitsL`:

```haskell
validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) ...
  where
    maxTxExUnits = pp ^. ppMaxTxExUnitsL
    totalExUnits = totExUnits tx
``` [3](#0-2) 

The minimum-fee calculation is equally blind to sub-transactions:

```haskell
alonzoMinFeeTx pp tx =
  ... <+> txscriptfee (pp ^. ppPricesL) allExunits
  where
    allExunits = totExUnits tx
``` [4](#0-3) 

The block-level check in `validateExUnits` (BBODY) also folds `totExUnits` over top-level transactions only:

```haskell
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
``` [5](#0-4) 

By contrast, the collateral check in Dijkstra **does** inspect sub-transactions:

```haskell
hasAnyRedeemers t =
  hasRedeemers t || any hasRedeemers (t ^. bodyTxL . subTransactionsTxBodyL)
``` [6](#0-5) 

This asymmetry confirms the design intent is to treat sub-transactions as part of the batch for validation purposes, but the ExUnits accounting was not extended to match.

Sub-transaction scripts are collected and executed via `plutusScriptsWithContextDijkstraStAnnTx`, which gathers scripts from both levels:

```haskell
plutusScriptsWithContextDijkstraStAnnTx stAnnTx =
  withBothTxLevels stAnnTx
    (\DijkstraStAnnTopTx {dsattPlutusScriptsWithContext} -> dsattPlutusScriptsWithContext)
    (\DijkstraStAnnSubTx {dsastPlutusScriptsWithContext} -> dsastPlutusScriptsWithContext)
``` [7](#0-6) 

So sub-transaction Plutus scripts are executed, but their ExUnits are not counted against any limit.

---

### Impact Explanation

**Medium** — Attacker-controlled transactions exceed intended validation limits and modify fees outside design parameters.

1. **Resource-limit bypass**: A single Dijkstra top-level transaction with N sub-transactions, each declaring `maxTxExUnits` worth of ExUnits, forces validators to execute up to `N × maxTxExUnits` of computation. Neither the per-transaction nor the per-block cap is enforced on the aggregate.

2. **Fee undercharge**: The minimum fee includes `txscriptfee prices (totExUnits tx)`, which is zero when all scripts are in sub-transactions. The attacker pays only the byte-size fee for the transaction, not the script-execution fee, while validators bear the full execution cost.

---

### Likelihood Explanation

**Medium**. The Dijkstra era is new and sub-transactions are a novel feature. Any DApp author or adversarial user who discovers that sub-transaction ExUnits are unchecked can exploit this immediately with a single crafted transaction. No privileged access, governance majority, or key compromise is required — only the ability to submit a valid Dijkstra-era transaction.

---

### Recommendation

Extend `totExUnits` (or introduce a Dijkstra-specific variant) to also sum ExUnits from all sub-transactions:

```haskell
totExUnitsBatch tx =
  totExUnits tx
    <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
```

Use this batch-aware function in:
1. `dijkstraUtxoTransition` for the `maxTxExUnits` check.
2. The minimum-fee calculation (`getMinFeeTxUtxo`) for Dijkstra-era transactions.
3. The BBODY `validateExUnits` fold, so `maxBlockExUnits` is also enforced over the full batch.

---

### Proof of Concept

1. Set `maxTxExUnits = ExUnits { memory = 14_000_000, steps = 10_000_000_000 }` (current mainnet values).
2. Craft a Dijkstra top-level transaction with zero top-level redeemers and 10 sub-transactions, each containing one Plutus script with a redeemer declaring `ExUnits { memory = 14_000_000, steps = 10_000_000_000 }`.
3. Submit the transaction. The UTXO check `validateExUnitsTooBigUTxO pp tx` sees `totExUnits tx = ExUnits 0 0 ≤ maxTxExUnits` — passes.
4. The minimum fee is computed without any script fee component — the attacker pays only the byte-size fee.
5. The UTXOS rule executes all 10 sub-transaction scripts, consuming up to `10 × maxTxExUnits` of validator CPU and memory.
6. The BBODY `validateExUnits` check also passes because it folds `totExUnits` over top-level transactions only.

The attacker has forced 10× the intended per-transaction computation budget to be consumed at the cost of a plain byte-size fee.
