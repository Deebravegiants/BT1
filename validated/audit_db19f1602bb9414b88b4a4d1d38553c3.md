### Title
Pool Admin Bypasses 20% Fee Cap via Uncapped Per-Bin Additional Fees - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap validation. These per-bin fees are added on top of the global spread fee in every swap step, allowing the pool admin to push the effective per-bin fee above the protocol-enforced 20% ceiling.

### Finding Description
The global admin spread fee is capped at `maxAdminSpreadFeeE6` (≤ `HARD_MAX_SPREAD_FEE_E6` = 200,000 E6 = 20%) and enforced in `setPoolAdminFees`:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

However, `setPoolBinAdditionalFees` performs no analogous check:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool's `setBinAdditionalFees` only validates the bin index:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
```

During every swap step the per-bin additional fee is added directly to the base fee before computing the swap output:

```solidity
// MetricOmmPool.sol:1177
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6),
// MetricOmmPool.sol:1088
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
```

`addFeeBuyE6` / `addFeeSellE6` are `uint16`, so the pool admin can set either to `type(uint16).max` = 65,535 (E6 units ≈ 6.55%) with a single call. Combined with the maximum global spread fee of 20%, the effective per-bin fee reaches ≈ 26.55%, exceeding the hard cap by ~32%.

### Impact Explanation
Every swap routed through the affected bin pays the inflated fee. Traders receive less output (exact-input) or must supply more input (exact-output) than the 20% cap permits. The excess fee accrues as LP fee inside the bin, constituting a direct, quantifiable loss of swap output for every user who trades through that bin. The loss is bounded by `uint16` headroom (~6.55% above the cap) but applies to the full notional of every affected swap.

### Likelihood Explanation
The pool admin is classified as "semi-trusted, bounded" and explicitly "cannot exceed caps." The contest Q&A states findings are valid when the pool admin bypasses caps and the impact is Medium or higher. No timelock, multisig requirement, or secondary approval gates `setPoolBinAdditionalFees`; a single transaction from the pool admin address is sufficient. The call is indistinguishable from a legitimate fee adjustment until traders observe degraded output.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees`) that ensures `addFeeBuyE6` and `addFeeSellE6` do not exceed `maxAdminSpreadFeeE6` (or a dedicated per-bin cap constant), preventing the combined effective fee from exceeding the protocol-enforced ceiling:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

### Proof of Concept

1. Pool is deployed with `adminSpreadFeeE6 = 200_000` (20%, the maximum) and `protocolSpreadFeeE6 = 0`.
2. Pool admin calls:
   ```solidity
   factory.setPoolBinAdditionalFees(pool, 0, 65_535, 65_535);
   ```
   No revert occurs; `_binStates[0].addFeeBuyE6 = 65_535` and `addFeeSellE6 = 65_535` are stored.
3. A trader calls `swap(…, zeroForOne=true, …)`. The swap loop enters bin 0 and computes:
   ```
   effectiveFeeX64 = baseFeeX64 + mulDiv(65_535, ONE_X64, 1e6)
                   ≈ (200_000/1e6 + 65_535/1e6) × 2^64
                   ≈ 26.55% fee
   ``` [1](#0-0) 
4. The trader receives ≈ 6.55% less token0 than the 20%-cap scenario, with no on-chain guard having fired.
