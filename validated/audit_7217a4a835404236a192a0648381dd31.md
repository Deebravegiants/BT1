### Title
Backup-CLI `secrets.json` Written Without Restricted File Permissions, Exposing Key-Share Decryption Key and Backup-Service Private Key - (File: `crates/backup-cli/src/adapters/secrets_storage.rs`)

---

### Summary

The `backup-cli` tool writes its `secrets.json` file — which contains the backup service's Ed25519 private key (`p2p_private_key`) and the AES-128 key used to decrypt locally stored keyshares (`local_storage_aes_key`) — using default OS permissions (typically `0o644`). Any local user on the same machine can read this file. An attacker who reads it can decrypt all keyshares the backup-cli has collected from MPC nodes, and/or impersonate the registered backup service to request additional keyshares from live MPC nodes. If the attacker accumulates ≥ threshold keyshares, they can reconstruct the MPC root private key and issue unauthorized signatures on any foreign chain.

---

### Finding Description

`JsonSecretsStorage::open_write()` in `crates/backup-cli/src/adapters/secrets_storage.rs` creates `secrets.json` using `tokio::fs::File::options()` without setting a restrictive mode:

```rust
// crates/backup-cli/src/adapters/secrets_storage.rs lines 55-63
let destination = File::options()
    .create(true)
    .truncate(true)
    .write(true)
    .open(file_path)          // ← no .mode(0o600)
    .await
    .map_err(Error::OpenFile)?;
``` [1](#0-0) 

The file created inherits the process umask (commonly `0o022`, yielding `0o644` — world-readable). The file contains:

```rust
// crates/backup-cli/src/types.rs lines 6-13
pub struct PersistentSecrets {
    pub p2p_private_key: SigningKey,       // mTLS identity registered on-chain
    pub near_signer_key: SigningKey,
    pub local_storage_aes_key: [u8; 16],  // decrypts locally stored keyshares
}
``` [2](#0-1) 

This is in direct contrast to the MPC node itself, which uses `write_secret_file()` with an explicit `0o600` mode for every secret it writes (`secrets.json`, `backup_encryption_key.hex`):

```rust
// crates/node/src/config.rs lines 192-208
fn write_secret_file(path: &Path, data: &[u8]) -> anyhow::Result<()> {
    let mut file = fs::OpenOptions::new()
        .write(true).create(true).truncate(true)
        .mode(0o600)
        .open(path)...
    fs::set_permissions(path, fs::Permissions::from_mode(0o600))...
}
``` [3](#0-2) 

The `backup-cli` is a production migration tool. Its `secrets.json` is generated by `generate-keys` and then used in `get-keyshares` / `put-keyshares` to authenticate to live MPC nodes and decrypt collected keyshares:

```rust
// crates/backup-cli/src/backup.rs lines 53-70
let secrets = ports::SecretsRepository::load_secrets(&secrets_storage)...
let mpc_p2p_client = adapters::p2p_client::MpcP2PClient::new(
    ..., secrets.p2p_private_key, backup_encryption_key,
);
let key_shares_storage = adapters::keyshare_storage::KeyshareStorageAdapter::new(
    home_dir.clone(), secrets.local_storage_aes_key,
)...
``` [4](#0-3) 

The documentation itself acknowledges the sensitivity: *"Keep the `secrets.json` file secure. Anyone with access to this file can authenticate as your backup service and decrypt any keyshares stored locally."* — but the code does not enforce this. [5](#0-4) 

---

### Impact Explanation

**Critical — Unauthorized access to MPC key shares enabling secret recovery.**

An attacker who reads the world-readable `secrets.json` gains two independent attack vectors:

1. **Decrypt locally stored keyshares**: The `local_storage_aes_key` is the sole protection over keyshares the backup-cli has already fetched from MPC nodes and stored on disk. The attacker decrypts them directly without any network interaction.

2. **Impersonate the registered backup service**: The `p2p_private_key` is the credential registered on-chain via `register_backup_service`. The MPC node authorizes keyshare export to any client presenting this key. The attacker can connect to any MPC node that has the backup service registered and request its keyshare.

If the attacker collects keyshares from ≥ threshold nodes (realistic during a multi-node migration window when the backup service is registered with all source nodes), they can reconstruct the MPC root private key and issue arbitrary threshold signatures on any foreign chain — constituting full theft of funds controlled by the MPC network.

---

### Likelihood Explanation

**Medium.** The attack requires local access to the same machine as the backup-cli operator (a second OS user account, a compromised process running as a different user, or a backup of the home directory). This is a realistic threat model for shared infrastructure, CI/CD runners, or cloud VMs where multiple services or users coexist. The migration window — when the backup service is registered with multiple nodes and keyshares are transiently stored on disk — is the highest-risk period, but the file persists indefinitely after migration.

---

### Recommendation

Apply `0o600` permissions when creating `secrets.json` in `JsonSecretsStorage::open_write()`, mirroring the pattern already used by the MPC node's `write_secret_file()`. On Unix, use `OpenOptionsExt::mode(0o600)` before calling `.open()`. Additionally, on `open_read()`, check the file's current permissions and emit an error or warning if they are broader than `0o600`, analogous to how `openssh` refuses to use a private key file with overly broad permissions.

### Proof of Concept

```bash
# Operator generates backup-cli keys (production step)
backup-cli --home-dir /home/operator/backup generate-keys

# Attacker (different local user) reads the world-readable file
ls -la /home/operator/backup/secrets.json
# -rw-r--r-- 1 operator operator ... secrets.json   ← 0o644, readable by all

cat /home/operator/backup/secrets.json
# {
#   "p2p_private_key": "ed25519:<base58-encoded-64-bytes>",
#   "near_signer_key":  "ed25519:<base58-encoded-64-bytes>",
#   "local_storage_aes_key": [<16 bytes>]
# }

# Attacker decrypts locally stored keyshares using local_storage_aes_key
# (same AES-128-GCM scheme used by KeyshareStorageAdapter)
# → obtains raw keyshare material without any network interaction

# OR: attacker uses p2p_private_key to open mTLS connection to MPC node
# migration endpoint (port 8079) and requests keyshares as the registered
# backup service → MPC node serves its keyshare because the public key
# matches the on-chain registration
```

The root cause is solely in `crates/backup-cli/src/adapters/secrets_storage.rs` lines 55–63: the absence of `.mode(0o600)` on the `File::options()` builder. No threshold collusion, TEE attack, or privileged access is required — only a second OS-level user account on the same host.
