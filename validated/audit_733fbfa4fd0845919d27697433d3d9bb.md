Audit Report

## Title
Zero-Minimum Withdrawal Queue Stuffing Delays Legitimate User Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`minRsEthAmountToWithdraw` is a plain mapping that defaults to `0` for every asset until an admin explicitly calls `setMinRsEthAmountToWithdraw`. Because the only guard in `initiateWithdrawal` is `rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]`, any unprivileged rsETH holder can enqueue arbitrarily many 1-wei withdrawal requests. The global FIFO queue cannot skip nonces, so legitimate users' requests queued after the dust entries are temporarily frozen until every preceding dust entry is processed by an operator.

## Finding Description

`minRsEthAmountToWithdraw` is declared as a plain mapping at line 35 and is never initialized in `initialize`:

```solidity
mapping(address asset => uint256) public minRsEthAmountToWithdraw;
```

The guard in `initiateWithdrawal` (lines 162â€“164) is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

When `minRsEthAmountToWithdraw[asset] == 0` (the default), the condition reduces to `rsETHUnstaked == 0`, so any call with `rsETHUnstaked = 1` passes. Each such call:

1. Transfers 1 wei rsETH from the caller to the contract (line 166).
2. Computes `expectedAssetAmount â‰ˆ 1 wei` via `getExpectedAssetAmount` (line 168, using `amount * rsETHPrice / assetPrice`).
3. Increments `assetsCommitted[asset]` by 1 wei (line 173).
4. Pushes a new entry into the sequential nonce queue via `_addUserWithdrawalRequest`, advancing `nextUnusedNonce[asset]` (lines 744â€“759).

The queue is processed strictly in FIFO order in `_unlockWithdrawalRequests` (lines 790â€“815). The loop iterates from `nextLockedNonce_` to `firstExcludedIndex` and uses `break` (not `continue`) on the delay check and the asset-availability check, meaning operators cannot skip nonces. Every dust entry must be advanced through before `nextLockedNonce` reaches a legitimate user's entry.

The `getAvailableAssetAmount` ceiling (line 170) is `totalAssets - assetsCommitted[asset]`. With 1 wei committed per dust request and a TVL of, say, 100,000 ETH (â‰ˆ 10Â²Â³ wei), the ceiling is not a practical barrier. Gas cost per call (~100,000â€“200,000 gas) is the only real constraint, making thousands of dust entries economically feasible for a motivated attacker.

## Impact Explanation

Legitimate users who call `initiateWithdrawal` after the attacker's dust entries are assigned higher nonces. Their rsETH is locked in the contract and their expected assets are committed via `assetsCommitted`, but `unlockQueue` cannot advance `nextLockedNonce` past the dust entries until each one is individually processed. This constitutes **temporary freezing of funds** for all users who queue after the attack begins, matching the allowed Medium impact.

## Likelihood Explanation

The default value of `minRsEthAmountToWithdraw` is `0` for every asset. Any newly deployed or upgraded contract is immediately vulnerable until an admin explicitly calls `setMinRsEthAmountToWithdraw`. An attacker needs only a trivial amount of rsETH (purchasable on the open market) and pays only gas per request. The attack is repeatable and requires no privileged access, no oracle manipulation, and no victim mistakes.

## Recommendation

Set a non-zero `minRsEthAmountToWithdraw` for every supported asset inside the `initialize` function (or a dedicated initializer), rather than relying on a post-deployment admin call. Additionally, consider adding a per-user cap on the number of pending withdrawal requests (e.g., checking `userAssociatedNonces[asset][msg.sender].length()` before enqueuing) to bound queue depth independently of the minimum amount.

## Proof of Concept

1. Deploy or upgrade `LRTWithdrawalManager`; `minRsEthAmountToWithdraw[ETH_TOKEN]` is `0` (default â€” admin has not called `setMinRsEthAmountToWithdraw`).
2. Attacker acquires `N` wei of rsETH (e.g., `N = 10_000`) and approves the contract.
3. Attacker calls `initiateWithdrawal(ETH_TOKEN, 1, "")` in a loop `N` times. Each call succeeds because `1 != 0` and `1 >= 0`.
4. `nextUnusedNonce[ETH_TOKEN]` advances to `N`; `assetsCommitted[ETH_TOKEN]` increases by `N` wei.
5. Victim calls `initiateWithdrawal(ETH_TOKEN, 1 ether, "")` â€” their request is assigned nonce `N`.
6. After `withdrawalDelayBlocks` (â‰ˆ 8 days), operator calls `unlockQueue(ETH_TOKEN, N+1, ...)`. The while-loop must iterate through all `N` dust entries before reaching the victim's entry at nonce `N`, consuming proportional gas across multiple batched calls.
7. Victim's 1 ETH worth of rsETH remains locked in the contract until all `N` dust entries are processed.

**Foundry test sketch:**
```solidity
function testQueueStuffing() public {
    // Attacker mints/buys N wei rsETH
    uint256 N = 10_000;
    rsETH.mint(attacker, N);
    vm.startPrank(attacker);
    rsETH.approve(address(withdrawalManager), N);
    for (uint256 i = 0; i < N; i++) {
        withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");
    }
    vm.stopPrank();
    assertEq(withdrawalManager.nextUnusedNonce(ETH_TOKEN), N);

    // Victim queues a real withdrawal
    rsETH.mint(victim, 1 ether);
    vm.prank(victim);
    rsETH.approve(address(withdrawalManager), 1 ether);
    vm.prank(victim);
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1 ether, "");
    assertEq(withdrawalManager.nextUnusedNonce(ETH_TOKEN), N + 1);

    // Advance past delay
    vm.roll(block.number + withdrawalManager.withdrawalDelayBlocks() + 1);

    // Operator must process all N dust entries before victim's entry is reachable
    // unlockQueue with firstExcludedIndex = N+1 iterates all N+1 entries
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH_TOKEN, N + 1, ...);
    // Victim's nonce N is only unlocked after all N dust entries are processed
}
```
