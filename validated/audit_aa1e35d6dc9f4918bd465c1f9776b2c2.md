### Title
Mandatory Storage Deposits for Non-Executed Update Proposals Are Permanently Trapped in `MpcContract` — (File: `crates/contract/src/update.rs`)

---

### Summary

`propose_update` enforces a mandatory storage deposit from every proposer. When any competing update reaches threshold votes and `do_update` fires, it bulk-clears **all** pending entries — including those of other proposers — without issuing any refund. Because `UpdateEntry` never records the proposer's identity or deposit amount, the funds are irrecoverable. This is a direct structural analog to the UMA "Trapped Proposer Reward" pattern: funds are collected by the contract, never forwarded or returned, and permanently frozen.

---

### Finding Description

**Collection path — `propose_update` (`crates/contract/src/lib.rs:1298–1334`)**

Every call to `propose_update` enforces a deposit equal to `ProposedUpdates::required_deposit(&update)`, which is computed as `env::storage_byte_cost() × bytes_used(update)`. For a full contract binary this can be tens of NEAR. Only the *excess* above the required amount is refunded to the proposer; the required portion is retained by the contract.

```rust
// lib.rs:1308-1316
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
if attached < required {
    return Err(InvalidParameters::InsufficientDeposit { ... }.into());
}
// lib.rs:1327-1331  — only excess is returned
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

The `UpdateEntry` struct that is stored contains only the update payload and the byte count — **no proposer account ID, no deposit amount**:

```rust
// update.rs:132-135
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
```

**Destruction path — `do_update` (`crates/contract/src/update.rs:195–227`)**

When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry and then **bulk-clears every other pending entry and all votes** with no refund loop:

```rust
// update.rs:195-200
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
```

There is no cancel/withdraw endpoint anywhere in the contract API that would let a proposer reclaim their deposit before or after execution.

---

### Impact Explanation

Any participant who proposed a competing update loses their mandatory storage deposit permanently the moment a different update is executed. Because `UpdateEntry` omits the proposer's identity, the contract cannot issue refunds even if the logic were patched at the call site. The funds accumulate silently in `MpcContract`'s balance with no accounting entry and no withdrawal path. This breaks the production safety invariant that every deposit must either be consumed by the storage it covers or returned when that storage is freed — exactly the class of accounting corruption described in the reference report.

Impact classification: **Medium** — balance and contract execution-flow invariant broken; proposers' NEAR permanently frozen without any network-level DoS or operator misconfiguration required.

---

### Likelihood Explanation

The MPC governance model explicitly supports concurrent proposals: multiple participants can call `propose_update` independently before consensus is reached. In any upgrade cycle where participants disagree on which binary or config to apply, competing proposals will coexist. The moment the winning proposal clears threshold, all competing deposits are silently forfeited. This is a normal governance scenario, not an edge case.

---

### Recommendation

1. Add `proposer: AccountId` and `deposit: NearToken` fields to `UpdateEntry` so the contract can issue refunds.
2. In `do_update`, iterate over the entries being cleared and emit `Promise::new(entry.proposer).transfer(entry.deposit)` for each non-executed entry before calling `self.entries.clear()`.
3. Alternatively, implement a `cancel_update(id: UpdateId)` endpoint that lets a proposer withdraw their deposit while the proposal is still pending, and document that deposits are forfeited on execution.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 kB contract binary; required deposit ≈ 5 NEAR. Deposit is kept; only excess refunded.
2. Participant B calls `propose_update` with a different 500 kB binary; same deposit mechanics apply.
3. Threshold participants vote for A's update via `vote_update(id_A)`.
4. `vote_update` calls `do_update(&id_A, gas)`.
5. `do_update` removes entry A, then calls `self.entries.clear()` — entry B is deleted with no refund.
6. Participant B's ≈ 5 NEAR is now held by `MpcContract` with no accounting record and no recovery path.
7. Calling any contract method to retrieve the deposit fails — no such endpoint exists. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn required_deposit(update: &Update) -> NearToken {
        required_deposit(bytes_used(update))
    }

    pub fn propose(&mut self, update: Update) -> UpdateId {
        let bytes_used = bytes_used(&update);

        let id = self.id.generate();
        self.entries.insert(id, UpdateEntry { update, bytes_used });

        id
    }
