### Title
Mempool Evicts Excess Transactions Due to Shortfall Miscalculation in `handle_capacity_overflow` - (`File: crates/apollo_mempool/src/mempool.rs`)

---

### Summary

`handle_capacity_overflow` passes the full net byte size of the incoming transaction (`tx.total_bytes() - freed_bytes`) to `try_make_space`, instead of the actual shortfall (`(current_size + tx_bytes - freed_bytes) - capacity`). This causes the mempool to evict far more valid transactions than necessary whenever a new transaction is added near capacity.

---

### Finding Description

When a new transaction is submitted and the mempool is over capacity, `add_tx_validations` calls `handle_capacity_overflow`. Inside that function, the amount of space to free is computed as:

```rust
// crates/apollo_mempool/src/mempool.rs, line 1046
let required_space = tx.total_bytes().saturating_sub(freed_bytes);
``` [1](#0-0) 

This is the full net byte size of the incoming transaction, not the actual shortfall. The actual shortfall — the minimum number of bytes that must be freed to make room — is:

```
shortfall = (self.size_in_bytes() + tx.total_bytes() - freed_bytes) - capacity
```

The `exceeds_capacity` guard that gates this call already computes this correctly:

```rust
// crates/apollo_mempool/src/mempool.rs, line 940-944
fn exceeds_capacity(&self, tx: &InternalRpcTransaction, freed_bytes: u64) -> bool {
    (self.size_in_bytes() + tx.total_bytes()).saturating_sub(freed_bytes)
        > self.config.static_config.capacity_in_bytes
}
``` [2](#0-1) 

But `handle_capacity_overflow` ignores `self.size_in_bytes()` and `capacity` entirely when computing `required_space`, passing the inflated value to `try_make_space`:

```rust
// crates/apollo_mempool/src/mempool.rs, line 992-1028
pub fn try_make_space(&mut self, required_space: u64) -> bool {
    let mut total_space_freed = 0;
    ...
    while total_space_freed < required_space {
        // evict whole accounts until required_space is met
    }
    total_space_freed >= required_space
}
``` [3](#0-2) 

**Concrete example:**
- `capacity = 1000`, `current_size = 800`, `tx_bytes = 300`, `freed_bytes = 0`
- `exceeds_capacity` fires: `800 + 300 = 1100 > 1000` ✓
- Actual shortfall: `1100 - 1000 = 100` bytes
- Code evicts: `300 - 0 = 300` bytes → 3× excess eviction

After the eviction, `current_size` drops to `500`, and adding the new tx yields `800`, leaving 200 bytes of capacity wasted and 200 bytes of valid transactions unnecessarily removed.

---

### Impact Explanation

Every time a transaction is added to a near-full mempool, `try_make_space` evicts `tx.total_bytes()` bytes worth of existing transactions instead of only the shortfall. The excess eviction scales with the size of the incoming transaction relative to the actual overflow. An unprivileged attacker can submit a single large transaction to cause disproportionate eviction of other users' valid transactions, preventing them from being sequenced. This directly matches the allowed impact: **"Mempool/gateway/RPC admission … rejects valid transactions before sequencing."**

---

### Likelihood Explanation

Any user can submit a transaction to the mempool. The bug triggers whenever the mempool is not empty and the incoming transaction's byte size exceeds the available free space. In a production mempool that is regularly near capacity, this fires on nearly every admission. No special privileges or coordination are required.

---

### Recommendation

Replace the `required_space` calculation in `handle_capacity_overflow` with the actual shortfall:

```rust
// crates/apollo_mempool/src/mempool.rs
let required_space = (self.size_in_bytes() + tx.total_bytes())
    .saturating_sub(freed_bytes)
    .saturating_sub(self.config.static_config.capacity_in_bytes);
``` [4](#0-3) 

This mirrors the fix recommended in the original report: withdraw only the shortfall, not the full requested amount.

---

### Proof of Concept

1. Configure a mempool with `capacity_in_bytes = 1000`.
2. Fill it to 800 bytes with 8 transactions of 100 bytes each (all from gap accounts so they are evictable).
3. Submit a new transaction of 300 bytes (no replacement, `freed_bytes = 0`).
4. `exceeds_capacity` fires: `800 + 300 = 1100 > 1000`.
5. `handle_capacity_overflow` calls `try_make_space(300)`.
6. `try_make_space` evicts 3 × 100-byte transactions (300 bytes freed ≥ 300 required).
7. **Observed**: 3 valid transactions evicted; mempool ends at `500 + 300 = 800` bytes (200 bytes wasted).
8. **Expected**: only 1 transaction (100 bytes) should have been evicted; mempool should end at `700 + 300 = 1000` bytes (exactly at capacity).

The existing test `rejects_or_accepts_tx_based_on_freed_space` in `crates/apollo_mempool/src/fee_mempool_test.rs` (lines 1894–1938) exercises the eviction path but does not assert that the *minimum* number of transactions is evicted, so the over-eviction goes undetected. [5](#0-4)

    pub fn try_make_space(&mut self, required_space: u64) -> bool {
        let mut total_space_freed = 0;
        let mut evicted_txs = Vec::new();

        while total_space_freed < required_space {
            let Some(address) = self.get_evictable_account() else {
                break;
            };

            let txs: Vec<_> = self.tx_pool.account_txs_sorted_by_nonce(address).copied().collect();
            for tx_ref in txs.iter().rev() {
                let tx = self
                    .tx_pool
                    .remove(tx_ref.tx_hash)
                    .expect("Transaction must exist in the pool.");
                total_space_freed += tx.total_bytes();
                evicted_txs.push(*tx_ref);
                metric_count_evicted_txs(1);
                self.decrement_stuck_txs_if_gap_account(address, 1);
                if total_space_freed >= required_space {
                    break;
                }
            }

            // Clean up if account is now empty.
            if !self.tx_pool.contains_account(address) {
                self.accounts_with_gap.swap_remove(&address);
            }
        }

        // Keep the queue consistent with the pool: the evicted txs were removed from the pool, so
        // drop their (now-orphaned) queue references too. In fee mode gap accounts are never
        // queued, so this is a no-op there; in Echonet/FIFO mode they are, and skipping this leaves
        // a dangling reference that panics the next `get_txs`.
        self.tx_queue.remove_txs(&evicted_txs);

        total_space_freed >= required_space

```rust
    fn handle_capacity_overflow(
        &mut self,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
        freed_bytes: u64,
    ) -> Result<(), MempoolError> {
        let address = tx.contract_address();

        let account_has_gap = self.accounts_with_gap.contains(&address);
        let account_has_txs = self.tx_pool.contains_account(address);
        let closing_gap = tx.nonce() == account_nonce;
        let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;

        // Only the net growth must be evicted: an accompanying replacement removal frees
        // `freed_bytes` (0 when there is no replacement).
        let required_space = tx.total_bytes().saturating_sub(freed_bytes);
        if !creating_gap && self.try_make_space(required_space) {
            return Ok(());
        }

        Err(MempoolError::MempoolFull)
    }
```
