### Title
Chainlink Price Feed Staleness Not Validated, Enabling Oracle Rate Abuse — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` and `answeredInRound` return values. The protocol accepts any price the feed returns — fresh or arbitrarily stale — without any freshness check. This is the direct on-chain analog of the SSL pinning bug: just as the mobile app trusts any CA-signed certificate without pinning to a specific trusted certificate, this contract trusts any price returned by the Chainlink aggregator without pinning to a freshness constraint.

---

### Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol`, the `getAssetPrice()` function fetches the latest round data from a Chainlink aggregator:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`latestRoundData()` returns five values: `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The implementation discards `updatedAt` (the timestamp of the last price update) and `answeredInRound` (the round in which the answer was computed). No staleness threshold is enforced, and no check verifies that `answeredInRound >= roundId`. A price that has not been updated for hours, days, or indefinitely is accepted as valid.

This stale price propagates through the entire pricing stack:

1. `ChainlinkPriceOracle.getAssetPrice(asset)` → returns stale price
2. `LRTOracle._getTotalEthInProtocol()` → multiplies stale price by total asset deposits to compute TVL
3. `LRTOracle._updateRsETHPrice()` → derives `newRsETHPrice` from stale TVL
4. `LRTDepositPool.getRsETHAmountToMint()` → uses `lrtOracle.getAssetPrice(asset) / lrtOracle.rsETHPrice()` to determine how many rsETH tokens to mint per deposited asset unit

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield / protocol insolvency**

If a supported LST's Chainlink feed goes stale at an **inflated** price:
- `_getTotalEthInProtocol()` overstates TVL
- `rsETHPrice` is computed too high
- New depositors receive fewer rsETH than they are entitled to; existing holders are enriched at depositors' expense (yield theft from depositors)

If a supported LST's Chainlink feed goes stale at a **deflated** price:
- `_getTotalEthInProtocol()` understates TVL
- `rsETHPrice` is computed too low
- New depositors receive excess rsETH relative to their deposit value, diluting all existing rsETH holders (theft of yield from existing holders, potential insolvency)

The `pricePercentageLimit` guard in `_updateRsETHPrice()` only triggers if the computed new price deviates from `highestRsethPrice` by more than the configured threshold. A gradually drifting stale price — or a stale price that was already near the current price at the time of staleness — bypasses this guard entirely.

---

### Likelihood Explanation

**Likelihood: Medium**

Chainlink feeds can go stale due to:
- Network congestion preventing oracle updates
- Chainlink node operator failures
- A feed being deprecated or paused without the protocol updating its configuration

The protocol supports multiple LST assets (stETH, ETHx, rETH, sfrxETH, swETH), each with its own price feed. The probability that at least one feed experiences a staleness event over the protocol's lifetime is non-trivial. No privileged action is required by the attacker — any depositor calling `depositAsset()` or `depositETH()` during a staleness window triggers the mispricing.

---

### Recommendation

Add staleness and round-completeness validation in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= STALENESS_THRESHOLD, "Stale price");
```

Set `STALENESS_THRESHOLD` per feed based on the feed's documented heartbeat (e.g., 3600 seconds for a 1-hour heartbeat feed, with a small buffer).
