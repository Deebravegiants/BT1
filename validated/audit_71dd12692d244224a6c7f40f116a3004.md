### Title
Missing `signer_id == receiver_id` Ownership Check in `WithdrawFromGasKey` Allows Any Account to Drain Another Account's Gas Key Balance - (File: `runtime/runtime/src/access_keys.rs`)

### Summary
`WithdrawFromGasKey` is intended as a self-service action for an account owner to reclaim funds from their own gas key. However, neither the action validator nor the runtime enforces that the transaction signer is the account owner (`signer_id == receiver_id`). Any unprivileged account holding a full-access key can send a transaction targeting a victim's account with a `WithdrawFromGasKey` action, draining the victim's gas key balance to zero without authorization. This corrupts the `gas_key_info.balance` trie entry on the victim's account and permanently disables the victim's gas key for transaction submission until manually re-funded — at which point the attacker can repeat the drain.

### Finding Description
`WithdrawFromGasKeyAction` carries an attacker-controlled `public_key` field and an `amount`. When a transaction is submitted with `signer_id = attacker` and `receiver_id = victim`, the runtime converts it to an action receipt that executes on the victim's account. The action handler `action_withdraw_from_gas_key` receives `account_id = victim` and looks up `action.public_key` on the victim's account:

The action validator for `WithdrawFromGasKey` performs only a protocol-feature gate — no check that `signer_id == receiver_id`:

```rust
fn validate_withdraw_from_gas_key_action(
    current_protocol_version: ProtocolVersion,
) -> Result<(), ActionsValidationError> {
    require_protocol_feature(ProtocolFeature::GasKeys, "GasKeys", current_protocol_version)?;
    Ok(())
}
``` [2](#0-1) 

The action dispatch in `lib.rs` passes `account_id` (the receipt receiver, i.e., the victim) directly to the handler with no ownership assertion: [3](#0-2) 

The action comment explicitly states the intent is self-service only:

> "This action must only be available via transactions, not via contract execution (there is no corresponding promise batch action host function)." [4](#0-3) 

But "only via transactions" does not enforce "only on the signer's own account." The missing invariant is `transaction.signer_id() == transaction.receiver_id()`.

### Impact Explanation
The attacker drains `gas_key_info.balance` on the victim's gas key to zero. The corrupted trie value is the `AccessKey` entry for `(victim_account_id, victim_gas_key_public_key)` — specifically its `GasKeyInfo.balance` field. The funds are not stolen (they move to the victim's main account balance), but the gas key becomes unusable: all subsequent gas-key transactions from the victim are rejected with `NotEnoughGasKeyBalance`. The `PendingTransactionQueue` also does not correctly account for cross-account `WithdrawFromGasKey` actions (it keys withdrawals by `(signer_id, public_key)`, not `(receiver_id, public_key)`), so the PTQ's gas-key balance enforcement for the victim's key is bypassed. [5](#0-4) 

The attacker can repeat the drain every time the victim re-funds their gas key, creating a persistent DoS on the victim's gas key transaction capability.

### Likelihood Explanation
Any account with a full-access key (the default for all NEAR accounts) can execute this attack against any other account that has a funded gas key. No special privileges are required. The attacker pays only the gas fee for the transaction. The attack is cheap, repeatable, and requires no coordination.

### Recommendation
Add a check in `validate_withdraw_from_gas_key_action` (or in the transaction verifier) that `transaction.signer_id() == transaction.receiver_id()` when a `WithdrawFromGasKey` action is present. This enforces the self-service intent documented in the action's comment and mirrors the ownership model of other self-modifying actions.

### Proof of Concept
1. Bob creates a gas key (`bob_gas_key`) on his account and funds it with 1 NEAR.
2. Alice (attacker) constructs a transaction:
   - `signer_id = alice.near`
   - `receiver_id = bob.near`
   - `actions = [WithdrawFromGasKey { public_key: bob_gas_key, amount: 1_NEAR }]`
   - Signed with Alice's full-access key.
3. Alice submits the transaction via RPC.
4. The transaction passes all validation (signature valid, Alice's access key exists, protocol feature enabled).
5. The action receipt executes on Bob's account: `gas_key_info.balance` is set to 0, Bob's main account balance increases by 1 NEAR.
6. Bob's gas key is now drained. Any gas-key transaction Bob submits is rejected with `NotEnoughGasKeyBalance`.
7. Alice repeats step 2–6 whenever Bob re-funds his gas key.

pub(crate) fn action_withdraw_from_gas_key(
    state_update: &mut TrieUpdate,
    account: &mut Account,
    result: &mut ActionResult,
    account_id: &AccountId,
    action: &WithdrawFromGasKeyAction,
) -> Result<(), RuntimeError> {
    let Some(mut access_key) = get_access_key(state_update, account_id, &action.public_key)? else {
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };
    let Some(gas_key_info) = access_key.gas_key_info_mut() else {
        // Key exists but is not a gas key
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };

    let Some(updated_balance) = gas_key_info.balance.checked_sub(action.amount) else {
        result.result = Err(ActionErrorKind::InsufficientGasKeyBalance {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
            balance: gas_key_info.balance,
            required: action.amount,
        }
        .into());
        return Ok(());
    };
    gas_key_info.balance = updated_balance;
    set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);

    let new_account_balance = account.amount().checked_add(action.amount).ok_or_else(|| {
        RuntimeError::StorageError(StorageError::StorageInconsistentState(
            "Account balance integer overflow".to_string(),
        ))
    })?;
    account.set_amount(new_account_balance);
    Ok(())
