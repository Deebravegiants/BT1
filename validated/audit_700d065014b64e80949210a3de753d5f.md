### Title
OmniBridgeWormhole: Excess ETH Not Returned to Caller When Overpaying Wormhole Message Fee — (`evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

### Summary

`OmniBridgeWormhole` contains three internal extension functions — `deployTokenExtension`, `logMetadataExtension`, and `finTransferExtension` — that each forward the caller's entire `msg.value` directly to `_wormhole.publishMessage`. The Wormhole `messageFee()` is a dynamic value that can change. When a caller sends more ETH than the current `messageFee()` (a rational defensive strategy to ensure the transaction succeeds), the excess is consumed by Wormhole and never returned to the caller.

### Finding Description

In `OmniBridgeWormhole.sol`, three extension hooks unconditionally forward `msg.value` to Wormhole:

- `deployTokenExtension` (called from the public `deployToken`): [1](#0-0) 

- `logMetadataExtension` (called from public `logMetadata` / `logMetadata1155`): [2](#0-1) 

- `finTransferExtension` (called from public `finTransfer`): [3](#0-2) 

In all three cases, `{value: msg.value}` is passed to `publishMessage`. The Wormhole contract only requires `messageFee()` worth of ETH; any surplus is retained by Wormhole and not returned to the caller. None of these three paths contain a check of the form `msg.value == _wormhole.messageFee()`, nor any refund of `msg.value - messageFee()` back to `msg.sender`.

The same structural issue exists in `initTransfer` for ERC20 tokens, where `extensionValue = msg.value - nativeFee` is forwarded to Wormhole: [4](#0-3) 

If `msg.value > nativeFee + messageFee()`, the surplus is silently consumed by Wormhole.

### Impact Explanation

Any ETH sent above `messageFee()` is routed to Wormhole instead of being returned to the caller. This is a **fee-routing / refund divergence** that sends value to the wrong party. The caller permanently loses the excess ETH with no on-chain recourse. This maps to the allowed High impact category: *"fee-routing, refund, or balance-accounting divergence … that sends value to the wrong party."*

### Likelihood Explanation

The Wormhole `messageFee()` is a governance-controlled dynamic value. Callers who query the fee off-chain and then add a buffer to guarantee execution (the same rational behavior described in the reference report) will overpay. Because `finTransfer`, `deployToken`, `logMetadata`, and `logMetadata1155` are all public `payable` entrypoints reachable by any unprivileged caller, the path is fully attacker- (or user-) reachable without any privileged role.

### Recommendation

After calling `_wormhole.publishMessage`, compute the actual fee consumed and refund the remainder:

```solidity
uint256 fee = _wormhole.messageFee();
_wormhole.publishMessage{value: fee}(wormholeNonce, payload, _consistencyLevel);
uint256 excess = msg.value - fee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "ETH refund failed");
}
```

Apply this pattern to `deployTokenExtension`, `logMetadataExtension`, and `finTransferExtension`. For `initTransferExtension`, enforce `value == _wormhole.messageFee()` or refund `value - messageFee()` to the original sender.

### Proof of Concept

1. Wormhole governance sets `messageFee()` to 0.01 ETH.
2. A relayer calls `finTransfer(signatureData, payload)` with `msg.value = 0.02 ETH` (2× the fee, to hedge against a fee increase between query and execution).
3. `finTransferExtension` executes `_wormhole.publishMessage{value: 0.02 ETH}(...)`.
4. Wormhole consumes 0.01 ETH for the fee and retains the remaining 0.01 ETH.
5. `OmniBridgeWormhole` never refunds the 0.01 ETH surplus to the relayer.
6. The relayer permanently loses 0.01 ETH per call with no recovery path. [5](#0-4) [6](#0-5)
