### Title
Stale Votes from Removed Participants Counted in TEE Governance Threshold Checks — (File: crates/contract/src/tee/measurements.rs)

---

### Summary

`MeasurementVotes::count_votes` tallies every entry in `vote_by_account` without filtering for current participants. After a resharing that removes participants, their pending votes for OS measurements (and launcher/docker-image hashes) remain in the map until a detached cleanup promise executes in the next block. During that inter-block window a remaining Byzantine participant can cast the final vote for a malicious OS measurement, and the stale votes from the removed participants inflate the count past the new (lower) threshold — adding the measurement without the required number of current-participant approvals.

---

### Finding Description

`MeasurementVotes::count_votes` in `crates/contract/src/tee/measurements.rs` counts every entry in `vote_by_account` that matches the action, with no filter on whether the voter is still a current participant:

```rust
fn count_votes(&self, action: &MeasurementVoteAction) -> u64 {
    u64::try_from(
        self.vote_by_account
            .values()
            .filter(|a| *a == action)
            .count(),
    )
    .expect("participant count should not overflow u64")
}
``` [1](#0-0) 

This is called by `MeasurementVotes::vote`, which is called by `TeeState::vote_measurement`, which is called by `vote_add_os_measurement` and `vote_remove_os_measurement`:

```rust

if votes >= self.threshold()?.value() {
    let added = self.tee_state.add_measurement(measurement);
``` [2](#0-1) 

After resharing completes in `vote_reshared`, the new `Running` state (with the reduced participant set and lower threshold) is committed immediately, but the stale-vote cleanup is only scheduled as a **detached promise** that executes in a subsequent receipt:

```rust
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        ...
    )
    .detach();
``` [3](#0-2) 

`clean_tee_status` calls `TeeState::clean_non_participant_votes`, which replaces the vote maps with filtered copies:

```rust
pub fn clean_non_participant_votes(&mut self, participants: &Participants) {
    self.votes = self.votes.get_remaining_votes(participants);
    self.launcher_votes = self.launcher_votes.get_remaining_votes(participants);
    self.measurement_votes = self.measurement_votes.get_remaining_votes(participants);
}
``` [4](#0-3) 

Between the resharing receipt and the cleanup receipt, `vote_by_account` still contains entries from removed participants. Any call to `vote_add_os_measurement` during that window counts those stale entries.

Contrast this with `ProviderVotes::vote` in `foreign_chain_rpc.rs`, which explicitly filters for current participants at count time:

```rust
voter_set.count_for(|(p, c)| {
    *c == chain && participants.is_participant_given_participant_id(&p.get())
})
``` [5](#0-4) 

`MeasurementVotes` (and `CodeHashesVotes`, `LauncherHashVotes`) lack this per-count participant filter.

**Concrete attack scenario (N=5, T=3 → N=2, T=2):**

1. Pre-resharing: Byzantine participant P1 calls `vote_add_os_measurement(M)`. Count = 1 < 3; measurement not applied.
2. Resharing removes P1, P2, P3. New set: {P4, P5}, new threshold T′ = 2. State is committed; cleanup promise is queued but not yet executed.
3. In the same block (before the cleanup receipt runs), Byzantine participant P4 calls `vote_add_os_measurement(M)`.
4. `count_votes` tallies P1's stale entry + P4's new entry = 2 ≥ T′ = 2. Measurement M is added to `allowed_measurements`.

The codebase's own test documents this exact stale-vote risk:

```
/// Stale CodeHashesVotes entries from removed participants must not count toward
/// quorum after resharing.
/// Scenario (N=5, T=3):
/// 1. P1 and P2 vote for malicious hash before resharing.
/// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
/// 3. clean_non_participant_votes removes stale votes.
/// 4. P3 votes for the same hash — only 1 vote, not 3.
``` [6](#0-5) 

The test verifies the cleanup path works correctly, but does not test the window before cleanup runs.

### Impact Explanation

OS measurements (MRTD, RTMR0-2, key-provider event digest) are the TEE identity gate: `submit_participant_info` accepts an attestation only if its measurements appear in `allowed_measurements`. Adding a malicious measurement M allows a node running a compromised OS image to pass attestation verification and be accepted as a valid TEE participant.
