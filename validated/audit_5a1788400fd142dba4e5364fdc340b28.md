### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions Based on Overly Broad Mempool Presence Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is intended to skip the account's `__validate__` entry point only when a `deploy_account` transaction for the sender is pending in the mempool (UX improvement for atomic deploy+invoke flows). However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the address, not specifically a `deploy_account` transaction. An unprivileged attacker who observes a pending `deploy_account` transaction for address A in the mempool can submit an invoke transaction with nonce=1 from address A carrying an arbitrary/invalid signature. The gateway will skip `__validate__` and admit the transaction to the mempool without any signature verification.

---

When `skip_validate = true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false` and calls `blockifier_validator.validate(account_tx)` — which returns `Ok(())` immediately without invoking `__validate__`: [2](#0-1) 

The guard `account_tx_in_pool_or_recent_block` is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`tx_pool.contains_account` checks for **any** transaction type from the address: [4](#0-3) 

The code comment acknowledges the assumption: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This assumption is false. A `deploy_account` transaction submitted by the legitimate owner of address A satisfies the check, but the check does not distinguish it from any other transaction type. [5](#0-4) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can submit an invoke transaction from an address they do not control (no valid private key) with an arbitrary signature. The gateway admits it to the mempool without running `__validate__`. Concrete consequences:

1. **Unsigned invoke transactions enter the mempool.** The batcher will execute them and they will revert (the blockifier runs `__validate__` during sequencing), but they consume block bouncer capacity and produce reverted receipts.
2. **Griefing / nonce-slot squatting.** If the attacker submits before the legitimate user's nonce=1 invoke, the mempool's duplicate-nonce rejection (or fee-escalation replacement) blocks or displaces the legitimate transaction. The legitimate user's nonce=1 invoke is either rejected or must escalate fees to reclaim the slot.

---

### Likelihood Explanation

**Medium.** The attacker only needs to:
1. Monitor the public mempool for pending `deploy_account` transactions (observable via the P2P gossip layer or gateway).
2. Construct an invoke transaction with `nonce=1` from the target address and any signature bytes.
3. Submit it to the gateway before the legitimate user submits their nonce=1 invoke.

No privileged access, no cryptographic material for the target account, and no special network position are required.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. Add a dedicated mempool query such as `deploy_account_tx_in_pool(sender_address)` that inspects the transaction type stored in `tx_pool`, and use that in `skip_stateful_validations` instead of the generic presence check.

Alternatively, mirror the `native_blockifier` approach in `PyValidator::should_run_stateful_validations`, which requires the caller to explicitly supply the `deploy_account_tx_hash` as proof — making the skip conditional on a concrete, caller-supplied hash rather than an indirect mempool presence signal. [6](#0-5) 

---

### Proof of Concept

1. Legitimate user submits `deploy_account` for address `A` (nonce=0). It enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker calls the gateway `add_tx` with an `Invoke V3` transaction: `sender_address=A`, `nonce=1`, `signature=[0x0, 0x0]` (invalid).
3. Gateway `run_pre_validation_checks`:
   - `validate_nonce`: `account_nonce=0`, `tx_nonce=1`, within `max_allowed_nonce_gap` → passes. [7](#0-6) 
   - `validate_by_mempool`: checks duplicate hash and nonce range only, no signature check → passes. [8](#0-7) 
   - `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` → returns `true`. [9](#0-8) 
4. `run_validate_entry_point` is called with `skip_validate=true`; `execution_flags.validate=false`; `__validate__` is never invoked. The transaction is admitted to the mempool.
5. If the attacker submits with a higher tip than the legitimate user's nonce=1 invoke, fee escalation replaces the legitimate transaction. The attacker's transaction executes, `__validate__` runs in the blockifier and fails (invalid signature), the transaction reverts, and the legitimate user's nonce=1 invoke is permanently lost from the mempool.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
