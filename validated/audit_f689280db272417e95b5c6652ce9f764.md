### Title
Unanimity Requirement for `vote_new_parameters` and `vote_add_domains` Exceeds Governance Threshold, Enabling Single-Participant Governance Freeze - (File: `crates/contract/src/state/running.rs`)

### Summary

Both `vote_new_parameters` and `vote_add_domains` require **unanimity** (all `n` proposed/current participants) to trigger a state transition, while the governance threshold is explicitly set between 60% and 100% of `n`. A single Byzantine participant strictly below the signing threshold can permanently block all governance operations — preventing participant-set changes and new domain additions — by simply refusing to vote.

### Finding Description

In `RunningContractState::process_new_parameters_proposal`, the resharing transition fires only when every proposed participant has voted:

```rust
let n_votes = self.parameters_votes.vote(proposal, candidate);
Ok(new_num_participants == n_votes)   // unanimity required
``` [1](#0-0) 

Similarly, `vote_add_domains` transitions only when every current participant has voted:

```rust
if self.parameters.participants().len() as u64 == n_votes {
``` [2](#0-1) 

The governance threshold, however, is validated to be between 60% and 100% of `n`:

```rust
pub(crate) fn governance_threshold_lower_relative_bound(n: u64) -> u64 {
    3_u64.saturating_mul(n).div_ceil(5)   // 60% rounded up
}

/// Currently set to 100% of participants but would be a discussion subject
/// to drop this upper bound down not to have problems with smart contract
/// being locked if t = n and if an operator stops voting
pub(crate) fn governance_threshold_upper_relative_bound(n: u64) -> u64 {
    n
}
``` [3](#0-2) 

The code's own comment acknowledges the locking risk when `t = n`. The mismatch is structural: the governance threshold (the intended security parameter, ≥60% of `n`) is validated and stored, but the actual quorum enforced for the two most critical governance operations is always 100% of `n` — unanimity. The ABI documentation compounds this by stating "If a threshold number of votes are reached on the exact same proposal, this will transition the contract into the Resharing state," which is inconsistent with the unanimity check in the implementation. [4](#0-3) 

### Impact Explanation

A single participant who refuses to call `vote_new_parameters` or `vote_add_domains` permanently blocks:

1. **Participant-set changes** — the network cannot add or remove any participant, including removing a compromised node.
2. **New domain additions** — no new signing keys (ECDSA, EdDSA, CKD) can be added via `vote_add_domains`.

This breaks the production safety invariant that the governance threshold (not unanimity) is sufficient for governance decisions. Existing signing operations in the `Running` state continue, but the governance layer is permanently frozen. If the blocking participant's key is later compromised, the network cannot remove them via resharing because the compromised participant can refuse to vote for any proposal that includes them (or any proposal at all for `vote_add_domains`).

This matches the allowed impact: **Medium — participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

### Likelihood Explanation

A single participant going offline, being decommissioned without coordination, or acting adversarially is a realistic operational scenario. No collusion, no key leakage, and no privileged access are required — the attacker simply does not submit a transaction. The participant set in production is small (order of tens of nodes), making any single node's absence impactful. The governance threshold being set below `n` (e.g., 60% of `n`) creates an explicit expectation that governance should tolerate some non-participation, but the unanimity requirement violates this expectation.

### Recommendation

Replace the unanimity check in `process_new_parameters_proposal` and `vote_add_domains` with the governance threshold:

- In `process_new_parameters_proposal`, change `Ok(new_num_participants == n_votes)` to compare `n_votes` against `proposal.threshold().value()` (the proposed governance threshold), counting only votes from current participants (old epoch) separately from new-participant votes.
- In `vote_add_domains`, change the unanimity check `self.parameters.participants().len() as u64 == n_votes` to compare against `self.parameters.threshold().value()`.
- Ensure the governance threshold lower bound (60%) is sufficient to prevent a minority from unilaterally triggering governance transitions.
- Update the ABI documentation to accurately reflect the actual quorum requirement.

### Proof of Concept

**Setup:** A Running contract with `n = 5` participants and governance threshold `t = 3` (60%).

**Attack:**
1. Participants P1–P4 agree on a `vote_new_parameters` proposal (e.g., to add a new participant or change the threshold) and each call `vote_new_parameters`. Four votes are cast; `n_votes = 4`.
2. The check `new_num_participants == n_votes` evaluates `5 == 4` → `false`. No transition.
3. P5 refuses to call `vote_new_parameters` (offline, decommissioned, or adversarial).
4. The resharing never triggers. The governance threshold of 3 is irrelevant — unanimity is required.
5. The same applies to `vote_add_domains`: P5 refusing to vote blocks all new domain additions indefinitely.

The contract remains in `Running` state with the old participant set, unable to evolve, regardless of how many of the other participants agree. [5](#0-4) [6](#0-5) [7](#0-6)

    /// Casts a vote for `proposal`, removing any previous votes by `env::signer_account_id()`.
    /// Fails if the proposal is invalid or the signer is not a proposed participant.
    /// Returns true if all participants of the proposed parameters voted for it.
    pub(super) fn process_new_parameters_proposal(
        &mut self,
        proposal: &ProposedThresholdParameters,
    ) -> Result<bool, Error> {
        // ensure the proposal is valid against the current parameters
        self.parameters
            .validate_incoming_proposal(proposal.parameters())?;

        // Validate effective per-domain thresholds (updates override, absent
        // domains keep theirs) against the proposed participant count.
        let new_num_participants =
            u64::try_from(proposal.participants().len()).expect("participant count fits in u64");
        let threshold_updates = proposal.per_domain_thresholds();
        // Reject unknown domain IDs: the loop below iterates existing domains, so
        // an unknown ID would otherwise be silently ignored here (it's caught at
        // the resharing transition, but we fail fast at vote acceptance).
        for id in threshold_updates.keys() {
            if self.domains.get_domain_by_domain_id(*id).is_none() {
                return Err(DomainError::UnknownDomainInProposal { domain_id: *id }.into());
            }
        }
        let effective_domains: Vec<DomainConfig> = self
            .domains
            .domains()
            .iter()
            .map(|domain| {
                let effective_threshold = threshold_updates
                    .get(&domain.id)
                    .copied()
                    .unwrap_or(domain.reconstruction_threshold);
                DomainConfig {
                    reconstruction_threshold: effective_threshold,
                    ..domain.clone()
                }
            })
            .collect();
        for domain in &effective_domains {
            validate_domain_threshold(domain, new_num_participants)?;
        }

        // The GovernanceThreshold must dominate every domain's effective ReconstructionThreshold;
        // enforced here so the state transition is self-contained (single source of truth).
        ThresholdParameters::validate_governance_against_reconstruction(
            new_num_participants,
            proposal.threshold(),
            max_reconstruction_threshold(&effective_domains),
        )?;

        // ensure the signer is a proposed participant
        let candidate = AuthenticatedAccountId::new(proposal.participants())?;

        // If the signer is not a participant of the current epoch, they can only vote after
        // `threshold` participant of the current epoch have casted their vote to admit them.
        if AuthenticatedAccountId::new(self.parameters.participants()).is_err() {
            let n_votes = self
                .parameters_votes
                .n_votes(proposal, self.parameters.participants());
            if n_votes < self.parameters.threshold().value() {
                return Err(VoteError::VoterPending.into());
            }
        }

        // finally, vote.
        let n_votes = self.parameters_votes.vote(proposal, candidate);
        Ok(new_num_participants == n_votes)
    }
