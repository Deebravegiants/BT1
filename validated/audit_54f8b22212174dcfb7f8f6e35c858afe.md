Audit Report

## Title
FIFO Queue Head Blocking Causes Temporary Freezing of Subsequent Withdrawal Requests - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`_unlockWithdrawalRequests` processes the withdrawal queue in strict FIFO order and unconditionally `break`s when the head-of-queue request requires more assets than the vault currently holds. Any unprivileged rsETH holder can place a large withdrawal request early in the queue, causing all subsequent smaller requests â€” which could individually be fulfilled â€” to remain locked until the vault accumulates enough liquid assets to cover the large request first.

## Finding Description

`_unlockWithdrawalRequests` (L770â€“816) iterates from `nextLockedNonce[asset]` forward. At L800, when the current request's `payoutAmount` exceeds `availableAssetAmount`, the loop unconditionally breaks:

```solidity
if (availableAssetAmount < payoutAmount) break; // Exit if not enough assets to cover this request
```

`availableAssetAmount` is sourced from `unstakingVault.balanceOf(asset)` at the time `unlockQueue` is called (L849), which reflects only the vault's current liquid balance â€” typically a fraction of total committed assets due to EigenLayer's multi-day unstaking delay.

Any rsETH holder can call `initiateWithdrawal` (L150â€“178), which is unrestricted. The only guard at L170 prevents committing more than the currently uncommitted total asset pool:

```solidity
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
```

This does not prevent a large request from being placed at the front of the nonce sequence. Once such a request occupies `nextLockedNonce[asset]`, every subsequent `unlockQueue` call hits the `break` immediately. `nextLockedNonce[asset]` never advances, so all later requests remain locked. Users behind the blocker who call `completeWithdrawal` hit the check at L707:

```solidity
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```

There is no mechanism to skip, reorder, or bypass an unfulfillable head-of-queue request.

## Impact Explanation

All withdrawal requests queued after the attacker's large request are temporarily frozen. Users with small, individually-fulfillable requests cannot call `completeWithdrawal` because `nextLockedNonce` never advances past the blocking request. The freeze persists until the vault's liquid balance reaches the attacker's full `payoutAmount`. Given EigenLayer's 7-day+ unstaking cycle, this translates to a multi-day to multi-week delay for all queued users behind the blocker. **Impact: Medium â€” Temporary freezing of funds.**

## Likelihood Explanation

Any rsETH holder can call `initiateWithdrawal` without any special privileges. The attacker does not lose funds; they eventually receive their full withdrawal. The only cost is locking rsETH for the withdrawal delay period. The attack is most effective when the attacker places the request before other users or when the queue is short. **Likelihood: Medium.**

## Recommendation

Replace the unconditional `break` at L800 with a `continue` so that smaller requests behind an unfulfillable large request can still be unlocked. If strict ordering is required for accounting reasons, introduce a per-request "skipped" flag and allow the operator to explicitly advance `nextLockedNonce` past a request that cannot currently be fulfilled. A complementary mitigation is to cap the maximum single withdrawal size relative to the vault's expected liquid balance.
