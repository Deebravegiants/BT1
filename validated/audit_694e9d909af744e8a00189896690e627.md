### Title
Single Attested Participant Can Submit Unverified CKD Response for `AppPublicKey` Variant, Bypassing Threshold Requirement - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract enforces a BLS12-381 pairing-based output check (`ckd_output_check`) for the `AppPublicKeyPV` (publicly verifiable) variant of CKD requests, but performs **no cryptographic verification whatsoever** for the `AppPublicKey` (privately verifiable) variant. A single malicious attested participant can call `respond_ckd` with an arbitrary fabricated `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will accept it, drain all queued yields with the fabricated output, and deliver a wrong confidential key to the requesting TEE application.

---

### Finding Description

In `respond_ckd`, after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
// crates/contract/src/lib.rs lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically ties the response to the MPC network's master secret key and the user's ephemeral key pair. [2](#0-1) 

For `AppPublicKey`, the arm is completely empty — any `(big_y, big_c)` pair is accepted without verification.

After the match, `resolve_yields_for` immediately drains **all** queued yields for the request key with the supplied response bytes, meaning the first node to respond wins and all pending callers receive that response: [3](#0-2) [4](#0-3) 

The `AppPublicKey` variant is the "privately verifiable" (legacy) variant, still actively supported and used in production: [5](#0-4) 

---

### Impact Explanation

The CKD protocol's security guarantee is that no single participant can unilaterally determine the confidential key derivation output — threshold cooperation is required. For `AppPublicKeyPV`, this guarantee is enforced on-chain via the pairing check. For `AppPublicKey`, it is not enforced at all.

A single malicious attested participant can:
1. Observe a pending `AppPublicKey` CKD request in the contract state.
2. Race to call `respond_ckd` with a fabricated `CKDResponse{big_y: arbitrary, big_c: arbitrary}` before honest nodes respond.
3. The contract accepts the fabricated response (empty match arm), drains all queued yields, and delivers the wrong `(big_y, big_c)` to the requesting TEE application.
4. The TEE app decrypts `big_c - big_y * a` and obtains a wrong confidential key — one that is not `hash_point * msk` as the protocol requires.

This is a **bypass of the threshold-signature/CKD requirement**: a single participant (strictly below threshold) can unilaterally corrupt confidential key derivation output without the required participant authorization. The TEE application receives a key that is not the deterministic secret it expects, breaking the core security property of the CKD feature.

**Impact classification: Critical** — unauthorized confidential key derivation output without the required participant authorization (threshold bypass).

---

### Likelihood Explanation

- Any single attested participant can perform this attack; no collusion above threshold is needed.
- The attacker only needs to submit their fabricated `respond_ckd` transaction before honest nodes do. In a network with a small number of participants, this is a realistic race condition.
- The `AppPublicKey` variant is the legacy/default variant and is actively used (the `ckd-example-cli` README demonstrates it as the primary flow). [6](#0-5) 

---

### Recommendation

1. **Require `AppPublicKeyPV` for all new CKD requests** and deprecate `AppPublicKey`. The `AppPublicKeyPV` variant was specifically designed to allow on-chain verification and closes this gap.
2. If `AppPublicKey` must be retained, add a threshold-based consensus mechanism: require at least `threshold` distinct attested participants to submit matching `(big_y, big_c)` values before resolving the yield, analogous to how threshold signatures require threshold cooperation before a signature is accepted.
3. At minimum, document clearly that `AppPublicKey` CKD responses are not verified on-chain and that users of this variant must independently verify the response after decryption.

---

### Proof of Concept

```rust
// Attacker is an attested participant.
// A user has submitted: contract.request_app_private_key(CKDRequestArgs {
//     app_public_key: CKDAppPublicKey::AppPublicKey(user_app_pk),
//     derivation_path: "mykey",
//     domain_id: bls_domain_id,
// });

// Attacker races to call respond_ckd with fabricated output:
let fabricated_response = CKDResponse {
    big_y: Bls12381G1PublicKey([0xde; 48]),  // arbitrary garbage
    big_c: Bls12381G1PublicKey([0xad; 48]),  // arbitrary garbage
};

// The contract's AppPublicKey arm is empty — no check performed.
// resolve_yields_for drains all pending yields with the fabricated response.
contract.respond_ckd(pending_ckd_request, fabricated_response);
// → User's TEE app receives wrong (big_y, big_c), derives wrong confidential key.
```

The existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` confirms this: it passes `big_y: [1u8; 48], big_c: [2u8; 48]` (not a valid protocol output) and the contract accepts it without complaint. [7](#0-6)

```rust
/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
