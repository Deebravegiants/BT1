### Title
Stale TEE Governance Votes from Removed Participants Count Toward Threshold After Failed Resharing Cleanup - (`crates/contract/src/lib.rs`)

### Summary

After a resharing that removes participants, `vote_code_hash`, `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, and `vote_remove_os_measurement` do not filter stale votes from removed participants at counting time. They rely entirely on a fire-and-forget detached cleanup promise (`CLEAN_TEE_STATUS`) to purge those votes. If that promise fails silently, removed participants' pre-resharing votes remain in storage and count toward the governance threshold, reducing the effective threshold for whitelisting or removing TEE image hashes.

### Finding Description

When resharing completes, `vote_reshared` transitions the contract to `Running` and then spawns six detached cleanup promises, including `CLEAN_TEE_STATUS`:

```rust
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        vec![],
        NearToken::from_yoctonear(0),
        Gas::from_tgas(self.config.clean_tee_status_tera_gas),
    )
    .detach();
``` [1](#0-0) 

`.detach()` means the promise is fire-and-forget: if it fails (e.g., gas exhaustion), the contract state is not rolled back and there is no retry mechanism. The `clean_tee_status` endpoint is `#[private]`, so only the contract itself can call it, and there is no public retry path.

The five TEE governance vote functions (`vote_code_hash`, `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, `vote_remove_os_measurement`) all follow the same pattern:

```rust
let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
let votes = self.tee_state.vote(code_hash, &participant);
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
``` [2](#0-1) 

`AuthenticatedParticipantId::new` prevents a removed participant from calling these functions after resharing. However, the raw vote count returned by `tee_state.vote()` includes **all stored votes**, including stale ones from removed participants. The `CodeHashesVotes` data structure does not filter by current participant membership at counting time.

This is confirmed by the test comment in `tee_state.rs`:

> "P2 votes for the same malicious hash — should be only 1 vote, not 3" [3](#0-2) 

The comment "not 3" explicitly acknowledges that without `clean_non_participant_votes`, the stale votes from P0 and P1 would still be counted.

**Contrast with correctly-mitigated functions:**

`vote_update` explicitly filters at counting time with a comment acknowledging the cleanup can fail:

```rust
// Filter votes to only count current participants voting for this specific update.
// This ensures correctness even if the cleanup promise in MpcContract::vote_reshared() fails.
let valid_votes_count = running_state.parameters.participants().participants()
    .iter()
    .filter(|(account_id, _, _)| { ... })
    .count();
``` [4](#0-3) 

`vote_tee_verifier_change` also filters via `count_for`:

```rust
let count_usize = {
    let voter_set = self.pending.vote(participant, proposal_hash);
    voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
};
``` [5](#0-4) 

The TEE governance vote functions (`vote_code_hash`, etc.) lack this same defense-in-depth filtering.

### Impact Explanation

If `CLEAN_TEE_STATUS` fails after a resharing that removes participant P0, P0's pre-resharing vote for a malicious Docker image hash `H_malicious` remains in `tee_state.votes`. A new participant P_new then only needs `(threshold - 1)` colluding current participants to vote for `H_malicious` to reach the threshold. Once `H_malicious` is whitelisted, a node running that malicious image can submit an attestation that passes verification, join the MPC network, and participate in threshold signing rounds — potentially learning key share material or influencing signing output.

This breaks the production safety invariant that TEE governance decisions require a full threshold of **current** participants.

### Likelihood Explanation

The `CLEAN_TEE_STATUS` gas budget is configurable (`self.config.clean_tee_status_tera_gas`). With a large participant set, the cleanup iterates over all stored votes and can exhaust its gas allocation, failing silently. Additionally, there is a structural race window: between the moment resharing completes and the cleanup promise executes, any new participant vote on a pending proposal will count stale votes. A participant who knows they are about to be removed can vote for a malicious hash immediately before resharing, and if cleanup fails, that vote persists indefinitely with no on-chain mechanism to remove it (since `clean_tee_status` is `#[private]`).

### Recommendation

Apply the same defense-in-depth filtering used by `vote_update` and `vote_tee_verifier_change` to all five TEE governance vote functions. At counting time, filter the raw vote count to include only votes from accounts that are current participants:

```rust
// In vote_code_hash, vote_add_launcher_hash, etc.:
let valid_votes = self.tee_state.votes.count_for_current_participants(
    code_hash,
    threshold_parameters.participants(),
);
if valid_votes >= self.threshold()?.value() { ... }
```

This makes correctness independent of whether the cleanup promise succeeded, matching the explicit design intent already documented in `vote_update`.

### Proof of Concept

1. Contract has N=5 participants {P0, P1, P2, P3, P4}, threshold T=3.
2. P0 (attacker, about to be removed) calls `vote_code_hash(H_malicious)`. Vote count = 1.
3. Resharing completes, removing P0. New set: {P1, P2, P3, P4}, T=3.
4. `CLEAN_TEE_STATUS` detached promise fails (gas exhaustion). P0's vote for `H_malicious` remains in `tee_state.votes`.
5. P1 calls `vote_code_hash(H_malicious)`. Raw count = 2 (P0 stale + P1).
6. P2 calls `vote_code_hash(H_malicious)`. Raw count = 3 (P0 stale + P1 + P2) ≥ threshold 3.
7. `whitelist_tee_proposal(H_malicious, ...)` executes — malicious image hash is now whitelisted.
8. Attacker runs a node with `H_malicious`, submits attestation that passes, and joins signing rounds with only 2 current-participant votes instead of the required 3. [6](#0-5) [7](#0-6)

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
