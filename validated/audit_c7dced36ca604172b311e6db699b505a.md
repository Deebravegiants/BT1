Audit Report

## Title
Reward Tokens Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` uses a Synthetix-style reward accumulator that freezes `rewardPerTokenStored` whenever `totalKernelStaked == 0`. Any staker can call `initiateWithdrawal` to immediately reduce `totalKernelStaked` to zero during an active reward period, causing all reward tokens accruing during the zero-staking window to be permanently locked in the contract with no on-chain recovery path.

## Finding Description
`rewardPerToken()` returns the frozen `rewardPerTokenStored` when `totalKernelStaked == 0`:

```solidity
// L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;  // accumulation halts
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

The `updateReward` modifier (L232-242) snapshots `updatedAt = lastTimeRewardApplicable()` before the function body executes. After `initiateWithdrawal` sets `totalKernelStaked = 0` (L326), every subsequent call to `rewardPerToken()` returns the same frozen value. The `rewardRate` continues to imply token emission, but those tokens are never attributed to any account.

`initiateWithdrawal` has no guard against zeroing `totalKernelStaked` during an active reward window:

```solidity
// L320-326
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    if (_amount == 0) revert AmountZero();
    if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
    if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;   // immediate, unchecked reduction
```

`notifyRewardAmount` (L570) guards against starting a period with zero staked, but provides no protection once a period is active. There is no `sweep`, `recoverERC20`, or admin rescue function for `rewardsToken` anywhere in the contract. The NatSpec at L14-23 explicitly acknowledges this limitation and states it is mitigated solely by off-chain operational discipline — there is no on-chain enforcement.

## Impact Explanation
Reward tokens (`rewardsToken`) deposited via `notifyRewardAmount` that correspond to any time interval where `totalKernelStaked == 0` are permanently frozen in `KernelDepositPool`. The stuck amount equals `rewardRate × (seconds of zero-staking)` in reward token units. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
Any unprivileged staker can trigger this by calling `initiateWithdrawal` for their full balance during an active reward period. No special role, flash loan, or coordination is required. In a single-large-staker scenario (common in early protocol deployment), one call is sufficient. The staker has a legitimate reason to withdraw (e.g., needing liquidity), making this a realistic, non-adversarial trigger path.

## Recommendation
Implement one or more of the following:
1. Add an admin-callable `recoverStuckRewards()` function that can reclaim undistributed reward tokens after `block.timestamp >= finishAt`.
2. In `initiateWithdrawal`, revert if the withdrawal would set `totalKernelStaked == 0` while `block.timestamp < finishAt`.
3. Track "lost" reward seconds when `totalKernelStaked == 0` and roll the corresponding token amount into the next `notifyRewardAmount` call.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 30 days`. `rewardRate = 1_000e18 / 30 days`.
2. Alice is the sole staker: `balanceOf[Alice] = 1_000e18`, `totalKernelStaked = 1_000e18`.
3. After 10 days, Alice calls `initiateWithdrawal(1_000e18)`. The `updateReward(Alice)` modifier correctly snapshots her 10-day earnings. Then `totalKernelStaked` is set to 0.
4. For the remaining 20 days, `rewardPerToken()` returns the frozen `rewardPerTokenStored`. No staker accumulates rewards.
5. `rewardRate × 20 days ≈ 666e18` reward tokens remain in the contract with no claimant and no recovery function.
6. Foundry invariant test: assert `rewardsToken.balanceOf(address(pool)) == 0` after all stakers have claimed — this invariant breaks after a zero-staking window.
