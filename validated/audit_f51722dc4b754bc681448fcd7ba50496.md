Audit Report

## Title
Fee Bypass via Integer Division Truncation in `viewSwapRsETHAmountAndFee` - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The fee calculation `fee = amount * feeBps / 10_000` truncates to zero for any deposit amount below `⌈10_000 / feeBps⌉`. Because the only deposit guard is `if (amount == 0) revert InvalidAmount()`, any unprivileged depositor can split a large deposit into many sub-threshold transactions and receive the full rsETH/wrsETH mint while the protocol collects zero fee revenue. The same pattern is confirmed in `RSETHPool.sol` and `AGETHPoolV3.sol`.

## Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256)` (line 300) and its token overload (line 324), the fee is computed as:

```solidity
fee = amount * feeBps / 10_000;
```

Solidity integer division truncates. When `amount * feeBps < 10_000`, `fee = 0` and `amountAfterFee = amount`, so the depositor receives the full rsETH mint with no fee deducted. The ETH deposit entry point (line 256) only checks `if (amount == 0) revert InvalidAmount()` — no minimum deposit amount is enforced. The `limitDailyMint` modifier (lines 96–125) tracks total rsETH minted per day but does not prevent fee-zero deposits; it only caps the aggregate daily mint volume. There is no `minAmount` variable in `RSETHPoolV3.sol` (confirmed by search). The identical truncation pattern exists at `RSETHPool.sol` lines 312 and 336, and `AGETHPoolV3.sol` lines 161 and 184.

**Exploit path:**
1. Attacker identifies `feeBps` (e.g., 5 → threshold = 1 999 wei for ETH).
2. Attacker calls `deposit{value: 1_999}(referralId)` in a loop on an L2 (Arbitrum/Optimism).
3. Each call: `fee = 1999 * 5 / 10000 = 0`; `feeEarnedInETH += 0`; full `1_999 wei` worth of wrsETH is minted.
4. After N iterations the attacker has deposited N × 1 999 wei and paid zero protocol fees.

## Impact Explanation
**High — Theft of unclaimed yield.** The protocol is designed to collect a basis-point fee on every deposit, accrued into `feeEarnedInETH` / `feeEarnedInToken`. By making sub-threshold deposits the attacker receives a larger rsETH/wrsETH mint than entitled (the fee that should have been deducted is not), and the protocol's fee revenue stream is drained to zero. The attacker does not lose principal; the loss falls entirely on the protocol's accrued fee income.

## Likelihood Explanation
These pools are deployed on L2 networks where gas per call is negligible (fractions of a cent on Arbitrum/Optimism). The threshold is small but the attack is economically rational: 500 000 calls at 1 999 wei each costs a few dollars in L2 gas while bypassing ~0.0005 ETH in fees on ~1 ETH of deposits. No special privilege is required — any EOA can call `deposit`. The attack is repeatable every day (the daily mint limit resets), and the threshold is fixed by `feeBps` which is a protocol-set constant.

## Recommendation
Enforce a minimum deposit amount so that the computed fee is always at least 1 unit:

```solidity
uint256 minDepositAmount = 10_000 / feeBps + 1;
if (amount < minDepositAmount) revert DepositBelowMinimum();
```

Or revert when the fee rounds to zero while `feeBps > 0`:

```solidity
fee = amount * feeBps / 10_000;
if (feeBps > 0 && fee == 0) revert DepositTooSmall();
```

Apply the same fix to both `viewSwapRsETHAmountAndFee` overloads in `RSETHPoolV3`, `RSETHPool`, `RSETHPoolV2`, `RSETHPoolNoWrapper`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV2NBA`, and `AGETHPoolV3`.

## Proof of Concept
**Foundry fork test (Arbitrum):**

```solidity
function testFeeBypass() public {
    uint256 feeBps = pool.feeBps(); // e.g. 5
    uint256 threshold = 10_000 / feeBps; // 1999 for feeBps=5

    uint256 feesBefore = pool.feeEarnedInETH();

    for (uint256 i = 0; i < 1000; i++) {
        pool.deposit{value: threshold}("ref");
    }

    uint256 feesAfter = pool.feeEarnedInETH();
    assertEq(feesAfter, feesBefore); // fees collected = 0
}
```

Each iteration: `fee = 1999 * 5 / 10000 = 0` (confirmed by the line 300 arithmetic). After 1 000 iterations, `feeEarnedInETH` is unchanged while the attacker has received 1 000 × full wrsETH mints. The same test applies to the token deposit path using `viewSwapRsETHAmountAndFee(amount, token)` at line 324.
