### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Replay of Stale Foreign-Chain Attestations — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the root public key, but never checks that `response.payload_hash` is the hash of a `ForeignTxSignPayload` derived from the `request` argument. Because `VerifyForeignTransactionRequest` carries no nonce or timestamp, the same foreign-chain query can be re-submitted after a timeout or reorganization, and a single malicious participant (below threshold) can satisfy the new pending yield by replaying any previously observed `(payload_hash, signature)` pair — delivering stale or incorrect extracted values to the bridge caller.

---

### Finding Description

**Root cause — missing binding between `payload_hash` and `request`**

In `respond_verify_foreign_tx` the contract performs exactly two checks before resolving all queued yields:

1. The caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the **root** public key (no tweak).

```
// crates/contract/src/lib.rs  lines 718-734
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,          // ← root key, no tweak
)
.is_ok()
```

There is **no check** that `payload_hash` equals `ForeignTxSignPayload::V1 { request: request.request, values: <anything> }.compute_msg_hash()`. The `request` argument is used only as a lookup key into `pending_verify_foreign_tx_requests`; its content is never bound to the hash that is signed and returned to the caller.

**Why replay is possible**

`VerifyForeignTransactionRequest` contains only `(request: ForeignChainRpcRequest, domain_id, payload_version)` — no nonce, no NEAR block hash, no timestamp:

```
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs  lines 124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

The conversion from args to request discards the caller's account ID entirely:

```
// crates/contract/src/dto_mapping.rs  lines 840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

After `resolve_yields_for` drains the queue, the map entry is removed. If the same `VerifyForeignTransactionRequest` is submitted again (new `verify_foreign_transaction` call), a fresh yield is enqueued under the identical key. Any previously observed `(payload_hash_old, sig_old)` pair — visible on-chain from the first `respond_verify_foreign_tx` call — is still cryptographically valid and will pass the contract's only check.

**Exploit path**

1. At time T₁, a bridge service calls `verify_foreign_transaction` for Bitcoin tx_id X (2 confirmations, `BlockHash` extractor). MPC nodes verify and sign; the leader calls `respond_verify_foreign_tx(request_X, {payload_hash_old, sig_old})` where `payload_hash_old = hash(ForeignTxSignPayloadV1 { request_X, [BlockHash(B_old)] })`. The yield resolves; the map entry is removed.

2. Bitcoin reorganizes. Tx X is now in a different block with hash `B_new ≠ B_old`.

3. The bridge service re-submits `verify_foreign_transaction` for the same `request_X` (same tx_id, same confirmations, same extractors). A new yield is enqueued under the same key.

4. A single malicious participant P (who observed step 1 on-chain) calls `respond_verify_foreign_tx(request_X, {payload_hash_old, sig_old})` before the honest nodes can respond.

5. The contract checks: (a) P is an attested participant ✓; (b) `sig_old` is valid over `payload_hash_old` under the root key ✓; (c) `request_X` is in the pending map ✓. It resolves the yield with the stale response.

6. The bridge contract receives `(payload_hash_old, sig_old)` — an MPC-signed attestation that Bitcoin tx X is in block `B_old` — even though the current canonical block is `B_new`. The bridge interprets this as a fresh, valid attestation and executes accordingly.

---

### Impact Explanation

The bridge caller receives a cryptographically valid MPC signature over a `ForeignTxSignPayload` that encodes **stale extracted values** (e.g., a pre-reorganization block hash). From the bridge contract's perspective the attestation is indistinguishable from a fresh one: the root key signed it, and the contract accepted it. This enables:

- **Invalid bridge execution**: a bridge that gates asset release on a specific block hash will release funds against a block that no longer exists in the canonical chain.
- **Double-spend conditions**: if the reorganized transaction was a deposit, the bridge may credit the deposit twice — once for the original confirmation and once for the replayed stale attestation after re-submission.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- The attacker must be an **attested participant** (required by `assert_caller_is_attested_participant_and_protocol_active`). This is a Byzantine participant strictly below the signing threshold — explicitly in scope.
- The old `(payload_hash, signature)` is **publicly visible** on-chain from the first `respond_verify_foreign_tx` call; no privileged access is needed to obtain it.
- Re-submission of the same foreign-chain query is a **normal operational event**: bridge services retry after NEAR yield timeouts, and chain reorganizations are routine on probabilistically-finalized chains (Bitcoin, EVM with `Latest`/`Safe` finality).
- The attacker only needs to **front-run** the honest nodes' response in the second round, which is feasible given that the attacker is already a network participant with low-latency access.

---

### Recommendation

In `respond_verify_foreign_tx`, reconstruct the minimum expected payload prefix from the `request` argument and verify that `response.payload_hash` is consistent with it. Concretely, for `ForeignTxPayloadVersion::V1`, the signed payload is `ForeignTxSignPayloadV1 { request: request.request, values: <node-observed> }`. The contract cannot know `values` ahead of time, but it **can** verify that `payload_hash` decodes to a payload whose embedded `request` field matches the `request` argument:

```diff
// After verifying the signature, also verify the payload is bound to this request:
+ let decoded_payload: ForeignTxSignPayload =
+     ForeignTxSignPayload::from_hash_preimage(&response.payload_hash)
+     .ok_or(RespondError::InvalidPayload)?;
+ if decoded_payload.request() != &request.request {
+     return Err(RespondError::PayloadRequestMismatch.into());
+ }
```

Alternatively, include a per-request nonce or the NEAR block hash at submission time inside `VerifyForeignTransactionRequest` so that re-submitted requests produce a distinct key and old responses cannot satisfy them:

```diff
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
+   pub nonce: CryptoHash,   // set to env::block_hash() at submission time
}
```

```rust

// Round 1: legitimate flow
contract.verify_foreign_transaction(request_X);          // bridge submits
// honest nodes respond:
contract.respond_verify_foreign_tx(

    VerifyForeignTransactionResponse {

    }

// --- Bitcoin reorganizes; B_old is no longer canonical ---

// Round 2: bridge re-submits the same request after timeout
contract.verify_foreign_transaction(request_X);          // new yield enqueued

// Malicious participant P replays the old response (observed on-chain from Round 1):
contract.respond_verify_foreign_tx(
    request_X,
    VerifyForeignTransactionResponse {
        payload_hash: payload_hash_old,   // stale — still valid signature
        signature: sig_old,
    }
);
// Contract checks:
//   (1) P is attested participant          ✓
//   (2) sig_old valid over payload_hash_old under root key  ✓
//   (3) request_X in pending map           ✓
// → yield resolved with stale response; bridge receives B_old as "current" block hash
```

The contract's verification logic at lines 718–734 of `crates/contract/src/lib.rs` passes all three checks without ever comparing `payload_hash_old` to anything derived from `request_X`, confirming the vulnerability is reachable with a single below-threshold participant and no cryptographic forgery.
