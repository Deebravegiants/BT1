### Title
Dual-Mapping DRep Delegation Inconsistency Allows Silent Vote-Delegation Wipe During Bootstrap Phase - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs`)

### Summary

The Conway era maintains DRep vote delegation in two separate data structures: a forward map (`AccountState.dRepDelegationAccountStateL` inside `Accounts`) and a reverse map (`DRepState.drepDelegs` inside `VState`). During the bootstrap phase (protocol version < 10), redelegation from one DRep to another only updates the forward map and adds to the new DRep's reverse set, but intentionally does **not** remove the stale entry from the old DRep's reverse set. When the old DRep later unregisters, the `ConwayUnRegDRep` handler uses its (stale) `drepDelegs` set to clear forward delegations — incorrectly wiping the current, legitimate delegation of any credential that had already redelegated away. This silently zeroes out those credentials' vote delegation, reducing the effective voting power of the DRep they legitimately delegated to.

### Finding Description

The `DRepState` record holds a `drepDelegs :: Set (Credential Staking)` field that is supposed to be the reverse index of all stake credentials currently delegating to that DRep. [1](#0-0) 

The `processDelegationInternal` function in the `DELEG` rule handles redelegation. When `preserveIncorrectDelegation` is `True` (set when `pvMajor pv < natVersion @10`, i.e., during bootstrap), the code explicitly skips removing the stale reverse-delegation entry from the old DRep: [2](#0-1) 

The comment at line 356–363 explicitly acknowledges this is the "incorrect behavior" being preserved. The bug reference is issue #4772.

When a DRep unregisters via `ConwayUnRegDRep`, the `GOVCERT` rule uses `drepDelegs` to clear forward delegations: [3](#0-2) 

Because `drepDelegs` contains stale entries (credentials that have already redelegated to a different DRep), `clearDRepDelegations` sets `dRepDelegationAccountStateL .~ Nothing` for those credentials — wiping their current, valid delegation to the new DRep.

The `DELEG` transition rule passes `preserveIncorrectDelegation = (pvMajor pv < natVersion @10)`: [4](#0-3) 

The `hardforkTransition` at version 10 calls `updateDRepDelegations` to reconcile the two maps, but this only runs at the hard fork boundary — not during the bootstrap phase itself: [5](#0-4) 

The test suite explicitly confirms the impact: after DRep X unregisters in bootstrap, `lookupDRepDelegation cred accounts` returns `Nothing` for a credential that had already redelegated to DRep Y: [6](#0-5) 

### Impact Explanation

The `dRepDelegationAccountStateL` field in `Accounts` is the authoritative source for computing each DRep's active voting power during governance ratification. When this field is incorrectly set to `Nothing` for credentials that legitimately delegated to DRep Y, DRep Y's effective voting stake is silently reduced. This can:

1. Cause a governance action that DRep Y supports to fail ratification (legitimate action blocked).
2. Cause a governance action that DRep Y opposes to be ratified (unauthorized governance action enacted).

This matches the **Critical** impact: *Unauthorized governance action is enacted* or a legitimate one is blocked due to an invalid ledger state transition.

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender during the Conway bootstrap phase (protocol version 9):

1. Attacker registers DRep X (permissionless, costs only the DRep deposit).
2. Attacker attracts stake credential delegations to DRep X (or waits for organic delegations).
3. Those credentials redelegate to DRep Y (a legitimate DRep with governance influence).
4. Attacker submits `UnRegDRepTxCert` for DRep X, reclaiming their deposit.
5. The `clearDRepDelegations` call wipes the forward delegations of all credentials in DRepX's stale `drepDelegs`, including those now delegating to DRep Y.

No privileged access, leaked keys, or supermajority is required. The attacker recovers their deposit upon unregistration, making the attack essentially free.

### Recommendation

During the bootstrap phase, `ConwayUnRegDRep` should validate each entry in `drepDelegs` against the current forward delegation before clearing it. Specifically, `clearDRepDelegations` should only set `dRepDelegationAccountStateL .~ Nothing` for credentials whose `dRepDelegationAccountStateL` still points to the unregistering DRep:

```haskell
clearDRepDelegations unregCred delegs accountsMap =
  foldr
    (Map.adjust (\as ->
       if as ^. dRepDelegationAccountStateL == Just (DRepCredential unregCred)
         then as & dRepDelegationAccountStateL .~ Nothing
         else as))
    accountsMap
    delegs
```

This fix should be applied regardless of protocol version, since the stale `drepDelegs` entries are the root cause of the incorrect clearing.

### Proof of Concept

The existing test at `eras/conway/impl/testlib/Test/Cardano/Ledger/Conway/Imp/DelegSpec.hs` (the "Redelegate vote" case) already demonstrates the bug: [7](#0-6) 

Concretely:
1. Register stake credential `cred`, register DRep `drepCred`, register DRep `drepCred2`.
2. Submit `RegDepositDelegTxCert cred (DelegVote (DRepCredential drepCred)) deposit` → `cred` delegates to `drepCred`.
3. Submit `DelegTxCert cred (DelegVote (DRepCredential drepCred2))` → `cred` redelegates to `drepCred2`. In bootstrap, `drepCred.drepDelegs` still contains `cred` (stale).
4. Submit `UnRegDRepTxCert drepCred refund` → `clearDRepDelegations {cred} accountsMap` sets `cred.dRepDelegationAccountStateL = Nothing`.
5. `lookupDRepDelegation cred accounts` returns `Nothing` — `cred`'s delegation to `drepCred2` has been silently wiped.

updateDRepDelegations :: ConwayEraCertState era => CertState era -> CertState era
updateDRepDelegations certState =
  let accountsMap = certState ^. certDStateL . accountsL . accountsMapL
      dReps =
        -- Reset all delegations in order to remove any inconsistencies
        -- Delegations will be reset accordingly below.
        Map.map (\dRepState -> dRepState {drepDelegs = Set.empty}) $
          certState ^. certVStateL . vsDRepsL
      (dRepsWithDelegations, accountsWithoutUnknownDRepDelegations) =
        Map.mapAccumWithKey adjustDelegations dReps accountsMap
      adjustDelegations ds stakeCred accountState =
        case accountState ^. dRepDelegationAccountStateL of
          Just (DRepCredential dRep) ->
            let addDelegation _ dRepState =
                  Just $ dRepState {drepDelegs = Set.insert stakeCred (drepDelegs dRepState)}
             in case Map.updateLookupWithKey addDelegation dRep ds of
                  (Nothing, _) -> (ds, accountState & dRepDelegationAccountStateL .~ Nothing)
                  (Just _, ds') -> (ds', accountState)
          _ -> (ds, accountState)
   in certState
        -- Remove dangling delegations to non-existent DReps:
        & certDStateL . accountsL . accountsMapL .~ accountsWithoutUnknownDRepDelegations
        -- Populate DRep delegations with delegatees
        & certVStateL . vsDRepsL .~ dRepsWithDelegations
