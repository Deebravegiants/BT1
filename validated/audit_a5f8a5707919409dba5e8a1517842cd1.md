### Title
Single Malicious Participant Can Submit Unverified CKD Response for `AppPublicKey` Variant, Bypassing Threshold Authorization and Consuming Pending Requests - (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_ckd` function in the MPC smart contract applies cryptographic output verification only for the `AppPublicKeyPV` (publicly verifiable) variant of CKD requests. For the `AppPublicKey` (privately verifiable) variant, **no on-chain verification is performed**. A single malicious attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse`, which the contract unconditionally accepts, drains the entire pending yield queue, and delivers to all waiting callers. The legitimate threshold-authorized response can never be delivered for that request.

---

### Finding Description

In `respond_ckd`, the response verification is gated on the `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For the `AppPublicKey` branch, the contract performs zero cryptographic validation of `big_y` and `big_c` before calling `resolve_yields_for`, which atomically removes the entire pending queue entry and resumes every queued yield with the supplied (potentially forged) bytes. [2](#0-1) 

By contrast, `respond` (for ECDSA/EdDSA signatures) always verifies the signature against the derived public key before resolving yields: [3](#0-2) 

And `respond_verify_foreign_tx` always verifies the ECDSA signature against the root public key: [4](#0-3) 

The `AppPublicKey` variant is described as "privately verifiable" because the app verifies correctness after decryption using its own secret key `a`. The on-chain contract cannot perform this check. However, the absence of any on-chain guard means the contract cannot distinguish a threshold-produced response from a single-participant forgery, and it resolves the request either way.

The `respond_ckd` caller must be an attested participant: [5](#0-4) 

There is no requirement that the caller be the leader, nor any threshold vote count. Any single attested participant satisfies the guard.

---

### Impact Explanation

**Vulnerability class:** request/accounting/state corruption — a single below-threshold operation manipulates request lifecycle state to bypass the threshold authorization invariant.

**Concrete impact:**

1. A single malicious attested participant calls `respond_ckd` with an arbitrary `CKDResponse{big_y: ..., big_c: ...}` targeting any live `AppPublicKey` CKD request.
2. The contract accepts the forged response without verification, calls `resolve_yields_for`, drains the entire pending yield queue for that request key, and delivers the forged bytes to every waiting caller.
3. The pending request entry is permanently removed from `pending_ckd_requests`. Legitimate MPC nodes that subsequently attempt to call `respond_ckd` for the same request receive `RequestNotFound` and cannot deliver the threshold-authorized response.
4. The caller (TEE app) receives a forged `(big_y, big_c)` pair. App-side verification (`e(H(pk, app_id), pk) = e(sig, G2)`) will fail because the attacker, holding only one key share, cannot produce a value of `sig = msk · H(pk, app_id)` consistent with the MPC master secret. The app cannot use the forged key.
5. The user must resubmit the request, which can be attacked again by the same participant indefinitely.

This breaks the production safety invariant that **a CKD response must be produced by a threshold number of participants**. A single Byzantine participant below the signing threshold can unilaterally resolve any pending `AppPublicKey` CKD request, permanently blocking the legitimate threshold response.

This maps to: **Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

---

### Likelihood Explanation

Any single attested participant in the MPC network can exploit this. The attacker only needs to:
- Monitor the NEAR blockchain for `request_app_private_key` calls using the `AppPublicKey` variant (observable on-chain).
- Call `respond_ckd` before the legitimate threshold response is submitted.

The `AppPublicKey` variant is the legacy/default variant and is widely used. The attack requires no special tooling beyond a standard NEAR transaction. The attacker does not need to collude with any other participant.

---

### Recommendation

Add on-chain verification for the `AppPublicKey` variant. One approach: require the response to be accompanied by a threshold number of partial-response commitments that the contract can aggregate and verify against the stored BLS public key. Alternatively, adopt a commit-reveal or threshold-vote scheme analogous to `vote_reshared`, where `respond_ckd` only resolves the yield after a threshold number of participants have submitted consistent partial outputs.

At minimum, document that the `AppPublicKey` variant provides no on-chain response integrity guarantee and that a single participant can forge and deliver an invalid response, so that callers are aware they must always verify the response before use.

---

### Proof of Concept

```rust
// Attacker is a single attested participant (participant_0).
// User has submitted request_app_private_key with AppPublicKey variant.

// 1. Attacker observes the pending CKD request on-chain.
let ckd_request = CKDRequest::new(
    CKDAppPublicKey::AppPublicKey(app_public_key),
    domain_id,
    &user_account_id,
    &derivation_path,
);

// 2. Attacker forges an arbitrary response (no threshold computation needed).
let forged_response = CKDResponse {
    big_y: dtos::Bls12381G1PublicKey([0xAB; 48]), // arbitrary
    big_c: dtos::Bls12381G1PublicKey([0xCD; 48]), // arbitrary
};

// 3. Attacker calls respond_ckd as a single attested participant.
//    No on-chain verification is performed for AppPublicKey variant.
//    The contract resolves all pending yields with the forged response.
contract.respond_ckd(ckd_request.clone(), forged_response)
    .expect("accepted without verification");

// 4. Legitimate MPC nodes can no longer respond — request is gone.
let result = contract.respond_ckd(ckd_request.clone(), legitimate_response);
assert!(matches!(result, Err(InvalidParameters::RequestNotFound)));

// 5. User receives forged (big_y, big_c); app-side pairing check fails.
//    User must resubmit — which can be attacked again.
```

The `AppPublicKeyPV` variant is immune because `ckd_output_check` enforces the pairing equation `e(C, G2) = e(Y, pk2) · e(H(pk, app_id), pk)` on-chain before resolving yields. [6](#0-5)

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
