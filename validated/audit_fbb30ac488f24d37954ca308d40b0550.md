Missing Fee Upper-Bound Validation in `initialize` While `setFeeBps` Enforces a Cap - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPoolV2.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV2NBA.sol, contracts/agETH/AGETHPoolV3.sol)

---

### Summary

Every pool contract in the LRT-rsETH suite accepts a `_feeBps` parameter in its `initialize` function and stores it directly without any upper-bound check. Each of those same contracts enforces a maximum in its `setFeeBps` setter. The asymmetry means the deployer can initialize the contract with a fee that is impossible to set post-deployment, silently overcharging every depositor from the first block.

---

### Finding Description

The pattern is identical across all pool contracts. During initialization `_feeBps` is stored without validation:

```solidity
// RSETHPoolV3.sol – initialize()
feeBps = _feeBps;   // no upper-bound check
``` [1](#0-0) 

Yet the post-deployment setter enforces a strict cap:

```solidity
// RSETHPoolV3.sol – setFeeBps()
function setFeeBps(uint256 _feeBps) external onlyRole(DEFAULT_ADMIN_ROLE) {
    if (_feeBps > 1000) revert InvalidFeeAmount();   // max 10 %
    feeBps = _feeBps;
}
``` [2](#0-1) 

The same gap exists in every other pool:

| Contract | `setFeeBps` cap | `initialize` cap |
|---|---|---|
| `RSETHPoolV3.sol` | 1 000 BPS (10 %) | none |
| `RSETHPoolV2ExternalBridge.sol` | 1 000 BPS (10 %) | none |
| `RSETHPoolV3WithNativeChainBridge.sol` | 1 000 BPS (10 %) | none |
| `RSETHPool.sol` | 10 000 BPS (100 %) | none |
| `RSETHPoolNoWrapper.sol` | 10 000 BPS (100 %) | none |
| `RSETHPoolV2NBA.sol` | 10 000 BPS (100 %) | none |
| `RSETHPoolV2.sol` | (setter present) | none |
| `AGETHPoolV3.sol` | 10 000 BPS (100 %) | none | [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

The fee is applied to every deposit:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [9](#0-8) 

---

### Impact Explanation

**Scenario A – feeBps > 10 000 (> 100 %):** `amountAfterFee = amount - fee` underflows and reverts in Solidity 0.8. Every `deposit` call reverts permanently until the admin calls `setFeeBps` with a valid value. This is a temporary freeze of the deposit path for all users.

**Scenario B – feeBps between the setter cap and 10 000 (e.g., 1 001–10 000 for V3 contracts):** Deposits succeed but users receive far fewer rsETH/agETH tokens than the exchange rate warrants. The excess ETH is silently accumulated in `feeEarnedInETH` and later withdrawn by the `BRIDGER_ROLE`. Every depositor from block 1 loses value. This constitutes theft of depositor yield.

Impact classification: **Low – Contract fails to deliver promised returns** (Scenario B); escalates to **Medium – Temporary freezing of funds** (Scenario A).

---

### Likelihood Explanation

The `initialize` function is called exactly once, by the deployer, with no on-chain enforcement of the fee ceiling. A deployment script error, copy-paste mistake, or unit mismatch (e.g., confusing 10 % = 1 000 BPS with 10 % = 10 000 BPS) is a realistic path to triggering Scenario B silently. The absence of the same guard that exists in `setFeeBps` makes this a latent misconfiguration risk on every new pool deployment.

---

### Recommendation

Apply the same upper-bound check used in `setFeeBps` inside every `initialize` function. For the V3 family (cap = 1 000):

```solidity
function initialize(..., uint256 _feeBps, ...) external initializer {
    if (_feeBps > 1000) revert InvalidFeeAmount();
    ...
    feeBps = _feeBps;
}
```

For the V2/Pool family (cap = 10 000), apply `if (_feeBps > 10_000) revert InvalidFeeAmount();` consistently.

---

### Proof of Concept

1. Deploy `RSETHPoolV3` proxy and call `initialize` with `_feeBps = 5000` (50 %).
2. `setFeeBps(5000)` would revert with `InvalidFeeAmount` (cap is 1 000), but `initialize` accepts it silently.
3. A user calls `deposit{value: 1 ether}(...)`.
4. `fee = 1e18 * 5000 / 10_000 = 0.5 ether`; `amountAfterFee = 0.5 ether`.
5. User receives rsETH worth 0.5 ETH instead of ~1 ETH — 50 % of their deposit is captured as protocol fee.
6. The admin cannot reproduce this state via `setFeeBps` after deployment; the only remedy is an upgrade or redeployment. [10](#0-9) [11](#0-10)
