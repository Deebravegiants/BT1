### Title
`DirectDeposits` amounts absent from produced-value accounting enables ADA creation from nothing — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `DirectDeposits` field in the transaction body that adds arbitrary ADA amounts directly to reward-account balances. However, these amounts are never included in the **produced-value** side of the ledger's value-conservation check. Because the check enforces `consumed == produced` without accounting for `DirectDeposits`, any unprivileged transaction author can credit reward accounts with arbitrarily large ADA amounts while only funding the UTxO outputs, fee, and deposits — effectively minting ADA from nothing.

---

### Finding Description

**Root cause — missing term in `dijkstraProducedValue`**

The Dijkstra era's produced-value function is defined in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody          -- outputs + fee + deposits + donation
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)
``` [1](#0-0) 

`conwayProducedValue` adds outputs, fee, deposits, and `treasuryDonation` — but **not** `DirectDeposits`:

```haskell
conwayProducedValue pp isRegPoolId txBody =
  getProducedMaryValue pp isRegPoolId txBody
    <+> inject (txBody ^. treasuryDonationTxBodyL)
``` [2](#0-1) 

The consumed-value function `getConsumedDijkstraValue` also omits `DirectDeposits`:

```haskell
txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
-- sums: UTxO inputs + refunds + withdrawals — no DirectDeposits
``` [3](#0-2) 

**How `DirectDeposits` are applied**

In `dijkstraEntitiesTransition`, after certificate processing, `applyDirectDeposits` unconditionally adds the specified amounts to account balances. The only guard is that target accounts must already exist:

```haskell
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [4](#0-3) 

`applyDirectDeposits` itself performs no amount validation:

```haskell
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [5](#0-4) 

**The broken invariant**

The ledger's value-conservation check (`validateValueNotConservedUTxO`) enforces:

```
consumed == produced
```

With `DirectDeposits` absent from both sides, the equation reduces to:

```
UTxO_inputs + refunds + withdrawals
  == UTxO_outputs + fee + deposits + treasury_donation
```

`DirectDeposits` are invisible to this check. A transaction that credits 1,000,000 ADA to a reward account via `DirectDeposits` passes the check as long as its UTxO balance is otherwise correct — the 1,000,000 ADA is never deducted from anywhere.

Compare with `treasuryDonation`, which is correctly included in `conwayProducedValue` and therefore forces the transaction to fund it from UTxO inputs. [6](#0-5) 

---

### Impact Explanation

**Critical — Direct creation of ADA from nothing.**

An attacker constructs a transaction whose UTxO inputs cover only outputs + fee + deposits. They set `DirectDeposits` to an arbitrarily large amount targeting any registered reward account they control. The value-conservation check passes; `applyDirectDeposits` then inflates the account balance by the specified amount. Repeating this across many transactions allows unbounded ADA inflation, violating the fundamental preservation-of-value invariant of the Cardano ledger.

---

### Likelihood Explanation

Any transaction author can include a `DirectDeposits` map in a Dijkstra-era transaction body. No special role, key, or privilege is required. The only precondition is that the target account is registered, which the attacker can arrange themselves. The attack is deterministic and requires no external cooperation.

---

### Recommendation

Include the total `DirectDeposits` amount in the produced-value calculation, mirroring the treatment of `treasuryDonation`:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <+> inject (foldMap id . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)
```

This forces the transaction to fund `DirectDeposits` from its UTxO inputs, restoring the preservation-of-value invariant. The same fix must be applied to `dijkstraSubTxProducedValue` for sub-transactions. [7](#0-6) 

---

### Proof of Concept

1. Register a stake credential and obtain its `AccountAddress` (e.g. `acct`).
2. Build a Dijkstra-era transaction:
   - **Inputs**: UTxO covering `outputs + fee` (e.g. 10 ADA in, 9 ADA out, 1 ADA fee).
   - **DirectDeposits**: `{ acct => 1_000_000_000_000 }` (1 million ADA).
3. Submit the transaction. `validateValueNotConservedUTxO` checks `10 ADA == 10 ADA` — passes.
4. `dijkstraEntitiesTransition` calls `applyDirectDeposits`; `acct`'s balance increases by 1,000,000,000,000 lovelace.
5. The attacker withdraws the inflated balance in a subsequent transaction, extracting ADA that was never present in the UTxO set.

The relevant production files are:
- `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` — missing `DirectDeposits` in `dijkstraProducedValue`
- `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs` — `applyDirectDeposits` called without amount validation
- `libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs` — `applyDirectDeposits` adds amounts unconditionally [1](#0-0) [4](#0-3) [8](#0-7)
