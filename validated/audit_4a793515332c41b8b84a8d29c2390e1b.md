### Title
Missing Vote Signature Verification Allows Any Network Peer to Forge Committee Votes and Trigger Spurious Block Commitment — (File: `crates/apollo_consensus/src/single_height_consensus.rs`)

### Summary

`handle_vote` in `SingleHeightConsensus` checks that `vote.voter` is a committee member address but never verifies the cryptographic signature on the vote. The `Vote` struct carries a `signature: RawSignature` field, and `verify_precommit_vote_signature` exists in `apollo_signature_manager`, but neither is called in the vote-handling path. Any network peer can craft prevote/precommit messages claiming any committee member address as `voter`, manufacture a fake quorum, and cause the target node to emit `DecisionReached` — committing a block to storage without genuine validator agreement.

### Finding Description

`handle_vote` in `crates/apollo_consensus/src/single_height_consensus.rs` performs two checks before accepting a vote:

1. Height match (`vote.height != height`)
2. Committee membership (`committee.members().iter().any(|s| s.address == vote.voter)`)

It explicitly defers signature verification with a `TODO`:

The `Vote` struct carries a `signature: RawSignature` field that is populated with `RawSignature::default()` (an all-zero signature) when votes are created locally, and is never verified when votes are received:

```rust

``` [2](#0-1) 

The `voter` field in `Vote` is a plain `ContractAddress` — it is not derived from the network peer's authenticated identity. The network-layer peer authentication (`verify_identity`) is a separate handshake that does not bind a peer's connection identity to any committee member address. Therefore, any connected peer can send a `Vote` message with `voter` set to any committee member's address.

The `cache_future_vote` function in the manager explicitly acknowledges this gap:

> "Since vote signatures are not yet verified, a peer can forge votes with arbitrary voter addresses" [3](#0-2) 

The `verify_precommit_vote_signature` function exists and is tested but is never called in the vote-handling path: [4](#0-3) 

### Impact Explanation

Once a local node has validated a proposal (stored `proposal_id` in `self.proposals` via `handle_finished_validation`), an attacker who can observe the network can:

1. Identify the `proposal_commitment` the target node has validated.
2. Send forged `Precommit` votes claiming to be from multiple committee member addresses for that commitment.
3. The `upon_decision` function fires when the forged votes accumulate to quorum weight, emitting `SMRequest::DecisionReached`. [5](#0-4) 

`DecisionReached` triggers `decision_reached()` in `SequencerConsensusContext`, which calls `batcher.decision_reached()` — an irreversible storage commit: [6](#0-5) 

The target node commits a block to storage and advances its chain state without the real committee members having voted. This causes the node to diverge from the canonical chain. The committed block carries a `ProposalCommitment` (partial block hash) derived from forged precommits, not from genuine validator agreement.

### Likelihood Explanation

Any peer that can establish a network connection to the target node can execute this attack. No validator private key is required. The attacker only needs to know the committee member addresses (public information from the staking contract) and observe which proposal the target node is currently validating (observable from the p2p proposal stream). The attack requires sending O(committee_size) forged vote messages.

### Recommendation

Call `verify_precommit_vote_signature` (already implemented in `apollo_signature_manager`) inside `handle_vote` before accepting a vote, using the committee member's `public_key` from the `Staker` struct:

```rust
pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
    // ... height check ...
    let staker = self.committee.members().iter().find(|s| s.address == vote.voter);
    let Some(staker) = staker else {
        debug!("Ignoring vote from non validator: vote={:?}", vote);
        return VecDeque::new();
    };
    match verify_precommit_vote_signature(
        BlockHash(vote.proposal_commitment.map_or(Felt::ZERO, |c| c.0)),
        vote.signature.clone(),
        staker.public_key,
    ) {
        Ok(true) => {}
        _ => {
            warn!("Invalid vote signature: {:?}", vote);
            return VecDeque::new();
        }
    }
    // ... rest of handling ...
}
```

Also resolve the companion `TODO` in `make_self_vote` to actually sign outgoing votes using `SignatureManager::sign_precommit_vote`. [7](#0-6) 

### Proof of Concept

```
Setup: 4-validator committee [P, V1, V2, V3], each weight 1. Byzantine quorum = 3/4.
Target node T is running as V1.

1. T receives and validates a proposal from P for height H, round 0.
   T stores proposal_commitment = X in self.proposals[0].

2. Attacker (any connected peer, not a validator) sends:
   - Precommit { voter: P,  height: H, round: 0, proposal_commitment: Some(X), signature: default() }
   - Precommit { voter: V2, height: H, round: 0, proposal_commitment: Some(X), signature: default() }
   - Precommit { voter: V3, height: H, round: 0, proposal_commitment: Some(X), signature: default() }

3. handle_vote accepts all three (address in committee, no signature check).
   T's own precommit (V1) + 3 forged = 4 votes = quorum.

4. upon_decision fires → DecisionReached(X) → batcher.decision_reached() → block committed.

Result: T commits block X without P, V2, V3 ever actually voting.
        T diverges from the canonical chain.
``` [8](#0-7)

    async fn decision_reached(

        height: BlockNumber,
        round: Round,
        commitment: ProposalCommitment,

        info!("Finished consensus for height: {height}. Agreed on block: {:#066x}", commitment.0);

        self.interrupt_active_proposal().await;
        let (init, transactions, proposal_id, finished_info) = {
            let mut proposals = self.valid_proposals.lock().unwrap();
            let (init, transactions, proposal_id, finished_info) =
