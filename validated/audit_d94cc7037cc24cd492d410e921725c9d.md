### Title
Missing Cryptographic Output Validation in `respond_ckd` for Legacy `AppPublicKey` Variant Allows Any Single Attested Participant to Corrupt CKD Responses - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` enforces cryptographic output validity only for the `AppPublicKeyPV` variant. For the legacy `AppPublicKey` variant, any single attested participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary `CKDResponse` and the contract will accept and deliver it to the waiting user without any verification.

### Finding Description

In `respond_ckd`, the contract branches on the `app_public_key` variant of the pending `CKDRequest`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the BLS pairing equation `e(C, G2) = e(H(pk, app_id), pk_mpc) · e(Y, A2)`, ensuring the encrypted output is mathematically bound to the MPC master secret and the user's public key. For `AppPublicKey` (the legacy, still-supported variant), the arm is an empty block — **no analogous check exists**.

The `respond_ckd` caller is only required to be an attested participant: [2](#0-1) 

All information needed to construct the matching `CKDRequest` key (`app_public_key`, `domain_id`, `predecessor_id`, `derivation_path`) is observable on-chain from the original `request_app_private_key` call: [3](#0-2) 

Once `resolve_yields_for` is called with the malicious response, the pending yield is consumed and the corrupted `CKDResponse` is delivered to the user: [4](#0-3) 

This is directly analogous to the reported `StandardToken::transferFrom` bug: a function that should enforce a validity/authorization check before completing a privileged state transition simply omits it for one code path.

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) can:

1. Observe any pending `request_app_private_key` call that uses the `AppPublicKey` variant.
2. Reconstruct the exact `CKDRequest` key from on-chain data.
3. Call `respond_ckd` with arbitrary `big_y` and `big_c` values before the legitimate coordinator does.
4. The contract accepts the response unconditionally and delivers it to the user.

The user receives a cryptographically invalid `(big_y, big_c)` pair. If the user performs the off-chain verification (`e(sig, G2) = e(H(pk, app_id), pk_mpc)`), they detect the corruption but the pending yield has already been consumed — they must resubmit. The attacker can repeat this indefinitely, permanently denying the user a valid CKD response. If the user skips verification and uses the corrupted output as a key, they derive a key that is not bound to the MPC master secret, breaking the confidentiality and integrity guarantees of the CKD protocol.

This breaks the production safety invariant that every delivered `CKDResponse` must be a valid encryption of `msk · H(pk, app_id)` under the user's public key.

**Impact class:** Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation

- Requires only a single attested participant (below threshold), not collusion.
- All inputs needed to construct the attack are publicly visible on-chain.
- The `AppPublicKey` variant is still accepted by `request_app_private_key` and used in production.
- The attacker only needs to front-run `respond_ckd` before the legitimate coordinator, which is straightforward since the attacker is already a participant monitoring the chain. [5](#0-4) 

### Recommendation

Apply the same cryptographic output validation to the `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. For the legacy variant, the contract should verify that the submitted `(big_y, big_c)` satisfies the BLS pairing equation with respect to the MPC public key and the `app_id`. If on-chain verification of the privately-verifiable variant is not feasible (because it requires the user's secret scalar), the contract should at minimum require threshold-weighted agreement from multiple participants before resolving the yield, mirroring the threshold requirement enforced at the signing layer.

### Proof of Concept

1. Alice calls `request_app_private_key` with `AppPublicKey` variant. The `CKDRequest` key is now visible on-chain.
2. Attacker (single attested participant) reconstructs the `CKDRequest` from on-chain data.
3. Attacker calls `respond_ckd(ckd_request, CKDResponse { big_y: [0u8;48].into(), big_c: [0u8;48].into() })`.
4. The contract passes the `AppPublicKey` arm (empty block), skips all validation, and calls `resolve_yields_for` — delivering the zeroed response to Alice.
5. Alice's yield resolves with `big_y = 0, big_c = 0`. She computes `sig = 0 - a·0 = 0`. The pairing check fails; her CKD request is permanently consumed with no valid key derived.
6. Repeating steps 1–5 for every resubmission constitutes a persistent, single-participant DoS of the CKD service for the legacy variant. [6](#0-5)
