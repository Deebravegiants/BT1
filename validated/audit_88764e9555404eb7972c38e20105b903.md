### Title
`respond_ckd` Skips Cryptographic Output Verification for `AppPublicKey` Variant, Allowing a Single Byzantine Participant to Corrupt CKD Responses - (File: `crates/contract/src/lib.rs`)

### Summary

`MpcContract::respond_ckd()` performs a cryptographic pairing check (`ckd_output_check`) on the submitted `CKDResponse` only when the request uses the `AppPublicKeyPV` variant. For the `AppPublicKey` (privately verifiable) variant, the arm is a no-op and the response is accepted and resolved unconditionally. A single attested participant — strictly below the signing threshold — can therefore submit an arbitrary `CKDResponse` for any pending `AppPublicKey` CKD request, bypassing the threshold-computation guarantee and delivering a corrupted encrypted key to the user.

### Finding Description

In `respond_ckd`, after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC network's master key and the user's `app_id`. [2](#0-1) 

For `AppPublicKey`, no equivalent check exists. The response values `big_y` and `big_c` are accepted as-is and immediately used to resolve the yield, delivering whatever the caller supplied to the waiting user.

The design rationale documented in the CKD spec is that `AppPublicKey` is "privately verifiable" — the user verifies correctness after decryption using their own secret scalar. However, this design intent does not prevent a malicious participant from front-running the honest leader's on-chain submission with a fabricated response. The contract has no mechanism to distinguish a legitimately computed response from an arbitrary one for this variant. [3](#0-2) 

The `respond` function for signature requests, by contrast, always verifies the signature against the derived public key before resolving: [4](#0-3) 

This creates a direct specification mismatch: signature responses are always cryptographically verified on-chain; CKD responses are only verified for one of the two request variants.

### Impact Explanation

A single Byzantine participant (one attested MPC node, well below the signing threshold) can:

1. Observe a pending `AppPublicKey` CKD request on-chain (the `CKDRequest` key — `app_public_key`, `app_id`, `domain_id` — is fully public).
2. Call `respond_ckd` with the correct `CKDRequest` key but with arbitrary `big_y` / `big_c` values before the honest leader submits the real response.
3. The contract accepts the fabricated response, resolves the yield, and delivers the corrupted `CKDResponse` to the user's callback.

The user receives an encrypted blob that does not correspond to any key derivable from the MPC network's master secret. Decryption with the user's private scalar produces garbage. The pending request is consumed and cannot be re-submitted with the same parameters while the yield is live, so the user's CKD request lifecycle is permanently corrupted for that invocation.

This breaks the core production safety invariant of the CKD subsystem: that the encrypted output delivered to the user is the threshold-computed encryption of the MPC-derived secret under the user's ephemeral key.

Impact classification: **Medium** — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation

Any single attested participant can execute this attack. The `CKDRequest` key is deterministic and observable on-chain from the `request_app_private_key` call. The attacker only needs to submit `respond_ckd` before the honest leader does, which is a straightforward NEAR transaction race. No threshold collusion, leaked keys, or privileged access beyond being an attested participant is required. [5](#0-4) 

### Recommendation

Apply the same cryptographic output check to `AppPublicKey` requests. The `AppPublicKey` variant carries only a G1 point (`pk1 = a·G1`), which is insufficient for the pairing check that requires a G2 component. Two options:

1. **Deprecate the unverifiable variant for on-chain use**: Require all new `request_app_private_key` calls to use `AppPublicKeyPV`, which provides the G2 component needed for on-chain verification. Existing `AppPublicKey` requests in flight can be grandfathered.
2. **Document and enforce the single-submitter trust assumption explicitly**: If the `AppPublicKey` variant is intentionally trust-delegated to the submitting participant, document this as a known trust assumption and add a comment in `respond_ckd` explaining why the empty arm is intentional, so future reviewers do not mistake it for a missing check.

Option 1 is strongly preferred because it closes the gap between the two variants and aligns the on-chain security model with the threshold guarantee users expect from the MPC network.

### Proof of Concept

```
1. User calls request_app_private_key({
       derivation_path: "mykey",
       app_public_key: AppPublicKey(pk1),   // privately verifiable variant
       domain_id: 2
   })
   → pending_ckd_requests[CKDRequest { app_public_key: AppPublicKey(pk1),
                                        app_id: derive_app_id(user, "mykey"),
                                        domain_id: 2 }] = [yield_id]

2. Attacker (single attested participant) observes the pending request on-chain.

3. Attacker calls respond_ckd(
       request = CKDRequest { app_public_key: AppPublicKey(pk1),
                               app_id: derive_app_id(user, "mykey"),
                               domain_id: 2 },
       response = CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }  // arbitrary garbage
   )

4. respond_ckd hits the AppPublicKey arm → no-op → resolve_yields_for resolves
   the yield with the garbage response.

5. User's callback receives CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }.
   Decryption with the user's private scalar a produces:
       confidential_key = big_c - a * big_y = 0 - 0 = 0  (not the MPC-derived secret)

6. Honest leader's subsequent respond_ckd call finds no pending request and fails.
   The user's CKD request is permanently resolved with corrupted output.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
