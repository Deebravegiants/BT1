### Title
Node Sends Zero Deposit to `submit_participant_info`, Permanently Blocking Automatic TEE Attestation Onboarding for New Participants - (File: `crates/node/src/tee/remote_attestation.rs`)

---

### Summary

The MPC node's `submit_remote_attestation` function dispatches `submit_participant_info` transactions with **zero attached deposit**. The contract enforces a storage-staking deposit for every new participant's first attestation submission. Because the node never attaches any deposit, every automatic attestation attempt by a new node is unconditionally rejected by the contract with `InsufficientDeposit`, permanently blocking automatic network onboarding. This is the direct analog of the Uniswap `amountInMax = 0` bug: a hardcoded zero for a required payment parameter causes a mandatory operation to always revert.

---

### Finding Description

**Root cause — node side (`crates/node/src/tee/remote_attestation.rs`):**

`submit_remote_attestation` constructs a `ChainSendTransactionRequest::SubmitParticipantInfo` and dispatches it via `tx_sender.send_and_wait`. No deposit is attached at any point in this call path. [1](#0-0) 

The function retries indefinitely with exponential backoff (100 ms → 60 s, capped at 12 h), but because the root cause is a missing deposit rather than a transient network error, every retry fails identically. [2](#0-1) 

**Root cause — contract side (`crates/contract/src/lib.rs`):**

`submit_participant_info` measures storage consumed by the new attestation entry and requires the caller to cover it when either (a) the attestation is brand-new (`NewlyInsertedParticipant`) or (b) the caller is not yet a protocol participant. For every first-time joiner, **both** conditions are true simultaneously. [3](#0-2) 

With `attached_deposit = 0` and `cost > 0`, the check at line 835 always fails and the contract returns `InvalidParameters::InsufficientDeposit`.

**Explicit acknowledgement in production documentation:**

The operator guide documents this as a known defect and instructs operators to work around it manually:

> *"first-time joiners must attach enough yoctoNEAR for storage; **the node attaches `0`**, so call `submit_participant_info` manually with `--deposit` once. Exact amount tracked in [#903](https://github.com/near/mpc/issues/903)."* [4](#0-3) 

The same zero-deposit path is also exercised by `monitor_attestation_removal`, which calls `validate_and_submit_remote_attestation → submit_remote_attestation` when a node's attestation is evicted and must be resubmitted. [5](#0-4) 

---

### Impact Explanation

**Medium — participant-state manipulation that breaks production safety/accounting invariants.**

Every new MPC node is permanently unable to complete automatic TEE attestation onboarding. Without a valid on-chain attestation, a node cannot be voted into the participant set. If the network must expand or replace nodes (e.g., after a node failure or a TEE upgrade cycle that evicts existing attestations), the automatic resubmission path silently fails for all affected nodes. If enough nodes cannot maintain valid attestations, the active participant count can fall below the signing threshold, blocking all `sign` and `request_app_private_key` requests and freezing user funds locked in pending cross-chain operations. The failure is not transient: the retry loop runs for up to 12 hours and then terminates, leaving the node permanently unattestation.

---

### Likelihood Explanation

**High.** The defect is triggered unconditionally for every new node on its very first `submit_participant_info` call — no special attacker action is required. The production documentation explicitly acknowledges the bug and tracks it as an open issue (#903). Any network expansion event (new participant vote, node replacement, TEE software upgrade requiring re-attestation of non-participant nodes) will reproduce the failure automatically.

---

### Recommendation

Attach a sufficient deposit when constructing the `SubmitParticipantInfo` transaction. The simplest safe fix — directly analogous to the Uniswap mitigation of using `type(uint256).max` — is to attach a conservatively large deposit (e.g., enough to cover several kilobytes of storage at the current `storage_byte_cost`) and rely on the contract's existing refund logic to return any excess:

```rust
// In submit_remote_attestation or the tx_sender layer:
// Attach e.g. 0.1 NEAR; the contract refunds the unused portion.
let deposit = NearToken::from_millinear(100);
ChainSendTransactionRequest::SubmitParticipantInfo { args, deposit }
```

Alternatively, query `env::storage_byte_cost()` off-chain before submission and compute the exact required amount. Either way, the contract's refund path at lines 844–848 ensures no NEAR is permanently lost. [6](#0-5) 

---

### Proof of Concept

1. A new MPC node starts, generates a valid TEE attestation, and calls `periodic_attestation_submission`.
2. `validate_and_submit_remote_attestation` → `submit_remote_attestation` is invoked.
3. A `ChainSendTransactionRequest::SubmitParticipantInfo` is dispatched with **0 yoctoNEAR** attached.
4. The contract executes `submit_participant_info`:
   - `add_participant` succeeds and returns `NewlyInsertedParticipant`.
   - `caller_is_not_participant = true` (node is not yet a protocol participant).
   - `attestation_storage_must_be_paid_by_caller = true`.
   - `storage_used > 0`, so `cost > 0`.
   - `attached (0) < cost` → `Err(InsufficientDeposit)`.
5. The node receives `TransactionStatus::NotExecuted`, logs `"attestation submission was not executed"`, and retries.
6. Every retry reproduces step 3–5 identically. After 12 hours the retry loop times out and the node exits with an error, never having joined the network.
