Audit Report

## Title
Permissionless `sendFunds()` in FeeReceiver Enables Front-Running of MEV Reward Distribution to Steal Unclaimed Yield - (File: contracts/FeeReceiver.sol)

## Summary

`FeeReceiver.sendFunds()` has no access control, allowing any external caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at will. An attacker who deposits ETH before calling `sendFunds()` and then calls `LRTOracle.updateRSETHPrice()` to update the stored exchange rate can initiate a withdrawal at the inflated price, capturing a disproportionate share of rewards that should have accrued to long-standing rsETH holders.

## Finding Description

`FeeReceiver.sendFunds()` is declared `external` with no role check:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

The receiving function in `LRTDepositPool` is also unrestricted:

```solidity
// contracts/LRTDepositPool.sol L61
function receiveFromRewardReceiver() external payable { }
```

Once ETH lands in `LRTDepositPool`, it is immediately counted in `getETHDistributionData()` via `address(this).balance` (L480), which feeds `getTotalAssetDeposits` → `_getTotalEthInProtocol()` in `LRTOracle`. The public `updateRSETHPrice()` function (L87-89 of `LRTOracle.sol`) recomputes and stores the new `rsETHPrice` from live balances. Since `rsETHPrice` is a stored state variable (not computed on-the-fly), the price only updates when `updateRSETHPrice()` is explicitly called — but that function is also public and permissionless.

The complete exploit path:
1. Attacker deposits ETH via `LRTDepositPool.depositETH()` at the current stored `rsETHPrice`, receiving rsETH.
2. Attacker calls `FeeReceiver.sendFunds()`, pushing accumulated MEV rewards into the pool.
3. Attacker calls `LRTOracle.updateRSETHPrice()`, which recomputes the price from the now-larger pool balance and stores the higher value.
4. Attacker calls `LRTWithdrawalManager.initiateWithdrawal()`, which locks in `expectedAssetAmount` using the new higher `rsETHPrice` (via `getExpectedAssetAmount` → `lrtOracle.rsETHPrice()`).
5. After the withdrawal delay, attacker calls `completeWithdrawal()` and receives more ETH than deposited.

The `_calculatePayoutAmount` function in `LRTWithdrawalManager` returns `min(expectedAssetAmount, currentReturn)`, so locking in the higher `expectedAssetAmount` at step 4 is the key to capturing the gain.

Existing checks are insufficient:
- `pricePercentageLimit` in `_updateRsETHPrice()` could block the price update if the jump is too large, but it defaults to 0 (no limit) and is ineffective for small reward-to-TVL ratios.
- The protocol fee mechanism in `_updateRsETHPrice()` takes a cut of rewards but does not prevent the attacker from capturing the remainder.
- The withdrawal delay reduces profitability but does not eliminate it, especially for large accumulated FeeReceiver balances.

## Impact Explanation

**High — Theft of unclaimed yield.** Long-standing rsETH holders accumulate yield from MEV/execution-layer rewards held in `FeeReceiver`. By front-running the reward flush, an attacker captures a fraction of those rewards proportional to their deposit relative to total TVL. The stolen yield is concrete and quantifiable (see PoC). This matches the allowed impact: *High. Theft of unclaimed yield.*

## Likelihood Explanation

- `FeeReceiver` accumulates ETH passively and its balance is publicly visible on-chain.
- `sendFunds()` requires no privilege, no signature, and no special state.
- `LRTOracle.updateRSETHPrice()` is public and permissionless.
- `LRTDepositPool.depositETH()` is open to any caller.
- The only friction is the withdrawal delay (~8 days default) and capital requirement, which reduce but do not eliminate profitability for large accumulated balances.
- **Likelihood: Medium** — requires capital and patience through the withdrawal delay, but is otherwise trivially executable by any on-chain observer.

## Recommendation

1. Add an access-control modifier to `sendFunds()` so only a trusted operator or keeper role can trigger it:
   ```solidity
   function sendFunds() external onlyRole(LRTConstants.MANAGER) { ... }
   ```
2. Optionally, also restrict `receiveFromRewardReceiver()` in `LRTDepositPool` to only accept calls from the registered `FeeReceiver` address, preventing direct ETH injection.
3. Consider a deposit fee or a minimum holding period before withdrawal to make the attack economically unattractive even if `sendFunds()` remains permissionless.

## Proof of Concept

**Setup:** FeeReceiver holds 100 ETH. Total protocol ETH = 10,000 ETH. rsETH supply = 9,500 rsETH. Stored `rsETHPrice` ≈ 1.0526 ETH/rsETH. `pricePercentageLimit` = 0.

1. **Attacker deposits 10,000 ETH** via `LRTDepositPool.depositETH()`. At the stored price they receive `10,000 / 1.0526 ≈ 9,500 rsETH`. Total ETH = 20,000, rsETH supply = 19,000, stored price unchanged.

2. **Attacker calls `FeeReceiver.sendFunds()`**. 100 ETH flows into the deposit pool. Total ETH = 20,100, rsETH supply = 19,000. Stored price still 1.0526 (not yet updated).

3. **Attacker calls `LRTOracle.updateRSETHPrice()`**. New stored price = `20,100 / 19,000 ≈ 1.0579 ETH/rsETH` (after protocol fee deduction, slightly less, but still higher).

4. **Attacker calls `LRTWithdrawalManager.initiateWithdrawal(ETH, 9500_rsETH)`**. `expectedAssetAmount` is locked at `9,500 × 1.0579 ≈ 10,050 ETH`.

5. **After withdrawal delay**, attacker calls `completeWithdrawal()` and receives ≈ 10,050 ETH.

6. **Net gain:** ~50 ETH captured from the 100 ETH reward that should have accrued pro-rata to the 9,500 pre-existing rsETH holders.

**Foundry fork test plan:**
- Fork mainnet, deploy/configure contracts.
- Seed `FeeReceiver` with 100 ETH.
- Attacker deposits 10,000 ETH, records rsETH received.
- Attacker calls `sendFunds()` then `updateRSETHPrice()`.
- Attacker initiates and completes withdrawal after delay.
- Assert attacker ETH balance > initial deposit, and existing holder's redeemable value decreased proportionally.
