### Title
Caller-Controlled `confirmations: 0` in `verify_foreign_transaction` Bypasses Bitcoin Finality Guarantee — (File: crates/contract/src/lib.rs)

---

### Summary

`verify_foreign_transaction` accepts a `BitcoinRpcRequest` with a fully caller-controlled `confirmations: BlockConfirmations` field and performs no minimum-value check. An unprivileged caller can set `confirmations: 0`, causing every MPC node to accept any Bitcoin transaction regardless of its actual on-chain confirmation depth and produce a valid threshold signature over it. Because the `confirmations` value is embedded verbatim in the signed payload, a bridge contract that does not independently re-enforce a minimum confirmation floor will treat the MPC signature as proof of finality for an unconfirmed or minimally-confirmed transaction, enabling double-spend conditions.

---

### Finding Description

`verify_foreign_transaction` in `crates/contract/src/lib.rs` validates only:
1. Domain purpose is `ForeignTx`.
2. The requested chain is in the supported-chains list.
3. Gas and deposit requirements. [1](#0-0) 

It performs **no validation** of the `confirmations` field inside `BitcoinRpcRequest`. The `BlockConfirmations` type is a plain `u64` with a JSON-schema minimum of `0.0`: [2](#0-1) 

The `BitcoinRpcRequest` struct stores `confirmations` as an unconstrained field: [3](#0-2) 

When a node processes the request in `execute_foreign_chain_request`, it passes the caller-supplied value directly to `BitcoinInspector::extract` as `block_confirmations_threshold`: [4](#0-3) 

The inspector's finality check is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [5](#0-4) 

With `block_confirmations_threshold = 0`, the condition `0 <= any_value` is always `true`. The inspector proceeds to extract values and the MPC nodes sign the payload `ForeignTxSignPayloadV1 { request: (tx_id, confirmations=0, ...), values: [...] }`.

The `ForeignChainRpcWhitelist` / `ChainEntry` governance structure stores only provider lists and RPC quorum — no per-chain minimum confirmation floor is voted in or enforced anywhere: [6](#0-5) 

---

### Impact Explanation

The signed payload commits to `confirmations=0`. The SDK's `ForeignChainSignatureVerifier::verify_signature` only verifies the cryptographic signature against the payload hash — it does not enforce a minimum `confirmations` value: [7](#0-6) 

The feature's stated primary use case is the Omnibridge inbound flow: *"Chain Signatures are required to attest that a foreign transaction finalized successfully."* A bridge contract built on this guarantee may reasonably assume the MPC network enforces a minimum confirmation depth and omit its own check. Such a contract will accept a `confirmations=0` MPC signature as proof of Bitcoin finality, allowing an attacker to claim NEAR-side funds before the Bitcoin transaction is irreversible and then reorganize/double-spend it.

**Impact: High** — light-client-style verification bypass causing invalid bridge execution or double-spend conditions.

---

### Likelihood Explanation

Any unprivileged NEAR account can call `verify_foreign_transaction` with `confirmations: 0`. No special privileges, collusion, key material, or operator misconfiguration are required. The `BlockConfirmations` type accepts `0` as a valid value at the JSON deserialization layer. The attack is trivially executable by any caller on a supported Bitcoin chain.

**Likelihood: High.**

---

### Recommendation

Enforce a minimum `confirmations` value in `verify_foreign_transaction` before the request is enqueued. The minimum should be governed on-chain — either:

1. **Extend `ChainEntry`** (in `ForeignChainRpcWhitelist`) with a `min_confirmations: u64` field voted in alongside the provider list, and reject requests below that floor in `verify_foreign_transaction`; or
2. **Hardcode a protocol-level minimum** (e.g., `confirmations >= 1`) directly in the contract as a short-term guard.

Apply the analogous fix for EVM chains: reject `EvmFinality::Latest` (which provides no reorg protection) unless the governance configuration explicitly permits it.

---

### Proof of Concept

```json
// Attacker calls verify_foreign_transaction with:
{
  "request": {
    "domain_id": <valid_ForeignTx_domain_id>,
    "payload_version": "V1",
    "request": {
      "Bitcoin": {
        "tx_id": "<attacker_tx_id>",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    }
  }
}
```

1. Contract accepts the request — no `confirmations` validation at `crates/contract/src/lib.rs:526-542`.
2. MPC nodes call `execute_foreign_chain_request`; `chain_is_supported` passes (Bitcoin is whitelisted).
3. `BitcoinInspector::extract` is called with `block_confirmations_threshold = 0`; `0 <= actual_confirmations` is always `true` — the inspector accepts the transaction even with 1 confirmation.
4. Nodes sign `ForeignTxSignPayloadV1 { request: (tx_id, confirmations=0, [BlockHash]), values: [BlockHash(<hash>)] }`.
5. Attacker receives a valid MPC threshold signature.
6. Attacker submits the signature to a bridge contract that trusts the MPC network to enforce finality.
7. Bridge releases NEAR-side funds.
8. Attacker double-spends the Bitcoin transaction (1-confirmation reorg is feasible for a well-resourced attacker).
