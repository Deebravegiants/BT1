### Title
`directDeposits` Excluded from Preservation-of-Value Check Enables ADA Creation from Thin Air — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in transaction bodies that adds ADA directly to registered account balances. However, this amount is not included on either side of the preservation-of-value (consumed == produced) check. An unprivileged transaction author can include arbitrarily large `directDeposits` in a balanced transaction, causing account balances to increase without any corresponding reduction in UTxO outputs — creating ADA from thin air.

---

### Finding Description

**Root cause — `directDeposits` applied but not accounted for in value conservation:**

The Dijkstra era `ENTITIES` rule (`dijkstraEntitiesTransition`) applies `directDeposits` to account balances:

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts

pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [1](#0-0) 

The only validation is that target accounts exist. There is no check that the transaction provides sufficient UTxO inputs to fund the deposits.

`applyDirectDeposits` unconditionally adds ADA to account balances:

```haskell
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [2](#0-1) 

**`directDeposits` is absent from both sides of the value-conservation equation:**

The Dijkstra consumed-value function aggregates UTxO inputs, withdrawals, and deposit refunds — no `directDeposits`:

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels txBody
    (\topTxBody -> txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody)
    txBodyConsumedValue
  where
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
``` [3](#0-2) 

The Dijkstra produced-value function aggregates UTxO outputs, fees, and certificate deposits — no `directDeposits`:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)
``` [4](#0-3) 

`conwayProducedValue` is a Conway-era function that knows nothing about the Dijkstra-era `directDeposits` field. `dijkstraProducedValue` does not add `directDeposits` on top of it.

The preservation-of-value check (`validateValueNotConservedUTxO`) therefore never sees `directDeposits`:

```haskell
validateValueNotConservedUTxO pp utxo certState txBody =
  failureUnless (consumedValue == producedValue) $
    ValueNotConservedUTxO Mismatch {mismatchSupplied = consumedValue, mismatchExpected = producedValue}
  where
    consumedValue = consumed pp certState utxo txBody
    producedValue = produced pp certState txBody
``` [5](#0-4) 

The analogous `treasuryDonation` field (introduced in Conway) is included in `conwayProducedValue`, confirming the established pattern: any new ledger-pot credit must appear on the produced side. `directDeposits` breaks this pattern.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker can craft a transaction whose UTxO inputs and outputs balance normally (consumed == produced for the UTxO portion), while simultaneously crediting an arbitrary amount of ADA to their own registered account addresses via `directDeposits`. Because `directDeposits` is invisible to the value-conservation check, the transaction is accepted as valid. The attacker's account balance grows by the deposited amount with no corresponding deduction anywhere in the ledger state. This violates the fundamental preservation-of-value invariant and constitutes unbounded ADA creation.

---

### Likelihood Explanation

**High.** The entry path requires only that the attacker:
1. Register one or more stake credentials (standard, permissionless operation).
2. Construct a Dijkstra-era transaction with a balanced UTxO body and a non-zero `directDeposits` map pointing to their own accounts.
3. Submit the transaction.

No privileged access, governance majority, or key compromise is required. The `directDeposits` field is a standard, serializable part of the Dijkstra transaction body, reachable by any transaction author.

---

### Recommendation

Include the total `directDeposits` amount on the **produced** side of the value-conservation equation, mirroring how `treasuryDonation` and certificate deposits are handled. Concretely, `dijkstraProducedValue` (or a Dijkstra-specific override of `getProducedValue`) should add:

```haskell
<> Val.inject (foldMap id . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
```

This ensures that every lovelace credited to an account via `directDeposits` must be funded by a corresponding UTxO input, preserving the ledger's accounting invariant.

---

### Proof of Concept

1. Register account address `A` for credential `k` (standard `RegTxCert`).
2. Construct a Dijkstra `TopTx` with:
   - UTxO inputs totalling `X` lovelace.
   - UTxO outputs totalling `X − fee` lovelace.
   - `fee` set to the minimum fee.
   - `directDeposits = { A → Y }` for any `Y > 0`.
3. Submit the transaction.
4. **Preservation-of-value check**: consumed = `X`, produced = `(X − fee) + fee = X`. Check passes.
5. **`ENTITIES` rule**: `directDepositsMissingAccounts` finds `A` is registered; `applyDirectDeposits` adds `Y` lovelace to `A`'s balance.
6. **Result**: Account `A` now holds `Y` lovelace that was never deducted from any UTxO or other ledger pot. Repeating this across many transactions allows unbounded ADA inflation. [4](#0-3) [1](#0-0) [6](#0-5)
