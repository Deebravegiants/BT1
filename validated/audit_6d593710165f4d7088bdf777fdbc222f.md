### Title
Propose-Update Deposits Permanently Frozen When a Competing Proposal Wins — (`File: crates/contract/src/update.rs`)

### Summary
When `propose_update` is called, a storage deposit proportional to the uploaded contract code size is collected from the proposer. The `UpdateEntry` struct does not record the proposer's account ID or the deposit amount. When `do_update` executes on threshold, it bulk-clears **all** pending entries and votes without issuing any refunds. Every deposit paid for a non-winning proposal is permanently locked in the contract with no recovery path.

### Finding Description
`propose_update` in `crates/contract/src/lib.rs` collects a deposit from the caller:

```rust
let required = ProposedUpdates::required_deposit(&update);
// ... stores only the excess back, keeps `required`
let id = self.proposed_updates.propose(update);
``` [1](#0-0) 

The `propose` helper inserts an `UpdateEntry` that stores only the update payload and the byte count — **no proposer account ID, no deposit amount**:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
``` [2](#0-1) 

When threshold votes are reached, `do_update` removes the winning entry and then bulk-clears every other pending entry and all votes:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ... no refund loop, no proposer lookup
``` [3](#0-2) 

Because neither the proposer's `AccountId` nor the deposit amount is stored in `UpdateEntry`, there is no information available to issue refunds to the owners of the cleared proposals. The storage those entries occupied is freed, but the NEAR tokens that were deposited to cover that storage remain in the contract balance permanently.

There is no `cancel_proposal`, `withdraw_proposal`, or equivalent function in the contract. The only deposit-related path is `remove_update_vote`, which removes a vote record but does not touch entries or deposits. [4](#0-3) 

### Impact Explanation
Every participant who proposed an update that was not selected loses their storage deposit permanently. For a contract-code update, the deposit is proportional to the WASM binary size (typically several hundred KB to several MB), which translates to several NEAR tokens per proposal. In a normal governance round where multiple participants each propose a version (as shown in sandbox tests where every participant proposes independently), all but one proposer's deposit is frozen with no recovery path. This breaks the accounting invariant that storage deposits are refundable when the storage they cover is freed.

This maps to the allowed impact: **Medium — balance/accounting invariant broken without relying on network-level DoS or operator misconfiguration.**

### Likelihood Explanation
The scenario is a normal part of the governance lifecycle. The sandbox tests explicitly show multiple participants each calling `propose_update` with their own deposit before votes converge on one proposal. Every successful contract upgrade where more than one proposal exists triggers this loss. Likelihood is **High** given the routine multi-proposer governance pattern. [5](#0-4) 

### Recommendation
Store the proposer's `AccountId` and the deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the refund pattern already used in `propose_update` for excess deposits and in `require_deposit` for sign/CKD requests. [6](#0-5) 

### Proof of Concept
1. Participant A calls `propose_update` with a 500 KB WASM, attaching the required deposit (~5 NEAR).
2. Participant B calls `propose_update` with a different 500 KB WASM, attaching ~5 NEAR.
3. Threshold participants vote for proposal B; `vote_update` calls `do_update(&B_id, ...)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry is deleted.
5. A's ~5 NEAR deposit remains in the contract balance. There is no function to recover it.
6. Repeat across multiple governance rounds: the contract accumulates permanently frozen NEAR from every non-winning proposer. [3](#0-2) [7](#0-6)
