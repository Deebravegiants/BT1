### Title
Missing On-Chain Verification for `CKDAppPublicKey::AppPublicKey` Variant in `respond_ckd()` Allows Single Attested Participant to Forge Confidential Key Derivation Output — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` in the MPC smart contract performs a cryptographic output check only for the `AppPublicKeyPV` (publicly verifiable) variant of `CKDAppPublicKey`. The `AppPublicKey` (privately verifiable, default) variant is silently accepted with **no on-chain verification whatsoever**. A single Byzantine attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will resolve the yield and deliver the attacker-controlled key material to the user, completely bypassing the threshold requirement.

---

### Finding Description

In `respond_ckd()`, after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKey` arm is an empty no-op. The contract then unconditionally calls `resolve_yields_for`, which removes the pending request from the map and resumes every queued yield with the supplied (unverified) response bytes: [2](#0-1) [3](#0-2) 

By contrast, `respond()` (for signatures) always verifies the cryptographic output against the domain public key for every scheme it handles: [4](#0-3) 

The `AppPublicKey` variant is the **default** usage path. The CLI and documentation show it as the primary non-`--publicly-verifiable` mode: [5](#0-4) 

The `ckd_output_check` function that is only applied to `AppPublicKeyPV` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC network's public key and the user's app identity: [6](#0-5) 

No equivalent binding exists for `AppPublicKey` requests.

---

### Impact Explanation

The CKD protocol is a threshold operation: no single node should be able to determine the output. For `AppPublicKey` requests, this guarantee is entirely absent at the contract layer. A single attested participant can:

1. Observe any pending `AppPublicKey` CKD request in the contract state.
2. Call `respond_ckd` with an arbitrary `CKDResponse` (`big_y`, `big_c` of their choosing).
3. The contract accepts it, removes the request from `pending_ckd_requests`, and resumes all queued yields with the forged bytes.
4. The user's TEE application receives attacker-controlled encrypted key material.

This is a **bypass of the threshold-signature/key-derivation requirement**: a single Byzantine participant, strictly below the signing threshold, unilaterally determines the CKD output for the default request variant. The user cannot distinguish a forged response from a legitimate one at the contract level (the `AppPublicKey` variant is designed for client-side verification, but the contract is supposed to guarantee the response originates from the threshold computation).

**Impact class:** Critical — unauthorized confidential key derivation output without required participant authorization; bypass of threshold requirements.

---

### Likelihood Explanation

- The `AppPublicKey` variant is the default and most common usage path (no `--publicly-verifiable` flag required).
- Any single attested participant can exploit this — no collusion, no threshold, no special privilege beyond being an active participant.
- The attacker only needs to race `respond_ckd` before the honest nodes do. Since the contract resolves on the first valid call, the attacker wins by being first.
- The exploit requires no off-chain capability beyond being an attested MPC node.

---

### Recommendation

Apply the same cryptographic output check to `AppPublicKey` requests. For the non-PV variant, the check must be adapted: since `AppPublicKey` only provides a G1 point (`pk1`) without a G2 counterpart (`pk2`), the contract cannot run the full pairing check. The fix is one of:

1. **Require `AppPublicKeyPV` for all on-chain CKD requests**, deprecating the unverifiable `AppPublicKey` variant entirely.
2. **Extend `AppPublicKey` to include a G2 component** (making it equivalent to `AppPublicKeyPV`) so the pairing check can be applied uniformly.
3. **Add a threshold-vote mechanism** for `AppPublicKey` responses (analogous to `vote_pk`/`vote_reshared`) so that a quorum of participants must agree on the response before it is accepted.

The minimal safe fix is option 1: reject `AppPublicKey` requests in `respond_ckd` until a verifiable variant is used, matching the pattern in `respond()` where every scheme is verified before acceptance.

---

### Proof of Concept

```
1. Contract is in Running state with participants [A, B, C], threshold = 2.
2. User submits request_app_private_key with AppPublicKey variant (the default).
   → pending_ckd_requests now contains the request.
3. Participant A (Byzantine, acting alone) calls:
     respond_ckd(request, CKDResponse { big_y: [attacker_g1_1], big_c: [attacker_g1_2] })
   where big_y and big_c are arbitrary G1 points chosen by A.
4. respond_ckd executes:
     match &request.app_public_key {
         AppPublicKey(_) => {}   // ← no check, falls through
     }
     resolve_yields_for(...)    // ← accepts and delivers forged response
5. The user's yield is resumed with attacker-controlled (big_y, big_c).
6. The user's TEE app decrypts garbage / attacker-chosen key material.
7. Participants B and C never participated; threshold was never reached.
```

The `AppPublicKey` arm at line 676 is the exact missing-case analog to the unreachable `0xF` opcode in the reference report: a variant that is present in the type but absent from the verification filter, allowing an unintended (unverified) code path to complete successfully. [1](#0-0)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
