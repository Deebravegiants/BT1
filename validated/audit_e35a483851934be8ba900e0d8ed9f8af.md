### Title
Byzantine New Participant Can Forge `AppPublicKey` CKD Responses During Resharing Window — (`crates/contract/src/lib.rs`)

### Summary

During resharing, `respond_ckd` authorizes callers via `assert_caller_is_attested_participant_and_protocol_active`, which calls `active_participants()`. In `Resharing` state, `active_participants()` returns the **new** participant set. For `CKDAppPublicKey::AppPublicKey` requests, there is no cryptographic output check. A Byzantine new participant — who does not hold the old key shares — can therefore call `respond_ckd` with arbitrary `(big_y, big_c)` values for any in-flight `AppPublicKey` CKD request, resolving it with attacker-controlled output.

### Finding Description

**Step 1 — `active_participants()` returns NEW participants during resharing.** [1](#0-0) 

During `Resharing`, the function returns `state.resharing_key.proposed_parameters().participants()` — the incoming participant set, not the old one.

**Step 2 — `assert_caller_is_attested_participant_and_protocol_active` uses `active_participants()`.** [2](#0-1) 

The guard checks only that the caller is an attested member of `active_participants()`. During resharing, this is the new set.

**Step 3 — `respond_ckd` calls this guard and then has a missing output check for `AppPublicKey`.** [3](#0-2) 

The match arm for `CKDAppPublicKey::AppPublicKey(_)` is an empty block — no cryptographic verification is performed. The `AppPublicKeyPV` arm calls `ckd_output_check`, which enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`. [4](#0-3) 

**Step 4 — `resolve_yields_for` unconditionally resumes all queued yields with the attacker-supplied bytes.** [5](#0-4) 

Once the attacker calls `respond_ckd` with the correct `CKDRequest` key (which is public, derived deterministically from `predecessor_id` and `derivation_path`) and arbitrary `CKDResponse`, the pending yield is resolved and the caller receives the forged `(big_y, big_c)`.

### Impact Explanation

The `AppPublicKey` CKD variant relies entirely on participant authorization to guarantee response integrity — there is no on-chain cryptographic binding between the response and the network's master secret key. A Byzantine new participant who does not hold the old key shares can:

1. Observe a pending `AppPublicKey` CKD request (the request key is deterministic and public).
2. During the resharing window, call `respond_ckd` with attacker-chosen `big_y = g1·r` and `big_c = g1·s` for arbitrary scalars `r, s`.
3. The contract resolves the yield; the requesting client receives `(big_y, big_c)` derived from attacker-controlled values.
4. Because the attacker chose `r` and `s`, they can compute the same derived key the client will use — this is an unauthorized confidential key derivation output.

This maps to the Critical allowed impact: **"Unauthorized … confidential key derivation output without the required participant authorization."**

### Likelihood Explanation

The resharing window is a normal operational event (participant rotation). Any single new participant who is attested but Byzantine can execute this attack unilaterally — no threshold collusion is required. The target request key is fully deterministic and observable on-chain. The attack requires only one malicious new participant and one in-flight `AppPublicKey` CKD request during the resharing window.

### Recommendation

Two complementary fixes:

1. **Restrict `respond_ckd` to OLD participants during resharing.** The function should check membership in `previous_running_state.parameters.participants()` when the state is `Resharing`, not `active_participants()`. This mirrors how `public_key_extended` already uses the old keyset during resharing. [6](#0-5) 

2. **Require `AppPublicKeyPV` for all new CKD requests, or add an equivalent output check for `AppPublicKey`.** The `AppPublicKey` variant provides no on-chain verifiability; if it must be retained, the contract should at minimum reject `respond_ckd` calls for `AppPublicKey` requests during resharing.

### Proof of Concept

```
1. Deploy contract in Running state with participant set {A, B, C} (old).
2. Submit a CKD request using CKDAppPublicKey::AppPublicKey(...) → yields a pending entry in pending_ckd_requests.
3. Transition to Resharing with new participant set {D, E, F}.
4. Attest participant D (new, does not hold old key shares).
5. As D, call respond_ckd(request, CKDResponse { big_y: garbage_g1, big_c: garbage_g1 }).
6. Assert: call succeeds (no panic, no error).
7. Assert: the yield is resolved with the garbage response.
8. Assert: old participants A, B, C can no longer respond (RequestNotFound).
```

The `AppPublicKey` arm at line 676 is an empty block, so step 5 passes the output check unconditionally. The participant check at line 666 passes because D is in `active_participants()` (new set) during resharing.

```rust
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
```

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
