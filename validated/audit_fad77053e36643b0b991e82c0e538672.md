### Title
Missing `address(0)` owner validation in `MetricOmmPool.addLiquidity` permanently locks deposited tokens - (File: metric-core/contracts/MetricOmmPool.sol)

### Summary
`MetricOmmPool.addLiquidity` accepts `owner = address(0)` without any zero-address guard. Because `removeLiquidity` enforces `msg.sender == owner`, and `msg.sender` is always non-zero in the EVM, any tokens deposited under `address(0)` as position owner are permanently irrecoverable from the pool.

### Finding Description
The core pool's `addLiquidity` entry point takes an arbitrary `owner` address and credits LP shares to `_positionBinShares[keccak256(abi.encode(owner, salt, binIdx))]` and increments `_binTotalShares[binIdx]`. No check is performed to reject `owner == address(0)`. [1](#0-0) 

The `removeLiquidity` path enforces:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [2](#0-1) 

Because the EVM guarantees `msg.sender != address(0)` for every call, the `NotPositionOwner` revert is unconditional when `owner` is `address(0)`. The shares and the underlying tokens they represent are permanently stranded.

The periphery helper `MetricOmmPoolLiquidityAdder` does guard against this via `_validateOwner`:

```solidity
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
``` [3](#0-2) 

However, the periphery is not the only entry point. Any EOA or contract can call `MetricOmmPool.addLiquidity` directly, bypassing the periphery guard entirely. The core pool is the authoritative contract and must enforce this invariant itself. [4](#0-3) 

### Impact Explanation
Tokens deposited under `owner = address(0)` are permanently locked inside the pool. The depositor suffers a direct, total loss of the principal they transferred via the `metricOmmModifyLiquidityCallback`. The `_binTotalShares` counter is inflated by the orphaned shares, which dilutes the fractional claim of every other LP in that bin until those shares are the last ones remaining — at which point the underlying tokens are unrecoverable by anyone.

### Likelihood Explanation
The trigger is reachable by any unprivileged caller who interacts with the core pool directly rather than through the periphery router. Integrators building custom routers, scripts, or on-chain adapters that call `MetricOmmPool.addLiquidity` directly and pass `address(0)` as `owner` (e.g., as a placeholder or by omission) will silently lose funds with no on-chain warning. The periphery guard creates a false sense of security that the core is also protected.

### Recommendation
Add a zero-address check at the top of `MetricOmmPool.addLiquidity`, mirroring what the periphery already does:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
+   require(owner != address(0), InvalidPositionOwner());
    if (deltas.binIdxs.length == 0) return (0, 0);
    ...
}
```

The `InvalidPositionOwner` error already exists in the periphery interface; it should be promoted to the core pool interface or a new equivalent error added there.

### Proof of Concept

```solidity
// Attacker/integrator calls the core pool directly, bypassing periphery
pool.addLiquidity(
    address(0),   // owner — zero address, no revert
    salt,
    deltas,
    callbackData, // callback pays real tokens
    ""
);

// Tokens are now in the pool, shares credited to address(0).
// Attempt to recover:
pool.removeLiquidity(
    address(0),   // owner
    salt,
    deltas,
    ""
);
// ↑ Always reverts: NotPositionOwner()
//   because msg.sender != address(0) is always true.
// Tokens are permanently locked.
``` [2](#0-1) [3](#0-2)
