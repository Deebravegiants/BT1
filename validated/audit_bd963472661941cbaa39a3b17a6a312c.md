### Title
Dev Measurements Accepted in Production Attestation Verification — (`crates/mpc-attestation/src/attestation.rs`)

### Summary

`default_measurements()` compiles both production and dev `ExpectedMeasurements` into the binary. The production contract seeds its `AllowedMeasurements` from this function on migration. As a result, `verify_any_measurements` accepts dev-TEE attestations in production, breaking the invariant that only production-grade nodes may join the participant set.

### Finding Description

`default_measurements()` returns a static slice of two entries: [1](#0-0) 

Both `assets/tcb_info.json` (prod) and `assets/tcb_info_dev.json` (dev) are compiled in. The `TODO(#1433)` comment explicitly acknowledges this as an unresolved security issue.

`verify_any_measurements` iterates the accepted set and returns `Ok` on the first match: [2](#0-1) 

Because dev measurements are in the accepted set, a node presenting dev-TEE measurements passes this check.

The contract's `AllowedMeasurements` is documented as being seeded from `default_measurements()` on migration: [3](#0-2) 

This means any migrated production contract carries dev measurements in its on-chain allowed set.

Additionally, the node-side pre-submission validation is **completely bypassed** — the result of `validate_remote_attestation` is discarded with `let _ = ...`, and submission proceeds regardless: [4](#0-3) 

### Impact Explanation

A node running a dev TEE image (with dev MRTD/RTMRs) can submit a `DstackAttestation` that passes `verify_any_measurements` because the dev `ExpectedMeasurements` are in the contract's allowed set. This allows a dev-grade node to be accepted as a valid participant, breaking the invariant that only production-approved TEE nodes may participate in threshold signing, key generation, or resharing.

### Likelihood Explanation

The preconditions are:
1. The production contract has been migrated (seeding dev measurements) — likely given the migration path described in the code.
2. The attacker controls a node with TDX hardware running a dev image.
3. The attacker's node uses the same MPC docker image hash as production (so `verify_mpc_hash` passes).

Condition 3 is the main practical constraint. If the production MPC docker image is publicly known (e.g., published image), an attacker with TDX hardware can run it under a dev TEE configuration. The node-side validation being silently discarded removes any last-resort defense.

### Recommendation

- Remove `include_measurements!("assets/tcb_info_dev.json")` from `default_measurements()` in production builds (resolve TODO #1433). Use a compile-time feature flag (`#[cfg(feature = "dev")]`) to gate dev measurements.
- Make `validate_and_submit_remote_attestation` return an error instead of logging and continuing when `validate_remote_attestation` fails.
- Add a contract-level invariant check that rejects any `AllowedMeasurements` entry that matches known dev measurement hashes.

### Proof of Concept

1. Build the MPC node binary with the production MPC docker image but under a dev TDX configuration (dev MRTD/RTMRs matching `assets/tcb_info_dev.json`).
2. Submit a `DstackAttestation` with the dev measurements to the production contract's `submit_participant_info` endpoint.
3. The contract calls `verify_any_measurements` with `default_measurements()` (or the migrated allowed set seeded from it).
4. The dev `ExpectedMeasurements` entry matches → `AcceptedDstackAttestation` is returned → the node is accepted into the participant set.
5. Assert the node appears in the contract's TEE accounts list despite running a dev image.

---

The vulnerability is real and locally testable. The root cause is confirmed at `crates/mpc-attestation/src/attestation.rs` lines 259–266, with the contract seeding path confirmed at `crates/contract/src/tee/measurements.rs` lines 97–108. The TODO #1433 comment is an explicit acknowledgment by the authors that this is an unresolved security gap.

    fn verify_any_measurements(
        &self,
        report_data: &TDReport10,
        tcb_info: &TcbInfo,
        accepted_measurements: &[ExpectedMeasurements],
    ) -> Result<ExpectedMeasurements, VerificationError> {
        for expected in accepted_measurements {
            if self
                .verify_static_rtmrs(report_data, tcb_info, expected)
                .is_ok()
                && self
                    .verify_key_provider_digest(tcb_info, &expected.key_provider_event_digest)
                    .is_ok()
            {
                return Ok(*expected); // found a valid match
            }
        }

        Err(VerificationError::WrongHash {
            name: "expected_measurements",
            expected: "one of the embedded TCB info sets (prod or dev)".into(),
            found: "none matched".into(),
        })
    }
