Audit Report

## Title
Push-Pattern ETH Transfer in `_processWithdrawalCompletion` Permanently Freezes Funds for Non-Payable Contract Withdrawers - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._processWithdrawalCompletion` uses a push pattern to deliver ETH via `_transferAsset`, which reverts with `EthTransferFailed` if the recipient contract cannot receive ETH. Because rsETH is burned and ETH is moved from `LRTUnstakingVault` in the prior `unlockQueue` transaction, a permanently failing `completeWithdrawal` leaves the ETH stranded in `LRTWithdrawalManager` with no admin recovery path. The user loses both their rsETH and their ETH.

## Finding Description
The withdrawal lifecycle spans two separate transactions:

**Transaction 1 — `unlockQueue` (operator-only):**
- Burns rsETH: `IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned)` [1](#0-0) 
- Moves ETH from vault: `unstakingVault.redeem(asset, assetAmountUnlocked)` [2](#0-1) 

After this transaction, rsETH is permanently burned and ETH sits in `LRTWithdrawalManager`.

**Transaction 2 — `completeWithdrawal` / `completeWithdrawalForUser`:**
Both call `_processWithdrawalCompletion`, which ends with: [3](#0-2) 

```solidity
_transferAsset(asset, user, request.expectedAssetAmount);
```

`_transferAsset` for ETH: [4](#0-3) 

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `user` is a contract without `receive()` (or with a reverting `receive()`), `sent = false` and the entire transaction reverts. The revert unwinds all state changes within that transaction (the `popFront`, `delete withdrawalRequests[requestId]`, and `unlockedWithdrawalsCount[asset]--`), so the request remains in the queue. Every subsequent retry fails identically.

**No recovery path exists:**
- `sweepRemainingAssets` is gated by `if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist()`. Since `unlockedWithdrawalsCount[asset]` can never decrement for the stuck request, this is permanently blocked for all users of that asset. [5](#0-4) 
- `completeWithdrawalForUser` (operator function) calls the same `_processWithdrawalCompletion` and hits the same revert. The NatDoc comment even acknowledges: *"Not expected to be used for ETH"*. [6](#0-5) 
- No function exists to redirect the payout to a different address or cancel an unlocked withdrawal.

## Impact Explanation
**Critical — Permanent freezing of funds.**

Once `unlockQueue` executes, the user's rsETH is irreversibly burned and the corresponding ETH is held in `LRTWithdrawalManager`. If `completeWithdrawal` permanently reverts, the ETH is frozen with no admin escape hatch. Additionally, `unlockedWithdrawalsCount[asset]` remains nonzero indefinitely, blocking `sweepRemainingAssets` for all users of that asset. This matches the allowed impact: *"Permanent freezing of funds."*

## Likelihood Explanation
**Medium.** Smart contract wallets (Gnosis Safe, multisigs, DAO treasuries) are primary holders of large rsETH positions and frequently lack a `receive()` function or implement one that conditionally reverts. The protocol performs no on-chain check at `initiateWithdrawal` time to verify the caller can receive ETH. No attacker is required — the scenario is triggered by ordinary protocol usage from a contract address. The condition is self-inflicted but irreversible once `unlockQueue` runs.

## Recommendation
Replace the push pattern for ETH with a pull pattern in `_processWithdrawalCompletion`. Instead of calling `_transferAsset(asset, user, request.expectedAssetAmount)` directly, record the owed amount in a mapping (e.g., `mapping(address => uint256) public pendingETHWithdrawals`) and emit an event. Provide a separate `claimETH()` function that the user calls to pull their ETH. A failed pull only affects the caller's own transaction and cannot freeze funds or block `sweepRemainingAssets`. For ERC-20 assets, the existing push pattern is safe since `safeTransfer` reverts are also retryable without the burned-token problem.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

contract NoReceive {
    // No receive() or fallback() — rejects all ETH

    function triggerFreeze(
        address withdrawalManager,
        address rsETH,
        uint256 rsETHAmount
    ) external {
        // Step 1: approve and initiate withdrawal
        IERC20(rsETH).approve(withdrawalManager, rsETHAmount);
        ILRTWithdrawalManager(withdrawalManager)
            .initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");

        // Step 2: operator calls unlockQueue externally
        //   -> rsETH is burned (separate tx)
        //   -> ETH moved to LRTWithdrawalManager (separate tx)

        // Step 3: completeWithdrawal always reverts with EthTransferFailed
        ILRTWithdrawalManager(withdrawalManager)
            .completeWithdrawal(ETH_TOKEN, "");
        // ^^^ ALWAYS REVERTS — ETH permanently frozen in LRTWithdrawalManager
    }
}
```

**Reproduction steps:**
1. Deploy `NoReceive`, fund it with rsETH.
2. Call `triggerFreeze` — `initiateWithdrawal` succeeds.
3. Operator calls `unlockQueue` — rsETH burned, ETH moved to `LRTWithdrawalManager`.
4. Any call to `completeWithdrawal` or `completeWithdrawalForUser` for this address reverts at `_transferAsset` with `EthTransferFailed`. [7](#0-6) 
5. ETH is permanently locked; `unlockedWithdrawalsCount[ETH_TOKEN]` never decrements; `sweepRemainingAssets` is permanently blocked for ETH. [8](#0-7)
