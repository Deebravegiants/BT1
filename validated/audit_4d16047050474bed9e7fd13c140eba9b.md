Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice()` Missing `minAnswer`/`maxAnswer` Validation Enables rsETH Over-Minting During LST Depeg — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and uses the returned price without validating it against the Chainlink aggregator's built-in `minAnswer`/`maxAnswer` circuit breakers. If a supported LST depegs below `minAnswer`, the oracle silently returns the floor price. An attacker who deposits the depegged LST in the window between the depeg and the next `updateRSETHPrice()` call receives rsETH priced at the stale (pre-depeg) `rsETHPrice` but with the inflated `minAnswer` in the numerator, yielding excess rsETH at the expense of existing depositors.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the price with no bounds validation:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Chainlink aggregators have a hardcoded `minAnswer`; when the true market price falls below it, `latestRoundData()` returns `minAnswer` rather than the actual price. No check against `aggregator.minAnswer()` or `aggregator.maxAnswer()` is performed.

This price flows into `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` / `_updateRsETHPrice()` is called. The minting formula therefore uses a **live** (inflated `minAnswer`) asset price in the numerator and a **stale** (pre-depeg) `rsETHPrice` in the denominator.

The downside-protection circuit in `_updateRsETHPrice()` (L270–281) pauses the protocol when the computed new price drops beyond `pricePercentageLimit`, but this only fires when `updateRSETHPrice()` is called — it does not prevent deposits in the preceding window. An attacker can deposit in the same block as the depeg, before any keeper or public caller triggers `updateRSETHPrice()`.

**Exploit flow:**
1. LST true market price drops to `P_true` < `minAnswer` = `P_min`.
2. `ChainlinkPriceOracle.getAssetPrice(LST)` returns `P_min` (inflated).
3. `rsETHPrice` is still the pre-depeg value `P_rsETH` (stale).
4. Attacker calls `LRTDepositPool.depositAsset(LST, amount, ...)`.
5. rsETH minted = `amount * P_min / P_rsETH` — over-valued relative to true deposit worth `amount * P_true`.
6. Attacker sells or redeems excess rsETH for healthy pool assets, extracting `amount * (P_min - P_true) / P_rsETH` ETH-equivalent profit from existing depositors.

Existing checks (`minRSETHAmountExpected`, deposit limits, reentrancy guard) do not constrain the oracle price path.

## Impact Explanation

**Critical — Direct theft of user funds.** Existing rsETH holders' share of the pool is diluted by the excess rsETH minted for the attacker. The attacker can immediately sell the excess rsETH on a secondary market or wait for `rsETHPrice` to be updated and redeem through the protocol, extracting real ETH-denominated value from the pool at the expense of all other depositors. The loss scales linearly with the size of the deposit and the gap between `minAnswer` and the true market price.

## Likelihood Explanation

**Medium.** LST depeg events are rare but precedented (stETH in May 2022). The protocol holds stETH, rETH, ETH-X, swETH, and sfrxETH, each with its own Chainlink feed and `minAnswer`. The attack window is the block interval between the depeg and the next `updateRSETHPrice()` call. `updateRSETHPrice()` is public and callable by anyone, but the attacker can act in the same block as the depeg before any keeper or MEV bot updates the price. The attack requires no special privileges and no capital beyond the depegged LST itself.

## Recommendation

In `ChainlinkPriceOracle.getAssetPrice()`, retrieve the aggregator's `minAnswer` and `maxAnswer` from the underlying `IAccessControlledOffchainAggregator` and revert if the returned price is at or outside those bounds:

```solidity
IAccessControlledOffchainAggregator aggregator =
    IAccessControlledOffchainAggregator(address(priceFeed.aggregator()));
int192 minAnswer = aggregator.minAnswer();
int192 maxAnswer = aggregator.maxAnswer();
if (price <= int256(minAnswer) || price >= int256(maxAnswer)) revert PriceOutOfBounds();
```

Additionally, validate `updatedAt` against a staleness threshold and check `answeredInRound >= roundId` (as already done in `ChainlinkOracleForRSETHPoolCollateral.getRate()`).

## Proof of Concept

**Foundry fork test outline:**

```solidity
// Fork mainnet at a block where stETH Chainlink feed is live.
// 1. Deploy a mock AggregatorV3 that returns minAnswer (e.g. 0.9e18) regardless of true price.
// 2. Call lrtManager.updatePriceFeedFor(stETH, mockFeed).
// 3. Record rsETHPrice (stale, e.g. 1.0e18) and attacker's stETH balance (1000e18, true value ~500 ETH).
// 4. attacker.depositAsset(stETH, 1000e18, 0, "").
// 5. Assert rsETH minted == 1000e18 * 0.9e18 / 1.0e18 == 900e18.
// 6. Assert true ETH value deposited (at true price 0.5e18) == 500 ETH.
// 7. Assert attacker holds 900 rsETH redeemable for ~900 ETH, netting ~400 ETH profit.
// 8. Call updateRSETHPrice() — protocol pauses due to pricePercentageLimit, confirming
//    the depeg was real and the pre-pause window was exploitable.
```

The test confirms that `getRsETHAmountToMint` uses the live inflated oracle price against the stale stored `rsETHPrice`, and that no existing guard prevents the deposit in the pre-update window.
