Audit Report

## Title
Merkle Proof Replay Across Multiple Distributor Instances Due to Missing Contract Address in Leaf Encoding - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
Every merkle distributor in the codebase computes its leaf hash without binding it to the verifying contract's address or chain ID. Because `isClaimed` state is stored per-contract instance, a proof that is valid for one distributor instance is mathematically identical to a proof valid for any other instance that sets the same merkle root, allowing a claimant to drain reward tokens from a second distributor they are not entitled to.

## Finding Description
All four distributor contracts omit `address(this)` and `block.chainid` from their leaf encoding:

- `MerkleDistributor.sol` L120: `keccak256(abi.encodePacked(index, account, cumulativeAmount))`
- `KernelMerkleDistributor.sol` L320: `keccak256(abi.encodePacked(index, account, cumulativeAmount))`
- `KernelTop100MerkleDistributor.sol` L293: `keccak256(abi.encodePacked(user, amount))`
- `MerkleBlastPointsDistributor.sol` L110–111: `keccak256(abi.encodePacked(index, account, cumulativeBlastPointAmount, cumulativeBlastGoldAmount))`

The `isClaimed` guard in each contract checks only its own `userClaims` mapping:

```solidity
// MerkleDistributor.sol L93
return userClaims[account].lastClaimedIndex >= index;
```

This provides zero cross-contract protection. If two instances share the same merkle root — a realistic scenario for a multi-chain protocol deploying the same reward snapshot on multiple chains — a proof accepted by instance A is accepted verbatim by instance B, because the leaf is identical. The `MerkleDistributor` is explicitly documented as a "Generic Merkle distributor contract" intended for repeated deployment. No `chainid` or contract-address binding exists anywhere in the codebase (confirmed by search).

Exploit path:
1. Protocol deploys `MerkleDistributor` on Chain A (`DistA`) and Chain B (`DistB`), both funded with reward tokens, both with the same merkle root at `currentIndex = 1`.
2. Alice holds a valid proof for `(index=1, account=Alice, cumulativeAmount=1000e18)`.
3. Alice calls `DistA.claim(1, Alice, 1000e18, proof)` → receives 1000 tokens; `DistA.userClaims[Alice].lastClaimedIndex = 1`.
4. Alice calls `DistB.claim(1, Alice, 1000e18, proof)` on Chain B → `DistB.isClaimed(1, Alice)` returns `false`; leaf `keccak256(abi.encodePacked(1, Alice, 1000e18))` is identical; proof verifies against the same root → Alice receives another 1000 tokens she is not entitled to.
5. Alice has stolen 1000 tokens from other users' allocations in `DistB`.

The same attack applies on a single chain if two `MerkleDistributor` instances share the same root (e.g., two reward campaigns with the same eligible set).

## Impact Explanation
The attacker receives reward tokens from a second distributor instance without any legitimate entitlement, directly draining the token balance that belongs to other users' unclaimed allocations. This matches **High: Theft of unclaimed yield**. The `MerkleDistributor` holds arbitrary ERC-20 reward tokens; `KernelMerkleDistributor` holds KERNEL tokens. Both are fully drained of other users' shares proportional to the attacker's allocation size and the number of instances sharing the same root.

## Likelihood Explanation
The LRT-rsETH protocol is explicitly multi-chain. Deploying the same reward distribution contract with the same merkle root on multiple chains is a standard operational pattern for protocols that compute rewards from a chain-agnostic snapshot (e.g., total points, rsETH holdings aggregated off-chain). The `MerkleDistributor` is documented as generic and designed for repeated deployment. The off-chain root-generation system is shared across deployments. Any user who holds a valid proof requires no privilege escalation — they simply submit the same calldata to the second instance. The attack is repeatable across every instance that shares the root.

## Recommendation
Include `address(this)` (and `block.chainid`) in the leaf hash so that a proof is cryptographically bound to the specific contract instance:

```solidity
// MerkleDistributor.sol
bytes32 node = keccak256(abi.encodePacked(address(this), block.chainid, index, account, cumulativeAmount));

// KernelTop100MerkleDistributor.sol
bytes32 leaf = keccak256(abi.encodePacked(address(this), block.chainid, user, amount));
```

The off-chain merkle tree generation must be updated to include the contract address and chain ID when building leaves, mirroring the EIP-712 domain separator pattern.

## Proof of Concept
Foundry fork test outline:

```solidity
// Fork Arbitrum and Optimism (or use two local instances)
MerkleDistributor distA = deploy(token, treasury, 0); // Arbitrum
MerkleDistributor distB = deploy(token, treasury, 0); // Optimism (same bytecode, different address)

// Fund both with 1000e18 tokens
// Build merkle tree: leaf = keccak256(abi.encodePacked(1, alice, 1000e18))
bytes32 root = buildMerkleRoot(leaf);
distA.setMerkleRoot(root);
distB.setMerkleRoot(root);

// Alice claims on distA
vm.prank(alice);
distA.claim(1, alice, 1000e18, proof);
assertEq(token.balanceOf(alice), 1000e18);

// Alice replays on distB — isClaimed returns false, proof is identical
vm.prank(alice);
distB.claim(1, alice, 1000e18, proof); // succeeds
assertEq(token.balanceOf(alice), 2000e18); // double payout confirmed
```
