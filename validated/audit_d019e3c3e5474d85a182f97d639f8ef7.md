Audit Report

## Title
Stale Cross-Chain Rate in `CrossChainRateReceiver.getRate()` Enables Over-Issuance of rsETH/wrsETH at Expense of Existing Holders - (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` variable with no check against `lastUpdated`, allowing the value to be arbitrarily stale. Both `RSETHPoolV3` and `RSETHPoolNoWrapper` use this rate as the sole denominator when computing how many rsETH/wrsETH tokens to issue per deposited ETH. When the L2 rate lags the true L1 price, any depositor receives more tokens than the current L1 rate entitles them to, diluting accrued yield of existing holders.

## Finding Description
`CrossChainRateReceiver` stores the rsETH/ETH rate in `rate` and records the update time in `lastUpdated`. The rate is written only inside `lzReceive()`, which fires only when a LayerZero message arrives from the off-chain keeper-triggered `RSETHMultiChainRateProvider`: [1](#0-0) 

`getRate()` returns the stored value unconditionally — `lastUpdated` is never consulted: [2](#0-1) 

Both pools delegate to this oracle via `IOracle(rsETHOracle).getRate()` and compute the mint/transfer amount as `amountAfterFee * 1e18 / rsETHToETHrate`: [3](#0-2) [4](#0-3) 

A stale (lower) `rsETHToETHrate` inflates `rsETHAmount`. In `RSETHPoolV3`, this inflated amount is freshly minted via `wrsETH.mint()`, directly expanding the wrsETH supply beyond what the deposited ETH justifies and diluting every existing wrsETH holder: [5](#0-4) 

`RSETHPoolNoWrapper` transfers pre-held rsETH from the pool's balance at the same inflated rate, causing the pool to dispense more rsETH per ETH than the true rate, depleting the pool's rsETH reserves faster than the bridged ETH can replenish them: [6](#0-5) 

`RSETHPoolNoWrapper` carries no `limitDailyMint` guard (unlike `RSETHPoolV3`), so the extraction is unbounded within a single block: [7](#0-6) [8](#0-7) 

No existing check — `nonReentrant`, `whenNotPaused`, `isEthDepositEnabled` — addresses rate staleness.

## Impact Explanation
**High — Theft of unclaimed yield.**

In `RSETHPoolV3`, new wrsETH is minted at a below-true-value rate. The total wrsETH supply grows beyond what the deposited ETH warrants; because the underlying rsETH backing does not grow proportionally, every existing wrsETH holder's redemption value is reduced — their accrued staking yield is transferred to the late depositor. In `RSETHPoolNoWrapper`, the pool's rsETH reserves are drained at a discount; the ETH bridged to L1 mints fewer rsETH than were dispensed, creating a structural shortfall that is ultimately borne by the protocol and its depositors. Both paths satisfy the "theft of unclaimed yield" criterion.

## Likelihood Explanation
The rate update is entirely off-chain and keeper-dependent. Keeper latency of several hours to days is operationally normal for cross-chain oracle systems. The staleness window is publicly observable by comparing `lastUpdated` on-chain against the current block timestamp and the L1 `LRTOracle.rsETHPrice`. No privileged access, no front-running of other users, and no oracle compromise is required — only a well-timed public `deposit()` call when the gap is widest. The attack is repeatable every keeper cycle.

**Likelihood: Medium.**

## Recommendation
Add a configurable `maxStaleness` threshold to `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public maxStaleness; // e.g., 86400 (24 hours)

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes all downstream pool deposits to revert when the oracle is stale, eliminating the exploitation window and creating a strong keeper incentive to publish timely updates.

## Proof of Concept
1. Observe `CrossChainRateReceiver.lastUpdated` is 48 hours old; L2 `rate` = 1.030e18, L1 true price = 1.050e18.
2. Call `RSETHPoolNoWrapper.deposit{value: 100 ether}(referralId)`.
3. `viewSwapRsETHAmountAndFee(100e18)` fetches stale rate 1.030e18 → `rsETHAmount = 100e18 * 1e18 / 1.030e18 ≈ 97.087e18`.
4. True entitlement at 1.050e18: `100e18 / 1.050e18 ≈ 95.238e18`. Excess: **≈ 1.849 rsETH**.
5. Keeper updates rate to 1.050e18. Attacker redeems 97.087 rsETH × 1.050 ≈ **101.94 ETH** — a ~1.94 ETH profit on 100 ETH deposited, funded by dilution of existing holders' staking rewards.
6. Repeat each keeper cycle for unbounded extraction via `RSETHPoolNoWrapper` (no daily mint cap).

**Foundry fork test outline:**
- Fork the target L2 at a block where `lastUpdated` is stale.
- Record attacker and victim rsETH balances.
- Call `deposit` with a large ETH value.
- Simulate keeper update via a direct `lzReceive` call with the true L1 rate.
- Assert attacker's rsETH redemption value exceeds deposited ETH; assert per-token value for pre-existing holders decreased.
