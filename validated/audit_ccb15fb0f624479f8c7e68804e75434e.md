Looking at the external report's vulnerability class — **stale-state / TOCTOU**: an update operation applies a value captured at submission time without re-validating the current state at execution time — I need to find an analog in the Cardano Ledger where a governance action or transaction is applied based on state that may have changed between proposal submission and enactment.

**Vulnerability class mapping:** Invalid state transition / funds-accounting bug caused by missing re-validation of current state at enactment time.

**Analysis of candidate sites:**

**1. `TreasuryWithdrawals` in `ENACT` / `RATIFY`**
The `withdrawalCanWithdraw` check in `ratifyTransition` uses `ensTreasury` from the running `EnactState`, which is decremented sequentially as each withdrawal is enacted. This correctly tracks the running balance across multiple proposals in the same epoch. No stale-state issue here. [1](#0-0)
