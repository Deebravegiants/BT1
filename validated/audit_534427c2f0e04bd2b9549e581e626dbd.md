### Title
Single Attested Participant Can Forge CKD Response for Legacy `AppPublicKey` Requests, Bypassing Threshold Requirement — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs no cryptographic verification of the CKD response when the request uses the legacy `CKDAppPublicKey::AppPublicKey` variant. A single Byzantine attested participant can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending legacy CKD request, delivering it to the user without any proof of threshold collaboration. This directly bypasses the t-of-n threshold requirement that is the security foundation of the MPC network.

---

### Finding Description

In `respond_ckd` at `crates/contract/src/lib.rs` lines 653–689, the contract branches on the request's `app_public_key` type:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← empty: no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies that the encrypted output is consistent with the distributed public key and the user's key pair — proving threshold collaboration. For `AppPublicKey` (legacy, "privately verifiable"), the branch is **empty**: any `CKDResponse` value is accepted and immediately delivered to the user via `resolve_yields_for`.

By contrast, `respond` (for signatures) always verifies the signature cryptographically before resolving yields, ensuring the response could only have been produced by t-of-n participants:

```rust
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
``` [2](#0-1) 

The `AppPublicKey` variant is still explicitly supported and documented as a live format in the contract README: [3](#0-2) 

The `request_app_private_key` endpoint accepts it without restriction, so pending legacy CKD requests exist in production.

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe any pending legacy CKD request on-chain — all request parameters (`domain_id`, `app_id`, `app_public_key`, `derivation_path`) are public.
2. Call `respond_ckd(request, forged_response)` with an arbitrary `CKDResponse`.
3. The contract accepts the forged response (no verification for `AppPublicKey`) and delivers it to the user via `resolve_yields_for`.

The user receives a wrong/forged confidential key derivation output. This is **confidential key derivation output without the required participant authorization** — the t-of-n threshold requirement is completely bypassed for all legacy CKD requests. The attacker does not need to know the user's private key to cause harm; delivering a garbage response is sufficient to deny the user their derived key and break the safety invariant that CKD outputs are produced by threshold collaboration.

**Impact: Critical** — matches "Bypass of threshold-signature requirements … or confidential key derivation output without the required participant authorization."

---

### Likelihood Explanation

The attacker must be an attested participant (TEE attestation + governance inclusion). However, only **one** such participant is needed — strictly below the threshold (≥60% of participants). The request parameters needed to target a specific pending request are fully visible on-chain. The `AppPublicKey` variant is still supported and used in production. Any single Byzantine node in the participant set can exploit this at any time a legacy CKD request is pending.

---

### Recommendation

1. **Deprecate and reject `AppPublicKey` in `respond_ckd`**: Since on-chain verification is impossible for this variant (no public key pair to check against), the contract should refuse to accept `respond_ckd` calls for `AppPublicKey` requests from a single participant. Instead, require threshold-based off-chain consensus before submission, or migrate all users to `AppPublicKeyPV`.

2. **Short-term mitigation**: Add a guard in `respond_ckd` that panics for `AppPublicKey` requests, forcing callers to use `AppPublicKeyPV`:
   ```rust
   dtos::CKDAppPublicKey::AppPublicKey(_) => {
       env::panic_str("AppPublicKey variant is not verifiable on-chain; use AppPublicKeyPV");
   }
   ```

3. **Align with `respond`**: The `respond` function for signatures proves threshold collaboration via cryptographic signature verification. `respond_ckd` must provide an equivalent proof for all supported variants.

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey` variant — request is stored in `pending_ckd_requests`.
2. Attacker (single attested participant) reads the pending request parameters from on-chain state.
3. Attacker calls `respond_ckd(request, CKDResponse { ... arbitrary bytes ... })`.
4. `respond_ckd` passes the `assert_caller_is_attested_participant_and_protocol_active()` check.
5. The `AppPublicKey` branch executes with no verification.
6. `resolve_yields_for` delivers the forged response to the user — the pending request is removed from the map.
7. User receives a forged confidential key; the threshold requirement was never enforced. [4](#0-3) [5](#0-4)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,

```rust
/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
