Audit Report

## Title
Stale Cross-Chain Rate Used Without Staleness Validation Allows Over-Minting of wrsETH - (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally, ignoring the `lastUpdated` timestamp that is written on every `lzReceive()` call. All L2 deposit pools (`RSETHPoolV2`, `RSETHPoolV3`, and their variants) call `getRate()` to price new deposits. When the L1 rsETH/ETH rate rises and the L2 rate has not yet been refreshed, any unprivileged depositor can exploit the artificially low stale rate to receive more wrsETH per ETH than the current exchange rate warrants, diluting the redemption value of all existing wrsETH holders.

## Finding Description

In `contracts/cross-chain/CrossChainRateReceiver.sol`, `lzReceive()` writes both `rate` and `lastUpdated` on every inbound message:

```solidity
// CrossChainRateReceiver.sol L95-97
rate = _rate;
lastUpdated = block.timestamp;
```

However, `getRate()` returns `rate` with no reference to `lastUpdated`:

```solidity
// CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;   // lastUpdated is never consulted
}
```

Every pool delegates to this function. In `RSETHPoolV2.sol`:

```solidity
// RSETHPoolV2.sol L201-203
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
```

And uses it to compute the mint amount:

```solidity
// RSETHPoolV2.sol L225-234
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // stale rate accepted silently
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

The identical pattern exists in `RSETHPoolV3.sol` (L299-308), `RSETHPoolV2ExternalBridge.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `RSETHPoolV2NBA.sol`. Because `rsETHAmount` is inversely proportional to `rsETHToETHrate`, a stale rate that is lower than the true current rate causes every pool to over-mint wrsETH. No existing guard in any pool checks the age of the rate before minting.

## Impact Explanation

**High — Theft of unclaimed yield.**

wrsETH represents a proportional claim on the rsETH pool. The yield accruing to existing wrsETH holders is the appreciation of the rsETH/ETH exchange rate over time. When an attacker deposits at a stale lower rate, they receive more wrsETH shares than the deposited ETH warrants at the true current rate. Upon rate update, those excess shares entitle the attacker to more rsETH than they contributed, extracting yield that belongs to pre-existing wrsETH holders. The surplus is bounded by the magnitude of the rate lag and the deposit size; with a 48-hour lag and a 4.76% rate appreciation, a 100 ETH deposit yields ≈4.33 excess wrsETH, redeemable for ≈4.76 ETH of rsETH value at the expense of other holders.

## Likelihood Explanation

LayerZero message delivery requires an off-chain keeper to call `updateRate()` on the provider side; it is not automatic. Any period of network congestion, keeper downtime, or rapid L1 rate appreciation creates an exploitable window. The attacker requires no privileged access: `deposit()` is fully public on all affected pools. The attacker only needs to compare the L1 rsETH oracle price against the L2 stored `rate` (both are publicly readable on-chain) and deposit when the gap exceeds transaction costs. The attack is repeatable every time the rate lags.

## Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the stored rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 24 hours, set by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

Alternatively, expose `lastUpdated` to the pool contracts and revert in `viewSwapRsETHAmountAndFee` if the rate age exceeds the threshold. As a defense-in-depth measure, the `PAUSER_ROLE` keeper should monitor rate lag and pause deposits automatically when the rate has not been refreshed within the expected window.

## Proof of Concept

1. At T₀, L1 rsETH/ETH = 1.05. LayerZero delivers the rate; `CrossChainRateReceiver.rate = 1.05e18`, `lastUpdated = T₀`.
2. At T₁ = T₀ + 48 h, L1 rsETH/ETH rises to 1.10. No keeper call has been made; L2 `rate` remains `1.05e18`.
3. Attacker calls `RSETHPoolV2.deposit{value: 100 ether}("")`.
4. `viewSwapRsETHAmountAndFee(100e18)` computes `rsETHAmount = 100e18 * 1e18 / 1.05e18 ≈ 95.238e18`.
5. Correct amount at true rate: `100e18 * 1e18 / 1.10e18 ≈ 90.909e18`.
6. Attacker receives **≈ 4.329 excess wrsETH** per 100 ETH deposited.
7. After the rate updates to 1.10, attacker redeems 95.238 wrsETH for ≈ 104.76 ETH worth of rsETH, profiting at the expense of existing holders.
