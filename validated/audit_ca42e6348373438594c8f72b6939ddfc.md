### Title
Wallet Permanently Abandons Valid Transactions When a Malicious Full Node Returns PENDING Six Times — (`File: chia/wallet/wallet_transaction_store.py`, `chia/wallet/transaction_record.py`)

### Summary

`WalletTransactionStore.increment_sent` appends every `transaction_ack` response — including `PENDING` — to the `sent_to` list without any cap or deduplication by status. `TransactionRecordOld.is_valid()` uses `len(sent_to) >= minimum_send_attempts (6)` as the sole trigger to permanently abandon a transaction. Because `PENDING` acks are not excluded from this count, a single malicious full-node peer that returns `PENDING` six consecutive times causes the wallet to mark the spend bundle as `confirmed=True, confirmed_at_height=0`, permanently abandoning it and cancelling any associated offer.

### Finding Description

**Root cause — `increment_sent` always appends, `is_valid()` counts PENDING as an attempt:** [1](#0-0) 

```python
sent_to = current.sent_to.copy()
...
sent_to.append(append_data)          # always appended, no cap, no status filter

tx = dataclasses.replace(current, sent=sent_count, sent_to=sent_to)
if not tx.is_valid():
    tx = dataclasses.replace(tx, confirmed=True, confirmed_at_height=uint32(0))
await self.add_transaction_record(tx)
```

**`is_valid()` treats any six `sent_to` entries — including `PENDING` — as exhausted attempts:** [2](#0-1) 

```python
def is_valid(self) -> bool:
    if len(self.sent_to) < minimum_send_attempts:   # 6
        return True
    if any(x[1] == MempoolInclusionStatus.SUCCESS.value for x in self.sent_to):
        return True
    if any(x[2] in {Err.INVALID_FEE_LOW_FEE.name, Err.INVALID_FEE_TOO_CLOSE_TO_ZERO.name}
           for x in self.sent_to):
        return True
    return False
``` [3](#0-2) 

**Why PENDING does not block re-sends:** `_messages_to_resend` only adds a peer to `already_sent` for `SUCCESS` or `FAILED` statuses, not `PENDING`. A peer that returned `PENDING` will receive the transaction again on the next resend cycle. [4](#0-3) 

**Attack path (single malicious peer):**

1. Wallet sends spend bundle to peer-1 → peer-1 returns `PENDING` → `sent_to` = 1 entry.
2. `_messages_to_resend` does **not** add peer-1 to `already_sent` (PENDING is excluded).
3. Wallet resends → peer-1 returns `PENDING` again → `sent_to` = 2 entries.
4. Repeat until `len(sent_to) == 6`.
5. `is_valid()` returns `False` → `increment_sent` writes `confirmed=True, confirmed_at_height=0`.
6. `get_not_sent` queries `WHERE confirmed=0` — the record is never returned again. [5](#0-4) 

The `tx_submitted` in-memory counter allows exactly `minimum_send_attempts` (6) sends within a 10-minute window, so the attacker has exactly the right number of opportunities to deliver 6 PENDING acks.

**Offer cancellation side-effect:** `remove_from_queue` calls `fail_pending_offer` for any trade linked to the abandoned transaction: [6](#0-5) 

### Impact Explanation

A single malicious full-node peer (no keys, no admin access required) can permanently corrupt the wallet's transaction state for any outgoing spend bundle:

- The transaction is marked `confirmed=True, confirmed_at_height=0` — a false "confirmed-but-failed" sentinel — and is never retried.
- Any offer/trade linked to the transaction is permanently cancelled via `fail_pending_offer`, even though the underlying coins remain unspent on-chain.
- The wallet's coin and transaction state diverges from chain state: the wallet believes the spend was attempted and failed, while the coins are still fully spendable on-chain. Recovery requires manual intervention.

This matches **High: Corruption of wallet sync state and offer/trade settlement state with direct security impact**, and **High: Permanent inability for honest wallets to process valid spend bundles under normal network assumptions**.

### Likelihood Explanation

Any full node a wallet connects to can exploit this. Wallets typically connect to a small number of peers (often 1–3). A single malicious peer is sufficient. The `PENDING` response is a legitimate protocol message (returned on 45-second timeout), so the wallet has no way to distinguish a malicious PENDING from a legitimate one. No cryptographic material or privileged access is needed.

### Recommendation

1. **Exclude `PENDING` from the `is_valid()` attempt count.** Only `FAILED` responses (excluding fee errors) should count toward exhaustion:

```python
def is_valid(self) -> bool:
    failed_non_fee = [
        x for x in self.sent_to
        if x[1] == MempoolInclusionStatus.FAILED.value
        and x[2] not in {Err.INVALID_FEE_LOW_FEE.name, Err.INVALID_FEE_TOO_CLOSE_TO_ZERO.name}
    ]
    if len(failed_non_fee) < minimum_send_attempts:
        return True
    if any(x[1] == MempoolInclusionStatus.SUCCESS.value for x in self.sent_to):
        return True
    ...
```

2. **Cap `sent_to` growth** or deduplicate by `(peer_id, status)` in `increment_sent` to prevent unbounded list growth.

3. **Do not mark a transaction `confirmed=True` solely because `is_valid()` is False.** Use a distinct `abandoned` flag so the wallet can distinguish "confirmed on-chain" from "gave up sending."

### Proof of Concept

```python
# Attacker controls a full node. Wallet connects to it.
# For each of 6 wallet sends of the same spend bundle:
#   full_node_api.send_transaction() times out (or returns PENDING directly)
#   → wallet receives TransactionAck(txid, PENDING, None)
#   → increment_sent appends ("peer_hex", uint8(2), None) to sent_to
#   → PENDING peer is NOT added to already_sent → wallet resends next cycle

# After 6 PENDING acks:
#   len(sent_to) == 6
#   is_valid() → False (no SUCCESS, no fee error)
#   tx marked confirmed=True, confirmed_at_height=0
#   offer (if any) → fail_pending_offer()
#   coins remain unspent on-chain; wallet permanently abandons the tx
``` [2](#0-1) [7](#0-6) [8](#0-7)

    async def increment_sent(
        self,
        tx_id: bytes32,
        name: str,
        send_status: MempoolInclusionStatus,
        err: Err | None,
    ) -> bool:
        """
        Updates transaction sent count (Full Node has received spend_bundle and sent ack).
        """

        current: TransactionRecord | None = await self.get_transaction_record(tx_id)
        if current is None:
            return False

        sent_to = current.sent_to.copy()

        current_peers = set()
        err_str = err.name if err is not None else None
        append_data = (name, uint8(send_status.value), err_str)

        for peer_id, status, error in sent_to:
            current_peers.add(peer_id)

        if name in current_peers:
            sent_count = uint32(current.sent)
        else:
            sent_count = uint32(current.sent + 1)

        sent_to.append(append_data)

        tx: TransactionRecord = dataclasses.replace(current, sent=sent_count, sent_to=sent_to)
        if not tx.is_valid():
            # if the tx is not valid due to repeated failures, we will confirm that we can't spend it
            log.info(f"Marking tx={tx.name} as confirmed but failed, since it is not spendable due to errors")
            tx = dataclasses.replace(tx, confirmed=True, confirmed_at_height=uint32(0))
        await self.add_transaction_record(tx)
        return True

    @metadata.request(peer_required=True)
    async def transaction_ack(self, ack: wallet_protocol.TransactionAck, peer: WSChiaConnection) -> None:
        """
        This is an ack for our previous SendTransaction call. This removes the transaction from
        the send queue if we have sent it to enough nodes.
        """
        async with self.wallet_node.wallet_state_manager.lock:
            assert peer.peer_node_id is not None
            name = peer.peer_node_id.hex()
            matched_in_flight_send = False
            if peer.peer_node_id in self.wallet_node._tx_messages_in_progress:
                in_progress = self.wallet_node._tx_messages_in_progress[peer.peer_node_id]
                if ack.txid in in_progress:
                    matched_in_flight_send = True
                    in_progress.remove(ack.txid)
                if in_progress == []:
                    del self.wallet_node._tx_messages_in_progress[peer.peer_node_id]

            # Ignore stale/unsolicited acknowledgements that don't correspond to an in-flight send.
            if not matched_in_flight_send:
                self.wallet_node.log.debug(
                    "Ignoring unsolicited transaction ack from peer %s for tx %s", name, ack.txid
                )
                return

            status = MempoolInclusionStatus(ack.status)
            try:
                wallet_state_manager = self.wallet_node.wallet_state_manager
            except RuntimeError as e:
                if "not assigned" in str(e):
                    return
                raise

            if status == MempoolInclusionStatus.SUCCESS:
                self.wallet_node.log.info(
                    f"SpendBundle has been received and accepted to mempool by the FullNode. {ack}"
                )
            elif status == MempoolInclusionStatus.PENDING:
                self.wallet_node.log.info(f"SpendBundle has been received (and is pending) by the FullNode. {ack}")
            else:
                if not self.wallet_node.is_trusted(peer) and ack.error == Err.NO_TRANSACTIONS_WHILE_SYNCING.name:
                    self.wallet_node.log.info(f"Peer {peer.get_peer_info()} is not synced, closing connection")
                    await peer.close()
                    return
                self.wallet_node.log.warning(f"SpendBundle has been rejected by the FullNode. {ack}")
            if ack.error is not None:
                await wallet_state_manager.remove_from_queue(ack.txid, name, status, Err[ack.error])
            else:
                await wallet_state_manager.remove_from_queue(ack.txid, name, status, None)
