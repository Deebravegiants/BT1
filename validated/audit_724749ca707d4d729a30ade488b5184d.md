### Title
Malicious Peer Can Inject Fee-Error `TransactionAck` to Keep Permanently Invalid Transactions Alive Indefinitely — (`chia/wallet/transaction_record.py`, `chia/wallet/wallet_node_api.py`)

### Summary

`TransactionRecordOld.is_valid()` checks only the error string (`x[2]`) in `sent_to` entries for fee-error names, without verifying the accompanying status code (`x[1]`). A malicious full-node peer that the wallet has sent a transaction to can respond with `TransactionAck(status=FAILED, error='INVALID_FEE_LOW_FEE')` for any transaction — including one that is permanently invalid — causing `is_valid()` to return `True` indefinitely, preventing the wallet from ever marking the transaction as failed and freeing its input coins.

### Finding Description

**Step 1 — Entry point: `transaction_ack` handler**

In `wallet_node_api.py`, the wallet processes `TransactionAck` messages from peers: [1](#0-0) 

The guard at lines 110–123 requires the ack to match an in-flight send (`matched_in_flight_send`). This means the attacker must be a peer the wallet actually sent the tx to — which is any peer the wallet connects to on the network (a normal, unprivileged full-node peer).

**Step 2 — Unvalidated `Err` lookup from peer-controlled string**

At line 146, the peer-supplied `ack.error` string is used directly as an enum key: [2](#0-1) 

`Err[ack.error]` performs a name-based lookup on the `Err` enum. If the peer sends `error='INVALID_FEE_LOW_FEE'`, this resolves to `Err.INVALID_FEE_LOW_FEE` with no further validation. The resulting `Err` value is passed to `remove_from_queue` → `increment_sent`.

**Step 3 — `increment_sent` stores the entry verbatim** [3](#0-2) 

`err_str = err.name` stores `'INVALID_FEE_LOW_FEE'` as the error string. The tuple `(peer_name, FAILED.value, 'INVALID_FEE_LOW_FEE')` is appended to `sent_to`. Then `is_valid()` is called on the updated record.

**Step 4 — `is_valid()` checks only the error string, not the status** [4](#0-3) 

Line 89 checks `x[2]` (the error string) against the fee-error set **without checking `x[1]`** (the status code). Even though `x[1] == MempoolInclusionStatus.FAILED.value`, the presence of `'INVALID_FEE_LOW_FEE'` in any `sent_to` entry causes `is_valid()` to return `True`.

**Step 5 — Transaction is never marked failed**

Because `is_valid()` returns `True`, the `if not tx.is_valid()` branch in `increment_sent` (line 205–208) is never taken. The transaction remains unconfirmed and is retried indefinitely. [5](#0-4) 

**Step 6 — One malicious peer is sufficient**

Even if 5 honest peers return `FAILED+DOUBLE_SPEND`, a single malicious peer returning `FAILED+INVALID_FEE_LOW_FEE` causes `any(x[2] in fee_errors for x in sent_to)` to be `True`, overriding the honest responses. `minimum_send_attempts = 6` is satisfied, but `is_valid()` still returns `True` due to line 89. [6](#0-5) 

### Impact Explanation

The wallet's input coins (removals) for the permanently invalid transaction are locked indefinitely. The wallet cannot reuse those coins for new transactions. The transaction is retried forever, consuming network resources. This constitutes a **long-lived inability for the wallet to process valid spend bundles** using the affected coins.

### Likelihood Explanation

Any peer the wallet connects to can execute this attack. The wallet connects to multiple full-node peers on the public network. An attacker simply needs to run a malicious full-node peer, wait for the wallet to connect and send a transaction, then respond with the crafted ack. No keys, admin access, or cryptographic breaks are required.

### Recommendation

Fix `is_valid()` to require that the fee-error check also verifies the status is actually `FAILED` (not `SUCCESS` or `PENDING`), and additionally require that the status is consistent with a fee error (i.e., the tx was rejected specifically due to fees, not some other reason). The simplest fix is to also check `x[1] == MempoolInclusionStatus.FAILED.value` AND cross-validate that no other non-fee `FAILED` entry exists for the same peer. Alternatively, the full node should be the authoritative source of the error classification, and the wallet should not trust the error string from an untrusted peer for validity decisions.

### Proof of Concept

1. Create a transaction spending an already-spent coin.
2. Run a malicious full-node peer that accepts wallet connections.
3. When the wallet sends `SendTransaction` for the tx, respond with `TransactionAck(txid=tx.name, status=MempoolInclusionStatus.FAILED, error='INVALID_FEE_LOW_FEE')`.
4. Observe that `increment_sent` is called with `err=Err.INVALID_FEE_LOW_FEE`.
5. Assert `tx.is_valid()` returns `True` (line 89 fires).
6. Assert the transaction remains in `confirmed=False` state indefinitely.
7. Assert the wallet cannot create new transactions spending the same coins (they are locked as pending removals).

    async def transaction_ack(self, ack: wallet_protocol.TransactionAck, peer: WSChiaConnection) -> None:

        This is an ack for our previous SendTransaction call. This removes the transaction from

                in_progress = self.wallet_node._tx_messages_in_progress[peer.peer_node_id]
                if ack.txid in in_progress:
                    matched_in_flight_send = True
                    in_progress.remove(ack.txid)
                if in_progress == []:
                    del self.wallet_node._tx_messages_in_progress[peer.peer_node_id]

            # Ignore stale/unsolicited acknowledgements that don't correspond to an in-flight send.
            if not matched_in_flight_send:
                self.wallet_node.log.debug(

                )
                return

            try:
                wallet_state_manager = self.wallet_node.wallet_state_manager
            except RuntimeError as e:
                if "not assigned" in str(e):
                    return
                raise

            if status == MempoolInclusionStatus.SUCCESS:
                self.wallet_node.log.info(
                    f"SpendBundle has been received and accepted to mempool by the FullNode. {ack}"

            elif status == MempoolInclusionStatus.PENDING:
                self.wallet_node.log.info(f"SpendBundle has been received (and is pending) by the FullNode. {ack}")

                await wallet_state_manager.remove_from_queue(ack.txid, name, status, Err[ack.error])

                await wallet_state_manager.remove_from_queue(ack.txid, name, status, None)

    def is_valid(self) -> bool:
        if len(self.sent_to) < minimum_send_attempts:
            # we haven't tried enough peers yet
            return True
        if any(x[1] == MempoolInclusionStatus.SUCCESS.value for x in self.sent_to):
            # we managed to push it to mempool at least once
            return True
        if any(x[2] in {Err.INVALID_FEE_LOW_FEE.name, Err.INVALID_FEE_TOO_CLOSE_TO_ZERO.name} for x in self.sent_to):
            # we tried to push it to mempool and got a fee error so it's a temporary error
            return True
        return False
