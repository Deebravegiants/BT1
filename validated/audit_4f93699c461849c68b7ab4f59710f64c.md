### Title
`respond_verify_foreign_tx` Accepts Mismatched `payload_hash` Enabling Cross-Request Attestation Replay - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the MPC signature in `response` is valid over `response.payload_hash`, but never checks that `response.payload_hash` actually commits to the `request` being resolved. A single Byzantine MPC participant (below threshold) can replay a threshold signature produced for a different foreign-chain request and have the contract accept it as a valid response to an unrelated pending request, delivering a forged attestation to the caller.

### Finding Description

In `respond_verify_foreign_tx`, the contract performs two independent operations:

1. **Signature check** — verifies `response.signature` is a valid ECDSA signature over `response.payload_hash` using the domain's MPC public key.
2. **Yield resolution** — resolves pending yields keyed on `request`, passing the full `response` (including `payload_hash`) back to the original caller. [1](#0-0) 

The critical gap: there is no assertion that `response.payload_hash == SHA-256(borsh(ForeignTxSignPayload { request: request, values: <any values> }))`. The `request` parameter is used only as a lookup key for the pending yield map — it is never used to validate the hash in the response.

Because `payload_hash` is computed from `(request, extracted_values)`, a valid `(payload_hash_X, signature_X)` pair produced by the MPC network for request X is also a cryptographically valid pair that the contract will accept when submitted alongside any other pending request Y. The contract never re-derives the hash from the submitted `request` to confirm they match.

### Impact Explanation

A single Byzantine MPC participant (below threshold) can:

1. Observe a pending `verify_foreign_tx` request for transaction T1 (e.g., a bridge deposit of token A for amount N).
2. Retrieve a previously computed `(payload_hash_T2, signature_T2)` from a different, already-resolved request T2 (e.g., a different transaction with different extracted values — a different block hash, program ID, or data hash).
3. Call `respond_verify_foreign_tx(request = T1_request, response = { payload_hash: payload_hash_T2, signature: signature_T2 })`.
4. The contract accepts: the signature is valid over the hash, and the caller is an attested participant.
5. The caller of T1 receives a response whose `payload_hash` commits to T2's extracted values, not T1's.

Any bridge contract that does not independently re-derive and check the `payload_hash` against its own expected `(request, values)` will accept this forged attestation and execute the bridge action based on wrong observed values — enabling double-spend or invalid bridge execution.

The SDK's `ForeignChainSignatureVerifier::verify_signature` does perform this check client-side: [3](#0-2) 

However, the contract itself provides no on-chain enforcement, so callers that skip or misuse the SDK are fully exposed.

### Likelihood Explanation

- The attack requires only one Byzantine attested participant — strictly below the signing threshold.
- The attacker needs no new cryptographic capability: they replay an existing, legitimately produced threshold signature.
- Pending requests are publicly observable on-chain, so the attacker can target any live request.
- Bridge contracts that trust the returned `payload_hash` without re-deriving it (a common pattern when integrating with oracle-style attestation services) are directly exploitable.

### Recommendation

Inside `respond_verify_foreign_tx`, after verifying the signature, re-derive the expected `payload_hash` prefix from the submitted `request` and assert the response hash commits to that same request:

```rust
// Verify payload_hash commits to the submitted request
// (values are unknown on-chain, but the request portion must match)
// Simplest enforcement: re-derive hash from (request, values_from_response)
// and compare, OR require the node to also submit the extracted values
// so the contract can verify hash = SHA256(borsh(ForeignTxSignPayload{request, values})).
```

Concretely, extend `VerifyForeignTransactionResponse` to include the `Vec<ExtractedValue>` alongside the hash, and have the contract recompute `SHA-256(borsh(ForeignTxSignPayload { request, values }))` and assert it equals `response.payload_hash` before accepting the response. This closes the gap between the signature check and the request being resolved. [4](#0-3)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
