### Title
Caller Identity Absent from `verify_foreign_transaction` Request Key Enables Queue Saturation and Unauthorized Signature Acquisition — (File: `crates/contract/src/lib.rs`)

### Summary
The `verify_foreign_transaction` function does not bind the caller's account ID into the pending-request map key, unlike `sign()` and `request_app_private_key()`. Any unprivileged caller can submit a verification request for any foreign transaction they did not initiate. By submitting up to `MAX_PENDING_REQUEST_FAN_OUT` (128) duplicate calls, an attacker saturates the fan-out queue for a targeted foreign transaction, causing the legitimate initiator's subsequent call to panic with `PendingRequestQueueFull` and permanently block their yield from being queued within the current yield window. The attacker simultaneously receives 128 MPC root-key signatures over the foreign transaction hash.

### Finding Description

**Root cause — missing caller binding in the request key.**

In `sign()`, the caller's identity is explicitly folded into the request key:

```rust
// crates/contract/src/lib.rs  ~line 379
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // ← caller identity bound into key
    &request.path,
);
```

In `verify_foreign_transaction()`, the conversion helper omits any caller identity entirely:

```rust
// crates/contract/src/dto_mapping.rs  line 840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
        // ← no predecessor / caller field
    }
}
```

The resulting `VerifyForeignTransactionRequest` is used directly as the `LookupMap` key in `pending_verify_foreign_tx_requests`. Two callers submitting the same foreign transaction therefore collide on the same key.

**Queue saturation path.**

`push_pending_yield` enforces a hard cap of 128 yields per key:

```rust
// crates/contract/src/pending_requests.rs  line 51-58
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull {
            limit: MAX_PENDING_REQUEST_FAN_OUT,
        }
        .to_string(),
    );
}
```

An attacker who observes a foreign transaction on the public foreign chain submits 128 `verify_foreign_transaction` calls for that transaction (minimum deposit: 1 yoctoNEAR each). The queue is now full. The legitimate user's subsequent call panics. MPC nodes then process the request once and deliver the signature to all 128 attacker yields via `resolve_yields_for`. The legitimate user's yield is never queued.

**Signature scheme note.** Unlike `respond()`, which verifies against a *derived* key, `respond_verify_foreign_tx()` verifies the signature against the **root public key**:

```rust
// crates/contract/src/lib.rs  ~line 729
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // ← root key, not derived
)
```

The attacker therefore receives 128 root-key signatures over the foreign transaction hash.

### Impact Explanation

This is a request-lifecycle manipulation that breaks the production safety invariant that the initiator of a foreign transaction can always queue their verification yield within the 200-block window (`REQUEST_EXPIRATION_BLOCKS = 200`). If the bridge protocol requires the claim to be submitted within that window, the legitimate user permanently loses their bridged funds for that transaction. Additionally, the attacker acquires 128 root-key signatures over the foreign transaction hash; if the broader bridge protocol uses these signatures to authorize NEAR-side fund releases, the attacker can front-run the legitimate user and claim funds they did not deposit.

### Likelihood Explanation

Foreign-chain transactions are publicly observable. The attacker needs only to watch the foreign chain, identify a pending bridge deposit, and submit 128 NEAR transactions (each costing gas but only 1 yoctoNEAR deposit) before the legitimate user calls `verify_foreign_transaction`. The attack is fully deterministic, requires no privileged access, no collusion, and no TEE compromise. The cost is bounded and predictable.

### Recommendation

Bind the caller's `predecessor_account_id` into the `VerifyForeignTransactionRequest` key, mirroring the pattern used in `sign()`:

```rust
// In verify_foreign_transaction(), after check_request_preconditions:
let request = VerifyForeignTransactionRequest {
    domain_id: request.domain_id,
    request: request.request,
    payload_version: request.payload_version,
    requester: predecessor,   // ← add caller identity
};
```

This ensures each caller's yield is keyed independently, preventing queue saturation and ensuring that only the caller who submitted the request receives the resulting signature.

### Proof of Concept

1. Legitimate user sends ETH to a NEAR bridge contract on Ethereum (transaction `tx_id` is public).
2. Attacker observes `tx_id` on Ethereum before the legitimate user acts on NEAR.
3. Attacker submits 128 calls to `verify_foreign_transaction({ domain_id, request: tx_id, payload_version })` — each costs gas + 1 yoctoNEAR.
4. `push_pending_yield` appends 128 `YieldIndex` entries under the same key in `pending_verify_foreign_tx_requests`.
5. Legitimate user calls `verify_foreign_transaction` with the same arguments → `push_pending_yield` sees `queue.len() == 128 >= MAX_PENDING_REQUEST_FAN_OUT` → panics with `PendingRequestQueueFull` → legitimate user's yield is never created.
6. MPC nodes verify `tx_id` on Ethereum, compute the signature, and call `respond_verify_foreign_tx`.
7. `resolve_yields_for` drains all 128 attacker yields, delivering the root-key signature to each.

pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
