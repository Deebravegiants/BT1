Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked` is zero, while `rewardPerToken()` freezes accumulation during that same interval. Any rewards accrued during a zero-staker window are permanently unclaimable because the time gap is consumed without distributing tokens, and no admin recovery function exists to retrieve them.

## Finding Description
The root cause is a mismatch between two components of the Synthetix-style accounting:

`rewardPerToken()` correctly short-circuits when there are no stakers:
```solidity
// L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // accumulation frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

But `updateReward` unconditionally advances the clock regardless:
```solidity
// L232-234
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();   // always advances
    ...
}
```

`initiateWithdrawal` can reduce `totalKernelStaked` to zero with no floor guard:
```solidity
// L320-326
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;
}
```

The `notifyRewardAmount` guard at L569-570 only prevents starting a period with zero stakers; it cannot prevent `totalKernelStaked` from reaching zero after the period has begun. The developer NatSpec at L18-22 explicitly acknowledges this gap and relies entirely on an off-chain operational assumption ("ensuring there are always some tokens staked"). There is no on-chain enforcement and no recovery function anywhere in the contract.

Exploit flow:
1. Admin calls `notifyRewardAmount` while stakers are present — passes the L570 guard.
2. All stakers call `initiateWithdrawal` — permissionless, no floor on `totalKernelStaked`.
3. `totalKernelStaked` reaches zero mid-period.
4. Any subsequent call that triggers `updateReward` (e.g., a new staker calling `stake`) sets `updatedAt = lastTimeRewardApplicable()`, consuming the zero-staker interval.
5. `rewardRate * (T2 - T1)` worth of reward tokens are permanently locked with no distribution path.

## Impact Explanation
This constitutes **permanent freezing of unclaimed yield** (Medium severity per the allowed impact scope). Reward tokens equal to `rewardRate * duration_of_zero_staker_gap` are locked in the contract forever. No staker can claim them, and no admin function exists to recover them. The developer comment at L18-22 confirms the protocol team is aware the tokens would be permanently stranded.

## Likelihood Explanation
The scenario is reachable by any unprivileged staker without any special access. It can occur organically during market stress (mass exit) or be deliberately triggered by a coordinated group of stakers. The only precondition is that a reward period is active, which is a normal operational state. The contract's own NatSpec acknowledges the risk, confirming the developers consider it a realistic scenario.

## Recommendation
In the `updateReward` modifier, only advance `updatedAt` when stakers are present:
```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```
This preserves the unallocated time window so that when stakers return, the accumulated rewards are distributed correctly. As a secondary safeguard, add an admin recovery function to sweep reward tokens that were never distributed, covering any residual edge cases.

## Proof of Concept
```
Setup:
  rewardRate = R (set by notifyRewardAmount at T=0, totalKernelStaked = S > 0)
  finishAt = T=0 + duration

T=10:  All stakers call initiateWithdrawal → totalKernelStaked = 0
         updateReward fires: rewardPerTokenStored updated for [0,10], updatedAt = T=10

T=20:  New staker calls stake(1e18) → updateReward fires:
         rewardPerToken() returns rewardPerTokenStored (totalKernelStaked == 0, no accumulation)
         updatedAt = lastTimeRewardApplicable() = T=20
         Gap [T=10, T=20] is consumed; R*10 seconds of rewards are permanently lost

T=20+: New staker's earned() uses updatedAt=T=20 as baseline, skipping the gap entirely.

Locked rewards = rewardRate * 10 = permanently unclaimable, no recovery path.
```

A Foundry invariant test can confirm this: assert that `rewardsToken.balanceOf(address(pool)) == sum_of_all_earned()` at all times; the invariant breaks after a zero-staker gap.
