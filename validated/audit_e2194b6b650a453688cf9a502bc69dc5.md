### Title
Reward Tokens Permanently Locked in `KernelDepositPool` When `totalKernelStaked` Drops to Zero - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary

`KernelDepositPool` implements a Synthetix-style staking rewards mechanism. When `totalKernelStaked` drops to zero during an active reward distribution window, the `rewardPerToken()` function returns `rewardPerTokenStored` unchanged while the `updateReward` modifier still advances `updatedAt` to `lastTimeRewardApplicable()`. The time elapsed with zero stakers is permanently consumed without distributing the corresponding rewards, and those reward tokens are never recovered or re-attributed. The contract's own NatSpec explicitly acknowledges this: *"any remaining rewards for that period will stay locked in the contract."*

### Finding Description

The `rewardPerToken()` function short-circuits when `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;  // accumulation halts
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

The `updateReward` modifier, called on every state-changing function, always updates `updatedAt`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();   // advances even when totalKernelStaked == 0
    ...
}
```

When `totalKernelStaked` is zero, `rewardPerTokenStored` stays flat but `updatedAt` advances. The next time a user stakes and `updateReward` fires, the formula `rewardRate * (lastTimeRewardApplicable() - updatedAt)` uses the *new* `updatedAt`, so the entire interval during which no one was staked is silently skipped. The `rewardRate * elapsed_zero_period` worth of reward tokens remain in the contract with no accounting entry pointing to them — they are unattributable and unrecoverable.

`initiateWithdrawal` immediately reduces `totalKernelStaked`:

```solidity
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;   // drops to 0 if last staker
    ...
}
```

Any user who is the last staker (or a coordinated group) can drive `totalKernelStaked` to zero, causing all rewards accruing during the zero-staking window to be permanently locked.

### Impact Explanation

Reward tokens (`rewardsToken`) allocated for the zero-staking interval are permanently frozen inside `KernelDepositPool`. No user can ever claim them; no admin function recovers them. This matches **permanent freezing of unclaimed yield** (Medium severity).

### Likelihood Explanation

Any unprivileged staker can call `initiateWithdrawal` to reduce their balance. If the last staker exits — whether organically, due to a market event, or deliberately — `totalKernelStaked` hits zero and the loss is triggered. The contract's own NatSpec at lines 17–22 explicitly acknowledges the scenario and relies solely on an off-chain operational promise ("ensuring there are always some tokens staked") rather than any code-level protection.

### Recommendation

When `totalKernelStaked == 0`, do not advance `updatedAt` in the `updateReward` modifier, so the unspent reward interval is preserved for future stakers:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {          // only advance when there are stakers
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```

Alternatively, roll over undistributed rewards into the next `notifyRewardAmount` call by tracking the "lost" interval explicitly.

### Proof of Concept

1. Admin calls `notifyRewardAmount` with `rewardRate = R`, `finishAt = T+D`.
2. Alice is the only staker; she calls `initiateWithdrawal` for her full balance at time `T+1`. `totalKernelStaked` drops to 0. `updateReward` fires: `updatedAt` advances to `T+1`.
3. No one stakes for the interval `[T+1, T+D]`. Each time `updateReward` is triggered (e.g., by any external call), `updatedAt` advances while `rewardPerTokenStored` stays flat.
4. At `T+D`, the reward window closes. `R * (D-1)` worth of reward tokens remain in the contract. No user has a claim to them; no function distributes or recovers them.
5. The contract's own comment at lines 17–22 confirms: *"any remaining rewards for that period will stay locked in the contract."* [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
