### Title
Missing Chainlink Price Validity Checks Allow Stale or Invalid Oracle Data to Corrupt rsETH Minting and Pricing — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness, completeness, or sign validation. This is the direct analog to the reported missing subgroup membership check: a value is consumed in a critical computation without first verifying it satisfies required validity constraints. The same protocol codebase already applies all three checks in `ChainlinkOracleForRSETHPoolCollateral.sol`, confirming the omission is unintentional.

---

### Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol`, `getAssetPrice()` reads from a Chainlink feed as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The return values `roundId`, `updatedAt`, and `answeredInRound` are silently discarded. No check is made for:

1. **Stale round** — `answeredInRound < roundId` indicates the answer was carried over from a prior round.
2. **Incomplete round** — `updatedAt == 0` indicates the round has not been completed.
3. **Non-positive price** — `price <= 0` is not rejected; casting a negative `int256` to `uint256` wraps to a near-`type(uint256).max` value.

By contrast, the protocol's own `ChainlinkOracleForRSETHPoolCollateral.sol` enforces all three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

`ChainlinkPriceOracle` is the oracle wired into `LRTOracle` for all supported LST assets via `assetPriceOracle[asset]`. Its output flows into two critical paths:

**Path 1 — rsETH minting:**
`LRTDepositPool.getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(asset)` to compute how many rsETH tokens to mint per deposited LST unit. [3](#0-2) 

**Path 2 — rsETH price update:**
`LRTOracle._getTotalEthInProtocol()` calls `getAssetPrice(asset)` for every supported asset to compute total protocol TVL, which then determines the rsETH/ETH exchange rate stored in `rsETHPrice`. [4](#0-3) 

---

### Impact Explanation

**Stale price scenario (High — Theft of unclaimed yield):**
If a Chainlink feed stalls (e.g., during network congestion or a sequencer outage), `getAssetPrice` returns the last cached price. If the true market price of an LST has dropped since the last update but the stale price is still high, `_updateRsETHPrice()` computes an inflated `totalETHInProtocol`, raising `rsETHPrice`. Any depositor who calls `depositAsset()` before the feed recovers receives fewer rsETH tokens than they should. Conversely, if the stale price is lower than the true price, depositors receive more rsETH than they should, diluting existing holders and constituting theft of their accrued yield.

**Negative/zero price scenario (Critical — Direct theft of funds):**
If `price` is returned as `≤ 0` (possible during a Chainlink circuit-breaker event or feed misconfiguration), `uint256(price)` wraps to an astronomically large value. `getRsETHAmountToMint` then computes `(amount * huge_number) / rsETHPrice`, minting a near-unbounded quantity of rsETH for a trivial deposit. This directly dilutes all existing rsETH holders to near-zero value. [5](#0-4) 

---

### Likelihood Explanation

Chainlink feeds do go stale during periods of network congestion, sequencer downtime (on L2 deployments), or when the deviation threshold is not crossed for extended periods. The stale-price path requires no attacker action — any depositor who transacts during a stale window is affected. The negative-price path is less frequent but has occurred historically during Chainlink circuit-breaker events on certain feeds. The entry point (`depositAsset`, `depositETH`, `updateRSETHPrice`) is fully permissionless and callable by any user.

---

### Recommendation

Apply the same three guards already present in `ChainlinkOracleForRSETHPoolCollateral.sol` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
```

Additionally, consider adding a configurable `heartbeat` staleness window (e.g., `block.timestamp - updatedAt > heartbeat`) per feed.

---

### Proof of Concept

1. A Chainlink LST/ETH feed stalls; `updatedAt` is 2 hours old but the true LST price has dropped 5%.
2. Anyone calls `updateRSETHPrice()` (permissionless via `LRTOracle.updateRSETHPrice()`).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(lst)`, which returns the stale high price with no revert.
4. `newRsETHPrice` is computed using the inflated TVL, setting `rsETHPrice` 5% above its true value.
5. A depositor calls `depositAsset(lst, amount, 0)`. `getRsETHAmountToMint` divides by the inflated `rsETHPrice`, minting fewer rsETH than deserved — or, in the inverse case, minting more rsETH than deserved, diluting all existing holders. [6](#0-5) [7](#0-6)

        uint256 supportedAssetCount = supportedAssets.length;
