Audit Report

## Title
Stale Cross-Chain Rate in `CrossChainRateReceiver.getRate()` Enables Over-Minting of wrsETH/rsETH on L2 Pools - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally with no staleness check against `lastUpdated`. All three L2 deposit pool variants (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`) consume this rate to compute how many wrsETH/rsETH tokens to mint per unit of ETH or LST deposited. When the stored rate is stale and lower than the true current rsETH/ETH exchange rate, the mint formula `amountAfterFee * 1e18 / rsETHToETHrate` yields a larger token amount than the deposit warrants, allowing any depositor to extract more value than they deposited.

## Finding Description

`CrossChainRateReceiver` stores `rate` and `lastUpdated` as state variables. `lastUpdated` is written on every `lzReceive` call but is never read by `getRate()`:

```solidity
// CrossChainRateReceiver.sol L95-97
rate = _rate;
lastUpdated = block.timestamp;
```

```solidity
// CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
``` [1](#0-0) [2](#0-1) 

There is no maximum-age guard, no circuit-breaker, and no revert path for a stale value. The rate is pushed from L1 via `CrossChainRateProvider.updateRate()` or `MultiChainRateProvider.updateRate()`, both of which are callable by anyone with no access control, but LayerZero message delivery is not instantaneous and can be delayed by minutes to hours under congestion. [3](#0-2) [4](#0-3) 

All three pool variants call `getRate()` through their local `getRate()` wrapper which delegates to `IOracle(rsETHOracle).getRate()`, and use the result directly in the mint formula:

- `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee` (ETH path, L282-285): `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`
- `RSETHPoolV3.viewSwapRsETHAmountAndFee` (L304, L307): same formula
- `RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee` (L423, L426): same formula [5](#0-4) [6](#0-5) [7](#0-6) 

The `RSETHRateReceiver` is a concrete deployment of `CrossChainRateReceiver` and inherits the same unguarded `getRate()`: [8](#0-7) 

rsETH accrues staking rewards monotonically; the rate always drifts upward between updates. A stale (lower-than-actual) `rsETHToETHrate` inflates `rsETHAmount`. The `RSETHPoolNoWrapper` variant transfers pre-minted rsETH directly from pool reserves (`rsETH.safeTransfer`), making the drain immediate and concrete. The `RSETHPoolV3` and `RSETHPoolV3ExternalBridge` variants mint new wrsETH (`wrsETH.mint`), diluting existing holders. [9](#0-8) [10](#0-9) 

The daily minting limit in `RSETHPoolV3` and `RSETHPoolV3ExternalBridge` partially mitigates scale but does not prevent the attack — it only caps the total rsETH minted per day, not the per-unit over-issuance. An attacker can still extract the full rate-discrepancy profit up to the daily cap. [11](#0-10) 

## Impact Explanation

**Critical — Direct theft of user funds.**

An attacker who deposits N ETH when the oracle rate is stale by δ receives `N / (rate - δ)` wrsETH/rsETH instead of the correct `N / rate`. On redemption at the corrected rate (on L1 or after the L2 rate is updated), they recover `(N / (rate - δ)) * rate` ETH > N ETH. The surplus is drawn from the pool's ETH/rsETH reserves, which are funded by other depositors. For `RSETHPoolNoWrapper`, the drain is immediate from the pool's pre-minted rsETH balance. This is a concrete, direct theft of user funds matching the Critical impact class.

## Likelihood Explanation

**Medium.**

The attacker needs only to monitor `CrossChainRateReceiver.lastUpdated` on-chain and deposit when the staleness gap is large enough to be profitable after fees. No privileged access, no oracle compromise, and no external protocol manipulation is required — only a public `deposit()` call. LayerZero message delivery is not guaranteed within any fixed window; network congestion, gas price spikes, or simply no one calling `updateRate()` for an extended period all produce a stale rate. rsETH appreciates monotonically, so the rate is always drifting upward between updates. The attack is repeatable across every update cycle.

## Recommendation

Add a staleness guard in `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours; // configurable by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

Alternatively, expose `lastUpdated` from the oracle interface and revert in each pool's `viewSwapRsETHAmountAndFee` if the oracle's `lastUpdated` is too old. The `MAX_RATE_AGE` constant should be configurable by the owner to allow adjustment based on observed update frequency.

## Proof of Concept

**Setup:** Deploy on a fork of the target L2. Configure `RSETHPoolNoWrapper` with `RSETHRateReceiver` as the oracle. Pre-fund the pool with rsETH.

**Steps:**

1. Simulate a stale rate by setting `CrossChainRateReceiver.rate = 1.0400e18` and `lastUpdated = block.timestamp - 6 hours` (via `vm.store` in Foundry).

2. The true L1 rsETH price has risen to `1.0405e18` (normal 6-hour staking accrual at ~4% APY).

3. Attacker calls `RSETHPoolNoWrapper.deposit{value: 1000 ether}("")`:
   - `viewSwapRsETHAmountAndFee(1000e18)` computes `rsETHAmount = 1000e18 * 1e18 / 1.0400e18 = 961.538...e18`
   - Pool transfers `961.538 rsETH` to attacker.

4. Correct amount at true rate: `1000e18 * 1e18 / 1.0405e18 = 961.076...e18`.

5. Attacker holds excess `~0.462 rsETH`. After the rate is updated on L2 (or attacker bridges to L1), they redeem `961.538 rsETH` for `961.538 * 1.0405 ≈ 1000.46 ETH`.

6. Net profit: `~0.46 ETH` on a 1000 ETH deposit (0.046%). Scaled to pool TVL in the tens of millions, this is a material theft per stale window, repeatable every update cycle.

**Foundry test skeleton:**

```solidity
function testStaleRateOverMint() public {
    // Fork target L2
    // vm.store to set stale rate and old lastUpdated
    vm.store(address(rateReceiver), RATE_SLOT, bytes32(uint256(1.0400e18)));
    vm.store(address(rateReceiver), LAST_UPDATED_SLOT, bytes32(block.timestamp - 6 hours));

    uint256 attackerBalanceBefore = rsETH.balanceOf(attacker);
    vm.prank(attacker);
    pool.deposit{value: 1000 ether}("");
    uint256 attackerBalanceAfter = rsETH.balanceOf(attacker);

    // Assert attacker received more rsETH than correct amount
    uint256 correctAmount = 1000e18 * 1e18 / 1.0405e18;
    assertGt(attackerBalanceAfter - attackerBalanceBefore, correctAmount);
}
```
