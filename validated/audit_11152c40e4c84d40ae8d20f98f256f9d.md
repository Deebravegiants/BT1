### Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` uses a structurally wrong condition for ETH that never includes the incoming deposit `amount`. Any unprivileged depositor can push total ETH deposits arbitrarily beyond the configured cap in a single `depositETH` call, minting rsETH beyond the protocol's intended ceiling.

### Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit`, the ETH branch is:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
``` [1](#0-0) 

The ETH branch tests only whether the current total **already** exceeds the limit. It never adds `amount`. For every LST the correct formula `totalAssetDeposits + amount > limit` is used. The ETH condition is therefore structurally analogous to the OrchidDirectory bug: a condition that can never fire for the intended case (a deposit that would push the total over the limit) causes the wrong branch to be taken, silently bypassing the invariant.

This check is the sole on-chain guard invoked by `_beforeDeposit` before minting rsETH:

```solidity
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
``` [2](#0-1) 

`depositETH` is the public entry point reachable by any user: [3](#0-2) 

### Impact Explanation
The deposit limit is the primary on-chain guard against over-minting rsETH relative to the protocol's capacity (EigenLayer strategy slots, node-delegator capacity). Bypassing it allows a single depositor to mint rsETH for an arbitrarily large ETH amount in one transaction, diluting existing rsETH holders and potentially driving the protocol into insolvency if backing assets cannot absorb the excess. This matches the **Low** impact tier (contract fails to deliver promised returns — the deposit cap — without necessarily losing value immediately) and can escalate to **Critical** (protocol insolvency) if the limit was sized to match EigenLayer capacity.

### Likelihood Explanation
No privilege, oracle manipulation, or front-running is required. The bypass is reachable by any depositor whenever `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset[ETH]`, which is the normal operating state of the protocol. A single large `depositETH` call is sufficient.

### Recommendation
Apply the same formula used for LSTs:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

### Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1_000 ether`.
2. Current `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.
3. Attacker calls `depositETH{value: 50_000 ether}(0, "ref")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999 ether > 1_000 ether` → `false` → no revert.
5. `getRsETHAmountToMint` mints rsETH for the full 50 000 ETH.
6. Total ETH deposits become 50 999 ether — 50× the intended cap — with a corresponding rsETH over-mint that dilutes all existing holders.

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
