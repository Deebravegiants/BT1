### Title
Deflationary/Fee-on-Transfer Token Causes wrsETH Over-Minting and Wrapper Insolvency - (File: contracts/L2/RsETHTokenWrapper.sol)

### Summary
`RsETHTokenWrapper._deposit` mints wrsETH 1:1 with the caller-supplied `_amount` without verifying the actual tokens received. If the underlying altRsETH token charges a transfer fee, the wrapper mints more wrsETH than it holds in collateral, making it insolvent and preventing last withdrawers from redeeming.

### Finding Description
In `RsETHTokenWrapper._deposit`, the contract calls `safeTransferFrom` with `_amount` and immediately mints exactly `_amount` of wrsETH to `_to`: [1](#0-0) 

`safeTransferFrom` does not verify that the contract actually received `_amount` tokens â€” it only checks that the call did not revert. If the underlying `_asset` is a fee-on-transfer token, the contract receives `_amount - fee` but mints `_amount` of wrsETH. The same pattern exists in `AGETHTokenWrapper._deposit`: [2](#0-1) 

The `_withdraw` path burns exactly `_amount` of wrsETH and transfers exactly `_amount` of altRsETH back: [3](#0-2) 

Because the wrapper's altRsETH balance is less than total wrsETH supply after any fee-on-transfer deposit, the last withdrawers will find the contract unable to fulfill their redemptions.

The `LRTDepositPool.depositAsset` has the same structural flaw â€” `rsethAmountToMint` is computed from the caller-supplied `depositAmount` before the transfer, and rsETH is minted for that full amount regardless of what is actually received: [4](#0-3) 

### Impact Explanation
**Temporary/Permanent Freezing of Funds (Mediumâ€“Critical).**

Every deposit with a fee-on-transfer altRsETH token inflates wrsETH supply relative to the actual collateral held. The shortfall accumulates with each deposit. When users attempt to withdraw, the contract will revert on `safeTransfer` once its altRsETH balance is exhausted, permanently freezing the remaining wrsETH holders' funds. In the `LRTDepositPool` case, rsETH is over-minted relative to actual LST value, leading to protocol insolvency.

### Likelihood Explanation
**Low.** The altRsETH tokens accepted by the wrapper are bridged representations of rsETH. Standard bridge implementations do not apply transfer fees today. However, the contract places no guard against fee-on-transfer tokens being added via `addAllowedToken` (TIMELOCK_ROLE), and the `allowedTokens` mapping is open-ended. If any accepted token enables fees in the future (as USDT has the ability to do), the vulnerability is immediately exploitable by any unprivileged depositor calling `deposit()` or `depositTo()`. [5](#0-4) 

### Recommendation
Use a balance-before/balance-after pattern to determine the actual received amount, and mint only that amount:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();

    uint256 balanceBefore = ERC20Upgradeable(_asset).balanceOf(address(this));
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    uint256 received = ERC20Upgradeable(_asset).balanceOf(address(this)) - balanceBefore;

    _mint(_to, received);
    emit Deposit(_asset, msg.sender, _to, received);
}
```

Apply the same fix to `AGETHTokenWrapper._deposit` and to `LRTDepositPool.depositAsset` (use the actual received amount to compute `rsethAmountToMint`). Additionally, document that rebasing tokens are not supported and enforce this at the token-allowlisting step.

### Proof of Concept

1. Admin adds a fee-on-transfer altRsETH token (1% fee) to `allowedTokens` via `addAllowedToken`.
2. Attacker (or any user) calls `deposit(asset, 1000e18)`.
3. `_deposit` executes `safeTransferFrom(msg.sender, address(this), 1000e18)` â€” contract receives `990e18` tokens.
4. `_mint(_to, 1000e18)` â€” `1000e18` wrsETH is minted.
5. Wrapper now holds `990e18` altRsETH but has `1000e18` wrsETH outstanding.
6. Repeat for N depositors; the shortfall grows linearly.
7. The last `~N * 10e18` worth of wrsETH holders call `withdraw()` â€” `safeTransfer` reverts because the contract's altRsETH balance is insufficient.
8. Those wrsETH tokens are permanently frozen. [1](#0-0) [3](#0-2)
