Let me check how `ecurve_secp256k1_t::verify` is used in the MPC protocol and look at the `ecdsa_signature_t::from_der` implementation.

### Title
Missing Low-S Normalization in `ecurve_secp256k1_t::verify` Enables ECDSA Signature Malleability - (`src/cbmpc/crypto/base_ecc_secp256k1.cpp`)

---

### Summary

`ecurve_secp256k1_t::verify` routes through `ossl_ecdsa_verify`, which only checks `0 < s < q` but never enforces `s ≤ q/2`. As a result, both `(r, s)` and `(r, q−s)` are accepted as valid signatures for the same message and public key. The bundled `secp256k1_ecdsa_verify` from the vendor library explicitly rejects high-S signatures via `!secp256k1_scalar_is_high(&s)`, but that guard is never invoked on the secp256k1 verification path.

---

### Finding Description

**Call chain:**

```
ecc_pub_key_t::verify(hash, sig)
  → ecurve_secp256k1_t::verify(P, hash, sig)          [base_ecc_secp256k1.cpp:342]
      → to_ossl_point(group, P.secp256k1)              [base_ecc_secp256k1.cpp:343]
      → ossl_ecdsa_verify(group, point, hash, sig)     [base_ecc.cpp:96]
          → ecdsa_signature_t::from_der(curve, sig)    [base_ecc.cpp:102]
          → if (r<=0||r>=q || s<=0||s>=q) return err   [base_ecc.cpp:107]  ← NO low-S check
          → u1=e*w, u2=r*w, R=mul_add(u1,Q,u2)
          → if (q.mod(R.get_x()) != r) return err
          → return SUCCESS                              ← accepts high-S
```

The guard at line 107 of `base_ecc.cpp` only rejects `s` outside `(0, q)`. It does **not** check `s ≤ q/2`. Standard ECDSA math is symmetric: if `(r, s)` verifies, so does `(r, q−s)`, because negating `s` negates `w = s⁻¹`, which negates `u1` and `u2`, which reflects `R` to `−R`, whose x-coordinate is identical mod `q`.

The vendor library's `secp256k1_ecdsa_verify` enforces the low-S rule explicitly:

```c
return (!secp256k1_scalar_is_high(&s) &&   // ← guard absent in ossl_ecdsa_verify
        secp256k1_pubkey_load(ctx, &q, pubkey) &&
        secp256k1_ecdsa_sig_verify(&r, &s, &q, &m));
```

`ecurve_secp256k1_t::verify` never calls `secp256k1_ecdsa_verify`; it converts the public key to an OpenSSL `EC_POINT` via `to_ossl_point` and delegates entirely to `ossl_ecdsa_verify`, bypassing the vendor guard.

`ecdsa_signature_t::from_der` uses OpenSSL's `d2i_ECDSA_SIG`, which also performs no low-S check, so the high-S value passes through parsing unchanged.

---

### Impact Explanation

Any caller of `ecurve_secp256k1_t::verify` (or the public `ecc_pub_key_t::verify` on a secp256k1 key) will accept both `(r, s)` and `(r, q−s)` as valid for the same message. The curve type is explicitly set to `ecurve_type_e::bitcoin` in the constructor. The MPC signing protocols (`ecdsa_mp.cpp`, `ecdsa_2p.cpp`) call `.verify()` to validate signatures produced or received during the protocol. A Byzantine participant can take any low-S signature `(r, s)` produced by the protocol, negate `s` to obtain `(r, q−s)`, and present it as a valid signature. Honest parties will accept it via `ecurve_secp256k1_t::verify`, but Bitcoin consensus nodes will reject it (they enforce low-S per BIP-62/BIP-146). This causes honest-party divergence: the MPC layer considers the signing round successful and the signature valid, while the Bitcoin network rejects the transaction carrying it.

---

### Likelihood Explanation

The attacker needs only a single valid signature `(r, s)` — which is the output of the MPC signing protocol itself — and the ability to negate `s` mod `q`, a trivial arithmetic operation. No secret material is required. The path is directly reachable through the public `ecc_pub_key_t::verify` API on any secp256k1 key.

---

### Recommendation

Add a low-S check inside `ossl_ecdsa_verify` immediately after the existing range check, specifically for secp256k1 (or any curve with `type == ecurve_type_e::bitcoin`):

```cpp
if (r <= 0 || r >= q || s <= 0 || s >= q)
    return coinbase::error(E_CRYPTO, "ossl_ecdsa_verify: invalid scalar");
// Add for secp256k1 / bitcoin curves:
if (curve.type() == ecurve_type_e::bitcoin && s > q / 2)
    return coinbase::error(E_CRYPTO, "ossl_ecdsa_verify: high-S signature rejected");
```

Alternatively, replace the `ossl_ecdsa_verify` call in `ecurve_secp256k1_t::verify` with a direct call to the vendor `secp256k1_ecdsa_verify`, which already enforces low-S.

### Proof of Concept

```cpp
// 1. Sign a message with a secp256k1 key (produces low-S signature)
ecc_prv_key_t prv;
prv.set(curve_secp256k1, some_scalar);
ecc_pub_key_t pub = prv.pub();
buf_t hash = sha256_t::hash(mem_t("test"));
buf_t der_low_s = prv.sign(hash);  // low-S by construction

// 2. Parse and negate s to produce high-S malleable counterpart
ecdsa_signature_t sig;
sig.from_der(curve_secp256k1, der_low_s);
bn_t r = sig.get_r();
bn_t s_high = curve_secp256k1.order() - sig.get_s();  // q - s
ecdsa_signature_t malleable(curve_secp256k1, r, s_high);
buf_t der_high_s = malleable.to_der();

// 3. Both verify successfully — demonstrates malleability
assert(pub.verify(hash, der_low_s)  == SUCCESS);  // expected
assert(pub.verify(hash, der_high_s) == SUCCESS);  // BUG: should fail
```

The second assertion passes because `ossl_ecdsa_verify` has no low-S guard, confirming that `ecurve_secp256k1_t::verify` accepts both forms.
