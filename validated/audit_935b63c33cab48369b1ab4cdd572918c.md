### Title
Gas Key Transactions Bypass Deploy Exclusivity Invariant in Pending Transaction Queue - (File: chain/client/src/pending_transaction_queue.rs)

### Summary
The `PendingTxSession::check_pending` function enforces the deploy exclusivity invariant only for access key transactions, gated on `!is_gas_key_tx`. Gas key transactions carrying a `DeployContract` action bypass this guard entirely. This is a direct structural analog to the VotingEscrow bug: a protection is only applied on one operation path (access key tx / `_transferFrom`) but not on an alternative path (gas key tx / split), allowing the invariant to be sidestepped by any unprivileged user who holds a gas key.

### Finding Description
In `chain/client/src/pending_transaction_queue.rs`, the deploy exclusivity invariant is intended to ensure that no `DeployContract` action coexists with other pending transactions from the same account in the uncertified window. The check is implemented in `PendingTxSession::check_pending`:

```rust
// Deploy exclusivity: a deploy cannot coexist with any other access
// key tx (including another deploy) in the pending window.
if !is_gas_key_tx {
    if total_deploy_count > 0 {
        return PendingTxCheckResult::Skip;
    }
    if tx_has_deploy && total_access_key_count > 0 {
        return PendingTxCheckResult::Skip;
    }
}
``` [1](#0-0) 

The entire guard is gated on `!is_gas_key_tx`. Gas key transactions (`nonce_index.is_some()`) skip this block entirely.

Meanwhile, `add_chunk_transactions` increments `deploy_tx_count` unconditionally for any transaction containing a deploy action, regardless of key type: [2](#0-1) 

The session update in `check_pending` also increments `session_deploy_tx_counts` unconditionally: [3](#0-2) 

This creates a one-directional asymmetry:
- A gas key deploy **does** increment `deploy_tx_count`, which blocks subsequent **access key** txs.
- But a gas key tx is **not** blocked by a pending deploy (access key or gas key).

The existing test `test_deploy_exclusivity_gas_key_deploy_blocks_access_key_tx` confirms the first direction (gas key deploy blocks access key txs), but the reverse — a gas key tx being blocked by a pending deploy — is never enforced: [4](#0-3) 

An unprivileged user who owns a gas key on their account can:
1. Submit an access key `DeployContract` tx (pending, uncertified).
2. Immediately submit a gas key `DeployContract` tx — the deploy exclusivity check is skipped, and the tx is admitted.
3. Both deploys are in-flight simultaneously, violating the invariant.

### Impact Explanation
The deploy exclusivity invariant in the SPICE Pending Transaction Queue is violated. Multiple `DeployContract` actions from the same account can be in-flight simultaneously. The final on-chain contract code for the account is determined by the order of chunk certification. The corrupted protocol value is the `deploy_tx_count` invariant tracked in `PendingTransactionQueue::pending_accounts`, which is supposed to guarantee deploy exclusivity across the uncertified window. This can lead to unpredictable final contract state for the account — the deployed code that ends up on-chain depends on non-deterministic certification ordering rather than the user's intent. [5](#0-4) 

### Likelihood Explanation
Any account holder who has added a gas key to their account (a normal, unprivileged `AddKey` operation) can trigger this bypass. No validator, node admin, or operator privileges are required. The attacker simply submits a gas key transaction with a `DeployContract` action while an access key deploy is pending. The gas key feature is a production feature gated on `ProtocolFeature::GasKeys`. [6](#0-5) 

### Recommendation
Remove the `!is_gas_key_tx` gate from the deploy exclusivity check in `PendingTxSession::check_pending`, or add a parallel check for gas key transactions with deploy actions. The invariant should be enforced uniformly regardless of transaction type. Specifically, the block at lines 493–502 should also apply when `is_gas_key_tx` is true, or a separate guard should be added for gas key deploys.

### Proof of Concept
1. Account `alice` has a full-access key and a gas key with `num_nonces = 1`.
2. Alice submits an access key `DeployContract` tx (nonce 1). It is included in a block (pending, uncertified). `deploy_tx_count` for alice becomes 1 via `add_chunk_transactions`.
3. Alice submits a gas key `DeployContract` tx (nonce_index 0, nonce 2). `check_pending` is called:
   - `is_gas_key_tx = true` (nonce_index is `Some(0)`)
   - The deploy exclusivity block (`if !is_gas_key_tx`) is skipped entirely
   - Gas key balance check passes
   - The tx is admitted with `PendingTxCheckResult::Admit`
4. Both deploys are now in-flight simultaneously. The final contract code on `alice`'s account is determined by which chunk is certified first, violating the deploy exclusivity invariant that the PTQ is designed to enforce. [7](#0-6)

        // Deploy exclusivity: a deploy cannot coexist with any other access
        // key tx (including another deploy) in the pending window.
        if !is_gas_key_tx {
            if total_deploy_count > 0 {
                return PendingTxCheckResult::Skip;
            }
            if tx_has_deploy && total_access_key_count > 0 {
                return PendingTxCheckResult::Skip;
            }
        }

        // P_MAX for contract accounts.
        if has_contract == HasContract::Yes && !is_gas_key_tx && total_access_key_count >= P_MAX {
            return PendingTxCheckResult::Skip;
        }

        // Build constraints for runtime validation.
        let gas_key = (signer_id.clone(), public_key.clone());
        let session_gas_key_withdrawal =
            self.session_gas_key_withdrawals.get(&gas_key).copied().unwrap_or(Balance::ZERO);
        let paid_from_gas_key =
            snapshot.pending_gas_key_cost.saturating_add(session_gas_key_withdrawal);

        // Update session state optimistically (assumes tx will be accepted).
        // If the runtime subsequently rejects the tx (e.g. insufficient
        // balance), these counts are not rolled back and the tx is discarded
        // (not reintroduced to the pool). This means a rejected tx may consume
        // a P_MAX or deploy exclusivity slot for the remainder of this chunk
        // production session, reducing throughput under high contention. The
        // risk is mitigated by the fact that check_pending is called after
        // signature verification and basic validation, so only transactions
        // with valid signatures can reach this point -- an adversary cannot
        // cheaply spam rejected txs to exhaust slots.
        if !is_gas_key_tx {
            *self.session_access_key_tx_counts.entry(signer_id.clone()).or_insert(0) += 1;
        }
        if tx_has_deploy {
            *self.session_deploy_tx_counts.entry(signer_id.clone()).or_insert(0) += 1;
        }
        // Track WithdrawFromGasKey amounts from this tx's actions.
        for action in tx.transaction.actions() {
            if let Action::WithdrawFromGasKey(withdraw) = action {
                let entry = self
                    .session_gas_key_withdrawals
                    .entry((signer_id.clone(), withdraw.public_key.clone()))
                    .or_insert(Balance::ZERO);
                *entry = entry.saturating_add(withdraw.amount);
            }
        }

        PendingTxCheckResult::Admit(PendingConstraints {
            paid_from_balance: snapshot.paid_from_balance,
            paid_from_gas_key,
            max_nonce: snapshot.max_nonce,
        })
    }
