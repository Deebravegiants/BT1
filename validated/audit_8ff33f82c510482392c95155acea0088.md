### Title
Storage Deposit Permanently Frozen on Update Execution — (`File: crates/contract/src/update.rs`)

### Summary

When a participant proposes a contract or config update via `propose_update()`, they must attach a NEAR storage deposit. When any one update reaches threshold and `do_update()` executes, it unconditionally clears **all** other pending proposals without refunding the storage deposits paid by their proposers. Because `UpdateEntry` does not record the proposer's account ID, those NEAR tokens are permanently trapped in the contract with no recovery path.

---

### Finding Description

`propose_update()` in `lib.rs` requires the caller to attach a deposit covering the storage cost of the proposal: [1](#0-0) 

The required deposit is computed from `bytes_used`, which for a contract update includes the full WASM binary plus overhead for up to 128 participant-vote slots: [2](#0-1) 

Excess is refunded to the proposer, but the required portion is retained by the contract. The `UpdateEntry` struct stores only the update payload and byte count — **not the proposer's account ID**: [3](#0-2) 

When `do_update()` is triggered by a successful `vote_update()`, it removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every other pending proposal: [4](#0-3) 

Because the proposer's identity is not stored in `UpdateEntry`, there is no mechanism to refund the deposits of the cleared proposals. The NEAR tokens paid by those proposers remain in the contract balance permanently.

---

### Impact Explanation

Every proposer of a non-executed update loses their storage deposit. For a contract update containing a typical NEAR WASM binary (~500 KB–1 MB), the deposit at NEAR's storage rate (~1 NEAR per 100 KB) is on the order of 5–10 NEAR per proposal. With multiple concurrent proposals (a realistic governance scenario), the aggregate loss is material. The tokens are irrecoverable: no withdrawal, no refund, and no cleanup function exists for them.

This breaks the production accounting invariant that deposited funds are either consumed for their stated purpose or returned to the depositor. Impact: **Medium** — balance/accounting invariant broken without requiring DoS or operator misconfiguration.

---

### Likelihood Explanation

This triggers during normal governance operation. Any time two or more participants independently propose different updates (e.g., one proposes a config change while another proposes a code upgrade), whichever reaches threshold first causes the other's deposit to be permanently lost. No adversarial intent is required; the loss is a structural consequence of the clearing logic.

---

### Recommendation

1. Add a `proposer: AccountId` field to `UpdateEntry` so the depositor's identity is preserved.
2. In `do_update()`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(required_deposit(entry.bytes_used))` for each.
3. Alternatively, track deposits in a separate `LookupMap<UpdateId, (AccountId, NearToken)>` and drain it on `do_update()`.

---

### Proof of Concept

1. Participant A calls `propose_update({ code: <1MB WASM> })` attaching ~10 NEAR. The deposit is accepted; `UpdateEntry { update: Contract(...), bytes_used: ~1_000_000 }` is stored.
2. Participant B calls `propose_update({ config: <small config> })` attaching ~0.1 NEAR. A second entry is stored.
3. Threshold participants vote for B's update via `vote_update(id_B)`. `do_update(&id_B, gas)` is called.
4. Inside `do_update`: `self.entries.remove(&id_B)` removes B's entry; `self.entries.clear()` silently drops A's entry. No refund is issued to A.
5. A's ~10 NEAR is permanently locked in the contract. A has no method to reclaim it. [5](#0-4) [6](#0-5)

    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
