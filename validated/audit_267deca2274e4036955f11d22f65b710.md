### Title
Missing On-Chain CKD Output Verification for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Forged Key Material - (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC contract performs a cryptographic pairing check (`ckd_output_check`) on the CKD response only when the request used the `AppPublicKeyPV` (publicly verifiable) variant. For the `AppPublicKey` (privately verifiable) variant, the match arm is **completely empty** — no verification of any kind is performed. A single Byzantine attested participant can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept and deliver it unconditionally.

---

### Finding Description

In `respond_ckd`, after authenticating the caller as an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC network's master key and the app's identity: [2](#0-1) 

For `AppPublicKey`, the arm is `{}` — the contract immediately proceeds to `resolve_yields_for`, delivering whatever `(big_y, big_c)` the caller supplied: [3](#0-2) 

The `AppPublicKey` variant is the **legacy and default** variant. It is used in the example CLI, in all node unit tests, and is the format accepted when a plain G1 key string is passed: [4](#0-3) [5](#0-4) 

The off-chain MPC computation for `AppPublicKey` runs the `ckd` protocol (not `ckd_pv`), which produces a correct `(big_y, big_c)` pair. But the contract never verifies this on-chain for this variant: [6](#0-5) 

By contrast, `respond` (for ECDSA/EdDSA signatures) and `respond_verify_foreign_tx` both perform cryptographic verification of the response before resolving yields: [7](#0-6) 

---

### Impact Explanation

**Critical — Confidential key derivation output without required participant authorization.**

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe a pending `AppPublicKey` CKD request in the contract's `pending_ckd_requests` map.
2. Call `respond_ckd` with an arbitrary `CKDResponse { big_y: <any G1>, big_c: <any G1> }`.
3. The contract passes all checks (attested participant, protocol running, domain valid) and resolves all queued yields for that request with the forged data.
4. The forged `(big_y, big_c)` is returned to the developer contract and forwarded to the TEE app.

The TEE app receives attacker-controlled encrypted key material. If the app implements the optional `decrypt_secret_and_verify` step (as shown in the example CLI), it will detect the forgery — but the yield is already consumed and the request is permanently resolved. The app must re-submit, and the Byzantine participant can repeat the attack indefinitely, causing a permanent denial of the CKD service for that `app_id`. If the app does not implement app-side verification (which the contract does not mandate), it silently derives a wrong key from the forged material. [8](#0-7) 

The threshold guarantee — that a correct CKD output requires collaboration of at least `t` honest participants — is entirely absent for `AppPublicKey` requests at the contract level.

---

### Likelihood Explanation

**High.** The `AppPublicKey` variant is the default and legacy format. Any attested MPC participant (one node, no collusion required) can exploit this. The attack requires only: (a) being an active attested participant, and (b) observing a pending `AppPublicKey` CKD request on-chain — both are trivially achievable by any participant. The Byzantine participant can race to call `respond_ckd` before the honest leader submits the correct response.

---

### Recommendation

Add a cryptographic verification step for the `AppPublicKey` variant in `respond_ckd`. Since `AppPublicKey` only provides a G1 point `pk1 = a·G1` (without `pk2 = a·G2`), the full pairing check used by `ckd_output_check` is not directly applicable. The recommended fix is to **require all new CKD requests to use `AppPublicKeyPV`** and deprecate `AppPublicKey`, or alternatively derive a verifiable commitment from `pk1` alone using a different check. At minimum, document clearly that `AppPublicKey` provides no on-chain response integrity guarantee and that app-side verification is mandatory for security.

---

### Proof of Concept

1. TEE app submits a CKD request using `AppPublicKey` (the default/legacy variant):
   ```json
   { "request": { "derivation_path": "mykey", "app_public_key": "bls12381g1:<pk1>", "domain_id": 2 } }
   ```

2. Byzantine attested participant observes the pending request in `pending_ckd_requests`.

3. Byzantine participant calls `respond_ckd` with forged values before honest nodes respond:
   ```json
   { "request": { "app_public_key": "bls12381g1:<pk1>", "app_id": "...", "domain_id": 2 },
     "response": { "big_y": "bls12381g1:<any_point>", "big_c": "bls12381g1:<any_point>" } }
   ```

4. Contract executes `respond_ckd`:
   - `assert_caller_is_attested_participant_and_protocol_active()` → passes (Byzantine node is attested)
   - `match &request.app_public_key { AppPublicKey(_) => {} ... }` → **empty arm, no check**
   - `resolve_yields_for(...)` → forged response delivered to all waiting yields [9](#0-8) 

5. TEE app receives `(big_y_forged, big_c_forged)`. If it skips app-side verification, it derives a wrong key. If it verifies, the request is permanently consumed and must be retried — where the attack repeats.

                    env::panic_str("CKD output check failed");

        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)

fn decrypt_secret_and_verify(
    big_y: &Bls12381G1PublicKey,
    big_c: &Bls12381G1PublicKey,
    private_key: Scalar,
    app_id: CkdAppId,
    mpc_public_key: &Bls12381G2PublicKey,
) -> Result<[u8; BLS12381G1_PUBLIC_KEY_SIZE]> {
    let big_y: G1Projective = big_y.try_into().context("invalid G1 point")?;
    let big_c: G1Projective = big_c.try_into().context("invalid G1 point")?;
    let mpc_public_key: G2Projective = mpc_public_key.try_into().context("invalid G2 point")?;

    // decrypt the secret
    let secret = big_c - big_y * private_key;

    // verify the secret
    if !verify(&mpc_public_key, app_id.as_ref(), &secret) {
        anyhow::bail!("Verification failed!");
    }

    // return the secret as bytes
    Ok(secret.to_compressed())
