### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Rewards via Unenforced `farmer_reward_address_override` — (File: `chia/farmer/farmer_api.py`)

---

### Summary

CHIP-22 introduced `farmer_reward_address_override`, a field in `NewProofOfSpace` and `RespondSignatures` that allows a third-party harvester to redirect the farmer's 1/8 block reward to itself as a fee. The fee quality threshold check in `notify_farmer_reward_taken_by_harvester_as_fee` is **advisory only** — it logs a warning but never blocks the override. In `_process_respond_signatures`, the farmer unconditionally substitutes the harvester-supplied address as `farmer_reward_address` in `DeclareProofOfSpace`, which is then broadcast to the full node. A malicious third-party harvester can therefore redirect 100% of the farmer's block rewards to any address with no enforcement.

---

### Finding Description

**Entry point — `NewProofOfSpace` with override set:**

When a harvester sends `NewProofOfSpace` with `farmer_reward_address_override != None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee` and then unconditionally continues the block-signing flow: [1](#0-0) 

**Advisory-only fee quality check:**

`notify_farmer_reward_taken_by_harvester_as_fee` computes a fee quality value and compares it to the harvester-supplied threshold. When the threshold is violated — or when `fee_info` is `None` — it only emits a `log.warning`. There is no `return`, no exception, and no mechanism to abort the signing flow: [2](#0-1) 

**Unconditional override in `_process_respond_signatures`:**

After the harvester returns `RespondSignatures`, the farmer replaces `farmer_reward_address` with whatever the harvester supplied, with no re-validation of the fee quality threshold: [3](#0-2) 

**`DeclareProofOfSpace` broadcast with attacker's address:**

The resulting `DeclareProofOfSpace` message carries the attacker-controlled `farmer_reward_address` and is sent to all connected full nodes: [4](#0-3) 

The full node then writes this address into `FoliageBlockData.farmer_reward_puzzle_hash`, which is consensus-enforced and determines where the 1/8 farmer block reward coin is created.

---

### Impact Explanation

A malicious third-party harvester can set `farmer_reward_address_override` to any puzzle hash it controls and set `fee_info=None` (bypassing even the advisory log path). The farmer will sign and broadcast a `DeclareProofOfSpace` that pays the farmer's entire block reward to the attacker's address. Every block won while the malicious harvester is connected results in permanent, irreversible XCH theft from the farmer. This matches the **High** impact category: *unauthorized payout redirection affecting XCH*. [5](#0-4) 

---

### Likelihood Explanation

The CHIP-22 third-party harvester model is an explicitly supported and documented use case. Farmers who delegate plot scanning to external harvester services are the direct target population. The attacker only needs to operate a harvester service that the farmer connects to — no key compromise, no cryptographic break, and no admin access is required. The private-CA SSL requirement means the attacker must have a certificate signed by the farmer's private CA, but this is satisfied by any harvester the farmer has legitimately enrolled. The attack is silent (only a log warning is emitted) and takes effect on the very first block won. [6](#0-5) 

---

### Recommendation

1. **Enforce the fee quality threshold**: In `new_proof_of_space`, after calling `notify_farmer_reward_taken_by_harvester_as_fee`, check whether `fee_quality <= fee_threshold` and abort (return `None`) if the threshold is violated or if `fee_info` is absent.
2. **Re-validate in `_process_respond_signatures`**: Before substituting `response.farmer_reward_address_override`, re-check the fee quality against the stored proof and challenge hash. If the check fails, use `self.farmer.farmer_target` instead and log an error.
3. **Optionally allow farmers to configure a whitelist of permitted override addresses** or a maximum fee rate, enforced in code rather than advisory logging.

---

### Proof of Concept

1. Attacker operates a harvester service and obtains a private-CA certificate (normal enrollment).
2. Farmer connects to the attacker's harvester.
3. Attacker's harvester, upon receiving `NewSignagePointHarvester`, responds with `NewProofOfSpace` where:
   - `farmer_reward_address_override = <attacker_puzzle_hash>`
   - `fee_info = None`
4. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee` → logs a warning → **continues**.
5. Farmer sends `RequestSignatures` to the harvester.
6. Harvester returns `RespondSignatures` with `farmer_reward_address_override = <attacker_puzzle_hash>`.
7. `_process_respond_signatures` sets `farmer_reward_address = attacker_puzzle_hash` unconditionally.
8. Farmer broadcasts `DeclareProofOfSpace` with `farmer_reward_address = attacker_puzzle_hash`.
9. Full node creates the farmer reward coin at `attacker_puzzle_hash`. The farmer receives nothing. [1](#0-0) [7](#0-6) [8](#0-7)

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
