### Title
Caller-Agnostic `verify_foreign_transaction` Key Allows Queue Exhaustion DoS — (`File: crates/contract/src/pending_requests.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint stores pending requests under a key that does **not** include the caller's account ID. Combined with the hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` on the per-key yield queue, any unprivileged account can fill the queue for a specific foreign-chain transaction with 128 cheap submissions (each costing 1 yoctoNEAR), permanently blocking all subsequent legitimate callers from queuing the same request until slots drain.

---

### Finding Description

The `sign()` function derives its storage key by hashing the caller's account ID into the key via `SignatureRequest::new`:

```rust
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // ← caller-specific tweak
    &request.path,
);
```

This means each caller gets their own independent queue slot.

The `verify_foreign_transaction()` function, however, maps its request through `args_into_verify_foreign_tx_request`, which produces a `VerifyForeignTransactionRequest` containing only `{ domain_id, request, payload_version }` — **no caller identity**:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

All callers submitting the same foreign-chain transaction share a **single queue entry** in `pending_verify_foreign_tx_requests`. The queue is capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
// ...
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string(),
    );
}
```

An attacker submits 128 `verify_foreign_transaction` calls for the same `(domain_id, ForeignChainRpcRequest, payload_version)` tuple, each with the minimum 1 yoctoNEAR deposit. The queue is now full. Any subsequent legitimate caller attempting to verify the same foreign-chain transaction receives `PendingRequestQueueFull` and their transaction reverts. The attacker can re-fill the queue immediately after each MPC response drains it, sustaining the DoS indefinitely.

The test at line 3242 in `lib.rs` even contains the comment: *"a different account would today be blocked from receiving a response by alice's submission"* — acknowledging the shared-key design.

---

### Impact Explanation

A legitimate bridge user who submitted a foreign-chain transaction (e.g., a Bitcoin deposit) and calls `verify_foreign_transaction` to obtain the MPC signature needed to complete the bridge operation will have their call permanently rejected as long as the attacker keeps the queue saturated. The bridge operation cannot complete; the user's funds on the foreign chain are effectively frozen from the MPC network's perspective. This breaks the production safety invariant that any valid `verify_foreign_transaction` request must eventually be processable.

**Impact class:** Medium — request-lifecycle manipulation that breaks production accounting invariants (bridge operations cannot complete) without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

- **Entry path:** Any unprivileged NEAR account. No special role, key, or collusion required.
- **Cost:** 128 × 1 yoctoNEAR + gas ≈ negligible. Gas on NEAR is cheap; 128 transactions cost on the order of a few cents.
- **Persistence:** The attacker can monitor the queue length (via `get_pending_verify_foreign_tx_request`) and re-submit as slots drain, sustaining the DoS indefinitely.
- **Targeting:** The attacker can selectively target specific foreign-chain transactions (e.g., a competitor's large withdrawal) by choosing the exact `(tx_id, domain_id, payload_version)` tuple.

---

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, analogous to how `SignatureRequest` incorporates a per-caller tweak. This gives each caller their own independent queue slot, eliminating the shared-key exhaustion vector. Alternatively, enforce a per-account submission limit per foreign-chain transaction key.

---

### Proof of Concept

1. Alice (legitimate user) submits a Bitcoin deposit on-chain and calls `verify_foreign_transaction` with `{ domain_id: 0, request: BitcoinRpcRequest { tx_id: [0xAB; 32], confirmations: 2, extractors: [BlockHash] }, payload_version: V1 }` and 1 yoctoNEAR deposit.

2. Attacker Bob, before Alice's call lands (or immediately after), submits the **identical** `verify_foreign_transaction` call 128 times from his own account (or a script), each with 1 yoctoNEAR. Total cost: 128 yoctoNEAR + gas.

3. The queue for that `VerifyForeignTransactionRequest` key is now at `MAX_PENDING_REQUEST_FAN_OUT = 128`.

4. Alice's call (or any subsequent call from any account for the same tx) hits `push_pending_yield` and panics with `PendingRequestQueueFull { limit: 128 }`. Alice's transaction reverts; she cannot obtain the MPC signature.

5. When MPC nodes eventually respond and drain the queue, Bob immediately re-submits 128 calls. Alice is permanently blocked.

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```

```rust
    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
```
