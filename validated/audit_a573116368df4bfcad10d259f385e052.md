Audit Report

## Title
Missing Staleness and Validity Checks in `ChainlinkPriceOracle.getAssetPrice()` Allows Stale Prices to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `answeredInRound`, and does not validate that `price > 0`. This allows a stale or incomplete Chainlink round price to flow directly into `LRTDepositPool.getRsETHAmountToMint()`, causing rsETH to be minted at an incorrect rate. The same oracle path affects `LRTWithdrawalManager.getExpectedAssetAmount()`, causing withdrawers to receive fewer underlying tokens than owed.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` reads the Chainlink feed as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The implementation drops `updatedAt` and `answeredInRound`, so:

- **No heartbeat/staleness check**: `block.timestamp - updatedAt > heartbeat` is never evaluated.
- **No round-completeness check**: `answeredInRound < roundId` (in-progress or failed round) is never caught.
- **No non-negative price guard**: a zero or negative `price` is cast directly to `uint256`.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository explicitly checks all three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The stale price flows into rsETH minting via the public call chain:

`depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice()` [3](#0-2) 

The `minRSETHAmountExpected` slippage guard in `_beforeDeposit()` does not protect against stale prices — when the stale price is inflated, the depositor receives *more* rsETH than expected, so the guard never triggers. [4](#0-3) 

## Impact Explanation
**Low — Contract fails to deliver promised returns.**

When a Chainlink feed for an LST (e.g., stETH/ETH) goes stale while the asset's real price has dropped (depeg scenario), `getAssetPrice()` returns the old, higher price. Depositors receive more rsETH than the deposited ETH value warrants, diluting all existing rsETH holders' share of the backing. Withdrawers using `getExpectedAssetAmount()` with a stale high asset price receive fewer underlying tokens than the rsETH they burn is worth. No funds are directly stolen, but the protocol fails to deliver correct exchange rates — matching the "Low: Contract fails to deliver promised returns" impact class.

## Likelihood Explanation
**Low.** Chainlink feeds for major LSTs (stETH, rETH) have 24-hour heartbeats and 0.5% deviation thresholds, making prolonged staleness uncommon under normal conditions. However, during network congestion, oracle node outages, or a rapid LST depeg event — exactly the scenario where correct pricing matters most — the feed can lag significantly before the heartbeat triggers an update. The exploit requires no special privileges; any unprivileged depositor can call `depositAsset()` while the feed is stale.

## Recommendation
Add staleness and validity guards to `ChainlinkPriceOracle.getAssetPrice()`, consistent with what `ChainlinkOracleForRSETHPoolCollateral` already does:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be set per-feed based on the Chainlink heartbeat (e.g., 25 hours for a 24-hour heartbeat feed).

## Proof of Concept
1. Chainlink's stETH/ETH feed stops updating (oracle node outage or network congestion). Last reported price: `1.05 ETH` per stETH.
2. stETH depegs on-chain to `0.95 ETH`.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `(1000e18 * 1.05e18) / rsETHPrice` — using the stale `1.05` price instead of the real `0.95`.
5. Attacker receives ~10.5% more rsETH than the deposited value warrants, diluting all existing rsETH holders.

**Foundry fork test plan:**
- Fork mainnet at a block where the stETH/ETH Chainlink feed has not updated recently (or mock `latestRoundData()` to return an `updatedAt` value 25+ hours in the past with a price of `1.05e18`).
- Call `depositAsset(stETH, 1000e18, 0, "")` and record `rsethAmountToMint`.
- Compare against the expected amount computed using the current real price (`0.95e18`).
- Assert that the attacker received more rsETH than the real price warrants, confirming the dilution.
