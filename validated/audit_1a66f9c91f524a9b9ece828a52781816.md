### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Pending `request` — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the ECDSA signature in the response is valid over `response.payload_hash`, but it never checks that `response.payload_hash` was actually derived from the `request` stored in the pending queue. A single Byzantine attested participant (strictly below the signing threshold) can therefore deliver a response for pending request X that carries the legitimately-produced MPC signature and `payload_hash` from a different pending request Y, causing the caller of `verify_foreign_transaction(X)` to receive a forged verification result.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs two independent checks:

1. The caller is an attested participant.
2. The ECDSA signature in `response` is valid over `response.payload_hash` under the domain's root public key. [1](#0-0) 

What it **never** checks is whether `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: <any values> }))`. The `request` argument is used only as a map key to locate and drain the pending yield queue: [2](#0-1) 

The `ForeignTxSignPayload` that nodes are supposed to sign binds both the original `ForeignChainRpcRequest` and the extracted on-chain values: [3](#0-2) 

The off-chain SDK helper `ForeignChainSignatureVerifier::verify_signature` does perform this binding check (`expected_payload_hash == response.payload_hash`), but it is an optional client-side utility, not enforced by the contract: [4](#0-3) 

The contract's own `respond_verify_foreign_tx` has no equivalent check.

### Impact Explanation

**Impact: High — Forged foreign-chain verification / verification bypass causing invalid bridge execution.**

A bridge contract that calls `verify_foreign_transaction(request_X)` and receives a `VerifyForeignTransactionResponse` trusts that the MPC network verified transaction X. Because the contract does not bind `payload_hash` to `request`, the response it delivers to the caller may contain a `payload_hash` (and valid MPC signature) that corresponds to a completely different transaction Y. Any bridge logic that does not independently re-derive and compare the expected hash (i.e., does not use `ForeignChainSignatureVerifier::verify_signature`) will accept this as proof that transaction X was verified, enabling double-spend or invalid bridge execution.

### Likelihood Explanation

**Likelihood: Medium.**

The attack requires only a **single** attested participant — strictly below the signing threshold — and two concurrently pending `verify_foreign_transaction` requests. In production, concurrent requests are routine. The attacker participates in the MPC signing for request Y (obtaining a valid `(payload_hash_Y, sig_Y)` tuple), then calls `respond_verify_foreign_tx(request=request_X, response={payload_hash: payload_hash_Y, signature: sig_Y})`. The contract accepts this because `sig_Y` is a valid MPC signature over `payload_hash_Y`, and `request_X` exists in the pending queue. No threshold collusion, no key leakage, and no privileged operator access is required.

### Recommendation

Inside `respond_verify_foreign_tx`, after extracting `payload_hash` from the response, recompute the expected hash from the pending request and assert equality before accepting the response:

```rust
// Derive the minimum expected hash prefix from the request alone.
// The contract cannot know the extracted values, but it can at least
// verify that payload_hash encodes the correct ForeignChainRpcRequest.
// Alternatively, store the expected payload_hash at request-submission
// time (computed by the node that first responds) and compare on every
// subsequent respond call.
```

The cleanest fix is to have the contract recompute `SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request.clone(), values: response.values })))` and compare it to `response.payload_hash`, requiring nodes to also include the `values` in the response. Alternatively, the contract can store the `payload_hash` on the first accepted response and reject any subsequent `respond_verify_foreign_tx` call for the same request that supplies a different `payload_hash`.

### Proof of Concept

1. Alice submits `verify_foreign_transaction(request_X)` (Bitcoin tx_id = `[0xAA; 32]`).
2. Bob submits `verify_foreign_transaction(request_Y)` (Bitcoin tx_id = `[0xBB; 32]`).
3. Both requests are pending in `pending_verify_foreign_tx_requests`.
4. The MPC network legitimately computes `payload_hash_Y = SHA-256(borsh(ForeignTxSignPayload { request: request_Y, values: [BlockHash([0xCC; 32])] }))` and produces MPC signature `sig_Y` over it.
5. A single Byzantine attested participant calls:
   ```
   respond_verify_foreign_tx(
     request = request_X,          // ← Alice's pending request
     response = {
       payload_hash: payload_hash_Y, // ← hash of Bob's transaction
       signature: sig_Y,             // ← valid MPC sig over payload_hash_Y
     }
   )
   ```
6. The contract checks: `verify_ecdsa_signature(sig_Y, payload_hash_Y, root_pk)` → **passes** (signature is genuinely valid).
7. The contract looks up `request_X` in `pending_verify_foreign_tx_requests` → **found**.
8. `resolve_yields_for` delivers `{payload_hash: payload_hash_Y, signature: sig_Y}` to Alice's waiting promise.
9. Alice's bridge contract receives a response claiming the MPC network verified her Bitcoin tx_id `[0xAA; 32]`, but the `payload_hash` actually encodes tx_id `[0xBB; 32]`. If Alice's bridge does not call `ForeignChainSignatureVerifier::verify_signature` (which is off-chain and optional), it proceeds with an invalid bridge execution. [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

        Ok(Hash256(hasher.finalize().into()))

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
