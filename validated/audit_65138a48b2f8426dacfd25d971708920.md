### Title
Off-by-One in `DelegateAction` Expiry Check Allows Relayer to Execute Meta-Transaction One Block Past User-Intended Deadline - (File: runtime/runtime/src/actions.rs)

### Summary

`apply_delegate_action` uses a strict `>` comparison to check whether a `DelegateAction` has expired, but the field's own documentation and the spec both state the action should be invalid when `block_height >= max_block_height`. A malicious relayer can hold a signed meta-transaction and submit it at exactly `block_height == max_block_height`, executing it one block after the user intended it to have expired.

### Finding Description

In `apply_delegate_action`, the expiry guard is:

```rust
if apply_state.block_height > delegate_action.max_block_height() {
    result.result = Err(ActionErrorKind::DelegateActionExpired.into());
    return Ok(());
}
``` [1](#0-0) 

The `DelegateAction` struct documents `max_block_height` as:

> "The maximal height of the block in the blockchain **below which** the given DelegateAction is valid." [2](#0-1) 

The runtime spec (`docs/RuntimeSpec/Actions.md`) states:

> "If the current block is **equal or greater than** `max_block_height`" → `DelegateActionExpired` [3](#0-2) 

Both the field description ("below which") and the spec ("equal or greater than") agree: the action should be invalid when `block_height >= max_block_height`. The code uses `>`, so when `block_height == max_block_height` the action is **not** rejected — it executes. This is the same class of off-by-one boundary error as the reported `ve_supply` bug.

The existing test `test_delegate_action_max_height` only tests `max_block_height + 1` (correctly fails), but `test_delegate_action` and `test_delegate_action_validate_sender_account` both call `create_apply_state(max_block_height)` and assert success — confirming the current code accepts execution at the boundary block. [4](#0-3) [5](#0-4) 

### Impact Explanation

A malicious relayer (an unprivileged user who submits transactions via the public RPC) can:

1. Receive a user-signed `DelegateAction` with `max_block_height = H`.
2. Withhold submission until block `H` is produced.
3. Submit the meta-transaction at block `H`.
4. The runtime executes it — even though the user set `max_block_height = H` expecting the action to be invalid at that height.

The user may have taken other on-chain actions between blocks `H-1` and `H` under the assumption the meta-transaction had expired (e.g., revoked an access key, changed a contract state, or transferred funds). The relayer's late execution can drain funds, replay an unintended transfer, or execute any inner action the user signed.

The corrupted protocol value is the **action execution result**: a receipt that should have been rejected as `DelegateActionExpired` is instead applied to state, mutating balances, access keys, or contract storage.

### Likelihood Explanation

**Medium.** The trigger requires a relayer who deliberately times submission to exactly `block_height == max_block_height`. This is a narrow window (one block), but block heights are public and predictable. Any relayer who receives a meta-transaction and wants to exploit the user only needs to monitor the chain and submit at the right height. Users who read the spec or field documentation and rely on `max_block_height` as an inclusive upper bound are directly vulnerable.

### Recommendation

Change the comparison from strict `>` to `>=`:

```diff
- if apply_state.block_height > delegate_action.max_block_height() {
+ if apply_state.block_height >= delegate_action.max_block_height() {
``` [6](#0-5) 

Update `test_delegate_action` and `test_delegate_action_validate_sender_account` to use `max_block_height - 1` as the valid apply-state height, and add a test asserting that `block_height == max_block_height` returns `DelegateActionExpired`.

### Proof of Concept

1. User signs a `DelegateAction` with `max_block_height = 1000` (e.g., a `Transfer` of 100 NEAR to a receiver).
2. User submits a separate transaction at block 999 revoking the access key used in the delegate action, expecting the meta-transaction to be dead at block 1000.
3. Malicious relayer monitors the chain and submits the `SignedTransaction` wrapping the `DelegateAction` at block 1000.
4. `apply_delegate_action` evaluates `1000 > 1000` → `false`, so no expiry error is returned.
5. The inner `Transfer` executes, draining 100 NEAR from the user's account despite the user's expectation that the action had expired. [7](#0-6) [8](#0-7)

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
```
