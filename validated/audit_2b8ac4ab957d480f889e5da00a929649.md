### Title
Missing Minimum Output Protection on All L2 Pool Deposit Functions — (File: contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

---

### Summary

Every L2 pool `deposit()` function accepts ETH or LSTs and returns wrsETH/rsETH at the current oracle rate, but none of them expose a `minRSETHAmountExpected` parameter. The L1 `LRTDepositPool` enforces this protection globally; the L2 pools do not. Any oracle rate movement between transaction submission and execution silently delivers fewer tokens to the depositor with no on-chain recourse.

---

### Finding Description

The L1 entry point `LRTDepositPool.depositETH()` and `LRTDepositPool.depositAsset()` both accept a `minRSETHAmountExpected` argument and revert inside `_beforeDeposit` if the computed mint amount falls below it:

```solidity
// LRTDepositPool.sol line 667
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [1](#0-0) 

Every L2 pool variant omits this control entirely. The ETH deposit path in `RSETHPool`:

```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [2](#0-1)

The output amount is computed as:

```solidity

`rsETHToETHrate` is fetched live from the oracle at execution time. If the rate rises between submission and execution, `rsETHAmount` decreases proportionally. The depositor has no on-chain mechanism to reject the outcome.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but does not lose value.**

A depositor who observes a rate of R at submission time and expects `amount * 1e18 / R` wrsETH may receive materially fewer tokens if the oracle rate is updated before their transaction is mined. The deposited ETH/LST is retained by the pool (to be bridged to L1), so no funds are destroyed, but the user receives fewer liquid restaking tokens than the protocol implicitly promised at submission time. This is the direct Solidity analog of the Android report's core finding: a security control that the L1 already enforces (`minRSETHAmountExpected`) is absent from every L2 pool, leaving users unprotected against rate drift on any current or future deposit path.

---

### Likelihood Explanation

**Medium.** The rsETH oracle rate is updated periodically by the protocol. On L2 chains with public mempools or sequencer ordering, a pending oracle update transaction is observable. A depositor whose transaction lands after an oracle update receives fewer tokens with no recourse. This is a routine operational condition, not a contrived edge case. The risk compounds as new tokens and oracles are added to pool variants (e.g., `addSupportedToken`), since each new oracle introduces an additional rate-drift surface with no slippage floor.

---

### Recommendation

Add a `minRSETHAmountExpected` parameter to every L2 pool `deposit()` overload and revert if the computed output falls below it, mirroring the existing L1 pattern:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same change to all token deposit overloads across `RSETHPool`, `RSETHPoolV3`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV2ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`.

---

### Proof of Concept

1. Oracle reports `rsETHToETHrate = 1.05e18` (1 rsETH = 1.05 ETH).
2. User submits `RSETHPoolV3.deposit{value: 1 ether}("ref")`, expecting `≈ 0.952 wrsETH`.
3. Before the transaction is mined, the oracle is updated to `rsETHToETHrate = 1.10e18`.
4. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 wrsETH`.
5. User receives `0.909 wrsETH` instead of `0.952 wrsETH` — a ~4.5% shortfall — with no revert and no on-chain protection.
6. The same outcome applies to every token deposit across all six L2 pool contracts, and to any new token added via `addSupportedToken` in the future. [8](#0-7) [11](#0-10)

    function _beforeDeposit(
        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
