Audit Report

## Title
Deposits Mint rsETH Using Stale Cached `rsETHPrice` Without Prior Oracle Update, Enabling Yield Extraction From Existing Holders - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.depositETH` and `depositAsset` compute the rsETH mint amount via `_beforeDeposit`, which calls `getRsETHAmountToMint`. That function divides by `lrtOracle.rsETHPrice()`, a plain storage variable in `LRTOracle` that is only written when `_updateRsETHPrice()` is explicitly invoked. Because neither deposit path triggers a price update before the calculation, any accrued staking rewards that have raised the true protocol TVL cause the stored price to be stale (too low), minting excess rsETH to the depositor and diluting existing holders' unclaimed yield.

## Finding Description
`LRTOracle.rsETHPrice` is a plain `uint256` storage variable: [1](#0-0) 

It is only written at the end of `_updateRsETHPrice()`: [2](#0-1) 

`getRsETHAmountToMint` reads this cached value directly: [3](#0-2) 

`_beforeDeposit` is declared `private view`, making it structurally incapable of calling any state-mutating function such as `updateRSETHPrice()`: [4](#0-3) 

Neither `depositETH` nor `depositAsset` calls `updateRSETHPrice()` before invoking `_beforeDeposit`: [5](#0-4) [6](#0-5) 

Additionally, when the true price has risen beyond `pricePercentageLimit`, the public `updateRSETHPrice()` reverts for non-managers, extending the staleness window: [7](#0-6) 

**Exploit path:**
1. Staking rewards accrue, increasing `totalETHInProtocol`. The stored `rsETHPrice` is not yet updated.
2. Attacker calls `depositETH` (or `depositAsset`). `getRsETHAmountToMint` computes `amount * assetPrice / rsETHPrice_stale`, yielding more rsETH than the deposit is worth at the true price.
3. Attacker (or anyone) calls `updateRSETHPrice()`. The price corrects upward.
4. Attacker's rsETH is now redeemable for more ETH than was deposited; the surplus was extracted from yield that had accrued to existing holders.

## Impact Explanation
This is **theft of unclaimed yield** (High). Existing rsETH holders have accrued yield reflected in the rising TVL but not yet in the stored price. A depositor who acts before the price update receives a larger ownership share than their deposit warrants. After the price corrects, that excess ownership is backed by yield that belonged to prior holders, whose proportional share is permanently diluted. The impact is concrete and quantifiable (as shown in the PoC), not hypothetical.

## Likelihood Explanation
`updateRSETHPrice()` is public with no access restriction, so any attacker can monitor on-chain state to detect staleness. EigenLayer staking rewards accrue continuously, meaning a staleness window exists after every reward event. The `pricePercentageLimit` guard can cause the public update to revert, extending the window further and preventing self-correction. No privileged access, special role, or victim mistake is required. The attack is repeatable on every reward cycle.

## Recommendation
Remove the `view` modifier from `_beforeDeposit` and call `updateRSETHPrice()` (or an internal equivalent) at its start, before any mint calculation:

```solidity
function _beforeDeposit(
    address asset,
    uint256 depositAmount,
    uint256 minRSETHAmountExpected
)
    private
    returns (uint256 rsethAmountToMint)
{
    // Refresh price before computing mint amount
    ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();

    if (depositAmount == 0 || depositAmount < minAmountToDeposit) revert InvalidAmountToDeposit();
    if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) revert MaximumDepositLimitReached();

    rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
    if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
}
```

This ensures the mint calculation always uses the current price, eliminating the staleness window.

## Proof of Concept
**State at t=0:** `rsETHPrice = 1.00 ETH`, TVL = 1000 ETH, rsETH supply = 1000.

**t=1:** 10 ETH of staking rewards accrue. True price = 1010/1000 = 1.01 ETH. `rsETHPrice` still = 1.00 (not updated).

**Attacker action:** Call `depositETH{value: 100 ETH}`.
- `rsethAmountToMint = 100 / 1.00 = 100 rsETH` (correct would be `100 / 1.01 ≈ 99.01 rsETH`).

**t=2:** Anyone calls `updateRSETHPrice()`.
- New supply = 1100 rsETH, new TVL = 1110 ETH.
- `rsETHPrice = 1110 / 1100 ≈ 1.009 ETH`.

**Result:** Attacker holds 100 rsETH worth `100 × 1.009 = 100.9 ETH` — deposited 100 ETH, extracted ~0.9 ETH of yield from existing holders.

A Foundry fork test can reproduce this by: (1) deploying the contracts, (2) simulating reward accrual by directly increasing the node delegator's ETH balance, (3) calling `depositETH` without a prior `updateRSETHPrice`, (4) calling `updateRSETHPrice`, and (5) asserting that the attacker's rsETH redemption value exceeds their deposit.
