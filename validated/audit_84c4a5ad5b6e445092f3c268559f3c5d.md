### Title
Negative `RequireMOf` Threshold in Native Scripts Unconditionally Bypasses M-of-N Witness Requirement - (File: `eras/allegra/impl/src/Cardano/Ledger/Allegra/Scripts.hs`)

---

### Summary

The `TimelockMOf` / `RequireMOf` constructor in Cardano native scripts stores the M-of-N threshold as a plain `Int`, which can be negative. The evaluator immediately returns `True` when the threshold `n ≤ 0`, making any `RequireMOf` script with a negative threshold unconditionally satisfied — no signatures or sub-script witnesses are required. The CDDL specification permits `int64` (and `int32` in Shelley) for this field, so any unprivileged transaction sender can submit a native script with a negative threshold on-chain. This is a direct analog to the reported Plonk bug: a user-supplied integer is used in a comparison-based validation without a lower-bound range check.

---

### Finding Description

**Root cause — `TimelockRaw` type and evaluator:**

In `eras/allegra/impl/src/Cardano/Ledger/Allegra/Scripts.hs`, the `TimelockMOf` constructor stores the threshold as `!Int`:

```haskell
| TimelockMOf !Int !(StrictSeq (Timelock era))
| -- Note that the Int may be negative in which case (TimelockMOf (-2) [..]) is always True
``` [1](#0-0) 

The evaluator `isValidMOf` inside `evalTimelock` uses `n <= 0` as its base case:

```haskell
isValidMOf n SSeq.Empty = n <= 0
isValidMOf n (ts SSeq.:<| tss) =
  n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
``` [2](#0-1) 

When `n` is any negative integer, `n <= 0` is immediately `True` on the very first call, so the entire `RequireMOf` branch short-circuits to `True` without evaluating a single sub-script or checking any key hash.

The same pattern is present in the Shelley `MultiSig` evaluator: [3](#0-2) 

And in the Dijkstra evaluator: [4](#0-3) 

**On-chain format permits negative values:**

The CDDL specifications across all eras explicitly allow negative thresholds. Shelley uses `int32`:

```
script_n_of_k = (3, n : int32, [* native_script])
int32 = -2147483648 .. 2147483647
``` [5](#0-4) 

From Allegra onwards, `int64` is used:

```
script_n_of_k = (3, n : int64, [* native_script])
int64 = min_int64 .. max_int64
min_int64 = -9223372036854775808
``` [6](#0-5) 

The deserialization path (`decRaw 3`) decodes the threshold with a plain `Ann From` — no non-negativity check is performed: [7](#0-6) 

**Discrepancy with the formal specification:**

The formal specification defines `MOfN ∈ N → seq(Timelock) → Timelock` where `N` is the natural numbers (non-negative integers): [8](#0-7) 

The implementation uses `Int` without enforcing this lower bound.

---

### Impact Explanation

An attacker can craft a native script `RequireMOf (-1) [RequireSignature k1, RequireSignature k2]`. This script:

1. **Always passes witness validation** — no key signatures are required, regardless of what keys are listed.
2. **Can serve as a minting policy** — the policy ID is `hash(script)`. Anyone can mint tokens under this policy ID without providing any of the listed key signatures. A protocol or user that accepts tokens under this policy ID, believing it is secured by those keys, can be deceived.
3. **Can serve as a UTxO spending condition** — any UTxO locked by such a script is freely spendable by anyone.

This is a **witness/script validation bypass**: attacker-controlled serialized inputs exceed intended validation limits (the threshold must be ≥ 0 per the formal spec) and allow minting/spending outside design parameters.

**Allowed impact matched**: Medium — "Attacker-controlled transactions, blocks, certificates, votes, proposals, scripts, witnesses, or serialized inputs exceed intended validation limits or modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters."

---

### Likelihood Explanation

The attack requires only the ability to submit a transaction — any unprivileged transaction sender can craft a native script with a negative threshold. The CDDL format explicitly permits the negative range, so no special encoding tricks are needed. The attacker does not need any keys, privileged access, or governance majority. The entry path is: craft CBOR for `[3, -1, [script1, script2]]`, include it as a native script in a transaction, and submit.

---

### Recommendation

Add a non-negativity check during deserialization of the `TimelockMOf` / `MultiSigMOf` threshold. In `decRaw 3` inside `DecCBOR (Annotator (TimelockRaw era))`:

```haskell
decRaw 3 = Ann (SumD TimelockMOf)
  <*! Ann (D (do { n <- decCBOR; when (n < 0) (fail "RequireMOf threshold must be non-negative"); pure n }))
  <*! D (sequence <$> decCBOR)
```

Alternatively, change the threshold type from `Int` to `Natural` or `Word32`, which would enforce non-negativity at the type level and align the implementation with the formal specification's `N` type.

The same fix should be applied to `MultiSigMOf` in `eras/shelley/impl/src/Cardano/Ledger/Shelley/Scripts.hs` and the Dijkstra evaluator.

---

### Proof of Concept

1. Construct a native script in CBOR: `[3, -1, [[0, <any_keyhash>]]]` — this is `RequireMOf (-1) [RequireSignature k]`.
2. Compute the script hash: `policy_id = blake2b_224(0x00 || cbor_script)`.
3. Submit a transaction with `mint = {policy_id: {"": 1000000}}` and include the script in the witness set, but provide **no** `vkeywitness` for `k`.
4. The ledger calls `evalTimelock` → `isValidMOf (-1) [RequireSignature k]` → `(-1) <= 0` → `True`.
5. The transaction is accepted and 1,000,000 tokens are minted under `policy_id` without the key `k`'s signature.
