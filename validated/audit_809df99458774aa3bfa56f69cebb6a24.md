Audit Report

## Title
ETH Donation via Open `receive()` Inflates TVL and Triggers Excess Protocol Fee Minting - (`contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary
`LRTDepositPool` exposes an unrestricted `receive() external payable {}` at line 58, and `getETHDistributionData()` measures TVL using the raw `address(this).balance`. Any unprivileged actor can donate ETH directly to the pool, inflating `totalETHInProtocol`. When `LRTOracle.updateRSETHPrice()` is subsequently called, the inflated TVL is treated as earned yield, causing excess protocol fee rsETH to be minted to the treasury and permanently diluting all existing rsETH holders.

## Finding Description
`_updateRsETHPrice()` computes `previousTVL = rsethSupply.mulWad(rsETHPrice)` at line 234. It then calls `_getTotalEthInProtocol()` (lines 331–349), which calls `ILRTDepositPool.getTotalAssetDeposits(ETH_TOKEN)`, which routes to `getETHDistributionData()`. That function reads `ethLyingInDepositPool = address(this).balance` at line 480 — a raw balance that includes any ETH sent via the open `receive()`.

When `totalETHInProtocol > previousTVL`, lines 244–246 compute:
```
rewardAmount     = totalETHInProtocol - previousTVL   // entirely the donation
protocolFeeInETH = rewardAmount * protocolFeeInBPS / 10_000
```
Lines 299–307 then mint `protocolFeeInETH.divWad(newRsETHPrice)` rsETH to the treasury. This rsETH is minted against ETH that was never earned as yield — it is the attacker's donation being misclassified as protocol revenue.

The `_checkAndUpdateDailyFeeMintLimit` guard (lines 204–209) caps the per-day mint amount but does not prevent the attack; it only bounds the per-day damage. The price inflation itself is unconditional and persists. The `pricePercentageLimit` check (lines 256–265) can cause a secondary DoS for non-manager callers if the donation is large, but does not block the fee mint path for smaller donations.

No privileged access is required. The entire exploit path is:
1. Attacker calls `LRTDepositPool.receive()` (or plain ETH transfer) — no access control.
2. `address(this).balance` increases by the donation amount.
3. `_getTotalEthInProtocol()` returns an inflated value.
4. `totalETHInProtocol > previousTVL` triggers fee computation on the donated ETH.
5. Excess rsETH is minted to treasury; `rsETHPrice` is updated to the inflated value.

## Impact Explanation
**High — Theft of unclaimed yield.** The excess rsETH minted to the treasury is funded by diluting all existing rsETH holders. Each holder's proportional claim on the underlying ETH is permanently reduced by `protocolFeeInBPS * donationAmount / totalETHInProtocol`. The attacker sacrifices the donated ETH but causes a permanent, irreversible redistribution of yield from rsETH holders to the treasury. This matches the allowed impact class "Theft of unclaimed yield."

## Likelihood Explanation
The attack requires only a plain ETH transfer to a public contract address — no privileged role, no front-running, no complex setup. The cost is the donated ETH; the damage is proportional to the donation and is permanent once `updateRSETHPrice()` is called. The attack is repeatable across days (subject to the daily fee mint cap). Any externally-owned account can execute it.

## Recommendation
1. Replace `address(this).balance` in `getETHDistributionData()` with an internal accounting variable (e.g., `totalETHDeposited`) incremented only through the authorized entry points: `depositETH`, `receiveFromRewardReceiver`, `receiveFromNodeDelegator`, `receiveFromLRTConverter`.
2. Replace `IERC20(asset).balanceOf(address(this))` in `getAssetDistributionData()` with a corresponding internal accounting variable for each supported LST.
3. Alternatively, restrict `receive()` to only accept ETH from known authorized senders (NodeDelegators, reward receivers, LRTConverter), reverting for all other callers.

## Proof of Concept
```
Initial state:
  rsethSupply      = 1000e18
  rsETHPrice       = 1.05e18
  previousTVL      = 1050e18
  protocolFeeInBPS = 1000 (10%)

Step 1: attacker sends 100 ETH to LRTDepositPool (open receive())
  address(LRTDepositPool).balance increases by 100e18

Step 2: anyone calls LRTOracle.updateRSETHPrice()
  totalETHInProtocol = 1150e18   (inflated)
  rewardAmount       = 100e18    (entirely the donation)
  protocolFeeInETH   = 10e18
  newRsETHPrice      = (1150 - 10) / 1000 = 1.14e18
  rsethAmountToMint  = 10e18 / 1.14e18 ≈ 8.77e18 rsETH minted to treasury

Result:
  Treasury receives ~8.77 rsETH it did not earn.
  All rsETH holders are diluted by ~0.877%.
  rsETHPrice is permanently inflated from 1.05 to 1.14.
```

Foundry fork test plan:
- Fork mainnet, deploy/use existing LRTDepositPool and LRTOracle.
- Record `rsETHPrice` and treasury rsETH balance before.
- `vm.deal(attacker, 100 ether); vm.prank(attacker); (bool ok,) = address(lrtDepositPool).call{value: 100 ether}("");`
- Call `lrtOracle.updateRSETHPrice()`.
- Assert treasury rsETH balance increased and existing holder's ETH-per-rsETH ratio decreased.
