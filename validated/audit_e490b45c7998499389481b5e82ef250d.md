### Title
Unvalidated `builder` field in `ProposalInit` allows a malicious proposer to redirect all block transaction fees to an arbitrary address - (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

### Summary

`is_proposal_init_valid` validates many fields of `ProposalInit` (timestamp, starknet_version, version_constant_commitment, height, l1_da_mode, l2_gas_price_fri, L1 gas prices, fee_proposal_fri) but never validates the `builder` field. That field is passed verbatim as `sequencer_address` into `BlockInfo`, which is the address that receives all transaction fees during block execution. A malicious proposer can set `builder` to any address they control, causing every validator to execute the block with the wrong sequencer address and commit a state where all fees flow to the attacker.

### Finding Description

`is_proposal_init_valid` in `validate_proposal.rs` performs the following checks on `ProposalInit`: [1](#0-0) 

It validates `height`, `l1_da_mode`, `l2_gas_price_fri`, L1 gas prices with a margin, `starknet_version`, `version_constant_commitment`, `timestamp` bounds, and `fee_proposal_fri` bounds. The `builder` field is never checked.

`convert_to_sn_api_block_info` then maps `init.builder` directly to `sequencer_address`: [2](#0-1) 

This `BlockInfo` is passed to the batcher via `initiate_validation`: [3](#0-2) 

The batcher executes every transaction in the block using this `BlockInfo`. In Starknet, the `sequencer_address` is the recipient of all transaction fees. Because every validator independently reconstructs `BlockInfo` from the received `ProposalInit.builder`, they all execute with the same (attacker-supplied) `sequencer_address`. The resulting state diff — including fee transfers to the attacker's address — is identical on every validator, so the batcher commitment matches `fin.proposal_commitment` and the proposal is accepted.

The same `init.builder` is also forwarded to state sync: [4](#0-3) 

The proposer sets `builder` from a static config field with an explicit TODO noting it should eventually come from the committee: [5](#0-4) [6](#0-5) 

No validator-side check ever compares the received `builder` against any locally-trusted value.

### Impact Explanation

A malicious proposer sets `builder` to an address they control. Every validator executes the block with `sequencer_address = attacker_address`. All transaction fees for the block are transferred to the attacker in the committed state. The legitimate sequencer receives nothing. The committed state diff, block header, and state sync record all reflect the wrong sequencer address. This is a direct, per-block economic attack with no cap on the fee amount stolen.

Impact category: **Critical — Incorrect fee/balance effect with economic impact.**

### Likelihood Explanation

The attacker must be the selected proposer for a round, which is a consensus-level privilege. However, once selected, exploitation requires only setting one field in `ProposalInit` to an arbitrary address. There is no guard, no margin check, and no committee-derived expected value to compare against. The attack is silent — the commitment check passes because all validators derive the same wrong state from the same wrong `builder`.

### Recommendation

Add a check in `is_proposal_init_valid` that rejects any `ProposalInit` whose `builder` does not match the locally-configured `builder_address` (or, once available, the committee-derived builder for the round). This mirrors the existing `proposer` check in `handle_proposal`: [7](#0-6) 

Until the committee provides the builder address, the validator should compare `init.builder` against its own `static_config.builder_address` and reject proposals that differ.

### Proof of Concept

1. Attacker is selected as proposer for height H, round R.
2. Attacker constructs `ProposalInit` with all fields set legitimately **except** `builder`, which is set to `attacker_wallet`.
3. Attacker streams transactions and sends `ProposalFin` with a `proposal_commitment` computed by their batcher (which used `sequencer_address = attacker_wallet`).
4. Each validator calls `is_proposal_init_valid` — `builder` is never checked, validation passes.
5. Each validator calls `batcher.validate_block` with `block_info.sequencer_address = attacker_wallet`.
6. The batcher executes all transactions; fee transfers go to `attacker_wallet` in the state diff.
7. The validator's batcher commitment matches the proposer's `fin.proposal_commitment` — `ProposalFinMismatch` is not triggered. [8](#0-7) 

8. Consensus reaches decision; `decision_reached` commits the block. The state now has `attacker_wallet` holding all fees from block H. The legitimate sequencer address received nothing.
