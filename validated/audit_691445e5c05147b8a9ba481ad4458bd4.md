### Title
`ForeignTxSignPayload` digest lacks contract identity and NEAR network binding, enabling cross-deployment replay — (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

`ForeignTxSignPayload::compute_msg_hash()` produces a hash that commits only to the foreign-chain request and extracted values. It omits the NEAR contract account ID, NEAR network identifier (mainnet/testnet), and `domain_id`. A Byzantine participant (below threshold) who observes a valid `VerifyForeignTransactionResponse` from one NEAR MPC deployment can replay it verbatim on a second deployment that shares the same MPC key material, causing the second contract to accept a forged foreign-chain verification attestation.

---

### Finding Description

The signed payload is constructed and hashed as follows:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
``` [1](#0-0) 

The resulting `msg_hash = SHA-256(borsh(ForeignTxSignPayload))` contains only `{request, values}`. Absent from the commitment are:

- The NEAR contract account ID (contract identity)
- The NEAR network/chain ID (`mainnet`, `testnet`, etc.)
- The `domain_id` that scopes the signing key

The node-side `build_signature_request` function feeds this hash directly into the threshold ECDSA signing pipeline:

```rust
fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    ...
    Ok(SignatureRequest {
        ...
        tweak: Tweak::new([0u8; 32]),   // zero tweak — no key derivation
        ...
    })
}
``` [2](#0-1) 

The contract's `respond_verify_foreign_tx` verifies the signature against the domain's root public key and resolves the pending yield:

```rust
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
``` [3](#0-2) 

Because `payload_hash` is identical for the same `(request, values)` tuple regardless of which NEAR contract or network produced it, any valid `VerifyForeignTransactionResponse` is portable across deployments that share the same MPC key.

The SDK-level verifier used by downstream bridge contracts has the same gap:

```rust
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: self.request,
    values: self.expected_extracted_values,
});
let expected_payload_hash = expected_payload.compute_msg_hash()...;
``` [4](#0-3) 

---

### Impact Explanation

A valid `VerifyForeignTransactionResponse` produced by the MPC network on deployment A (e.g., NEAR testnet) is cryptographically indistinguishable from one produced on deployment B (e.g., NEAR mainnet) if both deployments share the same MPC key material. A Byzantine participant below the signing threshold can:

1. Observe a legitimately produced response `R = (payload_hash, sig)` on deployment A.
2. Call `respond_verify_foreign_tx(request, R)` on deployment B, where the same foreign-chain request is pending.
3. The contract on B accepts `R` because `sig` verifies against B's domain public key (same key) and `payload_hash` matches (same hash construction).
4. All callers waiting on that request on B receive the forged attestation.

This constitutes **forged foreign-chain verification** and can cause **invalid bridge execution** — e.g., an Omnibridge inbound flow on mainnet is unlocked using a testnet-generated attestation, enabling a double-spend.

---

### Likelihood Explanation

The preconditions are:

1. **Same MPC key material across two NEAR deployments.** This occurs during contract migrations/upgrades where key shares are transferred to a new contract account, or when a staging/testnet cluster is promoted to production without re-running DKG. The NEAR MPC system explicitly supports resharing and migration (`crates/node/src/node_migration_service.rs`), making this a realistic operational scenario.
2. **A Byzantine participant below threshold.** The `respond_verify_foreign_tx` entry point requires `assert_caller_is_attested_participant_and_protocol_active`, so the replayer must be an active, attested participant — within the explicit Byzantine threat model of a threshold system.

Neither condition requires threshold-level collusion, leaked keys, or physical TEE attacks.

---

### Recommendation

Fold a domain separator into `compute_msg_hash` that commits to at minimum:

- The NEAR contract account ID (`env::current_account_id()` at request time, stored in the request or passed into the hash)
- The NEAR network identifier (e.g., `mainnet` / `testnet`)
- The `domain_id`

Concretely, extend `ForeignTxSignPayloadV1` (or introduce a wrapper struct used only for hashing) to include these fields before Borsh-serializing:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub contract_id: AccountId,   // NEAR contract account
    pub network_id: String,       // "mainnet" | "testnet" | ...
    pub domain_id: DomainId,
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

This mirrors the EIP-712 domain separator recommendation in the external report and ensures every signed attestation is single-instance–bound.

### Proof of Concept

```
1. Deploy MPC contract A at `mpc.testnet` (NEAR testnet), participants {P1, P2, P3}, threshold 2.
   DKG produces key K with public key PK.

2. Migrate key material to contract B at `mpc.mainnet` (NEAR mainnet), same participants, same K/PK.
   (Realistic during a testnet→mainnet promotion without re-running DKG.)

3. User U_testnet calls verify_foreign_transaction on A for Bitcoin tx T.
   MPC nodes sign SHA-256(borsh({request=T, values=[blockHash=H]})) → sig_T.
   Response R = (payload_hash, sig_T) is submitted to A and returned to U_testnet.

4. Byzantine participant P1 (below threshold) observes R on-chain (NEAR is public).

5. User U_mainnet independently calls verify_foreign_transaction on B for the same Bitcoin tx T.
   B queues the request under key (domain_id=0, payload_version=V1, request=T).

6. P1 calls respond_verify_foreign_tx(request=(domain_id=0, V1, T), response=R) on B.
   - assert_caller_is_attested_participant_and_protocol_active: P1 is attested on B ✓
   - verify_ecdsa_signature(sig_T, payload_hash, PK): same key PK, same hash → ✓
   - resolve_yields_for: U_mainnet's yield is resumed with R.

7. U_mainnet's bridge contract receives R, calls ForeignChainSignatureVerifier::verify_signature,
   which also uses compute_msg_hash without contract/network binding → accepts R as valid.

8. Bridge on mainnet unlocks funds for tx T using a testnet-generated attestation.

```rust
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
