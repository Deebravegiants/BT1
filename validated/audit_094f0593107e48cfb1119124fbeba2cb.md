Audit Report

## Title
`_updateRsETHPrice()` Reverts on Daily Fee Mint Limit Exceeded, Permanently Blocking rsETH Price Updates - (File: contracts/LRTOracle.sol)

## Summary

`_checkAndUpdateDailyFeeMintLimit()` unconditionally reverts with `DailyFeeMintLimitExceeded` when the computed protocol fee in rsETH exceeds the remaining daily budget. Because this call occurs at line 303 — before `rsETHPrice = newRsETHPrice` at line 313 — the price write is never reached. Both public entry points (`updateRSETHPrice()` and `updateRSETHPriceAsManager()`) share the same internal path with no bypass, leaving `rsETHPrice` stale for the remainder of the 24-hour period and permanently destroying the unclaimed protocol fee for that period.

## Finding Description

In `_updateRsETHPrice()`, after computing `newRsETHPrice` and `protocolFeeInETH`, the function calls `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` at line 303:

```solidity
// contracts/LRTOracle.sol lines 298-313
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee); // ← reverts here
    ...
} else {
    _checkAndUpdateDailyFeeMintLimit(0);
}
rsETHPrice = newRsETHPrice; // ← never reached on revert
``` [1](#0-0) 

`_checkAndUpdateDailyFeeMintLimit` reverts unconditionally when the fee exceeds the budget:

```solidity
// contracts/LRTOracle.sol lines 205-206
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
``` [2](#0-1) 

Both public entry points call `_updateRsETHPrice()` with no bypass or alternative path: [3](#0-2) 

The `else` branch (`_checkAndUpdateDailyFeeMintLimit(0)`) is safe because `currentPeriodMintedFeeAmount + 0 > maxFeeMintAmountPerDay` is always false given the invariant maintained by the check. The revert path is exclusively triggered when `protocolFeeInETH > 0` and the fee exceeds the remaining daily budget.

The downside-protection auto-pause (lines 270–281) executes before the fee check, so it is not blocked. However, the price write at line 313 is always blocked when the revert fires. [4](#0-3) 

## Impact Explanation

**Permanent freezing of unclaimed yield (Medium) and Temporary freezing of funds (Medium).**

1. **Permanent freezing of unclaimed yield**: The protocol fee for the blocked period is permanently unrecoverable. The fee is computed from `totalETHInProtocol - previousTVL`; once the period rolls over, `previousTVL` is recalculated from the now-stale `rsETHPrice`, erasing the missed fee delta. The treasury never receives the yield it was entitled to.

2. **Temporary freezing of funds**: `rsETHPrice` is frozen at its last successful value for up to 24 hours. `LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` and `LRTWithdrawalManager.getExpectedAssetAmount()` multiplies by it, so all deposits and withdrawals execute at the wrong rate during the blocked window, diluting or shortchanging users. [5](#0-4) 

## Likelihood Explanation

`maxFeeMintAmountPerDay` is a finite manager-set value. The fee per call is `(totalETHInProtocol - previousTVL) * protocolFeeInBPS / 10_000 / newRsETHPrice`. A single large reward event (EigenLayer distribution, LST price spike, or a long gap between updates) can produce a fee exceeding the daily cap in one call. `updateRSETHPrice()` is public and permissionless — any external caller triggers the revert. No privileged actor can bypass it through the same code path. The condition is repeatable every time the TVL delta is large relative to `maxFeeMintAmountPerDay`.

## Recommendation

Decouple the fee-minting limit check from the price-update logic. The simplest fix is to cap the minted fee at the remaining daily budget rather than reverting:

```solidity
uint256 remaining = maxFeeMintAmountPerDay - currentPeriodMintedFeeAmount;
uint256 mintable = rsethAmountToMintAsProtocolFee > remaining
    ? remaining
    : rsethAmountToMintAsProtocolFee;
currentPeriodMintedFeeAmount += mintable;
// mint `mintable`, not `rsethAmountToMintAsProtocolFee`
```

Alternatively, update `rsETHPrice` unconditionally first, then attempt the fee mint in a separate step (or wrap it in a try/catch) so a limit breach cannot roll back the price write.

## Proof of Concept

1. Manager calls `setMaxFeeMintAmountPerDay(1e18)` (1 rsETH cap per day).
2. EigenLayer rewards accrue; `totalETHInProtocol` grows by 200 ETH relative to `previousTVL`.
3. With `protocolFeeInBPS = 1000` (10%), `protocolFeeInETH = 20 ETH`.
4. At `newRsETHPrice ≈ 1.05e18`, `rsethAmountToMintAsProtocolFee ≈ 19.05e18 > 1e18`.
5. `_checkAndUpdateDailyFeeMintLimit(19.05e18)` reverts with `DailyFeeMintLimitExceeded`.
6. `updateRSETHPrice()` reverts; `rsETHPrice` at line 313 is never written.
7. For the next ~24 hours, all deposits and withdrawals use the stale price; the 20 ETH protocol fee is permanently lost.

**Foundry test sketch:**
```solidity
function test_priceUpdateBlockedByDailyLimit() public {
    vm.prank(manager);

    vm.expectRevert(LRTOracle.DailyFeeMintLimitExceeded.selector);

}
