### Title
Responder-Key `respond` Calls Always Fail Due to Attestation Public-Key Mismatch — (`crates/contract/src/tee/tee_state.rs`)

---

### Summary

Every node response function (`respond`, `respond_ckd`, `respond_verify_foreign_tx`) enforces `assert_caller_is_attested_participant_and_protocol_active()`, which requires the **signing key's public key** to exactly match the key stored during `submit_participant_info`. The design explicitly provisions separate *responder keys* (additional function-call access keys) for throughput when posting signatures on-chain. Because those responder keys carry a different public key than the attested node key, every `respond*` call submitted through a responder key is unconditionally rejected with `AttestationKeyMismatch`, permanently breaking the throughput path for signature delivery.

---

### Finding Description

**Attestation check in `is_caller_an_attested_participant`** [1](#0-0) 

The function reads `env::signer_account_pk()` — the public key of whichever NEAR access key signed the current transaction — and compares it against `attestation.node_id.account_public_key`, which was recorded when the node called `submit_participant_info`.

**Attestation is registered with the node's main account key** [2](#0-1) 

`submit_participant_info` captures `env::signer_account_pk()` at submission time and stores it as `account_public_key` inside the `NodeId`. This is the node's primary Ed25519 key.

**All three respond functions gate on this check** [3](#0-2) [4](#0-3) [5](#0-4) 

**Design explicitly provisions responder keys for throughput**

The design document states:

> "In addition — The node will generate multiple responder account keys as well. These are used to increase throughput when posting generated MPC signatures to the chain." [6](#0-5) 

When a responder key (a separate function-call access key) signs a `respond` transaction, `env::signer_account_pk()` returns the responder key's public key. That key was never submitted to `submit_participant_info`, so `attestation.node_id.account_public_key != signer_ed25519` is always true, and the call panics unconditionally.

This is structurally identical to the reference bug: `burn()` only allows `PolicyCenter` but `PayoutPool` must call it — here, `respond` only allows the attested key but responder keys must call it.

---

### Impact Explanation

**Medium — request-lifecycle invariant broken.**

Every `respond*` transaction submitted via a responder key is rejected. Under any non-trivial signing load the node falls back to serialising all responses through its single main account key, collapsing throughput. Pending `sign` / `request_app_private_key` / `verify_foreign_transaction` yields accumulate faster than they can be drained, causing them to hit the NEAR yield-resume timeout and fire `fail_on_timeout`, returning errors to every waiting caller. No funds are directly stolen, but the signing service's liveness guarantee is broken for any load that requires the responder-key throughput path.

---

### Likelihood Explanation

**High.** The responder-key mechanism is an explicit, documented design feature intended for production throughput. Any deployment that activates it (i.e., any production node under meaningful load) immediately triggers the mismatch on every responder-key-signed `respond` call. No special attacker action is required; the system's own normal operation reproduces the failure.

---

### Recommendation

One of the following fixes should be applied:

1. **Register responder keys in the attestation state.** Extend `submit_participant_info` (or add a new method) to record additional authorised signing keys per node, and update `is_caller_an_attested_participant` to accept any of them.
2. **Separate the throughput path from the attestation check.** Gate `respond*` on account-ID membership in the participant set (already checked) rather than on the exact signing-key public key. Reserve the stricter key-binding check for governance votes where confused-deputy risk is real.
3. **Issue responder keys from the same key material as the attested key.** If the node derives responder keys deterministically from the attested key, the stored `account_public_key` can be extended to a set, or the check can be relaxed to a key-family comparison.

---

### Proof of Concept

```
1. Node boots, generates main Ed25519 key K_main and TLS key K_tls.
2. Node calls submit_participant_info(attestation, K_tls) signed with K_main.
   → stored_attestations[K_tls].node_id.account_public_key = K_main.

3. Node generates responder key K_resp (different Ed25519 key) and registers it
   as a function-call access key on its NEAR account.

4. Node completes a signing round and calls respond(request, sig) signed with K_resp.
   → env::signer_account_pk() = K_resp
   → is_caller_an_attested_participant():
       signer_ed25519 = K_resp
       attestation.node_id.account_public_key = K_main
       K_resp ≠ K_main  →  Err(AttestationKeyMismatch)
   → assert_matches! panics: "Caller must be an attested participant"

5. The pending yield is never resumed; after ~200 blocks the runtime fires
   fail_on_timeout, returning RequestError::Timeout to the original sign() caller.
```

Every `respond`, `respond_ckd`, and `respond_verify_foreign_tx` call made through any responder key follows this path and always fails, regardless of whether the cryptographic signature in the response is valid.

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
