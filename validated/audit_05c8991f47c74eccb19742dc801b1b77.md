Audit Report

## Title
Global Daily Mint Limit Exhaustion Enables Block Stuffing DoS on ETH Deposits — (File: contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

## Summary
The `limitDailyMint` modifier enforces a global daily minting cap with no per-address sub-limit, allowing a single unprivileged caller to exhaust the entire `dailyMintLimit` in one `deposit()` call. On L2 chains where this contract is deployed, an attacker can follow this with block stuffing to prevent legitimate users from landing any `deposit()` transactions for the remainder of the 86400-second window, constituting a Low-severity block stuffing DoS.

## Finding Description
`getCurrentDay()` at L390–392 computes the epoch as `(block.timestamp - startTimestamp) / 1 days`. The `limitDailyMint` modifier at L108–137 resets `dailyMintAmount` only when `currentDay > lastMintDay`; otherwise it enforces a hard cap at L131–133 and accumulates at L135. There is no per-address tracking anywhere in the modifier or in `deposit(string)`. A single caller can send ETH equal to the full daily quota in one transaction, setting `dailyMintAmount == dailyMintLimit`. Every subsequent `deposit()` from any address then reverts with `DailyMintLimitExceeded` for the rest of the day. The `deposit(string)` function at L282–301 is fully permissionless — it requires only `whenNotPaused` and `isEthDepositEnabled`, with no role, whitelist, or per-user cap. On L2 chains (evidenced by `l1VaultETHForL2Chain` and the native bridge architecture), block stuffing is economically feasible due to lower block gas limits and cheap gas. After exhausting the daily limit, the attacker fills every subsequent block with high-gas dummy transactions, preventing legitimate users' `deposit()` calls from being included until the day boundary resets the limit. Note: the PoC's use of `vm.warp(frozenTs)` to pin `block.timestamp` overstates the "indefinite extension" claim — L2 sequencers advance timestamps regardless of block stuffing, so the DoS window is bounded to the current 86400-second epoch per attack cycle. The attacker must re-exhaust the limit each day to sustain the DoS.

## Impact Explanation
**Low — Block stuffing.** All ETH `deposit()` calls revert with `DailyMintLimitExceeded` for the duration of the stuffed window (up to 24 hours per cycle). The attacker receives wrsETH in exchange for their ETH deposit (no net loss on the limit exhaustion step), making the attack economically viable. The impact matches the explicitly allowed "Low. Block stuffing." impact class.

## Likelihood Explanation
The attack is fully permissionless — no special role is required. Preconditions are: `isEthDepositEnabled == true` (normal operating state), a finite `dailyMintLimit` (always true), and the attacker holding ETH sufficient to match the daily quota. On L2 chains with low block gas limits and cheap gas (e.g., Optimism, Base, Scroll), block stuffing is a documented and repeatable attack vector. The attacker recovers their deposited ETH value as wrsETH, reducing the net cost to only the block stuffing gas overhead.

## Recommendation
1. Add a per-address daily deposit cap inside `limitDailyMint` (e.g., `mapping(address => mapping(uint256 => uint256)) public userDailyMintAmount`) to prevent a single actor from consuming the global quota.
2. Enforce a maximum single-deposit size relative to `dailyMintLimit` (e.g., no single deposit may exceed a configurable percentage of the daily limit).
3. Consider a sliding-window rate limiter instead of a fixed epoch boundary to mitigate the cliff-reset griefing pattern.

## Proof of Concept
1. Deploy or fork the contract with `isEthDepositEnabled = true` and a finite `dailyMintLimit`.
2. Attacker calls `deposit{value: quota}("ref")` where `quota` is the ETH equivalent of `dailyMintLimit` — this sets `dailyMintAmount == dailyMintLimit` in one transaction.
3. Attacker submits high-gas dummy transactions to fill every subsequent block on the L2, preventing user transactions from being included.
4. Any victim call to `deposit{value: 1 ether}("ref")` either cannot land (block stuffing) or reverts with `DailyMintLimitExceeded` if it does land.
5. After 86400 seconds, `getCurrentDay() > lastMintDay` resets the limit — the attacker must repeat step 2 to sustain the DoS each day.

Foundry test sketch:
```solidity
uint256 quota = pool.dailyMintLimit();
vm.deal(attacker, quota);
vm.prank(attacker);
pool.deposit{value: quota}("ref");
assertEq(pool.dailyMintAmount(), quota);

// Simulate block stuffing: blocks advance but timestamp stays within same day
vm.roll(block.number + 1000);
// timestamp NOT warped past day boundary

vm.deal(victim, 1 ether);
vm.prank(victim);
vm.expectRevert(IPool.DailyMintLimitExceeded.selector);
pool.deposit{value: 1 ether}("ref");
```
