### Title
`create_block_context` Ignores Pending Block's `starknet_version`, Always Falls Back to `StarknetVersion::LATEST` for Pending Fee Estimation — (`crates/apollo_rpc_execution/src/lib.rs`)

---

### Summary

`create_block_context` resolves `starknet_version` by querying storage for the pending block number, which is never in storage, causing an unconditional fallback to `StarknetVersion::LATEST`. Gas prices, however, come from `pending_data`. For `block_id=Latest`, both gas prices and `starknet_version` come from the stored header. When the latest stored block's version differs from `LATEST`, the two calls produce different `VersionedConstants` and therefore different fee estimates for identical transactions and identical effective gas prices.

---

### Finding Description

In `create_block_context` (`crates/apollo_rpc_execution/src/lib.rs`), the pending branch sets `block_number = block_context_number.unchecked_next()` — the next (pending) block number, which has not yet been written to storage: [1](#0-0) 

Immediately after, `starknet_version` is resolved by querying storage for that same `block_number`: [2](#0-1) 

`get_starknet_version` returns `None` for any block number at or beyond the header marker (i.e., any block not yet committed to storage): [3](#0-2) 

So the `.unwrap_or(StarknetVersion::LATEST)` fallback always fires for the pending case. The `starknet_version` field that the feeder gateway provides in `pending_data.block.starknet_version()` is available in the client-side `PendingBlockOrDeprecated`: [4](#0-3) 

But it is **never transferred** into `ExecutionPendingData` during conversion: [5](#0-4) 

The `ExecutionPendingData` struct has no `starknet_version` field, so `create_block_context` has no way to use the pending block's declared version and always falls back to `LATEST`.

For `block_id=Latest`, `maybe_pending_data` is `None`, so `block_number` comes from the stored header and `get_starknet_version` returns the actual stored version — both gas prices and `starknet_version` are consistent. [6](#0-5) 

The resolved `starknet_version` is then used to select `VersionedConstants`: [7](#0-6) 

Different `VersionedConstants` encode different step costs, builtin costs, gas cost tables, and OS constants, all of which directly affect fee computation.

The `estimate_fee` RPC handler in `api_impl.rs` sets `maybe_pending_data = Some(...)` only for `block_id=Pending` and `None` otherwise, feeding directly into `create_block_context`: [8](#0-7) 

---

### Impact Explanation

Any unprivileged RPC client calling `starknet_estimateFee` (or `starknet_simulateTransactions`, `starknet_estimateMessageFee`) with `block_id=Pending` receives a fee estimate computed under `StarknetVersion::LATEST` versioned constants, while the same call with `block_id=Latest` uses the actual stored version's constants. When the node is running on a version where the latest committed block carries an older `starknet_version` (e.g., during a protocol upgrade transition), the two estimates diverge for identical transactions and identical gas prices. This is an authoritative-looking wrong value from the RPC fee estimation surface.

---

### Likelihood Explanation

The bug is latent and fires automatically whenever the latest committed block's `starknet_version` differs from `StarknetVersion::LATEST`. This is a normal condition during any protocol version upgrade window. No special attacker action is required beyond calling the public RPC endpoint with `block_id=Pending`.

---

### Recommendation

Add a `starknet_version: StarknetVersion` field to `ExecutionPendingData` in `crates/apollo_rpc_execution/src/objects.rs`. Populate it from `client_pending_data.block.starknet_version()` (parsed via `StarknetVersion::try_from` or falling back to `LATEST`) in `client_pending_data_to_execution_pending_data` (`crates/apollo_rpc/src/pending.rs`). In `create_block_context`, use `pending_data.starknet_version` directly instead of querying storage when `maybe_pending_data` is `Some`, eliminating the unconditional `LATEST` fallback.

---

### Proof of Concept

1. Write block N to storage with `starknet_version = V0_13_6`.
2. Set `pending_data` with the same gas prices as block N but `starknet_version = "0.13.6"` in the pending block header.
3. Call `starknet_estimateFee` with `block_id=Latest` → resolves `starknet_version = V0_13_6`, uses `VersionedConstants::get(V0_13_6)`.
4. Call `starknet_estimateFee` with `block_id=Pending` → `get_starknet_version(N+1)` returns `None`, falls back to `StarknetVersion::LATEST` (e.g., `V0_14_4`), uses `VersionedConstants::get(V0_14_4)`.
5. Assert the two fee estimates differ despite identical gas prices — the difference is entirely due to the `VersionedConstants` mismatch.
