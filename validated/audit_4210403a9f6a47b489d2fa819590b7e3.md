### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Rewards via Unenforced `farmer_reward_address_override` - (File: chia/farmer/farmer_api.py)

### Summary
A harvester connected to a farmer can set `farmer_reward_address_override` in its `NewProofOfSpace` or `RespondSignatures` protocol messages to redirect the farmer's XCH block reward to any arbitrary address. The farmer performs no enforcement — it only logs a warning — and unconditionally uses the harvester-supplied address in the `DeclareProofOfSpace` message sent to the full node.

### Finding Description

CHIP-22 defines a fee convention allowing third-party harvesters to take a portion of the farmer reward. The farmer is supposed to validate the fee quality against the harvester-reported threshold. However, the enforcement is entirely absent: the farmer only logs a warning when the threshold is violated and proceeds to use the overridden address regardless.

**Step 1 — Harvester sends `NewProofOfSpace` with `farmer_reward_address_override` set:**

In `chia/farmer/farmer_api.py`, `new_proof_of_space()` checks for the override and calls `notify_farmer_reward_taken_by_harvester_as_fee()`, which only logs: [1](#0-0) 

**Step 2 — `notify_farmer_reward_taken_by_harvester_as_fee` only logs, never blocks:**

Even when `fee_quality > fee_threshold` (harvester is taking more than agreed), the function logs a warning and returns normally — no exception, no rejection: [2](#0-1) 

**Step 3 — `_process_respond_signatures` unconditionally accepts the override from `RespondSignatures`:**

The harvester's `RespondSignatures` also carries `farmer_reward_address_override`. The farmer blindly substitutes it as the `farmer_reward_address` in `DeclareProofOfSpace`: [3](#0-2) 

**Step 4 — The override field is part of the harvester protocol with no signing constraint:**

`RespondSignatures.farmer_reward_address_override` is a plain `bytes32 | None` field with no cryptographic binding to the farmer's key or any agreed fee rate: [4](#0-3) 

### Impact Explanation

A malicious or compromised harvester can set `farmer_reward_address_override` to its own address on every winning proof, redirecting 100% of the farmer's XCH block rewards. The farmer has no mechanism to cap, reject, or enforce the fee threshold — only advisory logging. This constitutes unauthorized reward diversion of XCH.

### Likelihood Explanation

Any harvester connected to the farmer via the private CA SSL channel can exploit this. In the third-party harvester model (explicitly supported by CHIP-22), the farmer deliberately connects to external harvester services and grants them private CA certificates. A malicious third-party harvester service can abuse this trust to steal all block rewards, not just the agreed fee. The farmer operator has no on-chain or protocol-level protection.

### Recommendation

The farmer must enforce the fee quality check and **reject** (not merely log) any `farmer_reward_address_override` where `fee_quality > fee_threshold`. Specifically, in `notify_farmer_reward_taken_by_harvester_as_fee`, when the threshold is violated, the function should return a sentinel value or raise an exception that causes `new_proof_of_space` and `_process_respond_signatures` to discard the override and fall back to `self.farmer.farmer_target`. Additionally, `fee_info` should be required (not optional) when `farmer_reward_address_override` is set, so a harvester cannot omit the threshold entirely.

### Proof of Concept

1. Operator sets up a farmer and connects a third-party harvester (granting it a private CA certificate).
2. Malicious harvester, upon finding a winning proof, sends `NewProofOfSpace` with:
   - `farmer_reward_address_override = <attacker_puzzle_hash>`
   - `fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=0xFFFFFFFF)` (or `fee_info = None`)
3. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, logs a warning, and continues.
4. Harvester returns `RespondSignatures` with `farmer_reward_address_override = <attacker_puzzle_hash>`.
5. Farmer's `_process_respond_signatures` sets `farmer_reward_address = attacker_puzzle_hash` and sends `DeclareProofOfSpace` to the full node with the attacker's address as the farmer reward recipient.
6. The block is won and the farmer reward (currently 0.25 XCH) is paid to the attacker's address, not the farmer's configured `xch_target_address`. [5](#0-4) [6](#0-5) [7](#0-6)
