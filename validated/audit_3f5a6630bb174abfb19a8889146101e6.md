### Title
Missing Minimum Asset Output Protection in `instantWithdrawal` Allows Users to Burn rsETH for Fewer Assets Than Expected - (File: `contracts/LRTWithdrawalManager.sol`)

### Summary
`LRTWithdrawalManager.instantWithdrawal` lacks a minimum asset output parameter, directly analogous to the UMA vulnerability where liquidators could not specify a minimum collateral-to-token ratio. Users irreversibly burn rsETH without being able to specify a minimum asset amount they expect to receive, exposing them to oracle price changes between transaction submission and execution. The deposit path already implements this protection (`minRSETHAmountExpected`), but the withdrawal path does not.

### Finding Description
In `instantWithdrawalManager.instantWithdrawal`, the function signature accepts only `asset`, `rsETHUnstaked`, and `referralId`. There is no `minAssetAmount` parameter. [1](#0-0) 

The asset amount is computed at execution time using live oracle prices: [2](#0-1) 

via `getExpectedAssetAmount`: [3](#0-2) 

The rsETH burn is executed **before** any check that the user is satisfied with the output amount: [4](#0-3) 

By contrast, `LRTDepositPool.depositETH` and `depositAsset` both accept a `minRSETHAmountExpected` parameter and revert if the minted amount falls below it: [5](#0-4) [6](#0-5) 

This asymmetry means users are protected from oracle price slippage when depositing but completely unprotected when burning rsETH via instant withdrawal.

### Impact Explanation
If the rsETH/asset oracle price ratio changes between when the user submits the transaction and when it executes, the user's rsETH is burned irreversibly and they receive fewer assets than they anticipated. There is no mechanism to revert or recover. The user cannot set a floor on the asset amount they are willing to accept.

**Impact: Low** — Contract fails to deliver the promised return the user expected at submission time, though the user receives the oracle-fair value at execution time.

### Likelihood Explanation
The rsETH price (`lrtOracle.rsETHPrice()`) and asset price (`lrtOracle.getAssetPrice(asset)`) are both live oracle reads at execution time. Oracle prices update regularly, especially during volatile market conditions (e.g., LST depeg events, EigenLayer slashing, stETH rebases). A user who previews the expected output off-chain and then submits `instantWithdrawal` can receive materially fewer assets if the oracle updates in the same or a subsequent block before their transaction is included. No front-running by an adversary is required — natural oracle drift is sufficient.

### Recommendation
Add a `minAssetAmount` parameter to `instantWithdrawal` and revert if `assetAmountUnlocked` (before fee deduction) falls below it, consistent with the slippage protection already present in `LRTDepositPool`:

```solidity
function instantWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
    uint256 minAssetAmount,   // <-- add this
    string calldata referralId
) external ... {
    ...
    uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
    if (assetAmountUnlocked < minAssetAmount) revert SlippageExceeded(); // <-- add this
    IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
    ...
}
```

### Proof of Concept

1. rsETH oracle price = 1.05 ETH/rsETH; ETH asset price = 1 ETH/ETH.
2. User previews `instantWithdrawal(ETH, 1e18, "")` off-chain → expects ≈1.05 ETH.
3. User submits transaction to mempool.
4. Oracle updates rsETH price to 1.00 ETH/rsETH (legitimate update, e.g., after a slashing event).
5. User's transaction executes: `getExpectedAssetAmount` returns 1.00 ETH.
6. `burnFrom` burns 1e18 rsETH irreversibly.
7. User receives 1.00 ETH (minus fee) instead of the expected 1.05 ETH.
8. No recourse exists — the burn is complete and no minimum was enforced.

This is the direct analog of the UMA vulnerability: just as a liquidator burned synthetic tokens without a minimum collateral-to-token ratio and received insufficient compensation, an rsETH holder burns rsETH via `instantWithdrawal` without a minimum asset output and receives fewer assets than expected, with no ability to abort.     /// @param asset The address of the asset (ETH/LST) to withdraw
    /// @param rsETHUnstaked The amount of rsETH tokens to burn for withdrawal
    /// @param referralId The referral identifier for tracking
    /// @dev Uses the fee set at execution time. Managers can raise it right before this call, making withdrawals cost
    /// more than expected.
    function instantWithdrawal(
        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
        onlyInstantWithdrawalAllowed(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }
        if (IERC20(lrtConfig.rsETH()).balanceOf(msg.sender) < rsETHUnstaked) revert NotEnoughRsETH();
        uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
        IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
        if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
            revert CantInstantWithdrawMoreThanAvailable();
        }

        unstakingVault.redeem(asset, assetAmountUnlocked);

        uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
        uint256 userAmount = assetAmountUnlocked - fee;

        address feeRecipient = instantWithdrawalFeeRecipient;
        if (feeRecipient == address(0)) {
            // Backwards-compatible default: send fees to the protocol treasury
            feeRecipient = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        }
        if (fee > 0) {
            _transferAsset(asset, feeRecipient, fee);
            emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
        }

        _transferAsset(asset, msg.sender, userAmount);
        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(msg.sender, asset, rsETHUnstaked, userAmount);
    }

    function getExpectedAssetAmount(
        address asset,
        uint256 amount
    )
        public
        view
        override
        returns (uint256 underlyingToReceive)
    {
        // setup oracle contract
        ILRTOracle lrtOracle = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE));

        // calculate underlying asset amount to receive based on rsETH amount and asset exchange rate
        underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
    }
