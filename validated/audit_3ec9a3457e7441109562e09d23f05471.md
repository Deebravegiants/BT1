### Title
Hardcoded `Fee(1)` Placeholder in Consensus L1 Handler Conversion Produces Wrong Executable Payload and Proposer/Validator Divergence - (File: crates/apollo_transaction_converter/src/transaction_converter.rs)

### Summary

`TransactionConverter::convert_consensus_l1_handler_to_internal_l1_handler` unconditionally injects `Fee(1)` as `paid_fee_on_l1` for every L1 handler transaction received through the consensus wire, instead of using the actual fee paid on L1. The actual fee is never transmitted in the consensus protobuf message, so the validator always reconstructs the executable transaction with a fabricated fee. This is the direct Sequencer analog of the "unsafe transfer" pattern: a value that must be read from an authoritative source is instead substituted with a hardcoded placeholder, bypassing the real check.

### Finding Description

In `convert_consensus_l1_handler_to_internal_l1_handler`:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

The consensus protobuf serialization of `L1HandlerTransaction` omits `paid_fee_on_l1` entirely:

```rust
impl From<L1HandlerTransaction> for protobuf::L1HandlerV0 {
    fn from(value: L1HandlerTransaction) -> Self {
        Self {
            nonce: Some(value.nonce.0.into()),
            address: Some(value.contract_address.into()),
            entry_point_selector: Some(value.entry_point_selector.0.into()),
            calldata: value.calldata.0.iter().map(|calldata| (*calldata).into()).collect(),
        }
    }
}
``` [2](#0-1) 

The blockifier's L1 handler execution path checks `paid_fee_on_l1` directly:

```rust
let paid_fee = self.paid_fee_on_l1;
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
``` [3](#0-2) 

The proposer fetches L1 handler transactions from the L1 events provider with the real `paid_fee_on_l1`. The validator receives only the raw `L1HandlerTransaction` (no fee field) and reconstructs the executable transaction with `Fee(1)`. If the actual `paid_fee_on_l1` on an L1 event is `Fee(0)`:

- **Proposer**: `execute_raw` returns `Err(InsufficientFee)` → transaction is rejected, state changes are committed but execution fails.
- **Validator**: `execute_raw` returns `Ok(...)` because `Fee(1) != Fee(0)` → transaction is accepted, state changes committed, execution succeeds.

This produces divergent `execution_infos_and_signatures`, divergent `ThinStateDiff`, divergent `PartialBlockHashComponents`, and ultimately a different `ProposalCommitment` on the validator side, causing `ProposalFinMismatch` and consensus failure. [4](#0-3) 

### Impact Explanation

The validator binds the wrong executable payload (`paid_fee_on_l1 = Fee(1)`) to every L1 handler transaction received through consensus. When the actual L1 fee is zero, the proposer and validator produce different execution results, different state diffs, and different block commitments. This matches: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."** If the divergence causes the validator to accept a transaction the proposer rejected, it also matches: **"Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."**

### Likelihood Explanation

The trigger requires an L1 event with `paid_fee_on_l1 = 0` to reach the sequencer. Whether the Starknet core contract on L1 enforces a non-zero fee is outside this codebase's scope, but the Sequencer itself has no guard against it. The TODO comment confirms this is a known unresolved placeholder, not a deliberate design choice. Any L1 event that reaches the sequencer with a zero fee would trigger the divergence.

### Recommendation

1. Add `paid_fee_on_l1` to the consensus protobuf `L1HandlerV0` message so the actual fee is transmitted over the wire.
2. Replace `Fee(1)` in `convert_consensus_l1_handler_to_internal_l1_handler` with the deserialized value from the wire message.
3. Until the wire format is updated, add a validator-side guard that rejects any L1 handler transaction whose reconstructed `paid_fee_on_l1` would differ from the proposer's intent (e.g., by requiring the L1 events provider to confirm the fee before execution).

### Proof of Concept

1. An L1 event is emitted on Ethereum with `paid_fee_on_l1 = 0` and reaches the sequencer's L1 events provider.
2. The proposer includes the L1 handler transaction in a block. `execute_raw` is called with `paid_fee_on_l1 = Fee(0)`. After execution succeeds, the fee check fires: `return Err(InsufficientFee)`. The transaction is rejected; its state changes are committed but the execution info records a failure.
3. The proposer streams the transaction to validators via `ProposalPart::Transactions`. The protobuf `L1HandlerV0` carries no `paid_fee_on_l1` field.
4. The validator calls `convert_consensus_l1_handler_to_internal_l1_handler`, injecting `Fee(1)`. `execute_raw` is called with `paid_fee_on_l1 = Fee(1)`. The fee check passes. The transaction is accepted with a successful execution info.
5. The validator's `BlockExecutionArtifacts` contains a different `execution_infos_and_signatures` map, a different `ThinStateDiff`, and different `PartialBlockHashComponents` than the proposer's.
6. `PartialBlockHash::from_partial_block_hash_components` produces a different hash on the validator.
7. `validate_proposal` compares `built_block` (validator's hash) against `received_fin.proposal_commitment` (proposer's hash): they differ, `ProposalFinMismatch` is returned, and consensus cannot reach a decision on this block. [5](#0-4) [6](#0-5)
