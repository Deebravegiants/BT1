Audit Report

## Title
Fee Calculation Integer Truncation to Zero Allows Fee-Free Deposits for Small Amounts — (File: contracts/pools/RSETHPoolV3.sol)

## Summary
In `RSETHPoolV3.viewSwapRsETHAmountAndFee`, the fee is computed as `fee = amount * feeBps / 10_000`. Solidity integer division truncates to zero whenever `amount * feeBps < 10_000`, causing the protocol to collect zero fee and mint rsETH on the full deposit amount. This pattern is replicated verbatim across all L2 pool variants.

## Finding Description
At `RSETHPoolV3.sol` line 300, the fee calculation is:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

When `amount * feeBps < 10_000`, integer division yields `fee = 0`. `amountAfterFee` then equals the full `amount`, and rsETH is minted on the full deposit with no fee deducted. The `feeEarnedInETH` accumulator is incremented by zero. The only guard in `deposit()` is `if (amount == 0) revert InvalidAmount()` (line 256), which does not prevent deposits of 1–(10_000/feeBps − 1) wei. Concrete thresholds: `feeBps = 10` → any `amount < 1000 wei`; `feeBps = 100` → any `amount < 100 wei`; `feeBps = 1000` → any `amount < 10 wei`. The same truncating formula appears identically in `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`, and `RSETHPoolV2NBA`.

## Impact Explanation
The protocol's fee-collection mechanism silently fails for any deposit below the truncation threshold: zero fee is recorded, and rsETH is minted as if no fee applies. This matches the allowed impact **Low — Contract fails to deliver promised returns, but doesn't lose value**: existing rsETH holders are not materially harmed, but the protocol foregoes its promised fee revenue on these deposits.

## Likelihood Explanation
Low. Any unprivileged depositor can trigger this by sending a deposit of 1 to `(10_000/feeBps − 1)` wei. However, on L2 networks (Arbitrum, Base, Optimism, Scroll, Linea, Unichain), transaction gas costs far exceed the fee savings achievable at these amounts, making rational exploitation economically irrational. The structural bug is always present and reachable, but mass exploitation is self-defeating.

## Recommendation
Replace the truncating division with a ceiling fee calculation to ensure at least 1 wei of fee is charged whenever `feeBps > 0` and `amount > 0`:

```solidity
fee = (amount * feeBps + 9_999) / 10_000;
```

Alternatively, enforce a minimum deposit amount in all L2 pool contracts analogous to `minAmountToDeposit` in `LRTDepositPool` (line 657), guaranteeing `amount * feeBps >= 10_000` for any non-zero `feeBps`.

## Proof of Concept
1. Deploy `RSETHPoolV3` with `feeBps = 100` (1%).
2. Call `deposit{value: 99}("")` — passes the `amount == 0` guard at line 256.
3. `viewSwapRsETHAmountAndFee(99)` computes `fee = 99 * 100 / 10_000 = 0` (integer truncation).
4. `amountAfterFee = 99`; `rsETHAmount = 99 * 1e18 / rsETHToETHrate`.
5. `feeEarnedInETH += 0` — protocol collects no fee.
6. User receives rsETH equivalent to the full 99 wei deposit with no fee deducted.
7. Repeat for any `amount ∈ [1, 99]` — all produce `fee = 0`.

Foundry fuzz test sketch:
```solidity
function testFuzz_feeZeroTruncation(uint256 amount) public {
    vm.assume(amount > 0 && amount < 100); // feeBps = 100
    (uint256 rsETHAmount, uint256 fee) = pool.viewSwapRsETHAmountAndFee(amount);
    assertEq(fee, 0); // always zero in this range
}
```
