### Title
Missing State Diff Hash Verification Allows Malicious P2P Peer to Inject Wrong Contract Nonce via `unite_state_diffs` Split-Chunk False Negative — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

The p2p sync client assembles `ThinStateDiff` from peer-supplied `StateDiffChunk` messages but **never verifies the assembled result against the `state_diff_commitment` stored in the block header**. The `unite_state_diffs` function also has a false negative: when two `ContractDiff` chunks arrive for the same address and only the *second* carries a nonce, no conflict is raised, so an attacker-chosen nonce is silently accepted. The combined effect lets an unauthenticated p2p peer permanently store a wrong contract nonce in the node's state.

---

### Finding Description

**Step 1 — `unite_state_diffs` nonce conflict check is bypassable.** [1](#0-0) 

The guard `if let Some(nonce) = contract_diff.nonce` is skipped entirely when `nonce = None`. A subsequent chunk for the same address with `nonce = Some(WRONG_N)` calls `state_diff.nonces.insert(addr, WRONG_N)`, which returns `None` (no prior entry), so `ConflictingStateDiffParts` is never returned.

**Step 2 — `StateDiffChunk::len()` counts nonce only when `Some`.** [2](#0-1) 

Chunk 1 `{addr=A, nonce=None, storage={k:v}}` → `len = 1`.  
Chunk 2 `{addr=A, nonce=Some(WRONG_N)}` → `len = 1`.  
Total = 2 = `target_state_diff_len`. The length guard at line 99 passes.

**Step 3 — No state diff hash verification exists anywhere in the p2p sync client.** [3](#0-2) 

After the loop, `parse_data_for_block` calls only `validate_deprecated_declared_classes_non_conflicting` and returns. There is **zero** call to `calculate_state_diff_hash` anywhere under `crates/apollo_p2p_sync/` — confirmed by grep. The block header carries a `state_diff_commitment` (Poseidon hash) that is never compared against the assembled diff. [4](#0-3) 

**Step 4 — The corrupted diff is written directly to storage.** [5](#0-4) 

`append_state_diff` persists the nonce into the `nonces` table with no further validation. [6](#0-5) 

**Step 5 — The legitimate server never splits a single contract across two chunks.** [7](#0-6) 

`split_thin_state_diff` always emits one `ContractDiff` per address with all fields populated. Receiving two chunks for the same address is therefore an attacker-only pattern that the client does not reject.

---

### Impact Explanation

A syncing node stores a wrong nonce for a contract. Downstream effects:

- **RPC `get_nonce`** returns the attacker-chosen value — an authoritative-looking wrong answer.
- **Mempool / gateway admission** uses the stored nonce for replay-protection checks; a wrong nonce causes valid transactions to be rejected or invalid (replayed) transactions to be accepted.
- **Class sync** reads the stored state diff to determine which classes to fetch; a corrupted diff can cause class sync to skip or mis-fetch classes.

This maps to **High** impact under the scope rules ("RPC execution … returns an authoritative-looking wrong value"; "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing").

---

### Likelihood Explanation

Any unauthenticated peer on the p2p network can serve state diff responses. The attack requires crafting two `ContractDiff` protobuf messages for the same address — trivial with any p2p client. No cryptographic material, keys, or privileged access are needed. The only precondition is that the victim node is in the p2p sync path (i.e., not receiving the block internally).

---

### Recommendation

1. **Verify the assembled state diff against the header commitment** before storing it:

```rust
let commitment = calculate_state_diff_hash(&result);
if commitment != header.state_diff_commitment.unwrap() {
    return Err(ParseDataError::BadPeer(BadPeerError::WrongStateDiffHash { ... }));
}
```

2. **Reject duplicate contract addresses** in `unite_state_diffs` regardless of which fields are `Some`/`None` — a second `ContractDiff` for an already-seen address should always be an error, matching the server's invariant.
