### Title
Missing Deadline Check in `MetricOmmPoolLiquidityAdder` Allows Stale Liquidity Transactions to Execute at Unfavorable Pool States — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

### Summary

`MetricOmmSimpleRouter` correctly enforces a `deadline` on every swap entry point via `_checkDeadline`. The sibling periphery contract `MetricOmmPoolLiquidityAdder` exposes four public liquidity entry points — two overloads of `addLiquidityExactShares` and two overloads of `addLiquidityWeighted` — none of which accept or enforce a deadline. A pending liquidity transaction can therefore sit in the mempool and execute arbitrarily late, at a pool cursor position the LP never intended to target.

### Finding Description

`MetricOmmSwapRouterBase._checkDeadline` is called at the top of every swap function in the router: [1](#0-0) 

All four router entry points invoke it before touching the pool: [2](#0-1) [3](#0-2) 

None of the four `MetricOmmPoolLiquidityAdder` entry points contain any equivalent guard: [4](#0-3) [5](#0-4) 

`addLiquidityWeighted` does validate cursor bounds via `_validateBinAndBinPosition`, which reverts if the pool cursor has moved outside the LP's stated `[minimalCurBin, maximalCurBin]` window: [6](#0-5) 

However, this is not equivalent to a deadline:

- The cursor can drift significantly **within** the stated bounds while the transaction is pending. The probe then re-determines composition at the new (stale-to-the-LP) cursor position, and the LP receives a token split they did not intend.
- `addLiquidityExactShares` has **no cursor-bounds check at all**. The LP specifies exact per-bin shares; the token amounts owed are computed from the live pool state at execution time. A delayed transaction executes against whatever cursor position the pool has reached, with no revert guard.
- `maxAmountToken0` / `maxAmountToken1` caps prevent the LP from paying *more* than their stated maximums, but they do not prevent the LP from receiving a **worse share-to-token ratio** or from having their position placed at an unintended price range.
- Without a deadline, MEV searchers can observe the pending transaction, manipulate the pool cursor (via a swap) to a position that is still within the LP's cursor bounds but maximally unfavorable for the LP's intended composition, let the liquidity transaction execute, then reverse the cursor manipulation — a classic sandwich on a liquidity operation.

### Impact Explanation

An LP submitting `addLiquidityWeighted` or `addLiquidityExactShares` through `MetricOmmPoolLiquidityAdder` can have their transaction executed at a pool state materially different from the one they observed at submission time. The resulting position may be concentrated at a price range the LP did not intend, increasing their impermanent-loss exposure and reducing the value of their LP claim relative to what they would have received had the transaction been included promptly. MEV bots can amplify this by sandwiching the liquidity addition within the cursor-bounds window. The `maxAmount` caps bound the token outflow but do not bound the quality of the position received.

### Likelihood Explanation

Any liquidity transaction that spends time in the mempool — due to low gas, network congestion, or deliberate MEV delay — is affected. The `addLiquidityWeighted` path is particularly exposed because it is the recommended high-level entry point for proportional deposits, and its cursor-bounds window is typically wide enough to allow meaningful cursor drift. No special permissions or unusual conditions are required; any unprivileged LP using the standard periphery is at risk.

### Recommendation

Add a `uint256 deadline` parameter to all four public entry points in `MetricOmmPoolLiquidityAdder` and revert at the top of each function if `block.timestamp > deadline`, mirroring the pattern already established in `MetricOmmSwapRouterBase._checkDeadline`:

```diff
 function addLiquidityExactShares(
     address pool,
     address owner,
     uint80 salt,
     LiquidityDelta calldata deltas,
     uint256 maxAmountToken0,
     uint256 maxAmountToken1,
+    uint256 deadline,
     bytes calldata extensionData
 ) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
+    if (block.timestamp > deadline) revert TransactionExpired(deadline, block.timestamp);
     _validateOwner(owner);
     ...
 }
```

Apply the same change to the `msg.sender`-owner overload and both `addLiquidityWeighted` overloads. Update `IMetricOmmPoolLiquidityAdder` accordingly.

### Proof of Concept

1. LP calls `addLiquidityWeighted` with cursor bounds `[bin 0, bin 1]`, `maxAmountToken0 = X`, `maxAmountToken1 = Y`, targeting the current cursor at bin 0 / position 30 %.
2. Transaction sits in the mempool (low gas price or deliberate MEV delay).
3. A MEV bot swaps the pool, moving the cursor to bin 1 / position 70 % — still within the LP's stated bounds, so `_validateBinAndBinPosition` will not revert.
4. The LP's transaction executes. The probe now returns a composition heavily weighted toward token1 (the pool's current deficit side). Shares are scaled accordingly.
5. The LP's position is now concentrated at a price range they did not intend, with a token split they did not choose.
6. The MEV bot reverses its swap, profiting from the spread while the LP is left with a suboptimal position.

The same scenario applies to `addLiquidityExactShares` without even the cursor-bounds guard: the LP's exact shares are minted against whatever pool state exists at execution time, with no revert path if the cursor has moved far from the LP's intended target.

  function _validateBinAndBinPosition(
    address pool,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition
  ) internal view {
    if (minimalCurBin > maximalCurBin) {
      revert CursorOutOfBounds(0, 0, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }

    (, int8 curBinIdx, uint104 curPosInBin,,,) = PoolStateLibrary._slot0(pool);

    int256 curBin = curBinIdx;
    if (curBin < minimalCurBin || curBin > maximalCurBin) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == minimalCurBin && curPosInBin < minimalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == maximalCurBin && curPosInBin > maximalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
  }
