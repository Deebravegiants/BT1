### Title
`max_l2_gas_amount` Cap Not Enforced for `Declare` Transactions in Gateway Stateless Validator - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidatorConfig` defines a `max_l2_gas_amount` cap that is enforced for `Invoke` and `DeployAccount` transactions but is explicitly skipped for `Declare` transactions. A `Declare` transaction with an arbitrarily large `l2_gas.max_amount` (e.g., `u64::MAX`) passes all gateway admission checks and enters the mempool, violating the gateway's own admission policy.

### Finding Description

`StatelessTransactionValidatorConfig` declares `max_l2_gas_amount: u64` as a hard cap on the L2 gas amount a transaction may claim. In production this is set to `1,210,000,000`. [1](#0-0) 

The enforcement in `validate_resource_bounds` contains an explicit empty branch that silently skips the check for `Declare` transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [2](#0-1) 

The gap is intentionally tested and confirmed to pass: [3](#0-2) 

The stateful validator only checks the L2 gas *price*, not the gas *amount*, so it provides no compensating guard: [4](#0-3) 

### Impact Explanation

A `Declare` transaction with `l2_gas.max_amount = u64::MAX` (or any value above `max_l2_gas_amount`) passes both the stateless and stateful gateway validators and is admitted to the mempool. The blockifier's pre-validation only checks `actual_gas >= max_amount` (lower bound), not an upper bound, so execution proceeds normally. The fee charged is based on actual gas used, not the declared maximum, so the transaction produces correct state and receipts — but the gateway's admission invariant is broken: a transaction class that should be rejected is accepted.

**Impact category:** High — Mempool/gateway admission accepts invalid transactions before sequencing.

### Likelihood Explanation

Any unprivileged user can submit a `Declare` transaction with `l2_gas.max_amount` set above the configured cap. No special access, timing, or network position is required. The code path is unconditional and the bypass is confirmed by an existing test case.

### Recommendation

Remove the empty `Declare` branch and apply the same `max_l2_gas_amount` check uniformly to all transaction types:

```rust
fn validate_resource_bounds(&self, tx: &RpcTransaction) -> ... {
    // ...
    if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
        return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
            gas_amount: resource_bounds.l2_gas.max_amount,
            max_gas_amount: self.config.max_l2_gas_amount,
        });
    }
    Ok(())
}
```

If `Declare` transactions legitimately require a higher gas amount ceiling, introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.

### Proof of Concept

1. Construct a V3 `Declare` transaction with:
   - `l2_gas.max_amount = GasAmount(u64::MAX)` (far above the production cap of `1,210,000,000`)
   - `l2_gas.max_price_per_unit` at or above `min_gas_price` (e.g., `8_000_000_000`)
2. Submit to the gateway's `add_transaction` endpoint.
3. Observe: `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for `Declare`.
4. The transaction enters the mempool and is eligible for inclusion in a block proposal, bypassing the `max_l2_gas_amount` admission policy that applies to all other transaction types.

The existing test `valid_l2_gas_amount_on_declare` in `crates/apollo_gateway/src/stateless_transaction_validator_test.rs` (lines 173–201) already documents and confirms this behavior. [3](#0-2)
