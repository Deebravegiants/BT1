Audit Report

## Title
Immutable `i_maxSupply` in `WrappedRSETH` permanently blocks CCIP yield delivery when cap is saturated — (`contracts/ccip/WrappedRSETH.sol`)

## Summary

`WrappedRSETH` stores its supply ceiling in an `immutable` variable `i_maxSupply` with no owner-callable function to raise it after deployment. Once `totalSupply()` reaches the cap through legitimate CCIP bridge usage, every subsequent CCIP yield-delivery mint reverts with `MaxSupplyExceeded`, and because the cap cannot be raised, the failure is permanent. L2 `WrappedRSETH` holders permanently stop receiving their share of accrued EigenLayer/Lido restaking yield.

## Finding Description

`i_maxSupply` is declared `immutable` and set once in the constructor with no mutator:

```solidity
// contracts/ccip/WrappedRSETH.sol  line 41, 54
uint256 internal immutable i_maxSupply;
...
i_maxSupply = maxSupply_;
```

`mint` enforces the ceiling on every call:

```solidity
// contracts/ccip/WrappedRSETH.sol  line 138
if (i_maxSupply != 0 && totalSupply() + amount > i_maxSupply)
    revert MaxSupplyExceeded(totalSupply() + amount);
```

No `setMaxSupply`, `increaseMaxSupply`, or equivalent owner function exists anywhere in the contract. The entire Roles section (lines 143–195) only manages minter/burner addresses.

Exploit flow:
1. `WrappedRSETH` is deployed with a finite `maxSupply_` (non-zero), an explicitly supported deployment path advertised in the NatDoc on line 19.
2. Users legitimately bridge rsETH from L1 to L2 via CCIP; each successful bridge call invokes `WrappedRSETH.mint` through the CCIP token pool (the permissioned minter), incrementing `totalSupply()`.
3. Once `totalSupply() == i_maxSupply`, the MANAGER calls `L1VaultV2.bridgeRsETHToL2UsingCCIP` to deliver accrued yield. The CCIP router locks/burns rsETH on L1 and dispatches the cross-chain message.
4. On L2, the CCIP token pool calls `WrappedRSETH.mint`; the check at line 138 reverts with `MaxSupplyExceeded`.
5. The CCIP message enters a failed state. Manual-execution retries also revert for the same reason. Because `i_maxSupply` cannot be raised, every retry permanently fails.
6. There is no admin escape hatch: the owner cannot raise the cap, cannot redirect the yield, and cannot unblock the failed message.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

rsETH is a yield-bearing token; its exchange rate increases continuously as EigenLayer/Lido restaking rewards accrue. `L1VaultV2` periodically bridges additional rsETH to L2 to reflect that yield. Once the cap is saturated, none of those yield-delivery bridge calls can complete. L2 `WrappedRSETH` holders permanently stop receiving their proportional share of accrued yield with no recovery path available in the contract.

## Likelihood Explanation

The precondition is deployment with `maxSupply_ != 0`, which the contract explicitly supports and documents. Once deployed with a finite cap, the supply grows organically through normal user bridge activity (each user bridge call mints `WrappedRSETH` via the CCIP token pool). No privileged access or attacker action is required to saturate the cap — it fills through legitimate protocol usage. The condition is therefore reachable by any user who bridges rsETH to L2, and once reached, the yield freeze is irreversible.

## Recommendation

1. Replace the `immutable` field with a mutable storage variable and add an `onlyOwner` function `setMaxSupply(uint256 newCap)` that enforces `newCap >= totalSupply()` and `newCap >= i_maxSupply` (cap can only increase, never decrease).
2. Alternatively, deploy with `maxSupply_ = 0` (unlimited) and enforce supply limits at the pool or bridge layer rather than in the token contract.
3. Add a CCIP message-recovery path (e.g., a fallback that unlocks rsETH on L1 if the L2 mint permanently fails) so that in-flight funds are not stranded.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/ccip/WrappedRSETH.sol";

contract MaxSupplyFreezeTest is Test {
    WrappedRSETH token;
    address ccipPool = address(0xBEEF); // simulates CCIP token pool (permissioned minter)
    address user     = address(0xCAFE);

    function setUp() public {
        token = new WrappedRSETH("Wrapped rsETH", "wrsETH", 18, 1000e18, address(this));
        token.grantMintRole(ccipPool);
    }

    function test_yieldMintRevertsAfterCapSaturated() public {
        // Step 1: cap saturated through legitimate user bridging
        vm.prank(ccipPool);
        token.mint(user, 1000e18);          // totalSupply == i_maxSupply

        // Step 2: CCIP yield-delivery bridge tries to mint accrued yield (+1 wei)
        vm.prank(ccipPool);
        vm.expectRevert(
            abi.encodeWithSelector(WrappedRSETH.MaxSupplyExceeded.selector, 1000e18 + 1)
        );
        token.mint(user, 1);                // reverts — yield permanently frozen

        // Step 3: no admin escape hatch — cap cannot be raised
        // token.setMaxSupply(2000e18);     // does not compile; function does not exist
        assertEq(token.maxSupply(), 1000e18); // immutable, unchanged
    }
}
```

The test deploys `WrappedRSETH` with `i_maxSupply = 1000e18`, saturates it via the CCIP token pool (the only permissioned minter), then demonstrates that a single-wei yield mint reverts and cannot be unblocked because no `setMaxSupply` function exists.
