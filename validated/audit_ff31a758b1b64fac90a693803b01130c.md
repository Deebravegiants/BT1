### Title
Sub-Transaction Plutus Script ExUnits Excluded from Fee Calculation and `maxTxExUnits` Enforcement - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

In the Dijkstra era, sub-transactions (`SubTx`) can carry Plutus scripts with redeemers and associated `ExUnits` budgets. The `SUBUTXO` transition rule does not enforce the `maxTxExUnits` protocol parameter against sub-transaction redeemers, and the top-level minimum fee calculation (`minfee`) does not include sub-transaction `ExUnits`. An unprivileged transaction submitter can therefore force validators to execute an unbounded amount of Plutus script work across sub-transactions while paying fees that cover only the top-level transaction's (potentially zero) script execution cost.

---

### Finding Description

**Vulnerability class:** Resource-limit bug / fee accounting bypass.

**Root cause — `totExUnits` ignores sub-transaction redeemers:**

`totExUnits` is defined as:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It sums only the redeemers in the top-level transaction's own witness set. Sub-transactions embedded in `dtbrSubTransactions` each carry their own `TxWits` (including redeemers), but those are never folded into this sum. [1](#0-0) 

**Root cause — `validateExUnitsTooBigUTxO` only checks the top-level tx:**

The Dijkstra UTXO transition calls:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

`validateExUnitsTooBigUTxO` computes `totalExUnits = totExUnits tx`, which, as shown above, excludes sub-transaction redeemers entirely. [2](#0-1) [3](#0-2) 

**Root cause — `SUBUTXO` rule has no ExUnits check at all:**

The entire `dijkstraSubUtxoTransition` function contains no call to `validateExUnitsTooBigUTxO` or any equivalent. The `DijkstraSubUtxoPredFailure` type does not even include an `ExUnitsTooBigUTxO` constructor; the conversion function explicitly marks it as impossible:

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [4](#0-3) [5](#0-4) 

**Root cause — `minfee` excludes sub-transaction script fees:**

The minimum fee for the top-level transaction is:

```
minfee pp tx = a * txSize tx + b + txscriptfee (prices pp) (totExUnits tx)
```

Because `totExUnits tx` ignores sub-transaction redeemers, the `txscriptfee` component of `minfee` does not account for any Plutus execution performed inside sub-transactions. The top-level fee check (`validateFeeTooSmallUTxO`) therefore passes even when sub-transactions carry large ExUnits budgets. [6](#0-5) [7](#0-6) 

**Sub-transactions do carry Plutus scripts:**

The Dijkstra CDDL schema shows `sub_transaction_body` includes `? 11 : script_data_hash`, and the `transaction_witness_set` inside a sub-transaction includes redeemers. The `SUBUTXOW` rule calls `Alonzo.hasExactSetOfRedeemers` and `Alonzo.checkScriptIntegrityHash` for sub-transactions, confirming they are first-class Plutus script carriers. [8](#0-7) [9](#0-8) 

The `DijkstraStAnnTx SubTx` type carries `dsastPlutusScriptsWithContext`, confirming sub-transactions have Plutus scripts that are collected and executed during validation. [10](#0-9) 

---

### Impact Explanation

An attacker can submit a single top-level transaction with zero or minimal top-level ExUnits (passing `maxTxExUnits`) and embed N sub-transactions each carrying ExUnits approaching `maxTxExUnits`. The combined Plutus execution work imposed on validators is N × `maxTxExUnits`, but the fee charged covers only the top-level tx's (near-zero) script cost. This constitutes:

1. **Fee manipulation outside design parameters**: The `txscriptfee` component of `minfee` is systematically under-charged for any transaction that uses Plutus scripts exclusively inside sub-transactions.
2. **Intended validation limit bypass**: The `maxTxExUnits` protocol parameter, designed to bound per-transaction script execution, is bypassed because it is only applied to the top-level transaction's redeemers.

This maps to the **Medium** impact category: *Attacker-controlled transactions exceed intended validation limits or modify fees outside design parameters.*

---

### Likelihood Explanation

The Dijkstra era is production-bound code in this repository. Any unprivileged user who can submit a Dijkstra-era transaction can exploit this. No special role, key, or governance majority is required. The attack requires only the ability to construct a valid top-level transaction with embedded sub-transactions carrying Plutus scripts — a capability explicitly supported by the protocol.

---

### Recommendation

1. **Extend `totExUnits` to include sub-transaction redeemers.** The function should recursively sum ExUnits across all sub-transactions embedded in `dtbrSubTransactions`.

2. **Apply `validateExUnitsTooBigUTxO` to the combined ExUnits total** (top-level + all sub-transactions) in the Dijkstra UTXO rule, or add an equivalent check in `SUBUTXO` per sub-transaction.

3. **Include sub-transaction ExUnits in `minfee`.** The `getMinFeeTx` / `alonzoMinFeeTx` path used for Dijkstra must account for sub-transaction script fees so that the fee charged reflects the actual Plutus execution work imposed on validators.

---

### Proof of Concept

```
1. Construct a Dijkstra top-level transaction T with:
   - No top-level Plutus scripts (totExUnits T = 0)
   - N sub-transactions S₁…Sₙ, each with a Plutus script
     and ExUnits = (maxTxExUnits - 1)

2. Submit T.

3. Ledger validation:
   - validateExUnitsTooBigUTxO pp T  →  0 ≤ maxTxExUnits  ✓ (passes)
   - validateFeeTooSmallUTxO pp T    →  fee ≥ minfee(0 ExUnits)  ✓ (passes cheaply)
   - SUBUTXO for each Sᵢ            →  no ExUnits check  ✓ (passes)
   - Plutus scripts in S₁…Sₙ are executed by validators

4. Result: validators execute N × (maxTxExUnits - 1) ExUnits of Plutus work
   while the attacker pays fees for 0 ExUnits of script execution.
```

validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) $
