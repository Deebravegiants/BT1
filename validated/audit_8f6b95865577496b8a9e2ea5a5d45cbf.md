### Title
Single Attested Participant Can Forge CKD Response for `AppPublicKey` Variant, Delivering Attacker-Controlled Key Material — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC contract applies cryptographic verification of the CKD response **only** for the `AppPublicKeyPV` (publicly verifiable) variant. For the `AppPublicKey` (privately verifiable) variant — the default and most widely documented mode — any single attested participant can submit an entirely arbitrary `CKDResponse` (`big_y`, `big_c`) for any pending request, and the contract will accept and deliver it to the user. A single Byzantine participant, strictly below the signing threshold, can therefore unilaterally determine the CKD output, causing the requester to derive a key that the attacker fully controls and knows.

---

### Finding Description

The `respond_ckd` public method in `crates/contract/src/lib.rs` contains the following branch:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk, app_id), pk)` via `ckd_output_check`, which cryptographically binds the response to the MPC master secret key. [2](#0-1) 

For `AppPublicKey`, the arm is an empty no-op. The contract performs **zero** verification that `big_y` and `big_c` are related to the MPC master secret key, the user's ephemeral public key, or any honest computation. The response is immediately serialised and delivered to the user via `resolve_yields_for`. [3](#0-2) 

The existing unit test for the `AppPublicKey` path explicitly uses byte-array garbage (`[1u8; 48]`, `[2u8; 48]`) as the response and asserts it succeeds, confirming the absence of any check: [4](#0-3) 

The only gate on `respond_ckd` is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a single attested participant — not a threshold-many quorum. [5](#0-4) 

This is the direct analog of the original report's root cause: the access-control mechanism (here, "is an attested participant") grants a single role the ability to exercise a privileged function (`respond_ckd`) without the Principle of Least Privilege being enforced — specifically, without requiring that the response be cryptographically bound to the threshold-held master secret.

---

### Impact Explanation

The CKD protocol is designed so that the user computes:

```
sig = big_c − a · big_y  =  msk · H(pk, app_id)
```

where `a` is the user's ephemeral secret and `msk` is the MPC master secret. The security guarantee is that `sig` is determined by `msk`, which no single node knows.

If an attacker controls `big_y` and `big_c`, they can set:

- `big_y = identity` (the G1 identity point)
- `big_c = attacker_scalar · H(pk, app_id)` for any `attacker_scalar` they choose

The user then computes `sig = attacker_scalar · H(pk, app_id) − a · identity = attacker_scalar · H(pk, app_id)`. The attacker knows `attacker_scalar`, so they know `sig` and therefore the user's derived key `s = HKDF(sig)`. The confidentiality guarantee of the CKD protocol is completely broken for every `AppPublicKey` request answered by the Byzantine participant.

**Impact class**: Critical — unauthorized confidential key derivation output without the required participant authorization (threshold-many honest nodes). The user's derived secret is fully known to the attacker.

---

### Likelihood Explanation

- The `AppPublicKey` variant is the default, legacy, and most widely documented mode (all CLI examples, sandbox tests, and e2e tests use it).
- Only **one** attested participant needs to be Byzantine; no threshold collusion is required.
- The attacker's window is any time a `request_app_private_key` call is pending on-chain (visible in contract state).
- The attacker does not need to race against honest nodes: `resolve_yields_for` delivers the **first** response that arrives for a given request key, so the Byzantine participant simply needs to call `respond_ckd` before the honest coordinator does. [3](#0-2) 

---

### Recommendation

Apply the same pairing-based verification to the `AppPublicKey` variant. Because `AppPublicKey` does not include a G2 component (`pk2`), the current `ckd_output_check` cannot be applied directly. Two options:

1. **Deprecate the unverifiable variant**: Require all callers to use `AppPublicKeyPV`, which provides the G2 component needed for on-chain verification.
2. **Add a weaker binding check**: At minimum, verify that `big_c` and `big_y` are valid G1 points in the prime-order subgroup (preventing trivially malformed responses), and document that full cryptographic binding is only available via `AppPublicKeyPV`.

The long-term fix is option 1, consistent with the design intent stated in the CKD documentation that `AppPublicKeyPV` "allows anyone to verify correctness of the encrypted result on-chain."

### Proof of Concept

The existing unit test already demonstrates the vulnerability without modification:

```rust
// crates/contract/src/lib.rs, test respond_ckd__should_succeed_when_response_is_valid_and_request_exists
let response = CKDResponse {
    big_y: dtos::Bls12381G1PublicKey([1u8; 48]),  // garbage bytes
    big_c: dtos::Bls12381G1PublicKey([2u8; 48]),  // garbage bytes
};
with_active_participant_and_attested_context(&contract);
match contract.respond_ckd(ckd_request.clone(), response.clone()) {
    Ok(_) => { /* succeeds — no verification performed */ }
    Err(_) => panic!("respond_ckd should not fail"),
}
``` [4](#0-3) 

**Exploit steps**:

1. Attacker is one attested participant (e.g., a newly joined node that has submitted a valid TEE attestation).
2. A user submits `request_app_private_key` with `AppPublicKey` variant and ephemeral public key `A = a·G1`.
3. Attacker observes the pending `CKDRequest` in contract state.
4. Attacker calls `respond_ckd(request, CKDResponse { big_y: identity, big_c: attacker_scalar · H(pk, app_id) })` before the honest coordinator responds.
5. Contract accepts the response (no verification), resolves the yield, and delivers `(big_y, big_c)` to the user.
6. User computes `sig = big_c − a · big_y = attacker_scalar · H(pk, app_id)`.
7. Attacker, knowing `attacker_scalar`, computes the same `sig` and derives the same key `s = HKDF(sig)`.
8. The user's "confidential" key is now fully known to the attacker.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
