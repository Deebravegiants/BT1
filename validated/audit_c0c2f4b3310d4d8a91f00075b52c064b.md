### Title
Farmer Signs Arbitrary Foliage Block Data Hash Without Reward Address Verification — (`chia/farmer/farmer_api.py`)

### Summary

A malicious full node that receives a `DeclareProofOfSpace` from the farmer can send back a `RequestSignedValues` containing a `foliage_block_data_hash` computed from a `FoliageBlockData` with an attacker-controlled `farmer_reward_puzzle_hash`, with `foliage_block_data=None`. The farmer forwards the hash to the harvester for signing without verifying the reward address, and returns valid `SignedValues` to the attacker.

### Finding Description

In `FarmerAPI.request_signed_values` (farmer_api.py:722–769), the only guard is that `full_node_request.quality_string` must exist in `self.farmer.quality_str_to_identifiers`: [1](#0-0) 

After that check, the `foliage_block_data_hash` and `foliage_transaction_block_hash` are forwarded directly to the harvester for signing with no verification that they correspond to a block directing rewards to the farmer's configured `farmer_target`: [2](#0-1) 

The `message_data` mechanism (lines 734–747) is the only path that could carry source data for verification, but it is only populated when `foliage_block_data is not None`: [3](#0-2) 

When `foliage_block_data=None`, `message_data` stays `None` and no source-data verification is possible. The harvester's `request_signatures` handler signs whatever messages it receives without any reward-address check: [4](#0-3) 

The `RequestSignedValues` protocol message explicitly makes `foliage_block_data` optional: [5](#0-4) 

### Attack Path

1. Farmer's harvester finds a valid proof of space → farmer sends `DeclareProofOfSpace` (which includes `quality_string`) to all connected full nodes, including the malicious one.
2. Malicious full node constructs a `FoliageBlockData` with `farmer_reward_puzzle_hash` set to an attacker-controlled address, computes `malicious_hash = hash(malicious_foliage_block_data)`.
3. Malicious full node sends `RequestSignedValues(quality_string=<valid>, foliage_block_data_hash=malicious_hash, foliage_block_data=None)`.
4. Farmer passes `malicious_hash` to the harvester for signing without any reward-address check.
5. Harvester signs `malicious_hash`; farmer returns `SignedValues` to the attacker.
6. Malicious full node assembles a valid block with the malicious `FoliageBlockData` and the farmer's signature, redirecting the 0.25 XCH farmer reward.

### Impact Explanation

This is reward diversion: the farmer's signing key is used to authorize a block that pays the 0.25 XCH farmer coinbase reward to an attacker-controlled puzzle hash instead of the farmer's configured `farmer_target`. This falls squarely within the Critical scope (unauthorized reward diversion affecting XCH).

### Likelihood Explanation

Any full node the farmer is peered with can execute this attack whenever the farmer's harvester finds a winning proof. No leaked keys or broken cryptography are required — only a network-level peer connection and a valid quality string received via the normal `DeclareProofOfSpace` flow.

### Recommendation

In `request_signed_values`, before forwarding to the harvester:
- Require `foliage_block_data` to be non-None.
- Verify `std_hash(bytes(foliage_block_data)) == foliage_block_data_hash`.
- Verify `foliage_block_data.farmer_reward_puzzle_hash == self.farmer.farmer_target`.

Reject the request if any of these checks fail.
