### Title
Unconstrained `fee_proposal_fri` During Initiation Period Allows Malicious Proposer to Poison `fee_proposals_window` and Corrupt Future `l2_gas_price` — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

During the first `window_size` (10) blocks, `is_proposal_init_valid` skips the bounds check for `fee_proposal_fri` entirely when `fee_actual` is `None`. A malicious proposer can set `fee_proposal_fri` to any value (e.g., `u128::MAX`). Because the validator uses the proposer-supplied `fee_proposal_fri` to compute `batcher_block_commitment`, honest validators compute the same commitment and vote for it. The block is finalized, the malicious value is stored in `fee_proposals_window`, and after `window_size` blocks `compute_fee_actual` returns `u128::MAX` as the median, driving `l2_gas_price` to `u128::MAX` and making all transactions unaffordable.

---

### Finding Description

**Root cause — missing lower bound during initiation period.**

In `is_proposal_init_valid`, the bounds check is wrapped in an `if let` that fires only when both `fee_actual` and `fee_proposal` are `Some`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    ...
}
``` [1](#0-0) 

When `height < window_size` (10), `compute_fee_actual` returns `None` because the window is incomplete:

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ({window_size})");
    return None;
};
``` [2](#0-1) 

`fee_actual = None` is passed into `ProposalInitValidation.fee_actual` at both call sites: [3](#0-2) [4](#0-3) 

So during the first 10 blocks, `fee_proposal_fri` is completely unconstrained. The proposer's own fallback is `l2_gas_price`:

```rust
let Some(fee_actual) = fee_actual else {
    warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
    return self.l2_gas_price;
};
``` [5](#0-4) 

But the validator enforces no equivalent constraint on the proposer-supplied value.

**Commitment binding amplifies the impact.**

The validator computes `batcher_block_commitment` using the proposer's `fee_proposal_fri` directly from `ProposalInit`:

```rust
let batcher_block_commitment = proposal_commitment_from(
    finished_info.proposal_commitment.partial_block_hash,
    fee_proposal,   // ← args.init.fee_proposal_fri, proposer-supplied
);
``` [6](#0-5) 

`proposal_commitment_from` hashes the malicious value into the commitment:

```rust
pub(crate) fn proposal_commitment_from(
    partial: PartialBlockHash,
    fee_proposal: Option<GasPrice>,
) -> ProposalCommitment {
    let Some(fee_proposal) = fee_proposal else {
        return ProposalCommitment(partial.0);
    };
    ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
}
``` [7](#0-6) 

Because the validator accepted the `ProposalInit` with `fee_proposal_fri = u128::MAX`, it computes `Poseidon(partial_block_hash, u128::MAX)` — identical to the proposer's `ProposalFin.proposal_commitment`. The `ProposalFinMismatch` guard passes, the validator votes for the commitment, and the block is finalized.

**Window poisoning and `l2_gas_price` corruption.**

After finalization, `record_fee_proposal` stores the malicious `fee_proposal_fri` in `fee_proposals_window`. Once the malicious proposer has been selected for ≥ 6 of the first 10 blocks, `compute_fee_actual` returns `u128::MAX` as the median. `calculate_next_l2_gas_price_for_fin` uses `fee_actual` as a floor, setting `l2_gas_price = u128::MAX`. All subsequent transactions require paying `u128::MAX` FRI as gas.

---

### Impact Explanation

`l2_gas_price` is driven to `u128::MAX`, making every transaction unaffordable. This is an incorrect fee/gas accounting effect with direct economic impact: the network is effectively DoS'd for all users. The corrupted value also propagates to the cende pipeline via `FeeProposalInfo { fee_proposal_fri: init.fee_proposal_fri }`: [8](#0-7) 

---

### Likelihood Explanation

The attack requires a malicious proposer to be selected for more than half of the first `window_size` (10) blocks. In a small validator set (2–3 validators, as is typical at Starknet launch), a single malicious validator is selected as proposer for 3–5 of the first 10 blocks via round-robin. With 5 out of 10 slots, the median is exactly at the boundary; with 6 slots, the median is `u128::MAX`. No special network access is required beyond being a legitimate validator selected as proposer during the genesis window.

---

### Recommendation

During the initiation period (when `fee_actual` is `None`), enforce that `fee_proposal_fri` equals the validator's own `l2_gas_price` fallback — the same value the proposer is required to use. Concretely, in `is_proposal_init_valid`, add an `else` branch:

```rust
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    // existing margin check
} else if let Some(fee_proposal) = init_proposed.fee_proposal_fri {
    // Initiation period: no historical fee_actual, so the only valid
    // fee_proposal is the l2_gas_price fallback both sides must agree on.
    if fee_proposal != proposal_init_validation.l2_gas_price_fri {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "fee_proposal during initiation must equal l2_gas_price_fri \
                 ({}), got {}",
                proposal_init_validation.l2_gas_price_fri.0, fee_proposal.0
            ),
        ));
    }
}
```

This mirrors the existing `l2_gas_price_fri` equality check and closes the unconstrained window.

---

### Proof of Concept

1. Network starts at height 0; `window_size = 10`; `fee_proposal_window_size` is configured as 10. [9](#0-8) 

2. For heights 0–9, `compute_fee_actual` returns `None` (height < window_size). [2](#0-1) 

3. Malicious proposer (selected for ≥ 6 of the first 10 blocks) emits `ProposalInit { fee_proposal_fri: Some(GasPrice(u128::MAX)), ... }`.

4. Validator calls `is_proposal_init_valid`; the `if let (Some(fee_actual), ...)` guard is `false` because `fee_actual = None`. No error is returned. [1](#0-0)
