### Title
Untrusted Token Call Without Validation or Reentrancy Guard in `initTransfer` Enables Unbacked Wrapped-Token Minting — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` accepts any caller-supplied `tokenAddress` and calls `safeTransferFrom` on it without verifying the token is a registered, supported bridgeable asset. There is no reentrancy guard anywhere in the contract. A fully unprivileged attacker can supply a malicious ERC-20 whose `transferFrom` returns `true` without moving any value, causing the bridge to publish a Wormhole message claiming a deposit that never occurred. On NEAR, the bridge processes that message and mints wrapped tokens backed by nothing.

---

### Finding Description

`OmniBridge.initTransfer` branches on three token categories: [1](#0-0) 

```
if (customMinters[tokenAddress] != address(0)) { … }
else if (isBridgeToken[tokenAddress])            { … }
else {
    IERC20(tokenAddress).safeTransferFrom(       // ← arbitrary address, no whitelist check
        msg.sender, address(this), amount
    );
}
```

The `else` branch imposes **no requirement** that `tokenAddress` is a registered or supported bridgeable asset. Any address that satisfies the ERC-20 `transferFrom` ABI (returning `true`) is accepted. OpenZeppelin's `SafeERC20.safeTransferFrom` only checks that the call does not revert and that any returned bytes decode as `true`; it cannot detect a no-op transfer.

After the transfer call the function unconditionally publishes a Wormhole message via `initTransferExtension`: [2](#0-1) 

No reentrancy guard is present anywhere in the contract — confirmed by the complete absence of `ReentrancyGuard` / `nonReentrant` in all EVM production sources. The malicious token's `transferFrom` callback can therefore freely re-enter any `OmniBridge` function.

`logMetadata` is also fully permissionless and accepts any `tokenAddress`: [3](#0-2) 

This lets the attacker first register the fake token on NEAR via a Wormhole `LogMetadata` message, then trigger the unbacked `InitTransfer` message.

---

### Impact Explanation

An attacker can mint an arbitrary quantity of wrapped tokens on NEAR that are backed by zero real EVM-side collateral. This is unauthorized creation of wrapped bridge assets — a Critical impact under the allowed scope ("Unauthorized creation … of … wrapped bridge assets through … verification failure").

---

### Likelihood Explanation

All steps are permissionless and require no privileged role, leaked key, or external dependency compromise:

1. Deploy a malicious ERC-20 whose `transferFrom` returns `true` without moving tokens.
2. Call `logMetadata(maliciousToken)` — public, no access control — to publish a Wormhole `LogMetadata` message; NEAR bridge deploys a wrapped token for it.
3. Call `initTransfer(maliciousToken, amount, 0, nativeFee, nearRecipient, "")` — public, no access control — to publish a Wormhole `InitTransfer` message claiming `amount` was deposited.
4. NEAR bridge processes the message and mints `amount` of wrapped tokens to the attacker.

The reentrancy surface amplifies the attack: during step 3 the malicious `transferFrom` can re-enter `initTransfer` with a legitimate token, locking real user funds in the bridge while simultaneously publishing an additional unbacked message for the fake token.

---

### Recommendation

**Short term:**
- Add a token whitelist check in `initTransfer`'s `else` branch: revert if `tokenAddress` is not in a registry of approved bridgeable tokens (analogous to how `isBridgeToken` and `customMinters` gate the other branches).
- Add OpenZeppelin's `ReentrancyGuardUpgradeable` and apply `nonReentrant` to `initTransfer`, `initTransfer1155`, and `finTransfer`.

**Long term:**
- Enforce that `logMetadata` can only be called for tokens that pass a basic sanity check (e.g., non-zero code size, registered in a whitelist) to prevent fake-token registration on NEAR.
- Integrate Slither or Crytic into CI to flag reentrancy and arbitrary external calls automatically.

---

### Proof of Concept

```solidity
// Malicious token — transferFrom is a no-op
contract FakeToken {
    function transferFrom(address, address, uint256) external pure returns (bool) {
        return true; // no actual transfer
    }
    function name()     external pure returns (string memory) { return "Fake"; }
    function symbol()   external pure returns (string memory) { return "FAKE"; }
    function decimals() external pure returns (uint8)         { return 18; }
}

// Attack sequence (all calls are unprivileged):
// 1. Deploy FakeToken ft = new FakeToken();
// 2. bridge.logMetadata(address(ft));          // registers ft on NEAR via Wormhole
// 3. bridge.initTransfer{value: nativeFee}(
//        address(ft), 1_000_000e18, 0, nativeFee, "attacker.near", ""
//    );
// → Wormhole message published: "1_000_000e18 of ft deposited"
// → NEAR bridge mints 1_000_000e18 wrapped FAKE to attacker.near
// → EVM bridge holds 0 real tokens
```

The `safeTransferFrom` at line 407 succeeds (returns `true`), `initTransferExtension` publishes the Wormhole message at line 415, and the NEAR bridge has no EVM-side evidence that the transfer was a no-op. [4](#0-3) [5](#0-4)
