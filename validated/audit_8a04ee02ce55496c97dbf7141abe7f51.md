### Title
No Cancellation Mechanism for Pending Cross-Chain Transfers Causes Permanent Fund Lock - (File: near/omni-bridge/src/lib.rs)

### Summary
Once a user initiates a cross-chain transfer via `init_transfer` on any supported chain (NEAR, EVM, Starknet, Solana), their tokens are immediately and irrevocably burned or locked. No public cancel function exists anywhere in the protocol. If a transfer becomes stuck in `pending_transfers` — due to MPC signer unavailability, relayer downtime, or a destination-chain issue — the user has no mechanism to reclaim their principal.

### Finding Description

**NEAR origin (`near/omni-bridge/src/lib.rs`)**

`init_transfer_internal` immediately burns bridge tokens or locks native tokens, then inserts a `TransferMessage` into `pending_transfers`: [1](#0-0) 

The only code paths that ever remove a pending transfer are:

1. `sign_transfer_callback` — removes the entry only when `fee.is_zero()` and MPC signing succeeds: [2](#0-1) 

2. `claim_fee_callback` — removes the entry only after a trusted relayer submits a valid on-chain proof from the destination chain: [3](#0-2) 

3. `remove_transfer_message_without_refund` — called only on internal storage-accounting failures, never as a user-facing escape hatch: [4](#0-3) 

There is no public `cancel_transfer`, `reclaim`, or time-locked withdrawal function anywhere in the contract. A search of the entire public interface confirms zero such entry points.

**EVM origin (`evm/src/omni-bridge/contracts/OmniBridge.sol`)**

`initTransfer` burns bridge tokens or locks native ERC-20s into the contract with no cancel path: [5](#0-4) 

The contract exposes only `initTransfer`, `finTransfer`, `deployToken`, and admin functions. No cancel or refund function exists.

**Starknet origin (`starknet/src/omni_bridge.cairo`)**

`init_transfer` burns bridge tokens or transfers native tokens to the contract with no cancel path: [6](#0-5) 

The `IOmniBridge` interface exposes no cancel function: [7](#0-6) 

**Solana origin (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)**

Tokens are burned or transferred to the vault with no cancel path: [8](#0-7) 

### Impact Explanation

A transfer initiated from any chain enters a state where:
- The user's tokens are already burned or locked.
- Completion requires a trusted relayer to call `sign_transfer` (NEAR-origin) or `fin_transfer` (foreign-origin), and the MPC signer to produce a valid signature.

If either the relayer or MPC signer is unavailable for any sustained period, the `pending_transfers` entry persists indefinitely with no user recourse. For burned bridge tokens, the supply is permanently reduced with no corresponding mint on the destination chain. For locked native tokens, they remain frozen in the locker contract. In both cases the user suffers a total, irreversible loss of their bridged principal — not merely a fee — with no protocol-level escape hatch.

Additionally, `sign_transfer` panics if the token has no registered address on the destination chain (`FailedToGetTokenAddress`), meaning a transfer to a chain where the token is not yet deployed is permanently unrecoverable: [9](#0-8) 

### Likelihood Explanation

The trigger conditions are realistic operational events, not exotic attacks:
- MPC signer downtime during upgrades or key rotation.
- Relayer infrastructure failure.
- A destination chain experiencing a temporary or permanent halt.
- A user initiating a transfer to a destination chain where the token is not yet deployed (no check prevents this at `init_transfer` time).

Any of these leaves the user's funds permanently locked with no recourse.

### Recommendation

Implement a time-locked cancellation mechanism:

1. Record a `created_at` block timestamp alongside each `TransferMessage` in `pending_transfers`.
2. Expose a public `cancel_transfer(transfer_id)` function callable only by the original sender after a protocol-defined waiting period (e.g., 7 days).
3. On cancellation: remove the `pending_transfers` entry, re-mint burned bridge tokens to the sender, and unlock locked native tokens.
4. Apply the same pattern on EVM, Starknet, and Solana: store a timestamp at `initTransfer` time and allow the sender to reclaim after the timeout if no `finTransfer` has been executed.

### Proof of Concept

1. User calls `ft_transfer_call` on NEAR with a valid `InitTransferMsg` targeting an EVM chain. Tokens are burned/locked; a `TransferMessage` is inserted into `pending_transfers`.
2. The MPC signer goes offline (e.g., scheduled maintenance). `sign_transfer` is callable but the MPC cross-contract call fails; `sign_transfer_callback` receives `Err` and does nothing — the pending entry is never removed.
3. The user has no function to call. `pending_transfers` retains the entry indefinitely.
4. The user's tokens are permanently lost: burned bridge tokens cannot be re-minted without a destination-chain proof; locked native tokens cannot be unlocked without `claim_fee` which itself requires a destination-chain proof.
5. The same scenario applies on EVM/Starknet/Solana: once `initTransfer` executes, the user's assets are gone from their wallet with no cancel path if the NEAR-side `fin_transfer` is never submitted.

        fn init_transfer(
            ref self: ContractState,
            token_address: ContractAddress,
            amount: u128,
            fee: u128,
            native_fee: u128,
            recipient: ByteArray,
            message: ByteArray,
        ) {
            assert(!_is_paused(@self, PAUSE_INIT_TRANSFER), 'ERR_INIT_TRANSFER_PAUSED');

            assert(amount > 0, 'ERR_ZERO_AMOUNT');
            assert(fee < amount, 'ERR_INVALID_FEE');

            let origin_nonce = self.current_origin_nonce.read() + 1;
            self.current_origin_nonce.write(origin_nonce);

            let caller = get_caller_address();

            if self.is_bridge_token(token_address) {
                IBridgeTokenDispatcher { contract_address: token_address }
                    .burn(caller, amount.into());
            } else {
                let success = IERC20Dispatcher { contract_address: token_address }
                    .transfer_from(caller, get_contract_address(), amount.into());
                assert(success, 'ERR_TRANSFER_FROM_FAILED');
            }

            if native_fee > 0 {
                let native_token = self.strk_token_address.read();
                let success = IERC20Dispatcher { contract_address: native_token }
                    .transfer_from(caller, get_contract_address(), native_fee.into());
                assert(success, 'ERR_FEE_TRANSFER_FAILED');
            }

            self
                .emit(
                    Event::InitTransfer(
                        InitTransfer {
                            sender: caller,
                            token_address,
                            origin_nonce,
                            amount,
                            fee,
                            native_fee,
                            recipient,
                            message,
                        },
                    ),
                )
        }
