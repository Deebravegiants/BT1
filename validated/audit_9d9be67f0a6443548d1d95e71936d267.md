### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` is actually derived from the accompanying `request`. A single Byzantine attested participant (below threshold) can replay any previously published `(payload_hash, signature)` pair from a prior foreign-tx response to resolve an entirely different pending foreign-tx request, delivering a valid signature over the wrong payload to the waiting bridge contract.

---

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← caller-supplied

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← verified only against itself
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract confirms that `signature` is a valid ECDSA signature over `payload_hash` under the domain root key. It does **not** confirm that `payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the specific `request` being resolved.

Compare this with the regular `respond` function, where the payload is taken directly from the stored request — not from the response:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
``` [2](#0-1) 

In `respond_verify_foreign_tx`, `VerifyForeignTransactionRequest` stores only `(request, domain_id, payload_version)` — no committed payload hash: [3](#0-2) 

And `VerifyForeignTransactionResponse` carries a free-form `payload_hash` field that the contract never binds to the request: [4](#0-3) 

The `ForeignTxSignPayload::compute_msg_hash` function that produces the correct hash is only called off-chain by MPC nodes and optionally by bridge contracts using the SDK: [5](#0-4) 

The SDK's `ForeignChainSignatureVerifier::verify_signature` does perform this binding check client-side: [6](#0-5) 

But this check is entirely optional and is not enforced by the contract itself.

---

### Impact Explanation

**Impact: High** — Cross-chain replay / forged foreign-chain verification causing invalid bridge execution.

A Byzantine attested participant can:

1. Observe any previously successful `respond_verify_foreign_tx` on-chain and extract `(payload_hash_A, sig_A)` — both are public.
2. Identify any currently pending `VerifyForeignTransactionRequest` B (different tx, same domain).
3. Call `respond_verify_foreign_tx(request=B, response={payload_hash=payload_hash_A, signature=sig_A})`.
4. The contract accepts the call: `sig_A` is a valid root-key signature over `payload_hash_A`. The pending yield for B is resolved and the request is removed from the map.
5. The bridge contract waiting on B receives `(payload_hash_A, sig_A)` — a valid MPC signature over the wrong foreign-chain observation.

Bridge contracts that do not call `ForeignChainSignatureVerifier::verify_signature` (or implement their own verification without the payload-hash binding check) will accept this as proof that transaction B was verified, when in fact it was transaction A's data that was signed. This enables forged foreign-chain verification and potential double-spend or unauthorized bridge execution.

Even bridge contracts that do use the SDK's verifier are harmed: their pending request is consumed without a correct response, forcing a re-submission and deposit. [7](#0-6) 

---

### Likelihood Explanation

**Likelihood: Medium**

- The attacker must be an attested MPC participant (TEE-verified node), which is a non-trivial prerequisite.
- However, no threshold collusion is required — a single Byzantine participant below the signing threshold is sufficient.
- All inputs needed for the attack (`payload_hash_A`, `sig_A`, pending request B) are publicly observable on-chain.
- The attack is deterministic and requires no brute-force or timing luck.
- The Omnibridge inbound flow is an explicitly stated primary use case, making high-value pending requests a realistic target. [8](#0-7) 

---

### Recommendation

The contract must bind `response.payload_hash` to `request` before accepting the response. Since the contract cannot independently re-derive the extracted values, the minimum viable fix is to include the `request` fields (tx_id, chain, extractors, payload_version) as a domain-separation prefix inside the signed payload, and then verify on-chain that `response.payload_hash` begins with (or is committed to) the canonical encoding of `request`.

A stronger fix: include the `SignatureId` / receipt ID of the pending request inside `ForeignTxSignPayloadV1` so that each signature is cryptographically bound to exactly one request instance. The contract can then verify `payload_hash` encodes the correct request ID before resolving the yield. [9](#0-8)

### Proof of Concept

```
Setup:
  - Attacker = attested MPC participant P_evil (below threshold)
  - Domain = ForeignTx Secp256k1 domain, root public key K_root

Step 1 — Harvest a valid (payload_hash, signature) pair:
  Observe any past successful respond_verify_foreign_tx call on NEAR.
  Extract:
    payload_hash_A = SHA-256(borsh(ForeignTxSignPayload{request_A, values_A}))
    sig_A          = valid ECDSA signature over payload_hash_A under K_root

Step 2 — Identify a pending victim request:
  Read pending_verify_foreign_tx_requests from contract state.
  Pick any pending VerifyForeignTransactionRequest B (B ≠ A, same domain).

Step 3 — Submit the replayed response:
  P_evil calls:
    respond_verify_foreign_tx(
      request  = B,
      response = { payload_hash: payload_hash_A, signature: sig_A }
    )

Step 4 — Contract accepts:
  verify_ecdsa_signature(sig_A, payload_hash_A, K_root) → OK   ✓
  pending_verify_foreign_tx_requests[B] exists             → OK   ✓
  → resolve_yields_for(B, response{payload_hash_A, sig_A})

Step 5 — Bridge contract for B receives:
  VerifyForeignTransactionResponse {
    payload_hash: payload_hash_A,   // ← hash of tx A, not tx B
    signature:    sig_A,            // ← valid MPC signature
  }
  If bridge does not call ForeignChainSignatureVerifier::verify_signature:
    → accepts as proof that tx B was verified → invalid bridge execution
  If bridge does call verify_signature:
    → detects mismatch, rejects, but request B is already consumed → DoS
``` [10](#0-9) [11](#0-10)

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
