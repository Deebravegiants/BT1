Audit Report

## Title
SwapAllowlistExtension Per-User Gate Bypassed via MetricOmmSimpleRouter — Any User Can Circumvent Swap Allowlist - (File: `metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` gates swaps by checking the `sender` argument, which equals `msg.sender` of the pool's `swap` call. When any user routes through `MetricOmmSimpleRouter`, `sender` becomes the router's address. If the pool admin allowlists the router (required for any legitimate router-mediated swap), every non-allowlisted user can bypass the per-user gate by calling through the router, completely defeating the extension's access-control invariant.

## Finding Description
`SwapAllowlistExtension.beforeSwap` performs its check as:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

Here `msg.sender` is the pool (the extension caller) and `sender` is the argument forwarded from `MetricOmmPool.swap`:

```solidity
_beforeSwap(
    msg.sender,   // direct caller of pool.swap
    recipient, ...
);
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly:

```solidity
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
    .swap(params.recipient, params.zeroForOne, ...);
```

This makes `msg.sender = router` from the pool's perspective, so the extension sees `sender = router`. If the pool admin has called `setAllowedToSwap(pool, router, true)` — a necessary step for any legitimate user to swap via the router — then `allowedSwapper[pool][router]` is `true`, and the check passes for every caller of the router regardless of their own allowlist status. There is no mechanism in the router or extension to recover or verify the original end-user address. The `extensionData` field is passed through but the extension ignores it for identity purposes.

## Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (KYC'd counterparties, whitelisted market makers, protocol-controlled addresses) can be fully bypassed by any unprivileged user routing through `MetricOmmSimpleRouter`. The non-allowlisted user executes a swap and receives output tokens identically to an allowlisted user. This breaks the core access-control invariant of the extension and constitutes broken core pool functionality: the extension exists precisely to make certain swaps revert, and that guarantee is eliminated for all router-mediated paths. Pools relying on this extension for compliance or restricted-market operation are rendered functionally equivalent to open pools.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary user-facing swap interface in the periphery and is a public, permissionless contract. Any pool admin who enables router-mediated swaps for legitimate users must allowlist the router, at which point the bypass is immediately and unconditionally available to all users. Even a single legitimate allowlisted user routing through the router creates the bypass condition for all other users. No special privileges, flash loans, or contract deployment are required — a standard EOA call to `exactInputSingle` suffices.

## Recommendation
The extension must gate the actual end-user identity, not the direct caller of `pool.swap`. Two concrete approaches:

1. **Transient storage propagation**: The router already uses transient storage to pass the payer address for swap callbacks (`_setNextCallbackContext` stores `msg.sender`). A parallel slot can store the original caller's address, exposed via a standard interface that `SwapAllowlistExtension.beforeSwap` reads during its check instead of (or in addition to) `sender`.

2. **extensionData convention**: Define a standard encoding where the router encodes the original `msg.sender` into `extensionData`, and the extension decodes and verifies it. This requires a coordinated standard between router and extension but avoids transient storage complexity.

Note: `DepositAllowlistExtension` does not share this flaw because it gates the `owner` parameter (the position recipient), which is correctly passed through even when `msg.sender` is `MetricOmmPoolLiquidityAdder`.
