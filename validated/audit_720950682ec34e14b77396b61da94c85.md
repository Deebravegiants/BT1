Audit Report

## Title
Default `pricePercentageLimit = 0` Permanently Disables Auto-Pause Downside Protection in `_updateRsETHPrice()` — (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle` never initializes `pricePercentageLimit` in `initialize()`, leaving it at the Solidity default of `0`. Both the upside-revert and downside auto-pause guards inside `_updateRsETHPrice()` are gated on `pricePercentageLimit > 0`, so they short-circuit to `false` in the default state. Because `updateRSETHPrice()` is a public function callable by anyone, a significant price drop (e.g., from an EigenLayer slashing event) will be accepted and written to storage without pausing `LRTDepositPool` or `LRTWithdrawalManager`, allowing users to deposit at the depressed price and dilute existing rsETH holders.

## Finding Description
`initialize()` sets only `lrtConfig` and emits an event; `pricePercentageLimit` is never assigned:

```solidity
// contracts/LRTOracle.sol L64-68
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

`setPricePercentageLimit()` accepts any value including `0` with no lower-bound check:

```solidity
// contracts/LRTOracle.sol L125-128
function setPricePercentageLimit(uint256 _pricePercentageLimit) external onlyLRTAdmin {
    pricePercentageLimit = _pricePercentageLimit;
    emit PricePercentageLimitUpdate(_pricePercentageLimit);
}
```

Both guards inside `_updateRsETHPrice()` are conditioned on `pricePercentageLimit > 0`:

```solidity
// contracts/LRTOracle.sol L256-257
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);

// contracts/LRTOracle.sol L273-274
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
```

When `pricePercentageLimit == 0`, both evaluate to `false` unconditionally. The downside auto-pause block is therefore never reached:

```solidity
// contracts/LRTOracle.sol L277-282
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

The entry point is unrestricted:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`reinitialize()` (L72-79) also does not set `pricePercentageLimit`, confirming no upgrade path initializes it either.

**Exploit path:**
1. Protocol is deployed; `pricePercentageLimit` is `0`.
2. An EigenLayer slashing event reduces total ETH in the protocol, causing `newRsETHPrice` to fall significantly below `highestRsethPrice`.
3. Any external caller invokes `updateRSETHPrice()`.
4. `isPriceDecreaseOffLimit = (0 > 0) && ...` → `false`; the auto-pause block is skipped.
5. `rsETHPrice` is updated to the slashed value; `LRTDepositPool` and `LRTWithdrawalManager` remain unpaused.
6. A depositor calls `LRTDepositPool.depositAsset()` at the depressed price, minting more rsETH per ETH than the protocol's true backing supports.
7. When the price recovers, the depositor's rsETH is worth more than their deposit, diluting all pre-existing rsETH holders.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** Existing rsETH holders are entitled to the full appreciation of the protocol's ETH backing. When a new depositor mints rsETH at a depressed price and the price subsequently recovers, the new depositor captures a disproportionate share of the recovery, reducing the yield delivered to pre-existing holders. No funds are directly stolen, but the protocol fails to deliver the returns it promises to existing holders.

## Likelihood Explanation
**Medium.** The vulnerable state (`pricePercentageLimit == 0`) is the contract's default from the moment of deployment — no admin error is required. The protection is absent until an admin explicitly calls `setPricePercentageLimit` with a non-zero value. Additionally, an admin who later calls `setPricePercentageLimit(0)` — perhaps intending to "disable the limit" — silently re-enters the vulnerable state, as the function accepts `0` without complaint. The triggering action (`updateRSETHPrice()`) requires no privileges.

## Recommendation
1. Add a lower bound and upper bound in `setPricePercentageLimit()`:
```solidity
function setPricePercentageLimit(uint256 _pricePercentageLimit) external onlyLRTAdmin {
    if (_pricePercentageLimit == 0 || _pricePercentageLimit > 1e18) revert InvalidPricePercentageLimit();
    pricePercentageLimit = _pricePercentageLimit;
    emit PricePercentageLimitUpdate(_pricePercentageLimit);
}
```
2. Initialize `pricePercentageLimit` to a sensible default (e.g., `5e16` = 5%) inside `initialize()` so the protection is active from the first price update.

## Proof of Concept
**Foundry fork test outline:**
1. Deploy `LRTOracle` via proxy; confirm `pricePercentageLimit == 0`.
2. Seed `LRTDepositPool` with stETH; call `updateRSETHPrice()` to establish `highestRsethPrice`.
3. Simulate a 30% slashing event by reducing the mocked `getTotalAssetDeposits` return value by 30%.
4. Call `updateRSETHPrice()` from an unprivileged EOA.
5. Assert: `lrtDepositPool.paused() == false` and `rsETHPrice` has been updated to the slashed value (both confirming the auto-pause was bypassed).
6. Deposit ETH via `LRTDepositPool.depositAsset()` at the depressed price; record rsETH minted.
7. Restore the mocked TVL to its original value; call `updateRSETHPrice()` again.
8. Assert that the depositor's rsETH balance is worth more ETH than deposited, at the expense of pre-existing holders whose share value decreased.
