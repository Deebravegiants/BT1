Audit Report

## Title
ETH Push-Transfer Revert in `_transferAsset` Permanently Freezes User Funds After rsETH Is Burned - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTWithdrawalManager` uses a two-phase withdrawal design where rsETH is burned and ETH is redeemed from the vault in a separate operator transaction (`unlockQueue`, Phase 1), before the user calls `completeWithdrawal` to receive ETH (Phase 2). If the ETH push in Phase 2 always reverts — because the recipient address is a contract that cannot accept ETH — the Phase 2 transaction rolls back entirely, restoring the withdrawal request in the queue, but the rsETH burned in Phase 1 is irrecoverable. The ETH is permanently frozen in the contract with no rescue path.

## Finding Description

**Phase 1 — `unlockQueue` (operator-only, separate transaction):**
At lines 305–307, rsETH is burned from the withdrawal manager and ETH is redeemed from the unstaking vault into the contract. This transaction completes atomically and is irreversible.

**Phase 2 — `_processWithdrawalCompletion` (lines 699–738):**
The function pops the user's nonce (L705), deletes the request (L712), decrements `unlockedWithdrawalsCount[asset]` (L717), and then calls `_transferAsset` (L734).

`_transferAsset` (lines 876–883) sends ETH via a bare `.call{value:}("")` and hard-reverts on failure:
```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` is a contract with no `receive()` function or a reverting fallback, the entire Phase 2 transaction reverts. Solidity's atomicity restores the deleted request, the popped nonce, and the decremented counter — so the withdrawal request re-enters the queue. However, the rsETH burned in Phase 1 is not restored; it is permanently destroyed.

Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` (L192–203) passes the same `user` address (recorded at initiation time via `userAssociatedNonces[asset][msg.sender]` at L756) and will always revert identically. There is no mechanism to redirect the withdrawal to a different address.

Because `unlockedWithdrawalsCount[asset]` is never permanently decremented, `hasUnlockedWithdrawals(asset)` returns `true` indefinitely, blocking `sweepRemainingAssets` at line 403:
```solidity
if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();
```
The ETH is permanently locked with no administrative recovery path.

## Impact Explanation

**Critical — Permanent freezing of funds.** The user's rsETH is permanently destroyed (burned in Phase 1) and the corresponding ETH is permanently frozen inside the withdrawal manager. Neither the user nor any operator can recover the ETH: `completeWithdrawal` always reverts, `completeWithdrawalForUser` targets the same unrecoverable address, and `sweepRemainingAssets` is gated behind `hasUnlockedWithdrawals`. This matches the allowed impact: *Critical. Permanent freezing of funds.*

## Likelihood Explanation

Smart-contract wallets (Gnosis Safe, ERC-4337 account-abstraction wallets, protocol-owned multisigs, DAO treasuries) are the standard custody mechanism for DeFi protocols and power users. Many such contracts lack a `receive()` function or have a fallback that reverts under certain conditions (gas limits, reentrancy guards, paused state). `initiateWithdrawal` is fully permissionless — any address, including a contract, can call it. The operator-driven `unlockQueue` processes requests in FIFO order without inspecting recipient capabilities, so a single such request in the queue triggers the freeze. **Likelihood: Medium.**

## Recommendation

Replace the push-payment pattern with a pull-payment (claim) pattern: during `_processWithdrawalCompletion`, store the owed ETH in a `mapping(address => uint256) pendingETHClaims` instead of pushing it, and provide a separate `claimETH()` function that the user calls to pull their ETH. This decouples the accounting update (which must succeed) from the ETH delivery (which may fail for contract recipients). Alternatively, wrap the `.call` in a non-reverting path that stores unclaimed ETH and emits an event on failure, allowing a subsequent claim.

## Proof of Concept

1. Deploy `MaliciousWallet` — a contract with no `receive()` function (or one whose fallback always reverts).
2. `MaliciousWallet` calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to the withdrawal manager and the request is queued.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. At L305, rsETH is burned; at L307, ETH is redeemed from the vault into the withdrawal manager. This transaction succeeds and is final.
4. `MaliciousWallet` (or operator via `completeWithdrawalForUser`) calls `completeWithdrawal(ETH_TOKEN, "")`.
5. `_processWithdrawalCompletion` deletes the request (L712), decrements `unlockedWithdrawalsCount` (L717), then calls `_transferAsset` (L734).
6. `payable(MaliciousWallet).call{value: amount}("")` returns `sent = false`; `EthTransferFailed` is thrown; the entire transaction reverts. The request and counter are restored.
7. Steps 4–6 repeat indefinitely — every attempt reverts.
8. `hasUnlockedWithdrawals(ETH_TOKEN)` returns `true` forever; `sweepRemainingAssets` is blocked at L403.
9. The ETH is permanently frozen in the contract; the user's rsETH is permanently destroyed.

**Foundry test sketch:**
```solidity
function test_permanentFreeze() public {
    MaliciousWallet wallet = new MaliciousWallet(); // no receive()
    // Fund wallet with rsETH, approve withdrawal manager
    vm.prank(address(wallet));
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
    // Operator unlocks — rsETH burned here, ETH redeemed
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH_TOKEN, nextNonce, minRsETH, maxRsETH, minAsset, maxAsset);
    // Phase 2 always reverts
    vm.prank(address(wallet));
    vm.expectRevert(LRTWithdrawalManager.EthTransferFailed.selector);

    // ETH frozen, sweepRemainingAssets blocked
    vm.prank(manager);

}
```
