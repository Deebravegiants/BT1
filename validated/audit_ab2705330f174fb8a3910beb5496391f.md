### Title
Mempool eviction is not rolled back when the triggering transaction is rejected, allowing unprivileged callers to permanently remove other users' queued transactions — (File: `crates/apollo_mempool/src/mempool.rs`)

---

### Summary

`try_make_space` in `crates/apollo_mempool/src/mempool.rs` permanently evicts gap-account transactions from both the pool and the queue to make room for an incoming transaction. If the incoming transaction is ultimately rejected because even after eviction there is still insufficient space, the evictions are **not rolled back**. An attacker can submit a large-but-gateway-valid transaction that triggers eviction of other users' queued transactions and is itself rejected, permanently destroying those transactions at zero cost to the attacker.

---

### Finding Description

When the mempool is full and a new transaction arrives, `add_tx_validations` calls `handle_capacity_overflow`. That function computes `required_space = tx.total_bytes()` and calls `try_make_space(required_space)`. [1](#0-0) 

`try_make_space` iterates over `accounts_with_gap`, permanently removing their transactions from the pool and queue one by one until `total_space_freed >= required_space` or no more evictable accounts remain: [2](#0-1) 

If `try_make_space` returns `false` (not enough space freed), `handle_capacity_overflow` returns `MempoolError::MempoolFull` and the incoming transaction is rejected. **The evictions already committed to the pool and queue are not reversed.**

The test `rejects_or_accepts_tx_based_on_freed_space` in `crates/apollo_mempool/src/fee_mempool_test.rs` explicitly documents and asserts this behaviour: [3](#0-2) 

The comment reads: *"Transactions tx1 and tx2 are evicted regardless of whether large_tx is accepted or rejected. We do not revert the eviction attempt even if adding large_tx ultimately fails."*

The gateway's stateless validator enforces `max_signature_length = 4000` felts (≈ 128 KB per transaction): [4](#0-3) 

A transaction carrying a 4000-felt signature passes `validate_tx_signature_size`: [5](#0-4) 

and passes stateful validation (nonce check, balance check) without the fee ever being charged, because the transaction is rejected by the mempool before sequencing.

---

### Impact Explanation

**Impact: High — Mempool admission rejects valid transactions before sequencing.**

An attacker with a funded account submits a large invoke transaction (signature up to 4000 felts, ≈ 128 KB) whose nonce equals the account nonce (so `creating_gap = false` and `try_make_space` is reached). If the mempool is nearly full and the total size of evictable gap-account transactions is smaller than the attacker's transaction, `try_make_space` evicts those transactions but returns `false`. The attacker's transaction is rejected with `MempoolError::MempoolFull`, but the evicted transactions — which were previously admitted valid transactions — are permanently destroyed. Affected users must resubmit. The attacker pays no fee because the transaction never reaches the batcher.

The corrupted value is the mempool's transaction set: previously-admitted, gateway-valid transactions belonging to innocent users are irreversibly removed.

---

### Likelihood Explanation

**Likelihood: Medium.**

Requirements:
1. A funded account (to pass stateful balance/nonce checks at the gateway).
2. The mempool to be near capacity — common under load.
3. At least one gap account in the mempool whose total transaction bytes are less than the attacker's transaction size.
4. The attacker's transaction must be larger than the freed space so it is ultimately rejected.

All four conditions are achievable in production. The attack is repeatable and essentially free: no fee is charged on rejection, so the attacker only needs a funded account to pass stateful pre-validation.

---

### Recommendation

**Short term:** Before mutating state in `try_make_space`, perform a dry-run pass that sums the bytes of evictable transactions without removing them. Only proceed with actual eviction if the sum meets `required_space`. This eliminates the window where evictions are committed but the incoming transaction is still rejected.

**Long term:** Implement a two-phase eviction: (1) collect the set of transactions to evict and verify the freed space is sufficient; (2) atomically commit the evictions and insert the incoming transaction. If step 1 fails, no evictions occur.

---

### Proof of Concept

```
1. Fill the mempool to capacity with transactions from account "0x1"
   that have nonce 1 while account_nonce = 0 (gap account, evictable).
   Total size of these transactions: S bytes.

2. Craft an invoke transaction from a fresh account "0x0" with:
   - nonce = 0 (== account_nonce, so creating_gap = false)
   - signature = [0x0; 4000]  (4000 felts, ≈ 128 KB > S)
   This passes gateway stateless validation (max_signature_length = 4000).
   This passes gateway stateful validation (funded account, valid nonce).

3. Submit the transaction to the gateway → mempool.

4. In the mempool:
   - exceeds_capacity() = true  (mempool full + large tx)
   - handle_capacity_overflow() is called
   - creating_gap = false  (nonce 0 == account_nonce 0)
   - try_make_space(128 KB) is called
   - All S bytes of "0x1"'s transactions are evicted (S < 128 KB)
   - try_make_space returns false  (S < 128 KB)
   - handle_capacity_overflow returns MempoolError::MempoolFull
   - Attacker's transaction is rejected

5. Result: "0x1"'s transactions are permanently gone from pool and queue.
   Attacker paid no fee. Attack is repeatable.
```

```rust
    pub fn try_make_space(&mut self, required_space: u64) -> bool {
        let mut total_space_freed = 0;
        let mut evicted_txs = Vec::new();

        while total_space_freed < required_space {
            let Some(address) = self.get_evictable_account() else {
                break;
            };

            let txs: Vec<_> = self.tx_pool.account_txs_sorted_by_nonce(address).copied().collect();
            for tx_ref in txs.iter().rev() {
                let tx = self
                    .tx_pool
                    .remove(tx_ref.tx_hash)
                    .expect("Transaction must exist in the pool.");
                total_space_freed += tx.total_bytes();
                evicted_txs.push(*tx_ref);
                metric_count_evicted_txs(1);
                self.decrement_stuck_txs_if_gap_account(address, 1);
                if total_space_freed >= required_space {
                    break;
                }
            }

            // Clean up if account is now empty.
            if !self.tx_pool.contains_account(address) {
                self.accounts_with_gap.swap_remove(&address);
            }
        }

        // Keep the queue consistent with the pool: the evicted txs were removed from the pool, so
        // drop their (now-orphaned) queue references too. In fee mode gap accounts are never
        // queued, so this is a no-op there; in Echonet/FIFO mode they are, and skipping this leaves
        // a dangling reference that panics the next `get_txs`.
        self.tx_queue.remove_txs(&evicted_txs);

        total_space_freed >= required_space
    }
```

```rust
    fn handle_capacity_overflow(
        &mut self,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
        freed_bytes: u64,
    ) -> Result<(), MempoolError> {
        let address = tx.contract_address();

        let account_has_gap = self.accounts_with_gap.contains(&address);
        let account_has_txs = self.tx_pool.contains_account(address);
        let closing_gap = tx.nonce() == account_nonce;
        let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;

        // Only the net growth must be evicted: an accompanying replacement removal frees
        // `freed_bytes` (0 when there is no replacement).
        let required_space = tx.total_bytes().saturating_sub(freed_bytes);
        if !creating_gap && self.try_make_space(required_space) {
            return Ok(());
        }

        Err(MempoolError::MempoolFull)
    }
```

```rust
#[case::insufficient_eviction_space(20, false)]
#[case::sufficient_eviction_space(8, true)]
fn rejects_or_accepts_tx_based_on_freed_space(
    #[case] signature_size: usize,
    #[case] expect_success: bool,
) {
    let tx1 = add_tx_input!(tx_hash: 1, address: "0x1", tx_nonce: 1, account_nonce: 0);
    let tx2 = add_tx_input!(tx_hash: 2, address: "0x1", tx_nonce: 2, account_nonce: 0);

    let large_signature = vec![felt!("0x0"); signature_size];
    let large_tx = AddTransactionArgs {
        tx: internal_invoke_tx(invoke_tx_args!(
            tx_hash: tx_hash!(3),
            signature: TransactionSignature(large_signature.into())
        )),
        account_state: AccountState { address: contract_address!("0x0"), nonce: nonce!(0) },
    };

    let capacity = tx1.tx.total_bytes() + tx2.tx.total_bytes();
    let mut mempool = Mempool::new(
        MempoolConfig {
            static_config: MempoolStaticConfig {
                capacity_in_bytes: capacity,
                ..Default::default()
            },
            ..Default::default()
        },
        Arc::new(FakeClock::default()),
    );

    for tx in [&tx1, &tx2] {
        add_tx(&mut mempool, tx);
    }

    if expect_success {
        add_tx(&mut mempool, &large_tx);
        assert!(mempool.tx_pool.get_by_tx_hash(large_tx.tx.tx_hash()).is_ok());
    } else {
        add_tx_expect_error(&mut mempool, &large_tx, MempoolError::MempoolFull);
    }

    // Transactions tx1 and tx2 are evicted regardless of whether large_tx is accepted or rejected.
    // We do not revert the eviction attempt even if adding large_tx ultimately fails.
    assert!(!mempool.tx_pool.contains_account(contract_address!("0x1")));
```

```rust
    fn validate_tx_signature_size(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let signature = tx.signature();

        let signature_length = signature.0.len();
        if signature_length > self.config.max_signature_length {
            return Err(StatelessTransactionValidatorError::SignatureTooLong {
                signature_length,
                max_signature_length: self.config.max_signature_length,

        }

        Ok(())
```
