Audit Report

## Title
Uninitialized `withdrawalDelay` and `maxNumberOfWithdrawalsPerUser` in `KernelDepositPool` Allow Withdrawal Delay Bypass and Cause Temporary Fund Freeze - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` sets only token addresses and the admin role, leaving `withdrawalDelay` and `maxNumberOfWithdrawalsPerUser` at their Solidity defaults of zero. A zero `maxNumberOfWithdrawalsPerUser` causes every `initiateWithdrawal()` call to revert immediately, freezing all staked KERNEL until the admin intervenes. A zero `withdrawalDelay` sets `unlockTime` equal to `block.timestamp`, making the time-lock check in `claimWithdrawal()` trivially passable in the same or any subsequent block.

## Finding Description
`initialize()` (L259–271) does not assign `withdrawalDelay` or `maxNumberOfWithdrawalsPerUser`, leaving both at zero.

**Root cause 1 — fund freeze:** `initiateWithdrawal()` at L323 checks:
```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```
With `maxNumberOfWithdrawalsPerUser == 0`, the condition evaluates to `0 >= 0 == true` on every call, causing an unconditional revert. Any user who calls `stake()` before the admin calls `setMaxNumberOfWithdrawalsPerUser()` cannot retrieve their principal.

**Root cause 2 — delay bypass:** `initiateWithdrawal()` at L330 computes:
```solidity
uint256 unlockTime = block.timestamp + withdrawalDelay;
```
With `withdrawalDelay == 0`, `unlockTime == block.timestamp`. The guard in `claimWithdrawal()` at L355–357:
```solidity
if (block.timestamp < withdrawal.unlockTime) revert WithdrawalNotReady();
```
evaluates to `block.timestamp < block.timestamp == false`, so the withdrawal is immediately claimable with no enforced delay.

The setter functions `setWithdrawalDelay()` (L598–604) and `setMaxNumberOfWithdrawalsPerUser()` (L610–620) are admin-only and are not invoked during initialization. No guard in `stake()` or `initiateWithdrawal()` verifies these parameters are configured before allowing user operations.

## Impact Explanation
**Primary — Medium: Temporary freezing of funds.** Between deployment and the admin's call to `setMaxNumberOfWithdrawalsPerUser()`, every staked KERNEL token is frozen. Users who have called `stake()` cannot retrieve their principal. The freeze window is bounded only by admin responsiveness; there is no on-chain guarantee it closes promptly.

**Secondary — Low: Contract fails to deliver promised returns.** If `setWithdrawalDelay()` is not called before users begin staking, withdrawals initiated during that window carry a zero delay, permanently bypassing the lockup that later stakers must respect and violating the protocol's fairness invariant.

## Likelihood Explanation
The contract is deployed and initialized in a single transaction that does not set either parameter. Any user who stakes before the admin's follow-up configuration transactions will be affected. No special privilege, flash loan, or complex setup is required — only the ability to call `stake()` and observe that `initiateWithdrawal()` reverts. The attack window is open from the moment of deployment and is visible to any mempool observer.

## Recommendation
Set both parameters to safe non-zero defaults inside `initialize()`:
```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ...
    withdrawalDelay = 7 days;
    maxNumberOfWithdrawalsPerUser = 10;
}
```
Alternatively, add explicit guards in `initiateWithdrawal()` to revert if either parameter remains unconfigured:
```solidity
if (withdrawalDelay == 0) revert WithdrawalDelayNotSet();
if (maxNumberOfWithdrawalsPerUser == 0) revert WithdrawalLimitNotSet();
```

## Proof of Concept
```
// Scenario A — fund freeze
1. Admin deploys and calls initialize() — maxNumberOfWithdrawalsPerUser = 0.
2. Alice calls stake(100e18) — succeeds, balanceOf[Alice] = 100e18.
3. Alice calls initiateWithdrawal(100e18):
   → userWithdrawalIds[Alice].length == 0
   → 0 >= 0 (maxNumberOfWithdrawalsPerUser) → true → revert WithdrawalLimitReached()
4. Alice's 100e18 KERNEL is frozen until admin calls setMaxNumberOfWithdrawalsPerUser(N).

// Scenario B — delay bypass
1. Admin calls setMaxNumberOfWithdrawalsPerUser(10) but omits setWithdrawalDelay().
2. Bob calls stake(100e18) at block T.
3. Bob calls initiateWithdrawal(100e18):
   → unlockTime = block.timestamp + 0 = T
4. Bob calls claimWithdrawal(id) in the next block (T+1):
   → block.timestamp (T+1) < unlockTime (T) → false → no revert
   → Bob receives 100e18 KERNEL immediately, bypassing any intended lockup.
```
