### Title
Missing `observed_at` Timestamp in `ForeignTxSignPayloadV1` Enables Stale Foreign Chain Observation Attestation - (File: crates/near-mpc-contract-interface/src/types/foreign_chain.rs)

### Summary

The `ForeignTxSignPayloadV1` struct omits the `observed_at` field that the design specification explicitly requires. The signed payload commits only to `(request, values)` instead of the specified `(request, observed_values, observed_at)`. Bridge contracts receiving MPC attestations have no cryptographic means to verify when the foreign chain observation was made, enabling acceptance of stale attestations after a foreign chain reorganization.

### Finding Description

The design document for foreign chain transaction verification states:

> "If extraction succeeds, MPC signs a canonical encoding of `(request, observed_values, observed_at)` and returns the signature on-chain."

However, the actual implementation of `ForeignTxSignPayloadV1` contains only:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [1](#0-0) 

The `observed_at` field (block height or timestamp at which the observation was made) is entirely absent. The `compute_msg_hash()` function hashes only `(request, values)`: [2](#0-1) 

So the resulting MPC signature does not commit to the observation time. The design doc's `observed_at` is a dead letter in the implementation.

The `ForeignChainSignatureVerifier::verify_signature` in the SDK only checks signature validity and payload hash equality — no freshness check exists: [3](#0-2) 

Additionally, in `crates/node/src/mpc_client.rs`, the leader node caches its computed response in `computation_progress.computed_response` and reuses it on retry: [4](#0-3) 

If a reorg occurs after the threshold signature is computed but before the NEAR transaction is finalized, the bridge contract receives and processes a stale attestation with no way to detect it.

### Impact Explanation

A bridge contract (e.g., Omnibridge — the primary stated use case) receiving a `VerifyForeignTransactionResponse` cannot determine when the MPC network made its observation. If a foreign chain reorganization occurs after the MPC nodes sign the response but before the bridge contract processes it, the bridge contract accepts an attestation based on a reorganized (now-invalid) transaction. This enables a double-spend condition: a user submits a transaction on the foreign chain, the MPC network attests to it, the transaction is reorganized away, but the bridge still processes the transfer. The signed payload `(request, values)` remains cryptographically valid even after the observed block hash is no longer canonical.

This maps to: **forged foreign-chain verification that causes invalid bridge execution or double-spend conditions** (High impact).

### Likelihood Explanation

The attack requires a foreign chain reorganization to occur after the MPC nodes sign the response. For `EthereumFinality::Latest` or `EthereumFinality::Safe` — both supported finality levels: [5](#0-4) 

natural 1–2 block reorgs occur occasionally on Ethereum without attacker intervention. An attacker can submit many `verify_foreign_transaction` requests for recent blocks and wait for a natural reorg to coincide. No active chain manipulation is required for `Latest` finality. For `Finalized` or Bitcoin with 6 confirmations, likelihood is very low.

### Recommendation

Add an `observed_at` field (NEAR block height or block timestamp at signing time) to `ForeignTxSignPayloadV1` so the MPC signature commits to the observation time:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
    pub observed_at_block_height: u64,  // NEAR block height when observation was made
}
```

Bridge contracts should then enforce a maximum acceptable age for `observed_at_block_height` relative to the current block, rejecting attestations that are too old. The `ForeignChainSignatureVerifier` in the SDK should expose this field so callers can perform the freshness check.
