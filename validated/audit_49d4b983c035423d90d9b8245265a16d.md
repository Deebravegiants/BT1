### Title
Missing `curve.check()` after `from_bin` in `blob_to_keyset` allows infinity/invalid point to be returned as root public key — (`src/cbmpc/api/hd_keyset_ecdsa_2p.cpp`)

### Summary
`extract_root_public_key_compressed` deserializes a caller-supplied keyset blob and returns the root public key. The internal `blob_to_keyset` function calls `from_bin` on `root.Q` and `root.K` but never calls `curve.check()`, unlike the analogous `blob_to_key` in `ecdsa2pc.cpp`. For P-256/P-384/P-521 curves, `from_bin` accepts the infinity point encoding (leading `0x00` byte), so a crafted blob causes the function to return `SUCCESS` with an infinity-encoded public key.

### Finding Description
In `blob_to_keyset` (`src/cbmpc/api/hd_keyset_ecdsa_2p.cpp`, lines 98–100), after calling `keyset.root.Q.from_bin(icurve, blob.root_Q_compressed)`, there is no subsequent `icurve.check(keyset.root.Q)` call. The OSSL backend (`ecurve_ossl_t::from_bin`) explicitly accepts the infinity point encoding (a buffer of `1 + curve.size()` zero bytes). `curve.check()` is the only guard that rejects infinity and out-of-subgroup points. Its absence means a crafted blob with `root_Q_compressed` = 33 zero bytes (for P-256) passes deserialization, and `to_compressed_bin()` on the resulting infinity point is returned to the caller as the root public key with `SUCCESS`.

The analogous `blob_to_key` in `ecdsa2pc.cpp` correctly calls `curve.check()` after `from_bin` (line 82), making the omission in the HD keyset path a clear inconsistency.

### Impact Explanation
The public API `extract_root_public_key_compressed` returns `SUCCESS` with an invalid (infinity) public key encoding instead of an error. Any caller that trusts this output — e.g., for key display, cross-party comparison, or downstream verification — receives a semantically invalid value with no error signal. This constitutes a public-API-reachable validation bypass producing invalid cryptographic output.

### Likelihood Explanation
The attacker only needs to supply a crafted keyset blob to the public API. No threshold collusion, key compromise, or privileged access is required. The blob format is documented (versioned struct), making it straightforward to craft. The affected curves (P-256/P-384/P-521) are supported by the API.

### Recommendation
Add `curve.check()` calls after both `from_bin` calls in `blob_to_keyset`:
```cpp
error_t rv = keyset.root.Q.from_bin(icurve, blob.root_Q_compressed);
if (rv) return rv;
if (icurve.check(keyset.root.Q)) return coinbase::error(E_FORMAT, "invalid keyset blob Q");
rv = keyset.root.K.from_bin(icurve, blob.root_K_compressed);
if (rv) return rv;
if (icurve.check(keyset.root.K)) return coinbase::error(E_FORMAT, "invalid keyset blob K");
```
Apply the same fix to the EdDSA variant in `src/cbmpc/api/hd_keyset_eddsa_2p.cpp`.

### Proof of Concept
For P-256 (`curve_id::p256`):
1. Construct a valid `keyset_blob_v1_t` with all scalar fields in range, a valid Paillier key, and `root_Q_compressed` = 33 zero bytes, `root_K_compressed` = 33 zero bytes.
2. Serialize it and call `extract_root_public_key_compressed(blob, out)`.
3. Assert: the function returns `SUCCESS` and `out` contains the infinity encoding (1-byte `0x00`), rather than returning an error.
