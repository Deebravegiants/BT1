The code confirms this is a real structural issue. Here is the analysis:

---

**Key Code Facts**

`FeeReceiver.depositPool` is a standalone state variable, set at init and only updatable via `setDepositPool()` (MANAGER role): [1](#0-0) [2](#0-1) 

`sendFunds()` reads directly from `depositPool` — it never queries `LRTConfig`: [3](#0-2) 

`LRTConfig.setContract()` updates `contractMap[LRT_DEPOSIT_POOL]` independently, with no callback or notification to `FeeReceiver`: [4](#0-3) 

`receiveFromRewardReceiver()` on `LRTDepositPool` has no access control — any address can receive ETH into it: [5](#0-4) 

---

**Why the "Permanent Freezing" Claim is Overstated**

The scenario requires:
1. Admin calls `LRTConfig.setContract(LRT_DEPOSIT_POOL, newPool)` — a legitimate admin action.
2. MANAGER **omits** calling `FeeReceiver.setDepositPool(newPool)` — an operational oversight.
3. Anyone calls `sendFunds()` — no access control, so this is permissionless.

The structural divergence is real. However, the "permanent freezing" impact is not guaranteed:

- The old `LRTDepositPool` still has admin-controlled ETH recovery paths: `transferETHToNodeDelegator` and `transferETHToLRTUnstakingVault`, both gated by `onlyAssetTransferRole`. [6](#0-5) 
- The old pool's `lrtConfig` reference still resolves roles, so the admin retains the ability to drain ETH from the stale pool.
- ETH sent to the old pool is **recoverable** unless the old pool is decommissioned in a way that revokes all admin access or replaces the implementation with one lacking recovery functions.

---

**Verdict**

### Title
Stale `depositPool` in `FeeReceiver` Misdirects MEV Rewards After `LRTConfig` Pool Upgrade — (`contracts/FeeReceiver.sol`)

### Summary
`FeeReceiver` caches the deposit pool address locally and does not read it dynamically from `LRTConfig`. If the admin upgrades the deposit pool in `LRTConfig` without also calling `FeeReceiver.setDepositPool()`, all subsequent `sendFunds()` calls route MEV ETH to the decommissioned pool.

### Finding Description
`FeeReceiver.depositPool` is set at initialization and only updated via a separate MANAGER-gated call. `sendFunds()` uses this cached address unconditionally. `LRTConfig.setContract(LRT_DEPOSIT_POOL, newPool)` has no mechanism to notify or update `FeeReceiver`. The two addresses can silently diverge. [3](#0-2) [7](#0-6) 

### Impact Explanation
MEV/execution-layer rewards accumulate in the stale (old) `LRTDepositPool` rather than the active one. This misaccounts TVL and disrupts reward distribution. However, because the old pool retains admin-accessible ETH transfer functions, the funds are **recoverable** by the admin — making this a **temporary** rather than permanent freeze in the typical upgrade scenario. Permanent freezing would only occur if the old pool's admin access is also revoked or the implementation replaced without recovery functions.

**Adjusted impact: Medium — Temporary freezing of funds / misdirection of MEV rewards.**

### Likelihood Explanation
Requires a deposit pool upgrade (a rare but legitimate admin operation) combined with an omission of a second update call. The permissionless `sendFunds()` means any caller can trigger the misdirection immediately after the divergence occurs. Likelihood is low-to-medium given the operational dependency.

### Recommendation
`FeeReceiver.sendFunds()` should resolve the deposit pool address dynamically from `LRTConfig` rather than from a cached state variable:

```solidity
// Instead of:
ILRTDepositPool(depositPool).receiveFromRewardReceiver{value: balance}();

// Use:
address activePool = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
ILRTDepositPool(activePool).receiveFromRewardReceiver{value: balance}();
```

This eliminates the synchronization requirement entirely. `FeeReceiver` would need a reference to `LRTConfig` (similar to how `LRTDepositPool` uses `LRTConfigRoleChecker`). [8](#0-7) 

### Proof of Concept
1. Deploy `LRTConfig`, `FeeReceiver` (pointing to `poolV1`), and `poolV1`.
2. Admin calls `LRTConfig.setContract(LRT_DEPOSIT_POOL, poolV2)`.
3. MANAGER does **not** call `FeeReceiver.setDepositPool(poolV2)`.
4. Send ETH to `FeeReceiver` (simulating MEV rewards).
5. Call `FeeReceiver.sendFunds()`.
6. Assert: `poolV1.balance > 0` and `poolV2.balance == 0`.
7. Assert: `LRTConfig.getContract(LRT_DEPOSIT_POOL) == poolV2` (divergence confirmed).
