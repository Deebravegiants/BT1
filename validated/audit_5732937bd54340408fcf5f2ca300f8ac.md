Audit Report

## Title
Unprotected `updateRate()` Allows Any Caller to Push Stale rsETH/ETH Rate Cross-Chain — (File: `contracts/cross-chain/CrossChainRateProvider.sol`, `contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary
`CrossChainRateProvider.updateRate()` and `MultiChainRateProvider.updateRate()` are `external payable nonReentrant` with no access control, allowing any unprivileged caller to broadcast the stored `rsETHPrice` from `LRTOracle` to L2 receiver contracts via LayerZero. Because `rsETHPrice` is a cached value updated only on explicit calls to `updateRSETHPrice()`, an attacker can selectively push a stale (lower-than-current) rate to L2 during the normal staleness window, causing L2 depositors to receive more wrsETH per ETH than the protocol intends and diluting existing holders.

## Finding Description
`CrossChainRateProvider.updateRate()` at line 85 carries only `nonReentrant` — no `onlyOwner` or role check:

```solidity
function updateRate() external payable nonReentrant {
    uint256 latestRate = getLatestRate();
    ...
    ILayerZeroEndpoint(layerZeroEndpoint).send{ value: msg.value }(...);
}
```

`MultiChainRateProvider.updateRate()` at line 108 is identical in access posture, iterating over all registered `rateReceivers`.

`RSETHRateProvider.getLatestRate()` (line 27–29) and `RSETHMultiChainRateProvider.getLatestRate()` (line 26–28) both resolve to `ILRTOracle(rsETHPriceOracle).rsETHPrice()` — the *stored* value in `LRTOracle`, not a freshly computed one. This value is only refreshed when `LRTOracle.updateRSETHPrice()` is called (line 87), which is itself `public whenNotPaused` but is not called atomically with every block.

The exploit path:
1. `rsETHPrice` in `LRTOracle` is stale (lower than the true current value) — a routine condition between keeper refreshes.
2. Attacker calls `updateRate{value: lzFee}()` on `RSETHRateProvider` or `RSETHMultiChainRateProvider`, pushing the stale lower rate to the L2 `rateReceiver`.
3. L2 pool (`RSETHPoolV2`) reads the pushed rate via `IOracle(rsETHOracle).getRate()` and computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` (line 233).
4. With a stale lower rate, the division yields a larger `rsETHAmount`, over-minting wrsETH for L2 depositors and diluting the backing ratio for all existing holders.

The attacker has full timing discretion: they can wait for the maximum staleness window and push at the moment of greatest divergence, while the authorized keeper cannot prevent this without adding access control.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

No direct ETH theft occurs. The harm is dilution: existing rsETH/wrsETH holders' backing ratio is reduced because new depositors receive more wrsETH than the correct rate warrants. The protocol fails to maintain its exchange-rate guarantee for existing holders during the staleness window. This matches the allowed Low impact class exactly.

## Likelihood Explanation
**Medium.** The function is unconditionally callable by any EOA or contract. The only cost to the attacker is the LayerZero messaging fee (`msg.value`), which is modest. The stale-price precondition is a routine operational condition — oracle updates are not atomic with every block — so the window is reliably and repeatedly available. The attacker bears no protocol risk and can repeat the attack across every staleness window.

## Recommendation
Add `onlyOwner` (or an equivalent keeper role) to `updateRate()` in both base contracts, restricting rate propagation to authorized infrastructure:

```solidity
// CrossChainRateProvider.sol
function updateRate() external payable nonReentrant onlyOwner { ... }

// MultiChainRateProvider.sol
function updateRate() external payable nonReentrant onlyOwner { ... }
```

This ensures only the protocol's own automation can choose when to propagate the rate, eliminating the attacker's timing advantage.

## Proof of Concept
**Minimal Foundry fork test plan:**

1. Fork mainnet; deploy `RSETHRateProvider` pointing at the live `LRTOracle`.
2. Record `LRTOracle.rsETHPrice()` as `stalePrice`.
3. Advance time by several hours without calling `updateRSETHPrice()`, so the true computed price diverges upward from `stalePrice`.
4. From an unprivileged address (`vm.prank(attacker)`), call `rateProvider.updateRate{value: lzFee}()`.
5. Assert the L2 receiver's stored rate equals `stalePrice` (not the current computed price).
6. Simulate an L2 deposit: compute `rsETHAmount = amountAfterFee * 1e18 / stalePrice` vs. `amountAfterFee * 1e18 / currentPrice`; assert the former is strictly greater, confirming over-minting.

No privileged access, no external compromise, and no victim mistake is required — only the attacker paying the LayerZero fee.
