Audit Report

## Title
Zero-output guard missing in `deposit(token, amount, referralId)` allows permanent loss of deposited tokens — (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPool.sol`)

## Summary

All four pool variants compute `rsETHAmount` via integer division without a post-computation zero-output guard. For small deposits of low-decimal tokens, the division truncates to zero. The contract pulls the user's tokens in via `safeTransferFrom`, then mints or transfers zero rsETH out — with no revert and no refund. The deposited tokens are absorbed into the pool's general balance and subsequently bridged to L1, making recovery impossible.

## Finding Description

Every pool variant's `viewSwapRsETHAmountAndFee(amount, token)` computes:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, integer division yields `rsETHAmount = 0`.

The deposit flow in all four contracts follows the same pattern — tokens are pulled **before** the output amount is computed, and there is no guard on the result:

- `RSETHPoolV3.sol` L282–290: `safeTransferFrom` → compute → `wrsETH.mint(msg.sender, 0)` — succeeds silently because OpenZeppelin `_mint(to, 0)` does not revert.
- `RSETHPoolV3ExternalBridge.sol` L401–409: identical pattern, same `wrsETH.mint(msg.sender, 0)`.
- `RSETHPoolNoWrapper.sol` L260–268: `rsETH.safeTransfer(msg.sender, 0)` — standard ERC-20 `transfer(to, 0)` returns `true` and does not revert.
- `RSETHPool.sol` L294–302: `IERC20(address(wrsETH)).safeTransfer(msg.sender, 0)` — same.

The only existing guard is `if (amount == 0) revert InvalidAmount()`, which checks the **input** amount, not the **output** amount. No minimum-output check exists anywhere in the deposit path.

The absorbed tokens sit in the pool's general balance. The bridging functions (`bridgeTokens`, `moveAssetsForBridging`) sweep the entire token balance minus fees to L1, making the loss permanent and unrecoverable.

## Impact Explanation

A user who deposits a sub-threshold amount of a supported low-decimal token receives zero rsETH while their tokens are permanently transferred to the protocol's L1 vault. This is a direct, unrecoverable loss of user funds triggered by a public, permissionless call. This matches **Critical — Direct theft of any user funds, whether at-rest or in-motion**: the contract accepts the user's tokens and delivers nothing in return, with no mechanism for the user to recover the deposited assets.

## Likelihood Explanation

Any unprivileged user can trigger this by calling `deposit(token, amount, referralId)` with an amount below the truncation threshold. For a 6-decimal token (e.g., USDC) with `tokenToETHRate ≈ 3.33e14` and `rsETHToETHrate ≈ 1.05e18`, the threshold is approximately 3,154 token units (~$0.003). For a 2-decimal token (e.g., GUSD), the threshold rises to ~$0.30. The condition is reachable without any special privilege, oracle manipulation, or governance action — it is a pure arithmetic property of the existing rate formula. The loss per transaction is small but real, permanent, and repeatable.

## Recommendation

Add a zero-output guard immediately after computing `rsETHAmount` in every `deposit(token, amount, referralId)` path across all four pool variants:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

This check must be inserted **after** the `viewSwapRsETHAmountAndFee` call and **before** `safeTransferFrom` (or, if the transfer order cannot be changed, before the mint/transfer-out step with a revert that also returns the tokens).

## Proof of Concept

**Preconditions:** `RSETHPoolV3` deployed with USDC (6 decimals) as a supported token; oracle reports `tokenToETHRate = 3.33e14`; `rsETHToETHrate = 1.05e18`; `feeBps = 0`.

**Call sequence:**
1. User approves pool for 3000 USDC units.
2. User calls `deposit(USDC, 3000, "")`.
3. Inside `viewSwapRsETHAmountAndFee(3000, USDC)`:
   - `fee = 0`
   - `rsETHAmount = 3000 * 3.33e14 / 1.05e18 = 9.99e17 / 1.05e18 = 0` (integer truncation)
4. `safeTransferFrom(user, pool, 3000)` — 3000 USDC units leave the user.
5. `wrsETH.mint(user, 0)` — no revert, user receives nothing.
6. Transaction succeeds; user's 3000 USDC units are in the pool.
7. Next `moveAssetsForBridging(USDC)` call sweeps the balance to L1; user's tokens are permanently lost.

**Foundry fuzz test plan:** Fuzz `amount` over `[1, rsETHToETHrate / tokenToETHRate - 1]` for each supported token; assert that after `deposit`, `wrsETH.balanceOf(user) > 0`. The assertion will fail for all amounts in the fuzz range, confirming the silent zero-mint path.
