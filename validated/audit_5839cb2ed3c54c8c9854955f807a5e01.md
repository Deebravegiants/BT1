### Title
Missing Chainlink Staleness Check and Zero/Negative Price Validation in `ChainlinkPriceOracle.getAssetPrice` - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

---

### Summary

`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but discards `updatedAt` entirely and casts the raw `int256 price` to `uint256` without any validity check. This is the direct analog of the reported vulnerability: no staleness enforcement and no zero/negative price guard.

---

### Finding Description

`getAssetPrice` in `ChainlinkPriceOracle` reads the Chainlink feed as follows:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The implementation captures only `price` (the `answer`); `updatedAt` and `answeredInRound` are silently discarded. There is no check of the form `require(updatedAt + heartbeat >= block.timestamp)` and no check that `price > 0`.

By contrast, the newer `ChainlinkOracleForRSETHPoolCollateral` wrapper used for pool collateral explicitly validates all three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

`ChainlinkPriceOracle` is the oracle registered in `LRTOracle` for the core LST assets (stETH, rETH, cbETH, etc.) and is the one consumed during every deposit and withdrawal. [3](#0-2) 

---

### Impact Explanation

**Stale price path (Medium — Temporary freezing / mis-accounting of funds):**
If a Chainlink feed stops updating (network congestion, sequencer downtime, oracle operator issue), `getAssetPrice` silently returns the last stale price. Deposits made during this window mint rsETH at an incorrect rate, causing share/asset mis-accounting for all depositors.

**Zero price path (Medium — Temporary freezing of funds):**
Chainlink can return `answer = 0` during circuit-breaker events. `uint256(0)` propagates through `getRsETHAmountToMint`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
// = (amount * 0) / rsETHPrice = 0
``` [4](#0-3) 

A depositor transfers real LST tokens into the pool and receives 0 rsETH — their assets are locked in the protocol with no corresponding shares.

**Negative price path (Critical — Protocol insolvency):**
If `price` is negative (possible in misconfigured or circuit-broken aggregators), `uint256(negative_int256)` wraps to a value near `type(uint256).max`. The resulting `getAssetPrice` return value is astronomically large, causing `getRsETHAmountToMint` to mint an unbounded amount of rsETH for a trivial deposit, draining the protocol.

---

### Likelihood Explanation

The stale-price and zero-price scenarios are realistic: Chainlink feeds for LST/ETH pairs have historically paused or returned boundary values during extreme market events. The `ChainlinkPriceOracle` is the live oracle for all core LST assets on mainnet (address `0x78C12ccE8346B936117655Dd3D70a2501Fd3d6e6`). Any unprivileged depositor calling `depositAsset` triggers the vulnerable path with no preconditions.

---

### Recommendation

1. Capture and validate `updatedAt` and `answeredInRound` from `latestRoundData`, using a per-asset configurable heartbeat (e.g., 3600 s for ETH/ETH feeds, 86400 s for slower feeds).
2. Revert if `price <= 0`.
3. Align `ChainlinkPriceOracle` with the validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral`.

---

### Proof of Concept

1. A Chainlink LST/ETH feed stops updating (e.g., stETH/ETH feed goes stale after its 86400 s heartbeat).
2. Any user calls `LRTDepositPool.depositAsset(stETH, amount, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint` → `LRTOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)`.
4. `latestRoundData()` returns the stale price; `updatedAt` is discarded with no check.
5. rsETH is minted at the stale (incorrect) rate, mis-accounting shares for all protocol participants.

For the zero-price variant: if `latestRoundData` returns `answer = 0`, the user's LST tokens are transferred in at line 114 of `LRTDepositPool.sol` but `rsethAmountToMint = 0`, so `_mintRsETH(0)` is called and the user receives nothing — their funds are frozen in the pool. [5](#0-4) [6](#0-5)
