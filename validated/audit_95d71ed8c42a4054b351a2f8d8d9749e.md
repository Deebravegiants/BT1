### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Combined with Root-Key Attestation Enables Cross-Chain Replay by Unprivileged Front-Runner — (File: `crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `verify_foreign_transaction` flow uses a pending-request map keyed on `VerifyForeignTransactionRequest`, a struct that contains no caller identity (`predecessor_account_id` is absent). Any unprivileged account can submit an identical request and be queued under the same map entry. When MPC nodes respond, every queued caller receives the same `VerifyForeignTransactionResponse`. Critically, that response carries a signature produced over the **root public key** (no caller-specific tweak), so the attestation is universally reusable. An attacker who front-runs a legitimate bridge call obtains a valid MPC-signed foreign-chain attestation they did not earn, and can replay it in any downstream bridge contract to trigger unauthorized execution.

---

### Finding Description

**Root cause 1 — caller-agnostic request key.**

`VerifyForeignTransactionRequest` is defined as:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

There is no `predecessor_account_id` field. The contract's own test acknowledges this explicitly:

> *"a different account would today be blocked from receiving a response by alice's submission"* — both yields are queued under the single **(caller-agnostic)** request key. [1](#0-0) [2](#0-1) 

**Root cause 2 — attestation signed with the root key, no caller tweak.**

In `respond_verify_foreign_tx`, the contract verifies the MPC signature against the **root public key** with no derivation tweak:

```rust
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // ← root key, no tweak
)
```

Compare this to `respond` for regular `sign` requests, which derives a caller-specific key via `derive_key_secp256k1(&affine, &request.tweak)`. The foreign-tx path deliberately omits the tweak, making the attestation a global, caller-independent proof. [3](#0-2) [4](#0-3) 

**Root cause 3 — signed payload contains no caller identity.**

`ForeignTxSignPayloadV1` encodes only the chain request and extracted values:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

No `predecessor_account_id`, no nonce, no per-caller binding. The 32-byte `msg_hash = SHA-256(borsh(ForeignTxSignPayload))` is therefore identical for every caller who submits the same chain request. [5](#0-4) 

**Fan-out delivery path.**

`push_pending_yield` appends every caller's `YieldIndex` to the same map entry. `resolve_yields_for` drains the entire queue on a single `respond_verify_foreign_tx` call, delivering the identical `VerifyForeignTransactionResponse` to every queued caller — including the attacker.
