### Title
Unchecked `DirectDeposits` Field Allows Arbitrary ADA Creation via Missing Preservation-of-Value Accounting - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `DirectDeposits` field in transaction bodies that credits arbitrary ADA amounts directly to registered staking account balances. However, neither `getConsumedDijkstraValue` nor `dijkstraProducedValue` (nor any sub-function they call) includes the `DirectDeposits` total in the preservation-of-value equation. An unprivileged transaction sender can therefore declare arbitrarily large `DirectDeposits` targeting their own registered account, pass the UTxO balance check, and have the ENTITIES rule credit those amounts to their account balance — creating ADA from nothing.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — declared amount credited without verifying actual ADA was consumed.

**Analog mapping:** In MonoX, `addLiquidityPair` minted pool tokens proportional to a *declared* `tokenAmount` without checking the actual balance delta before/after `safeTransferFrom`. The fix was to measure the real delta. In Cardano Ledger's Dijkstra era, `DirectDeposits` credits account balances with a *declared* coin amount without that amount appearing on either side of the `consumed == produced` equation.

**Root cause — preservation-of-value omission:**

`getConsumedDijkstraValue` delegates to `getConsumedMaryValue` for every tx body level:

```
getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  consumedValue <> MaryValue mempty mintedMultiAsset
  where
    mintedMultiAsset = filterMultiAsset (\_ _ -> (> 0)) $ txBody ^. mintTxBodyL
    consumedValue =
      sumUTxO (txInsFilter utxo (txBody ^. inputsTxBodyL))
        <> inject (refunds <> withdrawals)
```

`DirectDeposits` are absent. [1](#0-0) 

`dijkstraProducedValue` aggregates `conwayProducedValue` plus sub-transaction produced values:

```
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)
```

`DirectDeposits` are absent here too. [2](#0-1) 

`dijkstraSubTxProducedValue` (for sub-transactions) also omits `DirectDeposits`:

```
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody <> txBody ^. treasuryDonationTxBodyL)
    <> burnedMultiAssets txBody
``` [3](#0-2) 

**Application of DirectDeposits — ENTITIES rule:**

After the UTxO preservation-of-value check passes, the ENTITIES rule applies `DirectDeposits` unconditionally to account balances. The only guard is that target accounts are registered:

```
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts

pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [4](#0-3) 

`applyDirectDeposits` adds the declared coin to each account balance with no further checks:

```
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [5](#0-4) 

The same omission exists in the `SUBENTITIES` rule for sub-transactions: [6](#0-5)

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

The preservation-of-value invariant — the foundational accounting property of the Cardano ledger — is violated. An attacker can credit their own staking account with an arbitrary quantity of ADA without providing any corresponding ADA in transaction inputs. The credited balance can then be withdrawn in a subsequent transaction via the normal withdrawal mechanism, permanently inflating the ADA supply. This is equivalent to minting ADA, which the protocol explicitly forbids.

---

### Likelihood Explanation

**High.** The entry path requires only:
1. Registering a staking credential (standard, low-cost operation).
2. Constructing a valid Dijkstra-era transaction with a non-empty `directDepositsTxBodyL` targeting the attacker's own registered account.
3. Submitting the transaction.

No privileged access, governance majority, leaked key, or external dependency is required. The attack is deterministic and repeatable.

---

### Recommendation

Include the total `DirectDeposits` coin in the `produced` side of the preservation-of-value equation, analogous to how certificate deposits are included. Specifically, `dijkstraProducedValue`, `dijkstraSubTxProducedValue`, and any other produced-value aggregation function must add `fold (unDirectDeposits (txBody ^. directDepositsTxBodyL))` to the produced coin. This mirrors the fix applied to MonoX: measure the actual balance delta rather than trusting the declared amount.

---

### Proof of Concept

```
1. Register staking credential C (standard RegDepositTxCert, costs ppKeyDeposit).

2. Construct a Dijkstra TopTx T:
     inputs  = { some UTxO input worth F ADA }
     outputs = { change output worth 0 ADA }
     fee     = F
     directDepositsTxBodyL = DirectDeposits
       { accountAddressOf(C) → 1_000_000_000_000 }   -- 1 million ADA

3. Sign and submit T.

   UTxO check:  consumed = F,  produced = 0 + F + 0 = F  ✓  (DirectDeposits absent from both sides)
   ENTITIES:    applyDirectDeposits credits account(C) with 1_000_000_000_000 lovelace.

4. In a subsequent transaction, withdraw the full balance of account(C):
     withdrawalsTxBodyL = Withdrawals { accountAddressOf(C) → 1_000_000_000_000 }
   
   The withdrawal passes (balance matches declared amount) and the ADA enters the UTxO.

Net result: 1_000_000_000_000 lovelace created from nothing.
```
