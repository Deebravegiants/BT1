### Title
`instantWithdrawal` Can Drain Vault Assets Reserved for Queued Withdrawal Requests Due to Static `queuedWithdrawalsBuffer` Not Tracking `assetsCommitted` - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

The `instantWithdrawal` function in `LRTWithdrawalManager` checks available vault assets using `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal`, which relies on a static `queuedWithdrawalsBuffer` set by an operator. This buffer is not automatically synchronized with `assetsCommitted`, the dynamically tracked amount committed to pending queued withdrawal requests. As a result, any rsETH holder can call `instantWithdrawal` to drain vault assets that are needed to service queued withdrawal requests, causing temporary freezing of funds for queued withdrawal users.

---

### Finding Description

`LRTWithdrawalManager` maintains `assetsCommitted[asset]` to track the total amount of assets committed to pending (locked) withdrawal requests.

When a user calls `initiateWithdrawal`, `assetsCommitted` is incremented:

```solidity
assetsCommitted[asset] += expectedAssetAmount;
``` [1](#0-0) 

When `_unlockWithdrawalRequests` processes requests, `assetsCommitted` is decremented:

```solidity
assetsCommitted[asset] -= request.expectedAssetAmount;
``` [2](#0-1) 

The `instantWithdrawal` function, however, does not interact with `assetsCommitted` at all. Instead, it checks `unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)`:

```solidity
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
unstakingVault.redeem(asset, assetAmountUnlocked);
``` [3](#0-2) 

`getAssetsAvailableForInstantWithdrawal` returns `vaultBalance - queuedWithdrawalsBuffer[asset]`:

```solidity
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
``` [4](#0-3) 

The `queuedWithdrawalsBuffer` is a static value set by an operator and is **not** automatically synchronized with `assetsCommitted`:

```solidity
function setQueuedWithdrawalsBuffer(address asset, uint256 buffer) external onlyLRTOperator ...
    queuedWithdrawalsBuffer[asset] = buffer;
``` [5](#0-4) 

This creates a structural desync: `assetsCommitted` grows dynamically as users queue withdrawal requests, but `queuedWithdrawalsBuffer` is a static snapshot. The default value of `queuedWithdrawalsBuffer` is `0`, meaning by default there is zero protection for queued withdrawal assets.

When `unlockQueue` is called, it uses the raw vault balance as `totalAvailableAssets`:

```solidity
totalAvailableAssets: unstakingVault.balanceOf(asset)
``` [6](#0-5) 

If instant withdrawals have drained the vault, `unlockQueue` will find `params.totalAvailableAssets == 0` and revert, or `_unlockWithdrawalRequests` will break immediately because `availableAssetAmount < payoutAmount`:

```solidity
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
``` [7](#0-6) 

```solidity
if (availableAssetAmount < payoutAmount) break;
``` [8](#0-7) 

---

### Impact Explanation

When instant withdrawals drain vault assets needed for queued requests, `unlockQueue` cannot unlock queued withdrawal requests. Users who called `initiateWithdrawal` and are waiting for their requests to be unlocked will have their funds temporarily frozen. Their rsETH has already been transferred to the withdrawal manager contract, and they cannot recover it until more assets are unstaked from EigenLayer and deposited into the vault. This is a **temporary freezing of funds** for an unbounded set of queued withdrawal users.

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

- `instantWithdrawal` is gated by `isInstantWithdrawalEnabled[asset]`, which must be enabled by the manager. Once enabled, **any rsETH holder** can call it.
- `queuedWithdrawalsBuffer` defaults to `0`. Even if set correctly at one point, it becomes stale as `assetsCommitted` grows with new queued requests.
- The desync is structural: `assetsCommitted` is in `LRTWithdrawalManager` and `queuedWithdrawalsBuffer` is in `LRTUnstakingVault`, with no on-chain link between them.
- No special role or privilege is required beyond holding rsETH.

**Likelihood: Medium.**

---

### Recommendation

The available-for-instant-withdrawal calculation must account for `assetsCommitted` rather than relying on a static `queuedWithdrawalsBuffer`. One approach is to compute the available amount directly inside `LRTWithdrawalManager.instantWithdrawal`:

```solidity
uint256 vaultBalance = unstakingVault.balanceOf(asset);
uint256 reservedForQueued = assetsCommitted[asset];
uint256 availableForInstant = vaultBalance > reservedForQueued
    ? vaultBalance - reservedForQueued
    : 0;
if (assetAmountUnlocked > availableForInstant) revert CantInstantWithdrawMoreThanAvailable();
```

This ensures instant withdrawals can never drain assets that are already committed to pending queued withdrawal requests.

---

### Proof of Concept

1. The `LRTUnstakingVault` holds 100 ETH (unstaked from EigenLayer, ready for distribution).
2. Multiple users have called `initiateWithdrawal`, so `assetsCommitted[ETH] = 80 ETH`.
3. `queuedWithdrawalsBuffer[ETH] = 0` (default, or stale from a prior operator setting).
4. Manager has enabled instant withdrawals: `isInstantWithdrawalEnabled[ETH] = true`.
5. Attacker holds rsETH equivalent to 100 ETH and calls `instantWithdrawal(ETH, rsETHAmount, "")`.
6. `getAssetsAvailableForInstantWithdrawal(ETH)` returns `100 - 0 = 100 ETH` — no protection for the 80 ETH committed to queued requests.
7. `unstakingVault.redeem(ETH, 100 ETH)` drains the vault to 0.
8. Operator calls `unlockQueue(ETH, ...)` — `params.totalAvailableAssets = 0`, reverts with `AmountMustBeGreaterThanZero`.
9. The 80 ETH worth of queued withdrawal requests cannot be unlocked. Users who transferred their rsETH to the withdrawal manager are stuck until new assets arrive in the vault from EigenLayer unstaking. [9](#0-8) [10](#0-9) [11](#0-10)
