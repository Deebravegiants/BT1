### Title
Deposit Refund Sent to Sender (Alice) Instead of Relayer When Meta-Transaction Inner Action Fails — (File: runtime/runtime/src/actions.rs)

---

### Summary

In `apply_delegate_action`, the inner receipt spawned from a `DelegateAction` (meta-transaction) is constructed with `predecessor_id = sender_id` (Alice). When that inner receipt fails on the receiver's shard, the protocol's deposit-refund mechanism sends the full attached deposit back to `predecessor_id` — Alice — not to the relayer who actually funded the deposit. An unprivileged user can exploit this to drain a relayer's balance by crafting meta-transactions with large deposits that are guaranteed to fail.

---

### Finding Description

`apply_delegate_action` in `runtime/runtime/src/actions.rs` constructs the forwarded receipt as a `Receipt::V0` with `ReceiptEnum::Action`:

```rust
let new_receipt = Receipt::V0(ReceiptV0 {
    predecessor_id: sender_id.clone(),          // Alice
    receiver_id: delegate_action.receiver_id(), // Bob
    receipt: ReceiptEnum::Action(ActionReceipt {
        signer_id: action_receipt.signer_id(),  // Relayer
        ...
        actions: delegate_action.get_actions(), // may carry deposit
    }),
});
``` [1](#0-0) 

When this receipt fails on Bob's shard, `refund_unspent_gas_and_deposits` computes `deposit_refund = total_deposit` and calls:

```rust
result.new_receipts.push(Receipt::new_balance_refund(
    receipt.balance_refund_receiver(),
    deposit_refund,
));
``` [2](#0-1) 

`balance_refund_receiver()` is:

```rust
pub fn balance_refund_receiver(&self) -> &AccountId {
    self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
}
``` [3](#0-2) 

`refund_to()` returns `&None` for `ReceiptEnum::Action` (only `ActionV2` carries a `refund_to` field):

```rust
ReceiptEnum::Action(_) | ... => &None,
ReceiptEnum::ActionV2(r) | ReceiptEnum::PromiseYieldV2(r) => &r.refund_to,
``` [4](#0-3) 

So the refund always falls back to `predecessor_id` = Alice. The relayer, who is `signer_id` and who paid for the deposit when the transaction was processed, receives only the gas refund — not the deposit refund.

The code comment at lines 471–475 explicitly acknowledges this asymmetry:

> "If something goes wrong, deposit is refunded to the predecessor, this is sender_id/Sender in DelegateAction. Gas is refunded to the signer, this is Relayer. Therefore Relayer should verify DelegateAction before submitting it because it spends the attached deposit." [5](#0-4) 

The architecture documentation confirms the exploitability: "there is a financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard." [6](#0-5) 

---

### Impact Explanation

The relayer's account balance is debited for the full deposit when the transaction is converted to a receipt. On failure, that deposit is credited to Alice's account, not the relayer's. The corrupted protocol value is the **account balance** of both the relayer (incorrectly reduced) and Alice (incorrectly increased) by the full deposit amount. There is no cap on the deposit size in a `DelegateAction`, so the loss is unbounded per transaction.

---

### Likelihood Explanation

Any unprivileged NEAR account (Alice) can sign a `DelegateAction` containing a `FunctionCall` with an arbitrarily large `deposit` targeting a method that does not exist on the receiver contract. The relayer cannot reliably detect this before submission because: (a) the receiver contract may not be inspectable, (b) the method may be conditionally absent, or (c) the receiver account may not exist. The attack requires only a valid signed `DelegateAction` submitted through the normal public RPC path.

---

### Recommendation

When constructing the forwarded receipt in `apply_delegate_action`, use `ReceiptEnum::ActionV2` and set `refund_to: Some(action_receipt.signer_id().clone())` so that on failure the deposit is returned to the relayer (the party who funded it), mirroring how gas refunds already correctly target `signer_id`.

---

### Proof of Concept

1. Alice signs a `DelegateAction` with `receiver_id = bob.near`, actions = `[FunctionCall { method_name: "nonexistent", deposit: 10_000 NEAR, ... }]`.
2. Relayer submits the meta-transaction. The relayer's balance is debited `10_000 NEAR` (plus gas) when the transaction is processed on Alice's shard.
3. On Bob's shard, the function call fails (method does not exist). `refund_unspent_gas_and_deposits` fires with `deposit_refund = 10_000 NEAR`.
4. `balance_refund_receiver()` returns `predecessor_id` = Alice.
5. A `Receipt::new_balance_refund(&alice, 10_000 NEAR)` is emitted. Alice's balance increases by `10_000 NEAR`; the relayer's balance is never restored.
