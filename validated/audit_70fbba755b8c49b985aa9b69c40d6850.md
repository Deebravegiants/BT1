Audit Report

## Title
Token Decimal Precision Not Normalized in `viewSwapRsETHAmountAndFee` Causes Drastically Undercalculated rsETH Minted for Non-18-Decimal Deposits - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The token-deposit overload of `viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18-decimal precision. Because `ChainlinkOracleForRSETHPoolCollateral.getRate()` always returns a 1e18-precision rate, any token with fewer than 18 decimals produces an rsETH output that is `10^(18 - tokenDecimals)` times smaller than correct. A depositor of a 6-decimal token (e.g., USDC) receives approximately 1e12 times less rsETH than owed, constituting a near-total permanent loss of deposited principal.

## Finding Description
The ETH-deposit overload correctly normalizes by multiplying by `1e18`: [1](#0-0) 

The token-deposit overload does not apply any decimal normalization: [2](#0-1) 

The oracle wrapper always returns a 1e18-precision value regardless of the underlying Chainlink feed's native decimals: [3](#0-2) 

So `tokenToETHRate` is always 1e18-scaled, but `amountAfterFee` is in the token's native decimals. For a 6-decimal token, the formula silently treats `1_000e6` (1000 USDC) as if it were `1_000e6` in 1e18 scale, producing a result ~1e12 times too small.

The same missing normalization is present identically in all five pool contracts: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

The `addSupportedToken` function performs no decimal check on the token being added: [8](#0-7) 

The deposit function transfers the full token amount from the user and mints the undercalculated rsETH amount with no recourse: [9](#0-8) 

## Impact Explanation
**Critical — Direct theft of user funds.** Once a non-18-decimal token is added as supported, any user calling `deposit(token, amount, referralId)` transfers their full token balance to the pool (bridged to L1 and permanently absorbed), but receives rsETH worth `10^(18 - tokenDecimals)` times less than fair value. For USDC (6 decimals), the shortfall factor is 1e12 — effectively 100% loss of deposited principal. The user holds insufficient rsETH to redeem anything meaningful. This matches the allowed impact: **Critical — Direct theft of any user funds in motion.** [9](#0-8) 

## Likelihood Explanation
The currently deployed supported token is wstETH (18 decimals), for which the formula is coincidentally correct. However, the `addSupportedToken` function accepts any ERC-20 and oracle pair with no decimal guard. The protocol's architecture explicitly anticipates future token expansion (multiple `reinitialize` versions already add tokens). Any legitimate governance action to add a common asset with fewer than 18 decimals (USDC, USDT, WBTC) immediately activates the bug for every subsequent depositor of that token. The admin action is a normal governance operation, not an attack — the loss is suffered by unprivileged users through the public `deposit` function. Likelihood is **Medium**. [8](#0-7) 

## Recommendation
Normalize `amountAfterFee` to 18-decimal precision before applying the rate ratio in all five pool contracts:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 amountNormalized = amountAfterFee * 1e18 / (10 ** tokenDecimals);
rsETHAmount = amountNormalized * tokenToETHRate / rsETHToETHrate;
```

Alternatively, enforce in `addSupportedToken` that only 18-decimal tokens are accepted, though this limits future extensibility.

## Proof of Concept
1. Admin calls `addSupportedToken(usdc, chainlinkUsdcEthOracle)` — no decimal check prevents this.
2. User calls `RSETHPoolV3.deposit(usdc, 1_000e6, "")` — depositing 1000 USDC (~0.4 ETH at $2500/ETH, fair rsETH ≈ 3.8e17).
3. `viewSwapRsETHAmountAndFee` computes:
   - `tokenToETHRate = 4e14` (oracle normalized to 1e18)
   - `rsETHToETHrate = 1.05e18`
   - `rsETHAmount = 1_000e6 * 4e14 / 1.05e18 ≈ 380_952`
4. User receives `380_952` wrsETH (≈ 3.8e-13 ETH value) instead of `≈ 3.8e17` wrsETH.
5. The 1000 USDC is bridged to L1 and absorbed. Loss is ~100% of deposited value.

**Foundry fork test plan:** Fork a network where USDC is available; deploy `ChainlinkOracleForRSETHPoolCollateral` pointing to the USDC/ETH feed; call `addSupportedToken`; impersonate a USDC holder; call `deposit`; assert `rsETHAmount < 1e6` while correct value is `> 3e17`.
