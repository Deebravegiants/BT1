Audit Report

## Title
Large `initiateWithdrawal` Commitment Blocks Other Users From Initiating Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`initiateWithdrawal` gates new withdrawal requests against `getAvailableAssetAmount`, which counts illiquid EigenLayer-staked assets as available capacity. A whale can commit to withdrawing nearly all protocol assets, inflating `assetsCommitted[asset]` and causing every subsequent `initiateWithdrawal` call to revert with `ExceedAmountToWithdraw`. The commitment is only released when `_unlockWithdrawalRequests` processes the request, which itself requires assets to be physically present in `LRTUnstakingVault` — a condition that cannot be met until after EigenLayer's 7-day unbonding period plus operator processing time.

## Finding Description
`initiateWithdrawal` enforces a capacity check before recording a new request:

```solidity
// LRTWithdrawalManager.sol:168-173
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` computes capacity using `getTotalAssetDeposits`:

```solidity
// LRTWithdrawalManager.sol:599-603
uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
```

`getTotalAssetDeposits` includes `assetStakedInEigenLayer + assetUnstakingFromEigenLayer` in its total:

```solidity
// LRTDepositPool.sol:394-396
uint256 effectiveAssetWithEigenLayer = assetStakedInEigenLayer + assetUnstakingFromEigenLayer;
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer + assetLyingInConverter + assetLyingUnstakingVault);
```

However, `_unlockWithdrawalRequests` can only process requests using assets physically present in `LRTUnstakingVault`, sourced via `_createUnlockParams`:

```solidity
// LRTWithdrawalManager.sol:849
totalAvailableAssets: unstakingVault.balanceOf(asset)
```

And `assetsCommitted` is only decremented inside `_unlockWithdrawalRequests`:

```solidity
// LRTWithdrawalManager.sol:802
assetsCommitted[asset] -= request.expectedAssetAmount;
```

The root cause is the mismatch between the two pools: `getAvailableAssetAmount` counts illiquid EigenLayer assets as available for accepting new commitments, but `_unlockWithdrawalRequests` can only consume assets already in `LRTUnstakingVault`. A whale commits against EigenLayer-locked assets, saturating `assetsCommitted`, while the decrement path is gated behind a 7-day unbonding cycle that the whale's own request must complete first.

## Impact Explanation
**Medium — Temporary freezing of funds.** Any rsETH holder attempting `initiateWithdrawal` after the whale's commitment will receive `ExceedAmountToWithdraw` and be unable to enter the withdrawal queue. Their rsETH remains inaccessible with no redemption path for a minimum of 7 days (EigenLayer unbonding) plus operator processing time. This directly matches the allowed impact "Medium. Temporary freezing of funds."

## Likelihood Explanation
`initiateWithdrawal` is publicly callable by any rsETH holder with no role restriction. Since EigenLayer assets typically represent the majority of protocol TVL, a single large holder (whale, protocol treasury, or DeFi vault) can trigger this condition unintentionally or deliberately by committing to a large fraction of total assets. The condition is straightforward to reach and the blocking window is structurally long due to EigenLayer's unbonding period.

## Recommendation
Decouple withdrawal commitment capacity from illiquid EigenLayer assets. `getAvailableAssetAmount` should only count assets realistically redeemable within the withdrawal delay window — i.e., assets in `LRTDepositPool`, NDCs, and `LRTUnstakingVault`. EigenLayer assets should not count toward available capacity until they have landed in `LRTUnstakingVault`. A minimal fix is to replace `lrtDepositPool.getTotalAssetDeposits(asset)` in `getAvailableAssetAmount` with a sum of only liquid asset locations.

## Proof of Concept
1. Protocol state: 10,000 stETH total, 9,500 stETH staked in EigenLayer, 500 stETH in `LRTUnstakingVault`. `assetsCommitted[stETH] = 0`.
2. `getAvailableAssetAmount(stETH)` returns `10,000 - 0 = 10,000`.
3. Whale calls `initiateWithdrawal(stETH, rsETHAmount)` where `rsETHAmount` corresponds to 9,800 stETH. Check passes; `assetsCommitted[stETH] = 9,800`.
4. `getAvailableAssetAmount(stETH)` now returns `10,000 - 9,800 = 200`.
5. Alice calls `initiateWithdrawal(stETH, rsETHAmount)` for 300 stETH. `300 > 200` → `ExceedAmountToWithdraw`. Alice is blocked.
6. Operator must queue 9,300 stETH from EigenLayer, wait 7 days, complete withdrawal, transfer to vault, then call `unlockQueue`. Only after this does `assetsCommitted` decrease and Alice can retry.

**Foundry test plan:** Deploy protocol with mock EigenLayer strategy holding 9,500 stETH. Call `initiateWithdrawal` as whale for 9,800 stETH equivalent. Assert `assetsCommitted[stETH] == 9800e18`. Call `initiateWithdrawal` as Alice for 300 stETH equivalent. Assert revert with `ExceedAmountToWithdraw`. Advance blocks past `withdrawalDelayBlocks` but without transferring assets to vault. Assert `unlockQueue` cannot process whale's request (vault balance insufficient). Confirm Alice remains blocked.
