Audit Report

## Title
ETH Permanently Frozen in LRTWithdrawalManager When Recipient Contract Rejects ETH After rsETH Burn - (File: contracts/LRTWithdrawalManager.sol)

## Summary

The withdrawal lifecycle splits rsETH burning (Phase 1, `unlockQueue`) and ETH delivery (Phase 2, `completeWithdrawal`) across two separate, independent transactions. Once `unlockQueue` irreversibly burns the user's rsETH and pulls ETH into `LRTWithdrawalManager`, any subsequent `completeWithdrawal` call that fails because the recipient contract rejects ETH will permanently strand the ETH in the contract with no on-chain recovery path.

## Finding Description

**Phase 1 — `unlockQueue` (irreversible):**

`unlockQueue` calls `_unlockWithdrawalRequests`, which advances `nextLockedNonce[asset]` and finalizes `request.expectedAssetAmount`. It then burns all accumulated rsETH and pulls the corresponding ETH from `LRTUnstakingVault` into `LRTWithdrawalManager` in the same transaction:

```solidity
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);
``` [1](#0-0) 

After this transaction commits, the rsETH is permanently destroyed and the ETH sits in `LRTWithdrawalManager`.

**Phase 2 — `_processWithdrawalCompletion` (can permanently revert):**

`_processWithdrawalCompletion` pops the user's nonce, deletes the request, decrements `unlockedWithdrawalsCount`, and then calls `_transferAsset`:

```solidity
uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
...
delete withdrawalRequests[requestId];
...
unlockedWithdrawalsCount[asset]--;
...
_transferAsset(asset, user, request.expectedAssetAmount);
``` [2](#0-1) 

`_transferAsset` for ETH uses a low-level call:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
``` [3](#0-2) 

If `to` is a contract that reverts on ETH receipt, `EthTransferFailed` is thrown, causing the entire Phase 2 transaction to revert. Solidity's revert semantics restore all Phase 2 state changes (nonce re-pushed, request restored, count restored), but **Phase 1 is already committed on a prior block and is unaffected**. The rsETH is gone and the ETH is stranded.

**No recovery path exists:**

- `sweepRemainingAssets` requires `hasUnlockedWithdrawals(asset) == false`, i.e., `unlockedWithdrawalsCount[asset] == 0`. Since Phase 2 always reverts for this user, the count is never decremented, permanently blocking the sweep. [4](#0-3) 

- `completeWithdrawalForUser` (operator-callable) calls the identical `_processWithdrawalCompletion` and fails the same way. [5](#0-4) 

- There is no function to update the recipient address on an existing request, cancel a request, or redirect ETH to an alternative address.

**FIFO queue amplifies impact:**

`popFront()` always processes the oldest request first. A permanently undeliverable oldest request blocks all of that user's newer requests in the same per-asset deque. [6](#0-5) 

## Impact Explanation

**Critical — Permanent freezing of funds.** The affected user loses their rsETH (burned in Phase 1, irreversible) and cannot receive the corresponding ETH (permanently stuck in `LRTWithdrawalManager`). All of the user's subsequent withdrawal requests in the same per-user deque are also blocked. Without a contract upgrade, there is no on-chain path to recover the ETH.

## Likelihood Explanation

**Low-Medium.** The triggering condition requires a contract address to have initiated a withdrawal and for that contract to permanently reject ETH at completion time. Realistic scenarios include: a smart-contract wallet (Gnosis Safe, ERC-4337 account) upgraded after `initiateWithdrawal` to reject ETH; a contract whose `receive()` or `fallback()` unconditionally reverts; or a contract that was self-destructed after initiating the withdrawal. Smart-contract wallets are common among DeFi power users who are the most likely holders of rsETH. No privileged access is required — any contract that holds rsETH and calls `initiateWithdrawal` can trigger this path.

## Recommendation

1. **Pull-payment pattern**: Do not deliver ETH in `_processWithdrawalCompletion`. Instead, credit the amount to a `claimable[user]` mapping and let users pull it via a separate `claimETH()` function. This decouples state mutation from transfer.
2. **Allow recipient update**: Add a function permitting the original withdrawer (authenticated by `msg.sender`) to update the destination address for their pending unlocked request.
3. **Operator rescue path**: Allow the operator to mark a permanently undeliverable request as failed, credit the ETH to a claimable mapping, and remove it from the FIFO queue so subsequent requests are not blocked.

## Proof of Concept

1. Alice (a smart-contract wallet) calls `initiateWithdrawal(ETH_TOKEN, 1e18, "")`. rsETH is transferred to `LRTWithdrawalManager`. Request stored with `expectedAssetAmount = X ETH`.
2. Alice's wallet is upgraded; its new implementation reverts on all ETH receives (or Alice's contract always had a reverting fallback).
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. `_unlockWithdrawalRequests` advances `nextLockedNonce`, sets `request.expectedAssetAmount = X ETH`, burns `1e18` rsETH from the contract, and pulls `X ETH` from `LRTUnstakingVault` into `LRTWithdrawalManager`. rsETH is now permanently destroyed.
4. Alice calls `completeWithdrawal(ETH_TOKEN, "")`. `_processWithdrawalCompletion` pops Alice's nonce, deletes the request, decrements `unlockedWithdrawalsCount`, then calls `_transferAsset(ETH_TOKEN, alice, X)`. The low-level call to Alice's contract reverts. `EthTransferFailed` is thrown. The entire transaction reverts — state is restored.
5. Step 4 can be repeated infinitely; it always reverts. `X ETH` is permanently frozen in `LRTWithdrawalManager`. Alice has lost `1e18` rsETH with no recourse.
6. Foundry fork test: deploy a `RejectETH` contract with a reverting `receive()`, fund it with rsETH, call `initiateWithdrawal`, advance blocks past `withdrawalDelayBlocks`, call `unlockQueue`, then assert `completeWithdrawal` always reverts with `EthTransferFailed` while the contract's ETH balance remains unchanged and `unlockedWithdrawalsCount[ETH_TOKEN] > 0` indefinitely.
