Audit Report

## Title
`LRTDepositPool.getAssetCurrentLimit` Ignores `RSETH` Daily Mint Cap, Returning Misleading Deposit Availability - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getAssetCurrentLimit()` reports remaining deposit capacity based solely on the per-asset deposit cap in `LRTConfig`, without consulting `RSETH.maxMintAmountPerDay`. When the daily rsETH mint cap is exhausted, `getAssetCurrentLimit()` returns a non-zero value while any subsequent `depositAsset()` or `depositETH()` call will revert inside `RSETH.mint()`, causing users to waste gas on transactions that cannot succeed.

## Finding Description
`getAssetCurrentLimit()` at `contracts/LRTDepositPool.sol:402-409` computes remaining capacity exclusively against `lrtConfig.depositLimitByAsset(asset)`:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) {
        return 0;
    }
    return lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;
}
```

The deposit execution path calls `_mintRsETH()` at `contracts/LRTDepositPool.sol:686-690`, which calls `IRSETH(rsethToken).mint()`. That function applies the `checkDailyMintLimit` modifier at `contracts/RSETH.sol:42-56`, which independently tracks `currentPeriodMintedAmount` and reverts with `DailyMintLimitExceeded` if the daily cap is exceeded. `RSETH` already exposes `remainingDailyMintLimit()` at `contracts/RSETH.sol:265-272` as a public view, but `getAssetCurrentLimit()` never calls it. The two limits are entirely independent state variables with no cross-reference.

The `_beforeDeposit` check at `contracts/LRTDepositPool.sol:661-663` only calls `_checkIfDepositAmountExceedesCurrentLimit`, which mirrors the same per-asset-only logic and also does not consult the daily mint cap. No existing guard bridges the two limits before the revert occurs inside `RSETH.mint()`.

## Impact Explanation
A user or integrator calls `getAssetCurrentLimit(asset)`, receives a positive value, and submits a deposit transaction. If `currentPeriodMintedAmount == maxMintAmountPerDay`, the transaction reverts with `DailyMintLimitExceeded` after passing all `_beforeDeposit` checks. No user funds are lost, but the contract fails to deliver the deposit it appeared to promise. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`maxMintAmountPerDay` is an active operational parameter set by `onlyLRTManager` via `setMaxMintAmountPerDay()` at `contracts/RSETH.sol:125-128`. During high-demand periods the daily cap can be exhausted while the per-asset deposit limit still has significant remaining room. Any depositor or on-chain integrator using `getAssetCurrentLimit()` as a pre-flight check — a standard and documented pattern — will receive a false positive. The condition is repeatable every day the cap is hit before the per-asset limit.

## Recommendation
Update `getAssetCurrentLimit()` to return the minimum of the per-asset deposit room and the asset-equivalent of `RSETH.remainingDailyMintLimit()`. Convert the remaining rsETH mint allowance to asset units using the existing oracle (e.g., `ILRTOracle.rsETHPrice()` and `ILRTOracle.getAssetPrice(asset)`). When `maxMintAmountPerDay == 0` (no daily cap configured), the function should treat the daily mint room as unlimited and fall back to the per-asset limit only.

## Proof of Concept
1. `RSETH.maxMintAmountPerDay` is set to `1000e18`. The current period has already minted `1000e18` rsETH (`currentPeriodMintedAmount == maxMintAmountPerDay`).
2. The per-asset deposit limit for stETH is `10000e18`; only `5000e18` has been deposited. `getAssetCurrentLimit(stETH)` returns `5000e18`.
3. A user calls `getAssetCurrentLimit(stETH)`, sees `5000e18`, and submits `depositAsset(stETH, 1e18, 0, "")`.
4. `_beforeDeposit` passes: `_checkIfDepositAmountExceedesCurrentLimit` returns false (per-asset limit not exceeded).
5. `_mintRsETH` calls `RSETH.mint()`; `checkDailyMintLimit` evaluates `1000e18 + rsethToMint > 1000e18` → true → reverts with `DailyMintLimitExceeded`.
6. Transaction reverts. User loses gas. `getAssetCurrentLimit` gave a false positive.
