### Title
Mid-Run Parameter Update Allows Treasury Withdrawal Under Lowered Threshold — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

### Summary

`reorderActions` sorts proposals by priority so `ParameterChange` (priority 4) is always processed before `TreasuryWithdrawals` (priority 5). When `ParameterChange` is enacted mid-run, `ratifyTransition` immediately propagates the updated `ensCurPParams` into the recursive `RatifyState`. Subsequent proposals in the same RATIFY run are then evaluated against the new parameters. An attacker who can gather enough DRep votes to pass a `ParameterChange` that lowers `dvtTreasuryWithdrawal` can pair it with a `TreasuryWithdrawals` proposal in the same epoch and have the withdrawal ratified under the new, lower threshold — bypassing the threshold that was in effect at the start of the epoch.

### Finding Description

**Step 1 — Priority ordering is deterministic and attacker-controlled.**

`reorderActions` unconditionally sorts the signal by `actionPriority`: [1](#0-0) 

`ParameterChange` = 4, `TreasuryWithdrawals` = 5, so `ParameterChange` is always first.

**Step 2 — `ParameterChange` is not a delaying action.** [2](#0-1) 

`delayingAction ParameterChange {} = False`, so `rsDelayed` remains `False` after enacting it. The `not rsDelayed` guard on line 339 does not block subsequent proposals.

**Step 3 — ENACT immediately mutates `ensCurPParams` in the returned state.** [3](#0-2) 

`applyPPUpdates` writes the attacker's new (lower) `dvtTreasuryWithdrawal` into `ensCurPParams`.

**Step 4 — The updated state is passed directly into the recursive RATIFY call.** [4](#0-3) 

`st'` carries `rsEnactStateL .~ newEnactState` (the post-`ParameterChange` params) and is the state used for all remaining proposals in the same run.

**Step 5 — `dRepAccepted` reads the threshold from the live `RatifyState`, not a snapshot.** [5](#0-4) 

`votingDRepThreshold rs govAction` reads `rs ^. rsEnactStateL . ensCurPParamsL`: [6](#0-5) 

When `rs` is `st'` (post-enactment), `dvtTreasuryWithdrawal` is already the attacker's lowered value.

**Step 6 — `TreasuryWithdrawals` has no `prevActionAsExpected` parent requirement.**

`withGovActionParent` returns the default `True` for `TreasuryWithdrawals`, so no ancestry chain check blocks it: [7](#0-6) 

### Impact Explanation

**Critical — Unauthorized treasury action enacted outside design parameters.**

The treasury withdrawal is ratified using a threshold that was not in effect at the start of the epoch boundary. An attacker who controls enough DRep stake to pass a `ParameterChange` (governance-group threshold) but not enough to pass a `TreasuryWithdrawals` under the original `dvtTreasuryWithdrawal` can exploit the ordering to extract treasury funds that the protocol, as configured at epoch start, would have rejected.

### Likelihood Explanation

The preconditions are reachable by any DRep coalition that meets the `ParameterChange` governance-group threshold:

- Submit `ParameterChange` lowering `dvtTreasuryWithdrawal` with the correct `prevGovActionId` (public on-chain state).
- Submit `TreasuryWithdrawals` in the same epoch with votes sufficient only under the new threshold.
- Both proposals are processed in a single `runConwayRatify` call at the epoch boundary.

No privileged key, supermajority, or external dependency is required beyond normal DRep participation.

### Recommendation

Snapshot `ensCurPParams` (or specifically the voting thresholds) at the **start** of each `runConwayRatify` invocation and use that snapshot for all `acceptedByEveryone` checks throughout the run, regardless of intermediate enactments. The `RatifyEnv` is the natural place to carry this immutable snapshot, since it is fixed for the entire run, unlike `RatifyState` which is intentionally mutable.

### Proof of Concept

```
Epoch N:
  tx1: submitGovAction (ParameterChange SNothing {dvtTreasuryWithdrawal: 0.51 -> 0.30} SNothing)
       -- prevGovActionId = current ensPrevPParamUpdateL (public)
  tx2: submitGovAction (TreasuryWithdrawals {addr: 1_000_000 ADA} SNothing)

  DRep votes: cast Yes on both proposals
  -- DRep stake ratio = 0.40 (above 0.30, below 0.51)

Epoch boundary (N -> N+1):
  reorderActions: [ParameterChange(4), TreasuryWithdrawals(5)]
  ratifyTransition:
    1. ParameterChange: acceptedByEveryone (0.40 >= original threshold for pparam governance group)
       -> ENACT: ensCurPParams.dvtTreasuryWithdrawal := 0.30
       -> st' carries new ensCurPParams
    2. TreasuryWithdrawals: dRepAccepted reads dvtTreasuryWithdrawal = 0.30 from st'
       -> 0.40 >= 0.30 => True => ENACTED
       -- Would have failed under original 0.51 threshold

Result: 1_000_000 ADA withdrawn from treasury with only 40% DRep approval,
        below the 51% threshold that was in effect at epoch start.
```
