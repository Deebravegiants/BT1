### Title
Missing Output Validation for `AppPublicKey` CKD Variant Allows Single Byzantine Participant to Substitute Arbitrary Response — (`crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` function in the chain-signature contract performs cryptographic output validation only for the `AppPublicKeyPV` variant of a CKD request. The `AppPublicKey` variant has an empty match arm with no validation whatsoever. Any single attested participant — below the signing threshold — can race the honest leader and resolve a pending CKD yield with an attacker-chosen `(big_y, big_c)` pair. The user receives fraudulent key material; the honest leader's subsequent call returns `RequestNotFound`.

### Finding Description

In `respond_ckd`, after confirming the caller is an attested participant, the contract branches on the request variant: [1](#0-0) 

The `AppPublicKeyPV` arm calls `ckd_output_check`, which performs a pairing-based cryptographic check binding the response to the MPC public key and the request parameters. The `AppPublicKey` arm is empty — no check of any kind is performed on `big_y` or `big_c`.

The only gate before this match is `assert_caller_is_attested_participant_and_protocol_active`: [2](#0-1) 

This confirms the caller is an attested participant, but it does **not** enforce that the caller is the designated leader for this request, nor does it require threshold agreement. A single attested participant passes this check.

After the (empty) match arm, `resolve_yields_for` is called unconditionally: [3](#0-2) 

`resolve_yields_for` resolves the pending yield with whatever bytes are serialized from `response`. The first caller wins; the request is removed from `pending_ckd_requests`. Any subsequent call — including the honest leader's — returns `RequestNotFound`.

The existing contract test for the `AppPublicKey` variant confirms the contract accepts all-arbitrary bytes (`[1u8; 48]`, `[2u8; 48]`) without complaint: [4](#0-3) 

### Impact Explanation

The user's pending yield is resolved with attacker-controlled `(big_y, big_c)`. For the `AppPublicKey` variant, the user has no independent means to verify the response (unlike `AppPublicKeyPV`, which is publicly verifiable via pairing). The user receives key material that does not correspond to `msk * H(pk, app_id)`. This breaks the core safety invariant: only the threshold-computed output should be delivered. A single below-threshold Byzantine participant can unilaterally corrupt every `AppPublicKey` CKD response it races.

**Impact class:** Critical — unauthorized confidential key derivation output without the required participant authorization.

### Likelihood Explanation

Any one of the MPC participants that is Byzantine (or compromised) can exploit this. The attacker only needs to observe a pending `AppPublicKey` CKD request on-chain and submit `respond_ckd` before the honest leader. No threshold collusion, no TEE compromise, and no operator misconfiguration is required. The race window is the block-time gap between the request being indexed and the honest leader's response transaction being finalized.

### Recommendation

Apply the same cryptographic output binding to the `AppPublicKey` variant that `AppPublicKeyPV` already has. Specifically, add a check that `big_y = msk_public_key * H(app_id, derivation_path)` (or the equivalent commitment equation) before calling `resolve_yields_for`. If a publicly verifiable check is not possible for the non-PV variant, the contract must at minimum require a threshold of participants to agree on the response before resolving the yield, rather than accepting the first responder's answer.

### Proof of Concept

1. Deploy the contract with one Byzantine attested participant `B` and honest participants.
2. A user calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk)`.
3. Before the honest leader submits its `respond_ckd`, participant `B` calls:
   ```rust
   contract.respond_ckd(ckd_request, CKDResponse {
       big_y: Bls12381G1PublicKey([0u8; 48]),
       big_c: Bls12381G1PublicKey([0u8; 48]),
   });
   ```
4. The contract executes the empty `AppPublicKey` match arm, skips all validation, and calls `resolve_yields_for` — returning `Ok(())`.
5. The user's yield is resolved with all-zero `(big_y, big_c)`.
6. The honest leader's subsequent `respond_ckd` call returns `Err(RequestNotFound)`.
7. Assert: the user received `big_y = [0u8; 48]`, which is not `msk * H(pk, app_id)`.

This is directly demonstrated by the existing test at line 3404, which already passes arbitrary bytes through `respond_ckd` for the `AppPublicKey` variant with no rejection. [5](#0-4)
