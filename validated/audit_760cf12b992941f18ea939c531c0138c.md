## Title
ADA Creation via Unguarded Pool Deposit in Subtransaction SUBUTXO — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

## Summary

The Dijkstra era's `SUBUTXO` rule calls `updateUTxOStateNoFees`, which unconditionally increases `utxosDeposited` by the pool deposit amount when a subtransaction contains a `RegPoolTxCert` for a new pool. However, `SUBUTXO` contains **no value-conservation check** (`validateValueNotConservedUTxO` is absent), and the top-level UTXO rule's value-conservation check only covers the **top-level transaction body's certificates**, not subtransaction certificates. An attacker can therefore submit a subtransaction whose inputs equal its outputs (no ADA deducted for the deposit) while still triggering the deposit-pot increase, creating ADA from nothing.

### Step 4 — SUBUTXO has no value-conservation check

`dijkstraSubUtxoTransition` (lines 231–278) performs structural checks (validity interval, input existence, output size, network IDs) but **never calls `validateValueNotConservedUTxO`**. Compare with every prior era's UTXO rule, which always includes:

```haskell

### Step 5 — `updateUTxOStateNoFees` blindly increases the deposit pot

When `isValid = True`, SUBUTXO calls:

```haskell
newState <-
  Shelley.updateUTxOStateNoFees
    pp utxoState txBody certState
    (utxosGovState utxoState)
    (tellEvent . TotalDeposits (hashAnnotated txBody))
    (\a b -> tellEvent $ TxUTxODiff a b)
``` [5](#0-4) 

Inside `updateUTxOStateNoFees`:

```haskell
totalDeposits = certsTotalDepositsTxBody pp certState txBody

...
utxosDeposited = utxosDeposited <> depositChange
``` [6](#0-5) 

For a `RegPoolTxCert` for a pool not yet in `psStakePools certState`, `certsTotalDepositsTxBody` returns `ppPoolDeposit` (via `shelleyTotalDepositsTxCerts`):

```haskell
numNewRegPoolCerts = Set.size (F.foldl' addNewPoolIds Set.empty certs)
addNewPoolIds regPoolIds = \case
  RegPoolTxCert (StakePoolParams {sppId})
    | not (isRegPoolRegistered sppId || Set.member sppId regPoolIds) -> Set.insert sppId regPoolIds

`updateUTxOStateNoFees` then increases `utxosDeposited` by `ppPoolDeposit` **without verifying** that the subtransaction's inputs minus outputs equals that amount.

## Impact Explanation

**Critical — direct ADA creation.**

Before the batch transaction:
```
Total ADA = UTxO_val + D + F + N
```

After a subtransaction with `RegPoolTxCert`, inputs = X, outputs = X (no deposit deducted):
```
UTxO change:        X removed, X added → net 0
utxosDeposited:     D + ppPoolDeposit
Total ADA:          UTxO_val + D + ppPoolDeposit + F + N
```

`ppPoolDeposit` ADA is created from nothing. The pool is registered with a deposit recorded in `PState.spsDeposit`, but no corresponding ADA was ever moved from the UTxO into the deposit pot. At the next POOLREAP epoch boundary, the deposit is refunded to the pool owner, destroying ADA from the deposit pot and leaving the system with a permanent deficit equal to `ppPoolDeposit` per exploit invocation.

---

## Likelihood Explanation

The Dijkstra era is new and subtransactions are a novel mechanism. Any unprivileged user who can submit a batch transaction can exploit this. No special keys, governance majority, or privileged access is required. The attack is repeatable: each subtransaction with a distinct pool key creates another `ppPoolDeposit` worth of ADA. The only cost is the top-level transaction fee.

---

## Recommendation

Add a value-conservation check to `dijkstraSubUtxoTransition` analogous to the one in every prior era's UTXO rule:

```haskell
-- In dijkstraSubUtxoTransition, before updateUTxOStateNoFees:
runTest $ Shelley.validateValueNotConservedUTxO pp (utxosUtxo utxoState) certState txBody
```

The check must use the subtransaction's own body and the certState snapshot passed to SUBUTXO (the pre-SUBENTITIES state, which is already what is passed). This ensures that `inputs + refunds = outputs + deposits` for every subtransaction, closing the ADA-creation gap.

## Proof of Concept

```
1. Obtain a UTxO entry U with value V ≥ minUTxOValue.
2. Construct a subtransaction sub_tx:
     inputs  = {U}
     outputs = {U' at attacker address, value V}   -- no deposit deducted
     certs   = [RegPoolTxCert poolParams]           -- new pool, not yet registered
3. Construct a top-level batch transaction top_tx:
     inputs  = {some other UTxO for fee}
     outputs = {change}
     fee     = minFee
     subtransactions = [sub_tx]
4. Submit top_tx.
5. Observe:
     - top-level validateValueNotConservedUTxO passes (no pool cert in top-level body)
     - SUBPOOL registers the pool
     - utxosDeposited increases by ppPoolDeposit
     - UTxO value is unchanged (V in, V out)
     - Total ADA in the system increased by ppPoolDeposit
6. Wait for POOLREAP; the deposit is refunded to the pool owner.
   The deposit pot is now short ppPoolDeposit, permanently.
```
