### Title
`KernelDepositPool`: `userWithdrawalIds` array never cleaned up on claim, permanently locking users out of future withdrawals — (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool.initiateWithdrawal` enforces a cap on open withdrawals by checking `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`. However, `claimWithdrawal` never removes the claimed withdrawal ID from `userWithdrawalIds`. Because the array only ever grows, once a user has initiated exactly `maxNumberOfWithdrawalsPerUser` withdrawals — even after claiming every one of them — they are permanently blocked from initiating any further withdrawal, freezing their staked KERNEL principal forever.

---

### Finding Description

`userWithdrawalIds` is declared as a plain dynamic array:

```solidity
mapping(address user => uint256[] withdrawalIds) public userWithdrawalIds;
``` [1](#0-0) 

`initiateWithdrawal` enforces the cap by checking the raw `.length` of this array and then appending to it:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
...
userWithdrawalIds[msg.sender].push(withdrawalId);
``` [2](#0-1) 

`claimWithdrawal` only sets `withdrawal.claimed = true` on the `Withdrawal` struct stored in `withdrawals[withdrawalId]`. It never touches `userWithdrawalIds`:

```solidity
if (withdrawal.claimed) {
    revert WithdrawalAlreadyClaimed();
}
``` [3](#0-2) 

The existence of the `claimed` boolean field on the struct is itself evidence that the array is not cleaned up — if the ID were removed from `userWithdrawalIds` on claim, the field would be unnecessary for double-claim protection.

The natspec comment on `maxNumberOfWithdrawalsPerUser` states its intent:

> "The maximum number of withdrawals that any user can have **open (unclaimed)** at any time" [4](#0-3) 

But the enforcement counts **all** withdrawal IDs ever created for the user, not just unclaimed ones. This is the root cause: accumulated state (`userWithdrawalIds.length`) is never reset when a withdrawal is completed, exactly mirroring M-13's `reviewsByStatus` counter not being reset on re-registration.

---

### Impact Explanation

After a user has initiated `maxNumberOfWithdrawalsPerUser` withdrawals and claimed all of them, `userWithdrawalIds[user].length` equals `maxNumberOfWithdrawalsPerUser`. Every subsequent call to `initiateWithdrawal` reverts with `WithdrawalLimitReached`. The user's staked KERNEL principal is permanently frozen — they can never exit their position. This satisfies **Critical: Permanent freezing of funds**.

The admin can raise `maxNumberOfWithdrawalsPerUser`, but the array continues to grow monotonically, so the user will hit any finite limit after enough withdrawal cycles. There is no on-chain mechanism to shrink `userWithdrawalIds`.

---

### Likelihood Explanation

The default `maxNumberOfWithdrawalsPerUser` is bounded by `MAX_WITHDRAWALS_PER_USER`:

```solidity
if (_maxNumberOfWithdrawalsPerUser == 0 || _maxNumberOfWithdrawalsPerUser > MAX_WITHDRAWALS_PER_USER) {
    revert InvalidMaxNumberOfWithdrawalsPerUser();
}
``` [5](#0-4) 

Any active staker who cycles through withdrawals — a normal, expected usage pattern — will eventually trigger this condition. No special privilege or external dependency is required; the attacker path is simply repeated `initiateWithdrawal` + `claimWithdrawal` calls by the victim themselves.

---

### Recommendation

In `claimWithdrawal`, remove the claimed withdrawal ID from `userWithdrawalIds[withdrawal.user]` (swap-and-pop pattern), so the length accurately reflects only open withdrawals:

```diff
+ // Remove the withdrawal ID from the user's list (swap-and-pop)
+ uint256[] storage ids = userWithdrawalIds[withdrawal.user];
+ for (uint256 i = 0; i < ids.length; i++) {
+     if (ids[i] == _withdrawalId) {
+         ids[i] = ids[ids.length - 1];
+         ids.pop();
+         break;
+     }
+ }
  withdrawal.claimed = true;
```

Alternatively, replace the `uint256[]` with a counter of **unclaimed** withdrawals per user, incrementing on `initiateWithdrawal` and decrementing on `claimWithdrawal`.

---

### Proof of Concept

1. Admin deploys `KernelDepositPool` with `maxNumberOfWithdrawalsPerUser = 5`.
2. Alice stakes 100 KERNEL.
3. Alice calls `initiateWithdrawal` five times (IDs 1–5). `userWithdrawalIds[Alice].length == 5`.
4. Alice waits for `withdrawalDelay` and calls `claimWithdrawal` for IDs 1–5. All five are marked `claimed = true`. `userWithdrawalIds[Alice].length` remains `5`.
5. Alice calls `initiateWithdrawal` again → `5 >= 5` → `WithdrawalLimitReached` revert.
6. Alice's remaining staked KERNEL is permanently frozen; she can never withdraw it.
