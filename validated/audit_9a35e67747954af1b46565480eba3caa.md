### Title
`verify_foreign_transaction` Signs with Zero Tweak (Root Key) Instead of Caller-Derived Key, Enabling Cross-Caller Attestation Replay - (File: crates/node/src/providers/verify_foreign_tx/sign.rs)

### Summary

The `verify_foreign_transaction` flow hardcodes `Tweak::new([0u8; 32])` (zero tweak = root ForeignTx key) in `build_signature_request` instead of deriving a caller-specific key from `(predecessor_id, derivation_path)` as the design document specifies. Combined with the caller-agnostic `VerifyForeignTransactionRequest` key (no `predecessor_id` field), every foreign-tx attestation is signed under the same shared root key. Any party who observes a valid attestation on-chain can replay it against any bridge contract that verifies against the root public key, enabling unauthorized fund release or double-spend conditions.

### Finding Description

The design document (`docs/foreign-chain-transactions.md`) explicitly specifies that `verify_foreign_transaction` must derive a caller-specific tweak using a foreign-tx-specific prefix:

```rust
// From docs/foreign-chain-transactions.md (design spec)
pub struct VerifyForeignTransactionRequestArgs {
    pub request: ForeignChainRpcRequest,
    pub derivation_path: String, // Key derivation path  ← MISSING in implementation
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}

pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub tweak: Tweak,            // ← MISSING in implementation
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

The actual implementation deviates at three points:

**1. `VerifyForeignTransactionRequestArgs` has no `derivation_path` field:** [1](#0-0) 

**2. `args_into_verify_foreign_tx_request` computes no tweak — it simply copies fields:** [2](#0-1) 

**3. `build_signature_request` hardcodes a zero tweak instead of a caller-derived one:** [3](#0-2) 

The zero tweak means the MPC nodes sign every `verify_foreign_transaction` response with the **root ForeignTx domain key**. The `respond_verify_foreign_tx` function is internally consistent with this — it verifies against the root public key — but this consistency is itself the vulnerability: the root key is shared by all callers and is publicly queryable via `public_key()`.

The request key is also caller-agnostic (no `predecessor_id` in `VerifyForeignTransactionRequest`), which the codebase explicitly acknowledges: [4](#0-3) 

Contrast this with `sign()` and `request_app_private_key()`, which both derive caller-specific tweaks/app-IDs from `predecessor_id`: [5](#0-4) [6](#0-5) 

### Impact Explanation

Because every `verify_foreign_transaction` attestation is signed under the same root ForeignTx key:

1. Alice calls `verify_foreign_transaction` for Bitcoin tx X. She receives a `VerifyForeignTransactionResponse` containing a signature under the root key.
2. NEAR transaction receipts are public. Bob reads Alice's response from the chain.
3. Bob presents Alice's signature to his own bridge contract.
4. Bob's bridge contract calls `public_key()` to obtain the root ForeignTx public key and verifies the signature — it passes, because the signature is under the root key, not Alice's derived key.
5. Bob's bridge contract releases funds based on Alice's attestation of Bitcoin tx X.

This is a concrete cross-caller replay / double-spend scenario. The signature covers the specific `(request, observed_values)` tuple, so the replay is constrained to the same foreign tx — but that is precisely the transaction a bridge contract acts on. Any bridge contract that does not independently track "which account's attestation was already consumed" is vulnerable.

The design doc's intended mitigation — caller-specific key derivation — was never implemented. [7](#0-6) 

### Likelihood Explanation

- The deviation from the design spec is complete: `derivation_path` is absent from the args struct, the tweak is absent from the request struct, and the node hardcodes zero. This is not a partial implementation.
- NEAR transactions are fully public; any observer can extract a valid attestation from a receipt.
- Bridge contracts are the primary intended consumer of `verify_foreign_transaction` (the design doc's stated use case is "Omnibridge inbound flow"). Such contracts routinely verify a signature and release funds without tracking per-caller consumption.
- The root public key is trivially obtainable via `public_key(domain_id)`.

### Recommendation

1. Add `derivation_path: String` to `VerifyForeignTransactionRequestArgs`.
2. In `args_into_verify_foreign_tx_request`, compute `tweak = derive_foreign_tx_tweak(&predecessor_id, &args.derivation_path)` using the foreign-tx-specific prefix from the design doc, and store it in `VerifyForeignTransactionRequest`.
3. In `build_signature_request` (node side), use `request.tweak` instead of `Tweak::new([0u8; 32])`.
4. In `respond_verify_foreign_tx`, verify the signature against the **derived** public key (apply the tweak to the root key before verification), not the raw root key.
5. Include `predecessor_id` (or the derived tweak) in the `VerifyForeignTransactionRequest` map key so that different callers' requests are isolated and cannot fan-out to each other.

## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.
