### Title
Missing CKD Output Verification on `AppPublicKey` Path Allows Single Leader to Forge Confidential Key Derivation Output — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` in `mpc-contract` enforces a cryptographic output check only for the `AppPublicKeyPV` variant of a CKD request. For the `AppPublicKey` (legacy) variant, the contract performs **no verification** of the submitted `CKDResponse`. A single attested-participant leader node — strictly below the signing threshold — can submit any arbitrary `(big_y, big_c)` pair and the contract will accept it, resolve the pending yield, and deliver the fraudulent key material to the requesting user. This directly mirrors the OptimismHubConnector pattern: one code path (PV) has a proper origin/content check; the other (non-PV) silently skips it.

---

### Finding Description

In `respond_ckd()`, after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check()` enforces the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(app_id, pk), mpc_pk)`, which cryptographically proves the response was produced using the threshold master secret. [2](#0-1) 

For `AppPublicKey`, the arm is an empty block `{}`. The contract immediately proceeds to `resolve_yields_for`, delivering whatever `(big_y, big_c)` the caller supplied. [3](#0-2) 

The `AppPublicKey` variant is the legacy "privately verifiable" form still accepted in production (the README labels it "legacy" and the e2e test `ckd_response__passes_cryptographic_verification` exercises it). [4](#0-3) 

The off-chain node leader is the sole submitter of `respond_ckd`. The leader is elected per-request and is a single attested participant — not a threshold quorum. [5](#0-4) 

---

### Impact Explanation

The threshold CKD protocol is designed so that no single node below the threshold can determine or forge the output `big_s = H(app_id, pk) · msk`. The on-chain `ckd_output_check` for `AppPublicKeyPV` enforces this invariant at the contract level. For `AppPublicKey` requests, that invariant is entirely absent from the contract. A single malicious leader can:

1. Skip the threshold CKD computation entirely.
2. Submit arbitrary `big_y` and `big_c` values (e.g., random group elements).
3. The contract resolves the yield and returns the fraudulent `CKDResponse` to the user.
4. The user's derived "private key" is `big_c - big_y · app_sk`, which is attacker-controlled garbage rather than the legitimate `big_s`.

If the user does not independently verify the output (which is the common case — the "private verifiability" is optional and off-chain), they will use a wrong key, potentially losing access to assets encrypted or controlled by the expected derived key. This constitutes **confidential key derivation output delivered without the required threshold participant authorization**, matching the Critical impact tier.

---

### Likelihood Explanation

- The `AppPublicKey` path is reachable by any single attested participant who is elected leader for a CKD request. No threshold collusion is required.
- The leader election is per-request and rotates; any participant can become leader.
- The `AppPublicKey` variant is still the default legacy format accepted by the contract and exercised in e2e tests.
- A Byzantine participant strictly below the signing threshold is the exact attacker model in scope.

---

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` branch. For the non-PV variant, the verifiable relationship is `e(big_c - big_y · app_pk1, g2) = e(H(app_id, pk), mpc_pk)` (i.e., verify `big_c = big_s + app_pk1 · y` using the public `app_pk1 = g1 · app_sk`). Alternatively, deprecate and remove the `AppPublicKey` variant entirely in favour of the publicly verifiable `AppPublicKeyPV` path, which already has the correct on-chain check.

---

### Proof of Concept

1. User submits a `request_app_private_key` call with `CKDAppPublicKey::AppPublicKey(pk1)`.
2. The request is queued in `pending_ckd_requests`.
3. A malicious leader (single attested participant) calls `respond_ckd` with the correct `CKDRequest` but a fabricated response: `CKDResponse { big_y: g1, big_c: g1 }` (or any arbitrary BLS12-381 G1 points).
4. The contract passes all checks (signer, attestation, protocol state) and reaches the `match` block. The `AppPublicKey(_) => {}` arm executes with no verification.
5. `resolve_yields_for` is called, resuming the user's yield with the fraudulent `(big_y, big_c)`.
6. The user receives `big_c - big_y · app_sk = g1 - g1 · app_sk`, which is not the legitimate `big_s`, and any key material derived from it is attacker-influenced.

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates this: it passes `big_y = [1u8;48]` and `big_c = [2u8;48]` (invalid BLS points) and the contract accepts them without complaint. [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Verify that a CKD response (AppPublicKey variant) is mathematically correct.
#[tokio::test]
#[expect(non_snake_case)]
async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),
        "CKD response failed cryptographic verification"
    );
}
