Audit Report

## Title
Missing L2 Sequencer Uptime Check Enables Stale-Price Exploitation During Sequencer Restart - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

## Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls Chainlink's `latestRoundData()` without verifying that the Arbitrum (or Unichain) sequencer is live. When the sequencer restarts after an outage, the feed may return a pre-outage price that has not yet caught up to the true market price. Any unprivileged depositor can exploit this window to receive more rsETH than their collateral is worth, extracting value from the pool's rsETH reserves at the expense of the protocol and other holders.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` fetches the collateral-to-ETH price via `latestRoundData()` and applies three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [1](#0-0) 

None of these guards detect a sequencer outage. When an Arbitrum sequencer goes offline, Chainlink L2 feeds stop updating; `updatedAt` freezes and `answeredInRound == roundID`, so all three checks pass silently while the price is stale.

This oracle is consumed by `RSETHPool` (Arbitrum) via `IOracle(supportedTokenOracle[token]).getRate()`: [2](#0-1) 

and by `RSETHPoolNoWrapper` (Arbitrum / Unichain) via the same pattern: [3](#0-2) 

The rsETH amount minted to a depositor is computed as:

```
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate
```

If `tokenToETHRate` (from the stale collateral oracle) is inflated relative to the true market price, the depositor receives more rsETH than the deposited collateral is worth. The surplus rsETH is drawn from the pool's own rsETH reserves.

Both `deposit(address token, uint256 amount, string referralId)` functions are callable by any unprivileged external user with no access control beyond `whenNotPaused` / `onlySupportedToken`. [4](#0-3) [5](#0-4) 

## Impact Explanation
**High — Theft of unclaimed yield.**

When the sequencer restarts and the collateral feed has not yet updated, an attacker deposits collateral at the stale (inflated) price and receives rsETH worth more than the deposited collateral. The excess rsETH is taken from the pool's reserves, constituting direct theft of yield/value that would otherwise accrue to the protocol and existing rsETH holders. The attacker retains the surplus rsETH permanently; there is no mechanism to claw it back.

## Likelihood Explanation
Arbitrum sequencer outages have occurred (December 2021, June 2023). The exploitable window opens immediately when the sequencer resumes and closes only after the Chainlink feed's next heartbeat update. No special permissions are required — any address can call `deposit()`. The attacker needs only to monitor the sequencer status and submit a deposit transaction in the first block(s) after restart. The attack is repeatable across every sequencer restart event.

## Recommendation
Follow the Chainlink L2 Sequencer Uptime Feed pattern. Before consuming `latestRoundData()` in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, query the sequencer uptime feed and revert if the sequencer is down or within the grace period after restart:

```solidity
AggregatorV3Interface sequencerFeed = AggregatorV3Interface(SEQUENCER_UPTIME_FEED);
(, int256 answer, uint256 startedAt,,) = sequencerFeed.latestRoundData();
if (answer != 0) revert SequencerDown();
if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();
```

The `SEQUENCER_UPTIME_FEED` address and `GRACE_PERIOD` (typically 3600 seconds) should be set at construction time and be immutable. Apply the same guard to any other oracle wrapper deployed on L2s.

## Proof of Concept
**Foundry fork test outline (Arbitrum mainnet fork):**

1. Fork Arbitrum at a block just after a historical sequencer restart (or simulate by manipulating `block.timestamp` and mocking the Chainlink feed to return a pre-outage price).
2. Deploy / reference the live `ChainlinkOracleForRSETHPoolCollateral` pointing at the wstETH/ETH feed.
3. Mock `latestRoundData()` to return a price 33% above the current true price (simulating a stale pre-outage value), with `answeredInRound == roundID` and `timestamp != 0` so all existing guards pass.
4. Call `RSETHPool.deposit(wstETH, 1e18, "")` as an unprivileged attacker.
5. Assert that `rsETHAmount` received exceeds the fair value computed using the true current price by ~33%.
6. Confirm the pool's rsETH reserve decreased by the full (inflated) amount, not the fair amount.

The three existing guards in `getRate()` — `answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0` — all pass because the stale round data is internally self-consistent; only the sequencer uptime check would catch this condition. [1](#0-0)

        (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =

        if (timestamp == 0) revert IncompleteRound();
