### Title
Immutable Oracle Addresses in `RSETHPriceFeed` Cannot Be Updated If External Contracts Change — (File: `contracts/oracles/RSETHPriceFeed.sol`)

---

### Summary

`RSETHPriceFeed` hardcodes both `ETH_TO_USD` (a Chainlink ETH/USD aggregator) and `RS_ETH_ORACLE` (the `LRTOracle` contract) as `immutable` state variables. The contract has no owner, no admin setter, and is not upgradeable. If either address ever needs to change — for example, if Chainlink deprecates the ETH/USD feed (which has occurred historically) or if the protocol redeploys `LRTOracle` — `RSETHPriceFeed` will permanently return incorrect rsETH/USD prices with no on-chain mechanism to correct it.

---

### Finding Description

`RSETHPriceFeed` is a Chainlink `AggregatorV3Interface`-compatible price feed for rsETH/USD, intended for consumption by external DeFi protocols (e.g., lending markets using rsETH as collateral).

Both of its critical external dependencies are declared `immutable` at construction time:

```solidity
AggregatorV3Interface public immutable ETH_TO_USD;
IRSETHOracle public immutable RS_ETH_ORACLE;
``` [1](#0-0) 

These are set once in the constructor and can never be changed:

```solidity
constructor(address ethToUSDAggregatorAddress, address rsETHOracle, string memory description_) {
    ETH_TO_USD = AggregatorV3Interface(ethToUSDAggregatorAddress);
    RS_ETH_ORACLE = IRSETHOracle(rsETHOracle);
    description = description_;
}
``` [2](#0-1) 

Both `latestRoundData()` and `getRoundData()` compose the rsETH/USD price by multiplying the ETH/USD Chainlink answer by the rsETH/ETH rate from `RS_ETH_ORACLE`:

```solidity
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
``` [3](#0-2) [4](#0-3) 

The contract has no `Ownable`, no `AccessControl`, no proxy, and no upgrade path. There is no function to update either address after deployment.

This is in direct contrast to other oracle-adjacent contracts in the same codebase. For example, `RSETHPoolV3` exposes `setRSETHOracle()` to update its rate source:

```solidity
function setRSETHOracle(address _rsETHOracle) external onlyRole(TIMELOCK_ROLE) {
    UtilLib.checkNonZeroAddress(_rsETHOracle);
    rsETHOracle = _rsETHOracle;
    emit OracleSet(_rsETHOracle);
}
``` [5](#0-4) 

And `LRTOracle` exposes `updatePriceOracleFor()` to update asset price oracles: [6](#0-5) 

`RSETHPriceFeed` has no equivalent.

---

### Impact Explanation

If the Chainlink ETH/USD aggregator address pointed to by `ETH_TO_USD` is deprecated (Chainlink has deprecated feeds on multiple occasions), `RSETHPriceFeed` will continue calling the old feed. Depending on the deprecated feed's behavior, it may revert (causing a DoS for all consumers), return a stale answer, or return zero — all of which produce incorrect rsETH/USD prices.

Similarly, if `LRTOracle` is ever redeployed to a new proxy address (e.g., due to a critical bug requiring a fresh deployment), `RS_ETH_ORACLE` will point to the abandoned contract, causing `rsETHPrice()` to return a permanently stale value.

Any external lending protocol or AMM using `RSETHPriceFeed` as its rsETH/USD price source would then operate on incorrect prices, causing the contract to fail to deliver its promised function: an accurate, up-to-date rsETH/USD rate.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

---

### Likelihood Explanation

Chainlink has a documented history of deprecating price feeds and migrating to new aggregator addresses. The ETH/USD feed is one of the most widely used feeds and has been migrated before. Additionally, the LRT-rsETH protocol uses upgradeable proxies for `LRTOracle`, but a sufficiently severe bug could necessitate a fresh proxy deployment. Either event would permanently break `RSETHPriceFeed` with no recovery path.

**Likelihood: Low-to-Medium** — the trigger requires an external feed migration or a protocol-level redeployment, both of which are realistic over a multi-year protocol lifetime.

---

### Recommendation

Add an owner or admin role to `RSETHPriceFeed` and expose setter functions for both `ETH_TO_USD` and `RS_ETH_ORACLE`, analogous to the pattern already used in `RSETHPoolV3.setRSETHOracle()` and `LRTOracle.updatePriceOracleFor()`. Alternatively, convert the contract to an upgradeable proxy so that the addresses can be updated via an upgrade if needed.

---

### Proof of Concept

1. `RSETHPriceFeed` is deployed with a specific Chainlink ETH/USD aggregator address and the current `LRTOracle` proxy address.
2. Chainlink deprecates the ETH/USD aggregator and deploys a new one at a different address.
3. `RSETHPriceFeed.latestRoundData()` continues to call the deprecated aggregator. Depending on the deprecated feed's behavior, it reverts, returns stale data, or returns zero.
4. Any external protocol (e.g., a lending market) using `RSETHPriceFeed` as its rsETH/USD oracle now receives incorrect prices.
5. There is no function in `RSETHPriceFeed` to update `ETH_TO_USD` to the new aggregator address, and the contract is not upgradeable, so the breakage is permanent unless the consuming protocol manually switches to a redeployed `RSETHPriceFeed`. [7](#0-6)
