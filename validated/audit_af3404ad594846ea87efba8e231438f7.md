### Title
Unprotected `init` Function Allows Front-Running of MPC Contract Initialization — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `init` function of the MPC contract carries no caller-identity restriction. Any NEAR account that observes the contract deployment transaction can race to call `init` first, installing an attacker-controlled participant set and threshold. Because NEAR's `#[init]` macro only prevents *re-initialization* (panics if state already exists), the first caller wins unconditionally. The legitimate operators' subsequent `init` call will fail, and the attacker becomes the sole controller of the MPC network.

---

### Finding Description

The `init` function is the sole public entry point for bootstrapping the MPC contract from a freshly deployed state: [1](#0-0) 

It carries `#[init]` (prevents double-initialization) but **no** `#[private]` attribute and no `assert_eq!(env::predecessor_account_id(), env::current_account_id())` guard. Any NEAR account may call it.

Compare with `init_running`, which is correctly restricted: [2](#0-1) 

The production deployment flow is a **two-step, two-transaction sequence**: first deploy the WASM, then call `init` in a separate transaction. This is visible in both the e2e test harness: [3](#0-2) [4](#0-3) 

and the localnet launch script: [5](#0-4) 

Between the deploy transaction and the `init` transaction there is an observable window on the NEAR blockchain. An attacker monitoring the chain can submit their own `init` call with an attacker-controlled `ThresholdParameters` (e.g., a single participant — themselves — with `threshold = 1`). The legitimate `init` call then fails because state already exists.

---

### Impact Explanation

After a successful front-run:

1. The attacker is the sole registered participant with threshold 1.
2. They call `vote_add_domains` to add signing domains.
3. The MPC nodes (which are not yet running) are irrelevant — the attacker controls the contract state and can submit `vote_pk` / `start_keygen_instance` responses themselves, or simply wait until they can run a node under their own account.
4. They obtain threshold signatures for arbitrary foreign-chain payloads via `sign` or `verify_foreign_transaction`, with no legitimate participant involved.

This constitutes **unauthorized threshold signature issuance** and **unauthorized access to MPC signing capability** — a Critical impact under the allowed scope. [6](#0-5) 

---

### Likelihood Explanation

- NEAR transactions are publicly visible before finalization; an attacker can watch for a `DeployContract` action targeting the known MPC contract account ID.
- The gap between deploy and init is at least one block (two separate async calls in all observed deployment paths).
- No special privilege, key material, or collusion is required — any NEAR account with enough gas can execute the attack.
- The attack is permanent: once `init` succeeds for the attacker, the legitimate operators cannot re-initialize without redeploying the contract to a new account.

---

### Recommendation

Add an explicit caller-identity check inside `init` that restricts invocation to the contract account itself (i.e., the account that owns the WASM), mirroring the `#[private]` guard already present on `init_running`:

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    // Add this guard:
    assert_eq!(
        env::predecessor_account_id(),
        env::current_account_id(),
        "init can only be called by the contract account itself"
    );
    // ... rest of function
}
```

Alternatively, use NEAR's `#[private]` attribute (which enforces `predecessor == current_account_id`) as is already done for `init_running`. Additionally, the deployment tooling should batch the deploy action and the `init` call into a single NEAR transaction using batch actions, eliminating the race window entirely.

---

### Proof of Concept

1. Operator broadcasts a `DeployContract` transaction to `mpc-contract.near`.
2. Attacker observes the transaction in the mempool / after the block that includes it.
3. Attacker submits:
   ```json
   near contract call-function as-transaction mpc-contract.near init \
     json-args '{
       "parameters": {
         "threshold": 1,
         "participants": {
           "next_id": 1,
           "participants": [["attacker.near", 0, {"tls_public_key": "<attacker_key>", "url": "https://attacker.example"}]]
         }
       }
     }' prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
     sign-as attacker.near network-config mainnet sign-with-keychain send
   ```
4. Operator's `init` call fails: `"Smart contract panicked: The contract has already been initialized"`.
5. Attacker, as the sole participant with threshold 1, proceeds to add domains and generate MPC keys, gaining full unauthorized signing authority over all foreign chains. [7](#0-6)
