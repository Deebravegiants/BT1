### Title
Direct Deposits Omitted from Preservation-of-Value Accounting Allows ADA Creation from Nothing — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the `dijkstraSubTxProducedValue` and `dijkstraProducedValue` functions compute the "produced" side of the preservation-of-value equation but omit `directDeposits` from the calculation. Because `directDeposits` are also absent from the "consumed" side, the `consumed == produced` check passes for any transaction that includes direct deposits. However, the direct deposits are then applied to reward accounts via `applyDirectDeposits`, adding ADA to those accounts without any corresponding deduction from the UTxO. The result is unbounded ADA creation from nothing.

---

### Finding Description

The Dijkstra era introduces a new `DirectDeposits` field in both top-level and sub-transaction bodies. A `DirectDeposits` map specifies amounts to be credited directly to registered reward accounts. These are applied in the `ENTITIES` / `SUBENTITIES` rules:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs:211-216
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [1](#0-0) 

`applyDirectDeposits` unconditionally adds coin to account balances:

```haskell

  updateAccountBalances

Neither `dijkstraSubTxProducedValue` nor `dijkstraProducedValue` (via `conwayProducedValue`) includes `directDepositsTxBodyL`. The consumed side (`getConsumedDijkstraValue` → `getConsumedMaryValue`) also does not include direct deposits, so the equation balances:

```
consumed  = UTxO inputs + refunds + withdrawals
produced  = outputs + fee + cert-deposits + treasury-donation + burned-assets
```

Direct deposits are applied to reward accounts outside this equation, with no corresponding deduction from either side. The ADA credited to reward accounts is not sourced from the transaction's inputs in the accounting sense — it is created from nothing.

The `DijkstraTxBodyRaw` confirms both top-level and sub-transaction bodies carry the `directDeposits` field:

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker submits a Dijkstra transaction (or sub-transaction) containing a `directDeposits` map targeting any registered reward accounts. The preservation-of-value check passes because direct deposits are absent from both sides of the equation. The `ENTITIES`/`SUBENTITIES` rule then credits the specified amounts to the target accounts. The UTxO is not reduced by the deposited amount. Repeating this across many transactions allows unbounded ADA inflation, directly violating the total-supply invariant of the Cardano ledger.

---

### Likelihood Explanation

The Dijkstra era is production code in this repository. The `directDeposits` field is serializable (CDDL field `? 25 : direct_deposits` in `dijkstra.cddl`), meaning any node operator or wallet can construct and submit such a transaction once the era is active. No privileged access, governance majority, or key compromise is required. The only prerequisite is that the target reward accounts are registered, which is trivially achievable by the attacker themselves. [8](#0-7) 

---

### Recommendation

`directDeposits` must be included in the produced-value calculation. The fix is to add the sum of all direct-deposit amounts to the produced side in both `dijkstraSubTxProducedValue` and `dijkstraProducedValue` (via `conwayProducedValue` or a Dijkstra-specific override):

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject ( getTotalDepositsTxBody pp isRegPoolId txBody
             <> txBody ^. treasuryDonationTxBodyL
             <> foldMap id (unDirectDeposits (txBody ^. directDepositsTxBodyL))  -- ADD THIS
              )
    <> burnedMultiAssets txBody
```

The same correction must be applied to the top-level `dijkstraProducedValue` path for `dtbrDirectDeposits`. A corresponding property-based test should verify that `consumed == produced` holds for all transactions containing non-empty `directDeposits`.

---

### Proof of Concept

1. Register two stake credentials `credA` (attacker-controlled) and `credB` (victim or attacker-controlled).
2. Construct a Dijkstra transaction with:
   - One UTxO input worth `X` lovelace.
   - One output returning `X - fee` lovelace to the attacker.
   - `directDeposits = { rewardAccountB → 1_000_000_000 }` (1,000 ADA).
3. Submit the transaction. The preservation-of-value check evaluates:
   - `consumed = X`
   - `produced = (X - fee) + fee = X` ✓ (direct deposits not counted)
4. The `ENTITIES` rule applies `applyDirectDeposits`, crediting 1,000 ADA to `rewardAccountB`.
5. The attacker's UTxO is unchanged in net (minus fee). `rewardAccountB` now holds 1,000 ADA that did not exist before.
6. The attacker withdraws from `rewardAccountB` in a subsequent transaction, extracting the newly created ADA. [4](#0-3) [1](#0-0) [2](#0-1)
