Audit Report

## Title
Uninitialized `rsETHPrice` Causes Division-by-Zero on All Deposit Calls Until `updateRSETHPrice()` Is Invoked - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.initialize()` never assigns a value to `rsETHPrice`, leaving it at the Solidity default of `0`. Every deposit in `LRTDepositPool` unconditionally calls `getRsETHAmountToMint()`, which divides by `lrtOracle.rsETHPrice()`. Until `updateRSETHPrice()` is explicitly called post-deployment, all deposits revert with a division-by-zero panic. No funds are lost because the revert prevents any state change, but the protocol cannot accept deposits during this window.

## Finding Description
`LRTOracle.initialize()` sets only `lrtConfig` and emits an event; `rsETHPrice` is never written: [1](#0-0) 

`rsETHPrice` is a plain `uint256` storage variable that defaults to `0`: [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` performs an unconditional division by `lrtOracle.rsETHPrice()`: [3](#0-2) 

This function is called from `_beforeDeposit()`: [4](#0-3) 

Which is called unconditionally from both `depositETH()` and `depositAsset()`: [5](#0-4) [6](#0-5) 

The only path that sets `rsETHPrice` to a non-zero value is `_updateRsETHPrice()`, which sets it to `1 ether` when `rsethSupply == 0`: [7](#0-6) 

This is only reachable via the permissionless `updateRSETHPrice()`: [8](#0-7) 

There are no guards in `_beforeDeposit()` or `getRsETHAmountToMint()` that check for a zero `rsETHPrice` before dividing.

## Impact Explanation
Between contract deployment and the first call to `updateRSETHPrice()`, every call to `depositETH()` or `depositAsset()` reverts with a division-by-zero panic. No user funds are at risk of loss because the revert prevents any state change. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The failure window opens on every fresh deployment or proxy upgrade that resets storage. Because `updateRSETHPrice()` is `public whenNotPaused`, any user who encounters the revert can self-remediate by calling it directly before retrying their deposit. The window is short in practice but is a real, reachable failure mode for any depositor who acts before the price is seeded.

## Recommendation
Initialize `rsETHPrice` (and `highestRsethPrice`) to `1 ether` inside `LRTOracle.initialize()`, mirroring the logic already present in `_updateRsETHPrice()` for the zero-supply case:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
+   rsETHPrice = 1 ether;
+   highestRsethPrice = 1 ether;
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

## Proof of Concept
1. Deploy `LRTOracle` proxy and call `initialize(lrtConfigAddr)`. `rsETHPrice` is `0`.
2. Do **not** call `updateRSETHPrice()`.
3. Call `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `getRsETHAmountToMint()` → `(1e18 * assetPrice) / 0` → EVM division-by-zero panic → revert.
5. Call `updateRSETHPrice()` (permissionless, no role required). `rsETHPrice` is now `1 ether`.
6. Repeat step 3 — deposit succeeds.
