### Title
Unauthenticated CAT Spend with Crafted REMARK Condition Forces Permanent CATWallet-to-CRCATWallet Conversion with Attacker-Controlled `authorized_providers` — (`chia/wallet/wallet_state_manager.py`, `chia/wallet/vc_wallet/cr_cat_drivers.py`, `chia/wallet/vc_wallet/cr_cat_wallet.py`)

---

### Summary

An unprivileged attacker who owns any CAT coin for a given TAIL can broadcast a spend that permanently converts a victim's plain `CATWallet` into a `CRCATWallet` with attacker-chosen `authorized_providers`. After conversion, the victim's wallet requires VC authorization from the attacker-controlled provider for all future CR-CAT spends, effectively blocking or hijacking the victim's CAT wallet.

---

### Finding Description

The vulnerability spans three functions with no authorization guard between them:

**Step 1 — `CRCAT.get_next_from_coin_spend()` blindly trusts REMARK conditions**

When the parent coin is a plain CAT (not a CR-CAT), the function enters the branch at line 345 and scans inner puzzle output conditions for a REMARK (opcode `1`): [1](#0-0) 

It extracts `new_inner_puzzle_hash`, `authorized_providers_as_prog`, and `proofs_checker` directly from the REMARK condition bytes with **zero validation**. Any spend submitter who controls the inner puzzle of a CAT coin can emit arbitrary values here.

**Step 2 — `handle_cat()` passes those values straight to `convert_to_cr()`**

`handle_cat()` calls `CRCAT.get_next_from_coin_spend(coin_spend)` and then checks only that the resulting `crcat.inner_puzzle_hash` is in the victim's derivation records: [2](#0-1) 

The attacker sets `inner_ph` in the REMARK to the victim's publicly known puzzle hash, so this check passes. The function then checks for an existing CRCAT wallet (lines 1292–1295); finding none (victim has only a plain `CATWallet`), it proceeds to the conversion branch: [3](#0-2) 

`crcat.authorized_providers` here is the attacker-supplied list extracted from the REMARK condition. There is no check that these providers are legitimate or match any on-chain TAIL issuer.

**Step 3 — `convert_to_cr()` persists the attacker-controlled state permanently** [4](#0-3) 

The wallet's database row is overwritten with `WalletType.CRCAT` and the attacker-supplied `authorized_providers`. The in-memory wallet object is also replaced. This is irreversible through normal wallet operation.

---

### Impact Explanation

After conversion:
- The victim's wallet now has `CRCATInfo.authorized_providers = [attacker_bytes32]`.
- All future CR-CAT spend attempts call `vc_wallet.get_vc_with_provider_in_and_proofs(self.info.authorized_providers, ...)`, which requires a VC issued by the attacker-controlled provider — one the victim cannot obtain.
- The victim's existing plain CAT coins (with the old puzzle hash) are no longer recognized by the converted wallet, making them effectively inaccessible through the wallet UI.
- This satisfies the **High** impact category: bypass of wallet authorization enabling unauthorized coin control and permanent corruption of wallet sync state with direct security impact.

---

### Likelihood Explanation

Preconditions are minimal:
1. Attacker owns **any** CAT coin for TAIL_X (purchasable on open markets for any widely-used CAT).
2. Attacker knows the victim's inner puzzle hash (publicly visible on-chain from any prior spend or hint).
3. Victim has a plain `CATWallet` for TAIL_X (the common case before CR-CAT adoption).

The attacker constructs a spend of their own CAT coin that:
- Creates a CRCAT output coin with puzzle hash `CAT(TAIL_X, CR_LAYER(attacker_providers, attacker_checker, victim_inner_ph))`.
- Includes a REMARK condition `(1 victim_inner_ph attacker_providers attacker_checker)`.
- Hints the output coin with `victim_inner_ph`.

The victim's wallet receives the coin state update via hint subscription, and the conversion fires automatically during sync.

---

### Recommendation

1. **Validate `authorized_providers` against the TAIL issuer**: The REMARK-based CR-CAT bootstrapping path in `get_next_from_coin_spend()` should not be trusted for wallet state mutations. At minimum, `handle_cat()` must not call `convert_to_cr()` based solely on REMARK-derived data from a third-party spend.
2. **Require explicit user confirmation** before converting a `CATWallet` to `CRCATWallet`. The conversion should not happen silently during sync.
3. **Bind `authorized_providers` to the TAIL**: If the TAIL issuer embeds `authorized_providers` in the TAIL program itself (verifiable on-chain), the wallet should only accept providers that match the TAIL-embedded list.
