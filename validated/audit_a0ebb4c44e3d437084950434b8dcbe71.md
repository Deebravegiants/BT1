The key question is whether a malicious peer can inject fabricated pool state through the `apply_state_transition` → `update_pool_config` path. Let me trace the full chain precisely.

## Full Trace

**Layer 1 — Peer fetch (no solution validation):**

`WalletStateManager._add_coin_states` detects a spent POOLING_WALLET coin and calls `fetch_coin_spend_for_coin_state(curr_coin_state, peer)`. [1](#0-0) 

`fetch_coin_spend` validates only two things about the peer's response:
- `puzzle.get_tree_hash() == coin.puzzle_hash` (puzzle must hash correctly)
- `response.coin_name == coin_id` (coin name must match)

**The solution field is accepted verbatim from the peer with zero validation.** [2](#0-1) 

**Layer 2 — `apply_state_transition` (no pool-state validation):**

The only guard is a coin-name continuity check (`spent_coin_name != new_state.coin.name()`). There is no call to `_verify_pool_state`, no signature check, and no CLVM execution to confirm the solution is valid for the puzzle. The spend is stored unconditionally after that single check. [3](#0-2) 

**Layer 3 — `update_pool_config` trusts the stored state:**

`update_pool_config` calls `get_current_state()` → `solution_to_pool_state()` on the stored spend, then writes the extracted fields directly to the `PoolingShareState` config file:
- `pool_url` ← `current_state.current.pool_url`
- `target_puzzle_hash` ← `current_state.current.target_puzzle_hash`
- `owner_public_key` ← `current_state.current.owner_pubkey` [4](#0-3) 

`solution_to_pool_state` simply parses the solution bytes; it does not verify the solution against the puzzle. [5](#0-4) 

**What a malicious peer can inject:**

A peer that the wallet connects to can return the correct puzzle (matching `coin.puzzle_hash`) paired with a fabricated solution containing arbitrary pool state. Because the puzzle hash check passes and the solution is never executed against the puzzle, the fabricated `pool_url`, `target_puzzle_hash`, and `owner_public_key` flow through all three layers into the on-disk config.

**What is NOT injectable:**

`payout_instructions` is generated locally from `action_scope.get_puzzle_hash(self.wallet_state_manager)` and is never taken from the peer-supplied solution. [6](#0-5) 

**Downstream impact in the farmer:**

The farmer reads `pool_url` from `PoolingShareState` and makes HTTP requests to it. A fabricated `pool_url` redirects the farmer to an attacker-controlled pool server. The farmer submits proofs of space to that server; the malicious server collects them without paying rewards. `owner_public_key` corruption causes `find_authentication_sk` to fail, breaking authentication with any legitimate pool. [7](#0-6) [8](#0-7)

## Assessment

The invariant violation is concrete: `apply_state_transition` is documented as the place where "all validation should be done," yet it performs no pool-state validation before storing the peer-supplied spend and calling `update_pool_config`. The receiving layer (`update_pool_config`) revalidates nothing — it blindly copies attacker-shaped fields from the parsed solution into the security-critical pool config.

However, the impact ceiling is bounded:

- **`payout_instructions` (the actual XCH reward address) is locally generated and is not injectable.** Direct XCH theft via this path is not possible.
- The attack requires the wallet to be connected to a malicious peer (within the stated threat model of "peer/mempool/sync path").
- The on-chain singleton state is unaffected; only the local wallet config is corrupted.

The reachable impact is **pool membership state corruption**: `pool_url` redirection causes the farmer to submit proofs to an attacker-controlled pool that withholds rewards, and `owner_public_key` corruption breaks pool authentication. This matches the "High: Corruption of... pool membership state... with direct security impact" category.

### Title
Malicious peer can inject fabricated pool state into `update_pool_config` via unvalidated solution — (`chia/pools/pool_wallet.py`)

### Summary
`apply_state_transition` fetches a `CoinSpend` from a peer, validates only the puzzle hash (not the solution), stores the spend, and calls `update_pool_config`. `update_pool_config` parses the solution to extract `pool_url`, `target_puzzle_hash`, and `owner_public_key` and writes them to the `PoolingShareState` config without any revalidation. A malicious peer can return the correct puzzle paired with a fabricated solution, injecting arbitrary values into the pool config.

### Finding Description
`fetch_coin_spend` checks `puzzle.get_tree_hash() == coin.puzzle_hash` but accepts the solution field verbatim. `apply_state_transition` performs only a coin-name continuity check before storing the spend. `update_pool_config` then reads `pool_url`, `target_puzzle_hash`, and `owner_public_key` from `solution_to_pool_state()` — which is a pure parser with no cryptographic verification — and writes them to disk. No layer between peer response and config write revalidates the solution against the puzzle or checks the embedded pool state for consistency with the on-chain singleton.

### Impact Explanation
A malicious peer injects a fabricated `pool_url` pointing to an attacker-controlled server. The farmer reads this URL from `PoolingShareState`, connects to the malicious server, and submits proofs of space. The attacker collects proofs without paying rewards. `owner_public_key` corruption additionally breaks `find_authentication_sk`, preventing the farmer from authenticating with any legitimate pool. `payout_instructions` (the actual XCH reward address) is locally generated and is not affected, so direct on-chain XCH theft is not achievable through this path alone.

### Likelihood Explanation
Requires the wallet to connect to a malicious peer. Wallets typically connect to their own full node, but they also connect to external peers during sync. The attack is passive (no user interaction needed beyond the wallet syncing a spent singleton) and leaves no obvious trace.

### Recommendation
In `apply_state_transition`, after storing the spend, call `_verify_pool_state` on the pool state extracted from the solution and reject (return `False`) if it fails. Additionally, verify the solution is consistent with the stored puzzle by running the CLVM puzzle against the solution and checking that the output conditions match the expected singleton structure, before calling `update_pool_config`.

### Proof of Concept
1. Operate a peer node that intercepts `request_puzzle_solution` for a known pool singleton coin.
2. Return the correct `puzzle_reveal` (matching `coin.puzzle_hash`) but substitute a crafted `solution` whose inner solution encodes `pool_url = "https://attacker.example.com"` and an arbitrary `target_puzzle_hash`.
3. Connect the victim wallet to this peer and trigger a sync that processes the spent singleton.
4. Observe that `PoolingShareState` on disk now contains `pool_url = "https://attacker.example.com"`.
5. Start the farmer; observe it contacts the attacker's server with valid proof-of-space submissions.

async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:
    solution_response = await peer.call_api(
        FullNodeAPI.request_puzzle_solution, RequestPuzzleSolution(coin.name(), height)
    )
    if solution_response is None or not isinstance(solution_response, RespondPuzzleSolution):
        raise PeerRequestException(f"Was not able to obtain solution {solution_response}")
    coin_id = coin.name()
    if solution_response.response.puzzle.get_tree_hash() != coin.puzzle_hash:
        raise PeerRequestException(f"Peer returned wrong puzzle hash for coin {coin_id}")
    if solution_response.response.coin_name != coin_id:
        raise PeerRequestException(f"Peer returned wrong coin name in puzzle solution for coin {coin_id}")

    return make_spend(
        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )

    async def apply_state_transition(
        self, new_state: CoinSpend, block_height: uint32, action_scope: WalletActionScope
    ) -> bool:
        """
        Updates the Pool state (including DB) with new singleton spends.
        The DB must be committed after calling this method. All validation should be done here. Returns True iff
        the spend is a valid transition spend for the singleton, False otherwise.
        """
        tip: tuple[uint32, CoinSpend] = await self.get_tip()
        tip_spend = tip[1]

        tip_coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(tip_spend)
        assert tip_coin is not None
        spent_coin_name: bytes32 = tip_coin.name()

        if spent_coin_name != new_state.coin.name():
            history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
            if new_state.coin.name() in [sp.coin.name() for _, sp in history]:
                self.log.info(f"Already have state transition: {new_state.coin.name().hex()}")
            else:
                self.log.warning(
                    f"Failed to apply state transition. tip: {tip_coin} new_state: {new_state} height {block_height}"
                )
            return False

        await self.wallet_state_manager.pool_store.add_spend(self.wallet_id, new_state, block_height)
        tip_spend = (await self.get_tip())[1]
        self.log.info(f"New PoolWallet singleton tip_coin: {tip_spend} farmed at height {block_height}")

        # If we have reached the target state, resets it to None. Loops back to get current state
        for _, added_spend in reversed(
            await self.wallet_state_manager.pool_store.get_spends_for_wallet(self.wallet_id)
        ):
            latest_state: PoolState | None = solution_to_pool_state(added_spend)
            if latest_state is not None:
                if self.target_state == latest_state:
                    self.target_state = None
                    self.next_transaction_fee = uint64(0)
                    self.next_tx_config = DEFAULT_TX_CONFIG
                break

        await self.update_pool_config(action_scope)
        return True

def solution_to_pool_state(full_spend: CoinSpend) -> PoolState | None:
    full_solution_ser: SerializedProgram = full_spend.solution
    full_solution: Program = Program.from_bytes(bytes(full_solution_ser))

    if full_spend.coin.puzzle_hash == SINGLETON_LAUNCHER_HASH:
        # Launcher spend
        extra_data: Program = full_solution.rest().rest().first()
        return pool_state_from_extra_data(extra_data)

    # Not launcher spend
    inner_solution: Program = full_solution.rest().rest().first()

    # Spend which is not absorb, and is not the launcher
    num_args = len(inner_solution.as_python())
    assert num_args in {2, 3}

    if num_args == 2:
        # pool member
        if inner_solution.rest().first().as_int() != 0:
            return None

        # This is referred to as p1 in the chialisp code
        # spend_type is absorbing money if p1 is a cons box, spend_type is escape if p1 is an atom
        # TODO: The comment above, and in the CLVM, seems wrong
        extra_data = inner_solution.first()
        if isinstance(extra_data.as_python(), bytes):
            # Absorbing
            return None
        return pool_state_from_extra_data(extra_data)
    else:
        # pool waitingroom
        if inner_solution.first().as_int() == 0:
            return None
        extra_data = inner_solution.rest().first()
        return pool_state_from_extra_data(extra_data)
