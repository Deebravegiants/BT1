Audit Report

## Title
Missing Chainlink Price Feed Staleness Validation in `getAssetPrice()` Allows Stale Prices to Inflate rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `roundId`, and `answeredInRound`, accepting stale or incomplete Chainlink rounds without reversion. The same codebase already implements all three validations in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the omission is a code-level defect in the protocol's own oracle wrapper. A stale inflated price flows directly into the rsETH minting formula, allowing any depositor to receive excess rsETH at the expense of existing holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at L52–54 reads:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`) are available but only `answer` is consumed. No check is made on:
- `updatedAt == 0` (incomplete round)
- `answeredInRound < roundId` (round not yet finalized / stale)
- `block.timestamp - updatedAt > heartbeat` (time-based staleness)

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` at L27–32 performs all three:

```solidity
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated price propagates through the following call chain:

1. `ChainlinkPriceOracle.getAssetPrice(asset)` → `LRTOracle.getAssetPrice(asset)` (L156–158)
2. `LRTOracle.getAssetPrice(asset)` → `LRTDepositPool.getRsETHAmountToMint()` (L519–521):
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```
3. `getRsETHAmountToMint()` → `_beforeDeposit()` → `depositAsset()` / `depositETH()` (L99–118, L648–670)

Additionally, `LRTOracle._getTotalEthInProtocol()` (L336–344) calls `getAssetPrice(asset)` for every supported asset to compute TVL, which feeds `_updateRsETHPrice()`. A stale inflated price here inflates `totalETHInProtocol`, causing excess fee rsETH to be minted to the treasury.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (L252–266) provides partial mitigation for the TVL path only — it reverts if the computed rsETH price increase exceeds the configured threshold. However, it does not protect the direct minting path in `depositAsset()`, where `rsETHPrice` is the stored (pre-update) value and is not recalculated during the deposit.

## Impact Explanation
**High — Theft of unclaimed yield.**

When a Chainlink feed for a supported LST (e.g., stETH/ETH) goes stale with an inflated last-known price, any depositor calling `depositAsset()` receives:

```
rsethAmountToMint = (depositAmount × staleInflatedAssetPrice) / rsETHPrice
```

The excess rsETH minted to the depositor dilutes the share value of all existing rsETH holders, constituting theft of unclaimed yield from existing holders. The same stale price fed into `_updateRsETHPrice()` inflates `totalETHInProtocol`, causing the protocol to compute a falsely elevated TVL increase and mint excess fee rsETH to the treasury, further diluting holders.

## Likelihood Explanation
Chainlink LST/ETH feeds (stETH/ETH, cbETH/ETH, rETH/ETH) on mainnet operate with 24-hour heartbeat intervals. During periods of low volatility, the heartbeat is the only trigger for an update, meaning a 24-hour stale window is a realistic and recurring scenario. No special role, setup, or attacker capability is required — any unprivileged user can call `depositAsset()` at any time. The condition is passively triggered by normal Chainlink oracle behavior, not by active manipulation.

## Recommendation
Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
// Optionally: if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Define `MAX_STALENESS` per asset to match the expected heartbeat interval (e.g., 24 hours + buffer for LST feeds).

## Proof of Concept

**Foundry fork test plan:**

```solidity
// Fork mainnet at a block where stETH/ETH Chainlink feed is stale
// (or mock the feed to return a stale round)
function test_stalePrice_inflatesRsETHMint() public {
    // 1. Deploy/fork with ChainlinkPriceOracle pointing to a mock feed
    MockChainlinkFeed feed = new MockChainlinkFeed();
    // Set answeredInRound < roundId (stale) and inflated price
    feed.setRoundData(
        roundId: 10,
        answer: 1.05e18,   // 5% above true price
        startedAt: block.timestamp - 48 hours,
        updatedAt: block.timestamp - 48 hours,  // 48h stale
        answeredInRound: 9   // answeredInRound < roundId
    );

    // 2. Record existing holder's share value before deposit
    uint256 rsETHPriceBefore = lrtOracle.rsETHPrice();

    // 3. Attacker deposits stETH
    vm.prank(attacker);
    lrtDepositPool.depositAsset(stETH, 1e18, 0, "");

    // 4. Update rsETH price to reflect new TVL
    lrtOracle.updateRSETHPrice();

    // 5. Assert rsETH price decreased (existing holders diluted)
    uint256 rsETHPriceAfter = lrtOracle.rsETHPrice();
    assertLt(rsETHPriceAfter, rsETHPriceBefore);

    // 6. Assert attacker received more rsETH than fair value
    uint256 fairRsETH = (1e18 * truePriceFeed.getAssetPrice(stETH)) / rsETHPriceBefore;
    assertGt(IRSETH(rsETH).balanceOf(attacker), fairRsETH);
}
```

The test demonstrates that without staleness validation, a depositor mints excess rsETH using a stale inflated price, reducing the per-share value for all existing rsETH holders.
