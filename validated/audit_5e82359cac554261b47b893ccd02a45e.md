### Title
Unrefunded Deposit in `submit_participant_info` Re-submission Path Causes Permanent Fund Loss - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `submit_participant_info` function is marked `#[payable]` but only executes deposit-refund logic in one of its two branches. When an existing participant re-submits their attestation, any NEAR tokens attached to the call are permanently locked in the contract with no recovery path. This is the direct analog to the external report's "uninitialized recipient causes funds to be lost" class: instead of fees routed to address(0), deposits are routed to the contract itself with no mechanism to retrieve them.

---

### Finding Description

In `crates/contract/src/lib.rs`, `submit_participant_info` (line 758) is decorated `#[payable]`, meaning it unconditionally accepts any attached deposit. After calling `self.tee_state.add_participant(...)`, the function computes two flags:

```rust
// lib.rs:817-824
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;
```

The entire deposit-handling block — including the refund — is gated behind this flag:

```rust
// lib.rs:826-849
if attestation_storage_must_be_paid_by_caller {
    // ... check deposit sufficiency and refund excess
}
```

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — which happens whenever the caller **is** a current participant (`voter_account()` succeeds) **and** the submission is a re-submission rather than a new entry (`is_new_attestation = false`) — the function:

1. Accepts any attached deposit (no guard, function is `#[payable]`)
2. Performs no sufficiency check
3. Issues **no refund**

Any NEAR tokens attached to a re-submission call by an existing participant are permanently locked in the contract. There is no administrative withdrawal function or recovery path for these funds.

This is a normal operational scenario: MPC nodes periodically re-submit their attestations to keep them fresh. The re-submission path is the expected steady-state call pattern for running participants.

---

### Impact Explanation

Attached deposits are permanently locked in the MPC contract. This breaks the accounting invariant that `#[payable]` functions must either consume deposits for a defined purpose or refund them. The impact maps to the allowed medium scope: **balance/accounting invariant violation that causes permanent fund loss without requiring network-level DoS or operator misconfiguration**. The funds are not recoverable by any on-chain mechanism.

---

### Likelihood Explanation

Existing participants re-submit attestations periodically as a normal operational duty. If a node operator, automated client, or misconfigured script attaches any non-zero deposit during a re-submission (e.g., following incorrect documentation, copying a first-submission call template, or due to a client bug), the funds are silently and permanently lost. The NEAR SDK does not warn callers that their deposit will not be refunded. Likelihood is **low-medium**: the scenario requires an accidental deposit attachment, but the re-submission code path is exercised continuously in production.

---

### Recommendation

Add an explicit refund for any attached deposit in the branch where `attestation_storage_must_be_paid_by_caller` is false:

```rust
if attestation_storage_must_be_paid_by_caller {
    // existing logic
} else {
    // Refund any accidentally attached deposit; re-submissions by existing
    // participants incur no storage cost and should not consume deposits.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, assert that no deposit is attached in the re-submission path to make the invariant explicit and fail fast.

---

### Proof of Concept

1. Deploy the MPC contract and initialize with participants A, B, C.
2. Participant A calls `submit_participant_info` with a valid attestation for the first time. `add_participant` returns `ParticipantInsertion::NewlyInsertedParticipant`, so `is_new_attestation = true`, the deposit block runs, and excess is refunded correctly.
3. Participant A calls `submit_participant_info` again (re-submission, e.g., attestation refresh) and accidentally attaches `1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR).
4. `add_participant` returns a non-`NewlyInsertedParticipant` variant → `is_new_attestation = false`.
5. `self.voter_account()` succeeds for A → `caller_is_not_participant = false`.
6. `attestation_storage_must_be_paid_by_caller = false || false = false`.
7. The deposit-handling block at line 826 is skipped entirely.
8. The call returns `Ok(())`. The 1 NEAR is now locked in the contract with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2)

            // `saturating_sub`: if a re-submission shrinks the entry, charge nothing
            // rather than underflow. Intentional asymmetry: we do not refund freed bytes
            // either — the caller already paid for the larger entry, and we'd rather
            // accept that asymmetry than open a refund path for payload-shrinking games.
            let storage_used = env::storage_usage().saturating_sub(initial_storage);
            let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
            let attached = env::attached_deposit();

            if attached < cost {
                return Err(InvalidParameters::InsufficientDeposit {
                    attached: attached.as_yoctonear(),
                    required: cost.as_yoctonear(),
                }
                .into());
            }

            // Refund the difference if the proposer attached more than required
            if let Some(diff) = attached.checked_sub(cost)
                && diff > NearToken::from_yoctonear(0)
            {
                Promise::new(account_id).transfer(diff).detach();
            }
