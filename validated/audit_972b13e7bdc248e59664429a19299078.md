### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Breaks Backing Guarantee — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

When a user calls `initTransfer` with a fee-on-transfer ERC20 token, the EVM bridge receives fewer tokens than the nominal `amount` due to the transfer fee, but the `InitTransfer` event records the full `amount`. The NEAR bridge processes this event and credits the user with the full `amount`, creating unbacked supply and permanently breaking the 1:1 backing guarantee.

---

### Finding Description

In `OmniBridge.initTransfer()`, for non-bridge ERC20 tokens (the `else` branch), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // nominal amount, not actual received
);
``` [1](#0-0) 

Immediately after, the function emits the `InitTransfer` event using the same nominal `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // nominal, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

For a fee-on-transfer ERC20, `safeTransferFrom(..., amount)` causes the bridge to receive only `amount - transferFee` tokens, but the event records `amount`. The NEAR bridge contract processes this event and stores a `TransferMessage` with `amount` as the credited value: [3](#0-2) 

The NEAR side then locks `amount` tokens in its accounting (`lock_tokens_if_needed`), while the EVM bridge only holds `amount - transferFee` as actual backing: [4](#0-3) 

There is no balance-before/balance-after check anywhere in `initTransfer` to detect the shortfall.

---

### Impact Explanation

**High — Asset-accounting divergence that breaks backing guarantees.**

Each `initTransfer` call with a fee-on-transfer token inflates the NEAR-side credit by exactly `transferFee` tokens relative to what the EVM bridge actually holds. This discrepancy accumulates across all such transfers. When users later bridge back (EVM `finTransfer`), the bridge attempts to release the full credited `amount`:

```solidity
IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
``` [5](#0-4) 

The EVM bridge's token reserves are insufficient to cover all outstanding NEAR-side credits, causing later withdrawals to fail or the bridge to become insolvent. Locked-token accounting on NEAR diverges from actual EVM holdings, permanently breaking the backing guarantee.

---

### Likelihood Explanation

Any unprivileged user can call `initTransfer` with any ERC20 token address — there is no whitelist check for the non-bridge-token path. The only prerequisite is that the token is registered in the bridge's cross-chain mapping (controlled by admin via `deployToken`/`addCustomToken`). Once a fee-on-transfer token is registered (a realistic scenario for tokens with protocol fees), every user transfer silently inflates NEAR-side supply beyond EVM backing. No special role, leaked key, or external dependency compromise is required.

---

### Recommendation

Use a balance-before/balance-after pattern to determine the actual received amount, and emit the event with the actual value:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "Zero received");
// Use actualReceived in place of amount for the event and downstream accounting
```

Alternatively, explicitly disallow fee-on-transfer tokens by requiring `actualReceived == amount`.

---

### Proof of Concept

1. Admin registers a fee-on-transfer ERC20 token (e.g., 1% fee) in the bridge via `addCustomToken` or `deployToken`.
2. Attacker (or any user) calls `initTransfer(feeToken, 1000, 0, 0, "alice.near", "")`.
3. Bridge receives `990` tokens (`safeTransferFrom` deducts 1% fee), but emits `InitTransfer` with `amount = 1000`.
4. NEAR bridge processes the event, credits `alice.near` with `1000` tokens, and locks `1000` in `locked_tokens`.
5. Repeat N times: EVM bridge holds `990*N` tokens; NEAR side has `1000*N` credits.
6. When users bridge back, the `(N * 10)`-token shortfall causes the last `~1%` of withdrawals to fail permanently — frozen redemption path and insolvency.

        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(current_amount) = self.locked_tokens.get(&key) else {

        let new_amount = current_amount
            .checked_add(amount)

            chain_kind,
            token_id: token_id.clone(),
