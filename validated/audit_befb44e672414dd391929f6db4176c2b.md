### Title
Dynamic Config Update Immediately Invalidates Queued Future-Round Proposals, Causing Consensus Liveness Failure — (File: crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs)

### Summary

When `validate_proposal` receives a proposal for a future round, it stores the raw `(ProposalInit, timeout, content_receiver)` tuple in `queued_proposals` **without** capturing the current `ProposalInitValidation` criteria. Later, when `set_height_and_round` advances to that round, it calls `update_dynamic_config()` **before** dequeuing and validating the stored proposal. The queued proposal is therefore validated against the **new** dynamic config — including a potentially changed `override_l2_gas_price_fri` or `gas_price_params` — rather than the config that was active when the proposal was received. A proposer that built its proposal under the old config will have its proposal rejected by a validator that has since updated, stalling consensus for the height.

### Finding Description

**Step 1 — Proposal queued without validation criteria snapshot.**

In `validate_proposal`, a future-round proposal is stored as a raw tuple:

```rust
std::cmp::Ordering::Greater => {
    trace!("Queueing proposal for future round.");
    self.queued_proposals
        .insert(init.round, ((init, timeout, content_receiver), fin_sender));
    fin_receiver
}
```

No `ProposalInitValidation` is computed at this point. [1](#0-0) 

**Step 2 — Dynamic config updated before dequeue.**

When `set_height_and_round` advances to a new round, it calls `update_dynamic_config()` first, then dequeues the stored proposal and constructs `ProposalInitValidation` from the **now-updated** config:

```rust
self.interrupt_active_proposal().await;
self.current_round = round;
self.update_dynamic_config().await;          // ← config replaced here

// ... then dequeues and builds validation criteria from new config:
let proposal_init_validation = ProposalInitValidation {
    ...
    l2_gas_price_fri: self
        .config
        .dynamic_config
        .override_l2_gas_price_fri
        .map(GasPrice)
        .unwrap_or(self.l2_gas_price),
    ...
};
``` [2](#0-1) 

**Step 3 — Exact-equality check rejects the proposal.**

`is_proposal_init_valid` performs an exact equality check on `l2_gas_price_fri`:

```rust
if !(init_proposed.height == proposal_init_validation.height
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
{
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
``` [3](#0-2) 

If `override_l2_gas_price_fri` changed from `None` (validator uses `self.l2_gas_price`) to `Some(X)` (validator uses `GasPrice(X)`), the proposer's `l2_gas_price_fri` — built under the old config — will not equal `GasPrice(X)`, and the proposal is rejected.

The same applies to `gas_price_params` (L1 gas multipliers, min/max bounds), which are also captured from the new config at dequeue time: [4](#0-3) 

**The `update_dynamic_config` mechanism** pulls a fresh `ContextDynamicConfig` from the `ConfigManager` on every height or round advance: [5](#0-4) 

The `ConfigManagerRunner` polls for config-file changes every 60 seconds by default and pushes updates immediately: [6](#0-5) 

**The `ValidationParams` type alias** confirms that no validation snapshot is stored alongside the queued proposal: [7](#0-6) 

### Impact Explanation

A valid proposal — one whose `l2_gas_price_fri` and L1 gas prices were computed correctly under the config active at receipt time — is rejected by the validator after a dynamic config update. The `fin_sender` is dropped, the `fin_receiver` returns `Canceled`, and consensus cannot commit the block for that height/round. All transactions in the proposal are not sequenced despite being valid. This matches **High: Mempool/gateway/RPC admission rejects valid transactions before sequencing** (extended to the consensus admission path per the Sequencer-Specific Audit Pivots).

### Likelihood Explanation

The `ConfigManagerRunner` is enabled in production deployments (`enable_config_updates: true`, 60-second poll interval). A config change to `override_l2_gas_price_fri` or any `gas_price_params` field during a multi-round height (e.g., during a timeout-induced round skip) will trigger the bug. The window is every round advance where a proposal was pre-queued. [8](#0-7) 

### Recommendation

Compute and store `ProposalInitValidation` at the time the proposal is queued (i.e., inside the `std::cmp::Ordering::Greater` arm of `validate_proposal`), using the config that is current at receipt time. When the proposal is later dequeued in `set_height_and_round`, pass the pre-computed `ProposalInitValidation` directly to `validate_current_round_proposal` instead of reconstructing it from the (now-updated) dynamic config. This mirrors the current-round path, which computes `ProposalInitValidation` immediately at receipt.

### Proof of Concept

1. Node is at height H, round 0. Dynamic config has `override_l2_gas_price_fri = None`; validator reference is `self.l2_gas_price = P`.
2. Proposer for round 1 sends a proposal with `l2_gas_price_fri = P`. The validator receives it while still at round 0 and queues it: `queued_proposals.insert(1, ((init_with_P, timeout, rx), fin_tx))`.
3. Round 0 times out. Consensus calls `set_height_and_round(H, 1)`.
4. Inside `set_height_and_round`, `update_dynamic_config()` fires. The config file now has `override_l2_gas_price_fri = Some(Q)` where `Q ≠ P`. `self.config.dynamic_config.override_l2_gas_price_fri` is now `Some(Q)`.
5. The queued proposal is dequeued. `ProposalInitValidation` is built with `l2_gas_price_fri = GasPrice(Q)`.
6. `is_proposal_init_valid` checks `init_proposed.l2_gas_price_fri (= P) == proposal_init_validation.l2_gas_price_fri (= Q)` → **false** → `InvalidProposalInit` error.
7. `fin_sender` is dropped; `fin_receiver` returns `Canceled`. Consensus cannot commit height H, round 1. The proposer's valid block is discarded.
