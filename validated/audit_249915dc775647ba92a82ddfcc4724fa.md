Audit Report

## Title
Uninitialized `tokenFeeBps` Mapping Allows Zero-Fee Token Deposits After `addSupportedToken()` - (File: contracts/pools/RSETHPool.sol)

## Summary

`RSETHPool.sol` stores per-token fee rates in `tokenFeeBps[token]`, but `addSupportedToken()` never initializes this mapping entry. Because Solidity mappings default to zero, any depositor who calls `deposit(token, amount, referralId)` before the admin separately calls `setTokenFeeBps(token, feeBps)` pays a 0% fee and receives the full token value converted to rsETH with no fee deducted, permanently depriving the protocol of that fee revenue.

## Finding Description

`tokenFeeBps` is declared at line 88 of `contracts/pools/RSETHPool.sol`:

```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
```

`addSupportedToken()` (lines 637–656) sets `supportedTokenOracle[token]` and `tokenBridge[token]` but never touches `tokenFeeBps[token]`. The token is immediately live for deposits via the `onlySupportedToken` modifier (which only checks `supportedTokenOracle[token] != address(0)`, line 101).

Setting the fee is a completely separate privileged call, `setTokenFeeBps()` (lines 583–594), gated by `DEFAULT_ADMIN_ROLE` — a different role from the `TIMELOCK_ROLE` required by `addSupportedToken()`. These two roles cannot atomically execute both calls in a single transaction without a batching script, and a timelock on `addSupportedToken` may further widen the gap.

The public `deposit(address token, uint256 amount, string referralId)` function (lines 284–305) calls `viewSwapRsETHAmountAndFee(amount, token)` (lines 326–347), which reads `tokenFeeBps[token]` directly:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];   // returns 0
fee = amount * feeBpsForToken / 10_000;         // fee = 0
uint256 amountAfterFee = amount - fee;          // full amount passes through
```

`feeEarnedInToken[token] += 0` — the protocol accumulates nothing. The depositor receives rsETH equivalent to the full token value. No existing guard prevents this: `whenNotPaused`, `nonReentrant`, and `onlySupportedToken` all pass normally.

## Impact Explanation

Every deposit made in the window between `addSupportedToken()` and `setTokenFeeBps()` pays zero protocol fee. The protocol's `feeEarnedInToken[token]` permanently records nothing for those deposits; the fee revenue is irrecoverably lost. This constitutes **theft of unclaimed yield** (protocol fee revenue), matching the allowed High-severity impact class.

## Likelihood Explanation

The window is structurally guaranteed every time a new token is listed: `addSupportedToken` (TIMELOCK_ROLE) and `setTokenFeeBps` (DEFAULT_ADMIN_ROLE) are separate roles requiring separate transactions. An attacker monitoring the mempool can front-run or immediately follow `addSupportedToken()` with a large deposit before `setTokenFeeBps()` is confirmed. Even without front-running, any organic deposit in the gap — which may span multiple blocks or longer if the admin delays — is affected. The attack requires no special privileges, only the ability to hold the newly supported token and call the public `deposit` function.

## Recommendation

Initialize `tokenFeeBps[token]` inside `addSupportedToken()` by accepting a `_feeBps` parameter:

```diff
function addSupportedToken(
    address token,
    address oracle,
-   address bridge
+   address bridge,
+   uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    // existing checks ...
+   if (_feeBps > 10_000) revert InvalidFeeAmount();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
+   tokenFeeBps[token] = _feeBps;
+   emit TokenFeeBpsSet(token, _feeBps);
    emit AddSupportedToken(token, oracle, bridge);
}
```

This eliminates the uninitialized window entirely, mirroring how `feeBps` is set atomically in `initialize()` for ETH deposits.

## Proof of Concept

**Setup:** Deploy `RSETHPool` on a local Arbitrum fork. Add `rETH` via `addSupportedToken(rETH, rETHOracle, rETHBridge)`. Do NOT call `setTokenFeeBps`.

**Foundry test sequence:**

```solidity
// After addSupportedToken, before setTokenFeeBps:
uint256 depositAmount = 100e18;
deal(address(rETH), attacker, depositAmount);
vm.startPrank(attacker);
rETH.approve(address(pool), depositAmount);
pool.deposit(address(rETH), depositAmount, "");
vm.stopPrank();

// Assert fee earned is zero
assertEq(pool.feeEarnedInToken(address(rETH)), 0);

// Now set fee and deposit again with same amount
vm.prank(admin);
pool.setTokenFeeBps(address(rETH), 50); // 0.5%
deal(address(rETH), attacker2, depositAmount);
vm.startPrank(attacker2);
rETH.approve(address(pool), depositAmount);
pool.deposit(address(rETH), depositAmount, "");
vm.stopPrank();

// Assert fee earned is now non-zero
assertGt(pool.feeEarnedInToken(address(rETH)), 0);
```

The first deposit yields `feeEarnedInToken[rETH] == 0`; the second yields the expected fee. The difference is the permanently lost protocol revenue.
