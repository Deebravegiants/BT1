### Title
Open Server-Side Request Forgery via Unvalidated Nginx Proxy — (`tools/debug-ui/nginx.conf`)

### Summary
The `tools/debug-ui` nginx configuration exposes an open HTTP proxy at `/api-proxy/<TARGET>/<PATH>` that forwards requests to any host specified in the URL path with no validation. An unprivileged external user who can reach the deployed debug-UI server can force it to make server-side HTTP requests to arbitrary hosts, including the cloud instance metadata endpoint (`169.254.169.254`), internal network services, or any other reachable address.

### Finding Description
`tools/debug-ui/nginx.conf` defines a catch-all proxy location:

```nginx
location ~ ^/api-proxy/([^/]+)/(.*)$ {
    set $target_host $1;
    set $api_path $2;
    proxy_pass http://$target_host/$api_path$is_args$args;
    ...
    add_header 'Access-Control-Allow-Origin' '*';
}
``` [1](#0-0) 

The `$target_host` variable is taken verbatim from the first path segment of the URL. There is no allowlist, blocklist, or any form of host validation. The service is containerized (a `Dockerfile` is present) and the nginx.conf comment explicitly states it is "suitable for Cloud Run to resolve external domains," confirming this is a production cloud deployment, not local-only tooling. [2](#0-1) 

The client-side code in `tools/debug-ui/src/api.tsx` routes all HTTPS traffic through this proxy, meaning the proxy is the intended production path for the deployed service:

```typescript
if (protocol === 'https:') {
    return getProxyUrl(addr, endpoint);  // → /api-proxy/${addr}/${endpoint}
}
``` [3](#0-2) 

### Impact Explanation
An attacker who can reach the debug-UI server (which is publicly deployed on Cloud Run with `Access-Control-Allow-Origin: *`) can issue:

```
GET /api-proxy/169.254.169.254/latest/meta-data/iam/security-credentials/ HTTP/1.1
Host: <debug-ui-cloud-run-host>
```

The nginx server will proxy this to `http://169.254.169.254/latest/meta-data/iam/security-credentials/` and return the response — including temporary AWS/GCP/Azure IAM credentials — directly to the attacker. On GCP Cloud Run specifically, the metadata server at `169.254.169.254` exposes OAuth2 access tokens for the service account, enabling full impersonation of the Cloud Run service identity. The attacker can also reach any internal VPC service reachable from the nginx container.

**Corrupted/leaked value**: Cloud instance IAM credentials (access tokens, secret keys), internal service responses, VPC-internal data.

### Likelihood Explanation
The debug-UI is deployed publicly on Cloud Run (confirmed by the nginx.conf resolver comment and the presence of a `Dockerfile`). The proxy endpoint requires no authentication and is reachable by any external user. The attack requires only a single HTTP GET request with a crafted path. No special knowledge of the system is needed beyond knowing the Cloud Run hostname. [4](#0-3) 

### Recommendation
Replace the open proxy with an allowlist of permitted target addresses (i.e., only known nearcore node addresses). Alternatively, validate that `$target_host` matches a strict pattern (e.g., `^[a-zA-Z0-9.-]+:[0-9]+$` combined with an IP/hostname allowlist) and explicitly deny requests to RFC 1918 private ranges, link-local (`169.254.0.0/16`), and loopback addresses. If the proxy is only needed to reach operator-configured nearcore nodes, the allowed targets should be hardcoded or injected at deploy time rather than taken from the request URL.

### Proof of Concept

```
# Reach cloud metadata on AWS/GCP/Azure/DigitalOcean:
GET /api-proxy/169.254.169.254/latest/meta-data/iam/security-credentials/ HTTP/1.1
Host: <debug-ui-cloud-run-hostname>

# Reach an internal VPC service:
GET /api-proxy/10.128.0.5:8080/internal/admin HTTP/1.1
Host: <debug-ui-cloud-run-hostname>
```

nginx resolves `$target_host` using the configured resolver (`8.8.8.8`) for hostnames, but for bare IP literals like `169.254.169.254` no DNS resolution is needed — nginx proxies directly. The response body is returned verbatim to the caller with `Access-Control-Allow-Origin: *`, making the data readable from any browser origin as well. [5](#0-4)

server {
    listen 80;
    root /var/www/html;
    
    # 1. Resolver is required when using variables in proxy_pass.
    # 8.8.8.8 is Google DNS; suitable for Cloud Run to resolve external domains.
    resolver 8.8.8.8; 

    location /static {
        alias /var/www/html/static;
    }

    # 2. The Proxy Block
    # Pattern matches: /api-proxy/TARGET_ADDRESS/REAL_API_PATH
    location ~ ^/api-proxy/([^/]+)/(.*)$ {
        # Capture the target address ($1) and the rest of the path ($2)
        set $target_host $1;
        set $api_path $2;

        # Proxy configuration
        proxy_pass http://$target_host/$api_path$is_args$args;
        
        # Standard proxy headers
        proxy_set_header Host $target_host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        # IMPORTANT: Hide the fact that this is a proxy from the browser
        # to avoid CORS issues (optional, depending on your API)
        add_header 'Access-Control-Allow-Origin' '*';
    }
```
