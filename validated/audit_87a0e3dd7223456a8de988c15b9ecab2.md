### Title
Unauthenticated Coordinator Header Allows External Clients to Bypass Sharded Query Routing - (File: chain/jsonrpc/src/lib.rs)

### Summary
The `rpc_handler` function in `chain/jsonrpc/src/lib.rs` determines whether an incoming HTTP request originates from a trusted internal coordinator or an external user solely by checking for the presence of a plain HTTP header (`SHARDED_RPC_COORDINATOR_HEADER`). No authentication or secret is required. Any unprivileged external client can set this header and be treated as a coordinator, bypassing the sharded query routing path and forcing local query handling on a node that may not track the relevant shard.

### Finding Description
In `rpc_handler` at line 2716, the request source is classified as follows:

```rust
let source = if headers.contains_key(SHARDED_RPC_COORDINATOR_HEADER) {
    RequestSource::Coordinator
} else {
    RequestSource::User
};
``` [1](#0-0) 

There is no secret, token, IP allowlist, or cryptographic check on this header. Any HTTP client that adds the coordinator header is unconditionally elevated to `RequestSource::Coordinator`.

This `source` value is then used in `process_request_internal` to route the `query` method down a different code path:

```rust
let result = match source {
    RequestSource::User => self.query_sharded(params).await,
    RequestSource::Coordinator => process_query_response(self.query(params).await),
};
``` [2](#0-1) 

`RequestSource` is used in 21 places across the file, meaning the coordinator/user distinction affects multiple RPC method behaviors beyond just `query`. [3](#0-2) 

### Impact Explanation
In a sharded RPC deployment, `query_sharded` routes the request to the shard node that actually tracks the relevant shard. When an external attacker forces `RequestSource::Coordinator`, the node instead calls `self.query()` — local handling — on a node that may not track the queried shard. This produces an incorrect or stale RPC result (wrong account state, wrong contract state, wrong access key data, etc.) for the attacker's own query, and more critically, it bypasses the sharded routing invariant that the coordinator header is supposed to be an internal-only signal. The corrupted value is the RPC query result returned to the caller.

### Likelihood Explanation
The public JSON-RPC endpoint (`/`) is reachable by any external user. Setting an arbitrary HTTP header requires no privileges. The attack requires only a single crafted HTTP request. Any user of a sharded NEAR RPC deployment can trigger this.

### Recommendation
Authenticate the coordinator header using a shared secret, bearer token, or by restricting the RPC endpoint to only accept coordinator-sourced requests from a trusted internal network address (e.g., via IP allowlist or mutual TLS). The header alone must not be sufficient to elevate trust level.

### Proof of Concept
```bash
# Any external user can impersonate the coordinator by adding the header:
curl -X POST https://<node-rpc-endpoint>/ \
  -H "Content-Type: application/json" \
  -H "X-Sharded-Rpc-Coordinator: true" \
  -d '{
    "jsonrpc": "2.0",
    "id": "dontcare",
    "method": "query",
    "params": {
      "request_type": "view_account",
      "finality": "final",
      "account_id": "victim.near"
    }
  }'
# The node processes this via self.query() (local handling) instead of
# self.query_sharded() (correct shard routing), returning stale or
# incorrect account state if the node does not track the relevant shard.
```
