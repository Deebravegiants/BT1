### Title
Single Malicious Participant Can Inject Arbitrary CKD Response for `AppPublicKey` Variant, Delivering Attacker-Controlled Key to User - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` in `crates/contract/src/lib.rs` performs a cryptographic output check (`ckd_output_check`) only when the request uses the `AppPublicKeyPV` variant. When the request uses the standard `AppPublicKey` variant, **no output verification is performed at all**. Any single attested participant can call `respond_ckd` with an arbitrary `CKDResponse` (`big_y`, `big_c`) for any pending `AppPublicKey` request, and the contract will accept and deliver it to the user. By setting `big_y = 0` (identity point) and `big_c = k·G1` for a scalar `k` of their choice, the attacker ensures the user decrypts to a key the attacker fully controls.

### Finding Description

The `respond_ckd` entrypoint contains a conditional output check:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` verifies the BLS pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC master key and the user's app identity. For `AppPublicKey`, the arm is a no-op — the response bytes are accepted unconditionally and forwarded to the user via `resolve_yields_for`.

The `AppPublicKey` variant is the default, non-publicly-verifiable variant used by TEE applications. The CKD decryption is `big_s = big_c − a · big_y`, where `a` is the user's private scalar. If an attacker submits `big_y = 0` (the G1 identity point) and `big_c = k·G1` for a scalar `k` they choose, the user decrypts to `k·G1` regardless of their private key `a`. The attacker knows `k` and therefore controls the user's derived key.

The analog to the external report is exact: `updateTokenPool` checks `sender == tokenAddress` (correct caller) but does not verify *which* FA1.2 function triggered the callback, so both `getBalance` and `getAllowance` satisfy the check and allow arbitrary value injection. Here, `respond_ckd` checks the caller is an attested participant (correct caller check) but does not verify the output for the `AppPublicKey` variant — both `AppPublicKey` and `AppPublicKeyPV` satisfy the caller check, but only `AppPublicKeyPV` triggers the output verification gate. ### Impact Explanation

**Critical. Unauthorized confidential key derivation output without the required participant authorization.**

A single malicious participant delivers a `CKDResponse` where `big_y = 0` and `big_c = k·G1`. The user's TEE application decrypts to `k·G1`, a key the attacker knows. Any assets or secrets the user subsequently protects with that derived key are under attacker control. The MPC network's threshold guarantee is entirely bypassed for the `AppPublicKey` variant: one participant out of `n` is sufficient to compromise the output.

### Likelihood Explanation

**High.** The `AppPublicKey` variant is the standard (non-publicly-verifiable) variant shown as the default in the CLI documentation and used in the primary e2e test. Any single attested participant — one that has passed TEE attestation and is registered in the contract — can execute this attack against any pending `AppPublicKey` CKD request. No coordination with other participants is required. The attacker only needs to submit their crafted `respond_ckd` call before the honest leader does. [3](#0-2) 

### Recommendation

Apply `ckd_output_check` unconditionally for both variants, or reject `AppPublicKey` requests at the `respond_ckd` level until a suitable on-chain binding check exists for that variant. The `AppPublicKeyPV` variant already has the correct pairing check; the `AppPublicKey` branch must not be a silent no-op.

Short term: require all CKD requests to use `AppPublicKeyPV` so the on-chain pairing check is always enforced. Long term: design an equivalent binding check for the `AppPublicKey` variant, or document that `AppPublicKey` responses must be verified client-side and enforce this at the contract level.

### Proof of Concept

1. User Alice (a TEE app) calls `request_app_private_key` with `AppPublicKey = a·G1` for some private scalar `a`.
2. The contract stores the pending `CKDRequest` in `pending_ckd_requests`.
3. Malicious participant Eve (an attested participant) calls `respond_ckd(request, CKDResponse { big_y: 0·G1, big_c: k·G1 })` for a scalar `k` she chooses.
4. `respond_ckd` checks Eve is an attested participant ✓, enters the `AppPublicKey` arm (no-op) ✓, and calls `resolve_yields_for` — delivering the crafted response to Alice.
5. Alice decrypts: `big_s = big_c − a · big_y = k·G1 − a·0 = k·G1`.
6. Alice's derived key is `k·G1`. Eve knows `k` and can sign on Alice's behalf, stealing any funds or secrets Alice associates with that key.

The existing test at line 4513–4521 inadvertently demonstrates this: it submits `CKDResponse { big_y: [1u8;48], big_c: [2u8;48] }` (arbitrary bytes) for an `AppPublicKey` request and the contract accepts it without error. [4](#0-3)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
