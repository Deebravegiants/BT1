### Title
Token-2022 MetadataPointer Allows Spoofed Name/Symbol in `log_metadata` via Cross-Mint Metadata Injection — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

### Summary
An unprivileged attacker can create a Token-2022 mint whose `MetadataPointer` extension points to a legitimate token's Metaplex metadata account (e.g., USDC's). Calling `log_metadata` with this mint causes the bridge to emit a Wormhole message associating the attacker's worthless mint address with the legitimate token's name and symbol. NEAR then deploys a bridged token for the attacker's mint under the spoofed identity.

### Finding Description

`LogMetadata::process()` branches on the token program. For Token-2022 mints, it reads the `MetadataPointer` extension from the mint's on-chain data: [1](#0-0) 

When the pointer is non-zero and non-self, it calls `parse_metadata_account` with the pointer address: [2](#0-1) 

`parse_metadata_account` only verifies two things: (1) the provided account's key matches the pointer address, and (2) the account is owned by `MetaplexID`: [3](#0-2) 

There is **no check** that the Metaplex metadata account's embedded mint field matches `self.mint.key()`. The attacker controls the `MetadataPointer` value because they created the mint. They set it to point to any legitimate token's Metaplex metadata PDA. The resulting payload uses the attacker's mint address but the victim token's name/symbol: [4](#0-3) 

Contrast this with the classic SPL path, which is safe because the metadata address is derived from `self.mint.key()` via `find_program_address` — making it cryptographically bound to the mint: [5](#0-4) 

The only constraint on the mint account is that the bridge authority is not the mint authority — no binding to any specific metadata account: [6](#0-5) 

### Impact Explanation

NEAR's `deploy_token_callback` consumes the `LogMetadataMessage` directly, using the `name`, `symbol`, and `decimals` fields from the Wormhole message to deploy a bridged token for the reported `token_address`: [7](#0-6) 

The attacker's worthless Token-2022 mint gets registered on NEAR as e.g. "USD Coin" / "USDC". Users or UIs that display token names rather than raw addresses will see the spoofed identity. This breaks the invariant that the name and symbol logged for a Solana token accurately reflect its on-chain metadata, constituting asset-identity divergence.

### Likelihood Explanation

Fully permissionless. Creating a Token-2022 mint with an arbitrary `MetadataPointer` is a standard operation requiring no special privileges. The `MetadataPointer` extension stores an arbitrary public key with no on-chain validation of the pointed-to account's content. The attacker needs only to:
1. Create a Token-2022 mint with `MetadataPointer` set to a legitimate token's Metaplex metadata PDA
2. Call `log_metadata` passing that mint and the legitimate metadata account

No admin keys, no leaked secrets, no oracle manipulation required.

### Recommendation

In the Token-2022 third-party metadata branch, after deserializing the `MplMetadata`, validate that the metadata's embedded `mint` field matches `self.mint.key()`:

```rust
// Third-party metadata
let (name, symbol) = self.parse_metadata_account(metadata_pointer.metadata_address.0)?;
// Add: verify the deserialized metadata actually belongs to this mint
// (parse_metadata_account should return the mint field and check it here)
```

Concretely, `parse_metadata_account` should be extended to also return (or check) `metadata.mint`, and the caller should assert `metadata.mint == self.mint.key()`. The classic SPL path already achieves this implicitly via PDA derivation from the mint key — the same guarantee must be enforced explicitly for the Token-2022 third-party path.

### Proof of Concept

```
1. Let USDC_MINT = <USDC mint pubkey>
2. Let USDC_META_PDA = find_program_address(
       [METADATA_SEED, MetaplexID, USDC_MINT], MetaplexID)
   // This is USDC's Metaplex metadata account, owned by MetaplexID,
   // containing name="USD Coin", symbol="USDC"

3. Attacker creates a Token-2022 mint EVIL_MINT with:
   - MetadataPointer extension: metadata_address = USDC_META_PDA
   - mint_authority = attacker (not bridge authority → passes constraint)
   - decimals = 6

4. Attacker calls log_metadata {
       mint: EVIL_MINT,
       metadata: USDC_META_PDA,   // UncheckedAccount, no Anchor validation
       token_program: Token-2022
   }

5. process() flow:
   - token_program == token_2022::ID → true
   - metadata_pointer.metadata_address.0 == USDC_META_PDA (≠ EVIL_MINT, ≠ default)
   → "Third-party metadata" branch
   - parse_metadata_account(USDC_META_PDA):
     - metadata.key() == USDC_META_PDA ✓
     - metadata.owner == MetaplexID ✓
     - deserialize → name="USD Coin", symbol="USDC"

6. LogMetadataPayload {
       token: EVIL_MINT,      // attacker's worthless mint
       name: "USD Coin",      // USDC's name
       symbol: "USDC",        // USDC's symbol
       decimals: 6
   } → post_message to Wormhole

7. NEAR deploy_token_callback registers EVIL_MINT as "USD Coin"/"USDC"
   → assert Wormhole message contains "USD Coin"/"USDC" for EVIL_MINT ✓
```
