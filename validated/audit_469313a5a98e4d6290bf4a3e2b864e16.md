### Title
Missing Vote Signature Verification Allows Any Peer to Forge Committee Votes and Corrupt Consensus Decisions - (File: crates/apollo_consensus/src/single_height_consensus.rs)

### Summary

`SingleHeightConsensus::handle_vote` checks only that the `voter` address appears in the committee member list, but never verifies the cryptographic signature carried in the `Vote` struct. This is the direct Sequencer analog of the Zetachain "jailed validator" bug: the authorization gate is incomplete — it enforces one condition (address in committee) while silently skipping the second condition (valid signature proving the sender controls that address). Any unprivileged network peer can forge prevotes and precommits attributed to legitimate committee members, accumulate a fraudulent 2/3 quorum, and cause the node — or the entire network — to commit a wrong `ProposalCommitment`.

---

### Finding Description

**Root cause — `handle_vote` in `SingleHeightConsensus`:** [1](#0-0) 

```rust
pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
    // TODO(Asmaa): verify the signature          // ← explicit acknowledgement of missing check
    trace!("Received {:?}", vote);
    let height = self.state_machine.height();
    if vote.height != height {
        warn!("Invalid vote height: expected {:?}, got {:?}", height, vote.height);
        return VecDeque::new();
    }
    if !self.committee.members().iter().any(|s| s.address == vote.voter) {
        debug!("Ignoring vote from non validator: vote={:?}", vote);
        return VecDeque::new();
    }
    // ... duplicate/conflict check, then accepted
```

The only guard is a linear scan of `committee_members` by address. The `Vote` struct carries a `signature: RawSignature` field (visible in simulation tests), but it is never read here.

**The infrastructure for verification already exists but is unused:**

`apollo_signature_manager` exposes `verify_precommit_vote_signature`: [2](#0-1) 

Each `Staker` in the committee carries a `public_key: Felt` field. The `apollo_signature_manager` crate provides `verify_precommit_vote_signature(block_hash, signature, public_key)`: [3](#0-2) 

Neither is called anywhere in the vote-handling path.

**Downstream: the state machine trusts the vote unconditionally**

Once `handle_vote` passes the address check, the vote is forwarded to `StateMachine::handle_prevote` / `handle_precommit`, which calls `vote_weight` and inserts the vote into the quorum maps: [4](#0-3) 

The `expect("voter must be in committee")` panic guard only confirms the address is present — it does not re-check any signature. The forged vote is now counted with the full staking weight of the impersonated validator.

**Quorum and decision logic uses these maps directly:** [5](#0-4) 

A forged vote's weight is summed identically to a legitimate vote's weight. If the attacker forges votes totalling ≥ 2/3 of `total_weight`, `upon_decision` fires and emits `SMRequest::DecisionReached` with the attacker-chosen `ProposalCommitment`.

---

### Impact Explanation

An attacker who is any reachable network peer (no staking required) can:

1. Read committee member addresses and their staking weights from the public staking contract for the current epoch.
2. Craft `Vote` messages with `voter` set to committee members whose combined weight exceeds the 2/3 Byzantine quorum threshold, and `proposal_commitment` set to an arbitrary wrong block hash.
3. Broadcast these forged votes to target nodes before the real validators' votes arrive (the conflict-detection logic discards the *second* vote for a given `(round, voter)` key, so the forged vote must arrive first).
4. The node's state machine accumulates the forged weight, fires `upon_decision`, and emits `DecisionReached` with the wrong `ProposalCommitment`.
5. `decision_reached` is called on the context, the batcher commits the wrong block, and the blockifier executes the wrong transaction set — producing wrong state roots, wrong receipts, wrong events, and wrong L1 messages.

The exact corrupted value is the `ProposalCommitment` (partial block hash + fee proposal) stored in `valid_proposals` and forwarded to the batcher's `decision_reached`, which anchors the entire state transition.

---

### Likelihood Explanation

- Committee addresses are public on-chain; no privileged access is needed.
- Any peer that can establish a gossipsub connection can inject vote messages; libp2p authenticates the *connection* by peer ID but does not bind the `voter` field in the message payload to the sender's key.
- The race condition (forged vote must arrive before the real one) is exploitable by a well-positioned adversary with low-latency paths to target nodes, or by targeting nodes that are slightly behind in receiving real validator messages.
- The conflict-detection guard (`VoteStatus::Conflict → ignore`) actually *helps* the attacker: once the forged vote is recorded, the legitimate vote from the real validator is silently dropped.

---

### Recommendation

In `handle_vote`, after confirming the voter address is in the committee, look up the corresponding `Staker::public_key` and call `verify_precommit_vote_signature` (or an equivalent prevote verifier) before forwarding the vote to the state machine. Reject votes with missing or invalid signatures.

```rust
// After the committee membership check:
let staker = self.committee.members().iter()
    .find(|s| s.address == vote.voter)
    .unwrap(); // already confirmed above
let public_key = PublicKey(staker.public_key);
if !verify_precommit_vote_signature(
        BlockHash(vote.proposal_commitment.unwrap_or_default().0),
        vote.signature.clone(),
        public_key)
    .unwrap_or(false)
{
    warn!("Rejecting vote with invalid signature: {:?}", vote);
    return VecDeque::new();
}
```

### Proof of Concept

```
1. Attacker queries the staking contract for epoch E:
   committee = [(addr_A, weight=40), (addr_B, weight=35), (addr_C, weight=25)]
   total_weight = 100, quorum = 67

2. Attacker crafts forged Prevote messages:
   Vote { height: H, round: 0, vote_type: Prevote,
          voter: addr_A, proposal_commitment: Some(WRONG_HASH),
          signature: RawSignature::default() }   // any bytes; never checked
   Vote { height: H, round: 0, vote_type: Prevote,
          voter: addr_B, proposal_commitment: Some(WRONG_HASH),
          signature: RawSignature::default() }
   // combined weight = 75 > 67 → quorum

3. Attacker broadcasts these to all nodes via gossipsub before real validators vote.

4. handle_vote passes the address check for addr_A and addr_B.
   StateMachine inserts both with weights 40 and 35.
   upon_prevote_quorum fires → node broadcasts its own Precommit for WRONG_HASH.

5. Attacker repeats for Precommit phase.
   upon_decision fires → DecisionReached(WRONG_HASH).

6. Batcher commits the wrong block; blockifier executes wrong transactions;
   state root, receipts, and events are corrupted.
```
