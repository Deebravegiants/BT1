### Title
Malicious Peer Can Corrupt CAT Lineage Proof Amount via Unvalidated `parent_coin_state` in `determine_coin_type` — (`chia/wallet/wallet_state_manager.py`, `chia/wallet/cat_wallet/cat_wallet.py`)

### Summary

`determine_coin_type` fetches the parent coin state from an untrusted peer without verifying that the returned coin's ID matches the requested coin ID. The unvalidated `coin.amount` from the peer's response is stored directly into `CATCoinData.amount`, which is then written into the wallet's `LineageProof` for the parent coin. A malicious peer can supply a wrong amount, causing every subsequent spend of the victim's CAT coins to be rejected by the full node.

### Finding Description

**Call chain:**

`_add_coin_states` → `determine_coin_type` → `coin_added` → `puzzle_solution_received` → `add_lineage`

**Step 1 — `determine_coin_type` (wallet_state_manager.py lines 908–930):**

The wallet queries the peer for the parent coin state:

```python
response: list[CoinState] = await self.wallet_node.get_coin_state(
    [coin_state.coin.parent_coin_info], peer=peer, fork_height=fork_height
)
parent_coin_state = response[0]
assert parent_coin_state.spent_height == coin_state.created_height   # only check
```

The sole guard is that `spent_height` matches `created_height`. There is **no check** that `parent_coin_state.coin.name() == coin_state.coin.parent_coin_info`. A malicious peer can return any `CoinState` whose `spent_height` equals the expected value, including one with a fabricated `coin.amount`. [1](#0-0) 

The `CATCoinData` is then built directly from the peer-supplied coin:

```python
cat_data: CATCoinData = CATCoinData(
    ...
    parent_coin_state.coin.parent_coin_info,
    uint64(parent_coin_state.coin.amount),   # ← attacker-controlled
)
``` [2](#0-1) 

**Step 2 — `coin_added` (cat_wallet.py lines 362–390):**

When `coin_data is not None` (pre-fetched from `determine_coin_type`), the wallet skips the peer re-query and the coin-ID validation that exists in the `coin_data is None` branch:

```python
if coin_data is None:
    coin_state = await ...get_coin_state([coin.parent_coin_info], peer=peer)
    assert coin_state[0].coin.name() == coin.parent_coin_info   # ← guard present here
    ...
await self.puzzle_solution_received(coin, coin_data)            # ← called with unvalidated data
``` [3](#0-2) 

**Step 3 — `puzzle_solution_received` (cat_wallet.py lines 401–404):**

The wrong amount is written unconditionally into the lineage store:

```python
await self.add_lineage(
    coin_name,
    LineageProof(data.parent_coin_id, data.inner_puzzle.get_tree_hash(), data.amount),
)
``` [4](#0-3) 

**Step 4 — `fetch_coin_spend_for_coin_state` (wallet_sync_utils.py lines 355–352):**

The coin spend is fetched using `parent_coin_state.coin` (the attacker-supplied coin). The puzzle-hash validation inside `fetch_coin_spend` checks the returned puzzle against `coin.puzzle_hash` — but `coin` is the attacker's fabricated coin, so the attacker controls both sides of that check. [5](#0-4) 

**Step 5 — Persistence:**

On subsequent calls to `coin_added`, line 369 checks `lineage = await self.get_lineage_proof_for_coin(coin)`. Because the wrong proof is already stored, `lineage is not None`, so `puzzle_solution_received` is never called again and the corrupt proof is never overwritten. [6](#0-5) 

### Impact Explanation

The CAT CLVM puzzle verifies the lineage proof by recomputing the parent coin's ID as `sha256(grandparent_id || cat_puzzle_hash(inner_puzzle_hash) || amount)` and checking it against the actual on-chain parent coin ID. A wrong `amount` in the stored `LineageProof` causes this check to fail for every spend attempt. The victim's CAT coins become permanently unspendable without manual DB intervention or a full wallet resync that overwrites the corrupt record — which the normal sync path will not do because the proof is already present.

### Likelihood Explanation

The wallet connects to untrusted peers by default. The attacker only needs to be the peer that responds to the `get_coin_state` call during `determine_coin_type`. No keys, admin access, or cryptographic breaks are required. The attack is triggered automatically during normal wallet sync when a new CAT coin is received.

### Recommendation

Add a coin-ID validation in `determine_coin_type` immediately after receiving the parent coin state, mirroring the guard already present in the `coin_data is None` branch of `coin_added`:

```python
parent_coin_state = response[0]
assert parent_coin_state.coin.name() == coin_state.coin.parent_coin_info, \
    "Peer returned wrong parent coin"
assert parent_coin_state.spent_height == coin_state.created_height
``` [7](#0-6) 

This is the same pattern already used correctly in `coin_added` at line 378. [8](#0-7) 

### Proof of Concept

1. Stand up a mock peer that responds to `get_coin_state([parent_coin_id])` with a `CoinState` where `coin.amount` is set to `parent_coin.amount + 1` (or any wrong value) but `spent_height` equals the real `created_height` of the child coin.
2. Have the mock peer return a valid CAT puzzle reveal (correct TAIL hash, any inner puzzle) for the fabricated coin when `request_puzzle_solution` is called.
3. Connect the wallet to this mock peer and receive a real CAT coin.
4. Inspect the lineage store: `lineage_proof.amount` will equal the fabricated amount, not the real on-chain parent coin amount.
5. Attempt to spend the CAT coin: the full node rejects the spend because the lineage proof amount does not match the actual parent coin ID on-chain.

async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:
    solution_response = await peer.call_api(
        FullNodeAPI.request_puzzle_solution, RequestPuzzleSolution(coin.name(), height)
    )
    if solution_response is None or not isinstance(solution_response, RespondPuzzleSolution):
        raise PeerRequestException(f"Was not able to obtain solution {solution_response}")
    coin_id = coin.name()
    if solution_response.response.puzzle.get_tree_hash() != coin.puzzle_hash:
        raise PeerRequestException(f"Peer returned wrong puzzle hash for coin {coin_id}")
    if solution_response.response.coin_name != coin_id:
        raise PeerRequestException(f"Peer returned wrong coin name in puzzle solution for coin {coin_id}")

    return make_spend(
        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )
