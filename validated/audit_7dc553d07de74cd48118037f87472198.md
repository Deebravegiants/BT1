Audit Report

## Title
Unprivileged Caller Can Trigger Protocol-Wide Pause via Public `updateRSETHPrice()` - (File: contracts/LRTOracle.sol)

## Summary

`updateRSETHPrice()` is declared `public` with no access control beyond `whenNotPaused`, allowing any external caller to invoke it. Its internal logic contains a downside-protection branch that, when triggered by a natural oracle price dip exceeding `pricePercentageLimit` below `highestRsethPrice`, atomically pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` itself. An unprivileged attacker can monitor public oracle feeds and submit a single call at the right moment to freeze all user deposits and withdrawals until three separate admin `unpause()` transactions are confirmed.

## Finding Description

`updateRSETHPrice()` is `public` with only `whenNotPaused`:

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The contrast with the privileged path is explicit — `updateRSETHPriceAsManager()` at L94 carries `onlyLRTManager`, yet both call the identical `_updateRsETHPrice()` which contains the pause side-effect.

Inside `_updateRsETHPrice()`, the upside-protection branch (L252–266) explicitly gates the dangerous path behind a role check: non-managers revert with `PriceAboveDailyThreshold` if the price increase exceeds the limit. The downside-protection branch (L270–282) has **no equivalent role check**:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

For this feature to function at all, `LRTOracle` must hold `PAUSER_ROLE` on both `LRTDepositPool` and `LRTWithdrawalManager` (their `pause()` functions at `LRTDepositPool.sol` L349 and `LRTWithdrawalManager.sol` L347 are gated by `onlyRole(LRTConstants.PAUSER_ROLE)`). With that role granted, any caller of `updateRSETHPrice()` who arrives when `newRsETHPrice < highestRsethPrice` by more than `pricePercentageLimit.mulWad(highestRsethPrice)` will atomically pause all three contracts.

`highestRsethPrice` is a monotonically increasing watermark (L293–296), so as the protocol accumulates staking rewards over time, the watermark grows and the gap required to trigger the branch shrinks relative to normal oracle volatility. The attacker requires no capital, no privileged role, and no oracle manipulation — only observation of public Chainlink/oracle feeds and a single zero-value transaction.

## Impact Explanation

When the branch fires, three contracts are paused atomically:
- `LRTDepositPool` — `depositETH()` and `depositAsset()` both carry `whenNotPaused`, blocking all new deposits.
- `LRTWithdrawalManager` — `initiateWithdrawal()`, `completeWithdrawal()`, and `instantWithdrawal()` all carry `whenNotPaused`, blocking all withdrawal initiation and completion.
- `LRTOracle` — `updateRSETHPrice()` becomes uncallable until admin unpauses.

Unpausing requires three separate `onlyLRTAdmin` calls. Until confirmed, all user funds in transit are frozen.

**Impact: Temporary freezing of funds (Medium).**

## Likelihood Explanation

The trigger condition — a price drop exceeding `pricePercentageLimit` from the all-time-high rsETH price — is a realistic market event. LST prices fluctuate due to oracle update cadence, brief depegs, or normal volatility. With `pricePercentageLimit` set to 1% (1e16), a 1.5% dip from the watermark (well within normal LST volatility) satisfies the condition. The attacker requires no capital, no privileged role, and no oracle manipulation. Cost is one gas payment; effect is a full protocol halt until three admin transactions respond. The attack is repeatable after each admin unpause cycle if market conditions recur.

## Recommendation

Apply the same asymmetric role check used in the upside-protection branch to the downside-protection branch: only allow the pause to be triggered if `msg.sender` holds `MANAGER` role (or a dedicated `KEEPER_ROLE`), mirroring the pattern at L263. Alternatively, restrict `updateRSETHPrice()` itself to a trusted caller (keeper/manager), leaving `updateRSETHPriceAsManager()` as the sole entry point for sensitive price updates with pause side-effects.

## Proof of Concept

1. Protocol has been running for several months; `highestRsethPrice = 1.05e18`.
2. Admin sets `pricePercentageLimit = 1e16` (1%).
3. A brief Chainlink stETH/ETH price update causes `newRsETHPrice = 1.034e18` (~1.5% drop from watermark).
4. Attacker calls `LRTOracle.updateRSETHPrice()` — no role required, no ETH value needed.
5. `_updateRsETHPrice()` computes `diff = 1.05e18 - 1.034e18 = 0.016e18 > pricePercentageLimit.mulWad(1.05e18) = 0.0105e18` → `isPriceDecreaseOffLimit = true`.
6. `lrtDepositPool.pause()`, `withdrawalManager.pause()`, `_pause()` execute.
7. All `depositETH`, `depositAsset`, `initiateWithdrawal`, `completeWithdrawal`, and `instantWithdrawal` calls revert with `Paused`.
8. Users cannot access funds until admin manually calls `unpause()` on all three contracts.

**Foundry test sketch:**
```solidity
function test_unprivilegedPause() public {
    // Setup: set pricePercentageLimit, seed highestRsethPrice above current oracle price
    lrtOracle.setPricePercentageLimit(1e16); // 1%

    mockOracle.setPrice(1.034e18);
