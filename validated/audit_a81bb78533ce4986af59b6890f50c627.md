### Title
Front-Running via `TransferToGasKey` Pushes Gas Key Balance Over Deletion Threshold, Permanently Blocking `DeleteKey`/`DeleteAccount` - (File: `runtime/runtime/src/access_keys.rs`)

### Summary
The `delete_gas_key` function and `action_delete_account` both gate key/account deletion on a hard threshold check: `gas_key_info.balance > GasKeyInfo::MAX_BALANCE_TO_BURN` (1 NEAR). Because any unprivileged account can send a `TransferToGasKey` action targeting another account's gas key, an attacker can front-run a victim's `DeleteKey` or `DeleteAccount` transaction by depositing a tiny amount into the victim's gas key, pushing its balance just above 1 NEAR and causing the deletion to revert with `GasKeyBalanceTooHigh`. The attacker can repeat this indefinitely, sustaining the denial.

### Finding Description

`delete_gas_key` in `runtime/runtime/src/access_keys.rs` enforces:

```rust
if gas_key_info.balance > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh { ... }.into());
    return Ok(());
}
``` [1](#0-0) 

`MAX_BALANCE_TO_BURN` is a hard constant of exactly 1 NEAR: [2](#0-1) 

The same guard appears in `action_delete_account` for the aggregate sum of all gas key balances on an account: [3](#0-2) 

`action_transfer_to_gas_key` unconditionally adds the caller-supplied `deposit` to the gas key balance stored on the **receiver** account — there is no restriction requiring `signer_id == receiver_id`: [4](#0-3) 

The Rosetta RPC adapter explicitly models `TransferToGasKey` with distinct `sender_account_id` and `receiver_account_id`, confirming cross-account funding is a supported code path: [5](#0-4) 

**Attack scenario (DeleteKey)**:
1. Victim Alice has a gas key with balance = 1 NEAR (exactly at the threshold; deletion would succeed because the check is `>`).
2. Alice submits a `DeleteKey` transaction.
3. Attacker observes the pending transaction and front-runs it with a `TransferToGasKey` action targeting Alice's gas key with a deposit of 1 yoctoNEAR.
4. Alice's gas key balance becomes `1 NEAR + 1 yoctoNEAR > MAX_BALANCE_TO_BURN`.
5. Alice's `DeleteKey` receipt executes and fails with `GasKeyBalanceTooHigh`.
6. Alice can issue a `WithdrawFromGasKey` to drain the excess, but the attacker can front-run that too, repeating the cycle at the cost of 1 yoctoNEAR per round.

**Attack scenario (DeleteAccount)**:
The same applies to `action_delete_account`, which sums all gas key balances via `compute_gas_key_balance_sum` and applies the identical `> MAX_BALANCE_TO_BURN` guard. An attacker can prevent account deletion entirely by keeping the aggregate gas key balance above 1 NEAR. [3](#0-2) 

### Impact Explanation
A victim who holds a gas key with balance near 1 NEAR cannot delete the key or their account. Every `DeleteKey` or `DeleteAccount` receipt fails with `GasKeyBalanceTooHigh`, wasting gas. The victim must issue a `WithdrawFromGasKey` first, but the attacker can front-run that withdrawal with another tiny `TransferToGasKey` deposit, sustaining the denial indefinitely. The corrupted protocol value is the `ActionResult::result` field — a legitimate action receipt that should succeed is forced to produce an `ActionError`, permanently altering the on-chain execution outcome for that receipt.

### Likelihood Explanation
Gas key public keys are publicly visible on-chain. Any account with a small NEAR balance can execute the attack. The cost per round is 1 yoctoNEAR plus gas fees, making sustained griefing cheap. The scenario is most realistic when a victim is trying to close an account or rotate keys, which are observable mempool events.

### Recommendation
Restrict `TransferToGasKey` so that the `signer_id` of the enclosing transaction must equal the `receiver_id` (i.e., only the account owner can fund their own gas key). Alternatively, if cross-account funding is intentional, remove the hard `MAX_BALANCE_TO_BURN` deletion guard and instead require the account owner to explicitly withdraw excess balance before deletion, without making the deletion itself fail based on a threshold that a third party can manipulate.

### Proof of Concept

```
// State: Alice's gas key balance = 1 NEAR (exactly at threshold)
// Alice submits:
DeleteKey { public_key: alice_gas_key }   // would succeed: 1 NEAR == MAX, not >

// Attacker front-runs with:
SignedTransaction {
    signer_id:   "attacker",
    receiver_id: "alice",          // Alice's account
    actions: [TransferToGasKey {
        public_key: alice_gas_key, // publicly visible
        deposit:    1,             // 1 yoctoNEAR
    }]
}

// Alice's gas key balance is now 1_000_000_000_000_000_000_000_001 yoctoNEAR
// Alice's DeleteKey receipt executes:
//   delete_gas_key checks: balance (1 NEAR + 1 yocto) > MAX_BALANCE_TO_BURN (1 NEAR) → true
//   → ActionErrorKind::GasKeyBalanceTooHigh { balance: 1_000...001 }
//   → receipt fails, gas wasted, key not deleted
