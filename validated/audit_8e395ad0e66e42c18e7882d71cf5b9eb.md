## Assessment

**The bug is structurally real**: the stale fallback unconditionally emits `Calldata` for `l1_da_mode`, and this value flows without transformation into the block context that governs DA gas cost calculation for `estimate_fee`, `simulate_transactions`, and `estimate_message_fee` on `Pending`.

**The critical conditional** is the value of `DONT_IGNORE_L1_DA_MODE`:
- If `DONT_IGNORE_L1_DA_MODE = false` ŌåÆ `override_kzg_da_to_false = false` ŌåÆ `use_kzg_da = l1_da_mode.is_use_kzg_da()` ŌåÆ the wrong `Calldata` DA mode propagates and fee estimation is wrong on a Blob-DA network.
- If `DONT_IGNORE_L1_DA_MODE = true` ŌåÆ `override_kzg_da_to_false = true` ŌåÆ `use_kzg_da = false` always ŌåÆ the stale fallback's wrong `l1_da_mode` is irrelevant for DA mode, but `l1_data_gas_price` is still 0 (also wrong).

I could not read the literal constant value in the available iterations. The TODO comment `// TODO(shahak): Remove this once we stop supporting rpc v0.6` strongly implies the override exists for v0.6 backward compatibility and is `false` for v0.8 (i.e., the actual DA mode is used). If that is the case, the vulnerability is fully reachable.

**The trigger condition** (stale pending data) is a normal operational window ŌĆö it occurs on every block commit until the pending sync loop fetches fresh data from the feeder gateway, which can be several seconds. Any unprivileged user calling `estimate_fee(Pending)` during that window on a Blob-DA network receives a wrong fee estimate.

### Title
Stale-pending fallback in `read_pending_data` hardcodes `l1_da_mode=Calldata`, causing `estimate_fee(Pending)` to use the wrong DA cost model on Blob-DA networks ŌĆö (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
`read_pending_data` constructs a synthetic `DeprecatedPendingBlock` when pending data is stale. `DeprecatedPendingBlock` has no `l1_da_mode` field; its `l1_da_mode()` accessor always returns `Calldata`. This value flows into the execution block context and sets `use_kzg_da = false`, causing DA gas costs to be computed using the Calldata model instead of the Blob model.

### Finding Description
`read_pending_data` (lines 1573ŌĆō1594) creates a stale fallback using `PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock { ..Default::default() })`. The `l1_da_mode()` method for the `Deprecated` variant is hardcoded to return `L1DataAvailabilityMode::Calldata` (pending_data.rs:172). This value is mapped into `ExecutionPendingData.l1_da_mode` by `client_pending_data_to_execution_pending_data` (pending.rs:21) and consumed in `prepare_block_execution_artifacts` (lib.rs:369) to set `use_kzg_da`. On a Blob-DA network, `use_kzg_da` is set to `false` instead of `true`, and `get_da_gas_cost` uses the wrong cost table.

### Impact Explanation
`estimate_fee`, `simulate_transactions`, and `estimate_message_fee` all call `read_pending_data` for `BlockId::Tag(Tag::Pending)` and pass the result through `client_pending_data_to_execution_pending_data`. The returned fee estimates are authoritative-looking wrong values. On a Blob-DA network, Calldata DA costs are higher, so estimates are inflated; users may set resource bounds too high or, if they trust the estimate exactly, waste funds.

### Likelihood Explanation
The stale window opens on every block commit and closes when the pending sync loop completes its next poll. This is a recurring, predictable window of several seconds. No special privileges are required ŌĆö any caller of `starknet_estimateFee` with `block_id = "pending"` is affected.

### Recommendation
In the stale fallback branch, read `l1_da_mode` from `latest_header.block_header_without_hash.l1_da_mode` and construct a `PendingBlock` (not `DeprecatedPendingBlock`) so that `l1_da_mode()` returns the correct value. Alternatively, populate a `PendingBlock` with all relevant fields from the latest committed header, including `l1_da_mode`, `l1_data_gas_price`, and `l2_gas_price`.

### Proof of Concept
1. Configure a node on a Blob-DA network (latest Starknet mainnet).
2. Commit a new block; immediately (before pending sync updates) call `starknet_estimateFee` with `block_id = "pending"`.
3. The stale fallback fires; `l1_da_mode()` returns `Calldata`; `use_kzg_da = false`.
4. Compare the returned fee with a call made after pending data is refreshed (where `use_kzg_da = true`).
5. The two estimates differ by the Blob-vs-Calldata DA cost delta, confirming the wrong model was used.
