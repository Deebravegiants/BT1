### Title
Unverified Legacy `respond_ckd` Allows Single Byzantine Participant to Deliver Arbitrary CKD Output During Participant-Set Transition — (`File: crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` function accepts a `CKDResponse` without any cryptographic verification when the request uses the legacy `AppPublicKey` variant. During resharing, `active_participants()` switches immediately to the **new proposed** participant set. A single Byzantine new participant — who holds a valid TEE attestation but does not yet possess key shares — can call `respond_ckd` with an arbitrary `(big_y, big_c)` pair for any pending legacy CKD request and have it accepted and delivered to the user, bypassing the threshold requirement entirely.

### Finding Description

`respond_ckd` in `lib.rs` contains the following branch:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For the `AppPublicKey` (legacy, non-PV) variant, the response bytes are passed directly to `resolve_yields_for` with zero cryptographic validation. [1](#0-0) 

The caller-authentication guard `assert_caller_is_attested_participant_and_protocol_active` delegates to `active_participants()`, which during `Resharing` state returns the **new proposed** participants from `resharing_key.proposed_parameters().participants()` — not the old set:

```rust
ProtocolContractState::Resharing(state) => {
    state.resharing_key.proposed_parameters().participants()
}
``` [2](#0-1) 

This is the direct analog of the NFT bug: the participant-set transition (resharing) does not check for pending legacy CKD requests, and the incoming participant set — which does not yet hold key shares — immediately gains the authority to resolve those requests with arbitrary data.

The `respond` (ECDSA/EdDSA) path is **not** affected because it validates the signature against the derived public key before resolving yields. [3](#0-2) 

The `AppPublicKeyPV` variant is also **not** affected because `ckd_output_check` enforces a BLS12-381 pairing equation. [4](#0-3) 

### Impact Explanation

A single Byzantine new participant (below the signing threshold) can:

1. Observe a pending `request_app_private_key` call using the `AppPublicKey` variant.
2. During resharing (when they are already `active_participants()` but key shares have not yet been distributed), call `respond_ckd` with attacker-chosen `big_y` and `big_c`.
3. The contract resolves all queued yields for that request with the forged output — the user's NEAR transaction receives the wrong confidential key.

Because `resolve_yields_for` drains the entire fan-out queue in one call, every duplicate submission of the same request is simultaneously poisoned. [5](#0-4) 

The user cannot distinguish a forged response from a legitimate one at the contract level; the `AppPublicKey` variant is described as "privately verifiable" (the user must check with their own scalar), but the contract provides no on-chain guarantee. Any downstream use of the derived key (e.g., decrypting data, signing with the derived secret) will silently fail or produce attacker-controlled output.

This maps to: **Critical — unauthorized confidential key derivation output without the required participant authorization**, because threshold consensus is bypassed by a single participant.

### Likelihood Explanation

- Resharing is a routine governance operation (adding/removing nodes).
- New participants submit TEE attestations before resharing completes, so they satisfy `is_caller_an_attested_participant` during the resharing window.
- The `AppPublicKey` (legacy) variant is still accepted by the contract and used by existing integrations.
- The attacker needs only one compromised or malicious new participant — strictly below the signing threshold — and must front-run the honest nodes' `respond_ckd` call, which is feasible since the honest nodes must complete the multi-round CKD protocol before responding.

### Recommendation

1. **Add cryptographic verification for the `AppPublicKey` variant.** The `AppPublicKey` path should verify that `big_y` is a valid BLS12-381 G1 point consistent with the network public key and the request's `app_id`, analogous to the `AppPublicKeyPV` pairing check.
2. **Alternatively, deprecate and remove the unverified `AppPublicKey` path** and require all callers to use `AppPublicKeyPV`.
3. **Gate `respond_ckd` during resharing** to only allow callers who are participants in the *previous* running state (i.e., those who actually hold key shares), mirroring the recommendation from the reference report to disable transfers when pending operations exist.

### Proof of Concept

```
State: Running (epoch N), 1 pending legacy CKD request for user Alice
       (AppPublicKey variant, request key R)

1. Governance votes to add Mallory as a new participant → Resharing begins.
   active_participants() = new proposed set (includes Mallory).
   Mallory has a valid TEE attestation → passes assert_caller_is_attested_participant.

2. Mallory calls:
     respond_ckd(
       request = R,
       response = CKDResponse { big_y: [0u8;48], big_c: [0u8;48] }
     )

3. Contract executes:
     match &request.app_public_key {
         AppPublicKey(_) => {}   // no check — falls through
     }
     resolve_yields_for(&mut self.pending_ckd_requests, &R, forged_bytes)
     // All queued yields for R are resumed with attacker-chosen output.

4. Alice's NEAR transaction resolves with (big_y=[0;48], big_c=[0;48]).
   Alice's application derives a wrong/attacker-controlled confidential key.
   Honest nodes' subsequent respond_ckd call returns RequestNotFound.
```

The existing unit test `test_respond_ckd_fails_for_attested_non_participant` confirms the participant check works, but no test covers the case of a *valid new participant* submitting a forged response during resharing for the legacy variant. [6](#0-5)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
