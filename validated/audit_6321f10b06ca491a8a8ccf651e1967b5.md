Audit Report

## Title
`addLiquidity` Accepts Pool's Own Address as `owner`, Permanently Locking Deposited Tokens — (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address and forwards it without validation to `LiquidityLib.addLiquidity`, which records the LP position under a key derived from `owner`. If `owner == address(pool)`, the resulting position is irrecoverable because `removeLiquidity` enforces `msg.sender == owner`, and the pool has no code path that causes it to call `removeLiquidity` on itself. Deposited tokens enter `binTotals` and `BinState` balances permanently, and the inflated `_binTotalShares` dilutes all other LPs in the affected bin.

## Finding Description
`MetricOmmPool.addLiquidity` (lines 182–196) performs no check on the `owner` parameter before delegating to `LiquidityLib.addLiquidity`:

```solidity
function addLiquidity(
    address owner,   // ← no owner != address(this) guard
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    ...
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
        _liquidityContext(), owner, salt, deltas, callbackData,
        binTotals, _binStates, _binTotalShares, _positionBinShares
    );
``` [1](#0-0) 

Inside `LiquidityLib.addLiquidity`, the position key is computed as `_positionBinKey(owner, salt, int8(binIdx))` and shares are written to `positionBinShares[posKey]`, while `binState.token0BalanceScaled`, `binState.token1BalanceScaled`, and `binTotals.scaledToken0/scaledToken1` are all incremented: [2](#0-1) 

The callback then pulls real tokens from the caller into the pool: [3](#0-2) 

`removeLiquidity` enforces `msg.sender == owner` with no exception:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [4](#0-3) 

When `owner == address(pool)`, satisfying this check requires the pool itself to be `msg.sender` on a `removeLiquidity` call. The pool has no function — neither in `MetricOmmPool`, `ExtensionCalling`, nor any factory path — that causes it to call `removeLiquidity` on itself. The periphery's `_validateOwner` guard exists only in `MetricOmmPoolLiquidityAdder` and is bypassed by any direct call to the pool. [5](#0-4) 

## Impact Explanation
Tokens deposited with `owner = address(pool)` enter `binTotals.scaledToken0/scaledToken1` and individual `BinState` balances permanently. The LP claim against those tokens (recorded in `_positionBinShares`) can never be redeemed, constituting a direct, irreversible loss of the depositor's principal. Additionally, `_binTotalShares[binIdx]` is inflated by the minted shares, diluting the pro-rata entitlement of all other LPs in that bin on future `removeLiquidity` calls — a secondary loss of owed LP assets for existing liquidity providers. This meets the "direct loss of user principal" and "pool insolvency: balances fail to cover LP claims" impact gates.

## Likelihood Explanation
The trigger is fully unprivileged: any external account can call `pool.addLiquidity(owner=address(pool), ...)` directly. No special role, no prior state, and no flash loan is required. A misconfigured router that passes `address(this)` as `owner` when `address(this)` happens to be the pool address would trigger this accidentally. A griefing attacker can trigger it deliberately and repeatably against any deployed pool.

## Recommendation
Add a guard at the entry of `MetricOmmPool.addLiquidity` before delegating to `LiquidityLib`:

```solidity
if (owner == address(this)) revert InvalidOwner();
```

Alternatively, enforce it inside `LiquidityLib.addLiquidity` (where `address(this)` resolves to the pool under DELEGATECALL):

```solidity
if (owner == address(this)) revert IMetricOmmPoolActions.InvalidOwner();
```

A corresponding `InvalidOwner` error should be declared in `IMetricOmmPoolActions`.

## Proof of Concept
1. Pool `P` is deployed with `TOKEN0 = USDC`, `TOKEN1 = WETH`.
2. Attacker calls `P.addLiquidity(owner=address(P), salt=0, deltas={binIdxs:[0], shares:[minimalMintableLiquidity]}, callbackData=..., extensionData="")`.
3. `LiquidityLib.addLiquidity` writes `positionBinShares[keccak256(abi.encode(P, 0, 0))] = minimalMintableLiquidity`, increments `binState.token0BalanceScaled` and `binTotals.scaledToken0`. The callback fires and the attacker transfers, e.g., 1000 USDC into the pool.
4. Anyone calls `P.removeLiquidity(owner=address(P), salt=0, ...)` → reverts `NotPositionOwner()` because `msg.sender != address(P)`.
5. No code path in `MetricOmmPool` or any factory/extension function satisfies `msg.sender == address(P)` for a `removeLiquidity` call.
6. The 1000 USDC is permanently locked. `_binTotalShares[0]` is inflated, diluting all other LPs' pro-rata share of bin 0 on every future `removeLiquidity`.
