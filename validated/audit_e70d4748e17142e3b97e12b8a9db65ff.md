### Title
Unwhitelisted Token Address in `logMetadata` + `initTransfer` Enables Unbacked Wrapped-Token Minting on NEAR — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`logMetadata` is a public, permissionless function that accepts any arbitrary ERC-20 address and publishes a Wormhole (or EVM-log) message that the NEAR bridge treats as a legitimate token-registration event. `initTransfer` similarly accepts any arbitrary ERC-20 address with no whitelist check. An unprivileged attacker can chain these two entry points to register a malicious ERC-20, fake a token lock, and cause the NEAR bridge to mint an unbounded supply of unbacked wrapped tokens.

---

### Finding Description

**Entry point 1 — `logMetadata`**

`logMetadata` carries no access-control modifier and no check that `tokenAddress` is a known or whitelisted token: [1](#0-0) 

It calls `name()`, `symbol()`, and `decimals()` on the caller-supplied address — arbitrary external code — then delegates to `logMetadataExtension`. In `OmniBridgeWormhole`, that extension publishes a `MessageType.LogMetadata` Wormhole message carrying the attacker-controlled address and metadata: [2](#0-1) 

On the NEAR side, `deploy_token_callback` processes this message. Its only guard is that the Wormhole emitter equals the registered factory — which it does, because `OmniBridgeWormhole` itself is the registered factory: [3](#0-2) 

So NEAR registers the attacker-controlled address as a valid bridge token and records its decimals in `token_decimals`.

**Entry point 2 — `initTransfer`**

`initTransfer` also carries no token-whitelist check. For any address that is neither a bridge token nor a custom-minter token, it simply calls `safeTransferFrom` on the caller-supplied address: [4](#0-3) 

A malicious ERC-20 can return `true` from `transferFrom` without moving any real value. `SafeERC20` accepts this. The bridge increments `currentOriginNonce` and `initTransferExtension` publishes a Wormhole `InitTransfer` message as if a real lock occurred: [5](#0-4) 

On the NEAR side, `fin_transfer_callback` again only checks that the emitter is the registered factory, then looks up decimals for the token (which now exist, registered in step 1), and mints wrapped tokens to the attacker's recipient: [6](#0-5) 

---

### Impact Explanation

An unprivileged attacker can mint an arbitrary quantity of wrapped tokens on NEAR that are backed by zero locked value on EVM. This directly satisfies the **Critical** impact criterion: *unauthorized creation of wrapped bridge assets through verification failure*. Every legitimate holder of the same wrapped token suffers dilution; redemption of real tokens becomes impossible once the unbacked supply exceeds the locked reserve.

---

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no collusion. It needs only:
1. Deploying a malicious ERC-20 (trivial).
2. Paying the Wormhole message fee twice (small ETH cost).
3. Submitting two Wormhole VAAs to NEAR (standard relayer operation).

Wormhole guardians attest to any message emitted by the registered emitter; they perform no token-level validation. The path is fully reachable from public protocol inputs.

---

### Recommendation

1. **Whitelist tokens in `logMetadata`**: require that `tokenAddress` is already registered (`isBridgeToken[tokenAddress]` or an admin-maintained allowlist) before emitting a cross-chain metadata message.
2. **Whitelist tokens in `initTransfer`**: reject any `tokenAddress` that is not in the bridge's known token set. The existing `isBridgeToken` and `customMinters` mappings already cover bridge-deployed and custom-minter tokens; add an explicit revert for all others, or maintain a separate `allowedNativeTokens` set populated by governance.
3. **Validate token registration on NEAR before minting**: in `fin_transfer_callback`, confirm that the token was registered through a governance-approved `deploy_token` path, not merely that `token_decimals` contains an entry.

---

### Proof of Concept

```
1. Attacker deploys MaliciousERC20:
   - name() / symbol() / decimals() return plausible values
   - transferFrom(from, to, amount) emits Transfer event, returns true, moves no real tokens

2. Attacker calls OmniBridgeWormhole.logMetadata(MaliciousERC20) with msg.value = wormhole fee
   → OmniBridge calls MaliciousERC20.name(), .symbol(), .decimals()
   → logMetadataExtension publishes Wormhole VAA_1 (MessageType.LogMetadata, token=MaliciousERC20)

3. Attacker submits VAA_1 to NEAR bridge via deploy_token(chain_kind=Eth, prover_args=VAA_1)
   → deploy_token_callback: emitter == registered factory ✓
   → NEAR registers MaliciousERC20 address, deploys wrapped token W, stores token_decimals

4. Attacker calls OmniBridgeWormhole.initTransfer(MaliciousERC20, 1_000_000e18, 0, 0, "near:attacker.near", "")
   → SafeERC20.safeTransferFrom calls MaliciousERC20.transferFrom → returns true, no real transfer
   → initTransferExtension publishes Wormhole VAA_2 (MessageType.InitTransfer, amount=1_000_000e18)

5. Attacker submits VAA_2 to NEAR bridge via fin_transfer(chain_kind=Eth, prover_args=VAA_2)
   → fin_transfer_callback: emitter == registered factory ✓, token in token_decimals ✓
   → NEAR mints 1_000_000 W tokens to attacker.near

Result: attacker holds 1_000_000 W tokens backed by zero locked EVM value.
```
