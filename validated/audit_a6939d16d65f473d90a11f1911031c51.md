### Title
Removing a Supported Asset While Withdrawal Requests Are Pending Freezes Users' rsETH in `LRTWithdrawalManager` - (File: contracts/LRTWithdrawalManager.sol, contracts/LRTConfig.sol)

---

### Summary

`LRTConfig.removeSupportedAsset` can be called while users have pending (locked) withdrawal requests in `LRTWithdrawalManager`. Once the asset is removed, the `unlockQueue` function — which is a mandatory prerequisite for completing any withdrawal — reverts due to its `onlySupportedAsset(asset)` modifier. Users who already transferred rsETH into the withdrawal manager cannot complete their withdrawals and have no mechanism to cancel and recover their rsETH.

---

### Finding Description

**Step 1 — User initiates a withdrawal.**

`LRTWithdrawalManager.initiateWithdrawal` transfers the user's rsETH into the contract and records a locked withdrawal request: [1](#0-0) 

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
// ...
assetsCommitted[asset] += expectedAssetAmount;
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
```

The user's rsETH is now held by the contract. The request is **locked** — it cannot be completed until an operator calls `unlockQueue`.

**Step 2 — Admin removes the asset.**

`LRTConfig.removeSupportedAsset` only guards against removal when `LRTDepositPool.getTotalAssetDeposits(asset)` exceeds `maxNegligibleAmount`: [2](#0-1) 

```solidity
if (ILRTDepositPool(depositPool).getTotalAssetDeposits(asset) > maxNegligibleAmount) {
    revert CannotRemoveAssetWithDeposits(asset);
}
delete isSupportedAsset[asset];
delete assetStrategy[asset];
```

This guard checks only the deposit pool (and node delegators staked in EigenLayer). It does **not** check `LRTWithdrawalManager.assetsCommitted[asset]` or the `LRTUnstakingVault` balance. During an asset deprecation lifecycle, all assets may have already been unstaked from EigenLayer and moved to the unstaking vault, making `getTotalAssetDeposits` return zero — while users still have pending withdrawal requests.

**Step 3 — `unlockQueue` is permanently blocked.**

`unlockQueue` carries the `onlySupportedAsset(asset)` modifier: [3](#0-2) 

```solidity
function unlockQueue(
    address asset, ...
)
    external
    nonReentrant
    onlySupportedAsset(asset)   // ← reverts after asset removal
    whenNotPaused
    onlyAssetTransferOrOperatorRole
```

After `removeSupportedAsset` sets `isSupportedAsset[asset] = false`, every call to `unlockQueue` for that asset reverts. The `onlySupportedAsset` modifier in `LRTConfigRoleChecker` delegates to `lrtConfig.isSupportedAsset`: [4](#0-3) 

**Step 4 — `completeWithdrawal` is unreachable.**

`completeWithdrawal` itself has no `onlySupportedAsset` guard, but it requires the request to be unlocked (`usersFirstWithdrawalRequestNonce < nextLockedNonce[asset]`): [5](#0-4) 

```solidity
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```

Since `unlockQueue` can never advance `nextLockedNonce[asset]`, all locked requests remain permanently locked. There is no `cancelWithdrawal` function in the contract, so users cannot recover their rsETH.

---

### Impact Explanation

Users who called `initiateWithdrawal` before the asset was removed have their rsETH irrecoverably held in `LRTWithdrawalManager`. The only recovery path is for the admin to re-add the asset via `addNewSupportedAsset` (which requires `TIME_LOCK_ROLE` and a timelock delay), making this a **temporary-to-indefinite freeze of user funds**.

**Impact: Medium — Temporary freezing of funds (rsETH stuck in `LRTWithdrawalManager`).**

---

### Likelihood Explanation

Asset deprecation is a normal administrative operation. During deprecation, the typical sequence is: stop new deposits → unstake from EigenLayer → move assets to unstaking vault → remove asset. At the point of removal, `getTotalAssetDeposits` returns zero (assets are in the unstaking vault, not the deposit pool), so the guard passes. If any users initiated withdrawals before the unstaking completed, their requests are now frozen. This is a realistic operational scenario with no malicious intent required.

---

### Recommendation

In `LRTConfig.removeSupportedAsset`, additionally verify that no pending withdrawal requests exist for the asset in `LRTWithdrawalManager`:

```solidity
address withdrawalManager = getContract(LRTConstants.LRT_WITHDRAW_MANAGER);
if (ILRTWithdrawalManager(withdrawalManager).assetsCommitted(asset) > 0) {
    revert CannotRemoveAssetWithPendingWithdrawals(asset);
}
```

Alternatively, remove the `onlySupportedAsset` guard from `unlockQueue` so that already-initiated withdrawal requests can always be processed regardless of the asset's current support status.

---

### Proof of Concept

1. Asset `stETH` is being deprecated. All stETH has been unstaked from EigenLayer and sits in `LRTUnstakingVault`. `LRTDepositPool.getTotalAssetDeposits(stETH)` returns `0`.
2. Alice calls `initiateWithdrawal(stETH, 1e18, "")`. Her `1e18` rsETH is transferred to `LRTWithdrawalManager`. `assetsCommitted[stETH] += expectedStETH`.
3. Admin calls `LRTConfig.removeSupportedAsset(stETH, 0)`. The guard passes (deposit pool shows 0). `isSupportedAsset[stETH]` is deleted.
4. Operator calls `unlockQueue(stETH, ...)` → **reverts** with `AssetNotSupported` from the `onlySupportedAsset` modifier.
5. Alice calls `completeWithdrawal(stETH, "")` → **reverts** with `WithdrawalLocked` because `nextLockedNonce[stETH]` was never advanced.
6. Alice's `1e18` rsETH is permanently stuck in `LRTWithdrawalManager` with no cancel path. [2](#0-1) [6](#0-5) [3](#0-2) [1](#0-0) [5](#0-4)

    function removeSupportedAsset(
        address asset,
        uint256 tokenIndex
    )
        external
        onlySupportedAsset(asset)
        onlyRole(DEFAULT_ADMIN_ROLE)
    {
        UtilLib.checkNonZeroAddress(asset);

        if (supportedAssetList[tokenIndex] != asset) {
            revert TokenNotFoundError();
        }

        address depositPool = getContract(LRTConstants.LRT_DEPOSIT_POOL);

        if (ILRTDepositPool(depositPool).getTotalAssetDeposits(asset) > maxNegligibleAmount) {
            revert CannotRemoveAssetWithDeposits(asset);
        }

        delete isSupportedAsset[asset];
        delete assetStrategy[asset];
        depositLimitByAsset[asset] = 0;

        supportedAssetList[tokenIndex] = supportedAssetList[supportedAssetList.length - 1];
        supportedAssetList.pop();

        emit RemovedSupportedAsset(asset);
    }
