### Title
`send_message_to_l1` Syscall Accepts Full 32-Byte `to_address` on L3 Chains but `to_ordered_tuple` Panics on Conversion to `EthAddress` — (File: `crates/blockifier/src/execution/call_info.rs`)

---

### Summary

On L3 chains the `send_message_to_l1` syscall intentionally skips the `EthAddress` range check and stores any 32-byte `L1Address` as the message destination. However, when the blockifier later flattens execution results into `StarknetAPIMessageToL1` via `OrderedL2ToL1Message::to_ordered_tuple`, it calls `.expect("Failed to convert L1Address to EthAddress")` — which panics for any address whose upper 12 bytes are non-zero. Any unprivileged user on an L3 chain can deploy a contract that calls `send_message_to_l1` with a full 32-byte Starknet address, causing the batcher to panic during block building and halting block production.

---

### Finding Description

**Step 1 — Syscall deliberately widens the accepted type for L3.**

`syscall_base.rs` `send_message_to_l1` stores the raw `L1Address` (a `Felt` wrapper, 32 bytes) without any range check when `is_l3 = true`: [1](#0-0) 

For non-L3 chains the guard `EthAddress::try_from(message.to_address)?` rejects addresses whose upper 12 bytes are non-zero. For L3 chains the guard is skipped entirely, so the full 32-byte `L1Address` is pushed into `l2_to_l1_messages`.

**Step 2 — `to_ordered_tuple` narrows back to 20 bytes with `.expect()`.**

When the blockifier collects L2→L1 messages from the call-info tree it calls `OrderedL2ToL1Message::to_ordered_tuple`, which converts the stored `L1Address` to `EthAddress` via `try_into().expect(...)`: [2](#0-1) 

`TryFrom<L1Address> for EthAddress` delegates to `TryFrom<Felt> for EthAddress`, which returns `Err(OutOfRange)` when the upper 12 bytes are non-zero: [3](#0-2) 

Because the call site uses `.expect()` instead of propagating the error, the thread panics.

**Step 3 — The type mismatch is structural.**

The blockifier's internal `MessageToL1` uses `L1Address` (32 bytes): [4](#0-3) 

But `starknet_api::transaction::MessageToL1` — the type that feeds receipts, commitments, and the OS output — uses `EthAddress` (20 bytes): [5](#0-4) 

The syscall layer widens the accepted type for L3 but the commitment layer never widens its target type, creating an irreconcilable mismatch identical in structure to the external `address`-vs-`bytes32` bug.

---

### Impact Explanation

When the batcher calls `sorted_items` to collect L2→L1 messages for a block that contains a transaction whose `send_message_to_l1` used a full 32-byte address, `to_ordered_tuple` panics. In Rust a panic unwinds the calling thread; if the batcher does not wrap this call in `catch_unwind` the entire batcher process crashes, halting block production. Even if the panic is caught, the transaction cannot be included in any block, producing a wrong execution result for an accepted transaction. This matches:

> *Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*

---

### Likelihood Explanation

Any unprivileged user on an L3 chain can deploy a Cairo contract with a single `send_message_to_l1` call whose `to_address` argument has non-zero upper 12 bytes (e.g., any Starknet contract address). No special permissions, consensus membership, or privileged access are required. The gateway and mempool accept the transaction normally; the failure occurs only during block building.

---

### Recommendation

Replace the `.expect()` in `to_ordered_tuple` with proper error propagation, and change `UnorderedItem` to use `L1Address` (or a new `MessageToL1` variant that holds `L1Address`) so that L3 messages with full 32-byte destinations can be represented without loss. Alternatively, add a post-execution validation step that rejects (reverts) any L3 transaction whose `send_message_to_l1` `to_address` does not fit in 20 bytes, keeping the invariant that `StarknetAPIMessageToL1.to_address` is always a valid `EthAddress`.

---

### Proof of Concept

1. Configure the sequencer with `chain_info.is_l3 = true`.
2. Deploy a Cairo 1 contract containing:
   ```cairo
   fn trigger(ref self: ContractState) {
       // 0x10000000000000000000000000000000000000001 — upper 12 bytes non-zero
       starknet::send_message_to_l1_syscall(
           0x10000000000000000000000000000000000000001_felt252,
           array![].span()
       ).unwrap();
   }
   ```
3. Submit an invoke transaction calling `trigger`.
4. The gateway and mempool accept the transaction.
5. The batcher pulls the transaction, the blockifier executes it, and `send_message_to_l1` succeeds (L3 path, no range check).
6. When the batcher calls `OrderedL2ToL1Message::sorted_items` to build the block receipt, `to_ordered_tuple` calls `L1Address(0x10000000000000000000000000000000000000001).try_into::<EthAddress>()`, which returns `Err(OutOfRange)`.
7. `.expect("Failed to convert L1Address to EthAddress")` panics, crashing the batcher thread.

The existing test `test_send_message_to_l1_invalid_address` confirms that `is_l3 = true` allows the syscall to succeed with `0x10000000000000000000000000000000000000000` — the same address class that triggers the panic in `to_ordered_tuple`: [6](#0-5)

impl TryFrom<Felt> for EthAddress {
    type Error = StarknetApiError;
    fn try_from(felt: Felt) -> Result<Self, Self::Error> {
        const COMPLIMENT_OF_H160: usize = std::mem::size_of::<Felt>() - H160::len_bytes();

        let bytes = felt.to_bytes_be();
        let (rest, h160_bytes) = bytes.split_at(COMPLIMENT_OF_H160);
        if rest != [0u8; COMPLIMENT_OF_H160] {
            return Err(StarknetApiError::OutOfRange { string: felt.to_string() });
        }

        Ok(EthAddress(H160::from_slice(h160_bytes)))
    }
