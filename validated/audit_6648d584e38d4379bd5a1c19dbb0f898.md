### Title
`pre_validate_block()` Propagates Unhandled `KeyError` Instead of Returning Structured Error Under Hard Fork 3.0 Consensus Mode — (`File: chia/consensus/multiprocess_validation.py`)

### Summary

In `chia/consensus/multiprocess_validation.py`, the async `pre_validate_block()` function catches only `ValueError` from `block_to_block_record()`. Under the 3.0 hard fork consensus mode, `block_to_block_record()` raises a `KeyError` (not a `ValueError`) when processing a block with an invalid sub-epoch summary hash. This `KeyError` is not caught, so it propagates as an unhandled exception to the full node's connection handler instead of returning a structured `PreValidationResult` error code. This is the direct Chia analog to the external report's bug class: a function expected to return a sentinel/error value instead raises an exception, causing callers to receive unexpected errors.

### Finding Description

In `pre_validate_block()`, the call to `block_to_block_record()` is wrapped in a narrow `except ValueError:` handler: [1](#0-0) 

`block_to_block_record()` explicitly raises `ValueError(Err.INVALID_SUB_EPOCH_SUMMARY)` when the computed sub-epoch summary hash does not match the one embedded in the block: [2](#0-1) 

However, before reaching that `raise ValueError`, `block_to_block_record()` calls `make_sub_epoch_summary()` with `make_challenge_root=True` (the 3.0 hard fork path): [3](#0-2) 

Under the 3.0 hard fork, `make_sub_epoch_summary()` calls `compute_challenge_merkle_root()`: [4](#0-3) 

`compute_challenge_merkle_root()` performs block record lookups that can raise a `KeyError` (e.g., a missing key in the block record store). This `KeyError` propagates through `block_to_block_record()` and is **not** caught by `except ValueError:` in `pre_validate_block()`. The test suite explicitly documents this known defect and works around it by excluding the 3.0 hard fork consensus mode from the affected test: [5](#0-4) 

The fix is also documented in the test comment: change `except ValueError:` to `except Exception as e:`.

The inner `_pre_validate_block()` (the sync worker function) does have a broad `except Exception:` guard that returns `Err.UNKNOWN`: [6](#0-5) 

But the `KeyError` is raised in the **async** `pre_validate_block()` function itself, before the work is dispatched to the thread pool executor, so that guard is never reached.

### Impact Explanation

When `pre_validate_block()` raises an unhandled `KeyError`, the exception propagates to the WebSocket connection handler (`_api_call`), which catches it, logs it, and closes the connection with a ban: [7](#0-6) 

An unprivileged attacker operating under the 3.0 hard fork consensus mode can craft a block with a malformed sub-epoch summary hash and broadcast it. Every full node that receives this block will:
1. Raise an unhandled `KeyError` from `pre_validate_block()` instead of returning a structured `Err.INVALID_SUB_EPOCH_SUMMARY` result.
2. Tear down the connection to the sending peer.

If the attacker controls multiple peers or relays the malformed block through honest peers, this can be used to repeatedly disrupt full node connections and block processing, constituting a High-severity denial-of-service against honest nodes' ability to process blocks under the 3.0 hard fork consensus mode.

### Likelihood Explanation

The 3.0 hard fork consensus mode is a production code path (not test-only). Any unprivileged peer can broadcast a crafted block. The triggering condition (a block with a sub-epoch summary hash that causes `compute_challenge_merkle_root()` to raise `KeyError`) is reachable without any privileged access. The defect is already acknowledged in the test suite as a known unfixed issue.

### Recommendation

Change the `except ValueError:` clause in `pre_validate_block()` to `except Exception:` (or at minimum `except (ValueError, KeyError):`) so that all exception types raised by `block_to_block_record()` are caught and converted to a structured `return_error(Err.INVALID_SUB_EPOCH_SUMMARY)` return, consistent with the function's contract of always returning a `PreValidationResult`:

```python
# chia/consensus/multiprocess_validation.py, line 258
# Before:
except ValueError:
# After:
except Exception:
```

This matches the fix already suggested in the test comment at `chia/_tests/blockchain/test_blockchain.py` lines 827–830.

### Proof of Concept

1. Operate a peer under the 3.0 hard fork consensus mode.
2. Craft a `FullBlock` that includes a `finished_sub_slots` entry with a non-`None` `subepoch_summary_hash` set to an arbitrary invalid value (e.g., `std_hash(b"0")`), matching the pattern in `test_empty_slot_no_ses`.
3. Broadcast this block to a target full node.
4. `pre_validate_block()` calls `block_to_block_record()`, which calls `make_sub_epoch_summary()` with `make_challenge_root=True`, which calls `compute_challenge_merkle_root()`, which raises `KeyError`.
5. The `except ValueError:` at line 258 does not catch `KeyError`; the exception propagates to `_api_call`, which closes and bans the connection instead of returning `Err.INVALID_SUB_EPOCH_SUMMARY`.
6. The block is not properly rejected with a structured error; the node's connection is disrupted.
