### Title
Wrong Receiver Identity Used in `DeterministicStateInit` Validation Inside `DelegateAction` — (`File: runtime/runtime/src/action_validation.rs`)

### Summary

`validate_delegate_action` validates a `DeterministicStateInit` inner action against the **outer transaction's receiver** (`receiver`, i.e. the delegate sender's account) instead of the **delegate action's own `receiver_id`** (the actual inner receipt target). This is the direct NEAR analog of the `tx.origin` misuse: the outer-context identity is substituted for the inner-context identity in an access-control check, allowing a crafted meta-transaction to pass initial tx validation while carrying a mismatched `state_init`/`receiver_id` pair.

### Finding Description

In `validate_delegate_action` (pre-`FixDelegatedDeterministicStateInit`), the inner actions are validated with the wrong receiver:

```rust
// runtime/runtime/src/action_validation.rs  lines 188-206
let inner_receiver =
    if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
        delegate_action.receiver_id()          // correct
    } else {
        receiver                               // BUG: outer tx receiver, not inner delegate receiver
    };
validate_actions_with_mode(limit_config, &actions, inner_receiver, ...)?;
```

`validate_deterministic_state_init` then checks `derive(state_init) == inner_receiver`. Because `inner_receiver` is the outer tx receiver (= `det_account_b = derive(state_init_b)`), the check passes even when the delegate action's own `receiver_id` is `det_account_a` (a completely different deterministic account). The mismatch is only caught later, at receipt execution, by `validate_receipt`. [1](#0-0) [2](#0-1) 

### Impact Explanation

An unprivileged user (Alice / meta-tx sender) can craft a `SignedDelegateAction` where:

- `delegate_action.sender_id` = `det_account_b` (a deterministic account Alice controls, `derive(state_init_b)`)
- `delegate_action.receiver_id` = `det_account_a` (a different deterministic account)
- inner action = `DeterministicStateInit { state_init: state_init_b, … }`

The outer tx sets `receiver_id = det_account_b`. Tx validation calls `validate_delegate_action(…, receiver = det_account_b)`, which checks `derive(state_init_b) == det_account_b` — this passes. The tx is accepted, the relayer's gas is purchased and partially burnt. When the resulting receipt arrives at `det_account_a`, `validate_receipt` checks `derive(state_init_b) == det_account_a` — this fails with `InvalidDeterministicStateInitReceiver`. The receipt outcome is `Failure(ActionError(NewReceiptValidationError(…)))` and the relayer's gas is consumed without the intended effect.

The corrupted protocol value is the **relayer's balance** (gas burnt for a tx that should have been rejected at admission) and the **receipt outcome** (failure instead of tx-level rejection). The deterministic account state itself is not corrupted because receipt validation catches the mismatch. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

Any unprivileged user who can submit a meta-transaction to a relayer can trigger this. The attacker only needs to:
1. Control a deployed deterministic account (`det_account_b`) with an access key.
2. Craft a `DelegateAction` targeting a different deterministic account (`det_account_a`) with a mismatched `state_init`.
3. Submit it to any relayer that does not independently re-derive and cross-check the inner `receiver_id` against the `state_init`.

The relayer is the financially harmed party. Relayers that trust the user's `DelegateAction` content (a common assumption in the meta-tx model) are directly exploitable. [5](#0-4) 

### Recommendation

Always use `delegate_action.receiver_id()` — not the outer tx receiver — when recursively validating inner actions of a `DelegateAction`. This is exactly what `ProtocolFeature::FixDelegatedDeterministicStateInit` implements. The fix should be applied unconditionally (or the feature flag promoted to always-on) so that the wrong-identity path is unreachable. [6](#0-5) 

### Proof of Concept

The repository's own integration test documents the exploit path:

```
test_deterministic_state_init_meta_tx_receiver_check_pre_fix
```

At `fix_version - 1` (pre-fix protocol version):
- `try_meta_tx_deterministic_receiver_exploit` submits a tx where `state_init_b` is wrapped in a delegate action targeting `det_account_a`.
- The tx **passes** initial validation (wrong receiver used in check).
- The final outcome is `Failure(ActionError { kind: NewReceiptValidationError(ActionsValidation(InvalidDeterministicStateInitReceiver { … })) })`.
- The relayer's gas is consumed. [7](#0-6) [8](#0-7)

fn validate_delegate_action(
    limit_config: &LimitConfig,
    delegate_action: VersionedDelegateActionRef<'_>,
    receiver: &AccountId,
    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,
) -> Result<(), ActionsValidationError> {
    let actions = delegate_action.get_actions();
    let inner_receiver =
        if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
            // This is the correct receiver id to use for the check.
            delegate_action.receiver_id()
        } else {
            // This is a bug fixed with `FixDelegatedDeterministicStateInit` that
            // validated against the wrong id. This makes it impossible to
            // initialize deterministic accounts from meta transactions.
            // The bug cannot be abused, if someone crafts a state init that passes
            // validation here, it will fail when it is checked as incoming receipt.
            receiver
        };
    validate_actions_with_mode(
        limit_config,
        &actions,
        inner_receiver,
        current_protocol_version,
        mode,
    )?;
    Ok(())
}

fn validate_deterministic_state_init(
    limit_config: &LimitConfig,
    action: &DeterministicStateInitAction,
    receiver_id: &AccountId,
) -> Result<(), ActionsValidationError> {
    validate_global_contract_identifier(action.state_init.code())?;

    let derived_id = derive_near_deterministic_account_id(&action.state_init);

    if derived_id != *receiver_id {
        return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver {
            derived_id,
            receiver_id: receiver_id.clone(),
        });

    protocol_version: ProtocolVersion,
) -> Result<FinalExecutionOutcomeView, InvalidTxError> {
    let mut env = TestEnv::setup_with_version(Balance::from_near(100), protocol_version);
    env.deploy_global_contract(GlobalContractDeployMode::AccountId);

    let (_state_init_a, det_account_a) = env.new_deterministic_account_with_data(small());
    let (state_init_b, det_account_b) = env.new_deterministic_account_with_data(big());
    assert_ne!(det_account_a, det_account_b);

    // Deploy det_account_b and add a full-access key so it can act as meta_tx_sender.
    let user_signer = create_user_test_signer(&env.user_account());
    let storage_balance = env.balance_for_storage(state_init_b.clone());
    let deploy_tx = SignedTransaction::deterministic_state_init(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        env.get_tx_block_hash(),
        state_init_b.clone(),
        storage_balance,
    );
    env.run_tx(deploy_tx);

    let meta_tx_sender_signer = create_user_test_signer(&det_account_b);
    let pk_base64 = near_primitives_core::serialize::to_base64(
        &borsh::to_vec(&meta_tx_sender_signer.public_key()).unwrap(),
    );
    let add_key_args = serde_json::json!([
        { "batch_create": { "account_id": det_account_b.as_str() }, "id": 0 },
        {
            "action_add_key_with_full_access": {
                "promise_index": 0,
                "public_key": pk_base64,
                "nonce": 0
            },
            "id": 0,
            "return": true
        }
    ]);
    let add_key_tx = SignedTransaction::call(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        Balance::from_near(2),
        "call_promise".to_owned(),
        serde_json::to_vec(&add_key_args).unwrap(),
        Gas::from_teragas(300),
        env.get_tx_block_hash(),
    );
    env.run_tx(add_key_tx);

    // Craft the exploit: outer_tx.receiver = det_account_b = derive(state_init_b).
    // Old check: det_account_b == derive(state_init_b) passes.
    // The delegate action targets det_account_a, which is the wrong account.
    // In no protocol version can this ever be allowed to be executed successfully.
    let relayer = env.independent_account();
    let relayer_signer = create_user_test_signer(&relayer);
    let inner_action = Action::DeterministicStateInit(Box::new(DeterministicStateInitAction {
        state_init: state_init_b,
        deposit: Balance::ZERO,
    }));
    let delegate_nonce = env.next_nonce_for(&det_account_b);
    let delegate_action = DelegateAction {
        sender_id: det_account_b.clone(),
        receiver_id: det_account_a,
        actions: vec![NonDelegateAction::try_from(inner_action).unwrap()],
        nonce: delegate_nonce,
        max_block_height: 1_000_000,
        public_key: meta_tx_sender_signer.public_key(),
    };
    let signed_delegate_action =
        SignedDelegateAction::sign(&meta_tx_sender_signer, delegate_action);
    let tx = SignedTransaction::from_actions(
        env.next_nonce(),
        relayer,
        det_account_b,
        &relayer_signer,
        vec![Action::Delegate(Box::new(signed_delegate_action))],
        env.get_tx_block_hash(),
    );
    env.try_execute_tx(tx)
}

## Relayer

Meta transactions only work with a relayer. This is an application layer
concept, implemented off-chain. Think of it as a server that accepts a
`SignedDelegateAction`, does some checks on them and eventually forwards it
inside a transaction to the blockchain network.

A relayer may choose to offer their service for free but that's not going to be
financially viable long-term. But they could easily have the user pay using
other means, outside of Near blockchain. And with some tricks, it can even be
paid using fungible tokens on Near.

In the example visualized above, the payment is done using \$FT. Together with
the transfer to John, Alice also adds an action to pay 0.1 \$FT to the relayer.
The relayer checks the content of the `SignedDelegateAction` and only processes
it if this payment is included as the first action. In this way, the relayer
will be paid in the same transaction as John.

Note that the payment to the relayer is still not guaranteed. It could be that
Alice does not have sufficient $FT and the transfer fails. To mitigate, the
relayer should check the $FT balance of Alice first.

Unfortunately, this still does not guarantee that the balance will be high
enough once the meta transaction executes. The relayer could waste NEAR gas
without compensation if Alice somehow reduces her \$FT balance in just the right
moment. Some level of trust between the relayer and its user is therefore
required.

The vision here is that there will be mostly application-specific relayers. A
general-purpose relayer is difficult to implement with just the MVP. See
limitations below.
