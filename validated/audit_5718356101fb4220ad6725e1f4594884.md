### Title
Missing Slippage Protection in `initiateWithdrawal` Allows Users to Receive Less Asset Than Expected - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

The `initiateWithdrawal` function in `LRTWithdrawalManager.sol` locks in an `expectedAssetAmount` at initiation time, but the actual payout is recalculated at the time the operator calls `unlockQueue`. Because `_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`, any adverse movement in the rsETH/asset exchange rate during the withdrawal delay window (up to 8 days) silently reduces the user's payout with no minimum-amount guard the user can set.

---

### Finding Description

The withdrawal lifecycle is a two-step process:

**Step 1 — `initiateWithdrawal`** (user-initiated): [1](#0-0) 

The function accepts `rsETHUnstaked` and a `referralId` but **no `minAssetAmount` parameter**. It calculates `expectedAssetAmount` from the oracle at the moment of the call and stores it in the request: [2](#0-1) 

**Step 2 — `unlockQueue`** (operator-initiated, called up to 8 days later): [3](#0-2) 

Inside `_unlockWithdrawalRequests`, the payout is recalculated at current oracle prices and the stored `expectedAssetAmount` is **overwritten** with the lower value: [4](#0-3) 

The helper that determines the final payout is: [5](#0-4) 

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

If `rsETHPrice` falls or `assetPrice` rises between initiation and unlock, `currentReturn < expectedAssetAmount` and the user silently receives less. The user has no mechanism to reject this outcome.

**Step 3 — `completeWithdrawal`** (user-initiated): [6](#0-5) 

The user receives whatever `request.expectedAssetAmount` was set to during `unlockQueue`, with no minimum-amount check: [7](#0-6) 

---

### Impact Explanation

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who initiates a withdrawal expecting `X` units of an LST (e.g., ETH, stETH, ETHx) may receive materially less if the rsETH/asset exchange rate moves adversely during the up-to-8-day delay. The rsETH is already burned from the user's wallet at initiation; the user cannot cancel or set a floor on the payout. The protocol itself does not become insolvent, but the individual user suffers a shortfall relative to the amount they were shown at initiation time.

---

### Likelihood Explanation

**Likelihood: Low.**

The withdrawal delay is configurable up to 16 days (`16 days / 12 seconds` blocks) and defaults to 8 days. [8](#0-7) 

Over an 8-day window, meaningful rsETH/LST rate divergence is plausible but not guaranteed. The operator's `unlockQueue` call does include price-band guards (`minimumAssetPrice`, `maximumRsEthPrice`, etc.), which bound the worst case, but the user has no independent control over where within that band their withdrawal is processed. [9](#0-8) 

---

### Recommendation

Add a `minAssetAmount` parameter to `initiateWithdrawal` and enforce it at the point where the final payout is transferred to the user (either in `_unlockWithdrawalRequests` or in `_processWithdrawalCompletion`). If the computed payout falls below `minAssetAmount`, the withdrawal should revert (or remain locked so the user can cancel and reclaim their rsETH).

```solidity
function initiateWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
+   uint256 minAssetAmount,   // slippage guard
    string calldata referralId
) external ...
```

---

### Proof of Concept

1. rsETH/ETH oracle rate at block N: `1.05 ETH per rsETH`.
2. User calls `initiateWithdrawal(ETH, 100e18, "")`. `expectedAssetAmount` is stored as `~105 ETH`.
3. Eight days pass; rsETH/ETH rate drops to `0.98 ETH per rsETH` (e.g., due to slashing or market movement).
4. Operator calls `unlockQueue` with price bounds that include `0.98`. `_calculatePayoutAmount` returns `min(105, 98) = 98 ETH`.
5. `request.expectedAssetAmount` is overwritten to `98 ETH`.
6. User calls `completeWithdrawal` and receives `98 ETH` — `7 ETH` less than the amount shown at initiation — with no recourse.

        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        override
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }

        IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);

        uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);

        if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();

        // preventing over-withdrawal.
        assetsCommitted[asset] += expectedAssetAmount;

        _addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
    }

        address asset,
        uint256 firstExcludedIndex,
        uint256 minimumAssetPrice,
        uint256 minimumRsEthPrice,
        uint256 maximumAssetPrice,
        uint256 maximumRsEthPrice
    )
        external
        nonReentrant
        onlySupportedAsset(asset)
        whenNotPaused
        onlyAssetTransferOrOperatorRole
        returns (uint256 rsETHBurned, uint256 assetAmountUnlocked)
    {
        ILRTOracle lrtOracle = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE));
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));

        UnlockParams memory params = _createUnlockParams(lrtOracle, unstakingVault, asset);

        _validatePrices(
            params.rsETHPrice,
            params.assetPrice,
            minimumRsEthPrice,
            maximumRsEthPrice,
            minimumAssetPrice,
            maximumAssetPrice
        );

        if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();

        // Updates and unlocks withdrawal requests up to a specified upper limit or until allocated assets are fully
        // utilized.
        (rsETHBurned, assetAmountUnlocked) = _unlockWithdrawalRequests(
            asset, params.totalAvailableAssets, params.rsETHPrice, params.assetPrice, firstExcludedIndex
        );

        if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
        //Take the amount to distribute from vault
        unstakingVault.redeem(asset, assetAmountUnlocked);

        // If Aave integration is enabled and asset is ETH, deposit to Aave
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN && assetAmountUnlocked > 0) {
            try this.depositToAaveExternal(assetAmountUnlocked) { }
            catch (bytes memory reason) {
                emit AaveDepositFailed(assetAmountUnlocked, reason);
                // Silently fail if Aave deposit fails (e.g., pool at max capacity)
                // Funds remain in contract for withdrawals
            }
        }

        emit AssetUnlocked(asset, rsETHBurned, assetAmountUnlocked, params.rsETHPrice, params.assetPrice);
    }
