Audit Report

## Title
`_checkAndUpdateDailyFeeMintLimit()` Unconditionally Reverts When `maxFeeMintAmountPerDay` Is Zero, Freezing Protocol Fee Minting and rsETH Price Updates - (File: contracts/LRTOracle.sol)

## Summary
When `maxFeeMintAmountPerDay` is zero (the Solidity default, never set by `reinitialize()`), `_checkAndUpdateDailyFeeMintLimit()` reverts on any non-zero `feeAmount` because `0 + feeAmount > 0` is always true. Since `updateRSETHPrice()` is public and calls this function whenever TVL has grown, the price update and fee-minting path is permanently blocked until a manager explicitly calls `setMaxFeeMintAmountPerDay()` with a non-zero value.

## Finding Description
`_checkAndUpdateDailyFeeMintLimit()` performs the following check unconditionally:

```solidity
// contracts/LRTOracle.sol L205-207
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
```

When `maxFeeMintAmountPerDay == 0` (default storage value) and `feeAmount > 0`, this simplifies to `feeAmount > 0`, which is always true, causing an unconditional revert. [1](#0-0) 

The upgrade entry-point `reinitialize()` only sets `feePeriodStartTime` and never initialises `maxFeeMintAmountPerDay`, leaving it at zero after every upgrade: [2](#0-1) 

`setMaxFeeMintAmountPerDay()` accepts zero without restriction, so a manager can re-trigger the condition at any time: [3](#0-2) 

In `_updateRsETHPrice()`, when `protocolFeeInETH > 0` (TVL has grown), `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` is called with a non-zero argument and reverts before `rsETHPrice = newRsETHPrice` is ever reached: [4](#0-3) 

The public `updateRSETHPrice()` has no access control, so any external caller can observe and trigger the revert: [5](#0-4) 

Note: when `protocolFeeInETH == 0` (TVL has not grown), the `else` branch calls `_checkAndUpdateDailyFeeMintLimit(0)`, which evaluates `0 > 0` â†’ false and does not revert. The bug is isolated to the fee-accrual path. [6](#0-5) 

## Impact Explanation
**Medium â€” Permanent freezing of unclaimed yield.**

Protocol treasury fees (`rsethAmountToMintAsProtocolFee`) are never minted while `maxFeeMintAmountPerDay == 0`. Because `rsETHPrice` is only written after the fee block succeeds, the stored price also becomes stale, causing depositors to receive more rsETH than entitled (at the stale lower price), diluting existing holders. The yield frozen during the broken period is partially unrecoverable due to this dilution. The condition persists indefinitely without admin intervention.

## Likelihood Explanation
The broken state is the default state after any upgrade that calls `reinitialize()` without a subsequent `setMaxFeeMintAmountPerDay()` call. No privileged action is required to trigger the revert â€” any external caller invoking the permissionless `updateRSETHPrice()` after TVL has grown will hit it. A manager can also re-trigger it at any time by calling `setMaxFeeMintAmountPerDay(0)`.

## Recommendation
Treat `maxFeeMintAmountPerDay == 0` as "no cap" (unlimited) in `_checkAndUpdateDailyFeeMintLimit()`:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (maxFeeMintAmountPerDay == 0) return; // 0 = unlimited; skip check

    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }

    if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
    }

    currentPeriodMintedFeeAmount += feeAmount;
}
```

Apply the same fix to `remainingDailyFeeMintLimit()`: return `type(uint256).max` when `maxFeeMintAmountPerDay == 0` instead of `0`. [7](#0-6) 

## Proof of Concept
1. Deploy/upgrade `LRTOracle`; call `reinitialize(_feePeriodStartTime)` â€” `maxFeeMintAmountPerDay` remains `0`.
2. Users deposit LSTs; `totalETHInProtocol` grows above `rsethSupply * rsETHPrice`.
3. Call `updateRSETHPrice()` (permissionless).
4. Inside `_updateRsETHPrice()`: `protocolFeeInETH > 0`, so `rsethAmountToMintAsProtocolFee > 0`.
5. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` evaluates `0 + rsethAmountToMintAsProtocolFee > 0` â†’ `true`.
6. Transaction reverts with `DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)`.
7. `rsETHPrice` is never updated; treasury fees are never minted.
8. Every subsequent call reverts identically until a manager calls `setMaxFeeMintAmountPerDay(nonZeroValue)`.

**Foundry test sketch:**
