Audit Report

## Title
Unguarded Division by Zero in `getRsETHAmountToMint` When `rsETHPrice` Is Uninitialized — (`contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle.rsETHPrice` is declared as a plain storage variable with no initialization in `initialize()`, leaving it at `0` until `updateRSETHPrice()` is explicitly called. `LRTDepositPool.getRsETHAmountToMint()` divides by this value without a zero-guard, causing every deposit to revert with a division-by-zero error during the post-deployment window. The secondary "Critical withdrawal fund loss" path claimed in the report is not independently valid at that severity, as analyzed below.

## Finding Description

**Root cause — primary path:**

`rsETHPrice` is declared at [1](#0-0)  with no assignment in `initialize()`: [2](#0-1) 

It is only written inside `_updateRsETHPrice()`, which is called by the public `updateRSETHPrice()`: [3](#0-2) 

`getRsETHAmountToMint()` performs an unchecked division by this value: [4](#0-3) 

Solidity 0.8 checked arithmetic causes this to revert whenever `rsETHPrice == 0`. Both `depositETH()` and `depositAsset()` call `_beforeDeposit()` → `getRsETHAmountToMint()`, so the entire deposit surface is blocked until an operator or any caller invokes `updateRSETHPrice()`.

**Secondary path — analysis:**

`getExpectedAssetAmount()` uses `rsETHPrice` as the numerator: [5](#0-4) 

`instantWithdrawal()` burns rsETH before checking the returned amount: [6](#0-5) 

For `rsETHPrice` to be `0` while rsETH is in circulation, `_updateRsETHPrice()` must compute `newRsETHPrice = 0`, which requires `totalETHInProtocol == 0` with `rsethSupply > 0`. The downside-protection circuit-breaker is gated on `pricePercentageLimit > 0`: [7](#0-6) 

Since `pricePercentageLimit` also defaults to `0` and is never set in `initialize()`, the pause is skipped and `rsETHPrice = 0` is written. However, `totalETHInProtocol == 0` with `rsethSupply > 0` means complete slashing of all protocol assets — at that point rsETH is already worth zero. Burning it for zero assets reflects the actual state rather than constituting independent theft. The "Critical" classification for this path is therefore not supported by the evidence; it conflates the slashing loss with the missing guard.

## Impact Explanation

**Medium — Temporary freezing of funds (primary path):** Every call to `depositETH()` and `depositAsset()` reverts with division-by-zero from the moment of deployment until `updateRSETHPrice()` is first called. No deposits can succeed during this window, temporarily freezing the deposit surface for all users.

The secondary withdrawal path does not independently reach Critical because the precondition (`totalETHInProtocol == 0` with live rsETH supply) already implies complete asset loss from slashing, not from this missing guard.

## Likelihood Explanation

The primary path is a near-certain transient condition on every fresh deployment or proxy upgrade: `rsETHPrice` starts at `0`, and any deposit attempted before the first `updateRSETHPrice()` call reverts. `updateRSETHPrice()` is public and permissionless, so any caller can resolve it, but the window is real and reproducible without any privileged action.

## Recommendation

Add an explicit zero-guard in `getRsETHAmountToMint()`:

```solidity
uint256 currentRsETHPrice = lrtOracle.rsETHPrice();
if (currentRsETHPrice == 0) revert InvalidRsETHPrice();
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / currentRsETHPrice;
```

Additionally, `LRTOracle.initialize()` should call `_updateRsETHPrice()` (or explicitly set `rsETHPrice = 1 ether`) so the variable is never left at `0` after deployment. Setting a non-zero default for `pricePercentageLimit` in `initialize()` would also enable the downside-protection circuit-breaker from the start.

## Proof of Concept

1. Deploy `LRTOracle` and `LRTDepositPool` without calling `updateRSETHPrice()`.
2. Confirm `lrtOracle.rsETHPrice() == 0`.
3. Any user calls `depositETH{value: 1 ether}(0, "")` or `depositAsset(asset, amount, 0, "")`.
4. Execution reaches `getRsETHAmountToMint` → `(amount * assetPrice) / 0` → Solidity 0.8 reverts with division-by-zero panic.
5. All deposits are blocked until any caller invokes `updateRSETHPrice()` (which is public and permissionless per `contracts/LRTOracle.sol` line 87).

Foundry test sketch:
```solidity
function test_depositRevertsWhenRsETHPriceZero() public {
    // rsETHPrice is 0 — updateRSETHPrice() never called
    assertEq(lrtOracle.rsETHPrice(), 0);
    vm.expectRevert(); // division by zero panic
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
}
```
