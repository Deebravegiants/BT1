Audit Report

## Title
Chainlink `minAnswer` Circuit Breaker Not Validated in `getAssetPrice()`, Enabling Excess rsETH Minting During LST Depeg — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and uses the returned price without checking it against the Chainlink aggregator's hardcoded `minAnswer` floor. When a supported LST depegs below `minAnswer`, the oracle silently returns the floor price rather than the true market price. Because `rsETHPrice` is a stored state variable updated only on explicit calls to `updateRSETHPrice()`, an attacker can deposit the depegged LST in the window before that update, receiving rsETH priced against the stale (pre-depeg) `rsETHPrice` but with the inflated `minAnswer` in the numerator, yielding excess rsETH redeemable for healthy pool assets.

## Finding Description

**Root cause.** `ChainlinkPriceOracle.getAssetPrice()` performs no bounds check:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Chainlink aggregators have a hardcoded `minAnswer`; when the true market price falls below it, `latestRoundData()` returns `minAnswer` rather than the actual price. No `IAccessControlledOffchainAggregator.minAnswer()` / `maxAnswer()` check is present.

**Price propagation.** The inflated price flows through `LRTOracle.getAssetPrice()` → `_getTotalEthInProtocol()` → `_updateRsETHPrice()`. Critically, `rsETHPrice` is a **stored state variable** updated only when `updateRSETHPrice()` is explicitly called:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
// ...
rsETHPrice = newRsETHPrice;  // L313 — written only here
```

**Minting calculation.** `LRTDepositPool.getRsETHAmountToMint()` reads the live asset price but the stale stored `rsETHPrice`:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Attack window.** Between the depeg event and the next `updateRSETHPrice()` call, `getAssetPrice(depeggedLST)` returns the inflated `minAnswer` while `rsETHPrice` remains the pre-depeg value. The ratio `minAnswer / staleRsETHPrice` is greater than `truePrice / staleRsETHPrice`, so the attacker receives more rsETH than the deposited assets are worth.

**Why existing guards fail.**

1. *Downside protection* (`LRTOracle.sol` L270-281): triggers only when `newRsETHPrice < highestRsethPrice`. Because the oracle returns an **inflated** `minAnswer`, `_getTotalEthInProtocol()` is inflated, making `newRsETHPrice` appear **higher** than the true price — the downside guard never fires.

2. *Upside protection* (`LRTOracle.sol` L252-266): if `pricePercentageLimit` is set and the inflated price pushes `newRsETHPrice` above `highestRsethPrice` by more than the limit, `updateRSETHPrice()` **reverts** for non-managers, actively **extending** the attack window.

3. *Withdrawal price validation* (`LRTWithdrawalManager.sol` L853-870): `_validatePrices()` is called only inside `unlockQueue()`, which is operator-gated. `instantWithdrawal()` and `initiateWithdrawal()` call `getExpectedAssetAmount()` which reads the same unchecked oracle, providing no protection.

**Redemption path.** `LRTWithdrawalManager.instantWithdrawal()` (L212-253) allows immediate redemption of rsETH for any supported asset held in the unstaking vault, subject only to available liquidity. `getExpectedAssetAmount()` computes `rsETHAmount * rsETHPrice / assetPrice` — with a stale (pre-depeg) `rsETHPrice` and a healthy asset price in the denominator, the attacker redeems excess rsETH for healthy assets at the inflated rate.

## Impact Explanation

**Critical — Direct theft of user funds.**

An attacker deposits a depegged LST at the inflated `minAnswer` price (before `rsETHPrice` is updated), receives excess rsETH, then redeems via `instantWithdrawal()` for healthy assets (ETH or other LSTs) held in the pool. The shortfall is borne by all existing rsETH holders whose share of the pool is diluted. The loss is bounded by available liquidity in the unstaking vault but is otherwise unbounded in principle.

## Likelihood Explanation

**Medium.** LST depeg events are rare but have occurred historically (stETH in May 2022). The protocol holds stETH, rETH, ETH-X, swETH, and sfrxETH — each with its own Chainlink feed and `minAnswer`. The attack window is the block interval between the depeg and the next `updateRSETHPrice()` call. `updateRSETHPrice()` is public but, if `pricePercentageLimit` is set, the upside revert for non-managers can extend this window. The SECURITY.md exclusion for "depegging of an external stablecoin" does not apply because LSTs are not stablecoins. The attack is executable by any unprivileged depositor with no special access.

## Recommendation

In `ChainlinkPriceOracle.getAssetPrice()`, retrieve the aggregator's `minAnswer` and `maxAnswer` from the `IAccessControlledOffchainAggregator` interface and revert if the returned price is at or outside those bounds:

```solidity
IAccessControlledOffchainAggregator aggregator =
    IAccessControlledOffchainAggregator(priceFeed.aggregator());
int192 minAnswer = aggregator.minAnswer();
int192 maxAnswer = aggregator.maxAnswer();
if (price <= minAnswer || price >= maxAnswer) revert PriceOutOfBounds();
```

Additionally, consider pausing deposits automatically when any asset price hits its circuit-breaker boundary.

## Proof of Concept

**Setup (fork mainnet, stETH feed `minAnswer` = 0.9e18):**

1. Deploy/fork with stETH as a supported asset; `rsETHPrice` = 1.0e18 (stored state).
2. Simulate a black-swan event: mock the stETH Chainlink aggregator to return `minAnswer` = 0.9e18 (true market price = 0.5e18).
3. Confirm `ChainlinkPriceOracle.getAssetPrice(stETH)` returns 0.9e18 — no revert.
4. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
   - `getRsETHAmountToMint` = `1000e18 * 0.9e18 / 1.0e18` = **900 rsETH**.
   - True value deposited = 500 ETH; attacker receives rsETH worth ~900 ETH at stale price.
5. Attacker calls `LRTWithdrawalManager.instantWithdrawal(ETH, 900e18, "")` (if instant withdrawal is enabled and ETH liquidity exists).
   - `getExpectedAssetAmount(ETH, 900e18)` = `900e18 * 1.0e18 / 1.0e18` = **900 ETH**.
   - Attacker receives ~900 ETH, netting ~400 ETH profit at the expense of existing depositors.
6. Confirm `updateRSETHPrice()` was never called between steps 2 and 5.

**Foundry fork test outline:**
