Looking at the actual code to verify each claim element.

Audit Report

## Title
`removeLiquidity` can be permanently DOSed by frontrunning `addLiquidity` to violate the `MinimalLiquidity` invariant — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`addLiquidity` accepts an arbitrary `owner` address with no `msg.sender == owner` guard, allowing any caller to credit shares to any position while paying the token cost themselves. `removeLiquidity` enforces that any non-zero remaining share balance must be ≥ `minimalMintableLiquidity`. An attacker can frontrun a victim's full-withdrawal transaction by adding a dust amount of shares to the victim's position, causing the remaining-shares check to revert and blocking the victim from exiting indefinitely.

## Finding Description

`MetricOmmPool.addLiquidity` passes `owner` directly to `LiquidityLib.addLiquidity` with no ownership restriction:

```solidity
// MetricOmmPool.sol L182-196 — no msg.sender == owner check
function addLiquidity(address owner, uint80 salt, ...) external nonReentrant(...) {
    ...
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
        _liquidityContext(), owner, salt, deltas, ...
    );
}
```

The callback is invoked on `msg.sender` (attacker pays), but shares are written to the victim's key:

```solidity
// LiquidityLib.sol L121
positionBinShares[posKey] = newUserShares;  // posKey = keccak256(owner, salt, bin)
```

`addLiquidity` only checks that the resulting total is ≥ `minimalMintableLiquidity`:

```solidity
// LiquidityLib.sol L76-79
uint256 newUserShares = userShares + sharesToAdd;
if (newUserShares < ctx.minimalMintableLiquidity) {
    revert MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

When the victim already holds `userShares >= minimalMintableLiquidity`, the attacker can add any `delta >= 1` and this check passes (`userShares + delta >= minimalMintableLiquidity`).

`removeLiquidity` then enforces:

```solidity
// LiquidityLib.sol L199-202
uint256 newUserShares = userShares - sharesToRemove;
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

When the victim submits a full withdrawal (`sharesToRemove == original userShares`), `newUserShares` would be 0, which passes (`0 > 0` is false). But after the attacker's frontrun adds `delta` shares, `newUserShares = delta`, and since `delta > 0 && delta < minimalMintableLiquidity`, the revert fires. The `msg.sender != owner` guard on `removeLiquidity` prevents the victim from being forced out; it only blocks their own exit.

## Impact Explanation

The victim's LP position is frozen: they cannot call `removeLiquidity` to recover their token principal. The attacker can repeat the frontrun for every retry the victim makes, making the DOS persistent. This constitutes an unusable core liquidity-withdrawal flow and temporary loss of access to user principal, matching the "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" impact criterion.

## Likelihood Explanation

- `addLiquidity` is fully permissionless with no owner check — any EOA or contract can invoke it for any `owner`/`salt`.
- The victim's `owner`, `salt`, and `binIdxs` are visible in pending transaction calldata on a public mempool.
- The cost per frontrun is the token value of `delta` shares (minimum 1 scaled unit, i.e., dust), making repeated griefing economically viable.
- The attacker need not steal funds; they only need to keep the victim's `newUserShares` in the range `(0, minimalMintableLiquidity)` after each removal attempt.

## Recommendation

Add an ownership check to `addLiquidity` so that only the designated `owner` (or an approved operator) can credit shares to a position:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
```

This mirrors the guard already present in `removeLiquidity` at `MetricOmmPool.sol` L206 and eliminates the ability for third parties to manipulate another user's share balance.

Alternatively, if permissionless liquidity-on-behalf-of is an intentional design feature, the `MinimalLiquidity` check in `removeLiquidity` should be skipped when `newUserShares == 0` (full exit), i.e., change the condition to only apply when the caller is partially reducing their position.

## Proof of Concept

1. Pool has `minimalMintableLiquidity = 100`. Victim holds `userShares = 1000` in bin 0 with `salt = 42`.
2. Victim submits `removeLiquidity(victim, 42, {binIdxs:[0], shares:[1000]}, "")`. Expected: `newUserShares = 0`, check `0 > 0` → false → passes.
3. Attacker observes the pending transaction, reads `owner = victim`, `salt = 42`, `binIdxs = [0]`.
4. Attacker submits `addLiquidity(victim, 42, {binIdxs:[0], shares:[1]}, ...)` with higher gas. In `addLiquidity`: `newUserShares = 1001 >= 100` → passes. Victim's `positionBinShares[key]` becomes `1001`.
5. Victim's transaction executes: `userShares = 1001`, `sharesToRemove = 1000`, `newUserShares = 1`. Check: `1 > 0 && 1 < 100` → **revert `MinimalLiquidity(1, 100)`**.
