### Title
Missing CKD Response Verification for Legacy `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Unauthorized Confidential Key Derivation Output — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function performs cryptographic output verification **only** for the `AppPublicKeyPV` (publicly verifiable) variant. For the legacy `AppPublicKey` variant, the match arm is empty — no verification is performed. A single malicious attested participant (strictly below the signing threshold) can front-run the legitimate MPC nodes and deliver a fabricated `CKDResponse` that the contract accepts and forwards to the user without any cryptographic check, bypassing the threshold requirement for confidential key derivation.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function handles responses to confidential key derivation requests. After confirming the caller is an attested participant and the domain key exists, it branches on the request's `app_public_key` variant: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

The function then unconditionally calls `resolve_yields_for`, delivering the (potentially fabricated) response to every waiting user yield: [2](#0-1) 

The `AppPublicKeyPV` path calls `ckd_output_check`, which enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`: [3](#0-2) 

For `AppPublicKey`, no analogous check exists. The contract cannot verify the response because it does not hold the user's private scalar `r`, but it also does not require any threshold agreement before accepting the response.

By contrast, the `respond` function for ECDSA/EdDSA signatures always verifies the signature cryptographically against the stored public key before accepting it: [4](#0-3) 

This asymmetry means a single malicious attested participant can deliver a fake CKD response for `AppPublicKey` requests, but **cannot** deliver a fake signature response (which would require forging a signature under the threshold public key).

The `resolve_yields_for` helper removes the request from the pending map on the first successful call, so the first `respond_ckd` call wins — legitimate nodes that respond later receive `RequestNotFound`: [5](#0-4) 

The `assert_caller_is_attested_participant_and_protocol_active` check only confirms the caller holds a valid TEE attestation and is in the active participant set — it does not require threshold agreement: [6](#0-5) 

During resharing, `active_participants()` returns the **new** (prospective) participant set: [7](#0-6) 

This means a newly added participant who has not yet received any key shares can immediately call `respond_ckd` with a fabricated response during resharing, widening the attack surface.

---

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) can:

1. Observe a pending `request_app_private_key` call using the `AppPublicKey` variant on-chain.
2. Immediately call `respond_ckd` with arbitrary `big_y` and `big_c` G1 points of their choosing.
3. The contract accepts the fabricated response (no verification for `AppPublicKey`).
4. `resolve_yields_for` resumes all queued user yields with the fake response.
5. The user receives a confidential key that the attacker chose, not the threshold-derived key.
6. Because the attacker selected `big_y` and `big_c`, they can compute the same "derived key" the user will unmask, enabling decryption of the user's confidential material.

This constitutes **confidential key derivation output without the required participant authorization** — a Critical impact under the allowed scope. The threshold requirement (which exists to prevent any single node from unilaterally producing key material) is completely bypassed for this request variant.

---

### Likelihood Explanation

**Medium.** The attacker must hold a valid TEE attestation and be in the active participant set — a non-trivial prerequisite. However, only **one** such participant needs to be malicious. NEAR transactions are publicly observable, making front-running feasible. The `AppPublicKey` variant is the legacy path and remains supported for existing integrations. During resharing, the attack window expands to include newly added participants who have not yet received key shares.

---

### Recommendation

1. **Require threshold agreement for `AppPublicKey` responses**: Collect responses from at least `t` distinct attested participants and only deliver the response when a quorum submits identical `(big_y, big_c)` values.
2. **Deprecate `AppPublicKey` in favor of `AppPublicKeyPV`**: The publicly verifiable variant allows on-chain verification via `ckd_output_check` and should be the only accepted path going forward.
3. **At minimum, add explicit documentation** explaining that `AppPublicKey` responses are unverified on-chain and that a single malicious participant can substitute an arbitrary response.

---

### Proof of Concept

```
1. Contract is Running with a BLS12-381 CKD domain and a valid keyset.
2. User calls request_app_private_key({
       derivation_path: "my/path",
       app_public_key: AppPublicKey(g1 * r),   // legacy variant
       domain_id: 0
   }) with 1 yoctoNEAR deposit.
   → Pending CKD request stored in pending_ckd_requests.

3. Malicious attested participant (participant index i, i < threshold) observes
   the pending request on-chain.

4. Malicious participant calls respond_ckd(request, CKDResponse {
       big_y: g1 * x,   // attacker-chosen scalar x
       big_c: g1 * y,   // attacker-chosen scalar y
   }).

5. respond_ckd checks:
   - assert_caller_is_signer()                          ✓ (participant calls directly)
   - is_running_or_resharing()                          ✓
   - accept_requests                                    ✓
   - assert_caller_is_attested_participant_...          ✓ (valid TEE attestation)
   - public_key_extended(domain_id)                     ✓ (BLS key exists)
   - match AppPublicKey(_) => {}                        ← NO VERIFICATION
   - resolve_yields_for(...)                            ← delivers fake response

6. User's yield resumes with CKDResponse { big_y: g1*x, big_c: g1*y }.
   User computes: confidential_key = CKDOutput::new(big_y, big_c).unmask(r)
   → wrong key, fully known to the attacker (who chose x and y).

7. Legitimate MPC nodes later call respond_ckd with the correct response
   → RequestNotFound (request already removed from map).
```

The analog to the L1ERC20Bridge vulnerability is direct: just as `deposit` proceeds with an uninitialized `l2Bridge = address(0)` causing funds to be locked irrecoverably, `respond_ckd` proceeds with an unverified `AppPublicKey` response, delivering attacker-controlled key material to the user with no on-chain recourse.

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
