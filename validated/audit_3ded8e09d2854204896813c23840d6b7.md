Audit Report

## Title
Censorable `rewardsToken` Permanently Freezes Staker Rewards in `KernelDepositPool` — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.getReward()` unconditionally calls `rewardsToken.safeTransfer(msg.sender, rewardAmount)` with no fallback or rescue path. If `rewardsToken` is a censorable ERC20 such as USDC or USDT and a staker's address is added to the token's blacklist, every subsequent call to `getReward()` reverts permanently, freezing that staker's accrued yield in the contract with no recovery mechanism. The same structural issue in `MerkleDistributor.claim()` can freeze all users' claims if `protocolTreasury` is blacklisted.

## Finding Description
`KernelDepositPool.getReward()` is the sole reward-claim path:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L382-390
function getReward() external nonReentrant updateReward(msg.sender) {
    uint256 rewardAmount = rewards[msg.sender];
    if (rewardAmount > 0) {
        rewards[msg.sender] = 0;
        rewardsToken.safeTransfer(msg.sender, rewardAmount); // reverts if blacklisted
        emit RewardsClaimed(msg.sender, rewardAmount);
    }
}
``` [1](#0-0) 

Because `rewards[msg.sender] = 0` and `safeTransfer` execute in the same transaction, a revert on the transfer atomically rolls back the zero-assignment, leaving `rewards[msg.sender]` intact but permanently unclaimable. `rewardsToken` is set once at initialization with no admin setter anywhere in the contract, so the token cannot be changed post-deployment. [2](#0-1) 

There is no alternative claim address, no try/catch, and no admin rescue function in the admin section of the contract. [3](#0-2) 

In `MerkleDistributor.claim()`, two sequential `safeTransfer` calls are made:

```solidity
// contracts/utils/MerkleDistributor/MerkleDistributor.sol L141-144
IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);
``` [4](#0-3) 

If `protocolTreasury` is blacklisted by the reward token, the second transfer reverts and every user's claim fails, regardless of whether the individual `account` is blacklisted.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** A staker who has legitimately earned rewards through `KernelDepositPool` permanently loses access to those rewards if their address is blacklisted by the reward token. The funds remain in the contract with no recovery path. This maps directly to the allowed impact: *Medium. Permanent freezing of unclaimed yield.* In the `MerkleDistributor` variant, blacklisting of `protocolTreasury` simultaneously freezes all users' unclaimed yield.

## Likelihood Explanation
USDC and USDT are the most widely used reward tokens in DeFi and both maintain active, on-chain blacklists exercised for regulatory and sanctions compliance. The protocol's `notifyRewardAmount` admin function accepts any ERC20 as `rewardsToken`, making USDC/USDT a realistic deployment choice. No malicious action by the protocol is required — a routine blacklisting event by the token issuer is sufficient to trigger the freeze. The scenario is foreseeable, documented, and has been accepted as valid in numerous prior audits of similar staking contracts.

## Recommendation
1. **Wrap the transfer in a try/catch** and retain `rewards[msg.sender]` if the transfer fails, allowing a future retry after the blacklist is lifted.
2. **Add an admin rescue function** that can redirect a specific address's frozen reward balance to an alternative recipient.
3. **For `MerkleDistributor`**, decouple the fee transfer from the user transfer — skip or defer the fee if the treasury transfer fails rather than reverting the entire claim.
4. Alternatively, restrict `rewardsToken` at initialization to a known non-censorable token.

## Proof of Concept
1. Deploy `KernelDepositPool` with USDC as `rewardsToken`.
2. Alice stakes KERNEL tokens; over time `rewards[alice]` accrues to `1000e6`.
3. USDC issuer blacklists Alice's address.
4. Alice calls `getReward()`.
5. `rewardsToken.safeTransfer(alice, 1000e6)` reverts; the entire transaction reverts including the `rewards[alice] = 0` assignment.
6. `rewards[alice]` remains `1000e6`; every subsequent call by Alice reverts identically.
7. No admin function exists to redirect or recover Alice's balance — her 1000 USDC is permanently frozen.

For `MerkleDistributor`: replace Alice with any user and blacklist `protocolTreasury` — the second `safeTransfer` at L144 reverts for every caller, freezing all pending claims simultaneously.
