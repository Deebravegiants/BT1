Audit Report

## Title
ETH Deposit Limit Bypass via Missing `amount` in `_checkIfDepositAmountExceedesCurrentLimit` - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` uses asymmetric logic between the ETH and ERC20 branches. The ETH branch evaluates `totalAssetDeposits > limit` without adding the incoming `amount`, so the limit check never accounts for the new deposit. Any depositor can push total ETH deposits arbitrarily above the admin-configured cap in a single transaction.

## Finding Description
At lines 676–682 of `contracts/LRTDepositPool.sol`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ERC20 branch correctly checks `totalAssetDeposits + amount > limit`. The ETH branch checks only `totalAssetDeposits > limit`, ignoring the incoming `amount` entirely. This means the guard only triggers if the running total has *already* exceeded the cap before the deposit, never if the new deposit is the one that crosses it.

The call chain is:
- `depositETH` (L76–93, public `payable`, no role restriction) passes `msg.value` as `depositAmount` to `_beforeDeposit` (L648–670).
- `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` at L661 and reverts with `MaximumDepositLimitReached` only if it returns `true`.
- Because `amount` is dropped in the ETH branch, the function returns `false` whenever `totalAssetDeposits ≤ limit`, regardless of how large `msg.value` is.

No existing guard compensates for this omission.

## Impact Explanation
The ETH deposit limit (`depositLimitByAsset`) is a protocol-level risk-management cap controlling how much ETH is restaked into EigenLayer strategies. Because the cap is not enforced for ETH deposits, any depositor can exceed it in a single call. The protocol fails to deliver its stated deposit ceiling — matching the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`depositETH` is a public `payable` function with no role restriction. The condition is met whenever `totalAssetDeposits` is at or near (but not yet past) the configured limit, which is the normal operating state. No special privileges, victim mistakes, or external conditions are required. Any depositor can trigger this repeatedly.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets ETH deposit limit to 1 000 ETH via `lrtConfig.depositLimitByAsset(ETH_TOKEN) = 1000e18`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns 999e18 (999 ETH already deposited).
3. Attacker calls `depositETH{value: 10 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999e18 > 1000e18` → `false` → no revert.
5. Deposit succeeds; total ETH deposits become 1 009 ETH — 9 ETH above the configured cap.

**Foundry test plan:**
```solidity
function test_ETHLimitBypass() public {
    // set limit to 1000 ether
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, 1000 ether);
    // deposit 999 ether to approach limit
    vm.deal(alice, 1009 ether);
    vm.prank(alice);
    depositPool.depositETH{value: 999 ether}(0, "");
    // single deposit of 10 ether should revert but does not
    vm.prank(alice);
    depositPool.depositETH{value: 10 ether}(0, ""); // expect revert MaximumDepositLimitReached — but it passes
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 1000 ether);
}
```
