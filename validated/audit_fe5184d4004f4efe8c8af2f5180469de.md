Audit Report

## Title
Missing Strategy Validation on Asset Addition Allows Deposits but Permanently Blocks Withdrawals Until Admin Intervention - (File: contracts/LRTConfig.sol / contracts/LRTWithdrawalManager.sol)

## Summary
`LRTConfig._addNewSupportedAsset()` sets `isSupportedAsset[asset] = true` and `depositLimitByAsset[asset]` but never sets `assetStrategy[asset]`, leaving it at `address(0)`. `LRTDepositPool.depositAsset()` gates only on `onlySupportedERC20Token`, which checks `isSupportedAsset` alone, so deposits succeed immediately. `LRTWithdrawalManager.initiateWithdrawal()` additionally enforces `onlySupportedStrategy`, which reverts for any non-ETH asset with `assetStrategy == address(0)`. Any LST deposited before the admin separately calls `updateAssetStrategy` is frozen in the protocol until that second admin transaction is issued.

## Finding Description

`_addNewSupportedAsset` sets only `isSupportedAsset` and `depositLimitByAsset`: [1](#0-0) 

`onlySupportedERC20Token` in `LRTConfigRoleChecker` checks only `isSupportedAsset` and the ETH exclusion â€” no strategy check: [2](#0-1) 

`depositAsset` uses only `onlySupportedERC20Token`, so it succeeds for any asset with `isSupportedAsset == true` regardless of whether a strategy is set: [3](#0-2) 

`onlySupportedStrategy` in `LRTWithdrawalManager` reverts with `StrategyNotSupported` for any non-ETH asset whose `assetStrategy` is `address(0)`: [4](#0-3) 

Both `initiateWithdrawal` and `instantWithdrawal` carry this modifier, so neither withdrawal path is available: [5](#0-4) [6](#0-5) 

`updateAssetStrategy` is a completely separate `DEFAULT_ADMIN_ROLE` call that must be issued after asset addition: [7](#0-6) 

The vulnerability is present from block 0: `initialize` adds stETH and ethX via `_addNewSupportedAsset` without setting any strategy for either: [8](#0-7) 

## Impact Explanation

Any LST deposited while `assetStrategy[asset] == address(0)` accumulates in the deposit pool and node delegators. The depositor receives rsETH but cannot call `initiateWithdrawal` or `instantWithdrawal` for that specific asset â€” both revert with `StrategyNotSupported`. The deposited LST tokens remain frozen in the protocol until the admin issues a separate `updateAssetStrategy` transaction. This matches **Medium â€” Temporary freezing of funds**: the freeze is bounded by admin intervention, not permanent, but it is externally observable and user-triggered without any attacker involvement.

## Likelihood Explanation

The vulnerable window opens at protocol initialization (stETH and ethX are added without strategies from block 0) and repeats for every future `addNewSupportedAsset` call. Deposits are open immediately after asset addition; the admin must remember to issue a second, separate transaction before any user deposits. The gap is observable on-chain, and normal users depositing a newly listed LST will trigger the freeze without any malicious intent. No special privileges or unrealistic assumptions are required on the user's side.

## Recommendation

Combine asset addition and strategy assignment into a single atomic operation to eliminate the window:

```solidity
// Option A: require strategy at asset-add time
function addNewSupportedAsset(
    address asset,
    uint256 depositLimit,
    address strategy
) external onlyRole(LRTConstants.TIME_LOCK_ROLE) {
    _addNewSupportedAsset(asset, depositLimit);
    // set strategy atomically
    assetStrategy[asset] = strategy;
    emit AssetStrategyUpdate(asset, strategy);
}
```

Alternatively, guard `depositAsset` to reject deposits for assets with no strategy set:

```solidity
// Option B: guard in depositAsset
if (lrtConfig.assetStrategy(asset) == address(0)) revert StrategyNotSet();
```

## Proof of Concept

1. Admin calls `LRTConfig.addNewSupportedAsset(newLST, 100_000 ether)` â€” `isSupportedAsset[newLST] = true`, `assetStrategy[newLST] = address(0)`.
2. User calls `LRTDepositPool.depositAsset(newLST, 10 ether, minRSETH, "")` â€” passes `onlySupportedERC20Token`, succeeds; user receives rsETH; 10 ether of `newLST` sits in the deposit pool.
3. User calls `LRTWithdrawalManager.initiateWithdrawal(newLST, rsETHAmount, "")` â€” `onlySupportedStrategy` fires before any state change: `lrtConfig.assetStrategy(newLST) == address(0)` â†’ **reverts `StrategyNotSupported`**.
4. The 10 ether of `newLST` is frozen in the protocol. The user's rsETH is not burned (modifier reverts before `safeTransferFrom`), but the deposited LST cannot be reclaimed via any withdrawal path until the admin calls `LRTConfig.updateAssetStrategy(newLST, strategyAddr)`.

**Foundry test sketch:**
```solidity
function test_depositWithoutStrategy_blocksWithdrawal() public {
    // admin adds asset without strategy
    vm.prank(timeLockAdmin);
    lrtConfig.addNewSupportedAsset(address(newLST), 100_000 ether);
    // assetStrategy[newLST] == address(0) confirmed
    assertEq(lrtConfig.assetStrategy(address(newLST)), address(0));

    // user deposits successfully
    vm.startPrank(user);
    newLST.approve(address(lrtDepositPool), 10 ether);
    lrtDepositPool.depositAsset(address(newLST), 10 ether, 0, "");
    vm.stopPrank();

    // user cannot withdraw
    vm.prank(user);
    vm.expectRevert(ILRTWithdrawalManager.StrategyNotSupported.selector);
    lrtWithdrawalManager.initiateWithdrawal(address(newLST), rsETHAmount, "");
}
```
