### Title
Impure `error` in `toPlutusScriptPurpose 'PlutusV4` Escapes `Either` Monad, Crashing Any Node That Processes a PlutusV4 Script in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs`)

---

### Summary

The `EraPlutusTxInfo 'PlutusV4 DijkstraEra` instance contains a stub implementation of `toPlutusScriptPurpose` that unconditionally calls `error "stub: PlutusV4 not yet implemented"`. This is an impure Haskell exception that escapes the `Either (ContextError era)` monad entirely. Any node that attempts to build a `PlutusWithContext` for a PlutusV4 script in Dijkstra era will crash rather than returning a `Left ContextError`, because the exception is thrown inside `toPlutusV4Args` which is called from `toPlutusWithContext` — a function whose return type is `Either (ContextError era) PlutusWithContext`. The `Either` wrapper provides no protection against impure exceptions.

  → Alonzo.toPlutusWithContext (Left p)
    → toPlutusArgs SPlutusV4 ...          -- line 118 of TxInfo.hs
      → toPlutusV4Args proxy pv txInfo ... -- line 576/590

`toPlutusTxInfo SPlutusV4 lti` (lines 500–574) is itself a proper `Either`-based computation and does **not** call `toPlutusScriptPurpose`. The exception is thrown one step later, inside `toPlutusArgs`.

**The `Either` wrapper provides no protection.** `toPlutusWithContext` has return type `Either (ContextError era) PlutusWithContext`, but Haskell's `do`-notation for `Either` only catches values returned via `Left`; it cannot intercept an `ErrorCall` thrown by `error`. The exception propagates up the call stack uncaught.

**`lookupTxInfoResultImpossible` is not involved here.** That function is a guard for calling `lookupTxInfoResult` with a language unsupported by the era. In Dijkstra era, `SPlutusV4` is fully supported and `lookupTxInfoResult SPlutusV4` returns the correct field — the crash happens later, in `toPlutusArgs`. [6](#0-5)

### Impact Explanation

Any honest node in Dijkstra era that receives a block containing a transaction with a PlutusV4 script will call `mkPlutusWithContext (DijkstraPlutusV4 p)`, which triggers the `error` stub. The node crashes (or, if the runtime catches the `ErrorCall` at a higher level, the block is rejected with an unspecified internal error). Nodes that handle the exception differently from nodes that crash will diverge on whether the block is accepted. This matches the critical impact: **honest nodes accept an invalid block or transaction causing permanent ledger divergence requiring a hard fork**, or alternatively a network-wide halt on any block containing a PlutusV4 script.

---

### Likelihood Explanation

PlutusV4 is explicitly listed as a supported language in Dijkstra era (`mkSupportedLanguage PlutusV4 = Just $ SupportedLanguage SPlutusV4`). Any unprivileged transaction author can submit a transaction that uses a PlutusV4 script. No special privileges, governance majority, or key compromise is required. The attacker only needs to craft a valid-looking transaction with a PlutusV4 script and submit it to the mempool.

### Recommendation

Replace the `error` stub with a proper `Either`-returning implementation:

```haskell
toPlutusScriptPurpose proxy pv purpose =
  Conway.transPlutusPurposeV3 proxy pv purpose
  -- or the correct V4-specific translation once implemented
```

Until the full PlutusV4 `toPlutusScriptPurpose` is implemented, the stub must at minimum return a `Left` error value rather than throwing an impure exception:

```haskell
toPlutusScriptPurpose _ _ _ =
  Left $ inject $ PlutusPurposeNotSupported ...
```

Additionally, audit all other `error` calls in the Dijkstra translation path — notably the catch-all in `transTxCert`: [7](#0-6)
