### Title
Stale `rsETHPrice` Used in `getRsETHAmountToMint` Allows Depositors to Steal Accrued Yield from Existing rsETH Holders — (`contracts/LRTDepositPool.sol`)

### Summary

`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, a stored state variable that is only updated when `updateRSETHPrice()` is explicitly called. Between updates, staking yield accrues and the true rsETH/ETH rate rises, but the stored price remains stale-low. Any depositor who deposits while the price is stale receives more rsETH than the current TVL justifies, diluting the unclaimed yield of all existing rsETH holders.

### Finding Description

`LRTOracle` stores `rsETHPrice` as a persistent state variable: [1](#0-0) 

It is only updated when `_updateRsETHPrice()` is called (via the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`): [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` uses this stored value directly to compute how many rsETH tokens to mint for a depositor: [3](#0-2) 

Between price updates, EigenLayer staking rewards cause the true TVL to grow. The true rsETH price (`totalETHInProtocol / rsethSupply`) rises above the stored `rsETHPrice`. Because the denominator in the minting formula is stale-low, every depositor during this window receives more rsETH than the current protocol state warrants.

Concretely, if yield `Y` has accrued since the last update (so true price = `(S·P + Y)/S` where `S` = supply, `P` = stale price), a depositor of `D` ETH receives `D/P` rsETH instead of the correct `D·S/(S·P + Y)`. The excess rsETH minted is:

```
excess = D/P − D·S/(S·P + Y) = D·Y / (P·(S·P + Y))
```

This excess is not backed by any ETH, so it dilutes the yield that existing holders had already earned but not yet had reflected in the price.

The `_updateRsETHPrice()` function itself also uses the stale price to compute `previousTVL`: [4](#0-3) 

When the price is eventually updated, `rewardAmount = totalETHInProtocol − previousTVL` correctly captures all yield since the last update. However, the rsETH supply has already been inflated by the stale-price deposits, so the new price is set lower than it would have been without those deposits — permanently transferring value from pre-existing holders to the late depositors.

Additionally, `LRTWithdrawalManager.getExpectedAssetAmount()` also reads the stale `rsETHPrice` to compute withdrawal amounts at `initiateWithdrawal` time: [5](#0-4) 

A user who initiates a withdrawal while `rsETHPrice` is stale-high (e.g., before a slashing event is reflected) locks in a higher-than-deserved asset amount, compounding the mis-accounting.

### Impact Explanation

Existing rsETH holders accumulate yield continuously as EigenLayer staking rewards increase the protocol TVL. That yield is only "crystallised" into the price when `updateRSETHPrice()` is called. Any deposit made while the price is stale mints excess rsETH that is not backed by ETH, permanently diluting the yield owed to pre-existing holders. This constitutes **theft of unclaimed yield** (High impact per the allowed scope).

### Likelihood Explanation

`updateRSETHPrice()` is not called atomically with deposits; it is an independent transaction. In any realistic deployment the price is updated on a schedule (e.g., daily or per-epoch). Every deposit made in the interval between two updates exploits the stale price. No special permissions, front-running, or external compromise are required — any unprivileged depositor calling `depositETH` or `depositAsset` during a stale-price window is affected. Likelihood is **Low** because the window is bounded by the update cadence and the yield rate, but the condition is structurally guaranteed to recur.

### Recommendation

Update `rsETHPrice` atomically inside `_beforeDeposit` (or at the start of `depositETH`/`depositAsset`) before computing `rsethAmountToMint`. Alternatively, compute the mint amount on-the-fly using the live TVL (`_getTotalEthInProtocol() / rsethSupply`) rather than the stored `rsETHPrice`, so that the minting rate always reflects the current state of the protocol.

### Proof of Concept

1. Protocol state: rsETH supply = 1 000, TVL = 1 100 ETH, stored `rsETHPrice` = 1.0 ETH (true price = 1.1 ETH; 100 ETH yield has accrued since last update).
2. Attacker calls `depositETH{value: 100 ETH}(0, "")`.
3. `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.0e18 = 100 rsETH`. Correct amount at true price would be `100e18 * 1e18 / 1.1e18 ≈ 90.9 rsETH`. Attacker receives **≈ 9.1 excess rsETH**.
4. Anyone calls `updateRSETHPrice()`. New supply = 1 100, TVL = 1 200 ETH. `previousTVL = 1 100 * 1.0 = 1 100`. `rewardAmount = 100 ETH` (the original yield). After fee, new price ≈ `1 200 / 1 100 ≈ 1.0909 ETH` — lower than the 1.1 ETH the original holders deserved, because the inflated supply diluted their yield.
5. Attacker holds 100 rsETH worth ≈ 109.09 ETH, having deposited 100 ETH — a risk-free gain of ≈ 9.09 ETH extracted from existing holders' unclaimed yield.
