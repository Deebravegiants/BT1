### Title
Sub-transaction ExUnits not included in `maxTxExUnits` / `maxBlockExUnits` validation in Dijkstra era - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

The Dijkstra era introduces batch transactions: a top-level `Tx TopTx era` may embed an ordered map of sub-transactions (`OMap TxId (Tx SubTx era)`). Each sub-transaction carries its own `TxWits` with its own redeemers and declared `ExUnits`. However, the two protocol-parameter guards that bound Plutus execution cost — the per-transaction limit (`maxTxExUnits`) and the per-block limit (`maxBlockExUnits`) — are evaluated using `totExUnits`, which only sums the redeemers of the single transaction object it receives. Sub-transactions' redeemers are never included. An unprivileged submitter can therefore craft a batch whose sub-transactions collectively declare ExUnits far exceeding both limits, causing block validators to perform arbitrarily more Plutus computation than the protocol parameters are designed to allow.

---

### Finding Description

**`totExUnits` is blind to sub-transactions.**

`totExUnits` is defined in `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It folds only over the redeemers stored in the single `tx` argument's witness set. [1](#0-0) 

There is no Dijkstra-era override of `totExUnits` that would also traverse `tx ^. bodyTxL . subTransactionsTxBodyL` and accumulate sub-transactions' redeemers. The `grep` results confirm `totExUnits` appears only in Alonzo-era files.

**Per-transaction ExUnits check ignores sub-transactions.**

`dijkstraUtxoTransition` enforces the per-tx limit with:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [2](#0-1) 

`validateExUnitsTooBigUTxO` calls `totExUnits tx`, which only counts the top-level transaction's redeemers. [3](#0-2) 

**Per-block ExUnits check ignores sub-transactions.**

`dijkstraBbodyTransition` enforces the per-block limit with:

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
``` [4](#0-3) 

`validateExUnits` computes `foldMap totExUnits txs`, summing only the top-level ExUnits of each transaction in the block. [5](#0-4) 

**SUBUTXO rule has no ExUnits check at all.**

`dijkstraSubUtxoTransition` performs no ExUnits validation. The error-mapping function for SUBUTXO explicitly marks `ExUnitsTooBigUTxO` as unreachable:

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [6](#0-5) 

The full `dijkstraSubUtxoTransition` transition rule contains no call to `validateExUnitsTooBigUTxO` or any equivalent. [7](#0-6) 

---

### Impact Explanation

The `maxTxExUnits` and `maxBlockExUnits` protocol parameters exist specifically to bound the CPU steps and memory that Plutus script evaluation may consume during block validation, ensuring that block validation time and memory usage remain within safe limits for all honest nodes. [8](#0-7) 

Because sub-transactions' declared ExUnits are never checked against either limit, an unprivileged submitter can include sub-transactions whose redeemers declare ExUnits of arbitrary magnitude. The Plutus interpreter uses the declared ExUnits as its execution budget; a sub-transaction declaring, e.g., 100× `maxTxExUnits` will cause the interpreter to run for up to 100× longer than the protocol intends to permit. Multiplied across multiple sub-transactions and multiple top-level transactions in a block, the total Plutus computation in a single block can exceed `maxBlockExUnits` by an unbounded factor.

This matches the allowed impact: **Medium — attacker-controlled transactions exceed intended validation limits.**

---

### Likelihood Explanation

The attack requires no special privilege. Any user who can submit a Dijkstra-era transaction can embed sub-transactions with arbitrarily large ExUnits. The sub-transaction body format (`sub_transaction_body`) includes a `script_data_hash` field and the witness set includes `redeemers` with `ex_units`, so the serialization path is fully available to any submitter. [9](#0-8) 

A block producer willing to include such a transaction (or a block producer who is themselves the attacker) can cause every validating node to perform far more Plutus computation than `maxBlockExUnits` permits.

---

### Recommendation

Replace the single-level `totExUnits` call in both the UTXO and BBODY rules with a batch-aware variant that also folds over sub-transactions' redeemers. Concretely:

1. **Define a Dijkstra-aware `batchTotExUnits`** in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` (or `Tx.hs`) that sums ExUnits across the top-level transaction and all sub-transactions:

```haskell
batchTotExUnits :: (AlonzoEraTxWits era, DijkstraEraTxBody era) => Tx TopTx era -> ExUnits
batchTotExUnits tx =
  totExUnits tx
    <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
```

2. **Override `validateExUnitsTooBigUTxO`** in `dijkstraUtxoTransition` to use `batchTotExUnits` instead of `totExUnits`.

3. **Override `validateExUnits`** in `dijkstraBbodyTransition` to use `foldMap batchTotExUnits txs` instead of `foldMap totExUnits txs`.

---

### Proof of Concept

A submitter constructs a Dijkstra-era top-level transaction whose body contains a sub-transaction. The sub-transaction's witness set includes a redeemer with `ExUnits { exUnitsMem = maxBound, exUnitsSteps = maxBound }`. The top-level transaction's own redeemers are empty (or within limits).

- `validateExUnitsTooBigUTxO pp tx` evaluates `totExUnits tx = mempty` (no top-level redeemers) — passes.
- `validateExUnits txs ppMax` evaluates `foldMap totExUnits txs = mempty` — passes.
- The SUBUTXO rule processes the sub-transaction with no ExUnits check.
- The Plutus interpreter is invoked with the declared `maxBound` budget, consuming resources far beyond `maxBlockExUnits`.

The entry path is: unprivileged user → submit Dijkstra `Tx TopTx` with sub-transactions carrying oversized ExUnits → accepted by UTXO and BBODY rules → Plutus interpreter runs with unchecked budget during block validation. [10](#0-9) [11](#0-10) [7](#0-6)

validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) $
    ExUnitsTooBigUTxO Mismatch {mismatchSupplied = totalExUnits, mismatchExpected = maxTxExUnits}
  where
    maxTxExUnits = pp ^. ppMaxTxExUnitsL
    -- This sums up the ExUnits for all embedded Plutus Scripts anywhere in the transaction:
    totalExUnits = totExUnits tx

dijkstraSubUtxoTransition = do
  TRC (SubUtxoEnv slot pp certState originalUtxo (IsValid isValid), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG

  let txBody = tx ^. bodyTxL

  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo
  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  let allSizedOutputs = txBody ^. allSizedOutputsTxBodyF

  runTest $ Alonzo.validateOutputTooBigUTxO pp allOutputs

  runTest $ Shelley.validateInputSetEmptyUTxO txBody

  let inputs = txBody ^. inputsTxBodyL
  let refInputs = txBody ^. referenceInputsTxBodyL
  runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxoState) inputs

  runTestOnSignal $ Shelley.validateOutputBootAddrAttrsTooBig allOutputs

  runTestOnSignal $ validateWrongNetworkInDirectDeposit netId txBody
  runTestOnSignal $ Alonzo.validateWrongNetworkInTxBody netId txBody

    then do

        Shelley.updateUTxOStateNoFees
          pp
          utxoState
          txBody
          certState
