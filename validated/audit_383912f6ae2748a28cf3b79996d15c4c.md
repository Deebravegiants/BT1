### Title
Stale DRep Reverse Delegation During Bootstrap Phase Enables Incorrect Vote Delegation Erasure, Corrupting Governance Ratification - (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version < 10), when a stake credential re-delegates its vote from one DRep credential (`drepA`) to another (`drepB`), the old DRep's reverse delegation index (`DRepState.drepDelegs`) is intentionally not cleaned up. When `drepA` subsequently unregisters, `clearDRepDelegations` iterates over the stale `drepDelegs` set and unconditionally sets `dRepDelegationAccountStateL .~ Nothing` for every credential in it — including credentials that have already validly re-delegated to `drepB`. This incorrectly erases the stake credential's current delegation, removing its stake from the DRep distribution and corrupting governance ratification.

---

### Finding Description

**Vulnerability class:** Delegation accounting bug — state recorded but not properly propagated; stale index causes incorrect erasure of valid state.

**Step 1 — Stale reverse delegation is created during re-delegation.**

In `processDelegationInternal`, when `preserveIncorrectDelegation` is `True` (i.e., `pvMajor pv < natVersion @10`), re-delegation to a new DRep credential only inserts the stake credential into the new DRep's `drepDelegs` set. It does **not** remove the credential from the old DRep's `drepDelegs` set: [1](#0-0) 

The condition at line 363 — `isNothing mAccountState || preserveIncorrectDelegation` — causes the `delegVote` branch to skip `unDelegReDelegDRep` (which would have removed the old reverse delegation) and instead only call `Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred` on the new DRep. The old DRep's `drepDelegs` set retains a stale entry for `stakeCred`.

The forward delegation in the account IS updated correctly: [2](#0-1) 

So after re-delegation: `accountState.dRepDelegationAccountStateL = Just drepB` (correct), but `drepA.drepDelegs` still contains `stakeCred` (stale).

**Step 2 — Old DRep unregistration incorrectly clears the valid delegation.**

When `drepA` unregisters via `ConwayUnRegDRep`, `clearDRepDelegations` iterates over `drepA.drepDelegs` and sets `dRepDelegationAccountStateL .~ Nothing` for every credential in the set: [3](#0-2) 

Because `stakeCred` is still in `drepA.drepDelegs` (stale), this call sets `stakeCred`'s `dRepDelegationAccountStateL` to `Nothing`, erasing its valid delegation to `drepB`.

**Step 3 — Corrupted DRep distribution.**

`computeDRepDistr` builds the DRep distribution by reading `accountState ^. dRepDelegationAccountStateL` for each account: [4](#0-3) 

After the erasure, `stakeCred`'s account has `dRepDelegationAccountStateL = Nothing`, so its stake is excluded from `drepB`'s distribution entirely. `drepB` loses voting power it legitimately held.

**Step 4 — Governance ratification uses the corrupted distribution.**

`dRepAcceptedRatio` iterates over `reDRepDistr` to compute the yes/total ratio: [5](#0-4) 

With `drepB`'s stake artificially reduced, governance actions that `drepB` votes on may fail to reach threshold even when they should pass, or the reduced denominator may allow other actions to pass that should not.

**The test suite explicitly confirms this behavior:** [6](#0-5) 

The comment "we cannot `expectNotDelegatedVote` because the delegation is still in the DRepState of the other drep" confirms that after `drepCred` unregisters, `cred`'s `dRepDelegationAccountStateL` is `Nothing` (delegation erased) while `drepCred2.drepDelegs` still contains `cred` (inconsistent state).

### Impact Explanation

An attacker can manipulate governance ratification by:
1. Registering an attacker-controlled DRep (`drepA`).
2. Registering stake credentials and delegating them to `drepA`.
3. Re-delegating those credentials to a target DRep (`drepB`) they wish to weaken.
4. Unregistering `drepA`.

After step 4, all re-delegated credentials have their delegation to `drepB` erased. `drepB`'s voting power in `reDRepDistr` is reduced by the sum of those credentials' stake. This can cause governance actions (protocol parameter changes, treasury withdrawals, committee updates, hard fork initiations, no-confidence motions) to be incorrectly ratified or blocked, constituting unauthorized governance action enactment.

This maps to: **Critical — Unauthorized governance action is enacted** (or blocked when it should pass), and **High — DRep distribution is modified outside design parameters**, corrupting the deterministic governance ratification outcome.

---

### Likelihood Explanation

- The bootstrap phase (protocol version 9) is the current Conway mainnet state.
- The attack requires only standard certificate transactions: `ConwayRegDRep`, `RegDepositDelegTxCert` (or `DelegTxCert`), and `ConwayUnRegDRep`. All are available to any unprivileged transaction sender.
- No privileged access, key leakage, or majority control is required.
- The attacker only needs to control `drepA` (their own DRep) and a set of stake credentials. The stake credentials can be freshly registered with minimal ADA deposits.
- The `drepDeposit` refund is returned on unregistration, so the cost is only transaction fees and key deposits.

---

### Recommendation

In `processDelegationInternal`, when `preserveIncorrectDelegation` is `True` and the account already has an existing DRep delegation (i.e., `mAccountState` is `Just accountState` with a non-`Nothing` `dRepDelegationAccountStateL`), the old DRep's `drepDelegs` entry should still be removed before adding to the new DRep. The `unDelegReDelegDRep` function already implements this correctly for the post-bootstrap path: [7](#0-6) 

Alternatively, `clearDRepDelegations` in `ConwayUnRegDRep` should verify that each credential in `drepDelegs` still has its `dRepDelegationAccountStateL` pointing to the unregistering DRep before clearing it, rather than unconditionally setting it to `Nothing`.

The hard fork transition at version 10 (`updateDRepDelegations`) already rebuilds the delegation index correctly: [8](#0-7) 

A targeted fix to `clearDRepDelegations` to guard against stale entries would mitigate the issue without requiring a hard fork.

---

### Proof of Concept

**Attacker-controlled entry path (all standard certificate transactions):**

1. Attacker registers `drepA` (`ConwayRegDRep`).
2. Attacker registers stake credential `cred` (`ConwayRegCert`).
3. Attacker delegates `cred` to `drepA` (`DelegTxCert cred (DelegVote (DRepCredential drepA))`).
4. Attacker re-delegates `cred` to target `drepB` (`DelegTxCert cred (DelegVote (DRepCredential drepB))`).
   - During bootstrap: `drepA.drepDelegs` still contains `cred` (stale); `cred.dRepDelegationAccountStateL = Just drepB` (correct).
5. Attacker unregisters `drepA` (`ConwayUnRegDRep`).
   - `clearDRepDelegations` sets `cred.dRepDelegationAccountStateL = Nothing`.
6. `computeDRepDistr` now excludes `cred`'s stake from `drepB`'s distribution.
7. `dRepAcceptedRatio` computes a lower ratio for `drepB`, potentially causing governance actions voted on by `drepB` to fail threshold checks they would otherwise pass.

The test at `eras/conway/impl/testlib/Test/Cardano/Ledger/Conway/Imp/DelegSpec.hs` lines 313–323 directly demonstrates this outcome in the bootstrap branch. [9](#0-8)

dRepAcceptedRatio ::
  forall era.
  RatifyEnv era ->
  Map (Credential DRepRole) Vote ->
  GovAction era ->
  Rational
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr

unDelegReDelegDRep ::
  ConwayEraAccounts era =>
  Credential Staking ->
  -- | Account that is losing its current delegation and/or acquiring a new one
  AccountState era ->
  -- | Potential new delegation. In case when stake credential unregisters this must be `Nothing`.
  Maybe DRep ->
  VState era ->
  VState era
unDelegReDelegDRep stakeCred accountState mNewDRep =
  fromMaybe (vsDRepsL %~ addNewDelegation) $ do
    dRep@(DRepCredential dRepCred) <- accountState ^. dRepDelegationAccountStateL
    pure $
      -- There is no need to update set of delegations if delegation is unchanged
      if Just dRep == mNewDRep
        then id
        else
          vsDRepsL %~ addNewDelegation . Map.adjust (drepDelegsL %~ Set.delete stakeCred) dRepCred
  where
    addNewDelegation =
      case mNewDRep of
        Just (DRepCredential dRepCred) ->
          Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
        _ -> id

updateDRepDelegations :: ConwayEraCertState era => CertState era -> CertState era
updateDRepDelegations certState =
  let accountsMap = certState ^. certDStateL . accountsL . accountsMapL
      dReps =
        -- Reset all delegations in order to remove any inconsistencies
        -- Delegations will be reset accordingly below.
        Map.map (\dRepState -> dRepState {drepDelegs = Set.empty}) $
          certState ^. certVStateL . vsDRepsL
      (dRepsWithDelegations, accountsWithoutUnknownDRepDelegations) =
        Map.mapAccumWithKey adjustDelegations dReps accountsMap
      adjustDelegations ds stakeCred accountState =
        case accountState ^. dRepDelegationAccountStateL of
          Just (DRepCredential dRep) ->
            let addDelegation _ dRepState =
                  Just $ dRepState {drepDelegs = Set.insert stakeCred (drepDelegs dRepState)}
             in case Map.updateLookupWithKey addDelegation dRep ds of
                  (Nothing, _) -> (ds, accountState & dRepDelegationAccountStateL .~ Nothing)
                  (Just _, ds') -> (ds', accountState)
          _ -> (ds, accountState)
   in certState
        -- Remove dangling delegations to non-existent DReps:
        & certDStateL . accountsL . accountsMapL .~ accountsWithoutUnknownDRepDelegations
        -- Populate DRep delegations with delegatees
        & certVStateL . vsDRepsL .~ dRepsWithDelegations
