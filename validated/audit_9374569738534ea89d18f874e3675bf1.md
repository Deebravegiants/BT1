### Title
Stale `target_shard_index` in `GetExecutionOutcome` produces invalid `outcome_root_proof` after resharding — (`chain/client/src/view_client_actor.rs`)

### Summary

In the `GetExecutionOutcome` RPC handler, `target_shard_index` is computed from the shard layout of the block containing the outcome (pre-resharding epoch). When `get_next_block_hash_with_new_chunk` crosses an epoch boundary with a new shard layout, it returns a post-resharding block and an updated child `shard_id`, but `target_shard_index` is never recomputed. The stale index is then used to select the Merkle path from `outcome_roots` collected from the post-resharding block, producing an incorrect `outcome_root_proof` that fails verification.

### Finding Description

`GetExecutionOutcome` computes `target_shard_index` at lines 1155–1158 using the pre-resharding `shard_layout`. It then calls `get_next_block_hash_with_new_chunk` (line 1159), which — when a resharding event is crossed — updates the tracked `shard_id` to a child shard and returns a block `h` in the post-resharding epoch. The handler updates `outcome_proof.block_hash` to `h` and collects `outcome_roots` from `h`'s chunks (post-resharding, more entries, different index mapping), but uses the original `target_shard_index` to index into `merklize(&outcome_roots).1`. For accounts that map to the second child shard (index = parent_index + 1 in the new layout), this selects the wrong Merkle path.

### Impact Explanation

Any call to `EXPERIMENTAL_light_client_proof` or `light_client_proof` for a transaction whose outcome block is in epoch N and whose "next block with new chunk" falls in epoch N+1 (post-resharding) will return an `outcome_root_proof` that fails `verify_path`. Light clients and cross-chain bridges relying on this proof to confirm transaction finality will reject valid proofs, breaking bridge operation for all accounts that map to the second child shard after resharding.

### Likelihood Explanation

Resharding is a planned, recurring event. Any transaction submitted in the last few blocks of epoch N, where the shard containing the sender's account is split at the N→N+1 boundary, triggers this bug. The condition is reachable by any unprivileged user simply by submitting a transaction and requesting a light client proof after resharding. The second child shard receives roughly half the accounts of the split parent, so approximately 50% of affected accounts trigger the wrong-index path.

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, target_shard_id)`, recompute `target_shard_index` using the shard layout of block `h`:

```rust

    let target_shard_index = h_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
```

### Proof of Concept

1. Configure a test-loop with two epochs: epoch 0 uses layout `[shard_A, shard_B]`; epoch 1 splits `shard_B` into `[child_1, child_2]`.
2. Submit a transaction from an account that maps to `child_2` (the second child, index = parent_index + 1 in the new layout) in the last block of epoch 0.
3. After epoch 1 starts, call `GetExecutionOutcome` for that transaction.
4. Observe that `target_shard_index` = parent's index (e.g., 1), but `outcome_roots` has 3 entries and the correct index for `child_2` is 2.
5. `verify_path(block_outcome_root, outcome_root_proof, chunk_outcome_root)` returns `false`.
