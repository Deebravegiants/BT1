### Title
Missing Payload-Hash-to-Request Binding in `respond_verify_foreign_tx` Enables Forged Foreign-Chain Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs only a one-directional check: it verifies that the submitted `response.payload_hash` carries a valid MPC threshold signature, but it never verifies that `response.payload_hash` was actually derived from the pending request's data (`SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))`). A single malicious MPC node acting as the signing leader can craft an arbitrary `payload_hash`, coordinate honest followers into signing it (followers do not independently verify the foreign chain transaction), and submit the forged response. The contract accepts it, resolves all queued yields with the forged data, and the pending request is permanently consumed.

---

### Finding Description

**Root cause — the missing check:**

In `respond_verify_foreign_tx` (lines 718–743), the contract extracts `payload_hash` directly from the caller-supplied response and verifies only that the signature is valid for that hash under the root public key:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

There is no check that `payload_hash == SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))`. The contract never reconstructs or bounds the hash to the pending request. [1](#0-0) 

**Contrast with the SDK's correct check:**

The `near-mpc-sdk` crate's `ForeignChainSignatureVerifier::verify_signature` performs both checks — it first verifies `expected_payload_hash == response.payload_hash`, then verifies the signature against the expected hash:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [2](#0-1) 

The contract performs only the second half of this check. The SDK performs both. The contract is the authoritative settlement layer; the SDK check is optional and caller-side only.

**Why followers sign an arbitrary hash:**

In `crates/node/src/providers/verify_foreign_tx/sign.rs`, the leader builds the `SignatureRequest` by calling `foreign_tx_payload.compute_msg_hash()` and passing the resulting hash into the threshold signing protocol. Followers participate in the CaitSith threshold ECDSA protocol and sign whatever hash the leader provides — they do not independently inspect the foreign chain or recompute the payload hash. A malicious leader can substitute any 32-byte value as the hash to sign. [3](#0-2) 

**Exploit flow:**

1. User calls `verify_foreign_transaction` for a real foreign-chain transaction (e.g., a Bitcoin UTXO). The request is stored in `pending_verify_foreign_tx_requests`.
2. A malicious MPC node is elected leader for this request (leader election is hash-based and rotates across participants).
3. The malicious leader crafts `payload_hash = [0u8; 32]` (or any arbitrary value) instead of computing `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))`.
4. The leader runs the CaitSith threshold signing protocol with honest followers. Followers sign `[0u8; 32]` — they have no mechanism to reject it.
5. The leader calls `respond_verify_foreign_tx` with `payload_hash = [0u8; 32]` and the valid threshold signature.
6. The contract verifies the signature is valid for `[0u8; 32]` under the root key — it is — and calls `pending_requests::resolve_yields_for`, draining all queued yields with the forged response. [4](#0-3) 

7. Every user who submitted the same request receives the forged `VerifyForeignTransactionResponse`. The pending map entry is permanently removed; no legitimate response can ever be submitted for this request.

---

### Impact Explanation

**Impact: High — Forged foreign-chain verification causing invalid bridge execution.**

- Any downstream contract that consumes the `VerifyForeignTransactionResponse` without independently calling `ForeignChainSignatureVerifier::verify_signature` (the SDK helper) will accept the forged verification as proof that the foreign-chain transaction occurred and had specific extracted values. This enables invalid bridge execution: e.g., minting wrapped tokens for a Bitcoin deposit that never happened, or releasing funds on the basis of a fabricated block hash.
- Even for callers that do use the SDK verifier, the pending request is permanently consumed by the forged response. No legitimate re-submission is possible because `resolve_yields_for` removes the map entry. All queued yields (potentially from multiple users who submitted the same request) are resolved with the forged data. [5](#0-4) 

---

### Likelihood Explanation

- The attacker is a single MPC node acting as leader — strictly below the signing threshold. No collusion with other participants is required; honest followers sign whatever hash the leader provides.
- Leader election rotates across participants. A malicious node will be elected leader for some fraction of requests proportional to its share of the participant set.
- The attack requires no privileged access beyond being an attested participant, which is the normal operating role of an MPC node.
- The contract provides no defense: the missing check is absent entirely, not gated behind a flag or configuration.

---

### Recommendation

Add a binding check inside `respond_verify_foreign_tx` that reconstructs the expected payload hash from the request and verifies it matches `response.payload_hash` before accepting the response. Since the contract does not receive `extracted_values`, the fix requires one of:

1. **Include `extracted_values` in the response** so the contract can recompute `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))` and compare it to `response.payload_hash`. This mirrors the SDK's `verify_signature` logic.
2. **Require followers to independently verify the foreign chain transaction** before participating in signing, so the threshold signature itself guarantees the hash is correct. This is a node-level fix and does not require contract changes, but it must be enforced by protocol.

Option 1 closes the gap at the contract layer (the authoritative settlement point). Option 2 closes it at the node layer but relies on all honest followers implementing the check correctly.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,

                &expected_payload_hash,

        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)

fn build_signature_request(

    foreign_tx_payload: &dtos::ForeignTxSignPayload,
