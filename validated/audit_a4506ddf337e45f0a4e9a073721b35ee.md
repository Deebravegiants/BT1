Let me look at a few more critical details before rendering a verdict.

### Title
Stale OS-measurement votes from removed participants are counted toward threshold in `vote_add_os_measurement` when `clean_tee_status` promise fails or races — (`crates/contract/src/tee/measurements.rs`, `crates/contract/src/lib.rs`)

---

### Summary

`MeasurementVotes::count_votes` counts every entry in `vote_by_account` that matches the proposed action, with no filter for whether those voters are still current participants. `vote_add_os_measurement` relies entirely on the detached `clean_tee_status` promise to purge stale votes before any new vote is counted. Because the promise is fire-and-forget (`.detach()`), a failure or a race window between resharing completion and promise execution leaves stale votes in storage. A single fresh vote from a current participant can then push the raw count over threshold, adding a malicious OS measurement with fewer than threshold current-participant votes.

---

### Finding Description

**Root cause — `MeasurementVotes::count_votes` has no participant filter** [1](#0-0) 

`count_votes` iterates `vote_by_account.values()` and counts every entry matching the action. It does not receive the current `Participants` set and cannot filter stale voters.

**`vote_add_os_measurement` uses this raw count directly** [2](#0-1) 

The returned `votes` value is compared against `self.threshold()?.value()` with no inline filtering. If stale entries remain in `measurement_votes.vote_by_account`, they are counted.

**Cleanup is a detached, fire-and-forget promise** [3](#0-2) 

The `CLEAN_TEE_STATUS` promise is spawned with `.detach()`. If it fails (e.g., out of gas, or the contract is in a non-Running state when the receipt executes), the resharing transaction still succeeds and stale votes remain indefinitely.

**`clean_tee_status` itself can fail** [4](#0-3) 

If the protocol state is not `Running` when the receipt executes (e.g., a second resharing started in the same block), `clean_tee_status` returns `Err(InvalidState::ProtocolStateNotRunning)` and the stale votes are never removed.

**`clean_non_participant_votes` is the only removal path** [5](#0-4) 

There is no inline filtering in `vote_measurement` or `vote_add_os_measurement`. The only way stale votes are removed is through this function, called exclusively from `clean_tee_status`.

**Contrast: every other threshold-gated vote is hardened against this exact failure**

`vote_update` explicitly filters inline: [6](#0-5) 

`ForeignChainRpcWhitelist::vote` uses `count_for` with a participant predicate: [7](#0-6) 

`TeeVerifierVotes::vote` uses `count_for` with a participant predicate: [8](#0-7) 

The comment in `vote_reshared` acknowledges this pattern for `vote_update` but does not apply it to `vote_add_os_measurement`: [9](#0-8) 

**Race window even when `clean_tee_status` succeeds**

The detached promise receipt is queued after the `vote_reshared` transaction completes. Other transactions — including `vote_add_os_measurement` — can be interleaved before the cleanup receipt executes. This is a NEAR-protocol-level ordering property, not a configuration issue.

---

### Impact Explanation

A Byzantine coalition of fewer than threshold current participants can add a malicious OS measurement (`ContractExpectedMeasurements`) to the on-chain allowed list. Once added, any node running software whose TEE measurements match that entry can submit a `submit_participant_info` call and receive a `VerifiedAttestation`. If that node is also admitted as a participant (through a subsequent resharing), it can participate in threshold signing with attacker-controlled software, enabling unauthorized threshold signature issuance or confidential key derivation.

---

### Likelihood Explanation

- Requires 2 Byzantine participants (below threshold T=3 in N=5) to vote for a malicious measurement before resharing — achievable by Byzantine behavior strictly below threshold.
- Requires a resharing that removes those 2 participants — a normal operational event.
- Requires either: (a) `clean_tee_status` to fail (out-of-gas with the 10 TGas default allocation, or a concurrent state transition), or (b) a `vote_add_os_measurement` call to land in the race window between `vote_reshared` completing and the cleanup receipt executing.
- Requires 1 additional participant to vote for the same measurement — achievable by social engineering (presenting M as a legitimate OS upgrade).

The race window exists on every resharing. The gas allocation (`DEFAULT_CLEAN_TEE_STATUS_TERA_GAS = 10`) may be insufficient if the participant set is large. [10](#0-9) 

---

### Recommendation

Apply the same inline-filtering pattern used by `vote_update`, `ForeignChainRpcWhitelist::vote`, and `TeeVerifierVotes::vote`:

In `MeasurementVotes::vote` (or in `vote_add_os_measurement` before the threshold check), filter `vote_by_account` to count only entries whose `AuthenticatedParticipantId` is still in the current `Participants` set. Pass the current `ThresholdParameters` (already retrieved at line 1510) into `vote_measurement` / `count_votes` so the count is always participant-scoped, making correctness independent of whether `clean_tee_status` has executed.

---

### Proof of Concept

```rust
// Unit test (no clean_non_participant_votes called between resharing and fresh vote)
let mut all_participants = Participants::new();
let mut account_ids = Vec::new();
for i in 0..5 { let (id, info) = gen_participant(i); account_ids.push(id.clone()); all_participants.insert(id, info).unwrap(); }

let mut tee_state = TeeState::default();
let malicious = make_measurement(0xAA);

// P0 and P1 (Byzantine) vote before resharing
for id in &account_ids[0..2] {
    testing_env!(VMContextBuilder::new().signer_account_id(id.clone()).build());
    let auth = AuthenticatedParticipantId::new(&all_participants).unwrap();
    tee_state.measurement_votes.vote(MeasurementVoteAction::Add(malicious.clone()), &auth);
}

// Resharing removes P0 and P1; clean_tee_status is NOT called (simulating failure/race)
let new_participants = all_participants.subset(2..5); // {P2, P3, P4}, T=3

// P2 (honest) votes for M
testing_env!(VMContextBuilder::new().signer_account_id(account_ids[2].clone()).build());
let auth_p2 = AuthenticatedParticipantId::new(&new_participants).unwrap();
let count = tee_state.measurement_votes.vote(MeasurementVoteAction::Add(malicious.clone()), &auth_p2);

// BUG: count == 3 (2 stale + 1 fresh), threshold == 3 → measurement is added
assert_eq!(count, 3); // should be 1
```
