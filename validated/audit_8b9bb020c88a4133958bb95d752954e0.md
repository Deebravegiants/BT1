### Title
Sierra-to-CASM Compilation Triggered Before Signature and Balance Verification in Declare Transaction Admission — (`File: crates/apollo_gateway/src/gateway.rs`)

---

### Summary

The Apollo gateway performs the CPU- and memory-intensive Sierra-to-CASM compilation for `Declare` transactions **before** verifying the sender's signature or account balance. An attacker can submit declare transactions with a valid Sierra class body but an invalid signature or zero balance, forcing the sequencer to compile the class at full cost while the transaction is ultimately rejected without any fee being charged.

---

### Finding Description

In `GenericGateway::add_tx_inner` (`crates/apollo_gateway/src/gateway.rs`), the processing order for a `Declare` transaction is:

1. `check_declare_permissions` — checks `block_declare` flag and `authorized_declarer_accounts` allowlist.
2. `stateless_tx_validator.validate(&tx)` — checks class size, Sierra version, resource-bound format, and min gas price. **Does not verify signature or balance.**
3. Acquire `compilation_permit` from `declare_compilation_semaphore` (bounded to `max_concurrent_declare_compilations`, default 40).
4. `convert_rpc_tx_to_internal_and_executable_txs` → `convert_rpc_tx_to_internal` → **`class_manager_client.add_class(tx.contract_class)`** — this triggers the full Sierra-to-CASM compilation pipeline inside `ClassManager::add_class`.
5. Drop `compilation_permit`.
6. `stateful_transaction_validator.extract_state_nonce_and_run_validations` — **only here** are nonce, balance, and the `__validate_declare__` entry point checked. [1](#0-0) 

The compilation itself is performed inside `ClassManager::add_class`: [2](#0-1) 

And is triggered from `TransactionConverter::convert_rpc_tx_to_internal`: [3](#0-2) 

The codebase itself explicitly documents this ordering problem: [4](#0-3) 

> "Compiling a declared Sierra class to CASM is CPU- and memory-intensive, and it happens during transaction ingestion **before the transaction's signature and balance are verified**."

The semaphore (`declare_compilation_semaphore`) is the only mitigation, bounding concurrent compilations to 40 by default: [5](#0-4) 

---

### Impact Explanation

An attacker can continuously drain sequencer CPU and memory by submitting declare transactions that pass stateless checks but carry an invalid signature or come from an account with zero balance. Each such transaction forces a full Sierra-to-CASM compilation (spawning a resource-limited subprocess, serializing/deserializing the class, computing the compiled class hash) before the gateway discovers the transaction is invalid and rejects it without charging any fee.

The semaphore caps the blast radius to 40 concurrent compilations per gateway instance, but the attacker can cycle continuously: submit 40 declares → wait for compilations to complete and fail → submit 40 more. There is no cooldown, no fee charged, and no state change that would prevent re-submission with a fresh class body.

This maps to: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing**, because the gateway performs the full admission-time compilation work for transactions that should have been rejected earlier (before the expensive step).

---

### Likelihood Explanation

The attack requires only:
- A valid Sierra class body (passes stateless size/version checks; any real contract class works).
- An invalid signature (e.g., all-zero felts) or a sender address with zero STRK balance.
- Resource bounds that meet the minimum gas price threshold.

No privileged access, no special network position, and no cryptographic capability is required. The attack is fully unprivileged and can be automated trivially.

---

### Recommendation

Move signature and balance pre-checks **before** the compilation step. Concretely:

- **Short term:** Reorder `add_tx_inner` so that `stateful_transaction_validator.extract_state_nonce_and_run_validations` (or at minimum the balance and nonce checks within it) runs **before** `convert_rpc_tx_to_internal_and_executable_txs` is called for `Declare` transactions. The compilation permit acquisition and compilation itself should only proceed after the sender is confirmed to have sufficient balance and a valid nonce.
- **Long term:** Consider a two-phase admission for declares: (1) validate signature and balance cheaply, then (2) compile. This mirrors the recommendation in the referenced report to "charge gas as early as possible."

---

### Proof of Concept

```
1. Craft a valid Sierra contract class C that passes stateless validation
   (size ≤ max_contract_bytecode_size, version in [min, max], entry points sorted).

2. Build a DeclareV3 RPC transaction T with:
   - contract_class = C
   - sender_address = any address with zero STRK balance (e.g., a freshly generated key)
   - signature = [0x0, 0x0]  (invalid)
   - resource_bounds.l2_gas.max_price_per_unit ≥ min_gas_price (8 gwei default)
   - compiled_class_hash = correct hash of C (passes the hash-mismatch check post-compilation)

3. Submit T to the gateway's add_tx endpoint up to 40 times concurrently
   (one per semaphore slot).

4. Observe: the gateway compiles C for each submission (CPU spike, memory spike),
   then rejects each transaction at the stateful validation step with
   "Account balance is smaller than the transaction's max_fee" or
   "Account validation failed" — no fee is charged.

5. Repeat immediately after the previous batch completes.
   The attacker pays zero; the sequencer bears the full compilation cost.
``` [6](#0-5) [7](#0-6)

    async fn extract_state_nonce_and_run_validations(
        &mut self,
        executable_tx: &ExecutableTransaction,
        mempool_client: SharedMempoolClient,
    ) -> StatefulTransactionValidatorResult<Nonce> {
        let account_nonce =
            self.get_nonce_from_state(executable_tx.contract_address()).await.map_err(|e| {
                // TODO(noamsp): Fix this. Need to map the errors better.
                StarknetError::internal_with_signature_logging(
                    format!(
                        "Failed to get nonce for sender address {}",
                        executable_tx.contract_address()
                    ),
                    &executable_tx.signature(),
                    e,
                )
            })?;
        let skip_validate =
            self.run_pre_validation_checks(executable_tx, account_nonce, mempool_client).await?;
        self.run_validate_entry_point(executable_tx, skip_validate).await?;
        Ok(account_nonce)
    }
