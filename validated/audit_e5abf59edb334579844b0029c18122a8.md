### Title
`AnchoredPriceProvider` uses L1-only staleness check, causing `FeedStalled` reverts on L2 networks — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` is described as "the one standard provider for public pools." The protocol is deployed on multiple L2 networks (Arbitrum, Base, Optimism, Linea). However, `AnchoredPriceProvider._isStale()` uses the strict L1 logic that treats any oracle `refTime` ahead of `block.timestamp` as stale. On L2 networks, sequencer clock drift routinely causes oracle publication timestamps to be slightly ahead of `block.timestamp`. This causes valid oracle data to be rejected, making every swap revert with `FeedStalled`.

---

### Finding Description

`AnchoredPriceProvider` contains an `_isStale` function explicitly annotated as L1-only:

```solidity
/// @dev Pure staleness check (L1). Any future refTime is stale.
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;   // ← hard rejection of any future timestamp
    return (nowTs - refTime) > maxDelta;
}
``` [1](#0-0) 

This check is called in `_readLeg`, which is the only path through which the anchor oracle data is consumed:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
``` [2](#0-1) 

When `_readLeg` returns `ok = false`, `_getBidAndAskPrice` returns the `(0, type(uint128).max)` sentinel, and `getBidAndAskPrice()` reverts with `FeedStalled`: [3](#0-2) 

The codebase already recognizes this L2 clock-drift problem. `PriceProviderL2` was created specifically to tolerate oracle timestamps slightly ahead of `block.timestamp` via a `FUTURE_TOLERANCE` parameter:

```solidity
/// @dev L2 sequencer timestamp can lag behind oracle publication time.
///      Allows refTime up to FUTURE_TOLERANCE seconds ahead of block.timestamp.
uint256 public immutable FUTURE_TOLERANCE;
``` [4](#0-3) 

```solidity
if (refTime > nowTs) {
    // refTime in the future: tolerate only within futureTol
    return (refTime - nowTs) > futureTol;
}
``` [5](#0-4) 

No equivalent `AnchoredPriceProviderL2` exists. The directory contains `AnchoredPriceProvider.sol` and `AnchoredProviderFactory.sol` only — no L2 variant:



The protocol is explicitly configured for L2 deployment. Feed configuration files exist for Arbitrum, Base, Optimism, and Linea:



---

### Impact Explanation

On any L2 network where the oracle's `refTime` is even 1 second ahead of `block.timestamp` (a normal condition due to sequencer clock skew), `AnchoredPriceProvider._isStale()` returns `true`, `_readLeg` returns `ok = false`, and `getBidAndAskPrice()` reverts with `FeedStalled`. Since `AnchoredPriceProvider` is the standard provider for public pools, this breaks all swaps on those pools for the duration of the clock skew. This is broken core pool functionality causing unusable swap flows, qualifying under the allowed impact gate.

---

### Likelihood Explanation

L2 sequencer clock drift is a well-documented, recurring phenomenon. The development team already addressed it for `PriceProvider` by creating `PriceProviderL2`, demonstrating awareness of the issue. The omission of the same fix in `AnchoredPriceProvider` — the more widely used "standard" provider — means any L2 deployment of a public pool using `AnchoredPriceProvider` is affected whenever the oracle node's clock is ahead of the sequencer's reported `block.timestamp`. This is not a rare edge case; it is a structural property of L2 networks.

---

### Recommendation

Create an `AnchoredPriceProviderL2` variant that adds a `FUTURE_TOLERANCE` immutable (bounded to ≤ 1 hour, matching `PriceProviderL2`) and replaces the `_isStale` body with the L2-aware version from `PriceProviderL2`:

```solidity
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta,
    uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;
}
```

Pass `FUTURE_TOLERANCE` from the constructor and thread it through `_readLeg`. Create a corresponding `AnchoredProviderFactoryL2` that deploys this variant for L2 networks.

---

### Proof of Concept

1. Deploy `AnchoredPriceProvider` on Arbitrum (or any L2) with `MAX_REF_STALENESS = 60 seconds`.
2. The Pyth/Chainlink oracle node publishes a price update with `refTime = block.timestamp + 2` (2 seconds ahead due to sequencer clock skew — a normal condition).
3. Call `getBidAndAskPrice()` on the pool.
4. Inside `_readLeg`, `_isStale(refTime, block.timestamp, MAX_REF_STALENESS)` evaluates `refTime > nowTs` → `true` → returns `(mid, spreadBps, refTime, false)`.
5. `_getBidAndAskPrice` returns `(0, type(uint128).max)`.
6. `getBidAndAskPrice()` reverts with `FeedStalled`.
7. All swaps on the pool revert for the duration of the clock skew, with no user recourse.
