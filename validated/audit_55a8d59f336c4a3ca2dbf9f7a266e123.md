### Summary
In the Dijkstra era, the `directDeposits` field of a transaction body is applied to account balances by the ENTITIES/SUBENTITIES rules, but is never included in either the `produced` or `consumed` value calculations used by the preservation-of-value (`consumed == produced`) check. An unprivileged transaction sender can include arbitrary `directDeposits` amounts in a transaction, causing ADA to be credited to account balances without being debited from any UTxO input â€” effectively creating ADA from nothing.

### Finding Description
The Dijkstra era introduces `DirectDeposits` â€” a `Map AccountAddress Coin` that, when present in a transaction body, credits the specified amounts to the corresponding account balances. This is applied in `dijkstraEntitiesTransition` (top-level transactions) and `dijkstraSubEntitiesTransition` (sub-transactions) via `applyDirectDeposits`. [1](#0-0) 

The preservation-of-value check (`validateValueNotConservedUTxO`) compares `consumed pp certState utxo txBody` against `produced pp certState txBody`. For Dijkstra:

**Consumed side** â€” `getConsumedDijkstraValue` calls `getConsumedMaryValue` for each tx body, summing UTxO inputs, refunds, withdrawals, and minted assets. `directDeposits` is absent. [2](#0-1) 

**Produced side (top-level tx)** â€” `dijkstraProducedValue` delegates to `conwayProducedValue` (outputs + fee + deposits + treasuryDonation + burned assets) and folds over sub-tx produced values. `directDeposits` is absent. [3](#0-2) 

**Produced side (sub-tx)** â€” `dijkstraSubTxProducedValue` sums outputs, deposits, treasuryDonation, and burned assets. `directDeposits` is absent. [4](#0-3) 

Compare to `treasuryDonation`, which is correctly included on the produced side in `conwayProducedValue`: [5](#0-4) 

Because `directDeposits` appears on neither side of the equation, the preservation-of-value check passes regardless of the `directDeposits` amounts. The ENTITIES rule then unconditionally applies those deposits to account balances, creating ADA that was never deducted from any UTxO input.

### Impact Explanation
**Critical â€” Direct creation of ADA through an invalid ledger state transition.**

An attacker can include a `directDeposits` map with arbitrarily large amounts in a valid Dijkstra transaction. The transaction passes all ledger checks including preservation of value, and the specified amounts are credited to account balances. This violates the fundamental invariant that total ADA in the system is conserved. The attacker can target any registered account address, not only their own.

### Likelihood Explanation
**High.** The attack requires only:
1. A registered account address (the attacker can register their own with a standard stake credential registration certificate)
2. Enough ADA to pay the minimum transaction fee
3. A structurally valid Dijkstra-era transaction

No privileged access, governance majority, or leaked keys are required. Any unprivileged transaction sender can exploit this on any Dijkstra-era node.

### Recommendation
Include the sum of all `directDeposits` amounts on the `produced` side of the preservation-of-value calculation, analogous to how `treasuryDonation` is included:

- In `dijkstraProducedValue` (top-level tx): add `inject (fold . unDirectDeposits $ txBody ^. directDepositsTxBodyL)` to the produced value alongside `treasuryDonationTxBodyL`.
- In `dijkstraSubTxProducedValue` (sub-tx): add `fold . unDirectDeposits $ txBody ^. directDepositsTxBodyL` to the injected coin sum alongside `treasuryDonationTxBodyL`.

This mirrors the pattern already used for `treasuryDonation` and ensures that any ADA credited to account balances via direct deposits must be funded by the transaction's UTxO inputs.

### Proof of Concept
1. Register a stake credential to create a valid account address `addr`.
2. Construct a Dijkstra transaction with:
   - One UTxO input worth `N` lovelace
   - One UTxO output worth `(N - fee)` lovelace (returning change to self)
   - `directDeposits = { addr: 1_000_000_000_000_000 }` (1 billion ADA)
   - Fee = minimum required
3. Submit the transaction.
4. The preservation-of-value check passes: `consumed = N = produced = (N - fee) + fee`.
5. The ENTITIES rule applies `directDeposits`, adding `1_000_000_000_000_000` lovelace to `addr`'s account balance.
6. 1 billion ADA has been created from nothing, with only a transaction fee paid.
