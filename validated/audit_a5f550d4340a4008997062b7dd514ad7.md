### Title
Blocked `protocolTreasury` Address Freezes All User Claims in `MerkleDistributor` - (File: `contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

---

### Summary

In `MerkleDistributor.claim()`, the fee transfer to `protocolTreasury` is executed unconditionally and atomically with the user's token transfer. If the distributed `token` is a blacklistable ERC20 (e.g., USDC, USDT) and `protocolTreasury` is added to that token's blacklist, every call to `claim()` reverts — permanently freezing all users' unclaimed yield until an admin intervenes.

---

### Finding Description

`MerkleDistributor.claim()` performs two sequential `safeTransfer` calls:

```solidity
// Line 141
IERC20(token).safeTransfer(account, amountToSend);

// Line 144 — unconditional, no fee > 0 guard
IERC20(token).safeTransfer(protocolTreasury, fee);
``` [1](#0-0) 

The second transfer to `protocolTreasury` is called **unconditionally** — even when `fee == 0` (i.e., when `feeInBPS == 0`). There is no `if (fee > 0)` guard and no error handling around this call.

`safeTransfer` propagates any revert from the underlying token contract. Tokens such as USDC and USDT implement address blacklists: any transfer to or from a blacklisted address reverts. If `protocolTreasury` is blacklisted by the token contract, `safeTransfer(protocolTreasury, fee)` reverts unconditionally, causing the entire `claim()` transaction to revert.

Because the `token` field is a generic, configurable ERC20 address: [2](#0-1) 

and can be set to any token via `setToken()`: [3](#0-2) 

the contract is fully exposed to this failure mode whenever a blacklistable token is used.

The analog in `LRTWithdrawalManager.instantWithdrawal()` is structurally identical: `_transferAsset(asset, feeRecipient, fee)` is called before the user's transfer, and `_transferAsset` uses `IERC20(asset).safeTransfer(to, amount)` for ERC20 assets. If `feeRecipient` is blacklisted and `fee > 0`, all instant withdrawals for that asset are blocked. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

When `protocolTreasury` is blacklisted by the distributed token:

- Every call to `claim()` reverts for every user, regardless of their individual claim validity.
- No user can receive their earned yield until an admin calls `setProtocolTreasury()` with a non-blacklisted address.
- The user's claim state is reverted along with the transaction (state update at lines 134–135 is not committed), so users can retry after the treasury is updated — but all yield is frozen during the window of inaction. [6](#0-5) 

---

### Likelihood Explanation

**Low-Medium.**

- USDC and USDT are the most widely distributed reward tokens in DeFi and both implement address blacklists enforced on every transfer, including zero-value transfers.
- The `token` field is configurable by the owner and can be set to any ERC20, including USDC.
- USDC blacklisting is a real, regularly exercised mechanism (OFAC sanctions compliance). A protocol treasury multisig address could be blacklisted due to regulatory action.
- No attacker action is required to trigger the freeze — the condition arises from the token contract's blacklist state, which is outside the protocol's control once set.
- The unconditional call (no `fee > 0` guard) means even a zero-fee configuration does not protect against this.

---

### Recommendation

1. Add a `fee > 0` guard before the treasury transfer to avoid unnecessary calls when fees are zero:

```solidity
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

2. Decouple fee accumulation from user claims. Instead of transferring fees atomically during `claim()`, accumulate them in a storage variable and expose a separate `sweepFees()` function callable by the owner. This ensures a failing fee transfer never blocks user claims.

3. Apply the same fix to `LRTWithdrawalManager.instantWithdrawal()`: accumulate fees internally and sweep them separately, rather than transferring to `feeRecipient` atomically within the user's withdrawal path.

---

### Proof of Concept

1. Deploy `MerkleDistributor` with `token = USDC` and `feeInBPS = 100` (1%).
2. Set `protocolTreasury` to address `T`.
3. USDC's compliance operator blacklists address `T` (e.g., OFAC sanction).
4. Any user calls `claim(index, account, cumulativeAmount, merkleProof)` with a valid proof.
5. Execution reaches line 144: `IERC20(token).safeTransfer(protocolTreasury, fee)`.
6. USDC's `transfer()` checks its blacklist, finds `T` is blacklisted, and reverts.
7. `safeTransfer` propagates the revert; the entire `claim()` transaction reverts.
8. The user receives nothing. The state update at lines 134–135 is also reverted.
9. Repeat for every user — all claims are blocked until admin calls `setProtocolTreasury()`.

Even with `feeInBPS = 0` (fee = 0), the call `safeTransfer(protocolTreasury, 0)` is still executed unconditionally at line 144, and USDC reverts on any transfer involving a blacklisted address regardless of amount. [7](#0-6)
