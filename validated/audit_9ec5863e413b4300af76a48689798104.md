### Title
AND-Node Additive Share Accumulation Uses Assignment Instead of Sum — (`File: src/cbmpc/protocol/ec_dkg.cpp`)

### Summary
`key_share_mp_t::reconstruct_additive_share` and `key_share_mp_t::reconstruct_pub_additive_shares` both handle `AND` nodes by **assigning** the last non-zero child's share to the output instead of **accumulating** (summing) all children's shares. This is the direct C++ analog of the GaugeController bug: a running total is computed from the most-recent delta rather than the full accumulated value.

### Finding Description

In `reconstruct_additive_share`, the `AND` node case reads:

```cpp
case node_e::AND:
  additive_share = 0;
  is_satisfied = true;
  for (int i = 0; i < n; i++) {
    bn_t additive_share_from_child;
    ...
    if (additive_share_from_child != 0) {
      additive_share = additive_share_from_child;   // ← assignment, not +=
    }
  }
  break;
``` [1](#0-0) 

The semantics of an AND node in the access structure is that the secret is split **additively** across all children: `x = x_child_1 + x_child_2 + … + x_child_n`. A party that appears in more than one child of an AND node (e.g., in two nested THRESHOLD sub-trees) holds a share from each child, and its total additive share must be their **sum**. The code instead silently discards every child's contribution except the last non-zero one.

The identical assignment-instead-of-accumulation bug exists in `reconstruct_pub_additive_shares` for the same `AND` case:

```cpp
if (!additive_share_from_child.is_infinity()) {
  pub_additive_shares = additive_share_from_child;  // ← assignment, not +=
}
``` [2](#0-1) 

Because both the scalar share and the EC-point share are truncated in the same way, the internal consistency check `x_share * G == Qis[party]` still passes — the wrong share is accepted silently and propagated into the signing path.

### Impact Explanation

`reconstruct_additive_share` is the sole implementation called by `to_additive_share`: [3](#0-2) 

`to_additive_share` is called directly from the three public signing APIs: [4](#0-3) 

For any access structure whose root (or any interior node) is an AND node with a party appearing in two or more children, `to_additive_share` silently returns a wrong private scalar and a wrong public-key-share map. The signing protocol then runs to completion with the wrong key material, producing an invalid signature — or, in protocols that do not verify the final signature internally, producing a structurally valid but cryptographically incorrect output that is accepted without error.

### Likelihood Explanation

AND nodes are a first-class, documented feature of the `ac_t` access structure. Any integrator who constructs an AND node whose children share at least one party (e.g., `AND(THRESHOLD(A,B,C), THRESHOLD(A,B,D))`) will silently receive wrong additive shares on every call to the ECDSA-MPC, EdDSA-MPC, or Schnorr-MPC sign API.

### Recommendation

Replace the assignment with modular accumulation in both functions:

```cpp
// reconstruct_additive_share — AND case
MODULO(q) additive_share += additive_share_from_child;

// reconstruct_pub_additive_shares — AND case
pub_additive_shares += additive_share_from_child;
```

This mirrors the correct pattern already used in `reconstruct_recursive` for AND nodes: [5](#0-4) 

### Proof of Concept

Consider an access structure `AND(THRESHOLD(2,[A,B,C]), THRESHOLD(2,[A,B,D]))`. Party A holds a share in both children. After DKG, call `to_additive_share` with a quorum that satisfies both children using A. The returned `x_share` equals only A's Lagrange-weighted share from the second THRESHOLD child; the contribution from the first child is silently dropped. Signing with this additive share produces a signature that fails standard ECDSA/EdDSA/Schnorr verification against the correct public key `Q`.

error_t key_share_mp_t::to_additive_share(const crypto::ss::ac_t ac, const std::set<crypto::pname_t>& quorum_names,
                                          key_share_mp_t& additive_share) {
  if (!ac.enough_for_quorum(quorum_names)) {
    return coinbase::error(E_INSUFFICIENT);
  }
  error_t rv = UNINITIALIZED_ERROR;
  const mod_t& q = curve.order();
  bn_t new_x_share;
  bool _ignore_is_satisfied = false;
  if (rv = reconstruct_additive_share(q, ac.root, quorum_names, new_x_share, _ignore_is_satisfied)) return rv;

  party_map_t<ecc_point_t> new_Qis;
  std::vector<crypto::pname_t> quorum_names_vec(quorum_names.begin(), quorum_names.end());

  for (size_t j = 0; j < quorum_names_vec.size(); j++) {
    crypto::vartime_scope_t vartime_scope;
    bool _ignore_is_satisfied = false;
    ecc_point_t new_Qi;
    if (rv = reconstruct_pub_additive_shares(ac.root, quorum_names, quorum_names_vec[j], new_Qi, _ignore_is_satisfied))
      return rv;
    new_Qis[quorum_names_vec[j]] = new_Qi;
  }

  additive_share.x_share = new_x_share;
  additive_share.Q = Q;
  additive_share.Qis = new_Qis;
  additive_share.curve = curve;
  additive_share.party_name = party_name;

  return SUCCESS;
