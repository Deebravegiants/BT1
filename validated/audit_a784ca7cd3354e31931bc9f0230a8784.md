### Title
`nativeFee` ETH Permanently Locked in EVM OmniBridge With No Withdrawal Path - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

Every call to `initTransfer` or `initTransfer1155` on the EVM `OmniBridge` contract requires the caller to include a `nativeFee` amount in `msg.value`. This ETH is retained by the contract and never forwarded anywhere, yet the contract contains no function to withdraw it. The ETH accumulates indefinitely with no recovery path.

---

### Finding Description

In `OmniBridge.initTransfer`, the `msg.value` is split into three parts: the bridged `amount` (for native ETH transfers), the Wormhole publishing fee (`extensionValue`), and the `nativeFee`. Only `extensionValue` is forwarded to the Wormhole relay via `initTransferExtension`:

```solidity
// OmniBridge.sol lines 386-413
if (tokenAddress == address(0)) {
    extensionValue = msg.value - amount - nativeFee;   // nativeFee stays
} else {
    extensionValue = msg.value - nativeFee;            // nativeFee stays
    ...
}
initTransferExtension(..., extensionValue);
```

In `OmniBridgeWormhole.initTransferExtension`, only `value` (i.e., `extensionValue`) is forwarded to Wormhole:

```solidity
// OmniBridgeWormhole.sol lines 143-144
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
```

The `nativeFee` portion of `msg.value` is silently retained by the contract. The contract has a bare `receive()` payable fallback but **no ETH withdrawal function of any kind**. The complete list of admin functions in `OmniBridge.sol` (`addCustomToken`, `removeCustomToken`, `acceptTokenOwnership`, `setMetadata`, `setNearBridgeDerivedAddress`, `pause`, `pauseAll`, `upgradeToken`) contains no mechanism to recover ETH.

The only path by which ETH ever leaves the contract is `finTransfer` when `payload.tokenAddress == address(0)` (native ETH bridge-back), which draws from the general contract balance — not from any tracked `nativeFee` pool. Accumulated `nativeFee` ETH is therefore permanently commingled and unrecoverable as protocol revenue.

---

### Impact Explanation

Every `initTransfer` or `initTransfer1155` call that includes a non-zero `nativeFee` permanently locks that ETH in the contract. Over the bridge's lifetime this constitutes an ever-growing pool of unclaimable protocol fee value. This matches the allowed impact: **"Irreversible fund lock … permanently unclaimable … protocol value in … fee … flows."**

---

### Likelihood Explanation

`nativeFee` is a standard parameter of the bridge's primary user-facing entry point. Any user bridging tokens from EVM to NEAR who pays a non-zero `nativeFee` contributes to the locked balance. This is a routine, high-frequency operation. No special attacker action is required; the lock occurs automatically on every such transfer.

---

### Recommendation

Add an admin-gated ETH withdrawal function to `OmniBridge.sol`, for example:

```solidity
function withdrawNativeFees(address payable recipient, uint256 amount)
    external onlyRole(DEFAULT_ADMIN_ROLE)
{
    (bool success, ) = recipient.call{value: amount}("");
    require(success, "ETH transfer failed");
}
```

Alternatively, track accumulated `nativeFee` in a dedicated storage variable and expose a targeted withdrawal for that amount only.

---

### Proof of Concept

1. User calls `OmniBridge.initTransfer(tokenAddress, 1e18, 0, 0.01e18, "near:recipient", "")` with `msg.value = 0.01e18` (ERC20 transfer, `nativeFee = 0.01 ETH`).
2. Inside `initTransfer`: `extensionValue = msg.value - nativeFee = 0`. The `nativeFee` (0.01 ETH) stays in the contract.
3. `initTransferExtension` is called with `value = 0`; Wormhole receives nothing from this call.
4. The 0.01 ETH is now in the contract. No admin function exists to retrieve it.
5. Repeat across thousands of transfers: the contract accumulates ETH with no recovery path.
