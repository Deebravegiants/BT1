### Title
Unauthenticated SQMR Peers Can Corrupt Canonical Block Body via Unverified `n_transactions` and Unvalidated Transaction Hashes — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

The p2p sync client accepts headers from unauthenticated SQMR peers with no cryptographic verification of the block hash or the `n_transactions` field. The transaction sync loop then uses `header.n_transactions` as the sole termination condition and accepts exactly that many peer-supplied transactions with no commitment verification. An attacker controlling both a header peer and a transaction peer can store an arbitrarily large set of fabricated transactions as the canonical block body for any synced block.

---

### Finding Description

**Step 1 — Forged header accepted without cryptographic verification.**

`HeaderStreamBuilder::parse_data_for_block` performs only two checks on a received `SignedBlockHeader`: [1](#0-0) 

- Block number matches the expected value.
- `signatures.len() == 1`.

There is no verification that the `block_hash` field is the correct Pedersen/Poseidon hash of the header fields, and no cryptographic verification that the signature is valid over that hash. The `n_transactions` field is accepted verbatim. The header is then written to storage as-is: [2](#0-1) 

**Step 2 — Transaction loop is bounded solely by the stored `n_transactions`.**

`TransactionStreamFactory::parse_data_for_block` reads `target_transaction_len` directly from the stored header: [3](#0-2) 

It then loops, collecting exactly `target_transaction_len` transactions from the peer: [4](#0-3) 

**Step 3 — No commitment verification on received transactions.**

Transaction hashes are pushed directly into `block_body.transaction_hashes` with an explicit TODO acknowledging the absence of validation: [5](#0-4) 

The `transaction_commitment` stored in the header is never compared against the Merkle root of the received transaction hashes. The `BlockBody` is written to storage unconditionally: [6](#0-5) 

---

### Impact Explanation

A syncing full node that processes this attack stores a `BlockBody` containing K attacker-chosen transactions (with attacker-chosen hashes, calldata, and outputs) as the canonical body for the targeted block. The `body_marker` advances past that block. Any subsequent RPC call that reads block transactions (`starknet_getBlockWithTxs`, `starknet_getTransactionByHash`, `starknet_getTransactionReceipt`, etc.) returns the fabricated data with full authority. This matches the **High** impact category: *RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

---

### Likelihood Explanation

Both the header peer and the transaction peer are unauthenticated SQMR connections — any network participant can act as either. The attacker needs to win the peer-selection race for both protocols for the target block range, which is feasible by operating multiple peers or by being the first to respond to a query. No privileged access is required.

---

### Recommendation

1. **Verify the block hash cryptographically** in `HeaderStreamBuilder::parse_data_for_block` before storing any header. Recompute the block hash from the received fields and reject headers where the computed hash does not match `signed_block_header.block_header.block_hash`.

2. **Verify `transaction_commitment`** in `TransactionStreamFactory::parse_data_for_block` after collecting all transactions. Recompute the Merkle commitment from the received transaction hashes and reject the peer if it diverges from `header.transaction_commitment`. The existing TODO at line 88 acknowledges this gap and must be resolved.

3. **Verify `n_transactions` against `concatenated_counts`** in the header path. The existing TODO in `convert_sync_block_to_block_data` at line 140–141 of `header.rs` also acknowledges this gap: [7](#0-6) 

---

### Proof of Concept

```
1. Attacker operates as an unauthenticated SQMR header peer.
2. When the syncing node queries for block N's header, attacker responds with:
     SignedBlockHeader {
       block_header: BlockHeader {
         block_number: N,
         n_transactions: 10_000,          // inflated
         transaction_commitment: <arbitrary felt>,
         block_hash: <arbitrary felt>,    // not verified
         ...
       },
       signatures: vec![BlockSignature { r: 1, s: 1 }],  // length==1, not verified
     }
3. Header passes both checks (block_number == N, signatures.len() == 1) and is stored.
4. Attacker operates as an unauthenticated SQMR transaction peer.
5. When the syncing node queries for block N's transactions, attacker responds with
   10,000 FullTransaction messages, each with arbitrary transaction data and hashes.
6. parse_data_for_block loops until current_transaction_len == 10_000 and returns
   Ok(Some((block_body, N))) with all 10,000 fabricated transactions.
7. write_to_storage calls append_body(N, block_body) and advances body_marker to N+1.
8. starknet_getBlockWithTxs(N) now returns 10,000 attacker-chosen transactions.
   The transaction_commitment in the stored header diverges from the actual Merkle
   root of those hashes, but this is never checked.
```
