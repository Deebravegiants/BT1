### Title
`compute_fee_target` consumes `get_strk_to_usd_rate` as USD-per-STRK but the trait documents it as STRK-per-USD — inverted oracle rate silently drives `fee_proposal_fri` to the wrong value - (File: `crates/apollo_consensus_orchestrator/src/dynamic_gas_price/mod.rs`)

---

### Summary

`compute_fee_target` divides by the oracle rate (`target * 10^18 / rate`), which is dimensionally correct only when `rate` is **USD per STRK** (e.g., 0.04 USD/STRK → `0.04 × 10^18`). The `L1GasPriceProviderClient::get_strk_to_usd_rate` trait documentation, however, explicitly states the oracle returns **STRK per USD** (e.g., "24.5 STRK per USD → `24.5 × 10^18`"). These are reciprocals. If the oracle endpoint is configured according to its own documentation, every `fee_proposal_fri` value embedded in a `ProposalCommitment` will be off by a factor of `(STRK_per_USD)²`, causing the SNIP-35 L2 gas price to converge to the wrong equilibrium and corrupting fee estimates returned by RPC.

---

### Finding Description

**The formula** in `compute_fee_target`:

```rust
// floor_fri = target_atto_usd_per_l2_gas * 10^18 / strk_usd_rate
let numerator = U256::from(target_atto_usd_per_l2_gas) * U256::from(FRI_DECIMALS_SCALE);
let floor = numerator / U256::from(strk_usd_rate);
```

Dimensional analysis shows this is correct **only** when `strk_usd_rate` = USD/STRK × 10^18:

```
(atto_USD/gas) × 10^18 / (USD/STRK × 10^18)
= (10^-18 USD/gas) × (STRK/USD)
= 10^-18 STRK/gas
= FRI/gas  ✓
```

The function's own test comment confirms this expectation:

```rust
// Target: $3e-9/gas = 3_000_000_000 atto-USD. STRK at $0.50 = 500_000_000_000_000_000.
// floor = 3_000_000_000 * 10^18 / 500_000_000_000_000_000 = 6_000_000_000.
```

"STRK at $0.50" = 0.5 USD per STRK → `rate = 0.5 × 10^18`. The formula is internally consistent with this interpretation.

**The trait documentation**, however, says the opposite:

```rust
/// STRK/USD rate as 18-decimal fixed-point integer (e.g. 24.5 STRK per USD → `24.5 * 10^18`).
async fn get_strk_to_usd_rate(&self, timestamp: u64) -> L1GasPriceProviderClientResult<u128>;
```

"24.5 STRK per USD" = STRK/USD (how many STRK you get for 1 USD). This is the **reciprocal** of what the formula needs.

This is consistent with the convention used by the sibling `get_rate` function:

```rust
/// ETH/FRI rate as 18-decimal fixed-point integer (e.g. 5000 STRK per ETH → `5000 * 10^18`).
async fn get_rate(&self, timestamp: u64) -> L1GasPriceProviderClientResult<u128>;
```

`get_rate` returns STRK per ETH (STRK/ETH), and `wei_to_fri` uses it as `wei × rate / 10^18`. An operator reading both docstrings would naturally configure `strk_to_usd_oracle_config` to return STRK per USD, following the same convention — which is exactly the wrong direction for `compute_fee_target`.

**The call chain** that reaches the wrong value:

1. `resolve_fee_target` calls `l1_gas_price_provider.get_strk_to_usd_rate(timestamp)` and passes the result directly to `compute_fee_target`
2. `compute_fee_target` divides by the rate (treating it as USD/STRK)
3. The result is passed to `compute_fee_proposal` and clamped to `fee_proposal_bounds(fee_actual, margin_ppt)`
4. The clamped value is stored as `fee_proposal_fri` in `ProposalInit`
5. `proposal_commitment_from` hashes `Poseidon(partial_block_hash, fee_proposal_fri)` into the `ProposalCommitment`

---

### Impact Explanation

If the oracle endpoint is configured to return STRK per USD (as documented), the fee target is:

```
computed = target_atto_usd / STRK_per_USD
correct  = target_atto_usd × STRK_per_USD
ratio    = 1 / (STRK_per_USD)²
```

For STRK at $0.04 (STRK_per_USD = 25): the fee target is **625× too low**. With `DEFAULT_SNIP35_TARGET_ATTO_USD_PER_L2_GAS = 880_000_000`:

- Correct target: `880_000_000 × 25 = 22 GFri`
- Wrong target: `880_000_000 / 25 = 35.2 MFri`

The `fee_proposal_bounds` clamp limits movement to `margin_ppt` per block, so the fee drifts slowly. Over `window_size` blocks the `fee_actual` (median of `fee_proposal` values) converges to the wrong equilibrium. Once converged:

- Every `fee_proposal_fri` committed into `ProposalCommitment` is wrong
- `starknet_estimateFee` / `starknet_simulateTransactions` return wrong L2 gas prices
- Actual transaction fees charged to users are wrong
- The `fee_proposal_fri` field is bound into the block commitment hash via `Poseidon(partial.0, fee_proposal_fri)`, so the committed state is permanently incorrect

---

### Likelihood Explanation

The `ExchangeRateOracleClient` is a generic HTTP client that parses a `price` field from a JSON response. The semantics of what the URL returns are entirely determined by the `strk_to_usd_oracle_config` URL. An operator configuring this URL would read the trait documentation ("24.5 STRK per USD") and configure an endpoint that returns STRK per USD — the wrong direction. The sibling `get_rate` uses the same "X per Y" convention, reinforcing the incorrect expectation. There is no runtime check that validates the direction of the rate.

---

### Recommendation

1. **Fix the trait documentation** to match the formula: change `"e.g. 24.5 STRK per USD → 24.5 * 10^18"` to `"e.g. 0.04 USD per STRK → 0.04 * 10^18"` (or equivalently `"e.g. STRK price in USD: if 1 STRK = $0.04, pass 40_000_000_000_000_000"`).
2. **Add a runtime sanity check**: on startup, verify that `strk_usd_rate` is in a plausible range for USD per STRK (e.g., between `10^14` and `10^21`), and log a prominent warning if it looks like STRK per USD.
3. **Add a dimensional-analysis comment** to `compute_fee_target` explicitly stating the required units: `strk_usd_rate` must be `(USD per STRK) × 10^18`.
4. **Rename** `get_strk_to_usd_rate` to `get_usd_per_strk_rate` to make the direction unambiguous, matching the formula's expectation.

pub fn compute_fee_target(
    target_atto_usd_per_l2_gas: u128,
    strk_usd_rate: u128,
) -> Option<GasPrice> {

        return None;

    Some(GasPrice(u128::try_from(floor).unwrap_or(u128::MAX)))
}
