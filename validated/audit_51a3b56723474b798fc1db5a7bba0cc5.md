Audit Report

## Title
ETH Withdrawal Permanently Frozen for Smart Contract Recipients Without `receive` Function — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager._transferAsset` pushes ETH to the recipient via `payable(to).call{value: amount}("")`, which reverts for any contract lacking a `receive`/`fallback` function. Because the rsETH burn occurs in a prior, separate `unlockQueue` transaction, the burn is never rolled back when `completeWithdrawal` reverts. The ETH is permanently locked in the contract with no admin recovery path.

## Finding Description
The withdrawal lifecycle is split across two independent transactions:

**Transaction 1 — `unlockQueue`:**
At `LRTWithdrawalManager.sol` L305–307, the operator burns the user's rsETH via `IRSETH.burnFrom` and calls `unstakingVault.redeem(asset, assetAmountUnlocked)`, which transfers ETH into `LRTWithdrawalManager`. After this transaction commits, the rsETH destruction is final and irreversible.

**Transaction 2 — `completeWithdrawal` / `completeWithdrawalForUser`:**
`_processWithdrawalCompletion` (L699–738) pops the nonce (L705), deletes the request (L712), decrements `unlockedWithdrawalsCount[asset]--` (L717), then calls `_transferAsset(asset, user, request.expectedAssetAmount)` (L734).

`_transferAsset` (L876–883) executes:
```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```
For a contract recipient without a `receive` function, `sent` is `false` and the function reverts. This reverts the entire Transaction 2, restoring the nonce, the request record, and the `unlockedWithdrawalsCount` — but **not** the rsETH burn from Transaction 1.

The withdrawal request is now permanently stuck: `completeWithdrawal` and `completeWithdrawalForUser` (L192–204, which calls the same internal function) will revert on every future attempt. `sweepRemainingAssets` (L395–414) is permanently blocked by the `hasUnlockedWithdrawals(asset)` guard at L403. A grep for `cancelWithdrawal`, `rescueETH`, `forceComplete`, `emergencyWithdraw`, and `recoverETH` returns no matches — there is no admin recovery path.

The developer comment at L191 ("Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH") acknowledges awareness of ETH transfer issues but incorrectly dismisses them, missing the permanent-freeze scenario.

## Impact Explanation
**Critical — Permanent freezing of funds.** The user's rsETH is irreversibly burned in Transaction 1. The corresponding ETH is permanently locked in `LRTWithdrawalManager` because every attempt to complete the withdrawal reverts, and no mechanism exists to cancel an already-unlocked withdrawal or recover the ETH while the unlocked withdrawal record persists. Both the rsETH and the ETH are lost to the user forever.

## Likelihood Explanation
**Medium.** Any smart contract that holds rsETH and calls `initiateWithdrawal` — a fully public, unprivileged function — can trigger this path. Common examples include DAO treasury contracts, yield aggregators, and any contract deployed without explicit ETH-acceptance logic. No attacker action is required; the victim contract simply needs to lack a `receive` function. The two-transaction split is what converts a recoverable revert into a permanent loss.

## Recommendation
Replace the push-payment model in `_processWithdrawalCompletion` with a pull-payment pattern: instead of calling `_transferAsset` directly, credit `request.expectedAssetAmount` to a per-user claimable balance mapping and emit an event. Add a separate `claimETH()` function that lets the user pull their ETH. This decouples accounting finalization from ETH delivery, so a failed transfer never leaves funds permanently frozen. Alternatively, before calling `_transferAsset` for ETH, check `to.code.length == 0` and revert with a descriptive error before any state is mutated if the recipient is a contract, preventing the two-transaction split from causing permanent loss.

## Proof of Concept
1. Deploy `VictimContract` (no `receive` function) holding rsETH.
2. `VictimContract` calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")` — rsETH transferred to withdrawal manager.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH burned (`burnFrom` at L305), ETH moved from `LRTUnstakingVault` into `LRTWithdrawalManager` (L307). Transaction commits; burn is final.
4. `VictimContract` calls `completeWithdrawal(ETH_TOKEN, "")` — `_transferAsset` at L878 executes `payable(VictimContract).call{value: amount}("")`, returns `(false, "")`, reverts with `EthTransferFailed()`. Transaction 2 is fully rolled back; rsETH burn is not.
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, address(VictimContract), "")` — identical revert path via L202.
6. `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist()` at L403.
7. `VictimContract`'s rsETH is permanently burned; ETH is permanently locked. No recovery path exists.

**Foundry test sketch:**
```solidity
function test_permanentFreeze() public {
    // Deploy victim with no receive()
    VictimContract victim = new VictimContract();
    deal(address(rsETH), address(victim), 1 ether);

    vm.prank(address(victim));
    rsETH.approve(address(withdrawalManager), 1 ether);
    vm.prank(address(victim));
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1 ether, "");

    // Operator unlocks — rsETH burned, ETH in manager
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH_TOKEN, ...);

    uint256 rsETHAfter = rsETH.balanceOf(address(victim));
    assertEq(rsETHAfter, 0); // burned

    // completeWithdrawal always reverts
    vm.prank(address(victim));
    vm.expectRevert(LRTWithdrawalManager.EthTransferFailed.selector);
    withdrawalManager.completeWithdrawal(ETH_TOKEN, "");

    // sweepRemainingAssets blocked
    vm.prank(manager);
    vm.expectRevert(LRTWithdrawalManager.PendingWithdrawalsExist.selector);
    withdrawalManager.sweepRemainingAssets(ETH_TOKEN);
}
```
