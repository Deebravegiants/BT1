### Title
`RequireGuard (KeyHashObj kh)` Native Script Satisfiable Without Key Authorization — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`, `Rules/Utxow.hs`)

---

### Summary

`evalDijkstraNativeScript` evaluates `RequireGuard cred` by checking only whether `cred` is a member of the tx body's `dtbGuards` OSet. For `KeyHashObj kh` credentials, no corresponding VKey witness is required. An unprivileged attacker can satisfy any `RequireGuard (KeyHashObj kh)` native script by simply listing `kh` in their own transaction's `dtbGuards` field, with no signature from the key holder.

---

### Finding Description

`evalDijkstraNativeScript` evaluates the `RequireGuard` branch as:

```haskell
RequireGuard cred -> cred `OSet.member` guards
``` [1](#0-0) 

The `guards` value is read directly from the attacker-controlled tx body field `dtbGuards`:

```haskell
validateDijkstraNativeScript tx =
  evalDijkstraNativeScript vhks (tx ^. bodyTxL . vldtTxBodyL) (tx ^. bodyTxL . guardsTxBodyL)
``` [2](#0-1) 

The Dijkstra UTXOW transition rule performs the following checks relevant to guards:

1. **VKey witness check** (`validateNeededWitnesses`) — uses `getWitsVKeyNeeded`, which does **not** include guard key hashes. [3](#0-2) 
2. **Script witness check** (`validateFailedBabbageScripts`) — validates scripts in `scriptsNeeded`, which includes `ScriptHashObj` guard credentials (they appear in `scriptsNeeded` via the guarding purpose), but **not** `KeyHashObj` guard credentials. [4](#0-3) 
3. **Sub-tx guard propagation check** — only verifies that guards required by sub-transactions are present in the top-level guards set; it does not validate that any guard credential has authorized anything. [5](#0-4) 

There is no rule anywhere in the UTXOW transition that requires a `KeyHashObj kh` appearing in `dtbGuards` to have a corresponding VKey witness in `addrTxWits`.

The asymmetry is stark: `RequireSignature kh` requires a valid signature from `kh` (enforced via `validateNeededWitnesses`), but `RequireGuard (KeyHashObj kh)` requires only that `kh` appears in the attacker-controlled `dtbGuards` field.

The `DijkstraNativeScriptRaw` type confirms `DijkstraRequireGuard` holds a `Credential Guard`, which can be either `KeyHashObj` or `ScriptHashObj`: [6](#0-5) 

---

### Impact Explanation

Any UTxO, minting policy, withdrawal, or certificate locked by a `DijkstraNativeScript` containing `RequireGuard (KeyHashObj kh)` — either directly or as part of a `RequireAllOf`/`RequireAnyOf`/`RequireMOf` composition — can be spent/executed by an attacker who simply includes `kh` in their transaction's `dtbGuards` field. No signature from the holder of `kh` is needed. This constitutes **direct, unauthorized theft of ADA and native assets** through an invalid ledger state transition.

---

### Likelihood Explanation

The exploit requires only constructing a standard Dijkstra-era transaction with a crafted `dtbGuards` field. No privileged access, governance majority, or leaked key is needed. The attacker only needs to know the `KeyHash` embedded in the target script (which is on-chain and publicly visible). The attack is deterministic and locally testable.

---

### Recommendation

For `KeyHashObj kh` credentials appearing in `dtbGuards`, the UTXOW rule must require that `kh` is present in `witsKeyHashes` (i.e., a valid VKey witness for `kh` over the tx body hash must be provided). Concretely, `getWitsVKeyNeeded` (or an equivalent Dijkstra-era override) should be extended to include all `KeyHashObj` credentials present in `dtbGuards`. This mirrors how `RequireSignature` is enforced and closes the gap between the two constructs.

---

### Proof of Concept

The existing test in the testlib directly demonstrates the vulnerability — it passes today:

```haskell
it "Spending inputs locked by script requiring a keyhash guard" $ do
  guardKeyHash <- KeyHashObj <$> freshKeyHash
  scriptHash <- impAddNativeScript (RequireGuard guardKeyHash)
  txIn <- produceScript scriptHash
  let tx = mkBasicTx (mkBasicTxBody & inputsTxBodyL .~ [txIn])
  -- Fails without guards (script not satisfied)
  submitFailingTx tx [...]
  -- Succeeds with guardKeyHash in dtbGuards — NO witness for guardKeyHash provided
  submitTx_ $ tx & bodyTxL . guardsTxBodyL .~ [guardKeyHash]
``` [7](#0-6) 

The `submitTx_` at line 38 succeeds without any VKey witness for `guardKeyHash`. An attacker who does not hold the private key for `guardKeyHash` can execute this exact transaction, spending the UTxO locked by `RequireGuard guardKeyHash` without authorization.
