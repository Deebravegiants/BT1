Audit Report

## Title
Unconditional `metricOmmSwapCallback` invocation with zero deltas causes `MetricOmmSimpleRouter` to always revert on no-op swaps - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool.swap()` unconditionally invokes `metricOmmSwapCallback` on `msg.sender` after every swap, including when `_executeSwap` returns `(0, 0, 0)`. `MetricOmmSimpleRouter.metricOmmSwapCallback()` reverts with `InvalidSwapDeltas` whenever `amount0Delta <= 0 && amount1Delta <= 0`, which is satisfied by `(0, 0)`. Any swap through the router that legitimately produces zero deltas — a documented, reachable state — permanently reverts, breaking the core swap path for that class of inputs.

## Finding Description
In `MetricOmmPool.swap()`, after `_executeSwap` returns, the callback is fired unconditionally in both branches: [1](#0-0) [2](#0-1) 

`_executeSwap` explicitly returns `(0, 0, 0)` when the price limit is immediately satisfied: [3](#0-2) 

When the pool calls `metricOmmSwapCallback(0, 0, data)` on the router, the guard at line 47 evaluates `0 <= 0 && 0 <= 0` as `true` and reverts: [4](#0-3) 

The contrast with `LiquidityLib.addLiquidity`, which guards its callback with `if (amount0Added > 0 || amount1Added > 0)`, confirms the pattern exists in the codebase but was not applied to the swap callback: [5](#0-4) 

## Impact Explanation
The core swap flow through `MetricOmmSimpleRouter` — the primary user-facing router — is broken for any input that produces zero deltas. This includes `exactInputSingle`, `exactOutputSingle`, and multi-hop `exactInput`/`exactOutput` paths where an intermediate hop hits the price limit immediately. This constitutes broken core pool functionality (unusable swap flow) matching the allowed impact gate.

## Likelihood Explanation
The zero-delta case is reachable by any unprivileged caller via a standard `exactInputSingle` call. It occurs when the caller supplies a `priceLimitX64` already satisfied by the current oracle price, or when the oracle price moves between transaction submission and execution. Both are normal live-trading conditions. No special privileges or malicious setup are required.

## Recommendation
**Option A (fix in pool):** Guard the callback, mirroring the `LiquidityLib` pattern:
```solidity
if (amount0Delta != 0 || amount1Delta != 0) {
    IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
}
```

**Option B (fix in router):** Change the guard to handle the zero-delta case gracefully:
```solidity
if (amount0Delta < 0 && amount1Delta < 0) revert InvalidSwapDeltas();
if (amount0Delta == 0 && amount1Delta == 0) return;
```

## Proof of Concept
1. Deploy pool with token0/token1 and add liquidity.
2. Deploy `MetricOmmSimpleRouter`.
3. Call `router.exactInputSingle(...)` with `zeroForOne=false` and `priceLimitX64=1` (immediately satisfied: `1 < initialPriceX64` for the `!zeroForOne` branch).
4. Pool calls `_executeSwap` → returns `(0, 0, 0)`.
5. Pool calls `metricOmmSwapCallback(0, 0, "")` on the router.
6. Router guard: `0 <= 0 && 0 <= 0` → `revert InvalidSwapDeltas()`.
7. Entire transaction reverts. [6](#0-5)
