Audit Report

## Title
Missing Chainlink `latestRoundData()` Staleness and Validity Checks Enable rsETH Over-Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and silently discards all validation return values (`updatedAt`, `answeredInRound`, price sign). The same repository already implements the correct validation pattern in `ChainlinkOracleForRSETHPoolCollateral`, confirming the protocol is aware of the requirement. The unvalidated price flows directly into rsETH minting at deposit time, allowing any depositor to over-mint rsETH against a stale inflated LST price, permanently diluting existing holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at L52 discards all validation fields:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository performs all three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unvalidated price propagates through two paths:

**Path 1 â€” rsETH minting per deposit:** `LRTDepositPool.getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(asset)` live at deposit time:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

**Path 2 â€” Protocol-wide rsETH price update:** `LRTOracle._getTotalEthInProtocol()` aggregates all asset values using the same unvalidated oracle: [4](#0-3) 

**Why existing guards are insufficient:** The `pricePercentageLimit` check in `_updateRsETHPrice()` only fires when the stored `rsETHPrice` is being updated. Individual deposits use the live `getAssetPrice()` return value directly against the stored `rsETHPrice`. Over-minting occurs continuously between price update calls, and the already-minted rsETH cannot be recalled. [5](#0-4) 

## Impact Explanation
**Critical â€” Protocol insolvency.**

When a Chainlink LST/ETH feed returns a stale inflated price (e.g., stETH/ETH has not updated after a slashing event within its 24-hour heartbeat window), `getAssetPrice(stETH)` returns the stale high value. Every depositor calling `depositAsset()` during this window receives more rsETH than the actual ETH value of their deposit. The total rsETH supply grows faster than the underlying ETH backing. The over-minted rsETH permanently dilutes all existing rsETH holders and cannot be recalled, constituting protocol insolvency. This matches the allowed Critical impact class.

## Likelihood Explanation
No attacker action is required. Any ordinary depositor transacting during a stale-price window triggers the over-minting. Chainlink heartbeat intervals for LST/ETH pairs are typically 24 hours. A slashing event, network congestion, or any significant LST price movement within the heartbeat window creates the condition. The scenario is realistic and has occurred on mainnet for other protocols. The SECURITY.md exclusion for "Incorrect data supplied by third-party oracles" does not apply here: the vulnerability is the contract's failure to consume the validation fields that Chainlink explicitly provides (`updatedAt`, `answeredInRound`), not the oracle providing wrong data. The protocol's own `ChainlinkOracleForRSETHPoolCollateral` demonstrates awareness of this requirement.

## Recommendation
Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0 || block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be configured per asset based on the Chainlink feed's documented heartbeat interval.

## Proof of Concept

1. Deploy a fork of mainnet with stETH/ETH Chainlink feed at price `1.05e18`. Warp time forward by 2 hours without triggering a feed update (simulating a slashing event that dropped true value to `1.00e18` but the feed has not updated within its 24-hour heartbeat).
2. Call `LRTDepositPool.depositAsset(stETH, 1e18, 0, "")` as an unprivileged depositor.
3. `getRsETHAmountToMint(stETH, 1e18)` â†’ `lrtOracle.getAssetPrice(stETH)` â†’ `ChainlinkPriceOracle.getAssetPrice(stETH)` â†’ returns stale `1.05e18`.
4. `rsethAmountToMint = (1e18 * 1.05e18) / rsETHPrice`. If `rsETHPrice = 1.05e18`, user receives `1e18` rsETH representing a `1.05 ETH` claim, while only `1.00 ETH` was deposited.
5. Repeat for multiple depositors. Each mints rsETH backed by `1.05 ETH` of claim while depositing `1.00 ETH` of actual value.
6. Call `updateRSETHPrice()`. The `pricePercentageLimit` guard may pause the protocol, but the over-minted rsETH already exists and cannot be recalled â€” existing holders are permanently diluted.

Foundry fork test: use `vm.mockCall` on the Chainlink feed to return a stale `updatedAt` timestamp and inflated price, then assert that `getRsETHAmountToMint` returns a value greater than the true ETH-equivalent amount, and that `rsETHPrice` after `updateRSETHPrice()` is lower than before the deposits.
