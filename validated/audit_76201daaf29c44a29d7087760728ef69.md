### Title
Unauthorized On-Behalf Invoke Transaction Bypasses Signature Validation at Gateway Admission — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function allows any unprivileged attacker to submit an invoke transaction with `sender_address = victim` and an invalid (or absent) signature, bypassing the gateway's `__validate__` entry-point check entirely. The gateway's only guard is `account_tx_in_pool_or_recent_block(tx.sender_address())`, which checks whether the victim's account has *any* transaction in the mempool — it does not verify that the current submitter is authorized to act on behalf of that account. The gateway therefore admits the attacker's transaction into the mempool without signature verification, a direct analog of the "on-behalf borrowing without authorization" bug class.

---

### Finding Description

`skip_stateful_validations` returns `true` (skip `__validate__`) when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
3. `mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`. [1](#0-0) 

Condition 3 is satisfied whenever the victim's address appears in the mempool or a recent committed block — it does **not** require the current submitter to be the account owner. An attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit an `invoke` transaction with `sender_address = victim`, `nonce = 1`, and an arbitrary invalid signature.

The gateway then calls `run_validate_entry_point` with `validate: false`: [2](#0-1) 

The account's `__validate__` entry point is never called, and the transaction is forwarded to the mempool without any signature check.

The analog to the external bug is exact:
- **External bug:** `borrow(_borrower=victim, _receiver=attacker)` — no check that `msg.sender == _borrower`.
- **Sequencer analog:** `invoke(sender_address=victim, nonce=1, signature=garbage)` — no check that the submitter owns the victim's account; the gateway skips `__validate__` solely because the victim has a tx in the mempool.

The `account_tx_in_pool_or_recent_block` implementation confirms it checks for *any* account transaction, not specifically a deploy_account owned by the submitter: [3](#0-2) 

---

### Impact Explanation

1. **Admission impact (High):** The gateway admits a transaction with an invalid signature into the mempool. This directly matches: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

2. **Nonce squatting / DoS:** The attacker's transaction occupies the victim's `nonce = 1` slot in the mempool. The victim's own legitimate `nonce = 1` transaction is either rejected as a duplicate nonce or must pay a higher fee to replace it via fee escalation.

3. **Fee drain:** When the batcher executes the block, it uses `AccountTransaction::new_for_sequencing`, which sets `validate: true`: [4](#0-3) 

The attacker's transaction then fails `__validate__` at execution time, reverts, but the victim's nonce is incremented and a fee is charged from the victim's balance (if the victim pre-funded their deterministic address before deployment, which is the standard Starknet UX pattern).

---

### Likelihood Explanation

- The attack requires only that the victim has submitted a `deploy_account` transaction (nonce=0) to the gateway, which is observable from the public mempool.
- No privileged access is needed; any user can submit transactions to the gateway endpoint.
- The victim's account address is deterministic and publicly derivable from the `deploy_account` transaction parameters (class hash, salt, constructor calldata).
- The `account_tx_in_pool_or_recent_block` check also returns `true` for accounts seen in recent committed blocks, widening the attack window beyond just the mempool. [3](#0-2) 

---

### Recommendation

1. **Verify deploy_account ownership:** When skipping `__validate__`, additionally verify that the incoming invoke transaction's `sender_address` matches the contract address computed from a `deploy_account` transaction that is *actually present in the mempool for that address* — not just any transaction from that address.

2. **Require co-submission:** Only skip validation when the `deploy_account` and `invoke` transactions are submitted in the same gateway request (atomically), so the gateway can confirm they originate from the same submitter.

3. **Alternatively, remove the skip entirely** and rely on the batcher's execution-time `__validate__` as the sole guard, accepting the UX trade-off of requiring the deploy_account to be confirmed before the first invoke can be submitted.

---

### Proof of Concept

```
1. Alice submits deploy_account tx (nonce=0) to the gateway.
   → Gateway validates and adds it to the mempool.
   → mempool.account_tx_in_pool_or_recent_block(alice_address) == true

2. Bob (attacker) submits:
   invoke_tx {
     sender_address: alice_address,
     nonce: 1,
     signature: [0x0, 0x0],   // invalid / garbage
     calldata: <arbitrary calldata>,
     resource_bounds: <valid bounds>,
   }

3. Gateway stateful validation path:
   - account_nonce = 0  (Alice not deployed yet)
   - tx.nonce() == Nonce(Felt::ONE)
   - account_tx_in_pool_or_recent_block(alice_address) == true
   → skip_stateful_validations returns true
   → run_validate_entry_point called with validate=false
   → __validate__ is NOT called
   → Bob's tx is accepted into the mempool

4. Alice's legitimate invoke tx (nonce=1) is now either:
   a. Rejected as duplicate nonce (if Bob's fee >= Alice's fee), or
   b. Must pay a higher fee to replace Bob's tx via fee escalation.

5. Batcher executes the block:
   - Alice's deploy_account tx (nonce=0) executes → Alice's account deployed.
   - Bob's invoke tx (nonce=1) executes via new_for_sequencing (validate=true):
     * perform_pre_validation_stage: nonce incremented to 2, fee bounds checked.
     * __validate__ called → fails (invalid signature) → tx reverts.
     * Alice's nonce is now 2; her balance is reduced by the revert fee.

6. Alice's legitimate nonce=1 tx is permanently blocked (nonce already consumed).
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
