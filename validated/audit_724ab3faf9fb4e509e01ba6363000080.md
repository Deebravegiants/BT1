Looking at the two key contracts:

### Title
Missing Zero-Address Guard on `eigenLayerRewardReceiver` in `processClaim` Causes Permanent Burning of EigenLayer Reward Yield - (File: contracts/NodeDelegator.sol)

---

### Summary

`LRTConfig.eigenLayerRewardReceiver` is never initialized in `initialize()` and defaults to `address(0)`. `NodeDelegator.processClaim` passes this value directly to `IRewardsCoordinator.processClaim` with no zero-address check. If `setEigenLayerRewardReceiver` is never called after deployment or upgrade, any operator-triggered claim will route all reward tokens to `address(0)`, permanently destroying them with no recovery path.

---

### Finding Description

`LRTConfig` declares `eigenLayerRewardReceiver` as a plain storage variable: [1](#0-0) 

It is never assigned in `initialize()`: [2](#0-1) 

The setter `setEigenLayerRewardReceiver` does enforce a non-zero check, but only at write time: [3](#0-2) 

`NodeDelegator.processClaim` reads the value and forwards it to EigenLayer with no guard: [4](#0-3) 

The `ILRTConfig` interface exposes `eigenLayerRewardReceiver()` as a plain view returning whatever is stored — zero if never set: [5](#0-4) 

Contrast this with all other critical addresses (e.g., `rewardsCoordinator`, `delegationManager`) which are stored in `contractMap` and retrieved via `getContract`, which internally calls `UtilLib.checkNonZeroAddress`: [6](#0-5) 

`eigenLayerRewardReceiver` bypasses this pattern entirely, leaving it silently zero until explicitly set.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

When `processClaim` is called with `eigenLayerRewardReceiver == address(0)`, one of two outcomes occurs depending on the EigenLayer `RewardsCoordinator` implementation:

1. **Tokens transferred to `address(0)`** — if the reward token does not revert on zero-address transfer (non-OZ-compliant tokens, native ETH wrappers, or custom reward tokens), all claimed tokens are burned with no recovery.
2. **Claim marked processed, then revert** — if the coordinator records the claim as consumed before executing the transfer, the Merkle leaf is spent and the claim can never be re-submitted, permanently losing the yield even after `setEigenLayerRewardReceiver` is later called.

Either path results in permanent loss of EigenLayer reward yield for the affected NDC.

---

### Likelihood Explanation

**Low-Medium.** The precondition is realistic:

- `eigenLayerRewardReceiver` is not set in `initialize()`, so any fresh deployment or proxy upgrade that does not include a `setEigenLayerRewardReceiver` call in the same transaction leaves the field at zero.
- The LRT operator (`onlyLRTOperator`) is a non-admin role that may be automated or scripted; it can call `processClaim` without awareness that the receiver was never configured.
- There is no on-chain check or event that would alert the operator to the misconfiguration before the irreversible call.

---

### Recommendation

Add a zero-address guard inside `processClaim` before forwarding to EigenLayer:

```solidity
function processClaim(IRewardsCoordinator.RewardsMerkleClaim calldata claim)
    external
    nonReentrant
    onlyLRTOperator
    whenNotPaused
{
    address receiver = lrtConfig.eigenLayerRewardReceiver();
    if (receiver == address(0)) revert EigenLayerRewardReceiverNotSet();
    IRewardsCoordinator(lrtConfig.rewardsCoordinator()).processClaim(claim, receiver);
}
```

Additionally, set `eigenLayerRewardReceiver` inside `initialize()` (or a dedicated `initialize2`-style reinitializer) so the invariant is enforced from deployment, consistent with how all other critical addresses are handled via `contractMap`.

// Pseudocode for a local fork/unit test
function test_processClaim_burnsTokensWhenReceiverIsZero() public {
    // 1. Deploy LRTConfig + NodeDelegator; do NOT call setEigenLayerRewardReceiver
    assertEq(lrtConfig.eigenLayerRewardReceiver(), address(0));

    // 2. Operator calls processClaim with a valid Merkle claim
    vm.prank(operator);
    nodeDelegator.processClaim(validClaim);

    // 3. Reward token balance of address(0) increased (tokens burned)
    assertGt(rewardToken.balanceOf(address(0)), 0);

    // 4. Attempt to re-claim the same leaf — reverts (already processed)
    vm.prank(operator);
    vm.expectRevert();
    nodeDelegator.processClaim(validClaim);
    // No recovery path exists.
}
