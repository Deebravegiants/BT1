### Title
Missing Zero-Value Guard on `keyDeposit` in `ppuWellFormed` Allows Governance Proposal to Eliminate Stake Credential Registration Cost — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/PParams.hs`)

---

### Summary

The `ppuWellFormed` function in both the Conway and Dijkstra eras validates that certain deposit-related protocol parameter updates are not set to zero. It correctly rejects proposals that set `poolDeposit`, `govActionDeposit`, or `dRepDeposit` to zero. However, it contains no equivalent guard for `keyDeposit` (the stake credential registration deposit). A governance `ParameterChange` proposal setting `keyDeposit = 0` passes the ledger's well-formedness check without triggering `MalformedProposal`, contrary to the design intent that all deposit parameters must remain positive.

---

### Finding Description

The `ppuWellFormed` method, defined in the `ConwayEraPParams` type class and implemented for both `ConwayEra` and `DijkstraEra`, is the ledger's gate for rejecting malformed `ParameterChange` governance actions before they enter the proposal queue. It is called from `actionWellFormed` in the `GOV` rule:

```haskell
actionWellFormed pv ga = failureUnless isWellFormed $ MalformedProposal ga
  where
    isWellFormed = case ga of
      ParameterChange _ ppd _ -> ppuWellFormed pv ppd
      _ -> True
```

The Conway implementation of `ppuWellFormed` checks three deposit parameters for non-zero:

```haskell
isValid (/= CompactCoin 0) ppuPoolDepositCompactL
isValid (/= CompactCoin 0) ppuGovActionDepositCompactL
isValid (/= CompactCoin 0) ppuDRepDepositCompactL
```

The `keyDeposit` parameter (`ppuKeyDepositL` / `ppuKeyDepositCompactL`) is **absent** from this list. A `ParameterChange` proposal containing `ppuKeyDeposit = SJust (Coin 0)` therefore passes `ppuWellFormed` and is accepted into the proposal queue as a valid governance action. The same omission is present in the Dijkstra era implementation.

The design intent is unambiguous: the Shelley delegation design specification states that the stake credential deposit "is to account for the costs of tracking the stake address and maintaining the corresponding reward account. It also incentivises de-registering stake addresses that are no longer required." The ADR on deposit tracking reinforces that these deposits are a critical economic mechanism. All other deposit parameters (`poolDeposit`, `govActionDeposit`, `dRepDeposit`) are protected by `ppuWellFormed`; `keyDeposit` is the sole exception.

If such a proposal is ratified and enacted, stake credential registration becomes free (beyond the transaction fee). Any party can then register an unbounded number of stake credentials at negligible cost, permanently bloating the `accounts` map in `DState`, removing the economic incentive to de-register unused credentials, and degrading ledger performance for all honest nodes.

---

### Impact Explanation

**Medium — Attacker-controlled governance proposal modifies deposits outside design parameters.**

Once `keyDeposit = 0` is enacted:

1. The `accounts` map in `DState` can be inflated without bound by submitting transactions containing `ConwayRegCert` or `ConwayRegDelegCert` certificates at zero deposit cost.
2. The economic incentive to de-register stake credentials (recovering the deposit) is eliminated, so bloat is permanent.
3. Reward distribution, stake snapshot computation, and ledger serialization all iterate over this map; unbounded growth degrades node performance across the network.
4. The deposit pot accounting is distorted: zero deposits are recorded per credential, so future attempts to raise `keyDeposit` back to a positive value would not recover any funds from existing registrations.

This matches the allowed Medium impact: *"Attacker-controlled... proposals... modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters."*

---

### Likelihood Explanation

A single unprivileged account holder can submit a `ParameterChange` governance action with `ppuKeyDeposit = SJust (Coin 0)` by paying the `govActionDeposit`. The proposal passes ledger validation without triggering `MalformedProposal`. Enactment requires DRep and Constitutional Committee ratification, which is a significant barrier in practice; however, the ledger-level defect (the missing guard) is the root cause, and the absence of the check means the proposal is never rejected at the point where the ledger is supposed to enforce it. The analogous checks for `poolDeposit`, `govActionDeposit`, and `dRepDeposit` demonstrate that the design intent is to block such proposals at submission time.

---

### Recommendation

Add a non-zero guard for `ppuKeyDepositCompactL` inside `ppuWellFormed` in both `ConwayEra` and `DijkstraEra`, consistent with the existing guards for the other deposit parameters:

```haskell
-- in ppuWellFormed, alongside the existing deposit checks:
, isValid (/= CompactCoin 0) ppuKeyDepositCompactL
```

This mirrors the treatment of `ppuPoolDepositCompactL`, `ppuGovActionDepositCompactL`, and `ppuDRepDepositCompactL`, and ensures that any governance proposal attempting to set `keyDeposit = 0` is rejected with `MalformedProposal` at submission time.
