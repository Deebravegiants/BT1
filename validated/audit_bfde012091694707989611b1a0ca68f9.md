### Title
Unbounded Iteration Over All Delegators During DRep Unregistration Exceeds Intended Validation Limits - (File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs)

---

### Summary

When a DRep unregisters via a `ConwayUnRegDRep` certificate, the GOVCERT transition rule iterates over every delegator stored in `drepDelegs` to clear their DRep delegation pointers. There is no protocol-enforced cap on the number of delegators per DRep. An attacker who controls a DRep can accumulate an unbounded number of delegators and then submit a single `ConwayUnRegDRep` certificate, forcing O(N) unbounded pure-Haskell computation during block validation. The transaction fee is calculated from transaction size and Plutus execution units only — it does not account for this ledger-rule work — allowing the attacker to impose arbitrarily expensive validation at near-zero marginal cost.

---

### Finding Description

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`, the `ConwayUnRegDRep` branch of the GOVCERT transition rule executes:

```haskell
let
  certState' =
    certState & certVStateL . vsDRepsL %~ Map.delete cred
  clearDRepDelegations delegs accountsMap =
    foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
pure $
  case mDRepState of
    Nothing -> certState'
    Just dRepState ->
      certState'
        & certDStateL . accountsL . accountsMapL
          %~ clearDRepDelegations (drepDelegs dRepState)
```

`clearDRepDelegations` calls `foldr (Map.adjust …) accountsMap delegs`, iterating over every element of `drepDelegs dRepState`. Each `Map.adjust` is O(log M) where M is the total number of registered accounts. Total work is therefore **O(N × log M)**, where N is the number of delegators to the unregistering DRep.

The `DRepState` type in `libs/cardano-ledger-core/src/Cardano/Ledger/DRep.hs` defines:

```haskell
data DRepState = DRepState
  { drepExpiry  :: !EpochNo
  , drepAnchor  :: !(StrictMaybe Anchor)
  , drepDeposit :: !(CompactForm Coin)
  , drepDelegs  :: !(Set (Credential Staking))   -- ← unbounded
  }
```

There is no field, protocol parameter, or ledger rule that caps the cardinality of `drepDelegs`. Delegators are added to this set in `processDelegationInternal` via `Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred` with no size check.

The `ConwayUnRegDRep` certificate itself is tiny (a 28–32-byte credential plus a `Coin`), so the `maxTxSize` protocol parameter does **not** bound the number of delegators. The delegator count lives entirely in the ledger state, not in the transaction.

The fee for the transaction is computed by `getConwayMinFeeTx` / `alonzoMinFeeTx`, which depends on transaction byte-size and Plutus `ExUnits`. Neither term reflects the O(N × log M) pure-Haskell work performed by `clearDRepDelegations`.

---

### Impact Explanation

**Impact: Medium — Attacker-controlled certificate exceeds intended validation limits.**

A single `ConwayUnRegDRep` certificate can force every validating node to perform O(N × log M) map operations, where N is unbounded. With N in the millions (achievable at ~2 ADA per delegator, all deposits recoverable), block validation for the containing block becomes extremely expensive. The attacker pays only the fixed transaction fee regardless of N. This exceeds the intended validation-cost model, which assumes per-transaction work is bounded by transaction size and declared Plutus budgets.

---

### Likelihood Explanation

The attack is straightforward and economically viable:

1. Register a DRep credential (DRep deposit, currently 500 ADA — **fully returned** on unregistration).
2. Register N stake credentials and delegate each to the DRep (key deposit ~2 ADA each — **recoverable** by later unregistering the stake keys).
3. Submit a single transaction containing `ConwayUnRegDRep`.

The DRep deposit is returned in the same transaction. Stake-key deposits are recoverable in subsequent transactions. The net unrecoverable cost is only transaction fees. No privileged access, governance majority, or external dependency is required. The attacker controls both the DRep and the delegating accounts.

---

### Recommendation

1. **Enforce a maximum delegator count per DRep** as a protocol parameter (e.g., `maxDRepDelegators`), checked at delegation time in `processDelegationInternal`.
2. Alternatively, **redesign DRep unregistration** to not require clearing all delegator pointers in a single transaction — for example, require delegators to explicitly re-delegate or undelegate before the DRep can unregister (analogous to how pool retirement works), or process the cleanup lazily at epoch boundaries.
3. **Charge fees proportional to the number of delegators** cleared, similar to how reference-script fees were introduced in Conway (ADR-009) to account for previously unpriced deserialization work.

---

### Proof of Concept

```
1. Register DRep credential D (deposit = 500 ADA, returned on step 4).
2. For i in 1..N:
     a. Register stake credential S_i (deposit = 2 ADA).
     b. Submit DelegTxCert S_i (DelegVote (DRepCredential D)).
        → drepDelegs D grows by 1 each time; no size check is performed.
3. After N delegations, drepDelegs D = {S_1, …, S_N}.
4. Submit a transaction containing ConwayUnRegDRep D (deposit = 500 ADA).
   → GOVCERT rule executes:
        clearDRepDelegations {S_1,…,S_N} accountsMap
      = foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing))
              accountsMap
              {S_1,…,S_N}
   → Performs N × O(log M) Map.adjust calls during block validation.
   → Transaction fee is fixed (based on ~200-byte tx size), independent of N.
5. Recover stake-key deposits by submitting UnRegDepositTxCert S_i for each i.
```

With N = 1,000,000 and M = 5,000,000 (realistic mainnet scale), step 4 requires ~23 million map operations in a single block-validation pass, with no corresponding fee charged to the submitter. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
