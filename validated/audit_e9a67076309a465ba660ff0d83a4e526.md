### Title
Node Ignores Per-Domain `reconstruction_threshold`, Using Global Governance Threshold for All Signing Operations — (`File: crates/node/src/providers/ecdsa.rs`, `crates/node/src/providers/robust_ecdsa/presign.rs`)

---

### Summary

The NEAR MPC contract stores a per-domain `reconstruction_threshold` in every `DomainConfig` and enforces it at the contract level. The MPC node, however, ignores this per-domain value entirely and uses the single global governance threshold (`mpc_config.participants.threshold`) for triple generation, presignature generation, and signing across all domains. This is the direct analog of the Sablier bug: a global/default value is used where a per-entity custom value should be consulted. Any domain whose `reconstruction_threshold` differs from the governance threshold will have its signing pipeline permanently broken — all sign requests to that domain will time out and fail.

---

### Finding Description

The contract's `DomainConfig` carries a per-domain `reconstruction_threshold`:

```rust
// crates/contract/src/primitives/domain.rs
pub struct DomainConfig {
    pub id: DomainId,
    pub protocol: Protocol,
    pub reconstruction_threshold: ReconstructionThreshold,  // per-domain
    pub purpose: DomainPurpose,
}
```

The contract validates and enforces this value at every mutation point (`vote_add_domains`, `vote_new_parameters`), and the invariant `governance_threshold >= max(reconstruction_threshold)` is enforced, meaning domains can legitimately be added with a reconstruction threshold strictly lower than the governance threshold.

The node's `EcdsaSignatureProvider::new` ignores all per-domain thresholds and creates a single triple store keyed on the global governance threshold:

```rust
// crates/node/src/providers/ecdsa.rs:69-84
// The set of distinct `t`s the node needs to serve is fully known at
// startup. Today every ECDSA domain shares the network-wide threshold;
// once #3164 lands and each domain may declare its own
// `reconstruction_threshold`, derive this set from the keyshares'
// domain configs instead.
let network_threshold = ReconstructionThreshold::new(mpc_config.participants.threshold);
triple_stores.insert(network_threshold, Arc::new(TripleStorage::new(..., network_threshold)?));
```

The same global threshold is used in background triple generation:

```rust
// crates/node/src/providers/ecdsa.rs:273-276
let threshold = ReconstructionThreshold::new(self.mpc_config.participants.threshold);
let triple_store = self.triple_store_for_t(threshold)?;
```

And in the signing path (both leader and follower):

```rust
// crates/node/src/providers/ecdsa/sign.rs:34-35
let threshold: usize = self.mpc_config.participants.threshold.try_into()?;
let threshold = ReconstructionThreshold::from(threshold);
```

For the `DamgardEtAl` (Robust ECDSA) provider, the same pattern appears in `compute_thresholds`, which the code itself annotates with a TODO:

```rust
    ...

```

And in the signing path for Robust ECDSA:

```rust
// crates/node/src/providers/robust_ecdsa/sign.rs:41-43
let threshold = self.mpc_config.participants.threshold.try_into()?;
let robust_ecdsa_threshold = translate_threshold(threshold, number_of_participants)?;
```

The key generation path, by contrast, correctly receives the per-domain `reconstruction_threshold` from the key event (passed through the coordinator). This means the keyshare is generated with `t_domain`, but all subsequent presignature and signing operations use `t_global`. When `t_domain != t_global`, the threshold embedded in the keyshare does not match the threshold used for presignatures, causing the signing protocol to fail.

---

### Impact Explanation

**Medium.** When a domain is added via `vote_add_domains` with a `reconstruction_threshold` lower than the governance threshold (a valid, contract-enforced governance action), the node generates triples and presignatures using the wrong (higher) threshold. The CaitSith presignature protocol requires the threshold to match the keyshare's threshold; a mismatch causes presignature generation to fail. For DamgardEtAl, the `MaxMalicious` parameter is derived from the wrong threshold, causing the presigning protocol to require a different participant count than the keyshare supports. In both cases, every sign request submitted to such a domain will permanently fail to produce a signature and will eventually time out via `fail_on_timeout`. This breaks the request-lifecycle safety invariant: a valid sign request to a valid, live domain is never fulfilled. The contract's per-domain threshold feature is rendered inoperative.

---

### Likelihood Explanation

The contract already supports and validates per-domain `reconstruction_threshold` values. Any threshold of participants can legitimately vote to add a domain with a lower reconstruction threshold (e.g., a domain intended for lower-security or higher-availability use cases). Once such a domain exists, every unprivileged user who submits a sign request to it triggers the failure. The code's own TODO comment at `crates/node/src/providers/robust_ecdsa/presign.rs:187` confirms this is a known gap between the contract's capability and the node's implementation.

---

### Recommendation

In `EcdsaSignatureProvider::new`, derive the set of triple stores from the per-domain `reconstruction_threshold` values in the keyshares' domain configs rather than from the single global threshold. In `spawn_background_tasks`, source `t` from `domain.reconstruction_threshold`. In `make_signature_leader_given_parameters` and `make_signature_follower_given_request`, look up the domain's `reconstruction_threshold` from the domain registry rather than reading `mpc_config.participants.threshold`. For `RobustEcdsaSignatureProvider`, pass the domain-specific threshold to `compute_thresholds` instead of the governance threshold. This is exactly the fix described in the codebase's own design document (`docs/design/domain-separation.md`, PR 7).

---

### Proof of Concept

1. Governance participants vote to add a new CaitSith domain with `reconstruction_threshold = 2` while the governance threshold is `3` (valid per contract invariant `governance_threshold >= max(reconstruction_threshold)`).
2. Key generation runs correctly with `t = 2` (the coordinator passes the domain's `reconstruction_threshold` to `run_key_generation_client`).
3. The node's `EcdsaSignatureProvider::new` creates a single triple store keyed on `t = 3` (governance threshold). [1](#0-0) 
4. Background triple generation produces triples with `t = 3`. [2](#0-1) 
5. An unprivileged user submits a sign request to the new domain.
6. The node attempts presignature generation using `t = 3` triples against a keyshare generated with `t = 2`. The CaitSith presign protocol enforces that the threshold matches the keyshare; the mismatch causes the presignature to fail. [3](#0-2) 
7. The sign request times out. The contract's `fail_on_timeout` fires, returning an error. The user's request is permanently unresolvable.
8. The same failure occurs for every subsequent sign request to this domain, rendering the domain permanently non-functional despite being valid on-chain.

The root cause is the direct analog of the Sablier bug: `mpc_config.participants.threshold` (global) is used where `domain.reconstruction_threshold` (per-domain) should be consulted. [4](#0-3) [5](#0-4)
