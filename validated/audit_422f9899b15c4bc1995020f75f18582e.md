### Title
Forged Foreign-Chain Verification via Unconstrained `payload_hash` in `respond_verify_foreign_tx` - (File: crates/contract/src/lib.rs)

### Summary
`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key. It does **not** verify that `response.payload_hash` is the canonical hash of `ForeignTxSignPayload{request.request, observed_values}`. A single malicious MPC leader node (below threshold) can replay a legitimately-obtained MPC signature from a previous session as the response to a different pending request, delivering a forged foreign-chain attestation to the caller.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract verifies `sig` over `payload_hash` using the root key, then resolves all pending yields for `request` with the full `response`: [2](#0-1) 

**What is missing:** the contract never verifies that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload{request.request, observed_values}))`. The `ForeignTxSignPayload` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

and `msg_hash = SHA-256(borsh(ForeignTxSignPayload))` per the design doc. [3](#0-2) 

Contrast this with the regular `respond` function, where the payload is part of the request itself and the contract directly verifies the signature against `request.payload`: [4](#0-3) 

For `respond_verify_foreign_tx`, the payload is computed off-chain by MPC nodes from foreign-chain data. The contract cannot independently reconstruct it, so it trusts whatever `payload_hash` the leader submits — as long as the signature is valid.

The `VerifyForeignTransactionResponse` returned to callers contains only `payload_hash` and `signature`, with no `observed_values`: [5](#0-4) 

This means callers cannot reconstruct the expected `payload_hash` from the response alone without independently querying the foreign chain.

### Impact Explanation

A malicious MPC leader node (single node, strictly below threshold) can:

1. Participate legitimately in signing session for **Request A** (e.g., Bitcoin `tx_id=X`) → obtains valid `(payload_hash_A, sig_A)`.
2. Wait for **Request B** (e.g., Bitcoin `tx_id=Y`) to appear as a pending request in `pending_verify_foreign_tx_requests`.
3. Call `respond_verify_foreign_tx(request=B, response={payload_hash_A, sig_A})`.
4. The contract checks: is `sig_A` valid over `payload_hash_A` under the root key? **YES** — it passes.
5. `resolve_yields_for` drains all yields queued under Request B, delivering `{payload_hash_A, sig_A}` to every caller of Request B.

The callers of Request B receive a `VerifyForeignTransactionResponse` where `payload_hash` corresponds to `tx_id=X`, not their `tx_id=Y`. A bridge contract that checks only "did I receive a valid MPC signature?" — without independently querying the foreign chain to reconstruct the expected `payload_hash` — would accept this as a valid attestation that `tx_id=Y` was verified, enabling invalid bridge execution (e.g., releasing funds on NEAR for a Bitcoin transaction that was never actually verified). [6](#0-5) 

### Likelihood Explanation

- Requires **one** malicious MPC leader node, strictly below the signing threshold. This is an explicitly modeled adversary in the NEAR MPC threat model.
- The leader needs only to save a `(payload_hash, signature)` pair from any prior legitimate signing session — trivially achievable in any production environment processing multiple requests.
- No cryptographic forgery is required; the attack is a pure application-layer replay.
- The `pending_verify_foreign_tx_requests` map is keyed by `VerifyForeignTransactionRequest` (not caller-specific), so any pending request from any caller is a valid target. [7](#0-6) 

### Recommendation

The contract must bind `response.payload_hash` to the `request` contents before accepting the response. Since the contract cannot independently compute the observed values, the recommended fix is to include the full `ForeignTxSignPayload` in the response and verify on-chain:

1. `payload_hash == SHA-256(borsh(payload))`
2. `payload.request == request.request`
3. `signature` is valid over `payload_hash`

If including the full payload exceeds NEAR's promise data limits, an alternative is to include a commitment to `request.request` inside the signed payload at the node level, and verify that commitment on-chain using only the `request` parameter already available to the contract.

### Proof of Concept

```
// Setup: two pending requests exist
// Request A: Bitcoin tx_id = [0xAA; 32], already processed
// Request B: Bitcoin tx_id = [0xBB; 32], pending

// Step 1: Leader participated in legitimate signing of Request A
// Obtained: payload_hash_A = SHA-256(borsh({tx_id=[0xAA;32], block_hash=H1}))
//           sig_A = MPC_sign(payload_hash_A)  [valid under root key]

// Step 2: Request B is pending in pending_verify_foreign_tx_requests
// request_B = VerifyForeignTransactionRequest { request: Bitcoin { tx_id=[0xBB;32], ... }, ... }

// Step 3: Malicious leader calls:
respond_verify_foreign_tx(
    request = request_B,
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,   // hash of tx_id=X, NOT tx_id=Y
        signature: sig_A,               // valid MPC sig over payload_hash_A
    }
)

// Step 4: Contract checks verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → OK
// Step 5: resolve_yields_for(request_B, response) → callers of request_B receive
//         {payload_hash_A, sig_A} — a forged attestation for tx_id=[0xBB;32]
```

The contract accepts the call at line 729–734 and resolves all yields at line 749–753, delivering the forged response to every caller waiting on Request B. [8](#0-7)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
