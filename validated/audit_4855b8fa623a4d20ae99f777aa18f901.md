### Title
Deposits Paid for Non-Winning `propose_update` Proposals Are Permanently Locked When `do_update` Clears All Entries — (`crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

---

### Summary

`propose_update` requires a NEAR token deposit proportional to the size of the proposed update (contract binary or config). When any update reaches threshold votes and `do_update` is called, it unconditionally clears **all** pending proposals and votes — but never refunds the deposits paid by the proposers of the non-winning entries. Those NEAR tokens are permanently locked in the contract with no recovery path.

---

### Finding Description

**Step 1 — Deposit is collected and kept on proposal.**

`propose_update` in `lib.rs` computes a required deposit from `bytes_used` (which includes the full contract binary size plus a 128-voter overhead), keeps exactly that amount, and only refunds the excess:

```rust
let required = ProposedUpdates::required_deposit(&update);
// ...
let id = self.proposed_updates.propose(update);
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

The `required` deposit for a 1 MB contract binary is ≈ 10+ NEAR (storage byte cost × bytes). [1](#0-0) 

**Step 2 — `do_update` clears every other proposal without any refund.**

Inside `ProposedUpdates::do_update`, after removing the winning entry, the entire `entries` map and all votes are wiped:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ...
}
```

No proposer address is stored per entry, so there is no way to issue refunds here. [2](#0-1) 

**Step 3 — No recovery mechanism exists.**

The `UpdateEntry` struct stores only `update` and `bytes_used`; it does not record the proposer's account ID:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
```

There is no `recover_token`, `withdraw_deposit`, or equivalent function anywhere in the contract. [3](#0-2) 

**Concrete scenario:**

1. Participant A proposes a 1 MB contract upgrade → pays ≈ 10 NEAR deposit.
2. Participant B proposes a config update → pays a smaller deposit.
3. Threshold participants vote for B's config update → `vote_update` calls `do_update`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry (and deposit) is gone.
5. A's ≈ 10 NEAR is permanently locked in the contract. No refund is ever issued.

This is the direct analog of the SpiceAuction bug: a cumulative accounting variable (`_totalAuctionTokenAllocation`) is not decremented when an allocation is deleted, causing tokens to be permanently locked. Here, the "accounting variable" is the contract's NEAR balance, which grows by each deposit but is never decremented when proposals are cleared.

---

### Impact Explanation

Every time a contract update reaches threshold and `do_update` fires, **all** competing proposals are silently discarded and their deposits are irrecoverably locked inside the contract. For a contract binary update (which can be hundreds of kilobytes to megabytes), the locked amount per proposal can be tens of NEAR. Over the protocol's lifetime, repeated update cycles accumulate permanently frozen funds with no administrative escape hatch.

This matches the allowed impact: **permanent freezing of funds controlled by the MPC network contract**.

---

### Likelihood Explanation

The MPC contract is a live, governance-driven system. Participants routinely propose and vote on contract upgrades and config changes. It is realistic — and expected — that multiple proposals coexist before one reaches threshold. Every such event silently destroys the deposits of the non-winning proposers. The likelihood is **high** given the protocol's normal operational cadence.

---

### Recommendation

Store the proposer's `AccountId` inside `UpdateEntry` at proposal time, and issue refunds for all cleared entries inside `do_update`:

```diff
pub(crate) struct UpdateEntry {
+   pub(super) proposer: AccountId,
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}

pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;

+   // Refund deposits for all non-winning proposals before clearing.
+   for (_, cleared_entry) in self.entries.iter() {
+       let refund = required_deposit(cleared_entry.bytes_used);
+       Promise::new(cleared_entry.proposer.clone()).transfer(refund).detach();
+   }

    self.entries.clear();
    self.vote_by_participant.clear();
    // ... rest unchanged
}
```

Also update `propose_update` in `lib.rs` to pass the proposer into `ProposedUpdates::propose`.

---

### Proof of Concept

1. Deploy the contract in Running state with 3 participants (threshold 2).
2. Participant A calls `propose_update` with a 500 KB contract binary, attaching the required deposit (~5 NEAR). Entry ID = 0 is created.
3. Participant B calls `propose_update` with a config update, attaching its required deposit. Entry ID = 1 is created.
4. Participants B and C call `vote_update(1)` — threshold reached, `do_update` fires.
5. `do_update` removes entry 1, then calls `self.entries.clear()` — entry 0 (A's proposal) is deleted.
6. Query the contract's NEAR balance: it has grown by A's full deposit with no corresponding refund promise ever issued.
7. A has no function to call to recover their deposit. The funds are permanently locked. [2](#0-1) [4](#0-3) [5](#0-4)

```rust
    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
```

```rust
fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
```
