### Title
FOT/Rebasing Token Deposit Mints Excess Wrapped Tokens, Causing Withdrawal Shortfall for Last Redeemers — (File: `contracts/agETH/AGETHTokenWrapper.sol`)

---

### Summary

`AGETHTokenWrapper._deposit` mints wrapped agETH 1:1 against the caller-supplied `_amount` rather than the actual tokens received. If the underlying altAgETH token is fee-on-transfer (FOT) or negatively rebasing, the contract holds fewer underlying tokens than the outstanding wrapped supply, making it impossible for the last redeemers to withdraw their full entitlement.

---

### Finding Description

`_deposit` executes a `safeTransferFrom` for `_amount` and immediately mints `_amount` wrapped tokens: [1](#0-0) 

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);          // mints _amount regardless of actual receipt
    emit Deposit(_asset, _to, _amount);
}
```

If the altAgETH token deducts a transfer fee, the contract receives `_amount − fee` but the user holds `_amount` wrapped tokens. The invariant `balanceOf(underlying) == totalSupply(wrapper)` is broken from the first deposit.

`_withdraw` burns `_amount` wrapped tokens and transfers exactly `_amount` underlying tokens: [2](#0-1) 

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);   // reverts if balance < _amount
    emit Withdraw(_asset, _to, _amount);
}
```

Because the contract holds less underlying than the wrapped supply, the last withdrawer's `safeTransfer` reverts (or, for a negatively rebasing token, the shortfall accumulates silently until the final redemption attempt fails). There is no `skim` or recovery function to reconcile the gap.

The same structural flaw exists in the public `depositTo` path: [3](#0-2) 

---

### Impact Explanation

**Medium — Temporary freezing of funds.**  
The last one or more users to call `withdraw`/`withdrawTo` will have their transactions revert because the contract's underlying balance is exhausted before all wrapped tokens are redeemed. Their wrapped agETH tokens are burned-eligible but the underlying cannot be returned, effectively freezing those funds until the protocol manually tops up the contract. Positive rebasing also silently locks excess tokens with no recovery path.

---

### Likelihood Explanation

**Low-Medium.**  
The allowed token set is fixed at initialization to a single altAgETH bridge token: [4](#0-3) 

There is no `addAllowedToken` function — only `removeAllowedToken`. If the bridge-wrapped altAgETH token is a standard non-rebasing ERC20, the bug is dormant. However, agETH is a yield-bearing LRT whose bridge representations on some L2s may carry rebasing or share-price-adjustment mechanics. Any future governance decision to allow a rebasing variant immediately activates the vulnerability for all depositors.

---

### Recommendation

Use a balance-before/after pattern to determine the actual amount received and mint only that amount:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    uint256 before = ERC20Upgradeable(_asset).balanceOf(address(this));
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    uint256 received = ERC20Upgradeable(_asset).balanceOf(address(this)) - before;
    _mint(_to, received);
    emit Deposit(_asset, _to, received);
}
```

Alternatively, add an explicit allowlist check that rejects FOT/rebasing tokens, and add a `skim` function to recover excess tokens from positive rebases.

---

### Proof of Concept

1. altAgETH token charges a 1% transfer fee.
2. Alice calls `deposit(altAgETH, 100e18)`.
   - Contract receives `99e18` underlying, but mints `100e18` wrapped agETH to Alice.
3. Bob calls `deposit(altAgETH, 100e18)`.
   - Contract receives `99e18` underlying (total `198e18`), mints `100e18` to Bob (total supply `200e18`).
4. Alice calls `withdraw(altAgETH, 100e18)` — succeeds, contract sends `100e18`, balance drops to `98e18`.
5. Bob calls `withdraw(altAgETH, 100e18)` — **reverts**: contract only holds `98e18` but must transfer `100e18`.
6. Bob's `100e18` wrapped tokens are permanently stuck; the `2e18` shortfall (fees paid on both deposits) is unrecoverable without an external top-up.
