### Title
Multiple Subtransactions Can Claim the Same Deposit Refund in Dijkstra Era — (`eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

### Summary

The Dijkstra era introduces subtransactions (`dtbSubTransactions`) embedded inside a top-level transaction. The consumed-value aggregation for a top-level Dijkstra transaction sums the consumed values of every subtransaction independently, using the same pre-transaction `lookupStakingDeposit` closure for each. No cross-subtransaction deduplication of deposit refunds is enforced. A developer-acknowledged pending test (`xit "Multiple subtransactions cannot get the same refund"`) confirms that the predicate failure that should reject such a transaction has not yet been implemented, meaning the ledger currently accepts a transaction in which two or more subtransactions each claim the full refund for the same staking credential.

### Finding Description

`getConsumedDijkstraValue` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` aggregates consumed value as:

```haskell
txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
```

where `subTransactionsConsumedValue` folds over every subtransaction and calls `getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo` for each independently. [1](#0-0) 

The `lookupStakingDeposit` closure is constructed once from the ledger state **before** any subtransaction is applied. If subtransaction `subTx1` contains `UnRegDepositTxCert stakingCred keyDeposit` and subtransaction `subTx2` contains the same certificate for the same credential, both calls to `lookupStakingDeposit stakingCred` return the same deposit amount, so the total consumed value is inflated by `keyDeposit` for each additional subtransaction that repeats the deregistration.

The corresponding predicate failure that should detect and reject this pattern is explicitly acknowledged as unimplemented:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
    ...
    submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [2](#0-1) 

The `xit` prefix disables the test precisely because the rejection logic does not exist yet, meaning the current code **accepts** the transaction.

This is structurally identical to the external report's bug: in the external report, two independent fee-bearing conditions (cross-chain + KYC) are collapsed into a single multiplier application; here, two independent refund-bearing subtransactions are each allowed to claim the full deposit, with no guard preventing double-counting.

The deposit accounting path that should enforce uniqueness is `conwayCertsTotalRefundsTxBody` / `shelleyTotalRefundsTxCerts`, but these functions operate per-`TxBody` in isolation and are never composed across subtransactions to deduplicate claims. [3](#0-2) [4](#0-3) 

### Impact Explanation

An attacker who has registered one staking credential (paying `keyDeposit` once) can construct a Dijkstra top-level transaction containing `n` subtransactions, each bearing `UnRegDepositTxCert stakingCred keyDeposit`. The preservation-of-value check (`consumed = produced`) passes because the consumed side is inflated by `n × keyDeposit` while the attacker places `n × keyDeposit` worth of outputs in the subtransactions. The deposit pot (`utxosDeposited`) is debited by `n × keyDeposit` even though only `1 × keyDeposit` was ever deposited. This constitutes a direct, attacker-controlled extraction of `(n-1) × keyDeposit` ADA from the deposit pot — an invalid ledger state transition that destroys ADA accounting integrity.

**Impact: Medium** — attacker-controlled transactions modify deposit refunds outside design parameters (and potentially Critical if the deposit pot can be fully drained).

### Likelihood Explanation

The entry path requires only the ability to submit a valid Dijkstra transaction with subtransactions — no privileged role, no governance majority, no key compromise. Any unprivileged transaction sender who has registered a staking credential can exploit this. The only prerequisite is that the Dijkstra era is active on the network. The vulnerability is acknowledged in the codebase with a pending TODO, confirming it is a known gap rather than a theoretical edge case.

**Likelihood: Medium** — straightforward to construct once the era is active; blocked only by era activation.

### Recommendation

Before activating the Dijkstra era, implement a cross-subtransaction deduplication check in the CERT/DELEG rule (or in the top-level UTXO rule) that tracks which staking and DRep credentials have already been deregistered by any subtransaction within the same top-level transaction, and rejects any subsequent subtransaction that attempts to claim a refund for the same credential. The pending test at line 53 of `CertSpec.hs` should be enabled (`it` instead of `xit`) once the predicate failure is implemented. [2](#0-1) 

### Proof of Concept

1. Register staking credential `C`, paying `keyDeposit = D` lovelace.
2. Construct subtransaction `subTx1` with `UnRegDepositTxCert C D` and a UTxO input worth `V1`.
3. Construct subtransaction `subTx2` with `UnRegDepositTxCert C D` and a UTxO input worth `V2`.
4. Construct top-level transaction `tx` with `subTransactionsTxBodyL = [subTx1, subTx2]`.
5. `getConsumedDijkstraValue` returns `V1 + D + V2 + D` (deposit counted twice).
6. Place outputs totalling `V1 + V2 + 2D` to satisfy preservation of value.
7. Submit `tx`. Because no predicate failure is implemented, the transaction is accepted.
8. `utxosDeposited` is decremented by `2D`; attacker receives `2D` in outputs having only deposited `D` — net gain of `D` lovelace extracted from the deposit pot. [5](#0-4) [2](#0-1)

getConsumedDijkstraValue ::
  forall era l.
  ( DijkstraEraTxBody era
  , EraUTxO era
  , Value era ~ MaryValue
  , STxLevel l era ~ STxBothLevels l era
  ) =>
  PParams era ->
  (Credential Staking -> Maybe Coin) ->
  (Credential DRepRole -> Maybe Coin) ->
  UTxO era ->
  TxBody l era ->
  Value era
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue :: forall m. TxBody m era -> Value era
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)

shelleyTotalRefundsTxCerts pp lookupDeposit = snd . F.foldl' accum (mempty, Coin 0)
  where
    keyDeposit = pp ^. ppKeyDepositL
    accum (!regCreds, !totalRefunds) cert =
      case lookupRegStakeTxCert cert of
        Just k ->
          -- Need to track new delegations in case that the same key is later deregistered in
          -- the same transaction.
          (Set.insert k regCreds, totalRefunds)
        Nothing ->
          case lookupUnRegStakeTxCert cert of
            Just cred
              -- We first check if there was already a registration certificate in this
              -- transaction.
              | Set.member cred regCreds -> (Set.delete cred regCreds, totalRefunds <+> keyDeposit)
              -- Check for the deposit left during registration in some previous
              -- transaction. This de-registration check will be matched first, despite being
              -- the last case to match, because registration is not possible without
              -- de-registration.
              | Just deposit <- lookupDeposit cred -> (regCreds, totalRefunds <+> deposit)
            _ -> (regCreds, totalRefunds)
