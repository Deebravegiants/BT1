Audit Report

## Title
Smart-contract ETH withdrawal recipients without a payable fallback permanently freeze funds in `LRTWithdrawalManager` — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
When a smart contract without a `payable` fallback calls `initiateWithdrawal` for ETH, its rsETH is escrowed and later burned during `unlockQueue`. The ETH equivalent is then irrecoverably stuck in `LRTWithdrawalManager` because every call to `completeWithdrawal` reverts when attempting to push ETH to the non-payable recipient, no cancellation path exists, and the administrative sweep is blocked by the outstanding unlocked withdrawal count.

## Finding Description

**Escrow on initiation.** `initiateWithdrawal` pulls rsETH from the caller into the contract with no undo path: [1](#0-0) 

**rsETH burned on unlock.** During `unlockQueue`, the escrowed rsETH is burned from the contract and the corresponding ETH is redeemed from the unstaking vault into `LRTWithdrawalManager`: [2](#0-1) 

At this point the user's rsETH is gone and the contract holds the ETH owed to them.

**ETH push reverts on completion.** `_processWithdrawalCompletion` ends with `_transferAsset`, which uses a low-level call for ETH: [3](#0-2) 

If the recipient has no `payable` fallback, `sent` is `false` and the function reverts with `EthTransferFailed`. Because Solidity reverts roll back all state changes in the same transaction, the `popFront`, `delete withdrawalRequests`, and `unlockedWithdrawalsCount[asset]--` at lines 705, 712, and 717 are all undone. The withdrawal request is restored to the queue and `unlockedWithdrawalsCount` remains > 0. [4](#0-3) 

**No cancellation path.** There is no function in the contract that allows a user to cancel a pending withdrawal and reclaim their rsETH (or receive ETH elsewhere). The only administrative escape, `sweepRemainingAssets`, is explicitly blocked while any unlocked withdrawal exists: [5](#0-4) 

Because the stuck withdrawal keeps `unlockedWithdrawalsCount[ETH_TOKEN] > 0`, this gate never opens.

**Operator path also fails.** `completeWithdrawalForUser` calls the same `_processWithdrawalCompletion` and reverts identically. The inline comment dismisses the risk as a gas-grief, which is incorrect — the failure is a permanent revert, not a gas cost: [6](#0-5) 

## Impact Explanation
The user's rsETH is burned and the ETH equivalent is permanently locked in `LRTWithdrawalManager` with no on-chain recovery path. This is **Critical — Permanent freezing of funds**.

## Likelihood Explanation
Smart contracts (multisigs, DAO treasuries, yield aggregators, automated rebalancers) routinely hold rsETH and interact with withdrawal queues programmatically. Many deliberately omit a `payable` fallback to prevent accidental ETH acceptance. `initiateWithdrawal` is fully permissionless — no special role is required. Any such contract calling it for ETH triggers the freeze. Likelihood is **Medium**, making overall severity **Critical**.

## Recommendation
1. **Add a withdrawal cancellation function** allowing the original requester to cancel a still-locked (pre-`unlockQueue`) request and reclaim their rsETH.
2. **Allow the user to specify a separate ETH recipient address** at completion time, so a smart contract can redirect ETH to an EOA or a contract that can receive it.
3. **Adopt a pull-payment pattern for ETH**: credit an internal claimable balance in `completeWithdrawal` instead of pushing ETH, and expose a separate `claimETH()` function. This eliminates the revert-on-receive risk entirely.
4. **Document the requirement** that callers initiating ETH withdrawals must be able to receive native ETH.

## Proof of Concept
Minimal call sequence (no fork required):

1. Deploy `NoPayableFallbackVault` (no `receive()` / `fallback() payable`).
2. Vault calls `IERC20(rsETH).approve(withdrawalManager, amount)` then `initiateWithdrawal(ETH_TOKEN, amount, "")` → rsETH escrowed at line 166.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)` → rsETH burned (line 305), ETH redeemed into contract (line 307), `unlockedWithdrawalsCount[ETH_TOKEN]++` (line 809).
4. Anyone calls `completeWithdrawal(ETH_TOKEN, "")` or `completeWithdrawalForUser(ETH_TOKEN, vault, "")` → `_transferAsset` attempts `call{value}("")` to vault → returns `false` → `EthTransferFailed` revert → all state rolled back.
5. Step 4 repeats indefinitely; it always reverts.
6. `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist` (line 403).
7. ETH is permanently locked; rsETH is already burned. No on-chain recovery path exists without a proxy upgrade.
