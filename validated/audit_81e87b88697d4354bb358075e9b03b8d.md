### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Byzantine Participant to Forge Key Derivation Response - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` in the MPC contract performs a cryptographic output check only for the `AppPublicKeyPV` (publicly verifiable) variant of a CKD request. The base `AppPublicKey` variant receives **no cryptographic verification** of the response. Any single attested MPC participant can call `respond_ckd` with an arbitrary `CKDResponse` for an `AppPublicKey` request, and the contract will accept and deliver it to the user without any check.

### Finding Description

In `respond_ckd`, the contract matches on the `app_public_key` variant of the pending request:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, binding the response to the MPC network's master secret key and the user's app public key. [2](#0-1) 

For `AppPublicKey` (the non-publicly-verifiable variant), the arm is empty — `{}` — meaning **any** `CKDResponse` (`big_y`, `big_c`) is accepted unconditionally. There is no equivalent pairing check or any other cryptographic binding of the response to the MPC master secret key.

The `AppPublicKey` variant is the primary/base CKD mode. It is used in the standard (non-PV) CKD flow, as seen in the node's `CKDComputation`: [3](#0-2) 

And in the sandbox test utilities: [4](#0-3) 

The structural parallel to the ERC721 report is exact: the "extension" variant (`AppPublicKeyPV`) has a compliance check; the "base" variant (`AppPublicKey`) does not, even though both variants flow through the same `respond_ckd` entry point and both deliver key material to the user.

### Impact Explanation

A single Byzantine attested MPC participant can:

1. Observe a pending `AppPublicKey` CKD request on-chain (the request is stored in `pending_ckd_requests`, which is publicly readable).
2. Call `respond_ckd` with an arbitrary `CKDResponse` — any `big_y` and `big_c` G1 points — before or instead of the honest leader.
3. The contract accepts the response without any cryptographic check and delivers it to the user via `pending_requests::resolve_yields_for`. [5](#0-4) 

The user's TEE application then decrypts `s = big_c - a · big_y` where `big_c` and `big_y` are attacker-chosen. The result is not `msk · H(pk || app_id)` — the legitimate MPC-derived secret — but an arbitrary value. The confidentiality guarantee of CKD is broken: the user derives a key that is not bound to the MPC network's master secret key.

This matches the allowed impact: **Critical — unauthorized confidential key derivation output without the required participant authorization.** A single Byzantine participant strictly below the signing threshold can forge the CKD output for any `AppPublicKey` request.

### Likelihood Explanation

- The attacker only needs to be an attested MPC participant (no threshold collusion required).
- Pending CKD requests are on-chain and publicly visible, so the attacker can race the honest leader.
- The `AppPublicKey` variant is the default/base CKD mode used by most callers (the PV variant requires extra gas and is opt-in).
- No special privileges, leaked keys, or physical TEE attacks are required.

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` variant. For the non-PV case, the verifiable relation is `e(big_c, g2) = e(big_y, g2) · e(hash_point, public_key)` (substituting `app_pk2 = g2` since `AppPublicKey` is `a · G1` with implicit `a · G2 = a · G2`). Concretely, add a check analogous to `ckd_output_check` that verifies the response is consistent with the MPC network's public key and the user's `app_public_key`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(app_pk) => {
        if !ckd_output_check_base(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

1. Deploy the MPC contract in Running state with a BLS12-381 CKD domain.
2. Submit a `request_app_private_key` call with `CKDAppPublicKey::AppPublicKey(some_g1_point)`.
3. As any attested participant, call `respond_ckd` with the matching `CKDRequest` and a fabricated `CKDResponse { big_y: [1u8; 48], big_c: [2u8; 48] }` (arbitrary G1 points).
4. Observe that `respond_ckd` returns `Ok(())` and the fabricated response is delivered to the user — no panic, no rejection.
5. Contrast with `AppPublicKeyPV`: submitting the same fabricated response panics with `"CKD output check failed"`. [1](#0-0) [6](#0-5)
