### Title
`ProposalInit.builder` Omitted from `is_proposal_init_valid` Allows Malicious Proposer to Commit Arbitrary `sequencer_address` into Block Hash — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates many `ProposalInit` fields but silently skips the `builder` field. Because `builder` is forwarded verbatim as `sequencer_address` into `BlockInfo`, then hashed into `PartialBlockHashComponents`, a legitimate-round proposer can inject any `sequencer_address` into the committed block hash and redirect all block fees to an arbitrary address. Every validator re-derives the same hash from the same `builder` value, so the `ProposalFinMismatch` guard never fires.

---

### Finding Description

`is_proposal_init_valid` enforces the following `ProposalInit` fields:

| Field | Check |
|---|---|
| `timestamp` | monotonicity + future-window |
| `starknet_version` | exact match |
| `version_constant_commitment` | sentinel match |
| `height`, `l1_da_mode`, `l2_gas_price_fri` | exact match |
| `l1_gas_price_{fri,wei}`, `l1_data_gas_price_{fri,wei}` | within-margin |
| `fee_proposal_fri` | presence + bounds |

The `builder` field is **never checked**. [1](#0-0) 

`builder` is then consumed by `convert_to_sn_api_block_info` as the block's `sequencer_address`: [2](#0-1) 

That `sequencer_address` flows into `PartialBlockHashComponents::new`: [3](#0-2) 

And is hashed into the block hash: [4](#0-3) 

The proposer sets `builder` to `args.builder_address` (its own address): [5](#0-4) 

A malicious proposer substitutes any address. The validator calls `convert_to_sn_api_block_info(init)` with the same tampered `init`, computes the same hash, and the `ProposalFinMismatch` guard passes: [6](#0-5) 

The only place `proposer` is checked is in `handle_proposal` at the consensus layer, but `builder` is never checked there either: [7](#0-6) 

---

### Impact Explanation

**Wrong block hash / wrong state commitment (Critical-class):** The `sequencer_address` slot in the Poseidon block hash is permanently set to the attacker-chosen value. Every downstream consumer (L1 verifier, state sync, RPC `starknet_getBlock`) sees a block header whose `sequencer_address` is incorrect.

**Fee theft (Critical-class):** The blockifier uses `BlockInfo.sequencer_address` to credit transaction fees. Setting `builder` to an attacker-controlled address redirects all fees for that block to that address, with no recovery path.

---

### Likelihood Explanation

The attacker must be the **legitimate proposer** for the current consensus round. In a decentralized validator set this is a normal, rotating role — not a privileged admin operation. The manipulation requires only changing one field in the locally-constructed `ProposalInit` before broadcasting; no cryptographic material needs to be forged.

---

### Recommendation

Add a `builder` check inside `is_proposal_init_valid`. The simplest invariant is that `builder` must equal `proposer` (the address already validated by the consensus layer):

```rust
if init_proposed.builder != init_proposed.proposer {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: proposer={:?}, builder={:?}",
            init_proposed.proposer, init_proposed.builder
        ),
    ));
}
```

If `builder` and `proposer` are intentionally decoupled in future designs, the validator must receive the expected `builder` address through a trusted out-of-band channel (e.g., the staking/committee registry) and enforce it here.

---

### Proof of Concept

1. Attacker is the legitimate proposer for round R at height H.
2. In `initiate_build`, attacker replaces `builder: args.builder_address` with `builder: ContractAddress::from(ATTACKER_WALLET)`.
3. Attacker broadcasts `ProposalPart::Init(tampered_init)`.
4. Each validator calls `is_proposal_init_valid` — passes (no `builder` check).
5. Each validator calls `convert_to_sn_api_block_info(&tampered_init)` → `sequencer_address = ATTACKER_WALLET`.
6. Batcher executes the block; blockifier credits all fees to `ATTACKER_WALLET`.
7. `PartialBlockHashComponents::new` hashes `ATTACKER_WALLET` as `sequencer`.
8. Both proposer and validator derive the same `PartialBlockHash` → `ProposalFinMismatch` guard passes.
9. Consensus reaches decision; block is committed with `sequencer_address = ATTACKER_WALLET` in the block hash.
10. All fees for that block are permanently credited to the attacker's wallet; the block hash is permanently wrong.
