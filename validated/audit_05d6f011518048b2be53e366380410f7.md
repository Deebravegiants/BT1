### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Redirect Block Fees — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` rigorously validates every economically-sensitive field of `ProposalInit` — `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices (FRI + WEI), `starknet_version`, `version_constant_commitment`, `fee_proposal_fri`, and `timestamp` — but leaves the `builder` field completely unchecked. `builder` is subsequently forwarded verbatim to `convert_to_sn_api_block_info(init)` inside `initiate_validation`, which maps it to the `sequencer_address` in the `BlockInfo` passed to the batcher. A validator who wins a proposer slot can set `builder` to any address; the batcher will execute the entire block with that address as the sequencer, directing all transaction fees to the attacker-controlled address.

---

### Finding Description

`validate_proposal` calls `is_proposal_init_valid` before handing the proposal to the batcher: [1](#0-0) 

Inside `is_proposal_init_valid`, the following fields are explicitly checked against locally-trusted reference values: [2](#0-1) [3](#0-2) [4](#0-3) 

The `ProposalInitValidation` struct — the sole source of locally-trusted reference values — has no `builder` member: [5](#0-4) 

After `is_proposal_init_valid` returns `Ok`, `initiate_validation` is called and passes the full, unmodified `init` (including the unchecked `builder`) to `convert_to_sn_api_block_info`: [6](#0-5) 

`convert_to_sn_api_block_info` maps `init.builder` to the `sequencer_address` field of Starknet's `BlockInfo`. The batcher then executes every transaction in the block with that address as the sequencer, which is the address that receives all transaction fees.

The `ProposalInit` wire format confirms `builder` is a fully proposer-controlled field transmitted over the network: [7](#0-6) 

The consensus manager does validate `proposer` identity against the committee: [8](#0-7) 

But `builder` is a separate field and receives no analogous check anywhere in the validation pipeline.

---

### Impact Explanation

Every transaction fee in the block is paid to `sequencer_address`. By setting `builder` to an attacker-controlled address, the proposer redirects 100 % of the block's fee revenue. The batcher computes the `proposal_commitment` (partial block hash) using the supplied `builder`/`sequencer_address`, so the commitment comparison at line 244 still passes — the validator accepts the proposal and the block is committed with the wrong sequencer address baked into the state.

Impact category: **Critical — Incorrect fee/balance effect with direct economic impact.**

---

### Likelihood Explanation

Any validator in the active committee can be selected as the proposer for a given round. No special privilege beyond normal validator participation is required. The attack is silent (no on-chain signal distinguishes a legitimate `builder` from a malicious one) and repeatable every round the attacker is selected.

---

### Recommendation

Add `builder` to `ProposalInitValidation` and verify it against the locally-known expected sequencer address inside `is_proposal_init_valid`, analogous to how `l1_da_mode` and `l2_gas_price_fri` are checked:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.expected_builder, init_proposed.builder
        ),
    ));
}
```

The `expected_builder` value should be derived from the node's own configuration (the sequencer's contract address), not from any proposer-supplied data.

---

### Proof of Concept

1. Attacker controls a validator that wins the proposer slot for height H, round R.
2. Attacker constructs a `ProposalInit` with all validated fields set to legitimate values (passing `is_proposal_init_valid`) but sets `builder = ATTACKER_ADDRESS`.
3. Validator node receives the proposal; `is_proposal_init_valid` passes because `builder` is never checked.
4. `initiate_validation` calls `convert_to_sn_api_block_info(init)`, producing a `BlockInfo` with `sequencer_address = ATTACKER_ADDRESS`.
5. Batcher executes all transactions with `sequencer_address = ATTACKER_ADDRESS`; all fees accumulate to that address.
6. Batcher returns a `proposal_commitment` computed over the block that includes `ATTACKER_ADDRESS` as sequencer.
7. The commitment comparison (`built_block != received_fin.proposal_commitment`) passes because both sides used the same `builder`.
8. Consensus reaches decision; block is committed. All fees for height H are permanently redirected to `ATTACKER_ADDRESS`. [9](#0-8)
