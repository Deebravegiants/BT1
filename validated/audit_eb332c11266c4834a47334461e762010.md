Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice()` Missing Chainlink Validity Checks Enables rsETH Price Manipulation and Withdrawal DoS - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validity fields (`roundId`, `startedAt`, `updatedAt`, `answeredInRound`), using only the raw `price` value with no staleness, completeness, or positivity checks. This contrasts directly with `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which applies all three checks. A stale or zero Chainlink price propagates into rsETH price computation, enabling an attacker to mint rsETH at a deflated price (diluting existing holders) and causing division-by-zero reverts in the withdrawal unlock path.

## Finding Description

**Root cause:** `ChainlinkPriceOracle.getAssetPrice()` at lines 52–54 of `contracts/oracles/ChainlinkPriceOracle.sol` silently discards all validity fields:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is made that `price > 0`, `answeredInRound >= roundId`, or `updatedAt != 0`. The same codebase's `ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 30–32) performs all three checks and reverts on failure.

**Path 1 — rsETH price deflation and theft:**

`LRTOracle._getTotalEthInProtocol()` (lines 336–343) calls `getAssetPrice(asset)` for every supported LST and accumulates `totalETHInProtocol`. If any asset's feed returns a stale or zero price, that asset's TVL contribution is zeroed or understated. `_updateRsETHPrice()` (line 250) then computes:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

A deflated `totalETHInProtocol` produces a deflated `newRsETHPrice`. `updateRSETHPrice()` is public and callable by anyone. The downside protection at lines 270–281 only pauses the protocol if the price drop exceeds `pricePercentageLimit`; if `pricePercentageLimit` is zero (default until set by admin) or the drop is within the configured limit, the protocol does not pause and the deflated price is committed.

An attacker then calls `LRTDepositPool.depositAsset()` with a different, validly-priced asset. `getRsETHAmountToMint()` (line 520) computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

With `rsETHPrice` artificially low, the attacker receives excess rsETH. When the oracle recovers and `rsETHPrice` is updated to its true value, the attacker's rsETH is worth more than deposited — extracting value from all existing rsETH holders.

**Path 2 — Withdrawal DoS via division by zero:**

`LRTWithdrawalManager._createUnlockParams()` (lines 846–848) passes `lrtOracle.getAssetPrice(asset)` as `assetPrice`. This value is used in `_calculatePayoutAmount()` (line 833):

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
```

If `assetPrice = 0` (Chainlink returns zero for a deprecated/invalid feed), this divides by zero and reverts, blocking all withdrawal unlocks for that asset. The `_validatePrices()` function (lines 864–869) only guards against this if `minimumAssetPrice` is configured to a non-zero value by the caller; with default zero parameters it provides no protection.

**Existing guards are insufficient:** The `pricePercentageLimit` downside protection only triggers a pause on large drops; small staleness (e.g., a few percent off) passes through silently. The `_validatePrices()` check is parameterized and provides no guarantee against zero prices unless explicitly configured.

## Impact Explanation

**Critical — Direct theft of user funds:** A stale Chainlink feed for any supported LST causes `rsETHPrice` to be computed below its true value. An unprivileged attacker mints rsETH at the deflated price using a different asset, then redeems at the recovered true price, extracting value from all existing rsETH holders. This matches the "Direct theft of any user funds" critical impact class.

**Medium — Temporary freezing of funds:** A zero price from any supported asset's Chainlink feed causes division-by-zero reverts in `_calculatePayoutAmount()`, blocking all withdrawal unlocks for that asset until the oracle recovers. This matches the "Temporary freezing of funds" medium impact class.

## Likelihood Explanation

Chainlink feeds can return stale data during network congestion, sequencer downtime (on L2), or feed deprecation — all externally observable, non-privileged conditions. `updateRSETHPrice()` is public with no access control, so any actor can trigger the price update at the moment of staleness. The deposit path is also fully public. No privileged access, victim mistake, or governance capture is required. The condition is repeatable whenever a feed goes stale.

## Recommendation

Apply the same validity checks already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, add a heartbeat staleness check (`block.timestamp - updatedAt > MAX_HEARTBEAT_DELAY`) per feed.

## Proof of Concept

1. Chainlink's stETH/ETH feed goes stale (`answeredInRound < roundId`, or returns `price = 0`).
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `0`.
4. `totalETHInProtocol` is computed excluding all stETH TVL (e.g., drops from 10,000 ETH to 3,000 ETH).
5. `newRsETHPrice` is set to `3000e18 / rsethSupply` instead of `10000e18 / rsethSupply`.
6. If `pricePercentageLimit == 0` or the drop is within the configured limit, the protocol does not pause and the deflated price is committed.
7. Attacker calls `LRTDepositPool.depositAsset(rETH, amount)` — `getRsETHAmountToMint` uses the deflated `rsETHPrice`, minting ~3.33× more rsETH than fair value.
8. Oracle recovers; `rsETHPrice` returns to true value; attacker's rsETH is worth more than deposited, diluting all other rsETH holders.

**Foundry fork test plan:** Fork mainnet, mock a Chainlink aggregator to return `answeredInRound < roundId` with `price = 0` for stETH feed, call `updateRSETHPrice()`, assert `rsETHPrice` is deflated, deposit rETH as attacker, assert rsETH minted exceeds fair value, restore oracle, call `updateRSETHPrice()` again, assert attacker profit at expense of existing holders.
