### Title
Block Producer-Manipulable Entropy Used as Presignature Rerandomization Input — (`crates/node/src/types.rs`, `crates/node/src/indexer/handler.rs`)

---

### Summary

Both `SignatureRequest.entropy` and `CKDRequest.entropy` are sourced directly from `block.header.random_value` (the NEAR block's VRF output). This value is used as the sole "unpredictable" entropy input to the presignature rerandomization scheme (GS21). A NEAR block producer can manipulate which block a signing request lands in, effectively selecting from a small set of `random_value`s, violating the explicit unpredictability requirement of the rerandomization scheme and weakening the Wagner-attack protection it is designed to provide.

---

### Finding Description

In `crates/node/src/indexer/handler.rs`, the block context is assembled from the raw NEAR streamer message:

```rust
entropy: streamer_message.block.header.random_value.into(),
``` [1](#0-0) 

This `BlockContext.entropy` is then directly assigned to every `SignatureRequest` and `CKDRequest` created from that block:

```rust
// SignatureRequest
entropy: block.entropy.clone().into(),
// CKDRequest
entropy: block.entropy.clone().into(),
``` [2](#0-1) [3](#0-2) 

This `entropy` field is then passed directly into `RerandomizationArguments`, which the code explicitly documents must be:

> **"Fresh, Unpredictable, and Public source of entropy"** [4](#0-3) 

The `derive_randomness()` function uses this entropy as the HKDF key material to derive the rerandomization scalar `δ`:

```rust
let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);
``` [5](#0-4) 

`δ` is then used to rerandomize the presignature nonce: `R → R^δ`, `k_i → k_i · δ⁻¹`, `σ_i → (σ_i + ε·k_i) · δ⁻¹`. [6](#0-5) 

The rerandomization is applied in both the OT-based ECDSA and Robust ECDSA signing paths: [7](#0-6) [8](#0-7)

### Impact Explanation

The GS21 rerandomization scheme's security proof requires that `ρ` (entropy) is **unpredictable at the time the presignature was generated**. NEAR's `block.header.random_value` is a VRF output produced by the block producer. A malicious block producer:

1. Sees a pending `sign()` call in the mempool.
2. Knows their own VRF output (i.e., the `random_value` of the block they are about to produce) before publishing it.
3. Can choose to **include or withhold** the signing request from their block, effectively selecting between the current block's `random_value` and a future block's `random_value`.

This breaks the "unpredictable" invariant. If the block producer also learns the presignature `R` (e.g., via a side channel, a compromised node, or a future disclosure), they can compute `δ = HKDF(entropy, pk, h, R, participants)` in advance, predict the rerandomized nonce `R' = R^δ`, and mount a Wagner-style adaptive chosen-message attack to extract the MPC private key shares. The signing documentation explicitly warns of this class of attack:

> "Following [GS21]'s recommendation, we rerandomize the presignature to make the Wagner attack practically infeasible." [9](#0-8) 

**Impact class**: Medium — breaks the production safety invariant of the rerandomization scheme (the unpredictability of `δ`). Escalates to Critical if presignature material is ever exposed, as it would enable MPC key share extraction.

---

### Likelihood Explanation

NEAR block producers are external, potentially adversarial actors relative to the MPC system. No threshold collusion is required — a single block producer controlling a block slot can perform this manipulation. The ability to selectively include or delay transactions is a standard capability of any block producer on any PoS chain. The attack is passive (no on-chain footprint) and requires only that the block producer observe the pending signing request and know their own VRF output, both of which are routine.

---

### Recommendation

Replace the sole reliance on `block.header.random_value` as the entropy source. Options include:

1. **Mix in node-local randomness**: Each MPC node contributes a locally-generated random value (e.g., from `rand::thread_rng()`) that is committed to before the signing session and revealed during it. The entropy becomes `H(block_random_value || node_contribution_1 || ... || node_contribution_n)`, which is unpredictable as long as at least one honest node participates.

2. **Use the signing request's receipt ID as additional entropy**: The receipt ID is determined before the block producer knows the `random_value`, providing some independence. However, this alone is insufficient.

3. **Derive entropy from a threshold-agreed value**: Have MPC nodes run a lightweight coin-toss sub-protocol to agree on fresh entropy for each signing session, independent of the block header.

---

### Proof of Concept

**Step 1**: A user submits a `sign()` call to the MPC contract. The transaction sits in the NEAR mempool.

**Step 2**: A malicious block producer (who controls the next block slot) computes their VRF output `v₁` for the block they are about to produce. They also compute `v₂` for the hypothetical next block (if they delay the request).

**Step 3**: The block producer evaluates which `entropy` value (`v₁` or `v₂`) is more favorable. Without knowing the presignature `R`, they cannot yet compute `δ`, but they can record both values.

**Step 4**: The block producer includes the signing request in the block with `random_value = v₁`. The MPC nodes read `entropy = v₁` from `block.header.random_value` and use it in `RerandomizationArguments`.

**Step 5**: If the block producer later learns the presignature `R` (e.g., through a compromised MPC node or a future protocol disclosure), they compute:
```
δ = HKDF(v₁, pk, h, R, participants)
R' = R^δ
```
They now know the rerandomized nonce `R'` before the signing session completes, enabling a Wagner-style nonce-reuse attack to extract the MPC key shares.

```rust
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
