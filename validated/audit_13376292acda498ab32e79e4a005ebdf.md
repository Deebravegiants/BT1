### Title
`respond_ckd` Skips Cryptographic Output Validation for `AppPublicKey` Variant While `respond` Always Validates — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond` function enforces a cryptographic validity check on every signature response before resolving any pending yield. The `respond_ckd` function enforces an equivalent check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`, and silently accepts any response for the `AppPublicKey` variant with no validation at all. A single Byzantine attested participant can exploit this inconsistency to deliver an arbitrary, forged CKD response for any in-flight `AppPublicKey` request, resolving all queued yields with incorrect key-derivation output before honest nodes can respond.

---

### Finding Description

**Root cause — inconsistent response-validity enforcement across two code paths that must enforce the same invariant.**

`respond` (lines 564–651) always validates the submitted signature against the on-chain derived public key before calling `resolve_yields_for`:

```rust
// crates/contract/src/lib.rs  lines 586-644
let signature_is_valid = match (&response, public_key) { … };
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
``` [1](#0-0) 

`respond_ckd` (lines 653–689) performs the analogous check only for `AppPublicKeyPV`. For `AppPublicKey` the match arm is an empty no-op — the response is forwarded to `resolve_yields_for` unconditionally:

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO VALIDATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [2](#0-1) 

After the match, `resolve_yields_for` is called unconditionally for both variants: [3](#0-2) 

The only authorization gate before this point is `assert_caller_is_attested_participant_and_protocol_active()` (line 666), which any single attested participant satisfies. [4](#0-3) 

**Exploit flow:**

1. A legitimate user calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(…)`. The request is enqueued in `pending_ckd_requests` and a yield is created.
2. A Byzantine attested participant (one node, well below the signing threshold) calls `respond_ckd` with a fabricated `CKDResponse` (arbitrary `big_y`, `big_c` values) before any honest node can respond.
3. Because the `AppPublicKey` arm performs no validation, the fabricated response passes all checks.
4. `resolve_yields_for` drains every queued yield for that request key, delivering the forged output to every caller who submitted the same request.
5. Honest nodes' subsequent `respond_ckd` calls find no pending entry and are silently ignored.

The `AppPublicKey` variant is a first-class production API path exposed through `request_app_private_key` / `CKDRequestArgs`. [5](#0-4) 

---

### Impact Explanation

**Critical — confidential key derivation output delivered without the required participant authorization.**

The CKD subsystem is described as "Deriving app-specific secrets without revealing master keys." A forged `CKDResponse` for an `AppPublicKey` request means the user receives a `(big_y, big_c)` pair that was not computed by the threshold MPC protocol. Any downstream use of that pair — key encapsulation, secret derivation, wallet key bootstrapping — operates on attacker-controlled material. A single Byzantine participant (strictly below the signing threshold) achieves this without any threshold collusion.

---

### Likelihood Explanation

Every attested participant is authorized to call `respond_ckd`. NEAR transaction ordering is determined by the block producer; a Byzantine participant who is also a block producer, or who submits with a higher gas price in a competitive slot, can reliably front-run honest nodes. Even without block-producer collusion, a Byzantine node that monitors the NEAR indexer for new `request_app_private_key` events can race to submit a forged response in the same or next block. The `AppPublicKey` variant requires no special setup by the attacker beyond holding a valid TEE attestation.

---

### Recommendation

Apply the same cryptographic output validation to the `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. Concretely, implement and call a validity check (analogous to `ckd_output_check`) for `AppPublicKey` responses inside `respond_ckd`, or unify both variants under a single mandatory validation path before `resolve_yields_for` is reached. The pattern already established in `respond` — where `if !signature_is_valid { return Err(…) }` gates every code path — should be mirrored here.

---
