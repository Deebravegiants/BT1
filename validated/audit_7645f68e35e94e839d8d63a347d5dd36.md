### Title
Excess Deposit Permanently Locked in Contract on Participant Re-Attestation Path — (`File: crates/contract/src/lib.rs`)

### Summary
`submit_participant_info` is a `#[payable]` method that conditionally handles deposit accounting. When an existing participant re-submits their attestation (not a new entry), the entire deposit-check-and-refund block is skipped, causing any attached NEAR deposit to be permanently absorbed by the contract with no refund path.

### Finding Description

`submit_participant_info` computes a boolean gate to decide whether to enforce deposit accounting:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check minimum, refund excess
}
``` [1](#0-0) 

When both conditions are false — the caller **is** an existing participant AND the submission **updates** an existing attestation entry (e.g., a TEE upgrade re-submission) — the `if` block is never entered. Because the function is marked `#[payable]`, the NEAR runtime accepts any attached deposit without complaint. No minimum is enforced, no excess is refunded, and the deposit is silently credited to the contract account with no mechanism to recover it.

The analogous pattern in the external report is `_transferRem` not being called after certain external methods: here, the equivalent "transfer remaining balance" logic (`Promise::new(account_id).transfer(diff).detach()`) is entirely absent for the re-submission code path. [2](#0-1) 

### Impact Explanation

Any NEAR tokens attached to a `submit_participant_info` call that lands on the `!attestation_storage_must_be_paid_by_caller` branch are permanently locked in the contract. The contract exposes no general withdrawal function. This breaks the production accounting invariant that excess deposits must be returned to callers, and results in permanent freezing of the caller's funds.

**Impact: Medium** — balance accounting invariant broken; funds permanently frozen without any theft or threshold bypass.

### Likelihood Explanation

The primary caller of `submit_participant_info` in production is the MPC node's `periodic_attestation_submission` task. During a TEE upgrade cycle, nodes re-submit their attestation while already being participants, landing squarely on the vulnerable branch. If the node's transaction construction attaches any non-zero deposit (e.g., due to a misconfiguration, a future code change, or a manual operator invocation), those funds are lost. The branch is reachable by any existing participant without any threshold collusion.

### Recommendation

Remove the conditional guard around deposit handling, or add an unconditional refund of any excess deposit at the end of `submit_participant_info` regardless of which branch is taken. The simplest fix mirrors the pattern already used in `propose_update`:

```rust
// After all logic, unconditionally refund any deposit not consumed by storage.
let attached = env::attached_deposit();
if attached > NearToken::from_yoctonear(0) {
    Promise::new(account_id).transfer(attached).detach();
}
```

Alternatively, restrict the function to reject non-zero deposits on the re-submission path, making the accounting explicit. [3](#0-2) 

### Proof of Concept

1. Deploy the MPC contract and initialize it with participant `alice.near`.
2. `alice.near` calls `submit_participant_info` with a valid attestation (first call) — this is the new-attestation path; deposit is checked and excess refunded correctly.
3. `alice.near` calls `submit_participant_info` again with an updated attestation (re-submission) while still a participant, attaching `1 NEAR` as deposit.
4. `is_new_attestation` = `false` (entry already exists), `caller_is_not_participant` = `false` (alice is a participant).
5. `attestation_storage_must_be_paid_by_caller` = `false` → the entire deposit block is skipped.
6. The call succeeds with `Ok(())`. Alice's `1 NEAR` is now in the contract account with no refund issued and no recovery path. [4](#0-3)
