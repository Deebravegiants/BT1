### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Binding It to the Stored Request — Cross-Request Signature Replay by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, and that the `request` key exists in `pending_verify_foreign_tx_requests`, but it never checks that `payload_hash` is actually derived from the stored `request`. A single malicious attested participant (below the signing threshold) can replay a legitimately produced signature from a previously resolved request against any currently pending request, causing the pending request's yield to resolve with a `payload_hash` that commits to a completely different foreign-chain transaction.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs two independent checks:

1. The ECDSA signature in `response` is valid over `response.payload_hash` under the domain's root public key.
2. The `request` argument exists as a key in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What it never does is verify that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request: <the stored request>, values: ... }))`. The hash is accepted as-is from the caller.

The canonical payload that nodes are supposed to sign is: [2](#0-1) 

`payload_hash` binds the `ForeignChainRpcRequest` (including `tx_id`, chain family, finality, extractors) and the `ExtractedValue` list. The contract never recomputes this hash from the stored request to confirm the binding.

**Attack path (single Byzantine participant below threshold):**

1. Request A (`tx_id = 0xAA`) is submitted and legitimately processed. The MPC network produces `(payload_hash_A, sig_A)` where `payload_hash_A = SHA-256(borsh({ request_A, values_A }))`. The leader calls `respond_verify_foreign_tx(request_A, { payload_hash_A, sig_A })`. This is accepted; request A is resolved and the response is emitted on-chain (fully public).

2. Request B (`tx_id = 0xBB`) is submitted and is now pending in `pending_verify_foreign_tx_requests`.

3. The malicious attested participant calls `respond_verify_foreign_tx(request_B, { payload_hash_A, sig_A })`.

4. The contract checks:
   - `sig_A` is valid over `payload_hash_A` under the root key → **passes** (the signature is genuine).
   - `request_B` exists in `pending_verify_foreign_tx_requests` → **passes** (it is pending).
   - `payload_hash_A` is derived from `request_B` → **never checked**.

5. Request B's yield resolves with `{ payload_hash: payload_hash_A, sig_A }`. The caller of request B receives a "verified" attestation that commits to request A's `tx_id` and extracted values, not their own.

### Impact Explanation

Any NEAR contract that uses `verify_foreign_transaction` to gate a bridge operation (e.g., Omnibridge inbound flow) and does not independently recompute the expected `payload_hash` from its own request will accept the forged response as proof that its foreign-chain transaction was verified. The attacker can cause the bridge contract to release funds or execute state transitions based on a fabricated attestation, constituting **forged foreign-chain verification and invalid bridge execution**.

The design documentation explicitly states that callers *can* verify the hash locally, but this is advisory, not enforced by the contract: [3](#0-2) 

The contract is the authoritative trust anchor; if it resolves the yield with a mismatched hash, downstream contracts have no on-chain mechanism to detect the forgery.

### Likelihood Explanation

- Requires only **one** malicious attested participant (strictly below the signing threshold). No threshold collusion is needed.
- The replayed signature is fully public: it appears in the NEAR transaction receipt of the legitimate `respond_verify_foreign_tx` call for request A.
- The attacker needs only to observe any previously resolved foreign-tx response and find any concurrently pending request to target.
- The window of opportunity exists whenever two different `verify_foreign_transaction` requests are pending or sequentially processed, which is the normal operating condition for a bridge.

### Recommendation

Inside `respond_verify_foreign_tx`, after the signature check, recompute the expected payload hash from the stored request and assert equality with `response.payload_hash`:

```rust
// After verifying the signature is valid:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    // values are embedded in payload_hash; recompute from response is not possible,
    // but the hash must at minimum commit to the correct request prefix.
    // Alternative: include the request hash in the response DTO and verify it here.
    ..
}).compute_msg_hash()?;
// OR: require the response to carry the full ForeignTxSignPayload (not just the hash)
// so the contract can recompute and verify the binding itself.
```

The cleanest fix is to change `VerifyForeignTransactionResponse` to carry the full `ForeignTxSignPayload` (or at minimum the `ForeignChainRpcRequest` and `Vec<ExtractedValue>`), let the contract recompute `payload_hash`, and verify it matches the stored request before resolving the yield. This mirrors the "balance before/after" pattern from the reference report: record the actual state, then verify the claimed value matches. [4](#0-3) 

### Proof of Concept

```
1. Alice submits verify_foreign_transaction({ tx_id: 0xAA, chain: Bitcoin, ... })
   → pending_verify_foreign_tx_requests[req_A] = [yield_A]

2. MPC network processes req_A, produces:
   payload_hash_A = SHA256(borsh(ForeignTxSignPayload::V1 { request: req_A, values: [block_hash_0xAA] }))
   sig_A = ECDSA_sign(root_key, payload_hash_A)

3. Legitimate leader calls respond_verify_foreign_tx(req_A, { payload_hash_A, sig_A })
   → yield_A resolved; response { payload_hash_A, sig_A } is emitted on-chain (public).

4. Bob submits verify_foreign_transaction({ tx_id: 0xBB, chain: Bitcoin, ... })
   → pending_verify_foreign_tx_requests[req_B] = [yield_B]

5. Malicious attested participant calls:
   respond_verify_foreign_tx(req_B, { payload_hash_A, sig_A })
   
   Contract checks:
   ✓ verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → OK
   ✓ pending_verify_foreign_tx_requests.get(req_B) → Some([yield_B])
   ✗ payload_hash_A == SHA256(borsh({ req_B, ... })) → NOT CHECKED

   → yield_B resolved with { payload_hash: payload_hash_A, sig_A }

6. Bob's NEAR contract receives payload_hash_A (commits to tx_id=0xAA),
   believes tx_id=0xBB was verified, releases bridge funds.
```

        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
