### Title
Sub-Transaction Plutus Execution Units Excluded from Minimum Fee Calculation — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the minimum fee calculation for a top-level transaction does not include the execution units declared in sub-transaction redeemers. Because collateral is required whenever *any* transaction in the batch has redeemers — confirming that sub-transaction Plutus scripts are executed — the omission of sub-transaction execution units from the fee formula allows an attacker to run arbitrarily expensive Plutus scripts at a cost that is lower than the protocol intends.

---

### Finding Description

The Dijkstra era introduces nested ("sub") transactions. A top-level `DijkstraTx` carries a set of sub-transactions in `dtbrSubTransactions`. Sub-transactions have their own witnesses, including redeemers for Plutus scripts.

The minimum fee for the entire batch is computed once, for the top-level transaction, via `getMinFeeTxUtxo`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` delegates to `getConwayMinFeeTx`, which calls `alonzoMinFeeTx`:

```haskell
getConwayMinFeeTx pp tx refScriptsSize =
  alonzoMinFeeTx pp tx <+> refScriptsFee
``` [2](#0-1) 

`alonzoMinFeeTx` computes the script execution fee as:

```haskell
alonzoMinFeeTx pp tx =
  shelleyMinFeeTx pp tx <+> txscriptfee (pp ^. ppPricesL) (totExUnits tx)
```

where `totExUnits tx` folds only over the **top-level** transaction's redeemer map:

```haskell
totExUnits tx = foldMap' exUnits (tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL)
```

Sub-transaction redeemers live in `subTx ^. witsTxL . rdmrsTxWitsL . unRedeemersL` and are never visited by this fold.

Meanwhile, `validateBatchCollateral` in the Dijkstra UTXO rule explicitly checks for redeemers in sub-transactions and requires collateral when any are present:

```haskell
hasAnyRedeemers t =
  hasRedeemers t || any hasRedeemers (t ^. bodyTxL . subTransactionsTxBodyL)
``` [3](#0-2) 

The requirement for collateral confirms that sub-transaction Plutus scripts are executed. Yet the fee check enforced in `dijkstraUtxoTransition`:

```haskell
{- minfee pp txTop utxo₀ ≤ txfee txb -}
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
``` [4](#0-3) 

uses the same `getMinFeeTxUtxo` that ignores sub-transaction execution units. The `SUBUTXO` rule, which processes each sub-transaction, calls `updateUTxOStateNoFees` and performs no fee check of its own:

```haskell
-- | Like 'updateUTxOState', but does not collect fees. This is used for sub-transactions
-- where fees are not applicable.
updateUTxOStateNoFees ...
``` [5](#0-4) 

The result is a structural gap identical to the reported `fillLPOrder` bug: one code path (top-level transaction) enforces the execution-unit fee; the other (sub-transactions) does not.

---

### Impact Explanation

An attacker can place arbitrarily expensive Plutus scripts exclusively in sub-transactions. The top-level transaction pays only the size-based fee (`a + b·size`) plus any top-level execution units, while the sub-transaction execution units — which can be set up to the per-transaction `maxTxExUnits` limit — are not priced. This allows script execution at a cost below the protocol-defined price, modifying fees outside design parameters.

**Impact class**: Medium — attacker-controlled transactions modify fees outside design parameters.

---

### Likelihood Explanation

The Dijkstra era is the current development era and has not yet been deployed to mainnet. Once deployed, any user who can construct a valid Dijkstra transaction can exploit this. No privileged access, governance majority, or key compromise is required. The attacker only needs to know the sub-transaction structure and place redeemers there instead of in the top-level transaction.

---

### Recommendation

Extend `totExUnits` (or the Dijkstra-specific fee calculation) to aggregate execution units from all sub-transaction redeemers in addition to the top-level redeemers. For example:

```haskell
totExUnitsDijkstra tx =
  totExUnits tx
    <> foldMap' totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
```

Override `getMinFeeTxUtxo` for `DijkstraEra` to use this extended function, ensuring the minimum fee check in `dijkstraUtxoTransition` covers the full execution cost of the batch.

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction `txTop` with **no redeemers** in its own witness set and a fee set to `minfee(pp, txTop)` (size-based only).
2. Embed a sub-transaction `txSub` whose witness set contains redeemers with `exUnits = maxTxExUnits pp` (maximum allowed execution budget).
3. Submit `txTop`. The fee check `validateFeeTooSmallUTxO pp txTop originalUtxo` passes because `totExUnits txTop = 0`, so `minfee` does not include any script fee.
4. `validateBatchCollateral` triggers because `hasAnyRedeemers txTop = True` (sub-transaction has redeemers), so collateral is required and provided — confirming the scripts will be executed.
5. The transaction is accepted. The attacker has executed `maxTxExUnits` worth of Plutus computation while paying zero execution-unit fee, paying only the size-based component.

The discrepancy between the fee actually charged and the fee that would be charged if the same scripts were placed in the top-level transaction can be as large as `txscriptfee(prices, maxTxExUnits)` per transaction.
