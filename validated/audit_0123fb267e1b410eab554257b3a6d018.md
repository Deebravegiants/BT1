Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` Operand in Bounds Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric bounds check: the ERC-20 branch correctly tests `totalAssetDeposits + amount > depositLimit`, but the ETH branch omits `amount` and only tests `totalAssetDeposits > depositLimit`. This allows any depositor to push the protocol's ETH TVL arbitrarily above the configured deposit cap in a single `depositETH` call, causing rsETH to be minted beyond the intended safety ceiling.

## Finding Description
The function at [1](#0-0)  evaluates the ETH branch as `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` (line 679), which only checks whether the **current** total already exceeds the limit — not whether the **post-deposit** total would. The ERC-20 branch on line 681 correctly includes `+ amount`. The function is the sole deposit-cap guard and is called unconditionally from `_beforeDeposit` at [2](#0-1) , which is invoked by every `depositETH` call at [3](#0-2) . No other guard exists to enforce the ETH deposit cap before rsETH is minted.

## Impact Explanation
**Low — Contract fails to deliver promised returns.** The deposit cap is an explicit protocol constraint (`depositLimitByAsset`). With `depositLimit = 1 000 ETH` and `totalAssetDeposits = 999 ETH`, the check `999 > 1 000` is `false`, so a depositor can send any amount (e.g., 500 ETH) in a single transaction. After the call, `totalAssetDeposits = 1 499 ETH` — 49.9% above the cap — and rsETH is minted for the full 500 ETH. The overshoot is bounded only by the depositor's balance, not the configured limit. This maps to **Low** (contract fails to deliver promised returns / deposit cap guarantee is broken), with potential escalation to **Medium** if excess ETH cannot be deployed into EigenLayer strategies and becomes stranded.

## Likelihood Explanation
`depositETH` is public and permissionless — no role, whitelist, or special condition is required. [4](#0-3)  Any depositor monitoring on-chain state can exploit this in a single transaction the moment `totalAssetDeposits` approaches `depositLimit`. Likelihood is **High**.

## Recommendation
Add `amount` to the ETH branch to mirror the ERC-20 branch:

```solidity
// contracts/LRTDepositPool.sol L678-L680
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [5](#0-4) 

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1_000 ether`.
2. Protocol accumulates `getTotalAssetDeposits(ETH) = 999 ether` through normal usage.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999 > 1_000` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the full 500 ETH.
6. `getTotalAssetDeposits(ETH)` now returns `1_499 ether`, 49.9% above the cap.
7. For comparison, the ERC-20 path for the same scenario evaluates `999 + 500 > 1_000` → `true` → `MaximumDepositLimitReached` revert, correctly blocking the deposit.

**Foundry test plan:**
```solidity
function test_ethDepositLimitBypass() public {
    vm.prank(admin);
    lrtConfig.setDepositLimitByAsset(LRTConstants.ETH_TOKEN, 1_000 ether);
    // Simulate 999 ETH already deposited

}
```
