### Title
Unvalidated `builder` Field in `ProposalInit` Allows Proposer to Redirect Sequencer Fees to Arbitrary Address - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary
The `builder` field in `ProposalInit` is accepted from the network and used as `sequencer_address` in block execution without any validation in `is_proposal_init_valid`. A malicious proposer can set `builder` to any arbitrary address, causing all transaction fees in the block to be collected by that address instead of the legitimate sequencer. The commitment comparison does not catch this because the validator itself uses the proposer-supplied `builder` value to re-execute the block.

### Finding Description
In `convert_to_sn_api_block_info` (`crates/apollo_consensus_orchestrator/src/utils.rs`, line 332), `init.builder` is mapped directly to `sequencer_address` in `BlockInfo`:

```rust
sequencer_address: init.builder,
```

This `BlockInfo` is passed to the batcher via `initiate_validation` → `batcher.validate_block(input)` where `input.block_info` carries the proposer-supplied `builder` as the sequencer address for all transaction execution in the block.

In `is_proposal_init_valid` (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, lines 253–419), the validator checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `timestamp`, `starknet_version`, `version_constant_commitment`, all four L1 gas price fields, and `fee_proposal_fri`. The `builder` field is **never checked**. The `ProposalInitValidation` struct (lines 75–85) contains no `builder` field, so there is no reference value to compare against.

The `partial_block_hash` is computed from `PartialBlockHashComponents` which includes `sequencer: SequencerContractAddress(block_info.sequencer_address)` (i.e., `builder`). Because the validator uses the proposer's `builder` value for its own re-execution, both sides compute the same `partial_block_hash`. The commitment comparison at line 244:

```rust
if built_block != received_fin.proposal_commitment {
    return Err(ValidateProposalError::ProposalFinMismatch);
}
```

passes unconditionally regardless of what `builder` contains, since the validator re-derived the commitment using the same attacker-controlled value.

By contrast, the `proposer` field is independently verified against the committee-derived expected proposer in `handle_proposal` (`crates/apollo_consensus/src/manager.rs`, line 860). No equivalent check exists for `builder`.

### Impact Explanation
`sequencer_address` is the address to which Starknet transaction fees (STRK) are paid during block execution. A malicious proposer setting `builder` to an arbitrary address causes every fee-paying transaction in the block to transfer fees to that address. This is a direct, per-block economic loss for the legitimate sequencer operator. Additionally, the committed block hash (which includes `sequencer_address`) and the L1 DA blob both record the wrong sequencer address, corrupting the authoritative on-chain view of who sequenced the block.

### Likelihood Explanation
Requires the attacker to be the elected proposer for a consensus round. In a BFT system Byzantine proposers are within the threat model (up to f = ⌊(n−1)/3⌋ Byzantine nodes). Any node that legitimately wins a proposer slot can exploit this without any additional capability. The attack is silent: no error is logged, no metric is incremented, and the block is committed normally.

### Recommendation
Add a `builder` field to `ProposalInitValidation` populated from the node's locally-configured builder address, and add an equality check in `is_proposal_init_valid`:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
```

This mirrors the existing `proposer` check in `handle_proposal` and closes the analogous gap for the builder address.

### Proof of Concept
1. A Byzantine node wins the proposer slot for height H, round R.
2. It constructs `ProposalInit` with `builder = <attacker_wallet>` (any arbitrary address).
3. The validator calls `is_proposal_init_valid` — `builder` is not in `ProposalInitValidation`, so no check fires.
4. The validator calls `convert_to_sn_api_block_info(init)` → `sequencer_address = <attacker_wallet>`.
5. The batcher executes all transactions with `sequencer_address = <attacker_wallet>`, collecting all fees there.
6. The batcher computes `partial_block_hash` including `sequencer = <attacker_wallet>`.
7. The validator computes `batcher_block_commitment` from this hash; it matches `received_fin.proposal_commitment` (the proposer also used `<attacker_wallet>`).

async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is in the future: now={}, block_timestamp_window_seconds={}, \
                 proposed={}",
                now,
                proposal_init_validation.block_timestamp_window_seconds,
                init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.starknet_version != proposal_init_validation.starknet_version {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "starknet_version mismatch: expected={:?}, proposed={:?}",
                proposal_init_validation.starknet_version, init_proposed.starknet_version
            ),
        ));
    }
    // `version_constant_commitment` is proposer-supplied (network-derived). It is not yet a real
    // commitment (see `expected_version_constant_commitment`): the only valid value is the
    // sentinel, so reject anything else. Enforcing the same value the proposer emits keeps the two
    // sides in lockstep, so a real value cannot ship on one side without the other.
    let expected_commitment = expected_version_constant_commitment();
    if init_proposed.version_constant_commitment != expected_commitment {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "version_constant_commitment mismatch: expected={expected_commitment:?}, \
                 proposed={:?}",
                init_proposed.version_constant_commitment
            ),
        ));
    }
    if !(init_proposed.height == proposal_init_validation.height
        && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
        && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
    {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            "ProposalInit validation failed".to_string(),
        ));
    }
