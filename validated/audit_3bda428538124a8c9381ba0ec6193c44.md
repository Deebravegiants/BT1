Audit Report

## Title
Stale Cross-Chain Rate in `AGETHRateReceiver` Allows Minting Excess agETH, Diluting Existing Holder Yield — (`contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/agETH/AGETHPoolV3.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the last received rate with no staleness validation despite storing a `lastUpdated` timestamp. `AGETHPoolV3.deposit()` mints agETH using this rate directly. Because `MultiChainRateProvider.updateRate()` is permissionless and LayerZero propagation is asynchronous, a structural lag window exists during which a depositor on L2 receives more agETH than the true rate warrants, extracting accrued yield from existing holders.

## Finding Description

`CrossChainRateReceiver` stores both `rate` and `lastUpdated` but `getRate()` returns the raw stored value unconditionally:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L13-16
uint256 public rate;
uint256 public lastUpdated;  // stored but never checked

// L103-105
function getRate() external view returns (uint256) {
    return rate;             // no max-age check
}
```

`AGETHPoolV3.deposit()` calls `viewSwapAgETHAmountAndFee()`, which fetches this rate and computes agETH to mint:

```solidity
// contracts/agETH/AGETHPoolV3.sol L160-168
function viewSwapAgETHAmountAndFee(uint256 amount) public view returns (uint256 agETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 agETHToETHrate = getRate();           // reads stale rate
    agETHAmount = amountAfterFee * 1e18 / agETHToETHrate;
}
```

`getRate()` on the pool delegates directly to `AGETHRateReceiver` (the `CrossChainRateReceiver`):

```solidity
// contracts/agETH/AGETHPoolV3.sol L104-106
function getRate() public view returns (uint256) {
    return IOracle(agETHOracle).getRate();
}
```

`MultiChainRateProvider.updateRate()` has no access control — anyone can call it, but must pay LayerZero fees and accept cross-chain delivery latency:

```solidity
// contracts/cross-chain/MultiChainRateProvider.sol L108-113
function updateRate() external payable nonReentrant {
    uint256 latestRate = getLatestRate();
    rate = latestRate;
    lastUpdated = block.timestamp;
    // ... sends LZ message to receivers
}
```

The exploit path: (1) agETH accrues yield on L1, rate increases from `R` to `R+δ`; (2) L2 `AGETHRateReceiver` still holds stale `R`; (3) attacker deposits `A` ETH on L2, receiving `(A-fee)*1e18/R` agETH instead of the correct `(A-fee)*1e18/(R+δ)`; (4) rate update arrives via LayerZero; (5) attacker's agETH balance, valued at `R+δ`, exceeds `A`. The excess `(A-fee)*1e18*δ/(R*(R+δ))` agETH is minted against no additional ETH, diluting all existing holders.

No existing check prevents this: there is no staleness guard in `getRate()`, no deposit circuit-breaker, and no on-chain enforcement of a maximum update interval.

## Impact Explanation

This is **theft of unclaimed yield** (High). Yield accrued by existing agETH holders — represented by the rate increase δ — is partially transferred to the depositor. The depositor's agETH, redeemable at the updated rate, is worth more ETH than they deposited. The shortfall is borne by existing holders whose redemption value is diluted. The impact scales with deposit size and δ, and is repeatable across every lag window.

## Likelihood Explanation

The lag window is structural and inherent to the cross-chain architecture. No admin compromise, governance capture, or mempool manipulation is required. An attacker only needs to monitor L1 for a rate increase and deposit on L2 before the LayerZero message arrives. The window can span minutes to hours depending on update cadence and LZ delivery latency. Because `updateRate()` is permissionless, the attacker cannot indefinitely suppress updates, but a single well-timed deposit during any lag window suffices to extract yield.

## Recommendation

1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**: define a `MAX_RATE_AGE` constant and revert if `block.timestamp - lastUpdated > MAX_RATE_AGE`.
2. **Add a staleness check in `AGETHPoolV3.deposit()`**: read `lastUpdated` from the oracle and revert if the rate is stale, as a defense-in-depth measure.
3. **Enforce a maximum update interval off-chain**: deploy a keeper that calls `updateRate()` on a fixed cadence (e.g., every hour) to minimize the exploitable window.

## Proof of Concept

```solidity
// Fork test on the L2 where AGETHPoolV3 is deployed
function testStaleRateYieldTheft() external {
    // 1. Do NOT call updateRate() — L2 rate is stale at 1.05e18
    //    True L1 rate = 1.06e18 (yield accrued, delta = 0.01e18)

    uint256 depositAmount = 100 ether;
    vm.deal(attacker, depositAmount);

    // 2. Attacker deposits at stale rate
    vm.prank(attacker);
    agETHPool.deposit{value: depositAmount}("ref");
    // agETHReceived ≈ 100e18 * 1e18 / 1.05e18 ≈ 95.238e18 agETH (minus fee)
    uint256 agETHReceived = agETH.balanceOf(attacker);

    // 3. Rate update arrives via LayerZero
    vm.prank(layerZeroEndpoint);
    agETHRateReceiver.lzReceive(srcChainId, srcAddressEncoded, 0, abi.encode(1.06e18));

    // 4. Attacker's agETH is worth more ETH than deposited
    uint256 trueRate = agETHRateReceiver.getRate(); // 1.06e18
    uint256 ethValue = agETHReceived * trueRate / 1e18;
    // ethValue ≈ 95.238e18 * 1.06e18 / 1e18 ≈ 100.952 ETH > 100 ETH deposited
    assertGt(ethValue, depositAmount, "Attacker extracted yield delta");
}
```

The assertion passes: the attacker's agETH balance valued at the updated rate exceeds their deposit, proving extraction of yield belonging to existing holders.
