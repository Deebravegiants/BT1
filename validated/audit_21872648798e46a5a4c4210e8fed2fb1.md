Audit Report

## Title
LST Deposit Limit Bypass via `assetLyingInConverter = 0` Hardcode — (`contracts/LRTDepositPool.sol`)

## Summary
`getAssetDistributionData` hardcodes `assetLyingInConverter = 0` for all non-ETH assets, so any LST transferred to `LRTConverter` via the routine `transferAssetFromDepositPool` operator call is silently excluded from `getTotalAssetDeposits(LST)`. This causes `_checkIfDepositAmountExceedesCurrentLimit` to undercount total LST holdings, allowing unprivileged depositors to push actual protocol LST holdings above the configured per-asset cap. rsETH backing is unaffected because the ETH-equivalent value is still counted through `ethValueInWithdrawal` in the ETH accounting path.

## Finding Description
`getAssetDistributionData` at line 460 of `contracts/LRTDepositPool.sol` explicitly sets `assetLyingInConverter = 0` for every non-ETH asset, with a comment acknowledging the design intent: the LST value is re-expressed as ETH and tracked via `getETHDistributionData`. However, this creates an asymmetry in the per-asset accounting path.

When an operator calls `transferAssetFromDepositPool` in `LRTConverter` (lines 140–142), two things happen: `ethValueInWithdrawal` is incremented by the oracle-priced ETH value of the LST, and the LST tokens are pulled from the deposit pool. From that point on, `getTotalAssetDeposits(stETH)` (lines 385–397) sums `assetLyingInDepositPool + assetLyingInNDCs + ... + assetLyingInConverter`, where `assetLyingInConverter` is always 0. The transferred LST is invisible to the stETH accounting path.

`_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682) and `getAssetCurrentLimit` (lines 402–409) both consume this understated total, so the effective remaining capacity is inflated by exactly the amount transferred to the converter. An unprivileged user can then call `depositAsset` and deposit up to that inflated capacity, pushing actual protocol-held stETH above the configured limit.

The ETH accounting path (`getETHDistributionData`, lines 498–499) correctly includes `ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal()`, so `_getTotalEthInProtocol` in `LRTOracle` (lines 336–343) still counts the full value, and rsETH price is unaffected.

## Impact Explanation
The per-asset deposit cap is not enforced once any LST is routed through `LRTConverter`. No user funds are lost and rsETH backing is intact, so this matches **Low — Contract fails to deliver promised returns, but doesn't lose value**. The deposit limit is a protocol-promised constraint; its silent relaxation is a concrete, demonstrable failure to deliver that promise.

## Likelihood Explanation
`transferAssetFromDepositPool` is a routine operator action performed as part of normal LST-to-ETH conversion. No malicious intent is required from the operator; every invocation passively relaxes the stETH deposit limit by the transferred amount. Once the limit is relaxed, any unprivileged depositor can exploit the inflated capacity. The bypass is repeatable and cumulative across multiple operator calls.

## Recommendation
In `getAssetDistributionData` (`contracts/LRTDepositPool.sol`, line 460), replace the hardcoded zero with the actual LST balance held by `LRTConverter`:

```solidity
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
assetLyingInConverter = IERC20(asset).balanceOf(lrtConverter);
```

This ensures `getTotalAssetDeposits(LST)` always reflects true protocol-wide LST holdings regardless of where the LST resides, keeping the deposit limit check accurate. Care should be taken to avoid double-counting in `_getTotalEthInProtocol`: if `assetLyingInConverter` is now non-zero for LSTs, the `ethValueInWithdrawal` component in `getETHDistributionData` should be adjusted to exclude the ETH-equivalent of LSTs still held in the converter (i.e., only count LSTs that have already been converted to ETH).

## Proof of Concept
```solidity
// Setup: depositLimit[stETH] = 100_000e18
// State:  90_000e18 stETH in DepositPool

// Step 1 — routine operator action
lrtConverter.transferAssetFromDepositPool(stETH, 10_000e18);
// ethValueInWithdrawal += 10_000e18 * price / 1e18
// stETH balance of DepositPool drops by 10_000e18

// Step 2 — getTotalAssetDeposits now undercounts
uint256 total = depositPool.getTotalAssetDeposits(stETH);
// returns 80_000e18 (converter balance excluded)

// Step 3 — getAssetCurrentLimit is inflated
uint256 limit = depositPool.getAssetCurrentLimit(stETH);
// returns 20_000e18 (should be 10_000e18)

// Step 4 — unprivileged user deposits 20_000 stETH, succeeds
depositPool.depositAsset(stETH, 20_000e18, 0, "");
// actual stETH in protocol = 80_000 (pool) + 10_000 (converter) + 20_000 (new) = 110_000
// configured limit = 100_000 — limit breached
```

A Foundry fork test can confirm this by: (1) deploying/forking the protocol, (2) seeding 90,000 stETH into the deposit pool, (3) calling `transferAssetFromDepositPool(stETH, 10_000e18)` as the asset-transfer role, (4) asserting `getTotalAssetDeposits(stETH) == 80_000e18`, and (5) asserting that a subsequent `depositAsset(stETH, 20_000e18, ...)` succeeds despite the 100,000 stETH cap.
