### Title
Missing Metaplex `mint` Field Validation in `parse_metadata_account` Allows Cross-Token Metadata Injection — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

### Summary

`LogMetadata::process` reads name/symbol from a Metaplex account whose address is taken from the Token-2022 `metadata_pointer` extension, but never verifies that the Metaplex account's embedded `mint` field matches `self.mint.key()`. An unprivileged attacker can create a Token-2022 mint with `decimals=0` and set its `metadata_pointer` to any existing legitimate token's Metaplex PDA, causing the emitted Wormhole payload to carry the victim token's name/symbol paired with the attacker mint's `decimals=0`. NEAR then registers a token with a mismatched identity, breaking the invariant that name, symbol, and decimals in a `log_metadata` payload must all describe the same token.

---

### Finding Description

**Root cause — `parse_metadata_account` (lines 72–90):**

```rust
// log_metadata.rs:78-86
require_keys_eq!(
    metadata.key(),
    address,                          // address = metadata_pointer.metadata_address.0
    ErrorCode::InvalidTokenMetadataAddress,
);
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let metadata = MplMetadata::try_deserialize(&mut data.as_ref())?;
    Ok((metadata.name.clone(), metadata.symbol.clone()))  // ← no check of metadata.mint
}
``` [1](#0-0) 

The only guard is `metadata.key() == address`. There is no assertion that `MplMetadata.mint == self.mint.key()`. The Metaplex `MetadataAccount` struct carries a `mint` field that identifies which mint the metadata belongs to; it is never read here.

**Payload construction (lines 130–134):**

```rust
let payload = LogMetadataPayload {
    token:    self.mint.key(),      // mint A's pubkey
    name:     name.trim_end_matches('\0').to_string(),   // token B's name
    symbol:   symbol.trim_end_matches('\0').to_string(), // token B's symbol
    decimals: self.mint.decimals,   // 0  ← always from mint A
}
``` [2](#0-1) 

**Attack setup:**

1. Attacker creates Token-2022 mint A with `decimals=0` and `metadata_pointer` set to token B's Metaplex PDA (a deterministic address the attacker can compute for any existing token).
2. Attacker calls `log_metadata` passing mint A as `mint` and token B's Metaplex PDA as the optional `metadata` account.
3. In `process()`, the Token-2022 branch fires (line 92), reads the `MetadataPointer` extension (line 98), finds `metadata_address != mint_A.key()` (line 99 false), enters the "third-party metadata" branch (line 104–106), and calls `parse_metadata_account(token_B_metaplex_PDA)`.
4. `parse_metadata_account` checks `metadata.key() == token_B_metaplex_PDA` — **passes**, because the attacker supplied exactly that account. It then checks `owner == MetaplexID` — **passes**, because it is a real Metaplex account. It deserializes and returns token B's name/symbol.
5. The payload is assembled with `name`/`symbol` from token B and `decimals=0` from mint A. [3](#0-2)

### Impact Explanation

On NEAR, `deploy_token_callback` receives the `LogMetadata` prover result and calls `deploy_token_internal` with `metadata.decimals` taken verbatim from the payload:

```rust
// near/omni-bridge/src/lib.rs:1169-1176
self.deploy_token_internal(
    chain,
    &metadata.token_address,
    BasicMetadata {
        name: metadata.name,
        symbol: metadata.symbol,
        decimals: metadata.decimals,   // ← 0, from mint A
    },
    attached_deposit,
)
``` [4](#0-3) 

`deploy_token_internal` stores both `decimals` and `origin_decimals` as 0:

```rust
// near/omni-bridge/src/lib.rs:2419-2424
self.add_token(
    &token_id,
    token_address,
    metadata.decimals,   // 0
    metadata.decimals,   // 0
);
``` [5](#0-4) 

The `normalize_amount` / `denormalize_amount` functions use `origin_decimals - decimals` as the scaling exponent:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [6](#0-5) 

With both set to 0, the exponent is 0 and no normalization occurs. Any user who bridges tokens using mint A (which they may identify by token B's name/symbol displayed in a UI) will have amounts processed with 0-decimal semantics instead of the real token's decimal count, causing direct value loss proportional to `10^(real_decimals)` per unit transferred.

---

### Likelihood Explanation

- `log_metadata` is fully permissionless — no role, no fee, no whitelist.
- Creating a Token-2022 mint with an arbitrary `metadata_pointer` is a standard, documented SPL operation requiring no special privileges.
- The Metaplex PDA for any token is deterministically computable off-chain.
- The attack requires only two transactions (create mint, call `log_metadata`) and is repeatable for any target token.

---

### Recommendation

In `parse_metadata_account`, after deserializing the Metaplex account, assert that its `mint` field matches `self.mint.key()`:

```rust
let mpl = MplMetadata::try_deserialize(&mut data.as_ref())?;
require_keys_eq!(
    mpl.mint,
    self.mint.key(),
    ErrorCode::InvalidTokenMetadataAddress,
);
Ok((mpl.name.clone(), mpl.symbol.clone()))
```

This ensures that the name/symbol read from a third-party Metaplex account actually belong to the mint being logged, closing the cross-token metadata injection path.

### Proof of Concept

```
1. Deploy Token-2022 mint A: decimals=0, metadata_pointer → Metaplex PDA of token B
   (token B is any legitimate token with, e.g., 18 decimals)

2. Compute token B's Metaplex PDA:
   find_program_address(
       [b"metadata", MetaplexID, token_B_mint.as_ref()],
       MetaplexID
   )

3. Call log_metadata {
       mint:     mint_A,
       metadata: Some(token_B_metaplex_PDA),
       token_program: Token-2022,
       ...
   }

4. Assert emitted Wormhole payload:
   - token   == mint_A.key()
   - name    == token_B.name    (e.g. "USD Coin")
   - symbol  == token_B.symbol  (e.g. "USDC")
   - decimals == 0              (from mint_A, not token_B)

5. Submit VAA to NEAR deploy_token → NEAR registers mint_A
   with USDC name/symbol but decimals=0.
   Any bridge transfer using mint_A is normalized with 10^0 = 1
   instead of 10^6, causing 1,000,000× accounting error.
```
