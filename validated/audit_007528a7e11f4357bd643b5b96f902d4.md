### Title
Permissionless `updateRSETHPrice()` Enables Selective Deposit Timing to Steal Yield from rsETH Holders - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle.updateRSETHPrice()` is a permissionless, fully deterministic function. Because `LRTDepositPool` mints rsETH using the stored (potentially stale) `rsETHPrice`, any depositor can simulate the price-update outcome entirely from public on-chain state and selectively deposit before or after the update to always obtain the most favorable exchange rate, stealing accrued yield from existing rsETH holders.

### Finding Description
`LRTOracle.updateRSETHPrice()` is declared `public whenNotPaused`: [1](#0-0) 

Its output — the new `rsETHPrice` — is entirely deterministic, depending only on publicly readable on-chain state: `totalETHInProtocol` (derived from `getTotalAssetDeposits` per asset), `rsETH.totalSupply()`, and `lrtConfig.protocolFeeInBPS()`.

`LRTDepositPool.getRsETHAmountToMint()` computes the rsETH to mint as: [2](#0-1) 

It uses `lrtOracle.rsETHPrice()` — the **stored** price — not a freshly computed value. The stored price is only updated when `updateRSETHPrice()` is explicitly called.

An attacker can:
1. Read `rsETHPrice` (stored, potentially stale)
2. Read all inputs to `_updateRsETHPrice()` on-chain: `getTotalAssetDeposits()` for each supported asset, `rsETH.totalSupply()`, `lrtConfig.protocolFeeInBPS()`
3. Simulate the exact new price using the same arithmetic as `_updateRsETHPrice()`: [3](#0-2) 

4. If `simulatedNewPrice > storedPrice` (rewards have accrued → price should rise), deposit **before** calling `updateRSETHPrice()` to receive more rsETH at the stale lower price
5. If `simulatedNewPrice < storedPrice` (price should fall), call `updateRSETHPrice()` first, then deposit at the lower price

The attacker always chooses the path that maximizes their rsETH allocation. This is directly analogous to the penguin game's predictable randomness: the outcome is fully simulatable before execution, and the attacker only proceeds when favorable.

An additional amplifier: the `pricePercentageLimit` guard prevents non-managers from calling `updateRSETHPrice()` when the price increase exceeds the configured threshold: [4](#0-3) 

When rewards are large, this guard blocks the price update for non-managers, causing the stale price to persist even longer and widening the exploitation window.

### Impact Explanation
The attacker receives more rsETH than the protocol intends for their deposit amount. The excess rsETH is backed by the same ETH pool as existing holders' rsETH, diluting their share of protocol assets. This constitutes **theft of unclaimed yield** from existing rsETH holders. The magnitude scales with deposit size and price staleness: a 0.1% daily reward rate on a 1,000 ETH deposit yields approximately 1 ETH of excess rsETH per cycle, repeatable indefinitely.

### Likelihood Explanation
The attack requires no privileged access, no special infrastructure, and no external dependencies. All required inputs are readable from public contract variables in a single view call. The attack is repeatable every time rewards accrue and `rsETHPrice` has not yet been updated. Likelihood is **high**.

### Recommendation
Force a price update atomically before computing the rsETH mint amount inside `_beforeDeposit()`, or replace the stored `rsETHPrice` read in `getRsETHAmountToMint()` with a freshly computed value. Alternatively, restrict `updateRSETHPrice()` to an authorized keeper and update on a fixed schedule to minimize the staleness window.

### Proof of Concept
```solidity
contract SelectiveDepositAttack {
    ILRTOracle   oracle;
    ILRTDepositPool pool;
    IRSETH       rsETH;
    ILRTConfig   config;
    address constant ETH_TOKEN = 0xEeeeeEeeeEeEeeEeEeEeeEEEeeeeEeeeeeeeEEeE;

    function executeAttack() external payable {
        uint256 depositAmount = msg.value;

        // Step 1: Read stored price
        uint256 storedPrice = oracle.rsETHPrice();

        // Step 2: Simulate new price from fully public on-chain state
        uint256 totalETH = pool.getTotalAssetDeposits(ETH_TOKEN);
        // (add other supported assets multiplied by their oracle prices)
        uint256 supply = rsETH.totalSupply();
        uint256 prevTVL = supply * storedPrice / 1e18;
        uint256 fee = totalETH > prevTVL
            ? ((totalETH - prevTVL) * config.protocolFeeInBPS()) / 10_000
            : 0;
        uint256 simulatedNewPrice = (totalETH - fee) * 1e18 / supply;

        // Step 3: Selectively execute for maximum rsETH
        if (simulatedNewPrice > storedPrice) {
            // Deposit at stale lower price → receive more rsETH
            pool.depositETH{value: depositAmount}(0, "");
            oracle.updateRSETHPrice(); // update after to lock in gain
        } else {
            // Update first to lower the price, then deposit
            oracle.updateRSETHPrice();
            pool.depositETH{value: depositAmount}(0, "");
        }
    }
}
```
