### Title
Attacker Can Permanently DoS `verify_foreign_transaction` for Any Specific Foreign Transaction by Saturating the Caller-Agnostic Fan-Out Queue - (File: `crates/contract/src/pending_requests.rs`)

### Summary

The `push_pending_yield` function panics with `PendingRequestQueueFull` when the fan-out queue for a given request key reaches `MAX_PENDING_REQUEST_FAN_OUT = 128`. For `verify_foreign_transaction`, the request key is **caller-agnostic** — it is derived solely from the foreign chain parameters (`domain_id`, `payload_version`, `tx_id`, chain, extractors), not from the submitting account. An unprivileged attacker can saturate the queue for any specific foreign transaction with 128 cheap submissions, permanently blocking all subsequent legitimate callers from enqueuing the same request until the queue drains. Because the attacker can immediately re-saturate after each drain, the DoS can be sustained indefinitely at negligible cost.

### Finding Description

`push_pending_yield` in `crates/contract/src/pending_requests.rs` enforces a hard cap:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
}
``` [1](#0-0) 

When the queue is full, any new submission for the same key panics and the caller's transaction is reverted. The caller loses gas but the 1 yoctoNEAR deposit is refunded on revert.

For `sign()` and `request_app_private_key()`, the request key includes `predecessor_account_id`, so each caller has an independent queue slot:

```rust
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // caller-specific
    &request.path,
);
``` [2](#0-1) 

For `verify_foreign_transaction()`, however, the key is built from `args_into_verify_foreign_tx_request`, which produces a `VerifyForeignTransactionRequest` containing only `(domain_id, payload_version, ForeignChainRpcRequest)` — **no caller identity**. The contract's own unit test confirms this explicitly:

```rust
// And: caller bob submits the identical request — a different account would today
// be blocked from receiving a response by alice's submission.
// ...
// Then: both yields are queued under the single (caller-agnostic) request key.
assert_eq!(
    contract.pending_verify_foreign_tx_requests.get(&request).map(|q| q.len()),
    Some(2),
    "duplicate foreign-tx requests from different callers should fan out",
);
``` [3](#0-2) 

Because the key is caller-agnostic, any account can consume all 128 slots for any specific foreign transaction. Once the cap is reached, every subsequent `verify_foreign_transaction` call for that tx panics:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    // ...
    let request = args_into_verify_foreign_tx_request(request);
    // ...
    self.enqueue_yield_request(
        method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
        callback_args,
        callback_gas,
        move |this, id| this.add_verify_foreign_tx_request(request, id),
    );
}
``` [4](#0-3) 

The queue drains only when MPC nodes call `respond_verify_foreign_tx`, which triggers `resolve_yields_for` and removes the map entry. The attacker can immediately re-saturate the queue after each drain, sustaining the DoS indefinitely.

### Impact Explanation

**Medium.** This is a request-lifecycle manipulation that breaks the production safety invariant: *any user who submits a valid `verify_foreign_transaction` request with sufficient deposit and gas must eventually receive a response.* An attacker can violate this invariant for any specific foreign transaction (identified by `tx_id` + chain + extractors) without any privileged access. In a bridge context, this can:

- Permanently block a specific cross-chain withdrawal or settlement from being verified and signed.
- Prevent time-sensitive bridge operations (e.g., liquidations, atomic swaps) from completing.
- Enable indirect financial gain by selectively blocking a competitor's bridge operations.

The impact does not require network-level DoS, operator misconfiguration, or threshold collusion.

### Likelihood Explanation

**Medium.** The attack requires submitting 128 `verify_foreign_transaction` transactions. Each costs only gas (≈10 Tgas per call) plus 1 yoctoNEAR deposit. On NEAR, 128 transactions cost on the order of a few cents. The attacker can automate re-saturation after each drain. The attack is fully permissionless and requires no special access. The only friction is the need to monitor the chain and re-submit quickly after each drain cycle.

### Recommendation

The root cause is that the fan-out queue cap is shared across all callers for `verify_foreign_transaction`. Two mitigations:

1. **Per-caller slot accounting**: Track how many queue slots each `predecessor_account_id` holds for a given request key and enforce a per-caller sub-limit (e.g., 1–2 slots per caller). This preserves the fan-out feature while preventing any single account from monopolizing the queue.

2. **Caller-keyed secondary map**: Store a `(request_key, caller)` → `YieldIndex` mapping alongside the fan-out map, and reject duplicate submissions from the same caller for the same request. This is the direct analog of replacing `safeApprove` with `safeIncreaseAllowance` — instead of a single shared state that any caller can exhaust, each caller manages their own independent slot.

### Proof of Concept

```
1. Attacker identifies a target foreign tx: Bitcoin tx_id = [0xAB; 32], domain_id = 0.

2. Attacker submits 128 verify_foreign_transaction calls with identical args
   (tx_id=[0xAB;32], domain_id=0, extractors=[BlockHash]), each with 1 yoctoNEAR deposit.
   Cost: ~128 × gas ≈ cents. All 128 succeed; queue length = 128.

3. Legitimate user (victim) submits verify_foreign_transaction with the same args.
   → push_pending_yield panics: "Pending-request queue is full (limit: 128)"
   → Victim's transaction reverts; they lose gas but get deposit back.
   → Victim cannot verify the foreign transaction.

4. MPC nodes observe the request and call respond_verify_foreign_tx.
   → resolve_yields_for drains all 128 attacker yields; map entry removed.

5. Attacker immediately re-submits 128 calls (automated).
   → Queue is full again before victim can retry.

6. Steps 3–5 repeat indefinitely. Victim is permanently blocked from verifying
   this specific foreign transaction.
```

    #[test]
    fn add_signature_request__should_panic_when_pending_queue_is_full() {
        // Given: a contract with a queue already at the fan-out cap for some request key.
        let (context, mut contract, _) = basic_setup(Curve::Secp256k1, &mut OsRng);
        let signature_request = SignatureRequest::new(
            DomainId::default(),
            Payload::from_legacy_ecdsa([3u8; 32]),
            &context.predecessor_account_id,
            "m/44'\''/60'\''/0'\''/0/0",
        );
        for i in 0..MAX_PENDING_REQUEST_FAN_OUT {
            contract.add_signature_request(signature_request.clone(), [i; 32]);
        }
        assert_eq!(
            contract
                .pending_signature_requests
                .get(&signature_request)
                .map(|q| q.len()),
            Some(usize::from(MAX_PENDING_REQUEST_FAN_OUT)),
        );

        // When: one more append is attempted.
        let result = panic::catch_unwind(panic::AssertUnwindSafe(|| {
            contract.add_signature_request(signature_request.clone(), [0xff; 32]);
        }));

        // Then: it panics with the typed cap-exceeded error and leaves the queue untouched.
        let err = result.expect_err("appending past the cap should panic");
        let msg = err
            .downcast_ref::<String>()
            .map(String::as_str)
            .or_else(|| err.downcast_ref::<&str>().copied())
            .unwrap_or_default();
        assert!(
            msg.contains("Pending-request queue is full"),
            "unexpected panic message: {msg}",
        );
        assert_eq!(
            contract
                .pending_signature_requests
                .get(&signature_request)
                .map(|q| q.len()),
            Some(usize::from(MAX_PENDING_REQUEST_FAN_OUT)),
            "queue should not have grown past the cap",
        );
