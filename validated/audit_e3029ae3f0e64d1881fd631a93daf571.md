### Title
Dijkstra Batch Minimum Fee Check Excludes Sub-Transaction Reference Script Costs - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the minimum fee validation for a batch transaction (`TopTx` with embedded sub-transactions) only accounts for reference scripts used by the top-level transaction. Reference scripts used exclusively by sub-transactions are not included in the fee calculation, allowing an attacker to submit batches with large sub-transaction reference scripts while paying a fee that only covers the top-level transaction's reference script cost.

---

### Finding Description

The Dijkstra era introduces sub-transactions (`SubTx`) embedded inside a top-level transaction (`TopTx`). Only the top-level transaction carries a fee field; sub-transactions are feeless. The minimum fee for the batch is therefore charged entirely at the top level and must cover the resource cost of the entire batch, including reference script deserialization for all sub-transactions.

The `EraUTxO` instance for `DijkstraEra` delegates `getMinFeeTxUtxo` directly to `getConwayMinFeeTxUtxo`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs, line 141
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` computes the reference-script size component using `txNonDistinctRefScriptsSize`, which only inspects the top-level transaction's own inputs and reference inputs:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx

txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [2](#0-1) 

Sub-transactions' `referenceInputsTxBodyL` and `inputsTxBodyL` are never consulted. The Dijkstra UTXO rule then enforces the fee using this incomplete calculation:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs, line 373
{- minfee pp txTop utxo₀ ≤ txfee txb -}
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
``` [3](#0-2) 

Meanwhile, the codebase already contains `batchNonDistinctRefScriptsSize`, which correctly aggregates reference script sizes across the top-level transaction **and all sub-transactions**:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs, lines 264-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [4](#0-3) 

This function exists but is **not used** in the fee validation path. The fee check is applied only to the partial (top-level) reference script cost, not the total batch cost — a direct structural analog to the Launchpad slippage bug where the check was applied to the bonding-curve portion of the cost only.

---

### Impact Explanation

The reference-script fee (`minFeeRefScriptCostPerByte` / tiered pricing) was introduced specifically to prevent DDoS attacks where an attacker submits transactions that are cheap to include but expensive for nodes to validate due to large script deserialization. By placing large reference scripts exclusively in sub-transactions, an attacker can force nodes to deserialize arbitrarily large scripts while paying a fee that only reflects the top-level transaction's (potentially zero) reference script usage. This bypasses the intended resource-limit protection and allows attacker-controlled transactions to modify fees outside design parameters.

This matches the **Medium** allowed impact: *"Attacker-controlled transactions… exceed intended validation limits or modify fees… outside design parameters."*

---

### Likelihood Explanation

Any unprivileged transaction submitter can craft a `DijkstraEra` batch transaction with sub-transactions that reference large scripts. No special role, key, or governance majority is required. The attacker only needs to construct a valid top-level transaction with sub-transactions whose `referenceInputsTxBodyL` point to UTxO entries containing large Plutus scripts. The top-level fee field can be set to the minimum fee computed by `getConwayMinFeeTxUtxo` (which ignores sub-transaction reference scripts), and the transaction will pass the `validateFeeTooSmallUTxO` check.

---

### Recommendation

Override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance to use `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
-- In eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo pparams tx utxo =
    getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx
```

This ensures the minimum fee check covers the full reference script cost of the entire batch, consistent with the intent of the tiered reference-script pricing mechanism.

---

### Proof of Concept

1. Construct a `DijkstraEra` top-level transaction with no reference inputs of its own.
2. Embed one or more sub-transactions whose `referenceInputsTxBodyL` point to UTxO entries containing large Plutus scripts (e.g., 25 KiB each).
3. Set the top-level `feeTxBodyL` to the value returned by `getConwayMinFeeTxUtxo` (which ignores sub-transaction reference scripts — effectively the base `a*size + b` fee with zero reference-script surcharge).
4. Submit the transaction. `validateFeeTooSmallUTxO` passes because it calls `getConwayMinFeeTxUtxo`, which only measures the top-level transaction's reference scripts (zero bytes).
5. Nodes must deserialize the large reference scripts from the sub-transactions during validation, incurring the full deserialization cost, while the attacker paid only the base fee — bypassing the tiered reference-script fee protection entirely.
