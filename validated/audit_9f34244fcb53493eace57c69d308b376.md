### Title
Fee-on-Transfer Token Deposit Over-Mints rsETH, Causing Protocol Insolvency - (File: contracts/LRTDepositPool.sol)

### Summary

`LRTDepositPool.depositAsset()` calculates `rsethAmountToMint` using the caller-supplied `depositAmount` **before** the `safeTransferFrom` call. If the deposited asset is a fee-on-transfer token, the contract receives fewer tokens than `depositAmount`, yet mints rsETH as if the full nominal amount arrived. This over-mints rsETH relative to actual backing assets, diluting all existing rsETH holders and driving the protocol toward insolvency.

### Finding Description

`depositAsset` follows this sequence:

1. `_beforeDeposit(asset, depositAmount, minRSETHAmountExpected)` — computes `rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount)` using the **nominal** `depositAmount`, then checks `rsethAmountToMint >= minRSETHAmountExpected`.
2. `IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount)` — the actual transfer; a fee-on-transfer token silently delivers `depositAmount − fee` to the contract.
3. `_mintRsETH(rsethAmountToMint)` — mints rsETH calculated from the **full** `depositAmount`, not the reduced amount actually received.

```
// contracts/LRTDepositPool.sol  lines 110-115
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
// ↑ rsethAmountToMint based on depositAmount (nominal)

IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
// ↑ contract receives depositAmount − transferFee  (actual)

_mintRsETH(rsethAmountToMint);
// ↑ mints for depositAmount, not for (depositAmount − transferFee)
```

`getRsETHAmountToMint` at line 520 confirms the calculation uses the raw `amount` argument:

```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The same pattern is present in `RSETHPoolV3.deposit(address token, …)` (line 284–290) and `RSETHPool.deposit(address token, …)` (line 296–302), where `viewSwapRsETHAmountAndFee(amount, token)` is called with the nominal `amount` after `safeTransferFrom` has already silently reduced the actual receipt.

### Impact Explanation

Every deposit of a fee-on-transfer LST mints more rsETH than the assets backing it. Because rsETH price is computed from total protocol TVL divided by total rsETH supply (`LRTOracle`), each such deposit lowers the per-share backing. Existing rsETH holders suffer permanent, compounding dilution. If the fee-on-transfer token constitutes a significant share of deposits, the protocol becomes insolvent: the rsETH supply exceeds the ETH-equivalent value of all underlying assets.

**Impact: Critical — Protocol insolvency.**

### Likelihood Explanation

The `LRTConfig.addNewSupportedAsset()` function allows governance to add any ERC-20 as a supported LST. Several real LSTs (e.g., stETH) have rebasing or transfer-tax mechanics. The code contains no guard against fee-on-transfer tokens, and the deposit path is fully permissionless once an asset is listed. Any user can trigger the over-mint immediately after a fee-on-transfer token is added.

### Recommendation

Measure the actual amount received by comparing the contract's token balance before and after `safeTransferFrom`, and use that delta — not the caller-supplied `depositAmount` — for both the `minRSETHAmountExpected` check and the rsETH mint calculation:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
_mintRsETH(rsethAmountToMint);
```

### Proof of Concept

1. Governance adds a fee-on-transfer LST (e.g., 1% transfer fee) as a supported asset via `LRTConfig.addNewSupportedAsset()`.
2. Attacker calls `depositAsset(feeToken, 1000e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` for `1000e18` and passes the `minRSETHAmountExpected` check.
4. `safeTransferFrom` delivers only `990e18` to the contract (1% fee deducted).
5. `_mintRsETH` mints rsETH equivalent to `1000e18` of the asset.
6. The protocol now holds `990e18` tokens but has issued rsETH backed by `1000e18` — a 1% over-issuance per deposit.
7. Repeated deposits compound the insolvency; honest rsETH holders redeem for less ETH than their shares represent. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```text
    function _beforeDeposit(
        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
```
