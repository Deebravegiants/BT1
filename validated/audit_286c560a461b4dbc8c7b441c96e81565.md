### Title
Fee-on-Transfer LST Causes rsETH Over-Minting in `depositAsset` - (File: `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool.depositAsset()` computes and mints rsETH based on the caller-supplied `depositAmount` parameter, not the actual tokens received after the `safeTransferFrom`. For any fee-on-transfer LST added as a supported asset, the protocol receives fewer tokens than it accounts for, inflating rsETH supply relative to actual backing and causing protocol insolvency.

---

### Finding Description

In `depositAsset()`, the rsETH mint amount is calculated from `depositAmount` in `_beforeDeposit()` **before** the transfer executes:

```solidity
// contracts/LRTDepositPool.sol:111-115
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);

// interactions
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
``` [1](#0-0) 

Inside `_beforeDeposit`, the mint amount is derived directly from the parameter:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
``` [2](#0-1) 

If `asset` is a fee-on-transfer token, `safeTransferFrom` delivers `depositAmount - fee` to the contract, but `rsethAmountToMint` was already computed from the full `depositAmount`. The actual received amount is never measured (no before/after balance check), and `_mintRsETH` mints the over-inflated amount unconditionally.

The same pattern exists in `RSETHPoolV3.deposit(address token, uint256 amount, ...)` on L2:

```solidity
// contracts/pools/RSETHPoolV3.sol:284-290
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
...
wrsETH.mint(msg.sender, rsETHAmount);
``` [3](#0-2) 

---

### Impact Explanation

**Critical â€” Protocol insolvency.**

The rsETH/ETH exchange rate is computed by `LRTOracle._getTotalEthInProtocol()`, which sums actual on-chain token balances across the deposit pool, NDCs, and EigenLayer strategies:

```solidity
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [4](#0-3) 

Because `getTotalAssetDeposits` reads real balances while rsETH supply was inflated by the over-mint, the computed `rsETHPrice` will be lower than it should be. Every existing rsETH holder is diluted. Repeated exploitation (each deposit with a fee-on-transfer LST mints excess rsETH) compounds the insolvency until the protocol's rsETH is worth materially less than its backing.

---

### Likelihood Explanation

**Medium.** The protocol is explicitly designed to support arbitrary ERC20 LSTs added by the admin via `LRTConfig`. The `depositAsset` function enforces only that the asset is in the supported list (`onlySupportedERC20Token`). No whitelist or fee-on-transfer guard exists. Any future LST with a transfer fee â€” or a rebasing token whose `transferFrom` delivers a dust-rounded amount â€” triggers this path. The entry point is fully permissionless for any depositor once such an asset is listed.

---

### Recommendation

Measure the actual received amount using a before/after balance check, and use that value for both the mint calculation and the deposit limit check:

```solidity
function depositAsset(...) external nonReentrant whenNotPaused onlySupportedERC20Token(asset) {
    uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
    IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
    uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

    uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);

    emit AssetDeposit(msg.sender, asset, actualReceived, rsethAmountToMint, referralId);
}
```

Apply the same fix to `RSETHPoolV3.deposit(address token, uint256 amount, ...)`.

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported, and add a runtime guard (e.g., verify `balanceOf` delta equals `depositAmount` after transfer, reverting otherwise).

---

### Proof of Concept

1. Admin adds `FoTLST` (a fee-on-transfer LST charging 5% per transfer) as a supported asset with a valid price oracle.
2. Attacker calls `depositAsset(FoTLST, 1000e18, 0, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` based on `1000e18`.
4. `safeTransferFrom` delivers only `950e18` to the deposit pool.
5. `_mintRsETH` mints rsETH equivalent to `1000e18` worth of FoTLST.
6. The oracle's next `updateRSETHPrice()` call reads actual balance (`950e18`) but divides by the inflated rsETH supply â€” rsETH price drops, diluting all holders.
7. Repeating this across many deposits progressively drains value from all rsETH holders. [5](#0-4)
