Audit Report

## Title
Missing L2 Sequencer Uptime Check Allows Stale-Price Swaps After Sequencer Recovery - (File: `smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` implement a timestamp-only staleness check (`_isStale`) that compares oracle `refTime` against `block.timestamp` within `MAX_TIME_DELTA` (up to 7 days). Neither contract checks whether the L2 sequencer is currently up or has recently recovered from downtime. After a sequencer outage shorter than `MAX_TIME_DELTA`, pre-outage oracle prices pass the staleness check and reach `MetricOmmPool.swap`, enabling a trader to execute at prices that no longer reflect current market conditions and extract value from LP reserves.

## Finding Description

Both L2 price providers implement an identical staleness check:

```solidity
// PriceProviderL2.sol L135-150, ProtectedPriceProviderL2.sol L138-153
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;
}
```

The constructor enforces `MAX_TIME_DELTA ∈ (0, 7 days]`:

```solidity
// PriceProviderL2.sol L92
if (_maxTimeDelta == 0 || _maxTimeDelta > 7 days) revert MaxTimeDeltaOutOfBounds();
```

`_getBidAndAskPrice()` in `PriceProviderL2` (L208-217) and `_computeBidAsk()` in `ProtectedPriceProviderL2` (L203-209) both gate on `_isStale` only — no sequencer liveness check exists anywhere in either contract.

When an L2 sequencer goes down: (1) no transactions process on-chain; (2) oracle publishers cannot push new price updates; (3) the last oracle update with its `refTime` and price remains frozen. When the sequencer recovers, `block.timestamp` resumes, but `refTime` is from before the outage. If `(block.timestamp - refTime) ≤ MAX_TIME_DELTA`, the staleness check passes and `_getBidAndAskPrice()` returns the pre-outage bid/ask.

The swap path in `MetricOmmPool.swap` (L228-248) consumes this stale quote without any additional guard:

```
swap → _getBidAndAskPriceX64() → IPriceProvider.getBidAndAskPrice()  // returns stale bid/ask
     → SwapMath.midAndSpreadFeeX64FromBidAsk(bid, ask)
     → _executeSwap(...)  // settles at stale price
```

The registry ABI (`smart-contracts-poc/contract-registry/versions/registry.json`) contains `ChainlinkVerifierL2` with `sequencerUptimeFeed` and `GRACE_PERIOD` fields, confirming the team is aware of the sequencer uptime pattern — but neither L2 price provider integrates it.

## Impact Explanation

A trader can call `MetricOmmPool.swap` immediately after sequencer recovery using a bid/ask that was valid before the outage but is now stale. If the real market price moved during the outage (e.g., a 5% drop), the trader buys token0 at the pre-crash ask price and sells at the real market price on a CEX, extracting value directly from LP reserves. This is a direct loss of user principal (LP assets) matching the allowed impact gate: **bad-price execution — stale bid/ask quote reaches a pool swap**.

## Likelihood Explanation

L2 sequencer outages are documented historical events on Arbitrum, Optimism, and Base. No privileged access is required — any public caller can invoke `swap` the moment the sequencer recovers. The exploitation window equals the outage duration. For pools using Chainlink feeds with `MAX_TIME_DELTA` set to the feed's heartbeat (1 h or 24 h), any outage shorter than that window is exploitable. The attack is repeatable across every sequencer recovery event.

## Recommendation

Add a sequencer uptime check inside `_getBidAndAskPrice()` (and `_computeBidAsk()` in `ProtectedPriceProviderL2`) following the pattern already present in the registry's `ChainlinkVerifierL2`:

```solidity
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600;

function _checkSequencer() internal view {
    if (address(sequencerUptimeFeed) == address(0)) return;
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer != 0 || block.timestamp - startedAt < GRACE_PERIOD)
        revert SequencerDown();
}
```

Call `_checkSequencer()` at the top of `_getBidAndAskPrice()` / `_computeBidAsk()` before the oracle read. If the sequencer is down or within the grace period, revert with `FeedStalled` (or a dedicated `SequencerDown` error), blocking all swaps until prices are fresh.

## Proof of Concept

```
Setup:
  - Deploy PriceProviderL2 on Arbitrum with MAX_TIME_DELTA = 3600 (1 hour)
  - Pool has ETH/USDC, oracle price = $3000

Attack:
  T=0:    Sequencer goes down. Last oracle refTime = T=0, price = $3000.
  T=0..D: Real ETH price drops to $2800 off-chain. No L2 txs possible.
  T=D:    Sequencer recovers (D = 30 min < MAX_TIME_DELTA = 1 hour).
          _isStale(T=0, D=1800, 3600, FUTURE_TOL) → (1800 - 0) = 1800 ≤ 3600 → NOT stale.
          getBidAndAskPrice() returns bid/ask based on $3000.
  T=D+1:  Attacker calls pool.swap(zeroForOne=true) — buys ETH at $3000 ask.
          Real market price is $2800. Attacker sells on CEX for $2800.
          Pool LPs absorb the $200/ETH loss.

Foundry fork test: fork Arbitrum at a block just after a historical sequencer recovery,
deploy PriceProviderL2 with MAX_TIME_DELTA=3600, warp block.timestamp to recovery time,
call getBidAndAskPrice() — confirm it returns non-zero bid/ask despite stale refTime.
```
