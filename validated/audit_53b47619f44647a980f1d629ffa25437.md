### Title
Missing NEAR Contract/Network Binding in `ForeignTxSignPayload` Enables Cross-Fork Replay of Foreign-Chain Verification Signatures - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary

`ForeignTxSignPayload::compute_msg_hash()` produces a hash that contains only the foreign-chain request and extracted values. It omits any NEAR-specific context — the contract account ID and the NEAR network ID. After a NEAR hard fork, both forks share identical key material and identical pending-request state, so a `VerifyForeignTransactionResponse` produced on one fork is cryptographically valid and structurally accepted on the other, enabling cross-fork replay that triggers invalid bridge execution.

### Finding Description

The canonical signed payload for foreign-chain transaction verification is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload))
```

where `ForeignTxSignPayload::V1` contains only `ForeignChainRpcRequest` and `Vec<ExtractedValue>`: [1](#0-0) 

Neither the NEAR contract account ID nor the NEAR network/chain ID is included in the serialized payload. The `respond_verify_foreign_tx` handler verifies only that `response.signature` is valid for `response.payload_hash` under the domain's public key, and that the `request` key exists in `pending_verify_foreign_tx_requests`: [2](#0-1) 

No check binds `payload_hash` to the specific NEAR contract instance or network. The domain ID is taken from the stored pending request, not from the signed payload itself.

### Impact Explanation

After a NEAR hard fork:

1. Both forks share identical contract state (including all pending `verify_foreign_transaction` requests) and identical MPC key material (generated before the fork).
2. The MPC network produces a `VerifyForeignTransactionResponse` on fork A: `{payload_hash, signature}`.
3. An attacker submits that same response on fork B. The signature verifies correctly (same key, same hash), and the pending request exists (same state at fork point). `respond_verify_foreign_tx` accepts it.
4. The bridge execution on fork B is triggered by a response that was produced for fork A's chain context, potentially with stale or mismatched foreign-chain state, causing invalid bridge execution or double-spend conditions.

This maps directly to the allowed High impact: **cross-chain replay causing invalid bridge execution**.

### Likelihood Explanation

NEAR hard forks are rare but not impossible — the same external-factor reasoning that led the original M-05 judge to rate it Medium applies here. Additionally, the attack surface is broader: any scenario where the same key material is reachable under two different NEAR contract deployments (e.g., a contract migration that preserves key shares via resharing) creates the same replay window. An unprivileged attacker needs only to observe a valid `VerifyForeignTransactionResponse` on one deployment and submit it to the other before the pending request is resolved there.

### Recommendation

Include the NEAR contract account ID and NEAR network ID in the signed payload, analogous to EIP-712's `verifyingContract` and `chainId` fields. For example, extend `ForeignTxSignPayloadV1` with a `context` field:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
    pub contract_id: AccountId,   // env::current_account_id()
    pub network_id: String,       // e.g. "mainnet", "testnet"
}
```

Because `ForeignTxSignPayload` is versioned (`V1`, `V2`, …) and Borsh field ordering is stability-critical, this should be introduced as a new `V2` variant rather than modifying `V1`, with nodes and the SDK updated to produce and verify `V2` hashes. [3](#0-2) 

### Proof of Concept

1. At block N, NEAR undergoes a hard fork producing chains A and B. Both have an identical pending `verify_foreign_transaction` request `R` in `pending_verify_foreign_tx_requests` and identical MPC key material for domain D.

2. On chain A, the MPC network queries the foreign chain, computes:
   ```
   payload = ForeignTxSignPayload::V1 { request: R.request, values: [v1, v2] }
   payload_hash = SHA-256(borsh(payload))   // no NEAR context included
   signature = MPC_sign(payload_hash, key_D)
   ```
   and calls `respond_verify_foreign_tx(R, {payload_hash, signature})`. Chain A resolves R.

3. On chain B (where R is still pending), the attacker calls `respond_verify_foreign_tx(R, {payload_hash, signature})` with the values observed on chain A.

4. `respond_verify_foreign_tx` on chain B:
   - Finds R in `pending_verify_foreign_tx_requests` ✓
   - Retrieves domain D's public key ✓
   - Verifies `signature` over `payload_hash` under key_D ✓ (same key, same hash)
   - Resolves R and delivers the response to the caller's promise ✓

5. Chain B's bridge contract receives a verification response that was produced for chain A's foreign-chain state, potentially authorizing a transfer that should not be authorized on chain B. [4](#0-3) [5](#0-4)

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
