Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Amount in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of its limit check, evaluating only `totalAssetDeposits > limit` instead of `totalAssetDeposits + amount > limit`. Any unprivileged caller can invoke `depositETH` with an arbitrarily large value while the current total is at or below the configured cap, bypassing the deposit limit entirely in a single transaction.

## Finding Description
The vulnerable function is at lines 676–682 of `contracts/LRTDepositPool.sol`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC20
}
``` [1](#0-0) 

This function is called from `_beforeDeposit` (lines 648–670), which gates both `depositETH` and `depositAsset`:

```solidity
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [2](#0-1) 

Because the ETH branch uses a strict greater-than without adding `amount`, the guard returns `false` (i.e., "limit not exceeded") whenever `totalAssetDeposits <= limit`, regardless of the size of the incoming deposit. The ERC20 branch correctly includes `amount` in the comparison, making the asymmetry a clear implementation defect rather than an intentional design choice.

The entry point is `depositETH`, a public `payable` function with no access control beyond `whenNotPaused` and `onlySupportedAsset`: [3](#0-2) 

## Impact Explanation
The ETH deposit cap configured via `lrtConfig.depositLimitByAsset` is the sole on-chain mechanism preventing over-exposure of ETH to EigenLayer strategies. The bug renders it ineffective: a depositor can mint rsETH against an arbitrarily large ETH deposit in a single call, pushing total ETH exposure far beyond the intended limit. No user funds are directly stolen, but the protocol fails to enforce its own invariant.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The trigger condition (`getTotalAssetDeposits(ETH_TOKEN) <= depositLimitByAsset(ETH_TOKEN)`) is true for the entire operational lifetime of the protocol until the limit is already breached. No flash loan, privileged role, multi-step sequence, or special precondition is required. Any external account can call `depositETH{value: X}(0, "")` with an arbitrarily large `X`.

**Likelihood: High.**

## Recommendation
Unify the check to include `amount` for both ETH and ERC20, mirroring the already-correct ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept

1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether`.
2. Legitimate users deposit until `getTotalAssetDeposits(ETH_TOKEN) == 99 ether`.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit`: `99 ether > 100 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 10 000 ETH; total ETH in protocol becomes 10 099 ETH.
6. All 10 099 ETH will eventually be forwarded to EigenLayer via `NodeDelegator`, far exceeding the risk limit the admin intended to enforce.

**Foundry test plan:** Deploy `LRTDepositPool` on a local fork, configure `depositLimitByAsset[ETH_TOKEN] = 100 ether`, simulate deposits up to 99 ether, then call `depositETH{value: 10_000 ether}` from an unprivileged address and assert that `getTotalAssetDeposits(ETH_TOKEN)` exceeds the configured limit without revert.
