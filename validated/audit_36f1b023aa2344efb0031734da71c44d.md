Audit Report

## Title
Incorrect `_sender` in `Withdraw` Event When `withdrawTo()` Is Called with a Different Recipient - (File: `contracts/agETH/AGETHTokenWrapper.sol`)

## Summary
`AGETHTokenWrapper._withdraw()` emits `Withdraw(_asset, _to, _amount)`, passing the recipient `_to` as the `_sender` field of the event. When `withdrawTo(asset, _to, _amount)` is called with `_to != msg.sender`, the event records the recipient address as the initiator rather than the actual caller. The on-chain token operations are unaffected; only the emitted event data is incorrect.

## Finding Description
The `Withdraw` event is declared with a `_sender` field:

```solidity
event Withdraw(address asset, address _sender, uint256 _amount); // L32
```

The internal `_withdraw()` function correctly burns from `msg.sender` and transfers to `_to`, but emits `_to` in the `_sender` position:

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    emit Withdraw(_asset, _to, _amount);   // _to used as _sender — incorrect
}
``` [1](#0-0) 

This is triggered by the public `withdrawTo()` entry point, callable by any token holder:

```solidity
function withdrawTo(address asset, address _to, uint256 _amount) external {
    _withdraw(asset, _to, _amount);
}
``` [2](#0-1) 

The sibling contract `RsETHTokenWrapper` demonstrates the correct pattern — it explicitly passes `msg.sender` as the sender and `_to` as the receiver:

```solidity
event Withdraw(address indexed asset, address indexed sender, address indexed receiver, uint256 amount);
// ...
emit Withdraw(_asset, msg.sender, _to, _amount);
``` [3](#0-2) [4](#0-3) 

`AGETHTokenWrapper` never records `msg.sender` in the `Withdraw` event at all. The bug is invisible when `withdraw()` is called (since `_to == msg.sender`), but manifests on every `withdrawTo()` call where the recipient differs from the caller.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** The contract's observable interface (its events) does not accurately reflect the actual operation performed. Any off-chain system consuming `Withdraw` events to attribute withdrawal activity — indexers, analytics dashboards, accounting tools, tax reporters, or protocol-level monitors — will record the recipient as the initiator. No funds are lost or frozen; the incorrect data is confined entirely to the emitted event.

## Likelihood Explanation
`withdrawTo()` is a public, permissionless function callable by any agETH wrapper token holder with no role requirement. Any user who calls `withdrawTo()` with a recipient address different from their own triggers the incorrect event. The condition is trivially reachable and repeatable. [2](#0-1) 

## Recommendation
Change `_withdraw()` to emit `msg.sender` as the sender, not `_to`. Optionally align the event signature with `RsETHTokenWrapper` to include both sender and receiver:

```solidity
event Withdraw(address asset, address sender, address receiver, uint256 amount);
// ...
emit Withdraw(_asset, msg.sender, _to, _amount);
``` [5](#0-4) 

## Proof of Concept
1. Alice holds 10 agETH wrapper tokens.
2. Alice calls `withdrawTo(altAgETH, bob, 10e18)`.
3. Inside `_withdraw()`:
   - `_burn(msg.sender, 10e18)` — burns 10 wrapper tokens from Alice. ✓
   - `safeTransfer(bob, 10e18)` — sends 10 altAgETH to Bob. ✓
   - `emit Withdraw(altAgETH, bob, 10e18)` — records **Bob** as `_sender`. ✗
4. Any indexer reading the `Withdraw` event concludes Bob withdrew 10 altAgETH; Alice's withdrawal activity is invisible in event logs.

A Foundry unit test can confirm this by calling `withdrawTo` with `_to != msg.sender`, capturing the emitted event via `vm.expectEmit`, and asserting that the `_sender` field equals `msg.sender` (Alice) rather than `_to` (Bob) — the test will fail against the current implementation, proving the bug.
