### Title
Single Participant Can Block All `verify_foreign_transaction` Requests via Empty Chain Registration - (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` gates every foreign-chain verification request on `get_supported_foreign_chains()`, which enforces a **strict all-participant intersection rule**. A single active participant calling `register_foreign_chain_support` with an empty set immediately drops every chain from the supported set, causing every subsequent `verify_foreign_transaction` call to revert with `ForeignChainNotSupported`. This requires only one Byzantine participant — strictly below the signing threshold — and is a direct analog to the reported `chainId`-not-configured liquidation block.

---

### Finding Description

`verify_foreign_transaction` in `crates/contract/src/lib.rs` computes the set of allowed chains at call time:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(&InvalidParameters::ForeignChainNotSupported { ... }.to_string());
}
``` [1](#0-0) 

`get_supported_foreign_chains()` returns only chains for which **every** active participant has registered support — a strict superset check:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
if all_active_nodes_supports_chain { Some(foreign_chain) } else { None }
``` [2](#0-1) 

Any active participant can overwrite their registration at any time via `register_foreign_chain_support`, which accepts an empty set with no validation:

```rust
pub fn register_foreign_chain_support(
    &mut self,
    foreign_chain_support: dtos::SupportedForeignChains,
) -> Result<(), Error> {
    let account_id = self.voter_or_panic();
    self.node_foreign_chain_support
        .foreign_chain_support_by_node
        .insert(account_id, foreign_chain_support);
    Ok(())
}
``` [3](#0-2) 

The project's own design document explicitly identifies this root cause:

> "Today, `get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains, and `verify_foreign_transaction` rejects any request whose target chain is not in it. A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down." [4](#0-3) 

A threshold-safe alternative (`get_available_foreign_chains()`) exists and is computed by `recompute_available_foreign_chains()`, but `verify_foreign_transaction` does **not** use it — it still calls the legacy strict-intersection function. [5](#0-4) 

---

### Impact Explanation

A single Byzantine participant (one is sufficient — no threshold collusion required) calls `register_foreign_chain_support` with an empty `SupportedForeignChains` set. From that moment, `get_supported_foreign_chains()` returns an empty set, and every call to `verify_foreign_transaction` — regardless of which chain is requested — reverts with `ForeignChainNotSupported`. All in-flight and future foreign-chain verification requests are permanently blocked until the malicious participant re-registers a non-empty set or is removed via resharing. This breaks the request-lifecycle and contract execution-flow invariants of the foreign-chain verification feature, matching the **Medium** allowed impact tier: *"request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

Any one of the `n` active participants can trigger this unilaterally. The call is a normal contract method (`register_foreign_chain_support`) that participants are expected to call regularly. A participant who turns Byzantine — or whose node is compromised — can issue this call at any time. No threshold collusion, no privileged access, and no external dependency is required. The attack is cheap (a single NEAR transaction) and immediately effective.

---

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which already implements the threshold-based availability check (`≥ signing_threshold` participants must cover a chain). This is the fix already designed and tracked in the project's own proposal (issue #3434): [6](#0-5) 

Concretely, in `verify_foreign_transaction`:
```rust
// Replace:
let supported_chains = self.get_supported_foreign_chains();
// With:
let supported_chains = self.get_available_foreign_chains();
```

This ensures that a single non-registering or maliciously-registering participant cannot drop a chain from the serviceable set, as long as at least `signing_threshold` participants cover it.

---

### Proof of Concept

1. Network has `n = 4` active participants, signing threshold `t = 3`. All four have previously registered `[Bitcoin, Ethereum]` via `register_foreign_chain_support`.
2. Participant `P1` (Byzantine, below threshold) calls:
   ```json
   register_foreign_chain_support({ "foreign_chain_support": [] })
   ```
3. `get_supported_foreign_chains()` now computes the intersection: `{Bitcoin, Ethereum} ∩ {} = {}` — empty set.
4. Any user calling `verify_foreign_transaction` for Bitcoin or Ethereum receives:
   ```
   Requested foreign chain, Bitcoin, is not supported.
   ``` [7](#0-6) 
5. The three honest participants (P2, P3, P4) cannot override P1's registration on P1's behalf. The feature remains blocked until P1 re-registers or is removed via a resharing ceremony.

The existing unit test `get_supported_foreign_chains__should_return_empty_when_no_votes` confirms that a single non-registering participant causes the entire supported set to be empty, validating the root cause. [8](#0-7)
