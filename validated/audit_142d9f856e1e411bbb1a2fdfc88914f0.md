### Title
Pool Waiting Room `ASSERT_HEIGHT_RELATIVE` Timer Reset via Unsigned Absorb Spends — (`File: chia/pools/pool_puzzles.py`)

### Summary

The pool waiting room puzzle enforces a `relative_lock_height` exit delay via `ASSERT_HEIGHT_RELATIVE`, measured from the **current singleton coin's confirmed block index**. Because absorb (reward-claim) spends in the waiting room state require no farmer signature and create a new singleton coin at the current block height, any party can repeatedly absorb farming rewards to reset the timer, permanently preventing the farmer from leaving the pool.

### Finding Description

When a farmer initiates a pool exit, the singleton transitions to `LEAVING_POOL` state, whose inner puzzle is the waiting room puzzle (`POOL_WAITINGROOM_INNERPUZ`). The exit path enforces `ASSERT_HEIGHT_RELATIVE relative_lock_height`, which is evaluated against the **coin being spent** — i.e., the current singleton coin's `confirmed_block_index`.

`create_absorb_spend` in `pool_puzzles.py` explicitly handles the waiting room state:

```python
elif is_pool_waitingroom_inner_puzzle(inner_puzzle):
    # inner sol is (spend_type, destination_puzhash, pool_reward_amount, pool_reward_height, extra_data)
    inner_sol = Program.to([0, reward_amount, height])
``` [1](#0-0) 

This absorb spend consumes the current singleton coin and creates a new one at the current block height, with the same waiting room inner puzzle. The new coin's `confirmed_block_index` becomes the new baseline for `ASSERT_HEIGHT_RELATIVE`.

Critically, absorb spends require **no farmer signature** when fee is zero:

```python
claim_spend = WalletSpendBundle(all_spends, G2Element())
# If fee is 0, no signatures are required to absorb
``` [2](#0-1) 

The wallet-side exit logic in `new_peak` also uses `tip_height` (the height of the last singleton spend) to decide when to submit the exit transaction:

```python
leave_height = tip_height + pool_wallet_info.current.relative_lock_height
if peak_height > leave_height + 2:
    # submit exit transaction
``` [3](#0-2) 

Each absorb updates `tip_height`, so the wallet perpetually defers the exit transaction.

The same reset applies to the on-chain `ASSERT_HEIGHT_RELATIVE` check, as confirmed by the lifecycle test which shows the exit fails with `ASSERT_HEIGHT_RELATIVE_FAILED` when the coin is too young: [4](#0-3) 

And that absorb while in the waiting room is a supported, tested code path: [5](#0-4) 

### Impact Explanation

A pool (or any on-chain participant) can monitor for new `p2_singleton` farming rewards belonging to a farmer in `LEAVING_POOL` state and immediately submit zero-fee absorb transactions. Each absorb:

1. Spends the current singleton coin (in waiting room state)
2. Creates a new singleton coin at the current block height
3. Resets the `ASSERT_HEIGHT_RELATIVE` timer to zero
4. Updates `tip_height` in the wallet's spend history, deferring the wallet's automatic exit submission

If absorbs are submitted every block (or whenever a farming reward appears), the farmer can never satisfy `ASSERT_HEIGHT_RELATIVE relative_lock_height` and is permanently locked in `LEAVING_POOL` state. This constitutes **permanent inability for a farmer to execute a valid pool state transition** and **corruption of pool membership state with direct security impact** — the farmer's XCH rewards continue flowing to the pool's `target_puzzle_hash` indefinitely.

### Likelihood Explanation

The pool operator is the most motivated attacker: they lose revenue when a farmer leaves. The attack requires only submitting zero-fee transactions whenever a farming reward appears at the farmer's `p2_singleton_puzzle_hash`. This is fully automatable, requires no privileged access, no keys, and no cryptographic capability beyond standard transaction submission.

### Recommendation

The waiting room puzzle's absorb path should not reset the `ASSERT_HEIGHT_RELATIVE` baseline. Options:

1. **Prohibit absorb in the waiting room state.** Remove the `is_pool_waitingroom_inner_puzzle` branch from `create_absorb_spend` so that absorbs are only valid in `FARMING_TO_POOL` state.
2. **Use an absolute height lock.** Record the block height at which the singleton entered the waiting room (e.g., via `ASSERT_HEIGHT_ABSOLUTE`) rather than a relative lock, so absorbs cannot shift the baseline.
3. **Require farmer signature for absorbs in waiting room state.** Add an `AGG_SIG_ME` condition to the waiting room absorb path so only the farmer can trigger it, removing the unsigned-absorb attack vector. [6](#0-5) 

### Proof of Concept

Attack sequence (on-chain):

1. Farmer calls `pw_self_pool` → singleton enters `LEAVING_POOL` at block `H`. Singleton coin `C0` is confirmed at block `H`.
2. Farmer farms a block at `H+1`, creating a `p2_singleton` reward coin `R1`.
3. Pool (attacker) submits `create_absorb_spend(last_coin_spend, LEAVING_POOL_state, ...)` with `G2Element()` (no signature, zero fee). This spends `C0` and creates `C1` confirmed at block `H+1`.
4. Farmer tries to exit at block `H + relative_lock_height`: the coin being spent is `C1` (confirmed at `H+1`), so only `relative_lock_height - 1` blocks have elapsed → `ASSERT_HEIGHT_RELATIVE_FAILED`.
5. Pool repeats step 3 for every new farming reward. The timer never expires.

The wallet-side guard in `self_pool` / `join_pool` also fails because `history[-1][0]` (the last spend height) is always the most recent absorb block: [7](#0-6)

def create_absorb_spend(
    last_coin_spend: CoinSpend,
    current_state: PoolState,
    launcher_coin: Coin,
    height: uint32,
    genesis_challenge: bytes32,
    delay_time: uint64,
    delay_ph: bytes32,
) -> list[CoinSpend]:
    inner_puzzle: Program = pool_state_to_inner_puzzle(
        current_state, launcher_coin.name(), genesis_challenge, delay_time, delay_ph
    )
    reward_amount: uint64 = calculate_pool_reward(height)
    if is_pool_member_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol: Program = Program.to([reward_amount, height])
    elif is_pool_waitingroom_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, destination_puzhash, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol = Program.to([0, reward_amount, height])
    else:
        raise ValueError
    # full sol = (parent_info, my_amount, inner_solution)
    coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(last_coin_spend)
    assert coin is not None

    if coin.parent_coin_info == launcher_coin.name():
        parent_info: Program = Program.to([launcher_coin.parent_coin_info, launcher_coin.amount])
    else:
        p = Program.from_bytes(bytes(last_coin_spend.puzzle_reveal))
        last_coin_spend_inner_puzzle: Program | None = get_inner_puzzle_from_puzzle(p)
        assert last_coin_spend_inner_puzzle is not None
        parent_info = Program.to(

                last_coin_spend.coin.parent_coin_info,
                last_coin_spend_inner_puzzle.get_tree_hash(),
                last_coin_spend.coin.amount,
            ]
        )
    full_solution: SerializedProgram = SerializedProgram.to([parent_info, last_coin_spend.coin.amount, inner_sol])
    full_puzzle: SerializedProgram = create_full_puzzle(inner_puzzle, launcher_coin.name()).to_serialized()
    assert coin.puzzle_hash == full_puzzle.get_tree_hash()

    reward_parent: bytes32 = pool_parent_id(height, genesis_challenge)
    p2_singleton_puzzle = create_p2_singleton_puzzle(
        SINGLETON_MOD_HASH, launcher_coin.name(), delay_time, delay_ph
    ).to_serialized()

    p2_singleton_solution = SerializedProgram.to([inner_puzzle.get_tree_hash(), reward_coin.name()])
    assert p2_singleton_puzzle.get_tree_hash() == reward_coin.puzzle_hash
    assert full_puzzle.get_tree_hash() == coin.puzzle_hash
    assert get_inner_puzzle_from_puzzle(Program.from_bytes(bytes(full_puzzle))) is not None

    coin_spends = [
        CoinSpend(coin, full_puzzle, full_solution),
        CoinSpend(reward_coin, p2_singleton_puzzle, p2_singleton_solution),
    ]
    return coin_spends
