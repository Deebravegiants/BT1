### Title
`starknet_traceTransaction` / `starknet_traceBlockTransactions` Returns Wrong Execution Trace for Internally-Synced Blocks — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary
When a sequencer node finalizes a block via the internal consensus path, `convert_sync_block_to_block_data` stores placeholder `invoke_tx(InvokeTxArgs::default())` transactions in the body storage instead of the real transactions. Any unprivileged RPC client that subsequently calls `starknet_traceTransaction` or `starknet_traceBlockTransactions` on a hash from such a block receives an authoritative-looking but entirely wrong execution trace.

### Finding Description

`TransactionStreamFactory::convert_sync_block_to_block_data` is called whenever a `SyncBlock` arrives via the internal channel (i.e., a block finalized by the local consensus engine rather than fetched from a p2p peer). [1](#0-0) 

It builds a `BlockBody` with the correct transaction hashes but fills every `transactions` slot with a deterministic, all-zero `InvokeV1` placeholder: [2](#0-1) 

This body is written to persistent storage via `append_body`: [3](#0-2) 

The `P2pSyncClient` run-loop fans every incoming `SyncBlock` out to the transaction stream through `InternalBlocksSenders::send`, which feeds `transaction_receiver`: [4](#0-3) 

On the RPC side, `starknet_traceBlockTransactions` reads `block_transactions` directly from storage and passes each one to `stored_txn_to_executable_txn`: [5](#0-4) 

`stored_txn_to_executable_txn` for `Transaction::Invoke` performs **no validation** — it wraps the value as-is: [6](#0-5) 

The same path is taken by `starknet_traceTransaction` when the hash resolves to a stored (non-pending) block: [7](#0-6) 

The placeholder transaction has a zero sender address, zero calldata, zero nonce, and zero signature. Executing it against the real chain state produces a trace that bears no relation to the original transaction.

### Impact Explanation

Any unprivileged caller of `starknet_traceTransaction` or `starknet_traceBlockTransactions` against a sequencer node that has processed at least one internally-synced block will receive a fabricated execution trace. The response is structurally valid JSON-RPC and indistinguishable from a correct trace to the caller. This satisfies the **High** impact criterion: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation

Every sequencer node running in consensus mode exercises this path for every finalized block. The `SyncBlock` type deliberately omits full transaction bodies (see the TODO at line 101), so the placeholder is used unconditionally. No special attacker capability is required beyond calling a public RPC endpoint with a known transaction hash.

### Recommendation

1. **Short-term**: Gate `starknet_traceTransaction` / `starknet_traceBlockTransactions` with a check that detects placeholder bodies (e.g., verify that the stored transaction hash matches the hash recomputed from the stored transaction body) and return an explicit error rather than a wrong trace.
2. **Long-term**: Resolve the TODO — include full transaction data in `SyncBlock` so `convert_sync_block_to_block_data` can store real transactions.

### Proof of Concept

1. Run a sequencer node in consensus mode so that at least one block is finalized via the internal path.
2. Obtain a real transaction hash `H` from that block (e.g., via `starknet_getBlockWithTxHashes`).
3. Call `starknet_traceTransaction` with `H`.
4. Observe that the returned `InvokeTransactionTrace` reflects execution of a zero-sender, zero-calldata `InvokeV1` transaction rather than the real transaction — the `validate_invocation` and `execute_invocation` entries will correspond to the placeholder, not the original call.
