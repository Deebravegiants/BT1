### Title

### Summary
The `limitDailyMint` modifier in the L2 pool contracts uses a discrete day counter anchored to `startTimestamp` to reset `dailyMintAmount`. Because the reset is a hard boundary at `startTimestamp + N * 1 days`, an unprivileged depositor can straddle that boundary and mint up to **2Ă— the intended daily limit** within a very short time window, directly analogous to the external report's time-window manipulation pattern.

### Finding Description
Every L2 pool contract (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV2ExternalBridge`) enforces a daily mint cap through the `limitDailyMint` modifier. The reset logic is:

```solidity
// RSETHPoolV3.sol â€“ limitDailyMint modifier
uint256 currentDay = getCurrentDay();          // (block.timestamp - startTimestamp) / 1 days
if (currentDay > lastMintDay) {
    lastMintDay = currentDay;
    dailyMintAmount = 0;                       // hard reset at exact 24-h boundary
}
if (dailyMintAmount + rsETHAmount > dailyMintLimit) revert DailyMintLimitExceeded();
dailyMintAmount += rsETHAmount;
``` [1](#0-0) 

`getCurrentDay()` returns `(block.timestamp - startTimestamp) / 1 days`, which increments by exactly 1 at `startTimestamp + N * 86400`. [2](#0-1) 

The same pattern is present in all other pool variants: [3](#0-2) [4](#0-3) [5](#0-4) 

**Attack pattern (analogous to the external report's buy/sell cycle):**

| Step | Action | `dailyMintAmount` | Notes |
|------|--------|-------------------|-------|
| T = `startTimestamp + N*86400 âˆ’ Îµ` | Deposit `dailyMintLimit` ETH | = `dailyMintLimit` | Day Nâˆ’1, limit fully consumed |
| T = `startTimestamp + N*86400` | Deposit `dailyMintLimit` ETH again | reset â†’ 0, then = `dailyMintLimit` | Day N just started, counter reset |

Total rsETH minted in â‰ˆ 2Îµ seconds: **2 Ă— `dailyMintLimit`**.

The external report's root cause â€” a sell period longer than the buy period â€” maps here to the fact that the pool tracks only the **deposit** side in a 24-hour window while the **reverse-swap** path (`swapAssetToPremintedRsETH`) never decrements `dailyMintAmount` and carries no time-window constraint of its own. The asymmetry means the deposit counter resets cleanly at the day boundary while no balancing "sell" counter exists to prevent the double-spend of the limit. [6](#0-5) 

### Impact Explanation
An attacker can mint up to **2Ă— `dailyMintLimit`** of wrsETH in a single atomic block pair straddling the day boundary. The daily mint cap is the protocol's primary rate-limiting control on L2 rsETH issuance; bypassing it means the protocol fails to enforce its stated per-day issuance ceiling. The attacker must supply real ETH/LST, so no funds are directly stolen, but the cap â€” a security invariant â€” is violated. This maps to **Low: Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
The boundary timestamp is fully deterministic and publicly readable via `getNextDailyLimitResetTimestamp()`. Any depositor can compute the exact block at which `currentDay` increments and submit two back-to-back transactions (or a single bundle via a block builder) to straddle it. No special role, flash loan, or oracle manipulation is required. Likelihood is **Medium**.

### Recommendation
Replace the discrete day-counter reset with a rolling 24-hour window, tracking `lastMintTimestamp` and accumulating only amounts deposited within the last `1 days`:

```solidity
// pseudocode
if (block.timestamp >= lastMintTimestamp + 1 days) {
    dailyMintAmount = 0;
    lastMintTimestamp = block.timestamp;
}
```

Alternatively, align `startTimestamp` with `RSETH.periodStartTime` so both limits reset simultaneously, and document the 2Ă— boundary-straddle risk as an accepted design trade-off (as the external report's developers did).
