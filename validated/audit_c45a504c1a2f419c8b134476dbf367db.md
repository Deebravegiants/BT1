### Title
Detached Refund Promises Silently Fail, Permanently Locking Excess Deposits — (File: `crates/contract/src/lib.rs`)

---

### Summary

The MPC contract uses `Promise::new(...).transfer(diff).detach()` for all excess-deposit refunds across three payable entry points. Because the promise is detached, its failure is never observed. If the recipient account does not exist at the time the refund receipt executes, the excess deposit is permanently locked inside the chain-signature contract with no recovery path.

---

### Finding Description

Three payable entry points refund excess deposits using the same detached-promise pattern:

**`require_deposit`** (called by `sign` and `request_app_private_key`): [1](#0-0) 

**`submit_participant_info`**: [2](#0-1) 

**`propose_update`**: [3](#0-2) 

In all three cases the pattern is identical:

```rust
Promise::new(recipient).transfer(diff).detach();
```

`.detach()` schedules the transfer receipt but discards the `Promise` handle, meaning the contract never inspects the outcome. On NEAR, a `Transfer` action to a non-existent account fails silently — the receipt is marked failed, the funds remain in the contract, and no error is surfaced to the caller or logged. There is no retry, no fallback accounting entry, and no administrative withdrawal function that could recover stranded amounts.

The contract README explicitly documents the refund guarantee:

> "Any excess deposit is automatically refunded." [4](#0-3) 

The code comment in `require_deposit` reinforces this: [5](#0-4) 

Neither the comment nor the implementation accounts for the failure branch.

---

### Impact Explanation

**Impact: Medium** — balance/accounting invariant break.

When a refund receipt fails, the excess deposit is permanently absorbed into the chain-signature contract's balance. The contract has no bookkeeping for failed refunds and no privileged withdrawal method, so the funds are irrecoverable. This breaks the documented production invariant that every excess deposit is returned to its sender.

The impact is bounded to excess deposits (the minimum required is 1 yoctonear, so only amounts above that threshold are at risk), but a caller that deliberately or accidentally attaches a large deposit — e.g., 1 NEAR — and whose account is absent at refund time loses the full excess permanently.

---

### Likelihood Explanation

**Likelihood: Low.**

A refund receipt fails only when the recipient account does not exist at execution time. The receipt is processed one or a few blocks after the originating transaction. The realistic trigger is a smart-contract caller that self-destructs (via `DeleteAccount`) in the same transaction batch as the `sign` / `request_app_private_key` call, or a user who deletes their account in the narrow window between submission and refund processing. Both scenarios are uncommon in normal operation but are fully reachable by an unprivileged caller without any special role or collusion.

---

### Recommendation

Replace every detached refund promise with a chained callback that checks the transfer result and either retries or emits a recoverable error. The idiomatic NEAR SDK pattern is:

```rust
// Instead of:
Promise::new(recipient).transfer(diff).detach();

// Use a chained callback:
Promise::new(recipient)
    .transfer(diff)
    .then(
        Promise::new(env::current_account_id())
            .function_call("on_refund".to_string(), args, NO_DEPOSIT, CALLBACK_GAS)
    );
```

Alternatively, maintain an internal `failed_refunds: LookupMap<AccountId, NearToken>` ledger and expose a permissionless `claim_refund` method so stranded deposits can always be recovered.

---

### Proof of Concept

1. Deploy a NEAR smart contract `attacker.near` that:
   - Calls `sign` on the MPC contract with `attached_deposit = 1_000_000_000_000_000_000_000_000` (1 NEAR) in one action.
   - Immediately issues a `DeleteAccount { beneficiary_id: "attacker.near" }` action in the same transaction (or a follow-up transaction submitted before the refund receipt lands).
2. The `sign` call succeeds; `require_deposit` fires `Promise::new("attacker.near").transfer(999_999_999_999_999_999_999_999).detach()`.
3. By the time the refund receipt executes, `attacker.near` no longer exists.
4. The transfer receipt fails silently. The 1 NEAR excess deposit remains in the MPC contract balance permanently.
5. No log entry, no error, no recovery path exists in the contract.

The same sequence applies to `submit_participant_info` (line 847) and `propose_update` (line 1330). [6](#0-5) [7](#0-6) [8](#0-7)

```rust
/// Checks that the caller attached at least `minimum_deposit` and refunds any excess.
///
/// A non-zero deposit is required so that the transaction must be signed by a
/// full-access key (or a function-call access key whose `deposit` allowance is
/// explicitly set). This prevents a **malicious frontend** from silently
/// submitting signature requests on behalf of a user via a restricted
/// function-call access key, because such keys cannot attach deposits by
/// default. In other words, requiring a deposit ensures the user (or their
/// full-access key) explicitly authorised the call.
///
/// See the "Deposit requirement" section in the contract README for more
/// details.
fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}
```
