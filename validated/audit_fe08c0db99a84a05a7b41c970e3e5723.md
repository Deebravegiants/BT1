### Title
Unchecked `payload_hash` Binding in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Malicious Attested Participant - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but it never verifies that `payload_hash` is actually the canonical hash of `ForeignTxSignPayload { request, values }` for the specific pending `request`. A single malicious attested participant (strictly below the signing threshold) can replay a valid MPC signature produced for a prior request to satisfy a completely different pending request, delivering a forged `payload_hash` to the original caller.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs the following checks:

1. Caller is an attested participant.
2. `verify_ecdsa_signature(signature_response, &payload_hash, &secp_pk)` — the signature is valid over the caller-supplied `payload_hash` against the domain root public key.
3. The `request` matches a pending yield entry. [1](#0-0) 

What it does **not** check is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` for the specific `request` being resolved. The `values` (the extracted foreign-chain data) are never submitted to the contract and are therefore never bound to the pending request on-chain.

The node-side code correctly computes `payload_hash` from `(request, values)` before signing: [3](#0-2) 

But the contract never reconstructs or validates this binding. It only checks that the signature is mathematically valid over whatever `payload_hash` the caller provides.

**Attack path:**

1. Attacker (a single malicious attested participant) submits `verify_foreign_transaction(request_A)` — a legitimate request for Bitcoin `tx_id=X`.
2. The MPC network (threshold nodes) signs the canonical payload → produces `(hash_A, sig_A)` where `hash_A = SHA-256(borsh(ForeignTxSignPayload::V1 { request_A, values_A }))`.
3. A victim submits `verify_foreign_transaction(request_B)` for a different Bitcoin `tx_id=Z`.
4. The malicious attested participant calls `respond_verify_foreign_tx(request=request_B, response={payload_hash=hash_A, signature=sig_A})`.
5. The contract checks: is `sig_A` valid over `hash_A` against the root public key? **Yes** — the signature is cryptographically valid.
6. The contract resolves the yield for `request_B` with `{payload_hash=hash_A, signature=sig_A}`.
7. The caller of `request_B` receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes `tx_id=X` (not `tx_id=Z`). [4](#0-3) 

### Impact Explanation

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow, where a bridge contract on NEAR reacts to a foreign-chain event only after the MPC network attests it. If the bridge contract does not independently recompute and compare `payload_hash` against the expected `(request, values)` — which the contract design leaves entirely to the caller — the malicious node's forged response is indistinguishable from a legitimate one. The bridge would process a foreign-chain event (e.g., a deposit of `tx_id=Z`) that was never actually verified, enabling invalid bridge execution or double-spend conditions.

This maps to: **High — forged foreign-chain verification that causes invalid bridge execution.**

### Likelihood Explanation

The attack requires only **one** malicious attested participant, which is strictly below the signing threshold. The attacker does not need to compromise the threshold signing protocol; they only need to reuse a signature the MPC network already produced legitimately for a different request. In production, attested participants must pass TEE attestation, which raises the bar, but a single compromised or malicious TEE node is sufficient. The attack is fully deterministic and requires no network-level interference.

### Recommendation

The contract should bind `payload_hash` to the pending `request` at response time. The simplest fix is to include the extracted `values` in `VerifyForeignTransactionResponse` and have the contract recompute and compare `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` against `response.payload_hash` before accepting the response. If on-chain recomputation is too expensive, the `values` should at minimum be committed to in the pending request entry (e.g., as a hash) so the contract can verify the binding without storing the full payload.

// Contract resolves yield for request_B with forged response
// Victim's bridge contract receives { payload_hash: hash_A, signature: sig_A }
// hash_A encodes tx_id=X, not tx_id=Z → forged attestation accepted if bridge skips payload_hash verification

    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
