### Title
Single Attested Participant Can Submit Unverified CKD Response for `AppPublicKey` Requests - (File: crates/contract/src/lib.rs)

### Summary
The `respond_ckd` function skips all cryptographic verification of the CKD response when the request uses the `AppPublicKey` variant. Unlike `respond` (for threshold signatures) and `respond_verify_foreign_tx`, which both verify the submitted signature on-chain against the MPC public key before resolving the yield, `respond_ckd` with `AppPublicKey` accepts any arbitrary `CKDResponse` from a single attested participant with no on-chain check. A Byzantine participant strictly below the signing threshold can forge the confidential key derivation output delivered to the user.

### Finding Description
In `respond_ckd`, the output check is conditional on the request variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant the contract calls `ckd_output_check`, which verifies the derived output against the BLS12-381 master public key and the user-supplied app public key. For the `AppPublicKey` variant the arm is an empty no-op: the contract immediately proceeds to resolve the pending yield with whatever bytes the caller supplied. [2](#0-1) 

Contrast this with `respond` for threshold signatures, which always verifies the submitted signature on-chain before resolving: [3](#0-2) 

And `respond_verify_foreign_tx`, which also verifies on-chain before resolving: [4](#0-3) 

The only authorization gate on `respond_ckd` is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a single attested participant — not a threshold of them. [5](#0-4) 

### Impact Explanation
A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe a pending `AppPublicKey` CKD request in the contract's `pending_ckd_requests` map.
2. Call `respond_ckd` with an arbitrary `CKDResponse` — for example, a derived key the attacker generated themselves, or a random invalid key.
3. The contract resolves the yield with the forged response with no verification.
4. The user's waiting `request_app_private_key` call returns the attacker-controlled derived key.

If the attacker supplies a key they know, they gain full control over the user's derived application secret. If they supply a random invalid key, the user's derived key is silently corrupted. Either outcome constitutes **confidential key derivation output without the required participant authorization**, matching the Critical impact tier.

### Likelihood Explanation
The `AppPublicKey` variant is a documented, supported API path (the contract README and `request_app_private_key` docstring both describe it). Any single attested participant — one that has passed `submit_participant_info` — can exploit this. No threshold collusion is required. The attacker only needs to race the honest nodes to call `respond_ckd` first, which is straightforward since NEAR transaction ordering is public.

### Recommendation
Apply the same defense used for threshold signatures: perform an on-chain cryptographic check before resolving the yield. For `AppPublicKey`, the response is encrypted to the user's app public key, so direct plaintext verification is not possible on-chain. Two mitigations are viable:

1. **Require threshold agreement before resolving**: accumulate `respond_ckd` calls from multiple participants (as a vote map keyed by request) and only resolve the yield once a threshold of identical responses has been received — mirroring how the off-chain MPC protocol already reaches consensus.
2. **Deprecate `AppPublicKey` in favor of `AppPublicKeyPV`**: the `AppPublicKeyPV` variant already supports on-chain verification via `ckd_output_check`. Document that `AppPublicKey` provides no on-chain integrity guarantee and remove it from the production API, or gate it behind an explicit user acknowledgement.

### Proof of Concept

```
1. User calls request_app_private_key({ app_public_key: AppPublicKey(user_pk), domain_id, derivation_path })
   → contract enqueues yield, stores CKDRequest in pending_ckd_requests

2. Malicious attested participant (account: evil.near) calls:
   respond_ckd(request=<same CKDRequest>, response=<attacker-chosen CKDResponse>)

3. Contract execution path in respond_ckd:
   - assert_caller_is_signer()                          ✓ (evil.near is signer)
   - is_running_or_resharing()                          ✓
   - accept_requests check                              ✓
   - assert_caller_is_attested_participant_...()        ✓ (evil.near is attested)
   - match request.app_public_key { AppPublicKey(_) => {} }  ← NO CHECK
   - resolve_yields_for(pending_ckd_requests, request, forged_response)  ← yield resolved

4. User's request_app_private_key call returns the attacker-supplied CKDResponse.
   If the attacker supplied a key they generated, they now share the user's derived secret.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
