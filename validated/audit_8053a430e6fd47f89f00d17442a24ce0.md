Audit Report

## Title
Stale Chainlink Price Accepted in `ChainlinkPriceOracle.getAssetPrice()` Corrupts rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards every return value except the raw `int256 price`, performing no staleness, round-completeness, or sign check. A stale Chainlink answer is silently accepted and propagated into `LRTOracle._updateRsETHPrice()`, which sets the global `rsETHPrice` used to mint rsETH for all depositors. Because `updateRSETHPrice()` is a public, permissionless function, an attacker can trigger a price update at the precise moment a feed is stale, then immediately deposit to receive excess rsETH, diluting all existing holders.

## Finding Description

`contracts/oracles/ChainlinkPriceOracle.sol` line 52 reads:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The code ignores `updatedAt` (staleness), `answeredInRound` (round completeness), and does not verify `price > 0`.

This contrasts directly with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 30–32, a sibling contract in the same repository that correctly validates all three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The exploit path is:

1. `LRTOracle.updateRSETHPrice()` (line 87) is `public whenNotPaused` — no access control.
2. It calls `_updateRsETHPrice()` (line 214), which calls `_getTotalEthInProtocol()` (line 231).
3. `_getTotalEthInProtocol()` (line 339) calls `getAssetPrice(asset)` for each supported LST, which routes to `ChainlinkPriceOracle.getAssetPrice()` — returning the stale price unchecked.
4. `newRsETHPrice` is computed at line 250 and written to the global `rsETHPrice` state variable at line 313.
5. `LRTDepositPool.getRsETHAmountToMint()` (line 520) computes:
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```
   If `rsETHPrice` was set using a stale (depressed) price but the feed has since updated to the true price, the attacker deposits at the true asset price divided by the stale (lower) `rsETHPrice`, receiving excess rsETH.

The `pricePercentageLimit` downside-protection check (lines 270–281) is a partial mitigation: it pauses the protocol if the price drop exceeds the configured threshold. However, it provides no protection when `pricePercentageLimit == 0` (its default uninitialized value), and it does not catch small deviations within the limit — precisely the scenario during a normal heartbeat window where the stale price lags the true price by a small but exploitable margin.

## Impact Explanation

**High — Theft of unclaimed yield.**

When a Chainlink feed goes stale and its last reported price is below the true current LST/ETH rate, `_getTotalEthInProtocol()` underestimates the protocol's TVL, causing `rsETHPrice` to be set below its true value. An attacker who calls `updateRSETHPrice()` at this moment locks in the depressed `rsETHPrice`. Once the feed updates to the true price, the attacker deposits LSTs via `LRTDepositPool.depositAsset()`: the numerator `lrtOracle.getAssetPrice(asset)` reflects the true (higher) price while the denominator `lrtOracle.rsETHPrice()` remains the stale (lower) value, minting excess rsETH. This dilutes all existing rsETH holders, constituting theft of their accrued yield.

## Likelihood Explanation

Chainlink LST/ETH feeds have documented heartbeat intervals (e.g., 24 hours) and deviation thresholds. During low-volatility periods, feeds can remain at their last reported value for the full heartbeat window while the true LST/ETH rate continues to accrue yield. Because `updateRSETHPrice()` is public and callable by anyone, an attacker can monitor on-chain feed `updatedAt` timestamps and call the function precisely when a feed is stale. No privileged access is required. The attack is repeatable every heartbeat cycle.

## Recommendation

Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_STALENESS_PERIOD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS_PERIOD` should be set per-asset based on the Chainlink feed's documented heartbeat interval.

## Proof of Concept

**Setup:** stETH/ETH Chainlink feed has a 24-hour heartbeat. Protocol has 1000 stETH deposited. At `T=0` the feed reports `0.9990e18`; `rsETHPrice` is set to `0.9990e18`. At `T=23h` the true rate is `1.0010e18` but the feed has not updated (`updatedAt` is 23 hours old).

**Attack sequence:**

1. At `T=23h`, attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control).
2. `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns stale `0.9990e18`.
3. `rsETHPrice` is written as `0.9990e18` (or lower if TVL is understated).
4. Feed updates to `1.0010e18` (e.g., at `T=23h+1s`).
5. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
6. `getRsETHAmountToMint` computes: `100e18 * 1.0010e18 / 0.9990e18 ≈ 100.2e18` rsETH instead of the fair `100e18`.
7. Attacker receives ~0.2 rsETH excess per 100 stETH deposited, diluting all existing holders.
