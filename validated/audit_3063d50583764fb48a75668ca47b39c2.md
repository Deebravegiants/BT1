Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Any Depositor to Exceed the Protocol Cap — (File: `contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` applies an asymmetric check: for LSTs it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it only tests `totalAssetDeposits > depositLimit`, omitting the incoming `amount`. Any depositor can push ETH deposits beyond the configured cap in a single call whenever the running total is at or below the limit, breaking the protocol's primary ETH exposure safety cap.

## Finding Description
`_beforeDeposit()` calls `_checkIfDepositAmountExceedesCurrentLimit()` before minting rsETH:

```solidity
// contracts/LRTDepositPool.sol L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← missing + amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

`totalAssetDeposits` is the pre-deposit total. For ETH, the function returns `true` (revert) only when the total **already** exceeds the limit. When `totalAssetDeposits == depositLimit`, the expression `depositLimit > depositLimit` evaluates to `false`, so the check passes and the deposit is accepted, pushing the total to `depositLimit + amount`.

The entry path is fully unprivileged:
```
depositETH() [public, payable]
  → _beforeDeposit()
    → _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)
      → returns false  ← incorrect when totalAssets == limit
  → _mintRsETH()      ← rsETH minted, limit exceeded
```

No existing guard compensates for this: `_beforeDeposit()` at L661-663 reverts only if `_checkIfDepositAmountExceedesCurrentLimit` returns `true`, which it never does at the boundary for ETH.

## Impact Explanation
**Low — Contract fails to deliver promised returns.** The deposit limit is the protocol's primary safety cap on ETH exposure (EigenLayer strategy capacity, slashing risk, liquidity constraints). Bypassing it allows the protocol to accept more ETH than it is designed to handle. Any depositor can exceed the cap by exactly one deposit's worth of ETH per transaction. The cap guarantee — a core protocol invariant — is broken without any loss of deposited value, matching the Low impact class exactly.

## Likelihood Explanation
High. The condition `totalAssetDeposits == depositLimit` is reachable in normal operation (the limit is designed to be hit). No special privileges, front-running, or external dependencies are required — any ETH depositor triggers the bypass automatically whenever the running total is at or below the limit. The condition can recur if assets are moved to NodeDelegators, reducing the pool balance counted by `getTotalAssetDeposits`.

## Recommendation
Apply the same `+ amount` inclusion used for LSTs:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets ETH deposit limit to `L` via `LRTConfig`.
2. Through normal deposits, `getTotalAssetDeposits(ETH_TOKEN)` reaches exactly `L`.
3. Any caller invokes `depositETH{value: Y}(0, "")` for any `Y > 0`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `L > L` → `false` → no revert.
5. `_mintRsETH` executes; total ETH deposits become `L + Y`, exceeding the configured cap.
6. Steps 3–5 can be repeated in subsequent blocks as long as `totalAssetDeposits <= L` at call time.

**Foundry fuzz test plan:**
```solidity
function testFuzz_ethDepositExceedsLimit(uint96 limitAmount, uint96 depositAmount) public {
    vm.assume(depositAmount > 0);
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, limitAmount);
    // fill deposits to exactly the limit

    // attempt one more deposit

    vm.prank(attacker);

}
```
