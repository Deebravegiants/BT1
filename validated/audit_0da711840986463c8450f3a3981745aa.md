### Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero Mid-Period - (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool` distributes ERC-20 reward tokens over a fixed time window set by `notifyRewardAmount`. When all stakers call `initiateWithdrawal` during an active reward period, `totalKernelStaked` immediately drops to zero. The `rewardPerToken()` function then freezes at the last stored value for the remainder of the window, and all reward tokens that should have been distributed during the zero-staked interval are permanently locked in the contract with no recovery path.

---

### Finding Description

`initiateWithdrawal` immediately decrements `totalKernelStaked` at the moment of initiation, not at claim time: [1](#0-0) 

`rewardPerToken()` handles the zero-staked case by returning the frozen accumulator: [2](#0-1) 

This means that for every second between the moment `totalKernelStaked` hits zero and `finishAt`, the quantity `rewardRate * elapsed` is silently discarded — it is never added to `rewardPerTokenStored` and can never be claimed by any address. The `updateReward` modifier snapshots `rewardPerToken()` before every state change, so the frozen value propagates into every subsequent user checkpoint.

`notifyRewardAmount` guards against starting a new period with zero stakers: [3](#0-2) 

But there is **no symmetric guard** in `initiateWithdrawal` that prevents reducing `totalKernelStaked` to zero while a reward period is still active (`block.timestamp < finishAt`). The contract's own NatDoc explicitly acknowledges the consequence: [4](#0-3) 

The proposed mitigation in that comment ("ensuring there are always some tokens staked") is a purely off-chain operational assumption with zero on-chain enforcement.

---

### Impact Explanation

Any reward tokens allocated to the zero-staked interval are permanently unclaimable. They remain in the contract balance but can never be distributed because:

1. `rewardPerTokenStored` is frozen at the value it held when `totalKernelStaked` last became zero.
2. When new stakers arrive after `finishAt`, `rewardRate` is no longer applied (the window has closed).
3. There is no admin sweep, rescue, or re-allocation function for stranded reward tokens.

This constitutes **permanent freezing of unclaimed yield**, matching the allowed Medium impact scope.

---

### Likelihood Explanation

The trigger is reachable by any combination of ordinary stakers exercising their normal right to withdraw. A single large staker who holds the majority of `totalKernelStaked` can unilaterally cause this by calling `initiateWithdrawal` for their full balance. No privileged role, oracle manipulation, or external protocol compromise is required. The scenario is especially plausible when:

- A reward period is long (e.g., 30 days) and market conditions change.
- A single whale dominates the staked supply.
- Stakers rationally exit after a price drop in the reward token.

---

### Recommendation

Add a check in `initiateWithdrawal` that prevents `totalKernelStaked` from reaching zero while a reward period is active, or alternatively, when `totalKernelStaked` would drop to zero mid-period, forward the remaining unallocated rewards (`rewardRate * (finishAt - block.timestamp)`) to a recoverable treasury address before zeroing out `rewardRate` and `finishAt`. A minimal on-chain guard:

```solidity
// In initiateWithdrawal, after decrementing totalKernelStaked:
if (totalKernelStaked == 0 && block.timestamp < finishAt) {
    // sweep remaining rewards to treasury or reset the period
    rewardRate = 0;
    finishAt = block.timestamp;
}
```

This mirrors the recommended fix for M-21: handle the "no participants" edge case explicitly inside the accounting update path rather than relying on off-chain operational discipline.

---

### Proof of Concept

1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 30 days`; `rewardRate = 1_000e18 / 30 days`.
2. Alice is the only staker with `balanceOf[Alice] = totalKernelStaked = 100e18`.
3. After 1 day, Alice calls `initiateWithdrawal(100e18)`.
   - `updateReward(Alice)` snapshots 1 day of rewards correctly into `rewards[Alice]`.
   - `totalKernelStaked` is set to `0`.
4. For the remaining 29 days, `rewardPerToken()` returns the frozen `rewardPerTokenStored` on every call.
5. At `finishAt`, `rewardRate * 29 days ≈ 966e18` reward tokens are permanently locked in the contract.
6. Alice calls `getReward()` and receives only her 1-day share (`≈ 33e18`).
7. No address can ever claim the remaining `≈ 966e18` tokens; there is no admin rescue function. [5](#0-4) [6](#0-5)
