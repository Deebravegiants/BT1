### Title
ETH Accidentally Sent to Payable Swap/Liquidity Functions Is Permanently Capturable by Any Subsequent WETH Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

Every user-facing entry point in `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` is marked `payable`. The shared `pay()` helper in `PeripheryPayments` silently consumes the contract's entire native-ETH balance whenever the token being paid is WETH, without distinguishing whose ETH it is. ETH accidentally sent alongside a non-WETH call is therefore permanently capturable by the next caller who performs a WETH-input swap or WETH-leg liquidity add.

---

### Finding Description

`PeripheryPayments.pay()` contains the following branch for WETH payments:

```solidity
// PeripheryPayments.sol lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // ← uses contract's ETH, not payer's
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

When `address(this).balance >= value`, the contract wraps its own ETH and forwards it to the pool. The `payer` field is completely bypassed — no `transferFrom` is issued against the actual caller.

All four swap entry points in `MetricOmmSimpleRouter` are `payable`:

```solidity
// MetricOmmSimpleRouter.sol lines 67, 92, 130, 154
function exactInputSingle(ExactInputSingleParams calldata params) external payable ...
function exactInput(ExactInputParams calldata params) external payable ...
function exactOutputSingle(ExactOutputSingleParams calldata params) external payable ...
function exactOutput(ExactOutputParams calldata params) external payable ...
```

And all four liquidity entry points in `MetricOmmPoolLiquidityAdder` are `payable`:

```solidity
// MetricOmmPoolLiquidityAdder.sol lines 56, 71, 88, 123
function addLiquidityExactShares(...) external payable ...
function addLiquidityWeighted(...) external payable ...
```

None of these functions enforce that `msg.value == 0` when the input token is not WETH. ETH sent alongside a non-WETH call is silently retained by the contract with no automatic refund.

The `refundETH()` function exists but is a separate, optional call. It does not protect against front-running between the victim's transaction and a subsequent attacker transaction.

---

### Impact Explanation

**Direct loss of user ETH principal.**

A victim who accidentally sends ETH with a non-WETH swap loses that ETH permanently unless they call `refundETH()` in the same multicall batch. Any attacker who observes the stranded ETH (via mempool or on-chain state) can immediately call `exactInputSingle` with `tokenIn = WETH` and `amountIn ≤ stranded ETH`. The `pay()` callback wraps the victim's ETH and delivers it to the pool as the attacker's input, giving the attacker the full swap output at zero cost. The same theft path applies inside `MetricOmmPoolLiquidityAdder` for WETH-leg liquidity adds.

---

### Likelihood Explanation

**Low.** The attack requires a user error: sending ETH with a non-WETH call. However, this is a realistic mistake given that all entry points are `payable` and the WETH-via-ETH pattern is explicitly documented and supported. Front-running stranded ETH is trivial once it appears on-chain.

---

### Recommendation

1. **Primary fix**: Add `if (msg.value > 0 && tokenIn != WETH) revert ETHNotAccepted();` at the top of each swap and liquidity function that does not use native ETH.
2. **Alternative**: Remove `payable` from functions that have no WETH leg (non-WETH pools). For functions that may or may not use ETH, enforce `msg.value == 0` when the relevant token is not WETH.
3. **Defense-in-depth**: Emit an event or revert in `pay()` if `address(this).balance > 0` and `payer != address(this)` and `token != WETH`, to surface unexpected ETH accumulation.

---

### Proof of Concept

```
Setup:
  - Pool A: token0 = USDC, token1 = DAI  (no WETH leg)
  - Pool B: token0 = WETH, token1 = DAI

Step 1 — Victim:
  victim calls exactInputSingle{value: 1 ether}(
      pool = Pool_A, tokenIn = USDC, tokenOut = DAI, amountIn = 1000e6, ...
  )
  → swap executes normally using USDC transferFrom
  → 1 ETH remains in MetricOmmSimpleRouter (not returned, no revert)

Step 2 — Attacker (same block or next):
  attacker calls exactInputSingle{value: 0}(
      pool = Pool_B, tokenIn = WETH, tokenOut = DAI, amountIn = 1e18, ...
  )
  → pool calls metricOmmSwapCallback
  → _justPayCallback calls pay(WETH, attacker, pool, 1e18)
  → address(this).balance == 1e18 >= 1e18
  → contract wraps victim's 1 ETH → 1 WETH → transfers to pool
  → attacker receives DAI output; attacker spent 0 ETH and 0 WETH

Result:
  Victim lost 1 ETH.
  Attacker gained DAI equivalent of 1 WETH at zero cost.
