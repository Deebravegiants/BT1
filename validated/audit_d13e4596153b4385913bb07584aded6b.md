### Title
Phase-2 Invalid Transactions Bypass Per-Transaction Reference Script Size Limit - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

### Summary
The `validateRefScriptSize` check in the Conway LEDGER rule is guarded by `isValidTxL == IsValid True`, meaning phase-2 invalid transactions (those with intentionally failing Plutus scripts) are never checked against the `maxRefScriptSizePerTx` protocol parameter. An unprivileged transaction sender can craft a phase-2 invalid transaction that includes reference scripts exceeding the 200 KiB per-transaction limit, bypassing the intended per-transaction validation bound while remaining under the 1 MiB block-level cap.

### Finding Description

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`, the `validateRefScriptSize` predicate is invoked only inside the `IsValid True` branch of `conwayLedgerTransitionTRC`:

```haskell
if tx ^. isValidTxL == IsValid True
  then do
    ...
    runTest $ validateRefScriptSize pp (utxoState ^. utxoL) tx
``` [1](#0-0) 

The predicate itself enforces the per-transaction cap:

```haskell
validateRefScriptSize pp utxo tx =
  let totalRefScriptSize = txNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $ ...
``` [2](#0-1) 

For a phase-2 invalid transaction (`IsValid False`), this branch is skipped entirely. The block-level check in `validateBodyRefScriptsSizeTooBig` does count reference scripts from all transactions (valid and invalid alike): [3](#0-2) 

And `totalRefScriptSizeInBlock` accumulates sizes across all transactions regardless of validity: [4](#0-3) 

So the block-level cap (1 MiB) is enforced for all transactions, but the per-transaction cap (200 KiB) is enforced only for valid ones. A single phase-2 invalid transaction can therefore carry up to the full block-level budget (~1 MiB) of reference scripts — five times the intended per-transaction limit.

The fee check (`feesOK`) does still run for phase-2 invalid transactions and calls `getConwayMinFeeTxUtxo`, which includes the tiered reference-script fee: [5](#0-4) [6](#0-5) 

So the attacker must declare a fee that covers the exponential tiered cost and provide collateral equal to `collateralPercent × fee`. The economic deterrent is real but does not substitute for the structural limit.

### Impact Explanation

An attacker-controlled phase-2 invalid transaction can include reference scripts totalling up to the block-level maximum (~1 MiB) in a single transaction, exceeding the intended 200 KiB per-transaction validation limit introduced specifically to bound per-transaction deserialization overhead. Every honest node processing the block must deserialize those scripts to compute the minimum fee, consuming more CPU and memory per transaction than the protocol parameter was designed to allow. This matches the **Medium** impact class: *attacker-controlled transactions exceed intended validation limits*.

### Likelihood Explanation

The attacker must:
1. Deploy a UTxO locked by a Plutus script that is known to fail (trivial).
2. Pre-populate the UTxO set with outputs carrying large reference scripts (> 200 KiB total).
3. Submit a transaction referencing those outputs with `isValidTxL = IsValid False`, paying the exponentially-priced fee and 150 % collateral.

The exponential `tierRefScriptFee` pricing makes the economic cost very high for large script sizes, significantly reducing practical likelihood. However, the structural bypass is unconditional and requires no privileged access — any transaction sender can trigger it.

### Recommendation

Move `validateRefScriptSize` (or an equivalent check using `txNonDistinctRefScriptsSize`) outside the `IsValid True` guard so that the per-transaction reference script size limit is enforced for all transactions, including phase-2 invalid ones. The check should mirror the unconditional enforcement already applied at the block level in `validateBodyRefScriptsSizeTooBig`.

### Proof of Concept

1. Deploy a UTxO `u_script` locked by `alwaysFailsNoDatum SPlutusV3`.
2. Deploy N UTxOs `u_ref_1 … u_ref_N`, each carrying a large reference script, such that `sum(originalBytesSize(script_i)) > maxRefScriptSizePerTx` (> 200 KiB) but `<= maxRefScriptSizePerBlock` (≤ 1 MiB).
3. Construct a transaction:
   - `inputsTxBodyL = {u_script}`
   - `referenceInputsTxBodyL = {u_ref_1, …, u_ref_N}`
   - `isValidTxL = IsValid False`
   - Fee set to cover `getConwayMinFeeTxUtxo` (which includes the tiered ref-script fee for the full N-script size).
   - Collateral ≥ `collateralPercent × fee`.
4. Submit the transaction. The LEDGER rule skips `validateRefScriptSize` because `isValidTxL = IsValid False`; the block-level check passes because the total is ≤ 1 MiB. The transaction is accepted, and every node deserializes > 200 KiB of reference scripts for this single transaction — exceeding the intended per-transaction bound. [7](#0-6) [8](#0-7)

totalRefScriptSizeInBlock ::
  (AlonzoEraTx era, BabbageEraTxBody era) => ProtVer -> StrictSeq (Tx TopTx era) -> UTxO era -> Int
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      let updatedUtxo = accUtxo `Map.union` unUTxO toAdd
          toAdd
            | IsValid True <- tx ^. isValidTxL = txouts $ tx ^. bodyTxL
            | otherwise = collOuts $ tx ^. bodyTxL
       in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
