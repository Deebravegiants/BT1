Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` Enables MEV Reward Dilution Attack Against Existing rsETH Holders - (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` has no access control, allowing any caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at an arbitrary time. An attacker can deposit ETH at the pre-reward `rsETHPrice`, trigger `sendFunds()` to inflate TVL, then initiate withdrawal after `rsETHPrice` is updated to the elevated rate — capturing a share of MEV rewards that belong exclusively to pre-existing rsETH holders. This constitutes theft of unclaimed yield.

## Finding Description

`FeeReceiver.sendFunds()` is declared `external` with no role modifier:

```solidity
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

Every other state-mutating function in `FeeReceiver` is gated behind `onlyRole(LRTConstants.MANAGER)` (e.g., `setDepositPool`), but `sendFunds()` is left completely open. [2](#0-1) 

`LRTDepositPool.receiveFromRewardReceiver()` also has no access control and accepts any ETH: [3](#0-2) 

The rsETH exchange rate (`rsETHPrice`) is a **stored state variable** in `LRTOracle`, updated only when `updateRSETHPrice()` is explicitly called: [4](#0-3) 

`updateRSETHPrice()` is itself a `public` function with no role restriction (only `whenNotPaused`): [5](#0-4) 

The mint rate for deposits uses the stored `rsETHPrice`: [6](#0-5) 

The withdrawal amount at `initiateWithdrawal()` is also computed from the stored `rsETHPrice` and locked into the request: [7](#0-6) [8](#0-7) 

**Exploit path:**

1. Observe that `FeeReceiver` holds a meaningful ETH balance (e.g., 10 ETH in MEV rewards). The stored `rsETHPrice` does not yet reflect these rewards.
2. Deposit ETH into `LRTDepositPool.depositETH()` — rsETH is minted at the current (pre-reward) `rsETHPrice`. This is price-neutral: TVL and supply increase proportionally.
3. Call `FeeReceiver.sendFunds()` — the 10 ETH flows into `LRTDepositPool`, increasing TVL without increasing rsETH supply.
4. Call `LRTOracle.updateRSETHPrice()` (public, no role required) — `rsETHPrice` updates to reflect the elevated TVL. If the price increase exceeds `pricePercentageLimit`, the attacker instead front-runs the manager's routine `updateRSETHPriceAsManager()` call.
5. Call `LRTWithdrawalManager.initiateWithdrawal()` — `expectedAssetAmount` is locked at the elevated `rsETHPrice`.
6. After the withdrawal delay, `completeWithdrawal()` returns more ETH than deposited, capturing a share of the 10 ETH MEV reward.

The `_calculatePayoutAmount` function at unlock time takes `min(expectedAssetAmount, currentReturn)`. Since MEV rewards only increase the price monotonically (absent slashing), the attacker's locked elevated amount is honored: [9](#0-8) 

**Numerical example:**
- State: 1000 rsETH, 1050 ETH in protocol, `rsETHPrice` = 1.05, FeeReceiver holds 10 ETH.
- Without attacker: after `sendFunds` + price update, existing 1000 rsETH holders gain ~10 ETH (minus protocol fee).
- With attacker depositing 100 ETH first: attacker holds ~95.24 rsETH (8.7% of 1095.24 total). After `sendFunds` + price update, attacker captures ~0.87 ETH of the 10 ETH reward; existing holders receive only ~9.13 ETH instead of ~10 ETH.

## Impact Explanation

**High — Theft of unclaimed yield.**

MEV/execution-layer rewards accumulate in `FeeReceiver` and represent yield owed proportionally to **all current rsETH holders at the time of distribution**. Because `sendFunds()` is permissionless, an attacker can insert themselves as a holder immediately before the reward flush, diluting the yield share of pre-existing holders. The stolen amount scales with the attacker's deposit size relative to total supply and the FeeReceiver balance. This is a direct, concrete, on-chain-provable transfer of yield from legitimate holders to the attacker.

## Likelihood Explanation

- Requires no special role, no governance capture, no leaked keys — only ETH to deposit.
- `FeeReceiver` accumulates rewards continuously and its balance is publicly visible on-chain.
- `updateRSETHPrice()` is public; the attacker can trigger the price update themselves when the price increase is within `pricePercentageLimit`, or front-run the manager's routine oracle update otherwise.
- The 8-day withdrawal delay locks capital but does not eliminate profit; the `expectedAssetAmount` is locked at initiation and the price is monotonically non-decreasing under normal conditions.
- If `instantWithdrawal` is enabled for ETH, the attack collapses to a single block with no delay risk.
- Any MEV bot monitoring the mempool can execute this profitably whenever the FeeReceiver balance justifies gas costs.

## Recommendation

Add an access-control modifier to `sendFunds()` so only an authorized role can trigger the reward flush:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

As defense-in-depth, restrict `receiveFromRewardReceiver()` in `LRTDepositPool` to only accept calls from the registered `FeeReceiver` address, preventing arbitrary ETH injection from any third party.

## Proof of Concept

```solidity
// Foundry fork test outline

function test_sendFunds_yieldTheft() public {
    // Setup: FeeReceiver has 10 ETH in MEV rewards
    vm.deal(address(feeReceiver), 10 ether);

    uint256 priceBefore = lrtOracle.rsETHPrice();

    // Step 1: Attacker deposits 100 ETH at pre-reward price
    vm.startPrank(attacker);
    lrtDepositPool.depositETH{value: 100 ether}(0, "");
    uint256 rsETHBalance = rsETH.balanceOf(attacker);

    // Step 2: Attacker triggers reward flush — no role required
    feeReceiver.sendFunds();

    // Step 3: Attacker updates oracle price (public function)
    lrtOracle.updateRSETHPrice();
    uint256 priceAfter = lrtOracle.rsETHPrice();
    assertGt(priceAfter, priceBefore);

    // Step 4: Attacker initiates withdrawal at elevated price
    rsETH.approve(address(withdrawalManager), rsETHBalance);
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHBalance, "");

    // expectedAssetAmount locked at elevated rate > 100 ETH
    (, uint256 expectedAmount,,) = withdrawalManager.getUserWithdrawalRequest(ETH_TOKEN, attacker, 0);
    assertGt(expectedAmount, 100 ether); // attacker profits from MEV rewards

    vm.stopPrank();
}
```
