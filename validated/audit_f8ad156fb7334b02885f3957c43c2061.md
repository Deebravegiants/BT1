### Title
Single Byzantine Participant Can Replay Old `VerifyForeignTransactionResponse` to Resolve Any Pending Foreign-Tx Request — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over `response.payload_hash`. It does **not** verify that `response.payload_hash` is the canonical hash of the `request` argument being resolved. A single attested-but-Byzantine MPC participant can therefore replay any previously produced, on-chain-visible `VerifyForeignTransactionResponse` to immediately drain any pending `verify_foreign_transaction` yield queue — bypassing the threshold signing requirement for the new request entirely.

---

### Finding Description

`respond_verify_foreign_tx` performs two independent checks and then resolves pending yields:

1. **Caller check** — the caller must be an attested participant.
2. **Signature check** — the ECDSA signature in `response` must verify against `response.payload_hash` under the domain's root public key. [1](#0-0) 

Critically, the contract never checks that `response.payload_hash` equals `hash(request, extracted_values)`. The `request` parameter is used **only** to look up the pending yield queue: [2](#0-1) 

Because every `VerifyForeignTransactionResponse` that has ever been submitted on-chain is publicly visible, any single attested participant can:

1. Collect a previously valid response `S_old` (produced by the threshold protocol for request `R_old`).
2. When a new `verify_foreign_transaction` request `R_new` is queued, immediately call `respond_verify_foreign_tx(R_new, S_old)`.
3. The contract accepts the call: the signature over `hash(R_old, values_old)` is valid under the root key, and `R_new` has a non-empty yield queue.
4. All pending yields for `R_new` are resolved with `S_old` — the response for a completely different (or stale) transaction.

This is the direct analog of the EBTCToken permit-replay issue: just as an issued permit could not be invalidated and could be used to restore a revoked approval, an issued `VerifyForeignTransactionResponse` cannot be invalidated and can be replayed against any future pending request.

The `ForeignTxSignPayloadV1` structure that determines `payload_hash` binds the hash to the original `request` (including `tx_id`) and `extracted_values`: [3](#0-2) 

But the contract never recomputes this hash from the incoming `request` to cross-check it against `response.payload_hash`.

---

### Impact Explanation

**High — Cross-chain replay / forged foreign-chain verification.**

A single Byzantine MPC participant (below signing threshold) can resolve any pending `verify_foreign_transaction` request with a stale or mismatched response, without the threshold of nodes ever verifying the target transaction. Concretely:

- A transaction that was reorganized out of the foreign chain can be "re-verified" by replaying its old response against a fresh request, producing a valid MPC-signed attestation that the bridge contract will accept.
- Any user's pending `verify_foreign_transaction` request can be front-run and resolved with an incorrect payload, causing the bridge to act on fabricated extracted values (e.g., a wrong block hash or log data).

This matches: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

**Medium.** Every `VerifyForeignTransactionResponse` is submitted as a NEAR transaction and is permanently visible on-chain. Any attested participant can collect these responses at zero cost. Front-running a new `verify_foreign_transaction` request requires only that the Byzantine node submits its replay call before the honest leader submits the legitimate response — a standard NEAR transaction-ordering race that is straightforward to win given the node's direct access to the NEAR RPC.

---

### Recommendation

Inside `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` and the `values` embedded in the response, and assert equality before accepting the call:

```rust
// Pseudocode
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),   // values must be part of the response DTO
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This binds each response irrevocably to the request it was produced for, making replay against a different request impossible — the same fix the EBTCToken report recommended (adding a nonce to invalidate stale permits).

---

### Proof of Concept

```
1. Honest flow: user submits verify_foreign_transaction(R_old).
   MPC threshold signs → respond_verify_foreign_tx(R_old, S_old) accepted on-chain.
   S_old = { payload_hash: hash(R_old, values_old), signature: σ } is now public.

2. Attack:
   a. Victim submits verify_foreign_transaction(R_new)  [different tx_id].
      Pending yield Y_new is queued under R_new.

   b. Byzantine node (single attested participant) calls:
        respond_verify_foreign_tx(R_new, S_old)

   c. Contract checks:
      - Caller is attested participant ✓
      - verify_ecdsa(σ, hash(R_old, values_old), root_pk) == Ok ✓
      - pending_verify_foreign_tx_requests[R_new] is non-empty ✓
      → resolve_yields_for(R_new, S_old) drains Y_new.

   d. Victim's NEAR transaction resolves with S_old:
        VerifyForeignTransactionResponse {
            payload_hash: hash(R_old, values_old),   // ← wrong transaction
            signature: σ,                             // ← valid MPC signature
        }

3. Victim (or attacker) presents S_old to the bridge contract as proof that
   R_new was verified. The MPC signature is genuine; the bridge cannot
   distinguish it from a legitimately produced response for R_new.
``` [4](#0-3) [5](#0-4)

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
