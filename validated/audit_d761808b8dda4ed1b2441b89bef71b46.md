### Title
Missing CKD Response Validation for Legacy `AppPublicKey` Variant Allows Single Byzantine Participant to Corrupt Pending Requests - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` contract method performs no cryptographic validation of the `CKDResponse` when the legacy `AppPublicKey` (single G1 point) variant is used. A single Byzantine attested participant can call `respond_ckd` directly with arbitrary `big_y` / `big_c` values, consuming the pending request and delivering a cryptographically invalid response to the user — without any threshold agreement.

---

### Finding Description

In `respond_ckd`, response validation is conditional on the `CKDAppPublicKey` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VALIDATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk||app_id), mpc_pk)`, which cryptographically binds the response to the master secret key and the user's ephemeral key. [2](#0-1) 

For `AppPublicKey` (the legacy "privately verifiable" variant), **no equivalent check exists**. The contract accepts any `big_y` and `big_c` byte strings without verifying they are valid G1 points, let alone that they satisfy the CKD relation. The `respond_ckd` function then immediately calls `resolve_yields_for`, which drains the entire pending-request queue and delivers the unvalidated bytes to all waiting callers. [3](#0-2) 

The `AppPublicKey` variant is still actively used in production (the node handles it, e2e tests cover it, and the README documents it as the "legacy" path). [4](#0-3) 

The `respond_ckd` method is a public contract entry point. The only gate is `assert_caller_is_attested_participant_and_protocol_active`. TEE attestation proves the node image is correct, but it does not prevent an attested participant from calling the contract method directly (outside the node software) with hand-crafted arguments. [5](#0-4) 

---

### Impact Explanation

**Medium — request-lifecycle and contract execution-flow manipulation breaking production safety invariants.**

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Monitor the NEAR chain for a pending `AppPublicKey` CKD request.
2. Race the honest nodes by calling `respond_ckd` directly with arbitrary `big_y` / `big_c` values.
3. The contract accepts the response unconditionally and calls `resolve_yields_for`, which drains the entire pending-request queue for that key and delivers the malicious bytes to the user's yield-resume promise. [6](#0-5) 

Consequences:
- The pending request is permanently consumed; honest nodes can no longer respond to it.
- The user receives a cryptographically invalid `(big_y, big_c)` pair. If the user's client verifies the response (as the example CLI does), the verification fails and the user must resubmit — losing their deposit and waiting again.
- If the user's client does not verify (e.g., a third-party integration), the user silently derives a wrong key that is not bound to the MPC master secret, breaking the confidentiality guarantee of CKD. [7](#0-6) 

The invariant broken is: *every resolved CKD response must be cryptographically bound to the MPC master secret and the user's ephemeral key*. This invariant is enforced for `AppPublicKeyPV` but not for `AppPublicKey`.

---

### Likelihood Explanation

**Medium.** The attacker must be a TEE-attested participant (a real network member), but only **one** is required — well below the signing threshold. The attack is a simple race: submit a direct contract call to `respond_ckd` before the honest leader node does. On NEAR, transaction ordering within a block is deterministic and observable, making the race feasible for a participant who monitors the mempool or block production.

---

### Recommendation

1. **Deprecate `AppPublicKey` in favour of `AppPublicKeyPV`**, which has full on-chain verification. Emit a deprecation warning or reject new `AppPublicKey` requests.
2. **If `AppPublicKey` must be retained**, require threshold agreement before accepting a response (e.g., collect `t` identical `(big_y, big_c)` submissions from distinct attested participants before resolving the yield), mirroring the threshold trust model used for signing.
3. **At minimum**, document explicitly that `AppPublicKey` CKD responses carry no on-chain integrity guarantee and that clients **must** perform client-side verification before using the derived key.

---

### Proof of Concept

1. Alice submits `request_app_private_key` with `AppPublicKey(pk1)` and `domain_id = 2`.
2. The contract stores the pending request and creates a yield-resume promise.
3. Byzantine participant Eve (attested, but acting outside the node software) observes the pending request on-chain.
4. Eve calls `respond_ckd(ckd_request, CKDResponse { big_y: [1u8;48], big_c: [2u8;48] })` directly.
5. The contract passes the `AppPublicKey(_) => {}` branch with no check, then calls `resolve_yields_for`, draining Alice's pending queue with Eve's malicious bytes.
6. Alice's yield-resume fires with `big_y = [1u8;48]`, `big_c = [2u8;48]` — neither a valid G1 point nor a valid CKD output.
7. The honest nodes' subsequent `respond_ckd` calls return `RequestNotFound` because the queue is already drained.

This is directly confirmed by the existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists`, which passes `[1u8;48]` / `[2u8;48]` (invalid G1 encodings) as the response for an `AppPublicKey` request and expects success — demonstrating the absence of any output validation. [8](#0-7)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

fn decrypt_secret_and_verify(
    big_y: &Bls12381G1PublicKey,
    big_c: &Bls12381G1PublicKey,
    private_key: Scalar,
    app_id: CkdAppId,
    mpc_public_key: &Bls12381G2PublicKey,
) -> Result<[u8; BLS12381G1_PUBLIC_KEY_SIZE]> {
    let big_y: G1Projective = big_y.try_into().context("invalid G1 point")?;
    let big_c: G1Projective = big_c.try_into().context("invalid G1 point")?;
    let mpc_public_key: G2Projective = mpc_public_key.try_into().context("invalid G2 point")?;

    // decrypt the secret
    let secret = big_c - big_y * private_key;

    // verify the secret
    if !verify(&mpc_public_key, app_id.as_ref(), &secret) {
        anyhow::bail!("Verification failed!");
    }

    // return the secret as bytes
    Ok(secret.to_compressed())
