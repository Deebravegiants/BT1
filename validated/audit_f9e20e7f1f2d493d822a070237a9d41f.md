### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` accepts a node-supplied `response.payload_hash` and only checks that the ECDSA signature is valid over that hash under the root public key. It never verifies that `payload_hash` is actually the canonical hash of `ForeignTxSignPayload{request, extracted_values}` for the specific `request` being resolved. A single Byzantine MPC participant node (below threshold) that possesses any previously produced threshold signature can replay it against a different pending request, delivering a forged `payload_hash` to the waiting user.

### Finding Description

In `respond_verify_foreign_tx` the contract performs:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // fully attacker-controlled

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // verified only against itself
    &secp_pk,               // root key, no tweak
)
.is_ok()
``` [1](#0-0) 

After this check passes, the entire `response` (including the unverified `payload_hash`) is serialised and delivered to every waiting yield:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),   // response.payload_hash is never bound to request
)
``` [2](#0-1) 

The canonical hash that *should* be verified is `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))`: [3](#0-2) 

The SDK-level verifier (`ForeignChainSignatureVerifier::verify_signature`) does enforce this binding, but it lives in the caller's contract, not in the MPC contract itself: [4](#0-3) 

The MPC contract therefore provides no on-chain guarantee that the `payload_hash` it delivers corresponds to the foreign transaction the user originally requested.

### Impact Explanation

A Byzantine MPC participant node that participated in signing request A obtains the threshold signature `sig_A` over `hash_A`. It then calls `respond_verify_foreign_tx(request_B, {payload_hash: hash_A, signature: sig_A})` for any other pending request B. The contract accepts the call (signature is valid over `hash_A` under the root key; request B exists in `pending_verify_foreign_tx_requests`). The yield for request B is resolved and the user receives `{payload_hash: hash_A, signature: sig_A}`.

- Any bridge contract that trusts the contract-delivered `payload_hash` without independently recomputing the expected hash receives a forged attestation — it is told that `hash_A` (the hash of a different transaction's payload) was verified for its own transaction. This enables invalid bridge execution or double-spend conditions.
- Even bridge contracts that do use the SDK verifier have their pending request consumed and must resubmit, constituting a targeted DoS against specific requests.

This maps to: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

- Only one attested participant node is required; no threshold collusion is needed.
- The node obtains `sig_A` legitimately by participating in the MPC signing protocol for any prior request.
- Any user can create a pending request B (costs 1 yoctoNEAR), giving the attacker a target.
- The attack is repeatable across all pending requests the attacker wishes to poison.

### Recommendation

Inside `respond_verify_foreign_tx`, recompute the expected payload hash from the `request` fields and reject any response whose `payload_hash` does not match. Because the contract does not know the `extracted_values` (they are off-chain), the simplest fix is to **not include `payload_hash` in the on-chain response at all** and instead have the contract recompute it from the request and the values the node must also submit on-chain. Alternatively, require the node to submit the `extracted_values` alongside the response so the contract can compute and verify `SHA-256(borsh(ForeignTxSignPayload{request, values})) == response.payload_hash` before resolving the yield.

### Proof of Concept

1. User Alice submits `verify_foreign_transaction` for Bitcoin tx `[0xAA; 32]` → pending request A is stored.
2. User Bob submits `verify_foreign_transaction` for Bitcoin tx `[0xBB; 32]` → pending request B is stored.
3. The MPC network honestly processes request A and produces `sig_A` over `hash_A = SHA-256(borsh(ForeignTxSignPayload{request_A, values_A}))`.
4. Malicious node Eve (one attested participant) calls:
   ```
   respond_verify_foreign_tx(
       request = request_B,          // Bob's pending request
       response = {
           payload_hash: hash_A,     // hash of Alice's transaction
           signature: sig_A,         // valid signature over hash_A
       }
   )
   ```
5. Contract checks: `verify_ecdsa_signature(sig_A, hash_A, root_pk)` → **passes**.
6. Contract resolves Bob's yield with `{payload_hash: hash_A, signature: sig_A}`.
7. Bob's bridge contract receives a response claiming `hash_A` was verified for his transaction `[0xBB; 32]`. If the bridge contract does not independently recompute the expected hash, it proceeds as if Bob's transaction was verified — it was not. [5](#0-4)
