### Title
Wrong Condition in ETH Deposit Limit Check Allows Limit Bypass - (File: contracts/LRTDepositPool.sol)

### Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an incorrect condition for ETH deposits: it omits the incoming `amount` from the comparison, so the deposit limit is never enforced against the deposit that actually pushes total deposits over the cap.

### Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit`, the ETH branch checks only whether the already-recorded total exceeds the limit, without adding the current deposit amount:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← missing `+ amount`
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC-20
}
``` [1](#0-0) 

For every ERC-20 asset the check is `totalAssetDeposits + amount > limit` (correct). For ETH it is `totalAssetDeposits > limit` (wrong). The result is that any ETH deposit whose amount alone would push the running total past the configured cap is accepted, because the check only fires once the total has *already* exceeded the limit from prior deposits.

The function is called unconditionally inside `_beforeDeposit`, which is the sole guard used by the public `depositETH` entry point:

```solidity
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [2](#0-1) 

`depositETH` is callable by any address with no role requirement: [3](#0-2) 

### Impact Explanation
The ETH deposit cap configured by the admin via `lrtConfig.depositLimitByAsset` is not enforced for the deposit that crosses the boundary. An unprivileged depositor can mint rsETH against ETH in excess of the intended protocol limit, inflating rsETH supply beyond what the cap was designed to allow. This constitutes the contract failing to deliver its promised deposit-limit guarantee.

**Impact: Low** — Contract fails to deliver promised returns (deposit limit enforcement), but no direct value is lost from existing holders in a single transaction; the over-issuance is bounded by one deposit's worth of ETH.

### Likelihood Explanation
Any depositor can trigger this by submitting an ETH deposit whose amount, when added to the current total, exceeds the limit. No special role, front-running, or external dependency is required. The condition is reachable on every call to `depositETH` when `totalAssetDeposits` is near the cap.

### Recommendation
Apply the same `+ amount` inclusion used for ERC-20 assets:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
1. Admin sets ETH deposit limit to 100 ETH via `lrtConfig`.
2. Current `getTotalAssetDeposits(ETH_TOKEN)` = 99 ETH.
3. Depositor calls `depositETH{value: 10 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)` evaluates `99 > 100` → `false` → no revert.
5. 10 ETH is accepted and rsETH is minted, bringing total to 109 ETH — 9 ETH above the configured cap.
6. For a non-ETH asset in the same scenario, step 4 would evaluate `99 + 10 > 100` → `true` → revert, correctly blocking the deposit.
