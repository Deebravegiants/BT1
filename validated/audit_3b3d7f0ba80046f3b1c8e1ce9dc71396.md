Audit Report

## Title
Stale `CrossChainRateReceiver` Rate Enables Unbounded agETH Over-Minting in `AGETHPoolV3` — (`contracts/agETH/AGETHPoolV3.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the last stored `rate` with no staleness enforcement, despite storing `lastUpdated`. `AGETHPoolV3` uses this rate as the denominator when computing agETH mint amounts, and unlike `RSETHPoolV3`, has no pause mechanism and no daily mint limit. Because agETH is yield-bearing, any staleness window guarantees the oracle rate is lower than the true rate, causing every deposit during that window to mint more agETH than the deposited collateral warrants, directly leading to protocol insolvency.

## Finding Description

**Root cause:** `CrossChainRateReceiver.getRate()` stores `lastUpdated` but never enforces a maximum age:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L102-105
function getRate() external view returns (uint256) {
    return rate;  // no staleness check
}
``` [1](#0-0) 

The rate is only updated when a LayerZero message arrives via `lzReceive()`. Between messages, `rate` is frozen at its last value: [2](#0-1) 

**Exploit path:**

`AGETHPoolV3.deposit(address, uint256, string)` is public with no pause gate and no mint cap: [3](#0-2) 

It calls `viewSwapAgETHAmountAndFee`, which computes:

```
agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate
```

where `agETHToETHrate` comes directly from the stale oracle: [4](#0-3) 

`AGETHPoolV3.getRate()` delegates to `IOracle(agETHOracle).getRate()` with no additional check: [5](#0-4) 

**Why existing checks are insufficient:**

`RSETHPoolV3` has both a `whenNotPaused` modifier and a `limitDailyMint` modifier on both `deposit()` overloads: [6](#0-5) [7](#0-6) 

`AGETHPoolV3` has neither. It has no `paused` state variable, no `whenNotPaused` modifier, no `dailyMintLimit`, and no `limitDailyMint` modifier anywhere in the contract. [8](#0-7) 

The collateral token oracle (`ChainlinkOracleForRSETHPoolCollateral`) does enforce staleness, creating the asymmetric scenario where the token rate is fresh but the agETH rate is stale: [9](#0-8) 

## Impact Explanation

**Critical — Protocol insolvency.**

Let `R_true` = current true agETH/ETH rate, `R_stale` = stale oracle rate (`R_stale < R_true` always, because agETH is yield-bearing and its rate monotonically increases).

For a deposit of `N` ETH-equivalent collateral:
- Correct mint: `N / R_true`
- Actual mint: `N / R_stale` (strictly larger)
- Excess agETH: `N * (R_true - R_stale) / (R_stale * R_true)`

The attacker bridges the excess agETH to L1 and redeems it for ETH. The protocol has issued more agETH than the deposited collateral backs. With no daily mint cap, a single large deposit extracts the full excess in one transaction, directly causing protocol insolvency.

## Likelihood Explanation

- Cross-chain oracle staleness is a natural, non-adversarial condition. LayerZero message delays, fee shortfalls on the provider side, or infrequent `updateRate()` calls are routine operational states.
- agETH's yield-bearing nature guarantees the stale rate is always lower than the true rate — the direction of the exploit is deterministic and requires no oracle manipulation.
- The attack requires no privileged role, no front-running, and no external protocol compromise — only a deposit during a staleness window.
- The `AGETHRateReceiver` constructor sets no update frequency enforcement: [10](#0-9) 

## Recommendation

1. **Enforce a staleness threshold in `CrossChainRateReceiver.getRate()`:**
   ```solidity
   uint256 public constant MAX_STALENESS = 24 hours;
   function getRate() external view returns (uint256) {
       require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate stale");
       return rate;
   }
   ```

2. **Add a staleness check in `AGETHPoolV3.viewSwapAgETHAmountAndFee()`** as defence-in-depth, reading `AGETHRateReceiver.lastUpdated` directly before using the rate.

3. **Add a `limitDailyMint` modifier to `AGETHPoolV3.deposit()`** analogous to the one present in `RSETHPoolV3`, to bound worst-case exposure during any staleness window.

4. **Add a `pause()` / `whenNotPaused` mechanism** to `AGETHPoolV3` so operators can halt deposits if a staleness event is detected.

## Proof of Concept

**Minimal differential test (no fork required):**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

contract StaleRatePoC {
    /// @notice Proves that a stale (lower) agETH rate always produces excess mint
    /// @param depositETH  collateral in ETH-equivalent (e.g. 1000e18)
    /// @param staleRate   agETH/ETH rate from 24h ago (e.g. 1.049863e18 at 5% APY)
    /// @param freshRate   current true agETH/ETH rate  (e.g. 1.050000e18)
    function prove(
        uint256 depositETH,
        uint256 staleRate,
        uint256 freshRate
    ) external pure returns (uint256 excess) {
        require(freshRate > staleRate, "yield-bearing invariant");
        uint256 mintedStale = depositETH * 1e18 / staleRate;
        uint256 mintedFresh = depositETH * 1e18 / freshRate;
        excess = mintedStale - mintedFresh;
        assert(excess > 0);
    }
}
```

**Concrete numbers:** agETH at ~5% APY, 24 h staleness → `R_true / R_stale ≈ 1.000137`. On a 1,000 ETH deposit, excess ≈ 0.137 agETH ≈ 0.137 ETH of unbacked issuance per deposit. With no daily mint cap, an attacker using a single large deposit (e.g. 100,000 ETH) extracts ~13.7 ETH of unbacked agETH in one transaction.

**Fork test path:**
1. Fork the deployment chain at a block where `AGETHRateReceiver.lastUpdated` is >24h old.
2. Call `AGETHPoolV3.deposit(token, largeAmount, "")` as an unprivileged EOA.
3. Compare `agETH.balanceOf(attacker)` against `largeAmount * tokenRate / trueAgETHRate` — the difference is the unbacked excess.
4. Confirm no revert occurs (no staleness guard, no mint cap, no pause).
