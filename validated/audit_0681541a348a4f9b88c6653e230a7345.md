### Title
`LRTOracle._updateRsETHPrice` Decrease Safety Check Uses All-Time-High as Reference, Creating Asymmetric and Imprecise Pause Trigger - (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle._updateRsETHPrice()` applies a `pricePercentageLimit` safety check to both price increases and decreases, but both branches measure deviation relative to `highestRsethPrice` (the all-time high). The decrease branch is imprecise: it measures the **cumulative drop from ATH**, not the **current price movement**. This causes the protocol to pause deposits and withdrawals when the current price movement is small but the cumulative deviation from ATH crosses the threshold — an asymmetric and imprecise safety check directly analogous to the `Buoy3Pool.safetyCheck` report.

---

### Finding Description

In `_updateRsETHPrice()`, the increase check and decrease check are:

**Increase check** (lines 252–267):
```solidity
uint256 priceDifference = newRsETHPrice - highestRsethPrice;
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```
This only fires when `newRsETHPrice > highestRsethPrice` (a new ATH is being set), and measures how much the new ATH exceeds the previous ATH.

**Decrease check** (lines 270–282):
```solidity
uint256 diff = highestRsethPrice - newRsETHPrice;
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
```
This fires whenever `newRsETHPrice < highestRsethPrice` (any time below ATH), and measures the **total cumulative drop from ATH**, not the drop from the last stored `rsETHPrice`.

The asymmetry:
- The increase check measures the **incremental** change from the previous ATH.
- The decrease check measures the **cumulative** deviation from ATH, regardless of how long ago the ATH was set or how small the current price movement is.

This means the same `pricePercentageLimit` value produces fundamentally different behavior depending on direction. A 1% price drop can trigger a protocol pause if the price has already been 9% below ATH, while a 9% drop from ATH itself does not trigger a pause. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

When `isPriceDecreaseOffLimit` is true, the function pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` itself:

```solidity
if (!lrtDepositPool.paused()) lrtDepositPool.pause();
if (!withdrawalManager.paused()) withdrawalManager.pause();
_pause();
return;
```

This freezes all user deposits and withdrawals. Because the check is imprecise, the protocol can be paused when the current price movement is minimal (e.g., 1–2%), causing **temporary freezing of funds** for all users.

**Impact**: Medium — Temporary freezing of funds (deposits and withdrawals). [3](#0-2) 

---

### Likelihood Explanation

- `updateRSETHPrice()` is a **public** function callable by any address, including unprivileged users.
- The scenario arises naturally: if the price has been below ATH for any period (e.g., 9% below ATH), a small further drop (e.g., 1.1%) crosses the 10% threshold and triggers the pause.
- No oracle manipulation or privileged access is required. The attacker only needs to call `updateRSETHPrice()` at the moment the cumulative drop from ATH crosses the threshold. [4](#0-3) 

---

### Recommendation

The decrease check should measure the drop from the **previous stored price** (`previousPrice`/`rsETHPrice`), not from `highestRsethPrice`. This makes the check symmetric with the increase check (both measure incremental movement) and prevents unnecessary pausing due to accumulated historical deviation:

```solidity
// Current (imprecise):
uint256 diff = highestRsethPrice - newRsETHPrice;
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);

// Recommended (measures current movement):
uint256 diff = previousPrice - newRsETHPrice;
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && previousPrice > 0 && diff > pricePercentageLimit.mulWad(previousPrice);
```

---

### Proof of Concept

1. Admin sets `pricePercentageLimit = 0.1e18` (10%).
2. State: `highestRsethPrice = 1.2e18`, `rsETHPrice = 1.09e18` (already 9.2% below ATH — within limit).
3. A routine price update computes `newRsETHPrice = 1.07e18` (a 1.8% drop from the current price).
4. Decrease check evaluates:
   - `diff = 1.2e18 − 1.07e18 = 0.13e18`
   - `limit = 0.1 × 1.2e18 = 0.12e18`
   - `0.13e18 > 0.12e18` → `isPriceDecreaseOffLimit = true`
5. Protocol pauses `LRTDepositPool` and `LRTWithdrawalManager`, freezing all user deposits and withdrawals.
6. The actual price movement was only **1.8%**, well within the intended 10% safety threshold.
7. Any user calling the public `updateRSETHPrice()` at this moment triggers the freeze. [5](#0-4) [6](#0-5)
