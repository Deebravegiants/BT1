### Title
Proposal Deposits Permanently Frozen on Update Execution — (`crates/contract/src/update.rs`)

### Summary

When `do_update` executes a winning contract/config proposal, it unconditionally clears **all** pending proposals. The NEAR storage-staking deposits paid by proposers of non-winning (cleared) proposals are never refunded, permanently locking those funds in the contract with no retrieval mechanism.

### Finding Description

`propose_update` requires each proposer to attach a deposit exactly equal to `ProposedUpdates::required_deposit(&update)`, which covers the storage cost of holding the proposal bytes on-chain. The excess is refunded immediately, but the exact required amount is retained by the contract. [1](#0-0) 

When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()` to wipe every other pending proposal: [2](#0-1) 

Clearing the entries frees the on-chain storage, but the NEAR tokens that were deposited to stake that storage are **not returned to the original proposers**. There is no rescue, withdrawal, or refund path in the contract for these accumulated tokens.

The deposit per contract-code proposal can be substantial. `bytes_used` accounts for up to 128 participant-vote slots plus the full contract binary: [3](#0-2) 

A 1.5 MB contract binary (the tested maximum) produces a required deposit on the order of **~15 NEAR** at current storage pricing. With multiple concurrent proposals — a normal governance pattern shown in `test_propose_update_contract_many` — every non-winning proposer permanently loses their deposit on each upgrade cycle.

### Impact Explanation

Every contract upgrade cycle that has more than one competing proposal permanently destroys the deposits of all non-winning proposers. The funds accumulate in the contract's balance with no on-chain path to recover them. This is a direct, permanent loss of NEAR tokens controlled by the chain-signature contract, matching the **Medium** impact tier: balance/accounting invariant broken in production safety logic.

### Likelihood Explanation

The governance flow explicitly supports multiple concurrent proposals (the README and tests demonstrate this). Any upgrade cycle where two or more participants independently propose different updates — a realistic scenario during active governance — triggers the loss. No adversarial action is required; the loss is a deterministic consequence of normal operation.

### Recommendation

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(proposer_account).transfer(deposit)` refund for each. The proposer's `AccountId` must be stored in `UpdateEntry` at proposal time (it is not currently stored there). Alternatively, track a `proposer → deposit` map alongside `entries` and drain it on `do_update`.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR deposit. Contract stores the entry and retains the deposit.
2. Participant B calls `propose_update` with a different binary, attaching ~10 NEAR deposit.
3. Threshold participants vote for B's proposal. `vote_update` calls `do_update(&B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry is deleted.
5. A's ~10 NEAR deposit remains in the contract balance. No refund is issued. No contract method exists to retrieve it. [4](#0-3) [5](#0-4)

    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
