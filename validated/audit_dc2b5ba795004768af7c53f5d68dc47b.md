Audit Report

## Title
Floor-Based Period Reset in `_checkAndUpdateDailyFeeMintLimit` Allows 2× Daily Fee Cap in 23 Hours — (`contracts/LRTOracle.sol`)

## Summary
When no `updateRSETHPrice()` call occurs for more than 24 hours, `_checkAndUpdateDailyFeeMintLimit` resets `feePeriodStartTime` to the integer-division floor of the elapsed day boundary via `getCurrentPeriodStartTime()`, not to `block.timestamp`. This anchors the new period one hour in the past (in the worst case), causing it to expire only 23 hours after the first reset. A second full `maxFeeMintAmountPerDay` mint becomes available at that point, yielding 2× the intended daily cap in a 23-hour window and diluting rsETH holders beyond the protocol's stated limit.

## Finding Description

`updateRSETHPrice()` is unrestricted — `public` with no role check: [1](#0-0) 

`getCurrentPeriodStartTime()` floors to the nearest completed day boundary using integer division: [2](#0-1) 

`_checkAndUpdateDailyFeeMintLimit` uses this floored value to set `feePeriodStartTime` on reset: [3](#0-2) 

**Concrete exploit path (T0 = initial `feePeriodStartTime`):**

| Time | Action | `feePeriodStartTime` after | Period expires |
|------|--------|---------------------------|----------------|
| T0 | Initialized | T0 | T0 + 24h |
| T0 + 25h | First `updateRSETHPrice()` | T0 + 24h (floored) | T0 + 48h |
| T0 + 48h | Second `updateRSETHPrice()` | T0 + 48h | T0 + 72h |

At T0+25h: `daysElapsed = (T0+25h − T0) / 86400 = 1` (integer division), so `getCurrentPeriodStartTime()` returns `T0 + 24h` — one hour in the past. The new period therefore expires at `T0 + 48h`, only **23 hours** after the first reset. The second full `maxFeeMintAmountPerDay` is available immediately at T0+48h.

The fee minting condition `totalETHInProtocol > previousTVL` is satisfied naturally over a 25-hour gap as staking rewards accrue: [4](#0-3) 

The daily cap check only guards against exceeding the limit within a single period; it does not prevent the period from being shortened by the floor reset: [5](#0-4) 

## Impact Explanation

`maxFeeMintAmountPerDay` is the sole on-chain protection against excessive rsETH dilution from protocol fee minting. This bug allows that cap to be consumed twice in 23 hours instead of 48 hours — a 2× acceleration of yield extraction from rsETH holders to the treasury. This matches **High: Theft of unclaimed yield**, as rsETH holders lose yield to the treasury at a rate exceeding the protocol's intended daily limit.

## Likelihood Explanation

The required precondition — a >24-hour gap between `updateRSETHPrice()` calls — is realistic: keeper/bot failures, network congestion, or deliberate withholding of updates can all produce this gap. Once the gap exists, the exploit requires only two calls to the public `updateRSETHPrice()` function with no special privileges. Any EOA can execute it. The TVL increase condition is satisfied automatically by natural staking reward accrual over the gap period.

## Recommendation

Replace the floor-based reset with a forward-anchored reset. Either set `feePeriodStartTime = block.timestamp` to guarantee a full 24-hour window from the moment of reset, or use `feePeriodStartTime + 1 days` to preserve calendar-day alignment without shortening the next period:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
-       feePeriodStartTime = getCurrentPeriodStartTime();
+       feePeriodStartTime = block.timestamp; // or feePeriodStartTime + 1 days
    }
    ...
}
```

Using `feePeriodStartTime + 1 days` is preferable if strict alignment is required, as it prevents any single period from being shorter than 24 hours regardless of when the call arrives.

## Proof of Concept

Foundry test demonstrating the invariant violation:

```solidity
function testDoubleFeeMintIn23Hours() public {
    uint256 T0 = oracle.feePeriodStartTime();

    // Simulate 25-hour gap (staking rewards accrue naturally)
    vm.warp(T0 + 25 hours);
    oracle.updateRSETHPrice();
    // feePeriodStartTime is now T0 + 24h (floored), not T0 + 25h
    assertEq(oracle.feePeriodStartTime(), T0 + 24 hours);
    assertEq(oracle.currentPeriodMintedFeeAmount(), maxFeeMintAmountPerDay);

    // Only 23 more hours pass — new period expires at T0+48h
    vm.warp(T0 + 48 hours);
    oracle.updateRSETHPrice();
    // Second full cap minted — 2× maxFeeMintAmountPerDay in 23 hours
    // Invariant violated: daily dilution cap bypassed
}
```

The test requires: oracle deployed with `feePeriodStartTime = T0`, `maxFeeMintAmountPerDay` set to a non-zero value, and TVL configured to increase by enough over the 25-hour gap to reach the fee cap on both calls.
