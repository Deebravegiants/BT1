### Title
Unprotected `init` Initializer Allows Any Account to Hijack MPC Network Participant Set - (File: `crates/contract/src/lib.rs`)

### Summary
The `MpcContract::init` function is decorated with `#[init]` but not `#[private]`. Because the deployment process is explicitly non-atomic (contract deployment and `init` are separate transactions), any account can race to call `init` before the legitimate operator does, installing an arbitrary participant set and threshold and seizing full control of the MPC network.

### Finding Description
`MpcContract::init` at line 1926 carries only the `#[init]` attribute, which in the NEAR SDK exclusively prevents *re-initialization* once state exists. It does **not** restrict the caller. Any NEAR account can invoke it on a freshly deployed, uninitialized contract.

```rust
// crates/contract/src/lib.rs:1924-1929
#[handle_result]
#[init]                          // ← no #[private], no owner assertion
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

By contrast, both sibling initializers are correctly restricted:

```rust
// init_running – line 1976
#[private]
#[init]
pub fn init_running(...) -> Result<Self, Error> { ... }

// migrate – line 2060
#[private]
#[init(ignore_state)]
pub fn migrate() -> Result<Self, Error> { ... }
``` [2](#0-1) [3](#0-2) 

Every documented deployment workflow separates the deploy step from the init step across distinct transactions. The localnet guide explicitly passes `without-init-call`:

```bash
# docs/localnet/localnet.md:131
near contract deploy mpc-contract.test.near use-file "$MPC_CONTRACT_PATH" \
  without-init-call network-config mpc-localnet sign-with-keychain send
``` [4](#0-3) 

The TEE cluster deployment script likewise separates `near_contract` (deploy) and `near_init` (init) into sequential, independently resumable phases:

```bash
# localnet/tee/scripts/rust-launcher/deploy-tee-cluster.sh:1669-1703
if should_run_from_start near_contract; then
    build_contract
    deploy_contract          # ← deploy only, no init

    init_contract            # ← separate transaction, later
``` [5](#0-4)

### Impact Explanation
An attacker who calls `init` first supplies their own `ThresholdParameters` — e.g., a single-participant set with threshold 1 pointing to an account they control. The contract transitions immediately to `Running` state with the attacker as the sole participant. From that position the attacker can:

- Call `vote_add_domains` to trigger key generation under their control.
- Call `respond` / `respond_ckd` to resolve any pending signature or CKD request with a forged response.
- Call `vote_new_parameters` unilaterally to lock out the legitimate operators permanently.

This constitutes **unauthorized threshold signature issuance and unauthorized access to MPC signing capability**, matching the Critical impact tier.

### Likelihood Explanation
NEAR transactions are publicly visible in the mempool and on-chain. Any observer can detect a contract deployment with no state and immediately submit an `init` call. The window exists on every fresh deployment and every contract migration that redeploys to a new account. No special privilege, leaked key, or TEE access is required — only a funded NEAR account.

### Recommendation
Add `#[private]` to `init`, mirroring the protection already applied to `init_running` and `migrate`:

```rust
#[handle_result]
#[private]          // ← add this
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

`#[private]` in the NEAR SDK asserts `predecessor_account_id == current_account_id`, so only the contract account itself (i.e., the deployer calling via a batch transaction or a self-call) can initialize it. Alternatively, the deployment scripts should be updated to use NEAR's `--initFunction` / `--initArgs` flags so that deployment and initialization are submitted as a single atomic batch transaction, eliminating the race window entirely.

### Proof of Concept
1. Operator submits: `near contract deploy v1.signer use-file mpc_contract.wasm without-init-call ...`
2. Attacker observes the deployment transaction finalized on-chain (contract state is empty).
3. Attacker submits:
   ```bash
   near contract call-function as-transaction v1.signer init \
     json-args '{"parameters":{"threshold":1,"participants":{"next_id":1,"participants":[["attacker.near",0,{"tls_public_key":"ed25519:ATTACKER_KEY","url":"https://attacker.example.com"}]]}}}' \
     prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
     sign-as attacker.near network-config mainnet sign-with-keychain send
   ```
4. The contract is now in `Running` state with `attacker.near` as the sole participant at threshold 1.
5. Operator's subsequent legitimate `init` call fails: `#[init]` panics because state already exists.
6. Attacker calls `vote_add_domains` to generate keys, then services all `sign` requests with attacker-controlled signatures. [7](#0-6)
