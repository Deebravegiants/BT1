### Title
Missing RSA Modulus Primality/Factor-Size Validation in `base_pke_rsa_ek_from_modulus` Allows Weak-Key PVE Ciphertext - (`src/cbmpc/api/pve_base_pke.cpp`)

### Summary

`base_pke_rsa_ek_from_modulus` accepts any 256-byte non-zero value as an RSA-2048 modulus with no primality or factor-size check. A malicious caller (or Byzantine protocol peer) can supply a smooth composite, obtain a valid `ek_blob`, pass it to `cbmpc_pve_encrypt`, and receive a well-formed PVE ciphertext. Because the modulus is trivially factorable, the attacker recovers the RSA-OAEP KEM shared secret and decrypts the AES-GCM payload.

### Finding Description

`base_pke_rsa_ek_from_modulus` performs only two checks on the caller-supplied modulus:

1. Byte-length == 256 (RSA-2048 size)
2. `BN_is_zero(n_bn)` — non-zero check [1](#0-0) 

After those two checks, the raw BIGNUM is handed directly to `rsa_pub_key_t::set`, which calls OpenSSL's `EVP_PKEY_fromdata` with only `n` and `e=65537`. OpenSSL does not validate that `n` is a semiprime, that its factors are large, or that it is odd (for a public-key-only import path). [2](#0-1) 

The resulting `rsa_pub_key_t` is serialized into a `base_pke_ek_blob_v1_t` and returned as a valid `ek_blob`. [3](#0-2) 

When that `ek_blob` is passed to `cbmpc_pve_encrypt` → `coinbase::api::pve::encrypt` → `pve_base_pke_rsa().encrypt` → `kem_pve_base_pke_t::encrypt` → `kem_aead_ciphertext_t::seal`, the encryption path calls `kem_policy_rsa_oaep_t::encapsulate`: [4](#0-3) 

`encapsulate` generates a random 32-byte `kem_ss`, then calls `pub_key.encrypt_oaep_with_seed(kem_ss, ...)`, which pads and performs a raw RSA modular exponentiation (`kem_ss^e mod n`) via `encrypt_raw`: [5](#0-4) 

OpenSSL's `EVP_PKEY_encrypt` with `RSA_NO_PADDING` succeeds for any odd modulus regardless of its factorization. The `kem_ss` is then used to derive an AES-GCM key via HKDF: [6](#0-5) 

The entire call chain produces a well-formed ciphertext with no error returned, even though `n` is a smooth composite.

### Impact Explanation

An attacker who supplies a smooth odd composite as the modulus (e.g., a product of many small primes totaling 2048 bits) can:

1. Call `cbmpc_pve_base_pke_rsa_ek_from_modulus(smooth_n, &ek)` → `SUCCESS`
2. Cause an honest party to call `cbmpc_pve_encrypt(NULL, curve, ek, label, x, &ct)` → `SUCCESS`
3. Factor `n` trivially (smooth numbers factor in polynomial time via trial division or ECM)
4. Compute `d = e^{-1} mod λ(n)` and decrypt the RSA-OAEP ciphertext to recover `kem_ss`
5. Derive the AES-GCM key via HKDF and decrypt the payload to recover the plaintext scalar `x`

In the PVE context, `x` is a private key share or equivalent secret material. This is a **Critical/High** impact: attacker-controlled input causes honest code to encrypt secret material under a key the attacker can break.

### Likelihood Explanation

The function `base_pke_rsa_ek_from_modulus` is explicitly designed for the HSM use case where only the public modulus is available externally. A Byzantine protocol participant or a malicious API caller can trivially construct a smooth 256-byte odd composite and pass it as the modulus. No special privileges are required. The attack requires only a single API call to construct the weak key and a second call to trigger encryption.

### Recommendation

Add the following validation inside `base_pke_rsa_ek_from_modulus` before constructing the key object:

1. **Odd check**: Reject `n` if `BN_is_odd(n_bn) == 0`.
2. **Minimum bit-length**: Reject if `BN_num_bits(n_bn) < RSA_KEY_LENGTH` (guards against leading-zero padding tricks).
3. **Small-factor trial division**: Run trial division against primes up to at least 2^16 (or use `BN_check_prime` / `RSA_check_key` with a primality test). OpenSSL's `EVP_PKEY_param_check` or `EVP_PKEY_public_check` with an appropriate provider can enforce this.
4. **Document the trust boundary**: If the modulus is expected to come from a trusted HSM, document that the caller is responsible for key quality and add an assertion or a compile-time note.

### Proof of Concept

```c
// Construct a smooth 256-byte odd composite:
// n = 3 * 5 * 7 * 11 * ... (product of small primes, padded to 256 bytes)
// For simplicity, use n = p1 * p2 * ... where all factors < 2^20.

uint8_t smooth_modulus[256] = { /* 256-byte smooth odd composite */ };

cmem_t ek = {nullptr, 0};
cbmpc_error_t rv = cbmpc_pve_base_pke_rsa_ek_from_modulus(
    cmem_t{smooth_modulus, 256}, &ek);
// rv == CBMPC_SUCCESS  <-- no rejection

uint8_t secret_scalar[32] = { /* private key share */ };
cmem_t ct = {nullptr, 0};
rv = cbmpc_pve_encrypt(
    NULL, CBMPC_CURVE_SECP256K1,
    ek, cmem_t{(uint8_t*)"label", 5},
    cmem_t{secret_scalar, 32}, &ct);
// rv == CBMPC_SUCCESS  <-- ciphertext produced under weak key

// Attacker factors smooth_modulus, recovers kem_ss from RSA-OAEP ct,
// derives AES-GCM key, decrypts payload -> recovers secret_scalar.
```

The expected behavior is that `cbmpc_pve_base_pke_rsa_ek_from_modulus` returns `E_BADARG` for any modulus that is not a valid RSA-2048 semiprime. Currently it does not.

error_t base_pke_rsa_ek_from_modulus(mem_t modulus, buf_t& out_ek) {
  if (const error_t rv = coinbase::api::detail::validate_mem_arg_max_size(modulus, "modulus",
                                                                          coinbase::api::detail::MAX_OPAQUE_BLOB_SIZE))
    return rv;

  constexpr int kExpectedModulusBytes = coinbase::crypto::RSA_KEY_LENGTH / 8;
  if (modulus.size != kExpectedModulusBytes)
    return coinbase::error(E_BADARG, "modulus must be exactly 256 bytes (RSA-2048)");

  constexpr unsigned long kDefaultExponent = 65537;

  BIGNUM* n_bn = BN_bin2bn(modulus.data, modulus.size, nullptr);
  if (!n_bn) return coinbase::error(E_GENERAL, "BN_bin2bn(modulus) failed");

  if (BN_is_zero(n_bn)) {
    BN_free(n_bn);
    return coinbase::error(E_BADARG, "modulus must not be zero");
  }

  BIGNUM* e_bn = BN_new();
  if (!e_bn) {
    BN_free(n_bn);
    return coinbase::error(E_GENERAL, "BN_new(e) failed");
  }
  BN_set_word(e_bn, kDefaultExponent);

  coinbase::crypto::rsa_pub_key_t pk;
  pk.set(n_bn, e_bn);
  BN_free(n_bn);
  BN_free(e_bn);

  base_pke_ek_blob_v1_t ek_blob;
  ek_blob.key_type = static_cast<uint32_t>(base_pke_key_type_v1::rsa_oaep_2048);
  ek_blob.rsa_ek = std::move(pk);

  out_ek = coinbase::convert(ek_blob);
  return SUCCESS;
}

error_t rsa_pub_key_t::encrypt_raw(mem_t in, buf_t& out) const {
  int n_size = size();
  if (n_size != in.size) return coinbase::error(E_CRYPTO);

  scoped_ptr_t<EVP_PKEY_CTX> ctx = EVP_PKEY_CTX_new(ptr, NULL);
  if (EVP_PKEY_encrypt_init(ctx) <= 0) return openssl_error("RSA encrypt RAW error");
  if (EVP_PKEY_CTX_set_rsa_padding(ctx, RSA_NO_PADDING) <= 0) return openssl_error("RSA encrypt RAW error");
  size_t outlen = n_size;
  if (EVP_PKEY_encrypt(ctx, out.alloc(n_size), &outlen, in.data, in.size) <= 0)
    return openssl_error("RSA encrypt RAW error");
  return SUCCESS;
}

void rsa_pub_key_t::set(RSA_BASE*& rsa, const BIGNUM* n, const BIGNUM* e) {
  cb_assert(n && e);
  OSSL_PARAM_BLD* param_bld = OSSL_PARAM_BLD_new();
  OSSL_PARAM_BLD_push_BN(param_bld, "n", n);
  OSSL_PARAM_BLD_push_BN(param_bld, "e", e);
  OSSL_PARAM* params = OSSL_PARAM_BLD_to_param(param_bld);

  scoped_ptr_t<EVP_PKEY_CTX> ctx = EVP_PKEY_CTX_new_from_name(NULL, "RSA", NULL);
  cb_assert(EVP_PKEY_fromdata_init(ctx) > 0);
  cb_assert(EVP_PKEY_fromdata(ctx, &rsa, EVP_PKEY_PUBLIC_KEY, params) > 0);

  OSSL_PARAM_free(params);
  OSSL_PARAM_BLD_free(param_bld);
}

  static error_t encapsulate(const ek_t& pub_key, buf_t& kem_ct, buf_t& kem_ss, drbg_aes_ctr_t* drbg) {
    const int sha256_size_bytes = hash_alg_t::get(hash_e::sha256).size;
    kem_ss = drbg ? drbg->gen(sha256_size_bytes) : gen_random(sha256_size_bytes);
