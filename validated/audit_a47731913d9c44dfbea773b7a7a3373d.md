## Title
ETH Deposit Limit Bypass at Exact Boundary via Strict Inequality - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses a strict `>` comparison for the ETH branch, meaning when `totalAssetDeposits` equals exactly `depositLimitByAsset(ETH_TOKEN)`, the function returns `false` and the deposit is not reverted. This allows any unprivileged depositor to push ETH deposits past the configured cap at the exact boundary, while `getAssetCurrentLimit` simultaneously reports zero remaining capacity.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch evaluates:

```solidity
return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));  // L679
```

When `totalAssetDeposits == depositLimit`, this expression is `false`, so `_beforeDeposit` (L661–663) does not revert with `MaximumDepositLimitReached`. The deposit proceeds and `_mintRsETH` mints rsETH for the caller.

The ERC-20 branch correctly includes `amount` in the comparison (`totalAssetDeposits + amount > limit`, L681), so it blocks deposits when the limit is exactly reached. The ETH branch omits `amount` entirely and only guards against an already-exceeded total.

`getAssetCurrentLimit` (L402–409) uses the same `>` threshold but returns `lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits`, which yields `0` when `totalAssetDeposits == depositLimit`. This creates a directly observable inconsistency: the public view signals zero remaining capacity while the enforcement gate still admits ETH deposits.

No other guard in the deposit flow compensates for this.

## Impact Explanation
An unprivileged depositor can mint rsETH against ETH that exceeds the protocol's configured deposit ceiling. The deposit limit is the primary mechanism bounding ETH intake; bypassing it at the exact boundary violates the protocol's stated deposit constraints. No funds are directly stolen, but the contract fails to enforce its promised deposit cap. This maps to **Low — contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The boundary condition `totalAssetDeposits == depositLimit` is reachable in two realistic ways: (a) an admin sets `depositLimitByAsset(ETH_TOKEN)` to exactly the current TVL to freeze new deposits — immediately exploitable by any depositor — or (b) a sequence of deposits naturally lands the total at the limit. In scenario (a) the bypass is directly and repeatably exploitable by any external caller with no special privileges. Likelihood is **Low** overall but non-negligible given scenario (a) is a plausible admin action.

## Recommendation
Change the strict inequality to a non-strict one in the ETH branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits >= lrtConfig.depositLimitByAsset(asset));
}
```

This aligns the enforcement gate with `getAssetCurrentLimit`, which already treats `totalAssetDeposits == depositLimit` as zero remaining capacity.

## Proof of Concept
1. Admin calls `setDepositLimitByAsset(ETH_TOKEN, X)` where `X` equals the current `getTotalAssetDeposits(ETH_TOKEN)`.
2. `getAssetCurrentLimit(ETH_TOKEN)` returns `0` — cap is reported as full.
3. Any depositor calls `depositETH{value: 1 ether}(0, "")`.
4. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether)`.
5. Evaluates `X > X` → `false` → no revert.
6. `_mintRsETH` mints rsETH; total ETH deposits become `X + 1 ether`, exceeding the configured limit.
