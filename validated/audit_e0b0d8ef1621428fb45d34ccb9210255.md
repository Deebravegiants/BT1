### Title
Unrestricted `updateRSETHPrice()` Allows Any Caller to Exhaust the Daily Fee Minting Limit, Permanently Suppressing Protocol Fee Collection - (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle.updateRSETHPrice()` is a public, permissionless function. Every call to it consumes the shared daily fee minting budget (`maxFeeMintAmountPerDay`). An unprivileged attacker who holds rsETH can call this function at a strategically chosen moment — when only a fraction of the day's yield has accrued — to partially exhaust the daily budget. All subsequent legitimate calls by the protocol operator that would push the cumulative minted amount above the cap revert with `DailyFeeMintLimitExceeded`, causing the protocol treasury to forfeit the remaining day's fees. The attack is repeatable every day, permanently suppressing protocol fee revenue while proportionally enriching rsETH holders.

---

### Finding Description

`updateRSETHPrice()` carries no access-control modifier:

```solidity
// contracts/LRTOracle.sol line 87
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Inside `_updateRsETHPrice()`, every execution path calls `_checkAndUpdateDailyFeeMintLimit`:

```solidity
// line 303-310
_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
// ...
} else {
    _checkAndUpdateDailyFeeMintLimit(0);
}
```

`_checkAndUpdateDailyFeeMintLimit` enforces a hard per-day cap:

```solidity
// lines 197-210
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }
    if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(...);
    }
    currentPeriodMintedFeeAmount += feeAmount;
}
```

Because `updateRSETHPrice()` is public, any address can trigger a fee-minting event at any time. The attacker does not need to supply any funds or privileged credentials.

**Mechanism of harm:**

The fee is computed as a fraction of the TVL increase since the last price update:

```solidity
// lines 244-246
if (!protocolPaused && totalETHInProtocol > previousTVL) {
    uint256 rewardAmount = totalETHInProtocol - previousTVL;
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}
```

After the call, `rsETHPrice` is updated to `newRsETHPrice`, resetting `previousTVL` for the next call. This means:

- **Call 1 (attacker, early in the day):** Only a fraction of the day's yield has accrued. Fee minted = X rsETH. `currentPeriodMintedFeeAmount = X`.
- **Call 2 (operator, later in the day):** More yield has accrued. Fee that would be minted = Y rsETH. If `X + Y > maxFeeMintAmountPerDay`, the call **reverts**. The Y rsETH in fees are never collected for that day.

The Y rsETH that the treasury forfeits is not burned — it remains as unrealised yield that inflates the rsETH price, directly benefiting rsETH holders including the attacker.

The attack is **repeatable every day** at zero cost to the attacker (only gas).

---

### Impact Explanation

**High — Theft of unclaimed yield (protocol fees).**

The protocol treasury permanently loses a portion of its daily fee revenue. An attacker who holds rsETH profits indirectly: every rsETH in fees that is not minted to the treasury instead accrues to existing rsETH holders via a higher `rsETHPrice`. Over time, sustained daily execution of this attack can eliminate a significant fraction of the protocol's fee income.

---

### Likelihood Explanation

**Medium.**

- The attacker requires no privileged access, no capital at risk beyond gas, and no off-chain infrastructure beyond a simple bot that monitors `totalETHInProtocol` vs. `previousTVL`.
- The attacker must hold rsETH to benefit, which is a low barrier for a motivated actor.
- The attack is most effective when yield accrues unevenly throughout the day (e.g., large EigenLayer reward distributions), which is a realistic on-chain condition.
- The only constraint is timing: the attacker must call before the operator does and before the full day's yield has accrued.

---

### Recommendation

Restrict `updateRSETHPrice()` to authorised callers (e.g., `onlyLRTOperator` or `onlyLRTManager`), consistent with the already-existing privileged variant:

```solidity
// contracts/LRTOracle.sol line 94
function updateRSETHPriceAsManager() external onlyLRTManager {
    _updateRsETHPrice();
}
```

The public `updateRSETHPrice()` should either be removed or gated behind the same role check. If permissionless price updates are desired for liveness, decouple the fee-minting step from the price-update step so that only authorised callers can trigger fee collection.

---

### Proof of Concept

**Setup:**
- `maxFeeMintAmountPerDay = 100 rsETH`
- `protocolFeeInBPS = 1000` (10%)
- Day starts; `currentPeriodMintedFeeAmount = 0`

**Step 1 — Early in the day, 500 rsETH worth of yield has accrued:**
- Attacker calls `updateRSETHPrice()`.
- `rewardAmount` = 500 rsETH-equivalent ETH → `protocolFeeInETH` = 50 ETH-equivalent.
- `rsethAmountToMintAsProtocolFee` ≈ 50 rsETH (< 100 limit) → minted to treasury.
- `currentPeriodMintedFeeAmount = 50`. `rsETHPrice` updated.

**Step 2 — Later in the day, another 600 rsETH worth of yield accrues:**
- Protocol operator calls `updateRSETHPrice()`.
- `rsethAmountToMintAsProtocolFee` ≈ 60 rsETH.
- `_checkAndUpdateDailyFeeMintLimit(60)`: `50 + 60 = 110 > 100` → **reverts** with `DailyFeeMintLimitExceeded`.
- The 60 rsETH in fees are **never collected**; they inflate `rsETHPrice` instead, benefiting the attacker's rsETH holdings.

**Net result:** Treasury collects 50 rsETH instead of the optimal 100 rsETH. The attacker repeats this every day. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
