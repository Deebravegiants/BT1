### Title
Single Attested Participant Can Deliver Arbitrary CKD Response to Any User's `AppPublicKey` Request — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_ckd` skips all cryptographic output verification when the request uses the `AppPublicKey` (non-PV) variant. Any single attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary `CKDResponse` for any pending CKD request, delivering garbage key material to the user without threshold authorization.

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` handles two variants of `CKDAppPublicKey`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, ensuring the response is cryptographically bound to the MPC network's root key and the user's app ID. [1](#0-0) 

For `AppPublicKey`, the arm is a no-op. No pairing check, no binding to the root key, no binding to the user's `app_id`. The response bytes are immediately passed to `resolve_yields_for`, which resumes all queued yield promises with whatever bytes the caller supplied. [2](#0-1) 

`resolve_yields_for` drains the entire fan-out queue for the request key in one call, delivering the attacker-controlled bytes to every waiting caller. [3](#0-2) 

The only gate is `assert_caller_is_attested_participant_and_protocol_active()`, which requires the caller to be a current attested participant — a single node, well below the signing threshold. [4](#0-3) 

The `CKDRequest` key is fully deterministic and observable on-chain: it is derived from the user's `predecessor_account_id`, `derivation_path`, `app_public_key`, and `domain_id`. [5](#0-4) 

### Impact Explanation

A user calling `request_app_private_key` with `AppPublicKey` expects to receive an encrypted derived private key `(big_y, big_c)` that was computed by the full MPC threshold. Instead, a single Byzantine participant can substitute any values for `big_y` and `big_c`. The user receives garbage key material. If the user uses the derived key to receive assets on a foreign chain (the primary use-case of CKD), those assets are permanently unrecoverable because the correct private key was never delivered. This is **unauthorized confidential key derivation output without the required participant authorization** — a Critical impact under the allowed scope.

### Likelihood Explanation

Any single attested participant in the network can execute this attack. The attacker only needs to:
1. Observe a pending `AppPublicKey` CKD request on-chain (public state).
2. Race the honest `respond_ckd` call with their own, supplying arbitrary `big_y`/`big_c` values.

Because `resolve_yields_for` is first-writer-wins (it removes the map entry on the first successful call), the attacker only needs to land their transaction before the honest nodes respond. This is a realistic race on NEAR's ~1-second block time. The attacker does not need any leaked keys, threshold collusion, or physical TEE access.

### Recommendation

Apply `ckd_output_check` unconditionally for all `CKDAppPublicKey` variants, not only `AppPublicKeyPV`. For the `AppPublicKey` (legacy) variant, a corresponding pairing-based output check should be defined and enforced before `resolve_yields_for` is called. Until such a check exists, the `AppPublicKey` variant should be deprecated or its `respond_ckd` path should require threshold-level authorization (e.g., requiring `t` independent `respond_ckd` calls with matching responses before resolving the yield).

### Proof of Concept

1. Alice calls `request_app_private_key` with `AppPublicKey(pk)`, `derivation_path = "wallet/0"`, `domain_id = 4`. The contract stores a pending yield under the key `CKDRequest { app_public_key: AppPublicKey(pk), app_id: derive_app_id(alice, "wallet/0"), domain_id: 4 }`. [6](#0-5) 

2. Mallory is one attested participant (e.g., participant index 0 in a 3-of-5 network). She observes Alice's pending request on-chain.

3. Mallory calls `respond_ckd(ckd_request, CKDResponse { big_y: [0u8;48], big_c: [0u8;48] })`. The contract executes:
   - `assert_caller_is_signer()` → passes (Mallory calls directly).
   - `assert_caller_is_attested_participant_and_protocol_active()` → passes (Mallory is attested).
   - `match &request.app_public_key { AppPublicKey(_) => {} }` → **no check performed**. [7](#0-6) 
   - `resolve_yields_for(...)` → resumes Alice's yield with the garbage bytes. [8](#0-7) 

4. Alice's `request_app_private_key` promise resolves with `CKDResponse { big_y: [0;48], big_c: [0;48] }`. Alice derives a key from this garbage material and uses it to receive 10 ETH on Ethereum. The ETH is permanently lost because the correct derived private key was never delivered.

5. The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` confirms that `CKDResponse { big_y: [1u8;48], big_c: [2u8;48] }` (arbitrary bytes) is accepted without error for the `AppPublicKey` variant. [9](#0-8)

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
