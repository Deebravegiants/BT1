### Title
`validateBatchWithdrawals` Uses Pre-Batch Account Balance, Incorrectly Rejecting Valid Dijkstra-Era Batch Transactions That Fund Withdrawals via Sub-Transaction Direct Deposits - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, `validateBatchWithdrawals` checks that total withdrawals across the entire batch (top-level tx + all sub-transactions) do not exceed the **original** (pre-batch) account balance. However, sub-transactions can include `directDeposits` that increase account balances before later sub-transactions execute. The actual execution model (via `SUBLEDGERS` → `SUBENTITIES`) applies direct deposits from sub-tx N before validating withdrawals in sub-tx N+1, making cross-sub-tx deposit-then-withdraw patterns valid at execution time. The pre-state balance check in `validateBatchWithdrawals` is overly strict and rejects these valid batch transactions.

---

### Finding Description

**Root cause** is in `validateBatchWithdrawals`: [1](#0-0) 

The function receives `accounts` which is explicitly the original (pre-batch) account state: [2](#0-1) 

This `certState` is passed from `dijkstraLedgerTransition` as `lsCertState ledgerState` — the original ledger state before any sub-ledger processing: [3](#0-2) 

The `SUBLEDGERS` rule is processed **before** `UTXOW`, and it applies direct deposits from each sub-transaction via `SUBENTITIES`: [4](#0-3) 

Within `dijkstraSubEntitiesTransition`, direct deposits are applied to accounts after each sub-tx's withdrawals, making the updated balance available to the next sub-tx: [5](#0-4) 

Both top-level and sub-transaction bodies support the `directDeposits` field (CDDL field 25): [6](#0-5) 

The `applyDirectDeposits` function adds coin to account balances: [7](#0-6) 

**The mismatch**: `SUBENTITIES` validates each sub-tx's withdrawals against the post-previous-sub-tx accounts (which include prior direct deposits), but `validateBatchWithdrawals` checks the sum of all withdrawals against the original balance, ignoring any direct deposits that occurred during batch execution.

---

### Impact Explanation

**Medium impact**: Valid Dijkstra-era batch transactions are incorrectly rejected by the ledger. Specifically, any batch where a sub-transaction's direct deposit funds a withdrawal in a later sub-transaction will fail `validateBatchWithdrawals` even though the actual execution (via `SUBENTITIES`) would succeed. This modifies the effective withdrawal behavior outside design parameters — the ledger rejects a state transition that is valid under the execution model it implements.

Concretely:
- Account A has original balance = 100 ADA
- Sub-tx 1: direct deposit of 50 ADA to account A → execution balance becomes 150 ADA
- Sub-tx 2: withdrawal of 120 ADA from account A → valid at execution (120 ≤ 150), but `validateBatchWithdrawals` computes 120 > 100 (original balance) and **rejects the batch**

The `SUBENTITIES` rule already prevents double-spend across sub-txs (it validates each sub-tx's withdrawal against the current post-previous-sub-tx balance), making `validateBatchWithdrawals` both redundant for the no-direct-deposit case and incorrectly strict for the direct-deposit case.

---

### Likelihood Explanation

Any Dijkstra-era user constructing a batch transaction with the direct-deposit-then-withdraw pattern across sub-transactions will trigger this. No special privileges are required — the entry path is an unprivileged transaction sender submitting a well-formed Dijkstra-era batch transaction. The Dijkstra era explicitly supports `directDeposits` in sub-transaction bodies (CDDL field 25), so this pattern is part of the intended design surface.

---

### Recommendation

`validateBatchWithdrawals` should compute the effective available balance for each account as:

```
effectiveBalance(account) = originalBalance(account) 
    + sum of directDeposits to account across all sub-txs
```

and check that total withdrawals do not exceed this effective balance. Alternatively, since `SUBENTITIES` already validates each sub-tx's withdrawals against the current (post-previous-sub-tx) balance, `validateBatchWithdrawals` could be removed entirely as it is redundant for the no-direct-deposit case and incorrect for the direct-deposit case.

---

### Proof of Concept

1. **Sub-tx 1** (`DijkstraSubTxBodyRaw`): includes `dstbrDirectDeposits = DirectDeposits { accountA → 50 ADA }` and spends a UTxO containing ≥ 50 ADA to fund the deposit.
2. **Sub-tx 2** (`DijkstraSubTxBodyRaw`): includes `dstbrWithdrawals = Withdrawals { accountA → 120 ADA }`.

Pre-condition: Account A has original balance = 100 ADA.

**Expected behavior** (per `SUBENTITIES` execution model):
- After sub-tx 1: account A balance = 150 ADA
- Sub-tx 2 withdrawal check in `SUBENTITIES`: 120 ≤ 150 → passes
- Batch succeeds

**Actual behavior** (due to `validateBatchWithdrawals`):
- `allWithdrawals = { accountA → 120 ADA }` (from sub-tx 2)
- `getAccountBalance accountA = 100 ADA` (original balance)
- `120 > 100` → `WithdrawalsExceedAccountBalance` failure
- Batch is **rejected**

```haskell
-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL
```
