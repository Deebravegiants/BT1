### Title
Unguarded NEAR Bridge `new()` Initializer Allows Front-Run Takeover of All Admin Roles — (File: near/omni-bridge/src/lib.rs)

### Summary
The `new()` constructor in `near/omni-bridge/src/lib.rs` is a public, caller-unrestricted function that unconditionally grants `acl_init_super_admin` and `Role::DAO` to `env::predecessor_account_id()`. Because NEAR's `#[init]` macro only prevents double-initialization — it does not restrict *who* may call the function — any unprivileged account that submits a call to `new()` between the WASM deployment transaction and the legitimate init transaction will seize permanent, irrevocable administrative control of the bridge. The same pattern exists in `near/token-deployer/src/lib.rs`.

---

### Finding Description

`near/omni-bridge/src/lib.rs` declares:

```rust
#[init]
pub fn new(mpc_signer: AccountId, wnear_account_id: AccountId) -> Self {
    let mut contract = Self { … };
    contract.acl_init_super_admin(near_sdk::env::predecessor_account_id());
    contract.acl_grant_role(Role::DAO.into(), near_sdk::env::predecessor_account_id());
    contract
}
``` [1](#0-0) 

There is no `require!(env::predecessor_account_id() == <expected_deployer>)` guard. The `#[init]` attribute only ensures the function runs exactly once (panics if state already exists); it places no restriction on the caller identity.

The identical structural problem exists in `near/token-deployer/src/lib.rs`:

```rust
#[init]
pub fn new(controller: AccountId, dao: AccountId, global_code_hash: Base58CryptoHash) -> Self {
    let mut contract = Self { global_code_hash: global_code_hash.into() };
    contract.acl_init_super_admin(near_sdk::env::predecessor_account_id());
    contract.acl_grant_role(Role::DAO.into(), dao.clone());
    contract.acl_grant_role(Role::Controller.into(), controller);
    contract.acl_transfer_super_admin(dao);
    contract
}
``` [2](#0-1) 

On NEAR, deploying a contract WASM and calling its initializer are two separate transactions. The project's own deployment tooling demonstrates this two-step pattern explicitly — the script deploys with `without-init-call`, sleeps, then calls the init function as a separate transaction:

```sh
near contract deploy "$CONTRACT_ID" use-file "$WASM_PATH" \
    without-init-call …
sleep 3
near contract call-function as-transaction "$CONTRACT_ID" \
    "$INIT_FUNCTION" …
``` [3](#0-2) 

Between the deploy transaction and the init transaction, the contract account exists with empty state and a live, callable `new()` function. Any NEAR account can submit a transaction to `new()` during this window.

By contrast, the EVM contracts are correctly protected: `OmniBridge`, `BridgeToken`, and `ENearProxy` all call `_disableInitializers()` in their constructors, and proxy deployment is atomic via `new ERC1967Proxy(impl, abi.encodeWithSelector(initialize.selector, …))` or `upgrades.deployProxy`. [4](#0-3) [5](#0-4) 

The Solana `Initialize` instruction requires `#[account(address = crate::ID)] pub program: Signer<'info>`, which enforces that the program's own keypair signs, preventing any unprivileged caller. [6](#0-5) 

---

### Impact Explanation

An attacker who wins the front-run on `near/omni-bridge::new()` becomes `acl_super_admin` and `Role::DAO`. With those roles they can:

- Register arbitrary factory addresses (`factories` map) so that forged cross-chain proofs are accepted as coming from a legitimate chain factory.
- Register a malicious prover, bypassing all proof verification in `fin_transfer`.
- Mint unbacked wrapped tokens to any recipient via `fin_transfer_callback` once a fake factory and prover are registered.
- Permanently freeze the bridge by pausing all operations, making all locked user funds unclaimable.

The legitimate deployer's subsequent `new()` call panics (`E_ALREADY_INITIALIZED` equivalent), forcing a full contract re-deployment — but by then the attacker's roles are already written to state and the original contract account is compromised.

An attacker who wins the front-run on `near/token-deployer::new()` controls `Role::Controller` and `Role::DAO`, allowing them to set `global_code_hash` to a malicious token implementation and deploy backdoored `OmniToken` contracts for any future NEAR-side token deployment. [7](#0-6) 

This satisfies:
- **Critical** — unauthorized creation of wrapped bridge assets through verification failure (forged factory + prover).
- **Critical** — irreversible fund lock / frozen redemption path.

---

### Likelihood Explanation

- NEAR has no traditional mempool, but transactions are publicly visible once submitted to the network. An attacker monitoring the chain for WASM deployments to known bridge account IDs can react within seconds.
- The project's own tooling inserts an explicit `sleep 3` between deploy and init, making the window at least three seconds wide and trivially exploitable by any automated bot.
- The attacker needs no privileged access, no leaked keys, and no colluding validators — only the ability to submit a standard NEAR transaction.

Likelihood: **High**.

---

### Recommendation

1. **Atomic deployment**: Use NEAR's `with-init-call` flag so the WASM upload and `new()` call are included in the same transaction, eliminating the window entirely.
2. **Caller guard in `new()`**: Add an explicit check inside the initializer:
   ```rust
   require!(
       env::predecessor_account_id() == env::current_account_id(),
       "Only the contract account may initialize"
   );
   ```
   or restrict to a known deployer account passed as a compile-time constant.
3. Apply the same fix to `near/token-deployer/src/lib.rs`.

---

### Proof of Concept

1. Deployer submits: `near contract deploy omni-bridge.near use-file omni_bridge.wasm without-init-call …`
2. Attacker observes the confirmed deploy transaction on-chain.
3. Attacker submits: `near contract call-function as-transaction omni-bridge.near new json-args '{"mpc_signer":"attacker.near","wnear_account_id":"wrap.near"}' sign-as attacker.near …`
4. Attacker's transaction is processed; `acl_init_super_admin(attacker.near)` and `acl_grant_role(DAO, attacker.near)` are written to state.
5. Deployer's legitimate `new()` call panics: contract state already exists.
6. Attacker calls `set_factory` and registers a malicious EVM factory address, then calls `fin_transfer` with a crafted proof accepted by their registered fake prover, minting unbacked tokens to any recipient. [1](#0-0) [2](#0-1) [3](#0-2)
