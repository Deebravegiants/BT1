### Title
Uninitialized `minAmountToDeposit` Disables Minimum Deposit Guard - (File: `contracts/LRTDepositPool.sol`)

### Summary
`LRTDepositPool` declares `minAmountToDeposit` but never initializes it in `initialize()`, leaving it at the Solidity default of `0`. The deposit guard that relies on this variable is therefore silently bypassed from the moment of deployment until an admin explicitly calls `setMinAmountToDeposit()`.

### Finding Description
`LRTDepositPool` declares the state variable on line 30:

```solidity
uint256 public minAmountToDeposit;
```

The `initialize()` function (lines 45–52) sets only `maxNodeDelegatorLimit = 10` and never assigns `minAmountToDeposit`, so it remains `0`. [1](#0-0) 

Every deposit path — `depositETH` and `depositAsset` — calls `_beforeDeposit`, which contains the guard:

```solidity
if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
    revert InvalidAmountToDeposit();
}
``` [2](#0-1) 

When `minAmountToDeposit == 0`, the sub-expression `depositAmount < minAmountToDeposit` is `depositAmount < 0`, which is always `false` for a `uint256`. The guard collapses to a pure zero-amount check and the intended minimum-deposit protection is completely inoperative. Any deposit of `1 wei` passes unchallenged.

The setter `setMinAmountToDeposit()` exists but is callable only by the LRT admin after deployment: [3](#0-2) 

Until that call is made there is an open window — identical in structure to the USDV `blockDelay = 0` window — during which the protection is absent.

### Impact Explanation
With `minAmountToDeposit == 0` any caller can submit arbitrarily small deposits (e.g., 1 wei). At extreme dust amounts the rsETH minted rounds to zero, meaning the depositor receives nothing while the protocol books a negligible asset credit. Repeated dust deposits bloat on-chain state and can cause share-accounting rounding drift over time. The contract does not deliver the promised minimum-deposit guarantee. No direct fund theft or permanent freeze results, placing this in the **Low** impact tier: *contract fails to deliver promised returns, but doesn't lose value*.

### Likelihood Explanation
The condition is present from the first block after deployment and persists until an admin transaction explicitly sets `minAmountToDeposit`. Any unprivileged depositor can exploit it immediately with no preconditions. Likelihood is **High** during the uninitialized window.

### Recommendation
Initialize `minAmountToDeposit` to a sensible non-zero value inside `initialize()`, mirroring how `maxNodeDelegatorLimit` is set:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    ...
    maxNodeDelegatorLimit = 10;
    minAmountToDeposit = 0.001 ether; // or another protocol-appropriate floor
    ...
}
```

This eliminates the deployment-to-configuration gap entirely.

### Proof of Concept
1. Deploy `LRTDepositPool` and call `initialize(lrtConfigAddr)`.
2. Observe `minAmountToDeposit == 0` (no admin call made yet).
3. Call `depositETH(0, "ref")` with `msg.value = 1` (1 wei).
4. Inside `_beforeDeposit`: `depositAmount == 0` → false; `depositAmount < minAmountToDeposit` → `1 < 0` → false. Guard does not revert.
5. `getRsETHAmountToMint` returns 0 (rounds down at 1 wei scale); `minRSETHAmountExpected = 0` passes the slippage check.
6. `_mintRsETH(0)` is called — the depositor loses 1 wei with no rsETH received, and the protocol silently accepts the dust deposit. [4](#0-3) [5](#0-4)
