Audit Report

## Title
Unchecked `binTotalShares` Overflow in `addLiquidity` Permanently Locks LP Funds — (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
The entire `addLiquidity` body in `LiquidityLib` executes inside an `unchecked` block, leaving `binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd` at line 120 with no overflow guard. When a bin's token balances are both zero (a routine post-swap state), the proportional token calculation resolves to zero, so the attacker pays nothing while supplying a crafted `sharesToAdd` that wraps `binTotalShares` to zero. Every legitimate LP in that bin is then permanently unable to call `removeLiquidity` due to division-by-zero.

## Finding Description

**Root cause — unchecked arithmetic with no input bound:**

The `addLiquidity` function body opens with `unchecked {` at line 51 and does not close until line 158. [1](#0-0) 

Inside that scope, `binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd` at line 120 has no cap on `sharesToAdd`. If the sum exceeds `type(uint256).max`, it silently wraps. [2](#0-1) 

**Why `_checkedMul` does not protect this path:**

`_checkedMul` at lines 261–263 is a separate function whose body is not inside `unchecked`, so in Solidity ≥0.8 its multiplication IS overflow-checked. However, when a bin's token balances are both zero, the call becomes `_checkedMul(0, sharesToAdd) = 0` — no overflow, no revert. [3](#0-2) 

The `else` branch at lines 108–111 is taken when `binTotalSharesVal > 0`. With `token0BalanceScaled = 0` and `token1BalanceScaled = 0`, both `amount0Scaled` and `amount1Scaled` resolve to zero, so the callback demands zero tokens from the attacker. [4](#0-3) 

**When can a bin have zero token balances with non-zero total shares?**

Swaps modify only `binState.token0BalanceScaled` / `token1BalanceScaled` inside `_binStates`. The `_binTotalShares` mapping is never touched during swaps. After enough swap volume fully traverses a bin, both token balances reach zero while `binTotalShares` retains the original LP share count. [5](#0-4) 

**Corrupted state breaks `removeLiquidity`:**

`removeLiquidity` at lines 205–206 divides by `binTotalSharesVal`. Division by zero always panics in Solidity (even inside `unchecked`), so every legitimate LP attempting to withdraw from the targeted bin receives a permanent revert. [6](#0-5) 

## Impact Explanation
Every LP who held shares in the targeted bin before the attack permanently loses the ability to call `removeLiquidity` for those shares. Their tokens remain locked in the pool contract with no recovery path. This constitutes broken core pool functionality causing loss of funds and unusable withdraw/liquidity flows, satisfying the allowed impact gate.

## Likelihood Explanation
- **Unprivileged trigger**: `addLiquidity` is `external` with no access control beyond `nonReentrant`. [7](#0-6) 
- **Precondition**: a bin with both token balances at zero and `binTotalShares > 0` — a routine post-swap state for any fully-traversed bin.
- **No cost**: the attacker pays zero tokens because `amount0Scaled = amount1Scaled = 0`.
- **Single transaction**: the entire attack fits in one `addLiquidity` call.

## Recommendation
1. Move `binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd` outside the `unchecked` block, or use an explicit overflow check: `require(binTotalSharesVal <= type(uint256).max - sharesToAdd, "shares overflow")`.
2. Add a guard for the zero-balance / non-zero-shares edge case: if `binTotalSharesVal > 0` but both token balances are zero, require `sharesToAdd` to be bounded by a safe maximum or reject the deposit entirely.
3. Consider enforcing a protocol-wide maximum shares-per-position (e.g., `type(uint128).max`) to limit the magnitude of any single `sharesToAdd`.

```
Attack (single tx)
──────────────────
sharesToAdd = type(uint256).max - S + 1

addLiquidity(owner=attacker, salt=0, deltas={binIdxs:[B], shares:[sharesToAdd]}, ...)

Inside addLiquidity (unchecked):
  binTotalSharesVal = S
  else-branch: _checkedMul(0, sharesToAdd) = 0  → amount0Scaled = 0, amount1Scaled = 0

  binTotalShares[B] = S + (2^256 - S + 1) = 2^256 ≡ 0   ← OVERFLOW
  positionBinShares[attackerKey] = sharesToAdd             ← huge value stored

Post-attack
───────────
• binTotalShares[B] = 0
• Legitimate LP calls removeLiquidity(shares=S):
    binTotalSharesVal = 0
    _checkedMul(token0BalanceScaled, S) / 0  → DIVISION BY ZERO → permanent revert
• LP funds permanently locked.
```
