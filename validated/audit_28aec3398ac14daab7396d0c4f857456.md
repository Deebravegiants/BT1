Audit Report

## Title
Stale `rsETHPrice` in `instantWithdrawal()` Enables Over-Extraction of Assets Before Oracle Update — (File: contracts/LRTWithdrawalManager.sol)

## Summary

`instantWithdrawal()` computes the asset payout using the stored `rsETHPrice` state variable from `LRTOracle`, which is only updated when `updateRSETHPrice()` is explicitly called. When actual TVL decreases (e.g., EigenLayer slashing) before the oracle is updated, the stored price is stale and inflated, allowing any rsETH holder to burn rsETH and receive more assets than their proportional entitlement. The shortfall is socialized among remaining rsETH holders when the oracle is eventually updated.

## Finding Description

`LRTOracle.updateRSETHPrice()` is a public function with no access control beyond `whenNotPaused`:

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The stored `rsETHPrice` state variable is only updated when this function is explicitly called. `_updateRsETHPrice()` computes the price from live on-chain TVL:

```solidity
// LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

`instantWithdrawal()` calls `getExpectedAssetAmount()` to compute the payout:

```solidity
// LRTWithdrawalManager.sol L228
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
```

`getExpectedAssetAmount()` reads the stored (potentially stale) `rsETHPrice` directly:

```solidity
// LRTWithdrawalManager.sol L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

There is no freshness check, no atomic price refresh, and no `min()` cap. By contrast, the regular queued withdrawal path applies a `min()` guard in `_calculatePayoutAmount()`:

```solidity
// LRTWithdrawalManager.sol L833-834
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

This guard is entirely absent from `instantWithdrawal()`. The only vault-side check is:

```solidity
// LRTWithdrawalManager.sol L231
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
```

This checks vault liquidity, not price correctness. If the vault holds sufficient assets (which it does — the slashing reduces EigenLayer-side shares, not the unstaking vault's token balance), this check passes even when the price is stale.

**Downside protection does not prevent the attack.** `_updateRsETHPrice()` contains a pause mechanism triggered when the price drop exceeds `pricePercentageLimit` (LRTOracle.sol L270-282). However, this pause only activates when `updateRSETHPrice()` is called. The attacker acts in the window before that call, so the pause is irrelevant to the extraction. For drops below `pricePercentageLimit`, no pause is triggered at all, and the stale price window is fully exploitable with no automatic protection.

**Exploit flow:**
1. EigenLayer slashing reduces `getEffectivePodShares()` for a NodeDelegator. Actual TVL drops. Stored `rsETHPrice` remains at pre-slash value `P_high`.
2. Attacker observes the on-chain state change. `updateRSETHPrice()` has not been called.
3. Attacker calls `instantWithdrawal(asset, rsETHAmount, "")`. `assetAmountUnlocked = rsETHAmount * P_high / assetPrice` — uses stale price. rsETH is burned; attacker receives `assetAmountUnlocked` from `LRTUnstakingVault`.
4. `updateRSETHPrice()` is called. New price `P_low = (TVL_reduced) / rsethSupply` is stored.
5. Remaining rsETH holders now hold rsETH backed by fewer assets per token. The attacker's excess `rsETHAmount * (P_high - P_low) / assetPrice` is a direct loss to them.

## Impact Explanation

**High — Theft of unclaimed yield.** The rsETH price appreciation represents staking yield accrued by all holders proportionally. By withdrawing at a stale inflated price, the attacker extracts more than their proportional share of this yield. The deficit is borne by remaining rsETH holders, whose tokens are now backed by fewer assets per unit. This is a concrete, quantifiable loss of yield belonging to other users, triggered by a public function callable by any rsETH holder without any privileged access.

## Likelihood Explanation

**Medium.** The attack window exists whenever actual TVL decreases before `updateRSETHPrice()` is called. EigenLayer slashing events are observable on-chain with no special access required. The attacker does not need to front-run any specific transaction — they only need to act in the block window between the TVL decrease and the oracle update. The prerequisite that `instantWithdrawal` is enabled for the target asset is a manager-controlled flag, but is a normal operational state. The attack is repeatable across any such event.

## Recommendation

1. **Atomically refresh the price inside `instantWithdrawal()`** by calling `ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice()` before computing `assetAmountUnlocked`, ensuring the price is always current at execution time.
2. Alternatively, apply the same `min(expectedAmount, currentReturn)` guard used in `_calculatePayoutAmount()` to `instantWithdrawal()`, computing `currentReturn` from live TVL rather than the stored price.
3. As defense-in-depth, require that `updateRSETHPrice()` was called within a recent block window before any instant withdrawal is processed.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// 1. Fork mainnet at a block where rsETHPrice = P_high
// 2. Simulate EigenLayer slashing: reduce getEffectivePodShares() for a NodeDelegator
//    (e.g., via vm.store on the EigenLayer DelegationManager)
// 3. Assert: lrtOracle.rsETHPrice() still returns P_high (not yet updated)
// 4. Attacker calls instantWithdrawal(asset, rsETHAmount, "")
//    Record: assetReceived = actual tokens transferred to attacker
// 5. Call updateRSETHPrice() → P_low is stored
// 6. Compute fair payout: fairAmount = rsETHAmount * P_low / assetPrice
// 7. Assert: assetReceived > fairAmount
//    Difference = rsETHAmount * (P_high - P_low) / assetPrice = stolen yield
// 8. Assert: remaining rsETH holders' effective backing per token has decreased
//    by exactly the stolen amount divided by remaining supply
```

The test concretely demonstrates that `getExpectedAssetAmount()` at L593 returns a value strictly greater than the fair proportional amount when `rsETHPrice` is stale, and that the excess is extracted from the pool backing remaining holders.
