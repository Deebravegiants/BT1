### Title
NFT Royalty Bypass via Integer Division Truncation in `compute_royalty_amount` - (File: `chia/wallet/nft_wallet/nft_wallet.py`)

### Summary

`compute_royalty_amount()` uses sequential integer floor-division to compute NFT royalties. When the offered fungible amount is small relative to `MAX_ROYALTY_BASIS_POINTS` (10 000), the result truncates to zero. The wallet then silently omits the royalty payment from the offer spend bundle, and the on-chain ownership-layer puzzle receives an empty `trade_prices_list`, so no royalty is enforced on-chain. An unprivileged buyer can craft an offer that transfers an NFT while paying the royalty recipient nothing.

### Finding Description

`compute_royalty_amount` in `chia/wallet/nft_wallet/nft_wallet.py` computes royalties with two sequential integer floor-divisions:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

`MAX_ROYALTY_BASIS_POINTS` is 10 000. [2](#0-1) 

When `abs(offered_amount) // royalty_split * percentage < 10 000`, the result is 0. The existing test suite explicitly documents and accepts this:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [3](#0-2) 

When `compute_royalty_amount` returns 0 for every NFT in the offer, `make_nft1_offer` skips the royalty coin spend entirely:

```python
if sum(p.amount for _, p in payments) == 0:
    continue
``` [4](#0-3) 

Simultaneously, the NFT singleton spend is constructed with a filtered `trade_prices_list` that drops any price whose royalty would be zero:

```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
``` [5](#0-4) 

With an empty `trade_prices_list`, the on-chain ownership-layer CLVM puzzle has no price entries to enforce, so the spend is valid on-chain with zero royalty paid. The NFT transfers to the buyer; the royalty recipient receives nothing.

The same truncation exists in the display-only `royalty_calculation` static method:

```python
"amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
``` [6](#0-5) 

This means the taker's wallet UI also shows 0 royalty, so the NFT owner sees no warning that royalties are being bypassed.

### Impact Explanation

An attacker who is the buyer (offer maker) can acquire an NFT while paying the royalty recipient 0 mojos. The royalty address embedded in the NFT singleton — which is the creator's income mechanism — receives nothing. This is an unauthorized accounting change affecting NFT offer settlement and singleton-controlled assets. The royalty recipient's funds are effectively diverted to the buyer.

### Likelihood Explanation

The NFT owner must accept the offer, which is a social-engineering dependency. However:

- For CATs with low royalty percentages (e.g., 1 basis point = 0.01%), the truncation threshold is `offered_amount < 10 000 mojos` = up to 10 CATs (at 1 000 mojos/CAT), which is not dust.
- The taker's wallet displays 0 royalty (same truncation in `royalty_calculation`), so the NFT owner receives no warning.
- A buyer can make many small-amount offers across many NFTs, systematically bypassing royalties at scale.

### Recommendation

Replace the sequential integer floor-division with a single multiplication before division to preserve precision:

```python
# Before (truncates early):
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS

# After (multiply first, divide once):
amount = abs(offered_amount) * percentage // (royalty_split * MAX_ROYALTY_BASIS_POINTS)
```

Apply the same fix to `royalty_calculation` at line 719. Additionally, consider enforcing a minimum offered amount or minimum royalty amount when creating offers for royalty-enabled NFTs.

### Proof of Concept

- NFT with 1 basis point (0.01%) royalty
- Buyer offers 9 999 mojos of a CAT (9.999 CATs at 1 000 mojos/CAT)
- `compute_royalty_amount(-9999, 1, 1)` → `9999 // 1 * 1 // 10000` → `9999 // 10000` → **0**
- `make_nft1_offer` skips the royalty coin spend; `trade_prices_list` is filtered to empty
- NFT transfers on-chain; royalty recipient receives 0 mojos
- Taker's wallet `royalty_calculation` also returns 0, so no UI warning is shown [1](#0-0) [7](#0-6)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
