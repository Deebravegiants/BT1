### Title
Unregistered ERC20 Tokens Permanently Locked in Bridge on `initTransfer()` Without Token Allowlist Check - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.initTransfer()` accepts any arbitrary ERC20 token address without verifying that the token is registered in the bridge's `ethToNearToken` mapping. Tokens sent for unregistered addresses are transferred into the contract and permanently locked, because the NEAR-side settlement callback hard-panics on missing token decimals and there is no admin rescue function in the contract.

---

### Finding Description

`initTransfer()` routes token handling through three branches: custom minter, bridge token (burn), and a catch-all `else` that calls `safeTransferFrom` into the contract for any other ERC20: [1](#0-0) 

The `else` branch at lines 406–411 fires for **any** ERC20 address that is not in `customMinters` and not in `isBridgeToken`. There is no guard checking `ethToNearToken[tokenAddress] != ""` (i.e., that the token has a registered NEAR counterpart). The `InitTransfer` event is emitted unconditionally: [2](#0-1) 

When a relayer submits this event to the NEAR bridge, `fin_transfer_callback` immediately panics on the unregistered token: [3](#0-2) 

The NEAR-side settlement is permanently blocked. The ERC20 tokens remain in the EVM contract. `OmniBridge.sol` contains no `withdrawERC20`, rescue, or sweep function — unlike the Gravity.sol contract in the reference report which at least allowed an admin to recover funds. Here there is no recovery path at all.

The identical pattern exists in the StarkNet bridge: `init_transfer()` also falls through to an unguarded `transfer_from` for any non-bridge token without checking registration: [4](#0-3) 

---

### Impact Explanation

**Critical — Irreversible fund lock.** Any user who calls `initTransfer` (or the StarkNet `init_transfer`) with an ERC20 address that has not been registered through the `logMetadata` → NEAR deploy → decimals-registration flow will have their tokens permanently locked in the bridge contract. There is no admin rescue function, no cancel-transfer mechanism visible in `OmniBridge.sol`, and the NEAR-side callback hard-panics rather than gracefully rejecting, so the transfer can never be completed or unwound.

---

### Likelihood Explanation

**High.** The function is fully public and requires no privileged role. A user who holds any ERC20 token — including a token that was previously supported but later de-registered, a token whose NEAR-side metadata has not yet been processed, or simply a wrong address — can trigger this path. The `else` branch is the intended path for legitimate native ERC20 lock-and-mint flows, so it is exercised in normal operation; the missing allowlist check is the only gate absent.

---

### Recommendation

Add an explicit registration check at the top of the `else` branch (or before the branch split) in `initTransfer()`:

```solidity
require(
    bytes(ethToNearToken[tokenAddress]).length > 0,
    "ERR_TOKEN_NOT_REGISTERED"
);
```

Apply the equivalent guard in the StarkNet `init_transfer()`:

```cairo
assert(self.is_bridge_token(token_address) || /* custom minter check */, 'ERR_TOKEN_NOT_REGISTERED');
```

Additionally, add an admin ERC20 rescue function to `OmniBridge.sol` as a defence-in-depth measure, analogous to the `withdrawERC20` present in the reference Gravity.sol contract, so that any tokens locked before this fix can be recovered.

---

### Proof of Concept

1. Deploy or obtain any ERC20 token `FAKE` that has never been passed through `logMetadata` on the EVM bridge.
2. Approve `OmniBridge` to spend `1000 FAKE`.
3. Call `OmniBridge.initTransfer(FAKE_address, 1000, 0, 0, "near_recipient.near", "")`.
4. The `else` branch executes: `IERC20(FAKE_address).safeTransferFrom(msg.sender, address(this), 1000)` succeeds; `InitTransfer` event is emitted.
5. A relayer submits the proof to the NEAR bridge. `fin_transfer_callback` calls `self.token_decimals.get(&init_transfer.token)` — returns `None` — and panics with `TokenDecimalsNotFound`.
6. The 1000 `FAKE` tokens are permanently locked in `OmniBridge`. No admin function exists to recover them. [5](#0-4) [3](#0-2)
