### Title
ETH Deposit Limit Check Missing Deposit Amount Allows Bypassing the ETH Deposit Cap - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool::_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric deposit-limit check: for LST assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it only tests `totalAssetDeposits > depositLimit`, omitting the incoming deposit amount. Any depositor can therefore push ETH deposits arbitrarily beyond the protocol-configured cap in a single call to `depositETH`.

### Finding Description
The internal helper `_checkIfDepositAmountExceedesCurrentLimit` is the sole gate that enforces per-asset deposit limits before rsETH is minted:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount NOT included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← amount included
}
```

For every LST asset the check is `totalAssetDeposits + amount > limit`, which correctly blocks a deposit that would push the running total over the cap. For ETH the check is `totalAssetDeposits > limit`, which only blocks deposits when the cap is *already* exceeded. A depositor who arrives when `totalAssetDeposits == limit` (or any value below it) passes the check regardless of how large `msg.value` is, and the full ETH amount is accepted and rsETH is minted.

The public entry point is `depositETH`, callable by any user:

```solidity
// contracts/LRTDepositPool.sol  lines 76-93
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    emit ETHDeposit(msg.sender, msg.value, rsethAmountToMint, referralId);
}
```

`_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit` and reverts on `true`. Because the ETH branch never returns `true` for a deposit that merely *reaches* the cap, the revert never fires and rsETH is minted without bound.

### Impact Explanation
The `depositLimitByAsset` cap for ETH is the primary risk-management control on how much ETH the protocol accepts and routes to EigenLayer. Bypassing it allows:

- Unlimited rsETH minting beyond the intended cap, diluting the backing ratio.
- The protocol to absorb more ETH than it can safely deploy to EigenLayer strategies, creating an operational imbalance.
- If the cap was set to bound protocol insolvency risk (e.g., EigenLayer slashing exposure), exceeding it undermines that protection.

Impact: **Low – Contract fails to deliver promised returns** (the deposit cap constraint is not enforced for ETH, breaking the protocol's stated invariant). In a scenario where the cap is a hard risk limit, the severity escalates toward protocol insolvency.

### Likelihood Explanation
The entry path is fully permissionless: any ETH holder can call `depositETH` with an arbitrarily large `msg.value`. No special role, front-running, or external dependency is required. The condition `totalAssetDeposits <= depositLimit` is the normal operating state of the protocol, so the bypass is reachable at any time before the cap is already exceeded.

### Recommendation
Add the incoming deposit amount to the ETH branch of the check, mirroring the LST branch:

```diff
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
-       return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+       return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept

1. Assume `depositLimitByAsset(ETH_TOKEN) = 1000 ether` and `totalAssetDeposits(ETH_TOKEN) = 999 ether`.
2. Attacker calls `depositETH{value: 500 ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `999 ether > 1000 ether` → `false` → no revert.
4. `_mintRsETH` mints rsETH for 500 ETH; `totalAssetDeposits` becomes 1499 ETH, 49.9% above the cap.
5. The same call can be repeated (or done in one shot from 0) to push deposits to any multiple of the cap.
