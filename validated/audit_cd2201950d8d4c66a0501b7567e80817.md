### Title
Missing Cross-Subtransaction Deduplication of Deposit Refunds Enables Double-Claiming in Dijkstra Era - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`)

---

### Summary

The Dijkstra era introduces nested transactions (TopTx/SubTx). The `dijkstraTotalRefundsTxCerts` function naively sums all `UnRegDepositTxCert` and `UnRegDRepTxCert` deposit amounts across a certificate list without deduplicating by credential. When two subtransactions within the same top-level transaction each contain an `UnRegDepositTxCert` for the same staking credential, the balance check for each subtransaction independently credits the full deposit refund. The predicate failure that should reject this scenario is explicitly acknowledged as unimplemented in the codebase. An unprivileged transaction author can therefore claim `2 × keyDeposit` in outputs while only one deposit exists in the ledger, directly creating ADA.

---

### Finding Description

`dijkstraTotalRefundsTxCerts` is the Dijkstra era's implementation of `getTotalRefundsTxCerts`:

```haskell
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert _ deposit -> deposit
  _ -> zero
``` [1](#0-0) 

This function is wired in as the era's refund calculator:

```haskell
getTotalRefundsTxCerts _ _ _ = dijkstraTotalRefundsTxCerts
``` [2](#0-1) 

Unlike the Conway/Shelley implementations (`shelleyTotalRefundsTxCerts`, `conwayDRepRefundsTxCerts`), which track seen credentials in an accumulator set to prevent double-counting within a single transaction, `dijkstraTotalRefundsTxCerts` performs no such deduplication. [3](#0-2) 

The Dijkstra era's `getTotalRefundsTxBody` calls this function only on the top-level transaction's own certificate list:

```haskell
getTotalRefundsTxBody pp lookupStakingDeposit lookupDRepDeposit txBody =
  getTotalRefundsTxCerts pp lookupStakingDeposit lookupDRepDeposit (txBody ^. certsTxBodyL)
``` [4](#0-3) 

Each subtransaction is validated independently by the SUBLEDGERS rule, and each subtransaction's own balance check uses `getTotalRefundsTxBody` on that subtransaction's body. There is no batch-level check that aggregates refunds across all subtransactions and compares them against the actual deposited amounts (contrast with `validateBatchWithdrawals`, which does perform such a cross-subtransaction check for withdrawals). [5](#0-4) 

The developers are aware of this gap. The test `"Multiple subtransactions cannot get the same refund"` is disabled with `xit` and ends with:

```haskell
submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [6](#0-5) 

The `error "TODO: predicate failure not yet implemented"` is the expected failure value passed to `submitFailingTx`. Because the test is disabled, the transaction currently **succeeds** when it should fail. The comment is an explicit acknowledgment that the rejection predicate has not been implemented.

---

### Impact Explanation

An attacker who registers a staking credential (paying `keyDeposit` once) can construct a top-level Dijkstra transaction containing two subtransactions, each with `UnRegDepositTxCert stakingCred keyDeposit`. Because the missing predicate failure is not enforced:

1. **subTx1** passes its balance check (credits `keyDeposit` as refund) and deregisters the credential.
2. **subTx2** also passes its balance check (credits `keyDeposit` as refund again) — the second deregistration of an already-deregistered credential is not rejected.

The attacker's outputs across the batch total `2 × keyDeposit` more than their inputs, but only `1 × keyDeposit` was ever deposited. The difference is drawn from the deposit pot, permanently destroying ADA that belongs to the protocol.

This matches: **Critical — Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition.**

---

### Likelihood Explanation

The attack requires no privileged access, no governance majority, and no external dependency. Any transaction author can register a staking credential and immediately craft the double-refund batch transaction. The Dijkstra era is the newest era (protocol version 12+), so the window exists from the moment it activates. The disabled test and the `TODO` comment confirm the developers have identified the scenario but have not yet closed it.

---

### Recommendation

1. **Add a batch-level refund check** analogous to `validateBatchWithdrawals`: aggregate all `UnRegDepositTxCert` and `UnRegDRepTxCert` refunds across the top-level transaction and all subtransactions, then verify that no credential's total claimed refund exceeds its actual recorded deposit.

2. **Fix `dijkstraTotalRefundsTxCerts`** to deduplicate by credential within a single certificate list, mirroring the accumulator pattern in `shelleyTotalRefundsTxCerts`. [3](#0-2) 

3. **Implement and enable the disabled test** `"Multiple subtransactions cannot get the same refund"` with the correct predicate failure once the fix is in place. [6](#0-5) 

---

### Proof of Concept

```
1. Register stakingCred in a prior transaction, paying keyDeposit.

2. Construct:
     subTx1 = mkBasicTx mkBasicTxBody
               & bodyTxL . certsTxBodyL .~ [UnRegDepositTxCert stakingCred keyDeposit]
     subTx2 = mkBasicTx mkBasicTxBody
               & bodyTxL . certsTxBodyL .~ [UnRegDepositTxCert stakingCred keyDeposit]
     topTx  = mkBasicTx mkBasicTxBody
               & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]

3. Submit topTx.

Expected (correct) result: transaction rejected — second deregistration of an
already-deregistered credential must fail.

Actual (current) result: transaction accepted — both subtransactions pass their
individual balance checks, each crediting keyDeposit as a refund. The attacker
receives 2 × keyDeposit in outputs while only 1 × keyDeposit was deposited,
directly creating ADA from the deposit pot.
```

The exact scenario is captured in the disabled test at [6](#0-5)  and the unguarded refund summation is at [1](#0-0) .

shelleyTotalRefundsTxCerts pp lookupDeposit = snd . F.foldl' accum (mempty, Coin 0)
  where
    keyDeposit = pp ^. ppKeyDepositL
    accum (!regCreds, !totalRefunds) cert =
      case lookupRegStakeTxCert cert of
        Just k ->
          -- Need to track new delegations in case that the same key is later deregistered in
          -- the same transaction.
          (Set.insert k regCreds, totalRefunds)
        Nothing ->
          case lookupUnRegStakeTxCert cert of
            Just cred
              -- We first check if there was already a registration certificate in this
              -- transaction.
              | Set.member cred regCreds -> (Set.delete cred regCreds, totalRefunds <+> keyDeposit)
              -- Check for the deposit left during registration in some previous
              -- transaction. This de-registration check will be matched first, despite being
              -- the last case to match, because registration is not possible without
              -- de-registration.
              | Just deposit <- lookupDeposit cred -> (regCreds, totalRefunds <+> deposit)
            _ -> (regCreds, totalRefunds)

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
