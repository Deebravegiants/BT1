### Title
TEE Safety Guard Blocks Resolution of Already-Pending CKD and Foreign-TX Requests When Attestations Expire Below Threshold — (File: `crates/contract/src/lib.rs`)

---

### Summary

When `verify_tee()` determines that kicking out participants with expired attestations would break the threshold relation, it sets `self.accept_requests = false`. This flag is checked not only for new signing requests but also inside `respond_ckd` and `respond_verify_foreign_tx` — the endpoints MPC nodes use to submit computed responses to **already-accepted** pending requests. When `accept_requests = false`, those response submissions are rejected, the pending yield-resume promises time out (~200 blocks), and the user's cross-chain operations permanently fail.

---

### Finding Description

`verify_tee()` is the TEE safety guard. When the number of participants with valid attestations falls below the threshold relation, it sets `self.accept_requests = false` and returns `Ok(false)`: [1](#0-0) 

This flag is then checked in `check_request_preconditions` to block new requests: [2](#0-1) 

But the same flag is **also** checked inside `respond_ckd` — the endpoint MPC nodes call to deliver a computed CKD response to a pending yield: [3](#0-2) 

And inside `respond_verify_foreign_tx`: [4](#0-3) 

These two `respond_*` methods resolve **already-accepted** pending requests — requests that were enqueued while `accept_requests` was `true`. Blocking them with the same flag conflates "stop accepting new work" with "abandon in-flight work."

The race window is:

1. User calls `sign`/`request_app_private_key`/`verify_foreign_transaction` → accepted, yield-resume promise created.
2. MPC nodes compute the response (takes several blocks).
3. Any single participant calls `verify_tee()` while enough attestations have expired → `accept_requests = false`.
4. MPC nodes call `respond_ckd` or `respond_verify_foreign_tx` → rejected with `TeeValidationFailed`.
5. After ~200 blocks the yield times out; `fail_on_timeout` fires. [5](#0-4) 

The user's request is permanently lost. There is no retry path once the yield has timed out.

The `verify_tee()` function is callable by **any single participant** (below threshold): [6](#0-5) 

The existing sandbox test explicitly confirms that `accept_requests = false` causes sign requests to be refused, and the same flag gates `respond_ckd` / `respond_verify_foreign_tx`: [7](#0-6) 

---

### Impact Explanation

**Medium — request-lifecycle manipulation that breaks production safety/accounting invariants.**

Users who have submitted CKD or foreign-chain verification requests lose those operations when `accept_requests` is set to `false` mid-flight. The pending yield-resume promises time out and cannot be recovered. For `verify_foreign_transaction` requests this means a foreign-chain transaction that was already verified by MPC nodes is never signed, leaving the user's cross-chain funds inaccessible. For CKD requests, derived key material is never returned, breaking any application that depends on it.

---

### Likelihood Explanation

- Attestation expiry is a routine operational event; if multiple participants submitted attestations at the same time, they expire together.
- Any single participant (strictly below threshold) can call `verify_tee()` at any moment — no collusion required.
- The computation window between request acceptance and `respond_*` submission spans multiple blocks, giving ample time for `verify_tee()` to fire in between.
- The contract's own test suite demonstrates the exact `accept_requests = false` state is reachable with a 3-participant / threshold-2 setup when 2 attestations expire. [8](#0-7) 

---

### Recommendation

Decouple "stop accepting new requests" from "stop resolving pending requests." Specifically:

- `respond_ckd` and `respond_verify_foreign_tx` should **not** check `accept_requests`; they are completing work that was already accepted and paid for.
- Only the user-facing entry points (`sign`, `request_app_private_key`, `verify_foreign_transaction`) should gate on `accept_requests`.

Alternatively, introduce a second boolean flag (e.g., `resolve_pending_requests`) that is never set to `false` by `verify_tee()`, and use it exclusively in the `respond_*` methods.

---

### Proof of Concept

```
Setup: 3 participants, GovernanceThreshold = 2, one CKD domain.

1. User calls request_app_private_key(domain_id=CKD, ...)
   → accepted (accept_requests = true), yield promise created.

2. MPC nodes compute the CKD response (takes ~5 blocks).

3. Meanwhile, 2 participants' attestations expire simultaneously.

4. Participant-0 calls verify_tee():
   → reverify_and_cleanup_participants returns Partial{1 valid participant}
   → validate_governance_against_reconstruction fails (1 < threshold 2)
   → self.accept_requests = false   ← safety guard fires

5. MPC nodes call respond_ckd(request, response):
   → if !self.accept_requests { return Err(TeeError::TeeValidationFailed) }
   → REJECTED

6. ~200 blocks later: yield times out → fail_on_timeout fires
   → user's CKD request permanently fails, derived key never returned.
```

The attacker-controlled entry path is `verify_tee()`, callable by any single participant. The necessary precondition (enough attestations expired) arises naturally from normal attestation lifecycle events and does not require threshold collusion.
