### Title
Compromised CC Hot Key Race Condition Allows Unauthorized Governance Votes Before Re-Authorization Confirms — (File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs)

---

### Summary

In the Conway era, a Constitutional Committee (CC) member authorizes a hot key to vote on governance actions on their behalf. If that hot key is compromised, the CC member can replace it by submitting a new `ConwayAuthCommitteeHotKey` certificate. However, there is a race condition: if the attacker casts a governance vote with the compromised hot key before the CC member's re-authorization transaction is confirmed **and** an epoch boundary (ratification) passes before the re-authorization, the attacker's vote is counted in governance ratification. The ledger provides no mechanism to invalidate in-flight votes when a hot key is replaced.

---

### Finding Description

The `conwayGovCertTransition` function in `GovCert.hs` handles `ConwayAuthCommitteeHotKey` by calling `checkAndOverwriteCommitteeMemberState`, which simply overwrites the `csCommitteeCreds` map entry for the cold credential with the new hot credential:

```haskell
ConwayAuthCommitteeHotKey coldCred hotCred ->
  checkAndOverwriteCommitteeMemberState coldCred $ CommitteeHotCredential hotCred
``` [1](#0-0) 

The `checkAndOverwriteCommitteeMemberState` helper performs no invalidation of existing votes; it only checks for prior resignation and membership, then inserts the new state:

```haskell
pure $
  certState
    & certVStateL . vsCommitteeStateL . csCommitteeCredsL %~ Map.insert coldCred newMemberState
``` [2](#0-1) 

At epoch-boundary ratification, `committeeAcceptedRatio` in `Ratify.hs` resolves each cold credential to its **current** hot key from `csCommitteeCreds` (the `reCommitteeState` snapshot taken at the epoch boundary), then looks that hot key up in `gasCommitteeVotes`:

```haskell
Just (CommitteeHotCredential hotKey) ->
  case Map.lookup hotKey votes of
    Nothing -> (yes, tot + 1) -- member hasn't voted, vote "no"
    Just VoteYes -> (yes + 1, tot + 1) -- member voted "yes"
``` [3](#0-2) 

The `votes` map (`gasCommitteeVotes`) is keyed by hot credential and is **never pruned** when a hot key is replaced. The race condition is:

1. CC member has hot key **H1** registered in `csCommitteeCreds`.
2. H1 is compromised; attacker submits a `CommitteeVoter H1` vote (VoteYes) on a critical governance action G. This vote is stored in `gasCommitteeVotes[G][H1] = VoteYes`.
3. CC member submits `AuthCommitteeHotKeyTxCert(cold, H2)` to replace H1.
4. **If the epoch boundary (RATIFY) fires before the CC member's re-authorization is included in a block**, `reCommitteeState` still maps `cold → H1`. The ratification logic finds `H1` in `gasCommitteeVotes` and counts the attacker's VoteYes.
5. The governance action may be ratified with the attacker's unauthorized vote.

If the CC member's re-authorization is confirmed before the epoch boundary, `reCommitteeState` maps `cold → H2`, H2 has no vote, and the attacker's H1 vote is silently discarded. The entire protection depends on the CC member winning a race against the epoch boundary — a window of up to the full epoch length (currently 5 days on mainnet).

The `CommitteeAuthorization` type and `CommitteeState` store only the current authorization; there is no tombstone or version counter for votes: [4](#0-3) 

The witness requirement for `ConwayAuthCommitteeHotKey` is the **cold** credential only:

```haskell
ConwayAuthCommitteeHotKey coldCred _hotCred -> credKeyHashWitness coldCred
``` [5](#0-4) 

This means the attacker holding only the compromised hot key H1 cannot submit the re-authorization themselves, but they can observe the CC member's pending re-authorization in the mempool and front-run it with a vote transaction.

---

### Impact Explanation

If the attacker's vote tips the balance on a governance action (treasury withdrawal, protocol parameter change, hard fork initiation, constitutional amendment, or committee update), that action is ratified and enacted without the CC member's consent. This satisfies:

> **Critical. Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.**

The `ENACT` rule applies the ratified action immediately and irreversibly at the epoch boundary. There is no rollback mechanism short of a hard fork.

---

### Likelihood Explanation

**Prerequisites:**
- A CC member's hot key must be compromised (e.g., via server breach, key exfiltration, or insider threat). Hot keys are kept online by design, making them a higher-risk target than cold keys.
- The attacker must submit a vote transaction that is included in a block before the CC member's re-authorization transaction.
- The epoch boundary must pass before the CC member's re-authorization is confirmed.

**Amplifying factors:**
- Cardano's epoch length is ~5 days, giving the attacker a large window.
- The attacker can monitor the mempool for the CC member's re-authorization and front-run it by paying a higher fee.
- CC members may not detect a hot key compromise immediately.
- A single CC vote can be decisive when the committee is near the ratification threshold.

Likelihood is **Low-Medium**: hot key compromise is a non-trivial prerequisite, but once achieved, the race window is wide and the front-running technique is straightforward.

---

### Recommendation

**Short term:** Document explicitly that CC members must re-authorize a new hot key **before the current epoch ends** after detecting a compromise, and that any votes cast by the compromised hot key during that epoch may be counted. Advise monitoring `gasCommitteeVotes` for unexpected votes from the current hot key.

**Long term:** When `ConwayAuthCommitteeHotKey` is processed, remove any existing vote entries in `gasCommitteeVotes` for the old hot key across all in-flight governance actions, or introduce a per-cold-credential vote-generation counter so that votes cast under a superseded hot key are rejected at ratification time. This would make hot key rotation atomic with vote invalidation.

### Proof of Concept

```
Epoch N, slot S:
  1. CC member cold=C, hot=H1 is registered.
     csCommitteeCreds[C] = CommitteeHotCredential H1

  2. Attacker (holding H1) submits:
       Tx_A: VoteTx { voter = CommitteeVoter H1, govActionId = G, vote = VoteYes }
     Tx_A is included in block at slot S+1.
     gasCommitteeVotes[G][H1] = VoteYes

  3. CC member submits:
       Tx_B: AuthCommitteeHotKeyTxCert(cold=C, hot=H2)
     Tx_B is pending in mempool.

  4. Epoch boundary fires at slot S+K (before Tx_B is confirmed).
     RATIFY snapshot: reCommitteeState[C] = CommitteeHotCredential H1
     committeeAcceptedRatio:
       member C → hotKey H1 → gasCommitteeVotes[G][H1] = VoteYes → counted

  5. Governance action G is ratified with the attacker's vote.
     ENACT applies G (e.g., treasury withdrawal, hard fork).

  6. Tx_B is confirmed in slot S+K+1 — too late.
     csCommitteeCreds[C] = CommitteeHotCredential H2
     H1's vote in gasCommitteeVotes is now orphaned but the damage is done.
```

committeeAcceptedRatio ::
  forall era.
  Map (Credential ColdCommitteeRole) EpochNo ->
  Map (Credential HotCommitteeRole) Vote ->
  CommitteeState era ->
  EpochNo ->
  Rational
committeeAcceptedRatio members votes committeeState currentEpoch =
  yesVotes %? totalExcludingAbstain
  where
    accumVotes ::
      (Integer, Integer) ->
      Credential ColdCommitteeRole ->
      EpochNo ->
      (Integer, Integer)
    accumVotes (!yes, !tot) member expiry
      | currentEpoch > expiry = (yes, tot) -- member is expired, vote "abstain" (don't count it)
      | otherwise =
          case Map.lookup member (csCommitteeCreds committeeState) of
            Nothing -> (yes, tot) -- member is not registered, vote "abstain"
            Just (CommitteeMemberResigned _) -> (yes, tot) -- member has resigned, vote "abstain"
            Just (CommitteeHotCredential hotKey) ->
              case Map.lookup hotKey votes of
                Nothing -> (yes, tot + 1) -- member hasn't voted, vote "no"
                Just Abstain -> (yes, tot) -- member voted "abstain"
                Just VoteNo -> (yes, tot + 1) -- member voted "no"
                Just VoteYes -> (yes + 1, tot + 1) -- member voted "yes"
    (yesVotes, totalExcludingAbstain) = Map.foldlWithKey' accumVotes (0, 0) members

data CommitteeAuthorization
  = -- | Member authorized with a Hot credential acting on behalf of their Cold credential
    CommitteeHotCredential !(Credential HotCommitteeRole)
  | -- | Member resigned with a potential explanation in Anchor
    CommitteeMemberResigned !(StrictMaybe Anchor)
  deriving (Eq, Ord, Show, Generic)

instance NoThunks CommitteeAuthorization

instance NFData CommitteeAuthorization

instance ToJSON CommitteeAuthorization

instance EncCBOR CommitteeAuthorization where
  encCBOR =
    encode . \case
      CommitteeHotCredential cred -> Sum CommitteeHotCredential 0 !> To cred
      CommitteeMemberResigned anchor -> Sum CommitteeMemberResigned 1 !> To anchor

instance DecCBOR CommitteeAuthorization where
  decCBOR =
    decode $ Summands "CommitteeAuthorization" $ \case
      0 -> SumD CommitteeHotCredential <! From
      1 -> SumD CommitteeMemberResigned <! From
      k -> Invalid k

newtype CommitteeState era = CommitteeState
  { csCommitteeCreds :: Map (Credential ColdCommitteeRole) CommitteeAuthorization
  }
