### Summary

The `is_caller_an_attested_participant` function, which gates every critical node operation (`respond`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`, `vote_abort_key_event_instance`), checks only for the **existence** of a stored attestation entry, never its **expiry or current validity**. Because attestation re-validation (`verify_tee`) is driven entirely by a periodic off-chain bot call, any window in which that call is delayed or fails leaves nodes whose TEE attestations have expired able to submit signature shares and vote on key events â€” bypassing the TEE security guarantee that is the system's primary trust anchor.

**The off-chain update pattern (direct analog to H-03)**

The design document states explicitly:

> *"Call verify_tee every 7 days â€” in order to trigger a re-validation of attestation information on the contract."*

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
