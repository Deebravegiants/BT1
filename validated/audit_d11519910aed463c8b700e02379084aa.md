Audit Report

## Title
`addLiquidity` Accepts Deposits Into a Paused Pool Without Enforcing `whenNotPaused` — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.addLiquidity` is missing the `whenNotPaused` modifier that `swap` correctly applies. When the pool is paused at level 1 or 2 (e.g., due to an oracle anomaly or active exploit), swaps are blocked but any address can still call `addLiquidity`, pulling real tokens from the caller and committing them to persistent pool state. This bypasses the intended safety boundary of the pause mechanism and puts depositor principal at direct risk.

## Finding Description
The `whenNotPaused` modifier (L174–177) calls `_checkNotPaused()` (L643–645), which reverts with `PoolPaused` whenever `pauseLevel != 0`. `swap` (L217–224) carries this modifier. `addLiquidity` (L182–196) carries only `nonReentrant(PoolActions.ADD_LIQUIDITY)` — no pause check. When `setPause` (L455–461) sets `pauseLevel` to 1 or 2, the swap path is correctly blocked, but the `addLiquidity` path remains fully open. `LiquidityLib.addLiquidity` is then called unconditionally, executing the token-pull callback and updating `binTotals`, `_binStates`, `_binTotalShares`, and `_positionBinShares` with the depositor's funds — all persistent state changes that commit principal to the pool. ## Impact Explanation
A depositor who calls `addLiquidity` while `pauseLevel` is 1 or 2 transfers real token balances into a pool halted for a safety reason. If the pause was triggered by oracle manipulation or a bin-price exploit, the freshly deposited tokens are immediately exposed to that same risk once the pool is unpaused or if another non-paused entry point is reachable. This constitutes a direct loss of user principal that bypasses the protocol's intended emergency safety boundary — a medium-to-high severity impact under Sherlock thresholds.

## Likelihood Explanation
No special privilege is required. Any EOA or bot monitoring mempool liquidity opportunities can call `addLiquidity` at any time, including while the pool is paused. Protocol-level pauses (level 2) are explicitly designed for emergencies, making the window of exposure real and repeatable. The missing guard requires zero attacker capability beyond submitting a standard `addLiquidity` transaction. [4](#0-3) 

## Recommendation
Add `whenNotPaused` to `addLiquidity`, mirroring the guard on `swap`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
```

`removeLiquidity` should remain unguarded so LPs can always exit a paused pool.

## Proof of Concept
1. Deploy pool; pool operates normally (`pauseLevel == 0`).
2. Factory calls `pool.setPause(2)` → `pauseLevel = 2`. `swap` now reverts with `PoolPaused`.
3. Alice calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`.
4. The call succeeds — no pause check exists on `addLiquidity`.
5. `LiquidityLib.addLiquidity` executes the callback, pulling Alice's tokens, and updates `binTotals.scaledToken0/1`, `_binStates`, `_binTotalShares`, and `_positionBinShares`.
6. Alice's principal is now locked in a pool paused for a safety reason. If the underlying issue (e.g., manipulated oracle price) is exploited before or after unpausing, Alice's deposit is drained through a subsequent swap at the bad price. [5](#0-4) [6](#0-5)
