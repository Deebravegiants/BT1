### Title
Zero-Amount wrsETH Mint on Dust Deposits Due to Unchecked Rounding-to-Zero in Rate Calculation - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPoolNoWrapper.sol, RSETHPool.sol)

### Summary

All L2 pool deposit functions compute `rsETHAmount` via a division that can silently round down to zero for dust-sized inputs. Unlike the reference report (which reverts on zero voting power), these pools have **no zero-check on `rsETHAmount`**, so the deposit proceeds: the user's ETH or tokens are accepted by the pool, but `wrsETH.mint(msg.sender, 0)` (or `safeTransfer(..., 0)`) is called, delivering nothing to the depositor.

### Finding Description

Every pool variant shares the same rate calculation pattern:

**ETH path:**
```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```
**Token path:**
```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

Because `rsETHToETHrate` grows over time as yield accrues (it starts at `1e18` and increases), any deposit where `amountAfterFee < rsETHToETHrate / 1e18` (i.e., `amountAfterFee = 1 wei` once the rate exceeds `1e18`) produces `rsETHAmount = 0`.

After the calculation, the deposit functions proceed unconditionally:

```solidity
// RSETHPoolV3 / RSETHPoolV3ExternalBridge / RSETHPoolV3WithNativeChainBridge
wrsETH.mint(msg.sender, rsETHAmount);   // mints 0 — no revert

// RSETHPool (Arbitrum)
IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);  // transfers 0 — no revert

// RSETHPoolNoWrapper
rsETH.safeTransfer(msg.sender, rsETHAmount);  // transfers 0 — no revert
``` [5](#0-4) [6](#0-5) 

There is no guard equivalent to the reference report's `if (votingPower == 0) revert ZeroVotingPowerError()`. The only input guard is `if (amount == 0) revert InvalidAmount()`, which checks the raw deposit amount, not the computed output. [7](#0-6) 

Additionally, the `limitDailyMint` modifier computes `rsETHAmount` and adds it to `dailyMintAmount`. When `rsETHAmount = 0`, the daily-limit accumulator is not incremented, meaning the daily cap is silently bypassed for dust deposits. [8](#0-7) 

### Impact Explanation

A depositor who sends a dust amount of ETH or a supported token (e.g., 1 wei) will have their funds accepted by the pool contract but will receive **zero wrsETH** in return. The deposited ETH/tokens remain in the pool (eventually bridged to L1), while the user holds nothing. This constitutes a direct, permanent loss of the deposited value for the affected user. The impact maps to **Low — Contract fails to deliver promised returns** (the user receives 0 wrsETH for a non-zero deposit), with a secondary bypass of the daily mint-limit accounting.

### Likelihood Explanation

The condition is reachable by any unprivileged depositor without front-running or special setup: simply calling `deposit{value: 1}("")` on any pool where `rsETHToETHrate > 1e18` (which is the normal operating state once any yield has accrued) triggers the zero-output path. No admin action or oracle manipulation is required. The practical monetary loss per incident is negligible (dust amounts), but the code path is unconditionally reachable.

### Recommendation

Add an explicit zero-check on the computed `rsETHAmount` before proceeding with the deposit, mirroring the reference report's fix:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert ZeroRsETHAmount();
```

Apply this check in all pool variants (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolNoWrapper`, `RSETHPool`) for both the ETH and token deposit paths. This ensures users always receive a non-zero amount of wrsETH for any accepted deposit, and prevents silent fund loss.

### Proof of Concept

```solidity
// Assume rsETHToETHrate = 1.05e18 (5% yield accrued, normal operating state)
// feeBps = 0 for simplicity

// User calls: pool.deposit{value: 1}("")
// Inside viewSwapRsETHAmountAndFee(1):
//   fee = 1 * 0 / 10_000 = 0
//   amountAfterFee = 1
//   rsETHAmount = 1 * 1e18 / 1.05e18 = 0  (rounds down)

// Back in deposit():
//   feeEarnedInETH += 0
//   wrsETH.mint(msg.sender, 0)   <-- user receives nothing
//   emit SwapOccurred(msg.sender, 0, 0, referralId)

// Result: 1 wei of ETH is permanently held by the pool;
//         the user's wrsETH balance is unchanged.
```
