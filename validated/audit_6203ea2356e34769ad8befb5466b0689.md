### Title
Stale-Pending Fallback Synthesizes `DeprecatedPendingBlock` With Zero `l2_gas_price`, Causing `starknet_estimateFee` to Return Structurally Incorrect Fee Estimates — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

When `read_pending_data` detects that the cached pending block's parent hash does not match the latest accepted block hash, it synthesizes a `DeprecatedPendingBlock` fallback. `DeprecatedPendingBlock` has no `l2_gas_price` field by design; the `l2_gas_price()` accessor on `PendingBlockOrDeprecated::Deprecated` unconditionally returns `GasPricePerToken::default()` (both fields zero). This zero propagates through `client_pending_data_to_execution_pending_data` into the `BlockContext` used by `exec_estimate_fee`, so the returned `FeeEstimation.l2_gas_price` is zero and the `overall_fee` undercounts any l2-gas component. Any unprivileged RPC caller using `BlockId::Tag(Tag::Pending)` during the stale-pending window receives this misleading response.

### Finding Description

**Step 1 — Fallback construction omits `l2_gas_price`** [1](#0-0) 

The fallback copies `eth_l1_gas_price` / `strk_l1_gas_price` from `latest_header`, but `DeprecatedPendingBlock` has no `l2_gas_price` field at all: [2](#0-1) 

**Step 2 — `l2_gas_price()` returns zero for every `Deprecated` variant** [3](#0-2) 

The comment "In older versions, L2 gas price was 0" is correct for historical blocks fetched from the feeder gateway, but the same code path is now used for the *current-pending* fallback, where the latest header's `l2_gas_price` is non-zero.

**Step 3 — Zero propagates into execution** [4](#0-3) 

`l2_gas_price: client_pending_data.block.l2_gas_price()` is passed directly into `ExecutionPendingData`, which becomes the `BlockContext` for `exec_estimate_fee`.

**Step 4 — `estimate_fee` reads `l2_gas_price` from `BlockContext`** [5](#0-4) 

`l2_gas_price` in the returned `FeeEstimation` comes from `block_context.block_info().gas_prices`, which was built from the zero value above. `overall_fee` is computed by the blockifier using the same zero price, so any l2-gas consumption is silently priced at zero.

**Step 5 — The RPC handler wires it all together** [6](#0-5) 

### Impact Explanation

Any unprivileged caller of `starknet_estimateFee` (or `starknet_simulateTransactions`) with `block_id = "pending"` during the stale-pending window receives a `FeeEstimation` where `l2_gas_price = 0` and `overall_fee` is undercounted by the full l2-gas component. Callers that use this estimate to set `max_fee` / resource bounds will submit transactions that revert or are rejected on-chain due to insufficient fee. The response is authoritative-looking (HTTP 200, well-formed JSON) with no indication that the price is synthetic.

This fits: **High — RPC fee estimation returns an authoritative-looking wrong value.**

### Likelihood Explanation

The stale-pending condition is a normal, recurring operational state: it occurs every time a new block is accepted before the pending-data cache is refreshed. On a busy network this window can be seconds wide and is not attacker-controlled, but it is also not rare. No privilege is required; any public RPC client can hit it.

### Recommendation

Replace the `DeprecatedPendingBlock` fallback with a `PendingBlock` (Current) that explicitly populates `l2_gas_price` from `latest_header.block_header_without_hash.l2_gas_price`, mirroring the existing handling of `l1_gas_price`:

```rust
// in read_pending_data, else branch
Ok(PendingData {
    block: PendingBlockOrDeprecated::Current(PendingBlock {
        parent_block_hash: latest_header.block_hash,
        l1_gas_price: latest_header.block_header_without_hash.l1_gas_price,
        l1_data_gas_price: latest_header.block_header_without_hash.l1_data_gas_price,
        l2_gas_price: latest_header.block_header_without_hash.l2_gas_price,
        l1_da_mode: latest_header.block_header_without_hash.l1_da_mode,
        timestamp: latest_header.block_header_without_hash.timestamp,
        sequencer_address: latest_header.block_header_without_hash.sequencer,
        starknet_version: latest_header.block_header_without_hash.starknet_version.to_string(),
        ..Default::default()
    }),
    ...
})
```

### Proof of Concept

The existing test suite already inadvertently documents the bug: tests that use a `DeprecatedPendingBlock` fixture explicitly set `pending_l2_gas_price` to `{price_in_wei:0, price_in_fri:0}` and treat that as the expected value: [7](#0-6) [8](#0-7) 

A concrete reproduction:
1. Write one accepted block with non-zero `l2_gas_price` in its header.
2. Set `pending_data` with `parent_block_hash` that does **not** match the accepted block's hash (triggering the fallback).
3. Call `starknet_estimateFee` with `BlockId::Tag(Tag::Pending)` for any v3 transaction.
4. Assert `result[0].l2_gas_price == 0` — the assertion passes, confirming the bug.

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
```
