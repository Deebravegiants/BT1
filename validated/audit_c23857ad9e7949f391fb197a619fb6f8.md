### Title
stETH `transferFrom` 1-2 Wei Rounding Causes rsETH Over-Minting in `depositAsset` - (File: contracts/LRTDepositPool.sol)

---

### Summary

`LRTDepositPool.depositAsset` calculates the rsETH amount to mint using the caller-supplied `depositAmount` **before** the stETH transfer occurs. Because stETH uses share-based accounting, the actual tokens received are `depositAmount - 1` or `depositAmount - 2` wei, yet rsETH is minted for the full `depositAmount`. This creates a persistent, accumulating deficit between actual stETH held and rsETH supply.

---

### Finding Description

In `LRTDepositPool.depositAsset`:

```solidity
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);

IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
```

`_beforeDeposit` calls `getRsETHAmountToMint(asset, depositAmount)`, which computes:

```
rsethAmountToMint = (depositAmount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()
```

This uses the user-supplied `depositAmount` as the basis for minting. However, stETH's `transferFrom` internally converts the amount to shares and back, and due to integer rounding, the contract receives `depositAmount - 1` or `depositAmount - 2` wei of stETH. The rsETH minted is still based on the original `depositAmount`, so the protocol mints rsETH for 1–2 wei of stETH it never actually received.

This is the same root cause documented in [Lido's own issue tracker](https://github.com/lidofinance/lido-dao/issues/442): stETH balances are share-denominated and `transferFrom` rounds down.

---

### Impact Explanation

Every stETH deposit creates a 1–2 wei gap between actual stETH held and the stETH value backing the minted rsETH. Over a large number of deposits, this accumulates:

- The rsETH price (`totalETHValue / totalRsETHSupply`) is fractionally lower than it should be.
- Existing rsETH holders are diluted by the over-minted rsETH.
- The protocol's stETH balance is systematically less than what its rsETH supply implies, constituting a slow-growing insolvency.
- The deposit limit check inside `_beforeDeposit` uses `depositAmount` (not actual received), so the effective deposit cap is silently exceeded by 1–2 wei per deposit.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

---

### Likelihood Explanation

stETH is a core supported asset in LRT-rsETH (referenced as `LRTConstants.ST_ETH_TOKEN` throughout the codebase). Every stETH deposit via `depositAsset` triggers this rounding. Likelihood is **High** — it is not a conditional edge case but a structural property of every stETH transfer.

---

### Recommendation

Use a balance-before/balance-after pattern (FoT approach) to determine the actual received amount, and use that for rsETH minting:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
_mintRsETH(rsethAmountToMint);
```

Alternatively, use Lido's `transferShares` function so the amount transferred is exact.

---

### Proof of Concept

1. Alice holds 1e18 stETH.
2. Alice calls `LRTDepositPool.depositAsset(stETH, 1e18, minRsETH, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` based on `1e18`.
4. `safeTransferFrom` executes; due to stETH share rounding, the contract receives `1e18 - 1` wei of stETH.
5. `_mintRsETH(rsethAmountToMint)` mints rsETH for `1e18` stETH worth of value.
6. The protocol now holds `1e18 - 1` stETH but has issued rsETH backed by `1e18` stETH — a 1 wei deficit.
7. Repeated across millions of deposits, the deficit accumulates, and the rsETH price is fractionally lower than it should be, diluting all existing rsETH holders. [1](#0-0) [2](#0-1) [3](#0-2)

    function _unstakeStEth(uint256 amountToUnstake) internal {
