### Title
Dijkstra Batch Transaction Minimum Fee Excludes Sub-Transaction Reference Script Costs — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The `EraUTxO` instance for `DijkstraEra` delegates `getMinFeeTxUtxo` to `getConwayMinFeeTxUtxo`, which computes the reference-script fee component using only the top-level transaction's inputs. Dijkstra batch transactions can embed sub-transactions that also reference scripts. A Dijkstra-aware helper `batchNonDistinctRefScriptsSize` exists and is correctly used for the per-transaction reference-script *size-limit* check, but it is **not** used for the minimum-fee check. An unprivileged sender can therefore craft a batch transaction whose sub-transactions carry large reference scripts, pay a fee that satisfies the (under-counting) minimum-fee check, and impose on validators the full deserialization cost of those scripts without paying the protocol-mandated surcharge.

---

### Finding Description

**Vulnerability class:** Resource-limit / fee-accounting bug — a fee floor that is enforced for one component (top-level transaction reference scripts) is missing for another component (sub-transaction reference scripts).

**Root cause — three code locations:**

1. `DijkstraEra` `EraUTxO` instance wires `getMinFeeTxUtxo` to the Conway implementation:

   ```haskell
   getMinFeeTxUtxo = getConwayMinFeeTxUtxo
   ``` [1](#0-0) 

2. `getConwayMinFeeTxUtxo` passes only the top-level transaction's inputs to `txNonDistinctRefScriptsSize`:

   ```haskell
   getConwayMinFeeTxUtxo pparams tx utxo =
     getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
   ``` [2](#0-1) 

   `txNonDistinctRefScriptsSize` only unions the top-level transaction's `inputsTxBodyL` and `referenceInputsTxBodyL`; it never descends into sub-transactions:

   ```haskell
   txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
     where
       inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
       refScripts = getReferenceScriptsNonDistinct utxo inputs
   ``` [3](#0-2) 

3. The Dijkstra UTXO transition rule invokes the fee check via `validateFeeTooSmallUTxO`, which internally calls `getMinFeeTxUtxo` — i.e., the Conway-only version above:

   ```haskell
   {- minfee pp txTop utxo₀ ≤ txfee txb -}
   runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
   ``` [4](#0-3) 

**The correct Dijkstra-aware function already exists** but is only used for the size-limit check, not the fee check:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [5](#0-4) 

The size-limit check in the Ledger rule correctly uses `batchNonDistinctRefScriptsSize`: [6](#0-5) 

The size-limit check prevents the most extreme cases (total reference script bytes across the batch cannot exceed `ppMaxRefScriptSizePerTxG`), but within that limit an attacker can still underpay the reference-script fee surcharge for every byte contributed by sub-transactions.

---

### Impact Explanation

The reference-script tiered fee (`tierRefScriptFee`) was introduced specifically to price the deserialization overhead of large Plutus scripts and prevent DDoS attacks (see ADR 009). [7](#0-6) 

Because `getConwayMinFeeTxUtxo` is used for the minimum-fee check in the Dijkstra UTXO rule, the reference-script surcharge is computed only over the top-level transaction's inputs. Sub-transaction reference scripts are invisible to the fee check. An attacker can therefore:

- Place large Plutus reference scripts exclusively in sub-transaction inputs/reference-inputs.
- Set the top-level transaction fee to satisfy the (under-counted) minimum-fee check.
- Force every validating node to deserialize those large scripts without paying the protocol-mandated surcharge.

This maps to the **Medium** allowed impact: *"Attacker-controlled transactions… modify fees… outside design parameters."*

---

### Likelihood Explanation

The attack requires only the ability to submit a valid Dijkstra batch transaction — no privileged role, no governance majority, no key compromise. The attacker controls the sub-transaction contents and can freely place reference-script inputs there. The only constraint is the per-transaction reference-script size cap (`ppMaxRefScriptSizePerTxG`), which limits the magnitude of each individual attack but does not prevent it. The attack is deterministic and repeatable.

---

### Recommendation

Override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance to use `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
-- In eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs

getDijkstraMinFeeTxUtxo ::
  ( EraTx era
  , DijkstraEraTxBody era
  ) =>
  PParams era ->
  Tx l era ->
  UTxO era ->
  Coin
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx

instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo
```

This ensures the minimum-fee check accounts for reference scripts in all sub-transactions, consistent with how `validateAllRefScriptSize` already counts them for the size-limit check.

---

### Proof of Concept

1. Construct a Dijkstra `TopTx` with no reference inputs and a trivial top-level spend input (no reference scripts at the top level).
2. Embed one or more `SubTx` entries in `subTransactionsTxBodyL`, each with `referenceInputsTxBodyL` pointing to UTxO outputs that carry large Plutus scripts (up to `ppMaxRefScriptSizePerTxG` bytes total across the batch).
3. Set `feeTxBodyL` to the value returned by `getConwayMinFeeTxUtxo` (which ignores sub-transaction reference scripts).
4. Submit the transaction. `validateFeeTooSmallUTxO` passes because it calls `getMinFeeTxUtxo = getConwayMinFeeTxUtxo`, which sees zero reference-script bytes at the top level. `validateAllRefScriptSize` passes because the total size is within the cap.
5. Every node must deserialize the large sub-transaction reference scripts, but the fee paid does not include the `tierRefScriptFee` surcharge for those bytes — the attacker has paid less than the protocol-mandated minimum fee for the actual validation work performed.

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
