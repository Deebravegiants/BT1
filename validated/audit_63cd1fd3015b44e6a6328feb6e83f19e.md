### Title
Missing Arbitrum Sequencer Liveness Check in `EvmInspector` Allows Signing Against Stale/Reorganizable Chain State - (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

### Summary
The `ArbitrumInspector` (a type alias for `EvmInspector<Client, Arbitrum>`) performs transaction finality verification and canonical-chain checks against Arbitrum RPC data without first verifying that the Arbitrum sequencer is active. When the sequencer is down, the RPC's view of the chain is frozen at the last sequencer-produced block. A caller using `EthereumFinality::Safe` or `EthereumFinality::Latest` can cause MPC nodes to sign a `ForeignTxSignPayload` for a transaction that appears finalized on the stale chain but may be reorganized once the sequencer recovers, producing a valid threshold signature attesting to a foreign-chain event that never became canonical.

### Finding Description

`ArbitrumInspector` is defined as a pure type alias over the shared `EvmInspector`: [1](#0-0) 

`EvmInspector::extract()` performs two checks before signing: [2](#0-1) 

`verify_finality_level()` fetches the chain head at the requested finality tag and checks that the receipt's block number is at or below it: [3](#0-2) 

`verify_block_is_canonical()` re-fetches the block by number and compares hashes: [4](#0-3) 

Neither function queries a sequencer-uptime oracle or checks the age/staleness of the returned block before accepting it as authoritative. The `EthereumFinality` enum exposes `Safe` and `Latest` variants that are explicitly accepted for Arbitrum: [5](#0-4) 

On Arbitrum, `Finalized` maps to L1-confirmed batches and is reorganization-safe regardless of sequencer status. However, `Safe` (sequencer-committed but not yet posted to L1) and `Latest` (most recent sequencer block) both depend on the sequencer being live and honest. When the sequencer is down:

- The RPC's `safe` and `latest` block tags freeze at the last block the sequencer produced before going offline.
- A transaction included in that frozen block passes `verify_finality_level` (its block number ≤ the frozen head) and `verify_block_is_canonical` (the hash matches the frozen canonical view).
- When the sequencer recovers, it may reorder or drop transactions from the pre-downtime window, reorganizing those blocks.

The finality level is caller-supplied in `VerifyForeignTransactionRequestArgs` and is not enforced to a minimum of `Finalized` for L2 chains by the contract: [6](#0-5) 

### Impact Explanation

**High** — Cross-chain replay / forged foreign-chain verification causing invalid bridge execution.

The MPC network produces a valid threshold signature over a `ForeignTxSignPayload` that encodes a specific Arbitrum transaction ID and its extracted log data. If that transaction is later reorganized out of the canonical Arbitrum chain, the signature remains valid and can be replayed against any downstream bridge or application that trusts MPC-attested foreign-chain events. This enables a double-spend or invalid bridge execution: the attacker redeems assets on NEAR (or another chain) based on an Arbitrum transaction that no longer exists on the canonical chain.

### Likelihood Explanation

**Medium** — Arbitrum sequencer downtime has occurred historically (e.g., the December 2021 outage). An attacker who can predict or induce a sequencer outage, or who simply monitors for one, can time a transaction submission to exploit the frozen `safe`/`latest` head. The attack requires no privileged access, no threshold collusion, and no TEE compromise — only the ability to call `verify_foreign_transaction` with `EthereumFinality::Safe` or `EthereumFinality::Latest` during a sequencer outage window.

### Recommendation

1. **Add a sequencer-uptime check** in `EvmInspector` (or a dedicated `ArbitrumInspector` override) by querying the Arbitrum sequencer uptime feed (e.g., the Chainlink L2 Sequencer Uptime Feed contract on L1, or the Arbitrum `NodeInterface` precompile) before accepting any RPC response for `Safe` or `Latest` finality.
2. **Restrict Arbitrum finality to `Finalized` only** at the contract level (`ForeignChainRpcWhitelist` / `ChainEntry`) so that callers cannot request `Safe` or `Latest` for L2 chains whose sequencer liveness is not verified.
3. **Add a block-timestamp staleness check**: reject any block whose timestamp is older than a configurable threshold (e.g., 10 minutes), which would catch a frozen chain head during sequencer downtime regardless of finality tag.

### Proof of Concept

1. Attacker submits transaction `T` on Arbitrum; it is included in block `B` (block number `N`).
2. Arbitrum sequencer goes offline. The RPC's `safe` head is frozen at block `N` (or higher if `T` was included before the outage).
3. Attacker calls `verify_foreign_transaction` on the NEAR MPC contract with:
   - `chain = Arbitrum`, `tx_id = T`, `finality = Safe`, `extractors = [Log { log_index: 0 }]`
4. Each MPC node runs `ArbitrumInspector::extract()` → `EvmInspector::extract()`:
   - `eth_getTransactionReceipt(T)` → returns receipt with `block_number = N`, `block_hash = H`.
   - `verify_finality_level(N, Safe)` → fetches `eth_getBlockByNumber("safe")` → returns frozen head at `N` or higher → check passes.
   - `verify_block_is_canonical(N, H)` → fetches `eth_getBlockByNumber(N)` → returns same frozen block → hash matches → check passes.
5. All nodes extract the same log data and produce a valid threshold signature over `ForeignTxSignPayload { tx_id: T, extracted: [log_data] }`.
6. Sequencer recovers and reorganizes block `B`; transaction `T` is dropped from the canonical chain.
7. Attacker holds a valid MPC threshold signature attesting that transaction `T` (with its log data) was finalized on Arbitrum — a transaction that no longer exists on the canonical chain — and redeems it against a downstream bridge. [2](#0-1) [1](#0-0)

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
    }
