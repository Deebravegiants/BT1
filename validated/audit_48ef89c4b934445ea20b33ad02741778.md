### Title
Deposits for Non-Executed Update Proposals Are Permanently Lost When `do_update` Clears All Entries - (`crates/contract/src/update.rs`)

---

### Summary

When `vote_update` reaches the signing threshold and triggers `do_update`, the function clears **all** competing proposal entries from storage — not just the winning one. The deposits paid by proposers of the non-executed proposals are never refunded. This creates a permanent accounting divergence: the contract's actual NEAR balance exceeds what it legitimately holds, and proposers of losing proposals suffer an unrecoverable loss of funds.

---

### Finding Description

The `propose_update` function in `crates/contract/src/lib.rs` requires each proposer to attach a deposit calculated by `ProposedUpdates::required_deposit(&update)`, which is `env::storage_byte_cost() * bytes_used(update)`. The deposit covers the storage cost of the proposal entry. Only the excess above the required amount is refunded at proposal time; the required portion is retained by the contract.

```rust
// lib.rs:1308-1331
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
if attached < required { return Err(...); }
let id = self.proposed_updates.propose(update);
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

When `vote_update` reaches threshold, it calls `do_update`, which:

1. Removes the winning entry: `self.entries.remove(id)`
2. **Clears ALL remaining entries**: `self.entries.clear()`
3. Clears all votes: `self.vote_by_participant.clear()`

```rust
// update.rs:195-200
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
```

The `UpdateEntry` struct stores `bytes_used` — the exact value used to compute the required deposit — but this field is **never consulted during cleanup**. No refund is issued to the proposers of the cleared non-executed proposals. Their deposits are permanently absorbed into the contract's balance.

The accounting invariant broken is: *"a deposit paid to cover storage for a proposal entry must be returned when that entry is removed from storage."* After `do_update`, the storage is freed (entries cleared), but the corresponding deposits are not returned, causing the contract's actual NEAR balance to exceed what it legitimately holds.

---

### Impact Explanation

**Medium — Balance/accounting invariant corruption that breaks production safety invariants.**

Every time an update reaches threshold while competing proposals exist, the proposers of all non-executed proposals permanently lose their deposits. The `bytes_used` field stored in `UpdateEntry` (which encodes the exact deposit amount owed) is discarded without being used for refunds. For a real contract upgrade (potentially hundreds of KB of WASM), the required deposit can be substantial (storage cost × code size + overhead for 128 participant-vote slots). The contract's balance permanently diverges upward from what it should hold, with no recovery path for affected proposers.

---

### Likelihood Explanation

Moderate. The scenario requires multiple competing proposals to exist simultaneously when one reaches threshold — a realistic governance situation (e.g., two participants propose different contract versions). The loss is triggered by the normal, correct execution of `vote_update` reaching threshold. No adversarial action is required; honest participants who propose competing updates are the victims. The `bytes_used` field being stored but never used for refunds confirms this is a structural omission, not an edge case.

---

### Recommendation

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue a refund for each one using the stored `bytes_used` value to reconstruct the deposit amount:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;

    // Refund deposits for all non-executed proposals before clearing
    for (_, remaining_entry) in self.entries.iter() {
        let refund = required_deposit(remaining_entry.bytes_used);
        // refund to the original proposer — requires storing proposer AccountId in UpdateEntry
        // Promise::new(remaining_entry.proposer.clone()).transfer(refund).detach();
    }

    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
```

This requires adding a `proposer: AccountId` field to `UpdateEntry` at proposal time so the refund recipient is known. Alternatively, store the deposit amount directly in `UpdateEntry` alongside the proposer.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 100 KB contract binary, paying deposit `D_A = storage_byte_cost * bytes_used(100KB_contract)`.
2. Participant B calls `propose_update` with a different 100 KB contract binary, paying deposit `D_B`.
3. Threshold participants vote for proposal A via `vote_update`.
4. `vote_update` calls `do_update(&id_A, gas)`:
   - `self.entries.remove(&id_A)` — removes A's entry (storage freed).
   - `self.entries.clear()` — removes B's entry (storage freed), **no refund issued**.
   - `self.vote_by_participant.clear()`.
5. Participant B's deposit `D_B` is permanently retained by the contract. B has no mechanism to recover it.
6. The contract's NEAR balance is now `D_B` higher than the storage it actually holds, breaking the deposit-equals-storage-cost invariant. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
