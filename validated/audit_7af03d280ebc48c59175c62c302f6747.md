Audit Report

## Title
LP Fee Sandwich Attack via Instant Liquidity Add/Remove — (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
LP fees from swaps are credited directly to per-bin token balances (`binState.token0BalanceScaled` / `binState.token1BalanceScaled`) with no per-share fee-growth checkpoint. Because `addLiquidity` and `removeLiquidity` impose no holding period or fee-growth snapshot, an attacker can front-run any large swap, acquire a dominant share position, let the swap deposit LP fees into the bin, then immediately back-run with `removeLiquidity` to extract a proportional share of those fees — stealing fee revenue that should accrue to existing LPs.

## Finding Description
**Fee accumulation:** In `SwapMath.buyToken0InBinSpecifiedIn`, the LP fee is folded directly into the bin's token balance:

```solidity
// SwapMath.sol L636-641
uint256 token1FeeScaled = lpFeeScaledFromGrossInput(totalIn1Scaled, currBinBuyFeeX64, onePlusBuyFeeX64);
uint256 protocolFeeAmountScaled = (token1FeeScaled * spreadFeeE6) / 1e6;
binState.token0BalanceScaled -= out0Scaled.toUint104();
binState.token1BalanceScaled =
    uint256((binState.token1BalanceScaled) + totalIn1Scaled - protocolFeeAmountScaled).toUint104();
```

After the swap, `binState.token1BalanceScaled` is permanently higher by `lpFee = token1FeeScaled - protocolFeeAmountScaled`. The same pattern applies in all four swap direction functions.

**Share-proportional withdrawal with no checkpoint:** `removeLiquidity` pays out strictly proportional to the current bin balance:

```solidity
// LiquidityLib.sol L205-206
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

There is no `feeGrowthInside` accumulator or snapshot at mint time. Any shares minted before a swap immediately entitle the holder to a proportional share of fees deposited by that swap.

**No time-lock or holding period:** `addLiquidity` carries no minimum holding period:

```solidity
// MetricOmmPool.sol L182-196
function addLiquidity(...) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    ...
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(...);
    _afterAddLiquidity(...);
}
```

`removeLiquidity` only checks `msg.sender == owner`:

```solidity
// MetricOmmPool.sol L206
if (msg.sender != owner) revert NotPositionOwner();
```

The `_beforeAddLiquidity` / `_afterAddLiquidity` extension hooks can optionally gate deposits, but no such extension is deployed by default and the core pool imposes none.

**Attack flow:**

Let bin have `B1` token1, total shares `S`, incoming swap LP fee `F`:

1. **Front-run:** attacker calls `addLiquidity` with `s` shares. Cost (ceiling): `B1 * s / S` token1. New state: `B1*(S+s)/S` token1, total shares `S+s`.
2. **Swap executes:** LP fee `F` added. Bin holds `B1*(S+s)/S + F` token1.
3. **Back-run:** attacker calls `removeLiquidity` with `s` shares. Receives: `(B1*(S+s)/S + F) * s/(S+s)` = `B1*s/S + F*s/(S+s)`.
4. **Attacker profit:** `F * s/(S+s)`. With `s >> S`, attacker captures ~100% of `F`.

Existing LPs receive only `F * S/(S+s)` instead of `F`.

## Impact Explanation
Direct, quantifiable loss of LP fee revenue for all existing liquidity providers in the sandwiched bin. The attacker extracts real token balances (`token1BalanceScaled` or `token0BalanceScaled`) credited to the pool by a legitimate swap. Pool solvency is not broken, but LP fee claims are diluted: the fee portion that should accrue to long-term LPs is captured by the sandwich attacker. This constitutes a direct loss of owed LP assets above Sherlock thresholds, qualifying as Critical/High severity.

## Likelihood Explanation
High. Any large swap broadcast to a public mempool is exploitable. The attacker requires only capital to add liquidity (no special role or permission), a standard flashbots bundle or priority-fee front/back-run, and three separate transactions in one block. The `MINIMAL_MINTABLE_LIQUIDITY` floor is a dust guard, not a meaningful barrier. No default extension enforces a holding period.

## Recommendation
1. **Per-share fee-growth checkpoint (preferred):** Track a `feeGrowthInside` accumulator per bin (analogous to Uniswap v3). Snapshot the accumulator at mint time; only fees accumulated *after* a position is opened are claimable by that position. This eliminates the sandwich entirely.
2. **Minimum holding period / withdrawal delay:** Enforce a block-number or timestamp lock on newly minted positions before `removeLiquidity` is callable. This raises the cost and risk of the attack but does not eliminate it.
3. **Private mempool guidance:** Advise integrators to route large swaps through private relays (Flashbots Protect, MEV Blocker) to prevent front-running visibility.

## Proof of Concept
```
Setup:
  Bin 0: token0Balance = 1_000_000 (scaled), token1Balance = 1_000_000 (scaled)
  Total shares S = 1_000_000
  Existing LP holds all S shares.
  Incoming swap: trader sends 500_000 token1 → receives token0.
  LP fee rate: 0.3% → lpFee F ≈ 1_500 token1 (scaled) stays in bin.

Step 1 — Attacker front-runs addLiquidity(bin=0, shares=s=9_000_000):
  Cost (ceiling): 1_000_000 * 9_000_000 / 1_000_000 = 9_000_000 token1 (scaled)
  New bin token1: 10_000_000 (scaled), total shares: 10_000_000

Step 2 — Swap executes:
  Bin token1 becomes: 10_000_000 + 1_500 = 10_001_500 (scaled)

Step 3 — Attacker back-runs removeLiquidity(bin=0, shares=9_000_000):
  Receives (floor): 10_001_500 * 9_000_000 / 10_000_000 = 9_001_350 token1 (scaled)
  Attacker profit: 9_001_350 − 9_000_000 = 1_350 token1 (scaled)
  (= F * s/(S+s) = 1_500 * 9/10 = 1_350 ✓)

Existing LP receives only:
  10_001_500 * 1_000_000 / 10_000_000 = 1_000_150 token1 (scaled)
  Fee captured: 150 instead of 1_500 — 90% stolen.
```

Foundry test plan: deploy pool, seed bin with known balances and shares, execute the three-step sequence in a single test function, assert attacker profit equals `F * s/(S+s)` and existing LP fee receipt equals `F * S/(S+s)`.
