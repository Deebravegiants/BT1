### Title
Zero-Confirmation Bitcoin Transaction Verification Bypass via Missing Minimum `BlockConfirmations` Bound - (File: `crates/contract/src/lib.rs`, `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint on the MPC contract accepts a `BitcoinRpcRequest` with a caller-controlled `confirmations: BlockConfirmations(u64)` field that has no enforced minimum. An unprivileged caller can set `confirmations: 0`, causing every MPC node to accept any Bitcoin transaction — including unconfirmed mempool transactions — as "verified," and the threshold network will produce a valid ECDSA signature attesting to that unconfirmed state. Any bridge contract consuming this signature (e.g., Omnibridge inbound flow) that does not independently re-validate the `confirmations` field in the signed payload is exposed to double-spend attacks.

---

### Finding Description

**Root cause — no lower bound on `confirmations` in `verify_foreign_transaction`:**

`BlockConfirmations` is defined as a plain `u64` newtype with no minimum constraint: [1](#0-0) 

The ABI schema confirms `minimum: 0.0` is the only constraint: [2](#0-1) 

The `verify_foreign_transaction` contract entry point performs no validation on the `confirmations` field — it only checks domain purpose, gas, deposit, and chain whitelist membership: [3](#0-2) 

**Node-side enforcement is equally absent:**

In the Bitcoin inspector, the confirmation check is `block_confirmations_threshold <= transaction_block_confirmation`. When the threshold is `0`, this condition is `0 <= N` which is always `true` for any `u64` value `N`, including transactions with 0 confirmations (i.e., unconfirmed mempool transactions): [4](#0-3) 

**Exploit path:**

1. Attacker broadcasts a Bitcoin transaction `T` to the mempool (0 confirmations).
2. Attacker immediately calls `verify_foreign_transaction` on the MPC contract with `BitcoinRpcRequest { tx_id: T, confirmations: 0, extractors: [BlockHash] }`.
3. Every MPC node queries its Bitcoin RPC provider. The transaction is found in the mempool with 0 confirmations. The check `0 <= 0` passes.
4. All nodes extract the block hash (or a mempool placeholder) and agree on the `ForeignTxSignPayload`.
5. The threshold network produces a valid ECDSA signature over `SHA-256(borsh(ForeignTxSignPayload))`.
6. The signed response is returned on-chain. The payload includes `confirmations: 0` in the request, but a bridge contract that does not re-inspect this field will treat the signature as proof of finality.
7. Attacker double-spends `T` on Bitcoin. The bridge has already released NEAR-side funds.

The signed payload does embed the full request (including `confirmations: 0`): [5](#0-4) 

However, the MPC contract itself provides no guarantee that the embedded `confirmations` value meets any security threshold, and the design documentation explicitly positions this feature as a replacement for a trusted relayer: [6](#0-5) 

---

### Impact Explanation

**Category: High — Cross-chain verification bypass causing double-spend conditions.**

The MPC network's `verify_foreign_transaction` feature is designed to let NEAR contracts react to foreign-chain events "without a trusted relayer." The implicit security guarantee is that the MPC signature attests to a transaction that has achieved meaningful finality. By accepting `confirmations: 0`, the contract allows the MPC network to sign off on transactions with zero finality, completely defeating the purpose of the feature. Any bridge contract that relies on the MPC signature as a finality proof — without independently re-checking the `confirmations` field embedded in the signed payload — is vulnerable to double-spend attacks where the attacker receives NEAR-side funds for a Bitcoin transaction that is subsequently reorganized out of the canonical chain.

---

### Likelihood Explanation

**High.** The attack requires no privileged access, no key material, and no collusion. Any NEAR account can call `verify_foreign_transaction` with `confirmations: 0` for a deposit of 1 yoctoNEAR. The attacker only needs to broadcast a Bitcoin transaction to the mempool and immediately submit the verification request before the transaction is confirmed. Bitcoin mempool transactions are visible to all RPC providers, so the extraction step succeeds deterministically. The attack is profitable whenever the bridge releases more value than the cost of the NEAR transaction plus the Bitcoin transaction fee.

---

### Recommendation

Enforce a protocol-level minimum on `BlockConfirmations` at the contract entry point in `verify_foreign_transaction`. Reject any `BitcoinRpcRequest` where `confirmations == 0` (or below a governance-configurable minimum, e.g., 1 for testnet, 6 for mainnet). This mirrors the pattern already used for `ChainEntry.quorum`, which rejects `quorum == 0`: [7](#0-6) 

A similar validation gate should be added for `BitcoinRpcRequest.confirmations` before the request is enqueued. Additionally, bridge contracts consuming `VerifyForeignTransactionResponse` should independently assert that the `confirmations` value in the embedded `ForeignTxSignPayload` meets their own security threshold, as defense in depth.

---

### Proof of Concept

```json
// Call verify_foreign_transaction with confirmations: 0
{
  "request": {
    "request": {
      "Bitcoin": {
        "tx_id": "<mempool_tx_id_hex>",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    },
    "derivation_path": "bridge-key",
    "domain_id": 0,
    "payload_version": "V1"
  }
}
```

With `confirmations: 0`, the Bitcoin inspector evaluates `0 <= rpc_response.confirmations` which is always `true`. The MPC network proceeds to sign the payload. The returned `VerifyForeignTransactionResponse` contains a valid threshold ECDSA signature over a payload that encodes `confirmations: 0` — a transaction with no finality guarantee. A bridge contract that does not re-inspect this field will accept the signature as proof of finality and release funds, enabling a double-spend when the attacker reorganizes the Bitcoin transaction.     pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
