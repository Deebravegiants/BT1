The code is clear. Let me trace the full attack path precisely.

**Header validation in `parse_data_for_block`** (header.rs lines 104–120):
- Checks `block_number` matches the header's block number field
- Checks `signatures.len() == ALLOWED_SIGNATURES_LENGTH` (count only, not cryptographic validity)
- No block hash verification against any trusted source
- No parent hash chain verification (explicit TODO at line 102)
- No signature cryptographic verification

**State diff consumption in `parse_data_for_block`** (state_diff.rs lines 62–70):
- Reads `target_state_diff_len` directly from the stored header's `state_diff_length` field
- Uses this as the sole gate for how many chunks to consume from the peer
- No validation of the assembled state diff against the header's `state_diff_commitment` field

**The attack chain is concrete and unbroken:**

1. Unauthenticated p2p peer sends `SignedBlockHeader` for block N with `state_diff_length = K` (arbitrary attacker-chosen value). The only checks are block number ordering and signature vector length == 1. Both pass trivially.
2. `write_to_storage` calls `append_header`, storing the attacker-controlled `state_diff_length = K` into the node's storage.
3. `StateDiffStreamBuilder` (bounded by `HeaderMarker`) advances to block N, reads `target_state_diff_len = K` from the stored header.
4. The attacker (same or colluding peer) sends exactly K fabricated `StateDiffChunk` entries. The loop at state_diff.rs line 72 exits when `current_state_diff_len == K`.
5. `append_state_diff` commits the fabricated state diff to storage.

**No existing validation prevents this.** The `state_diff_commitment` field is stored in the header but never checked against the assembled state diff anywhere in `StateDiffStreamBuilder.parse_data_for_block`.

### Title
Unauthenticated P2P Peer Can Corrupt Node State via Unverified `state_diff_length` in Stored Headers — (`crates/apollo_p2p_sync/src/client/header.rs`, `crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary
`HeaderStreamBuilder` accepts and stores headers from unauthenticated p2p peers with no cryptographic verification of the block hash or signatures. The `state_diff_length` field embedded in these headers is subsequently used by `StateDiffStreamBuilder` as the sole bound for how many state diff chunks to consume and store. A malicious peer can set `state_diff_length` to any value and supply a matching number of fabricated chunks, causing the node to commit an entirely attacker-controlled state diff to storage.

### Finding Description

`HeaderStreamBuilder` is declared with `BLOCK_NUMBER_LIMIT::Unlimited`, meaning it queries headers from peers with no upper bound. [1](#0-0) 

Its `parse_data_for_block` performs only two checks: that the block number matches the expected sequence position, and that exactly one signature is present (count only — no cryptographic verification). [2](#0-1) 

There is an explicit TODO acknowledging that parent hash chain verification is missing. [3](#0-2) 

The accepted header — including its attacker-controlled `state_diff_length` — is written unconditionally to storage. [4](#0-3) 

`StateDiffStreamBuilder` uses `BlockNumberLimit::HeaderMarker`, so it processes state diffs for every stored header. [5](#0-4) 

Its `parse_data_for_block` reads `state_diff_length` from the stored header and uses it as `target_state_diff_len` — the only termination condition for chunk consumption. [6](#0-5) 

The loop consumes chunks until `current_state_diff_len == target_state_diff_len`, then commits the result. The header's `state_diff_commitment` field is never checked against the assembled state diff. [7](#0-6) 

The fabricated state diff is then committed to storage via `append_state_diff`. [8](#0-7) 

### Impact Explanation

The node's committed state (storage slots, deployed contract addresses, class hashes, nonces) is fully attacker-controlled for any block whose header was served by the malicious peer. All subsequent execution by the blockifier reads from this corrupted state, producing wrong execution results, wrong fee computations, wrong class lookups, and wrong RPC responses for storage/class queries. The corruption is persistent and affects every block after the first poisoned header.

### Likelihood Explanation

P2P sync is designed to accept connections from unauthentced peers. No special privilege is required. The attacker only needs to be selected as the responding peer for a header query, which is a normal p2p network event. The attack requires no cryptographic material and no knowledge of real block content.

### Recommendation

1. **Verify block signatures cryptographically** in `HeaderStreamBuilder.parse_data_for_block` before accepting a header. The signature must be checked against the sequencer's known public key.
2. **Validate `state_diff_commitment`** in `StateDiffStreamBuilder.parse_data_for_block` after assembling the full state diff, before calling `append_state_diff`. The assembled diff's commitment must match the value stored in the header.
3. **Verify parent hash chain** (the existing TODO at header.rs line 102) to prevent insertion of headers that do not extend the canonical chain.

### Proof of Concept

```
1. Attacker peer connects to the syncing node via p2p.
2. Node sends a header query for blocks [N, N+1000).
3. Attacker responds with 1000 SignedBlockHeaders, each with:
   - block_number = N+i (correct, passes ordering check)
   - signatures = [BlockSignature::default()] (length 1, passes count check)
   - state_diff_length = Some(1000) (arbitrary, not verified)
   - block_hash = arbitrary (not verified)
4. All 1000 headers pass parse_data_for_block and are written to storage.
   Header marker advances to N+1000.
5. StateDiffStreamBuilder advances, reads target_state_diff_len = 1000 for block N.
6. Attacker sends 1000 fabricated StateDiffChunk entries (e.g., setting
   storage slot 0x1 = 0xdeadbeef for 1000 different contract addresses).
7. Loop exits: current_state_diff_len == 1000 == target_state_diff_len.
8. append_state_diff commits the fabricated diff. State marker advances.
9. Repeat for blocks N+1 through N+999.
10. Node now has 1,000,000 fabricated storage entries. All RPC storage queries
    and all blockifier executions against these contracts return attacker-chosen values.
```
