### Title
Single Byzantine Participant Can Permanently Block All Foreign-Chain Verification Requests via Empty Chain Registration — (`crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which computes the **strict intersection** of every active participant's registered chain set. A single Byzantine participant (strictly below the signing threshold) can call `register_foreign_chain_config` with an empty list, collapsing the intersection to the empty set and causing every subsequent `verify_foreign_transaction` call to panic. Already-queued requests that passed the gate before the sabotage can no longer be fulfilled by nodes (which perform the same chain-support check), causing them to time out and leaving deposited funds in limbo until the yield-timeout fires.

---

### Finding Description

`verify_foreign_transaction` in `crates/contract/src/lib.rs` calls `get_supported_foreign_chains()` at line 534 and panics if the requested chain is absent:

```rust
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported { requested: requested_chain }.to_string(),
    );
}
``` [1](#0-0) 

`get_supported_foreign_chains()` computes the intersection by requiring every active participant to appear in the supporters set for a chain:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
``` [2](#0-1) 

Any authenticated participant can call `register_foreign_chain_config` (or its alias `register_available_foreign_chain_config`) with an empty `ForeignChainConfiguration`. Because the intersection requires **all** active participants, one empty registration collapses the result to the empty set. The project's own design document explicitly acknowledges this:

> "A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down."



On the node side, `execute_foreign_chain_request` calls `chain_is_supported()` before producing any signature share:

```rust
chain_is_supported(&self.foreign_chain_policy_reader, request).await?;
``` [3](#0-2) 

`chain_is_supported` reads the on-chain supported-chain set and returns an error if the chain is absent:

```rust
if on_chain_foreign_chains_support.contains(&requested_chain) {
    Ok(())
} else {
    Err(ForeignChainSupportError::ChainNotSupported { requested: requested_chain })
}
``` [4](#0-3) 

The design document confirms that a failed node-side verification produces no on-chain failure response — the request simply times out:

> "A failed verification does **not** produce an on-chain failure response. The request eventually times out and fails with the standard timeout error."



---

### Impact Explanation

**Two classes of harm:**

1. **New requests (post-sabotage):** `verify_foreign_transaction` panics before `enqueue_yield_request` is reached. NEAR's panic semantics refund the attached deposit, so no direct fund loss here — but the feature is completely unavailable.

2. **Already-queued requests (pre-sabotage):** Requests that passed the gate before the empty registration are sitting in `pending_verify_foreign_tx_requests`. Every node now sees the chain as unsupported and refuses to produce a signature share. These requests time out after ~200 blocks. Whether the yield-callback refunds the `MINIMUM_SIGN_REQUEST_DEPOSIT` on timeout depends on the callback implementation; if it does not, those deposits are permanently locked. Even if refunded, the bridge/omnibridge flow that submitted the request (e.g., an inbound cross-chain transfer waiting for MPC attestation) is broken — the attested signature is never produced, and the foreign-chain funds that triggered the flow remain inaccessible.

This maps directly to the external report's pattern: a single unreliable external data source (here, one participant's chain registration) causes the verification/settlement step to fail, blocking the entire flow.

---

### Likelihood Explanation

- Any single active participant can call `register_foreign_chain_config` with an empty payload — no threshold collusion required.
- The call is idempotent and authenticated only by participant membership, not by any additional governance vote.
- The sabotage takes effect immediately on the next `verify_foreign_transaction` call.
- Recovery requires the sabotaging participant to re-register a non-empty list, or a resharing that removes them — neither is instantaneous.

---

### Recommendation

Replace the strict-intersection rule in `get_supported_foreign_chains()` with the threshold-based availability check already designed and tracked in the project (`get_available_foreign_chains()` / `get_whitelisted_foreign_chains()`):

- A chain is **available** iff ≥ `signing_threshold` active participants cover it.
- A single non-covering participant cannot drop a chain; only more than `n − signing_threshold` non-covering participants can.
- `verify_foreign_transaction` should gate on `get_available_foreign_chains()` instead of `get_supported_foreign_chains()`.

This migration is already designed in `docs/design/calculating-supported-foreign-chains.md` and tracked under issue #3434. Until it lands, the production contract remains vulnerable to a single-participant availability attack on the entire foreign-chain verification feature.

---

### Proof of Concept

1. Network has `n` participants, signing threshold `t < n`. All participants have registered Bitcoin as a supported chain. `get_supported_foreign_chains()` returns `{Bitcoin}`.

2. Attacker (participant `P`, one of the `n`) calls:
   ```
   register_foreign_chain_config({ foreign_chain_configuration: {} })
   ```
   with an empty map.

3. `get_supported_foreign_chains()` now computes: for Bitcoin, `nodes_supporting_chain` = `{all except P}`, which is **not** a superset of `active_participant_account_ids` (which includes `P`). Result: empty set. [2](#0-1) 

4. Any caller submitting `verify_foreign_transaction` for Bitcoin hits:
   ```
   env::panic_str("Requested foreign chain, Bitcoin, is not supported.")
   ``` [5](#0-4) 

5. Any request already in `pending_verify_foreign_tx_requests` is now unserviceable: every node's `chain_is_supported()` call returns `ChainNotSupported`, no signature shares are produced, and the request times out after ~200 blocks. [4](#0-3)

async fn chain_is_supported(
    policy_reader: &impl ReadSupportedForeignChain,
    request: &dtos::ForeignChainRpcRequest,
) -> Result<(), ForeignChainSupportError> {
    let on_chain_foreign_chains_support = policy_reader
        .get_supported_chains()
        .await
        .map_err(ForeignChainSupportError::FailedToReadContract)?;

    let requested_chain = request.chain();

    if on_chain_foreign_chains_support.contains(&requested_chain) {
        Ok(())
    } else {
        Err(ForeignChainSupportError::ChainNotSupported {
            requested: requested_chain,
        })
    }
