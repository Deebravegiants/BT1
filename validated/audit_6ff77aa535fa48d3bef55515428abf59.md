### Title
Single Byzantine Node Can Inject Arbitrary CKD Response for `AppPublicKey` Requests — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs a cryptographic pairing check (`ckd_output_check`) only when the stored request carries `CKDAppPublicKey::AppPublicKeyPV`. When the request carries `CKDAppPublicKey::AppPublicKey`, the match arm is an explicit no-op. A single attested-participant node can therefore call `respond_ckd` with any `big_y` / `big_c` values it chooses, and the contract will resolve the yield and deliver the fabricated response to the waiting caller — without threshold agreement and without any cryptographic binding to the network's BLS root key.

---

### Finding Description

`request_app_private_key` accepts both `AppPublicKey` (a single G1 point) and `AppPublicKeyPV` (a G1+G2 pair). For `AppPublicKeyPV`, the contract verifies the response on-chain at `respond_ckd` time via `ckd_output_check`, which checks the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, pk)`. [1](#0-0) 

For `AppPublicKey`, the corresponding match arm is empty:

```rust
dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // no check
``` [2](#0-1) 

After this match, `resolve_yields_for` is called unconditionally, resolving the pending yield with whatever `CKDResponse` the caller supplied. [3](#0-2) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active()` — i.e., the caller must be an attested participant node, not a threshold quorum. [4](#0-3) 

Compare this with `respond` for signatures: the contract verifies the ECDSA/EdDSA signature against the derived public key before resolving the yield, so a single Byzantine node cannot produce a valid signature without the threshold secret. No equivalent cryptographic gate exists for `AppPublicKey` CKD responses. [5](#0-4) 

The `ckd_output_check` function itself is correct and complete — it verifies the full pairing equation binding `big_c` and `big_y` to both the caller's app key and the network's root key. [6](#0-5) 

---

### Impact Explanation

A single Byzantine attested-participant node can:

1. Observe any pending `AppPublicKey` CKD request on-chain.
2. Call `respond_ckd(request, CKDResponse { big_y: arbitrary, big_c: arbitrary })`.
3. The contract accepts the call (attested-participant check passes, no output check is performed) and resolves the yield.
4. The waiting caller receives `big_y` / `big_c` values that are not cryptographically bound to the network's BLS root key or the caller's `app_public_key`.

This breaks the core invariant of the CKD protocol: that every delivered `CKDResponse` satisfies `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk ‖ app_id), pk)`. The caller's derived confidential key is therefore not a valid output of the threshold MPC protocol and carries no security guarantee.

---

### Likelihood Explanation

Any single attested participant that turns Byzantine can exploit this immediately against any `AppPublicKey` CKD request. No threshold collusion, no leaked key material, and no network-level attack is required. The attacker only needs to be an active attested participant and observe the NEAR chain for pending CKD requests.

---

### Recommendation

Apply `ckd_output_check` for `AppPublicKey` requests as well. Because `AppPublicKey` supplies only a G1 point (no G2 component), the pairing equation cannot be checked in its current form. Two options:

1. **Require `AppPublicKeyPV` for all CKD requests** — remove the `AppPublicKey` variant or reject it at `request_app_private_key` time, forcing callers to supply both G1 and G2 components so the on-chain check is always possible.
2. **Add a threshold-agreement layer in the contract** — require `t` distinct attested participants to submit the same `(request, response)` pair before resolving the yield, mirroring how the node layer is supposed to work but enforcing it on-chain.

Option 1 is simpler and consistent with the existing `AppPublicKeyPV` path.

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
