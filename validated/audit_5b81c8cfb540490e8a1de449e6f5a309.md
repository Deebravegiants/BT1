Audit Report

## Title
Permissionless `sendFunds()` in FeeReceiver Enables Yield Dilution Against Existing rsETH Holders - (File: contracts/FeeReceiver.sol)

## Summary

`FeeReceiver.sendFunds()` has no access control, allowing any caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at will. Because `LRTOracle.updateRSETHPrice()` is also publicly callable and reads live pool balances, an attacker can deposit ETH, trigger `sendFunds()`, then call `updateRSETHPrice()` to capture a disproportionate share of the accumulated rewards at the expense of long-standing rsETH holders. The impact is theft of unclaimed yield.

## Finding Description

`FeeReceiver.sendFunds()` is declared `external` with no role check:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

The contract inherits `AccessControlUpgradeable` and grants `LRTConstants.MANAGER` to a manager address during `initialize`, but `sendFunds()` carries no `onlyRole` modifier. Other state-mutating functions like `setDepositPool` are correctly gated with `onlyRole(LRTConstants.MANAGER)`.

Once ETH lands in `LRTDepositPool`, `getETHDistributionData()` immediately reflects it via `address(this).balance`:

```solidity
// contracts/LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

This feeds `getTotalAssetDeposits` → `_getTotalEthInProtocol()` in `LRTOracle`, which is used by the public `updateRSETHPrice()` to compute a new `rsETHPrice`. Critically, `rsETHPrice` is a **stored state variable** updated only on explicit calls to `updateRSETHPrice()` — it does not update automatically when ETH arrives. However, `updateRSETHPrice()` is itself public and callable by anyone when the price increase is within `pricePercentageLimit`:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The `pricePercentageLimit` guard only reverts non-managers when `newRsETHPrice > highestRsethPrice` by more than the configured threshold. For typical MEV reward sizes (small fraction of TVL), the price increase is within any reasonable limit, and the attacker can call `updateRSETHPrice()` themselves. If `pricePercentageLimit == 0`, the check is entirely bypassed.

`initiateWithdrawal` locks in the ETH amount at the current `rsETHPrice` at the time of the call via `getExpectedAssetAmount`, so the attacker secures the elevated redemption value before the delay period begins.

The complete attack path:
1. Attacker deposits ETH via `LRTDepositPool.depositETH()` at the current (pre-reward) rsETH price, receiving rsETH.
2. Attacker calls `FeeReceiver.sendFunds()` — accumulated MEV rewards flow into the pool.
3. Attacker calls `LRTOracle.updateRSETHPrice()` — price rises to reflect the new ETH in the pool.
4. Attacker calls `LRTWithdrawalManager.initiateWithdrawal()` — the elevated ETH amount is locked in.
5. After the withdrawal delay (~8 days), attacker claims ETH, receiving more than deposited.

No existing check prevents steps 2–4 from being executed atomically or in rapid succession by an unprivileged caller.

## Impact Explanation

This is **theft of unclaimed yield** (High severity per the allowed scope). MEV and execution-layer rewards accumulate in `FeeReceiver` and are intended to accrue pro-rata to all existing rsETH holders when distributed. By timing a large deposit immediately before triggering `sendFunds()` and `updateRSETHPrice()`, an attacker captures a fraction of those rewards proportional to their deposit relative to total TVL, diluting the yield that should have gone to pre-existing holders. The stolen amount scales linearly with the attacker's deposit size and the accumulated reward balance.

## Likelihood Explanation

- `FeeReceiver` accumulates ETH passively and its balance is publicly visible on-chain.
- `sendFunds()` requires no privilege, no signature, and no special state.
- `LRTDepositPool.depositETH` is open to any caller with no deposit fee.
- `LRTOracle.updateRSETHPrice()` is public and callable by anyone when the price increase is within `pricePercentageLimit` (or when `pricePercentageLimit == 0`).
- The only friction is the ~8-day withdrawal delay (`withdrawalDelayBlocks = 8 days / 12 seconds`), which reduces but does not eliminate profitability when the FeeReceiver holds a large accumulated balance.
- **Likelihood: Medium** — requires capital and patience through the withdrawal delay, but is otherwise trivially executable by any on-chain observer.

## Recommendation

1. Add an access-control modifier to `sendFunds()` so only a trusted operator or keeper role can trigger it:
   ```solidity
   function sendFunds() external onlyRole(LRTConstants.MANAGER) { ... }
   ```
2. Alternatively, route MEV reward distribution through a private/Flashbots relay so the transaction cannot be front-run.
3. Consider a deposit fee or a minimum holding period before withdrawal to make the attack economically unattractive even if `sendFunds()` remains permissionless.

## Proof of Concept

**Setup:** FeeReceiver holds 100 ETH. Total protocol ETH = 10,000 ETH. rsETH supply = 9,500 rsETH. Stored `rsETHPrice` ≈ 1.0526 ETH/rsETH. `pricePercentageLimit` is 0 or large enough to accommodate a ~1% price move.

1. **Attacker deposits 10,000 ETH** via `LRTDepositPool.depositETH`. At the current stored price they receive `10,000 / 1.0526 ≈ 9,500 rsETH`. Total ETH in pool = 20,000; rsETH supply = 19,000; stored price unchanged.

2. **Attacker calls `FeeReceiver.sendFunds()`**. 100 ETH flows into the deposit pool. Total ETH = 20,100; rsETH supply = 19,000.

3. **Attacker calls `LRTOracle.updateRSETHPrice()`**. New price = `20,100 / 19,000 ≈ 1.0579 ETH/rsETH`. Stored `rsETHPrice` is updated.

4. **Attacker calls `LRTWithdrawalManager.initiateWithdrawal()`** with their 9,500 rsETH. `getExpectedAssetAmount` locks in `9,500 × 1.0579 ≈ 10,050 ETH`.

5. **After ~8-day delay**, attacker claims ~10,050 ETH — a net gain of ~50 ETH extracted from the 100 ETH reward that should have accrued to the 9,500 pre-existing rsETH holders.

The attack scales linearly: the larger the attacker's deposit relative to existing TVL, the greater the fraction of rewards captured.
