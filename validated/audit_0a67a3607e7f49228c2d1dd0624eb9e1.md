### Title
Missing zero-scalar guard in `ec_pve_batch_t::encrypt` produces an unverifiable ciphertext — (`src/cbmpc/protocol/pve_batch.cpp`)

### Summary
`encrypt_batch` accepts a zero scalar (all-zero bytes input), reduces it modulo the curve order to produce `x=0`, computes `Q = 0·G = ∞`, and returns `SUCCESS` with a ciphertext embedding the infinity point as the public key. Every subsequent call to `verify_batch` on that ciphertext is guaranteed to fail because `ecurve_t::check` unconditionally rejects infinity. The encrypt and verify layers permanently disagree on validity.

### Finding Description
In `ec_pve_batch_t::encrypt`:

```cpp
for (int j = 0; j < n; j++) {
    x[j] = _x[j] % q;   // 0 % q = 0
    Q[j] = x[j] * G;    // 0 * G = infinity — no guard here
}
``` [1](#0-0) 

The API wrapper converts the raw bytes to a bignum and passes it straight through:

```cpp
for (int i = 0; i < n; i++)
    x_bn.push_back(coinbase::crypto::bn_t::from_bin(xs[i]));
// ...
rv = pve_ct.encrypt(bridge, ..., icurve, x_bn);
``` [2](#0-1) 

The only scalar-size guard is a byte-length check, not a value check:

```cpp
if (x.size > max_x_size) return coinbase::error(E_RANGE, "xs element too large");
``` [3](#0-2) 

`validate_mem_vec_arg` only rejects empty (`size <= 0`) elements, not zero-valued ones: [4](#0-3) 

Meanwhile, `ecurve_t::check` — called in both the API-layer `verify_batch` and the internal `ec_pve_batch_t::verify` — unconditionally rejects infinity:

```cpp
if (!thread_local_store_allow_ecc_infinity) {
    if (point.is_infinity()) return crypto::error("EC-point is infinity");
}
``` [5](#0-4) 

The internal verify also calls `curve.check(Q[i])` on the caller-supplied Qs before comparing them to the stored ones: [6](#0-5) 

There is no path through `verify_batch` that can succeed for a ciphertext whose stored `Q[i]` is infinity.

### Impact Explanation
`encrypt_batch` returns `SUCCESS` and writes a well-formed serialized blob, but the blob contains `Q=∞` and is permanently unverifiable. Any application that trusts the `SUCCESS` return code and stores or transmits the ciphertext will later find that `verify_batch` always fails. This is a concrete encrypt/verify disagreement on a public API: the encryption layer accepts and emits invalid cryptographic output that the verification layer is guaranteed to reject.

### Likelihood Explanation
The trigger is a single API call with `xs[0]` set to 32 zero bytes — a trivially constructible input. The API documentation states inputs are "reduced modulo the curve order" with no mention of a non-zero requirement, so callers have no documented reason to pre-validate. The issue is reachable from all three public API variants (`encrypt_batch` with default base PKE, custom base PKE, and the AC variant `encrypt_ac` which has the identical pattern). [7](#0-6) 

### Recommendation
Add a non-zero check immediately after the modular reduction in `ec_pve_batch_t::encrypt`:

```cpp
x[j] = _x[j] % q;
if (x[j].is_zero()) return coinbase::error(E_BADARG, "scalar must be non-zero");
Q[j] = x[j] * G;
```

Equivalently, the API wrapper in `encrypt_batch` can check `bn_t(0) == x_bn[i]` after `from_bin` and before calling the internal encrypt. The same fix is needed in `ec_pve_ac_t::encrypt`. [1](#0-0) 

### Proof of Concept
```cpp
// xs[0] = 32 zero bytes
std::array<uint8_t, 32> zero_bytes{};
std::vector<mem_t> xs;
xs.emplace_back(zero_bytes.data(), 32);

buf_t ct;
// Returns SUCCESS — produces ciphertext with Q[0] = infinity
assert(coinbase::api::pve::encrypt_batch(curve_id::secp256k1, ek, label, xs, ct) == SUCCESS);

// Extract the Q from the ciphertext
std::vector<buf_t> Qs;
assert(coinbase::api::pve::get_public_keys_compressed_batch(ct, Qs) == SUCCESS);
std::vector<mem_t> Qs_mem;
for (const auto& q : Qs) Qs_mem.emplace_back(q.data(), q.size());

// Always fails — ecurve_t::check rejects infinity, or Q mismatch
error_t rv = coinbase::api::pve::verify_batch(curve_id::secp256k1, ek, ct, Qs_mem, label);
assert(rv != SUCCESS);  // encrypt said SUCCESS; verify always says FAIL
```
