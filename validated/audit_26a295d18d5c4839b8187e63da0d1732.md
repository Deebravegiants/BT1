### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash`, but never verifies that `response.payload_hash` is the correct hash for the `request` being resolved. A single Byzantine attested participant (below the signing threshold) can replay a legitimately-produced threshold signature from one foreign-tx request to satisfy a completely different pending request, causing the contract to attest to a foreign-chain transaction that was never verified.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 691–754), the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract only checks: *"is this signature valid over this hash?"* It does **not** check: *"is this hash the correct hash for the request being responded to?"*

The `payload_hash` is entirely caller-supplied. The `VerifyForeignTransactionRequest` used as the map key contains `(domain_id, payload_version, ForeignChainRpcRequest{chain, tx_id, …})`, but `payload_hash` is not part of that key and is never cross-checked against it. [2](#0-1) 

By contrast, the regular `respond` function binds the signature to the request by verifying it against the **derived key** (keyed on `request.tweak`, which encodes the caller and path):

```rust
let expected_public_key = derive_key_secp256k1(&affine, &request.tweak)?;
let payload_hash = request.payload.as_ecdsa()?;
verify_ecdsa_signature(sig, payload_hash, &expected_public_key).is_ok()
``` [3](#0-2) 

For foreign-tx, the node intentionally uses a **zero tweak** (root key), so the signature is not bound to any per-request derivation:

```rust
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),
    ...
})
``` [4](#0-3) 

The canonical payload that nodes sign is `SHA256(borsh(ForeignTxSignPayload{request, observed_values}))`, which does embed the request. But the contract never reconstructs or checks this — it accepts any `payload_hash` as long as the signature over it is valid. [5](#0-4) 

---

### Impact Explanation

A single Byzantine attested participant can:

1. Participate in the honest signing protocol for a legitimate `verify_foreign_transaction(request_A)` (e.g., a real Bitcoin tx), obtaining the threshold signature `{payload_hash_A, sig_A}`.
2. Submit `verify_foreign_transaction(request_B)` for a fraudulent or non-existent transaction.
3. Call `respond_verify_foreign_tx(request_B, {payload_hash_A, sig_A})`.
4. The contract verifies `sig_A` over `payload_hash_A` using the root key — this passes.
5. The contract resolves the yield for `request_B` with `{payload_hash_A, sig_A}`.
6. The caller of `request_B` receives a valid MPC root-key signature, but it attests to `request_A`'s data, not `request_B`'s.

In a bridge context (the primary use case per the design doc), the caller uses this signature to claim a foreign-chain transaction was verified. The contract has now issued a valid MPC attestation for a transaction that was never inspected. This enables invalid bridge execution or double-spend conditions.

**Impact class:** High — forged foreign-chain verification / cross-chain replay causing invalid bridge execution.

---

### Likelihood Explanation

- Requires a single Byzantine attested participant (below threshold) — within the system's Byzantine fault-tolerance model.
- The attacker does not need to forge a threshold signature; they only need to replay one that was legitimately produced for a different request.
- Any attested participant who was part of the signing round for `request_A` has the threshold signature and can call `respond_verify_foreign_tx` for any other pending request.
- The `respond_verify_foreign_tx` function has no rate-limiting or deduplication guard beyond the pending-request map lookup. [6](#0-5) 

**Likelihood:** Medium — requires a single Byzantine node within the existing threat model, no threshold collusion needed.

---

### Recommendation

The contract must verify that `response.payload_hash` is the correct hash for the `request` being resolved. The cleanest fix is to require the response to carry the full `ForeignTxSignPayload` (not just the hash), and have the contract:

1. Assert `payload.request == request` (the embedded request matches the pending-map key).
2. Recompute `expected_hash = SHA256(borsh(payload))`.
3. Assert `response.payload_hash == expected_hash`.
4. Verify the signature over `expected_hash`.

This mirrors how `respond` binds the signature to the request via the derived key, and closes the cross-request replay window entirely.

---

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(request_A = Bitcoin tx 0xAAAA…)
   → pending_verify_foreign_tx_requests[request_A] = [yield_A]

2. MPC network signs: payload_hash_A = SHA256(borsh({request_A, [BlockHash(0xBBBB…)]}))
   → threshold signature sig_A produced; all signing participants hold (payload_hash_A, sig_A)

3. Byzantine attested participant Eve submits:
   verify_foreign_transaction(request_B = Bitcoin tx 0xDEAD… [non-existent])
   → pending_verify_foreign_tx_requests[request_B] = [yield_B]

4. Eve calls respond_verify_foreign_tx(request_B, {payload_hash_A, sig_A})
   → Contract checks: verify_ecdsa(sig_A, payload_hash_A, root_key) == OK  ✓
   → Contract resolves yield_B with {payload_hash_A, sig_A}

5. Bob (caller of request_B) receives {payload_hash_A, sig_A}
   → Valid root-key signature, but attesting to Bitcoin tx 0xAAAA…, not 0xDEAD…
   → Bob's bridge contract accepts the signature as proof that 0xDEAD… was finalized
   → Invalid bridge execution / double-spend
``` [7](#0-6) [8](#0-7)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
