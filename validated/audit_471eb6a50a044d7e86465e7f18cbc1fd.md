Audit Report

## Title
Stale `rsETHPrice` Allows Discounted rsETH Minting When Checkpoint Reward Exceeds `pricePercentageLimit` — (`contracts/LRTOracle.sol`)

## Summary
When an EigenLayer checkpoint credits consensus rewards large enough to push `newRsETHPrice` above `highestRsethPrice * (1 + pricePercentageLimit)`, the public `updateRSETHPrice()` reverts for all non-manager callers, leaving `rsETHPrice` at its previous lower value. Because `LRTDepositPool.getRsETHAmountToMint()` divides directly by the stale `lrtOracle.rsETHPrice()` with no freshness check, any depositor can mint rsETH at the old lower price during this window, diluting existing holders' unclaimed yield.

## Finding Description
`_updateRsETHPrice()` computes `newRsETHPrice` and checks whether the price increase exceeds `pricePercentageLimit`. If it does and the caller is not a manager, the function reverts at line 264 with `PriceAboveDailyThreshold`:

```solidity
// LRTOracle.sol L260-265
if (isPriceIncreaseOffLimit) {
    if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
        revert PriceAboveDailyThreshold();
    }
}
```

The `rsETHPrice = newRsETHPrice` assignment at line 313 is never reached, so `rsETHPrice` remains at its previous (lower) value. [1](#0-0) [2](#0-1) 

The deposit pool mints rsETH using that stored stale price:

```solidity
// LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

There is no freshness check on `rsETHPrice` anywhere in `_beforeDeposit()` or `getRsETHAmountToMint()`. Critically, the price-increase branch does **not** pause the deposit pool — only the price-decrease branch does (lines 278–280). Deposits remain fully open during the stale-price window. [4](#0-3) [6](#0-5) 

An attacker calls `depositETH()` or `depositAsset()` during this window, receiving `D / P_stale` rsETH instead of the correct `D / P_actual`. Once the manager calls `updateRSETHPriceAsManager()` to update the price, the attacker's rsETH is redeemable for more ETH than was deposited. The surplus is extracted from existing holders, whose per-token ETH entitlement is diluted because the rsETH supply was inflated beyond what the deposited ETH warranted. [5](#0-4) 

## Impact Explanation
**High — Theft of unclaimed yield.** The consensus rewards credited by the EigenLayer checkpoint represent yield accrued to existing rsETH holders. By minting rsETH at the stale lower price, the attacker captures a portion of that yield. After the manager updates the price, the attacker's rsETH is worth more ETH than deposited; the difference is borne by pre-existing holders whose share of the protocol's ETH is permanently diluted. This matches the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation
`pricePercentageLimit` is documented at 1% (`1e16`). EigenLayer checkpoints accumulate consensus rewards over time; a checkpoint not finalized for several weeks can easily credit more than 1% of the current price in a single update. No special privilege is required — `depositETH()` and `depositAsset()` are public. The stale-price window persists across multiple blocks until the manager acts, giving the attacker ample time to exploit it without any front-running requirement.

## Recommendation
1. **Pause deposits on price-increase threshold breach**: mirror the price-decrease behavior — if `isPriceIncreaseOffLimit` is true and the caller is not a manager, also call `lrtDepositPool.pause()` before reverting, preventing deposits at the stale price.
2. **Enforce price freshness at deposit time**: record a `lastPriceUpdateTimestamp` in `_updateRsETHPrice()` and revert in `getRsETHAmountToMint()` (or `_beforeDeposit()`) if the price is older than a configurable staleness threshold.
3. **Raise `pricePercentageLimit` conservatively**: set it high enough that a single realistic checkpoint cannot exceed it, or implement an incremental catch-up mechanism.

## Proof of Concept
```solidity
// Foundry fork test — preconditions:
//   pricePercentageLimit = 1e16 (1%)
//   A large EigenLayer checkpoint has been verified, crediting >1% rewards

function testStaleDepositExploit() public {
    // 1. Confirm public updateRSETHPrice() reverts for non-manager
    vm.expectRevert(ILRTOracle.PriceAboveDailyThreshold.selector);
    lrtOracle.updateRSETHPrice();

    uint256 stalePriceBeforeUpdate = lrtOracle.rsETHPrice();

    // 2. Attacker deposits at stale (lower) price
    uint256 attackerDeposit = 10 ether;
    vm.deal(attacker, attackerDeposit);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: attackerDeposit}(0, "");

    uint256 attackerRsETH = rsETH.balanceOf(attacker);
    // attackerRsETH = attackerDeposit / stalePriceBeforeUpdate
    // which is MORE than attackerDeposit / truePrice

    // 3. Manager updates price
    vm.prank(manager);
    lrtOracle.updateRSETHPriceAsManager();

    uint256 updatedPrice = lrtOracle.rsETHPrice();
    assertGt(updatedPrice, stalePriceBeforeUpdate);

    // 4. Attacker's rsETH is now worth more than deposited — profit at existing holders' expense
    uint256 attackerETHValue = attackerRsETH * updatedPrice / 1e18;
    assertGt(attackerETHValue, attackerDeposit);
}
