Audit Report

## Title
`tokenFeeBps` Defaults to Zero for Newly Added Tokens; Protocol Collects No Fee on Token Deposits Until Admin Intervenes - (File: contracts/pools/RSETHPool.sol)

## Summary

`RSETHPool.sol` maintains two separate fee state variables: `feeBps` for ETH deposits and `tokenFeeBps[token]` for token deposits. When a token is added via `addSupportedToken`, `tokenFeeBps[token]` is never initialized and defaults to `0`. Any depositor can call `deposit(token, amount, referralId)` and receive rsETH on the full amount with zero fee deducted, until an admin explicitly calls `setTokenFeeBps`. The non-zero `feeBps` global rate is never consulted for token deposits.

## Finding Description

`RSETHPool.sol` declares two fee variables:

- `feeBps` at L43 — global fee rate, set at initialization, used exclusively for ETH deposits.
- `tokenFeeBps[token]` at L88 — per-token fee mapping, a Solidity mapping that defaults to `0` for any key never written.

The ETH deposit path calls `viewSwapRsETHAmountAndFee(amount)` (L271), which reads `feeBps` at L312:
```solidity
fee = amount * feeBps / 10_000;
```

The token deposit path at L298 calls `viewSwapRsETHAmountAndFee(amount, token)`, which reads only `tokenFeeBps[token]` at L335–336:
```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```

`addSupportedToken` (L637–656) sets `supportedTokenOracle[token]` and `tokenBridge[token]` but never writes `tokenFeeBps[token]`, leaving it at the Solidity default of `0`. The only way to set it is via `setTokenFeeBps` (L583–594), which requires `DEFAULT_ADMIN_ROLE` and is a separate, non-atomic call. There is no on-chain enforcement requiring it to be called before or alongside `addSupportedToken`.

Exploit path:
1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0`.
2. Any depositor calls `deposit(wstETH, 10 ether, "ref")`.
3. `viewSwapRsETHAmountAndFee(10 ether, wstETH)` computes `fee = 10 ether * 0 / 10_000 = 0`.
4. Depositor receives rsETH for the full `10 ether`; `feeEarnedInToken[wstETH]` stays `0`.
5. The non-zero `feeBps` is never consulted.

No existing check prevents this. The `onlySupportedToken` modifier only verifies the oracle is set, not that a fee has been configured.

## Impact Explanation

The protocol fails to collect fee revenue on token deposits for any token whose `tokenFeeBps` has not been explicitly set. `feeEarnedInToken[token]` remains `0`, and the intended fee (as expressed by the non-zero `feeBps`) is silently bypassed. No user principal is lost — depositors receive more rsETH than intended, and the protocol receives less fee revenue than intended. This matches the **Low** impact tier: *Contract fails to deliver promised returns, but doesn't lose value.*

## Likelihood Explanation

This is the default state for every token added via `addSupportedToken`. The zero fee is publicly observable on-chain via `tokenFeeBps(token)` and `viewSwapRsETHAmountAndFee`. Any depositor can exploit the window between `addSupportedToken` and `setTokenFeeBps` — a window that exists by design since the two calls are separate and non-atomic. No special capability or capital is required beyond holding the supported token.

## Recommendation

Initialize `tokenFeeBps[token]` to `feeBps` inside `addSupportedToken`, so newly added tokens inherit the global fee rate by default:

```solidity
function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
    // ... existing checks ...
    tokenFeeBps[token] = feeBps; // inherit global fee rate
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    emit AddSupportedToken(token, oracle, bridge);
}
```

Alternatively, add a `_feeBps` parameter to `addSupportedToken` to allow explicit per-token fee configuration at addition time.

## Proof of Concept

Foundry test sequence (no fork required):

```solidity
// 1. Deploy and initialize RSETHPool with feeBps = 100 (1%)
// 2. Call addSupportedToken(wstETH, oracle, bridge)
// 3. Assert tokenFeeBps[wstETH] == 0  ✓
// 4. Depositor calls deposit(wstETH, 10e18, "ref")
// 5. viewSwapRsETHAmountAndFee(10e18, wstETH) returns fee = 0
// 6. Assert feeEarnedInToken[wstETH] == 0  ✓
// 7. Assert depositor received rsETH for full 10e18 (no fee deducted)  ✓
// 8. Assert feeBps == 100 (non-zero, but never consulted)  ✓
```

The test confirms: `addSupportedToken` at L637–656 never writes `tokenFeeBps[token]`; `viewSwapRsETHAmountAndFee(amount, token)` at L335–336 reads the zero default; `deposit(token, amount, referralId)` at L284–305 accumulates zero fee.
