Audit Report

## Title
`recipient` Parameter Silently Ignored in `bridgeTokenToL1` with No Access Control Allows External Callers to Lose Tokens to L1Vault - (File: contracts/bridges/SonicChainNativeTokenBridge.sol)

## Summary
`SonicChainNativeTokenBridge.bridgeTokenToL1` is callable by any external account with no access control, yet the `recipient` parameter it accepts — and which `IL2TokenBridge` explicitly documents as the L1 destination — is never forwarded to the underlying `sonicBridge.withdraw()` call. Any external caller who invokes the function directly will have their tokens routed to `L1Vault` via `SonicBridgeReceiver`, not to their specified address, and will have no mechanism to recover them without admin intervention.

## Finding Description
`IL2TokenBridge` documents `bridgeTokenToL1` as: *"Initiates a withdrawal of a specified amount of tokens to the specified recipient on L1"* and *"The address of the recipient on L1."* [1](#0-0) 

The implementation carries no access control modifier, making it reachable by any external account: [2](#0-1) 

Despite validating `recipient` and including it in UID generation and event emission, the actual bridge call omits it entirely: [3](#0-2) 

`ISonicBridge.withdraw` takes no recipient parameter by design — the Sonic gateway's self-claiming model means only the contract that called `withdraw` on L2 (i.e., `SonicChainNativeTokenBridge`, whose address equals `SonicBridgeReceiver` on L1) can claim the withdrawal. `SonicBridgeReceiver.claimAndTransferToVault` then unconditionally forwards all claimed tokens to `l1Vault`: [4](#0-3) 

`L1Vault` has no recovery path for arbitrary depositors — its token-handling functions (`depositAssetForL1Vault`, `bridgeRsETHToL2`) are gated by `MANAGER_ROLE` and operate on the vault's aggregate balance, not on per-user claims: [5](#0-4) 

The intended caller is `RSETHPool.bridgeTokens`, which is itself gated by `BRIDGER_ROLE` and always passes `l1VaultETHForL2Chain` as the recipient — consistent with the protocol's design: [6](#0-5) 

The missing access control on `bridgeTokenToL1` is the root cause: it exposes a function whose architectural constraints (self-claiming bridge, fixed L1Vault destination) are incompatible with the `IL2TokenBridge` interface's promise of arbitrary recipient delivery.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

An external caller who reads `IL2TokenBridge` and calls `bridgeTokenToL1(theirL1Address, amount)` directly will: (1) have `amount` tokens pulled from their wallet, (2) see those tokens bridged to `SonicBridgeReceiver` and forwarded to `L1Vault`, and (3) receive nothing at `theirL1Address`. The tokens are not destroyed — they enter the protocol's `L1Vault` — but the caller has no claim on them and no recovery path without admin intervention. The function fails to deliver the promised return (tokens at `recipient`) while the protocol retains the value.

## Likelihood Explanation
The function is `external` with no access control, callable by any account that holds the bridged token and approves the contract. The `IL2TokenBridge` interface creates a strong and documented expectation that `recipient` is honored. Any integrator or user who relies on the interface specification rather than reading the implementation NatSpec will trigger this on every direct call. The condition is always met — there is no edge case or race condition required.

## Recommendation
1. **Add access control** — apply `onlyRole(BRIDGER_ROLE)` (or equivalent) to `bridgeTokenToL1` so only the authorized pool contract can invoke it, eliminating the direct-user attack path and aligning with how `RSETHPool.bridgeTokens` already enforces `BRIDGER_ROLE`.
2. **Reconcile the interface** — update `IL2TokenBridge` NatSpec to reflect that for Sonic, the `recipient` parameter is informational only and the actual L1 destination is always `L1Vault`, or provide a Sonic-specific interface that omits the misleading parameter.

## Proof of Concept
1. Alice holds Sonic-bridged rsETH on Sonic chain and approves `SonicChainNativeTokenBridge` for 100e18 tokens.
2. Alice reads `IL2TokenBridge` and calls `SonicChainNativeTokenBridge.bridgeTokenToL1(aliceL1Address, 100e18)` directly.
3. The contract validates `aliceL1Address` (non-zero), pulls 100e18 tokens from Alice, and calls `sonicBridge.withdraw(uid, originalToken, 100e18)` — `aliceL1Address` is never passed.
4. On L1, a `CLAIMER_ROLE` operator calls `SonicBridgeReceiver.claimAndTransferToVault(...)`, which transfers the 100e18 tokens to `l1Vault`.
5. Alice's `aliceL1Address` receives 0 tokens. Alice has no mechanism to recover her funds from `L1Vault` without admin intervention.
6. The emitted `TokensBridgedToL1(aliceL1Address, 100e18, uid, bridgeReceiver)` event falsely implies Alice's address was the recipient.
