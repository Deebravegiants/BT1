### Title
Single Participant Can Permanently Block All `verify_foreign_transaction` Requests via Overly-Restrictive Superset Condition — (File: `crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which uses a strict all-participant intersection rule (`is_superset`). A single active participant calling `register_foreign_chain_support` with an empty set causes the intersection to collapse to empty, permanently blocking every foreign-chain verification request until the participant re-registers. This is the direct analog of the ZeroLend `&&`/`||` inversion: an overly-restrictive boolean condition that should require only a threshold of participants but instead requires unanimity, letting one sub-threshold actor freeze the entire feature.

### Finding Description

`get_supported_foreign_chains()` computes the set of chains that every active participant has registered:

```rust
// crates/contract/src/lib.rs  lines 2203–2217
foreign_chain_to_node_mapping
    .into_iter()
    .filter_map(|(foreign_chain, nodes_supporting_chain)| {
        let all_active_nodes_supports_chain =
            nodes_supporting_chain.is_superset(&active_participant_account_ids);
        if all_active_nodes_supports_chain {
            Some(foreign_chain)
        } else {
            None
        }
    })
    ...
```

`verify_foreign_transaction` then gates every incoming request on this set:

```rust
// crates/contract/src/lib.rs  lines 533–542
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported { requested: requested_chain }
            .to_string(),
    );
}
```

`register_foreign_chain_support` is callable by any active participant with no lower bound on the set size:

```rust
// crates/contract/src/lib.rs  lines 972–983
pub fn register_foreign_chain_support(
    &mut self,
    foreign_chain_support: dtos::SupportedForeignChains,
) -> Result<(), Error> {
    let account_id = self.voter_or_panic();
    self.node_foreign_chain_support
        .foreign_chain_support_by_node
        .insert(account_id, foreign_chain_support);
    Ok(())
}
```

Because the superset check requires every participant's entry to contain the chain, a single participant submitting `SupportedForeignChains::default()` (empty) causes `get_supported_foreign_chains()` to return the empty set. Every subsequent `verify_foreign_transaction` call then panics with `ForeignChainNotSupported`, regardless of how many other participants correctly registered the chain.

The project's own design document explicitly acknowledges this as a live defect:

> "Today, `get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains … A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down."
> — `docs/design/calculating-supported-foreign-chains.md`, lines 9–13

The proposed fix (`get_available_foreign_chains()`, threshold-based) exists in the contract but `verify_foreign_transaction` has not been migrated to use it.

### Impact Explanation

Any single active participant — strictly below the signing threshold — can call `register_foreign_chain_support({})` and immediately cause every `verify_foreign_transaction` call to revert. Because the registration is idempotent and persists until overwritten, the disruption lasts until the participant re-registers or is removed via resharing. This breaks the production safety invariant that foreign-chain verification should remain available as long as ≥ signing-threshold participants cover the chain. Bridge inbound flows (e.g., Omnibridge) that depend on `verify_foreign_transaction` to release funds on NEAR would be frozen for all users for the duration of the attack.

This maps to the **Medium** allowed impact: *request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.*

### Likelihood Explanation

The attack requires only one participant account — well below the signing threshold — to submit a single, valid, zero-cost contract call. No key material, no collusion, no special tooling. The call is indistinguishable from a legitimate re-registration with an updated (empty) chain list. The window of disruption is indefinite and fully controlled by the attacker.

### Recommendation

Replace the `is_superset` (unanimity) check in `get_supported_foreign_chains()` with the threshold-based `get_available_foreign_chains()` already implemented in the contract, and update `verify_foreign_transaction` to gate on it:

```diff
-        let supported_chains = self.get_supported_foreign_chains();
+        let supported_chains = self.get_available_foreign_chains();
         if !supported_chains.contains(&requested_chain) {
```

This matches the design intent documented in `docs/design/calculating-supported-foreign-chains.md`: a chain is available iff ≥ `signing_threshold` active participants cover it, so no single sub-threshold participant can remove it.

### Proof of Concept

```
Setup: 4 participants, signing threshold 3, Bitcoin whitelisted.

1. Participants 0–3 each call register_foreign_chain_support({Bitcoin}).
   → get_supported_foreign_chains() == {Bitcoin}
   → verify_foreign_transaction(Bitcoin, ...) succeeds.

2. Participant 0 calls register_foreign_chain_support({})   // empty set
   → node_foreign_chain_support[participant_0] = {}
   → get_supported_foreign_chains():
       nodes_supporting_bitcoin = {1, 2, 3}
       active_participants      = {0, 1, 2, 3}
       {1,2,3}.is_superset({0,1,2,3}) == false
   → returns {}

3. Any user calls verify_foreign_transaction(Bitcoin, ...).
   → panics: "ForeignChainNotSupported { requested: Bitcoin }"

4. Participants 1–3 (threshold = 3) cannot override participant 0's
   registration; they can only update their own entries.
   The chain remains blocked until participant 0 re-registers or is
   removed via a full resharing.
```

This is confirmed by the existing unit test `get_supported_foreign_chains__should_return_empty_when_no_votes` (line 7167) and the sandbox test `register_foreign_chain_config__returns_empty_when_not_all_registered` (line 205), both of which demonstrate that a single missing or empty registration drops the entire supported-chain set.     pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
```rust
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }
```
