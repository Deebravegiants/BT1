Audit Report

## Title
Missing Chainlink `latestRoundData()` Staleness and Validity Checks in `getAssetPrice()` — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards all four validity/staleness return values, using only the raw `answer`. This allows a stale, zero, or negative price to flow unchecked into `LRTDepositPool.getRsETHAmountToMint()`, enabling an unprivileged depositor to mint excess rsETH during any period of Chainlink feed staleness, or to trigger catastrophic over-minting if the feed returns a negative value.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at line 52 discards all validity indicators:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is performed for:
- **Stale round**: `answeredInRound < roundId` — answer is from a prior round
- **Incomplete round**: `updatedAt == 0` — round has not completed
- **Invalid price**: `price <= 0` — feed returned zero or negative

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository correctly validates all three conditions before using the price (lines 30–32).

The stale price propagates through `LRTOracle.getAssetPrice()` (line 157) into `LRTDepositPool.getRsETHAmountToMint()` (line 520):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` is a stored value updated separately by `_updateRsETHPrice()`. During a staleness window, `getAssetPrice(asset)` returns the inflated stale price while `rsETHPrice` reflects the last correctly-computed exchange rate, causing the numerator to be inflated and excess rsETH to be minted.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (lines 256–265) does not protect individual deposits — it only applies when the price update function is called by an operator, not during the deposit flow.

For a negative `price`, `uint256(negativeValue)` wraps to near-`type(uint256).max`, causing catastrophic over-minting in a single deposit call.

## Impact Explanation

**Critical — Direct theft of user funds / Protocol insolvency.**

- **Stale inflated price**: Attacker deposits LST during staleness window, receives excess rsETH proportional to the price inflation, then redeems for more ETH than deposited, extracting value from honest depositors.
- **Negative price**: A single deposit call wraps the negative value to `type(uint256).max`, minting an astronomically large rsETH amount, immediately rendering the protocol insolvent.
- **Zero price**: `rsethAmountToMint = 0`, silently minting nothing — permanent fund freeze for the depositor (deposited tokens transferred in, zero rsETH minted, no revert).

## Likelihood Explanation

Chainlink feeds go stale during network congestion, sequencer downtime (L2), or oracle node failures. The exploit requires no privileges — any caller of `depositAsset()` or `depositETH()` can trigger it. The vulnerability is active for the entire duration of any staleness period. The same codebase already demonstrates awareness of this risk by implementing the checks in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the omission in `ChainlinkPriceOracle` is an oversight, not a design choice.

## Recommendation

Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

**Stale price theft (Foundry fork test):**

1. Fork mainnet at a block where the `stETH/ETH` Chainlink feed is live.
2. Warp time forward past the feed's heartbeat interval without a new round being posted (simulating staleness).
3. Call `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")` as an unprivileged attacker.
4. `getRsETHAmountToMint` computes `1000e18 * stalePriceHigh / storedRsETHPrice`, minting excess rsETH.
5. Assert `rsethMinted > 1000e18 * truePrice / storedRsETHPrice` — the excess represents stolen value.
6. Confirm `ChainlinkPriceOracle.getAssetPrice()` never reverts despite the stale round.

**Negative price catastrophe (unit test):**

1. Deploy a mock `AggregatorV3Interface` that returns `answer = -1` from `latestRoundData()`.
2. Register it as the price feed for a supported asset.
3. Call `depositAsset()` — `uint256(-1)` = `type(uint256).max`, minting `~1.16e77` rsETH.
4. Assert protocol rsETH supply is now `type(uint256).max`, rendering all existing rsETH worthless.
