Audit Report

## Title
Zero rsETH/wrsETH Minted on Dust Deposits While User's ETH/Tokens Are Consumed - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The `deposit` functions across all L2 pool variants (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPool`, `RSETHPoolNoWrapper`) and the L1 `LRTDepositPool._beforeDeposit` path perform no check that the computed output amount is greater than zero before consuming the caller's ETH or ERC-20 tokens. When a deposit is small enough that integer division truncates `rsETHAmount` to zero, the user's funds are permanently absorbed by the pool while they receive nothing in return.

## Finding Description

**Root cause — `viewSwapRsETHAmountAndFee` (ETH path):**

`RSETHPoolV3.sol` lines 299–308 compute:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

When `amountAfterFee * 1e18 < rsETHToETHrate` (e.g., a 1-wei deposit when `rsETHToETHrate ≈ 1.05e18`), `rsETHAmount` truncates to 0. [1](#0-0) 

**Root cause — `viewSwapRsETHAmountAndFee` (token path):**

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

Same truncation applies when the token amount is small relative to the rate ratio. [2](#0-1) 

**Vulnerable call site — ETH deposit (`RSETHPoolV3`):**

The only guard is `if (amount == 0) revert InvalidAmount()`. There is no check that `rsETHAmount > 0` before calling `wrsETH.mint(msg.sender, rsETHAmount)`, which mints 0 tokens. [3](#0-2) 

**`limitDailyMint` modifier does not protect against this:**

The modifier adds `rsETHAmount` (which is 0) to `dailyMintAmount` and the limit check `dailyMintAmount + 0 > dailyMintLimit` passes without reverting. [4](#0-3) 

**Same pattern confirmed in all pool variants:**

- `RSETHPoolV3ExternalBridge.deposit` — mints 0 wrsETH after consuming ETH. [5](#0-4) 

- `RSETHPool.deposit` — calls `safeTransfer(msg.sender, 0)` (succeeds silently) after consuming ETH. [6](#0-5) 

- `RSETHPoolNoWrapper.deposit` — calls `rsETH.safeTransfer(msg.sender, 0)` after consuming ETH. [7](#0-6) 

**L1 analog — `LRTDepositPool._beforeDeposit`:**

`getRsETHAmountToMint` can return 0 for dust amounts. The only slippage guard is `if (rsethAmountToMint < minRSETHAmountExpected)`, which does not revert when both values are 0 (the default for many integrations). [8](#0-7) [9](#0-8) 

## Impact Explanation

A depositor who sends a dust amount of ETH or tokens receives zero rsETH/wrsETH while their input is permanently absorbed into the pool's balance (and subsequently bridged to L1 as protocol TVL). The depositor's funds are not recoverable. This matches the **Low** impact tier: *"Contract fails to deliver promised returns, but doesn't lose value"* — the protocol retains the absorbed ETH/tokens; the individual user loses their dust deposit with no recourse.

## Likelihood Explanation

Any unprivileged external caller can trigger this by calling `deposit{value: 1}("")` (or any amount below the truncation threshold) on any of the affected pool contracts while they are unpaused and deposits are enabled. No special role, precondition, or victim mistake is required. The truncation threshold grows as the rsETH/ETH rate appreciates over time, making the affected range larger as the protocol matures. The condition is fully deterministic and repeatable.

## Recommendation

Add an explicit zero-output guard immediately after computing `rsETHAmount` in each `deposit` function:

```solidity
// RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

For the L1 path, add a zero-output guard in `_beforeDeposit`:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint == 0) revert InvalidAmountToDeposit();
if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
```

## Proof of Concept

Assume `RSETHPoolV3` is deployed with `feeBps = 0` and the oracle returns `rsETHToETHrate = 1.05e18`.

1. Caller invokes `deposit{value: 1}("")`.
2. `viewSwapRsETHAmountAndFee(1)`:
   - `fee = 1 * 0 / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation)
3. `limitDailyMint`: `dailyMintAmount += 0` — no revert.
4. `wrsETH.mint(msg.sender, 0)` — 0 wrsETH minted.
5. The 1 wei of ETH remains in the pool and will be bridged to L1 as protocol TVL.
6. Caller holds 0 wrsETH and has permanently lost 1 wei.

**Foundry fuzz test plan:**

```solidity
function testFuzz_dustDepositMintsZero(uint256 amount) public {
    vm.assume(amount > 0 && amount < rsETHToETHrate / 1e18); // dust range
    uint256 balanceBefore = wrsETH.balanceOf(user);
    vm.deal(user, amount);
    vm.prank(user);
    pool.deposit{value: amount}("");
    assertEq(wrsETH.balanceOf(user), balanceBefore); // received 0
    assertEq(address(pool).balance, amount);          // ETH absorbed
}
```
