### Title
Multiple Sub-Transactions Can Claim the Same Deposit Refund, Creating ADA Out of Thin Air - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the `getConsumedDijkstraValue` function aggregates consumed values (including deposit refunds from `UnRegDepositTxCert` certificates) across all sub-transactions by summing them independently against the **original** `certState`. Multiple sub-transactions can each include `UnRegDepositTxCert` for the **same** staking credential, causing the deposit refund to be counted multiple times in the consumed-value calculation. No batch-level deduplication check exists for deposit refunds across sub-transactions, unlike the analogous `validateBatchWithdrawals` guard that exists for withdrawals. The developers have explicitly acknowledged this gap with a disabled test marked `xit "Multiple subtransactions cannot get the same refund"` and `error "TODO: predicate failure not yet implemented"`.

---

### Finding Description

The Dijkstra era introduces nested ("sub") transactions. The top-level LEDGER rule processes sub-transactions via `SUBLEDGERS` before running the top-level `UTXOW`/`UTXO` checks.

**Value conservation check** in `dijkstraUtxoTransition`:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [1](#0-0) 

This calls `getConsumedDijkstraValue`, which sums consumed values from the top-level body **and every sub-transaction body**:

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
``` [2](#0-1) 

The `lookupStakingDeposit` function is derived from the **original** `certState` (before any sub-transactions execute). If two sub-transactions both contain `UnRegDepositTxCert stakingCred keyDeposit`, `lookupStakingDeposit stakingCred` returns `Just keyDeposit` for **both**, so the deposit refund is counted **twice** in the consumed total.

By contrast, `validateBatchWithdrawals` correctly guards against over-withdrawal across sub-transactions:

```haskell
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals = Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance then Just ... else Nothing
          ) allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
``` [3](#0-2) 

No equivalent guard exists for deposit refunds across sub-transactions.

The developers have explicitly acknowledged this gap:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = mkBasicTx mkBasicTxBody
    & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)

  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [4](#0-3) 

The `xit` prefix disables the test; the `error "TODO: predicate failure not yet implemented"` confirms no rejection predicate has been implemented for this case.

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

An attacker registers a staking credential (paying `keyDeposit`), then submits a batch transaction with two sub-transactions each containing `UnRegDepositTxCert stakingCred keyDeposit`. The consumed-value calculation counts `keyDeposit` twice. The produced side (outputs + fees) can include `2 × keyDeposit` in outputs, but only `keyDeposit` was ever in the deposit pool. The ledger accepts the transaction, and `keyDeposit` ADA is created from nothing. This is a direct violation of ADA supply conservation.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Registering a staking credential (standard operation, costs `keyDeposit`)
2. Constructing a batch transaction with two sub-transactions each containing `UnRegDepositTxCert` for the same credential

No privileged access, governance majority, or leaked keys are required. The attacker is an ordinary transaction sender. The Dijkstra era is experimental/upcoming, but the vulnerability exists in the current codebase and the developers have already identified the missing check.

---

### Recommendation

Implement a batch-level check analogous to `validateBatchWithdrawals` that aggregates all `UnRegDepositTxCert` deposit refunds across the top-level transaction and all sub-transactions, and verifies that no staking credential's deposit is claimed more than once. Specifically:

- In `dijkstraUtxoTransition`, add a `validateBatchDepositRefunds` check that collects all `UnRegDepositTxCert` credentials across all sub-transaction bodies and rejects the batch if any credential appears more than once.
- Alternatively, fix `getConsumedDijkstraValue` to deduplicate deposit refunds across sub-transactions by tracking which credentials have already been counted.

The existing `validateBatchWithdrawals` pattern in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` is the correct model to follow.

---

### Proof of Concept

1. Register staking credential `C`, paying deposit `D` (e.g., 2 ADA).
2. Construct sub-transaction `subTx1` with body containing `UnRegDepositTxCert C D`.
3. Construct sub-transaction `subTx2` with body containing `UnRegDepositTxCert C D`.
4. Construct top-level transaction `tx` with `subTransactionsTxBodyL = [subTx1, subTx2]` and outputs totalling `2D` more than the UTxO inputs.
5. Submit `tx`.

`getConsumedDijkstraValue` counts `D` from `subTx1`'s body and `D` from `subTx2`'s body (both using the original `certState` where `C` is registered), yielding consumed = inputs + `2D`. The produced side includes outputs = inputs + `2D`. Value conservation passes. Only `D` was in the deposit pool; `D` ADA has been created from nothing. [5](#0-4) [4](#0-3)

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL
