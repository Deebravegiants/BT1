### Summary

An unprivileged transaction sender can craft a `DirectDeposits` entry whose `Coin` amount, when added to a target account's existing balance via `addCompactCoin`, silently wraps the underlying `Word64` representation. The root cause is two independent gaps: (1) `addCompactCoin` performs raw modular `Word64` addition with no overflow guard, and (2) the `DirectDeposits` field is entirely absent from both sides of the `validateValueNotConservedUTxO` check, so the deposit amount is unconstrained by the transaction's inputs. Together these allow an attacker to set any registered account's balance to an arbitrary small value, destroying the victim's ADA.

### Finding Description

**`addCompactCoin` — no overflow protection** [1](#0-0) 

`CompactForm Coin` wraps `Word64`. Haskell's `Word64` arithmetic is modular: `(maxBound :: Word64) + 1 = 0`. There is no saturation, no `Maybe`, no exception.

**`applyDirectDeposits` calls `addCompactCoin` directly** [2](#0-1) 

The lambda `(\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))` is passed straight to `updateAccountBalances`, which converts the `Coin` via `compactCoinOrError` and then calls the lambda with no bounds check. [3](#0-2) 

**`directDepositsMissingAccounts` checks only existence** [4](#0-3) 

The only predicate is `isAccountRegistered`. No balance-bound check exists.

**`dijkstraEntitiesTransition` — the call sequence** [5](#0-4) 

After the existence check passes, `applyDirectDeposits` is applied unconditionally.

**CBOR encoding caps the deposit at `maxBound :: Word64`**

`Coin` is decoded as `Coin . toInteger <$> Plain.decodeWord64`, so the maximum encodable deposit is `2^64 − 1` lovelace — well above the total ADA supply (~4.5 × 10^16 lovelace) but within the range accepted by `compactCoinOrError`.

### Impact Explanation

Let `B` be the target account's balance in lovelace (any positive value ≤ total ADA supply ≈ 4.5 × 10^16). The attacker sets the deposit to `D = (2^64 − B + k)` for any small `k ≥ 1`. Since `B` << `2^64`, `D` is a valid `Word64` value; `compactCoinOrError D` succeeds. Then:

```
addCompactCoin (CompactCoin D) (CompactCoin B)
  = CompactCoin ((D + B) mod 2^64)
  = CompactCoin (k − 1)
```

The account balance is silently set to `k − 1`, destroying `B − k + 1` lovelace. With `k = 1` the balance becomes 0. The ADA is not moved to any other pot; it simply vanishes from the ledger state. This is **direct destruction of ADA through an invalid ledger state transition** — Critical impact.

---

### Likelihood Explanation

- The attacker is fully unprivileged: only a standard transaction submission is required.
- The target account's balance is publicly observable on-chain.
- The deposit amount is a simple arithmetic calculation.
- No script execution, governance action, or privileged key is needed.
- The value-conservation check passes because `DirectDeposits` is excluded from both sides.
- The only guard (`directDepositsMissingAccounts`) passes as long as the account exists.

Likelihood is **high** for any account that has registered in the Dijkstra era.

---

### Recommendation

1. **Include `DirectDeposits` in the produced-value calculation** inside `getProducedDijkstraValue` / `dijkstraProducedValue`, so the total deposit amount must be funded by the transaction's inputs. This is the primary fix and closes the ADA-creation vector simultaneously.

2. **Add overflow protection to `addCompactCoin`** (or introduce a checked variant used by `applyDirectDeposits`). The simplest safe form is:
   ```haskell
   addCompactCoinChecked :: CompactForm Coin -> CompactForm Coin -> Maybe (CompactForm Coin)
   addCompactCoinChecked (CompactCoin x) (CompactCoin y)
     | x <= maxBound - y = Just (CompactCoin (x + y))
     | otherwise         = Nothing
   ```
   `applyDirectDeposits` should fail the transition (not silently wrap) if the check returns `Nothing`.

3. **Add a predicate failure** in `dijkstraEntitiesTransition` for deposits that would overflow the target balance, analogous to `WithdrawalAmountsExceedAccountBalances`.

  EraAccounts era =>
  DirectDeposits ->
  Accounts era ->

directDepositsMissingAccounts (DirectDeposits dds) accounts
  | Map.foldrWithKey' checkRegistered True dds = Nothing
  | otherwise = Just $ DirectDeposits $ Map.foldrWithKey' collectMissing Map.empty dds
  where
    isRegistered (AccountAddress _ (AccountId credential)) =
      isAccountRegistered credential accounts
    checkRegistered addr _ acc = acc && isRegistered addr
    collectMissing addr amount acc
      | isRegistered addr = acc
      | otherwise = Map.insert addr amount acc
