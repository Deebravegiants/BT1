### Title
Bootstrap-Phase DRep Re-Delegation Leaves Stale Reverse-Delegation Entries, Enabling Governance Voting Power Nullification via DRep Unregistration — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version < 10, i.e., current mainnet PV9), `processDelegationInternal` intentionally skips cleanup of the old DRep's reverse-delegation set (`drepDelegs`) when a staker re-delegates to a new credential DRep. When the old DRep later unregisters via `ConwayUnRegDRep`, the `clearDRepDelegations` helper iterates over the stale `drepDelegs` entries and unconditionally sets `dRepDelegationAccountStateL .~ Nothing` for every listed staker — including those who have already moved to a different DRep. This silently erases the staker's current delegation to the new DRep, permanently removing their stake from that DRep's voting-power distribution for the remainder of the epoch.

---

### Finding Description

**Root cause — `processDelegationInternal` skips old-DRep cleanup in bootstrap phase**

In `conwayDelegTransition`, the `ConwayDelegCert` branch calls `processDelegationInternal` with `preserveIncorrectDelegation = (pvMajor pv < natVersion @10)`: [1](#0-0) 

Inside `processDelegationInternal`, when `preserveIncorrectDelegation` is `True` and the new DRep is a credential DRep, the branch that removes the staker from the old DRep's `drepDelegs` is bypassed: [2](#0-1) 

The result is a split state:
- `accountState.dRepDelegationAccountStateL = Just (DRepCredential drepCred_B)` — forward pointer is correct.
- `drepState[drepCred_A].drepDelegs` still contains `stakeCred` — stale reverse pointer.
- `drepState[drepCred_B].drepDelegs` also contains `stakeCred` — correct reverse pointer.

**Propagation — `clearDRepDelegations` does not guard against stale entries**

When DRep A unregisters (`ConwayUnRegDRep`), the ledger clears delegations for every credential in the DRep's `drepDelegs` set without checking whether those credentials still point to the unregistering DRep: [3](#0-2) 

Because `drepDelegs dRepState` still contains `stakeCred` (stale entry), `Map.adjust (dRepDelegationAccountStateL .~ Nothing)` is applied to the staker's account, overwriting `Just (DRepCredential drepCred_B)` with `Nothing`.

**Voting-power impact — `computeDRepDistr` reads from `dRepDelegationAccountStateL`**

The DRep stake distribution used for ratification is built from each staker's `dRepDelegationAccountStateL`: [4](#0-3) 

Once the staker's field is `Nothing`, their stake is excluded from every DRep's distribution. DRep B loses the staker's contribution with no on-chain signal.

**Test confirms the behavior**

The existing test suite explicitly asserts this outcome in bootstrap mode: [5](#0-4) 

The comment "we cannot `expectNotDelegatedVote` because the delegation is still in the DRepState of the other drep" confirms the inconsistency: the staker's account shows `Nothing` while DRep B's `drepDelegs` still lists them.

---
