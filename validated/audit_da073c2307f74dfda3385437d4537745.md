### Title
ETH Deposit Limit Guard Omits the Deposit Amount, Allowing Unlimited Over-Deposit — (File: `contracts/LRTDepositPool.sol`)

---

### Summary

`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool` applies a different, weaker guard for ETH than for ERC-20 assets. The ETH branch omits the incoming `amount` from the comparison, so the check only detects a limit that is *already* exceeded — it never prevents a deposit that *would* exceed the limit. Any unprivileged depositor can push the protocol's ETH holdings arbitrarily beyond the configured `depositLimitByAsset`.

---

### Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` is the sole enforcement point for the per-asset deposit cap:

```solidity
// contracts/LRTDepositPool.sol  lines 676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount is ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for ERC-20
}
``` [1](#0-0) 

For every ERC-20 asset the guard correctly evaluates `totalAssetDeposits + amount > limit`. For ETH it evaluates only `totalAssetDeposits > limit`, discarding the `amount` argument entirely. This is structurally identical to the reported bug: a guard that is supposed to block an operation is rendered permanently ineffective because a critical parameter is never incorporated into the check.

The guard is called unconditionally from `_beforeDeposit`, which is the only pre-condition checked before minting rsETH:

```solidity
// contracts/LRTDepositPool.sol  lines 661-663
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [2](#0-1) 

`depositETH` passes `msg.value` as `depositAmount`, but that value is silently dropped inside the ETH branch:

```solidity
// contracts/LRTDepositPool.sol  lines 76-93
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    emit ETHDeposit(msg.sender, msg.value, rsethAmountToMint, referralId);
}
``` [3](#0-2) 

The deposit limit is stored in `LRTConfig.depositLimitByAsset` and is the primary risk-management knob for ETH exposure:

```solidity
// contracts/LRTConfig.sol  line 23
mapping(address token => uint256 amount) public depositLimitByAsset;
``` [4](#0-3) 

---

### Impact Explanation

The ETH deposit cap is the protocol's primary mechanism for bounding its EigenLayer exposure. With the guard broken, any depositor can push `totalAssetDeposits(ETH)` to an arbitrary multiple of the configured limit in a single transaction. The protocol will mint rsETH for every wei deposited, diluting the backing ratio for all existing holders and causing the contract to fail to deliver the risk-bounded returns it promises. This maps to **Low — Contract fails to deliver promised returns**, with a secondary path to **Medium — Temporary freezing of funds** if the excess ETH cannot be deployed into EigenLayer strategies (e.g., strategy cap reached), leaving it idle in the deposit pool earning no yield.

---

### Likelihood Explanation

The entry point is `depositETH`, a public payable function with no role restriction. Any wallet can call it at any time. No special conditions, front-running, or privileged access are required. The only prerequisite is that `totalAssetDeposits < depositLimitByAsset` (the normal operating state). Likelihood is **High**.

---

### Recommendation

Apply the same `+ amount` pattern used for ERC-20 assets to the ETH branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    // Unified check: include the incoming amount for both ETH and ERC-20
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This aligns the ETH guard with the ERC-20 guard and with the view helper `getAssetCurrentLimit`, which already computes the remaining headroom correctly as `depositLimit - totalAssetDeposits`. [5](#0-4)

### Proof of Concept

```
Setup:
  depositLimitByAsset[ETH_TOKEN] = 100 ether
  getTotalAssetDeposits(ETH_TOKEN) = 99 ether   // one wei below the cap

Attack:
  attacker calls depositETH{value: 50 ether}()

Execution of _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50 ether):
  totalAssetDeposits = 99 ether
  ETH branch: return (99 ether > 100 ether)  →  false   ← amount=50 ether is never used
  → revert MaximumDepositLimitReached() is NOT triggered

Result:
  50 ether deposited, rsETH minted
  getTotalAssetDeposits(ETH_TOKEN) = 149 ether  (49 ether over the cap)

Contrast with ERC-20 (e.g. stETH) under identical conditions:
  return (99 ether + 50 ether > 100 ether)  →  true  →  revert  ✓
```
