### Title
Single Attested Participant Can Submit Unverified CKD Response, Bypassing Threshold Authorization — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract resolves a pending Confidential Key Derivation (CKD) request when called by **any single attested participant**. For requests using the `AppPublicKey` (non-publicly-verifiable) mode, the contract performs **zero cryptographic verification** of the submitted `CKDResponse`. A single Byzantine participant — strictly below the signing threshold — can submit an arbitrary, attacker-controlled `(big_y, big_c)` pair and the contract will accept it, resolving the user's pending request with a forged derived key.

This is the direct analog to Compound's `approveThis` pattern: just as the Compound governor can unilaterally approve arbitrary asset transfers without threshold consensus, a single MPC participant can unilaterally resolve a CKD request with a forged output, bypassing the threshold-signature requirement that is the entire security foundation of the MPC network.

---

### Finding Description

In `respond` (for ECDSA/EdDSA signatures), the contract enforces the threshold requirement **implicitly** by verifying the submitted signature against the MPC network's public key: [1](#0-0) 

A single participant cannot forge a valid threshold signature, so the cryptographic check acts as an on-chain threshold enforcement gate.

In `respond_ckd`, the analogous gate is **absent for `AppPublicKey` mode**: [2](#0-1) 

The branch for `AppPublicKey` is an empty no-op (`{}`), while only `AppPublicKeyPV` (publicly verifiable) mode performs `ckd_output_check`. After this match, `resolve_yields_for` is called unconditionally, delivering whatever `CKDResponse` the caller supplied to the waiting user promise.

The access guard only requires the caller to be an attested participant: [3](#0-2) 

There is no threshold vote, no quorum check, and no cryptographic proof that the submitted `(big_y, big_c)` values were produced by the threshold of participants. A single malicious participant satisfies all preconditions.

---

### Impact Explanation

**Critical — Unauthorized confidential key derivation output without the required participant authorization.**

A malicious attested participant calls `respond_ckd` with a pending `CKDRequest` (submitted by any user via `request_app_private_key`) and supplies attacker-chosen `CKDResponse { big_y, big_c }` values. The contract resolves the user's yield with this forged response. The user's application receives a derived key that the attacker controls or knows, enabling:

- Decryption of any secrets the user encrypts to the forged derived key.
- Impersonation of the user's application-specific identity derived from the MPC network's master key.

The threshold-signature guarantee — the core security property of the MPC network — is completely bypassed for CKD in `AppPublicKey` mode.

---

### Likelihood Explanation

**Medium-High.** The attacker must be a registered, TEE-attested participant in the MPC network. This is a Byzantine participant strictly below the signing threshold (one node suffices). The MPC network is designed to tolerate up to `t-1` malicious participants; this vulnerability allows a single such participant to unilaterally corrupt all CKD responses in `AppPublicKey` mode without any off-chain collusion. The attack is silent — the contract emits no error, and the user has no on-chain way to detect the forgery.

---

### Recommendation

Apply the same implicit threshold enforcement used in `respond`: verify the CKD output cryptographically on-chain before resolving the yield. For `AppPublicKey` mode, either:

1. **Require `AppPublicKeyPV` mode** for all on-chain-resolved CKD requests, making `ckd_output_check` mandatory; or
2. **Add a BLS proof of correct derivation** for `AppPublicKey` mode that the contract can verify against the stored master public key, analogous to how ECDSA signature verification enforces the threshold in `respond`.

Document clearly that `AppPublicKey` mode currently provides no on-chain threshold enforcement and that users relying on it trust a single participant.

---

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key: AppPublicKey(pk)`, depositing 1 yoctoNEAR. A `CKDRequest` is stored and a yield is registered.
2. Malicious attested participant (one node, below threshold) calls:
   ```
   respond_ckd(
     request = <the pending CKDRequest>,
     response = CKDResponse { big_y: attacker_chosen_point, big_c: attacker_chosen_point }
   )
   ```
3. `assert_caller_is_attested_participant_and_protocol_active()` passes (one attested participant suffices).
4. The `AppPublicKey` branch executes `{}` — no verification.
5. `resolve_yields_for` delivers the forged `CKDResponse` to the user's pending promise.
6. The user's application receives `(big_y, big_c)` chosen by the attacker. The attacker knows the corresponding secret and can decrypt anything the user encrypts to it. [4](#0-3)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
