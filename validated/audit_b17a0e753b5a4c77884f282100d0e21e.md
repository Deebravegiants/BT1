### Title
Forged Foreign-Chain Verification via Cross-Request Signature Replay in `respond_verify_foreign_tx` — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function in the MPC contract verifies that a submitted signature is cryptographically valid over a caller-supplied `payload_hash`, but **never checks that `payload_hash` actually corresponds to the stored pending request's foreign-chain transaction data**. A single malicious attested participant (below the signing threshold) can replay any previously observed legitimate `ForeignTxSignPayload` signature as a response to a completely different pending request, causing the contract to emit a forged foreign-chain verification attestation.

---

### Finding Description

When a user calls `verify_foreign_transaction`, the contract stores a `VerifyForeignTransactionRequest` (containing `tx_id`, `extractors`, `domain_id`, `payload_version`) in `pending_verify_foreign_tx_requests` and issues a yield.

MPC nodes are expected to:
1. Fetch the foreign-chain transaction data
2. Compute `ForeignTxSignPayload::V1 { request, values }` from the observed data
3. Sign `SHA-256(borsh(ForeignTxSignPayload))` to produce `payload_hash` and `signature`
4. Call `respond_verify_foreign_tx(request, VerifyForeignTransactionResponse { payload_hash, signature })`

The contract's `respond_verify_foreign_tx` then performs:

```rust
// crates/contract/src/lib.rs:726-733
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

It verifies only that `signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. It does **not** recompute or validate that `payload_hash` is the correct hash of the foreign-chain data for the stored `request`. After this check passes, it unconditionally resolves the pending yield:

```rust
// crates/contract/src/lib.rs:749-753
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

The signed payload type itself also contains no NEAR contract address, no NEAR chain ID, and no domain ID:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:1499-1502
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

And the hash is computed as:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:1505-1509
pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
    let mut hasher = sha2::Sha256::new();
    borsh::BorshSerialize::serialize(self, &mut hasher)?;
    Ok(Hash256(hasher.finalize().into()))
}
```

Because `respond_verify_foreign_tx` arguments are public NEAR transactions, any observer can extract a previously emitted `(payload_hash_A, signature_A)` pair from chain history. A single malicious attested participant can then submit that pair as the response to any different pending request B, and the contract will accept it — resolving request B with a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to transaction A, not transaction B.

---

### Impact Explanation

**High — Forged foreign-chain verification / invalid bridge execution.**

The `verify_foreign_transaction` flow is explicitly designed to let bridge contracts trust the MPC network's attestation that a specific foreign-chain transaction occurred and had specific extracted values (e.g., block hash, log data). A forged response delivers a `VerifyForeignTransactionResponse` whose `payload_hash` and `signature` are cryptographically valid but correspond to a *different* transaction than the one the caller requested. Any downstream bridge contract that does not independently recompute and compare `payload_hash` against the expected transaction data will accept this as proof of a transaction that was never actually verified, enabling invalid fund releases or double-spend conditions.

---

### Likelihood Explanation

**Medium.**

The attack requires only a **single** malicious attested participant — well below the signing threshold. All `respond_verify_foreign_tx` calls are public NEAR transactions, so any attested participant can harvest `(payload_hash, signature)` pairs from chain history without any additional privilege. The attacker must wait for a suitable pending request to target, but the contract's fan-out queuing design (multiple callers can queue under the same request key) means there will frequently be pending requests available. No threshold collusion, key leakage, or TEE break is required.

---

### Recommendation

1. **Bind `payload_hash` to the stored request**: In `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the stored `VerifyForeignTransactionRequest` and the submitted `response.payload_hash`, or require nodes to submit the full `ForeignTxSignPayload` so the contract can verify `payload_hash == SHA-256(borsh(payload))` and that `payload.request == stored_request.request`.

2. **Include context in the signed payload**: Add the NEAR contract address (`env::current_account_id()`), domain ID, and a request nonce/receipt ID to `ForeignTxSignPayloadV1` so that signatures are bound to a specific contract instance, domain, and request. This prevents cross-deployment and cross-request replay at the cryptographic level.

---

### Proof of Concept

**Setup**: Two pending requests exist for different Bitcoin transactions:
- Request A: `tx_id = [0xAA; 32]`, pending in `pending_verify_foreign_tx_requests`
- Request B: `tx_id = [0xBB; 32]`, pending in `pending_verify_foreign_tx_requests`

**Step 1**: The MPC network legitimately processes request A. An honest node calls:
```
respond_verify_foreign_tx(
  request = { tx_id: [0xAA;32], domain_id: 0, ... },
  response = { payload_hash: H_A, signature: SIG_A }
)
```
This is a public NEAR transaction. The malicious participant records `(H_A, SIG_A)`.

**Step 2**: The malicious attested participant calls:
```
respond_verify_foreign_tx(
  request = { tx_id: [0xBB;32], domain_id: 0, ... },   // request B
  response = { payload_hash: H_A, signature: SIG_A }    // replayed from A
)
```

**Step 3**: The contract at `crates/contract/src/lib.rs:726-733` checks only:
- Is `SIG_A` a valid ECDSA signature over `H_A` under the domain root key? → **YES** (it was legitimately produced)
- Is request B pending? → **YES**

The contract resolves request B with `{ payload_hash: H_A, signature: SIG_A }`. The caller of request B receives a `VerifyForeignTransactionResponse` attesting to transaction A's data, not transaction B's. Any bridge contract consuming this response without independently verifying `payload_hash` will treat transaction B as verified. [1](#0-0) [2](#0-1) [3](#0-2)

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
