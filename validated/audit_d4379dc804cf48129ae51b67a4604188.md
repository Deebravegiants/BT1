Audit Report

## Title
Missing `msg.sender == account` Guard in `MerkleDistributor.claim()` Enables Forced Fee Extraction on Any User's Yield — (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` accepts a caller-supplied `account` address with no check that `msg.sender == account`. Any party holding a valid merkle proof for a victim can force-trigger the victim's claim, causing the protocol fee (up to 10%) to be deducted from the victim's allocation and permanently redirected to `protocolTreasury`. The victim's claim is then marked consumed and cannot be re-executed.

## Finding Description
`MerkleDistributor.claim()` takes `account` as an unconstrained parameter: [1](#0-0) 

The function verifies only that the merkle proof is valid for the `(index, account, cumulativeAmount)` tuple — it does not verify that the caller is `account`: [2](#0-1) 

After proof verification, a fee of up to 10% is deducted and sent to `protocolTreasury`, with the remainder sent to `account`. The claim state is permanently updated: [3](#0-2) 

By contrast, `KernelMerkleDistributor._processClaim()` enforces the caller identity check that `MerkleDistributor` omits: [4](#0-3) 

Merkle proofs are distributed off-chain to all eligible users, making them publicly observable. An attacker can collect any user's proof and call `claim()` on their behalf at any time, forcing the fee deduction without the victim's consent.

## Impact Explanation
**High — Theft of unclaimed yield.** The victim receives `claimableAmount - fee` instead of `claimableAmount`. The fee (up to 10%, `MAX_FEE_IN_BPS = 1000`) is permanently redirected to `protocolTreasury`. Because `userClaims[account].cumulativeAmount` is updated to `cumulativeAmount`, the claim is consumed and the victim has no recourse to recover the fee. [5](#0-4) 

## Likelihood Explanation
**High.** The attack requires no special role, no capital, and no on-chain state manipulation. The only prerequisite is a valid merkle proof for the victim, which is publicly distributed off-chain to all eligible users. Any observer of the proof distribution can immediately execute the attack against any unclaimed account. The attack is repeatable across all eligible users in every distribution round.

## Recommendation
Add a caller identity check at the top of `claim()`, mirroring `KernelMerkleDistributor._processClaim()`:

```solidity
if (account != msg.sender) revert Unauthorized();
```

This ensures only the rightful owner can trigger their own claim and decide when to accept the fee deduction. [4](#0-3) 

## Proof of Concept
1. Protocol publishes a merkle tree. Alice is entitled to `1000 TOKEN` at `index=5`, `cumulativeAmount=1000`. Her proof `P` is publicly available off-chain.
2. `feeInBPS = 500` (5%).
3. Attacker calls `MerkleDistributor.claim(5, alice, 1000, P)`.
4. Contract verifies proof (valid), computes `claimableAmount = 1000`, `fee = 50`, `amountToSend = 950`.
5. `950 TOKEN` is transferred to Alice; `50 TOKEN` is transferred to `protocolTreasury`.
6. `userClaims[alice].cumulativeAmount = 1000` — Alice's claim is permanently consumed.
7. Alice never consented to claiming, never received the full `1000 TOKEN`, and cannot recover the `50 TOKEN` fee.

**Foundry test sketch:**
```solidity
function testForcedClaim() public {
    // Setup: merkle tree with alice entitled to 1000 tokens
    // attacker calls claim on alice's behalf
    vm.prank(attacker);
    distributor.claim(index, alice, 1000e18, aliceProof);
    // Assert alice received 950e18, treasury received 50e18
    assertEq(token.balanceOf(alice), 950e18);
    assertEq(token.balanceOf(treasury), 50e18);
    // Assert alice's claim is consumed
    assertTrue(distributor.isClaimed(index, alice));
}
``` [6](#0-5)
