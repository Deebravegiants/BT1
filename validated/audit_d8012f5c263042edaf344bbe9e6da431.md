### Title
Fan-out Queue Exhaustion via Caller-Unbound `verify_foreign_transaction` Requests - (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction()` does not bind the pending-request map key to the caller's account ID. Any unprivileged caller can submit 128 identical requests for the same foreign-chain transaction, exhausting the `MAX_PENDING_REQUEST_FAN_OUT` cap and causing every subsequent legitimate request for that transaction to be rejected with `PendingRequestQueueFull`.

---

### Finding Description

The contract exposes three user-facing request methods. Two of them explicitly embed the caller's identity into the request key:

**`sign()`** — binds to caller:
```rust
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // ← caller account ID included in key
    &request.path,
);
``` [1](#0-0) 

**`request_app_private_key()`** — binds to caller:
```rust
let request = CKDRequest::new(
    request.app_public_key,
    domain_id,
    &predecessor,   // ← caller account ID included in key
    &request.derivation_path,
);
``` [2](#0-1) 

**`verify_foreign_transaction()`** — does **not** bind to caller:
```rust
let request = args_into_verify_foreign_tx_request(request);
``` [3](#0-2) 

The mapping function produces a key containing only `domain_id`, `request`, and `payload_version` — no caller identity:
```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [4](#0-3) 

The pending-request fan-out queue is hard-capped at 128 entries per key:
```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
// ...
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string(),
    );
}
``` [5](#0-4) 

Because the `VerifyForeignTransactionRequest` key is fully determined by the foreign-chain transaction data (which is public on-chain), any attacker who observes a pending foreign-chain transaction can compute the exact key the victim will use and pre-fill all 128 slots before the victim's call lands.

---

### Impact Explanation

Once the 128-slot queue is full, every subsequent `verify_foreign_transaction` call for that specific foreign-chain transaction panics with `PendingRequestQueueFull`. The victim's yield-resume promise is never created, so the MPC nodes never deliver a signature to the victim. The bridge flow for that transaction is permanently blocked until the attacker's 128 yields time out — and the attacker can immediately re-fill the queue at that point.

Additionally, the attacker's 128 queued yields all receive the MPC-produced signature when nodes call `respond_verify_foreign_tx`. Depending on how the consuming bridge contract uses that signature (e.g., releasing funds to whoever presents it), the attacker may be able to front-run the legitimate user and execute the bridge action themselves.

This maps to the **Medium** allowed impact: *request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.*

---

### Likelihood Explanation

- The minimum deposit per call is 1 yoctoNEAR (`MINIMUM_SIGN_REQUEST_DEPOSIT`), so filling 128 slots costs 128 yoctoNEAR — effectively free. [6](#0-5) 
- The full request key (`domain_id`, foreign-chain tx data, `payload_version`) is entirely derivable from public foreign-chain state; no privileged information is required.
- The attack does not require TEE access, participant collusion, or any special role — any NEAR account can execute it.

---

### Recommendation

Embed the caller's account ID into the `VerifyForeignTransactionRequest` key, mirroring the pattern used by `sign()` and `request_app_private_key()`. Concretely, `args_into_verify_foreign_tx_request` should accept the `predecessor` and include it in the returned struct, and `VerifyForeignTransactionRequest` should carry a `requester: AccountId` field that participates in `PartialEq`/`Ord`/`BorshSerialize` (i.e., the map key). This ensures that each caller's queue slot is isolated and an attacker cannot exhaust another user's quota.

---

### Proof of Concept

1. A legitimate user (`victim.near`) submits a transaction on Ethereum (tx hash `0xABCD…`).
2. `victim.near` prepares to call `verify_foreign_transaction({ domain_id: 3, request: EthTx(0xABCD…), payload_version: 0 })`.
3. Attacker (`attacker.near`) observes `0xABCD…` on Ethereum and computes the identical NEAR request key.
4. Attacker submits 128 `verify_foreign_transaction` calls with the same arguments, each attaching 1 yoctoNEAR. Total cost: 128 yoctoNEAR + gas.
5. `pending_verify_foreign_tx_requests[key].len()` reaches 128.
6. `victim.near` calls `verify_foreign_transaction` — `push_pending_yield` panics with `PendingRequestQueueFull`.
7. MPC nodes process the request and call `respond_verify_foreign_tx`; all 128 attacker yields are resolved and the attacker receives 128 copies of the signature.
8. `victim.near` can never obtain the signature for this transaction unless the attacker's yields time out — at which point the attacker can immediately repeat step 4. [7](#0-6) [8](#0-7)

        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,

            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
