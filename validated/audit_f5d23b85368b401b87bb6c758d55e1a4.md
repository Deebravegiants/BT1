### Title
Governance Proposal Deposit Permanently Lost to Treasury When Submitted with Unregistered Return Address During Bootstrap Phase — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version 9), the `ProposalReturnAccountDoesNotExist` guard is intentionally skipped in `conwayGovTransition`. This allows any unprivileged transaction sender to submit a governance proposal whose `pProcReturnAddr` points to an unregistered stake account. When the proposal later expires or is enacted, `returnProposalDeposits` in the `EPOCH` rule cannot credit the deposit to the unregistered address and silently routes it to the treasury instead. The depositor permanently loses their governance action deposit with no on-chain recourse short of a governance-approved treasury withdrawal or a hard fork.

This is the direct structural analog of the reported bug: a restriction mechanism (bootstrap-phase guards) is applied to some entry points (`checkBootstrapProposal` blocks non-bootstrap action types) but is **not** applied to the return-address validity check, allowing users to enter a state (locked deposit with an unclaimable return address) they cannot exit.

---

### Finding Description

**Root cause — missing guard in `conwayGovTransition`**

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`, the `processProposal` inner function wraps the return-address registration check inside an `unless (hardforkConwayBootstrapPhase …)` block:

```haskell
unless (hardforkConwayBootstrapPhase $ pp ^. ppProtocolVersionL) $ do
  let refundAddress = proposal ^. pProcReturnAddrL
  isAccountRegistered (refundAddress ^. accountAddressCredentialL) (certDState ^. accountsL)
    ?! (injectFailure . ProposalReturnAccountDoesNotExist) refundAddress
``` [1](#0-0) 

`hardforkConwayBootstrapPhase` is `True` exactly when `pvMajor pv == natVersion @9`: [2](#0-1) 

During bootstrap the check is therefore **never executed**, so a proposal whose `pProcReturnAddr` credential is not in the accounts map is accepted without error.

The allowed proposal types during bootstrap are `ParameterChange`, `HardForkInitiation`, and `InfoAction`: [3](#0-2) 

All three can be submitted with an arbitrary, unregistered return address during bootstrap.

**Consequence — deposit routed to treasury at epoch boundary**

At every epoch boundary the `EPOCH` rule calls `returnProposalDeposits` for all removed (expired or enacted) proposals:

```haskell
processProposal gas (!accounts, !unclaimed)
  | (Just _accountState, newAccounts) <- updateLookupAccountState addRefund cred accounts =
      (newAccounts, unclaimed)
  | otherwise = (accounts, Map.insert (gasId gas) (gasDeposit gas) unclaimed)
  where
    cred = gasReturnAddr gas ^. accountAddressCredentialL
``` [4](#0-3) 

If `cred` is not registered, the deposit falls into `unclaimed`. The epoch transition then adds `unclaimed` to the treasury:

```haskell
& casTreasuryL <>~ (utxoState0 ^. utxosDonationL <> fold unclaimed)
``` [5](#0-4) 

The deposit is gone from the user's perspective. The treasury can only be accessed via a ratified `TreasuryWithdrawals` governance action, which requires a supermajority of DReps, SPOs, and the Constitutional Committee — effectively permanent loss.

**Test evidence confirming the asymmetry**

The existing test suite explicitly encodes this asymmetry with `FailPostBootstrap`, meaning the proposal **succeeds** during bootstrap and **fails** post-bootstrap:

```haskell
submitBootstrapAwareFailingProposal_ proposal $
  FailPostBootstrap
    [injectFailure $ ProposalReturnAccountDoesNotExist unregisteredAccountAddress]
``` [6](#0-5) 

---

### Impact Explanation

A user who submits any bootstrap-allowed governance proposal (`ParameterChange`, `HardForkInitiation`, `InfoAction`) with an unregistered `pProcReturnAddr` during protocol version 9 permanently loses their entire governance action deposit to the treasury. On Cardano mainnet the `ppGovActionDepositL` parameter is 100,000 ADA. The deposit cannot be recovered without either a hard fork or a successful governance-approved treasury withdrawal (requiring supermajority ratification). This matches the **High** impact criterion: *permanent freezing of deposits where recovery requires a hard fork or equivalent extraordinary governance action*.

---

### Likelihood Explanation

The bootstrap phase (protocol version 9) was the live initial phase of the Conway era on mainnet and remains the starting point for any new Conway-compatible deployment or testnet. Any unprivileged transaction sender can trigger this path by constructing a `ProposalProcedure` with a `pProcReturnAddr` whose credential is not in the accounts map. No special privilege, key compromise, or consensus majority is required. The only prerequisite is that the network is at protocol version 9. Likelihood is **medium**: the mainnet bootstrap phase has passed, but the code path remains live on testnets and any future deployment starting at version 9.

---

### Recommendation

Apply the `ProposalReturnAccountDoesNotExist` check unconditionally, removing the `unless (hardforkConwayBootstrapPhase …)` wrapper around the return-address registration check in `conwayGovTransition`. The analogous fix in the external report was to apply the missing guard (`whenNotPaused`) to the previously unguarded entry point (`onERC721Received`). Here the fix is symmetric: apply `ProposalReturnAccountDoesNotExist` during bootstrap as well, so that no proposal with an unclaimable return address can ever be accepted by the ledger.

---

### Proof of Concept

1. Network is at protocol version 9 (bootstrap phase active; `hardforkConwayBootstrapPhase` returns `True`).
2. Attacker/user constructs a `ProposalProcedure`:
   - `pProcGovAction = InfoAction` (a bootstrap-allowed type per `isBootstrapAction`)
   - `pProcReturnAddr = AccountAddress { aaId = AccountId unregisteredCred, … }` where `unregisteredCred` has no entry in the accounts map
   - `pProcDeposit = ppGovActionDepositL pp` (correct deposit amount)
3. Submit the transaction. The `checkBootstrapProposal` check passes (InfoAction is allowed). The `ProposalReturnAccountDoesNotExist` check is **skipped** because `hardforkConwayBootstrapPhase` is `True`. The proposal is accepted and the deposit is locked.
4. Wait `ppGovActionLifetimeL + 1` epochs without ratification. The proposal expires.
5. At the epoch boundary, `returnProposalDeposits` looks up `unregisteredCred` in the accounts map, finds nothing, and inserts the deposit into `unclaimed`.
6. `unclaimed` is folded into `casTreasuryL`. The depositor's funds are permanently in the treasury.
