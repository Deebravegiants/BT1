### Title
Unauthenticated Peer Can Inflate `highest_height` to Trigger False `StaleSyncHash` and Cause Data Reset — (`chain/client/src/sync/state/mod.rs`)

### Summary

An unauthenticated peer can report an arbitrarily large `PeerChainInfoV2.height` during the handshake. This value flows unvalidated into `StateSync::run` as `highest_height`. If it exceeds `block_header.height() + chain.epoch_length + STALE_SYNC_HASH_THRESHOLD`, the node returns `StateSyncResult::StaleSyncHash`, which the sync handler converts to `SyncHandlerRequest::EpochSyncDataReset`, causing the node to write the EPOCH_SYNC_DATA_RESET marker and wipe its data directory — discarding all previously synced state.

### Finding Description

The stale sync hash check in `StateSync::run` is:

```rust
if highest_height > block_header.height() + chain.epoch_length + STALE_SYNC_HASH_THRESHOLD {
    return Ok(StateSyncResult::StaleSyncHash);
}
``` [1](#0-0) 

`highest_height` is a `u64` passed in from `SyncHandler::handle_sync_needed`, which receives it from the client actor's sync loop. The client actor derives this value from `HighestHeightPeerInfo`, which is populated from peer-reported `PeerChainInfoV2.height` during the handshake — a field that is not cryptographically verified and can be set to any value by any connecting peer. [2](#0-1) 

When `StaleSyncHash` is returned, the handler immediately returns `SyncHandlerRequest::EpochSyncDataReset`: [3](#0-2) 

The `EpochSyncDataReset` variant is documented as: "Write the data reset marker and shut down so the supervisor can wipe and restart." [4](#0-3) 

There is no guard requiring a minimum number of peers, no cryptographic proof of the claimed height, and no cross-validation against locally known block headers before the stale check fires.

### Impact Explanation

A single unauthenticated peer connecting to a node that is mid-state-sync can set `highest_block_height` to `sync_hash_height + epoch_length + STALE_SYNC_HASH_THRESHOLD + 1` (e.g., any large number). This immediately triggers the data reset path. The node writes the EPOCH_SYNC_DATA_RESET marker, shuts down, and on restart the supervisor wipes the data directory — destroying all previously downloaded state parts and forcing a full re-sync from scratch. This is permanent data loss, not a transient DoS.

### Likelihood Explanation

Any peer that can complete a TCP handshake with the target node can trigger this. The `PeerChainInfoV2.height` field is self-reported and requires no stake, no signature over a block, and no proof of work. The threshold `STALE_SYNC_HASH_THRESHOLD = 100` (production) is easily exceeded by reporting any height above `sync_hash_height + epoch_length + 101`. [5](#0-4) 

### Recommendation

Before triggering `StaleSyncHash`, require that the claimed `highest_height` be backed by a locally verified block header (i.e., the node has actually received and validated a header at or near that height). Alternatively, require a quorum of peers (e.g., a majority of connected peers) to agree on a height exceeding the threshold before treating the sync hash as stale. A single peer's self-reported height must never be sufficient to trigger a destructive data reset.

### Proof of Concept

1. Start a node that is mid-state-sync with a valid `sync_hash` at height `H`.
2. Connect a malicious peer that reports `highest_block_height = H + epoch_length + STALE_SYNC_HASH_THRESHOLD + 1` in its `PeerChainInfoV2`.
3. The peer manager includes this peer in `highest_height_peers` and sets `highest_height` to the reported value.
4. On the next call to `StateSync::run`, the condition `highest_height > block_header.height() + chain.epoch_length + STALE_SYNC_HASH_THRESHOLD` evaluates to `true`.
5. `StateSyncResult::StaleSyncHash` is returned, converted to `SyncHandlerRequest::EpochSyncDataReset`.
6. The node writes the EPOCH_SYNC_DATA_RESET marker and wipes its data directory.
