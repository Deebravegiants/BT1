### Title
Malicious SQMR Peer Can Misattribute Transactions Across Blocks via Count-Only Acceptance in `parse_data_for_block` — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts exactly `n_transactions` items from the peer's response stream and writes them to storage as the body of the queried block, with no validation of transaction content against the block's committed `transaction_commitment`. A malicious SQMR peer can send `n_transactions(N) − 1` legitimate transactions for block N followed by one transaction belonging to block N+1, causing the syncing node to store a wrong `BlockBody` for block N and a corrupted (shifted) body for block N+1.

---

### Finding Description

The loop in `parse_data_for_block` is purely count-driven: [1](#0-0) 

`target_transaction_len` is read from the locally stored block header's `n_transactions` field. The loop pulls items from the shared `transactions_response_manager` stream until the count is satisfied. The only rejection path is a premature `Fin` (None) marker: [2](#0-1) 

If the peer sends exactly `n_transactions(N)` `Some(FullTransaction)` items — even if the last one belongs to block N+1 — no error is raised, no peer report is filed, and the assembled `block_body` is written directly to storage: [3](#0-2) 

The developer is aware that transaction hashes are unvalidated from untrusted sources: [4](#0-3) 

There is no post-loop check of the assembled `block_body` against the `transaction_commitment` stored in the block header. The grep across the entire `apollo_p2p_sync` client confirms zero calls to any `transaction_commitment` verification after `parse_data_for_block` returns.

Because `client_response_manager` is a single shared stream consumed sequentially across all blocks in the query window, the item consumed for block N is permanently gone when `parse_data_for_block` is called for block N+1: [5](#0-4) 

---

### Impact Explanation

A syncing node that connects to a malicious peer stores a `BlockBody` for block N containing a transaction that belongs to block N+1. Downstream effects:

- `starknet_getBlockWithTxs(N)` returns an authoritative-looking wrong transaction set — **High** per scope ("RPC … returns an authoritative-looking wrong value").
- `starknet_getTransactionByHash` for the misplaced transaction returns the wrong block number and offset.
- Block N+1's body is also corrupted (its first transaction is missing; it receives a shifted item instead), cascading the misattribution across the entire query window `[N, N+K)`.
- Any state re-execution or fee estimation performed over the stored block bodies operates on wrong inputs.

---

### Likelihood Explanation

Any node in the P2P network can respond to SQMR transaction queries. No authentication or stake is required to become a peer. The attack requires only that the victim node selects the malicious peer for a transaction sync query, which is a normal network event. The malicious peer needs to send exactly the right count of items — a trivial protocol-level manipulation.

---

### Recommendation

After the loop completes and `block_body` is assembled, compute the Pedersen/Poseidon transaction commitment over `block_body.transaction_hashes` and compare it against `block_header.transaction_commitment`. Reject with `BadPeerError` and report the peer if the commitment does not match. This is the same pattern already used for state-diff commitments. The existing TODO at line 88 should be promoted to a blocking validation gate, not deferred.

---

### Proof of Concept

```
1. Sync node issues SQMR TransactionQuery for blocks [N, N+1).
   Block N header is already stored with n_transactions = 3,
   transaction_commitment = C_N.

2. Malicious peer sends:
   - Msg 1: Some(FullTransaction { tx_N_0, output_N_0, hash_N_0 })
   - Msg 2: Some(FullTransaction { tx_N_1, output_N_1, hash_N_1 })
   - Msg 3: Some(FullTransaction { tx_N1_0, output_N1_0, hash_N1_0 })
             ^^^ first transaction of block N+1
   - Msg 4: DataOrFin(None)   ← Fin

3. parse_data_for_block(N):
   current_transaction_len reaches 3 == target_transaction_len.
   Returns Ok(Some((block_body_with_tx_N1_0_as_third_tx, N))).

4. write_to_storage calls append_body(N, block_body).
   storage.get_block_transactions(N) now returns [tx_N_0, tx_N_1, tx_N1_0].

5. Assert: storage.get_block_transactions(N)[2] == tx_N_0_2  → FAILS.
   The N+1 transaction is stored under block N.
```
