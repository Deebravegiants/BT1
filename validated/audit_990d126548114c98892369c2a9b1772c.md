### Title
`accountBalanceIntervals` Transaction Body Field Is Parsed and Forwarded to Plutus Scripts but Never Validated Against Actual Ledger Account Balances — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces `accountBalanceIntervals` as a new transaction body field that maps staking credentials to balance intervals (`AccountBalanceLowerBound`, `AccountBalanceUpperBound`, or `AccountBalanceBothBounds`). The field is correctly serialized, deserialized, stored, and guarded from Plutus V1–V3 scripts. However, neither the top-level UTXO rule (`dijkstraUtxoTransition`) nor the sub-transaction UTXO rule (`dijkstraSubUtxoTransition`) ever checks that the actual on-chain account balances satisfy the declared intervals. Any unprivileged transaction sender can include arbitrary, false interval data and the ledger will accept the transaction unconditionally.

---

### Finding Description

**Data type and field definition**

`AccountBalanceInterval` is defined in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs` as a constraint type with three variants:

```haskell
data AccountBalanceInterval era
  = AccountBalanceLowerBound !(Inclusive Coin)
  | AccountBalanceUpperBound !(Exclusive Coin)
  | AccountBalanceBothBounds !(Inclusive Coin) !(Exclusive Coin)
``` [1](#0-0) 

It is wrapped in `AccountBalanceIntervals`, a map from `AccountId` to `AccountBalanceInterval`: [2](#0-1) 

The field is present in both `TopTx` and `SubTx` transaction body constructors: [3](#0-2) [4](#0-3) 

**Field is blocked for Plutus V1–V3 but available for V4**

`guardDijkstraFeaturesForPlutusV1toV3` explicitly rejects transactions that carry non-empty `accountBalanceIntervals` when a V1–V3 script is being evaluated, confirming the field is a live, semantically meaningful input for V4 scripts: [5](#0-4) 

**No enforcement in the top-level UTXO rule**

`dijkstraUtxoTransition` performs every standard check (fee, collateral, min-coin outputs, value conservation, withdrawal limits, network IDs, etc.) but contains no step that reads `accountBalanceIntervalsTxBodyL` and compares the declared intervals against the actual `Accounts` state: [6](#0-5) 

The `accounts` variable (the live ledger account map) is fetched at line 348 and used only for `validateBatchWithdrawals`. It is never consulted for `accountBalanceIntervals`.

**No enforcement in the sub-transaction UTXO rule**

`dijkstraSubUtxoTransition` likewise performs output-size, input-set, network-ID, and withdrawal checks but has no `accountBalanceIntervals` validation: [7](#0-6) 

**No `validateAccountBalanceIntervals` function exists anywhere in the production rule files.** A search across all Dijkstra rule modules (`Utxo.hs`, `SubUtxo.hs`, `Ledger.hs`, `SubLedger.hs`, `Entities.hs`, `SubEntities.hs`) returns no such function.

**Comparison with the correctly-enforced `validateBatchWithdrawals`**

For contrast, withdrawal amounts are validated against actual balances: [8](#0-7) 

The same pattern — read the declared value, look up the actual ledger state, reject on mismatch — is entirely absent for `accountBalanceIntervals`.

---

### Impact Explanation

**Impact: Medium — attacker-controlled transactions exceed intended validation limits.**

The `accountBalanceIntervals` field is structurally designed as a slippage-control / precondition mechanism: a transaction author declares "this transaction is only valid if account X holds between L and U lovelace." Because the ledger never enforces this, any sender can:

1. Craft a transaction with `accountBalanceIntervals` asserting that a target account holds an amount it does not actually hold.
2. Submit the transaction; the ledger accepts it without checking the actual balance.
3. A Plutus V4 script that reads `accountBalanceIntervals` from its context and uses it as a guard (e.g., "proceed only if the declared lower bound is satisfied") is presented with attacker-controlled data that the ledger has implicitly certified as unchecked.

If a V4 script uses `accountBalanceIntervals` to gate a treasury withdrawal, a governance vote, or a large ADA transfer, the missing enforcement allows the attacker to bypass that gate entirely, potentially escalating to a Critical impact (direct loss of ADA or unauthorized governance action). At minimum, the declared constraint is silently ignored, meaning transactions execute outside their stated design parameters.

---

### Likelihood Explanation

**High likelihood of reaching the vulnerable code path.** Any unprivileged transaction sender can include `accountBalanceIntervals` in a Dijkstra-era transaction body. No special role, key, or governance majority is required. The field is optional (omitted when empty), so adding it to an otherwise valid transaction costs only the marginal serialization bytes. The attack is deterministic and requires no timing, front-running, or probabilistic element.

---

### Recommendation

Add a dedicated validation function — analogous to `validateBatchWithdrawals` — that iterates over `accountBalanceIntervalsTxBodyL`, looks up each `AccountId` in the live `Accounts` state, and rejects the transaction if any actual balance falls outside the declared interval. This check must be applied in both `dijkstraUtxoTransition` (for top-level transactions) and `dijkstraSubUtxoTransition` (for sub-transactions), using the `accounts` value already available in both rules. A corresponding `DijkstraUtxoPredFailure` constructor (e.g., `AccountBalanceOutsideInterval`) should be added to carry the offending credential and mismatch details.

---

### Proof of Concept

1. **Setup**: Deploy a Plutus V4 script `S` that locks 10 000 ADA. `S`'s spending condition reads `accountBalanceIntervals` from the script context and requires that credential `C` has a declared lower bound ≥ 5 000 ADA (intended as a "liquidity check" before releasing funds).

2. **State**: Credential `C` actually holds 0 ADA in the ledger's `Accounts` map.

3. **Attack**: Construct a Dijkstra `TopTx` that:
   - Spends the UTxO locked by `S`.
   - Sets `accountBalanceIntervals = { C → AccountBalanceLowerBound (Inclusive 5000_000000) }`.
   - Provides a valid redeemer for `S`.

4. **Ledger processing**: `dijkstraUtxoTransition` runs all checks. `validateBatchWithdrawals` passes (no withdrawals). The `accountBalanceIntervals` field is decoded and stored but never compared against `C`'s actual balance of 0. The transaction is accepted.

5. **Script execution**: `S` receives a Plutus context in which `accountBalanceIntervals` shows `C ↦ LowerBound 5000 ADA`. `S` evaluates its condition as satisfied and approves the spend. The 10 000 ADA is transferred to the attacker.

6. **Root cause**: The missing enforcement step between lines 378 and 385 of `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` — where `accountBalanceIntervals` should be validated against `accounts` — is the necessary vulnerable step in the ledger's own code. [9](#0-8)

validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL

dijkstraSubUtxoTransition = do
  TRC (SubUtxoEnv slot pp certState originalUtxo (IsValid isValid), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG

  let txBody = tx ^. bodyTxL

  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo
  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  let allSizedOutputs = txBody ^. allSizedOutputsTxBodyF
  let allOutputs = fmap sizedValue allSizedOutputs
  runTest $ Alonzo.validateOutputTooBigUTxO pp allOutputs

  runTest $ Shelley.validateInputSetEmptyUTxO txBody

  let inputs = txBody ^. inputsTxBodyL
  let refInputs = txBody ^. referenceInputsTxBodyL
  runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxoState) inputs

  runTestOnSignal $ Shelley.validateOutputBootAddrAttrsTooBig allOutputs

  runTestOnSignal $ Babbage.validateOutputTooSmallUTxO pp allSizedOutputs

  netId <- liftSTS $ asks networkId
  runTestOnSignal $ Shelley.validateWrongNetwork netId allOutputs
  runTestOnSignal $ Shelley.validateWrongNetworkWithdrawal netId txBody
  runTestOnSignal $ validateWrongNetworkInDirectDeposit netId txBody
  runTestOnSignal $ Alonzo.validateWrongNetworkInTxBody netId txBody

  if isValid
    then do
      newState <-
        Shelley.updateUTxOStateNoFees
          pp
          utxoState
          txBody
          certState
          (utxosGovState utxoState)
          (tellEvent . TotalDeposits (hashAnnotated txBody))
          (\a b -> tellEvent $ TxUTxODiff a b)
      pure $ newState & utxosDonationL <>~ txBody ^. treasuryDonationTxBodyL
    else
      pure utxoState
