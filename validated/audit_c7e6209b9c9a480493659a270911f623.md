### Title
Deposits Paid for Non-Executed `propose_update` Proposals Are Permanently Frozen in the MPC Contract - (File: `crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

### Summary

`propose_update` requires a substantial NEAR deposit (proportional to the size of the proposed contract binary — up to ~40 NEAR for a max-size contract). When any one update reaches threshold and `do_update` executes, it unconditionally clears **all** pending proposals and their storage, but never refunds the deposits paid by the proposers of the non-executed proposals. Those NEAR tokens are permanently frozen in the contract with no recovery path.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` collects a storage-staking deposit from the proposer: [1](#0-0) 

The deposit is absorbed into the contract's balance. Critically, neither the `UpdateEntry` struct nor `ProposedUpdates` records the depositor's account ID or the deposit amount: [2](#0-1) 

When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()` on **all remaining proposals**: [3](#0-2) 

The freed storage bytes are reclaimed by the NEAR runtime, but the NEAR tokens that were deposited for those cleared proposals are never returned to their proposers. There is no `withdraw_proposal_deposit` method, no per-proposal deposit ledger, and no refund path anywhere in the contract.

The `required_deposit` for a max-size contract binary (~1.5 MB) is computed as: [4](#0-3) 

Tests confirm deposits of up to 40 NEAR are expected for large contract proposals: [5](#0-4) 

### Impact Explanation

Every participant who proposed an update that was not the one executed permanently loses their deposit. In a scenario where N participants each propose a different update (a normal governance pattern), N-1 proposers lose their full deposit when the winning update executes. For a max-size contract binary, each lost deposit is ~40 NEAR. The funds are irrecoverably frozen in the contract balance with no mechanism to retrieve them.

This breaks the production safety/accounting invariant that deposited funds are either consumed for their stated purpose or returned to the depositor.

**Impact level: Medium** — balance/accounting invariant violation causing permanent freezing of participant-deposited NEAR tokens.

### Likelihood Explanation

This is triggered by the normal governance flow: multiple participants propose competing updates (a realistic and documented operational pattern), one reaches threshold, and all others are silently cleared. No adversarial behavior is required. Any participant who has ever proposed an update that was outcompeted by another is affected. The `test_propose_update_contract_many` sandbox test even demonstrates multiple simultaneous proposals existing, confirming this is an expected operational scenario. [6](#0-5) 

### Recommendation

Track the depositor and deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) depositor: AccountId,   // add
    pub(super) deposit: NearToken,     // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.depositor).transfer(entry.deposit)` for each one. This mirrors the refund pattern already used in `propose_update` itself and in `submit_participant_info`.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~27 NEAR deposit. `UpdateEntry { update, bytes_used }` is stored; the 27 NEAR enters the contract balance. No record of A's account or deposit amount is kept.
2. Participant B calls `propose_update` with a different 1 MB binary, attaching ~27 NEAR.
3. Threshold participants vote for B's proposal. `vote_update` calls `do_update(&id_B, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry is deleted.
5. A's 27 NEAR remains in the contract balance. There is no function to retrieve it. A's deposit is permanently frozen. [7](#0-6)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}

async fn test_propose_update_contract_many() {
    let SandboxTestSetup {
        contract,
        mpc_signer_accounts,
        ..
    } = SandboxTestSetup::builder()
        .with_protocols(ALL_PROTOCOLS)
        .build()
        .await;
    dbg!(contract.id());

    const PROPOSAL_COUNT: usize = 2;
    let mut proposals = Vec::with_capacity(PROPOSAL_COUNT);
    // Try to propose multiple updates to check if they are being proposed correctly
    // and that we can have many at once living in the contract state.
    for i in 0..PROPOSAL_COUNT {
        let execution = mpc_signer_accounts[i % mpc_signer_accounts.len()]
            .call(contract.id(), method_names::PROPOSE_UPDATE)
            .args_borsh(current_contract_proposal())
            .max_gas()
            .deposit(CURRENT_CONTRACT_DEPLOY_DEPOSIT)
            .transact()
            .await
            .unwrap();

        assert!(
            execution.is_success(),
            "failed to propose update [i={i}]; {execution:#?}"
        );
        let proposal_id = execution.json().expect("unable to convert into UpdateId");
        proposals.push(proposal_id);
    }

    // Vote for the last proposal
    vote_update_till_completion(&contract, &mpc_signer_accounts, proposals.last().unwrap()).await;

    // Ensure all proposals are removed after update
    for proposal in proposals {
        let voter = mpc_signer_accounts.first().unwrap();
        let execution = voter
            .call(contract.id(), method_names::VOTE_UPDATE)
            .args_json(serde_json::json!({
                "id": proposal,
            }))
            .gas(GAS_FOR_VOTE_UPDATE)
            .transact()
            .await
            .unwrap();
        dbg!(&execution);

        assert!(execution.is_failure());
    }
