### Title
Missing CKD Response Verification for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs a cryptographic pairing-based validity check on the CKD response **only** when the `AppPublicKeyPV` variant is used. For the standard `AppPublicKey` variant, the match arm is an empty no-op, meaning any attested participant can submit an arbitrary forged `CKDResponse` that the contract accepts and delivers to the user without any on-chain verification.

---

### Finding Description

In `respond_ckd`, after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk, app_id), pk)`, which cryptographically proves the response encodes the correct BLS signature under the MPC master key. [2](#0-1) 

For `AppPublicKey`, **no equivalent check exists**. The response `(big_y, big_c)` is accepted and forwarded to the user unconditionally.

The `AppPublicKey` variant is the standard, most commonly used form (a plain G1 point). It is accepted as a legacy plain string or tagged enum: [3](#0-2) 

The `request_app_private_key` entry point also skips any check for `AppPublicKey` at submission time, so there is no upstream guard either.

The CKD protocol's security guarantee is that the output `(Y, C)` satisfies `C = msk · H(pk, app_id) + a · Y`, where `msk` is the MPC network's master secret and `a` is the user's private key. Without on-chain verification, a forged `(Y', C')` pair breaks this guarantee.

The `resolve_yields_for` call is a NEAR yield-resume mechanism: the **first** valid `respond_ckd` call resolves the pending request and delivers the response to the user. A racing malicious participant wins if they submit before the honest coordinator. [4](#0-3) 

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with arbitrary `big_y` and `big_c` values for any pending `AppPublicKey` CKD request. The contract accepts the forged response and delivers it to the user. The user's TEE application then computes `sig = big_c - a · big_y`, which does **not** equal `msk · H(pk, app_id)`. The user receives a key that is not derived from the MPC network's master secret — this is unauthorized confidential key derivation output without the required threshold participant authorization.

This matches the Critical impact class: **"Unauthorized … confidential key derivation output without the required participant authorization."**

---

### Likelihood Explanation

The `AppPublicKey` variant is the default and most widely used form of CKD request (plain G1 point, backward-compatible). Any single attested participant in the MPC network can exploit this. The attacker does not need to collude with others, compromise TEE hardware, or perform any network-level attack — they only need to race their `respond_ckd` call ahead of the honest coordinator's submission. Given that the honest coordinator must complete the multi-round MPC protocol before submitting, a malicious participant has a timing window to front-run the response.

---

### Recommendation

1. **Require `AppPublicKeyPV` for all CKD requests** so that on-chain pairing verification is always possible. Deprecate the `AppPublicKey` variant or gate it behind an explicit acknowledgment that no on-chain verification is performed.
2. **Alternatively**, if `AppPublicKey` must remain supported, document clearly that its security relies entirely on TEE attestation preventing any single participant from submitting a forged response — and add a contract-level guard that only the designated leader (coordinator) for a given request may call `respond_ckd`, reducing the attack surface to threshold collusion.

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk)` where `pk = a · G1` for some private scalar `a`.
2. Honest MPC nodes begin the multi-round `ckd` protocol to compute the correct `(Y, C)`.
3. Before the honest coordinator submits, a malicious attested participant calls:
   ```
   respond_ckd(ckd_request, CKDResponse { big_y: G1, big_c: G1 })
   ```
4. The contract's `respond_ckd` hits the empty `AppPublicKey(_) => {}` arm — no check is performed.
5. `resolve_yields_for` resolves the pending yield with the forged `(G1, G1)`.
6. The user's TEE app receives `(Y', C') = (G1, G1)` and computes:
   ```
   sig = C' - a · Y' = G1 - a · G1 = (1 - a) · G1
   ```
   This is not `msk · H(pk, app_id)`. The derived key is incorrect and attacker-influenced.

The test at line 3403–3441 confirms the contract accepts arbitrary byte values `[1u8; 48]` and `[2u8; 48]` for `big_y` and `big_c` in the `AppPublicKey` path without any rejection: [5](#0-4) 

Contrast with the `AppPublicKeyPV` path, which correctly rejects tampered responses: [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
