Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Mint Calculation Without Prior Oracle Update — (File: `contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, which returns a cached state variable that is only refreshed on explicit invocation of `updateRSETHPrice()`. No call to update the price occurs anywhere in the deposit flow. When `rsETHPrice` lags behind the true value (as rewards accrue), depositors receive excess rsETH, diluting the ETH-per-rsETH ratio for all existing holders and transferring their unclaimed yield to the new depositor.

## Finding Description
`LRTOracle` stores the rsETH/ETH exchange rate in the state variable `rsETHPrice` (L28). This value is computed only when `_updateRsETHPrice()` is invoked — either via the public permissionless `updateRSETHPrice()` (L87–89) or the manager-gated `updateRSETHPriceAsManager()` (L94–96). It is never computed on-the-fly.

The deposit entry points `depositAsset()` (L99–118) and `depositETH()` (L76–93) both delegate to `_beforeDeposit()` (L648–670), which calls `getRsETHAmountToMint()` (L506–521). That function computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

There is no call to `updateRSETHPrice()` anywhere in this path. As EigenLayer staking rewards accrue, the true rsETH/ETH rate rises, but `rsETHPrice` remains at its last-written value. A lower stale denominator inflates `rsethAmountToMint`.

When `updateRSETHPrice()` is eventually called after the deposit, `_updateRsETHPrice()` computes the new price as `(totalETHInProtocol - protocolFeeInETH) / rsethSupply` (L250). Because the inflated deposit minted excess rsETH into `rsethSupply`, the resulting price is lower than it would have been had the price been updated before the deposit. Every existing holder's rsETH is now backed by less ETH per token — their accrued yield has been transferred to the attacker.

An additional aggravating factor: if the accumulated price drift exceeds `pricePercentageLimit`, the public `updateRSETHPrice()` reverts for non-managers (L263), meaning the staleness window can be extended beyond normal keeper latency without any privileged action by the attacker.

Existing guards do not mitigate this:
- `minRSETHAmountExpected` is a depositor-side slippage guard; it protects the depositor from receiving *too little*, not existing holders from the depositor receiving *too much*.
- There is no timestamp or block-number freshness check on `rsETHPrice` anywhere in the deposit flow.

## Impact Explanation
**High — Theft of unclaimed yield.**

Existing rsETH holders have earned yield (reflected in the rising true rsETH/ETH rate) that has not yet been crystallised into `rsETHPrice`. A deposit against a stale price mints excess rsETH, permanently diluting the ETH backing per rsETH token. When the attacker redeems through the withdrawal pipeline, they extract ETH that rightfully belonged to prior holders. The magnitude scales linearly with deposit size and the staleness gap, making large targeted deposits economically rational.

## Likelihood Explanation
**Medium.**

`updateRSETHPrice()` is public and permissionless, so under normal keeper operation the staleness window is short. However: (1) the protocol enforces no on-chain freshness requirement before minting; (2) keeper downtime, network congestion, or deliberate griefing of the keeper creates exploitable windows; (3) if the drift exceeds `pricePercentageLimit`, the public update path reverts, extending the window further; (4) the attacker requires no privilege — only the ability to monitor the on-chain state and time a deposit. The attack is repeatable across every staleness window.

## Recommendation
Call `updateRSETHPrice()` (or an internal equivalent that skips the `whenNotPaused` guard) at the start of `depositAsset()` and `depositETH()`, before `_beforeDeposit()` computes the mint amount. Alternatively, record `block.timestamp` each time `rsETHPrice` is written and revert in `getRsETHAmountToMint()` if the price is older than a configurable maximum staleness window (e.g., 1 hour). Either approach eliminates the gap between the true rate and the minted amount.

## Proof of Concept

**State at time T:**
- `rsETHPrice = 1.01e18` (last written)
- True rsETH/ETH rate = `1.02e18` (rewards have accrued; `updateRSETHPrice()` not yet called)

**Attack sequence (no privileges required):**

1. Attacker observes `rsETHPrice` is stale by comparing it against the live TVL calculation off-chain.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint(stETH, 100e18)` executes:
   ```
   rsethAmountToMint = (100e18 * 1e18) / 1.01e18 ≈ 99.0099 rsETH
   ```
   Correct amount at true price: `(100e18 * 1e18) / 1.02e18 ≈ 98.0392 rsETH`
   **Excess minted: ~0.9707 rsETH**
4. Keeper (or anyone) later calls `updateRSETHPrice()`. The new price is computed over the inflated supply, permanently lowering the ETH-per-rsETH ratio for all prior holders.
5. Attacker initiates withdrawal, receiving ETH proportional to their excess rsETH balance — extracting yield that belonged to existing holders.
