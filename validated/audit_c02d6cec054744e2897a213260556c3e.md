### Title
Missing `parent_hash` Continuity Check in P2P Header Sync Allows Malicious Peer to Corrupt Stored Block Headers — (`crates/apollo_p2p_sync/src/client/header.rs`)

### Summary
`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` from an unauthenticated p2p peer and passes it directly to `write_to_storage` → `append_header` without verifying that the header's `parent_hash` matches the previously stored block's hash. A malicious peer can inject an arbitrary `parent_hash` for any block, which is then persisted and served authoritatively by RPC endpoints.

### Finding Description
In `parse_data_for_block`, the only validations performed on a peer-supplied `SignedBlockHeader` are:

1. `block_number` matches the expected sequence [1](#0-0) 
2. `signatures.len() == ALLOWED_SIGNATURES_LENGTH` [2](#0-1) 

There is an explicit TODO acknowledging the missing check:

```rust
// TODO(shahak): Check that parent_hash is the same as the previous block's hash
// and handle reverts.
``` [3](#0-2) 

The accepted header is then written unconditionally:

```rust
storage_writer
    .begin_rw_txn()?
    .append_header(
        self.block_header.block_header_without_hash.block_number,
        &self.block_header,
    )?
``` [4](#0-3) 

By contrast, the central sync path explicitly calls `verify_parent_block_hash` before any storage write: [5](#0-4) 

That function reads the stored hash of block N-1 and compares it against the incoming block's `parent_hash`, returning `StateSyncError::ParentBlockHashMismatch` on mismatch. [6](#0-5) 

No equivalent guard exists anywhere in the p2p sync header pipeline.

### Impact Explanation
The `parent_hash` field stored in `BlockHeaderWithoutHash` is read directly by the RPC layer and mapped into the `BlockHeader` response struct: [7](#0-6) 

A node syncing via p2p will serve the attacker-supplied `parent_hash` as authoritative data to any caller of `starknet_getBlockWithTxHashes`, `starknet_getBlockWithTxs`, or `starknet_getStateUpdate`. Clients relying on this field for chain continuity verification (e.g., light clients, bridges, explorers) will receive a structurally valid but chain-incorrect value.

Additionally, because `parent_hash` is an input to `calculate_block_hash` (it is the last chained field in the Poseidon hash), a corrupted `parent_hash` also means the stored `block_hash` for block N is inconsistent with the real chain's hash for that block, propagating the corruption forward. [8](#0-7) 

### Likelihood Explanation
Any node that enables p2p sync and connects to at least one malicious or compromised peer is exposed. The attacker only needs to respond to a header request with a correctly-sequenced `block_number` and a single valid-length signature vector — both trivially satisfied. No cryptographic capability is required.

### Recommendation
Before returning `Ok(Some(signed_block_header))` in `parse_data_for_block`, read the stored hash of `block_number - 1` from `_storage_reader` (which is already passed as a parameter but currently unused, named `_storage_reader`) and assert it equals `signed_block_header.block_header.block_header_without_hash.parent_hash`. Return `ParseDataError::BadPeer(...)` on mismatch, mirroring the logic in `verify_parent_block_hash` in the central sync path. The `_storage_reader` parameter is already present in the function signature for exactly this purpose. [9](#0-8) 

### Proof of Concept
Using the existing header test harness:

1. Set up a p2p sync client with a mock peer.
2. Have the mock peer respond to the header request for `BlockNumber(1)` with a `SignedBlockHeader` where `block_header_without_hash.block_number = 1` and `block_header_without_hash.parent_hash = BlockHash(Felt::from(0xdeadbeef_u64))`.
3. Let the sync loop process the response.
4. Assert: `storage_reader.begin_ro_txn().get_block_header(BlockNumber(1)).unwrap().unwrap().block_header_without_hash.parent_hash == BlockHash(Felt::from(0xdeadbeef_u64))`.
5. The assertion passes — the corrupted value is stored and will be returned by any RPC call that reads the header for block 1.
