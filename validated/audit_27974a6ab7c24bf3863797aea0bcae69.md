### Title
Missing Zero Address Check for `nearBridgeDerivedAddress_` in `OmniBridge.initialize()` Freezes All Incoming Bridge Transfers — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initialize()` accepts `nearBridgeDerivedAddress_` without a zero address check. If mistakenly set to `address(0)`, the ECDSA signature gate in both `finTransfer()` and `deployToken()` will always revert, freezing the entire incoming bridge transfer path. Tokens locked by users via `initTransfer()` become unclaimable until an admin upgrade is performed.

### Finding Description
`OmniBridge.initialize()` assigns `nearBridgeDerivedAddress` directly from the caller-supplied parameter with no validation:

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {
    tokenImplementationAddress = tokenImplementationAddress_;
    nearBridgeDerivedAddress = nearBridgeDerivedAddress_;   // ← no zero-address check
    ...
}
``` [1](#0-0) 

`nearBridgeDerivedAddress` is the **sole authorization gate** for both critical outbound paths:

- `finTransfer()` (line 311): `if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) revert InvalidSignature();`
- `deployToken()` (line 151): same check [2](#0-1) [3](#0-2) 

No valid ECDSA private key maps to `address(0)`. OpenZeppelin's `ECDSA.recover` reverts on any signature that would recover to `address(0)`. Therefore, if `nearBridgeDerivedAddress = address(0)`, both `finTransfer` and `deployToken` will **always** revert with `InvalidSignature`, regardless of the signature provided.

Note that `tokenImplementationAddress` already has a zero-address guard at line 139 (`if (tokenImplementationAddress == address(0)) revert TokenImplementationNotSet()`), but `nearBridgeDerivedAddress_` has no equivalent protection. [4](#0-3) 

The same root cause is present in `OmniBridgeWormhole.initializeWormhole()`, which delegates to `initialize()` and passes `nearBridgeDerivedAddress` through unchanged. [5](#0-4) 

### Impact Explanation
- `finTransfer()` is frozen: all tokens locked on EVM via `initTransfer()` cannot be released to recipients.
- `deployToken()` is frozen: no new bridge tokens can be deployed.
- Every user who calls `initTransfer()` after the broken deployment locks their assets with no redemption path until an admin upgrade is executed.
- This constitutes a frozen redemption path for all bridge assets on the affected EVM chain.

The `initialize()` function is callable exactly once (enforced by the `initializer` modifier), so the broken state cannot be corrected through re-initialization.

### Likelihood Explanation
- `initialize()` is called once at deployment time, typically via a deployment script that derives `nearBridgeDerivedAddress` off-chain. A scripting error, misconfiguration, or copy-paste mistake could supply `address(0)`.
- The contract is UUPS upgradeable, so an admin can deploy a corrected implementation, but the window between deployment and upgrade exposes any `initTransfer` callers to fund loss.
- The `initializer` modifier prevents any second call to fix the value in-place.

### Recommendation
Add an explicit zero address check in `initialize()`:

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {
    require(nearBridgeDerivedAddress_ != address(0), "ERR_ZERO_NEAR_BRIDGE_ADDRESS");
    require(tokenImplementationAddress_ != address(0), "ERR_ZERO_TOKEN_IMPL_ADDRESS");
    tokenImplementationAddress = tokenImplementationAddress_;
    nearBridgeDerivedAddress = nearBridgeDerivedAddress_;
    ...
}
```

Apply the same guard in `OmniBridgeWormhole.initializeWormhole()` for `wormholeAddress`.

### Proof of Concept
1. Deploy `OmniBridge` proxy calling `initialize(validImpl, address(0), chainId)`.
2. User calls `initTransfer(token, amount, fee, nativeFee, recipient, message)` — tokens are burned/locked on EVM; `InitTransfer` event is emitted.
3. NEAR side observes the event and the MPC network signs a `TransferMessagePayload`.
4. Relayer calls `finTransfer(validSignature, payload)`.
5. `ECDSA.recover(hashed, validSignature)` returns a valid non-zero address `A`.
6. `A != address(0)` → `revert InvalidSignature()`.
7. User's tokens are permanently locked with no redemption path until an admin upgrade corrects `nearBridgeDerivedAddress`. [6](#0-5)

        bytes calldata signatureData,
        BridgeTypes.TransferMessagePayload calldata payload

            revert NonceAlreadyUsed(payload.destinationNonce);
        }

        completedTransfers[payload.destinationNonce] = true;

        bytes memory borshEncoded = bytes.concat(
            bytes1(uint8(BridgeTypes.PayloadType.TransferMessage)),
            Borsh.encodeUint64(payload.destinationNonce),
            bytes1(payload.originChain),
            Borsh.encodeUint64(payload.originNonce),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(payload.tokenAddress),
            Borsh.encodeUint128(payload.amount),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(payload.recipient),
            bytes(payload.feeRecipient).length == 0 // None or Some(String) in rust
                ? bytes("\x00")

                    bytes("\x01"),
                    Borsh.encodeString(payload.feeRecipient)

        );
