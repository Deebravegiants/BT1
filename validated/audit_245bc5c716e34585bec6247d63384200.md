Audit Report

## Title
`LRTOracle._getTotalEthInProtocol` Lacks Error Handling for External Oracle Calls, Freezing rsETH Price Updates and Protocol Fee Minting - (File: contracts/LRTOracle.sol)

## Summary
`_getTotalEthInProtocol()` iterates over all supported assets and makes unguarded external calls to each asset's price oracle. If any single oracle reverts — including due to the protocol's own `ChainlinkPriceOracle` reverting on a negative `int256`→`uint256` cast — the entire `updateRSETHPrice()` call reverts. This permanently freezes protocol fee minting (unclaimed yield) and the rsETH price until an admin manually replaces the broken oracle.

## Finding Description
`_getTotalEthInProtocol()` (lines 331–349) loops over all supported assets and calls `getAssetPrice(asset)` for each, with no try-catch or low-level call wrapping:

```solidity
// contracts/LRTOracle.sol L336-L343
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);   // unguarded external call
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    totalETHInProtocol += totalAssetAmt.mulWad(assetER);
    unchecked { ++assetIdx; }
}
```

`getAssetPrice` (lines 156–158) delegates directly to the registered `IPriceFetcher`:

```solidity
return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
```

The registered oracle contracts make unguarded external calls to third-party contracts:
- `ChainlinkPriceOracle.getAssetPrice()` calls `priceFeed.latestRoundData()` and then performs `uint256(price)` (line 52–54). In Solidity 0.8.x, casting a negative `int256` to `uint256` causes a panic revert — a code-level bug in the protocol's own wrapper that does not require any external compromise.
- `RETHPriceOracle.getAssetPrice()` calls `IrETH(rETHAddress).getExchangeRate()` (line 39).
- `SfrxETHPriceOracle.getAssetPrice()` calls `ISfrxETH.pricePerShare()` (line 40).
- `SwETHPriceOracle.getAssetPrice()` calls `ISwETH.getRate()` (line 39).

`_getTotalEthInProtocol()` is called exclusively by `_updateRsETHPrice()` (line 231), which is called by the public `updateRSETHPrice()` (line 87–89) and manager-gated `updateRSETHPriceAsManager()` (line 94–96). A revert in any oracle propagates all the way up, preventing `rsETHPrice` from being updated and preventing the fee mint at lines 298–311 from ever executing.

## Impact Explanation
**Medium. Permanent freezing of unclaimed yield.**

Protocol fee minting occurs inside `_updateRsETHPrice()` at lines 298–311 (`IRSETH.mint(treasury, rsethAmountToMintAsProtocolFee)`). If any oracle reverts, `_updateRsETHPrice()` never completes, so the treasury never receives accrued protocol fees. This constitutes permanent freezing of unclaimed yield for the duration of the outage. The freeze has no on-chain time bound and persists until an admin manually calls `updatePriceOracleFor` to replace the broken oracle — a reactive, manual remediation with no automatic recovery path.

Additionally, the price-drop auto-pause safety mechanism (lines 277–281) cannot fire while the oracle is broken, removing a critical circuit-breaker.

## Likelihood Explanation
Any public caller can invoke `updateRSETHPrice()`. The revert condition is triggered by an external oracle failure, which is a realistic operational event:
- Chainlink periodically deprecates price feeds; a deprecated feed can return a negative `answer`, which causes the protocol's own `ChainlinkPriceOracle` to revert on the `uint256(price)` cast (Solidity 0.8.x panic) — no external compromise required, only normal Chainlink feed lifecycle.
- LST protocols (Rocket Pool, Frax, Swell) have historically paused or upgraded their rate contracts.
- As more assets are added to the supported list, the probability of at least one oracle being in a broken state at any given time increases.

The `ChainlinkPriceOracle` negative-price revert path is entirely within the protocol's own code and requires no external actor to act maliciously.

## Recommendation
Wrap each external oracle call in a `try-catch` inside `_getTotalEthInProtocol()`. On failure, either skip the asset and emit a named event, or revert with a specific error identifying the broken oracle:

```solidity
try IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset) returns (uint256 price) {
    totalETHInProtocol += totalAssetAmt.mulWad(price);
} catch {
    emit AssetPriceOracleFailed(asset);
    // protocol policy: skip asset or revert with named error
}
```

Additionally, `ChainlinkPriceOracle.getAssetPrice()` should validate that `price > 0` before casting to `uint256` to prevent a panic revert from a stale or deprecated Chainlink feed.

## Proof of Concept

1. Deploy the protocol with assets `[stETH, rETH]`, each with its respective oracle.
2. Simulate Chainlink returning `answer = -1` for the stETH feed (possible on deprecated feeds), or simulate `rETH.getExchangeRate()` reverting (e.g., Rocket Pool upgrade).
3. Any EOA calls `LRTOracle.updateRSETHPrice()`.
4. Execution enters `_getTotalEthInProtocol()` → `getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → `uint256(-1)` → **Solidity 0.8.x panic revert** (or for rETH: `IrETH.getExchangeRate()` → **revert**).
5. Revert bubbles through `_getTotalEthInProtocol` → `_updateRsETHPrice` → `updateRSETHPrice`.
6. `rsETHPrice` is never updated; `IRSETH.mint(treasury, ...)` is never reached; the price-drop auto-pause never fires.
7. State persists until admin calls `updatePriceOracleFor(stETH, newOracle)`.

**Foundry fork test plan:** Fork mainnet, call `vm.mockCallRevert` on the Chainlink feed's `latestRoundData()` for one asset, then call `lrtOracle.updateRSETHPrice()` and assert it reverts. Confirm `rsETHPrice` is unchanged and no `FeeMinted` event was emitted.
