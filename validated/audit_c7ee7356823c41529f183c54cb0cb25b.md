### Title
Single Attested Participant Can Forge CKD Response for `AppPublicKey` Variant Due to Missing On-Chain Output Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs **zero cryptographic verification** of the CKD output when the request uses the `AppPublicKey` (legacy, privately-verifiable) variant. Any single attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse`, bypassing the threshold security guarantee and delivering a confidential key of the attacker's choosing to the user.

---

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs:675-682`), the contract branches on `request.app_public_key`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For the `AppPublicKeyPV` variant, `ckd_output_check` (`crates/contract/src/primitives/ckd.rs:80-101`) enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` on-chain, binding the response to the MPC network's public key and the user's app key. For the `AppPublicKey` (legacy) variant, the arm is a no-op — the contract accepts **any** `big_y` and `big_c` values without any check.

This is the direct analog of the external report's root cause: just as `calcSingleInGivenPoolOut` silently returns 0 when `inRecord.balance == 0` and the caller never checks for it, `respond_ckd` silently skips all verification when `AppPublicKey` is used and the contract never enforces correctness.

**Exploit path:**

1. A user submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(app_pk)` (the legacy format, still accepted per `crates/contract/src/lib.rs:484-485`). The request is stored in `pending_ckd_requests`.
2. A single malicious attested participant (below threshold) observes the pending request on-chain.
3. The attacker constructs a forged `CKDResponse`:
   - Choose attacker scalar `y_a` and secret `s_a`.
   - Set `big_y = G1 * y_a`, `big_c = G1 * s_a + app_pk * y_a`.
4. The attacker calls `respond_ckd(request, forged_response)` before the honest coordinator. The contract passes `assert_caller_is_attested_participant_and_protocol_active()`, skips verification (empty arm), and calls `resolve_yields_for` which drains the queue and delivers the forged response to all waiting callers.
5. The user receives `{big_y, big_c}` and decrypts: `big_c - app_sk * big_y = G1 * s_a + app_pk * y_a - app_sk * G1 * y_a = G1 * s_a`. The user's "confidential key" is `G1 * s_a` — a value the attacker chose and knows.

The `respond` function for signatures (`crates/contract/src/lib.rs:586-644`) correctly verifies the ECDSA/EdDSA signature against the derived public key before accepting. `respond_ckd` provides no equivalent protection for the `AppPublicKey` variant. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Critical.** A single Byzantine participant (strictly below the signing threshold) can forge a CKD response for any pending `AppPublicKey` request. The user receives a confidential key whose discrete log is known to the attacker, breaking the core confidentiality guarantee of the CKD protocol. Any downstream system that relies on the derived key for encryption, authentication, or secret storage is compromised. The attacker can impersonate the user or decrypt data protected by the derived key. This is an unauthorized confidential key derivation output produced without the required threshold participant authorization. [4](#0-3) 

---

### Likelihood Explanation

**High.** The `AppPublicKey` variant is the documented legacy format, still accepted by the contract and used in production integrations (e.g., `crates/contract/tests/sandbox/utils/sign_utils.rs:421-451`). Any single attested participant — a role reachable without privileged assumptions — can exploit this. The attacker only needs to front-run the honest coordinator's `respond_ckd` call. On NEAR, transactions are visible before finalization, making front-running feasible. No threshold collusion, no leaked keys, and no privileged access are required. [5](#0-4) [6](#0-5) 

---

### Recommendation

**Short term:** Reject `respond_ckd` calls for `AppPublicKey` requests unless the response passes a threshold-agreement check (e.g., require `t` participants to submit matching responses before resolving). Alternatively, add a revert if the `AppPublicKey` variant is used in `respond_ckd` until a proper fix is in place.

**Long term:** Deprecate the `AppPublicKey` variant and require all new CKD requests to use `AppPublicKeyPV`, which enables the on-chain `ckd_output_check` pairing verification. Migrate existing integrations. The `AppPublicKeyPV` path already correctly enforces `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` on-chain, binding the response to the MPC network's public key. [7](#0-6) [1](#0-0)

### Proof of Concept

```
1. Deploy contract in Running state with a BLS12-381 CKD domain.
2. Attacker registers as an attested participant (one node, below threshold).
3. Victim calls:
     request_app_private_key({
       derivation_path: "my-path",
       app_public_key: AppPublicKey("bls12381g1:<victim_pk>"),
       domain_id: 0
     })
   with 1 yoctoNEAR deposit.
4. Attacker observes the pending CKDRequest on-chain.
5. Attacker computes:
     y_a  = random scalar
     s_a  = chosen scalar (attacker's backdoor)
     big_y = G1 * y_a
     big_c = G1 * s_a + victim_app_pk * y_a
6. Attacker calls respond_ckd(request, {big_y, big_c}).
   Contract passes participant check, skips verification (AppPublicKey arm is empty),
   resolves the yield, and delivers {big_y, big_c} to the victim.
7. Victim decrypts: big_c - victim_app_sk * big_y = G1 * s_a.
   Attacker knows s_a, so attacker knows victim's "confidential key" G1 * s_a.
``` [4](#0-3) [8](#0-7)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
