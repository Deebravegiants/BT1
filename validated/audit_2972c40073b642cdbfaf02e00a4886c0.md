### Title
Unbounded DRep Registration Enables Linear-Scan DoS on Governance Proposal Submission — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

Any transaction containing a governance proposal triggers `updateDormantDRepExpiry`, which performs an unconditional `Map.map` over every registered DRep in `VState.vsDReps`. Because DRep registration has no hard cap, an attacker can pre-register an arbitrarily large number of DReps and then force every subsequent governance-proposal transaction to perform O(N) work proportional to the total DRep count, potentially exceeding block-validation time budgets and preventing governance proposals from being processed.

---

### Finding Description

**Unbounded DRep registration.** The `ConwayRegDRep` branch of the `GOVCERT` rule inserts a new entry into `vsDReps` with no upper-bound check on the map's size: [1](#0-0) 

The only guards are "not already registered" and "deposit matches `ppDRepDeposit`". There is no `maxNumDReps` protocol parameter or any other cardinality limit. The `VState` type stores all registered DReps in a plain `Map`: [2](#0-1) 

**Linear scan triggered by every governance proposal.** `updateDormantDRepExpiries` fires whenever a transaction contains at least one proposal procedure: [3](#0-2) 

It delegates to `updateDormantDRepExpiry`, which calls `Map.map updateExpiry` over the **entire** `vsDReps` map: [4](#0-3) 

This O(N) traversal is invoked unconditionally (subject only to `numDormantEpochs /= 0`) inside the `CERTS` transition rule: [5](#0-4) 

and again in the `LEDGER` rule for protocol versions that have moved the check there: [6](#0-5) 

Unlike Plutus script execution, native ledger-rule Haskell is not metered by execution-unit budgets; its cost is bounded only by wall-clock block-validation time enforced by the consensus layer.

---

### Impact Explanation

**Medium — attacker-controlled transactions exceed intended validation limits.**

An attacker who has pre-registered N DReps forces every governance-proposal transaction to traverse all N entries. With a sufficiently large N, the wall-clock cost of `Map.map` over `vsDReps` can push block validation beyond the time budget tolerated by the consensus layer, causing block producers to be unable to include governance proposals and honest nodes to reject blocks that do contain them. This degrades or halts the Conway governance mechanism without requiring the attacker to hold a governance majority.

---

### Likelihood Explanation

DRep registration requires only a `ppDRepDeposit` deposit (a refundable amount set by governance). An attacker can register thousands of DReps across many transactions, recover the deposits later, and in the interim keep the `vsDReps` map inflated. The attack is cheap relative to the disruption caused: the attacker pays only transaction fees and the opportunity cost of locked ADA. The trigger condition (`numDormantEpochs > 0` and a proposal in the transaction) is routinely satisfied during normal governance activity.

---

### Recommendation

1. **Introduce a `maxNumDReps` protocol parameter** analogous to the `max_n_validators` fix in the Ditto patch, and enforce it in the `ConwayRegDRep` branch of `GOVCERT` before inserting into `vsDReps`.
2. **Avoid full-map traversal on the hot path.** Restructure `updateDormantDRepExpiry` so that the dormant-epoch adjustment is applied lazily (e.g., stored as a single offset and subtracted at read time) rather than eagerly rewriting every DRep entry on each proposal transaction.
3. **Require minimum stake or activity** for DRep registration to raise the economic cost of inflating the registry.

---

### Proof of Concept

```
-- Attacker setup (off-chain loop):
for i in 1..N:
    submit RegDRepTxCert(freshKeyHash_i, ppDRepDeposit, SNothing)
-- vsDReps now has N entries.

-- Trigger (any honest governance participant):
submit ProposalProcedure { pProcGovAction = InfoAction, ... }

-- During LEDGER/CERTS validation:
--   updateDormantDRepExpiries detects hasProposals = True
--   calls updateDormantDRepExpiry currentEpoch vState
--   which executes: vsDRepsL %~ Map.map updateExpiry   -- O(N) over N attacker DReps
--
-- With N large enough, this traversal exceeds the block-validation
-- time budget, causing the proposal transaction to be un-includable.
```

updateDormantDRepExpiry ::
  -- | Current Epoch
  EpochNo ->
  VState era ->
  VState era
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry
  where
    numDormantEpochs = vState ^. vsNumDormantEpochsL
    updateExpiry =
      drepExpiryL
        %~ \currentExpiry ->
          let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
           in if actualExpiry < currentEpoch
                then currentExpiry
                else actualExpiry
