### Title
Missing `omniBridgeChainId` Domain Separator in `deployToken` Signature Allows Cross-Chain Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.deployToken` verifies an MPC-signed `MetadataPayload` whose Borsh-encoded blob contains no `omniBridgeChainId` field. Because `nearBridgeDerivedAddress` is derived from a fixed path (`"bridge-1"`) and is therefore identical across every EVM deployment, a valid `deployToken` signature obtained on chain A is cryptographically indistinguishable from a valid signature on chain B. An unprivileged relayer can replay it verbatim, deploying a spurious canonical bridge-token identity on chain B that is backed by no locked collateral.

---

### Finding Description

**Root cause — missing chain-ID in the metadata signing blob**

`deployToken` constructs its signed blob as:

```
[PayloadType.Metadata | token | name | symbol | decimals]
``` [1](#0-0) 

`omniBridgeChainId` is stored in the contract but is never mixed into this blob.

By contrast, `finTransfer` injects `omniBridgeChainId` **twice** into its blob (once for the token address OmniAddress tag, once for the recipient tag): [2](#0-1) 

The same asymmetry is present on every other chain. Starknet's `MetadataPayload.to_borsh()` takes no argument and emits no chain ID: [3](#0-2) 

While `TransferMessagePayload.to_borsh(chain_id)` explicitly binds the chain: [4](#0-3) 

Aptos `metadata_to_borsh()` is identical — no chain ID — while `transfer_message_to_borsh(chain_id)` carries the comment *"preventing cross-chain replay"* for transfer messages but not for metadata: [5](#0-4) [6](#0-5) 

**Root cause — `nearBridgeDerivedAddress` is chain-agnostic**

The deployment script derives `nearBridgeDerivedAddress` from the NEAR bridge account ID and the **fixed** path `"bridge-1"`: [7](#0-6) 

This produces the same Ethereum address for every EVM chain. The NEAR bridge signs metadata with the same `SIGN_PATH` constant: [8](#0-7) 

The `MetadataPayload` serialized on NEAR also contains no chain ID: [9](#0-8) 

**Consequence**

Because the signed blob is chain-agnostic and the verifying key is the same on every chain, `ECDSA.recover(keccak256(borshEncoded), signatureData) == nearBridgeDerivedAddress` passes on every EVM deployment for the same `(signatureData, metadata)` tuple: [10](#0-9) 

---

### Impact Explanation

A successful replay on chain B:

1. Deploys a new `ERC1967Proxy` wrapping `BridgeToken` and sets `isBridgeToken[proxy] = true`, `nearToEthToken[metadata.token] = proxy`, `ethToNearToken[proxy] = metadata.token`. [11](#0-10) 

2. Permanently blocks the legitimate deployment of that NEAR token on chain B — the `"ERR_TOKEN_EXIST"` guard fires for any future call: [12](#0-11) 

3. Registers the spurious token as the canonical bridge-token identity for that NEAR token ID on chain B. If the NEAR bridge's off-chain relayer observes the `DeployToken` event and registers the spurious EVM address in its `token_id_to_address` mapping, subsequent `sign_transfer` calls will embed the spurious address in `finTransfer` signatures. Any `finTransfer` call on chain B will then call `IBridgeToken(spuriousProxy).mint(recipient, amount)` — minting tokens that have no locked collateral on NEAR. [13](#0-12) 

---

### Likelihood Explanation

- Every `deployToken` signature is broadcast on-chain and is trivially observable by any party.
- The attacker needs no privileged access, no leaked key, and no MPC collusion — only a publicly visible transaction on any chain where the token has already been legitimately deployed.
- The only precondition — that `nearBridgeDerivedAddress` is the same across chains — is the **intended production configuration**, confirmed by the deployment script.
- The replay succeeds as long as the target chain has not yet deployed that token (i.e., `nearToEthToken[metadata.token] == address(0)`), which is the normal state before a token is bridged to a new chain.

---

### Recommendation

Include `omniBridgeChainId` in the Borsh-encoded blob for `deployToken`, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // domain separator
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the same fix to the Starknet `MetadataPayload.to_borsh()` (add `chain_id` parameter, matching `TransferMessagePayload.to_borsh(chain_id)`) and to the Aptos `metadata_to_borsh()`. Update the NEAR `log_metadata_callback` to include the destination chain ID in the `MetadataPayload` before signing, so the MPC signature is bound to exactly one destination chain.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

// Setup: two OmniBridge proxies, same nearBridgeDerivedAddress, different omniBridgeChainId
// bridgeA.omniBridgeChainId = 1  (e.g. Ethereum)
// bridgeB.omniBridgeChainId = 2  (e.g. Base)

// Step 1: obtain a legitimate deployToken signature for chain A
// (e.g. by observing the on-chain tx or calling log_metadata on NEAR)
bytes memory signatureData = <valid_sig_from_chain_A>;
BridgeTypes.MetadataPayload memory metadata = BridgeTypes.MetadataPayload({
    token:    "wrap.near",
    name:     "Wrapped NEAR",
    symbol:   "wNEAR",
    decimals: 24
});

// Step 2: replay on chain B — identical calldata, different contract
address spuriousToken = bridgeB.deployToken(signatureData, metadata);

// Step 3: assert spurious token is registered as canonical on chain B
assert(bridgeB.isBridgeToken(spuriousToken));
assert(bridgeB.nearToEthToken("wrap.near") == spuriousToken);

// Step 4: legitimate deployment on chain B is now permanently blocked
vm.expectRevert("ERR_TOKEN_EXIST");
bridgeB.deployToken(signatureData, metadata); // or any other valid sig
```

The Borsh blob hashed on chain B is byte-for-byte identical to the one signed for chain A (no chain ID field), so `ECDSA.recover` returns `nearBridgeDerivedAddress` and the check at line 151 passes.
