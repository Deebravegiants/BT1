Audit Report

## Title
`getCurrentDay()` Reverts Before `startTimestamp`, Causing `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` to Revert - (File: contracts/pools/RSETHPoolV2.sol, contracts/pools/RSETHPoolV3.sol)

## Summary
`getCurrentDay()` performs an unchecked `block.timestamp - startTimestamp` subtraction in Solidity 0.8.27, which reverts with an arithmetic underflow panic whenever `block.timestamp < startTimestamp`. Because `reinitialize` enforces that `startTimestamp` must be at or after the current block, there is always a window during which `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` — both of which call `getCurrentDay()` unconditionally — revert for any caller. The state-changing `deposit()` path is separately protected by an explicit guard in `limitDailyMint`, but the view functions are not.

## Finding Description
In both `RSETHPoolV2` and `RSETHPoolV3`, `getCurrentDay()` is implemented as:

```solidity
function getCurrentDay() public view returns (uint256) {
    return (block.timestamp - startTimestamp) / 1 days;
}
```

`reinitialize` enforces that `startTimestamp` cannot be in the past:

```solidity
if (block.timestamp > _startTimestamp) {
    revert InvalidStartTimestamp();
}
```

This guarantees a window — from the moment `reinitialize` is called until `block.timestamp >= startTimestamp` — during which `block.timestamp - startTimestamp` underflows and reverts. Both public view functions call `getCurrentDay()` without any pre-condition guard:

```solidity
function remainingDailyMintLimit() external view returns (uint256) {
    uint256 effectiveDailyMintAmount = (getCurrentDay() > lastMintDay) ? 0 : dailyMintAmount;
    return dailyMintLimit - effectiveDailyMintAmount;
}

function getNextDailyLimitResetTimestamp() external view returns (uint256) {
    return startTimestamp + (getCurrentDay() + 1) * 1 days;
}
```

By contrast, the `limitDailyMint` modifier — the only state-changing caller of `getCurrentDay()` — is correctly guarded with `if (block.timestamp < startTimestamp) { revert MintBeforeStartTimestamp(); }` before invoking `getCurrentDay()`. The view functions have no equivalent guard.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` are public informational view functions intended to be queried by off-chain clients, integrators, and front-ends. During the pre-`startTimestamp` window, both revert unconditionally, returning no usable data. No funds are at risk and no state-changing path is broken, but the contract fails to deliver its promised informational interface.

## Likelihood Explanation
**Medium.** Every deployment that calls `reinitialize` with a future `startTimestamp` — which is the only valid input — creates this window. The window length equals `startTimestamp - block.timestamp` at the time of `reinitialize`, which can be hours or days. No special attacker action is required; any public caller triggers the revert simply by querying the view functions.

## Recommendation
Add a pre-condition guard in `getCurrentDay()` that returns a safe sentinel value when `block.timestamp < startTimestamp`, mirroring the guard already present in `limitDailyMint`:

```solidity
function getCurrentDay() public view returns (uint256) {
    if (block.timestamp < startTimestamp) return 0;
    return (block.timestamp - startTimestamp) / 1 days;
}
```

Alternatively, `remainingDailyMintLimit()` can explicitly return `dailyMintLimit` (full limit available) when `block.timestamp < startTimestamp`, and `getNextDailyLimitResetTimestamp()` can return `startTimestamp` directly in that case.

## Proof of Concept
1. Admin calls `reinitialize(dailyMintLimit, block.timestamp + 1 days)` on `RSETHPoolV2` or `RSETHPoolV3`.
2. `startTimestamp` is now set to `block.timestamp + 1 days`.
3. Any external caller immediately calls `remainingDailyMintLimit()`.
4. Internally, `getCurrentDay()` computes `block.timestamp - startTimestamp` = `block.timestamp - (block.timestamp + 1 days)`, a negative value, triggering an arithmetic underflow revert in Solidity 0.8.27.
5. `remainingDailyMintLimit()` reverts with a panic instead of returning the expected `dailyMintLimit`.
6. Same revert occurs for `getNextDailyLimitResetTimestamp()`.

Foundry test sketch:
```solidity
function test_remainingDailyMintLimit_revertsBeforeStart() public {
    uint256 futureStart = block.timestamp + 1 days;
    vm.prank(admin);
    pool.reinitialize(1 ether, futureStart);
    vm.expectRevert(); // arithmetic underflow panic
    pool.remainingDailyMintLimit();
}
```
