### Title
Zero `BlockConfirmations` Threshold Accepted Without Minimum Enforcement, Enabling Finality Bypass in Foreign-Chain Verification — (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, `crates/contract/src/lib.rs`)

### Summary

The `verify_foreign_transaction` contract method accepts a `BitcoinRpcRequest` whose `confirmations` field is fully caller-controlled with no minimum enforced anywhere in the contract or node pipeline. Setting `confirmations = 0` causes the `BitcoinInspector` to accept any transaction — including one with a single confirmation — as "finalized," and the MPC network will produce a valid threshold signature over it. In a bridge context this enables a double-spend: the attacker obtains a valid MPC-signed attestation for a barely-confirmed (or even mempool) Bitcoin transaction, redeems it on NEAR, then reorganizes the Bitcoin block.

### Finding Description

`BlockConfirmations` is defined as a plain `u64` wrapper with no lower-bound invariant:

```rust
pub struct BlockConfirmations(pub u64);
``` [1](#0-0) 

The ABI schema explicitly marks the minimum as `0.0`, confirming the contract accepts zero: [2](#0-1) 

Inside `verify_foreign_transaction`, the only guards are deposit, gas, chain-whitelist, and domain-existence checks. There is no validation that `request.confirmations > 0` (or any protocol-defined minimum): [3](#0-2) 

The node-side `BitcoinInspector::extract` enforces the threshold with a `<=` comparison:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [4](#0-3) 

When `block_confirmations_threshold = 0`, the condition `0 <= N` is trivially true for every `u64` value `N` returned by the RPC, so the finality gate is completely bypassed. The node then proceeds to `verify_block_is_canonical` and, if the block is currently canonical (even with 1 confirmation), produces a valid MPC signature over the `ForeignTxSignPayload`.

The signed payload is then delivered on-chain via `respond_verify_foreign_tx`, which only verifies the ECDSA signature against the domain public key — it does not re-examine the confirmation count: [5](#0-4) 

### Impact Explanation

In the primary use case documented for this feature — the Omnibridge inbound flow — a bridge contract calls `verify_foreign_transaction` and uses the returned MPC signature to release NEAR-side assets. If an attacker submits the request with `confirmations = 0`, the MPC will sign for a Bitcoin transaction that has only 1 confirmation (or even 0 if the RPC returns a mempool entry with a `blockhash`). The attacker can then:

1. Obtain the valid MPC-signed attestation and redeem NEAR-side assets.
2. Collude with miners or wait for a natural reorg to reverse the Bitcoin transaction.
3. Retain both the NEAR assets and the Bitcoin — a classic double-spend.

This matches the **High** impact category: *cross-chain verification bypass that causes invalid bridge execution or double-spend conditions*.

### Likelihood Explanation

The entry path requires no privilege: any NEAR account can call `verify_foreign_transaction` with a deposit of 1 yoctoNEAR. The `confirmations` field is a plain JSON integer with no contract-side floor. The attack is deterministic and requires only a Bitcoin transaction with ≥ 1 confirmation and a willing (or compromised) RPC provider that returns the expected response. For high-value bridges the economic incentive is significant.

### Recommendation

Enforce a protocol-defined minimum confirmation threshold on-chain before enqueuing the request. For Bitcoin, the widely accepted safe minimum is 6 confirmations. The check should be added inside `verify_foreign_transaction` when the request variant is `ForeignChainRpcRequest::Bitcoin`:

```rust
if request.confirmations.0 < MIN_BITCOIN_CONFIRMATIONS {
    env::panic_str("Bitcoin confirmations below protocol minimum");
}
```

Alternatively, store per-chain minimum confirmation requirements in the `ForeignChainRpcWhitelist` / `ChainEntry` and validate against them at request time, similar to how `quorum == 0` is already rejected: [6](#0-5) 

### Proof of Concept

1. Deploy the contract in running state with Bitcoin whitelisted.
2. Submit:
   ```json
   {
     "request": {
       "request": {
         "Bitcoin": {
           "tx_id": "<1-confirmation tx hash>",
           "confirmations": 0,
           "extractors": ["BlockHash"]
         }
       },
       "domain_id": <foreign_tx_domain>,
       "payload_version": 1
     }
   }
   ```
   with 1 yoctoNEAR attached.
3. The MPC nodes call `getrawtransaction`; the RPC returns `confirmations = 1`. The check `0 <= 1` passes.
4. `verify_block_is_canonical` passes (block is currently canonical).
5. MPC produces a valid threshold signature over the payload.
6. `respond_verify_foreign_tx` accepts the signature and resolves the yield.
7. The caller receives a valid `VerifyForeignTransactionResponse` for a 1-confirmation transaction, which can be used to unlock bridge funds before the transaction achieves safe finality.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
