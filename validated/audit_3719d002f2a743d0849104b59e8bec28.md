### Title
Unguarded Division by Zero in `getRsETHAmountToMint` When `rsETHPrice` Is Uninitialized — (`contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` without first verifying the value is non-zero. The `rsETHPrice` storage variable in `LRTOracle` is initialized to `0` by default and is only updated when `updateRSETHPrice()` is explicitly called. When `rsETHPrice == 0`, every user-facing deposit path reverts with a division-by-zero error, temporarily freezing all deposits.

---

### Finding Description

In `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` reads the `rsETHPrice` storage variable from `LRTOracle`. That variable is declared as:

```solidity
uint256 public override rsETHPrice;
```

with no initialization in `initialize()`. It is only written inside `_updateRsETHPrice()`, which is called by the public `updateRSETHPrice()`. Until that function is called, `rsETHPrice == 0`.

The `getRsETHAmountToMint()` function is invoked by `_beforeDeposit()`, which is called by both `depositETH()` and `depositAsset()` — the two primary user-facing deposit entry points. When `rsETHPrice == 0`, Solidity 0.8's checked arithmetic causes the division to revert, making every deposit attempt fail.

A secondary impact exists in `LRTWithdrawalManager.getExpectedAssetAmount()`:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

Here `rsETHPrice` is the **numerator**, so when it is `0` the function silently returns `0` instead of reverting. `initiateWithdrawal()` and `instantWithdrawal()` both call this function and use the result as the asset amount to disburse. A user who calls either function while `rsETHPrice == 0` will have their rsETH transferred/burned while receiving `0` underlying assets — a complete loss of principal.

---

### Impact Explanation

**Primary path — deposit freeze (Medium: Temporary freezing of funds):** All calls to `depositETH()` and `depositAsset()` revert with division-by-zero whenever `rsETHPrice == 0`, blocking the entire deposit surface until `updateRSETHPrice()` is called.

**Secondary path — withdrawal fund loss (Critical: Direct theft of user funds):** If `rsETHPrice == 0` while rsETH is in circulation, `getExpectedAssetAmount()` returns `0`. `instantWithdrawal()` burns the caller's rsETH and transfers `0` assets back. `initiateWithdrawal()` locks the caller's rsETH in the contract and records `expectedAssetAmount = 0`, so the completed withdrawal also yields `0` assets.

---

### Likelihood Explanation

`rsETHPrice` is `0` in two realistic situations:

1. **Initial / post-upgrade state:** Before `updateRSETHPrice()` is ever called, `rsETHPrice` remains at its default value of `0`. Any deposit attempted in this window reverts.
2. **Protocol-wide slashing to zero TVL:** If `updateRSETHPrice()` is called when `rsethSupply > 0` but `totalETHInProtocol == 0` (complete slashing), and `pricePercentageLimit == 0` (its default, disabling the downside-protection circuit-breaker), `_updateRsETHPrice()` computes `newRsETHPrice = 0` and writes it to storage without pausing the protocol.

Scenario 1 is a near-certain transient condition on every deployment or upgrade. Scenario 2 is low-probability but fully reachable without any privileged action once the state is reached.

---

### Recommendation

Add an explicit zero-guard in `getRsETHAmountToMint()` and in `getExpectedAssetAmount()`:

```solidity
// LRTDepositPool.getRsETHAmountToMint
uint256 currentRsETHPrice = lrtOracle.rsETHPrice();
if (currentRsETHPrice == 0) revert InvalidRsETHPrice();
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / currentRsETHPrice;
```

```solidity
// LRTWithdrawalManager.getExpectedAssetAmount
uint256 currentRsETHPrice = lrtOracle.rsETHPrice();
if (currentRsETHPrice == 0) revert InvalidRsETHPrice();
underlyingToReceive = amount * currentRsETHPrice / lrtOracle.getAssetPrice(asset);
```

Additionally, `LRTOracle.initialize()` should call `_updateRsETHPrice()` (or set `rsETHPrice = 1 ether`) so the variable is never left at `0` after deployment.

---

### Proof of Concept

**Deposit freeze:**

1. Deploy `LRTOracle` and `LRTDepositPool` without calling `updateRSETHPrice()`.
2. `rsETHPrice` is `0`.
3. Any user calls `depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `getRsETHAmountToMint` → `(1e18 * assetPrice) / 0` → Solidity 0.8 reverts with division-by-zero.
5. All deposits are blocked until an operator calls `updateRSETHPrice()`.

**Withdrawal fund loss (secondary):**

1. `rsETHPrice` is `0` (initial state or post-slashing with `pricePercentageLimit == 0`).
2. A user holding rsETH calls `instantWithdrawal(asset, rsETHAmount, "")`.
3. `getExpectedAssetAmount` returns `0`.
4. `IRSETH.burnFrom(msg.sender, rsETHAmount)` burns the user's rsETH.
5. `_transferAsset(asset, msg.sender, 0)` sends nothing.
