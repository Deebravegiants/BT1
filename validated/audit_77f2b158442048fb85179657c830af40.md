### Title
Missing Attestation Expiry Check in `is_caller_an_attested_participant` Allows Expired-Attestation Nodes to Participate in Threshold Signing - (File: crates/contract/src/tee/tee_state.rs)

### Summary
`TeeState::is_caller_an_attested_participant()`, which gates every signing-response submission (`respond`, `respond_ckd`, etc.), never checks whether the stored `VerifiedAttestation`'s `expiry_timestamp_seconds` has passed. A node whose TEE attestation has lapsed — because it stopped resubmitting, e.g. after switching to a non-approved image — can continue submitting signing responses and participating in threshold signature generation until someone manually calls `verify_tee()`. This is the direct analog of M-17: a freshness/staleness check is absent from the critical-path enforcement point, so stale (expired) state is silently trusted.

### Finding Description

`ValidatedDstackAttestation` carries an `expiry_timestamp_seconds` field stamped at submission time as `block_timestamp + DEFAULT_EXPIRATION_DURATION_SECONDS` (currently 1 day). [1](#0-0) [2](#0-1) 

The expiry is enforced in `VerifiedAttestation::re_verify()`, which is called only from `verify_tee()` and `clean_invalid_attestations()` — both are lazy, on-demand operations. [3](#0-2) 

However, `TeeState::is_caller_an_attested_participant()` — the function that gates every call to `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` — performs only identity and ownership checks. It never reads or compares `expiry_timestamp_seconds`: [4](#0-3) 

This function is invoked unconditionally before every signing response: [5](#0-4) 

The result is a gap: once a node's attestation expires, it remains in `stored_attestations` and continues to pass the attestation gate until `verify_tee()` is explicitly called. The docs describe `verify_tee()` as something "any participant can trigger" on a voluntary cadence — it is not automatic and has no on-chain enforcement interval. [6](#0-5) 

### Impact Explanation

The TEE attestation system's core invariant is: **only nodes running a governance-approved image inside a genuine TDX enclave may participate in threshold signing**. The missing expiry check in `is_caller_an_attested_participant` breaks this invariant. A node that has stopped resubmitting attestations — for example because it is now running a non-approved or compromised image — retains full signing-response capability for an unbounded period after its attestation lapses. During this window the node can contribute key-share material to live signing sessions, undermining the TEE security boundary that is the sole mechanism preventing a compromised node from participating in key-share operations. This maps to the **Medium** allowed impact: participant-state manipulation that breaks a production safety invariant (TEE-gated signing participation) without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation

A Byzantine participant below the signing threshold can exploit this without any collusion. The attacker needs only to:
1. Have been a legitimate participant with a valid attestation.
2. Switch to a non-approved image and stop resubmitting attestations.
3. Continue calling `respond()` — the contract accepts these calls because `is_caller_an_attested_participant` never checks expiry.

The window is open from the moment the attestation expires until `verify_tee()` is called. Because `verify_tee()` is voluntary and not automatically triggered, the window is unbounded in the worst case. Nodes resubmit hourly under normal operation, so the attack requires deliberate non-resubmission, making it a realistic adversarial scenario rather than an accidental one.

### Recommendation

Add an expiry check inside `is_caller_an_attested_participant` (or in a thin wrapper called at the same site) that reads `env::block_timestamp_ms() / 1_000` and compares it against the stored `expiry_timestamp_seconds`:

```rust
// After retrieving `attestation` from stored_attestations:
let now_seconds = env::block_timestamp_ms() / 1_000;
if let VerifiedAttestation::Dstack(ref d) = attestation.verified_attestation {
    if now_seconds >= d.expiry_timestamp_seconds {
        return Err(AttestationCheckError::AttestationExpired);
    }
}
```

This mirrors the check already present in `VerifiedAttestation::re_verify()` and closes the gap between lazy `verify_tee()` sweeps and real-time signing-response gating — exactly the "freshness check at the point of use" that M-17 recommended for the Curve oracle.

### Proof of Concept

1. Contract is `Running` with participants A, B, C (threshold 2). All three have submitted valid Dstack attestations; `expiry_timestamp_seconds` for each is `T_submit + 86400` (1 day).
2. At time `T_submit + 86401` (one second past expiry), node A's attestation has lapsed. Node A has stopped resubmitting (it is now running a non-approved image). `verify_tee()` has not been called.
3. A user calls `sign(payload)`. The contract emits a signing request.
4. Node A calls `respond(signature_share)`. The contract executes `assert_caller_is_attested_participant_and_protocol_active()` → `is_caller_an_attested_participant()`. The function finds A's entry in `stored_attestations`, confirms account/key ownership, and returns `Ok(())` — **no expiry check is performed**.
5. Node A's response is accepted. Combined with one honest node (B or C), the threshold is met and the signature is produced with participation from a node whose TEE guarantee has expired. [4](#0-3) [7](#0-6) [1](#0-0)

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
