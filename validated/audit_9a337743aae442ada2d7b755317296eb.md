### Title
Governance Threshold Used Instead of Per-Domain Reconstruction Threshold in Robust ECDSA Presign Computation â€” (`crates/node/src/providers/robust_ecdsa/presign.rs`)

### Summary

The `compute_thresholds` function and `run_presignature_generation_follower` in the robust ECDSA presign module both derive `MaxMalicious` from the **governance threshold** (`mpc_config.participants.threshold`) rather than the **per-domain reconstruction threshold** stored in `DomainConfig.reconstruction_threshold`. Because the contract now enforces `GovernanceThreshold >= max(ReconstructionThreshold)` as a valid invariant, these two values can legitimately differ. When they do, the presign phase uses a semantically wrong threshold, producing a mismatch with the threshold used during key generation, which breaks the signing flow for all requests routed to that domain.

---

### Finding Description

The contract stores two distinct threshold concepts:

- **`Threshold` (governance)** â€” how many participants must vote to approve governance actions. Enforced to be `>= max(ReconstructionThreshold)` across all domains.
- **`ReconstructionThreshold` (per-domain)** â€” the cryptographic `t` in a `t-of-n` scheme; the minimum shares needed to reconstruct the secret for a specific domain. [1](#0-0) 

The contract explicitly validates that `GovernanceThreshold >= max(ReconstructionThreshold)`, making configurations where they differ both valid and expected: [2](#0-1) 

Key generation for `RobustEcdsa` correctly receives a `ReconstructionThreshold` and translates it: [3](#0-2) 

However, the presign leader path (`compute_thresholds`) uses the **governance threshold** instead: [4](#0-3) 

The TODO comment in the code explicitly acknowledges this is wrong:

> `TODO(#3164): once the node supports per-domain thresholds, this should take the domain-specific threshold instead of the single governance threshold.`

The presign follower path has the same defect â€” it reads `self.mpc_config.participants.threshold` (governance) rather than the domain's reconstruction threshold: [5](#0-4) 

The `translate_threshold` function maps the input threshold `t` to `MaxMalicious = (t - 1) / 2`, and the presign protocol then enforces that exactly `2 * MaxMalicious + 1` participants are used: [6](#0-5) [7](#0-6) 

**Concrete mismatch scenario:**

| Step | Threshold used | `MaxMalicious` | Signers required |
|---|---|---|---|
| Key generation | `t_recon = 5`, `n = 7` | `(5-1)/2 = 2` | `2*2+1 = 5` |
| Presign (leader + follower) | `t_gov = 7`, `n = 7` | `(7-1)/2 = 3` | `2*3+1 = 7` |

The key was generated under a 5-of-7 scheme (Lagrange coefficients for 5 participants), but presigning assembles 7 participants and applies Lagrange interpolation for a 7-of-7 scheme. The resulting presignatures are cryptographically inconsistent with the key shares, producing invalid signatures.

---

### Impact Explanation

When the governance threshold exceeds the per-domain reconstruction threshold for a `DamgardEtAl` domain â€” a configuration the contract explicitly permits and validates â€” every presignature generated for that domain is computed with the wrong `MaxMalicious`. The presign protocol enforces `participants.len() == 2*MaxMalicious+1`, so the participant set assembled for presigning differs from the one used during key generation. Lagrange coefficients computed over the wrong participant set produce presignatures that are inconsistent with the distributed key, causing all subsequent signing operations on that domain to produce invalid signatures. This permanently breaks the signing flow for the affected domain until the node software is corrected, freezing all cross-chain transaction capability routed through it.

This matches the allowed impact: **Medium â€” request-lifecycle manipulation that breaks production safety/accounting invariants**.

---

### Likelihood Explanation

The contract's `validate_governance_against_reconstruction` enforces `GovernanceThreshold >= max(ReconstructionThreshold)`, making a divergence between the two values a **normal, expected, and validated configuration** â€” not a misconfiguration. Any operator who votes to add a `DamgardEtAl` domain with a reconstruction threshold lower than the current governance threshold (e.g., governance = 7, reconstruction = 5 for a 7-participant network) will silently trigger this mismatch. The TODO comment in the production code confirms the developers are aware the wrong value is being used. As `DamgardEtAl` moves from "Ongoing" to deployed, this becomes immediately reachable.

---

### Recommendation

In `compute_thresholds` and `run_presignature_generation_follower`, replace the governance threshold source with the per-domain `reconstruction_threshold` from `DomainConfig`:

```rust
// Instead of:
let threshold = self.mpc_config.participants.threshold.try_into()?;

// Use the domain-specific reconstruction threshold:
let domain_config = self.get_domain_config(domain_id)?;
let threshold: usize = domain_config.reconstruction_threshold.inner().try_into()?;
```

This aligns with the design document's proposed clean solution: [8](#0-7) 

---

### Proof of Concept

1. Deploy contract with 7 participants, governance threshold = 7, and a `DamgardEtAl` domain with `reconstruction_threshold = 5` (valid: `2*5-1=9 > 7` fails, so use `t=4`: `2*4-1=7 <= 7`, governance=7 >= reconstruction=4).
2. Key generation runs: `translate_threshold(4, 7)` â†’ `MaxMalicious(1)` â†’ `2*1+1 = 3` signers. Key shares are 3-of-7.
3. Presign runs: `compute_thresholds(governance=7, participants=7)` â†’ `translate_threshold(7, 7)` â†’ `MaxMalicious(3)` â†’ `2*3+1 = 7` signers required.
4. Presign assembles 7 participants and computes Lagrange coefficients for a 7-of-7 scheme over key shares designed for a 3-of-7 scheme.
5. Resulting presignatures are cryptographically invalid; all `sign()` requests on this domain produce invalid signatures or fail outright. [4](#0-3) [5](#0-4)

    /// Validates the GovernanceThreshold `k` against both the participant count and the
    /// largest ReconstructionThreshold across all domains. Layers the cross-domain rule
    /// `GovernanceThreshold >= max(ReconstructionThreshold)` on top of `validate_threshold`:
    /// the network must never be able to govern with fewer parties than are required to
    /// reconstruct any domain's key. Call this at every point where the GovernanceThreshold,
    /// a ReconstructionThreshold, or the participant set changes.
    pub fn validate_governance_against_reconstruction(
        num_participants: u64,
        governance: Threshold,
        max_reconstruction_threshold: Option<ReconstructionThreshold>,
    ) -> Result<(), Error> {
        Self::validate_threshold(num_participants, governance)?;
        if let Some(max_reconstruction_threshold) = max_reconstruction_threshold
            && governance.value() < max_reconstruction_threshold.inner()
        {
            return Err(InvalidThreshold::BelowReconstructionThreshold {
                reconstruction_threshold: max_reconstruction_threshold.inner(),
                governance_threshold: governance.value(),
            }
            .into());
        }
        Ok(())
    }

/// Computes `(num_signers, robust_ecdsa_threshold)` and validates the
/// `2 * max_malicious + 1 <= num_signers` invariant. Returns an error only if
/// the configured governance threshold is invalid for robust-ECDSA.
///
/// TODO(#3164): once the node supports per-domain thresholds, this should
/// take the domain-specific threshold instead of the single governance threshold.
fn compute_thresholds(
    governance_threshold: u64,
    num_running_participants: usize,
) -> anyhow::Result<(usize, MaxMalicious)> {
    let governance_threshold: usize = governance_threshold.try_into()?;
    let num_signers = get_number_of_signers(governance_threshold, num_running_participants)?;
    let robust_ecdsa_threshold =
        translate_threshold(governance_threshold, num_running_participants)?;
    anyhow::ensure!(
        robust_ecdsa_threshold
            .value()
            .checked_mul(2)
            .and_then(|v| v.checked_add(1))
            .is_some_and(|v| v <= num_signers)
    );
    Ok((num_signers, robust_ecdsa_threshold))
}
