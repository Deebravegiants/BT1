### Title
Missing On-Chain Cryptographic Verification of CKD Response for `AppPublicKey` Variant Allows Single Attested Participant to Deliver Forged Confidential Key - (File: crates/contract/src/lib.rs)

### Summary

The `respond_ckd` method in `MpcContract` performs a cryptographic output check only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`. For the legacy `AppPublicKey` variant, the check is entirely absent. Any single attested participant can call `respond_ckd` with an arbitrary, attacker-chosen `CKDResponse` (`big_y`, `big_c`) and the contract will accept it, resolve the pending yield, and deliver the forged confidential key to the requesting user — without any threshold-level authorization.

### Finding Description

In `respond_ckd`, after verifying the caller is an attested participant, the contract branches on the app public key type:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` — a cryptographic proof that the output was computed using the MPC network's master key. [2](#0-1) 

For `AppPublicKey`, the empty match arm means the contract performs **zero cryptographic verification** of the response. The `pending_requests::resolve_yields_for` call then immediately delivers whatever `big_y` and `big_c` the caller supplied to the waiting user. [3](#0-2) 

By contrast, `respond` (ECDSA/EdDSA) always verifies the signature against the stored public key before resolving the yield, making forgery impossible there. [4](#0-3) 

The `AppPublicKey` variant is explicitly documented as still accepted in production ("a plain G1 point string (privately verifiable, legacy)"), so this code path is live. [5](#0-4) 

### Impact Explanation

A single attested participant (strictly below the signing threshold) can:

1. Monitor the contract for a pending `CKDRequest` using the `AppPublicKey` variant.
2. Construct an arbitrary `CKDResponse` — for example, one where the attacker knows the discrete log of `big_y`, meaning they can later decrypt any data the user encrypts to the derived key.
3. Call `respond_ckd` before the legitimate MPC nodes respond.
4. The contract resolves the yield and delivers the forged confidential key to the user.

The user receives a "confidential key" that is not derived from the MPC network's master key and whose secret value is known to the attacker. Any data the user encrypts to this key is immediately readable by the attacker. This constitutes **confidential key derivation output without the required participant authorization** — a Critical impact under the allowed scope.

### Likelihood Explanation

- The `AppPublicKey` variant is still accepted by the production contract.
- The attacker only needs to be one attested participant — a role achievable by any node that has submitted a valid TEE attestation.
- The attacker can race the legitimate MPC nodes by monitoring `pending_ckd_requests` via the view method `get_pending_ckd_request` and submitting a forged response in the same or next block.
- No threshold collusion is required; a single Byzantine participant below the threshold suffices.

### Recommendation

Apply the same `ckd_output_check` to the `AppPublicKey` variant, or remove the `AppPublicKey` variant from `respond_ckd` entirely and require all new CKD requests to use `AppPublicKeyPV`. If backward compatibility must be preserved, at minimum document that `AppPublicKey` responses carry no on-chain integrity guarantee and gate the variant behind a feature flag or deprecation warning.

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {
        // AppPublicKey provides no on-chain verifiability;
        // either reject or apply an equivalent check.
        env::panic_str("AppPublicKey variant is not supported for respond_ckd; use AppPublicKeyPV");
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

1. Deploy the contract in `Running` state with a BLS12-381 CKD domain.
2. User calls `request_app_private_key` with `AppPublicKey(some_g1_point)` and 1 yoctoNEAR deposit. A pending yield is registered under `pending_ckd_requests`.
3. Attacker (any single attested participant) constructs a `CKDResponse` with arbitrary `big_y = G1::generator()` and `big_c = G1::generator()` (or any values where the attacker knows the relationship).
4. Attacker calls `respond_ckd(ckd_request, forged_response)` from their attested account.
5. The contract passes the `AppPublicKey` branch with no check, calls `resolve_yields_for`, and the user's yield resolves with the forged `CKDResponse`.
6. The user receives `{big_y: G1::generator(), big_c: G1::generator()}` — a key entirely controlled by the attacker — with no on-chain indication that anything is wrong.

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` confirms that arbitrary `big_y = [1u8;48]` and `big_c = [2u8;48]` values are accepted without error for the `AppPublicKey` variant, demonstrating the absence of any output check. [6](#0-5)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
