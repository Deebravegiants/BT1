### Title
Single Byzantine Participant Can Forge CKD Output for `AppPublicKey` Requests via Missing On-Chain Verification — (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` function in `mpc-contract` applies an on-chain output integrity check (`ckd_output_check`) only when the user's request uses the `AppPublicKeyPV` variant. When the user uses the `AppPublicKey` variant, the contract performs **no output verification at all** and accepts any response. A single Byzantine attested participant can race to call `respond_ckd` with a forged CKD response — delivering a key they know to the user — bypassing the threshold security guarantee with a single-participant attack.

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function handles CKD (Confidential Key Derivation) responses submitted by MPC nodes:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies that the CKD output is correctly computed against the domain public key and the user's key pair. For `AppPublicKey`, the arm is a no-op — the contract accepts **any** `CKDResponse` without verification.

The `respond_ckd` function is callable by any single attested participant: [2](#0-1) 

The first successful call drains the entire pending yield queue via `resolve_yields_for`, delivering the response to all waiting callers: [3](#0-2) 

**Attack path:**

1. User calls `request_app_private_key` with the `AppPublicKey` variant (a single BLS G1 point — currently supported, described as "privately verifiable, legacy" in the README).
2. Byzantine attested participant observes the pending request and the user's `app_public_key` (public on-chain).
3. Byzantine participant generates a fake key `K_fake` and encrypts it to the user's `app_public_key`.
4. Byzantine participant calls `respond_ckd` with the forged `CKDResponse` before honest nodes submit the correct result.
5. The contract passes `assert_caller_is_attested_participant_and_protocol_active()`, finds no output check for `AppPublicKey`, and calls `resolve_yields_for` — delivering the forged response to the user.
6. The user decrypts the response and receives `K_fake`, a key the attacker knows.

The root cause is structurally identical to the external report: the system checks one variant (`AppPublicKeyPV`) but leaves the other (`AppPublicKey`) entirely unchecked, creating a bypassable integrity guard. A user's choice of variant determines whether the threshold guarantee holds. [4](#0-3) 

### Impact Explanation

A single Byzantine participant (below the signing threshold) can forge a CKD output for any pending `AppPublicKey` request. The attacker delivers a key they know, encrypted to the user's `app_public_key`. The user receives and uses `K_fake` believing it is their legitimately derived confidential key. If `K_fake` is used as a signing key for a foreign chain, the attacker can sign transactions on the user's behalf; if used as an encryption key, the attacker can decrypt the user's data. This is a direct bypass of the threshold-signature requirement: forging a CKD output should require threshold-many participants to collude, but here a single participant suffices.

**Impact category:** Critical — unauthorized access to secret material / bypass of threshold-signature requirements.

### Likelihood Explanation

The `AppPublicKey` variant is a currently accepted, non-deprecated input path (the README describes it as "privately verifiable, legacy" but still lists it as a valid option). Any user who submits a `request_app_private_key` with `AppPublicKey` is vulnerable. The attacker must:

1. Be an attested participant (requires joining the MPC network with a valid TEE attestation — a realistic adversary profile per the scope).
2. Submit `respond_ckd` before honest nodes. Because the Byzantine participant is also running the MPC protocol, they know when the computation completes and can submit immediately, racing honest nodes. Low-latency access to the NEAR RPC makes this race winnable.

**Likelihood:** Medium-High for a motivated Byzantine participant.

### Recommendation

1. **Preferred:** Require `AppPublicKeyPV` for all new requests and reject `AppPublicKey` submissions. The `AppPublicKeyPV` path already has a correct on-chain integrity check.
2. **Alternative:** If `AppPublicKey` must be retained, add a threshold-response mechanism: require at least `t` identical `respond_ckd` calls before the response is delivered, so a single Byzantine participant cannot unilaterally determine the output.
3. **Minimum:** Document clearly that `AppPublicKey` requests carry no on-chain output guarantee and are vulnerable to a single malicious participant, so users can make an informed choice.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
