Audit Report

## Title
ETH Deposit Limit Check Ignores Deposit Amount, Allowing Limit Bypass - (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses a strict `>` comparison without adding the incoming `amount` for ETH deposits, while the ERC-20 branch correctly uses `totalAssetDeposits + amount >`. When `totalAssetDeposits` equals the configured limit exactly, the ETH check returns `false` and the deposit proceeds, pushing total ETH held above the governance-approved cap.

## Finding Description
At [1](#0-0)  the function branches on asset type. The ERC-20 path at line 681 adds `amount` before comparing against the limit. The ETH path at line 679 does not — it only checks whether the current total already exceeds the limit, never whether the incoming deposit would push it over.

The public entry point `depositETH` at [2](#0-1)  is permissionless and calls `_beforeDeposit` at [3](#0-2) , which reverts only when `_checkIfDepositAmountExceedesCurrentLimit` returns `true`. Because the ETH branch never returns `true` at the exact limit boundary, the revert never fires and `_mintRsETH` executes for the full deposit amount.

## Impact Explanation
The ETH deposit cap is the protocol's primary risk-management gate for native ETH exposure. Bypassing it silently allows TVL to grow beyond the governance-approved ceiling with no on-chain revert or event. This matches **Low — contract fails to deliver promised returns, but doesn't lose value**: the cap guarantee is broken, but no funds are directly stolen or frozen.

## Likelihood Explanation
The exploitable condition (`totalAssetDeposits == depositLimit`) is reached transiently on every deposit that fills the pool to the limit. No special role, prior setup, or privileged access is required. Any unprivileged depositor who observes or races to that moment can submit an arbitrarily large ETH deposit through the public `depositETH` function.

## Recommendation
Apply the same prospective check used for ERC-20 assets:

```diff
 if (asset == LRTConstants.ETH_TOKEN) {
-    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
 }
```

## Proof of Concept
1. Governance sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Legitimate deposits bring `getTotalAssetDeposits(ETH_TOKEN)` to exactly `1000 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → proceeds.
5. `_beforeDeposit` does not revert; `_mintRsETH` mints rsETH for 500 ETH.
6. `getTotalAssetDeposits(ETH_TOKEN)` is now `1500 ether`, 50% above the approved cap, with no revert or alert.

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
