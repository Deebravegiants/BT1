### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to `request` Content, Enabling Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` function verifies that the submitted ECDSA signature is valid over `response.payload_hash`, but it never verifies that `response.payload_hash` is actually derived from the `request` content (i.e., `SHA256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: ... }))`). A malicious attested participant who previously led a signing session for the same `VerifyForeignTransactionRequest` key can replay the old threshold signature against a new pending request, delivering stale or incorrect extracted values to the caller.

### Finding Description

When a user calls `verify_foreign_transaction`, the contract stores the request under a content-keyed map (`pending_verify_foreign_tx_requests`) using the full `VerifyForeignTransactionRequest` struct as the key: [1](#0-0) 

The `VerifyForeignTransactionRequest` key is: [2](#0-1) 

When an attested participant calls `respond_verify_foreign_tx`, the contract performs two checks:

1. The ECDSA signature is valid over `response.payload_hash` under the root public key.
2. The `request` matches a pending entry in `pending_verify_foreign_tx_requests`. [3](#0-2) 

**What is missing**: the contract never checks that `response.payload_hash` equals `SHA256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: <observed_values> }))`. The `payload_hash` is accepted as-is from the caller, as long as the signature over it is valid.

The canonical payload structure that nodes are supposed to sign is: [4](#0-3) 

The `payload_hash` is supposed to commit to both the `request` (tx_id, extractors, finality) **and** the `values` (extracted observations from the foreign chain). The contract only verifies the signature over whatever `payload_hash` is supplied; it does not re-derive the expected hash from `request`.

Any attested participant (not just the elected leader) can call `respond_verify_foreign_tx`: [5](#0-4) 

There is no check that the caller is the elected leader for this specific request.

### Impact Explanation

**Forged foreign-chain verification / invalid bridge execution (High).**

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow, where a NEAR bridge contract must verify that a foreign-chain transaction finalized before releasing funds: [6](#0-5) 

A malicious attested participant who replays a stale `payload_hash` causes the bridge contract to receive a valid MPC signature attesting to **old or incorrect extracted values** (e.g., a block hash from a re-orged chain state). If the bridge contract uses this signature to authorize a fund release, it may release funds based on a transaction that is no longer canonical on the foreign chain, enabling a double-spend or invalid bridge execution.

The SDK's `ForeignChainSignatureVerifier::verify_signature` does check `payload_hash` against expected values: [7](#0-6) 

However, this is an **application-level** check that bridge contracts must opt into. The protocol-level contract does not enforce it, leaving any bridge contract that does not independently reconstruct the expected `payload_hash` vulnerable.

### Likelihood Explanation

**Low-medium.** The attack requires:

1. A malicious attested participant who was previously the elected leader for a signing session involving the same `VerifyForeignTransactionRequest` key (same `tx_id`, `extractors`, `domain_id`, `payload_version`). As leader, they receive the final threshold signature and can store it.
2. A foreign-chain re-org or state change that alters the extracted values for the same `tx_id` (common for chains with probabilistic finality, e.g., Ethereum with `Latest` finality, Bitcoin with low confirmation counts).
3. A new pending request submitted under the same `VerifyForeignTransactionRequest` key (e.g., user retries after timeout, or a different user submits the same request).

Conditions 2 and 3 are realistic for bridge use cases. Condition 1 is the most restrictive but is achievable by any participant who has ever led a session for the same request type.

### Recommendation

Add a protocol-level binding between `response.payload_hash` and `request` in `respond_verify_foreign_tx`. Two approaches:

1. **Include extracted values in the response**: Require the MPC node to submit the `Vec<ExtractedValue>` alongside the signature. The contract re-derives `payload_hash = SHA256(borsh(ForeignTxSignPayload::V1 { request: request.request, values }))` and verifies it matches `response.payload_hash` before accepting.

2. **Store the expected payload hash at request time**: This is not feasible since the contract does not know the extracted values at submission time.

3. **Enforce application-level verification**: Document clearly that callers of `verify_foreign_transaction` MUST use `ForeignChainSignatureVerifier::verify_signature` from the SDK and must supply the expected extracted values. Add an on-chain mechanism (e.g., a required `expected_payload_hash` parameter) to make this binding explicit.

### Proof of Concept

**Setup**: Bitcoin tx_id=X is pending verification. The MPC network has previously signed `payload_hash_A = SHA256(borsh(ForeignTxSignPayload::V1 { request: {tx_id=X, extractors=[BlockHash], ...}, values: [BlockHash(block_A)] }))` and produced threshold signature `sig_A`. Malicious attested participant `M` was the leader and stored `(payload_hash_A, sig_A)`.

**Attack**:

1. Bitcoin re-orgs. tx_id=X is now in block `block_B ≠ block_A`.
2. User (bridge contract) submits `verify_foreign_transaction` for tx_id=X again (same `VerifyForeignTransactionRequest` key). The pending yield is queued.
3. `M` calls `respond_verify_foreign_tx(request={tx_id=X, ...}, response={payload_hash: payload_hash_A, signature: sig_A})`.
4. Contract checks:
   - `sig_A` is valid over `payload_hash_A` under root public key → **TRUE** (signature is genuine).
   - `{tx_id=X, ...}` is in `pending_verify_foreign_tx_requests` → **TRUE** (new request is pending).
   - `payload_hash_A` corresponds to `request` → **NOT CHECKED**.
5. Contract resolves the pending yield with `{payload_hash: payload_hash_A, signature: sig_A}`.
6. Bridge contract receives a valid MPC signature attesting to `block_A` (the re-orged, non-canonical block) instead of `block_B`.
7. If the bridge contract does not independently verify `payload_hash_A` against its expected `block_B`, it may authorize a fund release based on a stale/invalid chain state. [3](#0-2) [8](#0-7)

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
