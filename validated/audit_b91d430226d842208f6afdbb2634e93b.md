Audit Report

## Title
`RsETHTokenWrapper` 1:1 cross-token mint/burn with no per-token accounting enables direct fund theft when multiple altRsETH tokens are active - (File: contracts/L2/RsETHTokenWrapper.sol)

## Summary
`RsETHTokenWrapper` supports multiple allowed altRsETH tokens via `allowedTokens` and mints/burns `wrsETH` at a strict 1:1 ratio with any of them, with no exchange-rate check and no restriction on which token is chosen at withdrawal. Once a second token is added via `reinitialize` or `addAllowedToken`, any user can deposit the cheaper token, receive wrsETH 1:1, then immediately withdraw the more valuable token 1:1, extracting the price difference from other depositors. The `maxAmountToDepositBridgerAsset` function compounds the issue by comparing the full `totalSupply()` against only one token's balance, inflating the bridger deposit ceiling.

## Finding Description
`_deposit` mints exactly `_amount` wrsETH for exactly `_amount` of any allowed altRsETH with no exchange rate consulted: [1](#0-0) 

`_withdraw` burns exactly `_amount` wrsETH and transfers exactly `_amount` of any caller-chosen allowed altRsETH, again with no rate check: [2](#0-1) 

The public `withdraw` and `withdrawTo` functions impose no restriction on which allowed asset is selected: [3](#0-2) 

The contract explicitly provisions for multiple tokens. `reinitialize` (DEFAULT_ADMIN_ROLE) adds a second token as a documented upgrade path: [4](#0-3) 

`addAllowedToken` (TIMELOCK_ROLE) can add further tokens: [5](#0-4) 

Once two tokens are active, the wrapper becomes a zero-cost swap venue. `maxAmountToDepositBridgerAsset` uses `totalSupply()` (all wrsETH minted across all tokens) minus the balance of only one specific token, producing an inflated ceiling: [6](#0-5) 

The identical pattern exists in `AGETHTokenWrapper`: [7](#0-6) [8](#0-7) [9](#0-8) 

No existing guard prevents cross-token withdrawal. The only check in `_withdraw` is `allowedTokens[_asset]`, which passes for any registered token.

## Impact Explanation
**Critical — direct theft of user funds.** Any unprivileged user can deposit the cheaper altRsETH token, receive wrsETH 1:1, then call `withdraw` specifying the more valuable altRsETH token and receive it 1:1. The value difference is extracted directly from other depositors who contributed the more valuable token. No special role, flash loan, or oracle manipulation is required — only a standard `deposit` followed by a `withdraw` with a different asset address.

## Likelihood Explanation
**Medium.** The precondition is that a second allowed token is active, which requires a normal governance action (`reinitialize` by DEFAULT_ADMIN_ROLE or `addAllowedToken` by TIMELOCK_ROLE). This is the documented upgrade path, not an attack. Once a second token is live, different bridge representations of rsETH (e.g., LayerZero OFT vs. native bridge) routinely trade at small but non-zero discounts/premiums on secondary markets, providing the price differential needed to profit. The exploit is immediately available to any user with no setup beyond holding the cheaper token.

## Recommendation
1. **Per-token accounting**: maintain a `mapping(address => uint256) public wrsETHMintedFor` incremented in `_deposit` and decremented in `_withdraw`. Use this per-token balance (not `totalSupply()`) in `maxAmountToDepositBridgerAsset`.
2. **Enforce same-token withdrawal**: require that wrsETH can only be redeemed for the same altRsETH token it was minted against (track deposit token per user or per wrsETH unit), or introduce an oracle-based exchange rate between allowed tokens before permitting cross-token redemption.
3. Apply the same fix to `AGETHTokenWrapper`.

## Proof of Concept
**Setup**: Admin calls `reinitialize(altRsETH_B)` to add a second allowed token alongside the original `altRsETH_A`. `altRsETH_B` trades at 1.05× the value of `altRsETH_A` (realistic bridge premium).

```
// Step 1 — Alice deposits the valuable token
Alice.deposit(altRsETH_B, 100e18)
  → wrapper.balanceOf(altRsETH_B) = 100e18
  → Alice receives 100e18 wrsETH

// Step 2 — Bob deposits the cheap token
Bob.deposit(altRsETH_A, 100e18)
  → wrapper.balanceOf(altRsETH_A) = 100e18
  → Bob receives 100e18 wrsETH

// Step 3 — Bob withdraws the valuable token
Bob.withdraw(altRsETH_B, 100e18)   // freely chooses altRsETH_B
  → burns 100e18 wrsETH
  → Bob receives 100e18 altRsETH_B (worth 105e18 altRsETH_A equivalent)

// Result: Bob spent 100e18 altRsETH_A, received 100e18 altRsETH_B
// Net profit: ~5 ETH-equivalent
// Alice's 100e18 altRsETH_B is gone; wrapper holds only 100e18 altRsETH_A
// against Alice's 100e18 wrsETH — she cannot redeem at fair value
```

**Foundry invariant test**: assert `sum(wrsETHMintedFor[token] for all tokens) == totalSupply()` and `wrsETHMintedFor[token] <= token.balanceOf(wrapper)` for each token after any sequence of `deposit`/`withdraw` calls across two allowed tokens. The second invariant breaks immediately after Step 3 above.

```text
    function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
        if (!allowedTokens[_asset]) return 0;

        // get totalSupply of wrsETH minted
        uint256 wrsETHSupply = totalSupply();
        // balance of _asset with the contract
        uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));

        if (balanceOfAssetInWrapper > wrsETHSupply) return 0;

        return wrsETHSupply - balanceOfAssetInWrapper;
    }
```
