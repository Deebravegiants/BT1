### Title
Hardcoded 18-Decimal Assumption in `viewSwapRsETHAmountAndFee` Causes Near-Total Loss of Deposited Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

All L2 pool contracts compute the rsETH mint amount for ERC20 token deposits using a formula that silently assumes the deposited token has 18 decimals. The `addSupportedToken` function imposes no decimal restriction. If a token with fewer than 18 decimals (e.g., USDC with 6) is ever added, every depositor of that token receives a near-zero rsETH amount while their full token balance is permanently locked in the pool.

---

### Finding Description

In `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)`:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

`tokenToETHRate` is fetched from `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which normalises the Chainlink answer to a 1e18-scaled price representing the value of **one full token** in ETH:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [2](#0-1) 

For an 18-decimal token (e.g., wstETH), `amount` is also in 1e18 units, so the multiplication is dimensionally consistent and the result is correct.

For a 6-decimal token (e.g., USDC), `amount` is in 1e6 units while `tokenToETHRate` still represents the price of one full USDC (= 1e6 raw units) in ETH. The product `amountAfterFee * tokenToETHRate` is therefore 10^12 times smaller than the equivalent ETH value, and the resulting `rsETHAmount` is 10^12 times smaller than it should be.

**Concrete example (USDC, 6 decimals, 1 USDC ≈ 0.00033 ETH, rsETH rate = 1.05 ETH):**

| Variable | Value |
|---|---|
| `amountAfterFee` | 1e6 (1 USDC) |
| `tokenToETHRate` | 3.3e14 (0.00033 ETH, normalised) |
| `rsETHToETHrate` | 1.05e18 |
| Computed `rsETHAmount` | ≈ 314 (≈ 3.14e-16 rsETH) |
| Correct `rsETHAmount` | ≈ 3.14e14 (≈ 0.000314 rsETH) |

The formula is off by exactly 10^(18 − tokenDecimals) = 10^12.

The same structural flaw is present in every pool variant: [3](#0-2) [4](#0-3) [5](#0-4) 

`addSupportedToken` in every pool performs no decimal check — only a non-zero oracle rate check: [6](#0-5) 

---

### Impact Explanation

A depositor who calls `deposit(token, amount, referralId)` with a supported non-18-decimal token:

1. Transfers their full token balance to the pool (`safeTransferFrom`).
2. Receives a `rsETHAmount` that rounds to essentially zero (for tokens with < 18 decimals).
3. Has no user-accessible path to recover the deposited tokens — `moveAssetsForBridging` and `swapAssetToPremintedRsETH` are both role-restricted.

The deposited tokens are effectively frozen in the pool from the user's perspective. This constitutes **temporary (and practically permanent) freezing of user funds**.

---

### Likelihood Explanation

**Low.** Triggering the bug requires `TIMELOCK_ROLE` to call `addSupportedToken` with a non-18-decimal token. The protocol currently supports only 18-decimal LSTs (wstETH, etc.). However, the code contains no guard preventing this, and the protocol's multi-chain expansion roadmap makes future addition of stablecoins or other non-18-decimal collateral plausible.

---

### Recommendation

1. In `addSupportedToken`, assert `IERC20Metadata(token).decimals() == 18`, or store the token's decimals and apply a normalisation factor in `viewSwapRsETHAmountAndFee`:

```solidity
rsETHAmount = amountAfterFee
    * tokenToETHRate
    * (10 ** (18 - tokenDecimals[token]))
    / rsETHToETHrate;
```

2. Apply the same fix to `viewSwapAssetToPremintedRsETH` (reverse direction) to prevent over-payment of tokens when the operator swap path is used.

---

### Proof of Concept

1. Admin calls `addSupportedToken(USDC, usdcOracle, bridge)` on `RSETHPoolV3` (no decimal check blocks this).
2. User calls `deposit(USDC, 1_000_000, "ref")` (1 USDC = 1e6 raw units).
3. `viewSwapRsETHAmountAndFee(1_000_000, USDC)` executes:
   - `tokenToETHRate` = 3.3e14 (Chainlink USDC/ETH, normalised)
   - `rsETHToETHrate` = 1.05e18
   - `rsETHAmount` = 1e6 × 3.3e14 / 1.05e18 = **314** (≈ 0 wrsETH minted)
4. User's 1 USDC is transferred to the pool; user receives 314 wei of wrsETH (≈ $0).
5. User has no function to reclaim the USDC — funds are frozen.

        UtilLib.checkNonZeroAddress(oracle);

        if (supportedTokenOracle[token] != address(0)) {
            revert AlreadySupportedToken();
        }
