### Title
Sub-Transaction ExUnits Excluded from Per-Transaction and Per-Block Limit Checks — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, `totExUnits` only sums the redeemer ExUnits of the **top-level transaction**. Sub-transactions embedded in a Dijkstra batch carry their own redeemers with their own ExUnits, but those values are never counted against `maxTxExUnits` (per-transaction limit) or `maxBlockExUnits` (per-block limit). The minimum-fee calculation has the same gap. An unprivileged sender can therefore submit a batch whose sub-transactions collectively declare and consume far more script execution budget than the protocol parameters permit, while paying a fee that covers only the top-level transaction's scripts.

---

### Finding Description

`totExUnits` is defined as:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It reads only the redeemers attached to the single `Tx` value it receives. [1](#0-0) 

The Dijkstra UTXO rule enforces the per-transaction limit by calling:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

where `tx` is the top-level transaction only. [2](#0-1) 

The Dijkstra BBODY rule enforces the per-block limit by calling:

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
```

which expands to `foldMap totExUnits txs` — again summing only top-level redeemers across all transactions in the block. [3](#0-2) 

The `SUBUTXO` transition rule (`dijkstraSubUtxoTransition`) contains no ExUnits check at all. The mapping function `dijkstraUtxoToDijkstraSubUtxoPredFailure` explicitly marks `ExUnitsTooBigUTxO` as impossible for sub-transactions:

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [4](#0-3) 

The minimum-fee calculation for Dijkstra delegates to `getConwayMinFeeTxUtxo`:

```haskell
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [5](#0-4) 

`getConwayMinFeeTxUtxo` ultimately calls `alonzoMinFeeTx`, which computes `txscriptfee (pp ^. ppPricesL) allExunits` where `allExunits = totExUnits tx`. [6](#0-5) 

Sub-transactions do not have their own fee field; the top-level transaction's fee covers the entire batch. Because `totExUnits` ignores sub-transaction redeemers, the minimum fee is computed as if those scripts do not exist.

By contrast, the reference-script size check **does** aggregate across sub-transactions via `batchNonDistinctRefScriptsSize`, demonstrating that the pattern for batch-aware accounting is known and applied selectively. [7](#0-6) 

---

### Impact Explanation

An attacker can craft a top-level Dijkstra transaction whose own redeemers declare zero or minimal ExUnits, while embedding N sub-transactions each declaring ExUnits just below `maxTxExUnits`. The combined sub-transaction ExUnits can be N × `maxTxExUnits`, far exceeding both the per-transaction and per-block protocol limits. Honest nodes accept the block because neither `validateExUnitsTooBigUTxO` nor `validateExUnits` sees the sub-transaction budgets. The fee paid covers only the top-level scripts, so the attacker also underpays for script execution relative to what the protocol parameters require.

This matches the allowed impact: **Medium — attacker-controlled transactions exceed intended validation limits and modify fees outside design parameters.**

---

### Likelihood Explanation

The Dijkstra era is the newest era in this repository and introduces nested transactions as a novel feature. The ExUnits accounting gap is a straightforward consequence of reusing `totExUnits` without extending it to cover sub-transactions. Any user who can submit a valid Dijkstra transaction can exploit this; no privileged access, key compromise, or governance majority is required. The attack is deterministic and reproducible.

---

### Recommendation

Extend `totExUnits` (or introduce a Dijkstra-specific variant) to recursively sum ExUnits from all sub-transactions, mirroring the approach already used for reference-script size accounting in `batchNonDistinctRefScriptsSize`:

```haskell
dijkstraTotExUnits :: (AlonzoEraTxWits era, DijkstraEraTxBody era) => Tx TopTx era -> ExUnits
dijkstraTotExUnits tx =
  totExUnits tx
    <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
```

Apply this function in:
1. `validateExUnitsTooBigUTxO` within `dijkstraUtxoTransition` (replacing the bare `Alonzo.validateExUnitsTooBigUTxO pp tx` call).
2. `validateExUnits` within `dijkstraBbodyTransition` (replacing `foldMap totExUnits txs`).
3. The minimum-fee calculation for Dijkstra (override `getMinFeeTxUtxo` to use the batch-aware variant).

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction with zero redeemers (top-level ExUnits = 0).
2. Embed K sub-transactions, each with a Plutus script redeemer declaring `ExUnits { exUnitsMem = M, exUnitsSteps = S }` where `ExUnits M S` equals `maxTxExUnits`.
3. Submit the transaction. `validateExUnitsTooBigUTxO` checks `totExUnits topTx = ExUnits 0 0 ≤ maxTxExUnits` — passes. `validateExUnits` at the block level sums `totExUnits` over all top-level transactions, again seeing only zero — passes.
4. The ledger accepts the block. Total script budget consumed = K × `maxTxExUnits`, which can be made arbitrarily larger than `maxBlockExUnits` by choosing K. The fee paid covers zero ExUnits of script execution. [8](#0-7) [9](#0-8) [10](#0-9)

validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) $
