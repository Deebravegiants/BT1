Audit Report

## Title
`SwapAllowlistExtension` grants unrestricted swap access to all router users when router is allowlisted — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`MetricOmmPool.swap()` always passes `msg.sender` (the direct pool caller) as `sender` to extension hooks. When a user routes through `MetricOmmSimpleRouter`, `msg.sender` at the pool is the router contract. `SwapAllowlistExtension.beforeSwap()` evaluates `allowedSwapper[pool][router]` instead of the actual end-user's address, so any user who calls through an allowlisted router bypasses per-user swap restrictions on pools configured with this extension.

## Finding Description
`MetricOmmPool.swap()` unconditionally passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap()` checks this `sender` value against the per-pool allowlist: [2](#0-1) 

`MetricOmmSimpleRouter.exactInputSingle()` calls `pool.swap()` directly — the router is `msg.sender` at the pool. The actual end-user's address is stored only in transient callback context for payment purposes and is never forwarded to the pool as `sender`: [3](#0-2) 

The `IMetricOmmExtensions.beforeSwap` interface provides no `originator` or `payer` field, so extensions have no on-chain mechanism to recover the actual end-user when the direct caller is a router: [4](#0-3) 

The same pattern applies to `exactInput`, `exactOutputSingle`, and `exactOutput` — all call `pool.swap()` with the router as `msg.sender`. [5](#0-4) 

## Impact Explanation
A pool configured with `SwapAllowlistExtension` and `allowAllSwappers = false` is intended to restrict swaps to a curated set of addresses. If the pool admin allowlists `MetricOmmSimpleRouter` — a natural operational step to enable standard UI access — every address on-chain can bypass the per-user restriction by routing through it. This breaks the access-control invariant the pool admin established, exposes LP principal to unauthorized toxic flow from arbitrageurs or other excluded addresses, and constitutes a direct loss of LP assets above Sherlock thresholds.

## Likelihood Explanation
`SwapAllowlistExtension` is a first-party periphery contract deployed alongside the protocol. Pool admins routinely allowlist the canonical router to enable normal user access; no documentation warns that doing so grants unrestricted access to all users. The exploit requires a single `exactInputSingle` call and is permissionlessly repeatable by any address.

## Recommendation
1. **Add an `originator` field** to `beforeSwap`/`afterSwap` that the router populates (e.g., via `callbackData` or a dedicated transient slot) and that the pool forwards to extensions alongside `sender`.
2. **Alternatively, document the limitation clearly**: state that `sender` is always the direct pool caller and that `SwapAllowlistExtension` gates by contract, not by end-user. Warn admins that allowlisting the router grants access to all router users.
3. **Short-term mitigation**: pool admins should not allowlist `MetricOmmSimpleRouter` on restricted pools; instead, require users to call the pool directly or deploy a purpose-built allowlisted wrapper.

## Proof of Concept
```
Setup:
  pool = MetricOmmPool with SwapAllowlistExtension, allowAllSwappers = false
  allowedSwapper[pool][router] = true   // admin allowlists the router
  allowedSwapper[pool][alice]  = false  // alice is NOT individually allowed

Attack:
  alice calls router.exactInputSingle({pool: pool, ...})
    → pool.swap(recipient, ...) with msg.sender = router
    → _beforeSwap(sender=router, ...)
    → SwapAllowlistExtension checks allowedSwapper[pool][router] == true → PASSES
    → alice's swap executes against the restricted pool

Result:
  alice, explicitly excluded by the pool admin, successfully swaps.
  The allowlist invariant is broken; LP funds are exposed to unauthorized flow.
```
