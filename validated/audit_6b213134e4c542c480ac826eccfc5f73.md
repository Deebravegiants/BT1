### Title
Zero Tweak Hardcoded in `build_signature_request` Causes All Foreign-Tx Verifications to Sign Under the Root MPC Key — (`File: crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

`build_signature_request` in the node's `verify_foreign_tx` provider hardcodes `tweak: Tweak::new([0u8; 32])` instead of deriving the tweak from the request's predecessor account and derivation path. Because the tweak is the sole mechanism for per-user key derivation in the MPC signing protocol, a zero tweak collapses every foreign-chain verification signature onto the root MPC public key. This is the direct NEAR-MPC analog of the Tokensoft bug where `new bytes(0)` was substituted for the real data parameter, silently corrupting every downstream consumer.

---

### Finding Description

The `build_signature_request` function constructs the internal `SignatureRequest` that the MPC nodes use to run the threshold ECDSA signing protocol for `verify_foreign_transaction` requests. [1](#0-0) 

The field `tweak: Tweak::new([0u8; 32])` is a 32-byte all-zero value. In the signing protocol, the tweak is consumed as:

```
derived_public_key = root_public_key + tweak * G
``` [2](#0-1) 

With `tweak = 0`, `derived_public_key = root_public_key + 0·G = root_public_key`. Every foreign-tx verification signature is therefore produced under the root MPC key, regardless of which NEAR account submitted the request or what derivation path was specified.

The correct behavior is described in the design document and in the analogous `SignatureRequest::from_chain` implementation for regular `sign()` requests, which properly calls `derive_tweak(&predecessor_id, &request.path)`: [3](#0-2) 

The design document explicitly specifies that `verify_foreign_transaction()` must use a **different** tweak derivation prefix from `sign()` so that the same `(predecessor_id, derivation_path)` pair never yields the same derived key across the two purposes: [4](#0-3) 

The design also specifies that `VerifyForeignTransactionRequest` (the resolved form stored in contract state) must carry a `tweak: Tweak` field derived internally by the contract from `request.derivation_path`. In the current implementation, neither `VerifyForeignTransactionRequestArgs` nor the node-side `VerifyForeignTxRequest` carries a `derivation_path` or `tweak` field: [5](#0-4) [6](#0-5) 

The zero tweak is the only value used in production; `grep` confirms it appears exactly once in the entire codebase, in this function.

---

### Impact Explanation

**Impact: High — forged foreign-chain verification / invalid bridge execution.**

1. **Key isolation is completely absent.** Every `verify_foreign_transaction` call, regardless of caller or derivation path, produces a signature verifiable against the root MPC public key. The design's per-user key isolation (different derived key per `(predecessor_id, path)`) is never applied.

2. **Bridge contracts that verify against derived keys will reject every signature.** A bridge contract that calls `derived_public_key(path, predecessor)` to obtain the expected verifying key will compute `root_key + tweak·G` with the correct non-zero tweak, but the MPC network signed with `root_key + 0·G = root_key`. The signature will fail ECDSA verification, permanently breaking the foreign-chain verification flow for all users.

3. **Bridge contracts that accept the root key receive signatures with no per-user binding.** If a bridge contract is written to verify against the root key (matching the current broken behavior), then the key isolation guarantee is gone: the same root key signs every user's foreign-tx attestation, eliminating the cryptographic separation between users that the derivation path is supposed to provide.

4. **The root MPC key is directly exposed through signatures.** The root key is the most sensitive material in the system. Producing ECDSA signatures under it for arbitrary user-submitted foreign-tx payloads is contrary to the security model, which requires all user-facing signing to go through key derivation.

---

### Likelihood Explanation

**Likelihood: High.**

- The `verify_foreign_transaction` contract method is fully implemented and present in the production ABI snapshot.
- Every single call to `verify_foreign_transaction` triggers `build_signature_request`, which unconditionally uses the zero tweak.
- No special attacker capability is required; any unprivileged NEAR account can call `verify_foreign_transaction` with a 1 yoctoNEAR deposit.
- The bug is structural (a hardcoded constant), not conditional, so it fires on 100% of requests.

---

### Recommendation

1. Add `derivation_path: String` to `VerifyForeignTransactionRequestArgs` and propagate it through `VerifyForeignTransactionRequest` (both the contract-interface DTO and the node-side `VerifyForeignTxRequest`).
2. In the contract's `verify_foreign_transaction`, derive the tweak using a foreign-tx-specific prefix (e.g., `"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`) and store it in the resolved request, mirroring how `sign()` calls `SignatureRequest::new(…, predecessor_id, path)`.
3. In `build_signature_request`, read the tweak from the `VerifyForeignTxRequest` instead of hardcoding `[0u8; 32]`.

```markdown
## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.
```
