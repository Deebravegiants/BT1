### Title
Wrong `WalletType` Returned After RCAT Wallet Conversion Corrupts Wallet Sync State — (`File: chia/wallet/wallet_state_manager.py`)

### Summary

In `WalletStateManager.handle_cat`, when a received coin is identified as an R-CAT (Revocable CAT) and an existing plain CAT wallet is successfully converted to an `RCATWallet`, the function returns a `WalletIdentifier` stamped with `WalletType.CRCAT` instead of `WalletType.RCAT`. This is a direct principal-type mix-up: the RCAT branch returns the CRCAT type identifier, mirroring the Illuminate/Yield swap described in the external report.

### Finding Description

In `chia/wallet/wallet_state_manager.py`, the `handle_cat` method detects the sub-type of an incoming CAT coin and, when necessary, upgrades an existing plain `CATWallet` to the correct specialised wallet. The relevant branching logic is:

```python
if wallet_type is CRCATWallet:
    ...
    return WalletIdentifier(wallet_info.id, WalletType(WalletType.CRCAT))   # correct
elif wallet_type is RCATWallet:
    success = await RCATWallet.convert_to_revocable(...)
    if success:
        self.state_changed("converted cat wallet to revocable", wallet_info.id)
        return WalletIdentifier(wallet_info.id, WalletType(WalletType.CRCAT))  # ← BUG: should be WalletType.RCAT
    else:
        return None
``` [1](#0-0) 

`RCATWallet.convert_to_revocable` correctly rewrites the wallet record in the DB and replaces the in-memory wallet object with an `RCATWallet` instance (type `WalletType.RCAT`). [2](#0-1) 

However, the `WalletIdentifier` returned to the caller carries `WalletType.CRCAT`, not `WalletType.RCAT`. Every downstream consumer of that identifier — coin-record storage, balance accounting, coin-type tagging — will treat the newly converted RCAT coin as a CRCAT coin.

For contrast, the CRCAT branch immediately above it returns the correct type: [3](#0-2) 

### Impact Explanation

The `WalletIdentifier` returned from `handle_cat` is consumed by `_add_coin_states` to route the coin into the correct wallet and store its metadata. With the wrong type (`CRCAT`), the coin is tagged and stored as a CRCAT coin. CRCAT coins carry `CRCATMetadata` (lineage proof + inner puzzle hash) and are expected to go through the CR-CAT authorization flow (VC proof, `claim_pending_approval_balance`, etc.). An RCAT coin stored under CRCAT logic will have no valid CRCAT metadata, making it unspendable through normal wallet paths and corrupting the wallet's balance and coin records for that asset. This matches the "Corruption of coin records … wallet sync state … with direct security impact" criterion.

### Likelihood Explanation

The trigger is fully unprivileged: any network participant can send an RCAT coin whose TAIL hash matches a TAIL for which the victim already holds a plain CAT wallet. The conversion path is exercised automatically during normal wallet sync whenever such a coin is observed on-chain. No key material, admin access, or cryptographic break is required.

### Recommendation

Change line 1321 to return `WalletType.RCAT`:

```python
elif wallet_type is RCATWallet:
    success = await RCATWallet.convert_to_revocable(
        found_cat_wallet,
        hidden_puzzle_hash=revocation_layer_match[0],
    )
    if success:
        self.state_changed("converted cat wallet to revocable", wallet_info.id)
-       return WalletIdentifier(wallet_info.id, WalletType(WalletType.CRCAT))
+       return WalletIdentifier(wallet_info.id, WalletType.RCAT)
    else:
        return None
```

### Proof of Concept

1. Victim has a plain `CATWallet` for TAIL hash `T`.
2. Attacker (or any sender) mints an RCAT coin with TAIL `T` and sends it to the victim's address.
3. Victim's node syncs; `_add_coin_states` calls `handle_cat`.
4. `handle_cat` detects the revocation layer, sets `wallet_type = RCATWallet`, calls `RCATWallet.convert_to_revocable` (succeeds), then returns `WalletIdentifier(id, WalletType.CRCAT)`.
5. The coin is stored with CRCAT coin-type and CRCAT metadata expectations. Subsequent spend attempts via `CRCATWallet._generate_unsigned_spendbundle` fail because the coin has no valid `CRCATMetadata`, and the RCAT-specific `RCATWallet.make_inner_solution` path is never reached.
6. The victim's RCAT balance is permanently inaccessible through the wallet. [4](#0-3) [5](#0-4) [6](#0-5)

    @classmethod
    async def convert_to_revocable(
        cls,
        cat_wallet: CATWallet,
        hidden_puzzle_hash: bytes32,
    ) -> bool:
        if not await cat_wallet.lineage_store.is_empty():
            cat_wallet.log.error("Received a revocable CAT to a CAT wallet that already has CATs")
            return False
        replace_self = cls()
        replace_self.standard_wallet = cat_wallet.standard_wallet
        replace_self.log = logging.getLogger(cat_wallet.get_name())
        replace_self.log.info(f"Converting CAT wallet {cat_wallet.id()} to R-CAT wallet")
        replace_self.wallet_state_manager = cat_wallet.wallet_state_manager
        replace_self.lineage_store = cat_wallet.lineage_store
        replace_self.cat_info = cls.wallet_info_type(
            cat_wallet.cat_info.limitations_program_hash, None, hidden_puzzle_hash
        )
        await cat_wallet.wallet_state_manager.user_store.update_wallet(
            WalletInfo(
                cat_wallet.id(), cat_wallet.get_name(), uint8(cls.wallet_type.value), bytes(replace_self.cat_info).hex()
            )
        )
        updated_wallet_info = await cat_wallet.wallet_state_manager.user_store.get_wallet_by_id(cat_wallet.id())
        assert updated_wallet_info is not None
        replace_self.wallet_info = updated_wallet_info

        cat_wallet.wallet_state_manager.wallets[cat_wallet.id()] = replace_self
        await cat_wallet.wallet_state_manager.puzzle_store.delete_wallet(cat_wallet.id())
        result = await cat_wallet.wallet_state_manager.create_more_puzzle_hashes()
        await result.commit(cat_wallet.wallet_state_manager)
        return True
