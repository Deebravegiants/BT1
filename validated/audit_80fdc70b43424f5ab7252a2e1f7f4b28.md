Audit Report

## Title
Stale Cached Rate Returned Without Staleness Check Enables Over-Minting of wrsETH - (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the cached `rate` unconditionally, with no check against `lastUpdated`. Because `updateRate()` on the L1 `MultiChainRateProvider` is a manual, ETH-paid call, the L2 cache can lag arbitrarily behind the true rsETH/ETH rate. Any depositor calling `RSETHPoolV3.deposit()` during a stale-low period receives more `wrsETH` than the current rsETH value warrants, diluting yield owed to existing holders.

## Finding Description

`CrossChainRateReceiver` stores the rsETH/ETH rate in `rate` and records the update timestamp in `lastUpdated`:

```solidity
// CrossChainRateReceiver.sol L13-16
uint256 public rate;
uint256 public lastUpdated;
```

`lzReceive` updates both fields when a LayerZero message arrives:

```solidity
// CrossChainRateReceiver.sol L93-97
uint256 _rate = abi.decode(_payload, (uint256));
rate = _rate;
lastUpdated = block.timestamp;
```

However, `getRate()` returns the cached value unconditionally — `lastUpdated` is stored but never consulted:

```solidity
// CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

`RSETHPoolV3.getRate()` delegates directly to `IOracle(rsETHOracle).getRate()`, where `rsETHOracle` is set to a `RSETHRateReceiver` (the concrete subclass of `CrossChainRateReceiver`). `viewSwapRsETHAmountAndFee` uses this rate to compute the minting amount:

```solidity
// RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

`deposit()` is callable by any unprivileged user and mints `wrsETH` directly based on this formula. The L1 `MultiChainRateProvider.updateRate()` is a manual, ETH-paid function with no on-chain enforcement of a call frequency:

```solidity
// MultiChainRateProvider.sol L108-113
function updateRate() external payable nonReentrant {
    uint256 latestRate = getLatestRate();
    rate = latestRate;
    lastUpdated = block.timestamp;
```

There is no staleness guard anywhere in the consumer path. `RSETHRateReceiver` adds no additional check over `CrossChainRateReceiver`.

**Exploit path:**
1. L1 `rsETHPrice` increases (e.g., after a reward distribution).
2. `updateRate()` has not been called; L2 `CrossChainRateReceiver.rate` holds the old, lower value.
3. Attacker calls `RSETHPoolV3.deposit{value: 1 ether}("")`.
4. Pool calls `IOracle(rsETHOracle).getRate()` → returns stale low rate.
5. `rsETHAmount = 1e18 * 1e18 / staleRate` — larger than the correct amount at the current rate.
6. Attacker receives excess `wrsETH`, diluting yield owed to existing holders.

## Impact Explanation

**High — Theft of unclaimed yield.** When the L2 cached rate is stale-low, every depositor receives more `wrsETH` than the current rsETH value warrants. The excess `wrsETH` represents yield that belongs to existing rsETH/wrsETH holders but is instead captured by new depositors. This is a direct, on-chain-triggerable theft of unclaimed yield, matching the allowed High impact class. The SECURITY.md exclusion for "Incorrect data supplied by third-party oracles" does not apply: the oracle data itself is correct; the defect is the protocol's own missing staleness check on its internal cache.

## Likelihood Explanation

- `updateRate()` requires a manual call and ETH payment; it is not automated or enforced on-chain.
- LayerZero message delivery can be delayed during network congestion, extending the staleness window.
- rsETH appreciates continuously as staking rewards accrue; any gap between L1 reward distributions and L2 cache updates creates an exploitable window.
- Any unprivileged depositor can observe the stale rate on-chain (both `rate` and `lastUpdated` are public) and immediately exploit it by calling `deposit()`.
- The attack is repeatable every time the cache lags behind the true rate.

## Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the cached rate is too old:

```solidity
uint256 public maxStaleness; // e.g., 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

`maxStaleness` should be set by the owner and should reflect the expected update cadence with a reasonable buffer. This ensures that if the L1→L2 bridge update is delayed beyond the acceptable window, deposits revert rather than silently using a stale rate.

## Proof of Concept

**Foundry fork test outline:**

1. Fork the L2 deployment with `RSETHPoolV3` and `RSETHRateReceiver` deployed.
2. Record `CrossChainRateReceiver.rate` (e.g., `1.00e18`) and `lastUpdated`.
3. Warp `block.timestamp` forward by 48 hours (simulating no `updateRate()` call).
4. Manually set `CrossChainRateReceiver.rate` to `1.00e18` while the "true" rate (from L1 state) is `1.05e18` — or simply do not call `updateRate()` after a simulated L1 price increase.
5. Call `RSETHPoolV3.deposit{value: 1 ether}("")` as an unprivileged attacker.
6. Assert `wrsETH` minted = `1.000e18` (stale rate path).
7. Assert correct amount at true rate = `1e18 * 1e18 / 1.05e18 ≈ 0.952e18`.
8. Assert attacker received `~0.048e18` excess `wrsETH` (~5%), confirming yield theft from existing holders.

The test requires no privileged access — only a standard `deposit()` call — and is reproducible on any local fork where `lastUpdated` is in the past and `rate` has not been refreshed to match the current L1 value.
