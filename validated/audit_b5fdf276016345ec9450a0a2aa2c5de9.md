Audit Report

## Title
Missing Slippage Protection in `removeLiquidity` Exposes LPs to Composition Loss - (File: metric-core/contracts/MetricOmmPool.sol, metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
`MetricOmmPool.removeLiquidity` computes payout amounts from live bin balances at execution time with no caller-specified minimum output floor and no deadline. Any swap that crosses the target bin between transaction submission and inclusion shifts `token0BalanceScaled`/`token1BalanceScaled`, causing the LP to receive a composition materially different from what they observed, with no on-chain revert path. No periphery wrapper for liquidity removal exists to supply these guards off-chain.

## Finding Description
`MetricOmmPool.removeLiquidity` signature accepts only `owner`, `salt`, `deltas`, and `extensionData` — no `minAmount0Out`, `minAmount1Out`, or `deadline`: [1](#0-0) 

Inside `LiquidityLib.removeLiquidity`, the payout is computed directly from the live bin state: [2](#0-1) 

Both `token0BalanceScaled` and `token1BalanceScaled` are mutated by every swap crossing the bin. After computation, tokens are transferred immediately with no revert path: [3](#0-2) 

By contrast, `MetricOmmPoolLiquidityAdder` enforces `maxAmountToken0`/`maxAmountToken1` caps for `addLiquidity` in its callback: [4](#0-3) 

No `MetricOmmPoolLiquidityRemover` contract exists (glob search returns empty). `MetricOmmSimpleRouter` has `amountOutMinimum` and deadline checks for swaps but no liquidity-removal path: [5](#0-4) 

The `IMetricOmmPoolLiquidityAdder` interface exposes only `addLiquidityExactShares` and `addLiquidityWeighted`: [6](#0-5) 

## Impact Explanation
An LP removing liquidity from a bin that is subsequently drained of one token by a swap receives 0 of that token and a doubled amount of the other. If the drained token has higher oracle value, the LP suffers a direct, irreversible loss of principal within the transaction. This meets the "direct loss of user principal above Sherlock thresholds" criterion, as the loss is proportional to position size and can be total for one leg.

## Likelihood Explanation
The trigger is fully unprivileged: any trader executing a swap against the same bin between the LP's submission and inclusion causes the composition shift. In an oracle-driven pool with active trading this is routine. The absence of a deadline compounds exposure for transactions queued during high-gas periods. Every LP calling `removeLiquidity` directly on the pool — the only available path — is exposed on every withdrawal.

## Recommendation
1. Add `minAmount0Out`, `minAmount1Out`, and `deadline` parameters to `MetricOmmPool.removeLiquidity` and revert if computed amounts fall below the caller's floor or `block.timestamp > deadline`.
2. Enforce the check after `LiquidityLib.removeLiquidity` returns, before tokens are transferred.
3. Add a `MetricOmmPoolLiquidityRemover` periphery wrapper mirroring `MetricOmmPoolLiquidityAdder` to expose these guards to EOA callers.

## Proof of Concept
1. Pool bin 0: `token0BalanceScaled = 1000`, `token1BalanceScaled = 1000`, `binTotalShares = 10000`.
2. LP holds 5000 shares (50%); submits `removeLiquidity` expecting ~500 token0 and ~500 token1.
3. Trader executes a swap buying all token0 from bin 0 before LP tx is mined; bin becomes `token0BalanceScaled = 0`, `token1BalanceScaled ≈ 2000`.
4. LP's `removeLiquidity` executes: `amount0Scaled = (0 * 5000) / 10000 = 0`; `amount1Scaled = (2000 * 5000) / 10000 = 1000`.
5. LP receives 0 token0 and ~1000 token1. If token0 > token1 in oracle value, LP receives less total value than the 500/500 split they intended, with no on-chain recourse.
