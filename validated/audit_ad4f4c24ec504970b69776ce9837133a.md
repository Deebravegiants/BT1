### Title
Missing On-Chain CKD Output Verification for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Forged Key Material — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs no cryptographic verification of the `CKDResponse` when the request uses the `AppPublicKey` variant. A single attested Byzantine participant can call `respond_ckd` with arbitrary `big_y` / `big_c` G1 points, and the contract will accept and resolve the yield — delivering key material with no binding to the MPC master secret.

---

### Finding Description

In `respond_ckd`, the match on `request.app_public_key` has an explicit no-op arm for `AppPublicKey`: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(app_id, pk), msk·g2)`, which cryptographically binds the response to the MPC master secret: [2](#0-1) 

For `AppPublicKey`, no equivalent check exists. The contract proceeds directly to `resolve_yields_for` with whatever `big_y`/`big_c` the caller supplied: [3](#0-2) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active()`: [4](#0-3) 

This requires the caller to be a single attested participant — well below the signing threshold.

The existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates the exact attack path: it passes `[1u8; 48]` / `[2u8; 48]` (invalid G1 points) as `big_y` / `big_c` and the call returns `Ok`: [5](#0-4) 

---

### Impact Explanation

A single Byzantine attested participant can race to call `respond_ckd` for any pending `AppPublicKey` CKD request and supply arbitrary `big_y` / `big_c`. The contract resolves the yield with this garbage, delivering it to the user. The user computes `S = big_c − a·big_y`, obtaining a value with no relation to the MPC master secret. The threshold requirement for CKD output is effectively bypassed for all `AppPublicKey` requests.

This matches the allowed critical impact: **unauthorized confidential key derivation output without the required participant authorization**.

---

### Likelihood Explanation

- Requires only one Byzantine participant in the current attested set — strictly below threshold.
- The `AppPublicKey` variant is still a supported, non-deprecated production path (documented in the contract README as the "legacy" format).
- The attacker only needs to observe the mempool/contract state for a pending `AppPublicKey` CKD request and submit `respond_ckd` before the honest participants do.

---

### Recommendation

Apply the same pairing-based output check to `AppPublicKey` requests, or remove the `AppPublicKey` variant from production. If the G2 component is unavailable for `AppPublicKey`, the contract should at minimum require a threshold of participants to agree on the same `(big_y, big_c)` before resolving the yield, rather than accepting the first response unconditionally.

---

### Proof of Concept

The existing test at `crates/contract/src/lib.rs:3404` already proves the path:

1. Submit `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(...)`.
2. Call `respond_ckd` as a single attested participant with `big_y = [1u8; 48]`, `big_c = [2u8; 48]` (garbage bytes, not valid G1 points).
3. The call returns `Ok(())`.
4. `return_ck_and_clean_state_on_success` resolves the yield with the garbage response.
5. The pending request is cleared — no legitimate participant can overwrite it.
6. The user receives `big_c − a·big_y` as their "derived secret", which has no cryptographic relation to the MPC master secret.

The `AppPublicKeyPV` path rejects the same garbage at line 678–680 (`ckd_output_check` → `env::panic_str`), confirming the asymmetry is the root cause. [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
