### Title
`respond_verify_foreign_tx` Accepts Attacker-Supplied `payload_hash` Without Binding It to the Stored Request - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` function verifies the MPC signature against a `payload_hash` value that is entirely supplied by the responding node in the `VerifyForeignTransactionResponse`. The contract never independently derives or checks that this hash corresponds to the foreign-chain transaction stored in `pending_verify_foreign_tx_requests`. A single malicious attested participant (below the signing threshold) can replay a valid signature obtained from a prior legitimate signing round to a different pending request, causing the contract to resolve that request with a forged attestation.

### Finding Description

In `respond` (regular signing), the payload is taken from the **stored** `SignatureRequest` (committed by the user at submission time) and the signature is verified against a **derived** key:

```rust
// crates/contract/src/lib.rs:597-607
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
```

In `respond_verify_foreign_tx`, the `payload_hash` is taken entirely from the **response** (node-provided), and the signature is verified against the **root** key with no binding to the stored request:

```rust
// crates/contract/src/lib.rs:726-734
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

The stored `VerifyForeignTransactionRequest` contains only `{ request, domain_id, payload_version }` — no pre-committed payload hash. The contract therefore has no ground truth to compare `response.payload_hash` against. The check only confirms that the signature is a valid ECDSA signature over *some* 32-byte value under the root key; it does not confirm that value encodes the correct `(ForeignChainRpcRequest, extracted_values)` tuple for the pending request.

On the node side, `build_signature_request` uses a zero tweak:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs:43
tweak: Tweak::new([0u8; 32]),
```

meaning the signing key equals the root key, so any previously obtained foreign-tx signature is valid under the same root key and can be replayed to any other pending foreign-tx request.

The design document (`docs/foreign-chain-transactions.md` lines 254–286) explicitly planned a `derivation_path`/`tweak` field in `VerifyForeignTransactionRequestArgs` and `VerifyForeignTransactionRequest` to bind each response to a specific caller-controlled key. That binding was never implemented in the production structs:

```rust
// crates/contract/src/dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

No `tweak` or `derivation_path` is carried through.

### Impact Explanation

A single malicious attested MPC participant can:

1. Participate honestly in signing round R1 for foreign-chain transaction T1, obtaining a valid signature `sig_H1` over `H1 = hash(T1, extracted_values_T1)` under the root key.
2. While a victim's request R2 (for a different transaction T2) is pending in `pending_verify_foreign_tx_requests`, call `respond_verify_foreign_tx(R2, { payload_hash: H1, signature: sig_H1 })`.
3. The contract verifies `verify_ecdsa_signature(sig_H1, H1, root_key)` → passes, because `sig_H1` is a valid signature over H1.
4. The contract resolves R2's yield with `{ payload_hash: H1, signature: sig_H1 }`.
5. The victim's bridge contract receives a valid MPC-signed attestation that encodes T1's data, not T2's.

If the downstream bridge contract uses the returned `payload_hash` to determine which foreign-chain event was attested (the intended use), it will process T1's event instead of T2's. This enables cross-chain replay and forged foreign-chain verification, matching the "invalid bridge execution or double-spend conditions" impact class.

### Likelihood Explanation

The attacker must be a single attested MPC participant — within the standard Byzantine fault model (up to t < threshold malicious nodes). No threshold collusion is required; the attacker only needs a signature from a prior legitimate round, which every participant in that round possesses. The attack window is any time a victim's request is pending, which is the normal operating state of the contract.

### Recommendation

The contract must independently commit to the expected `payload_hash` at request submission time, or at minimum verify that `response.payload_hash` is consistent with the stored `VerifyForeignTransactionRequest`. Concretely:

- Implement the planned `derivation_path` → `tweak` derivation so each foreign-tx response is bound to a caller-specific derived key (as described in `docs/foreign-chain-transactions.md` lines 254–286), making cross-request replay impossible.
- Alternatively, have the contract compute `expected_payload_hash = ForeignTxSignPayload::V1 { request, values }.compute_msg_hash()` from the stored request and the node-supplied `values`, then verify `response.payload_hash == expected_payload_hash` before accepting the signature.

### Proof of Concept

1. Attacker (attested participant P) participates in signing round for request R1 (Bitcoin tx `[0xAA; 32]`). Obtains `sig_H1` over `H1 = hash(R1_request, [BlockHash([0xBB;32])])`.
2. Victim submits `verify_foreign_transaction` for request R2 (Bitcoin tx `[0xCC; 32]`). R2 is now pending.
3. Attacker calls:
   ```json
   respond_verify_foreign_tx(
     request = R2,
     response = { payload_hash: H1, signature: sig_H1 }
   )
   ```
4. Contract executes `verify_ecdsa_signature(sig_H1, H1, root_key)` → `Ok`.
5. Contract resolves R2's yield with `{ payload_hash: H1, signature: sig_H1 }`.

## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.
