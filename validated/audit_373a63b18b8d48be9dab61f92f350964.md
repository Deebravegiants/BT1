### Title
Consensus Vote Signature Not Verified in `handle_vote` — Malicious Peer Can Forge Votes for Any Validator, Permanently Stalling Block Production - (File: `crates/apollo_consensus/src/single_height_consensus.rs`)

---

### Summary

`SingleHeightConsensus::handle_vote` accepts incoming `Vote` messages from the network and records them in the state machine's vote maps keyed by `(round, voter)`, without ever verifying the `signature` field. An explicit `// TODO(Asmaa): verify the signature` comment at the top of the function confirms this check is absent. A single malicious peer can forge prevotes or precommits for every legitimate validator, occupying their `(round, voter)` slots before the real validators broadcast their own votes. When the real votes arrive they are silently dropped as `VoteStatus::Duplicate` or `VoteStatus::Conflict`, preventing quorum from ever being reached and permanently stalling block production.

---

### Finding Description

The `Vote` struct carries a `signature: RawSignature` field that is transmitted over the wire: [1](#0-0) 

The protobuf definition confirms the field is present in every network message: [2](#0-1) 

However, `handle_vote` skips signature verification entirely: [3](#0-2) 

The only identity check performed is that `vote.voter` is a committee member address — a check that is trivially satisfied by copying any legitimate validator's address into the forged vote. After that check, the vote is passed to `state_machine.received_vote`, which deduplicates by `(round, voter)` key: [4](#0-3) 

Once a `(round, voter)` slot is occupied — whether by a legitimate or forged vote — any subsequent vote from the same voter for the same round is either `Duplicate` (same commitment) or `Conflict` (different commitment), and both are silently dropped: [5](#0-4) 

Votes are also created with a default (empty) signature, confirming the signing side is equally unimplemented: [6](#0-5) 

A parallel `TODO` exists in `validate_proposal.rs` for the `ProposalFin` commitment comparison: [7](#0-6) 

---

### Impact Explanation

**Attack scenario — liveness destruction:**

1. A malicious peer monitors the broadcast channel and, as soon as a new height/round begins, immediately broadcasts forged `Prevote` and `Precommit` messages for every committee member, each carrying `proposal_commitment = None` (nil) and `signature = RawSignature::default()`.
2. Because `handle_vote` does not verify signatures, each forged vote is accepted and stored in the `prevotes`/`precommits` maps under `(round, validator_address)`.
3. When the real validators subsequently broadcast their actual prevotes/precommits for the real proposal, `received_vote` returns `VoteStatus::Conflict` (different commitment) and the real votes are dropped.
4. No quorum for the real proposal is ever reached. The round times out and advances.
5. The malicious peer repeats the attack for every round, permanently preventing any block from being decided.

**Attack scenario — quorum forgery:**

Alternatively, the malicious peer sends forged precommits for all validators with `proposal_commitment = Some(real_hash)` (matching the legitimate proposal). The real validators' precommits are dropped as `VoteStatus::Duplicate`. The node reaches a decision for `real_hash` based entirely on forged precommits with invalid signatures. The block is committed, but the consensus certificate (the set of precommit signatures) is cryptographically invalid, breaking any downstream L1 or proof-system consumer that verifies the validator set signed off on the block.

The `verify_precommit_vote_signature` function exists in `apollo_signature_manager` but is never called in the vote-handling path: [8](#0-7) 

---

### Likelihood Explanation

Any peer that can send messages on the consensus broadcast topic can execute this attack. The broadcast channel is open to all network participants. No privileged access is required. The attack requires only the ability to observe the current height/round (public information) and send crafted `Vote` protobuf messages with a valid committee member address and an empty signature. The `TODO(Asmaa): verify the signature` comment confirms the developers are aware the check is missing and it has not yet been implemented.

---

### Recommendation

Before recording any incoming vote in the state machine, verify the `vote.signature` against `vote.voter`'s public key over the canonical vote digest (type, height, round, proposal_commitment). The `verify_precommit_vote_signature` helper in `apollo_signature_manager` already implements the verification primitive and should be extended to cover prevotes as well. Votes with invalid or missing signatures must be rejected and the sending peer reported, analogous to how malformed protobuf messages are handled today.

The corresponding signing side (`make_self_vote` in `state_machine.rs`) must also be completed so that legitimate votes carry valid signatures before the verification gate is enforced.

    /// Handle vote messages from peer nodes.
    #[instrument(skip_all)]
    pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
        // TODO(Asmaa): verify the signature
        trace!("Received {:?}", vote);
        let height = self.state_machine.height();
        if vote.height != height {
            warn!("Invalid vote height: expected {:?}, got {:?}", height, vote.height);
            return VecDeque::new();
        }
        if !self.committee.members().iter().any(|s| s.address == vote.voter) {
            debug!("Ignoring vote from non validator: vote={:?}", vote);
            return VecDeque::new();
        }

        // Check if vote has already been received.
        match self.state_machine.received_vote(&vote) {
            VoteStatus::Duplicate => {
                // Duplicate - ignore.
                trace_every_n_ms!(
                    DUPLICATE_VOTE_LOG_PERIOD_MS,
                    "Ignoring duplicate vote: {vote:?}"
                );
                return VecDeque::new();
            }
            VoteStatus::Conflict(old_vote, new_vote) => {
                // Conflict - ignore and record.
                warn!("Conflicting votes: old={old_vote:?}, new={new_vote:?}");
                CONSENSUS_CONFLICTING_VOTES.increment(1);
                return VecDeque::new();
            }
            VoteStatus::New => {
                // Vote is new, proceed to process it.
            }
        }

        info!("Accepting {:?}", vote);
        let sm_vote = match vote.vote_type {
            VoteType::Prevote => StateMachineEvent::Prevote(vote),
            VoteType::Precommit => StateMachineEvent::Precommit(vote),
        };
        self.state_machine.handle_event(sm_vote)
    }
}

    /// Check if a vote has already been received (either in the vote maps or queued).
    /// Returns the status of the vote: NotReceived, Duplicate, or Conflict.
    pub(crate) fn received_vote(&self, vote: &Vote) -> VoteStatus {
        let determine_status = |old: &Vote, new: &Vote| {
            if old.proposal_commitment == new.proposal_commitment {
                VoteStatus::Duplicate
            } else {
                VoteStatus::Conflict(old.clone(), new.clone())
            }
        };

        // Check Map
        let key = (vote.round, vote.voter);
        let map_entry = match vote.vote_type {
            VoteType::Prevote => self.prevotes.get(&key),
            VoteType::Precommit => self.precommits.get(&key),
        };

        if let Some((old_vote, _)) = map_entry {
            return determine_status(old_vote, vote);
        }

        // Check Queue
        for event in &self.events_queue {
            let queued_vote = match (event, vote.vote_type) {
                (StateMachineEvent::Prevote(v), VoteType::Prevote) => v,
                (StateMachineEvent::Precommit(v), VoteType::Precommit) => v,
                _ => continue,
            };

            if queued_vote.round == vote.round && queued_vote.voter == vote.voter {
                return determine_status(queued_vote, vote);
            }
        }

        VoteStatus::New
    }
