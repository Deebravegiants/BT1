### Title
User-Controlled `EvmFinality::Latest` Allows MPC Network to Sign Reorg-able Foreign-Chain State — (`File: crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

Any unprivileged caller can submit a `verify_foreign_transaction` request with `EvmFinality::Latest`, causing MPC nodes to threshold-sign extracted values (block hash, event logs) from a non-finalized EVM block. Because `latest` blocks are subject to chain reorganizations, the resulting MPC signature attests to foreign-chain state that may subsequently cease to exist, enabling a bridge double-spend.

---

### Finding Description

`EvmFinality` is a caller-supplied field inside `EvmRpcRequest`, which is embedded in `VerifyForeignTransactionRequestArgs` and submitted directly to the NEAR MPC contract via `verify_foreign_transaction()`. The contract performs no validation on the finality level — it only checks gas attachment, deposit, domain purpose, and whether the chain is whitelisted. [1](#0-0) 

The `EvmFinality` enum exposes `Latest` as a first-class accepted variant alongside `Safe` and `Finalized`: [2](#0-1) 

Inside each MPC node, `execute_foreign_chain_request` converts the caller-supplied `EvmFinality` directly into the inspector's `EthereumFinality` without any floor enforcement: [3](#0-2) 

The EVM inspector's `verify_finality_level` then issues `eth_getBlockByNumber("latest")` — the current chain head — and accepts the transaction as long as the head block number is ≥ the receipt's block number: [4](#0-3) 

When `EthereumFinality::Latest` is used, the "finality head" is simply the most recently seen block, which carries no finality guarantee. The `verify_block_is_canonical` check that follows only confirms the block is on the current canonical chain at query time — it cannot protect against a future reorg: [5](#0-4) 

After both checks pass, the leader and all followers independently sign the same `ForeignTxSignPayloadV1{request, values}` hash, producing a valid threshold signature over the extracted values (block hash, log data): [6](#0-5) 

---

### Impact Explanation

The MPC network issues a threshold signature attesting that a specific foreign-chain transaction occurred with specific extracted values (e.g., a bridge deposit log). If the underlying block is subsequently reorganized away, the attested transaction no longer exists on the canonical chain, but the MPC signature remains valid and usable on NEAR. A bridge contract that trusts this signature will process a deposit that was never finalized on the foreign chain — a classic double-spend via forged foreign-chain verification.

This maps to the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

- The attack requires no privileged access: any account can call `verify_foreign_transaction` with `EvmFinality::Latest`.
- On Ethereum mainnet, `latest` blocks are reorganized infrequently but non-negligibly (especially on L2s like Arbitrum, Base, BNB, Polygon, Abstract, HyperEVM — all of which accept `EvmFinality::Latest` through the same code path).
- The attacker controls the timing: they submit the request immediately after broadcasting a deposit transaction, before it is finalized, and can coordinate a reorg (or exploit a natural one) after obtaining the MPC signature.
- The 5-second `FOREIGN_CHAIN_INSPECTION_TIMEOUT` means nodes query the chain promptly, making it straightforward to race the request against finality. [7](#0-6) 

---

### Recommendation

1. **Reject `EvmFinality::Latest` at the contract level** inside `verify_foreign_transaction`, returning an `InvalidParameters` error if the request carries `EvmFinality::Latest` for any EVM chain variant.
2. Alternatively, enforce a minimum finality floor inside `execute_foreign_chain_request` in the node, mapping `Latest` → `Safe` or `Finalized` before passing it to the inspector.
3. For Bitcoin, similarly enforce a minimum `BlockConfirmations` floor (e.g., ≥ 6) at the contract level, since `BlockConfirmations(0)` or `BlockConfirmations(1)` is equally reorg-able. [8](#0-7) 

---

### Proof of Concept

1. Attacker broadcasts a deposit transaction `T` on Ethereum (or any supported EVM chain) sending funds to a bridge contract.
2. Attacker immediately calls `verify_foreign_transaction` on the NEAR MPC contract with:
   ```json
   {
     "request": {
       "Ethereum": {
         "tx_id": "<hash of T>",
         "finality": "Latest",
         "extractors": [{"Log": {"log_index": 0}}]
       }
     },
     "domain_id": <foreign_tx_domain>,
     "payload_version": "V1"
   }
   ```
3. MPC nodes query `eth_getTransactionReceipt(<T>)` — succeeds because `T` is in the latest block.
4. `verify_finality_level` queries `eth_getBlockByNumber("latest")` — head ≥ receipt block, passes.
5. `verify_block_is_canonical` passes (block is canonical at query time).
6. All threshold nodes sign `ForeignTxSignPayloadV1{request, [log_data]}`. The NEAR contract returns the signature.
7. Attacker submits the MPC signature to the NEAR bridge contract, claiming the deposit.
8. A chain reorganization (natural or attacker-induced) removes block containing `T` from the canonical Ethereum chain.
9. The deposit never finalized on Ethereum, but the NEAR bridge has already credited the attacker — double-spend achieved. [9](#0-8) [2](#0-1)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
    }
