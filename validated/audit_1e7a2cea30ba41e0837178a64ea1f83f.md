Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice()` Accepts Stale Chainlink Prices Without Staleness Validation — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-detection return values (`updatedAt`, `answeredInRound`, `roundId`), returning the last known price unconditionally. A stale price propagates through `LRTOracle._getTotalEthInProtocol()` into `rsETHPrice`, which governs how much rsETH every depositor receives. The public, permissionless `updateRSETHPrice()` means any external caller can commit a stale price to storage.

## Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` destructures only the `answer` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`updatedAt` (timestamp of last oracle update) is never compared against `block.timestamp`, and `answeredInRound < roundId` (the standard Chainlink round-completeness guard) is never checked. If a feed stops updating, `getAssetPrice()` silently returns the frozen price.

The stale price flows through the full pricing pipeline:
- `LRTOracle.getAssetPrice(asset)` → `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)` (line 157)
- `_getTotalEthInProtocol()` → `totalETHInProtocol += totalAssetAmt.mulWad(assetER)` (line 343)
- `_updateRsETHPrice()` → `newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply)` (line 250)
- `rsETHPrice = newRsETHPrice` (line 313)

`updateRSETHPrice()` is `public whenNotPaused` (line 87), callable by any unprivileged user.

**Existing guards are insufficient.** `LRTOracle` has a `pricePercentageLimit` guard that pauses the protocol on large price decreases or reverts on large price increases (lines 252–281). However: (1) `pricePercentageLimit` defaults to `0`, disabling all guards entirely; (2) even when set, it only catches deviations exceeding the threshold relative to `highestRsethPrice` — small-to-moderate staleness within the band passes through silently; (3) it is not a substitute for feed-level staleness validation.

The inconsistency is confirmed by `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 30–32, which performs `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0` checks in its sibling `getRate()` function, demonstrating the team is aware of the pattern and intentionally applied it elsewhere.

## Impact Explanation

**Direction 1 — stale price frozen above actual (LST depegs, oracle freezes):** `totalETHInProtocol` is overestimated → `rsETHPrice` is inflated → depositors receive fewer rsETH tokens than their deposit is worth. This is a direct, quantifiable loss per depositor proportional to `(P_stale − P_actual) / P_stale`. Maps to: **Low — Contract fails to deliver promised returns**.

**Direction 2 — stale price frozen below actual (LST appreciates, oracle freezes):** `totalETHInProtocol` is underestimated → `rsETHPrice` is deflated → depositors receive more rsETH than the protocol's backing supports → rsETH supply becomes undercollateralized. Maps to: **Critical — Protocol insolvency**.

## Likelihood Explanation

Chainlink feeds have documented heartbeat intervals (1 hour for ETH/USD, up to 24 hours for some LST feeds). Network congestion, L2 sequencer downtime, or oracle node failures are documented operational risks that every Chainlink integration is expected to guard against. No special attacker capability is required: any user can call the public `updateRSETHPrice()` to commit the stale price. The protocol's own sibling contract confirms the team is aware of the staleness pattern, making the omission in `ChainlinkPriceOracle` an oversight rather than an intentional design choice.

## Recommendation

Add a configurable `stalePriceThreshold` per feed (matching the Chainlink heartbeat + buffer) and validate in `getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (block.timestamp - updatedAt > stalePriceThreshold) revert StalePrice();
if (price <= 0) revert InvalidPrice();
```

This mirrors the pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()` and adds the missing time-based staleness guard absent from that contract as well.

## Proof of Concept

1. A supported LST (e.g., stETH) has a Chainlink feed registered via `ChainlinkPriceOracle.updatePriceFeedFor()`.
2. The Chainlink feed stops updating; last recorded price is `P_stale`.
3. Actual market price moves to `P_actual` (depeg: `P_actual < P_stale`).
4. Any unprivileged caller invokes `LRTOracle.updateRSETHPrice()` (public, line 87).
5. `_getTotalEthInProtocol()` calls `getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `P_stale` with no revert.
6. `rsETHPrice` is set using the inflated `P_stale`.
7. A depositor calls `LRTDepositPool.depositAsset(stETH, amount)`; the deposit pool uses the inflated `rsETHPrice` to compute the rsETH mint amount, issuing fewer rsETH tokens than the deposit is worth — a direct, quantifiable loss.

**Foundry fork test plan:** Fork mainnet/Holesky with a registered stETH Chainlink feed. Use `vm.mockCall` to make `latestRoundData()` return a price with `updatedAt = block.timestamp - 2 days` (well past any heartbeat). Call `updateRSETHPrice()` as address(1). Assert `rsETHPrice` was updated to the stale value without revert. Then call `depositAsset` and assert the minted rsETH amount reflects the stale (inflated) price rather than the actual price.
