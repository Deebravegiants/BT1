### Title
`respond_verify_foreign_tx` Verifies Signature Against Attacker-Supplied `payload_hash` Instead of Request-Derived Hash — (File: crates/contract/src/lib.rs)

### Summary

In `respond_verify_foreign_tx`, the contract verifies the MPC signature against `response.payload_hash` — a value supplied entirely by the responding node — without ever checking that this hash actually corresponds to the `VerifyForeignTransactionRequest` being resolved. A single Byzantine attested participant who acts as leader for any one request can reuse the threshold signature produced for that request to close out a *different* pending request with a fabricated `payload_hash`, delivering a forged foreign-chain verification result to the waiting caller.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← from the response, not the request

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← attacker-controlled
    &secp_pk,               // root public key
)
.is_ok()
``` [1](#0-0) 

The contract only confirms that the ECDSA signature is self-consistent with the provided `payload_hash`; it never verifies that `payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the specific `request` argument passed to the call.

Compare this to `respond` (the sign-request path), where the payload is taken directly from the stored request — not from the response — and the signature is checked against a *derived* key that commits to the caller's path:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,           // ← from the request, not the response
    &expected_public_key,   // derived key, commits to tweak
)
``` [2](#0-1) 

For foreign-tx verification the MPC nodes sign `SHA-256(borsh(ForeignTxSignPayload { request, values }))`, where `values` are the on-chain-observed extracted values: [3](#0-2) 

The leader node computes this hash and submits it as `response.payload_hash`. Because the contract never recomputes the hash from the `request` field it already holds, the leader can substitute any hash for which it possesses a valid root-key signature.

### Impact Explanation

A Byzantine leader node (a single attested participant, strictly below the signing threshold) can:

1. Act as leader for pending request **R1** (e.g., "verify Bitcoin tx T1"). The MPC network collectively signs `H1 = SHA-256(borsh(ForeignTxSignPayload{R1, V1}))`. The leader assembles the full threshold signature `sig_H1`.
2. While a *different* pending request **R2** (e.g., "verify Bitcoin tx T2") is waiting, call `respond_verify_foreign_tx(request=R2, response={payload_hash=H1, signature=sig_H1})`.
3. The contract verifies `sig_H1` over `H1` under the root key — this passes — and calls `resolve_yields_for`, delivering `{payload_hash=H1, signature=sig_H1}` to every yield parked under R2.

The caller of R2 receives a response whose `payload_hash` commits to a completely different transaction (T1, not T2). Any downstream bridge contract or SDK consumer that does not independently recompute the expected hash and compare it to `response.payload_hash` will treat the response as a valid attestation of T2, enabling invalid bridge execution or double-spend conditions.

The `near-mpc-sdk` `ForeignChainSignatureVerifier::verify_signature` does perform this check client-side: [4](#0-3) 

However, the contract — the authoritative on-chain source of truth — does not enforce it, so any caller that omits the SDK-level re-verification is silently misled.

### Likelihood Explanation

The attacker is a single Byzantine attested participant who is elected leader for at least one request. Leader election is deterministic and rotates across participants; every participant will eventually be leader. No threshold collusion is required: the leader legitimately receives the assembled threshold signature as part of the normal protocol flow and can immediately repurpose it. The attack is silent (no on-chain error, no log anomaly) and requires no special timing beyond having two concurrent pending requests.

### Recommendation

The contract should recompute the expected `payload_hash` from the `request` and the `extracted_values` supplied in the response, then assert equality before accepting the response. Concretely, the `VerifyForeignTransactionResponse` should carry the full `ForeignTxSignPayload` (or at minimum the `Vec<ExtractedValue>`), and `respond_verify_foreign_tx` should:

1. Reconstruct `ForeignTxSignPayload { request: request.request.clone(), values: response.values }`.
2. Compute `expected_hash = payload.compute_msg_hash()`.
3. Assert `expected_hash == response.payload_hash` before the signature check.

This mirrors how `respond` binds the signature to the stored request payload rather than trusting the responder's claim.

### Proof of Concept

```
// Setup: two pending requests R1 (tx T1) and R2 (tx T2) are queued.
// Byzantine leader is elected for R1.

// Step 1 – normal MPC signing for R1 produces sig_H1 over H1.
let H1 = SHA256(borsh(ForeignTxSignPayload { request: R1, values: V1 }));
let sig_H1 = <threshold signature from MPC network>;

// Step 2 – Byzantine leader calls respond_verify_foreign_tx for R2 with R1's signature.
contract.respond_verify_foreign_tx(
    request = R2,                                    // pending request for tx T2
    response = { payload_hash: H1, signature: sig_H1 }  // H1 commits to T1, not T2
);

// Step 3 – Contract checks: is sig_H1 valid over H1 under root key? YES.
// Contract resolves R2 with { payload_hash: H1, signature: sig_H1 }.

// Step 4 – Caller of R2 receives a response attesting to T1, not T2.
// If caller skips ForeignChainSignatureVerifier re-verification, they
// treat the response as proof that T2 was observed on-chain.
```
