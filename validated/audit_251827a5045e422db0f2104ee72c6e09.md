Audit Report

## Title
Missing Staleness Validation on Chainlink `latestRoundData()` Enables Stale Prices to Corrupt rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and discards every return value except `answer`, performing no staleness, sign, or round-completeness checks. A stale Chainlink price propagates into both the deposit minting ratio and the stored `rsETHPrice`, creating a window in which depositors can mint more rsETH than they are entitled to, diluting the yield of all existing rsETH holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at line 52 reads:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`) are available via the declared interface but only `answer` is consumed. No check is made on `updatedAt`, `answeredInRound`, or the sign of `price`.

The sibling contract `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository performs three of these checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

This confirms the project is aware of the requirements but failed to apply them to `ChainlinkPriceOracle`.

**Exploit path:**

1. A Chainlink LST/ETH feed goes stale (OCR outage, gas spike) at a price below the true market price (e.g., `0.998e18`).
2. `LRTOracle.updateRSETHPrice()` is called (publicly callable) while the feed is stale. `_getTotalEthInProtocol()` calls `getAssetPrice(asset)` for each supported LST, receiving the stale low price. TVL is understated, so `rsETHPrice` is written to storage at a value below its true level. [3](#0-2) 
3. The Chainlink feed subsequently updates to the true price (`1.000e18`), but `rsETHPrice` in storage has not yet been corrected.
4. A depositor calls `LRTDepositPool.depositAsset()`. `getRsETHAmountToMint` computes:
   ```
   rsethAmountToMint = (amount × getAssetPrice(asset)) / rsETHPrice
                     = (amount × 1.000e18) / stale-low-rsETHPrice
   ```
   The numerator uses the now-fresh Chainlink price while the denominator is the stale-low stored `rsETHPrice`, yielding more rsETH than the depositor is entitled to. [4](#0-3) 
5. When `updateRSETHPrice()` is next called with the corrected feed, `rsETHPrice` rises, but the over-minted tokens already exist, permanently diluting all pre-existing rsETH holders.

Existing guards are insufficient: the `pricePercentageLimit` circuit-breaker in `_updateRsETHPrice()` only triggers on large deviations and only when `updateRSETHPrice()` is called, not during deposits. The `minRSETHAmountExpected` slippage parameter in `depositAsset()` protects the depositor from receiving *fewer* tokens but does not prevent over-minting.

## Impact Explanation

**High — Theft of unclaimed yield.** When a depositor mints rsETH against a stale-low `rsETHPrice`, the total rsETH supply increases by more than the deposited value warrants. Upon price correction, every pre-existing rsETH holder's share of the protocol TVL is permanently diluted. The magnitude scales with the size of the deposit and the degree of staleness; even a 0.2% stale deviation on a large deposit causes a measurable, irreversible transfer of accrued yield from existing holders to the over-minted depositor.

## Likelihood Explanation

Chainlink OCR feeds for LST/ETH pairs carry documented heartbeat windows (typically 1 hour). Network congestion or OCR node failures can delay updates beyond the heartbeat. The condition is not attacker-controlled, but any depositor who observes that the on-chain feed price lags the true market price can time a deposit to the window between feed recovery and the next `updateRSETHPrice()` call, requiring no special privileges. `updateRSETHPrice()` is public but not automatically called on every deposit, so the window can persist for an extended period.

## Recommendation

Add a configurable `maxStaleness` parameter to `ChainlinkPriceOracle` and validate all relevant fields from `latestRoundData()`, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(price > 0,                                    "Invalid price");
require(updatedAt != 0,                               "Incomplete round");
require(answeredInRound >= roundId,                   "Stale round");
require(block.timestamp - updatedAt <= maxStaleness,  "Price too stale");
```

`maxStaleness` should be set per feed based on its documented heartbeat (e.g., 3600 s for a 1-hour heartbeat, with a small buffer), and should be updatable by the LRT admin.

## Proof of Concept

**Foundry fork test outline:**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed is live.
2. Deploy or reference the existing `ChainlinkPriceOracle` and `LRTOracle`.
3. Use `vm.mockCall` to make `latestRoundData()` return a stale answer: `updatedAt = block.timestamp - 7200` (2 hours ago), `price = 0.998e18`, `answeredInRound < roundId`.
4. Call `LRTOracle.updateRSETHPrice()` — record `rsETHPrice` (stale-low).
5. Remove the mock so `latestRoundData()` returns the true price `1.000e18` with a fresh `updatedAt`.
6. Call `LRTDepositPool.depositAsset(stETH, largeAmount, 0, "")` as an unprivileged address.
7. Assert that the rsETH minted exceeds `largeAmount * 1e18 / trueRsETHPrice` — confirming over-minting.
8. Call `LRTOracle.updateRSETHPrice()` again to correct `rsETHPrice`.
9. Assert that the per-token ETH value of pre-existing rsETH holders has decreased relative to step 4, confirming yield dilution.
