### Title
Silently Swallowed L1 Events Provider Commit Failure Enables L1 Handler Transaction Re-execution — (File: crates/apollo_batcher/src/batcher.rs)

---

### Summary

In `commit_proposal_and_block`, after the block is irreversibly written to storage, the call to `l1_events_provider_client.commit_block` can fail. The code logs an error and increments a metric but **does not return an error and does not retry**, despite a comment at the failure site that reads *"Return error if the commit to the L1 provider failed."* This leaves the L1 events provider in a state where consumed L1 handler transactions are still marked as staged (not committed), making them eligible for re-inclusion in the next block proposal and causing double-execution of L1 messages.

---

### Finding Description

`commit_proposal_and_block` in `crates/apollo_batcher/src/batcher.rs` performs two sequential, non-atomic operations:

1. **Commit the block to storage** (line 1101–1106) — this is guarded with `?` and propagates errors correctly.
2. **Notify the L1 events provider** (lines 1118–1121) — this is *not* guarded; failures are silently swallowed. [1](#0-0) 

The critical section:

```rust
// Commit the proposal to the storage.
self.storage_writer
    .commit_proposal(height, state_diff, storage_commitment_block_hash)
    .map_err(|err| { ... })?;   // ← error propagated correctly

// Notify the L1 provider of the new block.
let l1_events_provider_result = self
    .l1_events_provider_client
    .commit_block(consumed_l1_handler_tx_hashes, rejected_l1_handler_tx_hashes, height)
    .await;

// Return error if the commit to the L1 provider failed.   ← comment is WRONG
if let Err(err) = l1_events_provider_result {
    // ... logs error, increments metric ...
    BATCHER_L1_EVENTS_PROVIDER_ERRORS.increment(1);
    // ← NO return, NO retry, NO panic
}
```

The L1 events provider is the sole authoritative gatekeeper for which L1 handler transactions are eligible for block inclusion. Its state machine transitions are:

- **Pending** → **Staged** (on `get_txs` / `validate`)
- **Staged** → **Committed** (on `commit_block`)
- **Staged** → **Pending** (on `start_block` for the next height, which resets the staging epoch)

When `commit_block` fails silently, consumed L1 handler transactions remain in **Staged** state. The next call to `start_block` (for block N+1) resets the staging epoch, returning those transactions to **Pending** — available for re-proposal. [2](#0-1) 

The blockifier has no independent deduplication of L1 handler transactions. It relies entirely on the L1 events provider to gate eligibility. If the provider's state is stale, the blockifier will execute the same L1 handler transaction a second time without error. [3](#0-2) 

---

### Impact Explanation

L1 handler transactions already executed and committed in block N can be re-included and re-executed in block N+1. This produces:

- **Wrong on-chain state**: storage writes from the L1 handler execute twice.
- **Wrong receipts and events**: the transaction appears in two blocks with two distinct receipts.
- **Wrong L1 message consumption records**: the L1 message is consumed twice on L2, violating the L1→L2 messaging invariant.

This matches the allowed Critical impact category: *"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."*

---

### Likelihood Explanation

The L1 events provider is a local async component. Failures can arise from:

1. **Channel/communication errors** between the batcher and the L1 events provider process.
2. **`UnexpectedHeight` errors** — explicitly matched in the error arm (lines 1126–1133), confirming this is a known, reachable failure mode.
3. **Any other `L1EventsProviderClientError`** — caught by the `other_err` arm.

The misleading comment *"Return error if the commit to the L1 provider failed"* at line 1123 strongly suggests the original intent was to propagate the error, but the `return Err(...)` was never written. This is a latent defect, not a deliberate design choice. [4](#0-3) 

---

### Recommendation

The fix must ensure the L1 events provider is always notified before `commit_proposal_and_block` returns `Ok`. Three options, in order of preference:

1. **Retry until success**: loop on `commit_block` with backoff before returning `Ok(())`. This mirrors the retry pattern used elsewhere in the codebase (e.g., `add_task_with_retries` in the commitment manager).
2. **Propagate the error**: return `Err(BatcherError::InternalError)` so the caller can handle it. Note: the block is already committed to storage at this point, so the caller must be prepared to handle the inconsistency.
3. **Panic**: treat this as an unrecoverable invariant violation, forcing a node restart and re-sync of the L1 events provider from storage.

The current behavior — log, increment metric, and silently continue — must be removed.

---

### Proof of Concept

1. A block is proposed and executed containing L1 handler transactions `T1`, `T2`, `T3`.
2. Consensus reaches a decision; `batcher.decision_reached` is called.
3. Inside `commit_proposal_and_block`, `storage_writer.commit_proposal(N, ...)` succeeds — block N is now permanently in storage.
4. `l1_events_provider_client.commit_block({T1, T2, T3}, {}, N)` fails (e.g., `UnexpectedHeight` or a channel error).
5. Lines 1123–1143 log the error and increment `BATCHER_L1_EVENTS_PROVIDER_ERRORS`, then fall through. `commit_proposal_and_block` returns `Ok(())`.
6. `decision_reached` returns `Ok(DecisionReachedResponse {...})` to the consensus orchestrator.
7. The consensus orchestrator calls `finalize_decision` and advances to height N+1.
8. The batcher calls `l1_events_provider_client.start_block(SessionState::Propose, N+1)`, which resets the staging epoch. `T1`, `T2`, `T3` transition back to **Pending**.
9. The proposer calls `get_txs` on the L1 events provider; `T1`, `T2`, `T3` are returned as eligible.
10. Block N+1 is built with `T1`, `T2`, `T3`. The blockifier executes them a second time, producing duplicate state writes, duplicate receipts, and duplicate L1 message consumption — all committed to the canonical chain. [5](#0-4)

    async fn commit_proposal_and_block(
        &mut self,
        height: BlockNumber,
        state_diff: ThinStateDiff,
        address_to_nonce: HashMap<ContractAddress, Nonce>,
        consumed_l1_handler_tx_hashes: IndexSet<TransactionHash>,
        rejected_tx_hashes: IndexSet<TransactionHash>,
        storage_commitment_block_hash: StorageCommitmentBlockHash,
    ) -> BatcherResult<()> {
        info!(
            "Committing block at height {} and notifying mempool & L1 event provider of the block.",
            height
        );
        trace!("Rejected transactions: {:#?}, State diff: {:#?}.", rejected_tx_hashes, state_diff);

        // Proposal commitment is the the partial block hash when it's available, and None
        // otherwise. The commitment is computed here to set the prev_proposal_commitment cache. As
        // this cache is only used for blocks obtained through the decision_reached flow, it can
        // be None for old (pre 0.13.2) blocks.
        let proposal_commitment = match &storage_commitment_block_hash {
            StorageCommitmentBlockHash::Partial(components) => Some((
                height,
                ProposalCommitment {
                    partial_block_hash: PartialBlockHash::from_partial_block_hash_components(
                        components,
                    )
                    .map_err(|e| {
                        error!("Failed to compute partial block hash: {}", e);
                        BatcherError::InternalError
                    })?,
                },
            )),
            StorageCommitmentBlockHash::ParentHash(_) => None,
        };

        // Commit the proposal to the storage.
        self.storage_writer
            .commit_proposal(height, state_diff, storage_commitment_block_hash)
            .map_err(|err| {
                error!("Failed to commit proposal to storage: {}", err);
                BatcherError::InternalError
            })?;
        info!("Successfully committed proposal for block {} to storage.", height);

        self.prev_proposal_commitment = proposal_commitment;

        // Notify the L1 provider of the new block.
        let rejected_l1_handler_tx_hashes = rejected_tx_hashes
            .iter()
            .copied()
            .filter(|tx_hash| consumed_l1_handler_tx_hashes.contains(tx_hash))
            .collect();

        let l1_events_provider_result = self
            .l1_events_provider_client
            .commit_block(consumed_l1_handler_tx_hashes, rejected_l1_handler_tx_hashes, height)
            .await;

        // Return error if the commit to the L1 provider failed.
        if let Err(err) = l1_events_provider_result {
            match err {
                L1EventsProviderClientError::L1EventsProviderError(
                    L1EventsProviderError::UnexpectedHeight { expected_height, got },
                ) => {
                    error!(
                        "Unexpected height while committing block in L1 provider: expected={:?}, \
                         got={:?}",
                        expected_height, got
                    );
                }
                other_err => {
                    error!(
                        "Unexpected error while committing block in L1 provider: {:?}",
                        other_err
                    );
                }
            }
            BATCHER_L1_EVENTS_PROVIDER_ERRORS.increment(1);
        }

        // Notify the mempool of the new block (skipped in validation-only mode).
        if let Some(mempool_client) = &self.mempool_client {
            let mempool_result = mempool_client
                .commit_block(CommitBlockArgs { address_to_nonce, rejected_tx_hashes })
                .await;
            if let Err(mempool_err) = mempool_result {
                // Recoverable error, mempool won't be updated with the new block.
                error!("Failed to commit block to mempool: {}", mempool_err);
            }
        }

        BUILDING_HEIGHT.increment(1);
        Ok(())
    }
