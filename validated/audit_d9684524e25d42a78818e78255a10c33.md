### Title
Excess NEAR Deposit Not Refunded in `submit_participant_info` Re-submission Path — (File: `crates/contract/src/lib.rs`)

### Summary

`submit_participant_info` is a `#[payable]` function that conditionally handles deposit refunds. When a current participant re-submits their attestation (the routine renewal path), the deposit-handling block is skipped entirely. Any NEAR tokens attached to that call are permanently locked in the contract with no refund issued.

### Finding Description

The function `submit_participant_info` computes a boolean gate:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... deposit check and refund logic ...
}
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — i.e., the caller is already a participant **and** the attestation is an update rather than a new insertion — the entire deposit-handling block is bypassed. The function is still marked `#[payable]`, so NEAR tokens can be attached freely. Any such tokens are silently absorbed by the contract; no refund path exists for this branch. [2](#0-1) 

The `require_deposit` helper used by `sign`, `request_app_private_key`, and `verify_foreign_transaction` always refunds excess:

```rust
Some(diff) => {
    if diff > NearToken::from_yoctonear(0) {
        log!("refund excess deposit {diff} to {predecessor}");
        Promise::new(predecessor.clone()).transfer(diff).detach();
    }
}
``` [3](#0-2) 

`submit_participant_info` does not apply this pattern in the `false` branch.

### Impact Explanation

Any NEAR attached to a re-submission call by an existing participant is permanently locked inside the contract. The contract's own accounting invariant — that excess deposits are always returned — is violated on this code path. Funds are not recoverable without a contract upgrade.

This matches: **Medium. Balance or contract execution-flow manipulation that breaks production safety/accounting invariants.**

### Likelihood Explanation

MPC nodes call `submit_participant_info` on a 1-hour cadence for attestation renewal. The node software controls the attached deposit, so accidental over-attachment requires a node-side bug or misconfiguration. However:

- The function is `#[payable]` with no upper-bound enforcement, so any deposit is silently accepted.
- A Byzantine participant (below signing threshold) could deliberately call this with a large deposit, permanently removing those funds from circulation.
- A node software regression (e.g., always attaching 1 yoctoNEAR as a "safe default") would silently drain node operator balances over time across every hourly renewal.

### Recommendation

Add an unconditional refund of any attached deposit in the `else` branch (or after the `if` block), mirroring the pattern already used in `require_deposit` and `propose_update`:

```rust
} else {
    // Caller is an existing participant re-submitting; no storage cost applies.
    // Refund any accidentally attached deposit.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, reject non-zero deposits outright in this branch to make the invariant explicit.

### Proof of Concept

1. Deploy the contract and register participant `alice` with a valid attestation (new insertion — deposit is handled correctly).
2. `alice` calls `submit_participant_info` again (re-submission, `is_new_attestation = false`) with `attached_deposit = 1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR).
3. `attestation_storage_must_be_paid_by_caller` evaluates to `false` because `alice` is already a participant and the attestation is not new.
4. The `if` block is skipped; no refund is issued.
5. `alice`'s balance decreases by 1 NEAR; the contract balance increases by 1 NEAR. The funds are permanently locked. [4](#0-3)
