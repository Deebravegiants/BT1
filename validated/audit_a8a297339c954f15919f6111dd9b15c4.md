### Title
Rewards Permanently Lost When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary

`KernelDepositPool` contains the same structural flaw as the BathBuddy report: when `totalKernelStaked` is zero during an active reward window, `rewardPerTokenStored` is frozen while `updatedAt` still advances, causing all rewards accrued during the zero-supply gap to be permanently unclaimable. The contract itself acknowledges this in its NatSpec but relies entirely on off-chain operational guarantees with no on-chain enforcement.

---

### Finding Description

`KernelDepositPool` is a Synthetix-style staking rewards contract. The `updateReward` modifier runs before every state-changing function:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // frozen if totalKernelStaked == 0
    updatedAt = lastTimeRewardApplicable();    // always advances
    ...
}
``` [1](#0-0) 

The `rewardPerToken()` view function returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
``` [2](#0-1) 

The consequence: when `totalKernelStaked == 0`, `rewardPerTokenStored` does not increase, but `updatedAt` is set to `lastTimeRewardApplicable()`. The time gap during which no one was staked is permanently skipped. Rewards that should have accrued during that window (`rewardRate × gap_seconds`) are never attributed to any staker and remain locked in the contract forever.

`notifyRewardAmount` has a guard against starting a period with zero stakers:

```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
``` [3](#0-2) 

But this only prevents the initial-period case. There is **no on-chain mechanism** preventing all stakers from withdrawing mid-period. The contract's own NatSpec explicitly acknowledges the unmitigated mid-period case:

> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract. In this deployment, we're avoiding this issue by ensuring there are always some tokens staked… for the entire duration of the reward period."* [4](#0-3) 

This is a pure off-chain operational assumption with no enforcement.

The withdrawal flow that triggers the gap: `initiateWithdrawal` calls `updateReward(msg.sender)` **before** decrementing `totalKernelStaked`. So when the last staker withdraws, `updatedAt` advances to the current time and `rewardPerTokenStored` is correctly updated up to that moment. After the decrement, `totalKernelStaked == 0`. From that point forward, any subsequent call to `updateReward` (e.g., when a new staker joins) will advance `updatedAt` again without increasing `rewardPerTokenStored`, permanently discarding the rewards for the empty interval. [5](#0-4) 

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Rewards deposited by the admin via `notifyRewardAmount` and intended for stakers are permanently locked in the contract for any time interval where `totalKernelStaked == 0`. No recovery mechanism exists; the tokens cannot be swept or redistributed. The magnitude scales with `rewardRate × zero_supply_duration`.

---

### Likelihood Explanation

The scenario is realistic and requires no privileged action:

- Any staker can call `initiateWithdrawal` at any time during an active reward period.
- If all stakers withdraw (e.g., due to a better yield opportunity, a market downturn, or a coordinated exit), `totalKernelStaked` reaches zero.
- No on-chain mechanism prevents this or pauses the reward clock.
- The contract's own comment confirms the developers are aware this can happen and are relying on social/operational coordination to prevent it.

---

### Recommendation

Enforce the invariant on-chain rather than relying on operational guarantees. Two standard approaches:

1. **Pause the reward clock when supply is zero**: In `rewardPerToken()`, also freeze `updatedAt` when `totalKernelStaked == 0` so the time gap is not consumed. This requires storing a separate "last non-zero timestamp" and using it in the accumulator calculation.

2. **Prevent withdrawals that would reduce `totalKernelStaked` to zero during an active reward period**: Add a check in `initiateWithdrawal` that reverts if `totalKernelStaked - _amount == 0 && block.timestamp < finishAt`.

Option 1 is the canonical fix used in the referenced Synthetix-style audits and preserves full user freedom to withdraw.

### Proof of Concept

```
Setup:
  - duration = 10,000 seconds
  - rewardAmount = 100e18 tokens
  - Alice stakes 1e18 KERNEL tokens
  - Admin calls notifyRewardAmount(100e18)
    → finishAt = T + 10,000, updatedAt = T, rewardRate = 100e18 / 10,000 = 1e16/s

At T + 5,000 (halfway):
  - Alice calls initiateWithdrawal(1e18)
    → updateReward(Alice) runs: rewardPerTokenStored += rewardRate * 5,000 * 1e18 / 1e18 = 5e19
    → updatedAt = T + 5,000
    → totalKernelStaked = 0

From T + 5,000 to T + 10,000 (empty period):
  - No stakers. No calls. updatedAt frozen at T + 5,000.

At T + 10,000:
  - Bob stakes 1e18 KERNEL tokens
    → updateReward(Bob) runs: totalKernelStaked == 0 → rewardPerTokenStored unchanged (stays 5e19)
    → updatedAt = lastTimeRewardApplicable() = finishAt = T + 10,000
    → Bob's userRewardPerTokenPaid = 5e19

  - Bob calls getReward():
    → earned(Bob) = 1e18 * (5e19 - 5e19) / 1e18 = 0

  - Alice calls getReward() (after claimWithdrawal):
    → earned(Alice) = 0 * (...) + rewards[Alice] = 5e19 (only first half)

Result:
  - Alice receives 50e18 tokens (first half of period)
  - Bob receives 0 tokens
  - 50e18 tokens permanently locked in KernelDepositPool
```

The second 50e18 tokens — corresponding to the 5,000-second empty window — are permanently unclaimable by any address.
