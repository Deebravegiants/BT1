### Title
Deposits for Non-Winning `propose_update` Proposals Are Permanently Locked in the MPC Contract - (File: `crates/contract/src/update.rs`)

### Summary
When `do_update` executes a winning contract-upgrade or config-update proposal, it unconditionally clears **all** other pending proposals via `self.entries.clear()` without refunding the NEAR deposits those proposers attached. Because `UpdateEntry` does not record the proposer's account ID or deposit amount, the funds are irrecoverable and permanently frozen inside the chain-signature contract.

### Finding Description
`propose_update` (`lib.rs:1298–1334`) is a `#[payable]` method that requires a deposit sized to cover on-chain storage for the proposal payload:

```rust
let required = ProposedUpdates::required_deposit(&update);
// ...
let id = self.proposed_updates.propose(update);
``` [1](#0-0) 

The deposit is consumed by the NEAR runtime (credited to the contract's balance). Only the excess above `required` is refunded to the proposer at call time. [2](#0-1) 

The stored `UpdateEntry` records only the update payload and its byte count — **not** the proposer's account ID or the deposit amount:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
``` [3](#0-2) 

When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry and then **bulk-clears every other pending entry and all votes** with no refund logic:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ... deploy/config promise, no refund of other proposers' deposits
}
``` [4](#0-3) 

Because the proposer identity was never stored, there is no information available to issue refunds. The contract README explicitly acknowledges the clearing behaviour ("all pending update proposals and votes are cleared") but does not address the deposit loss. [5](#0-4) 

The existing sandbox test `test_propose_update_contract_many` confirms that multiple concurrent proposals are a supported and tested scenario, making this a reachable production path. [6](#0-5) 

### Impact Explanation
NEAR tokens deposited by non-winning proposers are permanently frozen inside the chain-signature contract. The deposit for a contract-code proposal is proportional to the WASM binary size:

```rust
fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
``` [7](#0-6) 

`bytes_used` includes the full WASM payload plus a fixed overhead for 128 participant-vote slots: [8](#0-7) 

A typical MPC contract WASM is hundreds of KB to over 1 MB, placing the required deposit in the range of tens to hundreds of NEAR per proposal. With multiple competing proposals, the total permanently frozen amount scales accordingly. There is no admin escape hatch, no withdrawal function, and no migration path that recovers these funds. This constitutes **permanent freezing of funds controlled by the chain-signature contract**, matching the Critical impact tier.

### Likelihood Explanation
- Only current participants (voters) can call `propose_update`, so the entry path is restricted to the participant set — but it does not require threshold collusion or any privileged key.
- Multiple concurrent proposals are explicitly supported and tested (`test_propose_update_contract_many`).
- Any governance upgrade cycle where two or more participants independently propose different updates (e.g., one proposes a code upgrade, another proposes a config change) triggers the loss automatically upon the first proposal reaching threshold.
- No malicious intent is required; the loss occurs during normal governance operation.

Likelihood: **Medium** (requires ≥2 concurrent proposals, which is a realistic governance scenario).

### Recommendation
Store the proposer's `AccountId` and the exact deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the refund pattern already used in `propose_update` for excess deposits and in `submit_participant_info`. [2](#0-1) 

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM binary, attaching ~50 NEAR deposit (deposit is consumed, only excess refunded).
2. Participant B calls `propose_update` with a different 500 KB WASM binary, attaching ~50 NEAR deposit.
3. Threshold participants vote for B's proposal via `vote_update`.
4. `vote_update` calls `self.proposed_updates.do_update(&id, ...)`.
5. `do_update` removes B's entry, then executes `self.entries.clear()` — silently dropping A's entry.
6. A's ~50 NEAR deposit remains in the contract balance with no record of who deposited it or how much. No refund is ever issued.
7. A has no recourse: there is no `withdraw_proposal_deposit` function, no admin recovery path, and the proposer identity was never stored. [9](#0-8) [10](#0-9)

    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

Participants can propose and vote on contract updates (code or configuration changes). When an update receives sufficient votes and is executed (via the `vote_update` endpoint which calls `do_update` internally), all pending update proposals and votes are cleared as they are no longer be valid after the contract migration. The update ID counter is preserved across migrations as part of the contract state to avoid race conditions where multiple participants might propose updates with colliding IDs immediately after an upgrade.

async fn test_propose_update_contract_many() {
    let SandboxTestSetup {
        contract,
        mpc_signer_accounts,
        ..
    } = SandboxTestSetup::builder()
        .with_protocols(ALL_PROTOCOLS)
        .build()
        .await;
    dbg!(contract.id());

    const PROPOSAL_COUNT: usize = 2;
    let mut proposals = Vec::with_capacity(PROPOSAL_COUNT);
    // Try to propose multiple updates to check if they are being proposed correctly
    // and that we can have many at once living in the contract state.
    for i in 0..PROPOSAL_COUNT {
        let execution = mpc_signer_accounts[i % mpc_signer_accounts.len()]
            .call(contract.id(), method_names::PROPOSE_UPDATE)
            .args_borsh(current_contract_proposal())
            .max_gas()
            .deposit(CURRENT_CONTRACT_DEPLOY_DEPOSIT)
            .transact()
            .await
            .unwrap();

        assert!(
            execution.is_success(),
            "failed to propose update [i={i}]; {execution:#?}"
        );
        let proposal_id = execution.json().expect("unable to convert into UpdateId");
        proposals.push(proposal_id);
    }

    // Vote for the last proposal
    vote_update_till_completion(&contract, &mpc_signer_accounts, proposals.last().unwrap()).await;

    // Ensure all proposals are removed after update
    for proposal in proposals {
        let voter = mpc_signer_accounts.first().unwrap();
        let execution = voter
            .call(contract.id(), method_names::VOTE_UPDATE)
            .args_json(serde_json::json!({
                "id": proposal,
            }))
            .gas(GAS_FOR_VOTE_UPDATE)
            .transact()
            .await
            .unwrap();
        dbg!(&execution);

        assert!(execution.is_failure());
    }
