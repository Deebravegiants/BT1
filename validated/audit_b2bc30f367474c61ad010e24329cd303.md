Audit Report

## Title
FeeReceiver ETH Balance Excluded from TVL Enables Block Stuffing to Temporarily Underprice rsETH - (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` is an unpermissioned public function and the sole mechanism to move accumulated MEV/execution-layer rewards into `LRTDepositPool`. Until it is called, `FeeReceiver.balance` is invisible to the oracle because `getETHDistributionData()` explicitly excludes it. An attacker can use block stuffing to suppress `sendFunds()` and then call `updateRSETHPrice()` themselves, causing rsETH to be priced below its true value and allowing new depositors to receive more rsETH than they should, diluting existing holders.

## Finding Description
`FeeReceiver.sendFunds()` carries no access control: [1](#0-0) 

`LRTOracle._getTotalEthInProtocol()` iterates supported assets and calls `getTotalAssetDeposits(asset)` for each: [2](#0-1) 

For ETH, `getTotalAssetDeposits` delegates to `getAssetDistributionData`, which routes to `getETHDistributionData()`: [3](#0-2) 

`getETHDistributionData()` counts deposit pool balance, NDC balances, EigenLayer staked/unstaking ETH, converter ETH, and unstaking vault ETH — but never `FeeReceiver.balance`. The codebase itself documents this exclusion: [4](#0-3) [5](#0-4) 

`updateRSETHPrice()` is also callable by any unprivileged address: [6](#0-5) 

The price computed in `_updateRsETHPrice()` is `(totalETHInProtocol - protocolFeeInETH) / rsethSupply`, where `totalETHInProtocol` never includes `FeeReceiver.balance`: [7](#0-6) 

The downside-protection pause logic does not mitigate this: the attack does not push the price *below* `highestRsethPrice`; it merely prevents the price from rising to its true value, so no pause is triggered: [8](#0-7) 

**Exploit flow:**
1. MEV rewards accumulate in `FeeReceiver` (e.g., after a period of high MEV activity).
2. Attacker stuffs the preceding N blocks with high-gas transactions to crowd out any `sendFunds()` call.
3. In the first non-stuffed block, attacker calls `updateRSETHPrice()` directly.
4. Oracle prices rsETH without `FeeReceiver.balance`, producing a price lower than the true value.
5. Attacker (or any new depositor in that window) deposits ETH/LSTs and receives more rsETH than they should, diluting existing holders.

## Impact Explanation
The rsETH price is set below its true value for the duration until `sendFunds()` is eventually called and `updateRSETHPrice()` is called again. During this window, new depositors receive more rsETH than they should, diluting existing holders' share of accrued MEV rewards. This maps to **Low — Contract fails to deliver promised returns, but doesn't lose value** and **Low — Block stuffing**, both of which are in the allowed impact scope.

## Likelihood Explanation
Both `sendFunds()` and `updateRSETHPrice()` are public with no role restriction, so no privileged access is needed. Block stuffing on Ethereum mainnet is expensive but feasible when `FeeReceiver` has accumulated a large balance (e.g., after sustained high-MEV activity), making the profit from mispricing potentially exceed the stuffing cost. The attack is significantly more viable on L2 deployments where block stuffing is far cheaper. The attack is repeatable as long as MEV rewards continue to accumulate.

## Recommendation
Include `FeeReceiver.balance` directly in `getETHDistributionData()` by reading the registered reward receiver address from `LRTConfig` and adding its balance to the TVL computation. This eliminates the gap between accrued rewards and reflected TVL without requiring a separate `sendFunds()` call before each oracle update.

Alternatively, atomically call `sendFunds()` inside `updateRSETHPrice()` (or require it as a precondition) so the oracle always prices against the full ETH balance.

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Foundry fork test demonstrating rsETH price difference
// when sendFunds() is suppressed before updateRSETHPrice()

function test_blockStuffing_rsETHMispricing() public {
    // 1. Simulate MEV rewards accumulating in FeeReceiver
    uint256 mevRewards = 10 ether;
    vm.deal(address(feeReceiver), mevRewards);

    // 2. Attacker calls updateRSETHPrice() directly — FeeReceiver.balance excluded
    lrtOracle.updateRSETHPrice();
    uint256 priceWithoutRewards = lrtOracle.rsETHPrice();

    // 3. Now call sendFunds() and update price again
    feeReceiver.sendFunds();
    lrtOracle.updateRSETHPrice();
    uint256 priceWithRewards = lrtOracle.rsETHPrice();

    // 4. Assert price difference is at least mevRewards / totalSupply
    uint256 rsethSupply = rsETH.totalSupply();
    uint256 expectedDiff = mevRewards * 1e18 / rsethSupply;

    assertGe(
        priceWithRewards - priceWithoutRewards,
        expectedDiff,
        "rsETH price must reflect pending MEV rewards"
    );
}
```

`priceWithoutRewards < priceWithRewards` by at least `FeeReceiver.balance / rsethSupply`, confirming the invariant violation when block stuffing prevents `sendFunds()` from executing before the oracle update.
