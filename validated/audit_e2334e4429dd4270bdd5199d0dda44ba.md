Audit Report

## Title
Missing Chainlink Oracle Staleness and Validity Checks Allow Stale Price Acceptance - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `answer`, performing no validation on staleness (`updatedAt`), round completeness (`answeredInRound` vs `roundId`), or price sign (`price > 0`). The same repository already implements all three guards in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, making the omission a clear and demonstrable inconsistency. A stale inflated price fed into the deposit minting path allows an attacker to mint excess rsETH at the expense of existing holders.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at L52, `getAssetPrice()` destructures `latestRoundData()` as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

`roundId`, `startedAt`, `updatedAt`, and `answeredInRound` are all silently discarded. Three checks are absent:

1. **Staleness**: no `block.timestamp - updatedAt > threshold` guard.
2. **Round completeness**: no `answeredInRound < roundId` guard.
3. **Non-negative price**: `price` is cast directly to `uint256` with no `price <= 0` guard; a zero price yields zero rsETH minted (silent failure), and a negative `int256` wraps to a massive `uint256`.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` at L26–37 correctly implements all three:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

`ChainlinkPriceOracle.getAssetPrice()` is the price source consumed in two critical paths:

**Deposit minting** — `LRTDepositPool.getRsETHAmountToMint()` at L520:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

**rsETH price update** — `LRTOracle._getTotalEthInProtocol()` at L339–343:
```solidity
uint256 assetER = getAssetPrice(asset);
...
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [4](#0-3) 

Exploit flow for the staleness case:
1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale — frozen at 1.0 ETH — while the asset depegs to 0.97 ETH on the open market.
2. Attacker purchases stETH at 0.97 ETH.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, amount, ...)`. `getRsETHAmountToMint` reads the stale 1.0 ETH price, minting rsETH as if stETH is worth 1.0 ETH.
4. Attacker receives ~3% excess rsETH relative to the true value deposited.
5. Existing rsETH holders are diluted by the inflated NAV contribution.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` only fires when the *rsETH price itself* moves beyond a threshold during a price update call; it does not protect individual deposit transactions from consuming a stale asset price. [5](#0-4) 

## Impact Explanation
**High — Theft of unclaimed yield.** When a Chainlink feed is stale and the underlying asset has depegged, any depositor can mint rsETH at the inflated stale rate. The excess rsETH minted represents value extracted from existing rsETH holders' proportional share of the protocol's ETH backing. This is a direct, repeatable theft of yield/value from existing holders, matching the "Theft of unclaimed yield" impact class. A zero or negative price return additionally causes a silent zero-mint or uint256 wraparound, matching "Temporary freezing of funds."

## Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for stETH/ETH). During periods of low volatility the heartbeat may not trigger, leaving the price stale. The protocol is deployed on Arbitrum, Optimism, Base, and Scroll — all L2s where sequencer outages are a known, historically observed risk that can freeze oracle updates for extended periods. No privileged access is required; any external caller can invoke `depositAsset()` during the stale window. The attacker only needs to observe that the on-chain price diverges from the market price, which is publicly visible.

## Recommendation
Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`STALENESS_THRESHOLD` should be configurable per feed and set to the feed's documented heartbeat interval plus a reasonable buffer (e.g., heartbeat + 20%).

## Proof of Concept
**Fork test outline (Foundry):**

1. Fork a network where a stETH/ETH Chainlink feed is deployed.
2. Deploy the protocol contracts with `ChainlinkPriceOracle` pointing to the stETH/ETH feed.
3. Use `vm.mockCall` to make `latestRoundData()` return a price of `1e18` (1.0 ETH) with `updatedAt = block.timestamp - 2 hours` (past any reasonable heartbeat), simulating a stale feed.
4. Separately record the "true" market price as `0.97e18`.
5. Call `LRTDepositPool.depositAsset(stETH, 1e18, 0, "")` as the attacker.
6. Assert that `rsethAmountToMint` equals `1e18 / rsETHPrice` (calculated at 1.0 ETH rate) rather than `0.97e18 / rsETHPrice`.
7. The difference (~3%) represents the excess rsETH minted at the expense of existing holders, confirming the theft of yield.
