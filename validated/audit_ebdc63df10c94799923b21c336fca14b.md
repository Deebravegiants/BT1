### Title
NFT Royalty Bypass via Integer Division Truncation in `compute_royalty_amount` — (File: chia/wallet/nft_wallet/nft_wallet.py)

---

### Summary

`compute_royalty_amount` uses integer floor division (`//`) to compute NFT royalties. For small offered amounts, the royalty truncates to exactly 0. When this happens, the wallet silently skips the royalty spend, omits the royalty announcement assertion, and passes an empty `trade_prices_list` to the on-chain NFT transfer program — fully bypassing on-chain royalty enforcement. An unprivileged maker can exploit this to transfer a royalty-enabled NFT without paying any royalty to the creator.

---

### Finding Description

In `chia/wallet/nft_wallet/nft_wallet.py`, `compute_royalty_amount` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

Because both divisions are integer floor divisions, the result is 0 whenever:

```
abs(offered_amount) // royalty_split * percentage < 10000
```

For a 1 % royalty (100 basis points) this means any offer requesting **fewer than 100 mojos per NFT** produces a royalty of 0. The project's own test suite explicitly documents and asserts this behaviour:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

When `compute_royalty_amount` returns 0, three downstream guards all silently drop the royalty:

**1. Royalty spend is skipped entirely:**
```python
if sum(p.amount for _, p in payments) == 0:
    continue
``` [3](#0-2) 

**2. No royalty announcement is asserted in the offer coin spend:**
```python
for launcher_id, p in payments
    if p.amount > 0
``` [4](#0-3) 

**3. The trade price is filtered out of the `trade_prices_list` passed to the NFT spend:**
```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
``` [5](#0-4) 

The on-chain `NFT_OWNERSHIP_TRANSFER_PROGRAM_ONE_WAY_CLAIM_WITH_ROYALTIES` puzzle receives an empty `trade_prices_list` and therefore has no prices from which to compute or enforce a royalty payment. The NFT is transferred with no royalty coin created.

The same truncation exists in the display-path helper `royalty_calculation`:

```python
"amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
``` [6](#0-5) 

This means the CLI royalty summary shown to the taker also displays 0, so neither party is warned.

---

### Impact Explanation

An unprivileged maker can craft an offer requesting a fungible amount below the royalty-zero threshold (e.g., 99 mojos for a 1 % royalty NFT). The wallet constructs a valid, on-chain-accepted spend bundle with an empty `trade_prices_list`. The NFT is transferred and ownership updated on-chain, but the royalty address receives nothing. The NFT creator suffers a direct, permanent loss of their entitled royalty on every such trade. Because the attack requires only a standard offer creation RPC call with a small amount, it is fully reachable by any unprivileged wallet user.

This constitutes an unauthorized accounting change affecting NFT offer settlement — royalty coins that the protocol is designed to create are never created.

---

### Likelihood Explanation

The threshold is low and fixed by the royalty percentage:

| Royalty % | Threshold (mojos per NFT) |
|-----------|--------------------------|
| 0.01 % (1 bp) | < 1 000 000 |
| 1 % (100 bp) | < 100 |
| 10 % (1000 bp) | < 10 |

A colluding maker and taker agree on a real price off-chain, then the maker lists the NFT on-chain for 1–99 mojos. The taker accepts; the NFT transfers; the royalty address receives 0. The real payment settles off-chain. No special tooling is required — the standard wallet RPC `create_offer_for_ids` with a small amount is sufficient.

---

### Recommendation

Replace floor division with ceiling division so the royalty is never 0 unless the configured percentage is 0:

```python
# Current (vulnerable):
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS

# Fixed (ceiling division, pure integer arithmetic):
numerator = abs(offered_amount) // royalty_split * percentage
amount = (numerator + MAX_ROYALTY_BASIS_POINTS - 1) // MAX_ROYALTY_BASIS_POINTS
```

Apply the same fix to `royalty_calculation` (line 719) and add a guard that rejects offers where the computed royalty would be 0 but the configured percentage is non-zero.

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
