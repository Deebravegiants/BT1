Audit Report

## Title
Instant Withdrawal Fee Rounds to Zero for Dust Amounts Due to Integer Division and Uninitialized Minimum - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`instantWithdrawal()` computes the protocol fee as `(assetAmountUnlocked * instantWithdrawalFee) / 10_000`. For small `rsETHUnstaked` values, Solidity integer division truncates the fee to zero. The only guard against dust amounts — `rsETHUnstaked < minRsEthAmountToWithdraw[asset]` — is ineffective because `minRsEthAmountToWithdraw` is a storage mapping never initialized in `initialize()`, defaulting to zero for every asset. The result is that any positive rsETH amount passes the check, and sub-threshold amounts pay no fee.

## Finding Description

**Root cause — fee calculation (L237):**
```solidity
uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
```
Integer division truncates to zero whenever `assetAmountUnlocked * instantWithdrawalFee < 10_000`.

**`assetAmountUnlocked` derivation (L593):**
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```
For ETH (rsETHPrice ≈ assetPrice ≈ 1e18), `assetAmountUnlocked ≈ rsETHUnstaked`. So the fee truncates to zero when `rsETHUnstaked * instantWithdrawalFee < 10_000`.

**Existing guard is insufficient (L224–226):**
```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```
`minRsEthAmountToWithdraw` is declared at L35 as a plain mapping. `initialize()` (L90–98) never sets it, so it defaults to `0` for every asset. The condition reduces to `rsETHUnstaked == 0`, allowing any positive amount through.

**Fee is silently skipped when zero (L245–248):**
```solidity
if (fee > 0) {
    _transferAsset(asset, feeRecipient, fee);
    emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
}
```
No revert, no event — the fee recipient simply receives nothing.

**Exploit path:**
1. `instantWithdrawalFee = 100` (1%) is set by LRT manager.
2. `minRsEthAmountToWithdraw[ETH]` is never set → defaults to `0`.
3. Caller holds rsETH and calls `instantWithdrawal(ETH, 99, "")`.
4. `assetAmountUnlocked ≈ 99`, `fee = (99 × 100) / 10_000 = 0`.
5. Caller receives 99 wei of ETH; fee recipient receives 0.
6. Repeating across many calls extracts the full underlying with zero fee paid.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The protocol's fee mechanism fails to collect fees on sub-threshold amounts. The fee recipient (treasury or configured address) receives zero instead of the promised fee on every dust-sized instant withdrawal. No user funds are at risk; only protocol revenue is affected. The invariant that every instant withdrawal pays a fee is broken, and the protocol silently under-collects revenue on any call below the rounding threshold.

## Likelihood Explanation

Low. On mainnet, gas costs per call (~50,000 gas × prevailing gas price) vastly exceed the per-call fee savings (at most a few wei), making large-scale exploitation economically irrational. The condition is met by default for any asset until an admin explicitly calls `setMinRsEthAmountToWithdraw`. The vulnerability is passively present for all assets in their default state and requires no special attacker capability beyond holding rsETH.

## Recommendation

1. **Set a protocol-wide floor in `initialize()`** or require `minRsEthAmountToWithdraw[asset] > 0` before enabling instant withdrawal for an asset via `setInstantWithdrawalEnabled`.
2. **Round the fee up** to prevent zero-fee outcomes on dust amounts:
   ```solidity
   uint256 fee = (assetAmountUnlocked * instantWithdrawalFee + 9_999) / 10_000;
   ```
3. **Revert if fee is zero but `instantWithdrawalFee > 0`**, making the zero-fee case explicit rather than silent.

## Proof of Concept

**Foundry fuzz test plan:**

```solidity
function testFuzz_instantWithdrawalFeeRoundsToZero(uint256 rsETHAmount) public {
    // Bound to sub-threshold range for 1% fee: rsETHAmount in [1, 99]
    rsETHAmount = bound(rsETHAmount, 1, 99);

    // Setup: fee = 100 bps, minRsEthAmountToWithdraw[ETH] = 0 (default)
    vm.prank(lrtManager);
    withdrawalManager.setInstantWithdrawalFee(100);
    // minRsEthAmountToWithdraw never set → defaults to 0

    uint256 feeRecipientBefore = feeRecipient.balance;

    vm.prank(attacker);
    withdrawalManager.instantWithdrawal(ETH_TOKEN, rsETHAmount, "");

    // Fee recipient received nothing despite fee being enabled
    assertEq(feeRecipient.balance, feeRecipientBefore);
}
```

**Invariant test:** Assert that for any completed `instantWithdrawal` call where `instantWithdrawalFee > 0`, the `InstantWithdrawalFeeCollected` event is emitted. The invariant fails for all inputs where `assetAmountUnlocked * instantWithdrawalFee < 10_000`.
