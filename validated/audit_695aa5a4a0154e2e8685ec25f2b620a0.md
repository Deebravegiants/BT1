### Title
`directDeposits` Applied to Account Balances Without Inclusion in Produced-Value Balance Check, Enabling ADA Creation from Nothing — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in both top-level and sub-transaction bodies. These deposits are applied to account balances in the `ENTITIES`/`SUBENTITIES` rules via `applyDirectDeposits`. However, `directDeposits` are **not included** in either the `consumed` or `produced` value calculations used by the UTxO balance check (`validateValueNotConservedUTxO`). Any unprivileged transaction author can therefore include arbitrary `directDeposits` amounts, causing ADA to be credited to account balances without being deducted from UTxO inputs — a direct violation of the preservation-of-value invariant.

---

### Finding Description

**Root cause — balance check omits `directDeposits` on the produced side:**

`getConsumedDijkstraValue` computes consumed value as UTxO inputs + refunds + withdrawals (delegating to `getConsumedMaryValue`): [1](#0-0) 

`dijkstraProducedValue` computes produced value as UTxO outputs + fees + cert-deposits + treasury donation (delegating to `conwayProducedValue`), then folds over sub-transactions: [2](#0-1) 

Neither function references `directDepositsTxBodyL`. The `directDeposits` coin amounts are therefore invisible to the balance check.

**The Dijkstra UTxO rule enforces only this balance check:** [3](#0-2) 

**Root cause — `directDeposits` are unconditionally applied to account balances:**

In the `ENTITIES` rule, after cert processing, `applyDirectDeposits` is called with no authorization check and no requirement that UTxO inputs cover the deposited amounts. The only guard is that target accounts must already be registered: [4](#0-3) 

The same pattern applies to sub-transactions in `SUBENTITIES`: [5](#0-4) 

`applyDirectDeposits` unconditionally adds the specified coin to each target account's balance: [6](#0-5) 

**Contrast with treasury donation (correctly handled):** `conwayProducedValue` correctly includes `treasuryDonationTxBodyL` in the produced value, so the UTxO inputs must cover it: [7](#0-6) 

`directDeposits` receive no equivalent treatment.

**Analogy to the external report:** The external report describes a swap router that deducts a fee from the actual transfer but returns the gross amount to the caller, causing the vault to account for more tokens than received. Here the inversion is symmetric: `directDeposits` are credited to account balances (ADA is "received" by accounts) but the balance check does not require the UTxO inputs to cover them (the "fee" side is missing from `produced`). In both cases a value is applied to an accounting ledger without a corresponding deduction on the other side of the balance equation.

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

The preservation-of-value invariant (`consumed = produced`) is the foundational correctness property of the Cardano ledger. Because `directDeposits` are applied to account balances but absent from `produced`, total ADA in the system increases by the sum of all `directDeposits` in every Dijkstra transaction. An attacker can mint unbounded ADA into any registered reward account with a single transaction, constituting direct loss/creation of ADA through an invalid ledger state transition.

---

### Likelihood Explanation

**High.** The entry path requires only:
1. A registered stake credential (trivially obtained by any user).
2. A Dijkstra transaction with a `directDeposits` map targeting that credential.
3. Sufficient UTxO inputs to cover fees and any declared outputs — the `directDeposits` amount itself is unconstrained.

No privileged role, governance majority, or leaked key is required. The `directDeposits` field is freely settable by any transaction author.

---

### Recommendation

Include the sum of all `directDeposits` coin in the produced-value calculation, mirroring the treatment of `treasuryDonationTxBodyL` in `conwayProducedValue`. Concretely, `dijkstraProducedValue` (and `dijkstraSubTxProducedValue`) should add `inject (fold (unDirectDeposits (txBody ^. directDepositsTxBodyL)))` to the produced value, so that UTxO inputs must cover the deposited amounts and the preservation-of-value invariant is restored.

---

### Proof of Concept

1. Register stake credential `cred` (account balance starts at 0).
2. Craft a Dijkstra `TopTx` with:
   - UTxO inputs covering only the transaction fee (e.g., 1 ADA).
   - One UTxO output returning change (e.g., 0.8 ADA).
   - `fee = 0.2 ADA`.
   - `directDeposits = { AccountAddress(cred) → 1_000_000_000_000 }` (1 million ADA).
3. Submit the transaction.
4. **Balance check passes:** `consumed = 1 ADA = produced = 0.8 ADA + 0.2 ADA`. `directDeposits` is absent from both sides.
5. **ENTITIES rule fires:** `applyDirectDeposits` adds 1,000,000 ADA to `cred`'s account balance.
6. Total ADA in the system has increased by 1,000,000 ADA. The attacker can immediately withdraw via a standard withdrawal transaction.

getConsumedDijkstraValue ::
  forall era l.
  ( DijkstraEraTxBody era
  , EraUTxO era
  , Value era ~ MaryValue
  , STxLevel l era ~ STxBothLevels l era
  ) =>
  PParams era ->
  (Credential Staking -> Maybe Coin) ->
  (Credential DRepRole -> Maybe Coin) ->
  UTxO era ->
  TxBody l era ->
  Value era
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue :: forall m. TxBody m era -> Value era
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
