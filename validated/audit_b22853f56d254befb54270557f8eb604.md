### Title
Single Byzantine Participant Can Resolve Foreign-Tx Verification With Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid for the caller-supplied `response.payload_hash` under the root public key, but **never verifies that `payload_hash` was actually derived from the `request` argument**. A single Byzantine attested participant (below signing threshold) who has legitimately obtained a threshold signature for one request can replay it against any other pending foreign-tx verification request, causing the contract to resolve that request with a forged attestation.

---

### Finding Description

The intended invariant is that `payload_hash` = `SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))`, binding the signature to both the specific foreign transaction and the values the MPC nodes observed on-chain.

In `respond_verify_foreign_tx` the contract performs only two checks:

```rust
// crates/contract/src/lib.rs  lines 718-734
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,          // ← root key, no tweak
)
.is_ok()
``` [1](#0-0) 

It then resolves the pending yield keyed on `request`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

There is **no assertion** that `payload_hash` encodes `request`. Compare this with the regular `respond` path, which derives the expected public key from the root key **plus the request-specific tweak**, cryptographically binding the signature to the request:

```rust
// crates/contract/src/lib.rs  lines 597-607
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response, payload_hash, &expected_public_key,
).is_ok()
``` [3](#0-2) 

For foreign-tx, the node intentionally uses a zero tweak:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs  line 43
tweak: Tweak::new([0u8; 32]),
``` [4](#0-3) 

This means every foreign-tx signature is produced under the same root key with no per-request binding enforced on-chain.

**Attack path (single Byzantine participant, below threshold):**

1. Attacker participates honestly in the MPC signing round for legitimate **Request A** (e.g., Bitcoin tx `0xAA…`). The threshold protocol produces signature `S_A` over `H_A = SHA-256(borsh({request_A, values_A}))`.
2. A victim submits **Request B** (e.g., a fraudulent or non-existent Ethereum tx `0xBB…`). It enters `pending_verify_foreign_tx_requests`.
3. Attacker calls `respond_verify_foreign_tx(request_B, { payload_hash: H_A, signature: S_A })`.
4. The contract checks: is `S_A` a valid ECDSA signature for `H_A` under the root key? **Yes** — it was produced by the threshold protocol for request A.
5. The contract resolves Request B's yield queue with `{ payload_hash: H_A, signature: S_A }`.
6. Request B's caller receives a "verified" response whose `payload_hash` encodes **request A's transaction and extracted values**, not request B's.

The `ForeignTxSignPayloadV1` design explicitly commits the request inside the hash:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,   // ← tx_id, chain, extractors
    pub values: Vec<ExtractedValue>,
}
impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
``` [5](#0-4) 

The contract never recomputes or checks this hash against `request`.

---

### Impact Explanation

A bridge contract (the primary consumer of `verify_foreign_transaction`) that does not independently reconstruct and compare `payload_hash` against its expected value will accept the forged attestation. The design document acknowledges that callers *should* verify the hash off-chain:

> "The response contains the hash of the sign payload, so callers can verify the signature by checking it against the expected hash they reconstruct locally." [6](#0-5) 

But the contract provides no on-chain enforcement. A bridge contract that only checks `verify_ecdsa_signature(sig, payload_hash, root_pk)` — a natural and minimal check — would be fooled into releasing funds or executing a cross-chain action for a transaction that was never actually verified on the foreign chain. This constitutes **forged foreign-chain verification causing invalid bridge execution**.

---

### Likelihood Explanation

- Requires the attacker to be a single **attested participant** (Byzantine, below signing threshold) — an explicitly allowed entry path per the audit scope.
- The attacker needs only one legitimately produced threshold signature (obtainable by participating honestly in any prior foreign-tx signing round).
- No threshold collusion, no key leakage, no network-level attack is required.
- The attack is silent: Request A eventually times out; Request B's caller receives a structurally valid response with a valid signature.

---

### Recommendation

In `respond_verify_foreign_tx`, enforce the binding between `payload_hash` and `request` on-chain. Two options:

1. **Include extracted values in the response and verify on-chain.** Change `VerifyForeignTransactionResponse` to carry `values: Vec<ExtractedValue>` alongside `payload_hash`. The contract recomputes `expected_hash = SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))` and asserts `expected_hash == response.payload_hash` before accepting the response.

2. **Bind the signature to the request via a request-specific tweak.** Derive a per-request tweak from `hash(request)` (analogous to how `respond` uses `request.tweak`) and verify the signature against the derived key rather than the root key. This makes a signature produced for request A cryptographically invalid for request B without any additional data in the response.

Option 1 is more transparent (the extracted values become auditable on-chain); option 2 is cheaper in calldata but requires the node to use the same tweak derivation.

---

### Proof of Concept

The existing unit test at `crates/contract/src/lib.rs:3660` demonstrates the gap: it constructs `payload` from `request.request.clone()` and **arbitrary** `values` (`[42u8; 32]`), signs the resulting hash, and the contract accepts it without any check that those values correspond to what the foreign chain actually returned for that request.

```rust
// lib.rs ~3687-3712 (existing test, unmodified)
let payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: vec![ExtractedValue::BitcoinExtractedValue(
        BitcoinExtractedValue::BlockHash([42u8; 32].into()),  // ← arbitrary value
    )],
});
let payload_hash = payload.compute_msg_hash().unwrap().0;
// … sign with root key …
contract.respond_verify_foreign_tx(request.clone(), response)
    .expect("respond_verify_foreign_tx should succeed");  // ← succeeds
``` [7](#0-6) 

To demonstrate the cross-request replay, extend this test: after the first `respond_verify_foreign_tx` call succeeds for `request_A`, enqueue a second distinct `request_B`, then call `respond_verify_foreign_tx(request_B, same_response)`. The contract will resolve `request_B` with `request_A`'s `payload_hash` and signature, confirming the missing binding check.
