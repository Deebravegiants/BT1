### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Response Substitution by a Single Byzantine MPC Leader — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted `response.payload_hash` carries a valid MPC threshold signature. It never checks that `payload_hash` is actually the SHA-256 hash of `ForeignTxSignPayload::V1 { request: request.request, values: <observed> }`. A single malicious MPC leader (one Byzantine participant, strictly below the signing threshold) can take a legitimately-produced threshold signature for pending request **Y** and use it to resolve a completely different pending request **X**, delivering a forged foreign-chain attestation to the caller of **X**.

---

### Finding Description

`verify_foreign_transaction` stores a `VerifyForeignTransactionRequest` keyed only on `(ForeignChainRpcRequest, domain_id, payload_version)` — no caller account ID, no tweak. [1](#0-0) 

When a node calls `respond_verify_foreign_tx`, the contract performs two checks:

1. The caller is an attested participant.
2. `verify_ecdsa_signature(signature_response, &response.payload_hash, &root_pk)` succeeds. [2](#0-1) 

It then resolves all yields queued under `request` with the raw `response` bytes: [3](#0-2) 

**What is never checked**: that `response.payload_hash == SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: … }))` for any set of extracted values. The contract has no way to enforce this because the extracted values are not included in the response DTO: [4](#0-3) 

The MPC nodes independently compute the hash from actual foreign-chain RPC data: [5](#0-4) 

Both leader and follower independently execute the foreign-chain request and build the same `sign_request`: [6](#0-5) [7](#0-6) 

This means the threshold signature is over the correct hash for the request it was produced for. But the contract never binds that signature to its originating request when `respond_verify_foreign_tx` is called.

---

### Impact Explanation

A single malicious MPC leader (one Byzantine participant, below the signing threshold) can:

1. Observe two concurrent pending requests: **X** (e.g., a 100 ETH Ethereum deposit) and **Y** (e.g., a 1 ETH deposit).
2. Participate honestly in signing for **Y**, obtaining `sig_Y` over `hash_Y = SHA-256(borsh(ForeignTxSignPayload::V1 { request: Y, values: [block_hash_Y] }))`.
3. Call `respond_verify_foreign_tx(request: X, response: { payload_hash: hash_Y, signature: sig_Y })`.
4. The contract verifies `sig_Y` over `hash_Y` against the root key — this passes.
5. All yields queued under request **X** are resolved with `{ payload_hash: hash_Y, signature: sig_Y }`.

Every caller who submitted request **X** now receives a `VerifyForeignTransactionResponse` whose `payload_hash` attests to transaction **Y**, not **X**. A bridge contract that does not independently reconstruct and compare the expected hash (i.e., does not call `ForeignChainSignatureVerifier::verify_signature`) will accept this as a valid attestation of **X**, enabling invalid bridge execution (e.g., minting tokens for a deposit that was never made, or minting the wrong amount).

The SDK provides `ForeignChainSignatureVerifier` to catch this: [8](#0-7) 

But the contract itself does not enforce its use, and the `payload_hash` is not bound to the request on-chain.

---

### Likelihood Explanation

- Requires one malicious attested MPC participant acting as leader — a single Byzantine node strictly below the signing threshold.
- Requires two concurrent pending `verify_foreign_transaction` requests (common in any active bridge).
- No key forgery, no threshold collusion, no TEE break needed. The leader simply routes a legitimately-produced threshold signature to the wrong pending request.
- The `VerifyForeignTransactionRequest` key contains no caller identity, so all callers who submitted the same request **X** are simultaneously affected by one call. [9](#0-8) 

---

### Recommendation

The contract should verify that `response.payload_hash` is consistent with `request.request`. The most practical fix is to include the full `ForeignTxSignPayload` (or at minimum the `ForeignChainRpcRequest` it was derived from) in the response, and recompute the hash on-chain:

```rust
// In respond_verify_foreign_tx, after verifying the signature:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // add this field to the response
}).compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, the `VerifyForeignTransactionResponse` should include the extracted values so the contract can recompute and verify the hash, binding the signature to the specific request it was produced for.

---

### Proof of Concept

1. Alice calls `verify_foreign_transaction` for Ethereum tx **X** (100 ETH deposit). Request key = `(EvmRpcRequest { tx_id: X, … }, domain_id, V1)`.
2. Bob calls `verify_foreign_transaction` for Ethereum tx **Y** (1 ETH deposit). Request key = `(EvmRpcRequest { tx_id: Y, … }, domain_id, V1)`.
3. Both requests are pending in `pending_verify_foreign_tx_requests`.
4. The MPC network (with the malicious leader) processes request **Y** and produces `sig_Y` over `hash_Y = SHA-256(borsh(ForeignTxSignPayload::V1 { request: Y, values: [block_hash_Y] }))`.
5. The malicious leader calls:
   ```
   respond_verify_foreign_tx(
     request = VerifyForeignTransactionRequest { request: X, domain_id, V1 },
     response = VerifyForeignTransactionResponse { payload_hash: hash_Y, signature: sig_Y }
   )
   ```
6. The contract checks `verify_ecdsa_signature(sig_Y, hash_Y, root_pk)` — **passes**, because `sig_Y` is a valid threshold signature over `hash_Y`.
7. `resolve_yields_for(&mut pending_verify_foreign_tx_requests, &request_X, serialize(response))` — Alice's yield is resolved with the forged response.
8. Alice's NEAR bridge contract receives `{ payload_hash: hash_Y, signature: sig_Y }`. If it does not call `ForeignChainSignatureVerifier::verify_signature` with the expected 100 ETH values, it accepts this as proof that tx **X** (100 ETH) was verified, and mints 100 ETH worth of bridged tokens — while the MPC network only attested to a 1 ETH deposit.     #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
}
