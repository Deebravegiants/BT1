### Title
Single Attested Participant Can Inject Arbitrary CKD Output for Legacy `AppPublicKey` Requests, Bypassing Threshold Requirement - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` function in the MPC contract verifies the cryptographic correctness of a CKD response only when the request uses the `AppPublicKeyPV` (publicly verifiable) variant. For the legacy `AppPublicKey` variant, the response is accepted with **no cryptographic verification**. Any single attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary `CKDResponse` for any pending legacy CKD request, causing the user to receive attacker-controlled key material. This bypasses the threshold requirement that is the core security guarantee of the MPC network.

### Finding Description

In `respond_ckd`, after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` type: [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which performs a BLS12-381 pairing check verifying `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`: [2](#0-1) 

For `AppPublicKey` (the legacy variant), the branch body is **empty** — no check is performed. The arbitrary `response` is immediately serialized and used to resolve the pending yield: [3](#0-2) 

Contrast this with `respond` (ECDSA/EdDSA), where the signature is cryptographically verified against the derived public key before acceptance: [4](#0-3) 

A forged ECDSA/EdDSA signature cannot pass that check without threshold cooperation. No analogous check exists for legacy CKD responses.

The `AppPublicKey` variant is documented as "privately verifiable" — only the holder of the app secret key can verify the result off-chain. The contract designers accepted that on-chain verification is impossible for this variant. However, this design choice eliminates the threshold enforcement at the contract layer: the contract cannot distinguish a legitimately threshold-computed `(big_y, big_c)` from an arbitrary pair submitted by a single rogue participant.

The attacker entry path:
1. A user submits `request_app_private_key` with `AppPublicKey` variant and 1 yoctoNEAR deposit. The request is stored as a pending yield.
2. A single attested participant (one of the MPC nodes, below threshold) calls `respond_ckd` with the matching `CKDRequest` and an arbitrary `CKDResponse{big_y: <attacker_point>, big_c: <attacker_point>}`.
3. `assert_caller_is_attested_participant_and_protocol_active()` passes (the attacker is a legitimate participant).
4. The `AppPublicKey` branch executes with no check.
5. `resolve_yields_for` resolves the pending yield with the attacker-controlled bytes.
6. The user's transaction receives the attacker-controlled `(big_y, big_c)` as the CKD result. [5](#0-4) 

### Impact Explanation

The user computes their derived private key as `big_c − app_secret_key · big_y`. With attacker-controlled `big_y` and `big_c`, this yields an arbitrary point unrelated to the MPC network's actual threshold-derived key. The user's derived key is corrupted. This is **confidential key derivation output produced without the required threshold participant authorization** — a single Byzantine participant below the signing threshold unilaterally resolves a CKD request, bypassing the threshold guarantee that is the entire security basis of the MPC network.

### Likelihood Explanation

Any one of the current attested MPC participants can execute this attack against any pending legacy `AppPublicKey` CKD request. The `AppPublicKey` format is the default legacy format accepted by the contract and used by the reference CLI tool: [6](#0-5) 

The attacker needs only to be an attested participant — a condition already satisfied by every active MPC node. No threshold collusion, leaked keys, or privileged operator access is required.

### Recommendation

For the `AppPublicKey` variant, the contract cannot perform a pairing-based on-chain check because it does not have access to the app's secret key. Two mitigations are possible:

1. **Deprecate `AppPublicKey` in favor of `AppPublicKeyPV`**: Require all new CKD requests to use the publicly verifiable variant, which allows full on-chain verification. Reject `AppPublicKey` requests at `request_app_private_key` time.
2. **Require threshold-of-participants to submit matching responses**: Before resolving a legacy CKD yield, require that at least `threshold` distinct attested participants have submitted identical `(big_y, big_c)` values. Only resolve when the threshold is met. This mirrors how the off-chain MPC protocol actually works and prevents a single participant from unilaterally injecting a result.

### Proof of Concept

```
1. Deploy the MPC contract in Running state with threshold=2, participants=[A, B, C].
2. User calls request_app_private_key({app_public_key: AppPublicKey(<pk>), derivation_path: "x", domain_id: 0})
   with 1 yoctoNEAR deposit. A pending CKDRequest is stored.
3. Participant A (attested, but acting alone — below threshold=2) calls:
   respond_ckd(
     request = <the pending CKDRequest>,
     response = CKDResponse { big_y: [1u8;48], big_c: [2u8;48] }  // arbitrary garbage
   )
4. assert_caller_is_attested_participant_and_protocol_active() passes (A is attested).
5. AppPublicKey branch: no check executed.
6. resolve_yields_for resolves the yield with the garbage response.
7. User's transaction returns CKDResponse { big_y: [1u8;48], big_c: [2u8;48] }.
8. User computes derived_key = big_c - app_secret * big_y = garbage point, not the MPC-derived key.
   The threshold protocol was never run; participant B and C were never involved.
```

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
