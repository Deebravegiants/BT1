### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Redirect Sequencer Fees and Corrupt Block Hash — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates many fields of a received `ProposalInit` (height, L1 DA mode, L2 gas price, L1 gas prices, timestamp, starknet version, version constant commitment, fee proposal) but never validates the `builder` field. That field is passed verbatim into `convert_to_sn_api_block_info` as `sequencer_address`, which is the address that receives transaction fees and is committed into the block hash. A malicious consensus proposer can set `builder` to any address, causing every validator to execute the block with an attacker-controlled sequencer address, redirecting all block fees and producing a block hash that encodes the wrong sequencer.

---

### Finding Description

`ProposalInit` carries a `builder` field described as "Address of the one who builds/sequences the block." [1](#0-0) 

During proposal validation, `validate_proposal` calls `is_proposal_init_valid` before forwarding the proposal to the batcher: [2](#0-1) 

`is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, `timestamp`, all four L1 gas price fields, and `fee_proposal_fri`. The `builder` field is never checked: [3](#0-2) 

`ProposalInitValidation` — the struct that carries the validator's local reference values — has no `builder` field at all: [4](#0-3) 

After `is_proposal_init_valid` returns `Ok`, `initiate_validation` calls `convert_to_sn_api_block_info(init)`, which maps `init.builder` directly to `sequencer_address`: [5](#0-4) 

This `BlockInfo` (with the attacker-supplied `sequencer_address`) is sent to the batcher via `validate_block`: [6](#0-5) 

The batcher executes the entire block under this `BlockInfo`. In Starknet, `sequencer_address` is the fee recipient for every transaction in the block and is a committed component of the block hash. Because every validator uses the proposer's `builder` value, all validators compute the same (attacker-influenced) commitment, so the `ProposalFinMismatch` check passes: [7](#0-6) 

The proposer's `builder` is set from a local config value (`args.builder_address`) during `initiate_build`: [8](#0-7) 

A malicious proposer simply substitutes an attacker-controlled address for `builder` before broadcasting the `ProposalInit`.

---

### Impact Explanation

**Incorrect fee accounting with direct economic impact (Critical scope).** Every transaction fee in the block is paid to `sequencer_address`. By setting `builder` to an attacker-controlled address, the malicious proposer causes all validators to execute the block with that address as the fee recipient. All block fees are redirected to the attacker. Additionally, `sequencer_address` is part of `PartialBlockHashComponents`, so the committed block hash encodes the attacker's address rather than the legitimate sequencer's address, corrupting the canonical chain state.

---

### Likelihood Explanation

**Medium.** The attacker must be selected as the block proposer by the Tendermint consensus protocol. In a decentralized validator set, any validator can be elected proposer. No other precondition is required — the attack is a single-field substitution in the `ProposalInit` message the proposer broadcasts. The attack is silent: no error is raised, no metric fires, and the committed block looks structurally valid to all participants.

---

### Recommendation

Add `builder` to `ProposalInitValidation` and enforce it in `is_proposal_init_valid`:

```rust
// In ProposalInitValidation:
pub builder: ContractAddress,

// In is_proposal_init_valid, add:
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

The expected `builder` value should be populated from the node's own configured builder address when constructing `ProposalInitValidation` in `set_height_and_round` / `validate_current_round_proposal`, mirroring how `l2_gas_price_fri` and `l1_da_mode` are sourced from local config.

---

### Proof of Concept

1. Attacker is elected proposer for height H.
2. Attacker calls `build_proposal` normally but, before broadcasting `ProposalPart::Init`, replaces `init.builder` with `attacker_address`.
3. Validators receive the `ProposalInit`, call `is_proposal_init_valid` — no check on `builder` — and proceed.
4. `initiate_validation` calls `convert_to_sn_api_block_info(init)` → `sequencer_address = attacker_address`.
5. Batcher executes all transactions; every fee is credited to `attacker_address`.
6. Batcher returns `partial_block_hash` that encodes `attacker_address` as sequencer.
7. `proposal_commitment_from(partial_block_hash, fee_proposal)` matches the proposer's `ProposalFin.proposal_commitment` (both sides used the same `builder`).
8. `ProposalFinMismatch` check passes; block is committed with `attacker_address` as sequencer and all fees transferred to attacker.
