### Title
Single Reverting Price Oracle in Loop Permanently Blocks `updateRSETHPrice()` and Downside-Protection Pause — (File: `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle._getTotalEthInProtocol()` iterates over every supported asset and calls an external price oracle for each one with no `try/catch`. If any single oracle reverts (e.g., a Chainlink circuit-breaker, a stale-price guard, or a paused aggregator), the entire `updateRSETHPrice()` call reverts. Because the downside-protection auto-pause lives inside `_updateRsETHPrice()`, a stuck oracle also disables the protocol's only automated safety mechanism against slashing-induced price drops.

---

### Finding Description

`_getTotalEthInProtocol()` is a private helper called unconditionally by `_updateRsETHPrice()`:

```solidity
// LRTOracle.sol  lines 336-348
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);          // ← external call, no try/catch
    uint256 totalAssetAmt =
        ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    totalETHInProtocol += totalAssetAmt.mulWad(assetER);
    unchecked { ++assetIdx; }
}
```

`getAssetPrice(asset)` delegates to the registered `IPriceFetcher`:

```solidity
// LRTOracle.sol  line 157
return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
```

Chainlink-based fetchers (e.g., `ChainlinkPriceOracle.sol`) revert when the answer is stale, the sequencer is down, or the aggregator is paused — all normal market-stress conditions. A single such revert propagates up through `_getTotalEthInProtocol()` → `_updateRsETHPrice()` → `updateRSETHPrice()` / `updateRSETHPriceAsManager()`, making both entry points permanently unusable until the oracle recovers.

The downside-protection block that auto-pauses the protocol sits inside the same function that is now bricked:

```solidity
// LRTOracle.sol  lines 270-281
if (newRsETHPrice < highestRsethPrice) {
    ...
    if (isPriceDecreaseOffLimit) {
        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!withdrawalManager.paused()) withdrawalManager.pause();
        _pause();
        return;
    }
}
```

If the oracle for asset A reverts precisely because A's price is crashing (triggering a Chainlink circuit-breaker), the protocol can never compute the new lower rsETH price, the auto-pause never fires, and the stale (inflated) `rsETHPrice` stored on-chain remains the basis for all new deposits and withdrawals.

---

### Impact Explanation

**Severity: High**

Two compounding harms:

1. **Permanent freezing of price updates** — `rsETHPrice` is frozen at its last value. New depositors receive rsETH priced at the stale (higher) rate, diluting existing holders. New withdrawers receive more underlying than they are entitled to. This is direct, at-rest fund theft from existing rsETH holders.

2. **Bypass of downside-protection pause** — The automated circuit-breaker that should halt the protocol during a slashing event or rapid NAV decline cannot execute. The protocol continues to accept deposits and process withdrawals at an incorrect price, compounding losses.

---

### Likelihood Explanation

**Likelihood: Medium**

Chainlink aggregators revert or return stale data during:
- L2 sequencer outages (Arbitrum, Optimism — both chains where rsETH pools operate)
- Rapid price moves that exceed deviation thresholds before a new round is confirmed
- Aggregator contract pauses or upgrades

The protocol supports multiple LST assets (stETH, ETHx, sfrxETH, etc.), each with its own oracle. The probability that *at least one* oracle reverts during a market-stress event — exactly when the downside-protection pause is most needed — is non-trivial. The two failure modes (oracle stress and asset price crash) are positively correlated.

---

### Recommendation

Wrap each oracle call in a `try/catch` inside `_getTotalEthInProtocol()`. On revert, either skip the asset (treating its contribution as zero) or revert with a dedicated error that does **not** prevent the downside-protection path from executing. Alternatively, separate the price-aggregation logic from the auto-pause logic so that a single oracle failure cannot suppress the safety mechanism.

```solidity
try IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset) returns (uint256 price) {
    totalETHInProtocol += totalAssetAmt.mulWad(price);
} catch {
    // log and skip, or revert with a flag that still allows the pause path
}
```

---

### Proof of Concept

1. Protocol has two supported assets: stETH (oracle OK) and ETHx (Chainlink aggregator paused due to rapid price drop).
2. Anyone calls `updateRSETHPrice()`.
3. `_getTotalEthInProtocol()` reaches the ETHx iteration; `getAssetPrice(ETHx)` reverts.
4. The entire call reverts; `rsETHPrice` is not updated.
5. The downside-protection block (lines 270–281) never executes; `LRTDepositPool` and `LRTWithdrawalManager` are never paused.
6. Users continue depositing ETH/stETH and receive rsETH priced at the stale (pre-crash) rate, extracting value from existing holders.
7. The oracle recovers hours later; by then the damage is done.

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
