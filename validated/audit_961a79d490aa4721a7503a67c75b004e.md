### Title
Missing `builder` Address Validation in `is_proposal_init_valid` Allows Malicious Proposer to Commit Wrong Sequencer Address into Block Hash — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` validates many `ProposalInit` fields (timestamp, starknet version, L1/L2 gas prices, `l1_da_mode`, `version_constant_commitment`) but never validates the `builder` field. The `builder` address is directly used as the `sequencer_address` in `BlockInfo`, which is then hashed into `PartialBlockHashComponents` and committed as the `ProposalCommitment`. A malicious proposer can set `builder = ContractAddress::default()` (zero) or any arbitrary address; validators will accept the proposal, execute the block with the wrong sequencer address, and commit a wrong block hash.

### Finding Description

`ProposalInit.builder` is the sequencer/builder address for the block. In `convert_to_sn_api_block_info` it is mapped directly to `sequencer_address`:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs:329-332
Ok(starknet_api::block::BlockInfo {
    ...
    sequencer_address: init.builder,
    ...
})
```

This `BlockInfo` is passed to both `propose_block` and `validate_block` in the batcher. Inside `BlockExecutionArtifacts::new`, `PartialBlockHashComponents::new(&block_info, header_commitments)` captures `sequencer_address` and it is hashed into the final `ProposalCommitment` via `calculate_block_hash`.

The `ProposalInitValidation` struct used by validators contains no `builder` field:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:75-85
pub(crate) struct ProposalInitValidation {
    pub height: BlockNumber,
    pub block_timestamp_window_seconds: u64,
    pub previous_proposal_init: Option<PreviousProposalInitInfo>,
    pub l1_da_mode: L1DataAvailabilityMode,
    pub l2_gas_price_fri: GasPrice,
    pub starknet_version: StarknetVersion,
    pub fee_actual: Option<GasPrice>,
}
```

And `is_proposal_init_valid` never checks `init_proposed.builder`:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:312-321
if !(init_proposed.height == proposal_init_validation.height
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
{
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
```

`convert_to_sn_api_block_info` does log a warning for zero gas prices and then rejects them via `NonzeroGasPrice::new(...)`, but it performs **no check** on `init.builder` — a zero `ContractAddress` is silently accepted:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs:304-332
if init.l1_gas_price_fri.0 == 0 || ... {
    warn!("Zero gas price detected in block info: {:?}", init);
}
// ... NonzeroGasPrice::new rejects zero gas prices ...
// but:
sequencer_address: init.builder,   // no zero-check, no validation
```

### Impact Explanation

A malicious proposer sets `builder = ContractAddress::default()` (Felt::ZERO) in `ProposalInit`. Validators pass `is_proposal_init_valid` (no builder check), pass `convert_to_sn_api_block_info` (no builder check), and execute the block with `sequencer_address = 0`. Both proposer and validator compute the same `PartialBlockHashComponents` (both use the attacker-supplied zero builder), so the `ProposalFinMismatch` check also passes. The block is committed with:

1. **Wrong block hash** — `sequencer_address = 0` is hashed into `PartialBlockHashComponents` → wrong `ProposalCommitment` → wrong on-chain state root commitment.
2. **Wrong `get_sequencer_address` syscall result** — all contracts calling `get_sequencer_address` during that block receive `0` instead of the real sequencer address.
3. **Wrong fee recipient** — sequencer fees are credited to address `0`.

This matches: *Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*

### Likelihood Explanation

Requires a malicious consensus proposer (a validator-set participant). In BFT consensus the security model tolerates up to f < n/3 Byzantine validators; a single malicious proposer in its assigned round can trigger this. Validators perform no independent check on `builder`, so the attack succeeds unconditionally whenever the malicious node is the round proposer.

### Recommendation

1. Add a zero-address check for `builder` (and `proposer`) in `is_proposal_init_valid`:

```rust
if init_proposed.builder == ContractAddress::default() {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        "builder address must not be zero".to_string(),
    ));
}
```

2. Add `expected_builder: ContractAddress` to `ProposalInitValidation` so each validator can compare the received `builder` against its own configured `builder_address`, analogous to how `l2_gas_price_fri` and `l1_da_mode` are cross-checked.

3. Add a corresponding check in `convert_to_sn_api_block_info` to return an error (not just a warning) for a zero `builder`.

### Proof of Concept

1. Malicious proposer node is selected for round R at height H.
2. It constructs `ProposalInit { builder: ContractAddress::default(), ... }` with all other fields valid.
3. It streams this `ProposalInit` to peer validators.
4. Each validator calls `is_proposal_init_valid` → passes (no builder check).
5. Each validator calls `convert_to_sn_api_block_info` → succeeds, `sequencer_address = Felt::ZERO`.
6. Each validator calls `batcher.validate_block` with `block_info.sequencer_address = 0`.
7. Blockifier executes all transactions; `get_sequencer_address` syscall returns `0` for every transaction.
8. `PartialBlockHashComponents::new` captures `sequencer = SequencerContractAddress(Felt::ZERO)`.
9. `calculate_block_hash` hashes `Felt::ZERO` as the sequencer field → wrong `ProposalCommitment`.
10. Both proposer and validators compute the same wrong commitment → `ProposalFinMismatch` check passes.
11. Consensus reaches decision; `decision_reached` commits the block with wrong sequencer address and wrong block hash.

```rust
    })
```

```rust
pub fn calculate_block_hash(
    partial_block_hash_components: &PartialBlockHashComponents,
    state_root: GlobalRoot,
    previous_block_hash: BlockHash,
) -> StarknetApiResult<BlockHash> {
    let block_hash_version: BlockHashVersion =
        partial_block_hash_components.starknet_version.try_into()?;
    let block_commitments = &partial_block_hash_components.header_commitments;
    Ok(BlockHash(
        HashChain::new()
            .chain(&block_hash_version.clone().into())
            .chain(&partial_block_hash_components.block_number.0.into())
            .chain(&state_root.0)
            .chain(&partial_block_hash_components.sequencer.0)
            .chain(&partial_block_hash_components.timestamp.0.into())
            .chain(&block_commitments.concatenated_counts)
            .chain(&block_commitments.state_diff_commitment.0.0)
            .chain(&block_commitments.transaction_commitment.0)
            .chain(&block_commitments.event_commitment.0)
            .chain(&block_commitments.receipt_commitment.0)
            .chain_iter(
                gas_prices_to_hash(
                    &partial_block_hash_components.l1_gas_price,
                    &partial_block_hash_components.l1_data_gas_price,
                    &partial_block_hash_components.l2_gas_price,
                    &block_hash_version,
                )
                .iter(),
            )
            .chain(
                &Felt::try_from(&partial_block_hash_components.starknet_version)
                    .expect("Expect ASCII version"),
            )
            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)
            .get_poseidon_hash(),
    ))
}
```
