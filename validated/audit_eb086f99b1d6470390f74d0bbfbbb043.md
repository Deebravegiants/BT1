### Title
Queue Saturation DoS on `verify_foreign_transaction` Fan-Out Cap â€” (`File: crates/contract/src/pending_requests.rs`)

### Summary

The `verify_foreign_transaction` fan-out queue is keyed on `VerifyForeignTransactionRequest` which contains no caller identity. Any unprivileged account can fill the 128-slot cap for a specific foreign-chain transaction request, causing every subsequent legitimate submission for that same request to panic with `PendingRequestQueueFull`, permanently blocking the caller's bridge operation until the attacker stops refilling the queue.

### Finding Description

`push_pending_yield` enforces a hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` entries per request key: [1](#0-0) 

The map key for `verify_foreign_transaction` is `VerifyForeignTransactionRequest`, which contains only `(request, domain_id, payload_version)` â€” no caller account ID: [2](#0-1) 

This is by design: the contract explicitly allows different callers to share the same queue entry so one MPC response fans out to all of them: [3](#0-2) 

Because there is no per-account sub-limit inside `push_pending_yield`, a single attacker account can submit the same `verify_foreign_transaction` request 128 times in rapid succession, filling the queue entirely. Any subsequent call by a legitimate user for the same foreign-chain transaction will hit the cap and panic: [4](#0-3) 

After the MPC nodes respond and drain the queue via `resolve_yields_for`, the attacker can immediately resubmit 128 more requests before the legitimate user retries, sustaining the DoS indefinitely at negligible cost (128 Ă— 1 yoctonear + gas per cycle).

The `sign` endpoint is **not** affected because its key is `SignatureRequest { tweak, payload, domain_id }` where `tweak` is derived from `(predecessor_id, path)`, making each caller's queue entry unique: [5](#0-4) 

### Impact Explanation

A targeted attacker can prevent any specific foreign-chain transaction from being verified by a legitimate user. In a bridge context this means:

- A user's `verify_foreign_transaction` call panics with `PendingRequestQueueFull`.
- The user's NEAR transaction reverts; no yield is registered; the user receives no response.
- The MPC nodes still process and respond to the attacker's 128 queued yields (which are worthless to the attacker), draining the queue.
- The attacker immediately refills the queue, blocking the user's next retry.
- Bridge withdrawals or cross-chain operations that depend on this verification are permanently stalled for the targeted transaction.

This matches the **Medium** allowed impact: *"request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

### Likelihood Explanation

**Medium.** The attack requires:
- Knowledge of the specific `(ForeignChainRpcRequest, domain_id, payload_version)` tuple the victim will submit â€” observable on-chain or predictable from the foreign-chain transaction ID.
- 128 NEAR transactions each attaching 1 yoctonear. Cost is negligible.
- No privileged role, no TEE access, no threshold collusion.

The attacker must sustain the attack across MPC response cycles, but each cycle costs only gas. An attacker with a financial motive (e.g., shorting a bridged asset whose unlock depends on the blocked verification) has clear incentive.

### Recommendation

1. **Add a per-account sub-limit inside `push_pending_yield`**: store `Map<AccountId, u8>` alongside the yield vector and cap each account's contribution (e.g., 1â€“4 slots). This preserves the fan-out feature while preventing a single actor from monopolising the queue.

2. **Alternatively, include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key** (as is done for `sign` via the tweak). This gives each caller an independent queue, eliminating cross-caller interference entirely, at the cost of losing the fan-out deduplication benefit for foreign-tx requests.

3. **Increase the deposit requirement** for `verify_foreign_transaction` to make queue-filling economically costly.

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
