### Title
`psVRFKeyHashes` Reference Count Zeroed on Pool Re-registration When Duplicate VRF Keys Exist - (`File: eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

### Summary

The `POOLREAP` transition rule incorrectly removes an entire VRF key hash entry from `psVRFKeyHashes` (using `Map.withoutKeys`) when a pool re-registers with a new VRF key, even when another pool still holds the same VRF key. This breaks the reference-count invariant introduced at protocol version 11, allowing a new pool to register with a VRF key that is still actively used by another pool, bypassing the post-v11 duplicate-VRF-key enforcement.

### Finding Description

`PState` maintains two related mappings:

- `psStakePools :: Map (KeyHash StakePool) StakePoolState` — primary map; each `StakePoolState` embeds the pool's active VRF key via `spsVrf`.
- `psVRFKeyHashes :: Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)` — secondary reference-count index introduced at protocol version 11 to enforce the "no duplicate VRF keys" invariant. [1](#0-0) 

At the v11 hard fork, `populateVRFKeyHashes` correctly counts all VRF keys across `psStakePools` and `psFutureStakePoolParams`, including pre-v11 duplicates (count > 1): [2](#0-1) 

The test suite explicitly exercises the case where three pools share the same VRF key after the hard fork, confirming the developers anticipated pre-v11 duplicates: [3](#0-2) 

When a pool re-registers with a new VRF key, `poolTransition` stages the new params in `psFutureStakePoolParams` and updates `psVRFKeyHashes` via `updateFutureVRFKeyHash`: [4](#0-3) 

At the epoch boundary, `poolReapTransition` computes `danglingVRFKeyHashes` — the set of VRF keys that are in `psStakePools` but being replaced by a different VRF in `psFutureStakePoolParams`: [5](#0-4) 

It then removes these dangling VRF keys from `psVRFKeyHashes` using `Map.withoutKeys`: [6](#0-5) 

**The bug**: `Map.withoutKeys danglingVRFKeyHashes` deletes the entire map entry for the dangling VRF key, regardless of its reference count. If `psVRFKeyHashes[V1] = 2` (two pools share VRF `V1` from pre-v11 state) and one pool re-registers with `V2`, `V1` becomes "dangling" for that pool. At the epoch boundary, `Map.withoutKeys` removes `V1` entirely from `psVRFKeyHashes`, even though the second pool still holds `V1` in `psStakePools`.

Contrast this with the correct handling for pool retirement, which uses `removeVRFKeyHashOccurrences` (decrement-and-remove-if-zero): [7](#0-6) 

### Impact Explanation

After `V1` is incorrectly removed from `psVRFKeyHashes`, the `VRFKeyHashAlreadyRegistered` check in `poolTransition` passes for `V1` (since `Map.notMember V1 psVRFKeyHashes` is now `True`): [8](#0-7) 

A new pool can now register with `V1`, creating a post-v11 duplicate VRF key. Two pools sharing the same VRF key produce identical VRF proofs for the same slot, meaning both can claim slot leadership for the same slot. This violates the post-v11 ledger invariant and can cause deterministic disagreement between honest nodes during block validation and leader election — matching the **High** impact category: *"Deterministic disagreement between honest nodes from ledger rule evaluation, era transition, serialization, or script/witness validation."*

This requires a pre-v11 ledger state containing at least two pools with the same VRF key. populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
