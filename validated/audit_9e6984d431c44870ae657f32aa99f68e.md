### Title
No Minimum Output Amount in L2 Pool `deposit()` Functions Exposes Users to Oracle Rate Slippage - (File: contracts/pools/RSETHPool.sol, contracts/agETH/AGETHPoolV3.sol)

---

### Summary

The `deposit()` functions in `RSETHPool.sol` and `AGETHPoolV3.sol` accept ETH or supported tokens from users and return rsETH/agETH based on the oracle rate fetched at execution time. There is no `minOutputAmount` parameter, so users cannot enforce a minimum amount of rsETH/agETH they are willing to receive. If the oracle rate changes between when the user previewed the rate off-chain and when the transaction is mined, the user silently receives fewer tokens than expected with no on-chain recourse.

---

### Finding Description

In `RSETHPool.sol`, the ETH deposit path is:

```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [1](#0-0) 

`viewSwapRsETHAmountAndFee` computes the output amount using the live oracle rate:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [2](#0-1) 

The same pattern applies to the token deposit overload: [3](#0-2) 

And identically in `AGETHPoolV3.sol`: [4](#0-3) [5](#0-4) 

Neither function accepts a `minRsETHAmount` / `minAgETHAmount` parameter. The user has no on-chain mechanism to reject execution if the oracle rate has moved unfavorably since they previewed the swap.

By contrast, the mainnet `LRTDepositPool.sol` correctly accepts and enforces a `minRSETHAmountExpected` parameter: [6](#0-5) [7](#0-6) 

The L2 pool contracts are missing this protection entirely.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews the rate via `viewSwapRsETHAmountAndFee` off-chain and then submits a deposit transaction may receive fewer rsETH/agETH than expected if the oracle rate is updated before their transaction is mined. The user's ETH is not stolen — it is held in the pool — but the rsETH/agETH minted to them is less than the amount they agreed to accept. There is no on-chain slippage guard to revert the transaction in this case.

---

### Likelihood Explanation

**Medium.** Oracle rate updates for rsETH/agETH are routine (they track staking yield accrual). On L2 networks, transactions can sit in the mempool for multiple blocks, and oracle updates can occur in the interim. Any user who previews the rate and then deposits faces this exposure on every transaction. No attacker action is required — legitimate oracle updates are sufficient to trigger the discrepancy.

---

### Recommendation

Add a `minOutputAmount` parameter to all `deposit()` overloads in `RSETHPool.sol` and `AGETHPoolV3.sol`, and revert if the computed output falls below it:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert SlippageExceeded();
    ...
}
```

This mirrors the protection already present in `LRTDepositPool.sol`.

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and observes they will receive `X` rsETH at the current oracle rate.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, the rsETH oracle rate is updated upward (rsETH appreciates vs ETH — a routine event).
4. The deposit executes: `rsETHAmount = (1 ether - fee) * 1e18 / newHigherRate`, yielding `Y < X` rsETH.
5. The user receives `Y` rsETH with no revert and no recourse. The shortfall `(X - Y)` rsETH of expected value is silently lost to the rate movement.
6. No attacker action is required; the discrepancy arises from a legitimate oracle update between preview and execution.
