### Title
User-Controlled `confirmations` Parameter Bypasses Bitcoin Finality Enforcement in `verify_foreign_transaction` - (File: crates/contract/src/lib.rs)

### Summary

The `verify_foreign_transaction` contract entry point accepts a caller-supplied `BitcoinRpcRequest::confirmations` value with no minimum enforcement. Because the MPC nodes use this value verbatim as the finality threshold, a caller can set `confirmations: 0` (or `confirmations: 1`) and receive a valid MPC threshold signature attesting to a Bitcoin transaction that has not reached any meaningful finality. Downstream bridge contracts that trust the MPC attestation without independently re-validating the `confirmations` field in the signed payload will accept this as proof of finality, enabling double-spend attacks.

---

### Finding Description

`verify_foreign_transaction` in `crates/contract/src/lib.rs` accepts a `VerifyForeignTransactionRequestArgs` whose inner `BitcoinRpcRequest` carries a caller-chosen `confirmations: BlockConfirmations` field. The contract performs only three checks before enqueuing the request:

1. Domain purpose is `ForeignTx`
2. Gas and deposit requirements are met
3. The requested chain is in the supported-chains list [1](#0-0) 

There is **no minimum `confirmations` value enforced**. The value is stored verbatim in the pending request and forwarded to the MPC nodes.

On the node side, `execute_foreign_chain_request` in `crates/node/src/providers/verify_foreign_tx/sign.rs` extracts the caller-supplied value and passes it directly to `BitcoinInspector::extract` as `block_confirmations_threshold`: [2](#0-1) 

Inside `BitcoinInspector::extract`, the only finality check is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [3](#0-2) 

With `block_confirmations_threshold = 0`, this condition is `0 <= actual_confirmations`, which is trivially true for any transaction that has even one confirmation (or is in the mempool with 0 confirmations). The inspector then proceeds to sign the payload.

The signed payload (`ForeignTxSignPayloadV1`) commits to the full `request` including the `confirmations` field: [4](#0-3) 

So the MPC network produces a valid threshold signature over `(tx_id, confirmations: 0, block_hash)`. Any downstream bridge contract that does not independently re-validate the `confirmations` value in the signed payload — trusting the MPC network to have enforced a reasonable minimum — will accept this as proof of finality.

---

### Impact Explanation

This is a **forged foreign-chain verification / light-client-style verification bypass** that enables double-spend conditions on bridge protocols.

**Concrete attack scenario (Omnibridge inbound flow):**

1. Attacker broadcasts a Bitcoin transaction sending funds to the bridge deposit address. The transaction is in the mempool (0 confirmations) or has 1 confirmation.
2. Attacker calls `verify_foreign_transaction` with `confirmations: 0` (or `1`) for that `tx_id`.
3. MPC nodes query the Bitcoin RPC, find the transaction, pass the `0 <= 0` (or `0 <= 1`) check, and produce a valid threshold signature.
4. Attacker presents the MPC signature to the Omnibridge NEAR contract to claim the bridged funds.
5. Attacker simultaneously double-spends the Bitcoin transaction (RBF or miner collusion), invalidating the deposit.
6. The bridge has released NEAR-side funds for a Bitcoin transaction that never finalized.

The MPC network is the designated trusted verifier of foreign chain state — its signature is the root of trust for the bridge. By allowing the caller to set `confirmations: 0`, the protocol's finality guarantee is entirely nullified.

---

### Likelihood Explanation

- **Entry path**: Any unprivileged NEAR account can call `verify_foreign_transaction` with a 1 yoctoNEAR deposit. No special role or privilege is required.
- **No collusion needed**: The attack requires only a single caller and a Bitcoin transaction in the mempool. No MPC node collusion is needed — the nodes faithfully execute the caller-supplied threshold.
- **Economically motivated**: The attacker profits by claiming bridge funds for an unconfirmed or reorg-able Bitcoin transaction.
- **Realistic**: Bitcoin 1-confirmation reorgs occur in practice; 0-confirmation double-spends via RBF are trivial.

---

### Recommendation

Enforce a **minimum `confirmations` value** at the contract level before enqueuing the request. The minimum should be a governance-controlled parameter (e.g., stored in the contract's `config`) rather than a hardcoded constant, to allow adjustment as Bitcoin network conditions change. A reasonable default is 6 confirmations.

```rust
// In verify_foreign_transaction, before enqueue_yield_request:
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    let min_confirmations = self.config.bitcoin_min_confirmations; // e.g., 6
    if btc_req.confirmations.0 < min_confirmations {
        env::panic_str("Bitcoin confirmations below minimum required");
    }
}
```

Additionally, the `near-mpc-sdk`'s `ForeignChainSignatureVerifier::verify_signature` should document that callers **must** validate the `confirmations` field in the returned `request` against their own minimum, since the MPC contract does not currently enforce one.
