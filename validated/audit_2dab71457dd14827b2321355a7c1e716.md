Audit Report

## Title
Missing Minimum Output Guard in L2 Pool `deposit()` Functions Allows Silent Shortfall - (`contracts/pools/RSETHPoolV3.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPool.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPoolV2ExternalBridge.sol`)

## Summary
Every L2 pool `deposit()` function computes the wrsETH/rsETH output amount using a live oracle rate at execution time but accepts no `minRsETHAmountOut` parameter. Because the rsETH/ETH rate monotonically increases as staking rewards accrue and is pushed cross-chain on a regular schedule, any transaction that sits in the mempool between a user's preview call and its inclusion can silently deliver materially fewer tokens than the user expected, with no on-chain revert path. The L1 `LRTDepositPool` already enforces a `minRSETHAmountExpected` guard on its equivalent deposit functions, confirming the protocol is aware of this pattern and intentionally implemented it on L1 but omitted it on all L2 pool variants.

## Finding Description
In `RSETHPoolV3.sol`, `deposit(string)` calls `viewSwapRsETHAmountAndFee(amount)` which reads `getRate()` live and computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. The computed amount is minted directly with no floor check:

```solidity
// RSETHPoolV3.sol L258-262
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);
```

`viewSwapRsETHAmountAndFee` at L299-307 reads the oracle live:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The identical pattern is present in `RSETHPoolNoWrapper.sol` L231-243, `RSETHPool.sol` L265-278, `RSETHPoolV3ExternalBridge.sol` L366-384 and L390-412, and the remaining pool variants. None of these functions accept a caller-supplied minimum output amount.

By contrast, the L1 `LRTDepositPool.depositETH(uint256 minRSETHAmountExpected, ...)` and `depositAsset(address, uint256, uint256 minRSETHAmountExpected, ...)` both enforce the guard in `_beforeDeposit`:

```solidity
// LRTDepositPool.sol L667-668
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

The cross-chain oracle rate is pushed by `MultiChainRateProvider`/`CrossChainRateReceiver` on a regular cadence. Because rsETH is a yield-bearing token, its rate against ETH only increases over time. Any oracle push between a user's off-chain preview and on-chain execution increases the denominator in the output formula, reducing the minted amount below the previewed value. There is no existing check in any L2 pool `deposit()` path that would revert or protect the user in this scenario.

## Impact Explanation
A user who previews a deposit via `viewSwapRsETHAmountAndFee` and submits a transaction receives fewer wrsETH/rsETH than the preview showed if the oracle rate is updated before the transaction is mined. The user's ETH or LST is consumed and held in the pool, so no funds are lost outright, but the user receives fewer LRT shares than the contract implicitly promised at preview time. This maps to **Low: Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The rsETH/ETH oracle rate increases continuously as staking rewards accrue and is updated cross-chain on a regular schedule. On L2 networks with variable block times or congested mempools, a transaction can easily be delayed by minutes to hours. Any oracle update during that window silently reduces the output amount. This is a routine operational condition, not an edge case, and is repeatable on every deposit across all L2 pool variants.

## Recommendation
Add a `minRsETHAmountOut` parameter to each `deposit()` overload in all L2 pool contracts and revert if the computed amount falls below it, mirroring the existing guard in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountOut) external payable {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountOut) revert SlippageExceeded();
    wrsETH.mint(msg.sender, rsETHAmount);
    ...
}
```

Optionally add a `deadline` parameter and revert if `block.timestamp > deadline`.
