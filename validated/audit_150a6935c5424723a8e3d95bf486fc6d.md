### Title
Silent Fallthrough Permanently Locks User Tokens When `withdraw_relayer` Is Unset - (File: near/omni-token/src/lib.rs)

### Summary
`OmniToken.ft_transfer` contains a legacy bridging path gated on the `WITHDRAW_TO:` memo prefix. When the user signals intent to use this path but `withdraw_relayer` is not configured, the code silently falls through and executes a plain `ft_transfer` to `receiver_id = env::current_account_id()` — the token contract itself — permanently locking the user's tokens with no error emitted.

### Finding Description
In `near/omni-token/src/lib.rs`, the `ft_transfer` override implements a legacy Near Intents bridging flow:

```rust
fn ft_transfer(&mut self, receiver_id: AccountId, amount: U128, memo: Option<String>) {
    if receiver_id == env::current_account_id()
        && memo.as_ref().is_some_and(|m| m.starts_with(WITHDRAW_MEMO_PREFIX))
    {
        if let Some(withdraw_relayer) = Self::read_withdraw_relayer_address() {
            return self.token.ft_transfer(withdraw_relayer, amount, memo);
        }
        // ← NO revert here; falls through silently
    }
    self.token.ft_transfer(receiver_id, amount, memo);
}
``` [1](#0-0) 

The outer `if` fires when `receiver_id == current_account_id` **and** the memo starts with `"WITHDRAW_TO:"`. If `read_withdraw_relayer_address()` returns `None` (withdraw relayer not configured), the inner branch is skipped and execution falls through to `self.token.ft_transfer(receiver_id, amount, memo)` — where `receiver_id` is still the token contract itself. [2](#0-1) 

The NEP-141 standard allows `ft_transfer` to a contract's own account (it only prohibits `sender == receiver`). The tokens are credited to the token contract's own balance. The `OmniToken` contract has no self-recovery function: `burn` withdraws from `predecessor_account_id` (the caller), not from the contract's own balance. [3](#0-2) 

The `withdraw_relayer` is written only during `migrate_from_poa`. Freshly deployed tokens, or any token where the storage key was not written, have no relayer configured. [4](#0-3) 

### Impact Explanation
Tokens transferred to the token contract's own account are permanently unclaimable without out-of-band DAO intervention via `transfer_token_as_dao` on the bridge contract. This is an irreversible fund lock matching the **Critical** impact tier: "Irreversible fund lock, frozen redemption path, or permanently unclaimable user or protocol value."

### Likelihood Explanation
Any user of a token that (a) does not have `withdraw_relayer` configured and (b) uses the legacy `WITHDRAW_TO:` memo interface — e.g., a Near Intents client that has not been updated — will silently lose funds. The entry point is fully unprivileged (`ft_transfer` is public). The condition is reachable on any freshly deployed `OmniToken` that has never had `migrate_from_poa` called.

### Recommendation
Inside the outer `if` block, after the inner `if let Some(...)` fails to match, add an explicit revert:

```rust
if receiver_id == env::current_account_id()
    && memo.as_ref().is_some_and(|m| m.starts_with(WITHDRAW_MEMO_PREFIX))
{
    if let Some(withdraw_relayer) = Self::read_withdraw_relayer_address() {
        return self.token.ft_transfer(withdraw_relayer, amount, memo);
    }
    env::panic_str("ERR_WITHDRAW_RELAYER_NOT_SET");
}
self.token.ft_transfer(receiver_id, amount, memo);
```

This mirrors the M-05 recommendation: when the user's stated intent cannot be fulfilled, revert with an informative error rather than silently executing a different — and destructive — action.

### Proof of Concept
1. Deploy a fresh `OmniToken` (no `migrate_from_poa` called → `WITHDRAW_RELAYER_ADDRESS` storage key absent).
2. User calls `ft_transfer(receiver_id = <token_contract>, amount = X, memo = Some("WITHDRAW_TO:near:user.near"))`.
3. Outer `if` condition is true; `read_withdraw_relayer_address()` returns `None`; inner branch skipped.
4. Falls through to `self.token.ft_transfer(<token_contract>, X, memo)`.
5. `X` tokens are credited to the token contract's own balance.
6. No error is emitted; the user's transaction succeeds with a `true` return.
7. Tokens are permanently locked — no public function on `OmniToken` can retrieve them.
