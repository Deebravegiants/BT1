### Title
Fee-on-Transfer Token Inflates Bridged Amount — Unbacked Wrapped Token Minting - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.initTransfer` emits an `InitTransfer` event using the caller-supplied `amount` parameter without verifying the actual balance delta received by the contract. For any ERC20 token with a transfer fee (or a malicious token with a modified `transferFrom`), the bridge locks fewer tokens than the event declares, causing the NEAR side to mint a larger amount of wrapped tokens than are actually backed.

### Finding Description

In `OmniBridge.initTransfer`, when the token is neither a bridge token nor a custom-minter token, the contract calls `safeTransferFrom` with the caller-declared `amount` and then immediately emits `InitTransfer` with that same declared `amount`: [1](#0-0) [2](#0-1) 

`SafeERC20.safeTransferFrom` only checks that the call does not revert and that the return value (if any) is `true`. It does **not** verify that `balanceOf(address(this))` increased by exactly `amount`. A fee-on-transfer token (e.g., one that deducts 10% on every transfer) will succeed the call while delivering only `amount * 0.9` to the bridge. The event still records the full `amount`.

The `logMetadata` entry point is fully public with no token whitelist: [3](#0-2) 

Any caller can register an arbitrary ERC20 token with the NEAR bridge by emitting a `LogMetadata` event, then immediately call `initTransfer` with that token.

On the NEAR side, `ft_on_transfer` receives the `amount` field from the cross-chain `InitTransfer` event and uses it to credit the sender: [4](#0-3) 

Because NEAR trusts the event's `amount` field, it mints or releases the full declared amount to the recipient, even though the EVM bridge vault only holds a fraction of that.

### Impact Explanation

**Critical.** An attacker can create unbacked wrapped supply on NEAR. Each bridging operation with a fee-on-transfer token inflates the circulating supply of the wrapped asset beyond what is locked in the EVM vault. When legitimate users later bridge back, the vault is short, causing the last redeemers to receive nothing — a permanent, irreversible loss of funds for honest users. This matches the allowed impact: *unauthorized creation of wrapped bridge assets through settlement failure* and *irreversible fund lock / permanently unclaimable user value*.

### Likelihood Explanation

**High.** Fee-on-transfer tokens are a well-known and widely deployed ERC20 pattern (e.g., reflection tokens, deflationary tokens). No privileged role is required: `logMetadata` and `initTransfer` are both fully public. The attacker needs only to hold a fee-on-transfer ERC20 and call two public functions. The attack is repeatable and each invocation widens the backing gap.

### Recommendation

Record the vault's token balance before and after `safeTransferFrom`, and use the delta as the canonical `amount` for the emitted event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualAmount = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualAmount > fee, "ERR_INSUFFICIENT_AMOUNT_AFTER_FEE");
// use actualAmount in the event and downstream logic
```

This is exactly the fix applied to the MonoX vulnerability referenced in the external report.

### Proof of Concept

1. Attacker deploys `FeeToken`, an ERC20 that deducts 20% on every `transferFrom` (transfers `0.8 * value` to `to`, keeps `0.2 * value` in the contract).
2. Attacker calls `OmniBridge.logMetadata(feeTokenAddress)` — public, no restriction — causing the NEAR bridge to register `FeeToken`.
3. Attacker calls `OmniBridge.initTransfer(feeTokenAddress, 1_000e18, 0, 0, "attacker.near", "")`.
4. `safeTransferFrom` succeeds; bridge vault receives `800e18` tokens.
5. `InitTransfer` event is emitted with `amount = 1_000e18`.
6. NEAR bridge processes the event and mints `1_000e18` wrapped `FeeToken` to `attacker.near`.
7. Attacker now holds `1_000e18` wrapped tokens backed by only `800e18` locked tokens — `200e18` of unbacked supply created from nothing.
8. Attacker redeems `1_000e18` wrapped tokens; the EVM vault can only pay out `800e18`, leaving the last `200e18` of legitimate redemptions permanently unclaimable.

    pub fn ft_on_transfer(&mut self, sender_id: AccountId, amount: U128, msg: String) {
        let token_id = env::predecessor_account_id();
        let parsed_msg: BridgeOnTransferMsg = serde_json::from_str(&msg)
            .or_else(|_| serde_json::from_str(&msg).map(BridgeOnTransferMsg::InitTransfer))
            .near_expect(BridgeError::ParseMsg);

        // We can't trust sender_id to pay for storage as it can be spoofed.
        let signer_id = env::signer_account_id();
        let promise_or_promise_index_or_value = match parsed_msg {
            BridgeOnTransferMsg::InitTransfer(init_transfer_msg) => {
                self.init_transfer(sender_id, signer_id, token_id, amount, init_transfer_msg)
            }
            BridgeOnTransferMsg::FastFinTransfer(fast_fin_transfer_msg) => {
                self.fast_fin_transfer(token_id, amount, signer_id, fast_fin_transfer_msg)
            }
            BridgeOnTransferMsg::UtxoFinTransfer(utxo_fin_transfer_msg) => self.utxo_fin_transfer(
                token_id,
                amount,
                &signer_id,
                &sender_id,
                utxo_fin_transfer_msg,
            ),
            BridgeOnTransferMsg::SwapMigratedToken => {
                self.swap_migrated_token(sender_id, token_id, amount)
                    .detach();
                PromiseOrPromiseIndexOrValue::Value(U128(0))
            }
        };

        promise_or_promise_index_or_value.as_return();
    }
