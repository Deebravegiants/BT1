Audit Report

## Title
Uninitialized `minRsEthAmountToWithdraw` allows dust-request flooding to temporarily freeze legitimate user withdrawals — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`minRsEthAmountToWithdraw[asset]` defaults to `0` for every asset and is never set in `initialize()`. The only effective guard in `initiateWithdrawal()` is `rsETHUnstaked == 0`, so any amount ≥ 1 wei of rsETH is accepted. Because `_unlockWithdrawalRequests()` processes the FIFO queue strictly in ascending nonce order and uses `break` (not `continue`) on each blocking condition, an attacker who queues thousands of dust requests ahead of a legitimate user's request temporarily freezes that user's funds until the operator drains every preceding dust entry at significant gas cost.

## Finding Description
**Root cause — uninitialized minimum:**
`LRTWithdrawalManager.initialize()` sets only `withdrawalDelayBlocks` and `lrtConfig`; `minRsEthAmountToWithdraw` is left at the Solidity default of `0` for every asset.

```solidity
// contracts/LRTWithdrawalManager.sol L90-98
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    __Pausable_init();
    __ReentrancyGuard_init();
    withdrawalDelayBlocks = 8 days / 12 seconds;
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

**Collapsed guard in `initiateWithdrawal()`:**
```solidity
// L162-164
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```
When `minRsEthAmountToWithdraw[asset] == 0`, the expression `rsETHUnstaked < 0` is always `false` for `uint256`, so the check reduces to `rsETHUnstaked == 0`. Any value ≥ 1 wei passes.

**Strict FIFO with no skip:**
`_unlockWithdrawalRequests()` iterates from `nextLockedNonce` upward and uses `break` on both blocking conditions:
```solidity
// L795
if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;
// L800
if (availableAssetAmount < payoutAmount) break;
```
A request at nonce `N` cannot be unlocked until all requests at nonces `0 … N-1` have been processed. There is no mechanism to skip individual entries.

**Exploit path:**
1. Attacker deposits ETH into `LRTDepositPool.depositETH()` (also unguarded — `minAmountToDeposit` defaults to `0`) to obtain rsETH.
2. Attacker calls `initiateWithdrawal(asset, 1, "")` N times, filling nonces `0 … N-1` with 1-wei rsETH requests.
3. Victim calls `initiateWithdrawal(asset, largeAmount, "")` → queued at nonce `N`.
4. After `withdrawalDelayBlocks` pass, the operator calls `unlockQueue`. The loop must process all N dust entries before reaching nonce `N`. Due to block gas limits, this requires many separate `unlockQueue` transactions.
5. Victim cannot call `completeWithdrawal()` until `nextLockedNonce[asset] > N`, which requires the operator to drain all N dust entries first.

**`setMinRsEthAmountToWithdraw` accepts zero:**
```solidity
// L330-333
function setMinRsEthAmountToWithdraw(address asset, uint256 minRsEthAmountToWithdraw_) external onlyLRTAdmin {
    minRsEthAmountToWithdraw[asset] = minRsEthAmountToWithdraw_;
    emit MinAmountToWithdrawUpdated(asset, minRsEthAmountToWithdraw_);
}
```
No zero-value guard, so even if an admin sets a minimum it can be silently reset to `0`.

## Impact Explanation
**Temporary freezing of funds (Medium).** The victim's rsETH is transferred into `LRTWithdrawalManager` at `initiateWithdrawal` time and cannot be recovered until `completeWithdrawal` succeeds. `completeWithdrawal` requires `usersFirstWithdrawalRequestNonce < nextLockedNonce[asset]`, which cannot be satisfied until the operator processes every preceding dust entry. With EigenLayer's mandatory ~8-day `withdrawalDelayBlocks` already in play, the effective freeze extends by however long it takes the operator to drain the queue across many gas-expensive transactions.

**Unbounded gas consumption (Medium).** The operator must call `unlockQueue` repeatedly; total operator gas scales linearly with the number of dust entries the attacker creates.

## Likelihood Explanation
Any address holding rsETH can execute this attack. rsETH is freely obtainable via `LRTDepositPool.depositETH()` or `depositAsset()`. Because `minAmountToDeposit` also defaults to `0`, the attacker can acquire rsETH in arbitrarily small increments. The attacker's only cost is gas for `initiateWithdrawal` calls and the temporary lock-up of their own rsETH (which they eventually recover). The cost asymmetry strongly favours the attacker: N deposit + N `initiateWithdrawal` transactions for the attacker versus N `unlockQueue` iterations for the operator and an indefinite freeze for every legitimate user queued behind the dust entries.

## Recommendation
1. **Enforce a non-zero minimum in `initialize()`**: Set a sensible default for `minRsEthAmountToWithdraw` for each supported asset during initialisation (e.g., equivalent to ~0.001 ETH worth of rsETH).
2. **Guard `setMinRsEthAmountToWithdraw` against zero**: Add `if (minRsEthAmountToWithdraw_ == 0) revert InvalidMinAmount();` analogous to guards present elsewhere in the codebase.
3. **Guard `setMinAmountToDeposit` against zero** in `LRTDepositPool` for the same reason.
4. **Consider a `continue`-based skip** in `_unlockWithdrawalRequests()` for requests whose `payoutAmount` is zero, so that dust entries do not block subsequent legitimate requests.

## Proof of Concept
```
// Precondition: minRsEthAmountToWithdraw[ETH_TOKEN] == 0 (default, never set)

1. Attacker calls LRTDepositPool.depositETH{value: 1 ether}(0, "")
   → receives ~1e18 rsETH

2. Attacker approves LRTWithdrawalManager to spend rsETH.

3. Attacker calls initiateWithdrawal(ETH_TOKEN, 1, "") 1,000,000 times.
   Each call: rsETHUnstaked=1, guard check: (1==0 || 1<0) → false → no revert.
   Dust requests fill nonces 0 … 999,999.

4. Victim calls initiateWithdrawal(ETH_TOKEN, 1e18, "") → queued at nonce 1,000,000.

5. After withdrawalDelayBlocks (~57,600 blocks / ~8 days) pass:
   Operator calls unlockQueue(ETH_TOKEN, 1_000_001, ...).
   _unlockWithdrawalRequests iterates from nextLockedNonce=0 to 1_000_001.
   Block gas limit forces the operator to split this into many transactions.
   Victim's request at nonce 1,000,000 remains locked until all 1,000,000
   dust entries are drained.
   Victim cannot call completeWithdrawal() until then.
```
