### Title
Unvalidated Future Timestamp in `RespondPeers` Enables Permanent Peer Residence and DNS Seeder Poisoning — (`chia/seeder/crawl_store.py`, `chia/seeder/crawler.py`)

### Summary

The crawler accepts `response_peer.timestamp` values from remote peers without any upper-bound validation. A malicious peer can respond to the crawler's `RequestPeers` call with a `RespondPeers` message containing a far-future timestamp. This future value propagates through `best_timestamp_per_peer` → `CrawlStore.update_best_timestamp` → `PeerRecord.best_timestamp`, causing the peer to permanently survive all `prune_old_peers(cutoff)` calls. Over time, the peer accumulates reliability statistics and enters `good_peers`, causing the DNS seeder to serve the attacker's IP to every bootstrapping node.

### Finding Description

**Step 1 — Entry point:** The crawler actively connects to peers in `host_to_records` and calls `peer.call_api(FullNodeAPI.request_peers, ...)`. The response is stored in `self.peers_retrieved`. [1](#0-0) 

**Step 2 — No timestamp validation in `crawl()`:** For each `response_peer` in the collected responses, `best_timestamp_per_peer` is updated with `max(existing, response_peer.timestamp)`. There is no upper-bound check — a future timestamp (e.g., `now + 10*24*3600`) is accepted and stored. [2](#0-1) 

**Step 3 — Future timestamp propagates to `PeerRecord`:** `update_best_timestamp` performs no validation and directly replaces `best_timestamp` with the attacker-supplied value. [3](#0-2) 

**Step 4 — `prune_old_peers` never evicts the peer:** The cutoff is `now - older_than_days*86400`. Since `future_timestamp > cutoff`, the peer is never deleted from either the DB or the in-memory dicts. [4](#0-3) 

**Step 5 — Future timestamp also survives the in-memory cleanup:** The `best_timestamp_per_peer` dict is filtered to keep entries where `timestamp >= now - 5*24*3600`. A future timestamp trivially satisfies this. [5](#0-4) 

**Step 6 — Peer enters `good_peers`:** `load_reliable_peers_to_db` writes all peers for which `is_reliable()` returns `True` into the `good_peers` table. Because the peer is never pruned, the crawler keeps connecting to it, accumulating `tries`/`successes` and stat window values until `is_reliable()` returns `True`. [6](#0-5) 

**Step 7 — DNS seeder serves attacker IP:** `get_good_peers()` reads from the `good_peers` table and returns results to the DNS server, which serves them to every bootstrapping node. [7](#0-6) 

**`TimestampedPeerInfo` has no timestamp validation:** [8](#0-7) 

### Impact Explanation

A permanently resident attacker-controlled peer that accumulates reliability stats and enters `good_peers` causes the DNS seeder to serve the attacker's IP to all bootstrapping nodes. If the attacker controls enough IPs in `good_peers`, new nodes bootstrapping from the DNS seeder connect primarily to attacker-controlled peers, enabling a sustained eclipse attack. This qualifies as **High** impact: long-lived inability for honest bootstrapping nodes to reach the honest network and process valid blocks under normal network assumptions.

### Likelihood Explanation

The attacker only needs to run a node that gets gossiped to the crawler (an unprivileged action). Once the crawler connects and sends `RequestPeers`, the attacker responds with a crafted `RespondPeers` containing future timestamps. No keys, admin access, or cryptographic breaks are required. The attack is fully automated and persistent across crawler restarts (the future timestamp is persisted to the SQLite DB via `load_to_db`).

### Recommendation

In `crawl()`, clamp `response_peer.timestamp` to `min(response_peer.timestamp, int(time.time()))` before storing it in `best_timestamp_per_peer`. Apply the same clamp inside `update_best_timestamp`. This ensures no future timestamp can ever be written to `PeerRecord.best_timestamp`.

### Proof of Concept

```python
from chia_rs.sized_ints import uint32, uint64
from chia.seeder.crawl_store import CrawlStore
from chia.seeder.peer_record import PeerRecord, PeerReliability

async def test():
    db = await aiosqlite.connect(":memory:")
    store = await CrawlStore.create(db)
    future_ts = uint64(int(time.time()) + 10 * 24 * 3600)
    peer = PeerRecord("1.2.3.4","1.2.3.4",uint32(8444),False,
                      uint64(0),uint32(0),uint64(0),uint64(int(time.time())),
                      future_ts,"undefined",uint64(0),"unknown")
    rel = PeerReliability("1.2.3.4")
    await store.add_peer(peer, rel, save_db=True)
    await db.commit()
    for _ in range(5):
        await store.prune_old_peers(90)
    assert "1.2.3.4" in store.host_to_records, "Peer was pruned — expected to survive"
    print("PASS: peer with future best_timestamp survives prune_old_peers indefinitely")

asyncio.run(test())
```
