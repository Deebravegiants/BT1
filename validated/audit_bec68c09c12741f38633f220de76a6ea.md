### Title
Cross-Message Replay Poisons ChunkEndorsementTracker Cache, Silently Drops Legitimate Endorsements for Chunk B - (`chain/client/src/stateless_validation/chunk_endorsement.rs`)

---

### Summary

`ChunkEndorsementV2::verify()` validates `inner_signature` and `metadata_signature` independently against the same validator public key, but never checks that `inner.chunk_hash` corresponds to the chunk identified by `metadata.{shard_id, epoch_id, height_created}`. A malicious peer who has observed two legitimate endorsements from the same validator (for chunk A and chunk B) can splice them together — `inner_A + sig_inner_A + metadata_B + sig_meta_B` — producing a structurally valid endorsement that passes all checks. `process_chunk_endorsement` then caches this under chunk B's `ChunkProductionKey` with chunk A's `chunk_hash`. The deduplication early-exit subsequently rejects the legitimate endorsement for chunk B. `collect_chunk_endorsements` filters the poisoned entry out by `chunk_hash` mismatch, so validator V's stake never counts toward chunk B's endorsement threshold.

---

### Finding Description

**Root cause — missing cross-field invariant in `ChunkEndorsementV2::verify`:**

`inner` and `metadata` are signed and verified as two independent blobs. Nothing binds `inner.chunk_hash` to the `(shard_id, epoch_id, height_created)` triple in `metadata`. [1](#0-0) 

**Validation path does not add the missing check:**

`validate_chunk_endorsement_signature` simply calls `endorsement.verify()` and returns; no cross-field comparison is performed. [2](#0-1) 

**Cache insert stores mismatched fields:**

The cache key is derived from `metadata` (chunk B's `ChunkProductionKey`), but the stored value carries `chunk_hash()` from `inner` (chunk A's hash). [3](#0-2) 

**Deduplication early-exit locks out the legitimate endorsement:**

Once the poisoned entry is cached under chunk B's key for validator V, any subsequent legitimate endorsement from V for chunk B is silently dropped here: [4](#0-3) 

**Collection filters the poisoned entry out:**

`collect_chunk_endorsements` filters by `chunk_hash == chunk_header.chunk_hash()`. The cached entry carries chunk A's hash, so it is excluded and V's stake is not counted. [5](#0-4) 

---

### Impact Explanation

The endorsement cache for chunk B is permanently poisoned (until LRU eviction) for every targeted validator. The legitimate endorsement for chunk B from each targeted validator is silently discarded. If enough validators are targeted to push the remaining endorsing stake below the threshold, chunk B cannot be included in a block. The block producer's `collect_chunk_endorsements` call returns an `is_endorsed = false` state, stalling block production for that shard. [6](#0-5) 

---

### Likelihood Explanation

The attacker requires:
1. A network peer position on the routing path between validator V and the block producer (to observe both endorsements in transit).
2. Two legitimate endorsements from the same validator V — one for chunk A (shard S1, height H1) and one for chunk B (shard S2, height H2) — both observable in the same epoch. This is realistic when a validator is assigned to multiple shards.
3. The ability to deliver the crafted message to the block producer **before** the legitimate endorsement for chunk B arrives (network-level race).

No validator key material is needed. The attacker only recombines already-signed blobs. The attack is repeatable across multiple validators to amplify impact. [7](#0-6) 

---

### Recommendation

Bind `inner.chunk_hash` to `metadata` at signing time and enforce the binding at verification time. The simplest fix is to include `(shard_id, epoch_id, height_created)` inside `ChunkEndorsementInnerV1` (or sign a single combined struct), so that a signature over `inner` also commits to the metadata fields. Alternatively, add an explicit check in `ChunkEndorsementV2::verify` that the `chunk_hash` in `inner` matches the hash derivable from the `metadata` fields — though this requires the verifier to have access to the chunk header. [8](#0-7)

### Proof of Concept

// Pseudocode — no validator key needed; attacker only recombines observed wire messages.

// Step 1: observe two legitimate endorsements from validator V
let endorsement_A: ChunkEndorsement = /* observed from network for chunk A (shard S1, H1) */;
let endorsement_B: ChunkEndorsement = /* observed from network for chunk B (shard S2, H2) */;

// Step 2: splice inner from A with metadata from B — both sigs remain valid under V's key
let crafted = ChunkEndorsement::V2(ChunkEndorsementV2 {
    inner:              endorsement_A.inner,              // chunk_hash = hash_A
    signature:          endorsement_A.signature,          // valid sig over inner_A
    metadata:           endorsement_B.metadata,           // shard_id=S2, height=H2
    metadata_signature: endorsement_B.metadata_signature, // valid sig over metadata_B
});

// Step 3: send crafted endorsement to block producer before legitimate endorsement_B arrives
block_producer.process_chunk_endorsement(&crafted);
// -> passes verify() (both sigs valid), cached under key_B with hash_A

// Step 4: legitimate endorsement_B arrives — rejected by deduplication early-exit
block_producer.process_chunk_endorsement(&endorsement_B);
// -> "already received chunk endorsement" — silently dropped

// Step 5: block producer collects endorsements for chunk B
let state = tracker.collect_chunk_endorsements(&chunk_header_B);
// -> filter removes V's entry (hash_A != hash_B) -> V's stake not counted
assert!(!state.is_endorsed); // chunk B cannot be endorsed

        let metadata = borsh::to_vec(&self.metadata).unwrap();

    }

    pub fn process_chunk_endorsement(&self, endorsement: &ChunkEndorsement) -> Result<(), Error> {
        // Check if we have already received chunk endorsement from this validator.
        let key = endorsement.chunk_production_key();
        let account_id = endorsement.account_id();

        {
            let cache = self.chunk_endorsements.lock();
            if cache.peek(&key).is_some_and(|entry| entry.contains_key(account_id)) {
                tracing::debug!(target: "client", ?endorsement, "already received chunk endorsement");
                return Ok(());
            }
        }

        // Validate the chunk endorsement and store it in the cache.
        match validate_chunk_endorsement(self.epoch_manager.as_ref(), endorsement, &self.store)? {
            ChunkRelevance::Relevant => {
                let mut cache = self.chunk_endorsements.lock();
                cache.get_or_insert_mut(key, || HashMap::new()).insert(
                    account_id.clone(),
                    (endorsement.chunk_hash(), endorsement.signature()),
                );
                let shard_id = endorsement.shard_id();
                metrics::CHUNK_ENDORSEMENTS_ACCEPTED
                    .with_label_values(&[&shard_id.to_string()])
                    .inc();
            }
            irrelevant => {
                metrics::CHUNK_ENDORSEMENTS_REJECTED
                    .with_label_values::<&str>(&[
                        endorsement.shard_id().to_string().as_str(),
                        irrelevant.into(),
                    ])
                    .inc();
            }
        };
        Ok(())
    }
