### Title
Missing Slippage Protection in L2 Pool `deposit()` Functions Allows Users to Receive Fewer rsETH Tokens Than Expected - (File: contracts/pools/RSETHPoolNoWrapper.sol)

### Summary
The L2 deposit pool contracts (`RSETHPoolNoWrapper`, `RSETHPoolV2`, `RSETHPoolV3`, and variants) expose `deposit()` functions that compute the rsETH/wrsETH output amount from a live oracle rate at execution time, but accept no `minRsETHAmountOut` parameter. Users have no way to bound the minimum tokens they receive, exposing them to unfavorable rate movements between transaction submission and execution.

### Finding Description

The L1 `LRTDepositPool` correctly implements slippage protection: both `depositETH()` and `depositAsset()` accept a `minRSETHAmountExpected` parameter, and the internal `_beforeDeposit()` reverts with `MinimumAmountToReceiveNotMet` if the computed mint amount falls below it. [1](#0-0) [2](#0-1) 

The L2 pool contracts do not replicate this protection. In `RSETHPoolNoWrapper.sol`, both deposit overloads compute `rsETHAmount` from `viewSwapRsETHAmountAndFee()` which reads `getRate()` from the oracle at execution time, and immediately transfer that amount to the caller with no minimum-output check: [3](#0-2) [4](#0-3) [5](#0-4) 

The same pattern exists in `RSETHPoolV2.sol`: [6](#0-5) [7](#0-6) 

And in `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `RSETHPoolV2NBA.sol` — all follow the identical pattern. [8](#0-7) [9](#0-8) 

The oracle rate `rsETHToETHrate` is the ETH value of one rsETH. If this rate increases between the time a user submits their transaction and when it is mined, the user receives fewer rsETH/wrsETH tokens for the same ETH input, with no on-chain protection to revert the transaction.

### Impact Explanation

A user depositing ETH or a supported token into any L2 pool contract may receive fewer rsETH/wrsETH tokens than they observed when constructing the transaction. The rsETH/ETH rate increases monotonically over time as staking rewards accrue, so any delay in transaction inclusion (e.g., due to network congestion or mempool delay) directly reduces the output amount. The user suffers a loss relative to the rate they expected, with no recourse. The protocol does not lose value; only the depositor does.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

### Likelihood Explanation

Every unprivileged user calling `deposit()` on any deployed L2 pool contract is affected. The rsETH/ETH rate is updated periodically by the oracle, and any deposit transaction that is pending across an oracle update will silently receive a worse rate. This is a routine occurrence on congested L2 networks and requires no attacker — it is a structural property of the design.

### Recommendation

Add a `minRsETHAmountOut` parameter to all `deposit()` overloads in the L2 pool contracts, mirroring the existing `minRSETHAmountExpected` pattern in `LRTDepositPool`. Revert if the computed `rsETHAmount` is below the caller-specified minimum:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountOut) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountOut) revert SlippageExceeded();
    ...
}
```

Apply the same change to the token-deposit overload and to all pool contract variants.

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolNoWrapper` and observes `rsETHAmount = X` at oracle rate `R`.
2. User submits `deposit{value: 1 ether}(referralId)`.
3. Before the transaction is mined, the oracle rate is updated from `R` to `R'` where `R' > R` (rsETH appreciated).
4. The transaction executes: `rsETHAmount = (1 ether - fee) * 1e18 / R'`, which is strictly less than `X`.
5. User receives fewer rsETH/wrsETH than expected with no revert, and has no on-chain mechanism to prevent this outcome. [5](#0-4) [7](#0-6)

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
