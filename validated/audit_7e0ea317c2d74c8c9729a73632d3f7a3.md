### Title
Unauthenticated SQMR Peer Injects Arbitrary Transaction Hash→Body Mapping via Unvalidated `transaction_hash` in P2P Sync — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts `transaction_hash` from an unauthenticated SQMR peer and stores it directly as the lookup key in the `transaction_hash_to_idx` table without recomputing or verifying it against the transaction body. This allows any peer to inject an arbitrary mapping `H → T` such that `starknet_getTransactionByHash(H)` returns transaction body `T` even when `compute_hash(T) ≠ H`.

---

### Finding Description

In `parse_data_for_block`, the peer-supplied `transaction_hash` field is pushed verbatim into `block_body.transaction_hashes` with an explicit TODO acknowledging the missing validation: [1](#0-0) 

The `BlockBody` is then passed to `write_to_storage`, which calls `append_body`, which calls `write_transactions`. Inside `write_transactions`, the peer-supplied hash is inserted as the key into the `transaction_hash_to_idx_table`: [2](#0-1) 

The `get_transaction_idx_by_hash` reader performs a direct table lookup with no recomputation: [3](#0-2) 

The RPC handler `get_transaction_by_hash` uses this lookup and returns the stored transaction body paired with the caller-supplied hash — it never recomputes the hash from the body: [4](#0-3) 

**Why the block hash does not protect against this:** The block header stores an optional `transaction_commitment` (Merkle root over transaction hashes): [5](#0-4) 

However, `parse_data_for_block` only reads `n_transactions` from the header — it never verifies the received hashes against `transaction_commitment`. There is no call to `validate_transaction_hash` anywhere in the p2p sync transaction path.

---

### Impact Explanation

A malicious SQMR peer responding to a transaction sync query for block N can send `(transaction_body_T, arbitrary_hash_H)` pairs. The node stores `H → index(T)` in its database. Subsequently:

- `starknet_getTransactionByHash(H)` returns `T` with `transaction_hash = H`, even though `Poseidon(T) ≠ H`.
- `starknet_getTransactionReceipt(H)` returns a receipt for `T` under hash `H`.
- `starknet_getTransactionStatus(H)` reports the transaction as accepted.

This corrupts the authoritative RPC view served to block explorers, wallets, and other nodes. It also corrupts the `get_transaction_idx_by_hash` index used by fee estimation and simulation paths.

Impact: **High** — RPC returns an authoritative-looking wrong value; transaction hash/body binding is broken.

---

### Likelihood Explanation

Any node on the network can act as an SQMR peer. The syncing node sends a query and accepts responses from whichever peer it connects to. No authentication, signature, or proof-of-work is required to respond. The attack requires only that the attacker be reachable as a peer during the sync of any block.

---

### Recommendation

In `parse_data_for_block`, after receiving each `FullTransaction`, recompute the transaction hash from the transaction body and reject (report peer + retry) if it does not match the peer-supplied `transaction_hash`. The function `validate_transaction_hash` already exists in `starknet_api`: [6](#0-5) 

Additionally, after collecting all transaction hashes for a block, verify the resulting `transaction_commitment` against the value stored in the block header (when present) before writing to storage.

---

### Proof of Concept

1. Operate a node that responds to SQMR `TransactionQuery` for block N.
2. For each transaction slot, send `FullTransaction { transaction: T, transaction_output: O, transaction_hash: H }` where `H` is an arbitrary felt and `T` is any valid transaction body such that `compute_hash(T) ≠ H`.
3. The syncing node calls `parse_data_for_block`, pushes `H` into `block_body.transaction_hashes`, and writes via `append_body` → `write_transactions` → `transaction_hash_to_idx_table.insert(H, index)`.
4. Query the victim node: `starknet_getTransactionByHash(H)` returns `T` with `transaction_hash = H`.
5. Assert: `get_transaction_idx_by_hash(H)` returns `Some(index)` pointing to `T`, but `compute_hash(T) ≠ H`. The invariant is violated.
