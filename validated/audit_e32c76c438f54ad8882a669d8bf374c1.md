Audit Report

## Title
Missing Staleness and Validity Checks on Chainlink `latestRoundData()` Allows Stale Prices to Corrupt rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt`, `answeredInRound`, and does not validate that `price > 0`. A stale or frozen Chainlink feed silently propagates an incorrect price into rsETH minting via `LRTDepositPool.getRsETHAmountToMint()`, allowing an attacker to receive more rsETH than their deposit warrants and extract value from existing rsETH holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at L49–55 reads only the `price` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

`updatedAt` (staleness timestamp), `answeredInRound` (round completeness), and sign of `price` are all silently discarded. This contrasts with `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which checks `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0`: [2](#0-1) 

The stale price propagates through two paths:

**Path 1 — Minting (permissionless):**
`depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice(asset)`. [3](#0-2) 

The minting formula `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` uses the stale price directly in the numerator. If the stale price is inflated, the depositor receives more rsETH than the deposited value warrants. [4](#0-3) 

**Path 2 — rsETH price update (public, permissionless):**
`updateRSETHPrice()` is callable by anyone and feeds into `_getTotalEthInProtocol()`, which prices all supported assets via the same stale oracle. [5](#0-4) [6](#0-5) 

**Existing guards are insufficient:**
- The `minRSETHAmountExpected` slippage parameter in `depositAsset()` protects the depositor from receiving *less* than expected, but does not prevent receiving *more* than deserved.
- The `pricePercentageLimit` check in `_updateRsETHPrice()` only applies to the rsETH price update path, not to the minting path.
- No time-based or round-completeness guard exists anywhere in `ChainlinkPriceOracle`.

## Impact Explanation
**High — Theft of unclaimed yield.**

When a Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale with an inflated last price, an attacker can deposit at the inflated rate and receive more rsETH than the deposited value justifies. After the feed recovers and the rsETH price is updated to reflect true TVL, the attacker redeems their over-minted rsETH for more ETH than they deposited. The excess comes directly from the yield and principal accrued by existing rsETH holders, whose share of the pool is diluted. The attack is repeatable for every stale feed window and for every supported LST asset.

## Likelihood Explanation
Chainlink LST/ETH feeds (stETH/ETH, rETH/ETH, ETHx/ETH) have documented heartbeats of up to 24 hours and have historically lagged during network congestion or oracle node issues. `depositAsset()` and `updateRSETHPrice()` are both fully permissionless — no special role, whitelist, or off-chain keeper exclusivity is required. An attacker monitoring mempool or Chainlink feed health can time the deposit precisely during a stale window.

## Recommendation
Add staleness and validity checks inside `ChainlinkPriceOracle.getAssetPrice()`, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (price <= 0) revert InvalidPrice();
    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePriceFeed();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`STALENESS_THRESHOLD` should be configurable per asset and set to match the Chainlink feed's documented heartbeat (e.g., 86 400 s for daily feeds, 3 600 s for hourly feeds).

## Proof of Concept
1. Chainlink stETH/ETH feed stops updating. Last reported price: `1.05e18`. True current price: `0.98e18` (post-slashing event).
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.05e18` with no revert.
4. Attacker receives `1000 * 1.05 / rsETHPrice` rsETH instead of `1000 * 0.98 / rsETHPrice` — approximately 7.1% more rsETH than the deposited value justifies.
5. After the feed recovers, `updateRSETHPrice()` is called, correctly pricing stETH at `0.98e18` and reducing the rsETH price to reflect true TVL.
6. Attacker redeems their rsETH, extracting ~7% more ETH than deposited, at the expense of existing rsETH holders.

**Foundry fork test plan:**
- Fork mainnet at a block where a stETH/ETH Chainlink round is near its heartbeat limit.
- Mock `latestRoundData()` to return a stale `updatedAt` with an inflated price.
- Call `depositAsset()` and record rsETH minted vs. fair value.
- Call `updateRSETHPrice()` with the corrected price.
- Redeem rsETH and assert profit > 0, confirming extraction from existing holders.
