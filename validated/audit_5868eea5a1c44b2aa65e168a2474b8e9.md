Audit Report

## Title
Unprotected `sendFunds()` in `FeeReceiver` Allows Anyone to Force MEV Reward Distribution and Dilute Existing Holder Yield — (File: `contracts/FeeReceiver.sol`)

## Summary
`FeeReceiver.sendFunds()` has no access control, allowing any external caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at an arbitrary time. An attacker can front-run this flush by depositing ETH at the pre-reward price, triggering `sendFunds()` to move the reward ETH into the deposit pool, then calling the public `LRTOracle.updateRSETHPrice()` to realize the price increase — capturing a disproportionate share of yield that belonged to existing rsETH holders.

## Finding Description
`FeeReceiver.sendFunds()` at line 53 carries no access control modifier, while every other state-changing function in the same contract (`setDepositPool`) is gated by `onlyRole(LRTConstants.MANAGER)`:

```solidity
// contracts/FeeReceiver.sol:53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`LRTDepositPool.receiveFromRewardReceiver()` is equally unguarded:

```solidity
// contracts/LRTDepositPool.sol:61
function receiveFromRewardReceiver() external payable { }
```

When `sendFunds()` executes, the ETH moves from `FeeReceiver` (excluded from TVL per the comment at `LRTDepositPool.sol:465-466`) into `LRTDepositPool` where it is counted as `address(this).balance` in `getETHDistributionData()`. This increases `totalETHInProtocol` in `_getTotalEthInProtocol()`, raising the rsETH price on the next `updateRSETHPrice()` call — which is itself public (`LRTOracle.sol:87`):

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (lines 252–266) compares the new price against `highestRsethPrice` (the all-time high), not the previous price. When the accumulated reward `R` is small relative to total TVL `V`, the percentage increase stays within the configured limit and the check passes without reverting. An attacker can calibrate their deposit size `D` to ensure this condition holds.

## Impact Explanation
**High — Theft of unclaimed yield.**

With protocol TVL `V`, rsETH supply `N`, and FeeReceiver balance `R`:
- Attacker deposits `D` ETH → receives `D·N/V` rsETH at price `V/N`
- Attacker calls `sendFunds()` → `R` ETH enters the deposit pool
- Attacker calls `updateRSETHPrice()` → new price = `(V+D+R)/(N+D·N/V)`
- Attacker's rsETH is worth `D·(V+D+R)/(V+D)` ETH
- **Attacker profit = `D·R/(V+D)` ETH**, extracted from existing holders' MEV yield

Existing holders lose exactly `D·R/(V+D)` ETH of yield they had earned. The attack is repeatable every time rewards accumulate in `FeeReceiver`.

## Likelihood Explanation
**High.** No privileged access is required. `sendFunds()` is callable by any EOA. The `FeeReceiver` balance is publicly observable on-chain. `updateRSETHPrice()` is also public, so the attacker controls the full sequence — deposit, flush, price update — within a single transaction or block. The `pricePercentageLimit` check is bypassable whenever `R/V` is below the configured threshold, which is the normal operating condition between reward flushes.

## Recommendation
Add `onlyRole(LRTConstants.MANAGER)` to `sendFunds()`, consistent with every other privileged function in `FeeReceiver`:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Additionally, add a caller check to `LRTDepositPool.receiveFromRewardReceiver()` to ensure only the registered `FeeReceiver` contract can invoke it, preventing direct ETH injection from arbitrary callers.

## Proof of Concept
**Setup:** TVL = 1000 ETH, rsETH supply = 1000 rsETH (price = 1.0), FeeReceiver holds 50 ETH.

```
// Step 1: Attacker deposits
LRTDepositPool.depositETH{value: 1000 ETH}(minRsETH, "")
// Attacker receives 1000 rsETH; TVL = 2000 ETH, supply = 2000 rsETH

// Step 2: Attacker forces reward flush
FeeReceiver.sendFunds()
// 50 ETH moves FeeReceiver → LRTDepositPool; TVL = 2050 ETH, supply = 2000 rsETH

// Step 3: Attacker triggers price update
LRTOracle.updateRSETHPrice()
// New price = 2050/2000 = 1.025 ETH/rsETH
// Price increase = 2.5% from highestRsethPrice=1.0 — passes if pricePercentageLimit >= 2.5%

// Step 4: Attacker's 1000 rsETH is now worth 1025 ETH
// Attacker profit = 25 ETH (stolen from original holders' MEV yield)
// Original 1000 rsETH holders receive 1025 ETH instead of 1050 ETH
```

**Foundry fork test plan:** Deploy against a mainnet fork with a funded `FeeReceiver`. Assert that a non-privileged address can call `sendFunds()` successfully, that `LRTDepositPool.balance` increases by the flushed amount, and that `updateRSETHPrice()` succeeds and sets a higher `rsETHPrice` — with the attacker's rsETH redeemable for more ETH than deposited.
