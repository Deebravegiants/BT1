Audit Report

## Title
Rewards permanently lost when `totalKernelStaked` drops to zero mid-period via `initiateWithdrawal` - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` uses a Synthetix-style reward accumulator where `rewardPerToken()` short-circuits and returns the stored value unchanged when `totalKernelStaked == 0`, but the `updateReward` modifier unconditionally advances `updatedAt` to the current timestamp. Any staker holding 100% of `totalKernelStaked` can call `initiateWithdrawal` to drop supply to zero mid-period, causing all rewards accrued during the zero-supply window to be permanently frozen in the contract with no recovery path.

## Finding Description
`rewardPerToken()` at L408-413 returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`: [1](#0-0) 

The `updateReward` modifier at L232-242 always executes `updatedAt = lastTimeRewardApplicable()` regardless of supply: [2](#0-1) 

`initiateWithdrawal` at L325-326 decrements `totalKernelStaked` in the function body (after the modifier runs), so the modifier correctly snapshots rewards up to the withdrawal moment — but immediately after, `totalKernelStaked == 0` and `updatedAt == block.timestamp`: [3](#0-2) 

Every subsequent call that triggers `updateReward` (e.g., a new staker's `stake`) finds `totalKernelStaked == 0`, so `rewardPerTokenStored` stays frozen while `updatedAt` keeps advancing to the current time. When a new staker eventually arrives, `rewardPerToken()` computes rewards only from the moment they staked, skipping the entire zero-supply window. The `rewardRate * elapsed_zero_supply_seconds` worth of tokens remain locked in the contract permanently — there is no admin sweep, no rescue function, and no rollover mechanism.

`notifyRewardAmount` guards only against starting a period with zero supply; it provides no protection against supply dropping to zero mid-period: [4](#0-3) 

The contract's own NatSpec at L18-22 explicitly acknowledges this residual risk as an off-chain operational assumption, not an on-chain invariant: [5](#0-4) 

## Impact Explanation
Reward tokens accrued during the zero-supply interval are permanently frozen inside `KernelDepositPool`. The frozen amount scales linearly with `rewardRate` and the duration of the zero-supply window — up to 30 days of rewards per withdrawal cycle (`MAX_WITHDRAWAL_DELAY`). This matches **Medium — Permanent freezing of unclaimed yield**. [6](#0-5) 

## Likelihood Explanation
Any single staker holding 100% of `totalKernelStaked` can trigger this unilaterally by calling the public `initiateWithdrawal` function. No admin action, no governance, and no coordination is required. The scenario is realistic whenever the pool has few participants or one dominant staker, and the withdrawal delay (up to 30 days) means the zero-supply window can be substantial.

## Recommendation
Skip advancing `updatedAt` when supply is zero, mirroring the `rewardPerTokenStored` freeze:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures time during which no tokens are staked does not consume the reward budget — rewards remain distributable once stakers return.

## Proof of Concept
```
T=0:   Admin calls notifyRewardAmount(1_000e18), duration=30 days
       → rewardRate ≈ 385e12 tokens/sec
       → finishAt = T+30d, updatedAt = T=0

T=1d:  Alice (sole staker, 100 KERNEL) calls initiateWithdrawal(100e18)
       updateReward runs (totalKernelStaked still 100e18):
         rewardPerTokenStored += rewardRate*1d/100e18  ✓
         updatedAt = T=1d
       Function body: totalKernelStaked = 0

T=1d–T=10d: No stakers. Any updateReward call finds totalKernelStaked==0:
            rewardPerTokenStored stays frozen, updatedAt advances each call.

T=10d: Bob stakes 100 KERNEL.
       updateReward runs (totalKernelStaked still 0 before body):
         rewardPerTokenStored unchanged, updatedAt = T=10d

T=10d+: Bob earns rewards only from T=10d onward.
        9 days × rewardRate ≈ 9 × 385e12 × 86400 ≈ 299e18 tokens
        permanently locked in the contract with no recovery path.
```
