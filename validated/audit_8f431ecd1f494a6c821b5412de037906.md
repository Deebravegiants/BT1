### Title
Block-Level Reference Script Size Limit Bypassed via Sub-Transactions in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`, `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs`)

---

### Summary

The Dijkstra era introduces nested sub-transactions (`dtbrSubTransactions`) inside a top-level transaction. The per-transaction reference script size check correctly aggregates reference scripts from both the top-level transaction and all its sub-transactions via `batchNonDistinctRefScriptsSize`. However, the block-level check (`validateBodyRefScriptsSizeTooBig`) still uses `totalRefScriptSizeInBlock`, which calls `txNonDistinctRefScriptsSize` per top-level transaction only — completely ignoring sub-transaction reference scripts. An attacker can craft Dijkstra-era transactions that each individually pass the per-tx limit while collectively causing the block to deserialize far more reference script bytes than `maxRefScriptSizePerBlock` permits.

---

### Finding Description

**Root cause — two divergent size-counting functions:**

`txNonDistinctRefScriptsSize` (used by the block-level check) only counts reference scripts reachable from the top-level transaction's `inputsTxBodyL` and `referenceInputsTxBodyL`:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [1](#0-0) 

`batchNonDistinctRefScriptsSize` (used by the per-tx check) additionally sums reference scripts from every sub-transaction:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [2](#0-1) 

**Per-transaction check (correct — uses batch function):**

`validateAllRefScriptSize` in the Dijkstra LEDGER rule calls `batchNonDistinctRefScriptsSize`, so the per-tx limit is enforced over the full batch (top-level + all sub-txs): [3](#0-2) 

**Block-level check (incomplete — ignores sub-txs):**

`validateBodyRefScriptsSizeTooBig` calls `totalRefScriptSizeInBlock`, which iterates over top-level transactions and calls `txNonDistinctRefScriptsSize` for each — the non-batch variant that does not recurse into sub-transactions: [4](#0-3) [5](#0-4) 

The Dijkstra BBODY rule inherits this Conway block-level check without substituting a batch-aware variant.

**Attack path:**

1. Attacker creates N top-level Dijkstra transactions, each with zero or minimal top-level reference inputs.
2. Each top-level transaction embeds K sub-transactions, each referencing large Plutus scripts stored in UTxO outputs.
3. Each transaction's total batch size = (top-level ref scripts) + (sub-tx ref scripts) ≤ `maxRefScriptSizePerTx` → passes `validateAllRefScriptSize`.
4. The block-level check sees only the top-level reference scripts (≈ 0 bytes per tx) → total block size reported as ≈ 0 → passes `validateBodyRefScriptsSizeTooBig`.
5. Actual deserialization work = N × (sub-tx ref script bytes) >> `maxRefScriptSizePerBlock`.

---

### Impact Explanation

The `maxRefScriptSizePerBlock` protocol parameter was introduced specifically to prevent the June 2024 DDoS attack where large reference scripts caused disproportionate deserialization cost per block. [6](#0-5) 

By exploiting the gap between the per-tx batch check and the block-level non-batch check, an attacker can submit a block whose actual reference script deserialization burden is a multiple of `maxRefScriptSizePerBlock` (bounded only by `N × maxRefScriptSizePerTx`). This exceeds the intended validation resource limit, matching the **Medium** allowed impact: *"Attacker-controlled transactions... exceed intended validation limits."*

---

### Likelihood Explanation

- Requires no privileged access: any transaction author can craft a Dijkstra-era top-level transaction with sub-transactions.
- The Dijkstra era is the current production target era; sub-transactions are a first-class feature.
- The attacker pays fees proportional to the top-level transaction size and the per-tx ref script fee (`minFeeRefScriptCostPerByte`), but the block-level cap — the primary throughput guard — is not enforced for sub-tx scripts.
- The attack is deterministic and reproducible.

---

### Recommendation

Replace the call to `txNonDistinctRefScriptsSize` inside `totalRefScriptSizeInBlock` (or its Dijkstra-era override) with a batch-aware variant that also sums sub-transaction reference scripts when the era supports them. Concretely, the Dijkstra BBODY rule should override `validateBodyRefScriptsSizeTooBig` to use `batchNonDistinctRefScriptsSize` (or an equivalent block-level accumulator) instead of `txNonDistinctRefScriptsSize`.

---

### Proof of Concept

**Parameters (illustrative):**
- `maxRefScriptSizePerTx` = 200 KiB
- `maxRefScriptSizePerBlock` = 1 MiB
- Script size = 50 KiB (large Plutus script stored in UTxO)

**Attacker constructs 10 top-level transactions, each with:**
- 0 top-level reference inputs (0 bytes counted by block check)
- 4 sub-transactions, each referencing the 50 KiB script (200 KiB total per batch)

**Validation outcome:**
- Per-tx check: `batchNonDistinctRefScriptsSize` = 200 KiB ≤ 200 KiB → **passes**
- Block check: `totalRefScriptSizeInBlock` = 10 × 0 KiB = 0 KiB ≤ 1 MiB → **passes**
- Actual deserialization: 10 × 200 KiB = **2 MiB** — 2× the intended block limit

The block-level guard is fully bypassed. Each honest node validating this block must deserialize 2 MiB of reference scripts instead of the protocol-intended maximum of 1 MiB, replicating the resource-exhaustion class of the June 2024 DDoS attack through the new sub-transaction surface.

totalRefScriptSizeInBlock ::
  (AlonzoEraTx era, BabbageEraTxBody era) => ProtVer -> StrictSeq (Tx TopTx era) -> UTxO era -> Int
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      let updatedUtxo = accUtxo `Map.union` unUTxO toAdd
          toAdd
            | IsValid True <- tx ^. isValidTxL = txouts $ tx ^. bodyTxL
            | otherwise = collOuts $ tx ^. bodyTxL
       in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
