### Title
Fee Proposal Bounds Check Bypassed During Startup Window Allows Unconstrained Fee Market Manipulation — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` contains a bounds check that validates `fee_proposal_fri` against `fee_actual`. However, the check is gated on `fee_actual` being `Some`, and `fee_actual` is `None` for the first `fee_proposal_window_size` (10) blocks. During this startup window, a proposer can set `fee_proposal_fri` to any arbitrary value. Validators accept it unconditionally, the value is bound into the `ProposalCommitment` that consensus votes on, and it is recorded in `fee_proposals_window` — poisoning the fee market for all subsequent blocks once the window fills.

### Finding Description

In `is_proposal_init_valid`, the fee proposal bounds check is:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(...);
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
``` [1](#0-0) 

When `fee_actual` is `None` — which is the case for the first `fee_proposal_window_size` (= 10) blocks because `compute_fee_actual` returns `None` when the sliding window is incomplete — the entire `if let` body is skipped. The check appears to validate `fee_proposal_fri`, but during startup it provides no constraint whatsoever. [2](#0-1) 

The proposer side documents the intended fallback: when `fee_actual` is `None`, `compute_proposer_fee_proposal` freezes at `self.l2_gas_price`:

```rust
let Some(fee_actual) = fee_actual else {
    warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
    return self.l2_gas_price;
};
``` [3](#0-2) 

But the validator never enforces this rule. A malicious proposer can send any `fee_proposal_fri` value (e.g., `u128::MAX`) in `ProposalInit` during the startup window, and every validator will accept it.

The accepted `fee_proposal_fri` is then:

1. **Bound into `ProposalCommitment`** via `proposal_commitment_from(partial_block_hash, fee_proposal_fri)` — the value consensus votes on: [4](#0-3) 

2. **Stored in `fee_proposals_window`** via `record_fee_proposal(height, init.fee_proposal_fri)` at decision time: [5](#0-4) 

3. **Used to compute `fee_actual`** for all future blocks once the window fills: [6](#0-5) 

4. **Propagated to state sync** via `BlockHeaderWithoutHash.fee_proposal_fri` and to the cende blob pipeline.

The `ProposalInitValidation` struct confirms that `fee_actual` is the only field that gates the bounds check, and it is explicitly `None` during startup: [7](#0-6) 

The window size is 10 blocks per the versioned constants: [8](#0-7) 

### Impact Explanation

A malicious proposer who controls any of the first 10 blocks can inject an arbitrary `fee_proposal_fri` (e.g., `u128::MAX`). Once the window fills, `fee_actual` is computed as the median of the poisoned window. All subsequent proposers must then propose `fee_proposal_fri` within a narrow margin of the manipulated `fee_actual`, or their proposals are rejected by validators. This forces the L2 gas price to converge toward the attacker-chosen value, causing sustained incorrect fee pricing with direct economic impact on all users. The manipulated value is permanently committed into the block hash chain.

This matches the allowed impact: **Critical — Incorrect fee, gas, resource accounting with economic impact.**

### Likelihood Explanation

The attack requires being the proposer for at least one of the first `fee_proposal_window_size` (10) blocks. In Tendermint-style BFT, any validator node can be selected as proposer for a given round. This is a normal, non-privileged operation. The attack is most effective if the attacker controls multiple proposer slots during the startup window, but even a single poisoned block shifts the median.

### Recommendation

When `fee_actual` is `None`, the validator should enforce the same fallback rule the proposer is documented to follow: `fee_proposal_fri` must equal `l2_gas_price_fri` (the value from `ProposalInitValidation`). Concretely, add a check in `is_proposal_init_valid`:

```rust
if fee_proposal_required {
    match (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri) {
        (None, Some(fee_proposal)) => {
            // Startup window: proposer must freeze at l2_gas_price.
            if fee_proposal != proposal_init_validation.l2_gas_price_fri {
                return Err(ValidateProposalError::InvalidProposalInit(...));
            }
        }
        (Some(fee_actual), Some(fee_proposal)) => {
            // Normal window: enforce margin bounds.
            ...
        }
        ...
    }
}
```

### Proof of Concept

1. Start the sequencer at genesis (or after a restart with an empty `fee_proposals_window`).
2. As the proposer for block 0 (Starknet version ≥ V0_14_3), construct a `ProposalInit` with `fee_proposal_fri = GasPrice(u128::MAX)`.
3. Send the proposal to validators. `is_proposal_init_valid` is called; `proposal_init_validation.fee_actual` is `None` (window empty), so the `if let` body is skipped entirely.
4. The proposal passes validation. `handle_proposal_part` computes `batcher_block_commitment = proposal_commitment_from(partial_block_hash, Some(u128::MAX))`. The proposer's `ProposalFin` carries the same commitment. The comparison at line 244 passes.
5. `decision_reached` calls `record_fee_proposal(BlockNumber(0), Some(GasPrice(u128::MAX)))`.
6. Repeat for blocks 1–9 (or as many as the attacker controls as proposer).
7. At block 10, `compute_fee_actual` returns `Some(GasPrice(u128::MAX))` (or a median skewed toward `u128::MAX`).
8. All subsequent proposers must propose `fee_proposal_fri` within `fee_proposal_margin_ppt` (2 per-thousand) of `u128::MAX`, or their proposals are rejected. The L2 gas price converges to `u128::MAX`, making all transactions prohibitively expensive. [9](#0-8) [10](#0-9)

pub fn compute_fee_actual(
    fee_proposals_window: &BTreeMap<BlockNumber, Option<GasPrice>>,
    height: BlockNumber,
    window_size: u64,
) -> Option<GasPrice> {
    let Some(start) = height.0.checked_sub(window_size) else {
        warn!(
            "Cannot compute fee_actual for height {height}: height is below window_size \
             ({window_size})"
        );
        return None;
    };
    let window_size_usize = usize::try_from(window_size).expect("window_size fits in usize");
    let mut window = Vec::with_capacity(window_size_usize);
    for source_height in (start..height.0).map(BlockNumber) {
        match fee_proposals_window.get(&source_height) {
            Some(Some(price)) => window.push(*price),
            Some(None) | None => {
                warn!(
                    "Cannot compute fee_actual for height {height}: fee_proposals_window has no \
                     recorded fee_proposal for height {source_height}"
                );
                return None;
            }
        }
    }
    window.sort();
    let mid = window_size_usize / 2;
    let median = if window_size_usize.is_multiple_of(2) {
        // Even: average of the two middle values, rounded down.
        // Overflow-safe averaging: a + (b - a) / 2 (safe because sorted, so b >= a).
        GasPrice(window[mid - 1].0 + (window[mid].0 - window[mid - 1].0) / 2)
    } else {
        window[mid]
    };
    Some(median)
