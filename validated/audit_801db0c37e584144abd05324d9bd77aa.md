### Title
Double-Claiming of Deposit Refunds Across Subtransactions in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, `getConsumedDijkstraValue` aggregates consumed values from all subtransactions using the **pre-transaction** ledger state for deposit lookups. Two subtransactions can both claim a refund for the same staking credential, double-counting the deposit in the consumed-value balance check. A disabled test in the repository explicitly acknowledges this gap with `error "TODO: predicate failure not yet implemented"`.

---

### Finding Description

`getConsumedDijkstraValue` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` computes the total consumed value for a Dijkstra top-level transaction by summing the top-level body's consumed value with each subtransaction body's consumed value:

```haskell
subTransactionsConsumedValue topTxBody =
  foldMap'
    (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
    (topTxBody ^. subTransactionsTxBodyL)
``` [1](#0-0) 

The `lookupStakingDeposit` and `lookupDRepDeposit` closures are derived once from the **pre-transaction** `certState` and passed unchanged to every subtransaction's consumed-value calculation:

```haskell
consumed = conwayConsumed   -- uses lookupDepositDState from pre-tx certState
``` [2](#0-1) 

`conwayConsumed` calls `getConsumedValue` with `lookupDepositDState` from the pre-transaction `certState`: [3](#0-2) 

The preservation-of-value check in the Dijkstra UTXO rule fires against this aggregated consumed value:

```
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [4](#0-3) 

If two subtransactions each contain `UnRegDepositTxCert stakingCred keyDeposit` for the **same** staking credential, `lookupStakingDeposit` returns the deposit for both (the credential is still registered in the pre-transaction snapshot), so the deposit is counted twice in `consumed`. For the balance equation to hold, the transaction's outputs may include the deposit amount twice, extracting it twice from the deposit pot while it was paid only once.

The developers have already identified this exact scenario. The test is written but disabled (`xit`) with an explicit acknowledgement that the blocking predicate failure does not yet exist:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  tx = mkBasicTx mkBasicTxBody
         & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [5](#0-4) 

The same structural gap exists for DRep deposits: `conwayDRepRefundsTxCerts` also uses the pre-transaction `lookupDRepDeposit` for every subtransaction. [6](#0-5) 

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker who registers a staking credential (paying `keyDeposit` once) can craft a top-level Dijkstra transaction containing two subtransactions that both deregister the same credential. The `consumed` calculation counts the deposit refund twice. For the balance equation to hold, the transaction outputs can include `2 × keyDeposit`, extracting twice the amount that was ever deposited. The deposit pot (`utxosDeposited`) is decremented twice for a deposit that was paid only once, permanently destroying the accounting invariant and creating lovelace from nothing.

The same attack applies to DRep deposits (`ppDRepDepositL`) and pool deposits, wherever the pre-transaction lookup is reused across subtransactions.

---

### Likelihood Explanation

The Dijkstra era introduces subtransactions as a novel feature. The attack path is a single crafted transaction submitted by any unprivileged user — no special role, key, or governance majority is required. The disabled test with `error "TODO: predicate failure not yet implemented"` confirms the developers are aware the guard is absent. Any user who can submit a Dijkstra-era transaction can attempt this.

---

### Recommendation

1. **Track claimed deposits across subtransactions.** Before aggregating subtransaction consumed values, maintain a set of credentials whose deposits have already been claimed. Reject (or zero out) any subsequent claim for the same credential within the same batch.
2. **Implement the missing predicate failure.** The disabled test at `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53` should be completed with a concrete `DijkstraLedgerPredFailure` constructor (e.g., `DuplicateDepositRefundInSubTransactions`) and enabled.
3. **Apply the same fix to DRep and pool deposit lookups** in `conwayDRepRefundsTxCerts` and `shelleyTotalRefundsTxCerts` when called from the Dijkstra subtransaction path.

---

### Proof of Concept

1. Register staking credential `C`, paying `keyDeposit = D` lovelace.
2. Craft a top-level Dijkstra transaction:
   - `subTx1`: inputs = `{utxo1}`, certs = `[UnRegDepositTxCert C D]`, outputs = `[addr ↦ value1 + D]`
   - `subTx2`: inputs = `{utxo2}`, certs = `[UnRegDepositTxCert C D]`, outputs = `[addr ↦ value2 + D]`
   - Top-level body: no inputs, no outputs, fee = 0
3. `getConsumedDijkstraValue` computes:
   - `consumed(subTx1)` = `value1 + D` (refund counted once)
   - `consumed(subTx2)` = `value2 + D` (refund counted again — same pre-tx lookup)
   - Total consumed = `value1 + value2 + 2D`
4. Total produced = `(value1 + D) + (value2 + D)` = `value1 + value2 + 2D` ✓ balance holds.
5. The deposit pot is decremented by `2D` while only `D` was ever deposited, creating `D` lovelace from nothing. [7](#0-6)

  forall era l.
  ( DijkstraEraTxBody era
  , EraUTxO era
  , Value era ~ MaryValue
  , STxLevel l era ~ STxBothLevels l era
  ) =>
  PParams era ->
  (Credential Staking -> Maybe Coin) ->
  (Credential DRepRole -> Maybe Coin) ->
  UTxO era ->
  TxBody l era ->
  Value era
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody

conwayDRepRefundsTxCerts ::
  (Foldable f, ConwayEraTxCert era) =>
  (Credential DRepRole -> Maybe Coin) ->
  f (TxCert era) ->
  Coin
conwayDRepRefundsTxCerts lookupDRepDeposit = snd . F.foldl' go (Map.empty, Coin 0)
  where
    go accum@(!drepRegsInTx, !totalRefund) = \case
      RegDRepTxCert cred deposit _ ->
        -- Track registrations
        (Map.insert cred deposit drepRegsInTx, totalRefund)
      UnRegDRepTxCert cred _
        -- DRep previously registered in the same tx.
        | Just deposit <- Map.lookup cred drepRegsInTx ->
            (Map.delete cred drepRegsInTx, totalRefund <+> deposit)
        -- DRep previously registered in some other tx.
        | Just deposit <- lookupDRepDeposit cred -> (drepRegsInTx, totalRefund <+> deposit)
      _ -> accum
