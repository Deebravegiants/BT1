### Title
Unchecked Detached Promise Transfer Silently Drops Excess-Deposit Refunds — (`crates/contract/src/lib.rs`)

---

### Summary

Two locations in the MPC contract issue a NEAR token refund via `Promise::new(...).transfer(diff).detach()` without ever checking whether the transfer succeeded. If the promise fails, the excess deposit is permanently lost to the caller with no revert and no error signal. This is the direct NEAR analog of the M-08 unchecked low-level call return value.

---

### Finding Description

**Location 1 — `require_deposit` (called from every user-facing request method)** [1](#0-0) 

```rust
Some(diff) => {
    if diff > NearToken::from_yoctonear(0) {
        log!("refund excess deposit {diff} to {predecessor}");
        Promise::new(predecessor.clone()).transfer(diff).detach(); // ← result discarded
    }
}
```

`require_deposit` is invoked inside `check_request_preconditions`, which is called unconditionally by `sign`, `request_app_private_key`, and `verify_foreign_transaction`. [2](#0-1) 

**Location 2 — `submit_participant_info`** [3](#0-2) 

```rust
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach(); // ← result discarded
}
```

In both cases `.detach()` is the NEAR SDK's explicit "fire-and-forget" mechanism: the runtime schedules the promise but the contract never receives or inspects its outcome. If the transfer action fails (e.g., the recipient account is deleted between the call and promise execution, or the runtime rejects the action for any reason), the failure is silently swallowed. The outer transaction has already succeeded, the deposit has been consumed, and the caller has no recourse.

The correct pattern is to **not** call `.detach()` and instead return the refund promise (or chain it as a dependency), so that a failed transfer causes the entire transaction to revert.

---

### Impact Explanation

Any caller of `sign`, `request_app_private_key`, `verify_foreign_transaction`, or `submit_participant_info` who attaches more than the minimum required deposit may permanently lose the excess. The contract's accounting invariant — "excess deposits are always returned" — is broken without any on-chain signal. Funds are neither returned to the caller nor credited anywhere; they remain in the contract balance, effectively stolen from the user.

This maps to the **Medium** allowed impact: *balance/accounting invariant broken without requiring network-level DoS or operator misconfiguration*.

---

### Likelihood Explanation

The entry path requires no privilege: any unprivileged account can call `sign` or `submit_participant_info` with an excess deposit. The failure condition (a detached transfer promise failing) is rare under normal NEAR operation but is not impossible — account deletion in the same block, runtime-level action rejection, or future protocol changes could trigger it. More importantly, the code is structurally incorrect regardless of current failure rates: the contract makes a financial promise it cannot enforce.

---

### Recommendation

Remove `.detach()` and propagate the refund promise as the return value of the enclosing method, or chain it so that a failed transfer reverts the transaction. For example, in `require_deposit`:

```rust
// Before (silent failure):
Promise::new(predecessor.clone()).transfer(diff).detach();

// After (failure reverts the call):
// Return the Promise from require_deposit and chain it in the caller,
// or use env::promise_return to make it the transaction's outcome.
```

For `submit_participant_info`, the refund promise should be returned from the `#[payable]` method so the NEAR runtime enforces its success.

---

### Proof of Concept

1. Alice calls `sign(request)` attaching `10 NEAR` (minimum is 1 yoctoNEAR).
2. `require_deposit` computes `diff = 10 NEAR - 1 yoctoNEAR` and fires `Promise::new(alice).transfer(diff).detach()`.
3. Between the call and promise execution, Alice's account is deleted (e.g., via a concurrent `delete_account` action in the same block).
4. The transfer promise fails silently; `.detach()` means the contract never observes the failure.
5. The outer `sign` transaction succeeds, the yield is enqueued, and Alice's `10 NEAR` excess is permanently locked in the contract with no refund path.

The same scenario applies to `submit_participant_info` at line 847. [4](#0-3) [5](#0-4)
