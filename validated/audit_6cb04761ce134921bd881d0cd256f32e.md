### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Enables Queue-Saturation DoS — (File: `crates/contract/src/lib.rs`)

### Summary
The `verify_foreign_transaction` function stores pending requests under a key that contains no caller identity. Any unprivileged NEAR account can pre-fill the 128-slot fan-out queue for any specific foreign-chain transaction, causing every subsequent legitimate submission for that transaction to panic with `PendingRequestQueueFull` and permanently stall the request lifecycle for that foreign tx until the queue drains via timeouts — which the attacker can continuously prevent.

### Finding Description

**Root cause — caller-agnostic request key**

`verify_foreign_transaction` in `crates/contract/src/lib.rs` (line 549) converts the user-supplied args via `args_into_verify_foreign_tx_request()`:

```rust
// crates/contract/src/dto_mapping.rs  lines 840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

The resulting `VerifyForeignTransactionRequest` struct (defined in `crates/near-mpc-contract-interface/src/types/foreign_chain.rs` lines 124-128) contains only `(domain_id, request, payload_version)` — **no predecessor/caller identity**. This is the map key used in `pending_verify_foreign_tx_requests`.

Contrast with `sign()`, which derives a `tweak` from `(predecessor_id, path)` before storing the request:

```rust
// crates/near-mpc-crypto-types/src/sign.rs  lines 118-125
pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
    let tweak = crate::kdf::derive_tweak(predecessor_id, path);
    SignatureRequest { domain_id: domain, tweak, payload }
}
```

Because `SignatureRequest` embeds the caller-specific tweak, no two different callers share the same map key for `sign()`. `verify_foreign_transaction` has no such protection.

**Queue saturation**

`push_pending_yield` in `crates/contract/src/pending_requests.rs` (lines 43-60) panics when the queue reaches `MAX_PENDING_REQUEST_FAN_OUT = 128`:

```rust
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }
            .to_string(),
    );
}
```

An attacker who submits 128 `verify_foreign_transaction` calls for the same `(domain_id, tx_id, confirmations, extractors, payload_version)` tuple saturates the queue. Every subsequent call from any account — including the legitimate bridge caller — panics and is reverted.

The codebase's own test at `crates/contract/src/lib.rs` lines 3208-3263 explicitly acknowledges the caller-agnostic key and that "a different account would today be blocked from receiving a response by alice's submission," confirming the design is aware of cross-caller queue sharing.

**Sustained attack**

NEAR yield-resume promises time out after a protocol-defined window. `pop_oldest_pending_yield` removes one entry per timeout. The attacker can continuously resubmit to keep the queue at 128, sustaining the DoS indefinitely at a cost of 1 yoctoNEAR + gas per slot per timeout cycle.

### Impact Explanation

Any bridge or dApp that calls `verify_foreign_transaction` for a predictable foreign-chain transaction (e.g., a Bitcoin tx whose `tx_id` is visible in the mempool) can be permanently blocked from completing that request. The request lifecycle for that specific foreign tx is broken: the user's yield-resume promise is never created, so no signature is ever returned, and the bridge action fails. This is a request-lifecycle invariant violation that does not require network-level DoS or operator misconfiguration — it is triggered entirely through the permissionless contract API.

### Likelihood Explanation

- **Permissionless entry**: `verify_foreign_transaction` requires only 1 yoctoNEAR deposit and sufficient gas; no privileged role is needed.
- **Observable target**: Foreign-chain tx IDs (Bitcoin, EVM, etc.) are public. An attacker monitoring the mempool or a bridge's pending queue can identify the exact parameters before the legitimate caller submits.
- **Low cost**: Saturating the queue costs 128 × (1 yoctoNEAR + ~10 TGas) — negligible on NEAR.
- **Sustainable**: The attacker can refill slots as they drain, maintaining the DoS indefinitely.

### Recommendation

Include the caller's account ID in the stored request key for `verify_foreign_transaction`, analogous to how `sign()` embeds the caller-derived tweak. One approach: add a `predecessor_id` field to `VerifyForeignTransactionRequest` (or derive a nonce from it), so each caller occupies a distinct map key and cannot saturate another caller's queue. Alternatively, enforce a per-account submission limit before the queue cap is reached.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
