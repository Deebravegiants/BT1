### Title
Trailing-Byte Acceptance in `coinbase::convert<T>` Enables Key-Blob Message Malleability Across All Public API Deserialization Paths — (`File: include-internal/cbmpc/internal/core/convert.h`)

### Summary
The `coinbase::convert<T>(T& dst, mem_t src)` template function, used by every key-blob deserialization function in the public API, does not enforce a trailing-byte check. The library's own `coinbase::deser` explicitly rejects trailing bytes with the comment "Strict deserialization: reject trailing bytes — This prevents message malleability where two different byte sequences could deserialize to the same value." Because all key-blob deserialization paths call `convert` instead of `deser`, any key blob with appended garbage bytes is silently accepted as valid, producing the same parsed key as the canonical blob. This is the direct analog of M-04: excess input is not rejected at the boundary.

### Finding Description

`coinbase::deser` enforces a strict trailing-byte check:

```cpp
// include-internal/cbmpc/internal/core/convert.h, lines 253-266
template <typename... ARGS>
error_t deser(mem_t bin, ARGS&... args) {
  converter_t converter(bin);
  converter.convert(args...);
  error_t rv = converter.get_rv();
  if (rv != SUCCESS) return rv;
  // Strict deserialization: reject trailing bytes
  if (converter.get_offset() != converter.get_size()) {
    return coinbase::error(E_BADARG);
  }
  return SUCCESS;
}
```

`coinbase::convert<T>` does not:

```cpp
// include-internal/cbmpc/internal/core/convert.h, lines 276-282
template <typename T>
error_t convert(T& dst, mem_t src) {
  if (src.size < 0 || (src.size && !src.data)) return coinbase::error(E_BADARG);
  converter_t converter(src);
  converter.convert(dst);
  return converter.get_rv();   // ← no offset == size check
}
```

Every public-API key-blob deserialization function calls `coinbase::convert`, not `coinbase::deser`:

### Impact Explanation

Any caller can supply a key blob with arbitrary trailing bytes appended. The library accepts it without error, parses the same key material as the canonical blob, and proceeds with DKG refresh, signing, or HD derivation. This creates a one-to-many mapping from byte sequences to parsed keys:

- **Unsafe state acceptance**: The public API boundary accepts structurally invalid blobs (those with unconsumed bytes) as valid, violating the invariant the library itself documents in `deser`.
- **Message malleability**: Two distinct byte sequences deserialize to the same `key_t`. Any application that hashes, commits to, or compares key blobs by their raw bytes (e.g., for deduplication, audit logs, or PVE ciphertext binding) will diverge between a canonical blob and a padded one that the library treats as identical.
- **Forward-version confusion**: A blob crafted to look like a future format version (with extra fields in the trailing region) is silently accepted under the current schema, creating a path for version-confusion attacks if the format is ever extended.

### Likelihood Explanation

The attack surface is the entire public C++ and C API. Any caller that stores, transmits, or receives key blobs from an untrusted source (the documented use case for PVE recovery and multi-party refresh) can supply a padded blob. No threshold collusion or key leakage is required. The root cause is a single missing two-line check that the library already applies correctly in `deser` but omits in `convert`.

### Recommendation

Replace every `coinbase::convert(blob, in)` call in key-blob deserialization functions with `coinbase::deser(in, blob)`, which already enforces the trailing-byte check. Alternatively, add the offset check directly to `coinbase::convert<T>`:

```cpp
template <typename T>
error_t convert(T& dst, mem_t src) {
  if (src.size < 0 || (src.size && !src.data)) return coinbase::error(E_BADARG);
  converter_t converter(src);
  converter.convert(dst);
  if (converter.get_rv() != SUCCESS) return converter.get_rv();
  if (converter.get_offset() != converter.get_size()) return coinbase::error(E_BADARG);
  return SUCCESS;
}
```
