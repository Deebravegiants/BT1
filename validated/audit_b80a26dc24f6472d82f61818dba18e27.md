### Title
Missing Caller Identity in `verify_foreign_transaction` Signed Payload Enables Cross-Caller Signature Replay - (File: crates/contract/src/lib.rs)

### Summary

The `verify_foreign_transaction` function does not bind the caller's (`predecessor_id`) identity into the signed payload or the stored request key. Unlike `sign()`, which derives a caller-specific tweak from `(predecessor_id, path)` and embeds it in `SignatureRequest`, the foreign-tx flow stores a caller-agnostic `VerifyForeignTransactionRequest` and signs with a hard-coded zero tweak. Any party who obtains the resulting MPC signature can replay it as if they had initiated the verification, breaking the caller-authentication invariant that the design document explicitly required.

### Finding Description

**`sign()` correctly binds the caller:**

`SignatureRequest::new` derives a tweak from `(predecessor_id, path)` and stores it in the request key. The MPC nodes sign with the derived key, so the signature is cryptographically bound to the caller. [1](#0-0) [2](#0-1) 

**`verify_foreign_transaction()` drops the caller entirely:**

The function reads `env::predecessor_account_id()` only for logging. It is never embedded in the stored request or the signed payload. [3](#0-2) 

`VerifyForeignTransactionRequest` has no `predecessor_id` or `tweak` field: [4](#0-3) 

On the node side, `build_signature_request` hard-codes a zero tweak, meaning the root key signs the payload directly: [5](#0-4) 

**Design document explicitly required caller-specific tweak derivation:**

The design doc specifies a `derivation_path` field in `VerifyForeignTransactionRequestArgs` and a `FOREIGN_TX_TWEAK_DERIVATION_PREFIX` to produce a caller-specific tweak — precisely to prevent the same `(predecessor_id, path)` from yielding the same key across `sign()` and `verify_foreign_transaction()`. The actual implementation omits both. [6](#0-5) 

**Fan-out confirms the caller-agnostic key:**

The contract explicitly tests and documents that different callers submitting the same foreign-tx request receive the identical signature response, confirming the request key is caller-agnostic by design in the current implementation: [7](#0-6) 

### Impact Explanation

The MPC signature over `ForeignTxSignPayloadV1 { request, values }` carries no caller identity. Any party who observes the signature on-chain (e.g., by watching NEAR receipts) can present it to a downstream bridge contract as if they had initiated the verification. In the primary stated use case — Omnibridge inbound flow where a user submits `verify_foreign_transaction` to prove a Bitcoin/EVM deposit and then claims funds on NEAR — an adversary can front-run or replay the signature to redirect the claim. This breaks the production safety invariant that the verification result is bound to the initiating account, enabling invalid bridge execution and potential double-spend conditions.

### Likelihood Explanation

The attack requires only that an adversary monitor NEAR receipts for a `return_verify_foreign_tx_and_clean_state_on_success` callback carrying the signature, then submit a competing claim to the bridge contract before the legitimate user. No privileged access, threshold collusion, or TEE compromise is needed. The fan-out behavior (multiple callers receiving the same response) is explicitly tested, confirming the surface is reachable by any unprivileged caller.

### Recommendation

Bind the caller's identity into the signed payload by:

1. Restoring the `derivation_path` field to `VerifyForeignTransactionRequestArgs` and `VerifyForeignTransactionRequest` as specified in the design document.
2. Deriving a caller-specific tweak from `(predecessor_id, derivation_path)` using the intended `FOREIGN_TX_TWEAK_DERIVATION_PREFIX` and storing it in `VerifyForeignTransactionRequest`.
3. Passing the tweak to `build_signature_request` on the node side instead of `Tweak::new([0u8; 32])`.

This ensures the MPC signature is cryptographically bound to the initiating account, matching the security model of `sign()` and the original design intent.

### Proof of Concept

1. Alice calls `verify_foreign_transaction({ bitcoin_tx_id: X, ... })` with 1 yoctoNEAR deposit.
2. MPC nodes verify the Bitcoin transaction and call `respond_verify_foreign_tx`, which resumes Alice's yield with `VerifyForeignTransactionResponse { payload_hash, signature }`.
3. Bob monitors NEAR receipts and extracts the `signature` from Alice's receipt outcome.
4. Bob presents `signature` to the bridge contract claiming the Bitcoin deposit for his own NEAR account.
5. Because `signature` covers only `ForeignTxSignPayloadV1 { request: { tx_id: X, ... }, values: [...] }` with no caller field, the bridge contract cannot distinguish Bob's claim from Alice's — the signature is equally valid for both.
6. Bob receives Alice's bridged funds; Alice's claim fails or is pre-empted.

## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.
