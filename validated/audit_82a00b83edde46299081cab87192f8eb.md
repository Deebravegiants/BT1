### Title
Mock Attestation Bypass of TEE Verification Accepted by Production Contract - (File: `crates/contract/src/lib.rs`, `crates/mpc-attestation/src/attestation.rs`)

### Summary
The production MPC contract's `submit_participant_info` endpoint unconditionally accepts `Attestation::Mock(MockAttestation::Valid)` from any caller. `MockAttestation::Valid` always passes verification with no real TEE checks. This is development/testnet code that was never removed from the production interface, directly analogous to the pSTAKE `_mint(msg.sender, ...)` testnet placeholder. A participant (or prospective participant) can register a permanently-valid fake attestation, bypassing the TEE security model entirely.

### Finding Description

The `Attestation` enum in `crates/mpc-attestation/src/attestation.rs` exposes a `Mock` variant alongside the real `Dstack` variant:

```rust
pub enum Attestation {
    Dstack(DstackAttestation),
    Mock(MockAttestation),
}
``` [1](#0-0) 

`MockAttestation::Valid` unconditionally returns `Ok(())` during verification:

```rust
MockAttestation::Valid => Ok(()),
``` [2](#0-1) 

The DTO mapping layer in `dto_mapping.rs` passes `Mock` attestations through to the internal type without any guard:

```rust
dtos::Attestation::Mock(mock_attestation) => {
    Attestation::Mock(mock_attestation.into_contract_type())
}
``` [3](#0-2) 

The public `submit_participant_info` method in `lib.rs` accepts this attestation from any caller and stores it as a verified attestation: [4](#0-3) 

The stored `VerifiedAttestation::Mock(MockAttestation::Valid)` also always passes **re-verification** in `re_verify`:

```rust
Self::Mock(mock_attestation) => mock_attestation.verify_constraints(...)
// MockAttestation::Valid => Ok(())
``` [5](#0-4) 

Additionally, `default_measurements()` in the same file compiles **dev TCB measurements** into the production binary with an explicit security acknowledgment that they must be removed before mainnet:

```rust
// TODO(#1433): Security - remove dev measurements from production builds after testing is complete
include_measurements!("assets/tcb_info_dev.json"),
``` [6](#0-5) 

Furthermore, `TeeState::with_mocked_participant_attestations` seeds the initial `TeeState` with `VerifiedAttestation::Mock(MockAttestation::Valid)` for all initial participants, with a TODO acknowledging this is a placeholder that must be replaced: [7](#0-6) 

### Impact Explanation

A participant (or any account that is subsequently voted in as a participant) can call `submit_participant_info` with `Attestation::Mock(Valid)`. Their stored attestation permanently passes all re-verification checks. The `assert_caller_is_attested_participant_and_protocol_active` gate — which guards `respond`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, and all other privileged node operations — will pass for this account even though it is not running inside a real TEE. This breaks the foundational security invariant of the MPC network: that all signing participants are running authorized code inside a Trusted Execution Environment. A participant operating outside a TEE can access signing flows and contribute key-event votes, undermining the confidentiality and integrity guarantees of the threshold key shares.

**Impact: High** — participant-state and attestation-authorization bypass that breaks production TEE safety invariants without requiring threshold collusion.

### Likelihood Explanation

The `submit_participant_info` endpoint is publicly callable by any NEAR account. The `dtos::Attestation::Mock` variant is part of the published contract interface (it appears in the ABI snapshot). Any existing participant who wishes to avoid TEE overhead, or any account being onboarded as a new participant, can trivially submit `Mock(Valid)` instead of a real Dstack attestation. No special tooling is required — a standard NEAR CLI call suffices.

### Recommendation

1. **Remove `Attestation::Mock` from the production contract interface.** Gate the `Mock` variant behind a compile-time feature flag (e.g., `#[cfg(feature = "test-utils")]`) so it is excluded from the WASM binary deployed to mainnet.
2. **Remove dev TCB measurements** (`tcb_info_dev.json`) from `default_measurements()` in production builds, as the `TODO(#1433)` comment explicitly requires.
3. **Replace `with_mocked_participant_attestations`** with a real attestation submission flow during initialization, as tracked by `TODO(#1087)`.
4. Add a contract-level guard in `submit_participant_info` that rejects any `Attestation::Mock` variant with a clear error, as a defense-in-depth measure even if the DTO type is restricted.

### Proof of Concept

```bash
# Any NEAR account can call this with a mock attestation
near call <mpc_contract> submit_participant_info '{
  "proposed_participant_attestation": {"Mock": "Valid"},
  "tls_public_key": "<attacker_ed25519_key>"
}' --accountId attacker.near --deposit 0.1
```

After this call, `tee_state.stored_attestations` contains an entry for `attacker.near` with `VerifiedAttestation::Mock(MockAttestation::Valid)`. Every subsequent call to `reverify_participants` for this entry returns `TeeQuoteStatus::Valid` unconditionally, satisfying `assert_caller_is_attested_participant_and_protocol_active` for all privileged node operations once the account is a participant.     pub fn submit_participant_info(
        &mut self,
        proposed_participant_attestation: dtos::Attestation,
        tls_public_key: dtos::Ed25519PublicKey,
    ) -> Result<(), Error> {
        let proposed_participant_attestation =
            proposed_participant_attestation.try_into_contract_type()?;

        let account_key = env::signer_account_pk();
        let account_id = Self::assert_caller_is_signer();

        log!(
            "submit_participant_info: signer={}, proposed_participant_attestation={:?}, account_key={:?}",
            account_id,
            proposed_participant_attestation,
            account_key
        );

        // Save the initial storage usage to know how much to charge the proposer for the storage
        // used
        let initial_storage = env::storage_usage();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        // The node always signs submissions with an Ed25519 key
        // (`near_signer_key`), so the signer key here is Ed25519 in practice.
        // Reject non-Ed25519 signer keys rather than silently storing a value
        // we could never match against in `is_caller_an_attested_participant`.
        let account_public_key = dtos::Ed25519PublicKey::try_from(&account_key).map_err(|_| {
            InvalidParameters::InvalidTeeRemoteAttestation {
                reason: "signer account key must be Ed25519".to_string(),
            }
        })?;

        // Add the participant information to the contract state
        let attestation_insertion_result = self
            .tee_state
            .add_participant(
                NodeId {
                    account_id: account_id.clone(),
                    tls_public_key,
                    account_public_key,
                },
                proposed_participant_attestation,
                tee_upgrade_deadline_duration,
            )
            .map_err(|err| {
                let reason = match &err {
                    AttestationSubmissionError::InvalidAttestation(_) => {
                        format!("TeeQuoteStatus is invalid: {err}")
                    }
                    AttestationSubmissionError::TlsKeyOwnedByOtherAccount => err.to_string(),
                };
                InvalidParameters::InvalidTeeRemoteAttestation { reason }
            })?;

    /// Creates a [`TeeState`] with an initial set of participants that will receive a valid mocked attestation.
    pub(crate) fn with_mocked_participant_attestations(participants: &Participants) -> Self {
        let mut tee_state = Self::default();

        for (account_id, _, participant_info) in participants.participants() {
            let tls_public_key = participant_info.tls_public_key.clone();
            // TODO(#1087): replace account_public_key with a real account public
            // key passed in by the caller. `Participants` does not currently
            // carry the operator's account public key, so a mocked entry
            // cannot record the real one and we use the TLS key as a unique
            // per-participant placeholder. The mock keeps the
            // participant from being kicked out of an empty `TeeState` until
            // a real `submit_participant_info` call replaces it (keyed by
            // TLS), but any caller-facing check that compares
            // `signer_account_pk` against the stored key will fail until
            // then. #1087 tracks threading real attestations through
            // initialization so this sentinel can go away.
            let node_id = NodeId {
                account_id: account_id.clone(),
                tls_public_key: tls_public_key.clone(),
                // Use tls_public_key as account_public_key instead of hardcoded
                // Ed25519PublicKey::from([0u8; 32]) so that same account public
                // key isn't associated with different tls keys.
                // This is not a fix for above issue: #1087, which should be
                // addressed outside this PR.
                account_public_key: tls_public_key.clone(),
            };

            tee_state.stored_attestations.insert(
                tls_public_key,
                NodeAttestation {
                    node_id,
                    verified_attestation: VerifiedAttestation::Mock(
                        attestation::MockAttestation::Valid,
                    ),
                },
            );
        }

        tee_state
    }
