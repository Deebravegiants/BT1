### Title
Zero-Fee Transfer Causes Unconditional `safeTransfer(protocolTreasury, 0)` to Revert, Permanently Freezing User Claims — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

---

### Summary

`MerkleDistributor.claim()` unconditionally calls `safeTransfer(protocolTreasury, fee)` even when `fee == 0`. Because `feeInBPS` is explicitly allowed to be zero (no lower-bound check), and because certain ERC-20 tokens (e.g. USDT) revert on zero-value transfers, every claim attempt will revert whenever the protocol has set a zero fee, permanently freezing all unclaimed yield for every user of that distributor instance.

---

### Finding Description

In `MerkleDistributor.sol`, `feeInBPS` is initialized and updated with only an upper-bound check: [1](#0-0) [2](#0-1) 

This means `feeInBPS = 0` is a valid, intended configuration (the protocol owner can forgo the fee). When a user calls `claim()`, the fee is computed and then transferred unconditionally: [3](#0-2) 

When `feeInBPS == 0`, `fee` evaluates to `0`, and `safeTransfer(protocolTreasury, 0)` is called. Tokens such as USDT revert on zero-value `transfer()` calls. Because the state update (marking the claim) happens **before** the transfer, the transaction reverts after state is written, but since Solidity reverts the entire transaction, the user's claim is never recorded — yet the call always reverts, making the claim permanently inaccessible.

The sibling contract `KernelTop100MerkleDistributor.sol` already demonstrates the correct fix: [4](#0-3) 

`MerkleDistributor.sol` lacks this guard entirely.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

All users of a `MerkleDistributor` instance whose `feeInBPS` is set to `0` and whose `token` is a zero-transfer-reverting ERC-20 (e.g. USDT) are completely unable to claim their merkle-distributed rewards. The contract holds their tokens but every `claim()` call reverts. No admin action can rescue the situation without an upgrade, because `feeInBPS` being set back to non-zero would fix the revert but would incorrectly charge users a fee they were never supposed to pay.

---

### Likelihood Explanation

- `feeInBPS = 0` is an explicitly supported configuration (no lower-bound guard, owner can call `setFeeInBPS(0)`).
- The `token` address is settable to any ERC-20 via `setToken()`, and USDT is a common reward token in LRT ecosystems.
- Any unprivileged user calling `claim()` triggers the revert — no special role or timing is required.
- The combination is realistic and the code path is always executed.

---

### Recommendation

Guard the fee transfer with a zero-check, mirroring the pattern already used in `KernelTop100MerkleDistributor.sol`:

```solidity
// In MerkleDistributor.claim()
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

---

### Proof of Concept

1. Deploy `MerkleDistributor` with a USDT-like token (reverts on zero-value transfer) and `feeInBPS = 0`.
2. Set a valid merkle root with a leaf for `alice` with `cumulativeAmount = 100e6`.
3. Alice calls `claim(1, alice, 100e6, proof)`.
4. Execution reaches line 138: `fee = (100e6 * 0) / 10_000 = 0`.
5. `safeTransfer(alice, 100e6)` succeeds.
6. `safeTransfer(protocolTreasury, 0)` reverts (USDT reverts on zero-value transfer).
7. Entire transaction reverts. Alice's claim is never recorded and she can never claim her tokens. [3](#0-2)
