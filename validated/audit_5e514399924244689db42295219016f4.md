### Title
Missing Chainlink Price Feed Validation Allows Stale or Zero Prices to Corrupt Core Deposit and Withdrawal Accounting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

---

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but discards all validation fields (`roundId`, `startedAt`, `updatedAt`, `answeredInRound`), and does not check whether the returned `price` is positive. This is the direct analog of the external report's pattern: an external call's return value is used without validating correctness, leading to incorrect protocol behavior. The price is consumed by every depositor and withdrawer path in the protocol.

---

### Finding Description
`contracts/oracles/ChainlinkPriceOracle.sol` line 52 reads only the `price` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No checks are performed on:
- `updatedAt` — whether the price is stale (e.g., Chainlink heartbeat missed)
- `answeredInRound >= roundId` — whether the round completed
- `price > 0` — whether the price is valid

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` (lines 30–32) in the same repository explicitly performs all three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated `getAssetPrice()` result flows into:

1. `LRTDepositPool.getRsETHAmountToMint()` (line 520):
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```
2. `LRTWithdrawalManager.getExpectedAssetAmount()` (line 593):
   ```solidity
   underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
   ```
3. `LRTWithdrawalManager._unlockWithdrawalRequests()` via `_createUnlockParams()` (line 848).

---

### Impact Explanation
**Scenario A — Zero price returned**: If `price == 0` (possible during Chainlink circuit-breaker events or feed misconfiguration), `getAssetPrice` returns 0. In `getExpectedAssetAmount`, this causes a division-by-zero revert, permanently blocking `initiateWithdrawal` for the affected asset until the oracle recovers. Users with rsETH cannot queue withdrawals — **temporary freezing of funds**.

**Scenario B — Stale price returned**: If the Chainlink heartbeat is missed (e.g., 24-hour heartbeat feeds during network congestion), the stale price is used to compute rsETH minted on deposit and assets owed on withdrawal. A stale price that is lower than the true price causes `getExpectedAssetAmount` to return more assets than the user is entitled to, allowing a withdrawer to drain more LST than their rsETH represents — **theft of unclaimed yield / temporary incorrect accounting**.

Impact classification: **Medium** (temporary freezing of funds) and **Low** (contract fails to deliver promised returns).

---

### Likelihood Explanation
Chainlink price feeds for LST/ETH pairs (stETH/ETH, ETHx/ETH, rETH/ETH) use heartbeat-based updates (typically 24 hours). During periods of network congestion, oracle node downtime, or Chainlink circuit-breaker activation, `updatedAt` can lag significantly. This is a known, documented failure mode. The protocol's own pool oracle (`ChainlinkOracleForRSETHPoolCollateral`) guards against it, confirming the team is aware of the risk. Likelihood: **Low to Medium**.

---

### Recommendation
Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > HEARTBEAT_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

### Proof of Concept

1. Chainlink's stETH/ETH feed misses its 24-hour heartbeat; `updatedAt` is now 25 hours old and `price` reflects a value from before a significant stETH depeg event.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, largeAmount, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)`.
4. `ChainlinkPriceOracle` returns the stale (pre-depeg, higher) price without any staleness check.
5. `rsethAmountToMint = largeAmount * staleHighPrice / rsETHPrice` — attacker receives more rsETH than the current fair value of their stETH deposit.
6. Attacker immediately calls `LRTWithdrawalManager.instantWithdrawal` (if enabled) or queues a withdrawal for a different asset at the inflated rsETH balance, extracting value from other rsETH holders.

Alternatively, for the zero-price DoS path:
1. Chainlink feed returns `price = 0` (circuit breaker or misconfiguration).
2. Any call to `initiateWithdrawal(stETH, rsETHAmount, "")` reaches `getExpectedAssetAmount` → `lrtOracle.getAssetPrice(stETH)` → returns 0 → division by zero → revert.
3. All stETH withdrawal requests are blocked for the duration of the oracle failure.
