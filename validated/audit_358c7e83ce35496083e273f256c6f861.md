### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Pending `request` — (File: `crates/contract/src/lib.rs`)

---

### Summary

`MpcContract::respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key, but it never checks that `response.payload_hash` is the correct hash for the supplied `request`. A single malicious attested participant (strictly below the signing threshold) can replay any previously observed valid `(payload_hash, signature)` pair against a *different* pending request, causing the contract to resolve that request's yield with fabricated foreign-chain data.

---

### Finding Description

The intended execution flow for foreign-chain verification is:

1. A user calls `verify_foreign_transaction(request)`. The contract stores the request in `pending_verify_foreign_tx_requests` and opens a yield.
2. MPC nodes query the foreign chain, extract values, compute `payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))`, and collectively sign it with the root key (zero tweak).
3. One node calls `respond_verify_foreign_tx(request, { payload_hash, signature })`. The contract verifies the signature and resolves the yield, returning the response to the caller.

The contract's signature check in `respond_verify_foreign_tx` is:

```rust
// crates/contract/src/lib.rs  lines 718-734
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

The contract only checks: *"is `response.signature` a valid signature for `response.payload_hash` under the root key?"* It does **not** check: *"does `response.payload_hash` equal `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` for the supplied `request`?"*

Because `ForeignTxSignPayload` commits to the `ForeignChainRpcRequest` (including the transaction ID) and the extracted `values`, a signature produced for request A with values V_A is valid for `payload_hash_A = SHA-256(borsh({request_A, V_A}))`. That same `(payload_hash_A, sig_A)` pair can be submitted as the response to a completely different pending request B, and the contract will accept it — because the signature is cryptographically valid for `payload_hash_A` under the root key, and the contract never verifies that `payload_hash_A` encodes `request_B`.

The `VerifyForeignTransactionRequest` struct stored as the map key contains only `{request, domain_id, payload_version}`:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs  lines 124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

The `values` (extracted chain data) are never stored on-chain and are never re-derived by the contract during `respond_verify_foreign_tx`. The contract has no way to reconstruct the expected `payload_hash` from the `request` alone, and it makes no attempt to do so.

Contrast this with `respond` for regular signing, where the contract derives the expected public key from the request's `tweak` and verifies the signature against it — binding the response to the specific request. No equivalent binding exists for `respond_verify_foreign_tx`.

---

### Impact Explanation

A malicious attested participant (a single node, strictly below threshold) who has observed any prior valid `respond_verify_foreign_tx` on-chain response can:

1. Record `(payload_hash_A, sig_A)` from a previous call for request A (e.g., Bitcoin tx_id_A confirmed in block B_A).
2. Wait for a new pending request B (e.g., Bitcoin tx_id_B) to appear in `pending_verify_foreign_tx_requests`.
3. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
4. The contract verifies `sig_A` against `payload_hash_A` under the root key — this passes.
5. The yield for request B is resolved with `{ payload_hash: payload_hash_A, signature: sig_A }`.

The caller of `verify_foreign_transaction(request_B)` receives a response asserting that transaction B was verified with the extracted values from transaction A (e.g., a different block hash). A bridge service that trusts the contract's verification step and uses `response.payload_hash` to look up the attested block hash will act on fabricated data, enabling invalid bridge execution or double-spend conditions.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution.**

---

### Likelihood Explanation

- The attacker needs only to be a single attested participant — no threshold collusion is required.
- The replayed `(payload_hash, signature)` pair is publicly visible on-chain from any prior `respond_verify_foreign_tx` call.
- The attack is executable any time a new `verify_foreign_transaction` request is pending, which is the normal operating condition of any active bridge.
- No special tooling or privileged access beyond being an attested node is required.

---

### Recommendation

Inside `respond_verify_foreign_tx`, after looking up the domain's public key, recompute the minimum expected payload hash from the `request` and verify that `response.payload_hash` starts with the correct `request` prefix. Because the `values` are not stored on-chain, the contract cannot fully reconstruct `payload_hash`; however, it can enforce that the `ForeignChainRpcRequest` embedded in the signed payload matches the pending `request`. The cleanest fix is to include the `request` hash as a domain-separation prefix in the signed payload and verify it on-chain:

```rust
// Verify that the payload_hash commits to the correct request
let request_hash = env::sha256_array(
    borsh::to_vec(&request.request).expect("borsh serialization must succeed")
);
// The payload_hash must be derived from a payload whose request field matches
// the pending request. Nodes must include request_hash in the signed payload
// so the contract can verify this binding without knowing the extracted values.
```

Alternatively, store the `request` hash alongside the yield index in `pending_verify_foreign_tx_requests` and require that `response.payload_hash` begins with (or is derived from) that stored hash, making cross-request replay cryptographically impossible.

### Proof of Concept

**Setup:** Contract is in Running state with one attested participant `node_A`. A `ForeignTxDomain` exists.

**Step 1 — Observe a prior valid response:**
```
// On-chain: a previous call succeeded for request_A (Bitcoin tx_id = [0x01; 32])
// The response is publicly visible:
payload_hash_A = SHA-256(borsh(ForeignTxSignPayload::V1 {
    request: Bitcoin { tx_id: [0x01; 32], ... },
    values: [BlockHash([0xAA; 32])],
}))
sig_A = <valid ECDSA signature over payload_hash_A under root key>
```

**Step 2 — Submit a new request:**
```
// User submits verify_foreign_transaction for a different tx
verify_foreign_transaction({ request: Bitcoin { tx_id: [0x02; 32], ... }, domain_id, payload_version })
// Contract stores request_B = { request: Bitcoin { tx_id: [0x02; 32], ... }, ... } in pending map
```

**Step 3 — Malicious node replays old response:**
```
// node_A calls respond_verify_foreign_tx with request_B but response from request_A
respond_verify_foreign_tx(
    request = { request: Bitcoin { tx_id: [0x02; 32], ... }, domain_id, payload_version },
    response = { payload_hash: payload_hash_A, signature: sig_A }
)
```

**Step 4 — Contract accepts:**
```rust
// crates/contract/src/lib.rs lines 729-734
near_mpc_signature_verifier::verify_ecdsa_signature(
    sig_A,          // valid signature
    &payload_hash_A, // valid hash — but for a DIFFERENT request
    &secp_pk,       // root public key
).is_ok()           // → true
// Yield for request_B is resolved with { payload_hash_A, sig_A }
```

**Result:** The caller of `verify_foreign_transaction` for tx_id `[0x02; 32]` receives a response asserting that the transaction was confirmed in block `[0xAA; 32]` — which is the block hash from a completely different transaction.     #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
