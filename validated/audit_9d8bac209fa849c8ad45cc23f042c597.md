Audit Report

## Title
Zero-Initialized `rate` in `CrossChainRateReceiver` Causes Division-by-Zero Panic in All Deposit Paths Before First LayerZero Message — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.rate` is declared as `uint256 public rate` with no initializer, so it is `0` at deployment. Every pool variant's `viewSwapRsETHAmountAndFee` divides by this value without a zero-guard, causing an arithmetic panic (Solidity 0.8 checked arithmetic) on any deposit call until the first LayerZero rate message is delivered. All deposit transactions revert cleanly; no funds are lost.

## Finding Description

`CrossChainRateReceiver` declares `uint256 public rate` with no initial value, so it is `0` at construction. [1](#0-0) 

`RSETHRateReceiver`'s constructor sets `rateInfo`, `srcChainId`, `rateProvider`, and `layerZeroEndpoint`, but never sets `rate`. [2](#0-1) 

`getRate()` returns the raw stored value with no zero-check. [3](#0-2) 

`RSETHPoolV3.viewSwapRsETHAmountAndFee` (ETH overload) divides `amountAfterFee * 1e18` by `rsETHToETHrate` with no zero-guard: [4](#0-3) 

The token overload has the same unguarded division: [5](#0-4) 

The `limitDailyMint` modifier calls `viewSwapRsETHAmountAndFee` before the function body executes, meaning the panic fires before any `safeTransferFrom` for token deposits — the revert is fully atomic. [6](#0-5) 

A zero-check `if (rsETHToETHrate == 0) revert UnsupportedOracle()` exists **only** in `viewSwapAssetToPremintedRsETH`, not in the deposit path: [7](#0-6) 

The same unguarded division is present in `RSETHPoolV2.viewSwapRsETHAmountAndFee`: [8](#0-7) 

The same pattern applies across all pool variants: `RSETHPool`, `RSETHPoolV2`, `RSETHPoolV2NBA`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`.


## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

When `rate == 0`, every call to `deposit()` panics at the division and the entire transaction reverts. For ETH deposits, `msg.value` is returned to the caller by the revert. For token deposits, the `limitDailyMint` modifier triggers the panic before `safeTransferFrom` executes, so the token transfer is never initiated. No user funds are frozen or lost; deposits simply revert cleanly. The condition resolves automatically once the first `lzReceive` call sets `rate` to a non-zero value.

## Likelihood Explanation

The vulnerable window spans from pool deployment until the first successful LayerZero rate message. In normal operation this is short. However, if the LZ message is delayed — due to insufficient gas fee paid to the provider, LZ relayer lag, or misconfigured `srcChainId`/`rateProvider` — the pool is entirely non-functional for deposits during that period. No attacker action is required; any user calling `deposit` triggers the revert.

## Recommendation

Add a zero-rate guard in both overloads of `viewSwapRsETHAmountAndFee`, consistent with the guard already present in `viewSwapAssetToPremintedRsETH`:

```solidity
uint256 rsETHToETHrate = getRate();
if (rsETHToETHrate == 0) revert UnsupportedOracle();
```

Alternatively, initialize `rate` to a safe bootstrap value (e.g., `1e18`) in the `RSETHRateReceiver` constructor, to be overwritten by the first real LZ message.

## Proof of Concept

```solidity
// 1. Deploy RSETHRateReceiver — rate == 0
// 2. Deploy RSETHPoolV3 (proxy) with rsETHOracle = address(rateReceiver)
// 3. Call pool.deposit{value: 1 ether}("ref")
//    → limitDailyMint modifier calls viewSwapRsETHAmountAndFee(1 ether)
//    → getRate() returns 0
//    → amountAfterFee * 1e18 / 0 → arithmetic panic (Solidity 0.8 checked math)
//    → transaction reverts, 1 ether returned to caller
// 4. Simulate lzReceive setting rate = 1.05e18
// 5. Call pool.deposit{value: 1 ether}("ref") again → succeeds
```
