### Title
Treasury Donation Front-Running Invalidates `currentTreasuryValue` Exact-Equality Check, Blocking Governance Proposal Transactions - (File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs)

---

### Summary

The `validateTreasuryValue` function in the Conway/Dijkstra LEDGER rule enforces an **exact equality** check between the `currentTreasuryValue` field embedded in a transaction body and the live treasury balance (`casTreasuryL`). Because any unprivileged user can donate ADA to the treasury via `treasuryDonationTxBodyL`, an attacker can shift the treasury balance at every epoch boundary for the cost of 1 lovelace, permanently invalidating any pending governance-proposal transaction that carries a `currentTreasuryValue` assertion. This is the direct Cardano analog of the pre-swap output-balance front-running pattern described in the external report.

---

### Finding Description

**Root cause — exact equality check on a mutable ledger value**

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`, `validateTreasuryValue` is called unconditionally for every valid transaction in the LEDGER transition:

```haskell
runTest $ validateTreasuryValue txBody (chainAccountState ^. casTreasuryL)
``` [1](#0-0) 

The check itself is:

```haskell
validateTreasuryValue txBody actualTreasuryValue =
  case txBody ^. currentTreasuryValueTxBodyL of
    SNothing -> pure ()
    SJust submittedTreasuryValue ->
      failureUnless (submittedTreasuryValue == actualTreasuryValue) $
        ConwayTreasuryValueMismatch ...
``` [2](#0-1) 

The same check is re-used verbatim in the Dijkstra LEDGER rule and in the Dijkstra SUBLEDGER rule (for sub-transactions): [3](#0-2) [4](#0-3) 

**Why `currentTreasuryValue` must be present in real transactions**

The field is the sole mechanism by which Plutus V3 guardrail scripts can observe the treasury balance. The `txInfoCurrentTreasuryAmount` field in the Plutus script context is populated directly from `currentTreasuryValueTxBodyL`:

```haskell
PV3.txInfoCurrentTreasuryAmount =
    strictMaybe Nothing (Just . transCoinToLovelace) $
        txBody ^. currentTreasuryValueTxBodyL
``` [5](#0-4) 

The canonical guardrail script `ensureTreasuryReserve` (shipped in the repository) reads `txInfoCurrentTreasuryAmount` and fails if the field is absent:

```haskell
case PV3D.txInfoCurrentTreasuryAmount txInfo of
  Just treasury -> treasury P.- totalWithdrawal P.>= 100_000_000
  _ -> False
``` [6](#0-5) 

Any governance proposal transaction that must satisfy such a guardrail script is therefore **required** to include `currentTreasuryValue`, making it subject to the exact-equality check.

**The attacker-controlled mutation path**

Any transaction may include a `treasuryDonationTxBodyL` field. In the UTXO rule, `updateTreasuryDonation` accumulates the donation into `utxosDonationL`:

```haskell
updateTreasuryDonation tx utxos =
  case tx ^. isValidTxL of
    IsValid True -> utxos & utxosDonationL <>~ tx ^. bodyTxL . treasuryDonationTxBodyL
    IsValid False -> utxos
``` [7](#0-6) 

At every epoch boundary, the EPOCH rule moves the accumulated donation into the live treasury (`casTreasuryL`):

```haskell
chainAccountState3 =
  chainAccountState2
    & casTreasuryL <>~ (utxoState0 ^. utxosDonationL <> fold unclaimed)
``` [8](#0-7) 

Because `casTreasuryL` is the value passed to `validateTreasuryValue`, any donation submitted in epoch N causes the treasury to differ from the value a user observed and embedded in their transaction body, making that transaction fail in epoch N+1 with `ConwayTreasuryValueMismatch`.

**Attack flow**

1. User constructs a treasury-withdrawal governance proposal transaction with `currentTreasuryValue = T` (the treasury balance observed at construction time) and submits it to the mempool.
2. Attacker submits a transaction with `treasuryDonation = 1` lovelace in the same epoch. No special privilege is required.
3. At the epoch N → N+1 boundary, `casTreasuryL` becomes `T + 1`.
4. The user's transaction is validated in epoch N+1; `validateTreasuryValue` fires `ConwayTreasuryValueMismatch { mismatchSupplied = T, mismatchExpected = T+1 }` and the transaction is rejected.
5. The user reconstructs the transaction with the new treasury value and resubmits. The attacker donates again. The cycle repeats indefinitely at a cost of 1 lovelace per epoch to the attacker.

---

### Impact Explanation

An unprivileged attacker can persistently block any governance proposal transaction that is required to carry `currentTreasuryValue` — most critically, treasury-withdrawal proposals guarded by a script such as `ensureTreasuryReserve`. Because the attacker's cost per epoch is 1 lovelace and the victim must pay full transaction fees on every resubmission, the economic asymmetry makes indefinite blocking practical. This falls under the **Medium** allowed impact: *"Attacker-controlled transactions… modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters."* Treasury donations are being weaponised to invalidate the treasury-value assertion in a way that is outside the intended design of the field.

---

### Likelihood Explanation

The attack requires only:
- Monitoring the mempool for transactions containing `currentTreasuryValue` (trivially observable from the serialised transaction body).
- Submitting a single transaction per epoch with `treasuryDonation ≥ 1` lovelace.

No key compromise, governance majority, or Sybil attack is needed. The attacker is a standard transaction sender. The attack is therefore **medium likelihood**: it requires active mempool monitoring but is otherwise trivially executable by any ADA holder.

---

### Recommendation

Replace the exact-equality check with a **lower-bound check**: require only that `submittedTreasuryValue ≤ actualTreasuryValue`. This preserves the guardrail script's ability to assert a minimum treasury balance while making the check immune to additive manipulation (donations can only increase the treasury, so a lower-bound assertion remains valid after a donation). Alternatively, expose the intra-epoch donation accumulator (`utxosDonation`) to the script context so that scripts can reason about the post-donation treasury without requiring the transaction to commit to an exact pre-donation value.

---

### Proof of Concept

```
Epoch N:
  Treasury balance (casTreasury) = 1_000_000 ADA

  User constructs tx_proposal:
    currentTreasuryValue = 1_000_000 ADA   ← exact value required by guardrail script
    proposalProcedures   = [TreasuryWithdrawal ...]

  Attacker constructs tx_grief:
    treasuryDonation = 1 lovelace           ← no privilege required

  Both transactions enter the mempool.
  tx_grief is included in a block in epoch N.
  utxosDonation += 1 lovelace.

Epoch N → N+1 boundary (EPOCH rule):
  casTreasury = 1_000_000 ADA + 1 lovelace

Epoch N+1:
  tx_proposal is validated:
    validateTreasuryValue:
      submittedTreasuryValue = 1_000_000 ADA
      actualTreasuryValue    = 1_000_000 ADA + 1 lovelace
      → FAIL: ConwayTreasuryValueMismatch

  User must reconstruct and resubmit.
  Attacker donates 1 lovelace again. Cycle repeats.
```
