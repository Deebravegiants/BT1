### Title
Missing CKD Response Verification for `AppPublicKey` Variant Enables Single Byzantine Node to Forge Confidential Key Derivation Output — (`crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract contains an empty match arm for the `AppPublicKey` variant of CKD requests, performing **zero cryptographic verification** on the submitted `CKDResponse`. Unlike the `AppPublicKeyPV` variant (which has on-chain verification via `ckd_output_check`) and unlike `respond` (which verifies the threshold signature against the derived public key), the `AppPublicKey` path accepts any `big_y` / `big_c` pair unconditionally. A single Byzantine attested participant can therefore submit a forged CKD response, bypassing the threshold requirement and delivering an attacker-controlled derived key to the requesting user.

---

### Finding Description

In `respond_ckd` the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKeyPV` arm calls `ckd_output_check`, which cryptographically verifies that the submitted `(big_y, big_c)` pair is consistent with the domain's BLS12-381 public key and the request's `app_id`. The `AppPublicKey` arm is a no-op.

Contrast this with `respond`, which derives the expected public key from the stored root key and the request's tweak, then verifies the submitted ECDSA signature against it:

```rust
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
// ...
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response, payload_hash, &expected_public_key,
).is_ok()
``` [2](#0-1) 

Because producing a valid threshold ECDSA signature requires threshold-many honest participants, the `respond` path implicitly enforces the threshold requirement. The `respond_ckd` / `AppPublicKey` path has no equivalent enforcement.

After the (absent) check, the contract unconditionally resolves all queued yields with the submitted response bytes:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [3](#0-2) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active()`, which a single attested participant satisfies. [4](#0-3) 

---

### Impact Explanation

A single Byzantine attested MPC participant can:

1. Observe any pending `CKDRequest` whose `app_public_key` is the `AppPublicKey` variant.
2. Call `respond_ckd` with an arbitrary `CKDResponse` — any `big_y` and `big_c` of their choosing.
3. The contract accepts the response without verification and resumes all queued yield promises with the forged bytes.
4. The requesting user receives an attacker-controlled `(big_y, big_c)` pair and derives a wrong application key from it.

Because the attacker controls `big_y = g·y′` and `big_c = big_s′ + app_pk·y′` for arbitrary `big_s′`, the user's derived secret `big_s′` is entirely under attacker control. This constitutes **confidential key derivation output produced without the required threshold-participant authorization** — a Critical impact under the allowed scope.

---

### Likelihood Explanation

The attacker path requires only that the adversary is an attested MPC participant, which is a normal operational role. No threshold collusion, no key leakage, and no network-level attack is needed. The attack is a single on-chain transaction. Any one of the `n` participants who turns Byzantine can execute it against any `AppPublicKey` CKD request at any time the contract is in `Running` or `Resharing` state.

---

### Recommendation

Because the `AppPublicKey` variant intentionally hides the application public key from on-chain observers, a direct algebraic check analogous to `ckd_output_check` is not possible without the user's private key. Two mitigations are viable:

1. **Threshold voting**: Require at least `threshold` distinct attested participants to submit identical `(big_y, big_c)` values before resolving the yield. The contract already has the participant set and threshold stored in `protocol_state`.
2. **Commit-then-reveal**: Participants first commit a hash of their response; after a quorum of matching commits is reached, any participant reveals the preimage and the contract verifies the hash before resolving.

Either approach restores the threshold guarantee that the `AppPublicKeyPV` variant already enjoys via `ckd_output_check`.

---

### Proof of Concept

```
Given: contract in Running state, threshold = t, n attested participants.

1. Alice calls `request_app_private_key` with AppPublicKey(app_pk).
   → contract enqueues a yield for Alice's transaction.

2. Byzantine participant Bob calls:
     respond_ckd(
       request = <Alice's CKDRequest>,
       response = CKDResponse { big_y: [0u8;48], big_c: [0u8;48] }
     )
   → assert_caller_is_attested_participant_and_protocol_active() passes (Bob is attested).
   → AppPublicKey arm: no check, falls through.
   → resolve_yields_for resumes Alice's yield with the forged bytes.

3. Alice's transaction completes with CKDResponse { big_y: 0, big_c: 0 }.
   Alice computes big_s = big_c - app_sk * big_y = 0 - 0 = 0.
   Alice's derived application key is the identity element — entirely wrong.

4. Any cryptographic operation Alice performs with this derived key
   (signing, encryption, authentication) produces incorrect or attacker-predictable output.
``` [5](#0-4)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
