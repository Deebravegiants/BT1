### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Allows Any Unprivileged Actor to Obtain a Valid MPC Signature Over Any Foreign Transaction - (File: crates/contract/src/lib.rs)

### Summary

`verify_foreign_transaction` omits the caller's account ID from the `VerifyForeignTransactionRequest` map key. Because the pending-request fan-out map is keyed only on `(domain_id, request, payload_version)`, any unprivileged actor can submit an identical request and be queued alongside the original caller. When MPC nodes call `respond_verify_foreign_tx`, every queued yield is drained simultaneously, delivering the same `VerifyForeignTransactionResponse` — including the MPC-signed payload — to all callers, including the attacker.

### Finding Description

`sign()` explicitly binds each request to its caller by including `predecessor` in the `SignatureRequest` key:

```rust
// lib.rs:379-384
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // ← caller is part of the key
    &request.path,
);
```

`verify_foreign_transaction()` calls `check_request_preconditions` but **discards the returned `(domain_config, predecessor)` tuple entirely**:

```rust
// lib.rs:526-531
self.check_request_preconditions(
    request.domain_id,
    DomainPurpose::ForeignTx,
    Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
    MINIMUM_SIGN_REQUEST_DEPOSIT,
);
// predecessor is never captured
```

The request is then built without any caller binding:

```rust
// dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
        // no predecessor / caller field
    }
}
```

`VerifyForeignTransactionRequest` has no caller field:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:124-128

    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,

```

`resolve_yields_for` drains **all** queued yields for the key in one pass:

```rust
// pending_requests.rs:74-81

    .remove(request)
    .unwrap_or_default()

    .map(|YieldIndex { data_id }| {
        env::promise_yield_resume(&data_id, response_bytes.clone());
```


### Impact Explanation

`VerifyForeignTransactionResponse` contains a valid threshold signature produced by the MPC network's root key over the foreign transaction's `payload_hash`. Any contract or account that receives this response can present it as proof that the MPC network attested to the foreign transaction, and use it to authorize bridge operations downstream.

Because the request key is caller-agnostic, an attacker who observes a pending `verify_foreign_transaction` request on-chain (all NEAR state is public) can submit the identical request for 1 yoctoNEAR. When MPC nodes respond, the attacker's yield is resolved with the same signed response as the victim. The attacker now holds a valid MPC signature over the foreign transaction without the victim's consent and without performing the full signing cost. This signature can be replayed to:

- Execute bridge operations the MPC network did not intend to authorize for the attacker.
- Trigger double-spend conditions by presenting the same signed foreign-tx proof to multiple bridge contracts.

This matches the allowed impact: **High — Cross-chain replay, forged foreign-chain verification, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

The attack requires no special privilege. All NEAR contract state, including `pending_verify_foreign_tx_requests`, is publicly readable. An attacker only needs to:

1. Watch the chain for a `verify_foreign_transaction` call.
2. Submit the same `VerifyForeignTransactionRequestArgs` with 1 yoctoNEAR deposit.
3. Wait for the MPC nodes to respond.

No threshold collusion, TEE access, or key material is required. The attacker is a standard unprivileged NEAR account.

### Recommendation

Include the caller's `predecessor_account_id` in `VerifyForeignTransactionRequest`, mirroring the pattern used by `sign`:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    pub caller: AccountId,   // ← add this
}
```

Capture and forward the predecessor in `verify_foreign_transaction`:

```rust
let (_, predecessor) = self.check_request_preconditions(...);
let request = VerifyForeignTransactionRequest {
    domain_id: args.domain_id,
    request: args.request,
    payload_version: args.payload_version,
    caller: predecessor,
};
```

This ensures each caller's yield is keyed independently, preventing cross-caller fan-out and eliminating the ability for an attacker to piggyback on another contract's request.

### Proof of Concept

1. Victim contract `A` calls `verify_foreign_transaction` with `{domain_id: 0, request: Bitcoin{tx_id: [7u8;32], ...}, payload_version: V1}` and attaches 1 yoctoNEAR. Its yield is stored under key `K`.

2. Attacker account `B` calls `verify_foreign_transaction` with the identical arguments. Because `VerifyForeignTransactionRequest` has no caller field, `B`'s yield is appended to the same queue under key `K`. `pending_verify_foreign_tx_requests[K].len() == 2`.

3. MPC nodes call `respond_verify_foreign_tx(K, response)`. `resolve_yields_for` drains both yields, delivering the `VerifyForeignTransactionResponse` (containing the MPC root-key signature over `payload_hash`) to both `A` and `B`.

4. `B` now holds a valid MPC signature over the Bitcoin transaction. `B` presents this signature to a bridge contract to execute a bridge operation, or replays it to a second bridge contract to double-spend.

The unit test at `lib.rs:3208–3298` (`verify_foreign_transaction__should_queue_duplicates_from_different_callers`) already demonstrates steps 1–3 as passing behavior, confirming the root cause is present in production code. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

```rust
    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
```

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
