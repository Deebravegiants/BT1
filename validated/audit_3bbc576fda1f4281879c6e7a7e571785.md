### Title
Zero `confirmations` Threshold Bypasses Bitcoin Finality Verification in `verify_foreign_transaction` — (`crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

An unprivileged caller can submit a `verify_foreign_transaction` request with `confirmations: 0` for a Bitcoin transaction. Neither the on-chain contract nor the node-side inspector enforces a minimum confirmation threshold. The finality check `block_confirmations_threshold <= transaction_block_confirmation` trivially passes when the threshold is zero, causing MPC nodes to sign a payload for an unconfirmed (or zero-confirmation) Bitcoin transaction. In a bridge context this enables a double-spend: the attacker obtains a valid MPC signature before the Bitcoin transaction finalizes, then the transaction is reorganized away.

---

### Finding Description

The `verify_foreign_transaction` contract method accepts a `BitcoinRpcRequest` whose `confirmations` field is a plain `u64` newtype (`BlockConfirmations`) with no minimum value enforced anywhere in the call path.

**Contract entry point** — no lower-bound check on `confirmations`: [1](#0-0) 

The request is forwarded verbatim to the MPC nodes. The node-side inspector performs the only finality check: [2](#0-1) 

When `block_confirmations_threshold` is `0`, the expression `0 <= transaction_block_confirmation` is always `true` for any `u64` value returned by the RPC, so the guard is a no-op. The inspector then proceeds to verify canonical-chain membership and returns extracted values, after which the MPC network signs the payload.

The `BlockConfirmations` type is a transparent `u64` wrapper with no invariant: [3](#0-2) 

The existing tests only exercise thresholds ≥ 1 and never assert that `confirmations: 0` is rejected: [4](#0-3) 

---

### Impact Explanation

This maps to the allowed High impact: **forged foreign-chain verification / light-client-style verification bypass that causes invalid bridge execution or double-spend conditions.**

In the Omnibridge inbound flow (Bitcoin → NEAR), a relayer submits `verify_foreign_transaction` to prove a Bitcoin deposit finalized. With `confirmations: 0`, the MPC network signs the payload for a transaction that has zero block confirmations — i.e., it may still be in the mempool or in a block that has not yet been buried. If the Bitcoin chain reorganizes and evicts that transaction, the attacker holds a valid MPC-signed attestation for a deposit that never finalized, and can use it to claim bridge funds on NEAR. This is a direct double-spend against the bridge's locked funds.

---

### Likelihood Explanation

Any unprivileged NEAR account can call `verify_foreign_transaction` with a deposit of 1 yoctoNEAR. The `confirmations` field is a free integer parameter with no on-chain validation. No special role, key, or threshold collusion is required. The attack is trivially reproducible by any caller who can construct a valid `BitcoinRpcRequest` JSON.

---

### Recommendation

1. **Enforce a minimum at the contract level**: In `verify_foreign_transaction`, reject any `BitcoinRpcRequest` whose `confirmations` field is zero (or below a protocol-defined minimum, e.g., 1 for testnet, 6 for mainnet). This mirrors the fix recommended in the original report — assert the critical safety parameter is non-zero before enqueuing the request.

2. **Enforce a minimum in `BlockConfirmations`**: Add a constructor that panics or returns `Err` for `confirmations == 0`, making the invariant impossible to violate at the type level.

3. **Add a test**: Assert that `verify_foreign_transaction` with `confirmations: 0` is rejected at the contract level, analogous to the existing `InsufficientDeposit` test.

---

### Proof of Concept

```json
{
  "request": {
    "request": {
      "Bitcoin": {
        "tx_id": "<any 32-byte tx hash>",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    },
    "domain_id": 3,
    "payload_version": 1
  }
}
```

Call `verify_foreign_transaction` with the above JSON and `--deposit-yocto 1`. The contract enqueues the request. MPC nodes call `BitcoinInspector::extract` with `block_confirmations_threshold = BlockConfirmations(0)`. The check at line 51–52 of `inspector.rs` evaluates `0 <= N` (always true), so the inspector proceeds to sign the payload regardless of the actual confirmation count of the transaction — including 0 confirmations (mempool). The attacker receives a valid MPC signature for an unconfirmed Bitcoin transaction, enabling a double-spend if the transaction is later reorganized.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
