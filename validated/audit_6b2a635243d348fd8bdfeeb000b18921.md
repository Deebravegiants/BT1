### Summary

The Conway era `RATIFY` rule processes multiple `TreasuryWithdrawals` governance actions sequentially within a single epoch boundary. The guardrails script â€” the constitution's hard constraint on treasury withdrawals â€” is evaluated only at **proposal submission time** (in the `GOV` rule), not at **ratification time**. Because the script receives only the treasury balance at the moment of submission, multiple proposals can each individually satisfy the guardrails constraint while their combined enactment in the same epoch violates it. This is the direct Cardano analog of the Arrakis M-2 finding: each individual operation is within the permitted bound, but the absence of a cumulative rate limit allows the bound to be exceeded in aggregate.

### Finding Description

**Guardrails script is checked at submission time, not ratification time.**

When a `TreasuryWithdrawals` proposal is submitted, the `GOV` rule verifies that the proposal's guardrails script hash matches the constitution's hash: [1](#0-0) 

The actual Plutus guardrails script is then executed as part of transaction validation. The script receives `txInfoCurrentTreasuryAmount` â€” the treasury balance **at the time of the submitting transaction**. A canonical example of such a script is `ensureTreasuryReserve`, which enforces `treasury - withdrawal >= reserve`: [2](#0-1) 

**The `RATIFY` rule processes multiple `TreasuryWithdrawals` proposals sequentially, updating `ensTreasury` after each enactment.**

  [d|
    ensureTreasuryReserve :: P.BuiltinData -> P.BuiltinUnit
    ensureTreasuryReserve context =
      P.check $
        case unsafeFromBuiltinData context of
          PV3D.ScriptContext
            txInfo
            _
            (PV3D.ProposingScript _ (PV3D.ProposalProcedure _ _ (PV3D.TreasuryWithdrawals withdrawals _))) ->
