### Title
`revert_state_diff` Unconditionally Deletes Sierra/CASM Data for Classes Declared in Earlier Blocks When a Migration Block Is Reverted — (`File: crates/apollo_storage/src/state/mod.rs`)

### Summary

`revert_state_diff` contains two helper functions — `delete_declared_classes` and `delete_compiled_classes` — that delete Sierra class and CASM data for every class hash present in the reverted block's `class_hash_to_compiled_class_hash` map, without checking whether the class was actually *first declared* in that block. Because the CASM-hash migration feature causes a later block's state diff to include entries for classes declared in earlier blocks, reverting such a migration block permanently destroys the Sierra and CASM storage entries for those earlier-declared classes, leaving the state corrupted.

### Finding Description

**`append_state_diff` correctly guards the declaration-block table:** [1](#0-0) 

When a class hash already exists in `declared_classes_block_table`, the insert is skipped. So for a migration entry (a class declared in block N that appears again in block M's state diff), `declared_classes_block_table[C]` remains `N`.

**`delete_declared_classes_block` correctly honours this guard during revert:** [2](#0-1) 

It only deletes the block-number record when `class_block_entry == block_number`, so it correctly skips class `C` when reverting block M.

**`delete_declared_classes` has no such guard:** [3](#0-2) 

It iterates over every key in `class_hash_to_compiled_class_hash` and deletes the Sierra class from `declared_classes_table` unconditionally. For a migration entry, the Sierra class stored in block N is still present in `declared_classes_table` and is deleted.

**`delete_compiled_classes` has the same missing guard:** [4](#0-3) 

It deletes the CASM entry for every class hash in the iterator, again without checking whether the class was first declared in the block being reverted.

**Both are called unconditionally from `revert_state_diff`:** [5](#0-4) 

**Migration entries are a confirmed production scenario.** The `native_blockifier` storage layer explicitly identifies and separates migration entries — class hashes present in `class_hash_to_compiled_class_hash` but *not* in the newly declared classes: [6](#0-5) 

The blockifier test confirms that a block's state diff can contain migration entries for classes declared in earlier blocks: [7](#0-6) 

And `enable_casm_hash_migration` is present in every versioned-constants file from v0.13.0 through v0.14.4, confirming it is a live production flag.

### Impact Explanation

**Scenario:**

1. Block N: Class `C` is declared. `declared_classes_table[C]` = Sierra class; `compiled_classes_table[C]` = CASM V1; `declared_classes_block_table[C]` = N.
2. Block M (M > N): Migration block. State diff contains `{C: CASM_V2}`. `compiled_classes_table[C]` is updated to CASM V2. `declared_classes_block_table[C]` remains N (correctly not overwritten).
3. Block M is reverted (normal consensus rejection path via `revert_block` → `revert_state_diff`).
4. `delete_declared_classes_block`: Sees `declared_classes_block_table[C] = N ≠ M`, correctly skips. ✓
5. `delete_declared_classes`: Finds `declared_classes_table[C]` (the Sierra class from block N), **deletes it**. ✗
6. `delete_compiled_classes`: Finds `compiled_classes_table[C]` (CASM V2 from block M), **deletes it**. ✗

**Post-revert state is corrupted:** `declared_classes_block_table[C] = N` (says class exists) but `declared_classes_table[C]` and `compiled_classes_table[C]` are both gone. Any subsequent transaction invoking a contract deployed with class `C` will fail with a class-not-found error from the blockifier, even though class `C` was validly declared in block N which was never reverted.

This is a **Critical** wrong-state result: "Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."

### Likelihood Explanation

Block reverts are a normal part of consensus operation (a proposed block that fails validation is reverted). The CASM migration feature is enabled in all production versioned constants. Any node that processes a migration block and then reverts it — a sequence that can occur without any privileged access — will silently corrupt its class storage. The corruption is permanent until the node is re-synced.

### Recommendation

Pass `declared_classes_block_table` and `block_number` into both `delete_declared_classes` and `delete_compiled_classes`, and add the same guard that `delete_declared_classes_block` already uses:

```rust
// Only delete if the class was first declared in this block.
if declared_classes_block_table.get(txn, class_hash)? == Some(block_number) {
    declared_classes_table.delete(txn, class_hash)?;
    // ... same for compiled_classes_table
}
```

This mirrors the existing correct pattern in `delete_deprecated_declared_classes`: [8](#0-7) 

### Proof of Concept

```
1. Declare class C in block 0:
   append_state_diff(0, { class_hash_to_compiled_class_hash: {C: CASM_V1} })
   append_classes(0, [(C, sierra_class)], [])
   append_casm(C, casm_v1)
   → declared_classes_block_table[C] = 0
   → declared_classes_table[C] = sierra_class
   → compiled_classes_table[C] = casm_v1

2. Migration block 1 (enable_casm_hash_migration=true, C executed):
   append_state_diff(1, { class_hash_to_compiled_class_hash: {C: CASM_V2} })
   append_casm(C, casm_v2)
   → declared_classes_block_table[C] = 0  (unchanged, not_declared=false)
   → compiled_classes_table[C] = casm_v2

3. Revert block 1:
   revert_state_diff(1)
   → delete_declared_classes_block: class_block_entry=0 ≠ 1, SKIP ✓
   → delete_declared_classes: declared_classes_table[C] found, DELETE ✗
   → delete_compiled_classes: compiled_classes_table[C] found, DELETE ✗

4. State after revert:
   declared_classes_block_table[C] = 0  (says class exists)
   declared_classes_table[C] = MISSING  (Sierra class gone)
   compiled_classes_table[C] = MISSING  (CASM gone)

5. Execute invoke tx calling contract deployed with class C:
   blockifier calls get_compiled_class(C) → None → execution error
   → Transaction that should succeed now fails with wrong revert result
```

fn delete_declared_classes_block<'env>(
    txn: &'env DbTransaction<'env, RW>,
    thin_state_diff: &ThinStateDiff,
    declared_classes_block_table: &'env DeclaredClassesBlockTable<'env>,
    block_number: BlockNumber,
) -> StorageResult<Vec<ClassHash>> {
    let mut deleted_data = Vec::new();
    for class_hash in thin_state_diff.class_hash_to_compiled_class_hash.keys() {
        let class_block_entry =
            declared_classes_block_table.get(txn, class_hash)?.ok_or_else(|| {
                StorageError::DBInconsistency {
                    msg: format!(
                        "Attempting to revert declaration of class {class_hash} but it doesn't \
                         exist in the DB"
                    ),
                }
            })?;
        if class_block_entry == block_number {
            declared_classes_block_table.delete(txn, class_hash)?;
            deleted_data.push(*class_hash);
        }
    }
    Ok(deleted_data)

fn delete_compiled_classes<'a, 'env>(
    txn: &'env DbTransaction<'env, RW>,
    class_hashes: impl Iterator<Item = &'a ClassHash>,
    compiled_classes_table: &'env CompiledClassesTable<'env>,
    file_handlers: &FileHandlers<RW>,
) -> StorageResult<IndexMap<ClassHash, CasmContractClass>> {
    let mut deleted_data = IndexMap::new();
    for class_hash in class_hashes {
        let Some(compiled_class_location) = compiled_classes_table.get(txn, class_hash)?
        // No compiled class means the rest of the compiled classes weren't downloaded yet.
        else {
            break;
        };
        compiled_classes_table.delete(txn, class_hash)?;
        deleted_data
            .insert(*class_hash, file_handlers.get_casm_unchecked(compiled_class_location)?);
    }

    Ok(deleted_data)
}
