### Title
Unprotected `init()` Allows Front-Running to Hijack MPC Participant Set and Signing Capability - (File: `crates/contract/src/lib.rs`)

### Summary
The `MpcContract::init()` function carries no caller restriction. All production deployment procedures deploy the contract bytecode and call `init()` in **separate transactions**, creating a window in which any unprivileged account can race the legitimate operator, call `init()` first with an attacker-controlled `ThresholdParameters`, and permanently seize control of the MPC network's participant set and signing capability.

### Finding Description
`MpcContract::init()` is decorated only with NEAR SDK's `#[init]` attribute, which prevents *re-initialization* but places **no restriction on who may call it**: [1](#0-0) 

The function accepts arbitrary `ThresholdParameters` — participant set, threshold value, and configuration — and immediately transitions the contract into the `Running` state with those parameters: [2](#0-1) 

Critically, `init()` calls `TeeState::with_mocked_participant_attestations(initial_participants)`, meaning the initial participants receive **mocked (always-valid) TEE attestations** regardless of whether they are real TEE nodes: [3](#0-2) 

Every documented production deployment procedure separates the `deploy` and `init` steps into distinct transactions:

- Localnet guide: `without-init-call` then a separate `init` call [4](#0-3) 

- Testnet TEE guide: same two-step pattern [5](#0-4) 

- TEE cluster deployment script: `deploy_contract()` and `init_contract()` are separate sequential phases [6](#0-5) 

- `launch-localnet.sh`: same pattern [7](#0-6) 

- The `mpc-devnet` CLI exposes `DeployContract` and `InitContract` as **separate subcommands**, making the gap explicit [8](#0-7) 

The second initialization function, `init_running`, is correctly protected with `#[private]` (callable only by the contract itself via a cross-contract call): [9](#0-8) 

`init()` has no equivalent protection.

### Impact Explanation
An attacker who calls `init()` before the legitimate operator:

1. Supplies `ThresholdParameters` with themselves as the sole participant and `threshold = 1`.
2. The contract transitions to `Running` with the attacker's participant set; the legitimate `init()` call subsequently panics because the contract is already initialized.
3. Because `with_mocked_participant_attestations` grants the initial participants unconditional TEE validity, the attacker's node passes all TEE checks without a real enclave.
4. The attacker runs their own MPC node, which sees itself as the only participant. DKG with `n=1, t=1` completes trivially, generating a key share the attacker fully controls.
5. The attacker's node can now respond to any `sign()` request and produce valid threshold signatures — **unauthorized threshold signature issuance** — while the legitimate MPC network is permanently locked out.

This satisfies the **Critical** impact class: *Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization*, and *permanent freezing of funds controlled by the MPC network* (the legitimate network can never be started).

### Likelihood Explanation
The attack requires only:
- Monitoring the NEAR chain for a `deploy` transaction targeting the MPC contract account (trivially observable on-chain).
- Submitting a single `init()` call before the operator's `init` transaction is included.

No privileged access, TEE hardware, key material, or collusion is required. The window exists on every fresh deployment and every contract upgrade that redeploys to a new account. The devnet CLI's explicit separation of `DeployContract` and `InitContract` confirms this is the standard operational procedure.

### Recommendation
1. **Enforce caller restriction on `init()`**: Check that `env::predecessor_account_id() == env::current_account_id()` (i.e., the contract account itself must be the caller), mirroring the `#[private]` guard on `init_running`.
2. **Atomic deploy-and-init**: Use NEAR's `with-init-call` flag in all deployment scripts so the bytecode upload and `init()` are included in the same transaction, eliminating the race window entirely.
3. **Audit all deployment scripts** (`scripts/launch-localnet.sh`, `localnet/tee/scripts/rust-launcher/deploy-tee-cluster.sh`, testnet guides) to replace the two-step pattern with atomic deployment.

### Proof of Concept

**Step 1 — Operator deploys bytecode (observable on-chain):**
```bash
near contract deploy $MPC_CONTRACT_ACCOUNT use-file mpc_contract.wasm \
  without-init-call network-config testnet sign-with-keychain send
```

**Step 2 — Attacker races to call `init()` before the operator:**
```bash
near contract call-function as-transaction $MPC_CONTRACT_ACCOUNT init \
  json-args '{
    "parameters": {
      "threshold": 1,
      "participants": {
        "next_id": 1,
        "participants": [["attacker.testnet", 0, {"tls_public_key": "ed25519:ATTACKER_P2P_KEY", "url": "https://attacker.example.com"}]]
      }
    }
  }' \
  prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
  sign-as attacker.testnet network-config testnet sign-with-keychain send
```

**Step 3 — Operator's `init()` fails** with "contract already initialized."

**Step 4 — Attacker runs their own MPC node** registered as `attacker.testnet`. Because `with_mocked_participant_attestations` grants unconditional TEE validity to initial participants, the node passes all attestation checks. With `n=1, t=1`, DKG completes and the attacker holds the sole key share.

**Step 5 — Attacker's node responds to `sign()` requests**, issuing threshold signatures for arbitrary payloads — full unauthorized signing capability over the MPC network. [10](#0-9)

add_node_keys_from_file() {
  local keys_file="$1"
  log "Adding node keys to NEAR accounts using $keys_file"
  [ -f "$keys_file" ] || { err "Missing keys file at $keys_file. Run collect phase first."; exit 1; }

  jq -c '.[]' "$keys_file" | while read -r row; do
    local acct signer responder
    acct="$(echo "$row" | jq -r .account)"
    signer="$(echo "$row" | jq -r .signer_pk)"
    responder="$(echo "$row" | jq -r .responder_pk)"

    log "$acct: add signer key"
    near_add_key_skip_if_exists "$acct" "$signer" "signer"

    log "$acct: add responder key"
    near_add_key_skip_if_exists "$acct" "$responder" "responder"
  done
}
