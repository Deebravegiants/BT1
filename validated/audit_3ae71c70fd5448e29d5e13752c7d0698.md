### Title
Reference Script Fee Omits Sub-Transaction Inputs in Dijkstra Era Minimum Fee Calculation — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces sub-transactions (`subTransactionsTxBodyL`) embedded inside a top-level transaction body. The minimum fee calculation for a Dijkstra batch transaction uses `getConwayMinFeeTxUtxo`, which calls `txNonDistinctRefScriptsSize` — a function that only inspects the **top-level** transaction's inputs for reference scripts. Sub-transactions' inputs are never examined. A dedicated function `batchNonDistinctRefScriptsSize` exists in the same file and correctly aggregates reference script sizes across the top-level transaction and all sub-transactions, but it is **never called** during fee validation. This is a direct accounting mismatch: the approved/validated fee covers only part of the actual computational cost, analogous to the EverclearBridge bug where only `amount` was approved but `amount + fee` was required.

---

### Finding Description

**Root cause file:** `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`

The `EraUTxO DijkstraEra` instance sets:

```haskell
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` is defined as:

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
``` [2](#0-1) 

`txNonDistinctRefScriptsSize` only looks at the top-level transaction body's inputs:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

Sub-transactions are stored in `tx ^. bodyTxL . subTransactionsTxBodyL` and are **never visited** by this function.

In contrast, `batchNonDistinctRefScriptsSize` — defined in the same Dijkstra UTxO module — correctly aggregates across the whole batch:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [4](#0-3) 

This function is exported but **never used** in the fee validation path.

The fee validation in `dijkstraUtxoTransition` calls:

```haskell
{- minfee pp txTop utxo₀ ≤ txfee txb -}
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
``` [5](#0-4) 

`validateFeeTooSmallUTxO` dispatches to `getMinFeeTxUtxo pp tx utxo`, which for `DijkstraEra` resolves to `getConwayMinFeeTxUtxo` — the function that ignores sub-transaction reference scripts.

The reference script fee was introduced in Conway specifically to deter DDoS attacks via large, cheap-to-include but expensive-to-deserialize scripts (see ADR-009): [6](#0-5) 

The tiered pricing formula is applied in `getConwayMinFeeTx`:

```haskell
getConwayMinFeeTx pp tx refScriptsSize =
  alonzoMinFeeTx pp tx <+> refScriptsFee
  where
    refScriptCostPerByte = unboundRational (pp ^. ppMinFeeRefScriptCostPerByteL)
    refScriptsFee = tierRefScriptFee ...
``` [7](#0-6) 

When `refScriptsSize` is computed only from the top-level transaction, the tiered fee for sub-transaction reference scripts is zero — regardless of how large those scripts are.

---

### Impact Explanation

**Allowed impact matched:** *Medium — Attacker-controlled transactions modify fees outside design parameters.*

An unprivileged transaction author can craft a Dijkstra top-level transaction that:
- Carries minimal or zero reference scripts at the top level (low `txNonDistinctRefScriptsSize`)
- Embeds sub-transactions that reference large Plutus scripts via `referenceInputsTxBodyL`

The ledger accepts the batch with a fee computed only from the top-level reference scripts. The sub-transaction reference script deserialization cost — which the tiered fee formula was designed to price — is entirely unaccounted for. This allows systematic underpayment of fees for computationally expensive batches, undermining the economic deterrent against the reference-script DDoS vector that was patched in Conway.

---

### Likelihood Explanation

Any transaction author can exploit this without any privileged access. The Dijkstra era is the active development target in this repository. The mismatch is structural: `batchNonDistinctRefScriptsSize` exists and is exported precisely because sub-transaction reference scripts must be counted, yet the fee validation path never calls it. The entry point is a standard transaction submission.

---

### Recommendation

Replace the `getMinFeeTxUtxo` instance for `DijkstraEra` with an implementation that uses `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
-- In eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
getDijkstraMinFeeTxUtxo :: (EraTx era, DijkstraEraTxBody era) => PParams era -> Tx l era -> UTxO era -> Coin
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx

instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo
```

This ensures the tiered reference script fee covers all reference scripts in the batch, consistent with the design intent documented in ADR-009.

---

### Proof of Concept

1. Construct a Dijkstra `TopTx` with empty `referenceInputsTxBodyL` and `inputsTxBodyL` at the top level (so `txNonDistinctRefScriptsSize = 0`).
2. Embed a `SubTx` whose `referenceInputsTxBodyL` points to UTxO entries containing large Plutus scripts (e.g., 100 KiB each, up to the per-tx limit).
3. Set `feeTxBodyL` to the minimum fee computed by `getConwayMinFeeTxUtxo` (which returns only the base linear fee, with zero reference script surcharge).
4. Submit the transaction. `validateFeeTooSmallUTxO` calls `getMinFeeTxUtxo = getConwayMinFeeTxUtxo`, computes `minFee` without any sub-transaction reference script cost, and the check passes.
5. The ledger accepts a batch whose actual deserialization cost is equivalent to a transaction with 100+ KiB of reference scripts, but the fee paid corresponds to a transaction with 0 KiB of reference scripts — a fee undercharge of potentially thousands of ADA at current `minFeeRefScriptCostPerByte` parameters. ## Context

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. Taking this fact into consideration, a decision was made to set this parameter to a fairly moderate value to at l ... (truncated)

Unfortunately things did not go exactly as planned, because on [June 25th 2024 an attack like this actually took place](https://cardanospot.io/news/ddos-attack-on-the-cardano-blockchain-mRIKAzZTNnzq5NGd). This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
