### Title
Unvalidated `key_event_timeout_blocks` in `InitConfig` Enables Permanent Key-Generation Freeze via Front-Run of Unguarded `init` - (File: crates/contract/src/lib.rs)

### Summary

The `init` function of `MpcContract` accepts an `InitConfig` whose `key_event_timeout_blocks` field is stored without any minimum-value check. Because `init` carries no `#[private]` guard, any unprivileged account can call it before the deployer does. Setting `key_event_timeout_blocks = 0` causes every `KeyEventInstance` to expire within the same block it is created, making it impossible for nodes to ever vote a key-generation or resharing attempt to completion. Without keys, no threshold signatures can be issued — permanently freezing the MPC network's signing capability.

### Finding Description

The `init` function is decorated with `#[init]` but **not** `#[private]`: [1](#0-0) 

It accepts an optional `InitConfig` whose fields are applied verbatim to the internal `Config` with no range validation: [2](#0-1) [3](#0-2) 

The `key_event_timeout_blocks` value flows directly into `KeyEventInstance::new`: [4](#0-3) 

With `timeout_blocks = 0`, `expires_on = block_height + 1`. The `active()` predicate is: [5](#0-4) 

A NEAR transaction executes in a single block. Any subsequent node vote arrives in a later block, at which point `block_height >= expires_on`, so `active()` returns `false` and `cleanup_if_timed_out()` discards the instance. The leader retries indefinitely but every new instance expires before any vote can land.

The same `key_event_timeout_blocks` is used for both keygen and resharing: [6](#0-5) 

The `Config` struct itself has no validation layer: [7](#0-6) 

A secondary impact exists via `return_signature_and_clean_state_on_success_call_tera_gas = 0`: the yield-callback for every `sign` call would receive zero gas, causing OOG and permanently locking all pending signature requests: [8](#0-7) 

### Impact Explanation

Setting `key_event_timeout_blocks = 0` at initialization permanently prevents Distributed Key Generation and resharing from completing. Without a generated key, the contract can never reach a state where `sign`, `request_app_private_key`, or `request_verify_foreign_tx` can produce a valid threshold signature. All funds and cross-chain flows that depend on the MPC network's signing capability are permanently frozen. This matches the allowed impact: **permanent freezing of funds controlled by the MPC network / chain-signature contract**.

### Likelihood Explanation

NEAR allows contract deployment and `init` to be issued as separate transactions (as seen in the localnet script and sandbox test helpers): [9](#0-8) 

Any account that observes the deployment transaction in the mempool can submit a competing `init` call with a higher gas price and `key_event_timeout_blocks: 0` before the deployer's own `init` lands. Because `#[init]` only enforces single-call semantics (panics if already initialized), the attacker's call succeeds and the deployer's call is permanently rejected. The attack requires no special privilege, no key material, and no threshold collusion — only mempool observation and a single transaction.

### Recommendation

1. **Add minimum-value guards in `init`** before applying `InitConfig`:
   ```rust
   if let Some(ref cfg) = init_config {
       if cfg.key_event_timeout_blocks == Some(0) {
           return Err(Error::InvalidParameters(...));
       }
       // similarly for return_signature_and_clean_state_on_success_call_tera_gas, etc.
   }
   ```
2. **Add `#[private]` to `init`** (or enforce that only the contract account can call it), mirroring the guard already present on `init_running`: [10](#0-9) 
3. **Validate `Config` in `update_config`** with the same bounds, so a governance-voted config update cannot introduce the same inconsistent state.

### Proof of Concept

```
# 1. Deployer deploys the contract (tx A)
near deploy mpc-contract.near mpc_contract.wasm

# 2. Attacker front-runs the deployer's init call (tx B, higher gas price)
near call mpc-contract.near init \
  '{"parameters": <valid_params>, "init_config": {"key_event_timeout_blocks": 0}}' \
  --gas 300000000000000

# 3. Deployer's init call (tx C) panics: "Contract is already initialized"

# 4. Any subsequent start_keygen_instance call creates a KeyEventInstance
#    with expires_on = block_height + 1.
#    All node votes arrive in later blocks → instance always expired → 
#    key generation never completes → no signatures ever issued.
``` [11](#0-10) [12](#0-11)

    /// Start a new key event instance as the leader, if one isn't already active.
    /// The leader is always the participant with the lowest participant ID.
    pub fn start(&mut self, key_event_id: KeyEventId, timeout_blocks: u64) -> Result<(), Error> {
        self.cleanup_if_timed_out();
        if self.instance.is_some() {
            return Err(KeyEventError::ActiveKeyEvent.into());
        }
        let expected_key_event_id =
            KeyEventId::new(self.epoch_id, self.domain.id, self.next_attempt_id);
        if key_event_id != expected_key_event_id {
            return Err(KeyEventError::KeyEventIdMismatch.into());
        }
        self.verify_leader()?;
        self.instance = Some(KeyEventInstance::new(self.next_attempt_id, timeout_blocks));
        self.next_attempt_id = self.next_attempt_id.next();
        Ok(())
    }
