Audit Report

## Title
Stale `rsETHPrice` After Early Return in `_updateRsETHPrice()` Enables Over-Commitment and Over-Payout on Withdrawal - (File: contracts/LRTOracle.sol)

## Summary
In `LRTOracle._updateRsETHPrice()`, when a price drop exceeds `pricePercentageLimit`, the function pauses the protocol and returns early at line 281, permanently skipping the `rsETHPrice = newRsETHPrice` assignment at line 313. After the admin unpauses the contracts, the stale (pre-drop, inflated) `rsETHPrice` is immediately live. An unprivileged user can call `initiateWithdrawal()` using the stale price to over-commit `assetsCommitted`, and if `unlockQueue()` is called before `updateRSETHPrice()`, withdrawers receive more underlying assets than their rsETH entitles them to.

## Finding Description
`_updateRsETHPrice()` (contracts/LRTOracle.sol, lines 277–281) contains:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;   // ← exits before rsETHPrice = newRsETHPrice at line 313
}
```

The `return` at line 281 exits before `rsETHPrice = newRsETHPrice` at line 313, leaving the stored price at the pre-slashing value for the entire pause duration and beyond.

After the admin calls `lrtOracle.unpause()` and `withdrawalManager.unpause()` (independent transactions with no on-chain ordering constraint relative to `updateRSETHPrice()`), the stale price is immediately consumable:

1. **`initiateWithdrawal()`** (public, `whenNotPaused`) calls `getExpectedAssetAmount()` → `amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)` (line 593), producing an inflated `expectedAssetAmount`, which is added to `assetsCommitted[asset]` (line 173).

2. **`unlockQueue()`** calls `_createUnlockParams()` which reads `lrtOracle.rsETHPrice()` (line 847). `_calculatePayoutAmount()` computes `currentReturn = rsETHUnstaked * rsETHPrice / assetPrice` (line 833). Since both `expectedAssetAmount` and `currentReturn` are derived from the same stale price, `min(expectedAssetAmount, currentReturn) = expectedAssetAmount` — the full inflated amount is paid out.

The `_validatePrices` guard in `unlockQueue()` does not prevent this: the operator naturally queries `lrtOracle.rsETHPrice()` to set their price bounds, which returns the stale value, so the check passes with the stale price.

## Impact Explanation
**Scenario A — Temporary freezing of funds (Medium):** An unprivileged user calls `initiateWithdrawal()` after unpause with the stale price. `assetsCommitted[asset]` is over-inflated by `(stalePriceRatio - truePriceRatio) * rsETHUnstaked / assetPrice`, causing `getAvailableAssetAmount()` to return a deflated figure and blocking other users from initiating withdrawals until `unlockQueue()` processes the request and decrements `assetsCommitted`.

**Scenario B — Theft of unclaimed yield (High):** If `unlockQueue()` is called before `updateRSETHPrice()`, `_calculatePayoutAmount()` uses the stale price for `currentReturn`. Since `expectedAssetAmount == currentReturn` (both from the stale price), the payout equals the inflated amount. The attacker receives excess underlying assets extracted from the pool at the expense of remaining rsETH holders.

## Likelihood Explanation
The trigger condition — a price drop exceeding `pricePercentageLimit` — corresponds to a real-world EigenLayer slashing event: plausible but infrequent. The exploitation window opens whenever the admin unpauses the withdrawal manager before calling `updateRSETHPrice()`. There is no on-chain enforcement of ordering between `unpause()` and `updateRSETHPrice()`. An operator calling `unlockQueue()` as a routine post-unpause step would query `lrtOracle.rsETHPrice()` for their price bounds, receiving the stale value and inadvertently enabling the over-payout. A sophisticated user monitoring the mempool for unpause transactions can front-run the `updateRSETHPrice()` call. Likelihood: Low-Medium.

## Recommendation
Before the `return` statement in the `isPriceDecreaseOffLimit` branch, update `rsETHPrice` to the newly computed value:

```solidity
if (isPriceDecreaseOffLimit) {
    rsETHPrice = newRsETHPrice;   // ← add this line
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

This ensures the stored price always reflects the latest known state, even when the protocol is paused due to a large price drop. Alternatively, enforce at the contract level that `updateRSETHPrice()` must succeed before the withdrawal manager can be unpaused, or add a staleness check in `initiateWithdrawal()` and `unlockQueue()` that reverts if `rsETHPrice` has not been refreshed since the last pause event.

## Proof of Concept
1. Protocol operating normally: `rsETHPrice = 1.05 ether`, `highestRsethPrice = 1.05 ether`, `pricePercentageLimit = 0.05e18` (5%).
2. Slashing event reduces true ETH backing per rsETH to `0.98 ether` (6.7% drop, exceeding 5% limit).
3. Anyone calls `updateRSETHPrice()`:
   - `newRsETHPrice = 0.98 ether`, `isPriceDecreaseOffLimit = true`
   - Deposit pool, withdrawal manager, and oracle are paused.
   - **Early return fires — `rsETHPrice` remains `1.05 ether` in storage.**
4. Admin unpauses all three contracts (three separate transactions, no ordering constraint with `updateRSETHPrice()`).
5. Attacker calls `initiateWithdrawal(ETH, X_rsETH)`:
   - `getExpectedAssetAmount` = `X * 1.05 / 1.0` = `1.05X ETH` (stale price)
   - True entitlement = `X * 0.98 / 1.0` = `0.98X ETH`
   - `assetsCommitted[ETH] += 1.05X` (over-committed by `0.07X ETH`)
6. Operator calls `unlockQueue(ETH, ...)` using `lrtOracle.rsETHPrice()` = `1.05 ether` as price bound reference (before `updateRSETHPrice()` is called):
   - `_createUnlockParams` reads `rsETHPrice = 1.05 ether`
   - `_validatePrices` passes (bounds set around stale price)
   - `_calculatePayoutAmount`: `currentReturn = X * 1.05 / 1.0 = 1.05X`
   - `min(1.05X, 1.05X) = 1.05X ETH` paid out
   - Attacker receives `0.07X ETH` more than their rsETH is worth, extracted from remaining holders.

**Foundry test plan:** Deploy `LRTOracle` and `LRTWithdrawalManager` with `pricePercentageLimit = 0.05e18`. Set `rsETHPrice = 1.05 ether`. Mock `_getTotalEthInProtocol()` to return a value yielding `newRsETHPrice = 0.98 ether`. Call `updateRSETHPrice()` → assert contracts paused, assert `rsETHPrice == 1.05 ether` (unchanged). Unpause all contracts. Call `initiateWithdrawal()` → assert `assetsCommitted` inflated. Call `unlockQueue()` with bounds `[1.04, 1.06]` → assert payout = `1.05X` instead of `0.98X`.
