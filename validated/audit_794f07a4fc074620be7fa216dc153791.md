### Title
Missing State Diff Commitment Verification in P2P Sync Allows Forged State Storage ŌĆö (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

`parse_data_for_block` in the p2p sync client validates only the *length* of assembled state diff chunks against the stored header's `state_diff_length`, but never verifies the assembled `ThinStateDiff` against the header's `state_diff_commitment`. A malicious unauthenticated peer can send chunks whose lengths sum to `N` (passing the length check) but whose assembled diff hashes to `H_forged ŌēĀ H_canonical`, causing the forged diff to be committed to storage. Downstream RPC calls then return values derived from `H_forged` while the stored header retains `state_diff_commitment = H_canonical`.

---

### Finding Description

In `parse_data_for_block`:

```
target_state_diff_len = header.state_diff_length   // only length is read
...
while current_state_diff_len < target_state_diff_len { ... }
if current_state_diff_len != target_state_diff_len { return Err(...) }
// ŌåÉ no calculate_state_diff_hash(result) == header.state_diff_commitment check
Ok(Some((result, block_number)))
``` [1](#0-0) 

The header's `state_diff_commitment` field is read from storage (it was stored verbatim from the peer-supplied `SignedBlockHeader`) but is **never consulted** during state diff assembly. The only guard is the integer length check at line 99. [2](#0-1) 

`write_to_storage` then blindly commits whatever `ThinStateDiff` was assembled: [3](#0-2) 

The header parsing path (`HeaderStreamBuilder::parse_data_for_block`) also performs no cryptographic verification ŌĆö it only checks block number ordering and that exactly one signature is present; it does **not** call `verify_block_signature` and does not verify the block hash: [4](#0-3) 

`verify_block_signature` exists in the API crate but is not invoked anywhere in the p2p sync client path: [5](#0-4) 

The `calculate_state_diff_hash` function and the `verify_state_diff_hash` config flag exist in the committer, but the p2p sync client writes directly to `StorageWriter::append_state_diff` and never goes through the committer's hash-verification path: [6](#0-5) 

---

### Impact Explanation

After the forged diff is stored, any RPC call that reads state ŌĆö `starknet_getStorageAt`, `starknet_getNonce`, `starknet_getClassHashAt`, `starknet_getClassAt` ŌĆö returns values derived from `H_forged`. The stored header still carries `state_diff_commitment = H_canonical`, creating a permanent, silent inconsistency. The node presents authoritative-looking wrong state to callers.

This fits: **High ŌĆö RPC execution / pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

The attacker only needs to be a reachable p2p peer. No validator key, sequencer operator access, or node admin privilege is required. The SQMR protocol is unauthenticated at the application layer; any peer that wins the query race can supply both the header and the state diff chunks. The forged data is not rejected, rate-limited, or ignored ŌĆö it is accepted and durably committed.

---

### Recommendation

After assembling the full `ThinStateDiff` in `parse_data_for_block`, compute `calculate_state_diff_hash(&result)` and compare it against `header.state_diff_commitment`. Return `ParseDataError::BadPeer(...)` on mismatch, mirroring the check already present in the committer: [7](#0-6) 

Additionally, `HeaderStreamBuilder::parse_data_for_block` should call `verify_block_signature` to prevent a peer from supplying an arbitrary `state_diff_commitment` in the header in the first place.

---

### Proof of Concept

1. Store a `BlockHeader` with `state_diff_length = 2` and `state_diff_commitment = calculate_state_diff_hash(&canonical_diff)`.
2. Feed `parse_data_for_block` two `StateDiffChunk` items whose combined `len()` equals 2 but whose content differs from `canonical_diff` (e.g., different storage key/value).
3. The function returns `Ok(Some((forged_diff, block_number)))`.
4. Call `calculate_state_diff_hash(&forged_diff)` and assert it does **not** equal the stored `state_diff_commitment` ŌĆö demonstrating the inconsistency is silently committed to storage.
