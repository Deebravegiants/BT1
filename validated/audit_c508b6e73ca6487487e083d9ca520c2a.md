## Math: Does the stale price cancel out?

Let:
- `P_s` = stale-high stETH price, `P_a` = actual price, `k = P_s/P_a > 1`
- `x` = stETH TVL at actual price, `y` = other-asset TVL (ETH, ETHx, etc.)
- `S` = rsETH supply

After `updateRSETHPrice()` with stale price:
```
rsETHPrice_inflated = (k·x + y) / S
```

Attacker deposits `d` stETH:
```
rsethMinted = (d · P_s) / rsETHPrice_inflated
            = (d · k·P_a · S) / (k·x + y)
```

Correct amount (accurate price):
```
rsethMinted_correct = (d · P_a · S) / (x + y)
```

Ratio:
```
rsethMinted / rsethMinted_correct = k·(x + y) / (k·x + y)
```

For `k > 1` and `y > 0` (other assets exist): **ratio > 1** — attacker mints more rsETH than deserved. The excess is extracted from existing rsETH holders via dilution.

If `y = 0` (only stETH in pool), the ratio equals 1 — no benefit. The attack requires a multi-asset pool, which is the normal production state.

## Guard Evaluation

**`pricePercentageLimit` check:** [7](#0-6) 

This only fires when `newRsETHPrice > highestRsethPrice` AND the increase exceeds `pricePercentageLimit`. It is bypassed entirely when `pricePercentageLimit == 0` (default until explicitly set). Even when set, the stETH/ETH feed's 0.5% deviation threshold means a stale price can drift up to ~0.5% before the feed would have self-updated — which may be within a configured 1% limit.

**Downside protection** (auto-pause on price drop) does not apply here since the stale-high scenario causes a price *increase*.

**`minRSETHAmountExpected` slippage parameter** in `depositAsset()` protects the depositor from receiving *fewer* rsETH than expected, but does not prevent receiving *more* — it is not a guard against this attack.

## Conclusion

### Title
Missing Chainlink Staleness Check Allows Stale stETH/ETH Price to Inflate rsETH Minting Rate — (`contracts/oracles/ChainlinkPriceOracle.sol`)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` discards `updatedAt` and `answeredInRound` from `latestRoundData()`. A stale-high stETH/ETH price inflates `rsETHPrice` via `_updateRsETHPrice()`, and because the same stale price appears in the numerator of `getRsETHAmountToMint()` while the inflated rsETHPrice appears in the denominator, the inflation does not cancel when other assets are present in the pool. An attacker mints rsETH at a rate that exceeds the true stETH/ETH exchange rate, extracting value from existing rsETH holders.

### Finding Description
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and reads only `answer`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

No check on `updatedAt` against a heartbeat, and no check that `answeredInRound >= roundId`. The stETH/ETH Chainlink feed has a 24-hour heartbeat and a 0.5% deviation threshold; during low-volatility periods the feed can go stale for up to 24 hours while the last recorded price remains above the current market price.

The public `updateRSETHPrice()` → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` path uses this unchecked price to compute `totalETHInProtocol`, which is then divided by `rsethSupply` to set `rsETHPrice`. A subsequent `depositAsset(stETH, ...)` call uses the same stale price in the numerator and the inflated stored `rsETHPrice` in the denominator. Because the denominator is only partially inflated (other assets dilute the effect), the attacker receives more rsETH than the deposited stETH is worth.

### Impact Explanation
Existing rsETH holders are diluted: the attacker's rsETH represents a claim on protocol assets exceeding the value deposited. For a protocol with $100M TVL (50% stETH, 50% other assets), a 0.5% stale deviation, and a $10M deposit, the attacker extracts approximately $25,000 per transaction. The attack is repeatable and scales with deposit size.

### Likelihood Explanation
The stETH/ETH Chainlink feed (24h heartbeat, 0.5% deviation threshold) can remain stale for up to 24 hours during low-volatility periods. `updateRSETHPrice()` is permissionless. The `pricePercentageLimit` guard is bypassed when set to 0 (the default) or when the stale deviation falls within the configured limit. The multi-asset pool state (ETH + multiple LSTs) is the normal production configuration.

### Recommendation
Add staleness and round-completeness checks to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();
require(answeredInRound >= roundId, "Stale price: incomplete round");
require(block.timestamp - updatedAt <= HEARTBEAT, "Stale price: heartbeat exceeded");
require(price > 0, "Invalid price");
```

Store a per-feed heartbeat value (e.g., 3600s for 1h feeds, 86400s for 24h feeds) and enforce it. This mirrors the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`.
