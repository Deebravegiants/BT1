Audit Report

## Title
ETH Deposit Limit Is Never Enforced Due to Missing `+ amount` in Boundary Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool.sol` omits the incoming `amount` from the ETH branch comparison, evaluating only `totalAssetDeposits > limit` instead of `totalAssetDeposits + amount > limit`. As a result, the `depositLimitByAsset` cap for ETH is never enforced during a deposit, and any unprivileged caller can push the protocol's total ETH holdings arbitrarily above the configured ceiling while receiving rsETH in return.

## Finding Description
At lines 676–682 of `contracts/LRTDepositPool.sol`, the function reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The ERC-20 branch correctly adds `amount` before comparing against the limit. The ETH branch does not. The parameter `amount` is accepted but silently discarded for ETH.

The call chain is:
1. `depositETH` (L76–93) is `external payable` with no role restriction.
2. It calls `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected)` (L87).
3. `_beforeDeposit` (L648–670) calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` (L661) and reverts only if it returns `true`.
4. For ETH, the function returns `true` only when `totalAssetDeposits` has **already** exceeded the limit from a prior deposit. While `totalAssetDeposits ≤ limit` (the normal operating state), it always returns `false` regardless of `msg.value`.
5. `_mintRsETH` is called and rsETH is issued to the depositor with no cap enforcement.

No existing guard compensates for this: `onlySupportedAsset` only checks that ETH is a supported asset, `nonReentrant` and `whenNotPaused` are unrelated, and the `minAmountToDeposit` check (L657) only enforces a floor, not a ceiling.

## Impact Explanation
The `depositLimitByAsset` cap for `ETH_TOKEN` is the protocol's primary mechanism for bounding ETH intake and downstream EigenLayer exposure. Because the check never includes the incoming `amount`, a depositor can call `depositETH` with an arbitrarily large `msg.value` and receive rsETH in return, minting rsETH well beyond the intended ceiling. This constitutes unbounded over-issuance of rsETH relative to the protocol's risk parameters. The contract fails to deliver its promised deposit-limit guarantee for ETH — matching the allowed impact **"Contract fails to deliver promised returns"** (Low). At sufficient deposit volume, the excess minting could escalate toward protocol insolvency (Critical), but the immediate, directly provable impact is the broken limit guarantee.

## Likelihood Explanation
`depositETH` is `public payable` with no role or whitelist requirement. Any ETH holder can trigger the path. The defect is present in every deployment of the current code and requires no precondition beyond `totalAssetDeposits ≤ limit`, which is the normal operating state. The exploit is repeatable on every block. Likelihood is **High**.

## Recommendation
Mirror the ERC-20 branch for ETH by including `amount` in the comparison. Using `>=` also closes the off-by-one that allows one final deposit exactly at the limit:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount >= lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
**Setup:** `depositLimitByAsset[ETH_TOKEN] = 1000 ether`, `totalAssetDeposits(ETH) = 999 ether`.

**Steps:**
1. Call `LRTDepositPool.depositETH{value: 500 ether}(0, "")` from any EOA.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)`.
3. ETH branch evaluates `999 ether > 1000 ether` → `false`; `amount` (500 ether) is never used.
4. `_mintRsETH` executes; total ETH becomes 1499 ether — 499 ETH above the cap.
5. Repeat with any `msg.value`; the check never triggers while `totalAssetDeposits ≤ limit`.

**Expected (fixed) behaviour:** `999 + 500 = 1499 ether >= 1000 ether` → `true` → revert `MaximumDepositLimitReached`.

**Foundry fuzz test sketch:**
```solidity
function testFuzz_ethDepositLimitBypass(uint96 amount) public {
    vm.assume(amount > 0);
    // set limit to 1000 ether, seed pool with 999 ether
    // any amount should revert MaximumDepositLimitReached
    vm.deal(attacker, amount);
    vm.prank(attacker);
    vm.expectRevert(LRTDepositPool.MaximumDepositLimitReached.selector);
    depositPool.depositETH{value: amount}(0, "");
}
```
Under the current code this test fails for all `amount` values while `totalAssetDeposits ≤ limit`, confirming the bypass.
