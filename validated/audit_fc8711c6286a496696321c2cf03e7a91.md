Audit Report

## Title
`instantWithdrawal` Pays Out at Stale Pre-Slashing `rsETHPrice`, Enabling Theft from Remaining rsETH Holders — (`contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTWithdrawalManager.instantWithdrawal()` computes the asset payout using the cached `rsETHPrice` state variable in `LRTOracle`, which is only updated when `updateRSETHPrice()` is explicitly called. Because `NodeDelegatorHelper.getAssetBalance` reads live data from EigenLayer's `DelegationManager.getWithdrawableShares`, an EigenLayer slashing event immediately reduces the protocol's true TVL while `rsETHPrice` remains at the pre-slashing value. Any rsETH holder can exploit this window to withdraw at the inflated stale price, receiving more assets than their proportional share and concentrating the slashing loss on remaining holders.

## Finding Description

`instantWithdrawal` computes the payout at line 228 via `getExpectedAssetAmount`:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

`lrtOracle.rsETHPrice()` is a plain state variable updated only inside `_updateRsETHPrice()`, which is only triggered by an explicit call to `updateRSETHPrice()` or `updateRSETHPriceAsManager()`. There is no automatic or atomic update.

The live TVL path is: `getTotalAssetDeposits` → `getAssetDistributionData` → `INodeDelegator.getAssetBalance` → `NodeDelegatorHelper.getAssetBalance` → `NodeDelegatorHelper.getWithdrawableShare` → `DelegationManager.getWithdrawableShares`. This chain reads EigenLayer's live accounting, so a slashing event is immediately reflected in the true TVL — but only when `_updateRsETHPrice()` is called to recompute `rsETHPrice`.

The queued withdrawal path uses `_calculatePayoutAmount`, which takes `min(expectedAssetAmount, currentReturn)` at unlock time, ensuring slashing is always reflected before payout. `instantWithdrawal` has no equivalent guard: it burns rsETH and immediately transfers assets at the stale price with no recalculation.

The `pricePercentageLimit` downside-pause mechanism only activates inside `_updateRsETHPrice()`. Since the oracle has not been updated, the protocol is not paused, and `instantWithdrawal` (which is `whenNotPaused`) proceeds freely.

**Exploit path:**
1. EigenLayer operator is slashed; `DelegationManager.getWithdrawableShares` for the NDC immediately decreases.
2. `rsETHPrice` in `LRTOracle` still holds the pre-slashing value.
3. Attacker (any rsETH holder) calls `instantWithdrawal(asset, largeRsETHAmount, ...)`.
4. `assetAmountUnlocked` is computed at the stale inflated price.
5. Attacker's rsETH is burned and they receive the inflated asset amount from `LRTUnstakingVault`.
6. `updateRSETHPrice()` is eventually called; the new price reflects the TVL drop.
7. Remaining rsETH holders now hold tokens worth less than their proportional share, having absorbed the attacker's portion of the slashing loss.

**Preconditions:** `isInstantWithdrawalEnabled[asset] == true` (manager-controlled toggle) and sufficient assets in `LRTUnstakingVault` for instant withdrawal.

## Impact Explanation

This is **Critical — Direct theft of user funds**. The attacker burns rsETH at a stale inflated price and receives more underlying assets than their fair share. The deficit is borne by remaining rsETH holders as a larger-than-proportional price drop when the oracle is next updated. This is a direct, concrete loss of principal for remaining holders, not merely unclaimed yield.

## Likelihood Explanation

EigenLayer slashing events are observable on-chain the moment they occur via `AllocationManager` or `DelegationManager` events. `updateRSETHPrice()` is a public function but is not called atomically with slashing; there is always a non-zero block window. A user monitoring EigenLayer events can detect the discrepancy and call `instantWithdrawal` in the same or next block. The only gating condition is `isInstantWithdrawalEnabled`, which is a manager toggle that may be enabled for ETH or LSTs in normal operation. **Likelihood: Medium.**

## Recommendation

1. **Force a fresh price computation inside `instantWithdrawal`**: Call `_updateRsETHPrice()` (or an equivalent internal view that computes the current price without caching) before computing `assetAmountUnlocked`, so the payout always reflects the current TVL.
2. **Add a staleness bound**: Revert if `block.timestamp` exceeds the last oracle update timestamp by more than a configurable threshold (e.g., 1 hour).
3. **Apply the same `min(expectedAmount, currentReturn)` guard used in `_calculatePayoutAmount`**: Compute the expected amount at initiation and again at execution, paying the lesser of the two. This is the existing pattern for queued withdrawals and should be applied consistently.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// 1. Fork mainnet at a block where an EigenLayer operator is about to be slashed.
// 2. Simulate slashing: call AllocationManager.slashOperator() reducing the NDC's
//    getWithdrawableShares by X%.
// 3. Assert: lrtOracle.rsETHPrice() still returns the pre-slashing value.
// 4. Assert: lrtDepositPool.getTotalAssetDeposits(asset) now returns a lower value
//    (reflecting the live getWithdrawableShares reduction).
// 5. Attacker calls instantWithdrawal(asset, largeAmount, "").
// 6. Record attacker's received asset amount.
// 7. Call updateRSETHPrice(); record new rsETHPrice.
// 8. Compute what the attacker *should* have received at the post-slashing price.
// 9. Assert: attacker received more than the post-slashing fair value.
// 10. Assert: remaining rsETH totalSupply * newRsETHPrice < pre-attack TVL - attacker's fair share.
```

The differential between step 6 and step 8 quantifies the theft. The test is reproducible on any fork where EigenLayer slashing is active and `isInstantWithdrawalEnabled` is true for the target asset.
