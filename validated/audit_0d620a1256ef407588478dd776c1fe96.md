### Title
NFT Royalty Payment Sent to Zero Puzzle Hash Burns XCH/CAT During Offer Settlement - (`chia/wallet/nft_wallet/nft_wallet.py`, `chia/wallet/nft_wallet/nft_puzzle_utils.py`)

---

### Summary

An NFT minted with `royalty_puzzle_hash = bytes32.zeros` (all-zero puzzle hash) will have its royalty payment permanently burned during offer settlement. The offer-settlement code in `make_nft1_offer` creates a `CreateCoin` condition targeting the zero puzzle hash without any guard, and the minting guard in `create_ownership_layer_puzzle` fails to catch `bytes32.zeros` because Python's truthiness check treats non-empty bytes as truthy.

---

### Finding Description

**Root cause 1 — Minting guard is ineffective for `bytes32.zeros`**

In `create_ownership_layer_puzzle`:

```python
if not royalty_puzzle_hash:
    royalty_puzzle_hash = p2_puzzle.get_tree_hash()
```

`bytes32.zeros` is a 32-byte non-empty object. Python evaluates `not bytes32.zeros` as `False`, so the guard never fires and the zero puzzle hash is baked into the NFT's transfer program via `NFT_TRANSFER_PROGRAM_DEFAULT.curry(singleton_struct, royalty_puzzle_hash, percentage)`. [1](#0-0) 

**Root cause 2 — Offer settlement creates `CreateCoin` to zero puzzle hash without address validation**

In `make_nft1_offer`, the `royalty_address` is read directly from the NFT's on-chain transfer program and used unconditionally to build a `CreateCoin`:

```python
required_royalty_info.append(
    (asset, bytes32(transfer_info["transfer_program"]["royalty_address"]), uint16(royalty_percentage))
)
...
payment_list.append((launcher_id, CreateCoin(address, extra_royalty_amount, [address])))
```

There is no check for `address == bytes32.zeros`. The only existing filter skips payments where `p.amount == 0`, but if `royalty_percentage > 0` the amount is non-zero and the payment proceeds. [2](#0-1) [3](#0-2) 

**Entry point — RPC minting accepts arbitrary royalty address**

`nft_mint_nft` in `wallet_rpc_api.py` decodes whatever `royalty_address` the caller supplies and passes it directly to `generate_new_nft`:

```python
if request.royalty_address is not None:
    royalty_puzhash = decode_puzzle_hash(request.royalty_address)
else:
    royalty_puzhash = await action_scope.get_puzzle_hash(...)
```

Any unprivileged user can supply the bech32 encoding of `bytes32.zeros` as `royalty_address`. [4](#0-3) 

---

### Impact Explanation

When a taker accepts an offer for such an NFT, `make_nft1_offer` constructs a spend bundle that includes a `CreateCoin(bytes32.zeros, royalty_amount, [bytes32.zeros])` condition. The zero puzzle hash has no known preimage; the coin created at that address is permanently unspendable. The XCH or CAT royalty amount is irreversibly burned. The NFT transfer itself completes, so the taker receives the NFT while losing the royalty amount on top of the purchase price.

This is an accounting change affecting XCH/CATs during offer settlement — matching the Critical impact category. [5](#0-4) 

---

### Likelihood Explanation

The attack requires minting an NFT with `royalty_puzzle_hash = bytes32.zeros` and `royalty_percentage > 0`. This is reachable by any unprivileged wallet user via the `nft_mint_nft` RPC or the `chia wallet nft mint` CLI command with `--royalty-address <bech32(zeros)> --royalty-percentage-fraction <N>`. No special privileges, leaked keys, or cryptographic breaks are required. The NFT can then be listed on any marketplace or offered peer-to-peer. [6](#0-5) [7](#0-6) 

---

### Recommendation

1. **In `create_ownership_layer_puzzle`**: Replace the falsy check with an explicit `None` check and add a separate guard for the zero puzzle hash:
   ```python
   if royalty_puzzle_hash is None or royalty_puzzle_hash == bytes32.zeros:
       royalty_puzzle_hash = p2_puzzle.get_tree_hash()
   ``` [8](#0-7) 

2. **In `make_nft1_offer`**: Before appending to `required_royalty_info`, skip or substitute any entry where `royalty_address == bytes32.zeros`:
   ```python
   if bytes32(transfer_info["transfer_program"]["royalty_address"]) == bytes32.zeros:
       continue  # or redirect to seller
   ``` [9](#0-8) 

3. **In `nft_mint_nft` RPC**: Reject `royalty_address` that decodes to `bytes32.zeros` when `royalty_percentage > 0`. [10](#0-9) 

---

### Proof of Concept

1. Mint an NFT via RPC:
   ```json
   {
     "wallet_id": 2,
     "royalty_address": "<bech32 encoding of 0x0000...0000>",
     "royalty_percentage": 500,
     "hash": "...",
     "uris": ["..."],
     "did_id": "<valid DID>"
   }
   ```
   The NFT is minted with `royalty_puzzle_hash = bytes32.zeros` baked into its transfer program because `if not bytes32.zeros` evaluates to `False`. [1](#0-0) 

2. A second wallet takes an offer for this NFT offering 1 XCH. `make_nft1_offer` reads `royalty_address = bytes32.zeros` from the driver dict, computes `royalty_amount = 1_000_000_000_000 * 500 / 10000 = 50_000_000_000` mojos, and appends `CreateCoin(bytes32.zeros, 50_000_000_000, [bytes32.zeros])` to the spend bundle. [11](#0-10) 

3. The spend bundle is submitted and confirmed. The taker's wallet loses 50 billion mojos (0.05 XCH) to the zero puzzle hash permanently. The NFT transfer succeeds. [3](#0-2)
