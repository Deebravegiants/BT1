### Title
`fee_proposal_fri` Bound Enforcement Silently Skipped During Startup Window — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The `is_proposal_init_valid` function in the proposal validation path intentionally skips the `fee_proposal_fri` bound check when `fee_actual` is `None`. `fee_actual` is `None` whenever the sliding window of past `fee_proposal` values is incomplete — specifically for the first `fee_proposal_window_size` blocks after genesis or after a node restart/revert that skips `initialize_fee_proposals_window`. During this window, a malicious proposer can publish an arbitrarily large or small `fee_proposal_fri` value in `ProposalInit`, and every validator will accept it without any range check. Because `fee_proposal_fri` is hashed into the `ProposalCommitment` via `Poseidon(partial_block_hash, fee_proposal_fri)`, the accepted extreme value is permanently committed to the chain and seeded into the `fee_proposals_window` of all nodes. This poisons the median computation for the next `window_size` blocks, allowing a malicious proposer to drive `fee_actual` — and therefore the L2 base fee — to an arbitrary value.

### Finding Description

**The guard that is supposed to prevent this:**

In `is_proposal_init_valid`, the fee_proposal bound check is:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    // ... range check ...
}
``` [1](#0-0) 

The `if let` pattern silently does nothing when `proposal_init_validation.fee_actual` is `None`. The comment explicitly acknowledges this: *"During initiation (fee_actual is None, <window_size blocks), bounds are not enforced."*

**How `fee_actual` becomes `None`:**

`fee_actual` is computed by `compute_fee_actual`, which returns `None` if any height in the window `[height - window_size, height - 1]` is missing from `fee_proposals_window` or recorded as `None`:

```rust
Some(None) | None => {
    warn!(...);
    return None;
}
``` [2](#0-1) 

This `None` is passed directly into `ProposalInitValidation.fee_actual`:

```rust
fee_actual: compute_fee_actual(
    &self.fee_proposals_window,
    init.height,
    VersionedConstants::latest_constants().fee_proposal_window_size,
),
``` [3](#0-2) 

**The proposer side also uses `fee_actual = None` as a fallback:**

When `fee_actual` is `None`, the proposer freezes `fee_proposal` at `self.l2_gas_price`:

```rust
let Some(fee_actual) = fee_actual else {
    warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
    return self.l2_gas_price;
};
``` [4](#0-3) 

An honest proposer freezes at `l2_gas_price`. But a malicious proposer can set `fee_proposal_fri` to any value — `u128::MAX`, `1`, or any extreme — and the validator will accept it because the `if let` guard is skipped.

**The accepted value is committed and poisons future blocks:**

`fee_proposal_fri` is bound into the `ProposalCommitment` via Poseidon hash:

```rust
ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
``` [5](#0-4) 

After `decision_reached`, the accepted `fee_proposal_fri` is recorded into `fee_proposals_window`:

```rust
self.record_fee_proposal(height, init.fee_proposal_fri);
``` [6](#0-5) 

Once `window_size` blocks have been committed with the attacker-controlled values, `fee_actual` becomes `Some(extreme_value)`, and the EIP-1559-style `calculate_next_l2_gas_price_for_fin` uses it as a floor:

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),
    None => config_min,
};
``` [7](#0-6) 

### Impact Explanation

A malicious proposer who controls the first `window_size` blocks (or any `window_size`-block window where `fee_actual` is `None` due to a restart/revert) can:

1. Publish `fee_proposal_fri = u128::MAX` in every `ProposalInit` during the unguarded window.
2. All validators accept these proposals without any range check.
3. After `window_size` blocks, `fee_actual` becomes `Some(u128::MAX)`.
4. `calculate_next_l2_gas_price_for_fin` uses `u128::MAX` as the effective minimum, driving the L2 base fee to `u128::MAX`.
5. All user transactions become economically impossible to execute (fees are astronomically high), or conversely, setting `fee_proposal_fri = 1` drives fees to near-zero, enabling economic attacks.

This matches the **High** impact category: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

### Likelihood Explanation

The window is `fee_proposal_window_size` blocks long (a protocol constant). It is triggered:
- At genesis (first `window_size` blocks of the chain).
- After any node restart where `initialize_fee_proposals_window` is not called or fails.
- After a chain revert that clears the window.

The attacker only needs to be the proposer for one block during this window to inject a poisoned value. Since the window spans many blocks, the probability of a validator-set member being the proposer at least once is high. The attack requires no special privileges beyond being a legitimate proposer.

### Recommendation

Replace the silent skip with an explicit fallback bound. When `fee_actual` is `None`, validate `fee_proposal_fri` against a hard absolute cap (e.g., `max_l2_gas_price` from versioned constants or a configured ceiling) rather than skipping validation entirely:

```rust
let effective_fee_actual = proposal_init_validation.fee_actual
    .unwrap_or(MAX_ALLOWED_FEE_PROPOSAL_DURING_STARTUP);

let (lower_bound, upper_bound) = fee_proposal_bounds(
    effective_fee_actual,
    VersionedConstants::latest_constants().fee_proposal_margin_ppt,
);
if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
    return Err(...);
}
```

Alternatively, during the startup window, clamp the accepted `fee_proposal_fri` to `[min_gas_price, max_gas_price]` before recording it into `fee_proposals_window`, so that even if the bound check is skipped, the recorded value cannot be extreme.

### Proof of Concept

**Setup:** Chain at genesis, `fee_proposal_window_size = W`. `fee_proposals_window` is empty, so `compute_fee_actual` returns `None` for all heights `< W`.

**Step 1:** Malicious proposer at height `H < W` constructs a `ProposalInit` with `fee_proposal_fri = u128::MAX`.

**Step 2:** `validate_proposal` calls `is_proposal_init_valid`. The check at line 398 evaluates `if let (Some(fee_actual), Some(fee_proposal)) = (None, Some(u128::MAX))` — the pattern does not match, the body is skipped, `Ok(())` is returned. [1](#0-0) 

**Step 3:** `handle_proposal_part` calls `proposal_commitment_from(partial, fee_proposal=Some(u128::MAX))`, producing `Poseidon(partial, u128::MAX)` as `batcher_block_commitment`. This matches `fin.proposal_commitment` (since the malicious proposer also computed it correctly), so `ProposalFinMismatch` is not triggered. [8](#0-7) 

**Step 4:** `decision_reached` calls `finalize_decision`, which calls `self.record_fee_proposal(height, Some(u128::MAX))`. [6](#0-5) 

**Step 5:** After `W` such blocks, `compute_fee_actual` returns `Some(u128::MAX)`. `calculate_next_l2_gas_price_for_fin` uses `max(config_min, u128::MAX) = u128::MAX` as the effective minimum, permanently setting the L2 base fee to `u128::MAX`. [9](#0-8)
