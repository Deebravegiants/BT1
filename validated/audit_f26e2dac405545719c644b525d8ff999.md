### Title
Attached NEAR Deposit Permanently Locked in `submit_participant_info` When Caller Is an Existing Participant Re-Submitting Attestation - (File: `crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is marked `#[payable]` and contains a conditional refund path. When an existing participant re-submits their attestation (e.g., after a TEE upgrade), the branch that handles refunds is skipped entirely, causing any accidentally attached NEAR deposit to be permanently locked in the contract.

---

### Finding Description

In `crates/contract/src/lib.rs`, the function `submit_participant_info` is decorated with `#[payable]`, meaning it unconditionally accepts any attached NEAR deposit from the caller. [1](#0-0) 

After inserting or updating the attestation, the function computes a boolean flag: [2](#0-1) 

The deposit check and refund logic is gated entirely on this flag: [3](#0-2) 

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — which happens when the caller **is** an existing participant (`caller_is_not_participant == false`) **and** the attestation is an update rather than a new insertion (`is_new_attestation == false`) — the entire `if` block is skipped. There is no `else` branch that either rejects a non-zero deposit or refunds it. Any NEAR tokens attached to the call in this scenario are silently absorbed by the contract with no recovery path.

This is the direct NEAR analog of M-03: just as the Solidity contract locked ETH when `msg.value > 0` was sent alongside an ERC20 transfer (the non-ETH branch had no `require(msg.value == 0)`), this contract locks NEAR when `attached_deposit > 0` is sent alongside a re-attestation update (the non-storage-charging branch has no rejection or refund).

---

### Impact Explanation

Any NEAR tokens attached by an existing participant when calling `submit_participant_info` to refresh or update their TEE attestation are permanently locked in the MPC contract. There is no administrative withdrawal function, no sweep mechanism, and no way for the caller to recover the funds after the transaction completes. This constitutes a permanent, irreversible loss of user funds held by the contract.

This matches the allowed impact: **Medium — balance or contract execution-flow manipulation that breaks production safety/accounting invariants.**

---

### Likelihood Explanation

Re-attestation is a routine, expected operation for MPC participants — it occurs after TEE image upgrades, certificate rotations, or node restarts. Participants are accustomed to attaching a deposit on their *first* call (to cover storage costs) and may repeat that pattern on subsequent calls. The NEAR SDK does not warn callers when a `#[payable]` function silently discards their deposit. The scenario is realistic and requires no adversarial intent — a well-meaning participant following their own prior procedure is sufficient to trigger it.

---

### Recommendation

Add an explicit guard in the `else` branch (when `attestation_storage_must_be_paid_by_caller` is `false`) that either rejects a non-zero deposit or refunds it unconditionally:

```rust
} else {
    // Caller is an existing participant re-submitting; no storage cost applies.
    // Reject or refund any accidentally attached deposit to prevent fund lock.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, assert `attached_deposit == 0` in this branch and return an error, mirroring the M-03 mitigation pattern exactly.

---

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` for the first time with `1 NEAR` attached (to cover storage). The call succeeds; excess is refunded via the `if attestation_storage_must_be_paid_by_caller` branch.

2. After a TEE image upgrade, `alice.near` calls `submit_participant_info` again with `0.5 NEAR` attached (habit from step 1). At this point:
   - `is_new_attestation` = `false` (alice already has a stored attestation entry — it is being updated, not newly inserted).
   - `caller_is_not_participant` = `false` (alice is a current participant).
   - Therefore `attestation_storage_must_be_paid_by_caller` = `false`.

3. The `if` block at line 826 is skipped entirely. [4](#0-3) 

4. The `0.5 NEAR` is credited to the contract's balance with no refund promise issued and no error returned. The call returns `Ok(())`. Alice's `0.5 NEAR` is permanently locked.
