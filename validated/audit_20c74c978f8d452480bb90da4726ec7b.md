### Title
Governance Threshold Silently Reduced by Integer Division in Robust ECDSA Signing — (`File: crates/node/src/providers/robust_ecdsa.rs`)

### Summary

`translate_threshold()` converts the governance threshold to a `MaxMalicious` value for the DamgardEtAl (Robust ECDSA) protocol using integer division `(threshold - 1) / 2`. For any **even** governance threshold, this silently reduces the effective signing threshold by one participant. The signing protocol then enforces exactly `2 * max_malicious + 1` participants — a value strictly less than the governance threshold — allowing a Byzantine coalition one participant below the governance threshold to produce a valid threshold signature without the required number of cooperating parties.

---

### Finding Description

`translate_threshold` in `crates/node/src/providers/robust_ecdsa.rs` computes:

```rust
Ok(MaxMalicious::from((number_of_signers - 1) / 2))
``` [1](#0-0) 

`number_of_signers` is set equal to the governance threshold by `get_number_of_signers`: [2](#0-1) 

For an **even** governance threshold `T`, integer division yields:

```
MaxMalicious = (T - 1) / 2  →  floor((T-1)/2)
```

The signing protocol then enforces **exactly** `2 * MaxMalicious + 1` participants: [3](#0-2) 

Concrete reduction table:

| Governance Threshold (T) | MaxMalicious | Actual signers needed (2M+1) | Reduction |
|---|---|---|---|
| 5 | 2 | 5 | none |
| **6** | **2** | **5** | **−1** |
| 7 | 3 | 7 | none |
| **8** | **3** | **7** | **−1** |
| **10** | **4** | **9** | **−1** |

This is confirmed by the existing test fixture: [4](#0-3) 

`threshold=6` maps to `MaxMalicious=2`, so only **5** participants are required to sign — one below the governance threshold of 6.

The `compute_thresholds` guard only checks `2*M+1 <= num_signers` (inequality), not equality, so it passes silently: [5](#0-4) 

This same `translate_threshold` is called in the leader signing path, the follower signing path, and the presignature generation path: [6](#0-5) [7](#0-6) 

Additionally, the presignature is generated with `num_signers = T` participants (e.g., 6), but the signing phase uses exactly `2*M+1 = 5` participants. This creates `N1 > N2`, which the protocol's own security documentation explicitly forbids: [8](#0-7) 

> "Do not allow N1 > 2t + 1 … Allowing larger presigning sets or smaller signing sets enables split-view and presignature-reuse attacks."

---

### Impact Explanation

**Critical — Bypass of threshold-signature requirements.**

For any even governance threshold `T` on the DamgardEtAl (Robust ECDSA) domain, a Byzantine coalition of exactly `T − 1` participants can:

1. Participate in presignature generation (they are within the `T`-participant presign set).
2. Drive the signing phase with only `2*((T-1)/2)+1 = T-1` participants — the exact count the protocol enforces.
3. Produce a cryptographically valid ECDSA signature that passes the on-chain `respond()` verification. [9](#0-8) 

The on-chain verifier checks only cryptographic validity of the signature, not how many MPC participants contributed. A signature produced by `T-1` colluding nodes is indistinguishable from one produced by `T` honest nodes.

Furthermore, the `N1 > N2` condition (presign with 6, sign with 5) opens the split-view attack vector documented in the protocol, where a malicious coordinator can present different `(msg_hash, tweak)` pairs to different signing subsets derived from the same presignature, enabling secret key recovery.

---

### Likelihood Explanation

The governance threshold is constrained to `[ceil(0.6n), n]`. For any participant count `n ≥ 5`, even values in this range are common (e.g., n=10 → valid thresholds 6, 7, 8, 9, 10; three of five are even). The DamgardEtAl domain is actively being deployed to production: [10](#0-9) 

The code itself acknowledges the `translate_threshold` function is a known hack pending resolution of two open issues, meaning the reduction is not an intentional security design: [11](#0-10) 

Any deployment with an even governance threshold and an active DamgardEtAl domain is immediately affected.

---

### Recommendation

1. **Enforce `2*MaxMalicious+1 == num_signers`** (equality, not `<=`) in `compute_thresholds`, so any even threshold that would silently reduce security is rejected at startup rather than silently accepted.
2. **Round up** instead of truncating: use `threshold / 2` (ceiling for odd) or require the governance threshold to always be odd when a DamgardEtAl domain is active.
3. **Validate N1 = N2**: ensure the participant set used for presignature generation is exactly `2*MaxMalicious+1`, matching the signing set, as required by the protocol security proof.

---

### Proof of Concept

**Setup**: 10 participants, governance threshold `T = 6` (valid: `ceil(0.6*10) = 6`), one DamgardEtAl domain.

**Step 1** — `translate_threshold(6, 10)` returns `MaxMalicious::from(2)` (confirmed by test `#[case(6, 10, false, 2)]`).

**Step 2** — `compute_thresholds(6, 10)` returns `(num_signers=6, robust_ecdsa_threshold=MaxMalicious(2))`. The guard `2*2+1=5 <= 6` passes silently.

**Step 3** — Presignature generation runs with 6 participants (N1=6).

**Step 4** — At signing time, `sign()` enforces `participants.len() == 2*2+1 = 5` (N2=5). Only 5 participants are needed.

**Step 5** — A Byzantine coalition of 5 participants (one below the governance threshold of 6) drives the signing phase. They produce a valid ECDSA signature.

**Step 6** — The signature is submitted via `respond()`. The on-chain verifier checks only `verify_ecdsa_signature(signature, payload_hash, expected_public_key)` — it passes. The pending request is resolved and the signature is delivered to the user.

**Result**: A threshold-signature was issued with `T-1 = 5` participants, bypassing the governance threshold of 6. With `N1=6 > N2=5`, the split-view attack surface documented in `signing.md` is also open, enabling potential secret key extraction across two signing sessions. /// This function translates the current threshold from the contract
/// to the threshold expected by the robust-ecdsa scheme, which
/// is semantically different.
/// The function should be no longer needed when these issues are solved:
/// https://github.com/near/threshold-signatures/issues/255
/// https://github.com/near/mpc/issues/1649
pub(super) fn translate_threshold(
    threshold: usize,
    number_of_participants: usize,
) -> anyhow::Result<MaxMalicious> {
    let number_of_signers = get_number_of_signers(threshold, number_of_participants)?;
    anyhow::ensure!(
        number_of_signers >= 5,
        "Robust ECDSA requires the threshold to be at least 2, which implies that the number of signers needs to be at least 5"
    );
    Ok(MaxMalicious::from((number_of_signers - 1) / 2))
}

/// Computes `(num_signers, robust_ecdsa_threshold)` and validates the
/// `2 * max_malicious + 1 <= num_signers` invariant. Returns an error only if
/// the configured governance threshold is invalid for robust-ECDSA.
///
/// TODO(#3164): once the node supports per-domain thresholds, this should
/// take the domain-specific threshold instead of the single governance threshold.
fn compute_thresholds(
    governance_threshold: u64,
    num_running_participants: usize,
) -> anyhow::Result<(usize, MaxMalicious)> {
    let governance_threshold: usize = governance_threshold.try_into()?;
    let num_signers = get_number_of_signers(governance_threshold, num_running_participants)?;
    let robust_ecdsa_threshold =
        translate_threshold(governance_threshold, num_running_participants)?;
    anyhow::ensure!(
        robust_ecdsa_threshold
            .value()
            .checked_mul(2)
            .and_then(|v| v.checked_add(1))
            .is_some_and(|v| v <= num_signers)
    );
    Ok((num_signers, robust_ecdsa_threshold))
}
