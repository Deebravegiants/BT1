Audit Report

## Title
Missing Slippage Protection in L2 Pool `deposit()` Functions Allows Users to Receive Less rsETH Than Expected - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol)

## Summary
All three L2 pool contracts expose public `deposit()` entry points that compute the minted rsETH/wrsETH amount using a live oracle rate at execution time, with no `minRsETHAmount` guard. A user who previews the expected output via `viewSwapRsETHAmountAndFee()` off-chain and then submits a deposit can receive materially fewer tokens than previewed if the oracle rate increases between preview and execution, with no on-chain recourse. The L1 `LRTDepositPool` already enforces a `minRSETHAmountExpected` parameter for the identical scenario, confirming the L2 omission is an inconsistency rather than a design choice.

## Finding Description
Every L2 pool variant exposes a public `deposit()` entry point callable by any unprivileged user:

- `RSETHPoolV3ExternalBridge.deposit(string referralId)` (ETH path, L366–384) and `deposit(address token, uint256 amount, string referralId)` (LST path, L390–412)
- `RSETHPool.deposit(string referralId)` (L265–278)
- `RSETHPoolNoWrapper.deposit(string referralId)` (L231–244)

In all cases the minted amount is computed as:
```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // viewSwapRsETHAmountAndFee
```
where `rsETHToETHrate = getRate()` is read live from the oracle at execution time (e.g., `IOracle(rsETHOracle).getRate()`). There is no `minRsETHAmount` parameter or guard anywhere in the call path.

The oracle rate (`getRate()`) reflects the rsETH/ETH exchange rate, which increases monotonically as EigenLayer staking rewards accrue and is propagated cross-chain via `RSETHMultiChainRateProvider` / `RSETHRateReceiver`. Whenever the L2 oracle is updated to a higher rate before a user's pending deposit transaction is included, the user receives strictly fewer tokens than `viewSwapRsETHAmountAndFee` showed at preview time, with no ability to revert.

By contrast, `LRTDepositPool.depositETH()` (L76–93) accepts `minRSETHAmountExpected` and enforces it in `_beforeDeposit()` (L665–669):
```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```
The L2 pools provide no equivalent protection.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The user's ETH/LST is consumed in full. The shortfall in wrsETH/rsETH tokens is permanent and unrecoverable within the protocol. The user does not lose ETH value in absolute terms (the fewer tokens they receive are each worth more ETH at the new rate), but the contract fails to deliver the token quantity that `viewSwapRsETHAmountAndFee` — the protocol's own preview function — indicated the user would receive. This matches the allowed Low impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
The rsETH oracle rate increases monotonically and is updated periodically as staking rewards accrue and the L2 oracle receiver is refreshed. Any user who previews the rate off-chain and submits a deposit during an oracle update window will receive fewer tokens than previewed. No privileged access, capital, or special conditions are required beyond normal deposit timing. An MEV searcher can also observe a pending oracle update transaction and order a user's deposit after it in the same block, guaranteeing the user receives the post-update (worse) rate. The attack requires no capital beyond gas and is repeatable for every deposit during every oracle update cycle.

## Recommendation
Add a `minRsETHAmount` parameter to all `deposit()` overloads in every L2 pool contract, mirroring the pattern already used in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert SlippageExceeded();

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same change to the token `deposit()` overload in `RSETHPoolV3ExternalBridge` and to `RSETHPool` and `RSETHPoolNoWrapper`.

## Proof of Concept
1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and observes she will receive `X` wrsETH at current rate `R`.
2. Alice submits `deposit{value: 1 ether}("ref")`.
3. Before Alice's transaction is included, the L2 oracle receiver is updated to rate `R' > R` (rsETH is now worth more ETH per token).
4. Alice's transaction executes: `rsETHAmount = 1e18 * 1e18 / R'` — strictly less than `X`.
5. Alice receives fewer wrsETH than previewed with no recourse, as there is no `minRsETHAmount` check to revert the transaction.

**Foundry fork test plan:**
```solidity
function testSlippageNoProtection() public {
    uint256 depositAmount = 1 ether;
    // 1. Record preview amount at current rate
    (uint256 expectedRsETH,) = pool.viewSwapRsETHAmountAndFee(depositAmount);
    // 2. Simulate oracle rate increase (prank oracle manager, call setRate with higher value)
    vm.prank(oracleManager);
    interimOracle.setRate(currentRate * 101 / 100); // +1%
    // 3. Execute deposit
    vm.prank(alice);
    pool.deposit{value: depositAmount}("ref");
    // 4. Assert Alice received fewer tokens than previewed
    uint256 actualRsETH = wrsETH.balanceOf(alice);
    assertLt(actualRsETH, expectedRsETH); // passes — no slippage protection
}
```
