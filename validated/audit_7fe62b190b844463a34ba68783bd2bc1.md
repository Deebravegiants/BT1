### Title
Single Attested Participant Can Submit Arbitrary CKD Response for `AppPublicKey` Requests, Controlling the Derived Secret - (File: crates/contract/src/lib.rs)

### Summary

In `respond_ckd`, when a `CKDRequest` uses the `AppPublicKey` (privately verifiable) variant, the contract performs **no cryptographic validation** of the submitted `CKDResponse`. Any single attested participant can call `respond_ckd` with fabricated `big_y` and `big_c` values, causing the user to receive an attacker-controlled derived key. The attacker can choose `big_y` and `big_c` such that they know the exact secret the user will derive.

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` contains an explicit branch that skips all validation for the `AppPublicKey` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no validation whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKeyPV` variant carries both a G1 and G2 component (`pk1 = a·G1`, `pk2 = a·G2`), enabling an on-chain pairing check via `ckd_output_check`. The `AppPublicKey` variant carries only `pk1 = a·G1`, so no pairing check is possible on-chain. The design intent is that the app verifies correctness privately after decryption. However, the contract still accepts and forwards whatever `big_y` / `big_c` a single participant submits, with no threshold enforcement on the response itself.

`respond_ckd` only requires the caller to be **any** single attested participant — not specifically the coordinator, and not a threshold quorum: [2](#0-1) 

The first `respond_ckd` call that matches a pending request resolves the yield and removes the entry from `pending_ckd_requests`, so a malicious participant who front-runs the legitimate coordinator wins.

**Attacker-controlled key derivation math:**

The correct CKD output satisfies:
- `big_y = y·G1` (random)
- `big_c = msk·H(app_id) + y·A` where `A = a·G1` is the user's public key

The user decrypts as: `big_c − a·big_y = msk·H(app_id)`

A malicious participant who knows `A` (it is on-chain in the `CKDRequest`) can instead submit:
- `big_y = y'·G1` (attacker-chosen)
- `big_c = s + y'·A` (where `s` is any attacker-chosen scalar)

The user then decrypts: `big_c − a·big_y = s + y'·A − a·y'·G1 = s + y'·a·G1 − a·y'·G1 = s`

The attacker has made the user's derived secret equal to `s`, a value the attacker chose and knows.

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Monitor the chain for `request_app_private_key` calls using the `AppPublicKey` variant.
2. Immediately call `respond_ckd` with fabricated `{big_y, big_c}` before the legitimate threshold protocol completes.
3. The contract accepts the response with no validation.
4. The user's TEE application receives an attacker-controlled derived key `s`.

The attacker knows `s` and can therefore impersonate the TEE application, decrypt its data, or cause it to derive a key that maps to attacker-controlled material. This is unauthorized access to confidential key derivation output — the core security guarantee of the CKD subsystem is broken by a single participant.

**Allowed impact matched:** *Critical — Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery.*

### Likelihood Explanation

- The attacker must be an attested participant, which requires passing TEE attestation. This is a meaningful barrier but is the exact attacker model the system must defend against ("Byzantine participant strictly below the signing threshold").
- Once a participant, the attack is mechanically simple: watch for `AppPublicKey` CKD requests on-chain and immediately call `respond_ckd` with crafted values.
- Front-running the legitimate coordinator is feasible because NEAR block times are ~1 second and the threshold protocol takes multiple rounds of off-chain communication before the coordinator can submit.
- The `AppPublicKey` variant is the legacy/default variant and is actively used in production. [3](#0-2) 

### Recommendation

Apply the same principle used for `AppPublicKeyPV`: enforce a cryptographic bound on the response even for `AppPublicKey` requests. Options:

1. **Require `AppPublicKeyPV` for all new requests.** Deprecate `AppPublicKey` so that every CKD response is publicly verifiable on-chain via `ckd_output_check`.
2. **Store a commitment to the expected response.** During the threshold CKD protocol, have participants vote on the response hash before any single node can submit it, analogous to how `vote_pk` works for key generation.
3. **Restrict `respond_ckd` to the designated coordinator.** Record the coordinator identity at request time and enforce it in `respond_ckd`, so a non-coordinator participant cannot submit a response at all.

The root cause mirrors the SeaDrop finding exactly: an "allowed" role (attested participant) has unconstrained control over a critical output parameter (`big_y`, `big_c`) with no on-chain enforcement of the constraints that the threshold protocol is supposed to guarantee.

### Proof of Concept

```
1. Attacker is an attested participant (passes TEE check).

2. User calls request_app_private_key({
       app_public_key: AppPublicKey(A),   // A = a·G1, a is user's secret
       derivation_path: "mykey",
       domain_id: 2
   }) with 1 yoctoNEAR deposit.

3. Attacker observes the CKDRequest on-chain (A is public).

4. Attacker picks s = 0xdeadbeef... (any scalar they want the user's key to be).
   Attacker picks random y'.
   Attacker computes:
       big_y = y' · G1
       big_c = s·G1 + y' · A      (note: s here is treated as a scalar, big_c is a G1 point)

   Wait — more precisely, the CKD output is in G1:
       big_c = S + y'·A  where S = s·G1 is the attacker's chosen G1 point
   User decrypts: big_c − a·big_y = s·G1 + y'·a·G1 − a·y'·G1 = s·G1
   So the user's derived key (as a G1 element) is s·G1, and the attacker knows s.

5. Attacker calls respond_ckd(request, {big_y, big_c}) before the legitimate
   threshold protocol finishes.

6. Contract executes:
       match &request.app_public_key {
           AppPublicKey(_) => {}   // no check — passes immediately
       }
       resolve_yields_for(...)     // yields the fabricated response to the user

7. User's TEE app receives {big_y: y'·G1, big_c: s·G1 + y'·A}.
   App computes: big_c − a·big_y = s·G1.
   App uses s·G1 as its derived secret — a value the attacker chose and knows.
``` [4](#0-3)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
