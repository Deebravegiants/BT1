### Title
`setWithdrawalDelayBlocks` Takes Effect Immediately on Already-Queued Withdrawal Requests, Temporarily Freezing User Funds - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager.setWithdrawalDelayBlocks` can be called by the Manager role at any time and takes effect immediately. Because `withdrawalDelayBlocks` is read at completion time (not recorded at request time), increasing the delay retroactively extends the lock period for all already-queued withdrawal requests, temporarily freezing users' rsETH that has already been transferred to the contract.

### Finding Description

When a user calls `initiateWithdrawal`, their rsETH is immediately transferred to the `LRTWithdrawalManager` contract and a `WithdrawalRequest` is stored with `withdrawalStartBlock = block.number`. [1](#0-0) 

The withdrawal request records only `withdrawalStartBlock`; it does **not** snapshot the `withdrawalDelayBlocks` value at request time. [2](#0-1) 

Both `_processWithdrawalCompletion` (user-facing `completeWithdrawal`) and `_unlockWithdrawalRequests` (operator-facing `unlockQueue`) read the **current** `withdrawalDelayBlocks` at execution time: [3](#0-2) [4](#0-3) 

The Manager can update `withdrawalDelayBlocks` to any value up to 16 days / 12 seconds at any time: [5](#0-4) 

There is no mechanism to apply the new delay only to future requests. The change is global and retroactive.

### Impact Explanation

A user who calls `initiateWithdrawal` expecting an 8-day delay (the default, set at initialization) has their rsETH transferred to the contract immediately. If the Manager subsequently increases `withdrawalDelayBlocks` to the maximum of 16 days / 12 seconds, the user's `completeWithdrawal` call will revert with `WithdrawalDelayNotPassed` for up to 8 additional days beyond what was expected. The user's rsETH is already held by the contract and cannot be recovered until the new delay passes. This constitutes a **temporary freezing of funds** (Medium severity).

### Likelihood Explanation

The Manager role is an operational role expected to adjust protocol parameters in normal operation. Increasing the withdrawal delay is a plausible security response (e.g., to a suspected exploit or oracle anomaly). The Manager does not need to act maliciously — any legitimate increase to `withdrawalDelayBlocks` retroactively extends the lock on all pending requests. The default is 8 days and the cap is 16 days, so the maximum additional freeze is 8 days.

### Recommendation

Snapshot `withdrawalDelayBlocks` into each `WithdrawalRequest` at the time `initiateWithdrawal` is called, and use the stored value (rather than the current global value) when checking whether the delay has passed in both `_processWithdrawalCompletion` and `_unlockWithdrawalRequests`. This ensures parameter changes only affect future requests, not already-queued ones.

### Proof of Concept

1. Default `withdrawalDelayBlocks = 8 days / 12 seconds = 57,600 blocks`.
2. User calls `initiateWithdrawal(stETH, 1e18, "")` at block `N`. Their 1 rsETH is transferred to the contract. `withdrawalRequests[id].withdrawalStartBlock = N`.
3. Manager calls `setWithdrawalDelayBlocks(115_200)` (16 days / 12 seconds) — a legitimate security action.
4. At block `N + 57,600` (8 days later), user calls `completeWithdrawal(stETH, "")`.
5. The check `block.number < request.withdrawalStartBlock + withdrawalDelayBlocks` evaluates as `N + 57,600 < N + 115,200` → **true** → reverts with `WithdrawalDelayNotPassed`.
6. The user's rsETH remains locked in the contract for an additional 8 days beyond their original expectation, with no recourse. [6](#0-5) [7](#0-6)
