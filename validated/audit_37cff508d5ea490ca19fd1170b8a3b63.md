### Title
Multiple Sub-Transactions Can Double-Claim the Same Deposit Refund, Violating ADA Preservation - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era's batch transaction model, `getConsumedDijkstraValue` computes the total consumed value for a top-level transaction by summing the consumed values of every sub-transaction independently, each evaluated against the **original** (pre-batch) cert state. If two sub-transactions both include `UnRegDepositTxCert` for the same staking credential, the deposit refund `R` is counted twice in the consumed value. Simultaneously, the `SUBCERT` rule lacks a check preventing a second sub-transaction from deregistering an already-deregistered credential. Together, these two gaps allow an attacker to extract `R` ADA from the deposit pot that was never deposited twice, violating the preservation-of-value invariant. The developers have acknowledged this as an open bug via a skipped test marked `xit "Multiple subtransactions cannot get the same refund"` with `error "TODO: predicate failure not yet implemented"`.

---

### Finding Description

**Root cause 1 — `getConsumedDijkstraValue` uses the original cert state for every sub-transaction**

`getConsumedDijkstraValue` in `UTxO.hs` folds over all sub-transactions and calls `getConsumedValue` for each one, passing the same `lookupStakingDeposit` closure derived from the **original** cert state:

```haskell
subTransactionsConsumedValue topTxBody =
  foldMap'
    (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
    (topTxBody ^. subTransactionsTxBodyL)
``` [1](#0-0) 

`getConsumedMaryValue` calls `getTotalRefundsTxBody`, which calls `lookupStakingDeposit` for every `UnRegDepositTxCert` in each sub-transaction body. Because the same original cert state is used for every sub-transaction, if two sub-transactions both contain `UnRegDepositTxCert cred R`, `lookupStakingDeposit cred` returns `R` for both, and the total consumed value includes `2*R`. [2](#0-1) 

**Root cause 2 — `validateValueNotConservedUTxO` is called with the original cert state**

In `dijkstraLedgerTransition`, the `UTXOW` rule (which contains `UTXO` and therefore `validateValueNotConservedUTxO`) is invoked with `lsCertState ledgerState` — the cert state **before** any sub-transactions ran:

```haskell
-- Call UTXOW with DijkstraUtxoEnv, passing the original UTxO and original certState
utxoStateFinal <-
  trans @(EraRule "UTXOW" era) $
    TRC
      ( DijkstraUtxoEnv slot pp (lsCertState ledgerState) originalUtxo
      , utxoStateBeforeUtxow
      , stAnnTx
      )
``` [3](#0-2) 

Inside `dijkstraUtxoTransition`, the comment confirms this is intentional but the consequence is that the balance check sees the double-counted refund:

```haskell
-- this is the original Accounts, before any transactions were applied
let accounts = certState ^. certDStateL . accountsL
...
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [4](#0-3) 

**Root cause 3 — `SUBCERT` does not prevent double-deregistration across sub-transactions**

Sub-transactions are processed sequentially in `SUBLEDGERS` before `UTXOW` runs. Sub-tx 1 deregisters `cred` and removes deposit `R`. Sub-tx 2 then attempts to deregister the same `cred` again. The `SUBCERT` rule currently lacks a guard that rejects this second deregistration, as acknowledged by the skipped test:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [5](#0-4) 

**Attack path**

1. Attacker registers staking credential `cred`, paying deposit `R` into the deposit pot.
2. Attacker constructs a batch transaction:
   - Sub-tx 1: inputs worth `V1`, cert `UnRegDepositTxCert cred R`
   - Sub-tx 2: inputs worth `V2`, cert `UnRegDepositTxCert cred R`
   - Top-level tx: outputs worth `V1 + V2 + 2*R − fees`
3. `SUBLEDGERS` processes sub-tx 1: `cred` deregistered, deposit `R` removed from pot.
4. `SUBLEDGERS` processes sub-tx 2: `cred` is already deregistered, but the missing guard allows it to proceed; deposit `R` is removed again (pot goes to `−R`).
5. `validateValueNotConservedUTxO` runs with the original cert state:
   - consumed = `V1 + V2 + 2*R` (double-counted)
   - produced = `(V1 + V2 + 2*R − fees) + fees = V1 + V2 + 2*R`
   - Check passes.
6. Attacker's outputs contain `2*R − fees` more ADA than their inputs, having only ever deposited `R`. Net gain: `R` stolen from the deposit pot.

---

### Impact Explanation

This is a **Critical** impact: **direct loss of ADA through an invalid ledger state transition**. The deposit pot is reduced by `2*R` when only `R` was ever deposited, and the attacker's UTxO outputs are inflated by the difference. The preservation-of-value invariant — the foundational correctness property of the Cardano ledger — is violated. Any honest node applying this batch transaction would accept an invalid state, and the deposit pot balance would underflow.

---

### Likelihood Explanation

The Dijkstra era is not yet deployed on mainnet, but the vulnerability is reachable by any unprivileged transaction author who can construct a batch transaction. No special keys, governance majority, or privileged access is required. The attacker only needs to have previously registered a staking credential (a routine, permissionless operation). The attack is deterministic and reproducible.

---

### Recommendation

Two complementary fixes are needed:

1. **In `SUBCERT` / `SUBDELEG`**: Add a guard that rejects `UnRegDepositTxCert cred _` if `cred` is no longer present in the current (post-previous-sub-tx) cert state. This is the fix the developers have already identified as needed (`TODO: predicate failure not yet implemented`).

2. **In `getConsumedDijkstraValue`**: Track which credentials have already been claimed for a refund within the batch and deduplicate, or compute the consumed value using the cert state as it evolves through sub-transactions rather than always using the original snapshot. This prevents the balance check from being satisfied with a double-counted refund even if the `SUBCERT` guard is bypassed.

---

### Proof of Concept

The developers' own skipped test in `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` (lines 53–75) directly encodes this scenario. Enabling it (changing `xit` to `it`) and running it against the current codebase would demonstrate that `submitFailingTx` does not observe the expected failure — i.e., the transaction currently succeeds when it should be rejected.

The structural proof follows from the code:

- `getConsumedDijkstraValue` (`UTxO.hs:88–91`) sums sub-tx refunds using the original cert state, producing `2*R` for two sub-txs claiming the same credential.
- `validateValueNotConservedUTxO` (`Utxo.hs:380–381`) is satisfied because consumed `= V1 + V2 + 2*R =` produced.
- `dijkstraSubLedgersTransition` (`SubLedger.hs:233–294`) processes sub-txs sequentially with no cross-sub-tx deduplication of deregistration claims.
- The `SUBCERT` rule (via `SUBCERTS`, `SubCerts.hs:114–136`) processes each certificate against the current cert state but currently lacks a predicate failure for re-deregistration of an already-deregistered credential, as confirmed by `CertSpec.hs:53–75`.   forall era l.
  ( DijkstraEraTxBody era
  , EraUTxO era
  , Value era ~ MaryValue
  , STxLevel l era ~ STxBothLevels l era
  ) =>
  PParams era ->
  (Credential Staking -> Maybe Coin) ->
  (Credential DRepRole -> Maybe Coin) ->
  UTxO era ->
  TxBody l era ->
  Value era
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody

  consumedValue <> MaryValue mempty mintedMultiAsset

dijkstraSubCertsTransition ::
  forall era.
  ( ConwayEraGov era
  , ConwayEraCertState era
  , EraRule "SUBCERTS" era ~ SUBCERTS era
  , EraRule "SUBCERT" era ~ SUBCERT era
  , Embed (EraRule "SUBCERT" era) (SUBCERTS era)
  ) =>
  TransitionRule (EraRule "SUBCERTS" era)
dijkstraSubCertsTransition = do
  TRC
    ( env@(SubCertsEnv _tx pp currentEpoch committee committeeProposals)
      , certState
      , certificates
      ) <-
    judgmentContext
  case certificates of
    Empty -> pure certState
    gamma :|> txCert -> do
      certStateRest <-
        trans @(SUBCERTS era) $ TRC (env, certState, gamma)
      trans @(EraRule "SUBCERT" era) $
        TRC (Conway.CertEnv pp currentEpoch committee committeeProposals, certStateRest, txCert)
