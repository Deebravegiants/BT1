### Title
Unchecked `farmer_reward_address_override` in `RespondSignatures` Allows Connected Harvester to Redirect Farmer Block Reward â€” (`chia/farmer/farmer_api.py`)

### Summary

A harvester connected to a farmer can unconditionally redirect the farmer's block reward to an attacker-controlled puzzle hash by setting `farmer_reward_address_override` in a `RespondSignatures` message. The farmer accepts this override without any authorization check, fee-quality enforcement, or comparison against its own configured `farmer_target`. The resulting `DeclareProofOfSpace` carries the attacker's puzzle hash as `farmer_puzzle_hash`, and the full node mints the farmer reward coin to that address.

---

### Finding Description

**Entry point â€” `NewProofOfSpace` (harvester â†’ farmer)**

When a harvester wins a block, it sends `NewProofOfSpace` to the farmer. [1](#0-0) 

If `farmer_reward_address_override` is non-`None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, which **only logs** the event and checks a fee-quality heuristic. It never blocks the override, even when the fee quality check fails. [2](#0-1) 

**The actual override is applied in `_process_respond_signatures` (farmer â†’ full node)**

After the farmer sends `RequestSignatures` to the harvester, the harvester replies with `RespondSignatures`, which also carries `farmer_reward_address_override`: [3](#0-2) 

Inside `_process_respond_signatures`, the farmer unconditionally replaces its own `farmer_target` with whatever the harvester supplied:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [4](#0-3) 

This address is then placed directly into `DeclareProofOfSpace.farmer_puzzle_hash`: [5](#0-4) 

**No cross-message consistency check exists.** The harvester can send `NewProofOfSpace` with `farmer_reward_address_override = None` (bypassing even the logging in `notify_farmer_reward_taken_by_harvester_as_fee`) and then inject the attacker address only in `RespondSignatures`. The farmer never compares the two values.

**Full node accepts any `farmer_puzzle_hash` without validation**

The full node's `declare_proof_of_space` handler reads the puzzle hash directly from the request:

```python
farmer_ph = request.farmer_puzzle_hash
``` [6](#0-5) 

It then passes `farmer_ph` into `create_unfinished_block`, which embeds it in `FoliageBlockData`. The full node never checks that this puzzle hash corresponds to any key the farmer controls. [7](#0-6) 

The subsequent `signed_values` handler verifies the farmer's signature over `foliage_block_data.get_hash()`, but by that point the farmer has already committed to the attacker's address inside the foliage â€” the signature proves the farmer signed the block, not that the farmer authorized the reward address. [8](#0-7) 

---

### Impact Explanation

Every block won by the farmer while a malicious harvester is connected will have its farmer reward coin minted to the attacker's puzzle hash instead of the farmer's configured address. This is a direct, permanent financial loss: the farmer reward (currently 0.25 XCH per block) is irreversibly diverted. Coin records are corrupted in the sense that reward coins are minted to an unauthorized puzzle hash, violating the invariant that only the farmer controls its reward address.

---

### Likelihood Explanation

The precondition â€” a harvester connected to the farmer â€” is the normal operating state for any farmer using a third-party harvester (the explicit use case of CHIP-22). The attack requires no leaked keys, no broken cryptography, and no external infrastructure. Any third-party harvester operator can silently redirect rewards on every block won, with no action required from the farmer beyond the existing connection.

---

### Recommendation

1. **Enforce the override at the `NewProofOfSpace` boundary.** Store the `farmer_reward_address_override` from `NewProofOfSpace` alongside the proof in `self.farmer.proofs_of_space`, and in `_process_respond_signatures` reject (or ignore) any `RespondSignatures.farmer_reward_address_override` that does not match the stored value.

2. **Block the override when fee quality fails.** `notify_farmer_reward_taken_by_harvester_as_fee` currently only logs a warning when the fee threshold is violated. It should return a boolean indicating whether the override is legitimate, and `new_proof_of_space` should abort the block-making flow (not send `RequestSignatures`) when the check fails.

3. **Optionally, require farmer-side configuration to opt in.** The farmer config could list allowed harvester puzzle hashes or a maximum fee rate; any override outside those bounds would be rejected before `RequestSignatures` is sent.

---

# Step 2: In request_signatures handler, respond with attacker address
attacker_ph = bytes32(b"\xde\xad" * 16)
respond_sig = harvester_protocol.RespondSignatures(
    plot_identifier="attacker.plot",
    challenge_hash=valid_challenge_hash,
    sp_hash=valid_sp_hash,
    local_pk=local_pk,
    farmer_pk=farmer_pk,
    message_signatures=valid_sigs,

# â†’ DeclareProofOfSpace.farmer_puzzle_hash == attacker_ph

```

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
