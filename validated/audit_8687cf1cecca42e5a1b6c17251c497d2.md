### Title
Unvalidated Negative `m` in `RequireMOf` Native Scripts Allows Signature Requirement Bypass — (File: `eras/allegra/impl/src/Cardano/Ledger/Allegra/Scripts.hs`)

---

### Summary

The `m` threshold parameter in `RequireMOf`/`TimelockMOf` native scripts is decoded from CBOR as a signed `Int` with no non-negativity check. When `m ≤ 0`, the evaluator immediately returns `True` regardless of witnesses. An unprivileged transaction author can craft a script with `m = -1` that always validates, bypassing any embedded `RequireSignature` constraints. This is the direct analog of the external report's "insufficient validation of signature parameters at initialization time."

---

### Finding Description

The `TimelockRaw` and `MultiSigRaw` data types store the threshold as `!Int` (a signed integer): [1](#0-0) 

The CBOR decoder for `TimelockRaw` reads `m` with no bounds check: [2](#0-1) 

The same pattern holds for `MultiSigRaw`: [3](#0-2) 

And for `DijkstraNativeScriptRaw`: [4](#0-3) 

The evaluator's `isValidMOf` short-circuits to `True` the moment `n ≤ 0`: [5](#0-4) 

The same logic appears in `evalMultiSig`: [6](#0-5) 

The codebase even documents this behavior explicitly: [7](#0-6) 

No validation gate exists anywhere between CBOR deserialization and script evaluation to reject a negative `m`. The `validateNativeScript` / `validateTimelock` path in the UTXOW rule calls `evalTimelock` directly: [8](#0-7) 

And `validateFailedNativeScripts` in the UTXOW STS rule passes the result of `validateNativeScript` straight to the predicate check: [9](#0-8)

### Impact Explanation

**Medium — Attacker-controlled scripts exceed intended validation limits.**

An attacker can publish a minting policy or spending credential whose CBOR encodes `RequireMOf (-1) [RequireSignature <keyHash>]`. The script hash is computed over the raw bytes, so the policy ID is unique and valid. The script always evaluates to `True` at execution time, meaning:

- **Minting policy**: The attacker can mint an unlimited supply of tokens under a policy that visually contains a `RequireSignature` clause. Downstream tooling or users inspecting the script may believe the policy is key-gated, while the attacker can mint freely without any signature.
- **Spending credential**: Any UTxO locked by such a script can be spent by anyone without providing the embedded key's signature.

This directly matches the external report's impact: a parameter that should constrain cryptographic authorization is left unvalidated, allowing crafted objects that bypass the intended authorization check.

---

### Likelihood Explanation

**Medium.** Crafting the CBOR is trivial — the attacker simply encodes a negative integer for the `m` field (CBOR major type 1, negative integer). No privileged access, key material, or consensus participation is required. The attack is fully reachable by an unprivileged transaction sender. The main limiting factor is that victims must interact with the attacker's script (e.g., by sending funds to an address derived from it, or by trusting tokens minted under it).

---

### Recommendation

Add a non-negativity check for `m` at deserialization time in all three decoders (`MultiSigRaw`, `TimelockRaw`, `DijkstraNativeScriptRaw`). For example, after decoding `m <- decCBOR`, assert `m >= 0` and fail with a `DecoderErrorCustom` if not. This mirrors the recommendation in the external report to validate parameters as soon as the object is initialized, preventing unvalidated values from propagating into evaluation. ### Proof of Concept

1. Construct a `TimelockRaw` value `TimelockMOf (-1) [TimelockSignature <anyKeyHash>]` and serialize it to CBOR: `[3, -1, [[0, <keyHash>]]]`.
2. Compute `scriptHash = hashScript (fromNativeScript script)`.
3. Create a transaction output locked by `scriptHash` (or a minting policy using `scriptHash`).
4. In a subsequent transaction, reference the script in the witness set **without** providing a `WitVKey` for `<anyKeyHash>`.
5. Submit the transaction. `evalTimelock` calls `isValidMOf (-1) [...]`, which immediately returns `True` because `-1 ≤ 0`, and the script validates successfully despite the missing signature.

The entry path is: serialized transaction → CBOR decoder (`decCBOR` for `TimelockRaw`) → `validateNativeScript` → `evalTimelock` → `isValidMOf` → `True`. [13](#0-12) [14](#0-13)

evalTimelock vhks (ValidityInterval txStart txExp) = go
  where
    -- The important part of this validator is that it will stop as soon as it reaches the
    -- required number of valid scripts
    isValidMOf n SSeq.Empty = n <= 0
    isValidMOf n (ts SSeq.:<| tss) =
      n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
    go = \case
      RequireTimeStart lockStart -> lockStart `lteNegInfty` txStart
      RequireTimeExpire lockExp -> txExp `ltePosInfty` lockExp
      RequireSignature hash -> hash `Set.member` vhks
      RequireAllOf xs -> all go xs
      RequireAnyOf xs -> any go xs
      RequireMOf m xs -> isValidMOf m xs
      _ -> error "Impossible: All NativeScripts should have been accounted for"

validateVerifiedWits :: EraTx era => Tx l era -> Test (ShelleyUtxowPredFailure era)
validateVerifiedWits tx =
  failureOnNonEmpty (failed <> failedBootstrap) InvalidWitnessesUTXOW
  where
    txBody = tx ^. bodyTxL
    txBodyHash = extractHash (hashAnnotated txBody)
    wvkKey (WitVKey k _) = k
    failed =
      wvkKey
        <$> filter
          (not . verifyWitVKey txBodyHash)
          (Set.toList $ tx ^. witsTxL . addrTxWitsL)
    failedBootstrap =
      bwKey
        <$> filter
          (not . verifyBootstrapWit txBodyHash)
          (Set.toList $ tx ^. witsTxL . bootAddrTxWitsL)
