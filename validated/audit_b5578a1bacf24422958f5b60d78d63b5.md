### Title
Merkle Leaf Hash Excludes Contract Address, Enabling Cross-Instance Proof Replay for KERNEL Token Claims - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

---

### Summary
`KernelMerkleDistributor._processClaim()` computes the Merkle leaf without binding it to `address(this)`. In a multi-chain deployment where the same Merkle root is set on multiple distributor instances, a user's on-chain proof (revealed by their first claim) is valid on every other instance sharing that root, allowing them to drain KERNEL tokens from each.

---

### Finding Description

In `KernelMerkleDistributor._processClaim()`, the leaf node is constructed as:

```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
``` [1](#0-0) 

The encoding contains only `index`, `account`, and `cumulativeAmount`. It does **not** include `address(this)` (the deploying contract's address) or `block.chainid`. The `isClaimed` guard is stored in per-contract storage (`userClaims[account]`):

```solidity
mapping(address user => UserClaim userClaim) public userClaims;
``` [2](#0-1) 

This guard is therefore **not shared** across different deployed instances. Once a user submits a valid `(index, account, cumulativeAmount, merkleProof)` tuple to one instance, all parameters are permanently public on-chain. Any other `KernelMerkleDistributor` instance whose `currentMerkleRoot` was built from the same off-chain dataset will accept the identical proof, because the leaf hash is identical.

The same structural issue exists in `KernelTop100MerkleDistributor._verifyClaimProof()`:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
``` [3](#0-2) 

---

### Impact Explanation

**Impact: High ‚Äî Theft of unclaimed yield.**

Kelp DAO is a multi-chain protocol. If the off-chain reward system generates one Merkle root and the admin sets it on `KernelMerkleDistributor` instances on two or more chains (e.g., Ethereum mainnet and Arbitrum), a user can:

1. Call `claim()` on chain A ‚Äî proof is now public.
2. Replay the identical `(index, account, cumulativeAmount, merkleProof)` on chain B.
3. Receive KERNEL tokens on chain B despite having already claimed the full allocation.

Each additional instance multiplies the attacker's yield extraction. The stolen asset is KERNEL tokens held by each distributor contract.

---

### Likelihood Explanation

**Likelihood: Low.**

Exploitation requires the same Merkle root to be active on at least two `KernelMerkleDistributor` instances simultaneously. This is a realistic operational pattern for a multi-chain protocol distributing the same reward epoch to users across chains, but it is not guaranteed to occur in every deployment. The window exists whenever the off-chain tooling does not embed the target contract address into each leaf before generating the root.

---

### Recommendation

Include `address(this)` (and optionally `block.chainid`) in the leaf encoding so that a proof generated for one contract instance is cryptographically invalid on any other:

```solidity
// Before (vulnerable)
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));

// After (fixed)
bytes32 node = keccak256(abi.encodePacked(address(this), index, account, cumulativeAmount));
```

Apply the same fix to `KernelTop100MerkleDistributor._verifyClaimProof()`:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(address(this), user, amount));
```

The off-chain Merkle tree generation must be updated to include the target contract address when building leaves.

---

### Proof of Concept

1. Admin deploys `KernelMerkleDistributor` at address `0xAAA` on Ethereum and `0xBBB` on Arbitrum.
2. Off-chain system generates Merkle root `R` from leaves `keccak256(abi.encodePacked(1, alice, 1000e18))` (among others) and calls `setMerkleRoot(R)` on both `0xAAA` and `0xBBB`.
3. Alice calls `claim(1, alice, 1000e18, proof)` on `0xAAA` (Ethereum). Transaction is mined; `proof` is now public.
4. Alice calls `claim(1, alice, 1000e18, proof)` on `0xBBB` (Arbitrum) with the identical arguments.
5. `_processClaim` on `0xBBB` recomputes `node = keccak256(abi.encodePacked(1, alice, 1000e18))` ‚Äî identical to the leaf in `R` ‚Äî and `MerkleProofUpgradeable.verify` returns `true`.
6. `isClaimed(1, alice)` on `0xBBB` returns `false` (fresh storage).
7. Alice receives 1000e18 KERNEL on Arbitrum in addition to what she already received on Ethereum. [4](#0-3)

    function _processClaim(
        uint256 index,
        address account,
        uint256 cumulativeAmount,
        bytes32[] calldata merkleProof
    )
        internal
        returns (uint256)
    {
        UtilLib.checkNonZeroAddress(account);

        if (currentMerkleRoot == bytes32(0)) {
            revert ZeroValueProvided();
        }

        if (index == 0 || index > currentIndex) {
            revert InvalidIndex();
        }

        if (account != msg.sender) {
            revert Unauthorized();
        }

        if (isClaimed(index, account)) {
            revert AlreadyClaimed();
        }

        // Verify the merkle proof
        bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
        if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
            revert InvalidMerkleProof();
        }

        // Calculate the claimable amount
        uint256 claimableAmount = cumulativeAmount - userClaims[account].cumulativeAmount;

        // Ensure there is something to claim
        if (claimableAmount == 0) {
            revert NoTokensToClaim();
        }

        // Update user claim info
        userClaims[account].lastClaimedIndex = index;
        userClaims[account].cumulativeAmount = cumulativeAmount;

        // Calculate the fee and the amount to send
        uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;
        uint256 amountToSend = claimableAmount - fee;

        if (fee > 0) {
            kernel.safeTransfer(protocolTreasury, fee);
        }

        return amountToSend;
    }
```
