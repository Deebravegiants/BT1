### Title
Sub-transaction Plutus Script Execution Costs Excluded from Minimum Fee Calculation â€” (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, the minimum fee validation for a batch transaction only accounts for the **top-level** transaction's Plutus script execution units (`totExunits`). Sub-transactions embedded in the batch can carry their own Plutus scripts with independent `ExUnits`, but those costs are never aggregated into the minimum fee check. An unprivileged transaction author can therefore submit a batch where all expensive Plutus scripts reside in sub-transactions, paying a fee that covers only the top-level (minimal or zero) script costs, while the actual computational work performed by sub-transaction scripts goes uncompensated.

---

### Finding Description

**Vulnerability class**: Fee accounting bypass / resource-limit bypass.

The Dijkstra era introduces nested ("sub") transactions. The top-level UTXO transition rule (`dijkstraUtxoTransition`) enforces the minimum fee with:

```haskell
{- minfee pp txTop utxoâ‚€ â‰¤ txfee txb -}
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
``` [1](#0-0) 

`validateFeeTooSmallUTxO` calls `getMinFeeTxUtxo`, which for `DijkstraEra` is bound to `getConwayMinFeeTxUtxo`:

```haskell
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [2](#0-1) 

`getConwayMinFeeTxUtxo` computes the script-fee component as `txscriptfee(prices, totExunits(tx))`, where `totExunits` sums only the redeemers present in the **top-level** transaction's `TxWits`. Sub-transaction redeemers live in separate `TxWits` structures inside each sub-transaction and are never aggregated.

Additionally, the ExUnits cap check also only covers the top-level transaction:

```haskell
{- totExunits tx â‰¤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [3](#0-2) 

The `SUBUTXO` transition rule (`dijkstraSubUtxoTransition`) contains **no** `validateExUnitsTooBigUTxO` check and **no** fee check: [4](#0-3) 

The mapping function `dijkstraUtxoToDijkstraSubUtxoPredFailure` explicitly marks `FeeTooSmallUTxO` and `ExUnitsTooBigUTxO` as impossible for sub-transactions:

```haskell
FeeTooSmallUTxO _ -> error "Impossible: `FeeTooSmallUTxO` for SUBUTXO"
...
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [5](#0-4) 

Meanwhile, `validateBatchCollateral` confirms that sub-transaction redeemers **are** recognized and scripts **are** executed as part of the batch:

```haskell
hasAnyRedeemers t =
  hasRedeemers t || any hasRedeemers (t ^. bodyTxL . subTransactionsTxBodyL)
``` [6](#0-5) 

So sub-transaction scripts are executed, but their execution cost is not reflected in the minimum fee.

The `dijkstraSubTxProducedValue` function confirms that sub-transactions have no `txfee` field in their produced-value accounting:

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody <> txBody ^. treasuryDonationTxBodyL)
    <> burnedMultiAssets txBody
``` [7](#0-6) 

The top-level `conwayProducedValue` includes `txfee`, but the sub-transaction produced value does not. The value-conservation check (`validateValueNotConservedUTxO`) at the top level aggregates both, meaning the top-level fee must cover the "gap" â€” but the minimum fee floor is computed without sub-transaction script costs, so the floor is systematically too low.

### Impact Explanation

**Medium â€” Attacker-controlled transactions modify fees outside design parameters.**

An attacker can submit a Dijkstra batch transaction where:
- The top-level transaction carries zero or minimal Plutus scripts (low `totExunits`, low script fee).
- Sub-transactions carry arbitrarily expensive Plutus scripts (high `ExUnits`, no per-sub-tx ExUnits cap).

The ledger accepts the transaction as long as `txfee â‰Ą minfee(top-level only)`. The script execution fee that should compensate block producers for running sub-transaction scripts is never enforced. This directly modifies the effective fee paid below the protocol-intended minimum, constituting a fee parameter violation outside design parameters.

---

### Likelihood Explanation

**Medium.** The Dijkstra era is the current development era and is not yet deployed on mainnet. However, the code is present in the repository and will be activated via a hard fork. Any user who can submit a valid Dijkstra-era transaction (an unprivileged transaction author) can exploit this without any special privilege. The attack requires only knowledge of the sub-transaction structure, which is publicly documented in the CDDL spec. [8](#0-7) 

---

### Recommendation

1. **Aggregate sub-transaction ExUnits into the minimum fee calculation.** Override `getMinFeeTxUtxo` for `DijkstraEra` to sum `totExunits` across the top-level transaction and all sub-transactions before computing `txscriptfee`.

2. **Enforce a per-batch ExUnits cap.** Extend `validateExUnitsTooBigUTxO` (or introduce a new `validateBatchExUnitsTooBigUTxO`) to sum ExUnits across the entire batch and compare against `maxTxExUnits pp`.

3. **Add a sub-transaction ExUnits check in `dijkstraSubUtxoTransition`** analogous to the top-level check, or enforce it at the batch level in `dijkstraUtxoTransition`.

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction `txTop` with:
   - One spending input.
   - `txfee = minfee(pp, txTop)` computed with zero top-level redeemers (script fee = 0).
   - One sub-transaction `txSub` containing a Plutus V3 script spending an input, with redeemers specifying `ExUnits { memory = maxBound, steps = maxBound }`.

2. Submit the batch. `dijkstraUtxoTransition` runs:
   - `validateFeeTooSmallUTxO pp txTop originalUtxo` â†’ passes, because `totExunits txTop = 0`, so `minfee = a*size + b` (no script fee component).
   - `validateBatchCollateral` â†’ passes if collateral is provided (required because `txSub` has redeemers).
   - `validateExUnitsTooBigUTxO pp txTop` â†’ passes, because `totExunits txTop = 0`.

3. `dijkstraSubUtxoTransition` runs for `txSub`:
   - No ExUnits check.
   - No fee check.
   - `updateUTxOStateNoFees` is called â€” no fee collected.

4. The batch is accepted. The attacker has executed an arbitrarily expensive Plutus script while paying only the size-based minimum fee, with the script execution fee component entirely bypassed.
