Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` Enables Stale-Price Deposit to Steal Unclaimed Yield — (`contracts/FeeReceiver.sol`, `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access control, allowing any caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at will. Because `LRTOracle.rsETHPrice` is a lazily-updated storage variable, a deposit made immediately after `sendFunds()` uses a stale (pre-reward) price, minting more rsETH than the deposited ETH is worth. When the price is subsequently updated, the attacker's excess rsETH captures yield that belonged to existing holders.

## Finding Description

**Root cause 1 — `sendFunds()` is permissionless:**
`contracts/FeeReceiver.sol` lines 53–58 expose `sendFunds()` as a plain `external` function with no `onlyRole` or any other guard. Any EOA or contract can call it at any time to push the contract's entire ETH balance into `LRTDepositPool`.

**Root cause 2 — TVL is read from live balance, but the mint price is a stale storage slot:**
`getETHDistributionData()` returns `address(this).balance` directly (`LRTDepositPool.sol` L480), so the deposit pool's ETH balance immediately reflects the newly received rewards. However, `getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` (`LRTDepositPool.sol` L520), which is a storage variable last written by a prior `_updateRsETHPrice()` call. The price has not been updated to account for the new rewards, so it is stale (too low).

**Root cause 3 — `updateRSETHPrice()` is public:**
`LRTOracle.sol` L87 exposes `updateRSETHPrice()` as `public whenNotPaused`. Any caller can trigger the price update after depositing.

**Exploit flow (single block or consecutive blocks):**
1. Call `FeeReceiver.sendFunds()` → deposit pool balance increases by R; stored `rsETHPrice` still reflects the pre-reward TVL X/S.
2. Call `LRTDepositPool.depositETH{value: D}(0, "")` → attacker receives `D * S/X` rsETH (computed using stale price X/S instead of the fair post-reward price (X+R)/S).
3. Call `LRTOracle.updateRSETHPrice()` → new price = `(X+R+D) / (S + D*S/X)`.

Attacker's rsETH value after step 3 = `D * (X+R+D)/(X+D)` = `D + D*R/(X+D)`.  
**Profit = `D*R/(X+D)`** — extracted directly from existing holders' unclaimed yield.

**Why the `pricePercentageLimit` guard is insufficient:**
The guard at `LRTOracle.sol` L252–266 only reverts `updateRSETHPrice()` if the price jump exceeds the configured threshold and the caller is not a manager. For typical MEV rewards (small relative to TVL), the jump stays within the limit and the call succeeds. Even if the threshold is breached and the public call reverts, the attacker's rsETH was already minted at the stale price in step 2; when the manager eventually calls `updateRSETHPriceAsManager()`, the attacker's inflated position is already locked in.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders are entitled to the yield accumulated in `FeeReceiver`. The attack siphons a portion of that yield (`D*R/(X+D)`) to the attacker on every execution. The attack is repeatable whenever the `FeeReceiver` holds a non-trivial balance, which is a routine operational state between keeper calls. No funds are permanently frozen and the protocol remains solvent, placing this squarely in the "Theft of unclaimed yield" category.

## Likelihood Explanation
- `sendFunds()` requires zero preconditions and is callable by any address.
- `depositETH()` is the standard user-facing deposit function.
- `updateRSETHPrice()` is public.
- No flash loan is required; the attacker only needs to hold ETH.
- The attack is profitable whenever `FeeReceiver` holds a non-trivial balance, a routine operational state.
- The only partial mitigation (`pricePercentageLimit`) does not prevent minting at the stale price; it only conditionally delays price recognition, which does not protect existing holders.

## Recommendation
1. **Add access control to `sendFunds()`** — restrict it to `MANAGER` or `LRT_OPERATOR` role so only trusted keepers can flush rewards into the deposit pool.
2. **Alternatively, call `_updateRsETHPrice()` atomically inside `receiveFromRewardReceiver()`** — update the stored price before any subsequent deposit can use the new balance.
3. **Deposit-time price freshness check** — reject deposits if `rsETHPrice` was last updated more than N blocks/seconds ago relative to the current TVL, forcing a price refresh before minting.

## Proof of Concept
```solidity
// Foundry fork test
function testSandwichYield() public {
    address attacker = address(0xBEEF);
    vm.deal(attacker, 1 ether);
    vm.deal(address(feeReceiver), 10 ether); // simulate accumulated rewards

    uint256 stalePriceBefore = lrtOracle.rsETHPrice();

    // Step 1: flush rewards — permissionless
    feeReceiver.sendFunds();

    // Step 2: deposit at stale price
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
    uint256 rsethMinted = rsETH.balanceOf(attacker);

    // Step 3: update price — permissionless
    lrtOracle.updateRSETHPrice();
    uint256 newPrice = lrtOracle.rsETHPrice();

    // Attacker's rsETH is worth more than 1 ETH
    uint256 attackerEthValue = rsethMinted * newPrice / 1e18;
    assertGt(attackerEthValue, 1 ether);

    // Attacker received more rsETH than the fair share at the new price
    uint256 fairRseth = 1 ether * 1e18 / newPrice;
    assertGt(rsethMinted, fairRseth);
}
```
The assertions pass because `rsethMinted` was computed using `stalePriceBefore` (lower, pre-reward), while `newPrice` (higher, post-reward) is used to value the position. The difference `rsethMinted - fairRseth` represents yield stolen from existing holders.
