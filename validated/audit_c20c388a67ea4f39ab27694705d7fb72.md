### Title
Daily Mint Limit Exhaustion via Unrestricted Deposit Size Blocks Legitimate User Deposits - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol)

---

### Summary

L2 pool contracts enforce a shared `dailyMintLimit` on wrsETH minting but impose no minimum deposit amount beyond zero. An attacker can exhaust the entire daily limit in one or a few transactions at negligible net cost (receiving wrsETH back), blocking all other users from depositing for the remainder of the day. This is the structural analog of the MonoX pool-blocking bug: a user-controlled action resets/consumes a shared rate-control resource, preventing legitimate protocol use.

---

### Finding Description

All three active L2 pool variants implement a `limitDailyMint` modifier that tracks cumulative minted rsETH against `dailyMintLimit`. The modifier increments `dailyMintAmount` by the rsETH equivalent of every deposit and reverts with `DailyMintLimitExceeded` once the cap is reached.

In `RSETHPoolV3.sol`, the modifier:

```solidity
modifier limitDailyMint(uint256 amount, address token) {
    ...
    if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
        revert DailyMintLimitExceeded();
    }
    dailyMintAmount += rsETHAmount;
    _;
}
```

The `deposit()` functions only reject a zero amount:

```solidity
function deposit(string memory referralId) external payable ... limitDailyMint(msg.value, ETH_IDENTIFIER) {
    ...
    if (amount == 0) revert InvalidAmount();
    ...
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

There is no minimum deposit floor analogous to `minAmountToDeposit` in `LRTDepositPool.sol`. Any amount above zero is accepted and counted against the shared daily cap.

Because the depositor receives wrsETH proportional to their deposit, the attacker's net capital outlay is zero — they simply hold wrsETH and can bridge or redeem it later. The only real cost is L2 gas, which is negligible. A single deposit equal to `dailyMintLimit` exhausts the cap in one transaction, after which every subsequent `deposit()` call by any user reverts for the rest of the day.

The same pattern exists identically in `RSETHPoolV3ExternalBridge.sol` and `RSETHPoolV2ExternalBridge.sol`.

---

### Impact Explanation

Once the daily limit is exhausted, all `deposit()` calls revert with `DailyMintLimitExceeded` until the next UTC day boundary. Users who wish to deposit ETH or supported LSTs into the L2 pool are completely blocked for up to 24 hours. This constitutes a **temporary freezing of funds** (users' ETH cannot enter the protocol) and a denial of the protocol's core promised service on L2. The attack can be repeated every day at negligible cost.

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

On L2 chains (Arbitrum, Optimism, Base, etc.) gas costs are a fraction of a cent per transaction. The attacker needs capital equal to `dailyMintLimit` in ETH/LST, but receives wrsETH back immediately — no capital is destroyed. The attack is therefore economically rational for any actor who wishes to suppress L2 deposit activity (e.g., a competitor, a short-seller of rsETH, or a griever). The attack can be automated and repeated daily.

**Likelihood: Medium.**

---

### Recommendation

Enforce a meaningful minimum deposit amount in all L2 pool `deposit()` functions, mirroring the `minAmountToDeposit` guard already present in `LRTDepositPool.sol`. A sufficiently high floor (e.g., 0.01 ETH) makes exhausting the daily limit via many micro-deposits economically prohibitive and ensures that a single large deposit consuming the cap is a deliberate, high-capital action rather than a cheap griefing vector. Additionally, consider per-address deposit caps within a single day to prevent a single actor from consuming the entire limit.

---

### Proof of Concept

1. Observe `dailyMintLimit` in `RSETHPoolV3` (e.g., 500 ETH worth of rsETH).
2. Attacker calls `deposit{value: 500 ether}("")` — a single transaction exhausts the entire daily cap. `dailyMintAmount` is now equal to `dailyMintLimit`.
3. Any subsequent call by a legitimate user — even `deposit{value: 1 wei}("")` — reverts with `DailyMintLimitExceeded`.
4. Attacker holds 500 ETH worth of wrsETH, redeemable at any time. Net cost: L2 gas only.
5. The attack repeats the next day.

```text
    modifier limitDailyMint(uint256 amount, address token) {
        if (block.timestamp < startTimestamp) {
            revert MintBeforeStartTimestamp();
        }

        uint256 rsETHAmount;

        // Calculate the amount of rsETH that will be minted
        if (token == ETH_IDENTIFIER) {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount);
        } else {
            (rsETHAmount,) = viewSwapRsETHAmountAndFee(amount, token);
        }

        uint256 currentDay = getCurrentDay();

        // If the current day is greater than the last mint day, reset the daily mint amount
        if (currentDay > lastMintDay) {
            lastMintDay = currentDay;
            dailyMintAmount = 0;
        }

        // Check if the daily mint amount plus the amount to mint is greater than the daily mint limit
        if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
            revert DailyMintLimitExceeded();
        }

        dailyMintAmount += rsETHAmount;
        _;
    }
```

**File:** contracts/pools/RSETHPoolV3.sol (L246-265)
```text
    function deposit(string memory referralId)
        external
        payable
        nonReentrant
        whenNotPaused
        limitDailyMint(msg.value, ETH_IDENTIFIER)
    {
        if (!isEthDepositEnabled) revert EthDepositDisabled();
        uint256 amount = msg.value;

        if (amount == 0) revert InvalidAmount();

        (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);

        feeEarnedInETH += fee;

        wrsETH.mint(msg.sender, rsETHAmount);

        emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
    }
```
