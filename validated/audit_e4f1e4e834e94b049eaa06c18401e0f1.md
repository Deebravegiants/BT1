### Title
Expired DRep Self-Resurrects via Vote Submission, Bypassing DRep Activity Enforcement — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

`updateVotingDRepExpiries` refreshes the on-chain expiry of every `DRepVoter` found in a transaction's voting procedures without first checking whether that DRep is already expired. Because the GOV rule's voter-existence check only verifies registration (not active status), an expired DRep can submit a vote transaction, have its expiry silently reset to a future epoch, and have its vote counted in the next ratification pass — bypassing the DRep activity mechanism entirely.

---

### Finding Description

**Root cause — `updateVotingDRepExpiries` does not guard against expired DReps**

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`, `updateVotingDRepExpiries` iterates over every voter in the transaction's `VotingProcedures` and, for each `DRepVoter`, calls `Map.adjust` to overwrite `drepExpiryL` with a freshly computed future epoch: [1](#0-0) 

`Map.adjust` silently succeeds for any key that exists in the map. There is no guard of the form `currentEpoch <= drepExpiry drepState` before the write. An expired DRep (one whose `drepExpiry` is already in the past) is still present in `vsDReps` — it has not been unregistered — so `Map.adjust` finds the key and overwrites the expiry with `computeDRepExpiry drepActivity currentEpoch numDormantEpochs`, a value in the future.

**Why the GOV rule does not block the vote**

The GOV rule's voter-existence check builds `knownDReps` directly from `vsDReps`: [2](#0-1) [3](#0-2) 

`VotersDoNotExist` fires only when the credential is absent from `knownDReps`: [4](#0-3) 

`checkVotersAreValid` for a `DRepVoter` calls only `isDRepVotingAllowed (gasAction gas)`, which checks the *action type*, not the DRep's expiry: [5](#0-4) 

There is no check equivalent to `reCurrentEpoch > drepExpiry drepState` at vote-submission time.

**Contrast with ratification — the check that should have been mirrored**

At ratification, `dRepAcceptedRatio` correctly excludes expired DReps: [6](#0-5) 

But because `updateVotingDRepExpiries` already refreshed the expiry *during the epoch*, by the time ratification runs at the epoch boundary the DRep's stored `drepExpiry` is a future epoch, so the guard `reCurrentEpoch > drepExpiry drepState` evaluates to `False` and the vote is counted.

**End-to-end exploit path**

1. DRep registers; `drepExpiry` is set to `E0 + drepActivity`.
2. DRep goes dormant; `drepExpiry` passes; `reCurrentEpoch > drepExpiry` becomes `True` at ratification — DRep is excluded.
3. In epoch `E1 > E0 + drepActivity`, the expired DRep submits a transaction containing a `DRepVoter` vote on any live governance action.
4. GOV accepts the vote (`VotersDoNotExist` does not fire; `checkVotersAreValid` passes).
5. `updateVotingDRepExpiries` overwrites `drepExpiry` with `E1 + drepActivity - numDormantEpochs` — a future epoch.
6. At the `E1 → E1+1` epoch boundary, `reDRepState` reflects the refreshed expiry; `reCurrentEpoch > drepExpiry` is `False`; the DRep's vote is counted in `dRepAcceptedRatio`.
7. The DRep can repeat this indefinitely, staying perpetually active without ever submitting a `ConwayUpdateDRep` certificate.

---

### Impact Explanation

The DRep activity mechanism is a core governance invariant: DReps that do not participate expire and lose voting power. By self-resurrecting through a vote transaction, an expired DRep can:

- Inflate the "yes" or "no" stake counted in `dRepAcceptedRatio`, tipping ratification of parameter changes, treasury withdrawals, committee updates, or hard-fork initiations.
- Permanently avoid expiry, making the `ppDRepActivityL` parameter ineffective for that DRep.

This matches the allowed impact: **Medium — attacker-controlled transactions modify governance ratification outside design parameters**, with potential escalation to **Critical — unauthorized governance/treasury/protocol-parameter action is enacted** if the DRep holds sufficient delegated stake to swing a vote.

---

### Likelihood Explanation

**Medium.** Any previously registered DRep whose expiry has passed can perform this attack unilaterally. No privileged access, leaked keys, or majority collusion is required. The only precondition is having been a registered DRep at some point. The transaction cost is a normal fee-bearing vote transaction.

---

### Recommendation

In `updateVotingDRepExpiries`, guard the expiry refresh with an expiry check before calling `Map.adjust`:

```haskell
DRepVoter cred ->
  Map.adjust
    (\drepState ->
       if currentEpoch > drepExpiry drepState
         then drepState  -- already expired; do not refresh
         else drepState & drepExpiryL .~ computeDRepExpiry drepActivity currentEpoch numDormantEpochs
    )
    cred
    dreps
```

This mirrors the guard already present in `dRepAcceptedRatio` and ensures that only currently-active DReps can refresh their expiry by voting. Expired DReps must use the explicit `ConwayUpdateDRep` certificate path (which also checks registration but is the intended re-activation mechanism) or re-register.

### Proof of Concept

```
Epoch 0:  DRep D registers; drepExpiry = 0 + drepActivity (e.g., 4) = epoch 4.
Epochs 1–4: D does not vote; no proposals exist; numDormantEpochs increments.
Epoch 5:  reCurrentEpoch (5) > drepExpiry (4) → D is expired at ratification.
           D submits Tx containing VotingProcedures { DRepVoter D → { gaId → VoteYes } }.
           GOV: knownDReps contains D → VotersDoNotExist does not fire.
           CERTS/updateVotingDRepExpiries: Map.adjust overwrites drepExpiry to
             computeDRepExpiry 4 5 4 = (5+4) - 4 = epoch 5  [or higher depending on numDormantEpochs].
Epoch 6:  ratification: reCurrentEpoch (6) > drepExpiry (5)? → False if drepActivity > 1.
           D's VoteYes is counted in dRepAcceptedRatio.
           D repeats each epoch to stay perpetually active.

-- | Update DRep expiry for all DReps that are voting in this transaction. This
-- will execute in mutual-exclusion to the updates to the dormant DRep expiry,
-- because if there are no proposals to vote on, there will be no votes either.
updateVotingDRepExpiries ::
  ( EraTx era
  , ConwayEraTxBody era
  , ConwayEraCertState era
  ) =>
  Tx l era -> EpochNo -> EpochInterval -> CertState era -> CertState era
updateVotingDRepExpiries tx currentEpoch drepActivity certState =
  let numDormantEpochs = certState ^. certVStateL . vsNumDormantEpochsL
      updateVSDReps vsDReps =
        Map.foldlWithKey'
          ( \dreps voter _ -> case voter of
              DRepVoter cred ->
                Map.adjust
                  (drepExpiryL .~ computeDRepExpiry drepActivity currentEpoch numDormantEpochs)
                  cred
                  dreps
              _ -> dreps
          )
          vsDReps
          (unVotingProcedures $ tx ^. bodyTxL . votingProceduresTxBodyL)
   in certState & certVStateL . vsDRepsL %~ updateVSDReps

```haskell
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
