### Title
Missing CKD Output Correctness Check for Legacy `AppPublicKey` Path Allows Single Byzantine Participant to Forge Derived Key Material - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` enforces a cryptographic output-correctness check (`ckd_output_check`) only for the `AppPublicKeyPV` (publicly verifiable) variant. The legacy `AppPublicKey` variant has an empty match arm — no check at all. Because `respond_ckd` requires only a single attested participant to call it (not a threshold), a single Byzantine node below the signing threshold can submit an arbitrary, attacker-controlled `CKDResponse` for any pending `AppPublicKey` request and the contract will accept it unconditionally.

---

### Finding Description

In `respond_ckd` the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` performs a BLS12-381 pairing equation: `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`. This cryptographically proves the response was produced by the MPC network using the correct master key. [2](#0-1) 

For `AppPublicKey`, the arm is empty — the contract calls `resolve_yields_for` immediately with whatever `big_y` and `big_c` the caller supplied, without any cryptographic verification. [3](#0-2) 

The `respond_ckd` entry point only requires `assert_caller_is_signer()` and `assert_caller_is_attested_participant_and_protocol_active()` — a single attested participant, not a threshold of them. [4](#0-3) 

The yield/resume pattern means the **first** valid `respond_ckd` call for a given request resolves it. A malicious attested node can therefore front-run the legitimate leader and inject arbitrary `big_y`/`big_c` values.

The unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` confirms this: it passes `CKDResponse { big_y: [1u8;48], big_c: [2u8;48] }` — obviously invalid curve points — and the contract accepts them without complaint. [5](#0-4) 

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe any pending `request_app_private_key` call that uses the `AppPublicKey` variant.
2. Call `respond_ckd` with attacker-chosen `big_y` and `big_c` before the legitimate leader does.
3. The contract resolves the yield and returns the forged `CKDResponse` to the user.

The user receives a derived key that was not produced by the MPC network's threshold protocol. This is **confidential key derivation output without the required participant authorization** — the threshold requirement is bypassed entirely for this code path.

---

### Likelihood Explanation

The `AppPublicKey` variant is the legacy/default path and is actively used in production (the e2e test `ckd_response__passes_cryptographic_verification` exercises it). [6](#0-5) 

Any single attested MPC participant can exploit this. The attacker does not need to collude with others, break TEE isolation, or perform any off-chain attack — they only need to submit a NEAR transaction to `respond_ckd` before the legitimate leader does.

---

### Recommendation

Apply the same `ckd_output_check` to the `AppPublicKey` arm. For the legacy variant, the check reduces to verifying `e(big_c, g2) = e(big_y, pk1) · e(hash_point, public_key)` where `pk1` is the app's G1 public key. Alternatively, require that the response be submitted only by the elected leader and that it carries a threshold-aggregated proof, consistent with how `respond` handles ECDSA/EdDSA signatures.

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk)`.
2. Attacker (any single attested participant) calls:
   ```
   respond_ckd(
     request = <the pending CKDRequest>,
     response = CKDResponse { big_y: [0u8;48], big_c: [0u8;48] }
   )
   ```
3. Contract executes the `AppPublicKey(_) => {}` arm — no check — and calls `resolve_yields_for`, returning the forged response to the user.
4. User's derived key material (`big_c − a·big_y`) is now attacker-controlled.

The `AppPublicKeyPV` path would reject step 2 with `"CKD output check failed"` because `ckd_output_check` would fail the pairing equation. The `AppPublicKey` path has no such guard. [7](#0-6)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Verify that a CKD response (AppPublicKey variant) is mathematically correct.
#[tokio::test]
#[expect(non_snake_case)]
async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),
        "CKD response failed cryptographic verification"
    );
}
