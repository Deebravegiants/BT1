Looking at the full data flow from the p2p sync `SignedBlockHeader` through to the cende `parent_proposal_commitment`, I need to trace every step carefully.

### Title
Unauthenticated P2P Peer Can Inject Arbitrary `fee_proposal_fri` into `fee_proposals_window`, Corrupting `parent_proposal_commitment` Sent to Cende Pipeline — (`crates/apollo_protobuf/src/converters/header.rs`, `crates/apollo_p2p_sync/src/client/header.rs`, `crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

---

### Summary

The `fee_proposal_fri` field (field 22) in the p2p sync `SignedBlockHeader` protobuf message is explicitly acknowledged as **not part of the block hash** and therefore not covered by any signature. The p2p sync client accepts and stores this field verbatim without any validation. When the consensus orchestrator initializes or catches up via `try_sync`, it reads `fee_proposal_fri` from state-sync storage (which was populated by the p2p sync path) and records it into the in-memory `fee_proposals_window`. The `parent_proposal_commitment` sent to the cende pipeline is then computed as `Poseidon(partial_block_hash, parent_fee_proposal)` using the value from that window. An unauthenticated p2p peer can therefore cause the sequencer to send a wrong `parent_proposal_commitment` to the cende pipeline.

---

### Finding Description

**Step 1 — Unauthenticated injection via p2p sync**

The proto definition explicitly warns:

```
// WARNING: this field is currently not part of the block hash, so the value must be trusted.
optional Uint128 fee_proposal_fri = 22;
``` [1](#0-0) 

The `TryFrom<protobuf::SignedBlockHeader>` converter stores the field verbatim with no validation:

```rust
let fee_proposal_fri = value.fee_proposal_fri.map(|v| GasPrice(u128::from(v)));
``` [2](#0-1) 

**Step 2 — No validation in `parse_data_for_block`**

The p2p sync client's `parse_data_for_block` only checks block number ordering and signature vector length. It does not verify the block hash against header content, does not call `verify_block_signature`, and does not validate `fee_proposal_fri` in any way. [3](#0-2) 

The block signature itself only covers `Poseidon(block_hash, state_diff_commitment)` — `fee_proposal_fri` is outside the signed surface. [4](#0-3) 

**Step 3 — Corrupted value written to Apollo storage**

`write_to_storage` calls `append_header` with the full `BlockHeader` including the attacker-controlled `fee_proposal_fri`, persisting it to the shared Apollo storage. [5](#0-4) 

**Step 4 — Corrupted value enters `fee_proposals_window` via two paths**

*Path A — `initialize_fee_proposals_window`* (called before consensus starts):

```rust
Ok(block) => self.record_fee_proposal(
    block_number,
    block.block_header_without_hash.fee_proposal_fri,
),
``` [6](#0-5) 

*Path B — `try_sync`* (called when the orchestrator is behind state-sync, e.g. after restart):

```rust
self.record_fee_proposal(height, sync_block.block_header_without_hash.fee_proposal_fri);
``` [7](#0-6) 

Both paths read from `state_sync_client.get_block()`, which reads from the same Apollo storage that the p2p sync client wrote to.

**Step 5 — Corrupted `parent_fee_proposal` used to reconstruct `parent_proposal_commitment`**

In `finalize_decision`, the parent block's `fee_proposal_fri` is read from `fee_proposals_window`:

```rust
let parent_fee_proposal = height
    .prev()
    .and_then(|parent_height| self.fee_proposals_window.get(&parent_height).copied())
    .flatten();
``` [8](#0-7) 

This is then used to compute the `parent_proposal_commitment` sent to the cende pipeline:

```rust
parent_proposal_commitment: central_objects
    .parent_proposal_commitment
    .map(|c| proposal_commitment_from(c.partial_block_hash, parent_fee_proposal)),
``` [9](#0-8) 

`proposal_commitment_from` computes `Poseidon(partial.0, fee_proposal_fri)` for V0_14_3+ blocks: [10](#0-9) 

If `parent_fee_proposal` is attacker-controlled, the resulting `parent_proposal_commitment` is wrong.

---

### Impact Explanation

The `parent_proposal_commitment` in the cende blob is the cryptographic link the centralized cende pipeline uses to chain consecutive blocks. A wrong value causes the pipeline to receive a commitment that does not match the one consensus signed over for the parent block. Depending on the cende pipeline's behavior, this either causes the blob to be rejected (liveness failure) or causes the pipeline to anchor to the wrong L1 blob (integrity failure). Either outcome is Critical: the sequencer either stalls or writes incorrect data to L1.

Additionally, the corrupted `fee_proposals_window` skews `compute_fee_actual` (the sliding-window median), causing the node to compute a wrong `fee_actual`. This makes the validator reject valid proposals or accept out-of-bounds proposals, breaking consensus liveness.

---

### Likelihood Explanation

The attack is reachable whenever the sequencer node catches up via p2p sync — specifically:
- On restart, when `initialize_fee_proposals_window` reads the window from state-sync storage that was populated by p2p sync.
- During live operation, when `try_sync` is triggered because the orchestrator falls behind state-sync.

An unauthenticated p2p peer only needs to be connected to the syncing node and respond to header queries with a `SignedBlockHeader` carrying a spoofed `fee_proposal_fri`. No validator, operator, or privileged access is required.

---

### Recommendation

1. **Validate `fee_proposal_fri` against the block hash**: Either include `fee_proposal_fri` in the block hash computation (making it part of the signed surface), or derive it independently from the `fee_proposals_window` of already-validated blocks rather than trusting the p2p-supplied value.

2. **Cross-check on `initialize_fee_proposals_window` / `try_sync`**: When reading `fee_proposal_fri` from state-sync storage that was populated by p2p sync, verify it is within the expected bounds derived from the preceding window (using `fee_proposal_bounds`), or refuse to use it for `parent_proposal_commitment` reconstruction.

3. **Remove the "must be trusted" design gap**: The proto comment acknowledges the field is outside the signed surface. This should be treated as a known vulnerability to fix, not a permanent design choice.

---

### Proof of Concept

```rust
// Attacker sends a SignedBlockHeader for block N with fee_proposal_fri = Some(1)
// when the honest value is Some(1_000_000_000_000).
let honest_fee = GasPrice(1_000_000_000_000);
let attacker_fee = GasPrice(1);
let partial = PartialBlockHash(Felt::from(42u64));

let honest_commitment = proposal_commitment_from(partial, Some(honest_fee));
let attacker_commitment = proposal_commitment_from(partial, Some(attacker_fee));

// The two commitments are different Felt values.
assert_ne!(honest_commitment, attacker_commitment);

// The cende pipeline receives attacker_commitment as parent_proposal_commitment
// instead of honest_commitment, anchoring to the wrong L1 blob.
```
