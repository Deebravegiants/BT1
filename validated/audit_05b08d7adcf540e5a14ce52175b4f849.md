Audit Report

## Title
ETH Deposit Limit Check Missing `amount` in Comparison, Allowing Deposits Beyond Configured Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit()` applies an asymmetric check: for LST assets it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it omits `amount` and only tests `totalAssetDeposits > depositLimit`. Any unprivileged depositor can bypass the ETH deposit cap in a single transaction, causing the protocol to accept and mint rsETH for ETH amounts that exceed the admin-configured limit.

## Finding Description
The bug is confirmed at [1](#0-0)  — the ETH branch at line 679 returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without including `amount`, while the LST branch at line 681 correctly uses `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`.

The public entry point is `depositETH()` at [2](#0-1)  which is `external payable` with no access control beyond `whenNotPaused` and `onlySupportedAsset`. It calls `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected)`, which internally calls `_checkIfDepositAmountExceedesCurrentLimit`. Because `amount` (`msg.value`) is never added to `totalAssetDeposits` in the ETH branch, the check passes as long as the current total has not already exceeded the cap — regardless of how large the incoming deposit is.

The view helper `getAssetCurrentLimit()` at [3](#0-2)  correctly computes remaining room as `depositLimit - totalAssetDeposits`, but this value is never enforced in the ETH deposit path.

## Impact Explanation
The deposit cap is a protocol safety invariant. Bypassing it causes the contract to accept ETH and mint rsETH beyond the configured limit, directly violating the protocol's promised behavior. This concretely matches **Low — Contract fails to deliver promised returns**: the contract guarantees enforcement of `depositLimitByAsset` for ETH but silently fails to do so. The claim's assertion of Medium (Temporary freezing of funds) via downstream EigenLayer capacity exhaustion is plausible but depends on external protocol behavior and is not concretely proven by the submitted PoC.

## Likelihood Explanation
The exploit requires no special role, no front-running, and no external dependency. Any depositor who observes that `getTotalAssetDeposits(ETH_TOKEN)` is below the cap can send a single `depositETH` transaction with `msg.value` exceeding the remaining room. The condition is easily observable on-chain and repeatable.

## Recommendation
Add `amount` to the ETH branch to match the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999 ether > 1000 ether` → `false` → deposit proceeds.
5. Protocol now holds `1499 ether` of ETH with rsETH minted for the full `500 ether`, 49.9% above the configured cap.

A Foundry fuzz test parameterizing `amount` over `[1, type(uint128).max]` with `totalAssetDeposits` initialized just below `depositLimit` would confirm the bypass for all values of `amount > 0`.

    function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
        }
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
