### Title
Unbounded `entries` Map in `ProposedUpdates::propose()` Allows Byzantine Participant to Permanently Block Contract Upgrade Governance - (`File: crates/contract/src/update.rs`)

---

### Summary

`ProposedUpdates` maintains two parallel maps: `vote_by_participant` (implicitly bounded — one entry per participant) and `entries` (completely unbounded — no cap on proposals). A single Byzantine participant below the signing threshold can call `propose_update` repeatedly, flooding `entries`. When legitimate participants later reach threshold on a valid proposal, `do_update` calls `self.entries.clear()`, which iterates over every accumulated entry and exhausts gas, permanently blocking the governance upgrade path.

---

### Finding Description

`ProposedUpdates` in `crates/contract/src/update.rs` holds two maps:

```rust
pub struct ProposedUpdates {
    pub(super) vote_by_participant: IterableMap<AccountId, UpdateId>,  // bounded: 1 per participant
    pub(super) entries: IterableMap<UpdateId, UpdateEntry>,            // UNBOUNDED: no cap
    pub(super) id: UpdateId,
}
```

`vote_by_participant` is naturally bounded because each participant can hold at most one vote at a time. `entries`, however, has no cap. `propose_update` in `lib.rs` inserts into `entries` unconditionally:

```rust
pub fn propose(&mut self, update: Update) -> UpdateId {
    let bytes_used = bytes_used(&update);
    let id = self.id.generate();
    self.entries.insert(id, UpdateEntry { update, bytes_used }); // no cap check
    id
}
```

When threshold is reached and `do_update` is called, it clears **all** entries:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    self.entries.clear();           // O(N) over ALL accumulated entries
    self.vote_by_participant.clear();
    // ...
}
```

If `entries` has been flooded with thousands of proposals, `entries.clear()` exhausts the 300 TGas budget of `vote_update`, causing the transaction to fail. Because the failure is transactional, the bloated `entries` map persists, and every subsequent `vote_update` call that reaches threshold will also fail. The governance upgrade mechanism is permanently bricked.

---

### Impact Explanation

A Byzantine participant below the signing threshold permanently blocks the contract upgrade governance path (`propose_update` → `vote_update` → `do_update`). This prevents security patches, parameter changes, and config updates from ever being applied. If a critical vulnerability is discovered in the running contract, the inability to upgrade it could lead to fund loss or unauthorized signing — escalating the effective impact beyond a simple governance DoS.

This matches the allowed impact: **Medium — contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

---

### Likelihood Explanation

- `propose_update` is callable by any current participant (enforced by `voter_or_panic()`).
- A single Byzantine participant below threshold is sufficient — no collusion required.
- The deposit per config proposal is small (storage cost for config bytes + a fixed overhead of `128 * size_of::<AccountId>()` bytes). Flooding with thousands of config proposals costs on the order of tens of NEAR — economically feasible.
- The attack is permanent: once `entries` is large enough to exhaust gas in `entries.clear()`, no upgrade can ever succeed without a contract migration that bypasses the existing state.

---

### Recommendation

Add a cap on the number of concurrent proposals in `ProposedUpdates::propose()`, analogous to how `push_pending_yield` caps the fan-out queue at `MAX_PENDING_REQUEST_FAN_OUT`:

```rust
const MAX_PROPOSED_UPDATES: usize = 32; // or a config-driven value

pub fn propose(&mut self, update: Update) -> UpdateId {
    if self.entries.len() >= MAX_PROPOSED_UPDATES {
        env::panic_str("ProposedUpdates: maximum number of concurrent proposals reached");
    }
    let bytes_used = bytes_used(&update);
    let id = self.id.generate();
    self.entries.insert(id, UpdateEntry { update, bytes_used });
    id
}
```

Alternatively, restrict each participant to at most one outstanding proposal at a time (mirroring the one-vote-per-participant invariant already enforced on `vote_by_participant`).

---

### Proof of Concept

1. Contract is in `Running` state with N participants and threshold T.
2. Malicious participant P (one of the N, below threshold) calls `propose_update` with a minimal config payload K times, paying the small per-proposal deposit each time. Each call inserts a new entry into `entries`.
3. Legitimate participants agree on a valid upgrade and call `vote_update` until T votes are collected.
4. On the T-th call, `vote_update` calls `do_update`, which calls `self.entries.clear()` over K + 1 entries. With K large enough (empirically, a few thousand config entries), this exhausts the 300 TGas budget.
5. The transaction reverts. `entries` still contains K + 1 entries. Every future `vote_update` that reaches threshold will also revert. The upgrade path is permanently blocked.

        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );
