### Title
Stale Vote Accumulation in TEE Governance Functions Allows Threshold Bypass After Resharing — (`crates/contract/src/lib.rs`)

---

### Summary

`vote_code_hash`, `vote_add_launcher_hash`, and `vote_add_os_measurement` compare the raw accumulated vote count returned by `tee_state.vote*()` against the **current** threshold without filtering to only count votes from **current** participants. After a resharing that lowers the governance threshold, votes cast before the resharing by participants who remain in the new set retroactively satisfy the new (lower) threshold, allowing a malicious TEE image hash to be whitelisted without the required number of current-epoch participants ever agreeing.

---

### Finding Description

**Root cause — live accumulated value vs. tracked per-epoch value**

The external report's root cause is: a security check uses `address(this).balance` (a live, externally-inflatable value) instead of `_auction.amount` (the value captured at the time of the operation). The fix is to compare against the tracked state variable, not the live one.

The exact structural analog exists in the NEAR MPC contract.

`vote_code_hash` (and the launcher/OS-measurement siblings) do:

```rust
// lib.rs ~1406-1431
let votes = self.tee_state.vote(code_hash, &participant);
// …
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
```

`votes` is the **raw accumulated count** of all votes ever stored for `code_hash` — it is not filtered to the current participant set. `self.threshold()?.value()` is the **current** threshold, which can be lowered by a resharing.

Contrast this with `vote_update`, which explicitly filters and carries a comment explaining exactly why:

```rust
// lib.rs ~1361-1374
// Filter votes to only count current participants voting for this specific update.
// This ensures correctness even if the cleanup promise in MpcContract::vote_reshared() fails.
let valid_votes_count = running_state
    .parameters
    .participants()
    .participants()
    .iter()
    .filter(|(account_id, _, _)| {
        self.proposed_updates
            .vote_by_participant
            .get(account_id)
            .is_some_and(|voted_id| *voted_id == id)
    })
    .count();
```

The TEE governance functions (`vote_code_hash`, `vote_add_launcher_hash`, `vote_add_os_measurement`) lack this filtering entirely.

**Why the cleanup does not save them**

After a successful resharing, `vote_reshared` spawns a detached promise to `clean_tee_status`:

```rust
// lib.rs ~1186-1193
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        …
    )
    .detach();
```

`clean_tee_status` calls `self.tee_state.clean_non_participant_votes(participants)`, which removes votes from accounts that are **no longer** participants. It does **not** remove votes from accounts that remain participants. Therefore, if the colluding participants survive the resharing (which is the normal case when the resharing only removes a third party or lowers the threshold), their pre-resharing votes persist and are counted at full face value against the new, lower threshold.

---

### Impact Explanation

A set of T−1 colluding participants (strictly below the signing threshold T) can cause a malicious TEE image hash to be whitelisted by the contract. Once whitelisted, nodes running that malicious image can submit attestations that pass `is_caller_an_attested_participant`, satisfy `assert_caller_is_attested_participant_and_protocol_active`, and be proposed as participants in a future resharing. A malicious participant in the signing set can participate in threshold-signature rounds, potentially leaking key-share material or producing unauthorized signatures. This breaks the TEE security invariant that only audited, approved code may participate in the MPC network.

Impact category: **Medium** (participant-state manipulation that breaks a production safety invariant) with a credible escalation path to **Critical** (unauthorized signing capability) once the malicious node is admitted via resharing.

---

### Likelihood Explanation

The attack requires two sequential conditions:

1. T−1 colluding participants vote for a malicious code hash before the resharing (below the current threshold, so the hash is not yet whitelisted).
2. A legitimate resharing occurs that lowers the governance threshold to T−1 (e.g., removing a failed or departing node, which is a routine operational event).

The resharing itself requires T votes from current participants, so the colluding minority cannot force it alone — but they do not need to. They only need to wait for an honest resharing that happens to lower the threshold. This is a realistic operational scenario (e.g., a 5-of-4 network shrinks to 3-of-3 after two nodes leave). The colluding participants do not need to take any extraordinary action; they simply cast their pre-resharing votes and wait.

---

### Recommendation

Apply the same filtering pattern used in `vote_update` to all three TEE governance voting functions. Count only votes from accounts that are current participants at the time the threshold check is evaluated:

```diff
// vote_code_hash (and analogously for vote_add_launcher_hash, vote_add_os_measurement)
- let votes = self.tee_state.vote(code_hash, &participant);
- if votes >= self.threshold()?.value() {
-     self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
- }
+ let _ = self.tee_state.vote(code_hash, &participant);
+ let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();
+ let valid_votes = self.tee_state.count_votes_for_code_hash_from(
+     code_hash,
+     threshold_parameters.participants(),
+ );
+ if valid_votes >= self.threshold()?.value() {
+     self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
+ }
```

Alternatively, expose a `count_valid_votes` helper on `TeeState` that accepts the current `Participants` and filters internally, mirroring the approach in `TeeVerifierVotes::vote` which already does:

```rust
// verifier_votes.rs ~76
voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
```

### Proof of Concept

**Setup**: 5 participants A, B, C, D, E; governance threshold T = 4.

1. A, B, C each call `vote_code_hash(malicious_hash)`. Vote count = 3 < 4 → hash not yet whitelisted.
2. A, B, C, D call `vote_new_parameters` proposing a new epoch with participants {A, B, C, D} and threshold 3 (e.g., removing E for a legitimate operational reason). Four votes meet the current threshold of 4 → resharing begins and completes.
3. `clean_tee_status` runs post-resharing. A, B, C, D are all still participants → their code-hash votes are **not** cleaned.
4. A calls `vote_code_hash(malicious_hash)` again. `tee_state.vote()` returns the accumulated count for `malicious_hash` across A, B, C (= 3). New threshold = 3. Condition `3 >= 3` is true → `whitelist_tee_proposal` is called.
5. `malicious_hash` is now whitelisted. Nodes running that image can submit passing attestations and be proposed as participants in a future resharing.

The colluding set {A, B, C} is of size T−1 = 3, strictly below the signing threshold of 4 at the time the votes were cast. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
        log!(
            "vote_code_hash: signer={}, code_hash={:?}",
            env::signer_account_id(),
            code_hash,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let votes = self.tee_state.vote(code_hash, &participant);

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        // If the vote threshold is met and the new Docker hash is allowed by the TEE's RTMR3,
        // update the state
        if votes >= self.threshold()?.value() {
            self.tee_state
                .whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
        }

        Ok(())
    }
