### Title
Deflation Token Transfer Inflates Cross-Chain Amount, Creating Unbacked Supply — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter directly in the cross-chain event and message payload after calling `safeTransferFrom`. For deflation (fee-on-transfer) ERC20 tokens, the bridge vault actually receives fewer tokens than `amount`, but the cross-chain settlement message claims the full `amount` was locked. This causes the destination chain to mint or release more tokens than are actually held in the EVM vault, breaking the 1:1 backing guarantee.

### Finding Description

In `OmniBridge.initTransfer`, the non-bridge-token, non-custom-minter branch locks tokens by calling:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

Immediately after, the function emits the cross-chain event and calls `initTransferExtension` using the original caller-supplied `amount`, not the actual balance delta received by the contract:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← user-supplied, not actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

There is no balance-before / balance-after check anywhere in `initTransfer` or `initTransferExtension`. [3](#0-2) 

For a deflation token that burns `δ` on every transfer, the vault receives `amount − δ` but the cross-chain message carries `amount`. The NEAR side (or any destination) will mint/release `amount` tokens backed by only `amount − δ` locked tokens.

### Impact Explanation

This is a balance-accounting divergence that breaks the bridge's backing guarantee. Every `initTransfer` with a deflation token inflates the circulating wrapped supply on the destination chain relative to the collateral held on EVM. Over time, the vault becomes undercollateralized: the last redeemers cannot withdraw because the vault holds fewer tokens than the outstanding wrapped supply. This matches the allowed impact: *"Asset-identity, token-mapping, decimals, fee-routing, refund, or balance-accounting divergence that breaks backing guarantees or sends value to the wrong party."*

### Likelihood Explanation

The `initTransfer` function imposes no allowlist check on `tokenAddress` for the non-bridge-token path — any ERC20 can be passed. Fee-on-transfer tokens (e.g., PAXG, STA, tokens with reflection mechanics) are widely deployed. A user (or attacker) simply calls `initTransfer` with such a token; no privileged role is required. The vulnerability is triggered on every such call, not just adversarial ones.

### Recommendation

Measure the actual balance delta received by the vault and use that as the settled amount:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
    amount = uint128(received); // use actual received amount downstream
}
```

This ensures the emitted event and cross-chain payload reflect only the tokens actually held in the vault. Alternatively, explicitly document and enforce (via a registry check) that deflation tokens are not supported, reverting if the received amount differs from `amount`.

### Proof of Concept

1. Deploy a deflation ERC20 token `D` that burns 1% on every `transferFrom`.
2. Call `OmniBridge.initTransfer(D, 1000, 0, 0, "alice.near", "")`.
3. The vault receives `990` tokens (`safeTransferFrom` burns 10).
4. The `InitTransfer` event is emitted with `amount = 1000`.
5. The NEAR bridge processes the event and mints `1000` wrapped-D to `alice.near`.
6. Repeat: after enough transfers the vault holds far fewer tokens than the outstanding wrapped supply.
7. When users bridge back, `finTransfer` attempts `safeTransfer(recipient, 1000)` but the vault only holds `990 * n` for `n` round trips — the last redeemers receive nothing or the call reverts, permanently locking their wrapped tokens. [4](#0-3)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
        );
    }
