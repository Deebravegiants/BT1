### Title
Unnecessary `payable` modifier on `depositETHForL1VaultETH()` allows manager to accidentally inflate deposit amount and trigger `MaximumDepositLimitReached` revert — (`contracts/L1VaultV2.sol`)

### Summary
`depositETHForL1VaultETH()` is marked `payable` despite having no legitimate reason to accept ETH. If the manager accidentally attaches `msg.value`, the full `address(this).balance` (vault ETH + `msg.value`) is forwarded to `lrtDepositPool.depositETH`, potentially pushing `getTotalAssetDeposits(ETH)` above the configured limit and causing a revert that blocks the intended deposit.

### Finding Description
`depositETHForL1VaultETH()` is designed to convert the vault's accumulated ETH into rsETH by calling `lrtDepositPool.depositETH`. [1](#0-0) 

It reads `address(this).balance` rather than tracking a pre-call balance, so any ETH sent with the call is silently included. [2](#0-1) 

Inside `LRTDepositPool.depositETH`, `_beforeDeposit` is called with `msg.value` as the deposit amount. [3](#0-2) 

`_checkIfDepositAmountExceedesCurrentLimit` for ETH computes `getTotalAssetDeposits(ETH)` — which includes `address(depositPool).balance`, already updated with the incoming ETH — and checks `totalAssetDeposits > depositLimit`, ignoring the `amount` argument entirely. [4](#0-3) 

If the inflated total exceeds the limit, `_beforeDeposit` reverts with `MaximumDepositLimitReached`. [5](#0-4) 

### Impact Explanation
The deposit reverts. The vault's ETH (and the manager's accidentally sent ETH, which is returned on revert) remains idle, delaying rsETH minting and L2 delivery. No funds are lost, but the vault fails to deliver its promised function until the manager retries without `msg.value`. This matches **Low — contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation
Requires the manager to accidentally attach ETH to the call (e.g., copy-paste error, tooling misconfiguration, or confusion with `bridgeRsETHToL2` which legitimately requires `msg.value`). The condition is further constrained by needing total deposits near the limit. Likelihood is low but non-zero given the unnecessary `payable` modifier creates the footgun. [6](#0-5) 

### Recommendation
Remove the `payable` modifier from `depositETHForL1VaultETH()`. The function has no legitimate use for `msg.value`; the vault's ETH balance is already present before the call. Removing `payable` makes accidental ETH attachment a compile-time/runtime revert rather than a silent inclusion.
