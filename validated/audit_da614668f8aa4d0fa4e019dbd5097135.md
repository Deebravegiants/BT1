Audit Report

## Title
Attacker Can Permanently Lock Reward Tokens by Withdrawing All Stake After `notifyRewardAmount` - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary

`KernelDepositPool.notifyRewardAmount()` guards against zero-staked reward periods with `if (totalKernelStaked == 0) revert NoStakedTokens()`, but this check is only a point-in-time snapshot. An unprivileged attacker who is the sole staker (or who stakes any nonzero amount when no one else is staked) can call `initiateWithdrawal` immediately after `notifyRewardAmount`, reducing `totalKernelStaked` to zero. Once zero, `rewardPerToken()` permanently returns the frozen `rewardPerTokenStored` while `updateReward` continues advancing `updatedAt`, irrecoverably skipping all rewards that should have accrued during the zero-staked window.

## Finding Description

The `rewardPerToken()` function short-circuits when `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // frozen — no accumulation
    }
    ...
}
``` [1](#0-0) 

The `updateReward` modifier always advances `updatedAt` to `lastTimeRewardApplicable()`, regardless of whether `totalKernelStaked` is zero:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();
    ...
}
``` [2](#0-1) 

`initiateWithdrawal` decrements `totalKernelStaked` immediately (no delay), after `updateReward` has already snapshotted the pre-withdrawal state:

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
``` [3](#0-2) 

**Exploit flow:**

1. Attacker is the only staker (e.g., at launch), with `totalKernelStaked = X`.
2. Admin calls `notifyRewardAmount(large_amount)` — passes the `totalKernelStaked > 0` guard, sets `rewardRate`, `finishAt`, `updatedAt = block.timestamp`.
3. Attacker calls `initiateWithdrawal(X)`:
   - `updateReward` runs: `rewardPerToken()` computes normally (but delta since step 2 is ~0 if same block), `updatedAt = block.timestamp`.
   - `totalKernelStaked` drops to 0.
4. All subsequent calls to `rewardPerToken()` return the frozen `rewardPerTokenStored`.
5. When the next legitimate user calls `stake(amount)`, `updateReward` runs with `totalKernelStaked == 0` → `rewardPerTokenStored` stays frozen, `updatedAt` advances to now.
6. All rewards that should have accrued during the zero-staked window (step 3 → step 5) are permanently locked in the contract — no user can ever claim them.

The contract's own NatSpec explicitly acknowledges this risk and relies entirely on an off-chain operational assumption to prevent it:

```
* @dev If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards
*      for that period will stay locked in the contract. In this deployment, we're avoiding this issue by
*      ensuring there are always some tokens staked...
``` [4](#0-3) 

The `notifyRewardAmount` guard only checks the moment of the call: [5](#0-4) 

There is no on-chain mechanism preventing a staker from withdrawing immediately after `notifyRewardAmount` is called.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** All reward tokens deposited for the current period that should have accrued during the zero-staked window are permanently locked in `KernelDepositPool`. No present or future staker can ever claim them. The locked amount equals `rewardRate * (duration of zero-staked window)`, which can be the entire period's allocation if the attacker acts immediately after `notifyRewardAmount`.

## Likelihood Explanation

Medium. The attacker must be the sole staker (or reduce `totalKernelStaked` to zero) at the time of withdrawal. This is realistic at protocol launch when staking is sparse, or when the attacker is the only participant. The cost is negligible (any nonzero KERNEL amount). The attacker can also monitor the mempool and back-run `notifyRewardAmount` with `initiateWithdrawal`. The attack is repeatable each reward period.

## Recommendation

Replace the point-in-time `totalKernelStaked == 0` guard with a meaningful minimum threshold that makes it economically infeasible to drain `totalKernelStaked` to zero after `notifyRewardAmount`:

```solidity
uint256 public constant MIN_STAKED_FOR_REWARDS = 1e18; // e.g., 1 KERNEL

function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
    if (_amount == 0) revert AmountZero();
    if (totalKernelStaked < MIN_STAKED_FOR_REWARDS) revert InsufficientStakedTokens();
    ...
}
```

Additionally, consider tracking a `minimumStakeSnapshot` at `notifyRewardAmount` time and reverting withdrawals that would push `totalKernelStaked` below a safe floor during an active reward period.

## Proof of Concept

Minimal Foundry test sequence:

```solidity
// 1. Attacker stakes 1 wei
vm.prank(attacker);
kernelDepositPool.stake(1);

// 2. Admin notifies reward amount (passes guard since totalKernelStaked == 1)
vm.prank(admin);
rewardsToken.approve(address(kernelDepositPool), 1_000_000e18);
vm.prank(admin);
kernelDepositPool.notifyRewardAmount(1_000_000e18);

// 3. Attacker immediately withdraws (same block or next block)
vm.prank(attacker);
kernelDepositPool.initiateWithdrawal(1);
// totalKernelStaked == 0 now

// 4. Time passes (half the reward period)
vm.warp(block.timestamp + duration / 2);

// 5. Legitimate user stakes
vm.prank(legitimateUser);
kernelDepositPool.stake(1000e18);

// 6. Wait for period to end, user claims
vm.warp(block.timestamp + duration);
vm.prank(legitimateUser);
kernelDepositPool.getReward();

// Assert: legitimateUser received only ~half the rewards (from step 5 onward)
// The other ~500,000 reward tokens are permanently locked in the contract
assertApproxEqAbs(
    rewardsToken.balanceOf(address(kernelDepositPool)),
    500_000e18,  // locked forever
    1e18
);
```
