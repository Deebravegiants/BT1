### Title
Pool Operator Can Indefinitely Reset Waiting-Room Timelock via Permissionless Absorb Spends - (File: chia/pools/pool_puzzles.py)

### Summary
The legacy pool wallet's `LEAVING_POOL` (waiting-room) state enforces a `relative_lock_height` exit delay using `ASSERT_HEIGHT_RELATIVE`, which is measured from the **current singleton coin's** confirmed block index. Because absorb spends (reward claims) are permissionless and each one destroys the current singleton coin and creates a new one at the current block height, a malicious pool operator can continuously absorb p2_singleton rewards to keep minting fresh singleton coins, perpetually resetting the relative timelock and trapping the farmer in the pool indefinitely.

### Finding Description

When a farmer initiates a pool exit, the singleton transitions to the `LEAVING_POOL` state using the `POOL_WAITINGROOM_INNERPUZ` inner puzzle, which is curried with `relative_lock_height`:

```python
# chia/pools/pool_puzzles.py L51-64
def create_waiting_room_inner_puzzle(..., relative_lock_height, ...) -> Program:
    ...
    return POOL_WAITING_ROOM_MOD.curry(
        target_puzzle_hash, p2_singleton_puzzle_hash, bytes(owner_pubkey),
        pool_reward_prefix, relative_lock_height
    )
```

To exit the waiting room, the farmer must spend the current singleton coin with `ASSERT_HEIGHT_RELATIVE(relative_lock_height)`. Per `compute_assert_height` in `mempool_manager.py`, this condition resolves to:

```
valid_at_height = removal_coin_records[coin_id].confirmed_block_index + relative_lock_height
```

The wallet-side check in `pool_wallet.py` mirrors this:

```python
# chia/pools/pool_wallet.py L695-703
last_height: uint32 = history[-1][0]
if (
    await self.wallet_state_manager.blockchain.get_finished_sync_up_to()
    <= last_height + current_state.current.relative_lock_height
):
    raise ValueError(...)
```

Critically, absorb spends (claiming p2_singleton farming rewards) are **permissionless** — they require no signature (`G2Element()` empty sig suffices), as confirmed by the lifecycle test:

```python
# chia/_tests/pools/test_pool_puzzles_lifecycle.py L340-352
coin_sols: list[CoinSpend] = create_absorb_spend(
    travel_coinsol, target_pool_state, launcher_coin, 3, ...
)
coin_db.update_coin_store_for_spend_bundle(
    SpendBundle(coin_sols, G2Element()), time, ...  # no signature needed
)
```

Each absorb spend destroys the current singleton coin and creates a new one at the current block height. The `ASSERT_HEIGHT_RELATIVE` for the farmer's exit is then measured from this new coin's `confirmed_block_index`, resetting the countdown.

### Impact Explanation

A malicious pool operator monitors the blockchain for a farmer entering `LEAVING_POOL` state. Whenever the timelock is about to expire, the pool broadcasts a new absorb spend against any pending p2_singleton reward coin. This creates a new singleton coin at the current block height, resetting `relative_lock_height` blocks of waiting. Since the farmer continues farming (and thus producing p2_singleton rewards) while waiting to leave, the pool always has material to absorb. The farmer is permanently trapped: their block rewards continue flowing to the pool's `target_puzzle_hash` and they cannot redirect them to self-pooling.

This matches the High impact category: **permanent inability for a farmer to execute a valid pool exit action**, and **unauthorized prevention of a protected singleton state transition**.

### Likelihood Explanation

The pool operator has direct financial incentive (retaining the farmer's rewards) and the technical capability (absorb spends are permissionless, requiring only knowledge of the p2_singleton puzzle hash, which is public). The attack requires no key compromise, no cryptographic break, and no admin access — only the ability to broadcast transactions, which any network participant can do.

### Recommendation

The timelock for exiting the waiting room should be anchored to the **block height at which the singleton first entered the `LEAVING_POOL` state**, not the height of the most recently created singleton coin. One approach: record the entry height as an immutable field in the waiting-room puzzle (curried in at transition time) and use `ASSERT_HEIGHT_ABSOLUTE(entry_height + relative_lock_height)` instead of `ASSERT_HEIGHT_RELATIVE`. This prevents absorb spends from resetting the countdown because the absolute target height is fixed at entry.

### Proof of Concept

1. Farmer joins pool with `relative_lock_height = 1440` (~1 day).
2. Farmer calls `pw_self_pool`, singleton transitions to `LEAVING_POOL` at block **B**.
3. Pool operator observes the transition and waits until block **B + 1430** (10 blocks before expiry).
4. Pool operator broadcasts an absorb spend against any pending p2_singleton reward coin. New singleton coin is created at block **B + 1430**.
5. Farmer's exit is now gated on block **(B + 1430) + 1440 = B + 2870**.
6. Pool repeats step 4 every ~1430 blocks. The farmer can never satisfy `ASSERT_HEIGHT_RELATIVE(1440)` because the base coin is always refreshed before the condition can be met. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
