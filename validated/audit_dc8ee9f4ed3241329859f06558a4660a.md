### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Submitted `request` — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the supplied signature is valid over `response.payload_hash`, but never validates that `response.payload_hash` actually corresponds to the `request` key used to look up the pending yield. Any single attested participant can replay a legitimately-produced threshold signature from a prior request to resolve a completely different pending request, delivering a forged verification result to the waiting caller.

### Finding Description

In `respond_verify_foreign_tx` the contract performs two independent operations:

1. **Signature check** — verifies `response.signature` over `response.payload_hash` against the root public key.
2. **Yield resolution** — looks up `pending_verify_foreign_tx_requests[request]` and resolves it with the full `response`. [1](#0-0) 

The `payload_hash` in the response is the SHA-256 of `borsh(ForeignTxSignPayload{request, extracted_values})`. The contract never recomputes this hash from the supplied `request` and never asserts `response.payload_hash == expected_hash(request)`. The `request` field is used **only** as a map key to locate and drain the pending yield queue; it plays no role in the cryptographic check. [2](#0-1)

### Impact Explanation

**Forged foreign-chain verification / cross-chain replay by a single below-threshold participant.**

Attack path:

1. User A submits `verify_foreign_transaction(request_A)` (e.g., Bitcoin tx A). The MPC network legitimately produces `sig_A` over `payload_hash_A` and the leader calls `respond_verify_foreign_tx(request_A, {payload_hash_A, sig_A})`. Both `sig_A` and `payload_hash_A` are now permanently visible on-chain.

2. User B submits `verify_foreign_transaction(request_B)` (Bitcoin tx B). The yield for `request_B` is now pending.

3. A **single malicious attested participant** (below threshold) reads `sig_A` / `payload_hash_A` from chain history and calls:
   ```
   respond_verify_foreign_tx(request_B, {payload_hash_A, sig_A})
   ```

4. The contract checks: is `sig_A` a valid signature over `payload_hash_A` under the root key? **Yes** — it was legitimately produced. The check passes.

5. The contract resolves the yield for `request_B` and delivers `{payload_hash_A, sig_A}` to User B's waiting transaction.

6. User B's bridge contract receives a response whose `payload_hash` corresponds to tx A, not tx B. If the bridge contract does not independently recompute and compare the expected hash (using the SDK's `ForeignChainSignatureVerifier`), it will treat the MPC network as having verified tx B when it actually verified tx A — enabling double-spend or invalid bridge execution. [4](#0-3) 

The SDK helper that *would* catch this mismatch lives in the **caller's** contract, not in the MPC contract itself. The MPC contract provides no on-chain guarantee.

### Likelihood Explanation

- Any single attested participant can call `respond_verify_foreign_tx`; there is no restriction to the leader node.
- All prior signatures are permanently public on-chain; no key material needs to be stolen.
- The attacker only needs to front-run the legitimate response for `request_B`, which is feasible when foreign-chain RPC latency is high or when the attacker controls a NEAR validator.
- The attack is profitable whenever the bridge contract omits the payload-hash cross-check, which is an easy integration mistake given that the MPC contract itself does not enforce it.

### Recommendation

Inside `respond_verify_foreign_tx`, recompute the expected payload hash from the `request` and the `extracted_values` included in the response, then assert equality before accepting the response:

```rust
// Pseudocode
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // add this field to the response DTO
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, store the expected `payload_hash` in the pending-request map at submission time (once the MPC nodes have agreed on the extracted values) so the contract can validate it without trusting the responder.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
