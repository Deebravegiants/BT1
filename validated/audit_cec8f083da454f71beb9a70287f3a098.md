Audit Report

## Title
Missing Staleness Check in `getRate()` Enables Over-Minting of wrsETH at Existing Holders' Expense - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver` records `lastUpdated` on every LayerZero message receipt but never reads it inside `getRate()`, which unconditionally returns the stored `rate`. Because rsETH is a monotonically appreciating asset, any period of oracle staleness (LayerZero congestion, relayer downtime) leaves the stored rate below the true rate, causing all pool variants to mint more wrsETH per deposited ETH than the depositor is entitled to, diluting existing holders' accrued yield.

## Finding Description
`CrossChainRateReceiver.lzReceive()` writes both `rate` and `lastUpdated` on receipt of a LayerZero message:

```solidity
// CrossChainRateReceiver.sol L95-97
rate = _rate;
lastUpdated = block.timestamp;
```

However, `getRate()` ignores `lastUpdated` entirely:

```solidity
// CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

Every pool variant (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`) computes the mint amount as:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

where `rsETHToETHrate` is fetched via `IOracle(rsETHOracle).getRate()`. Because rsETH accrues EigenLayer restaking rewards continuously, its ETH-denominated rate increases over time. A stale (lower-than-current) rate makes the divisor smaller, inflating `rsETHAmount`. The `dailyMintLimit` guard caps per-day exposure but does not prevent the attack from being repeated across days or at the limit boundary. The `paused` flag requires a privileged `PAUSER_ROLE` holder to intervene; it is not an automatic protection. `InterimRSETHOracle.getRate()` has the identical structural gap (no expiry on the manually set rate), though that path involves a trusted MANAGER.

The SECURITY.md exclusion for "incorrect data supplied by third-party oracles" does not apply here: the oracle sent correct data; the vulnerability is the receiver contract's own failure to enforce a freshness requirement on data it already holds.

## Impact Explanation
**High — Theft of unclaimed yield.**

Existing wrsETH holders have a proportional claim on the underlying rsETH pool. When a depositor mints wrsETH at a stale (lower) rate, they receive more wrsETH tokens than their ETH deposit warrants. After the oracle refreshes to the true (higher) rate, the over-issued wrsETH represents more ETH-equivalent value than was deposited. The surplus is extracted directly from the yield that had accrued to prior holders between the last oracle update and the deposit. This matches the allowed impact "Theft of unclaimed yield."

## Likelihood Explanation
**Medium.** LayerZero relayer congestion and brief downtime are documented real-world events for cross-chain protocols. rsETH's rate grows continuously (EigenLayer restaking rewards), so any gap between oracle updates creates a measurable, exploitable delta. No attacker capability beyond a standard `deposit()` call is required; the attacker only needs to observe that `CrossChainRateReceiver.lastUpdated` is sufficiently old and act before the next update arrives. The attack is permissionless and repeatable.

## Recommendation
Add a configurable maximum staleness period to `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public maxStaleness = 24 hours; // configurable by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

Apply the same guard to `InterimRSETHOracle.getRate()` by storing a `lastUpdated` timestamp on each `setRate()` call and checking it in `getRate()`. This causes deposits to revert when the oracle is stale rather than silently minting at an incorrect rate.

## Proof of Concept

**Foundry fork test outline:**

1. Fork an L2 where `RSETHPoolV3` is deployed with `CrossChainRateReceiver` as the oracle.
2. Warp time forward by 30+ hours without delivering a new LayerZero message (simulating relayer downtime), so `lastUpdated` is stale but `rate` remains at the old lower value.
3. Record `rateBefore = CrossChainRateReceiver.rate` and `trueRate` (fetched from L1 rsETH rate provider via a separate read or hardcoded from a recent block).
4. Call `RSETHPoolV3.deposit{value: 10 ether}("")` as an unprivileged attacker address.
5. Assert `wrsETH.balanceOf(attacker) > 10e18 * 1e18 / trueRate` — i.e., attacker received more wrsETH than the true rate warrants.
6. Deliver the updated LayerZero message (call `lzReceive` with `trueRate`); oracle refreshes.
7. Assert that attacker's wrsETH balance, valued at the new rate, exceeds 10 ETH — confirming extraction of accrued yield from prior holders.

The `dailyMintLimit` check at `RSETHPoolV3.sol` L119-121 caps single-day exposure but does not prevent the attack from being repeated on subsequent days or across all deployed pool variants sharing the same stale oracle.
