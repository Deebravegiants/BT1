### Title
Gateway Admits Invoke Transaction with Arbitrary Signature via Validation-Skip Path — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
An unprivileged HTTP client can submit an invoke transaction with nonce=1 and an arbitrary (invalid) signature for any account whose address appears in the mempool or recent-block state. The `skip_stateful_validations` function returns `true`, causing `run_validate_entry_point` to set `execution_flags.validate = false`, which causes `StatefulValidator::perform_validations` to return `Ok(())` before calling `__validate__`. The transaction is accepted into the mempool without any cryptographic signature check.

`skip_stateful_validations` is a UX feature that allows a user to submit a `deploy_account` + `invoke` pair atomically, before the deploy is processed. The skip condition is: [1](#0-0)

### Impact Explanation

An attacker who observes that account address `A` has a `deploy_account` transaction in the mempool (on-chain nonce = 0) can submit an invoke with `nonce=1` and a completely arbitrary signature for `A`. The gateway accepts it into the mempool without any signature verification. This is a confirmed **High** impact: the gateway/mempool admission path accepts a cryptographically invalid transaction.

The transaction will fail during block building when `__validate__` is called by the batcher (the skip is gateway-only). However, the accepted invalid invoke can:
- Displace the legitimate user's nonce=1 invoke via fee-escalation rules, causing the legitimate transaction to be rejected
- Pollute the mempool with an invalid transaction that wastes batcher resources

The claim in the question that this "corrupts the state diff" is **incorrect** — the transaction fails at block-building time and does not produce a committed state change. The confirmed impact is admission of an invalid transaction, not state corruption.

### Likelihood Explanation

The attack requires only:
1. Observing that a `deploy_account` for address `A` is pending in the mempool (publicly observable via mempool APIs or p2p gossip)
2. Constructing an invoke with `nonce=1`, `sender_address=A`, and any signature bytes
3. POSTing to `/gateway/add_transaction`

No privileged access is required. The condition `nonce==1 && account_nonce==0 && account_in_mempool` is reliably triggerable whenever a user deploys a new account.

### Recommendation

The skip should be conditioned on the presence of a **deploy_account specifically** for the sender address, not just any account presence. The mempool should expose a dedicated `has_deploy_account_in_pool(address)` query, or the gateway should verify that the matching mempool entry is a `deploy_account` transaction type. Alternatively, the skip should only apply when the transaction's signature is structurally consistent with the account class being deployed (e.g., verified against the constructor arguments of the pending deploy_account).

### Proof of Concept

1. Submit a valid `deploy_account` for address `A` (attacker controls the key, or observes a legitimate user's deploy_account in the mempool)
2. Submit an invoke: `sender_address=A`, `nonce=1`, `signature=[0xdead, 0xbeef]` (arbitrary bytes)
3. `skip_stateful_validations` returns `true` → `execution_flags.validate = false` → `__validate__` skipped
4. Gateway returns HTTP 200 with a transaction hash
5. Assert: the transaction hash is accepted; the mempool contains the invalid-signature invoke
6. Observe: when the batcher processes the block, the invoke fails at `__validate__` and is dropped — confirming the signature was never checked at admission
