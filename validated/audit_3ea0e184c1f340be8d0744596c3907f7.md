### Title
Missing `directDeposits` in Dijkstra Era Preservation-of-Value Check Allows ADA Creation - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a new `directDeposits` transaction body field that allows a transaction author to credit ADA directly to staking account balances. However, `dijkstraProducedValue` — the function used in the preservation-of-value (consumed = produced) check — does not include `directDeposits` in the produced side. Because the ENTITIES rule unconditionally applies `directDeposits` to account balances after the UTxO balance check passes, an unprivileged transaction sender can create ADA from nothing by including a non-zero `directDeposits` map in their transaction body.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — missing special case for a new ADA-transfer mechanism in the preservation-of-value accounting function.

The Dijkstra era adds `directDeposits :: DirectDeposits` to both top-level and sub-transaction bodies: [1](#0-0) [2](#0-1) 

`DirectDeposits` is a `Map AccountAddress Coin` that is applied to account balances in the ENTITIES rule via `applyDirectDeposits`: [3](#0-2) 

This function unconditionally adds the specified coin amounts to account balances. It does not deduct from any UTxO output or other pot.

The ENTITIES rule applies `directDeposits` after the UTxO rule has already validated the preservation of value: [4](#0-3) 

The same pattern exists in the SUBENTITIES rule for sub-transactions: [5](#0-4) 

The Dijkstra produced-value function used in the UTxO balance check is: [6](#0-5) 

It delegates to `conwayProducedValue` (outputs + fee + cert deposits) and aggregates sub-transaction produced values. **`directDeposits` is absent from both branches.**

Similarly, the consumed-value function does not include `directDeposits`: [7](#0-6) 

The ledger transition order in `dijkstraLedgerTransition` confirms that the UTXO rule (which enforces consumed = produced) runs **before** the ENTITIES rule (which applies `directDeposits`): [8](#0-7) 

**Root cause:** `directDeposits` is a new ADA-transfer mechanism analogous to `withdrawals` (which moves ADA from accounts to UTxO and is correctly included in `consumed`). The symmetric operation — moving ADA from UTxO to accounts — requires `directDeposits` to appear in `produced`. Its absence means the UTxO balance check enforces only:

```
inputs = outputs + fee + cert_deposits
```

while the ENTITIES rule additionally credits `sum(directDeposits)` to account balances, violating the global preservation-of-value invariant:

```
inputs = outputs + fee + cert_deposits + directDeposits   ← NOT enforced
```

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

An attacker who submits a Dijkstra transaction with `directDeposits = {addr: N}` will:
1. Pass the UTxO preservation-of-value check (consumed = produced, without `directDeposits`).
2. Have `N` lovelace credited to the target account balance by the ENTITIES rule.
3. Net effect: `N` lovelace is created from nothing — the UTxO is not reduced by `N`, yet the account balance increases by `N`.

The attacker can then withdraw the inflated balance. Repeated exploitation inflates the total ADA supply, directly violating the ledger's core accounting invariant.

---

### Likelihood Explanation

**High.** The attack requires no privileged access, no governance majority, and no leaked keys. Any unprivileged transaction sender can craft a valid Dijkstra transaction body with a non-empty `directDeposits` map. The `directDeposits` field is a standard, serializable part of the transaction body: [9](#0-8) 

The ENTITIES rule validates only that target accounts are registered (`directDepositsMissingAccounts`), not that the deposited amounts are funded by the transaction's inputs: [10](#0-9) 

---

### Recommendation

Include `directDeposits` in the produced-value calculation for both top-level and sub-transactions in `dijkstraProducedValue` (and `dijkstraSubTxProducedValue`). Specifically, add `inject (sum (unDirectDeposits (txBody ^. directDepositsTxBodyL)))` to the produced value, mirroring how `withdrawals` are included in `consumed`. This ensures the transaction author must provide sufficient UTxO inputs to fund both outputs and direct deposits, preserving the global ADA supply invariant.

---

### Proof of Concept

1. Construct a Dijkstra transaction with:
   - Inputs: 10 ADA
   - Outputs: 9 ADA (to self)
   - Fee: 1 ADA
   - `directDeposits`: `{registeredAccount: 5_000_000}` (5 ADA)
2. The UTxO rule checks: consumed (10 ADA) = produced (9 ADA + 1 ADA fee) = 10 ADA → **passes**.
3. The ENTITIES rule applies `applyDirectDeposits`, adding 5 ADA to `registeredAccount`'s balance.
4. Result: 10 ADA in, 9 ADA UTxO output + 1 ADA fee + 5 ADA in account = 15 ADA out. **5 ADA created from nothing.**
5. The attacker withdraws the 5 ADA from the account in a subsequent transaction. [11](#0-10) [6](#0-5)
