### Title
Plutus Guard Datum Consistency Check Silently Bypassed When Script Absent from Sub-Transaction's `scriptsProvided` - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxow.hs`)

---

### Summary

In the Dijkstra era's `SUBUTXOW` rule, `validateGuardDatums` is responsible for enforcing that every `ScriptHashObj` credential in a sub-transaction's `requiredTopLevelGuards` map that resolves to a Plutus script must carry a datum (`SJust`). When the script hash is not found in the sub-transaction's `scriptsProvided` — which happens whenever the Plutus guard script is supplied only as a reference script at the top-level transaction rather than in the sub-transaction's own inputs — the function silently returns `acc` (no error). This is the direct analog of the "unvalidated initiator / unconsumed receiver" footgun: a required validation step is skipped because the entity being validated is not locally visible, mirroring the flash-loan pattern where the callback does not verify the initiator.

---

### Finding Description

`validateGuardDatums` iterates over `requiredTopLevelGuards` and, for each `ScriptHashObj` credential, looks up the script in the sub-transaction's `scriptsProvided`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxow.hs
accum acc cred mbDatum =
  case credScriptHash cred of
    Nothing ->
      case mbDatum of
        SNothing -> acc
        SJust _  -> Set.insert cred acc
    Just scriptHash ->
      case Map.lookup scriptHash scripts of   -- scripts = sub-tx's scriptsProvided
        Just script
          | isNativeScript script ->
              case mbDatum of
                SNothing -> acc
                SJust _  -> Set.insert cred acc
          | otherwise ->                      -- Plutus: datum MUST be SJust
              case mbDatum of
                SJust _  -> acc
                SNothing -> Set.insert cred acc
        Nothing -> acc                        -- ← BUG: silently passes
``` [1](#0-0) 

The `Nothing -> acc` branch is reached whenever the script hash is not present in the sub-transaction's own `scriptsProvided`. The sub-transaction's `scriptsProvided` is computed by `getBabbageScriptsProvided utxo subTx`, which only aggregates scripts from the sub-transaction's own witness set and its own reference/spending inputs: [2](#0-1) 

A Plutus guard script that is attached only as a reference script in the **top-level** transaction's inputs is therefore absent from the sub-transaction's `scriptsProvided`, triggering the silent pass. The check is invoked in `dijkstraSubUtxowTransition` using exactly this per-sub-tx view: [3](#0-2) 

The function's own documentation states the invariant that is violated:

> *"Plutus script credentials must have a datum, key/native script credentials must not."* [4](#0-3) 

---

### Impact Explanation

The `requiredTopLevelGuards` map (`Map (Credential Guard) (StrictMaybe (Data era))`) is the mechanism by which a sub-transaction author declares which top-level guards must authorize the sub-transaction and supplies the datum each Plutus guard script will receive at execution time. [5](#0-4) 

By exploiting the `Nothing -> acc` bypass, a sub-transaction author can declare a `ScriptHashObj` guard credential with `SNothing` (no datum) for a Plutus script that is only present as a top-level reference script. The phase-1 check passes. The guard script then executes at the top level via the `GuardingPurpose` redeemer without the datum the script was designed to receive.

For **Plutus V3+** scripts (CIP-0069 removes the mandatory datum argument), the script can succeed without a datum. If the guard script's authorization logic depends on the datum to verify conditions about the sub-transaction (e.g., checking an authorization token, a price oracle value, or a time-lock), that check is silently omitted. This constitutes a **witness/script validation bypass**: an attacker-controlled sub-transaction exceeds the intended validation limits of the guard mechanism, causing the guard to execute outside its designed parameters.

For **Plutus V1/V2** scripts, the missing datum causes a phase-2 failure, which takes collateral from the top-level transaction author — a denial-of-service / collateral-drain vector against the top-level transaction author who is compelled to include the guard by the `MissingRequiredGuards` check. [6](#0-5) 

This matches the **Medium** allowed impact: *"Attacker-controlled transactions … exceed intended validation limits … outside design parameters."*

---

### Likelihood Explanation

The attack is reachable by any unprivileged sub-transaction author. The only precondition is that the Plutus guard script is supplied as a reference script in the top-level transaction's inputs rather than in the sub-transaction's own inputs — a natural and common pattern for reusing on-chain scripts without re-attaching them in every sub-transaction. No privileged access, key compromise, or consensus-level attack is required. The sub-transaction body is attacker-controlled serialized input accepted by the ledger.

---

### Recommendation

Replace the silent `Nothing -> acc` branch with a conservative treatment. Two options:

1. **Treat unknown scripts as Plutus** (safe default): if the script hash is not found in `scriptsProvided`, require `SJust` datum, since the script type cannot be determined and the safe assumption is Plutus.

2. **Extend the lookup to the top-level `scriptsProvided`**: pass the aggregated top-level `scriptsProvided` (which includes reference scripts from all inputs) into `validateGuardDatums` so that the script type can always be resolved.

```haskell
-- Proposed fix (option 1):
Nothing ->
  -- Script not locally visible; conservatively require datum
  -- (Plutus scripts provided at top level still need datum declared here)
  case mbDatum of
    SJust _  -> acc
    SNothing -> Set.insert cred acc
```

### Proof of Concept

1. Deploy a Plutus V3+ guard script `G` as a reference script in UTxO output `refOut`.
2. Craft a sub-transaction `subTx` with:
   - `requiredTopLevelGuards = { ScriptHashObj (hash G) → SNothing }` (no datum)
   - No reference input pointing to `refOut` (so `G` is absent from `subTx`'s `scriptsProvided`)
3. Craft a top-level transaction `topTx` with:
   - `subTransactions = { subTx }`
   - `guards = [ ScriptHashObj (hash G) ]` (forced by `MissingRequiredGuards` check)
   - `referenceInputs = { refOut }` (provides `G` at top level)
4. Submit `topTx`. Phase-1 validation runs `validateGuardDatums` on `subTx`; the `Nothing -> acc` branch fires because `G` is not in `subTx`'s `scriptsProvided`. No `SubMalformedGuardDatums` failure is raised.
5. Phase-2 executes `G` with `GuardingPurpose` redeemer and **no datum**. For a CIP-0069-compliant Plutus V3+ script, execution succeeds. The guard's datum-dependent authorization logic is never evaluated. [7](#0-6) [8](#0-7)
