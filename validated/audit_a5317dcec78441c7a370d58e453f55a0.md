### Title
Attached NEAR Deposit Permanently Locked in WalletContract on `SelfBaseTokenTransfer` No-Op - (File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs)

### Summary
When an external caller (relayer) attaches a NEAR deposit to `rlp_execute` and the Ethereum transaction resolves to a `SelfBaseTokenTransfer` (a no-op), the attached deposit is captured in `CallerDeposit` but is silently discarded and permanently locked inside the wallet contract. The `rlp_execute_callback` early-returns at `n == 0` before the refund logic is reached, mirroring the M-08 pattern exactly: a secondary value channel is ignored when a specific action type is processed.

### Finding Description
`rlp_execute` is marked `#[payable]`, so any external caller may attach NEAR tokens. [1](#0-0) 

Inside `inner_rlp_execute`, the attached deposit is captured into `CallerDeposit` for any non-self external caller with a non-zero deposit: [2](#0-1) 

`CallerDeposit::new` stores the predecessor account and the yoctoNEAR amount: [3](#0-2) 

When the Ethereum transaction is a self-transfer (Ethereum `to == wallet address`), the code takes the `SelfBaseTokenTransfer` branch and calls `rlp_execute_callback` **directly** (no preceding promise): [4](#0-3) 

Inside `rlp_execute_callback`, when called directly (`n == 0`), the function returns immediately with `success: true` **before** the `caller_deposit` refund logic is ever reached: [5](#0-4) 

The refund logic only executes when `n == 1` and `PromiseResult::Failed`: [6](#0-5) 

Because `SelfBaseTokenTransfer` is a no-op (no action is dispatched, no deposit is forwarded to any recipient), the attached NEAR tokens are neither consumed by an action nor returned to the caller — they are permanently locked in the wallet contract's account balance.

### Impact Explanation
The corrupted protocol value is the **caller's NEAR token balance**: any NEAR attached to `rlp_execute` for a self-transfer is irrecoverably locked in the eth-implicit wallet contract. There is no recovery path: the `SelfBaseTokenTransfer` branch always succeeds, so the deposit refund branch (`PromiseResult::Failed`) is never triggered, and the `n == 0` early return skips the refund unconditionally.

### Likelihood Explanation
`rlp_execute` is `#[payable]` and the `CallerDeposit` mechanism was explicitly designed for external relayers who attach NEAR deposits. A relayer may attach NEAR when:
- The Ethereum transaction carries a non-zero `value` field (as shown in the existing test at line 70 of `emulation.rs`, where `value: Wei::new_u128(TRANSFER_AMOUNT...)` is used for a self-transfer)
- The relayer follows a general pattern of attaching deposits for all transactions
- The relayer is trying to forward the Ethereum transaction's value to the wallet

The `test_base_token_transfer` test already demonstrates a self-transfer with a non-zero Ethereum `value`, confirming this is a realistic user-facing scenario. [7](#0-6) 

### Recommendation
In `rlp_execute_callback`, when `n == 0` (the `SelfBaseTokenTransfer` path), refund the `caller_deposit` before returning:

```rust
if n == 0 {
    // Refund the caller's deposit since the self-transfer is a no-op
    if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
        let refund_promise = env::promise_batch_create(&account_id);
        env::promise_batch_action_transfer(
            refund_promise,
            NearToken::from_yoctonear(yocto_near.into()),
        );
    }
    return ExecuteResponse { success: true, success_value: None, error: None };
}
```

### Proof of Concept
1. Deploy the wallet contract to an eth-implicit account `0xABCD...`.
2. An external relayer calls `rlp_execute(target="0xABCD...", tx_bytes_b64=<self-transfer-tx>)` with `attached_deposit = 3 NEAR`.
3. `inner_rlp_execute` captures `CallerDeposit { account_id: relayer, yocto_near: 3e24 }`.
4. The Ethereum transaction's `to` field equals the wallet address → `SelfBaseTokenTransfer` branch.
5. `rlp_execute_callback(Some(caller_deposit))` is scheduled directly.
6. Callback executes: `n = 0` → early return `ExecuteResponse { success: true }`.
7. The 3 NEAR is now locked in the wallet contract. The relayer's balance is permanently reduced by 3 NEAR with no recourse.
