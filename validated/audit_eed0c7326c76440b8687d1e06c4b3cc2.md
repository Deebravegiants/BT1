### Title
Missing Chain Identifier in `DelegateAction` Signed Payload Enables Cross-Network Signature Replay - (File: `core/primitives/src/action/delegate.rs`)

### Summary
`DelegateAction` and `DelegateActionV2` (meta-transaction payloads) are signed without any chain or network identifier. A malicious relayer who receives a user's `SignedDelegateAction` intended for one NEAR network (e.g., testnet) can submit it to a different NEAR network (e.g., mainnet) where the same account and key exist, causing unauthorized execution of the inner actions on the victim's account on the wrong network.

### Finding Description
In NEAR's meta-transaction system (NEP-366), a user creates and signs a `DelegateAction` off-chain and hands it to a relayer. The relayer wraps it in an outer `SignedTransaction` and submits it to the chain. The outer transaction includes a `block_hash` that ties it to a specific chain, but this is signed by the **relayer**, not the user.

The user's signature covers only the `DelegateAction` payload, hashed via `get_nep461_hash()`:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
```

The `SignableMessage` prepends a 4-byte `MessageDiscriminant` (NEP number) to distinguish message types, but contains no chain identifier:

```rust
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub msg: &'a T,
}
```

The signed `DelegateAction` struct itself contains no chain binding:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
}
```

The same is true for `DelegateActionV2`. Neither struct nor the `SignableMessage` wrapper includes a `chain_id`, genesis hash, or any network-scoping field.

The verification in `apply_delegate_action` checks the signature, nonce, `max_block_height`, and `sender_id`, but none of these checks are network-scoped:

```rust
pub(crate) fn apply_delegate_action(...) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() { ... }
    if apply_state.block_height > delegate_action.max_block_height() { ... }
    if delegate_action.sender_id().as_str() != sender_id.as_str() { ... }
    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
    ...
}
```

### Impact Explanation
A malicious relayer who receives a victim's `SignedDelegateAction` intended for testnet can submit it to mainnet (or any other NEAR-compatible network) by wrapping it in a fresh outer `SignedTransaction` with a valid mainnet `block_hash`. If the victim's account exists on mainnet with the same key pair and the nonce is valid on mainnet, the inner actions execute on mainnet without the victim's consent. The corrupted protocol values are: the victim's mainnet account balance, contract state, and access key nonces — all modified by actions the victim only authorized for a different network.

### Likelihood Explanation
The attack requires a malicious relayer (the natural recipient of `SignedDelegateAction` in the meta-transaction flow), the victim having the same account ID and key pair on multiple NEAR networks (common for developers and dApp users who use the same wallet across testnet/mainnet), and a valid nonce on the target network. The nonce constraint is the primary mitigation, but if the victim's nonce on mainnet is lower than the one used in the testnet delegate action, the replay succeeds. The `max_block_height` is a block height, not a timestamp, and block heights across networks are independent, so this provides limited cross-network protection.

### Recommendation
Add a `chain_id` or genesis block hash field to `DelegateAction` and `DelegateActionV2`, or incorporate it into the `SignableMessage` discriminant / `MessageDiscriminant`. This mirrors how `SignedTransaction` binds to a specific chain via `block_hash`. The `SignableMessage` wrapper in `core/primitives/src/signable_message.rs` is the natural place to add a network-scoping field, since it already provides the domain-separation prefix.

### Proof of Concept
1. Alice signs a `DelegateAction` on testnet: `sender_id=alice.near`, `receiver_id=token.near`, `actions=[Transfer(100 NEAR)]`, `nonce=5`, `max_block_height=10_000_000`, `public_key=ed25519:AliceKey`. She sends the `SignedDelegateAction` to a relayer.
2. The malicious relayer, instead of submitting to testnet, constructs a new outer `SignedTransaction` with a valid mainnet `block_hash`, wrapping Alice's unchanged `SignedDelegateAction`.
3. On mainnet, `apply_delegate_action` calls `signed_delegate_action.verify()` — this passes because the signature is over the chain-agnostic payload. It checks `max_block_height` against mainnet's block height — passes if mainnet height < 10,000,000. It checks the nonce against Alice's mainnet access key nonce — passes if Alice's mainnet nonce < 5.
4. Alice's mainnet account transfers 100 NEAR to `token.near` on mainnet, an action she never authorized for mainnet. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }
