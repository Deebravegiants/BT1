Audit Report

## Title
Missing Minimum Output (Slippage) Protection in L2 RSETHPool `deposit()` Functions - (File: contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol)

## Summary
All public-facing `deposit()` functions across the L2 RSETHPool family compute the rsETH/wrsETH output at execution time using a live oracle rate but accept no `minRsETHAmountExpected` parameter. If the oracle rate increases between transaction submission and execution, the user receives fewer rsETH/wrsETH tokens than anticipated with no on-chain mechanism to revert. The L1 counterpart `LRTDepositPool.sol` already enforces this guard, confirming the omission is an oversight rather than a design choice.

## Finding Description
Every `deposit()` overload in the L2 pool family follows the same pattern: compute `rsETHAmount` from the live oracle rate, then immediately transfer or mint that amount to the caller with no floor check.

**`RSETHPool.sol` ETH path** ( [1](#0-0) ):
```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount); // no min check
```

**`RSETHPool.sol` token path** ( [2](#0-1) ): identical pattern.

**`RSETHPoolNoWrapper.sol`** ( [3](#0-2) ): identical pattern for both ETH and token paths.

**`RSETHPoolV2ExternalBridge.sol`** ( [4](#0-3) ): identical pattern, mints instead of transfers.

**`RSETHPoolV3ExternalBridge.sol`** ( [5](#0-4) ): identical pattern for both ETH and token paths.

The rate formula is `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. Any increase in `rsETHToETHrate` between submission and execution silently reduces `rsETHAmount`.

By contrast, `LRTDepositPool.sol` accepts and enforces `minRSETHAmountExpected` in `_beforeDeposit` ( [6](#0-5) ):
```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [7](#0-6) 

The L2 pool contracts have no equivalent guard. No privileged access or attacker action is required; normal oracle updates during routine protocol operation are sufficient to trigger the condition.

## Impact Explanation
A depositor submits a transaction expecting a specific rsETH/wrsETH amount based on the rate visible at submission time. If the oracle updates before the transaction is mined, the user receives fewer tokens than expected with no ability to revert. The deposited ETH/token remains in the pool; the user cannot recover the shortfall. This matches the allowed Low impact: **Contract fails to deliver promised returns, but doesn't lose value** — the user's deposited principal is not stolen, but the rsETH output is silently reduced below their expectation.

## Likelihood Explanation
No attacker is required. Oracle rate updates occur periodically as part of normal protocol operation. On L2 chains with short block times, a pending deposit transaction can be executed in the same block as or immediately after an oracle update. Every depositor on every supported L2 chain is affected on any deposit that races with an oracle update. The condition is repeatable and requires no special setup.

## Recommendation
Add a `minRsETHAmountExpected` parameter to all `deposit()` overloads in every RSETHPool variant, mirroring the pattern already used in `LRTDepositPool.sol`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    feeEarnedInETH += fee;
    IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same pattern to all token-path overloads and to `RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`, and `RSETHPoolV3ExternalBridge`.

## Proof of Concept
1. Observe `rsETHToETHrate = 1.05e18` on-chain. Call `RSETHPool.deposit{value: 1 ether}("ref")`, expecting ≈ 0.952 wrsETH (`1e18 * 1e18 / 1.05e18`).
2. Before the transaction is mined, the oracle updates `rsETHToETHrate` to `1.10e18`.
3. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 wrsETH`.
4. User receives ~4.5% fewer wrsETH than expected; the 1 ETH remains in the pool with no recovery path.
5. Repeat for token-path deposits in `RSETHPool`, `RSETHPoolNoWrapper`, and `RSETHPoolV3ExternalBridge`.

**Foundry fork test plan:** Fork an L2 where an RSETHPool is deployed. Call `viewSwapRsETHAmountAndFee` to record the expected output. Use `vm.store` to advance the oracle rate. Call `deposit` and assert the received amount is less than the pre-recorded expectation, with no revert.
