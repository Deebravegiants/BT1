### Title
FunctionCallPermission Allowance Race Condition Allows Key Holder to Spend Old + New Allowance - (`runtime/runtime/src/verifier.rs`)

### Summary

The `FunctionCallPermission.allowance` field in NEAR's access key system has no atomic "change allowance" operation. The only way to change an allowance is to delete the old key and add a new one. A key holder who observes a pending allowance-reduction transaction can frontrun it — spending the full old allowance N before the delete lands, then spending the new allowance M after the re-add — consuming N+M tokens from the account instead of the intended M.

### Finding Description

`FunctionCallPermission` carries an `allowance` field that limits how much NEAR can be spent on gas fees by the holder of that key. The protocol documentation explicitly states:

> "NOTE: To change or increase the allowance, the old access key needs to be deleted and a new access key should be created." [1](#0-0) 

This means the only way for an account owner to reduce a key's allowance from N to M is to submit a transaction containing `[DeleteKey(K), AddKey(K, allowance=M)]`. Even though these two actions are atomic within a single transaction, the transaction itself is not yet included in a block when it enters the mempool. A key holder who monitors the mempool can observe this pending transaction and immediately submit their own transactions using key K — spending up to N tokens on gas fees — before the delete+add transaction is included.

After the delete+add transaction lands, the key K is re-created with allowance M. The key holder can then spend M more tokens on gas fees. The net result is N+M tokens spent instead of M.

The allowance check is performed in `check_and_compute_new_allowance`: [2](#0-1) 

This function correctly enforces the current allowance at transaction-verification time, but it cannot prevent a key holder from racing against a pending allowance-change transaction. The allowance is decremented per-transaction and written back via `AccessKeyUpdate::Regular { nonce, new_allowance }`: [3](#0-2) 

The new key is added with the reduced allowance via `action_add_key`: [4](#0-3) 

There is no protocol-level "compare-and-set allowance" or "decrease allowance by delta" operation that would prevent the race.

### Impact Explanation

The account owner's NEAR balance is reduced by more than intended. Gas fees for function calls using a FunctionCall key are deducted from both the account balance and the key's allowance. By spending the old allowance N before the delete lands, the key holder drains N tokens from the account's balance in gas fees. After the re-add with allowance M, they drain M more. The corrupted DB entry is the account's `amount` field (balance) in the trie state, which ends up lower than the account owner intended to permit.

### Likelihood Explanation

The key holder already possesses the private key K and can submit transactions at any time. Observing a pending `[DeleteKey, AddKey]` transaction in the mempool is straightforward via any RPC node. The key holder simply needs to submit one or more function-call transactions using key K before the allowance-change transaction is included. NEAR's mempool does not provide ordering guarantees between transactions from different accounts, so the race is winnable by any key holder who acts promptly.

### Recommendation

Introduce an atomic `ChangeAllowance` action (or `DecreaseAllowance` / `IncreaseAllowance` pair) that modifies the allowance in-place without deleting and re-creating the key. This eliminates the window between delete and add. Alternatively, follow the ERC-20 mitigation pattern: require the caller to supply the expected current allowance, and reject the change if the on-chain allowance does not match, making the operation a compare-and-swap rather than a blind set.

### Proof of Concept

1. Account A creates FunctionCall key K for User B with `allowance = N` (e.g., 1 NEAR).
2. Account A decides to reduce the allowance to `M = 0.1 NEAR` and submits:
   `Transaction { actions: [DeleteKey(K), AddKey(K, allowance=0.1 NEAR)] }`
3. User B monitors the mempool, sees the pending transaction, and immediately submits function-call transactions signed with key K, consuming up to N = 1 NEAR in gas fees before the delete lands.
4. Account A's transaction is included: key K is deleted and re-added with allowance 0.1 NEAR.
5. User B submits more function-call transactions, consuming 0.1 NEAR more in gas fees.
6. Total gas fees charged to Account A's balance: 1.1 NEAR instead of the intended 0.1 NEAR.

The broken invariant is enforced at: [5](#0-4) 

which correctly enforces the *current* allowance but has no mechanism to prevent the race against a pending allowance-change transaction.

fn check_and_compute_new_allowance(
    access_key: &AccessKey,
    account_id: &AccountId,
    public_key: &PublicKey,
    total_cost: Balance,
) -> Result<Option<Balance>, InvalidTxError> {
    let Some(fc) = access_key.permission.function_call_permission() else {
        return Ok(None);
    };
    let Some(allowance) = fc.allowance else {
        return Ok(None);
    };
    let new_allowance = allowance.checked_sub(total_cost).ok_or_else(|| {
        InvalidTxError::InvalidAccessKeyError(InvalidAccessKeyError::NotEnoughAllowance {
            account_id: account_id.clone(),
            public_key: public_key.clone().into(),
            allowance,
            cost: total_cost,
        })
    })?;
    Ok(Some(new_allowance))
}

pub(crate) fn action_add_key(
    apply_state: &ApplyState,
    state_update: &mut TrieUpdate,
    account: &mut Account,
    result: &mut ActionResult,
    account_id: &AccountId,
    add_key: &AddKeyAction,
) -> Result<(), StorageError> {
    if get_access_key(state_update, account_id, &add_key.public_key)?.is_some() {
        result.result = Err(ActionErrorKind::AddKeyAlreadyExists {
            account_id: account_id.to_owned(),
            public_key: add_key.public_key.clone().into(),
        }
        .into());
        return Ok(());
    }

    let fee_config = &apply_state.config.fees;

    if let Some(gas_key_info) = add_key.access_key.gas_key_info() {
        add_gas_key(
            fee_config,
            state_update,
            account,
            account_id,
            &add_key.public_key,
            &add_key.access_key,
            gas_key_info,
            apply_state.block_height,
        )?;
    } else {
        add_regular_key(
            fee_config,
            state_update,
            account,
            account_id,
            &add_key.public_key,
            &add_key.access_key,
            apply_state.block_height,
        )?;
    }

    Ok(())
}
