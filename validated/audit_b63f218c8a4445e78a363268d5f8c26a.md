### Title
Pending `sign` requests become permanently unresolvable after epoch resharing — (File: `crates/contract/src/lib.rs`)

---

### Summary
The `respond` function verifies a submitted threshold signature against the **current** domain public key at response time. The `SignatureRequest` struct does not record the epoch or the specific public key active at submission time. After resharing completes and the contract transitions to a new `RunningContractState` with a new keyset, any in-flight sign requests from the previous epoch can never be fulfilled: nodes computed their signatures under old key shares, but the contract now verifies against the new public key. Every such request is permanently stuck until the NEAR yield-resume timeout fires and the user's transaction fails.

---

### Finding Description

When a user calls `sign`, the contract creates a `SignatureRequest` containing `domain_id`, `payload`, and a `tweak` derived from the caller and path. It does **not** record the epoch ID or the public key active at submission time. [1](#0-0) 

When a node later calls `respond`, the contract fetches the public key at response time:

```rust
let domain = request.domain_id;
let public_key = self.public_key_extended(domain)?;
``` [2](#0-1) 

`public_key_extended` dispatches to `ProtocolContractState::public_key`:

```rust
pub fn public_key(&self, domain_id: DomainId) -> Result<PublicKeyExtended, Error> {
    match self {
        ProtocolContractState::Running(state) => state.keyset.public_key(domain_id),
        ProtocolContractState::Resharing(state) => {
            state.previous_keyset().public_key(domain_id)
        }
        _ => Err(InvalidState::ProtocolStateNotRunningNorResharing.into()),
    }
}
``` [3](#0-2) 

During resharing the previous keyset is used (correct). But once `vote_reshared` accumulates enough votes and the state transitions back to `Running` with the new keyset:

```rust
return Ok(Some(RunningContractState::new(
    new_domains,
    Keyset::new(self.prospective_epoch_id(), self.reshared_keys.clone()),
    ...
)));
``` [4](#0-3) 

…the contract now returns the **new** public key for every subsequent `respond` call. Any node that computed a threshold signature under the old key shares (epoch E) and submits it after resharing completes (epoch E+1) will fail signature verification:

```rust
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
``` [5](#0-4) 

The pending yield is never resumed. After ~200 blocks the NEAR runtime fires the yield timeout, `return_signature_and_clean_state_on_success` is called with `Err(PromiseError::Failed)`, and the user's transaction fails. [6](#0-5) 

The same stale-key problem applies to `respond_ckd`, which also calls `self.public_key_extended(request.domain_id)` at response time with no epoch anchor in the stored `CKDRequest`. [7](#0-6) 

---

### Impact Explanation

Any sign (or CKD) request accepted during epoch E whose threshold computation completed before resharing finishes becomes permanently unresolvable once the contract enters epoch E+1. The user's foreign-chain transaction cannot be signed; the user must resubmit. The contract accepted the request and the nodes performed the cryptographic work, but the result can never be delivered. This breaks the request-lifecycle safety invariant without any direct fund theft.

**Impact: Medium** — request-lifecycle invariant broken; accepted requests are silently dropped after resharing.

---

### Likelihood Explanation

Resharing is a normal, expected protocol operation triggered by a threshold
