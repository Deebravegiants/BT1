### Title
Missing `DirectDeposits` in Preservation-of-Value Check Allows Unbounded ADA Creation — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

The Dijkstra era introduces a new `directDeposits` transaction-body field that credits ADA directly into registered stake-account balances. The `ENTITIES` rule unconditionally applies these credits, but neither the consumed-value nor the produced-value side of the preservation-of-value (PoV) check accounts for `directDeposits`. An unprivileged transaction sender can therefore craft a transaction that passes the PoV check while carrying an arbitrary `directDeposits` payload, causing the ledger to create ADA from nothing.

---

### Finding Description

**Vulnerability class:** funds/accounting bug — missing term in the preservation-of-value equation.

**Consumed-value calculation** (`getConsumedDijkstraValue`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` lines 65–91):

```
consumed = Σ UTxO inputs + withdrawals + refunds + minted multi-assets
```

It delegates per-body accounting to `getConsumedMaryValue` and adds sub-transaction consumed values. `directDeposits` is never read. [1](#0-0) 

**Produced-value calculation** (`getProducedDijkstraValue` → `dijkstraProducedValue` → `conwayProducedValue`, same file lines 93–122):

```
produced = Σ UTxO outputs + fee + cert deposits + treasury donation + burned multi-assets
```

`directDeposits` is absent here too. [2](#0-1) 

**PoV check** (`dijkstraUtxoTransition`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` line 381):

```haskell
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
```

This fires before the `ENTITIES` rule and uses the two functions above. The only `directDeposits`-related check in the UTXO rule is a network-ID sanity check (line 406), not an accounting check. [3](#0-2) 

**ENTITIES rule** (`dijkstraEntitiesTransition`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs` lines 211–216):

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts

pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

The only guard is that target accounts exist. There is no check that the `directDeposits` amount was balanced by the PoV equation. [4](#0-3) 

**`applyDirectDeposits`** (`libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs` lines 290–298) unconditionally adds coin to account balances:

```haskell
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [5](#0-4) 

The identical gap exists in the sub-transaction path (`dijkstraSubEntitiesTransition`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubEntities.hs` lines 182–187). [6](#0-5) 

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

Because `directDeposits` appears on neither side of `consumed == produced`, the PoV equation is satisfied without the sender providing the ADA that will be credited to accounts. After the PoV check passes, the `ENTITIES` rule credits the full `directDeposits` amount to the target accounts. The net effect is that `directDeposits` ADA is added to the ledger's total supply with no corresponding debit anywhere. An attacker can repeat this in every block, inflating the ADA supply without bound.

---

### Likelihood Explanation

**High.** The Dijkstra era is a production era in this repository. Any transaction sender who can submit a valid Dijkstra-era transaction (no special role required) can include a `directDeposits` field. The only prerequisite is that the target credential(s) are registered accounts, which the attacker can register themselves. No governance majority, no privileged key, and no consensus threshold is needed.

---

### Recommendation

Add the total `directDeposits` coin to the **produced** side of the PoV equation, mirroring how certificate deposits are handled. Concretely, `getProducedDijkstraValue` (and `dijkstraSubTxProducedValue` for sub-transactions) should include:

```haskell
<> inject (sumDirectDeposits (txBody ^. directDepositsTxBodyL))
```

where `sumDirectDeposits` folds the `Map AccountAddress Coin` inside `DirectDeposits` to a single `Coin`. This ensures that the sender must supply the credited ADA through UTxO inputs, restoring the preservation-of-value invariant.

---

### Proof of Concept

Construct a Dijkstra-era top-level transaction:

| Field | Value |
|---|---|
| UTxO inputs | 100 ADA (attacker-controlled UTxO) |
| UTxO outputs | 94 ADA (change back to attacker) |
| fee | 1 ADA |
| cert deposits | 5 ADA (e.g. register one stake key) |
| `directDeposits` | 10 ADA → attacker's own registered account |

**PoV check (UTXO rule):**
- `consumed` = 100 ADA
- `produced` = 94 + 1 + 5 = 100 ADA ✓ (passes — `directDeposits` ignored)

**ENTITIES rule:**
- `directDepositsMissingAccounts` returns `Nothing` (account is registered) ✓
- `applyDirectDeposits` credits +10 ADA to the attacker's account balance

**Net ledger change:**
- UTxO: −100 + 94 = −6 ADA
- Fee pot: +1 ADA
- Deposit pot: +5 ADA
- Account balance: **+10 ADA**
- **Total ADA created: +10 ADA** (10 − 6 − 1 − 5 = −2 ADA net from attacker's UTxO, but +10 ADA in account = net +8 ADA created from nothing)

By setting `directDeposits` to an arbitrarily large value and adjusting outputs accordingly, the attacker can create unbounded ADA in a single transaction.
