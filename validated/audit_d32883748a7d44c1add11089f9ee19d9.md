### Title
`EraUTxO DijkstraEra` Uses Conway-Only `getMinFeeTxUtxo`, Omitting Sub-Transaction Reference Script Fees - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The `EraUTxO DijkstraEra` typeclass instance sets `getMinFeeTxUtxo = getConwayMinFeeTxUtxo`, which only measures reference-script size for the **top-level transaction**. Dijkstra introduces sub-transactions, and a dedicated helper `batchNonDistinctRefScriptsSize` already exists to aggregate reference-script sizes across all sub-transactions. However, it is never wired into `getMinFeeTxUtxo`. The result is that sub-transaction reference scripts are entirely free from the fee perspective, while the size-limit guard (`validateAllRefScriptSize`) correctly uses `batchNonDistinctRefScriptsSize`. This is a direct analog of the Fairswap interface-inconsistency class: the same conceptual operation (reference-script cost accounting) is implemented differently in two places that are supposed to be consistent.

---

### Finding Description

**Root cause — inconsistent implementations of the same interface method:**

`EraUTxO` defines `getMinFeeTxUtxo` as the single authoritative method for minimum-fee computation. For `DijkstraEra` the instance is:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs  line 141
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` delegates to `txNonDistinctRefScriptsSize`, which only inspects the top-level transaction's `inputsTxBodyL` and `referenceInputsTxBodyL`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs  lines 174-175
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
``` [2](#0-1) 

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs  lines 183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

Dijkstra already provides `batchNonDistinctRefScriptsSize` that correctly aggregates across all sub-transactions:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs  lines 271-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [4](#0-3) 

The **size-limit guard** in the LEDGER rule correctly uses `batchNonDistinctRefScriptsSize`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs  lines 321-322
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
``` [5](#0-4) 

So the two implementations of the same conceptual operation diverge:

| Check | Function used | Sub-tx ref scripts counted? |
|---|---|---|
| Minimum fee (`getMinFeeTxUtxo`) | `txNonDistinctRefScriptsSize` | **No** |
| Size limit (`validateAllRefScriptSize`) | `batchNonDistinctRefScriptsSize` | **Yes** |

This is the direct Cardano Ledger analog of the Fairswap interface inconsistency: the same conceptual operation (`reference-script cost accounting`) is implemented differently across two places that are supposed to be consistent, with no shared interface enforcing the correct behavior.

---

### Impact Explanation

The reference-script fee was introduced specifically to prevent DDoS attacks where large scripts are cheap to submit but expensive to deserialize and validate (see ADR-009). [6](#0-5) 

Because `getMinFeeTxUtxo` for `DijkstraEra` ignores sub-transaction reference scripts, an attacker can:

1. Construct a top-level transaction with no reference scripts (fee = base fee only).
2. Embed sub-transactions that each reference large Plutus scripts up to the per-transaction size limit (`ppMaxRefScriptSizePerTxG`).
3. The UTXO rule enforces the minimum fee using `getMinFeeTxUtxo`, which charges **zero** for the sub-transaction reference scripts.
4. The node must still deserialize and validate every sub-transaction reference script, incurring full computational cost.

This allows attacker-controlled transactions to modify fees outside design parameters — the fee paid is lower than the cost imposed on validators. This matches the **Medium** impact: *attacker-controlled transactions modify fees outside design parameters*.

---

### Likelihood Explanation

- Any unprivileged transaction submitter can craft such a transaction.
- No special keys, governance majority, or privileged access is required.
- The Dijkstra era is the current production-bound era in this repository, making this reachable on mainnet once deployed.
- The attack pattern is well-understood (it mirrors the June 2024 DDoS attack that motivated ADR-009).

---

### Recommendation

Override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance to use `batchNonDistinctRefScriptsSize` instead of the inherited Conway implementation:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo pp tx utxo =
    getMinFeeTx pp tx (batchNonDistinctRefScriptsSize utxo tx)
```

This mirrors the fix recommended in the Fairswap report: enforce the shared interface in a single place so all implementations are consistent. The `EraUTxO` typeclass is the correct enforcement point — just as Fairswap's fix was to inherit from a shared interface file.

### Proof of Concept

**Attacker constructs a Dijkstra `TopTx`:**

```
TopTx {
  inputs        = [some_utxo],
  referenceInputs = [],          -- no top-level ref scripts → fee = base fee
  fee           = base_fee_only,
  subTransactions = [
    SubTx { referenceInputs = [utxo_with_200KB_plutus_script] },
    SubTx { referenceInputs = [utxo_with_200KB_plutus_script] },
    ...  -- up to ppMaxRefScriptSizePerTx
  ]
}
```

**Fee check path** (`UTXO` rule → `getMinFeeTxUtxo`):
- Calls `getConwayMinFeeTxUtxo pp tx utxo`
- Calls `txNonDistinctRefScriptsSize utxo tx` → sees 0 bytes of ref scripts (top-level only)
- Minimum fee = `base_fee_only` ✓ (passes)

**Size limit check path** (`LEDGER` rule → `validateAllRefScriptSize`):
- Calls `batchNonDistinctRefScriptsSize utxo tx` → sees all sub-tx ref scripts
- If total ≤ `ppMaxRefScriptSizePerTx` → passes ✓

**Result:** Transaction accepted with base fee only, but the node must deserialize and validate all sub-transaction reference scripts — the same DDoS vector that ADR-009 was designed to close. [7](#0-6) [8](#0-7)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability.
