### Title
Backup-CLI `secrets.json` Written Without Restrictive File Permissions, Exposing MPC Keyshare Decryption Key and mTLS Authentication Credential ŌĆö (File: `crates/backup-cli/src/adapters/secrets_storage.rs`)

---

### Summary

The `backup-cli` tool writes its `secrets.json` file ŌĆö which contains both the mTLS private key used to authenticate as the registered backup service and the AES-128 key used to decrypt locally stored MPC keyshares ŌĆö using default OS file permissions (no explicit `mode` set). This contrasts with the MPC node's own `write_secret_file`, which explicitly enforces `0o600`. An attacker with filesystem read access (e.g., via a misconfigured umask, compromised co-process, or unencrypted backup) can steal both the authentication credential and the keyshare decryption key from a single file, enabling unauthorized access to a node's MPC key share.

---

### Finding Description

**Root cause ŌĆö `crates/backup-cli/src/adapters/secrets_storage.rs`, lines 55ŌĆō65:**

```rust

    let destination = File::options()

        .truncate(true)

``` [1](#0-0) 

No explicit permission mode is set. Under a typical Linux umask of `0o022`, the resulting file is `0o644` ŌĆö world-readable.

**What the file contains ŌĆö `crates/backup-cli/src/types.rs`, lines 6ŌĆō13:**

```rust

    pub p2p_private_key: SigningKey,       // mTLS key registered on-chain

The `local_storage_aes_key` is the exact key passed to `LocalPermanentKeyStorageBackend` to decrypt the AES-128-GCM-encrypted keyshare files stored on disk: [3](#0-2) 

**Contrast with the MPC node's correct approach ŌĆö `crates/node/src/config.rs`, lines 192ŌĆō208:**

```rust
fn write_secret_file(path: &Path, data: &[u8]) -> anyhow::Result<()> {
    let mut file = fs::OpenOptions::new()
        .write(true).create(true).truncate(true)
        .mode(0o600)          // ŌåÉ explicit owner-only
        .open(path)?;
    file.write_all(data)?;
    fs::set_permissions(path, fs::Permissions::from_mode(0o600))?;
    Ok(())
}
``` [4](#0-3) 

The MPC node's `secrets.json` is written with `0o600`. The backup-cli's `secrets.json` is not.

**How the `p2p_private_key` enables direct keyshare exfiltration:**

The MPC node's migration server authenticates the connecting backup service by comparing the peer's TLS public key against the contract-registered value: [5](#0-4) 

Because the stolen `p2p_private_key` is the exact key registered on-chain, an attacker presenting it passes this check and can request keyshares directly from the MPC node's migration endpoint ŌĆö without needing the encrypted keyshare files at all.

---

### Impact Explanation

An attacker who reads the backup-cli's `secrets.json` (world-readable under default umask) obtains:

1. **`p2p_private_key`** ŌĆö the on-chain-registered mTLS credential. The attacker can impersonate the backup service, connect to the MPC node's migration endpoint, and receive the node's keyshare directly.
2. **`local_storage_aes_key`** ŌĆö the AES-128 key protecting any keyshare files already stored on disk. The attacker can decrypt them offline.

Either path yields unauthorized access to one MPC node's key share. This constitutes unauthorized access to MPC key material as defined in the Critical impact scope.

---

### Likelihood Explanation

- The backup-cli is explicitly documented as running on the operator's standard (non-TEE) infrastructure during Soft Launch.
- The documentation itself warns: *"Keep the `secrets.json` file secure. Anyone with access to this file can authenticate as your backup service and decrypt any keyshares stored locally."*
- Under a default Linux umask (`0o022`), the file is created as `0o644` (world-readable), meaning any local user or process on the same machine can read it without elevated privileges.
- Unencrypted filesystem backups (e.g., rsync, tar archives) would also expose the file. [6](#0-5) 

---

### Recommendation

Apply the same pattern used by the MPC node's `write_secret_file`. In `JsonSecretsStorage::open_write`, set explicit `0o600` permissions at file creation and enforce them after writing:

```rust
use std::os::unix::fs::OpenOptionsExt;

let destination = tokio::fs::OpenOptions::new()
    .create(true)
    .truncate(true)
    .write(true)
    .mode(0o600)          // owner read/write only
    .open(file_path)
    .await
    .map_err(Error::OpenFile)?;
```

Additionally, after writing, call `set_permissions` to enforce `0o600` even if the file pre-existed with looser permissions, mirroring the MPC node's double-enforcement pattern. [7](#0-6)

### Proof of Concept

1. Operator runs `backup-cli --home-dir /backup generate-keys`. The file `/backup/secrets.json` is created with permissions `0o644` (world-readable under default umask).
2. Operator runs `backup-cli get-keyshares ...`, which stores the encrypted keyshare in `/backup/permanent_keys/`.
3. Attacker (any local user or process) reads `/backup/secrets.json`, extracting `p2p_private_key` and `local_storage_aes_key`.
4. **Path A (offline):** Attacker reads `/backup/permanent_keys/` encrypted keyshare files and decrypts them using the stolen `local_storage_aes_key` via `Aes128Gcm::new(&local_storage_aes_key.into())` + `db::decrypt(...)`.
5. **Path B (online):** Attacker uses the stolen `p2p_private_key` to establish a mTLS connection to the MPC node's migration endpoint (`port 8079`). The node's `authenticate_peer` check passes because the TLS public key matches the on-chain-registered backup service key. The attacker issues `GET /get_keyshares` and receives the AES-256-encrypted keyshare, then decrypts it using the separately provided `backup_encryption_key_hex`. [8](#0-7) [9](#0-8)

pub fn authenticate_peer(

    expected_peer_public_key: &VerifyingKey,

    let peer_public_key = mpc_tls::tls::extract_public_key(common_state)?;

    if peer_public_key != *expected_peer_public_key {
