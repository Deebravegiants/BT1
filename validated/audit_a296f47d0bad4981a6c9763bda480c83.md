### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Allows Any Caller to Obtain MPC-Signed Foreign-Chain Verification Without Authorization — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` entry point constructs its pending-request map key without the caller's account ID, making it caller-agnostic. Any unprivileged NEAR account can submit the identical foreign-chain verification request and receive the same MPC-signed response that was produced for the original requester. Because the signed payload also omits the caller's identity, the resulting signature is reusable by anyone who holds it, enabling cross-chain replay and invalid bridge execution.

---

### Finding Description

**Root cause — missing caller binding in the request key**

`sign` explicitly binds its request key to the caller:

```rust
// lib.rs:379-384
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // ← caller account ID is part of the key
    &request.path,
);
``` [1](#0-0) 

`verify_foreign_transaction` does not:

```rust
// lib.rs:549-556
let request = args_into_verify_foreign_tx_request(request);   // no predecessor
let callback_args = serde_json::to_vec(&(&request,)).unwrap();
self.enqueue_yield_request(
    method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
    callback_args,
    callback_gas,
    move |this, id| this.add_verify_foreign_tx_request(request, id),
);
``` [2](#0-1) 

The codebase itself acknowledges this in the sandbox test:

```rust
// lib.rs:3255
// Then: both yields are queued under the single (caller-agnostic) request key.
assert_eq!(
    contract.pending_verify_foreign_tx_requests.get(&request).map(|q| q.len()),
    Some(2),
    "duplicate foreign-tx requests from different callers should fan out",
);
``` [3](#0-2) 

**Root cause — signed payload also omits caller identity**

`respond_verify_foreign_tx` verifies the MPC signature against the **root** public key, not a per-caller derived key:

```rust
// lib.rs:729-734
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // ← root key, no derivation from caller
)
.is_ok()
``` [4](#0-3) 

The `ForeignTxSignPayload` that the MPC nodes sign contains only the foreign-chain request data and extracted values — no NEAR account ID:

```rust
// lib.rs:3266-3272 (test, showing payload structure)
let payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: vec![ExtractedValue::BitcoinExtractedValue(...)],
});
``` [5](#0-4) 

When `resolve_yields_for` drains the queue it resumes **every** queued yield with the same serialised response bytes:

```rust
// pending_requests.rs:74-81
let resumed = requests
    .remove(request)
    .unwrap_or_default()
    .into_iter()
    .map(|YieldIndex { data_id }| {
        env::promise_yield_resume(&data_id, response_bytes.clone());
    })
    .count();
``` [6](#0-5) 

Every caller who queued under the same key receives an identical, valid MPC signature.

---

### Impact Explanation

A bridge contract that calls `verify_foreign_transaction` to gate fund releases receives a signed attestation that a foreign-chain transaction (e.g., a Bitcoin UTXO spend) occurred. Because the attestation is not bound to any NEAR account, an attacker who obtains the same signed bytes can present them to the bridge contract as if they were the original requester. If the bridge contract does not independently re-derive the intended beneficiary from the foreign-chain transaction data, the attacker can trigger an invalid fund release — a double-spend or cross-chain replay condition.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

The attack requires no special privilege:

1. The NEAR blockchain is public; the attacker watches for `verify_foreign_transaction` calls.
2. The attacker submits the identical `VerifyForeignTransactionRequestArgs` (same `domain_id`, `payload_version`, and `request` fields) before the MPC nodes respond — a window of several seconds to minutes.
3. The attacker's yield is queued under the same caller-agnostic key and receives the same signed response.
4. The attacker presents the response to any bridge contract that does not re-check the beneficiary.

The attack is automatable, requires only 1 yoctoNEAR deposit, and the foreign-chain transaction data needed to replicate the request is fully public.

---

### Recommendation

Mirror the design of `sign`: include `env::predecessor_account_id()` in the `VerifyForeignTransactionRequest` key and in the signed payload (`ForeignTxSignPayload`). This binds each MPC-issued attestation to the specific NEAR account that requested it, preventing any other account from reusing the signature. Bridge contracts can then verify the caller binding on-chain without additional trust assumptions.

---

### Proof of Concept

The existing unit test at `crates/contract/src/lib.rs:3209–3297` already demonstrates the full attack surface:

1. Alice (`alice.near`) calls `verify_foreign_transaction` with `request_args`.
2. Bob (`bob.near`) calls `verify_foreign_transaction` with the **identical** `request_args`.
3. Both yields are stored under the same key (`pending_verify_foreign_tx_requests` length = 2).
4. A single `respond_verify_foreign_tx` call drains both yields; both Alice and Bob receive the identical MPC-signed `VerifyForeignTransactionResponse`.
5. Bob — who has no legitimate claim to the foreign-chain transaction — now holds a valid, root-key-signed attestation he can replay against any bridge contract. [7](#0-6)

    fn verify_foreign_transaction__should_queue_duplicates_from_different_callers() {
        // Given: two different callers will submit the same foreign-tx verification request.
        let mut rng = rand::rngs::StdRng::from_seed([42u8; 32]);
        let (context, mut contract, secret_key) =
            basic_setup_with_protocol(Protocol::CaitSith, DomainPurpose::ForeignTx, &mut rng);
        register_supported_chains(&mut contract, [dtos::ForeignChain::Bitcoin]);
        let SharedSecretKey::Secp256k1(secret_key) = secret_key else {
            unreachable!();
        };

        let request_args = VerifyForeignTransactionRequestArgs {
            domain_id: DomainId::default().0.into(),
            payload_version: ForeignTxPayloadVersion::V1,
            request: dtos::ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
                tx_id: [7u8; 32].into(),
                confirmations: 2.into(),
                extractors: vec![BitcoinExtractor::BlockHash],
            }),
        };
        let request = args_into_verify_foreign_tx_request(request_args.clone());

        // When: caller alice submits the request.
        let alice = AccountId::from_str("alice.near").unwrap();
        testing_env!(
            VMContextBuilder::new()
                .signer_account_id(alice.clone())
                .predecessor_account_id(alice)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()
        );
        contract.verify_foreign_transaction(request_args.clone());

        // And: caller bob submits the identical request — a different account would today
        // be blocked from receiving a response by alice's submission.
        let bob = AccountId::from_str("bob.near").unwrap();
        testing_env!(
            VMContextBuilder::new()
                .signer_account_id(bob.clone())
                .predecessor_account_id(bob)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()
        );
        contract.verify_foreign_transaction(request_args);

        // Then: both yields are queued under the single (caller-agnostic) request key.
        assert_eq!(
            contract
                .pending_verify_foreign_tx_requests
                .get(&request)
                .map(|q| q.len()),
            Some(2),
            "duplicate foreign-tx requests from different callers should fan out",
        );

        // When: a single valid response is delivered.
        let payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: request.request.clone(),
            values: vec![ExtractedValue::BitcoinExtractedValue(
                BitcoinExtractedValue::BlockHash([42u8; 32].into()),
            )],
        });
        let payload_hash_arr = payload.compute_msg_hash().unwrap().0;
        let secret_key_ec: elliptic_curve::SecretKey<Secp256k1> =
            elliptic_curve::SecretKey::from_bytes(&secret_key.to_bytes()).unwrap();
        let signing_key = SigningKey::from_bytes(&secret_key_ec.to_bytes()).unwrap();
        let (signature, recovery_id) = signing_key
            .sign_prehash_recoverable(&payload_hash_arr)
            .unwrap();
        let response = VerifyForeignTransactionResponse {
            payload_hash: payload.compute_msg_hash().unwrap(),
            signature: dtos::SignatureResponse::Secp256k1(
                dtos::K256Signature::from_ecdsa_recoverable(&signature, recovery_id),
            ),
        };

        with_active_participant_and_attested_context(&contract);
        contract
            .respond_verify_foreign_tx(request.clone(), response)
            .expect("respond_verify_foreign_tx should succeed");

        // Then: both queued yields are drained from the single map entry.
        assert!(
            contract
                .pending_verify_foreign_tx_requests
                .get(&request)
                .is_none()
        );
