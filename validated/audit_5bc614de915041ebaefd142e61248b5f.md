Audit Report

## Title
Missing Chainlink `latestRoundData()` Return Value Validation Enables Stale-Price-Triggered Protocol Pause — (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `answeredInRound`, and does not check that `price > 0`. Because `LRTOracle.updateRSETHPrice()` is an unrestricted public function, any caller can invoke it while a Chainlink LST/ETH feed is stale, causing `_getTotalEthInProtocol()` to return an artificially low TVL, which triggers the downside-protection logic and pauses `LRTDepositPool` and `LRTWithdrawalManager`, temporarily freezing all user funds.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` (L49–55) fetches the asset/ETH exchange rate used throughout the protocol:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Three return values are discarded with no checks:
- `updatedAt` — not compared against a maximum staleness window
- `answeredInRound` — not compared to `roundId` to detect an incomplete round
- `price` — not checked to be `> 0` before casting to `uint256`

The protocol's own `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L26–36) demonstrates the correct pattern that is absent here:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The exploit path is:
1. A Chainlink LST/ETH feed (e.g., stETH/ETH, 24 h heartbeat) goes stale — `updatedAt` lags beyond the heartbeat, `answeredInRound < roundId`.
2. The stale price is below the true market rate (e.g., 5% lower).
3. Any unprivileged caller invokes `LRTOracle.updateRSETHPrice()` (public, `whenNotPaused`).
4. `_updateRsETHPrice()` calls `_getTotalEthInProtocol()`, which iterates over all supported assets and calls `getAssetPrice(asset)` for each, returning the stale value without revert.
5. `totalETHInProtocol` is underestimated; `newRsETHPrice` drops below `highestRsethPrice`.
6. If the drop exceeds `pricePercentageLimit`, the downside-protection block (L270–281) executes: `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()` are called, freezing all deposits and withdrawals.

Existing guards are insufficient: `pricePercentageLimit` is a threshold on the *computed* price drop, not a guard against stale oracle data. The `whenNotPaused` modifier on `updateRSETHPrice()` only prevents calls after a pause is already active, not before.

## Impact Explanation

**Medium — Temporary freezing of funds.**

A stale Chainlink price that causes `newRsETHPrice` to fall below `highestRsethPrice` by more than `pricePercentageLimit` triggers an automatic pause of `LRTDepositPool` and `LRTWithdrawalManager` (L277–280), preventing all user deposits and withdrawals until an admin manually unpauses. This matches the allowed impact "Temporary freezing of funds." A secondary Low impact exists if the stale price is within the limit: `rsETHPrice` is set too low, and subsequent withdrawals pay out less than users are owed ("Contract fails to deliver promised returns").

## Likelihood Explanation

No attacker action is required beyond calling the public `updateRSETHPrice()` function. Chainlink LST/ETH feeds (e.g., stETH/ETH on mainnet) have documented 24-hour heartbeat intervals. During periods of L1 congestion, sequencer downtime on L2, or feed deprecation, `updatedAt` can lag well beyond the heartbeat. This is a documented, recurring operational condition, not a theoretical edge case. Any user, bot, or keeper that calls `updateRSETHPrice()` during such a window triggers the impact.

## Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();
    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, add a per-feed configurable `maxStaleness` parameter and check `block.timestamp - updatedAt <= maxStaleness`.

## Proof of Concept

**Foundry fork test outline:**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed's `updatedAt` is within the heartbeat.
2. Warp `block.timestamp` forward by 25+ hours (beyond the 24 h heartbeat) without advancing the feed's `updatedAt`.
3. Call `LRTOracle.updateRSETHPrice()` from an unprivileged EOA.
4. Assert that `LRTDepositPool.paused()` returns `true` and `LRTWithdrawalManager.paused()` returns `true`.
5. Confirm that a subsequent `depositAsset()` call reverts with `Pausable: paused`.

This sequence requires no privileged access and is reproducible on a local fork with standard Foundry cheatcodes (`vm.warp`, `vm.roll`).
