### Title
Wrapped Mint Account Not Validated Against Signed Payload Token in `FinalizeTransfer` — (`solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

---

### Summary

The `FinalizeTransfer` instruction accepts a caller-supplied `mint` account with no constraint binding it to the token identifier in the signed payload. For the bridged-token path, only the mint authority is checked. An unprivileged attacker who holds a valid signed payload for wrapped token A can substitute the wrapped mint for token B, causing the bridge to mint the wrong token to the recipient and emit a falsified `FinalizeTransferResponse` back to NEAR.

---

### Finding Description

`DeployToken` creates every wrapped (bridged) mint as a deterministic PDA:

```rust
#[account(
    init,
    payer = common.payer,
    seeds = [WRAPPED_MINT_SEED, data.payload.token.to_hashed_bytes().as_ref()],
    bump,
    ...
)]
pub mint: Box<Account<'info, Mint>>,
``` [1](#0-0) 

`FinalizeTransfer`, however, accepts the `mint` account with only a token-program interface check:

```rust
#[account(
    mut,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
``` [2](#0-1) 

When the vault is absent (bridged-token path), `process` only verifies the mint authority:

```rust
require!(
    self.mint.mint_authority.contains(self.authority.key),
    ErrorCode::InvalidBridgedToken
);
mint_to(...)?;
``` [3](#0-2) 

There is no check that `self.mint.key()` equals the PDA derived from `data.token` (i.e., `find_program_address([WRAPPED_MINT_SEED, data.token.to_hashed_bytes()], program_id)`). Any bridge-controlled mint passes the authority check, regardless of which NEAR token it represents.

The response emitted back to NEAR uses the caller-supplied mint key, not the payload's token:

```rust
let payload = FinalizeTransferResponse {
    token: self.mint.key(),   // attacker-controlled
    ...
}
``` [4](#0-3) 

---

### Impact Explanation

An attacker who substitutes wrapped mint B for the payload's token A causes:

1. **Unauthorized minting of token B** — the bridge mints `data.amount` of token B to the recipient instead of token A, creating unbacked supply of token B on Solana.
2. **Falsified cross-chain accounting** — the `FinalizeTransferResponse` sent to NEAR reports `token: mint_B`, so NEAR's state records that token B was settled, not token A. This breaks the 1:1 backing invariant for token B.
3. **Nonce consumed** — the original payload's nonce is marked used, so the legitimate token A transfer can never be finalized; those funds are permanently locked on NEAR.

This satisfies both "unauthorized creation/release of wrapped bridge assets through verification failure" (Critical) and "asset-identity divergence that breaks backing guarantees" (High).

---

### Likelihood Explanation

The attacker only needs to:
- Hold any amount of NEAR token A and initiate a legitimate cross-chain transfer to Solana (a fully public, permissionless protocol action).
- Receive the resulting signed `FinalizeTransferPayload`.
- Call `FinalizeTransfer` substituting the wrapped mint for a more valuable token B.

No privileged access, leaked keys, or colluding parties are required. The entry point is the standard user-facing bridge flow.

---

### Recommendation

In `FinalizeTransfer::process`, after the vault-presence branch is resolved to the bridged path, add an explicit PDA derivation check:

```rust
let (expected_mint, _bump) = Pubkey::find_program_address(
    &[WRAPPED_MINT_SEED, data.token.to_hashed_bytes().as_ref()],
    ctx.program_id,
);
require_keys_eq!(self.mint.key(), expected_mint, ErrorCode::InvalidBridgedToken);
```

Alternatively, if the `FinalizeTransferPayload` already carries the Solana mint address, add a `require_keys_eq!(self.mint.key(), data.token)` guard before the `mint_to` call. Either approach closes the substitution path.

Also audit `update_metadata.rs`, which similarly accepts `mint` with only `mint::token_program = token_program` and no PDA seed constraint, to confirm no analogous privileged-path issue exists there. [5](#0-4) 

---

### Proof of Concept

1. Attacker holds 100 units of NEAR token A (low value) and initiates a bridge transfer to Solana, producing a valid `SignedPayload<FinalizeTransferPayload>` with `nonce = N`, `amount = 100`, `token = A`.
2. The bridge has previously deployed wrapped mint A (`PDA([WRAPPED_MINT_SEED, hash(A)])`) and wrapped mint B (`PDA([WRAPPED_MINT_SEED, hash(B)])`), both with `mint_authority = authority`.
3. Attacker calls `FinalizeTransfer` with:
   - `data` = the signed payload for token A
   - `mint` = wrapped mint B (a different bridge-controlled PDA)
   - `vault` = `None` (omitted, forcing the bridged path)
   - `token_account` = attacker's ATA for mint B
4. Anchor validates: `mint::token_program` passes; `vault` is absent → bridged path; `mint_authority == authority` passes for mint B.
5. `mint_to` mints 100 units of token B to the attacker's account.
6. `FinalizeTransferResponse { token: mint_B, amount: 100, ... }` is posted to Wormhole and relayed to NEAR.
7. Result: attacker receives 100 token B (potentially higher value); nonce N is consumed; NEAR records token B as settled; token A's 100 units remain locked on NEAR permanently.
