### Title
Overly Strict Foreign-Chain Availability Check Allows Sub-Threshold Byzantine Participant to Block All `verify_foreign_transaction` Requests - (File: `crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which requires **every** active participant to have registered a chain. A single Byzantine participant below the signing threshold can suppress the entire foreign-chain verification feature by simply not registering (or registering an empty set), causing all valid `verify_foreign_transaction` requests to be rejected even when a signing-threshold quorum of honest nodes is ready to serve them.

### Finding Description

`verify_foreign_transaction` calls `get_supported_foreign_chains()` to decide whether to accept a request:

```rust
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(&InvalidParameters::ForeignChainNotSupported { ... }.to_string());
}
``` [1](#0-0) 

`get_supported_foreign_chains()` computes the **strict intersection** of all active participants' registered chains:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
``` [2](#0-1) 

The correct invariant for the MPC to actually produce a signature is that **≥ signing threshold** participants cover the chain — not all of them. The contract already implements this correct check in `get_available_foreign_chains()` / `update_available_chains_config_cache()`:

```rust
.filter_map(|(chain, count)| (count >= threshold).then_some(chain))
``` [3](#0-2) 

But `verify_foreign_transaction` never calls `get_available_foreign_chains()`; it calls the strict-intersection view. The design documentation explicitly acknowledges this as a known defect:

> "A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down." [4](#0-3) 

The analog to M-8 is exact: M-8 checks collateral against the wrong (too broad) set of borrows; here the contract checks chain availability against the wrong (too broad) set of participants.

### Impact Explanation

A Byzantine participant strictly below the signing threshold can:
1. Register an empty `ForeignChainsConfig` (or simply never call `register_foreign_chains_config`).
2. `get_supported_foreign_chains()` returns an empty set (or a set missing the targeted chain).
3. Every subsequent `verify_foreign_transaction` call panics with `ForeignChainNotSupported`, regardless of how many honest nodes are ready to serve the request.

For bridge use-cases (Omnibridge inbound flow), this means foreign-chain transactions that have already been submitted and are waiting for MPC attestation cannot be completed, breaking the request lifecycle and potentially leaving bridge users unable to finalize cross-chain transfers. This breaks the production safety invariant that a request accepted by the contract can reach the signing threshold.

### Likelihood Explanation

The attacker only needs to be one active participant — a role that is privileged but explicitly listed as a valid adversary in the scope ("Byzantine participant strictly below the signing threshold"). The attack requires no collusion, no key material, and no network-level action: simply omitting the `register_foreign_chains_config` transaction (or sending one with an empty set) is sufficient. The effect is immediate and persists until the participant registers correctly.

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which already implements the correct threshold-based check. The contract already computes and caches this set in `ForeignChainsMetadata::available_foreign_chains` and exposes it via `get_available_foreign_chains()`. [5](#0-4) 

The migration path is already documented: keep `get_supported_foreign_chains()` for backward compatibility, add `get_available_foreign_chains()` (already done), then switch `verify_foreign_transaction` to gate on the threshold-based set. [6](#0-5) 

### Proof of Concept

Given a running contract with 4 participants and signing threshold 3:

1. Participants 0, 1, 2 call `register_foreign_chains_config` with `{Bitcoin}`.
2. Participant 3 (Byzantine, below threshold) never calls `register_foreign_chains_config`.
3. `get_supported_foreign_chains()` returns `{}` because participant 3 has no entry → `is_superset` fails.
4. `verify_foreign_transaction(Bitcoin, ...)` panics with `ForeignChainNotSupported`.
5. `get_available_foreign_chains()` (the correct check) would return `{Bitcoin}` because 3 ≥ threshold of 3.

The existing unit test `get_available_foreign_chains__should_include_chain_when_at_least_threshold_participants_cover_it` confirms step 5 is correct, and the existing test `get_supported_foreign_chains__should_exclude_chains_not_supported_by_all` confirms step 3 is the current production behavior. ## Migration

`get_supported_foreign_chains()` stays working throughout, so the new node version
can roll out before the contract upgrade (node and contract migrate independently):

1. Keep `get_supported_foreign_chains()` unchanged.
2. Add `get_whitelisted_foreign_chains()` and `get_available_foreign_chains()` (additive).
3. Vote the RPC providers / chains into the whitelist.
4. Upgrade the contract.
5. Switch node code to the new methods and deprecate `get_supported_foreign_chains()`.
```
