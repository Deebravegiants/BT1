### Title
ETH Deposit Limit Check Ignores Deposit Amount, Allowing Unlimited ETH Deposits Beyond Configured Cap - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check for ETH vs. ERC-20 assets. For ERC-20 tokens it correctly tests `totalAssetDeposits + amount > depositLimit`, but for ETH it only tests `totalAssetDeposits > depositLimit`, omitting the incoming `amount`. This mirrors the M-13 root cause class: a limit/ceiling value is used incorrectly in a comparison, causing the guard to be ineffective.

### Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));  // ← missing + amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch returns `true` (i.e., "limit exceeded, revert") only when the current total **already** exceeds the cap. It never accounts for the incoming `amount`. Consequently:

- When `totalAssetDeposits == depositLimit`, the check returns `false` (limit not exceeded), and the deposit proceeds, pushing the total to `depositLimit + amount`.
- When `totalAssetDeposits < depositLimit`, a deposit of any size — even one that is orders of magnitude larger than the remaining capacity — passes the check.

The companion view function `getAssetCurrentLimit` correctly computes `depositLimit - totalAssetDeposits` (the remaining capacity), but the enforcement function never uses this value for ETH. [1](#0-0)

### Impact Explanation
`depositLimitByAsset[ETH_TOKEN]` is the protocol's primary risk-management cap on ETH exposure. Bypassing it allows any depositor to push the protocol's ETH TVL arbitrarily above the intended ceiling. All excess ETH is forwarded to `NodeDelegator` contracts and staked into EigenLayer strategies. If those strategies incur a loss (slashing, smart-contract failure), the protocol holds more ETH at risk than its risk parameters permit, potentially causing **protocol insolvency** (Critical) or at minimum a state where rsETH is backed by more ETH-at-risk than the protocol's design allows (Low — contract fails to deliver promised returns).

### Likelihood Explanation
The entry path is fully permissionless: any user calling `depositETH` with `msg.value > 0` when `totalAssetDeposits <= depositLimit` triggers the bug. No special role, front-running, or external dependency is required. The condition is met naturally whenever the ETH deposit pool is at or near its configured cap, which is a normal operating state. [4](#0-3) 

### Recommendation
Apply the same `+ amount` logic to the ETH branch as is already used for ERC-20 tokens:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This unifies the check and ensures the incoming deposit amount is always considered against the cap, regardless of asset type.

### Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. Protocol accumulates `totalAssetDeposits(ETH) = 1000 ether` through normal deposits.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 10_000 ether)` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. 10,000 ETH is accepted, rsETH is minted, and the ETH is eventually staked into EigenLayer — 10× the intended cap.
