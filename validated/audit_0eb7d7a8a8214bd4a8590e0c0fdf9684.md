### Title
Per-Transaction Reference Script Size Limit Bypassed for `IsValid False` Transactions - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

In the Conway (and Dijkstra) LEDGER rule, the per-transaction reference script size check (`validateRefScriptSize`) is guarded inside an `if tx ^. isValidTxL == IsValid True` branch. Transactions tagged `IsValid False` (i.e., those whose Plutus phase-2 scripts fail) skip this check entirely, allowing an attacker to submit a transaction that references more than `maxRefScriptSizePerTx` bytes of scripts while paying only collateral.

---

### Finding Description

`conwayLedgerTransitionTRC` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs` branches on the `IsValid` tag before running several LEDGER-level checks:

```haskell
if tx ^. isValidTxL == IsValid True
  then do
    let txBody = tx ^. bodyTxL
    runTest $ validateTreasuryValue txBody (chainAccountState ^. casTreasuryL)
    runTest $ validateRefScriptSize pp (utxoState ^. utxoL) tx   -- ← only here
    ...
  else pure (utxoState, certState)   -- ← validateRefScriptSize never called
```

`validateRefScriptSize` enforces the protocol-parameter cap `maxRefScriptSizePerTx` (currently hard-coded at 200 KiB in Conway, promoted to a proper parameter in Dijkstra):

```haskell
validateRefScriptSize pp utxo tx =
  let totalRefScriptSize = txNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $
        ConwayTxRefScriptsSizeTooBig Mismatch { ... }
```

Because `feesOK` (called unconditionally in the UTXO rule) does enforce `minFee ≤ txfee` for all transactions—including the tiered reference-script component—the attacker must still pay a fee proportional to the script size. However, the **hard cap** on per-transaction script bytes is a separate, independent guard whose purpose is to bound worst-case deserialization and validation time regardless of fee level. That guard is silently skipped for every `IsValid False` transaction.

The per-block cap (`validateBodyRefScriptsSizeTooBig` in the BBODY rule) still applies to all transactions, so the block-level bound is not bypassed. But the per-transaction bound—which exists precisely to prevent any single transaction from consuming disproportionate node resources—is not enforced on the `IsValid False` path.

---

### Impact Explanation

**Medium.** An attacker-controlled transaction with `IsValid False` can exceed the intended per-transaction reference script size limit (`maxRefScriptSizePerTx`). This falls squarely within the allowed Medium impact: *"Attacker-controlled transactions … exceed intended validation limits … outside design parameters."*

Concretely, a single `IsValid False` transaction may carry reference-script bytes up to the per-block cap (1 MiB) rather than the per-transaction cap (200 KiB), forcing nodes to deserialize and hash up to 5× more script data per transaction than the protocol intends to permit. The tiered fee formula makes this economically expensive but does not eliminate the resource asymmetry, because the fee is paid in collateral at epoch end while the deserialization cost is paid by every validating node immediately.

---

### Likelihood Explanation

**Medium.** Any unprivileged transaction author can set `isValid = False` on a transaction (the block producer is responsible for setting the tag correctly, but a malicious block producer below the consensus threshold can include a self-crafted `IsValid False` transaction). Constructing a transaction with large reference inputs is trivial. The attacker must lock up and forfeit collateral proportional to the fee, which grows super-linearly with script size, making sustained exploitation expensive but not impossible for a well-funded adversary.

---

### Recommendation

Move `validateRefScriptSize` (and `validateTreasuryValue`) outside the `IsValid True` guard so they are enforced for every transaction processed by the LEDGER rule:

```haskell
conwayLedgerTransitionTRC ... = do
  let tx = stAnnTx ^. txStAnnTxG
      txBody = tx ^. bodyTxL

  -- Phase-1 checks that must hold regardless of script validity
  runTest $ validateRefScriptSize pp (utxoState ^. utxoL) tx

  (utxoState', certStateAfterCERTS) <-
    if tx ^. isValidTxL == IsValid True
      then do
        runTest $ validateTreasuryValue txBody (chainAccountState ^. casTreasuryL)
        ...
      else pure (utxoState, certState)
  ...
```

The same fix should be applied to the Dijkstra LEDGER rule (`validateAllRefScriptSize`).

---

### Proof of Concept

1. Deploy any always-failing Plutus V3 script to the UTxO as a reference script (size ≈ 250 KiB, exceeding `maxRefScriptSizePerTx = 200 KiB`).
2. Construct a transaction that:
   - Spends a UTxO locked by a different Plutus script (so redeemers are present).
   - Includes the 250 KiB script as a reference input.
   - Sets `isValid = False`.
   - Provides collateral ≥ `collateralPercent × txfee / 100` where `txfee` covers the tiered reference-script fee for 250 KiB.
3. Submit the transaction. The LEDGER rule skips `validateRefScriptSize` (line 365 is inside the `IsValid True` branch at line 361), so `ConwayTxRefScriptsSizeTooBig` is never raised. The transaction is accepted despite carrying 250 KiB of reference scripts—25% above the per-transaction cap—forcing every node to deserialize and hash the oversized script payload.
