### Title
`respond_ckd` Accepts Unvalidated CKD Output for `AppPublicKey` Variant, Enabling a Single Byzantine Participant to Deliver Fraudulent Key Derivation - (File: crates/contract/src/lib.rs)

### Summary
The `respond_ckd` function performs cryptographic output verification only for the `AppPublicKeyPV` variant of CKD requests. For the legacy `AppPublicKey` variant, the response is accepted and returned to the user with **no on-chain validation**. A single malicious attested participant — strictly below the signing threshold — can call `respond_ckd` with arbitrary `big_y`/`big_c` values, bypassing the threshold requirement and delivering a fraudulent confidential key derivation output to the requesting user.

### Finding Description

In `crates/contract/src/lib.rs`, `respond_ckd` branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO VALIDATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, cryptographically proving the output was computed from the MPC master key with the correct threshold of shares. [2](#0-1) 

For `AppPublicKey`, the empty arm means the contract immediately calls `resolve_yields_for` with whatever `big_y`/`big_c` the caller supplied — no pairing check, no point validity check, nothing. [3](#0-2) 

The only gate is `assert_caller_is_attested_participant_and_protocol_active()`. There is no restriction on *which* participant may respond, and no check that the response is the product of a threshold computation. [4](#0-3) 

**Attack path:**
1. User calls `request_app_private_key` with the `AppPublicKey` variant (still actively supported; listed first in the README).
2. A single malicious attested participant calls `respond_ckd` with crafted `big_y`/`big_c` before the legitimate coordinator.
3. The contract accepts the fraudulent response without any validation and resolves the yield.
4. The user receives the fraudulent output; the legitimate coordinator's subsequent call fails ("request not found").

The `AppPublicKey` variant is still in production use: [5](#0-4) 

The existing unit test at line 3403 demonstrates that completely arbitrary byte values (`[1u8; 48]`, `[2u8; 48]`) are accepted without error for the `AppPublicKey` variant: [6](#0-5) 

By contrast, the `AppPublicKeyPV` test at line 3596 shows the contract correctly panics on a tampered response: [7](#0-6) 

### Impact Explanation

A single Byzantine participant (strictly below the signing threshold) can deliver a fraudulent confidential key derivation output to any user who submits an `AppPublicKey` CKD request. The user receives incorrect `big_y`/`big_c` values, meaning:
- The derived application-specific private key is wrong.
- Data encrypted under the expected key cannot be decrypted.
- Signatures produced with the expected key are invalid.

This is **unauthorized confidential key derivation output without the required threshold participant authorization** — matching the Critical impact category.

### Likelihood Explanation

High. The `AppPublicKey` variant is still actively supported and is the first option listed in the README. Any single attested participant can call `respond_ckd` at any time for any pending request. No collusion is required. The attack is a straightforward front-run of the legitimate coordinator, executable by any one of the n MPC participants.

### Recommendation

The contract cannot perform the full pairing check for `AppPublicKey` (it requires the user's private key). Mitigations:

1. **Deprecate and remove** the `AppPublicKey` variant in favour of `AppPublicKeyPV`, which provides on-chain verifiable output integrity.
2. Until removal, **document explicitly** that `AppPublicKey` CKD requests carry no on-chain output integrity guarantee and are vulnerable to a single malicious participant.
3. As a partial mitigation, add a minimum validity check (e.g., that `big_y` and `big_c` are valid non-identity G1 points) to prevent trivially malformed responses.

### Proof of Concept

```
1. Deploy MPC contract with n participants, threshold t.
2. User calls request_app_private_key with AppPublicKey variant.
3. Malicious participant (any single attested node) calls respond_ckd with:
     request  = <the pending CKDRequest>
     response = CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }
4. Contract accepts the response — no validation for AppPublicKey.
5. User's yield resolves with fraudulent big_y / big_c.
6. Legitimate coordinator's respond_ckd call fails: request already resolved.
```

The existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already proves step 4: it passes `[1u8; 48]` / `[2u8; 48]` (cryptographically meaningless values) and the call succeeds. [8](#0-7)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

**File:** crates/contract/README.md (L118-121)
```markdown
- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.
