### Title
Caller-Controlled Finality Parameters in `verify_foreign_transaction` Allow Signatures on Unfinalized Foreign-Chain Transactions - (File: `crates/contract/src/lib.rs`)

### Summary
`verify_foreign_transaction` is a permissionless function that accepts user-controlled finality/confirmation parameters (`EvmFinality`, `SolanaFinality`, `BlockConfirmations`) with no minimum enforcement in the contract. Any caller can set `EvmFinality::Latest`, `SolanaFinality::Processed`, or `BlockConfirmations(0)`, causing the MPC network to produce a valid threshold signature over a transaction that has not been finalized on the foreign chain. This is a direct structural analog to the `withdrawDonation` / `_minAmount = 0` pattern: a permissionless entry point with a caller-controlled protection parameter that can be set to its weakest value.

---

### Finding Description

`verify_foreign_transaction` is callable by any NEAR account with a 1 yoctonear deposit: [1](#0-0) 

The function validates only that the requested chain is in the supported set. It performs **no validation of the finality or confirmation parameters** embedded in the `ForeignChainRpcRequest`. These parameters are defined as fully caller-controlled:

- `EvmFinality` (Ethereum, BNB, Base, Arbitrum, Polygon, HyperEvm, Abstract): `Latest | Safe | Finalized`
- `SolanaFinality`: `Processed | Confirmed | Finalized`
- `BlockConfirmations` for Bitcoin: a plain `u64` wrapping type with no lower bound [2](#0-1) [3](#0-2) 

The MPC nodes receive the request exactly as submitted and use the caller-supplied threshold when querying the foreign chain. For Bitcoin, the inspector enforces the caller-supplied `confirmations` value as the threshold — meaning `BlockConfirmations(0)` passes immediately for any mempool transaction: [4](#0-3) 

The signed payload (`ForeignTxSignPayload::V1`) includes the full `ForeignChainRpcRequest` (including the finality field), so the finality level is committed into the hash that MPC signs. However, the contract never enforces a minimum — it is left entirely to the caller to choose. [5](#0-4) 

---

### Impact Explanation

**High — Forged foreign-chain verification enabling double-spend / invalid bridge execution.**

A bridge contract consuming the MPC signature (e.g., an Omnibridge inbound flow) receives a valid threshold signature over `(request, extracted_values)`. If the consuming contract does not independently re-validate the finality level embedded in the signed payload — a reasonable assumption, since the MPC network is the trusted verifier — the attacker can:

1. Submit a `verify_foreign_transaction` with `EvmFinality::Latest` or `BlockConfirmations(0)` for a transaction that is still reversible.
2. Receive a valid MPC threshold signature.
3. Submit the signature to the bridge contract to claim funds on NEAR.
4. Wait for (or cause) a reorg on the foreign chain that reverses the original deposit transaction.
5. Reuse the foreign-chain funds — a classic double-spend via forged/premature foreign-chain verification.

This matches the allowed impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

**Medium.** The `verify_foreign_transaction` endpoint is explicitly designed for bridge use cases (Omnibridge inbound flow is the primary stated use case). Any unprivileged NEAR account can call it. The only cost is 1 yoctonear. The attacker does not need any special role, key, or threshold collusion — they only need to craft a request with a weak finality parameter. On EVM chains, `Latest` finality is a standard RPC tag; on Bitcoin, setting `confirmations: 1` (or `0`) is trivially expressible. The profit motive (double-spending a bridge deposit) is concrete and direct.

---

### Recommendation

The contract should enforce a **minimum finality level per supported foreign chain** stored in the on-chain `ForeignChainsMetadata` configuration (already used for the RPC provider whitelist). Participants vote on this configuration, so the minimum can be updated without a code migration. The `verify_foreign_transaction` handler should reject requests whose finality/confirmation parameter falls below the configured minimum for that chain before enqueuing the yield request.

For example:
- Ethereum: reject `EvmFinality::Latest`; require at least `Safe` or `Finalized`.
- Bitcoin: reject `BlockConfirmations` below a configured floor (e.g., 6).
- Solana: reject `SolanaFinality::Processed`; require at least `Confirmed` or `Finalized`.

---

### Proof of Concept

1. Alice controls a bridge contract on NEAR that calls `verify_foreign_transaction` and releases NEAR tokens upon receiving a valid MPC signature.
2. Attacker deposits 1 ETH to the bridge's Ethereum deposit address in block N (unfinalized, `Latest` only).
3. Attacker immediately calls `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "Ethereum": {
         "tx_id": "<deposit_tx_hash>",
         "extractors": [{"BlockHash": {}}],
         "finality": "Latest"
       }
     },
     "domain_id": <foreign_tx_domain>,
     "payload_version": "V1"
   }
   ```
   with 1 yoctonear deposit. The contract accepts it — no finality floor check exists.
4. MPC nodes query the RPC, find the transaction at `Latest`, extract the block hash, and produce a threshold signature over `SHA-256(borsh(ForeignTxSignPayload { request, values }))`.
5. Attacker submits the MPC signature to Alice's bridge contract, which trusts the MPC network as the finality oracle and releases NEAR tokens.
6. Attacker (or a colluding miner) causes a reorg that drops block N, reversing the ETH deposit.
7. Attacker retains both the NEAR tokens and the ETH — double-spend complete.

The root cause is in `verify_foreign_transaction` at: [6](#0-5) 

which never validates the caller-supplied `finality` / `confirmations` field against any protocol-enforced minimum, mirroring exactly the `_minAmount = 0` pattern from the reference report.

    #[handle_result]
    #[payable]
    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
