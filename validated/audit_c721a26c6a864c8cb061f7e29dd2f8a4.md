Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Causes Permanent DoS on `initiateWithdrawal()` Until Admin Intervention - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal()` evaluates `userWithdrawalIds[msg.sender].length (0) >= maxNumberOfWithdrawalsPerUser (0)` as `true` for every caller, reverting with `WithdrawalLimitReached()`. All staked funds are frozen until an admin manually calls `setMaxNumberOfWithdrawalsPerUser()` with a non-zero value.

## Finding Description
`initialize()` sets roles and token addresses but never assigns `maxNumberOfWithdrawalsPerUser`: [1](#0-0) 

The variable therefore holds its default value of `0`. In `initiateWithdrawal()`, the guard at line 323 is: [2](#0-1) 

For any fresh caller, `userWithdrawalIds[msg.sender].length == 0`. With `maxNumberOfWithdrawalsPerUser == 0`, the condition `0 >= 0` is unconditionally `true`, causing an immediate revert. The only recovery path is an admin calling `setMaxNumberOfWithdrawalsPerUser()`: [3](#0-2) 

This function enforces `_maxNumberOfWithdrawalsPerUser != 0`, confirming that `0` is an invalid operational value, yet `initialize()` leaves it there.

## Impact Explanation
Every user who has staked via `stake()` or `stakeFor()` is unable to call `initiateWithdrawal()` from the moment of deployment. Their principal remains locked in the contract with no user-accessible exit path. This constitutes **temporary freezing of funds** (Medium impact per the allowed scope), as the freeze persists until admin intervention.

## Likelihood Explanation
The condition is triggered unconditionally on the very first call to `initiateWithdrawal()` by any user, immediately after deployment. No attacker setup, special conditions, or economic assumptions are required â€” any ordinary staker attempting to exit hits the revert. The likelihood is high.

## Recommendation
Set a safe default in `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
``` [4](#0-3) 

## Proof of Concept
1. Deploy `KernelDepositPool` behind a proxy and call `initialize(admin, kernelToken, rewardToken)`.
2. Read `maxNumberOfWithdrawalsPerUser` â€” returns `0` (never set).
3. User calls `stake(1e18)` â€” succeeds; `balanceOf[user] == 1e18`.
4. User calls `initiateWithdrawal(1e18)`.
5. Execution reaches line 323: `userWithdrawalIds[user].length (0) >= maxNumberOfWithdrawalsPerUser (0)` â†’ `true` â†’ reverts `WithdrawalLimitReached()`.
6. User's `1e18` KERNEL is locked; `claimWithdrawal()` is unreachable because no withdrawal record is ever created.
7. Funds remain frozen until admin calls `setMaxNumberOfWithdrawalsPerUser(N)` where `0 < N â‰¤ 100`.
