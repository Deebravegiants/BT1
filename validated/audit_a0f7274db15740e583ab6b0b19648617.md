### Title
VRF Key Hash Reference Count Incorrectly Zeroed on Pool Re-registration, Allowing Duplicate VRF Keys After Protocol Version 11 — (File: `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

`POOLREAP` removes "dangling" VRF key hashes from `psVRFKeyHashes` using `Map.withoutKeys` (complete deletion) instead of decrementing the reference count. When two pools share the same VRF key (permitted before protocol version 11) and one of them re-registers with a new VRF after pv11, the shared VRF key's count is wiped to zero at the epoch boundary even though the other pool still holds it. A subsequent `RegPool` certificate for that VRF key then passes the `VRFKeyHashAlreadyRegistered` guard, installing two active pools with identical VRF keys and breaking the uniqueness invariant introduced at pv11.

---

### Finding Description

`PState` maintains two related mappings:

```haskell
data PState era = PState
  { psVRFKeyHashes         :: Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
  , psStakePools           :: Map (KeyHash StakePool) StakePoolState
  , psFutureStakePoolParams :: Map (KeyHash StakePool) StakePoolParams
  , psRetiring             :: Map (KeyHash StakePool) EpochNo
  }
``` [1](#0-0) 

`psStakePools` maps each pool ID to its state (which embeds the pool's VRF key), while `psVRFKeyHashes` maps each VRF key to a reference count tracking how many active pools use it. This is the exact dual-mapping pattern from the external report: one mapping from identifier → resource, another from resource → data.

**Registration path (pv ≥ 11).** When a pool re-registers with a new VRF key and has no pending future params (`Nothing` branch), `poolTransition` inserts the new VRF key with count 1 but does **not** decrement the old VRF key's count:

```haskell
case Map.lookup sppId psFutureStakePoolParams of
  Nothing -> Map.insert sppVrf (knownNonZeroBounded @1)   -- old VRF count untouched
  Just futureStakePoolParams
    | futureStakePoolParams ^. sppVrfL /= sppVrf ->
        Map.insert sppVrf (knownNonZeroBounded @1)
          . Map.delete (futureStakePoolParams ^. sppVrfL)
    | otherwise -> id
``` [2](#0-1) 

**Epoch-boundary path (POOLREAP).** `poolReapTransition` computes `danglingVRFKeyHashes` — the set of VRF keys whose pool has a pending re-registration with a *different* VRF — and removes them from `psVRFKeyHashes` with `Map.withoutKeys`, which **deletes the key entirely regardless of its count**:

```haskell
danglingVRFKeyHashes =
  Set.fromList $ Map.elems $
    Map.merge Map.dropMissing Map.dropMissing
      ( Map.zipWithMaybeMatched $ \_ sps sppF ->
          if sps ^. spsVrfL /= sppF ^. sppVrfL then Just (sps ^. spsVrfL) else Nothing
      )
      (ps0 ^. psStakePoolsL)
      (ps0 ^. psFutureStakePoolParamsL)
``` [3](#0-2) 

```haskell
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)   -- ← complete deletion, not decrement
     )
``` [4](#0-3) 

Contrast this with the retirement path, which correctly uses `removeVRFKeyHashOccurrences` (decrement-then-delete-if-zero):

```haskell
removeVRFKeyHashOccurrence =
  Map.update (mapNonZero (\n -> n - 1))
``` [5](#0-4) 

**Concrete exploit sequence:**

1. **Before pv11** (no VRF uniqueness enforcement): pool `kh1` and pool `kh2` both register with VRF key `vrf`. This was valid.
2. **Hard fork to pv11**: `populateVRFKeyHashes` correctly sets `psVRFKeyHashes[vrf] = 2`.
3. **After pv11**: `kh1` submits a `RegPool` certificate with a fresh VRF key `vrfNew`.
   - `psFutureStakePoolParams[kh1] = {vrf: vrfNew}`
   - `psVRFKeyHashes[vrfNew] = 1`; `psVRFKeyHashes[vrf]` remains 2.
4. **Epoch boundary (POOLREAP)**:
   - `danglingVRFKeyHashes = {vrf}` (kh1's current VRF ≠ its future VRF).
   - `Map.withoutKeys {vrf}` **completely removes** `vrf` from `psVRFKeyHashes`.
   - `kh2` still holds `vrf` in `psStakePools`, but `psVRFKeyHashes[vrf]` is now absent.
5. **New registration**: pool `kh3` submits `RegPool` with `vrf`.
   - `Map.notMember vrf psVRFKeyHashes` → `True` → `VRFKeyHashAlreadyRegistered` check passes.
   - `kh3` is registered; `psVRFKeyHashes[vrf] = 1`.
6. **Result**: `kh2` and `kh3` both have `vrf` in `psStakePools`, violating the pv11 uniqueness invariant.

---

### Impact Explanation

Two active pools sharing the same VRF key produce identical VRF proofs for every slot. Both pools are therefore eligible for the same slot leadership slots. They will produce structurally different blocks (different pool IDs, different cold-key signatures) for the same slot. Honest nodes following the Praos chain-selection rule may accept different blocks for the same slot, causing a deterministic fork that cannot be resolved without a hard fork. This matches:

> **High. Deterministic disagreement between honest nodes from ledger rule evaluation.**

Additionally, the attacker who registers `kh3` can use the same VRF signing key as `kh2` to produce valid block headers indistinguishable from `kh2`'s, enabling impersonation of `kh2`'s slot leadership.

---

### Likelihood Explanation

- **Pre-condition**: Two pools sharing a VRF key before pv11. VRF uniqueness was not enforced before pv11, so such pools can exist on mainnet. The test at `eras/conway/impl/testlib/Test/Cardano/Ledger/Conway/Imp/HardForkSpec.hs:54–85` explicitly exercises this scenario.
- **Trigger**: One of those pools re-registers with a new VRF after pv11 — a routine pool-operator action.
- **Exploit**: Any unprivileged user registers a new pool with the now-untracked VRF key after the epoch boundary.
- No privileged access, governance majority, or key compromise is required. All three steps are reachable via ordinary signed transactions.

---

### Recommendation

Replace the `Map.withoutKeys danglingVRFKeyHashes` call with `removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)` so that the dangling VRF key's count is decremented (and only deleted when it reaches zero), consistent with how `retiredVRFKeyHashes` are handled:

```haskell
-- Before (incorrect):
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )

-- After (correct):
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
     )
``` [6](#0-5)

### Proof of Concept

```
State before pv11 hard fork:
  psStakePools      = { kh1 → {vrf: vrf, ...}, kh2 → {vrf: vrf, ...} }
  psVRFKeyHashes    = {}   (not yet populated)
  psFutureStakePoolParams = {}

After populateVRFKeyHashes at pv11 hard fork:
  psVRFKeyHashes    = { vrf → 2 }

Epoch N, after pv11: kh1 submits RegPool with vrfNew
  poolTransition (re-register, Nothing branch):
    psFutureStakePoolParams = { kh1 → {vrf: vrfNew, ...} }
    psVRFKeyHashes          = { vrf → 2, vrfNew → 1 }

Epoch N→N+1 boundary (POOLREAP):
  danglingVRFKeyHashes = { vrf }   (kh1: psStakePools.vrf ≠ psFutureStakePoolParams.vrf)
  Map.withoutKeys { vrf }:
    psVRFKeyHashes = { vrfNew → 1 }   ← vrf completely removed, kh2 still holds it!

Epoch N+1: kh3 submits RegPool with vrf
  Map.notMember vrf psVRFKeyHashes → True → check passes
  psVRFKeyHashes = { vrfNew → 1, vrf → 1 }
  psStakePools   = { kh1 → {vrf: vrfNew}, kh2 → {vrf: vrf}, kh3 → {vrf: vrf} }

Invariant violated: kh2 and kh3 share VRF key `vrf` under pv11.
```
