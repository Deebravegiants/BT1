### Title
Single Byzantine Participant Can Forge CKD Response for `AppPublicKey` Variant, Corrupting Key Derivation - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function performs no cryptographic verification of the response when the CKD request uses the `AppPublicKey` (privately verifiable) variant. Any single attested participant can call `respond_ckd` with arbitrary `big_y` and `big_c` values, resolving the pending yield with a forged response before honest participants can respond — directly analogous to the `CreatorStory` pattern where a role-restricted action is accessible to a broader set of callers.

---

### Finding Description

In `respond_ckd`, the contract performs a `ckd_output_check` **only** for the `AppPublicKeyPV` variant. For the `AppPublicKey` variant, the match arm is empty — no verification is performed: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

`resolve_yields_for` resolves the yield on the **first** successful call and removes the pending request from state. A single Byzantine attested participant can therefore race to call `respond_ckd` with arbitrary garbage `big_y`/`big_c` values, consuming the pending request before honest participants respond. [2](#0-1) 

Contrast this with the `respond` function for signatures, where the contract verifies the ECDSA/EdDSA signature on-chain before resolving the yield — a forged signature is rejected outright: [3](#0-2) 

For CKD with `AppPublicKey`, there is no equivalent on-chain check. The `AppPublicKey` variant is documented as "privately verifiable" (verification is performed by the app after decryption), but this design choice leaves the contract unable to reject a forged response from any single participant. [4](#0-3) 

The `respond_ckd` function is gated by `assert_caller_is_attested_participant_and_protocol_active()`, which confirms the caller is an attested participant — but does not require threshold agreement among participants before the yield is resolved. [5](#0-4) 

---

### Impact Explanation

The requester receives a garbage encrypted key. The pending CKD request is consumed and removed from state. The requester must submit a new `request_app_private_key` call, which the malicious participant can again corrupt. This breaks the CKD request-lifecycle invariant: the contract is supposed to deliver the threshold-computed derived key, but a single participant can substitute an arbitrary value. For TEE apps relying on CKD for deterministic secrets (e.g., signing keys, encryption keys), this is a persistent denial of the correct key material. If the app does not independently verify the decrypted output, the attacker can additionally supply a known value as `big_c` with `big_y = identity`, causing the app to use a key the attacker controls.

This maps to the **Medium** allowed impact: *"request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

A single Byzantine attested participant (strictly below the signing threshold) can execute this attack. The `CKDRequest` parameters (`app_id`, `domain_id`, `app_public_key`, `derivation_path`) are all visible on-chain in the original `request_app_private_key` transaction, so the attacker can trivially reconstruct the correct request object. The attacker only needs to submit `respond_ckd` before honest participants do. This is a realistic race condition for any Byzantine participant monitoring the chain. [6](#0-5) 

---

### Recommendation

1. **Require `AppPublicKeyPV` for all CKD requests** — deprecate `AppPublicKey`. The PV variant can be verified on-chain via the pairing check, closing the forgery window entirely.
2. **Add threshold-based response collection** — require multiple participants to submit the same `(big_y, big_c)` pair before resolving the yield, mirroring how `vote_pk` and `vote_reshared` require threshold agreement.
3. **Leader-only restriction** — restrict `respond_ckd` to the designated leader (as `start_keygen_instance` does), so only one trusted node submits the aggregated result. [7](#0-6) 

---

### Proof of Concept

1. User submits `request_app_private_key` with `AppPublicKey` variant; a pending `CKDRequest` entry is created in `pending_ckd_requests`.
2. Malicious attested participant observes the pending request on-chain and reconstructs the `CKDRequest` from the transaction arguments.
3. Malicious participant calls `respond_ckd(ckd_request, CKDResponse { big_y: [0u8;48], big_c: [0u8;48] })`.
4. Contract passes `assert_caller_is_attested_participant_and_protocol_active()`, enters the `AppPublicKey` match arm with no check, and calls `resolve_yields_for`.
5. The yield is resolved with the forged response; the pending request is removed from state.
6. User receives garbage key material. Honest participants' subsequent `respond_ckd` calls return an error because the request is no longer pending. [8](#0-7)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
