Deposit Refund Double-Claim Across Sub-Transactions in Dijkstra Era Batch Transactions - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`)

### Summary

In the Dijkstra era, a top-level transaction can embed multiple sub-transactions. Each sub-transaction may include an `UnRegDepositTxCert` certificate to deregister a stake credential and claim its deposit refund. No batch-level guard aggregates and bounds deposit refunds across all sub-transactions against the original deposit state, unlike the analogous `validateBatchWithdrawals` guard that exists for reward withdrawals. An attacker can therefore include the same `UnRegDepositTxCert` in two or more sub-transactions, causing the deposit pot to be debited multiple times for a single deposit, creating ADA from nothing.

---

### Finding Description

The Dijkstra era introduces nested ("sub") transactions. The top-level UTXO rule (`dijkstraUtxoTransition`) contains an explicit batch-level guard for reward withdrawals:

```haskell
-- validateBatchWithdrawals: aggregates withdrawals across top-level + all sub-txs
-- and checks the sum does not exceed the original account balance.
runTest $ validateBatchWithdrawals accounts tx
``` [1](#0-0) 

No equivalent guard exists for deposit refunds. The `dijkstraTotalRefundsTxCerts` function, which computes the refund contribution to the consumed-value balance for each sub-transaction body, reads the refund amount directly from the certificate field without consulting the live ledger state:

```haskell
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit   -- reads from cert, not from state
  UnRegDRepTxCert _ deposit -> deposit
  _ -> zero
``` [2](#0-1) 

This is the Dijkstra-era design: the certificate carries the claimed refund amount and the ledger validates it matches the stored deposit. However, there is no cross-sub-transaction guard that prevents the same credential's deposit from being claimed by more than one sub-transaction in the same batch.

The developers have explicitly acknowledged this gap with a disabled test:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  tx = ... & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [3](#0-2) 

The `xit` prefix disables the test because the predicate failure that should reject this transaction is not yet implemented. The comment `"TODO: predicate failure not yet implemented"` confirms the check is absent and the transaction currently passes validation.

The `getConsumedDijkstraValue` function aggregates consumed value (including refunds) across all sub-transactions using the original cert state:

```haskell
subTransactionsConsumedValue topTxBody =
  foldMap'
    (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
    (topTxBody ^. subTransactionsTxBodyL)
``` [4](#0-3) 

Each sub-transaction's individual value-conservation check (SUBUTXO) also uses the original cert state, so both sub-tx1 and sub-tx2 independently satisfy their own conservation checks with `keyDeposit` counted as consumed. The deposit pot is debited once per sub-transaction that processes the certificate, yielding a net debit of `n × keyDeposit` for `n` sub-transactions, while only `1 × keyDeposit` was ever deposited.

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

An attacker registers a stake credential (depositing `keyDeposit` ADA into the deposit pot), then submits a single Dijkstra batch transaction containing `n` sub-transactions each bearing `UnRegDepositTxCert stakingCred keyDeposit`. The deposit pot is debited `n × keyDeposit` while only `1 × keyDeposit` was held. The surplus `(n-1) × keyDeposit` ADA is created from nothing and flows into the attacker's sub-transaction outputs. The total ADA supply is inflated, violating the preservation-of-value invariant that is a foundational security property of the Cardano ledger.

---

### Likelihood Explanation

**High.** The Dijkstra era is the current development frontier and sub-transactions are a new, explicitly supported feature. Any user can construct a valid Dijkstra transaction with multiple sub-transactions; no special role, key, or governance threshold is required. The attack requires only: (1) registering a stake credential, (2) constructing a batch transaction with two or more sub-transactions each containing the deregistration certificate, and (3) balancing the outputs to absorb the double refund. The developers have already identified the scenario and noted the missing guard, confirming the path is reachable and the check is absent.

---

### Recommendation

Add a batch-level refund guard in `dijkstraUtxoTransition` analogous to `validateBatchWithdrawals`. Before sub-transactions are processed, aggregate all `UnRegDepositTxCert` and `UnRegDRepTxCert` certificates across the top-level tx and all sub-transactions, keyed by credential. For each credential, verify that at most one sub-transaction claims its refund and that the claimed amount matches the stored deposit. This mirrors the existing pattern: [5](#0-4) 

A new `validateBatchRefunds` function should:
1. Collect all deregistration certificates from the top-level tx body and every sub-transaction body.
2. Detect any credential that appears in more than one certificate.
3. Fail with a new `DijkstraUtxoPredFailure` constructor (e.g., `DuplicateRefundsInBatch`) if duplicates are found.

---

### Proof of Concept

1. Register stake credential `C` with deposit `D` (e.g., 2 ADA).
2. Construct sub-tx1 with `UnRegDepositTxCert C D` and an output absorbing `D`.
3. Construct sub-tx2 with `UnRegDepositTxCert C D` and an output absorbing `D`.
4. Construct top-level tx with `subTransactions = [subTx1, subTx2]`.
5. Submit the transaction.
6. Observe: credential `C` is deregistered, but `2D` ADA has been extracted from the deposit pot while only `D` was deposited. The attacker nets `D` ADA of newly created value.

The developers' own disabled test at [3](#0-2) 

encodes exactly this scenario and explicitly marks it as currently unguarded (`"TODO: predicate failure not yet implemented"`).

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL
```
