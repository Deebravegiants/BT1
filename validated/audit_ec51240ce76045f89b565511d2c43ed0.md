### Title
Missing `state_diff_commitment` Verification in P2P State Diff Sync Allows Malicious Peer to Corrupt Contract Nonces — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`parse_data_for_block` in the p2p sync client validates only the **count** of state diff entries against `state_diff_length` from the stored block header. It never computes `calculate_state_diff_hash` on the assembled `ThinStateDiff` and never compares the result against the `state_diff_commitment` that is also present in the stored header. A malicious p2p peer can therefore send a `ContractDiff` chunk with `contract_address=A` and `nonce=Some(WRONG_NONCE)` whose total chunk length equals `target_state_diff_len`, pass every guard in `parse_data_for_block`, and cause `write_to_storage` → `append_state_diff` → `write_nonces` to persist `WRONG_NONCE` into the `nonces_table` for address `A`.

---

### Finding Description

**Entry point:** An unauthenticated p2p peer responding to a `StateDiffsRequest`.

**Step 1 — length-only guard in `parse_data_for_block`:**

`parse_data_for_block` reads `target_state_diff_len` from the stored block header's `state_diff_length` field and loops until `current_state_diff_len == target_state_diff_len`. The only structural checks are:
- chunk is non-empty
- no duplicate address in the same field (nonce, class_hash, storage key)
- total count equals `target_state_diff_len`

There is no call to `calculate_state_diff_hash` and no comparison against `state_diff_commitment`. [1](#0-0) 

**Step 2 — nonce inserted verbatim in `unite_state_diffs`:**

```rust
if let Some(nonce) = contract_diff.nonce {
    if state_diff.nonces.insert(contract_diff.contract_address, nonce).is_some() {
        return Err(BadPeerError::ConflictingStateDiffParts);
    }
}
```

The only rejection is a duplicate address. Any `Nonce` value is accepted. [2](#0-1) 

**Step 3 — unconditional write to storage:**

`write_to_storage` calls `append_state_diff` with no hash verification step. [3](#0-2) 

`append_state_diff` then calls `write_nonces`, which writes the attacker-supplied nonce directly into `nonces_table`. [4](#0-3) 

**The missing check — `state_diff_commitment` is stored but never used here:**

`calculate_state_diff_hash` chains nonces into a Poseidon hash and produces a `StateDiffCommitment`. The block header already stores this commitment. The `apollo_committer` has a `verify_state_diff_hash` config flag that performs exactly this check, but the p2p sync client path never invokes it. [5](#0-4) [6](#0-5) 

A grep of `crates/apollo_p2p_sync/**/*.rs` for `state_diff_commitment` returns only one hit — in `header.rs` where the commitment is *stored* — and zero hits in `state_diff.rs` where it should be *verified*. [7](#0-6) 

---

### Impact Explanation

A node syncing via p2p will store an attacker-chosen nonce for any contract address `A`. Every subsequent call to `starknet_getNonce` for `A` on that node returns `WRONG_NONCE`. If the node also serves as a gateway or mempool, transaction admission will use the corrupted nonce to validate incoming transactions: a nonce set too high causes all valid transactions from `A` to be rejected; a nonce set too low causes replay-window confusion. This is a **High** impact: RPC returns an authoritative-looking wrong value, and mempool/gateway admission rejects valid transactions.

---

### Likelihood Explanation

The p2p network is open. Any node can connect as a peer and respond to `StateDiffsRequest` messages. The attacker only needs to know `state_diff_length` for the target block (available from the public block header) and craft a `ContractDiff` whose chunk length equals that value. No cryptographic material, no privileged access, and no special timing is required.

---

### Recommendation

After assembling the final `ThinStateDiff` in `parse_data_for_block` (after the length check at line 99), compute `calculate_state_diff_hash(&result)` and compare it against `block_header.state_diff_commitment`. Reject the peer with `BadPeerError` if the hashes do not match. This mirrors the existing `verify_state_diff_hash` logic already present in `apollo_committer`.

```rust
// After line 106 (validate_deprecated_declared_classes_non_conflicting)
let expected_commitment = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .state_diff_commitment
    .ok_or(/* OldHeaderInStorage */)?;
let actual_commitment = calculate_state_diff_hash(&result);
if actual_commitment != expected_commitment {
    return Err(ParseDataError::BadPeer(BadPeerError::WrongStateDiffHash { ... }));
}
```
