### Title
Excess Deposit Permanently Locked in Contract on Existing-Participant Attestation Re-submission — (File: `crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is `#[payable]` but only executes its deposit-refund logic when `attestation_storage_must_be_paid_by_caller` is `true`. When an existing participant re-submits their attestation (the normal periodic re-attestation path), this flag is `false` and the entire deposit-handling block is skipped. Any NEAR tokens attached to that call are permanently locked in the contract with no refund path.

---

### Finding Description

`submit_participant_info` computes a boolean gate at lines 823–824:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;
```

`is_new_attestation` is `true` only when `add_participant` returns `ParticipantInsertion::NewlyInsertedParticipant`; `caller_is_not_participant` is `true` only when `self.voter_account().is_err()`. When an existing participant re-submits (the routine hourly re-attestation cycle), `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` and the caller is already a voter, so both sub-conditions are `false`.

The deposit-handling block at lines 826–849 is therefore entirely skipped:

```rust
if attestation_storage_must_be_paid_by_caller {
    // ...
    // Refund the difference if the proposer attached more than required
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
```

Because the function is `#[payable]`, the NEAR runtime accepts any attached deposit. When the `if` block is skipped, `env::attached_deposit()` is never read, never checked, and never refunded. The tokens are credited to the contract account and have no recovery path — there is no skim/sweep function in the contract.

This is structurally identical to the WETHGateway analog: a payable entry-point that only conditionally refunds excess, leaving funds stranded in the contract on the common re-submission path.

---

### Impact Explanation

Any NEAR tokens attached to a re-submission call by an existing participant are permanently locked in the `mpc-contract` account. The contract has no skim or recovery function. The accounting invariant — "deposits are either consumed for storage or refunded" — is broken on every `UpdatedExistingParticipant` call that carries a non-zero deposit. This maps to the **Medium** allowed impact: *balance/accounting invariant break that corrupts contract execution-flow accounting without requiring DoS or operator misconfiguration*.

---

### Likelihood Explanation

The automated `mpc-node` path (`periodic_attestation_submission`) attaches `0` deposit for re-submissions, so the bug is dormant under fully automated operation. However:

1. The operator guide explicitly instructs first-time joiners to call `submit_participant_info` manually with `--deposit` (documented in `docs/running-an-mpc-node-in-tdx-external-guide.md`). A participant who has already registered once and re-runs that manual command will hit the `UpdatedExistingParticipant` branch with a non-zero deposit — losing those funds silently.
2. Any integrator or script that defensively attaches a deposit "to be safe" on every call will permanently lose the excess on every re-submission.
3. The function is public and `#[payable]` with no documentation warning that deposits are silently consumed on re-submissions.

---

### Recommendation

Unconditionally refund any attached deposit that exceeds the actual storage cost (zero on re-submissions). The simplest fix mirrors the existing pattern used in `require_deposit` and `propose_update`:

```rust
// After the if block, always refund any unspent deposit.
let attached = env::attached_deposit();
if !attestation_storage_must_be_paid_by_caller && attached > NearToken::from_yoctonear(0) {
    Promise::new(account_id).transfer(attached).detach();
}
```

Alternatively, restructure the deposit logic to always compute `cost` (zero when storage delta is zero) and always refund `attached - cost`, eliminating the conditional entirely. Additionally, consider adding a skim/recovery function callable by the contract owner to recover any tokens already locked.

---

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` for the first time with `1 NEAR` attached. `is_new_attestation = true`, so the deposit block runs, storage cost is charged, and the excess is refunded. Alice's attestation is stored.
2. Alice's node re-submits (routine re-attestation) and a human operator manually runs the same CLI command with `--deposit 1 NEAR` again.
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant`. `is_new_attestation = false`. Alice is a voter, so `caller_is_not_participant = false`. `attestation_storage_must_be_paid_by_caller = false`.
4. The `if` block at line 826 is skipped. `env::attached_deposit()` (1 NEAR) is never read or refunded.
