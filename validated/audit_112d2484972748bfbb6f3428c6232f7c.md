### Title
Deposits Paid by Non-Winning `propose_update` Proposers Are Permanently Lost When Any Update Is Executed - (`crates/contract/src/update.rs`)

### Summary

When `do_update` executes a winning contract-upgrade or config-update proposal, it bulk-clears all other pending proposals from storage without refunding the storage-staking deposits those proposers paid. Because `UpdateEntry` does not record the proposer's account ID or deposit amount, the funds are irrecoverable.

### Finding Description

`propose_update` requires a deposit proportional to the size of the proposed contract binary or config blob: [1](#0-0) 

The required deposit is computed as `env::storage_byte_cost() × bytes_used`, which for a full contract WASM binary (up to ~1.5 MB) can reach tens of NEAR tokens. Only the *excess* above the required amount is refunded at proposal time: [2](#0-1) 

The required portion is retained by the contract. The `UpdateEntry` struct that is stored records only the update payload and its byte count — **not** the proposer's account ID or the deposit amount: [3](#0-2) 

When threshold votes are reached and `do_update` fires, it removes the winning entry and then bulk-clears every other pending entry and all votes: [4](#0-3) 

No refund is issued to the proposers of the cleared entries. Because the proposer identity and deposit amount were never stored in `UpdateEntry`, there is no information available to perform a refund even if the code attempted one.

The `propose` helper that inserts entries confirms no proposer metadata is saved: [5](#0-4) 

### Impact Explanation

Every participant whose proposal is cleared by a competing `do_update` permanently loses their storage-staking deposit. For a contract binary proposal the deposit is `storage_byte_cost × (sizeof(UpdateEntry) + 128 × sizeof(AccountId) + code.len())`. At NEAR's current storage cost (~10 yoctoNEAR/byte) a 1 MB WASM binary costs roughly 10 NEAR. Multiple participants can each propose different binaries simultaneously; when one wins, all others forfeit their deposits with no recovery path. This breaks the balance/accounting invariant: funds paid into the contract for a specific purpose (storage staking for a proposal) are not returned when that purpose is cancelled by an external event (a competing proposal winning).

### Likelihood Explanation

This is triggered by the normal governance flow. Any time a contract upgrade is successfully executed — a routine operation — every other pending proposal is silently cleared. Participants who proposed competing upgrades (a realistic scenario during contested governance) lose their deposits automatically. No adversarial action is required beyond submitting a competing proposal and having it win.

### Recommendation

Store the proposer's `AccountId` and the exact deposit amount inside `UpdateEntry`. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the pattern already used for excess-deposit refunds in `propose_update`.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR deposit. Entry `id=0` is stored.
2. Participant B calls `propose_update` with a different binary, attaching ~10 NEAR deposit. Entry `id=1` is stored.
3. Threshold participants vote for `id=1`. `vote_update` calls `do_update(&id=1, gas)`.
4. `do_update` removes entry `id=1`, then calls `self.entries.clear()` — entry `id=0` is deleted with no refund.
5. Participant A's ~10 NEAR deposit remains locked in the contract permanently. There is no function to recover it. [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
