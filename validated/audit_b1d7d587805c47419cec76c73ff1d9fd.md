### Title
Unrefunded Attached Deposit Permanently Stuck in Contract on Existing-Participant Re-submission Path — (`crates/contract/src/lib.rs`)

### Summary

`submit_participant_info` is marked `#[payable]` and therefore accepts an attached NEAR deposit on every call. However, when an existing participant re-submits their attestation (the `attestation_storage_must_be_paid_by_caller == false` branch), the entire deposit-handling block is skipped: no deposit is required, but no deposit is refunded either. Any NEAR attached to such a call is permanently locked in the contract with no recovery path.

### Finding Description

`submit_participant_info` computes a boolean flag to decide whether the caller must cover storage costs:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check deposit sufficiency, then refund excess
}
// ← no else branch: if flag is false, attached deposit is silently consumed
``` [1](#0-0) 

When the flag is `false` — i.e., the caller is an existing protocol participant **and** the insertion is an update (not a new entry) — the `if` block is never entered. The function returns `Ok(())` without inspecting `env::attached_deposit()` at all. Because the function is `#[payable]`, the NEAR runtime accepts any deposit the caller attaches; that deposit is credited to the contract account and there is no subsequent transfer back to the caller. [2](#0-1) 

By contrast, every other payable entry point in the contract either calls `require_deposit` (which always refunds the excess) or explicitly computes and transfers the difference:

- `sign` / `request_app_private_key` / `verify_foreign_transaction` → `require_deposit` → unconditional refund of excess. [3](#0-2) 
- `propose_update` → explicit `checked_sub` + `Promise::transfer`. [4](#0-3) 

`submit_participant_info` is the only `#[payable]` function that has a success path where the deposit is neither consumed for a defined purpose nor returned.

### Impact Explanation

Any NEAR attached to a re-submission call by an existing participant is permanently locked in the contract. NEAR has no built-in sweep or rescue mechanism for funds credited to a contract account; the contract itself contains no admin withdrawal function. The accounting invariant — that deposits are either consumed for storage or refunded — is violated, and the stuck balance cannot be recovered without a contract upgrade.

This matches the **Medium** allowed impact: *"Balance … manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

### Likelihood Explanation

The re-submission path is the normal TEE-upgrade path: existing participants periodically re-submit attestations when their TEE measurement changes. The call is made by automated node software. A misconfigured script, a copy-paste from a new-participant setup (which does require a deposit), or a wrapper contract that always attaches a small deposit would silently lose NEAR on every periodic re-attestation. The function being `#[payable]` gives no indication to callers that attaching NEAR on the re-submission path is harmful.

### Recommendation

Add an unconditional refund of any excess deposit at the end of `submit_participant_info`, mirroring the pattern already used in `require_deposit`:

```rust
// After the conditional storage-cost block:
let attached = env::attached_deposit();
if attached > NearToken::from_yoctonear(0) {
    Promise::new(account_id).transfer(attached).detach();
}
```

Alternatively, restructure the function so that `require_deposit(NearToken::from_yoctonear(0), &account_id)` is called unconditionally before the storage-cost block, which will refund any non-zero deposit on the free re-submission path.

### Proof of Concept

1. Alice is an existing protocol participant (`voter_account()` succeeds).
2. Alice's TEE measurement changes; she calls `submit_participant_info` with a new attestation, attaching `1 NEAR` (e.g., copied from a new-participant setup script).
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` → `is_new_attestation = false`.
4. `caller_is_not_participant = false` (Alice is a voter).
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The `if` block is skipped entirely; `Ok(())` is returned.
7. Alice's `1 NEAR` is now credited to the contract account with no refund promise scheduled.
8. `env::account_balance()` of the contract has increased by `1 NEAR`; Alice's balance has decreased by `1 NEAR` plus gas. There is no contract method to recover these funds. [5](#0-4)

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {

                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
