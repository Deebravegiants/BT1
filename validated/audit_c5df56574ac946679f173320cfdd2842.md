### Title
Fee Applied at Claim-Time Allows Mid-Vesting Fee Increase to Reduce User's Net Received Tokens — (`contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

---

### Summary

`setFeeInBPS` takes immediate global effect with no per-user snapshot or timelock. Because `KernelTop100MerkleDistributor` is a **time-based vesting contract** (30-day linear schedule) that expects users to make multiple partial claims, a fee change between claims silently reduces the net tokens a user receives relative to what was calculable at the time of their first claim.

---

### Finding Description

`setFeeInBPS` writes directly to the global `feeInBPS` storage variable with no guard beyond the 10% cap: [1](#0-0) 

Every call to `claim` reads `feeInBPS` at execution time: [2](#0-1) 

The vesting model is explicitly multi-claim: `_getUnclaimedVestedAmount` computes only the slice vested since the last claim timestamp, so users are structurally expected to call `claim` multiple times over the 30-day `VESTING_DURATION`: [3](#0-2) 

The merkle leaf commits only to `(user, amount)` — the gross allocation — not to any fee: [4](#0-3) 

There is no per-user fee snapshot, no fee-lock-on-first-claim, and no timelock on `setFeeInBPS`.

---

### Impact Explanation

Concrete scenario with `totalAmount = 1000 KERNEL`:

| Event | feeInBPS | Vested slice | Fee taken | User receives |
|---|---|---|---|---|
| t=15d: `claim()` | 0 | 500 | 0 | 500 |
| Owner: `setFeeInBPS(1000)` | — | — | — | — |
| t=30d: `claim()` | 1000 | 500 | 50 | 450 |
| **Total** | | 1000 | 50 | **950** |

The user expected 1000 (fee=0 at first claim) but received 950. The 50 tokens are not lost from the system — they go to `protocolTreasury` — so this matches the Low scope: **contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

The path is fully reachable without any key compromise. The owner calls `setFeeInBPS` — a legitimate, unguarded admin function — between two user claims. No front-running, no brute force, no external dependency. The 30-day vesting window makes multi-claim patterns the norm, not the exception.

---

### Recommendation

Snapshot the effective fee per user on their **first claim** and apply that snapshot for all subsequent claims:

```solidity
struct UserClaim {
    uint256 lastClaimTimestamp;
    uint256 amountClaimed;
    uint256 feeInBPSAtFirstClaim;   // add this
    bool    hasClaimedOnce;          // add this
}
```

In `claim` / `claimAndStake`, before computing the fee:

```solidity
if (!userClaims[user].hasClaimedOnce) {
    userClaims[user].feeInBPSAtFirstClaim = feeInBPS;
    userClaims[user].hasClaimedOnce = true;
}
uint256 effectiveFee = userClaims[user].feeInBPSAtFirstClaim;
uint256 fee = (claimableAmount * effectiveFee) / FEE_DENOMINATOR;
```

Alternatively, add a timelock to `setFeeInBPS` so users have advance notice before any fee change takes effect.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Pseudocode — run on a local fork or Foundry test

function testFeeChangeBetweenClaims() public {
    // Setup: user has merkle allocation of 1000e18 KERNEL, feeInBPS = 0
    // vestingStartTimestamp = block.timestamp (already started)

    // t = 15 days: user claims first half
    vm.warp(vestingStart + 15 days);
    uint256 balBefore1 = kernel.balanceOf(user);
    vm.prank(user);
    distributor.claim(1000e18, proof);
    uint256 received1 = kernel.balanceOf(user) - balBefore1;
    assertEq(received1, 500e18); // no fee

    // Owner raises fee to 10%
    vm.prank(owner);
    distributor.setFeeInBPS(1000);

    // t = 30 days: user claims second half
    vm.warp(vestingStart + 30 days);
    uint256 balBefore2 = kernel.balanceOf(user);
    vm.prank(user);
    distributor.claim(1000e18, proof);
    uint256 received2 = kernel.balanceOf(user) - balBefore2;
    assertEq(received2, 450e18); // 10% fee applied

    uint256 totalReceived = received1 + received2;
    assertEq(totalReceived, 950e18); // user expected 1000e18
    // Invariant broken: total < merkle-committed amount * (1 - fee_at_first_claim)
}
```
