### Title
In-Flight `ParameterChange` Proposals Bypass Newly Enacted Guardrails Script ŌĆö (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`, `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

The Conway ledger enforces the guardrails script constraint on `ParameterChange` and `TreasuryWithdrawals` proposals only at **proposal-submission time** (the `GOV` rule). It is never re-checked at **ratification time** (the `RATIFY` rule) or **enactment time** (the `ENACT` rule). Because governance proposals can remain in-flight for multiple epochs, a `ParameterChange` submitted when no guardrails script existed (or with an old guardrails script hash) can be ratified and enacted after a `NewConstitution` action has established a new, stricter guardrails script ŌĆö completely bypassing the guardrails constraint. This is the direct analog of the external report's pattern: the "enabling" of a constraint and the "constrained action" are set in separate governance actions, creating a window where the constraint is active but in-flight proposals are immune to it.

---

### Finding Description

**Root cause ŌĆö guardrails check is submission-only:**

In `conwayGovTransition` (the `GOV` rule), when a proposal is submitted, the guardrails script hash is checked: [1](#0-0) 

```haskell
-- Guardrails script hash check
runTest $ checkGuardrailsScriptHash @era constitutionPolicy proposalPolicy
...
ParameterChange _ _ proposalPolicy ->
    runTest $ checkGuardrailsScriptHash @era constitutionPolicy proposalPolicy
```

`checkGuardrailsScriptHash` simply compares the proposal's embedded policy hash against the current constitution's guardrails script hash: [2](#0-1) 

**Root cause ŌĆö no re-check at ratification:**

In `ratifyTransition` (the `RATIFY` rule), the five conditions checked before enacting a proposal are: [3](#0-2) 

```haskell
if prevActionAsExpected gas ensPrevGovActionIds
    && validCommitteeTerm govAction ensCurPParams reCurrentEpoch
    && not rsDelayed
    && withdrawalCanWithdraw govAction ensTreasury
    && acceptedByEveryone env st gas
```

There is no guardrails script hash check here. A `ParameterChange` proposal that was submitted with `SNothing` (or an old `SJust oldHash`) policy is ratified and enacted regardless of what the current constitution's guardrails script hash is.

**The two related state changes in separate governance actions:**

- **State change 1 (enabling the constraint):** A `NewConstitution` governance action is ratified and enacted, setting `constitutionGuardrailsScriptHash` to `SJust newScriptHash`. This is a delaying action. [4](#0-3) 

- **State change 2 (the constrained action):** A `ParameterChange` governance action was submitted in a prior epoch with `SNothing` policy (when no guardrails existed), and it remains in-flight. [5](#0-4) 

**Concrete attack scenario:**

1. **Epoch N:** No guardrails script exists (`constitutionGuardrailsScriptHash = SNothing`). A `ParameterChange` proposal is submitted with `SNothing` policy ŌĆö it passes the `checkGuardrailsScriptHash` check trivially.
2. **Epoch N+1:** A `NewConstitution` action is ratified and enacted, setting `constitutionGuardrailsScriptHash = SJust restrictiveScript`. The restrictive script would reject the in-flight `ParameterChange` (e.g., it enforces bounds on `ppMinFeeA`, `ppKeyDeposit`, etc.).
3. **Epoch N+2:** The in-flight `ParameterChange` (with `SNothing` policy) is ratified and enacted. The `RATIFY` rule does not re-check the guardrails. The restrictive Plutus script is never executed. The parameter change is enacted outside the bounds the guardrails were designed to enforce.

The same scenario applies in reverse: a `ParameterChange` submitted with `SJust oldHash` can be enacted after a `NewConstitution` replaces the guardrails with `SJust newHash`.

---

### Impact Explanation

**Medium.** An attacker-controlled `ParameterChange` or `TreasuryWithdrawals` proposal, submitted before guardrails were established (or before they were updated), can be ratified and enacted after the guardrails are in place, modifying protocol parameters (fees, deposits, execution unit prices, etc.) or treasury withdrawals outside the bounds the guardrails script was designed to enforce. This matches the allowed impact: *"Attacker-controlled transactions... modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters."*

---

### Likelihood Explanation

Realistic. Governance proposals have a configurable lifetime (`ppGovActionLifetimeL`, default 6 epochs on mainnet). A `ParameterChange` submitted in epoch N remains valid until epoch N+6. A `NewConstitution` establishing guardrails can be enacted at any point during that window. The ordering of these two governance actions across epochs is a normal operational scenario, not a contrived edge case. Any DRep or SPO coalition that can ratify a `ParameterChange` can exploit this window.

---

### Recommendation

Re-check the guardrails script hash (and execute the guardrails Plutus script if present) at **ratification time** in `ratifyTransition`, using the `ensCurPParams`-derived constitution policy that is already available in the `EnactState`. Specifically, add a check analogous to `checkGuardrailsScriptHash` inside `ratifyTransition` for `ParameterChange` and `TreasuryWithdrawals` actions, comparing the proposal's embedded policy hash against the current `constitutionGuardrailsScriptHash` at the moment of ratification. This mirrors the existing submission-time check and closes the window.
