### Title
Undetected `ft_transfer` Failure in `fin_transfer_send_tokens_callback` Permanently Locks User Funds — (`near/omni-bridge/src/lib.rs`)

### Summary

When finalizing a cross-chain transfer to a NEAR recipient using a non-deployed (locked) token with no message, the bridge calls `ft_transfer` and then `fin_transfer_send_tokens_callback`. The callback's `is_refund_required` helper unconditionally returns `false` for plain `ft_transfer` paths, so a failed `ft_transfer` promise is silently ignored. The transfer ID is already recorded in `finalised_transfers` before the token send, so the transfer can never be replayed. The tokens remain in the bridge contract with no on-chain recovery path.

### Finding Description

`process_fin_transfer_to_near` calls `add_fin_transfer` to mark the transfer as finalized, then calls `send_tokens`, then chains `fin_transfer_send_tokens_callback`. [1](#0-0) [2](#0-1) 

Inside `send_tokens`, for a non-deployed token with an empty `msg`, the code issues a plain `ft_transfer`: [3](#0-2) 

The callback receives `is_ft_transfer_call = false` (because `msg.is_empty()` is `true`). `is_refund_required` then unconditionally returns `false` for this branch: [4](#0-3) 

Because `is_refund_required` returns `false`, the callback skips the refund branch entirely and proceeds to emit `FinTransferEvent` and pay the fee recipient — regardless of whether the `ft_transfer` promise actually succeeded or failed: [5](#0-4) 

The `remove_fin_transfer` call that would un-finalize the transfer and allow a retry is never reached. The transfer ID stays in `finalised_transfers` permanently. [6](#0-5) 

### Impact Explanation

Tokens belonging to the user are held by the bridge contract. After a failed `ft_transfer`, they remain there indefinitely. Because the transfer ID is already in `finalised_transfers`, no relayer can replay `fin_transfer` with the same proof. There is no privileged recovery function that removes a finalized transfer and re-sends tokens. The result is an **irreversible fund lock** matching the Critical impact category.

### Likelihood Explanation

Any condition that causes `ft_transfer` to revert after the storage deposit step succeeds triggers this path. Realistic triggers include:

- A token contract with a pause or blacklist feature (common for regulated or upgradeable tokens bridged through Omni Bridge).
- A token contract that is upgraded or self-destructs between the time the user locks funds on the source chain and when `fin_transfer` is called on NEAR.
- Insufficient bridge-side token balance due to an accounting edge case (e.g., decimal normalization rounding).

None of these require the user or relayer to be malicious; they are operational conditions that can arise in production.

### Recommendation

The callback must inspect the `ft_transfer` promise result and treat a failure as a refund condition, mirroring the existing `ft_transfer_call` handling. Concretely:

1. Change `is_refund_required` to also check `env::promise_result_checked(0, ...)` when `is_ft_transfer_call` is `false`, treating a `Failed` promise result as requiring a refund.
2. Alternatively, restructure `send_tokens` so that all paths (plain `ft_transfer`, `ft_transfer_call`, `mint`) go through a unified callback that always checks the promise outcome before finalizing state.

The pull-pattern fix referenced in the external report (removing inline transfers in favor of claimable balances) would also eliminate this class of bug entirely.

### Proof of Concept

1. User initiates a transfer of token `T` (non-deployed, locked) from Ethereum to a NEAR account `alice.near`.
2. Token `T` has a pause mechanism; its admin pauses it between the Ethereum lock event and the NEAR `fin_transfer` call.
3. Relayer calls `fin_transfer` with a valid proof and a `StorageDepositAction` for `alice.near`.
4. `fin_transfer_callback` → `process_fin_transfer_to_near`:
   - `add_fin_transfer` inserts the transfer ID into `finalised_transfers`. [1](#0-0) 
   - `send_tokens` dispatches `ft_transfer(alice.near, amount)` to the paused token contract. [3](#0-2) 
5. Token contract panics (paused). Promise result = `Failed`.
6. `fin_transfer_send_tokens_callback` is called with `is_ft_transfer_call = false`.
7. `is_refund_required(false)` returns `false`. [4](#0-3) 
8. Callback emits `FinTransferEvent` and pays the fee recipient. Tokens remain in the bridge.
9. Any subsequent `fin_transfer` call with the same proof is rejected: `NonceAlreadyUsed` / transfer already in `finalised_transfers`.
10. `alice.near` never receives tokens. No recovery path exists on-chain.
