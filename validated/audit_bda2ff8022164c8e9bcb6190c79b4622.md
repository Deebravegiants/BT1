### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Single-Participant Forged Foreign-Chain Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash`. It never checks that `payload_hash` was actually derived from the `request` argument. A single malicious TEE-attested participant who possesses any legitimately-produced root-key signature can replay it against an unrelated pending `verify_foreign_transaction` request, resolving that request with a forged payload hash.

---

### Finding Description

In `respond_verify_foreign_tx` the contract extracts the payload hash exclusively from the response object and verifies the ECDSA signature against it:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract never reconstructs or bounds-checks what `payload_hash` should be for the given `request`. Compare this with the regular `respond` path, where the payload hash is taken directly from the request, not from the response:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [2](#0-1)
