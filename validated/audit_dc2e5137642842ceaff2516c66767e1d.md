### Title
`pricePercentageLimit` Cap Not Enforced for Price Increases Below All-Time High (`highestRsethPrice`) - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle` defines `pricePercentageLimit` as a cap on per-update price changes. The cap is enforced for price decreases unconditionally (relative to `highestRsethPrice`), but the corresponding upside check is gated behind `if (newRsETHPrice > highestRsethPrice)`. When the rsETH price is recovering from a prior dip and is still below `highestRsethPrice`, any magnitude of price increase passes through `_updateRsETHPrice()` without triggering `PriceAboveDailyThreshold`, defeating the stated safety invariant.

### Finding Description
`LRTOracle._updateRsETHPrice()` contains two asymmetric branches:

**Downside branch** (lines 270–291): always evaluated when `newRsETHPrice < highestRsethPrice`; pauses the protocol if the drop exceeds `pricePercentageLimit`.

**Upside branch** (lines 252–267): only evaluated when `newRsETHPrice > highestRsethPrice`. If the new price is anywhere in the range `(previousPrice, highestRsethPrice]`, the entire upside check is skipped. [1](#0-0) 

The `pricePercentageLimit` setter places no lower bound on the value and documents it as a per-update percentage guard: [2](#0-1) 

The downside branch correctly measures the gap from `highestRsethPrice`: [3](#0-2) 

Because the upside branch is absent for the sub-peak recovery range, a price jump of, say, 15% from `previousPrice` to a value just below `highestRsethPrice` is accepted silently, while a 1.1% jump that crosses `highestRsethPrice` would be caught. The cap is defined but not enforced in the recovery path.

### Impact Explanation
**Low — Contract fails to deliver promised returns.**

The `pricePercentageLimit` mechanism is the protocol's primary on-chain guard against anomalous per-update price spikes (e.g., from a misbehaving asset price oracle or a sudden large deposit). When the price is below its all-time high, this guard is entirely absent. A large upward price move in a single `updateRSETHPrice()` call will be committed to `rsETHPrice` and immediately used by `LRTDepositPool.getRsETHAmountToMint()` and the L2 pool `getRate()` path, causing depositors in that block to receive fewer rsETH tokens than the protocol's safety model intends to guarantee. [4](#0-3) 

### Likelihood Explanation
**Medium.** The scenario — price drops below `highestRsethPrice` then recovers — is a normal operating condition (e.g., after a slashing event is resolved, or after a temporary oracle dip). Any caller of the permissionless `updateRSETHPrice()` can trigger the update at any time. [5](#0-4) 

### Recommendation
Remove the `if (newRsETHPrice > highestRsethPrice)` gate on the upside check and instead measure the per-update price change relative to `previousPrice` (or at minimum apply the cap symmetrically for any upward move, not only moves that set a new all-time high). Alternatively, if the intent is to cap only new all-time-high excursions, document this explicitly and add a separate per-update increase guard relative to `previousPrice`.

### Proof of Concept
Assume:
- `highestRsethPrice = 1.10e18`
- `rsETHPrice (previousPrice) = 1.00e18`
- `pricePercentageLimit = 1e16` (1%)

A single `updateRSETHPrice()` call that computes `newRsETHPrice = 1.09e18` (a 9% jump from `previousPrice`):

1. `newRsETHPrice (1.09e18) > highestRsethPrice (1.10e18)` → **false** → upside branch skipped entirely.
2. `newRsETHPrice (1.09e18) < highestRsethPrice (1.10e18)` → downside branch entered; `diff = 0.01e18`; `pricePercentageLimit.mulWad(highestRsethPrice) = 1e16 * 1.10e18 / 1e18 = 1.10e16`; `0.01e18 > 1.10e16` → **true** → protocol pauses.

Wait — in this specific example the downside branch fires and pauses. Adjust: `newRsETHPrice = 1.099e18` (9.9% jump, still below `highestRsethPrice`):

1. Upside branch: `1.099e18 > 1.10e18` → false → skipped.
2. Downside branch: `1.099e18 < 1.10e18` → `diff = 0.001e18`; threshold = `1.10e16`; `0.001e18 (1e15) > 1.10e16` → **false** → no pause.
3. `rsETHPrice` is updated to `1.099e18` — a 9.9% single-block increase — with no revert and no pause, bypassing the 1% `pricePercentageLimit` cap entirely. [1](#0-0) [3](#0-2) [6](#0-5)
