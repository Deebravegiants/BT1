Audit Report

## Title
Stale Cross-Chain Rate Enables Over-Minting of wrsETH, Diluting Existing Holders' Yield — (`contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/pools/RSETHPoolV3.sol`)

## Summary

`CrossChainRateReceiver` stores `lastUpdated` on every `lzReceive` call but never enforces a staleness threshold in `getRate()`. `RSETHPoolV3.viewSwapRsETHAmountAndFee()` consumes this unchecked rate to compute wrsETH minting amounts. During the inherent LayerZero delivery window after an L1 rsETH price increase, any depositor receives more wrsETH than their ETH can back at the true rate, permanently diluting existing holders' accrued yield.

## Finding Description

`CrossChainRateReceiver` records `lastUpdated` at line 97 on every `lzReceive` call but `getRate()` at lines 103–105 returns `rate` unconditionally with no comparison against `lastUpdated`:

```solidity
// CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;  // lastUpdated is stored but never checked here
}
```

`RSETHPoolV3.getRate()` (lines 235–237) delegates directly to this oracle via `IOracle(rsETHOracle).getRate()`. `viewSwapRsETHAmountAndFee()` (lines 299–308) then uses the potentially stale rate:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The rate update path is: L1 `RSETHRateProvider.getLatestRate()` reads `ILRTOracle.rsETHPrice()` → `CrossChainRateProvider.updateRate()` (permissionless, caller pays LZ fees) → LayerZero message → L2 `CrossChainRateReceiver.lzReceive()`. There is always a non-zero delivery window (minutes to hours). During this window the L2 receiver holds the old (lower) rate while the L1 oracle has already moved higher. A depositor calling `deposit()` during this window receives `amountAfterFee * 1e18 / staleRate` wrsETH instead of the correct `amountAfterFee * 1e18 / trueRate`, minting excess tokens. The ETH is later bridged to L1 and buys only `ETH / trueRate` rsETH, leaving the pool with more wrsETH outstanding than rsETH backing — a permanent shortfall borne by all existing holders. No existing check prevents this: `whenNotPaused` and `limitDailyMint` do not validate oracle freshness.

## Impact Explanation

**High. Theft of unclaimed yield.**

Concrete arithmetic: true L1 rate = `1.1e18`, stale L2 rate = `1.0e18`, deposit = 10 ETH (fee = 0). wrsETH minted = `10e18 * 1e18 / 1.0e18 = 10e18`. Correct mint = `10e18 * 1e18 / 1.1e18 ≈ 9.09e18`. Over-mint ≈ `0.91e18` (~10%). The 10 ETH bridged to L1 buys only ≈ 9.09 rsETH. Pool now has 10 wrsETH backed by 9.09 rsETH — a permanent shortfall. Existing holders' per-token redemption value is permanently reduced; the yield they accrued from the rate increase (1.0 → 1.1) is redistributed to the new depositor. The over-minted tokens remain in circulation and cannot be clawed back.

## Likelihood Explanation

Rate increases are routine (EigenLayer rewards accrue continuously). LZ message delivery takes minutes to hours; the staleness window is structural to the design. No special privileges are required — any address can call `deposit()`. An attacker only needs to monitor the L1 oracle (`ILRTOracle.rsETHPrice()`) and deposit on L2 before the LZ message arrives. The attacker can also extend the window by not calling `updateRate()` (it is permissionless but requires the caller to supply ETH for LZ fees, so no one is obligated to push the update). The `lastUpdated` field being stored but never used in any guard confirms this is a design omission. The exploit is repeatable on every rate increase cycle.

## Recommendation

1. **Add a staleness guard in `CrossChainRateReceiver.getRate()`:**
   ```solidity
   uint256 public constant MAX_RATE_AGE = 1 hours;

   function getRate() external view returns (uint256) {
       require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
       return rate;
   }
   ```
   This causes `deposit()` to revert when the rate is stale, preventing over-minting.

2. **Alternatively, expose `lastUpdated` from the receiver and add a staleness check inside `RSETHPoolV3.viewSwapRsETHAmountAndFee()`**, reverting or returning zero when the oracle is stale.

3. **Automate rate pushes** via a keeper/bot to minimize the staleness window and reduce the exploitable surface.

## Proof of Concept

```solidity
// Foundry fork test (L2 fork, no public mainnet required)
function testStaleRateDilutesExistingHolders() public {
    // 1. Existing holder deposited 10 ETH at rate 1.0e18 → holds 10e18 wrsETH
    //    Pool backing: 10 rsETH

    // 2. Simulate L1 rate increase to 1.1e18 (EigenLayer rewards distributed)
    //    LZ message NOT yet delivered — CrossChainRateReceiver still holds 1.0e18

    // 3. Attacker deposits 10 ETH at stale rate
    vm.deal(attacker, 10 ether);
    vm.prank(attacker);
    pool.deposit{value: 10 ether}("ref");

    // wrsETH minted to attacker = 10e18 * 1e18 / 1.0e18 = 10e18
    assertEq(wrsETH.balanceOf(attacker), 10e18);

    // 4. ETH bridged to L1 buys rsETH at true rate 1.1e18
    //    rsETH bought = 10e18 * 1e18 / 1.1e18 ≈ 9.09e18

    // 5. Total wrsETH supply: 20e18; total rsETH backing: ≈19.09e18
    //    Existing holder redemption per wrsETH: 19.09/20 * 1.1 ETH ≈ 1.05 ETH
    //    Expected (no dilution): 1.1 ETH
    //    Yield stolen per existing wrsETH: ≈0.05 ETH (~4.5%)

    assertLt(existingHolderRedemptionValue, 1.1 ether);
}
```

The test requires forking L2 with `CrossChainRateReceiver` holding a stale rate (achievable by simply not calling `updateRate()` after a simulated L1 price increase via `vm.mockCall` on the LRT oracle).
