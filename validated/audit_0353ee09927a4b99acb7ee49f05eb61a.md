### Title
Negative Threshold in `RequireMOf` / `TimelockMOf` Native Script Bypasses Signature Validation — (File: `eras/allegra/impl/src/Cardano/Ledger/Allegra/Scripts.hs`, `eras/shelley/impl/src/Cardano/Ledger/Shelley/Scripts.hs`)

---

### Summary

The `RequireMOf` (Shelley) and `TimelockMOf` (Allegra through Dijkstra) native script constructors store the threshold `m` as a signed `Int`. When `m` is negative, the `isValidMOf` evaluator immediately returns `True` without checking any sub-scripts or signatures, making the script an unconditional "always pass" script. A script author can craft a `RequireMOf (-1) [RequireSignature alice_key]` that appears to be a legitimate single-key multisig but requires zero witnesses. Any funds locked to such a script address can be spent by anyone without providing any signatures.

---

### Finding Description

**Root type — Shelley (`MultiSigMOf`):**

`eras/shelley/impl/src/Cardano/Ledger/Shelley/Scripts.hs:103` stores the threshold as a signed `Int`:

```haskell
MultiSigMOf !Int !(StrictSeq (MultiSig era))
``` [1](#0-0) 

The CBOR decoder at lines 274–277 decodes `m` via `decCBOR @Int`, which accepts any CBOR integer including negative values — no non-negativity guard is applied:

```haskell
3 -> do
  m <- decCBOR          -- Int: accepts negative CBOR integers
  multiSigs <- sequence <$> decCBOR
  pure (3, MultiSigMOf m <$> multiSigs)
``` [2](#0-1) 

The evaluator `isValidMOf` inside `evalMultiSig`:

```haskell
isValidMOf n StrictSeq.Empty = n <= 0
isValidMOf n (msig StrictSeq.:<| msigs) =
  n <= 0 || if go msig then isValidMOf (n - 1) msigs else isValidMOf n msigs
``` [3](#0-2) 

When `n = -1`, the first branch `n <= 0` evaluates to `True` immediately, short-circuiting all sub-script evaluation. The script passes with zero witnesses.

**Spec divergence:**

The formal specification at `eras/shelley/formal-spec/crypto-primitives.tex:189` types the threshold as `\N` (natural numbers):

```
\type{RequireMOf}~\N~[\Script]
``` [8](#0-7) 

The `decodeNatural` function in `libs/cardano-ledger-binary/src/Cardano/Ledger/Binary/Decoding/Decoder.hs:1479–1484` shows the correct pattern for enforcing non-negativity — it is simply not applied to the `m` field:

```haskell
decodeNatural = do
  !n <- decodeInteger
  if n >= 0
    then return $! fromInteger n
    else cborError $ DecoderErrorCustom "Natural" "got a negative number"
``` [9](#0-8)

### Impact Explanation

A script author submits a serialized native script `[3, -1, [[0, <alice_keyhash>]]]` (CBOR for `RequireMOf (-1) [RequireSignature alice]`). The script hash is computed from these bytes and a script address is derived. Funds sent to that address are guarded by a script that evaluates to `True` for any transaction, regardless of witnesses. The attacker — or any third party — can drain those funds in a transaction carrying zero key witnesses. This is a direct, unconditional loss of ADA caused by an invalid ledger state transition: the ledger accepts a spend that the script's apparent semantics forbid.

The vulnerability spans every era from Shelley through Dijkstra because `MultiSig` scripts are forward-compatible with `Timelock` evaluation (the Allegra formal spec notes that existing Shelley multisig outputs are spent by interpreting the encoded script as a Timelock script).

---

### Likelihood Explanation

The attacker role is "script author" — explicitly in scope. The attack path is: craft a script with a negative threshold, present it to a counterparty (e.g., as a "1-of-2 multisig" for a shared wallet or DeFi vault), and wait for funds to be deposited. The negative `m` is invisible to anyone who only inspects the key list and does not parse the integer field. No privileged access, consensus majority, or key compromise is required. The CBOR encoding of a negative integer is a single extra byte compared to the positive case, making detection by casual inspection unlikely.

---

### Recommendation

1. Change the field type in `MultiSigMOf` and `TimelockMOf` from `!Int` to `!Natural` (or `!Word`).
2. In the CBOR decoders for both `MultiSigRaw` and `TimelockRaw`, replace `m <- decCBOR` with `m <- decCBOR @Natural` (or add an explicit `when (m < 0) $ fail "RequireMOf threshold must be non-negative"` guard), mirroring the existing `decodeNatural` pattern.
3. Update the CDDL specifications to use `uint` instead of `int64` for the `script_n_of_k` threshold field, aligning the binary format with the formal spec's `\N` type.

---

### Proof of Concept

```
-- 1. Construct the malicious script in CBOR:
--    [3, -1, [[0, <alice_keyhash_28bytes>]]]
--    tag=3 → RequireMOf, m=-1, sub-scripts=[RequireSignature alice]

-- 2. Compute script hash:
--    scriptHash = blake2b_224(0x00 || script_cbor_bytes)
--    (0x00 is the Shelley native-script prefix tag)

-- 3. Derive script address and send ADA to it.

-- 4. Submit a spending transaction with:
--    inputs  = [utxo_at_script_address]
--    outputs = [attacker_address]
--    witnesses = { scripts: [malicious_script], vkeyWitnesses: [] }
--    -- zero key witnesses

-- 5. Ledger evaluation:
--    evalMultiSig {} (RequireMOf (-1) [RequireSignature alice])
--    → isValidMOf (-1) [RequireSignature alice]
--    → (-1) <= 0  ← True, short-circuit
--    → script passes; transaction accepted; funds stolen.
```
