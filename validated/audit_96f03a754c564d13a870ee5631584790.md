### Title
Enacted `ParameterChange` Immediately Lowers Voting Thresholds for Subsequent Same-Epoch Ratification, Enabling Unauthorized Treasury Withdrawal - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

In the Conway `RATIFY` rule, when a `ParameterChange` governance action is ratified, the updated `EnactState` (containing new protocol parameters, including new DRep voting thresholds) is immediately used to evaluate all subsequent governance actions in the **same ratification round**. Because `ParameterChange` (priority 4) is always sorted and processed before `TreasuryWithdrawals` (priority 5) by `reorderActions`, an attacker who controls enough DRep stake to ratify a `ParameterChange` that lowers `dvtTreasuryWithdrawal` can cause a `TreasuryWithdrawals` action to be ratified in the same epoch under the new, lower threshold — bypassing the threshold that was in effect when the withdrawal was proposed and voted on.

---

### Finding Description

The `ratifyTransition` function in `Ratify.hs` processes governance actions sequentially in priority order. When a `ParameterChange` action passes all ratification checks, it is immediately enacted via `trans @(EraRule "ENACT" era)`, which calls `enactmentTransition` and applies `applyPPUpdates` to `ensCurPParams`. The resulting `newEnactState` — with the updated parameters — is then threaded into the recursive call for the remaining actions: [1](#0-0) 

The voting threshold for `TreasuryWithdrawals` is read directly from `ensCurPParams` at the time of evaluation: [2](#0-1) 

The action priority ordering guarantees `ParameterChange` is always processed before `TreasuryWithdrawals`: [3](#0-2) 

`reorderActions` is called unconditionally inside `runConwayRatify` before the `RATIFY` rule is invoked: [4](#0-3) 

Critically, `ParameterChange` is **not** a delaying action, so subsequent actions in the same ratification round are not blocked: [5](#0-4) 

The `enactmentTransition` for `ParameterChange` immediately overwrites `ensCurPParams`: [6](#0-5) 

The result is that the `TreasuryWithdrawals` action is evaluated against the **post-`ParameterChange`** threshold, not the threshold that was in effect when the withdrawal was submitted and voted on.

---

### Impact Explanation

**Critical — Unauthorized treasury action is enacted.**

An attacker controlling enough DRep stake to ratify a `ParameterChange` that modifies `dvtTreasuryWithdrawal` (a governance-group parameter) can drain the treasury in the same epoch with fewer votes than the original `dvtTreasuryWithdrawal` threshold required. Concretely:

- Suppose `dvtPPGovGroupL` = T₁ and `dvtTreasuryWithdrawal` = T₂, with T₁ < T₂.
- An attacker with T₁ ≤ X% < T₂ DRep stake submits a `ParameterChange` lowering `dvtTreasuryWithdrawal` to X%, and simultaneously submits a `TreasuryWithdrawals` action.
- Both are voted on with X% DRep stake.
- At the epoch boundary: `ParameterChange` is ratified (X% ≥ T₁), `ensCurPParams` is updated, then `TreasuryWithdrawals` is ratified (X% ≥ X%).
- The treasury withdrawal is enacted with fewer votes than T₂ required — an unauthorized governance action.

This constitutes a direct, attacker-controlled loss of ADA from the treasury through an invalid ledger state transition (the withdrawal would have been rejected under the pre-existing threshold).

---

### Likelihood Explanation

**Medium.** The attacker needs:
1. Enough DRep stake to ratify a `ParameterChange` modifying governance-group parameters (threshold `dvtPPGovGroupL`).
2. Committee approval for both the `ParameterChange` and the `TreasuryWithdrawals` (same committee threshold applies to both).
3. Both actions to be active at the same epoch boundary.

The condition `dvtPPGovGroupL` < `dvtTreasuryWithdrawal` must hold. This is a configurable state, but the attack is realistic for any governance configuration where governance-parameter changes are easier to ratify than treasury withdrawals — a common design intent. No privileged key or supermajority is required beyond what is needed to ratify the `ParameterChange` itself.

---

### Recommendation

1. **Apply a delaying flag to `ParameterChange` actions that modify voting thresholds.** If a `ParameterChange` modifies any DRep or SPO voting threshold, treat it as a delaying action (set `rsDelayed = True`), preventing other actions from being ratified in the same epoch under the new thresholds.

2. **Alternatively, snapshot the initial `EnactState` for threshold evaluation.** Evaluate all ratification thresholds against the `EnactState` at the *start* of the ratification round (before any enactments), rather than the incrementally updated state. Only non-threshold state (e.g., treasury balance for `withdrawalCanWithdraw`) needs to be updated incrementally.

3. **At minimum, document the intended behavior** in the formal spec and confirm it is consistent with the Agda conformance specification.

---

### Proof of Concept

**Setup:**
- `dvtPPGovGroupL` = 51% (threshold for governance-group `ParameterChange`)
- `dvtTreasuryWithdrawal` = 67% (original threshold for treasury withdrawals)
- Attacker controls 55% of active DRep stake and has committee approval.

**Steps:**

1. Attacker submits `ParameterChange` (call it `PC`) setting `dvtTreasuryWithdrawal := 51%`.
2. Attacker submits `TreasuryWithdrawals` (call it `TW`) draining the treasury.
3. Attacker votes `VoteYes` on both `PC` and `TW` with 55% DRep stake; committee votes yes on both.
4. At the epoch boundary, `runConwayRatify` calls `reorderActions`, placing `PC` (priority 4) before `TW` (priority 5).
5. `ratifyTransition` evaluates `PC`: `prevActionAsExpected = True`, `acceptedByEveryone = True` (55% ≥ 51%). `PC` is enacted; `ensCurPParams` now has `dvtTreasuryWithdrawal = 51%`.
6. `ratifyTransition` evaluates `TW` with the updated `st'`: `votingDRepThreshold` reads `dvtTreasuryWithdrawal = 51%` from the new `ensCurPParams`. `dRepAccepted = True` (55% ≥ 51%). `TW` is enacted.
7. At the `EPOCH` boundary, `applyEnactedWithdrawals` transfers the treasury funds.

**Result:** The treasury is drained with 55% DRep stake, even though the threshold at the time of proposal and voting was 67%. Under the original threshold, `TW` would have been rejected (55% < 67%). [7](#0-6) [8](#0-7) [4](#0-3) [6](#0-5)

votingDRepThresholdInternal ::
  ConwayEraPParams era =>
  PParams era ->
  Bool ->
  GovAction era ->
  VotingThreshold
votingDRepThresholdInternal pp isElectedCommittee action =
  let thresholds@DRepVotingThresholds
        { dvtCommitteeNoConfidence
        , dvtCommitteeNormal
        , dvtMotionNoConfidence
        , dvtUpdateToConstitution
        , dvtHardForkInitiation
        , dvtTreasuryWithdrawal
        } -- We reset all (except InfoAction) DRep thresholds to 0 during bootstrap phase
          | hardforkConwayBootstrapPhase (pp ^. ppProtocolVersionL) = def
          | otherwise = pp ^. ppDRepVotingThresholdsL

        UpdateCommittee {} ->

            if isElectedCommittee
