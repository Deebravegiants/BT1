### Title
Incomplete Per-Entry Zero-Amount Validation in `TreasuryWithdrawals` Governance Proposals — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

---

### Summary

The `ZeroTreasuryWithdrawals` guard in the Conway `GOV` rule validates only that the **aggregate sum** of all withdrawal amounts is non-zero. It does not validate that each **individual** withdrawal entry is non-zero. An unprivileged proposer can therefore submit a `TreasuryWithdrawals` governance action containing one or more zero-amount entries alongside at least one positive entry. The proposal passes all ledger validation, is ratified, and is enacted — with the zero-amount entries persisted into `ensWithdrawals` and processed at epoch boundary — bypassing the design intent of the `ZeroTreasuryWithdrawals` predicate failure.

---

### Finding Description

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`, the only guard against zero-value treasury withdrawals is:

```haskell
-- The sum of all withdrawals must be positive
F.fold wdrls /= mempty ?! (injectFailure . ZeroTreasuryWithdrawals) pProcGovAction
```

`F.fold` on `Map AccountAddress Coin` sums the `Coin` values via the `Foldable`/`Monoid` instance. If the map contains `{addr1: Coin 0, addr2: Coin 100_000}`, the sum is `Coin 100_000 /= mempty`, so the check passes. The zero-amount entry for `addr1` is never individually inspected.

The existing test in `Test/Cardano/Ledger/Conway/Imp/GovSpec.hs` explicitly demonstrates this gap:

```haskell
wdrls <- mkTreasuryWithdrawalsGovAction $ withdrawals ++ [(accountAddress2, Coin 100_000)]
-- where withdrawals = [(accountAddress1, zero), (accountAddress2, zero)]
proposal <- mkProposal wdrls
submitBootstrapAwareFailingProposal_ proposal $
  FailBootstrap [disallowedProposalFailure proposal]
```

Post-bootstrap, this proposal (containing `{addr1: 0, addr2: 100_000}`) succeeds — only the bootstrap path rejects it (for an unrelated reason). No `ZeroTreasuryWithdrawals` failure is raised.

When enacted, `enactmentTransition` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Enact.hs` processes the action:

```haskell
TreasuryWithdrawals wdrls _ ->
  let wdrlsAmount = fold wdrls
      wdrlsNoNetworkId = Map.mapKeys (^. accountAddressCredentialL) wdrls
   in st
        { ensWithdrawals = Map.unionWith (<>) wdrlsNoNetworkId $ ensWithdrawals st
        , ensTreasury = ensTreasury st <-> wdrlsAmount
        }
```

All entries — including zero-amount ones — are merged into `ensWithdrawals`. At epoch boundary, `applyEnactedWithdrawals` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Epoch.hs` processes every entry in `ensWithdrawals`, calling `compactCoinOrError` and `addToBalanceAccounts` for zero-amount registered accounts. The `withdrawalCanWithdraw` ratification guard in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs` also only checks the aggregate:

```haskell
withdrawalCanWithdraw (TreasuryWithdrawals m _) treasury =
  Map.foldr' (<+>) zero m <= treasury
```

The same aggregate-only pattern is present in both the proposal validation and the ratification guard.

---

### Impact Explanation

**Medium.** An attacker-controlled governance proposal can include zero-amount individual withdrawal entries alongside positive entries, bypassing the intended `ZeroTreasuryWithdrawals` validation. When enacted:

- Zero-amount entries are written into `ensWithdrawals` and processed at epoch boundary, causing unnecessary state traversal.
- The proposal on-chain appears to grant treasury disbursements to certain addresses that in fact receive nothing — enabling social-engineering or misleading governance records.
- The treasury is debited only by the non-zero sum, so no ADA is directly lost; however, the enacted governance action contains entries that violate the design constraint that every individual withdrawal must be positive.

This matches the Medium impact criterion: *attacker-controlled proposals modify withdrawals outside design parameters*.

---

### Likelihood Explanation

**Low-to-Medium.** Any participant with sufficient ADA to pay the governance action deposit (`ppGovActionDepositL`) can submit a `TreasuryWithdrawals` proposal. No privileged role, leaked key, or consensus majority is required. The proposal must still be ratified by DReps and the Constitutional Committee, but the zero-amount entries are invisible to ratifiers who only see the aggregate amount. The barrier is the governance deposit and the ratification threshold, not any technical access control.

---

### Recommendation

Replace the aggregate-sum check with a per-entry check that also validates each individual amount:

```haskell
-- Require every individual withdrawal to be positive
let zeroEntries = Map.filter (== mempty) wdrls
failOnNonEmpty (Map.keys zeroEntries) (injectFailure . ZeroTreasuryWithdrawals)
-- Also require the total to be positive (handles the empty-map case)
F.fold wdrls /= mempty ?! (injectFailure . ZeroTreasuryWithdrawals) pProcGovAction
```

Or equivalently, require `all (> mempty) (Map.elems wdrls)` before the existing aggregate check. This mirrors the analogous `decodeNonZeroAmount` guard already applied to `MultiAsset` quantities in `eras/mary/impl/src/Cardano/Ledger/Mary/Value.hs`.

### Proof of Concept

1. Post-bootstrap Conway/Dijkstra ledger.
2. Register two stake credentials: `addr1`, `addr2`.
3. Submit a `TreasuryWithdrawals` governance action: `{addr1: Coin 0, addr2: Coin 1_000_000}`.
4. Observe: the GOV rule accepts the proposal — no `ZeroTreasuryWithdrawals` failure is raised.
5. Ratify and enact the proposal (DRep + CC votes).
6. At epoch boundary, `applyEnactedWithdrawals` processes both entries; `addr1` receives `Coin 0`, `addr2` receives `Coin 1_000_000`.
7. The enacted governance action on-chain lists `addr1` as a withdrawal recipient despite receiving nothing, violating the design invariant enforced by `ZeroTreasuryWithdrawals`.
