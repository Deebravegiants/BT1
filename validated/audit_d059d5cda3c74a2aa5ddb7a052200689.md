### Title
`respond()` Bypasses TEE Attestation Re-Verification, Allowing Nodes with Stale/Expired Attestations to Issue Threshold Signatures - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` functions check the `accept_requests` flag but never re-verify the calling node's TEE attestation freshness. The `accept_requests` flag is only updated by `verify_tee()`, which must be explicitly called. Between `verify_tee()` calls, a node whose attestation has expired or whose docker image has been removed from the whitelist can still call `respond()` and participate in threshold signing, bypassing the TEE security model.

### Finding Description

The contract's TEE security model relies on two mechanisms:

1. **`accept_requests` flag** — set by `verify_tee()` after calling `reverify_and_cleanup_participants()`. If fewer than threshold participants have valid attestations, `accept_requests` is set to `false`.
2. **`is_caller_an_attested_participant()`** — called by `assert_caller_is_attested_participant_and_protocol_active()` inside `respond()`. This function only checks that a stored attestation *exists* and that the stored identity fields match the caller. It does **not** call `reverify_participants()` to check whether the attestation is still valid (not expired, docker image still whitelisted, measurements still accepted). [1](#0-0) 

The `is_caller_an_attested_participant()` function checks account identity but never calls `reverify_participants()`: [2](#0-1) 

By contrast, `verify_tee()` and `vote_new_parameters()` both call `reverify_and_cleanup_participants()` before making trust decisions: [3](#0-2) [4](#0-3) 

The `respond()` function checks `accept_requests` and calls `assert_caller_is_attested_participant_and_protocol_active()`, but neither of these re-verifies attestation freshness: [5](#0-4) [6](#0-5) 

**Vulnerable window:** After a node's attestation expires or its docker image is removed from the whitelist, but before `verify_tee()` is called, `accept_requests` remains `true` and the stale stored attestation still passes `is_caller_an_attested_participant()`. The node can call `respond()` successfully.

### Impact Explanation

A node running a docker image that has been removed from the whitelist (e.g., due to a discovered vulnerability enabling key extraction) can still call `respond()` to contribute a threshold signature share. The contract accepts the response because it reads stale attestation state — the stored `VerifiedAttestation` entry — rather than re-verifying the node's current TEE status. This breaks the production safety invariant that only nodes with currently-valid TEE attestations may participate in signing, and can enable unauthorized threshold signature issuance from a node that should have been excluded.

This maps to the allowed impact: **contract execution-flow manipulation that breaks production safety/accounting invariants** (Medium), and potentially **unauthorized threshold signature issuance** (Critical) if the excluded node's image has a key-extraction vulnerability.

### Likelihood Explanation

- Docker image hashes are removed from the whitelist via `vote_remove_launcher_hash()` / `vote_remove_os_measurement()`, which requires all participants to vote. This is a realistic operational event (security patch rollout).
- Attestation certificates expire on a schedule; the window between expiry and the next `verify_tee()` call is non-zero and depends on operator cadence.
- `verify_tee()` is not called automatically; it must be triggered by a participant. In a busy network, this call may be delayed.
- An adversarial node can time its `respond()` call to land in this window.

### Recommendation

Add a per-caller attestation re-verification step at the start of `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()`. Specifically, call `reverify_participants()` for the calling node's `NodeId` before accepting the response, and return an error if the result is `TeeQuoteStatus::Invalid`. This mirrors the "pull accrual" fix recommended in the Licredity report.

```rust
// At the start of respond(), after assert_caller_is_attested_participant_and_protocol_active():
let signer_id = env::signer_account_id();
let participants = self.protocol_state.active_participants();
let info = participants.info(&signer_id).expect("already checked");
let node_id = self.tee_state
    .find_node_id_by_tls_key(&info.tls_public_key)
    .expect("already checked");
let tee_upgrade_deadline = Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);
if !matches!(
    self.tee_state.reverify_participants(&node_id, tee_upgrade_deadline),
    TeeQuoteStatus::Valid
) {
    return Err(TeeError::TeeValidationFailed.into());
}
```

### Proof of Concept

1. Deploy the contract with participants A, B, C (threshold = 2).
2. All participants submit valid attestations via `submit_participant_info()`.
3. Call `verify_tee()` → `accept_requests = true`.
4. Participants vote to remove participant A's docker image hash from the whitelist via `vote_remove_launcher_hash()` (all-participant vote).
5. Do **not** call `verify_tee()` again.
6. Participant A calls `respond(request, valid_signature)`.
7. **Expected:** `respond()` should fail because A's attestation no longer passes re-verification against the updated whitelist.
8. **Actual:** `respond()` succeeds. `accept_requests` is still `true`, and `is_caller_an_attested_participant()` only checks the stored attestation entry (which still exists), not whether it re-verifies against the current whitelist. Participant A's signature share is accepted and the yield is resolved. [7](#0-6) [8](#0-7)

    /// reverifies stored participant attestations.
    pub(crate) fn reverify_participants(
        &self,
        node_id: &NodeId,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeQuoteStatus {
        let allowed_mpc_docker_image_hashes =
            self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
        let allowed_launcher_compose_hashes = self.get_allowed_launcher_compose_hashes();
        let allowed_measurements = self.get_accepted_measurements();

        let participant_attestation = self.stored_attestations.get(&node_id.tls_public_key);
        let Some(participant_attestation) = participant_attestation else {
            return TeeQuoteStatus::Invalid("participant has no attestation".to_string());
        };

        // Verify the attestation quote
        let time_stamp_seconds = Self::current_time_seconds();
        match participant_attestation.verified_attestation.re_verify(
            time_stamp_seconds,
            &allowed_mpc_docker_image_hashes,
            &allowed_launcher_compose_hashes,
            &allowed_measurements,
        ) {
            Ok(()) => TeeQuoteStatus::Valid,
            Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
        }
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
