### Title
Missing Request-to-Response Hash Binding in `respond_verify_foreign_tx` Enables Single-Node Forged Foreign-Chain Verification - (File: crates/contract/src/lib.rs)

### Summary

The `respond_verify_foreign_tx` function verifies that a signature is valid for a caller-supplied `payload_hash`, but never checks that `payload_hash` actually corresponds to the pending `VerifyForeignTransactionRequest` being resolved. A single Byzantine attested MPC participant can replay any previously observed valid `(payload_hash, signature)` pair to satisfy any pending foreign-transaction verification request, delivering a forged verification result to bridge callers.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` accepts two independent parameters: a `VerifyForeignTransactionRequest` (used as the map key to look up and drain pending yields) and a `VerifyForeignTransactionResponse` (containing a free-form `payload_hash` and `signature`). [1](#0-0) 

The contract's only cryptographic check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

This confirms the signature is valid for *some* hash against the root public key, but it never verifies that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` for the specific `request` being resolved. [2](#0-1) 

The canonical hash is defined as `SHA-256(borsh(ForeignTxSignPayload))`, which binds the `ForeignChainRpcRequest` (tx_id, extractors, etc.) and the `values` (extracted on-chain data). The contract never reconstructs or checks this binding at response time. [3](#0-2) 

Contrast this with the regular `respond` function, where the payload hash is taken directly from `request.payload` — the hash is structurally bound to the request and cannot be substituted: [4](#0-3) 

The `respond_verify_foreign_tx` function then unconditionally resolves all pending yields for the supplied `request` key with the mismatched response bytes: [5](#0-4) 

### Impact Explanation

A single Byzantine attested MPC participant can:

1. Observe any previously completed `respond_verify_foreign_tx` call on-chain (all calls are public), recording a valid `(payload_hash_Y, sig_Y)` pair for foreign transaction Y.
2. Wait for a new `verify_foreign_transaction` request for a different transaction X to appear in `pending_verify_foreign_tx_requests`.
3. Call `respond_verify_foreign_tx(request_X, { payload_hash_Y, sig_Y })`.
4. The contract accepts the call — `sig_Y` is a valid MPC signature for `payload_hash_Y` — and drains all pending yields for `request_X` with the mismatched response.

Bridge contracts receive a `VerifyForeignTransactionResponse` where `payload_hash` does not correspond to the transaction they requested. Bridge contracts that do not use the SDK's `ForeignChainSignatureVerifier.verify_signature` (which does check `expected_payload_hash == response.payload_hash`) will accept the forged verification, potentially releasing funds or executing bridge logic for a transaction that was never actually verified on the foreign chain. [6](#0-5) 

This matches the allowed impact: **High — forged foreign-chain verification causing invalid bridge execution**.

### Likelihood Explanation

- Requires only **one** Byzantine attested MPC participant — well below the signing threshold.
- The attacker needs no special capability beyond being an active participant: all `respond_verify_foreign_tx` calls are on-chain and publicly observable, so collecting valid `(payload_hash, signature)` pairs is trivial.
- The `ForeignTxSignPayload` hash includes no NEAR contract address, chain ID, or nonce, so any previously produced signature is permanently replayable within the same epoch (same root key).
- The `VerifyForeignTransactionRequest` key contains no caller identity or nonce, so any pending request for the same `(domain_id, tx_id, extractors)` tuple is a valid target. [7](#0-6) 

### Recommendation

In `respond_verify_foreign_tx`, require the responder to supply the `values` (extracted values) alongside the response, then recompute the expected hash on-chain and assert it matches `response.payload_hash`:

```rust
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: provided_values,
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This binds the response hash to the specific pending request, closing the replay window. The SDK's `ForeignChainSignatureVerifier` already performs this check at the caller level; the contract must enforce it at the protocol level. [8](#0-7) 

### Proof of Concept

1. Alice's bridge contract calls `verify_foreign_transaction` for Bitcoin `tx_id=X`. The contract stores a pending yield keyed by `request_X = { domain_id, tx_id=X, extractors=[BlockHash] }`.

2. The MPC network previously processed a request for `tx_id=Y` and produced `payload_hash_Y = SHA256(borsh({ request_Y, [block_hash_B] }))` with a valid MPC signature `sig_Y`. This response is visible on-chain.

3. Byzantine node calls:
   ```
   respond_verify_foreign_tx(
     request = request_X,          // pending request for tx_id=X
     response = {
       payload_hash: payload_hash_Y,  // hash of tx_id=Y's payload
       signature: sig_Y               // valid MPC sig for tx_id=Y
     }
   )
   ```

4. The contract at line 729 verifies `sig_Y` is valid for `payload_hash_Y` against the root key — **passes**.

5. The contract at line 749 resolves all pending yields for `request_X` with `{ payload_hash_Y, sig_Y }`.

6. Alice's bridge contract receives `{ payload_hash_Y, sig_Y }`. If it skips the `expected_hash == response.payload_hash` check and only verifies the signature, it concludes that `tx_id=X` was verified — but the MPC network never actually inspected `tx_id=X`. [9](#0-8)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
