### Title
`is_caller_an_attested_participant` Does Not Check Attestation Expiry, Allowing Nodes with Expired TEE Attestations to Submit Signatures - (File: `crates/contract/src/tee/tee_state.rs`)

### Summary

The `is_caller_an_attested_participant` function, which gates all `respond`, `respond_ckd`, and `respond_verify_foreign_tx` calls, verifies only that an attestation *exists* in storage for the caller. It never checks whether that attestation has expired. The expiry check exists only in `reverify_and_cleanup_participants`, which is invoked exclusively by `verify_tee`. This creates a window — from the moment a node's attestation expires until `verify_tee` is next called — during which a node whose TEE guarantee is no longer valid can still submit signatures accepted by the contract.

### Finding Description

`is_caller_an_attested_participant` in `crates/contract/src/tee/tee_state.rs` performs four checks:

1. Is the caller in the active participant set?
2. Does a stored attestation exist for the caller's TLS key?
3. Does the attestation's `account_id` match the signer?
4. Does the attestation's `account_public_key` match the signer's key? [1](#0-0) 

It does **not** check whether the attestation's expiry timestamp has passed. The expiry check is implemented only in `reverify_and_cleanup_participants`, called exclusively from `verify_tee`: [2](#0-1) 

`assert_caller_is_attested_participant_and_protocol_active` — which calls `is_caller_an_attested_participant` — is the sole TEE gate for `respond`, `respond_ckd`, and `respond_verify_foreign_tx`: [3](#0-2) 

The analog to M-2 is direct: just as `priceCollateralToUSD` always subtracts the withdrawal fee while the actual Lyra pool only does so when live boards exist, `is_caller_an_attested_participant` always accepts a stored attestation while `reverify_and_cleanup_participants` conditionally rejects it when expired. The contract applies the "attestation is valid" conclusion unconditionally at the `respond` path, but conditionally (with expiry) only at the `verify_tee` path.

During the `Running` state, `active_participants` includes the node with the expired attestation: [4](#0-3) 

The `respond` function checks `accept_requests` and `is_running_or_resharing`, but neither of those flags reflects per-node attestation expiry: [5](#0-4) 

### Impact Explanation

During the window between a node's attestation expiry and the next `verify_tee` call, a node whose TEE guarantee is no longer valid can call `respond` (or `respond_ckd` / `respond_verify_foreign_tx`) and have its response accepted by the contract. The contract's invariant — "only nodes currently running inside a valid TEE can contribute to threshold signatures" — is broken. A node that has exited its TEE environment (e.g., migrated to a non-TEE host) but whose attestation has not yet been cleaned up retains full signing authority. This breaks the participant-state safety invariant that TEE attestation validity is continuously enforced at the point of signature submission, not only at periodic `verify_tee` sweeps.

This matches the allowed Medium impact: *participant-state manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.*

### Likelihood Explanation

Attestation certificates have finite lifetimes. `verify_tee` is not called automatically by the contract — it must be triggered externally by a participant. Any gap between expiry and the next `verify_tee` call is an open window. The window length is operator-dependent but is a normal operational condition, not an edge case. A node operator who intentionally or accidentally lets their attestation lapse retains signing capability during this gap.

### Recommendation

Add an expiry check inside `is_caller_an_attested_participant` that mirrors the logic already present in `reverify_and_cleanup_participants`. Specifically, retrieve the stored attestation's expiry timestamp and compare it against `env::block_timestamp()` before returning `Ok(())`. This ensures the TEE validity invariant is enforced at every `respond` call, not only at `verify_tee` sweeps.

### Proof of Concept

1. Node A is a current participant with a stored attestation whose `expiry_timestamp_seconds` is `T`.
2. Block time advances past `T`. `verify_tee` has not yet been called.
3. Node A calls `respond(request, signature)`.
4. `assert_caller_is_attested_participant_and_protocol_active` calls `is_caller_an_attested_participant`.
5. The function finds the stored attestation, confirms `account_id` and `account_public_key` match, and returns `Ok(())` — no expiry check is performed.
6. The signature is accepted and the pending request is resolved, despite Node A's TEE guarantee having lapsed. [6](#0-5)

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
