### Title
Unprotected `init` Function Allows Any Caller to Seize Full MPC Participant Control — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `init` function on the MPC contract carries no `#[private]` guard and performs no caller identity check. Because the production deployment scripts deploy the contract binary and call `init` as two separate transactions, a window exists in which any external account can race to call `init` first, install themselves as the sole participant with threshold 1, and receive mocked-valid TEE attestations — giving them complete, uncontested control over the MPC signing network.

---

### Finding Description

The `init` function is the sole public entry point that transitions the contract from `NotInitialized` to `Running` and establishes the authoritative participant set and threshold:

```rust
// crates/contract/src/lib.rs:1924-1945
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    ...
    let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);
    Ok(Self { protocol_state: ProtocolContractState::Running(...), ... })
}
```

There is no `#[private]` attribute and no check on `env::predecessor_account_id()` or `env::signer_account_id()`. Any NEAR account can call it.

The NEAR `#[init]` macro only prevents a *second* call once state is written; it does not restrict *who* may make the first call. This is confirmed by the ABI snapshot, which lists `init` with only the `"init"` modifier, while the analogous `init_running` and `migrate` functions both carry `"init"` **and** `"private"`:

```
// crates/contract/tests/snapshots/abi__abi_has_not_changed.snap
{ "name": "init",         "modifiers": ["init"]           }   // ← no "private"
{ "name": "init_running", "modifiers": ["init","private"] }   // ← protected
{ "name": "migrate",      "modifiers": ["init","private"] }   // ← protected
```

The production deployment script explicitly separates deployment from initialization using `without-init-call`:

```bash
# localnet/tee/scripts/rust-launcher/deploy-tee-cluster.sh:1142-1144
near contract deploy "$MPC_CONTRACT_ACCOUNT" use-file "$MPC_CONTRACT_PATH" \
  without-init-call network-config "$NEAR_NETWORK_CONFIG" sign-with-keychain send
```

`init_contract` is then called as a separate phase. Between these two transactions the contract is live on-chain but uninitialized, and `init` is callable by anyone.

A critical amplifier: `init` calls `TeeState::with_mocked_participant_attestations(initial_participants)`, which immediately marks every supplied participant as TEE-attested without any real quote verification. An attacker who calls `init` with themselves as the sole participant (threshold = 1) is therefore immediately recognized as an attested participant by `assert_caller_is_attested_participant_and_protocol_active`, which gates `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`, and all `respond*` callbacks.

---

### Impact Explanation

An attacker who wins the race gains:

1. **Full participant authority** — they are the only entry in `parameters.participants()` and the only attested node in `tee_state`.
2. **Unilateral key generation** — with threshold = 1 they can call `vote_add_domains`, `start_keygen_instance`, and `vote_pk` alone, generating MPC key shares that only they hold.
3. **Unauthorized signature issuance** — they can call `respond` with any signature they compute off-chain, resolving any pending `sign` yield and delivering a forged signature to the waiting caller.
4. **Unauthorized CKD output** — same path via `respond_ckd`.
5. **Permanent lock-out of legitimate operators** — because `#[init]` prevents re-initialization, the legitimate deployer cannot recover without redeploying to a new account, invalidating all existing user integrations.

This matches **Critical: Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization** and **Critical: Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material**.

---

### Likelihood Explanation

- The production deployment script (`deploy-tee-cluster.sh`) explicitly uses `without-init-call`, confirming the two-step pattern is the real operational procedure.
- The localnet and testnet guides (`docs/localnet/localnet.md`, `docs/testnet/setup-guide-for-testnet-with-tee-support.md`) both show `init` called as a separate CLI transaction after deployment.
- Any attacker watching the NEAR blockchain for a `DeployContract` action on the known contract account can immediately submit a competing `init` call in the same or next block.
- No special privilege, key material, or collusion is required — a single NEAR account and a small gas fee suffice.

---

### Recommendation

Add `#[private]` to `init`, consistent with `init_running` and `migrate`. In NEAR, `#[private]` restricts the caller to `predecessor_account_id == current_account_id`, meaning only the contract account itself (i.e., a `DeployContract + FunctionCall` batch action signed by the contract's own key) can invoke it. This eliminates the race window entirely.

Alternatively, combine deployment and initialization into a single atomic transaction using NEAR's `DeployContract` + `FunctionCall` batch action, so the contract is never observable in an uninitialized state.

---

### Proof of Concept

1. Operator deploys the contract binary to `v1.signer` using `without-init-call` (as the production script does).
2. Attacker observes the `DeployContract` receipt on-chain.
3. Attacker submits:
   ```json
   near contract call-function as-transaction v1.signer init \
     json-args '{"parameters":{"threshold":1,"participants":{"next_id":1,"participants":[["attacker.near",0,{"tls_public_key":"ed25519:ATTACKER_KEY","url":"https://attacker.example"}]]}}}' \
     prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
     sign-as attacker.near network-config mainnet sign-with-keychain send
   ```
4. `init` succeeds; `attacker.near` is now the sole participant with a mocked-valid attestation.
5. Attacker calls `vote_add_domains` (threshold = 1, passes immediately), then `start_keygen_instance` and `vote_pk` with a key pair they generated locally.
6. Contract transitions to `Running` with the attacker's public key as the MPC root key.

// Contract developer helper API
#[near]
impl MpcContract {
    #[handle_result]
    #[init]
    pub fn init(
        parameters: dtos::ThresholdParameters,
        init_config: Option<dtos::InitConfig>,
    ) -> Result<Self, Error> {
        let parameters: ThresholdParameters = parameters.try_into_contract_type()?;
        // Log participant count and hash - full parameters exceed NEAR's 16KB log limit at ~100 participants
        let params_hash = env::sha256_array(borsh::to_vec(&parameters).unwrap());
        log!(
            "init: signer={}, num_participants={}, parameters_hash={:?}, init_config={:?}",
            env::signer_account_id(),
            parameters.participants().len(),
            params_hash,
            init_config,
        );
        parameters.validate()?;

        // TODO(#1087): Every participant must have a valid attestation, otherwise we risk
        // participants being immediately kicked out once contract transitions into running.
        let initial_participants = parameters.participants();
        let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);

  local keys_file="$1"
  log "Adding node keys to NEAR accounts using $keys_file"
  [ -f "$keys_file" ] || { err "Missing keys file at $keys_file. Run collect phase first."; exit 1; }

  jq -c '.[]' "$keys_file" | while read -r row; do
    local acct signer responder
    acct="$(echo "$row" | jq -r .account)"
    signer="$(echo "$row" | jq -r .signer_pk)"
    responder="$(echo "$row" | jq -r .responder_pk)"

    log "$acct: add signer key"
    near_add_key_skip_if_exists "$acct" "$signer" "signer"

    log "$acct: add responder key"
    near_add_key_skip_if_exists "$acct" "$responder" "responder"
  done
}
