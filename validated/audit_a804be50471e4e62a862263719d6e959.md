Looking at the `propose_update` / `do_update` flow in detail to confirm the finding.

### Title
Propose-Update Deposits Permanently Locked When Competing Update Is Executed — (`File: crates/contract/src/update.rs`)

---

### Summary

`propose_update` collects a storage-proportional NEAR deposit from each proposer. When any update reaches threshold and `do_update` fires, it bulk-clears **all** remaining proposals without refunding their deposits. Because `UpdateEntry` never records the proposer's `AccountId`, no refund path exists. The deposited NEAR is permanently locked in the contract.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` enforces a deposit equal to `env::storage_byte_cost() × bytes_used(update)`. [1](#0-0) 

Only the excess above `required` is returned to the caller; the exact required amount is retained by the contract. [2](#0-1) 

The stored `UpdateEntry` records `bytes_used` but **not** the proposer's `AccountId`: [3](#0-2) 

When `do_update` executes the winning proposal it calls `self.entries.clear()` and `self.vote_by_participant.clear()`, silently discarding every other pending proposal: [4](#0-3) 

No refund is issued for the cleared entries, and there is no `cancel_update`, `withdraw_deposit`, or equivalent function anywhere in the contract.

---

### Impact Explanation

For a contract-code update, `bytes_used` includes the full WASM bytecode plus a fixed overhead of `128 × sizeof(AccountId)` bytes: [5](#0-4) 

A 500 KB WASM binary requires roughly 5 NEAR in deposit at NEAR's storage byte cost. Multiple participants may each propose a competing upgrade simultaneously; when one wins, every other proposer's deposit is silently consumed by the contract with no recovery path. This permanently freezes participant funds held by the MPC contract, matching the **Medium** impact class: *balance/accounting invariant broken without relying on network-level DoS or operator misconfiguration*.

---

### Likelihood Explanation

The scenario is routine: during any upgrade cycle, multiple participants may independently propose updates (e.g., different binary versions or config variants). The first proposal to reach threshold executes and wipes the rest. No adversarial intent is required — normal governance activity is sufficient to trigger the loss. The only prerequisite is that more than one `propose_update` call is in flight, which is expected in a multi-participant network.

---

### Recommendation

1. **Store the proposer's `AccountId` in `UpdateEntry`** so a refund target is available.
2. **Refund all cleared entries in `do_update`** by iterating the remaining `entries` map before calling `clear()` and scheduling `Promise::new(entry.proposer).transfer(required_deposit(entry.bytes_used))` for each.
3. Alternatively, expose a `cancel_update(id)` function that lets a proposer withdraw their own deposit before an update is executed.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM binary, attaching ~5 NEAR deposit. `UpdateEntry { update, bytes_used: ~500_000 }` is stored; A's account is debited 5 NEAR.
2. Participant B calls `propose_update` with a different binary, attaching another ~5 NEAR.
3. Participants reach threshold on B's proposal and call `vote_update(B_id)`.
4. `do_update` executes:
   - Removes B's entry.
   - Calls `self.entries.clear()` — A's entry is dropped with no refund.
   - Calls `self.vote_by_participant.clear()`.
5. A's 5 NEAR is now permanently locked in the contract. No function exists to retrieve it. [4](#0-3) [6](#0-5)

    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
