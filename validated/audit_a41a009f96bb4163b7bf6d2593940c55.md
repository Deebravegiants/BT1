### Title
Unconditional `metricOmmSwapCallback` Invocation on Zero-Delta Swaps Causes DoS via `InvalidSwapDeltas` Revert — (`metric-core/contracts/MetricOmmPool.sol` / `metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

---

### Summary

`MetricOmmPool.swap` unconditionally invokes `metricOmmSwapCallback` on the caller regardless of whether the swap produced any non-zero token deltas. `MetricOmmSimpleRouter.metricOmmSwapCallback` unconditionally reverts with `InvalidSwapDeltas` when both deltas are `<= 0`. When a swap legitimately produces `(0, 0)` deltas — because the price limit is already exceeded at the current cursor position, or the pool has no liquidity in reachable bins — the pool calls the router's callback with `(0, 0)`, the router reverts, and the entire swap transaction reverts with a misleading error. Users who set `amountOutMinimum = 0` (accepting any output) or who use `priceLimitX64` defensively cannot execute swaps through the router in these conditions.

---

### Finding Description

**Root cause in `MetricOmmPool.swap` (lines 250–278):**

```solidity
if (zeroForOne) {
    if (amount1Delta < 0) {
        transferToken1(recipient, uint256(-amount1Delta));
    }
    uint256 balance0Before = balance0();
    // ← called unconditionally, even when amount0Delta == 0 && amount1Delta == 0
    IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
    if (amount0Delta > 0 && balance0Before + uint256(amount0Delta) > balance0()) {
        revert IncorrectDelta();
    }
}
```

There is no guard of the form `if (amount0Delta != 0 || amount1Delta != 0)` before the callback.

**Triggering condition in the swap math:** Each of the four internal swap helpers (`_swapToken0ForToken1SpecifiedInput`, `_swapToken0ForToken1SpecifiedOutput`, `_swapToken1ForToken0SpecifiedInput`, `_swapToken1ForToken0SpecifiedOutput`) returns `(0, 0, 0[, 0])` immediately when the price limit is already satisfied at the initial cursor position:

```solidity
if (params.priceLimitX64 >= initialPriceX64) {
    return (0, 0, 0, 0);   // zero-output, zero-input
}
```

After `deltasScaledToExternal`, both `amount0Delta` and `amount1Delta` are `0`.

**Revert in `MetricOmmSimpleRouter.metricOmmSwapCallback` (line 47):**

```solidity
function metricOmmSwapCallback(int256 amount0Delta, int256 amount1Delta, bytes calldata data) external override {
    if (amount0Delta <= 0 && amount1Delta <= 0) revert InvalidSwapDeltas();
    ...
}
```

This guard is unconditional — it does not distinguish between a caller-supplied invalid swap and a pool-reported zero-output swap.

**Analogous pattern to the external report:** `_migrateTo` unconditionally calls `_postDeposit` even when `collatToAdd == 0`, causing a revert in the staking contract that requires `_amount > 0`. Here, `swap` unconditionally calls `metricOmmSwapCallback` even when both deltas are zero, causing a revert in the router that requires at least one positive delta.

---

### Impact Explanation

Any user routing through `MetricOmmSimpleRouter` (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) whose swap resolves to zero output — due to a price limit already exceeded at the current bin cursor, or a pool with no liquidity in reachable bins — receives an `InvalidSwapDeltas` revert instead of a graceful zero-output result. This breaks the core swap flow for the router, which is the primary user-facing entry point. Users who set `amountOutMinimum = 0` (explicitly accepting zero output) are still blocked. The error message is misleading: the user did not supply invalid deltas; the pool computed them as zero.

---

### Likelihood Explanation

The condition is reachable by any unprivileged caller:
1. A pool whose current bin cursor is at a price already beyond `priceLimitX64` (common during volatile oracle moves).
2. A pool with no liquidity in the current bin and a price limit that prevents bin traversal.
3. Any `exactInput` multi-hop where an intermediate pool returns zero output.

No special setup or privileged role is required.

---

### Recommendation

Guard the callback invocation in `MetricOmmPool.swap` to skip it when both deltas are zero:

```solidity
// Only invoke callback when there is a non-zero settlement obligation
if (amount0Delta != 0 || amount1Delta != 0) {
    IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
}
```

Alternatively, relax the `InvalidSwapDeltas` guard in `MetricOmmSimpleRouter.metricOmmSwapCallback` to treat `(0, 0)` as a no-op rather than an error, consistent with the pool's legitimate zero-output behavior.

---

### Proof of Concept

1. Deploy a pool with `spreadFeeE6` such that the current `initialPriceX64` for a `zeroForOne` swap already satisfies `priceLimitX64 >= initialPriceX64`.
2. Call `MetricOmmSimpleRouter.exactInputSingle` with `amountOutMinimum = 0` and the same `priceLimitX64`.
3. Inside `pool.swap`, `_swapToken0ForToken1SpecifiedInput` returns `(0, 0, 0)` immediately.
4. `amount0Delta = 0`, `amount1Delta = 0`.
5. `pool.swap` calls `router.metricOmmSwapCallback(0, 0, "")`.
6. Router reverts: `if (amount0Delta <= 0 && amount1Delta <= 0) revert InvalidSwapDeltas()`.
7. `exactInputSingle` reverts with `InvalidSwapDeltas` despite `amountOutMinimum = 0`.
