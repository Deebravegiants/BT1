### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Malicious Participant — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the caller-supplied `response.payload_hash` is correctly signed by the MPC root key, but never checks that `payload_hash` is actually derived from the `request` argument supplied in the same call. A single attested participant who has obtained a valid threshold signature for foreign-chain request A can call `respond_verify_foreign_tx(request_B, {payload_hash: payload_hash_A, signature: sig_A})`, causing every queued caller of request B to receive a forged verification response whose payload hash corresponds to a completely different foreign transaction.

---

### Finding Description

In `respond_verify_foreign_tx` the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // taken from caller input
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // never tied to `request`
    &secp_pk,
).is_ok()
``` [1](#0-0) 

The contract then resolves every queued yield for `request` with the raw `response` bytes:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

Compare this with the ordinary `respond` function for threshold signatures, which derives the hash **from the request itself** (`request.payload_hash()`), not from the response. That binding is absent in `respond_verify_foreign_tx`.

The `VerifyForeignTransactionResponse` that callers receive contains both `payload_hash` and `signature`: [3](#0-2) 

Because the fan-out queue is caller-agnostic (multiple callers for the same foreign-chain request share one queue entry), every caller waiting on request B receives the forged `{payload_hash_A, sig_A}` response: [4](#0-3) 

The `ForeignTxSignPayload` that the MPC network actually signs encodes `(request_content, extracted_values)`. Because the contract never stores extracted values, it cannot independently recompute the expected `payload_hash` — but it **can** reject any `payload_hash` that does not embed the correct `request` content. It currently does neither.

---

### Impact Explanation

**High — forged foreign-chain verification enabling invalid bridge execution.**

A bridge contract that calls `verify_foreign_transaction` for Bitcoin tx_id B and receives a `VerifyForeignTransactionResponse` will typically:
1. Check that the MPC signature is valid for `payload_hash` — **passes** (sig_A is valid for payload_hash_A).
2. Use `payload_hash` as the canonical proof that tx B was observed on-chain.

Because `payload_hash_A` encodes tx A's content, the bridge contract is accepting a proof for the wrong transaction. This enables double-spend or invalid inbound bridge execution: an attacker can make the NEAR side believe a foreign deposit (tx B) was confirmed when only a different transaction (tx A) was actually verified by the MPC network.

---

### Likelihood Explanation

**Medium.** The attack requires exactly one malicious attested participant. In the Cait-Sith signing protocol the final reconstructed signature is broadcast to all co-signers before the leader submits it on-chain. Any participant who co-signed request A therefore holds `sig_A`. Because `respond_verify_foreign_tx` is callable by **any** attested participant (not only the leader), that participant can immediately replay `sig_A` against any concurrently pending request B. No threshold collusion is needed; no privileged role is required beyond being an active participant. [5](#0-4) 

---

### Recommendation

Derive the expected payload hash from the `request` argument rather than accepting it from the caller. The contract should reconstruct `ForeignTxSignPayload::V1 { request: request.request.clone(), values: response.values }` (requiring the responder to also supply the extracted values), compute its hash, and assert `computed_hash == response.payload_hash` before verifying the signature. This mirrors the pattern already used in `respond`, where `request.payload_hash()` — not a caller-supplied field — is the input to signature verification.

---

### Proof of Concept

```
Setup:
  - 3 participants, threshold 2, one foreign-tx domain (Secp256k1).
  - Alice submits verify_foreign_transaction(Bitcoin, tx_id=[1;32])  → request_A pending.
  - Bob   submits verify_foreign_transaction(Bitcoin, tx_id=[2;32])  → request_B pending.

MPC network signs request_A:
  - payload_hash_A = hash(ForeignTxSignPayloadV1 { request_A_content, values_A })
  - sig_A = threshold_sign(payload_hash_A)   // all 3 participants hold sig_A

Malicious participant (participant 1, below threshold) calls:
  respond_verify_foreign_tx(
      request  = request_B,          // Bob's pending request
      response = VerifyForeignTransactionResponse {
          payload_hash: payload_hash_A,   // hash of tx [1;32]
          signature:    sig_A,            // valid threshold sig for tx [1;32]
      }
  )

Contract checks:
  verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → OK   ✓
  pending_verify_foreign_tx_requests.get(request_B)      → found ✓
  → resolves Bob's yield with {payload_hash_A, sig_A}

Result:
  Bob's NEAR contract receives a VerifyForeignTransactionResponse
  whose payload_hash encodes tx [1;32], not tx [2;32].
  Any downstream bridge logic that trusts this response
  will incorrectly treat tx [2;32] as verified.
```

```rust
/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
