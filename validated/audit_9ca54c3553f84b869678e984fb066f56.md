### Title
Reference Script Fee Charged Only on Top-Level Transaction, Not Sub-Transactions in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the per-transaction reference script size limit is enforced over the combined size of the top-level transaction **and all its sub-transactions** (`batchNonDistinctRefScriptsSize`), but the minimum fee calculation charges the reference script fee only against the top-level transaction's reference scripts (`txNonDistinctRefScriptsSize`). An attacker can craft a top-level transaction with sub-transactions that carry large reference scripts up to the size limit, paying zero reference script fee for the sub-transaction scripts, while the node must still deserialize all of them.

---

### Finding Description

The Conway era introduced a tiered reference script fee (`minFeeRefScriptCostPerByte`) to compensate nodes for the deserialization cost of reference scripts. The fee is computed in `getConwayMinFeeTxUtxo`:

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

`txNonDistinctRefScriptsSize` only inspects the top-level transaction's `inputsTxBodyL` and `referenceInputsTxBodyL`:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

The Dijkstra era added `batchNonDistinctRefScriptsSize` specifically to account for sub-transactions:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

This function is used **only** in `validateAllRefScriptSize` (the size limit check in the Dijkstra LEDGER rule):

```haskell
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $ ...
```

It is **not** used in the fee calculation. The fee check (`feesOK` in the UTXO rule) calls `getMinFeeTxUtxo`, which resolves to `getConwayMinFeeTxUtxo` → `txNonDistinctRefScriptsSize` (top-level only). Sub-transaction reference scripts are therefore deserialized by every validating node but are not included in the fee base. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**Medium — Attacker-controlled transactions modify fees outside design parameters.**

The reference script fee was introduced precisely to make large-script deserialization attacks economically infeasible (see ADR 009). By placing large reference scripts exclusively in sub-transactions, an attacker pays zero reference script fee for those scripts while every validating node must still deserialize them. The size limit (`ppMaxRefScriptSizePerTx` enforced via `batchNonDistinctRefScriptsSize`) caps the total bytes, so the attack is bounded, but the fee paid is systematically less than the design intends. The attacker can include up to `ppMaxRefScriptSizePerTx` bytes of reference scripts in sub-transactions at no additional fee cost beyond the base transaction size fee. [5](#0-4) 

---

### Likelihood Explanation

Any Dijkstra era transaction author can exploit this. No privileged access, governance majority, or key compromise is required. The attacker simply constructs a top-level transaction with sub-transactions (`subTransactionsTxBodyL`) whose inputs reference UTxO entries containing large Plutus or native scripts. The ledger accepts the transaction as long as the batch size is within `ppMaxRefScriptSizePerTx` and the top-level fee satisfies `feesOK` (which only checks top-level ref script size). This is reachable from any unprivileged transaction sender in the Dijkstra era. [6](#0-5) 

---

### Recommendation

Replace the call to `txNonDistinctRefScriptsSize` inside `getConwayMinFeeTxUtxo` (or its Dijkstra-era override of `getMinFeeTxUtxo`) with `batchNonDistinctRefScriptsSize` so that the fee base includes reference scripts from all sub-transactions. Alternatively, define a Dijkstra-specific `getMinFeeTxUtxo` implementation that delegates to `batchNonDistinctRefScriptsSize`:

```haskell
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx
```

This ensures the fee charged is consistent with the resource cost actually imposed on validating nodes, matching the design intent of the reference script fee mechanism. [7](#0-6) 

---

### Proof of Concept

1. Attacker creates a UTxO entry containing a large Plutus V4 script (e.g., 100 KiB, well within `ppMaxRefScriptSizePerTx`).
2. Attacker constructs a Dijkstra top-level transaction with **no** reference inputs at the top level (zero ref script fee contribution from the top-level tx).
3. Attacker embeds a sub-transaction (`subTransactionsTxBodyL`) whose `referenceInputsTxBodyL` references the large-script UTxO entry.
4. The top-level transaction's fee is computed via `getConwayMinFeeTxUtxo` → `txNonDistinctRefScriptsSize` → 0 bytes of ref scripts → zero ref script fee surcharge.
5. `validateAllRefScriptSize` checks `batchNonDistinctRefScriptsSize` = 100 KiB ≤ `ppMaxRefScriptSizePerTx` → passes.
6. Every validating node deserializes the 100 KiB script from the sub-transaction, but the attacker paid only the base byte-size fee for the transaction body, not the tiered `minFeeRefScriptCostPerByte` surcharge that the ADR mandates for this deserialization cost. [8](#0-7) [9](#0-8) [10](#0-9)

## Context

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
