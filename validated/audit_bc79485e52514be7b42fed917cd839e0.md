### Title
`env::block_timestamp_ms()` Used as "Current Time" for TEE Attestation Expiry Checks Offers No Manipulation Protection — (File: crates/contract/src/tee/tee_state.rs)

---

### Summary

`TeeState::current_time_seconds()` derives the "current time" exclusively from `env::block_timestamp_ms()` — the NEAR equivalent of EVM's `block.timestamp`. This value is used as the authoritative clock for both initial attestation acceptance (`add_participant` / `submit_participant_info`) and periodic re-verification (`reverify_participants` / `verify_tee`). In NEAR's PoS model, block producers set the block timestamp and can delay transaction inclusion to a block whose timestamp falls past an attestation's certificate expiry, causing a valid attestation to be rejected. A Byzantine MPC participant who is also a block producer (or who can influence one) can exploit this to trigger forced resharing or freeze the network below threshold.

---

### Finding Description

`TeeState::current_time_seconds()` is defined as:

```rust
fn current_time_seconds() -> u64 {
    let current_time_milliseconds = env::block_timestamp_ms();
    current_time_milliseconds / 1_000
}
``` [1](#0-0) 

This value is passed directly to `attestation.verify_locally()` during `add_participant`:

```rust
attestation.verify_locally(
    expected_report_data.into(),
    Self::current_time_seconds(),   // ← block timestamp
    ...
)?;
``` [2](#0-1) 

And again to `re_verify()` during `reverify_participants`:

```rust
let time_stamp_seconds = Self::current_time_seconds();
match participant_attestation.verified_attestation.re_verify(
    time_stamp_seconds,
    ...
)
``` [3](#0-2) 

`reverify_participants` is called from `reverify_and_cleanup_participants`, which is called from the public `verify_tee()` method: [4](#0-3) 

The expiry check inside `VerifiedAttestation::re_verify` for Dstack attestations is:

```rust
let attestation_has_expired = *expiration_timestamp_seconds < timestamp_seconds;
if attestation_has_expired {
    return Err(VerificationError::Custom(...));
}
``` [5](#0-4) 

Additionally, `Timestamp::now()` — used in `AllowedDockerImageHashes::valid_entries()` to compute docker-image-hash grace-period deadlines — also reads `env::block_timestamp()`: [6](#0-5) 

A compounding concern is the explicit TODO in `ValidatedDstackAttestation`:

```rust
// TODO(#1639): This timestamp can not come from the contract,
// but should be extracted from the certificate itself.
pub expiry_timestamp_seconds: u64,
``` [7](#0-6) 

This indicates the stored expiry may not faithfully reflect the certificate chain's actual validity window, making the block-timestamp dependency the sole arbiter of expiry.

---

### Impact Explanation

A malicious NEAR block producer (or a Byzantine MPC participant who controls one) can:

1. **Delay a `submit_participant_info` transaction** until a block whose timestamp exceeds the attestation certificate's expiry. The contract then rejects a cryptographically valid attestation, preventing a legitimate node from joining or renewing. If the node is a current participant whose renewal is blocked, `verify_tee` will subsequently kick it out.

2. **Time a `verify_tee` call** to a block whose timestamp is just past one or more participants' attestation expiries, triggering forced resharing. If the number of participants with "expired" attestations drops the surviving set below threshold, `verify_tee` sets `accept_requests = false`, halting all signature production.

3. **Manipulate docker-image-hash grace-period expiry** via `Timestamp::now()`, prematurely expiring a valid image hash and blocking all attestation renewals that reference it.

The worst-case outcome — halting signature production or permanently freezing the network — maps to the Critical/High allowed impacts. The realistic near-term outcome (forcing resharing, blocking individual node renewals) maps to Medium: participant-state and contract execution-flow manipulation that breaks production safety invariants.

---

### Likelihood Explanation

NEAR's PoS protocol allows each block producer to set the block timestamp within a bounded window (must be strictly greater than the previous block and within a few seconds of wall-clock time). A single block producer acting alone — without threshold collusion — can delay transaction inclusion by choosing not to include a transaction in their assigned slot, effectively deferring it to a later block with a higher timestamp. MPC participants who are also NEAR validators (a realistic operational overlap) have this capability without any additional privilege escalation.

---

### Recommendation

1. **Do not use `env::block_timestamp_ms()` as the sole clock for attestation expiry.** Accept a caller-supplied `deadline` parameter in `submit_participant_info` (analogous to the recommended fix in the external report), or bound the check to a range anchored at submission time rather than the block timestamp alone.

2. **Resolve TODO(#1639):** Extract `expiry_timestamp_seconds` from the DCAP certificate chain itself (via `dcap_qvl` output), not from the block timestamp or any contract-side value. Until this is done, the expiry stored in `ValidatedDstackAttestation` is not a faithful representation of the certificate's actual validity window.

3. **For `verify_tee` / `reverify_participants`:** Consider requiring a minimum number of independent participant calls before acting on an expiry-triggered resharing, reducing the ability of a single actor to force resharing by timing a single transaction.

---

### Proof of Concept

```
1. Operator node N has a Dstack attestation whose certificate expires at T+60s.
2. Node N submits `submit_participant_info` at wall-clock time T+30s (30 seconds before expiry).
3. Malicious block producer M holds the transaction out of its block.
4. At wall-clock time T+61s, M includes the transaction in a block with timestamp T+61s.
5. `TeeState::current_time_seconds()` returns T+61 (seconds).
6. `attestation.verify_locally(..., T+61, ...)` calls `verify_dcap_quote(T+61)`.
7. The certificate chain is expired at T+61 → `VerificationError::ExpiredCertificate` is returned.
8. `submit_participant_info` returns `InvalidTeeRemoteAttestation`.
9. Node N cannot renew; on the next `verify_tee` call its stored attestation also fails
   `re_verify(current_time_seconds(), ...)` → `TeeQuoteStatus::Invalid`.
10. `reverify_and_cleanup_participants` returns `TeeValidationResult::Partial` excluding N.
11. If removing N drops the participant count below threshold,
    `verify_tee` sets `self.accept_requests = false` → all `sign` calls are rejected.
```
