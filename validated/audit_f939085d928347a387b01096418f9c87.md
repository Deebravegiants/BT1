### Title
Unbounded O(n) Iteration Over All Registered DReps Per Governance Proposal Transaction in `updateDormantDRepExpiry` - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

The `updateDormantDRepExpiry` function iterates over the entire `vsDReps` map (all registered DReps) on every transaction that contains at least one governance proposal and has accumulated dormant epochs. Because the DRep registry grows over time and is unbounded in size, this creates an O(n_dreps) per-transaction validation cost that scales with the total number of registered DReps on-chain, not with the size of the transaction itself.

---

### Finding Description

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`, the function `updateDormantDRepExpiries` is called on every transaction processed through the `CERTS` rule (pre-hardfork) and the `LEDGER` rule (post-hardfork):

```haskell
updateDormantDRepExpiries tx currentEpoch =
  let hasProposals = not . OSet.null $ tx ^. bodyTxL . proposalProceduresTxBodyL
   in if hasProposals
        then certVStateL %~ updateDormantDRepExpiry currentEpoch
        else id
```

When `hasProposals` is true and `numDormantEpochs ≠ 0`, it calls `updateDormantDRepExpiry`, which performs a full `Map.map` over the entire `vsDReps` map:

```haskell
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry
```

The `Map.map updateExpiry` call is O(n) in the number of registered DReps. This is invoked inside the per-transaction `LEDGER` transition rule at:

```haskell
pure $
  certState
    & updateDormantDRepExpiries tx curEpochNo
    & updateVotingDRepExpiries tx curEpochNo (pp ^. ppDRepActivityL)
    & certDStateL . accountsL %~ drainAccounts withdrawals
```

The `vsDReps` map lives in `VState` and is part of the `ConwayCertState`. The number of registered DReps is unbounded by protocol design — any credential can register as a DRep by paying the `ppDRepDeposit`. As the DRep registry grows, every governance proposal transaction triggers a full linear scan of all registered DReps whenever dormant epochs have accumulated.

The same pattern is also present in the Dijkstra era's `SUBENTITIES` rule via `dijkstraSubEntitiesTransition`, which also calls `Conway.updateDormantDRepExpiries`.

---

### Impact Explanation

**Medium. Attacker-controlled governance proposal transactions exceed intended validation limits.**

As the number of registered DReps grows, the per-transaction cost of processing any governance proposal transaction grows linearly. A sufficiently large DRep registry combined with accumulated dormant epochs causes every proposal-containing transaction to perform an O(n_dreps) map traversal during block validation. This can push block validation time beyond the slot budget (1 second on mainnet), causing honest block producers to miss slots or causing slower nodes to fail to validate blocks within the required window. The `utxosDepositedL .~ totalObligation certState2 govState1` call in the EPOCH rule also iterates over all proposals via `obligationGovState` (which calls `foldMap' gasDeposit $ proposalsActions`), compounding the epoch-boundary cost.

---

### Likelihood Explanation

The DRep deposit (`ppGovActionDeposit` / `ppDRepDeposit`) limits the rate of DRep registration but does not cap the total count. On mainnet, DRep count is already in the thousands and growing. The dormant-epoch condition (`numDormantEpochs ≠ 0`) is triggered naturally whenever no active governance proposals exist for a full epoch — a common occurrence. Any user submitting a governance proposal transaction after a dormant period triggers the full DRep map traversal. No privileged access is required; any unprivileged transaction author can submit a `ProposalProcedure`.

---

### Recommendation

Replace the full `Map.map updateExpiry` traversal in `updateDormantDRepExpiry` with an O(1) approach: store the cumulative dormant-epoch offset as a separate field and apply it lazily at DRep lookup time (similar to how `vsNumDormantEpochs` is already tracked). The actual expiry adjustment can be computed on-demand when a DRep's expiry is queried (e.g., in `vsActualDRepExpiry`), rather than eagerly rewriting every entry in the map on each proposal transaction. This mirrors the recommendation from the original report: precalculate necessary data so the per-transaction operation has O(1) complexity.

---

### Proof of Concept

1. Register N DReps (each paying `ppDRepDeposit`). N can be in the tens of thousands over time through normal governance participation.
2. Allow one or more epochs to pass with no active governance proposals, causing `vsNumDormantEpochs` to increment to a value > 0.
3. Submit any governance proposal transaction (`ProposalProcedure` in `ctbrProposalProcedures`).
4. The `LEDGER` rule calls `updateDormantDRepExpiries`, which detects `hasProposals = True` and `numDormantEpochs ≠ 0`, and executes `vsDRepsL %~ Map.map updateExpiry` — a full O(N) traversal of all registered DReps.
5. With N large enough, this traversal dominates block validation time for any block containing such a transaction.

updateDormantDRepExpiry ::
  -- | Current Epoch
  EpochNo ->
  VState era ->
  VState era
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry
  where
    numDormantEpochs = vState ^. vsNumDormantEpochsL
    updateExpiry =
      drepExpiryL
        %~ \currentExpiry ->
          let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
           in if actualExpiry < currentEpoch
                then currentExpiry
                else actualExpiry
