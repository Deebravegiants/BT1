### Title
User-Supplied Refund Amount Used Directly in UTxO Balance Accounting Without Complete Multi-Sub-Transaction Validation - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`)

### Summary

In the Dijkstra era, `dijkstraTotalRefundsTxCerts` computes the total refunds for UTxO balance conservation by reading the deposit/refund amount directly from the user-supplied certificate fields, completely ignoring the lookup functions that retrieve the actual stored deposit from ledger state. The comment in the code asserts "ledger rules will validate that they are accurate," but this guarantee is incomplete for the multi-sub-transaction case introduced by Dijkstra's nested transaction feature.

### Finding Description

In Conway era, `conwayTotalRefundsTxCerts` correctly uses `lookupStakingDeposit` and `lookupDRepDeposit` — functions that retrieve the actual deposit stored in the ledger state — to compute refunds for the UTxO balance check: [1](#0-0) 

In Dijkstra era, this is replaced by `dijkstraTotalRefundsTxCerts`, which ignores both lookup functions entirely and reads the `Coin` value directly from the certificate as supplied by the transaction author: [2](#0-1) 

This is wired into the `EraTxCert` instance for `DijkstraEra` with the lookup arguments discarded: [3](#0-2) 

For a **single transaction**, the `DELEG` rule (which reuses `Conway.conwayDelegTransition`) does validate the user-supplied refund against the stored deposit and rejects mismatches: [4](#0-3) 

However, Dijkstra introduces **nested sub-transactions** processed by the `SUBLEDGER` rule. In `dijkstraSubLedgersTransition`, the `SUBENTITIES` rule (which includes `SUBDELEG`) runs first and produces an updated `certStateFinal`. But the subsequent `SUBUTXOW` call — which performs the UTxO balance check via `updateUTxOStateNoFees` — is passed the **original** `certState`, not `certStateFinal`: [5](#0-4) 

The `SUBUTXO` transition calls `Shelley.updateUTxOStateNoFees`, which internally calls `certsTotalRefundsTxBody`, which calls `getTotalRefundsTxCerts` — resolved to `dijkstraTotalRefundsTxCerts` for `DijkstraEra`. This function reads the refund amount from the certificate, not from the ledger state: [6](#0-5) [7](#0-6) 

The `SUBDELEG` rule for sub-transactions also reuses `Conway.conwayDelegTransition`: [8](#0-7) 

The critical question is whether the outer `LEDGER` rule properly threads the updated `certState` (output of one `SUBLEDGER` invocation) as the input `certState` for the next sub-transaction. If it does not — i.e., each sub-transaction sees the same original `certState` — then:

1. An attacker registers a staking credential, paying deposit `D`.
2. The attacker submits an outer transaction containing `N` sub-transactions, each with `UnRegDepositTxCert cred D`.
3. Each sub-transaction's `SUBDELEG` sees the credential as still registered (original `certState`) and validates the refund `D` as correct.
4. Each sub-transaction's `SUBUTXO` balance check counts `D` as consumed (refund side).
5. The attacker receives `N × D` ADA in refunds while having paid only `D` in deposit.

The `SUBLEDGER` rule does return `LedgerState utxoStateAfterSubUtxow certStateFinal` with the updated cert state, so if the outer `LEDGER` rule threads this correctly, the second sub-transaction's `SUBDELEG` would fail because the credential is already unregistered. The outer `Ledger.hs` rule was not fully examined in this analysis, so the exploitability depends on that threading behavior.

### Impact Explanation

If the outer `LEDGER` rule does not thread `certState` between sub-transactions, an attacker can drain the deposit pot by claiming the same deposit `N` times through `N` sub-transactions. This constitutes a direct loss of ADA through an invalid ledger state transition (deposit pot decremented by `N × D` while only `D` was ever deposited). This matches the **Critical** impact category: direct destruction of ADA through an invalid ledger state transition.

### Likelihood Explanation

The Dijkstra era introduces nested transactions as a new feature. Any transaction author can construct a transaction with multiple sub-transactions. The attacker only needs a registered staking credential (trivially obtainable) and the ability to submit a transaction — no privileged access required. The attack is deterministic and repeatable. Likelihood is **High** if the certState threading gap exists in the outer `LEDGER` rule.

### Recommendation

1. The outer `LEDGER` rule must strictly thread the `certState` output of each `SUBLEDGER` invocation as the input for the next, so that a credential unregistered in sub-transaction `i` is seen as unregistered by sub-transaction `i+1`.
2. Alternatively, add an explicit cross-sub-transaction uniqueness check: a given `UnRegDepositTxCert` credential must not appear in more than one sub-transaction within the same outer transaction.
3. Consider whether `dijkstraTotalRefundsTxCerts` should be changed to use the lookup functions (as Conway does) rather than trusting the certificate-embedded value, so that the UTxO balance check is independently grounded in ledger state rather than relying solely on `DELEG` validation.

### Proof of Concept

1. Register staking credential `cred` with deposit `D = ppKeyDeposit`.
2. Construct outer transaction `tx` with `subTransactionsTxBodyL` containing `N` sub-transactions `[subTx_1, ..., subTx_N]`, each with `certsTxBodyL = [UnRegDepositTxCert cred D]` and sufficient UTxO inputs to balance each sub-transaction individually (each sub-transaction's consumed side includes `D` as a refund).
3. Submit `tx`. If certState is not threaded between sub-transactions, all `N` sub-transactions pass `SUBDELEG` (each sees `cred` as registered) and each `SUBUTXO` balance check accepts `D` as a valid refund.
4. The deposit pot is decremented by `N × D`; the attacker's outputs across all sub-transactions total `N × D` more than their inputs, net of the single `D` deposit paid at registration.
