Let me look at the specific details of the Ton/Ethereum/Solana handling and the contract's supported chains mechanism more carefully.

### Title
Node `execute_foreign_chain_request` Unconditionally Bails for Contract-Supported Chains (Ethereum, Solana, Ton), Permanently Stalling All `verify_foreign_transaction` Requests for Those Chains — (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

The MPC node's `execute_foreign_chain_request` function explicitly bails for `Ethereum`, `Solana`, and `Ton` chain variants. However, the contract imposes no such restriction: any participant can register these chains as supported via `register_foreign_chain_config` / `register_foreign_chain_support`, and the contract's `verify_foreign_transaction` will accept user requests for them. Once these chains are in the supported set, the node's own `chain_is_supported` guard passes — but the subsequent chain-dispatch match arm immediately bails. Every `verify_foreign_transaction` request for these chains is accepted on-chain, silently dropped by every node, and left to time out, permanently breaking the request lifecycle for those chains.

---

### Finding Description

**Contract side — no restriction on which chains can be registered as supported:**

`get_supported_foreign_chains()` returns the strict intersection of every active participant's registered chains. [1](#0-0) 

Any participant can call `register_foreign_chain_support` with any `ForeignChain` variant, including `Ethereum`, `Solana`, and `Ton`. The contract imposes no allowlist at registration time. The sandbox test `verify_foreign_transaction__should_succeed` explicitly exercises Ethereum and Ton as supported chains: [2](#0-1) 

The helper `register_foreign_chain_configuration` registers any `ForeignChain` variant for all participants: [3](#0-2) 

**Node side — chain-dispatch bails for three supported variants:**

Inside `execute_foreign_chain_request`, after the `chain_is_supported` guard passes, the match arm for `Ethereum`, `Solana`, and `Ton` unconditionally bails:

```rust
dtos::ForeignChainRpcRequest::Ethereum(_request) => {
    bail!("ForeignChainRpcRequest::Ethereum is unsupported")
}
dtos::ForeignChainRpcRequest::Solana(_request) => {
    bail!("ForeignChainRpcRequest::Solana is unsupported")
}
// ...
dtos::ForeignChainRpcRequest::Ton(_request) => {
    bail!("ForeignChainRpcRequest::Ton is unsupported")
}
``` [4](#0-3) [5](#0-4) 

The `chain_is_supported` guard that precedes the match reads from the contract's supported-chain set: [6](#0-5) 

So the guard passes whenever the chain is in the supported set — which is exactly the condition under which the bail is reached. The two checks are contradictory: the contract says the chain is supported; the node says it is not.

**The discrepancy is structural, not accidental:**

The `ForeignChainRpcRequest` enum is `#[non_exhaustive]` and includes `Ethereum`, `Solana`, and `Ton` as first-class variants: [7](#0-6) 

The contract's `verify_foreign_transaction` accepts any chain in the supported set without further restriction: [8](#0-7) 

---

### Impact Explanation

**Impact: Medium — request-lifecycle invariant break.**

The production safety invariant is: *"a request accepted by the contract will eventually be fulfilled or explicitly rejected by the network."* When Ethereum, Solana, or Ton are in the supported set, every `verify_foreign_transaction` request for those chains is accepted on-chain (deposit taken, yield queued), but every node silently bails during `execute_foreign_chain_request`. No node ever calls `respond_verify_foreign_tx`. The request stalls until the yield-resume timeout fires and the transaction fails. The user receives no signature and no actionable error — only a timeout failure. This breaks the request-lifecycle accounting invariant for those chains and permanently prevents the `verify_foreign_transaction` feature from functioning for them.

---

### Likelihood Explanation

**Likelihood: High (conditional on chain registration).**

The contract imposes no guard preventing Ethereum, Solana, or Ton from being registered as supported. The sandbox test suite explicitly registers Ethereum and Ton as supported chains and asserts that `verify_foreign_transaction` succeeds at the contract level — demonstrating that this is an intended, tested code path, not an edge case. Any operator following the documented registration flow for these chains triggers the condition. Once triggered, every unprivileged user who submits a `verify_foreign_transaction` request for those chains is affected.

---

### Recommendation

1. **Enforce the restriction at registration time in the contract.** Add a contract-level allowlist of chains for which the node has a working implementation. Reject `register_foreign_chain_support` calls that include chains not yet implemented in the node (Ethereum, Solana, Ton). This closes the gap between what the contract advertises as supported and what the node can actually process.

2. **Alternatively, implement the missing node-side handlers.** Add working `execute_foreign_chain_request` arms for `Ethereum`, `Solana`, and `Ton` analogous to the existing `Abstract`, `Bnb`, `Base`, etc. arms.

3. **At minimum, surface the bail as an explicit on-chain failure** rather than a silent timeout, so callers receive an actionable error instead of waiting for the yield-resume timeout.

---

### Proof of Concept

1. All active participants call `register_foreign_chain_support` with `ForeignChain::Ethereum` (or `Solana` / `Ton`). The contract's `get_supported_foreign_chains()` now includes Ethereum. [9](#0-8) 

2. Any unprivileged user calls `verify_foreign_transaction` with `ForeignChainRpcRequest::Ethereum(...)`. The contract's chain-support check passes; the request is queued. [8](#0-7) 

3. Each MPC node picks up the request. In `execute_foreign_chain_request`, `chain_is_supported` passes (Ethereum is in the supported set), then the match arm immediately bails: [10](#0-9) 

4. No node calls `respond_verify_foreign_tx`. The yield-resume timeout fires; the user's transaction fails with a timeout error. The request lifecycle is broken for all Ethereum (or Solana / Ton) requests. [11](#0-10)

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }

async fn chain_is_supported(
    policy_reader: &impl ReadSupportedForeignChain,
    request: &dtos::ForeignChainRpcRequest,
) -> Result<(), ForeignChainSupportError> {
    let on_chain_foreign_chains_support = policy_reader
        .get_supported_chains()
        .await
        .map_err(ForeignChainSupportError::FailedToReadContract)?;

    let requested_chain = request.chain();

    if on_chain_foreign_chains_support.contains(&requested_chain) {
        Ok(())
    } else {
        Err(ForeignChainSupportError::ChainNotSupported {
            requested: requested_chain,
        })
    }
}
