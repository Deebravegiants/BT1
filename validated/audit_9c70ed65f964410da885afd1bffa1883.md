Audit Report

## Title
Zero wrsETH Minted on Dust Deposits Due to Missing Zero-Amount Guard - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The `deposit` functions across `RSETHPoolV3`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge` compute the wrsETH/rsETH output via integer division in `viewSwapRsETHAmountAndFee`. When `amountAfterFee * 1e18 < rsETHToETHrate`, Solidity truncation produces `rsETHAmount = 0`, yet no guard reverts the transaction. A depositor sending a dust amount receives zero wrsETH while their ETH is permanently absorbed into the pool.

## Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee` (ETH path):
```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```
Once `rsETHToETHrate > 1e18` (normal post-launch state), any deposit where `amountAfterFee < rsETHToETHrate / 1e18` (i.e., `amountAfterFee < 1` for a rate just above 1e18, or more generally any sub-threshold dust) yields `rsETHAmount = 0` via truncation.

The ETH deposit function in `RSETHPoolV3` then proceeds unconditionally:
```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // mints 0 — no revert
```
The only existing guard is `if (amount == 0) revert InvalidAmount()`, which checks the input, not the output. The same pattern is present in all four pool variants and in both the ETH and token deposit paths. `RSETHPoolNoWrapper` calls `rsETH.safeTransfer(msg.sender, 0)` instead of minting, which also succeeds silently. The deposited ETH is retained by the contract and accrues to existing wrsETH holders.

## Impact Explanation
**Low. Contract fails to deliver promised returns.** A user who sends a non-zero ETH amount receives zero wrsETH in return. The transaction succeeds without revert, and the deposited ETH is permanently held in the pool. The per-transaction loss is bounded to dust amounts (amounts below `rsETHToETHrate / 1e18` wei), making the absolute value negligible, but the contract demonstrably fails to deliver its promised return without informing the caller.

## Likelihood Explanation
The condition is reachable by any unprivileged external caller via the public `deposit` functions. No special role, front-running, or oracle manipulation is required. The condition activates as soon as `rsETHToETHrate > 1e18`, which is the normal operating state after any yield has accrued. A scripting error, test transaction, or wallet rounding that sends 1 wei will silently result in zero wrsETH minted.

## Recommendation
Add a zero-output guard immediately after computing `rsETHAmount` in each deposit function across all four pool variants:
```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
require(rsETHAmount != 0, "zero wrsETH minted");
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);
```
Apply the same guard to the token deposit path (`deposit(address,uint256,string)`) in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`, and to the ETH and token paths in `RSETHPoolNoWrapper`.

## Proof of Concept
Assume `rsETHToETHrate = 1.05e18` (5% yield accrued).

1. Alice calls `RSETHPoolV3.deposit{value: 1}("")`.
2. `fee = 1 * feeBps / 10_000 = 0` (truncates to zero).
3. `amountAfterFee = 1`.
4. `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (Solidity truncation).
5. `feeEarnedInETH += 0`.
6. `wrsETH.mint(Alice, 0)` — Alice receives 0 wrsETH.
7. Alice's 1 wei is permanently held in the pool.

**Foundry fuzz test plan:**
```solidity
function testFuzz_dustDepositMintsZero(uint256 amount) public {
    vm.assume(amount > 0 && amount < rsETHToETHrate / 1e18);
    uint256 balanceBefore = wrsETH.balanceOf(alice);
    vm.deal(alice, amount);
    vm.prank(alice);
    pool.deposit{value: amount}("");
    assertEq(wrsETH.balanceOf(alice), balanceBefore); // zero minted
}
```
This fuzz test will find inputs where the deposit succeeds but mints zero wrsETH, confirming the missing guard.
