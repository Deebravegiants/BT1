### Title
Caller's Attached NEAR Deposit Permanently Stuck in Wallet Contract on Early Return from `rlp_execute` - (File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs)

### Summary
`rlp_execute` is marked `#[payable]` and accepts an attached NEAR deposit from any external caller. When `has_in_flight_tx` is `true`, the function returns a `PromiseOrValue::Value(...)` immediately — a successful receipt from the runtime's perspective — without issuing any refund. The attached deposit is silently absorbed into the wallet contract's balance and is permanently inaccessible to the caller.

### Finding Description
`rlp_execute` in the `WalletContract` is the main entry point for ETH-implicit account transactions. It is decorated `#[payable]`, meaning any caller may attach a NEAR deposit. The contract has a `CallerDeposit` tracking mechanism that records an external caller's deposit and refunds it via `rlp_execute_callback` if the downstream cross-contract call fails. However, this mechanism is only reached if the function proceeds past the `has_in_flight_tx` guard and into `inner_rlp_execute`.

When `has_in_flight_tx == true`, the function returns at line 97–104 with `PromiseOrValue::Value(ExecuteResponse { success: false, ... })`. From the NEAR runtime's perspective this is a **successful** receipt execution (the function did not panic), so the runtime does not generate an automatic deposit refund to the predecessor. The attached deposit is transferred to the wallet contract's balance and the caller has no mechanism to recover it.

The same deposit-loss path exists at line 126 (`Err(e) => PromiseOrValue::Value(e.into())`) for non-Relayer errors returned by `inner_rlp_execute` (e.g., `AccountNonceExhausted`, `UserError`), because `CallerDeposit` is constructed inside `inner_rlp_execute` and never returned to the outer function on the error path.

### Impact Explanation
The corrupted value is the **caller's NEAR balance**: the deposit is debited from the caller and credited to the wallet contract without the intended action being executed. The caller (typically a relayer) permanently loses the attached NEAR. The wallet contract owner benefits from the unintended credit. This is a direct balance corruption from the caller's perspective.

### Likelihood Explanation
Any unprivileged external account can call `rlp_execute` with an attached deposit. The `has_in_flight_tx` flag is `true` whenever a prior call has produced an unresolved promise chain. A relayer who attaches NEAR to cover a user's intended transfer (as shown in `test_caller_refunds`) and races with an in-flight transaction loses their deposit. The `rlp_execute` function is `#[payable]` with no guard preventing a deposit when `has_in_flight_tx` is true, so the window is reachable in normal operation.

### Recommendation
Add a refund of `env::attached_deposit()` to the caller (`env::predecessor_account_id()`) before returning `PromiseOrValue::Value(...)` in all early-exit paths of `rlp_execute` where no promise is created. Alternatively, assert `env::attached_deposit() == NearToken::from_yoctonear(0)` (panic) in those paths so the runtime automatically refunds the deposit via the failed-receipt mechanism. The `has_in_flight_tx` guard and the `Err(e) => PromiseOrValue::Value(e.into())` arm both need this fix.

### Proof of Concept

1. Deploy the wallet contract on an ETH-implicit account.
2. Submit a valid `rlp_execute` call that creates a long-running promise chain, setting `has_in_flight_tx = true`.
3. Before the promise chain resolves, submit a second `rlp_execute` call from any external account with `attached_deposit = 5 NEAR`.
4. The guard at line 97 fires and returns `PromiseOrValue::Value(ExecuteResponse { success: false, ... })`.
5. The NEAR runtime treats this as a successful receipt; no deposit refund receipt is generated.
6. The caller's 5 NEAR is now part of the wallet contract's balance. The caller's balance is permanently reduced by 5 NEAR with no corresponding action executed.

/// A data type to keep track of the deposit given by an external caller.
/// This allows us to refund the caller's deposit if the cross-contract call fails.
#[derive(Debug, PartialEq, Eq, Clone, serde::Serialize, serde::Deserialize)]
pub struct CallerDeposit {
    pub account_id: AccountId,
    pub yocto_near: NonZeroU128,
}

impl CallerDeposit {
    pub fn new(context: &ExecutionContext) -> Option<Self> {
        // Only track for external (non-self) callers
        if context.current_account_id == context.predecessor_account_id {
            return None;
        }

        NonZeroU128::new(context.attached_deposit.as_yoctonear()).map(|yocto_near| Self {
            account_id: context.predecessor_account_id.clone(),
            yocto_near,
        })
    }
