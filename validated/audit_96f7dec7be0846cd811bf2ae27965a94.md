### Title
Deposit Refund Permanently Burnt When `promise_set_refund_to` Targets a Non-Existent Named Account - (`runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The `promise_set_refund_to` host function allows a contract to redirect deposit refunds for a sub-promise to an arbitrary account ID. No on-chain existence check is performed at call time. If the specified account does not exist when the deposit refund receipt is later executed, the refund receipt fails and the entire deposit refund amount is permanently burnt — mirroring the L1GraphTokenGateway bug where a refund address that cannot receive funds causes value to be blackholed.

---

### Finding Description

`promise_set_refund_to` (exposed via `VMLogic::promise_set_refund_to` and the wasmtime shim) accepts any syntactically valid `AccountId` and stores it in the receipt's `refund_to` field without verifying the account exists on-chain. [1](#0-0) 

The stored value is written into `ActionReceiptV2::refund_to` via `ReceiptManager::set_refund_to`: [2](#0-1) 

When the sub-promise fails, `refund_unspent_gas_and_deposits` in the runtime creates a balance-refund receipt addressed to `receipt.balance_refund_receiver()`, which returns `refund_to` when set: [3](#0-2) [4](#0-3) 

The refund receipt is a system-originated `Transfer` action. When it executes against a non-existent named account, `check_transfer_to_nonexisting_account` returns `AccountDoesNotExist` because refund receipts are explicitly excluded from implicit-account creation: [5](#0-4) 

The runtime's documented and coded behavior for a failing refund receipt is to burn the amount: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

The corrupted protocol value is the **NEAR token balance**: the deposit that should be returned to the predecessor (or the designated `refund_to` account) is instead permanently burnt and credited to `stats.balance.other_burnt_amount`. The user who attached the deposit loses it with no recourse. This is a direct balance loss, not merely a gas cost.

---

### Likelihood Explanation

Any deployed contract can call `promise_set_refund_to`. Realistic triggers include:

1. **Accidental**: A contract sets `refund_to` to an account that exists at call time but is deleted before the refund receipt is processed (race condition across blocks/shards).
2. **Misconfiguration**: A contract sets `refund_to` to a named account that has not yet been created (e.g., expecting the account to be created in a parallel promise that fails or is reordered).
3. **Malicious**: A contract deliberately sets `refund_to` to a non-existent account to burn a user's deposit refund rather than returning it.

Multisig contracts and DeFi protocols that use `promise_set_refund_to` to redirect refunds to a treasury or DAO account are the most realistic victims of cases 1 and 2.

---

### Recommendation

1. At `promise_set_refund_to` call time, validate that the `refund_to` account exists in the current state (similar to how `predecessor_account_id` is always a live account). If the account does not exist, return a `HostError` rather than silently accepting the ID.
2. Alternatively, if existence cannot be checked cheaply at call time, document clearly that the caller is responsible for ensuring the account exists when the refund is processed, and consider falling back to `predecessor_id` if the `refund_to` account does not exist at refund execution time (instead of burning).

---

### Proof of Concept

1. Contract `A` is deployed and holds user funds.
2. User calls `A` with a 10 NEAR deposit.
3. `A` creates a sub-promise to contract `B` with the 10 NEAR deposit forwarded, and calls `promise_set_refund_to(promise_idx, "nonexistent.near")`.
4. The sub-promise to `B` fails (e.g., `B` has no matching method).
5. The runtime generates a deposit-refund receipt addressed to `"nonexistent.near"` via `Receipt::new_balance_refund(receipt.balance_refund_receiver(), 10 NEAR)`.
6. The refund receipt executes: `Transfer` to `"nonexistent.near"` fails with `AccountDoesNotExist` (not an implicit account, refunds cannot create accounts).
7. The runtime enters the `predecessor_id().is_system() && result.is_err()` branch and adds 10 NEAR to `other_burnt_amount`.
8. The user's 10 NEAR is permanently burnt. Neither the user nor `A` recovers it.
