### Title
Proof-Verification Tasks for Truncated Transactions Are Still Awaited, Allowing a Malicious Proposer to Force Validators to Reject Valid Proposals - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

When a validator processes a `ProposalPart::Fin` with `executed_transaction_count < n_received_txs`, it correctly truncates `content` to the declared count. However, the parallel `verify_and_store_proof_tasks` list — which holds proof-verification futures for **all** received transactions — is never truncated. All tasks, including those for transactions that were discarded by the truncation, are awaited before the proposal is finalized. A malicious proposer can exploit this by appending a transaction with an invalid proof beyond `final_n_executed_txs`, causing every validator to fail proof verification and reject an otherwise-valid proposal.

### Finding Description

In `handle_proposal_part` (`validate_proposal.rs`), when a `ProposalPart::Transactions` batch arrives, every transaction is converted via `convert_consensus_tx_to_internal_consensus_tx`. For any transaction carrying non-empty `proof_facts`, `spawn_verify_and_store_proof` is called and the resulting `JoinHandle` is appended to `verify_and_store_proof_tasks`: [1](#0-0) 

The transactions themselves are pushed to `content` at the same time: [2](#0-1) 

When `ProposalPart::Fin` arrives, `content` is truncated to `executed_txs_count`: [3](#0-2) 

`truncate_to_executed_txs` only operates on `content`; it has no knowledge of `verify_and_store_proof_tasks`: [4](#0-3) 

Immediately after truncation, **all** tasks in `verify_and_store_proof_tasks` are drained and awaited — including those belonging to the transactions that were just discarded: [5](#0-4) 

If any awaited task returns an error, the handler immediately returns `HandledProposalPart::Failed`: [6](#0-5) 

The proof-verification logic itself will return `Err(VerifyProofError::Verification(_))` for any proof that does not pass the circuit verifier: [7](#0-6) 

The two data structures — `content` and `verify_and_store_proof_tasks` — are maintained as independent flat collections with no shared index. Truncating one does not affect the other.

### Impact Explanation

A malicious proposer appends one or more transactions with syntactically valid `proof_facts` but cryptographically invalid proofs **after** the transactions it intends to include in the block. It then sets `executed_transaction_count` in `ProposalFin` to exclude those tail transactions. Every honest validator:

1. Receives the full batch, spawns proof-verification tasks for all transactions including the invalid-proof tail.
2. Receives `Fin`, truncates `content` to the declared count (correctly excluding the tail).
3. Awaits **all** proof tasks — including the tail's — and encounters a verification failure.
4. Returns `HandledProposalPart::Failed`, which propagates as `ValidateProposalError::ProposalPartFailed`.
5. Aborts the proposal.

The result is that a valid block (whose included transactions are entirely legitimate) is rejected by all validators. Consensus for that round fails, and the network must advance to the next round with a new proposer. Repeated across rounds, this can stall block production.

### Likelihood Explanation

Any validator that wins a proposer slot can trigger this. No privileged access is required beyond being selected as proposer by the consensus protocol. The attack requires only that the proposer craft a transaction with non-empty `proof_facts` and an invalid proof payload, which is trivially constructable. The `executed_transaction_count` bound check (line 522–528) only verifies the count does not exceed the number of received transactions; it does not prevent the count from being set below the number of proof-carrying transactions. [8](#0-7) 

### Recommendation

Correlate `verify_and_store_proof_tasks` with transaction positions so that tasks for truncated transactions are discarded rather than awaited. One concrete approach: instead of a flat `Vec<VerifyAndStoreProofTask>`, maintain a `Vec<(usize, VerifyAndStoreProofTask)>` where the `usize` is the cumulative transaction index at the time the task was spawned. After `truncate_to_executed_txs` reduces `content` to `final_n_executed_txs`, drop (abort or simply discard) every task whose index is `>= final_n_executed_txs` before entering the await loop.

### Proof of Concept

```
Proposer (malicious):
  1. Build a valid block with 2 transactions: [tx_valid_1, tx_valid_2].
  2. Craft tx_invalid: an InvokeV3 with non-empty proof_facts but a garbage proof blob.
  3. Stream ProposalPart::Transactions([tx_valid_1, tx_valid_2, tx_invalid]).
  4. Stream ProposalPart::Fin { executed_transaction_count: 2, ... }.

Validator (honest):
  handle_proposal_part(Transactions):
    - Converts tx_valid_1, tx_valid_2, tx_invalid.
    - tx_invalid has proof_facts → spawn_verify_and_store_proof spawned → task T3 added to verify_and_store_proof_tasks.
    - content = [[tx_valid_1, tx_valid_2, tx_invalid]].

  handle_proposal_part(Fin { executed_transaction_count: 2 }):
    - n_received_txs = 3, executed_txs_count = 2 → guard passes (2 <= 3).
    - content = truncate_to_executed_txs(content, 2) → [[tx_valid_1, tx_valid_2]].
    - verify_and_store_proof_tasks = [T3]  ← NOT truncated.
    - Await T3: run_proof_verification(tx_invalid.proof_facts, garbage_proof) →
        verify_proof returns Err(VerifyProofError::Verification(_)).
    - await_verify_and_store_proof_task(T3) returns Err("...").
    - Returns HandledProposalPart::Failed("...").

  validate_proposal returns Err(ValidateProposalError::ProposalPartFailed(...)).
  → Valid proposal rejected; consensus round fails.
```

The root cause is the structural mismatch between `content` (truncated) and `verify_and_store_proof_tasks` (not truncated) after `ProposalFin` is processed. [9](#0-8)

pub(crate) fn truncate_to_executed_txs(
    content: &mut Vec<Vec<InternalConsensusTransaction>>,
    final_n_executed_txs: usize,
) -> Vec<Vec<InternalConsensusTransaction>> {
    let content = std::mem::take(content);
    // Truncate `content` to keep only the first `final_n_executed_txs`, preserving batch
    // structure.
    let mut executed_content: Vec<Vec<InternalConsensusTransaction>> = Vec::new();
    let mut remaining_tx_count = final_n_executed_txs;

    for batch in content {
        if remaining_tx_count < batch.len() {
            executed_content.push(batch.into_iter().take(remaining_tx_count).collect());
            break;
        } else {
            remaining_tx_count -= batch.len();
            executed_content.push(batch);
        }
    }

    executed_content
}

pub fn verify_proof(proof_facts: ProofFacts, proof: Proof) -> Result<(), VerifyProofError> {
    // Reject empty proof payloads before running the verifier.
    if proof.is_empty() {
        return Err(VerifyProofError::EmptyProof);
    }

    let proof_version_felt = proof_facts.0.first().copied().unwrap_or_default();
    let proof_version = ProofVersion::try_from(proof_version_felt)
        .map_err(|()| VerifyProofError::InvalidProofVersion { actual: proof_version_felt })?;

    let output_preimage = reconstruct_output_preimage(&proof_facts)?;
    // TODO(Avi): Avoid cloning the proof.
    let proof_bytes = proof.0.to_vec();

    match proof_version {
        // V0 proofs are no longer verifiable: the v0 circuit was removed. V0 proof facts are only
        // tolerated by the blockifier (gated per protocol version) for replaying historical blocks.
        ProofVersion::V0 => {
            return Err(VerifyProofError::InvalidProofVersion { actual: proof_version_felt });
        }
        ProofVersion::V1 => {
            let proof_output = privacy_circuit_verify_v1::PrivacyProofOutput {
                proof: proof_bytes,
                output_preimage,
            };
            privacy_circuit_verify_v1::verify_recursive_circuit(&proof_output)
                .map_err(|e| VerifyProofError::Verification(e.to_string()))?;
        }
    }

    Ok(())
