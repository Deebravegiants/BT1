### Title
`respond_verify_foreign_tx()` Does Not Verify That `response.payload_hash` Corresponds to the Resolved `request` - (File: crates/contract/src/lib.rs)

### Summary

`MpcContract::respond_verify_foreign_tx()` verifies that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash`, but never checks that `payload_hash` is actually the hash of a `ForeignTxSignPayload` derived from the `request` being resolved. A single Byzantine participant (below the signing threshold) can redirect a legitimate MPC signature — produced for one pending foreign-chain request — to resolve a completely different pending request, delivering a forged verification attestation to the bridge or dApp that submitted it.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs the following checks:

1. Caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key.
3. `request` exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What it **does not** check is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: <any valid extracted values> }))`. The `payload_hash` is entirely caller-controlled and is only verified for signature validity, not for semantic binding to the `request` key used to look up and drain the pending queue. [2](#0-1) 

The correct payload hash construction is defined in `ForeignTxSignPayload::compute_msg_hash()`: [3](#0-2) 

The `near-mpc-sdk`'s `ForeignChainSignatureVerifier::verify_signature` does perform this check on the client side: [4](#0-3) 

But the **contract itself** — the on-chain trust anchor — does not enforce it, leaving the invariant entirely to off-chain callers.

### Impact Explanation

A Byzantine participant `P` (a single node, strictly below the signing threshold) can execute the following attack:

1. Two pending requests exist: `request_A` (for foreign tx `0xAA`) and `request_B` (for foreign tx `0xBB`). `P` can trivially create `request_B` by calling `verify_foreign_transaction` with a 1 yoctonear deposit.
2. `P` participates honestly in the MPC signing protocol for `request_A`, contributing their share. The network produces a valid threshold signature `sig_A` over `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload { request_A.request, extracted_values_A }))`.
3. `P` calls `respond_verify_foreign_tx(request = request_B, response = { payload_hash: payload_hash_A, signature: sig_A })`.
4. The contract checks: `sig_A` valid over `payload_hash_A`? ✓. `request_B` in pending map? ✓. It resolves `request_B` with `{ payload_hash_A, sig_A }`.
5. The bridge or dApp that submitted `request_B` receives a response where the MPC signature attests to `0xAA`'s data, not `0xBB`'s.

Any bridge that checks only signature validity (rather than also reconstructing and comparing the expected `payload_hash`) will accept this as proof that `0xBB` was verified, enabling invalid bridge execution or double-spend conditions. [5](#0-4) 

### Likelihood Explanation

- Requires only a single attested participant — no threshold collusion.
- The attacker does not forge any cryptographic material; they redirect a legitimately produced MPC signature.
- Submitting a second pending request costs only 1 yoctonear.
- The attack is executable in a single NEAR transaction after the MPC network completes signing for `request_A`.
- Bridges that rely on the contract as the verification trust anchor (rather than re-running `ForeignChainSignatureVerifier::verify_signature` off-chain) are directly exploitable.

### Recommendation

In `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` field and reject any response whose `payload_hash` does not match. Because the `extracted_values` are not stored on-chain (only the `ForeignChainRpcRequest` is), the contract cannot reconstruct the full payload hash itself. Two mitigations are possible:

1. **Store the expected payload hash on-chain at request time** (not feasible without node cooperation at submission time).
2. **Require the responding participant to also submit the `extracted_values`**, and have the contract compute `SHA-256(borsh(ForeignTxSignPayload { request.request, extracted_values }))` and assert it equals `response.payload_hash` before accepting the response. This is the direct analog of the Velodrome fix: cache the binding parameters at deposit time and verify them on respond.
3. **Alternatively**, bind the `payload_hash` to the `request` key at the map level by including the `request` in the signed payload in a way the contract can verify — e.g., by requiring `payload_hash` to commit to `domain_id` and `ForeignChainRpcRequest` in a verifiable way.

### Proof of Concept

```
// Setup: two pending requests exist
// request_A: Bitcoin tx 0xAA (submitted by anyone)
// request_B: Bitcoin tx 0xBB (submitted by bridge)

// Step 1: Byzantine participant P participates in MPC signing for request_A.
// MPC network produces: (payload_hash_A, sig_A)
// where payload_hash_A = SHA256(borsh(ForeignTxSignPayload { Bitcoin(0xAA), [BlockHash(0x...)] }))

// Step 2: P calls respond_verify_foreign_tx with mismatched request/response:
contract.respond_verify_foreign_tx(
    request  = request_B,   // Bitcoin tx 0xBB — exists in pending map
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,  // hash of 0xAA's data
        signature:    sig_A,           // valid MPC sig over payload_hash_A
    }
)

// Contract accepts:
//   verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) == Ok(())  ✓
//   pending_verify_foreign_tx_requests.get(request_B) == Some(...)    ✓
// Resolves request_B with { payload_hash_A, sig_A }

// Bridge receives a "verified" response for 0xBB
// but the signature attests to 0xAA's extracted values.
// If bridge checks only signature validity → deceived.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
