### Title
Wallet NFT Sync Crash via `KeyError` in `prepend_value` on Unknown Metadata Key in `-24` Condition — (`chia/wallet/nft_wallet/nft_puzzle_utils.py`)

### Summary

`prepend_value` unconditionally indexes `metadata[key]` without checking whether `key` exists in the dict. An NFT owner can craft a spend bundle whose p2 puzzle solution emits a `-24` (metadata-update) condition referencing a key absent from the NFT's on-chain metadata. The on-chain CLVM metadata updater silently ignores such updates (confirmed by the lifecycle test), so the spend is accepted on-chain. When the victim wallet later processes the parent coin spend through `handle_nft` → `get_metadata_and_phs` → `update_metadata` → `prepend_value`, an unhandled `KeyError` is raised, permanently preventing the wallet from recording the post-spend NFT state.

---

### Finding Description

**Root cause — `prepend_value` (line 161):**

```python
def prepend_value(key: bytes, value: Program, metadata: dict[bytes, Any]) -> None:
    if value != Program.to(0):
        if metadata[key] == b"":          # KeyError if key absent
            metadata[key] = [value.as_python()]
        else:
            metadata[key].insert(0, value.as_python())
``` [1](#0-0) 

`metadata[key]` raises `KeyError` when `key` is not present in the dict produced by `nft_program_to_metadata`. There is no `key in metadata` guard and no `try/except`.

**Call chain — no exception handling at any level:**

`update_metadata` calls `prepend_value` directly: [2](#0-1) 

`get_metadata_and_phs` calls `update_metadata` for every `-24` condition without any guard: [3](#0-2) 

`handle_nft` in `WalletStateManager` calls `get_metadata_and_phs` with no `try/except`: [4](#0-3) 

`puzzle_solution_received` in `NFTWallet` also calls `get_metadata_and_phs` with no `try/except`: [5](#0-4) 

**On-chain CLVM accepts the spend:** The lifecycle test proves that the default on-chain `NFT_METADATA_UPDATER` CLVM puzzle **silently ignores** updates to any key not in its internal whitelist (`u`, `lu`, `mu`). The spend is accepted on-chain with no metadata change; the CLVM does not raise: [6](#0-5) 

This means a spend with a `-24` condition targeting `b"custom_key"` (absent from the NFT metadata dict) is **valid on-chain** but **crashes the wallet** during sync.

---

### Impact Explanation

When the victim wallet processes the confirmed parent coin spend:
1. `get_metadata_and_phs` runs the p2 puzzle locally and sees the attacker-injected `-24` condition.
2. `update_metadata` → `prepend_value` raises `KeyError` on `metadata[b"custom_key"]`.
3. The unhandled exception propagates through `handle_nft` / `puzzle_solution_received`.
4. The wallet permanently fails to record the post-spend NFT coin state; the NFT is absent from `nft_store` and unspendable from the wallet.

In an offer scenario: the victim pays XCH, the spend settles on-chain, but the victim's wallet never tracks the received NFT. This constitutes corruption of wallet NFT sync state with direct security impact (loss of asset control from the wallet's perspective).

---

### Likelihood Explanation

The attacker only needs to own any NFT (self-minted). They craft a spend with a `-24` condition using a key absent from the NFT's metadata dict, sign it with their own key, and submit it. No leaked keys, no admin access, and no broken cryptography are required. The attack is trivially reproducible with the existing simulation framework.

---

### Recommendation

Add a key-existence guard in `prepend_value`:

```python
def prepend_value(key: bytes, value: Program, metadata: dict[bytes, Any]) -> None:
    if value != Program.to(0):
        if key not in metadata:
            return          # silently ignore unknown keys, matching on-chain CLVM behavior
        if metadata[key] == b"":
            metadata[key] = [value.as_python()]
        else:
            metadata[key].insert(0, value.as_python())
```

Additionally, wrap `get_metadata_and_phs` calls in `handle_nft` and `puzzle_solution_received` with `try/except` to prevent a single malformed spend from permanently breaking wallet sync.

---

### Proof of Concept

1. Mint an NFT with metadata `{b"u": [...], b"h": <hash>}` (standard keys only).
2. Craft a spend bundle for that NFT coin whose p2 puzzle solution includes:
   ```
   [-24, NFT_METADATA_UPDATER, (b"custom_key", "http://evil")]
   [51, <recipient_puzhash>, 1, [<recipient_puzhash>]]
   ```
3. Sign and submit the spend bundle (attacker owns the NFT).
4. Confirm the spend is accepted on-chain (CLVM silently ignores `b"custom_key"`).
5. Observe that the recipient wallet raises `KeyError: b'custom_key'` inside `prepend_value` during `handle_nft` / `puzzle_solution_received`.
6. Assert the NFT is absent from `nft_store` after the block is processed.
