### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function performs cryptographic output verification **only** for the `CKDAppPublicKey::AppPublicKeyPV` variant. When a user submits a CKD request using the `CKDAppPublicKey::AppPublicKey` variant, the contract accepts any response from any single attested participant with zero verification. A single Byzantine participant below the signing threshold can forge the CKD output and deliver an attacker-controlled derived key to the user. This is a direct analog to the missing-documentation finding: the empty match arm carries no comment explaining the security implication, making the asymmetry invisible to reviewers and future maintainers.

---

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 675–682), the contract branches on the `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKeyPV` arm calls `ckd_output_check`, which verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, binding the response to the network's BLS12-381 master public key and the user's app identity. [2](#0-1) 

The `AppPublicKey` arm is an empty block. No equivalent check exists. The contract immediately proceeds to `resolve_yields_for`, which resumes the user's yield promise with the unverified bytes. [3](#0-2) 

By contrast, the `respond` function for threshold signatures always verifies the ECDSA or EdDSA signature against the derived public key before resolving the yield: [4](#0-3) 

The `request_app_private_key` entry point accepts both variants and documents that `AppPublicKeyPV` identity points are allowed for "use cases where the derived key is intentionally public," but provides no documentation on the security consequence of using `AppPublicKey` — namely that the output is entirely unverified: [5](#0-4) 

The `AppPublicKey` variant is a `Bls12381G1PublicKey` (a single G1 point). Because the pairing check requires a G2 component (`pk2`) to bind the response to the network key, no equivalent on-chain verification is possible for this variant without a protocol change. The empty match arm is therefore not a placeholder — it is the permanent code path, and it is entirely undocumented. [6](#0-5) 

---

### Impact Explanation

**Critical — Confidential key derivation output without the required participant authorization.**

A single Byzantine attested participant can call `respond_ckd` with an arbitrary `CKDResponse` (any `big_c` and `big_y` G1 points of their choosing) for any pending `AppPublicKey` CKD request. The contract will accept the forged response and deliver it to the waiting user via `promise_yield_resume`. The user receives an attacker-controlled derived key believing it was produced by the threshold MPC network. Depending on how the user employs the derived key (encryption, signing, authentication), the attacker gains the ability to decrypt user data, forge signatures, or impersonate the user on foreign chains — all without the threshold of honest participants ever participating in the derivation.

---

### Likelihood Explanation

**Medium.** The attacker must be a single attested MPC participant (i.e., hold a valid TEE attestation accepted by the contract). This is strictly below the signing threshold. The `AppPublicKey` variant is a supported, documented code path available to any user of the `request_app_private_key` API. A Byzantine node monitoring the NEAR chain can observe pending `AppPublicKey` CKD requests and race to call `respond_ckd` with a forged response before the honest nodes do. Because `respond_ckd` resolves the yield on first call (the queue is drained), the first valid attested participant to respond wins.

---

### Recommendation

1. **Require `AppPublicKeyPV` for all CKD requests** that expect a security guarantee on the output. Deprecate or remove `AppPublicKey` from the `request_app_private_key` API, or gate it behind an explicit "unverified / best-effort" flag with a clear on-chain warning.
2. **If `AppPublicKey` must remain**, add a prominent NatSpec/Rust doc comment on the empty match arm in `respond_ckd` explicitly stating: *"No output verification is possible for this variant; the response is accepted on trust from the first responding attested participant."* This at minimum surfaces the security assumption to future reviewers.
3. **Consider a protocol extension** where nodes include a zero-knowledge proof of correct derivation even for the `AppPublicKey` variant, so the contract can verify correctness without requiring the G2 component from the user.

---

### Proof of Concept

**Setup:** Contract is in `Running` state. Attacker controls one attested MPC participant account (`attacker.near`). Victim calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(legitimate_g1_pk)`.

**Attack steps:**

1. Attacker observes the pending `CKDRequest` on-chain (the `CKDRequest` key is deterministic from `app_public_key`, `domain_id`, `predecessor_id`, and `derivation_path`).
2. Attacker constructs a forged `CKDResponse`:
   ```
   big_y = G1::generator()   // arbitrary
   big_c = G1::generator()   // arbitrary, attacker-chosen
   ```
3. Attacker calls `respond_ckd(request, forged_response)` from `attacker.near`.
4. Contract executes:
   - `assert_caller_is_signer()` → passes (attacker is a signer)
   - `is_running_or_resharing()` → passes
   - `accept_requests` → passes
   - `assert_caller_is_attested_participant_and_protocol_active()` → passes
   - `match &request.app_public_key { AppPublicKey(_) => {} ... }` → **empty, no check**
   - `resolve_yields_for(...)` → resumes victim's yield with forged bytes
5. Victim's callback receives the attacker-controlled `CKDResponse` and uses it as their derived secret key. [7](#0-6)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(app_pk1, g2) = e(g1, app_pk2)`.
///
/// Point validation is fully delegated to the host: the decompression
/// functions abort execution on malformed or off-curve encodings, and
/// `bls12381_pairing_check` returns `false` when a point is outside its
/// prime-order subgroup.
pub(crate) fn app_public_key_check(app_public_key: &dtos::CKDAppPublicKeyPV) -> bool {
    let pk1 = env::bls12381_p1_decompress(&app_public_key.pk1);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);

    let pairing_input = [
        pk1.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        G1_GENERATOR_UNCOMPRESSED.as_slice(),
        pk2.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,

    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);

        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
