### Title
Unauthenticated P2P peer can poison `block_hash_to_number` table by supplying an arbitrary `block_hash` in `SignedBlockHeader`, causing RPC hash-based queries to return wrong block data — (`crates/apollo_p2p_sync/src/client/header.rs`)

### Summary
The P2P sync header client accepts the `block_hash` field of a `SignedBlockHeader` from an untrusted peer without any cryptographic verification. It only checks that the `block_number` field matches the expected cursor. The unverified hash is written directly into the `block_hash_to_number` reverse-lookup table. Any RPC query that resolves a block by hash (e.g., `starknet_getBlockWithTxHashes`, `starknet_getStateUpdate`) will then return the wrong block's data.

### Finding Description
`HeaderStreamBuilder::parse_data_for_block` validates only that `block_number` matches the sync cursor and that exactly one signature is present. It does not:
- Compute the expected block hash from the header fields and compare it to `block_header.block_hash`
- Verify the `BlockSignature` against a known sequencer public key

The accepted `SignedBlockHeader` is passed to `write_to_storage`, which calls `append_header`. Inside `append_header`, `update_hash_mapping` inserts `(block_header.block_hash → block_number)` into the `block_hash_to_number` LMDB table. A malicious peer can set `block_hash` to any arbitrary value — including the known hash of a different block — and the node will store that mapping without complaint.

### Impact Explanation
After poisoning, any RPC call using `BlockId::HashOrNumber(BlockHashOrNumber::Hash(poisoned_hash))` resolves to the wrong block number via `get_block_number_by_hash`, and the node returns that block's state diff, transactions, receipts, and class data as authoritative responses. This is a **High** impact: RPC returns authoritative-looking wrong values for hash-based queries.

### Likelihood Explanation
Any node that syncs headers over P2P (the default for non-central-sync deployments) is exposed. The attacker only needs to be a peer that the victim connects to — no special privileges required. The attack requires no cryptographic capability; the attacker simply sets the `block_hash` field in the protobuf to an arbitrary value.

### Recommendation
In `parse_data_for_block`, after receiving a `SignedBlockHeader`:
1. Recompute the expected block hash from the header fields using `calculate_block_hash` and reject the header if it does not match the received `block_hash`.
2. Verify the `BlockSignature` against the known sequencer public key using `verify_block_signature` before accepting the header.
3. Implement the existing TODO: verify `parent_hash` matches the previously stored block's hash.

### Proof of Concept
```
1. Node starts syncing headers from block 0 via P2P.
2. Attacker (malicious peer) responds to the HeaderQuery for block 1 with:
     SignedBlockHeader {
       block_header: BlockHeader {
         block_hash: <known hash of block 0>,   // ← poisoned
         block_header_without_hash: BlockHeaderWithoutHash {
           block_number: 1,                     // ← passes the only check
           ...
         },
         ...
       },
       signatures: [<any single signature>],
     }
3. parse_data_for_block passes: block_number(1) == block_header_without_hash.block_number(1). ✓
4. write_to_storage → append_header → update_hash_mapping stores:
     block_hash_to_number[hash_of_block_0] = 1
5. RPC call: starknet_getStateUpdate(block_id = {block_hash: hash_of_block_0})
   → get_block_number_by_hash(hash_of_block_0) returns 1
   → node returns block 1's state diff as if it were block 0's.
6. Assert: returned block_number is 1, not 0. Corruption confirmed.
```
