Based on my analysis of the code, this is a real vulnerability. Let me explain the findings precisely.

**The call path is concrete and reachable:**

1. A syncing node's `P2pSyncClient` issues SQMR queries for transactions
2. An unauthenticated peer responds with `FullTransaction` messages containing arbitrary `transaction_output` values
3. `parse_data_for_block` accepts them with zero validation and pushes them directly to `block_body.transaction_outputs`
4. `write_to_storage` calls `append_body`, which persists them via `write_transactions`
5. `starknet_getTransactionReceipt` reads `get_transaction_output` directly from that storage and returns it

The TODO comment at line 88 explicitly acknowledges missing validation: `// TODO(eitan): Validate transaction hash from untrusted sources`.

**However, the impact claim in the question is partially wrong:**

The question claims this corrupts `receipt_commitment` and `event_commitment` used by `calculate_block_commitments`. This is **incorrect**. Those commitment values are stored in the block header (synced separately), and the syncing node does not recompute them from stored transaction outputs. The `calculate_block_commitments` path is only used during block **building** by the batcher/sequencer, not during P2P sync.

The **actual** impact is narrower but still real: `starknet_getTransactionReceipt` (and `starknet_getBlockWithReceipts`) on a syncing full node will return fabricated events, wrong `actual_fee`, and wrong execution resources — an authoritative-looking wrong value per the scope rules.

### Title
Unauthenticated P2P Peer Can Inject Fabricated `TransactionOutput` Data Served by `starknet_getTransactionReceipt` — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary
`TransactionStreamFactory::parse_data_for_block` accepts `transaction_output` from any SQMR peer without validating it against the block header's pre-committed `event_commitment` or `receipt_commitment`. The fabricated outputs are persisted to storage and served verbatim by the RPC layer.

### Finding Description
In `parse_data_for_block`, the only check performed is that the count of received transactions matches `header.n_transactions`. The `transaction_output` field of each `FullTransaction` — which contains `events`, `actual_fee`, `execution_resources`, `messages_sent`, and `execution_status` — is pushed directly to `block_body.transaction_outputs` with no integrity check. [1](#0-0) 

`write_to_storage` then calls `append_body`, which stores the fabricated outputs via `write_transactions` and indexes events from them. [2](#0-1) [3](#0-2) 

The RPC `get_transaction_receipt` reads `get_transaction_output` directly from this storage and returns it to callers. [4](#0-3) 

The block header's `event_commitment` and `receipt_commitment` are stored separately from the synced header and are **not** recomputed from the stored transaction outputs, so those header fields are unaffected. The corruption is confined to the body storage and RPC responses. [5](#0-4) 

### Impact Explanation
Any client querying `starknet_getTransactionReceipt` or `starknet_getBlockWithReceipts` on a compromised syncing node receives fabricated events, wrong `actual_fee`, wrong execution resources, and potentially a wrong execution status. This fits the "High. RPC ... returns an authoritative-looking wrong value" criterion. The sequencer's own block building is unaffected since it uses its own execution results, not P2P-synced outputs.

### Likelihood Explanation
The attacker only needs to be a peer that the syncing node connects to — no authentication is required. The node actively queries peers for transaction data, so any peer that responds to the SQMR query can inject data. The TODO comment confirms the developers are aware validation is absent. [6](#0-5) 

### Recommendation
After collecting all `FullTransaction` responses for a block, recompute `event_commitment` and `receipt_commitment` from the received outputs using `calculate_block_commitments` and compare against the values already stored in the block header. Reject the entire block body and penalize the peer if there is a mismatch. [7](#0-6) 

### Proof of Concept
1. Start a syncing node with `P2pSyncClient`.
2. Act as a peer; respond to the node's transaction SQMR query for block N with a `FullTransaction` where `transaction_output = TransactionOutput::Invoke(InvokeTransactionOutput { actual_fee: Fee(u128::MAX), events: vec![fabricated_event], ... })`.
3. Send the correct `transaction_hash` and `transaction` body so the count check passes.
4. After the node stores the block body, call `starknet_getTransactionReceipt` with the transaction hash.
5. Observe that the response contains the fabricated `actual_fee` and events rather than the true on-chain execution result.
