Audit Report

## Title
Unchecked Multiplication Overflow in `_checkedMul` Enables Zero-Cost Share Inflation and LP Fund Drain — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`_checkedMul` performs a bare `a * b` with no overflow guard, and is called inside an `unchecked` block that wraps the entire `addLiquidity` function body. An attacker can supply a crafted `sharesToAdd` value that causes `token0BalanceScaled * sharesToAdd` to wrap to zero modulo 2^256, resulting in `amount0Scaled = 0`. The pool mints an arbitrarily large share count for zero tokens, diluting all existing LPs to near-zero and enabling full bin drainage via sub-overflow-threshold removal chunks.

## Finding Description

**Root cause — `_checkedMul` is a bare multiplication:**

`_checkedMul` at lines 261–263 performs `return a * b` with no overflow check: [1](#0-0) 

The entire `addLiquidity` body is wrapped in `unchecked { ... }` starting at line 51, disabling Solidity ≥0.8's default overflow revert: [2](#0-1) 

**Vulnerable multiplication (lines 109–110):**

When `binTotalSharesVal > 0`, the token cost is computed as: [3](#0-2) 

`binState.token0BalanceScaled` is `uint104` (max ≈ 2^104): [4](#0-3) 

`sharesToAdd` comes from `LiquidityDelta.shares[]`, typed `uint256[]` with no upper-bound validation: [5](#0-4) 

**Only guard is a minimum floor, not a ceiling:** [6](#0-5) 

**Overflow condition:** For `T = binState.token0BalanceScaled` and attacker-chosen `Y = sharesToAdd`, if `T × Y ≡ 0 (mod 2^256)`, the product wraps to 0. For any `T = 2^k * m` (odd `m`), the attacker picks `Y = 2^(256−k)` (a valid `uint256`). Then `ceilDiv(0, binTotalSharesVal) = 0`, so `amount0Scaled = 0` and `amount1Scaled = 0`.

**Callback is skipped when both amounts are zero:** [7](#0-6) 

The attacker pays zero tokens. Meanwhile, `binTotalShares[binIdx]` is incremented by the full `Y` (inside `unchecked`, no overflow on the addition since `Y < 2^256`): [8](#0-7) 

**Existing LPs are wiped in `removeLiquidity`:** The same `_checkedMul` pattern appears at lines 205–206: [9](#0-8) 

With `binTotalSharesVal ≈ Y` (enormous), a legitimate LP holding `S` original shares receives `T × S / (S + Y) ≈ 0` — complete loss of principal.

**Attacker drains the bin:** The attacker removes their `Y` shares in chunks `R` small enough that `T × R < 2^256` (i.e., `R < 2^(256−104) = 2^152`). Each chunk yields `T × R / Y` tokens. Summing over `Y / R` chunks recovers approximately `T` tokens — the full original bin balance.

**No guard in `MetricOmmPool.addLiquidity`:** The pool entry point at lines 182–196 performs no validation on share magnitudes before delegating to `LiquidityLib`: [10](#0-9) 

## Impact Explanation

- **Direct loss of LP principal**: existing LPs receive zero tokens on `removeLiquidity` after the attack — their proportional claim is diluted to near-zero by the inflated `binTotalShares`.
- **Pool insolvency**: `token0BalanceScaled` / `token1BalanceScaled` no longer cover LP claims after the attacker drains via chunked removal.
- **Attacker profit**: the attacker recovers the full bin token balance at zero token cost.
- All bins with `binTotalSharesVal > 0` are simultaneously vulnerable.

This meets the Critical threshold: direct, complete loss of LP principal and pool insolvency.

## Likelihood Explanation

- **Unprivileged trigger**: any EOA or contract can call `addLiquidity` directly on `MetricOmmPool` with an arbitrary `sharesToAdd` value.
- **No front-running required**: the attacker reads `binState.token0BalanceScaled` on-chain to compute the exact `Y` needed, then executes atomically.
- **No existing ceiling guard**: the only share validation is a minimum floor (`minimalMintableLiquidity`), not a maximum cap.
- **Repeatable**: the attack can be applied to every bin independently.

## Recommendation

1. **Remove the `unchecked` wrapper from `addLiquidity` and `removeLiquidity`**, or scope it only to arithmetic that is provably safe (e.g., loop counter increments).
2. **Replace `_checkedMul` with `Math.mulDiv`** for the share-to-token conversion in both `addLiquidity` (lines 109–110) and `removeLiquidity` (lines 205–206):
   ```solidity
   amount0Scaled = Math.mulDiv(binState.token0BalanceScaled, sharesToAdd, binTotalSharesVal, Math.Rounding.Ceil);
   amount1Scaled = Math.mulDiv(binState.token1BalanceScaled, sharesToAdd, binTotalSharesVal, Math.Rounding.Ceil);
   ```
3. **Rename or remove `_checkedMul`** — the name implies safety that does not exist.
4. **Apply the same fix to the `binTotalSharesVal == 0` branch** (lines 87–89), which also calls `_checkedMul` inside `unchecked`.

## Proof of Concept

```
Setup:
  Pool bin B, token0 (18 decimals), TOKEN_0_SCALE_MULTIPLIER = 1
  Legitimate LP adds shares → binTotalShares[B] = 10_000
  token0BalanceScaled = T = 1_000_000 (= 2^6 * 5^6)

Attack:
  gcd(T, 2^256) = 2^6 = 64
  Choose Y = 2^256 / 64 = 2^250
  Verify: T * Y = 2^6 * 5^6 * 2^250 = 5^6 * 2^256 ≡ 0 (mod 2^256) ✓

Step 1 — Attacker calls addLiquidity(binIdx=B, sharesToAdd=Y=2^250):
  _checkedMul(1_000_000, 2^250) = 0 (wraps mod 2^256)
  amount0Scaled = ceilDiv(0, 10_000) = 0
  → callback skipped, attacker pays 0 tokens
  → binTotalShares[B] = 10_000 + 2^250 ≈ 2^250
  → positionBinShares[attacker] = 2^250

Step 2 — Legitimate LP removes 10_000 shares:
  amount0Scaled = 1_000_000 * 10_000 / (10_000 + 2^250) ≈ 0
  → LP receives 0 tokens (full loss)

Step 3 — Attacker removes Y shares in chunks R = 2^149:
  T * R = 1_000_000 * 2^149 = 2^6 * 5^6 * 2^149 = 5^6 * 2^155 < 2^256 (no overflow)
  Each chunk: 5^6 * 2^155 / 2^250 = 5^6 / 2^95 ≈ 0 (integer division)
  → Adjust R upward until each chunk yields nonzero tokens
  → Over Y/R chunks, total recovered ≈ T = 1_000_000 scaled units (full bin drain)

Foundry test skeleton:
  1. Deploy pool, mint token0 to LP and attacker.
  2. LP calls addLiquidity with 10_000 shares; verify binTotalShares = 10_000.
  3. Attacker calls addLiquidity with sharesToAdd = 2^250; assert amount0Added == 0.
  4. Assert binTotalShares[B] ≈ 2^250.
  5. LP calls removeLiquidity with 10_000 shares; assert amount0Removed == 0.
  6. Attacker removes in chunks; assert cumulative amount0Removed ≈ original T.
```
