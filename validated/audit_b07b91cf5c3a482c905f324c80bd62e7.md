### Title
Single Byzantine Participant Can Inject Malicious CKD Response for Legacy `AppPublicKey` Requests - (File: `crates/contract/src/lib.rs`)

### Summary
The `respond_ckd` endpoint accepts a Confidential Key Derivation (CKD) response from **any single attested participant** for legacy `AppPublicKey` requests, with zero cryptographic output verification. A Byzantine participant strictly below the signing threshold can race to inject an arbitrary response before honest nodes, delivering a corrupted derived key to the user — without the threshold of participants ever authorizing the output.

### Finding Description
In `respond_ckd` (`crates/contract/src/lib.rs:675-682`), the response verification is gated on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For the `AppPublicKeyPV` (publicly verifiable) variant, `ckd_output_check` is called and the response is rejected if it does not cryptographically match the user's public key and the MPC root key. For the legacy `AppPublicKey` variant, the arm is a no-op — **any `CKDResponse` with arbitrary `big_y` and `big_c` BLS12-381 G1 points is unconditionally accepted**.

Contrast this with the `respond` function for threshold signatures (`crates/contract/src/lib.rs:586-644`), where the signature is verified against the derived public key before `resolve_yields_for` is called. That cryptographic check is what makes a single-node forgery impossible for ECDSA/EdDSA: producing a valid signature requires threshold key-share cooperation. No equivalent enforcement exists for `AppPublicKey` CKD responses.

Once `resolve_yields_for` is called (`crates/contract/src/lib.rs:684-688`), the yield is drained and the request is removed from `pending_ckd_requests`. There is no retry path for the user — the malicious response is final.
