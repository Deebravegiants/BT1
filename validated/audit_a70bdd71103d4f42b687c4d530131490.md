Audit Report

## Title
ETH Donation via Open `receive()` Inflates TVL and Triggers Excess Protocol Fee Minting — (`contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary
`LRTDepositPool` exposes an unrestricted `receive() external payable {}` function, and `getETHDistributionData()` measures TVL using the raw `address(this).balance`. Any unprivileged actor can donate ETH directly to the pool, inflating the reported TVL. When `LRTOracle.updateRSETHPrice()` is subsequently called, the inflated TVL is treated as earned yield, causing excess protocol fee rsETH to be minted to the treasury and permanently diluting all existing rsETH holders.

## Finding Description
`LRTDepositPool` accepts arbitrary ETH via its bare `receive()` at line 58:

```solidity
receive() external payable { }   // LRTDepositPool.sol:58
```

`getETHDistributionData()` reads the raw contract balance at line 480:

```solidity
ethLyingInDepositPool = address(this).balance;
```

This value flows through `getTotalAssetDeposits(ETH_TOKEN)` → `_getTotalEthInProtocol()` → `_updateRsETHPrice()`. In `_updateRsETHPrice()` (lines 244–246), any increase in `totalETHInProtocol` over `previousTVL` is treated as earned yield:

```solidity
uint256 rewardAmount = totalETHInProtocol - previousTVL;
protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
```

The protocol then mints `protocolFeeInETH / newRsETHPrice` rsETH to the treasury (lines 301–307). Because the donated ETH was never earned yield, this fee mint is entirely spurious. The `_checkAndUpdateDailyFeeMintLimit` guard (lines 197–209) caps the per-day fee mint but does not prevent the attack — an attacker can repeat the donation across multiple days or stay under the daily cap. The `pricePercentageLimit` guard only blocks non-manager callers when the price increase is large; small donations bypass it entirely.

## Impact Explanation
**High — Theft of unclaimed yield.** The donated ETH is real ETH that enters the protocol and should accrue entirely to rsETH holders as yield. Instead, `protocolFeeInBPS / 10_000` of it is diverted to the treasury as spurious protocol fee, permanently diluting every rsETH holder's share of the underlying ETH. The dilution is proportional to `protocolFeeInBPS * donationAmount / totalETHInProtocol` and is irreversible once `updateRSETHPrice()` is called. The attacker sacrifices the donated ETH but causes a permanent, measurable loss to all rsETH holders.

## Likelihood Explanation
The attack requires only a direct ETH transfer to `LRTDepositPool`, which is permissionless via the open `receive()` function. No privileged access, no front-running, and no complex setup is needed. The attacker can size the donation to stay under both the `pricePercentageLimit` threshold and the `maxFeeMintAmountPerDay` cap, making the attack repeatable across multiple days. The cost to the attacker is the donated ETH, but the damage is permanent and scales with the donation.

## Recommendation
1. Replace raw `address(this).balance` reads with an internal accounting variable (e.g., `totalETHDeposited`) that is incremented only through authorized deposit paths (`depositETH`, `receiveFromRewardReceiver`, `receiveFromNodeDelegator`, `receiveFromLRTConverter`) and decremented on withdrawals.
2. Similarly, replace `IERC20(asset).balanceOf(address(this))` reads with internal accounting variables for LST balances.
3. Alternatively, restrict the bare `receive()` function to revert unconditionally, forcing all ETH inflows through the named, access-controlled receive functions.

## Proof of Concept
```
State before attack:
  rsethSupply      = 1000e18 rsETH
  rsETHPrice       = 1.05e18
  previousTVL      = 1050 ETH
  protocolFeeInBPS = 1000 (10%)
  maxFeeMintAmountPerDay = large (not a binding constraint)

Step 1: Attacker sends 10 ETH to LRTDepositPool via receive().
  address(this).balance increases by 10 ETH.

Step 2: Anyone calls LRTOracle.updateRSETHPrice().
  totalETHInProtocol = 1060 ETH  (inflated by donation)
  rewardAmount       = 1060 - 1050 = 10 ETH
  protocolFeeInETH   = 10 * 10% = 1 ETH
  newRsETHPrice      = (1060 - 1) / 1000 = 1.059e18
  rsethAmountToMintAsProtocolFee = 1e18 / 1.059e18 ≈ 0.944 rsETH minted to treasury

Result:
  - Treasury receives ~0.944 rsETH it did not earn.
  - All existing rsETH holders are permanently diluted.
  - Attack is repeatable daily, staying under maxFeeMintAmountPerDay.

Foundry test outline:
  1. Fork mainnet or deploy protocol locally.
  2. Record rsETHPrice and treasury rsETH balance.
  3. vm.deal(attacker, 10 ether); vm.prank(attacker); address(depositPool).call{value: 10 ether}("");
  4. Call lrtOracle.updateRSETHPrice().
  5. Assert treasury rsETH balance increased by ~0.944 rsETH.
  6. Assert per-holder ETH claim is lower than it would have been without the fee.
```
