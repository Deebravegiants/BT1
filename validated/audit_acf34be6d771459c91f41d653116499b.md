### Title
Caller Deposit Permanently Stuck in `WalletContract` on Intermediate Callback Failure - (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary
`WalletContract.address_check_callback()` and `WalletContract.nep_141_storage_balance_callback()` both return early on permanent failure paths without refunding the external caller's attached deposit (`CallerDeposit`). The deposit is absorbed into the wallet contract's balance with no recovery mechanism, mirroring the StargateLbpHelper stuck-token pattern.

### Finding Description

`WalletContract.rlp_execute()` is `#[payable]` and accepts an attached NEAR deposit from an external caller. The deposit is tracked in a `CallerDeposit` struct and is supposed to be refunded if the downstream cross-contract call fails. [1](#0-0) 

The correct refund path exists in `rlp_execute_callback`: [2](#0-1) 

However, two intermediate callbacks that sit **before** `rlp_execute_callback` in the promise chain have early-return failure paths that drop `caller_deposit` without issuing any refund:

**`address_check_callback` — registrar call fails or returns malformed JSON:** [3](#0-2) 

Both `PromiseResult::Failed` and the JSON-parse-error arm return `PromiseOrValue::Value(...)` immediately, discarding `caller_deposit` entirely.

**`nep_141_storage_balance_callback` — `storage_balance_of` call fails or returns malformed JSON:** [4](#0-3) 

Same pattern: both failure arms return early without touching `caller_deposit`.

The promise chains that lead to these callbacks are set up in `inner_rlp_execute`: [5](#0-4) 

When either intermediate callback fires with a failure, the caller's deposit is already held in the wallet contract's balance. The wallet contract exposes no admin withdrawal function; the only egress is through `rlp_execute` itself, which requires a valid signed Ethereum transaction from the wallet owner — not the external caller whose deposit is stuck.

### Impact Explanation

**Impact: High.** An external caller who attaches NEAR tokens to `rlp_execute` for an EOA base-token transfer (address-check path) or an ERC-20 transfer (NEP-141 storage-balance path) permanently loses those tokens if the registrar or token contract fails. The wallet contract's balance increases by the stuck amount, and no function exists to return it to the caller. The corrupted value is the caller's NEAR balance (a concrete on-chain balance entry).

### Likelihood Explanation

**Likelihood: Medium.** The address registrar and NEP-141 token contracts are external dependencies. Either can fail due to: gas exhaustion on the registrar/token side, a contract panic, a temporary outage, or a malicious token contract that deliberately panics `storage_balance_of`. Any of these conditions is reachable by an unprivileged external user simply by calling `rlp_execute` with an attached deposit at an inopportune moment.

### Recommendation

In both `address_check_callback` and `nep_141_storage_balance_callback`, before each early-return failure path, issue a refund transfer to the caller if `caller_deposit` is `Some`, mirroring the pattern already used in `rlp_execute_callback`:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund_promise = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(
        refund_promise,
        NearToken::from_yoctonear(yocto_near.into()),
    );
}
```

This must be applied to all four early-return arms (two in each callback).

### Proof of Concept

1. External caller `alice` calls `rlp_execute` on a deployed `WalletContract` with `attached_deposit = 5 NEAR`, submitting an ERC-20 transfer Ethereum transaction (triggers the NEP-141 path).
2. `inner_rlp_execute` creates `CallerDeposit { account_id: alice, yocto_near: 5e24 }` and schedules `storage_balance_of` on the token contract followed by `nep_141_storage_balance_callback`.
3. The token contract's `storage_balance_of` panics (e.g., out of gas, or a malicious token contract).
4. `nep_141_storage_balance_callback` fires with `PromiseResult::Failed` and returns `PromiseOrValue::Value(ExecuteResponse { success: false, ... })` at line 204–209, discarding `caller_deposit`.
5. Alice's 5 NEAR is now part of the wallet contract's balance. No function in the wallet contract allows Alice to recover it. The deposit is permanently lost. [6](#0-5)

/// A data type to keep track of the deposit given by an external caller.
/// This allows us to refund the caller's deposit if the cross-contract call fails.
#[derive(Debug, PartialEq, Eq, Clone, serde::Serialize, serde::Deserialize)]
pub struct CallerDeposit {
    pub account_id: AccountId,
    pub yocto_near: NonZeroU128,
}

impl CallerDeposit {
    pub fn new(context: &ExecutionContext) -> Option<Self> {
        // Only track for external (non-self) callers
        if context.current_account_id == context.predecessor_account_id {
            return None;
        }

        NonZeroU128::new(context.attached_deposit.as_yoctonear()).map(|yocto_near| Self {
            account_id: context.predecessor_account_id.clone(),
            yocto_near,
        })
    }
