### Title
Zero-agETH Mint on Truncation Permanently Freezes Deposited Tokens — (`contracts/agETH/AGETHPoolV3.sol`)

---

### Summary

`deposit(address,uint256,string)` transfers the caller's tokens into the contract **before** computing `agETHAmount`. When integer division in `viewSwapAgETHAmountAndFee` truncates to zero, the function proceeds to mint 0 agETH and emits the event — with no revert, no refund, and no recovery path for the depositor.

---

### Finding Description

The token-deposit flow is:

1. **Transfer tokens in** — line 145
2. **Compute agETH amount** — line 147 → `viewSwapAgETHAmountAndFee(amount, token)`
3. **Accumulate fee** — line 149
4. **Mint agETH** — line 151 [1](#0-0) 

Inside `viewSwapAgETHAmountAndFee(uint256,address)` the final amount is:

```
agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate;
``` [2](#0-1) 

This is plain integer division with **no floor-check**. Whenever

```
amountAfterFee * tokenToETHRate  <  agETHToETHrate
```

the result is 0. The function returns `(0, fee)`, the caller's tokens are already in the contract, `feeEarnedInToken[token]` is incremented by `fee` (which may itself be 0 when `feeBps == 0`), and `agETH.mint(msg.sender, 0)` is called. There is **no guard** of the form `if (agETHAmount == 0) revert` anywhere in the deposit path. [3](#0-2) 

The only privileged functions that touch the token balance are `withdrawFees` (moves `feeEarnedInToken[token]`) and `moveAssetsForBridging(token)` (moves `balance − feeEarnedInToken[token]`). Neither refunds the original depositor. [4](#0-3) 

---

### Impact Explanation

Deposited tokens are permanently stranded in the contract. The depositor receives 0 agETH and has no on-chain mechanism to recover their tokens. The magnitude of the loss scales with the ratio `agETHToETHrate / tokenToETHRate`:

| Token decimals | Typical `tokenToETHRate` | Max `amount` that truncates to 0 |
|---|---|---|
| 18 (e.g. wstETH ≈ 1 ETH) | ~1.05 × 10¹⁸ | 1 wei — dust only |
| 6 (e.g. USDC at $3 000/ETH) | ~3.33 × 10¹⁴ | ~3 153 token-wei ≈ $0.003 |
| 6 (low-value token, $0.001/ETH) | ~3.33 × 10¹¹ | ~3.15 × 10⁶ token-wei ≈ $3.15 |

For standard 18-decimal tokens the frozen amount is dust. For 6-decimal tokens with low unit value, or for any token whose price crashes after oracle registration, the frozen amount per transaction grows. Because the condition is purely arithmetic and requires no privileged action, **any user can trigger it on any supported token by choosing a sufficiently small deposit amount**, making the loss repeatable across many transactions.

---

### Likelihood Explanation

- No role or access restriction is needed; any caller can invoke `deposit(token, amount, referralId)` on any supported token.
- The only prerequisite is that `amount > 0` (enforced) and the token is supported (enforced by `onlySupportedToken`).
- The truncation condition is reachable with dust-sized deposits on any supported token, and with larger deposits if a supported token has low unit value or its oracle rate falls.
- The `addSupportedToken` guard only checks `getRate() != 0` at registration time; it does not bound the ratio `agETHToETHrate / tokenToETHRate`. [5](#0-4) 

---

### Recommendation

Add a zero-amount guard immediately after computing `agETHAmount`, before any state mutation:

```solidity
(uint256 agETHAmount, uint256 fee) = viewSwapAgETHAmountAndFee(amount, token);
if (agETHAmount == 0) revert InvalidAmount();   // ← add this
feeEarnedInToken[token] += fee;
agETH.mint(msg.sender, agETHAmount);
```

Alternatively, move the `safeTransferFrom` call **after** the amount calculation so that a revert never leaves tokens stranded.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Assume:
//   agETHToETHrate  = 1.05e18  (agETH oracle)
//   tokenToETHRate  = 1e8      (very low-value 18-dec token, ~$0.0000003/token)
//   feeBps          = 0
//   amount          = 1e9      (1 000 000 000 wei of token)
//
// agETHAmount = 1e9 * 1e8 / 1.05e18
//             = 1e17 / 1.05e18
//             = 0   (integer truncation)
//
// Result: 1e9 token-wei transferred to contract, 0 agETH minted, no refund.

function testZeroMint() external {
    uint256 amount          = 1e9;
    uint256 tokenToETHRate  = 1e8;
    uint256 agETHToETHrate  = 1.05e18;
    uint256 feeBps          = 0;

    uint256 fee           = amount * feeBps / 10_000;   // 0
    uint256 amountAfterFee = amount - fee;               // 1e9
    uint256 agETHAmount   = amountAfterFee * tokenToETHRate / agETHToETHrate;

    assert(agETHAmount == 0);   // passes — user loses 1e9 token-wei
}
```

The same assertion holds for any `(amount, tokenToETHRate, agETHToETHrate)` triple satisfying `amount * tokenToETHRate < agETHToETHrate`, which is trivially reachable with small deposits on any supported token.
