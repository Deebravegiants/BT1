Audit Report

## Title
Cross-Token Drain via Missing Per-Token Collateral Accounting in `_withdraw` - (File: contracts/L2/RsETHTokenWrapper.sol)

## Summary

`RsETHTokenWrapper` allows any `wrsETH` holder to redeem any allowed token regardless of which token they originally deposited. Because `_deposit` mints `wrsETH` 1:1 for any allowed token and `_withdraw` releases any caller-chosen allowed token 1:1 with no per-token balance tracking, a user who deposits a lower-value rsETH variant can withdraw a higher-value rsETH variant, directly stealing collateral from honest depositors.

## Finding Description

The contract tracks acceptable tokens via a flat boolean mapping with no quantity tracking: [1](#0-0) 

`_deposit` mints `wrsETH` 1:1 for any allowed token: [2](#0-1) 

`_withdraw` burns `wrsETH` and transfers any caller-chosen allowed token. The sole guard is an `allowedTokens` membership check — there is no check that the contract holds sufficient balance of the specific token relative to what was deposited against it: [3](#0-2) 

Both public entry points are permissionless: [4](#0-3) 

`maxAmountToDepositBridgerAsset` compounds the issue by computing headroom per-asset in isolation against the total `wrsETH` supply, ignoring balances of all other allowed tokens. This can allow over-collateralization with one token while another remains fully withdrawable: [5](#0-4) 

The contract is explicitly designed to hold multiple allowed tokens: `reinitialize` adds a second token and `addAllowedToken` (TIMELOCK_ROLE) can add more: [6](#0-5) [7](#0-6) 

## Impact Explanation

**Critical — Direct theft of user funds at rest.**

When two allowed tokens diverge in price (bridge latency, liquidity imbalance, or partial depeg on one source chain), an attacker deposits `N` units of the cheaper token, receives `N` `wrsETH`, then calls `withdraw(expensiveToken, N)` to burn those `wrsETH` and receive `N` units of the more expensive token. The price difference is extracted directly from honest depositors' collateral. At scale this fully drains the more valuable token from the wrapper. This matches the allowed impact class: *Direct theft of any user funds, whether at-rest or in-motion*.

## Likelihood Explanation

No privileged access is required. The `withdraw` and `withdrawTo` functions are fully public. The only precondition is that two allowed tokens exist (an intended, documented use case) and that a transient price divergence occurs between them — a realistic condition given bridge latency, cross-chain liquidity differences, or a partial depeg on one source chain. The SECURITY.md exclusion for stablecoin depegging does not apply here; rsETH variants are liquid staking tokens, not stablecoins. The attack is repeatable and atomic within a single transaction.

## Recommendation

Introduce per-token deposit accounting. Add a `mapping(address => uint256) public tokenDeposited` state variable. In `_deposit`, increment `tokenDeposited[_asset] += _amount`. In `_withdraw`, require `tokenDeposited[_asset] >= _amount` before transferring, then decrement `tokenDeposited[_asset] -= _amount`. This ensures `wrsETH` minted against token A can only be redeemed for token A. Update `maxAmountToDepositBridgerAsset` to use `tokenDeposited[_asset]` as the baseline rather than the raw ERC-20 balance, so bridger headroom is also computed per-token.

## Proof of Concept

**Foundry test outline:**

```solidity
// Setup
address rsETH_A = deploy MockERC20("rsETH_A");
address rsETH_B = deploy MockERC20("rsETH_B");
wrapper.addAllowedToken(rsETH_A);   // via TIMELOCK_ROLE
wrapper.addAllowedToken(rsETH_B);   // via TIMELOCK_ROLE

// Honest user deposits rsETH_B
rsETH_B.mint(honestUser, 1000e18);
vm.prank(honestUser);
wrapper.deposit(rsETH_B, 1000e18);
// wrapper holds: rsETH_B=1000, wrsETH supply=1000

// Attacker deposits rsETH_A (cheaper token)
rsETH_A.mint(attacker, 1000e18);
vm.prank(attacker);
wrapper.deposit(rsETH_A, 1000e18);
// wrapper holds: rsETH_A=1000, rsETH_B=1000, wrsETH supply=2000

// Attacker withdraws rsETH_B (more expensive token)
vm.prank(attacker);
wrapper.withdraw(rsETH_B, 1000e18);
// wrapper holds: rsETH_A=1000, rsETH_B=0, wrsETH supply=1000
// attacker holds: 1000 rsETH_B (worth more than the 1000 rsETH_A deposited)
// honestUser's rsETH_B is gone; they hold 1000 wrsETH backed only by rsETH_A

assertEq(rsETH_B.balanceOf(attacker), 1000e18);  // passes
assertEq(rsETH_B.balanceOf(address(wrapper)), 0); // passes — honest user drained
```

The `_withdraw` call succeeds because `allowedTokens[rsETH_B] == true` is the only guard checked. [8](#0-7)

```text
    function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
        if (!allowedTokens[_asset]) return 0;

        // get totalSupply of wrsETH minted
        uint256 wrsETHSupply = totalSupply();
        // balance of _asset with the contract
        uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));

        if (balanceOfAssetInWrapper > wrsETHSupply) return 0;

        return wrsETHSupply - balanceOfAssetInWrapper;
    }
```
