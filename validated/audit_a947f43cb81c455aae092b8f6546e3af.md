Audit Report

## Title
Stale `rsETHPrice` Allows Deposit-Before-Update / Withdraw-After-Update Yield Extraction — (File: contracts/LRTOracle.sol)

## Summary

`LRTOracle.rsETHPrice` is a cached storage variable that diverges from the true TVL-based rate whenever staking rewards accrue. Because `LRTDepositPool.getRsETHAmountToMint` and `LRTWithdrawalManager.getExpectedAssetAmount` both consume the cached value without refreshing it, an attacker can deposit at the stale (lower) price, trigger or wait for the price update, and initiate a withdrawal at the corrected (higher) price — extracting a portion of the accrued yield that belongs to existing rsETH holders.

## Finding Description

`LRTOracle.updateRSETHPrice()` is public with no access control beyond `whenNotPaused`:

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [1](#0-0) 

The deposit path mints rsETH using the stored `rsETHPrice`:

```solidity
// LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

The withdrawal initiation path locks in `expectedAssetAmount` using the same stored value:

```solidity
// LRTWithdrawalManager.sol L592-593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [3](#0-2) 

This `expectedAssetAmount` is stored immutably in the withdrawal request at `initiateWithdrawal` time:

```solidity
// LRTWithdrawalManager.sol L168
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
``` [4](#0-3) 

```solidity
// LRTWithdrawalManager.sol L751-753
withdrawalRequests[requestId] = WithdrawalRequest({
    rsETHUnstaked: rsETHUnstaked, expectedAssetAmount: expectedAssetAmount, withdrawalStartBlock: block.number
});
``` [5](#0-4) 

The `_calculatePayoutAmount` used at `unlockQueue` time returns `min(request.expectedAssetAmount, currentReturn)`:

```solidity
// LRTWithdrawalManager.sol L833-834
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
``` [6](#0-5) 

This minimum is a downside-protection mechanism (guards against slashing), but it does **not** prevent the upside extraction: when the price at unlock time equals or exceeds the price at initiation time (the normal case after rewards), the attacker receives the full `expectedAssetAmount` computed at the higher post-update price.

The `pricePercentageLimit` guard inside `_updateRsETHPrice` only restricts non-managers from calling `updateRSETHPrice()` when the price jump exceeds the configured threshold:

```solidity
// LRTOracle.sol L256-264
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
if (isPriceIncreaseOffLimit) {
    if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
        revert PriceAboveDailyThreshold();
    }
}
``` [7](#0-6) 

This guard is bypassed entirely when `pricePercentageLimit == 0`, and is irrelevant when the attacker simply deposits before the manager's own price-update transaction (front-running) rather than calling `updateRSETHPrice()` themselves.

**Exploit flow:**

1. Observe: true TVL = 1 010 ETH, rsETH supply = 1 000, stored `rsETHPrice` = 1.000e18 (stale).
2. Attacker calls `depositETH{value: 100 ether}` → mints `100 * 1e18 / 1.000e18 = 100 rsETH` (correct amount would be ≈99.01 rsETH).
3. `updateRSETHPrice()` executes (by attacker or manager) → new price = 1 110 ETH / 1 100 rsETH ≈ 1.009e18.
4. Attacker calls `initiateWithdrawal(ETH, 100e18)` → `expectedAssetAmount = 100 * 1.009e18 / 1e18 = 100.9 ETH` stored.
5. After `withdrawalDelayBlocks`, operator calls `unlockQueue`; `_calculatePayoutAmount` returns `min(100.9, 100.9) = 100.9 ETH`.
6. Attacker calls `completeWithdrawal` → receives 100.9 ETH. Net profit ≈ 0.9 ETH extracted from the 10 ETH reward pool.

## Impact Explanation

**High — Theft of unclaimed yield.**

Staking rewards that should accrue pro-rata to all existing rsETH holders are partially siphoned by an attacker who deposits at the stale price and redeems at the corrected price. The surplus rsETH minted at the stale price represents a dilution of existing holders' claims; when redeemed at the updated price, the excess is paid out of the reward pool. The attack is repeatable on every reward epoch.

## Likelihood Explanation

**Medium.**

- No privileged access is required; `depositETH`, `initiateWithdrawal`, and `completeWithdrawal` are all public.
- The price gap opens automatically on every staking reward epoch; the attacker only needs to monitor on-chain TVL vs. stored `rsETHPrice`.
- The attacker does not need to call `updateRSETHPrice()` themselves — depositing before the manager's routine update and initiating withdrawal after is sufficient, bypassing the `pricePercentageLimit` guard entirely.
- The 8-day withdrawal delay reduces frequency but does not eliminate profitability; the payout is locked in at `initiateWithdrawal` time.
- Profitability scales with deposit size and reward epoch magnitude.

## Recommendation

1. **Atomically refresh the price before computing mint/redeem amounts.** Call `_updateRsETHPrice()` (or compute the live TVL-based rate inline) at the start of `depositETH`, `depositAsset`, and `initiateWithdrawal`, so the stored value is never stale at the moment of user interaction.
2. **Alternatively, restrict `updateRSETHPrice()` to a trusted keeper or the deposit/withdrawal contracts**, ensuring the stored price advances only in lockstep with protocol operations and cannot be exploited via front-running.

## Proof of Concept

**Minimal call sequence (fork test):**

```
// Setup: 1000 rsETH outstanding, 1000 ETH TVL, rsETHPrice = 1.000e18
// Simulate reward: add 10 ETH to protocol (e.g., via receiveFromRewardReceiver)
// Now true price ≈ 1.010e18 but rsETHPrice storage = 1.000e18

// Step 1: Attacker deposits
vm.prank(attacker);
depositPool.depositETH{value: 100 ether}(0, "");
// Attacker receives 100 rsETH (excess ≈ 0.99 rsETH vs. fair share)

// Step 2: Price update (attacker or manager)
lrtOracle.updateRSETHPrice();
// rsETHPrice ≈ 1110/1100 ≈ 1.009e18

// Step 3: Attacker initiates withdrawal
vm.prank(attacker);
rsETH.approve(address(withdrawalManager), 100e18);
withdrawalManager.initiateWithdrawal(ETH_TOKEN, 100e18, "");
// expectedAssetAmount = 100 * 1.009e18 / 1e18 = 100.9 ETH stored

// Step 4: Advance blocks past withdrawalDelayBlocks
vm.roll(block.number + withdrawalDelayBlocks + 1);

// Step 5: Operator unlocks queue
vm.prank(operator);
withdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, 0, type(uint256).max, 0, type(uint256).max);

// Step 6: Attacker completes withdrawal
vm.prank(attacker);
withdrawalManager.completeWithdrawal(ETH_TOKEN, "");
// Attacker receives 100.9 ETH; net profit ≈ 0.9 ETH from existing holders' yield

assertGt(attacker.balance, 100 ether); // passes
```
