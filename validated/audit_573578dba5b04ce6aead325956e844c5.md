### Title
`respond_ckd` Skips Cryptographic Output Verification for `AppPublicKey` Requests, Enabling a Single Byzantine Participant to Deliver a Forged CKD Response — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` enforces a BLS12-381 pairing check (`ckd_output_check`) only when the pending request used the `AppPublicKeyPV` variant. When the request used the legacy `AppPublicKey` variant, the verification arm is an empty no-op. A single attested participant — strictly below the signing threshold — can therefore call `respond_ckd` with an arbitrary `(big_y, big_c)` pair for any pending `AppPublicKey` CKD request and the contract will accept and deliver the forged output to the user.

---

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` dispatches on the `app_public_key` field of the stored `CKDRequest`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV` requests, `ckd_output_check` verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the master public key and the user's ephemeral key pair: [2](#0-1) 

For `AppPublicKey` requests the arm is empty — no pairing check, no point validation, no binding to the master key. Any `CKDResponse` struct is accepted verbatim and forwarded to the user via `resolve_yields_for`. [3](#0-2) 

The `AppPublicKey` variant is the **default and legacy path** used by the example CLI and the sandbox test suite: [4](#0-3) 

By contrast, `respond` (for ECDSA/EdDSA signatures) always validates the cryptographic output against the derived public key before resolving the yield — there is no unverified fast path: [5](#0-4) 

This is the direct structural analog of M-05: one request variant (`AppPublicKeyPV`) has the invariant enforced; the other (`AppPublicKey`) silently bypasses it, breaking the security guarantee that the delivered CKD output was honestly computed by the threshold network.

---

### Impact Explanation

A single Byzantine attested participant calls `respond_ckd` with a pending `AppPublicKey` request and supplies an attacker-chosen `(big_y, big_c)`. The contract resolves the yield and returns the forged pair to the user. The user's derived secret — computed as `HKDF(big_c − a·big_y)` where `a` is their ephemeral private key — is now fully attacker-controlled. Any asset, wallet, or secret protected by that derived key is compromised.

This matches the allowed Critical impact: **confidential key derivation output without the required participant authorization** — a single participant below the signing threshold can unilaterally forge a CKD result that should require an honest threshold computation.

---

### Likelihood Explanation

The attacker entry path requires only that the caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`). No threshold collusion is needed. Any single participant who turns Byzantine — or whose TEE is compromised — can exploit this against every concurrent `AppPublicKey` CKD request. The `AppPublicKey` variant is the default and most widely documented path, so the population of vulnerable requests is large. [6](#0-5) 

---

### Recommendation

Apply the same cryptographic binding check to `AppPublicKey` responses that is already applied to `AppPublicKeyPV` responses. Because `AppPublicKey` supplies only a G1 point (`pk1`) and not the G2 companion (`pk2`), the existing `ckd_output_check` pairing cannot be used directly. The mitigation options are:

1. **Deprecate `AppPublicKey` and require `AppPublicKeyPV`** for all new requests, so the on-chain pairing check is always available. This is consistent with the contract's ability to verify `respond` outputs for signatures.
2. **If `AppPublicKey` must be retained**, document explicitly that users of this variant receive no on-chain integrity guarantee and must verify the response themselves before using the derived key — and add a prominent warning in the contract ABI and README.

The structural fix mirrors the M-05 recommendation: make the verified path the only path, or make the unverified path impossible to reach without explicit user opt-in and documented risk acceptance.

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk1)` and `domain_id` pointing to the BLS12-381 CKD domain. A `CKDRequest` is stored in `pending_ckd_requests`. [7](#0-6) 

2. Byzantine attested participant calls `respond_ckd(ckd_request, CKDResponse { big_y: [1u8;48], big_c: [2u8;48] })`. The `AppPublicKey` arm is empty; no check fires. [8](#0-7) 

3. `resolve_yields_for` resumes the yield with the forged response. The user's callback receives `big_y = [1u8;48], big_c = [2u8;48]`. [3](#0-2) 

4. The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates this path succeeds with `big_y = [1u8;48], big_c = [2u8;48]` — values that are not valid BLS12-381 points and bear no relation to the master key — confirming the absence of any output check. pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
