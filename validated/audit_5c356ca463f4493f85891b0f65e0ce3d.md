Audit Report

## Title
Protocol Fee rsETH Dilution Rate Is Super-Linear in `updateRSETHPrice()` Call Frequency - (File: contracts/LRTOracle.sol)

## Summary
In `LRTOracle._updateRsETHPrice()`, the ETH-denominated protocol fee is converted to rsETH using the post-fee price (`newRsETHPrice`). Because this price is lower when the function is called more frequently (less reward has accumulated per call), more rsETH is minted per ETH of fee on each invocation. Across N calls covering the same total reward, cumulative rsETH minted is strictly greater than across a single call, causing existing holders to be diluted beyond the nominal `protocolFeeInBPS` rate. Since `updateRSETHPrice()` is permissionless, any external caller can increase call frequency to amplify this effect.

## Finding Description
The fee conversion at line 301 uses `newRsETHPrice`, which is the post-fee price computed at line 250:

```solidity
// LRTOracle.sol L244-250
uint256 rewardAmount = totalETHInProtocol - previousTVL;
protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
// ...
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

```solidity
// LRTOracle.sol L301
uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
```

The ETH fee is always exactly `protocolFeeInBPS / 10_000` of incremental reward — this part is correct. However, the rsETH amount minted equals `protocolFeeInETH / newRsETHPrice`. When called more frequently, `newRsETHPrice` is lower at each call (less reward has compounded), so the denominator is smaller and more rsETH is minted per ETH of fee. This is a path-dependent, non-linear relationship.

`updateRSETHPrice()` at line 87 carries no access control:

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Any unprivileged caller can invoke it at will, increasing call frequency and thus total rsETH dilution.

The daily fee mint limit (`_checkAndUpdateDailyFeeMintLimit`, lines 197–210) provides a partial cap on total rsETH minted per day, but does not fix the fundamental math: within the daily cap, more frequent calls still mint more rsETH per ETH of fee than fewer calls would. If `maxFeeMintAmountPerDay` is set generously (or based on expected infrequent-call amounts), the attacker can exhaust the daily budget faster while having minted more rsETH per unit of reward than the protocol intended.

## Impact Explanation
Existing rsETH holders are diluted by the treasury fee mint. The degree of dilution per unit of protocol reward is not fixed — it is higher when `updateRSETHPrice()` is called more frequently. The protocol fails to deliver the exact promised fee rate to holders: the effective dilution rate is super-linear in call frequency rather than being a flat `protocolFeeInBPS` of rewards. No funds are lost outright, but holders receive fewer net rewards than the nominal fee parameter implies when the function is called at high cadence.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
`updateRSETHPrice()` is a permissionless public function callable by any address. Any depositor, rsETH holder, or bot can call it at will. The protocol is expected to be called regularly by keeper bots, and call frequency is not bounded on-chain. The condition is always reachable by any unprivileged caller without any special setup.

## Recommendation
Convert the fee to rsETH using the **pre-fee** price (`totalETHInProtocol / rsethSupply`) rather than the post-fee price `newRsETHPrice`. This makes the rsETH fee amount independent of call cadence:

```solidity
uint256 preFeePrice = totalETHInProtocol.divWad(rsethSupply);
uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(preFeePrice);
```

Alternatively, adopt a continuous-compounding or fixed-period fee model so the effective rate is independent of call cadence, analogous to EIP-4626 streaming fee patterns.

## Proof of Concept
Assume `rsethSupply = 100`, initial `rsETHPrice = 1.0`, `protocolFeeInBPS = 1000` (10%), TVL grows from 100 → 120 ETH.

**Single call (TVL 100 → 120):**
- `rewardAmount = 20`, `protocolFeeInETH = 2`
- `newRsETHPrice = (120 − 2) / 100 = 1.18`
- `rsethFee = 2 / 1.18 ≈ 1.695 rsETH` minted

**Two calls (TVL 100 → 110, then 110 → 120):**

*Call 1:*
- `rewardAmount = 10`, `protocolFeeInETH = 1`
- `newRsETHPrice = (110 − 1) / 100 = 1.09`
- `rsethFee₁ = 1 / 1.09 ≈ 0.917 rsETH`; supply → 100.917, price → 1.09

*Call 2:*
- `previousTVL = 100.917 × 1.09 ≈ 110`, `rewardAmount = 10`, `protocolFeeInETH = 1`
- `newRsETHPrice = (120 − 1) / 100.917 ≈ 1.1792`
- `rsethFee₂ = 1 / 1.1792 ≈ 0.848 rsETH`; supply → 101.765

**Result:** Two calls mint ≈ **1.765 rsETH** vs. one call minting ≈ **1.695 rsETH** — a ~4% increase in dilution for the same total reward, caused solely by higher call frequency. A Foundry fuzz test parameterizing call count N over a fixed total reward interval would confirm the monotonically increasing total rsETH minted as N increases, with the single-call case as the minimum.
