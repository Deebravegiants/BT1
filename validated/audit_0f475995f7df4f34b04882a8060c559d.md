Audit Report

## Title
Unverified Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Tokens — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address in every public entry point without verifying it against the factory registry. A malicious contract posing as a pool can exploit the callback settlement flow to pull up to `maxAmountToken0`/`maxAmountToken1` of any tokens the victim has approved to the adder. The contract's own NatSpec explicitly documents this gap.

## Finding Description
Every public entry point (`addLiquidityExactShares`, `addLiquidityWeighted`) forwards the caller-supplied `pool` address directly into `_addLiquidity` without any factory membership check. The constructor stores no factory reference: [1](#0-0) 

The NatSpec explicitly acknowledges the missing check: [2](#0-1) 

Inside `_addLiquidity`, the unverified pool is stored as the expected callback caller in transient storage, then called: [3](#0-2) 

In `metricOmmModifyLiquidityCallback`, the guard `msg.sender != expectedPool` passes because the malicious pool **is** the stored expected pool. The callback then trusts `IMetricOmmPool(msg.sender).getImmutables()` for token addresses and calls `pay`: [4](#0-3) 

`pay` resolves to `safeTransferFrom(payer, recipient, value)` — transferring tokens from the victim to the malicious pool: [5](#0-4) 

The factory exposes `isPool(address)` returning `poolToIdx[pool] != 0`, a ready-made registry check that is never called by the adder: [6](#0-5) 

For `addLiquidityWeighted`, the attack is especially clean: the malicious pool's `addLiquidity` reverts with `LiquidityProbe(need0, need1)` during the probe phase (matching the expected revert selector), causing the adder to compute scaled shares and proceed to the paying phase where the drain occurs: [7](#0-6) 

## Impact Explanation
A victim who has approved `MetricOmmPoolLiquidityAdder` for token0 and/or token1 (required for normal use) can have up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 drained in a single transaction. Because the malicious pool controls the token addresses returned by `getImmutables()`, it can target **any** token the victim has approved to the adder — not just a pool's native pair. This is a direct loss of user principal with no recovery path, meeting the Critical/High threshold for direct loss of user funds.

## Likelihood Explanation
Any unprivileged attacker can deploy a malicious contract implementing `IMetricOmmPoolActions` and `IMetricOmmPool`. The victim only needs to be directed to call `addLiquidityExactShares` or `addLiquidityWeighted` with the malicious address — a realistic phishing or front-end injection vector. Users who have pre-approved the adder (the normal usage pattern) are permanently at risk until they revoke approvals. The attack is repeatable and requires no privileged access.

## Recommendation
Store the factory address as an immutable in the constructor (analogous to how `MetricOmmSimpleRouter` stores it via `MetricOmmSwapRouterBase`): [8](#0-7) 

Add a factory membership check at the top of `_addLiquidity`:

```solidity
function _addLiquidity(address pool, ...) internal returns (...) {
    if (!IMetricOmmPoolFactory(FACTORY).isPool(pool)) revert NotAFactoryPool();
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

## Proof of Concept
1. Deploy `MaliciousPool(adder, victimToken, attacker)` implementing `IMetricOmmPoolActions` and `IMetricOmmPool`:
   - `addLiquidity(...)` calls back `metricOmmModifyLiquidityCallback(MAX0, 0, abi.encode(uint8(1)))` then returns `(MAX0, 0)`
   - `getImmutables()` returns `PoolImmutables{token0: victimToken, ...}`
2. Victim (who has approved the adder for `victimToken`) calls `adder.addLiquidityExactShares(maliciousPool, victim, 0, deltas, MAX0, 0, "")`.
3. `_addLiquidity` stores `maliciousPool` as `expectedPool` in transient storage and calls `maliciousPool.addLiquidity(...)`.
4. `maliciousPool` calls back `metricOmmModifyLiquidityCallback(MAX0, 0, abi.encode(1))`.
5. Guard `msg.sender != expectedPool` passes; `amount0Delta <= max0` passes.
6. Callback calls `pay(victimToken, victim, maliciousPool, MAX0)` → `safeTransferFrom(victim, maliciousPool, MAX0)`.
7. `MAX0` of `victimToken` is transferred from victim to malicious pool.
