### Title
`verify_foreign_transaction` Gates on Deprecated `get_supported_foreign_chains()` Instead of `get_available_foreign_chains()`, Allowing a Single Byzantine Participant to Block All Foreign-Chain Verification - (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` enforces chain eligibility using `get_supported_foreign_chains()` — the legacy all-participant intersection rule — instead of `get_available_foreign_chains()`, which is the threshold-based availability check the design specification mandates. A single Byzantine participant strictly below the signing threshold can block every `verify_foreign_transaction` request for any chain by simply not registering via the old `register_foreign_chain_config` mechanism, even when the chain is properly governance-whitelisted and covered by a signing-threshold of nodes.

---

### Finding Description

In `verify_foreign_transaction` the contract reads:

```rust
let supported_chains = self.get_supported_foreign_chains();   // line 534
if !supported_chains.contains(&requested_chain) {
    env::panic_str(...ForeignChainNotSupported...);
}
```

`get_supported_foreign_chains()` computes the **strict intersection** of every active participant's chains registered via the old `register_foreign_chain_config` / `node_foreign_chain_support` data path:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
```

A chain is returned only if **every** active participant has registered it. One absent registration drops the chain to zero.

The design specification (`docs/design/calculating-supported-foreign-chains.md`) is unambiguous:

> `verify_foreign_transaction(C)` is **rejected unless `C` is available**

> The legacy `get_supported_foreign_chains()` (the intersection rule) is **to be deprecated** in favour of the two views above.

`get_available_foreign_chains()` — the correct gate — returns chains that are governance-whitelisted **and** covered by ≥ `signing_threshold` active participants, reading from the new `foreign_chains.available_foreign_chains` data structure:

```rust
pub fn get_available_foreign_chains(&self) -> dtos::AvailableForeignChains {
    self.foreign_chains.get().available_foreign_chains.clone()
}
```

The two methods read from entirely separate storage fields and can diverge. The contract's own test suite documents the regression explicitly:

> "Bitcoin is available. A single non-covering node cannot take it down — **the regression the legacy intersection rule had**."

That test validates `get_available_foreign_chains()` in isolation, but `verify_foreign_transaction` never calls it; it still calls the deprecated method.

---

### Impact Explanation

**Medium — request-lifecycle manipulation that breaks production safety/accounting invariants.**

A chain that is governance-whitelisted and covered by a signing-threshold of nodes (i.e., fully serviceable) is permanently rejected at the contract gate if even one active participant has not registered via the old `register_foreign_chain_config` path. Every `verify_foreign_transaction` call for that chain panics with `ForeignChainNotSupported`, the yield slot is never opened, and the deposited funds are returned — but the bridge flow (e.g., Omnibridge inbound) is stalled indefinitely. The attacker does not need to compromise any key or exceed the threshold; they only need to withhold a single registration call.

---

### Likelihood Explanation

A Byzantine participant strictly below the signing threshold can trigger this by omitting or clearing their `register_foreign_chain_config` registration. No special privilege, no key material, and no network-level attack is required. The participant's node remains online and participates in all other signing; only the foreign-chain registration is withheld. Because the old and new registration paths are independent storage fields, a node that has correctly registered via the new `register_foreign_chains_config` / `vote_update_foreign_chain_providers` path may still have an empty entry in `node_foreign_chain_support`, causing the intersection to exclude the chain.

---

### Recommendation

Replace the deprecated gate in `verify_foreign_transaction` with the threshold-based availability check:

```rust
// Before (incorrect — legacy intersection rule):
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) { ... }

// After (correct — threshold-based availability):
let available_chains = self.get_available_foreign_chains();
if !available_chains.contains(&requested_chain) { ... }
```

This aligns the production code with the design specification and with the guarantees already validated by the `get_available_foreign_chains` test suite.

---

### Proof of Concept

**Setup**: 4-participant network, signing threshold 3. Bitcoin is governance-whitelisted via `vote_update_foreign_chain_providers` (threshold vote passed). Participants 1–3 register Bitcoin via `register_foreign_chains_config` (new path). Participant 4 (Byzantine, below threshold) never calls `register_foreign_chain_config` (old path), leaving `node_foreign_chain_support` empty for their account.

**State**:
- `get_available_foreign_chains()` → `{Bitcoin}` ✓ (whitelisted + 3/4 ≥ threshold coverage)
- `get_supported_foreign_chains()` → `{}` ✗ (participant 4 absent from old intersection)

**Attack**: Any user calls `verify_foreign_transaction` with a Bitcoin request.

**Result** (`crates/contract/src/lib.rs`, lines 533–541):
```rust
let requested_chain = request.request.chain();          // Bitcoin
let supported_chains = self.get_supported_foreign_chains(); // {} — empty
if !supported_chains.contains(&requested_chain) {       // true — panics
    env::panic_str(&InvalidParameters::ForeignChainNotSupported { ... });
}
```

The request is rejected with `ForeignChainNotSupported` despite Bitcoin being fully available. The bridge flow is blocked. Participant 4 has achieved a complete denial of foreign-chain verification using only a withheld registration call, with no threshold collusion required.     pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
```rust
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }
```
