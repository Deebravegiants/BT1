### Title
Unconditional `updatedAt` advance in `updateReward` modifier when `totalKernelStaked == 0` permanently freezes streamed rewards - (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool`'s `updateReward` modifier unconditionally advances `updatedAt = lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. Because `rewardPerToken()` returns the stored value unchanged when supply is zero, the elapsed time window is consumed without crediting any accumulator. Rewards that streamed during the zero-supply gap are permanently locked in the contract and can never be claimed by any staker.

---

### Finding Description

The `updateReward` modifier in `KernelDepositPool` runs on every state-changing call (`stake`, `stakeFor`, `initiateWithdrawal`, `getReward`, `notifyRewardAmount`):

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // ← may return unchanged value
    updatedAt = lastTimeRewardApplicable();    // ← always advances
    ...
}
``` [1](#0-0) 

`rewardPerToken()` correctly short-circuits when supply is zero:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // no accumulation
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
``` [2](#0-1) 

But the modifier still executes `updatedAt = lastTimeRewardApplicable()` regardless. This means the time interval `[updatedAt_before, now]` is permanently consumed: `rewardPerTokenStored` does not increase, yet `updatedAt` jumps forward. Any `rewardRate * elapsed` rewards that accrued during that window are silently discarded — they remain as token balance in the contract but are unreachable by any position.

The withdrawal path that reduces `totalKernelStaked` to zero is `initiateWithdrawal`:

```solidity
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;   // ← can reach 0
    ...
}
``` [3](#0-2) 

The contract's own NatSpec acknowledges the issue but relies entirely on off-chain operational discipline:

```
* @dev If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards
*      for that period will stay locked in the contract. In this deployment, we're avoiding this issue by
*      ensuring there are always some tokens staked...
``` [4](#0-3) 

No on-chain enforcement prevents the last staker from withdrawing during an active reward period.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

When `totalKernelStaked` drops to zero mid-period, every subsequent call that triggers `updateReward` (including the next `stake`) advances `updatedAt` without accumulating `rewardPerTokenStored`. The rewards for the entire zero-supply gap are permanently stranded in the contract balance. Stakers who re-enter after the gap cannot claim those rewards; they are not recoverable by any user or admin function.

---

### Likelihood Explanation

Any unprivileged staker who is the last remaining depositor can trigger this by calling `initiateWithdrawal` for their full balance during an active reward period. No special role, governance action, or external dependency is required. The scenario is realistic whenever staking participation is low or a single large staker exits.

---

### Recommendation

Mirror the fix from the referenced report: do not advance `updatedAt` when `totalKernelStaked == 0`. The modifier should preserve the time window so rewards are credited once stakers return:

```diff
modifier updateReward(address _account) {
+   if (totalKernelStaked > 0) {
        rewardPerTokenStored = rewardPerToken();
        updatedAt = lastTimeRewardApplicable();
+   }

    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures the elapsed window is only consumed when there are stakers to receive it, preserving the pending rewards for future participants.

---

### Proof of Concept

1. Admin calls `notifyRewardAmount` with `rewardRate > 0` and `finishAt = now + 7 days`.
2. Alice is the only staker (`totalKernelStaked = Alice's balance`).
3. Alice calls `initiateWithdrawal(fullBalance)`:
   - `updateReward` runs: `rewardPerToken()` accumulates correctly (supply still > 0 at modifier entry), then `updatedAt = block.timestamp`.
   - `totalKernelStaked` becomes 0.
4. 3 days pass. `rewardRate * 3 days` worth of rewards accrue but are credited to no one.
5. Bob calls `stake(amount)`:
   - `updateReward` runs: `rewardPerToken()` returns `rewardPerTokenStored` unchanged (supply == 0), but `updatedAt` advances to `block.timestamp`.
   - The 3-day reward window is permanently consumed.
6. Bob calls `getReward()` — receives only rewards from step 5 onward, not the 3-day gap.
7. The contract holds `rewardRate * 3 days` tokens that can never be claimed by any address.
