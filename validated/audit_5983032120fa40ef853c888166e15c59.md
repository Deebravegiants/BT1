Audit Report

## Title
Uninitialized `maxFeeMintAmountPerDay` Causes DoS on `updateRSETHPrice()` When Rewards Accrue — (`contracts/LRTOracle.sol`)

## Summary
`reinitialize()` sets only `feePeriodStartTime` and never initializes `maxFeeMintAmountPerDay`, leaving it at the Solidity default of `0`. Once staking rewards push `totalETHInProtocol` above `previousTVL`, every call to the public `updateRSETHPrice()` permanently reverts with `DailyFeeMintLimitExceeded`, freezing `rsETHPrice` at a stale value until a manager manually calls `setMaxFeeMintAmountPerDay`.

## Finding Description
`reinitialize()` is the upgrade-time setup entry point. It accepts and stores `feePeriodStartTime` but never touches `maxFeeMintAmountPerDay`: [1](#0-0) 

`maxFeeMintAmountPerDay` therefore remains `0` (Solidity default) unless the manager separately calls `setMaxFeeMintAmountPerDay`: [2](#0-1) 

Inside `_updateRsETHPrice()`, whenever the protocol is unpaused and `totalETHInProtocol > previousTVL`, a non-zero `protocolFeeInETH` is computed and `_checkAndUpdateDailyFeeMintLimit` is called unconditionally: [3](#0-2) 

`_checkAndUpdateDailyFeeMintLimit` evaluates `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay`. With `maxFeeMintAmountPerDay == 0` and any `feeAmount > 0`, this is `0 + feeAmount > 0` → always `true` → always reverts: [4](#0-3) 

Because both the public `updateRSETHPrice()` and the manager-only `updateRSETHPriceAsManager()` route through the same `_updateRsETHPrice()` internal, neither can succeed: [5](#0-4) 

The revert occurs before `rsETHPrice = newRsETHPrice` at line 313, so the stored price is never updated. Note that `remainingDailyFeeMintLimit()` already handles the zero case by returning `0` early, but `_checkAndUpdateDailyFeeMintLimit` has no equivalent guard: [6](#0-5) 

## Impact Explanation
**Medium — Temporary freezing of funds.**

`rsETHPrice` is frozen at its pre-reward value. Withdrawers who redeem rsETH while the price is stale receive less ETH than they are entitled to (the accrued yield is not reflected), making their yield temporarily inaccessible. The freeze persists for an indeterminate duration until a privileged manager calls `setMaxFeeMintAmountPerDay` with a non-zero value. Because the fix requires a privileged transaction but no upgrade, the impact does not reach Critical/Permanent.

## Likelihood Explanation
Moderate. The `reinitialize()` function is the canonical upgrade-time setup call and it structurally omits `maxFeeMintAmountPerDay`. Any deployment or upgrade that does not follow up with a separate `setMaxFeeMintAmountPerDay` call leaves the contract in the broken state. Staking rewards accrue continuously on Ethereum, so the first oracle update after any reward accrual will trigger the revert. No attacker capability is required — the condition is met passively by normal protocol operation, and any unprivileged address can call `updateRSETHPrice()` to confirm the DoS.

## Recommendation
**Option 1 (preferred):** Add `maxFeeMintAmountPerDay` as a parameter to `reinitialize()` and set it there, ensuring the upgrade atomically configures the limit.

**Option 2:** Treat `maxFeeMintAmountPerDay == 0` as "no cap" inside `_checkAndUpdateDailyFeeMintLimit`, consistent with the intent implied by `remainingDailyFeeMintLimit()`:
```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (maxFeeMintAmountPerDay == 0) return; // 0 = unlimited
    ...
}
```

## Proof of Concept
```
Preconditions:
  - LRTOracle upgraded; reinitialize() called → maxFeeMintAmountPerDay == 0
  - protocolFeeInBPS > 0 (protocol fee is configured)
  - setMaxFeeMintAmountPerDay() never called

Trigger sequence:
  1. Staking rewards accrue: totalETHInProtocol > rsethSupply * rsETHPrice
  2. Any address calls updateRSETHPrice()
  3. _updateRsETHPrice() computes protocolFeeInETH > 0
  4. rsethAmountToMintAsProtocolFee = protocolFeeInETH / newRsETHPrice > 0
  5. _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee):
       0 + rsethAmountToMintAsProtocolFee > 0  →  true
       revert DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)
  6. rsETHPrice is never written; all subsequent calls also revert

Foundry test plan:
  - Deploy LRTOracle proxy; call reinitialize() with a valid feePeriodStartTime
  - Do NOT call setMaxFeeMintAmountPerDay
  - Mock totalETHInProtocol > previousTVL and protocolFeeInBPS > 0
  - vm.expectRevert(DailyFeeMintLimitExceeded.selector); oracle.updateRSETHPrice();
  - Assert rsETHPrice unchanged after the revert
  - Call setMaxFeeMintAmountPerDay(1e18); oracle.updateRSETHPrice(); — assert succeeds
```
