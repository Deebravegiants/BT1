Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice` Uses Unvalidated Chainlink Price Without Staleness or Validity Checks - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but discards `updatedAt`, `answeredInRound`, and does not guard against `price <= 0`. This unvalidated price is used directly in `LRTDepositPool.getRsETHAmountToMint` to determine how many rsETH tokens to mint per deposited LST. When a Chainlink feed is stale at a price higher than the true current price, depositors receive more rsETH than the ETH-equivalent value they deposited, diluting all existing rsETH holders.

## Finding Description
`contracts/oracles/ChainlinkPriceOracle.sol` line 52 reads only the `answer` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The three discarded return values — `updatedAt` (staleness timestamp), `answeredInRound` vs `roundId` (incomplete round), and the sign of `price` — are all validated in the sibling oracle `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unvalidated price flows through `LRTOracle.getAssetPrice` → `LRTDepositPool.getRsETHAmountToMint`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

And into `LRTOracle._getTotalEthInProtocol` which computes the TVL used to update `rsETHPrice`: [4](#0-3) 

**Exploit path:**
1. At T0: stETH/ETH Chainlink feed reports `1.05e18`. `updateRSETHPrice()` is called; `rsETHPrice` is stored as `1.05e18`.
2. At T1: True stETH price drops to `1.00e18`, but the Chainlink feed goes stale (network congestion, circuit-breaker, sequencer outage) and still reports `1.05e18`.
3. Attacker calls `depositAsset(stETH, 100e18, 0, "")`.
4. `getAssetPrice(stETH)` returns `1.05e18` (stale). `rsETHPrice` is `1.05e18` (last stored correctly).
5. `rsethAmountToMint = (100e18 * 1.05e18) / 1.05e18 = 100e18` rsETH minted.
6. Correct amount: `(100e18 * 1.00e18) / 1.05e18 ≈ 95.24e18` rsETH.
7. Attacker receives `~4.76 rsETH` excess, representing a claim on ~5 ETH of value they did not deposit.

No existing guard prevents this: `minRSETHAmountExpected` is a depositor-controlled slippage parameter (set to 0 by the attacker), and the `pricePercentageLimit` check in `_updateRsETHPrice` only applies when `updateRSETHPrice()` is called, not during deposits. [5](#0-4) 

## Impact Explanation
The attacker mints rsETH tokens representing a claim on more ETH than they deposited. When the feed recovers and `rsETHPrice` is recalculated correctly, the attacker's excess rsETH dilutes all existing holders' share of the underlying ETH. This constitutes **direct theft of user funds (Critical)**: existing rsETH holders' redemption value is permanently reduced by the amount over-minted to the attacker.

The secondary impact from `price == 0` is a DoS: `uint256(0)` causes `rsethAmountToMint = 0`, freezing deposits for any depositor who does not set `minRSETHAmountExpected = 0`. A negative `price` causes `uint256(price)` to wrap to `2^256-1`, which overflows in the multiplication under Solidity 0.8.x checked arithmetic, reverting all deposit calls — a **temporary freezing of funds**.

## Likelihood Explanation
Chainlink feeds have gone stale during high network congestion and have returned clamped values during extreme market events (LUNA collapse, USDC depeg). `updateRSETHPrice()` is public and permissionless, so any actor can lock in a stale price into `rsETHPrice`. The attack requires no privileged access: any depositor can call `depositAsset` while a feed is stale. The condition (feed stale while `rsETHPrice` reflects a prior correct value) is a realistic, recurring scenario on mainnet and L2s.

## Recommendation
Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, add a configurable per-asset `heartbeat` staleness threshold (e.g., `block.timestamp - updatedAt > heartbeat`) to catch feeds that are technically complete but outdated beyond their expected update interval.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// Fork mainnet at a block where stETH/ETH feed is live
// 1. Record current rsETHPrice (set by a prior updateRSETHPrice() call)
// 2. Mock the stETH/ETH Chainlink aggregator to return a stale answer:
//    - roundId = 100, answeredInRound = 99, updatedAt = block.timestamp - 4 hours
//    - answer = 1.05e8 (above true price of 1.00e8)
// 3. Call depositAsset(stETH, 100e18, 0, "") as attacker
// 4. Assert rsETH minted > (100e18 * truePriceNow) / rsETHPrice
// 5. Assert the delta equals the over-minted amount (~4.76e18 rsETH)
// 6. Call updateRSETHPrice() after feed recovers (mock returns 1.00e8)
// 7. Assert rsETHPrice decreased, confirming dilution of existing holders
```

The test is reproducible on a mainnet fork by warping `block.timestamp` past the Chainlink feed's `updatedAt` and using `vm.mockCall` to return a stale answer from the aggregator, then verifying the minted rsETH exceeds the fair-value amount.
