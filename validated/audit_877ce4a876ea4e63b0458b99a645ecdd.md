### Title
Malicious Peer Can Corrupt DL Singleton Root via Crafted Solution in `singleton_removed` — (`chia/data_layer/data_layer_wallet.py`)

### Summary

`DataLayerWallet.singleton_removed` extracts the next-generation root hash from the hint field of a `CREATE_COIN` condition produced by running the peer-supplied solution through `run_with_cost`. The only guard in the call chain validates the **puzzle_reveal** against the coin's known `puzzle_hash`, but the **solution** is accepted verbatim from the peer. A malicious full-node peer can return the correct puzzle_reveal (passing the hash check) paired with a crafted solution whose delegated puzzle emits a `CREATE_COIN` with an arbitrary 32-byte root hash in `condition[3][1]`, causing the wallet to persist a phantom root in the singleton record and permanently desync the local Data Layer store from on-chain state.

---

### Finding Description

**Call chain:**

```

  → fetch_coin_spend_for_coin_state(coin_state, peer)   # validates puzzle hash only

  → DataLayerWallet.singleton_removed(singleton_spend, height)

**Why the crafted solution works:**

A DL singleton's inner puzzle is typically a standard p2 (BLS-signed) puzzle. The standard p2 puzzle takes a *delegated puzzle* from the solution and runs it to produce conditions. The `AGG_SIG_ME` condition it emits is ignored by `singleton_removed` (which only scans for `CREATE_COIN`). An attacker can supply a crafted solution whose delegated puzzle is:

```clvm
(q . ((51 <real_child_puzhash> 1 (<launcher_id> <attacker_root> <inner_puzhash>))))
```

This passes `run_with_cost` successfully (the puzzle_reveal is valid), and `condition[3][1]` becomes the attacker-controlled `<attacker_root>`. No signature is checked by the wallet during sync.

Note also that `match_dl_singleton` only validates the puzzle structure (curried mod hashes), not the solution:

```python
matched, _ = match_dl_singleton(puzzle)
``` [5](#0-4) 

The curried root inside the puzzle_reveal (the *current* generation's root) is never compared against the hint-derived root (the *next* generation's root) that gets stored.

---

### Impact Explanation

The phantom root is stored as `confirmed=True` in the DL store. All subsequent wallet operations that read the latest singleton record — including proof-of-inclusion verification, DL offer settlement (`finish_graftroot_solutions`), and any RPC that returns the current root — will operate against the attacker-chosen value instead of the actual Merkle root committed on-chain. This constitutes **Data Layer root/store state corruption with direct security impact** (High scope per the rules).

Additionally, `full_puzzle_hash` and `inner_puzzle_hash` are also taken from the same untrusted conditions, so the attacker can simultaneously cause the wallet to register interest in a non-existent coin ID, permanently stalling further sync for that singleton.

---

### Likelihood Explanation

Any peer the wallet connects to can execute this attack. Wallet nodes connect to full-node peers over the standard Chia protocol; a malicious peer is a realistic adversary. No key material, admin access, or broken cryptography is required — only the ability to respond to `request_puzzle_solution` with a valid puzzle_reveal and a crafted solution.

---

### Recommendation

After `run_with_cost`, cross-validate the extracted `full_puzzle_hash` and `amount` against the actual on-chain children of the spent coin (already available via `children` in `_add_coin_states`). Specifically:

1. Confirm that a coin with `(parent=parent_name, puzzle_hash=full_puzzle_hash, amount=amount)` actually exists on-chain before writing the singleton record.
2. Derive the expected root from the child coin's `puzzle_hash` by uncurrying it with `match_dl_singleton` and reading the curried root argument — this root is cryptographically committed by the puzzle_hash and cannot be forged.
3. Use the curried root from the child's puzzle_reveal (fetched and hash-validated the same way) rather than the hint field, or at minimum assert they match.

---

# Step 3: Wallet calls singleton_removed with this CoinSpend.
#         run_with_cost produces condition[3][1] == attacker_root.

The integration test would mock the peer's `request_puzzle_solution` handler to return the crafted solution, trigger a sync cycle, and assert the mismatch between the stored root and the root derivable from the actual child coin's puzzle_hash.
