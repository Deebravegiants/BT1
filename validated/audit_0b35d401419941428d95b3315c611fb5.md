### Title
Unbounded Nested-Loop Gas Consumption in Publicly Callable `updateRSETHPrice()` Without Rate Limiting — (File: `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle.updateRSETHPrice()` is `public` with no access control beyond `whenNotPaused`. Any unprivileged caller triggers a nested loop over every supported asset × every node delegator, issuing multiple external calls per iteration, with no rate limiting, cooldown, or gas guard applied before the computation begins. This is the direct on-chain analog of the JSON-RPC report's "parsing before rate limiting" pattern: expensive work is performed unconditionally before any limiting check, and the work scales with protocol-state dimensions that are unbounded from the caller's perspective.

---

### Finding Description

`updateRSETHPrice()` at line 87 of `LRTOracle.sol` is declared `public whenNotPaused`, making it callable by any EOA or contract with no further restriction. [1](#0-0) 

It immediately delegates to `_updateRsETHPrice()`, which calls the private helper `_getTotalEthInProtocol()`: [2](#0-1) 

Inside `_getTotalEthInProtocol()`, the outer loop iterates over every entry in `lrtConfig.getSupportedAssetList()`. For each asset it calls `ILRTDepositPool.getTotalAssetDeposits(asset)`, which resolves to `getAssetDistributionData(asset)` in `LRTDepositPool.sol`: [3](#0-2) 

That inner loop makes **three external calls per node delegator** (`balanceOf`, `getAssetBalance`, `getAssetUnstaking`). The ETH path (`getETHDistributionData`) similarly loops over every NDC with three external calls each: [4](#0-3) 

The upper bound on `nodeDelegatorQueue.length` is `maxNodeDelegatorLimit`, which has **no ceiling** in its setter: [5](#0-4) 

**Gas amplification formula**: `3 × supportedAssets × nodeDelegators` external calls per invocation. At 5 assets and 10 NDCs this is 150 external calls; at 10 assets and 50 NDCs (a realistic growth trajectory) this is 1,500 external calls, approaching or exceeding the 30 M-gas block limit.

There is **no rate limiting, no per-caller cooldown, and no semaphore** applied before this computation. The price-threshold check that could revert the call (`PriceAboveDailyThreshold`) is evaluated only *after* the full nested loop completes: [6](#0-5) 

This mirrors the JSON-RPC report exactly: the expensive work (`serde_json::from_slice` / nested NDC loop) executes unconditionally before any limiting control is applied.

---

### Impact Explanation

**Medium — Unbounded gas consumption.**

As the protocol adds supported assets and node delegators through normal governance, the gas cost of `updateRSETHPrice()` grows as `O(assets × NDCs)`. Once the cost exceeds the block gas limit, no caller—including the protocol's own keeper—can successfully invoke the function. A stale `rsETHPrice` then propagates to every deposit (`getRsETHAmountToMint` reads `lrtOracle.rsETHPrice()`) and every withdrawal (`getExpectedAssetAmount` uses the oracle price), causing incorrect rsETH minting and incorrect payout calculations for all users. The protocol does not lose funds immediately, but it fails to deliver correct exchange-rate accounting, satisfying the "contract fails to deliver promised returns" Low/Medium criterion; the gas-exhaustion path itself is the explicit Medium criterion.

---

### Likelihood Explanation

The protocol already supports multiple LSTs (stETH, ETHx, sfrxETH, ETH) and multiple NDCs. `maxNodeDelegatorLimit` has no on-chain ceiling and can be raised by the LRT admin at any time as EigenLayer capacity grows. The function is already called by off-chain keepers on a regular cadence. As the protocol scales, the gas cost grows deterministically and the window for a griefing caller to push the function over the block gas limit narrows. Any unprivileged address can call `updateRSETHPrice()` at any time to consume a full block's worth of gas at the caller's expense, or to race a legitimate keeper call.

---

### Recommendation

1. **Add a per-call cooldown**: record `lastPriceUpdateTimestamp` and revert if called again within a minimum interval (e.g., 1 hour) by non-manager callers.
2. **Restrict public access**: make `updateRSETHPrice()` callable only by addresses holding a designated `PRICE_UPDATER_ROLE`, keeping `updateRSETHPriceAsManager()` for the manager override path.
3. **Cap `maxNodeDelegatorLimit`**: enforce an on-chain ceiling (e.g., 20) in `updateMaxNodeDelegatorLimit` to bound the worst-case loop cost.
4. **Cache distribution data**: consider a single aggregated TVL storage variable updated lazily rather than recomputing the full nested loop on every price update call.

### Proof of Concept

```
Attacker (any EOA):
  loop {
    call LRTOracle.updateRSETHPrice()
      → _updateRsETHPrice()
        → _getTotalEthInProtocol()
          for each asset in supportedAssets:          // e.g. 5 assets
            getTotalAssetDeposits(asset)
              getAssetDistributionData(asset)
                for each NDC in nodeDelegatorQueue:   // e.g. 10 NDCs
                  IERC20(asset).balanceOf(NDC)        // external call 1
                  NDC.getAssetBalance(asset)          // external call 2
                  NDC.getAssetUnstaking(asset)        // external call 3
          // 150 external calls total, no rate limit applied before this point
        → price threshold check (too late)
  }
```

With `maxNodeDelegatorLimit` raised to 50 and 6 supported assets: **900 external calls per invocation**, consuming ~1.9 M gas for the calls alone (cold-slot pricing), making the function uncallable within normal block gas limits and permanently freezing the rsETH price feed.
