Audit Report

## Title
Raw `address(this).balance` in Deposit-Limit Guard Allows Donation Attack to Freeze ETH Deposits — (File: `contracts/LRTDepositPool.sol`)

## Summary

`LRTDepositPool` accepts ETH from any sender via an unrestricted `receive()` fallback. `getETHDistributionData()` measures the pool's ETH using the raw `address(this).balance`, which is consumed by the deposit-limit guard in `_checkIfDepositAmountExceedesCurrentLimit()`. Because the ETH branch of that guard omits the incoming `amount` from its comparison (the deposited ETH is already in `address(this).balance` at call time), a donation of any size that pushes the raw balance above `depositLimitByAsset(ETH_TOKEN)` causes every subsequent `depositETH()` call to revert with `MaximumDepositLimitReached`. A secondary consequence is that the inflated balance propagates into `LRTOracle._getTotalEthInProtocol()`, causing excess protocol-fee rsETH to be minted to the treasury and diluting existing rsETH holders.

## Finding Description

**Root cause — unrestricted ETH ingress with raw-balance accounting:**

`LRTDepositPool` accepts ETH from any caller with no accounting side-effects:

```solidity
// contracts/LRTDepositPool.sol L58
receive() external payable { }
```

`getETHDistributionData()` reads the raw native balance as the amount of ETH in the pool:

```solidity
// contracts/LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

This value flows through `getTotalAssetDeposits()` (L385–397) and into the deposit-limit guard:

```solidity
// contracts/LRTDepositPool.sol L676-681
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

For ETH, `amount` is intentionally excluded from the comparison because `msg.value` is already reflected in `address(this).balance` at call time. This is correct for normal deposits but creates the attack surface: any ETH donated via `receive()` is indistinguishable from a legitimate deposit in the accounting, and once the raw balance exceeds the cap, the strict `>` check causes every `depositETH()` call to revert at L661–663.

**Secondary path — oracle fee inflation:**

`LRTOracle._getTotalEthInProtocol()` (L331–348) calls `getTotalAssetDeposits()` for every supported asset. The inflated ETH balance raises `totalETHInProtocol`, which is compared against `previousTVL = rsethSupply * rsETHPrice` at L244–246. The difference is treated as protocol yield, and protocol-fee rsETH is minted to the treasury proportional to the artificial reward. The `_checkAndUpdateDailyFeeMintLimit()` guard (L303) caps per-day minting but does not eliminate the impact when `maxFeeMintAmountPerDay` is set to a non-trivial value; it also causes `updateRSETHPrice()` to revert if the inflated fee exceeds the daily cap, adding a secondary griefing vector against the oracle update path.

## Impact Explanation

**Primary — Medium: Temporary freezing of funds.**
Once `address(this).balance` exceeds `depositLimitByAsset(ETH_TOKEN)`, every `depositETH()` call reverts. The deposit path is frozen for all users until an admin raises the limit via governance. This is a concrete, protocol-wide denial of service on the deposit function.

**Secondary — High: Theft of unclaimed yield.**
The donated ETH is counted as protocol yield. The oracle mints protocol-fee rsETH to the treasury proportional to the artificial reward, diluting existing rsETH holders' share of real underlying yield. The attacker permanently loses the donated ETH, but the treasury captures fee rsETH at the expense of existing holders.

## Likelihood Explanation

`receive()` is callable by any EOA or contract with no preconditions. The cost of the attack scales with the gap between current total deposits and the configured cap. When the protocol operates near its cap (a common production state for a popular LRT), the donation required to tip it over is small. The attack is immediately effective, repeatable after each admin limit increase, and requires no privileged access or victim cooperation.

## Recommendation

Replace raw-balance reads with an internal accounting variable incremented only on legitimate ETH inflows and decremented on outflows:

```solidity
// storage
uint256 public ethDepositedInPool;

// in depositETH / receiveFromNodeDelegator / receiveFromRewardReceiver / receiveFromLRTConverter
ethDepositedInPool += msg.value;

// in transferETHToNodeDelegator / transferETHToLRTUnstakingVault / swapAssetForETHWithinDepositPool
ethDepositedInPool -= amount; // or returnAmount

// in getETHDistributionData
ethLyingInDepositPool = ethDepositedInPool;
```

Apply the same pattern to `assetLyingInDepositPool` (use `mapping(address => uint256) public assetDepositedInPool`) and to the NDC and unstaking vault balances. This ensures that only ETH entering through tracked, legitimate paths is counted toward the deposit limit and TVL.

## Proof of Concept

```solidity
// Precondition: depositLimitByAsset[ETH_TOKEN] = 10_000 ether
//               getTotalAssetDeposits(ETH_TOKEN) = 9_999 ether

// Step 1: attacker donates 2 ETH directly — no deposit, no rsETH minted
(bool ok,) = address(lrtDepositPool).call{value: 2 ether}("");
require(ok);

// Step 2: address(this).balance is now 10_001 ether
// getTotalAssetDeposits(ETH_TOKEN) returns 10_001 ether
// _checkIfDepositAmountExceedesCurrentLimit: 10_001 > 10_000 → true

// Step 3: any user's depositETH() now reverts
vm.prank(user);
vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
lrtDepositPool.depositETH{value: 1 ether}(0, "");

// Step 4: oracle update treats donation as yield
// totalETHInProtocol inflated by 2 ETH
// rewardAmount = 2e18; protocolFeeInETH = 2e18 * feeBPS / 10_000
// excess rsETH minted to treasury, diluting existing holders
lrtOracle.updateRSETHPrice();
```
