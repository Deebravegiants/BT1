Audit Report

## Title
`addLiquidityWeighted` Reverts with `MinimalLiquidity` When Scaled Shares Fall Below Pool Floor — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPoolLiquidityAdder.addLiquidityWeighted` uses a probe-then-scale pattern where `_scaleWeightsToShares` multiplies weight shares by `min(max0/need0, max1/need1)`. The only guard on the scaled result is `SharesRoundedToZero` (catches `== 0`). When the scale factor is small but non-zero, scaled shares can land in `[1, MINIMAL_MINTABLE_LIQUIDITY − 1]`, causing the subsequent `addLiquidity` call to revert with `MinimalLiquidity`. The weighted liquidity path — the primary ergonomic entry point for LPs specifying token budgets — is broken for this cap range with no recovery path through the weighted interface.

## Finding Description
In `_scaleWeightsToShares` (L238–241), after computing `out.shares[i] = Math.mulDiv(w.shares[i], scaleWad, WAD)`, the only guard is:
```solidity
if (w.shares[i] != 0 && out.shares[i] == 0) revert SharesRoundedToZero();
```
Any value in `[1, MINIMAL_MINTABLE_LIQUIDITY − 1]` passes silently. The scaled `LiquidityDelta` is forwarded to `_addLiquidity`, which calls `LiquidityLib.addLiquidity`. Inside that function (L76–79):
```solidity
uint256 newUserShares = userShares + sharesToAdd;
if (newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```
This revert is caught by `_addLiquidity`'s catch block (L201–206) and re-reverted verbatim to the caller. The transient pay context is cleared, so no state is permanently corrupted, but the caller receives an opaque `MinimalLiquidity` error from the pool layer with no actionable path through `addLiquidityWeighted` for that `(max0, max1)` combination.

## Impact Explanation
The weighted liquidity path is unusable for any LP whose `maxAmountToken0`/`maxAmountToken1` caps produce a scale factor mapping weight shares into `[1, MINIMAL_MINTABLE_LIQUIDITY − 1]`. This constitutes broken core periphery liquidity functionality. No funds are lost, but the primary ergonomic LP entry point reverts with a confusing pool-level error rather than a descriptive periphery-level error, and the caller cannot recover without abandoning the weighted path entirely and computing raw shares manually via `addLiquidityExactShares`.

## Likelihood Explanation
The condition is reachable by any unprivileged LP. It occurs whenever `max0` (or `max1`) is small relative to `need0` (or `need1`) but not so small that shares round to zero. With `MINIMAL_MINTABLE_LIQUIDITY = 1000` and weight shares of `5_000_000`, a cap producing `scaleWad ≈ 0.0002e18` yields `out.shares ≈ 1000`; a cap slightly below that yields `out.shares = 999`, which reverts. This is a normal operating range for users depositing small amounts into a pool with large existing liquidity. No special privileges or adversarial setup are required.

## Recommendation
In `_scaleWeightsToShares`, after computing `out.shares[i]`, add a check that the result is either zero or at least `MINIMAL_MINTABLE_LIQUIDITY`. Read `MINIMAL_MINTABLE_LIQUIDITY` from the pool via `IMetricOmmPool(pool).getImmutables().minimalMintableLiquidity` (already called in the callback at L169) and revert with a descriptive error (e.g., `SharesBelowMinimalLiquidity(out.shares[i], minimalMintableLiquidity)`) before the paying `addLiquidity` call is attempted. This surfaces the failure at the periphery layer with an actionable error rather than letting the inner pool function revert with a confusing error.

## Proof of Concept
1. Deploy a pool with `MINIMAL_MINTABLE_LIQUIDITY = 1000`.
2. Seed the pool so that a probe with weight `[bin=4, shares=5_000_000]` returns `need0 = 500_000`.
3. Call `addLiquidityWeighted` with `maxAmountToken0 = 99` → `scaleWad = mulDiv(99, 1e18, 500_000) ≈ 198e12`.
4. `_scaleWeightsToShares` computes `out.shares[0] = mulDiv(5_000_000, 198e12, 1e18) = 990`.
5. `990 != 0` → `SharesRoundedToZero` does not fire.
6. The paying `addLiquidity` call reaches `LiquidityLib` and reverts: `MinimalLiquidity(990, 1000)`.
7. `_addLiquidity` re-reverts; caller receives `MinimalLiquidity` with no tokens moved and no actionable periphery-level error.
