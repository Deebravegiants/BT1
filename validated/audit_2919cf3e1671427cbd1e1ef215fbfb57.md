Audit Report

## Title
ETH Deposit Limit Check Omits `amount`, Allowing Deposits Beyond the Configured Cap — (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison: the ERC-20 branch correctly evaluates `totalAssetDeposits + amount > limit`, but the ETH branch evaluates only `totalAssetDeposits > limit`, ignoring the incoming deposit amount. As a result, any depositor can push ETH holdings above the configured `depositLimitByAsset` cap.

## Finding Description
In `contracts/LRTDepositPool.sol` at line 676–682, the function `_checkIfDepositAmountExceedesCurrentLimit` contains the following logic:

```solidity
// L678-681
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // missing `+ amount`
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

The ETH branch returns `true` only when the limit is **already** exceeded before the current call. When `totalAssetDeposits == depositLimit` (the cap is exactly met), the expression evaluates to `false`, so `_beforeDeposit` (L661–663) does not revert, and the deposit proceeds — pushing total ETH deposits above the cap.

The public entry point `depositETH` (L76–93) passes `msg.value` as `depositAmount` to `_beforeDeposit`, which forwards it to this check. No other guard exists for the ETH deposit limit.

## Impact Explanation
The `depositLimitByAsset` cap is the protocol's primary risk-management control for ETH exposure (EigenLayer strategy capacity, slashing risk). Bypassing it causes the protocol to silently accept more ETH than its risk parameters allow, violating the invariant `getTotalAssetDeposits(ETH) ≤ depositLimit`. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`depositETH` is public, requires no special role, and is callable by any user. The vulnerable condition (`totalAssetDeposits == depositLimit`) is a normal operational state reached during regular protocol use. Once the cap is exactly met, every subsequent ETH deposit bypasses the limit check. Likelihood is **High**.

## Recommendation
Apply the same `+ amount` pattern to the ETH branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Legitimate deposits bring `getTotalAssetDeposits(ETH_TOKEN)` to exactly `1000 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false`; no revert.
5. `_mintRsETH` executes; `getTotalAssetDeposits(ETH_TOKEN)` becomes `1500 ether` — 50% above the cap.
6. Repeat indefinitely; the cap is permanently ineffective for ETH.

**Foundry test sketch:**
```solidity
function test_ethDepositExceedsLimit() public {
    uint256 limit = 1000 ether;
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, limit);
    // fill to exactly the limit via prior deposits
    depositPool.depositETH{value: limit}(0, "");
    // next deposit should revert but does not
    depositPool.depositETH{value: 500 ether}(0, ""); // passes — bug confirmed
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), limit);
}
```
