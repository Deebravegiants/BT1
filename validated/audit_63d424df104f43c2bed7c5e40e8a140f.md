### Title
Unauthenticated P2P Peer Can Inject Arbitrary Gas Prices into Block Header Storage, Corrupting RPC Fee Estimation — (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

The p2p sync client's `parse_data_for_block` for `SignedBlockHeader` performs only two checks before accepting a header from a low-trust peer: block number ordering and signature vector length. It performs **no cryptographic verification** of the block signature and **no validation of gas price fields**. The header is then committed verbatim to `StorageBlockHeader` via `write_to_storage`. The RPC `estimate_fee` and `simulate_transactions` paths read `l1_gas_price`, `l1_data_gas_price`, and `l2_gas_price` directly from that stored header to construct the `BlockContext` used for fee calculation, so any value injected by the peer flows through to RPC responses.

---

### Finding Description

**Step 1 — Attacker entry point: unauthenticated p2p peer**

`parse_data_for_block` in `crates/apollo_p2p_sync/src/client/header.rs` receives a `SignedBlockHeader` from any peer on the p2p network. The only guards are:

```rust
if block_number != signed_block_header.block_header.block_header_without_hash.block_number { … }
if signed_block_header.signatures.len() != ALLOWED_SIGNATURES_LENGTH { … }
``` [1](#0-0) 

There is no call to `verify_block_signature` or any check against a `SequencerPublicKey`. A grep for `verify_block_signature`, `sequencer_pub_key`, or `SequencerPublicKey` across the entire `crates/apollo_p2p_sync/` tree returns zero matches. The gas price fields (`l1_gas_price`, `l1_data_gas_price`, `l2_gas_price`) are decoded from protobuf as raw `u128` values with no range or sanity checks. [2](#0-1) 

**Step 2 — Unconditional write to storage**

`write_to_storage` calls `append_header` with the full, unvalidated `BlockHeader`, including the attacker-controlled gas price fields: [3](#0-2) 

`append_header` copies `l1_gas_price`, `l1_data_gas_price`, and `l2_gas_price` verbatim into `StorageBlockHeader`: [4](#0-3) 

**Step 3 — RPC fee estimation reads gas prices from storage**

`create_block_context` in `crates/apollo_rpc_execution/src/lib.rs` reads the stored header and uses its gas price fields to build the `BlockContext` passed to blockifier for fee estimation: [5](#0-4) 

This `BlockContext` is used by `exec_estimate_fee` called from the RPC `estimate_fee` handler: [6](#0-5) 

**Step 4 — Why the block signature does not protect gas prices**

Even if signature verification were added, it would not protect gas prices. The `BlockSignature` is defined as `poseidon_hash(block_hash, state_diff_commitment)` — it does not commit to gas price fields: [7](#0-6) 

Gas prices are part of `BlockHeaderWithoutHash`, which is excluded from the block hash computation by design.

---

### Impact Explanation

A low-trust p2p peer can inject a `SignedBlockHeader` with `l1_gas_price.price_in_fri = GasPrice(u128::MAX)` (or any extreme value). After the header is committed to storage, any RPC call to `starknet_estimateFee` or `starknet_simulateTransactions` referencing that block number will use the injected gas price as the authoritative price for fee calculation, returning a wildly incorrect fee estimate to RPC clients. This matches the allowed impact: **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

Any node running in p2p sync mode (i.e., with `p2p_sync_client_config` enabled rather than `central_sync_client_config`) is exposed. The attacker only needs to be a peer on the p2p network — no special privileges are required. The attack is trivially constructable by crafting a valid protobuf `SignedBlockHeader` message with the correct `block_number` and a single-element `signatures` vector (content irrelevant, since it is never verified).

---

### Recommendation

1. **Verify the block signature** against the known sequencer public key before calling `write_to_storage`. The `verify_block_signature` function already exists in `starknet_api::block`; it should be called in `parse_data_for_block` or `write_to_storage`.
2. **Validate gas price ranges** — reject headers with zero or implausibly extreme gas prices.
3. **Verify the block hash** matches the header contents before storage, so that any tampered field (including gas prices) causes rejection.

---

### Proof of Concept

1. Connect a malicious peer to a node running in p2p sync mode.
2. When the node sends a `BlockHeadersRequest` for block N, respond with a `SignedBlockHeader` where:
   - `number = N` (passes the block number check)
   - `signatures = [<any single ConsensusSignature>]` (passes the length check)
   - `l1_gas_price_fri = Uint128 { low: u64::MAX, high: u64::MAX }` (i.e., `u128::MAX`)
3. Observe that `write_to_storage` commits the header without error.
4. Call `starknet_estimateFee` with `block_id = { block_number: N }`.
5. Assert that the returned `l1_gas_price` in the `FEE_ESTIMATE` response equals `u128::MAX` and that `overall_fee` is correspondingly inflated.
