### Title
`directDeposits` Amounts Not Included in Value Conservation Equation Allows Unbounded ADA Creation - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in the transaction body that allows ADA to be deposited directly into staking accounts. However, these amounts are applied to account balances by the `ENTITIES` rule without being included in the `consumed`/`produced` value conservation equation enforced by the `UTXO` rule. An unprivileged transaction submitter can declare arbitrary `directDeposits` amounts, causing the ledger to credit those amounts to target accounts without any corresponding debit from the UTxO — effectively creating ADA from thin air.

---

### Finding Description

The Dijkstra era adds `dtbrDirectDeposits :: !DirectDeposits` to `DijkstraTxBodyRaw TopTx` and `dstbrDirectDeposits :: !DirectDeposits` to `DijkstraTxBodyRaw SubTx`. [1](#0-0) [2](#0-1) 

The `UTXO` transition rule enforces value conservation via:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [3](#0-2) 

This calls `consumed` and `produced` from the `EraUTxO` instance for `DijkstraEra`:

```haskell
consumed = conwayConsumed
getConsumedValue = getConsumedDijkstraValue
getProducedValue = getProducedDijkstraValue
``` [4](#0-3) 

`getConsumedDijkstraValue` delegates to `getConsumedMaryValue` for each tx body (top-level and sub-transactions): [5](#0-4) 

`getConsumedMaryValue` computes: UTxO inputs + minted multi-assets + refunds + withdrawals — **no `directDeposits`**. [6](#0-5) 

`dijkstraProducedValue` computes: UTxO outputs + fee + certificate deposits + treasury donation (via `conwayProducedValue`) — **no `directDeposits`**. [7](#0-6) 

After the `UTXO` rule passes, the `ENTITIES` rule applies the direct deposits unconditionally to account balances:

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts

pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [8](#0-7) 

The only check is `directDepositsMissingAccounts` — verifying that target accounts exist — not that the amounts are funded. `applyDirectDeposits` then unconditionally adds the declared amounts to account balances: [9](#0-8) 

The same gap exists for sub-transactions in `dijkstraSubEntitiesTransition`: [10](#0-9) 

**The accounting invariant that must hold for value preservation is:**

```
UTxO_inputs + withdrawals + refunds = UTxO_outputs + fee + deposits + directDeposits
```

**But the check only enforces:**

```
UTxO_inputs + withdrawals + refunds = UTxO_outputs + fee + deposits
```

The `directDeposits` term is missing from the produced side. The ledger state therefore increases in total ADA by exactly `sum(directDeposits)` for every transaction that includes a non-empty `directDeposits` map.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker submits a Dijkstra-era transaction with:
- Minimal UTxO inputs (just enough to cover outputs + fee)
- `directDeposits = {victimOrAttackerAccount: N lovelace}`

The `UTXO` rule's value conservation check passes because `directDeposits` is absent from `produced`. The `ENTITIES` rule then credits `N` lovelace to the target account. The attacker can repeat this to mint unbounded ADA, then withdraw it via the normal withdrawal mechanism. This violates the fundamental preservation-of-value property of the Cardano ledger.

---

### Likelihood Explanation

**Low** — The Dijkstra era is not yet deployed on mainnet. Exploitation requires:
1. Dijkstra era activation.
2. Knowledge of the missing accounting term.
3. A registered target account (enforced by `directDepositsMissingAccounts`).

No privileged access, governance majority, or key compromise is required. Any transaction submitter can trigger this.

---

### Recommendation

Include the total `directDeposits` amount on the **produced** side of the value conservation equation. In `dijkstraProducedValue` (and the analogous sub-transaction function), add:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> Val.inject (foldMap id . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
    <> foldMap'
         (\subTx -> getProducedValue pp isRegPoolId (subTx ^. bodyTxL)
                    <> Val.inject (foldMap id . unDirectDeposits $ subTx ^. bodyTxL . directDepositsTxBodyL))
         (txBody ^. subTransactionsTxBodyL)
```

This mirrors how `txfee` and `totalDeposits` are already included in `shelleyProducedValue`: [11](#0-10)

### Proof of Concept

1. Register a staking credential `C` and obtain its `AccountAddress` `A`.
2. Construct a Dijkstra-era transaction `tx` with:
   - `inputs`: one UTxO output worth exactly `fee` lovelace
   - `outputs`: empty (or a zero-value output)
   - `fee`: the minimum fee
   - `directDeposits`: `{A: 10_000_000_000}` (10,000 ADA)
3. Submit `tx`. The `UTXO` rule's `validateValueNotConservedUTxO` passes because `consumed = UTxO_input_value` and `produced = fee`, and `UTxO_input_value = fee` satisfies the equation.
4. The `ENTITIES` rule calls `applyDirectDeposits`, crediting 10,000 ADA to account `A`.
5. Submit a withdrawal transaction draining account `A` of 10,000 ADA into a UTxO output.
6. 10,000 ADA has been created from thin air. [8](#0-7) [12](#0-11)
