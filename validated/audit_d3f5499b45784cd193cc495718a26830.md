### Title
Stale `unlockedCount` seeded by `processChunk` causes permanent `unlockedWithdrawalsCount` inflation, blocking `sweepRemainingAssets` — (`contracts/utils/UnlockedWithdrawalsInitializer.sol`)

### Summary
`processChunk` accumulates `unlockedCount[asset]` by scanning `withdrawalRequests` at scan time. If users call `completeWithdrawal` between the last `processChunk` call and `finalizeInitialize2`, their requests are deleted on-chain (setting `expectedAssetAmount = 0`) but `unlockedCount[asset]` is not decremented. `finalizeInitialize2` then seeds `LRTWithdrawalManager.unlockedWithdrawalsCount[asset]` with the inflated value. After all remaining users complete, the counter never reaches zero, permanently blocking `sweepRemainingAssets`.

### Finding Description
`processChunk` iterates from `processedIndex[asset]` to `nextLockedNonce[asset]` and counts requests where `expectedAssetAmount > 0`, accumulating the result in `unlockedCount[asset]`. [7](#0-6) 

`finalizeInitialize2` reads this accumulated value and passes it directly to `initialize2`: [3](#0-2) 

Between `processChunk` and `finalizeInitialize2`, any user can call `completeWithdrawal`, which executes `_processWithdrawalCompletion`. This deletes the request (`expectedAssetAmount → 0`) but does **not** decrement `unlockedCount[asset]` in the initializer contract: [2](#0-1) 

`initialize2` then sets `unlockedWithdrawalsCount[asset]` to the inflated value. Each subsequent `completeWithdrawal` decrements it by 1, but since the starting value is too high, the counter reaches `(inflated - actual_remaining)` after all real requests are completed — never reaching 0.

`sweepRemainingAssets` checks `hasUnlockedWithdrawals(asset)` which returns `unlockedWithdrawalsCount[asset] > 0`, so it permanently reverts with `PendingWithdrawalsExist`: [5](#0-4) 

The `getUnlockedWithdrawalsCount` view function re-scans live state and would return the correct (lower) value, but `finalizeInitialize2` never calls it — it only reads the stale `unlockedCount` storage variable. [8](#0-7) 

### Impact Explanation
`sweepRemainingAssets` is permanently blocked for the affected asset. Any ETH/ETHx/stETH balance remaining in `LRTWithdrawalManager` after all real withdrawals are completed cannot be swept to the treasury. This constitutes permanent freezing of unclaimed yield.

### Likelihood Explanation
The migration is a multi-transaction process: `processChunk` may be called across multiple blocks (gas-limited chunking), and `finalizeInitialize2` is called separately by the manager. The window between the last `processChunk` and `finalizeInitialize2` can span many blocks. `completeWithdrawal` is a public, permissionless function callable by any user at any time. Any user with a pending unlocked withdrawal completing during this window triggers the bug. Given that the migration is designed for a live deployment with active users, this is a realistic and likely scenario.

### Recommendation
In `finalizeInitialize2`, replace the use of the stale `unlockedCount[asset]` with a live re-scan using `getUnlockedWithdrawalsCount(asset)`:

```solidity
function finalizeInitialize2() external onlyLRTManager onlyBeforeInitialize2 {
    if (!isAssetComplete(_ethX()) || !isAssetComplete(_stETH()) || !isAssetComplete(_eth())) {
        revert PendingWithdrawalsExist();
    }

    // Use live count instead of stale accumulated count
    uint256 countETHx = getUnlockedWithdrawalsCount(_ethX());
    uint256 countSTETH = getUnlockedWithdrawalsCount(_stETH());
    uint256 countETH = getUnlockedWithdrawalsCount(_eth());

    _withdrawalManager().initialize2(countETHx, countSTETH, countETH);
    emit Finalized(countETHx, countSTETH, countETH);
}
```

Alternatively, add a consistency check asserting `unlockedCount[asset] == getUnlockedWithdrawalsCount(asset)` before finalizing, reverting if they diverge so the operator can re-run `processChunk` to reconcile.

### Proof of Concept
```solidity
// 1. processChunk scans 10 unlocked requests → unlockedCount[ETH] = 10
initializer.processChunk(ETH, 100);
assert(initializer.unlockedCount(ETH) == 10);

// 2. 3 users complete their withdrawals (deletes their requests)
vm.prank(user1); withdrawalManager.completeWithdrawal(ETH, "");
vm.prank(user2); withdrawalManager.completeWithdrawal(ETH, "");
vm.prank(user3); withdrawalManager.completeWithdrawal(ETH, "");

// 3. finalizeInitialize2 seeds with stale count=10 (live count is 7)
vm.prank(manager); initializer.finalizeInitialize2();
assert(withdrawalManager.unlockedWithdrawalsCount(ETH) == 10); // seeded with 10, not 7

// 4. Remaining 7 users complete
for (uint i = 0; i < 7; i++) {
    vm.prank(users[i]); withdrawalManager.completeWithdrawal(ETH, "");
}

// 5. Counter is now 3, not 0 — sweepRemainingAssets permanently reverts
assert(withdrawalManager.unlockedWithdrawalsCount(ETH) == 3);
vm.expectRevert(ILRTWithdrawalManager.PendingWithdrawalsExist.selector);
withdrawalManager.sweepRemainingAssets(ETH);
```
