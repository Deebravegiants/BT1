### Title
Unbounded VLQ Decoder Loop with Silent Integer Overflow in Pointer Address Deserialization - (File: `libs/cardano-ledger-core/src/Cardano/Ledger/Address.hs`)

### Summary
`decodeVariableLengthWord64` uses Haskell's `fix` combinator to create an unbounded recursive loop with no explicit iteration cap. Unlike the sibling decoders `decodeVariableLengthWord16` (hard-capped at 3 iterations) and `decodeVariableLengthWord32` (hard-capped at 5 iterations), the Word64 variant will consume bytes indefinitely as long as the MSB continuation bit is set. This allows an attacker to craft a pointer-address payload with more than 9 continuation bytes, silently overflowing the `Word64` accumulator and producing an attacker-influenced `Ptr` value that is stored in the UTxO.

### Finding Description

`decodeVariableLengthWord64` is defined as:

```haskell
-- | This decoder is here only with the purpose of preserving old buggy behavior.
decodeVariableLengthWord64 name buf = fix (decode7BitVarLength name buf) 0
``` [1](#0-0) 

`fix` applies `decode7BitVarLength` recursively with no depth limit. Each call reads one byte; if the MSB is set it recurses. The only termination condition is running out of buffer bytes (via `guardLength`). Compare this to the bounded siblings:

- `decodeVariableLengthWord16`: `d7 (d7 d7last) 0` — exactly 3 calls max
- `decodeVariableLengthWord32`: `d7 (d7 (d7 (d7 d7last))) 0` — exactly 5 calls max [2](#0-1) 

`decodeVariableLengthWord64` is called by `decodePtrLenient`, which decodes all three fields of a `Ptr` (SlotNo, TxIx, CertIx):

```haskell
decodePtrLenient buf =
  mkPtrNormalized
    <$> decodeVariableLengthWord64 "SlotNo" buf
    <*> decodeVariableLengthWord64 "TxIx" buf
    <*> decodeVariableLengthWord64 "CertIx" buf
``` [3](#0-2) 

`decodePtrLenient` is activated whenever `isPtrLenient = True` in `decodeStakeReference`: [4](#0-3) 

This flag is set to `True` in two live code paths:

1. `fromCborBackwardsBothAddr` (protocol version < 7, pre-Babbage) — `decodeAddrStateLenientT True True`
2. `fromCborRigorousBothAddr True` (protocol version 7–8, Babbage) — `decodeAddrStateLenientT True False` [5](#0-4) 

Additionally, `decompactAddr` — called whenever a stored `CompactAddr` is expanded from the UTxO — always uses `decodeAddrStateLenientT True True`, meaning any pointer address already in the UTxO from a prior era is decoded through this path in every era including Conway: [6](#0-5) 

**Overflow mechanics:** The `Word64` accumulator shifts left 7 bits per byte. After 9 bytes with MSB set, the accumulator holds 63 bits of data. The 10th continuation byte causes `shiftL 7` to push bits past the 64-bit boundary, silently wrapping. The resulting `Word64` value is then passed to `mkPtrNormalized`:

```haskell
mkPtrNormalized slotNo txIx certIx =
  fromMaybe (Ptr (SlotNo32 0) (TxIx 0) (CertIx 0)) $ do
    slotNo32 <- integralToBounded slotNo
    txIx16   <- integralToBounded txIx
    certIx16 <- integralToBounded certIx
    pure $ Ptr (SlotNo32 slotNo32) (TxIx txIx16) (CertIx certIx16)
``` [7](#0-6) 

`mkPtrNormalized` clamps values that exceed `Word32`/`Word16` bounds to `Ptr 0 0 0`. However, a carefully chosen byte sequence can overflow the `Word64` accumulator to a value that **falls within** the valid `Word32` range, bypassing the clamp entirely and producing an attacker-chosen `Ptr`.

### Impact Explanation

A `Ptr` encodes `(slotNo, txIx, certIx)` — the location of a stake registration certificate. The ledger uses this to resolve stake delegation for reward distribution. An attacker who can place a crafted pointer address into the UTxO (via a pre-Babbage or Babbage era transaction output) can cause the decoded `Ptr` to point to an arbitrary certificate position. If that position coincidentally or deliberately corresponds to a valid stake registration, the funds in that address are delegated to the wrong pool, modifying reward distribution outside design parameters. This falls under:

> **Medium. Attacker-controlled transactions... modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters.**

### Likelihood Explanation

Pointer addresses are deprecated but still accepted in pre-Babbage and Babbage era blocks. Any node replaying historical chain history or processing Babbage-era transactions exercises this path. The `decompactAddr` path means even Conway-era nodes hit this decoder when reading pointer-address UTxO entries. Crafting the overflow requires computing a specific byte sequence — feasible offline with no privileged access. The attacker only needs to submit a transaction output to a pointer address.

### Recommendation

Apply an explicit iteration cap to `decodeVariableLengthWord64` matching the maximum number of 7-bit groups needed to represent a `Word64` (ceil(64/7) = 10 bytes). The bounded pattern already used by `decodeVariableLengthWord16` and `decodeVariableLengthWord32` should be replicated:

```haskell
-- Bounded: at most 10 iterations for Word64
decodeVariableLengthWord64 name buf = d7 (d7 (d7 (d7 (d7 (d7 (d7 (d7 (d7 (d7last))))))))) 0
  where
    d7 = decode7BitVarLength name buf
    d7last acc = decode7BitVarLength name buf (\_ -> failDecoding name "too many bytes.") acc
```

This eliminates both the unbounded iteration and the overflow, consistent with how `decodeVariableLengthWord16` and `decodeVariableLengthWord32` are implemented.

### Proof of Concept

Craft a Shelley-era pointer address (header byte `0x40` = mainnet, payment key, stake-by-pointer) with the following structure:

```
[0x40]                          -- header: pointer address, mainnet, key payment
[28 bytes]                      -- payment key hash
[0xFF 0xFF 0xFF 0xFF 0xFF       -- SlotNo: 5 continuation bytes (MSB set)
 0xFF 0xFF 0xFF 0xFF 0x7F]      -- 4 more + terminator = 10 bytes total → Word64 overflow
[0x00]                          -- TxIx: 0
[0x00]                          -- CertIx: 0
```

With 10 bytes for SlotNo where the first 9 have MSB set and data bits chosen to produce a wrapped `Word64` value in `[0, 2^32)`, `integralToBounded` succeeds and `mkPtrNormalized` constructs `Ptr <attacker_chosen_slot> 0 0` — a Ptr that was never encoded in the address bytes as intended. Submit this as a transaction output in a Babbage-era block. The UTxO entry persists and is decoded via `decompactAddr` → `decodePtrLenient` → `decodeVariableLengthWord64` on every subsequent access, including in Conway-era nodes.
