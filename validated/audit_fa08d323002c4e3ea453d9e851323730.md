### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Pending Request, Enabling Cross-Request Signature Replay by a Byzantine Participant - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` contract method verifies that a submitted ECDSA signature is valid over the submitted `payload_hash` under the MPC root public key, but **never verifies that `payload_hash` actually corresponds to the specific pending request being resolved**. A single Byzantine attested participant (below signing threshold) can replay a valid MPC threshold signature obtained for one foreign transaction as a response to a completely different pending request, causing the contract to attest to foreign chain state that was never verified for that request.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 692–754), the contract performs two checks:

1. The caller is an attested participant.
2. The submitted `signature` is a valid ECDSA signature over `response.payload_hash` under the MPC root public key.

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

After these two checks pass, the contract unconditionally resolves the pending yield for `request`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

**The contract never verifies that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload{request: pending_request.request, values: ...}))` for the specific pending request being resolved.**

The signed payload structure is:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [3](#0-2) 

The hash is computed as `SHA-256(borsh(ForeignTxSignPayload))`, which binds it to the specific `tx_id` and extracted values. However, the contract does not enforce this binding when accepting a response — it only checks that the signature is valid over *whatever hash the caller supplies*.

The `VerifyForeignTransactionResponse` contains only `payload_hash` and `signature`, not the extracted values:

```rust
pub struct VerifyForeignTransactionResponse {
    pub payload_hash: Hash256,
    pub signature: SignatureResponse,
}
``` [4](#0-3) 

Because the extracted values are not included in the response (by design, to stay within NEAR promise data limits), the contract has no way to recompute the expected hash and compare it. This creates a structural gap: any valid MPC signature over any `ForeignTxSignPayload` hash can be submitted as a response to any pending request.

**Attack path:**

1. Byzantine attested participant (below threshold) submits `verify_foreign_transaction` for a controlled foreign transaction `tx_id=A`.
2. The MPC network (including honest nodes) verifies `tx_id=A` via RPC and runs the threshold ECDSA protocol, producing a valid threshold signature `sig_A` over `hash_A = SHA-256(borsh({request: {tx_id=A, ...}, values: [real_values_A]}))`.
3. The attacker receives `{payload_hash: hash_A, signature: sig_A}` as the return value of their own request.
4. The attacker calls `respond_verify_foreign_tx(request_B, {payload_hash: hash_A, signature: sig_A})` where `request_B` is a *different* pending request (for `tx_id=B`, a transaction that may not exist, may have failed, or may have different extracted values).
5. The contract checks: `verify_ecdsa_signature(sig_A, hash_A, root_pk)` → **valid** (the signature is genuinely from the MPC network).
6. The contract resolves the pending yield for `request_B` with `{payload_hash: hash_A, signature: sig_A}`.
7. The caller of `request_B` receives a response containing a valid MPC root-key signature, but the `payload_hash` encodes `tx_id=A`'s state, not `tx_id=B`'s.

The node-side leader and follower both independently execute the foreign chain request and build the signature request:

```rust
let response_payload = self
    .execute_foreign_chain_request(
        &foreign_tx_request.request,
        foreign_tx_request.payload_version,
    )
    .await?;
let sign_request = build_signature_request(&foreign_tx_request, &response_payload)?;
``` [5](#0-4) 

The honest nodes sign the correct hash for their request. But the contract does not enforce that the submitted `payload_hash` matches what the honest nodes signed for the *specific pending request being resolved*.

---

### Impact Explanation

The primary production use case is the Omnibridge inbound flow: a bridge contract calls `verify_foreign_transaction` and, upon receiving a valid MPC-signed response, mints tokens or unlocks funds on NEAR. If the bridge contract does not independently recompute the expected `payload_hash` for its specific `tx_id` and compare it to `response.payload_hash`, it will accept the forged response and execute the bridge action for a transaction that was never verified.

This enables:
- **Minting tokens on NEAR for a foreign transaction that failed, was never submitted, or belongs to a different user.**
- **Double-spend**: the attacker's controlled `tx_id=A` is used to satisfy multiple pending requests for different `tx_id`s.
- **Cross-chain state corruption**: any application that relies on the MPC network's foreign-chain attestation without re-verifying the hash binding is vulnerable.

The design doc states: *"The response contains the hash of the sign payload, so callers can verify the signature by checking it against the expected hash they reconstruct locally."* [6](#0-5) 

This places the binding verification burden entirely on callers, but the contract does not enforce it. Any bridge contract that trusts the contract's acceptance of the response without re-verifying the hash is exploitable.

---

### Likelihood Explanation

- **Attacker requirement**: A single Byzantine attested MPC participant (below signing threshold). This is explicitly within the allowed attacker model.
- **Signature acquisition**: The attacker submits a legitimate `verify_foreign_transaction` for a transaction they control on the foreign chain. The MPC network produces a valid threshold signature. The attacker receives it as the return value of their own request — no interception or key compromise required.
- **Replay**: The attacker calls `respond_verify_foreign_tx` for a different pending request. This is a public contract method callable by any attested participant.
- **Bridge contract dependency**: The attack succeeds if the bridge contract does not re-verify the `payload_hash` binding. Given that the contract itself does not enforce this, and the design doc only recommends it, newly written bridge contracts are at risk of omitting this check.

---

### Recommendation

1. **Include extracted values in the response** and have the contract recompute and verify `payload_hash`:
   ```rust
   let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
       request: request.request.clone(),
       values: response.values.clone(),
   }).compute_msg_hash()?;
   if response.payload_hash != expected_hash {
       return Err(RespondError::PayloadHashMismatch.into());
   }
   ```
   This requires adding `values: Vec<ExtractedValue>` to `VerifyForeignTransactionResponse`.

2. **Include a request nonce or `SignatureId`** in `ForeignTxSignPayload` so that each signature is cryptographically bound to a specific request instance, preventing cross-request replay even if extracted values are not included in the response.

3. **Document the hash-binding verification as mandatory** in the contract ABI and SDK, and enforce it in the reference bridge contract implementation.

### Proof of Concept

```
1. Attacker (Byzantine attested participant) calls:
   verify_foreign_transaction({
     domain_id: ForeignTx_domain,
     request: Bitcoin({tx_id: ATTACKER_CONTROLLED_TX, confirmations: 1, extractors: [BlockHash]}),
     payload_version: V1,
   })
   → MPC network verifies ATTACKER_CONTROLLED_TX, produces:
     hash_A = SHA-256(borsh(V1({request: {tx_id: ATTACKER_CONTROLLED_TX, ...}, values: [block_hash_A]})))
     sig_A  = threshold_sign(hash_A)
   → Attacker receives {payload_hash: hash_A, signature: sig_A}

2. A victim bridge contract has a pending verify_foreign_transaction for:
   request_B = {tx_id: VICTIM_TX, confirmations: 6, extractors: [BlockHash]}
   (VICTIM_TX may be a failed or non-existent transaction)

3. Attacker calls:
   respond_verify_foreign_tx(
     request = request_B,
     response = {payload_hash: hash_A, signature: sig_A}
   )

4. Contract checks:
   verify_ecdsa_signature(sig_A, hash_A, root_pk) → OK  ✓
   (No check that hash_A corresponds to request_B)

5. Contract resolves pending yield for request_B with {payload_hash: hash_A, signature: sig_A}

6. Victim bridge contract receives response, verifies signature → valid
   If bridge does not check: hash_A == SHA-256(borsh({request: request_B.request, values: expected_values}))
   → Bridge mints tokens / unlocks funds for VICTIM_TX that was never verified
```
