Audit Report

## Title
Stale `rsETHPrice` Read Without Prior Update in Deposit Functions Allows Excess rsETH Minting — (File: `contracts/LRTDepositPool.sol`)

## Summary
`depositETH` and `depositAsset` compute the rsETH mint amount via `getRsETHAmountToMint`, which divides by `lrtOracle.rsETHPrice()` — a cached state variable updated only when `_updateRsETHPrice()` is explicitly called. Neither deposit function triggers a price update before reading it. When staking rewards have accrued since the last update, the stored price is lower than the true rate, causing depositors to receive more rsETH than they are entitled to, diluting existing holders' unclaimed yield.

## Finding Description
`getRsETHAmountToMint` at `LRTDepositPool.sol:520` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` is a storage variable in `LRTOracle` written only at `LRTOracle.sol:313` inside `_updateRsETHPrice()`. The public entry point `updateRSETHPrice()` at `LRTOracle.sol:87` is not called by either `depositETH` (`LRTDepositPool.sol:87`) or `depositAsset` (`LRTDepositPool.sol:111`) before invoking `_beforeDeposit` → `getRsETHAmountToMint`.

As staking rewards accrue, `_getTotalEthInProtocol()` grows, but `rsETHPrice` remains at its last-written (lower) value. A depositor who deposits D ETH while the price is stale (P₀ < P₁) receives `D/P₀` rsETH instead of the correct `D/P₁`. The fee mechanism in `_updateRsETHPrice()` (`LRTOracle.sol:244–250`) does not mitigate this: `previousTVL` is computed as `rsethSupply * rsETHPrice`, so the attacker's deposit increases both `rsethSupply` and `totalETHInProtocol` by the same proportional amount, leaving `rewardAmount = TVL₁ - S*P₀` unchanged. After the keeper calls `updateRSETHPrice()`, the new price is set over the inflated supply, and the attacker's excess rsETH redeems for more ETH than was deposited.

The `pricePercentageLimit` guard (`LRTOracle.sol:260–265`) can cause `updateRSETHPrice()` to revert for non-manager callers when the price increase exceeds the threshold, potentially extending the staleness window and increasing the exploitable gap.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders' accrued staking rewards are diluted by the excess rsETH minted to the attacker. After `updateRSETHPrice()` is called, the attacker's excess rsETH redeems for more underlying ETH than was deposited, extracting value that belongs to existing holders. The magnitude scales linearly with deposit size and the duration of price staleness.

## Likelihood Explanation
`updateRSETHPrice()` is a public function with no on-chain enforcement of call frequency. The protocol relies entirely on off-chain keepers. Any gap between keeper calls creates an exploitable window. An attacker can compute the true price off-chain by reading `getTotalAssetDeposits` and `rsETH.totalSupply()`, compare it to the stored `rsETHPrice`, and deposit a large amount immediately before the next keeper update. The attack is repeatable every keeper cycle, requires no special privileges, and is executable by any external caller.

## Recommendation
Call `updateRSETHPrice()` (or an internal equivalent that skips the `whenNotPaused` modifier) at the start of `depositETH` and `depositAsset` before computing the mint amount, ensuring `rsETHPrice` reflects the current TVL before any rsETH is issued. Alternatively, compute the mint amount on-the-fly from live TVL and supply rather than from the cached `rsETHPrice`.

## Proof of Concept
1. At block T₀, keeper calls `updateRSETHPrice()`; `rsETHPrice` is set to P₀ = 1.01e18.
2. Staking rewards accrue. At block T₁ (e.g., 24 hours later), true rate is P₁ = 1.012e18, but `rsETHPrice` still stores P₀.
3. Attacker calls `depositETH{value: 1000 ether}(minRSETH, "")`.
4. `getRsETHAmountToMint` returns `1000e18 * 1e18 / P₀ ≈ 990.099e18` rsETH instead of the correct `1000e18 * 1e18 / P₁ ≈ 988.142e18` rsETH — an excess of ≈1.957 rsETH.
5. Keeper calls `updateRSETHPrice()`. New price is computed over the inflated supply; attacker's excess rsETH now represents a claim on ≈1.957 × P₁ ≈ 1.98 ETH of value extracted from existing holders.
6. Foundry fork test: fork mainnet, advance time by 24 hours without calling `updateRSETHPrice()`, call `depositETH` with a large value, then call `updateRSETHPrice()`, and assert that the attacker's rsETH balance redeems for strictly more ETH than deposited while existing holders' per-rsETH ETH entitlement has decreased.
