### Title
Irremovable Address-to-Account Mapping in `AddressRegistrar` Enables Permanent Fund Misdirection via Stale Registration - (File: `runtime/near-wallet-contract/implementation/address-registrar/src/lib.rs`)

### Summary

The `AddressRegistrar` contract (`address-map.near`) exposes a `register` method that permanently inserts entries into a `LookupMap<Address, AccountId>` with no corresponding removal method. Additionally, `register` does not verify that the caller owns or controls the registered `account_id`. The `WalletContract` relies on this registrar to route ETH-style transfers: when a relayer submits `rlp_execute` with a target that is an ETH-implicit account, the wallet contract queries the registrar to determine whether a named NEAR account should receive the transfer instead. A stale or attacker-planted registration permanently corrupts this routing decision, enabling fund misdirection to an attacker-controlled account.

### Finding Description

The `AddressRegistrar.register` function: [1](#0-0) 

- Accepts any `account_id` from any caller (no ownership check on `env::predecessor_account_id()`)
- Computes `address = keccak256(account_id)[12..32]` and inserts `address → account_id` into the `LookupMap`
- Has no `unregister`, `remove`, or update method anywhere in the contract [2](#0-1) 

The `WalletContract.inner_rlp_execute` function calls `address_registrar.lookup(address)` when the relayer routes to an ETH-implicit account: [3](#0-2) 

The callback `address_check_callback` then uses the registrar result to decide routing: [4](#0-3) 

If `maybe_account_id.is_some()`, the relayer is penalized and the transaction fails. If `maybe_account_id.is_none()`, the transfer proceeds to the ETH-implicit account. An honest relayer who sees `Some(named_account)` in the registrar will route to `named_account` directly (bypassing the ETH-implicit path), causing the transfer to land on whatever account currently holds that name.

The registrar account ID is hardcoded: [5](#0-4) 

### Impact Explanation

**Concrete corrupted value:** The `lookup` RPC result returned by `address-map.near` for a given 20-byte address, which directly determines where ETH-style NEAR transfers are sent.

**Attack path:**

1. Attacker calls `register("victim.near")` on `address-map.near`, paying the small storage deposit. No ownership check prevents this. The mapping `keccak256("victim.near")[12..32] → victim.near` is permanently written.
2. `victim.near` is later deleted (e.g., the account owner migrates to a new name, or storage rent forces deletion).
3. Attacker recreates `victim.near` (NEAR allows account name reuse after deletion).
4. An ETH user signs a transaction with `tx.to = keccak256("victim.near")[12..32]`.
5. An honest relayer queries the registrar, receives `Some("victim.near")`, and routes `target = "victim.near"`.
6. The wallet contract validates that `keccak256("victim.near")[12..32]` matches `tx.to` ✓, and executes `Promise::new("victim.near").transfer(deposit)`.
7. Funds land in the attacker's recreated `victim.near`.

The deposit paid by the original registrant is also permanently locked since there is no refund path.

### Likelihood Explanation

- **Entry barrier:** Low. Any unprivileged account can call `register` with any `account_id` for a small NEAR deposit (~320 yoctoNEAR per byte of storage).
- **Account deletion:** NEAR accounts can be deleted via `DeleteAccount` action. Popular contracts are sometimes migrated or deprecated.
- **Account recreation:** NEAR allows recreation of deleted account names. An attacker monitoring the chain can detect deletion and immediately recreate the name.
- **Relayer behavior:** Honest relayers are designed to consult the registrar and route to named accounts when found — this is the intended protocol behavior, making the misdirection automatic.

### Recommendation

1. Add an `unregister(account_id: AccountId)` method that allows only the currently registered account (verified via `env::predecessor_account_id() == stored_account_id`) to remove its own entry and reclaim the storage deposit.
2. Add an ownership check in `register`: require `env::predecessor_account_id() == account_id` so only the account itself can register its address.
3. Consider adding a validity check in `address_check_callback` that verifies the named account returned by the registrar actually exists on-chain before routing to it.

    pub fn register(&mut self, account_id: AccountId) -> Option<String> {
        // It is not allowed to register eth-implicit accounts because the purpose
        // of the registry is to allow looking up the named account associated with
        // an address obtained via hashing, but eth-implicit accounts are already
        // parsable as addresses.
        if is_eth_implicit(&account_id) {
            let log_message = format!("Refuse to register eth-implicit account {account_id}");
            env::log_str(&log_message);
            return None;
        }

        // Must store the address and the account id
        let bytes_to_store = 20 + (account_id.len() as u128);
        let required_deposit =
            NearToken::from_yoctonear(env::storage_byte_cost().as_yoctonear() * bytes_to_store);
        let given_deposit = env::attached_deposit();
        // The caller must pay for the storage cost of registering.
        if given_deposit < required_deposit {
            let message = format!(
                "Insufficient deposit to cover storage cost. Given={} Expected={}",
                given_deposit.as_yoctonear(),
                required_deposit.as_yoctonear(),
            );
            env::panic_str(&message);
        }

        let address = account_id_to_address(&account_id);

        match self.addresses.entry(address) {
            Entry::Vacant(entry) => {
                let address = format!("0x{}", hex::encode(address));
                let log_message = format!("Added entry {} -> {}", address, account_id);
                entry.insert(account_id);
                env::log_str(&log_message);
                Some(address)
            }
            Entry::Occupied(entry) => {
                let log_message = format!(
                    "Address collision between {} and {}. Keeping the former.",
                    entry.get(),
                    account_id
                );
                env::log_str(&log_message);
                // Transfer the deposit back to the caller since no storage was updated.
                let refund_promise = env::promise_batch_create(&env::predecessor_account_id());
                env::promise_batch_action_transfer(refund_promise, given_deposit);
                None
            }
        }
    }
