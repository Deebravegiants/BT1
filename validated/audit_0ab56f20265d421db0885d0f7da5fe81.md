### Title
Missing Output Integrity Check in `respond_ckd()` for `AppPublicKey` Variant Allows Malicious Participant to Corrupt CKD Response - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd()` enforces a cryptographic pairing-based output check (`ckd_output_check`) only when the request carries `CKDAppPublicKey::AppPublicKeyPV`. When the request carries `CKDAppPublicKey::AppPublicKey`, the branch is a no-op and the response is accepted unconditionally. A single Byzantine attested participant can therefore call `respond_ckd` with an arbitrary `CKDResponse`, resolve the pending yield with garbage or attacker-chosen BLS12-381 points, and permanently consume the user's request without delivering a correct derived key.

### Finding Description

In `respond_ckd()`, after verifying the caller is an attested participant and the protocol is active, the contract branches on the request's `app_public_key` variant:

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which proves the ciphertext is a well-formed ElGamal encryption of the correct derived key share under the user's public key:

```rust

``` [2](#0-1) 

No equivalent check exists for `AppPublicKey`. The `CKDResponse` fields `big_y` and `big_c` are accepted as-is and forwarded to the user via `resolve_yields_for`:

```rust

    &request,
    serde_json::to_vec(&response).unwrap(),

``` [3](#0-2) 

The `AppPublicKey` variant is the standard (non-publicly-verifiable) CKD mode used by default in the example CLI:

```rust
// crates/ckd-example-cli/src/ckd.rs  lines 31-33
let (scalar, pk) = generate_ephemeral_key(&mut OsRng);
(scalar, CKDAppPublicKey::AppPublicKey(pk))
``` [4](#0-3) 

**Attack path:**

1. A Byzantine attested participant (below the signing threshold) observes a pending `AppPublicKey` CKD request in `pending_ckd_requests`.
2. It calls `respond_ckd(request, CKDResponse { big_y: <attacker_chosen>, big_c: <attacker_chosen> })`.
3. The contract passes all checks (caller is attested participant, protocol is running) and skips the output check because `app_public_key` is `AppPublicKey`.
4. The yield is resolved; the user's callback fires with the attacker-supplied `{big_y, big_c}`.
5. The request is permanently consumed from `pending_ckd_requests`.

In the most damaging form: the attacker sets `big_y` to the BLS12-381 G1 identity element (point at infinity). The user's decryption then reduces to `big_s = big_c − pk_private · identity = big_c`. If the attacker sets `big_c = G1 · known_scalar`, the user's derived key material becomes `G1 · known_scalar`, a value whose discrete log the attacker knows — enabling key material theft.

### Impact Explanation

**Medium** (request/accounting/state corruption breaking production safety invariants).

The correctness invariant of the CKD protocol — that the returned ciphertext is a valid encryption of the MPC-derived key share — is entirely unenforced for the `AppPublicKey` path. A single Byzantine participant can:

- Deliver a corrupted or attacker-controlled derived key to any user who submitted an `AppPublicKey` CKD request.
- Permanently consume the request so the user cannot retry.
- In the worst case (identity-point attack), learn the discrete log of the user's derived key material, constituting key material disclosure.

This maps to the allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

**Medium.** The attacker must be a single attested MPC participant — a Byzantine node below the signing threshold. This is explicitly within the MPC threat model. The `AppPublicKey` variant is the default mode (used by the example CLI and likely by most integrators), so the vulnerable code path is exercised in normal production use. No threshold collusion, leaked keys, or privileged operator access is required.

### Recommendation

Apply the same cryptographic output check to `AppPublicKey` requests. Because `AppPublicKey` does not carry a G2 component (`pk2`), the existing `ckd_output_check` pairing equation cannot be used directly. Options:

1. **Require `AppPublicKeyPV` for all on-chain CKD requests**, deprecating the unverifiable `AppPublicKey` variant.
2. **Add a G2 component to `AppPublicKey`** (i.e., require callers to supply both `pk1` and `pk2 = g2 · sk`) so the same pairing check can be applied, making all CKD responses publicly verifiable on-chain.
3. **At minimum**, document that `AppPublicKey` responses are unverified on-chain and that callers must perform client-side verification before trusting the derived key.

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}

                    env::panic_str("CKD output check failed");
                }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
