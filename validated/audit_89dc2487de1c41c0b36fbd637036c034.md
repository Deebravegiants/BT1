Audit Report

## Title
Stale `processChunk` Counts Permanently Over-Seed `unlockedWithdrawalsCount` After `initialize2` — (`contracts/utils/UnlockedWithdrawalsInitializer.sol`)

## Summary

`processChunk` accumulates unlocked-withdrawal counts into `unlockedCount[asset]` and permanently advances `processedIndex[asset]` past scanned indices. If `completeWithdrawal` executes between a `processChunk` call for one asset and `finalizeInitialize2`, the already-scanned indices are never re-examined. `finalizeInitialize2` passes the stale `unlockedCount` values directly to `initialize2`, which overwrites the live `unlockedWithdrawalsCount` in `LRTWithdrawalManager` with an inflated figure. The result is a permanently inflated counter that blocks `sweepRemainingAssets` for the affected asset.

## Finding Description

**Root cause — `processChunk` locks in a count that can become stale.**

For each index in `[processedIndex[asset], nextLockedNonce[asset])`, `processChunk` reads `withdrawalRequests[requestId].expectedAssetAmount`. If non-zero, the request is counted as unlocked. The result is added to `unlockedCount[asset]` and `processedIndex[asset]` is advanced past those indices permanently — they will never be re-scanned. [1](#0-0) 

**`completeWithdrawal` deletes request data after `processChunk` has already counted it.**

`_processWithdrawalCompletion` calls `delete withdrawalRequests[requestId]`, zeroing `expectedAssetAmount`, then decrements the live `unlockedWithdrawalsCount[asset]`. [2](#0-1) 

**`isAssetComplete` does not detect staleness.**

It only checks `processedIndex[asset] >= nextLockedNonce[asset]`. Completions do not change `nextLockedNonce`, so the check passes even when `unlockedCount[asset]` is stale. [3](#0-2) 

**`finalizeInitialize2` uses the stale accumulated count, not a live scan.**

The contract already exposes `getUnlockedWithdrawalsCount` (lines 128–137) which performs a fresh on-chain scan, but `finalizeInitialize2` ignores it and passes `unlockedCount[asset]` directly. [4](#0-3) 

**`initialize2` overwrites the live counter with the stale value.**

The assignment (`=`) replaces whatever the live `unlockedWithdrawalsCount` was, discarding any decrements that occurred between `processChunk` and `finalizeInitialize2`. [5](#0-4) 

**`sweepRemainingAssets` is permanently blocked.**

After all legitimate completions decrement the inflated counter down to the inflation amount, no unlocked requests remain but `unlockedWithdrawalsCount[asset] > 0`. `sweepRemainingAssets` reverts on `hasUnlockedWithdrawals` forever. [6](#0-5) 

**Why existing guards are insufficient.**

`onlyBeforeInitialize2` only prevents `processChunk` from running after `initialize2` is called — it does not prevent `completeWithdrawal` from running between `processChunk` calls. `completeWithdrawal` is gated only by `nonReentrant` and `whenNotPaused`, both of which are satisfied during normal operation. [7](#0-6) 

## Impact Explanation

`sweepRemainingAssets` for the affected asset is permanently blocked. Any residual balance in `LRTWithdrawalManager` (dust from rounding or over-funded unlocks) can never be recovered to the treasury. No user withdrawal funds are lost. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation

The precondition is that `unlockQueue` is called after the upgrade but before `initialize2` (so that `unlockedWithdrawalsCount > 0`, allowing `completeWithdrawal` to succeed without an arithmetic underflow). This is a normal operational scenario: the initializer is explicitly designed to run in chunks across multiple transactions, and the protocol is expected to continue unlocking new requests during that window. No attacker action is required; ordinary user `completeWithdrawal` calls are sufficient. The one-time nature of `initialize2` (`reinitializer(2)`) makes the miscounting permanent. [8](#0-7) 

## Recommendation

Replace the stale `unlockedCount[asset]` values in `finalizeInitialize2` with a fresh call to `getUnlockedWithdrawalsCount` at finalization time:

```solidity
function finalizeInitialize2() external onlyLRTManager onlyBeforeInitialize2 {
    if (!isAssetComplete(_ethX()) || !isAssetComplete(_stETH()) || !isAssetComplete(_eth())) {
        revert PendingWithdrawalsExist();
    }

    uint256 countETHx  = getUnlockedWithdrawalsCount(_ethX());
    uint256 countSTETH = getUnlockedWithdrawalsCount(_stETH());
    uint256 countETH   = getUnlockedWithdrawalsCount(_eth());

    _withdrawalManager().initialize2(countETHx, countSTETH, countETH);
    emit Finalized(countETHx, countSTETH, countETH);
}
```

This makes `processChunk` / `processedIndex` serve only as a completeness gate (ensuring the full queue has been scanned at least once) while the actual seeded values always reflect on-chain state at the moment of finalization. [9](#0-8) 

## Proof of Concept

```
Setup (fork or local):
1. Deploy LRTWithdrawalManager + UnlockedWithdrawalsInitializer (upgrade to v2).
   → unlockedWithdrawalsCount[stETH] = 0; N pre-existing unlocked stETH requests exist.

2. Call unlockQueue(stETH, ...) to unlock M additional stETH requests.
   → nextLockedNonce[stETH] = N+M, unlockedWithdrawalsCount[stETH] = M.

3. Call processChunk(stETH, N+M).
   → All N+M requests have expectedAssetAmount > 0.
   → unlockedCount[stETH] = N+M, processedIndex[stETH] = N+M.

4. Call completeWithdrawal(stETH) K times (K ≤ M, so no underflow).
   → withdrawalRequests for those K indices deleted (expectedAssetAmount = 0).
   → unlockedWithdrawalsCount[stETH] = M-K (live, correct).

5. Call processChunk(ETHx, ...) and processChunk(ETH, ...) to completion.

6. Assert isAssetComplete(stETH) == true          ← passes (processedIndex >= nextLockedNonce)
7. Assert unlockedCount[stETH] == N+M             ← stale (K completions not reflected)
8. Assert getUnlockedWithdrawalsCount(stETH) == N+M-K  ← live, correct

9. Call finalizeInitialize2().
   → initialize2 seeds unlockedWithdrawalsCount[stETH] = N+M (not N+M-K).

10. Complete the remaining N+M-K stETH withdrawals.
    → unlockedWithdrawalsCount[stETH] = (N+M) - (N+M-K) = K.

11. Assert hasUnlockedWithdrawals(stETH) == true  ← BUG: no requests remain but counter = K > 0
12. sweepRemainingAssets(stETH) → reverts with PendingWithdrawalsExist  ← permanently blocked
```
