Audit Report

## Title
ETH Transfer to Contract Recipient Without `receive` Permanently Freezes User Funds After rsETH Burn - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._transferAsset` delivers ETH via `payable(to).call{value: amount}("")`. If the recipient is a smart contract without a `receive` or `fallback` function, this call returns `false` and the function reverts with `EthTransferFailed`. Because rsETH is irreversibly burned in a prior `unlockQueue` transaction, the user's rsETH is permanently destroyed while the corresponding ETH allocation is permanently frozen inside `LRTWithdrawalManager` with no recovery path.

## Finding Description
The ETH disbursement primitive is `_transferAsset` at L876-883:

```solidity
function _transferAsset(address asset, address to, uint256 amount) internal {
    if (asset == LRTConstants.ETH_TOKEN) {
        (bool sent,) = payable(to).call{ value: amount }("");
        if (!sent) revert EthTransferFailed();
    } else {
        IERC20(asset).safeTransfer(to, amount);
    }
}
```

It is called at L734 inside `_processWithdrawalCompletion`. The withdrawal lifecycle spans two separate, non-atomic transactions:

1. **`initiateWithdrawal`** (L150-178): user's rsETH is pulled into `LRTWithdrawalManager`; a `WithdrawalRequest` is recorded.
2. **`unlockQueue`** (L305): `IRSETH.burnFrom` permanently destroys the rsETH held by the contract; ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager`.
3. **`completeWithdrawal` / `completeWithdrawalForUser`** (L183-204): calls `_processWithdrawalCompletion`, which attempts to push ETH to `user`.

In step 3, `_processWithdrawalCompletion` (L699-738) performs state mutations — `popFront()` at L705, `delete withdrawalRequests[requestId]` at L712, `unlockedWithdrawalsCount[asset]--` at L717 — before calling `_transferAsset` at L734. If `_transferAsset` reverts, all those mutations are rolled back, leaving the request intact in the queue. However, the rsETH burned in step 2 is not restored; that state change occurred in a separate committed transaction.

Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` for this user will revert identically, because the `user` address is hardcoded in the stored `WithdrawalRequest` and neither function accepts an alternative recipient. The NatSpec comment at L191 (`/// @dev Not expected to be used for ETH`) confirms the developers did not design `completeWithdrawalForUser` as an ETH recovery path.

The only other potential escape, `sweepRemainingAssets` (L395-414), is gated by `if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist()` at L403. Because `unlockedWithdrawalsCount[asset]` is never decremented (every attempt reverts), this check permanently blocks the sweep. There is no admin function to redirect a locked withdrawal to an alternative address.

## Impact Explanation
**Critical — Permanent freezing of funds.** The user's rsETH is irreversibly burned in `unlockQueue`. The corresponding ETH sits in `LRTWithdrawalManager` but can never be delivered to the user or recovered by the protocol. This matches the allowed impact class "Permanent freezing of funds."

## Likelihood Explanation
**Medium.** Smart-contract wallets, protocol treasuries, EigenLayer operator contracts, and DeFi vaults routinely hold and withdraw rsETH. Any such contract that does not implement `receive() external payable` will trigger this freeze upon attempting an ETH withdrawal. The condition is self-inflicted by the victim contract's design, but the pattern is common enough in production DeFi that the likelihood is non-trivial. No privileged role or attacker is required; the victim contract itself triggers the freeze by calling `initiateWithdrawal`.

## Recommendation
Replace the push-ETH pattern with a pull pattern: record the owed ETH amount per user in a mapping and expose a separate `claimETH()` function that the user calls to withdraw. Alternatively, wrap ETH into WETH before transferring (`IWETH.deposit{value: amount}(); IERC20(WETH).safeTransfer(to, amount)`), which always succeeds regardless of the recipient's fallback implementation.

## Proof of Concept
1. Deploy `VaultUser` — a contract with no `receive` function — that holds rsETH.
2. `VaultUser` calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`; a `WithdrawalRequest` is stored.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. At L305, `IRSETH.burnFrom` permanently destroys the rsETH. At L307, ETH is redeemed into `LRTWithdrawalManager`.
4. `VaultUser` calls `completeWithdrawal(ETH_TOKEN, "")`. Inside `_processWithdrawalCompletion`, `_transferAsset` executes `payable(VaultUser).call{value: amount}("")`. Because `VaultUser` has no `receive`, the call returns `false`; `EthTransferFailed` is thrown; the transaction reverts. All state mutations in `_processWithdrawalCompletion` are rolled back.
5. Step 4 can be repeated indefinitely — it always reverts. `VaultUser`'s rsETH is permanently burned and the ETH is permanently locked in `LRTWithdrawalManager`. `sweepRemainingAssets` is also permanently blocked because `unlockedWithdrawalsCount[ETH_TOKEN] > 0`.

**Foundry test sketch:**
```solidity
contract NoReceive {
    function initiateWithdrawal(address wm, address eth, uint256 amt) external {
        ILRTWithdrawalManager(wm).initiateWithdrawal(eth, amt, "");
    }
    function completeWithdrawal(address wm, address eth) external {
        ILRTWithdrawalManager(wm).completeWithdrawal(eth, "");
    }
    // No receive() function
}

function test_permanentFreeze() public fork {
    NoReceive vault = new NoReceive();
    deal(address(rsETH), address(vault), 1 ether);
    vm.prank(address(vault));
    rsETH.approve(address(withdrawalManager), 1 ether);
    vault.initiateWithdrawal(address(withdrawalManager), ETH_TOKEN, 1 ether);
    // operator unlocks
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, ...);
    // complete always reverts
    vm.expectRevert(ILRTWithdrawalManager.EthTransferFailed.selector);
    vault.completeWithdrawal(address(withdrawalManager), ETH_TOKEN);
    // ETH locked, rsETH gone
    assertGt(address(withdrawalManager).balance, 0);
    assertEq(rsETH.balanceOf(address(withdrawalManager)), 0);
}
```
