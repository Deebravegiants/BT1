### Title
Cross-Request Replay of `respond_verify_foreign_tx` Enables Forged Foreign-Chain Verification - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over the caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` is the correct hash for the `request` being resolved. A single malicious attested MPC node (below the signing threshold) can replay a previously observed, legitimately produced `(payload_hash, signature)` pair against any currently pending `verify_foreign_transaction` request, causing the contract to resolve that request with fabricated verification data.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract performs the following signature check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the root public key. It then resolves all pending yields for `request` with this response:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

**The missing check:** the contract never verifies that `response.payload_hash` is the correct hash for the given `request`. The `payload_hash` is a free parameter in the response struct — any valid `(payload_hash, signature)` pair signed by the MPC root key will pass the check, regardless of which request it was originally produced for.

Contrast this with the `respond` function for regular sign requests, where the payload hash is derived directly from `request.payload` (part of the request key itself), making the signature cryptographically bound to the specific request:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [3](#0-2) 

The `ForeignTxSignPayload` does embed the `request` in its hash computation:

```rust
let payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: extracted_values,
});
let payload_hash = payload.compute_msg_hash().unwrap();
``` [4](#0-3) 

So a `payload_hash_A` produced for `request_A` embeds `request_A`'s chain data — but the contract never enforces this binding when accepting a response for `request_B`.

---

### Impact Explanation

A bridge contract that calls `verify_foreign_transaction(request_B)` (e.g., "verify Bitcoin tx Y was finalized") receives a `VerifyForeignTransactionResponse` containing a valid MPC root-key signature, but over `payload_hash_A` — the hash of a completely different foreign transaction (`request_A`). The MPC contract has resolved `request_B`'s pending yield with fabricated verification data. Any downstream bridge logic that trusts the returned `(payload_hash, signature)` without independently re-deriving the expected `payload_hash` from the original request will incorrectly conclude that Bitcoin tx Y was verified, potentially releasing funds or executing bridge logic for a transaction that was never actually verified. This matches the **High** impact class: forged foreign-chain verification that causes invalid bridge execution.

---

### Likelihood Explanation

**Medium.** The attacker must be a single attested MPC node (a Byzantine participant strictly below the signing threshold). The attacker does not need to forge any cryptographic material — the valid `(payload_hash_A, signature_A)` pair is publicly visible on-chain after any prior successful `respond_verify_foreign_tx` call. The attacker only needs to:
1. Observe a past valid response on-chain (trivially available).
2. Identify a currently pending `verify_foreign_transaction` request (also publicly visible on-chain).
3. Call `respond_verify_foreign_tx(request_B, response_A)` — a single NEAR transaction.

The `assert_caller_is_attested_participant_and_protocol_active` guard is the only barrier, and it is satisfied by any single attested node. [5](#0-4) 

---

### Recommendation

In `respond_verify_foreign_tx`, verify that `response.payload_hash` is consistent with the `request` being resolved. Since `ForeignTxSignPayloadV1` includes `request.request` (the `ForeignChainRpcRequest`) in its hash, the contract can enforce a partial binding by requiring the MPC nodes to also submit the `extracted_values` alongside the response, then independently computing and comparing `H(request.request, extracted_values)` against `response.payload_hash`.

Alternatively, restructure the signing payload so that the `request` key (or its hash) is a domain-separating prefix of the signed message, allowing the contract to verify the binding without knowing the `extracted_values`. This mirrors how `respond` binds the signature to `request.payload` directly.

---

### Proof of Concept

1. A legitimate `verify_foreign_transaction(request_A)` (e.g., Ethereum tx X) is processed by the MPC network. An attested node submits `respond_verify_foreign_tx(request_A, {payload_hash_A, signature_A})` on-chain. This transaction is publicly visible.

2. A bridge contract submits `verify_foreign_transaction(request_B)` (e.g., Bitcoin tx Y), creating a pending yield in `pending_verify_foreign_tx_requests`.

3. A single malicious attested MPC node calls:
   ```
   respond_verify_foreign_tx(request_B, {payload_hash_A, signature_A})
   ```

4. The contract at line 729 verifies `verify_ecdsa_signature(signature_A, payload_hash_A, root_pk)` — this passes because `signature_A` is a genuine MPC signature over `payload_hash_A`.

5. `resolve_yields_for` at line 749 drains all pending yields for `request_B`, delivering `{payload_hash_A, signature_A}` to the bridge contract.

6. The bridge contract receives a response with a valid root-key signature, but `payload_hash_A` encodes `request_A`'s chain data (Ethereum tx X), not Bitcoin tx Y. If the bridge does not re-derive the expected `payload_hash` from `request_B`, it incorrectly concludes that Bitcoin tx Y was verified and executes the bridge action. [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
