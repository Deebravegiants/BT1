### Title
Depositors Can Exploit Stale `rsETHPrice` to Steal Accrued Yield from Existing rsETH Holders — (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTOracle.updateRSETHPrice()` is a public, permissionless function that updates the stored `rsETHPrice` state variable. Between updates, when EigenLayer rewards cause the actual protocol TVL to exceed `rsethSupply × rsETHPrice`, the stored price is stale (lower than actual). Because `LRTDepositPool` mints rsETH using this stale stored price, an attacker can deposit at the artificially low price, then trigger the price update, capturing a portion of the accrued yield that rightfully belongs to existing rsETH holders.

---

### Finding Description

`LRTOracle.updateRSETHPrice()` is declared `public` with no access restriction: [1](#0-0) 

The function updates the stored `rsETHPrice` state variable based on the current total ETH in the protocol: [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` uses this stored (potentially stale) `rsETHPrice` to calculate how many rsETH tokens to mint per deposit: [3](#0-2) 

There is no mechanism to refresh the price atomically within a deposit. The `minRSETHAmountExpected` slippage guard only protects the depositor from receiving *fewer* tokens than expected — it does not prevent the depositor from receiving *more* tokens than the actual current price warrants: [4](#0-3) 

When EigenLayer staking rewards accrue (ETH balance in NodeDelegators increases), `_getTotalEthInProtocol()` returns a value greater than `rsethSupply × rsETHPrice`, meaning the stored price is below the actual per-share value: [5](#0-4) 

An attacker who deposits during this stale-price window receives more rsETH than the actual current exchange rate warrants. After `updateRSETHPrice()` is called (by anyone, including the attacker), the price rises to reflect the accrued rewards, and the attacker's rsETH is worth more ETH than was deposited — at the expense of existing holders whose per-share value is diluted.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield.**

Existing rsETH holders earn yield as EigenLayer rewards increase the protocol's total ETH. This yield is reflected when `rsETHPrice` is updated upward. An attacker who deposits at the stale (pre-update) price dilutes the supply, capturing a fraction of the pending yield that should accrue exclusively to pre-existing holders.

Concretely: if 5 ETH in rewards has accrued on a 100-rsETH / 105-ETH base, and an attacker deposits 10 ETH at the stale price of 1.05 ETH/rsETH (receiving 9.524 rsETH), the post-update price becomes ≈1.0956 ETH/rsETH instead of 1.10 ETH/rsETH. The attacker's 9.524 rsETH is worth ≈10.44 ETH — a profit of ≈0.44 ETH extracted from existing holders, who collectively lose ≈0.44 ETH of yield they had earned.

The attack scales with the attacker's deposit size relative to TVL and with the magnitude of accrued rewards.

---

### Likelihood Explanation

**Likelihood: Low.**

The attack requires no special privileges — only the ability to monitor on-chain state (NodeDelegator ETH balances vs. stored `rsETHPrice`) and submit two transactions in order. The attacker can compute the pending price increase off-chain by simulating `_getTotalEthInProtocol()` and comparing to `rsethSupply × rsETHPrice`. The `pricePercentageLimit` guard partially mitigates the attack by reverting `updateRSETHPrice()` for non-managers when the price increase exceeds the threshold: [6](#0-5) 

However, this only forces the attacker to wait for a manager to call `updateRSETHPriceAsManager()` — the deposit at the stale price is already committed. For reward accruals within the daily threshold, the attacker can call `updateRSETHPrice()` themselves immediately after depositing. The attack is repeatable every reward cycle.

---

### Recommendation

1. **Refresh price before minting**: Call `_updateRsETHPrice()` (or an equivalent read-only computation) inside `getRsETHAmountToMint()` to use the current actual price rather than the stored stale value.
2. **Deposit fee**: Introduce a small deposit fee that exceeds the expected per-block reward accrual, making the attack economically unviable.
3. **Access-restrict price updates**: Restrict `updateRSETHPrice()` to a trusted keeper role, reducing the attacker's ability to time the update immediately after depositing.

---

### Proof of Concept

**Setup:**
- `rsETHPrice` (stored) = 1.05 ETH/rsETH
- `rsethSupply` = 100 rsETH
- `totalETHInProtocol` (actual) = 110 ETH (5 ETH rewards accrued, price not yet updated)

**Attack:**

1. Attacker calls `LRTDepositPool.depositETH{value: 10 ether}(0, "")`.
   - `getRsETHAmountToMint` computes: `10e18 * 1e18 / 1.05e18 = 9.524 rsETH` minted to attacker.
   - New `rsethSupply` = 109.524 rsETH; new `totalETH` = 120 ETH.

2. Attacker calls `LRTOracle.updateRSETHPrice()`.
   - `previousTVL = 109.524 × 1.05 = 115 ETH`
   - `rewardAmount = 120 − 115 = 5 ETH`
   - `newRsETHPrice = 120 / 109.524 ≈ 1.0956 ETH/rsETH`

3. Attacker's 9.524 rsETH is now worth `9.524 × 1.0956 ≈ 10.435 ETH`.
   - **Attacker profit: ≈ 0.435 ETH** (deposited 10 ETH, holds 10.435 ETH equivalent).
   - **Existing holders' loss**: their 100 rsETH is worth `100 × 1.0956 = 109.56 ETH` instead of the `110 ETH` they would have had without the attack.

The attack is executable in two sequential transactions with no privileged access, exploiting the gap between `LRTOracle.rsETHPrice` (stale) and the actual protocol TVL. [1](#0-0) [3](#0-2) [7](#0-6)

    function _updateRsETHPrice() internal {
        address rsETHTokenAddress = lrtConfig.rsETH();
        uint256 rsethSupply = IRSETH(rsETHTokenAddress).totalSupply();

        if (rsethSupply == 0) {
            rsETHPrice = 1 ether;
            highestRsethPrice = 1 ether;
            return;
        }

        if (highestRsethPrice == 0) {
            highestRsethPrice = rsETHPrice;
        }

        uint256 previousPrice = rsETHPrice;

        // get total ETH in the protocol (normalized to 1e18)
        uint256 totalETHInProtocol = _getTotalEthInProtocol();

        // calculate previousTVL using rsethSupply multiplied by rsETHPrice
        uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);

        IPausable lrtDepositPool = IPausable(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        IPausable withdrawalManager = IPausable(lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER));

        // determine if the protocol is active (not paused)
        bool protocolPaused = lrtDepositPool.paused() || withdrawalManager.paused() || paused;

        // only take fee if TVL increased and protocol is not paused
        uint256 protocolFeeInETH = 0;
        if (!protocolPaused && totalETHInProtocol > previousTVL) {
            uint256 rewardAmount = totalETHInProtocol - previousTVL;
            protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
        }

        // compute new rsETH price based on total ETH minus fee
        uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
