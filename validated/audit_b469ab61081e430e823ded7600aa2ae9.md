### Title
Malicious Third-Party Harvester Can Unconditionally Redirect Farmer Block Reward Due to Non-Enforced CHIP-22 Fee Quality Check - (`File: chia/farmer/farmer_api.py`)

---

### Summary

CHIP-22 introduced a convention allowing third-party harvesters to redirect the farmer's block reward to themselves as a fee, gated by a probabilistic "fee quality" check. However, the farmer's enforcement of this check is **logging-only**: when a harvester sends a `farmer_reward_address_override`, the farmer unconditionally accepts it regardless of whether the fee quality threshold is satisfied. A malicious harvester can always redirect 100% of the farmer's block reward to an arbitrary address.

---

### Finding Description

CHIP-22 defines a fee convention between farmers and third-party harvesters. When a harvester finds a valid proof of space, it may set `farmer_reward_address_override` in `RespondSignatures` to redirect the farmer's block reward to itself as a fee. The convention requires that the harvester only do this when `fee_quality <= applied_fee_threshold`, where `fee_quality = calculate_harvester_fee_quality(proof, challenge)`.

In `_process_respond_signatures()` in `chia/farmer/farmer_api.py`, when the harvester provides a `farmer_reward_address_override`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()` (which only logs a warning if the threshold is violated) and then **unconditionally** uses the override as the `farmer_reward_address` in the `DeclareProofOfSpace` message sent to the full node:

```python
# chia/farmer/farmer_api.py lines 916-919
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

The `notify_farmer_reward_taken_by_harvester_as_fee()` function in `chia/farmer/farmer.py` only emits log warnings when the fee quality check fails — it does not return a value, raise an exception, or cause the farmer to reject the override:

```python
# chia/farmer/farmer.py lines 920-928
else:
    self.log.warning(
        f"Invalid fee threshold for challenge '{challenge_str}': ..."
    )
    self.log.warning(
        "Harvester illegitimately took a fee reward that "
        + "did not belong to it or it incorrectly applied the fee convention."
    )
```

The full node at `chia/full_node/full_node_api.py` line 1069 simply uses whatever `farmer_puzzle_hash` the farmer sends, with no independent validation of the fee quality convention:

```python
farmer_ph = request.farmer_puzzle_hash
```

There is no consensus-level or protocol-level enforcement of the CHIP-22 fee quality convention. The farmer's only defense is a log message.

---

### Impact Explanation

A malicious third-party harvester can redirect the entire farmer block reward (currently 0.25 XCH per block) to any address it controls, every time the harvester finds a winning proof of space. The farmer receives nothing. This constitutes unauthorized payout redirection of XCH, a Critical/High impact per the allowed scope.

---

### Likelihood Explanation

Any farmer who connects to a third-party harvester service is exposed. The attacker needs only to operate a harvester service and set `farmer_reward_address_override` to their own address on every `RespondSignatures` message. No special privileges, leaked keys, or cryptographic breaks are required. The attacker is an unprivileged network peer (harvester) that the farmer has voluntarily connected to.

---

### Recommendation

In `_process_respond_signatures()`, after calling `notify_farmer_reward_taken_by_harvester_as_fee()`, the farmer should **reject** the `farmer_reward_address_override` (i.e., fall back to `self.farmer.farmer_target`) if the fee quality check fails. Specifically:

```python
fee_quality = calculate_harvester_fee_quality(proof_bytes, sp.challenge_hash)
if response.fee_info is None or fee_quality > response.fee_info.applied_fee_threshold:
    # Fee quality check failed: do not honor the override
    farmer_reward_address = self.farmer.farmer_target
else:
    farmer_reward_address = response.farmer_reward_address_override
```

The `notify_farmer_reward_taken_by_harvester_as_fee()` function should be refactored to return a boolean indicating whether the override is legitimate, and the caller should act on that result.

---

### Proof of Concept

1. Attacker deploys a harvester service.
2. Victim farmer connects their harvester to the attacker's service.
3. When the attacker's harvester finds a valid proof of space (winning block), it sends `RespondSignatures` with `farmer_reward_address_override` set to the attacker's puzzle hash and `fee_info` with `applied_fee_threshold = 0xFFFFFFFF` (always passes) or even `fee_info = None`.
4. The farmer's `_process_respond_signatures()` at lines 916–919 unconditionally sets `farmer_reward_address = response.farmer_reward_address_override`.
5. The farmer sends `DeclareProofOfSpace` to the full node with the attacker's address as `farmer_puzzle_hash`.
6. The full node at line 1069 uses `farmer_ph = request.farmer_puzzle_hash` (the attacker's address) to create the block.
7. The farmer's block reward (0.25 XCH) is paid to the attacker. The farmer receives nothing.

def calculate_harvester_fee_quality(proof: bytes, challenge: bytes32) -> uint32:
    """
    This calculates the 'fee quality' given a convention between farmers and third party harvesters.
    See CHIP-22: https://github.com/Chia-Network/chips/pull/88
    """
    return uint32(int.from_bytes(std_hash(proof + challenge)[32 - 4 :], byteorder="big", signed=False))
