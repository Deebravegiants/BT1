Negative `n` in `RequireMOf` Native Script Bypasses All Signature Requirements - (File: `eras/shelley/impl/src/Cardano/Ledger/Shelley/Scripts.hs`, `eras/allegra/impl/src/Cardano/Ledger/Allegra/Scripts.hs`)

### Summary

The `RequireMOf` native script constructor accepts a signed integer `n` (threshold). The CDDL schema encodes `n` as `int32` (Shelley) or `int64` (Allegra through Dijkstra), both of which permit negative values. The evaluator's base case `isValidMOf n _ = n <= 0` causes any script with `n < 0` to unconditionally return `True`, bypassing all signature requirements. No deserialization-time or evaluation-time guard rejects negative `n`. This is a direct analog to the reported bug: the system accepts an out-of-range count (negative instead of >12) and silently treats it as satisfying the condition, producing behavior that diverges from the formal specification.

---

### Finding Description

The formal specification defines `RequireMOf m ts` with `m ∈ ℕ` (natural numbers, i.e., `m ≥ 0`). The Haskell implementation uses `Int` (a signed type) for `m` across all eras.

**Shelley** CDDL:
```
script_n_of_k = (3, n : int32, [* native_script])
int32 = -2147483648 .. 2147483647
``` [1](#0-0) 

**Allegra through Dijkstra** CDDL:
```
script_n_of_k = (3, n : int64, [* native_script])
int64 = min_int64 .. max_int64   ; min_int64 = -9223372036854775808
``` [2](#0-1) 

The evaluator in every era shares the same structure:

```haskell
isValidMOf n StrictSeq.Empty = n <= 0
isValidMOf n (msig StrictSeq.:<| msigs) =
  n <= 0 || if go msig then isValidMOf (n - 1) msigs else isValidMOf n msigs
``` [3](#0-2) [4](#0-3) [5](#0-4) 

When `n < 0`, the condition `n <= 0` is `True` at every recursive step, so the function returns `True` immediately regardless of the script list or the witnesses present. A script `RequireMOf (-1) [RequireSignature k1, RequireSignature k2]` is indistinguishable from `RequireAllOf []` — it always passes.

The deserialization path for Shelley `MultiSig` decodes `m` with a plain `decCBOR` call on `Int` with no range check:

```haskell
3 -> do
  m <- decCBOR
  multiSigs <- sequence <$> decCBOR
  pure (3, MultiSigMOf m <$> multiSigs)
``` [6](#0-5) 

No validation of `m >= 0` exists anywhere in the production evaluation or deserialization path.

---

### Impact Explanation

A UTxO, withdrawal, or governance action locked by a native script with `n < 0` can be spent or executed by **any transaction with zero witnesses**. The ledger accepts the transaction as valid because `evalMultiSig` / `evalTimelock` returns `True`. This constitutes a direct loss of ADA or native assets through an invalid ledger state transition: funds are transferred without satisfying the script's intended authorization condition.

The same bypass applies to stake withdrawals, pool registrations, and Conway-era governance actions that use native scripts as credentials, enabling unauthorized governance or treasury actions.

---

### Likelihood Explanation

The attack requires only that a UTxO be locked by a native script with a negative threshold. This can occur via:

1. **Attacker-crafted script presented as legitimate multisig**: An attacker serializes `RequireMOf (-1) [RequireSignature victim_key]`, computes its script hash, and convinces a counterparty to send funds to the corresponding address (e.g., in a DeFi or escrow context). The attacker then drains the address with a zero-witness transaction.
2. **Wallet/tooling bug**: A wallet that serializes `n` as a signed integer and underflows (e.g., wraps from `0` to `-1` due to an off-by-one) produces a permanently drainable address.

The CBOR encoding of a negative `Int` is valid and accepted by the ledger without error. No special privilege is required.

---

### Recommendation

**Short term**: Add a deserialization guard that rejects `n < 0` for `RequireMOf` in all eras:

```haskell
3 -> do
  m <- decCBOR
  when (m < 0) $ fail "RequireMOf: negative threshold is not permitted"
  multiSigs <- sequence <$> decCBOR
  pure (3, MultiSigMOf m <$> multiSigs)
```

**Long term**: Change the type of the threshold field from `Int` to `Natural` (or `Word`) in `MultiSigRaw` and `TimelockRaw` to make the invariant structural. Update the CDDL to use `uint` instead of `int32`/`int64` for `script_n_of_k`. This aligns the implementation with the formal specification's `m ∈ ℕ`.

---

### Proof of Concept

Construct the following CBOR-encoded native script (Shelley era):

```
# RequireMOf (-1) [RequireSignature <any_keyhash>]
# CBOR: 82 03 82 21 58 1c <28-byte-keyhash>
#   where 21 = negative(0) = -1 in CBOR major type 1
```

1. Compute the script hash of this script.
2. Send ADA to the corresponding script address.
3. Submit a transaction spending that UTxO with an **empty** witness set.
4. The ledger evaluates `evalMultiSig {} (RequireMOf (-1) [...])` → `isValidMOf (-1) [...] = True` → transaction accepted.
5. Funds are transferred with zero signatures.

The same technique applies to `Timelock` scripts in Allegra through Dijkstra eras using `int64` encoding of `-1` (`0x20` in CBOR). [7](#0-6) [8](#0-7)

  (ShelleyEraScript era, NativeScript era ~ MultiSig era) =>
  Set.Set (KeyHash Witness) ->
  NativeScript era ->

evalMultiSig vhks = go
  where
    -- The important part of this validator is that it will stop as soon as it reaches the
    -- required number of valid scripts
    isValidMOf n StrictSeq.Empty = n <= 0
    isValidMOf n (msig StrictSeq.:<| msigs) =
      n <= 0 || if go msig then isValidMOf (n - 1) msigs else isValidMOf n msigs
    go = \case

      RequireAllOf msigs -> all go msigs
      RequireAnyOf msigs -> any go msigs
      RequireMOf m msigs -> isValidMOf m msigs
