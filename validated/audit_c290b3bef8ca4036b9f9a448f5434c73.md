### Title
Consensus Vote Signature Verification Completely Absent — Any Network Peer Can Forge Votes for Any Committee Member - (`crates/apollo_consensus/src/single_height_consensus.rs`)

### Summary

The `handle_vote` function in `SingleHeightConsensus` accepts incoming consensus votes from the network without ever verifying the cryptographic signature on the vote. Simultaneously, the `StateMachine` emits self-votes with `RawSignature::default()` (an all-zero signature). Any network peer who knows the committee member addresses (which are public) can send forged prevote or precommit messages claiming to be any validator, bypassing the BFT safety guarantee and forcing consensus to decide on a `ProposalCommitment` without a real quorum of honest validators.

### Finding Description

**Root cause — missing signature check in `handle_vote`:**

In `crates/apollo_consensus/src/single_height_consensus.rs`, the `handle_vote` function performs only two checks before accepting a vote: (1) the height matches, and (2) the `vote.voter` address is a member of the committee. The cryptographic signature carried in `vote.signature` is never verified:

```rust
pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
    // TODO(Asmaa): verify the signature      ← signature is NEVER checked
    ...
    if !self.committee.members().iter().any(|s| s.address == vote.voter) {
        debug!("Ignoring vote from non validator: vote={:?}", vote);
        return VecDeque::new();
    }
    // vote is accepted and fed into the state machine
    self.state_machine.handle_event(sm_vote)
}
```

**Companion gap — votes are emitted with zero signatures:**

In `crates/apollo_consensus/src/state_machine.rs`, `make_self_vote` constructs every outgoing vote with `RawSignature::default()`:

```rust
let vote = Vote {
    ...
    voter: self.id,
    // TODO(Asmaa): sign the vote
    signature: RawSignature::default(),   ← zero/empty signature
};
```

The `SignatureManager` crate (`crates/apollo_signature_manager/`) already exposes `sign_precommit_vote` and `verify_precommit_vote_signature`, but neither is wired into the consensus vote path.

**Analog to H-30:**

| H-30 (Astaria) | Sequencer analog |
|---|---|
| `delegate` defaults to `address(0)` | Votes carry `RawSignature::default()` (zero) |
| `ecrecover` returns `address(0)` for invalid `v` | No `ecrecover`/ECDSA call is made at all |
| Check `recovered == delegate` passes trivially | Check is simply absent (`// TODO`) |
| Attacker sets `strategist = address(0)` | Attacker sets `vote.voter = <any committee address>` |

In both cases the "zero default" of the credential field is never validated against the claimed identity, allowing an unauthenticated actor to pass as any authorized party.

### Impact Explanation

An attacker who can send messages on the consensus broadcast channel can:

1. Observe the current height, round, and committee member addresses (all public).
2. Observe a `ProposalCommitment` value that the local node has already validated (broadcast in `ProposalFin`).
3. Send forged `Precommit` votes claiming to be from enough committee members to satisfy the quorum threshold (>2/3 of total weight for Byzantine quorum).
4. The state machine's `upon_decision` fires, emitting `SMRequest::DecisionReached` with the attacker-chosen `ProposalCommitment`.
5. The sequencer commits a block whose content was never agreed upon by the real validator set.

This breaks the BFT safety invariant: a block can be finalized without a genuine quorum of honest validators, leading to wrong committed state, wrong receipts, wrong events, and wrong storage values — matching **Critical impact #2** ("Wrong state, receipt, event, L1 message, class hash, storage value, or revert result").

### Likelihood Explanation

- **Unprivileged trigger**: any peer on the consensus gossip network can send `Vote` protobuf messages; no special key or role is required.
- **Committee addresses are public**: they appear in `ProposalInit.proposer`, in committee provider responses, and in broadcast votes.
- **No network-layer authentication for votes**: the `BroadcastVoteChannel` in `manager.rs` delivers deserialized `Vote` structs directly to `handle_vote` with no prior authentication step.
- **Duplicate-vote guard is bypassable**: the guard only fires if the same `(round, voter)` pair is seen twice. An attacker forging a vote for a validator who has not yet voted for that round will not be caught.

### Recommendation

1. **Verify the signature in `handle_vote`** before the vote is fed into the state machine:
   ```rust
   pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
       let public_key = self.committee.members()
           .iter()
           .find(|s| s.address == vote.voter)
           .map(|s| s.public_key);
       let Some(pk) = public_key else {
           return VecDeque::new();
       };
       let block_hash = BlockHash(vote.proposal_commitment
           .map(|c| c.0).unwrap_or_default());
       match verify_precommit_vote_signature(block_hash, vote.signature.clone(), pk) {
           Ok(true) => {}
           _ => { warn!("Invalid vote signature"); return VecDeque::new(); }
       }
       ...
   }
   ```
2. **Sign votes in `make_self_vote`** using `SignatureManager::sign_precommit_vote`.
3. **Store validator public keys** in the `CommitteeMember` struct so they are available at verification time.
