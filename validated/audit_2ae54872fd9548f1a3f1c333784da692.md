### Title
Native SOL Fees Accumulate in `sol_vault` With No Withdrawal Path — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`, `init_transfer_sol.rs`)

---

### Summary

When a user initiates a Solana-side transfer and specifies a non-zero `native_fee`, the SOL is deposited into the `sol_vault` PDA. No instruction exists anywhere in the program to withdraw or route those accumulated lamports to the intended fee recipient (relayer). The SOL is permanently locked.

---

### Finding Description

In `init_transfer.rs`, when `payload.native_fee > 0`, the user's SOL is transferred directly into the `sol_vault` PDA: [1](#0-0) 

In `init_transfer_sol.rs`, the entire `native_fee + amount` is deposited into `sol_vault` in a single transfer: [2](#0-1) 

On the redemption side, `finalize_transfer_sol.rs` only transfers `data.amount` back out of `sol_vault` to the recipient — the `native_fee` portion is never disbursed: [3](#0-2) 

The `FinalizeTransferResponse` carries a `fee_recipient` field that is posted back to NEAR as a Wormhole message, and the NEAR side's `send_fee_internal` handles the fee by minting a wrapped-SOL token on NEAR — it does **not** release the actual SOL lamports sitting in `sol_vault`: [4](#0-3) 

The complete set of admin instructions contains no withdrawal function:



The four admin instructions (`change_config`, `initialize`, `pause`, `update_metadata`) provide no path to drain `sol_vault` of accumulated fee lamports.

---

### Impact Explanation

Every `native_fee` paid in SOL by any user calling `init_transfer` or `init_transfer_sol` is permanently locked in the `sol_vault` PDA. There is no instruction — privileged or otherwise — that can move those lamports to a relayer or protocol treasury. This is an irreversible fund lock of user-paid protocol value, matching the allowed impact: *"Irreversible fund lock … permanently unclaimable user or protocol value in bridge … fee … flows."*

---

### Likelihood Explanation

Any user who sets `native_fee > 0` when calling `init_transfer` or `init_transfer_sol` triggers the lock. This is a normal, documented user action (paying a relayer incentive). The path is fully unprivileged and reachable from day one of mainnet deployment. Every such transfer permanently forfeits the fee.

---

### Recommendation

Add a dedicated admin instruction (e.g., `withdraw_fees`) that transfers the accumulated `native_fee` lamports from `sol_vault` to a configured fee-recipient account, analogous to the `collectFees` method Gondi added as their mitigation. The instruction should be gated to an admin/DAO role and should only withdraw the fee surplus (i.e., `sol_vault.lamports - rent_exempt_minimum`), leaving the vault rent-exempt.

---

### Proof of Concept

1. User calls `init_transfer` with `native_fee = 1_000_000` lamports (0.001 SOL).
2. `init_transfer.rs` line 76–85 transfers 1_000_000 lamports from user → `sol_vault`.
3. NEAR side eventually calls `finalize_transfer_sol` to release the bridged SOL to the recipient; only `data.amount` leaves `sol_vault` — the 1_000_000 lamport fee remains.
4. Inspect all four admin instructions: none contain a `sol_vault` debit.
5. The 1_000_000 lamports are unrecoverable. Repeat for every user who ever pays a non-zero `native_fee`.
