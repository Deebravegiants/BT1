### Title
Missing `minAmountOut` Slippage Protection in L2 Pool `deposit` Functions - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3.sol)

---

### Summary

All L2 pool `deposit` functions compute the wrsETH/rsETH output amount from a live oracle rate at execution time but accept no `minAmountOut` parameter. A user who previews the swap via `viewSwapRsETHAmountAndFee` and then submits a `deposit` transaction can silently receive fewer tokens than expected if the oracle rate is updated between preview and execution. The L1 `LRTDepositPool` already implements this protection via `minRSETHAmountExpected`, making the omission on L2 an inconsistency with the protocol's own design intent.

---

### Finding Description

Every L2 pool variant exposes public `deposit` functions that:

1. Accept ETH or a supported LST from the caller.
2. Compute `rsETHAmount` by calling `viewSwapRsETHAmountAndFee`, which internally reads the live oracle rate via `getRate()`.
3. Immediately mint or transfer that amount of wrsETH/rsETH to the caller.
4. Emit `SwapOccurred`.

None of these functions accept a caller-supplied minimum output amount.

**`RSETHPoolV3ExternalBridge.deposit` (token path):** [2](#0-1) 

The same pattern is present in every other L2 pool:

- `RSETHPool.deposit` (ETH and token): [3](#0-2)

- `RSETHPoolV3WithNativeChainBridge.deposit`, `RSETHPoolV3.deposit`, `RSETHPoolV2ExternalBridge.deposit` — identical structure.

By contrast, the L1 `LRTDepositPool` explicitly enforces a caller-supplied minimum:

```solidity
// contracts/LRTDepositPool.sol
function depositETH(uint256 minRSETHAmountExpected, ...) external payable { ... }
function depositAsset(address asset, uint256 depositAmount, uint256 minRSETHAmountExpected, ...) external { ... }
``` [5](#0-4) 

The enforcement happens in `_beforeDeposit`: [6](#0-5) 

The oracle rate consumed by the L2 pools is read from `IOracle(rsETHOracle).getRate()`: [7](#0-6) 

This rate is updated via cross-chain messages (LayerZero/CCIP) from the L1 oracle. A rate update arriving in the same block as, or just before, a user's `deposit` transaction will silently reduce the wrsETH minted with no on-chain recourse for the user.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who calls `viewSwapRsETHAmountAndFee` off-chain to preview the swap and then submits `deposit` may receive fewer wrsETH/rsETH than the preview indicated. Because the rsETH/ETH rate is monotonically non-decreasing (it is a yield-bearing rate), every oracle update that occurs between preview and execution silently reduces the user's output. The user's deposited ETH/LST is not lost — it is correctly accounted for in the pool — but the user receives fewer liquid tokens than they were shown, with no ability to set a floor.

---

### Likelihood Explanation

**Low.**

The rsETH/ETH rate increases gradually and predictably. A meaningful discrepancy between preview and execution requires a rate update to land in the same block or in a narrow window before the user's transaction. Cross-chain rate messages are not attacker-controlled. However, this is a normal operational event (rate updates happen regularly), and the absence of any protection means every depositor is silently exposed to it on every transaction. No privileged role or external compromise is required — the condition arises from ordinary protocol operation.

---

### Recommendation

Add a `minRSETHAmountExpected` parameter to all L2 pool `deposit` functions, mirroring the L1 `LRTDepositPool` pattern:

```solidity
function deposit(
    string memory referralId,
    uint256 minRSETHAmountExpected   // <-- add
) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same change to the token-deposit overload and to all pool variants (`RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`).

---

### Proof of Concept

1. User calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV3ExternalBridge`. Oracle rate is `1.05e18`; preview returns `~0.952 wrsETH`.
2. A cross-chain rate message arrives and updates the oracle to `1.06e18` before the user's `deposit` transaction is mined.
3. User's `deposit{value: 1 ether}(referralId)` executes. `viewSwapRsETHAmountAndFee` now reads `1.06e18` and computes `~0.943 wrsETH`.
4. `wrsETH.mint(msg.sender, rsETHAmount)` mints `~0.943 wrsETH` — roughly 0.9% less than the user expected — with no revert and no recourse.
5. The L1 equivalent (`LRTDepositPool.depositETH`) would have reverted with `MinimumAmountToReceiveNotMet` if the user had passed `0.952e18` as `minRSETHAmountExpected`. [8](#0-7) [9](#0-8)
