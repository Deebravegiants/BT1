### Title
Stale `accept_requests` Flag Allows Signature Requests and Responses When TEE Health Has Silently Degraded Below Threshold — (`crates/contract/src/lib.rs`)

---

### Summary

`MpcContract.accept_requests` is a stored boolean that is only mutated when `verify_tee()` is explicitly called by a participant. TEE attestations carry expiry timestamps that advance independently of any contract call. Between successive `verify_tee()` invocations the flag is never recomputed, so the contract continues accepting `sign()` / `respond()` traffic even after enough attestations have expired to drop the live TEE-attested participant count below the signing threshold — the exact safety invariant the flag is meant to enforce.

---

### Finding Description

`MpcContract` stores `accept_requests: bool` as a persistent contract field. [1](#0-0) 

It is set to `true` unconditionally at initialization: [2](#0-1) [3](#0-2) 

It is the sole network-wide gate checked in every user-facing and node-facing request path: [4](#0-3) [5](#0-4) 

The flag is only updated inside `verify_tee()`, which must be explicitly called by a participant: [6](#0-5) 

`verify_tee()` itself calls `reverify_and_cleanup_participants()`, which does perform a real-time expiry check against `env::block_timestamp()`: [7](#0-6) 

The problem is structural: the real-time validity computation lives inside `verify_tee()` and its result is cached in `accept_requests`. Nothing forces `verify_tee()` to be called before each `sign()` or `respond()`. Any gap between the last `verify_tee()` call and the moment enough attestations expire is a window during which `accept_requests` is stale-`true` while the live TEE-attested participant count is below threshold.

This is the direct analog of the SwapUtils H-03 finding: `tokenPrecisionMultipliers` was a stored value updated only on admin calls, while the correct value required real-time interpolation. Here, `accept_requests` is a stored value updated only on explicit `verify_tee()` calls, while the correct value requires real-time attestation re-verification.

---

### Impact Explanation

**Request-lifecycle safety invariant broken (Medium).**

During the stale window:

1. Any unprivileged caller can invoke `sign()` / `verify_foreign_transaction()` / `request_app_private_key()` and have the request enqueued, even though the network no longer has a TEE-attested quorum capable of fulfilling it. Requests will time out after ~200 blocks, consuming gas and locking deposits until the yield-resume timeout fires.

2. More critically, `respond()` also gates on `accept_requests` *after* `assert_caller_is_attested_participant_and_protocol_active()`. If the per-caller attestation check does not independently re-verify expiry in real-time (i.e., it only confirms the node is in the participant set without re-running `reverify_participants()`), a node whose attestation has since expired can still submit a valid `respond()` call and have its signature accepted by the contract. This constitutes a bypass of the TEE enforcement layer: a node that is no longer provably running inside a trusted execution environment can contribute a threshold signature that the contract accepts as legitimate. [8](#0-7) 

---

### Likelihood Explanation

TEE attestations have finite lifetimes (expiry timestamps embedded in the quote). The `verify_tee()` function is not called automatically on any schedule — it requires an explicit participant transaction. Any real-world deployment gap between attestation expiry and the next `verify_tee()` call opens the window. No special privilege is required: any NEAR account can call `sign()` with a 1-yoctonear deposit. The attack is purely passive — wait for attestations to expire, then submit requests before `verify_tee()` is called.

---

### Recommendation

Replace the stored `accept_requests` flag with an inline real-time TEE health check inside `check_request_preconditions()` (and the `respond*` entry points), analogous to the SwapUtils recommendation of computing `tokenPrecisionMultipliers` in real-time rather than reading the cached value:

```rust
// Instead of:
if !self.accept_requests {
    env::panic_str(&TeeError::TeeValidationFailed.to_string())
}

// Compute dynamically:
let tee_upgrade_deadline = Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);
if let ProtocolContractState::Running(ref state) = self.protocol_state {
    match self.tee_state.reverify_and_cleanup_participants(
        state.parameters.participants(),
        tee_upgrade_deadline,
    ) {
        TeeValidationResult::Full => {}
        TeeValidationResult::Partial { .. } => {
            env::panic_str(&TeeError::TeeValidationFailed.to_string())
        }
    }
}
```

If the inline check is too gas-expensive for every call, at minimum add a block-timestamp-based staleness guard: if the last `verify_tee()` was called more than `tee_upgrade_deadline_duration_seconds` ago, treat `accept_requests` as `false` until re-verified.

---

### Proof of Concept

1. Deploy contract with 3 participants, threshold 2; all submit valid TEE attestations.
2. Call `verify_tee()` → `accept_requests = true`.
3. Advance block time past the expiry of 2 participants' attestations (leaving only 1 valid, below threshold 2).
4. Do **not** call `verify_tee()`.
5. Call `sign(...)` with 1 yoctonear deposit — succeeds because `accept_requests` is still stale-`true`.
6. The request is enqueued. The network cannot produce a threshold signature (only 1 TEE-valid node remains). The yield times out after ~200 blocks; the user loses gas.
7. If `assert_caller_is_attested_participant_and_protocol_active()` does not re-verify expiry, a node with an expired attestation can additionally call `respond()` — which also passes the stale `accept_requests` check — and have its signature accepted, completing a TEE-bypass signing flow.     pub fn reverify_and_cleanup_participants(
        &mut self,
        participants: &Participants,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeValidationResult {
        self.allowed_docker_image_hashes
            .cleanup_expired_hashes(tee_upgrade_deadline_duration);

        let participants_with_valid_attestation: Vec<_> = participants
            .participants()
            .iter()
            .filter(|(_, _, participant_info)| {
                // Use the stored NodeId (keyed by TLS public key) so the real
                // `account_public_key` participates in re-verification. If
                // there is no stored attestation for this TLS key, the
                // participant is invalid.
                let Some(node_id) = self.find_node_id_by_tls_key(&participant_info.tls_public_key)
                else {
                    return false;
                };

                let tee_status =
                    self.reverify_participants(&node_id, tee_upgrade_deadline_duration);

                matches!(tee_status, TeeQuoteStatus::Valid)
            })
            .cloned()
            .collect();

        if participants_with_valid_attestation.len() != participants.len() {
            let participants_with_valid_attestation =
                Participants::init(participants.next_id(), participants_with_valid_attestation);

            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            }
        } else {
            TeeValidationResult::Full
        }
