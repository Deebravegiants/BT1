Audit Report

## Title
Missing Minimum Withdrawal Amount Allows Queue Inflation and Temporary Withdrawal Freeze - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`minRsEthAmountToWithdraw[asset]` defaults to `0` for every asset because the `initialize` function never sets it, and no non-zero default exists. An unprivileged rsETH holder can call `initiateWithdrawal` with 1-wei amounts an arbitrary number of times, inflating the global FIFO withdrawal queue. Legitimate users whose requests land behind the attacker's dust entries cannot call `completeWithdrawal` until the operator advances `nextLockedNonce[asset]` past every preceding dust slot, causing a temporary freeze of their funds.

## Finding Description
`minRsEthAmountToWithdraw` is declared as a plain mapping with no initializer assignment:

```solidity
mapping(address asset => uint256) public minRsEthAmountToWithdraw; // line 35
```

`initialize` (lines 90–98) sets only `withdrawalDelayBlocks` and `lrtConfig`; it never writes to `minRsEthAmountToWithdraw`. The only way to set a non-zero minimum is via the admin-only `setMinRsEthAmountToWithdraw` (line 330). Until that call is made, every asset has a minimum of `0`.

`initiateWithdrawal` enforces (lines 162–164):
```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```
When the minimum is `0`, the condition `rsETHUnstaked < 0` is vacuously false for any `uint256`, so any amount ≥ 1 wei passes. For 1-wei inputs, `getExpectedAssetAmount` (line 593) rounds down to `0`, so `expectedAssetAmount = 0`, and the `ExceedAmountToWithdraw` guard at line 170 also passes. Each call pushes a new nonce (lines 756–757):
```solidity
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
```

`_unlockWithdrawalRequests` (lines 790–815) iterates in strict FIFO order from `nextLockedNonce` to `firstExcludedIndex` with no mechanism to skip entries. Dust entries with `payoutAmount == 0` do not trigger the `availableAssetAmount < payoutAmount` break, so they are individually processable but still occupy nonce slots the operator must advance through one by one.

`_processWithdrawalCompletion` (line 707) enforces:
```solidity
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```
A legitimate user whose nonce is `N` cannot complete their withdrawal until `nextLockedNonce[asset] > N`, meaning every attacker-created nonce below `N` must first be processed by the operator.

## Impact Explanation
**Medium — Temporary freezing of funds.** Legitimate users whose withdrawal requests are queued after the attacker's dust entries cannot call `completeWithdrawal` until the operator exhausts the entire attacker-inflated prefix of the queue. With thousands of dust entries, this requires many `unlockQueue` calls (each gas-bounded), delaying user access to their funds for an extended but finite period. Funds are not permanently lost, placing this squarely in the "Temporary freezing of funds" impact class.

## Likelihood Explanation
The attack requires only that the attacker holds a non-zero amount of rsETH (or can acquire it cheaply) and pays gas per call. `initiateWithdrawal` is fully public and permissionless. Any asset for which the admin has not yet called `setMinRsEthAmountToWithdraw` is permanently vulnerable to this attack. The cost to the attacker is low (gas only; the 1-wei rsETH is locked in the contract but retrievable via the normal withdrawal flow), while the cost to the operator and legitimate users is proportional to the number of dust entries created.

## Recommendation
1. In `initialize`, set a non-zero default for `minRsEthAmountToWithdraw` for each supported asset, or add a precondition in `initiateWithdrawal` that `minRsEthAmountToWithdraw[asset] > 0` before the asset is usable.
2. Add a check after computing `expectedAssetAmount` that `expectedAssetAmount > 0`, rejecting requests that round down to zero asset payout.
3. Consider adding a per-user rate limit or per-block request cap to bound queue growth.

## Proof of Concept
1. Deploy with default state: `minRsEthAmountToWithdraw[ETH_TOKEN] == 0` (no admin call to `setMinRsEthAmountToWithdraw`).
2. Attacker acquires a small amount of rsETH (e.g., 10,000 wei).
3. Attacker calls `initiateWithdrawal(ETH_TOKEN, 1, "")` 10,000 times, creating 10,000 dust entries at nonces `[0, 9999]`. Each call transfers 1 wei of rsETH to the contract and records `expectedAssetAmount = 0`.
4. Legitimate user calls `initiateWithdrawal(ETH_TOKEN, 1 ether, "")`, receiving nonce `10000`.
5. After the 8-day withdrawal delay, operator calls `unlockQueue` repeatedly; each call advances `nextLockedNonce` through a gas-bounded batch of attacker dust entries.
6. Until all 10,000 dust entries are processed (advancing `nextLockedNonce` to at least `10001`), the legitimate user's `completeWithdrawal` reverts with `WithdrawalLocked` — their funds are temporarily frozen.

**Foundry test sketch:**
```solidity
function test_queueInflation() public {
    // attacker creates 10_000 dust requests
    vm.startPrank(attacker);
    for (uint i = 0; i < 10_000; i++) {
        withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");
    }
    vm.stopPrank();

    // legitimate user queues 1 ETH withdrawal → nonce 10_000
    vm.prank(user);
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1 ether, "");

    // advance blocks past delay
    vm.roll(block.number + withdrawalDelayBlocks + 1);

    // user cannot complete — locked behind dust entries
    vm.prank(user);
    vm.expectRevert(WithdrawalLocked.selector);
    withdrawalManager.completeWithdrawal(ETH_TOKEN, "");
}
```
