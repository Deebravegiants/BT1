Audit Report

## Title
Unsafe `int256` to `uint256` Cast Without Negativity Check Freezes Protocol - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` casts the `int256 price` returned by Chainlink's `latestRoundData()` directly to `uint256` with no positivity guard. If the feed returns a zero or negative value, the bitwise reinterpretation produces a value near `2**256`, and the subsequent `* 1e18` multiplication overflows under Solidity 0.8's checked arithmetic, reverting every protocol function that reads the price. The same codebase already applies the correct guard in `ChainlinkOracleForRSETHPoolCollateral`, confirming developer awareness of the pattern.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at lines 52–54:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

There is no check that `price > 0` before the cast. If `price == -1`, then `uint256(-1) == 2**256 - 1`, and `(2**256 - 1) * 1e18` overflows and reverts under Solidity 0.8 checked arithmetic.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` at line 32 explicitly guards:

```solidity
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The revert propagates through four confirmed call chains:

- **Price update:** `LRTOracle._getTotalEthInProtocol()` calls `getAssetPrice(asset)` at line 339. [3](#0-2) 
- **Deposits:** `LRTDepositPool.getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(asset)` at line 520. [4](#0-3) 
- **Withdrawals:** `LRTWithdrawalManager.getExpectedAssetAmount()` calls `lrtOracle.getAssetPrice(asset)` at line 593. [5](#0-4) 
- **Unlock queue:** `LRTWithdrawalManager._createUnlockParams()` calls `lrtOracle.getAssetPrice(asset)` at line 848. [6](#0-5) 

No existing check in `ChainlinkPriceOracle` intercepts a non-positive price before the cast.

## Impact Explanation
**Medium — Temporary freezing of funds.** If any supported asset's Chainlink feed returns a non-positive price for even a single block, all of `updateRSETHPrice()`, `depositETH()`, `depositAsset()`, `initiateWithdrawal()`, and `unlockQueue()` revert. Funds already committed to pending withdrawal requests are frozen until the feed recovers or governance replaces the oracle. This matches the allowed impact class "Temporary freezing of funds."

## Likelihood Explanation
Chainlink feeds for ETH/LST assets do not routinely return negative values, but the scenario is not purely theoretical: Chainlink has historically returned zero or anomalous values during oracle incidents. No privileged access is required; the freeze is triggered automatically the moment any public caller (depositor, withdrawer, or `updateRSETHPrice()` caller) invokes a function that reads the affected feed. The absence of any validation in `ChainlinkPriceOracle` while the identical guard exists in `ChainlinkOracleForRSETHPoolCollateral` confirms the omission is an oversight rather than an intentional design choice.

## Recommendation
Add a positivity check before the cast, mirroring the pattern in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
if (price <= 0) revert InvalidPrice();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Additionally, add staleness checks (`answeredInRound >= roundId`, `updatedAt != 0`) consistent with `ChainlinkOracleForRSETHPoolCollateral.getRate()`. [7](#0-6) 

## Proof of Concept
1. Deploy a mock Chainlink feed for a supported asset (e.g., stETH) that returns `price = -1` from `latestRoundData()`.
2. Register it via `ChainlinkPriceOracle.updatePriceFeedFor(stETH, mockFeed)`.
3. Call `LRTDepositPool.depositAsset(stETH, amount, minRSETH, "")` as any user.
4. Execution reaches `ChainlinkPriceOracle.getAssetPrice(stETH)`: `uint256(-1) * 1e18` overflows → revert.
5. Repeat for `initiateWithdrawal`, `unlockQueue`, and `updateRSETHPrice` — all revert identically.
6. A Foundry fork test can confirm by mocking `latestRoundData` to return `-1` and asserting all four entry points revert with an arithmetic overflow panic.
