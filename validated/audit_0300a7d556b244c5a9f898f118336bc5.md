### Title
Wrong `receiver_id` Used in `validate_delegate_action` for `DeterministicStateInit` Inner-Action Validation — (`runtime/runtime/src/action_validation.rs`)

### Summary

In `validate_delegate_action`, when validating the inner actions of a `DelegateAction` (meta transaction), the function accepts `delegate_action.receiver_id()` as the user-supplied target but uses `receiver` — the outer transaction's `receiver_id`, which equals `delegate_action.sender_id` — to validate `DeterministicStateInit` inner actions. This is the exact same bug class as the external report: a user-supplied parameter is accepted but silently substituted with a different value, corrupting the validity decision for the transaction.

### Finding Description

`validate_delegate_action` in `runtime/runtime/src/action_validation.rs` receives two distinct account IDs:

- `receiver: &AccountId` — the outer transaction's `receiver_id`, which is the `DelegateAction`'s **sender** (the account that unpacks the delegate action)
- `delegate_action.receiver_id()` — the **inner** target account to which the delegate action's inner actions are dispatched

For `DeterministicStateInit` inner actions, the correct invariant is:

```
derive_near_deterministic_account_id(state_init) == delegate_action.receiver_id()
```

However, in the pre-fix code path (protocol version < 85), the function selects `receiver` (the outer tx's receiver, i.e., the delegate sender) instead of `delegate_action.receiver_id()` as `inner_receiver`:

```rust
let inner_receiver =
    if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
        delegate_action.receiver_id()   // correct
    } else {
        receiver                        // BUG: outer tx receiver, not delegate target
    };
validate_actions_with_mode(limit_config, &actions, inner_receiver, ...)?;
```

This means `validate_deterministic_state_init` checks `derive(state_init) == outer_tx.receiver_id` instead of `derive(state_init) == delegate_action.receiver_id`. The user-supplied `delegate_action.receiver_id` — the parameter that carries the intended target — is completely ignored for this check. [1](#0-0) 

### Impact Explanation

Two concrete validity-decision corruptions result:

1. **Malformed meta-tx incorrectly accepted at tx validation**: An attacker crafts a `DelegateAction` where `state_init` derives to `outer_tx.receiver_id` (= `delegate_action.sender_id`) but `delegate_action.receiver_id` points to a different deterministic account. The wrong check `derive(state_init) == outer_tx.receiver_id` passes. The transaction is admitted to the block. It then fails at receipt execution with `InvalidDeterministicStateInitReceiver` — wasting gas and producing a spurious on-chain failure outcome. The corrupted protocol value is the **tx validity decision**: a transaction that must be rejected at validation is instead accepted.

2. **Legitimate meta-tx incorrectly rejected**: A user legitimately wraps a `DeterministicStateInit` targeting `det_account_X` inside a `DelegateAction`. The correct `derive(state_init) == det_account_X` holds, but the wrong check `derive(state_init) == outer_tx.receiver_id` (= `delegate_action.sender_id` ≠ `det_account_X`) fails. The transaction is rejected with `InvalidDeterministicStateInitReceiver` even though it is valid. The corrupted protocol value is again the **tx validity decision**.

The codebase's own test `try_meta_tx_deterministic_receiver_exploit` confirms scenario 1: without the fix, the exploit tx passes initial tx validation and is only caught at receipt execution. [2](#0-1) [3](#0-2) 

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a signed meta transaction containing a `DelegateAction` with a `DeterministicStateInit` inner action. No validator, node-admin, or trusted-service privileges are required. The entry path is a standard public RPC transaction submission. The bug is active on nodes running protocol version < 85 (the `FixDelegatedDeterministicStateInit` feature is assigned to protocol version 85). [4](#0-3) 

### Recommendation

Use `delegate_action.receiver_id()` — not the outer `receiver` — as the `inner_receiver` when validating inner actions of a `DelegateAction`. This is already implemented behind `ProtocolFeature::FixDelegatedDeterministicStateInit` at protocol version 85. The fix should be the unconditional code path. [5](#0-4) 

### Proof of Concept

The exploit scenario is fully exercised by `try_meta_tx_deterministic_receiver_exploit` in `test-loop-tests/src/tests/deterministic_account_id.rs`:

- `det_account_b = derive(state_init_b)` is deployed and given a full-access key.
- A relayer wraps `DeterministicStateInit(state_init_b)` inside a `DelegateAction` with `sender_id = det_account_b` and `receiver_id = det_account_a` (wrong target).
- The outer tx targets `det_account_b`, so `outer_tx.receiver_id = det_account_b`.
- Pre-fix check: `derive(state_init_b) == det_account_b` → **passes** (wrong).
- The tx is admitted; at receipt execution, `derive(state_init_b) == det_account_a` fails → `InvalidDeterministicStateInitReceiver`. [6](#0-5) [7](#0-6)

    action: &DeterministicStateInitAction,
    receiver_id: &AccountId,

    validate_global_contract_identifier(action.state_init.code())?;

    let derived_id = derive_near_deterministic_account_id(&action.state_init);

    if derived_id != *receiver_id {
        return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver {
            derived_id,
            receiver_id: receiver_id.clone(),
