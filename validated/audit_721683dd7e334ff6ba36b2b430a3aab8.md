**Key facts established:**

1. `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` at [1](#0-0)  hardcodes `only_query=false` for both `Invoke` and `DeployAccount` variants, with a developer TODO explicitly acknowledging the problem: `// TODO(yair): pass the right value for only_query field.`

2. `calc_tx_hash` at [2](#0-1)  passes the stored `only_query` flag directly into `TransactionOptions { only_query }` when calling `get_transaction_hash`. The flag is what controls whether the query-version high bit is included in the hash computation.

3. `apply_on_transaction` at [3](#0-2)  confirms the `only_query` field is threaded through the entire execution pipeline per-variant.

**Analysis:**

When a client submits a `BroadcastedInvokeTransaction` with a query-version (e.g., `version=0x100000000000000000000000000000001`) to `starknet_simulateTransactions` or `starknet_estimateFee`, the conversion hardcodes `only_query=false`. This causes `calc_tx_hash` to compute the hash **without** the query-version high bit. However, the client's account contract signed the transaction using the hash **with** the query-version high bit. The result is that `__validate__` (when not skipped) receives the wrong `transaction_hash`, and any contract reading the hash via `get_execution_info` during simulation also sees the wrong value.

This is a real behavioral mismatch, but its impact is bounded:
- It only affects `starknet_simulateTransactions` / `starknet_estimateFee` (simulation/estimation paths), not actual on-chain execution or consensus.
- When `SkipValidate` is set, `__validate__` is bypassed, so the wrong hash only affects contracts that explicitly read `get_execution_info` during execution.
- The attacker can only corrupt their own simulation results; no other user's state, block hash, or receipt is affected.
- The TODO comment confirms the developers are aware of this gap.

The impact fits "High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value." However, the corrupted value is the simulation trace/fee estimate returned to the **submitting client only** — it does not propagate to consensus, block commitments, or other users' state.

### Title
Wrong `transaction_hash` in simulation trace for query-version transactions due to hardcoded `only_query=false` — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

### Summary
`TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` hardcodes `only_query=false` for `Invoke` and `DeployAccount` variants. When a client submits a query-version transaction (high bit set in version), `calc_tx_hash` computes the hash without the query bit, producing a hash that differs from what the account contract signed. Simulation traces and fee estimates reflect this wrong hash.

### Finding Description
At [4](#0-3) , `only_query` is unconditionally `false` regardless of the version field in the incoming `BroadcastedTransaction`. The `calc_tx_hash` function at [5](#0-4)  then computes the hash using `TransactionOptions { only_query: false }`, omitting the query-version high bit. The Starknet transaction hash specification requires the high bit to be present in the version when `only_query=true`, so the resulting hash differs from the one the client's account contract expects.

### Impact Explanation
Simulation traces returned by `starknet_simulateTransactions` contain a `transaction_hash` that does not match the hash the account contract signed. If `__validate__` is not skipped, it will fail (wrong hash seen). If skipped, any contract reading `get_execution_info` during simulation sees the wrong hash. Fee estimates derived from such simulations are unreliable for query-version transactions.

### Likelihood Explanation
Any unprivileged RPC client that submits a query-version transaction (the standard way to call `starknet_estimateFee` / `starknet_simulateTransactions` per the Starknet spec) triggers this path. The developer TODO at [6](#0-5)  confirms the issue is known but unresolved.

### Recommendation
Derive `only_query` from the version field of the incoming `BroadcastedTransaction` before constructing `ExecutableTransactionInput`. Check whether the high bit (`0x100000000000000000000000000000000`) is set in the version and pass `only_query=true` accordingly, then strip the high bit from the stored version field so the hash is computed correctly.

### Proof of Concept
Submit `starknet_simulateTransactions` with an `InvokeV1` transaction where `version=0x100000000000000000000000000000001`. Observe that the `transaction_hash` in the returned trace equals the hash computed with `only_query=false` (no high bit), not the hash the account contract signed with `only_query=true`. A property test fuzzing the version high bit and asserting `trace.transaction_hash == hash(tx, only_query=true)` will fail for all query-version inputs.
