### Title
Missing ECDSA message hash size enforcement allows signing over non-standard message domains — (`src/cbmpc/c_api/ecdsa_mp.cpp`, `src/cbmpc/api/ecdsa_mp.cpp`)

### Summary
`cbmpc_ecdsa_mp_sign_additive` (and `cbmpc_ecdsa_mp_sign_ac`) accept `msg_hash` inputs of any size in [1, 64] bytes. No layer enforces the standard 32-byte size required for ECDSA over secp256k1/P-256. The internal `curve_msg_to_bn` utility silently truncates inputs > 32 bytes to 32 bytes, and treats inputs < 32 bytes as smaller scalars. The protocol produces a valid DER signature in both cases.

### Finding Description
The call chain is:

1. `cbmpc_ecdsa_mp_sign_additive` → `validate_cmem(msg_hash)`: accepts any size ≥ 0 with non-null data. [6](#0-5) 
2. `coinbase::api::ecdsa_mp::sign_additive` → `validate_mem_arg_max_size(msg, "msg", 64)`: accepts any size in [1, 64]. [2](#0-1) 
3. `coinbase::mpc::ecdsampc::sign` → `curve_msg_to_bn`: truncates to `curve.size()` (32) if larger, or uses raw bytes if smaller. [3](#0-2) 
4. The protocol consistency check hashes `msg` as-is, so all parties agree on the non-standard input and the protocol completes successfully. [5](#0-4) 

The `MAX_MESSAGE_DIGEST_SIZE = 64` constant is documented as "64 bytes (e.g., SHA-512 / SHA3-512)", confirming the intent to allow SHA-512 digests — but ECDSA over secp256k1/P-256 with a 64-byte input silently signs only the first 32 bytes. [7](#0-6) 

### Impact Explanation
- **33–64 byte input**: `curve_msg_to_bn` silently truncates to the first 32 bytes. The returned DER signature is a valid ECDSA signature over `msg[0..31]`, not over the full input. Two distinct 63-byte messages sharing the same first 32 bytes produce identical signatures.
- **1–31 byte input**: The message scalar is a very small integer (e.g., 0–255 for 1 byte), which is non-standard and weakens the message entropy assumption of ECDSA.

In both cases, `CBMPC_SUCCESS` is returned and a valid DER signature is produced. Honest callers that pass a SHA-512 hash (64 bytes) believing the full hash is signed are silently misled — only the first 32 bytes are signed.

### Likelihood Explanation
Any caller with a valid key blob can trigger this. The Schnorr API's explicit `msg.size != 32` guard shows the codebase is aware of this requirement; its absence from the ECDSA path is an oversight. The `MAX_MESSAGE_DIGEST_SIZE = 64` comment explicitly invites SHA-512 use, making the truncation path likely to be hit in practice.

### Recommendation
Add an explicit size check in `coinbase::api::ecdsa_mp::sign_additive` (and `sign_ac`) mirroring the Schnorr guard:
```cpp
if (msg.size != curve_order_size) return coinbase::error(E_BADARG, "ECDSA requires a hash matching the curve order size");
```
Alternatively, document precisely which hash sizes are accepted and what truncation behavior applies, and add a corresponding check in the C API wrapper. The `MAX_MESSAGE_DIGEST_SIZE = 64` constant should not be used as the upper bound for ECDSA message hashes without an accompanying exact-size check.
