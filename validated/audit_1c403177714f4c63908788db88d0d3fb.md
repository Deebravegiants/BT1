### Title
Users permanently lose bridged assets when `normalize_amount` rounds transfer amount to zero — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a NEAR-to-other-chain transfer with an amount smaller than the decimal conversion factor (`10^(origin_decimals - decimals)`), their tokens are irreversibly burned or locked inside `init_transfer_internal` before any normalization check occurs. The subsequent `sign_transfer` call then always panics with `InvalidAmountToTransfer`, and no cancel or recovery path exists, permanently trapping the user's funds.

---

### Finding Description

**Step 1 — Entry point: `init_transfer`**

The public `init_transfer` function (called via `ft_on_transfer` when a user sends tokens to the bridge) validates only that `fee < amount`: [1](#0-0) 

There is no check that the amount, after decimal normalization to the destination chain, would be greater than zero.

**Step 2 — Tokens are burned/locked before normalization**

`init_transfer_internal` immediately burns (for deployed bridge tokens) or locks (for native tokens) the full user amount and returns `U128(0)` to the `ft_on_transfer` caller, meaning the NEP-141 protocol keeps the tokens — no refund is issued: [2](#0-1) 

**Step 3 — `normalize_amount` uses floor division**

The normalization function divides by `10^(origin_decimals - decimals)` with integer (floor) division: [3](#0-2) 

For a token with `origin_decimals = 18` and `decimals = 6` (a common pairing), any amount below `10^12` normalizes to zero.

**Step 4 — `sign_transfer` panics, but funds are already gone**

The trusted relayer calls `sign_transfer`, which normalizes the amount and then requires it to be non-zero: [4](#0-3) 

This panic is permanent — the transfer message stays in storage, the tokens remain burned/locked, and no recovery mechanism exists. The user cannot call `sign_transfer` themselves (it is gated by `#[trusted_relayer]`), and there is no `cancel_transfer` function visible in the contract.

---

### Impact Explanation

**Critical — Irreversible fund lock.**

Any user who initiates a NEAR-to-EVM (or NEAR-to-Solana, etc.) transfer with an amount below the decimal-conversion threshold loses their tokens permanently:
- Deployed bridge tokens are burned and cannot be re-minted.
- Native/locked tokens are locked in the bridge contract with no withdrawal path.

The transfer message is stored but can never be finalized, and the user has no privileged recourse.

---

### Likelihood Explanation

**Medium.** The condition is reachable by any unprivileged user through the standard `ft_transfer_call` → `ft_on_transfer` → `init_transfer` path. It requires only that:
1. The token has a decimal difference between origin and destination chains (e.g., 18 vs 6, a common real-world pairing).
2. The user sends an amount below the conversion threshold (e.g., any amount < `10^12` for an 18→6 decimal token).

No special knowledge, role, or collusion is required. A user unfamiliar with decimal normalization semantics can trigger this accidentally.

---

### Recommendation

Add a normalization check inside `init_transfer` (or at the latest inside `init_transfer_internal`) **before** burning or locking tokens. Retrieve the destination-chain decimals for the token and verify that `normalize_amount(amount - fee, decimals) > 0`; revert with a clear error if not. This mirrors the guard already present in `sign_transfer` but moves it to the point where funds are still safe.

---

### Proof of Concept

1. A token is registered with `origin_decimals = 18`, `decimals = 6` on the destination chain.
2. Alice calls `ft_transfer_call` on the token contract, sending `999_999_999_999` units (≈ 1 micro-token in 18-decimal terms) to the bridge with a valid `InitTransferMsg` targeting an EVM recipient.
3. The bridge's `ft_on_transfer` calls `init_transfer`, which passes the `fee < amount` check and calls `init_transfer_internal`.
4. `init_transfer_internal` burns Alice's `999_999_999_999` tokens and returns `U128(0)` — the NEP-141 protocol keeps the tokens.
5. The relayer calls `sign_transfer` for Alice's transfer ID.
6. `normalize_amount(999_999_999_999, Decimals { origin_decimals: 18, decimals: 6 })` → `999_999_999_999 / 10^12 = 0`.
7. `require!(amount_to_transfer > 0, ...)` panics.
8. Alice's `999_999_999_999` tokens are permanently burned; she receives nothing on the destination chain and has no recovery path.

        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
