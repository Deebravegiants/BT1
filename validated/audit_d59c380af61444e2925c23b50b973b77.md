### Title
`MissingRequiredGuards` Check Downgraded to `runTestOnSignal` in `dijkstraUtxowTransition`, Bypassing Guard Enforcement During Block Replay - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxow.hs`)

### Summary

In the Dijkstra era's `UTXOW` rule, the check that enforces sub-transaction `requiredTopLevelGuards` are present in the top-level transaction's `guards` field is executed with `runTestOnSignal` instead of `runTest`. This is inconsistent with every other security-critical check in the same rule (script validation, datum checks, redeemer checks, script integrity hash, needed witnesses) which all use `runTest`. The `runTestOnSignal` label marks a check as "static" — only depending on the transaction signal, not on ledger state — which causes it to be **skipped entirely during block replay**. An attacker can craft a top-level transaction whose sub-transactions declare `requiredTopLevelGuards` credentials that are absent from the top-level `guards` field, and this violation will be accepted by nodes replaying the chain, causing a deterministic ledger divergence between nodes that validate fresh and nodes that replay.

### Finding Description

The Dijkstra era introduces sub-transactions (`SubTx`) that can declare `requiredTopLevelGuards`: a map of `Credential Guard → StrictMaybe Datum` that the enclosing top-level transaction must include in its own `guards` field. This is the mechanism by which sub-transactions assert that certain guard credentials have been authorized at the top level.

The enforcement rule is in `dijkstraUtxowTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxow.hs`:

```haskell
{- concatMapˡ (λ txSub → mapˢ proj₁ (TopLevelGuardsOf txSub)) (SubTransactionsOf txTop) ⊆ GuardsOf txTop -}
let requiredGuardsBySubTxs =
      foldMap (Map.keysSet . (^. bodyTxL . requiredTopLevelGuardsL)) subTxs
    topLevelGuards = OSet.toSet (txBody ^. guardsTxBodyL)
    missingGuards = requiredGuardsBySubTxs `Set.difference` topLevelGuards
runTestOnSignal $ failureOnNonEmptySet missingGuards MissingRequiredGuards
``` [1](#0-0) 

The check is wrapped in `runTestOnSignal` instead of `runTest`. The distinction is defined in `ValidationMode.hs`:

- `runTest` runs the check unconditionally.
- `runTestOnSignal` labels the check with `lblStatic`, meaning it is only run when the signal (transaction) is being validated for the first time. During block replay (re-application of already-validated blocks), static checks are **skipped**. [2](#0-1) 

Every other security-critical check in the same `dijkstraUtxowTransition` function uses `runTest`:
- `Babbage.validateFailedBabbageScripts` — `runTest`
- `Babbage.babbageMissingScripts` — `runTest`
- `Alonzo.missingRequiredDatums` — `runTest`
- `Alonzo.hasExactSetOfRedeemers` — `runTest`
- `Shelley.validateNeededWitnesses` — `runTest`
- `Babbage.validateScriptsWellFormed` — `runTest`
- `Alonzo.checkScriptIntegrityHash` — `runTest` [3](#0-2) 

Only `validateVerifiedWits` (cryptographic signature verification) and `validateMetadata` (metadata hash consistency) legitimately use `runTestOnSignal`, because those checks depend only on the transaction bytes and not on ledger state — they are genuinely static. The `MissingRequiredGuards` check, however, depends on the **ledger state** (the current UTxO and the sub-transaction bodies embedded in the top-level transaction body), making it non-static and incorrect to label as such.

The `MissingRequiredGuards` check is also explicitly marked as impossible to inject into the `SUBUTXOW` rule:

```haskell
MissingRequiredGuards _ -> error "Impossible: `MissingRequiredGuards` for SUBUTXOW"
``` [4](#0-3) 

This confirms that `MissingRequiredGuards` is exclusively a top-level UTXOW concern, and its enforcement is the sole mechanism preventing sub-transactions from bypassing the guard authorization model.

The `validateDijkstraNativeScript` function, which evaluates `RequireGuard` native scripts, checks `cred \`OSet.member\` guards` where `guards` comes from `tx ^. bodyTxL . guardsTxBodyL`: [5](#0-4) [6](#0-5) 

If a sub-transaction declares `requiredTopLevelGuards = {credX → datum}` but the top-level transaction omits `credX` from its `guards`, the `MissingRequiredGuards` check should reject the transaction. With `runTestOnSignal`, this check is skipped on replay, meaning a block producer can include such a transaction in a block, and replaying nodes will accept it without enforcing the guard constraint.

### Impact Explanation

**High — Deterministic disagreement between honest nodes from ledger rule evaluation.**

A block producer submits a top-level Dijkstra transaction containing sub-transactions whose `requiredTopLevelGuards` declare credentials not present in the top-level `guards`. On first validation (mempool or block application), the `MissingRequiredGuards` check fires and rejects the transaction. However, if the block producer is also a node that can bypass mempool validation (e.g., by constructing a block directly), or if the check is not enforced at the mempool layer, the transaction can be included in a block. When other nodes replay that block, the `runTestOnSignal` label causes the check to be skipped, and the invalid state transition is accepted. This creates a permanent ledger divergence between nodes that validated the block fresh and nodes that replayed it, matching the "High — Deterministic disagreement between honest nodes from ledger rule evaluation" impact category.

Additionally, if the guard authorization model is the security boundary preventing unauthorized script execution or unauthorized sub-transaction effects (e.g., unauthorized minting, spending, or certificate issuance within sub-transactions), bypassing it constitutes an unauthorized ledger state transition.

### Likelihood Explanation

The Dijkstra era is experimental/pre-mainnet. However, the vulnerability class is a straightforward coding error: using `runTestOnSignal` where `runTest` is required. Any block producer running a Dijkstra node can exploit this by constructing a block directly (bypassing mempool). The attacker-controlled entry path is: craft a `TopTx` with sub-transactions declaring `requiredTopLevelGuards` credentials absent from the top-level `guards`, include it in a block, and broadcast the block. Replaying nodes will accept it.

### Recommendation

Change `runTestOnSignal` to `runTest` for the `MissingRequiredGuards` check in `dijkstraUtxowTransition`:

```haskell
-- Before (incorrect):
runTestOnSignal $ failureOnNonEmptySet missingGuards MissingRequiredGuards

-- After (correct):
runTest $ failureOnNonEmptySet missingGuards MissingRequiredGuards
``` [7](#0-6) 

This aligns the guard enforcement check with all other security-critical checks in the same transition rule, ensuring it is enforced both during initial validation and during block replay.

### Proof of Concept

1. Construct a `DijkstraEra` top-level transaction `txTop` with:
   - At least one sub-transaction `txSub` whose body contains `requiredTopLevelGuards = Map.singleton credX SNothing`
   - The top-level `guards` field of `txTop` is **empty** (does not contain `credX`)
2. Submit `txTop` directly as part of a block (bypassing mempool, or in a context where the mempool check is not enforced).
3. On first application, `dijkstraUtxowTransition` computes `missingGuards = {credX}` and calls `runTestOnSignal $ failureOnNonEmptySet {credX} MissingRequiredGuards`. Because this is labeled static, whether it fires depends on the validation mode.
4. When a second node replays the block, the static label causes the check to be skipped entirely, and the block is accepted with the guard constraint violated.
5. The two nodes now have divergent ledger states regarding the validity of this block. [8](#0-7)
