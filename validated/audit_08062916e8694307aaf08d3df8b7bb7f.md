### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Binding It to the Pending Request, Enabling Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` endpoint verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root MPC key, but **never checks that `response.payload_hash` encodes the submitted `request`**. A single Byzantine attested participant (below threshold) can replay a valid threshold signature produced for a previously resolved request R1 to resolve a different, currently pending request R2, delivering a forged foreign-chain verification attestation to the user who submitted R2.

---

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs the following signature check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract verifies only that `signature` is a valid ECDSA signature over `response.payload_hash` under the root key. It does **not** verify that `response.payload_hash` is the correct hash for the submitted `request`.

Per the design specification, the `payload_hash` is supposed to be:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

The `VerifyForeignTransactionResponse` struct carries both `payload_hash` and `signature` as caller-supplied fields: [3](#0-2) 

Because the contract never recomputes or constrains `payload_hash` from the `request` argument, any valid `(payload_hash, signature)` pair — regardless of which request it was originally produced for — will pass the check and cause `resolve_yields_for` to drain the pending queue for the target request R2 with the wrong response data. [4](#0-3) 

This is the direct analog of the stETH report's root cause: one side of a paired state update (resolving the pending yield for R2) is executed without verifying the corresponding invariant (that `payload_hash` encodes R2's actual foreign-chain observation).

---

### Impact Explanation

**High — Forged foreign-chain verification / invalid bridge execution.**

The user who submitted R2 receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes the foreign-chain state of R1 (a different transaction, different extracted values such as a different Bitcoin block hash). Any downstream NEAR smart contract (e.g., an Omnibridge inbound handler) that trusts the MPC attestation to unlock funds will act on the wrong foreign-chain state. Concretely:

- A bridge contract unlocking NEAR-side assets upon Bitcoin finality could be made to accept a signature attesting to a *different* Bitcoin transaction's block hash, enabling invalid bridge execution.
- Because the MPC root key signed H1 (for R1), the signature is cryptographically valid; the forgery is undetectable by any verifier that only checks `verify(sig, payload_hash, root_pk)` without also reconstructing the expected hash from the original request.

This matches the allowed impact: *"Cross-chain replay, forged foreign-chain verification, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

**Moderate.** The attack requires:

1. A single Byzantine attested participant (below threshold) — one compromised or malicious TEE node suffices.
2. At least one previously resolved `verify_foreign_transaction` request whose threshold signature (`sig_over_H1`) is publicly visible on-chain.
3. A concurrently pending request R2 targeting the same domain.

In a production environment processing many bridge requests, conditions 2 and 3 are continuously satisfied. The attacker does not need to forge a threshold signature; they only need to observe one that was already produced and submitted on-chain for any prior request.

---

### Recommendation

The contract must bind `response.payload_hash` to the submitted `request`. Two complementary approaches:

1. **Request-hash prefix commitment**: Change the payload format so that `msg_hash = SHA-256(borsh(request) || SHA-256(borsh(values)))`. The contract can then independently compute `SHA-256(borsh(request))` from the `request` argument and verify that the payload hash is consistent with it (e.g., via a two-level hash structure where the first level is verifiable on-chain).

2. **Include extracted values in the response**: Require nodes to submit the `Vec<ExtractedValue>` alongside the signature. The contract recomputes `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` and asserts it equals `response.payload_hash` before accepting the response. This is the most direct fix and eliminates the trust assumption on the responding node.

---

### Proof of Concept

```
1. User A submits verify_foreign_transaction(R1) for Bitcoin tx X1.
   → R1 is added to pending_verify_foreign_tx_requests.

2. MPC network (threshold) signs:
   H1 = SHA-256(borsh(ForeignTxSignPayload::V1 { request: R1, values: [BlockHash(B1)] }))
   producing threshold signature sig_over_H1.

3. Honest leader submits respond_verify_foreign_tx(R1, {payload_hash: H1, sig: sig_over_H1}).
   → R1 resolved correctly. sig_over_H1 is now publicly visible on-chain.

4. User B submits verify_foreign_transaction(R2) for Bitcoin tx X2.
   → R2 is added to pending_verify_foreign_tx_requests.

5. Byzantine attested participant reads sig_over_H1 from on-chain history.
   Submits: respond_verify_foreign_tx(R2, {payload_hash: H1, sig: sig_over_H1})

6. Contract checks:
   a. Caller is attested participant ✓
   b. Protocol is running ✓
   c. verify_ecdsa(sig_over_H1, H1, root_pk) == true ✓  ← no check that H1 encodes R2
   d. resolve_yields_for(pending_verify_foreign_tx_requests, R2, serialize({H1, sig_over_H1}))

7. User B's yield is resumed with {payload_hash: H1, signature: sig_over_H1}.
   User B's bridge contract receives a valid MPC attestation for Bitcoin tx X1's
   block hash, not X2's — and may unlock NEAR-side assets incorrectly.
``` [5](#0-4)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
