### Title
Missing CKD Output Verification for `AppPublicKey` Variant in `respond_ckd` — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_ckd` performs a cryptographic output check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`, leaving the `AppPublicKey` variant with **no response integrity verification**. A single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will accept and deliver it to the waiting user without any cryptographic validation.

---

### Finding Description

In `respond_ckd`, the response verification branch is:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically binds the `CKDResponse` (`big_y`, `big_c`) to the requester's app public key and the network's root BLS key, enforcing that the response is the genuine threshold-computed output. [1](#0-0) 

For `AppPublicKey`, the arm is a no-op. The `CKDResponse` fields (`big_y: Bls12381G1PublicKey`, `big_c: Bls12381G1PublicKey`) are accepted unconditionally and immediately delivered to the yield-resume promise. [2](#0-1) 

The parallel in the request path (`request_app_private_key`) also skips the `app_public_key_check` for `AppPublicKey`, confirming the asymmetry is structural, not incidental: [3](#0-2) 

The `respond` and `respond_verify_foreign_tx` counterparts both verify the signature against a derived or root public key before resolving yields. `respond_ckd` for `AppPublicKey` has no equivalent guard. [4](#0-3) 

---

### Impact Explanation

**Critical — Unauthorized confidential key derivation output without required participant authorization.**

The threshold model requires that no single participant can unilaterally determine the output of a multi-party computation. For `AppPublicKeyPV` this is enforced on-chain. For `AppPublicKey` it is not. A single Byzantine attested participant can:

1. Observe a pending `AppPublicKey` CKD request in `pending_ckd_requests`.
2. Call `respond_ckd` with an arbitrary `CKDResponse{big_y: ..., big_c: ...}` of their choosing.
3. The contract passes all guards (`assert_caller_is_attested_participant_and_protocol_active`, `is_running_or_resharing`, `accept_requests`) and resolves the yield with the forged response.
4. The requesting user receives incorrect key material derived from the attacker's chosen values rather than the genuine threshold-computed output.

This bypasses the threshold-signature requirement for CKD: one participant out of `n` (where `n-1` may be honest) can forge the output. [5](#0-4) 

---

### Likelihood Explanation

Any single attested participant is a sufficient attacker. The attacker needs only:
- A valid TEE attestation (already required to be a participant).
- Knowledge of a pending `AppPublicKey` CKD request (observable on-chain via `pending_ckd_requests`).

No collusion, no key leakage, no network-level attack is required. The attack is a single on-chain transaction. [6](#0-5) 

---

### Recommendation

Add an equivalent output integrity check for the `AppPublicKey` variant in `respond_ckd`. If `AppPublicKey` does not carry a verifiable binding point (unlike `AppPublicKeyPV`), the contract should either:

1. Require a threshold-quorum of identical responses before resolving the yield (analogous to how `vote_pk` collects threshold votes before accepting a public key), or
2. Derive and verify the expected output from the root BLS key and the request parameters before accepting any single node's response.

The fix should mirror the existing `ckd_output_check` guard already applied to `AppPublicKeyPV`:

```rust
dtos::CKDAppPublicKey::AppPublicKey(app_pk) => {
    if !ckd_output_check_for_plain_key(&request.app_id, &response, app_pk, &public_key) {
        env::panic_str("CKD output check failed for AppPublicKey");
    }
}
``` [1](#0-0) 

---

### Proof of Concept

1. Contract is in `Running` state with a BLS12-381 CKD domain. Attacker is one of `n` attested participants.
2. Honest user calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(some_bls_key)`. A yield is queued in `pending_ckd_requests`.
3. Attacker calls `respond_ckd(request, CKDResponse { big_y: [0u8;48].into(), big_c: [0u8;48].into() })`.
4. `respond_ckd` passes all guards. The `AppPublicKey` match arm is a no-op. `resolve_yields_for` delivers the forged response.
5. The user's callback receives `big_y = [0;48], big_c = [0;48]` — attacker-chosen garbage — as their derived key material, with no error. [7](#0-6)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
