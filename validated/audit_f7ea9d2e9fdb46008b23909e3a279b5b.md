### Title
`directDeposits` Excluded from Preservation-of-Value Check Allows Unbounded ADA Creation — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in the transaction body that allows a transaction author to directly credit arbitrary amounts of ADA into registered account balances. The `ENTITIES` rule applies these deposits to account balances via `applyDirectDeposits`, but the preservation-of-value (`consumed == produced`) check in the UTXO rule does not include `directDeposits` in the `produced` side. An unprivileged transaction author can therefore specify arbitrarily large direct deposits without a corresponding reduction in UTxO outputs, creating ADA out of thin air.

---

### Finding Description

**Vulnerability class:** funds/accounting bug — analogous to the BlueBerry Bank phantom-collateral issue, where a deduction applied in one subsystem is not reflected in the accounting check of another.

**Root cause — step 1: `directDeposits` are applied to account balances**

In `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`, the `ENTITIES` transition rule applies direct deposits after processing certificates:

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts

pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [1](#0-0) 

`applyDirectDeposits` unconditionally adds each specified amount to the matching account balance:

```haskell
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [2](#0-1) 

The only validation performed before applying the deposits is a check that target accounts are registered (`directDepositsMissingAccounts`) and that the network ID matches (`validateWrongNetworkInDirectDeposit`). **No check is made that the deposit amounts are covered by the UTxO inputs.**

**Root cause — step 2: `directDeposits` are absent from the `produced` side of the preservation-of-value equation**

The UTXO rule enforces `consumed == produced` via `updateUTxOStateNoFees`:

```haskell
totalRefunds  = certsTotalRefundsTxBody  pp certState txBody
totalDeposits = certsTotalDepositsTxBody pp certState txBody
depositChange = totalDeposits <-> totalRefunds
``` [3](#0-2) 

`certsTotalDepositsTxBody` for the Dijkstra era is inherited from Conway's `conwayTotalDepositsTxBody`, which sums only certificate deposits (stake key, pool, DRep, proposal):

```haskell
conwayTotalDepositsTxBody pp isPoolRegisted txBody =
  getTotalDepositsTxCerts pp isPoolRegisted (txBody ^. certsTxBodyL)
    <+> conwayProposalsDeposits pp txBody
``` [4](#0-3) 

`directDeposits` are not a certificate type and are not counted by `conwayTotalDepositsTxBody`. They are therefore **absent from `produced`**.

The formal preservation-of-value equation that should hold is:

```
UTxO_inputs + withdrawals + refunds
  == UTxO_outputs + fees + cert_deposits + direct_deposits
```

But the check that is actually enforced is:

```
UTxO_inputs + withdrawals + refunds
  == UTxO_outputs + fees + cert_deposits
```

The `direct_deposits` term on the right-hand side is missing. The UTXO rule's own post-condition assertion confirms the scope of the check:

```haskell
PostCondition "Should preserve value in the UTxO state"
  ( \(TRC (_, us, stAnnTx)) us' ->
      let tx = stAnnTx ^. txStAnnTxG
       in utxoBalance us <> withdrawals (tx ^. bodyTxL) == utxoBalance us'
  )
``` [5](#0-4) 

`utxoBalance` is defined as `Val.inject (utxosDeposited <> utxosFees) <> sumUTxO utxosUtxo` — it does not include account balances. Because `directDeposits` increase account balances without touching `utxosDeposited` or `utxosUtxo`, the assertion passes even when ADA is created.

The same gap exists for sub-transactions via `dijkstraSubEntitiesTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubEntities.hs`:

```haskell
pure $ certStateAfterSubCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [6](#0-5) 

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker can submit a valid-looking Dijkstra transaction that:
1. Spends UTxO inputs totalling N lovelace.
2. Produces UTxO outputs totalling N − fee lovelace (satisfying `consumed == produced`).
3. Specifies `directDeposits` of an arbitrarily large amount M to one or more registered accounts.

After the transaction is accepted, account balances increase by M lovelace while the UTxO decreases by only fee lovelace. The ledger has created M − fee lovelace from nothing, violating the fundamental preservation-of-value invariant that underpins all of Cardano's monetary policy.

---

### Likelihood Explanation

Any transaction author who can submit a Dijkstra-era transaction can exploit this. No privileged role, governance majority, or key compromise is required. The only precondition is that at least one registered account address exists as a deposit target, which is trivially satisfiable (the attacker can register their own stake credential in the same transaction). The `directDeposits` field is part of the serialized transaction body (CDDL field 25), so it is fully attacker-controlled. [7](#0-6) 

---

### Recommendation

Include the sum of `directDeposits` amounts in the `produced` side of the preservation-of-value check. Concretely, `certsTotalDepositsTxBody` (or an equivalent Dijkstra-era override) should add the total of all `directDeposits` entries:

```haskell
totalDirectDeposits txBody =
  fold (unDirectDeposits (txBody ^. directDepositsTxBodyL))
```

This value must be added to `produced` so that the check becomes:

```
consumed == UTxO_outputs + fees + cert_deposits + direct_deposits
```

The same fix must be applied to the sub-transaction path in `dijkstraSubEntitiesTransition`.

---

### Proof of Concept

```
Attacker registers stake credential C (paying keyDeposit D).

Transaction T:
  inputs:  { utxo_in  → 1_000_000 lovelace }
  outputs: { utxo_out → 999_000  lovelace }   -- 1_000 lovelace fee
  fee:     1_000 lovelace
  certs:   []
  directDeposits: { AccountAddress(C) → 1_000_000_000_000 lovelace }

Preservation-of-value check (as currently implemented):
  consumed = 1_000_000
  produced = 999_000 + 1_000 = 1_000_000  ✓  (passes)

Actual ledger state change:
  UTxO:             −1_000 lovelace  (net)
  Fees:             +1_000 lovelace
  Account balance C: +1_000_000_000_000 lovelace  ← created from nothing

Net ADA created: 1_000_000_000_000 lovelace (1 million ADA).
```
