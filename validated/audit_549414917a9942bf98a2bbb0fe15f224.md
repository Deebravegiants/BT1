### Title
Stale Cross-Chain Rate Allows L2 Depositors to Mint Excess rsETH, Diluting Existing Holders - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

---

### Summary

The `CrossChainRateReceiver` stores the last rsETH/ETH rate pushed from L1 via LayerZero. All L2 pool variants (`RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV2ExternalBridge`, etc.) call `getRate()` on this receiver to determine how much rsETH to mint per ETH deposited. There is no staleness check anywhere in this path. When the actual rsETH price on L1 has risen (due to staking rewards accruing to underlying LSTs) but the L2 rate has not been updated, any depositor on L2 receives more rsETH than they are entitled to, diluting all existing rsETH holders.

---

### Finding Description

`CrossChainRateReceiver.getRate()` simply returns the stored `rate` field with no age validation:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol
uint256 public rate;
uint256 public lastUpdated;

function getRate() external view returns (uint256) {
    return rate;   // no staleness check
}
``` [1](#0-0) [2](#0-1) 

This `rate` is only updated when `lzReceive` is called by the LayerZero endpoint, which happens only when someone calls `updateRate()` on the L1 `MultiChainRateProvider` (or `CrossChainRateProvider`) and pays the LayerZero messaging fee:

```solidity
// contracts/cross-chain/MultiChainRateProvider.sol
function updateRate() external payable nonReentrant {
    uint256 latestRate = getLatestRate();
    rate = latestRate;
    lastUpdated = block.timestamp;
    // ... sends via LayerZero
}
``` [3](#0-2) 

Every L2 pool variant uses this stale rate directly in its minting formula. For example, `RSETHPoolV2.viewSwapRsETHAmountAndFee()`:

```solidity
uint256 rsETHToETHrate = getRate();   // potentially stale
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [4](#0-3) 

The same pattern is replicated in `RSETHPoolV3`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolV2NBA`. [5](#0-4) [6](#0-5) 

The L1 `RSETHMultiChainRateProvider.getLatestRate()` reads `ILRTOracle(rsETHPriceOracle).rsETHPrice()`, which is itself a cached value updated by `updateRSETHPrice()`: [7](#0-6) [8](#0-7) 

So the full chain is: `LRTOracle.rsETHPrice` (cached) → `RSETHMultiChainRateProvider.getLatestRate()` → LayerZero message → `CrossChainRateReceiver.rate` (cached) → L2 pool minting. Both ends are cached with no on-chain freshness enforcement.

---

### Impact Explanation

When the true rsETH/ETH rate has risen (e.g., from 1.00 to 1.05 due to staking rewards) but the L2 receiver still holds the old rate of 1.00:

- A depositor sending 1 ETH receives `1e18 / 1.00e18 = 1.0 rsETH`
- They should receive `1e18 / 1.05e18 ≈ 0.952 rsETH`
- The excess 0.048 rsETH is minted from thin air, diluting every existing rsETH holder proportionally

This is a direct theft of accrued yield from existing holders. The `wrsETH.mint()` call issues real, redeemable rsETH backed by no additional ETH deposited into the protocol. [9](#0-8) 

---

### Likelihood Explanation

- The L2 rate is only updated when someone pays LayerZero fees to call `updateRate()`. During periods of high gas prices or low keeper activity, the rate can lag by hours or days.
- rsETH accrues staking rewards continuously from underlying LSTs (stETH, ETHx, etc.), so the true rate drifts upward predictably over time.
- An attacker can monitor the L1 `rsETHPrice` vs. the L2 `CrossChainRateReceiver.rate` and deposit on L2 precisely when the gap is largest, before anyone pushes an update.
- No special privileges are required; any unprivileged depositor can exploit this.

---

### Recommendation

Add a maximum staleness check in `CrossChainRateReceiver.getRate()` (or in each pool's minting path):

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
``` [10](#0-9) 

Alternatively, integrate a Chainlink or other on-chain oracle as a fallback or sanity bound on the L2 side.

---

### Proof of Concept

1. At T=0, L1 `rsETHPrice = 1.00e18`. Rate is pushed to L2 receiver. `CrossChainRateReceiver.rate = 1.00e18`, `lastUpdated = T`.
2. Over the next 48 hours, staking rewards accrue. L1 `rsETHPrice` rises to `1.05e18`. No one pays to push the update to L2.
3. At T+48h, attacker calls `RSETHPoolV2.deposit{value: 100 ether}("")`.
4. `viewSwapRsETHAmountAndFee(100e18)` fetches `getRate() = 1.00e18` (stale).
5. `rsETHAmount = 100e18 * 1e18 / 1.00e18 = 100e18` rsETH minted.
6. Correct amount would be `100e18 * 1e18 / 1.05e18 ≈ 95.24e18` rsETH.
7. Attacker receives ~4.76 excess rsETH. All existing holders are diluted by ~4.76 / (totalSupply + 100) of their holdings.
8. Attacker can immediately bridge the rsETH to L1 and redeem at the true 1.05 rate, extracting ~5 ETH of value from existing holders. [11](#0-10) [4](#0-3)

        uint256 latestRate = getLatestRate();

        rate = latestRate;

        lastUpdated = block.timestamp;

        bytes memory _payload = abi.encode(latestRate);

        uint256 rateReceiversLength = rateReceivers.length;

        for (uint256 i; i < rateReceiversLength;) {
            uint16 dstChainId = uint16(rateReceivers[i]._chainId);

            bytes memory remoteAndLocalAddresses = abi.encodePacked(rateReceivers[i]._contract, address(this));

            (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
                .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

            ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
                dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
            );

            unchecked {
                ++i;
            }
        }

        emit RateUpdated(rate);
    }
