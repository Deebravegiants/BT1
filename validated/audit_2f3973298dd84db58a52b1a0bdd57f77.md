I need to carefully trace the `valid_entries` logic and verify the off-by-one claim against the actual code.

### Title
Off-by-one in `AllowedDockerImageHashes::valid_entries` includes the most-recently-expired TEE image hash as valid — (`crates/contract/src/tee/proposal.rs`)

---

### Summary

`valid_entries` uses `rposition` to find the rightmost expired entry, then slices from that index inclusive (`get(cutoff_index..)`). This means the expired entry at `cutoff_index` is always returned as valid. A Byzantine participant running an image whose grace period has fully elapsed continues to pass `reverify_participants` and remains an authorized threshold signer.

---

### Finding Description

The logic in `valid_entries`:

```rust
let cutoff_index = self
    .allowed_tee_proposals
    .iter()
    .rposition(|allowed_docker_image| {
        ...
        grace_period_deadline < current_time   // true = expired
    })
    .unwrap_or(0);

self.allowed_tee_proposals
    .get(cutoff_index..)   // ← includes the expired entry at cutoff_index
    .unwrap_or(&[])
    .to_vec()
``` [1](#0-0) 

`rposition` returns the index of the **last** (rightmost) entry satisfying the predicate — i.e., the last expired entry. `get(cutoff_index..)` is an inclusive range, so that expired entry is included in the output.

**Concrete trace** with `allowed_tee_proposals = [H1 (index 0, expired), H2 (index 1, valid)]`:

| Step | Action | Result |
|------|--------|--------|
| `rposition` checks H2 (idx 1) | `grace_period_deadline < now` → false | skip |
| `rposition` checks H1 (idx 0) | `grace_period_deadline < now` → true | returns `Some(0)` |
| `cutoff_index = 0` | | |
| `get(0..)` | returns `[H1, H2]` | **H1 (expired) included** |

The correct slice is `get(cutoff_index + 1..)`.

The existing `test_clean_expired` does **not** catch this because it sets the clock so that **both** H1 and H2 are expired at test time, causing `rposition` to land on H2 (index 1) and `get(1..)` to return only `[H2]` — the test passes for the wrong reason. [2](#0-1) 

---

### Impact Explanation

`valid_entries` feeds directly into `get_allowed_mpc_docker_image_hashes`, which is used in both `add_participant` and `reverify_participants`: [3](#0-2) [4](#0-3) 

A participant running image H1 whose grace period has fully elapsed will still pass `reverify_participants` because H1 is incorrectly present in the allowed-hash list. `reverify_and_cleanup_participants` will not remove them: [5](#0-4) 

This is a direct **attestation/TEE-authorization bypass**: the contract's enforcement that all signers must run a currently-approved image is defeated for the most-recently-expired hash, for as long as a successor hash exists and has not itself expired.

---

### Likelihood Explanation

The precondition is the normal operational state during any TEE image upgrade cycle: H1 is whitelisted, threshold votes whitelist H2, H1's grace period elapses. No timestamp manipulation is required; the bug manifests from the natural passage of time. A single Byzantine participant (below signing threshold) who refuses to upgrade exploits this automatically.

---

### Recommendation

Change the slice to exclude the expired entry at `cutoff_index`:

```rust
// Before
self.allowed_tee_proposals.get(cutoff_index..)

// After — skip the expired entry; keep at least one if all are expired
let start = if cutoff_index + 1 < self.allowed_tee_proposals.len() {
    cutoff_index + 1
} else {
    cutoff_index   // all expired: preserve the "keep at least one" invariant
};
self.allowed_tee_proposals.get(start..)
```

Add a unit test with exactly two entries where H1 is expired and H2 is valid, asserting H1 is absent from the result.

    pub fn reverify_and_cleanup_participants(
        &mut self,
        participants: &Participants,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeValidationResult {
        self.allowed_docker_image_hashes
            .cleanup_expired_hashes(tee_upgrade_deadline_duration);

        let participants_with_valid_attestation: Vec<_> = participants
            .participants()
            .iter()
            .filter(|(_, _, participant_info)| {
                // Use the stored NodeId (keyed by TLS public key) so the real
                // `account_public_key` participates in re-verification. If
                // there is no stored attestation for this TLS key, the
                // participant is invalid.
                let Some(node_id) = self.find_node_id_by_tls_key(&participant_info.tls_public_key)
                else {
                    return false;
                };

                let tee_status =
                    self.reverify_participants(&node_id, tee_upgrade_deadline_duration);

                matches!(tee_status, TeeQuoteStatus::Valid)
            })
            .cloned()
            .collect();

        if participants_with_valid_attestation.len() != participants.len() {
            let participants_with_valid_attestation =
                Participants::init(participants.next_id(), participants_with_valid_attestation);

            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            }
        } else {
            TeeValidationResult::Full
        }
    }
