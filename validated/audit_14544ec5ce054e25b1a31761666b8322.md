### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the On-Chain Request, Enabling Cross-Request Signature Replay - (File: crates/contract/src/lib.rs)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over the caller-supplied `response.payload_hash`, but never checks that `payload_hash` was actually derived from the `request` argument that identifies the pending foreign-chain verification. A single malicious attested participant (strictly below the signing threshold) can replay any previously observed valid `{payload_hash, signature}` pair against a different pending request, causing all callers waiting for that request to receive a forged `VerifyForeignTransactionResponse`.

---

### Finding Description

The `respond_verify_foreign_tx` function at `crates/contract/src/lib.rs:691-754` performs the following signature check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // taken from the response, not the request
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,                                            // root public key
)
.is_ok()
``` [1](#0-0) 

The contract only asserts that `signature` is a valid ECDSA signature over `response.payload_hash` under the root public key. It never verifies that `response.payload_hash` was computed from the `request` that is being resolved.

The intended invariant is that `payload_hash = SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))`, where `request` is the `ForeignChainRpcRequest` embedded in the pending `VerifyForeignTransactionRequest`. [2](#0-1) 

The node-side code correctly computes this hash before submitting the response: [3](#0-2) 

But the contract never re-derives or cross-checks the hash against the `request` it received.

**Contrast with `respond`** (regular signature requests): there, the payload is read directly from `request.payload` — it is structurally impossible to supply a mismatched payload:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(signature_response, payload_hash, ...)
``` [4](#0-3) 

No equivalent binding exists in `respond_verify_foreign_tx`.

After the signature check passes, `resolve_yields_for` is called, which drains **all** queued yield promises for `request` and delivers the raw serialized `response` (including the attacker-chosen `payload_hash`) to every waiting caller: [5](#0-4) 

---

### Impact Explanation

**Impact: High — forged foreign-chain verification enabling invalid bridge execution.**

A malicious attested participant (one node, below threshold) can:

1. Observe any previously finalized `respond_verify_foreign_tx` transaction on-chain. All such calls are public NEAR transactions; the attacker extracts `{payload_hash_A, sig_A}` for `request_A` (e.g., Bitcoin `tx_id_A` with `block_hash_X`).
2. Wait for a new `verify_foreign_transaction` request `request_B` (e.g., Bitcoin `tx_id_B` with `block_hash_Y`) to appear in the pending map.
3. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
4. The contract accepts the call: the caller is an attested participant, and `sig_A` is a valid signature over `payload_hash_A` under the root key.
5. Every NEAR account that submitted `request_B` receives `VerifyForeignTransactionResponse { payload_hash: payload_hash_A, signature: sig_A }` — a response that attests to `request_A`'s foreign-chain state, not `request_B`'s.

Bridge contracts or NEAR smart contracts consuming this response and trusting the MPC attestation without independently recomputing the expected `payload_hash` from their own request will accept a forged verification. This can enable:
- Claiming a bridge deposit on NEAR for a foreign-chain transaction that was never actually verified (or was verified with different extracted values, e.g., a different block hash or confirmation count).
- Double-spend: reusing the attestation of one finalized foreign transaction to unlock funds for a different, unverified transaction.

---

### Likelihood Explanation

**Likelihood: Medium.**

- Requires exactly one malicious attested participant — strictly below the signing threshold. No collusion is needed.
- The attacker does not need to forge any cryptographic material; they only replay an existing, legitimately produced signature that is already public on-chain.
- The attack window is any time a new `verify_foreign_transaction` request is pending, which is the normal operating state of the bridge.
- The only prerequisite is that at least one prior `respond_verify_foreign_tx` has been finalized (trivially satisfied in production).

---

### Recommendation

The contract must verify that `response.payload_hash` is consistent with `request`. The cleanest fix is to include the extracted `values` in the response and have the contract recompute and compare the hash:

```rust
// In VerifyForeignTransactionResponse, add:
pub values: Vec<ExtractedValue>,

// In respond_verify_foreign_tx, after receiving the response:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),
}).compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
// then verify signature over response.payload_hash as before
```

This mirrors how `respond` binds the signature to `request.payload` without relying on the caller to supply the correct preimage.

### Proof of Concept

Using the existing test harness pattern from `crates/contract/src/lib.rs`:

1. Complete a legitimate `verify_foreign_transaction` for `request_A` (Bitcoin `tx_id_A`). Record the resulting `{payload_hash_A, sig_A}` from the on-chain `respond_verify_foreign_tx` call.

2. Submit a new `verify_foreign_transaction` for `request_B` (Bitcoin `tx_id_B`, different tx).

3. As a single attested participant, call:
   ```rust
   contract.respond_verify_foreign_tx(
       request_B,                                          // different request
       VerifyForeignTransactionResponse {
           payload_hash: payload_hash_A,                  // hash of request_A
           signature: sig_A,                              // valid sig over payload_hash_A
       },
   )
   ```

4. The call succeeds (signature check passes, `request_B` is in the pending map).

5. All callers waiting for `request_B` receive `{payload_hash_A, sig_A}` — a response that attests to `request_A`'s foreign-chain state.

The contract's `respond_verify_foreign_tx` at line 718 accepts any `payload_hash` for which a valid root-key signature exists, regardless of whether it was derived from the `request` being resolved. [6](#0-5) [7](#0-6) [2](#0-1)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
