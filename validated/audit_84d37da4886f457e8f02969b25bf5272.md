### Title
Token-2022 TransferFee Extension Causes Recipient Shortfall and Accounting Divergence in `finalize_transfer` — (`solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

### Summary

The native-path of `FinalizeTransfer::process` calls `transfer_checked` with the full `data.amount` and then unconditionally posts `FinalizeTransferResponse { amount: data.amount }` to Wormhole. For any Token-2022 mint that carries the `TransferFee` extension, the SPL Token-2022 program silently withholds a fee from the recipient's token account during `transfer_checked`, so the recipient's spendable balance is `data.amount − fee_withheld` while NEAR is told the full `data.amount` was delivered. No check for the `TransferFee` extension exists anywhere in the registration (`log_metadata`) or settlement (`finalize_transfer`) path.

### Finding Description

**Step 1 — Vault registration has no TransferFee guard.**

`log_metadata` creates the vault PDA for any mint whose `mint_authority` is not the bridge authority. The only constraint is:

```rust
constraint = !mint.mint_authority.contains(authority.key),
``` [1](#0-0) 

There is no inspection of Token-2022 extensions. A mint with `TransferFeeConfig` set to any non-zero basis-point rate passes this check and gets a vault.

**Step 2 — `transfer_checked` silently withholds fees.**

In the native path of `process`, the program calls:

```rust
transfer_checked(
    CpiContext::new_with_signer(...),
    data.amount.try_into()...?,
    self.mint.decimals,
)?;
``` [2](#0-1) 

Under Token-2022 `TransferFee`, the SPL runtime deducts `fee_withheld = data.amount × fee_bps / 10_000` from the recipient's token account at the moment of transfer. The recipient's *spendable* balance increases by only `data.amount − fee_withheld`. The withheld amount sits in the recipient's account but is exclusively harvestable by the mint's fee authority — it is not accessible to the recipient.

**Step 3 — The Wormhole response always encodes the full amount.**

Immediately after the transfer, regardless of what the recipient actually received:

```rust
let payload = FinalizeTransferResponse {
    token: self.mint.key(),
    amount: data.amount,          // ← always the pre-fee amount
    fee_recipient: data.fee_recipient.unwrap_or_default(),
    transfer_id: data.transfer_id,
}
.serialize_for_near(())?;
self.common.post_message(payload)?;
``` [3](#0-2) 

NEAR receives `amount = data.amount` and marks the transfer complete. There is no post-transfer balance read, no fee-withheld reconciliation, and no revert path.

**Step 4 — The SECURITY.md gap.**

The project's security notes acknowledge that Token-2022 *transfer hooks* are unsupported and "fail at runtime (denial, not fund loss)": [4](#0-3) 

`TransferFee` is a completely different extension. It requires **no extra account metas** and is applied transparently by the Token-2022 runtime inside `transfer_checked` — it does not cause a runtime failure. It is not mentioned in SECURITY.md and is not guarded against anywhere in the program.

### Impact Explanation

- **Recipient shortfall**: For every `finalize_transfer` on a fee-bearing mint, the recipient receives `data.amount − fee_withheld` spendable tokens while NEAR records `data.amount` as settled. The delta is permanently withheld in the recipient's ATA and flows to the mint's fee authority.
- **Backing guarantee broken**: The cross-chain invariant — *amount credited on NEAR = amount received on Solana* — is violated. NEAR's ledger shows a larger outstanding liability than the Solana side can honour.
- **Nonce consumed, no retry**: `use_nonce` marks `destination_nonce` used before the transfer executes. The shortfall cannot be corrected by resubmitting; the settlement is final.
- **Compounding with `init_transfer`**: The same `transfer_checked` call in `init_transfer` means the vault receives `data.amount − fee_withheld` when a user deposits, yet NEAR credits the full `data.amount`. Both directions compound the divergence.

### Likelihood Explanation

- Any Token-2022 mint with `TransferFeeConfig` can be registered via `log_metadata` — no privileged action required.
- Real-world Token-2022 tokens with transfer fees exist (e.g., tokens using the fee extension for protocol revenue).
- An attacker who controls the fee authority of such a mint can set the fee to the maximum (10 000 bps = 100%) and drain the entire transferred amount into the withheld account on every `finalize_transfer`, while NEAR marks each transfer as fully settled.
- Even without a malicious fee authority, any legitimate fee-bearing token registered on the bridge silently misbehaves.

### Recommendation

1. **Reject fee-bearing mints at registration time.** In `log_metadata`, unpack the Token-2022 mint data and reject any mint that carries a `TransferFeeConfig` extension with a non-zero maximum fee:

   ```rust
   if let Ok(fee_config) = mint_with_extension.get_extension::<TransferFeeConfig>() {
       require!(
           u16::from(fee_config.older_transfer_fee.transfer_fee_basis_points) == 0
               && u16::from(fee_config.newer_transfer_fee.transfer_fee_basis_points) == 0,
           ErrorCode::UnsupportedMintExtension
       );
   }
   ```

2. **Alternatively, measure actual received amount.** Read the recipient's token account balance before and after `transfer_checked` and use the delta — not `data.amount` — in `FinalizeTransferResponse`. This is more complex but allows fee-bearing tokens to be supported with correct accounting.

3. **Add the same guard to `init_transfer`** to prevent the symmetric divergence on the deposit side.

### Proof of Concept

```
1. Deploy a Token-2022 mint M with TransferFeeConfig = 100 bps (1%), fee_authority = attacker.
2. Call log_metadata(M) → vault PDA is created. No error.
3. Fund the vault with 10_000 tokens (e.g., via a direct SPL transfer from any holder).
4. On NEAR, initiate a transfer of 10_000 M-tokens to a Solana recipient R.
   NEAR locks 10_000 tokens and emits a signed FinalizeTransferPayload { amount: 10_000, ... }.
5. Relayer submits finalize_transfer with the signed payload.
   - use_nonce marks destination_nonce used.
   - transfer_checked(vault → R_ATA, amount=10_000) executes.
     Token-2022 withholds 100 tokens; R_ATA.amount = 9_900 spendable, 100 withheld.
   - FinalizeTransferResponse { amount: 10_000 } is posted to Wormhole.
6. NEAR receives the VAA, reads amount=10_000, marks transfer complete.
7. Assert: R_ATA spendable balance == 9_900 (not 10_000).
8. Assert: Wormhole VAA encodes amount == 10_000.
9. Assert: destination_nonce is consumed; no retry possible.
10. Attacker calls harvestWithheldTokensToMint / harvestWithheldTokensToAuthority
    and collects 100 tokens. NEAR's ledger is permanently 100 tokens over-credited.
```

The delta is neither refunded nor reconciled. Repeating across many transfers scales the unbacked supply linearly with volume.
