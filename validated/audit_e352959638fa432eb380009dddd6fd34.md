### Title
Unbounded Loop Over Attacker-Controlled `WeightProof` Lists Blocks Full Node Event Loop During Sync — (`File: chia/full_node/weight_proof.py`)

### Summary
A malicious peer can send a crafted `RespondProofOfWeight` message containing a `WeightProof` with an arbitrarily large `sub_epochs` or `recent_chain_data` list. The full node's `validate_weight_proof` calls `_validate_sub_epoch_summaries` **synchronously** (no `await`) before offloading work to a subprocess, causing `_map_sub_epoch_summaries` and `_validate_summaries_weight` to iterate over the entire attacker-controlled list while blocking the asyncio event loop. During this time the node cannot process incoming blocks, transactions, or peer messages.

### Finding Description

`validate_weight_proof` in `chia/full_node/weight_proof.py` calls `_validate_sub_epoch_summaries` synchronously on line 611, before any executor is involved:

```python
summaries, sub_epoch_weight_list = _validate_sub_epoch_summaries(self.constants, weight_proof)
await asyncio.sleep(0)  # break up otherwise multi-second sync code
``` [1](#0-0) 

`_validate_sub_epoch_summaries` calls `_map_sub_epoch_summaries` which iterates over every element of `weight_proof.sub_epochs` with no upper-bound check:

```python
for idx, data in enumerate(sub_epoch_data):
    ses = SubEpochSummary(...)
    ...
    ses_hash = std_hash(ses)
    summaries.append(ses)
``` [2](#0-1) 

`_validate_summaries_weight`, also called inside `_validate_sub_epoch_summaries`, iterates over the entire `weight_proof.recent_chain_data` list with no size limit:

```python
for block in weight_proof.recent_chain_data:
    if block.reward_chain_block.height == ses_end_height:
        curr = block
``` [3](#0-2) 

The `WeightProof` type declares both fields as unbounded lists:

```python
class WeightProof(Streamable):
    sub_epochs: list[SubEpochData]
    sub_epoch_segments: list[SubEpochChallengeSegment]
    recent_chain_data: list[HeaderBlock]
``` [4](#0-3) 

The only pre-validation checks in `request_validate_wp` verify the **last** element's height and weight, not the list sizes:

```python
if response.wp.recent_chain_data[-1].reward_chain_block.height != peak_height:
    ...
if response.wp.recent_chain_data[-1].reward_chain_block.weight != peak_weight:
    ...
``` [5](#0-4) 

An attacker can craft a proof where the last `recent_chain_data` entry is valid (correct height/weight) but the list contains millions of preceding entries, and `sub_epochs` contains millions of entries. Both loops run synchronously in the event loop before any timeout or subprocess boundary is reached.

### Impact Explanation

While the node is executing the synchronous `_map_sub_epoch_summaries` / `_validate_summaries_weight` loops, the asyncio event loop is fully blocked. The node cannot:
- Accept or process incoming `FullBlock` messages from peers
- Respond to signage point or end-of-slot messages
- Process mempool transactions
- Respond to other peer connections

This constitutes a **long-lived inability for honest nodes to process valid blocks and sync updates** under normal network assumptions, matching the High impact tier.

### Likelihood Explanation

Any peer that the syncing node selects for weight proof download can trigger this. The attacker only needs to be one of the peers the victim node connects to during sync. No keys, admin access, or cryptographic breaks are required — only the ability to respond to a `RequestProofOfWeight` message with a crafted `RespondProofOfWeight`.

### Recommendation

1. **Enforce an upper bound on `weight_proof.sub_epochs` and `weight_proof.recent_chain_data` sizes** before calling any validation loop. The expected maximum number of sub-epochs is bounded by `peak_height / SUB_EPOCH_BLOCKS`; reject proofs exceeding a reasonable multiple of this.
2. **Move `_validate_sub_epoch_summaries` into the `ProcessPoolExecutor`** alongside the rest of the heavy validation, so it cannot block the event loop regardless of input size.
3. **Add periodic `await asyncio.sleep(0)` yields inside `_map_sub_epoch_summaries`** as a short-term mitigation if the above refactor is not immediately feasible.

### Proof of Concept

1. Attacker node receives `RequestProofOfWeight(peak_height, peak_header_hash)` from a syncing victim.
2. Attacker constructs a `WeightProof` where:
   - `recent_chain_data[-1]` has the correct `height` and `weight` (copied from a real block header)
   - `recent_chain_data` contains 10,000,000 dummy `HeaderBlock` entries before the valid last entry
   - `sub_epochs` contains 10,000,000 dummy `SubEpochData` entries
3. Attacker sends `RespondProofOfWeight(weight_proof, peak_header_hash)`.
4. Victim's `request_validate_wp` passes the height/weight checks and calls `validate_weight_proof(response.wp)`.
5. `_validate_sub_epoch_summaries` → `_map_sub_epoch_summaries` iterates 10,000,000 times synchronously, blocking the event loop for minutes.
6. During this window the victim node drops all incoming block and transaction messages, stalling its chain tip.

def _map_sub_epoch_summaries(
    sub_blocks_for_se: uint32,
    ses_hash: bytes32,
    sub_epoch_data: list[SubEpochData],
    curr_difficulty: uint64,
) -> tuple[list[SubEpochSummary], uint128, list[uint128]]:
    total_weight: uint128 = uint128(0)
    summaries: list[SubEpochSummary] = []
    sub_epoch_weight_list: list[uint128] = []
    for idx, data in enumerate(sub_epoch_data):
        ses = SubEpochSummary(
            ses_hash,
            data.reward_chain_hash,
            data.num_blocks_overflow,
            data.new_difficulty,
            data.new_sub_slot_iters,
            data.challenge_merkle_root,
        )

        if idx < len(sub_epoch_data) - 1:
            delta = 0
            if idx > 0:
                delta = data.num_blocks_overflow
            log.debug(f"sub epoch {idx} start weight is {total_weight + curr_difficulty} ")
            sub_epoch_weight_list.append(uint128(total_weight + curr_difficulty))
            total_weight = uint128(
                total_weight
                + curr_difficulty * (sub_blocks_for_se + sub_epoch_data[idx + 1].num_blocks_overflow - delta)
            )

        # if new epoch update diff and iters
        if data.new_difficulty is not None:
            curr_difficulty = data.new_difficulty

        # add to dict
        summaries.append(ses)
        ses_hash = std_hash(ses)
    # add last sub epoch weight
    sub_epoch_weight_list.append(uint128(total_weight + curr_difficulty))
    return summaries, total_weight, sub_epoch_weight_list
```
