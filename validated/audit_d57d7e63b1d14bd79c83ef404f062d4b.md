### Title
Single Attested Participant Can Deliver Arbitrary CKD Response for Legacy `AppPublicKey` Requests Without On-Chain Verification - (File: `crates/contract/src/lib.rs`)

### Summary
`respond_ckd` in the MPC contract performs **no cryptographic verification** of the supplied `CKDResponse` when the pending request used the legacy `AppPublicKey` variant. A single attested participant — strictly below the signing threshold — can race the honest MPC network, submit any fabricated `(big_c, big_y)` pair, and have the contract accept and deliver it to the waiting user. Setting `big_y` to the G1 identity element reduces the user's derived secret to `big_c` alone, a value the attacker chose and therefore knows, constituting a full confidential-key disclosure.

### Finding Description
`respond_ckd` (lines 653–689, `crates/contract/src/lib.rs`) handles two `app_public_key` variants differently:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For the **publicly-verifiable** (`AppPublicKeyPV`) variant the contract calls `ckd_output_check`, which verifies the BLS12-381 pairing equation and confirms the response is consistent with the MPC master public key. For the **legacy** (`AppPublicKey`) variant the branch is an empty no-op: the contract immediately proceeds to `resolve_yields_for`, resuming every queued yield with whatever bytes the caller supplied.

The only guards before that point are:
- `assert_caller_is_signer()` — caller's signer key must equal predecessor key (prevents confused-deputy forwarding).
- `assert_caller_is_attested_participant_and_protocol_active()` — caller must be an attested participant in the current epoch.

Neither guard constrains the *content* of `response`. A single attested participant satisfies both guards and can supply `big_y = G1_identity` and `big_c = P` for any G1 point `P` of their choice.

### Impact Explanation
The user's application computes the derived secret as:

```
s = big_c − a · big_y
```

where `a` is the user's ephemeral private scalar. When `big_y = 0` (G1 identity):

```
s = big_c − a · 0 = big_c
```

The attacker chose `big_c`, so they know `s` exactly. The user then feeds `s` into HKDF to produce their application key. Because the attacker knows `s`, they derive the identical key — a complete confidential-key disclosure for that `(account_id, derivation_path)` pair.

Even without the identity-point trick, the attacker can deliver any garbage `(big_c, big_y)` pair, causing the user to derive a key that is neither the correct MPC-computed secret nor one the attacker knows — silently corrupting the user's deterministic key without any on-chain error. The request is simultaneously removed from `pending_ckd_requests` by `resolve_yields_for`, so honest nodes can no longer respond; the forged answer is final.

This bypasses the threshold-signature requirement: the MPC protocol is designed so that no fewer than `t` participants can reconstruct or influence the derived secret, yet a single participant can unilaterally determine the output delivered to the user for all legacy CKD requests.

### Likelihood Explanation
- Any single attested participant can execute this attack; no collusion is required.
- The attacker monitors the NEAR chain for `request_app_private_key` calls using the legacy `AppPublicKey` variant (visible in transaction arguments).
- The attacker submits `respond_ckd` with a crafted response immediately, before the honest MPC nodes finish the threshold protocol (which involves multiple rounds of P2P communication).
- The race window is realistic: threshold signing takes multiple network round-trips, while the attacker's forged `respond_ckd` is a single on-chain transaction.
- Legacy `AppPublicKey` requests are the default variant shown in the contract README and the `ckd-example-cli`, making them the common case in production.

### Recommendation
Apply the same on-chain verification to the legacy `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. Concretely:

1. **Require the publicly-verifiable variant** for all new CKD requests, deprecating `AppPublicKey`. The PV variant already provides the necessary on-chain pairing check.
2. **If the legacy variant must be retained**, add a BLS12-381 pairing check inside the `AppPublicKey` branch of `respond_ckd` to verify `e(big_c − a·big_y, G2) = e(H(app_id), msk_pk)` — analogous to `ckd_output_check` — before calling `resolve_yields_for`.
3. **Validate that `big_y` is not the G1 identity element** in both variants to close the zero-blinding attack path.

### Proof of Concept
1. User calls `request_app_private_key` with `AppPublicKey` variant; a pending CKD yield is registered in `pending_ckd_requests`.
2. Attacker (single attested participant) observes the transaction on-chain and immediately calls:
   ```
   respond_ckd(
     request = <matching CKDRequest>,
     response = CKDResponse {
       big_y: G1_identity,   // 48-byte encoding of the point at infinity
       big_c: P,             // any G1 point the attacker chooses
     }
   )
   ```
3. `respond_ckd` passes both guards (attested participant, signer == predecessor), enters the `AppPublicKey` branch (empty no-op), and calls `resolve_yields_for`, which resumes the user's yield with the forged bytes and removes the request from the map.
4. The user's application receives `(big_c=P, big_y=0)` and computes `s = P − a·0 = P`.
5. The attacker, who chose `P`, derives the identical application key via HKDF(`P`, …), achieving full confidential-key disclosure without any threshold cooperation. [1](#0-0) [2](#0-1) [3](#0-2)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
