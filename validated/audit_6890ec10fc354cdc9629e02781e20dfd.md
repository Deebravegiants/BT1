### Title
Receipt Size Limit Bypass via `promise_return` and `value_return` — (`runtime/runtime/src/lib.rs`, `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

An unprivileged user can deploy a NEAR contract that causes the runtime to produce and forward receipts exceeding `max_receipt_size` (4 MB). The root cause is that receipt size validation occurs before `output_data_receivers` are appended to a receipt (in the `promise_return` path), and that `DataReceipt` size is never validated against `max_receipt_size` (in the `value_return` path). This is a confirmed, open bug (nearcore issue #12606) with a concrete attacker-controlled entry path.

---

### Finding Description

**Path 1 — `promise_return` (ReceiptIndex result)**

When a contract executes `promise_return(C)`, the runtime validates the new receipt `C` for size during action execution at: [1](#0-0) 

This validation uses `ValidateReceiptMode::NewReceipt`, which enforces `max_receipt_size`. However, **after** this validation, the runtime appends the parent receipt's `output_data_receivers` to receipt `C`: [2](#0-1) 

If receipt `C` was crafted to be exactly at `max_receipt_size` (e.g., by filling its `args` field), appending even a single `DataReceiver` (≈ 40 bytes) pushes it over the limit. The size check is never re-run after this mutation.

**Path 2 — `value_return` (Value result)**

`value_return` enforces `max_length_returned_data` (4 MB) but does **not** check the resulting `DataReceipt` size against `max_receipt_size`: [3](#0-2) 

A contract returning exactly 4 MB of data causes the runtime to wrap it in a `DataReceipt` envelope. The borsh-serialized `DataReceipt` (with its `Receipt` wrapper, predecessor/receiver IDs, etc.) exceeds `max_receipt_size`. The `validate_data_receipt` function only checks `max_length_returned_data`, not the total receipt size: [4](#0-3) 

The codebase explicitly acknowledges this as a bug and uses a relaxed `ValidateReceiptMode::ExistingReceipt` mode to tolerate oversized receipts already in state: [5](#0-4) 

---

### Impact Explanation

Oversized receipts (exceeding `max_receipt_size`) bypass the protocol's size invariant and are stored in the trie and forwarded cross-shard. Concrete consequences:

1. **Protocol invariant violation**: The `max_receipt_size` limit — a protocol parameter — is not enforced, allowing receipts larger than 4 MB to enter the delayed receipt queue and outgoing receipt buffers.
2. **Stateless validation witness inflation**: Oversized receipts included in chunk state witnesses can push the witness past `main_storage_proof_size_soft_limit` (4 MB) or `per_receipt_storage_proof_size_limit`, potentially causing stateless validation failures or inconsistencies across validators.
3. **Congestion control undercount**: `DataReceipt` congestion gas is counted as `Gas::ZERO` by design, so oversized data receipts are invisible to the congestion control system, allowing a user to inflate the delayed receipt queue's actual byte size beyond what `receipt_bytes` tracks. [6](#0-5) 

---

### Likelihood Explanation

Any unprivileged user can trigger this by:
1. Deploying a contract (standard `DeployContract` action).
2. Calling a method that either:
   - Creates a promise chain `A → B`, where `A` creates receipt `C` at exactly `max_receipt_size` and calls `promise_return(C)`, or
   - Creates a promise chain `A → B`, where `A` calls `value_return` with exactly `max_length_returned_data` bytes.

The test suite confirms this is reproducible and that oversized receipts do occur in practice: [7](#0-6) [8](#0-7) 

---

### Recommendation

1. **Re-validate receipt size after appending `output_data_receivers`** in the `promise_return` path (lines 1028–1037 of `runtime/runtime/src/lib.rs`). If the receipt exceeds `max_receipt_size` after mutation, treat it as a `NewReceiptValidationError`.
2. **Validate `DataReceipt` total size** (not just `data` length) against `max_receipt_size` in `validate_data_receipt` or at the point where data receipts are emitted.
3. Consider pre-reserving space in a receipt for the `output_data_receivers` that will be appended, so the size check at creation time accounts for the final size.

---

### Proof of Concept

**Path 1 (`promise_return`)**:
- Deploy `rs_contract` (already in the test suite).
- Call `max_receipt_size_promise_return_method1` with `args_size = max_receipt_size - base_receipt_size`.
- The runtime produces and forwards a receipt exceeding `max_receipt_size`.

**Path 2 (`value_return`)**:
- Deploy `rs_contract`.
- Call `max_receipt_size_value_return_method` with `value_size = 4_194_304`.
- The resulting `DataReceipt` exceeds `max_receipt_size` and is forwarded without rejection.

Both are confirmed by `assert_oversized_receipt_occurred` in the test suite: [9](#0-8)

/// Validates given data receipt. Checks validity of the length of the returned data.
fn validate_data_receipt(
    limit_config: &LimitConfig,
    receipt: &DataReceipt,
) -> Result<(), ReceiptValidationError> {
    let data_len = receipt.data.as_ref().map(|data| data.len()).unwrap_or(0);
    if data_len as u64 > limit_config.max_length_returned_data {
        return Err(ReceiptValidationError::ReturnedValueLengthExceeded {
            length: data_len as u64,
            limit: limit_config.max_length_returned_data,
        });
    }
    Ok(())
}

/// Assert that there was an incoming receipt with size above max_receipt_size
fn assert_oversized_receipt_occurred(node: &TestLoopNode<'_>) {
    let client = node.client();
    let chain = &client.chain;
    let epoch_manager = &*client.epoch_manager;

    let tip = chain.head().unwrap();
    let epoch_id = epoch_manager.get_epoch_id(&tip.last_block_hash).unwrap();
    let protocol_version = epoch_manager.get_epoch_protocol_version(&epoch_id).unwrap();
    let runtime_config = client.runtime_adapter.get_runtime_config(protocol_version);
    let max_receipt_size = runtime_config.wasm_config.limit_config.max_receipt_size;

    let mut block = chain.get_block(&tip.last_block_hash).unwrap();

    // Go over all blocks down to genesis looking for a receipt above max_receipt_size.
    loop {
        if block.header().is_genesis() {
            panic!("Didn't find receipt with size above max_receipt_size!");
        }
        let prev_block = chain.get_block(block.header().prev_hash()).unwrap();

        let shard_layout = epoch_manager
            .get_shard_layout(&epoch_manager.get_epoch_id(block.hash()).unwrap())
            .unwrap();

        let oversized = if ProtocolFeature::Spice.enabled(protocol_version) {
            // With spice chunks are executed asynchronously and their produced receipts are
            // persisted as receipt proofs keyed by the block in which the chunk was applied,
            // rather than as incoming receipts on the following block.
            shard_layout.shard_ids().any(|shard_id| {
                chain
                    .chain_store()
                    .iter_receipt_proofs_for_shard(block.hash(), shard_id)
                    .iter()
                    .flat_map(|proof| proof.0.iter())
                    .any(|receipt| receipt_is_oversized(receipt, max_receipt_size))
            })
        } else {
            block.chunks().iter_new().any(|new_chunk| {
                let shard_id = new_chunk.shard_id();
                let prev_shard_index = epoch_manager
                    .get_prev_shard_id_from_prev_hash(block.header().prev_hash(), shard_id)
                    .unwrap()
                    .2;
                let prev_height_included =
                    prev_block.chunks().get(prev_shard_index).unwrap().height_included();
                let incoming_receipts_proofs = get_incoming_receipts_for_shard(
                    &chain.chain_store,
                    epoch_manager,
                    shard_id,
                    &shard_layout,
                    *block.hash(),
                    prev_height_included,
                    ReceiptFilter::TargetShard,
                )
                .unwrap();
                incoming_receipts_proofs
                    .iter()
                    .flat_map(|response| response.1.iter())
                    .flat_map(|proof| proof.0.iter())
                    .any(|receipt| receipt_is_oversized(receipt, max_receipt_size))
            })
        };

        if oversized {
            return;
        }

        block = prev_block;
    }
}
