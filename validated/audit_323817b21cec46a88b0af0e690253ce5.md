### Title
Silent `Word64` Overflow in `sumAllCoin` Enables ADA Creation via Crafted Transaction Outputs - (File: `libs/cardano-ledger-core/src/Cardano/Ledger/State/UTxO.hs`)

---

### Summary

`sumAllCoin` accumulates transaction output coin values using `Sum Word64`, which wraps silently on overflow. This function is used in the `produced` side of the value-conservation check (`consumed == produced`). An attacker can craft transaction outputs whose `Word64` sum wraps to exactly equal the `consumed` value, causing the ledger to accept a transaction that creates ADA out of thin air.

---

### Finding Description

`sumAllCoin` in `libs/cardano-ledger-core/src/Cardano/Ledger/State/UTxO.hs` accumulates coin values from `TxOut`s using `Sum Word64`:

```haskell
sumAllCoin :: (EraTxOut era, Foldable f) => f (TxOut era) -> Coin
sumAllCoin = fromCompact . CompactCoin . getSum . foldMap' getCoinWord64
  where
    getCoinWord64 txOut =
      case txOut ^. compactCoinTxOutL of
        CompactCoin w64 -> Sum w64
```

`Sum Word64` uses Haskell's modular `Word64` addition — it wraps silently at `2^64`. The function itself carries an explicit warning:

> *"Care should be taken since it is susceptible to integer overflow, therefore make sure this function is not applied to unvalidated `TxOut`s"*

`sumCoinUTxO` (which delegates to `sumAllCoin`) carries the same warning:

> *"This function cannot be applied to an untrusted `UTxO`, since it is susceptible to overflow"*

Despite these warnings, `producedTxBody` in `eras/shelley/impl/src/Cardano/Ledger/Shelley/AdaPots.hs` calls `sumCoinUTxO` directly on the unvalidated outputs of the transaction body:

```haskell
producedTxBody txBody pp dpstate =
  Produced
    { proOutputs = sumCoinUTxO (txouts txBody)
    ...
```

The existing unit test `testProducedOverMaxWord64` in `eras/shelley/test-suite/test/Test/Cardano/Ledger/Shelley/UnitTests.hs` explicitly acknowledges that `produced` can exceed `maxBound :: Word64`:

```haskell
& feeTxBodyL .~ Coin 1 -- @produced@ will return biggestCoin + 1, which is > 2^64.
```

The test only verifies that evaluation does not throw an exception (`pure $! rnf st`). It does **not** verify that the transaction is correctly rejected when the wrapped overflow value accidentally equals the `consumed` value.

The `consumed` side of the check uses `sumCoinUTxO` on inputs drawn from the existing UTxO. Because the total ADA supply is ~45 × 10¹⁵ lovelace — far below `maxBound :: Word64` (~1.8 × 10¹⁹) — the `consumed` calculation never overflows. The asymmetry between a correct `consumed` and a potentially wrapped `produced` is the root cause.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

If an attacker crafts outputs whose `Word64` sum wraps to exactly the `consumed` value, the `consumed == produced` conservation check passes. The ledger accepts the transaction and records outputs containing far more lovelace than the inputs provided. This is direct, attacker-controlled ADA creation from nothing, constituting an invalid state transition.

---

### Likelihood Explanation

**Medium.** The attack requires no privileged access, no governance majority, and no key compromise. Any unprivileged transaction sender who controls a UTxO of value `V` can construct the attack. The arithmetic to find the required output values is trivial. The only constraint is that each individual output `Coin` must fit in a `Word64` (which is satisfied since `maxBound :: Word64` ≈ 18.4 × 10¹⁸ lovelace is a valid CBOR-encoded `Word64`). The ledger imposes no per-output upper bound beyond what fits in a `Word64`.

---

### Recommendation

Replace the `Sum Word64`
