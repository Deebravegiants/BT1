### Title
Unbounded Index Into `summaries` During Weight Proof Validation Crashes Syncing Node — (File: `chia/full_node/weight_proof.py`)

### Summary
In `_validate_sub_epoch_segments`, the attacker-controlled `sub_epoch_n` field from a crafted `WeightProof` is used directly to index into the `summaries` list without a bounds check. A malicious peer can send a weight proof whose segment carries a `sub_epoch_n` value ≥ `len(summaries)`, triggering an unhandled `IndexError` that crashes the weight-proof validation coroutine and permanently blocks the node from completing sync.

### Finding Description
`_validate_sub_epoch_segments` builds two independent structures from the received `WeightProof`:

1. `summaries` — derived from `summaries_bytes` (the proof's `sub_epochs` field), length N.
2. `segments_by_sub_epoch` — derived from `sub_epoch_segments.challenge_segments` (the proof's segment field), whose keys are the `sub_epoch_n` values embedded in each segment.

The loop then directly indexes `summaries` with the attacker-supplied key:

```python
prev_ses = summaries[sub_epoch_n - 1]          # line 983
if not summaries[sub_epoch_n].reward_chain_hash == rc_sub_slot_hash:  # line 985
```

No check is performed that `sub_epoch_n < len(summaries)` before either access. A crafted proof with a segment whose `sub_epoch_n` ≥ N raises an unhandled `IndexError`.

The same pattern exists in `validate_recent_blocks` at line 1276, where `ses_idx` is incremented for every sub-epoch summary hash found in the attacker-supplied recent chain data, and then used to index `summaries[ses_idx]` without a bounds check. [1](#0-0) 

### Impact Explanation
`_validate_sub_epoch_segments` is called directly (not inside the process-pool executor) inside `validate_weight_proof_inner`: [2](#0-1) 

An `IndexError` propagates as an unhandled exception through `validate_weight_proof_inner` and up into the full node's sync loop (`sync_from_fork_point`). The node cannot complete sync against any peer that serves the crafted proof, satisfying the **High** impact criterion: *permanent or long-lived inability for honest nodes to process sync updates under normal network assumptions*.

### Likelihood Explanation
Any peer on the network can serve a `WeightProof` during the sync handshake (`respond_proof_of_weight`). No authentication or privilege is required. The attacker only needs to set `sub_epoch_n` in one `SubEpochChallengeSegment` to a value larger than the number of `SubEpochSummary` entries in the same proof. This is trivially constructable.

### Recommendation
Before the loop body accesses `summaries[sub_epoch_n]`, add an explicit bounds check:

```python
if sub_epoch_n >= len(summaries):
    log.error(f"sub_epoch_n {sub_epoch_n} out of range for summaries (len={len(summaries)})")
    return None
```

Apply the same guard in `validate_recent_blocks` before `summaries[ses_idx]` at line 1276. [3](#0-2) 

### Proof of Concept
1. Craft a `WeightProof` whose `sub_epochs` list has N entries (e.g., N = 2).
2. In `sub_epoch_segments`, include one `SubEpochChallengeSegment` with `sub_epoch_n = N` (e.g., 2).
3. Serve this proof to a syncing full node via `respond_proof_of_weight`.
4. The node calls `_validate_sub_epoch_segments`; the loop reaches `sub_epoch_n = 2`, executes `summaries[2]` on a list of length 2, and raises `IndexError`.
5. The exception propagates through `validate_weight_proof_inner`, crashing the sync coroutine. The node cannot advance its chain tip. [4](#0-3)
