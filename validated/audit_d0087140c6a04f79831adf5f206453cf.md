### Title
`TransferToGasKey`/`WithdrawFromGasKey` Exec Fee Uses Hardcoded Minimum Size, Undercharging Gas for `GasKeyFunctionCall` Keys - (File: `core/parameters/src/cost.rs`)

### Summary

`gas_key_transfer_exec_fee()` computes the execution fee for `TransferToGasKey` and `WithdrawFromGasKey` actions using `AccessKey::min_gas_key_borsh_len()` as the estimated trie value length. This is the serialized size of the smallest possible gas key variant (`GasKeyFullAccess`). When the actual key is a `GasKeyFunctionCall` — which additionally stores a `FunctionCallPermission` containing a `receiver_id` string and a `method_names` vector — the actual trie value is substantially larger, causing the exec fee to be systematically undercharged.

### Finding Description

In `core/parameters/src/cost.rs`, the exec fee for `TransferToGasKey` and `WithdrawFromGasKey` is computed as:

```rust
pub fn gas_key_transfer_exec_fee(...) -> GasKeyTransferFee {
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len(); // hardcoded minimum
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    ...
}
``` [1](#0-0) 

`AccessKey::min_gas_key_borsh_len()` is explicitly documented as the size for `GasKeyFullAccess` — the **smallest** gas key variant:

```rust
/// Minimum borsh-serialized size of an AccessKey with a gas key permission.
/// This is the size for GasKeyFullAccess (the smallest gas key variant).
pub fn min_gas_key_borsh_len() -> usize {
    borsh::object_length(&Self::gas_key_full_access(0)).unwrap()
}
``` [2](#0-1) 

However, `AccessKeyPermission::GasKeyFunctionCall` stores an additional `FunctionCallPermission` struct containing `receiver_id: String` (up to 64 bytes) and `method_names: Vec<String>` (variable length): [3](#0-2) 

The exec fee is used in `runtime/runtime/src/config.rs` for both `TransferToGasKey` and `WithdrawFromGasKey`: [4](#0-3) 

And in the VM host function `promise_batch_action_transfer_to_gas_key`: [5](#0-4) 

### Impact Explanation

The exec fee is supposed to cover the cost of reading and writing the access key trie entry. For a `GasKeyFunctionCall` key with a long `receiver_id` (e.g., 64 chars) and multiple method names, the actual borsh-serialized value stored in the trie is significantly larger than `min_gas_key_borsh_len()`. The undercharge per operation is:

```
undercharge = gas_key_byte_exec_fee × (actual_value_len − min_gas_key_borsh_len)
```

This means the network performs more trie I/O work than it is compensated for. The corrupted protocol value is the gas accounting for these actions: the `gas_burnt` and `gas_used` recorded in the receipt outcome are lower than the actual computational cost, breaking the 1ms = 1 Tgas invariant for these operations.

### Likelihood Explanation

Any unprivileged user can:
1. Create a `GasKeyFunctionCall` access key with a maximally long `receiver_id` (64 chars) and many method names via a standard `AddKey` transaction.
2. Fund it via `TransferToGasKey` or drain it via `WithdrawFromGasKey`, both of which are charged using the underestimated exec fee.

This is directly reachable through public RPC transactions with no special privileges. The `GasKeyFunctionCall` permission type is a supported, validated action type.

### Recommendation

Replace the hardcoded `AccessKey::min_gas_key_borsh_len()` with a fee model that accounts for the actual key type. Since the actual value length is not known at fee-computation time (the exec fee is computed from the action fields, not the trie state), the fee should use a conservative upper-bound estimate or be split into a base fee plus a per-byte fee that scales with the `receiver_id` length and number of method names provided in the action, similar to how `add_function_call_key_byte` already scales with method name bytes.

### Proof of Concept

1. Create a `GasKeyFunctionCall` key with `receiver_id = "a" * 64` and 16 method names of maximum length.
2. Submit a `TransferToGasKey` action targeting this key.
3. Observe that `gas_burnt` in the receipt outcome equals the fee computed with `min_gas_key_borsh_len()`, not the actual larger value size.
4. Compare against a `GasKeyFullAccess` key: the gas charged is identical despite the `GasKeyFunctionCall` trie value being substantially larger, confirming the undercharge. [6](#0-5) [7](#0-6)

/// Exec fee for TransferToGasKey / WithdrawFromGasKey actions.
/// Based on the access key trie key length + estimated value length (what the
/// receiver needs to read/write in the trie).
pub fn gas_key_transfer_exec_fee(
    cfg: &RuntimeFeesConfig,
    account_id_len: usize,
    public_key_len: usize,
) -> GasKeyTransferFee {
    let base = cfg.fee(ActionCosts::gas_key_transfer_base).exec_fee();
    let trie_key_len = access_key_key_len(account_id_len, public_key_len);
    let estimated_value_len = AccessKey::min_gas_key_borsh_len();
    let per_byte = cfg
        .fee(ActionCosts::gas_key_byte)
        .exec_fee()
        .checked_mul((trie_key_len + estimated_value_len) as u64)
        .unwrap();
    GasKeyTransferFee { base, per_byte }
}
```
