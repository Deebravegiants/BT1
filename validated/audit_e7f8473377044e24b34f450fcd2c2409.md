### Title
`L1EventsScraperConfig` Default `finality: 0` Admits L1 Handler Transactions From Unconfirmed Blocks, Enabling Reorg-Corrupted L2 State - (File: `crates/apollo_l1_events_config/src/config.rs`)

---

### Summary

`L1EventsScraperConfig::default()` sets `finality: 0`, meaning the L1 events scraper reads `LogMessageToL2` events from the very latest Ethereum block with zero confirmation buffer. An L1 reorg that reorganizes the block containing a `LogMessageToL2` event after the scraper has already forwarded it to the L1 provider — but before the next scraping cycle detects the reorg — allows a phantom L1 handler transaction to be proposed, validated by all honest validators, and committed into a finalized L2 block. The resulting L2 state contains an L1 message that never existed on the canonical Ethereum chain.

---

### Finding Description

`L1EventsScraperConfig` is defined in `crates/apollo_l1_events_config/src/config.rs`. Its `Default` implementation sets `finality: 0`:

```rust

    fn default() -> Self {
        Self {
            ...
            finality: 0,
            ...
        }
    }
}
``` [1](#0-0) 

This default is also the authoritative value in the node's config schema:

```json

"l1_events_scraper_config.finality": {
  "description": "Number of blocks to wait for finality",

``` [2](#0-1) 

The `fetch_events()` function in `crates/apollo_l1_events/src/l1_scraper.rs` applies this value as a ceiling subtraction:

```rust
// l1_scraper.rs, line 276-281
let latest_l1_block_number = latest_l1_block_number
    .checked_sub(self.config.finality)
    .ok_or(L1EventsScraperError::LatestBlockNumberTooLow { ... })?;
``` [3](#0-2) 

With `finality: 0`, `latest_l1_block_number - 0 = latest_l1_block_number`, so events are scraped all the way up to and including the chain tip.

The reorg guard, `assert_no_l1_reorgs()`, only checks whether the **previously scraped** block's hash still matches the current chain:

```rust
// l1_scraper.rs, line 430-468
async fn assert_no_l1_reorgs(&mut self) -> ... {
    let last_processed_l1_block_number = scrape_from_this_l1_block.number;
    let last_processed_l1_block_hash = scrape_from_this_l1_block.hash;
    let last_block_processed_fresh = self.base_layer.l1_block_at(last_processed_l1_block_number)...;
    if last_block_processed_fresh.hash != last_processed_l1_block_hash {
        return Err(L1EventsScraperError::L1ReorgDetected { ... });
    }
    Ok(())
}
``` [4](#0-3) 

The execution order in `send_events_to_l1_events_provider` is:

1. `assert_no_l1_reorgs()` — checks the **old** cursor block
2. `fetch_events()` — fetches events up to the **new** latest block (tip, with `finality: 0`)
3. `add_events(events)` — forwards events to the L1 provider [5](#0-4) 

The reorg check in step 1 cannot protect against a reorg of the **newly fetched** blocks in step 2. Events from the tip block are forwarded to the provider in step 3, and `scrape_from_this_l1_block` is updated to that tip. Only on the **next** polling cycle (30 seconds later by default) will `assert_no_l1_reorgs()` detect that the tip block was reorged.

During that 30-second window, the L1 provider holds the phantom L1 handler transaction in its pending pool. The batcher's `ProposeTransactionProvider` calls `get_txs()` on the provider and includes it in a block proposal. All validators call `validate()` on the provider — which returns `ValidationStatus::Validated` because the transaction is present in the provider's records — and the block is committed to L2 state. [6](#0-5) 

When the reorg is finally detected, the scraper terminates with `L1EventsScraperError::L1ReorgDetected` and the error message instructs operators to "Restart both the L1 provider and the scraper." However, the L2 block containing the phantom L1 handler transaction has already been committed. [7](#0-6) 

A deployment config overlay at `crates/apollo_deployments/resources/app_configs/l1_events_scraper_config.json` sets `finality: 10` for the reference deployment, but this is a separate file that must be explicitly loaded. Any deployment that relies on the schema default — including test networks, new chain deployments, or misconfigured nodes — inherits `finality: 0`. [8](#0-7) 

---

### Impact Explanation

An L1 handler transaction derived from a `LogMessageToL2` event in a reorged Ethereum block is admitted to the L1 provider, proposed by the sequencer, validated by all honest validators (because the provider returns `Validated`), and committed into a finalized L2 block. The L2 state then contains a wrong L1 message — an L1→L2 message that was never emitted on the canonical Ethereum chain. This corrupts the L2 state diff, the receipt commitment, and the event commitment for that block. The Starknet OS proof for that block will fail because the L1 handler transaction cannot be verified against the Starknet core contract on L1, leaving the L2 chain in an unprovable state.

This matches: **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

---

### Likelihood Explanation

Ethereum proof-of-stake reorgs are rare but not impossible; single-block reorgs occur several times per month on mainnet. With `finality: 0` and a 30-second polling interval, every L1 handler transaction scraped from the tip block is at risk during the entire polling window. The `l1_handler_proposal_cooldown_seconds: 70` default adds a 70-second delay before a scraped transaction can be proposed, which provides approximately 5–6 Ethereum blocks of additional buffer, but this is not a formal finality guarantee and does not prevent the race condition. Any deployment that does not explicitly load the deployment config overlay (e.g., test networks, new chain deployments, or operator misconfiguration) is exposed with `finality: 0`.

---

### Recommendation

1. **Change the `Default` implementation** of `L1EventsScraperConfig` to set `finality` to a safe value (e.g., `10` or higher, matching the deployment config overlay) rather than `0`. Zero confirmations provides no protection against reorgs.

2. **Add a validator** (using the existing `#[validate]` attribute infrastructure) that rejects `finality: 0` at startup with a clear error message, preventing silent misconfiguration.

3. **Update the config schema** default from `0` to `10` so that the schema itself is safe out of the box.

```rust
// Recommended fix in crates/apollo_l1_events_config/src/config.rs
impl Default for L1EventsScraperConfig {
    fn default() -> Self {
        Self {
            finality: 10,  // was 0; 10 Ethereum blocks (~2 min) is safe against reorgs
            ...
        }
    }
}
``` [1](#0-0)

### Proof of Concept

**Setup:** Deploy the sequencer using only the schema defaults (no deployment config overlay), so `l1_events_scraper_config.finality = 0`.

**Steps:**

1. Submit a `sendMessageToL2` transaction to the Starknet core contract on an Ethereum testnet. It is included in block N (the current tip).

2. The L1 events scraper polls (every 30 seconds). `fetch_events()` computes `latest = N - 0 = N`, fetches the `LogMessageToL2` event from block N, and calls `add_events([L1HandlerTransaction])` on the L1 provider. The transaction enters the provider's pending pool.

3. The batcher calls `start_block(Propose, height)` and `get_txs(n, height)`. The provider returns the L1 handler transaction (cooldown of 70 seconds may delay this, but does not prevent it). The batcher includes it in a block proposal.

4. All validators call `validate(tx_hash, height)` on their local L1 providers. Because the transaction was scraped from the same L1 tip, all providers return `ValidationStatus::Validated`. Consensus is reached and the block is committed to L2 state.

5. Ethereum block N is reorged (e.g., a competing block at the same height wins). The `LogMessageToL2` event no longer exists on the canonical chain.

6. On the next scraping cycle, `assert_no_l1_reorgs()` detects that block N's hash has changed and returns `L1EventsScraperError::L1ReorgDetected`. The scraper crashes.

7. **Result:** The committed L2 block contains an L1 handler transaction for a message that never existed on the canonical Ethereum chain. The L2 state is corrupted and the block cannot be proven.
