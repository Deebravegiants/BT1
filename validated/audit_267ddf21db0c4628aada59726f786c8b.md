### Title
Pool Can Permanently Prevent User Exit by Resetting `ASSERT_HEIGHT_RELATIVE` via Absorb Spends on `LEAVING_POOL` Singleton — (`chia/pools/pool_puzzles.py`, `chia/pools/pool_wallet.py`)

---

### Summary

`create_absorb_spend` in `pool_puzzles.py` explicitly supports absorb spends on singletons in the `LEAVING_POOL` (waiting room) state. Each absorb spend destroys the current singleton coin and recreates it at the current block height, resetting the `ASSERT_HEIGHT_RELATIVE` countdown that enforces the `relative_lock_height` exit delay. A malicious pool can continuously perform absorb spends to keep resetting this countdown, permanently preventing the user from exiting the pool and indefinitely redirecting farming rewards to the pool.

---

### Finding Description

When a user initiates pool exit, the singleton transitions to `LEAVING_POOL` state, which uses the `pool_waitingroom_inner_puzzle`. The user must wait `relative_lock_height` blocks (measured from the singleton coin's `confirmed_block_index`) before the exit spend can be confirmed.

`create_absorb_spend` in `pool_puzzles.py` explicitly handles the waiting room case:

```python
elif is_pool_waitingroom_inner_puzzle(inner_puzzle):
    # inner sol is (spend_type, destination_puzhash, pool_reward_amount, pool_reward_height, extra_data)
    inner_sol = Program.to([0, reward_amount, height])
``` [1](#0-0) 

This means absorb spends are valid on `LEAVING_POOL` singletons. Each absorb spend spends the current singleton coin and creates a new one at the current block height. Because `ASSERT_HEIGHT_RELATIVE` is evaluated relative to the `confirmed_block_index` of the coin being spent (as computed in `compute_assert_height`), the countdown resets to zero with every absorb. [2](#0-1) 

The wallet-level exit guard in `self_pool` uses `history[-1][0]` — the height of the last singleton spend — which is also updated by every absorb:

```python
history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
last_height: uint32 = history[-1][0]
if (
    await self.wallet_state_manager.blockchain.get_finished_sync_up_to()
    <= last_height + current_state.current.relative_lock_height
):
    raise ValueError(...)
``` [3](#0-2) 

Similarly, `new_peak` uses `tip_height` from `get_tip()` to compute `leave_height`, and `get_tip()` returns the last spend in the pool store — updated by every absorb:

```python
tip_height, tip_spend = await self.get_tip()
...
leave_height = tip_height + pool_wallet_info.current.relative_lock_height
``` [4](#0-3) 

The absorb spend itself requires **no signature from the user**:

```python
# If fee is 0, no signatures are required to absorb
if fee > 0:
    await self.generate_fee_transaction(...)
``` [5](#0-4) 

The pool controls the farming plots and therefore controls when `p2_singleton` reward coins are created. It can use each new farming reward to trigger an absorb spend on the `LEAVING_POOL` singleton, resetting the lock height each time.

---

### Impact Explanation

The `relative_lock_height` mechanism is the protocol's sole guarantee that a user can exit a pool within a bounded number of blocks. By continuously performing absorb spends, the pool resets the `ASSERT_HEIGHT_RELATIVE` countdown at both the CLVM level (the new singleton coin's `confirmed_block_index` advances) and the wallet level (`tip_height` / `last_height` advance). The user is permanently trapped in `LEAVING_POOL` state, with farming rewards continuing to flow to the pool's `target_puzzle_hash` indefinitely. This constitutes unauthorized payout redirection and a permanent inability to perform a protected pool state transition. [6](#0-5) 

---

### Likelihood Explanation

The pool is a semi-trusted entity that already controls the farming plots. It has direct economic incentive to prevent high-value farmers from leaving. The attack requires only that the pool farm blocks (its normal operation) and submit absorb spends on the `LEAVING_POOL` singleton — no special access, no user signature, and no cryptographic capability beyond what the pool already possesses. The `MAXIMUM_RELATIVE_LOCK_HEIGHT` of 1000 blocks means the pool only needs to perform one absorb spend every ~1000 blocks to maintain the lock indefinitely. [7](#0-6) 

---

### Recommendation

The waiting room inner puzzle (`pool_waitingroom_inner_puzzle`) should not allow absorb spends that recreate the singleton at a new coin height while the singleton is in `LEAVING_POOL` state. Options:

1. **Disallow absorb spends in the waiting room puzzle entirely.** Rewards that arrive during the exit window can be claimed after the exit completes.
2. **Preserve the original exit-initiation height.** Record the block height at which the singleton first entered `LEAVING_POOL` state (e.g., in the puzzle or solution) and assert `ASSERT_HEIGHT_ABSOLUTE` against that fixed height rather than using `ASSERT_HEIGHT_RELATIVE` on the current coin.
3. **Wallet-level guard.** In `create_absorb_spend`, reject absorb spends when `current_state.state == LEAVING_POOL`, so the wallet never constructs such a spend bundle. [8](#0-7) 

---

### Proof of Concept

```
Block N:   User calls self_pool() → singleton transitions to LEAVING_POOL
           Singleton coin S₀ created at block N
           Exit requires: block N + relative_lock_height

Block N+1: Pool farms a block → p2_singleton reward coin R₁ created
           Pool submits absorb spend: spends S₀ + R₁ → creates S₁ at block N+1
           apply_state_transition() records new tip at height N+1
           Exit now requires: block N+1 + relative_lock_height

Block N+2: Pool farms another block → R₂ created
           Pool submits absorb spend: spends S₁ + R₂ → creates S₂ at block N+2
           Exit now requires: block N+2 + relative_lock_height

... repeat every block ...

User can never satisfy ASSERT_HEIGHT_RELATIVE.
self_pool() always raises: "Cannot self pool until height <current+relative_lock_height>"
new_peak() never triggers the second travel transaction.
``` [9](#0-8) [10](#0-9)

    async def apply_state_transition(
        self, new_state: CoinSpend, block_height: uint32, action_scope: WalletActionScope
    ) -> bool:
        """
        Updates the Pool state (including DB) with new singleton spends.
        The DB must be committed after calling this method. All validation should be done here. Returns True iff
        the spend is a valid transition spend for the singleton, False otherwise.
        """
        tip: tuple[uint32, CoinSpend] = await self.get_tip()
        tip_spend = tip[1]

        tip_coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(tip_spend)
        assert tip_coin is not None
        spent_coin_name: bytes32 = tip_coin.name()

        if spent_coin_name != new_state.coin.name():
            history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
            if new_state.coin.name() in [sp.coin.name() for _, sp in history]:
                self.log.info(f"Already have state transition: {new_state.coin.name().hex()}")
            else:
                self.log.warning(
                    f"Failed to apply state transition. tip: {tip_coin} new_state: {new_state} height {block_height}"
                )
            return False

        await self.wallet_state_manager.pool_store.add_spend(self.wallet_id, new_state, block_height)
        tip_spend = (await self.get_tip())[1]
        self.log.info(f"New PoolWallet singleton tip_coin: {tip_spend} farmed at height {block_height}")

        # If we have reached the target state, resets it to None. Loops back to get current state
        for _, added_spend in reversed(
            await self.wallet_state_manager.pool_store.get_spends_for_wallet(self.wallet_id)
        ):
            latest_state: PoolState | None = solution_to_pool_state(added_spend)
            if latest_state is not None:
                if self.target_state == latest_state:
                    self.target_state = None
                    self.next_transaction_fee = uint64(0)
                    self.next_tx_config = DEFAULT_TX_CONFIG
                break

        await self.update_pool_config(action_scope)
        return True

class PoolSingletonState(IntEnum):
    """
    From the user's point of view, a pool group can be in these states:
    `SELF_POOLING`: The singleton exists on the blockchain, and we are farming
        block rewards to a wallet address controlled by the user

    `LEAVING_POOL`: The singleton exists, and we have entered the "escaping" state, which
        means we are waiting for a number of blocks = `relative_lock_height` to pass, so we can leave.

    `FARMING_TO_POOL`: The singleton exists, and it is assigned to a pool.

    `CLAIMING_SELF_POOLED_REWARDS`: We have submitted a transaction to sweep our
        self-pooled funds.
    """

    SELF_POOLING = 1
    LEAVING_POOL = 2
    FARMING_TO_POOL = 3
