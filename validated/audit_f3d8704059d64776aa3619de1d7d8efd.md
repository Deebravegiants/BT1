### Title
Missing `recompute_available_foreign_chains` Invocation After Resharing Cancellation Leaves Available-Chain Cache Stale - (File: `crates/contract/src/lib.rs`)

### Summary

`vote_cancel_resharing` transitions the contract back to the previous running state but omits the `recompute_available_foreign_chains` call and the `CLEAN_FOREIGN_CHAIN_DATA` cleanup promise that `vote_reshared` always performs. This leaves the `available_foreign_chains` cache reflecting availability computed during the resharing period — including coverage registrations from prospective participants who are no longer participants after cancellation — causing `verify_foreign_transaction` to accept requests for chains that are no longer actually serviceable.

### Finding Description

`vote_reshared` (the successful-resharing path) performs two critical post-transition operations:

1. Calls `self.recompute_available_foreign_chains()` synchronously to rebuild the availability cache from the new participant set.
2. Spawns a detached `CLEAN_FOREIGN_CHAIN_DATA` promise to remove stale per-node foreign-chain registrations from `node_foreign_chain_support` for non-participants. [1](#0-0) 

`vote_cancel_resharing`, which also transitions the protocol state (back to the previous running state), performs neither: [2](#0-1) 

During a resharing, prospective new participants can call `register_available_foreign_chain_config` (or its equivalent), inserting entries into `node_foreign_chain_support`. Each such registration triggers `recompute_available_foreign_chains`, potentially marking chains as "available" that were not available before resharing began. When resharing is subsequently cancelled:

- The prospective participants' entries remain in `node_foreign_chain_support` (no `CLEAN_FOREIGN_CHAIN_DATA` call).
- The `available_foreign_chains` cache is not rebuilt (no `recompute_available_foreign_chains` call).
- The contract reverts to the old participant set, but the availability cache still reflects the resharing-era participant coverage.

The `verify_foreign_transaction` entry point gates on this cached set: [3](#0-2) 

The design document confirms the invariant: a chain is "available" iff ≥ signing-threshold active participants currently cover it, and `verify_foreign_transaction` must reject unavailable chains.



### Impact Explanation

After resharing cancellation, the `available_foreign_chains` cache can show chains as available that are no longer covered by a signing threshold of current participants. Any unprivileged user calling `verify_foreign_transaction` for such a chain will have their request accepted by the contract (deposit taken, yield slot created), but no signing threshold of nodes can fulfill it — the request times out. This breaks the production safety invariant that `verify_foreign_transaction` must only accept requests for chains that the current participant set can actually service, corrupting the request lifecycle and potentially causing user funds to be locked until the yield-resume timeout fires.

### Likelihood Explanation

Resharing cancellation is a normal protocol operation triggered whenever a resharing stalls (e.g., a prospective participant goes offline). Any participant can initiate resharing and any threshold of old participants can cancel it. The stale cache persists until the next `recompute_available_foreign_chains` call (which only happens on `vote_reshared` or `vote_update_foreign_chain_providers`), meaning the window can be arbitrarily long. No privileged access is required to exploit the stale cache — any user can call `verify_foreign_transaction`.

### Recommendation

Add the missing operations to `vote_cancel_resharing` to mirror what `vote_reshared` does on state transition:

```rust
pub fn vote_cancel_resharing(&mut self) -> Result<(), Error> {
    Self::assert_caller_is_signer();
    log!("vote_cancel_resharing: signer={}", env::signer_account_id());

    if let Some(new_state) = self.protocol_state.vote_cancel_resharing()? {
        self.protocol_state = new_state;
        // Rebuild the available-chain cache for the reverted participant set.
        self.recompute_available_foreign_chains();
        // Clean up stale foreign-chain registrations from prospective participants.
        Promise::new(env::current_account_id())
            .function_call(
                method_names::CLEAN_FOREIGN_CHAIN_DATA.to_string(),
                vec![],
                NearToken::from_yoctonear(0),
                Gas::from_tgas(self.config.clean_foreign_chain_data_tera_gas),
            )
            .detach();
    }
    Ok(())
}
```

### Proof of Concept

1. Contract is in Running state; participants P1, P2, P3 cover chains A and B. Chain C is not covered (not available).
2. Resharing begins to add prospective participant P4.
3. P4 calls `register_available_foreign_chain_config` covering chain C. `recompute_available_foreign_chains` is called internally; chain C now appears available in the cache (P4 + enough others cover it).
4. P4 goes offline; resharing stalls. P1 and P2 call `vote_cancel_resharing`; threshold is reached.
5. Contract reverts to Running with {P1, P2, P3}. `recompute_available_foreign_chains` is **not** called. P4's entry remains in `node_foreign_chain_support`. Chain C still appears available in the cache.
6. An unprivileged user calls `verify_foreign_transaction` targeting chain C. The contract checks `available_foreign_chains`, sees C as available, accepts the request, and creates a yield slot.
7. No current participant covers chain C; no signing threshold can be reached. The request times out after ~200 blocks, breaking the request-lifecycle safety invariant. [4](#0-3) [5](#0-4)

    /// Private endpoint to clean up foreign chain policy votes and node configurations
    /// for non-participants after resharing.
    #[private]
    #[handle_result]
    pub fn clean_foreign_chain_data(&mut self) -> Result<(), Error> {
        log!(
            "clean_foreign_chain_data: signer={}",
            env::signer_account_id()
        );

        let participants = match &self.protocol_state {
            ProtocolContractState::Running(state) => state.parameters.participants(),
            _ => {
                return Err(InvalidState::ProtocolStateNotRunning.into());
            }
        };

        let participant_accounts: std::collections::HashSet<dtos::AccountId> = participants
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect();

        let active_tls_keys: std::collections::BTreeSet<dtos::Ed25519PublicKey> = participants
            .participants()
            .iter()
            .map(|(_, _, info)| info.tls_public_key.clone())
            .collect();

        let non_participant_configs: Vec<dtos::AccountId> = self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .keys()
            .filter(|account| !participant_accounts.contains(*account))
            .cloned()
            .collect();
        for account in &non_participant_configs {
            self.node_foreign_chain_support
                .foreign_chain_support_by_node
                .remove(account);
        }

        self.foreign_chains
            .get_mut()
            .remove_stale_configs(&active_tls_keys);

        self.foreign_chains
            .get_mut()
            .rpc_whitelist
            .votes
            .retain(participants);

        Ok(())
