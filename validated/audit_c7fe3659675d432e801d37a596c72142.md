### Title
Double Deposit Refund via Sub-Transaction Batch in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces nested (sub-)transactions. The `dijkstraUtxoTransition` function adds a batch-level check (`validateBatchWithdrawals`) to prevent double-withdrawal of reward-account balances across sub-transactions, but no equivalent batch check exists for deposit refunds from deregistration certificates. Two sub-transactions in the same top-level batch can each include `UnRegDepositTxCert` for the same staking credential, each independently passing the per-sub-transaction balance equation using the original (pre-batch) `certState`, thereby claiming the same deposit refund twice. The developers have explicitly acknowledged this gap with a disabled test marked `"TODO: predicate failure not yet implemented"`.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — double-claiming of deposit refunds.

In `dijkstraUtxoTransition`, the UTXO rule performs two relevant checks:

1. **`validateBatchWithdrawals`** (line 378): aggregates withdrawals from the top-level tx and all sub-transactions using `Map.unionsWith (<>)`, then checks the total against the **original** account balance. This correctly prevents double-withdrawal of rewards across sub-transactions.

2. **`validateValueNotConservedUTxO`** (line 381): checks the balance equation using `originalUtxo`, `certState` (original, pre-batch), and `txBody` (top-level body only). Sub-transaction bodies are **not** included. [1](#0-0) 

Each sub-transaction's balance equation is checked independently. The "consumed" side of a sub-transaction's balance equation includes `certsTotalRefundsTxBody`, which counts the deposit refund for any `UnRegDepositTxCert` in that sub-transaction's certificate list. Because the check uses the **original** `certState` (before any sub-transaction has been applied), two sub-transactions that each contain `UnRegDepositTxCert stakingCred keyDeposit` for the same credential will each independently pass the balance equation check — both include the same refund in their "consumed" side.

There is no `validateBatchRefunds` equivalent to `validateBatchWithdrawals`. The structural gap is: [2](#0-1) 

`validateBatchWithdrawals` exists and aggregates across sub-transactions; no analogous function exists for deposit refunds.

The developers have explicitly acknowledged this gap in a disabled test: [3](#0-2) 

The test is marked `xit` (pending/disabled) with the comment `error "TODO: predicate failure not yet implemented"`, confirming that the specific predicate failure to block this scenario has not yet been implemented.

The `applyWithdrawals` call in `dijkstraEntitiesTransition` only applies the **top-level** transaction's withdrawals to the accounts before processing certificates: [4](#0-3) 

Sub-transaction withdrawals and certificate effects are processed separately, and the original `accounts` snapshot used for batch validation does not reflect intermediate state changes from earlier sub-transactions.

---

### Impact Explanation

An attacker who controls a staking credential can register it, then submit a Dijkstra-era top-level transaction containing two sub-transactions, each with `UnRegDepositTxCert stakingCred keyDeposit`. Both sub-transactions pass the per-sub-transaction balance equation check (each independently includes the refund in "consumed" using the original `certState`). The deposit pot is debited twice for a single credential's deposit, creating ADA out of thin air in the attacker's outputs. This is a **direct loss/creation of ADA through an invalid ledger state transition** — matching the Critical impact tier.

---

### Likelihood Explanation

The entry path requires only an unprivileged transaction sender operating in the Dijkstra era. No privileged role, governance majority, or key compromise is needed. The attacker needs only to:
1. Register a staking credential (paying one deposit).
2. Construct a top-level Dijkstra transaction with two sub-transactions, each containing `UnRegDepositTxCert` for the same credential.
3. Balance each sub-transaction's outputs to absorb the claimed refund.

The Dijkstra era is present in the production codebase. The gap is structural and the developers have already identified it as a known TODO.

---

### Recommendation

Add a `validateBatchRefunds` check in `dijkstraUtxoTransition`, analogous to `validateBatchWithdrawals`, that:
- Aggregates all deposit refunds claimed across the top-level transaction and all sub-transactions (by iterating over `UnRegDepositTxCert` / `UnRegTxCert` certificates in each sub-transaction body).
- Verifies that no staking credential appears in more than one deregistration certificate across the entire batch.
- Alternatively, verifies that the total refunds claimed for each credential do not exceed the deposit recorded for that credential in the original `certState`.

This mirrors the pattern already established by `validateBatchWithdrawals`: [5](#0-4) 

The disabled test at `Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53–75` should be re-enabled once the predicate failure is implemented.

---

### Proof of Concept

```
1. Register staking credential C with deposit D (keyDeposit).

2. Construct:
   subTx1:
     inputs:  { utxoInput1 }
     certs:   [ UnRegDepositTxCert C D ]
     outputs: [ addr1 ← (value1 + D) ]   -- absorbs refund

   subTx2:
     inputs:  { utxoInput2 }
     certs:   [ UnRegDepositTxCert C D ]
     outputs: [ addr2 ← (value2 + D) ]   -- absorbs refund

   topTx:
     inputs:  {}
     subTransactions: { subTx1, subTx2 }

3. Submit topTx.

Expected (correct) behavior: transaction rejected with a "double refund" predicate failure.
Actual (buggy) behavior:     both sub-transactions pass the balance equation check
                              (each independently includes D in "consumed" using original certState);
                              no batch-level refund check exists to catch the duplication;
                              attacker receives 2×D in outputs while only one deposit D was paid.
```

The structural evidence is the absence of any `validateBatchRefunds` call in `dijkstraUtxoTransition` alongside the existing `validateBatchWithdrawals` call, and the explicitly disabled test `xit "Multiple subtransactions cannot get the same refund"` with `error "TODO: predicate failure not yet implemented"`. -- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL

dijkstraUtxoTransition = do
  TRC (DijkstraUtxoEnv slot pp certState originalUtxo, utxos, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG
  -- this is the original Accounts, before any transactions were applied
  let accounts = certState ^. certDStateL . accountsL

  let txBody = tx ^. bodyTxL

  {- inInterval (SlotOf Γ) (ValidIntervalOf txTop) -}
  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo

  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  {- SpendInputs ≠ ∅ -}
  runTestOnSignal $ Shelley.validateInputSetEmptyUTxO txBody

  let allInputs = txBody ^. allInputsTxBodyF
      inputs = txBody ^. inputsTxBodyL

  {- SpendInputsOf txTop ∪ RefInputsOf txTop ∪ CollInputsOf txTop ⊆ dom(utxo₀) -}
  runTest $ Shelley.validateBadInputsUTxO originalUtxo allInputs

  {- SpendInputsOf txTop ⊆ dom(utxo_s) — prevents double-spend with subtxs -}
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxos) inputs

  {- minfee pp txTop utxo₀ ≤ txfee txb -}
  runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo

  {- (RedeemersOf txTop ≠ ∅ ⊎ Any (λ txSub → RedeemersOf txSub ≠ ∅) subtxs) → collateralCheck -}
  validate $ validateBatchCollateral pp tx originalUtxo

  runTest $ validateBatchWithdrawals accounts tx

  {- consumed pp utxo₀ txb = produced pp certState txb -}
  runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
