### Title
Missing shard_id binding in `set_state_header` allows cross-shard state substitution during state sync — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a requested `shard_id` without ever verifying that the embedded chunk's `shard_id` matches the requested one. Because `verify_path` only proves membership in the block's chunk-headers merkle tree (not position), a malicious peer can substitute a chunk from shard_id=1 into a response for shard_id=0, pass all validation, and cause the syncing node to apply shard_id=1's state to shard_id=0's trie.

### Finding Description

`set_state_header` performs five validation steps before persisting the header:

1. **`validate_chunk_proofs`** — checks internal chunk consistency (hash, tx root, receipts root). No `shard_id` check. [1](#0-0) 

2. **`verify_path` (step 3a)** — proves the chunk hash is reachable from `chunk_headers_root` via the supplied path. Uses plain `verify_path`, not `verify_path_with_index`, so it does **not** check the leaf position (i.e., the shard index). [2](#0-1) [3](#0-2) 

3. **Receipt proof validation (step 4)** — hashes receipts with the *requested* `shard_id`, not the chunk's `shard_id`, so valid receipt proofs for shard_id=0 can coexist with a chunk for shard_id=1. [4](#0-3) 

4. **State root node validation (step 5)** — validates the `state_root_node` against `chunk_inner.prev_state_root()`, which is shard_id=1's state root. A valid node for shard_id=1 passes. [5](#0-4) 

5. **Persistence** — the header is stored under `StateHeaderKey(shard_id=0, sync_hash)` with no check that `chunk.shard_id() == shard_id`. [6](#0-5) 

There is no line anywhere in `set_state_header` that asserts `chunk.shard_id() == shard_id`.

### Impact Explanation

After the poisoned header is stored, `set_state_finalize` derives `shard_uid` from the *requested* shard_id=0 but reads `prev_state_root` from the stored chunk (shard_id=1's root): [7](#0-6) 

The runtime then applies shard_id=1's state parts to shard_id=0's trie. The syncing node ends up with a completely wrong state for shard_id=0: account balances, nonces, access keys, and contract storage for all accounts in shard_id=0 are replaced with shard_id=1's data. Every subsequent transaction validation, RPC query, and receipt execution for shard_id=0 accounts operates on corrupted state.

### Likelihood Explanation

Any node can join the NEAR peer network and serve state sync responses. State sync explicitly downloads headers from peers via `StateSyncDownloadSource`. The `ensure_shard_header` loop in `downloader.rs` retries until a header passes `set_state_header` validation — a malicious peer that is the only available source will have its crafted header accepted on the first attempt. [8](#0-7) 

### Recommendation

Add an explicit shard_id binding check at the top of `set_state_header`, immediately after extracting the chunk:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, the `verify_path` call at step 3a should be replaced with `verify_path_with_index` using the shard index derived from `shard_id` and the epoch's shard layout, so that the merkle proof is bound to the correct leaf position.

### Proof of Concept

1. Run a malicious peer that intercepts `StateRequestHeader { shard_id: 0, sync_hash }`.
2. Construct a `ShardStateSyncResponseHeader::V2` where:
   - `chunk` = a valid `ShardChunk` for shard_id=1 (obtained from the canonical chain).
   - `chunk_proof` = the valid merkle path for shard_id=1's chunk in `sync_prev_block.chunk_headers_root` (position 1, not 0).
   - `incoming_receipts_proofs` / `root_proofs` = valid proofs for shard_id=0's actual incoming receipts.
   - `state_root_node` = valid node for shard_id=1's `prev_state_root`.
3. Serve this response to a syncing node requesting shard_id=0.
4. Observe that `set_state_header` returns `Ok(())` and stores the header under `StateHeaderKey(0, sync_hash)`.
5. Observe that `set_state_finalize` applies shard_id=1's state to shard_id=0's trie, producing a wrong state root for shard_id=0.
6. Assert that any account balance or nonce query for a shard_id=0 account now returns shard_id=1's data.

    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash

    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {

            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };

        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
