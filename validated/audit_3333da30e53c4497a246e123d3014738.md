### Title
Hardcoded `Fee(1)` in Consensus-Path L1 Handler Conversion Bypasses Fee Adequacy Check and Commits Wrong `paid_fee_on_l1` to Block — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

`convert_consensus_l1_handler_to_internal_l1_handler` unconditionally injects `paid_fee_on_l1 = Fee(1)` for every L1 handler transaction that arrives through the consensus path (validator receiving a proposal). The blockifier's only fee guard for L1 handlers checks `paid_fee != Fee(0)`, so `Fee(1)` trivially satisfies it regardless of the actual fee paid on L1. Additionally, the `CentralL1HandlerTransaction` block-commitment record serialises `paid_fee_on_l1` directly from the executable transaction, so every consensus-path L1 handler is committed to the block with the wrong fee value.

---

### Finding Description

**Root cause — `transaction_converter.rs`:**

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

This function is called from `convert_consensus_tx_to_internal_consensus_tx` for every `ConsensusTransaction::L1Handler` variant:

```rust
ConsensusTransaction::L1Handler(tx) => {
    let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
    Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))
}
``` [2](#0-1) 

**Fee check in the blockifier — `l1_handler_transaction.rs`:**

The only fee guard for L1 handlers is:

```rust
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
``` [3](#0-2) 

Because the converter always supplies `Fee(1)`, the condition `paid_fee == Fee(0)` is never true, so the check is trivially bypassed for every consensus-path L1 handler, regardless of what was actually paid on L1.

**Wrong value committed to the block — `central_objects.rs`:**

`CentralL1HandlerTransaction` serialises `paid_fee_on_l1` directly from the executable transaction:

```rust
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,
            ...
        }
    }
}
``` [4](#0-3) 

Because `paid_fee_on_l1` is always `Fee(1)` on the validator side, every L1 handler transaction in a consensus-proposed block is committed to the cende with `paid_fee_on_l1 = 1` instead of the actual fee paid on L1.

**Contrast with the non-consensus (native_blockifier / Python) path**, which reads the real value from the L1 event:

```rust
let paid_fee_on_l1 = Fee(py_attr::<u128>(py_tx, "paid_fee_on_l1")?);
Ok(L1HandlerTransaction { tx, tx_hash, paid_fee_on_l1 })
``` [5](#0-4) 

The two paths produce different `paid_fee_on_l1` values for the same transaction, breaking the invariant that all nodes agree on the same execution result.

---

### Impact Explanation

Two concrete impacts:

1. **Wrong receipt/block-commitment value (Critical).** `paid_fee_on_l1` is part of the `CentralL1HandlerTransaction` record written to the block. Every validator commits `Fee(1)` instead of the real L1 fee, producing a wrong authoritative value in the block data. Any downstream consumer (prover, explorer, bridge verifier) that reads `paid_fee_on_l1` from the committed block will see an incorrect value.

2. **Fee adequacy check trivially bypassed (High).** The blockifier's guard `if paid_fee == Fee(0)` is the only in-sequencer check that an L1 handler paid something on L1. With `Fee(1)` hardcoded, this guard is permanently satisfied for all consensus-path L1 handlers, meaning the sequencer cannot distinguish a legitimately-paid handler from one that somehow arrived with zero actual payment.

---

### Likelihood Explanation

Every L1 handler transaction processed by a validator (non-proposer) node goes through `convert_consensus_tx_to_internal_consensus_tx`, which unconditionally calls `convert_consensus_l1_handler_to_internal_l1_handler`. There is no code path that avoids the hardcoded `Fee(1)`. The trigger requires no special privilege — it fires for every L1 handler in any consensus proposal.

---

### Recommendation

1. Include the real `paid_fee_on_l1` in the consensus wire message (`ConsensusTransaction::L1Handler`) so validators receive the actual value from the proposer.
2. Until (1) is done, at minimum add an assertion or log warning when `paid_fee_on_l1 == Fee(1)` is used as a placeholder, so the wrong value is not silently committed.
3. Strengthen the blockifier check from `paid_fee != Fee(0)` to `paid_fee >= actual_fee` once the real value is available, removing the reliance on the L1 contract as the sole enforcement point.

---

### Proof of Concept

**Step 1.** Proposer builds a block containing an L1 handler transaction with `paid_fee_on_l1 = Fee(1_000_000)` (as seen in OS flow tests).

**Step 2.** Proposer streams the proposal to validators. The consensus wire type `ConsensusTransaction::L1Handler(tx)` carries only `transaction::L1HandlerTransaction`, which has no `paid_fee_on_l1` field.

**Step 3.** Each validator calls `convert_consensus_tx_to_internal_consensus_tx`:

```rust
ConsensusTransaction::L1Handler(tx) => {
    let internal_tx = self.convert_consensus_l1_handler_to_internal_l1_handler(tx)?;
    // internal_tx.paid_fee_on_l1 == Fee(1)  ← always
``` [2](#0-1) 

**Step 4.** The blockifier executes the L1 handler. The fee check `if paid_fee == Fee(0)` evaluates to `false` (because `Fee(1) != Fee(0)`), so execution succeeds. [6](#0-5) 

**Step 5.** The validator commits `CentralL1HandlerTransaction { paid_fee_on_l1: Fee(1), ... }` to the block, while the proposer committed `paid_fee_on_l1: Fee(1_000_000)`. The two nodes have agreed on the same block hash (because `paid_fee_on_l1` is not part of the transaction hash) but have committed different `paid_fee_on_l1` values to their block records — a wrong authoritative receipt value. [4](#0-3)
