Audit Report

## Title
Missing L2 Sequencer Uptime Check Allows Stale-Price Swaps After Sequencer Recovery - (File: `smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary
`PriceProviderL2` and `ProtectedPriceProviderL2` are deployed on L2 chains but contain no sequencer uptime or grace-period check. Their only freshness guard is `_isStale`, which accepts any oracle `refTime` within `MAX_TIME_DELTA` (up to 7 days). When the L2 sequencer goes down and recovers, the pre-outage oracle price passes `_isStale` and is fed directly into `MetricOmmPool.swap`, allowing a trader to execute at a price that no longer reflects market conditions and extracting value from LP reserves.

## Finding Description
Both contracts implement an identical staleness check:

```solidity
// PriceProviderL2.sol L135-150 / ProtectedPriceProviderL2.sol L138-153
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;
}
```

`FUTURE_TOLERANCE` (≤ 1 hour) addresses sequencer **clock skew** — it does not address sequencer **downtime**. When the sequencer is down, no L2 transactions execute and oracle publishers cannot push new prices. The last oracle update (with its `refTime` and price) is frozen in the oracle contract. When the sequencer recovers, `block.timestamp` resumes and `(block.timestamp - refTime)` equals the outage duration. If that duration is less than `MAX_TIME_DELTA`, `_isStale` returns `false` and `_getBidAndAskPrice()` returns the pre-outage bid/ask.

`PriceProviderFactoryL2.createPriceProvider` deploys `PriceProviderL2` with no `sequencerUptimeFeed` parameter at all — there is no path to inject a sequencer check at construction time.

The stale quote then flows unguarded through the pool:

```
MetricOmmPool.swap
  → _getBidAndAskPriceX64()
    → IPriceProvider(activePriceProvider).getBidAndAskPrice()  // returns stale bid/ask
  → SwapMath.midAndSpreadFeeX64FromBidAsk(bid, ask)
  → _executeSwap(...)                                          // settles at stale price
  → token transfer to recipient
```

The registry ships `ChainlinkVerifierL2` with `sequencerUptimeFeed` and `GRACE_PERIOD`, and a separate registry entry for a more feature-rich L2 provider also includes `sequencerUptimeFeed` — confirming the team is aware of the pattern. Neither source contract integrates it.

## Impact Explanation
A trader can call `MetricOmmPool.swap` immediately after sequencer recovery using a bid/ask that was valid before the outage but is now stale. If the real market price moved during the outage (e.g., ETH dropped 5%), the trader buys token0 at the pre-crash ask price and immediately sells at the real market price, extracting value from LP reserves. This is a direct loss of LP principal and matches the allowed impact gate: **bad-price execution — stale bid/ask quote reaches a pool swap**.

## Likelihood Explanation
L2 sequencer outages are documented historical events on Arbitrum, Optimism, and Base. No privileged access is required — any public caller can invoke `swap` the moment the sequencer recovers. The exploitation window equals the outage duration. For pools using Chainlink feeds with `MAX_TIME_DELTA` set to the feed's heartbeat (1 h or 24 h), any outage shorter than that window is exploitable. The `PriceProviderFactoryL2` is permissionless, so pools using these providers are widely deployable.

## Recommendation
Add an immutable `sequencerUptimeFeed` (nullable) and a `GRACE_PERIOD` constant to both `PriceProviderL2` and `ProtectedPriceProviderL2`. Call a `_checkSequencer()` guard at the top of `_getBidAndAskPrice()` / `_computeBidAsk()` before the oracle read:

```solidity
function _checkSequencer() internal view returns (bool blocked) {
    if (address(sequencerUptimeFeed) == address(0)) return false;
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer != 0) return true;                                    // sequencer down
    if (block.timestamp - startedAt < GRACE_PERIOD) return true;    // grace period
    return false;
}
```

If `_checkSequencer()` returns `true`, return `(0, type(uint128).max)` so `getBidAndAskPrice()` reverts with `FeedStalled`, blocking all swaps until prices are fresh. Update `PriceProviderFactoryL2.createPriceProvider` to accept and forward `_sequencerUptimeFeed`.

## Proof of Concept
```
Setup:
  - Deploy PriceProviderL2 on Arbitrum with MAX_TIME_DELTA = 3600 (1 hour)
  - Pool has ETH/USDC, oracle price = $3000, refTime = T=0

Attack:
  T=0:      Sequencer goes down. Last oracle refTime = T=0, price = $3000.
  T=0..D:   Real ETH price drops to $2800 off-chain. No L2 txs possible.
  T=D:      Sequencer recovers (D = 30 min < MAX_TIME_DELTA = 1 hour).
            block.timestamp = D, oracle refTime = 0.
            _isStale(0, D, 3600, FUTURE_TOL) → (D - 0) = 1800 ≤ 3600 → NOT stale.
            getBidAndAskPrice() returns bid/ask based on $3000.
  T=D+1:    Attacker calls pool.swap(zeroForOne=true) — buys ETH at $3000 ask.
            Real market price is $2800. Attacker sells on CEX for $2800.
            Pool LPs absorb the $200/ETH loss.

Foundry fork test: fork Arbitrum at a block just after a historical sequencer recovery,
warp block.timestamp to recovery time, set oracle refTime to pre-outage value within
MAX_TIME_DELTA, call pool.swap, assert trade executes at stale price.
```
