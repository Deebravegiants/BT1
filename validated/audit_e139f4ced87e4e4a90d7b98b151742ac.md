### Title
Validator Proposals Not Rolled Back on Failed Receipt Execution, Causing Staking Invariant Violation at Epoch Boundary - (`runtime/runtime/src/lib.rs`)

### Summary

In `runtime/runtime/src/lib.rs`, when an action receipt fails (e.g., due to `LackBalanceForState` after a `StakeAction` succeeds), the runtime correctly rolls back trie state changes but **unconditionally appends validator proposals** to the block-level `validator_proposals` vector before checking the receipt result. This mirrors the external report's pattern exactly: a multi-step operation modifies two independent state surfaces (trie state and validator proposals), but only one surface is restored on failure. The orphaned proposal propagates into `EpochInfo.stake_change`, and at the next epoch boundary `update_validator_accounts` enforces the staking invariant `account.locked() >= max_of_stakes`, which now fails because the trie state was rolled back but the proposal was not. This causes a `StorageInconsistentState` FATAL error on every node, halting the chain.

### Finding Description

**Root Cause**

In `apply_action_receipt` (`runtime/runtime/src/lib.rs`), after all actions are executed, the code path is:

1. **Lines 834–873**: Actions are executed one by one. `action_stake` (`runtime/runtime/src/actions.rs:75–79`) pushes a `ValidatorStake` into `result.validator_proposals` when the stake action itself succeeds (balance check passes, minimum stake check passes).

2. **Lines 876–898**: `check_storage_stake` is called on the in-memory modified account. If the staked amount leaves `account.amount()` below the storage staking requirement, `result.set_error(LackBalanceForState)` is called, setting `result.result = Err(...)`.

3. **Line 943**: `validator_proposals.append(&mut result.validator_proposals)` — **unconditional**, happens before the Ok/Err branch.

4. **Lines 946–955**: Only now is `result.result` checked. On `Err`, `state_update.rollback()` is called, reverting trie changes. But the validator proposal is already in the outer `validator_proposals` vector and is **not removed**.

```rust
// Moving validator proposals
validator_proposals.append(&mut result.validator_proposals);  // line 943 — unconditional

// Committing or rolling back state.
match &result.result {
    Ok(_) => { state_update.commit(...); }
    Err(_) => { state_update.rollback(); }   // trie rolled back, but proposal already appended
};
```

The same path is reachable via a multi-action receipt: `[StakeAction(X), AddKeyAction(existing_key)]` — the stake action succeeds and pushes a proposal, the second action fails, the receipt fails, trie is rolled back, but the proposal remains.

**Propagation to Epoch Boundary**

The `validator_proposals` from `ApplyResult` are stored in `BlockInfo.proposals` by the chain layer. The `EpochInfoAggregator` collects them at `epoch_info_aggregator.rs:201`:

```rust
for proposal in block_info.proposals_iter() {
    self.all_proposals.entry(proposal.account_id().clone()).or_insert(proposal);
}
```

At epoch finalization, `collect_blocks_info` uses `all_proposals` to compute `EpochInfo.stake_change`. Then `compute_stake_return_info` (`chain/epoch-manager/src/lib.rs:1360–1399`) computes `max_of_stakes` as the maximum stake across the past three epochs for each account in `stake_change`.

At the first block of the new epoch, `update_validator_accounts` (`runtime/runtime/src/lib.rs:1584–1680`) enforces:

```rust
if account.locked() < *max_of_stakes {
    return Err(StorageError::StorageInconsistentState(format!(
        "FATAL: staking invariant does not hold. \
         Account stake {} is less than maximum of stakes {} in the past three epochs",
        account.locked(), max_of_stakes)).into());
}
```

Because the trie was rolled back, `account.locked()` is the pre-stake value (e.g., 0), while `max_of_stakes` reflects the fake proposal (e.g., 9.5 NEAR). Every node applying this epoch-boundary block hits the same FATAL error.

### Impact Explanation

The corrupted protocol value is `validator_proposals` in `ApplyResult`, which propagates into `EpochInfo.stake_change` and ultimately into `ValidatorAccountsUpdate.stake_info`. At the epoch boundary, `update_validator_accounts` returns `RuntimeError::StorageError(StorageInconsistentState(...))`. This causes chunk application to fail on every node processing the epoch-boundary block, halting the chain. The invariant broken is: *every account in `stake_info` must have `locked >= max_of_stakes`*, which is violated because the trie rollback restored `locked` but the proposal was not removed.

### Likelihood Explanation

An unprivileged user can trigger this with a single signed transaction. The `check_storage_stake` path requires engineering an account where staking almost all available balance leaves `amount` below the storage staking threshold (e.g., account with 10 NEAR, stake 9.5 NEAR, storage requirement ~1 NEAR). The multi-action path requires only a `StakeAction` followed by any deterministically failing action (e.g., `AddKeyAction` with a key that already exists). Both are achievable without validator, node-admin, or any privileged access.

### Recommendation

Move `validator_proposals.append(...)` inside the `Ok(_)` branch, so proposals are only promoted when the receipt succeeds:

```rust
match &result.result {
    Ok(_) => {
        validator_proposals.append(&mut result.validator_proposals);
        state_update.commit(StateChangeCause::ReceiptProcessing {
            receipt_hash: receipt.get_hash(),
        });
    }
    Err(_) => {
        result.validator_proposals.clear(); // discard proposals from failed receipt
        state_update.rollback();
    }
}
```

### Proof of Concept

1. Create account `attacker` with `amount = 10 NEAR`, `locked = 0`, storage usage ~100 bytes (storage staking requirement ~1 NEAR).
2. Submit a transaction from `attacker` to `attacker` with a single `StakeAction { stake: 9.5 NEAR, public_key: <validator_key> }`.
3. Runtime executes `action_stake`: `increment = 9.5 NEAR`, `new_balance = 0.5 NEAR`, check passes → `result.validator_proposals.push(ValidatorStake { account_id: attacker, stake: 9.5 NEAR })`, `account.amount = 0.5 NEAR`, `account.locked = 9.5 NEAR` (in-memory only).
4. `check_storage_stake(account, 0.5 NEAR, config)` fails → `result.result = Err(LackBalanceForState)`.
5. `validator_proposals.append(&mut result.validator_proposals)` — proposal for 9.5 NEAR is now in the block's proposals.
6. `state_update.rollback()` — trie unchanged; `attacker.locked` remains 0 in state.
7. Block is produced and accepted. `EpochInfoAggregator.all_proposals[attacker] = 9.5 NEAR`.
8. At epoch boundary: `stake_change[attacker] = 9.5 NEAR`, `max_of_stakes = 9.5 NEAR`, `account.locked() = 0`.
9. `update_validator_accounts` returns `Err(StorageInconsistentState("FATAL: staking invariant does not hold..."))`.
10. All nodes fail to apply the epoch-boundary block → chain halt.

    pub fn compute_stake_return_info(
        &self,
        last_block_hash: &CryptoHash,
    ) -> Result<(HashMap<AccountId, Balance>, HashMap<AccountId, Balance>), EpochError> {
        let next_next_epoch_id = EpochId(*last_block_hash);
        let validator_reward = self.get_epoch_info(&next_next_epoch_id)?.validator_reward().clone();

        let next_epoch_id = self.get_next_epoch_id(last_block_hash)?;
        let epoch_id = self.get_epoch_id(last_block_hash)?;
        tracing::debug!(target: "epoch_manager",
            epoch_id = ?next_next_epoch_id,
            prev_epoch_id = ?next_epoch_id,
            prev_prev_epoch_id= ?epoch_id,
        );

        // Since stake changes for epoch T are stored in epoch info for T+2, the one stored by epoch_id
        // is the prev_prev_stake_change.
        let prev_prev_stake_change = self.get_epoch_info(&epoch_id)?.stake_change().clone();
        let prev_stake_change = self.get_epoch_info(&next_epoch_id)?.stake_change().clone();
        let stake_change = self.get_epoch_info(&next_next_epoch_id)?.stake_change().clone();
        tracing::debug!(target: "epoch_manager",
            ?prev_prev_stake_change,
            ?prev_stake_change,
            ?stake_change,
        );
        let all_stake_changes =
            prev_prev_stake_change.iter().chain(&prev_stake_change).chain(&stake_change);
        let all_keys: HashSet<&AccountId> = all_stake_changes.map(|(key, _)| key).collect();

        let mut stake_info = HashMap::new();
        for account_id in all_keys {
            let new_stake = *stake_change.get(account_id).unwrap_or(&Balance::ZERO);
            let prev_stake = *prev_stake_change.get(account_id).unwrap_or(&Balance::ZERO);
            let prev_prev_stake = *prev_prev_stake_change.get(account_id).unwrap_or(&Balance::ZERO);
            let max_of_stakes =
                vec![prev_prev_stake, prev_stake, new_stake].into_iter().max().unwrap();
            stake_info.insert(account_id.clone(), max_of_stakes);
        }
        tracing::debug!(target: "epoch_manager", ?stake_info, ?validator_reward);
        Ok((stake_info, validator_reward))
