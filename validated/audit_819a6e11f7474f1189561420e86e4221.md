Audit Report

## Title
Missing Non-Positive Price Validation in `ChainlinkPriceOracle.getAssetPrice()` - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` casts the `int256` return value of `latestRoundData()` directly to `uint256` without verifying `price > 0`, and does not check whether the price has been clamped to Chainlink's `minAnswer`/`maxAnswer` circuit breaker bounds. This is inconsistent with the sibling contract `ChainlinkOracleForRSETHPoolCollateral`, which explicitly checks `if (ethPrice <= 0) revert InvalidPrice()`. The corrupted price flows into `LRTOracle._getTotalEthInProtocol()` and ultimately into `newRsETHPrice`, causing incorrect rsETH exchange rate computation.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` lines 52–54:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is performed on `price` before the cast. Two failure modes exist:

**Zero price:** `uint256(0) * 1e18 / decimals = 0`, silently zeroing that asset's entire TVL contribution in `_getTotalEthInProtocol()` (line 343: `totalETHInProtocol += totalAssetAmt.mulWad(assetER)`), deflating `newRsETHPrice` (line 250).

**Negative price:** In Solidity 0.8.x, `uint256(negative_int256)` wraps to near-`type(uint256).max`; the subsequent multiplication overflows and reverts, causing a DoS on `updateRSETHPrice()` (public, line 87).

**Circuit breaker:** If a supported LST depegs severely, Chainlink clamps the returned price to `minAnswer` (a floor above the true market price). The code accepts this inflated value, causing `_getTotalEthInProtocol()` to overstate TVL and `newRsETHPrice` to be set above the true backing value.

The contrast with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` line 32 (`if (ethPrice <= 0) revert InvalidPrice()`) confirms the project is aware of this pattern but did not apply it to `ChainlinkPriceOracle`.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (lines 252–266, 270–281) only mitigates these scenarios when the limit is non-zero and the price deviation exceeds it; if `pricePercentageLimit == 0` (the default), all checks are bypassed (`pricePercentageLimit > 0 && ...`).

## Impact Explanation
**Low — Contract fails to deliver promised returns.**

- **Circuit breaker / inflated price:** New depositors calling `LRTDepositPool.depositAsset()` receive fewer rsETH shares than the true asset value warrants, because `newRsETHPrice` is set above the true backing. Existing holders are unjustly enriched at new depositors' expense.
- **Zero price / deflated price:** Existing rsETH holders receive less ETH on withdrawal than they are owed, because `newRsETHPrice` is set below the true backing.
- **Negative price:** `updateRSETHPrice()` reverts, temporarily freezing price updates (DoS on a public function).

None of these scenarios result in direct fund theft or permanent freezing, placing the impact squarely in the **Low** tier.

## Likelihood Explanation
`updateRSETHPrice()` is public with no access control (line 87), so any external caller can trigger the mispricing. Chainlink circuit breakers have triggered historically (e.g., LUNA crash, November 2022 LUNA/UST event). The supported assets are LSTs whose ETH-denominated feeds could plausibly hit `minAnswer` during a severe depeg. The zero/negative price scenarios are less likely from a well-operated Chainlink feed but are not impossible (e.g., feed deprecation, misconfiguration). The inconsistency with the sibling contract confirms this is an oversight rather than an intentional design choice.

## Recommendation
Add the following checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();

    if (price <= 0) revert InvalidPrice();

    // Optional: reject prices at Chainlink circuit breaker bounds
    // int192 minAnswer = IChainlinkAggregator(address(priceFeed)).minAnswer();
    // int192 maxAnswer = IChainlinkAggregator(address(priceFeed)).maxAnswer();
    // if (price <= minAnswer || price >= maxAnswer) revert PriceOutOfBounds();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept
**Circuit breaker path (most realistic):**
1. A supported LST (e.g., stETH) suffers a severe depeg; its true ETH price falls to 0.01e18.
2. Chainlink's aggregator clamps the returned price to `minAnswer` (e.g., 0.5e18).
3. Any caller invokes `LRTOracle.updateRSETHPrice()` (public, no access control).
4. Call chain: `updateRSETHPrice()` → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns 0.5e18 instead of 0.01e18.
5. `totalETHInProtocol` is inflated by `(0.5 - 0.01) * totalStETHDeposits`.
6. `newRsETHPrice` is set above the true backing value (line 250).
7. New depositors calling `depositAsset()` receive fewer rsETH shares than their deposit warrants.

- Set `mockAggregator.latestRoundData()` to return `minAnswer` (e.g., 5e17).
