### Title
Unprotected `init` Function Allows Frontrunning to Seize MPC Participant Control - (File: `crates/contract/src/lib.rs`)

### Summary

The MPC contract's `init` function lacks `#[private]` access control and is deployed in a separate transaction from the contract itself. An attacker who monitors the NEAR blockchain can front-run the legitimate `init` call and install an arbitrary participant set, gaining full control over the MPC network's key generation and signing capability.

### Finding Description

The deployment flow for the MPC contract is a two-step process. First, the contract is deployed **without** an initialization call: [1](#0-0) 

Then, in a separate transaction, `init` is called: [2](#0-1) 

The same two-step pattern appears in the testnet and TEE production deployment scripts: [3](#0-2) 

The `init` function in the contract carries only the `#[init]` NEAR SDK modifier, which solely prevents re-initialization after the first call. It does **not** carry `#[private]`, which would restrict the caller to `env::current_account_id()`. This is confirmed by the ABI snapshot, where `init` lists only `"init"` as a modifier, while `init_running` correctly lists both `"init"` and `"private"`: [4](#0-3) 

Because `init` accepts arbitrary `parameters` (the full `ThresholdParameters` struct including the participant set and threshold value), any account that calls it first wins: [5](#0-4) 

### Impact Explanation

An attacker who front-runs `init` can supply a `ThresholdParameters` value containing only their own account as the sole participant with `threshold = 1`. From that point:

- The contract enters the `Initializing` state with the attacker's participant set.
- The attacker participates in DKG as the only node, generating MPC key shares they fully control.
- The contract transitions to `Running` state with keys the attacker owns outright.
- The attacker can then issue threshold signatures for any foreign-chain transaction without any legitimate participant authorization.

This satisfies the **Critical** impact class: *Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization*, and *Bypass of threshold-signature requirements or unauthorized access to MPC key shares*.

### Likelihood Explanation

NEAR transactions are publicly visible in the mempool. Any attacker watching for the MPC contract bytecode hash being deployed can immediately submit a competing `init` call. The window between `deploy` and `init` is at least one block (and in practice several blocks given the scripts include `sleep` delays). No special privilege or collusion is required — only the ability to submit a NEAR transaction.

### Recommendation

Apply `#[private]` to the `init` function so that only the contract account itself (`env::current_account_id() == env::predecessor_account_id()`) can call it. Alternatively, combine deployment and initialization into a single atomic transaction using NEAR's `--initFunction` / `--initArgs` flags in the deployment call, eliminating the window entirely. The `init_running` function already demonstrates the correct pattern with `#[private]`: [6](#0-5) 

The `init` function should be updated to match this pattern.

### Proof of Concept

1. Attacker watches NEAR for a transaction deploying the MPC contract bytecode (identifiable by its known code hash).
2. In the same block or the next block, attacker submits:
   ```
   near contract call-function as-transaction <mpc_contract_account> init \
     json-args '{"parameters": {"participants": {"next_id": {"id": 1}, "participants": [["attacker.near", {"id": 0}, {"url": "http://attacker.example.com", "cipher_pk": "...", "sign_pk": "..."}]]}, "threshold": {"value": 1}}, "init_config": null}' \
     prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
     sign-as attacker.near network-config mainnet sign-with-keychain send
   ```
3. If this transaction is included before the legitimate deployer's `init` call, the contract is initialized with the attacker as the sole participant.
4. The attacker's node participates in DKG alone, generating key shares it fully controls.
5. The attacker can now produce threshold signatures for arbitrary foreign-chain transactions, stealing any funds or approvals directed at the MPC network's derived addresses. [7](#0-6)
