Audit Report

## Title
Zero Rate Propagation via Unguarded `lzReceive` Bricks Pool Deposits — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`MultiChainRateProvider.updateRate()` is permissionless and broadcasts whatever `getLatestRate()` returns, including zero, via LayerZero. `CrossChainRateReceiver.lzReceive()` stores the decoded rate unconditionally with no zero-check. Once a zero rate is stored, every call to `RSETHPool.viewSwapRsETHAmountAndFee()` and `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee()` reverts with a division-by-zero panic, blocking all ETH and token deposits until a new non-zero rate is bridged.

## Finding Description
`MultiChainRateProvider.updateRate()` carries no access control — it is `external payable` with only a `nonReentrant` guard. [1](#0-0) 

`RSETHMultiChainRateProvider.getLatestRate()` delegates directly to `ILRTOracle.rsETHPrice()` with no zero-check. [2](#0-1) 

The result is encoded and sent via LayerZero unconditionally, even if it is zero. [3](#0-2) 

On the receiving side, `lzReceive()` validates only that `msg.sender == layerZeroEndpoint`, `_srcChainId == srcChainId`, and `srcAddress == rateProvider` — all of which are satisfied by a legitimate LayerZero delivery of a zero-rate message. It then writes the decoded value directly to `rate` with no zero-check. [4](#0-3) 

`RSETHPool.viewSwapRsETHAmountAndFee()` divides by the stored rate with no guard: [5](#0-4) 

`RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee()` has the identical pattern: [6](#0-5) 

Both `deposit()` functions call `viewSwapRsETHAmountAndFee()` internally, so a zero rate causes every deposit to revert. [7](#0-6) [8](#0-7) 

## Impact Explanation
All ETH and token deposits into `RSETHPool` and `RSETHPoolNoWrapper` revert with a division-by-zero panic for as long as the zero rate is stored. No funds already in the pool are lost, but the pool fails to deliver its core promised function (accepting deposits) for the duration. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`updateRate()` is permissionless, so any external caller can trigger propagation the moment `rsETHPrice()` returns 0. A zero return from `rsETHPrice()` is a realistic transient condition during oracle upgrades, migrations, or first-deployment initialization. Once the zero-rate LayerZero message is delivered, the condition persists until a new non-zero `updateRate()` call is made and the cross-chain message is delivered, which introduces a non-trivial recovery window.

## Recommendation
Add a zero-check in `lzReceive()` before storing the rate:

```solidity
uint256 _rate = abi.decode(_payload, (uint256));
require(_rate > 0, "Rate must be non-zero");
rate = _rate;
```

Optionally, also guard `updateRate()` before broadcasting:

```solidity
uint256 latestRate = getLatestRate();
require(latestRate > 0, "Rate must be non-zero");
```

## Proof of Concept
```solidity
// 1. Deploy MockLRTOracle returning 0 for rsETHPrice()
// 2. Deploy RSETHMultiChainRateProvider with MockLRTOracle
// 3. Deploy CrossChainRateReceiver (concrete subclass) pointing to a mock LZ endpoint
// 4. Call provider.updateRate{value: fee}() — succeeds with no access control
// 5. Impersonate the LayerZero endpoint; call receiver.lzReceive(srcChainId, srcAddress, 0, abi.encode(uint256(0)))
//    — passes all three require checks, sets rate = 0
// 6. Assert receiver.getRate() == 0
// 7. Call RSETHPool.viewSwapRsETHAmountAndFee(1 ether)
//    — reverts: division by zero at `amountAfterFee * 1e18 / rsETHToETHrate`
// 8. Call RSETHPool.deposit{value: 1 ether}("ref")
//    — reverts for the same reason, confirming deposit DoS
```
