Audit Report

## Title
`LRTDepositPool._beforeDeposit` — Remaining LST Deposit Capacity Becomes Permanently Inaccessible When `depositLimit - totalDeposits < minAmountToDeposit` - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool` enforces a per-asset deposit ceiling (`depositLimitByAsset`) and a global minimum deposit floor (`minAmountToDeposit`). When the remaining capacity for an LST asset falls below `minAmountToDeposit`, the two sequential checks in `_beforeDeposit` create a mutually exclusive revert condition: no valid deposit amount exists. The remaining capacity is permanently inaccessible via the public deposit path until an admin intervenes, while `getAssetCurrentLimit` continues to advertise the capacity as available.

## Finding Description
`_beforeDeposit` (L648–670) applies two sequential checks:

```solidity
// L657-659
if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
    revert InvalidAmountToDeposit();
}
// L661-663
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
```

For LST assets, `_checkIfDepositAmountExceedesCurrentLimit` (L681) evaluates:
```solidity
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

When `depositLimitByAsset[asset] - totalAssetDeposits < minAmountToDeposit`:
- Any `depositAmount >= minAmountToDeposit` causes `totalAssetDeposits + depositAmount > depositLimitByAsset` → `MaximumDepositLimitReached`
- Any `depositAmount < minAmountToDeposit` → `InvalidAmountToDeposit`

No valid deposit amount exists. Meanwhile, `getAssetCurrentLimit` (L402–409) returns `depositLimitByAsset - totalAssetDeposits` without accounting for `minAmountToDeposit`, advertising a non-zero available capacity that is unreachable.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** The deposit pool cannot be filled to its stated limit for the affected LST asset. Users who attempt to deposit are blocked even though the protocol's own accounting (`getAssetCurrentLimit`) shows available capacity, creating a misleading and stuck state. No funds are lost, but the protocol fails to accept deposits it claims to support.

## Likelihood Explanation
No malicious actor is required. As deposits for a given LST accumulate organically toward `depositLimitByAsset`, the remaining capacity will eventually fall below `minAmountToDeposit`. Since `minAmountToDeposit` is a single global value applied to all assets and `depositLimitByAsset` is per-asset, the two parameters can diverge without any coordination check. The condition is not self-healing — it persists until an admin calls `setMinAmountToDeposit` (L282–285) or `updateAssetDepositLimit` (LRTConfig.sol L123–133). Any unprivileged depositor will encounter this state during normal protocol usage.

## Recommendation
Two viable fixes:
1. In `_beforeDeposit`, when `depositAmount >= minAmountToDeposit` but `totalAssetDeposits + depositAmount > depositLimitByAsset`, cap the deposit at the remaining capacity (`depositLimitByAsset - totalAssetDeposits`) provided it is non-zero, rather than reverting.
2. Enforce at configuration time (in `setMinAmountToDeposit` and `updateAssetDepositLimit`) that `depositLimitByAsset[asset] - currentTotalDeposits >= minAmountToDeposit` for all supported assets, preventing the stuck state from being entered.

Additionally, `getAssetCurrentLimit` should return `0` (or the actual usable limit) when the remaining capacity is below `minAmountToDeposit`, to avoid advertising unreachable capacity.

## Proof of Concept
**Setup:**
- `minAmountToDeposit = 1e18`
- `depositLimitByAsset[stETH] = 100e18`
- `totalAssetDeposits(stETH) = 99.5e18`
- Remaining capacity: `0.5e18`

**Attempt 1 — deposit `1e18` (at minimum):**
```
totalAssetDeposits + 1e18 = 100.5e18 > 100e18 → MaximumDepositLimitReached
```

**Attempt 2 — deposit `0.5e18` (exactly the remaining capacity):**
```
0.5e18 < minAmountToDeposit (1e18) → InvalidAmountToDeposit
```

Both paths revert. `getAssetCurrentLimit(stETH)` still returns `0.5e18`. The pool is stuck until admin intervention.

**Foundry test plan:**
```solidity
function test_depositStuck() public {
    // set minAmountToDeposit = 1e18, depositLimit = 100e18
    // deposit 99.5e18 (passes both checks)
    // attempt deposit of 1e18 → expect revert MaximumDepositLimitReached
    // attempt deposit of 0.5e18 → expect revert InvalidAmountToDeposit
    // assert getAssetCurrentLimit(stETH) == 0.5e18 (misleading non-zero)
}
```
