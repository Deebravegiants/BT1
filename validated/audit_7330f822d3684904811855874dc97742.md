Audit Report

## Title
`LRTDepositPool#getRsETHAmountToMint` divides a live asset price by a stale cached `rsETHPrice`, minting excess rsETH to depositors at the expense of existing holders — (File: `contracts/LRTDepositPool.sol`)

## Summary

`getRsETHAmountToMint` computes the rsETH mint amount as `(amount × lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`. The numerator uses a live Chainlink price while the denominator reads a cached state variable that is only refreshed by an explicit external call to `updateRSETHPrice()`. Any depositor who deposits while the cached price is stale receives more rsETH than their proportional share, permanently diluting the yield that has accrued to existing holders since the last oracle update.

## Finding Description

`LRTDepositPool.getRsETHAmountToMint` at line 520 computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`lrtOracle.getAssetPrice(asset)` delegates to a live `IPriceFetcher` (Chainlink) and always returns the current market rate: [2](#0-1) 

`lrtOracle.rsETHPrice()` reads a plain storage variable: [3](#0-2) 

This variable is only written inside `_updateRsETHPrice()`, which is invoked only when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is called externally: [4](#0-3) [5](#0-4) 

The deposit entry point `_beforeDeposit` is a `view` function that calls `getRsETHAmountToMint` without triggering any price refresh: [6](#0-5) 

Because LSTs (stETH, etc.) accrue staking rewards continuously, `getAssetPrice(asset)` rises every block while `rsETHPrice` remains frozen at its last-updated value. The ratio `getAssetPrice / rsETHPrice` therefore exceeds 1 for any depositor who acts between two keeper calls, and the excess rsETH minted is backed by no additional ETH value — it is extracted from the yield that has accrued to existing holders.

No existing guard prevents this: `minRSETHAmountExpected` is a slippage floor set by the depositor themselves, deposit-limit checks are purely quantity-based, and `nonReentrant`/`whenNotPaused` are irrelevant to the price mismatch.

## Impact Explanation

Every deposit made while `rsETHPrice` is stale and the underlying LST price has risen mints rsETH in excess of the depositor's true proportional claim. When `updateRSETHPrice()` is eventually called, the new price is computed over the inflated supply, permanently reducing the per-token value for all pre-existing holders. This is a concrete, on-chain transfer of accrued staking yield from existing rsETH holders to the depositor.

**Impact: High — Theft of unclaimed yield.**

## Likelihood Explanation

- stETH and other LSTs increase in ETH value every block due to continuous staking rewards; the price gap opens immediately after each keeper call.
- `updateRSETHPrice()` is called on a periodic off-chain schedule, not on every deposit, so the window is always open between calls.
- `depositAsset` is fully permissionless — any EOA or contract can call it with no special role or setup.
- No front-running, flash loans, or external compromise is required; a normal deposit at any point in the keeper interval passively captures the excess.
- The gain scales linearly with deposit size and time elapsed since the last update, making it attractive for large depositors to time their entry.

**Likelihood: High.**

## Recommendation

Before computing `rsethAmountToMint`, refresh `rsETHPrice` by calling `_updateRsETHPrice()` (made `internal`) at the start of `depositAsset` or inside `_beforeDeposit`. This ensures the denominator reflects the same live asset prices as the numerator. Alternatively, compute the mint amount entirely from live prices without relying on the cached `rsETHPrice` state variable (e.g., derive the current rsETH price on-the-fly from `_getTotalEthInProtocol() / rsethSupply` at deposit time).

## Proof of Concept

**State at T0 (last `updateRSETHPrice` call):**
- Protocol holds 100 stETH; `rsethSupply = 100`; `rsETHPrice = 1.050e18` (stETH/ETH = 1.050)

**State at T1 (staking rewards accrued, keeper not yet called):**
- Chainlink stETH/ETH feed returns `1.060e18`; `rsETHPrice` still `1.050e18`

**Alice deposits 10 stETH:**
1. `depositAsset(stETH, 10e18, 0, "")` → `_beforeDeposit` → `getRsETHAmountToMint`
2. `rsethAmountToMint = 10e18 × 1.060e18 / 1.050e18 = 10.0952e18`
3. Alice receives **10.0952 rsETH**; fair amount is **10.0 rsETH**

**After deposit, keeper calls `updateRSETHPrice`:**
- `totalETHInProtocol = 110 × 1.060 = 116.6 ETH`
- `rsethSupply = 110.0952`
- `newRsETHPrice = 116.6 / 110.0952 ≈ 1.05908e18`
- Existing 100 rsETH is worth `100 × 1.05908 = 105.908 ETH`
- Fair value (without Alice's dilution): `100 × (116.6 / 110) = 106.0 ETH`
- **Existing holders permanently lost ~0.092 ETH of yield to Alice**

A Foundry fork test can reproduce this by: (1) forking mainnet, (2) recording `rsETHPrice` and the live stETH/ETH Chainlink answer, (3) calling `depositAsset` without calling `updateRSETHPrice` first, (4) asserting that `rsethAmountToMint > depositAmount` and that after a subsequent `updateRSETHPrice` call the per-token value for pre-existing holders is lower than it would have been had the price been refreshed before the deposit.
