### Title
`detach_private_scalar` accepts attacker-crafted blobs with `x_share=0` and a mismatched `Qi_self`, silently emitting a wrong private scalar — (`src/cbmpc/api/eddsa_mp.cpp`)

### Summary
`detach_private_scalar` lacks the `x_share * G == Qi_self` consistency check that `attach_private_scalar` enforces. An attacker-controlled blob with `x_share=0` and any valid non-infinity `Qi_self` passes all guards in `detach_private_scalar` and causes it to return `SUCCESS` while emitting a zero scalar. The emitted scalar is then permanently irreconcilable with the blob's public share, because `attach_private_scalar` enforces the missing check and rejects it.

### Finding Description

`detach_private_scalar` performs three checks before emitting the scalar:

1. Version check (line 464–466) [1](#0-0) 
2. Curve check (line 466–468)
3. Range check: `if (!q.is_in_range(blob.x_share))` (line 471) [2](#0-1) 

`is_in_range` is defined as `a.sign() >= 0 && a < m`, so `x_share = 0` passes trivially. [3](#0-2) 

After passing these checks, the function unconditionally emits the scalar and returns `SUCCESS`:

```cpp
out_private_scalar_fixed = blob.x_share.to_bin(order_size);
blob.x_share = 0;
out_public_key_blob = coinbase::convert(blob);
return SUCCESS;
``` [4](#0-3) 

There is **no** `x_share * G == Qi_self` check here. That check exists only in `attach_private_scalar`:

```cpp
if (x * G != Qi_self) return coinbase::error(E_FORMAT, "x_share mismatch key blob");
``` [5](#0-4) 

### Impact Explanation

An attacker who can supply a `key_blob` (e.g., a malicious peer in a key-export or blob-storage flow) crafts a blob with `x_share=0` and a valid non-infinity `Qi_self` (e.g., the generator point). `detach_private_scalar` returns `SUCCESS` and emits a zero scalar. The honest caller:

- Receives a false `SUCCESS` signal, believing the blob is valid.
- Holds a zero private scalar that does not correspond to any legitimate key share.
- Cannot reattach it: `attach_private_scalar` will reject it at line 523 because `0 * G ≠ Qi_self`.
- If the zero scalar is forwarded into a signing protocol, it produces an invalid or trivially forgeable partial signature.

This falls under **High** impact: an attacker-controlled blob is accepted under the wrong key, and the API emits an invalid private scalar with a `SUCCESS` return code.

### Likelihood Explanation

Any caller that accepts a `key_blob` from an external source (network peer, storage, IPC) and passes it to `detach_private_scalar` is reachable. The craft is trivial: set `x_share=0` in the serialized blob while keeping a valid `Qi_self`. No threshold collusion or privileged access is required.

### Recommendation

Add the same `x_share * G == Qi_self` consistency check to `detach_private_scalar` before emitting the scalar, mirroring the guard already present in `attach_private_scalar` at line 523. Additionally, consider rejecting `x_share = 0` explicitly, since a zero private scalar share is cryptographically invalid in any threshold scheme.

### Proof of Concept

```cpp
// 1. Obtain or construct a legitimate key_blob_v1_t for ed25519.
// 2. Overwrite x_share = 0 in the serialized blob, keeping Qi_self = some valid non-infinity point.
// 3. Call detach_private_scalar with the crafted blob.
buf_t pub_blob, scalar;
error_t rv = detach_private_scalar(crafted_blob, pub_blob, scalar);
assert(rv == SUCCESS);           // passes — missing guard
assert(scalar == zero_32_bytes); // emits zero scalar

// 4. Attempt to reattach: fails because 0*G != Qi_self.
buf_t out_blob;
rv = attach_private_scalar(pub_blob, scalar, original_Qi_self_compressed, out_blob);
assert(rv != SUCCESS);           // E_FORMAT: "x_share mismatch key blob"
// API disagrees: detach accepted the blob, attach rejects the scalar.
```
