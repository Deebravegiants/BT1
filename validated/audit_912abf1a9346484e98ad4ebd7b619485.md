Audit Report

## Title
Minimum Deposit Floor and Per-Asset Deposit Ceiling Create Permanent Deadlock Near Capacity - (File: contracts/LRTDepositPool.sol)

## Summary
In `LRTDepositPool._beforeDeposit`, a `minAmountToDeposit` floor check and a `depositLimitByAsset` ceiling check interact such that when remaining deposit capacity for an LST asset falls below `minAmountToDeposit`, no deposit amount can satisfy both guards simultaneously. The remaining capacity becomes permanently inaccessible, and the protocol can never reach its configured deposit limit for that asset.

## Finding Description
`_beforeDeposit` enforces two sequential guards:

```solidity
// LRTDepositPool.sol L657-663
if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
    revert InvalidAmountToDeposit();
}
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [1](#0-0) 

For LST assets, the ceiling check is:

```solidity
// LRTDepositPool.sol L681
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
``` [2](#0-1) 

Let `R = depositLimitByAsset(asset) - totalAssetDeposits` (remaining capacity). When `0 < R < minAmountToDeposit`:
- Any `d < minAmountToDeposit` → reverts `InvalidAmountToDeposit` (floor check fails).
- Any `d >= minAmountToDeposit` → `totalAssetDeposits + d > depositLimitByAsset` → reverts `MaximumDepositLimitReached` (ceiling check fails).

No value of `d` satisfies both guards. The state variables involved are confirmed in the codebase: [3](#0-2) [4](#0-3) 

## Impact Explanation
The protocol permanently under-collects relative to its configured `depositLimitByAsset` ceiling once remaining capacity drops below `minAmountToDeposit`. No user funds are lost or frozen, but the protocol fails to reach its configured fundraising target. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
No attacker action is required. As organic depositors bring `totalAssetDeposits` close to `depositLimitByAsset`, the remaining capacity naturally shrinks. The moment it falls below `minAmountToDeposit` (a non-trivial value settable by admin, e.g. 0.01 ETH), the deadlock is permanent until an admin raises the limit or lowers the minimum. The entry path is the public `depositAsset` function callable by any user. [5](#0-4) 

## Recommendation
In `_beforeDeposit`, relax the minimum deposit check when the deposit amount exactly fills the remaining capacity:

```solidity
uint256 remainingCapacity = lrtConfig.depositLimitByAsset(asset) - getTotalAssetDeposits(asset);
if (depositAmount == 0 || (depositAmount < minAmountToDeposit && depositAmount != remainingCapacity)) {
    revert InvalidAmountToDeposit();
}
```

This allows a depositor to fill the last slot even if it is smaller than `minAmountToDeposit`. [6](#0-5) 

## Proof of Concept
1. Admin sets `depositLimitByAsset(stETH) = 100_000e18` and `minAmountToDeposit = 0.01e18`.
2. Organic depositors bring `totalAssetDeposits(stETH)` to `99_999.995e18`. Remaining capacity `R = 0.005e18`.
3. Alice calls `depositAsset(stETH, 0.005e18, ...)`:
   - `0.005e18 < 0.01e18` → reverts `InvalidAmountToDeposit`.
4. Alice calls `depositAsset(stETH, 0.01e18, ...)`:
   - `99_999.995e18 + 0.01e18 = 100_000.005e18 > 100_000e18` → reverts `MaximumDepositLimitReached`.
5. No deposit amount can succeed. The remaining `0.005e18` capacity is permanently locked and the protocol never reaches its 100,000 stETH target.
