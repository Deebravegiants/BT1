Audit Report

## Title
Stale Cross-Chain Oracle Rate Allows Over-Minting of wrsETH on L2 Pools - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally with no staleness check, despite tracking `lastUpdated`. All L2 pool variants call this oracle to compute wrsETH mint amounts by dividing deposited ETH by the rate. When the L2 oracle lags behind the true L1 rsETH price, depositors receive more wrsETH than their ETH is worth at the current exchange rate, extracting accrued yield from existing holders.

## Finding Description
`CrossChainRateReceiver.lzReceive` stores the rate and timestamp on receipt:

```solidity
rate = _rate;
lastUpdated = block.timestamp;
``` [1](#0-0) 

`getRate()` returns the stored value with no staleness guard:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
``` [2](#0-1) 

`RSETHRateReceiver` inherits this implementation directly and adds no additional logic: [3](#0-2) 

Every L2 pool's `getRate()` delegates to this oracle: [4](#0-3) 

`viewSwapRsETHAmountAndFee` uses the stale rate as the denominator:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [5](#0-4) 

The same pattern is present in `RSETHPoolV3ExternalBridge`: [6](#0-5) 

And in `RSETHPoolV3WithNativeChainBridge`, which stores `rsETHOracle` identically: [7](#0-6) 

The `deposit()` function mints wrsETH directly from this computed amount with no additional rate-freshness check: [8](#0-7) 

The L1 rsETH price is monotonically non-decreasing as rewards accrue: [9](#0-8) 

The only existing guard is a `dailyMintLimit`, which caps total daily volume but does not prevent the rate discrepancy from being exploited within that cap. [10](#0-9) 

## Impact Explanation
Every wrsETH token is a proportional claim on the L1 rsETH pool. Over-minting wrsETH at a stale (lower) rate dilutes the claims of all existing wrsETH/rsETH holders. The attacker bridges or redeems the surplus wrsETH on L1 at the true current rsETH price, extracting ETH value that rightfully belongs to prior depositors as accrued yield. This matches the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation
The L2 oracle is updated only when a LayerZero message is relayed from L1. Any gap — network congestion, relayer downtime, or infrequent pushes — creates an exploitable window. Because rsETH price increases continuously with EigenLayer rewards, even a few hours of staleness produces a measurable discrepancy. An attacker can monitor `lastUpdated` on-chain and deposit precisely when the gap is largest. No privileged access is required; any external caller can invoke `deposit()`.

## Recommendation
Add a configurable maximum staleness threshold and revert in `getRate()` if the rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 1 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

Alternatively, each L2 pool's deposit path should check `CrossChainRateReceiver.lastUpdated` before accepting a deposit.

## Proof of Concept
1. L1 `LRTOracle.updateRSETHPrice()` is called; `rsETHPrice` rises from `1.05e18` to `1.10e18`.
2. The LayerZero relayer has not yet delivered the updated rate to `RSETHRateReceiver` on L2; `rate` still reads `1.05e18`.
3. Attacker calls `RSETHPoolV3.deposit{value: 1 ether}("")`.
4. `viewSwapRsETHAmountAndFee(1 ether)` computes `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.9524 wrsETH`.
5. Fair amount at current rate: `1e18 * 1e18 / 1.10e18 ≈ 0.9091 wrsETH`.
6. Attacker receives **≈ 0.0433 wrsETH surplus** — worth ≈ 0.0476 ETH at the true rate — at the expense of existing holders.
7. Attacker bridges or redeems the wrsETH on L1 for the full current rsETH value, realising the profit.

**Foundry fork test plan**: Fork an L2 where `RSETHRateReceiver` is deployed. Warp `block.timestamp` forward by several hours without triggering `lzReceive`. Call `RSETHPoolV3.deposit` and assert that the minted `wrsETH` amount exceeds `depositedETH * 1e18 / currentL1Rate`, confirming over-minting relative to the true rate.
