### Title
Unguarded `init` Allows Unprivileged Pre-Initialization, Permanently Locking Out Legitimate MPC Network Setup - (File: crates/contract/src/lib.rs)

### Summary

The MPC contract's `init` function lacks any access-control guard (`#[private]` or caller check). Any NEAR account can call it before the legitimate deployer, permanently locking in an attacker-controlled participant set and threshold. Because NEAR's `#[init]` macro prevents re-initialization once state exists, the legitimate `init` call will always fail after a successful attacker call, and the attacker gains permanent control over who can issue threshold signatures.

### Finding Description

The `init` function is the sole entry point for bootstrapping the MPC contract from `NotInitialized` to `Running`: [1](#0-0) 

It carries no `#[private]` decorator and performs no caller check — any NEAR account can invoke it. Compare this with `init_running`, which is correctly restricted: [2](#0-1) 

The deployment workflow calls `init` as a **separate transaction** from contract deployment: [3](#0-2) 

This creates a window between deployment and initialization. An attacker monitoring the chain for the deployment receipt can submit an `init` call with attacker-controlled `parameters` (e.g., `participants = [attacker_account]`, `threshold = 1`) before the legitimate deployer does. The call succeeds because `parameters.validate()` only checks structural validity, not participant legitimacy: [4](#0-3) 

Once the attacker's `init` succeeds, the contract state exists. Every subsequent `init` call — including the legitimate deployer's — panics with "contract already initialized" (NEAR SDK's `#[init]` invariant). The sandbox test confirms this behavior: [5](#0-4) 

The attacker's participant set is now canonical. The `ProtocolContractState` is permanently `Running` with the attacker's parameters: [6](#0-5) 

### Impact Explanation

With the attacker as the sole participant at threshold 1, they can:

1. **Issue arbitrary threshold signatures** via `respond` — the attacker's account passes `assert_caller_is_attested_participant_and_protocol_active` because they are the only participant in the stored `Participants` set.
2. **Permanently suppress the legitimate MPC network** — the real operators can never call `init` again; `init_running` is `#[private]` and also `#[init]`, so it too fails once state exists.
3. **Control all governance votes** — `vote_new_parameters`, `vote_add_domains`, etc., all require the caller to be in the participant set, which the attacker controls.

This maps to the Critical allowed impact: *Unauthorized threshold signature issuance and bypass of threshold-signature requirements without the required participant authorization.*

### Likelihood Explanation

- **No special resources required**: any NEAR account with enough gas can call `init`.
- **Trustless and cheap**: a single transaction, no token transfers, no approvals.
- **Observable trigger**: contract deployment is a public on-chain event; an attacker can watch for it and submit the front-running `init` in the next block.
- **Irreversible**: NEAR's `#[init]` guard makes the lock-in permanent without a full contract re-deploy and state wipe, which itself requires the contract account's full-access key.

### Recommendation

Add `#[private]` to `init` so only the contract account itself can call it (matching the pattern already used on `init_running`):

```rust
#[handle_result]
#[init]
#[private]          // ← add this
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

Alternatively, assert `env::predecessor_account_id() == env::current_account_id()` at the top of `init`. Either fix closes the window entirely.

### Proof of Concept

1. Attacker observes the MPC contract deployment transaction on-chain.
2. Attacker immediately submits:
   ```
   near contract call-function as-transaction <mpc_contract_account> init \
     json-args '{"parameters": {"participants": [{"account_id": "attacker.near", ...}], "threshold": 1}}' \
     sign-as attacker.near ...
   ```
3. Attacker's `init` succeeds; contract state is now `Running` with `participants = [attacker.near]`, `threshold = 1`.
4. Legitimate deployer submits their `init` call → panics: *"The contract has already been initialized"*.
5. Attacker calls `submit_participant_info` with a valid (or mocked) attestation for `attacker.near`.
6. Attacker calls `respond(request, forged_signature)` for any pending `sign` request — passes all participant and attestation checks because they are the only participant. [7](#0-6)
