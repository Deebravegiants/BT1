### Title
Unbounded `sub_epoch_n` Index Into `summaries` List Causes Unhandled `IndexError` During Weight Proof Validation — (`chia/full_node/weight_proof.py`)

---

### Summary

In `_validate_sub_epoch_segments`, the `sub_epoch_n` value is read directly from attacker-controlled `WeightProof.sub_epoch_segments` and used to index into the `summaries` list (derived from `WeightProof.sub_epochs`) without any bounds check. A malicious peer can craft a `WeightProof` where `sub_epoch_n` in a segment exceeds `len(summaries)`, triggering an unhandled `IndexError` that propagates out of the validation function.

---

### Finding Description

`_validate_sub_epoch_segments` in `chia/full_node/weight_proof.py` iterates over segments grouped by their `sub_epoch_n` field:

```python
for sub_epoch_n, segments in segments_by_sub_epoch.items():
    ...
    if sub_epoch_n > 0:
        ...
        prev_ses = summaries[sub_epoch_n - 1]   # line 983
        ...
    if not summaries[sub_epoch_n].reward_chain_hash == rc_sub_slot_hash:  # line 985
``` [1](#0-0) 

The `summaries` list is built from `weight_proof.sub_epochs` (the `sub_epoch_data` field), which has exactly `len(weight_proof.sub_epochs)` entries. The `sub_epoch_n` values come from the `sub_epoch_n` field of each `SubEpochChallengeSegment` inside `weight_proof.sub_epoch_segments` — a completely separate, independently attacker-controlled list. [2](#0-1) 

There is **no check** that `sub_epoch_n < len(summaries)` before either index operation. The only guard present is a per-sub-epoch segment count check (`len(segments) > max_segments`), which does not protect against an out-of-range `sub_epoch_n`. [3](#0-2) 

The `_validate_sub_epoch_summaries` call that precedes `_validate_sub_epoch_segments` validates the summaries themselves but does not cross-check `sub_epoch_n` values in segments against `len(summaries)`. [4](#0-3) 

---

### Impact Explanation

A malicious peer sends a crafted `WeightProof` (via `RespondProofOfWeight`) where:
- `sub_epochs` contains N entries (so `summaries` has N entries after validation)
- `sub_epoch_segments` contains at least one `SubEpochChallengeSegment` with `sub_epoch_n >= N`

When `_validate_sub_epoch_segments` processes this segment, `summaries[sub_epoch_n]` at line 985 raises an unhandled `IndexError`. This exception is not caught inside `_validate_sub_epoch_segments` (the function only returns `None` on logical failures, not on exceptions).

In the `validate_weight_proof` path, this runs inside a `ProcessPoolExecutor` subprocess via `validate_weight_proof_inner`; the subprocess crashes and the awaited future re-raises the exception in the main process. [5](#0-4) 

In the `validate_weight_proof_single_proc` path (used in tests and potentially in sync), the exception propagates directly to the caller. [6](#0-5) 

If the sync code does not catch this exception, the full node crashes. Even if caught, a malicious peer can repeatedly send crafted weight proofs to prevent a syncing node from completing sync — a long-lived inability to process sync updates.

---

### Likelihood Explanation

Any peer on the network can send a `RespondProofOfWeight` message. No keys, admin access, or special privileges are required. The crafted proof only needs to pass `_validate_sub_epoch_summaries` (which validates the `sub_epochs` list independently) before reaching the vulnerable indexing code. The attacker simply needs to include a segment with a `sub_epoch_n` value larger than the number of entries in `sub_epochs`.

---

### Recommendation

Add an explicit bounds check at the start of the `sub_epoch_n` loop body in `_validate_sub_epoch_segments`:

```python
if sub_epoch_n >= len(summaries):
    log.error(f"sub_epoch_n {sub_epoch_n} out of range for summaries of length {len(summaries)}")
    return None
```

This mirrors the existing guard for segment count and ensures the function returns `None` (graceful failure) rather than raising an unhandled exception. [7](#0-6) 

---

### Proof of Concept

Craft a `WeightProof` object where:
1. `sub_epochs` contains exactly 2 `SubEpochData` entries (minimum to pass the `len(summaries) < 2` guard at line 582).
2. `sub_epoch_segments` contains one `SubEpochChallengeSegment` with `sub_epoch_n = 100`.
3. `recent_chain_data` is populated to pass `_validate_sub_epoch_summaries`.

When `validate_weight_proof_single_proc` (or `validate_weight_proof`) processes this proof, `_validate_sub_epoch_segments` reaches line 985 and executes `summaries[100]` on a list of length 2, raising `IndexError: list index out of range`. [8](#0-7)
