Audit Report

## Title
Strict FIFO `break` in `_unlockWithdrawalRequests` Allows Large Head-of-Queue Request to Block All Subsequent Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`_unlockWithdrawalRequests` processes withdrawal requests in strict FIFO order and hard-`break`s the entire loop when the head-of-queue request cannot be covered by the vault's current liquid balance. Because `nextLockedNonce` is not advanced past the blocking request, every subsequent request — regardless of size — is frozen until the vault accumulates enough assets to satisfy the head request first. Users whose rsETH was already transferred at `initiateWithdrawal` time have no recourse during this freeze.

## Finding Description
`_unlockWithdrawalRequests` (lines 790–815) iterates from `nextLockedNonce[asset]` upward. At line 800, when `availableAssetAmount < payoutAmount`, the loop immediately `break`s and writes the unchanged `nextLockedNonce_` back to storage at line 815. There is no `continue` path, no skip mechanism, and no way for the operator-supplied `firstExcludedIndex` to advance past the blocking head entry.

The `availableAssetAmount` fed into this loop comes from `unstakingVault.balanceOf(asset)` via `_createUnlockParams` (lines 846–850) — the vault's current liquid balance. However, `initiateWithdrawal` (line 170) gates new requests against `getAvailableAssetAmount`, which at lines 599–603 computes `totalAssets - assetsCommitted[asset]` using `getTotalAssetDeposits` — the full protocol TVL including assets still delegated in EigenLayer. This structural mismatch means a request can be legitimately queued for an amount far exceeding what the vault holds at any given time.

When such a large request lands at the head of the queue, the loop breaks on its first iteration. `nextLockedNonce[asset]` remains at the blocking nonce. Any user whose nonce is greater than or equal to `nextLockedNonce[asset]` is rejected at line 707 (`if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked()`) when they call `completeWithdrawal`, even if their individual payout would be trivially satisfiable.

## Impact Explanation
Users queued behind a large blocking request have their rsETH locked in the contract with no ability to complete withdrawal. EigenLayer unstaking carries a multi-day delay and the vault fills incrementally, so the freeze persists until the vault accumulates enough to satisfy the head request. This constitutes **temporary freezing of funds** — a Medium impact under the allowed scope.

## Likelihood Explanation
Any unprivileged rsETH holder can call `initiateWithdrawal` with a large amount whenever total protocol TVL is high, because the admission check uses total deposits rather than vault liquidity. The vault routinely holds far less than total TVL since most assets remain staked in EigenLayer. This is a normal operating condition, not an edge case, and requires no special privileges or coordination.

## Recommendation
Replace the hard `break` at line 800 with a `continue` so the loop skips under-funded requests and processes smaller ones behind them. If strict ordering must be preserved for accounting reasons, introduce an operator-callable function that advances `nextLockedNonce` past a provably under-funded request without discarding it, or maintain a separate "pending" state for requests that cannot yet be satisfied.

## Proof of Concept
1. Protocol holds 2 000 ETH total deposits; `unstakingVault` holds 500 ETH liquid.
2. Alice calls `initiateWithdrawal(ETH, rsETH_for_1000_ETH)`. `getAvailableAssetAmount` returns `2000 - 0 = 2000 ≥ 1000` → passes. Nonce 0 assigned. `assetsCommitted[ETH] = 1000`.
3. Bob calls `initiateWithdrawal(ETH, rsETH_for_1_ETH)`. `getAvailableAssetAmount` returns `2000 - 1000 = 1000 ≥ 1` → passes. Nonce 1 assigned. `assetsCommitted[ETH] = 1001`.
4. Operator calls `unlockQueue(ETH, 2, ...)`. `_createUnlockParams` sets `totalAvailableAssets = 500` (vault balance).
5. Loop, nonce 0: `payoutAmount ≈ 1000 ETH`, `500 < 1000` → `break` at line 800. `nextLockedNonce[ETH]` stays at 0.
6. Bob calls `completeWithdrawal(ETH)`. His nonce is 1. Check at line 707: `1 >= 0` → `revert WithdrawalLocked()`. Bob's 1 ETH is frozen until the vault accumulates ≥ 1 000 ETH to unblock Alice's request. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
