Audit Report

## Title
Unbounded Gas Consumption in `updateRSETHPrice()` Due to Triple-Nested Iteration Over Assets, NDCs, and EigenLayer Queued Withdrawals - (File: contracts/LRTOracle.sol)

## Summary
`updateRSETHPrice()` is a public, permissionless function that triggers a triple-nested unbounded iteration: over all supported assets, over all NDCs per asset, and over all EigenLayer queued withdrawals per NDC. As queued withdrawals accumulate through normal protocol usage, the gas cost of this function grows without bound and can exceed the block gas limit, permanently freezing the rsETH price oracle at its last stored value.

## Finding Description
The call chain is confirmed in the codebase:

**Step 1:** `updateRSETHPrice()` (public, no access control) calls `_updateRsETHPrice()`, which calls `_getTotalEthInProtocol()`. [1](#0-0) [2](#0-1) 

**Step 2:** `_getTotalEthInProtocol()` iterates over all supported assets and calls `getTotalAssetDeposits()` for each, which internally calls `getAssetDistributionData()`. [3](#0-2) 

**Step 3:** `getAssetDistributionData()` iterates over all NDCs (`nodeDelegatorQueue.length`) and calls `getAssetUnstaking()` on each NDC for every asset. [4](#0-3) 

The same NDC loop also runs for the ETH asset path via `getETHDistributionData()`. [5](#0-4) 

**Step 4:** `getAssetUnstaking()` in `NodeDelegator.sol` calls `getQueuedWithdrawals()` on EigenLayer's `DelegationManager` and iterates over every queued withdrawal and every strategy within each withdrawal. [6](#0-5) 

The total gas cost scales as **O(assets × NDCs × queued_withdrawals_per_NDC)**. The number of EigenLayer queued withdrawals per NDC is entirely unbounded — it grows organically as users initiate unstaking and is never pruned until withdrawals are completed. There is no guard, cap, or pagination on the `getQueuedWithdrawals()` iteration. `maxNodeDelegatorLimit` is initialized to 10 but is admin-adjustable upward, and even at 10 NDCs the queued withdrawal dimension alone is sufficient to cause OOG. [7](#0-6) 

## Impact Explanation
**Medium — Unbounded gas consumption.** When `updateRSETHPrice()` reverts with OOG, `rsETHPrice` is frozen at its last stored value. All downstream consumers of `rsETHPrice` — deposit minting via `getRsETHAmountToMint()`, withdrawal payout calculations in `LRTWithdrawalManager`, and the downside protection pause logic in `_updateRsETHPrice()` — operate on a stale price. Protocol fee minting also halts. This matches the allowed impact "Medium. Unbounded gas consumption" exactly.

## Likelihood Explanation
No privileged action is required. Any unprivileged address can call `updateRSETHPrice()`. The queued withdrawal count grows through normal user behavior: every user who initiates an unstake from EigenLayer adds to the queue of the relevant NDC. During periods of elevated withdrawal demand (e.g., market stress), dozens to hundreds of withdrawals can accumulate per NDC. With N=3 assets, M=10 NDCs, and K=50 queued withdrawals each, the function executes ~1,500 external calls to EigenLayer storage — sufficient to approach or exceed Ethereum's ~30M block gas limit. The condition is self-reinforcing: once OOG is reached, no one can unfreeze the price without completing enough withdrawals to reduce K, which itself requires protocol functionality that depends on a live price.

## Recommendation
1. **Restructure `getAssetUnstaking()`** to use an incremental accounting pattern: track the unstaking amount in a storage variable updated on `queueWithdrawal` and `completeWithdrawal` events rather than re-reading all queued withdrawals on every call.
2. **Cache `totalAssetDeposits` per asset** and update it incrementally on deposit/withdrawal events rather than recomputing from scratch on every price update.
3. **Enforce a hard, non-adjustable cap on `maxNodeDelegatorLimit`** and document the maximum safe number of NDCs and supported assets relative to the block gas limit.
4. Consider a two-phase price update pattern: accumulate TVL data in a preparatory call (or off-chain), then finalize the price update in a separate transaction to avoid unbounded on-chain computation.

## Proof of Concept
**Minimal Foundry fork test plan:**
1. Fork mainnet with an active rsETH deployment.
2. Deploy or configure 3 supported assets and 10 NDCs.
3. For each NDC, simulate 50 users each calling `queueWithdrawal()` on EigenLayer's `DelegationManager` (standard unstaking flow, no privileges required).
4. Call `LRTOracle.updateRSETHPrice()` and measure gas via `vm.expectRevert()` or `gasleft()` instrumentation.
5. Observe OOG revert as gas consumption exceeds 30M.

**Alternatively, a differential/model test:** instrument `getAssetUnstaking()` with a mock `DelegationManager` that returns K withdrawals, sweep K from 1 to 200 across 3 assets × 10 NDCs, and plot gas vs. K to demonstrate the linear growth toward the block gas limit.

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
