### Title
Attacker-Controlled `hidden_puzzle_hash` Injected into Victim's `RCATInfo` via On-Chain RCAT Spend — (`chia/wallet/wallet_state_manager.py`, `chia/wallet/cat_wallet/r_cat_wallet.py`)

---

### Summary

`handle_cat` in `WalletStateManager` extracts `hidden_puzzle_hash` directly from the **parent coin's on-chain puzzle reveal** and passes it, without any ownership verification, to `RCATWallet.convert_to_revocable`. An attacker who holds any RCAT coin for TAIL X can spend it, creating a new RCAT coin hinted to the victim's puzzle hash but embedding an attacker-chosen `hidden_puzzle_hash` in the revocation layer. The victim's wallet processes this coin state, converts its `CATWallet` to an `RCATWallet`, and permanently stores the attacker's `hidden_puzzle_hash` in `RCATInfo`. All future RCAT puzzle hashes generated by the victim's wallet then embed the attacker's revocation key.

---

### Finding Description

**Entry point — `handle_cat`:**

At line 1263, the wallet uncurries the **parent** coin's puzzle reveal and extracts the revocation layer's `hidden_puzzle_hash`:

```python
revocation_layer_match = match_revocation_layer(uncurry_puzzle(uncurried_puzzle_reveal.args.at("rrf")))
``` [1](#0-0) 

`match_revocation_layer` simply reads the first curried argument of the revocation layer mod — whatever the attacker baked into their RCAT coin's puzzle:

```python

This value is then passed directly to `convert_to_revocable`:

``` [3](#0-2) 

**`convert_to_revocable` — no ownership check:**

The only guard is whether the lineage store is empty. There is no check that the wallet owner controls the `hidden_puzzle_hash`:

```python

After this, `puzzle_for_pk` and `puzzle_hash_for_pk` permanently embed the attacker's `hidden_puzzle_hash` into every new RCAT puzzle the victim's wallet generates:

```python
def puzzle_for_pk(self, pubkey: G1Element) -> Program:
    inner_puzzle = create_revocation_layer(
        self.cat_info.hidden_puzzle_hash, self.standard_wallet.puzzle_hash_for_pk(pubkey)
    )
``` [5](#0-4) 

**The hint guard does not protect against this:**

The only pre-conversion check is that the hint on the new coin matches a derivation record in the victim's puzzle store (line 1248). The attacker satisfies this trivially by setting the hint to any of the victim's known puzzle hashes (observable on-chain). This check confirms the coin is *addressed to* the victim, but says nothing about who controls the revocation key. [6](#0-5)

### Impact Explanation

After the conversion, the victim's wallet generates all future RCAT receive addresses as:

```
CAT(TAIL_X, REVOCATION_LAYER(attacker_hidden_puzzle_hash, victim_p2_hash))
```

The attacker, who knows the preimage of `attacker_hidden_puzzle_hash`, can spend any of these coins via the hidden puzzle path at any time — stealing or destroying the victim's RCAT holdings. This is an unauthorized spend of CAT assets, matching the Critical/High scope.

---

### Likelihood Explanation

Preconditions are realistic:
1. Victim has a `CATWallet` for TAIL X with an empty lineage store (common for newly created wallets or wallets that haven't yet received coins).
2. Attacker holds any RCAT coin for TAIL X — obtainable by participating in the token economy.
3. Attacker knows one of the victim's puzzle hashes — public on-chain information.

The attack requires a single confirmed spend bundle. No leaked keys, no admin access, no broken cryptography.

---

### Recommendation

Before calling `convert_to_revocable`, verify that the wallet owns the `hidden_puzzle_hash` — i.e., that it corresponds to a puzzle the wallet can construct or a key it controls. One approach: only accept the conversion if the `hidden_puzzle_hash` matches a value derived from the wallet's own key material (e.g., a well-known revocation puzzle hash stored at wallet creation time). Alternatively, refuse conversion entirely if the wallet did not initiate the RCAT relationship (e.g., require an explicit user confirmation or a pre-registered `hidden_puzzle_hash`).

### Proof of Concept

1. Victim creates a `CATWallet` for TAIL X (empty lineage store).
2. Attacker creates an RCAT coin with TAIL X, using `hidden_puzzle_hash = H(attacker_revocation_puzzle)`.
3. Attacker spends their RCAT coin, producing a new RCAT coin with:
   - Inner puzzle hash = victim's known p2 puzzle hash (hint set to same value)
   - Revocation layer = `REVOCATION_LAYER(attacker_hidden_puzzle_hash, victim_p2_hash)`
4. Victim's wallet receives the coin state (hinted to victim's puzzle hash → passes derivation record check).
5. `handle_cat` calls `match_revocation_layer` on the parent spend's puzzle reveal → returns `(attacker_hidden_puzzle_hash, ...)`.
6. `convert_to_revocable(found_cat_wallet, hidden_puzzle_hash=attacker_hidden_puzzle_hash)` is called.
7. `lineage_store.is_empty()` → `True` → conversion proceeds.
8. `RCATInfo(limitations_program_hash, None, attacker_hidden_puzzle_hash)` is written to DB.
9. Assert: `victim_wallet.cat_info.hidden_puzzle_hash == attacker_hidden_puzzle_hash` ✓
10. Attacker spends any future RCAT coin sent to the victim using the hidden puzzle path.
