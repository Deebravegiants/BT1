Audit Report

## Title
Integer Division Truncation in `viewSwapRsETHAmountAndFee` Silently Accepts Deposits While Minting Zero rsETH - (File: `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`)

## Summary
All four L2 deposit pool variants compute rsETH output via integer division (`amountAfterFee * 1e18 / rsETHToETHrate`) without guarding against a zero result. When a sufficiently small ETH or token amount is deposited, the division truncates to zero, yet the deposit function accepts the user's funds and transfers/mints zero rsETH/wrsETH in return. The depositor's ETH is permanently absorbed into the pool with no recovery path.

## Finding Description
In every L2 pool, `viewSwapRsETHAmountAndFee` computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because `rsETHToETHrate` is always strictly greater than `1e18` in normal operation (rsETH is yield-bearing), any `amountAfterFee` satisfying `amountAfterFee * 1e18 < rsETHToETHrate` will truncate to zero. For example, with `rsETHToETHrate = 1.05e18`, depositing 1 wei gives `1 * 1e18 / 1.05e18 = 0`.

The deposit functions in all four contracts only guard against `amount == 0`:

```solidity
if (amount == 0) revert InvalidAmount();
```

There is no subsequent check for `rsETHAmount == 0`. Execution continues:
- `RSETHPoolNoWrapper.sol`: `rsETH.safeTransfer(msg.sender, 0)` — transfers zero rsETH; user's ETH is retained.
- `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`: `wrsETH.mint(msg.sender, 0)` — mints zero wrsETH; user's ETH is retained.

The retained ETH is included in the pool's aggregate balance and bridged to L1 as undifferentiated pool ETH, with no per-user accounting or recovery mechanism.

By contrast, `LRTDepositPool._beforeDeposit` enforces both a `minAmountToDeposit` floor and a caller-supplied `minRSETHAmountExpected` slippage guard before accepting funds. None of the L2 pool contracts implement either protection.

## Impact Explanation
A depositor who sends a dust amount (e.g., 1 wei of ETH) receives 0 rsETH or wrsETH while their ETH is permanently absorbed into the pool. The protocol does not lose value — the ETH remains in the system — but the depositor loses their entire deposit with no recourse. This matches the **Low** allowed impact: *Contract fails to deliver promised returns, but doesn't lose value.*

## Likelihood Explanation
Any unprivileged external caller can trigger this by calling `deposit()` with a non-zero but sufficiently small `msg.value`. No special role, governance action, or front-running is required. The condition is deterministic and reproducible at any time the rsETH rate exceeds `1e18`, which is always true in normal operation. Likelihood is **Low** because the amounts involved are dust-level and accidental triggering is rare, but deliberate triggering is trivially easy.

## Recommendation
Add a zero-output guard in every `deposit` function (or inside `viewSwapRsETHAmountAndFee`) across all pool variants:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, enforce a minimum deposit floor analogous to `LRTDepositPool`'s `minAmountToDeposit`, or require callers to supply a `minRsETHAmountExpected` slippage parameter checked before accepting funds.

## Proof of Concept
Applies to `RSETHPoolNoWrapper` (and identically to all other pool variants):

1. Deploy or use an existing `RSETHPoolNoWrapper` instance where `rsETHToETHrate > 1e18` (normal operating condition).
2. Call `deposit{value: 1}("")` — sending exactly 1 wei of ETH.
3. Inside `viewSwapRsETHAmountAndFee`:
   - `fee = 1 * feeBps / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / rsETHToETHrate = 0` (truncates)
4. `rsETH.safeTransfer(msg.sender, 0)` executes without revert.
5. Caller's 1 wei is now held by the pool; caller holds 0 additional rsETH.
6. The 1 wei is included in the next `bridgeAssets` call and sent to L1 as undifferentiated pool ETH — permanently unrecoverable by the depositor.

**Foundry fuzz test plan:**
```solidity
function testFuzz_dustDepositMintsZero(uint256 amount) public {
    // rsETHToETHrate > 1e18 always in production
    vm.assume(amount > 0 && amount * 1e18 < rsETHToETHrate);
    uint256 balanceBefore = rsETH.balanceOf(user);
    vm.prank(user);
    pool.deposit{value: amount}("");
    assertEq(rsETH.balanceOf(user), balanceBefore); // user received nothing
    assertGt(address(pool).balance, 0);             // pool retained ETH
}
