### Title
Stale `rsETHPrice` at `initiateWithdrawal` permanently locks users into a below-fair-value payout — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal` computes `expectedAssetAmount` using the stored `lrtOracle.rsETHPrice()` without triggering a price refresh. Because `_calculatePayoutAmount` (called at `unlockQueue` time) returns `min(expectedAssetAmount, currentReturn)`, a stale (lower-than-true) price at initiation permanently caps the user's payout below the fair value of their rsETH, even after the oracle is later updated with the correct price.

---

### Finding Description

**Entry point — `initiateWithdrawal`:**

`getExpectedAssetAmount` reads the cached `rsETHPrice` directly:

```solidity
// LRTWithdrawalManager.sol:593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [1](#0-0) 

This value is stored as `request.expectedAssetAmount` and added to `assetsCommitted[asset]`:

```solidity
// LRTWithdrawalManager.sol:168-173
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
...
assetsCommitted[asset] += expectedAssetAmount;
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
``` [2](#0-1) 

**Payout cap at `unlockQueue` — `_calculatePayoutAmount`:**

When the operator later calls `unlockQueue` with fresh prices, the payout is:

```solidity
// LRTWithdrawalManager.sol:833-834
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
``` [3](#0-2) 

If `rsETHPrice` was stale (lower) at initiation:
- `expectedAssetAmount` = `rsETHUnstaked × stalePrice / assetPrice` (lower)
- `currentReturn` = `rsETHUnstaked × truePrice / assetPrice` (higher)
- `min(lower, higher)` = `lower` → user is permanently capped at the stale amount

`request.expectedAssetAmount` is then overwritten with this lower value and transferred at `completeWithdrawal`:

```solidity
// LRTWithdrawalManager.sol:804, 734
request.expectedAssetAmount = payoutAmount;   // set to stale-capped amount
...
_transferAsset(asset, user, request.expectedAssetAmount);
``` [4](#0-3) [5](#0-4) 

**`updateRSETHPrice()` is public but never called automatically:**

```solidity
// LRTOracle.sol:87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [6](#0-5) 

`initiateWithdrawal` does not call it, and there is no staleness guard anywhere in the withdrawal initiation path.

---

### Impact Explanation

A user who calls `initiateWithdrawal` while `rsETHPrice` has not been refreshed (e.g., after several days of yield accrual with no oracle update) will have their payout permanently capped at the stale lower amount. The rsETH they transfer in is burned at `unlockQueue` time, and they receive fewer underlying assets than the rsETH is currently worth. No principal is lost in the sense that the protocol remains solvent, but the user fails to receive the returns their rsETH has accrued — matching the **Low: Contract fails to deliver promised returns, but doesn't lose value** scope.

---

### Likelihood Explanation

`updateRSETHPrice()` is permissionless and public, so any user can call it before initiating a withdrawal. However, the contract provides no on-chain guarantee or automatic trigger that the price is fresh at initiation time. In practice, if the off-chain keeper misses updates for even a short period during which EigenLayer strategies accrue yield, any user who initiates a withdrawal during that window is silently harmed. The user has no indication from the contract that the price is stale, and the loss is irreversible once the request is queued.

---

### Recommendation

Call `updateRSETHPrice()` (or an equivalent internal price refresh) at the start of `initiateWithdrawal` before computing `expectedAssetAmount`, or add a staleness check that reverts if `rsETHPrice` has not been updated within an acceptable window (e.g., 24 hours). This ensures the committed amount always reflects the current fair value of rsETH.

---

### Proof of Concept

```solidity
// Local unit test (no mainnet required)
function test_stalePrice_locksLowerPayout() public {
    // 1. Deploy contracts with mock oracle returning stalePrice = 1.00e18
    uint256 stalePrice  = 1.00e18;
    uint256 truePrice   = 1.05e18; // 5% yield accrued, not yet reflected
    uint256 assetPrice  = 1.00e18; // asset == ETH, 1:1

    mockOracle.setRsETHPrice(stalePrice);   // oracle NOT updated

    // 2. User initiates withdrawal of 1 rsETH
    uint256 rsETHAmount = 1e18;
    vm.prank(user);
    withdrawalManager.initiateWithdrawal(asset, rsETHAmount, "");

    // expectedAssetAmount stored = 1e18 * 1.00e18 / 1.00e18 = 1.00e18

    // 3. Oracle is now updated to true price
    mockOracle.setRsETHPrice(truePrice);

    // 4. Operator unlocks queue with fresh prices
    withdrawalManager.unlockQueue(asset, type(uint256).max, 0, 0, type(uint256).max, type(uint256).max);

    // _calculatePayoutAmount: currentReturn = 1e18 * 1.05e18 / 1.00e18 = 1.05e18
    // min(1.00e18, 1.05e18) = 1.00e18  ← stale amount wins

    // 5. User completes withdrawal
    vm.prank(user);
    withdrawalManager.completeWithdrawal(asset, "");

    // Assert: user received 1.00e18, not the fair 1.05e18
    assertEq(asset.balanceOf(user), 1.00e18);
    // Fair value shortfall = 0.05e18 per rsETH withdrawn
}
```
