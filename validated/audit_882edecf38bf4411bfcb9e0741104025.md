### Title
Wrong Transaction Hash in DeclareV1 Simulation/Fee Estimation Due to `ClassHash::default()` — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

### Summary

When converting a `BroadcastedDeclareV1Transaction` for use in `starknet_estimateFee` or `starknet_simulateTransactions`, the `class_hash` field of the resulting `DeclareTransactionV0V1` is unconditionally set to `ClassHash::default()` (i.e., `Felt::ZERO`). Because `get_declare_transaction_v1_hash` directly chains `transaction.class_hash.0` into the Pedersen hash, the blockifier computes and presents a transaction hash of `H(..., 0)` instead of `H(..., real_class_hash)` to the `__validate__` syscall and to the simulation trace returned to the caller.

---

### Finding Description

In the `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput` implementation: [1](#0-0) 

The `class_hash` field is hardcoded to `ClassHash::default()` with the comment "The blockifier doesn't need the class hash." However, `get_declare_transaction_v1_hash` — the function that computes the transaction hash for a V1 declare — directly uses `transaction.class_hash.0`: [2](#0-1) 

Specifically, line 556 chains the class hash into the Pedersen hash:
```
.chain(&HashChain::new().chain(&transaction.class_hash.0).get_pedersen_hash())
```

With `class_hash = Felt::ZERO`, the computed hash is `H(declare, version, sender, 0, H(0), max_fee, chain_id, nonce)` instead of `H(declare, version, sender, 0, H(real_class_hash), max_fee, chain_id, nonce)`.

---

### Impact Explanation

Any unprivileged RPC client calling `starknet_estimateFee` or `starknet_simulateTransactions` with a `BroadcastedDeclareV1Transaction` receives:

1. **Wrong transaction hash in the simulation trace** — the `TransactionTrace` returned by `starknet_simulateTransactions` reflects a hash computed with `class_hash=0`, not the hash the client actually signed or intends to submit.
2. **Wrong `transaction_hash` syscall value in `__validate__`** — if the account's `__validate__` entrypoint reads `get_tx_info().transaction_hash`, it sees the zero-class-hash variant. An account that validates the transaction hash (e.g., for replay protection or custom logic) will behave differently during simulation than during actual execution, making the simulation result misleading.
3. **Potentially wrong fee estimate** — if `__validate__` reverts or branches differently due to the wrong hash, the fee estimate is incorrect for the real transaction.

This falls squarely under: *"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

- The path is unconditional: every `BroadcastedDeclareV1Transaction` submitted to these two RPC endpoints hits this code.
- No privilege is required; any RPC client can trigger it.
- The only mitigation would be if the blockifier recomputes the class hash from the contract class before hashing — but the comment explicitly says it does not need the class hash, and the struct field is what gets hashed.

---

### Recommendation

Compute the real class hash from the contract class before constructing the `DeclareTransactionV0V1`. The deprecated contract class hash is the Starknet Pedersen hash of the contract definition. Replace:

```rust
class_hash: ClassHash::default(),
```

with:

```rust
class_hash: compute_deprecated_class_hash(&sn_api_contract_class)
    .map_err(internal_server_error)?,
```

using the appropriate hash function from `starknet_api` or `blockifier`. This ensures the transaction hash computed during simulation matches the hash the client would sign and submit.

---

### Proof of Concept

```rust
// Pseudocode unit test
let contract_class = /* known deprecated contract class */;
let real_class_hash = compute_deprecated_class_hash(&contract_class);

let tx = BroadcastedDeclareV1Transaction {
    contract_class: compress(contract_class),
    sender_address: ACCOUNT,
    nonce: Nonce(0),
    max_fee: Fee(1_000_000),
    signature: vec![],
    r#type: _,
};

let result = exec_simulate_transactions(vec![tx], ...);
let simulated_hash = result[0].transaction_trace.validate_invocation.transaction_hash;

let expected_hash = get_declare_transaction_v1_hash(
    &DeclareTransactionV0V1 { class_hash: real_class_hash, ... },
    &chain_id,
    &version,
);

// This assertion FAILS — simulated_hash uses class_hash=0
assert_eq!(simulated_hash, expected_hash);

pub(crate) fn get_declare_transaction_v1_hash(
    transaction: &DeclareTransactionV0V1,
    chain_id: &ChainId,
    transaction_version: &TransactionVersion,
) -> Result<TransactionHash, StarknetApiError> {
    Ok(TransactionHash(
        HashChain::new()
        .chain(&DECLARE)
        .chain(&transaction_version.0)
        .chain(transaction.sender_address.0.key())
        .chain(&Felt::ZERO) // No entry point selector in declare transaction.
        .chain(&HashChain::new().chain(&transaction.class_hash.0).get_pedersen_hash())
        .chain(&transaction.max_fee.0.into())
        .chain(&Felt::try_from(chain_id)?)
        .chain(&transaction.nonce.0)
        .get_pedersen_hash(),
    ))
}
