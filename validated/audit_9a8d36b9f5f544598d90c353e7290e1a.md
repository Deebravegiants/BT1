### Title
`vote_add_domains` Enforces Unanimity Instead of Threshold, Allowing a Single Byzantine Participant to Permanently Block Key Generation - (File: crates/contract/src/state/running.rs)

### Summary

The `vote_add_domains` function in `RunningContractState` requires **all** participants to vote for the same domain proposal before transitioning to the `Initializing` state. The on-chain ABI documentation and docstrings explicitly state "threshold number of votes," but the implementation compares the vote count to the total participant count. A single Byzantine participant strictly below the signing threshold can permanently prevent domain addition, leaving the MPC network without signing keys and unable to fulfill any signature request.

### Finding Description

In `crates/contract/src/state/running.rs`, the transition guard for `vote_add_domains` is:

```rust
let n_votes = self.add_domains_votes.vote(domains.clone(), &participant);
if self.parameters.participants().len() as u64 == n_votes {   // line 237
    // transition to Initializing
}
``` [1](#0-0) 

This requires every participant to cast an identical vote. The on-chain ABI docstring for the same function reads:

> "If a **threshold** number of votes are reached on the exact same proposal, this will transition the contract into the Initializing state to generate keys for the new domains." [2](#0-1) 

The discrepancy is structural: `vote_new_parameters` uses `ThresholdParametersVotes`, which counts votes against the governance threshold, while `vote_add_domains` uses `AddDomainsVotes.vote()` whose return value is then compared to `participants().len()` — unanimity, not threshold. [3](#0-2) 

The `AddDomainsVotes.vote()` method returns the count of participants who have voted for the exact same proposal. Comparing that count to `participants().len()` means the transition fires only when every registered participant has voted identically — a unanimity requirement that is nowhere documented as intentional and is inconsistent with every other governance vote in the contract.

The contract is initialized with an empty `DomainRegistry` and `accept_requests: true`: [4](#0-3) 

Domains must be added via `vote_add_domains` before any signing key exists. If the unanimity bar is never cleared, the contract remains in `Running` state with zero domains indefinitely, and every `sign`, `request_app_private_key`, or `verify_foreign_transaction` call panics with `DomainNotFound` at: [5](#0-4) 

### Impact Explanation

Without domains the MPC network has no key material. All user-facing request methods fail immediately. Yield promises created by `enqueue_yield_request` are never resolved, permanently locking any attached deposits or cross-chain operations that depend on the signature. This breaks the request lifecycle and accounting invariants of the chain-signature contract without requiring any privileged access.

Impact classification: **Medium** — contract execution-flow manipulation that breaks production safety/accounting invariants (request lifecycle permanently stalled) without relying on network-level DoS or operator misconfiguration.

### Likelihood Explanation

The governance threshold is enforced at ≥ 60 % of participants. Up to 40 % of participants can be Byzantine without violating the signing threshold. Any **single** such participant — strictly below the threshold — can block `vote_add_domains` indefinitely by simply not submitting the transaction. No collusion, no privileged role, no key material is required. The attacker's entry point is the absence of a call, which is trivially achievable and undetectable on-chain until the network fails to become operational.

### Recommendation

Replace the unanimity check with a threshold check, consistent with `vote_new_parameters`:

```rust
// current (unanimity — incorrect)
if self.parameters.participants().len() as u64 == n_votes {

// corrected (threshold)
if self.parameters.threshold().value() <= n_votes {
```

Additionally, the `init` function should document (or enforce) that `vote_add_domains` requires only threshold votes, and the ABI docstring already reflects this correct intent. [6](#0-5) 

### Proof of Concept

1. Deploy the MPC contract with `n = 10` participants and governance threshold `k = 6` (60 %).
2. Nine participants call `vote_add_domains` with an identical `DomainConfig` proposal. `n_votes` reaches 9.
3. The check `10 == 9` is false; the contract does **not** transition to `Initializing`.
4. The tenth participant (Byzantine, well below the threshold of 6) never calls `vote_add_domains`.
5. The contract remains in `Running` state with `DomainRegistry::default()` (zero domains) indefinitely.
6. Any user calling `sign(...)` receives `DomainNotFound`; yield promises are never resolved; attached deposits are locked.
7. The only remediation path is a contract upgrade — there is no on-chain escape hatch. [7](#0-6) [8](#0-7)

    /// If the number of votes for the same set of new domains reaches the number of participants,
    /// returns the InitializingContractState we should transition into to generate keys for these
    /// new domains.
    pub fn vote_add_domains(
        &mut self,
        domains: Vec<DomainConfig>,
    ) -> Result<Option<InitializingContractState>, Error> {
        if domains.is_empty() {
            return Err(DomainError::AddDomainsMustAddAtLeastOneDomain.into());
        }
        let num_participants = u64::try_from(self.parameters.participants().len())
            .expect("participant count fits in u64");
        for domain in &domains {
            validate_domain_purpose(domain)?;
            validate_domain_threshold(domain, num_participants)?;
        }
        // Keep trust assumptions consistent: a domain must never require more shares to
        // reconstruct than the GovernanceThreshold demands to govern. Route through the
        // canonical helper so the cross-domain invariant has a single source of truth.
        ThresholdParameters::validate_governance_against_reconstruction(
            num_participants,
            self.parameters.threshold(),
            max_reconstruction_threshold(&domains),
        )?;
        let participant = AuthenticatedParticipantId::new(self.parameters.participants())?;
        let n_votes = self.add_domains_votes.vote(domains.clone(), &participant);
        if self.parameters.participants().len() as u64 == n_votes {
            let new_domains = self.domains.add_domains(domains.clone())?;
            Ok(Some(InitializingContractState {
                generated_keys: self.keyset.domains.clone(),
                domains: new_domains,
                epoch_id: self.keyset.epoch_id,
                generating_key: KeyEvent::new(
                    self.keyset.epoch_id,
                    domains[0].clone(),
                    self.parameters.clone(),
                ),
                cancel_votes: BTreeSet::new(),
            }))
        } else {
            Ok(None)
        }
    }

impl AddDomainsVotes {
    /// Votes for the proposal, returning the total number of voters so far who
    /// have proposed the exact same domains to add.
    /// If the participant had voted already, this replaces the existing vote.
    pub fn vote(
        &mut self,
        proposal: Vec<DomainConfig>,
        participant: &AuthenticatedParticipantId,
    ) -> u64 {
        if self
            .proposal_by_account
            .insert(participant.clone(), proposal.clone())
            .is_some()
        {
            log!("removed old vote for signer");
        }
        let total = self
            .proposal_by_account
            .values()
            .filter(|&prop| prop == &proposal)
            .count() as u64;
        log!("total votes for proposal: {}", total);
        total
    }
