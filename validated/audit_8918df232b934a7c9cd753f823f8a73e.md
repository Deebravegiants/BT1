### Title
ETH Withdrawals Temporarily Frozen When Aave Integration Is Enabled After `unlockQueue` - (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

When `isAaveIntegrationEnabled` is `true`, `unlockQueue` redeems ETH from the unstaking vault and immediately deposits it into Aave. The ETH is then held as `aWETH` in Aave, not as native ETH in the contract. When users subsequently call `completeWithdrawal`, the contract attempts to transfer native ETH that is no longer present, causing the transfer to fail until an admin manually withdraws from Aave.

---

### Finding Description

`unlockQueue` in `LRTWithdrawalManager` processes pending ETH withdrawal requests in three sequential steps:

1. Marks requests as unlocked and calculates `assetAmountUnlocked` via `_unlockWithdrawalRequests`.
2. Redeems ETH from the unstaking vault: `unstakingVault.redeem(asset, assetAmountUnlocked)` — ETH arrives in the withdrawal manager.
3. If `isAaveIntegrationEnabled` is `true`, deposits the just-redeemed ETH into Aave:

```solidity
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN && assetAmountUnlocked > 0) {
    try this.depositToAaveExternal(assetAmountUnlocked) { }
    catch (bytes memory reason) {
        emit AaveDepositFailed(assetAmountUnlocked, reason);
        // Silently fail if Aave deposit fails (e.g., pool at max capacity)
        // Funds remain in contract for withdrawals
    }
}
``` [1](#0-0) 

The inline comment is the critical tell: **"Funds remain in contract for withdrawals"** is stated only as the consequence of a *failed* Aave deposit. When the deposit *succeeds*, the ETH leaves the contract's native balance and is held as `aWETH`. At this point, users whose requests were just unlocked call `completeWithdrawal` → `_processWithdrawalCompletion`, which must transfer native ETH to the user. There is no automatic Aave withdrawal in the completion path; the only mechanisms to recover ETH from Aave are the admin-gated `collectInterestAndWithdrawFromAave` and the pauser-gated `emergencyWithdrawFromAave`: [2](#0-1) 

Because `_processWithdrawalCompletion` has no fallback to pull from Aave, every `completeWithdrawal` call for ETH will revert (insufficient native balance) until an admin manually drains Aave back into the contract.

This is the direct analog to the `GlpStrategy` bug: there, `_currentBalance()` included pending WETH rewards, but `_withdraw` only converted them to GLP when `shouldBuyGLP == true`; here, `unlockQueue` moves ETH out of the contract into Aave when `isAaveIntegrationEnabled == true`, but `completeWithdrawal` has no path to retrieve it.

---

### Impact Explanation

All users with unlocked ETH withdrawal requests are temporarily unable to complete their withdrawals. Their rsETH has already been burned (at `unlockQueue` time) and the ETH is inaccessible until an admin intervenes. This constitutes a **temporary freezing of funds** (Medium severity per the allowed impact scope).

---

### Likelihood Explanation

`isAaveIntegrationEnabled` is set to `true` by the LRT manager via `configureAaveIntegration` — a routine operational configuration, not an attack. Any time the Aave integration is active and `unlockQueue` is called for ETH, the entire unlocked batch is deposited to Aave, blocking all subsequent `completeWithdrawal` calls for that batch. Likelihood is **Medium**. [3](#0-2) 

---

### Recommendation

Before transferring ETH to the user in `_processWithdrawalCompletion`, check whether the contract holds sufficient native ETH. If not, and if Aave integration is enabled, automatically withdraw the shortfall from Aave before proceeding with the transfer. Alternatively, modify `unlockQueue` to deposit to Aave only the ETH that exceeds the total amount needed to satisfy all currently unlocked (but not yet completed) withdrawal requests, ensuring the contract always retains enough native ETH to service pending completions.

---

### Proof of Concept

1. Manager calls `configureAaveIntegration(...)` and sets `isAaveIntegrationEnabled = true`.
2. Multiple users call `initiateWithdrawal(ETH_TOKEN, rsETHAmount, ...)` — rsETH is escrowed, `assetsCommitted[ETH]` grows.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`:
   - ETH is redeemed from the unstaking vault into the withdrawal manager.
   - ETH is immediately deposited into Aave via `depositToAaveExternal`.
   - Withdrawal requests are marked unlocked; `unlockedWithdrawalsCount[ETH]` increments.
4. Users call `completeWithdrawal(ETH_TOKEN, ...)`:
   - `_processWithdrawalCompletion` attempts to transfer native ETH.
   - Contract native ETH balance is 0 (all in Aave).
   - Transfer reverts.
5. Users are frozen until an admin calls `collectInterestAndWithdrawFromAave` or `emergencyWithdrawFromAave` to return ETH to the contract. [4](#0-3) [5](#0-4)

    function unlockQueue(
        address asset,
        uint256 firstExcludedIndex,
        uint256 minimumAssetPrice,
        uint256 minimumRsEthPrice,
        uint256 maximumAssetPrice,
        uint256 maximumRsEthPrice
    )
        external
        nonReentrant
        onlySupportedAsset(asset)
        whenNotPaused
        onlyAssetTransferOrOperatorRole
        returns (uint256 rsETHBurned, uint256 assetAmountUnlocked)
    {
        ILRTOracle lrtOracle = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE));
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));

        UnlockParams memory params = _createUnlockParams(lrtOracle, unstakingVault, asset);

        _validatePrices(
            params.rsETHPrice,
            params.assetPrice,
            minimumRsEthPrice,
            maximumRsEthPrice,
            minimumAssetPrice,
            maximumAssetPrice
        );

        if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();

        // Updates and unlocks withdrawal requests up to a specified upper limit or until allocated assets are fully
        // utilized.
        (rsETHBurned, assetAmountUnlocked) = _unlockWithdrawalRequests(
            asset, params.totalAvailableAssets, params.rsETHPrice, params.assetPrice, firstExcludedIndex
        );

        if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
        //Take the amount to distribute from vault
        unstakingVault.redeem(asset, assetAmountUnlocked);

        // If Aave integration is enabled and asset is ETH, deposit to Aave
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN && assetAmountUnlocked > 0) {
            try this.depositToAaveExternal(assetAmountUnlocked) { }
            catch (bytes memory reason) {
                emit AaveDepositFailed(assetAmountUnlocked, reason);
                // Silently fail if Aave deposit fails (e.g., pool at max capacity)
                // Funds remain in contract for withdrawals
            }
        }

        emit AssetUnlocked(asset, rsETHBurned, assetAmountUnlocked, params.rsETHPrice, params.assetPrice);
    }

    /// @notice Emergency withdrawal from Aave in case of issues
    /// @param amount Amount to withdraw (use type(uint256).max for full withdrawal)
    /// @dev Only callable by PAUSER_ROLE
    /// @dev Collects accrued interest to treasury before withdrawing principal
    function emergencyWithdrawFromAave(uint256 amount) external nonReentrant onlyRole(LRTConstants.PAUSER_ROLE) {
        if (!isAaveIntegrationEnabled) revert AaveIntegrationNotEnabled();

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // First collect any accrued interest to treasury
        _collectInterestToTreasury();

        uint256 withdrawnAmount = _withdrawFromAave(amount);

        emit EmergencyWithdrawFromAave(withdrawnAmount, address(this));
    }
