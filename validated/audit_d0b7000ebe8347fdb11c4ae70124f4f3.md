### Title
`ec_pve_batch_t::convert` guard fires pre-deserialization, leaving `Q.size() < n` undetected, causing OOB read in `restore_from_decrypted` — (`include-internal/cbmpc/internal/protocol/pve_batch.h`, `src/cbmpc/protocol/pve_batch.cpp`)

### Summary
The `Q.size() != n` guard in `ec_pve_batch_t::convert` is evaluated before the vector deserializer overwrites `Q`. An attacker who supplies a crafted ciphertext blob with `batch_count=N` in the outer wrapper but fewer than `N` entries in the inner `Q` vector passes the guard, causes `coinbase::convert` to return `SUCCESS`, and then triggers an out-of-bounds read on `Q[i]` inside `restore_from_decrypted`.

### Finding Description

`ec_pve_batch_t::convert` is a single method used for both serialization and deserialization. Its invariant check:

```cpp
if (int(Q.size()) != n) { converter.set_error(); return; }
```

is evaluated at entry, when `Q.size() == n` because the constructor called `Q.resize(n)`. The subsequent call `converter.convert(Q, L, b)` then clears `Q` and resizes it to the count encoded in the blob. If the blob encodes 2 entries but `n == 3`, `Q.size()` becomes 2 after deserialization with no error set. `converter.get_rv()` returns `SUCCESS`. The caller in `decrypt_batch` checks the return value and finds no error, so it proceeds to call `pve_ct.decrypt(..., skip_verify=true)`. Inside `decrypt`, `restore_from_decrypted` loops `for (int i = 0; i < n; i++)` and accesses `Q[i]` — undefined behavior when `i >= Q.size()`.

### Impact Explanation

This is a public-API-reachable memory-safety invariant break. Any caller of `cbmpc_pve_batch_decrypt` / `decrypt_batch` / `decrypt_batch_rsa_oaep_hsm` / `decrypt_batch_ecies_p256_hsm` that supplies an attacker-controlled ciphertext blob can trigger the OOB read. The immediate effect is undefined behavior: most likely a crash (DoS) or silent read of adjacent heap memory. Because `Q[i]` is an `ecc_point_t` object, reading past the vector's storage could expose heap metadata or adjacent allocations to the point-comparison branch, creating a potential information-disclosure path.

### Likelihood Explanation

The attack requires only a crafted byte blob passed to a public decrypt API. The attacker needs the public key `ek` (which is not secret) to produce a valid base-PKE ciphertext that decrypts to `n * curve_size` bytes, satisfying the size guard before the OOB loop. No private key material or threshold collusion is required.

### Recommendation

Add a post-deserialization invariant check inside `ec_pve_batch_t::convert` when in read mode:

```cpp
void convert(coinbase::converter_t& converter) {
    if (int(Q.size()) != n) { converter.set_error(); return; }
    converter.convert(Q, L, b);
    // NEW: re-validate after deserialization overwrites Q
    if (!converter.is_write() && int(Q.size()) != n) {
        converter.set_error();
        return;
    }
    for (int i = 0; i < kappa; i++) { ... }
}
```

Alternatively, remove the pre-deserialization guard (it is vacuously true during reads) and rely solely on the post-deserialization check.
