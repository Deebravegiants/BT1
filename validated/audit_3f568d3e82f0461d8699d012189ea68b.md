### Title
Unenforced Harvester Fee-Threshold Allows Unconditional Farmer Reward Diversion - (`chia/farmer/farmer_api.py`)

### Summary
The farmer unconditionally accepts a `farmer_reward_address_override` supplied by any connected harvester in `RespondSignatures`, redirecting the farmer's block reward (XCH) to an arbitrary puzzle hash. The fee-quality threshold check introduced by CHIP-22 is only logged as a warning and is never enforced; the override is always applied regardless of whether the threshold is met.

### Finding Description

CHIP-22 (`farmer_reward_address_override`) allows a third-party harvester to redirect the farmer's block reward to itself as a fee, subject to a fee-quality convention. The farmer is supposed to enforce a threshold: only accept the override when `fee_quality <= applied_fee_threshold`.

In `FarmerAPI.new_proof_of_space()`, when a harvester sends `NewProofOfSpace` with `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [1](#0-0) 

That function only **logs** a warning when the threshold is violated — it never returns a rejection signal and never prevents the override from being used: [2](#0-1) 

Later, in `_process_respond_signatures()`, the farmer **unconditionally** replaces its own configured `farmer_target` with whatever `farmer_reward_address_override` the harvester placed in `RespondSignatures`: [3](#0-2) 

This overridden address is then embedded in the `DeclareProofOfSpace` message broadcast to the full node: [4](#0-3) 

The full node creates the farmer coinbase reward coin using exactly the `farmer_puzzle_hash` declared in that message: [5](#0-4) 

There is no on-chain or protocol-level check that the declared puzzle hash matches the farmer's configured `farmer_target`. The full node accepts whatever the farmer declares.

### Impact Explanation

A malicious harvester can set `farmer_reward_address_override` to any arbitrary puzzle hash in both `NewProofOfSpace` and `RespondSignatures`. The farmer will use that address as the farmer reward destination in every block it wins, permanently diverting XCH block rewards away from the farmer's configured address to the attacker's address. The farmer's only signal is a log warning that is not actionable in real time.

This matches: **Critical — reward diversion affecting XCH**.

### Likelihood Explanation

Any harvester connected to the farmer can exploit this. In the CHIP-22 third-party harvester model, the farmer deliberately connects to external harvesters. The harvester needs no special keys or privileges beyond the existing connection — it simply sets the override field in the protocol message. The farmer has no enforcement mechanism to reject an override that violates the configured threshold.

### Recommendation

In `_process_respond_signatures()`, before accepting `farmer_reward_address_override`, the farmer should recompute `fee_quality` for the proof and verify that `fee_quality <= applied_fee_threshold` from `fee_info`. If the threshold is not met (or `fee_info` is absent), the override must be **rejected** and the farmer's own `farmer_target` used instead. The current call to `notify_farmer_reward_taken_by_harvester_as_fee()` in `new_proof_of_space()` should be moved or duplicated into `_process_respond_signatures()` and made to return a boolean that gates the override.

### Proof of Concept

1. Farmer connects to a malicious third-party harvester.
2. Harvester finds a valid proof of space and sends `NewProofOfSpace` with `farmer_reward_address_override = attacker_puzzle_hash` and `fee_info.applied_fee_threshold = 0xFFFFFFFF` (claiming 100% fee rate).
3. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()` — this logs a warning if `fee_quality > threshold`, but does **not** block the flow.
4. Farmer sends `RequestSignatures` to the harvester.
5. Harvester responds with `RespondSignatures` containing `farmer_reward_address_override = attacker_puzzle_hash`.
6. `_process_respond_signatures()` sets `farmer_reward_address = attacker_puzzle_hash` unconditionally.
7. Farmer broadcasts `DeclareProofOfSpace` with `farmer_puzzle_hash = attacker_puzzle_hash` to the full node.
8. Full node creates the farmer coinbase coin at `attacker_puzzle_hash`; the farmer receives nothing. [6](#0-5) [7](#0-6) [8](#0-7)
