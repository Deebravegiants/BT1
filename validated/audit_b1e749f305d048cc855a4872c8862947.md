Audit Report

## Title
Missing Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Near-Total Loss of User Funds for Low-Decimal Token Deposits - (File: contracts/pools/RSETHPoolV3.sol)

## Summary

The `viewSwapRsETHAmountAndFee(uint256 amount, address token)` function computes rsETH output as `amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18 decimals. Because `tokenToETHRate` from `ChainlinkOracleForRSETHPoolCollateral.getRate()` is always 1e18-scaled (price of 1 token unit in ETH), the formula is only dimensionally correct for 18-decimal tokens. For a 6-decimal token like USDC, the formula under-mints by a factor of 10^12, causing depositors to receive near-zero wrsETH while their full token balance is transferred to the pool. The identical bug exists across all five pool contracts.

## Finding Description

**Root cause — `ChainlinkOracleForRSETHPoolCollateral.getRate()`** normalizes the Chainlink answer to 1e18 precision regardless of the token's own decimals:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [1](#0-0) 

This returns the ETH value of **1 token unit** (in the token's native denomination), scaled to 1e18. For stETH (18 decimals) this is ~1e18; for USDC (6 decimals) this is ~5e14.

**The broken formula** in `RSETHPoolV3.viewSwapRsETHAmountAndFee`:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

For stETH: `1e18 (wei) × 1e18 / 1.05e18 ≈ 9.52e17` ✓  
For USDC: `1e6 (wei USDC) × 5e14 / 1.05e18 ≈ 476 wei wrsETH` — correct is `4.76e14 wei wrsETH` — **under-mint by 10^12**.

The same formula appears identically in all five pool contracts: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

**No decimal check exists** anywhere in the pool contracts — `grep` for `decimals` in `contracts/pools/*.sol` returns zero matches outside the oracle wrapper. 

**No `rsETHAmount > 0` guard** in `deposit`: the only input guard is `if (amount == 0) revert InvalidAmount()`, applied to the deposited token amount, not to the computed rsETH output. A dust USDC deposit (< 2100 wei) produces `rsETHAmount = 0` and mints nothing while transferring tokens to the pool. [7](#0-6) 

The inverse formula in `viewSwapAssetToPremintedRsETH` (`tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate`) is broken in the opposite direction for low-decimal tokens, over-paying by the same factor. [8](#0-7) 

## Impact Explanation

**Critical — Direct theft of user funds.**

Every depositor who calls `deposit(token, amount, referralId)` with a low-decimal token transfers their full token balance to the pool but receives `10^(18 − tokenDecimals)` times less wrsETH than the economically correct amount. For USDC (6 decimals) this is a ~99.9999% loss per deposit; for WBTC (8 decimals) ~99.9999999%. Deposited tokens accumulate in the pool and are not recoverable by the user. For dust deposits, the loss is 100% with no revert. This matches the allowed impact: **Critical — Direct theft of any user funds**.

## Likelihood Explanation

**Medium.** The precondition is that an admin calls `addSupportedToken` with a low-decimal token. This is a normal operational action — the function has no decimal restriction and the protocol is explicitly designed to be extensible to new collateral types. No malicious intent is required; a well-intentioned admin adding USDC or WBTC as L2 pool collateral triggers the bug for all subsequent depositors. Once the precondition is met, every unprivileged depositor is affected automatically and repeatably.

## Recommendation

1. **Normalize `amountAfterFee` to 18 decimals** before applying the formula in all five pool contracts:
   ```solidity
   uint8 tokenDecimals = IERC20Metadata(token).decimals();
   uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
   rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
   ```
2. **Enforce 18-decimal tokens** in `addSupportedToken` / `setSupportedTokenOracle` by reverting if `IERC20Metadata(token).decimals() != 18`.
3. **Add a `rsETHAmount > 0` guard** in `deposit` to prevent silent total-loss deposits.
4. Apply the symmetric fix to `viewSwapAssetToPremintedRsETH` to avoid over-payment in the reverse direction.

## Proof of Concept

**Setup**: Admin calls `addSupportedToken(USDC, chainlinkUSDCETHOracle)` on `RSETHPoolV3`. USDC has 6 decimals; oracle returns `tokenToETHRate = 5e14`. `rsETHToETHrate = 1.05e18`.

**Exploit sequence** (unprivileged depositor):

1. User approves 2000 USDC (`2000e6` wei) to `RSETHPoolV3`.
2. User calls `deposit(USDC, 2000e6, "")`.
3. `viewSwapRsETHAmountAndFee` computes (fee = 0 for simplicity):
   - `rsETHAmount = 2000e6 * 5e14 / 1.05e18 = 1e21 / 1.05e18 ≈ 952 wei wrsETH`
4. User receives 952 wei wrsETH ≈ 9.52e-16 wrsETH.
5. Correct amount: 2000 USDC × (5e14 / 1e18) ETH/USDC / (1.05e18 / 1e18) ETH/rsETH = ~9.52e17 wei wrsETH.
6. **Loss: ~99.9999999% of deposit value.** 2000 USDC remain locked in the pool.

**Foundry fork test plan**:
```solidity
function testUSDCDepositLoss() public {
    // Fork mainnet/L2, deploy RSETHPoolV3 with USDC oracle
    vm.prank(admin);
    pool.addSupportedToken(USDC, usdcEthOracle);
    
    deal(USDC, user, 2000e6);
    vm.startPrank(user);
    IERC20(USDC).approve(address(pool), 2000e6);
    pool.deposit(USDC, 2000e6, "");
    vm.stopPrank();
    
    uint256 received = wrsETH.balanceOf(user);
    // received ≈ 952 wei; correct ≈ 9.52e17 wei
    assertLt(received, 1000); // dust
    assertGt(IERC20(USDC).balanceOf(address(pool)), 1999e6); // tokens locked
}
```
