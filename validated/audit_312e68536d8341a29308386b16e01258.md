Audit Report

## Title
Missing Chainlink Staleness Check Enables Phantom Fee Minting — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt` and `answeredInRound`, accepting any price regardless of age. Because `updateRSETHPrice()` is permissionless, an attacker can call it while the oracle is stale-high — causing `_getTotalEthInProtocol()` to overstate TVL, triggering fee minting on phantom yield, and diluting all rsETH holders.

## Finding Description
**Root cause — `contracts/oracles/ChainlinkPriceOracle.sol` line 52:**
```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```
All five return values of `latestRoundData()` — `(roundId, answer, startedAt, updatedAt, answeredInRound)` — are available, but only `answer` is used. `updatedAt` and `answeredInRound` are discarded with no staleness or round-completeness check.

**Exploit path:**
1. Oracle price for a supported asset is at P2 (after a legitimate price increase). `updateRSETHPrice()` has not yet been called to capture it.
2. Real price drops back toward P1. Chainlink's deviation threshold is not crossed (or the attacker stuffs blocks to prevent the keeper update), so the oracle remains stale at P2.
3. Attacker calls the permissionless `updateRSETHPrice()` (`contracts/LRTOracle.sol` line 87).
4. `_getTotalEthInProtocol()` multiplies each asset's deposit balance by the stale P2 price, inflating `totalETHInProtocol`.
5. `previousTVL = rsethSupply.mulWad(rsETHPrice)` uses the stored price (based on P1), so `totalETHInProtocol > previousTVL`.
6. `protocolFeeInETH = rewardAmount * protocolFeeInBPS / 10_000` is computed on the phantom delta.
7. `rsethAmountToMintAsProtocolFee` is minted to treasury, increasing total supply without any real ETH backing.

**Why existing guards fail:**
- `pricePercentageLimit`: checked against `newRsETHPrice > highestRsethPrice` (line 252–257). If `pricePercentageLimit == 0` the check is entirely skipped. If the stale inflation is within the configured limit (e.g., 0.3% when limit is 1%), the transaction proceeds and fee is minted.
- `maxFeeMintAmountPerDay`: caps total daily minting but does not prevent phantom fee minting up to that cap. An attacker can drain the full daily allowance with stale data.

## Impact Explanation
Excess rsETH is minted to the treasury on phantom TVL growth. Total rsETH supply increases without a corresponding increase in underlying ETH, reducing NAV per rsETH for all existing holders. This is a direct, concrete theft of unclaimed yield from rsETH holders — matching the **High: Theft of unclaimed yield** impact class.

## Likelihood Explanation
Two realistic trigger conditions exist:
1. **Natural staleness:** During price volatility, an asset price can spike and retrace within a single Chainlink heartbeat window (e.g., 1 hour for ETH-denominated feeds). If the deviation threshold is not crossed on the retrace, the oracle remains stale-high. Any EOA can then call `updateRSETHPrice()` to capture the phantom fee — no block stuffing required.
2. **Block stuffing:** An attacker can deliberately delay the keeper update by filling blocks, then call `updateRSETHPrice()` before the oracle corrects. This is expensive on mainnet but profitable at sufficient TVL.

The attack is repeatable each time the oracle lags behind a price retrace, and requires no privileged access.

## Recommendation
Add staleness and round-completeness validation in `ChainlinkPriceOracle.getAssetPrice()`:
```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();
require(price > 0, "Non-positive price");
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
```
`MAX_STALENESS` should be set per feed based on its documented heartbeat (e.g., 3600 s for a 1-hour heartbeat feed, with a small buffer). Consider storing it as a per-asset configurable parameter alongside `assetPriceFeed`.

## Proof of Concept
**Minimal Foundry fork test outline:**
1. Fork mainnet; deploy `LRTOracle` + `ChainlinkPriceOracle` pointing to a `MockStaleFeed`.
2. Deposit assets so `rsethSupply > 0` and `rsETHPrice` is stored at 1e18 (based on oracle price P1 = 1e18).
3. Configure `MockStaleFeed` to return `price = 1.005e18` (0.5% above P1, within a 1% `pricePercentageLimit`), with `updatedAt = block.timestamp - 2 hours`.
4. Call `updateRSETHPrice()` as an unprivileged EOA.
5. Assert `FeeMinted` event is emitted with `amount > 0`.
6. Assert `rsETH.totalSupply()` increased by `rsethAmountToMintAsProtocolFee` with no real ETH yield having accrued (underlying deposit balances unchanged).
7. Assert rsETH NAV per token decreased, confirming holder dilution.

The `MockStaleFeed` from the submission is sufficient; no privileged role is needed to trigger the mint.
