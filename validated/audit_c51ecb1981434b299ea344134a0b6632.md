### Title
Unrefunded Proposer Deposits Permanently Locked on `do_update` Mass-Clear — (`File: crates/contract/src/update.rs`)

---

### Summary

When `do_update` executes a winning contract-upgrade or config-update proposal, it unconditionally clears **all** other pending proposals and their votes. However, the storage-staking deposits paid by the proposers of those cleared proposals are never returned. Because `UpdateEntry` does not record the proposer's `AccountId`, there is no mechanism — neither automatic nor manual — to refund those deposits. The funds are permanently locked inside the contract balance.

---

### Finding Description

`propose_update` is a payable function that requires a deposit proportional to the byte-size of the proposed contract binary or config blob:

```rust
// crates/contract/src/lib.rs  ~L1308-1331
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
if attached < required { return Err(...); }
let id = self.proposed_updates.propose(update);
// Refund only the *excess* above required:
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

The `required` portion stays in the contract. The `UpdateEntry` stored for each proposal contains only the update payload and its byte count — **no proposer `AccountId`**:

```rust
// crates/contract/src/update.rs  ~L132-135
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
```

When threshold votes accumulate for any one proposal, `vote_update` calls `do_update`, which removes the winning entry and then **clears every other entry and all votes**:

```rust
// crates/contract/src/update.rs  ~L195-225
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
```

No deposit refund is issued for any of the cleared proposals. Because the proposer's `AccountId` was never stored in `UpdateEntry`, it is impossible to reconstruct who should receive a refund even if the code were patched post-hoc. There is also no `cancel_update` or `remove_update` endpoint that would let a proposer voluntarily withdraw their proposal and reclaim the deposit before execution.

---

### Impact Explanation

A contract binary for a production MPC node is typically hundreds of kilobytes. NEAR storage staking costs ~1 NEAR per 10 KB, so a 500 KB binary requires ~50 NEAR as a deposit. When two or more participants simultaneously propose competing upgrades (a realistic governance scenario), the participant whose proposal loses has their deposit permanently locked in the contract with no recovery path. The contract's balance grows by the sum of all losing deposits, and those funds are irrecoverable by the rightful owners.

This breaks the production accounting invariant that deposits paid for a specific purpose (storage staking for a proposal) must be returned when that purpose is voided (the proposal is cleared). The impact maps to:

> **Medium. Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants.**

---

### Likelihood Explanation

Multiple simultaneous proposals are explicitly supported and tested (`test_propose_update_contract_many`, `only_one_vote_from_participant`). Any governance round where two participants independently propose different upgrades — a routine occurrence during protocol evolution — triggers the loss. No adversarial intent is required; the loss is a structural consequence of the current design.

---

### Recommendation

1. **Store the proposer's `AccountId` in `UpdateEntry`** so that a refund target is always available.
2. **Refund cleared proposals in `do_update`**: iterate over `self.entries` before calling `clear()` and schedule a `Promise::transfer` to each stored proposer for their `required_deposit(bytes_used)`.
3. **Add a `cancel_update` endpoint** that lets a proposer withdraw their own proposal and receive an immediate refund, mirroring the pattern used in `remove_update_vote`.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB binary, attaching 50 NEAR. The required deposit (50 NEAR) stays in the contract; `UpdateEntry { update: Contract([...]), bytes_used: 500_000 }` is stored with no reference to A.
2. Participant B calls `propose_update` with a different binary, attaching another 50 NEAR.
3. Threshold participants vote for B's proposal. `vote_update` calls `do_update(&B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — silently dropping A's entry — and `self.vote_by_participant.clear()`.
5. No `Promise::transfer` is issued for A's 50 NEAR. The contract balance is 50 NEAR higher than it should be, and A has no mechanism to recover the funds. [1](#0-0) [2](#0-1) [3](#0-2)
