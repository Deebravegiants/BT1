Audit Report

## Title
Unhandled Revert in `ChainlinkPriceOracle.getAssetPrice()` on Deprecated Feed Freezes Deposits and Withdrawals - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `priceFeed.latestRoundData()` with no `try/catch` or error handling. When Chainlink deprecates a price feed by setting its underlying aggregator to `address(0)`, `latestRoundData()` reverts with empty revert data. This bare revert propagates uncaught through `LRTOracle.getAssetPrice()` and into every user-facing function that depends on asset pricing, temporarily freezing deposits, withdrawal initiations, instant withdrawals, and the operator-driven unlock queue that gates pending withdrawal completions.

## Finding Description

**Root cause:** `ChainlinkPriceOracle.getAssetPrice()` at `contracts/oracles/ChainlinkPriceOracle.sol` L49–55 calls `priceFeed.latestRoundData()` directly with no error handling:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

When Chainlink deprecates a feed, `latestRoundData()` on the deprecated aggregator reverts with empty data. This propagates upward through `LRTOracle.getAssetPrice()` at L156–158, which delegates directly with no protection: [2](#0-1) 

`LRTOracle._getTotalEthInProtocol()` at L336–344 iterates over all supported assets and calls `getAssetPrice()` for each — a single deprecated feed causes the entire loop to revert: [3](#0-2) 

**Affected user-facing call paths:**

- `LRTDepositPool.depositETH()` / `depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice()` (L520): [4](#0-3) 

- `LRTWithdrawalManager.initiateWithdrawal()` / `instantWithdrawal()` → `getExpectedAssetAmount()` → `lrtOracle.getAssetPrice()` (L593): [5](#0-4) 

- `LRTWithdrawalManager.unlockQueue()` → `_createUnlockParams()` → `lrtOracle.getAssetPrice()` (L848): [6](#0-5) 

No existing guard, circuit breaker, or fallback oracle exists anywhere in this call chain to catch or isolate the revert.

## Impact Explanation

**Medium — Temporary freezing of funds.**

If any Chainlink feed registered in `ChainlinkPriceOracle` is deprecated, all of the following revert until an admin replaces the oracle via `updatePriceOracleFor()`:

- `depositETH` / `depositAsset` — users cannot deposit new funds.
- `initiateWithdrawal` / `instantWithdrawal` — users cannot initiate new withdrawals.
- `unlockQueue` — the operator cannot unlock pending withdrawal requests, which blocks `completeWithdrawal` for users already in the queue whose requests have not yet been unlocked.

Users with funds already committed to the withdrawal queue have those funds temporarily frozen because `unlockQueue` cannot execute. The freeze is not permanent (admin can replace the oracle), but it is unilaterally triggered by an external event with no on-chain mitigation path available to users.

## Likelihood Explanation

No attacker action is required. Chainlink has a documented and observed practice of deprecating feeds by setting the underlying aggregator to `address(0)` — confirmed on Polygon mainnet in January 2023. The LRT-rsETH protocol registers Chainlink feeds for LST assets (e.g., stETH, ETHx, rETH). Deprecation of any single registered feed triggers the freeze passively. The condition is externally triggered, repeatable, and requires no special privileges or victim mistakes.

## Recommendation

Wrap the `latestRoundData()` call in a `try/catch` in `ChainlinkPriceOracle.getAssetPrice()` and revert with a descriptive custom error so callers can distinguish oracle failure from other errors:

```solidity
error OracleCallFailed(address priceFeed);

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    try priceFeed.latestRoundData() returns (uint80, int256 price, uint256, uint256, uint80) {
        return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
    } catch {
        revert OracleCallFailed(address(priceFeed));
    }
}
```

Additionally, `LRTOracle._getTotalEthInProtocol()` should wrap each per-asset `getAssetPrice()` call in a `try/catch` so that a single deprecated feed does not freeze pricing for all other assets.

## Proof of Concept

1. Deploy a mock `AggregatorV3Interface` whose `latestRoundData()` always reverts with empty data (simulating a deprecated Chainlink feed with aggregator set to `address(0)`).
2. Register this mock as the price feed for stETH in `ChainlinkPriceOracle` via `updatePriceFeedFor`.
3. Call `ChainlinkPriceOracle.getAssetPrice(stETH)` — observe revert with empty data.
4. Call `LRTOracle._getTotalEthInProtocol()` (via `updateRSETHPrice()`) — observe revert propagation.
5. Call `LRTDepositPool.depositETH{value: 1 ether}(0, "")` — observe revert; deposit is blocked.
6. Call `LRTWithdrawalManager.initiateWithdrawal(stETH, amount, "")` — observe revert; withdrawal initiation is blocked.
7. Call `LRTWithdrawalManager.unlockQueue(stETH, ...)` — observe revert; queued withdrawals cannot be unlocked, blocking `completeWithdrawal` for users already in the queue.

A Foundry fork test on Ethereum mainnet can reproduce steps 3–7 by pointing the stETH price feed address to a mock reverting contract and asserting all listed calls revert.
