### Title
`WalletContract` Caller Deposit Not Refunded on Intermediate Callback Failure - (`File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary
The `WalletContract` (deployed as a global contract for ETH implicit accounts) has a `CallerDeposit` mechanism designed to refund an external caller's attached NEAR deposit when a cross-contract call fails. However, multiple early-return paths in `nep_141_storage_balance_callback` and `address_check_callback` return error responses without issuing the refund, causing the caller's deposit to be permanently trapped in the wallet contract.

### Finding Description
`CallerDeposit` is constructed in `inner_rlp_execute` from the full `attached_deposit` of any external (non-self) caller and is threaded through the promise chain to be refunded in `rlp_execute_callback` if the final cross-contract call fails. [1](#0-0) 

The refund is correctly issued in `rlp_execute_callback` on `PromiseResult::Failed`: [2](#0-1) 

However, `nep_141_storage_balance_callback` has three early-return paths that silently drop `CallerDeposit`:

1. When `storage_balance_of` itself fails: [3](#0-2) 

2. When the JSON response from `storage_balance_of` cannot be parsed: [4](#0-3) 

3. When the action is not a `FunctionCall` (unexpected action type): [5](#0-4) 

Similarly, `address_check_callback` has three early-return paths that drop `CallerDeposit`:

1. When the address registrar call fails: [6](#0-5) 

2. When the JSON response from the registrar cannot be parsed: [7](#0-6) 

3. When the target resolves to an existing named account and the signer is not the wallet itself: [8](#0-7) 

In all six paths, `PromiseOrValue::Value(ExecuteResponse { success: false, ... })` is returned. Because the NEAR runtime only refunds a function call's `attached_deposit` on a panic, not on a successful return of an error value, the deposit received by `rlp_execute` stays permanently in the wallet contract.

### Impact Explanation
An external caller who attaches NEAR tokens to `rlp_execute` for an ERC-20 transfer (which requires at least 1 yoctoNEAR for `ft_transfer`) or an EOA base-token transfer with an address-registrar check loses their entire attached deposit if any intermediate call (`storage_balance_of` or registrar `lookup`) fails or returns an unparseable response. The deposit is credited to the wallet contract's balance and is unrecoverable by the caller. The corrupted protocol value is the **caller's NEAR balance**, which is reduced by the full deposit amount that should have been refunded.

### Likelihood Explanation
The trigger requires an intermediate call to fail. Realistic scenarios include:
- A token contract that does not implement `storage_balance_of` (returns a panic)
- A token contract that is paused or rate-limited
- A token contract that returns a non-standard JSON response for `storage_balance_of`
- An address registrar that is temporarily unavailable

Any external relayer or user who calls `rlp_execute` with a non-zero deposit targeting such a token contract will lose their deposit. The scenario is reachable by an unprivileged external user through a standard public RPC transaction.

### Recommendation
Each early-return path in `nep_141_storage_balance_callback` and `address_check_callback` that returns a failure response should first issue a refund promise for the `CallerDeposit`, mirroring the pattern already used in `rlp_execute_callback`:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund_promise = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(
        refund_promise,
        NearToken::from_yoctonear(yocto_near.into()),
    );
}
```

This should be applied before every `return PromiseOrValue::Value(ExecuteResponse { success: false, ... })` in both callbacks.

### Proof of Concept
1. External caller attaches 3 NEAR and calls `rlp_execute` targeting a token contract for an ERC-20 transfer.
2. The wallet contract calls `storage_balance_of` on the token contract.
3. The token contract panics (e.g., it is paused).
4. `nep_141_storage_balance_callback` is invoked with `PromiseResult::Failed`.
5. The callback returns `PromiseOrValue::Value(ExecuteResponse { success: false, ... })` at line 205 without creating any refund promise.
6. The caller's 3 NEAR deposit remains in the wallet contract's balance permanently.

The `test_caller_refunds` test in `runtime/near-wallet-contract/implementation/wallet-contract/src/tests/sanity.rs` demonstrates that the refund mechanism works correctly when the *final* cross-contract call fails (via `rlp_execute_callback`), but does not cover the intermediate-callback failure paths described above. [9](#0-8)

// An external caller gets its deposit back if the cross-contract call fails.
#[tokio::test]
async fn test_caller_refunds() -> anyhow::Result<()> {
    let TestContext { worker, wallet_contract, wallet_sk, address_registrar, .. } =
        TestContext::new().await?;

    let caller = worker.root_account()?;
    let deposit_amount = NearToken::from_near(3);
    let create_tx = |receiver_id: &AccountId, nonce: u64| {
        let method = "register";
        let args = br#"{"account_id": "birchmd.near"}"#;
        let action = Action::FunctionCall {
            receiver_id: receiver_id.to_string(),
            method_name: method.into(),
            args: args.to_vec(),
            gas: Gas::from_tgas(10).as_gas(),
            yocto_near: 0,
        };
        utils::create_signed_transaction(
            nonce,
            receiver_id,
            Wei::new_u128(deposit_amount.as_yoctonear() / (MAX_YOCTO_NEAR as u128)),
            action,
            &wallet_sk,
        )
    };

    // External caller gets a refund when the cross-contract call fails
    let pre_tx_account_balance = caller.view_account().await?.balance;
    let receiver_id: AccountId = "fake.near".parse()?;
    let result = wallet_contract
        .rlp_execute_from(
            &caller,
            receiver_id.as_str(),
            &create_tx(&receiver_id, 0),
            deposit_amount,
        )
        .await?;
    assert!(!result.success);
    let post_tx_account_balance = caller.view_account().await?.balance;
    assert!(
        pre_tx_account_balance.as_yoctonear() - post_tx_account_balance.as_yoctonear()
            < deposit_amount.as_yoctonear()
    );
