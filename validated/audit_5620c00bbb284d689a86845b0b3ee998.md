Audit Report

## Title
No Staleness Check on Chainlink Price Data Allows Stale Asset Prices to Corrupt rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-related return values, returning the last recorded price with no validation. A stale Chainlink feed will silently propagate an outdated asset price into `rsETHPrice`, the exchange rate used for all deposits and withdrawals. The public `updateRSETHPrice()` entry point allows any caller to commit the stale price to state.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at L52, the call to `latestRoundData()` destructures only `answer` and discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

There is no `answeredInRound < roundId` check, no `updatedAt == 0` check, and no time-based heartbeat check. This price flows through `LRTOracle.getAssetPrice()` (L156–158) → `_getTotalEthInProtocol()` (L339–343) → `_updateRsETHPrice()` (L250), which writes the resulting `newRsETHPrice` to `rsETHPrice` (L313).

The public `updateRSETHPrice()` (L87–89) has only a `whenNotPaused` modifier and is callable by any address. The `pricePercentageLimit` guard in `_updateRsETHPrice()` (L256–266) provides partial protection against large sudden deviations, but is ineffective when: (a) `pricePercentageLimit` is 0 (unset), or (b) the stale price deviation is within the configured limit. The guard also only compares against `highestRsethPrice`, not the previous price, so a feed frozen below the all-time high passes unchecked.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` at L30–32 applies `answeredInRound < roundID` and `timestamp == 0` checks, confirming the project is aware of the staleness pattern but did not apply it to the primary oracle.

**Exploit path:**
1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale — last updated 30+ hours ago.
2. Any unprivileged caller invokes `LRTOracle.updateRSETHPrice()`.
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice()` for each supported asset.
4. The stale price is returned without revert, inflating or deflating `totalETHInProtocol`.
5. `rsETHPrice` is updated to the incorrect value and used for all subsequent deposits and withdrawals.

## Impact Explanation
**High: Theft of unclaimed yield.** When a stale price is lower than the true asset price, `newRsETHPrice` is computed below its correct value. New depositors receive more rsETH than they are entitled to at the expense of existing rsETH holders, whose proportional claim on protocol assets is diluted — constituting theft of unclaimed yield from existing holders. In the opposite direction (stale price above true value), depositors receive fewer rsETH tokens than owed, which maps to **Low: Contract fails to deliver promised returns**.

## Likelihood Explanation
Chainlink feeds have historically gone stale during network congestion, sequencer downtime (on L2s), or oracle node outages. The affected oracle is the primary price source for all supported LST assets. `updateRSETHPrice()` is public with no access control beyond `whenNotPaused`, so the stale price can be committed to state by any caller at any time without admin intervention. Likelihood is Low (requires a feed outage), but the trigger requires no special privileges.

## Recommendation
Add round-completeness and time-based heartbeat checks in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_HEARTBEAT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_HEARTBEAT` should be configurable per feed or set conservatively (e.g., 24 hours for LST/ETH feeds on Ethereum mainnet).

## Proof of Concept
**Foundry fork test outline:**

1. Fork Ethereum mainnet at a block where a supported LST/ETH Chainlink feed has not been updated for >24 hours (or mock `latestRoundData()` to return a stale `updatedAt`).
2. Call `LRTOracle.updateRSETHPrice()` from an unprivileged EOA.
3. Assert that `rsETHPrice` was updated to a value that differs from the correct price computed using the live feed.
4. Simulate a deposit at the stale price and show the depositor receives more rsETH than the correct amount, diluting existing holders.
5. Confirm the call does not revert, demonstrating the absence of any staleness guard.

The absence of any staleness check is directly visible at `contracts/oracles/ChainlinkPriceOracle.sol` L52, and the contrast with the existing guard at `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` L30–32 makes the omission unambiguous.
