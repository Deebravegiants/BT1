### Title
Missing `curve.check()` on deserialized root public key allows infinity-point injection in HD key derivation — (`src/cbmpc/api/hd_keyset_ecdsa_2p.cpp`)

### Summary

`blob_to_keyset` calls `ecc_point_t::from_bin` on `root_Q_compressed` but never calls `ecurve_t::check()` on the resulting point. For OpenSSL-backed curves (P-256, P-384, P-521), `ecurve_ossl_t::from_bin` explicitly accepts the infinity encoding (a zero-prefixed byte string). `ecurve_t::check()` is the only guard that rejects infinity. Because it is absent, an attacker-controlled keyset blob with `root_Q_compressed = 0x00 || 0x00...` (33 or 65 zero bytes for P-256) passes deserialization and enters `derive_keys` with `keyset.root.Q = ∞`.

### Finding Description

**Deserialization path — `blob_to_keyset`:** [1](#0-0) 

`from_bin` is called and its error is checked, but `curve.check()` is never called. Compare with `ecc_point_t::convert_fixed_curve`, which is the only other deserialization path and explicitly calls `curve.check()` after `from_bin`: [2](#0-1) 

**`ecurve_ossl_t::from_bin` accepts infinity:** [3](#0-2) 

A blob with `root_Q_compressed` = 33 zero bytes (P-256) passes this function and sets `P` to the infinity point.

**`ecurve_t::check()` is the only infinity guard:** [4](#0-3) 

Because `blob_to_keyset` never calls it, infinity propagates into `keyset.root.Q`.

**`derive_keys` computes `Q_derived = ∞ + delta*G = delta*G`:** [5](#0-4) 

`delta` and `chain_code` are both derived from the VRF output `Z = Z1 + Z2`, which is known to both protocol participants. So `Q_derived = delta*G` is a value fully determined by the interactive hardened-derivation output — the root key `Q` is completely discarded.

**`non_hard_derive` uses `Q_derived` as HMAC data:** [6](#0-5) 

`HMAC-SHA512(key=chain_code, data=Q_temp || index)` — both `chain_code` and `Q_temp = delta*G` are known to both parties. Every `non_hard_delta[i]` is therefore fully determined by the hardened-derivation output, with no contribution from the root key `Q`.

**secp256k1 is NOT affected** — `ecurve_secp256k1_t::from_bin` delegates to `secp256k1_eckey_pubkey_parse`, which rejects any encoding that is not a valid 33-byte compressed or 65-byte uncompressed point, so infinity cannot be injected on that curve: [7](#0-6) 

### Impact Explanation

For P-256, P-384, and P-521 keysets, a party that supplies a crafted keyset blob with `root_Q_compressed = 0x00...` causes `derive_ecdsa_2p_keys` to:

1. Accept the blob without error.
2. Compute all derived key public keys as `delta*G + non_hard_delta[i]*G`, where every scalar is determined solely by the VRF output `Z` (known to both participants).
3. Produce derived key blobs whose public keys bear no relationship to the root key `Q`.
4. Allow the crafting party to predict every derived key public key before the protocol completes, since they know `delta` and `chain_code`.

This is a public-API-reachable validation bypass that produces accepted invalid cryptographic output (derived keys structurally detached from the root key), satisfying the High impact criterion.

### Likelihood Explanation

The attacker must be one of the two protocol participants and must supply a crafted keyset blob. This is a realistic threat model for a Byzantine participant. The crafted blob requires only setting `root_Q_compressed` to the correct-length zero string; all other fields can be valid. No threshold collusion or host compromise is required.

### Recommendation

In `blob_to_keyset`, call `curve.check()` on both `keyset.root.Q` and `keyset.root.K` immediately after `from_bin` succeeds:

```cpp
error_t rv = keyset.root.Q.from_bin(icurve, blob.root_Q_compressed);
if (rv) return rv;
if (rv = icurve.check(keyset.root.Q)) return rv;   // ADD THIS
rv = keyset.root.K.from_bin(icurve, blob.root_K_compressed);
if (rv) return rv;
return icurve.check(keyset.root.K);                // ADD THIS
```

`ecurve_t::check` already rejects infinity, off-curve points, and wrong-curve points. [8](#0-7) 

### Proof of Concept

1. Choose curve P-256. Obtain or construct a valid `keyset_blob_v1_t` for that curve.
2. Replace `root_Q_compressed` with 33 zero bytes (`0x00 * 33`).
3. Call `derive_ecdsa_2p_keys` with this blob and any hardened + non-hardened paths.
4. Observe that `deserialize_keyset_blob` returns `SUCCESS`.
5. Record `delta` and `chain_code` from the VRF output `Z` (both parties compute these).
6. Independently compute `Q_derived = delta * G` and run `non_hard_derive(Q_derived, chain_code, paths)`.
7. Verify that the `non_hard_delta` values match those embedded in the returned derived key blobs, confirming that all non-hardened child keys are fully determined by the hardened-derivation output and the root key `Q` was silently ignored.

error_t ecurve_t::check(const ecc_point_t& point) const {
  if (!point.valid()) return crypto::error("EC-point invalid");
  if (point.get_curve() != *this) return crypto::error("EC-point of wrong curve");
  if (!point.is_in_subgroup()) return crypto::error("EC-point is not on curve");

  if (!thread_local_store_allow_ecc_infinity) {
    if (point.is_infinity()) return crypto::error("EC-point is infinity");
  }
  return SUCCESS;
}
