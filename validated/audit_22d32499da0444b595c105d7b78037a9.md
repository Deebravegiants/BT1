Audit Report

## Title
Unbounded `merkleProof` Length in `_verifyClaimProof` Enables O(N) Gas Consumption — (`contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

## Summary
`_verifyClaimProof` accepts a `bytes32[] calldata merkleProof` parameter with no upper-bound check on its length. Any unprivileged caller can pass an arbitrarily large proof array to `claim()` or `claimAndStake()`, causing `MerkleProofUpgradeable.verify()` to iterate over every element and compute a `keccak256` hash per step before reverting with `InvalidMerkleProof`. This enables sustained O(N) gas consumption per transaction with no state written.

## Finding Description
`_verifyClaimProof` performs three guards before the expensive operation: a non-zero address check, a non-zero `merkleRoot` check, and a non-zero `amount` check. [1](#0-0) 

None of these guards bound `merkleProof.length`. The call then reaches `MerkleProofUpgradeable.verify()`, which delegates to `processProof()`: [2](#0-1) 

This loop runs exactly `merkleProof.length` times with no early exit, calling `_hashPair` → `_efficientHash` (inline `keccak256` via assembly) on each iteration: [3](#0-2) 

After all N hashes, the computed root will not match `merkleRoot`, so the function reverts with `InvalidMerkleProof`. No state is written. The public entrypoints exposing this path are `claim()` and `claimAndStake()`, both `external` with only `nonReentrant whenNotPaused` guards and no role or allowlist restriction: [4](#0-3) [5](#0-4) 

## Impact Explanation
**Medium — Unbounded gas consumption.** An attacker supplying N = 10,000 proof elements consumes approximately 5.6 M gas per transaction (calldata + computation). At the Ethereum block gas limit (~30 M gas), five such transactions saturate a block, preventing all legitimate `claim()` calls from being included. The attacker can repeat this every block, causing sustained temporary denial of the claim flow. This maps exactly to the allowed impact "Medium. Unbounded gas consumption."

## Likelihood Explanation
No role, allowlist, or signature requirement gates `claim()` or `claimAndStake()`. The only prerequisites are that `merkleRoot` is non-zero (set at initialization) and `amount > 0` (any non-zero value). The attack is executable by any EOA immediately after deployment. Economic cost is non-trivial on mainnet but not prohibitive for a motivated griever, and is significantly cheaper on L2 deployments.

## Recommendation
Add a maximum proof-length guard at the top of `_verifyClaimProof`. For a 100-leaf tree, a valid proof is at most `ceil(log2(100)) = 7` elements; a ceiling of 20 is generous and eliminates the attack surface entirely:

```solidity
uint256 public constant MAX_PROOF_LENGTH = 20;

function _verifyClaimProof(
    address user,
    uint256 amount,
    bytes32[] calldata merkleProof
) internal view {
    if (merkleProof.length > MAX_PROOF_LENGTH) revert InvalidMerkleProof();
    // ... existing checks
}
```

## Proof of Concept
Deploy `KernelTop100MerkleDistributor` with a non-zero `merkleRoot`. Run the following Foundry fuzz test:

```solidity
function testFuzz_unboundedProofGas(uint16 proofLen) public {
    vm.assume(proofLen > 0 && proofLen <= 10_000);
    bytes32[] memory proof = new bytes32[](proofLen);
    for (uint256 i = 0; i < proofLen; i++) {
        proof[i] = bytes32(uint256(i + 1));
    }
    uint256 gasBefore = gasleft();
    try distributor.claim(1 ether, proof) {} catch {}
    uint256 gasUsed = gasBefore - gasleft();
    emit log_named_uint("proofLen", proofLen);
    emit log_named_uint("gasUsed", gasUsed);
}
```

`gasUsed` will grow linearly with `proofLen` with no enforced ceiling, confirming the unbounded gas consumption invariant violation.

            revert InvalidMerkleProof();
        }
    }
