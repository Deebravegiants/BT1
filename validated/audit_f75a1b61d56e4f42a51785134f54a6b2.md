### Title
Withdrawal Payout Capped at Initiation-Time Oracle Price, Causing Withdrawing Users to Lose Accrued Yield - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

In `LRTWithdrawalManager`, when a user initiates a withdrawal, the `expectedAssetAmount` is calculated using the rsETH oracle price at initiation time and stored in the `WithdrawalRequest` struct. When the withdrawal is later unlocked by the operator, `_calculatePayoutAmount` returns `min(expectedAssetAmount_at_initiation, currentReturn_at_unlock)`. Because rsETH price continuously increases as EigenLayer restaking yield accrues, `currentReturn_at_unlock` will always exceed `expectedAssetAmount_at_initiation`, so the user is permanently capped at the stale initiation-time value. The yield that accrued on the user's locked rsETH position during the waiting period is silently absorbed by the protocol, benefiting remaining rsETH holders at the expense of withdrawing users.

---

### Finding Description

When `initiateWithdrawal` is called, the user's rsETH is transferred into the contract and `expectedAssetAmount` is computed from the current oracle price and stored:

```solidity
// LRTWithdrawalManager.sol:168-175
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
...
withdrawalRequests[requestId] = WithdrawalRequest({
    rsETHUnstaked: rsETHUnstaked,
    expectedAssetAmount: expectedAssetAmount,   // ← locked at initiation price
    withdrawalStartBlock: block.number
});
``` [1](#0-0) 

When the operator later calls `unlockQueue`, `_calculatePayoutAmount` is invoked:

```solidity
// LRTWithdrawalManager.sol:824-835
function _calculatePayoutAmount(
    WithdrawalRequest storage request,
    uint256 rsETHPrice,
    uint256 assetPrice
) private view returns (uint256) {
    uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
    return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
}
``` [2](#0-1) 

The function returns `min(expectedAssetAmount_at_initiation, currentReturn_at_unlock)`. Because rsETH price monotonically increases as restaking yield accrues, `currentReturn_at_unlock > expectedAssetAmount_at_initiation` in the normal case, so the user always receives the stale initiation-time amount. The rsETH burned at unlock time is worth more than what the user receives; the surplus value remains in the protocol.

The rsETH is not burned at initiation — it is burned at unlock time:

```solidity
// LRTWithdrawalManager.sol:305
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
``` [3](#0-2) 

During the entire waiting period (initiation → unlock, which can span many days given `withdrawalDelayBlocks = 8 days / 12 seconds`), the rsETH sitting in the contract appreciates in value, but the withdrawing user receives none of that appreciation. [4](#0-3) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A user who initiates a withdrawal when rsETH price is `P₀` and whose withdrawal is unlocked when rsETH price is `P₁ > P₀` receives `rsETHUnstaked × P₀ / assetPrice` instead of `rsETHUnstaked × P₁ / assetPrice`. The difference `rsETHUnstaked × (P₁ - P₀) / assetPrice` is yield that accrued on the user's position during the waiting period and is permanently lost to the user. Over an 8-day delay with typical EigenLayer restaking APY (~4%), this represents a material and quantifiable loss per withdrawal.

---

### Likelihood Explanation

**High.** rsETH price increases continuously as EigenLayer restaking rewards accrue and as MEV/execution-layer rewards are forwarded via `FeeReceiver.sendFunds()`. Every withdrawal that experiences any non-zero waiting period between initiation and unlock will be affected. The withdrawal delay is initialized to 8 days and can be up to 16 days, making this loss unavoidable for all normal withdrawals. [5](#0-4) 

---

### Recommendation

At unlock time, compute the payout using the current oracle price rather than the stale initiation-time `expectedAssetAmount`. The `min()` guard against slashing is still valid, but the cap should be the current value, not the initiation-time value. Specifically, `_calculatePayoutAmount` should return `currentReturn` when `currentReturn >= expectedAssetAmount_at_initiation`, so that users receive the full value of their rsETH at the time of actual payment processing — analogous to the Wildcat fix of updating `scaleFactor` to `block.timestamp` rather than `expiry` before processing the batch.

---

### Proof of Concept

1. rsETH price = 1.10 ETH/rsETH. User calls `initiateWithdrawal(ETH, 100e18 rsETH)`.
   - `expectedAssetAmount = 100e18 × 1.10 / 1.0 = 110 ETH` stored in request.
   - 100 rsETH transferred to `LRTWithdrawalManager`.

2. 8 days pass. EigenLayer restaking yield accrues. rsETH price = 1.12 ETH/rsETH.

3. Operator calls `unlockQueue(ETH, ...)`.
   - `currentReturn = 100e18 × 1.12 / 1.0 = 112 ETH`
   - `payoutAmount = min(110 ETH, 112 ETH) = 110 ETH` ← user capped at stale price
   - 100 rsETH (worth 112 ETH) is burned; user receives only 110 ETH.

4. User calls `completeWithdrawal(ETH, ...)` and receives 110 ETH.

5. **Loss**: 2 ETH of yield that accrued on the user's position during the 8-day waiting period is absorbed by the protocol. At scale (e.g., 10,000 ETH in the withdrawal queue), this represents 200 ETH of yield stolen from withdrawing users per unlock cycle. [6](#0-5)
