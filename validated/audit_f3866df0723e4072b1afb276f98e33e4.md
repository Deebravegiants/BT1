### Title
Uncheckpointed EigenPod ETH Excluded from `getEffectivePodShares()`, Allowing New Depositors to Receive Inflated rsETH at Existing Holders' Expense — (`contracts/NodeDelegator.sol`, `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

---

### Summary

`NodeDelegator.getEffectivePodShares()` only sums `stakedButUnverifiedNativeETH` and the DelegationManager's `withdrawableShares` for the `beaconChainETHStrategy`. ETH that has landed in the EigenPod contract via execution-layer rewards or partial withdrawals (emitting `NonBeaconChainETHReceived`) is not reflected in either term until a checkpoint is started and finalized. Because `LRTDepositPool.getETHDistributionData()` feeds directly into `LRTOracle._getTotalEthInProtocol()`, the oracle can be made to undercount total protocol ETH. Any deposit made while the price is in this understated state mints more rsETH than the depositor's proportional share of real assets, diluting existing holders.

---

### Finding Description

**Step 1 — `getEffectivePodShares()` accounting** [1](#0-0) 

The function returns exactly two terms:
- `stakedButUnverifiedNativeETH` — incremented only by `stake32Eth()` and decremented by `verifyWithdrawalCredentials()`.
- `withdrawableShare` — the DelegationManager's view of withdrawable shares for `beaconChainETHStrategy`, which is only updated when EigenLayer formally credits shares (after `verifyWithdrawalCredentials` or after a checkpoint is finalized).

ETH that arrives at the EigenPod address directly (execution-layer rewards, partial beacon-chain withdrawals) sits in the EigenPod contract's balance and is not credited as EigenLayer shares until `startCheckpoint` + `verifyCheckpointProofs` completes. Neither term in `getEffectivePodShares()` captures it.

**Step 2 — Propagation to the oracle**

`LRTDepositPool.getETHDistributionData()` calls `getEffectivePodShares()` for every NDC: [2](#0-1) 

The result feeds into `getTotalAssetDeposits(ETH_TOKEN)`: [3](#0-2) 

Which is consumed by `LRTOracle._getTotalEthInProtocol()`: [4](#0-3) 

**Step 3 — Price is stored and publicly updatable**

`updateRSETHPrice()` is a public, permissionless function (when not paused): [5](#0-4) 

The stored `rsETHPrice` is what `getRsETHAmountToMint` divides by: [6](#0-5) 

**Step 4 — Mint inflation**

If `rsETHPrice` is set while uncheckpointed pod ETH exists, the price is lower than the true backing ratio. Any deposit during this window mints `amount / understatedPrice` rsETH — more than the depositor's true proportional claim. After the checkpoint finalizes, `totalETHInProtocol` jumps, the price rises, but the extra rsETH already minted dilutes all prior holders.

---

### Impact Explanation

The uncheckpointed ETH in the EigenPod represents accrued yield (execution-layer staking rewards, partial withdrawals). New depositors who enter while the price is understated effectively capture a share of that yield without having contributed to earning it. Existing rsETH holders bear the dilution. This is a direct, quantifiable theft of unclaimed yield.

---

### Likelihood Explanation

- Execution-layer rewards flow to EigenPods continuously on mainnet; the window between receipt and checkpoint finalization is routine and can span hours to days.
- `updateRSETHPrice()` is permissionless — an attacker can call it immediately after ETH arrives at the pod (observable on-chain) to lock in the understated price, then deposit.
- No privileged role, no governance capture, and no external oracle compromise is required.
- The only prerequisite is that the NDC has an active EigenPod receiving rewards, which is the normal operating state of the protocol.

---

### Recommendation

Include the EigenPod's raw ETH balance in the accounting. In `getEffectivePodShares()` (or a new helper), add `address(eigenPod).balance` to the returned value. This ensures that ETH sitting in the pod but not yet checkpointed is still counted as protocol-owned ETH, keeping the rsETH price accurate at all times.

Alternatively, enforce that `updateRSETHPrice()` can only be called after a checkpoint has been finalized (i.e., `eigenPod.currentCheckpointTimestamp() == 0`), preventing price updates while a checkpoint is in progress or while uncheckpointed pod ETH exists.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Pseudocode unit test (local fork or mock environment)

function test_uncheckpointedPodEthInflatesMint() public {
    // 1. Setup: NDC has an EigenPod with 1 active validator (32 ETH verified).
    //    rsETHPrice is 1.0 ETH, totalSupply = 100 rsETH, TVL = 100 ETH.

    // 2. Simulate 1 ETH of execution-layer rewards arriving at the EigenPod
    //    (NonBeaconChainETHReceived). No checkpoint has been started.
    vm.deal(address(eigenPod), 1 ether);

    // 3. Attacker calls updateRSETHPrice() — price is computed on understated TVL (100 ETH, not 101 ETH).
    lrtOracle.updateRSETHPrice();
    assertEq(lrtOracle.rsETHPrice(), 1.0 ether); // price unchanged, pod ETH ignored

    // 4. Attacker deposits 1 ETH.
    //    rsethToMint = 1e18 * 1e18 / 1.0e18 = 1.0 rsETH
    //    True fair share would be 1/101 * 101 = ~0.99 rsETH
    uint256 minted = depositPool.getRsETHAmountToMint(ETH_TOKEN, 1 ether);
    assertGt(minted, 0.99 ether); // attacker gets ~1.0 rsETH instead of ~0.99 rsETH

    // 5. Operator finalizes checkpoint — EigenLayer credits 1 ETH as shares.
    //    TVL is now 102 ETH, rsETH supply is 101 rsETH.
    finalizeCheckpoint();
    lrtOracle.updateRSETHPrice();

    // 6. New price = 102/101 ≈ 1.0099 ETH per rsETH.
    //    Attacker's 1.0 rsETH is now worth ~1.0099 ETH — they captured yield they didn't earn.
    //    Original 100 rsETH holders: expected 101/100 = 1.01 ETH each, got 102/101 ≈ 1.0099 ETH each.
    assertGt(lrtOracle.rsETHPrice(), 1.0 ether);
}
```

```text
    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
```
