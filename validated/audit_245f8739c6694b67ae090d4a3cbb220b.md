### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Enables Queue-Saturation DoS on Foreign Transaction Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction()` entry point stores pending yields under a `VerifyForeignTransactionRequest` key that contains **no caller identity**. Because the fan-out queue is capped at 128 slots and panics on overflow, any unprivileged account can pre-fill the queue for any specific foreign-chain transaction request, permanently blocking legitimate callers from submitting that request until the attacker stops refilling.

---

### Finding Description

**Root cause — caller-agnostic request key**

`SignatureRequest` (used by `sign()`) derives its map key from a `tweak` that is computed from `predecessor_id` and `path`: [1](#0-0) 

This makes every caller's queue entry unique; a malicious user cannot fill another user's slot.

`VerifyForeignTransactionRequest` (used by `verify_foreign_transaction()`) contains only `request`, `domain_id`, and `payload_version` — **no caller identity**: [2](#0-1) 

All callers submitting the same foreign-chain transaction share a single queue entry. The contract's own test acknowledges this asymmetry: [3](#0-2) 

**Queue cap enforcement — hard panic on overflow**

`push_pending_yield` panics (aborting the caller's transaction) when the queue reaches `MAX_PENDING_REQUEST_FAN_OUT = 128`: [4](#0-3) 

A caller whose submission panics is **not** enqueued and receives no response when MPC nodes later drain the queue via `resolve_yields_for`.

**Attack path**

1. Attacker observes (or predicts) a target `VerifyForeignTransactionRequest` key (e.g., a specific Bitcoin `tx_id` + `domain_id` + `payload_version`).
2. Attacker submits `verify_foreign_transaction(request_X)` 128 times (cost: 128 × 1 yoctoNEAR ≈ free).
3. Queue for `request_X` is saturated.
4. Legitimate user calls `verify_foreign_transaction(request_X)` → contract panics with `PendingRequestQueueFull`; user is not queued.
5. MPC nodes respond → all 128 attacker yields are drained; attacker receives 128 copies of the signature.
6. Attacker immediately re-submits 128 requests → queue is full again before the legitimate user can retry.
7. Cycle repeats indefinitely at negligible cost.

The `pop_oldest_pending_yield` timeout path removes only the head entry per timeout event: [5](#0-4) 

The attacker can replace each timed-out slot faster than the legitimate user can claim it.

---

### Impact Explanation

`verify_foreign_transaction` is the on-chain entry point for bridge withdrawal flows (Bitcoin, Ethereum, Polygon, etc.). A sustained queue-saturation attack on a specific foreign-chain transaction prevents the MPC network from ever delivering a signature to the legitimate bridge user for that transaction. The user's deposited foreign-chain funds are effectively frozen for as long as the attacker maintains the attack. This is a **request-lifecycle manipulation that breaks the production safety invariant** that any user can always obtain a signature for a valid foreign-chain transaction they submitted.

---

### Likelihood Explanation

- **No privilege required**: any NEAR account can call `verify_foreign_transaction`.
- **Cost is negligible**: 128 yoctoNEAR per saturation cycle.
- **Target is predictable**: foreign-chain transaction IDs are public on-chain; the attacker can observe the mempool or the foreign chain to know the exact request key before the victim submits.
- **Sustained attack is trivial**: the attacker simply re-submits 128 requests each time MPC drains the queue.

---

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, mirroring the `tweak`-based isolation already used by `SignatureRequest`. Alternatively, enforce a per-caller submission cap (e.g., 1 pending slot per account per request key) so a single attacker cannot monopolise the 128-slot queue.

---

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```

```rust
pub(crate) fn pop_oldest_pending_yield<K>(requests: &mut LookupMap<K, Vec<YieldIndex>>, request: &K)
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let Some(queue) = requests.get_mut(request) else {
        return;
    };
    if queue.is_empty() {
        requests.remove(request);
        return;
    }
    queue.remove(0);
    if queue.is_empty() {
        requests.remove(request);
    }
}
```
