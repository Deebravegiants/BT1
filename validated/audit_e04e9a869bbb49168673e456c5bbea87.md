Audit Report

## Title
Stale `rsETHPrice` Read in `instantWithdrawal` Enables Over-Redemption from Unstaking Vault - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.instantWithdrawal()` and `initiateWithdrawal()` both call `getExpectedAssetAmount()`, which reads `LRTOracle.rsETHPrice` — a stored, manually-updated state variable — without first invoking `updateRSETHPrice()`. When `rsETHPrice` is stale-higher than the true current rate (e.g., after an EigenLayer slashing event that has not yet been reflected by an oracle update), `instantWithdrawal` over-calculates `assetAmountUnlocked`, allowing a caller to burn rsETH and redeem more assets from the unstaking vault than the current exchange rate justifies. The downside-protection pause in `_updateRsETHPrice()` is never triggered because `updateRSETHPrice()` is never called before the withdrawal executes.

## Finding Description

`LRTOracle.rsETHPrice` is a plain storage variable written only inside `_updateRsETHPrice()`:

```solidity
// LRTOracle.sol:87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
// LRTOracle.sol:313
rsETHPrice = newRsETHPrice;
```

`getExpectedAssetAmount()` reads this stored value directly without refreshing it:

```solidity
// LRTWithdrawalManager.sol:590-594
ILRTOracle lrtOracle = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE));
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

Note that `lrtOracle.getAssetPrice(asset)` fetches a **live** price from an external price fetcher (Chainlink or equivalent), while `lrtOracle.rsETHPrice()` returns the **stored** stale value.

Both withdrawal entry points call `getExpectedAssetAmount()` without a prior price refresh:

- `initiateWithdrawal` (line 168): `uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);`
- `instantWithdrawal` (line 228): `uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);`

**Exploit path for `instantWithdrawal` (Scenario B — stale price higher than actual):**

1. An EigenLayer strategy suffers a slashing loss. The true rsETH/ETH rate drops from, say, 1.05 to 1.00, but `rsETHPrice` in `LRTOracle` still stores 1.05 because `updateRSETHPrice()` has not been called.
2. The downside-protection logic in `_updateRsETHPrice()` (lines 270–281) — which would pause `LRTDepositPool` and `LRTWithdrawalManager` if the price drop exceeds `pricePercentageLimit` — is never reached because `updateRSETHPrice()` is never called before `instantWithdrawal` executes.
3. Attacker calls `instantWithdrawal(asset, rsETHUnstaked, ...)`.
4. `assetAmountUnlocked = rsETHUnstaked * 1.05e18 / assetPrice` — over-calculated by ~5%.
5. The check `assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)` limits the per-transaction drain but does not correct the rate mismatch.
6. `unstakingVault.redeem(asset, assetAmountUnlocked)` transfers the inflated amount to the withdrawal manager, which then pays the attacker.
7. The attacker receives more assets than their rsETH is worth at the current rate, extracting value from the unstaking vault.

**Exploit path for `initiateWithdrawal` (Scenario A — stale price lower than actual):**

1. Rewards accrue in EigenLayer; true rsETH price is 1.05 but stored `rsETHPrice` is 1.00.
2. User calls `initiateWithdrawal(asset, 100e18, ...)`.
3. `expectedAssetAmount = 100e18 * 1.00e18 / assetPrice` — under-calculated.
4. `assetsCommitted[asset] += expectedAssetAmount` is under-counted, inflating `getAvailableAssetAmount`, allowing more withdrawals to be queued than the vault can service.
5. When `unlockQueue` eventually runs with the refreshed price, `_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn) = expectedAssetAmount` (the stale-lower amount). The user is systematically underpaid.

## Impact Explanation

**Primary (High — Theft of unclaimed yield):** In Scenario B, an unprivileged caller can exploit the stale-higher `rsETHPrice` during `instantWithdrawal` to redeem more assets from the unstaking vault than their burned rsETH is worth at the current rate. The excess assets are extracted from the vault, which holds protocol-owned assets accrued for rsETH holders. This constitutes theft of unclaimed yield bounded by `getAssetsAvailableForInstantWithdrawal` per transaction but repeatable across multiple transactions within the stale-price window.

**Secondary (Low — Contract fails to deliver promised returns):** In Scenario A, users who initiate withdrawals during a stale-lower price window are locked into an under-calculated `expectedAssetAmount`. When `unlockQueue` runs with the refreshed price, `_calculatePayoutAmount` caps their payout at the stale-lower amount, delivering fewer assets than the current rsETH/ETH rate entitles them to.

## Likelihood Explanation

`rsETHPrice` is updated on a periodic operator schedule, not atomically before every withdrawal. Any user can call `updateRSETHPrice()` themselves, but nothing in `initiateWithdrawal` or `instantWithdrawal` enforces this. For Scenario B, the attacker must observe an EigenLayer slashing event and act within the window before an operator (or anyone) calls `updateRSETHPrice()`. This window can span hours. For Scenario A, the condition (rewards accruing between oracle updates) is the normal operating state of the protocol, making it continuously present. Likelihood is **Medium** for Scenario B and **High** for Scenario A.

## Recommendation

Call `updateRSETHPrice()` (or an internal equivalent) at the start of both `initiateWithdrawal` and `instantWithdrawal` before reading `lrtOracle.rsETHPrice()`:

```solidity
// At the top of initiateWithdrawal and instantWithdrawal:
ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
```

This ensures the price used for `expectedAssetAmount` and `assetAmountUnlocked` always reflects the latest protocol state, and also ensures the downside-protection pause fires before any withdrawal can execute against a post-slashing stale price.

## Proof of Concept

**Scenario B — `instantWithdrawal` over-redemption:**

1. Deploy fork of mainnet. Set `rsETHPrice = 1.05e18` in `LRTOracle` (simulating a state where rewards have accrued and price was last updated at 1.05).
2. Simulate an EigenLayer slashing event that reduces total ETH in protocol such that the true rsETH price is now 1.00e18. Do NOT call `updateRSETHPrice()`.
3. Confirm `lrtOracle.rsETHPrice()` still returns `1.05e18`.
4. Attacker holds `100e18` rsETH. Call `instantWithdrawal(ETH, 100e18, "")`.
5. Observe `assetAmountUnlocked = 100e18 * 1.05e18 / 1e18 = 105e18` (assuming ETH asset price = 1e18).
6. Observe attacker receives `105 ETH` (minus fee) while burning rsETH worth only `100 ETH` at the current rate.
7. Call `updateRSETHPrice()` after the fact; confirm the protocol would have paused (if price drop exceeded `pricePercentageLimit`) or reflected the lower price — too late to prevent the over-redemption.

**Scenario A — `initiateWithdrawal` underpayment:**

1. Set `rsETHPrice = 1.00e18` (stale-lower; true rate is 1.05e18 after reward accrual).
2. User calls `initiateWithdrawal(ETH, 100e18, "")`. `expectedAssetAmount = 100 ETH` stored in request.
3. Call `updateRSETHPrice()`; `rsETHPrice` updates to `1.05e18`.
4. Operator calls `unlockQueue`. `_calculatePayoutAmount` returns `min(100, 105) = 100 ETH`.
5. User receives `100 ETH` despite having burned rsETH worth `105 ETH` at the current rate. The 5 ETH shortfall is retained by the protocol.
