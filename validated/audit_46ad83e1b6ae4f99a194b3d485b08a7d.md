### Title
Sub-Transaction Reference Scripts Bypass Minimum Fee Calculation in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces nested sub-transactions (`subTransactionsTxBodyL`). The minimum fee validation for a Dijkstra top-level transaction uses `getConwayMinFeeTxUtxo`, which only measures reference script sizes from the **top-level** transaction's inputs via `txNonDistinctRefScriptsSize`. Reference scripts attached to sub-transaction inputs are entirely excluded from the fee calculation. A helper function `batchNonDistinctRefScriptsSize` that correctly aggregates reference script sizes across the whole batch exists in the same file but is never wired into the fee-validation path. An unprivileged sender can therefore include arbitrarily large reference scripts in sub-transactions, forcing nodes to deserialize them at full cost while paying only the fee for the top-level transaction's (small) reference scripts.

---

### Finding Description

**Root cause — `getMinFeeTxUtxo` ignores sub-transaction reference scripts**

`DijkstraEra`'s `EraUTxO` instance delegates minimum-fee computation to `getConwayMinFeeTxUtxo`:

```haskell
instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getConwayMinFeeTxUtxo   -- ← Conway's function, top-level only
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` passes the reference-script size to `getMinFeeTx` using `txNonDistinctRefScriptsSize`:

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
``` [2](#0-1) 

`txNonDistinctRefScriptsSize` only unions the **top-level** `referenceInputsTxBodyL` and `inputsTxBodyL`:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

Sub-transactions are never visited here.

**The correct batch-wide function exists but is unused in fee validation**

`batchNonDistinctRefScriptsSize` correctly sums reference script sizes across the top-level transaction and all sub-transactions:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [4](#0-3) 

This function is defined but never called from `getMinFeeTxUtxo` or any fee-validation rule.

**Fee check in `dijkstraUtxoTransition` uses the incomplete function**

The Dijkstra UTXO rule enforces the minimum fee via:

```haskell
{- minfee pp txTop utxo₀ ≤ txfee txb -}
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
``` [5](#0-4) 

`validateFeeTooSmallUTxO` calls `getMinFeeTxUtxo pp tx utxo`, which resolves to `getConwayMinFeeTxUtxo` — the top-level-only version. [6](#0-5) 

**Sub-transactions do have reference inputs that are processed**

`DijkstraSubTxBodyRaw` carries its own `dstbrReferenceInputs` field:

```haskell
DijkstraSubTxBodyRaw ::
  { dstbrSpendInputs    :: !(Set TxIn)
  , dstbrReferenceInputs :: !(Set TxIn)
  ...
``` [7](#0-6) 

The `SUBUTXO` rule validates those inputs against `originalUtxo` and calls `Shelley.updateUTxOStateNoFees`, meaning the scripts attached to those UTxO entries are deserialized and processed by every validating node:

```haskell
runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
``` [8](#0-7) 

**Exploit path**

1. Attacker creates UTxO entries containing large Plutus scripts (up to the per-transaction 200 KiB limit each).
2. Attacker constructs a top-level Dijkstra transaction with a minimal top-level body (small reference inputs → low `txNonDistinctRefScriptsSize`).
3. The top-level body embeds N sub-transactions, each referencing those large-script UTxO entries via `dstbrReferenceInputs`.
4. The fee check passes because only the top-level reference scripts are measured.
5. Every validating node must deserialize all sub-transaction reference scripts (potentially hundreds of KiB) to validate the batch, while the attacker pays only the fee for the top-level transaction's negligible reference scripts.

---

### Impact Explanation

**Medium — Attacker-controlled transactions modify fees outside design parameters.**

The `minFeeRefScriptCostPerByte` / tiered-pricing mechanism was introduced in Conway specifically to close the DDoS vector where large reference scripts could be included cheaply but were expensive to deserialize (see ADR-009). [9](#0-8) 

The Dijkstra era re-opens this vector for sub-transaction reference scripts: an attacker can force nodes to perform expensive deserialization work while paying a fee that does not reflect that work. This allows sustained, cheap-to-submit but expensive-to-validate transactions, degrading block validation throughput across all honest nodes.

---

### Likelihood Explanation

**Medium-High.** Any unprivileged transaction sender can craft this transaction. No special keys, governance access, or privileged roles are required. The attacker only needs:
- ADA to fund the top-level fee (which is artificially low due to the bug).
- Pre-existing UTxO entries containing large scripts (which the attacker can create themselves in prior transactions).

The Dijkstra era is experimental/upcoming, but the code path is present and the fee-validation gap is structural.

---

### Recommendation

Replace `getConwayMinFeeTxUtxo` with a Dijkstra-specific implementation that uses `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
-- In Cardano.Ledger.Dijkstra.UTxO
getDijkstraMinFeeTxUtxo ::
  ( EraTx era, BabbageEraTxBody era, DijkstraEraTxBody era ) =>
  PParams era -> Tx l era -> UTxO era -> Coin
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx

instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo  -- replaces getConwayMinFeeTxUtxo
```

This ensures the tiered reference-script fee applies to the entire batch, consistent with the Conway-era DDoS mitigation.

### Proof of Concept

```
-- Setup: create a UTxO entry with a large (e.g. 50 KiB) Plutus script S.
-- txIn_large_script points to that UTxO.

-- Top-level transaction (minimal reference scripts → low fee):
topTx = DijkstraTxBody
  { dtbSpendInputs        = { txIn_funding }
  , dtbReferenceInputs    = {}          -- no large scripts at top level
  , dtbTxfee              = minFeeForSmallTx  -- passes validateFeeTooSmallUTxO
  , dtbSubTransactions    = OMap.fromList
      [ subTx1   -- dstbReferenceInputs = { txIn_large_script }
      , subTx2   -- dstbReferenceInputs = { txIn_large_script }
      , ...      -- repeat N times
      ]
  , ...
  }

-- Result:
--   Fee paid   = minFeeForSmallTx  (top-level only, ~0 ref-script cost)
--   Work done  = N × deserialize(50 KiB script)  (all sub-tx ref scripts)
--   Fee deficit = N × tierRefScriptFee(50_000) - 0
```

Each validating node must deserialize the large script N times (once per sub-transaction reference), while the attacker pays only the base transaction fee. With N sub-transactions and a 50 KiB script, the attacker avoids paying `N × tierRefScriptFee(50_000)` lovelace in reference-script fees, which at current parameters (`minFeeRefScriptCostPerByte = 15`) amounts to roughly `N × 750_000` lovelace of unpaid fee per batch.

```markdown
It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. Unfortunately things did not go exactly as planned, because on [June 25th 2024 an attack like this actually took place](https://cardanospot.io/news/ddos-attack-on-the-cardano-blockchain-mRIKAzZTNnzq5NGd). This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
