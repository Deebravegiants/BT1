### Title
Attacker Can DoS Governance Proposals via Treasury Donation Frontrunning - (File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs)

### Summary
The `validateTreasuryValue` function in the Conway/Dijkstra LEDGER rule performs an **exact equality check** between the `currentTreasuryValue` field embedded in a transaction body and the actual live treasury balance. Any unprivileged user can frontrun a governance proposal transaction by submitting a treasury donation of as little as 1 lovelace, shifting the treasury balance by that amount and causing the governance proposal to be rejected with `ConwayTreasuryValueMismatch`. The proposer must rebuild and re-sign the transaction with the updated value, and the attacker can repeat the disruption indefinitely at minimal cost.

### Finding Description

In `conwayLedgerTransitionTRC`, for every valid transaction the ledger calls:

```haskell
runTest $ validateTreasuryValue txBody (chainAccountState ^. casTreasuryL)
```

The validation is:

```haskell
validateTreasuryValue :: ConwayEraTxBody era => TxBody l era -> Coin -> Test (ConwayLedgerPredFailure era)
validateTreasuryValue txBody actualTreasuryValue =
  case txBody ^. currentTreasuryValueTxBodyL of
    SNothing -> pure ()
    SJust submittedTreasuryValue ->
      failureUnless (submittedTreasuryValue == actualTreasuryValue) $
        ConwayTreasuryValueMismatch
          ( Mismatch
              { mismatchSupplied = submittedTreasuryValue
              , mismatchExpected = actualTreasuryValue
              }
          )
```

The `currentTreasuryValue` field (`currentTreasuryValueTxBodyL`) is the Conway-era mechanism by which a transaction author asserts the current treasury balance so that guardrail Plutus scripts can receive it as `txInfoCurrentTreasuryAmount` in their script context. The guardrail script example in the codebase (`ensureTreasuryReserve`) explicitly returns `False` when `txInfoCurrentTreasuryAmount` is `Nothing`, making the field **effectively mandatory** for any governance proposal whose guardrail script inspects the treasury balance.

The treasury balance can be changed by any unprivileged user at any time via a treasury donation (`treasuryDonationTxBodyL`). The donation amount can be as small as 1 lovelace. Because the check is strict equality (`==`), a donation of 1 lovelace is sufficient to invalidate any pending transaction that embeds the pre-donation treasury value.

**Attack path:**
1. Alice builds a governance proposal transaction (e.g., a `TreasuryWithdrawals` proposal) whose guardrail script checks the treasury balance. She reads the current treasury value `T` and sets `currentTreasuryValue = T` in the transaction body, then signs it.
2. Bob observes Alice's transaction in the public mempool.
3. Bob submits a transaction with `treasuryDonation = 1` lovelace. This is a valid, cheap transaction.
4. Bob's donation is applied first (or in the same block before Alice's transaction). The treasury becomes `T + 1`.
5. Alice's transaction is applied: `submittedTreasuryValue (T) ≠ actualTreasuryValue (T + 1)` → `ConwayTreasuryValueMismatch` → transaction rejected.
6. Alice must rebuild and re-sign with `currentTreasuryValue = T + 1`.
7. Bob donates another 1 lovelace. Repeat indefinitely.

This is structurally identical to the external report: a user-supplied value (the embedded treasury snapshot) is validated against a live ledger quantity (the actual treasury balance) using a strict equality check, and an unprivileged third party can shift that live quantity by a minimal amount to invalidate the check.

### Impact Explanation

**Medium.** An attacker can persistently block any governance proposal that relies on a guardrail script inspecting the treasury balance. Because the `currentTreasuryValue` field is part of the signed transaction body, each disruption forces the proposer to rebuild and re-sign the transaction, incurring repeated effort and fees. The attacker's cost per disruption is one transaction fee plus 1 lovelace. This falls within the allowed Medium impact: *"Attacker-controlled transactions… modify… treasury donations… outside design parameters."* The design intent of treasury donations is to fund the treasury, not to weaponize the exact-equality treasury snapshot check against governance participants.

### Likelihood Explanation

**Medium.** The attack requires only a standard Cardano node with mempool visibility and the ability to submit transactions. No privileged access, key compromise, or consensus majority is needed. The attacker can sustain the disruption indefinitely at negligible cost. The proposer has no on-chain recourse other than using a private mempool (if available) or waiting for a block in which no donation precedes their transaction.

### Recommendation

Replace the strict equality check in `validateTreasuryValue` with a check that accepts any `submittedTreasuryValue` that is **at most** the actual treasury value (i.e., `submittedTreasuryValue <= actualTreasuryValue`). This preserves the security property that guardrail scripts cannot be fed an inflated treasury figure (which would allow bypassing a minimum-reserve guardrail), while eliminating the DoS vector from small donations. Alternatively, pass the actual treasury value directly to the script context regardless of what the transaction body asserts, removing the need for the field entirely.

### Proof of Concept

```
Initial state: treasury = 5_000_000 lovelace

Step 1 – Alice builds governance proposal tx:
  currentTreasuryValue = SJust (Coin 5_000_000)
  proposalProcedures   = [TreasuryWithdrawals ...]
  (tx is signed and broadcast to mempool)

Step 2 – Bob observes Alice's tx in mempool, submits:
  treasuryDonation = Coin 1
  (Bob's tx is applied first; treasury becomes 5_000_001)

Step 3 – Alice's tx is applied:
  validateTreasuryValue:
    submittedTreasuryValue = Coin 5_000_000
    actualTreasuryValue    = Coin 5_000_001
    5_000_000 ≠ 5_000_001  →  ConwayTreasuryValueMismatch
  → Alice's governance proposal is REJECTED

Step 4 – Alice rebuilds with currentTreasuryValue = SJust (Coin 5_000_001), re-signs.
Step 5 – Bob donates 1 lovelace again. Repeat from Step 3.
```
