### Title
Insufficient Rounding Tolerance in `_checkAaveHealth()` Permanently Blocks `collectInterestToTreasury()` After Multiple Deposits — (`contracts/LRTWithdrawalManager.sol`)

---

### Summary

`_checkAaveHealth()` uses a hardcoded 2-wei tolerance to accept `aaveBalance < totalETHDepositedToAave`. Aave v3's scaled-balance accounting introduces up to 1 wei of rounding loss **per deposit**. After three or more deposits, the cumulative rounding deficit exceeds 2 wei, causing `_checkAaveHealth()` to return `false` and `collectInterestToTreasury()` to revert with `AaveHealthCheckFailed` on every call — permanently blocking yield collection through the normal operator path.

---

### Finding Description

`_checkAaveHealth()` compares the live aWETH balance against the contract-tracked `totalETHDepositedToAave`:

```solidity
// contracts/LRTWithdrawalManager.sol : 925-933
function _checkAaveHealth() internal view returns (bool healthy) {
    if (!isAaveIntegrationEnabled) return false;
    uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
    uint256 principal = totalETHDepositedToAave;
    // Allow small rounding differences (up to 2 wei)
    if (principal > aaveBalance && principal - aaveBalance > 2) return false;
    return true;
}
``` [1](#0-0) 

`totalETHDepositedToAave` is incremented by the exact ETH amount on every deposit:

```solidity
// contracts/LRTWithdrawalManager.sol : 897-898
aaveWETHGateway.depositETH{ value: amount }(aavePool, address(this), 0);
totalETHDepositedToAave += amount;
``` [2](#0-1) 

Aave v3 aTokens use a scaled-balance model: `scaledBalance = deposit / liquidityIndex` (integer division, rounded down), and `balanceOf = scaledBalance * liquidityIndex` (rounded down again). Each deposit can therefore produce a `balanceOf` that is **1 wei less** than the deposited amount. With N deposits, the cumulative deficit is up to N wei. Once N ≥ 3, `principal - aaveBalance > 2` becomes true and the health check fails.

`collectInterestToTreasury()` gates on this check with a hard revert:

```solidity
// contracts/LRTWithdrawalManager.sol : 540-545
function collectInterestToTreasury() external nonReentrant onlyLRTOperator returns (uint256 interestAmount) {
    if (!_checkAaveHealth()) revert AaveHealthCheckFailed();
    return _collectInterestToTreasury();
}
``` [3](#0-2) 

There is no admin function to reset or adjust `totalETHDepositedToAave`. The only workarounds require the `LRT_MANAGER` role to call `setAaveIntegrationEnabled(false)` (which internally calls `_collectInterestToTreasury()` without the health check) or `emergencyWithdrawFromAave` (PAUSER_ROLE), both of which are disruptive and not the intended yield-collection path.

---

### Impact Explanation

The protocol's promised yield-collection mechanism (`collectInterestToTreasury`) is permanently blocked for the operator role after a small number of normal deposits. Accrued interest sits in Aave and cannot be routed to the treasury via the standard path. No user funds are lost, but the contract fails to deliver its promised yield-forwarding function. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

The Aave integration is designed to receive ETH from multiple `unlockQueue` calls and `depositIdleETHToAave` calls over time. Three deposits is a very low threshold — this condition is virtually guaranteed to be reached in normal production operation. No attacker action is required; it is a natural consequence of repeated use.

---

### Recommendation

Replace the hardcoded 2-wei constant with a tolerance that scales with the number of deposits, or track the cumulative rounding loss explicitly. A simple fix is to increase the tolerance to a value that accounts for realistic deposit counts (e.g., `totalETHDepositedToAave / 1e15` as a relative tolerance), or to remove the strict health check from `collectInterestToTreasury()` entirely and instead emit a warning event when `aaveBalance < principal`, reserving the hard revert only for cases where the deficit exceeds a meaningful threshold (e.g., 1% of principal).

---

### Proof of Concept

```solidity
// Pseudocode for a local Foundry fork test
// Setup: deploy LRTWithdrawalManager with mock aWETH that returns
//        balanceOf = deposited - 1 per deposit (simulating Aave rounding)

// Deposit 1: totalETHDepositedToAave = 1e18, aaveBalance = 1e18 - 1  (deficit: 1)
manager.depositIdleETHToAave(1e18);

// Deposit 2: totalETHDepositedToAave = 2e18, aaveBalance = 2e18 - 2  (deficit: 2)
manager.depositIdleETHToAave(1e18);

// Deposit 3: totalETHDepositedToAave = 3e18, aaveBalance = 3e18 - 3  (deficit: 3 > 2)
manager.depositIdleETHToAave(1e18);

// Now simulate interest accrual: aaveBalance = 3e18 - 3 + 1e15 (interest)
// _checkAaveHealth(): principal(3e18) - aaveBalance(3e18 - 3 + 1e15) = 3 - 1e15
// Wait — principal > aaveBalance? No, aaveBalance > principal here.
// Correct scenario: interest has NOT yet accrued, or deficit > interest
// aaveBalance = 3e18 - 3, principal = 3e18 → deficit = 3 > 2 → returns false

bool healthy = manager.aaveHealthCheck(); // returns false
vm.expectRevert(AaveHealthCheckFailed.selector);
manager.collectInterestToTreasury(); // reverts
```
