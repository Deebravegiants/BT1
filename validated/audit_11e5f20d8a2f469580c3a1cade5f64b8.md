### Title
`respond_verify_foreign_tx` accepts caller-supplied `payload_hash` without binding it to the pending request — (File: `crates/contract/src/lib.rs`)

---

### Summary

`MpcContract::respond_verify_foreign_tx` verifies that the ECDSA signature in the response is valid for `response.payload_hash` under the domain's root public key, but never checks that `response.payload_hash` is derived from the `ForeignChainRpcRequest` stored in the pending-request map for the supplied `request` key. A single Byzantine participant (strictly below the signing threshold) can replay any previously observed valid `(payload_hash, signature)` pair against a different, currently-pending `VerifyForeignTransactionRequest`, causing the contract to deliver a forged verification result to the waiting caller.

---

### Finding Description

**Root cause — missing request-binding check in `respond_verify_foreign_tx`**

In `respond` (the plain signing path), the hash that is verified is taken directly from the stored request:

```rust
// crates/contract/src/lib.rs  line 600
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,          // ← comes from the pending-request map entry
    &expected_public_key,
)
```

The payload is fixed by what the user originally submitted; a node cannot substitute a different hash.

In `respond_verify_foreign_tx`, the hash is taken from the **caller-supplied response**, not from the pending-request map:

```rust
// crates/contract/src/lib.rs  lines 726-734
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← fully attacker-controlled

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← no check that this matches request.request
    &secp_pk,
)
.is_ok()
```

The contract then resolves **all** yields queued under the supplied `request` key with the full `response` (including the attacker-chosen `payload_hash`):

```rust
// crates/contract/src/lib.rs  lines 749-753
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),   // ← forged payload_hash delivered verbatim
)
```

The canonical signed payload is defined as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }))
```

The `ForeignChainRpcRequest` (tx_id, extractors, finality) is included in the hash, so signatures for different foreign transactions produce different hashes. However, the contract never verifies that `response.payload_hash` is consistent with `request.request` (the `ForeignChainRpcRequest` stored in the pending map). There is no NEAR-request-ID or receipt-ID binding in `ForeignTxSignPayload` either, so a hash produced for one NEAR request is cryptographically indistinguishable from a hash produced for any other NEAR request that happens to share the same `ForeignChainRpcRequest`.

---

### Impact Explanation

A single Byzantine participant can:

1. Observe a legitimately produced `(H₁, sig_H₁)` pair from a prior `respond_verify_foreign_tx` call for request R₁ (all on-chain data is public).
2. While a different request R₂ (different `ForeignChainRpcRequest`, same `domain_id`) is pending, call `respond_verify_foreign_tx(request=R₂, response={payload_hash=H₁, signature=sig_H₁})`.
3. The contract passes the signature check (sig_H₁ is valid for H₁ under the domain root key) and resolves every yield queued under R₂ with the forged response.
4. Every caller waiting on R₂ receives `{payload_hash=H₁, signature=sig_H₁}` — a signed attestation that appears to cover R₂'s foreign transaction but actually covers R₁'s.

Because the `extracted_values` are determined by the MPC nodes (not the user), the user cannot reconstruct the expected `payload_hash` without those values. They must trust the hash returned by the contract. A downstream bridge contract that verifies only the ECDSA signature (not the hash-to-request binding) will accept the forged attestation and may release funds or execute a cross-chain action based on incorrect foreign-chain state.

This matches the allowed impact: **forged foreign-chain verification that causes invalid bridge execution or double-spend conditions**.

---

### Likelihood Explanation

- The attacker is a **single** attested MPC participant — strictly below the signing threshold.
- No key material needs to be leaked; the attacker simply reads a prior `respond_verify_foreign_tx` transaction from the public NEAR blockchain.
- The attack window is any time a second, different `VerifyForeignTransactionRequest` is pending in the contract for the same domain.
- No coordination with other participants is required.

---

### Recommendation

Bind the signed payload to the specific NEAR request by including the `VerifyForeignTransactionRequest` (or at minimum its Borsh hash / the NEAR receipt ID) inside `ForeignTxSignPayloadV1`:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub near_request: VerifyForeignTransactionRequest,  // ← add this
    pub values: Vec<ExtractedValue>,
}
```

With this change, the contract can recompute the minimum expected prefix of the hash from the stored request key and reject any response whose `payload_hash` could not have been produced for that specific request. Alternatively, store the `receipt_id` (already available in the node's `VerifyForeignTxRequest`) in the payload so the contract can verify it against the yield's data-id lineage.

---

### Proof of Concept

```
1. User A calls verify_foreign_transaction({domain_id=D, request=BTC_TX_1}).
   → R1 is queued; MPC nodes sign H1 = SHA-256(borsh({BTC_TX_1, [block_hash_A]})).
   → Honest node calls respond_verify_foreign_tx(R1, {payload_hash=H1, sig=sig_H1}).
   → H1 and sig_H1 are now permanently visible on-chain.

2. User B calls verify_foreign_transaction({domain_id=D, request=BTC_TX_2}).
   → R2 is queued (different tx_id → different map key).

3. Malicious participant calls:
     respond_verify_foreign_tx(
       request = R2,                              // valid pending key
       response = { payload_hash = H1,            // replayed from step 1
                    signature   = sig_H1 }        // replayed from step 1
     )

4. Contract check (lib.rs:729-734):
     verify_ecdsa_signature(sig_H1, H1, root_pk_D) → OK  ✓
   No check that H1 is derived from BTC_TX_2.

5. resolve_yields_for(R2, serialize({payload_hash=H1, sig=sig_H1}))
   → User B's yield is resumed with the forged response.

6. User B receives {payload_hash=H1, sig=sig_H1}.
   Downstream bridge verifies sig_H1 over H1 → valid.
   Bridge believes MPC attested BTC_TX_2 with block_hash_A,
   but the attestation actually covers BTC_TX_1.
