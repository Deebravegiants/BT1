### Title
Missing Deposit Refund for Cleared Proposals in `do_update` â€” (`File: crates/contract/src/update.rs`)

### Summary

When `do_update` executes a winning contract/config update, it unconditionally clears **all** pending proposals via `self.entries.clear()`. Every other proposer paid a storage-staking deposit via `propose_update`, but `do_update` never refunds those deposits. The funds are permanently locked inside the MPC contract with no recovery path.

### Finding Description

`propose_update` enforces a deposit proportional to the storage consumed by the proposal (contract binary bytes + vote-slot overhead): [1](#0-0) 

The deposit is accepted and stored in the contract's balance. Only the excess above `required` is refunded to the proposer: [2](#0-1) 

When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry and then **clears every other pending entry and every vote** with no refund: [3](#0-2) 

`do_update` contains no `Promise::new(proposer).transfer(deposit)` call for the cleared entries. The storage bytes are freed (so the contract's storage obligation shrinks), but the NEAR tokens that were deposited to cover that storage remain in the contract's balance permanently.

The analog to the Rubicon bug is exact: just as `maxSellAllAmount` was missing the `safeTransferFrom` that should have moved funds before the external call, `do_update` is missing the deposit-refund transfers that should accompany the `entries.clear()` call. In both cases a prerequisite action (moving funds to the right party) is absent, causing a permanent accounting discrepancy.

### Impact Explanation

Every participant who proposed an update that is not the winning one loses their full deposit. For a contract-binary proposal the deposit is `env::storage_byte_cost() Ă— (sizeof(UpdateEntry) + 128Ă—sizeof(AccountId) + code.len())`. A 1 MB WASM binary at the current NEAR storage price (~10 yoctoNEAR/byte) costs roughly **10 NEAR** per proposal. Multiple concurrent proposals (the codebase explicitly tests and supports this scenario) multiply the loss. [4](#0-3) 

The locked funds are controlled by the MPC contract and cannot be recovered by any existing contract method, satisfying the "permanent freezing of funds controlled by the MPC network" criterion.

### Likelihood Explanation

The trigger is a completely normal governance operation: any threshold-reaching `vote_update` call. No collusion, no privileged access, and no adversarial input is required. Concurrent proposals are explicitly supported and tested: [5](#0-4) 

Any participant who proposes an update while another proposal is already pending (or proposes one that loses the vote race) will have their deposit permanently locked the moment the winning update executes.

### Recommendation

Before calling `self.entries.clear()`, iterate over the remaining entries and refund each proposer's deposit. The `UpdateEntry` struct should store the proposer's `AccountId` and the exact deposit amount paid (mirroring the `PendingAttestation` pattern already used in the attestation flow):

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;

    // Refund deposits for all proposals that will be cleared
    for (_, cleared_entry) in self.entries.iter() {
        if let Some(proposer) = &cleared_entry.proposer {
            Promise::new(proposer.clone())
                .transfer(cleared_entry.deposit_paid)
                .detach();
        }
    }

    self.entries.clear();
    self.vote_by_participant.clear();
    // ... rest of do_update
}
```

`UpdateEntry` must be extended to record `proposer: AccountId` and `deposit_paid: NearToken` at `propose` time, and `propose_update` in `lib.rs` must pass those values through.

### Proof of Concept

1. Participant Alice calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR deposit. `entries` now contains `{id=0 â†’ entry_A}`.
2. Participant Bob calls `propose_update` with a config change, attaching a smaller deposit. `entries` now contains `{id=0 â†’ entry_A, id=1 â†’ entry_B}`.
3. Threshold participants vote for `id=1` (Bob's proposal). `vote_update` calls `do_update(&1, gas)`.
4. `do_update` removes entry 1, then calls `self.entries.clear()` â€” entry 0 (Alice's) is deleted with no refund.
5. Alice's ~10 NEAR is permanently locked in the MPC contract. No contract method exists to recover it. [6](#0-5) [7](#0-6)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
