### Title
`RequireGuard` Native Script Satisfiable Without Credential Signature, Enabling Unauthorized UTxO Spending - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`)

---

### Summary

In the Dijkstra era, the `RequireGuard` native script evaluates to `True` whenever the required credential appears in the transaction's `guards` field. For key-hash credentials (`KeyHashObj`), the ledger imposes **no witness (signature) requirement** on entries in the `guards` field. Any unprivileged transaction author can therefore insert an arbitrary key-hash credential into `guards` and satisfy a `RequireGuard` script that was intended to gate access to that credential's holder — enabling unauthorized spending of UTxOs, minting of tokens, or execution of certificates locked by such scripts.

---

### Finding Description

`evalDijkstraNativeScript` evaluates `RequireGuard cred` as a simple set-membership check:

```haskell
RequireGuard cred -> cred `OSet.member` guards
``` [1](#0-0) 

Here `guards :: OSet (Credential Guard)` is taken directly from the transaction body field `guardsTxBodyL`, which the transaction author populates freely. [2](#0-1) 

For `ScriptHashObj` guards the ledger correctly enforces that the referenced script is provided as a witness and must validate. For `KeyHashObj` guards, however, `getConwayWitsVKeyNeeded` — which Dijkstra inherits — never adds the guard key hash to the required-witness set:

```haskell
getConwayWitsVKeyNeeded utxo txBody =
  getShelleyWitsVKeyNeededNoGov utxo txBody
    `Set.union` Set.map asWitness (txBody ^. reqSignerHashesTxBodyG)
    `Set.union` voterWitnesses txBody
``` [3](#0-2) 

The `guards` field is absent from this computation. No Dijkstra-era override adds it. The existing test confirms the consequence: a UTxO locked by `RequireGuard (KeyHashObj freshKeyHash)` is successfully spent by a transaction that merely sets `guardsTxBodyL .~ [guardKeyHash]` with no corresponding signature in `addrTxWitsL`:

```haskell
submitTx_ $ tx & bodyTxL . guardsTxBodyL .~ [guardKeyHash]
``` [4](#0-3) 

The asymmetry is explicit in the test suite: script-hash guards require the script to be witnessed and to validate (lines 40–53), while key-hash guards require nothing beyond their presence in the `guards` field. [5](#0-4) 

The `DijkstraNativeScriptRaw` constructor that encodes this guard type is:

```haskell
DijkstraRequireGuard (Credential Guard)
``` [6](#0-5) 

---

### Impact Explanation

Any UTxO, minting policy, certificate, or governance action protected solely by `RequireGuard (KeyHashObj victim)` can be seized or executed by an attacker who knows `victim`'s key hash (public on-chain information). This constitutes a direct, unauthorized transfer of ADA or native assets — matching the **Critical: Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition** impact category.

---

### Likelihood Explanation

The attacker requires:
1. Knowledge of a key-hash credential `target` used in a deployed `RequireGuard` script (visible on-chain).
2. Construction of a transaction spending the locked UTxO with `guardsTxBodyL .~ [KeyHashObj target]`.
3. Submission of the transaction.

No private key material, privileged access, governance majority, or Sybil attack is required. The entry path is a standard unprivileged transaction submission.

---

### Recommendation

Add key-hash credentials from the `guards` field to the required-witness set in the Dijkstra `getWitsVKeyNeeded` implementation, analogous to how `voterWitnesses` adds voter key hashes for Conway:

```haskell
-- Dijkstra extension of getConwayWitsVKeyNeeded
getDijkstraWitsVKeyNeeded utxo txBody =
  getConwayWitsVKeyNeeded utxo txBody
    `Set.union` guardWitnesses txBody

guardWitnesses :: DijkstraEraTxBody era => TxBody l era -> Set (KeyHash Witness)
guardWitnesses txBody =
  foldr accum mempty (txBody ^. guardsTxBodyL)
  where
    accum cred khs = maybe khs (`Set.insert` khs) (credKeyHashWitness cred)
```

This mirrors the pattern already used for voter witnesses and ensures that `RequireGuard (KeyHashObj kh)` cannot be satisfied without a valid signature from the holder of `kh`.

### Proof of Concept

The existing test in `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/UtxowSpec.hs` (lines 30–38) demonstrates the issue directly. An attacker who does **not** hold the private key for `guardKeyHash` can replicate this pattern against any on-chain UTxO locked by `RequireGuard (KeyHashObj guardKeyHash)`:

```haskell

-- No witness for guardKeyHash is added; transaction succeeds:

evalDijkstraNativeScript ::
  (DijkstraEraScript era, NativeScript era ~ DijkstraNativeScript era) =>
  Set.Set (KeyHash Witness) ->
  ValidityInterval ->
  OSet (Credential Guard) ->
  NativeScript era ->
  Bool
evalDijkstraNativeScript keyHashes (ValidityInterval txStart txExp) guards = go
  where
    -- the evaluation will stop as soon as it reaches the required number of valid scripts
    isValidMOf n SSeq.Empty = n <= 0
    isValidMOf n (ts SSeq.:<| tss) =
      n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
    go = \case
      RequireTimeStart lockStart -> lockStart `lteNegInfty` txStart
      RequireTimeExpire lockExp -> txExp `ltePosInfty` lockExp
      RequireSignature hash -> hash `Set.member` keyHashes
      RequireAllOf xs -> all go xs
      RequireAnyOf xs -> any go xs
      RequireMOf m xs -> isValidMOf m xs
      RequireGuard cred -> cred `OSet.member` guards
      _ -> error "Impossible: All NativeScripts should have been accounted for"
```
