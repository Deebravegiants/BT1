### Title
`completeUnstaking(receiveAsTokens=false)` Decrements `uncompletedWithdrawalCount` Without Delivering Tokens to Vault — (`contracts/NodeDelegator.sol`)

---

### Summary

`NodeDelegator.completeUnstaking` exposes a three-argument `public` overload that accepts a caller-supplied `receiveAsTokens` flag. When an LRT Operator calls it with `receiveAsTokens=false`, EigenLayer re-credits shares back to the NDC's StrategyManager position instead of transferring tokens. The vault's `uncompletedWithdrawalCount` is still decremented unconditionally, desyncing the counter while no assets arrive in `LRTUnstakingVault`.

---

### Finding Description

`NodeDelegator` exposes two overloads of `completeUnstaking`:

- The one-argument version (line 336–338) hardcodes `receiveAsTokens=true` — the safe path.
- The three-argument version (lines 346–400) is `public onlyLRTOperator` and accepts any `receiveAsTokens` value. [1](#0-0) 

Inside the three-argument version, `decreaseUncompletedWithdrawalCount()` is called **unconditionally** at line 384, before the `if (receiveAsTokens)` branch that actually transfers tokens to the vault: [2](#0-1) 

When `receiveAsTokens=false`:
1. EigenLayer's `completeQueuedWithdrawal(..., false)` re-credits shares to the NDC's StrategyManager account — no tokens are sent anywhere.
2. `_getUnstakingVault().decreaseUncompletedWithdrawalCount()` fires, reducing the counter as if the withdrawal was fully serviced.
3. The `if (receiveAsTokens)` block is skipped — `LRTUnstakingVault` receives nothing. [3](#0-2) 

The result: `uncompletedWithdrawalCount` is lower than the actual number of pending EigenLayer withdrawals, and the vault holds no new assets to service queued user withdrawals that were unlocked against the expected incoming balance.

---

### Impact Explanation

Assets are not lost — they are re-credited as shares in the NDC's EigenLayer position. However, the vault's accounting is broken: `uncompletedWithdrawalCount` under-counts real pending withdrawals, and the vault balance does not increase. Users whose withdrawal requests were unlocked against the expected vault balance cannot complete their withdrawals until the count is manually corrected via `setUncompletedWithdrawalCount()`. [4](#0-3) 

This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

---

### Likelihood Explanation

The `onlyLRTOperator` modifier restricts the call to the LRT Operator role. This is a trusted but operationally active role (it also calls `initiateUnstaking`, `depositAssetIntoStrategy`, etc.). The three-argument overload is `public` with no NatSpec warning against `receiveAsTokens=false` for LST strategies. An operator could invoke it accidentally (e.g., intending to re-stake shares) or through a misconfigured script. No key compromise is required — only a valid operator address calling an existing, accessible function. [5](#0-4) 

---

### Recommendation

Add a guard in the three-argument `completeUnstaking` that rejects `receiveAsTokens=false` for non-beaconChain strategies, or move `decreaseUncompletedWithdrawalCount()` inside the `if (receiveAsTokens)` block so the counter is only decremented when tokens actually arrive in the vault:

```solidity
// Only decrement when tokens are actually delivered
if (receiveAsTokens) {
    _getUnstakingVault().decreaseUncompletedWithdrawalCount();
    // ... transfer loop ...
}
```

Alternatively, restrict the three-argument overload to `internal` and remove the `receiveAsTokens=false` path entirely for LST strategies, since the protocol has no documented use-case for re-crediting shares instead of withdrawing tokens.

---

    function setUncompletedWithdrawalCount() external onlyLRTManager {
        ILRTDepositPool lrtDepositPool = ILRTDepositPool(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        IDelegationManager delegationManager =
            IDelegationManager(lrtConfig.getContract(LRTConstants.EIGEN_DELEGATION_MANAGER));
        address[] memory nodeDelegatorQueue = lrtDepositPool.getNodeDelegatorQueue();
        uint256 totalQueued;
        for (uint256 i = 0; i < nodeDelegatorQueue.length; i++) {
            address nodeDelegator = nodeDelegatorQueue[i];
            (IDelegationManager.Withdrawal[] memory queuedWithdrawals,) =
                delegationManager.getQueuedWithdrawals(nodeDelegator);
            totalQueued += queuedWithdrawals.length;
        }

        uncompletedWithdrawalCount = totalQueued;

        emit UncompletedWithdrawalCountSet(totalQueued);
    }
