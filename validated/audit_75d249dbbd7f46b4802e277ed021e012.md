Audit Report

## Title
Lack of Slippage Protection in L2 Pool `deposit()` Functions Allows Users to Receive Fewer rsETH Tokens Than Quoted - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPoolNoWrapper.sol)

## Summary
All L2 pool `deposit()` functions compute the rsETH output amount from the live oracle rate at execution time but accept no `minRSETHAmountExpected` parameter. The mainnet `LRTDepositPool` explicitly enforces such a minimum in `_beforeDeposit()`. When the oracle rate is updated between a user's call to `viewSwapRsETHAmountAndFee` and the on-chain inclusion of their deposit transaction, the user silently receives fewer rsETH/wrsETH tokens than the quoted amount with no on-chain recourse.

## Finding Description
`LRTDepositPool.depositETH()` accepts a `minRSETHAmountExpected` argument and enforces it:

```solidity
// contracts/LRTDepositPool.sol L667-668
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

Every L2 pool `deposit()` function omits this guard. In `RSETHPoolV3`:

```solidity
// contracts/pools/RSETHPoolV3.sol L258-262
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
```

`viewSwapRsETHAmountAndFee` derives the output purely from the live oracle rate:

```solidity
// contracts/pools/RSETHPoolV3.sol L299-307
uint256 rsETHToETHrate = getRate();   // live oracle read
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because rsETH is yield-bearing, `getRate()` monotonically increases as EigenLayer staking rewards accrue. Any oracle update that lands before the user's deposit transaction will silently reduce the minted amount below the quoted value. The identical pattern is confirmed in `RSETHPoolV3ExternalBridge.deposit()` (L366-412), `RSETHPoolV3WithNativeChainBridge.deposit()` (L282-329), and `RSETHPoolNoWrapper.deposit()` (L231-271). Existing checks (`nonReentrant`, `whenNotPaused`, `limitDailyMint`, zero-amount guard) do not address this discrepancy.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The user's deposited ETH value is preserved in aggregate (fewer tokens, each worth proportionally more ETH), but the contract does not honour the token quantity the user was shown by the protocol's own view function. Users who rely on the quoted token amount for downstream operations (e.g., providing liquidity, collateralising a loan) may find themselves short of the expected balance with no on-chain protection and no revert.

## Likelihood Explanation
**Medium.** The rsETH oracle rate is updated on a regular cadence (at minimum daily, often more frequently on L2s). L2 chains with variable block times or sequencer congestion widen the window between transaction submission and inclusion. Any user who previews a quote via `viewSwapRsETHAmountAndFee` and then submits a deposit is exposed to this discrepancy. No attacker capability is required; normal protocol operation (reward accrual → oracle update) is sufficient to trigger it.

## Recommendation
Add a `minRSETHAmountExpected` parameter to every public `deposit()` overload in all L2 pool contracts, mirroring the pattern already used in `LRTDepositPool._beforeDeposit()`:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same change to the token-deposit overloads in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`.

## Proof of Concept
1. User calls `RSETHPoolV3.viewSwapRsETHAmountAndFee(1 ether)` at oracle rate `R` and receives quote `X` wrsETH.
2. User submits `RSETHPoolV3.deposit{value: 1 ether}("ref")`.
3. Before inclusion, the rsETH oracle is updated to rate `R' > R` (normal reward accrual — no attacker required).
4. Deposit executes: `rsETHAmount = (1 ether - fee) * 1e18 / R'`, strictly less than `X`.
5. User receives fewer wrsETH tokens than quoted; no revert occurs.

**Contrast:** calling `LRTDepositPool.depositETH{value: 1 ether}(X, "ref")` with the same quoted minimum would revert at step 4 via `MinimumAmountToReceiveNotMet`.

**Foundry fork test plan:**
- Fork an L2 where `RSETHPoolV3` is deployed.
- Call `viewSwapRsETHAmountAndFee(1 ether)` → record `expectedAmount`.
- `vm.prank(oracle_updater)` advance the oracle rate by a small increment (simulating one reward accrual).
- Call `deposit{value: 1 ether}("ref")` from a user address.
- Assert `wrsETH.balanceOf(user) < expectedAmount` — no revert, silent shortfall confirmed.
