### Title
`verify_foreign_transaction()` Gates on Intersection Rule Instead of Threshold+Whitelist, Enabling Single-Participant DoS on Whitelisted Chains — (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction()` uses `get_supported_foreign_chains()` — a strict intersection requiring **all** participants to register a chain — instead of `get_available_foreign_chains()` — which requires the chain to be whitelisted by governance vote **and** covered by ≥ signing-threshold participants. A single Byzantine participant strictly below the signing threshold can block all foreign-chain verification requests for any whitelisted chain by simply not registering it, breaking the production safety invariant. The inverse also holds: a non-whitelisted chain can be processed if all participants happen to register it, bypassing the governance whitelist entirely.

---

### Finding Description

In `verify_foreign_transaction()`, the chain-support gate is:

```rust
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(...)
}
``` [1](#0-0) 

`get_supported_foreign_chains()` computes the **strict intersection** of every active participant's registered chains:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
``` [2](#0-1) 

This means a chain is "supported" only if **every single active participant** has registered it. One participant omitting a chain drops it to zero availability.

The contract already implements the correct check — `get_available_foreign_chains()` — which requires a chain to be whitelisted (voted in by threshold) **and** covered by ≥ signing-threshold participants:

```rust
pub fn get_available_foreign_chains(&self) -> dtos::AvailableForeignChains {
    self.foreign_chains.get().available_foreign_chains.clone()
}
``` [3](#0-2) 

The design specification is unambiguous:

> `verify_foreign_transaction(C)` is **rejected unless `C` is available** … The legacy `get_supported_foreign_chains()` (the intersection rule) is **to be deprecated** in favour of the two views above. [4](#0-3) 

The design doc also explicitly names the root cause of the DoS:

> A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down. That is what this proposal fixes. [5](#0-4) 

**Inverse problem (governance bypass):** `get_supported_foreign_chains()` does not check the on-chain whitelist at all. If all participants register a chain that has never been voted into the whitelist, `verify_foreign_transaction` accepts the request and nodes process it using their local RPC configs — bypassing the governance model entirely.

---

### Impact Explanation

**Medium.** A Byzantine participant strictly below the signing threshold manipulates the request lifecycle by withholding its chain registration. Every `verify_foreign_transaction` call for the targeted whitelisted chain is rejected at the contract level, even though the chain is governance-approved and threshold-many participants cover it. This breaks the production safety invariant:

> `available ⊆ whitelisted` always; `verify_foreign_transaction(C)` is rejected unless `C` is available. [6](#0-5) 

The inverse impact — accepting requests for non-whitelisted chains — breaks the invariant that only governance-approved chains may receive MPC attestations, potentially enabling bridge flows over chains that have not been vetted by the network.

---

### Likelihood Explanation

**Medium.** The DoS path requires only one participant to omit a chain registration (or call `register_foreign_chain_config` with an empty list). This is a single, low-effort, on-chain transaction. No threshold collusion is needed. The governance-bypass path requires all participants to register a chain without a whitelist vote, which is plausible during a migration window or operator misconfiguration. Both paths are reachable by an unprivileged caller or a Byzantine participant below the signing threshold.

---

### Recommendation

Replace `self.get_supported_foreign_chains()` with `self.get_available_foreign_chains()` in `verify_foreign_transaction()`:

```rust
// Before (wrong):
let supported_chains = self.get_supported_foreign_chains();

// After (correct):
let supported_chains = self.get_available_foreign_chains();
```

`get_available_foreign_chains()` already exists, is tested, and enforces both the whitelist requirement and the threshold-coverage requirement. This is exactly the migration step described in the design document. [7](#0-6) 

---

### Proof of Concept

1. Network: 4 participants, signing threshold = 3. Chain C is whitelisted (voted in by threshold).
2. Participants P1, P2, P3 register chain C (threshold covered).
3. Byzantine participant P4 (below threshold) calls `register_foreign_chain_config` with an empty list.
4. `get_supported_foreign_chains()` computes the intersection: P4 did not register C → C is absent from the result.
5. Any user calling `verify_foreign_transaction(C)` hits:
   ```
   env::panic_str("ForeignChainNotSupported { requested: C }")
   ``` [8](#0-7) 
6. All requests for chain C are permanently rejected at the contract level. The chain is effectively offline despite being whitelisted and covered by threshold participants.
7. Recovery requires social coordination: P4 must re-register the chain, mirroring the OmniPortal report's recovery requirement.

The analog to the OmniPortal finding is exact: a wrong/missing check before accepting a state-modifying request (yield enqueue) allows a sub-threshold actor to corrupt the request-acceptance invariant, causing a DoS that requires out-of-band coordination to resolve.

## Migration

`get_supported_foreign_chains()` stays working throughout, so the new node version
can roll out before the contract upgrade (node and contract migrate independently):

1. Keep `get_supported_foreign_chains()` unchanged.
2. Add `get_whitelisted_foreign_chains()` and `get_available_foreign_chains()` (additive).
3. Vote the RPC providers / chains into the whitelist.
4. Upgrade the contract.
5. Switch node code to the new methods and deprecate `get_supported_foreign_chains()`.
