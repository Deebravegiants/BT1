Audit Report

## Title
`depositAsset()` / `depositETH()` Mint rsETH Against Stale Cached `rsETHPrice`, Enabling Theft of Unclaimed Yield — (File: contracts/LRTDepositPool.sol)

## Summary

`LRTOracle` stores `rsETHPrice` as a state variable updated only by explicit calls to `updateRSETHPrice()` or `updateRSETHPriceAsManager()`. Neither `depositAsset()` nor `depositETH()` triggers a price update before computing the rsETH mint amount. When protocol TVL has grown (e.g., stETH rebase) but the stored price has not yet been refreshed, a depositor receives more rsETH than their deposit is worth at the true current exchange rate, diluting existing holders and capturing their accrued yield.

## Finding Description

`LRTOracle.rsETHPrice` is a plain storage variable declared at line 28 of `contracts/LRTOracle.sol`. It is written only inside `_updateRsETHPrice()`, which is invoked by the public `updateRSETHPrice()` (guarded by `whenNotPaused`) or the manager-only `updateRSETHPriceAsManager()`. The deposit flow never calls either:

1. `depositAsset()` / `depositETH()` → `_beforeDeposit()` (declared `private view`, so it cannot make state-changing calls) → `getRsETHAmountToMint()`
2. `getRsETHAmountToMint()` at line 520 of `contracts/LRTDepositPool.sol` computes:
   ```
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```
   reading the cached `rsETHPrice` directly with no prior refresh.

When stETH rebases (every ~24 h) or EigenLayer rewards accumulate, `_getTotalEthInProtocol()` would return a higher value than `rsethSupply × rsETHPrice`, meaning the true price is above the stored value. Because `_beforeDeposit` is `view`, it structurally cannot call `updateRSETHPrice()`, and neither deposit entry point does so before invoking `_beforeDeposit`. The stale-low denominator inflates `rsethAmountToMint`, minting excess shares to the depositor. When `updateRSETHPrice()` is eventually called, the new price is computed over the enlarged supply, permanently diluting all pre-existing holders.

Additionally, `_updateRsETHPrice()` contains safety logic (auto-pause on price drop beyond `pricePercentageLimit`, fee minting, `highestRsethPrice` tracking) that is entirely bypassed during deposits.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders have an accrued but not-yet-reflected claim on the TVL growth (the "unclaimed yield"). A depositor who enters during the stale-price window receives rsETH computed against the lower cached price, obtaining more shares than their deposit warrants at the true rate. After the price is updated, those extra shares permanently dilute every prior holder's proportional claim on the TVL, transferring a portion of their accrued yield to the depositor. The loss to existing holders is bounded by the TVL growth since the last price update and the size of the deposit, but is repeatable on every rebase cycle.

## Likelihood Explanation

**Medium.** No special privileges are required — only a standard `depositAsset()` or `depositETH()` call. stETH rebases occur on a predictable ~24-hour schedule; EigenLayer rewards accrue continuously. A sophisticated actor can monitor the on-chain stETH balance held by NodeDelegators (public `balanceOf`) against the stored `rsETHPrice` to identify and time the maximum-spread window. The attack is repeatable every rebase cycle and requires no flash loans, governance access, or oracle manipulation.

## Recommendation

Refactor `depositAsset()` and `depositETH()` so that `updateRSETHPrice()` is called (or its logic is inlined) before `_beforeDeposit()` computes the mint amount. Because `_beforeDeposit` is currently `private view`, it must be changed to non-`view` (or the price-update call must be placed in the deposit entry points themselves, before the `_beforeDeposit` call). This ensures the price used for minting reflects current TVL, accrued rewards, fee minting, and any required safety pauses.

## Proof of Concept

**Setup (fork mainnet or local):**

1. Deploy protocol; call `updateRSETHPrice()`. Stored `rsETHPrice = 1.05e18`. Protocol TVL = 1050 ETH, rsETH supply = 1000.

2. Advance time past a stETH rebase. Protocol TVL grows to 1060 ETH (stETH balance of NodeDelegators increases). `rsETHPrice` in storage remains `1.05e18`. True price = `1060e18 / 1000 = 1.06e18`.

3. Attacker calls `depositAsset(stETH, 100e18, 0, "")`.
   - `getRsETHAmountToMint` computes: `(100e18 × 1e18) / 1.05e18 ≈ 95.238e18` rsETH minted.
   - At the true price `1.06e18`, the correct amount is `100e18 / 1.06e18 ≈ 94.340e18`.
   - Attacker receives **~0.898 rsETH extra**.

4. Keeper calls `updateRSETHPrice()`. New TVL = 1160 ETH, new supply = 1095.238 rsETH. New price = `1160 / 1095.238 ≈ 1.0591e18`.

5. Original 1000 rsETH holders' claim: `1000 × 1.0591 = 1059.1 ETH` — down from the 1060 ETH they were entitled to. The ~0.9 ETH difference was captured by the attacker's extra rsETH.

**Foundry test sketch:**
```solidity
function test_stalePrice_yieldTheft() public fork {
    // 1. record rsETHPrice after last update
    uint256 stalePriceBefore = lrtOracle.rsETHPrice();
    // 2. simulate stETH rebase (warp + deal stETH to NodeDelegator)
    vm.warp(block.timestamp + 1 days);
    deal(stETH, nodeDelegator, stETHBalance + rebaseAmount);
    // 3. deposit without calling updateRSETHPrice
    uint256 minted = depositPool.depositAsset(stETH, 100e18, 0, "");
    // 4. update price
    lrtOracle.updateRSETHPrice();
    uint256 correctMint = (100e18 * 1e18) / lrtOracle.rsETHPrice();
    // 5. assert attacker received more than correct amount
    assertGt(minted, correctMint);
}
```
