### Title
Missing Zero-Value Guard for `maxRefScriptSizePerBlock` / `maxRefScriptSizePerTx` in `DijkstraEra` `ppuWellFormed` — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/PParams.hs`)

---

### Summary

In the Dijkstra era, `maxRefScriptSizePerBlock` and `maxRefScriptSizePerTx` are promoted from hardcoded Conway constants to proper governance-updatable protocol parameters. The `ppuWellFormed` function for `DijkstraEra` validates that many critical parameters cannot be set to zero (e.g., `maxBBSize`, `maxTxSize`, `maxBHSize`), but it omits equivalent guards for these two new parameters. An unprivileged governance proposal author can therefore submit a well-formed `ParameterChange` proposal setting either limit to zero. If ratified, every transaction that references any reference script would be rejected by the ledger, permanently freezing all UTxOs locked by scripts that depend on reference inputs.

---

### Finding Description

**Conway era — hardcoded constants (not updatable):**

In `ConwayEra`, `ppMaxRefScriptSizePerTxG` and `ppMaxRefScriptSizePerBlockG` are implemented as compile-time constants:

```haskell
ppMaxRefScriptSizePerTxG   = L.to . const $ 200 * 1024   -- 200 KiB
ppMaxRefScriptSizePerBlockG = L.to . const $ 1024 * 1024  -- 1 MiB
``` [1](#0-0) 

Because they are constants, no governance proposal can change them, so no zero-value guard is needed in Conway.

**Dijkstra era — promoted to protocol parameters, guard omitted:**

In `DijkstraEra`, these become real protocol parameters stored in `DijkstraPParams`:

```haskell
dppMaxRefScriptSizePerBlock :: !(THKD ('PPGroups 'NetworkGroup 'SecurityGroup) f Word32)
dppMaxRefScriptSizePerTx    :: !(THKD ('PPGroups 'NetworkGroup 'SecurityGroup) f Word32)
``` [2](#0-1) 

The `ppuWellFormed` implementation for `DijkstraEra` validates many parameters against zero, but **does not include** `ppuMaxRefScriptSizePerBlockL` or `ppuMaxRefScriptSizePerTxL`:

```haskell
instance ConwayEraPParams DijkstraEra where
  ppuWellFormed _pv ppu =
    and
      [ isValid (/= 0) ppuMaxBBSizeL
      , isValid (/= 0) ppuMaxTxSizeL
      , isValid (/= 0) ppuMaxBHSizeL
      , isValid (/= 0) ppuMaxValSizeL
      , isValid (/= 0) ppuCollateralPercentageL
      ...
      -- ppuMaxRefScriptSizePerBlockL and ppuMaxRefScriptSizePerTxL are ABSENT
      ]
``` [3](#0-2) 

The `emptyDijkstraPParams` initialises both to zero:

```haskell
dppMaxRefScriptSizePerBlock = THKD 0
dppMaxRefScriptSizePerTx    = THKD 0
``` [4](#0-3) 

**Enforcement path that would break:**

`validateAllRefScriptSize` in the Dijkstra LEDGER rule reads `maxRefScriptSizePerTx` directly from the protocol parameters:

```haskell
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $
        DijkstraTxRefScriptsSizeTooBig ...
``` [5](#0-4) 

If `maxRefScriptSizePerTx = 0`, then `totalRefScriptSize <= 0` is false for any transaction that touches even one reference script, causing an unconditional `DijkstraTxRefScriptsSizeTooBig` failure.

Similarly, `validateBodyRefScriptsSizeTooBig` in the BBODY rule reads `ppMaxRefScriptSizePerBlockG`:

```haskell
maxSize = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerBlockG
in totalSize <= maxSize ?! injectFailure (BodyRefScriptsSizeTooBig ...)
``` [6](#0-5) 

If `maxRefScriptSizePerBlock = 0`, every block containing a reference-script transaction is rejected.

**`ppuWellFormed` is the ledger's gate for `ParameterChange` proposals:**

```haskell
actionWellFormed pv ga = failureUnless isWellFormed $ MalformedProposal ga
  where
    isWellFormed = case ga of
      ParameterChange _ ppd _ -> ppuWellFormed pv ppd
      _ -> True
``` [7](#0-6) 

Because `ppuWellFormed` for `DijkstraEra` does not reject zero values for these two parameters, a `ParameterChange` proposal setting them to zero passes `actionWellFormed` and is accepted into the proposal queue.

---

### Impact Explanation

**Allowed impact class: High — Permanent freezing of funds where recovery requires a hard fork.**

If a governance proposal setting `maxRefScriptSizePerTx = 0` or `maxRefScriptSizePerBlock = 0` is enacted:

- Every transaction that includes any reference input pointing to a UTxO with a reference script will be rejected at the LEDGER level (`DijkstraTxRefScriptsSizeTooBig`).
- Every block containing such a transaction will be rejected at the BBODY level (`BodyRefScriptsSizeTooBig`).
- All UTxOs locked by Plutus scripts that are spent via reference inputs become permanently unspendable.
- Recovery requires a hard fork to restore the parameter to a non-zero value, because the enactment itself is a valid ledger state transition.

---

### Likelihood Explanation

**Low.** Enacting the proposal requires governance ratification (DRep threshold, SPO threshold, and committee approval). However, the ledger-level defect — accepting the proposal as well-formed — is reachable by any unprivileged governance proposal author. The missing guard in `ppuWellFormed` is the necessary vulnerable step: without it, the ledger cannot distinguish a safe proposal from a destructive one at submission time, and the proposal enters the ratification queue where social-engineering or coordination failures could lead to enactment.

---

### Recommendation

Add non-zero guards for the two new Dijkstra parameters inside `ppuWellFormed` for `DijkstraEra`, mirroring the existing pattern for `maxBBSize`, `maxTxSize`, etc.:

```haskell
, isValid (/= 0) ppuMaxRefScriptSizePerBlockL
, isValid (/= 0) ppuMaxRefScriptSizePerTxL
```

This ensures any `ParameterChange` proposal attempting to set either limit to zero is rejected with `MalformedProposal` at submission time, before it can enter the ratification queue.

---

### Proof of Concept

1. In Dijkstra era, construct a `ParameterChange` governance action:
   ```haskell
   ppu = emptyPParamsUpdate
           & ppuMaxRefScriptSizePerTxL .~ SJust 0
   ga  = ParameterChange SNothing ppu SNothing
   ```
2. Submit the proposal. `actionWellFormed` calls `ppuWellFormed _pv ppu`. Because `ppuWellFormed` for `DijkstraEra` has no check for `ppuMaxRefScriptSizePerTxL`, the proposal is accepted (no `MalformedProposal` failure).
3. Once ratified and enacted, `pp ^. ppMaxRefScriptSizePerTxG` returns `0`.
4. Any subsequent transaction with `totalRefScriptSize > 0` hits `failureUnless (totalRefScriptSize <= 0)` in `validateAllRefScriptSize` and is rejected with `DijkstraTxRefScriptsSizeTooBig`.
5. All UTxOs locked by scripts requiring reference inputs are permanently frozen.
