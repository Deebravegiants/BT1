### Title
Sub-Transaction ExUnits Bypass `maxTxExUnits` Limit and Fee Calculation in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, the per-transaction execution-unit limit check (`maxTxExUnits`) and the minimum-fee calculation both rely on `totExUnits`, which only sums redeemers from the **top-level transaction's witnesses**. Sub-transactions embedded in the top-level transaction body each carry their own witnesses and redeemers with independent ExUnits budgets, but those budgets are never aggregated into the limit check or the fee. An unprivileged transaction author can therefore declare arbitrarily large total ExUnits across sub-transactions while keeping the top-level redeemer budget at zero, bypassing the protocol-parameter cap and underpaying the script-execution fee.

---

### Finding Description

**Root cause — `totExUnits` ignores sub-transaction redeemers**

`totExUnits` is defined in `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It reads only `tx ^. witsTxL` — the top-level transaction's witness set. Sub-transactions live in `tx ^. bodyTxL . subTransactionsTxBodyL` and each carries its own `TxWits` with its own `Redeemers` map. Those redeemers are invisible to `totExUnits`. [1](#0-0) 

**The limit check in the Dijkstra UTXO rule**

`dijkstraUtxoTransition` calls the Alonzo validator unchanged:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

`validateExUnitsTooBigUTxO` computes `totalExUnits = totExUnits tx`, so it only sees the top-level redeemers. [2](#0-1) [3](#0-2) 

**Sub-transactions provably carry redeemers**

`validateBatchCollateral` explicitly iterates sub-transactions to detect redeemers:

```haskell
hasAnyRedeemers t =
  hasRedeemers t || any hasRedeemers (t ^. bodyTxL . subTransactionsTxBodyL)
hasRedeemers = not . null . (^. witsTxL . rdmrsTxWitsL . unRedeemersL)
```

This confirms the ledger itself treats sub-transaction redeemers as real, script-executing redeemers — yet they are excluded from the ExUnits cap. [4](#0-3) 

**Fee calculation also uses `totExUnits`**

`alonzoMinFeeTx` (the base for all era minimum-fee calculations) computes the script-execution fee component as:

```haskell
txscriptfee (pp ^. ppPricesL) allExunits
  where allExunits = totExUnits tx
```

Sub-transaction ExUnits are therefore also excluded from the fee, meaning the transaction pays less than the cost of the scripts it causes to execute. [5](#0-4) 

**Block-level check is similarly incomplete**

`validateExUnits` in the Alonzo BBODY rule aggregates per-transaction totals with `foldMap totExUnits txs`. Because `totExUnits` ignores sub-transactions, the block-level `maxBlockExUnits` cap is also bypassable. [6](#0-5) 

---

### Impact Explanation

**Medium — Attacker-controlled transactions exceed intended validation limits and modify fees outside design parameters.**

1. **ExUnits cap bypass**: A top-level transaction with zero redeemers passes `validateExUnitsTooBigUTxO`. Each embedded sub-transaction can independently declare up to `maxTxExUnits` ExUnits. With N sub-transactions the effective execution budget is `N × maxTxExUnits`, far exceeding the protocol-parameter intent.

2. **Fee underpayment**: Because `totExUnits` omits sub-transaction redeemers, `minfee` does not include the `txscriptfee` component for those scripts. The transaction is accepted with a fee that does not cover the actual script-execution cost, violating the economic invariant that script execution is fully paid for by the submitter.

3. **Block-level cap bypass**: The same gap in `foldMap totExUnits txs` means a block can carry far more aggregate script work than `maxBlockExUnits` is intended to allow, potentially degrading block-validation performance.

---

### Likelihood Explanation

The Dijkstra era is the only era that introduces sub-transactions. Any unprivileged user who can submit a Dijkstra transaction can exploit this: they simply embed sub-transactions with large redeemer ExUnits budgets while keeping the top-level redeemer map empty. No special privilege, key, or governance action is required.

---

### Recommendation

1. **Aggregate ExUnits across the batch before the limit check.** Define a `batchTotExUnits` that sums `totExUnits` over the top-level transaction and all sub-transactions, and use it in `validateExUnitsTooBigUTxO` for the Dijkstra UTXO rule.

2. **Include sub-transaction ExUnits in the minimum-fee calculation.** Override `getMinFeeTxUtxo` for the Dijkstra era so that `txscriptfee` is computed over the full batch ExUnits.

3. **Apply the same fix to the block-level check.** The Dijkstra BBODY rule (or its inherited Alonzo/Conway variant) should use a batch-aware ExUnits aggregation when computing the block total.

---

### Proof of Concept

A Dijkstra top-level transaction is constructed with:
- Empty `rdmrsTxWitsL` in the top-level witnesses → `totExUnits tx = ExUnits 0 0` → passes `validateExUnitsTooBigUTxO`.
- N sub-transactions each with `rdmrsTxWitsL` containing a redeemer with `ExUnits maxMem maxSteps` → each sub-transaction's scripts are executed with the full budget.

The total execution budget consumed is `N × ExUnits maxMem maxSteps`, which can be made arbitrarily large. The fee paid covers only the top-level `ExUnits 0 0`, so the script-execution fee component is zero regardless of how many sub-transaction scripts run.

validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) $
    ExUnitsTooBigUTxO Mismatch {mismatchSupplied = totalExUnits, mismatchExpected = maxTxExUnits}
  where

-- | Validate that total execution units (all transactions) do not exceed block limit.
-- ∑(tx ∈ txs)(totExunits tx) ≤ maxBlockExUnits pp
validateExUnits ::
  forall era.
  ( AlonzoEraTx era
  , InjectRuleFailure "BBODY" AlonzoBbodyPredFailure era
  ) =>
  StrictSeq.StrictSeq (Tx TopTx era) ->
  -- | Max block exunits protocol parameter.
  ExUnits ->
  Rule (EraRule "BBODY" era) 'Transition ()
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
   in pointWiseExUnits (<=) txTotal ppMax
        ?! injectFailure
          ( TooManyExUnits $
              Mismatch
                { mismatchSupplied = txTotal
                , mismatchExpected = ppMax
                }
          )
