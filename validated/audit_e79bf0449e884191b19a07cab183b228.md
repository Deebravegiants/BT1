Audit Report

## Title
`RSETHPriceFeed.getRoundData` Returns Corrupted Historical Prices by Mixing Historical ETH/USD with Current rsETH/ETH Rate - (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary
`RSETHPriceFeed` implements `AggregatorV3Interface` and is deployed as a production Chainlink-compatible RSETH/USD feed. Its `getRoundData` function retrieves the historical ETH/USD price for a given `_roundId` but always multiplies by the **current** rsETH/ETH rate from `RS_ETH_ORACLE.rsETHPrice()`, producing a fabricated answer that is neither the historical nor the current RSETH/USD price. Any external protocol that calls `getRoundData` for historical deviation or staleness validation will receive corrupted data, potentially causing incorrect liquidations or temporary freezing of rsETH-collateralized positions.

## Finding Description
In `contracts/oracles/RSETHPriceFeed.sol` lines 53–61:

```solidity
function getRoundData(uint80 _roundId)
    external view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.getRoundData(_roundId);
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
}
```

`ETH_TO_USD.getRoundData(_roundId)` correctly returns the historical ETH/USD price for round `_roundId`. However, `RS_ETH_ORACLE` is `IRSETHOracle`, which exposes only `rsETHPrice() external view returns (uint256)` — a single storage slot (`LRTOracle.rsETHPrice`) that holds only the **current** rate, updated by `updateRSETHPrice()`. There is no historical rsETH/ETH rate storage or lookup mechanism anywhere in the codebase. The result is:

```
returned_answer = current_rsETH/ETH × historical_ETH/USD
```

instead of the correct:

```
correct_answer = historical_rsETH/ETH × historical_ETH/USD
```

`latestRoundData` (lines 63–70) is unaffected because both components are current there. The contract is confirmed deployed at `0x4B9C66c2C0d3706AabC6d00D2a6ffD2B68A4E383` (RSETHPriceFeed Morph) per the README, confirming production use.

## Impact Explanation
**Medium — Temporary freezing of funds.** `RSETHPriceFeed` is explicitly deployed for consumption by external DeFi protocols as a Chainlink-compatible feed. Any integrating protocol that calls `getRoundData` to perform historical round deviation checks (a standard Chainlink integration pattern) will receive a fabricated answer. Depending on the direction of rsETH/ETH rate drift since the queried round, the returned "historical" price will be inflated or deflated relative to the true historical value. This can cause deviation circuit-breakers to trigger falsely, temporarily freezing rsETH-collateralized positions held by rsETH holders, who are explicitly in scope.

## Likelihood Explanation
The contract is a deployed production oracle registered for external protocol consumption. `getRoundData` is a standard, permissionlessly callable function of `AggregatorV3Interface`. Any protocol performing historical round validation — a widely recommended Chainlink integration best practice — will trigger this path. The rsETH/ETH rate increases monotonically as staking rewards accrue, so the discrepancy between current and historical rsETH/ETH rate grows continuously over time, making the corrupted answer increasingly divergent from the true historical value.

## Recommendation
Since no historical rsETH/ETH rate is stored on-chain, `getRoundData` cannot be correctly implemented. The safest fix is to revert with an explicit `NotSupported()` error so integrating protocols cannot silently consume incorrect data:

```solidity
error NotSupported();

function getRoundData(uint80) external pure returns (...) {
    revert NotSupported();
}
```

Alternatively, store historical rsETH/ETH prices keyed by ETH/USD roundId at each `updateRSETHPrice()` call and use the stored value in `getRoundData`.

## Proof of Concept
1. At time T0: rsETH/ETH = 1.05, ETH/USD round R = 2000. True RSETH/USD at R = 2100.
2. At time T1: rsETH/ETH = 1.10 (staking rewards accrued), ETH/USD latest round R+N = 1800.
3. External protocol calls `RSETHPriceFeed.getRoundData(R)` to check historical deviation against `latestRoundData`.
4. `getRoundData(R)` returns: `1.10 × 2000 = 2200` (current rate × historical ETH price).
5. `latestRoundData()` returns: `1.10 × 1800 = 1980`.
6. Protocol computes deviation: `(2200 − 1980) / 2200 ≈ 10%` — a false spike that may trigger a circuit-breaker, freezing rsETH-collateralized positions.

Foundry fork test plan: fork mainnet, deploy a mock consumer that calls `getRoundData` on the deployed `RSETHPriceFeed` at `0x4B9C66c2C0d3706AabC6d00D2a6ffD2B68A4E383` with a historical round ID, then assert that the returned `answer` differs from `historical_ETH_USD × historical_rsETH_ETH` by the accumulated rate drift since that round.
