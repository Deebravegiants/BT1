### Title
Missing On-Chain Cryptographic Validation of Legacy CKD Response Enables Single-Participant Forgery - (`File: crates/contract/src/lib.rs`)

### Summary

The `respond_ckd()` function in the MPC contract applies a cryptographic pairing-check (`ckd_output_check`) only for the publicly-verifiable `AppPublicKeyPV` variant of CKD requests. For the legacy `AppPublicKey` (single G1 point) variant, the response is accepted with **zero cryptographic validation**. A single malicious attested participant — strictly below the signing threshold — can call `respond_ckd()` with an arbitrary forged `CKDResponse`, and the contract will resolve the pending request and deliver the forged output to the user.

### Finding Description

The `request_app_private_key` endpoint accepts two forms of app public key:

- **`AppPublicKeyPV`** (publicly verifiable): a `(pk1, pk2)` pair on G1 × G2. At request time, `app_public_key_check` verifies `e(pk1, g2) = e(g1, pk2)`. At response time, `ckd_output_check` verifies `e(big_c, g2) = e(big_y, pk2) · e(H(pk||app_id), msk_pk)`.
- **`AppPublicKey`** (legacy): a single G1 point. **Neither the request nor the response receives any cryptographic validation.**

The asymmetry is explicit in the contract code:

### Impact Explanation

The CKD protocol's security guarantee is that the correct output requires threshold-many participants to cooperate (each contributes a share of the BLS master secret). For the PV variant this guarantee is enforced on-chain via the pairing equation. For the legacy variant it is not enforced at all on-chain.

A single malicious attested participant can:
1. Observe any pending legacy CKD request in the contract state.
2. Call `respond_ckd(request, CKDResponse { big_c: <arbitrary>, big_y: <arbitrary> })`.
3. The contract resolves the yield and delivers the forged output to the user.

The user receives a `(big_c, big_y)` pair that does not encode `msk · H(pk || app_id)`. They cannot unmask it to obtain the correct confidential key. Any secret or asset that depends on the derived key is permanently inaccessible. This is a bypass of the threshold-signature requirement for CKD operations: one participant achieves what should require `t` participants.

### Likelihood Explanation

The legacy `AppPublicKey` format is the default and most widely documented format (it is the first example in the contract README and the `ckd-example-cli`). It is therefore the format most users will submit. Any single attested participant who wishes to deny a specific user their derived key — or to deliver a predictable wrong key — can do so unilaterally and without detection, since the contract emits no on-chain evidence of which participant submitted the response. [6](#0-5) 

### Recommendation

Apply the same cryptographic response validation to the legacy variant that is already applied to the PV variant. Because the legacy variant does not carry `pk2`, the pairing equation used by `ckd_output_check` cannot be applied directly. Two options:

1. **Deprecate the legacy variant** and require all new requests to use `AppPublicKeyPV`, which already has full on-chain verification.
2. **Add a threshold-vote mechanism** for the legacy response: require `t` distinct attested participants to submit the same `(big_c, big_y)` before the contract resolves the request, analogous to how `vote_pk` works for key generation.

### Proof of Concept

```
1. Alice submits:
   request_app_private_key({
     derivation_path: "mykey",
     app_public_key: "bls12381g1:<valid_g1_point>",   // AppPublicKey (legacy)
     domain_id: 2
   })
   with 1 yoctonear deposit.

2. Mallory (a single attested participant, below threshold) observes the
   pending CKDRequest in contract state.

3. Mallory calls:
   respond_ckd(
     request = <Alice's CKDRequest>,
     response = CKDResponse {
       big_y: Bls12381G1PublicKey([0xAB; 48]),   // arbitrary garbage
       big_c: Bls12381G1PublicKey([0xCD; 48]),   // arbitrary garbage
     }
   )

4. The contract executes lines 675-682 of lib.rs:
   match &request.app_public_key {
       AppPublicKey(_) => {}   // no check — falls through
       ...
   }
   pending_requests::resolve_yields_for(...)   // request resolved

5. Alice's promise resolves with the forged (big_y, big_c).
   Alice cannot unmask it; her confidential key is permanently lost.
```

The attack requires only that Mallory be an attested participant — a single Byzantine node below the signing threshold — which is the standard adversarial assumption for MPC systems.

pub(crate) fn app_public_key_check(app_public_key: &dtos::CKDAppPublicKeyPV) -> bool {
    let pk1 = env::bls12381_p1_decompress(&app_public_key.pk1);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);

    let pairing_input = [
        pk1.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        G1_GENERATOR_UNCOMPRESSED.as_slice(),
        pk2.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
