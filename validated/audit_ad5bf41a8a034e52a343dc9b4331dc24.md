Audit Report

## Title
Missing Chainlink `latestRoundData` Validation Allows Stale/Invalid Price Consumption - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validation return values — no staleness check, round-completeness check, or positive-price check is performed. The same codebase already implements the correct pattern in `ChainlinkOracleForRSETHPoolCollateral.getRate()`. A stale price fed into the deposit minting path causes incorrect rsETH issuance, diluting existing rsETH holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` (L49–55) fetches the price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values are available but only `answer` is consumed. In contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L27–32) performs all three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unvalidated price flows into two critical paths:

**Path 1 — Deposit minting:** `depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice()` → `ChainlinkPriceOracle.getAssetPrice()`. The returned price directly determines rsETH minted per deposited LST:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

**Path 2 — rsETH price update:** `_getTotalEthInProtocol()` iterates all supported assets and calls `getAssetPrice()` for each, summing their ETH value to set the global `rsETHPrice`: [4](#0-3) 

No existing check in `_beforeDeposit()` guards against a stale or invalid oracle price — the only downstream check is `minRSETHAmountExpected`, which protects the depositor from receiving *less* than expected but does not protect existing holders from dilution when the depositor receives *more* than correct. [5](#0-4) 

## Impact Explanation

**High — Theft of unclaimed yield.**

When a Chainlink feed returns a stale price that is *higher* than the true market price of the LST (e.g., the feed has not updated during a rapid LST price decline), `getRsETHAmountToMint()` returns more rsETH than the deposited assets are worth. The depositor receives excess rsETH backed by fewer real assets, directly diluting the share value of all existing rsETH holders — this is theft of unclaimed yield from existing holders. The effect is compounded if `rsETHPrice` was last set with the correct price while the per-asset feed is now stale, because the denominator remains accurate while the numerator is inflated.

The zero-price scenario (price = 0 → rsethAmountToMint = 0) would freeze deposited funds, but is mitigated in practice by a non-zero `minRSETHAmountExpected` parameter, making it dependent on a victim mistake and therefore not independently scored here.

The negative-price scenario (uint256 wrap) is theoretically Critical but requires a Chainlink feed to return a negative value, which is not a realistic operational condition for LST/ETH feeds and is therefore not used to upgrade severity.

## Likelihood Explanation

Chainlink feed staleness is a documented operational risk: L2 sequencer downtime, network congestion, and heartbeat misses during low-volatility periods have all caused stale rounds historically. The protocol supports multiple LST assets each with independent feeds, increasing the attack surface. No privileged access is required — any unprivileged user can call `depositAsset()` during a feed disruption. The attacker need only observe that a feed's `updatedAt` timestamp is stale and that the last reported price is above the current true price before depositing.

## Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`STALENESS_THRESHOLD` should be configured per-feed based on the feed's documented heartbeat interval.

## Proof of Concept

1. A supported LST/ETH Chainlink feed (e.g., stETH/ETH) enters a stale round — `updatedAt` is beyond the heartbeat window but `answeredInRound == roundId` so the round-completeness check (if it existed) would not trigger. The last reported price is 2% above the current true market price (e.g., the LST has depreciated since the last update).
2. `rsETHPrice` was last updated correctly and reflects the true NAV.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `_beforeDeposit()` calls `getRsETHAmountToMint(stETH, 100e18)`.
5. `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns the stale high price (1.02 ETH per stETH instead of 1.00 ETH).
6. `rsethAmountToMint = (100e18 * 1.02e18) / rsETHPrice` — attacker receives ~2% more rsETH than the deposited assets are worth.
7. No revert occurs; `minRSETHAmountExpected = 0` passes the slippage check.
8. Existing rsETH holders' share value is diluted by the excess issuance.

**Foundry fork test plan:** Fork mainnet/L2 at a block where a target LST/ETH feed's `updatedAt` is stale. Call `ChainlinkPriceOracle.getAssetPrice(stETH)` and compare the returned value against the true on-chain price from a non-stale source. Then call `depositAsset` and assert that `rsethAmountToMint` exceeds the correct amount, confirming dilution of existing holders.
