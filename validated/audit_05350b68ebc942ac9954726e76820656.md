### Title
Permanent Freezing of Unclaimed Yield via `setToken()` Token Rotation Without Recovery Path — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

### Summary
`setToken()` can be called at any time to replace the distributed token address. If called while the contract holds unclaimed balance of the old token, all pending claims permanently revert (insufficient balance of the new token), and the old token balance is irrecoverable because no sweep/rescue function exists.

### Finding Description
`setToken()` performs a bare address swap with no check for pending claimable balances:

```solidity
function setToken(address _token) external onlyOwner {
    if (_token == address(0)) { revert ZeroValueProvided(); }
    token = _token;          // ← overwrites token globally, no balance guard
    emit TokenUpdated(_token);
}
``` [1](#0-0) 

`claim()` then unconditionally transfers `IERC20(token)` — the new address — which holds zero balance:

```solidity
IERC20(token).safeTransfer(account, amountToSend);   // reverts: newToken balance = 0
IERC20(token).safeTransfer(protocolTreasury, fee);
``` [2](#0-1) 

The old token balance is now permanently locked: `claim()` reverts for every user (wrong token), and there is no `recoverTokens()`, `sweep()`, or any other rescue function in the contract. [4](#0-3) 

The `initialize()` comment explicitly acknowledges post-deployment token setting as an intended workflow ("token can be set later"), making this a realistic operational path, not a theoretical one. [3](#0-2) 

### Impact Explanation
All unclaimed yield denominated in the old token is permanently frozen in the contract with no on-chain recovery path. Affected users cannot claim, and the owner cannot rescue the funds. Impact: **Medium — Permanent freezing of unclaimed yield**.

### Likelihood Explanation
Token rotation is an explicitly supported admin operation. It can be triggered accidentally during a token migration/upgrade without the admin realizing unclaimed balances remain. No malicious intent or key compromise is required — a routine configuration change suffices.

### Recommendation
1. Add a balance guard to `setToken()`: revert if `IERC20(token).balanceOf(address(this)) > 0`.
2. Alternatively, add a `recoverTokens(address token_, address recipient, uint256 amount) external onlyOwner` rescue function so stuck balances can always be recovered.
3. Consider emitting the old token address in `TokenUpdated` to aid off-chain monitoring.

### Proof of Concept
```solidity
// 1. Deploy MerkleDistributor with tokenA
// 2. Transfer 1000e18 tokenA to the distributor
