### Title
Uninitialized `parent_hash` and `state_root` Fields in `update_state_sync_with_new_block` Produce Zero-Valued Block Headers in State Sync Storage - (File: crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs)

### Summary

In `update_state_sync_with_new_block`, the `BlockHeaderWithoutHash` struct is constructed with `..Default::default()`, leaving `parent_hash` and `state_root` as `Felt::ZERO`. An explicit TODO comment acknowledges the omission. The resulting `SyncBlock` is written to state sync via `add_new_block`, and the gateway's `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` reads `starknet_version` directly from that same struct. Any downstream consumer of state sync (RPC, gateway stateful validation) receives authoritative-looking but structurally wrong block header data.

A block stored with `parent_hash = Felt::ZERO` and `state_root = Felt::ZERO` will be returned verbatim by `starknet_getBlockWithTxHashes`, `starknet_getBlockWithTxs`, and `starknet_getStateUpdate`. ### Impact Explanation

Every block committed through the consensus `decision_reached` path in P2P-sync mode is stored in state sync with:

- `parent_hash = BlockHash(Felt::ZERO)` — wrong chain linkage
- `state_root = GlobalRoot(Felt::ZERO)` — wrong global state root

**RPC impact (High):** `starknet_getBlockWithTxHashes` and `starknet_getStateUpdate` return `parent_hash = 0x0` and `new_root = 0x0` for every sequencer-produced block. Clients relying on these fields for chain verification, proof generation, or state-root anchoring receive authoritative-looking but incorrect data.

**Gateway stateful validation impact:** `starknet_version` is also left as `Default::default()`. The gateway maps this directly into `BlockInfo.starknet_version`, which selects `VersionedConstants` for fee and resource-bound validation. If `StarknetVersion::default()` diverges from the actual protocol version at the time of block production, the gateway applies wrong fee/resource limits, potentially admitting transactions that should be rejected or rejecting valid ones.

### Likelihood Explanation

The trigger is automatic: every block produced by the sequencer and committed via `decision_reached` calls `update_state_sync_with_new_block`. No attacker action is required. The bug fires unconditionally in any deployment where state sync is not in Central Sync (dev_null) mode — i.e., in P2P sync mode, which is the production topology for a live sequencer node. The TODO comment confirms the omission is known but unresolved.

### Recommendation

Populate the missing fields from available sources at the call site:

- `parent_hash`: read the previous block's hash from state sync (`get_block_hash(height.prev())`) or pass it through from the `DecisionReachedResponse`.
- `state_root`: read from the committer's `FinalBlockCommitment.global_root` after the commitment task completes, or pass it through `DecisionReachedResponse`.
- `starknet_version`: use `init.starknet_version` (already present in `ProposalInit`) rather than relying on `Default`.

Remove the `..Default::default()` fallback once all fields are explicitly assigned, so future additions to `BlockHeaderWithoutHash` produce a compile error rather than a silent zero.

### Proof of Concept

1. Run the sequencer in P2P sync mode so `add_new_block` is not a no-op.
2. Let consensus commit one block (height N) via `decision_reached`.
3. Call `starknet_getBlockWithTxHashes` for block N via the RPC.
4. Observe `parent_hash = "0x0"` and `new_root = "0x0"` in the response, regardless of the actual parent hash and state root.
5. Alternatively, inspect the `SyncBlock` written to state sync: `block_header_without_hash.parent_hash == BlockHash(Felt::ZERO)` and `block_header_without_hash.state_root == GlobalRoot(Felt::ZERO)`. [7](#0-6)
