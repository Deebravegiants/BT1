### Title
`HyperliquedBridgeToken` 3-arg `mint` permanently routes bridged tokens to `_systemAddress` when `finTransfer` is called with a non-empty `message` field — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`OmniBridge.finTransfer` dispatches to the 3-arg `IBridgeToken.mint(account, value, message)` whenever `payload.message.length != 0`. For `HyperliquedBridgeToken`, that overload mints `value` tokens to `account` and then immediately calls `_update(account, _systemAddress, value)`, transferring the entire minted balance away from the recipient. The net credit to the recipient is zero; `_systemAddress` absorbs the full amount.

---

### Finding Description

**Dispatch logic in `OmniBridge.finTransfer`** (lines 337–349):

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient,
            payload.amount
        );
    } else {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient,
            payload.amount,
            payload.message   // ← any non-empty bytes triggers this
        );
    }
}
``` [1](#0-0) 

**`HyperliquedBridgeToken.mint` (3-arg)** (lines 76–83):

```solidity
function mint(address account, uint256 value, bytes memory) external override onlyOwner {
    _mint(account, value);
    _update(account, _systemAddress, value);  // ← full balance immediately swept away
}
``` [2](#0-1) 

`_mint` increases `account`'s balance by `value`. `_update(account, _systemAddress, value)` is an ERC-20 internal transfer that moves the same `value` from `account` to `_systemAddress`. The two operations cancel out: `account` ends with a net change of zero, and `_systemAddress` receives the full minted amount.

This 3-arg path is intentionally designed for HyperCore-side minting (where `_systemAddress` acts as a pool mirror), but `finTransfer` reaches it for any ordinary cross-chain transfer that carries a non-empty `message` bytes field — a field that is user-controlled on the originating chain.

The `message` field is part of the MPC-signed payload:

```solidity
bytes(payload.message).length == 0
    ? bytes("")
    : Borsh.encodeBytes(payload.message)
``` [3](#0-2) 

So the MPC legitimately signs payloads with non-empty messages whenever the originating-chain user includes one. No signature forgery is required.

---

### Impact Explanation

Any cross-chain transfer to a `HyperliquedBridgeToken` address that carries a non-empty `message` field results in:

- `payload.recipient` receives **0 tokens** (minted then immediately swept)
- `_systemAddress` absorbs the full `payload.amount`
- The originating-chain tokens are already burned/locked; the loss is permanent and irreversible

This breaks the core bridge invariant: a `finTransfer` must credit `payload.recipient` with `payload.amount` of the bridged token.

---

### Likelihood Explanation

The `message` field is a documented, user-facing parameter of the bridge (`initTransfer` accepts it as `string calldata message`). [4](#0-3) 

Any user who:
1. Initiates a transfer from NEAR (or another chain) to a `HyperliquedBridgeToken` on EVM, and
2. Includes any non-empty `message` bytes (e.g., for application-level metadata, or simply by mistake)

will have their tokens permanently misrouted. Additionally, a malicious sender can deliberately include `message = [0x01]` when sending tokens to a victim's address, causing the victim to receive nothing while the tokens accumulate at `_systemAddress`. No privileged access is required — only the ability to initiate a standard bridge transfer.

---

### Recommendation

In `OmniBridge.finTransfer`, always call the 2-arg `mint` for `isBridgeToken` tokens, regardless of `message` content. If the `message` field needs to trigger HyperCore-specific behavior, that routing decision should be made inside the token contract based on a separate, authenticated signal (e.g., a dedicated entry point callable only by the system address), not by the presence of arbitrary user-supplied bytes in the transfer payload.

Alternatively, `HyperliquedBridgeToken` should override the 3-arg `mint` to revert unconditionally (since HyperCore minting should only occur via `coreReceiveWithData`), preventing the bridge from accidentally invoking the HyperCore path.

---

### Proof of Concept

```solidity
// 1. Deploy HyperliquedBridgeToken; register it as isBridgeToken in OmniBridge.
// 2. On NEAR, initiate a transfer to `victim` with message = hex"01".
// 3. MPC signs the payload (legitimate operation).
// 4. Relayer calls OmniBridge.finTransfer(sig, payload) where payload.message = hex"01".
// 5. finTransfer takes the `else` branch → calls mint(victim, amount, hex"01").
// 6. HyperliquedBridgeToken.mint: _mint(victim, amount) then _update(victim, _systemAddress, amount).
// 7. Assert: token.balanceOf(victim) == 0
//    Assert: token.balanceOf(_systemAddress) == amount
```
