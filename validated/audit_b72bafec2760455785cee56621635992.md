### Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Blocks All Withdrawal Initiations, Temporarily Freezing Staked KERNEL Tokens Until Admin Intervention - (File: contracts/KERNEL/KernelDepositPool.sol)

---

### Summary

`KernelDepositPool.initiateWithdrawal()` guards against exceeding a per-user withdrawal cap with a check against `maxNumberOfWithdrawalsPerUser`. Because that variable is never set in `initialize()`, it defaults to `0`. The guard therefore evaluates `0 >= 0 → true` for every caller, causing every withdrawal initiation to revert with `WithdrawalLimitReached()`. Any user who stakes KERNEL tokens before an admin manually sets the variable to a non-zero value has their funds temporarily frozen in the contract.

---

### Finding Description

`KernelDepositPool` declares two distinct withdrawal-cap identifiers:

- A constant `MAX_WITHDRAWALS_PER_USER = 100` (line 38) — never used in the withdrawal guard.
- A mutable state variable `maxNumberOfWithdrawalsPerUser` (line 108) — used in the guard but **never initialized**. [1](#0-0) [2](#0-1) 

The `initialize()` function sets `kernelToken`, `rewardsToken`, and access-control roles, but never assigns `maxNumberOfWithdrawalsPerUser`: [3](#0-2) 

`initiateWithdrawal()` then performs:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
``` [4](#0-3) 

For every user, `userWithdrawalIds[msg.sender].length` starts at `0`. With `maxNumberOfWithdrawalsPerUser == 0`, the condition `0 >= 0` is always `true`, so the function always reverts. Meanwhile, `stake()` and `stakeFor()` impose no such dependency and succeed normally: [5](#0-4) 

The result is a one-way door: tokens flow in via `stake()` but cannot flow out via `initiateWithdrawal()` until an admin separately calls the setter for `maxNumberOfWithdrawalsPerUser`.

---

### Impact Explanation

Staked KERNEL tokens are temporarily frozen. Users cannot initiate the withdrawal process at all — not even for a partial amount — because the guard fires before any balance check. The only recovery path is admin intervention to set `maxNumberOfWithdrawalsPerUser` to a non-zero value. This matches **Medium — Temporary freezing of funds**.

---

### Likelihood Explanation

The contract is upgradeable and deployed via a proxy. Any deployment or upgrade that calls `initialize()` without a subsequent admin call to configure `maxNumberOfWithdrawalsPerUser` leaves the contract in the broken state. Because `stake()` works normally, users have no on-chain signal that withdrawals are blocked until they attempt one. The constant `MAX_WITHDRAWALS_PER_USER = 100` exists in the same file and is the natural value to use, making the omission easy to overlook during deployment. Likelihood is **Medium**.

---

### Recommendation

Initialize `maxNumberOfWithdrawalsPerUser` to a sensible non-zero default (e.g., `MAX_WITHDRAWALS_PER_USER`) inside `initialize()`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... existing setup ...
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // add this line
}
```

This mirrors the pattern used for `withdrawalDelay` in `LRTWithdrawalManager`, which is explicitly set during initialization: [6](#0-5) 

---

### Proof of Concept

1. Admin deploys `KernelDepositPool` proxy and calls `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` remains `0`.
2. User approves the contract and calls `stake(100e18)`. Succeeds — 100 KERNEL tokens are transferred to the contract and `balanceOf[user] = 100e18`.
3. User calls `initiateWithdrawal(100e18)`. Execution reaches:
   ```
   userWithdrawalIds[user].length  →  0
   maxNumberOfWithdrawalsPerUser   →  0
   0 >= 0                          →  true  →  revert WithdrawalLimitReached()
   ```
4. User's 100 KERNEL tokens are stuck. `claimWithdrawal()` is also unreachable because no withdrawal record was ever created.
5. Recovery requires the admin to call the setter for `maxNumberOfWithdrawalsPerUser` with a value `> 0`, after which step 3 succeeds. [7](#0-6)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
```
