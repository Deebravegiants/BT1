Audit Report

## Title
Missing Staleness and Validity Checks on Chainlink `latestRoundData()` in `ChainlinkPriceOracle.getAssetPrice()` Enables Incorrect rsETH Minting — (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validation fields (`updatedAt`, `answeredInRound`, `roundId`), accepting stale or zero/negative prices without any check. This unvalidated price flows directly into `LRTDepositPool.getRsETHAmountToMint()`, causing depositors to receive incorrect rsETH amounts. Over-minting dilutes existing rsETH holders' accrued yield — a direct theft of unclaimed yield — while under-minting harms depositors. The same codebase already ships `ChainlinkOracleForRSETHPoolCollateral.getRate()` with full validation, confirming the protocol is aware of these checks but failed to apply them in the core deposit oracle.

## Finding Description

**Root cause:** `ChainlinkPriceOracle.getAssetPrice()` at line 52 silently discards all validation return values from `latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

No check is performed for:
- **Staleness**: `block.timestamp - updatedAt > heartbeat`
- **Incomplete round**: `updatedAt == 0`
- **Stale round**: `answeredInRound < roundId`
- **Non-positive price**: `price <= 0`

In contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same codebase performs all three critical checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L26-37
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

**Propagation path:** The stale price flows directly into the minting formula:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

`lrtOracle.getAssetPrice(asset)` delegates to `ChainlinkPriceOracle.getAssetPrice()`, returning the unvalidated stale price. This value directly determines how many rsETH tokens are minted to the depositor.

**Call chain:** `depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice()`. [4](#0-3) 

**Existing guards are insufficient:** The `_beforeDeposit()` check only enforces `minRSETHAmountExpected`, which a depositor exploiting an inflated stale price would set to match the over-minted amount. No protocol-level guard validates the oracle price before minting. [5](#0-4) 

## Impact Explanation

**Impact: High — Theft of unclaimed yield.**

When a Chainlink LST/ETH feed (e.g., stETH/ETH, rETH/ETH) goes stale with an inflated price, any depositor calling `depositAsset()` receives more rsETH than the actual ETH value of their deposit warrants. This over-minting dilutes all existing rsETH holders proportionally. When `updateRSETHPrice()` is next called, the true TVL is lower than the inflated rsETH supply implies, and the rsETH price drops — existing holders lose accrued yield and principal value. This constitutes a direct, concrete theft of unclaimed yield from existing rsETH holders, matching the allowed High impact scope. The inverse (stale deflated price) harms depositors by under-minting, matching the Low impact scope of "contract fails to deliver promised returns."

## Likelihood Explanation

**Likelihood: Low.**

Chainlink feeds can go stale during network congestion, oracle node disruptions, or feed deprecation events. The stETH/ETH feed has a 24-hour heartbeat, creating a meaningful window during which the vulnerability is fully exploitable. No special attacker capability is required — any depositor can call the public `depositAsset()` function during a stale-price window. While not a routine occurrence, Chainlink feed staleness is a documented and realistic failure mode. The vulnerability is passively exploitable by any depositor, not just a targeted attacker.

## Recommendation

Apply the same staleness validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale price: round not complete");
    require(updatedAt != 0, "Stale price: incomplete round");
    require(price > 0, "Invalid price");
    // Optional: require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too stale");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

## Proof of Concept

**Foundry fork test plan:**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed is live.
2. Deploy or reference the existing `ChainlinkPriceOracle` pointing to the stETH/ETH feed.
3. Use `vm.mockCall` to mock `latestRoundData()` on the feed to return: `roundId=10`, `price=1.05e18`, `startedAt=block.timestamp - 25 hours`, `updatedAt=block.timestamp - 25 hours`, `answeredInRound=9` (stale: `answeredInRound < roundId`, and `updatedAt` beyond heartbeat).
4. Call `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")` as an unprivileged depositor.
5. Assert that `rsethAmountToMint` is computed using the stale `1.05e18` price rather than reverting.
6. Compare the minted rsETH amount against what would be minted at the true price `1.00e18` — the difference (~5%) represents the over-minted rsETH diluting existing holders.
7. Call `LRTOracle.updateRSETHPrice()` and assert the rsETH price has dropped, confirming existing holders lost value.
