### Title
Missing Access Control on `updateRSETHPrice()` Allows Anyone to Trigger Protocol Fee Minting — (File: contracts/LRTOracle.sol)

---

### Summary
`LRTOracle.updateRSETHPrice()` is declared `public` with only a `whenNotPaused` guard and no role-based access control. Any unprivileged caller can invoke it at will. Internally it calls `_updateRsETHPrice()`, which mints protocol fees as rsETH to the treasury whenever TVL has grown since the last update, diluting rsETH holders and extracting yield on the attacker's schedule rather than the protocol's intended cadence.

---

### Finding Description
`updateRSETHPrice()` at line 87 of `LRTOracle.sol` carries no role restriction:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [1](#0-0) 

`_updateRsETHPrice()` (line 214) performs three security-sensitive actions:

1. **Fee minting** — when `totalETHInProtocol > previousTVL`, it computes `protocolFeeInETH` and mints `rsethAmountToMintAsProtocolFee` rsETH directly to the treasury: [2](#0-1) 

2. **Price update** — it writes the new `rsETHPrice` used by every deposit and withdrawal calculation: [3](#0-2) 

3. **Protocol pause** — if the price drops beyond `pricePercentageLimit`, it pauses `LRTDepositPool`, `LRTWithdrawalManager`, and itself: [4](#0-3) 

The daily fee cap (`maxFeeMintAmountPerDay`) limits total minting per 24-hour window but does not prevent an unprivileged caller from triggering fee minting at any moment within that window.

The protocol itself recognises that price-update authority is sensitive: `updateRSETHPriceAsManager()` is correctly gated with `onlyLRTManager`: [5](#0-4) 

The public variant has no equivalent guard.

---

### Impact Explanation
**High — Theft of unclaimed yield.**

Every call to `updateRSETHPrice()` that finds `totalETHInProtocol > previousTVL` mints new rsETH to the treasury. This dilutes all existing rsETH holders: yield that would otherwise remain in the rsETH price (accruing to holders) is instead extracted as protocol fees. An attacker can call the function repeatedly — up to the daily cap — to maximise fee extraction on their own schedule, front-running the protocol's intended update cadence and accelerating dilution of rsETH holders.

---

### Likelihood Explanation
**High.** The function is `public` with no caller restriction. Any EOA or contract can call it at any time the contract is not paused. No capital, special role, or setup is required.

---

### Recommendation
Add an access control modifier to `updateRSETHPrice()` to restrict it to an authorised caller (e.g., a dedicated keeper role or `onlyLRTManager`), consistent with `updateRSETHPriceAsManager()`:

```diff
- function updateRSETHPrice() public whenNotPaused {
+ function updateRSETHPrice() external whenNotPaused onlyLRTManager {
      _updateRsETHPrice();
  }
```

If permissionless price updates are desired for liveness, introduce a separate keeper role with no fee-minting authority, or decouple fee minting from the public price-update path.

---

### Proof of Concept
1. Protocol TVL increases (e.g., staking rewards accrue), so `totalETHInProtocol > previousTVL`.
2. Attacker calls `LRTOracle.updateRSETHPrice()` — no role check, passes `whenNotPaused`.
3. `_updateRsETHPrice()` computes `protocolFeeInETH = (rewardAmount * protocolFeeInBPS) / 10_000`.
4. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` passes (within daily cap).
5. `IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee)` executes — new rsETH is minted to the treasury, diluting all rsETH holders.
6. Attacker repeats until the daily cap is exhausted, maximising fee extraction and rsETH holder dilution entirely outside the protocol's intended update schedule. [6](#0-5)
