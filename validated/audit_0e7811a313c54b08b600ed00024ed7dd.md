### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Enables Front-Running to Obtain Transferable Bridge Attestations - (File: crates/contract/src/lib.rs, crates/contract/src/dto_mapping.rs)

### Summary

The `verify_foreign_transaction` endpoint uses a request key that does not include the NEAR caller's identity. Any unprivileged account can observe a pending verification request (NEAR transactions are visible before finalization), submit the identical request, and receive the same MPC-signed attestation. Because the attestation is signed over the root key with no caller-binding tweak, it is fully transferable. An attacker who front-runs a legitimate bridge user's request receives an identical, valid MPC attestation and can use it to claim bridge funds before the legitimate user.

### Finding Description

`verify_foreign_transaction` converts user arguments into a `VerifyForeignTransactionRequest` via `args_into_verify_foreign_tx_request`. That function drops the caller's account ID entirely:

```rust
// crates/contract/src/dto_mapping.rs:840-848

    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [1](#0-0) 

The resulting `VerifyForeignTransactionRequest` struct has no `predecessor` / caller field:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [2](#0-1) 

This struct is used as the map key in `pending_verify_foreign_tx_requests`. The fan-out queue design explicitly allows different callers to share the same key and all receive the same response:

```rust
// "duplicate foreign-tx requests from different callers should fan out"
assert_eq!(
    contract.pending_verify_foreign_tx_requests.get(&request).map(|q| q.len()),
    Some(2),
    ...
);
``` [3](#0-2) 

The MPC signature produced by `respond_verify_foreign_tx` is verified against the **root public key with no derivation tweak**, unlike regular `sign` responses which use a per-caller derived key:

```rust
// respond_verify_foreign_tx: root key, no tweak
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // root key — no caller-binding tweak
)
``` [4](#0-3) 

Compare to `respond` for regular signatures, which derives a per-caller key:

```rust
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
``` [5](#0-4) 

The signed payload is `SHA-256(borsh(ForeignTxSignPayload::V1 { request, observed_values }))` — it contains the foreign chain `tx_id` and extracted values, but **no NEAR caller identity**. The attestation is therefore fully transferable: whoever holds the `VerifyForeignTransactionResponse` can present it to any downstream bridge contract.

### Impact Explanation

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow: a user sends funds on a foreign chain (e.g., Bitcoin), then calls `verify_foreign_transaction` on NEAR to obtain an MPC attestation proving the foreign transaction finalized, which a bridge contract uses to release NEAR-side funds.

**Attack path:**

1. Alice sends 1 BTC to the bridge address on Bitcoin and submits `verify_foreign_transaction` on NEAR with the Bitcoin `tx_id`.
2. Eve observes Alice's pending NEAR transaction (visible in the mempool / before finalization) and submits the identical `verify_foreign_transaction` call.
3. Both are queued under the same caller-agnostic key. MPC nodes respond once; both Alice and Eve receive the same `VerifyForeignTransactionResponse` (same `payload_hash` + same MPC signature).
4. Eve's NEAR transaction completes first (higher gas price / earlier block). Eve presents her `VerifyForeignTransactionResponse` to the bridge contract and claims Alice's 1 BTC equivalent in NEAR-side tokens.
5. When Alice's transaction completes, the bridge contract has already processed the attestation and Alice's claim fails or is double-spent.

This is a **forged foreign-chain verification / participant authorization bypass** that causes **invalid bridge execution and double-spend conditions**, matching the High impact tier.

### Likelihood Explanation

- NEAR transactions are publicly visible before finalization, making the front-running window observable.
- The attacker needs only 1 yoctoNEAR deposit and standard gas — no privileged access, no threshold collusion.
- The attack is profitable whenever the bridged value exceeds gas costs, which is true for any non-trivial bridge transfer.
- The fan-out behavior is explicitly tested and documented as intentional, so no special exploit tooling is needed.

### Recommendation

**Short term:** Include the NEAR caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key (analogous to how `SignatureRequest::new` binds the request to the predecessor). This prevents different callers from sharing the same queue slot and receiving the same attestation.

**Medium term:** Bind the MPC signature to the NEAR caller by incorporating the predecessor account ID into the signed payload (`ForeignTxSignPayloadV1`), so the attestation is only usable by the account that requested it.

**Long term:** Bridge contracts consuming `VerifyForeignTransactionResponse` should independently verify that the NEAR caller who presents the attestation is the same account that submitted the original `verify_foreign_transaction` request, or that the foreign chain transaction's recipient address maps to the NEAR caller.

### Proof of Concept

The existing sandbox test `verify_foreign_transaction__should_fan_out_response_to_duplicates_from_different_callers` already demonstrates the core mechanism: [6](#0-5) 

Alice and Bob (different accounts) submit the same request; both receive the identical `VerifyForeignTransactionResponse`. The test asserts this as correct behavior. An attacker exploiting this would:

1. Monitor the NEAR mempool for `verify_foreign_transaction` calls.
2. Resubmit the same `{request, domain_id, payload_version}` with higher gas.
3. Receive the MPC-signed attestation.
4. Call the downstream bridge contract with the attestation before the victim's transaction settles.

The root cause is confirmed at:
- `args_into_verify_foreign_tx_request` dropping the caller: [1](#0-0) 
- `verify_foreign_transaction` not passing `predecessor` into the request key: [7](#0-6) 
- `VerifyForeignTransactionRequest` having no caller field: [2](#0-1) 
- `respond_verify_foreign_tx` signing with the root key (no caller-binding tweak): [8](#0-7)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

#[tokio::test]
async fn verify_foreign_transaction__should_fan_out_response_to_duplicates_from_different_callers()
{
    // Given
    let rpc_request = bitcoin_request();
    let extracted_values = bitcoin_extracted_values();
    let chain = rpc_request.chain();
    let setup = SandboxTestSetup::builder()
        .with_foreign_tx_domain()
        .build()
        .await;
    let foreign_tx_key = setup.foreign_tx_key();
    register_foreign_chain_configuration(chain, &setup.contract, &setup.mpc_signer_accounts).await;

    let alice = setup.worker.dev_create_account().await.unwrap();
    let bob = setup.worker.dev_create_account().await.unwrap();
    let domain_id = dtos::DomainId(foreign_tx_key.domain_id().0);
    let request_args = dtos::VerifyForeignTransactionRequestArgs {
        domain_id,
        payload_version: ForeignTxPayloadVersion::V1,
        request: rpc_request.clone(),
    };
    let verify_request = VerifyForeignTransactionRequest {
        domain_id,
        payload_version: ForeignTxPayloadVersion::V1,
        request: rpc_request,
    };

    // When
    let status_alice = alice
        .call(
            setup.contract.id(),
            method_names::VERIFY_FOREIGN_TRANSACTION,
        )
        .args_json(json!({ "request": request_args }))
        .deposit(NearToken::from_yoctonear(1))
        .max_gas()
        .transact_async()
        .await
        .unwrap();
    let status_bob = bob
        .call(
            setup.contract.id(),
            method_names::VERIFY_FOREIGN_TRANSACTION,
        )
        .args_json(json!({ "request": request_args }))
        .deposit(NearToken::from_yoctonear(1))
        .max_gas()
        .transact_async()
        .await
        .unwrap();
    await_pending_foreign_tx_request_observed_on_contract(&setup.contract, &verify_request).await;

    let (payload, response) = sign_foreign_tx_response(
        &verify_request.request,
        extracted_values,
        foreign_tx_key.as_secp256k1(),
    );
    let respond_result = setup.mpc_signer_accounts[0]
        .call(setup.contract.id(), method_names::RESPOND_VERIFY_FOREIGN_TX)
        .args_json(json!({
            "request": verify_request,
            "response": response,
        }))
        .max_gas()
        .transact()
        .await
        .unwrap()
        .into_result();

    // Then
    assert!(
        respond_result.is_ok(),
        "single respond_verify_foreign_tx should drain both queued yields: {respond_result:?}",
    );
    let expected_payload_hash = payload.compute_msg_hash().unwrap();

    let exec_alice = status_alice.await.unwrap().into_result().unwrap();
    let returned_alice: VerifyForeignTransactionResponse = exec_alice.json().unwrap();
    assert_eq!(
        returned_alice.payload_hash, expected_payload_hash,
        "alice's tx should receive the fanned-out response",
    );

    let exec_bob = status_bob.await.unwrap().into_result().unwrap();
    let returned_bob: VerifyForeignTransactionResponse = exec_bob.json().unwrap();
    assert_eq!(
        returned_bob.payload_hash, expected_payload_hash,
        "bob's tx should receive the same fanned-out response and not be displaced by alice",
    );
}
