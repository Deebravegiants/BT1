### Title
P2P Peer-Supplied `state_diff_length=0` in `SignedBlockHeader` Causes Silent Empty `ThinStateDiff` Committed to Storage — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

An unprivileged p2p peer can serve a `SignedBlockHeader` with `state_diff_length=0` for any block. Because the p2p header sync path never cryptographically verifies the block signature, the forged header is stored verbatim. The state diff sync then reads `target_state_diff_len=0` from that stored header, the while loop in `parse_data_for_block` exits immediately without consuming any chunks, and `ThinStateDiff::default()` (fully empty) is committed to storage via `append_state_diff`. No `BadPeerError` is raised; the peer is not reported.

---

### Finding Description

**Step 1 — Header accepted without signature verification.**

`HeaderStreamBuilder::parse_data_for_block` performs only two checks on the incoming `SignedBlockHeader`:

- `block_number` matches the expected sequence number
- `signatures.len() == ALLOWED_SIGNATURES_LENGTH` (exactly 1 element) [1](#0-0) 

`verify_block_signature` (which computes `Poseidon(block_hash, state_diff_commitment)` and checks it against the sequencer's ECDSA key) is never called in this path. [2](#0-1) 

The attacker-supplied header — including its `state_diff_length=0` and any `state_diff_commitment` — is written directly to storage: [3](#0-2) 

**Step 2 — State diff sync reads the poisoned length.**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` from the stored header: [4](#0-3) 

**Step 3 — While loop exits immediately; empty diff committed.**

When `target_state_diff_len == 0`, the condition `current_state_diff_len < target_state_diff_len` is false from the start. The loop body never executes, no chunks are consumed, and the function returns `Ok(Some((ThinStateDiff::default(), block_number)))`: [5](#0-4) 

That empty diff is then committed unconditionally: [6](#0-5) 

There is no post-write check that the stored `ThinStateDiff` matches the `state_diff_commitment` field in the header.

---

### Impact Explanation

Every contract storage update, nonce change, class deployment, and class declaration in the real block is silently dropped. The node's state trie diverges from the canonical chain. All subsequent RPC state queries (`get_storage_at`, `get_nonce`, `get_class_hash_at`) return wrong values. Any block built on top of this node's state will use the corrupted base state, propagating the corruption forward.

---

### Likelihood Explanation

The p2p network is permissionless; any node can connect and serve headers. The attack requires only that the victim node selects the attacker as a peer for the header query covering the target block. No key material, validator role, or operator privilege is needed. The attacker supplies a single well-formed protobuf `SignedBlockHeader` with `state_diff_commitment.state_diff_length = 0` and any one-element `signatures` array.

---

### Recommendation

1. **Verify the block signature** in `HeaderStreamBuilder::parse_data_for_block` using `verify_block_signature` against the known sequencer public key before storing the header.
2. **Cross-validate the state diff commitment** after assembling the full `ThinStateDiff`: compute `calculate_state_diff_hash(&result)` and compare it to the `state_diff_commitment` stored in the header. Return `BadPeerError` on mismatch.
3. **Treat `target_state_diff_len == 0` as a special case** that still requires a `Fin` message from the peer before returning `Ok(Some(...))`, preventing the silent-success path.

---
