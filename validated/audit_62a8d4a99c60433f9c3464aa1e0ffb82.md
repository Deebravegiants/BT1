### Title
Missing Caller Identity Binding in `ForeignTxSignPayload` Enables Cross-Caller Reuse of Foreign-Chain Verification Proofs - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`, `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

The `verify_foreign_transaction` flow produces an MPC threshold signature over a `ForeignTxSignPayload` that contains **no caller identity** (`predecessor_id`). The pending-request map key (`VerifyForeignTransactionRequest`) is also caller-agnostic. Any unprivileged NEAR account can submit the identical request and receive the same MPC-signed attestation. Because the signed payload carries no binding to the originating NEAR account, downstream bridge contracts cannot use the MPC signature to authenticate which specific caller is entitled to claim the benefit of the verified foreign transaction, enabling double-spend or unauthorized fund claims.

---

### Finding Description

**Root cause 1 — `ForeignTxSignPayload` contains no `predecessor_id`.**

`ForeignTxSignPayloadV1` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [1](#0-0) 

The 32-byte `msg_hash` that all MPC nodes sign is `SHA-256(borsh(ForeignTxSignPayload))`. Because `predecessor_id` is absent, the hash — and therefore the threshold signature — is **identical for every NEAR account** that submits the same foreign-chain request parameters.

**Root cause 2 — Zero tweak used for all foreign-tx signatures.**

In `build_signature_request`, the node-side signing request is constructed with a hardcoded zero tweak:

```rust
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),   // ← no per-caller key derivation
    ...
})
``` [2](#0-1) 

This means every `verify_foreign_transaction` call, regardless of caller, is signed under the **same root key** with no per-caller child-key derivation. (Contrast with `sign()`, where `derive_tweak(predecessor_id, path)` binds the derived key to the caller.)

**Root cause 3 — Pending-request map key excludes `predecessor_id`.**

`VerifyForeignTransactionRequest` — the map key — contains only `request`, `domain_id`, and `payload_version`:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [3](#0-2) 

The contract explicitly queues different callers under the same key and fans out the single MPC response to all of them. This is confirmed by the test comment: *"both yields are queued under the single (caller-agnostic) request key."* [4](#0-3) 

**Root cause 4 — `verify_foreign_transaction` discards `predecessor_id` entirely.**

The contract reads `predecessor_id` only for the deposit refund, then calls `args_into_verify_foreign_tx_request(request)` which produces a `VerifyForeignTransactionRequest` with no caller field: [5](#0-4) 

**Root cause 5 — The SDK verifier has no caller-binding check.**

`ForeignChainSignatureVerifier::verify_signature` only checks that the signature is valid over the expected payload hash. It has no mechanism to verify which NEAR account is entitled to use the proof: [6](#0-5) 

---

### Impact Explanation

A bridge contract on NEAR that uses `verify_foreign_transaction` to gate token minting or fund release cannot rely on the MPC signature to authenticate the specific NEAR caller. Because the signed payload carries no `predecessor_id`, the same MPC attestation is valid for any presenter. An attacker who submits the same foreign-tx verification request (or who observes the response on-chain) obtains an indistinguishable proof and can present it to the bridge contract to claim funds they are not entitled to. This enables **double-spend / unauthorized fund claims** — the exact impact class of "forged foreign-chain verification causing invalid bridge execution."

---

### Likelihood Explanation

The attack requires no special privilege. Any NEAR account can:
1. Observe a pending `verify_foreign_transaction` request (all NEAR state is public)
2. Submit the identical `VerifyForeignTransactionRequestArgs` with 1 yoctoNEAR deposit
3. Receive the same MPC threshold signature via the fan-out queue
4. Present the signature to any bridge contract that uses `ForeignChainSignatureVerifier`

The fan-out queue explicitly supports this pattern (up to 128 concurrent callers per request key). [7](#0-6) 

---

### Recommendation

1. **Include `predecessor_id` in `ForeignTxSignPayload`**: Add the caller's NEAR account ID to `ForeignTxSignPayloadV1` so the MPC signature cryptographically binds to the specific requester. This is analogous to how `sign()` uses `derive_tweak(predecessor_id, path)`.

2. **Include `predecessor_id` in the request map key**: Add `predecessor_id` to `VerifyForeignTransactionRequest` so each caller's request is tracked independently, preventing cross-caller fan-out of the same signature.

3. **Use a per-caller tweak**: Replace `Tweak::new([0u8; 32])` in `build_signature_request` with `derive_foreign_tx_tweak(predecessor_id, derivation_path)` (as already designed in the documentation but not yet implemented).

4. **Update `ForeignChainSignatureVerifier`**: Expose the `predecessor_id` field from the verified payload so downstream contracts can enforce caller-specific authorization.

---

### Proof of Concept

```
1. Alice calls verify_foreign_transaction({tx_id: X, chain: Ethereum, ...}) with 1 yoctoNEAR.
   → Queued under key K = hash(tx_id=X, chain=Ethereum, domain=0, version=V1).

2. Bob calls verify_foreign_transaction({tx_id: X, chain: Ethereum, ...}) with 1 yoctoNEAR.
   → Also queued under the same key K (caller-agnostic).
   [Confirmed by test: verify_foreign_transaction__should_queue_duplicates_from_different_callers]

3. MPC nodes sign ForeignTxSignPayload{request: {tx_id: X, ...}, values: [BlockHash=H]}.
   → tweak = [0u8; 32] (zero, no caller binding).
   → sig = ECDSA(root_key, SHA256(borsh(payload))).

4. respond_verify_foreign_tx() fans out the same (payload_hash, sig) to BOTH Alice and Bob.

5. Bob presents (payload_hash, sig) to the bridge contract.
   → ForeignChainSignatureVerifier::verify_signature() passes (sig is valid over payload_hash).
   → Bridge contract cannot distinguish Bob's claim from Alice's.
   → Bob claims Alice's deposit. Double-spend achieved.
``` [8](#0-7) [9](#0-8) [10](#0-9)

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }

/// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
