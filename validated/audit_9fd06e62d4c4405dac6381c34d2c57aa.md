Let me check where `remove_fast_transfer` is called and what happens after `mark_fast_transfer_as_finalised` for the other-chain case.

### Title
UTXO Replay After Fast-Transfer Fee Claim Bypasses `finalised_utxo_transfers` Guard — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a UTXO fast transfer targets a non-NEAR destination chain, `utxo_fin_transfer_fast` marks the fast-transfer entry as finalised but **never inserts the UTXO into `finalised_utxo_transfers`**. After `claim_fee_callback` subsequently removes that fast-transfer entry, the bridge holds no record of the UTXO in any replay-prevention set. A second submission of the identical `UtxoFinTransferMsg` is accepted as a fresh transfer, producing a second on-chain settlement event and double token credit on the destination chain.

---

### Finding Description

**Step 1 — Fast transfer (UTXO → other chain)**

`fast_fin_transfer_to_other_chain` stores a `FastTransferStatus { finalised: false }` entry in `fast_transfers` and a `TransferMessage` (with `origin_transfer_id = Some(utxo_id)`) in `pending_transfers`. [1](#0-0) 

**Step 2 — UTXO arrives; `utxo_fin_transfer` dispatches to `utxo_fin_transfer_fast`**

Because a fast-transfer entry exists, the function returns early at line 2490 **before** reaching the `add_fin_utxo_transfer` call at line 2494. The UTXO is therefore never recorded in `finalised_utxo_transfers`. [2](#0-1) 

Inside `utxo_fin_transfer_fast`, the `destination != Near` branch calls `mark_fast_transfer_as_finalised` (sets `finalised = true`, keeps the entry) and sends tokens to the relayer. No UTXO finality record is written. [3](#0-2) 

**Step 3 — Fee claim removes the fast-transfer entry**

`claim_fee_callback` removes the `TransferMessage` from `pending_transfers`, then finds the fast-transfer entry with `finalised = true` and calls `remove_fast_transfer`, deleting it entirely. [4](#0-3) 

**After step 3, the bridge state contains:**
- `finalised_utxo_transfers`: does **not** contain the UTXO (never inserted)
- `fast_transfers`: does **not** contain the entry (just removed)

**Step 4 — Second submission of the same UTXO**

`utxo_fin_transfer` is called again with the identical `UtxoFinTransferMsg`:
- `get_fast_transfer_status` returns `None` → no early return
- `add_fin_utxo_transfer` succeeds (UTXO absent from `finalised_utxo_transfers`)
- `utxo_fin_transfer_to_other_chain` creates a new `TransferMessage`, locks tokens, and emits a `UtxoTransferEvent` [5](#0-4) 

The destination chain observes a second valid transfer event and credits tokens again.

**The existing double-finalization test (`fails_on_double_finalization`) does not cover this path** — it only tests a second submission while the fast-transfer entry is still present (which correctly rejects with `ERR_FAST_TRANSFER_ALREADY_FINALISED`). It does not test the post-`claim_fee` window. [6](#0-5) 

---

### Impact Explanation

Double-settlement of the same UTXO produces a second `UtxoTransferEvent` on NEAR. Off-chain relayers observing this event will submit a second finalization proof to the destination chain, minting or releasing tokens a second time with no corresponding locked collateral. This directly breaks the backing guarantee and inflates the bridged token supply.

---

### Likelihood Explanation

The window opens only after `claim_fee_callback` is called for the second-leg transfer. In production the UTXO connector is a separate contract; the bridge relies on `finalised_utxo_transfers` as the authoritative replay guard. The mock connector (`mock-utxo-connector`) has zero replay protection — any account can call `verify_deposit` with any `UtxoFinTransferMsg`. [7](#0-6) 

Even if the production connector has its own guard, the bridge's own invariant is broken: a connector restart, bug, or compromise would allow exploitation. The sequence (fast transfer → UTXO finalization → fee claim → replay) is a normal operational flow, making the window reachable without any privileged access.

---

### Recommendation

In `utxo_fin_transfer_fast`, unconditionally insert the UTXO into `finalised_utxo_transfers` regardless of destination chain, mirroring the normal path:

```rust
// near/omni-bridge/src/lib.rs — utxo_fin_transfer_fast
fn utxo_fin_transfer_fast(...) -> ... {
    require!(!fast_transfer_status.finalised, ...);

    // ADD: record UTXO as finalised before any branching
    let origin_chain = fast_transfer.transfer_id.origin_chain;
    self.add_fin_utxo_transfer(&utxo_fin_transfer_msg.get_transfer_id(origin_chain));

    let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
        self.remove_fast_transfer(&fast_transfer.id());
        fast_transfer.amount
    } else {
        self.mark_fast_transfer_as_finalised(&fast_transfer.id());
        ...
    };
    ...
}
```

This ensures that once a UTXO is processed (fast or slow path), it is permanently recorded in `finalised_utxo_transfers` and any subsequent submission is rejected with `ERR_UTXO_TRANSFER_ALREADY_FINALISED`.

---

### Proof of Concept

```
1. Setup: fast transfer for UTXO X → EVM chain (creates fast_transfers[X] = {finalised:false})
2. Submit UtxoFinTransferMsg(X) via connector → utxo_fin_transfer_fast:
     mark_fast_transfer_as_finalised(X)  // fast_transfers[X].finalised = true
     send_tokens(relayer, amount-fee)
     // finalised_utxo_transfers does NOT contain X
3. call claim_fee with proof of EVM finalization:
     remove_transfer_message(...)
     remove_fast_transfer(X)             // fast_transfers[X] deleted
     // Now: neither fast_transfers nor finalised_utxo_transfers has X
4. Submit identical UtxoFinTransferMsg(X) again via connector:
     get_fast_transfer_status(X) == None  → no early return
     add_fin_utxo_transfer(X)            → SUCCEEDS (X not in set)
     utxo_fin_transfer_to_other_chain(X) → new TransferMessage, new UtxoTransferEvent
5. Destination chain relayer finalizes second event → double token credit
```
