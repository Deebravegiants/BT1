### Title
Pool Operator Can Silently Redirect Farmer to Attacker-Controlled Pool URL Without User Confirmation — (`File: chia/farmer/farmer.py`)

### Summary
The `update_pool_state()` method in `chia/farmer/farmer.py` automatically and silently rewrites the farmer's persisted `pool_url` configuration when the pool server responds to `GET /pool_info` with an HTTP 301 or 308 redirect. No user notification, confirmation, or domain-validation guard is applied before the new URL is written to disk. A malicious or compromised pool operator can exploit this to redirect all future partial submissions to an attacker-controlled pool endpoint, causing the farmer to lose accumulated pool rewards (payout redirection).

### Finding Description

`_pool_get_pool_info()` issues `GET {pool_url}/pool_info` and inspects the final response URL after aiohttp follows redirects: [1](#0-0) 

If the redirect chain consists exclusively of 301 or 308 responses, `new_pool_url` is set to the final destination URL. Back in `update_pool_state()`, this value is written unconditionally and silently to the on-disk `PoolingShareState`: [2](#0-1) 

`PoolingShareState.acquire()` is a context manager that flushes the mutated object back to `pooling_share_state.yaml` on exit: [3](#0-2) 

After the URL is overwritten, every subsequent call to `update_pool_state()` contacts the attacker's endpoint. The farmer then submits signed partials to the attacker's pool via `new_proof_of_space`: [4](#0-3) 

There is no check that the new URL belongs to the same domain, no HTTPS-enforcement re-check against the new URL before writing it, and no user-facing notification of any kind.

### Impact Explanation

Once the farmer's `pool_url` is silently overwritten:

1. All future partials are submitted to the attacker's pool. The attacker accumulates the farmer's proof-of-space contributions.
2. The attacker's pool can refuse to pay out the farmer's accumulated rewards, or record a different `payout_instructions` address, diverting the farmer's pool payout share.
3. The `payout_instructions_update_required` logic at lines 672–675 only triggers a corrective `PUT /farmer` if the attacker's pool returns a mismatched `payout_instructions` in `GET /farmer`; a cooperative attacker simply echoes back the correct address while withholding payment. [5](#0-4) 

This maps directly to the allowed High impact: **pool payout redirection** — the farmer's accumulated pool rewards are diverted to an attacker-controlled pool without the farmer's knowledge or consent.

### Likelihood Explanation

The pool operator is a semi-trusted party the farmer has explicitly joined. However:
- The pool could be malicious from inception, or legitimately acquired/compromised later.
- The redirect fires automatically every `UPDATE_POOL_INFO_INTERVAL` (3600 s) without any user interaction.
- No cryptographic break or key compromise is required — only control of the pool's HTTP server.
- The farmer has no way to detect the change without manually inspecting `pooling_share_state.yaml`. [6](#0-5) 

### Recommendation

1. **Require user confirmation** before persisting a changed `pool_url`. At minimum, log a prominent warning and require an explicit operator acknowledgment (e.g., via the farmer RPC or a config flag).
2. **Validate the redirect domain**: only accept redirects within the same registered domain as the original `pool_url`.
3. **Re-enforce HTTPS** on the new URL before writing it, not just on the original URL.
4. **Emit a `state_changed` event** so the GUI/monitoring layer can alert the farmer operator of the URL change.

### Proof of Concept

1. Farmer joins `https://legitimate-pool.example.com` and `pool_url` is stored in `pooling_share_state.yaml`.
2. The pool operator configures their web server to respond to `GET /pool_info` with an HTTP 301 redirect to `https://attacker.example.com/pool_info`.
3. On the next `update_pool_state()` tick (within one hour), `_pool_get_pool_info()` follows the redirect, detects `new_pool_url = "https://attacker.example.com"`, and `update_pool_state()` writes it to disk without any user prompt:
