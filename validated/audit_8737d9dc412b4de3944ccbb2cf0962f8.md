### Title
Missing Unelected-Committee-Member Vote Check in Consensus-Critical GOV Rule Before Protocol Version 11 — (File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs)

### Summary

Before protocol version 11, the check that prevents unelected Constitutional Committee (CC) members from casting votes exists only in the `MEMPOOL` rule, not in the consensus-critical `GOV` rule. A block producer can bypass the `MEMPOOL` rule entirely and include a transaction containing votes from unelected CC members directly in a block. The `GOV` rule accepts and records these votes. Once the member's election is enacted, their pre-cast vote counts toward ratification of governance actions, enabling unauthorized governance outcomes.

### Finding Description

The codebase contains two parallel implementations of the same authorization check — one in `MEMPOOL` and one in `GOV` — but they are not symmetric across protocol versions.

**In `Mempool.hs` (lines 123–135):**

```haskell
unless (hardforkConwayDisallowUnelectedCommitteeFromVoting protVer) $
  -- This check can completely be removed once mainnet switches to protocol
  -- version 11, since the same check has been implemented in the GOV rule.
  --
  -- Disallow votes by unelected committee members
  failOnNonEmpty
    ( unelectedCommitteeVoters ... )
    (ConwayMempoolFailure . addPrefix . T.pack . show . NE.toList)
```

**In `Gov.hs` (lines 478–481):**

```haskell
when (hardforkConwayDisallowUnelectedCommitteeFromVoting $ pp ^. ppProtocolVersionL) $
  failOnNonEmpty
    (unelectedCommitteeVoters committee committeeState gsVotingProcedures)
    (injectFailure . UnelectedCommitteeVoters)
```

The protocol-version gate is inverted between the two rules:

| Protocol Version | MEMPOOL check active? | GOV check active? |
|---|---|---|
| < 11 (e.g., PV 10) | YES | **NO** |
| >= 11 | NO | YES |

`hardforkConwayDisallowUnelectedCommitteeFromVoting` is defined as:

```haskell
hardforkConwayDisallowUnelectedCommitteeFromVoting pv = pvMajor pv > natVersion @10
``` [1](#0-0) 

The `MEMPOOL` rule is only invoked for transactions entering the node's local mempool. It is **not** part of the consensus-critical block-validation path. A block producer constructs blocks by calling the `LEDGER` rule (which calls `GOV`) directly, bypassing `MEMPOOL` entirely. Before PV 11, the `GOV` rule contains no check for unelected committee voters, so any such vote is silently accepted and written into the governance proposals state. [2](#0-1) [3](#0-2) 

`authorizedElectedHotCommitteeCredentials` restricts the authorized set to hot credentials whose cold key is present in the **currently enacted** committee:

```haskell
authorizedElectedHotCommitteeCredentials committee committeeState =
  case committee of
    SNothing -> Set.empty
    SJust electedCommiteee ->
      authorizedHotCommitteeCredentials $
        CommitteeState $
          csCommitteeCreds committeeState `Map.intersection` committeeMembers electedCommiteee
``` [4](#0-3) 

A member whose election proposal is pending but not yet enacted is therefore "unelected" — their cold key is not yet in `committeeMembers` of the enacted committee. Before PV 11, the `GOV` rule uses `authorizedHotCommitteeCredentials` (line 474) — which does not filter by election status — only for the `VotersDoNotExist` check, not for the unelected-member check. The unelected-member check is entirely absent from `GOV` before PV 11. [5](#0-4) 

### Impact Explanation

The test `GovCertSpec.hs` "at protocol version 10, the vote is counted once the committee update is enacted" (lines 210–228) explicitly confirms the impact:

1. A `TreasuryWithdrawals` governance action is proposed.
2. A proposed-but-not-yet-enacted CC member (`proposedMember`) registers a hot key and casts a YES vote via `submitYesVote_` (which goes through the full `LEDGER`/`GOV` path, not just `MEMPOOL`).
3. Before the election is enacted: `isCommitteeAccepted gaiWithdrawal` returns `False`.
4. After one epoch (election enacted): `isCommitteeAccepted gaiWithdrawal` returns `True`.
5. The treasury withdrawal governance action is subsequently ratified and enacted. [6](#0-5) 

This means an unelected CC member's vote, once recorded in the governance state, counts toward ratification of governance actions (treasury withdrawals, parameter changes, hard-fork initiations, new constitutions) as soon as their election is enacted. If the unelected member's vote is the deciding vote, a governance action that would otherwise fail ratification is enacted.

The impact matches: **Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.**

### Likelihood Explanation

The attacker requires two capabilities:

1. **Control of an unelected CC member's hot key**: An adversary who has submitted (or colluded with the submitter of) an `UpdateCommittee` proposal adding their cold key, and has registered a hot key, satisfies this. This is a normal, permissionless on-chain action.
2. **Block production**: The attacker must produce a block containing the vote transaction, bypassing `MEMPOOL`. A stake pool operator with any non-zero stake can produce blocks. No consensus majority is required — a single block suffices.

These two roles can be held by the same entity (a stake pool operator who is also a proposed CC member) or by colluding parties. The on-chain actions required (submitting an `UpdateCommittee` proposal, registering a hot key, producing a block) are all permissionless and require no privileged access.

Likelihood: **Medium** — requires block production capability combined with a pending CC election, both of which are reachable without privileged access.

### Recommendation

The `unelectedCommitteeVoters` check must be present in the consensus-critical `GOV` rule for **all** protocol versions in which the Conway governance system is active, not only from PV 11 onward. The `when (hardforkConwayDisallowUnelectedCommitteeFromVoting ...)` guard in `conwayGovTransition` should be removed, making the check unconditional. The corresponding `unless` guard in `mempoolTransition` can then also be removed as the comment already anticipates.

### Proof of Concept

The existing test suite directly demonstrates the bypass. In `LedgerSpec.hs` lines 313–318, at PV 10:

```haskell
txFixed <-
  submitFailingMempoolTx "unallowed votes" tx $
    NonEmpty.singleton . injectFailure . ConwayMempoolFailure $
      "Unelected committee members are not allowed to cast votes: " <> ...
withNoFixup $ submitTx_ txFixed   -- succeeds: GOV has no check before PV 11
``` [7](#0-6) 

`submitFailingMempoolTx` confirms the `MEMPOOL` rule rejects the transaction. `withNoFixup $ submitTx_` submits the same transaction through the full `LEDGER`→`GOV` path and it **succeeds**, confirming the `GOV` rule has no equivalent check before PV 11. Combined with the `GovCertSpec.hs` test showing the vote is subsequently counted once the election is enacted, the full exploit chain is proven within the existing test suite.
