### Title
Missing Response Verification for Privately-Verifiable CKD Variant Allows Single Byzantine Participant to Forge CKD Output - (File: crates/contract/src/lib.rs)

### Summary
The `respond_ckd` contract function performs cryptographic output verification only for the `AppPublicKeyPV` (publicly verifiable) variant of CKD requests. For the `AppPublicKey` (privately verifiable, legacy) variant, the match arm is empty — no on-chain check is performed. A single attested MPC participant, strictly below the signing threshold, can race to call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending privately-verifiable CKD request. The contract accepts it unconditionally, resolves the yield, and delivers the forged output to the user. The request is permanently consumed and cannot be retried.

### Finding Description
In `crates/contract/src/lib.rs`, the `respond_ckd` function contains the following match block:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing relationship between the response, the app's public key pair, and the MPC network's BLS public key. For `AppPublicKey`, the arm is a no-op. After this match, `pending_requests::resolve_yields_for` is called unconditionally, resolving the yield and delivering whatever `CKDResponse` was supplied. [2](#0-1) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active()`, which requires the caller to be a currently attested MPC participant — a single participant, not a threshold quorum. [3](#0-2) 

By contrast, `respond` (for signature requests) always verifies the cryptographic signature before resolving any yield, regardless of scheme: [4](#0-3) 

### Impact Explanation
A single Byzantine MPC participant (strictly below the signing threshold) can:
1. Observe any pending `AppPublicKey` CKD request on-chain.
2. Immediately call `respond_ckd` with a crafted `CKDResponse` containing arbitrary `big_y` and `big_c` values.
3. The contract accepts the response without any cryptographic check, resolves the yield, and delivers the forged encrypted key material to the user's callback.
4. The yield is permanently consumed; the user cannot resubmit the same request and receive a correct response.

The user's TEE application receives an incorrect encrypted blob. Depending on the application, this can cause silent key corruption (the app derives a wrong key and uses it for security-sensitive operations) or a hard failure (decryption produces garbage). Either outcome constitutes unauthorized confidential key derivation output delivered without the required threshold participant authorization.

This matches the allowed impact: **Critical — Unauthorized confidential key derivation output without the required participant authorization**, and at minimum **Medium — request-lifecycle manipulation that breaks production safety invariants**.

### Likelihood Explanation
Any single attested MPC participant can exploit this. The attack requires no collusion, no privileged operator access, and no network-level capability beyond submitting a NEAR transaction. The attacker only needs to front-run the honest leader's `respond_ckd` call, which is straightforward since NEAR transaction ordering within a block is observable. The `AppPublicKey` variant is the legacy/default variant and is actively used in production.

### Recommendation
Add the same style of cryptographic output verification for the `AppPublicKey` variant in `respond_ckd`. For the privately verifiable variant, the contract already holds the BLS network public key (`public_key` extracted at line 668–673). A verification step should confirm that `big_y = msk · H(app_id)` and `big_c = big_y + a·big_y` (i.e., the standard CKD correctness relation) using the network public key, analogous to what `ckd_output_check` does for the PV variant. Alternatively, deprecate the unverified `AppPublicKey` variant and require all new requests to use `AppPublicKeyPV`. [5](#0-4) 

### Proof of Concept
1. User submits `request_app_private_key` with `app_public_key: AppPublicKey(pk)` and `domain_id` pointing to a BLS CKD domain. A yield is registered in `pending_ckd_requests`.
2. Attacker (a single attested MPC participant) observes the pending request on-chain.
3. Attacker calls `respond_ckd(ckd_request, CKDResponse { big_y: [1u8;48].into(), big_c: [2u8;48].into() })`.
4. `assert_caller_is_signer()` passes (attacker is a signer). `is_running_or_resharing()` passes. `accept_requests` is true. `assert_caller_is_attested_participant_and_protocol_active()` passes.
5. The match on `app_public_key` hits the `AppPublicKey(_) => {}` arm — no check performed.
6. `resolve_yields_for` resolves the yield with the forged response bytes.
7. The user's callback receives `CKDResponse { big_y: [1u8;48], big_c: [2u8;48] }` — attacker-controlled garbage — as the confidential key derivation output.
8. The honest MPC nodes' subsequent `respond_ckd` call finds no pending yield and silently does nothing (the request is already consumed). [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
