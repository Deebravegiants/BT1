Audit Report

## Title
Zero rsETH Minted on Dust Deposit Due to Unchecked Integer Division Truncation — (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`)

## Summary
Every L2 pool deposit function computes `rsETHAmount` via integer division that rounds down. When the deposited amount (after fee) is smaller than `rsETHToETHrate / 1e18` (ETH path) or `rsETHToETHrate / tokenToETHRate` (token path), the division truncates to zero. The deposit proceeds without reverting, the user's ETH or token is consumed by the pool, and zero rsETH/wrsETH is minted or transferred. No minimum-output guard exists in any L2 pool deposit path, unlike the L1 `LRTDepositPool` which enforces `minRSETHAmountExpected`.

## Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee` (L299–308) and `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee` (L277–286), the ETH-path output is computed as:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

When `amountAfterFee * 1e18 < rsETHToETHrate`, Solidity integer division yields `rsETHAmount = 0`.

The calling deposit functions perform only an input check (`if (amount == 0) revert InvalidAmount()`) and then unconditionally proceed:

- `RSETHPoolV3.deposit` (L256–264): calls `wrsETH.mint(msg.sender, 0)` — user receives nothing.
- `RSETHPoolNoWrapper.deposit` (L231–244): calls `rsETH.safeTransfer(msg.sender, 0)` — user receives nothing.

The deposited ETH remains in the pool, is eventually bridged to L1, and increases the total ETH backing rsETH without increasing rsETH supply — silently enriching existing holders at the depositor's expense. The same truncation applies to the token path (`amountAfterFee * tokenToETHRate / rsETHToETHrate`) in both pool variants. By contrast, `LRTDepositPool._beforeDeposit` (L665–669) enforces `if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet()`, a guard entirely absent from all L2 pool variants.

## Impact Explanation
**Low — Contract fails to deliver promised returns.** A depositor sends a non-zero ETH or token amount and receives zero rsETH/wrsETH. The deposited value is permanently absorbed into the pool's ETH balance, increasing the backing for all existing rsETH holders without minting new supply. The depositor suffers a permanent, irrecoverable loss of their deposited amount with no on-chain recourse.

## Likelihood Explanation
For ETH deposits, the rounding threshold is `amountAfterFee < rsETHToETHrate / 1e18 ≈ 1 wei` (since `rsETHToETHrate ≈ 1.0x × 1e18`), making accidental triggering extremely rare for normal deposits. For token deposits with low token-to-ETH rates (e.g., `tokenToETHRate = 1e15`, i.e., 0.001 ETH/token), the threshold rises to ~1050 token-wei, which is more meaningful for 6-decimal tokens. Any unprivileged user can trigger this by depositing a dust amount; no special role or precondition is required. The absence of any `minRsETHAmountExpected` parameter means users have no on-chain slippage protection regardless of deposit size.

## Recommendation
Add a zero-output guard in every pool deposit function after computing `rsETHAmount`, mirroring the protection already present in `LRTDepositPool._beforeDeposit`:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

Optionally, expose a `minRsETHAmountExpected` parameter so callers can enforce their own slippage tolerance, consistent with `LRTDepositPool.depositETH` / `depositAsset`.

## Proof of Concept
Assume `rsETHToETHrate = 1.05e18` (rsETH has accrued 5% staking yield) and `feeBps = 0`.

1. User calls `RSETHPoolV3.deposit{value: 1}("")` — deposits 1 wei of ETH.
2. `viewSwapRsETHAmountAndFee(1)` computes:
   - `fee = 1 * 0 / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation)
3. `feeEarnedInETH += 0`
4. `wrsETH.mint(msg.sender, 0)` — user receives 0 wrsETH
5. The 1 wei ETH remains in the pool; transaction succeeds with no revert.

**Foundry fuzz test plan:**
```solidity
