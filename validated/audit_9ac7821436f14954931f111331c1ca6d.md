Audit Report

## Title
Abandoned Unlocked Withdrawals Temporarily Freeze Assets in `LRTWithdrawalManager` and Block `sweepRemainingAssets()` — (`contracts/LRTWithdrawalManager.sol`)

## Summary
After `unlockQueue()` irrevocably burns rsETH and pulls underlying assets from `LRTUnstakingVault` into `LRTWithdrawalManager`, users face no on-chain obligation to call `completeWithdrawal()`. Any user who initiates a withdrawal and then never claims leaves assets frozen in `LRTWithdrawalManager`, permanently blocks `sweepRemainingAssets()` for the affected asset, and forces the operator to pay unrecoverable gas via `completeWithdrawalForUser()`. For ETH withdrawals where the user address is a contract that reverts on ETH receipt, even the operator's force-completion path reverts, making the freeze permanent.

## Finding Description
The withdrawal lifecycle has three phases. In phase 2, `unlockQueue()` atomically burns rsETH held in the contract and redeems the underlying asset from the vault:

```solidity
// LRTWithdrawalManager.sol L305-307
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);
```

Simultaneously, `_unlockWithdrawalRequests()` increments `unlockedWithdrawalsCount[asset]` for each unlocked request (L809). After this point, rsETH is gone and the underlying asset sits in `LRTWithdrawalManager`. There is no timeout, no cancellation path, and no mechanism to redirect assets back to the vault.

The only way to decrement `unlockedWithdrawalsCount[asset]` is through `_processWithdrawalCompletion()` (L717), which is called by either `completeWithdrawal()` (user-initiated) or `completeWithdrawalForUser()` (operator-initiated). `sweepRemainingAssets()` hard-reverts if any unlocked withdrawals remain:

```solidity
// LRTWithdrawalManager.sol L402-403
if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();
```

If a user never calls `completeWithdrawal()` — for any reason (lost wallet, economic abandonment, deliberate griefing) — the operator must call `completeWithdrawalForUser()` for each abandoned request. For ETH withdrawals, `_transferAsset()` uses a raw `.call{value}()` and reverts on failure (L877-879), so if the user's address is a contract that rejects ETH, even the operator's force-completion path is permanently blocked, making the ETH freeze irrecoverable.

## Impact Explanation
**Medium — Temporary freezing of funds** (escalating to **Critical — Permanent freezing of funds** for ETH with a rejecting-contract user address).

For ERC20 assets (stETH, ETHx): assets are frozen in `LRTWithdrawalManager`, not in the vault or EigenLayer, not earning yield, and `sweepRemainingAssets()` is blocked until the operator force-completes every abandoned request at their own gas expense. The freeze is temporary only because the operator has a recovery path.

For ETH: if the user address is a contract that reverts on ETH receipt, `completeWithdrawalForUser()` itself reverts (L877-879), and there is no alternative transfer path. The ETH is permanently locked in `LRTWithdrawalManager` with no on-chain recovery mechanism.

## Likelihood Explanation
Any user who initiates a withdrawal and simply does not call `completeWithdrawal()` triggers this — no special conditions, no depeg required. Lost wallets, forgotten claims, or deliberate abandonment all suffice. The ETH permanent-freeze variant requires the initiating address to be a contract that rejects ETH, which is a realistic pattern for smart-contract wallets or multisigs without a fallback. The condition is reachable by any unprivileged external caller through the normal `initiateWithdrawal()` → `unlockQueue()` → (no `completeWithdrawal()`) sequence.

## Recommendation
1. **Timeout/reclaim mechanism**: After a configurable grace period post-unlock, allow the operator (or anyone) to redirect unclaimed assets back to `LRTUnstakingVault` and re-mint rsETH to the original user's address, decrementing `unlockedWithdrawalsCount` and unblocking the queue.
2. **ETH-specific fallback**: For ETH force-completions, wrap the transfer in a try/catch or use a pull-payment pattern (deposit to a claimable escrow) so that a rejecting contract address does not permanently brick the operator's recovery path.
3. **Alternative `sweepRemainingAssets()` guard**: Consider allowing the sweep after a timeout even if `unlockedWithdrawalsCount > 0`, provided the unclaimed amounts are accounted for separately.

## Proof of Concept
1. User (or a contract that rejects ETH) calls `initiateWithdrawal(ETH, 1e18, "")` — rsETH transferred to `LRTWithdrawalManager`.
2. Operator calls `unlockQueue(ETH, ...)` — rsETH burned (L305), ETH pulled from vault (L307), `unlockedWithdrawalsCount[ETH]` incremented to 1 (L809).
3. User never calls `completeWithdrawal()`.
4. ETH is frozen in `LRTWithdrawalManager`; `unlockedWithdrawalsCount[ETH] == 1`.
5. Manager calls `sweepRemainingAssets(ETH)` → reverts with `PendingWithdrawalsExist` (L403).
6. Operator calls `completeWithdrawalForUser(ETH, user, "")` → `_transferAsset` executes `.call{value}()` to the rejecting contract → `sent == false` → reverts with `EthTransferFailed` (L879). ETH is permanently frozen.

For ERC20 assets, step 6 succeeds but the operator bears unrecoverable gas costs for every abandoned request, and assets remain non-yield-bearing until force-completed.
