Audit Report

## Title
Missing Minimum Output Slippage Guard on L2 Pool Deposits - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPool.sol)

## Summary
The L2 pool deposit functions in `RSETHPoolV3` and `RSETHPool` accept no `minRsETHAmountExpected` parameter and perform no minimum-output check before minting or transferring rsETH to the caller. The minted amount is derived from a live oracle rate that can increase between transaction submission and execution, leaving depositors with no on-chain protection against receiving fewer rsETH tokens than they observed when constructing the transaction.

## Finding Description
`RSETHPoolV3.deposit(string)` and `RSETHPoolV3.deposit(address,uint256,string)` both call `viewSwapRsETHAmountAndFee`, which reads the live oracle rate via `getRate()`, and immediately mint the resulting `rsETHAmount` with no floor check:

```solidity
// RSETHPoolV3.sol L258-262
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
```

`viewSwapRsETHAmountAndFee` computes the output as `amountAfterFee * 1e18 / rsETHToETHrate` where `rsETHToETHrate = getRate()` is a live cross-chain oracle read. Because rsETH/ETH rate increases monotonically as staking rewards accrue, any rate update pushed on-chain between tx submission and execution silently reduces the minted amount.

The same pattern is present in `RSETHPool.deposit(string)` (L265-278) and `RSETHPool.deposit(address,uint256,string)` (L284-305), which use `safeTransfer` instead of `mint` but apply the same unguarded oracle-derived amount.

By contrast, `LRTDepositPool._beforeDeposit` (L648-670) explicitly enforces:
```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```
showing the protocol already has the correct pattern on L1 but omitted it on L2.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A depositor who previewed a rate and submitted a transaction expecting a specific rsETH amount may receive materially fewer tokens if the oracle rate is updated upward before the transaction lands. The deposited ETH is not lost (it is held in the pool and eventually bridged to L1), but the user receives fewer rsETH than intended with no on-chain recourse. This matches the allowed impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
The rsETH/ETH exchange rate increases continuously as staking rewards accrue and is pushed on-chain by the cross-chain rate provider on a regular schedule. Any deposit transaction that sits in the mempool across a rate update will be affected. On congested L2s, this window can span several minutes. The entry path is fully permissionless — any depositor calling `deposit()` is exposed without any special preconditions.

## Recommendation
Add a `minRsETHAmountExpected` parameter to both `deposit` overloads in `RSETHPoolV3` and `RSETHPool`, mirroring the pattern already used in `LRTDepositPool._beforeDeposit`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    wrsETH.mint(msg.sender, rsETHAmount);
    ...
}
```

Apply the same guard to the token deposit overload and to the equivalent functions in `RSETHPool`.

## Proof of Concept
1. Oracle rate is 1.05 ETH/rsETH. User calls `RSETHPoolV3.deposit{value: 1 ether}("ref")` expecting ≈952 rsETH (after fee).
2. Transaction sits in the mempool. The cross-chain rate provider pushes a new rate of 1.10 ETH/rsETH.
3. Transaction executes. `viewSwapRsETHAmountAndFee(1 ether)` now returns ≈909 rsETH.
4. `wrsETH.mint(msg.sender, 909e18)` is called with no revert — the user receives ~43 fewer rsETH than expected (~4.5% shortfall) with no on-chain protection.

**Contrast:** The same user depositing via `LRTDepositPool.depositETH(952e18, "ref")` would have the transaction revert at `_beforeDeposit` line 667 (`rsethAmountToMint < minRSETHAmountExpected`), protecting them from the adverse rate.

A Foundry fork test can reproduce this by: (a) deploying/forking the pool, (b) recording the initial oracle rate, (c) pranking the oracle to return a higher rate, (d) calling `deposit` and asserting the minted amount is below the originally expected value with no revert.     function _beforeDeposit(
        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
