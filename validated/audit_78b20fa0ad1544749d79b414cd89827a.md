Audit Report

## Title
`SfrxETHPriceOracle.getAssetPrice()` Returns sfrxETH/frxETH Rate Instead of sfrxETH/ETH Rate, Enabling Over-Minting of rsETH - (File: contracts/oracles/SfrxETHPriceOracle.sol)

## Summary
`SfrxETHPriceOracle.getAssetPrice()` delegates entirely to `ISfrxETH.pricePerShare()`, which is the ERC4626 vault share price denominated in **frxETH**, not ETH. Because frxETH is a distinct LST that can trade at a discount to ETH on secondary markets, the oracle systematically overstates the sfrxETH/ETH exchange rate. This inflated rate propagates into rsETH minting and the rsETH price calculation, allowing any depositor to receive more rsETH than the true ETH value of their sfrxETH, diluting existing holders and risking protocol insolvency.

## Finding Description

`SfrxETHPriceOracle.getAssetPrice()` is implemented as:

```solidity
// contracts/oracles/SfrxETHPriceOracle.sol L35-41
function getAssetPrice(address asset) external view returns (uint256) {
    if (asset != sfrxETHContractAddress) {
        revert InvalidAsset();
    }
    return ISfrxETH(sfrxETHContractAddress).pricePerShare();
}
```

The interface comment at line 9 of the same file is self-contradictory and confirms the denomination error:

```solidity
/// @notice How much frxETH is 1E18 sfrxETH worth. Price is in ETH, not USD
```

The first sentence correctly states the return value is in **frxETH**; the second sentence incorrectly claims it is in ETH. `pricePerShare()` is the standard ERC4626 function returning the amount of the underlying asset (frxETH) per share (sfrxETH). frxETH is Frax's synthetic LST — a separate ERC20 token that can and does trade at a discount to ETH.

**Exploit path:**

When frxETH trades at `P_frx` ETH (where `P_frx < 1`):

- True sfrxETH/ETH rate = `pricePerShare() × P_frx`
- Oracle-reported rate = `pricePerShare()` (inflated by factor `1/P_frx`)

**Path 1 — rsETH minting** (`LRTDepositPool.getRsETHAmountToMint`, line 520):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The numerator uses the inflated `getAssetPrice(sfrxETH)`, so the depositor receives `1/P_frx` times more rsETH than the true ETH value of their deposit.

**Path 2 — rsETH price update** (`LRTOracle._getTotalEthInProtocol`, lines 339–343):

```solidity
uint256 assetER = getAssetPrice(asset);
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```

The total ETH backing rsETH is overstated by the sfrxETH holdings multiplied by the inflated rate, which inflates `rsETHPrice`, compounding the over-minting effect.

No existing guard corrects for this: `updatePriceOracleForValidated` only checks that the price is between `1e16` and `1e19` (a sanity range), not that it is denominated in ETH.

## Impact Explanation

**Critical — Direct theft of user funds / Protocol insolvency.**

When frxETH trades at `P_frx < 1 ETH`:
1. Attacker deposits `N` sfrxETH with true ETH value `N × pricePerShare() × P_frx`.
2. Oracle prices it at `N × pricePerShare()` (inflated).
3. Attacker receives rsETH worth `N × pricePerShare()` in rsETH terms.
4. On redemption, attacker extracts `N × pricePerShare() × (1 - P_frx)` ETH more than deposited, directly from the pool at the expense of all other rsETH holders.

At scale this constitutes direct theft of funds from existing rsETH holders and, if the depeg is sustained, renders the protocol insolvent (rsETH backed by less ETH than its stated price implies). This matches the Critical impact class: "Direct theft of any user funds."

## Likelihood Explanation

frxETH has historically depegged 0.5–2% from ETH during periods of market stress or liquidity crises. The vulnerability is always present — it requires no special access, no governance compromise, and no setup beyond the frxETH/ETH spot price being below 1. Any unprivileged external caller can trigger it by calling `LRTDepositPool.depositAsset(sfrxETH, amount, 0, "")`. The exploit is repeatable for as long as the depeg persists and deposit limits are not exhausted.

## Recommendation

Replace the single `pricePerShare()` call with a two-leg calculation that multiplies the sfrxETH/frxETH vault rate by the frxETH/ETH market rate from a Chainlink feed:

```solidity
uint256 sfrxETHPerFrxETH = ISfrxETH(sfrxETHContractAddress).pricePerShare();
uint256 frxETHPerETH = IChainlinkFeed(frxETHEthFeed).latestAnswer(); // 18-decimal feed
return sfrxETHPerFrxETH * frxETHPerETH / 1e18;
```

Alternatively, use a direct Chainlink `sfrxETH/ETH` feed if one is available, bypassing the intermediate frxETH step entirely. The `ChainlinkPriceOracle` contract already exists in the codebase and can be reused for this purpose.

## Proof of Concept

1. Deploy a fork of mainnet with frxETH trading at 0.99 ETH (a historically observed depeg level).
2. Note `SfrxETHPriceOracle.getAssetPrice(sfrxETH)` returns `sfrxETH.pricePerShare()` ≈ 1.05e18 (frxETH per sfrxETH), confirmed by the interface comment at `contracts/oracles/SfrxETHPriceOracle.sol` line 9: *"How much frxETH is 1E18 sfrxETH worth."*
3. True sfrxETH/ETH = 1.05 × 0.99 = 1.0395e18; oracle reports 1.05e18 — a ~1% overstatement.
4. Call `LRTDepositPool.depositAsset(sfrxETH, 1000e18, 0, "")`. The minting formula at line 520 of `contracts/LRTDepositPool.sol` mints `(1000e18 × 1.05e18) / rsETHPrice` rsETH instead of `(1000e18 × 1.0395e18) / rsETHPrice`.
5. Redeem the rsETH. The attacker recovers ~1% more ETH-equivalent than deposited, extracted from the pool.
6. Foundry fork test: assert `oracle.getAssetPrice(sfrxETH) > sfrxETH.pricePerShare() * frxETHEthPrice / 1e18` when `frxETHEthPrice < 1e18`, and assert the attacker's post-redemption ETH balance exceeds their pre-deposit ETH balance. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
