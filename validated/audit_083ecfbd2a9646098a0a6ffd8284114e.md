Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Limit to Be Exceeded - (File: contracts/LRTDepositPool.sol)

## Summary

`_checkIfDepositAmountExceedesCurrentLimit()` in `LRTDepositPool.sol` uses an asymmetric comparison for ETH versus ERC20 tokens. The ETH branch omits the incoming `amount` from the comparison, meaning the deposit cap is only enforced after it has already been exceeded, not at the boundary. Any unprivileged depositor can push total ETH deposits above the configured limit.

## Finding Description

The function `_checkIfDepositAmountExceedesCurrentLimit` at lines 676–682 contains the sole pre-deposit cap check:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for ERC20
}
``` [1](#0-0) 

For ERC20 assets the post-deposit total (`totalAssetDeposits + amount`) is compared against the limit — correctly blocking any deposit that would breach the cap. For ETH, only the pre-deposit total is compared. When `totalAssetDeposits == limit` the expression `totalAssetDeposits > limit` evaluates to `false`, the check passes, and the deposit is accepted, pushing the total above the cap by the full deposit amount.

This check is called unconditionally from `_beforeDeposit` at line 661, which gates both `depositETH` and `depositAsset`. [2](#0-1) 

## Impact Explanation

The ETH deposit cap configured by the admin via `updateAssetDepositLimit` is not enforced for the deposit that reaches the limit. The protocol fails to deliver its promised deposit-limit guarantee for ETH. No funds are directly stolen or frozen, placing this squarely in the **Low — Contract fails to deliver promised returns, but doesn't lose value** impact class.

## Likelihood Explanation

No special privileges, coordination, or front-running are required. The condition is triggered naturally as the protocol fills: the moment cumulative ETH deposits equal the configured cap, the very next `depositETH` call of any size bypasses the limit. Any ordinary depositor calling `depositETH` at that moment triggers it.

## Recommendation

Apply the same formula used for ERC20 tokens to the ETH branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [3](#0-2) 

## Proof of Concept

1. Admin calls `updateAssetDepositLimit(ETH_TOKEN, 100 ether)` — cap is 100 ETH.
2. Users deposit until `getTotalAssetDeposits(ETH_TOKEN) == 100 ether`.
3. Any user calls `depositETH{value: 50 ether}(referralId)`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50 ether)`:
   - `totalAssetDeposits = 100 ether`
   - ETH branch: `100 ether > 100 ether` → `false` → deposit is **not** blocked.
5. 50 ETH is accepted; `getTotalAssetDeposits(ETH_TOKEN)` becomes 150 ETH — 50 ETH above the cap.
6. For comparison, an ERC20 deposit at the same state evaluates `100 ether + 50 ether > 100 ether` → `true` → correctly reverted.

**Foundry test plan:** Deploy `LRTDepositPool` on a local fork, set ETH deposit limit to `X`, deposit `X` ETH via multiple calls, then assert that a further `depositETH` call reverts with `MaximumDepositLimitReached`. With the current code the assertion fails; after applying the fix it passes.
