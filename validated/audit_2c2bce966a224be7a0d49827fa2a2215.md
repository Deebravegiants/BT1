### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Single-Participant Forged Foreign-Chain Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is valid over `response.payload_hash` using the root public key, but never verifies that `response.payload_hash` is derived from the pending `request`. Because `payload_hash` is entirely caller-supplied, a single Byzantine attested participant can replay any previously observed valid `(payload_hash, signature)` pair as a response to a different pending request, causing the contract to deliver a forged foreign-chain verification result to the waiting user.

---

### Finding Description

**Vulnerable function** — `respond_verify_foreign_tx`, `crates/contract/src/lib.rs` lines 691–754:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← fully attacker-controlled

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,                                            // ← ROOT key, not derived
)
.is_ok()
```

The contract checks only that `signature` is a valid ECDSA signature over `payload_hash` under the root public key. It does **not** check that `payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the specific `request` being resolved.

**Contrast with `respond`** (lines 597–608), which derives the expected public key from the request-specific tweak:

```rust
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak)...;   // tweak = H(predecessor, path)
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,                               // ← DERIVED key, bound to request
)
```

In `respond`, the tweak cryptographically binds the signature to the specific request. In `respond_verify_foreign_tx`, no such binding exists.

**Analog to HoneyJarPortal**: The `_from != _msgSender()` check appeared to verify NFT ownership but was redundant because `_from` was never used in the burn. Here, the signature check appears to validate the response's integrity but is redundant as a *binding* check because `payload_hash` is never tied to the `request`.

**Attack path** (single Byzantine attested participant, no threshold collusion required):

1. Observe a previous on-chain `respond_verify_foreign_tx` call for request R1 that produced `(payload_hash_R1, sig_R1)` — both are public on-chain data.
2. Submit a new `verify_foreign_transaction` for a different (e.g., non-existent or fraudulent) foreign-chain transaction R2.
3. Call `respond_verify_foreign_tx(R2, { payload_hash: payload_hash_R1, signature: sig_R1 })`.
4. Contract checks: `verify_ecdsa_signature(sig_R1, payload_hash_R1, root_pk)` → **valid** ✓ (signature was honestly produced for R1).
5. Contract calls `resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &R2, ...)` → resolves R2's yield with the forged response.
6. User's contract receives `VerifyForeignTransactionResponse { payload_hash: payload_hash_R1, signature: sig_R1 }` for request R2.

The user's contract receives a cryptographically valid signature over a hash that does not correspond to R2's actual foreign-chain data.

---

### Impact Explanation

**Critical/High — Forged foreign-chain verification enabling invalid bridge execution.**

A user's contract that relies on `verify_foreign_transaction` to gate a bridge action (e.g., releasing funds after confirming a Bitcoin deposit) receives a response whose `payload_hash` does not correspond to the transaction it submitted. If the user contract does not independently reconstruct and compare the expected hash (which the docs note is the caller's responsibility but the MPC contract does not enforce), it will accept a forged attestation. This enables double-spend or invalid bridge execution: an attacker can make the MPC contract "confirm" a fraudulent or already-used foreign-chain transaction.

---

### Likelihood Explanation

**High.** The attack requires only:
- Being a single attested participant (no threshold collusion).
- Access to any previously published `(payload_hash, signature)` pair, which is permanently visible on-chain from any prior `respond_verify_foreign_tx` call.

No special cryptographic capability is needed. The attacker does not need to produce a new signature — they replay an existing one.

---

### Recommendation

The contract must bind `response.payload_hash` to the pending `request`. Two approaches:

1. **Include extracted values in the response**: Change `VerifyForeignTransactionResponse` to include the `values` field alongside `payload_hash`. The contract then recomputes `expected_hash = SHA-256(borsh(ForeignTxSignPayload { request, values }))` and asserts `response.payload_hash == expected_hash` before accepting the signature.

2. **Derive a request-specific key** (analogous to `respond`): Derive a tweak from the `VerifyForeignTransactionRequest` (e.g., `H(domain_id, chain, tx_id, extractors)`) and require the MPC nodes to sign with the derived key rather than the root key. This cryptographically binds the signature to the specific request, making replay impossible.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
