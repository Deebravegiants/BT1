Audit Report

## Title
Integer Division Truncation Allows Zero agETH Minted for Small Token Deposits — (`contracts/agETH/AGETHPoolV3.sol`)

## Summary
`deposit(address token, uint256 amount, string referralId)` transfers the user's tokens before computing `agETHAmount` via integer division. When `amountAfterFee * tokenToETHRate < agETHToETHrate`, the division truncates to zero. No post-computation guard exists, so `agETH.mint(msg.sender, 0)` is called silently — the user's tokens are consumed and zero agETH is returned.

## Finding Description
In `viewSwapAgETHAmountAndFee(uint256 amount, address token)` at [1](#0-0)  the computation is:

```solidity
agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate;
```

Both rates are 1e18-scaled. Truncation to zero occurs when `amountAfterFee * tokenToETHRate < agETHToETHrate`.

The ETH-path overload at [2](#0-1)  uses `amountAfterFee * 1e18 / agETHToETHrate`, which is equivalent to the token path only when `tokenToETHRate == 1e18`. For tokens with a lower rate (e.g., 6-decimal USDC where `tokenToETHRate ≈ 3e14`), the truncation threshold is orders of magnitude higher.

In `deposit(address token, ...)`, the only input validation is `amount == 0` at [3](#0-2) . The token transfer executes at line 145 **before** `agETHAmount` is computed at line 147, and there is no check that `agETHAmount > 0` before the mint at line 151. [4](#0-3) 

A standard ERC-20 `mint(addr, 0)` does not revert, so the entire call succeeds silently with the user's tokens permanently held by the pool.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A depositor who sends a non-zero but below-threshold token amount loses their tokens: the contract accepts the transfer, mints zero agETH, and emits a `SwapOccurred` event with `agETHAmount = 0`. The tokens remain in the pool (no protocol-level loss), but the user receives nothing, violating the invariant that any non-zero deposit must yield a non-zero output.

## Likelihood Explanation
The truncation threshold is `amountAfterFee < agETHToETHrate / tokenToETHRate`.

- **18-decimal token** (e.g., stETH, `tokenToETHRate ≈ 0.9e18`): threshold ≈ 1–2 wei. Only dust amounts are affected; practically negligible.
- **6-decimal token** (e.g., USDC, `tokenToETHRate ≈ 3e14` at ~$3,000/ETH): threshold ≈ `1.1e18 / 3e14 ≈ 3,667` wei ≈ 0.003667 USDC. Any deposit below this amount silently yields zero agETH. This is a realistic user mistake (UI rounding, copy-paste of a small test amount, etc.).

The function is public and callable by any unprivileged user with no special preconditions beyond holding a supported token.

## Recommendation
Add a zero-output guard immediately after computing `agETHAmount` in `deposit(address token, ...)`:

```solidity
(uint256 agETHAmount, uint256 fee) = viewSwapAgETHAmountAndFee(amount, token);
if (agETHAmount == 0) revert InvalidAmount();
```

Alternatively, enforce a protocol-level minimum deposit per token that guarantees at least 1 wei of agETH output.

## Proof of Concept
```solidity
// Preconditions:
//   tokenToETHRate  = 0.9e18  (token oracle, e.g. stETH)
//   agETHToETHrate  = 1.1e18  (agETH oracle)
//   feeBps          = 0
//   amount          = 1 wei

// viewSwapAgETHAmountAndFee(1, token):
//   fee            = 0
//   amountAfterFee = 1
//   agETHAmount    = 1 * 0.9e18 / 1.1e18 = 0  ← integer truncation

// deposit(token, 1, ""):
//   safeTransferFrom(msg.sender, pool, 1)  ← token taken
//   agETH.mint(msg.sender, 0)              ← no-op, no revert
//   emit SwapOccurred(..., agETHAmount=0)

// For USDC (6 decimals, tokenToETHRate ≈ 3e14):
//   Any amount < 3667 wei (< 0.003667 USDC) triggers this silently.
```

A Foundry fuzz test asserting `agETHAmount > 0` for all `amount ∈ [1, agETHToETHrate / tokenToETHRate]` will find failing cases immediately.
