### Title
Hardcoded Gas Forwarding in ERC-20 Emulation Causes Permanent `ft_transfer` Failure for NEP-141 Contracts Requiring More Than 10 Tgas — (`runtime/near-wallet-contract/implementation/wallet-contract/src/eth_emulation.rs`)

---

### Summary

The Ethereum Wallet Contract's ERC-20 emulation layer forwards a hardcoded, fixed gas amount to NEP-141 `ft_transfer` cross-contract calls. Any NEP-141 token contract whose `ft_transfer` implementation requires more than 10 Tgas will permanently fail when called through the wallet contract, with no user-controlled override possible. This is the direct NEAR analog of Solidity's `transfer()` forwarding a fixed 2300 gas stipend.

---

### Finding Description

In `eth_emulation.rs`, the constant `FIVE_TERA_GAS` is defined as exactly 5 Tgas: [1](#0-0) 

The `try_emulation` function constructs `FunctionCall` actions with hardcoded gas values for all three ERC-20 selectors:

- `ft_balance_of` → `FIVE_TERA_GAS` (5 Tgas)
- `ft_total_supply` → `FIVE_TERA_GAS` (5 Tgas)
- `ft_transfer` → `2 * FIVE_TERA_GAS` (10 Tgas) [2](#0-1) 

These `gas` fields are embedded directly into the `Action::FunctionCall` struct and are later retrieved unchanged by `action.gas()`: [3](#0-2) 

When the ERC-20 transfer path is taken in `inner_rlp_execute`, the action's gas is passed directly to the NEP-141 contract promise with no dynamic adjustment: [4](#0-3) 

In the `nep_141_storage_balance_callback` path (receiver not yet registered), the `ft_transfer` call is chained after `storage_deposit` and still uses the same hardcoded 10 Tgas from the action: [5](#0-4) 

The user's Ethereum `gas_limit` field is only used to validate that the relayer attached sufficient total gas to the outer `rlp_execute` call — it is never forwarded to the inner NEP-141 cross-contract call: [6](#0-5) 

There is no mechanism for the user to specify a different gas amount for the `ft_transfer` call when using the ERC-20 emulation path. The only way to specify custom gas is to use the `FUNCTION_CALL_SELECTOR` native action path, which bypasses ERC-20 emulation entirely.

---

### Impact Explanation

Any NEP-141 token contract whose `ft_transfer` implementation consumes more than 10 Tgas will always fail when called through the wallet contract's ERC-20 emulation. This includes contracts with:

- Transfer hooks or callbacks to other contracts
- Complex storage management logic beyond the standard NEP-145 pattern
- Access control checks involving additional state reads
- Any future protocol parameter changes that increase base operation costs

When the `ft_transfer` promise fails, `rlp_execute_callback` receives `PromiseResult::Failed` and returns `success: false`: [7](#0-6) 

The concrete corrupted result is: the user's NEP-141 token balance is not updated (transfer does not occur), the user's NEAR gas is consumed, and the wallet nonce is permanently incremented — making the failed transaction unreplayable. The user has no recourse through the ERC-20 emulation interface.

---

### Likelihood Explanation

The trigger is reachable by any unprivileged Ethereum user holding an eth-implicit account on NEAR. The user signs a standard ERC-20 `transfer(address,uint256)` transaction and submits it through any relayer. No special privileges are required. The failure is deterministic and permanent for any NEP-141 contract that exceeds the 10 Tgas ceiling — the user cannot increase the forwarded gas by any means within the ERC-20 emulation interface.

---

### Recommendation

Replace the hardcoded `2 * FIVE_TERA_GAS` constant with a dynamic gas allocation using `promise_batch_action_function_call_weight` (NEP-264), forwarding a proportional share of the remaining unspent gas to the `ft_transfer` call. This mirrors the fix recommended in the original report: use the equivalent of `call()` (dynamic gas) rather than `transfer()` (fixed gas stipend). Alternatively, expose a user-configurable gas field in the ERC-20 emulation ABI so that users can specify the required gas for complex token contracts.

---

### Proof of Concept

1. Deploy a NEP-141 token contract whose `ft_transfer` method performs additional cross-contract calls or storage operations consuming more than 10 Tgas.
2. Mint tokens to an eth-implicit wallet account.
3. Sign an Ethereum transaction with selector `0xa9059cbb` (ERC-20 `transfer`) targeting the token contract.
4. Submit via `rlp_execute`. The wallet contract constructs a `ft_transfer` promise with exactly `2 * FIVE_TERA_GAS = 10 Tgas`.
5. The `ft_transfer` receipt aborts with `GasExceeded`. `rlp_execute_callback` returns `success: false`. The token balance is unchanged, the wallet nonce is incremented, and the user's NEAR gas is consumed with no recourse.
