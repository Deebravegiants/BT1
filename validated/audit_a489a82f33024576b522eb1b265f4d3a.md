### Title
Missing `runTest` for `validateTooManyCollateralInputs` in Alonzo UTxO Transition — (`File: eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs`)

---

### Summary

The `validateTooManyCollateralInputs` function is defined, exported, and enforced in the Babbage and Dijkstra era UTxO transition rules, but its return value (`Test`) is **never passed to `runTest`** in the Alonzo era's `utxoTransition`. A placeholder comment marks exactly where the call should appear, but the enforcement call is absent. This allows any transaction submitted during the Alonzo era to carry an unbounded number of collateral inputs, bypassing the `ppMaxCollateralInputsL` protocol parameter.

---

### Finding Description

`validateTooManyCollateralInputs` is defined at lines 470–481 of `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs`:

```haskell
validateTooManyCollateralInputs ::
  AlonzoEraTxBody era =>
  PParams era ->
  TxBody TopTx era ->
  Test (AlonzoUtxoPredFailure era)
validateTooManyCollateralInputs pp txBody =
  failureUnless (numColl <= maxColl) $
    TooManyCollateralInputs Mismatch {mismatchSupplied = numColl, mismatchExpected = maxColl}
  where
    maxColl, numColl :: Word16
    maxColl = pp ^. ppMaxCollateralInputsL
    numColl = fromIntegral . Set.size $ txBody ^. collateralInputsTxBodyL
``` [1](#0-0) 

In `utxoTransition` (the Alonzo era's UTXO transition rule), the comment for this check is present at line 570, but no `runTest` call follows it — the function proceeds directly to the `UTXOS` sub-transition:

```haskell
  {-   totExunits tx ≤ maxTxExUnits pp    -}
  runTest $ validateExUnitsTooBigUTxO pp tx

  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  -- ← NO runTest $ validateTooManyCollateralInputs pp txBody HERE

  updatedGovState <-
    trans @(EraRule "UTXOS" era) $
      TRC (UtxosEnv slot pp certState, utxosGovState utxos, stAnnTx)
``` [2](#0-1) 

By contrast, the Babbage era's `babbageUtxoValidation` correctly enforces the check:

```haskell
  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  runTest $ Alonzo.validateTooManyCollateralInputs pp txBody
``` [3](#0-2) 

And the Dijkstra era also enforces it: [4](#0-3) 

The function is also explicitly exported from the Alonzo module, confirming it is intended to be used: [5](#0-4)

### Impact Explanation

**Medium — Attacker-controlled transactions exceed intended validation limits.**

The `ppMaxCollateralInputsL` protocol parameter is the ledger's mechanism for bounding the number of collateral inputs per transaction. Without the enforcement call, any transaction submitted during the Alonzo era can carry an arbitrarily large collateral input set. This:

1. Violates the resource-limit invariant established by the protocol parameter.
2. Allows transactions to exceed the `maxCollInputs` bound that all other eras enforce, meaning the Alonzo era's ledger state can be reached via a path that later eras would reject — a divergence in what constitutes a valid ledger history.
3. During chain replay from genesis (which every full node must perform), Alonzo-era blocks containing over-collateralised transactions would be accepted, while a hypothetically corrected implementation would reject them — a deterministic disagreement in ledger rule evaluation across implementations or versions.

---

### Likelihood Explanation

The Alonzo era is no longer the active era on Cardano mainnet (the chain is currently in the Conway era). However:

- Every full node replays the entire chain from genesis, including all Alonzo-era blocks. A transaction with excess collateral inputs submitted during the Alonzo era would be accepted by all current nodes but rejected by a corrected implementation, creating a permanent divergence in chain-replay validation.
- Any testnet, private network, or staging environment still operating in the Alonzo era is directly exploitable by any transaction author (unprivileged sender).
- The attacker-controlled entry path requires only constructing a transaction with `collateralInputsTxBodyL` containing more entries than `ppMaxCollateralInputsL` — no special privilege is needed.

---

### Recommendation

Add the missing enforcement call in `utxoTransition` immediately after the existing comment, mirroring the Babbage and Dijkstra implementations:

```haskell
  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  runTest $ validateTooManyCollateralInputs pp txBody
```

This should be placed between `runTest $ validateExUnitsTooBigUTxO pp tx` (line 568) and the `updatedGovState <-` binding (line 572).

### Proof of Concept

1. Construct an Alonzo-era transaction with `collateralInputsTxBodyL` containing `N` inputs where `N > ppMaxCollateralInputsL`.
2. Submit it to a node running in the Alonzo era (or replay an Alonzo-era block containing such a transaction).
3. Observe that the transaction is accepted without a `TooManyCollateralInputs` predicate failure, because `validateTooManyCollateralInputs` is never invoked in `utxoTransition`.
4. Confirm that the same transaction submitted in the Babbage era would be rejected with `TooManyCollateralInputs` via `babbageUtxoValidation` at line 412 of `eras/babbage/impl/src/Cardano/Ledger/Babbage/Rules/Utxo.hs`.
