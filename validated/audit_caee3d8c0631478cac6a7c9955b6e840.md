Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks `initiateWithdrawal` Until Admin Intervenes — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool.initialize` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal` guards with `>=`, the check `0 >= 0` evaluates to `true` for every caller, causing an immediate `WithdrawalLimitReached` revert. Tokens staked via `stake` or `KernelMerkleDistributor.claimAndStake` before the admin calls `setMaxNumberOfWithdrawalsPerUser` are temporarily unwithdrawable.

## Finding Description
`initialize` (lines 259–271) sets `kernelToken`, `rewardsToken`, and access-control roles, but never assigns `maxNumberOfWithdrawalsPerUser`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... no assignment to maxNumberOfWithdrawalsPerUser
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
}
```

`initiateWithdrawal` (line 323) uses `>=`:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```

A fresh user has `userWithdrawalIds[msg.sender].length == 0` and `maxNumberOfWithdrawalsPerUser == 0`, so `0 >= 0` is always `true` and the function always reverts.

The only setter, `setMaxNumberOfWithdrawalsPerUser` (lines 610–620), explicitly rejects `0`:

```solidity
if (_maxNumberOfWithdrawalsPerUser == 0 || _maxNumberOfWithdrawalsPerUser > MAX_WITHDRAWALS_PER_USER) {
    revert InvalidMaxNumberOfWithdrawalsPerUser();
}
```

This means `0` is exclusively an uninitialized sentinel — it can never be a valid configured value, and there is no in-contract enforcement that it must be set before staking is enabled.

`stakeFor` (lines 296–314) and `stake` (lines 281–289) have no guard on `maxNumberOfWithdrawalsPerUser`, so tokens can be deposited freely while withdrawals are fully blocked. `KernelMerkleDistributor.claimAndStake` (lines 280–284) calls `stakeFor` directly, creating a path where users claim and stake tokens in one transaction, only to find their funds locked when they attempt to withdraw.

## Impact Explanation
Any user who stakes tokens (directly or via `claimAndStake`) before the admin calls `setMaxNumberOfWithdrawalsPerUser` has `balanceOf[user] > 0` but cannot call `initiateWithdrawal` — every attempt reverts with `WithdrawalLimitReached`. This is a **temporary freezing of funds** (Medium), matching the allowed impact scope. The freeze lifts once the admin configures the limit, so it is not permanent.

## Likelihood Explanation
The deployment flow has no enforced ordering: `KernelMerkleDistributor` can be live and accepting `claimAndStake` calls the moment both contracts are deployed and a merkle root is set, with no dependency on the withdrawal limit being configured. An admin who follows only the `initialize` call and then enables the distributor will inadvertently freeze all staked funds. This is a realistic deployment-ordering gap, not a contrived scenario.

## Recommendation
Set a safe default for `maxNumberOfWithdrawalsPerUser` inside `initialize`:

```solidity
// In initialize():
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER;
```

Alternatively, block staking entirely until the withdrawal limit is configured:

```solidity
// In stake() and stakeFor():
if (maxNumberOfWithdrawalsPerUser == 0) revert WithdrawalLimitNotConfigured();
```
