Audit Report

## Title
Price Protection Guards Silently Disabled When `pricePercentageLimit` Is Zero — (`contracts/LRTOracle.sol`)

## Summary
`LRTOracle._updateRsETHPrice()` gates both the upside price-increase guard and the downside auto-pause trigger behind the compound condition `pricePercentageLimit > 0 && …`. Because `pricePercentageLimit` is a plain storage variable that defaults to `0` and can be freely reset to `0` by an admin with no validation, both guards are permanently `false` in that state. The downside guard failure means a significant price drop — caused by EigenLayer slashing, a NodeDelegator accounting error, or a temporary oracle deviation — never triggers the protocol pause, allowing an attacker to deposit at the depressed `rsETHPrice` and extract yield from existing holders when the price recovers.

## Finding Description
`pricePercentageLimit` is declared as a plain `uint256` storage variable with no initializer:

```solidity
// contracts/LRTOracle.sol L29
uint256 public pricePercentageLimit;
```

`initialize()` sets only `lrtConfig` and never assigns `pricePercentageLimit`, so it remains `0` from deployment until an admin explicitly calls `setPricePercentageLimit`. That setter accepts any value including `0` with no validation:

```solidity
// contracts/LRTOracle.sol L125-127
function setPricePercentageLimit(uint256 _pricePercentageLimit) external onlyLRTAdmin {
    pricePercentageLimit = _pricePercentageLimit;
    emit PricePercentageLimitUpdate(_pricePercentageLimit);
}
```

Inside `_updateRsETHPrice()`, both safety checks share the same flawed short-circuit:

**Upside guard (L256-257):**
```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```

**Downside guard (L273-274):**
```solidity
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
```

When `pricePercentageLimit == 0`, both evaluate to `false` regardless of the actual price movement. The auto-pause block (L277-282) is therefore never reached:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

`updateRSETHPrice()` is a public, permissionless function (only `whenNotPaused`), so any address can call it and commit the depressed price to storage. Deposits then use the stale low price:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

A lower `rsETHPrice` denominator mints proportionally more rsETH per deposited asset. When the price recovers, the over-minted shares redeem for more underlying than was deposited, extracting yield from existing holders.

## Impact Explanation
**High — Theft of unclaimed yield.**

The auto-pause mechanism exists specifically to halt deposits when `rsETHPrice` drops significantly, preventing the over-minting of rsETH at a depressed price. When `pricePercentageLimit == 0` silently disables this mechanism, an attacker can deposit during a price-depressed window and redeem after recovery for a profit funded by diluting existing rsETH holders. This matches the allowed impact "Theft of unclaimed yield."

The upside guard bypass (non-managers committing large price increases) is a secondary Medium-class auth bypass, but the price is computed from actual on-chain TVL via `_getTotalEthInProtocol()`, so the caller cannot manufacture an artificial increase; this impact is less concrete.

## Likelihood Explanation
`pricePercentageLimit` is `0` on every fresh deployment and after any admin call to `setPricePercentageLimit(0)`. No constructor or initializer sets a non-zero value. The vulnerable window spans the entire period from deployment until the admin explicitly configures the limit, and any subsequent period after it is reset. A price drop trigger (EigenLayer slashing, NodeDelegator accounting error, temporary oracle deviation) is a realistic, non-adversarial event. The attacker action — calling `updateRSETHPrice()` followed by `depositAsset()` or `depositETH()` — requires no special privileges and is a normal public interaction.

## Recommendation
Treat `pricePercentageLimit == 0` as "limit not configured" and revert early in `_updateRsETHPrice()`:

```solidity
if (pricePercentageLimit == 0) revert PricePercentageLimitNotSet();
```

Remove the `pricePercentageLimit > 0 &&` short-circuit from both guards so they evaluate unconditionally. Additionally, add a non-zero validation to `setPricePercentageLimit` and set a safe default in `initialize()` or `reinitialize()`.

## Proof of Concept
1. Deploy `LRTOracle` without calling `setPricePercentageLimit` (or call `setPricePercentageLimit(0)`). `pricePercentageLimit` is `0`.
2. A slashing event reduces the ETH value held by a `NodeDelegator`, causing `_getTotalEthInProtocol()` to return a value significantly lower than `rsethSupply.mulWad(rsETHPrice)`.
3. Any unprivileged address calls `updateRSETHPrice()`.
4. Inside `_updateRsETHPrice()`, `newRsETHPrice < highestRsethPrice` is true and `diff` is large, but `isPriceDecreaseOffLimit = (0 > 0) && … = false`. The auto-pause block at L277-282 is skipped entirely.
5. `rsETHPrice` is updated to the depressed value and the function returns normally.
6. An attacker immediately calls `LRTDepositPool.depositAsset()` or `depositETH()`. Because `rsETHPrice` is now lower, `getRsETHAmountToMint` at L520 mints more rsETH per deposited asset than the attacker's fair share.
7. When the price recovers (slashing resolved, oracle corrected), the attacker's over-minted rsETH redeems for more underlying than was deposited, extracting yield from existing holders.
