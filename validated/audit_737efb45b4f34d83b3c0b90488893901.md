### Title
Stale `parameters_votes` Persists After Resharing Cancellation, Reducing Effective Governance Threshold — (File: `crates/contract/src/state/resharing.rs`)

---

### Summary

When resharing is cancelled via `vote_cancel_resharing`, the returned `previous_running_state` retains accumulated `parameters_votes` from re-proposal attempts made during the resharing phase. These stale votes carry forward into the new running state, reducing the number of fresh votes required to trigger a subsequent resharing — breaking the governance invariant that all proposed participants must independently vote.

---

### Finding Description

**Root cause — incomplete state cleanup on resharing cancellation.**

When the contract transitions from Running → Resharing, `previous_running_state` is constructed via `RunningContractState::new()`, which initialises `parameters_votes` to `ThresholdParametersVotes::default()`: [1](#0-0) 

During resharing, participants may call `vote_new_parameters` (re-proposal). This delegates directly into `self.previous_running_state.process_new_parameters_proposal(proposal)`, which mutates `previous_running_state.parameters_votes`: [2](#0-1) 

If the re-proposal does not reach consensus (fewer than `new_num_participants` votes), the partial votes remain in `previous_running_state.parameters_votes`. When resharing is subsequently cancelled, `vote_cancel_resharing` returns the running state as a plain clone — **without clearing `parameters_votes`**: [3](#0-2) 

The only field that is updated is `previously_cancelled_resharing_epoch_id`: [4](#0-3) 

**Why the stale votes remain active in the new running state.**

After cancellation, `prospective_epoch_id()` in the returned running state evaluates to `resharing_epoch + 1`: [5](#0-4) 

The stale votes in `parameters_votes` were cast for proposals whose `prospective_epoch_id` was also `resharing_epoch + 1` (the re-proposal epoch checked in `ResharingContractState::vote_new_parameters`): [6](#0-5) 

Because `parameters_votes` stores votes keyed only by proposal hash (not by epoch), and because the epoch check in `RunningContractState::vote_new_parameters` validates only the caller-supplied `prospective_epoch_id` parameter — not the stored votes — the stale votes are silently counted when any participant submits the same proposal in the new running state: [7](#0-6) 

The consensus check in `process_new_parameters_proposal` is: [8](#0-7) 

If `k` participants voted during resharing re-proposal, only `new_num_participants − k` fresh votes are needed after cancellation to satisfy `new_num_participants == n_votes`.

---

### Impact Explanation

This breaks the governance invariant that **all** proposed participants must independently cast a fresh vote to trigger resharing. A participant who voted during the resharing re-proposal phase and then voted to cancel resharing can, in the returned running state, trigger a new resharing with a single additional vote — even though the governance threshold nominally requires unanimous consent from all proposed participants. This is participant-state manipulation that corrupts the contract's accounting of consent, matching the Medium allowed impact: *"participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The scenario is reachable without any privileged access beyond being a participant:

1. Resharing is triggered (all current participants vote for proposal P1).
2. During resharing, participants A, B, C vote for re-proposal P2 (k = 3 out of 4 required votes).
3. Resharing is cancelled via `vote_cancel_resharing` (threshold votes from the previous running state).
4. In the returned running state, participant D votes for P2 with `prospective_epoch_id = resharing_epoch + 1`.
5. Stale votes (3) + fresh vote (1) = 4 = `new_num_participants` → resharing is triggered immediately with only **one** fresh vote in the running state.

No collusion above the signing threshold is required; the attacker only needs to have legitimately participated in the re-proposal during resharing.

---

### Recommendation

Clear `parameters_votes` before returning the running state from `vote_cancel_resharing`:

```rust
// crates/contract/src/state/resharing.rs — vote_cancel_resharing
let running_state = if threshold_cancellation_votes_reached {
    let mut previous_running_state = self.previous_running_state.clone();
    let prospective_epoch_id = self.prospective_epoch_id();
    previous_running_state.previously_cancelled_resharing_epoch_id =
        Some(prospective_epoch_id);
+   previous_running_state.parameters_votes = ThresholdParametersVotes::default();
    Some(previous_running_state)
} else {
    None
};
```

This mirrors the pattern already used in `RunningContractState::new()`, which always initialises `parameters_votes` to `default()`, and in `ResharingContractState::vote_new_parameters`, which constructs a fresh `RunningContractState::new(...)` (and thus a fresh `parameters_votes`) when a re-proposal succeeds. [9](#0-8)

### Proof of Concept

```
State:  Running  (participants: A, B, C, D; threshold = 4; epoch = E0)
Step 1: All 4 vote for proposal P1 → Resharing (epoch E1, prospective = E1+1)
Step 2: A, B, C call vote_new_parameters(prospective=E1+1, P2)
        → previous_running_state.parameters_votes now has 3 votes for P2
        → consensus not reached (need 4), resharing continues
Step 3: ≥threshold old participants call vote_cancel_resharing
        → returned running state = previous_running_state.clone()
        → parameters_votes still has 3 votes for P2
        → previously_cancelled_resharing_epoch_id = E1
        → prospective_epoch_id() = E1+1
Step 4: D calls vote_new_parameters(prospective=E1+1, P2) in running state
        → epoch check passes (E1+1 == E1+1)
        → parameters_votes.vote(P2, D) returns 4 (3 stale + 1 fresh)
        → 4 == new_num_participants → resharing triggered
        → Only 1 fresh vote was cast in running state, not 4
```
