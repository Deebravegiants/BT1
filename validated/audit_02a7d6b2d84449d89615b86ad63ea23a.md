### Title
`log_metadata` Missing Pause Guard Allows Vault Creation and Native Token Registration During Full Pause — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

### Summary
The `log_metadata` instruction has no pause check. Every other user-facing instruction (`init_transfer`, `init_transfer_sol`, `finalize_transfer`, `finalize_transfer_sol`) gates execution behind a `config.paused` bitmask check. `log_metadata` does not. An unprivileged caller can invoke it while `config.paused == ALL_PAUSED`, causing a vault token account PDA to be created (`init_if_needed`) for an arbitrary SPL mint and a `LogMetadata` Wormhole message to be emitted to NEAR, registering that mint as a native bridge asset.

### Finding Description

**Pause bitmask constants** — `ALL_PAUSED = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED = 0b00000011`: [1](#0-0) 

**Every other user instruction is gated**, e.g.: [2](#0-1) 

**`log_metadata` has no such gate**: [3](#0-2) 

**The `LogMetadata` accounts struct unconditionally creates the vault PDA** via `init_if_needed`: [4](#0-3) 

**The only mint constraint** is that the bridge authority is *not* the mint authority — i.e., any independently-issued SPL token qualifies: [5](#0-4) 

**`process()` then posts a Wormhole message** carrying the token's mint address, name, symbol, and decimals to NEAR: [6](#0-5) 

### Impact Explanation
When the bridge is paused (typically in response to a security incident), the intent is to halt all bridge state changes. `log_metadata` bypasses this entirely: it (a) allocates a new on-chain vault PDA for an arbitrary mint and (b) emits a signed Wormhole VAA that NEAR will process as a legitimate native-token registration. If NEAR accepts the first registration of a given Solana mint as authoritative, an attacker can pre-register a token with manipulated or misleading metadata (name/symbol/decimals) before the legitimate operator can act, poisoning the token identity mapping on NEAR. This falls under **unauthorized creation of a native bridge asset** and **asset-identity divergence that breaks backing guarantees**.

### Likelihood Explanation
The call requires no privilege — only a funded payer account and a reference to any SPL mint where the bridge is not the mint authority. The pause window is precisely the period of highest risk (active incident response), making this a realistic attack vector.

### Recommendation
Add the same pause guard used by all other user instructions at the top of the `log_metadata` handler in `lib.rs`:

```rust
pub fn log_metadata(ctx: Context<LogMetadata>) -> Result<()> {
    require!(
        ctx.accounts.common.config.paused & ALL_PAUSED == 0,
        error::ErrorCode::Paused
    );
    msg!("Logging metadata");
    ctx.accounts.process()?;
    Ok(())
}
```

Alternatively, define a dedicated `LOG_METADATA_PAUSED` bit so operators can pause metadata registration independently of transfers.

### Proof of Concept
1. `pausable_admin` calls `pause()` → `config.paused = ALL_PAUSED = 3`.
2. Attacker constructs a `LogMetadata` context with any SPL mint `M` where `M.mint_authority ≠ bridge_authority` and the vault PDA for `M` does not yet exist.
3. Attacker calls `log_metadata` — no pause check fires.
4. Anchor's `init_if_needed` allocates the vault PDA for `M`.
5. `process()` serializes `LogMetadataPayload { token: M, name, symbol, decimals }` and calls `post_message` via Wormhole CPI.
6. NEAR receives and processes the VAA, registering `M` as a native Solana bridge asset with attacker-chosen metadata.
7. The vault PDA now exists on-chain; the token is registered on NEAR — both irreversible without admin intervention — while the bridge was supposed to be fully paused.
