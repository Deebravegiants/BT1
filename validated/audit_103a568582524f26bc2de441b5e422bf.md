### Title
Forged Foreign-Chain Verification via Cross-Request Response Injection by a Single Byzantine Participant — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is valid over `response.payload_hash` using the root public key, but **never verifies that `response.payload_hash` was derived from the stored request's `ForeignChainRpcRequest`**. A single malicious attested participant (below the signing threshold) can take a legitimately-produced threshold signature for pending request R1 (tx\_id=X) and submit it as the response to a different pending request R2 (tx\_id=Y). The contract accepts it, resolves R2 with the forged response, and the user's contract receives a cryptographically-valid attestation for the wrong foreign-chain transaction.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 691–753), the contract performs the following steps:

1. Asserts the caller is a single attested participant.
2. Extracts `payload_hash` directly from the caller-supplied `response`:
   ```rust
   let payload_hash: [u8; 32] = response.payload_hash.0;
   ```
3. Verifies the signature over that hash against the root public key:
   ```rust
   near_mpc_signature_verifier::verify_ecdsa_signature(
       signature_response,
       &payload_hash,
       &secp_pk,
   )
   .is_ok()
   ```
4. Resolves the pending yield keyed on the caller-supplied `request`:
   ```rust
   pending_requests::resolve_yields_for(
       &mut self.pending_verify_foreign_tx_requests,
       &request,
       serde_json::to_vec(&response).unwrap(),
   )
   ``` [1](#0-0) 

The `payload_hash` is designed to commit to the foreign-chain transaction data: per the protocol documentation, `msg_hash = SHA-256(borsh(ForeignTxSignPayloadV1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }))`. The `ForeignChainRpcRequest` (including `tx_id`) is embedded in the hash. [2](#0-1) 

However, the contract **never checks** that `response.payload_hash` was computed from the `request` parameter's `ForeignChainRpcRequest`. The `request` is used only as a map key to look up and drain pending yields. The response data (including `payload_hash`) is passed through verbatim to the user's callback. [3](#0-2) 

Contrast this with the regular `respond` function, where the payload is taken from the **stored** request, not from the caller-supplied response:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
``` [4](#0-3) 

This asymmetry is the root cause.

---

### Impact Explanation

**Attack scenario:**

1. User A submits `verify_foreign_transaction` for Bitcoin `tx_id=X` → pending request R1.
2. User B submits `verify_foreign_transaction` for Bitcoin `tx_id=Y` → pending request R2.
3. The MPC network runs the threshold signing protocol for R1. The malicious participant (one of the MPC nodes) observes the final threshold signature `sig_X` over `payload_hash_X = SHA-256(borsh({tx_id=X, extracted_values_X}))`.
4. The malicious participant calls `respond_verify_foreign_tx(R2, {payload_hash: payload_hash_X, signature: sig_X})` — using R2 as the request key but the response for R1.
5. The contract verifies `sig_X` over `payload_hash_X` → valid. It resolves R2's pending yield with the forged response.
6. User B's contract receives `VerifyForeignTransactionResponse { payload_hash: payload_hash_X, signature: sig_X }` — a cryptographically valid MPC attestation, but for `tx_id=X`, not `tx_id=Y`.
7. Honest nodes later call `respond_verify_foreign_tx(R1, {payload_hash_X, sig_X})` → R1 is also resolved correctly.

If User B's bridge contract does not independently reconstruct and compare the expected `payload_hash` for `tx_id=Y`, it will treat the forged attestation as proof that `tx_id=Y` was verified and release funds accordingly. This enables **invalid bridge execution** — funds released on NEAR for a foreign-chain deposit that was never verified. [5](#0-4) 

---

### Likelihood Explanation

- Requires **one** malicious attested participant — strictly below the signing threshold.
- The malicious participant participates in the honest MPC signing protocol for R1 and observes the resulting threshold signature before it is submitted on-chain (standard in MPC node operation).
- The malicious participant races to submit the forged response for R2 before honest nodes submit the correct response for R2. Once R2 is resolved, honest nodes' correct response for R2 fails with "request not found."
- No key forgery, no threshold collusion, no privileged access beyond being an attested participant.
- The pending request map is public on-chain, so the attacker can trivially identify suitable R2 targets.

---

### Recommendation

Bind the response to the stored request by including the `ForeignChainRpcRequest` (or a hash of the stored `VerifyForeignTransactionRequest`) in the signed payload **and** verifying this binding on-chain in `respond_verify_foreign_tx`. Concretely:

1. **Protocol level**: MPC nodes should sign `SHA-256(borsh(request_key || extracted_values))` where `request_key` is a canonical encoding of the full `VerifyForeignTransactionRequest` (including `domain_id` and `payload_version`), not just the `ForeignChainRpcRequest`. This ensures the signature is bound to the specific NEAR request.

2. **Contract level**: Store a hash of the `VerifyForeignTransactionRequest` alongside the pending yield. In `respond_verify_foreign_tx`, recompute the expected request commitment from the caller-supplied `request` and verify that `response.payload_hash` encodes it. This prevents a response for R1 from being accepted as a response for R2.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
