### Title
Missing State Diff Commitment Verification in P2P Sync Allows Unauthenticated Peer to Corrupt Node State — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary
`StateDiffStreamBuilder::parse_data_for_block` assembles a `ThinStateDiff` from peer-supplied `StateDiffChunk` messages and validates only that the assembled length equals `header.state_diff_length`. It never computes the Poseidon hash of the assembled diff and never compares it against `header.state_diff_commitment`. Any unauthenticated p2p peer can therefore supply state diff chunks whose total length is correct but whose storage values, nonces, deployed contracts, or class hashes are arbitrary. The corrupted `ThinStateDiff` is written unconditionally to the node's LMDB storage, poisoning every subsequent `starknet_getStorageAt` and `starknet_getNonce` RPC response served from that storage.

### Finding Description
In `crates/apollo_p2p_sync/src/client/state_diff.rs`, `parse_data_for_block` (lines 51–110):

- Reads `target_state_diff_len` from `header.state_diff_length` (line 66).
- Loops accumulating `StateDiffChunk` messages until `current_state_diff_len == target_state_diff_len` (lines 72–97).
- After the loop, checks only `current_state_diff_len != target_state_diff_len` (line 99) and calls `validate_deprecated_declared_classes_non_conflicting` (line 106).
- Returns `Ok(Some((result, block_number)))` — the assembled `ThinStateDiff` — with no hash computation and no comparison against `header.state_diff_commitment`.

`write_to_storage` (lines 28–39) then calls `append_state_diff` directly, which writes all storage diffs, nonces, deployed contracts, and class hashes into the node's persistent LMDB tables with no further integrity check.

The header itself is also unprotected: `HeaderStreamBuilder::parse_data_for_block` (lines 82–123) stores any peer-supplied `SignedBlockHeader` after checking only block number ordering and signature vector length — it never calls `verify_block_signature`. So the attacker controls both the `state_diff_commitment` value in the stored header and the state diff content, making the missing commitment check the sole gate between the network and persistent state.

### Impact Explanation
A malicious p2p peer that the syncing node connects to can:
1. Send a `SignedBlockHeader` with an arbitrary `state_diff_commitment` and a chosen `state_diff_length` (e.g., 3).
2. Send exactly 3 length-units of `StateDiffChunk` messages carrying wrong storage values or nonces.
3. The node writes the corrupted `ThinStateDiff` to storage.
4. All subsequent `starknet_getStorageAt` and `starknet_getNonce` RPC calls for affected contracts return attacker-chosen values.
5. Fee estimation, simulation, and pending-state views derived from the same storage are equally corrupted.

The global state root stored in the header is inconsistent with the actual persisted state, so the node presents an internally incoherent view of the chain to all RPC consumers.

### Likelihood Explanation
The p2p sync client connects to peers discovered through the network layer with no authentication requirement. Any node that can establish a p2p connection and respond to `StateDiffQuery` messages can execute this attack. No validator, operator, or privileged key is required.

### Recommendation
After the assembly loop in `parse_data_for_block`, compute `calculate_state_diff_hash(&result)` and compare it against `header.state_diff_commitment`. If the values differ, return `ParseDataError::BadPeer(...)` and report the peer. This mirrors the guard already present in `apollo_committer::Committer::commit_or_load` (lines 265–280). [4](#0-3) 

### Proof of Concept
1. Stand up a target node configured to p2p-sync state diffs.
2. Run a malicious peer that responds to `StateDiffQuery` for block 0 with:
   - A `SignedBlockHeader` where `state_diff_length = 1` and `state_diff_commitment` is any value.
   - One `StateDiffChunk::ContractDiff` carrying a storage key `0x1` → attacker-chosen value `0xdeadbeef` (length = 1, satisfying the length check).
   - A `DataOrFin(None)` fin message.
3. After sync completes, call `starknet_getStorageAt` for the affected contract and key.
4. Observe the RPC returns `0xdeadbeef` instead of the correct value, confirming the corrupted diff was persisted without commitment verification.
