### Title
Unvalidated Off-Chain CKD Output for `AppPublicKey` Requests Allows Single Byzantine Participant to Forge Confidential Key Derivation Response — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs **no on-chain cryptographic validation** of the `CKDResponse` when the request uses the `AppPublicKey` (legacy, privately-verifiable) variant. Any single attested participant can submit an entirely fabricated `(big_y, big_c)` pair and the contract will accept it, delivering a forged confidential key to the user. This is the direct analog of the reported pool-rebalancing bug: the correct output is computed off-chain by the threshold protocol, but the contract never verifies that the submitted value actually matches that computation, allowing a Byzantine actor to substitute a stale or fabricated value before the legitimate leader can respond.

---

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no validation at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract runs `ckd_output_check`, which verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` — a cryptographic proof that the response was produced using the network's master secret key. [2](#0-1) 

For `AppPublicKey`, the arm is an empty block. The contract fetches the current BLS public key (line 668–673) but never uses it to validate the response. [3](#0-2) 

The only gate is `assert_caller_is_attested_participant_and_protocol_active`, which confirms the caller is an attested participant — not that the response is the genuine output of the threshold protocol. [4](#0-3) 

The unit test at line 3403 confirms the contract accepts completely arbitrary byte arrays (`[1u8; 48]`, `[2u8; 48]`) as a valid `AppPublicKey` CKD response: [5](#0-4) 

The `assert_caller_is_signer` check only enforces that `signer_account_id == predecessor_account_id`; it does not restrict which participant may call `respond_ckd` or enforce leader-only submission. [6](#0-5) 

---

### Impact Explanation

A single Byzantine attested participant can call `respond_ckd` with an arbitrary `CKDResponse` for any pending `AppPublicKey` CKD request. The contract resolves the yield and delivers the forged key to the user. The user receives a `(big_y, big_c)` pair that was not produced by the threshold protocol and does not encode a key derived from the network's master secret. Because `AppPublicKey` is "privately verifiable," the user cannot detect the forgery on-chain; they can only discover it off-chain after attempting to use the derived key. This constitutes **unauthorized confidential key derivation output without the required participant authorization**, matching the Critical impact tier.

---

### Likelihood Explanation

The attacker must be an attested participant (requires a valid TEE attestation). However, once attested, any participant can call `respond_ckd` for any pending `AppPublicKey` request at any time — there is no leader-only enforcement at the contract level for this method. The attacker simply monitors the NEAR mempool for `request_app_private_key` transactions using the `AppPublicKey` variant and races to submit a forged `respond_ckd` before the legitimate leader. Because NEAR transaction ordering within a block is not guaranteed, a Byzantine participant with a direct RPC connection can reliably win this race. The `AppPublicKey` variant is described as "legacy" but remains fully supported and is used in production e2e tests. [7](#0-6) 

---

### Recommendation

Apply the same `ckd_output_check` pairing verification to `AppPublicKey` requests that is already applied to `AppPublicKeyPV` requests. For `AppPublicKey`, the check degenerates to verifying `e(big_c, g2) = e(big_y, app_pk1) · e(hash_point, public_key)` using only the G1 app public key. This moves the correctness check on-chain, mirroring the fix described in the referenced report ("computation for the rebalancing amount should be on-chain"). Alternatively, deprecate and remove the `AppPublicKey` variant entirely, requiring all callers to use `AppPublicKeyPV`.

---

### Proof of Concept

1. Attacker is an attested participant in the MPC network.
2. User submits `request_app_private_key` with `AppPublicKey: "bls12381g1:<some_point>"`.
3. Before the legitimate leader calls `respond_ckd`, the attacker calls:
   ```
   respond_ckd(
     request = <the pending CKDRequest>,
     response = CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }
   )
   ```
4. The contract executes lines 675–676 (`AppPublicKey` arm → empty), skips all validation, and calls `pending_requests::resolve_yields_for`, resolving the yield with the forged response. [8](#0-7) 

5. The user's `request_app_private_key` promise resolves with the attacker-chosen `(big_y, big_c)` — a key that was never derived from the network's master secret. The legitimate leader's subsequent `respond_ckd` call finds no pending request and silently fails.

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Verify that a CKD response (AppPublicKey variant) is mathematically correct.
#[tokio::test]
#[expect(non_snake_case)]
async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),
        "CKD response failed cryptographic verification"
    );
}
