### Title
Multiple Sub-Transactions Can Claim the Same Stake Deposit Refund in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubDeleg.hs`)

---

### Summary

The Dijkstra era's sub-transaction processing lacks a cross-sub-transaction uniqueness check for `UnRegDepositTxCert` certificates. Multiple sub-transactions within a single batch transaction can each include an `UnRegDepositTxCert` for the same stake credential, potentially allowing an attacker to claim the same deposit refund multiple times, creating ADA from nothing.

---

### Finding Description

The external report describes a missing identity check on a receipt: `repay_flash_loan` accepts a `FlashLoanReceipt` without verifying that its `order_id` matches the specific order that issued it, allowing the receipt to be "spent" against a different order. The analog in Cardano Ledger is that a deposit-refund claim (`UnRegDepositTxCert`) can be included in multiple sub-transactions within a single Dijkstra-era batch transaction, without any cross-sub-transaction uniqueness enforcement.

The Dijkstra era introduces nested (sub-)transactions. Each sub-transaction is processed through its own `SUBDELEG` rule, which simply delegates to `Conway.conwayDelegTransition`: [1](#0-0) 

`Conway.conwayDelegTransition` checks whether the stake credential is currently registered and, if so, unregisters it and issues the refund. However, there is no check that prevents two sub-transactions in the same batch from each presenting an `UnRegDepositTxCert` for the same credential.

The developers have explicitly acknowledged this gap in a disabled test: [2](#0-1) 

The test is marked `xit` (skipped/pending) and its expected failure is `error "TODO: predicate failure not yet implemented"` — confirming that the ledger rule to reject such a transaction does not yet exist. The transaction currently succeeds.

The `SUBUTXO` rule processes each sub-transaction's UTxO accounting independently: [3](#0-2) 

The `certState` passed into each sub-transaction's environment is the state as updated by prior sub-transactions. However, the `updateUTxOStateNoFees` call computes `totalRefunds` from the sub-transaction's own certificate list without checking whether the same credential's refund was already claimed by a sibling sub-transaction: [4](#0-3) 

If the first sub-transaction successfully unregisters the credential (removing it from `certState`), the second sub-transaction's `UnRegDepositTxCert` for the same credential would fail with `StakeKeyNotRegisteredDELEG` at the CERT level — but the `utxosDeposited` accounting in `updateUTxOStateNoFees` uses `certsTotalRefundsTxBody`, which computes refunds from the certificate list alone, not from the live `certState`. This creates a window where the deposit accounting diverges from the credential registration state, depending on the order of rule application.

The test comment confirms the developers believe the transaction currently succeeds when it should not.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker who registers a stake credential (paying deposit `D`) can construct a Dijkstra-era batch transaction containing two sub-transactions, each with `UnRegDepositTxCert` for the same credential. If both sub-transactions' refunds are credited, the attacker receives `2D` while having paid only `D`, creating `D` ADA from nothing. This violates the preservation-of-value invariant and constitutes a direct, attacker-controlled creation of ADA.

---

### Likelihood Explanation

The Dijkstra era is currently in active development and not yet deployed on mainnet. However, the vulnerability is reachable by any unprivileged transaction sender who can construct a valid Dijkstra-era batch transaction. No privileged access, governance majority, or key compromise is required. The attack requires only:
1. Registering a stake credential (paying the deposit).
2. Constructing a batch transaction with two sub-transactions each containing `UnRegDepositTxCert` for that credential.

The developers have already identified the scenario and noted the missing predicate failure, making the root cause confirmed rather than speculative.

---

### Recommendation

Implement a cross-sub-transaction uniqueness check for `UnRegDepositTxCert` (and analogously `UnRegDRepTxCert`) certificates within a batch transaction. Before processing sub-transactions, collect the set of all credentials being deregistered across all sub-transactions and reject the batch if any credential appears more than once. This check should be added to the `SUBLEDGERS` or `SUBLEDGER` rule, analogous to the fix described in the external report:

```
assert!(order_id == id(limit_order), EMismatchedOrder);
```

The Cardano Ledger equivalent would be:

```haskell
-- In SubLedgers transition, before processing sub-transactions:
let allUnregCreds = foldMap (collectUnregCreds . (^. bodyTxL . certsTxBodyL)) subTxs
failUnless (Set.size allUnregCreds == length (concatMap unregCerts subTxs))
  DuplicateUnregCertInSubTransactions
```

### Proof of Concept

1. Attacker registers stake credential `C` paying deposit `D = ppKeyDeposit`.
2. Attacker constructs a Dijkstra-era top-level transaction with:
   - `subTx1`: inputs = `{utxo1}`, certs = `[UnRegDepositTxCert C D]`
   - `subTx2`: inputs = `{utxo2}`, certs = `[UnRegDepositTxCert C D]`
3. Attacker submits the batch transaction.
4. Because no predicate failure prevents it (as confirmed by the `xit` test at line 53 of `CertSpec.hs`), both sub-transactions are accepted.
5. The deposit pot is reduced by `2D` while only `D` was originally deposited, and the attacker's reward account receives `2D`. [2](#0-1) [5](#0-4) [4](#0-3)
