Audit Report

## Title
Stale Cross-Chain Rate Enables agETH Over-Issuance via Block Stuffing — (`contracts/agETH/AGETHPoolV3.sol`)

## Summary

`AGETHPoolV3.deposit()` mints agETH using a rate fetched from `agETHOracle` (an `AGETHRateReceiver` / `CrossChainRateReceiver`) with no staleness guard. Because `MultiChainRateProvider.updateRate()` is permissionless, an attacker can use block stuffing to suppress rate updates, holding the receiver's stored rate artificially below the live rate. Deposits made against the stale-low rate receive more agETH than the deposited ETH backs at the live rate, degrading the pool's ETH-per-agETH backing ratio.

## Finding Description

**Rate delivery chain:**

`AGETHMultiChainRateProvider.updateRate()` is `external payable` with no access control — any caller can trigger it, and any caller can also prevent it by stuffing blocks. [1](#0-0) 

`CrossChainRateReceiver` stores `rate` and `lastUpdated` on each LayerZero delivery, but `getRate()` returns `rate` unconditionally — `lastUpdated` is never checked by any consumer. [2](#0-1) 

`AGETHPoolV3.getRate()` blindly forwards whatever `agETHOracle.getRate()` returns. [3](#0-2) 

`viewSwapAgETHAmountAndFee()` uses that rate directly to compute the mint amount with no freshness check. [4](#0-3) 

`deposit()` calls `viewSwapAgETHAmountAndFee()` and mints the result unconditionally. [5](#0-4) 

**Exploit flow:**

1. agETH is yield-bearing; its ETH rate drifts monotonically upward (e.g., `1.00e18` → `1.05e18`).
2. Attacker stuffs blocks on the source or destination chain to prevent `updateRate()` from landing, keeping `AGETHRateReceiver.rate` at stale value `S < L` (live rate).
3. Attacker calls `AGETHPoolV3.deposit{value: V}()`.
4. Minted: `agETHAmount = (V - fee) * 1e18 / S` — inflated vs. correct `(V - fee) * 1e18 / L`.
5. Over-issuance: `Δ = (V - fee) * 1e18 * (L - S) / (S * L)` — attacker holds agETH backed by less ETH than deposited at live rate.

**Why existing checks fail:** `CrossChainRateReceiver` does record `lastUpdated` on delivery, but neither `getRate()` nor any caller in `AGETHPoolV3` ever reads or enforces a maximum age on that timestamp. [6](#0-5) 

## Impact Explanation

The pool mints agETH in excess of the ETH deposited at the live rate. The ETH-per-agETH backing ratio decreases for all outstanding agETH holders. No single-event principal loss to existing holders, but repeated exploitation continuously degrades the backing ratio. This matches **Low. Block stuffing** and **Low. Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation

`updateRate()` is permissionless, so block stuffing is the only required mechanism — no key compromise or admin collusion needed. On L2 sequencer environments (Arbitrum, Optimism), suppressing a specific transaction for several blocks is significantly cheaper than on Ethereum mainnet. The agETH rate drifts upward continuously with staking yield, so any staleness window creates an exploitable gap. The attacker only needs ETH for gas and block-stuffing costs.

## Recommendation

Expose `lastUpdated` from `CrossChainRateReceiver` to consumers and enforce a maximum rate age in `AGETHPoolV3`:

```solidity
uint256 public constant MAX_RATE_AGE = 1 hours;

function getRate() public view returns (uint256) {
    uint256 lastUpdated = ICrossChainRateReceiver(agETHOracle).lastUpdated();
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Stale rate");
    return IOracle(agETHOracle).getRate();
}
```

`CrossChainRateReceiver.lastUpdated` is already set on each `lzReceive` delivery — it only needs to be read and enforced by `AGETHPoolV3`. [7](#0-6) 

## Proof of Concept

```solidity
// Foundry fork test — deploy AGETHPoolV3 pointing to AGETHRateReceiver

uint256 liveRate  = 1.05e18;  // current agETH/ETH rate
uint256 staleRate = 1.00e18;  // rate frozen by block stuffing (no updateRate called)
uint256 deposit   = 10 ether;
uint256 feeBps    = 10;       // 0.1%

uint256 fee       = deposit * feeBps / 10_000;
uint256 afterFee  = deposit - fee;

uint256 agETH_stale = afterFee * 1e18 / staleRate;  // ~9.99 agETH
uint256 agETH_live  = afterFee * 1e18 / liveRate;   // ~9.514 agETH

uint256 overIssuance = agETH_stale - agETH_live;     // ~0.476 agETH
assert(overIssuance > 0); // always true when staleRate < liveRate

// AGETHPoolV3.deposit() does NOT revert — no staleness check exists.
// Confirmed: viewSwapAgETHAmountAndFee() calls getRate() which calls
// CrossChainRateReceiver.getRate() returning stale `rate` unconditionally.
```

The `deposit()` call succeeds with no revert path related to rate freshness. The over-issuance is reachable on unmodified code whenever `lastUpdated` on the receiver lags behind the live rate. [4](#0-3) [8](#0-7)
