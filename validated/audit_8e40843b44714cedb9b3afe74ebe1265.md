### Title
`FeeEstimation` omits `l2_gas_consumed`, causing `overall_fee` to silently exceed the documented breakdown for V3 transactions — (File: `crates/apollo_rpc_execution/src/objects.rs`)

### Summary
`tx_execution_output_to_fee_estimation` builds a `FeeEstimation` response that includes `l2_gas_price` but omits `l2_gas_consumed`. For any V3 (`AllResources`) transaction that consumes L2 gas, the returned `overall_fee` is computed from the full three-component gas vector (L1 + L1-data + L2), yet the breakdown fields (`gas_consumed`, `data_gas_consumed`) cover only L1 and L1-data gas. The struct's own doc-comment and the published OpenAPI spec both assert `overall_fee = gas_consumed * gas_price + data_gas_consumed * data_gas_price`, which is arithmetically false whenever `l2_gas > 0`. Any wallet or tool that reconstructs the fee from the breakdown, or that uses the breakdown to derive L2 gas bounds for a V3 transaction, receives an authoritative-looking but wrong value.

### Finding Description

`FeeEstimation` is defined in `crates/apollo_rpc_execution/src/objects.rs`:

```rust
pub struct FeeEstimation {
    pub gas_consumed: Felt,        // L1 gas only
    pub l1_gas_price: GasPrice,
    pub data_gas_consumed: Felt,   // L1 data gas only
    pub l1_data_gas_price: GasPrice,
    // TODO(Tzahi): Add l2_gas_consumed. ...
    pub l2_gas_price: GasPrice,    // price present, amount absent
    /// The total amount of fee. This is equal to:
    /// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
    pub overall_fee: Fee,
    pub unit: PriceUnit,
}
``` [1](#0-0) 

The constructor `tx_execution_output_to_fee_estimation` populates `overall_fee` from `receipt.fee`, which is computed by `GasVector::cost` over all three gas components:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_price,
    overall_fee: tx_execution_output.execution_info.receipt.fee,  // includes l2_gas
    unit: tx_execution_output.price_unit,
})
``` [2](#0-1) 

`GasVector::cost` adds the tipped L2 gas cost:

```rust
let tipped_l2_gas_price = gas_prices.l2_gas_price.checked_add(tip.into())...;
// ...
(self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
``` [3](#0-2) 

So the true identity is:

```
overall_fee = l1_gas × l1_gas_price
            + l1_data_gas × l1_data_gas_price
            + l2_gas × (l2_gas_price + tip)
```

but the struct comment and the published OpenAPI schema both declare:

```
overall_fee = gas_consumed × gas_price + data_gas_consumed × data_gas_price
``` [4](#0-3) 

The gap equals `l2_gas_consumed × (l2_gas_price + tip)`, which is non-zero for every V3 transaction that executes Sierra code.

`TransactionContext::initial_sierra_gas` confirms that for `AllResources` transactions the initial gas is taken directly from `l2_gas.max_amount`, so L2 gas is always consumed and always contributes to `receipt.fee`: [5](#0-4) 

### Impact Explanation

**Incorrect fee breakdown returned to callers of `starknet_estimateFee` / `starknet_simulateTransactions`.**

1. A wallet that reconstructs the fee as `gas_consumed × l1_gas_price + data_gas_consumed × l1_data_gas_price` obtains a value strictly less than `overall_fee`. The wallet believes the fee is lower than it actually is.
2. A wallet that uses the breakdown fields to derive V3 resource bounds receives no `l2_gas_consumed` value. It cannot set a correct `l2_gas.max_amount`. Setting it to zero causes the transaction to revert with "Insufficient max L2Gas"; setting it to an arbitrary large value causes the user to over-commit funds.
3. The `l2_gas_price` field is present without a matching consumed-amount field, making the response internally inconsistent and misleading.

This matches: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

Every V3 (`AllResources`) transaction that executes any Sierra code consumes L2 gas. This is the dominant transaction type on Starknet post-0.13.3. The condition is triggered unconditionally for all such transactions, with no special setup required by the caller.

### Recommendation

Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it from `gas_vector.l2_gas.0` in `tx_execution_output_to_fee_estimation`. Update the struct doc-comment and the OpenAPI `FEE_ESTIMATE` schema description to reflect the three-component formula including the tipped L2 gas term. The existing TODO comment at line 104 already tracks this gap.

### Proof of Concept

```
1. Submit starknet_estimateFee for any V3 invoke transaction
   (resource_bounds = AllResources, any non-trivial calldata).

2. Receive FeeEstimation {
       gas_consumed      = A   (L1 gas)
       l1_gas_price      = P1
       data_gas_consumed = B   (L1 data gas)
       l1_data_gas_price = P2
       l2_gas_price      = P3
       overall_fee       = F
   }

3. Compute breakdown = A*P1 + B*P2.
   Observe breakdown < F.
   The hidden term is l2_gas_consumed * (P3 + tip),
   which is absent from the response.

4. A wallet that sets l2_gas.max_amount = 0 and submits the
   transaction receives:
     "Insufficient max L2Gas: max amount: 0, actual amount: N"
   where N > 0 is the actual L2 gas consumed.
```

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,

    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),

        l1_data_gas_price,

    })

    pub fn cost(&self, gas_prices: &GasPriceVector, tip: Tip) -> Fee {
        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }

                l2_gas.max_amount
            }
        }
    }
