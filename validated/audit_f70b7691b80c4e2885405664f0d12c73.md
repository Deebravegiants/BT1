### Title
Unchecked `binTotalShares` Overflow in `addLiquidity` Permanently Locks LP Funds — (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

---

### Summary

`LiquidityLib.addLiquidity` runs entirely inside an `unchecked` block. The line `binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd` (line 120) has no upper-bound guard on `sharesToAdd`. When a bin's token balances are both zero (a normal post-swap state), the only overflow-reverting guard — `_checkedMul` — is bypassed because `0 × anything = 0`. An attacker can therefore supply a crafted `sharesToAdd` that silently wraps `binTotalShares` to `0`, permanently bricking every existing LP's `removeLiquidity` call for that bin via division-by-zero.

---

### Finding Description

**Root cause — unchecked arithmetic with no input bound:** [1](#0-0) 

The entire `addLiquidity` body is wrapped in `unchecked { … }`. [2](#0-1) 

`binTotalSharesVal + sharesToAdd` is evaluated inside that `unchecked` scope with no cap on `sharesToAdd`. If the sum exceeds `type(uint256).max` it silently wraps to a small value (including `0`).

**Why `_checkedMul` does not protect this path:** [3](#0-2) 

`_checkedMul` is a separate internal function whose body is *not* inside `unchecked`, so it does revert on overflow. However, when a bin's token balances are both `0`, the call is `_checkedMul(0, sharesToAdd) = 0` — no overflow, no revert. [4](#0-3) 

Both `amount0Scaled` and `amount1Scaled` resolve to `0`, so the callback demands `0` tokens from the attacker.

**When can a bin have zero token balances with non-zero total shares?**

Swaps drain `token0BalanceScaled` / `token1BalanceScaled` from bins but never touch `binTotalShares`. After enough swap volume, a bin above (or below) the current price can be fully drained while existing LPs still hold recorded shares. [5](#0-4) 

**Corrupted state breaks `removeLiquidity`:** [6](#0-5) 

`removeLiquidity` computes `_checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal`. Once `binTotalSharesVal` is `0`, this is a division-by-zero revert for every legitimate LP trying to withdraw from that bin.

---

### Impact Explanation

Every LP who held shares in the targeted bin before the attack permanently loses the ability to call `removeLiquidity` for those shares. Their tokens remain locked in the pool contract with no recovery path. This satisfies **"broken core pool functionality causing loss of funds or unusable withdraw/liquidity flows"** and **"pool insolvency: balances fail to cover LP claims"** from the allowed impact gate.

---

### Likelihood Explanation

- **Unprivileged trigger**: any address can call `addLiquidity`.
- **Precondition**: a bin with both token balances at `0` and `binTotalShares > 0`. This is a routine post-swap state for any bin that has been fully traversed by the swap cursor.
- **No cost**: the attacker pays `0` tokens because `amount0Scaled = amount1Scaled = 0`.
- **Single transaction**: the entire attack fits in one `addLiquidity` call.

---

### Recommendation

1. **Add an explicit upper-bound check on `sharesToAdd`** before the `unchecked` block, e.g. `require(sharesToAdd <= type(uint128).max, "shares overflow")`, or enforce a protocol-wide maximum shares-per-position.
2. **Move the `binTotalShares` update outside the `unchecked` block**, or use checked arithmetic: `binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd;` with Solidity's default overflow protection.
3. **Add a guard for the zero-balance / non-zero-shares edge case**: if `binTotalSharesVal > 0` but both token balances are `0`, require `sharesToAdd` to be bounded by a safe maximum.

---

### Proof of Concept

```
Setup
─────
• Pool has bin B with token0BalanceScaled = 0, token1BalanceScaled = 0,
  binTotalShares[B] = S  (e.g. S = 1_000, left by legitimate LPs after swaps drained the bin)

Attack (single tx)
──────────────────
sharesToAdd = type(uint256).max - S + 1

addLiquidity(owner=attacker, salt=0, deltas={binIdxs:[B], shares:[sharesToAdd]}, ...)

Inside addLiquidity (unchecked):
  binTotalSharesVal = S
  _checkedMul(0, sharesToAdd) = 0          → amount0Scaled = 0, amount1Scaled = 0
  callback demands 0 tokens                → attacker pays nothing
  binTotalShares[B] = S + (2^256 - S + 1) = 2^256 ≡ 0   ← OVERFLOW
  positionBinShares[attackerKey] = sharesToAdd             ← huge

Post-attack
───────────
• binTotalShares[B] = 0
• Legitimate LP calls removeLiquidity(shares=S):
    binTotalSharesVal = 0
    _checkedMul(token0BalanceScaled, S) / 0  → DIVISION BY ZERO → revert
• LP funds permanently locked.
``` [7](#0-6)
