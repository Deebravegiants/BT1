Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero Mid-Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` uses a Synthetix-style accumulator where `rewardPerToken()` short-circuits to the frozen `rewardPerTokenStored` whenever `totalKernelStaked == 0`, while the `updateReward` modifier unconditionally advances `updatedAt`. Any `rewardRate * T` tokens that accrue during a zero-staking window are permanently unattributable and irrecoverable, as no sweep or rescue function exists in the contract.

## Finding Description
`rewardPerToken()` at L408-413 returns the frozen `rewardPerTokenStored` when `totalKernelStaked == 0`, skipping the accumulation of elapsed rewards. The `updateReward` modifier at L232-241 always executes `updatedAt = lastTimeRewardApplicable()`, advancing the timestamp even when no accumulation occurred. `initiateWithdrawal()` at L320-326 immediately decrements `totalKernelStaked` before the withdrawal delay elapses, meaning a sole staker can reduce `totalKernelStaked` to zero while a reward period is active. The guard in `notifyRewardAmount` at L566-570 only prevents initiating a new period with zero stakers; it does not prevent stakers from withdrawing to zero after a period has started. Once `totalKernelStaked` hits zero mid-period, every subsequent call to any `updateReward`-gated function advances `updatedAt` without accumulating `rewardPerTokenStored`, permanently losing `rewardRate * elapsed_zero_staking_time` tokens. The contract exposes no admin sweep, rescue, or recovery function — the only admin functions are `setRewardsDuration`, `notifyRewardAmount`, `setWithdrawalDelay`, and `setMaxNumberOfWithdrawalsPerUser`. The NatSpec at L18-22 explicitly acknowledges this risk and states it is mitigated purely operationally, not by code enforcement.

## Impact Explanation
Every `rewardRate * T` tokens (where `T` is the duration of the zero-staking window within an active reward period) are permanently frozen inside `KernelDepositPool`. They cannot be claimed by any user and cannot be recovered by any admin. This constitutes **Medium — Permanent freezing of unclaimed yield** per the allowed impact scope.

## Likelihood Explanation
The trigger is a standard, permissionless user action: calling `initiateWithdrawal()`. A single staker holding 100% of `totalKernelStaked` can reduce it to zero unilaterally at any time during an active reward period. In low-participation scenarios (early in an epoch, or after a large coordinated exit), this is realistic and repeatable across every reward period. The protocol's only mitigation is operational ("ensure tokens are always staked"), which is not enforced by code.

## Recommendation
Add a token-rescue function restricted to `DEFAULT_ADMIN_ROLE` that transfers surplus `rewardsToken` balance — i.e., `rewardsToken.balanceOf(address(this))` minus the sum of all pending `rewards[user]` entries — to a treasury address. Alternatively, track unallocated reward amounts explicitly and allow them to be rolled into the next reward period via `notifyRewardAmount`, similar to how remaining rewards are already rolled over at L582-583.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` while `totalKernelStaked > 0`. `rewardRate` is set, `finishAt = block.timestamp + duration`.
2. The sole staker calls `initiateWithdrawal(totalKernelStaked)`. `totalKernelStaked` becomes 0. The `updateReward` modifier fires: `rewardPerTokenStored` stays frozen (L409-410), `updatedAt` advances to `block.timestamp` (L234).
3. Time `T` passes. `rewardRate * T` tokens accumulate in the contract but `rewardPerToken()` keeps returning the frozen value.
4. A new staker calls `stake(1)`. `updateReward` fires again: `rewardPerTokenStored` is still frozen, `updatedAt` advances to now. The new staker's `userRewardPerTokenPaid` equals the frozen `rewardPerTokenStored` (L238).
5. All future rewards are calculated from the current frozen baseline. The tokens that accrued during step 3 are permanently unattributable.
6. `rewardsToken.balanceOf(address(this))` exceeds the sum of all claimable `rewards[user]` by exactly `rewardRate * T`. No function exists to recover the difference.

**Foundry invariant test plan:** Deploy `KernelDepositPool`, call `notifyRewardAmount`, have a single staker call `initiateWithdrawal` for full balance, warp forward, then assert `rewardsToken.balanceOf(pool) > sum(rewards[user] for all users)` — the invariant will be violated with no recovery path.
