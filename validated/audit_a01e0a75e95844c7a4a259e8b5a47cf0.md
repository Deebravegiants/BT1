### Title
Unprivileged Contract Can Cause Epoch Finalization to Abort via Duplicate Boundary Account in Dynamic Resharding - (`core/store/src/trie/split.rs`, `core/primitives/src/shard_layout/v3.rs`, `chain/epoch-manager/src/lib.rs`)

### Summary

`find_trie_split` can deterministically produce a boundary account (via the Split B "-0" suffix path in `best_split_at_current_path`) that already exists as a boundary account in the current `ShardLayoutV3`. No guard exists between `find_trie_split` and `ShardLayoutV3::derive_impl` to detect this collision. When `finalize_epoch` calls `next_next_shard_layout` → `derive_v3` with the duplicate boundary account, `derive_impl` returns `DuplicateBoundaryAccount`, which propagates as `EpochError::ShardingError` through `finalize_epoch`'s `?` operator, aborting epoch finalization on every honest node and halting the chain.

### Finding Description

**The Split B ("-0" suffix) path in `best_split_at_current_path`:** [1](#0-0) 

The function appends the literal bytes `"-0"` to the current nibble path and calls `nibbles_to_account_id`. This check is purely syntactic — it validates that the resulting bytes form a valid `AccountId` string, but does **not** check whether that account ID already exists as a boundary account in the current `ShardLayoutV3`.

**`derive_impl` rejects duplicate boundary accounts:** [2](#0-1) 

**`next_next_shard_layout` propagates the error fatally:** [3](#0-2) 

**`finalize_epoch` propagates the error with `?`:** [4](#0-3) 

**`compute_proposed_split` has no duplicate-boundary guard:** [5](#0-4) 

Non-`Storage` `FindSplitError` variants are swallowed, but `DuplicateBoundaryAccount` is a `ShardLayoutError` that only surfaces later in `finalize_epoch`, where it is not swallowed.

### Impact Explanation

Every honest node calls `finalize_epoch` when processing the first block of a new epoch. If `next_next_shard_layout` → `derive_v3` returns `DuplicateBoundaryAccount`, the `?` in `finalize_epoch` propagates the error, making it impossible to finalize the epoch on any node. Since `find_trie_split` is deterministic (all nodes compute the same result from the same trie state), all nodes fail identically. The chain cannot progress past the epoch boundary — a complete consensus halt.

### Likelihood Explanation

**Attacker-controlled inputs:**
1. `VMLogic::storage_write` (via `RuntimeExt::storage_set`) writes `TrieKey::ContractData { account_id, key }` entries — a normal, permissioned contract operation available to any deployed contract. [6](#0-5) [7](#0-6) 

2. The attacker reads the current `ShardLayoutV3` boundary accounts (public chain state). Suppose "ab-0" is a boundary account (confirmed possible by the existing test): [8](#0-7) 

3. The attacker creates account "ab" in the shard containing accounts < "ab-0" (Shard L). Account "ab" < "ab-0" lexicographically, so it resides in Shard L.

4. The attacker deploys a contract on "ab" and writes enough `CONTRACT_DATA` to make the Split B candidate ("ab-0") have a lower `mem_diff` than Split A ("ab"). The `attached_data_mem_usage` calculation includes CONTRACT_DATA weight: [9](#0-8) 

5. When Shard L's total memory exceeds `memory_usage_threshold`, `check_dynamic_resharding` calls `find_trie_split`: [10](#0-9) 

6. `find_trie_split` descends to path "ab" in the ACCOUNT subtree, evaluates Split B = "ab-0", selects it as optimal, and returns it.

7. "ab-0" is embedded in the chunk header's `proposed_split`, then in the block header's `shard_split` (validated deterministically by all nodes via `validate_block_shard_split`): [11](#0-10) 

8. At epoch boundary, `finalize_epoch` calls `derive_v3("ab-0")`, which fails with `DuplicateBoundaryAccount`.

**Barrier**: The attacker needs enough NEAR tokens to push Shard L's trie memory above `memory_usage_threshold`. This is a financial cost, not a protocol privilege. A well-funded attacker can achieve this.

### Recommendation

Add a guard in `compute_proposed_split` (or in `check_dynamic_resharding` / `find_trie_split`) that checks whether the proposed `boundary_account` already exists in the current shard layout's boundary accounts before returning it. If a collision is detected, either skip that candidate or return `None` (no split). This check is cheap (binary search on the sorted `boundary_accounts` vec) and eliminates the collision before it reaches `finalize_epoch`.

Alternatively, add a recovery path in `next_next_shard_layout` that logs a warning and returns `next_shard_layout` unchanged (no split) instead of propagating the error, preventing chain halt even if the upstream guard is missed.

### Proof of Concept

1. Observe current `ShardLayoutV3` boundary accounts; find one with "-0" suffix, e.g., "ab-0".
2. Create account "ab" in the shard containing accounts < "ab-0".
3. Deploy a contract on "ab"; call `storage_write` in a loop to accumulate CONTRACT_DATA until Shard L's total trie memory exceeds `memory_usage_threshold` and the Split B candidate "ab-0" has lower `mem_diff` than Split A "ab".
4. Wait for the epoch boundary. `compute_proposed_split` calls `find_trie_split`, which returns `TrieSplit { boundary_account: "ab-0", ... }`.
5. The block producer embeds `shard_split = Some((shard_id, "ab-0"))` in the last block of the epoch. All nodes validate and accept this block.
6. All nodes call `finalize_epoch`. `next_next_shard_layout` calls `derive_v3("ab-0")`. `derive_impl` finds "ab-0" in `boundary_accounts` via `binary_search`, returns `Err(DuplicateBoundaryAccount { account_id: "ab-0" })`. `finalize_epoch` propagates the error. Epoch finalization aborts on all nodes. Chain halts. [12](#0-11) [13](#0-12) [14](#0-13)

    fn best_split_at_current_path(&self) -> FindSplitResult<Option<TrieSplit>> {
        let split_a = self.current_account().map(|account_id| {
            // When splitting at the current path, middle memory goes to the right, as the boundary
            // account is included in the right child.
            TrieSplit::new(account_id, self.left_memory, self.right_memory + self.middle_memory)
        });

        let mut ext_nibbles = self.nibbles.clone();
        ext_nibbles.extend(bytes_to_nibbles("-0".as_bytes()));
        let split_b = nibbles_to_account_id(&ext_nibbles)
            .map(|account_id| {
                // When splitting at the suffixed path, nodes at the current path go to the left
                // together with their 'attached data'. All children accounts go to the right.
                let curr_nodes_mem = self.current_nodes_mem_usage()?;
                let attached_data_mem = self.attached_data_mem_usage()?;
                let left = self.left_memory + curr_nodes_mem + attached_data_mem;
                let right =
                    self.right_memory + self.middle_memory - curr_nodes_mem - attached_data_mem;
                Ok::<_, FindSplitError>(TrieSplit::new(account_id, left, right))
            })
            .transpose()?;

        Ok([split_a, split_b].into_iter().flatten().min_by_key(|split| split.mem_diff()))
    }

pub fn validate_block_shard_split(
    epoch_manager: &dyn EpochManagerAdapter,
    header: &BlockHeader,
    chunk_headers: &[ShardChunkHeader],
) -> Result<(), Error> {
    let is_last_block = epoch_manager.is_produced_block_last_in_epoch(
        header.height(),
        header.prev_hash(),
        header.last_final_block(),
    )?;

    let expected_shard_split = if is_last_block {
        let protocol_version = epoch_manager.get_epoch_protocol_version(header.epoch_id())?;
        epoch_manager.get_upcoming_shard_split(
            protocol_version,
            header.prev_hash(),
            chunk_headers,
        )?
    } else {
        None
    };

    let header_shard_split = header.shard_split();
    if header_shard_split != expected_shard_split.as_ref() {
        DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["block_header"]).inc();
        return Err(Error::InvalidBlockHeaderShardSplit(format!(
            "header has {:?}, expected {:?} (block hash: {:?} height: {:?})",
            header_shard_split,
            expected_shard_split,
            header.hash(),
            header.height(),
        )));
    }

    Ok(())
