### Title
`respond_ckd` Has Two Structurally Divergent Response Paths — `AppPublicKey` Variant Accepts Any Arbitrary Output Without Cryptographic Verification, Bypassing Threshold Integrity - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in `MpcContract` contains two code paths for finalizing a Confidential Key Derivation (CKD) response. For `AppPublicKeyPV` requests, the contract enforces a full BLS12-381 pairing check (`ckd_output_check`) before resolving the yield. For `AppPublicKey` requests, the match arm is an empty no-op — no cryptographic verification is performed at all. This structural divergence means a single attested MPC participant can call `respond_ckd` with any arbitrary `(big_y, big_c)` pair for an `AppPublicKey` request, and the contract will unconditionally accept it and deliver the fraudulent key material to the waiting caller.

---

### Finding Description

In `crates/contract/src/lib.rs` at lines 675–682, the response verification is gated entirely on the variant of `CKDAppPublicKey` embedded in the request:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty: no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` (defined in `crates/contract/src/primitives/ckd.rs` lines 80–102) verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` using the network's BLS12-381 public key. This cryptographically binds the response to the MPC network's master secret and the specific `app_id`.

For `AppPublicKey`, the empty arm `{}` performs no such check. The contract proceeds directly to `resolve_yields_for` (line 684), serializing and delivering whatever `big_y` and `big_c` the caller supplied.

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active()` (line 666), which requires the caller to be a registered, attested participant — but does **not** require threshold agreement, does not verify the response is the output of the threshold computation, and does not restrict which attested participant may call the function (any one of them may, not only the designated leader).

The existing unit test at lines 3403–3441 directly confirms the issue: it passes `big_y = [1u8; 48]` and `big_c = [2u8; 48]` — raw garbage byte arrays that are not valid BLS12-381 G1 points — and the call succeeds without error.

---

### Impact Explanation

A single malicious attested MPC participant can race to call `respond_ckd` before the honest leader for any pending `AppPublicKey` CKD request. The contract will accept the fraudulent `(big_y, big_c)` pair and resolve the yield, delivering attacker-controlled key material to the requesting TEE application. The user's application will compute `derived_key = big_c_attacker − big_y_attacker · a`, which is not `hash(app_id, pk) · msk` — the correct threshold-derived secret. The user receives a wrong key with no on-chain signal of failure.

This breaks the core safety invariant of the CKD protocol: that the derived key is the output of a threshold computation over the MPC network's master secret. A single participant below threshold can unilaterally substitute the output.

**Matched impact class:** Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants (the threshold integrity of CKD output) without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

- Requires one attested MPC participant to act maliciously — strictly below the signing threshold.
- No special timing window is needed beyond submitting `respond_ckd` before the honest leader does for a given request. Since the contract imposes no leader-only restriction on `respond_ckd`, any attested participant may call it at any time while the request is pending.
- The `AppPublicKey` variant is the legacy/default variant (it is the plain G1 key format accepted by the old untagged JSON path per `crates/near-mpc-crypto-types/src/ckd.rs` lines 29–53), so it is the most commonly used variant in practice.

---

### Recommendation

Add on-chain output verification for `AppPublicKey` requests in `respond_ckd`. Since `AppPublicKey` does not carry a G2 component (`pk2`), the existing `ckd_output_check` pairing equation cannot be applied directly. Options include:

1. **Require `AppPublicKeyPV` for all new CKD requests** and deprecate `AppPublicKey`, ensuring every response is verifiable on-chain.
2. **Introduce a threshold-vote mechanism** for `AppPublicKey` responses (analogous to `vote_pk`/`vote_reshared`) so that the contract only resolves the yield once a quorum of attested participants have submitted the same `(big_y, big_c)`.
3. At minimum, **document explicitly** that `AppPublicKey` responses carry no on-chain integrity guarantee and that callers must perform their own post-decryption verification.

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(some_g1_point)`. A pending yield is registered in `pending_ckd_requests`.
2. Malicious attested participant calls `respond_ckd(ckd_request, CKDResponse { big_y: [1u8;48], big_c: [2u8;48] })`.
3. `respond_ckd` passes all guards (signer check, protocol state, TEE flag, attestation check).
4. The match arm at line 676 is `{}` — no verification occurs.
5. `resolve_yields_for` at line 684 resolves the yield with the fraudulent response bytes.
6. The user's TEE application receives `big_y = [1u8;48]`, `big_c = [2u8;48]` and derives a wrong key.

This is directly confirmed by the passing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` at lines 3403–3441, which supplies exactly these garbage byte arrays and asserts `Ok(())`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
