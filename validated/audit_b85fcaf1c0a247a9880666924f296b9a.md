### Title
Pending Sign/CKD/ForeignTx Requests Become Unresolvable When Contract Transitions to `Initializing` State ŌĆö (`crates/contract/src/lib.rs`)

---

### Summary

When the MPC contract transitions from `Running` to `Initializing` via `vote_add_domains`, all in-flight sign, CKD, and foreign-tx requests stored in the pending-request maps become permanently unresolvable for the duration of keygen. The `respond()` / `respond_ckd()` / `respond_verify_foreign_tx()` entry points all gate on `is_running_or_resharing()`, which returns `false` for `Initializing`. Requests that were accepted under `Running` assumptions are now incompatible with the new contract mode, mirroring the `matchingEnabled` toggle class exactly: state created under mode A silently breaks under mode B.

---

### Finding Description

`respond()` (and its siblings) contain an explicit mode guard:

```rust
if !self.protocol_state.is_running_or_resharing() {
    return Err(InvalidState::ProtocolStateNotRunning.into());
}
```

`is_running_or_resharing()` is defined as:

```rust
pub fn is_running_or_resharing(&self) -> bool {
    matches!(
        self,
        ProtocolContractState::Running(_) | ProtocolContractState::Resharing(_)
    )
}
```

`Initializing` is not matched, so every `respond*` call returns `Err` while keygen is in progress.

Meanwhile, `sign()` / `request_app_private_key()` / `verify_foreign_transaction()` call `check_request_preconditions`, which calls `domain_registry()`:

```rust
let domains = match self.protocol_state.domain_registry() {
    Ok(domains) => domains,
    Err(err) => env::panic_str(&err.to_string()),
};
```

`domain_registry()` for `Initializing` returns `Err(ProtocolStateNotRunningNorResharing)`, so new requests are also rejected. The result is a window ŌĆö potentially spanning the entire keygen ceremony ŌĆö during which:

1. No new requests can be submitted.
2. All previously accepted requests (stored in `pending_signature_requests`, `pending_ckd_requests`, `pending_verify_foreign_tx_requests`) cannot be fulfilled.

The NEAR yield-resume mechanism has a fixed timeout. If keygen outlasts that timeout, the yield callback fires with `Err(PromiseError::Failed)`, the pending-request entry is cleaned up, and the user receives a failure response with no recourse other than resubmission.

---

### Impact Explanation

**Medium.** This is a request-lifecycle and accounting-invariant break. The contract's documented guarantee is that a submitted request will eventually be fulfilled. When `vote_add_domains` fires, that guarantee is silently voided for every in-flight request. Users lose their 1 yoctoNEAR deposit and must resubmit after keygen completes. For high-value cross-chain operations (e.g., a time-sensitive Bitcoin transaction that depends on the MPC signature), the forced timeout can cause irreversible downstream harm beyond the deposit itself. The impact matches the allowed scope: "request-lifecycle ŌĆ” manipulation that breaks production safety/accounting invariants."

---

### Likelihood Explanation

`vote_add_domains` requires **all** current participants to cast identical votes before the transition fires. This is a unanimous-consent governance action and is a routine operational step (adding new cryptographic domains). Any production deployment that adds a domain while users are actively submitting requests will hit this window. The transition is not adversarially gated ŌĆö it is a normal protocol operation ŌĆö so the window opens whenever governance and user activity overlap, which is expected in a live network.

---

### Recommendation

1. **Drain or pause before transitioning.** Before the contract moves to `Initializing`, either (a) reject new requests at the `sign` / `request_app_private_key` / `verify_foreign_transaction` layer once a `vote_add_domains` quorum is forming, or (b) allow `respond*` to proceed during `Initializing` for requests that were accepted under the previous `Running` state.

2. **Extend `is_running_or_resharing` to cover `Initializing` for respond paths.** The `respond*` functions do not use any keyset or domain state that changes during keygen; they only need the public key of the domain referenced by the request, which is unchanged. Allowing `respond*` during `Initializing` for pre-existing requests would eliminate the incompatibility window entirely.

3. **Document the invariant break.** At minimum, operators should be warned that triggering `vote_add_domains` while requests are in flight will cause those requests to time out.

---

### Proof of Concept

```
1. Contract is in Running state; domain 0 (Secp256k1/Sign) exists.

2. User calls sign({ domain_id: 0, payload: X, path: "m/44'/60'/0'/0/0" })
   ŌåÆ yield promise created; request stored in pending_signature_requests.

3. All N participants call vote_add_domains([{ id: 1, protocol: Frost, ... }])
   ŌåÆ unanimous vote reached; contract transitions to Initializing.

4. MPC nodes observe the pending sign request and call respond(request, sig).
   ŌåÆ respond() hits:
       if !self.protocol_state.is_running_or_resharing() {
           return Err(InvalidState::ProtocolStateNotRunning.into());
       }
   ŌåÆ returns Err; the yield is NOT resolved.

5. NEAR yield timeout fires.
   ŌåÆ return_signature_and_clean_state_on_success(request, Err(PromiseError::Failed))
   ŌåÆ pending_signature_requests entry removed; user's promise resolves as failure.

6. User's cross-chain operation fails; user must resubmit after keygen completes.
   Any time-sensitive downstream action (e.g., a Bitcoin HTLC with a locktime)

    pub fn vote_add_domains(
        &mut self,
        domains: Vec<DomainConfig>,
    ) -> Result<Option<InitializingContractState>, Error> {
        if domains.is_empty() {
            return Err(DomainError::AddDomainsMustAddAtLeastOneDomain.into());
        }
        let num_participants = u64::try_from(self.parameters.participants().len())
            .expect("participant count fits in u64");
        for domain in &domains {
            validate_domain_purpose(domain)?;
            validate_domain_threshold(domain, num_participants)?;
        }
        // Keep trust assumptions consistent: a domain must never require more shares to
        // reconstruct than the GovernanceThreshold demands to govern. Route through the
        // canonical helper so the cross-domain invariant has a single source of truth.
        ThresholdParameters::validate_governance_against_reconstruction(
            num_participants,
            self.parameters.threshold(),
            max_reconstruction_threshold(&domains),
        )?;
        let participant = AuthenticatedParticipantId::new(self.parameters.participants())?;
        let n_votes = self.add_domains_votes.vote(domains.clone(), &participant);
        if self.parameters.participants().len() as u64 == n_votes {
            let new_domains = self.domains.add_domains(domains.clone())?;
            Ok(Some(InitializingContractState {
                generated_keys: self.keyset.domains.clone(),
                domains: new_domains,
                epoch_id: self.keyset.epoch_id,
                generating_key: KeyEvent::new(
                    self.keyset.epoch_id,
                    domains[0].clone(),
                    self.parameters.clone(),
                ),
                cancel_votes: BTreeSet::new(),
            }))
        } else {
            Ok(None)
        }
    }
