Audit Report

## Title
Permissionless `sendFunds()` susceptible to block stuffing, temporarily suppressing rsETH rate — (`contracts/FeeReceiver.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access control and is the sole mechanism by which MEV/execution-layer rewards are reflected in protocol TVL and the rsETH price. Because `getETHDistributionData()` explicitly excludes the FeeReceiver balance from TVL until `sendFunds()` is called, an attacker can perform block stuffing to delay this call, temporarily suppressing the rsETH exchange rate and denying rsETH holders their promised yield accrual for the duration of the attack.

## Finding Description
`FeeReceiver.sendFunds()` is declared `external` with no role modifier:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`LRTDepositPool.getETHDistributionData()` explicitly documents that FeeReceiver balances are excluded from TVL until forwarded (L464-466), and only counts `address(this).balance` — the deposit pool itself — plus NDCs, EigenLayer, unstaking vault, and converter. `LRTOracle._getTotalEthInProtocol()` (L331-349) computes rsETH price by calling `getTotalAssetDeposits(asset)` for each supported asset, which routes through `getETHDistributionData()` for ETH. As a result, any ETH sitting in `FeeReceiver` is invisible to the oracle until `sendFunds()` is executed. An attacker who monitors the mempool for a pending `sendFunds()` call can submit high-gas-price filler transactions to fill each block's gas limit for consecutive blocks, excluding `sendFunds()` from inclusion. During this window, `FeeReceiver.balance` grows but `LRTDepositPool` receives nothing, so `_getTotalEthInProtocol()` returns a stale lower value and `rsETHPrice` is not updated to reflect accrued rewards.

## Impact Explanation
No principal is lost; the ETH remains in `FeeReceiver` and will eventually be forwarded. However, rsETH holders do not see the rate increase they are entitled to during the attack window. This matches two allowed impact classes: **Low. Block stuffing** and **Low. Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
Block stuffing on Ethereum mainnet requires filling ~30M gas per block at competitive base fee + priority fee, making it expensive. The attack is economically rational only when accumulated MEV rewards in `FeeReceiver` are large enough to justify the cost, or when the attacker profits from a secondary position (e.g., a short on rsETH rate or a competing LRT). The permissionless nature of `sendFunds()` and the explicit TVL exclusion of `FeeReceiver` balance make the structural precondition sound. Likelihood is **Low** but non-zero.

## Recommendation
Include the FeeReceiver balance directly in `getETHDistributionData()` / `_getTotalEthInProtocol()` by reading `address(feeReceiver).balance` as part of TVL, so accrued rewards are reflected in the rsETH rate regardless of whether `sendFunds()` has been called. Alternatively, restrict `sendFunds()` to a trusted keeper role and ensure the keeper infrastructure is robust against censorship.

## Proof of Concept
```solidity
// Fork test (Foundry)
function testBlockStuffingDelaySendFunds() public {
    // 1. Simulate MEV rewards accumulating in FeeReceiver
    vm.deal(address(feeReceiver), 10 ether);

    // 2. Record current rsETH price (stale, excludes FeeReceiver balance)
    uint256 priceBefore = lrtOracle.rsETHPrice();

    // 3. Simulate N blocks passing without sendFunds() being included
    //    (attacker fills blocks; modeled by simply not calling sendFunds())
    vm.roll(block.number + 50);
    vm.warp(block.timestamp + 600); // ~50 blocks * 12s

    // 4. Assert FeeReceiver still holds the balance
    assertEq(address(feeReceiver).balance, 10 ether);

    // 5. Assert LRTDepositPool ETH balance is unchanged
    assertEq(address(lrtDepositPool).balance, depositPoolBalanceBefore);

    // 6. Update oracle — price does NOT reflect the 10 ETH sitting in FeeReceiver
    lrtOracle.updateRSETHPrice();
    uint256 priceAfterDelay = lrtOracle.rsETHPrice();
    assertEq(priceAfterDelay, priceBefore); // rate unchanged despite 10 ETH in FeeReceiver

    // 7. Now call sendFunds() — price jumps
    feeReceiver.sendFunds();
    lrtOracle.updateRSETHPrice();
    uint256 priceAfterForward = lrtOracle.rsETHPrice();
    assertGt(priceAfterForward, priceBefore); // rate now reflects rewards
}

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
