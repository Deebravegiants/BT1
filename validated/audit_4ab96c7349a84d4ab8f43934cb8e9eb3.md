### Title
`is_proposal_init_valid` Skips `fee_proposal` Bounds Enforcement When `fee_actual` Is `None`, Allowing Arbitrary Fee Injection During V0_14_3 Transition — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` uses an `if let (Some(fee_actual), Some(fee_proposal)) = ...` guard to enforce that a proposer's `fee_proposal_fri` lies within a margin of `fee_actual`. When `fee_actual` is `None` — which occurs during genesis and during the V0_14_3 protocol-upgrade transition window — the guard short-circuits and **no bound is enforced at all**. A malicious proposer can publish any `fee_proposal_fri` value (e.g., `u128::MAX` or `0`) during this window; every honest validator will accept it, the value is committed into the block and stored in `fee_proposals_window`, and it permanently skews future `fee_actual` (the SNIP-35 base fee) computations for the subsequent `window_size` blocks.

---

### Finding Description

**The guard that silently drops the check:** [1](#0-0) 

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(...);
    }
}
```

When `proposal_init_validation.fee_actual` is `None`, the `if let` pattern does not match and the entire bounds check is bypassed. There is no fallback assertion — not even a check that `fee_proposal == l2_gas_price` (the value an honest proposer is required to use in this state).

**When `fee_actual` is `None`:**

`fee_actual` is computed by `compute_fee_actual`, which returns `None` in two production-reachable cases:

1. **Genesis / near-genesis** (`height < window_size`): the sliding window cannot cover `window_size` prior heights.
2. **V0_14_3 upgrade transition**: `compute_fee_actual` returns `None` for any height whose window `[height − window_size, height − 1]` contains even one block with `fee_proposal_fri = None` (i.e., a pre-V0_14_3 block). This condition persists for exactly `window_size` (= 10) blocks after the upgrade. [2](#0-1) 

**What an honest proposer does when `fee_actual` is `None`:** [3](#0-2) 

The honest proposer freezes at `self.l2_gas_price`. The code comment says "the validator derives the same fallback so both sides agree." But the validator does **not** enforce `fee_proposal == l2_gas_price`; it accepts any value.

**How `fee_proposal` is committed and persisted:**

After consensus, `finalize_decision` unconditionally stores the proposer-supplied value: [4](#0-3) 

This value then feeds future `fee_actual` computations (median of the window): [5](#0-4) 

**`fee_proposal` is also bound into the `ProposalCommitment`:** [6](#0-5) 

Because validators skip the bounds check when `fee_actual` is `None`, they compute the same commitment as the malicious proposer and consensus succeeds.

---

### Impact Explanation

A malicious proposer who is selected during the V0_14_3 transition window (10 blocks) can set `fee_proposal_fri` to any value — e.g., `u128::MAX` or `0`. All honest validators accept the proposal (bounds check skipped). The injected value is stored in `fee_proposals_window` and enters the median computation for the next `window_size` blocks. If the attacker controls ≥ 5 of the 10 transition blocks, they can set `fee_actual` to an arbitrary value, directly manipulating the SNIP-35 base fee charged to all users for the subsequent window. This is an incorrect fee/gas accounting effect with direct economic impact.

**Impact class:** Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.

---

### Likelihood Explanation

The V0_14_3 upgrade is a known, scheduled, network-wide event. The 10-block transition window is predictable. A malicious validator who is selected as proposer during any of those 10 blocks can inject an extreme value with zero additional cost. Controlling 5+ of the 10 blocks requires repeated proposer selection, which is unlikely for a single validator in a large committee but feasible in a small or partially-compromised committee. Even a single injected extreme value shifts the median by one position, which may be sufficient to move `fee_actual` outside the honest range once the window fills.

---

### Recommendation

When `fee_actual` is `None`, enforce that `fee_proposal_fri` equals the fallback value `l2_gas_price_fri` (the value honest proposers are required to use). Replace the silent skip with an explicit equality check:

```rust
match (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri) {
    (Some(fee_actual), Some(fee_proposal)) => {
        let (lower_bound, upper_bound) = fee_proposal_bounds(
            fee_actual,
            VersionedConstants::latest_constants().fee_proposal_margin_ppt,
        );
        if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
            return Err(ValidateProposalError::InvalidProposalInit(...));
        }
    }
    (None, Some(fee_proposal)) => {
        // Window incomplete: honest proposer must freeze at l2_gas_price.
        if fee_proposal != proposal_init_validation.l2_gas_price_fri {
            return Err(ValidateProposalError::InvalidProposalInit(
                ...,
                format!(
                    "fee_proposal must equal l2_gas_price_fri when fee_actual is unavailable: \
                     expected={}, proposed={}",
                    proposal_init_validation.l2_gas_price_fri.0, fee_proposal.0
                ),
            ));
        }
    }
    _ => {}
}
```

This mirrors the ERC-4626 fix: when the "window is paused/incomplete," enforce the specific fallback value rather than accepting any value.

---

### Proof of Concept

1. Network upgrades to V0_14_3 at block `N`. Blocks `N − 9` through `N − 1` have `fee_proposal_fri = None` (pre-V0_14_3).

2. For blocks `N` through `N + 9`, `compute_fee_actual` returns `None` because the window `[height − 10, height − 1]` contains at least one `None` entry. [7](#0-6) 

3. Malicious proposer at block `N` sets `init.fee_proposal_fri = Some(GasPrice(u128::MAX))`.

4. Validator constructs `ProposalInitValidation` with `fee_actual = None`: [8](#0-7) 

5. `is_proposal_init_valid` reaches the bounds check. The `if let (Some(fee_actual), Some(fee_proposal))` pattern does not match (`fee_actual` is `None`). The check is silently skipped. The proposal is accepted. [9](#0-8) 

6. `finalize_decision` stores `GasPrice(u128::MAX)` in `fee_proposals_window`: [10](#0-9) 

7. At block `N + 10`, the window is fully V0_14_3. `compute_fee_actual` returns `Some(median)` where the median is skewed by the injected `u128::MAX`. All transactions in blocks `N + 10` through `N + 19` are charged fees based on this corrupted `fee_actual`.
