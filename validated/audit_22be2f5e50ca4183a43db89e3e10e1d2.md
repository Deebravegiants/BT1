### Title
Byzantine Last-Quorum-Party Equivocates `c_all`/`h_all` to Split Non-Quorum Parties onto a Different Public Key - (`src/cbmpc/protocol/ec_dkg.cpp`)

### Summary

In `key_share_mp_t::dkg_or_refresh_ac`, the `h_all` consistency check at line 377 only enforces agreement among quorum parties by comparing each quorum party's `h_all` against the value received from `representative_quorum_pid_index` (the highest-indexed quorum party). The non-quorum party check at line 401 independently verifies the representative's `h_all` against the `cs` values the non-quorum party itself collected. Because `plain_broadcast` is a point-to-point send with no equivocation detection, a Byzantine representative can send a different `c_all` (and matching `h_all`) to non-quorum parties than to quorum parties. All per-party checks pass, but non-quorum parties derive a different public key `Q` than quorum parties.

### Finding Description

**`representative_quorum_pid_index` selection:**

The loop at lines 284–292 unconditionally overwrites `representative_quorum_pid_index` on every quorum member, so it always ends up as the **highest-indexed quorum party**. [1](#0-0) 

**`plain_broadcast` has no equivocation protection:**

`plain_broadcast` resolves to `group_message(party_set_t::all(), party_set_t::all(), msgs...)`, which is a raw point-to-point send/receive. A Byzantine party can send different serialized bytes to different recipients. [2](#0-1) 

**`uniform_msg_t` packs the same `msg` field for all recipients when honest**, but a Byzantine party controls its own transport and can violate this. [3](#0-2) 

**The quorum-side h_all check is vacuous for the representative itself:**

```cpp
if (h_all._j != h_all.received(representative_quorum_pid_index))
    return coinbase::error(E_CRYPTO, "h_all mismatch");
```

When `j == representative_quorum_pid_index`, this reduces to `h_all._j != h_all._j`, which is always false — no error is ever raised. For other honest quorum parties `j`, the check compares their locally-computed `h_all` against the representative's value, but only within the quorum bubble. [4](#0-3) 

**The non-quorum check is self-referential and can be satisfied with forged inputs:**

```cpp
if (!quorum_party_set.has(i)) {
    if (h_all.received(representative_quorum_pid_index) != crypto::sha256_t::hash(cs, quorum_pids, sid))
        return coinbase::error(E_CRYPTO, "h_all mismatch");
}
```

`cs[representative_quorum_pid_index]` is populated from `c_all._j` (line 389), which is the `c_all` the non-quorum party received from the Byzantine representative. The Byzantine representative controls both `h_all.received(representative_quorum_pid_index)` and `cs[representative_quorum_pid_index]`, so it can trivially satisfy this check by sending a consistent `(c_all_fake, h_all_fake)` pair. [5](#0-4) 

**All downstream checks also pass:**

- **Commitment opening (line 387):** The Byzantine representative chose `c_all_fake = commitment(Rs_fake, rho_fake)`, so `com_R_tag.open(Rs_fake, pi_r_fake)` succeeds.
- **ZK proof (line 391):** The Byzantine representative knows the discrete logs of `Rs_fake` and produces a valid `pi_r_fake`.
- **Share verification (line 396):** The Byzantine representative sends a consistent `(xij_fake, ac_pub_all_fake)` pair.
- **Final Q consistency (lines 428–435):** Both `Q` and `Qis` are derived from `ac_pub_all._j` which includes the fake data; the Byzantine representative can make them internally consistent. [6](#0-5) [7](#0-6) 

### Impact Explanation

Non-quorum parties store a different `key.Q` and `key.Qis` than quorum parties after `dkg_ac` or `refresh_ac` completes. Any subsequent operation that depends on a consistent public key across all parties — including signature verification, `to_additive_share`, or a follow-on `refresh_ac` — will silently operate on the wrong key material for non-quorum parties. This is a correctness break in the AC-DKG output that falls under the "High" impact scope: a public API reachable validation bypass in DKG creates accepted invalid cryptographic output.

### Likelihood Explanation

The attacker must be the highest-indexed quorum party (a Byzantine participant within the quorum). This is a specific but realistic condition: in any deployment where the quorum is not all-honest, the last quorum party by index can mount this attack. No threshold collusion is required — a single Byzantine quorum party suffices.

### Recommendation

Replace the single-representative `h_all` consistency check with a proper equivocation-resistant broadcast. The standard pattern already used elsewhere in this codebase (e.g., `committed_group_broadcast` in `committed_broadcast.h`) hashes all received `c_all` values into a shared transcript that every party — quorum and non-quorum alike — independently verifies against the same value. Concretely:

1. After the `c_all` broadcast, have every party (including non-quorum) compute `h_all = hash(all_received_c_s, quorum_pids, sid)` and broadcast it.
2. Every party checks that **all** received `h_all` values equal its own locally-computed value, not just the representative's.

This eliminates the equivocation window entirely. [8](#0-7) 

### Proof of Concept

5-party setup: parties `{p0, p1, p2, p3, p4}`, quorum `{p0, p1, p2}` (so `representative_quorum_pid_index = 2`), non-quorum `{p3, p4}`. Party `p2` is Byzantine.

1. `p2` generates two valid AC-sharing tuples: `(Rs_real, rho_real, pi_real, xij_real)` and `(Rs_fake, rho_fake, pi_fake, xij_fake)` where `Rs_fake ≠ Rs_real`.
2. `p2` sends `c_all_real = commitment(Rs_real, rho_real)` to `p0`, `p1`.
3. `p2` sends `c_all_fake = commitment(Rs_fake, rho_fake)` to `p3`, `p4`.
4. `p2` computes `h_all_quorum = hash({c_all_real, c_all_p0, c_all_p1}, quorum_pids, sid)` and sends it to `p0`, `p1`.
5. `p2` computes `h_all_nonquorum = hash({c_all_fake, c_all_p0, c_all_p1}, quorum_pids, sid)` and sends it to `p3`, `p4`.
6. `p2` sends `(ac_pub_all_real, pi_real, rho_real, xij_real)` to `p0`, `p1` and `(ac_pub_all_fake, pi_fake, rho_fake, xij_fake)` to `p3`, `p4`.

All checks pass for all parties. `p0` and `p1` compute `Q_quorum`; `p3` and `p4` compute `Q_nonquorum ≠ Q_quorum`. The DKG completes with `SUCCESS` on all five parties but with inconsistent public keys. [9](#0-8)

template <typename... ARGS>
inline error_t committed_group_broadcast(job_mp_t& job, ARGS&... args) {
  error_t rv = UNINITIALIZED_ERROR;

  const crypto::mpc_pid_t& pid = job.get_pid();
  coinbase::crypto::commitment_t com(pid);
  // effectively, it does com.gen(args.msg...)
  std::apply([&com](auto&&... args) { com.gen(args...); },
             map_args_to_tuple([](auto& arg) { return arg.msg; }, args...));

  auto com_rand = job.uniform_msg<buf256_t>(com.rand);
  auto com_msg = job.uniform_msg<buf_t>(com.msg);

  if (rv = job.plain_broadcast(com_msg)) return rv;

  auto v = job.uniform_msg<buf256_t>(crypto::sha256_t::hash(com_msg.all_received()));

  if (rv = job.plain_broadcast(v, com_rand, args...)) return rv;

  for (int i = 0; i < job.get_n_parties(); i++) {
    if (v.msg != v.received(i)) return rv = job.mpc_abort(E_CRYPTO, "received hash mismatch");

    coinbase::crypto::commitment_t com(job.get_pid(i));
    com.set(com_rand.received(i), com_msg.received(i));
    rv = std::apply([&com](auto&&... args) { return com.open(args...); },
                    map_args_to_tuple([i](auto& arg) { return arg.received(i); }, args...));
    if (rv) return rv;
  }

  return SUCCESS;
}
