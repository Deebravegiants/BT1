### Title
Block-Level Reference Script Size Check Omits Sub-Transaction Scripts in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, the per-transaction reference script size check correctly uses `batchNonDistinctRefScriptsSize` to sum reference scripts across both the top-level transaction and all its sub-transactions. However, the block-level check reuses the Conway implementation (`totalRefScriptSizeInBlock`), which calls `txNonDistinctRefScriptsSize` and counts only top-level transaction reference scripts, silently ignoring sub-transaction reference scripts entirely. An unprivileged attacker can craft batched transactions whose sub-transactions collectively carry reference script data far exceeding `maxRefScriptSizePerBlock`, forcing every validating node to deserialize and process more script bytes per block than the protocol parameter is designed to allow.

---

### Finding Description

**Dijkstra per-transaction check** (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`, lines 313–329):

```haskell
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $
        DijkstraTxRefScriptsSizeTooBig ...
``` [1](#0-0) 

`batchNonDistinctRefScriptsSize` sums the top-level transaction's reference scripts **plus** every sub-transaction's reference scripts:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum (foldMap' (Sum . txNonDistinctRefScriptsSize utxo)
                       (tx ^. bodyTxL . subTransactionsTxBodyL))
``` [2](#0-1) 

**Dijkstra block-level check** (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`, line 363):

```haskell
Conway.validateBodyRefScriptsSizeTooBig @era pp blockBody (ls ^. utxoL)
``` [3](#0-2) 

This delegates to the Conway implementation, which calls `totalRefScriptSizeInBlock`:

```haskell
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      ...
      in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
``` [4](#0-3) 

`txNonDistinctRefScriptsSize` only inspects the top-level transaction body's inputs and reference inputs — it has no knowledge of sub-transactions:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [5](#0-4) 

The result is a structural asymmetry: the per-transaction gate counts sub-transaction reference scripts; the block gate does not.

---

### Impact Explanation

The `maxRefScriptSizePerBlock` parameter (default `1 MiB`) was introduced specifically to cap the total deserialization cost imposed on nodes per block, after a real DDoS attack on June 25, 2024 exploited the absence of such a limit. [6](#0-5) 

With the Dijkstra block-level check blind to sub-transaction reference scripts, an attacker can submit a block containing `k` top-level transactions each carrying sub-transactions whose combined reference script size approaches `maxRefScriptSizePerTx` (200 KiB). The block-level check observes 0 bytes from those sub-transactions and passes, while nodes must actually deserialize up to `k × 200 KiB` of script bytes. With `maxRefScriptSizePerBlock = 1 MiB`, the effective bypass factor is unbounded by the block check alone (bounded only by block size and per-tx fee cost). This matches the allowed impact: **attacker-controlled transactions exceed intended validation limits** (Medium).

---

### Likelihood Explanation

- The attacker is an ordinary transaction submitter — no privileged access is required.
- The attacker must first create UTxO entries containing large reference scripts (a normal, fee-paying operation).
- They then craft top-level Dijkstra transactions whose sub-transactions reference those UTxO entries.
- The per-transaction check passes because `batchNonDistinctRefScriptsSize` ≤ `maxRefScriptSizePerTx`.
- The block-level check passes because `totalRefScriptSizeInBlock` sees only the top-level inputs (which may carry zero reference scripts).
- The attack is repeatable every block at the cost of normal transaction fees, making it economically viable for sustained resource pressure.

---

### Recommendation

Replace the call to `Conway.validateBodyRefScriptsSizeTooBig` in `dijkstraBbodyTransition` with a Dijkstra-specific block-level check that uses `batchNonDistinctRefScriptsSize` (or an equivalent block-level accumulator) so that sub-transaction reference scripts are included in the per-block total. Concretely, `totalRefScriptSizeInBlock` should be extended or overridden for Dijkstra to call `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize` when accumulating the block total. [7](#0-6) 

---

### Proof of Concept

1. Deploy N UTxO entries each containing a large Plutus reference script (e.g., 200 KiB each).
2. Construct a Dijkstra top-level transaction with:
   - Zero top-level reference inputs (so `txNonDistinctRefScriptsSize` = 0).
   - One sub-transaction whose `referenceInputsTxBodyL` points to the 200 KiB script UTxO.
   - `batchNonDistinctRefScriptsSize` = 0 + 200 KiB = 200 KiB ≤ `maxRefScriptSizePerTx` → per-tx check passes.
3. Pack 10 such transactions into one block.
   - `totalRefScriptSizeInBlock` sums only top-level inputs: 10 × 0 = 0 bytes ≤ `maxRefScriptSizePerBlock` (1 MiB) → block check passes.
   - Actual reference script bytes deserialized by nodes: 10 × 200 KiB = 2 MiB — twice the intended block limit.
4. Repeat every block to sustain elevated deserialization load on all validating nodes. [4](#0-3) [2](#0-1)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability.
