Audit Report

## Title
Unrestricted `sendFunds()` in `FeeReceiver` Enables Stale-Price Deposit to Steal Unclaimed Yield - (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` carries no access-control modifier, allowing any caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at will. Because `LRTOracle.updateRSETHPrice()` is also publicly callable and reverts for non-managers when the resulting price increase exceeds `pricePercentageLimit`, an attacker can force the TVL spike, cause the oracle update to revert, and then deposit at the stale (under-valued) `rsETHPrice` — minting excess rsETH at the expense of all existing holders.

## Finding Description
`FeeReceiver.sendFunds()` is declared `external` with no role check: [1](#0-0) 

It pushes the full ETH balance into `LRTDepositPool` via the equally unrestricted `receiveFromRewardReceiver()`: [2](#0-1) 

Once the ETH lands in the pool, `getETHDistributionData()` counts it immediately as `ethLyingInDepositPool`: [3](#0-2) 

This feeds `_getTotalEthInProtocol()`, which drives `newRsETHPrice` in `_updateRsETHPrice()`. The public `updateRSETHPrice()` entry point: [4](#0-3) 

…calls `_updateRsETHPrice()`, which checks whether the price increase exceeds `pricePercentageLimit` and reverts for non-managers: [5](#0-4) 

The revert unwinds the entire call, so `rsETHPrice` (written only at line 313) is **never updated** — it remains at the old, lower value. Deposits then compute rsETH to mint using this stale denominator: [6](#0-5) 

A lower `rsETHPrice` denominator means the depositor receives more rsETH per ETH than the true exchange rate warrants. When the manager later calls `updateRSETHPriceAsManager()` to push the price to its correct level, the attacker's inflated rsETH balance appreciates at the expense of all pre-existing holders.

## Impact Explanation
**High — Theft of unclaimed yield.**

Existing rsETH holders are entitled to the price appreciation that accumulated MEV rewards represent. By depositing at a stale price after forcing the TVL spike, the attacker captures a disproportionate share of that appreciation. The attacker's rsETH balance is inflated relative to the true TVL, permanently diluting every existing holder's claim on the underlying assets. This matches the allowed impact class "Theft of unclaimed yield."

## Likelihood Explanation
**Medium.**

- `pricePercentageLimit` is explicitly configurable via `setPricePercentageLimit()` and the guard code is written to enforce it, making a non-zero value likely in production.
- MEV/execution-layer rewards accumulate continuously in `FeeReceiver`; over days or weeks the balance can be large enough to push a single-block TVL spike past the configured threshold.
- The attack requires only two public calls (`sendFunds()` + `updateRSETHPrice()`) and a subsequent deposit — no special privileges, no flash loan, no front-running of a specific transaction.
- The attack is repeatable each time enough rewards accumulate.

## Recommendation
Restrict `sendFunds()` to an authorized role:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Optionally, also restrict `receiveFromRewardReceiver()` in `LRTDepositPool` to only accept calls from the registered `FeeReceiver` address, preventing direct ETH injection from arbitrary callers. [2](#0-1) 

## Proof of Concept
```
Setup:
  FeeReceiver holds 500 ETH in accumulated MEV rewards.
  rsETHPrice = highestRsethPrice = 1.05e18.
  pricePercentageLimit = 1e16 (1%).
  rsETH totalSupply = S.

Step 1 — Attacker calls FeeReceiver.sendFunds():
  → 500 ETH transferred to LRTDepositPool via receiveFromRewardReceiver().
  → LRTDepositPool.address(this).balance increases by 500 ETH.
  → getETHDistributionData() now returns ethLyingInDepositPool += 500 ETH.

Step 2 — Attacker calls LRTOracle.updateRSETHPrice():
  → _getTotalEthInProtocol() returns ~500 ETH more than before.
  → newRsETHPrice = (totalETH) / S >> 1.05e18 * 1.01.
  → isPriceIncreaseOffLimit = true.
  → msg.sender is not MANAGER → revert PriceAboveDailyThreshold().
  → rsETHPrice remains at 1.05e18 (stale).

Step 3 — Attacker calls LRTDepositPool.depositETH{ value: X }(minRSETH, ""):
  → getRsETHAmountToMint uses rsETHPrice = 1.05e18 (stale).
  → rsethAmountToMint = X * 1e18 / 1.05e18 (inflated vs. true price).
  → Attacker receives excess rsETH representing the 500 ETH reward bump.

Step 4 — Manager calls LRTOracle.updateRSETHPriceAsManager():
  → rsETHPrice updates to the true higher value.
  → Attacker's rsETH is now worth more than X ETH deposited,
    at the expense of all pre-existing rsETH holders.

Foundry test outline:
  1. Fork mainnet or deploy local fixture with FeeReceiver + LRTDepositPool + LRTOracle.
  2. Deal 500 ETH to FeeReceiver.
  3. Record existing holder's rsETH balance and rsETHPrice.
  4. Prank attacker: call sendFunds(), then updateRSETHPrice() (expect revert).
  5. Prank attacker: depositETH with value X, record rsethReceived.
  6. Prank manager: call updateRSETHPriceAsManager().
  7. Assert attacker's rsETH * newRsETHPrice > X (profit confirmed).
  8. Assert existing holder's share of TVL decreased (dilution confirmed).
```
