Audit Report

## Title
Floor Division in `removeLiquidity` Burns LP Shares and Reduces Bin Balances While Returning Zero Native Tokens - (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`LiquidityLib.removeLiquidity` performs two sequential floor divisions: first converting shares to scaled amounts, then converting scaled amounts to native token units. For tokens with large `token0ScaleMultiplier` (e.g., USDC with 6 decimals → multiplier = 10¹²), the scaled balance removed from the bin can be non-zero while the native transfer rounds to zero. The LP's shares are burned and `binTotals.scaledToken0` is permanently decremented, but no tokens are transferred. The stranded scaled balance becomes an untracked surplus that is swept to protocol/admin fee collectors on the next `collectFees` call.

## Finding Description

**Step 1 — share-to-scaled floor division** (`LiquidityLib.sol` lines 205–206):

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

The bin's `token0BalanceScaled` and `token1BalanceScaled` are immediately decremented by these floored values (lines 210–213), and `positionBinShares[posKey]` is set to `newUserShares` (line 214). The share burn is unconditional.

**Step 2 — scaled-to-native floor division** (`LiquidityLib.sol` lines 239–240, `_deltasScaledToExternal` lines 274–276):

```solidity
(amount0Removed, amount1Removed) =
    _deltasScaledToExternal(totalToken0ToRemoveScaled, totalToken1ToRemoveScaled, ctx, Math.Rounding.Floor);
// ...
deltaAmount0 = scaledDeltaAmount0 / ctx.token0ScaleMultiplier;
```

`binTotals.scaledToken0` is decremented unconditionally at line 232 before this conversion. If `totalToken0ToRemoveScaled < token0ScaleMultiplier`, `amount0Removed = 0` and the `safeTransfer` at line 242–244 is skipped entirely. The pool's `binTotals.scaledToken0` was already decremented, so the scaled units are permanently orphaned — they no longer appear in bin accounting but remain in the ERC-20 balance.

**Surplus flows to fee collectors** (`MetricOmmPool.sol` lines 385–388):

```solidity
uint256 surplus0Scaled =
    balance0() * TOKEN_0_SCALE_MULTIPLIER - uint256(binTotals.scaledToken0) - notionalFee0AmountScaled;
```

Since `binTotals.scaledToken0` was decremented but no ERC-20 tokens left the pool, `balance0() * TOKEN_0_SCALE_MULTIPLIER` remains unchanged while `binTotals.scaledToken0` is lower, inflating `surplus0Scaled`. This surplus is then distributed to admin and protocol fee collectors.

**Existing guards are insufficient**: The `MINIMAL_MINTABLE_LIQUIDITY` guard at line 200 only prevents leaving a dust *remainder* of shares; it does not prevent removing a quantity of shares whose total scaled value is below `token0ScaleMultiplier`. The asymmetric rounding (`Math.Rounding.Ceil` on `addLiquidity` at line 142, `Math.Rounding.Floor` on `removeLiquidity` at line 240) guarantees the LP always pays at least 1 native unit when adding but may receive 0 when removing.

## Impact Explanation

Direct, unrecoverable loss of LP principal. An LP burns valid shares and has their bin balance permanently reduced, but receives zero native tokens. The lost value is silently redirected to fee collectors with no revert or warning. This constitutes a broken core pool invariant: LP claims are not fully covered by the pool's token balances after `removeLiquidity`. Severity: **High**.

## Likelihood Explanation

- Any LP interacting with a pool whose token has fewer than 18 decimals (USDC, USDT, WBTC) is exposed.
- No privileged access is required; any LP triggers this by calling `removeLiquidity` with a small share count.
- The asymmetric rounding guarantees the LP always pays at least 1 native unit when adding but may receive 0 when removing, making a 100% loss of a small deposit reproducible.
- The condition `totalToken0ToRemoveScaled < token0ScaleMultiplier` is easily reachable for USDC pools (multiplier = 10¹²) with small positions.

## Recommendation

1. **Revert on zero-output removal**: After computing `amount0Removed` and `amount1Removed`, revert if either is zero while the corresponding scaled amount is non-zero:
   ```solidity
   if (totalToken0ToRemoveScaled > 0 && amount0Removed == 0) revert ZeroTokenOutput();
   if (totalToken1ToRemoveScaled > 0 && amount1Removed == 0) revert ZeroTokenOutput();
   ```
2. **Alternatively**, expose `minAmount0Out` / `minAmount1Out` parameters so callers can enforce a minimum and the revert is user-controlled.
3. **For the share-to-scaled step**, consider using `Math.ceilDiv` symmetrically with `addLiquidity` so the LP always recovers at least 1 scaled unit per share removed.

## Proof of Concept

**Setup**: USDC/ETH pool, `token0 = USDC` (6 decimals), `token0ScaleMultiplier = 10¹²`, `initialScaledToken0PerShareE18 = 1e18`, `MINIMAL_MINTABLE_LIQUIDITY = 1000`.

**Add liquidity** (1000 shares, empty bin):
```
amount0Scaled = ceilDiv(1e18 × 1000, 1e18) = 1000   // scaled units added to bin
amount0Added  = ceilDiv(1000, 1e12)         = 1       // 1 native unit (0.000001 USDC) paid by LP
```

**Remove liquidity** (all 1000 shares):
```
amount0Scaled          = 1000 × 1000 / 1000 = 1000   // scaled units, bin balance decremented
binTotals.scaledToken0 decremented by 1000            // accounting permanently reduced
amount0Removed         = 1000 / 1e12        = 0       // 0 native units transferred to LP
```

LP paid 1 native unit, received 0. The 1000 scaled units are orphaned in the pool's ERC-20 balance and will be swept to fee collectors on the next `collectFees` call via the surplus calculation at `MetricOmmPool.sol` lines 385–388.

A Foundry test can reproduce this by deploying a USDC/ETH pool, calling `addLiquidity` with 1000 shares, then `removeLiquidity` with 1000 shares, and asserting that `amount0Removed == 0` while the LP's USDC balance is unchanged and `collectFees` subsequently transfers the orphaned value to the fee destination.
