Audit Report

## Title
Stale Cross-Chain Oracle Rate Allows Excess wrsETH Minting on L2 - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored rsETH/ETH rate with no staleness check despite tracking `lastUpdated`. Because rsETH is yield-bearing and its rate monotonically increases, a stale (lower-than-current) L2 rate causes all L2 pool contracts to mint more wrsETH per deposited ETH than the current L1 rate justifies. An unprivileged attacker can exploit this to extract yield from existing rsETH holders.

## Finding Description

`CrossChainRateReceiver` stores the rate received from L1 via LayerZero and tracks when it was last updated:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L13-16
uint256 public rate;
uint256 public lastUpdated;
```

`lzReceive` sets both fields on each L1 push:

```solidity
// L95-97
rate = _rate;
lastUpdated = block.timestamp;
```

But `getRate()` returns the stored rate unconditionally:

```solidity
// L102-105
function getRate() external view returns (uint256) {
    return rate;  // lastUpdated is never consulted
}
```

Every L2 pool calls this through `IOracle(rsETHOracle).getRate()` to compute the mint amount. In `RSETHPoolV3`:

```solidity
// contracts/pools/RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

The same pattern is present in `RSETHPoolNoWrapper` (L277-286). `deposit()` then mints directly from this calculation with no additional rate validation:

```solidity
// RSETHPoolV3.sol L258-264
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);
```

The `dailyMintLimit` modifier limits total daily minting volume but does not prevent the rate-based over-minting — it only caps the scale of a single day's exploitation, not the per-unit excess.

Rate updates are push-based: the L1 `RSETHRateProvider` must be actively called to send a LayerZero message. There is no on-chain enforcement of update frequency. During bridge downtime, network congestion, or simply infrequent pushes, `lastUpdated` can lag arbitrarily behind real time while `getRate()` continues returning the outdated lower rate.

## Impact Explanation

**High — Theft of unclaimed yield.**

When the L2 oracle is stale, `rsETHAmount = amountAfterFee * 1e18 / staleRate` produces more wrsETH than the deposited ETH is worth at the current L1 rate. The attacker bridges wrsETH to L1, unwraps to rsETH, and redeems via `LRTWithdrawalManager` for more ETH than deposited. The surplus is extracted from the protocol's TVL, diluting all existing rsETH holders' accrued yield. This is a direct, repeatable theft of yield belonging to other users, matching the "Theft of unclaimed yield" (High) impact class.

## Likelihood Explanation

LayerZero message delivery requires active rate pushes from L1; there is no heartbeat or on-chain enforcement of update frequency. Any period of bridge downtime, network congestion, or operator inactivity creates a stale oracle window. rsETH accrues approximately 4% APY, so a 7-day stale rate produces ~0.077% excess minting per deposit. This is exploitable by any unprivileged depositor with no capital risk beyond gas, and is repeatable for the entire duration the oracle remains stale. The attack requires no special privileges, no victim mistakes, and no external protocol compromise — only a normal `deposit()` call.

## Recommendation

Add a staleness threshold check inside `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

This causes all L2 pool deposits to revert when the oracle is stale, preventing excess minting until a fresh rate is received from L1. The threshold should be calibrated to the expected update frequency with sufficient margin.

## Proof of Concept

1. L1 rsETH/ETH rate is `1.05e18` (reflecting accumulated staking rewards).
2. L2 `CrossChainRateReceiver` holds stale rate `1.04e18` (`lastUpdated` is 7 days ago; `getRate()` never checks it).
3. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("ref")` on L2.
4. `viewSwapRsETHAmountAndFee(100 ether)` computes: `rsETHAmount = 100e18 * 1e18 / 1.04e18 ≈ 96.153 wrsETH`.
5. Fair wrsETH at current L1 rate: `100e18 * 1e18 / 1.05e18 ≈ 95.238 wrsETH`.
6. Attacker received `~0.915 wrsETH` excess (~0.96 ETH of value at L1 rate).
7. Attacker bridges wrsETH to L1, unwraps, redeems via `LRTWithdrawalManager` for `~100.96 ETH`, netting `~0.96 ETH` profit at the expense of existing rsETH holders.
8. Repeatable for the full duration the oracle remains stale.

**Foundry fork test plan**: Fork an L2 where `RSETHRateReceiver` is deployed. Warp `block.timestamp` forward by 7 days without triggering `lzReceive`. Call `deposit{value: 100 ether}` and assert the minted wrsETH exceeds `100e18 * 1e18 / currentL1Rate`, confirming the over-mint. Compare against a baseline where `lzReceive` is called with the current rate before deposit.
