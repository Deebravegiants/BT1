### Title
`respond_verify_foreign_tx` Accepts Attacker-Supplied `payload_hash` Without Binding It to the Submitted Request — (`crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` contract function verifies that a signature is valid over `response.payload_hash`, but never checks that `response.payload_hash` was actually derived from the `request` stored in the pending queue. A single Byzantine MPC participant (strictly below the signing threshold) can replay a legitimately-produced `(payload_hash, signature)` pair from one pending request onto a different pending request, causing the contract to resolve that second request with forged foreign-chain attestation data.

### Finding Description

In `crates/contract/src/lib.rs`, `respond_verify_foreign_tx` performs the following verification:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract only checks that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. It does **not** verify that `response.payload_hash` is the canonical hash of `ForeignTxSignPayload::V1 { request: <the stored request>, values: <...> }`.

The signed payload is defined as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

Because `request` is embedded inside the hash, a `payload_hash` produced for `request_B` is cryptographically distinct from one produced for `request_A`. The contract never enforces this binding.

Contrast this with the regular `respond` function, which derives the payload directly from the **stored** request, not from the response:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [3](#0-2) 

In `respond`, the payload comes from the stored request. In `respond_verify_foreign_tx`, the payload hash comes from the response argument — fully attacker-controlled.

The node-side signing code confirms that each node independently computes `payload_hash` from the foreign chain result and signs it:

```rust
let response_payload = self
    .execute_foreign_chain_request(
        &foreign_tx_request.request,
        foreign_tx_request.payload_version,
    )
    .await?;

let sign_request = build_signature_request(&foreign_tx_request, &response_payload)?;
``` [4](#0-3) 

After the threshold signing protocol completes, **all** participants know the resulting `(payload_hash, signature)` pair. Any single participant can then call `respond_verify_foreign_tx` on the contract with an arbitrary `request` argument while supplying the legitimately-produced signature.

### Impact Explanation

**High — Forged foreign-chain verification / invalid bridge execution.**

Attack scenario:

1. Two requests are pending: `request_A` (e.g., verify a valid Bitcoin deposit tx) and `request_B` (e.g., verify a different, possibly invalid or already-used tx).
2. The MPC network legitimately processes `request_B` and all participants learn `(payload_hash_B, signature_B)`.
3. A single malicious participant calls `respond_verify_foreign_tx(request_A, { payload_hash_B, signature_B })`.
4. The contract checks: does `request_A` exist in `pending_verify_foreign_tx_requests`? Yes. Is `signature_B` valid over `payload_hash_B` under the root key? Yes. Both checks pass.
5. `request_A` is permanently resolved and removed from the pending queue with the forged response.
6. The caller of `request_A` receives `{ payload_hash_B, signature_B }`.

If the receiving contract does not independently verify that `payload_hash_B` encodes `request_A` (a check the SDK provides but does not enforce on-chain), it will accept a forged attestation. A bridge contract that mints tokens upon receiving a valid MPC attestation of a foreign deposit could be tricked into minting for a transaction it never actually verified.

Even if the caller does check the payload hash, `request_A` is permanently consumed from the queue — the user cannot resubmit and will never receive a valid response.

### Likelihood Explanation

- Requires exactly **one** malicious MPC participant, strictly below the signing threshold.
- No key forgery, no threshold collusion, no TEE bypass needed.
- The attacker only needs to observe the output of a legitimate signing session (which all participants see) and race to submit `respond_verify_foreign_tx` with a mismatched `request` argument before the honest leader does.
- Bridge contracts that do not use `ForeignChainSignatureVerifier::verify_signature` from the SDK are directly exploitable for invalid bridge execution.

### Recommendation

The contract should bind `response.payload_hash` to the stored `request` before accepting the response. The simplest fix is to require the responding node to supply the extracted `values` alongside the hash, then recompute and compare:

```rust
// Recompute the expected hash from the stored request + provided values
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(), // new field
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This mirrors the SDK-side `ForeignChainSignatureVerifier::verify_signature` logic: [5](#0-4) 

and enforces the binding on-chain rather than relying on each caller to implement it correctly.

### Proof of Concept

1. Submit two `verify_foreign_transaction` requests: `request_A` (Bitcoin tx `0xAA…`) and `request_B` (Bitcoin tx `0xBB…`), both with `confirmations = 1`.
2. The MPC network processes `request_B`; all participants observe `payload_hash_B = SHA256(borsh(V1 { request_B, [block_hash_BB] }))` and `signature_B`.
3. Malicious participant calls:
   ```
   respond_verify_foreign_tx(
       request = request_A,          // ← the victim's request
       response = {
           payload_hash: payload_hash_B,   // ← from request_B's signing session
           signature: signature_B,          // ← legitimately produced by the MPC network
       }
   )
   ```
4. The contract at lines 718–734 verifies `verify_ecdsa_signature(signature_B, payload_hash_B, root_key)` — this passes because the signature is genuine.
5. `resolve_yields_for` at line 749 resolves `request_A`'s pending yield with `response = { payload_hash_B, signature_B }` and removes it from the queue.
6. The caller of `request_A` receives a response whose `payload_hash` encodes `request_B`, not `request_A`. Any bridge contract that does not re-derive and compare the hash will accept this as a valid attestation of Bitcoin tx `0xAA…`. [6](#0-5)
