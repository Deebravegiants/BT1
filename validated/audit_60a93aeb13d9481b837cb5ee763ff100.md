### Title
Missing G2 Subgroup Membership Check in `alt_bn128_pairing_check` Allows Cofactor-Point Bypass — (`runtime/near-vm-runner/src/logic/alt_bn128.rs`)

### Summary

`decode_g2` delegates point validation entirely to `bn::AffineG2::new`, which only verifies that the supplied coordinates satisfy the twist-curve equation (curve membership). It performs **no prime-order subgroup check** (`r * Q == O`). An attacker can supply a cofactor point — a point on E'(Fq2) whose order divides the cofactor h₂ but not r — and the Ate pairing of any G1 point against it evaluates to 1 in GT, causing `pairing_check` to return `true` for an input that the spec explicitly requires to be rejected.

### Finding Description

The `bn` crate used is `zeropool-bn` v0.5.11 (`Cargo.toml` line 155):

```
bn = { package = "zeropool-bn", version = "0.5.11", default-features = false }
```

`decode_g2` in `alt_bn128.rs`:

```rust
fn decode_g2(raw: &[u8; 2 * POINT_SIZE]) -> Result<bn::G2, InvalidInput> {
    let (x, y) = stdx::split_array(raw);
    let x = decode_fq2(x)?;
    let y = decode_fq2(y)?;
    if x.is_zero() && y.is_zero() {
        Ok(bn::G2::zero())
    } else {
        bn::AffineG2::new(x, y)          // ← only checks y² = x³ + b/ξ
            .map_err(|_err| InvalidInput::new("invalid g2", raw))
            .map(bn::G2::from)
    }
}
```

`bn::AffineG2::new` in `zeropool-bn` checks only that the point satisfies the curve equation over Fq2. It does **not** multiply by r to verify the point is in the prime-order subgroup. [1](#0-0) 

The NEAR spec (doc-comment on `alt_bn128_pairing_check`) explicitly states:

> "If point coordinates are not on curve, **point is not in the subgroup**, scalar is not in the field or data are wrong serialized… the function returns `AltBn128InvalidInput`." [2](#0-1) 

The implementation never enforces the subgroup condition. Compare with the BLS12-381 pairing check, which explicitly calls `blst_p1_affine_in_g1` and `blst_p2_affine_in_g2` before proceeding: [3](#0-2) 

No analogous guard exists for alt_bn128.

### Impact Explanation

**Mathematical root cause.** The BN128 twist curve E'(Fq2) has order h₂ · r. A cofactor point Q satisfies h₂ · Q = O but r · Q ≠ O. For the Ate pairing:

```
e(P, Q)^h₂ = e(P, h₂·Q) = e(P, O) = 1  (bilinearity holds over the full group)
```

Since GT has order r and gcd(h₂, r) = 1, the only h₂-th root of unity in GT is 1, so **e(P, Q) = 1** for any P ∈ G1. Therefore `pairing_check` returns `true` (1) for the pair (G1_generator, cofactor_point), even though the cofactor point is not in G2.

**Practical impact.** Any on-chain smart contract that uses `alt_bn128_pairing_check` as its ZK proof verification primitive (Groth16 verifier, light-client bridge, epoch-proof verifier) can be attacked:

- For a single-pair check `e(A, B) == 1`: set B = cofactor_point → always passes.
- For multi-pair Groth16 checks: an attacker who controls proof elements can substitute cofactor points for G2 proof components, collapsing those pairing factors to 1 and potentially satisfying the verification equation with an invalid witness.

The outcome is that the contract accepts an invalid proof, producing incorrect state changes committed to the outcome root. [4](#0-3) 

### Likelihood Explanation

- The attack is fully unprivileged: any account can call a contract that invokes `alt_bn128_pairing_check`.
- Constructing a cofactor point for BN128 is straightforward: take any point T on E'(Fq2) and compute Q = h₂ · T. If Q ≠ O, it is a cofactor point.
- The cofactor h₂ for BN128 is publicly known.
- No special node access, validator key, or admin privilege is required.

### Recommendation

After `bn::AffineG2::new` succeeds, multiply the resulting point by the group order r and verify the result is the point at infinity:

```rust
let p = bn::AffineG2::new(x, y)
    .map_err(|_| InvalidInput::new("invalid g2", raw))?;
let g2 = bn::G2::from(p);
// Subgroup check: r * Q must be the identity
if g2 * bn::Fr::from_str("21888...r...").unwrap() != bn::G2::zero() {
    return Err(InvalidInput::new("g2 point not in prime-order subgroup", raw));
}
Ok(g2)
```

Alternatively, use a cofactor-clearing multiplication (multiply by h₂) and verify the result equals the original point (only valid for r-torsion points). This mirrors the explicit subgroup checks already present in the BLS12-381 path.

### Proof of Concept

1. Pick any non-identity point T on E'(Fq2) (e.g., the standard G2 generator).
2. Compute Q = h₂ · T (multiply by the BN128 twist cofactor). Q is on the curve but not in G2.
3. Encode Q as little-endian Fq2 bytes (two 256-bit field elements each for x and y).
4. Call `alt_bn128_pairing_check` with the pair (G1_generator_bytes ‖ Q_bytes).
5. Observe the returned `u64` is `1` (true), despite Q not being in the prime-order subgroup.

The call path is: `contract_call` → `alt_bn128_pairing_check` (logic.rs:1119) → `alt_bn128::pairing_check` (alt_bn128.rs:77) → `decode_g2` (alt_bn128.rs:131) → `bn::AffineG2::new` (no subgroup check) → `bn::pairing_batch` → returns `Gt::one()` → `pairing_check` returns `true`. [5](#0-4) [1](#0-0)

pub(crate) fn pairing_check(
    elements: &[[u8; PAIRING_CHECK_ELEMENT_SIZE]],
) -> Result<bool, InvalidInput> {
    let elements: Vec<(bn::G1, bn::G2)> = elements
        .iter()
        .map(|chunk| {
            let (g1, g2) = stdx::split_array(chunk);
            let g1 = decode_g1(g1)?;
            let g2 = decode_g2(g2)?;
            Ok((g1, g2))
        })
        .collect::<Result<Vec<_>, InvalidInput>>()?;

    let res = bn::pairing_batch(&elements) == bn::Gt::one();

    Ok(res)
}
