### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Attested Participant — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the domain's root public key, but **never checks that `response.payload_hash` was actually derived from the `request` parameter supplied in the same call**. A single attested MPC participant (strictly below the signing threshold) can reuse a valid `(payload_hash, signature)` pair obtained from one legitimately-processed foreign-chain verification request to resolve a completely different pending request, delivering fabricated verification data to the waiting caller.

---

### Finding Description

The `respond_verify_foreign_tx` entry point in `crates/contract/src/lib.rs` performs the following steps:

1. Authenticates the caller as an attested participant.
2. Looks up the root public key for `request.domain_id`.
3. Verifies `response.signature` over `response.payload_hash` with that root key.
4. Calls `pending_requests::resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &request, ...)` — resolving the yield keyed on `request` with the full `response` bytes. [1](#0-0) 

The signature check at lines 726–734 only proves that *some* MPC-network-produced signature covers *some* 32-byte hash. It does **not** prove that the hash was computed from the `ForeignTxSignPayload` that corresponds to the `request` argument. The contract has no access to the `values: Vec<ExtractedValue>` that the nodes fetched from the foreign chain, so it cannot recompute the expected hash itself.

The canonical hash the nodes are supposed to sign is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

Because `ForeignTxSignPayloadV1` contains `request` and `values`, two requests with different `ForeignChainRpcRequest` fields produce different `payload_hash` values. The contract, however, accepts any `payload_hash` for which a valid root-key signature exists, regardless of which request it was originally produced for.

The node-side signing path confirms the zero-tweak design (root key, no derivation): [3](#0-2) 

---

### Impact Explanation

A malicious attested participant can:

1. Submit a legitimate `verify_foreign_transaction` call for **Request A** (e.g., Bitcoin `tx_id = [0xAA;32]`).
2. Wait for the honest MPC network to collectively sign and produce `(payload_hash_A, sig_A)` — a valid root-key signature over the hash of `{request_A, values_A}`.
3. Submit (or observe) a second pending **Request B** (e.g., Bitcoin `tx_id = [0xBB;32]`) from any user.
4. Call `respond_verify_foreign_tx(request = B, response = {payload_hash: payload_hash_A, signature: sig_A})`.
5. The contract verifies `sig_A` over `payload_hash_A` with the root key → **passes**.
6. The contract resolves Request B's yield with `(payload_hash_A, sig_A)`.

The caller of Request B receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes the extracted values of **Request A's transaction**, not their own. Any downstream bridge or smart contract that trusts this response without independently recomputing the expected hash will act on fabricated foreign-chain state — enabling invalid bridge execution or double-spend conditions.

This matches the **High** allowed impact: *"forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- **Attacker requirement**: a single attested MPC participant. No threshold collusion is needed; the attacker does not forge a signature — they reuse one the honest network already produced.
- **Opportunity**: any time two different `verify_foreign_transaction` requests are pending simultaneously (a normal production condition), the attack window is open.
- **Profit motive**: a bridge contract that pays out based on a verified foreign-chain deposit can be made to pay out for a transaction that was never confirmed, or for a different transaction than the one the user submitted.

---

### Recommendation

Inside `respond_verify_foreign_tx`, after verifying the signature, recompute the minimum binding check the contract *can* perform: require the caller to supply the full `ForeignTxSignPayload` (not just its hash), verify `payload_hash == payload.compute_msg_hash()`, and verify `payload.request() == request.request`. This forces the responder to commit to the actual extracted values and binds the signature to the specific pending request.

Concretely, change the response type to include the full payload:

```rust
pub struct VerifyForeignTransactionResponse {
    pub payload: ForeignTxSignPayload,   // full payload, not just hash
    pub signature: SignatureResponse,
}
```

Then in `respond_verify_foreign_tx`:

```rust
// 1. Recompute hash from the supplied payload
let payload_hash = response.payload.compute_msg_hash()?;

// 2. Verify the payload is bound to the request
ensure!(response.payload.request() == &request.request, ...);

// 3. Verify signature over the recomputed hash
verify_ecdsa_signature(&sig, &payload_hash.0, &secp_pk)?;
```

This closes the replay window without requiring threshold changes.

### Proof of Concept

```
// Setup: two pending requests for different Bitcoin tx_ids, same domain.
let request_A = VerifyForeignTransactionRequest { request: Bitcoin(tx_id=[0xAA;32]), ... };
let request_B = VerifyForeignTransactionRequest { request: Bitcoin(tx_id=[0xBB;32]), ... };

// Step 1: honest network processes request_A, attacker observes the response.
let (payload_hash_A, sig_A) = honest_mpc_response_for(request_A);

// Step 2: attacker (single attested participant) calls respond for request_B
//         but supplies the response from request_A.
contract.respond_verify_foreign_tx(
    request_B,
    VerifyForeignTransactionResponse { payload_hash: payload_hash_A, signature: sig_A }
);
// → succeeds: sig_A is valid over payload_hash_A under the root key.
// → request_B's yield is resolved with payload_hash_A (wrong foreign-chain data).
// → caller of request_B receives fabricated verification of tx_id=[0xBB;32].

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
