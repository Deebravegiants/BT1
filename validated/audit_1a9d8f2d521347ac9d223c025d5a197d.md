### Title
Fee-on-Transfer ERC-20 Accounting Divergence Enables Unbacked NEAR Credit and EVM Fund Lock or Drain ŌĆö (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` records the caller-supplied `amount` in the `InitTransfer` event without measuring the actual tokens received. For fee-on-transfer ERC-20 tokens the bridge holds `amount ŌłÆ fee_taken` but NEAR credits the full `amount`, creating unbacked wrapped supply. On the return path `finTransfer` attempts to release the full credited amount, which either reverts (permanently locking user funds) or silently drains other depositors' locked tokens.

---

### Finding Description

In `initTransfer`, the lock-path for a plain ERC-20 (neither a bridge token nor a custom-minter token) is:

```solidity
// OmniBridge.sol lines 406-411
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ŌåÉ caller-supplied, not balance-checked
    );
}
```

Immediately after, the function emits:

```solidity
// OmniBridge.sol lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,             // ŌåÉ same caller-supplied value, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [1](#0-0) [2](#0-1) 

No balance snapshot is taken before or after the `safeTransferFrom` call. For a fee-on-transfer token the bridge receives `amount ŌłÆ token_fee` but the event records `amount`.

The NEAR bridge's `fin_transfer_callback` decodes this event and constructs a `TransferMessage` using `init_transfer.amount` verbatim:

```rust
// near/omni-bridge/src/lib.rs lines 726-729
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [3](#0-2) 

NEAR therefore credits the user with the full `amount`, not the actual received amount. The discrepancy is now baked into the pending transfer record.

When the user later bridges back, `finTransfer` on EVM attempts:

```solidity
// OmniBridge.sol lines 350-354
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount   // ŌåÉ full credited amount, bridge holds less
    );
}
``` [4](#0-3) 

The bridge's actual balance for that token is `amount ŌłÆ token_fee`, so the release either:
- **Reverts** ŌĆö permanently locking the user's funds in the NEAR pending transfer, or
- **Succeeds by consuming other depositors' tokens** ŌĆö draining the shared lock pool.

---

### Impact Explanation

Two distinct impacts, both in-scope:

1. **Critical ŌĆö Irreversible fund lock.** If the bridge's token balance is exactly what was deposited (single depositor scenario), `safeTransfer` reverts. The NEAR transfer record is already finalised; there is no rollback path. User funds are permanently unclaimable.

2. **Critical ŌĆö Unauthorized release of locked assets.** If other users have deposited the same fee-on-transfer token, the bridge holds a surplus that masks the shortfall. `finTransfer` succeeds, silently transferring tokens that belong to other depositors. Repeated exploitation drains the entire lock pool, breaking the 1:1 backing guarantee for all holders of the wrapped NEAR token.

---

### Likelihood Explanation

- No token allowlist exists in `initTransfer`; any ERC-20 address is accepted.
- Fee-on-transfer tokens (e.g., STA, PAXG, tokens with configurable fees) are deployed on mainnet and are publicly known.
- The attacker needs only to call `initTransfer` with such a token ŌĆö a fully unprivileged, public entrypoint.
- The discrepancy grows linearly with each deposit, making repeated exploitation straightforward.

---

### Recommendation

1. **Measure actual received amount** using a balance snapshot pattern in `initTransfer`:
   ```solidity
   uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
   IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
   uint128 actualReceived = uint128(
       IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore
   );
   // use actualReceived in place of amount for the event and extension call
   ```
2. **Emit the actual received amount** in `InitTransfer` so NEAR credits only what the bridge holds.
3. **Implement a token allowlist** (analogous to the external report's recommendation) so only audited, non-rebasing, non-fee tokens can be locked via the canonical bridge.

---

### Proof of Concept

1. Deploy a fee-on-transfer ERC-20 `FeeToken` with a 10% transfer fee on mainnet.
2. Call `OmniBridge.initTransfer(FeeToken, 1000, 0, 0, "near:alice.near", "")`.
   - Bridge receives `900` tokens; event emits `amount = 1000`.
3. NEAR relayer calls `fin_transfer` with the proof; NEAR credits `alice.near` with `1000` wrapped `FeeToken`.
4. `alice.near` calls NEAR bridge to initiate withdrawal of `1000` back to EVM.
5. NEAR `sign_transfer` produces a signed payload for `amount = 1000`.
6. Attacker calls `OmniBridge.finTransfer(sig, payload)`.
   - Bridge attempts `safeTransfer(alice_evm, 1000)` but holds only `900`.
   - **If no other depositors**: reverts ŌåÆ `alice.near`'s `1000` wrapped tokens are permanently locked on NEAR with no redemption path.
   - **If other depositors exist**: succeeds, consuming `100` tokens belonging to other users ŌåÆ backing deficit accumulates.
