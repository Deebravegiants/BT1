### Title
Retroactive `accept_requests = false` Permanently Blocks In-Flight Yield-Resume Sign Requests From Being Fulfilled — (File: `crates/contract/src/lib.rs`)

### Summary

When `verify_tee()` determines that fewer than `threshold` participants hold valid TEE attestations, it sets `accept_requests = false`. The `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` methods all gate on this flag **after** the yield-resume promise has already been created and stored. Sign requests that were submitted and queued while the contract was healthy are retroactively blocked from ever being fulfilled, mirroring the vlPenpie pattern where a condition introduced after a user action prevents that action from completing.

### Finding Description

`verify_tee()` sets `accept_requests = false` when TEE validation fails: [1](#0-0) 

Every `respond*` method then checks this flag and returns an error before it can call `resolve_yields_for`: [2](#0-1) [3](#0-2) 

The yield-resume entries for those requests are stored in `pending_signature_requests`, `pending_ckd_requests`, and `pending_verify_foreign_tx_requests`: [4](#0-3) 

`push_pending_yield` appended the `YieldIndex` at request-submission time, before any TEE state change: [5](#0-4) 

There is no code path that resolves or cancels these stored yields when `accept_requests` transitions to `false`. The only exit is the NEAR runtime's yield-resume timeout, which fires `fail_on_timeout` — but that callback is not guaranteed to execute promptly, and during the entire window the MPC nodes are unable to call `respond*` for any queued request regardless of when it was submitted.

### Impact Explanation

Every pending sign/CKD/foreign-tx request that was legitimately queued before the TEE failure is frozen. MPC nodes cannot call `respond*` to deliver the computed signature; the request-lifecycle invariant (a submitted request must eventually be fulfilled or explicitly rejected) is broken. For time-sensitive cross-chain operations this means the foreign-chain transaction the user was trying to authorize is left in an ambiguous state until the NEAR yield timeout fires. This matches the **Medium** allowed impact: *"request-lifecycle … manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

`verify_tee()` is callable by any single participant (`voter_or_panic()` — below threshold). TEE attestations expire on a 7-day rolling window (`tee_upgrade_deadline_duration_seconds = 604800`). Any routine upgrade cycle where one or more nodes temporarily lose valid attestation status can trigger `accept_requests = false`. Because sign requests are continuously submitted by users, there will almost always be in-flight yield-resume entries at the moment the flag flips. [6](#0-5) [7](#0-6) 

### Recommendation

Apply the `accept_requests` gate only to **new** incoming requests (in `check_request_preconditions`), not to `respond*` callbacks that are fulfilling requests already in the queue. Alternatively, when `accept_requests` transitions to `false`, immediately drain all stored `YieldIndex` entries by calling `promise_yield_resume` with an explicit failure payload so callers receive a deterministic error rather than waiting for the runtime timeout.

### Proof of Concept

1. User calls `sign(payload, domain_id)` at block N. Contract is healthy (`accept_requests = true`). A `YieldIndex` is stored in `pending_signature_requests`.
2. At block N+1, a participant calls `verify_tee()`. One node's attestation has expired. `remaining < threshold` → `accept_requests = false`.
3. MPC nodes compute the signature off-chain and call `respond(request, signature)`.
4. `respond()` hits the guard at line 579–581 and returns `Err(TeeError::TeeValidationFailed)`. The yield is never resumed.
5. The user's cross-chain transaction is left unresolved until the NEAR yield-resume timeout fires `fail_on_timeout`, which may be many blocks later. During this window the user cannot determine whether to retry on the foreign chain, creating a double-spend risk. [8](#0-7) [9](#0-8)
