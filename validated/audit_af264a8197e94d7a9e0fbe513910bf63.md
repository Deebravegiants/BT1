Audit Report

## Title
Wrong Inequality in ETH Deposit Limit Check Allows Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit()` omits the incoming `amount` from the ETH branch comparison, evaluating `totalAssetDeposits > depositLimit` instead of `totalAssetDeposits + amount > depositLimit`. This allows ETH deposits to proceed even when the cap is fully consumed or when a single deposit would push total ETH holdings arbitrarily above the configured limit, while the equivalent LST path correctly blocks such deposits.

## Finding Description
In `contracts/LRTDepositPool.sol` at L676–682, the function branches on `asset == LRTConstants.ETH_TOKEN`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // `amount` absent
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch never incorporates `amount`. `depositETH()` (L76–93) calls `_beforeDeposit()` (L648–670), which calls this function and reverts with `MaximumDepositLimitReached` only when it returns `true`. Because the ETH branch ignores `amount`, it returns `false` whenever `totalAssetDeposits <= depositLimit`, regardless of how large `amount` is. No other guard in the ETH deposit path compensates for this omission.

## Impact Explanation
The deposit limit is a protocol-level safety invariant bounding EigenLayer strategy exposure. Any ETH depositor can bypass it, minting rsETH and pushing total ETH deposits above the cap by an arbitrary margin. No direct fund theft or loss of value occurs from a single transaction; the protocol simply accepts more ETH than the cap was designed to allow. This matches the allowed Low impact: **Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
No special role, front-running, or external dependency is required. Any unprivileged user calling `depositETH{value: X}()` when `totalAssetDeposits >= depositLimit - X` (including when the cap is exactly reached) triggers the bypass. The condition is reachable in normal protocol operation and is repeatable indefinitely.

## Recommendation
Mirror the LST branch in the ETH path by adding `amount` to the comparison:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Protocol accumulates `totalAssetDeposits(ETH) = 1000 ether` (cap exactly reached).
3. `getAssetCurrentLimit(ETH)` returns `0`.
4. Attacker calls `depositETH{value: 500 ether}(0, "")`.
5. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `1000 ether > 1000 ether` → `false` → no revert.
6. 500 ETH is accepted and rsETH is minted; total ETH deposits become 1500 ether — 50% above cap.
7. Equivalent LST call would evaluate `1000 ether + 500 ether > 1000 ether` → `true` → reverts with `MaximumDepositLimitReached`.

**Foundry test plan:** Deploy `LRTDepositPool` on a local fork, set `depositLimitByAsset(ETH_TOKEN) = 1000 ether`, simulate deposits totalling 1000 ether, then call `depositETH{value: 1}()` and assert it does not revert (demonstrating the bypass). Confirm the same call with an LST reverts.
