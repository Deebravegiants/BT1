### Title
Stale `drepDelegs` Reverse-Index Causes `ConwayUnRegDRep` to Permanently Clear Delegation of Wrong Accounts - (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version < 10), when a stake credential redelegates its vote from DRep A to DRep B, the old DRep A's reverse-delegation set (`drepDelegs`) is not cleaned up. When DRep A subsequently unregisters, the `ConwayUnRegDRep` transition uses this stale `drepDelegs` set to clear account delegations, incorrectly setting `dRepDelegationAccountStateL = Nothing` for credentials that have already moved to DRep B. The hard-fork migration to PV10 (`updateDRepDelegations`) does not restore these cleared delegations, making the loss permanent. Affected credentials can no longer withdraw rewards in Conway (which requires an active DRep delegation), permanently freezing their reward balances.

---

### Finding Description

**Root cause — stale reverse-delegation during bootstrap (`Deleg.hs`)**

In `processDelegationInternal`, when `preserveIncorrectDelegation = True` (i.e., `pvMajor pv < natVersion @10`) and the new delegatee is a credential DRep, the old DRep's `drepDelegs` entry is never removed: [1](#0-0) 

The guard `isNothing mAccountState || preserveIncorrectDelegation` causes the code to only *add* the stake credential to the new DRep's `drepDelegs` without removing it from the old DRep's `drepDelegs`. This is the intentionally preserved bug referenced as issue #4772. [2](#0-1) 

**Trigger — `ConwayUnRegDRep` uses the stale index (`GovCert.hs`)**

When a DRep unregisters, `clearDRepDelegations` iterates over `drepDelegs` and sets `dRepDelegationAccountStateL .~ Nothing` for every credential in that set: [3](#0-2) 

Because `drepDelegs` still contains credentials that have already moved to DRep B, those credentials have their forward delegation cleared to `Nothing` even though they are no longer delegated to the unregistering DRep. This is the exact analog of the FrankenDAO bug: the state update (clearing delegation) is applied to the wrong accounts (those that have already moved away), not the accounts that are actually still delegated.

**Hard-fork migration does not repair cleared delegations (`HardFork.hs`)**

`updateDRepDelegations` (run at the PV9→PV10 transition) rebuilds `drepDelegs` from accounts, but only processes accounts whose `dRepDelegationAccountStateL` is `Just (DRepCredential _)`. Accounts already cleared to `Nothing` are silently skipped: [4](#0-3) 

The `_ -> (ds, accountState)` branch at the bottom of `adjustDelegations` leaves `Nothing`-delegation accounts untouched, so the damage is permanent across the hard fork.

**Test confirms the behavior**

The existing test suite explicitly acknowledges this inconsistency during bootstrap: [5](#0-4) 

After unregistering DRep A, `lookupDRepDelegation cred accounts` returns `Nothing` (account delegation cleared), while DRep B's `drepDelegs` still contains `cred` — an inconsistent state that persists through the hard fork.

---

### Impact Explanation

**High. Permanent freezing of rewards/withdrawals where recovery requires a hard fork.**

1. A credential `C` delegates to DRep A, then redelegates to DRep B during bootstrap.
2. DRep A unregisters (voluntarily or by an attacker who controls DRep A).
3. `clearDRepDelegations` uses DRep A's stale `drepDelegs` to set `C`'s `dRepDelegationAccountStateL = Nothing`.
4. `C` now has no DRep delegation. In Conway, reward withdrawals require an active DRep delegation (`ConwayWdrlNotDelegatedToDRep`). `C`'s accumulated rewards are permanently frozen.
5. The PV10 hard fork does not restore `C`'s delegation, so recovery requires a further protocol intervention (another hard fork or a new migration rule).

Additionally, `C`'s stake is no longer counted toward DRep B's voting power, silently distorting governance ratification calculations.

---

### Likelihood Explanation

**Medium.** The bootstrap phase (PV9) is the current live phase on mainnet. Any DRep that has received delegations and then had those delegations moved away can trigger this by unregistering — either maliciously or accidentally. The attacker's cost is zero (they recover their DRep deposit on unregistration). The victim's loss is permanent. No privileged access, governance majority, or key compromise is required; only a standard `ConwayUnRegDRep` certificate signed by the DRep's own credential.

---

### Recommendation

In `ConwayUnRegDRep` processing in `GovCert.hs`, `clearDRepDelegations` should only clear the `dRepDelegationAccountStateL` of an account if that account's current forward delegation actually points to the unregistering DRep (i.e., `dRepDelegationAccountStateL == Just (DRepCredential cred)`). A stale `drepDelegs` entry should not be trusted as authoritative. The fix should be:

```haskell
clearDRepDelegations delegs accountsMap =
  foldr
    (\stakeCred ->
       Map.adjust
         (\acct ->
            if acct ^. dRepDelegationAccountStateL == Just (DRepCredential cred)
              then acct & dRepDelegationAccountStateL .~ Nothing
              else acct)
         stakeCred)
    accountsMap
    delegs
```

Additionally, `updateDRepDelegations` should be extended to detect and repair accounts whose `dRepDelegationAccountStateL` was incorrectly cleared (i.e., accounts whose credential appears in a live DRep's `drepDelegs` but whose forward delegation is `Nothing`).

---

### Proof of Concept

**Sequence of on-chain transactions (all valid, no privileged access required):**

1. **Epoch N, bootstrap phase (PV9):**
   - Attacker registers DRep A (`ConwayRegDRep`).
   - Victim registers stake credential `C` and delegates to DRep A (`ConwayRegDelegCert C (DelegVote (DRepCredential A)) deposit`).

2. **Epoch N+1:**
   - Victim redelegates to DRep B (`ConwayDelegCert C (DelegVote (DRepCredential B))`).
   - Due to bug #4772: DRep A's `drepDelegs` still contains `C`; DRep B's `drepDelegs` also contains `C`; `C`'s account has `dRepDelegationAccountStateL = Just (DRepCredential B)`.

3. **Epoch N+2:**
   - Attacker unregisters DRep A (`ConwayUnRegDRep A deposit`).
   - `clearDRepDelegations (drepDelegs dRepState_A)` is called; `drepDelegs` contains `C`.
   - `C`'s account: `dRepDelegationAccountStateL = Nothing`. ← **Wrong account cleared.**

4. **Epoch N+3 onward (including after PV10 hard fork):**
   - Victim attempts to withdraw rewards: fails with `ConwayWdrlNotDelegatedToDRep`.
   - `updateDRepDelegations` at PV10 does not restore `C`'s delegation.
   - Victim's rewards are permanently frozen. [6](#0-5) [7](#0-6) [4](#0-3)

```haskell
updateDRepDelegations :: ConwayEraCertState era => CertState era -> CertState era
updateDRepDelegations certState =
  let accountsMap = certState ^. certDStateL . accountsL . accountsMapL
      dReps =
        -- Reset all delegations in order to remove any inconsistencies
        -- Delegations will be reset accordingly below.
        Map.map (\dRepState -> dRepState {drepDelegs = Set.empty}) $
          certState ^. certVStateL . vsDRepsL
      (dRepsWithDelegations, accountsWithoutUnknownDRepDelegations) =
        Map.mapAccumWithKey adjustDelegations dReps accountsMap
      adjustDelegations ds stakeCred accountState =
        case accountState ^. dRepDelegationAccountStateL of
          Just (DRepCredential dRep) ->
            let addDelegation _ dRepState =
                  Just $ dRepState {drepDelegs = Set.insert stakeCred (drepDelegs dRepState)}
             in case Map.updateLookupWithKey addDelegation dRep ds of
                  (Nothing, _) -> (ds, accountState & dRepDelegationAccountStateL .~ Nothing)
                  (Just _, ds') -> (ds', accountState)
          _ -> (ds, accountState)
   in certState
        -- Remove dangling delegations to non-existent DReps:
        & certDStateL . accountsL . accountsMapL .~ accountsWithoutUnknownDRepDelegations
        -- Populate DRep delegations with delegatees
        & certVStateL . vsDRepsL .~ dRepsWithDelegations
```
