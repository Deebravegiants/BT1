### Title
Missing CKD Output Verification for `AppPublicKey` Branch Allows Single Byzantine Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` in the MPC contract performs a full cryptographic pairing-based output check for `CKDAppPublicKey::AppPublicKeyPV` requests, but performs **no verification whatsoever** for `CKDAppPublicKey::AppPublicKey` requests. A single attested participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary, fabricated `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will unconditionally accept and deliver it to the waiting user.

---

### Finding Description

In `respond_ckd()`, after authenticating the caller as an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC network's master key and the user's ephemeral key pair. [2](#0-1) 

For `AppPublicKey`, the branch body is empty (`{}`). The contract immediately proceeds to `resolve_yields_for`, delivering whatever `big_c` and `big_y` the caller supplied, with no cryptographic binding to the MPC key or the user's request. [3](#0-2) 

The only gate before this branch is `assert_caller_is_attested_participant_and_protocol_active()`, which requires the caller to be a single registered, attested participant — not a threshold quorum. [4](#0-3) 

The `AppPublicKey` variant is the standard (non-publicly-verifiable) CKD mode, where the user supplies only a G1 point `pk = a·G1`. The node-side `CKDComputation` runs the `ckd()` protocol (not `ckd_pv()`) for this variant, producing `(big_y, big_c)` that encodes the derived key material. [5](#0-4) 

Because the contract cannot perform the pairing check without the G2 component, the `AppPublicKey` branch was left empty — but this means the threshold guarantee is entirely absent for this code path.

---

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request in `pending_ckd_requests`.
2. Construct an arbitrary `CKDResponse { big_c: ..., big_y: ... }` with values of their choosing.
3. Call `respond_ckd(request, forged_response)` directly.
4. The contract accepts the response unconditionally and resumes all queued yield promises with the forged output.

The user receives forged key material. Since the derived key is computed from `big_c` and `big_y` using the user's private scalar `a` (which the attacker does not know), the attacker cannot directly recover the resulting key — but they can deliver a key that is cryptographically garbage, permanently locking any funds the user subsequently sends to an address derived from it. This constitutes **permanent freezing of funds** and **confidential key derivation output without the required participant authorization**, bypassing the threshold-signature requirement entirely.

This maps to the allowed Critical impact: *"Bypass of threshold-signature requirements … or confidential key derivation output without the required participant authorization"* and *"Theft, direct loss, or permanent freezing of funds controlled by the MPC network."*

---

### Likelihood Explanation

The `AppPublicKey` variant is the default, non-PV CKD mode and is the path taken by the standard CLI and SDK. Any single attested participant — a role achievable by any node that has submitted a valid TEE attestation — can exploit this. No collusion above threshold is required. The attacker only needs to race the legitimate `respond_ckd` call, which is straightforward since the pending request map is public on-chain state. Likelihood is **High**.

---

### Recommendation

Add a cryptographic output check for the `AppPublicKey` branch in `respond_ckd`. Since the G2 component is absent, a direct pairing check is not possible. The recommended mitigations are:

1. **Require `AppPublicKeyPV` for all CKD requests** so the contract can always verify the response on-chain. Deprecate the unverifiable `AppPublicKey` variant.
2. **Alternatively**, implement a threshold-vote mechanism for `AppPublicKey` responses (analogous to `respond` for signatures), requiring at least `t` participants to submit the same `(big_c, big_y)` before the contract accepts it, restoring the threshold guarantee.

---

### Proof of Concept

```
1. User calls request_app_private_key({
       app_public_key: AppPublicKey(pk),   // standard non-PV mode
       derivation_path: "my/path",
       domain_id: 0
   }) with 1 yoctoNEAR deposit.
   → Contract stores the pending CKDRequest and yields.

2. Attacker (a single attested participant, account: attacker.near) observes
   the pending request on-chain.

3. Attacker calls respond_ckd(
       request = <the observed CKDRequest>,
       response = CKDResponse {
           big_c: Bls12381G1PublicKey([0xde; 48]),  // arbitrary garbage
           big_y: Bls12381G1PublicKey([0xad; 48]),  // arbitrary garbage
       }
   ).

4. respond_ckd passes assert_caller_is_attested_participant_and_protocol_active().
   The AppPublicKey branch executes `{}` — no check.
   resolve_yields_for() resumes the user's yield with the forged response.

5. User receives (big_c=0xde..., big_y=0xad...) and derives a key from it.
   The key is cryptographically garbage; any funds sent to the derived address
   are permanently frozen.
```

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates this: it passes `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (invalid BLS points) for an `AppPublicKey` request and the contract accepts them without error. [6](#0-5)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
