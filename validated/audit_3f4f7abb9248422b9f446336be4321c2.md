**`ecurve_ossl_t::from_bin` explicitly accepts infinity** (`src/cbmpc/crypto/base_ecc.cpp:317`):

  if (bin.size > 0 && bin[0] == 0)  // infinity
  {

    for (int i = 0; i < bin.size; i++)
      if (bin[i]) return coinbase::error(E_CRYPTO);
    bin.size = 1;
  }
  if (0 >= EC_POINT_oct2point(...)) return openssl_error(...);
  return SUCCESS;
}
```

A 33-byte all-zero buffer (for P-256) passes this check and `EC_POINT_oct2point` decodes it to the point at infinity — returning `SUCCESS`.

**`ecurve_t::check` is what would catch this**, but it is never called:

```cpp
error_t ecurve_t::check(const ecc_point_t& point) const {
  if (!point.is_in_subgroup()) return crypto::error("EC-point is not on curve");
  if (point.is_infinity()) return crypto::error("EC-point is infinity");
  return SUCCESS;
}
```

**Contrast with other functions that do it correctly:**

- `ecdsa2pc.cpp:82`: `if (curve.check(key.Q)) return coinbase::error(E_FORMAT, "invalid key blob");` [1](#0-0) 
- `eddsa_mp.cpp:137`: `if (curve.check(Q)) return coinbase::error(E_FORMAT, "invalid key blob");` [2](#0-1) 
- `ecdsa_mp.cpp:get_public_key_compressed`: **no `curve.check(Q)` call** [3](#0-2) 

**For OSSL curves (P-256, P-384, P-521), `is_in_subgroup` is just `is_on_curve`** (cofactor = 1, so no low-order points exist other than infinity):

```cpp
bool ecurve_ossl_t::is_in_subgroup(const ecc_point_t& P) const { return is_on_curve(P); }
``` [4](#0-3) 

OpenSSL considers infinity to be on the curve, so `is_in_subgroup` returns `true` for infinity — meaning only the explicit `is_infinity()` check inside `curve.check()` would catch it, and that check is absent here.

**ECDSA forgery with Q = ∞:** When the public key is the point at infinity, ECDSA verification computes `u1·G + u2·O = u1·G`. An attacker can forge a valid signature for any message `e` by picking any point `R = (r, y)` and setting `s = e·r⁻¹ mod q`, yielding `u1 = e·s⁻¹ = r`, so `u1·G = R` and the check `R.x mod q == r` passes.

### Title
Missing subgroup/infinity check in `get_public_key_compressed` allows crafted blob to return point-at-infinity as valid public key — (`src/cbmpc/api/ecdsa_mp.cpp`)

### Summary
`ecdsa_mp::get_public_key_compressed` calls `Q.from_bin()` on the attacker-controlled `Q_compressed` field of a key blob but never calls `curve.check(Q)`. The underlying `ecurve_ossl_t::from_bin` explicitly accepts the infinity encoding (all-zero bytes), so an attacker can supply a crafted blob whose `Q_compressed` encodes the point at infinity, causing the function to return `SUCCESS` with `pub_key` set to the compressed encoding of infinity.

### Finding Description
`get_public_key_compressed` at `src/cbmpc/api/ecdsa_mp.cpp:459` deserializes `Q_compressed` from the caller-supplied blob and calls `Q.from_bin(curve, blob.Q_compressed)`. [5](#0-4) 

`ecurve_ossl_t::from_bin` at `src/cbmpc/crypto/base_ecc.cpp:317` explicitly handles the infinity encoding: a buffer of the correct size (33 or 65 bytes for P-256) with all bytes zero is accepted and decoded to the point at infinity via `EC_POINT_oct2point`. [6](#0-5) 

`ecurve_t::check` at `src/cbmpc/crypto/base_ecc.cpp:592` is the guard that rejects infinity and non-subgroup points, but it is never called in this function. [7](#0-6) 

Every other key-blob parsing function in the codebase calls `curve.check()` after `from_bin()` (e.g., `ecdsa2pc.cpp:82`, `eddsa_mp.cpp:137`, `ecdsa_mp.cpp:attach_private_scalar:538`). [1](#0-0) [8](#0-7) 

The `key_blob_v1_t` struct stores `Q_compressed` as a raw `buf_t` with no point validation during deserialization, so the attacker has full control over its content. [9](#0-8) 

Both v1 and v2 (access-control) blobs are accepted by this function. [10](#0-9) 

### Impact Explanation
A caller that uses the returned "public key" (infinity) for ECDSA signature verification is vulnerable to universal signature forgery: with Q = ∞, the verification equation reduces to checking whether `u1·G = R`, which an attacker can satisfy for any message by choosing `s = e·r⁻¹ mod q` for any chosen point `R = (r, y)`. The function returns `SUCCESS` with an invalid public key, giving the caller no indication that the blob was crafted.

### Likelihood Explanation
The attacker only needs to supply a key blob with `Q_compressed` set to 33 bytes of zeros (for P-256). No cryptographic knowledge is required. The function is a public API entry point (`include/cbmpc/api/ecdsa_mp.h`) callable by any API consumer who can supply a key blob.

### Recommendation
Add `if (rv = curve.check(Q)) return coinbase::error(rv, "invalid key blob");` immediately after the `Q.from_bin(...)` call at line 475, matching the pattern used in every other key-blob parsing function in the codebase.

### Proof of Concept
1. Serialize a `key_blob_v1_t` with `version=1`, a valid `curve` (e.g., P-256), any `party_name`, `x_share=0`, `Qis_compressed` with one entry, and `Q_compressed` = 33 bytes of `0x00`.
2. Call `ecdsa_mp::get_public_key_compressed(crafted_blob, pub_key)`.
3. Assert the return value is `SUCCESS` and `pub_key` is 33 bytes of `0x00` (compressed infinity).
4. Pass `pub_key` to any ECDSA verifier; forge a signature for an arbitrary message `e` by picking any `R = r·G` and setting `s = e·r_x⁻¹ mod q`; observe verification succeeds.
