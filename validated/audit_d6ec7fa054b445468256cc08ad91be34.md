### Title
Single Attested Participant Can Inject Arbitrary `payload_hash` via Signature Replay in `respond_verify_foreign_tx` — (`crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never checks that `response.payload_hash` equals `compute_msg_hash(ForeignTxSignPayload{request, actual_observed_values})`. Because `payload_hash` is fully caller-controlled, a single Byzantine attested participant can replay any previously-produced valid MPC signature (publicly visible on-chain) against a different pending request, causing the contract to resolve that request's yields with a hash that does not correspond to the actual foreign-chain observation.

### Finding Description

In `respond_verify_foreign_tx`, the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // caller-supplied
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // verified only against this caller-supplied value
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

If the check passes, the entire `response` (including the caller-supplied `payload_hash`) is serialised and used to resolve all pending yields for the request:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),   // response.payload_hash is untouched
)
``` [2](#0-1) 

There is no step that recomputes `compute_msg_hash(ForeignTxSignPayload{request, values})` and asserts equality with `response.payload_hash`. The contract has no access to the actual extracted values, so it cannot perform this check itself.

`compute_msg_hash` is defined as `SHA-256(borsh(ForeignTxSignPayload))` and is only called by the MPC node before signing and by the off-chain SDK verifier — never inside the contract: [3](#0-2) 

The SDK's `ForeignChainSignatureVerifier::verify_signature` does perform the binding check (`expected_payload_hash == response.payload_hash`), but this is a client-side helper that callers must invoke themselves — it is not enforced by the contract: [4](#0-3) 

**Attack path:**

Every valid `respond_verify_foreign_tx` call submits a `VerifyForeignTransactionResponse` (including `payload_hash` and `signature`) to the NEAR chain, where it is permanently visible. A single attested participant can:

1. Observe any prior on-chain response `{payload_hash: H_prior, signature: sig(H_prior)}` for a completed request R_prior.
2. While a new pending request R_current exists, call `respond_verify_foreign_tx(request=R_current, response={payload_hash: H_prior, signature: sig(H_prior)})`.
3. The contract verifies `sig(H_prior)` against the domain public key — passes, because it is a genuine MPC signature.
4. The contract resolves all pending yields for R_current with `payload_hash = H_prior`.
5. The caller's bridge contract receives a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to R_prior's observation, not R_current's.

The gate `assert_caller_is_attested_participant_and_protocol_active` requires only a single attested participant — no threshold: [5](#0-4) 

Producing the original `sig(H_prior)` required threshold cooperation, but reusing it requires only one participant. The final signature is publicly visible on-chain after any prior `respond_verify_foreign_tx` call succeeds.

### Impact Explanation

A single Byzantine attested participant can make the contract deliver a cryptographically valid but semantically false attestation for any pending foreign-chain verification request. The returned `payload_hash` is bound to a different transaction's observation, not the one the user requested. Bridge contracts that rely on the contract's response without independently recomputing and checking the hash (i.e., without using `ForeignChainSignatureVerifier`) will accept a forged attestation, potentially enabling invalid bridge execution or double-spend conditions.

### Likelihood Explanation

The precondition — possession of a valid MPC signature over any hash — is trivially satisfied by observing any prior on-chain `respond_verify_foreign_tx` call. No key material needs to be stolen; no threshold collusion is needed beyond what already occurred in normal operation. Any single attested participant can execute this attack at any time a pending request exists.

### Recommendation

The contract cannot independently recompute the correct `payload_hash` because it does not receive the extracted values. Two mitigations are possible:

1. **Include extracted values in the response**: Change `VerifyForeignTransactionResponse` to include the `Vec<ExtractedValue>`, and have the contract recompute and assert `compute_msg_hash(ForeignTxSignPayload{request, values}) == response.payload_hash` before accepting the response. This is the strongest fix.

2. **Require threshold-many participants to submit the response**: Require `t` independent `respond_verify_foreign_tx` calls (each from a different attested participant) with matching `payload_hash` before resolving yields, analogous to a vote. This prevents a single participant from injecting a false hash.

### Proof of Concept

```rust
// 1. Obtain a valid prior MPC signature (publicly visible on-chain from any prior session)
let prior_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: prior_request.clone(),
    values: prior_values.clone(),
});
let h_prior = prior_payload.compute_msg_hash().unwrap();
// sig_prior was produced by the MPC network and is on-chain

// 2. A new pending request exists for a different tx
contract.verify_foreign_transaction(new_request_args.clone());

// 3. Single attested participant replays the prior signature for the new request
let forged_response = VerifyForeignTransactionResponse {
    payload_hash: h_prior,          // hash of a DIFFERENT request's observation
    signature: sig_prior.clone(),   // valid MPC signature over h_prior
};
with_active_participant_and_attested_context(&contract);
contract
    .respond_verify_foreign_tx(new_request.clone(), forged_response)
    .expect("contract accepts the replayed signature");

// 4. Caller receives payload_hash = h_prior, which does NOT correspond to new_request
// assert: returned payload_hash != compute_msg_hash(ForeignTxSignPayload{new_request, actual_values})
```

The contract's signature check at line 729 passes because `sig_prior` is a genuine signature over `h_prior`. No check exists to bind `h_prior` to `new_request`'s actual observation. [6](#0-5)
