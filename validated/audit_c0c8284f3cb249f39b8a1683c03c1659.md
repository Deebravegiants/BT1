### Title
Missing HTTPS Enforcement on Redirect URL Persisted to Pool Config — (`chia/farmer/farmer.py`)

### Summary

`update_pool_state` enforces HTTPS on the **current** `pool_config.pool_url` before calling `_pool_get_pool_info`, but applies **no HTTPS check** on the redirect-derived `new_pool_url` before writing it to the persistent YAML state via `PoolingShareState.acquire`. A malicious pool server returning a 301 redirect to an `http://` URL permanently corrupts the stored `pool_url`, causing the farmer to silently stop submitting partials on mainnet on every subsequent cycle.

### Finding Description

**Step 1 — HTTPS guard on the old URL (passes):**

In `update_pool_state`, the HTTPS check at line 601 only validates the *current* `pool_config.pool_url`. If the farmer is already joined to `https://legitimate-pool.com`, this check passes. [1](#0-0) 

**Step 2 — Redirect detection with no HTTPS validation:**

`_pool_get_pool_info` follows 301/308 redirects via aiohttp (which follows redirects by default). If the final URL differs from the original and all hops were 301/308, `new_pool_url` is set to the redirect destination — with no check that it starts with `https://`. [2](#0-1) 

**Step 3 — Unconditional write to persistent YAML:**

Back in `update_pool_state`, if `new_pool_url` is non-`None`, it is written directly to the YAML state file via `PoolingShareState.acquire` without any HTTPS enforcement: [3](#0-2) 

`PoolingShareState.acquire` writes the mutated object back to disk on successful exit: [4](#0-3) 

**Step 4 — Permanent breakage on next cycle:**

On the next `update_pool_state` call, `pool_config.pool_url` is now `http://attacker.com`. The HTTPS check at line 601 fires, logs an error, and `continue`s — skipping all partial submission logic for this pool indefinitely. [5](#0-4) 

### Impact Explanation

The farmer permanently stops submitting partials to the pool on mainnet. The corrupted `pool_url` persists in `pooling/pooling_share_state.yaml` across restarts. The farmer loses all pool rewards until the YAML is manually corrected. This is a **long-lived inability for an honest farmer to process valid pool actions**, matching the High impact criterion.

### Likelihood Explanation

Any pool server the farmer has joined can trigger this by returning a single 301 redirect response. No key material, admin access, or cryptographic break is required — only control of the HTTP response from the pool endpoint. A pool that goes rogue, is compromised, or is malicious from the start can exploit this.

### Recommendation

Add an HTTPS enforcement check on `pool_info_result.new_pool_url` before persisting it, mirroring the existing guard:

```python
if pool_info_result is not None and pool_info_result.new_pool_url is not None:
    new_url = pool_info_result.new_pool_url
    if enforce_https and not new_url.startswith("https://"):
        self.log.error(f"Redirect pool URL must be HTTPS on mainnet: {new_url}, ignoring.")
    else:
        with PoolingShareState.acquire(...) as editable_pool_config:
            editable_pool_config.pool_url = new_url
```
