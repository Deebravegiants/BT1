### Title
Wrong-Curve Public Share Accepted in DKG Due to Missing Curve Binding in ZK Verify - (`src/cbmpc/protocol/ec_dkg.cpp`)

### Summary

`key_share_mp_t::dkg` relies on a comment claiming that `uc_dl_t::verify` performs the curve check on `Qi._j`. It does not. `uc_dl_t::verify` infers the curve from the point itself via `Q.get_curve()`, so a Byzantine peer that sends `Qi._j` on a different curve (e.g., ed25519 in a secp256k1 session) will have its ZK proof verified against the wrong curve's generator and accepted.

### Finding Description

**Serialization path — curve is attacker-controlled**

`ecc_point_t::convert` embeds the curve ID in the wire format: [1](#0-0) 

On deserialization, `convert_fixed_curve` reads the curve from the wire, calls `from_bin` with that curve, then calls `curve.check(*this)` — which only verifies the point is on the curve the attacker declared, not on the session curve: [2](#0-1) 

**DKG round 2 — no explicit curve binding before ZK verify**

After receiving `Qi._j`, the DKG code has a comment that is factually wrong: [3](#0-2) 

There is no `if (Qi._j.get_curve() != curve) return error(...)` guard anywhere between receiving `Qi._j` and calling `pi._j.verify`.

**ZK verify — curve inferred from the attacker-controlled point**

`uc_dl_t::verify` derives its working curve from `Q.get_curve()`: [4](#0-3) 

`curve.check(Q)` at line 66 only confirms Q is on the curve Q itself claims to be on. The generator `G` used for the Fischlin check at line 93 is then the ed25519 generator, not secp256k1's: [5](#0-4) 

An attacker who crafts a valid ed25519 ZK proof for their ed25519 `Qi._j` will pass this check entirely.

**`h_consistency` does not close the gap**

Round 1 broadcasts `sha256(curve.get_name())` and checks all peers agree: [6](#0-5) 

A Byzantine party sends the correct secp256k1 hash in round 1, then sends an ed25519 point in round 2. The consistency check passes; it binds the curve name string, not the curve of `Qi._j`.

**Commitment does not bind the curve either**

The commitment opened at line 155 is over the raw bytes of `Qi._j`. An attacker who committed to an ed25519 point in round 1 and reveals the same bytes in round 2 opens the commitment successfully.

**Downstream acceptance**

After the ZK verify returns `SUCCESS`, the wrong-curve point is stored unconditionally: [7](#0-6) 

`key.Qis` now contains a mix of secp256k1 and ed25519 points. `SUM(key.Qis)` invokes `ecc_point_t::operator+=` across curves; `ecurve_t::check` confirms the two curves differ: [8](#0-7) 

Depending on the implementation path taken by `operator+=`, this either crashes, silently produces a garbage result, or returns an error that aborts the DKG. In all cases the ZK guard — the only cryptographic check meant to validate `Qi._j` — was bypassed and returned `SUCCESS` for a wrong-curve point.

### Impact Explanation

A single Byzantine peer below threshold can inject a public key share on an arbitrary curve into an otherwise honest DKG session. The ZK proof that is supposed to bind `Qi._j` to the session curve is verified against the attacker's chosen curve's generator and passes. This is a **High** impact finding: attacker-controlled points are accepted under the wrong curve in DKG, violating the protocol's curve-binding invariant.

### Likelihood Explanation

Any Byzantine participant in a multi-party DKG session can trigger this. No threshold collusion is required — a single malicious party suffices. The attack requires only crafting a valid ZK proof for a different curve, which is trivial for the party who knows the discrete log.

### Recommendation

Add an explicit curve check immediately after receiving `Qi._j`, before the ZK verify call:

```cpp
// In key_share_mp_t::dkg, after the commitment open:
if (Qi._j.get_curve() != curve)
    return coinbase::error(E_CRYPTO, "Qi._j is on wrong curve");
if (rv = pi._j.verify(Qi._j, sid, j)) return rv;
```

The same pattern applies to `key_share_mp_t::refresh` (line 231) and `dkg_or_refresh_ac` (line 391), which carry the same comment and the same missing guard.

### Proof of Concept

1. Run a 3-party secp256k1 DKG.
2. Party 1 is Byzantine: in round 1 it sends `h_consistency = sha256("secp256k1")` (correct) and commits to an ed25519 point `Qi_evil`.
3. In round 2 it sends `Qi_evil` (curve ID = ed25519 in the wire encoding) together with a valid ed25519 ZK proof of discrete log.
4. Honest parties call `pi._j.verify(Qi._j, sid, 1)`. Inside `uc_dl_t::verify`, `curve = Qi._j.get_curve()` returns ed25519; `curve.check(Qi._j)` passes; the Fischlin check passes against ed25519's generator.
5. Assert: `pi._j.verify` returns `SUCCESS` despite `Qi._j` being on ed25519 in a secp256k1 session — the guard is bypassed.

      return;
