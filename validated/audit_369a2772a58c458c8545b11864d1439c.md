### Title
`PriceVelocityGuardExtension` Uses `block.number` Instead of `block.timestamp`, Making the Velocity Guard Chain-Dependent — (`File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` measures elapsed time between oracle price updates using `block.number` differences. Because block production rates vary across EVM chains (Ethereum ~12 s, BSC ~3 s, Avalanche ~1 s), the effective per-real-time-unit allowance diverges from the admin's intent. On fast-block chains the guard is far more permissive than intended (security weakened); on slow-block chains it is far more restrictive than intended (legitimate swaps revert).

---

### Finding Description

`PriceVelocityGuardExtension.beforeSwap` records `lastUpdateBlock = uint64(block.number)` and computes the allowed price-change budget as:

```
allowedSq = maxChangePerBlockE18² × (1 + blockDiff)
```

where `blockDiff = block.number - prevBlock`. [1](#0-0) 

The parameter is named `maxChangePerBlockE18` and is documented as a per-block cap. An admin calibrating it on Ethereum (12 s/block) who wants to allow at most `X%` price movement per 12 seconds will set `maxChangePerBlockE18 = X`. On Avalanche (1 s/block), the same value allows `X%` per 1 second — a 12× looser guard per unit of real time. On a 60 s/block chain the same value allows only `X%` per 60 seconds — a 5× tighter guard, causing `PriceVelocityExceeded` reverts on legitimate oracle updates.

The `setLastMidPrice` admin function also writes `block.number` into `lastUpdateBlock`, so the reference point is always block-indexed. [2](#0-1) 

The protocol explicitly targets multi-chain deployment (L1 and L2 deployment scripts exist under `smart-contracts-poc/script/`).



---

### Impact Explanation

**Fast-block chains (BSC 3 s, Avalanche 1 s):** The velocity guard is 4–12× more permissive per real-time second than the admin intended. An attacker who can influence the oracle feed (or who simply benefits from a rapidly moving market) can push the mid-price through a much larger range within a single real-time window without triggering the guard. This allows bad-price execution — swaps execute at oracle prices that have moved far beyond the intended safety envelope, causing the pool to pay out more tokens than it should.

**Slow-block chains (some L2s, testnets):** The guard is proportionally tighter. Legitimate oracle price updates that reflect real market moves will trigger `PriceVelocityExceeded`, reverting every swap until the admin manually resets `lastMidPrice`. This breaks core pool swap functionality.

Both outcomes are contest-relevant: the first is a swap conservation / bad-price execution failure; the second is broken core pool functionality causing loss of usability.

---

### Likelihood Explanation

The protocol is designed for multi-chain deployment. Any pool that enables `PriceVelocityGuardExtension` on a non-Ethereum chain is affected. The extension is a first-class periphery contract, not a test mock. The admin who sets `maxChangePerBlockE18` has no on-chain mechanism to know the chain's block time, so miscalibration is the default outcome.

---

### Recommendation

Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension`. Rename the parameter to `maxChangePerSecondE18` (or per-minute, per-hour) and update the allowed-change formula:

```solidity
uint256 timeDiff = block.timestamp - s.lastUpdateTimestamp;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + timeDiff);
```

Store `lastUpdateTimestamp` as a `uint64` (seconds since epoch fits comfortably). This makes the guard chain-agnostic and consistent with how `PriceProvider._isStale` already measures staleness — using `block.timestamp` and `MAX_TIME_DELTA` in seconds. [3](#0-2) 

---

### Proof of Concept

**Setup:** Deploy `PriceVelocityGuardExtension` on Avalanche (1 s/block). Admin sets `maxChangePerBlockE18 = 1e15` (0.1% per block, intending 0.1% per 12 s on Ethereum → ~0.83% per minute).

**On Avalanche:** 60 blocks pass in 60 seconds (1 block/s).

```
blockDiff = 60
allowedSq = (1e15)² × 61 = 6.1e31
sqrt(allowedSq) / 1e18 ≈ 7.8%   ← allowed change in 60 real seconds
```

**On Ethereum:** 5 blocks pass in 60 seconds (12 s/block).

```
blockDiff = 5
allowedSq = (1e15)² × 6 = 6e30
sqrt(allowedSq) / 1e18 ≈ 2.4%   ← allowed change in 60 real seconds
```

The Avalanche deployment allows **3.2× more price movement per real-time minute** than the Ethereum deployment with identical configuration. An oracle price that moves 5% in one minute — blocked on Ethereum — passes the guard on Avalanche, and the swap executes at the manipulated price. [4](#0-3)
