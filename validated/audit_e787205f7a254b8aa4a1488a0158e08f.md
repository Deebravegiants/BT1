### Title
rsETH Share Price Inflation Attack via Zero `pricePercentageLimit` Default — (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle.pricePercentageLimit` is never set in `initialize()` and therefore defaults to `0`. This completely disables the price-increase guard. An attacker who is the first depositor can mint a minimal amount of rsETH, donate a large amount of ETH directly to `LRTDepositPool` (which has an open `receive()` payable), then call the permissionless `updateRSETHPrice()` to inflate the stored `rsETHPrice` by an arbitrary factor. Every subsequent depositor then receives drastically fewer rsETH tokens due to integer-division truncation, losing the bulk of their deposited value.

---

### Finding Description

**Root cause — disabled price guard**

In `LRTOracle._updateRsETHPrice()` the price-increase guard is:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 &&
    priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [1](#0-0) 

Because `pricePercentageLimit` is never initialised in `initialize()`, it is `0` at deployment. The short-circuit `pricePercentageLimit > 0` makes `isPriceIncreaseOffLimit` permanently `false`, so any price jump — no matter how large — is accepted without restriction. [2](#0-1) 

**Root cause — permissionless price update**

`updateRSETHPrice()` is `public` and callable by anyone:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [3](#0-2) 

**Root cause — open ETH receiver on deposit pool**

`LRTDepositPool` accepts raw ETH from any sender:

```solidity
receive() external payable { }
```
