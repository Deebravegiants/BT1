### Title
Unbounded Recursive CBOR Deserialization of Transaction Metadata Enables Stack Exhaustion Within Transaction Size Limit — (File: `libs/cardano-ledger-core/src/Cardano/Ledger/Metadata.hs`)

---

### Summary

The `decodeMetadatum` function performs mutually recursive CBOR deserialization of transaction auxiliary-data metadata with **no nesting depth limit**. An unprivileged transaction sender can craft a transaction whose metadata is a maximally deep nested CBOR list or map structure that fits within the protocol's `maxTxSize` byte limit (currently 16 KB) yet consumes unbounded Haskell call-stack space during deserialization. This is the direct Cardano analog of the JSON-RPC memory-amplification class: a compact serialized representation is accepted by the byte-size gate but causes disproportionate resource consumption during parsing, before any semantic validation is applied.

---

### Finding Description

`decodeMetadatum` in `libs/cardano-ledger-core/src/Cardano/Ledger/Metadata.hs` is mutually recursive with `decodeListN`, `decodeListIndefLen`, `decodeMapN`, and `decodeMapIndefLen`. None of these helpers carry or check a depth counter. [1](#0-0) 

The recursive call chain is:

```
decodeMetadatum
  → decodeListN n []
      → decodeMetadatum          ← no depth guard
          → decodeListN (n-1) []
              → decodeMetadatum  ← unbounded
``` [2](#0-1) 

The `Metadatum` type is a self-referential algebraic data type: [3](#0-2) 

The `NFData` instance for `Metadatum` is also unboundedly recursive via `rnf`: [4](#0-3) 

The CDDL specification for `metadatum` is self-referential with no depth bound: [5](#0-4) 

The only enforced resource bound on metadata is the overall transaction byte size (`maxTxSize`). The ledger rules check this bound **after** full deserialization has already occurred: [6](#0-5) 

The `validateTxAuxData` for the Shelley era unconditionally returns `True`, performing no structural validation of metadata depth: [7](#0-6) 

**Amplification arithmetic**: A single-element CBOR list is encoded as one byte (`0x81`). A 16 KB transaction can therefore encode approximately 16,000 levels of nesting. Each Haskell stack frame for `decodeMetadatum` consumes several hundred bytes. At 16,000 levels, stack consumption reaches several megabytes — at or beyond GHC's default 8 MB stack limit — before the byte-size check is ever evaluated.

The test infrastructure itself acknowledges the need for a depth bound when *generating* metadata, capping it at `maxMetadatumDepth = 2`, but no equivalent cap exists in the production decoder: [8](#0-7) 

---

### Impact Explanation

**Classification: Medium — Attacker-controlled serialized inputs exceed intended validation limits.**

The `maxTxSize` protocol parameter is the intended resource bound for transaction metadata. A deeply nested metadata structure within that byte limit causes unbounded Haskell call-stack consumption during CBOR deserialization. This exceeds the implicit computational resource limit (stack depth) that the byte-size gate was designed to enforce.

A GHC `StackOverflow` async exception thrown during block deserialization is not caught by the `Either DecoderError a` return type of `decodeFull`/`decodeFullDecoder`: [9](#0-8) 

If uncaught, the exception propagates and crashes the processing thread or node process. Because GHC's stack size is configurable per deployment (via `+RTS -K`), nodes with different RTS configurations may handle the same crafted transaction differently — some crashing, others rejecting gracefully — which constitutes a **deterministic disagreement between honest nodes from serialization**, elevating the impact toward High.

---

### Likelihood Explanation

- Any unprivileged transaction sender can include arbitrary auxiliary data in a transaction.
- The attack requires a single transaction, not a flood.
- The crafted payload is trivially constructable: a CBOR byte string of the form `0x81 0x81 0x81 ... 0x00` (nested single-element lists) fits in under 1 KB for 1,000 levels of nesting.
- The `maxTxSize` limit (16 KB) permits ~16,000 nesting levels, far exceeding any realistic stack budget.
- No special privileges, keys, or governance access are required.

---

### Recommendation

1. **Add a depth counter to `decodeMetadatum`**: Pass an `Int` depth parameter and fail with a `DecoderErrorCustom` when it exceeds a safe bound (e.g., 64 or 128 levels, consistent with the existing 64-byte leaf-size limits).

2. **Apply the same fix to `NFData (Metadatum)`**: The `rnf` instance is also unboundedly recursive; consider a depth-limited fold or a continuation-based traversal.

3. **Validate depth at the `validateTxAuxData` boundary**: Even if decoding succeeds, a post-decode depth check in `validateTxAuxData` would provide a defense-in-depth layer.

---

### Proof of Concept

The following CBOR hex encodes 5,000 levels of nested single-element lists terminating in integer `0`, totalling ~5,001 bytes — well within `maxTxSize`:

```
# Python construction:
payload = b'\x81' * 5000 + b'\x00'
# Wrap as metadata: { 0: <payload> }
# Submit as transaction auxiliary data
```

When a node deserializes a block containing this transaction, `decodeMetadatum` recurses 5,000 times before reaching the leaf. With GHC's default 8 MB stack and typical frame sizes, this causes a `StackOverflow` exception during block processing, before `validateMaxTxSizeUTxO` is ever reached. [10](#0-9) [2](#0-1)

decodeMetadatum :: Decoder s Metadatum
decodeMetadatum = do
  dv <- getDecoderVersion
  let checkSizes = dv > natVersion @2
  tkty <- peekTokenType
  case tkty of
    -- We support -(2^64) .. 2^64-1, but not big integers
    -- not even big integer representation of values within range
    TypeUInt -> I <$> decodeInteger
    TypeUInt64 -> I <$> decodeInteger
    TypeNInt -> I <$> decodeInteger
    TypeNInt64 -> I <$> decodeInteger
    TypeBytes -> do
      !ba <- decCBOR
      when (checkSizes && Prim.sizeofByteArray ba > 64) $
        decodeError "bytes .size (0..64): bytestring exceeds 64 bytes"
      return (B ba)
    TypeBytesIndef -> do
      decodeBytesIndef
      !ba <- decodeBytesIndefLen []
      when (checkSizes && Prim.sizeofByteArray ba > 64) $
        decodeError "bytes .size (0..64): bytestring exceeds 64 bytes"
      return (B ba)
    TypeString -> do
      !x <- decodeString
      when (checkSizes && TF.lengthWord8 x > 64) $
        decodeError "text .size (0..64): text exceeds 64 bytes"
      return (S x)
    TypeStringIndef -> do
      decodeStringIndef
      !x <- decodeStringIndefLen []
      when (checkSizes && TF.lengthWord8 x > 64) $
        decodeError "text .size (0..64): text exceeds 64 bytes"
      return (S x)

    -- Why does it work to do the same thing here for 32 and 64bit list len
    -- tokens? On 32bit systems the decodeListLen will fail if the value
    -- really is bigger than maxBound :: Int, and on 64bit systems if a value
    -- that big is provided, then it'll fail when it runs out of input for
    -- such a big list. Hence we can do exactly the same for the 32bit and
    -- 64bit cases.
    TypeListLen -> do
      n <- decodeListLen
      xs <- decodeListN n []
      return (List xs)
    TypeListLen64 -> do
      n <- decodeListLen
      xs <- decodeListN n []
      return (List xs)
    TypeListLenIndef -> do
      decodeListLenIndef
      xs <- decodeListIndefLen []
      return (List xs)

    -- Same logic applies as above for large lists.
    TypeMapLen -> do
      n <- decodeMapLen
      xs <- decodeMapN n []
      return (Map xs)
    TypeMapLen64 -> do
      n <- decodeMapLen
      xs <- decodeMapN n []
      return (Map xs)
    TypeMapLenIndef -> do
      decodeMapLenIndef
      xs <- decodeMapIndefLen []
      return (Map xs)
    _ -> decodeError ("Unsupported token type " <> T.pack (show tkty))
  where
    decodeError msg = cborError (DecoderErrorCustom "metadata" msg)

decodeFullDecoder ::
  -- | Protocol version to be used during decoding.
  Version ->
  -- | Label for error reporting
  Text ->
  -- | The parser for the @ByteString@ to decode. It should decode the given
  -- @ByteString@ into a value of type @a@
  (forall s. Decoder s a) ->
  -- | The @ByteString@ to decode
  BSL.ByteString ->
  Either DecoderError a
decodeFullDecoder version lbl decoder bs =
  case deserialiseDecoder version decoder bs of
    Right (x, leftover) ->
      if BS.null leftover
        then pure x
        else Left $ DecoderErrorLeftover lbl leftover
    Left (e, _) -> Left $ DecoderErrorDeserialiseFailure lbl e
{-# INLINE decodeFullDecoder #-}
