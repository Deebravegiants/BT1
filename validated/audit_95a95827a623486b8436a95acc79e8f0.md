### Title
Attacker-Crafted PVE Blob Bypasses Proof Verification to Force Decryption of Attacker-Chosen Scalar тАФ (`src/cbmpc/protocol/pve.cpp`)

---

### Summary

The public API `decrypt` in `src/cbmpc/api/pve_base_pke.cpp` unconditionally calls `ec_pve_t::decrypt` with `skip_verify=true`. An attacker who controls the serialized `ec_pve_t` blob can craft a ciphertext that causes the legitimate decryptor to recover an attacker-chosen scalar `x_attacker` satisfying `x_attacker * G = Q_attacker`, where `Q_attacker` is also attacker-chosen. The `x_value * G == Q` guard in `restore_from_decrypted` is not a defense because both `x_value` and `Q` are derived from attacker-controlled fields.

### Proof of Concept

**Attacker inputs known:** `ek` (public key, always public), `label` (public context string), target `x_attacker` of their choice.

**Craft the blob:**

1. Pick `x_attacker` freely; compute `Q_attacker = x_attacker * G`.
2. Set `b = 0` (all 128 bits zero тЖТ `bi=0` for every row).
3. Pick any `r[0]` (e.g., 16 zero bytes).
4. Compute `x_bi = DRBG_AES_CTR(r[0]).gen_bn(q)` тАФ fully deterministic, attacker can compute this offline.
5. Compute `x_bi_bar = (x_attacker - x_bi) mod q`.
6. Compute `inner_label = label + "-" + hex(SHA256(Q_attacker))`.
7. Encrypt `x_bi_bar.to_bin()` under `ek` with associated data `inner_label` тЖТ `c[0]`.
8. Set blob fields: `Q = Q_attacker`, `L = label`, `b = 0`, `r[0] = chosen_r`, `c[0] = crafted_ct`. All other `c[i]` can be garbage (they will fail `base_pke.decrypt` and be skipped via `continue`).

**Decryptor calls `decrypt(dk, ek, crafted_blob, label, out_x)`:**

- `skip_verify=true` тЖТ no proof check.
- `inner_label = genPVELabelWithPoint(label, Q_attacker)` тАФ matches what attacker used to encrypt.
- `base_pke.decrypt(dk, inner_label, c[0], x_buf)` тЖТ succeeds, returns `x_bi_bar`.
- `restore_from_decrypted(0, x_bi_bar, curve, x_out)`:
  - `bi = b.get_bit(0) = 0`
  - `x_bi = DRBG(r[0]).gen_bn(q)` тАФ same value attacker computed
  - `x_value = x_bi_bar + x_bi = x_attacker mod q`
  - `x_value * G = Q_attacker` тЬУ тЖТ returns `SUCCESS`
- `out_x = x_attacker` тАФ attacker-chosen value returned to caller.

### Impact Explanation

The decryptor recovers `x_attacker` (an attacker-chosen scalar) and the blob's `Q_attacker = x_attacker * G`. Because the decryptor has no prior knowledge of the secret (that is the purpose of PVE recovery), they cannot detect the substitution. Any subsequent protocol operation that uses the recovered `x` тАФ signing, DKG share reconstruction, key derivation тАФ will operate on the attacker-controlled key. This is a complete key substitution in the PVE recovery path.

---

### Likelihood Explanation

The public API `decrypt` hardcodes `skip_verify=true` and is the only shipped decryption entry point. Any caller of the public API is affected. The attacker only needs: (a) the ability to supply a crafted ciphertext blob (a malicious serialized-input provider), (b) knowledge of `ek` (always public), and (c) knowledge of `label` (a public context string). No threshold collusion, no private key material, and no side-channel access is required.

---

### Recommendation

1. **Remove `skip_verify=true` from the public API `decrypt`**, or perform an equivalent binding check inline: verify that `b == H(Q, label, c0, c1, X0, X1)` before accepting any decrypted output.
2. **Bind `Q` to caller-supplied expected value**: the public API should accept an expected `Q` (or derive it from `dk`) and reject blobs where `pve_ct.get_Q() != expected_Q` before decryption.
3. **Add a label binding check** in the `decrypt` path (not only in `verify`): reject if `label != L` even when `skip_verify=true`.
