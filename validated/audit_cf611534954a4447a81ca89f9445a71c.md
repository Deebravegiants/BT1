### Title
Reward Tokens Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

In `KernelDepositPool`, the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. Because `rewardPerToken()` returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`, any rewards that accrue during the zero-stake window are permanently skipped and locked in the contract with no recovery path.

---

### Finding Description

The `updateReward` modifier executes two independent assignments on every call:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // (A)
    updatedAt = lastTimeRewardApplicable();    // (B)
    ...
}
``` [1](#0-0) 

Line (A) calls `rewardPerToken()`, which short-circuits and returns the stored value unchanged when `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // no accumulation
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
``` [2](#0-1) 

Line (B), however, **always** advances `updatedAt` to the current time regardless of whether `totalKernelStaked` is zero. The two assignments are not coupled: `rewardPerTokenStored` stays flat while `updatedAt` leaps forward.

**Consequence:** When the next user action occurs after a zero-stake gap, the `rewardRate * (lastTimeRewardApplicable() - updatedAt)` term in `rewardPerToken()` starts from the new `updatedAt`, permanently erasing the rewards that accrued during the empty window. Those tokens remain in the contract balance but can never be claimed by anyone, and there is no admin rescue function.

The contract's own NatSpec acknowledges the root cause but relies on an off-chain operational promise rather than an on-chain invariant:

> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract. In this deployment, we're avoiding this issue by ensuring there are always some tokens staked…"* [3](#0-2) 

The contract does not enforce this invariant. Any staker can call `initiateWithdrawal()`, which immediately reduces `totalKernelStaked`:

```solidity
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;
    ...
}
``` [4](#0-3) 

If the last staker withdraws, `totalKernelStaked` drops to zero and the freeze condition is triggered.

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Reward tokens transferred into the contract via `notifyRewardAmount` and accrued during any zero-stake window are permanently unclaimable. There is no sweep, rescue, or re-notify mechanism that can recover them. The `rewardsToken` balance of the contract will exceed the sum of all claimable `rewards[user]` values by exactly the amount frozen during the empty period.

---

### Likelihood Explanation

The trigger is a normal, permissionless user action (`initiateWithdrawal`). It requires only that the last remaining staker withdraws while a reward period is active — a realistic scenario during low-participation periods, market stress, or protocol migration. The `notifyRewardAmount` guard (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new reward period with zero stake; it does not prevent stake from draining to zero after a period has started. [5](#0-4) 

---

### Recommendation

Decouple the `updatedAt` advancement from the zero-stake short-circuit. Only advance `updatedAt` when `totalKernelStaked > 0`, so that the elapsed time during the empty window is not silently discarded:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```

This preserves the unaccumulated time so that when stakers return, the rewards for the empty window are distributed to them rather than frozen. Alternatively, add an admin-callable `recoverUnallocatedRewards()` function that computes and sweeps any surplus `rewardsToken` balance that exceeds the sum of all pending rewards.

---

### Proof of Concept

**Setup:** `duration = 7 days`, `rewardRate = 1000 / 7 days`.

1. **T0** — Admin calls `notifyRewardAmount(1000)`. `finishAt = T0 + 7 days`, `updatedAt = T0`.
2. **T0** — Alice stakes 100 KERNEL. `totalKernelStaked = 100`.
3. **T0 + 1 day** — Alice calls `initiateWithdrawal(100)`.
   - `updateReward(Alice)` fires: `rewardPerTokenStored` accumulates 1 day of rewards; `updatedAt = T0 + 1 day`. Alice's earned rewards are saved.
   - `totalKernelStaked = 0`.
4. **T0 + 2 days** — Bob calls `stake(100)`.
   - `updateReward(Bob)` fires: `rewardPerToken()` returns `rewardPerTokenStored` unchanged (zero-stake branch). **`updatedAt` is advanced to `T0 + 2 days`**, silently discarding 1 day of rewards.
   - `totalKernelStaked = 100`.
5. **T0 + 3 days** — Bob calls `getReward()`.
   - Bob earns rewards for only 1 day (`T0+2` → `T0+3`), not 2 days.
   - The rewards for the window `T0+1` → `T0+2` (~143 tokens) are permanently frozen in the contract. [1](#0-0) [2](#0-1) [4](#0-3)

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
