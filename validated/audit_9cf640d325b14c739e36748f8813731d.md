### Title
Caller-Agnostic Fan-Out Queue Key in `verify_foreign_transaction` Enables 1-yoctonear Front-Running to Block Specific Bridge Transaction Verification - (File: `crates/contract/src/lib.rs`, `crates/contract/src/pending_requests.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint uses a **caller-agnostic** request key for its fan-out queue. Any unprivileged account can fill the 128-slot queue for any specific foreign-chain transaction by submitting 128 identical requests at 1 yoctonear each (~128 yoctonear total cost). Once the queue is saturated, every subsequent legitimate submission for that same transaction is rejected with `PendingRequestQueueFull`, blocking the bridge verification flow for that transaction until all attacker yields time out — at which point the attack can be repeated at the same negligible cost.

---

### Finding Description

The `sign` function constructs a `SignatureRequest` that includes the caller's `predecessor_account_id` as part of the map key, so each caller's queue is isolated:

```rust
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // caller is baked into the key
    &request.path,
);
``` [1](#0-0) 

By contrast, `verify_foreign_transaction` converts the user-supplied args into a `VerifyForeignTransactionRequest` that contains **no caller identity**:

```rust
let request = args_into_verify_foreign_tx_request(request);
// ...
move |this, id| this.add_verify_foreign_tx_request(request, id),
``` [2](#0-1) 

The contract's own test acknowledges this property explicitly:

```rust
// And: caller bob submits the identical request — a different account would today
// be blocked from receiving a response by alice's submission.
``` [3](#0-2) 

All three request types share the same `push_pending_yield` helper, which enforces a hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` per request key:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
// ...
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
}
``` [4](#0-3) 

The minimum deposit required per call is 1 yoctonear:

```rust
const MINIMUM_SIGN_REQUEST_DEPOSIT: NearToken = NearToken::from_yoctonear(1);
``` [5](#0-4) 

**Attack path:**

1. Attacker observes a foreign-chain transaction (e.g., a Bitcoin tx_id) that a bridge user intends to verify via `verify_foreign_transaction`.
2. Attacker submits 128 identical `verify_foreign_transaction` calls with the same `(domain_id, chain, tx_id, confirmations, extractors)` parameters, each with 1 yoctonear deposit. Total cost: 128 yoctonear + gas (~negligible).
3. The fan-out queue for that request key is now saturated at `MAX_PENDING_REQUEST_FAN_OUT`.
4. The legitimate user's `verify_foreign_transaction` call panics with `PendingRequestQueueFull`.
5. The attacker's 128 yields time out after the NEAR yield-resume timeout (~200 blocks, ~4 minutes).
6. The attacker repeats from step 2, indefinitely blocking the specific transaction at near-zero cost per cycle.

---

### Impact Explanation

`verify_foreign_transaction` is the on-chain entry point for the bridge flow: MPC nodes verify the foreign-chain transaction and then produce a threshold signature authorizing the corresponding action. If this endpoint is permanently blocked for a specific transaction, the bridge cannot process it. A user who has already committed funds on the foreign chain (e.g., sent Bitcoin to a bridge address) cannot get the MPC network to sign the corresponding NEAR-side action. This breaks the request lifecycle for that transaction and can result in funds being effectively frozen in the bridge until the attacker stops the attack. This matches the allowed Medium impact: **"request-lifecycle manipulation that breaks production safety/accounting invariants."**

---

### Likelihood Explanation

The attack requires no privileged access, no threshold collusion, and no TEE bypass. Any NEAR account with a trivial balance (~128 yoctonear + gas per wave) can execute it. The target transaction parameters are observable on the foreign chain before the victim submits their NEAR-side request, making front-running straightforward. The attack is repeatable indefinitely at negligible cost.

---

### Recommendation

1. **Include the caller's account ID in the `VerifyForeignTransactionRequest` key**, analogous to how `SignatureRequest` includes `predecessor_account_id`. This isolates each caller's queue and prevents cross-caller queue saturation.

2. **Alternatively, enforce a per-account rate limit** on `verify_foreign_transaction` submissions to bound the number of queue slots any single account can occupy for a given request key.

3. **Raise the minimum deposit** for `verify_foreign_transaction` to make filling 128 slots economically meaningful, reducing the incentive for spam.

---

### Proof of Concept

The contract's own unit test demonstrates the caller-agnostic queue behavior:

```rust
// alice submits
contract.verify_foreign_transaction(request_args.clone());

// bob submits the identical request — different caller, same queue slot
contract.verify_foreign_transaction(request_args);

// Both yields are queued under the single (caller-agnostic) request key.
assert_eq!(
    contract.pending_verify_foreign_tx_requests.get(&request).map(|q| q.len()),
    Some(2),
);
``` [6](#0-5) 

Extending this to 128 submissions (the cap) from a single attacker account saturates the queue:

```rust
for i in 0..MAX_PENDING_REQUEST_FAN_OUT {
    contract.add_signature_request(signature_request.clone(), [i; 32]);
}
// One more append panics with "Pending-request queue is full"
``` [7](#0-6) 

The same cap applies to `pending_verify_foreign_tx_requests` via the shared `push_pending_yield` helper. [8](#0-7)

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
}
