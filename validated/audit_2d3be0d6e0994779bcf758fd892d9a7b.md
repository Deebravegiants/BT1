### Title
Broken Pigeonhole Invariant in `set_state_header` Allows Peer to Omit Receipts and Corrupt Syncing Node State Root - (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

The receipt-proof completeness check in `set_state_header` relies on a Pigeonhole argument that is structurally broken. The block's `prev_chunk_outgoing_receipts_root` Merkle tree covers **all** shards (new and repeated), but the count check only enforces `receipt_proofs.len() == chunks_included` (the number of shards with **new** chunks). Because `from_shard_id` is never verified to correspond to the actual position of the supplied `root` in the Merkle tree, a malicious peer can substitute the proof for a shard that had real outgoing receipts with a proof for a shard that had none, omitting receipts entirely while passing every validation step.

---

### Finding Description

`set_state_header` validates incoming receipt proofs in four sub-steps:

**4c** – count check: `receipt_proofs.len() == block_header.chunks_included()` [1](#0-0) 

**4d** – uniqueness: `from_shard_id` values are inserted into a `HashSet<ShardId>` [2](#0-1) 

**4e** – receipt Merkle proof: `verify_path(*root, proof, receipts_hash)` where `receipts_hash = hash(ReceiptList(shard_id, receipts))` [3](#0-2) 

**4f** – root Merkle proof: `verify_path(block_header.prev_chunk_outgoing_receipts_root(), block_proof, root)` [4](#0-3) 

The code comment explains the intended security argument: [5](#0-4) 

**The flaw:** The block's `prev_chunk_outgoing_receipts_root` is a Merkle root over **all** shards' `prev_outgoing_receipts_root` values — including shards whose chunks were **not** new in this block:

```rust
pub fn compute_chunk_prev_outgoing_receipts_root(&self) -> CryptoHash {
    merklize(&self.iter().map(|chunk| *chunk.prev_outgoing_receipts_root()).collect_vec()).0
}
``` [6](#0-5) 

So the Merkle tree in step 4f has `num_shards` leaves, not `chunks_included` leaves. The Pigeonhole argument requires that the N supplied `from_shard_id` values are drawn from the same N-element set as the included chunks. But:

1. `from_shard_id` is **never** verified to correspond to the position of `root` in the Merkle tree.
2. `root` (from `RootProof`) is **never** verified to equal `block.chunks()[shard_index_of(from_shard_id)].prev_outgoing_receipts_root()`.
3. Step 4e uses `shard_id` (the target shard), not `from_shard_id`, so `from_shard_id` plays no cryptographic role in either 4e or 4f.

Compare with the honest node's construction, which **does** bind `root` to `from_shard_id`:

```rust
let from_shard_index = prev_shard_layout.get_shard_index(*from_shard_id)?;
let root_proof = *block.chunks()[from_shard_index].prev_outgoing_receipts_root();
root_proofs_cur.push(RootProof(root_proof, block_receipts_proofs[from_shard_index].clone()));
``` [7](#0-6) 

The verifier never enforces this binding.

### Impact Explanation

**Concrete attack (4-shard network, shards S0–S3):**

- Block B has new chunks for S0 and S1 (`chunks_included = 2`).
- S1 has outgoing receipts to target shard S0 (the shard being synced).
- S2 has no outgoing receipts to S0 in this block.

Attacker sends `incoming_receipts_proofs` for block B with two entries:
- Entry 1: `from_shard_id = S0`, actual receipts from S0→S0, valid proof against `root_S0`, `RootProof(root_S0, proof_S0_in_block)`.
- Entry 2: `from_shard_id = S2`, **empty** receipts, valid proof against `root_S2` (S2's `prev_outgoing_receipts_root`), `RootProof(root_S2, proof_S2_in_block)`.

All checks pass:
- 4c: 2 == `chunks_included` (2) ✓
- 4d: {S0, S2} unique ✓
- 4e entry 2: `verify_path(root_S2, proof, hash(ReceiptList(S0, [])))` — valid because S2 had no receipts to S0 ✓
- 4f entry 2: `verify_path(block_receipts_root, proof_S2_in_block, root_S2)` — valid because `root_S2` IS a leaf in the block's Merkle tree ✓

The crafted header is written to `DBCol::StateHeaders`. [8](#0-7) 

`set_state_finalize` then reads the stored header, calls `collect_receipts_from_response` on the tampered `incoming_receipts_proofs`, and passes the wrong receipt set to `apply_chunk`: [9](#0-8) 

S1's receipts to S0 are silently omitted. `apply_chunk` produces a state root that diverges from every honest node. There is no post-finalization check that compares the resulting state root against any on-chain commitment.

---

### Likelihood Explanation

Any peer that a syncing node contacts for state sync can trigger this. State sync peers are not authenticated or trusted; the syncing node accepts `ShardStateSyncResponseHeader` from any responding peer. The construction requires only knowledge of the block's chunk layout and the ability to compute standard Merkle proofs over public on-chain data — no validator keys or privileged access needed.

---

### Recommendation

In `set_state_header`, after step 4d, verify that `root` in `RootProof(root, block_proof)` equals the `prev_outgoing_receipts_root` of the chunk at the shard index corresponding to `from_shard_id`:

```rust
let from_shard_index = shard_layout.get_shard_index(*from_shard_id)?;
let expected_root = block.chunks()[from_shard_index].prev_outgoing_receipts_root();
if root != expected_root {
    return Err(Error::Other("set_shard_state failed: root does not match from_shard_id".into()));
}
```

Additionally, verify that the set of `from_shard_id` values equals exactly the set of shard IDs that had new chunks in the block (i.e., `chunk_mask[shard_index] == true`), not merely that they are distinct.

### Proof of Concept

In a multi-shard `TestEnv`:

1. Produce blocks until a sync point where shard S1 has outgoing receipts to shard S0.
2. Obtain an honest `ShardStateSyncResponseHeader` for S0.
3. Mutate it: replace the `ReceiptProofResponse` entry for S1 with one for S2 (a shard with no receipts to S0), providing `RootProof(root_S2, block_proof_S2)` and an empty receipt list with a valid proof against `root_S2`.
4. Call `set_state_header(S0, sync_hash, mutated_header)` — assert it returns `Ok(())`.
5. Apply state parts, call `set_state_finalize(S0, sync_hash)`.
6. Assert that the resulting `ChunkExtra.state_root` for the syncing node differs from the honest node's `ChunkExtra.state_root` for the same block. [10](#0-9) [9](#0-8) [6](#0-5)
