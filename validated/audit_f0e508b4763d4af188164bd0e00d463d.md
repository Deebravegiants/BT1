Audit Report

## Title
Missing Staleness Check in `CrossChainRateReceiver.getRate()` Allows Stale Rate to Over-Mint wrsETH - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver.getRate()` returns the cached `rate` state variable without ever validating it against the stored `lastUpdated` timestamp. All L2 pool contracts (`RSETHPool`, `RSETHPoolV2`, `RSETHPoolV3`, and their variants) call this function to price deposits. When the cross-chain rate has not been refreshed recently, the stale (lower) rate causes new depositors to receive more wrsETH than the current exchange rate entitles them to, diluting existing holders' unclaimed yield.

## Finding Description
`CrossChainRateReceiver` stores both `rate` and `lastUpdated` as state variables: [1](#0-0) 

`lzReceive` correctly updates both when a LayerZero message arrives: [2](#0-1) 

However, `getRate()` returns `rate` unconditionally with no staleness check: [3](#0-2) 

Every L2 pool contract calls this function to price deposits. In `RSETHPoolV3`, `getRate()` delegates directly to the oracle: [4](#0-3) 

The returned rate is used in `viewSwapRsETHAmountAndFee` to compute the wrsETH mint amount: [5](#0-4) 

The identical pattern exists in `RSETHPool.viewSwapRsETHAmountAndFee`: [6](#0-5) 

And in `RSETHPoolV2.viewSwapRsETHAmountAndFee`: [7](#0-6) 

The rate update path originates in `MultiChainRateProvider.updateRate()`, which is a permissionless but off-chain-triggered function: [8](#0-7) 

There is no on-chain enforcement of update frequency. The presence of `lastUpdated` in the receiver confirms the developers anticipated tracking freshness, but the check was never implemented in `getRate()`.

## Impact Explanation
rsETH/ETH rate monotonically increases as staking rewards accrue. When `rate` is stale (lower than the true current rate), the formula `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` produces a larger `rsETHAmount` than warranted. New depositors receive excess wrsETH, which dilutes the share of staking yield belonging to all existing rsETH/wrsETH holders. This constitutes **High — Theft of unclaimed yield** from existing holders, directly matching an allowed impact class.

## Likelihood Explanation
Any normal depositor calling `deposit()` on any L2 pool contract can trigger this without any special privileges. The precondition — a stale rate — requires only that the off-chain keeper has not called `updateRate()` recently, or that a LayerZero message delivery has been delayed. Keeper outages and delivery delays are realistic operational scenarios. The window of staleness can persist for hours or days with no on-chain mechanism to detect or block it. The exploit is repeatable by any depositor during the stale window.

## Recommendation
Add a configurable `maxStaleness` parameter and enforce it inside `getRate()`:

```solidity
uint256 public maxStaleness; // e.g., 86400 (24 hours)

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

`maxStaleness` should be set by the owner and should reflect the expected keeper update cadence with a reasonable buffer.

## Proof of Concept
1. At time `T`, `lzReceive` delivers `rate = 1.05e18` and sets `lastUpdated = T`.
2. Staking rewards accrue; the true L1 rsETH price rises to `1.07e18` by `T + 2 days`.
3. The keeper does not call `updateRate()` during this window.
4. A user calls `RSETHPoolV3.deposit{value: 1 ether}()` at `T + 2 days`.
5. `getRate()` returns the stale `1.05e18` instead of `1.07e18`.
6. `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.952 wrsETH`; correct would be `≈ 0.935 wrsETH`.
7. The user receives ~1.8% more wrsETH than entitled, at the expense of all existing holders.

**Foundry fork test plan**: Fork an L2 (e.g., Arbitrum) where `RSETHPoolV3` is deployed with `CrossChainRateReceiver` as the oracle. Record the current `rate` and `lastUpdated`. Use `vm.warp(block.timestamp + 2 days)` to simulate staleness. Call `deposit{value: 1 ether}()` and assert that the minted `wrsETH` amount exceeds what the current L1 rate (fetched via a separate L1 fork call) would entitle the depositor to.

        rate = latestRate;

        uint256 rateReceiversLength = rateReceivers.length;

        for (uint256 i; i < rateReceiversLength;) {
            uint16 dstChainId = uint16(rateReceivers[i]._chainId);

            bytes memory remoteAndLocalAddresses = abi.encodePacked(rateReceivers[i]._contract, address(this));

            (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
                .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

                dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
            );

            unchecked {
                ++i;
            }
        }

        emit RateUpdated(rate);
    }
