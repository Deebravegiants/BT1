### Title
`ForeignTxSignPayloadV1` Lacks Caller Binding and Nonce — Cross-Caller Replay of Foreign-Chain Attestation Enables Bridge Double-Spend - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The payload signed by the MPC network for `verify_foreign_transaction` requests — `ForeignTxSignPayloadV1` — contains only the foreign-chain RPC request and the extracted values. It does not include the NEAR caller's account ID, a nonce, or any NEAR-side context. Combined with a caller-agnostic pending-request key and a zero tweak hardcoded in the node, the resulting `VerifyForeignTransactionResponse` is a globally reusable attestation: any NEAR account can replay it against any downstream contract that uses the SDK's `verify_signature` helper, and the same response can be submitted an unlimited number of times.

---

### Finding Description

**Root cause 1 — `ForeignTxSignPayloadV1` contains no caller or nonce field**

The struct that the MPC network hashes and signs is:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

The hash is `SHA-256(borsh(ForeignTxSignPayload))`. There is no `predecessor_id`, no nonce, no NEAR contract address, and no per-request unique identifier in the signed material. [1](#0-0) 

**Root cause 2 — Zero tweak hardcoded in the node**

When the node builds the internal `SignatureRequest` for a foreign-tx verification, the tweak is unconditionally set to `[0u8; 32]`, meaning the MPC network always signs with the root key — no key derivation binds the signature to any caller or path:

```rust
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),
    ...
})
``` [2](#0-1) 

The contract's `respond_verify_foreign_tx` confirms this: it verifies the signature against the **root** public key (`secp_pk`), not a derived key. [3](#0-2) 

**Root cause 3 — Caller-agnostic pending-request key**

`VerifyForeignTransactionRequest` (the map key in `pending_verify_foreign_tx_requests`) contains no caller account ID:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [4](#0-3) 

This is explicitly tested and confirmed: alice and bob queue under the **same** key for the same foreign tx. [5](#0-4) 

**Root cause 4 — No completed-request tracking; no replay protection in the SDK**

After `respond_verify_foreign_tx` resolves all queued yields, the entry is removed from `pending_verify_foreign_tx_requests`. There is no "used attestations" set. The contract can accept the same foreign-tx request again immediately. [6](#0-5) 

The SDK helper `ForeignChainSignatureVerifier::verify_signature` only checks that the signature is cryptographically valid against the MPC public key. It performs no replay check and provides no nonce or used-response tracking: [7](#0-6) 

---

### Impact Explanation

The primary documented use case for `verify_foreign_transaction` is the Omnibridge inbound flow: a user proves a foreign-chain deposit to claim wrapped tokens on NEAR. Because the signed attestation is not bound to any caller or nonce:

1. **Cross-caller replay**: Alice obtains a `VerifyForeignTransactionResponse` for Bitcoin tx X. Bob can submit the same response to the bridge contract to claim tokens for a deposit he did not make.
2. **Same-caller double-spend**: Alice submits the same response twice to the bridge contract, claiming tokens twice for a single on-chain deposit.

The bridge contract's only defense would be to maintain its own used-response registry keyed on `payload_hash`. The SDK provides no such utility and does not document this requirement. Any bridge contract that calls `verify_signature` and then acts on the result without additional replay protection is directly exploitable.

This matches the allowed impact: **High — cross-chain replay causing double-spend / invalid bridge execution**.

---

### Likelihood Explanation

- The attack requires no privileged access: any NEAR account can call `verify_foreign_transaction` and any NEAR account can replay the response.
- The primary documented use case (Omnibridge) is exactly the scenario where double-spend is profitable.
- The SDK's `verify_signature` is the intended integration point for downstream contracts; its absence of replay protection makes vulnerable implementations the natural default.
- The contract itself re-accepts the same request after completion, so even the MPC nodes will re-sign the same foreign tx on demand.

---

### Recommendation

1. **Include caller binding in `ForeignTxSignPayloadV1`**: Add `predecessor_id: AccountId` (the NEAR account that submitted the request) to the signed payload. This binds each attestation to a specific caller.
2. **Add a per-request nonce**: Include a unique identifier (e.g., the NEAR receipt ID of the `verify_foreign_transaction` call) in `ForeignTxSignPayloadV1`. The receipt ID is already available in the node's `VerifyForeignTxRequest` struct and is globally unique.
3. **Track completed requests on-chain**: Maintain a set of used `payload_hash` values (or receipt IDs) in the contract to prevent re-submission of the same foreign-tx request after completion.
4. **Update the SDK**: `ForeignChainSignatureVerifier::verify_signature` should document the replay risk and, where possible, accept a nonce or used-response registry to enforce single-use semantics.

---

### Proof of Concept

**Step 1**: Alice submits a `verify_foreign_transaction` request for Bitcoin tx `X` and receives a `VerifyForeignTransactionResponse { payload_hash, signature }`.

**Step 2**: Alice calls the bridge contract with the response. The bridge contract calls `ForeignChainSignatureVerifier::verify_signature`, which succeeds (valid signature over `SHA256(borsh(ForeignTxSignPayloadV1 { request: X, values: [...] }))`). The bridge mints 1 NEAR-BTC to Alice.

**Step 3**: Alice calls the bridge contract again with the **identical** response. `verify_signature` succeeds again — the signature is still valid, the payload hash still matches, and neither the MPC contract nor the SDK has recorded the response as used. The bridge mints another 1 NEAR-BTC to Alice.

**Step 4 (cross-caller variant)**: Bob, who never submitted any request, calls the bridge contract with Alice's response. `verify_signature` succeeds because the signed payload contains no Alice-specific context. The bridge mints 1 NEAR-BTC to Bob for a deposit he did not make.

The signed payload `ForeignTxSignPayloadV1 { request, values }` is identical in all four calls; the MPC root key signature over it is valid in all four calls; and no on-chain state prevents any of them. [1](#0-0) [7](#0-6) [6](#0-5)

    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
