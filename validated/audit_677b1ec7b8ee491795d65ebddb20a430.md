Audit Report

## Title
Polynomial Gas Scaling in `updateRSETHPrice()` via Repeated `getQueuedWithdrawals` Calls Across Assets × NDCs — (`contracts/LRTOracle.sol`, `contracts/NodeDelegator.sol`)

## Summary
`updateRSETHPrice()` is publicly callable and its gas cost scales as O(assets × NDCs × withdrawals × strategies) because `NodeDelegator.getAssetUnstaking()` is invoked once per (asset, NDC) pair, each time fetching the full queued-withdrawal list from EigenLayer. The protocol's own code acknowledges the function breaks above ~120 uncompleted withdrawals and caps the count at 80, but this margin shrinks as more assets or NDCs are added through normal protocol growth, and `maxNodeDelegatorLimit` carries no hard upper bound.

## Finding Description
The call chain is confirmed by the code:

1. `updateRSETHPrice()` (public, `whenNotPaused`) → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` loops over every supported asset and calls `ILRTDepositPool.getTotalAssetDeposits(asset)` per asset.

2. `getTotalAssetDeposits` → `getAssetDistributionData(asset)` (for non-ETH) or `getETHDistributionData()` (for ETH). Both functions loop over every NDC in `nodeDelegatorQueue` and call `INodeDelegator(ndc).getAssetUnstaking(asset)` per NDC.

3. `NodeDelegator.getAssetUnstaking(asset)` calls `_getDelegationManager().getQueuedWithdrawals(address(this))` — one external EigenLayer call — and then iterates over every queued withdrawal and every strategy within it, calling `strategy.sharesToUnderlyingView(shares)` for matching strategies.

The result is that `getQueuedWithdrawals` is called **N × M** times (N = supported assets, M = NDCs), even though the result for a given NDC is identical across all N asset queries. Each call returns up to W withdrawals with S strategies each.

`LRTUnstakingVault.setMaxUncompletedWithdrawalCount` hard-caps the total queued withdrawal count at 80, with the inline comment: *"120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price."* This cap was calibrated for a specific asset/NDC count. `updateMaxNodeDelegatorLimit` in `LRTDepositPool` enforces no upper bound on `maxNodeDelegatorLimit` (initialized to 10 but raiseable without limit by admin), and the supported-asset list has no cap either. As the protocol adds assets and NDCs through normal growth, the 80-withdrawal safety margin erodes: with 5 assets and 10 NDCs the function already issues 50 external `getQueuedWithdrawals` calls before any withdrawal iteration begins.

## Impact Explanation
If `updateRSETHPrice()` reverts out-of-gas, `rsETHPrice` becomes permanently stale. Every deposit (`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()`) and every withdrawal accounting path that reads the stored price will operate on an incorrect value. This constitutes **Medium — Unbounded gas consumption** with a secondary effect of incorrect share/asset accounting for all users once the price is stale.

## Likelihood Explanation
`updateRSETHPrice()` is permissionlessly callable by any address. The gas cost is a function of protocol state (asset count, NDC count, queued withdrawal count) that grows with normal protocol operation and expected admin configuration changes (adding assets, adding NDCs). No malicious actor is required; the protocol reaching its own configured limits is sufficient. The team's own comment confirms awareness that the function breaks above ~120 uncompleted withdrawals, and the current 80-count cap provides a margin that shrinks multiplicatively as assets and NDCs are added.

## Recommendation
1. **Deduplicate `getQueuedWithdrawals` calls**: In `getAssetDistributionData` / `getETHDistributionData`, fetch the queued-withdrawal list once per NDC and compute the unstaking amount for all assets in a single pass, rather than calling `getAssetUnstaking(asset)` separately for each asset.
2. **Maintain an on-chain `assetUnstaking[asset]` accumulator**: Update it at `initiateUnstaking` / `completeUnstaking` time instead of re-deriving it from live EigenLayer queries on every price update.
3. **Hard-cap `maxNodeDelegatorLimit`**: Add an enforced upper bound in `updateMaxNodeDelegatorLimit` analogous to the 80-count cap on `maxUncompletedWithdrawalCount`.
4. **Recalibrate the withdrawal cap** to account for the multiplicative effect of asset count and NDC count, not just the raw withdrawal count.

## Proof of Concept
Call sequence (no privileges required):

```
// Any EOA calls:
LRTOracle.updateRSETHPrice()
  → _updateRsETHPrice()
    → _getTotalEthInProtocol()
      → for each asset[i] in supportedAssetList (N iterations):
          LRTDepositPool.getTotalAssetDeposits(asset[i])
            → getAssetDistributionData(asset[i])   // or getETHDistributionData for ETH
              → for each NDC[j] in nodeDelegatorQueue (M iterations):
                  NodeDelegator[j].getAssetUnstaking(asset[i])
                    → DelegationManager.getQueuedWithdrawals(NDC[j])  // external call, repeated N×M times
                    → for each withdrawal[k] (W iterations):
                        for each strategy[l] (S iterations):
                            strategy.sharesToUnderlyingView(shares)
```

With 5 supported assets, 10 NDCs, 80 total queued withdrawals (the current cap), and 5 strategies per withdrawal:
- External `getQueuedWithdrawals` calls: 5 × 10 = **50**
- Inner `sharesToUnderlyingView` calls: 5 × 80 × 5 = **2,000** (assuming worst-case distribution)

A Foundry fork test against mainnet EigenLayer with the above state parameters would demonstrate the gas consumption. As `maxNodeDelegatorLimit` is raised (e.g., to 25 NDCs) or additional assets are added, the 50-call baseline scales proportionally, pushing the transaction toward the block gas limit even before any withdrawal iteration.
