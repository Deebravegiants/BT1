### Title
Contract Accepts `verify_foreign_transaction` for Node-Unsupported Chains (Ethereum, Solana, Ton), Causing Silent Request Timeouts — (`crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

The `ForeignChainRpcRequest` enum includes `Ethereum`, `Solana`, and `Ton` as valid variants that participants can register on-chain. The contract's `verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which returns the intersection of all participants' registered chains. If all participants register these chains (which the node config and startup flow support), the contract accepts requests for them. However, the node-side `execute_foreign_chain_request` explicitly bails for all three, causing every such accepted request to time out silently rather than being rejected upfront. Users lose gas fees and bridge operations fail without a clear error.

---

### Finding Description

The `ForeignChain` enum contains `Ethereum`, `Solana`, and `Ton` as valid, serializable variants. [1](#0-0) 

The contract's `verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which computes the **strict intersection** of every active participant's registered chains: [2](#0-1) [3](#0-2) 

The `ForeignChainsConfig` struct has `ethereum: Option<ForeignChainConfig>` and `solana: Option<ForeignChainConfig>` fields that operators can populate in their `foreign_chains.yaml`. The node config tester explicitly exercises these fields: [4](#0-3) 

Per the design docs, on startup each node submits `register_available_foreign_chain_config` derived from its local config. If all operators configure Ethereum or Solana in their yaml, the node registers those chains on-chain, and `get_supported_foreign_chains()` returns them as supported. The contract then accepts `verify_foreign_transaction` requests for those chains.

However, the node-side `execute_foreign_chain_request` explicitly bails for these chains: [5](#0-4) [6](#0-5) 

The `ForeignChainInspectors` struct also has no inspector fields for Ethereum, Solana, or Ton: [7](#0-6) 

The design doc confirms that a failed verification produces **no on-chain failure response** — the request simply times out:

> "A failed verification does not produce an on-chain failure response. The request eventually times out and fails with the standard timeout error." [8](#0-7) 

This is the direct analog of the external report: the contract accepts a request it cannot fulfill (because an optional/incomplete configuration is silently tolerated), and the user receives no output — only a timeout.

---

### Impact Explanation

**Medium.** This breaks the request-lifecycle and contract execution-flow invariant: the contract accepts a `verify_foreign_transaction` request, takes the user's deposit and gas, enqueues the yield, but the MPC nodes will always bail during `execute_foreign_chain_request`. The request times out. For bridge use cases (e.g., Omnibridge inbound flow where a foreign-chain transaction must be verified before NEAR-side funds are released), this means the bridge operation fails silently. If the bridge contract has already locked funds on the foreign-chain side, those funds are stuck until the timeout expires. The 1 yoctonear deposit is refunded on timeout, but gas fees are lost and the bridge operation must be retried or abandoned.

---

### Likelihood Explanation

The `ForeignChainsConfig` struct exposes `ethereum` and `solana` as first-class optional fields. The config tester tool exercises them and only marks them as skipped with a comment — there is no enforcement at the contract or node startup level preventing registration. An operator who reads the config documentation, sees `ethereum: Option<ForeignChainConfig>`, and populates it (reasonably expecting Ethereum to be supported) will trigger this path. If all operators in a deployment do so, the contract will accept Ethereum/Solana requests that the nodes cannot process. This is analogous to the external report's scenario where a project owner omits `prices` thinking one currency is sufficient.

---

### Recommendation

1. **Short term**: Add a contract-level check that rejects `verify_foreign_transaction` requests for chains that have no node-side implementation (Ethereum, Solana, Ton). This can be a hardcoded set of "node-implemented chains" checked before enqueuing the yield.
2. **Short term**: Prevent nodes from registering chains they cannot process. The node startup registration logic should filter out chains for which no inspector is built before calling `register_available_foreign_chain_config`.
3. **Long term**: Implement Ethereum, Solana, and Ton inspectors in `execute_foreign_chain_request` to close the gap between what the contract accepts and what the node can process.

---

### Proof of Concept

1. All `n` participants configure `ethereum` in their `foreign_chains.yaml` (a valid config option).
2. On startup, each node calls `register_available_foreign_chain_config` including `ForeignChain::Ethereum`.
3. `get_supported_foreign_chains()` now returns `Ethereum` (all participants registered it).
4. A bridge service calls `verify_foreign_transaction` with `ForeignChainRpcRequest::Ethereum(...)` and attaches 1 yoctonear deposit.
5. The contract passes the `supported_chains.contains(&requested_chain)` check and enqueues the yield.
6. Each MPC node receives the request and calls `execute_foreign_chain_request`.
7. The node hits `bail!("ForeignChainRpcRequest::Ethereum is unsupported")` and produces no signature share.
8. The yield times out; the user's bridge operation fails silently with a generic timeout error. Gas fees are lost. If the bridge has already locked funds on Ethereum, those funds are stuck.

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
