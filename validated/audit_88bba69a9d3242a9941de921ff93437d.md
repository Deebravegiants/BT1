### Title
Gateway Resource Bounds Validation Uses Stale Previous-Block L2 Gas Price for Admission Decision - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
`StatefulTransactionValidator::validate_resource_bounds` gates mempool admission on the **previous committed block's** L2 gas price instead of the **next block's** L2 gas price. This is the direct sequencer analog of the "outdated exchange rate" class: a stale cached value drives a critical admission decision, causing the gateway to accept transactions that will fail execution (or reject transactions that would succeed).

### Finding Description

In `validate_resource_bounds` the gateway reads `previous_block_l2_gas_price` from `gateway_fixed_block_state_reader.get_block_info()`:

```rust

    executable_tx.resource_bounds(),

`gateway_fixed_block_state_reader` is `GatewayFixedBlockSyncStateClient`, which is constructed with a fixed `block_number` (the latest committed block at validator-instantiation time) and caches the result in a `OnceCell<BlockInfo>` that is never refreshed:

```rust

    state_sync_client: SharedStateSyncClient,
    block_number: BlockNumber,
    block_info_cache: OnceCell<BlockInfo>,

``` [2](#0-1)

The threshold the gateway enforces is:

```
threshold = (min_gas_price_percentage / 100) × prev_block_l2_gas_price
``` [4](#0-3) 

The blockifier's execution-time check, however, enforces:

```
tx.max_price_per_unit >= next_block_l2_gas_price
``` [5](#0-4) 

The next block's L2 gas price is computed by the consensus orchestrator via EIP-1559-style adjustment and is **not** the same as the previous block's price whenever block gas usage deviates from the target.

The TODO comment in the source code explicitly acknowledges the wrong value is being used:

```rust

``` [6](#0-5) 

### Impact Explanation

**Case 1 – Gateway admits, execution rejects (invalid tx accepted):**
If the next block's L2 gas price rises above the previous block's (high-demand scenario), any transaction with `max_price_per_unit` in the range `[threshold_prev, next_price)` passes the gateway check but fails the blockifier's pre-validation with `ResourceBoundsError::MaxGasPriceTooLow`. The transaction occupies mempool space and is never sequenced.

**Case 2 – Gateway rejects, execution would accept (valid tx rejected):**
If the next block's L2 gas price falls below the previous block's (low-demand scenario), a transaction with `max_price_per_unit` in the range `[next_price, threshold_prev)` is rejected by the gateway even though it would succeed execution. A valid user transaction is silently dropped.

Both outcomes match the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The L2 gas price adjusts every block whenever actual gas usage differs from the target (EIP-1559 mechanism). On a live network with variable demand this is the common case, not the exception. The magnitude per block is bounded by the adjustment factor, but the discrepancy accumulates whenever demand is consistently above or below target.

### Recommendation

Replace the stale previous-block price with the next block's computed L2 gas price. The consensus orchestrator already calculates this value (`calculate_next_l2_gas_price_for_fin`); it should be made available to the gateway (e.g., via the batcher or a dedicated RPC endpoint) so `validate_resource_bounds` can use the authoritative forward-looking price:

```diff
- // TODO(Arni): getnext_l2_gas_price from the block header.
- let previous_block_l2_gas_price = self
-     .gateway_fixed_block_state_reader
-     .get_block_info()
-     .await?
-     .gas_prices
-     .strk_gas_prices
-     .l2_gas_price;
+ let next_block_l2_gas_price = self
+     .next_l2_gas_price_provider
+     .get_next_l2_gas_price()
+     .await?;
  self.validate_tx_l2_gas_price_within_threshold(
      executable_tx.resource_bounds(),
-     previous_block_l2_gas_price,
+     next_block_l2_gas_price,
  )?;
```

### Proof of Concept

1. Block N is committed with `l2_gas_price = 100 FRI` and high gas usage.
2. The consensus orchestrator computes `next_block_l2_gas_price = 150 FRI` for block N+1.
3. A user submits an invoke transaction with `max_price_per_unit = 120 FRI`.
4. Gateway check (using stale prev-block price, `min_gas_price_percentage = 100`):
   `120 >= 100% × 100 = 100` → **admitted**.
5. Transaction enters the mempool and is forwarded to the batcher.
6. Blockifier pre-validation for block N+1 (using live next-block price):
   `120 < 150` → `ResourceBoundsError::MaxGasPriceTooLow` → **execution rejected**.
7. The transaction is never sequenced; the user's submission is wasted and mempool capacity is consumed.

The reverse path (Case 2) is symmetric: set `next_block_l2_gas_price = 70 FRI` and `max_price_per_unit = 80 FRI`; the gateway rejects (`80 < 100`) a transaction that would have executed successfully.
