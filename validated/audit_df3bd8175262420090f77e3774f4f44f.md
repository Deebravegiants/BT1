### Title
Unguarded `fee_proposal_fri` During Startup Window Allows Malicious Proposer to Permanently Poison the L2 Gas Price Floor - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

During the first `fee_proposal_window_size` (= 10) blocks of the chain, the `fee_proposal_fri` field carried in `ProposalInit` is accepted with **no bounds enforcement**. The guard that enforces the ±0.2 ppt margin is explicitly disabled when `fee_actual` is `None`. A malicious proposer who controls a majority of the first 10 proposal slots can inject an extreme `fee_proposal_fri` (e.g., `u128::MAX`) into the `fee_proposals_window`. Once the window fills, `fee_actual` is computed as the median of those poisoned values and is then used as the **floor** for the EIP-1559 gas price calculation. Because future `fee_proposal_fri` values are bounded to ±0.2 ppt of the poisoned `fee_actual`, recovery is extremely slow and the L2 gas price floor is effectively locked near `u128::MAX`, making all user transactions economically impossible.

---

### Finding Description

**Root cause — the disabled guard:**

In `is_proposal_init_valid`, the bounds check on `fee_proposal_fri` is wrapped in an `if let` that only fires when `fee_actual` is `Some`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    ...
}
``` [1](#0-0) 

`fee_actual` is `None` whenever `height < fee_proposal_window_size` (= 10), because `compute_fee_actual` returns `None` when the window is not yet full:

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ({window_size})");
    return None;
};
``` [2](#0-1) 

**Poisoned values enter the window at decision time:**

At `finalize_decision`, the proposer-supplied `init.fee_proposal_fri` is unconditionally recorded into `fee_proposals_window`:

```rust
self.record_fee_proposal(height, init.fee_proposal_fri);
``` [3](#0-2) 

**`fee_actual` becomes the EIP-1559 floor:**

In `calculate_next_l2_gas_price_for_fin`, `fee_actual` is used as the minimum gas price floor:

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),
    None => config_min,
};
calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
``` [4](#0-3) 

When `price < min_gas_price`, the EIP-1559 function ratchets the price upward by at most `price / 333` per block — a slow but irreversible climb toward `effective_min`:

```rust
if price < min_gas_price {
    let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
    let adjusted = price.0 + max_increase;
    let adjusted_price = adjusted.min(min_gas_price.0);
    return GasPrice(adjusted_price);
}
``` [5](#0-4) 

**Future proposals are locked near the poisoned value:**

Once `fee_actual` is poisoned, future `fee_proposal_fri` values are bounded to `fee_actual * (1 ± margin_ppt/1000)` = ±0.2 ppt per block. With `margin_ppt = 2`, halving the price takes ~347 blocks; reaching a safe value from `u128::MAX` is practically impossible without operator intervention: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

The corrupted `fee_actual` becomes the floor passed to `calculate_next_base_gas_price`. With `effective_min = u128::MAX`, the EIP-1559 ratchet drives `next_l2_gas_price` toward `u128::MAX` over thousands of blocks. The gateway's stateful validator rejects any transaction whose `max_price_per_unit` falls below `previous_block_l2_gas_price * threshold`: [8](#0-7) 

Once the gas price floor is near `u128::MAX`, no user transaction can satisfy the threshold, effectively halting the chain economically. Recovery requires operator intervention via `override_l2_gas_price_fri`.

This matches the allowed impact: **Critical — Incorrect fee/gas/resource accounting with economic impact.**

---

### Likelihood Explanation

With `window_size = 10` (even), the median is the average of the 5th and 6th sorted values. A malicious proposer needs to control **≥ 5 of the first 10 blocks** to shift the median to an extreme value. In a round-robin BFT system with 2 validators, one validator proposes blocks 0, 2, 4, 6, 8 — exactly 5 of the first 10. With 3+ validators a single actor controls fewer slots and cannot shift the median alone, but a colluding minority (e.g., 2 of 4 validators) can. The attack window is narrow (first 10 blocks only) but permanently effective once executed.

---

### Recommendation

1. **Short term**: When `fee_actual` is `None` (startup), enforce an absolute bound on `fee_proposal_fri` anchored to the current `l2_gas_price` (the honest proposer's fallback value, already used in `compute_proposer_fee_proposal`). Reject any `fee_proposal_fri` outside `[l2_gas_price / (1 + margin), l2_gas_price * (1 + margin)]` even during startup.

2. **Long term**: Treat the startup window as a known attack surface. Document it explicitly, add a test that verifies extreme `fee_proposal_fri` values are rejected during the first `window_size` blocks, and consider initializing the window from genesis state (e.g., from a hard-coded genesis `fee_proposal_fri`) so `fee_actual` is never `None`.

### Proof of Concept

**Setup**: 2-validator BFT network starting at genesis (height 0). `fee_proposal_window_size = 10`, `fee_proposal_margin_ppt = 2`.

**Steps**:

1. Malicious validator V1 is the round-robin proposer for blocks 0, 2, 4, 6, 8.
2. For each of those 5 blocks, V1 sends `ProposalInit` with `fee_proposal_fri = u128::MAX`.
3. `is_proposal_init_valid` is called. `fee_actual = None` (height < 10), so the bounds check at lines 398–416 is **skipped**. The proposal passes validation.
4. At decision time, `record_fee_proposal(height, Some(u128::MAX))` is called for heights 0, 2, 4, 6, 8.
5. At height 10, `compute_fee_actual` collects the window `[MAX, honest, MAX, honest, MAX, honest, MAX, honest, MAX, honest]`. Sorted: `[honest×5, MAX×5]`. Median = average of 5th and 6th = `(honest + MAX) / 2 ≈ MAX/2`.
6. `effective_min = max(config_min, MAX/2) = MAX/2`.
7. `calculate_next_base_gas_price(3_gwei, ..., MAX/2)`: since `3_gwei < MAX/2`, the ratchet fires: `adjusted = 3_gwei + 3_gwei/333`. The price begins climbing.
8. Future proposers must set `fee_proposal_fri` within ±0.2 ppt of `MAX/2`. They cannot reduce it significantly. The window stays poisoned for the next 10 blocks, keeping `fee_actual ≈ MAX/2`.
9. After ~12,000 blocks, `l2_gas_price` reaches `MAX/2`. All user transactions are rejected by the gateway's gas price threshold check.

**Corrupted value**: `next_l2_gas_price` in every block header from height 10 onward, stored in `StorageBlockHeader.next_l2_gas_price` and used as the admission floor for all subsequent transactions. [9](#0-8) [1](#0-0) [10](#0-9)

pub(crate) fn fee_proposal_bounds(fee_actual: GasPrice, margin_ppt: u128) -> (u128, u128) {
    let denom = U256::from(PPT_DENOMINATOR);
    let scaled = denom + U256::from(margin_ppt);
    let fee_actual_u256 = U256::from(fee_actual.0);
    let upper = u128::try_from(fee_actual_u256 * scaled / denom).unwrap_or(u128::MAX);
    let lower = u128::try_from(fee_actual_u256 * denom / scaled).unwrap_or(0);
    (lower, upper)
}

pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,
) -> GasPrice {
    if let Some(override_value) = override_l2_gas_price_fri {
        info!(
            "L2 gas price ({}) is not updated, remains on override value of {override_value} fri",
            current_l2_gas_price.0
        );
        return GasPrice(override_value);
    }
    let gas_target = VersionedConstants::latest_constants().gas_target;
    let config_min = get_min_gas_price_for_height(height, min_l2_gas_price_per_height);
    let effective_min = match fee_actual {
        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}
