### Title
Unvalidated Peer-Supplied Transaction Hash Stored Verbatim, Corrupting `starknet_getTransactionByHash` RPC Responses — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary
The p2p sync client's `TransactionStreamFactory::parse_data_for_block` accepts `FullTransaction` messages from unauthenticated peers and pushes the peer-supplied `transaction_hash` directly into `BlockBody::transaction_hashes` without recomputing or verifying it against the transaction body. The resulting `BlockBody` is committed to storage via `append_body`, which inserts the attacker-controlled hash as the key in `transaction_hash_to_idx_table`. The RPC layer then serves this corrupted mapping verbatim to callers of `starknet_getTransactionByHash` and `starknet_getTransactionByBlockIdAndIndex`.

### Finding Description

**Step 1 — Unvalidated hash accepted from peer.**

In `parse_data_for_block`, the `transaction_hash` field of each `FullTransaction` protobuf message is pushed directly into `block_body.transaction_hashes` with an explicit TODO acknowledging the missing validation: [1](#0-0) 

No call to `get_transaction_hash` or `validate_transaction_hash` (both of which exist in `crates/starknet_api/src/transaction_hash.rs`) is made. The peer is not reported for a hash mismatch because the mismatch is never detected — the only `BadPeer` errors raised are for wrong transaction count or a missing `Fin` message.

**Step 2 — Corrupted `BlockBody` committed to storage.**

`write_to_storage` calls `append_body`, which calls `write_transactions`. For every transaction, the attacker-supplied `tx_hash` is inserted as the lookup key: [2](#0-1) 

The `TransactionMetadata` struct also stores `tx_hash: *tx_hash`, so both the forward lookup table (`transaction_hash_to_idx_table`) and the per-index metadata record carry the wrong hash.

**Step 3 — RPC serves the corrupted hash.**

`get_transaction_by_hash` resolves the caller-supplied hash through `get_transaction_idx_by_hash` (which queries `transaction_hash_to_idx_table`) and returns `TransactionWithHash { transaction, transaction_hash }` where `transaction_hash` is whatever the caller passed in: [3](#0-2) 

Consequences:
- `starknet_getTransactionByHash(attacker_hash)` → returns the real transaction body paired with the attacker-chosen hash (authoritative-looking wrong value).
- `starknet_getTransactionByHash(real_hash)` → `TRANSACTION_HASH_NOT_FOUND` (real hash was never inserted as a key).
- `starknet_getTransactionByBlockIdAndIndex` → returns the transaction with the wrong hash via `get_transaction_hash_by_idx`, which reads `tx_hash` from `TransactionMetadata`.

**Step 4 — No cross-verification exists.**

The block header (synced independently via `HeaderStreamBuilder`) carries a `transaction_commitment` Merkle root over the correct transaction hashes. However, there is no code anywhere in the p2p sync client that verifies the stored transaction hashes against this commitment after storage. The two sync streams (header and transaction) are completely independent with no post-storage integrity check. [4](#0-3) 

### Impact Explanation
Any node running the p2p sync client (the `StateSyncRunner` with `p2p_sync_client_config` enabled) that connects to a malicious peer will permanently store attacker-controlled transaction hashes. The RPC server backed by that storage will return authoritative-looking wrong values for `starknet_getTransactionByHash` and `starknet_getTransactionByBlockIdAndIndex`. The real transaction hashes become unfindable by hash. This is a persistent storage corruption, not a transient resource-waste issue.

### Likelihood Explanation
The p2p network is open to unauthenticated peers. A single malicious peer that wins the SQMR query race for a given block's transactions can corrupt all transaction hash entries for that block. The peer is never reported for this misbehavior because no hash validation is performed. The TODO comment confirms the developers are aware the validation is missing.

### Recommendation
After receiving each `FullTransaction` from a peer, recompute the transaction hash using `get_transaction_hash` (or `validate_transaction_hash` for blocks below `MAINNET_TRANSACTION_HASH_WITH_VERSION`) and compare it against the peer-supplied value. If they differ, treat it as a `BadPeerError` and report the peer. The function already exists in `crates/starknet_api/src/transaction_hash.rs`. [5](#0-4) 

### Proof of Concept
1. Stand up a node with `p2p_sync_client_config` enabled.
2. Intercept or replace the SQMR transaction responder with a mock that, for block N, returns `FullTransaction { transaction: <real_tx>, transaction_output: <real_output>, transaction_hash: <arbitrary_felt> }`.
3. Let the sync client process the response.
4. Call `starknet_getTransactionByHash(<real_hash>)` → observe `TRANSACTION_HASH_NOT_FOUND`.
5. Call `starknet_getTransactionByHash(<arbitrary_felt>)` → observe the real transaction body returned with the attacker-chosen hash.
6. Call `starknet_getTransactionByBlockIdAndIndex(N, 0)` → observe the transaction returned with the attacker-chosen hash.

    fn parse_data_for_block<'a>(
        transactions_response_manager: &'a mut ClientResponsesManager<DataOrFin<FullTransaction>>,
        block_number: BlockNumber,
        storage_reader: &'a StorageReader,
    ) -> BoxFuture<'a, Result<Option<Self::Output>, ParseDataError>> {
        async move {
            let mut block_body = BlockBody::default();
            let mut current_transaction_len = 0;
            let target_transaction_len = storage_reader
                .begin_ro_txn()?
                .get_block_header(block_number)?
                .expect("A header with number lower than the header marker is missing")
                .n_transactions;
            while current_transaction_len < target_transaction_len {
                let maybe_transaction = transactions_response_manager.next().await.ok_or(
                    ParseDataError::BadPeer(BadPeerError::SessionEndedWithoutFin {
                        type_description: Self::TYPE_DESCRIPTION,
                    }),
                )?;
                let Some(FullTransaction { transaction, transaction_output, transaction_hash }) =
                    maybe_transaction?.0
                else {
                    if current_transaction_len == 0 {
                        return Ok(None);
                    } else {
                        return Err(ParseDataError::BadPeer(BadPeerError::NotEnoughTransactions {
                            expected: target_transaction_len,
                            actual: current_transaction_len,
                            block_number: block_number.0,
                        }));
                    }
                };
                block_body.transactions.push(transaction);
                block_body.transaction_outputs.push(transaction_output);
                // TODO(eitan): Validate transaction hash from untrusted sources
                block_body.transaction_hashes.push(transaction_hash);
                current_transaction_len += 1;
            }
            Ok(Some((block_body, block_number)))
        }
        .boxed()
    }

pub fn validate_transaction_hash(
    transaction: &Transaction,
    block_number: &BlockNumber,
    chain_id: &ChainId,
    expected_hash: TransactionHash,
    transaction_options: &TransactionOptions,
) -> Result<bool, StarknetApiError> {
    let mut possible_hashes = get_deprecated_transaction_hashes(
        chain_id,
        block_number,
        transaction,
        transaction_options,
    )?;
    possible_hashes.push(get_transaction_hash(transaction, chain_id, transaction_options)?);
    Ok(possible_hashes.contains(&expected_hash))
}
