Audit Report

## Title
No Staleness Validation on Chainlink Price Feed Allows Stale Price to Inflate rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt` and `answeredInRound`, accepting any price regardless of age. This unvalidated price flows directly into rsETH minting calculations and the global rsETH/ETH rate update, enabling an attacker to receive excess rsETH backed by no real assets (diluting existing holders' yield) or to trigger a protocol-wide pause via a stale low price.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, the five-value destructure from `latestRoundData()` drops all freshness fields:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

No check of the form `block.timestamp - updatedAt > MAX_DELAY` or `answeredInRound < roundId` is present. This contrasts directly with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 30–32, which validates both conditions before accepting a price.

The stale price propagates through two critical paths:

1. **Deposit minting** — `LRTDepositPool.getRsETHAmountToMint()` (line 520) computes `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`. `lrtOracle.getAssetPrice()` (LRTOracle line 157) delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, which resolves to the unvalidated `ChainlinkPriceOracle`. A stale high price inflates the numerator, minting excess rsETH.

2. **rsETH price update** — `LRTOracle._getTotalEthInProtocol()` (line 339) calls `getAssetPrice(asset)` for every supported asset to compute total protocol TVL. A stale low price deflates TVL, causing `newRsETHPrice` to fall below `highestRsethPrice` by more than `pricePercentageLimit`, triggering the auto-pause at lines 277–281.

`updateRSETHPrice()` (line 87) is a public function with no access control, so any caller can trigger path 2.

## Impact Explanation
**Primary — High: Theft of unclaimed yield.** When a feed is stale and its last reported price exceeds the true current price (e.g., an LST has depegged but the feed has not updated), a depositor receives `rsethAmountToMint = amount * stalePriceHigh / rsETHPrice`. The excess rsETH is backed by no real assets; its value is extracted from the yield pool of existing rsETH holders, whose rsETH now redeems for proportionally less ETH.

**Secondary — Medium: Temporary freezing of funds.** When a feed is stale and its last reported price is below the true price, any caller invoking `updateRSETHPrice()` causes `_getTotalEthInProtocol()` to undercount TVL. If the resulting `newRsETHPrice` drops more than `pricePercentageLimit` below `highestRsethPrice`, lines 278–281 pause the deposit pool, withdrawal manager, and oracle simultaneously, freezing all user deposits and withdrawals until an admin manually unpauses.

## Likelihood Explanation
Chainlink feeds operate on heartbeat intervals (e.g., 1 hour for stETH/ETH on mainnet, up to 24 hours for some feeds). During network congestion, oracle node outages, or L2 sequencer downtime, feeds routinely go stale within their heartbeat window while the underlying asset price moves. No special privilege is required: `depositAsset()` and `depositETH()` are callable by any user; `updateRSETHPrice()` has no access control. An attacker only needs to observe a stale feed and submit a transaction — no victim interaction, no governance capture, no oracle operator compromise is needed.

## Recommendation
Add staleness and completeness checks to `ChainlinkPriceOracle.getAssetPrice()`, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS_DELAY) revert StalePrice();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS_DELAY` should be configured per-feed based on its documented heartbeat interval (e.g., 3600 seconds for a 1-hour heartbeat feed, with a small buffer).

## Proof of Concept

**Setup:** stETH/ETH Chainlink feed last updated 2 hours ago; last reported price is `1.02e18`; true current price is `1.00e18` (2% depeg). `rsETHPrice` is `1e18`.

**Yield theft path:**
1. Attacker observes the stale feed still reports `1.02e18`.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint(stETH, 100e18)`:
   - `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.02e18`
   - `rsethAmountToMint = 100e18 * 1.02e18 / 1e18 = 102e18`
4. Attacker receives 102 rsETH for 100 stETH (true value 100 ETH). The 2 rsETH excess is unbacked — existing holders' rsETH is diluted by 2 ETH of phantom value.

**Pause path:**
1. Stale feed reports `0.98e18` for stETH/ETH (true price `1.00e18`). `pricePercentageLimit = 1e16` (1%). `highestRsethPrice = 1.01e18`.
2. Any caller invokes `updateRSETHPrice()`.
3. `_getTotalEthInProtocol()` uses stale `0.98e18` → `newRsETHPrice` drops >1% below `highestRsethPrice`.
4. Lines 278–281 execute: deposit pool, withdrawal manager, and oracle are all paused.
5. All user deposits and withdrawals are frozen until admin manually unpauses.

**Foundry fork test plan:** Fork mainnet, mock the stETH/ETH Chainlink aggregator to return a round with `updatedAt = block.timestamp - 2 hours` and `price = 1.02e18`. Call `depositAsset` and assert `rsethAmountToMint > amount`. Then mock `price = 0.98e18`, call `updateRSETHPrice()`, and assert `lrtDepositPool.paused() == true`.
