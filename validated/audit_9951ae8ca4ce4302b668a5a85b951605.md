### Title
Unprotected `init()` Allows Any Caller to Seize Control of the MPC Participant Set — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `init()` function in the MPC contract carries the `#[init]` attribute (preventing re-initialization) but lacks the `#[private]` attribute (which would restrict callers to the contract account itself). Any external account can race to call `init()` between contract deployment and the legitimate operator call, supplying an arbitrary `parameters` set and installing themselves as the sole participant(s) with a threshold of 1.

---

### Finding Description

`init()` is declared at line 1924 of `crates/contract/src/lib.rs`:

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

The NEAR SDK's `#[init]` macro only guarantees the function runs at most once (it panics if contract state already exists). It does **not** restrict who may call the function. The `#[private]` macro is the mechanism that enforces `predecessor_account_id == current_account_id`, blocking external callers.

Contrast with `init_running()`, which is correctly guarded:

```rust
#[private]
#[init]
#[handle_result]
pub fn init_running( ... )
``` [2](#0-1) 

`init()` has no such guard. There is no `assert_caller_is_signer()`, no `predecessor == current_account_id` check, and no `predecessor == deployer` check anywhere inside the function body. [3](#0-2) 

---

### Impact Explanation

An attacker who wins the race to call `init()` supplies their own `ThresholdParameters`, installing themselves as the only participant with `threshold = 1`. From that point:

- They are the sole voter for `vote_add_domains`, `vote_new_parameters`, `vote_pk`, `vote_reshared`, `propose_update`, and `vote_update`.
- They can add signing domains, drive DKG to completion, and become the only entity capable of calling `respond()` to resolve signature requests.
- They can issue threshold signatures for any payload submitted by any user, constituting **unauthorized threshold signature issuance**.
- The legitimate operator's subsequent `init()` call fails with "contract already initialized," permanently locking them out.

This matches the allowed Critical impact: *Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization.*

---

### Likelihood Explanation

NEAR contract deployment and initialization are two separate transactions. The window between them is observable on-chain by any party monitoring the network. A bot watching for `DeployContract` actions targeting the known MPC contract account ID can immediately submit a crafted `init()` call in the same or next block. No privileged access, leaked key, or collusion is required — only a funded NEAR account and knowledge of the contract's ABI (which is public).

---

### Recommendation

Add the `#[private]` attribute to `init()`, mirroring the protection already applied to `init_running()`:

```rust
#[private]
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

This restricts the call to `predecessor_account_id == current_account_id`, meaning only the contract account itself (via a self-call or a promise from the deployer's transaction) can initialize it. Alternatively, add an explicit check that `env::predecessor_account_id() == env::current_account_id()` at the top of the function body.

---

### Proof of Concept

1. Operator deploys the MPC contract to `mpc-contract.near` (transaction T1).
2. Attacker observes T1 on-chain and immediately submits:
   ```json
   near contract call-function as-transaction mpc-contract.near init \
     json-args '{
       "parameters": {
         "participants": {
           "next_id": 1,
           "participants": [["attacker.near", 0, { ... attacker_info ... }]]
         },
         "threshold": 1
       }
     }' \
     sign-as attacker.near ...
   ```
3. Attacker's `init()` call lands before the operator's, setting `attacker.near` as the sole participant with threshold 1.
4. Operator's `init()` call fails: "The contract has already been initialized."
5. Attacker calls `vote_add_domains` to add a Secp256k1 signing domain, drives DKG as the sole participant, and gains exclusive control over threshold signing for all user `sign()` requests. [4](#0-3)
