Audit Report

## Title
Fee Calculation Truncates to Zero via Integer Division, Enabling Fee-Free Deposits - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All L2 deposit pool contracts compute the protocol fee as `amount * feeBps / 10_000` using Solidity integer division, which truncates toward zero. When `amount * feeBps < 10_000`, the fee evaluates to exactly `0`, and the depositor receives the full rsETH/wrsETH/agETH output while the protocol collects nothing. Any depositor can exploit this by splitting deposits into sub-threshold amounts, systematically eliminating all fee revenue across every deployed pool.

## Finding Description
The fee computation in `viewSwapRsETHAmountAndFee` across all pool variants is:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
```

Confirmed at:
- `RSETHPoolV3.sol` L300, L324 [1](#0-0) 
- `RSETHPoolV2.sol` L226 [2](#0-1) 
- `RSETHPool.sol` L312, L336 [3](#0-2) 

The only guard in the deposit path is `if (amount == 0) revert InvalidAmount()` — there is no minimum deposit enforcing `amount * feeBps >= 10_000`. [4](#0-3) 

When `amount * feeBps < 10_000`, Solidity truncates the result to `0`. `amountAfterFee` then equals `amount` in full, and `feeEarnedInETH += 0` — the protocol records zero revenue for that deposit. [5](#0-4) 

## Impact Explanation
The protocol fee is the sole revenue mechanism for L2 deposits. Systematic zero-fee deposits constitute theft of unclaimed yield from the protocol: the protocol is owed fees per its configured `feeBps` rate, but collects nothing. This maps directly to **High — Theft of unclaimed yield** in the allowed impact scope. The attack scales across all six pool contracts and all supported tokens.

## Likelihood Explanation
No special privilege, capital, or front-running is required. The threshold below which fees round to zero is `floor(10_000 / feeBps)` units of the deposit token. At `feeBps = 5`, any deposit under 2000 wei of ETH pays zero fee. On L2s where gas costs fractions of a cent per transaction, splitting a 1 ETH deposit into ~501 transactions of 1999 wei each costs negligible gas while eliminating the entire 0.05% fee (~500,000 wei). The attack is permissionless, repeatable, and economically rational for any depositor on cheap-gas L2s.

## Recommendation
Replace truncating division with ceiling division so any non-zero `feeBps` on any non-zero `amount` always yields at least 1 unit of fee:

```solidity
// Current (rounds down):
fee = amount * feeBps / 10_000;

// Fix (rounds up):
fee = (amount * feeBps + 9_999) / 10_000;
// or using OpenZeppelin Math:
fee = Math.mulDiv(amount, feeBps, 10_000, Math.Rounding.Ceil);
```

Apply this change to all six pool contracts at every fee computation site.

## Proof of Concept
Deploy `RSETHPoolV3` on a local fork with `feeBps = 5`. Call `deposit{value: 1999}("")` — `viewSwapRsETHAmountAndFee(1999)` returns `fee = 1999 * 5 / 10_000 = 9995 / 10_000 = 0`. Repeat 501 times to deposit ~1 ETH total. Verify `feeEarnedInETH == 0` while the depositor holds the full rsETH equivalent of ~1 ETH. Expected fee at 0.05% would be ~500,000 wei. A Foundry fuzz test asserting `fee > 0` whenever `amount > 0 && feeBps > 0` will fail on any input where `amount * feeBps < 10_000`.
