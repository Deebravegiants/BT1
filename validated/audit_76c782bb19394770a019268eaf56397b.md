Audit Report

## Title
`NodeDelegator.getAssetUnstaking()` Uses Raw `scaledShares` Without Applying `slashingFactor`, Inflating `rsETHPrice` Post-Slash and Enabling Direct Theft of At-Rest User Funds - (`contracts/NodeDelegator.sol`)

## Summary

`NodeDelegator.getAssetUnstaking()` reads raw `scaledShares` from EigenLayer's `getQueuedWithdrawals()` and converts them to underlying tokens without applying the operator's current `slashingFactor`. After an operator slash, this inflates `getTotalAssetDeposits()`, which propagates into `LRTOracle._updateRsETHPrice()`, overstating `rsETHPrice`. An attacker can call the public `updateRSETHPrice()` to lock in the inflated price, then `initiateWithdrawal()` to commit a larger `expectedAssetAmount` than the vault will actually receive from EigenLayer, draining at-rest funds from other depositors when the withdrawal is completed.

## Finding Description

**Root cause — `getAssetUnstaking()` ignores `slashingFactor`**

`getAssetUnstaking()` calls `_getDelegationManager().getQueuedWithdrawals(address(this))` and uses the returned `withdrawalShares` directly:

```solidity
uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
amount += strategyAsset == LRTConstants.ETH_TOKEN
    ? sharesToUnstake
    : strategy.sharesToUnderlyingView(sharesToUnstake);
``` [1](#0-0) 

Per the `Withdrawal` struct in `IDelegationManager.sol`, the stored field is `scaledShares` — raw values that must be multiplied by the operator's current `maxMagnitude * beaconChainScalingFactor` at completion time: [2](#0-1) 

`SlashingLib.scaleForCompleteWithdrawal()` makes the required multiplication explicit: [3](#0-2) 

`getAssetUnstaking()` never calls this; it passes raw `scaledShares` straight into `strategy.sharesToUnderlyingView()`, returning a pre-slash token amount even after the operator has been slashed.

**Contrast with `getAssetBalance()`**

`NodeDelegatorHelper.getAssetBalance()` correctly calls `getWithdrawableShares()` → `DelegationManager.getWithdrawableShares()`, which applies the current `slashingFactor` internally: [4](#0-3) 

**Propagation through accounting**

`LRTDepositPool.getAssetDistributionData()` sums both values: [5](#0-4) 

`getTotalAssetDeposits()` adds them together: [6](#0-5) 

`LRTOracle._getTotalEthInProtocol()` uses this inflated total: [7](#0-6) 

`updateRSETHPrice()` is public and callable by anyone: [8](#0-7) 

**Withdrawal commitment at inflated price**

`initiateWithdrawal()` locks in `expectedAssetAmount` using the current (inflated) stored `rsETHPrice`: [9](#0-8) 

`getAvailableAssetAmount()` also uses `getTotalAssetDeposits()`, so the over-commitment guard is bypassed: [10](#0-9) 

**`unlockQueue()` uses stored (inflated) price**

`_createUnlockParams()` reads `lrtOracle.rsETHPrice()` (the stored, already-inflated value) and `unstakingVault.balanceOf(asset)` (actual vault balance, which includes other users' funds): [11](#0-10) 

`_calculatePayoutAmount()` takes the minimum of `expectedAssetAmount` and `currentReturn`. Since both are computed at the same inflated price, the payout equals the inflated `expectedAssetAmount`: [12](#0-11) 

When `completeUnstaking()` later settles with EigenLayer, the vault receives only `scaledShares × slashingFactor` tokens — less than what was committed — and the shortfall is covered by other depositors' funds already in the vault.

**Circuit-breaker insufficient**

The price-decrease circuit-breaker compares the computed (inflated) price against `highestRsethPrice`. Because `getAssetUnstaking()` inflates the total, the computed price is higher than actual, making the circuit-breaker less likely to fire for moderate slashes: [13](#0-12) 

## Impact Explanation

**Critical — Direct theft of at-rest user funds.**

After a slash, the attacker commits to receiving more asset than the vault will actually receive from EigenLayer. The shortfall is covered by other depositors' funds already sitting in the unstaking vault. The invariant `sum(tokens_received_by_all_withdrawers) ≤ sum(tokens_ever_deposited)` breaks. This matches the allowed Critical impact: "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation

- EigenLayer operator slashing is a live, permissionless protocol event requiring no admin compromise.
- `updateRSETHPrice()` is public; any EOA can call it immediately after a slash to lock in the inflated price.
- `initiateWithdrawal()` is public; any rsETH holder can exploit the inflated price.
- The `getAvailableAssetAmount` guard uses the same inflated accounting and does not prevent over-commitment.
- The circuit-breaker is less likely to fire for moderate slashes precisely because the inflation masks the true price drop.
- The EigenLayer withdrawal delay (typically 7 days) provides a window during which the price remains inflated and `unlockQueue()` can be called at the inflated price.

## Recommendation

Apply the current `slashingFactor` inside `getAssetUnstaking()`. For each queued withdrawal, retrieve the operator's current `maxMagnitude` from the `AllocationManager` and multiply `scaledShares` by it before converting to underlying tokens — mirroring what `SlashingLib.scaleForCompleteWithdrawal()` does at completion time. Alternatively, call `DelegationManager.getWithdrawableShares()` for the queued withdrawal strategies and use those already-adjusted values, consistent with how `getAssetBalance()` is implemented in `NodeDelegatorHelper`.

## Proof of Concept

```
Setup (fork EigenLayer with slashing support):
  1. Deploy NDC, delegate to operator O.
  2. Deposit 100 stETH into strategy; NDC holds 100 shares.
  3. NDC.initiateUnstaking(strategy, 100 shares)
     → queued withdrawal with scaledShares = 100.
  4. Slash operator O by 20% → slashingFactor = 0.8e18.
     EigenLayer burns 20 shares; vault will receive only 80 stETH on completion.

Accounting check:
  - getAssetBalance(stETH)   → 0   (shares removed from strategy on queue)
  - getAssetUnstaking(stETH) → 100 stETH  (raw scaledShares, no slashingFactor)
  - getTotalAssetDeposits()  → 100 stETH  (inflated; actual = 80)

Attack:
  5. Attacker calls LRTOracle.updateRSETHPrice()
     → totalETHInProtocol = 100 (inflated) → rsETHPrice inflated.
  6. Attacker calls LRTWithdrawalManager.initiateWithdrawal(stETH, rsETHAmount)
     → expectedAssetAmount = 100 stETH committed.
  7. Operator calls NDC.completeUnstaking()
     → vault receives 80 stETH from EigenLayer.
     → vault also holds 20 stETH from other depositors (total 100 stETH).
  8. Operator calls unlockQueue() at still-inflated price
     → _calculatePayoutAmount returns 100 stETH (min of 100, 100).
  9. Attacker calls completeWithdrawal()
     → receives 100 stETH, draining 20 stETH belonging to other users.

Assert: attacker received 100 stETH; other users are short 20 stETH.
Invariant broken: sum(received) = 100 > sum(actual_vault_assets_from_attacker) = 80.
```
