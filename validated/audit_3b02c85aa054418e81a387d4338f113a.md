Audit Report

## Title
Stale Cached `rsETHPrice` Allows Depositors to Mint Excess rsETH, Stealing Yield from Existing Holders - (File: contracts/LRTDepositPool.sol)

## Summary
`getRsETHAmountToMint` divides a live Chainlink asset price by a cached storage variable `rsETHPrice` that is only updated on explicit calls to `updateRSETHPrice()`. Because `depositAsset` and `depositETH` never call `updateRSETHPrice()` before minting, any depositor who acts after the Chainlink stETH/ETH price has moved (e.g., after a stETH rebase) but before the next `updateRSETHPrice()` call receives more rsETH than their deposit warrants, permanently diluting prior holders' unclaimed yield.

## Finding Description
`getRsETHAmountToMint` in `LRTDepositPool.sol` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, which for stETH calls `ChainlinkPriceOracle.getAssetPrice`, reading `latestRoundData()` from the live Chainlink stETH/ETH feed. This value updates whenever the feed's deviation threshold or heartbeat triggers â€” including after stETH rebases.

`lrtOracle.rsETHPrice()` reads the storage variable `uint256 public override rsETHPrice` in `LRTOracle`, which is only written inside `_updateRsETHPrice()`. That function is only reachable via the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`. Neither `depositETH` nor `depositAsset` calls either function before computing `rsethAmountToMint`.

`_updateRsETHPrice()` computes the correct price as `totalETHInProtocol / rsethSupply`, where `totalETHInProtocol` is derived from live `getAssetPrice` calls and live `balanceOf` reads. After a stETH rebase, the Chainlink stETH/ETH rate increases, so `getAssetPrice(stETH)` returns a higher value than it did at the last `rsETHPrice` update. The numerator of the minting formula therefore reflects the post-rebase rate while the denominator remains at the pre-rebase cached price, causing the depositor to receive excess rsETH.

No existing guard prevents this: `_beforeDeposit` only checks deposit limits and a minimum rsETH slippage floor (`minRSETHAmountExpected`), neither of which constrains the stale-price over-mint.

## Impact Explanation
**High â€” Theft of unclaimed yield.**

When `rsETHPrice` is stale (lower than the true current price), new depositors receive more rsETH than their contribution warrants. When `updateRSETHPrice()` is subsequently called, the new price is computed over the inflated supply, permanently reducing the ETH-per-rsETH ratio for all prior holders. The difference represents yield that had accrued to existing holders but was transferred to the new depositor. This matches the allowed impact "Theft of unclaimed yield."

## Likelihood Explanation
stETH rebases approximately every 24 hours. `updateRSETHPrice()` is a separate public transaction that must be submitted and mined independently of any deposit. The window between a Chainlink stETH/ETH price update (triggered by the rebase) and the next `updateRSETHPrice()` call is a recurring, predictable condition requiring no special permissions, no front-running, and no victim mistakes. Any unprivileged user calling `depositAsset(stETH, ...)` during this window exploits the stale price. The condition repeats daily and the magnitude scales with deposit size.

## Recommendation
Call `_updateRsETHPrice()` (or an equivalent internal refresh) at the start of `_beforeDeposit` before computing `rsethAmountToMint`, ensuring the price used for minting always reflects the current TVL. Alternatively, compute `rsETHPrice` on-the-fly within `getRsETHAmountToMint` using the same `_getTotalEthInProtocol()` logic rather than reading the cached storage variable.

## Proof of Concept
1. Protocol state: 100 stETH deposited, 100 rsETH minted, `rsETHPrice = 1e18`.
2. stETH rebases; Chainlink stETH/ETH feed updates to `1.01e18`. Actual rsETH price = `101/100 = 1.01e18`. `rsETHPrice` in storage remains `1e18`.
3. Attacker calls `depositAsset(stETH, 10e18, 0, "")`.
4. `_beforeDeposit` calls `getRsETHAmountToMint(stETH, 10e18)`:
   - `getAssetPrice(stETH)` â†’ `1.01e18` (live Chainlink)
   - `rsETHPrice()` â†’ `1e18` (stale storage)
   - minted = `10e18 * 1.01e18 / 1e18 = 10.1e18` rsETH
5. Correct amount: `10e18 * 1.01e18 / 1.01e18 = 10e18` rsETH. Attacker receives `0.1e18` excess rsETH.
6. `updateRSETHPrice()` is called: TVL = `111.1 ETH`, supply = `110.1 rsETH`, new price = `~1.009e18`.
7. Original 100 rsETH holders now hold `100 * 1.009e18 â‰ˆ 100.9 ETH` instead of the `101 ETH` they were entitled to â€” ~0.1 ETH of yield stolen.
