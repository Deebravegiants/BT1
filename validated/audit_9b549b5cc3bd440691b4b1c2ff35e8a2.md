Audit Report

## Title
Unchecked Chainlink Oracle Response: Missing Staleness and Validity Checks Allow Stale/Zero Price to Corrupt rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but silently discards `updatedAt`, `answeredInRound`, and `roundId`, and performs no `price > 0` check. A stale or zero Chainlink price is accepted and propagated directly into rsETH minting calculations in `LRTDepositPool.getRsETHAmountToMint`, enabling an attacker to mint excess rsETH against depegged collateral during an oracle staleness window, causing protocol insolvency and dilution of all rsETH holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice` fetches the price as:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All three validation fields — `updatedAt` (staleness), `answeredInRound`, and `roundId` — are discarded. No check is made that `price > 0`.

The same codebase's `ChainlinkOracleForRSETHPoolCollateral.getRate()` correctly validates all three:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The stale price flows directly into rsETH minting:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` resolves to `ChainlinkPriceOracle.getAssetPrice` for Chainlink-backed assets. The `minRSETHAmountExpected` slippage guard in `_beforeDeposit` (L667) provides no protection because the attacker sets it themselves and benefits from the inflated rate.

## Impact Explanation

**Critical — Protocol insolvency / direct theft from rsETH holders:** When the Chainlink oracle goes stale while an LST depegs, the last reported (pre-depeg) price remains in effect. An attacker deposits depegged LST and receives rsETH computed at the stale inflated price, minting excess rsETH backed by less-than-equivalent collateral. This dilutes all existing rsETH holders and renders the protocol insolvent. This matches the allowed impact: *Direct theft of any user funds* and *Protocol insolvency*.

**Medium — Temporary freezing of funds:** If `price == 0` is returned, `getAssetPrice` returns 0, causing `getRsETHAmountToMint` to return 0, which causes `_beforeDeposit` to revert on `minRSETHAmountExpected` (if non-zero) or mint 0 rsETH while transferring assets in. This matches *Temporary freezing of funds*.

## Likelihood Explanation

Chainlink oracle staleness is a documented, historically observed failure mode (network congestion, sequencer downtime on L2, oracle deprecation). No attacker action is required to cause staleness — it occurs passively when the oracle fails to update within its heartbeat window. The attacker only needs to submit a normal `depositAsset` call during the stale window with a depegged LST. The attack is repeatable for every supported Chainlink-backed asset and requires no special privileges.

## Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
// Optional: if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

## Proof of Concept

1. Chainlink oracle for stETH/ETH goes stale at price `1e18` (1:1 peg) due to network congestion.
2. stETH depegs to `0.5e18` on-chain (real market price).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1e18`.
5. `rsethAmountToMint = (1000e18 * 1e18) / rsETHPrice` — computed at the stale 1:1 rate.
6. Attacker receives rsETH worth ~1000 ETH while depositing stETH worth only ~500 ETH.
7. Existing rsETH holders are diluted; protocol becomes insolvent by ~500 ETH.

**Foundry fork test plan:**
- Fork mainnet at a block where a Chainlink feed has not updated for > heartbeat.
- Deploy/use existing `ChainlinkPriceOracle` pointed at the stale feed.
- Call `getAssetPrice` and assert it returns the stale value without reverting.
- Call `depositAsset` with the depegged asset and assert rsETH minted exceeds fair value.
- Assert `answeredInRound < roundId` or `block.timestamp - updatedAt > heartbeat` holds, confirming staleness. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
