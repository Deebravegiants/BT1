### Title
Empty Extractor List Bypasses Foreign-Chain Value Verification, Enabling Forged Attestation - (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`)

### Summary

The `verify_foreign_transaction` flow accepts a user-supplied `extractors: vec![]` (empty list) without any minimum-length enforcement at the contract or node level. This allows any unprivileged caller to obtain a valid MPC threshold signature over a foreign-chain transaction that attests to **zero** observed values — a meaningless attestation. Consuming bridge contracts that do not independently validate the presence of expected extracted values in the response can be tricked into accepting a forged attestation, enabling invalid bridge execution.

### Finding Description

The `verify_foreign_transaction` entry point on the contract accepts a `VerifyForeignTransactionRequestArgs` whose inner `ForeignChainRpcRequest` variants all carry an `extractors` field typed as a plain `Vec<_>` with no lower-bound constraint:

```rust
pub struct EvmRpcRequest {
    pub tx_id: EvmTxId,
    pub extractors: Vec<EvmExtractor>,   // plain Vec — zero is valid
    pub finality: EvmFinality,
}
pub struct BitcoinRpcRequest {
    pub tx_id: BitcoinTxId,
    pub confirmations: BlockConfirmations,
    pub extractors: Vec<BitcoinExtractor>, // plain Vec — zero is valid
}
``` [1](#0-0) 

The contract's `verify_foreign_transaction` method performs no validation of the extractor count before enqueuing the request: [2](#0-1) 

On the node side, `execute_foreign_chain_request` passes the user-supplied extractor list directly to the chain inspector without checking that it is non-empty: [3](#0-2) 

The inspectors are explicitly designed to return an empty `Vec<ExtractedValue>` when given an empty extractor list — confirmed by first-party tests: [4](#0-3) [5](#0-4) [6](#0-5) 

The resulting signed payload is:

```rust
ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: <original request with extractors: []>,
    values: vec![],   // nothing extracted
})
``` [7](#0-6) 

The MPC network then signs `SHA-256(borsh(this payload))` and returns the signature on-chain: [8](#0-7) 

### Impact Explanation

The stated primary use case of `verify_foreign_transaction` is the **Omnibridge inbound flow** — a bridge contract on NEAR that must verify a foreign-chain deposit event (amount, token, recipient) before minting or releasing funds. The MPC signature is the trust anchor for this verification.

When an attacker submits `extractors: []`, the MPC network produces a valid threshold signature that attests only: *"transaction `tx_id` exists on chain `C` with the required finality."* It does **not** attest to any transfer amount, recipient address, token type, or log event. If the consuming bridge contract:

1. Verifies the MPC signature is valid ✓
2. Checks `payload_hash` matches the reconstructed payload ✓
3. **Fails to check** that `values` contains the expected extractor results ✗

…then the attacker can present a valid MPC-signed attestation for any real transaction on the supported chain, with no verified content, and trigger an invalid bridge execution (e.g., minting tokens without a real deposit, or double-spending by reusing a transaction ID with a different extractor set in a separate legitimate request).

This maps to: **High — forged foreign-chain verification that causes invalid bridge execution.**

### Likelihood Explanation

- **Entry path**: Any unprivileged NEAR account can call `verify_foreign_transaction` with `extractors: []`. No special role or collusion is required.
- **Cost**: Only the standard 1 yoctoNEAR deposit plus gas.
- **Precondition**: The attacker needs a real transaction ID on a supported chain (trivially obtained — any confirmed transaction qualifies).
- **Exploitation condition**: A consuming bridge contract that does not validate the `values` array. Given that the MPC SDK's own `ForeignChainRequestBuilder` explicitly supports building requests with zero extractors (confirmed by SDK tests), this is a realistic developer mistake. [9](#0-8) 

### Recommendation

1. **Enforce a minimum of one extractor at the contract level** in `verify_foreign_transaction`. Reject any `ForeignChainRpcRequest` whose `extractors` list is empty with a clear error message.

2. **Enforce the same check at the node level** in `execute_foreign_chain_request` as a defense-in-depth measure, so a node never signs a payload with an empty `values` vector.

3. **Type-level enforcement**: Change the `extractors` field in all `*RpcRequest` structs from `Vec<_>` to a `NonEmptyBoundedVec` (the codebase already has `NonEmptyBTreeSet` and `NonEmptyBTreeMap` in `near-mpc-bounded-collections`), so the constraint is enforced at deserialization time.

4. **SDK guidance**: Update `ForeignChainRequestBuilder` to require at least one extractor before `build()` succeeds, preventing downstream developers from accidentally constructing empty-extractor requests.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
