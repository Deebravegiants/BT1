### Title
Stale TEE-vote State from Removed Participants Bypasses Threshold Requirement in `vote_code_hash` / `vote_add_launcher_hash` — (`crates/contract/src/lib.rs`)

---

### Summary

After a resharing that removes one or more participants, their pre-resharing votes for TEE image hashes and launcher hashes remain in contract storage. The cleanup is performed by a **detached, fire-and-forget promise** (`clean_tee_status`). If that promise fails, the stale votes are never removed. Unlike `vote_update`, which explicitly re-filters votes against the current participant set on every call, `vote_code_hash` and `vote_add_launcher_hash` count **all** entries in the vote map — including stale entries from removed participants — when checking whether the governance threshold has been reached. A single colluding participant who is about to be removed can therefore pre-cast a vote for a malicious TEE image hash, and after resharing only `T-1` current participants (instead of `T`) are needed to whitelist it.

---

### Finding Description

**Root cause — no per-call participant filtering in TEE vote functions**

`vote_update` (the contract-upgrade vote) was hardened with an explicit re-filter:

```rust
// crates/contract/src/lib.rs  ~line 1362
// Filter votes to only count current participants voting for this specific update.
// This ensures correctness even if the cleanup promise in MpcContract::vote_reshared() fails.
let valid_votes_count = running_state
    .parameters
    .participants()
    .participants()
    .iter()
    .filter(|(account_id, _, _)| {
        self.proposed_updates
            .vote_by_participant
            .get(account_id)
            .is_some_and(|voted_id| *voted_id == id)
    })
    .count();
```

`vote_code_hash` and `vote_add_launcher_hash` have **no equivalent filter**:

```rust
// crates/contract/src/lib.rs  ~line 1418
let votes = self.tee_state.vote(code_hash, &participant);
// `votes` is the raw count of ALL entries in the map, including stale ones
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
```

```rust
// crates/contract/src/lib.rs  ~line 1452
let votes = self.tee_state.vote_launcher(action, &participant);
if votes >= self.threshold()?.value() {
    let added = self.tee_state.add_launcher_image(launcher_hash, tee_upgrade_deadline_duration);
}
```

The underlying `count_votes` helper in `LauncherHashVotes` and the analogous counter in `CodeHashesVotes` iterate the entire `vote_by_account` map without checking whether each voter is still a current participant:

```rust
// crates/contract/src/tee/proposal.rs  ~line 112
fn count_votes(&self, action: &LauncherVoteAction) -> u64 {
    u64::try_from(
        self.vote_by_account
            .values()
            .filter(|a| *a == action)
            .count(),
    )
    .expect("participant count should not overflow u64")
}
```

**Cleanup is best-effort only**

After resharing, `vote_reshared` spawns `clean_tee_status` as a detached promise:

```rust
// crates/contract/src/lib.rs  ~line 1186
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        vec![],
        NearToken::from_yoctonear(0),
        Gas::from_tgas(self.config.clean_tee_status_tera_gas),
    )
    .detach();   // fire-and-forget; failure is silent
```

The `.detach()` call means the resharing transaction succeeds regardless of whether the cleanup promise succeeds. If the gas budget (`clean_tee_status_tera_gas`) is insufficient, or if the promise fails for any other reason, the stale votes remain in storage indefinitely.

The developers themselves documented this risk for `vote_update` and added the re-filter as a defense-in-depth measure. No equivalent defense exists for the TEE vote functions.

**The test suite confirms the attack**

The unit test `test_clean_non_participant_votes_removes_stale_votes` (tee_state.rs ~line 1496) explicitly demonstrates that without cleanup, a removed participant's vote still counts:

```rust
// P2 votes for the same malicious hash — should be only 1 vote, not 3
let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
```

The comment "not 3" confirms that without cleanup the stale votes from P0 and P1 would be counted.

---

### Impact Explanation

A malicious TEE image hash whitelisted with fewer current-participant votes than the governance threshold requires undermines the TEE enforcement layer. Nodes running the malicious image can submit valid attestations, be accepted as participants, and participate in threshold signing rounds. This breaks the production safety invariant that only nodes running verified, governance-approved code may join the MPC network. The impact is classified **Medium**: it is a participant-state and contract execution-flow manipulation that breaks a production safety/accounting invariant (the TEE governance threshold) without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

The attack requires:
1. A participant who knows they will be removed (or who can influence resharing) to pre-cast a vote for a malicious hash.
2. The `clean_tee_status` detached promise to fail (e.g., due to an under-configured `clean_tee_status_tera_gas` value, or a transient runtime error).
3. `T-1` remaining current participants to vote for the same hash (either colluding or deceived).

Step 2 is the weakest link but is realistic: the gas budget is operator-configurable and the promise is fire-and-forget. The developers already acknowledged this failure mode for `vote_update` and added a compensating filter there, confirming the scenario is considered realistic.

---

### Recommendation

Apply the same defense-in-depth pattern used in `vote_update` to `vote_code_hash`, `vote_add_launcher_hash`, `vote_remove_launcher_hash`, and `vote_add_os_measurement` / `vote_remove_os_measurement`: before comparing the vote count against the threshold, re-filter the vote map to count only entries whose `AuthenticatedParticipantId` corresponds to a current participant. This makes the threshold check correct regardless of whether the post-resharing cleanup promise succeeded.

---

### Proof of Concept

**Setup**: N=5, T=3. Participants: {P0, P1, P2, P3, P4}.

1. **Pre-resharing**: P0 (malicious, knows it will be removed) calls `vote_code_hash(malicious_hash)`. The vote is recorded in `tee_state.votes.proposal_by_account` under P0's `AuthenticatedParticipantId`. Vote count = 1.

2. **Resharing**: Governance votes to remove P0. `vote_reshared` completes, transitioning to Running with {P1, P2, P3, P4}, T=3. `clean_tee_status` is spawned as a detached promise but fails (e.g., gas exhausted). P0's vote remains in storage.

3. **Post-resharing**: P1 calls `vote_code_hash(malicious_hash)`. `count_votes` iterates the full map: finds P0's stale entry + P1's new entry → returns 2.

4. P2 calls `vote_code_hash(malicious_hash)`. `count_votes` returns 3 (P0 stale + P1 + P2). `3 >= threshold(3)` → `whitelist_tee_proposal` is called. The malicious hash is whitelisted with only **2 current-participant votes** instead of the required 3.

5. Nodes running the malicious image submit attestations, are accepted by `verify_tee`, and join the MPC network. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {

            code_hash,

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;

        // update the state
        if votes >= self.threshold()?.value() {
            self.tee_state
                .whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
        }

        Ok(())
    }

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
