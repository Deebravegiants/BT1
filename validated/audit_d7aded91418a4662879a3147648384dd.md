### Title
Unguarded `init` Allows Any Caller to Seize Full MPC Participant Control Before Legitimate Initialization - (File: crates/contract/src/lib.rs)

### Summary

The `init` function of the MPC contract carries no access-control modifier (`#[private]`). Combined with the production deployment scripts that explicitly deploy the contract *without* an init call (`without-init-call`), there is a window in which any unprivileged NEAR account can race to call `init` with attacker-controlled parameters, installing themselves as the sole participant with a threshold of 1 and seizing complete control of the MPC signing network.

### Finding Description

The `init` function is the sole entry point that transitions the contract from `ProtocolContractState::NotInitialized` to `ProtocolContractState::Running`. [1](#0-0) 

It carries `#[init]` (which only prevents a *second* call once state exists) but **no** `#[private]` attribute. `#[private]` in the NEAR SDK restricts a method so that only the contract account itself can invoke it. Without it, the function is callable by any NEAR account. [2](#0-1) 

The production and localnet deployment scripts deploy the contract binary in a separate step from initialization, using `without-init-call`: [3](#0-2) 

The `init` call is then issued in a subsequent transaction: [4](#0-3) 

This two-step pattern is also present in `scripts/launch-localnet.sh`: [5](#0-4) 

Between the deploy transaction and the legitimate `init` transaction, the contract sits in `ProtocolContractState::NotInitialized`. [6](#0-5) 

An attacker who monitors the chain for the deployment event can immediately submit their own `init` call with:
- `parameters.participants` = a single entry pointing to the attacker's account
- `parameters.threshold` = 1

Because `#[init]` prevents re-initialization, the legitimate operator's subsequent `init` call will fail. The attacker is now the only recognized participant.

### Impact Explanation

Once the attacker controls the sole participant slot with threshold 1, they can:

1. **Issue unauthorized threshold signatures** — call `vote_add_domains` to add signing domains, then have MPC nodes (which they control) call `respond` to resolve any signature request they choose. This constitutes unauthorized threshold signature issuance over all foreign chains (Bitcoin, Ethereum, etc.) managed by the MPC network.

2. **Permanently freeze legitimate use** — the legitimate operator cannot re-initialize, cannot add themselves as a participant, and cannot recover the contract without a full redeployment. All pending and future user signature requests are either hijacked or permanently frozen.

3. **Drain funds** — by issuing arbitrary signatures for foreign-chain transactions, the attacker can redirect any assets held in MPC-controlled addresses on Bitcoin, Ethereum, or any other supported chain.

This matches the **Critical** impact class: *Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization* and *Theft, direct loss, or permanent freezing of funds controlled by the MPC network*.

### Likelihood Explanation

- The attack requires no special privilege — any NEAR account suffices.
- The deployment scripts make the two-step deploy-then-init pattern the standard procedure, guaranteeing the vulnerable window exists on every production deployment.
- Monitoring NEAR for a new contract deployment at a known account ID is trivial with any NEAR indexer.
- The attacker's `init` call is a simple JSON transaction with no deposit requirement.

### Recommendation

1. **Add `#[private]` to `init`** so only the contract account itself can call it. The contract account can then call `init` atomically in the same deployment transaction.

2. **Deploy with an init call** — replace `without-init-call` with an inline init in all deployment scripts, eliminating the uninitialized window entirely.

3. As a defense-in-depth measure, validate inside `init` that `env::predecessor_account_id() == env::current_account_id()`, mirroring the pattern already used by `init_running`: [7](#0-6)

# Step 4 – Attacker's node is the only participant; threshold = 1
# Step 5 – Attacker calls vote_add_domains, then responds to any sign request,
#           issuing arbitrary signatures for foreign-chain transactions.

    #[handle_result]
    #[init]
    pub fn init(
        parameters: dtos::ThresholdParameters,
        init_config: Option<dtos::InitConfig>,
    ) -> Result<Self, Error> {
        let parameters: ThresholdParameters = parameters.try_into_contract_type()?;
        // Log participant count and hash - full parameters exceed NEAR's 16KB log limit at ~100 participants
        let params_hash = env::sha256_array(borsh::to_vec(&parameters).unwrap());
        log!(
            "init: signer={}, num_participants={}, parameters_hash={:?}, init_config={:?}",
            env::signer_account_id(),
            parameters.participants().len(),
            params_hash,
            init_config,
        );
        parameters.validate()?;

        // TODO(#1087): Every participant must have a valid attestation, otherwise we risk
        // participants being immediately kicked out once contract transitions into running.
        let initial_participants = parameters.participants();
        let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);

        Ok(Self {
            protocol_state: ProtocolContractState::Running(RunningContractState::new(
                DomainRegistry::default(),
                Keyset::new(EpochId::new(0), Vec::new()),
                parameters,
                AddDomainsVotes::default(),
            )),
            pending_signature_requests: LookupMap::new(StorageKey::PendingSignatureRequestsV4),
            pending_ckd_requests: LookupMap::new(StorageKey::PendingCKDRequestsV3),
            pending_verify_foreign_tx_requests: LookupMap::new(
                StorageKey::PendingVerifyForeignTxRequestsV2,
            ),
            proposed_updates: ProposedUpdates::default(),
            config: init_config.map(Into::into).unwrap_or_default(),
            tee_state,
            accept_requests: true,
            node_migrations: NodeMigrations::default(),
            metrics: Default::default(),
            node_foreign_chain_support: Default::default(),

                StorageKey::ForeignChainMetadata,

            tee_verifier_account_id: None,

    }
