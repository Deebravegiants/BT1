Audit Report

## Title
Stale `rsETHPrice` Stored Variable Allows Depositors to Steal Accrued Yield from Existing rsETH Holders — (`contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary
`LRTOracle` stores the rsETH/ETH exchange rate in a public state variable `rsETHPrice` that is only updated when `updateRSETHPrice()` is explicitly called. `LRTDepositPool.getRsETHAmountToMint` reads this stale stored value directly when computing how many rsETH tokens to mint for a depositor. When staking rewards accrue between oracle updates, the stored price is lower than the true price, causing depositors to receive more rsETH than their deposit is worth — extracting accrued yield that belongs to existing holders.

## Finding Description
`LRTOracle` holds the exchange rate as a mutable state variable updated only on explicit calls:

```solidity
// LRTOracle.sol:28
uint256 public override rsETHPrice;
```

The only way to refresh it is via `updateRSETHPrice()` → `_updateRsETHPrice()`, which computes the true price using `_getTotalEthInProtocol()`. That helper is `private` with no public view equivalent, and `ILRTOracle` exposes only the stored value:

```solidity
// ILRTOracle.sol:31
function rsETHPrice() external view returns (uint256);
```

`LRTDepositPool.getRsETHAmountToMint` reads the stored (potentially stale) price:

```solidity
// LRTDepositPool.sol:520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

This is called unconditionally from `_beforeDeposit`, which is invoked by both `depositAsset` and `depositETH` with no prior call to `updateRSETHPrice()`. Neither function refreshes the price before computing the mint amount.

**Exploit path:**
1. Staking rewards accrue over time; true rsETH price rises to 1.005 ETH while stored `rsETHPrice` remains 1.00 ETH.
2. Attacker calls `depositAsset(stETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint` computes `(1000e18 × 1e18) / 1.00e18 = 1000 rsETH` instead of the fair `≈995.02 rsETH`.
4. Attacker receives ~4.98 excess rsETH representing accrued yield belonging to prior holders.
5. When `updateRSETHPrice()` is subsequently called, the new price reflects the diluted supply, permanently reducing the value of all pre-existing rsETH holders' positions.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` limits how much the price can jump per oracle update call, but it does not prevent deposits from using the stale price during the gap between updates. The `minRSETHAmountExpected` slippage parameter protects the depositor, not existing holders.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: **Theft of unclaimed yield**. Accrued staking yield that belongs to existing rsETH holders is captured by a new depositor who deposits against a stale (lower) rsETH price. The excess rsETH minted is backed by yield that was already earned by prior holders, permanently diluting their positions once the price is updated.

## Likelihood Explanation
`updateRSETHPrice()` is permissionless but is not called atomically with deposits. Oracle updates are triggered off-chain on a periodic schedule, creating a predictable and observable staleness window. The gap between the stored price and the true price is computable entirely from on-chain data (total asset balances × asset prices ÷ rsETH supply). No special access, flash loans, or privileged roles are required — any depositor can exploit this by timing a deposit during the staleness window. The window recurs after every oracle update cycle, making the attack repeatable.

## Recommendation
Add a public view function to `LRTOracle` that computes the current rsETH price on-the-fly by replicating the logic of `_getTotalEthInProtocol()` and the price formula in `_updateRsETHPrice()`, without writing state. Expose it through `ILRTOracle`. Then update `getRsETHAmountToMint` and `getExpectedAssetAmount` to call this live-price function instead of reading the stored `rsETHPrice`. Alternatively, call `updateRSETHPrice()` atomically at the start of `depositAsset`, `depositETH`, and `initiateWithdrawal` before any price-dependent calculation.

## Proof of Concept
**Setup (Foundry fork test):**
1. Fork mainnet at a block where rsETH has accrued staking rewards since the last `updateRSETHPrice()` call.
2. Record `storedPrice = lrtOracle.rsETHPrice()`.
3. Compute `truePrice = totalETHInProtocol / rsETH.totalSupply()` from on-chain balances.
4. Assert `truePrice > storedPrice` (staleness confirmed).

**Attack sequence:**
1. Attacker holds 1000 stETH (worth ~1000 ETH).
2. Attacker calls `lrtDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. Record `rsethMinted` from the `AssetDeposit` event.
4. Call `lrtOracle.updateRSETHPrice()` to refresh the price.
5. Compute `fairMint = (1000e18 * stETHPrice) / truePrice`.
6. Assert `rsethMinted > fairMint` — the difference is stolen yield.
7. Assert that existing holders' rsETH is now worth less per unit than before the deposit (price diluted).
