### Title
`respond` Checks Wrong Participant Set During Resharing, Blocking Valid Signature Submissions — (File: `crates/contract/src/state.rs`, `crates/contract/src/lib.rs`)

---

### Summary

During `Resharing` state, `respond`, `respond_ckd`, and `respond_verify_foreign_tx` call `assert_caller_is_attested_participant_and_protocol_active()`, which internally calls `self.protocol_state.active_participants()`. During resharing, `active_participants()` returns the **new (prospective)** participant set, not the **old (current key-holding)** participant set. Because only old participants hold the current key shares and can produce cryptographically valid signatures, this check is performed against the wrong participant set. Old participants being rotated out are blocked from calling `respond`, while new participants being rotated in cannot produce valid signatures (they have no key shares yet). The net effect is that all pending signature requests are silently unresolvable for the duration of any resharing that changes the participant set.

---

### Finding Description

`assert_caller_is_attested_participant_and_protocol_active` is the gate used by all three `respond*` methods:

```rust
// crates/contract/src/lib.rs ~2389
fn assert_caller_is_attested_participant_and_protocol_active(&self) {
    let participants = self.protocol_state.active_participants(); // <-- wrong set during Resharing
    Self::assert_caller_is_signer();
    let attestation_check = self.tee_state.is_caller_an_attested_participant(participants);
    assert_matches::assert_matches!(attestation_check, Ok(()), "Caller must be an attested participant");
}
```

`active_participants()` is defined as:

```rust
// crates/contract/src/state.rs ~255
pub fn active_participants(&self) -> &Participants {
    match self {
        ProtocolContractState::Resharing(state) => {
            state.resharing_key.proposed_parameters().participants() // NEW participants
        }
        ...
    }
}
```

The `respond` function explicitly permits the `Resharing` state:

```rust
// crates/contract/src/lib.rs ~575
if !self.protocol_state.is_running_or_resharing() {
    return Err(InvalidState::ProtocolStateNotRunning.into());
}
```

Yet the participant check immediately above it uses the **new** participant set. The correct set to check for `respond` during resharing is the **old** participant set (`previous_running_state.parameters.participants()`), which is exactly what `authenticate_update_vote` uses for governance operations during resharing:

```rust
// crates/contract/src/state.rs ~232
ProtocolContractState::Resharing(state) => {
    AuthenticatedParticipantId::new(
        state.previous_running_state.parameters.participants(), // OLD participants
    )?;
}
```

The inconsistency is structural: governance votes correctly use old participants during resharing, but `respond*` methods use new participants.

`is_caller_an_attested_participant` enforces membership in the passed `participants` set:

```rust
// crates/contract/src/tee/tee_state.rs ~476
let info = participants
    .info(&signer_id)
    .ok_or(AttestationCheckError::CallerNotParticipant)?;
```

An old participant not present in the new set will receive `CallerNotParticipant` and be rejected.

---

### Impact Explanation

**Medium.** During any resharing that rotates out one or more participants, the following holds for the entire resharing window:

- Old participants (who hold key shares and can produce valid signatures) are blocked by the wrong participant check.
- New participants (who pass the check) have no key shares yet and cannot produce cryptographically valid signatures — the signature verification in `respond` would reject them anyway.

The combined effect is that **no node can successfully call `respond`** during resharing with participant set changes. All pending `sign`, `request_app_private_key`, and `verify_foreign_transaction` requests will time out. This breaks the production safety invariant that the MPC network remains live and able to fulfill requests during resharing — a request-lifecycle manipulation that corrupts the contract's execution flow without requiring any network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

**Medium.** Resharing is a normal, planned operational event (adding/removing nodes, changing thresholds). Any governance-triggered resharing that changes the participant set triggers this condition. No adversarial action is required; the bug fires automatically on the next `vote_new_parameters` that alters membership.

---

### Recommendation

In `assert_caller_is_attested_participant_and_protocol_active`, when the protocol is in `Resharing` state, validate the caller against the **old** participant set (`previous_running_state.parameters.participants()`), not the new one. Alternatively, accept callers who are in **either** the old or new participant set during resharing, consistent with `is_existing_or_prospective_participant`. The governance path (`authenticate_update_vote`) already demonstrates the correct pattern for the resharing case.

---

### Proof of Concept

1. Contract is `Running` with participants `{A, B, C}`.
2. Governance votes trigger `vote_new_parameters` to replace `C` with `D`, transitioning to `Resharing`. New participant set = `{A, B, D}`.
3. A user submits a `sign` request. The request is queued.
4. Nodes `A`, `B`, `C` (old set, holding key shares) complete the threshold signing protocol and attempt to call `respond`.
5. `A` and `B` pass the check (they are in both old and new sets). `C` is rejected with `CallerNotParticipant` because `active_participants()` returns `{A, B, D}` and `C` is absent.
6. `D` (new participant) is in the new set and passes the participant check, but `D` has no key shares yet and cannot produce a valid signature — `respond` rejects `D` with `InvalidSignature`.
7. If the threshold requires all three of `{A, B, C}` (e.g., 3-of-3), no valid `respond` can be submitted. The request times out.
8. Even at lower thresholds, any scenario where removed participants are needed to meet the signing threshold causes the same failure. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```rust
    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
```
