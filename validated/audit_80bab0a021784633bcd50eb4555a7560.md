### Title
`lastUpdated` Reflects L2 Delivery Time Instead of L1 Rate Capture Time Due to Missing Timestamp in Payload — (`contracts/cross-chain/RSETHRateReceiver.sol` / `CrossChainRateReceiver.sol`)

### Summary
`CrossChainRateReceiver.lastUpdated` is set to `block.timestamp` on the destination chain when `lzReceive` is called. Because the L1 capture timestamp is never included in the LayerZero payload, any delay between rate capture on L1 and message delivery on L2 causes `lastUpdated` to misrepresent the actual age of the rate.

### Finding Description
In `CrossChainRateProvider.updateRate()`, the payload is constructed as:

```solidity
bytes memory _payload = abi.encode(latestRate);
``` [3](#0-2) 

The L1 `block.timestamp` at the moment of rate capture is **never serialized into the payload**. On the receiver side, `lzReceive` decodes only the rate and stamps `lastUpdated = block.timestamp` using the L2 clock at delivery time: [4](#0-3) 

If the LayerZero relayer queues the message for N hours (due to congestion, gas price spikes, or network issues), the receiver will store a rate that is N hours stale while `lastUpdated` appears fresh (equal to the recent L2 delivery time). There is no mechanism in the contract to detect or signal this discrepancy.

### Impact Explanation
Any integrator (on-chain or off-chain) that reads `lastUpdated` to determine whether the rsETH/ETH rate is fresh will receive a false signal. The rate could be hours old while `lastUpdated` is within the last block. This can lead to mispriced operations in downstream protocols (e.g., lending markets, vaults, or AMMs) that use this rate receiver as a price oracle and gate operations on `block.timestamp - lastUpdated < threshold`.

**Scope match:** Low — Contract fails to deliver promised returns (`lastUpdated` does not reflect when the rate was actually sampled), but no direct value loss occurs within the receiver contract itself.

### Likelihood Explanation
LayerZero message delays are a known operational reality. Congestion, relayer downtime, or insufficient gas fees on the destination chain can delay delivery by hours. This is not a theoretical edge case; it is a documented operational characteristic of cross-chain messaging. The precondition requires no attacker — it occurs passively under normal network stress.

### Recommendation
Include the L1 capture timestamp in the payload and use it on the receiver side:

**Provider (`CrossChainRateProvider.updateRate`):**
```solidity
bytes memory _payload = abi.encode(latestRate, block.timestamp);
```

**Receiver (`CrossChainRateReceiver.lzReceive`):**
```solidity
(uint256 _rate, uint256 _capturedAt) = abi.decode(_payload, (uint256, uint256));
rate = _rate;
lastUpdated = _capturedAt;  // L1 capture time, not L2 delivery time
```

This ensures `lastUpdated` accurately reflects when the rate was sampled on L1, regardless of relay delay.

### Proof of Concept
1. Deploy `RSETHRateProvider` on a fork of Ethereum mainnet.
2. Deploy `RSETHRateReceiver` on a fork of an L2 (e.g., Arbitrum).
3. Call `updateRate()` on the provider at L1 timestamp T0; record the encoded payload.
4. Simulate a relay delay by advancing L2 time by N hours before calling `lzReceive` with the same payload.
5. Assert: `receiver.lastUpdated() == T0 + N * 3600` (L2 delivery time) while the rate was captured at T0.
6. Assert: `receiver.lastUpdated() - T0 > acceptable_staleness_threshold` — the freshness invariant is broken. [5](#0-4) [6](#0-5)
