### Title
Sub-transactions bypass `maxTxExUnits` and `maxBlockExUnits` execution-unit limits in Dijkstra era — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The Dijkstra era introduces nested ("sub") transactions. The top-level UTXO rule enforces `totExUnits tx ≤ maxTxExUnits pp` and the BBODY rule enforces `∑ totExUnits txs ≤ maxBlockExUnits pp`. Neither check is applied to sub-transactions: the `SUBUTXO` rule omits `validateExUnitsTooBigUTxO` entirely, and the block-level `validateExUnits` uses `foldMap totExUnits txs` which only sums top-level redeemers. As a result, an unprivileged transaction sender can embed Plutus scripts with arbitrarily large ExUnits budgets inside sub-transactions, bypassing both per-transaction and per-block resource limits, and paying no script-execution fee for those scripts.

---

### Finding Description

**Top-level UTXO rule (`dijkstraUtxoTransition`)** enforces two resource limits:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs:414-415
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

and the minimum-fee check implicitly includes script fees via `txscriptfee prices (totExUnits tx)`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs:372-373
{- minfee pp txTop utxo₀ ≤ txfee txb -}
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
```

**Sub-transaction UTXO rule (`dijkstraSubUtxoTransition`)** applies no equivalent check. The entire `dijkstraSubUtxoTransition` body contains no call to `validateExUnitsTooBigUTxO`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs:231-278
dijkstraSubUtxoTransition = do
  ...
  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody
  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx
  runTest $ Alonzo.validateOutputTooBigUTxO pp allOutputs
  runTest $ Shelley.validateInputSetEmptyUTxO txBody
  ...
  -- NO validateExUnitsTooBigUTxO call
```

The `DijkstraSubUtxoPredFailure` type has no `ExUnitsTooBig` constructor, and the injection mapping explicitly marks it impossible:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs:340
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
```

**Block-level BBODY rule (`dijkstraBbodyTransition`)** calls:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs:361
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
```

which internally does `foldMap totExUnits txs`. The `totExUnits` function (defined in `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`) sums only the redeemers in the top-level transaction's own witness set. Sub-transaction redeemers live in separate per-sub-tx witness sets and are never folded in.

The sub-transaction body format explicitly supports a `mint` field and redeemers (field `? 9 : mint` and `? 5 : redeemers` in `sub_transaction_body`):

```
-- eras/dijkstra/impl/cddl/data/dijkstra.cddl:788-808
sub_transaction_body =
  {   0  : set<transaction_input>
  ,   1  : [* transaction_output]
  , ? 9  : mint
  , ? 11 : script_data_hash
  ...
  }
```

---

### Impact Explanation

An attacker can construct a top-level transaction with N sub-transactions, each containing Plutus scripts whose redeemers declare ExUnits = `(maxNatural, maxNatural)`. Because:

1. **`maxTxExUnits` is not checked per sub-tx**: each sub-tx can declare and execute scripts with ExUnits far exceeding the protocol limit.
2. **`maxBlockExUnits` is not aggregated over sub-txs**: the block-level check only sums top-level tx ExUnits, so a single block can contain unbounded sub-tx script work.
3. **Script execution fees are not charged for sub-txs**: `minfee` uses `totExUnits` of the top-level tx only; sub-tx script fees are never included in the required fee.

This maps to: **Medium — attacker-controlled transactions exceed intended validation limits and modify fees outside design parameters** (sub-tx script execution fees are zero regardless of declared ExUnits budget).

---

### Likelihood Explanation

**High.** Any unprivileged transaction sender can craft a top-level Dijkstra transaction with sub-transactions containing Plutus scripts. No special keys, governance role, or majority are required. The bypass is structural and deterministic: every honest node running the Dijkstra ledger rules will accept the transaction identically, so there is no divergence risk from the bypass itself — but the resource-limit and fee-accounting invariants are violated on every such transaction.

---

### Recommendation

1. In `dijkstraSubUtxoTransition`, add a per-sub-tx ExUnits check analogous to the top-level check:
   ```haskell
   runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
   ```
   and add a corresponding `SubExUnitsTooBigUTxO` constructor to `DijkstraSubUtxoPredFailure`.

2. In `dijkstraBbodyTransition`, replace `Alonzo.validateExUnits @era txs` with a Dijkstra-specific variant that aggregates ExUnits across both top-level and sub-transaction redeemers before comparing against `maxBlockExUnits`.

3. In the minimum-fee calculation (`minfee` / `getMinFeeTxUtxo`) for the Dijkstra era, include `totExUnits` of all sub-transactions so that script execution fees are charged for sub-tx scripts.

---

### Proof of Concept

```
Top-level tx:
  inputs:  [utxo_A]
  fee:     minFee(top-level tx with zero redeemers)   -- no sub-tx script fees charged
  sub_transactions:
    subTx_1:
      inputs: [utxo_B]
      redeemers: { Spend(0) -> (data, ExUnits(maxNat, maxNat)) }
      scripts:   { alwaysSucceeds_PlutusV3 }
    subTx_2:
      inputs: [utxo_C]
      redeemers: { Spend(0) -> (data, ExUnits(maxNat, maxNat)) }
      scripts:   { alwaysSucceeds_PlutusV3 }
    ... (repeat N times)
```

- `validateExUnitsTooBigUTxO` is called only on the top-level tx, which has zero redeemers → passes.
- Each `dijkstraSubUtxoTransition` invocation has no ExUnits check → all sub-txs pass.
- `Alonzo.validateExUnits` in BBODY sums `totExUnits` of the top-level tx only (= 0) → passes.
