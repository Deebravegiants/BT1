### Title
Broken Pigeonhole Receipt-Completeness Check in `set_state_header` Allows Peer to Omit Incoming Receipts, Causing Wrong State Root After State Sync — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

The `set_state_header` function relies on a Pigeonhole argument to guarantee that all incoming receipts for the synced shard are present in the peer-supplied `ShardStateSyncResponseHeader`. The argument is unsound: `from_shard_id` is not cryptographically bound to the Merkle proof, and the block-level Merkle tree position is not verified to be unique across proofs. A malicious peer can supply N receipt proofs all backed by a single shard's outgoing-receipts root (one that has no receipts to the target shard), with N distinct `from_shard_id` values, passing every validation check while omitting all actual incoming receipts. `set_state_finalize` then applies the empty receipt set, producing a state root that diverges from the canonical chain.

### Finding Description

In `set_state_header` (`chain/chain/src/state_sync/adapter.rs`, lines 438–503), step 4 validates `incoming_receipts_proofs` as follows:

- **4c** (line 464–465): `receipt_proofs.len() == block_header.chunks_included()` — count matches.
- **4d** (lines 479–486): `from_shard_id` values are unique across proofs.
- **4e** (line 490): `verify_path(*root, proof, &receipts_hash)` — receipts hash is a leaf in the outgoing-receipts Merkle tree rooted at `root`.
- **4f** (lines 495–499): `verify_path(*block_header.prev_chunk_outgoing_receipts_root(), block_proof, root)` — `root` is a leaf in the block-level Merkle tree. [1](#0-0) 

The comment at lines 470–474 states the Pigeonhole rationale. The flaw is that neither check 4d nor 4e/4f binds `from_shard_id` to the Merkle tree position. `verify_path` (line 113 in `core/primitives/src/merkle.rs`) only checks that the path leads to the root — it does not verify which leaf position is being proven. [2](#0-1) 

The `receipts_hash` is `hash(ReceiptList(target_shard_id, receipts))` — `from_shard_id` is absent from the hash: [3](#0-2) 

Compare with `ReceiptProof::verify_against_receipt_root` which also hashes only `to_shard_id`: [4](#0-3) 

The block-level Merkle tree is built from all chunks' `prev_outgoing_receipts_root` values indexed by shard index. The `block_proof` proves `root` is at some position in that tree, but the validation never checks that the position corresponds to `from_shard_id`. Two proofs with different `from_shard_id` values can share the same `root` and `block_proof` and both pass 4f.

### Impact Explanation

`set_state_finalize` reads the stored header from the DB and calls `collect_receipts_from_response` on the peer-supplied `incoming_receipts_proofs`, then passes the result directly to `apply_chunk`: [5](#0-4) 

No re-validation of receipts against canonical chain data occurs. The `apply_chunk` call at line 540 uses the attacker-controlled (empty) receipt list. The resulting `apply_result` state root is saved as the chunk extra via `apply_chunk_postprocessing` (line 551). There is no post-hoc check that the computed state root matches the canonical chain's expected state root. The node's state permanently diverges from the canonical chain for that shard. [6](#0-5) 

### Likelihood Explanation

State sync is a standard production code path triggered whenever a node falls behind. Any peer that the syncing node contacts for state sync data can serve a crafted `ShardStateSyncResponseHeader`. No validator or admin privileges are required — any network participant running a node can act as a state sync peer. The crafted proof requires only publicly available block data (chunk outgoing-receipts roots and their Merkle paths), which are derivable from synced block headers.

### Recommendation

Replace the Pigeonhole uniqueness check with a check that verifies each `from_shard_id` is the actual shard ID of the chunk at the corresponding Merkle tree position. Concretely, for each proof at index `j`, retrieve the block's chunk at shard index `j` (or the shard ID corresponding to the Merkle leaf position proven by `block_proof`) and assert `from_shard_id == chunk_at_position.shard_id()`. Alternatively, use `verify_path_with_index` (which already exists in `core/primitives/src/merkle.rs` at line 121) for the block-level proof to bind each `root` to a specific leaf index, then verify that index maps to `from_shard_id`. [7](#0-6) 

The stateless-validation path in `chunk_validation.rs` already does this correctly — it checks `receipt_proof.1.from_shard_id != from_chunk.shard_id()` explicitly: [8](#0-7) 

The state-sync path should apply the same discipline.

### Proof of Concept

Setup: 2 shards (shard 0, shard 1), target shard = shard 1, both chunks included (`chunks_included = 2`). Shard 0 has receipts `[r1, r2]` to shard 1. Shard 1 has no receipts to shard 1.

Canonical `incoming_receipts_proofs`:
- Proof 0: `from_shard_id=0`, `receipts=[r1,r2]`, `root=root_0`, `block_proof=path_to_root_0`, `proof=path_to_hash(ReceiptList(shard_1,[r1,r2]))` in shard 0's Merkle tree
- Proof 1: `from_shard_id=1`, `receipts=[]`, `root=root_1`, `block_proof=path_to_root_1`, `proof=path_to_hash(ReceiptList(shard_1,[]))` in shard 1's Merkle tree

Attacker's crafted `incoming_receipts_proofs` (both proofs use shard 1's root):
- Proof 0: `from_shard_id=0`, `receipts=[]`, `root=root_1`, `block_proof=path_to_root_1`, `proof=path_to_hash(ReceiptList(shard_1,[]))` in shard 1's Merkle tree
- Proof 1: `from_shard_id=1`, `receipts=[]`, `root=root_1`, `block_proof=path_to_root_1`, `proof=path_to_hash(ReceiptList(shard_1,[]))` in shard 1's Merkle tree

Validation results:
- 4c: `2 == chunks_included(2)` ✓
- 4d: `from_shard_id` values `{0,1}` are unique ✓
- 4e: `verify_path(root_1, proof, hash(ReceiptList(shard_1,[])))` ✓ (shard 1 has no receipts to shard 1, so this leaf exists)
- 4f: `verify_path(block_receipts_root, path_to_root_1, root_1)` ✓

`set_state_header` stores the crafted header. `set_state_finalize` applies zero receipts. The resulting state root differs from the canonical chain's state root (which applied `[r1, r2]`). The node diverges from consensus.
