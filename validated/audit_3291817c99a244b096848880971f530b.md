### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Breaks Bridge Backing Guarantees — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter — not the actual balance change — as the canonical bridged amount when locking native ERC-20 tokens. For fee-on-transfer tokens the vault receives `amount − transferFee`, yet the bridge emits and forwards `amount` to the destination chain, permanently undercollateralising the vault and eventually causing irreversible fund lock when redemptions are attempted.

---

### Finding Description

In `OmniBridge.sol::initTransfer`, the non-bridge, non-custom-minter ERC-20 path is:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  L406-L412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← requested amount, not measured balance change
    );
}
```

Immediately after, the same caller-supplied `amount` is forwarded verbatim to the cross-chain messaging layer and emitted in the event:

```solidity
// L415-L436
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // ← overstated for fee-on-transfer tokens
    fee, nativeFee, recipient, message, extensionValue
);

emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // ← same overstated value
    fee, nativeFee, recipient, message
);
``` [1](#0-0) [2](#0-1) 

No balance-before / balance-after measurement is performed. For fee-on-transfer tokens (e.g. tokens with built-in transfer taxes), `safeTransferFrom` delivers `amount − transferFee` to the bridge, but the bridge records and broadcasts `amount`.

The NEAR-side `fin_transfer_callback` reads the amount directly from the proof produced by that event and uses it without any independent verification:

```rust
// near/omni-bridge/src/lib.rs  L726-L736
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [3](#0-2) 

So the destination chain credits the full `amount` while the EVM vault holds only `amount − transferFee`.

The same structural flaw exists on the Solana side: `init_transfer.rs` passes `payload.amount` directly to `post_message` after `transfer_checked`, without measuring the vault's actual balance increase, which matters for Token-2022 mints that carry a transfer-fee extension. [4](#0-3) 

---

### Impact Explanation

**Backing guarantee broken (High/Critical).**

Every `initTransfer` with a fee-on-transfer token creates a shortfall: the EVM vault holds `Σ(amount_i − fee_i)` but the destination chain has credited `Σ(amount_i)`. When any user later bridges back and `finTransfer` is called on EVM:

```solidity
// OmniBridge.sol  L351-L354
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount   // ← exceeds actual vault balance
);
``` [5](#0-4) 

`safeTransfer` reverts because the vault is short. The last users to redeem cannot withdraw their funds — **irreversible fund lock** for the shortfall amount. Additionally, the destination chain has minted/released unbacked supply equal to the cumulative transfer-fee shortfall, breaking the 1:1 collateralisation invariant.

---

### Likelihood Explanation

- `initTransfer` is a public, permissionless entry point — any unprivileged user can call it with any ERC-20 token address.
- No whitelist or fee-on-transfer guard exists in the function.
- Fee-on-transfer tokens are a well-known, deployed token class (e.g. tokens with configurable transfer taxes, PAXG-style fee tokens, or any ERC-20 that deducts a fee inside `transfer`/`transferFrom`).
- A single legitimate user bridging such a token is sufficient to trigger the shortfall; no adversarial intent is required, making accidental triggering realistic.

---

### Recommendation

Measure the actual balance change instead of trusting the caller-supplied `amount`:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
    amount = uint128(actualReceived); // use actual received amount downstream
}
```

Apply the same pattern on Solana by reading the vault's token-account balance before and after `transfer_checked` and using the delta as the canonical amount posted to NEAR.

---

### Proof of Concept

1. Deploy or use any ERC-20 token that deducts a 1% fee on every `transferFrom`.
2. Register the token with the bridge (call `logMetadata`; wait for NEAR-side registration).
3. Call `initTransfer(tokenAddress, 1000e18, 0, 0, nearRecipient, "")`.
   - Bridge vault receives `990e18` (1% fee deducted).
   - `InitTransfer` event emits `amount = 1000e18`.
4. NEAR relayer picks up the event, proves it, and calls `fin_transfer` on NEAR — recipient receives credit for `1000e18`.
5. Repeat step 3 many times; vault shortfall accumulates.
6. When any user bridges `1000e18` back from NEAR to EVM, `finTransfer` calls `safeTransfer(recipient, 1000e18)` but the vault only holds `990e18` → **revert, funds permanently locked**.
