### Title
Permissionless `claimAndStake()` Forces Any User's KERNEL Tokens Into Locked Staking Without Consent - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

---

### Summary

`KernelMerkleDistributor.claimAndStake()` accepts a caller-supplied `account` parameter with no access control. Because merkle proofs are public off-chain data, any unprivileged actor can call `claimAndStake` for an arbitrary victim, forcing that victim's unclaimed KERNEL allocation to be staked in `KernelDepositPool` — subject to a mandatory withdrawal delay of up to 30 days — without the victim's consent.

---

### Finding Description

`KernelMerkleDistributor.claimAndStake` is declared `external` with only `nonReentrant` and `whenNotPaused` guards:

```solidity
// contracts/KERNEL/KernelMerkleDistributor.sol
function claimAndStake(
    uint256 index,
    address account,          // ← attacker-supplied victim address
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
)
    external
    nonReentrant
    whenNotPaused
{
    uint256 amountToStake = _processClaim(index, account, cumulativeAmount, merkleProof);
    IKernelDepositPool(kernelDepositPool).stakeFor(account, amountToStake);
    emit ClaimedAndStaked(index, account, amountToStake);
}
```

`_processClaim` verifies the merkle proof against `currentMerkleRoot` and marks the claim as consumed for `account`. The proof data (`index`, `account`, `cumulativeAmount`, `merkleProof`) is published publicly so that users can self-claim; any observer can replay it against `claimAndStake` instead of `claim`.

After `_processClaim` succeeds, `stakeFor(account, amountToStake)` is called on `KernelDepositPool`. `KernelMerkleDistributor` holds `STAKE_FOR_ROLE`, so the call succeeds and the victim's tokens are deposited into the staking contract under the victim's balance:

```solidity
// contracts/KERNEL/KernelDepositPool.sol
function stakeFor(address _account, uint256 _amount)
    external
    nonReentrant
    onlyRole(STAKE_FOR_ROLE)
    updateReward(_account)
{
    ...
    balanceOf[_account] += _amount;
    totalKernelStaked += _amount;
    kernelToken.safeTransferFrom(msg.sender, address(this), _amount);
    ...
}
```

To recover the tokens, the victim must call `initiateWithdrawal`, then wait for `withdrawalDelay` (admin-configurable, up to `MAX_WITHDRAWAL_DELAY = 30 days`), then call `claimWithdrawal`.

By contrast, `KernelTop100MerkleDistributor.claimAndStake` correctly uses `msg.sender` as the beneficiary, preventing third-party invocation:

```solidity
// contracts/KERNEL/KernelTop100MerkleDistributor.sol
function claimAndStake(uint256 amount, bytes32[] calldata merkleProof)
    external nonReentrant whenNotPaused
{
    address user = msg.sender;   // ← no caller-supplied account
    ...
}
```

The inconsistency between the two distributors confirms the design intent was self-claim only.

---

### Impact Explanation

An attacker forces any victim's unclaimed KERNEL allocation into `KernelDepositPool` without consent. The victim's tokens are immediately subject to the withdrawal delay (up to 30 days). During that window the victim cannot transfer, sell, or use those tokens. This constitutes **temporary freezing of funds** — matching the Medium impact tier.

---

### Likelihood Explanation

Exploitation requires only:
1. Reading the publicly published merkle proof data for any target account.
2. Calling `claimAndStake` with that data.

No special role, no prior approval, no capital at risk for the attacker. Any unprivileged external actor can execute this against any account that has an unclaimed allocation in the current merkle root. Likelihood is **high**.

---

### Recommendation

Restrict `claimAndStake` so that only the beneficiary can invoke it for themselves, mirroring `KernelTop100MerkleDistributor`:

```solidity
function claimAndStake(
    uint256 index,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external nonReentrant whenNotPaused {
    address account = msg.sender;   // enforce self-only
    uint256 amountToStake = _processClaim(index, account, cumulativeAmount, merkleProof);
    IKernelDepositPool(kernelDepositPool).stakeFor(account, amountToStake);
    emit ClaimedAndStaked(index, account, amountToStake);
}
```

The plain `claim` function may retain the `account` parameter (tokens are sent directly to the account with no lock-up), but `claimAndStake` must not.

---

### Proof of Concept

1. Admin publishes a new merkle root. Off-chain proof data for `victim` is publicly available: `(index=5, account=victim, cumulativeAmount=1000e18, merkleProof=[...])`.
2. Attacker calls:
   ```solidity
   kernelMerkleDistributor.claimAndStake(5, victim, 1000e18, merkleProof);
   ```
3. `_processClaim` verifies the proof, deducts the fee, marks the claim consumed for `victim`, and returns `amountToStake`.
4. `KernelDepositPool.stakeFor(victim, amountToStake)` executes; `balanceOf[victim]` increases and tokens are transferred from `KernelMerkleDistributor` to `KernelDepositPool`.
5. `victim` now has no liquid KERNEL. To recover funds, victim must call `initiateWithdrawal` and wait up to 30 days before `claimWithdrawal` becomes callable. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
