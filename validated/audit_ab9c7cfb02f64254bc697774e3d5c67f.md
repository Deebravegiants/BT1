### Title
Dijkstra Era Sub-Transaction Execution Units Bypass `maxTxExUnits` Validation — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces nested sub-transactions (`SubTx`) inside a top-level transaction (`TopTx`). The `dijkstraUtxoTransition` rule enforces `maxTxExUnits` by calling `Alonzo.validateExUnitsTooBigUTxO`, which internally uses `totExUnits tx`. However, `totExUnits` only sums redeemers from the **top-level transaction's witnesses** — it is completely blind to the redeemers embedded in each sub-transaction's own witness set. Because sub-transactions can carry Plutus scripts with their own redeemers, and neither the `SUBUTXO` rule nor any other rule checks sub-transaction execution units against `maxTxExUnits` or `maxBlockExUnits`, an unprivileged submitter can craft a `TopTx` whose aggregate Plutus execution budget across all sub-transactions is an unbounded multiple of the intended per-transaction and per-block limits.

---

### Finding Description

**Root cause — `totExUnits` is blind to sub-transaction redeemers**

`totExUnits` is defined in the Alonzo layer and is not overridden for the Dijkstra era:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
``` [1](#0-0) 

For a `Tx TopTx DijkstraEra`, `witsTxL` resolves to the **top-level** `dtWits` field. Sub-transactions live in the body field `dtbrSubTransactions :: OMap TxId (Tx SubTx era)`, each carrying its own independent witness set `dstWits` with its own redeemers. `totExUnits` never traverses these. [2](#0-1) 

**The UTXO rule only checks top-level execution units**

`dijkstraUtxoTransition` calls:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [3](#0-2) 

This check passes as long as the top-level transaction's redeemers are within budget. Sub-transaction redeemers are never summed.

**The SUBUTXO rule has no ExUnits check**

`DijkstraSubUtxoPredFailure` enumerates every failure the sub-UTxO rule can raise. There is no `SubExUnitsTooBigUTxO` constructor: [4](#0-3) 

Sub-transactions do carry Plutus scripts with context (field `dsastPlutusScriptsWithContext` in `DijkstraStAnnSubTx`), confirming they are executed during validation: [5](#0-4) 

**The BBODY block-level check is equally blind**

The Alonzo BBODY `validateExUnits` also uses `foldMap totExUnits txs`, so sub-transaction execution units are excluded from the `maxBlockExUnits` check as well: [6](#0-5) 

**Attack path**

1. Attacker constructs a `TopTx` with zero or minimal top-level redeemers (passes `maxTxExUnits`).
2. The body includes N sub-transactions (`dtbrSubTransactions`), each containing a Plutus script redeemer declaring `maxTxExUnits` execution units.
3. Total execution budget = N × `maxTxExUnits`. With `maxTxSize` = 16 384 bytes and a ~20-byte minimum redeemer, N ≈ 800, yielding ~800× the intended limit.
4. The transaction passes all ledger validation checks and is included in a block.
5. Every validating node must execute all sub-transaction Plutus scripts, spending CPU far beyond the intended per-transaction and per-block budgets.

---

### Impact Explanation

**Medium — Attacker-controlled transactions exceed intended validation limits.**

The `maxTxExUnits` and `maxBlockExUnits` protocol parameters are the ledger's explicit resource-limit controls for Plutus script execution. A single crafted `TopTx` can force every validating node to execute Plutus scripts consuming an unbounded multiple of those limits, degrading block-processing throughput and potentially causing slot-leader nodes to miss their production windows. This matches the allowed impact: *"Attacker-controlled transactions… exceed intended validation limits."*

---

### Likelihood Explanation

Any unprivileged user can submit a Dijkstra `TopTx` with sub-transactions. No special role, key, or governance majority is required. The only cost is the transaction fee (proportional to `maxTxSize`), making repeated exploitation cheap. Likelihood is **medium-high** once the Dijkstra era is live.

---

### Recommendation

Override `totExUnits` for the Dijkstra era (or introduce a dedicated `validateBatchExUnits` check in `dijkstraUtxoTransition`) to sum execution units across the top-level transaction **and all sub-transactions**:

```haskell
-- In dijkstraUtxoTransition, replace the existing ExUnits check with:
let batchExUnits =
      totExUnits tx
        <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
runTest $ failureUnless (pointWiseExUnits (<=) batchExUnits (pp ^. ppMaxTxExUnitsL))
            (ExUnitsTooBigUTxO Mismatch { mismatchSupplied = batchExUnits
                                        , mismatchExpected = pp ^. ppMaxTxExUnitsL })
```

The same fix must be applied to the BBODY-level `validateExUnits` so that `maxBlockExUnits` is also enforced over the full batch.

---

### Proof of Concept

1. **Confirm `totExUnits` ignores sub-transaction redeemers**: `totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL` — `witsTxL` for a `Tx TopTx DijkstraEra` resolves to `dtWits`, not to any sub-transaction witness. [1](#0-0) 

2. **Confirm sub-transactions carry independent Plutus scripts**: `DijkstraStAnnSubTx` has `dsastPlutusScriptsWithContext`, proving sub-transaction Plutus scripts are collected and executed. [5](#0-4) 

3. **Confirm no ExUnits check in SUBUTXO**: `DijkstraSubUtxoPredFailure` has no `SubExUnitsTooBigUTxO` constructor; the sub-UTxO transition rule omits any execution-unit validation. [4](#0-3) 

4. **Confirm the UTXO rule only checks top-level units**: `dijkstraUtxoTransition` calls `Alonzo.validateExUnitsTooBigUTxO pp tx` with the unmodified `totExUnits`. [3](#0-2) 

5. **Confirm sub-transactions are an unbounded collection within `maxTxSize`**: `dtbrSubTransactions :: OMap TxId (Tx SubTx era)` has no protocol-parameter-enforced count limit; the only bound is the serialized transaction size. [2](#0-1)

-- | Validate that total execution units (all transactions) do not exceed block limit.
-- ∑(tx ∈ txs)(totExunits tx) ≤ maxBlockExUnits pp
validateExUnits ::
  forall era.
  ( AlonzoEraTx era
  , InjectRuleFailure "BBODY" AlonzoBbodyPredFailure era
  ) =>
  StrictSeq.StrictSeq (Tx TopTx era) ->
  -- | Max block exunits protocol parameter.
  ExUnits ->
  Rule (EraRule "BBODY" era) 'Transition ()
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
   in pointWiseExUnits (<=) txTotal ppMax
        ?! injectFailure
          ( TooManyExUnits $
              Mismatch
                { mismatchSupplied = txTotal
                , mismatchExpected = ppMax
                }
          )
