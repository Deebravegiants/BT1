### Title
Cache-Poisoning via Mismatched `inner.chunk_hash`/`metadata` in `ChunkEndorsementV2::verify` Drops Legitimate Validator Endorsements — (`core/primitives/src/stateless_validation/chunk_endorsement.rs`)

### Summary

`ChunkEndorsementV2::verify` verifies the inner signature (over `chunk_hash`) and the metadata signature (over `shard_id`/`epoch_id`/`height_created`) **independently**, with no cross-binding between them. A malicious peer who has observed validator V's endorsements for two different chunks can splice them into a hybrid that passes `verify`, poisons the endorsement cache under C1's `ChunkProductionKey` with C2's `chunk_hash`, and causes V's legitimate C1 endorsement to be rejected as a duplicate — effectively dropping it from `collect_chunk_endorsements`.

### Finding Description

`ChunkEndorsementV2` has two independently signed components:

- `inner: ChunkEndorsementInnerV1` — contains `chunk_hash` + `signature_differentiator`, signed by `signature`
- `metadata: ChunkEndorsementMetadata` — contains `account_id`, `shard_id`, `epoch_id`, `height_created`, signed by `metadata_signature`

`ChunkEndorsementMetadata` contains **no `chunk_hash`**: [5](#0-4) 

`ChunkEndorsementV2::verify` checks both signatures against the same public key but never asserts that `inner.chunk_hash` corresponds to the chunk described by `metadata`: [6](#0-5) 

A hybrid message constructed as:
- `inner` = V's legitimate inner for C2 (chunk_hash = H(C2))
- `signature` = V's legitimate signature over that inner
- `metadata` = V's legitimate metadata for C1 (shard_id/epoch_id/height_created of C1)
- `metadata_signature` = V's legitimate metadata signature for C1

...passes `verify` because both signatures are cryptographically valid from V's key.

After passing `validate_chunk_endorsement`, the cache insert at: [3](#0-2) 

stores the entry under `ChunkProductionKey` derived from `metadata` (C1's key) but with `chunk_hash = H(C2)`.

The deduplication guard: [7](#0-6) 

then causes V's legitimate C1 endorsement (arriving later) to be silently dropped as a duplicate.

When `collect_chunk_endorsements` runs for C1: [4](#0-3) 

the filter `chunk_hash == chunk_header.chunk_hash()` (C1's hash) does not match the stored H(C2), so V's entry is excluded from the endorsement state computation.

### Impact Explanation

V's endorsement for C1 is permanently dropped from the cache for the lifetime of that cache entry. If V's stake is needed to cross the endorsement threshold for C1, the chunk may fail to collect sufficient endorsements, delaying or preventing block production for that slot. An attacker targeting multiple validators simultaneously (by collecting their endorsements for other chunks and racing the hybrid messages) could suppress enough stake to stall chunk endorsement collection — a liveness attack on consensus.

### Likelihood Explanation

**Preconditions:**
1. The attacker must be a connected peer who can observe P2P endorsement messages from validator V for two different chunks (C1 and C2).
2. V must be a chunk validator for both C1 and C2 (different `ChunkProductionKey`s), which is normal across epochs/heights/shards.
3. The attacker must send the hybrid to the target node **before** V's legitimate C1 endorsement arrives — a race condition that is feasible for a well-positioned peer (e.g., one with lower latency to the target than V has).

These preconditions are achievable by a malicious peer with no validator privileges. The attack window is narrow per-chunk but repeatable across every chunk production cycle.

### Recommendation

Bind `inner.chunk_hash` to the metadata at signing and verification time. The simplest fix is to include `chunk_hash` in `ChunkEndorsementMetadata` and sign a single unified structure, or to add an explicit assertion in `ChunkEndorsementV2::verify` that the `chunk_hash` in `inner` matches the expected hash derivable from the metadata (if the chunk hash can be derived from the key). Alternatively, sign a single combined struct containing both the chunk hash and the production key fields, eliminating the two-independent-signature design entirely.

// Attacker intercepts both endorsements from the network.
let endorsement_c1: ChunkEndorsementV2 = /* observed from network */;
let endorsement_c2: ChunkEndorsementV2 = /* observed from network */;

// Construct hybrid: inner from C2, metadata from C1

    inner: endorsement_c2.inner.clone(),       // chunk_hash = H(C2)
    signature: endorsement_c2.signature.clone(), // valid over H(C2)
    metadata: endorsement_c1.metadata.clone(),  // C1's shard/epoch/height
    metadata_signature: endorsement_c1.metadata_signature.clone(), // valid over C1 metadata
});

// Both signatures verify independently — no cross-check exists
assert!(hybrid.verify(v_public_key)); // passes

// Cache now stores: key=C1's ChunkProductionKey, value=(H(C2), sig_c2)
// V's legitimate C1 endorsement is rejected as duplicate
// collect_chunk_endorsements for C1 filters out V's entry (H(C2) != H(C1))
