### Title
Division by Zero in `getRsETHAmountToMint` When `rsETHPrice` Is Uninitialized — (`contracts/LRTDepositPool.sol`)

---

### Summary

`LRTOracle.rsETHPrice` is a storage variable that defaults to `0` and is only set by calling `updateRSETHPrice()`. `LRTDepositPool.getRsETHAmountToMint` divides by `rsETHPrice` without a zero-guard. If `updateRSETHPrice()` has not been called before deposits are opened, every user deposit reverts with a division-by-zero panic, temporarily freezing all deposit activity.

---

### Finding Description

`LRTOracle` declares `rsETHPrice` as a plain `uint256` storage variable: [1](#0-0) 

Its `initialize` function never assigns a value to it: [2](#0-1) 

The only code path that sets `rsETHPrice` to a non-zero value is `_updateRsETHPrice`, which is reached only when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is called externally: [3](#0-2) 

`LRTDepositPool.getRsETHAmountToMint` uses `rsETHPrice` as the denominator with no zero-check: [4](#0-3) 

Both user-facing deposit entry points call this function unconditionally: [5](#0-4) [6](#0-5) 

There is no `require` or `if`-guard anywhere in the call chain that prevents execution when `rsETHPrice == 0`.

---

### Impact Explanation

If `updateRSETHPrice()` has not been called before the first user deposit, every call to `depositETH` and `depositAsset` reverts with a Solidity division-by-zero panic. This blocks all deposit activity until an operator manually calls `updateRSETHPrice()`. The impact is **temporary freezing of funds** (no user can deposit; funds already in the protocol are unaffected).

---

### Likelihood Explanation

The protocol has no on-chain enforcement that `updateRSETHPrice()` must be called before deposits are enabled. A deployment or upgrade sequence that opens the deposit pool before the oracle price is seeded — a realistic operational mistake — triggers the freeze for all depositors until the price is updated. The likelihood is **low** but the scenario is realistic and requires no attacker: any depositor who calls `depositETH` or `depositAsset` in this window will experience the revert.

---

### Recommendation

1. In `LRTOracle.initialize`, call `_updateRsETHPrice()` (or set `rsETHPrice = 1 ether` directly) so the price is never left at zero after deployment.
2. Add a zero-guard in `getRsETHAmountToMint`:

```solidity
uint256 price = lrtOracle.rsETHPrice();
require(price != 0, "rsETH price not initialized");
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / price;
```

---

### Proof of Concept

1. Deploy `LRTOracle` and `LRTDepositPool` without calling `updateRSETHPrice()`.
2. `LRTOracle.rsETHPrice()` returns `0`.
3. Any user calls `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `getRsETHAmountToMint` → `(1e18 * assetPrice) / 0` → EVM division-by-zero panic → revert.
5. All deposits are blocked until an operator calls `lrtOracle.updateRSETHPrice()`. [7](#0-6) [2](#0-1)
