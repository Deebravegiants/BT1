Audit Report

## Title
ETH Deposit Limit Check Omits New Deposit Amount, Allowing Limit to Be Exceeded - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric ceiling check: the ETH branch evaluates `totalAssetDeposits > limit` (missing `+ amount`), while the ERC-20 branch correctly evaluates `totalAssetDeposits + amount > limit`. When total ETH deposits exactly equal the configured limit, the ETH check returns `false`, allowing `depositETH` to proceed and mint rsETH beyond the intended ceiling.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch at L679 omits the incoming `amount`:

```solidity
// L678-681
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← missing + amount
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
``` [1](#0-0) 

This function is called from `_beforeDeposit` (L661), which reverts with `MaximumDepositLimitReached` only when it returns `true`. [2](#0-1) 

When `totalAssetDeposits == depositLimitByAsset(ETH)`, the expression `totalAssetDeposits > limit` evaluates to `false`, so `_beforeDeposit` does not revert. `depositETH` then proceeds to call `_mintRsETH(rsethAmountToMint)` at L90, minting rsETH for the full `msg.value` and pushing total ETH deposits above the ceiling. [3](#0-2) 

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** The `depositLimitByAsset` ceiling is the protocol's primary risk-management control for ETH exposure. Bypassing it causes rsETH to be minted against ETH the protocol did not intend to accept, violating the enforced accounting invariant. No direct theft occurs in a single transaction.

## Likelihood Explanation
Any unprivileged external caller can trigger this via a standard `depositETH{value: X}()` call. The boundary condition `totalAssetDeposits == depositLimitByAsset(ETH)` is a natural state that arises whenever the pool approaches capacity. No special permissions, flash loans, or oracle manipulation are required.

## Recommendation
Mirror the ERC-20 branch in the ETH branch by adding `+ amount`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Protocol accumulates `getTotalAssetDeposits(ETH) = 1000 ether` through normal usage.
3. Attacker calls `depositETH{value: 1 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 1 ETH; total ETH deposits become 1001 ether, exceeding the ceiling.
6. The ERC-20 equivalent path would have reverted at step 4 (`1000 + 1 > 1000` → `true`).

**Foundry test plan:** Deploy `LRTDepositPool` with a mock config returning `depositLimitByAsset(ETH) = 1000 ether`. Seed `getTotalAssetDeposits(ETH)` to exactly `1000 ether`. Call `depositETH{value: 1 ether}(0, "")` and assert it does not revert and that `getTotalAssetDeposits(ETH)` returns `1001 ether` post-call. Confirm the analogous ERC-20 call reverts with `MaximumDepositLimitReached`.
