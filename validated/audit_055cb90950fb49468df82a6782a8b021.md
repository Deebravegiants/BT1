### Title
Unpermissioned `directDeposits` Enables Frontrunning to Invalidate Victim's Reward Withdrawals in Legacy Mode ÔÇö (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

In the Dijkstra era, the `directDeposits` field of a transaction body allows any unprivileged transaction submitter to increase the balance of any registered account without the account owner's consent. In legacy-mode withdrawal validation (`withdrawalsThatDoNotDrainAccounts`), reward withdrawals must exactly drain the full account balance. An attacker can front-run a victim's pending withdrawal by depositing a minimal amount (e.g., 1 lovelace) to the victim's account, causing the victim's pre-signed transaction to fail with `IncompleteWithdrawals`. The attacker must spend the deposited ADA (it goes to the victim), but the cost is negligible and the attack can be repeated indefinitely.

---

### Finding Description

**Root cause ÔÇö no authorization check on `directDeposits`**

`dijkstraEntitiesTransition` in `Entities.hs` reads `directDeposits` from the transaction body and applies them unconditionally to account balances after certificate processing:

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

The only checks performed are (1) network-ID validation (`validateWrongNetworkInDirectDeposit`) and (2) that the target accounts are registered (`directDepositsMissingAccounts`). There is no check that the transaction submitter owns or has any relationship to the target accounts. Any party can include `directDeposits` targeting any registered account in their transaction.

`applyDirectDeposits` simply adds the deposited coin to the target account's balance:

```haskell
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
```

**How this breaks withdrawals in legacy mode**

`validateWithdrawals` is called earlier in the same rule with the pre-deposit account state:

```haskell
validateWithdrawals legacyMode network withdrawals accounts
```

When `legacyMode = True`, it delegates to `withdrawalsThatDoNotDrainAccounts`, which requires the withdrawal amount to **exactly equal** the account balance:

```haskell
withdrawalsThatDoNotDrainAccounts =
  categorizeWithdrawals
    ( \withdrawalAmount account ->
        withdrawalAmount == fromCompact (account ^. balanceAccountStateL)
    )
```

If an attacker's `directDeposits` transaction is included in a block before the victim's withdrawal transaction, the victim's account balance in the `certState` passed to the ENTITIES rule will already include the attacker's deposit. The victim's pre-signed withdrawal (specifying the old exact balance) then fails with `IncompleteWithdrawals`.

The `validateBatchWithdrawals` check in the UTXO rule uses `Ôëñ` (not `=`), so it passes ÔÇö the withdrawal amount does not exceed the new (higher) balance. The failure occurs only in the ENTITIES rule's legacy-mode exact-drain check, making the failure non-obvious.

**Contrast with SUBENTITIES**

`dijkstraSubEntitiesTransition` always uses `withdrawalsThatExceedAccountBalance` (the `Ôëñ` check), so sub-transactions are not affected. The vulnerability is specific to the top-level ENTITIES rule when `legacyMode = True`.

---

### Impact Explanation

An attacker can repeatedly front-run any victim's reward withdrawal at a cost of 1 lovelace per attempt (the deposited amount is credited to the victim, not destroyed). Each attempt forces the victim to re-sign their withdrawal transaction with the updated balance. If the victim's withdrawal is embedded in a time-sensitive or script-constrained transaction (e.g., a Plutus script that asserts a specific withdrawal amount), the attacker can cause that transaction to fail permanently within its validity window, effectively freezing the victim's rewards for the duration of the attack.

This matches the **Medium** allowed impact: *"Attacker-controlled transactionsÔÇª modifyÔÇª rewardsÔÇª outside design parameters."* The victim's reward balance is modified by an unauthorized third party, disrupting the victim's ability to withdraw on their own terms.

---

### Likelihood Explanation

The attack requires the attacker's `directDeposits` transaction to be ordered before the victim's withdrawal in the same block. Block producers control intra-block transaction ordering, and any block producer can trivially execute this. A non-block-producer attacker must rely on mempool monitoring and transaction submission timing, which is feasible given Cardano's ~20-second slot interval. The cost is 1 lovelace per disruption, making sustained griefing economically viable.

---
