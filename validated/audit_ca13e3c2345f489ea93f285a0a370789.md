The code is traceable and the gap is confirmed. Here is the complete analysis:

**Exact call chain:**

1. `pub_from_der` (`src/cbmpc/crypto/base_eddsa.cpp:191-195`) checks the X.509 prefix, then calls `from_bin` — no subgroup check.
2. `from_bin` (`src/cbmpc/crypto/base_eddsa.cpp:167-174`) delegates to `ec25519_core::from_bin` (`src/cbmpc/crypto/ec25519_core.cpp:872-912`), which only verifies the point satisfies the curve equation (valid x,y), not subgroup membership.
3. `ecurve_ed_t::verify` (`src/cbmpc/crypto/base_eddsa.cpp:204-210`) re-encodes the stored point to 32 bytes and passes it directly to `ED25519_verify` — again, no subgroup check.
4. `ecurve_t::check()` (`src/cbmpc/crypto/base_ecc.cpp:592-601`) **does** call `is_in_subgroup()` and would reject a torsion point, but it is **never called** in the `pub_from_der → verify` path.

The existing test `RejectTorsionAndFixInfinityEq` (`tests/unit/crypto/test_eddsa.cpp:14-37`) explicitly confirms: `from_bin` returns `SUCCESS` for the order-2 torsion point (x=0, y=−1), while `curve.check()` returns non-SUCCESS. This is the exact gap.

**Forgery feasibility under `ED25519_verify`:**

`ED25519_verify` is implemented as `ossl_ed25519_verify` (OpenSSL ≥ 3.0, `src/cbmpc/crypto/ec25519_core.cpp:1019-1022`). OpenSSL's Ed25519 verification does **not** check subgroup membership of the public key. For the order-2 torsion point T:

- `[k]T` = identity when k is even, T when k is odd.
- Non-cofactor form `[s]B = R + [k]T`: if k is even, the equation reduces to `[s]B = R`. The attacker picks any s, computes R = [s]B, computes k = SHA-512(R ‖ T ‖ M) mod ℓ. If k is even (~50% probability), (R, s) is a valid signature. Expected ~2 attempts.
- Cofactor form `[8][s]B = [8]R + [8][k]T`: since `[8]T = identity` (T has order 2, 8 = 2³), the equation reduces to `[8][s]B = [8]R`, which is trivially satisfied for any (R, s) with R = [s]B — **zero-attempt forgery**.

Either way, the forgery is computationally trivial.

### Title
Missing subgroup check in `pub_from_der` enables torsion-point public key acceptance and EdDSA signature forgery — (`src/cbmpc/crypto/base_eddsa.cpp`)

### Summary
`ecurve_ed_t::pub_from_der` imports a DER-encoded Ed25519 public key by checking only the X.509 prefix and curve-equation validity, without checking subgroup membership. An attacker who supplies a DER blob encoding a torsion point (e.g., the order-2 point (0, −1)) obtains a successfully parsed `ecc_pub_key_t`. The subsequent `ecurve_ed_t::verify` path also performs no subgroup check before calling `ED25519_verify`, which itself does not check subgroup membership. Because the torsion point has order dividing 8, the Ed25519 verification equation degenerates and the attacker can forge a valid (R, s) pair for any message with at most a handful of attempts.

### Finding Description
`pub_from_der` (`src/cbmpc/crypto/base_eddsa.cpp:191-195`) validates only the DER prefix and delegates to `from_bin`, which calls `ec25519_core::from_bin` (`src/cbmpc/crypto/ec25519_core.cpp:872-912`). That function solves the curve equation for x given y and returns `SUCCESS` for any valid curve point, including torsion points outside the prime-order subgroup. No call to `ecurve_t::check()` or `is_in_subgroup()` is made.

`ecurve_ed_t::verify` (`src/cbmpc/crypto/base_eddsa.cpp:204-210`) re-serializes the stored point to 32 bytes and passes it to `ED25519_verify` without any subgroup check. `ED25519_verify` is `ossl_ed25519_verify` under OpenSSL 3.x (`src/cbmpc/crypto/ec25519_core.cpp:1019-1022`), which decompresses the public key and checks curve membership but not subgroup membership.

`ecurve_t::check()` (`src/cbmpc/crypto/base_ecc.cpp:592-601`) calls `is_in_subgroup()` and would reject the torsion point, but it is not invoked anywhere in the `pub_from_der → verify` path.

The existing test suite (`tests/unit/crypto/test_eddsa.cpp:14-37`) explicitly documents that `from_bin` returns `SUCCESS` for the order-2 torsion point while `curve.check()` returns non-SUCCESS — confirming the gap is known at the `from_bin` level but not closed at the `pub_from_der` level.

### Impact Explanation
An attacker who can supply a DER-encoded public key to any code path that calls `pub_from_der` followed by `verify` can:
1. Provide the order-2 torsion point T = (0, −1) as their "public key."
2. For any target message M, pick any scalar s, compute R = [s]B, compute k = SHA-512(R ‖ T ‖ M) mod ℓ. If k is even (≈50%), the pair (R, s) satisfies `[s]B = R + [k]T` (since `[k]T = identity`). Under cofactor verification the forgery is unconditional.
3. `verify` returns `SUCCESS` for the forged (R, s) under T.

The result is that honest code that imports a peer-supplied public key via `pub_from_der` and then calls `verify` will accept a forged EdDSA signature, violating the expected authentication guarantee.

### Likelihood Explanation
Any API caller or protocol peer that can supply a DER-encoded public key blob can trigger this. The forgery requires at most ~2 hash evaluations (expected). The torsion point encoding is a fixed 32-byte value that is trivially constructed.

### Recommendation
Add a subgroup check immediately after `from_bin` in `pub_from_der`:

```cpp
error_t ecurve_ed_t::pub_from_der(ecc_pub_key_t& P, mem_t der) const {
  if (der.size != ed25519::x509_prefix.size + ed25519::pub_compressed_bin_size())
    return coinbase::error(E_FORMAT);
  if (0 != memcmp(ed25519::x509_prefix.data, der.data, ed25519::x509_prefix.size))
    return coinbase::error(E_FORMAT);
  error_t rv = from_bin(P, der.skip(ed25519::x509_prefix.size));
  if (rv) return rv;
  // NEW: reject torsion points
  if (!is_in_subgroup(P)) return coinbase::error(E_CRYPTO);
  return SUCCESS;
}
```

Alternatively, call `curve.check(P)` after `from_bin`, which already enforces subgroup membership and non-infinity. Similarly, `ecurve_ed_t::verify` should guard against a non-subgroup public key before invoking `ED25519_verify`.

static error_t from_bin(point_t& R, mem_t bin) {
  if (bin.size != 32) return coinbase::error(E_FORMAT);

  buf_t buf = bin.rev();
  uint8_t neg = buf[0] >> 7;
  buf[0] &= 0x7f;
  fe_t y = fe_t::to_fe(uint256_t::from_bin(buf));

  // x² = (y² - 1) / (dy² + 1)

  fe_t u, v, w, vxx, check;

  u = y * y;
  v = u * formula_t::get_d();
  u -= fe_t::one();       // u = y^2-1
  v += fe_t::one();       // v = dy^2+1
  w = u * v;              // w = u*v
  fe_t x = w.pow22523();  // x = w^((q-5)/8)
  x *= u;                 // x = u * w^((q-5)/8)

  vxx = x * x;
  vxx *= v;
  check = vxx - u;  // vx^2-u
  if (!check.is_zero()) {
    check = vxx + u;  // vx^2+u
    if (!check.is_zero()) {
      return coinbase::error(E_CRYPTO);
    }
    static const fe_t sqrtm1 =
        fe_t::from_bn(bn_t::from_hex("2b8324804fc1df0b2b4d00993dfbd7a72f431806ad2fe478c4ee1b274a0ea0b0"));
    x *= sqrtm1;
  }

  uint256_t x_val = x.from_fe();
  if (neg != (x_val.w0 & 1)) x = -x;

  R.x = x;
  R.y = y;
  R.z = fe_t::one();
  return 0;
}
