### Title
Cross-Request Signature Replay in `respond_verify_foreign_tx` Delivers Stale Foreign-Chain Attestation Without Re-Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function verifies that the MPC signature in the response is valid over `response.payload_hash` under the root public key, but it never verifies that `response.payload_hash` actually corresponds to the current pending `request`. Because `ForeignTxSignPayload` contains no nonce, receipt ID, or timestamp, a previously-produced valid `VerifyForeignTransactionResponse` remains cryptographically valid forever. A Byzantine attested participant (strictly below the signing threshold) can replay a captured response to satisfy any future pending request with identical parameters, delivering a stale foreign-chain attestation without triggering actual re-verification by the MPC nodes.

---

### Finding Description

**Root cause — `respond_verify_foreign_tx` does not bind `payload_hash` to the pending request:**

```
crates/contract/src/lib.rs  lines 718–743
```

The function performs two checks:

1. The caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`).
2. The signature in `response` is valid over `response.payload_hash` under the root MPC public key. [1](#0-0) 

It then calls `resolve_yields_for` keyed on `request`, which drains the pending queue and removes the entry. [2](#0-1) 

**What is missing:** the function never checks that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the current `request`. The `ForeignTxSignPayload` that the MPC network signs contains only `(request, extracted_values)` — no nonce, no receipt ID, no timestamp. [3](#0-2) 

**What `VerifyForeignTransactionRequest` looks like (the pending-map key):** [4](#0-3) 

No nonce, no receipt ID, no caller account ID. The key is purely `(domain_id, payload_version, ForeignChainRpcRequest)`.

**`resolve_yields_for` removes the entry after draining, but a new submission recreates it:** [5](#0-4) 

Once the entry is gone, any new `verify_foreign_transaction(request_A)` call creates a fresh pending entry under the same key. At that point, the old `(request_A, response_A)` pair is still cryptographically valid and can be submitted again.

---

### Impact Explanation

A Byzantine attested participant (below the signing threshold) who has observed a prior `respond_verify_foreign_tx(request_A, response_A)` on-chain can:

1. Wait for (or themselves submit) a new `verify_foreign_transaction(request_A)`.
2. Immediately call `respond_verify_foreign_tx(request_A, response_A)` with the captured old response.
3. The contract accepts it — `verify_ecdsa_signature(sig_A, payload_hash_A, root_pk)` is still valid.
4. The pending yield is drained; the caller receives `response_A`.

The user receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes extracted values from a **past** chain state, not the current one. For the primary use case — bridge inbound flows where a NEAR contract reacts to a foreign-chain event — this enables:

- **Forged foreign-chain verification**: the contract's invariant "MPC nodes will verify the transaction on the foreign chain before signing" is violated; no re-verification occurred.
- **Double-spend / invalid bridge execution**: if a Bitcoin transaction was confirmed in block H1 (response_A), then reorganized out, a bridge that re-queries `verify_foreign_transaction` to confirm the transaction is still final receives the stale H1 attestation and may release funds a second time.

This matches the allowed impact: **High — cross-chain replay / forged foreign-chain verification causing invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

- The attacker must be an attested MPC participant (below threshold). This is a realistic Byzantine-node scenario explicitly in scope.
- The captured response is publicly visible on-chain after the first honest `respond_verify_foreign_tx` call.
- Bridge retry patterns (re-verifying after a suspected reorg) are the exact scenario where the same `ForeignChainRpcRequest` is submitted a second time.
- The race between the Byzantine participant's replay and honest nodes re-running verification is won trivially if the Byzantine participant acts in the same block as the new `verify_foreign_transaction` submission.

---

### Recommendation

Include a unique per-request identifier in `ForeignTxSignPayload` so that each MPC signature is bound to exactly one request instance. The node-side `VerifyForeignTxRequest` already carries `receipt_id` and `id` (a `CryptoHash`) that uniquely identify each on-chain request: [6](#0-5) 

The fix has two parts:

1. **Extend `ForeignTxSignPayloadV1`** to include the `receipt_id` (or a contract-assigned nonce) so the signed hash is unique per request instance.
2. **In `respond_verify_foreign_tx`**, reconstruct the expected `payload_hash` from `(request, receipt_id, extracted_values)` and assert it equals `response.payload_hash` before accepting the response.

This mirrors the standard nonce-based mitigation recommended in the external report and ensures a valid signature for one request instance cannot satisfy a different instance.

---

### Proof of Concept

**Setup**: attested participant P (below threshold), honest MPC network, bridge contract B.

1. User U calls `verify_foreign_transaction({ request: bitcoin_tx_X, confirmations: 6, extractors: [BlockHash] })`. A pending entry is created under key `K = (domain_id, V1, bitcoin_tx_X)`.
2. Honest MPC nodes query Bitcoin RPC: tx X is in block H1 with 8 confirmations. They sign `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload { request: K, values: [BlockHash(H1)] }))`.
3. An honest participant calls `respond_verify_foreign_tx(K, { payload_hash: payload_hash_A, signature: sig_A })`. The queue is drained; U receives `response_A`. Bridge B releases funds.
4. Bitcoin reorg: tx X is reorganized into block H2 (or dropped entirely).
5. Bridge B calls `verify_foreign_transaction(K)` again to re-confirm before a second release. A new pending entry is created under the same key K.
6. Byzantine participant P calls `respond_verify_foreign_tx(K, { payload_hash: payload_hash_A, signature: sig_A })` — the exact same arguments as step 3.
7. Contract check: `verify_ecdsa_signature(sig_A, payload_hash_A, root_pk)` → **valid** (signature has not expired or been revoked).
8. `resolve_yields_for(K, response_A)` drains the new pending entry. Bridge B receives `response_A` claiming tx X is confirmed in block H1.
9. Bridge B releases funds a second time. **Double-spend achieved.**

The contract never detected that `payload_hash_A` encodes stale extracted values inconsistent with the current chain state, because `respond_verify_foreign_tx` performs no such binding check.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
