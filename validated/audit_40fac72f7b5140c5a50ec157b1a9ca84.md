### Title
`respond_verify_foreign_tx` Accepts Replayed Foreign-Chain Verification Responses — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash`, but never checks that `response.payload_hash` actually corresponds to the `request` being answered. A single malicious attested MPC participant can replay a valid `(payload_hash, signature)` pair from any prior foreign-chain verification response to resolve a completely different pending request, causing the contract to deliver a forged attestation to the caller.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract performs the following signature check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // taken from the response
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // NOT derived from `request`
    &secp_pk,
)
``` [1](#0-0) 

The `payload_hash` is caller-supplied (part of `VerifyForeignTransactionResponse`). The contract only checks that the signature is a valid ECDSA signature over that hash under the domain's root public key. It never reconstructs the expected hash from the `request` argument and compares.

Contrast this with `respond()` for regular signatures, where the payload hash is derived directly from the request:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,   // derived from the request itself
    &expected_public_key,
)
``` [2](#0-1) 

The `ForeignTxSignPayload` that nodes sign contains only `(ForeignChainRpcRequest, Vec<ExtractedValue>)` — it includes no NEAR contract address, no unique request ID, and no nonce:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [3](#0-2) 

The hash is computed as `SHA-256(borsh(ForeignTxSignPayload))` with no domain separator binding it to a specific NEAR contract instance or pending request: [4](#0-3) 

After the (insufficient) signature check, the contract resolves the pending yield for `request` with the attacker-supplied response data: [5](#0-4) 

---

### Impact Explanation

A malicious attested MPC participant (a single node, strictly below the signing threshold) can:

1. Observe any previously accepted `respond_verify_foreign_tx` call on-chain, extracting a valid `(payload_hash_A, sig_A)` pair for foreign transaction `tx_id=A`.
2. Wait for a new pending request `R_B` for a different transaction `tx_id=B` (e.g., a bridge deposit).
3. Call `respond_verify_foreign_tx(R_B, {payload_hash: payload_hash_A, signature: sig_A})`.
4. The contract accepts: `sig_A` is a valid MPC signature over `payload_hash_A` ✓, `R_B` exists in `pending_verify_foreign_tx_requests` ✓.
5. `R_B`'s caller receives `{payload_hash: payload_hash_A, signature: sig_A}` — an attestation that corresponds to `tx_id=A`, not `tx_id=B`.

For the primary use case described in the design docs (Omnibridge inbound flow), a bridge contract that does not independently verify the returned `payload_hash` against its expected value would process a deposit for `tx_id=B` based on a forged attestation, enabling double-spend or unauthorized fund release.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.** [6](#0-5) 

---

### Likelihood Explanation

- The attacker only needs to be a single attested MPC participant — no threshold collusion required.
- Valid `(payload_hash, signature)` pairs are permanently visible on-chain after any successful `respond_verify_foreign_tx` call.
- The attack requires no special timing, no key material, and no off-chain coordination.
- The `verify_foreign_transaction` endpoint is explicitly designed for bridge use cases where callers may not independently verify the `payload_hash`.

---

### Recommendation

Reconstruct the expected `payload_hash` from the `request` on the contract side and reject any response whose `payload_hash` does not match. Since the contract does not have the extracted values, the fix should instead bind the signed payload to the specific request by including a unique, request-scoped identifier (e.g., the NEAR yield receipt ID or a contract-address + request-nonce) inside `ForeignTxSignPayload`. This is the direct analog of initializing `DOMAIN_SEPARATOR` in EIP-712: the signed message must be bound to the specific context in which it is valid.

Concretely:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
    pub near_contract_id: AccountId,   // binds to this contract deployment
    pub receipt_id: CryptoHash,        // binds to this specific pending request
}
```

The contract can then reconstruct the expected hash prefix from `request.receipt_id` (available in the pending-request map entry) and reject any response whose `payload_hash` does not include the correct receipt binding.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
