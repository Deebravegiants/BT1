### Title
Stale Cached `rsETHPrice` Used in Deposit Minting Without Prior Update - (`contracts/LRTOracle.sol` / `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTOracle` stores `rsETHPrice` as a cached state variable updated only when `updateRSETHPrice()` is called explicitly. `LRTDepositPool.depositETH()` and `depositAsset()` compute the rsETH amount to mint using this stale cached price without triggering an update first. This is the direct analog of the Carapace bug: a cached state is read without being refreshed, allowing a depositor to receive more rsETH than the current TVL justifies, at the expense of existing holders.

---

### Finding Description

`LRTOracle` stores the rsETH/ETH exchange rate in a persistent state variable: [1](#0-0) 

This value is only updated when `updateRSETHPrice()` (or `updateRSETHPriceAsManager()`) is called explicitly — it is never called inside the deposit flow: [2](#0-1) 

When a user calls `depositETH()` or `depositAsset()`, the internal `_beforeDeposit()` computes the rsETH amount to mint via `getRsETHAmountToMint()`: [3](#0-2) 

`getRsETHAmountToMint()` divides by the **cached** `rsETHPrice`, not the live computed price: [4](#0-3) 

The numerator uses a live oracle call (`lrtOracle.getAssetPrice(asset)`), but the denominator uses the stale stored `rsETHPrice`. If TVL has grown since the last `updateRSETHPrice()` call (e.g., staking rewards accrued, LST prices rose), the cached `rsETHPrice` is lower than the real current price, causing the division to yield a larger rsETH amount than is correct.

---

### Impact Explanation

When `rsETHPrice` is stale and lower than the real current price, a depositor receives more rsETH than their deposit is worth at the current exchange rate. The excess rsETH represents a proportional claim on the protocol's TVL that was not backed by the depositor's contribution — it is effectively extracted from the yield that belongs to existing rsETH holders. This is a **theft of unclaimed yield** (High impact).

---

### Likelihood Explanation

Staking rewards accrue continuously on EigenLayer and underlying LSTs. `updateRSETHPrice()` is called on a schedule, not on every deposit, so there is always a window — potentially hours — where `rsETHPrice` is stale. An attacker can:

1. Monitor the on-chain TVL (via `getTotalAssetDeposits`) and the stored `rsETHPrice`.
2. Observe that TVL has grown since the last price update.
3. Call `depositETH()` or `depositAsset()` before calling `updateRSETHPrice()`.
4. Receive excess rsETH minted at the stale lower price.
5. Call `updateRSETHPrice()` themselves to lock in the gain.

No privileged access is required. The attacker controls the entire sequence. Likelihood is **Medium** (requires monitoring but no special permissions).

---

### Recommendation

In `LRTDepositPool._beforeDeposit()`, call `ILRTOracle(lrtOracleAddress).updateRSETHPrice()` before computing `getRsETHAmountToMint()`. This mirrors the Carapace recommendation of calling `_assessState` inside the protection-buying path to ensure the state is fresh before it is used.

---

### Proof of Concept

1. At time T, `updateRSETHPrice()` is called. `rsETHPrice = 1.05e18` (TVL = 105 ETH, supply = 100 rsETH).
2. Staking rewards accrue. TVL grows to 106 ETH. Real price = `1.06e18`. `rsETHPrice` is still `1.05e18`.
3. Attacker calls `depositETH{value: 1 ether}("")`.
4. `getRsETHAmountToMint` computes: `(1e18 * 1e18) / 1.05e18 ≈ 0.952 rsETH` — but the correct amount at the real price is `(1e18 * 1e18) / 1.06e18 ≈ 0.943 rsETH`.
5. Attacker receives ~0.009 rsETH extra per ETH deposited, extracted from existing holders' accrued yield.
6. Attacker calls `updateRSETHPrice()`. Price updates to `1.06e18`. Attacker's rsETH is now correctly priced, but they received it at a discount. [5](#0-4) [6](#0-5) [2](#0-1) [7](#0-6)

        view

        // setup oracle contract
        address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
        ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);

        // calculate rseth amount to mint based on asset amount and asset exchange rate
        rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

            revert MinimumAmountToReceiveNotMet();
