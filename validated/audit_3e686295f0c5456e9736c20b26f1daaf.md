### Title
Single Committee Member Can Inflate CC Vote Count by Aliasing Cold Credentials to a Shared Hot Credential — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

The Conway governance ratification logic in `committeeAcceptedRatio` counts votes by iterating over cold credentials and looking up each one's associated hot credential, then looking up the vote for that hot credential. Because the `GOVCERT` rule places no uniqueness constraint on hot credentials, a malicious committee member can point their own cold credential to another member's already-registered hot credential. The result is that a single vote cast by that hot key is counted once per cold credential that maps to it, inflating the yes-vote ratio and potentially ratifying governance actions that should fail.

---

### Finding Description

**Root cause 1 — no hot-credential uniqueness check in `GOVCERT`:**

`conwayGovCertTransition` in `GovCert.hs` handles `ConwayAuthCommitteeHotKey coldCred hotCred` by calling `checkAndOverwriteCommitteeMemberState`. The only checks performed are:

1. The cold credential has not previously resigned (`ConwayCommitteeHasPreviouslyResigned`).
2. The cold credential is a known current or potential future committee member (`ConwayCommitteeIsUnknown`). [1](#0-0) 

There is no check that `hotCred` is not already registered as the hot credential for a different cold credential. Any committee member can therefore submit a valid `AuthCommitteeHotKeyTxCert` pointing their cold key to a hot key already in use by another member.

**Root cause 2 — no deduplication by hot credential in `committeeAcceptedRatio`:**

`committeeAcceptedRatio` folds over the `members` map (cold credential → expiry epoch). For each cold credential it resolves the associated hot credential from `CommitteeState`, then looks up the vote for that hot credential: [2](#0-1) 

If `N` cold credentials all map to the same hot credential `H`, and `H` casts a single `VoteYes`, the accumulator increments `yes` and `tot` by `N` — one increment per cold credential — even though only one actual vote was cast.

**The behavior is explicitly exercised in tests:**

The test "Many CC Cold Credentials map to the same Hot Credential act as many votes" demonstrates that registering all committee cold credentials to the same hot key causes a single vote from that hot key to ratify a governance action that would otherwise fail: [3](#0-2) 

The conformance test "Duplicate CC hot keys count as separate votes" further confirms this is the current implemented behavior: [4](#0-3) 

**Concrete attack scenario:**

- Committee: 5 members A, B, C, D, E; threshold 3/5 (60%).
- D and E are malicious committee members. A's hot credential `hotCredA` is public on-chain.
- D submits `AuthCommitteeHotKeyTxCert coldCredD hotCredA` — accepted by `GOVCERT`.
- E submits `AuthCommitteeHotKeyTxCert coldCredE hotCredA` — accepted by `GOVCERT`.
- A votes Yes on a `TreasuryWithdrawals` action using `hotCredA`.
- `committeeAcceptedRatio` counts: coldCredA → `hotCredA` → Yes (+1,+1); coldCredD → `hotCredA` → Yes (+1,+1); coldCredE → `hotCredA` → Yes (+1,+1). Result: 3/5 = 60% ≥ threshold → ratified.
- Without the attack, only 1/5 = 20% < threshold → rejected.

The `CommitteeState` type stores a `Map (Credential ColdCommitteeRole) CommitteeAuthorization`, which is a many-to-one mapping from cold to hot credentials with no inverse uniqueness constraint: [5](#0-4) 

---

### Impact Explanation

A malicious committee member (or a coalition of fewer members than the threshold) can cause any governance action to be ratified by the Constitutional Committee component alone, including:

- `TreasuryWithdrawals` — direct loss of ADA from the treasury.
- `NewConstitution` — unauthorized constitution replacement.
- `UpdateCommittee` — unauthorized committee membership change.
- `HardForkInitiation` — unauthorized hard fork.
- `ParameterChange` — unauthorized protocol parameter modification.

This matches **Critical: Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted**, and **Critical: Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition**.

---

### Likelihood Explanation

Any elected committee member can execute this attack unilaterally. The attacker needs only to:
1. Observe another member's hot credential on-chain (it is public).
2. Submit a single valid transaction containing `AuthCommitteeHotKeyTxCert`.

No key compromise, no majority collusion, and no privileged access beyond holding a committee cold key is required. The transaction passes all current ledger validation rules.

---

### Recommendation

Enforce hot-credential uniqueness in the `GOVCERT` rule: before inserting `CommitteeHotCredential hotCred` for a cold credential, check that `hotCred` is not already the authorized hot credential for any other cold credential in `csCommitteeCreds`. If it is, reject the certificate with a new predicate failure (e.g., `ConwayCommitteeHotKeyAlreadyAuthorized`).

Alternatively, fix `committeeAcceptedRatio` to deduplicate by hot credential before counting votes — i.e., group cold credentials by their resolved hot credential and count each unique hot credential's vote only once. [6](#0-5) 

---

### Proof of Concept

The existing test at `RatifySpec.hs:52–68` already demonstrates the mechanism. To demonstrate the attack:

1. Elect a committee of 5 members with threshold 3/5.
2. Register all 5 cold credentials to the same hot credential `H` (each member submits `AuthCommitteeHotKeyTxCert coldCredN H`).
3. Submit a `TreasuryWithdrawals` governance action.
4. Cast a single `VoteYes` from `CommitteeVoter H`.
5. Observe that `committeeAcceptedRatio` returns 5/5 = 1 ≥ threshold, and the action is ratified and enacted, draining the treasury.

The ledger accepts all five `AuthCommitteeHotKeyTxCert` certificates because `conwayGovCertTransition` only checks that each cold credential is a known member and has not resigned — it never checks whether the supplied hot credential is already in use. [1](#0-0) [7](#0-6)

```haskell
committeeAcceptedRatio members votes committeeState currentEpoch =
  yesVotes %? totalExcludingAbstain
  where
    accumVotes ::
      (Integer, Integer) ->
      Credential ColdCommitteeRole ->
      EpochNo ->
      (Integer, Integer)
    accumVotes (!yes, !tot) member expiry
      | currentEpoch > expiry = (yes, tot) -- member is expired, vote "abstain" (don't count it)
      | otherwise =
          case Map.lookup member (csCommitteeCreds committeeState) of
            Nothing -> (yes, tot) -- member is not registered, vote "abstain"
            Just (CommitteeMemberResigned _) -> (yes, tot) -- member has resigned, vote "abstain"
            Just (CommitteeHotCredential hotKey) ->
              case Map.lookup hotKey votes of
                Nothing -> (yes, tot + 1) -- member hasn't voted, vote "no"
                Just Abstain -> (yes, tot) -- member voted "abstain"
                Just VoteNo -> (yes, tot + 1) -- member voted "no"
                Just VoteYes -> (yes + 1, tot + 1) -- member voted "yes"
    (yesVotes, totalExcludingAbstain) = Map.foldlWithKey' accumVotes (0, 0) members
```
