### Title
Deposits for Non-Winning Update Proposals Are Permanently Locked on `do_update()` - (File: `crates/contract/src/update.rs`)

---

### Summary

When `do_update()` executes a winning contract/config update, it clears all competing proposals via `self.entries.clear()` without refunding the NEAR deposits that were paid by their proposers. Those deposits are permanently locked in the contract with no recovery path.

---

### Finding Description

`propose_update()` in `crates/contract/src/lib.rs` requires each proposer to attach a NEAR deposit calculated as `env::storage_byte_cost() * bytes_used(update)`. Only the excess above the required amount is refunded at proposal time; the required portion is retained by the contract to cover storage staking. [1](#0-0) 

The `UpdateEntry` struct explicitly stores `bytes_used` alongside the update payload, which is the exact information needed to compute and return the deposit: [2](#0-1) 

When `do_update()` fires after threshold votes are reached, it removes the winning entry, then calls `self.entries.clear()` and `self.vote_by_participant.clear()` to purge all competing proposals: [3](#0-2) 

No refund is issued to the proposers of the cleared entries. The NEAR they deposited stays in the contract balance permanently. There is no `cancel_update`, `withdraw_deposit`, or any other recovery function in the contract.

The analog to the D3Vault bug is direct: in D3Vault, `withdrawReserves()` moves funds out without updating the internal balance variable, corrupting downstream accounting. Here, `do_update()` frees storage (clearing entries) without returning the storage-staking deposits to their owners, permanently misaccounting the contract's NEAR balance relative to what proposers are owed.

---

### Impact Explanation

Every MPC participant who proposed a competing update that was not selected permanently loses their deposited NEAR. For a contract binary update, `bytes_used` is computed as: [4](#0-3) 

For a 500 KB contract binary, `bytes_used ≈ 500,000 + overhead`, and at NEAR's storage cost of ~10 yoctoNEAR/byte, the deposit is on the order of **5 NEAR per proposal**. With N participants each proposing a different update, N−1 participants lose their full deposits when any one update passes.

This breaks the production safety/accounting invariant that deposited NEAR for storage staking must be returned when the storage is freed. It maps to the allowed impact: **Medium — balance/accounting invariant broken without relying on network-level DoS or operator misconfiguration.**

---

### Likelihood Explanation

This triggers in the normal governance flow. The README explicitly states that all proposals are cleared on execution: [5](#0-4) 

In any upgrade cycle where more than one participant proposes a different update (a common scenario when participants disagree on the upgrade target), all non-winning proposers lose their deposits. No adversarial setup is required — the loss occurs in routine operation. A Byzantine participant below the signing threshold can also deliberately propose a large contract binary to maximize the deposit they cause others to lose, then vote for a different proposal.

---

### Recommendation

Before calling `self.entries.clear()`, iterate over the remaining entries and refund each proposer's deposit. The `bytes_used` field stored in `UpdateEntry` provides the exact amount to refund. The proposer's `AccountId` must be stored in `UpdateEntry` at proposal time (it is currently not stored there, so this requires adding a `proposer: AccountId` field to `UpdateEntry` in `propose()`).

```rust
// In do_update(), before self.entries.clear():
for (_, entry) in self.entries.iter() {
    let refund = required_deposit(entry.bytes_used);
    Promise::new(entry.proposer.clone()).transfer(refund).detach();
}
self.entries.clear();
```

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, attaching ~5 NEAR deposit. `UpdateEntry { update, bytes_used: ~500_000 }` is stored.
2. Participant B calls `propose_update` with a different binary, attaching ~5 NEAR deposit.
3. Threshold participants vote for B's proposal via `vote_update`.
4. `do_update` is called: B's entry is removed and executed; `self.entries.clear()` silently drops A's entry.
5. A's ~5 NEAR deposit remains in the contract balance forever. A has no function to call to recover it.

The test `test_proposed_updates_do_update_clears_all_state` confirms that all entries are cleared on execution but makes no assertion about deposit refunds, demonstrating the gap: [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

## Contract Updates

Participants can propose and vote on contract updates (code or configuration changes). When an update receives sufficient votes and is executed (via the `vote_update` endpoint which calls `do_update` internally), all pending update proposals and votes are cleared as they are no longer be valid after the contract migration. The update ID counter is preserved across migrations as part of the contract state to avoid race conditions where multiple participants might propose updates with colliding IDs immediately after an upgrade.
