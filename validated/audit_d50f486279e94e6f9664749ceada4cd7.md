### Title
Stale `vote_add_domains` Votes from Removed Participants Persist After Resharing, Enabling Unauthorized Domain Addition — (File: `crates/contract/src/state/running.rs`)

---

### Summary

After a resharing completes, stale `vote_add_domains` votes from removed participants are not immediately purged. The cleanup is dispatched as a **detached promise** (a separate asynchronous NEAR receipt). In the one-block window before that receipt executes, a single current participant can call `vote_add_domains` with the same domain configuration that removed participants previously voted for, causing the unanimity check to pass with far fewer legitimate votes than required.

---

### Finding Description

`vote_reshared` transitions the contract to the new Running state and then spawns several cleanup promises as detached calls: [1](#0-0) 

The comment on line 1176 explicitly acknowledges that only `vote_update` has an inline filter to tolerate cleanup failure:

> *"Note: MpcContract::vote_update uses filtering to ensure correctness even if this cleanup fails."*

No equivalent filter exists for `vote_add_domains`. Its vote-counting path in `RunningContractState::vote_add_domains` is: [2](#0-1) 

```rust
let participant = AuthenticatedParticipantId::new(self.parameters.participants())?;
let n_votes = self.add_domains_votes.vote(domains.clone(), &participant);
if self.parameters.participants().len() as u64 == n_votes {
```

`AuthenticatedParticipantId::new` only validates that the **caller** is a current participant. The `n_votes` value returned by `vote()` is the raw total count for that proposal, including votes from participants who were removed during resharing. The `tee_state.rs` test for the structurally identical `CodeHashesVotes` type confirms this behaviour explicitly: [3](#0-2) 

> *"P2 votes for the same malicious hash — should be only 1 vote, not 3"* — the comment documents that without cleanup the count would be 3 (2 stale + 1 fresh).

---

### Impact Explanation

**Concrete scenario (N=5 → N=3 resharing):**

1. Participants P1–P5 are running. `vote_add_domains` requires unanimity (all 5 votes).
2. P1 and P2 vote for domain configuration D before resharing.
3. Resharing removes P1 and P2; new participant set is {P3, P4, P5} (N=3).
4. The cleanup promise (`CLEAN_TEE_STATUS` / `remove_non_participant_update_votes`) is scheduled but executes in the **next block**.
5. In the same block as resharing completion, P3 calls `vote_add_domains(D)`.
6. `n_votes = 3` (P1 stale + P2 stale + P3 new); `participants().len() = 3` → `3 == 3` → unanimity check passes.
7. The contract transitions to `InitializingContractState` and begins key generation for domain D — authorized by only **1 of 3** current participants.

This breaks the production safety invariant that all current participants must consent to domain addition, and triggers unauthorized key generation for the new domain.

**Impact class:** Medium — contract execution-flow manipulation that breaks governance/accounting invariants (unauthorized `InitializingContractState` transition and key generation) without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

- The attack window is exactly one NEAR block (~1 second), but NEAR transactions can be pre-signed and submitted immediately after the resharing receipt finalizes.
- The attacker only needs to be **one** current participant (no threshold collusion required).
- The attacker must know which domain configuration removed participants voted for — this is on-chain public state readable from `running.add_domains_votes.proposal_by_account`.
- Resharing events are observable on-chain, so the timing is predictable.

Likelihood: **Medium** — requires a malicious current participant and precise timing, but both conditions are realistic in an adversarial setting.

---

### Recommendation

Apply the same inline filtering used by `vote_update` to `vote_add_domains` (and `vote_new_parameters`): when computing `n_votes`, count only votes whose voter is still present in `self.parameters.participants()`. This eliminates the dependency on the asynchronous cleanup promise for correctness.

Alternatively, perform the stale-vote cleanup **synchronously** inside `vote_reshared` before returning, rather than via a detached promise, so no window exists. [4](#0-3) 

---

### Proof of Concept

```
State: Running, participants = {P1, P2, P3, P4, P5}, unanimity threshold = 5

Block B:
  P1 → vote_add_domains(D)   // add_domains_votes: {P1→D}
  P2 → vote_add_domains(D)   // add_domains_votes: {P1→D, P2→D}

Block B+k:
  vote_reshared() completes  // new participants = {P3, P4, P5}
                             // cleanup promise scheduled for block B+k+1
  P3 → vote_add_domains(D)   // n_votes = 3 (P1 stale + P2 stale + P3)
                             // participants().len() = 3  →  3 == 3  →  TRANSITION
                             // → InitializingContractState (key generation begins)

Block B+k+1:
  cleanup promise runs       // too late; state already transitioned
```

The sandbox test `add_domain_votes_from_kicked_out_participants_are_cleared_after_resharing` verifies cleanup works correctly **after** the promise runs, but does not test the one-block window before it executes. [5](#0-4)

        domains: Vec<DomainConfig>,

        }
        let num_participants = u64::try_from(self.parameters.participants().len())
            .expect("participant count fits in u64");
        for domain in &domains {
            validate_domain_purpose(domain)?;
            validate_domain_threshold(domain, num_participants)?;
        }
        // Keep trust assumptions consistent: a domain must never require more shares to
        // reconstruct than the GovernanceThreshold demands to govern. Route through the
        // canonical helper so the cross-domain invariant has a single source of truth.
        ThresholdParameters::validate_governance_against_reconstruction(
            num_participants,
            self.parameters.threshold(),
            max_reconstruction_threshold(&domains),
        )?;
        let participant = AuthenticatedParticipantId::new(self.parameters.participants())?;
        let n_votes = self.add_domains_votes.vote(domains.clone(), &participant);

                generated_keys: self.keyset.domains.clone(),
                domains: new_domains,

                    self.keyset.epoch_id,
                    domains[0].clone(),
                    self.parameters.clone(),

                cancel_votes: BTreeSet::new(),

            Ok(None)
