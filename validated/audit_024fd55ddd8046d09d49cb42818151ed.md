### Title
TOCTOU Double-Settlement: `fast_fin_transfer_to_near_callback` Does Not Re-Check `finalised_transfers`, Enabling Concurrent `fin_transfer` to Double-Credit the Same Transfer — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`fast_fin_transfer` performs its uniqueness check against `finalised_transfers` synchronously at call time, but the state-writing counterpart (`add_fast_transfer` into `fast_transfers`) happens in an async callback two receipt-hops later. `fin_transfer` writes to `finalised_transfers` in its own async callback. Because `fin_transfer` has fewer cross-contract hops than `fast_fin_transfer` (for a NEAR recipient), a trusted relayer can submit both for the same transfer in the same block and have `fin_transfer_callback` commit first. When `fast_fin_transfer_to_near_callback` subsequently runs, it checks only `fast_transfers` (not `finalised_transfers`), finds nothing, and sends tokens a second time — producing double-credit.

---

### Finding Description

**Entrypoint and async gap in `fast_fin_transfer`**

`fast_fin_transfer` (called from `ft_on_transfer`) performs its only dedup check at line 782:

```rust
if self.is_unified_transfer_finalised(&fast_fin_transfer_msg.transfer_id) {
    env::panic_str(BridgeError::TransferAlreadyFinalised...);
}
``` [1](#0-0) 

`is_unified_transfer_finalised` reads `finalised_transfers`: [2](#0-1) 

After the check passes, for a NEAR recipient the function schedules a two-hop async chain — `check_or_pay_ft_storage` → `fast_fin_transfer_to_near_callback` — without writing anything to `finalised_transfers` or `fast_transfers` in the synchronous frame: [3](#0-2) 

**`fast_fin_transfer_to_near_callback` has no re-check**

When the callback eventually executes, it calls `add_fast_transfer` and immediately sends tokens. There is no guard against `finalised_transfers`: [4](#0-3) 

`add_fast_transfer` only checks for a duplicate key in `fast_transfers`, not in `finalised_transfers`: [5](#0-4) 

**`fin_transfer_callback` writes `finalised_transfers` and sends tokens independently**

`process_fin_transfer_to_near` calls `add_fin_transfer` (which inserts into `finalised_transfers`) and then checks `fast_transfer_status`. If no fast transfer entry exists yet, it sends tokens to the original recipient: [6](#0-5) 

`add_fin_transfer` is the only guard against duplicate `fin_transfer` calls: [7](#0-6) 

**Receipt-hop count creates a deterministic ordering window**

For a NEAR-recipient fast transfer, the hop chain is:

```
ft_transfer_call → ft_on_transfer (fast_fin_transfer runs) → check_or_pay_ft_storage → fast_fin_transfer_to_near_callback
```

That is **2 additional hops** after `fast_fin_transfer` runs.

For `fin_transfer`:

```
fin_transfer → verify_proof → fin_transfer_callback
```

That is also **2 hops**, but starting from a later block if submitted after `fast_fin_transfer` runs. If the attacker submits `fin_transfer` in the same block that `fast_fin_transfer` executes, `fin_transfer_callback` arrives at the bridge contract in the same block as `fast_fin_transfer_to_near_callback`. Receipt ordering within that block determines which runs first. If `fin_transfer_callback` runs first:

1. `add_fin_transfer` inserts `TransferId{Eth, N}` into `finalised_transfers`.
2. `get_fast_transfer_status` returns `None` (callback hasn't run yet).
3. Tokens are sent to the original recipient. ← **CREDIT 1**

Then `fast_fin_transfer_to_near_callback` runs:

4. `add_fast_transfer` checks `fast_transfers` — empty, succeeds.
5. Tokens are sent to the recipient again. ← **CREDIT 2**

The two maps (`finalised_transfers` and `fast_transfers`) are disjoint; neither callback blocks on the other's map.

---

### Impact Explanation

The recipient receives `2 × amount_without_fee` tokens. The bridge has minted/unlocked tokens once (via `fin_transfer`) and also disbursed the relayer's deposited tokens (via `fast_fin_transfer`). The net effect is that the bridge's token supply is inflated by one full transfer amount with no corresponding locked collateral on the source chain. If the attacker controls the recipient account, they extract `amount_without_fee` tokens at the bridge's expense per exploited transfer.

---

### Likelihood Explanation

The attacker must be a trusted relayer (permissionless staking of ~1000 NEAR, 7-day waiting period) and must possess a valid proof for the transfer. Both conditions are achievable by any well-funded participant. The timing requirement (submitting `fin_transfer` within the same block that `fast_fin_transfer` executes) is straightforward to engineer by monitoring the mempool and submitting both transactions in the same block. No admin approval, key compromise, or external oracle manipulation is required.

---

### Recommendation

Add a `finalised_transfers` re-check at the top of `fast_fin_transfer_to_near_callback` before calling `add_fast_transfer`:

```rust
// In fast_fin_transfer_to_near_callback, before add_fast_transfer:
if self.is_unified_transfer_finalised(&fast_transfer.transfer_id) {
    env::panic_str(BridgeError::TransferAlreadyFinalised.to_string().as_str());
}
```

Alternatively, move the `add_fast_transfer` call (or an equivalent reservation into `finalised_transfers`) into the synchronous frame of `fast_fin_transfer` before any async hop, so the dedup state is committed atomically with the check.

---

### Proof of Concept

```
Block B:   Attacker calls ft_transfer_call(token, amount, FastFinTransfer{transfer_id: Eth:N, ...})
Block B+1: ft_on_transfer → fast_fin_transfer runs:
             - is_unified_transfer_finalised(Eth:N) → false (passes)
             - schedules check_or_pay_ft_storage → fast_fin_transfer_to_near_callback
           Attacker also calls fin_transfer(proof_for_Eth:N) in same block:
             - schedules verify_proof → fin_transfer_callback
Block B+2: check_or_pay_ft_storage executes (token contract)
           verify_proof executes (prover contract)
Block B+3: fin_transfer_callback executes FIRST:
             - add_fin_transfer(Eth:N) → inserts into finalised_transfers ✓
             - get_fast_transfer_status → None (fast_transfers still empty)
             - send_tokens(recipient, amount_without_fee)  ← CREDIT 1
           fast_fin_transfer_to_near_callback executes SECOND:
             - add_fast_transfer → checks fast_transfers only → succeeds
             - send_tokens(recipient, amount_without_fee)  ← CREDIT 2

Assert: recipient.balance == 2 × amount_without_fee  (double-credit confirmed)
```

A stateful near-workspaces integration test can reproduce this by submitting both transactions in the same block and asserting the recipient's final token balance equals exactly `amount_without_fee` (not `2 × amount_without_fee`).
