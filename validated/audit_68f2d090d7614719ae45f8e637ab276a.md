### Title
`updateRSETHPrice()` and `updateRSETHPriceAsManager()` can be permanently blocked by ETH donation that causes fee to exceed `maxFeeMintAmountPerDay` — (File: `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle._updateRsETHPrice()` computes a protocol fee proportional to the increase in `totalETHInProtocol` and then calls `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)`. If the computed fee exceeds `maxFeeMintAmountPerDay`, the call reverts with `DailyFeeMintLimitExceeded`. Because both the public `updateRSETHPrice()` and the manager-gated `updateRSETHPriceAsManager()` route through the same internal `_updateRsETHPrice()`, an attacker who donates enough ETH directly to `LRTDepositPool` can permanently block all price updates until governance manually raises the daily fee cap.

---

### Finding Description

`LRTDepositPool` exposes a public `receive()` function with no access control. [1](#0-0) 

`getETHDistributionData()` counts the raw ETH balance of the deposit pool as part of the protocol's total ETH: [2](#0-1) 

`_getTotalEthInProtocol()` aggregates this into `totalETHInProtocol`, which is then compared against `previousTVL = rsethSupply * rsETHPrice` to derive `rewardAmount`: [3](#0-2) 

The protocol fee is computed from `rewardAmount` and then converted to an rsETH amount: [4](#0-3) 

`_checkAndUpdateDailyFeeMintLimit` reverts unconditionally when the fee amount exceeds `maxFeeMintAmountPerDay`: [5](#0-4) 

Both externally callable price-update functions call the same internal `_updateRsETHPrice()`: [6](#0-5) 

The daily reset sets `currentPeriodMintedFeeAmount = 0`, but the donated ETH remains in the pool, so the same oversized fee is recalculated on every subsequent call, making the block persistent across days. [7](#0-6) 

---

### Impact Explanation

With price updates blocked:

- `rsETHPrice` is frozen at a stale (below-actual) value.
- New depositors receive more rsETH than they should (`rsethAmountToMint = amount * assetPrice / rsETHPrice`), diluting existing holders.
- Withdrawers receive less ETH than they are owed because the withdrawal manager prices rsETH at the stale rate.
- The block persists indefinitely until the manager calls `setMaxFeeMintAmountPerDay` to raise the cap — an out-of-band governance action that is not self-healing.

**Impact: Medium — temporary (practically permanent until admin action) freezing of the price update mechanism, with secondary share/asset mis-accounting for all deposits and withdrawals made during the freeze.**

---

### Likelihood Explanation

The attacker must donate enough ETH so that `(donation * protocolFeeInBPS / 10_000) / newRsETHPrice > maxFeeMintAmountPerDay`. The required donation scales inversely with `protocolFeeInBPS` and directly with `maxFeeMintAmountPerDay`. For a protocol with a modest fee rate (e.g., 5 %) and a conservatively set daily fee cap, the required donation is achievable by a well-funded attacker. The attacker permanently loses the donated ETH (it stays in the pool), making this a griefing attack with a real cost.

**Likelihood: Low.**

---

### Recommendation

1. **Decouple fee minting from the price-update path.** Instead of reverting when the daily fee cap is exceeded, cap the fee at the remaining daily allowance and carry the excess forward, or skip fee minting for that call and emit an event.
2. **Alternatively**, make `_checkAndUpdateDailyFeeMintLimit` a soft guard: if the fee would exceed the cap, mint only up to the cap and update the price anyway, so the price-update path is never blocked by the fee limit.
3. **Separately**, consider whether `LRTDepositPool.receive()` should be restricted to known callers (node delegators, reward receivers, etc.) to prevent arbitrary ETH inflation of `totalETHInProtocol`.

---

### Proof of Concept

1. Assume `protocolFeeInBPS = 500` (5 %), `maxFeeMintAmountPerDay = 10 ether` (in rsETH terms), `rsETHPrice ≈ 1.05 ETH`, and current TVL = 10 000 ETH.
2. Attacker sends `~210 ETH` directly to `LRTDepositPool` via `receive()`.
3. `totalETHInProtocol` increases by 210 ETH → `rewardAmount = 210 ETH`.
4. `protocolFeeInETH = 210 * 500 / 10_000 = 10.5 ETH`.
5. `rsethAmountToMintAsProtocolFee = 10.5 / 1.05 = 10 rsETH` — just over the 10 rsETH cap.
6. `_checkAndUpdateDailyFeeMintLimit(10 rsETH + ε)` reverts with `DailyFeeMintLimitExceeded`.
7. Both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` revert.
8. The donated ETH remains in the pool; the same revert occurs every day until the manager raises `maxFeeMintAmountPerDay`. [8](#0-7) [9](#0-8) [10](#0-9)

    function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
