### Title
Missing Refund of Excess Deposit in `submit_participant_info` Re-submission Path — (`File: crates/contract/src/lib.rs`)

---

### Summary

`MpcContract::submit_participant_info` is a `#[payable]` function that conditionally handles deposit validation and refunds. When an **existing participant** re-submits their attestation, the entire deposit-handling block is skipped. Any NEAR tokens attached to such a call are accepted by the contract and permanently locked — never refunded.

---

### Finding Description

`submit_participant_info` is marked `#[payable]`, meaning it accepts an arbitrary NEAR token deposit from the caller. After inserting the attestation, the function computes a boolean guard:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... deposit sufficiency check ...
    // ... refund of excess deposit ...
}
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — which happens when the caller **is** an existing voter/participant **and** the attestation is an update (not a first-time insertion) — the entire block is bypassed. The function returns `Ok(())` without ever reading `env::attached_deposit()` or scheduling a refund `Promise`. Any deposit attached to the call is silently absorbed by the contract.

Contrast this with the two other payable paths in the same file:

- `require_deposit` (used by `sign` / `request_app_private_key`) always refunds excess. [2](#0-1) 
- `propose_update` explicitly refunds excess after the storage cost is deducted. [3](#0-2) 

The re-submission branch of `submit_participant_info` has no equivalent refund path.

---

### Impact Explanation

Any NEAR tokens attached to a re-submission call by an existing participant are permanently locked in the contract with no recovery mechanism. This breaks the balance/accounting invariant that a `#[payable]` function should either consume the deposit for a defined purpose or return the excess to the caller. The impact maps to:

> **Medium. Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants.**

---

### Likelihood Explanation

MPC nodes call `submit_participant_info` on a periodic cadence (approximately every hour) and on attestation-removal events, as documented in the design doc. [4](#0-3) 

The function is `#[payable]`, so the NEAR SDK does not reject calls that carry a deposit. A node misconfiguration, SDK default, or future refactor that attaches even 1 yoctoNEAR to a re-submission call would silently lock those funds. Because the periodic submission is automated and the re-submission path is the common steady-state path (nodes re-attest continuously), the surface is exercised frequently.

---

### Recommendation

Add an unconditional refund of any excess deposit **outside** the `if attestation_storage_must_be_paid_by_caller` block, mirroring the pattern already used in `require_deposit`:

```rust
// After the conditional storage-payment block:
let attached = env::attached_deposit();
if attached > NearToken::from_yoctonear(0) {
    // Refund any deposit not consumed by storage costs
    // (for the re-submission path, the full deposit is excess)
    Promise::new(account_id).transfer(attached).detach();
}
```

Alternatively, require `env::attached_deposit() == NearToken::from_yoctonear(0)` in the re-submission branch and panic if a deposit is attached, preventing accidental fund loss.

---

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` for the first time. Storage cost is charged and excess is refunded correctly (the `if` block executes).
2. `alice.near` is now a voter (`self.voter_account().is_ok()`), so `caller_is_not_participant = false`.
3. `alice.near` calls `submit_participant_info` again (periodic re-attestation). `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` (not `NewlyInsertedParticipant`), so `is_new_attestation = false`.
4. `attestation_storage_must_be_paid_by_caller = false || false = false`.
5. The `if` block is skipped entirely. [5](#0-4) 
6. If `alice.near`'s node attached any deposit (e.g., 1 NEAR due to misconfiguration), `env::attached_deposit()` is never read, no refund `Promise` is scheduled, and the 1 NEAR is permanently locked in the contract.

The only caller of `submit_participant_info` in production is `mpc-node`'s `periodic_attestation_submission` task, which resubmits on a 1-hour cadence and on attestation-removal events. It already polls contract state to confirm the attestation is actually stored, with exponential backoff (100 ms → 60 s, capped at 12 h). That polling-based success criterion is what makes the sync→async change transparent. Under yield-resume the returned `Promise` also resolves with the actual outcome — success, a verifier-rejection error, or a post-DCAP-failure error (each as soon as the verifier answers), or a timeout error after ~200 blocks if it never does — so any future caller that wants to await the result synchronously can, without changing the contract.
