Audit Report

## Title
Missing `amount` in ETH Deposit Limit Check Allows Deposit Cap Bypass — (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison for ETH versus ERC-20 assets: the ETH branch checks only `totalAssetDeposits > limit` while the ERC-20 branch correctly checks `totalAssetDeposits + amount > limit`. As a result, any depositor can push the protocol's ETH holdings arbitrarily above the configured cap in a single transaction, as long as the cap has not already been exceeded before the call.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, `_checkIfDepositAmountExceedesCurrentLimit` contains the following logic:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
``` [1](#0-0) 

The ETH branch evaluates `totalAssetDeposits > limit`, which is `false` whenever the cap has not yet been reached. The incoming `amount` is never added, so the check never tests whether the deposit itself would breach the cap. The function returns `false`, and `_beforeDeposit` (lines 648–670) does not revert. [2](#0-1) 

The public entry point `depositETH` (lines 76–93) is `payable`, `nonReentrant`, and `whenNotPaused` — no role or special state is required. [3](#0-2) 

**Exploit flow:**
1. `totalAssetDeposits(ETH) = X` where `X ≤ depositLimit`.
2. Attacker calls `depositETH{value: Y}("")` with arbitrarily large `Y`.
3. Check evaluates `X > depositLimit` → `false` → no revert.
4. `_mintRsETH` mints rsETH proportional to `Y`.
5. `totalAssetDeposits(ETH)` becomes `X + Y`, potentially orders of magnitude above the cap.

## Impact Explanation
The ETH deposit cap is a risk-management invariant. Bypassing it allows a single depositor to inflate the protocol's ETH TVL without bound. No funds are directly stolen and the depositor receives rsETH at the prevailing rate, but the protocol operates outside its designed safety envelope: restaking capacity may be exceeded and subsequent depositors interact with a protocol whose risk parameters have been violated. This matches **Low — contract fails to deliver its promised invariant (the deposit cap) without losing value**.

## Likelihood Explanation
The entry point is the public, permissionless `depositETH` function. No privileged role, no special state, and no front-running is required. The condition is met whenever `totalAssetDeposits(ETH) ≤ depositLimit`, which is the normal operating state of the protocol. Any depositor with sufficient ETH can trigger this in a single transaction.

## Recommendation
Apply the same forward-looking check to ETH that is already applied to ERC-20 assets:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
```
depositLimit(ETH)       = 100 ETH
totalAssetDeposits(ETH) = 50 ETH  (before attack)

Attacker calls: depositETH{value: 10_000 ETH}("")

_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ETH):
  totalAssetDeposits = 50 ETH
  return (50 > 100)  →  false   ← cap check passes, amount ignored

_mintRsETH mints rsETH for 10_000 ETH at current rate.

totalAssetDeposits(ETH) after = 10_050 ETH  (100× over the cap)
```

**Foundry test plan:** Deploy `LRTDepositPool` with `depositLimit(ETH) = 100 ether`. Call `depositETH{value: 10_000 ether}("")` from an unprivileged address. Assert that `getTotalAssetDeposits(ETH_TOKEN)` equals `10_050 ether` (i.e., far above the cap) and that no revert occurred, confirming the cap was bypassed.
