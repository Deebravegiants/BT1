### Title
Detached Refund Transfer Return Value Not Checked — Excess Deposits Silently Locked on Transfer Failure - (`File: crates/contract/src/lib.rs`)

---

### Summary

The NEAR MPC contract schedules excess-deposit refunds using `Promise::new(...).transfer(diff).detach()` in three locations. Calling `.detach()` is the NEAR analog of Solidity's unchecked low-level `.call{value: ...}()`: the promise is fire-and-forget, its success or failure is never observed, and any failure is silently swallowed. If the refund transfer fails, the excess deposit is permanently locked inside the contract with no revert and no error surfaced to the caller.

---

### Finding Description

Three refund sites in `crates/contract/src/lib.rs` use the detached-promise pattern:

**1. `require_deposit` (line 137)** — called from every user-facing payable entry point (`sign`, `verify_foreign_transaction`, `request_app_private_key`):

```rust
Promise::new(predecessor.clone()).transfer(diff).detach();
``` [1](#0-0) 

**2. `submit_participant_info` (line 847)** — storage-cost overpayment refund:

```rust
Promise::new(account_id).transfer(diff).detach();
``` [2](#0-1) 

**3. `propose_update` (line 1330)** — excess deposit refund to the proposing participant:

```rust
Promise::new(proposer).transfer(diff).detach();
``` [3](#0-2) 

In all three cases the promise result is never inspected. NEAR's runtime will silently drop the failed receipt; the contract state is not rolled back, and the caller receives no indication that the refund did not land.

The primary failure condition is that the recipient account no longer exists when the scheduled receipt executes. In NEAR, a contract account can schedule a `delete_account` action in a subsequent receipt within the same transaction block. A calling contract that sends excess deposit and then self-destructs (or whose account is deleted by a concurrent transaction) will find its refund receipt failing silently, with the excess NEAR permanently retained by the MPC contract.

---

### Impact Explanation

**Medium — balance accounting invariant violation.**

The contract's documented invariant is that any deposit above the required minimum is returned to the caller. When the detached transfer fails, this invariant is broken: the MPC contract retains funds it is not entitled to, and the caller has no recourse. For `sign` / `verify_foreign_transaction` / `request_app_private_key`, the minimum deposit is 1 yoctoNEAR, so any caller who attaches a non-trivial deposit (e.g., to satisfy a function-call access-key allowance) and whose account is subsequently deleted loses the entire excess. For `propose_update`, the required deposit is the storage cost of the code blob, which can be substantial; an overpaying participant whose account is deleted loses the difference permanently.

The locked funds accumulate in the contract balance and are not recoverable without a governance upgrade.

---

### Likelihood Explanation

**Low.** The failure window is narrow: the recipient account must be absent when the refund receipt executes, which in practice requires the account to be deleted in the same or an immediately following block. This is realistic for:

- A calling contract that schedules `delete_account` in the same transaction as the `sign` call (the delete receipt runs after the sign receipt but before the detached refund receipt in some scheduling orders).
- An operator account that is deleted by a concurrent governance action between the `propose_update` call and the refund receipt.

Neither scenario is common in normal operation, but both are reachable without any privileged access.

---

### Recommendation

Replace each `.detach()` refund with a chained callback that verifies the transfer succeeded, or — following the pattern already used elsewhere in the contract — return the refund as part of the main promise chain so that a failed transfer causes the entire receipt to revert:

```rust
// Instead of:
Promise::new(predecessor.clone()).transfer(diff).detach();

// Use (chain the refund so failure is observable):
return Promise::new(predecessor.clone()).transfer(diff);
// or assert the result in a .then callback
```

For entry points that already return a `PromiseOrValue`, the refund promise can be chained as the final step so that its failure propagates to the caller.

---

### Proof of Concept

1. Deploy a calling contract `attacker.near` that:
   - Calls `mpc_contract.sign(...)` attaching 1 NEAR (excess above the 1 yoctoNEAR minimum).
   - In the same transaction, schedules `Promise::new(env::current_account_id()).delete_account(beneficiary)`.
2. The NEAR runtime executes receipts in dependency order. The `sign` receipt runs first; `require_deposit` schedules a detached refund receipt for `attacker.near` and enqueues the yield.
3. The `delete_account` receipt runs next, removing `attacker.near`.
4. The detached refund receipt executes and attempts `transfer(~1 NEAR)` to the now-deleted `attacker.near`. The receipt fails silently.
5. The MPC contract retains the ~1 NEAR excess deposit. No error is surfaced; the `sign` yield proceeds normally.

The same sequence applies to `propose_update` (line 1330) and `submit_participant_info` (line 847). [4](#0-3) [5](#0-4) [6](#0-5)

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}

    /// endpoint.
    #[payable]
    #[handle_result]
    pub fn submit_participant_info(
        &mut self,
        proposed_participant_attestation: dtos::Attestation,
        tls_public_key: dtos::Ed25519PublicKey,
    ) -> Result<(), Error> {
        let proposed_participant_attestation =
            proposed_participant_attestation.try_into_contract_type()?;

        let account_key = env::signer_account_pk();
        let account_id = Self::assert_caller_is_signer();

        log!(
            "submit_participant_info: signer={}, proposed_participant_attestation={:?}, account_key={:?}",
            account_id,
            proposed_participant_attestation,
            account_key
        );

        // Save the initial storage usage to know how much to charge the proposer for the storage
        // used
        let initial_storage = env::storage_usage();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        // The node always signs submissions with an Ed25519 key
        // (`near_signer_key`), so the signer key here is Ed25519 in practice.
        // Reject non-Ed25519 signer keys rather than silently storing a value
        // we could never match against in `is_caller_an_attested_participant`.
        let account_public_key = dtos::Ed25519PublicKey::try_from(&account_key).map_err(|_| {
            InvalidParameters::InvalidTeeRemoteAttestation {
                reason: "signer account key must be Ed25519".to_string(),
            }
        })?;

        // Add the participant information to the contract state
        let attestation_insertion_result = self
            .tee_state
            .add_participant(
                NodeId {
                    account_id: account_id.clone(),
                    tls_public_key,
                    account_public_key,
                },
                proposed_participant_attestation,
                tee_upgrade_deadline_duration,
            )
            .map_err(|err| {
                let reason = match &err {
                    AttestationSubmissionError::InvalidAttestation(_) => {
                        format!("TeeQuoteStatus is invalid: {err}")
                    }
                    AttestationSubmissionError::TlsKeyOwnedByOtherAccount => err.to_string(),
                };
                InvalidParameters::InvalidTeeRemoteAttestation { reason }
            })?;

        let caller_is_not_participant = self.voter_account().is_err();
        let is_new_attestation = matches!(
            attestation_insertion_result,
            ParticipantInsertion::NewlyInsertedParticipant
        );

        let attestation_storage_must_be_paid_by_caller =
            is_new_attestation || caller_is_not_participant;

        if attestation_storage_must_be_paid_by_caller {
            // `saturating_sub`: if a re-submission shrinks the entry, charge nothing
            // rather than underflow. Intentional asymmetry: we do not refund freed bytes
            // either — the caller already paid for the larger entry, and we'd rather
            // accept that asymmetry than open a refund path for payload-shrinking games.
            let storage_used = env::storage_usage().saturating_sub(initial_storage);
            let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
            let attached = env::attached_deposit();

            if attached < cost {
                return Err(InvalidParameters::InsufficientDeposit {
                    attached: attached.as_yoctonear(),
                    required: cost.as_yoctonear(),
                }
                .into());
            }

            // Refund the difference if the proposer attached more than required
            if let Some(diff) = attached.checked_sub(cost)
                && diff > NearToken::from_yoctonear(0)
            {
                Promise::new(account_id).transfer(diff).detach();
            }
        }

        Ok(())
    }
