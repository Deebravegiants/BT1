Audit Report

## Title
Incorrect ETH Deposit Limit Check Omits Current Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison: the ETH branch checks `totalAssetDeposits > depositLimit` (omitting `amount`), while the ERC20 branch correctly checks `totalAssetDeposits + amount > depositLimit`. When `totalAssetDeposits == depositLimit`, the ETH check returns `false` and the deposit proceeds, pushing total ETH deposits above the configured cap and minting rsETH beyond the intended ceiling.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit` does not include the incoming `amount` in its comparison:

```solidity
// L678-681
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount omitted
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
```

This function is called unconditionally from `_beforeDeposit` (L661), which is the sole guard for `depositETH` (L76–93, public, payable, no role restriction). When `totalAssetDeposits == depositLimit`, the ETH check evaluates `depositLimit > depositLimit` → `false`, so `_beforeDeposit` does not revert and the deposit proceeds. After the call, `totalAssetDeposits` becomes `depositLimit + msg.value`, exceeding the cap. The ERC20 path correctly evaluates `depositLimit + amount > depositLimit` → `true` and reverts. No other guard in `depositETH` or `_beforeDeposit` catches this case.

## Impact Explanation
The deposit limit is the primary mechanism bounding protocol ETH exposure. Bypassing it causes the contract to mint rsETH beyond the intended ceiling, violating the invariant `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`. No funds are directly stolen, but the protocol delivers more rsETH than the limit permits. This matches the allowed impact: **Low — Contract fails to deliver promised returns**.

## Likelihood Explanation
The condition `totalAssetDeposits == depositLimit` is reachable in normal operation whenever the cap is approached. Any depositor monitoring on-chain state can time a single `depositETH` call at that exact moment. No special role, flash loan, or multi-step setup is required. After the first over-limit deposit, subsequent calls will be blocked (since `totalAssetDeposits > depositLimit` now returns `true`), but the cap has already been breached by one deposit.

## Recommendation
Apply the same `+ amount` inclusion to the ETH branch, unifying the check:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This also aligns with `getAssetCurrentLimit` (L399–409), which already returns `0` when `totalAssetDeposits >= depositLimit`.

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Protocol accumulates `totalAssetDeposits(ETH) = 1000 ether` through normal deposits.
3. Attacker calls `depositETH{value: 10 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the attacker.
6. `totalAssetDeposits(ETH)` is now `1010 ether`, exceeding the configured cap.
