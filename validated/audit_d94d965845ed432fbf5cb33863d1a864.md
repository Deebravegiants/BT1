### Title
External Caller's Attached NEAR Deposit Permanently Stuck in WalletContract on Intermediate Callback Early-Return Paths - (File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs)

### Summary

The `WalletContract` (ETH-implicit account wallet) implements a `CallerDeposit` mechanism to refund external callers' attached NEAR deposits when a cross-contract call fails. However, multiple early-return error paths in `address_check_callback` and `nep_141_storage_balance_callback` silently drop the `caller_deposit` without issuing a refund. Any NEAR tokens attached by an external caller to `rlp_execute` are permanently absorbed into the wallet contract's balance when these paths are triggered.

### Finding Description

`rlp_execute` is `#[payable]` and accepts an attached deposit from any external caller. The deposit is tracked via `CallerDeposit::new` and is supposed to be refunded in `rlp_execute_callback` if the downstream promise fails. [1](#0-0) [2](#0-1) 

The refund is correctly implemented in `rlp_execute_callback`: [3](#0-2) 

However, two intermediate callbacks have early-return paths that never reach `rlp_execute_callback`, causing the `caller_deposit` to be silently dropped:

**`address_check_callback` — three unrefunded early returns:**

1. Registrar call fails (`PromiseResult::Failed`): [4](#0-3) 

2. Registrar returns unparseable JSON: [5](#0-4) 

3. Target address is registered but relayer is not using an access key: [6](#0-5) 

**`nep_141_storage_balance_callback` — three unrefunded early returns:**

4. NEP-141 `storage_balance_of` call fails: [7](#0-6) 

5. NEP-141 contract returns unparseable JSON: [8](#0-7) 

6. Action is not a `FunctionCall` (unexpected action type in ERC-20 path): [9](#0-8) 

In all six cases, the `caller_deposit: Option<CallerDeposit>` parameter is received but dropped without issuing a transfer back to the caller. The NEAR tokens are already in the wallet contract's balance (transferred at `rlp_execute` call time) and have no recovery path.

### Impact Explanation

The corrupted protocol value is the external caller's NEAR account balance. Tokens attached to `rlp_execute` are permanently absorbed into the ETH-implicit wallet contract's balance. The caller cannot recover them: the wallet contract has no withdrawal method for arbitrary balances, and the wallet owner (the ETH key holder) has no incentive or mechanism to return them. The affected balance entry is the caller's `account.amount()` in the NEAR state trie.

### Likelihood Explanation

The `EOABaseTokenTransfer` with `address_check: Some(address)` path is triggered whenever a relayer sends a base-token transfer to another eth-implicit account whose address might be registered. The registrar is an external contract; it can fail due to temporary unavailability, a panic in its logic, or being upgraded with breaking changes. Similarly, any NEP-141 token contract can return a failed promise or non-standard JSON from `storage_balance_of`. An external DeFi integration or relayer service that attaches NEAR as a deposit to `rlp_execute` (e.g., to cover a fee) and hits either of these failure modes loses its deposit permanently. This is a realistic production scenario.

### Recommendation

In every early-return error path inside `address_check_callback` and `nep_141_storage_balance_callback`, issue a refund transfer to the caller before returning, mirroring the pattern already used in `rlp_execute_callback`:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund_promise = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(
        refund_promise,
        NearToken::from_yoctonear(yocto_near.into()),
    );
}
```

This should be added to all six early-return sites identified above.

### Proof of Concept

1. External caller (e.g., a DeFi relayer) calls `rlp_execute` on a victim ETH-implicit wallet contract, attaching 3 NEAR as a deposit. The Ethereum transaction encodes a base-token transfer to another eth-implicit account whose address might be registered.
2. `inner_rlp_execute` detects `EOABaseTokenTransfer { address_check: Some(address), .. }` and schedules a registrar lookup followed by `address_check_callback`. [10](#0-9) 
3. The registrar contract call fails (e.g., registrar is temporarily unavailable or panics).
4. `address_check_callback` is invoked with `PromiseResult::Failed`. It returns early at line 142–147 without touching `caller_deposit`. [11](#0-10) 
5. `rlp_execute_callback` is never scheduled; the 3 NEAR deposit remains in the wallet contract's balance permanently. The caller's NEAR account balance is reduced by 3 NEAR with no recourse.
