### Title
Response-to-Request Binding Missing in `respond_verify_foreign_tx` Allows Byzantine Node to Inject Cross-Request Responses — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` validates the cryptographic signature and the pending-request key **independently**, not as a bound pair. A single Byzantine MPC node (below the signing threshold) can take a valid threshold signature produced for `request_A` and submit it as the response to a different pending `request_B`. The contract accepts the call because the signature is valid over `response.payload_hash` and `request_B` exists in the pending map — but it never checks that `response.payload_hash` is the correct hash for `request_B`.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs` lines 691–754), the two critical checks are:

1. **Signature check** (lines 718–743): verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key.
2. **Pending-request lookup** (lines 749–753): resolves the yield queue keyed on `request`. [1](#0-0) 

There is **no check** that `response.payload_hash` equals `hash(request.request, extracted_values)` — i.e., that the signed hash actually corresponds to the pending `request`. The `payload_hash` field is entirely attacker-supplied and is only required to have a valid signature over it.

Compare this to the regular `respond` function, which derives the payload directly from `request.payload` (a field of the request struct itself), making it impossible to inject a mismatched payload: [2](#0-1) 

The `VerifyForeignTransactionRequest` struct does not contain the extracted values (those are determined by MPC nodes at query time), so the contract cannot independently recompute the expected `payload_hash` from the request alone: [3](#0-2) 

The `ForeignTxSignPayload` that nodes sign includes both the request and the extracted values, but only the hash is returned in the response — the contract has no way to reconstruct it: [4](#0-3) 

---

### Impact Explanation

A single Byzantine MPC node (an attested participant, strictly below the signing threshold) can:

1. Wait for the MPC network to legitimately produce a threshold signature `response_A` for `request_A` (e.g., verifying Bitcoin tx_A).
2. Call `respond_verify_foreign_tx(request_B, response_A)` where `request_B` is a different victim's pending request (e.g., verifying Bitcoin tx_B).
3. The contract accepts: signature over `response_A.payload_hash` is valid ✓, `request_B` exists in the pending map ✓.
4. The victim's yield is drained and they receive `response_A` — a valid MPC signature, but over the hash of a **different transaction's** extracted values.

The victim's bridge contract receives a `VerifyForeignTransactionResponse` with a valid threshold signature that attests to the wrong transaction. If the bridge contract only verifies the signature (without independently recomputing the expected `payload_hash` from the request and known extracted values), it will accept a forged foreign-chain verification — enabling invalid bridge execution or double-spend conditions.

Even if the bridge contract uses `ForeignChainSignatureVerifier::verify_signature` from the SDK (which does check `payload_hash`), the victim's pending yield is permanently consumed with the wrong response, requiring resubmission and breaking the request-lifecycle accounting invariant. [5](#0-4) 

---

### Likelihood Explanation

- The attacker is a single Byzantine MPC node — an attested participant, no threshold collusion required.
- Two concurrent pending `verify_foreign_transaction` requests are sufficient (common in any active bridge deployment).
- The Byzantine node only needs to reuse an existing valid threshold signature in the wrong context — no cryptographic forgery is required.
- The attack is reachable via a direct on-chain call to `respond_verify_foreign_tx` with a crafted `(request_B, response_A)` pair.

---

### Recommendation

Bind the response to the request by storing the expected `payload_hash` alongside the pending yield at submission time, or by requiring the responding node to include the full `ForeignTxSignPayload` (not just its hash) so the contract can recompute and verify the hash against the stored request.

Concretely, when `verify_foreign_transaction` enqueues the request, also store a commitment to the expected signing context (e.g., the `ForeignChainRpcRequest` borsh hash). In `respond_verify_foreign_tx`, verify that `response.payload_hash` is consistent with that stored commitment before resolving the yield. [6](#0-5) 

---

### Proof of Concept

```
Setup:
  - Two pending requests exist:
      request_A = VerifyForeignTransactionRequest { request: Bitcoin(tx_id=0xAA...), domain_id: 0, ... }
      request_B = VerifyForeignTransactionRequest { request: Bitcoin(tx_id=0xBB...), domain_id: 0, ... }
  - MPC network computes response_A for request_A:
      response_A.payload_hash = SHA256(borsh(ForeignTxSignPayloadV1 { request: Bitcoin(0xAA...), values: [BlockHash(0x11...)] }))
      response_A.signature    = valid threshold ECDSA sig over response_A.payload_hash

Attack (single Byzantine node):
  1. Byzantine node calls:
       respond_verify_foreign_tx(request = request_B, response = response_A)
  2. Contract checks:
       - verify_ecdsa_signature(response_A.signature, response_A.payload_hash, root_pk) → OK ✓
       - pending_verify_foreign_tx_requests.contains(request_B) → OK ✓
       - NO check: response_A.payload_hash == hash(request_B, ...) ← MISSING
  3. Contract resolves request_B's yield with response_A.
  4. Victim's bridge contract receives response_A (valid sig, wrong payload_hash).
  5. If bridge contract only checks signature validity → accepts forged verification of tx_B.
  6. Byzantine node then calls respond_verify_foreign_tx(request_A, response_A) → also succeeds.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
