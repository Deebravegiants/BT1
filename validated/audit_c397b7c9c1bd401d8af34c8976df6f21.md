### Title
Forged Foreign-Chain Verification via Cross-Request Signature Replay in `respond_verify_foreign_tx` — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies the MPC signature against a **caller-supplied** `payload_hash` without checking that the hash corresponds to the stored pending request. A single Byzantine MPC leader who has legitimately obtained a threshold signature for one request can replay it as a response to any other pending request, delivering a forged foreign-chain verification to waiting callers.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← entirely caller-supplied
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

After this check passes, the contract resolves all pending yields for the supplied `request` with the full `response`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The contract **never verifies** that `response.payload_hash` is the correct hash for the stored `request`. The correct hash is `SHA-256(borsh(ForeignTxSignPayload::V1 { request: stored_request, values: observed_values }))`, where `values` are the foreign-chain extracted values the MPC nodes observed. Because the contract does not know `values`, it cannot recompute the expected hash — but it also does not enforce any binding between the supplied hash and the stored request key.

The analogous check-effects-interactions flaw: the contract checks the signature (against caller-supplied data) and then commits state (resolves yields) without verifying the caller-supplied data is consistent with the stored request. This is the NEAR analog of the Tracer `withdraw()` pattern where a validity check runs against stale/unverified data before state is finalized.

**Cross-request replay attack path (single Byzantine leader, no threshold collusion):**

1. User B submits `verify_foreign_transaction(request_B)` → stored in `pending_verify_foreign_tx_requests[request_B]`.
2. The Byzantine leader participates honestly in the MPC signing session for `request_A` (a different, legitimate request). The t-of-n threshold protocol produces `(payload_hash_A, signature_A)` where `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload::V1 { request: request_A, values: values_A }))`.
3. The Byzantine leader calls `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: signature_A })`.
4. The contract checks: is `signature_A` valid over `payload_hash_A` under the root key? **YES** — it is a genuine threshold signature.
5. The contract checks: is `request_B` in the pending map? **YES**.
6. `resolve_yields_for` removes `request_B` from the map and resumes all waiting yields with `{ payload_hash: payload_hash_A, signature: signature_A }`.
7. User B's caller receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes `request_A`'s foreign-chain state, not `request_B`'s. [3](#0-2) 

---

### Impact Explanation

User B's bridge or application contract receives a `VerifyForeignTransactionResponse` that is cryptographically valid (the signature checks out) but semantically wrong: the `payload_hash` encodes the foreign-chain state for `request_A`, not `request_B`. Any bridge contract that does not independently reconstruct the expected hash from `request_B` and the expected extracted values — and compare it against `response.payload_hash` — will accept the forged attestation. This enables invalid bridge execution: the bridge releases funds or triggers actions based on a foreign-chain event that was never actually verified for the submitted transaction.

The `near-mpc-sdk` provides `ForeignChainSignatureVerifier::verify_signature` which does perform this check:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
``` [4](#0-3) 

However, this is an optional SDK-level check. The MPC contract itself does not enforce it, and callers who do not use the SDK or omit the check are exposed.

**Impact class:** High — forged foreign-chain verification enabling invalid bridge execution.

---

### Likelihood Explanation

The attack requires exactly **one** Byzantine MPC leader — a single compromised attested participant who happens to be elected leader for any signing session. The leader legitimately obtains a threshold signature (with the cooperation of t-1 honest nodes who sign `request_A` in good faith) and then replays it against any concurrently pending `request_B`. No threshold collusion is required; the honest nodes never learn their partial signatures were misused. Given that the MPC network is designed to tolerate up to t-1 Byzantine participants, a single compromised leader is within the explicit threat model.

---

### Recommendation

The contract should bind the response to the stored request. Since the contract cannot independently compute `values` (the foreign-chain extracted data), the cleanest fix is to require the responder to supply the full `ForeignTxSignPayload` (not just the hash), so the contract can:

1. Verify `payload.request == stored_request` (the request field matches what is in the pending map).
2. Recompute `expected_hash = SHA-256(borsh(payload))` and verify `expected_hash == response.payload_hash`.
3. Verify the signature over `expected_hash`.

This eliminates the ability to replay a signature from one request as a response to another, because the contract independently recomputes the hash from the stored request and the supplied payload.

Alternatively, at minimum, the contract should document and enforce (via a required SDK wrapper) that callers **must** verify `payload_hash` against their expected values before acting on the response.

### Proof of Concept

```
// Setup
User B calls: verify_foreign_transaction(request_B)
  → pending_verify_foreign_tx_requests[request_B] = [yield_B]

// Byzantine leader participates in honest signing for request_A
// Honest nodes sign payload_hash_A = SHA256(borsh(ForeignTxSignPayload::V1 { request_A, values_A }))
// Leader obtains: (payload_hash_A, signature_A)  ← valid threshold signature

// Attack
Byzantine leader calls:
  respond_verify_foreign_tx(
    request  = request_B,          // ← pending request (exists in map)
    response = {
      payload_hash: payload_hash_A, // ← hash for request_A, not request_B
      signature:    signature_A,    // ← valid threshold sig over payload_hash_A
    }
  )

// Contract execution
verify_ecdsa_signature(signature_A, payload_hash_A, root_pk) → OK  ✓
resolve_yields_for(pending[request_B], serialize(response))   → resumes yield_B

// User B receives:
VerifyForeignTransactionResponse {
  payload_hash: payload_hash_A,  // encodes request_A's foreign-chain state
  signature:    signature_A,
}
// Signature is valid. payload_hash is wrong. Bridge contract is deceived.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
