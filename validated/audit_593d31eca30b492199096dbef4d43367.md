### Title
Unprivileged Attacker Can Saturate `verify_foreign_transaction` Fan-Out Queue, Blocking Legitimate Bridge Callers — (`crates/contract/src/pending_requests.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint uses a **caller-agnostic** request key (no predecessor account ID), unlike `sign` and `request_app_private_key` which bind the key to the caller. Any unprivileged NEAR account can submit 128 identical `verify_foreign_transaction` requests for a specific foreign transaction (e.g., a Bitcoin `tx_id`), saturating the fan-out queue to its hard cap. Once full, every subsequent legitimate caller attempting to verify the same transaction receives a `PendingRequestQueueFull` panic. The attacker can continuously refill the queue at near-zero cost, permanently blocking bridge users from completing cross-chain transfers that depend on verifying that specific transaction.

---

### Finding Description

**Caller-agnostic request key for `verify_foreign_transaction`**

The `sign` and `request_app_private_key` functions both bind their pending-request map key to the caller's account ID:

```rust
// sign():
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // ← caller-bound
    &request.path,
);

// request_app_private_key():
let app_id = derive_app_id(predecessor_id, derivation_path);  // ← caller-bound
``` [1](#0-0) [2](#0-1) 

By contrast, `verify_foreign_transaction` converts its arguments into a `VerifyForeignTransactionRequest` that contains only `(domain_id, payload_version, request)` — no caller field:

```rust
// verify_foreign_transaction():
let request = args_into_verify_foreign_tx_request(request);
// ... queued under caller-agnostic key
``` [3](#0-2) 

The contract's own unit test explicitly documents this behavior:

> "a different account would today be blocked from receiving a response by alice's submission" [4](#0-3) 

**Fan-out queue cap enforcement**

`push_pending_yield` panics with `PendingRequestQueueFull` when the queue for a given key reaches `MAX_PENDING_REQUEST_FAN_OUT = 128`:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

pub(crate) fn push_pending_yield<K>(...) {
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }
                .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
}
``` [5](#0-4) 

**Attack path**

1. Attacker identifies a foreign transaction (e.g., a Bitcoin `tx_id`) that a bridge user needs to verify to complete a cross-chain transfer.
2. Attacker submits 128 `verify_foreign_transaction` calls for that exact `(domain_id, payload_version, tx_id, extractors)` tuple, each with 1 yoctonear deposit. Total cost: 128 yoctonear + gas ≈ negligible.
3. The queue for that request key is now at capacity.
4. The legitimate bridge user's `verify_foreign_transaction` call panics with `PendingRequestQueueFull`.
5. The attacker continuously resubmits as yields time out, sustaining the block indefinitely.

---

### Impact Explanation

This is a **Medium** impact: request-lifecycle manipulation that breaks the production safety invariant — any caller who submits a valid `verify_foreign_transaction` request with sufficient deposit must be able to get it processed — without relying on network-level DoS or operator misconfiguration.

In a bridge context (Omnibridge inbound flow), blocking verification of a specific foreign transaction prevents the bridge user from completing their cross-chain transfer. If the bridge has time-sensitive settlement windows, this can cause the user's funds to become inaccessible.

---

### Likelihood Explanation

- **Entry path**: Fully open — any NEAR account can call `verify_foreign_transaction` with 1 yoctonear.
- **Cost**: 128 yoctonear + gas per saturation cycle; trivially sustainable.
- **Target selection**: The attacker only needs to know the `tx_id` of the transaction to block, which is public on the foreign chain.
- **No privileged access required**: Unlike the Lens report's whitelisted profile creator, this requires zero special permissions.

---

### Recommendation

Bind the `verify_foreign_transaction` pending-request key to the caller's account ID (as `sign` and `request_app_private_key` already do), so that an attacker can only saturate their own queue slot, not a shared one. If caller-agnostic fan-out is intentional (to share MPC computation across callers for the same tx), introduce a per-caller sub-queue or a per-caller submission cap to prevent a single account from monopolizing the shared queue.

---

                .to_string(),

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
