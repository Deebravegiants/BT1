### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions via Deploy-Account Mempool Race — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function skips the `__validate__` entry-point call for an invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that check returns `true` for **any** transaction from the address — not specifically a deploy-account — an unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can inject a fake-signature invoke (nonce=1, higher fee) for the same address, bypass all cryptographic validation at the gateway, displace the victim's legitimate invoke via fee escalation, and cause the victim to pay fees for a reverted transaction they never authorized.

---

### Finding Description

**Trigger path:**

1. Alice submits `deploy_account(A)` + `invoke(A, nonce=1, calldata=X, fee=F)` simultaneously. The gateway admits the invoke because `account_tx_in_pool_or_recent_block(A)` is `true` (Alice's own deploy-account is in the pool), so `skip_validate=true` and `__validate__` is not called.

2. The attacker observes Alice's `deploy_account` in the public mempool, computes address `A`, and submits `invoke(A, nonce=1, calldata=DRAIN, fee=F+1, sig=GARBAGE)`.

3. The gateway evaluates the attacker's invoke:
   - Stateless validation passes (signature is a valid felt array, no cryptographic check).
   - `validate_nonce`: `account_nonce=0`, `tx_nonce=1` — within the allowed gap, passes.
   - `validate_by_mempool` → `mempool.validate_tx` checks only for duplicate hash and nonce range, not signature.
   - `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
   - `run_validate_entry_point` is called with `skip_validate=true`, which sets `execution_flags.validate=false` and returns `Ok(())` after `perform_pre_validation_stage` (nonce/fee-bound check only).

4. The attacker's invoke enters the mempool. Because it has a higher fee, `validate_fee_escalation` allows it to replace Alice's invoke at `(A, nonce=1)`.

5. At block execution: `deploy_account(A)` runs (Alice's account is deployed), then the attacker's `invoke(A, nonce=1, GARBAGE_SIG)` runs. The blockifier calls `__validate__` on Alice's deployed account; the signature is invalid; the transaction reverts. Starknet still charges the fee from Alice's pre-funded balance and increments her nonce to 2.

**Broken invariant:** Every invoke transaction admitted to the mempool must have passed cryptographic signature verification (via `__validate__`). The `skip_stateful_validations` path breaks this invariant by relying on a coarse address-presence check rather than verifying that the pending transaction for that address is specifically a deploy-account owned by the same signer.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The comment at line 441–443 states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is incorrect: the check does not distinguish between Alice's own deploy-account and any other transaction from address `A`, and it does not verify that the pending transaction was signed by the same key that will own the account.

**`account_tx_in_pool_or_recent_block` implementation:** [2](#0-1) 

Returns `true` for any transaction from the address, regardless of type or signer.

**`run_validate_entry_point` with `skip_validate=true`:** [3](#0-2) 

When `skip_validate=true`, `execution_flags.validate=false` is set and `StatefulValidator::perform_validations` returns after the pre-validation stage (nonce/fee bounds only), never calling `__validate__`. [4](#0-3) 

---

### Impact Explanation

- **Admission of invalid transaction:** The gateway accepts an invoke with an arbitrary (attacker-controlled) signature without any cryptographic check, violating the "High. Mempool/gateway/RPC admission accepts invalid transactions" impact criterion.
- **Economic loss:** Alice's pre-funded account balance is drained by the fee charged for the attacker's reverted transaction. The attacker can set `resource_bounds` to the maximum Alice pre-funded, maximizing the theft.
- **Nonce consumption:** Alice's nonce=1 is permanently consumed by the attacker's failed transaction; Alice must resubmit with nonce=2.
- **Legitimate transaction displacement:** Alice's original invoke is evicted from the mempool via fee escalation and never executed.

---

### Likelihood Explanation

- The attack requires only that the victim submits `deploy_account + invoke` simultaneously (the documented and tested UX pattern).
- The attacker needs only to monitor the public mempool for `deploy_account` transactions — no privileged access required.
- The fee escalation threshold is low; the attacker pays only a marginally higher fee than Alice.
- The attack window is the entire time Alice's `deploy_account` is pending in the mempool (potentially many blocks). [5](#0-4) 

The integration test explicitly documents and exercises this exact pattern, confirming it is a production flow.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account transaction** for the sender address is pending in the mempool. Concretely:

1. Add a `deploy_account_in_pool(address)` query to the mempool that returns `true` only if the pending transaction for `address` at nonce=0 is of type `DeployAccount`.
2. In `skip_stateful_validations`, replace the current check with this stricter query.
3. Alternatively, do not skip `__validate__` at the gateway at all; instead, allow the blockifier to skip it at execution time (where the deploy-account has already been confirmed to be in the same block).

---

### Proof of Concept

```
// Setup: Alice's address A is derived from (class_hash, salt, ctor_calldata, deployer=0)
// Alice pre-funds address A with balance B.

// Step 1: Alice submits to gateway

gateway.add_tx(invoke(sender=A, nonce=1, calldata=ALICE_CALL, fee=F, sig=ALICE_SIG))
// Both admitted; invoke skips __validate__ because deploy_account is in mempool.

// Step 2: Attacker observes deploy_account(A) in mempool, submits:
gateway.add_tx(invoke(sender=A, nonce=1, calldata=ATTACKER_CALL, fee=F+1, sig=GARBAGE))
// Gateway: account_tx_in_pool_or_recent_block(A) == true → skip_validate=true
// → __validate__ NOT called → admitted with GARBAGE signature
// Mempool: fee escalation replaces Alice's invoke with attacker's invoke.

// Step 3: Block is built
// execute deploy_account(A)  → Alice's account deployed, nonce=1
// execute invoke(A, nonce=1, GARBAGE_SIG)
//   → blockifier calls __validate__ on Alice's account
//   → signature invalid → REVERT
//   → fee F+1 charged from Alice's balance
//   → Alice's nonce incremented to 2

// Result: Alice loses fee F+1, nonce=1 consumed, legitimate invoke never executed.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

/// The first invoke_tx can be inserted to the first block right after the deploy_tx due to

    let deploy_tx = undeployed_account_tx_generator.generate_deploy_account();
