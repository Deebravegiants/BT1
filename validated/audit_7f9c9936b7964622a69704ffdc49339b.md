Audit Report

## Title
`updateRSETHPrice()` DoS When `maxFeeMintAmountPerDay` Is Zero and `protocolFeeInBPS` Is Nonzero — (`contracts/LRTOracle.sol`)

## Summary
`maxFeeMintAmountPerDay` defaults to `0` and is never set by `reinitialize()`. When `protocolFeeInBPS > 0` and TVL has grown, `_updateRsETHPrice()` computes a nonzero `rsethAmountToMintAsProtocolFee` and unconditionally passes it to `_checkAndUpdateDailyFeeMintLimit()`, which reverts because `0 + feeAmount > 0` is always true. Both `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` share this path and are equally blocked, leaving `rsETHPrice` stale until a manager explicitly calls `setMaxFeeMintAmountPerDay()`.

## Finding Description
`maxFeeMintAmountPerDay` is declared as a plain `uint256` storage variable, defaulting to `0`: [1](#0-0) 

`reinitialize()` sets `feePeriodStartTime` but never initializes `maxFeeMintAmountPerDay`: [2](#0-1) 

Inside `_updateRsETHPrice()`, when the protocol is unpaused and TVL has grown, `protocolFeeInETH > 0` is computed and `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` is called unconditionally: [3](#0-2) 

Inside `_checkAndUpdateDailyFeeMintLimit()`, there is no guard for `maxFeeMintAmountPerDay == 0`. With `maxFeeMintAmountPerDay == 0` and any `feeAmount > 0`, the condition `0 + feeAmount > 0` is always true and the function always reverts: [4](#0-3) 

This is an internal inconsistency: `remainingDailyFeeMintLimit()` already treats `maxFeeMintAmountPerDay == 0` as a special case (returning `0`), but `_checkAndUpdateDailyFeeMintLimit()` does not apply the same guard: [5](#0-4) 

Both public entry points share the same internal path: [6](#0-5) 

## Impact Explanation
**Medium. Temporary freezing of funds.**

While the DoS is active, `rsETHPrice` is never updated. Deposits use `lrtOracle.rsETHPrice()` to compute rsETH minting amounts: [7](#0-6) 

Withdrawal unlocking also reads the stale `rsETHPrice` to calculate payouts: [8](#0-7) 

With a stale (lower-than-actual) price, new depositors receive more rsETH than they should, diluting existing holders' accrued yield. The freeze is not permanent: a manager can call `setMaxFeeMintAmountPerDay()` with any nonzero value to immediately unblock the function. [9](#0-8) 

## Likelihood Explanation
Moderate. The `reinitialize()` upgrade path explicitly omits setting `maxFeeMintAmountPerDay`. Any deployment or upgrade where `protocolFeeInBPS > 0` is configured but `setMaxFeeMintAmountPerDay()` is not called before the first price update after TVL growth will trigger the revert. This is a realistic operational omission requiring no attacker — the condition is met automatically as rewards accrue.

## Recommendation
1. **In `reinitialize()`**, require or set a nonzero `maxFeeMintAmountPerDay` as part of the upgrade initialization.
2. **In `_checkAndUpdateDailyFeeMintLimit()`**, treat `maxFeeMintAmountPerDay == 0` as "no limit" (skip the cap check entirely), consistent with how `remainingDailyFeeMintLimit()` already handles it:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }
    if (maxFeeMintAmountPerDay != 0) {
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }
    }
    currentPeriodMintedFeeAmount += feeAmount;
}
```

## Proof of Concept
```
Preconditions:
  maxFeeMintAmountPerDay == 0  (never set after upgrade)
  protocolFeeInBPS == 100      (1% fee, set in LRTConfig)
  rsethSupply > 0
  totalETHInProtocol > rsethSupply * rsETHPrice  (rewards accrued)

Step 1: Deploy/upgrade LRTOracle; call reinitialize() — maxFeeMintAmountPerDay stays 0
Step 2: Simulate ETH rewards so totalETHInProtocol > previousTVL
Step 3: Call updateRSETHPrice() (public, no privilege required)
  → protocolFeeInETH = rewardAmount * 100 / 10000 > 0
  → rsethAmountToMintAsProtocolFee > 0
  → _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)
  → 0 + rsethAmountToMintAsProtocolFee > 0  ✓
  → revert DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)
Step 4: Assert rsETHPrice unchanged
Step 5: Manager calls setMaxFeeMintAmountPerDay(1e18) → updateRSETHPrice() succeeds
```
