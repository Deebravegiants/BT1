### Title
Expired DRep Delegation Bypasses Withdrawal Governance Participation Check — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

The Conway era requires that stake credentials delegated to a DRep must have an active DRep delegation before rewards can be withdrawn (`ConwayWdrlNotDelegatedToDRep`). However, `validateWithdrawalsDelegated` only checks whether a DRep delegation field is non-`Nothing` — it does not verify that the referenced DRep is still active (i.e., not expired). A stake credential holder whose DRep has expired can still withdraw rewards without re-delegating to an active DRep, bypassing the governance participation requirement.

---

### Finding Description

`validateWithdrawalsDelegated` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs` implements the withdrawal DRep-delegation check:

```haskell
isNotDRepDelegated keyHash = isNothing $ do
  accountState <- lookupAccountState (KeyHashObj keyHash) accounts
  accountState ^. dRepDelegationAccountStateL
``` [1](#0-0) 

The predicate returns `False` (delegation considered valid) whenever `dRepDelegationAccountStateL` is `Just _` — regardless of whether the DRep credential it points to is still registered and active. DRep expiry is tracked separately in `DRepState.drepExpiry` inside `VState.vsDReps`, and an expired DRep is **not** removed from the map nor does it clear delegators' `dRepDelegationAccountStateL` fields. Only an explicit `ConwayUnRegDRep` certificate clears delegations via `clearDRepDelegations`:

```haskell
clearDRepDelegations delegs accountsMap =
  foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
``` [2](#0-1) 

When a DRep simply expires (epoch passes `drepExpiry`), no such cleanup occurs. The `dRepDelegationAccountStateL` of every delegator still points to the expired DRep credential, so `validateWithdrawalsDelegated` passes for all of them.

The ratification logic correctly excludes expired DReps from vote counting:

```haskell
| reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
``` [3](#0-2) 

But the withdrawal check does not mirror this logic.

The formal specification disagrees with this behavior. The test `"Withdraw from a key delegated to a DRep that expired after delegation"` is explicitly disabled with `disableInConformanceIt` and references formal-ledger-specifications issue #635:

```haskell
-- https://github.com/IntersectMBO/formal-ledger-specifications/issues/635
-- TODO: Re-enable after issue is resolved, by removing this override
disableInConformanceIt "Withdraw from a key delegated to a DRep that expired after delegation" $ do
  ...
  isDRepExpired drep `shouldReturn` True
  submitTx_ $ ...  -- succeeds in implementation, should fail per spec
``` [4](#0-3) 

A second passing test confirms that delegating to an already-expired DRep and then withdrawing also succeeds: [5](#0-4) 

The same `validateWithdrawalsDelegated` function is reused verbatim in the Dijkstra era's `ENTITIES` and `SUBENTITIES` rules: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Medium — Attacker-controlled transactions modify withdrawals outside design parameters.**

The Conway era governance participation requirement mandates that key-hash stake credentials must be delegated to an active DRep to withdraw rewards. This is a protocol-level design parameter enforced since protocol version 10 (`hardforkConwayBootstrapPhase`). A stake credential holder whose DRep has expired can withdraw accumulated rewards without re-delegating to any active DRep, circumventing this requirement. The withdrawal modifies the reward balance outside the intended validation constraint. The same flaw propagates to the Dijkstra era. [8](#0-7) 

---

### Likelihood Explanation

**Medium.** DRep expiry is a normal protocol event governed by `ppDRepActivityL`. Any DRep that stops voting or submitting update certificates will eventually expire. Any stake credential holder delegated to such a DRep — which can happen passively without any action by the delegator — gains the ability to withdraw rewards without satisfying the governance participation requirement. No special privileges, leaked keys, or coordination are required. The attacker-controlled entry path is a standard withdrawal transaction.

---

### Recommendation

`validateWithdrawalsDelegated` should additionally verify that the DRep the credential is delegated to is not expired at the time of withdrawal. Concretely, after resolving `dRepDelegationAccountStateL` to a `DRepCredential cred`, the function should look up `cred` in `vsDReps` and check that `reCurrentEpoch <= drepExpiry drepState` (mirroring the ratification logic in `dRepAcceptedRatio`). Predefined DReps (`DRepAlwaysAbstain`, `DRepAlwaysNoConfidence`) have no expiry and should continue to pass unconditionally.

---

### Proof of Concept

1. Register a stake credential `cred` and earn rewards.
2. Register a DRep with `ppDRepActivityL = EpochInterval 1`.
3. Submit a governance proposal and let it expire; pass enough epochs so the DRep expires (`isDRepExpired drep == True`).
4. Delegate `cred` to the expired DRep via `DelegTxCert cred (DelegVote (DRepCredential drep))`.
5. Submit a withdrawal transaction for `cred`'s reward balance.

**Result**: The transaction is accepted. `validateWithdrawalsDelegated` passes because `dRepDelegationAccountStateL` is `Just (DRepCredential drep)` (non-`Nothing`), even though the DRep is expired. The governance participation requirement is bypassed.

This is directly demonstrated by the existing (passing) test: [5](#0-4) 

and the disabled conformance test confirming the formal spec disagrees: [4](#0-3)
