### Title
`FanOut` Inspector Accepts a Single RPC Provider's Response Without Quorum Enforcement, Enabling Forged Foreign-Chain Transaction Attestation - (File: `crates/foreign-chain-inspector/src/lib.rs`)

---

### Summary

The `FanOut` inspector, which is the sole mechanism by which each MPC node verifies foreign-chain transactions before contributing a signature share, accepts a single successful RPC provider response as ground truth even when multiple providers are configured. The on-chain `ChainEntry.quorum` field — which is intended to require a minimum number of agreeing providers — is never passed to or enforced by `FanOut`. This is the direct analog of Peggo's single-oracle vulnerability: a single compromised or manipulated RPC provider, combined with transient failures of the remaining providers, is sufficient to cause a node to sign a forged foreign-chain observation.

---

### Finding Description

The `FanOut<Inspector>` struct in `crates/foreign-chain-inspector/src/lib.rs` fans out queries to all configured RPC providers concurrently and collects their results. Its acceptance logic is: [1](#0-0) 

If **any** subset of inspectors returns `Ok` with the same values, `FanOut` immediately returns those values as verified — regardless of how many providers succeeded. Transient errors (`ClientError`, `RpcRequestFailed`, `NotFinalized`, `NotEnoughBlockConfirmations`) are explicitly tolerated and do not count as substantive verdicts: [2](#0-1) 

This means: if a node has three providers configured but two fail transiently (network timeout, rate limit, etc.), the single surviving provider's response is accepted as the verified truth. There is no minimum quorum check anywhere in `FanOut`.

The design explicitly specifies a `quorum` field in `ChainEntry` that is voted on-chain: [3](#0-2) 

`build_fanout` constructs a `FanOut::new(inspectors)` from the provider list but **never passes the quorum value** to `FanOut`. The `FanOut` struct has no `quorum` field and no quorum-enforcement logic. The design documentation explicitly acknowledges this gap:

> "This sub-quorum outcome must be terminal — the leader must not re-attempt the request. **Implementation requirement, not current behavior.**" [4](#0-3) 

The same document also states that participant election for foreign-tx signing must select only nodes that cover the chain with ≥ `rpc_quorum(C)` providers, but "today the signing set is inherited from a presignature, whose participants were chosen for liveness, not chain coverage." [5](#0-4) 

Each node independently executes `execute_foreign_chain_request` in both the leader and follower paths before contributing a signature share: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

If a threshold of MPC nodes each have their non-primary RPC providers fail transiently while a single compromised provider returns forged transaction data (e.g., claiming a Bitcoin or EVM transaction exists and is finalized when it does not), each affected node independently accepts the forged observation and contributes a signature share. Because all nodes are fed the same forged data by the same compromised provider, their independently computed `ForeignTxSignPayload` values agree, and the threshold signature is produced. The resulting on-chain signature attests to a foreign-chain transaction that never occurred, enabling invalid bridge execution (e.g., minting bridged assets on NEAR for a deposit that never happened on the foreign chain).

This matches the **High** impact category: forged foreign-chain verification that causes invalid bridge execution.

---

### Likelihood Explanation

The attack requires two simultaneous conditions per node: (1) at least one configured provider returns attacker-controlled data, and (2) all other providers for that node fail transiently. Condition (2) is achievable via targeted DDoS or rate-limit exhaustion against the non-target providers. Condition (1) requires compromising a single widely-shared RPC provider (e.g., a public Alchemy or Infura endpoint used by multiple operators). Because the design doc notes that "currently, if a single RPC provider is unresponsive, then a foreign chain validation will fail" — operators are incentivized to configure only a small number of providers — the attack surface is narrow but realistic. The likelihood is **Medium**. [8](#0-7) 

---

### Recommendation

1. **Short term**: Pass `ChainEntry.quorum` through `build_fanout` into `FanOut` and enforce it in `FanOut::extract`: require that `successes.len() >= quorum` before returning `Ok`. If fewer than `quorum` providers succeed, return a non-transient error so the request is not retried.

2. **Short term**: Ensure the participant election for foreign-tx signing selects only nodes that cover the requested chain with at least `rpc_quorum(C)` providers, as specified in the design but not yet implemented.

3. **Long term**: Implement the on-chain RPC provider whitelist (PR 3 of issue #3208) so that the full URL is assembled from on-chain `base_url` + `chain_routing`, preventing a malicious operator from pointing a node at an arbitrary RPC server. Until then, a single operator's config file is the only barrier against a forged RPC endpoint. [9](#0-8) 

---

### Proof of Concept

1. A node is configured with three Bitcoin RPC providers: `provider-A` (attacker-controlled), `provider-B`, `provider-C`.
2. The attacker DDoSes `provider-B` and `provider-C`, causing them to return `RpcRequestFailed` (transient).
3. `provider-A` returns a fabricated `getrawtransaction` response claiming transaction `0xdeadbeef...` is confirmed in block `X` with 6 confirmations.
4. `FanOut::extract` receives: `successes = [(0, [BlockHash([0xaa;32])])]`, `first_transient_error = Some(RpcRequestFailed)`.
5. The check at line 130 finds `successes.first()` is `Some`, `all_successes_agree` is trivially `true` (only one success), and returns `Ok([BlockHash([0xaa;32])])`.
6. The node computes `ForeignTxSignPayload::V1 { request, values: [BlockHash([0xaa;32])] }` and contributes a signature share.
7. If a threshold of nodes are similarly misled (each via their own single surviving attacker-controlled provider), the MPC network produces a valid threshold signature attesting to the forged transaction.
8. The on-chain contract receives `respond_verify_foreign_tx` with a valid signature, resolves the yield, and the caller receives a signed attestation for a transaction that never existed. [10](#0-9) [11](#0-10)

```markdown
### Security
By requiring every node to have a valid quorum of RPC providers for a verification, each node does not need to trust the other nodes' RPC configurations. Putting the connection config (`base_url`, `auth_scheme`, `chain_routing`) on chain also closes the gap where a TEE-attested binary trusts its operator's local config — the operator can no longer point the node at an arbitrary URL, only at one the network has voted to trust for that chain.
```

```markdown
### Individual node quorum of RPC providers for verification requests

When a foreign TX verification request is processed by a set of nodes, every node individually queries its locally-configured RPC providers for that chain. A node considers the foreign TX verified iff at least the per-chain RPC quorum of providers agreed. If fewer than the RPC quorum agree, the node errors out and produces no signature share. This sub-quorum outcome must be **terminal** for the request — the leader does not re-attempt it. This is an implementation requirement, not current behavior. See [Calculating the whitelisted and available foreign-chain sets](calculating-supported-foreign-chains.md#verification-behavior).

The quorum value comes from on-chain `ChainEntry.quorum`, voted in as part of the same `ChainVote` that voted the chain's provider list — so participants agree on both "which providers are trusted" and "how many of them must concur" in one round.
```
