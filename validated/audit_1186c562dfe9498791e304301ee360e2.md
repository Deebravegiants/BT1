### Title
State Mutation Before Deposit Validation Allows Free Attestation Storage — (`File: crates/contract/src/lib.rs`)

### Summary

`submit_participant_info` mutates contract state (stores the attestation in `stored_attestations`) **before** validating that the caller attached sufficient deposit. In NEAR SDK, returning `Err` from a `#[handle_result]` function does **not** roll back state — only `env::panic_str` / a Rust panic does. As a result, a caller with a valid TEE attestation can permanently register their attestation without paying the required storage-staking deposit, breaking the contract's accounting invariant.

### Finding Description

`submit_participant_info` follows this order:

1. Snapshot `initial_storage = env::storage_usage()` [1](#0-0) 
2. Call `self.tee_state.add_participant(...)`, which unconditionally inserts into `stored_attestations` on success [2](#0-1) 
3. Compute `storage_used` and `cost` from the delta [3](#0-2) 
4. Check `attached < cost` and, if true, **return `Err`** [4](#0-3) 

`add_participant` writes the new `NodeAttestation` into the `IterableMap<Ed25519PublicKey, NodeAttestation>` field `stored_attestations` [5](#0-4) 

Because the function is decorated `#[handle_result]` [6](#0-5) , returning `Err` causes the SDK to serialize the error and return normally — the NEAR runtime then **commits** the in-memory state to storage. Only a panic would prevent the write. The deposit check at line 835 therefore fires too late: the attestation is already durably stored.

The condition that gates the deposit check is:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;
``` [7](#0-6) 

Both branches — a brand-new participant and any non-participant — reach the deposit check, meaning both are affected.

### Impact Explanation

An attacker with a valid TEE attestation (i.e., a legitimate or prospective MPC node) can:

1. Call `submit_participant_info` with `attached_deposit = 0`.
2. `add_participant` succeeds; the attestation is written to `stored_attestations`.
3. The deposit check fires, finds `0 < cost`, and returns `Err(InsufficientDeposit)`.
4. The NEAR runtime commits the state — the attestation is permanently stored without payment.

Consequences:
- The contract's storage grows without the corresponding NEAR balance increase, violating the storage-staking invariant. Repeated exploitation across many nodes degrades the contract's available balance relative to its storage usage.
- The participant is effectively registered (their attestation is present in `stored_attestations`, which is the authoritative source for `is_caller_an_attested_participant`) without having paid the required deposit. This breaks the production accounting invariant that every byte of contract storage is backed by a deposit.

This maps to the allowed Medium impact: *"participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

Any MPC node operator with a valid TEE attestation (the normal production case) can trigger this by simply omitting the deposit on their `submit_participant_info` call. No threshold collusion, no privileged access, and no network-level attack is required. The call is publicly accessible and the only prerequisite — a valid attestation — is held by every legitimate node.

### Recommendation

Move the deposit check **before** `add_participant`, using a pre-computed upper-bound estimate (as `propose_update` does with `bytes_used`) [8](#0-7) , or replace `return Err(...)` with `env::panic_str(...)` so that an insufficient deposit causes a panic and rolls back the state mutation. The cleanest fix is the former: estimate the maximum storage cost upfront, require the deposit before touching state, and refund the excess after the actual storage delta is known.

### Proof of Concept

```
// Attacker: a node with a valid TEE attestation
// 1. Construct a valid Attestation (real DCAP or Mock in test environments)
// 2. Call submit_participant_info with attached_deposit = 0
near call <mpc_contract> submit_participant_info \
  '{"proposed_participant_attestation": <valid_attestation>, "tls_public_key": <tls_pk>}' \
  --deposit 0 \
  --accountId <attacker_node>

// Result: transaction receipt shows Err(InsufficientDeposit)
// BUT: get_attestation(<tls_pk>) now returns Some(...) — attestation is stored
near view <mpc_contract> get_attestation '{"tls_public_key": <tls_pk>}'
// => Some(VerifiedAttestation { ... })   <-- stored without payment
```

The root cause is identical in structure to M-06: the "available balance" check (deposit sufficiency) is performed against a reference that already includes the committed state change (the storage written by `add_participant`), so the check cannot prevent the invariant violation — it fires after the damage is done.
