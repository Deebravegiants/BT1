### Title
Cross-Chain Replay of `SignedDelegateAction` Due to Missing Chain Binding in Signed Payload - (File: core/primitives/src/signable_message.rs, core/primitives/src/action/delegate.rs)

### Summary

The `DelegateAction` (NEP-366 meta transaction) and `DelegateActionV2` (NEP-611 gas-key meta transaction) signed payloads are constructed via `SignableMessage`, whose domain discriminant encodes only a fixed NEP number — no chain ID, genesis hash, or any network-specific identifier. Because regular `SignedTransaction` binds to a specific chain via `block_hash` in the signed body, but `SignedDelegateAction` has no equivalent binding, a valid signature produced on one NEAR network (e.g., testnet) is cryptographically valid on any other NEAR network (e.g., mainnet) where the same account and key exist with a compatible nonce.

### Finding Description

**Root cause — `SignableMessage` carries no chain binding:**

`DelegateAction::get_nep461_hash()` constructs the signed payload as:

```
borsh( MessageDiscriminant(2^30 + 366) || DelegateAction { sender_id, receiver_id, actions, nonce, max_block_height, public_key } )
``` [1](#0-0) 

`VersionedDelegateActionPayload::get_nep461_hash()` (for `DelegateActionV2`) does the same with discriminant `2^30 + 611`: [2](#0-1) 

`SignableMessage` itself contains only a `MessageDiscriminant` (a fixed `u32` NEP number) and the message body: [3](#0-2) 

The discriminant is a compile-time constant — it does not include `chain_id`, `genesis_id`, or any network-specific value: [4](#0-3) 

**Contrast with regular transactions:** A `SignedTransaction` includes `block_hash` in the signed body, which is chain-specific (a block hash from mainnet cannot exist on testnet). `DelegateAction` has no equivalent field — only `max_block_height`, which is a plain integer shared across all NEAR networks. [5](#0-4) 

**Exploit path:**

1. Alice signs a `DelegateAction` on NEAR testnet (e.g., to transfer 100 NEAR to Bob, or call `ft_transfer`). The signature covers `(discriminant=2^30+366, sender_id, receiver_id, actions, nonce=N, max_block_height=H, public_key)`.
2. An attacker observes this `SignedDelegateAction` from the public testnet.
3. The attacker submits the identical `SignedDelegateAction` to a relayer on NEAR mainnet.
4. `apply_delegate_action` calls `signed_delegate_action.verify()`, which recomputes the same hash (same discriminant, same payload fields) and verifies the same signature — **it passes**, because no chain-specific data was signed.
5. The nonce check (`nonce > current_key_nonce_on_mainnet`) passes if Alice's mainnet key nonce is below `N` (common for new accounts or infrequently used keys).
6. The `max_block_height` check passes if mainnet block height is below `H`.
7. Alice's mainnet account executes the inner actions without her consent. [6](#0-5) 

### Impact Explanation

An attacker who observes any `SignedDelegateAction` from one NEAR network can replay it on another NEAR network where the same account and key exist. The corrupted protocol values are:

- **Account balance**: NEAR tokens transferred without the sender's consent on the target chain.
- **Account state**: Arbitrary contract calls (e.g., `ft_transfer`, `DeleteAccount`, `AddKey`) executed on the target chain under the sender's identity.
- **Access key nonce**: Consumed on the target chain, potentially invalidating legitimate pending delegate actions.

This is a direct financial loss and unauthorized state mutation for any user whose `DelegateAction` is observed on one network and replayed on another.

### Likelihood Explanation

- NEAR account IDs are human-readable strings (e.g., `alice.near`). The same account ID commonly exists on both mainnet and testnet.
- Many users generate keys once and register them on multiple networks.
- Testnet transactions and receipts are publicly observable.
- The nonce precondition (mainnet nonce < delegate action nonce) is satisfied for any account that has used the key less on mainnet than on testnet, or for newly created mainnet accounts.
- `max_block_height` is a plain block height; mainnet and testnet block heights are independent integers that can easily overlap.
- No privileged access is required — the attacker only needs to observe public testnet data and submit to a public relayer RPC.

### Recommendation

Include the chain's `genesis_id` (or `chain_id` string + genesis block hash) in the `SignableMessage` payload, analogous to EIP-712's domain separator. The `GenesisId` type already exists in the codebase: [7](#0-6) 

`SignableMessage` should be extended to include a `genesis_id: GenesisId` field (or at minimum a `chain_id: String`) so that a signature produced on testnet is cryptographically invalid on mainnet. This is a protocol-level change requiring a new `SignableMessageType` variant and a protocol version gate.

### Proof of Concept

```
1. On testnet, Alice (alice.near, nonce=5 on key K) signs:
   DelegateAction { sender_id="alice.near", receiver_id="bob.near",
                    actions=[Transfer{100 NEAR}], nonce=6,
                    max_block_height=200_000_000, public_key=K }
   → hash = sha256(borsh(0x4000016E || delegate_action_bytes))
   → signature S = sign(hash, alice_secret_key)

2. Attacker observes (DelegateAction, S) from testnet.

3. On mainnet, alice.near exists with key K at nonce=3 (< 6).
   Mainnet block height = 150_000_000 (< 200_000_000).

4. Attacker wraps (DelegateAction, S) in a mainnet SignedTransaction
   (relayer pays gas) and submits to mainnet RPC.

5. apply_delegate_action calls signed_delegate_action.verify():
   recomputes sha256(borsh(0x4000016E || same_delegate_action_bytes)) → same hash
   signature.verify(hash, K) → true  ← no chain binding, passes

6. Nonce check: 6 > 3 → passes
   max_block_height check: 150M < 200M → passes

7. 100 NEAR transferred from alice.near on mainnet without Alice's consent.
```

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }

    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
