### Title
Attached NEAR Deposit Permanently Locked in `submit_participant_info` When Called by Existing Participant — (`File: crates/contract/src/lib.rs`)

### Summary

`submit_participant_info` is marked `#[payable]` and conditionally handles the attached deposit only when `attestation_storage_must_be_paid_by_caller` is `true`. When an **existing participant** re-submits their attestation (e.g., during a TEE upgrade cycle), this flag evaluates to `false` and the entire deposit-handling block is skipped — no sufficiency check, no refund. Any NEAR attached to the call is permanently locked in the chain-signature contract with no recovery path.

---

### Finding Description

In `crates/contract/src/lib.rs`, `submit_participant_info` is declared `#[payable]`: [1](#0-0) 

After inserting the attestation, the contract computes a boolean gate: [2](#0-1) 

The deposit-handling block — which both validates sufficiency and issues a refund — is guarded entirely by this flag: [3](#0-2) 

When `attestation_storage_must_be_paid_by_caller` is `false` (i.e., `is_new_attestation == false` **AND** `caller_is_not_participant == false`), the block is skipped entirely. `env::attached_deposit()` is never read, never validated, and never refunded. The NEAR is absorbed by the contract with no mechanism to recover it.

The condition evaluates to `false` precisely when an **already-registered participant** re-submits their attestation — the normal operational path during TEE software upgrades, key-rotation events, or attestation renewals. The node software (`crates/node/src/indexer/tx_sender.rs`) calls `submit_participant_info` programmatically; if it attaches any deposit to cover potential storage costs (a natural defensive pattern given that new submissions *do* require a deposit), the funds are silently locked on every re-submission. [4](#0-3) 

The refund `Promise` at line 847 is only reachable inside the guarded block. There is no unconditional refund path anywhere in the function.

---

### Impact Explanation

Any NEAR attached to a re-submission call by an existing participant is permanently frozen inside the chain-signature contract. There is no administrative withdrawal function, no sweep mechanism, and no way for the participant to recover the funds. This constitutes **permanent freezing of funds controlled by the chain-signature contract**, matching the Critical allowed impact tier. The severity is moderated to **High** because the locked amount per call is bounded by whatever deposit the caller attaches (not an unbounded drain of contract reserves), and the immediate victim is the calling participant rather than end-users.

---

### Likelihood Explanation

TEE upgrades and attestation renewals are **routine operational events** for every MPC participant. The node daemon (`crates/node/src/indexer/tx_sender.rs`) calls `submit_participant_info` automatically. Because new submissions require a deposit and the function is `#[payable]`, the node software or a human operator following documentation may defensively attach NEAR on every call. Every such re-submission by an existing participant silently locks the attached funds. The trigger requires no adversarial action — it fires on normal node operation.

---

### Recommendation

Add an unconditional refund of any excess deposit **outside** the `attestation_storage_must_be_paid_by_caller` guard, mirroring the pattern already used in `require_deposit`:

```rust
// After the guarded block, always refund any deposit not consumed by storage costs.
let consumed = if attestation_storage_must_be_paid_by_caller { cost } else { NearToken::from_yoctonear(0) };
let attached = env::attached_deposit();
if let Some(diff) = attached.checked_sub(consumed) {
    if diff > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(diff).detach();
    }
}
```

Alternatively, reject non-zero deposits when `!attestation_storage_must_be_paid_by_caller` with an explicit error, so callers are not silently penalised.

---

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` for the first time with a valid attestation and 1 NEAR deposit → succeeds, storage cost is charged, excess refunded correctly.
2. A TEE software upgrade occurs. `alice.near`'s node daemon calls `submit_participant_info` again (re-submission) with 0.5 NEAR attached (defensive deposit).
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` → `is_new_attestation = false`.
4. `self.voter_account()` succeeds (alice is a participant) → `caller_is_not_participant = false`.
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The deposit block is skipped. `env::attached_deposit()` (0.5 NEAR) is never refunded.
7. `assert_eq!(contract_balance_after - contract_balance_before, 0.5 NEAR)` — funds are locked forever. [5](#0-4)
