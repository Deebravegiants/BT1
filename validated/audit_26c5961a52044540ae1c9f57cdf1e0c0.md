### Title
Proposal Inserted into `valid_proposals` Before `ProposalFinMismatch` Check Leaves Stale Commitment Entry on Validation Failure - (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

### Summary

In `validate_proposal`, the call to `valid_proposals.insert_proposal(...)` is made **before** the `ProposalFinMismatch` guard. When the batcher's locally-computed commitment (`built_block`) differs from the proposer's network-supplied commitment (`received_fin.proposal_commitment`), the function returns `Err(ProposalFinMismatch)` — but the proposal has already been written into `valid_proposals` under the batcher's commitment. This is the direct sequencer analog of the `YRizStrategy.getAllPoolDistributions` bug: data is written to a collection before the validity check, leaving an invalid entry in the collection when the check fails.

### Finding Description

In `validate_proposal`:

```rust
// Line 240-247 of validate_proposal.rs
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info); // ← INSERT FIRST

// TODO(matan): Switch to signature validation.
if built_block != received_fin.proposal_commitment {                                    // ← CHECK AFTER
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);                             // ← RETURN WITH STALE ENTRY
}
Ok(built_block)
``` [1](#0-0) 

`insert_proposal` stores the proposal keyed by `(height, round)` with the batcher's computed `ProposalCommitment` (derived from `finished_info.proposal_commitment.partial_block_hash`):

```rust
// sequencer_consensus_context.rs lines 195-198
let previous = by_round.insert(
    round,
    (proposal_commitment, (init, transactions, *proposal_id, finished_info)),
);
``` [2](#0-1) 

When `ProposalFinMismatch` fires, the batcher's commitment `X` is stored for `(height, round)`, but the function returns an error. The `fin_sender` in `validate_and_send` is never called, so consensus receives `Canceled` and the validator does not vote. [3](#0-2) 

### Impact Explanation

Two concrete failure modes follow from the stale entry:

**1. Panic in `decision_reached` (node crash).**
`decision_reached` calls `get_proposal(height, round, commitment)`, which asserts that the stored commitment equals the requested one:

```rust
assert_eq!(
    stored_commitment, commitment,
    "Proposal commitment mismatch for height {height} round {round}: ..."
);
``` [4](#0-3) 

If the stale entry stores commitment `X` for `(height, round)` but consensus later decides on commitment `Y` (the proposer's value, voted for by 2/3 of other validators who did not encounter `ProposalFinMismatch`), the assert panics and crashes the node.

**2. Panic in `insert_proposal` (node crash).**
`insert_proposal` asserts `previous.is_none()` — at most one proposal per `(height, round)`. If a `ProposalFinMismatch` already wrote an entry for `(height, round)`, any subsequent insertion for the same slot (e.g., a retry or a race) panics unconditionally. [5](#0-4) 

Both paths are liveness failures: the node crashes and cannot participate in consensus.

### Likelihood Explanation

`ProposalFinMismatch` is reachable by any network participant acting as proposer:

- A Byzantine proposer sends a syntactically valid transaction batch but crafts the `ProposalFin.proposal_commitment` to be a value `Y` that differs from what the batcher will compute (`X`). The existing guard at lines 522–528 only rejects `executed_transaction_count > n_received_txs`; it does not prevent a mismatched commitment value. [6](#0-5) 

- Non-determinism in the batcher (e.g., ordering of concurrent execution results) can also produce `X ≠ Y` across validators without any malicious actor, causing a subset of validators to store under `X` while others vote for `Y`.

The comment `// TODO(matan): Switch to signature validation.` confirms the check is a known placeholder, making the mismatch path a live code branch in production. [7](#0-6) 

### Recommendation

Move `insert_proposal` to **after** the `ProposalFinMismatch` check. The comment explains the insert must precede `fin_sender.send()` to avoid a repropose race condition — but `fin_sender.send()` is called in `validate_and_send` only after `validate_proposal` returns `Ok`, so inserting at the end of the success path still satisfies that constraint:

```rust
// Check first
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}

// Insert only on success — still before fin_sender.send() in validate_and_send
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

Ok(built_block)
```

This mirrors the fix described in the external report: check initialization before writing to the collection, not after.

### Proof of Concept

1. A Byzantine proposer at height `H`, round `R` streams a valid transaction batch `[T1, T2, T3]` to a validator node.
2. The batcher executes the batch and computes `partial_block_hash = X`, yielding `built_block = commitment_from(X, fee_proposal)`.
3. The proposer sends `ProposalFin { proposal_commitment: Y, executed_transaction_count: 3 }` where `Y ≠ built_block`.
4. `handle_proposal_part` returns `HandledProposalPart::Finished(built_block=X_commitment, fin, finished_info)`.
5. `validate_proposal` calls `valid_proposals.insert_proposal(init, content, proposal_id, finished_info)` — the entry `(H, R) → (X_commitment, ...)` is written.
6. The check `built_block != received_fin.proposal_commitment` fires; `Err(ProposalFinMismatch)` is returned. The stale entry remains.
7. Meanwhile, 2/3 of other validators (running a batcher that agrees with the proposer's `Y`) vote for `Y`. Consensus decides on `Y`.
8. The local node receives the decision and calls `decision_reached(H, R, Y)`.
9. `get_proposal(H, R, Y)` finds the entry for `(H, R)` with stored commitment `X_commitment ≠ Y` and panics: `"Proposal commitment mismatch for height H round R: stored=X_commitment, requested=Y"`.
10. The node crashes. [8](#0-7) [9](#0-8)
