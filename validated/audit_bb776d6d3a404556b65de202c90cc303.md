### Title
Inconsistent CKD Response Validation Between `AppPublicKey` and `AppPublicKeyPV` Paths Allows Single Byzantine Coordinator to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` applies a full BLS12-381 pairing-based cryptographic output check only when the request carries an `AppPublicKeyPV` key, but performs **zero cryptographic validation** when the request carries the plain `AppPublicKey` variant. A single Byzantine MPC participant acting as the signing coordinator can call `respond_ckd()` with an entirely fabricated `CKDResponse` for any `AppPublicKey`-type request and the contract will accept and deliver it to the user, bypassing the threshold MPC protocol entirely.

---

### Finding Description

In `respond_ckd()`, the contract branches on the `CKDAppPublicKey` variant of the pending request:

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically proves the response was produced by the threshold MPC protocol using the shared master secret key. [2](#0-1) 

For `AppPublicKey`, the arm is an empty block `{}`. Any `big_c` and `big_y` values pass unconditionally. The only gate before `resolve_yields_for` is `assert_caller_is_attested_participant_and_protocol_active()`, which requires only a single attested participant — not a threshold quorum. [3](#0-2) 

The `AppPublicKey` variant is the default/legacy form used in the majority of documented examples and SDK usage. [4](#0-3) 

---

### Impact Explanation

A single Byzantine MPC participant who is elected coordinator for a CKD task can call `respond_ckd` with arbitrary `big_c` / `big_y` values for any `AppPublicKey` request. The contract resolves the pending yield and delivers the fabricated response to the user as if it were the genuine threshold-derived output. The user's derived application secret key is therefore entirely attacker-controlled. This constitutes **unauthorized confidential key derivation output without the required participant authorization**, matching the Critical impact tier.

---

### Likelihood Explanation

The `AppPublicKey` variant is the primary, non-PV path used by most callers (the PV variant requires the user to supply a matching G1/G2 key pair and is an opt-in feature). Any single participant who becomes the coordinator for a CKD round — a role that rotates and is observable on-chain — can exploit this. No collusion above one participant is required. The attacker needs only to be an active, attested participant in the current epoch.

---

### Recommendation

Apply the same cryptographic output check to the `AppPublicKey` path. Because `AppPublicKey` does not supply a G2 component (`pk2`), the existing `ckd_output_check` cannot be used directly. The recommended fix is one of:

1. **Require `AppPublicKeyPV` for all new requests** and deprecate the unverifiable `AppPublicKey` variant.
2. **Derive a synthetic `pk2`** from the supplied G1 key using a deterministic hash-to-G2 (so the pairing equation can still be checked), or store the G2 component alongside the G1 key at request time.
3. **Require threshold-many `respond_ckd` calls** (analogous to how `vote_pk` / `vote_reshared` aggregate votes) before resolving the yield, so a single coordinator cannot unilaterally deliver a response.

---

### Proof of Concept

1. User submits `request_app_private_key` with `AppPublicKey(some_g1_point)` and `domain_id` pointing to the BLS12-381 CKD domain.
2. The contract stores the pending yield under the `CKDRequest` key.
3. Byzantine coordinator (a single attested participant) calls:
   ```
   respond_ckd(
     request = <the pending CKDRequest>,
     response = CKDResponse { big_y: [0x01; 48], big_c: [0x02; 48] }  // arbitrary garbage
   )
   ```
4. The `AppPublicKey(_) => {}` branch executes with no check.
5. `resolve_yields_for` resolves the pending yield and returns the fabricated `CKDResponse` to the user.
6. The user's application derives a secret key from `big_c` and `big_y` that is entirely controlled by the attacker, not by the MPC network's shared master secret. [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
