### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Original Request — Forged Foreign-Chain Verification by a Single Malicious Participant - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never checks that `response.payload_hash` is actually the hash of `ForeignTxSignPayload { request: <the_original_request>, values: ... }`. A single malicious attested participant that leads any other pending request can obtain a legitimate MPC signature over a different payload hash and submit it as the resolution of an unrelated pending request, causing the contract to deliver a misleading attestation to the caller.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs the following checks:

1. Caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's **root** public key.
3. `request` matches a pending yield in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What it does **not** check is that `response.payload_hash` is the SHA-256 hash of `borsh(ForeignTxSignPayload::V1 { request: request.request, values: <any_values> })`. The `payload_hash` field is accepted as-is from the caller, and only its internal consistency with the signature is verified — not its binding to the original request.

The canonical payload structure is: [2](#0-1) 

The hash is computed as `SHA-256(borsh(ForeignTxSignPayload))`. The contract never recomputes this hash from `request.request` to compare against `response.payload_hash`.

### Impact Explanation

A malicious attested participant (single node, below threshold) that is selected as leader for a **different** pending request R2 obtains a legitimate threshold signature over `hash(ForeignTxSignPayload { request: R2, values: <R2_observed_values> })`. It then calls:

```
respond_verify_foreign_tx(
    request = R1,   // a different pending request
    response = {
        payload_hash: hash_of_R2_payload,   // hash of R2's payload
        signature: sig_over_hash_of_R2,     // valid MPC signature
    }
)
```

The contract accepts this because:
- The signature is valid over `payload_hash` ✓
- `request = R1` matches a pending yield ✓
- The `payload_hash` is never checked against `R1.request` ✗

R1's caller receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes R2's transaction data (different `tx_id`, different extracted values). Any bridge contract that does not independently recompute the expected `payload_hash` from the original request and compare it against `response.payload_hash` will accept this as proof that R1's transaction was verified. This enables forged foreign-chain verification and invalid bridge execution (e.g., releasing funds for a transaction that was never actually verified, or releasing funds of the wrong amount).

The `near-mpc-sdk` does provide an off-chain `ForeignChainSignatureVerifier::verify_signature` that performs this check: [3](#0-2) 

However, this is an optional off-chain SDK helper. The on-chain contract itself does not enforce the invariant, so any bridge contract that omits this check is vulnerable.

### Likelihood Explanation

- The attacker is a single malicious attested participant (below threshold).
- The attacker only needs to be selected as leader for **any** other pending `verify_foreign_transaction` request to obtain a valid MPC signature over a different payload hash.
- Leader selection is deterministic but rotates across nodes, so a malicious node will eventually lead some request.
- No threshold collusion is required; the malicious node uses a legitimately produced signature from a different computation.
- The attack is silent — the contract emits no error and the caller has no on-chain indication that the `payload_hash` does not correspond to their request.

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, recompute the expected payload hash prefix from `request.request` and verify that `response.payload_hash` is consistent with it. Concretely, the contract should verify:

```rust
// Reconstruct the canonical payload from the request and verify the hash prefix
// (the contract cannot know the extracted values, but it can verify the request field)
// OR: require the response to include the full ForeignTxSignPayload so the contract
// can recompute and verify the hash.
```

The cleanest fix is to change `VerifyForeignTransactionResponse` to include the full `ForeignTxSignPayload` (not just the hash), so the contract can:
1. Recompute `expected_hash = SHA256(borsh(payload))`.
2. Verify `expected_hash == response.payload_hash`.
3. Verify that `payload.request == request.request`.

This ensures the signature attests to the correct request, analogous to how EIP-4626 vaults must revert when the actual assets delivered do not match the requested amount.

### Proof of Concept

1. Alice submits `verify_foreign_transaction(request=R1)` for Bitcoin `tx_id=X`. The contract stores a pending yield keyed by R1.
2. Bob submits `verify_foreign_transaction(request=R2)` for Bitcoin `tx_id=Y`. The contract stores a pending yield keyed by R2.
3. Malicious node M is selected as leader for R2. It executes the foreign chain inspection for R2, gets the MPC network to sign `hash_of_R2_payload`, and obtains the final signature `sig_R2`.
4. M calls `respond_verify_foreign_tx(request=R1, response={payload_hash: hash_of_R2_payload, signature: sig_R2})`.
5. The contract checks: `verify_ecdsa_signature(sig_R2, hash_of_R2_payload, root_pk)` → valid ✓. Resolves R1's yield with this response.
6. Alice's contract receives `VerifyForeignTransactionResponse { payload_hash: hash_of_R2_payload, signature: sig_R2 }` — a signature attesting to `tx_id=Y`, not `tx_id=X`.
7. If Alice's bridge contract does not call `ForeignChainSignatureVerifier::verify_signature` with the expected extracted values for `tx_id=X`, it accepts this as proof that `tx_id=X` was verified and releases funds. [4](#0-3) [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
