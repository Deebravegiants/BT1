### Title
Missing CKD Response Validation for `AppPublicKey` Variant Allows Single Byzantine Node to Forge Confidential Key Derivation Output - (File: crates/contract/src/lib.rs)

### Summary
`respond_ckd()` in the MPC contract performs no cryptographic validation of the submitted `CKDResponse` when the request uses the `CKDAppPublicKey::AppPublicKey` variant. Any single attested participant can call `respond_ckd` with an arbitrary forged response and the contract will accept it, resume all waiting yields, and deliver the forged confidential key to every queued caller — bypassing the threshold-signature requirement entirely.

### Finding Description
In `respond_ckd`, the contract branches on the `app_public_key` variant of the request:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no validation whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` verifies the BLS proof of correct derivation against the domain's public key, so a forged response is rejected on-chain. For the `AppPublicKey` ("privately verifiable", legacy) variant, the branch body is empty — the contract performs zero validation and immediately calls `resolve_yields_for`, which resumes every queued yield with the attacker-supplied bytes. [2](#0-1) 

By contrast, `respond` (ECDSA/EdDSA signatures) always validates the submitted signature cryptographically against the derived public key before resuming yields, providing an implicit threshold-enforcement guarantee even without an explicit quorum check. [3](#0-2) 

The `respond_ckd` path has no equivalent safeguard for `AppPublicKey` requests.

### Impact Explanation
A single Byzantine attested participant (strictly below the signing threshold) can:
1. Observe any pending `AppPublicKey` CKD request in the contract's `pending_ckd_requests` map.
2. Construct an arbitrary `CKDResponse { big_y, big_c }` — values that do not correspond to any honest MPC computation.
3. Call `respond_ckd(request, forged_response)` from their attested account.
4. The contract accepts the call, drains the entire fan-out queue for that request, and delivers the forged encrypted key to every caller who submitted the same request.

The callers receive a `CKDResponse` that is not derived from the MPC network's master BLS key. Any secret material they decrypt from it is attacker-controlled. This constitutes **unauthorized confidential key derivation output without the required participant authorization**, matching the Critical impact tier.

### Likelihood Explanation
- The `AppPublicKey` variant is explicitly supported and documented as the legacy path; real user traffic uses it.
- Only one attested participant is required — no collusion above threshold is needed.
- The attacker's node is already inside the TEE attestation perimeter (a Byzantine-but-attested participant), which is the standard adversary model for threshold systems.
- The attack requires no special timing, no network-level access, and no privileged operator role beyond being an attested participant.

### Recommendation
Apply the same pattern used for `AppPublicKeyPV`: require a verifiable proof of correct derivation for every `respond_ckd` call, or reject `AppPublicKey` requests at the contract level if on-chain verification is impossible for that variant. At minimum, document that `AppPublicKey` requests carry no on-chain threshold enforcement and gate their acceptance behind an explicit governance flag. The analog fix in the ERC20 context was adopting `SafeERC20`; here the analog is adopting a mandatory on-chain output check (as already exists for `AppPublicKeyPV`) for all CKD response variants before resuming yields.

### Proof of Concept
1. User submits `request_app_private_key` with `app_public_key: AppPublicKey(some_bls_g1_point)`. The request enters `pending_ckd_requests`.
2. Attacker (a single attested participant) calls:
   ```
   respond_ckd(
     request = <the observed CKDRequest>,
     response = CKDResponse { big_y: [attacker_bytes; 48], big_c: [attacker_bytes; 48] }
   )
   ```
3. `respond_ckd` passes all guards (attested participant, running state, `AppPublicKey` branch is empty), reaches `resolve_yields_for`, and resumes all queued yields with the forged bytes.
4. The user's `request_app_private_key` promise resolves with the attacker-supplied `CKDResponse`. The user decrypts attacker-controlled key material, believing it to be their MPC-derived application secret.

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
