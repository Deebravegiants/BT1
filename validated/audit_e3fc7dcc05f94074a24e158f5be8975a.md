### Title
Caller-Controlled Finality Bypass in `verify_foreign_transaction` Enables Double-Spend via Block Reorganization - (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`, `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` flow accepts caller-supplied finality parameters (`EvmFinality`, `SolanaFinality`, `BlockConfirmations`) without enforcing any minimum threshold at either the contract or node layer. Any unprivileged caller can submit a request with the weakest available finality level (`EvmFinality::Latest`, `SolanaFinality::Processed`, or `BlockConfirmations(1)`), causing the MPC network to produce a valid threshold signature attesting to a transaction that has not reached sufficient finality. If the underlying foreign-chain transaction is subsequently reorganized away, the signed attestation remains valid and can be used to claim bridge funds — a double-spend.

---

### Finding Description

The `ForeignChainRpcRequest` DTO carries a caller-controlled finality field:

```rust
pub struct EvmRpcRequest {
    pub tx_id: EvmTxId,
    pub extractors: Vec<EvmExtractor>,
    pub finality: EvmFinality,   // ← entirely caller-supplied
}

pub enum EvmFinality { Latest, Safe, Finalized }
``` [1](#0-0) 

The contract's `verify_foreign_transaction` entry point performs no minimum-finality validation — it only checks that the chain is whitelisted and the domain is correct: [2](#0-1) 

The finality value is forwarded verbatim to the node-side inspector. In `EvmInspector::verify_finality_level`, when the caller passes `EvmFinality::Latest`, the node queries `eth_getBlockByNumber("latest")` and accepts the transaction as long as the latest block number is ≥ the receipt's block number — a check that is trivially satisfied for any mined transaction: [3](#0-2) 

For Bitcoin, the caller supplies `BlockConfirmations` directly in the request, and the inspector enforces only that threshold — meaning a caller can set `confirmations: 1` (or even `0`) and the inspector will accept it: [4](#0-3) 

The signed payload commits to the full `ForeignChainRpcRequest` (including the finality field), so the signature is cryptographically valid even when produced for a `Latest`-finality or 1-confirmation request: [5](#0-4) 

---

### Impact Explanation

The MPC network's `verify_foreign_transaction` feature is explicitly designed for bridge inbound flows where NEAR contracts must react to foreign-chain events without a trusted relayer. Bridge contracts that consume the resulting attestation reasonably expect the MPC network to enforce minimum finality requirements. Because the MPC network signs for any finality level the caller requests, an attacker can:

1. Send a transaction on an EVM chain (e.g., Polygon, which has had multi-block reorgs historically).
2. Immediately call `verify_foreign_transaction` with `EvmFinality::Latest` before the transaction is finalized.
3. Receive a valid MPC threshold signature attesting the transaction occurred.
4. Use the attestation to claim bridge funds on NEAR.
5. If a reorg removes the original transaction, the attacker retains both the NEAR-side funds and the foreign-chain funds — a double-spend.

This matches the **High** impact category: *forged foreign-chain verification / light-client-style verification bypass that causes invalid bridge execution or double-spend conditions.*

---

### Likelihood Explanation

- **No privileged role required.** Any account can call `verify_foreign_transaction` with a 1 yoctoNEAR deposit.
- **Finality parameters are fully attacker-controlled.** The contract and nodes impose no floor.
- **Reorgs are realistic.** Polygon (a supported chain) has experienced reorgs of 100+ blocks. EVM `Latest` blocks are reorganizable by definition. Bitcoin 1-confirmation transactions are routinely reorganized.
- **The attack window is short but automatable.** The attacker submits the MPC request in the same block or immediately after the foreign-chain transaction, before finality is reached.

---

### Recommendation

1. **Enforce a per-chain minimum finality level on-chain.** The `ChainEntry` (or equivalent on-chain foreign-chain configuration) should carry a `minimum_finality` field voted in by participants. `verify_foreign_transaction` should reject requests whose finality parameter is weaker than the on-chain minimum.

2. **Reject `EvmFinality::Latest` and `SolanaFinality::Processed` at the contract level** for production chains, or at minimum document that bridge contracts must validate the finality level in the signed payload before releasing funds.

3. **Enforce a minimum `BlockConfirmations` for Bitcoin** (e.g., 6) in the on-chain chain configuration, and have the contract reject requests below that floor.

4. **Share the finality-enforcement logic** between the contract and the node inspectors, analogous to the ZetaChain long-term recommendation to share code between `InboundTracker` and `ObserveTxIn`.

---

### Proof of Concept

The following demonstrates that the contract accepts and enqueues a request with `EvmFinality::Latest` without any rejection:

```rust
// Any unprivileged caller can submit this:
let request_args = VerifyForeignTransactionRequestArgs {
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Ethereum(EvmRpcRequest {
        tx_id: EvmTxId(tx_hash_bytes),
        extractors: vec![EvmExtractor::BlockHash],
        finality: EvmFinality::Latest,   // weakest finality — no minimum enforced
    }),
};
// Contract accepts this; MPC nodes will sign for a non-finalized block.
contract.verify_foreign_transaction(request_args);
```

The node-side `verify_finality_level` call with `EthereumFinality::Latest` resolves to `FinalityTag::Latest`, queries the current chain head, and passes as long as the head block number ≥ the receipt's block number — a condition satisfied immediately after mining: [6](#0-5) 

The resulting `VerifyForeignTransactionResponse` carries a valid MPC threshold signature over a payload that includes `EvmFinality::Latest`, which a bridge contract cannot distinguish from a `Finalized`-level attestation unless it explicitly inspects the finality field in the signed payload.             callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );

        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

    async fn extract(
        &self,
        transaction: BitcoinTransactionHash,
        block_confirmations_threshold: BlockConfirmations,
        extractors: Vec<BitcoinExtractor>,
    ) -> Result<Vec<BitcoinExtractedValue>, ForeignChainInspectionError> {
        let request_parameters = GetRawTransactionArgs {
            transaction_hash: TransportBitcoinTransactionHash::from(*transaction),
            verbose: VERBOSE_RESPONSE,
        };

        // TODO(#1978): add retry mechanism if the error from the request is transient
        let rpc_response: GetRawTransactionVerboseResponse = self
            .client
            .request(GET_RAW_TRANSACTION_METHOD, &request_parameters)
            .await?;

        let transaction_block_confirmation = rpc_response.confirmations.into();
        let enough_block_confirmations =
            block_confirmations_threshold <= transaction_block_confirmation;

        if !enough_block_confirmations {
            return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations {
                expected: block_confirmations_threshold,
                got: transaction_block_confirmation,
            });
        }

        self.verify_block_is_canonical(rpc_response.blockhash)
            .await?;

        let extracted_values = extractors
            .iter()
            .map(|extractor| extractor.extract_value(&rpc_response))
            .collect();

        Ok(extracted_values)
    }
