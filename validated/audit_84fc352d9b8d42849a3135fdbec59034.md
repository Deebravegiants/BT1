### Title
`valid_proposals` Mutated Before Commitment-Match Guard in `validate_proposal` — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

In `validate_proposal.rs`, the validator inserts the incoming proposal into the shared `valid_proposals` store **before** checking whether the batcher-computed commitment matches the proposer-supplied `ProposalFin.proposal_commitment`. A malicious proposer can exploit this ordering to plant an orphaned entry in `valid_proposals` under a real commitment value, potentially poisoning the reproposal path for a later legitimate proposal that produces the same commitment.

### Finding Description

The CEI (Checks-Effects-Interactions) violation is at lines 238–247 of `validate_proposal.rs`:

```rust
// Update valid_proposals before sending fin to avoid a race condition
// with `repropose` being called before `valid_proposals` is updated.
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

// TODO(matan): Switch to signature validation.
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}

Ok(built_block)
``` [1](#0-0) 

The **Effect** (`insert_proposal`) executes before the **Check** (`built_block != received_fin.proposal_commitment`). When the check fails, the function returns `Err(ProposalFinMismatch)` and the `fin_sender` is never written to, so consensus never votes for this proposal. However, the entry is already resident in `valid_proposals` under the batcher-computed commitment `built_block` (carried inside `finished_info`).

The `valid_proposals` store is an `Arc<Mutex<BuiltProposals>>` shared between the proposer path (`build_proposal`) and the validator path (`validate_proposal`), and it is the sole source of truth for the `repropose` call. [2](#0-1) 

The `ProposalInitValidation` struct that governs what the validator accepts is constructed from live node state (`previous_proposal_init`, `l1_da_mode`, `l2_gas_price_fri`, `fee_actual`) at the moment `validate_current_round_proposal` is called: [3](#0-2) 

### Impact Explanation

**Impact: High — reproposal path serves wrong `ProposalInit` metadata.**

Attack scenario:

1. **Round R (malicious proposer):** The attacker proposes transactions `T1, T2`. The batcher computes commitment `C1`. The attacker deliberately sets `ProposalFin.proposal_commitment = C2 ≠ C1`. The validator calls `insert_proposal(init_R, [T1,T2], pid_R, finished_info{C1})`, then detects the mismatch and returns `Err(ProposalFinMismatch)`. An orphaned entry keyed to `C1` is now in `valid_proposals` with `init_R` (round R, attacker as proposer).

2. **Round R+1 (legitimate proposer):** A legitimate proposer sends the same transactions `T1, T2`. The batcher again computes `C1`. `ProposalFin.proposal_commitment = C1` (correct). The validator calls `insert_proposal(init_R+1, [T1,T2], pid_R+1, finished_info{C1})`. Depending on whether `BuiltProposals` allows overwrite or first-write-wins for the same commitment key, the orphaned entry from round R may shadow the legitimate entry.

3. **Reproposal:** Consensus votes for `C1` (from the legitimate round R+1 proposal) and later calls `repropose(C1)`. If `valid_proposals` returns the orphaned entry (`init_R`, attacker as proposer, round R), the re-streamed `ProposalInit` carries wrong round, wrong proposer identity, and potentially wrong gas-price metadata. Validators receiving the reproposal will reject it (wrong proposer for round R+1), causing consensus to stall or time out for that height.

The `pending_validation_rounds` guard in `single_height_consensus.rs` prevents duplicate validation of the same round, but it does not prevent the cross-round poisoning described above. [4](#0-3) 

### Likelihood Explanation

**Likelihood: Medium.** The attacker must be the elected proposer for at least one round at a given height (a normal Tendermint condition when a validator is Byzantine). The attack requires predicting the commitment that the next round's legitimate proposer will produce, which is feasible by observing the public mempool. No privileged access beyond being a scheduled proposer is required.

### Recommendation

Move `valid_proposals.insert_proposal` to **after** the commitment-match check, following the CEI pattern:

```rust
// Check first
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}

// Effect only on success
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

Ok(built_block)
```

The comment justifying the current order ("avoid a race condition with `repropose`") applies only to the **proposer** path (`build_proposal.rs` lines 134–136), where the fin is sent immediately after insertion and a concurrent `repropose` call is genuinely possible. In the **validator** path the fin is sent by `validate_and_send` only after `validate_proposal` returns `Ok`, so there is no race condition that requires pre-insertion. [5](#0-4) 

### Proof of Concept

1. Deploy a Byzantine validator node that is scheduled as proposer for round 0 at height H.
2. In round 0, broadcast a `ProposalPart::Transactions` containing transactions `T1, T2`, followed by `ProposalPart::Fin { proposal_commitment: WRONG_HASH, executed_transaction_count: 2 }`.
3. Every honest validator node will: (a) execute `T1, T2` via the batcher and obtain the real commitment `C_real`; (b) call `valid_proposals.insert_proposal(init_round0, [T1,T2], pid_0, finished_info{C_real})`; (c) detect `C_real ≠ WRONG_HASH` and return `Err(ProposalFinMismatch)` — but the orphaned entry under `C_real` is already stored.
4. In round 1, the legitimate proposer broadcasts the same `T1, T2` with the correct `ProposalFin { proposal_commitment: C_real }`. Honest validators insert a second entry for `C_real` into `valid_proposals`.
5. Consensus reaches a precommit quorum for `C_real`. When `repropose` is triggered (e.g., on a new round after a timeout), the node looks up `C_real` in `valid_proposals`. If the store is first-write-wins, it returns `init_round0` (Byzantine proposer, round 0) instead of `init_round1`, causing the re-streamed `ProposalInit` to carry the wrong proposer and round, which honest validators reject. [6](#0-5)
