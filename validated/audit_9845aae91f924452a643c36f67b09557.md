### Title
Unguarded `l2bridge` Parameter in `sendETHToL1ViaBridge` Enables Block Stuffing — (`contracts/bridges/UnichainMessenger.sol`)

### Summary
`UnichainMessenger.sendETHToL1ViaBridge` accepts an arbitrary `l2bridge` address from any caller with no access control. An attacker can supply a gas-bomb contract as `l2bridge`, causing the function to forward nearly all block gas to a maximally consuming `bridgeETHTo` implementation, enabling block stuffing on Unichain.

### Finding Description
`UnichainMessenger` exposes `sendETHToL1ViaBridge` as a public `external payable` function with only a `nonReentrant` guard and a `msg.value == value` check:

```solidity
// contracts/bridges/UnichainMessenger.sol:24-27
function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value)
    external payable nonReentrant
{
    if (msg.value != value) revert MismatchedMsgValue();
    IUnichainMessenger(l2bridge).bridgeETHTo{ value: value }(target, DEFAULT_GAS_LIMIT, bytes(""));
}
```

There is no `onlyRole`, no allowlist, and no validation of `l2bridge`. The `DEFAULT_GAS_LIMIT = 200_000` constant is forwarded as the `_minGasLimit` argument to the callee — it is a hint to the bridge protocol, **not** an EVM gas cap on the call. The EVM forwards `gasleft() - some_overhead` to the external call. A malicious `bridgeETHTo` implementation can consume all forwarded gas via an infinite loop or `SSTORE` spam.

The intended callers (`RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge`, `RSETHPool.bridgeAssetsViaNativeBridge`, etc.) are correctly gated with `onlyRole(BRIDGER_ROLE)` and pass a governance-set `l2Bridge` address. However, the messenger contract itself is unprotected, so an attacker bypasses the pool entirely by calling the messenger directly.

### Impact Explanation
An attacker can repeatedly call `sendETHToL1ViaBridge(gasBomb, anyAddress, 1)` with `msg.value = 1 wei`, each transaction consuming nearly the full Unichain block gas limit. By submitting enough such transactions per block (paying the required gas fees), the attacker can saturate blocks and delay or prevent the `BRIDGER_ROLE` operator's legitimate `bridgeAssetsViaNativeBridge` transaction from being included. This is a **Low — Block Stuffing** impact as defined in the allowed scope.

### Likelihood Explanation
The function is publicly callable with no preconditions beyond providing `msg.value == value` (minimum 1 wei). The attacker only needs to deploy a gas-bomb contract and call the messenger directly. The cost is the gas fees paid, making sustained stuffing expensive but not impossible for a motivated attacker targeting a specific bridge window.

### Recommendation
Add an access control guard to `sendETHToL1ViaBridge` so only authorized callers (e.g., the corresponding pool contract or accounts with `BRIDGER_ROLE`) can invoke it. Alternatively, validate `l2bridge` against an immutable or governance-set allowlist of legitimate bridge addresses. The same fix should be applied to `BaseMessenger`, `OptimismMessenger`, `ArbitrumMessenger`, and `ScrollMessenger`, which share the identical pattern.

        uint256 i;
        while (true) { unchecked { i++; } } // infinite loop, consumes all gas
    }
    receive() external payable {}

    function exploit(address messenger) external payable {
        GasBomb bomb = new GasBomb();
        // Call with 1 wei; the gas bomb consumes all remaining block gas

            address(bomb), address(0xdead), 1
        );
    }
}
