Audit Report

## Title
Missing Staleness, Round Completeness, and Zero-Price Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables rsETH Over-Minting at Stale Prices - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, applying no staleness threshold (`updatedAt`), no round completeness guard (`answeredInRound >= roundId`), and no zero-price assertion. This price feeds directly into `LRTDepositPool.getRsETHAmountToMint()`, meaning a depositor can mint rsETH against a stale, inflated LST price, receiving more rsETH than the deposited asset is worth and diluting all existing rsETH holders.

## Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, the implementation is:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All four other return values — `roundId`, `startedAt`, `updatedAt`, `answeredInRound` — are silently discarded. There is no `block.timestamp - updatedAt <= heartbeat` check, no `answeredInRound >= roundId` guard, and no `price > 0` assertion.

This price is consumed by `LRTDepositPool.getRsETHAmountToMint()` at line 520:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The contrast with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` (lines 30–32) is direct: that contract checks `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0`. `ChainlinkPriceOracle` — the contract used for L1 deposit pricing — applies none of these guards.

**Exploit path:**
1. A supported LST (e.g., stETH) depegs below its Chainlink-reported price (e.g., true value 0.97 ETH, oracle still reports 1.0 ETH, within the 24-hour heartbeat / 0.5% deviation threshold).
2. An unprivileged attacker calls `LRTDepositPool.depositAsset(stETH, amount, minRSETH, referralId)`.
3. `getRsETHAmountToMint` queries the stale price (1.0 ETH) and mints rsETH as if the full 1.0 ETH/stETH value was deposited.
4. The attacker receives rsETH worth more ETH than they deposited; the shortfall is borne by all existing rsETH holders through dilution of the backing ratio.
5. The attack is repeatable for the entire duration of the staleness window with no privileged access.

No existing check in `_beforeDeposit` or `getRsETHAmountToMint` mitigates this; the `minRSETHAmountExpected` slippage guard only protects the depositor, not existing holders.

## Impact Explanation

**High — Theft of unclaimed yield / value dilution of existing rsETH holders.**

When a stale, inflated price is used for a deposited LST, the minted rsETH exceeds the ETH value actually added to the protocol. Every existing rsETH holder's share is diluted by the difference. Across multiple depositors during a staleness window, the cumulative extraction of value from existing holders is material and constitutes theft of their accrued yield and backing value.

## Likelihood Explanation

**Medium.** Chainlink LST/ETH feeds on mainnet (e.g., stETH/ETH) have 24-hour heartbeat intervals and 0.5% deviation thresholds. A depeg of less than 0.5% within a 24-hour window will not trigger an oracle update. LST depeg events are historically documented (stETH during the Merge, ETHx during Stader incidents). Network congestion can further delay updates. The attacker requires no special privileges — only the ability to call `depositAsset()`.

## Recommendation

Replace the bare `latestRoundData()` call in `ChainlinkPriceOracle.getAssetPrice()` with full validation, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= MAX_STALENESS_SECONDS, "Price too old");
```

`MAX_STALENESS_SECONDS` should be configured per-feed based on the Chainlink heartbeat (e.g., 86 400 s for a 24-hour feed, with a small buffer), stored alongside the feed address in the `assetPriceFeed` mapping or a parallel `assetHeartbeat` mapping.

## Proof of Concept

**Foundry fork test outline:**

1. Fork Ethereum mainnet at a block where the stETH/ETH Chainlink feed's `updatedAt` is near its 24-hour heartbeat limit.
2. Warp `block.timestamp` forward by 23 hours (still within heartbeat, no revert).
3. Separately manipulate or observe a stETH market price below the last reported Chainlink price (simulating a depeg).
4. Call `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")` from an unprivileged address.
5. Assert that the rsETH minted corresponds to the stale oracle price (1.0 ETH/stETH), not the true market price (e.g., 0.97 ETH/stETH).
6. Assert that the rsETH/ETH backing ratio has decreased for pre-existing holders, confirming dilution.
7. Confirm the same call reverts if the fix (staleness check) is applied.
