### Title
L2 Pool `deposit()` Functions Lack Slippage Protection, Allowing Users to Receive Fewer `wrsETH` Than Expected When Oracle Rate Updates - (`contracts/pools/RSETHPoolV3.sol`)

### Summary

All L2 pool `deposit()` functions compute the `wrsETH` amount to mint using the live oracle rate at execution time, but accept no `minRsETHAmountExpected` parameter. A user who previews their output via `viewSwapRsETHAmountAndFee()` and then submits a `deposit()` transaction can silently receive fewer `wrsETH` tokens than expected if the oracle rate is updated between their off-chain query and on-chain execution. The L1 `LRTDepositPool` enforces this protection; the L2 pools do not.

### Finding Description

Every L2 pool variant (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolNoWrapper`, `RSETHPoolV2`, `RSETHPoolV2ExternalBridge`) exposes a `deposit()` function that internally calls `viewSwapRsETHAmountAndFee()`, which divides the input amount by the live `getRate()` value from the oracle:

```solidity
// RSETHPoolV3.sol – deposit (ETH path)
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
wrsETH.mint(msg.sender, rsETHAmount);
```

```solidity
// viewSwapRsETHAmountAndFee
uint256 rsETHToETHrate = getRate();          // live oracle rate
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The oracle rate on L2 is stored in `RSETHRateReceiver` and is updated asynchronously via LayerZero whenever anyone calls `updateRate()` on the L1 `RSETHRateProvider` / `RSETHMultiChainRateProvider`. Because the delivery of a LayerZero message is a separate on-chain transaction, the rate stored in the receiver can change at any block, including the block in which a user's `deposit()` is mined.

A user who calls `viewSwapRsETHAmountAndFee()` off-chain to preview their output, then submits `deposit()`, has no mechanism to enforce a minimum output. If the rate increases between preview and execution, the user receives fewer `wrsETH` than expected with no revert or warning.

By contrast, the L1 `LRTDepositPool` explicitly accepts and enforces a `minRSETHAmountExpected` parameter in both `depositETH()` and `depositAsset()`, reverting with `MinimumAmountToReceiveNotMet` if the minted amount falls below the caller's threshold:

```solidity
// LRTDepositPool.sol – _beforeDeposit
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

This protection is entirely absent from all L2 pool deposit paths.

### Impact Explanation

**Low – Contract fails to deliver promised returns, but doesn't lose value.**

A depositor receives fewer `wrsETH` tokens than the amount returned by `viewSwapRsETHAmountAndFee()` at query time. In pure ETH-value terms the user is not worse off (fewer tokens each worth proportionally more), but the contract does not honour the output amount the user observed and planned around. There is no mechanism for the user to protect themselves or to have the transaction revert if the output is unacceptable.

### Likelihood Explanation

The rsETH/ETH rate monotonically increases as staking rewards accrue, and `updateRate()` on the provider is permissionless (callable by anyone who pays the LayerZero fee). Rate updates are routine operational events. Any user who previews output and submits a deposit in a different block from the preview is exposed. The gap between preview and execution is unavoidable for EOA users.

### Recommendation

Add a `minRsETHAmountExpected` parameter to all L2 pool `deposit()` overloads, mirroring the L1 `LRTDepositPool` pattern:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused ...
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert SlippageExceeded();
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

### Proof of Concept

1. The rsETH/ETH rate on L2 is currently `1.05e18` (105 ETH per 100 rsETH).
2. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain → preview shows `~0.952 wrsETH`.
3. Before the user's `deposit(1 ether, referralId)` tx is mined, a LayerZero message (sent earlier by anyone calling `updateRate()` on L1) is processed, updating the L2 rate to `1.10e18`.
4. The user's `deposit()` executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 = ~0.909 wrsETH`.
5. The user receives `~0.909 wrsETH` instead of the expected `~0.952 wrsETH` — a ~4.5% shortfall — with no revert and no recourse.
