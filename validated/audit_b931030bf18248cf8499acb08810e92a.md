### Title
Deposits for Competing `propose_update` Proposals Are Permanently Stuck in the Contract When Any Update Executes - (`File: crates/contract/src/update.rs`)

---

### Summary

When `do_update` executes a winning contract-upgrade proposal, it unconditionally clears **all** other pending proposals via `self.entries.clear()`. Because `UpdateEntry` never records the proposer's account ID or the deposit amount, the NEAR tokens paid by proposers of the cleared (non-winning) proposals are permanently locked in the contract with no retrieval path.

---

### Finding Description

`propose_update` requires a storage-proportional deposit from each proposer: [1](#0-0) 

The deposit is accepted and only the excess is refunded. The `required` amount stays in the contract. However, the `UpdateEntry` struct that is stored records only the update payload and its byte size — **not** the depositor's identity or the deposit amount: [2](#0-1) 

When threshold votes are reached and `do_update` fires, it removes the winning entry and then bulk-clears every other pending proposal: [3](#0-2) 

The storage bytes are freed (reducing the contract's storage obligation), but the NEAR tokens that were paid to cover those bytes are **never returned** to the proposers of the cleared entries. Because neither the account ID nor the deposit amount was ever persisted in `UpdateEntry`, there is no information available to issue refunds. The freed deposit value simply accumulates in the contract's own balance with no withdrawal mechanism.

---

### Impact Explanation

Every time a contract upgrade reaches threshold while competing proposals exist, the deposits of all non-winning proposers are permanently lost. For a large contract binary (e.g., 1 MB WASM), `required_deposit` equals `env::storage_byte_cost() × bytes_used`, which at NEAR's current storage cost (~10 yoctoNEAR/byte) amounts to roughly **10 NEAR per proposal**. Multiple competing proposals in a contested governance round multiply this loss. The funds are irrecoverably locked in the contract balance — there is no admin withdrawal, no refund path, and no record of who paid what.

This matches the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The trigger is the normal governance flow: any participant can call `propose_update` (paying the deposit), and any other participant can propose a competing update. This is explicitly supported — the contract allows multiple simultaneous proposals. Whenever a threshold vote executes one of them, all others are silently cleared. No adversarial action is required; the loss occurs as a side-effect of routine governance.

---

### Recommendation

Persist the proposer's `AccountId` and the exact deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over the remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the pattern already used in `propose_update` for excess-deposit refunds.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, paying ~5 NEAR deposit.
2. Participant B calls `propose_update` with a different 500 KB binary, paying ~5 NEAR deposit.
3. Threshold participants vote for B's proposal; `vote_update` calls `do_update(&B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry (and its 5 NEAR deposit) is erased with no refund.
5. A's 5 NEAR is now part of the contract's balance. No function exists to recover it. [4](#0-3) [5](#0-4)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
