### Title
ERC1155 Deterministic Address Collision Enables Cross-Token Asset Substitution - (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.deriveDeterministicAddress` truncates a 256-bit keccak256 hash to 160 bits to produce a virtual token identifier for ERC1155 assets. This is collision-prone by the birthday paradox. Because `initTransfer1155` does not verify that the computed deterministic address matches the already-registered `multiTokens` entry, an attacker who finds a collision between two `(tokenAddress, tokenId)` pairs can lock Token B while the bridge records the transfer as Token A, then redeem Token A from other users' locked deposits.

### Finding Description

`deriveDeterministicAddress` is defined as:

```solidity
function deriveDeterministicAddress(
    address tokenAddress,
    uint256 tokenId
) public pure returns (address) {
    return address(bytes20(keccak256(abi.encodePacked(tokenAddress, tokenId))));
}
``` [1](#0-0) 

`bytes20(keccak256(...))` takes the first 20 bytes of a 32-byte hash, discarding 96 bits. For every 160-bit output address there are 2^96 distinct inputs that produce it. By the birthday paradox, an attacker needs only ~2^80 hash evaluations to find two distinct `(tokenAddress, tokenId)` pairs that collide to the same deterministic address.

`logMetadata1155` writes the first pair that arrives into `multiTokens[det_addr]` and rejects any subsequent call with a different pair:

```solidity
if (multiToken.tokenAddress == address(0)) {
    multiToken.tokenAddress = tokenAddress;
    multiToken.tokenId = tokenId;
} else {
    if (multiToken.tokenAddress != tokenAddress || multiToken.tokenId != tokenId) {
        revert ERC1155MappingMismatch();
    }
}
``` [2](#0-1) 

`initTransfer1155`, however, performs **no such consistency check**. It computes `det_addr`, locks the caller-supplied ERC1155 token, and emits `InitTransfer` with `det_addr` as the token identifier — without verifying that `multiTokens[det_addr]` already maps to the same `(tokenAddress, tokenId)`:

```solidity
address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);
IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");
// ...
emit BridgeTypes.InitTransfer(msg.sender, deterministicToken, currentOriginNonce, amount, ...);
``` [3](#0-2) 

`finTransfer` resolves the actual ERC1155 token to release by reading `multiTokens[payload.tokenAddress]`:

```solidity
MultiTokenInfo memory multiToken = multiTokens[payload.tokenAddress];
// ...
} else if (multiToken.tokenAddress != address(0)) {
    IERC1155(multiToken.tokenAddress).safeTransferFrom(
        address(this), payload.recipient, multiToken.tokenId, payload.amount, ""
    );
``` [4](#0-3) 

### Impact Explanation

An attacker who finds a collision `deriveDeterministicAddress(A, idA) == deriveDeterministicAddress(B, idB) == det_addr` can:

1. Call `logMetadata1155(A, idA)` → `multiTokens[det_addr] = {A, idA}` (registers Token A as the canonical asset for `det_addr`)
2. Call `initTransfer1155(B, idB, amount, ...)` → locks Token B in the bridge, emits `InitTransfer` with `det_addr`
3. NEAR side observes the `InitTransfer` event for `det_addr` and credits the attacker
4. Attacker initiates a return transfer from NEAR; NEAR MPC signs a `finTransfer` for `det_addr`
5. `finTransfer` reads `multiTokens[det_addr] = {A, idA}` and releases Token A to the attacker

The attacker deposits Token B (which they own) and withdraws Token A (which belongs to other bridge users). This is an unauthorized custody escape of locked bridge assets.

**Impact category:** Critical — unauthorized release/withdrawal of locked bridge assets through token-mapping divergence.

### Likelihood Explanation

The birthday attack on a 160-bit output space requires approximately 2^80 hash evaluations. The attacker controls both `tokenAddress` (160 bits) and `tokenId` (256 bits), giving 416 bits of freely variable input — more freedom than the analogous NapierRouter case (320 bits). The attack is currently expensive (~$1–2M equivalent in compute) but is feasible for a well-resourced attacker and becomes cheaper as computing power advances. The Sherlock judging panel accepted the same class of attack as Medium severity in the NapierRouter and Arcadia contests.

### Recommendation

Add a consistency check inside `initTransfer1155` to verify that if `multiTokens[det_addr]` is already populated, it matches the caller-supplied `(tokenAddress, tokenId)`:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    ...
    address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);

+   MultiTokenInfo storage multiToken = multiTokens[deterministicToken];
+   if (multiToken.tokenAddress != address(0)) {
+       require(
+           multiToken.tokenAddress == tokenAddress && multiToken.tokenId == tokenId,
+           "ERC1155MappingMismatch"
+       );
+   }

    IERC1155(tokenAddress).safeTransferFrom(...);
    ...
}
```

A stronger long-term fix is to replace the truncated-hash derivation with a registry-based approach: store the `(tokenAddress, tokenId)` → `det_addr` mapping explicitly (e.g., using a counter-based nonce) so that no two distinct pairs can ever share the same identifier, eliminating the collision surface entirely.

### Proof of Concept

1. Attacker pre-computes offline: find `(A, idA)` and `(B, idB)` such that `keccak256(abi.encodePacked(A, idA))[:20] == keccak256(abi.encodePacked(B, idB))[:20]`. Birthday attack cost: ~2^80 keccak256 evaluations.
2. Attacker calls `logMetadata1155(A, idA)` on `OmniBridge`. `multiTokens[det_addr]` is now `{A, idA}`.
3. Attacker calls `initTransfer1155(B, idB, amount, fee, 0, "attacker.near", "")`. The bridge locks `amount` of Token B from the attacker. The `InitTransfer` event is emitted with `tokenAddress = det_addr`.
4. NEAR bridge observes the event and mints/credits `amount` of `det_addr`-mapped tokens to `attacker.near`.
5. Attacker initiates a withdrawal from NEAR back to EVM. NEAR MPC signs a `finTransfer` payload with `tokenAddress = det_addr`.
6. Attacker calls `finTransfer(signature, payload)` on EVM. The bridge reads `multiTokens[det_addr] = {A, idA}` and executes `IERC1155(A).safeTransferFrom(bridge, attacker_evm, idA, amount, "")`.
7. Attacker receives Token A (locked by legitimate users) while having only deposited Token B. [1](#0-0) [5](#0-4) [6](#0-5) [4](#0-3)
