### Title
`maxFeeMintAmountPerDay` Uninitialized to Zero Causes `updateRSETHPrice()` to Permanently Revert When Protocol Fees Are Due - (File: contracts/LRTOracle.sol)

---

### Summary

`LRTOracle` contains a fee-minting mechanism that mints rsETH to the protocol treasury whenever TVL increases. A critical guard variable, `maxFeeMintAmountPerDay`, is never initialized in `initialize()` or `reinitialize()`, defaulting to `0`. When `protocolFeeInBPS > 0` is set in `LRTConfig` and TVL grows, `_updateRsETHPrice()` computes a non-zero `rsethAmountToMintAsProtocolFee` and passes it to `_checkAndUpdateDailyFeeMintLimit()`. Because `maxFeeMintAmountPerDay == 0`, the limit check always reverts, permanently blocking both protocol fee minting and rsETH price updates.

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield + temporary freezing of funds.**

1. Protocol fees (rsETH minted to treasury) can never be collected while `maxFeeMintAmountPerDay == 0` and `protocolFeeInBPS > 0` — permanent freezing of unclaimed yield.
2. `updateRSETHPrice()` reverts for every caller (public function, no privileged bypass) whenever TVL increases, leaving the rsETH price stale. A stale price distorts deposit and withdrawal amounts for all users — temporary freezing of funds.

---

### Likelihood Explanation

**Medium.**

- `protocolFeeInBPS` is set in `LRTConfig` by the manager role; `maxFeeMintAmountPerDay` is a separate knob in `LRTOracle` that must be configured independently.
- Neither `initialize()` nor `reinitialize()` sets `maxFeeMintAmountPerDay`, so any deployment that enables fees without explicitly calling `setMaxFeeMintAmountPerDay()` triggers the revert.
- The two settings live in different contracts with no cross-check, making the oversight realistic.

---

### Recommendation

Initialize `maxFeeMintAmountPerDay` to a safe non-zero default in `initialize()` (or `reinitialize()`), or modify `_checkAndUpdateDailyFeeMintLimit` to treat `maxFeeMintAmountPerDay == 0` as "unlimited" (skip the cap check) rather than "zero cap":

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }
    // Treat 0 as "no cap" to avoid reverting when uninitialized
    if (maxFeeMintAmountPerDay != 0 &&
        currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
    }
    currentPeriodMintedFeeAmount += feeAmount;
}
```

### Proof of Concept

1. Deploy `LRTConfig` and `LRTOracle`; call `initialize()` on both.
2. Call `lrtConfig.setProtocolFeeBps(100)` — 1% fee enabled.
3. Do **not** call `lrtOracle.setMaxFeeMintAmountPerDay(...)` — `maxFeeMintAmountPerDay` remains `0`.
4. Simulate TVL growth: asset prices rise so `totalETHInProtocol > previousTVL`.
5. Call `lrtOracle.updateRSETHPrice()` (public, callable by anyone).
6. Transaction reverts: `DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)`.
7. rsETH price is permanently stale; protocol fees are never minted to treasury.
