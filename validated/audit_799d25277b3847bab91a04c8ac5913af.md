### Title
User-Controlled `BlockConfirmations: 0` Bypasses Bitcoin Finality Check Before MPC Signing - (File: crates/foreign-chain-inspector/src/bitcoin/inspector.rs)

### Summary
The `verify_foreign_transaction` flow accepts a user-supplied `confirmations` field in `BitcoinRpcRequest` with no minimum enforcement. Setting `confirmations: 0` causes the `BitcoinInspector` to skip the confirmation threshold check entirely, allowing MPC nodes to produce a valid threshold signature over an unconfirmed or barely-confirmed Bitcoin transaction. In a bridge context this enables a double-spend: the attacker obtains a signed payload for a transaction that is subsequently reorged.

### Finding Description

The `BitcoinRpcRequest` struct exposes a `confirmations` field of type `BlockConfirmations(u64)` that is passed directly from the user's `verify_foreign_transaction` call through to the off-chain inspector without any minimum-value enforcement.

In `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, the confirmation check is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
``` [1](#0-0) 

When `block_confirmations_threshold = 0`, the condition `0 <= any_value` is always `true`, so the guard never fires. The inspector proceeds to extract values and the MPC nodes sign the payload.

The `BlockConfirmations` type carries no minimum constraint:

```json
"BlockConfirmations": {
  "type": "integer",
  "format": "uint64",
  "minimum": 0.0
}
``` [2](#0-1) 

The contract's `verify_foreign_transaction` method validates domain, gas, deposit, and chain support — but never inspects the `confirmations` value inside the request body: [3](#0-2) 

The `BlockConfirmations` type definition confirms the inner `u64` is fully public with no invariant: [4](#0-3) 

This is the direct analog of `amountOutMinimum: 0` in the external report: a protection parameter that should be bounded below is left entirely at the caller's discretion, with no floor enforced by the protocol.

### Impact Explanation

Any NEAR account can call `verify_foreign_transaction` with `confirmations: 0` for a Bitcoin transaction that has as few as 1 confirmation (or even 0, if the RPC node has it in its mempool and the response deserializes). The MPC network will produce a valid threshold signature over the payload `(tx_id, block_hash, ...)`. A bridge contract that trusts this signature to release funds on another chain will do so before the Bitcoin transaction is safely final. If the Bitcoin block is subsequently reorged, the attacker retains both the released bridge funds and the re-spent Bitcoin — a classic double-spend.

The impact maps to: **High — forged foreign-chain verification / light-client-style verification bypass causing double-spend conditions**.

### Likelihood Explanation

- Entry path is fully unprivileged: any NEAR account with 1 yoctoNEAR deposit can submit the request.
- Bitcoin 1-block reorgs occur in practice (roughly once every few weeks on mainnet); shallow reorgs are the standard attack surface for confirmation-bypass exploits.
- The attacker controls the `confirmations` field directly in the JSON request; no special tooling is required.
- Likelihood: **Medium** (requires a reorg, but the attacker can time the submission to a block they are attempting to reorg, or simply race against natural reorgs).

### Recommendation

1. **Enforce a protocol-level minimum in the contract.** Reject any `BitcoinRpcRequest` whose `confirmations` field is below a safe floor (e.g., 1, or a configurable per-chain minimum stored in `ForeignChainsMetadata`).

```rust
// In verify_foreign_transaction, before enqueue_yield_request:
if let ForeignChainRpcRequest::Bitcoin(ref btc) = request.request {
    if btc.confirmations.0 < MIN_BITCOIN_CONFIRMATIONS {
        env::panic_str("confirmations below minimum");
    }
}
```

2. **Enforce a minimum in the inspector as a defence-in-depth.** Even if the contract validates, the inspector should independently reject a threshold of 0.

3. **Consider making `BlockConfirmations` a newtype with a validated constructor** that panics or returns `Err` on 0, mirroring how `Threshold` enforces `MIN_THRESHOLD_ABSOLUTE = 2`. [5](#0-4) 

### Proof of Concept

1. Attacker broadcasts a Bitcoin transaction `T` spending output `O`.
2. Attacker calls `verify_foreign_transaction` on the NEAR MPC contract with:
   ```json
   {
     "request": {
       "Bitcoin": {
         "tx_id": "<hash of T>",
         "confirmations": 0,
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": <foreign_tx_domain>,
     "payload_version": 1
   }
   ```
   with 1 yoctoNEAR attached.
3. MPC nodes pick up the request. Each node calls `BitcoinInspector::extract(tx_id, BlockConfirmations(0), ...)`. The check `0 <= actual_confirmations` passes unconditionally.
4. Nodes sign the payload `(tx_id, block_hash)` and submit `respond_verify_foreign_tx`. The contract resolves the yield and returns the signature to the attacker.
5. Attacker presents the signature to a bridge contract to claim funds on NEAR (or another chain).
6. Attacker simultaneously mines a competing chain that omits `T`, reorging the original block.
7. Attacker re-spends output `O` on Bitcoin. Double-spend complete; bridge funds already released.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
