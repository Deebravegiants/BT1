Now I have enough information to make a determination. Let me verify one more critical detail — whether signature verification is actually called anywhere in the p2p sync header acceptance path.

### Title
Unauthenticated P2P Peer Can Write Arbitrary State Diff to Storage via Inflated `state_diff_length` in `SignedBlockHeader` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` from any p2p peer without cryptographically verifying the block signature. The peer-supplied `state_diff_length` field is stored verbatim. `StateDiffStreamBuilder::parse_data_for_block` then uses that stored value as `target_state_diff_len` to drive a loop that accumulates `StateDiffChunk` responses. No verification of the assembled `ThinStateDiff` against the stored `state_diff_commitment` is ever performed. An attacker who controls a p2p peer can therefore write an arbitrary `ThinStateDiff` — with attacker-chosen storage diffs, nonces, deployed contracts, and class hashes — into the node's persistent storage for any block number.

---

### Finding Description

**Step 1 — Header accepted without signature verification.**

`HeaderStreamBuilder::parse_data_for_block` performs only two checks on the incoming `SignedBlockHeader`:

- `block_number` matches the expected sequence number.
- `signatures.len() == ALLOWED_SIGNATURES_LENGTH` (exactly 1 element present). [1](#0-0) 

`verify_block_signature` (which hashes `block_hash || state_diff_commitment` and checks the ECDSA signature against the sequencer's known public key) is **never called** in this path. A grep across the entire `apollo_p2p_sync` crate confirms zero references to `verify_block_signature`. [2](#0-1) 

The header — including the attacker-chosen `state_diff_length` and `state_diff_commitment` — is written to storage unconditionally: [3](#0-2) 

**Step 2 — `state_diff_length` drives the chunk-accumulation loop.**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` directly from the stored header: [4](#0-3) 

It then loops, pulling `StateDiffChunk` messages from the peer until `current_state_diff_len >= target_state_diff_len`: [5](#0-4) 

If the attacker sends exactly `N` units of chunks (matching the inflated `state_diff_length = N`), the final equality check at line 99 passes, no `BadPeerError` is raised, and `Ok(Some((result, block_number)))` is returned. [6](#0-5) 

**Step 3 — No state diff commitment verification.**

`calculate_state_diff_hash` is never called anywhere in `apollo_p2p_sync`. The `state_diff_commitment` stored in the header (also peer-supplied) is never compared against the Poseidon hash of the assembled `ThinStateDiff`. The function exists: [7](#0-6) 

but has zero references in the sync client.

**Step 4 — Corrupted diff written to all state tables.**

`write_to_storage` calls `append_state_diff`, which writes the attacker-controlled `ThinStateDiff` into the deployed-contracts table, contract-storage table, nonces table, and compiled-class-hash table: [8](#0-7) 

When `parse_data_for_block` returns `Ok(Some(...))`, the stream builder yields the output and calls `write_to_storage` — no peer report is filed, no retry occurs: [9](#0-8) 

---

### Impact Explanation

After the write, every RPC call that reads state for the affected block number returns attacker-controlled data:

- `starknet_getStateUpdate` — returns the fabricated `ThinStateDiff`.
- `starknet_getStorageAt` / `starknet_getNonce` — return attacker-written values because `append_state_diff` updates the live storage and nonce tables.
- Fee estimation, simulation, and tracing that read contract storage or nonces for those contracts will produce wrong results.

This satisfies: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

Any node that participates in p2p sync and connects to an attacker-controlled peer is vulnerable. The attacker needs only to:

1. Operate a reachable p2p node.
2. Serve a `SignedBlockHeader` with an arbitrary `state_diff_length` (the signature bytes can be any 64-byte value — only their presence is checked).
3. Serve exactly `state_diff_length` units of malicious `StateDiffChunk` messages.

No privileged access, no validator key, no sequencer operator role is required.

---

### Recommendation

1. **Verify the block signature** in `HeaderStreamBuilder::parse_data_for_block` by calling `verify_block_signature` with the sequencer's known public key before storing the header. Reject headers whose signature does not verify.

2. **Verify the state diff commitment** in `StateDiffStreamBuilder::parse_data_for_block` after assembling the full `ThinStateDiff`: call `calculate_state_diff_hash(&result)` and compare against `header.state_diff_commitment`. Treat a mismatch as `BadPeerError`.

---

### Proof of Concept

```
1. Attacker peer sends SignedBlockHeader {
       block_number: N,
       state_diff_length: 1000,          // inflated
       state_diff_commitment: <anything>,
       signatures: [<any 64-byte value>], // only length is checked
       ...
   }

2. Victim stores header; target_state_diff_len = 1000.

3. Attacker sends 1000 StateDiffChunk::ContractDiff messages,
   each with len() == 1 and attacker-chosen storage_diffs.

4. current_state_diff_len reaches 1000 == target_state_diff_len.
   No error. Ok(Some((result, N))) returned.

5. write_to_storage writes attacker's ThinStateDiff for block N.

6. starknet_getStateUpdate(block_id=N) returns attacker's diff.
   starknet_getStorageAt for affected contracts returns attacker's values.
```
