Based on my analysis of the full vote handling pipeline, this is a real vulnerability. Here is the complete finding:

---

### Title
Unauthenticated Vote Impersonation Allows Fabricated Consensus Quorum — (`crates/apollo_consensus/src/single_height_consensus.rs`)

### Summary
Any peer with a valid libp2p identity can publish `Vote` messages claiming to be any validator (`vote.voter = <validator ContractAddress>`). Because the Starknet-level vote signature is never verified and the libp2p `originated_peer_id` is never bound to `vote.voter`, the consensus layer accepts and counts these forged votes toward quorum.

### Finding Description

The attack path has three concrete steps:

**Step 1 — libp2p signing does not authenticate the Starknet voter identity.**
`gossipsub_impl.rs` configures `MessageAuthenticity::Signed(keypair)`, which proves only that the gossipsub message was signed by the sender's libp2p keypair. It says nothing about which Starknet `ContractAddress` the sender is entitled to vote as. [1](#0-0) 

**Step 2 — `originated_peer_id` is extracted but never forwarded to vote validation.**
In `manager.rs::handle_vote`, the `BroadcastedMessageMetadata` (which carries `originated_peer_id`) is used only for `continue_propagation` and `report_peer`. At line 996, only the bare `Vote` struct is passed to `shc.handle_vote(message)` — the peer identity is silently dropped. [2](#0-1) 

**Step 3 — `single_height_consensus::handle_vote` has no signature verification and no peer-to-voter binding.**
The only checks are: height matches, and `vote.voter` is a member of the committee. The `// TODO(Asmaa): verify the signature` comment explicitly confirms that the `Vote.signature` field is never verified. There is no check that the peer who sent the message is actually authorized to cast a vote on behalf of `vote.voter`. [3](#0-2) 

The `Vote` struct carries a `signature: RawSignature` field that is always `RawSignature::default()` in practice and is never validated anywhere in the consensus pipeline. [4](#0-3) 

### Impact Explanation

An unprivileged peer (any node that can connect to the gossipsub mesh) can:
1. Enumerate the current validator committee (public information).
2. Craft `Vote` messages with `voter` set to each validator's `ContractAddress`.
3. Broadcast these votes over gossipsub (libp2p signing is satisfied by the attacker's own keypair).
4. The receiving node accepts each vote, inserts it into the prevote/precommit maps keyed by `(round, voter)`, and counts the attacker-controlled weight toward quorum. [5](#0-4) 

With enough forged votes (≥ 2/3 weight), the attacker can force a `DecisionReached` event for any `ProposalCommitment` the attacker chooses, breaking Byzantine fault-tolerance guarantees and potentially causing nodes to commit a block that the honest validator set never agreed to.

### Likelihood Explanation

- **Attacker requirement**: only a valid libp2p identity — no validator key, no operator access.
- **No rate-limiting or rejection** of well-formed votes from non-validator peers exists; the only guard is the committee membership check on `vote.voter`, which the attacker satisfies by using a real validator's address.
- The missing signature verification is explicitly acknowledged as a TODO, confirming the gap is known but unimplemented. [6](#0-5) 

### Recommendation

1. **Implement `Vote.signature` verification** in `single_height_consensus::handle_vote` before the committee membership check. The signature must be over the canonical vote fields (`vote_type`, `height`, `round`, `proposal_commitment`) and must verify against the public key associated with `vote.voter` in the staking registry.
2. **Bind `originated_peer_id` to `vote.voter`**: maintain a mapping from libp2p `PeerId` to validator `ContractAddress` (derived from the validator's registered public key) and reject any vote where `originated_peer_id` does not correspond to `vote.voter`.
3. Pass `BroadcastedMessageMetadata` (and thus `originated_peer_id`) all the way into `SingleHeightConsensus::handle_vote` so the binding check can be enforced at the point of acceptance. [7](#0-6)
