### Title
Missing `whenNotPaused` on `swapAssetToPremintedRsETH` Allows Pause Bypass by Whitelisted Users - (File: contracts/pools/RSETHPoolV2ExternalBridge.sol)

### Summary
`RSETHPoolV2ExternalBridge.swapAssetToPremintedRsETH` lacks the `whenNotPaused` modifier, while the corresponding `deposit` function is correctly gated. Any address holding `WHITELISTED_USER_ROLE` can continue to extract ETH from the pool even after the contract has been paused in an emergency, bypassing the protocol's only emergency-stop mechanism for this asset flow.

### Finding Description
`RSETHPoolV2ExternalBridge` implements a custom boolean pause flag and a `whenNotPaused` modifier. The user-facing `deposit` function correctly applies this modifier. However, `swapAssetToPremintedRsETH` — which transfers ETH out of the pool in exchange for rsETH — does not apply `whenNotPaused`:

```solidity
// deposit — correctly gated
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value)
{ ... }

// swapAssetToPremintedRsETH — pause NOT enforced
function swapAssetToPremintedRsETH(address rsETH, uint256 rsETHAmount)
    external
    nonReentrant
    onlyOperatorOrWhitelisted(msg.sender)   // <-- no whenNotPaused
{ ... }
```

The `onlyOperatorOrWhitelisted` modifier grants access to any account holding either `OPERATOR_ROLE` **or** `WHITELISTED_USER_ROLE`. The `WHITELISTED_USER_ROLE` is a user-level role that can be granted to external depositors, not just internal protocol operators. When the contract is paused — the intended emergency stop — these accounts retain full ability to call `swapAssetToPremintedRsETH` and drain the pool's ETH balance.

The root cause is the omission of `whenNotPaused` from `swapAssetToPremintedRsETH` at line 418, while `deposit` at line 289 correctly includes it. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
When the `PAUSER_ROLE` triggers a pause (e.g., due to a detected oracle anomaly, accounting discrepancy, or active exploit), the pause is supposed to halt all asset flows. Because `swapAssetToPremintedRsETH` is not gated, a whitelisted user can continue to call it, transferring ETH out of the pool at the rate returned by `rsETHOracle.getRate()`. If the pause was triggered precisely because the oracle is returning a manipulated rate, the whitelisted user can exploit that rate to extract more ETH than the rsETH they provide is worth — constituting direct theft of user funds held in the pool. Even absent oracle manipulation, draining the pool's ETH during a pause leaves other users unable to complete legitimate operations, causing temporary freezing of funds. [4](#0-3) 

### Likelihood Explanation
The `WHITELISTED_USER_ROLE` is a role that can be granted to external users beyond the core operator set. A pause event is a realistic scenario (the role exists precisely for emergencies). Any whitelisted account that is online at the time of a pause can immediately call `swapAssetToPremintedRsETH` before the pause takes effect on their path, or continuously after the pause is set. No additional privilege escalation is required beyond holding the whitelisted role. [5](#0-4) [6](#0-5) 

### Recommendation
Add the `whenNotPaused` modifier to `swapAssetToPremintedRsETH` in `RSETHPoolV2ExternalBridge`, consistent with how `deposit` is protected:

```solidity
function swapAssetToPremintedRsETH(address rsETH, uint256 rsETHAmount)
    external
    nonReentrant
    whenNotPaused                          // add this
    onlyOperatorOrWhitelisted(msg.sender)
{ ... }
```

The same gap exists in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge` where `swapAssetToPremintedRsETH` is restricted to `OPERATOR_ROLE` only; those variants should also receive `whenNotPaused` for defence-in-depth. [7](#0-6) [8](#0-7) 

### Proof of Concept
1. Admin grants `WHITELISTED_USER_ROLE` to address `W`.
2. An oracle anomaly is detected; `PAUSER_ROLE` calls `pause()` — `paused` is set to `true`.
3. `W` calls `swapAssetToPremintedRsETH(rsETH, largeAmount)`. The `whenNotPaused` check is absent, so execution proceeds.
4. `IERC20(rsETH).safeTransferFrom(W, wrapper, largeAmount)` succeeds.
5. `getETHBalanceMinusFees()` returns the pool's ETH balance; `viewSwapAssetToPremintedRsETH` computes `ethAmount` using the (potentially manipulated) oracle rate.
6. ETH is transferred to `W` via `payable(msg.sender).call{value: ethAmount}("")`.
7. The pool's ETH is drained while the contract is nominally paused, leaving other users unable to receive ETH from the pool. [9](#0-8)

        address rsETH,
        uint256 rsETHAmount
    )
        external
        nonReentrant
        onlyOperatorOrWhitelisted(msg.sender)
    {
        UtilLib.checkNonZeroAddress(rsETH);

        IRsETHTokenWrapper wrapper = IRsETHTokenWrapper(address(wrsETH));

        if (!wrapper.allowedTokens(rsETH)) revert TokenNotAllowedInWrapper();
        if (rsETHAmount == 0) revert InvalidAmount();
        if (rsETHAmount > wrapper.maxAmountToDepositBridgerAsset(rsETH)) revert ExceedsMaxAmountToDepositInWrapper();

        // Get the amount of ETH to transfer to the user for the given amount of rsETH provided
        uint256 ethAmount = viewSwapAssetToPremintedRsETH(rsETHAmount);

        // Transfer rsETH from sender to the wrapper
        IERC20(rsETH).safeTransferFrom(msg.sender, address(wrapper), rsETHAmount);

        // Transfer the ETH from the pool to the sender
        if (getETHBalanceMinusFees() < ethAmount) revert InsufficientETHBalanceForReverseSwap();
        (bool success,) = payable(msg.sender).call{ value: ethAmount }("");
        if (!success) revert TransferFailed();

        emit ReverseSwapOccurred(msg.sender, rsETH, rsETHAmount, ethAmount);
    }
