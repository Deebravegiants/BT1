Audit Report

## Title
Signature Replay in `allowPushers` Lets Creator Silently Re-Establish Delegation After Pusher Revokes — (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

## Summary

`CompressedOracle.allowPushers` validates a pusher's EIP-191 consent signature but never marks it as consumed. Because `revokePusher` only zeroes `namespaceRemapping[msg.sender]` without invalidating outstanding signatures, a malicious creator can immediately replay the same `(deadline, pusher, signature)` tuple to re-establish delegation after the pusher revokes, silently redirecting the pusher's future oracle updates into the creator's namespace and corrupting the bid/ask prices consumed by pools.

## Finding Description

`allowPushers` constructs the signed digest as:

```solidity
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

There is no nonce, no per-`(pusher, creator)` counter, and no used-signature bitmap. The only freshness gate is `_ensureDeadline(deadline)`, which rejects calls made *after* the deadline but does nothing to prevent the same signature from being submitted multiple times *before* the deadline.

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

But it has no effect on the already-issued signature. The creator immediately calls `allowPushers` again with the identical tuple, and the mapping is restored to `msg.sender`. The pusher's revocation is silently undone.

The NatSpec comment acknowledges the replay risk but incorrectly claims the deadline fully mitigates it — the deadline only closes the window after expiry; within the window the replay is unrestricted. [3](#0-2) 

After re-establishing delegation, the pusher's subsequent `fallback()` calls resolve the namespace to the creator:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

All slot writes from the pusher land in the creator's namespace, not the pusher's own.

## Impact Explanation

`CompressedOracleV1` is the price-delivery layer consumed by `IPriceProvider → MetricOmmPool._getBidAndAskPriceX64`. Every swap executes against the bid/ask returned by this oracle. If a creator re-establishes delegation after the pusher revoked, the pusher's subsequent pushes land in the creator's namespace, updating the creator's oracle feed with data the pusher no longer intends to provide to that creator. Pools bound to the creator's price provider receive corrupted bid/ask prices, causing swaps to execute at wrong prices — satisfying the **bad-price execution** impact gate and potentially **pool insolvency**.

## Likelihood Explanation

- The pusher must have signed a consent with a non-trivial deadline (e.g., 1 day, 7 days — common practice).
- The creator must be malicious or compromised.
- No special privilege beyond being the original `msg.sender` creator is required; the replay call is a plain external transaction using a publicly visible signature.
- The pusher has no on-chain way to detect or prevent the replay short of waiting for the deadline to expire.

Likelihood: **Medium** (requires a malicious creator, but the attack is trivially executable with the original public signature).

## Recommendation

Add a per-`(pusher, creator)` nonce to the signed digest and increment it on every successful `allowPushers` call:

```solidity
mapping(address pusher => mapping(address creator => uint256)) public nonces;

bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid,
        address(this),
        deadline,
        pusher,
        msg.sender,
        nonces[pusher][msg.sender]++
    ))
);
```

This ensures each consent signature is single-use. After `revokePusher`, the nonce is already incremented, so the old signature is permanently invalid even if the deadline has not expired.

## Proof of Concept

```solidity
// 1. Pusher signs consent once with a 1-day deadline.
// 2. Creator calls allowPushers — delegation established.
// 3. Pusher calls revokePusher — namespaceRemapping[pusher] = address(0).
// 4. Creator calls allowPushers again with the SAME (deadline, pusher, sig) — no revert,
//    namespaceRemapping[pusher] = creator again.
// 5. Pusher's next fallback() push lands in creator's namespace.
// 6. oracle.getOracleData(feedIdOf(creator, slotId, posId)).price > 0
//    — creator feed poisoned; pool consumes corrupted bid/ask.
```

The exact Foundry test provided in the submission correctly demonstrates steps 1–6 and will pass against the current production code, confirming the replay is unrestricted within the deadline window.
