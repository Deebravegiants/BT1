### Title
Irreversible Fund Lock via Zero-Normalized Net Amount in `init_transfer` — (`near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` on the NEAR bridge accepts and locks user tokens without verifying that the net transfer amount (`amount - fee`) survives decimal normalization to the destination chain. When `normalize_amount(amount - fee, decimals) == 0`, the mandatory follow-up call `sign_transfer` permanently reverts with `ERR_INVALID_AMOUNT_TO_TRANSFER`, and no cancel or refund path exists. The user's tokens are irreversibly locked.

### Finding Description

The NEAR→EVM transfer flow is a two-step state machine:

**Step 1 — `init_transfer`** (via `ft_transfer_call`): tokens are locked in the bridge and a `TransferMessage` is stored in `pending_transfers`.

The only fee guard at this step is:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

There is no check that `normalize_amount(amount - fee, decimals) > 0`.

**Step 2 — `sign_transfer`**: the relayer calls this to obtain the MPC signature needed to release tokens on the destination chain. It computes:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

`normalize_amount` performs floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [3](#0-2) 

If `amount - fee < 10^(origin_decimals - decimals)`, the result is `0`, and `sign_transfer` reverts every time it is called for that transfer.

`amount_without_fee` uses `checked_sub`, so it returns `None` only when `fee > amount`; the `fee < amount` invariant from step 1 guarantees it returns `Some`, but the value can still be below the normalization threshold. [4](#0-3) 

**No recovery path exists.** `remove_transfer_message` is only called in `sign_transfer_callback` (when `fee.is_zero()`, which requires `sign_transfer` to succeed first) and in `claim_fee_callback` (which requires a proof of finalization on the destination chain — impossible if the transfer was never signed). [5](#0-4) 

`update_transfer_fee` cannot rescue the transfer: it can only increase the fee up to `amount - 1`, but if `1 < 10^diff`, the normalized net amount is still `0`. [6](#0-5) 

### Impact Explanation

**Critical — Irreversible fund lock.** Any user who initiates a NEAR→EVM transfer where `amount - fee < 10^(origin_decimals - decimals)` will have their tokens permanently locked in the bridge with no redemption path. For a NEAR-native token (24 decimals) bridging to an EVM chain (18 decimals), the threshold is `10^6` base units (~0.000001 tokens). For tokens with a larger decimal gap (e.g., 24 vs 6), the threshold is `10^18` base units.

### Likelihood Explanation

**Medium.** The entry point is fully public and unprivileged — any user calling `ft_transfer_call` with a small amount and any valid fee triggers the condition. The protocol silently accepts the transfer at step 1 without warning. Users bridging dust amounts or setting fees close to the transfer amount are at risk. The condition is easy to hit accidentally for tokens with large decimal differences.

### Recommendation

Add a normalization check inside `init_transfer_internal` (or at the end of `init_transfer`) before storing the transfer message:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let net = transfer_message
            .amount_without_fee()
            .near_expect(BridgeError::InvalidFee);
        require!(
            Self::normalize_amount(net, decimals) > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );
    }
}
```

Alternatively, add a `cancel_transfer` function that allows the original sender to reclaim locked tokens for transfers that have never been signed, providing a recovery path even if the upfront check is not added.

### Proof of Concept

Assume a NEAR-native token with `origin_decimals = 24`, bridging to an EVM chain where `decimals = 18` (diff = 6, divisor = 1,000,000).

1. User calls `ft_transfer_call` with `amount = 1_000_000` and `fee = 1`.
   - `init_transfer` check: `1 < 1_000_000` ✓ — transfer is accepted, tokens locked.
2. Relayer calls `sign_transfer` for this transfer.
   - `amount_without_fee()` = `1_000_000 - 1` = `999_999`
   - `normalize_amount(999_999, {origin: 24, dest: 18})` = `999_999 / 1_000_000` = `0`
   - `require!(0 > 0, ...)` → **panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`**
3. Every subsequent call to `sign_transfer` for this transfer ID reverts identically.
4. The transfer message remains in `pending_transfers` forever; the user's `1_000_000` base units are permanently locked. [2](#0-1) [1](#0-0) [3](#0-2)
