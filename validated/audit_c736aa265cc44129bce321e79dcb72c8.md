### Title
Attacker-Controlled `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Session Signature Replay to Forge Foreign-Chain Verification Results - ([File: crates/contract/src/lib.rs])

### Summary

`respond_verify_foreign_tx` verifies only that the submitted `response.payload_hash` is covered by a valid ECDSA signature under the domain's root public key. It does **not** verify that `response.payload_hash` is the canonical hash of `ForeignTxSignPayload{request: request.request, values: ...}` for the pending request being resolved. A single Byzantine attested participant can replay a `(payload_hash, signature)` pair obtained from any prior legitimate signing session to resolve an unrelated pending `verify_foreign_transaction` request with a fabricated payload hash, delivering a forged verification attestation to the waiting bridge contract.

### Finding Description

`respond_verify_foreign_tx` performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract accepts any `(payload_hash, signature)` pair where the signature is valid over `payload_hash` under the root key, regardless of whether `payload_hash` is the hash of the correct `ForeignTxSignPayload` for the `request` being resolved. The `request` parameter is used only as a map key to locate and drain the pending yield queue:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The node-side signing path confirms that foreign-tx signing uses a zero tweak (`Tweak::new([0u8; 32])`), meaning the root key signs the payload hash directly:

```rust
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),
    ...
})
``` [3](#0-2) 

This means any `(payload_hash, signature)` pair produced by the threshold network for **any** prior foreign-tx request is a valid credential that passes the contract's signature check.

By contrast, `respond` (for regular `sign` requests) derives the expected public key from the request's tweak before verifying, binding the signature to the specific request's key derivation path: [4](#0-3) 

No equivalent binding exists in `respond_verify_foreign_tx`.

### Impact Explanation

A single malicious attested participant that has participated in any prior legitimate `verify_foreign_transaction` signing session possesses a valid `(payload_hash_old, sig_old)` pair. It can call:

```
respond_verify_foreign_tx(
  request = R_victim,          // a currently-pending request
  response = {
    payload_hash: payload_hash_old,   // hash from a different, past session
    signature:    sig_old,            // valid signature over payload_hash_old
  }
)
```

The contract verifies the signature, finds it valid, and resolves all yields queued under `R_victim` with the fabricated response. The bridge contract waiting on `R_victim` receives `{payload_hash: payload_hash_old, signature: sig_old}`.

- **If the bridge contract uses `ForeignChainSignatureVerifier::verify_signature`** (from `crates/near-mpc-sdk`), it will detect the mismatch between `payload_hash_old` and the expected hash for `R_victim` and reject the response. The victim's call fails and the request slot is permanently consumed, forcing a resubmission and wasting a presignature.
- **If the bridge contract trusts the contract-returned `payload_hash` without re-deriving it** (a realistic omission for contracts not using the SDK), it accepts a forged attestation — the MPC network appears to have verified foreign-chain data it never actually observed. This enables invalid bridge execution or double-spend conditions. [5](#0-4) 

### Likelihood Explanation

- The attacker is a single Byzantine attested participant — strictly below the signing threshold.
- No threshold collusion is required for the malicious action; the attacker only needs to have participated in **one** prior legitimate signing session (normal operation) to obtain a replayable credential.
- `respond_verify_foreign_tx` is callable by any attested participant, not only the designated leader.
- The `VerifyForeignTransactionRequest` map key does not include the caller's account ID, so the attacker can target any pending request from any user. [6](#0-5) 

### Recommendation

Require the responding node to submit the full `ForeignTxSignPayload` (including the extracted `values`) alongside the signature, and have the contract recompute `expected_hash = SHA-256(borsh(ForeignTxSignPayload{request: request.request, values: submitted_values}))` and assert `expected_hash == response.payload_hash` before accepting the response. This binds the attested `payload_hash` to the specific pending request and its observed foreign-chain data, eliminating the replay surface.

2. Victim submits `verify_foreign_transaction` for request `R2` (Bitcoin tx `T2`). The contract stores a pending yield under key `R2`.

3. Attacker calls:
