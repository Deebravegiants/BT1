### Title
Single Byzantine Participant Can Block All Foreign-Chain Verification Requests via Strict Intersection in `get_supported_foreign_chains` - (File: crates/contract/src/lib.rs)

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which requires **every** active participant to have registered a chain. A single Byzantine participant (strictly below the signing threshold) can call `register_foreign_chain_support` with an empty set, collapsing the intersection to empty and causing every `verify_foreign_transaction` call to revert with `ForeignChainNotSupported`. This is the direct analog of the ERC20/ERC777 `preventLocking = true` bug: a stricter-than-necessary check is applied to a code path that should use a threshold-based (permissive) check.

### Finding Description

`verify_foreign_transaction` at `crates/contract/src/lib.rs:533-542` calls `get_supported_foreign_chains()` to decide which chains are serviceable:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported { requested: requested_chain }.to_string(),
    );
}
```

`get_supported_foreign_chains()` at `crates/contract/src/lib.rs:2203-2217` computes the **strict intersection** — a chain is included only if **all** active participants have registered it:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
if all_active_nodes_supports_chain { Some(foreign_chain) } else { None }
```

`register_foreign_chain_support` at `crates/contract/src/lib.rs:972-983` accepts any `SupportedForeignChains` value from any authenticated participant, including an empty set:

```rust
pub fn register_foreign_chain_support(
    &mut self,
    foreign_chain_support: dtos::SupportedForeignChains,
) -> Result<(), Error> {
    let account_id = self.voter_or_panic();
    self.node_foreign_chain_support
        .foreign_chain_support_by_node
        .insert(account_id, foreign_chain_support);
    Ok(())
}
```

A threshold-based alternative already exists: `get_available_foreign_chains()` (backed by `ForeignChainsMetadata::update_available_chains_config_cache` at `crates/contract/src/foreign_chains_metadata.rs:41-66`) requires only `≥ signing_threshold` participants to cover a chain. However, `verify_foreign_transaction` does not use it.

The project's own design document (`docs/design/calculating-supported-foreign-chains.md`) explicitly acknowledges this root cause:

> "Today, `get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains, and `verify_foreign_transaction` rejects any request whose target chain is not in it. A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down."

### Impact Explanation

A single Byzantine participant (below the signing threshold) can call `register_foreign_chain_support` with an empty `SupportedForeignChains`. Because `get_supported_foreign_chains()` requires unanimity, the intersection immediately becomes empty. Every subsequent `verify_foreign_transaction` call — regardless of chain — is rejected with `ForeignChainNotSupported`. This permanently freezes the bridge's inbound flow (Omnibridge foreign-chain → NEAR) until the Byzantine participant re-registers a non-empty set or is removed via resharing. Funds in transit that depend on `verify_foreign_transaction` completing cannot be claimed.

This matches the **Medium** allowed impact: request-lifecycle manipulation that breaks production safety/accounting invariants without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation

Any single authenticated participant can execute this attack with one on-chain transaction. No collusion, no key material, and no privileged access beyond participant status is required. The attack is reversible only by the attacker re-registering or by a resharing that removes them — both of which require coordination and time, during which the bridge is frozen.

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which already implements the correct threshold-based check (`≥ signing_threshold` participants must cover a chain). This mirrors the ERC20/ERC777 fix: use the permissive (threshold) check for the user-facing request path, not the strict (unanimity) check.

```rust
// In verify_foreign_transaction (crates/contract/src/lib.rs ~line 533):
// Before:
let supported_chains = self.get_supported_foreign_chains();
// After:
let supported_chains = self.get_available_foreign_chains();
```

### Proof of Concept

1. Network has `n` participants with signing threshold `t < n`. All participants have registered Bitcoin as a supported chain. `verify_foreign_transaction` for Bitcoin succeeds.
2. Byzantine participant `P` (one of the `n`) calls:
   ```
   register_foreign_chain_support({ foreign_chain_support: [] })
   ```
3. `get_supported_foreign_chains()` now computes: `nodes_supporting_bitcoin = {all except P}`, which is **not** a superset of `active_participant_account_ids` (which includes `P`). Bitcoin is dropped from the result.
4. Any user calling `verify_foreign_transaction` for Bitcoin now receives `ForeignChainNotSupported`, even though `t` honest participants are fully capable of verifying and signing the request.
5. The attack requires exactly one transaction from one participant. The `get_available_foreign_chains()` path (used by the new design) would correctly report Bitcoin as available since `n-1 ≥ t` participants still cover it. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```rust
    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
```
