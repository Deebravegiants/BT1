Audit Report

## Title
`LRTDepositPool._beforeDeposit` Allows Zero rsETH Minting — Depositor Loses Assets Without Receiving Shares - (File: contracts/LRTDepositPool.sol)

## Summary
`getRsETHAmountToMint` uses integer division that truncates to zero for dust deposits when `amount * assetPrice < rsETHPrice`. `_beforeDeposit` only guards against `rsethAmountToMint < minRSETHAmountExpected`, so when a caller passes `minRSETHAmountExpected = 0`, the guard is trivially satisfied even when `rsethAmountToMint = 0`. The depositor's ETH or LST is absorbed into the protocol TVL while zero rsETH is minted in return.

## Finding Description
`getRsETHAmountToMint` computes:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
This is integer division. For any deposit where `amount * assetPrice < rsETHPrice`, the result is 0.

`_beforeDeposit` then checks:
```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```
There is no `require(rsethAmountToMint > 0)`. With `minRSETHAmountExpected = 0`, the condition `0 < 0` is false and execution proceeds. For `depositETH`, ETH is already in the contract via `msg.value`; for `depositAsset`, the LST is pulled via `safeTransferFrom`. In both cases `_mintRsETH(0)` is called, minting nothing to the depositor. `minAmountToDeposit` is initialized to 0 and never set in `initialize()`, so any non-zero dust deposit passes the `depositAmount < minAmountToDeposit` check.

## Impact Explanation
**Low — Contract fails to deliver promised returns.**
A depositor who calls `depositETH` or `depositAsset` with `minRSETHAmountExpected = 0` and a deposit amount below the integer-division threshold permanently loses their deposited assets. The ETH or LST is absorbed into the protocol TVL, increasing the rsETH share price for all existing holders, while the depositor receives zero rsETH with no recovery path. In the default protocol state (rsETHPrice = 1e18, assetPrice = 1e18), only 1-wei deposits are affected; as rsETH appreciates, the threshold rises, but meaningful amounts still require price manipulation.

## Likelihood Explanation
Any unprivileged depositor can trigger this in normal protocol operation by passing `minRSETHAmountExpected = 0` (the default for integrating contracts that omit slippage) with a dust deposit. No privileged role, front-running, or external compromise is required. The condition becomes more reachable over time as rsETHPrice grows above 1e18. Likelihood is Low because only dust amounts are affected without price manipulation.

## Recommendation
Add an explicit zero-share guard immediately after computing `rsethAmountToMint` in `_beforeDeposit`:
```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint == 0) {
    revert ZeroRsETHMinted();
}

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```
This ensures no depositor can lose assets while receiving zero shares, regardless of the `minRSETHAmountExpected` argument.

## Proof of Concept
Assume:
- `rsETHPrice = 1.05e18` (rsETH has appreciated 5%)
- `assetPrice(ETH) = 1e18`
- `minAmountToDeposit = 0` (default, never initialized)

```
depositETH{value: 1}(minRSETHAmountExpected: 0, referralId: "")
```

1. `_beforeDeposit(ETH, 1, 0)` is called.
2. `depositAmount (1) == 0` → false; `1 < minAmountToDeposit (0)` → false → passes.
3. `getRsETHAmountToMint(ETH, 1)` → `(1 * 1e18) / 1.05e18 = 0` (integer truncation).
4. `rsethAmountToMint (0) < minRSETHAmountExpected (0)` → `false` → no revert.
5. `_mintRsETH(0)` → `IRSETH.mint(msg.sender, 0)` → depositor receives 0 rsETH.
6. The 1 wei ETH remains in the deposit pool, increasing TVL and benefiting all existing rsETH holders.
