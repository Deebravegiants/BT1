Audit Report

## Title
`_checkAndUpdateDailyFeeMintLimit` Reverts on Zero `maxFeeMintAmountPerDay`, Permanently Blocking `updateRSETHPrice` and Protocol Fee Collection - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle._checkAndUpdateDailyFeeMintLimit` lacks a guard for `maxFeeMintAmountPerDay == 0`, causing it to unconditionally revert with `DailyFeeMintLimitExceeded` whenever a non-zero fee amount is passed. Because `maxFeeMintAmountPerDay` defaults to `0` and is not set by `reinitialize()`, any deployment window where `protocolFeeInBPS > 0` but `setMaxFeeMintAmountPerDay` has not yet been called — or where a manager deliberately sets it to `0` — permanently blocks `updateRSETHPrice()`, freezing protocol fee collection and staling the rsETH price.

## Finding Description
`maxFeeMintAmountPerDay` is a storage variable that defaults to `0` after `initialize()` and is not set by `reinitialize()`. It requires an explicit call to `setMaxFeeMintAmountPerDay` (LRT manager only).

The internal write path `_checkAndUpdateDailyFeeMintLimit` has no zero-value guard:

```solidity
// contracts/LRTOracle.sol L197-210
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }
    // maxFeeMintAmountPerDay == 0, feeAmount > 0:
    // 0 + feeAmount > 0  →  always true  →  always reverts
    if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(...);
    }
    currentPeriodMintedFeeAmount += feeAmount;
}
```

Since `feePeriodStartTime` is also `0` by default, the period-reset branch always fires first (resetting `currentPeriodMintedFeeAmount` to `0`), so the check reduces to `feeAmount > 0` → revert on every call with a non-zero fee.

The call chain is: `updateRSETHPrice()` (public) → `_updateRsETHPrice()` → when `protocolFeeInBPS > 0` and TVL increases, `protocolFeeInETH > 0` → `rsethAmountToMintAsProtocolFee > 0` → `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` → revert.

The sibling view function `remainingDailyFeeMintLimit` short-circuits with `return 0` when `maxFeeMintAmountPerDay == 0`, but the write path has no equivalent guard.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

When triggered, every call to `updateRSETHPrice()` reverts. Protocol fees that should be minted to the treasury are never collected — the treasury's unclaimed yield is permanently frozen for the duration of the blockage. Additionally, `rsETHPrice` becomes stale: deposits via `LRTDepositPool.getRsETHAmountToMint` and withdrawals via `LRTWithdrawalManager.getExpectedAssetAmount` both read the stored `lrtOracle.rsETHPrice()`, so users transact at an outdated rate, potentially diluting existing rsETH holders if new deposits occur while the price is stale.

## Likelihood Explanation
The trigger requires two conditions to hold simultaneously: `maxFeeMintAmountPerDay == 0` (the default) and `protocolFeeInBPS > 0` (set separately via `LRTConfig.setProtocolFeeBps`). Any deployment window between enabling protocol fees and calling `setMaxFeeMintAmountPerDay`, or any manager action setting `maxFeeMintAmountPerDay` back to `0` intending "no limit," triggers the bug. No privilege is required to observe or trigger the revert — `updateRSETHPrice()` is callable by any EOA. TVL increasing is the normal operating state of the protocol.

## Recommendation
Mirror the guard already present in `remainingDailyFeeMintLimit`. Add an early return when `maxFeeMintAmountPerDay == 0`, treating zero as "no limit enforced":

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
+   if (maxFeeMintAmountPerDay == 0) return; // 0 means no limit; skip enforcement

    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }

    if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
    }

    currentPeriodMintedFeeAmount += feeAmount;
}
```

## Proof of Concept
1. Deploy `LRTOracle` and call `initialize(lrtConfigAddr)`. `maxFeeMintAmountPerDay` is `0`, `feePeriodStartTime` is `0`.
2. Call `LRTConfig.setProtocolFeeBps(500)` (5% fee). Do **not** call `setMaxFeeMintAmountPerDay`.
3. Allow TVL to increase (staking rewards accrue), so `totalETHInProtocol > previousTVL`.
4. Call `updateRSETHPrice()` from any EOA.
5. `protocolFeeInETH > 0` → `rsethAmountToMintAsProtocolFee > 0`.
6. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` is called; period reset fires (since `feePeriodStartTime == 0`), setting `currentPeriodMintedFeeAmount = 0`.
7. Check: `0 + rsethAmountToMintAsProtocolFee > 0` → `true` → revert `DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)`.
8. `rsETHPrice` is never updated; protocol fees are never minted to treasury.
9. All subsequent calls to `updateRSETHPrice()` revert identically until an admin calls `setMaxFeeMintAmountPerDay(nonZeroValue)`.
