### Title
Stale Cross-Chain Oracle Rate Allows Depositors to Drain wrsETH Reserves at Inflated Exchange Rate - (`contracts/pools/RSETHPool.sol`)

### Summary
`RSETHPool.deposit()` computes wrsETH output using `rsETHToETHrate` from `rsETHOracle.getRate()` with no staleness validation. The cross-chain oracle (`CrossChainRateReceiver`) stores `lastUpdated` but never enforces freshness in `getRate()`. When the L2 oracle rate lags behind the true L1 rate — a normal operational condition for cross-chain systems — any unprivileged depositor can receive more wrsETH than the ETH-equivalent value they contributed, directly draining the pool's wrsETH reserves.

### Finding Description

`RSETHPool` is deployed on Arbitrum. Its `rsETHOracle` is a `RSETHRateReceiver` (or `InterimRSETHOracle`), which receives the rsETH/ETH rate via LayerZero cross-chain messages from L1.

`CrossChainRateReceiver.lzReceive()` updates `rate` and `lastUpdated` when a message arrives: [1](#0-0) 

However, `getRate()` returns the stored rate unconditionally, with no check against `lastUpdated`: [2](#0-1) 

`RSETHPool.viewSwapRsETHAmountAndFee()` consumes this rate directly: [3](#0-2) 

And `deposit()` transfers wrsETH based on that computed amount: [4](#0-3) 

Since rsETH is a yield-bearing token, its rate vs ETH monotonically increases over time. A stale (lagging) rate is always lower than the true current rate. A lower `rsETHToETHrate` in the division `amountAfterFee * 1e18 / rsETHToETHrate` produces a larger `rsETHAmount`, giving the depositor more wrsETH than their ETH contribution is worth at the true rate.

The `InterimRSETHOracle` has the same issue — `getRate()` returns the stored `rate` with no timestamp validation: [5](#0-4) 

### Impact Explanation

An attacker who deposits 1 ETH when the oracle rate is 10% stale (e.g., oracle reports `1.05e18` while true rate is `1.155e18`) receives approximately `1e18 / 1.05e18 ≈ 0.952e18` wrsETH instead of the correct `≈ 0.866e18`. The excess `≈ 0.086e18` wrsETH is taken directly from the pool's wrsETH reserves. The attacker can immediately sell the wrsETH on secondary markets (Uniswap, etc.) at the true market rate, realizing a profit. This constitutes **direct theft of funds at-rest in the pool** (the wrsETH balance held by `RSETHPool`).

### Likelihood Explanation

Cross-chain oracle lag is a routine operational condition, not an attack or oracle operator compromise. LayerZero message delivery can be delayed by minutes to hours under normal network conditions. An attacker only needs to:
1. Monitor `CrossChainRateReceiver.lastUpdated` on-chain
2. Compare the stored `rate` against the live L1 rate (publicly readable)
3. Call `deposit()` during any staleness window

No special privileges, front-running, or external protocol compromise is required. The `lastUpdated` field is already stored on-chain, making detection trivial.

### Recommendation

1. **Add a staleness threshold in `CrossChainRateReceiver.getRate()`:**
   ```solidity
   uint256 public constant MAX_STALENESS = 24 hours;
   function getRate() external view returns (uint256) {
       require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate stale");
       return rate;
   }
   ```
2. **Add a minimum rate check in `viewSwapRsETHAmountAndFee`** to revert if the oracle rate deviates significantly from an expected floor.
3. Consider adding a `minRsETHRate` parameter to `deposit()` so depositors can specify slippage protection.

### Proof of Concept

```solidity
// Fork test on Arbitrum
function testStaleOracleExploit() public {
    // 1. Record current oracle rate (true rate = 1.155e18)
    uint256 trueRate = 1.155e18;
    
    // 2. Simulate oracle staleness: warp time forward without updating oracle
    //    Oracle still reports 1.05e18 (stale, ~10% below true rate)
    vm.mockCall(
        address(rsETHOracle),
        abi.encodeWithSelector(IOracle.getRate.selector),
        abi.encode(1.05e18)
    );
    
    // 3. Attacker deposits 1 ETH
    vm.deal(attacker, 1 ether);
    vm.prank(attacker);
    pool.deposit{value: 1 ether}("ref");
    
    // 4. Assert attacker received more wrsETH than fair value
    uint256 fairAmount = 1e18 * 1e18 / trueRate; // ~0.866e18
    uint256 received = wrsETH.balanceOf(attacker);
    assertGt(received, fairAmount); // received ~0.952e18 > fair ~0.866e18
    
    // 5. Excess wrsETH (~0.086e18) was taken from pool reserves
}
```
