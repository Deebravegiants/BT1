### Title
2P EdDSA `attach_private_scalar` accepts any (x, x·G) pair without binding x to the blob's stored public key — (`src/cbmpc/api/eddsa2pc.cpp`)

### Summary
`eddsa_2p::attach_private_scalar` only verifies that the caller-supplied scalar `x` satisfies `x*G == public_share_compressed`, where both inputs are attacker-controlled. Because the 2P blob stores no per-party `Qi` value, there is no stored anchor to compare against. Any valid (x, x·G) pair is accepted, allowing an attacker to inject an arbitrary scalar into a legitimate public blob.

### Finding Description
The 2P key blob (`key_blob_v1_t` in `eddsa2pc.cpp`) stores only the aggregate public key `Q_compressed`, `role`, `curve`, and `x_share`. It does not store the party's individual share public point `Qi`. [1](#0-0) 

The MP variant stores `Qis_compressed` (a `party_name → compressed Qi` map) and uses it in `attach_private_scalar` to enforce that the caller-supplied `public_share_compressed` matches the blob's own `Qi_self`: [2](#0-1) 

The 2P `attach_private_scalar` has no equivalent guard. It parses `public_share_compressed` as a curve point, then checks `x * G == Qi_self` — but since both `x` and `Qi_self` come from the caller, this check is trivially satisfied for any chosen scalar: [3](#0-2) 

The resulting blob stores `x_attacker` alongside `Q_original`. At sign time, `blob_to_key` validates only that `x_share` is in range and `Q_compressed` is a valid point — it never checks `x * G == Q` or `x * G == Qi`: [4](#0-3) 

The corrupted blob therefore passes all deserialization checks and is used for signing.

### Impact Explanation
An attacker who can call `attach_private_scalar` with an attacker-chosen scalar (e.g., a malicious process with API access, or a compromised backup-restore pipeline) can silently replace P1's legitimate share with an arbitrary scalar. The resulting blob is accepted by `sign` and `refresh`, but all signatures produced will be invalid against the original public key `Q`. This constitutes acceptance of an attacker-controlled scalar under the wrong key — fitting the **High** impact category ("attacker-controlled scalars accepted under the wrong key").

### Likelihood Explanation
The `detach_private_scalar` / `attach_private_scalar` API is explicitly designed for backup/restore workflows where the public blob is stored separately from the private scalar. Any component in that pipeline that can call `attach_private_scalar` with arbitrary inputs can trigger this. The API is reachable from both the C++ API (`eddsa_2p::attach_private_scalar`) and the C API (`cbmpc_eddsa_2p_attach_private_scalar`). [5](#0-4) 

### Recommendation
Store `Qi_self_compressed` (i.e., `x_share * G` at DKG/refresh time) in the 2P blob, mirroring the MP blob's `Qis_compressed` field. In `attach_private_scalar`, add the same guard the MP variant uses: reject the call if `public_share_compressed != stored_Qi_self_compressed`. Additionally, `blob_to_key` should verify `x_share * G == stored_Qi_self` to catch any in-blob inconsistency at deserialization time.
