### Title
Missing Token Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Severe rsETH Under-Minting for Non-18-Decimal Collateral Tokens - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

The `viewSwapRsETHAmountAndFee(uint256 amount, address token)` function across all L2 pool variants silently assumes the deposited token amount is expressed in 18 decimals. The oracle wrapper `ChainlinkOracleForRSETHPoolCollateral.getRate()` returns the price of **one whole token** in ETH (normalized to 1e18), but the formula multiplies this by the raw token-unit amount without normalizing for the token's actual decimal count. When a non-18-decimal token (e.g., USDC with 6 decimals) is added as a supported collateral, depositors receive a minted rsETH amount that is `10^(18 − tokenDecimals)` times too small, while their full token balance is irrecoverably transferred to the pool and bridged to L1.

---

### Finding Description

The minting formula in `viewSwapRsETHAmountAndFee`:

```solidity
// contracts/pools/RSETHPoolV3.sol L334
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

is dimensionally consistent only when `amountAfterFee` is in 18-decimal units.

`tokenToETHRate` is sourced from `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

This returns the price of **one whole token** in ETH, expressed in 1e18 precision. For a Chainlink USDC/ETH feed (feed decimals = 18), `normalizedPrice ≈ 4e14` (≈ 0.0004 ETH per 1 USDC).

For an 18-decimal token (stETH), `amountAfterFee = 1e18` for 1 stETH, and `tokenToETHRate ≈ 1e18`, so the product `1e18 × 1e18 / rsETHToETHrate` is correct.

For a 6-decimal token (USDC), `amountAfterFee = 1e6` for 1 USDC, but `tokenToETHRate ≈ 4e14`. The product `1e6 × 4e14 = 4e20` is `10^12` times smaller than the correct ETH-equivalent `1e18 × 4e14 = 4e32`, causing the minted rsETH to be `10^12` times too small.

The same uncorrected formula appears identically in:
- `contracts/pools/RSETHPoolV3ExternalBridge.sol` (line 452)
- `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol` (line 370)
- `contracts/pools/RSETHPool.sol` (line 346)
- `contracts/pools/RSETHPoolNoWrapper.sol` (line 311)

The `addSupportedToken` function in all these contracts performs no decimal validation on the token being added:

```solidity
// contracts/pools/RSETHPoolV3.sol L541-554
function addSupportedToken(address token, address oracle) external onlyRole(TIMELOCK_ROLE) {
    ...
    if (IOracle(oracle).getRate() == 0) revert UnsupportedOracle();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    ...
}
```

---

### Impact Explanation

**Critical — Direct theft / permanent freezing of user funds.**

A user depositing 1,000 USDC (worth ≈ 0.4 ETH at 0.0004 ETH/USDC) via `deposit(USDC, 1000e6, referralId)`:

| Variable | Value |
|---|---|
| `amountAfterFee` | ≈ 1,000e6 (6-decimal units) |
| `tokenToETHRate` | ≈ 4e14 (0.0004 ETH/USDC, 1e18 precision) |
| `rsETHToETHrate` | ≈ 1.05e18 |
| **Computed `rsETHAmount`** | **≈ 380 units (≈ 3.8 × 10⁻¹⁶ rsETH)** |
| **Correct `rsETHAmount`** | **≈ 3.8 × 10¹⁷ units (≈ 0.38 rsETH)** |

The user's 1,000 USDC is transferred to the pool via `safeTransferFrom` and subsequently bridged to L1. The user receives essentially zero rsETH with no mechanism to recover the deposited tokens. The funds are permanently lost to the user.

---

### Likelihood Explanation

**Low.** The current deployment targets ETH and LSTs (stETH, rETH, etc.), all of which have 18 decimals, so the formula is currently correct for all live supported tokens. The vulnerability activates the moment a TIMELOCK_ROLE holder adds any non-18-decimal token (e.g., USDC, USDT, WBTC) as a supported collateral — a plausible and legitimate protocol expansion that carries no obvious warning in the code. No malicious intent is required; the admin action is routine and the code provides no guard.

---

### Recommendation

Normalize `amountAfterFee` to 18 decimals before computing the rsETH output:

```solidity
// Before (incorrect for non-18-decimal tokens):
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;

// After (correct for any ERC-20 decimal count):
uint8 tokenDecimals = IERC20Metadata(token).decimals();
rsETHAmount = amountAfterFee * tokenToETHRate * 1e18
              / rsETHToETHrate / (10 ** tokenDecimals);
```

Apply the same fix to `viewSwapAssetToPremintedRsETH` (reverse direction) in all affected pool contracts, and add a decimal check (e.g., `require(decimals <= 18)`) inside `addSupportedToken`.

---

### Proof of Concept

1. Admin calls `addSupportedToken(USDC, chainlinkUsdcEthOracle)` on `RSETHPoolV3`. Oracle returns `getRate() ≈ 4e14` (0.0004 ETH/USDC). Check passes.
2. User calls `deposit(USDC, 1_000e6, "ref")` — depositing 1,000 USDC.
3. Pool executes `IERC20(USDC).safeTransferFrom(user, pool, 1_000e6)`. User's 1,000 USDC is now in the pool.
4. `viewSwapRsETHAmountAndFee(1_000e6, USDC)` computes:
   - `fee = 1_000e6 * feeBps / 10_000` (negligible)
   - `amountAfterFee ≈ 1_000e6`
   - `tokenToETHRate = 4e14`
   - `rsETHToETHrate = 1.05e18`
   - `rsETHAmount = 1_000e6 * 4e14 / 1.05e18 ≈ 380`
5. `wrsETH.mint(user, 380)` — user receives 380 rsETH units (≈ 3.8 × 10⁻¹⁶ rsETH, economically zero).
6. The 1,000 USDC is later bridged to L1 by the BRIDGER_ROLE. User has no recourse.

**Correct expected output:** `1_000e6 * 4e14 * 1e18 / 1.05e18 / 1e6 ≈ 3.8e17` units (≈ 0.38 rsETH). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    function getRate() public view returns (uint256) {
        (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
            AggregatorV3Interface(oracle).latestRoundData();

        if (answeredInRound < roundID) revert StalePrice();
        if (timestamp == 0) revert IncompleteRound();
        if (ethPrice <= 0) revert InvalidPrice();

        uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());

        return normalizedPrice;
    }
