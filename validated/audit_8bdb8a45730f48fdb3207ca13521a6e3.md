### Title
Sub-Transactions in Dijkstra Era Bypass `maxTxExUnits` Execution Unit Limit - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The Dijkstra era introduces nested (sub-)transactions. The top-level transaction's total Plutus execution units are validated against the `maxTxExUnits` protocol parameter in `dijkstraUtxoTransition`. However, the sub-transaction validation rule `dijkstraSubUtxoTransition` contains no equivalent check. An attacker can craft sub-transactions whose redeemers declare execution units exceeding `maxTxExUnits`, bypassing the per-transaction resource limit entirely.

---

### Finding Description

**Top-level transaction path** (`Utxo.hs`, `dijkstraUtxoTransition`):

The check `totExunits tx ≤ maxTxExUnits pp` is enforced:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [1](#0-0) 

`validateExUnitsTooBigUTxO` sums the declared `ExUnits` from the top-level transaction's redeemers (`totExUnits tx`) and rejects the transaction if they exceed `pp ^. ppMaxTxExUnitsL`:

```haskell
validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) $
    ExUnitsTooBigUTxO ...
  where
    maxTxExUnits = pp ^. ppMaxTxExUnitsL
    totalExUnits = totExUnits tx
``` [2](#0-1) 

`totExUnits` only sums the redeemers in `tx ^. witsTxL`, which is the top-level transaction's witness set. Sub-transactions carry their own separate `transaction_witness_set` (confirmed by the CDDL spec) and are not included in this sum. [3](#0-2) 

**Sub-transaction path** (`SubUtxo.hs`, `dijkstraSubUtxoTransition`):

The entire `dijkstraSubUtxoTransition` function (lines 231–278) contains no call to `validateExUnitsTooBigUTxO` or any equivalent check. The `DijkstraSubUtxoPredFailure` sum type has no `SubExUnitsTooBigUTxO` constructor: [4](#0-3) 

The injection function `dijkstraUtxoToDijkstraSubUtxoPredFailure` explicitly marks the `ExUnitsTooBigUTxO` case as a runtime `error`, confirming the check is intentionally absent and not merely unreachable:

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [5](#0-4) 

The `validateAllRefScriptSize` check in `Ledger.hs` uses `batchNonDistinctRefScriptsSize` to aggregate ref-script sizes across the whole batch (top-level + sub-transactions), showing the design intent is to apply batch-level limits. No analogous batch-level aggregation of execution units is performed for sub-transactions. [6](#0-5) 

---

### Impact Explanation

**Medium.** An attacker-controlled sub-transaction can declare Plutus redeemer `ExUnits` exceeding `maxTxExUnits` without rejection. Because the `SUBUTXOW` rule executes sub-transaction Plutus scripts using the declared budget, a single top-level transaction can carry multiple sub-transactions each with execution units far above the per-transaction limit. This allows a transaction submitter to exceed the intended per-transaction computational resource cap, violating the `maxTxExUnits` design parameter. This matches the allowed impact: *"Attacker-controlled transactions … exceed intended validation limits … outside design parameters."*

---

### Likelihood Explanation

The Dijkstra era is the only era supporting sub-transactions. Any unprivileged transaction sender can construct a `Tx TopTx era` with an `OMap` of `Tx SubTx era` values, each carrying a `transaction_witness_set` with redeemers whose declared `ExUnits` exceed `maxTxExUnits`. No special role, key, or governance threshold is required. The attacker only needs to pay the top-level transaction fee and provide valid collateral for the batch.

---

### Recommendation

Add an execution-unit limit check inside `dijkstraSubUtxoTransition` in `SubUtxo.hs`, analogous to the check in `dijkstraUtxoTransition`:

```haskell
{- totExunits subTx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

A `SubExUnitsTooBigUTxO` constructor should be added to `DijkstraSubUtxoPredFailure`, and the `error "Impossible"` branch in `dijkstraUtxoToDijkstraSubUtxoPredFailure` should be replaced with a proper mapping.

Alternatively, if the design intent is to apply the limit at the batch level, `validateAllRefScriptSize`'s pattern should be followed: aggregate `totExUnits` across all sub-transactions and the top-level transaction in `dijkstraLedgerTransition`, and compare the total against `maxTxExUnits`.

---

### Proof of Concept

1. Obtain the current `maxTxExUnits` value, e.g. `ExUnits { exUnitsMem = 14_000_000, exUnitsSteps = 10_000_000_000 }`.
2. Construct a `Tx SubTx DijkstraEra` whose `witsTxL . rdmrsTxWitsL` contains a redeemer with declared `ExUnits` of `ExUnits { exUnitsMem = 14_000_001, exUnitsSteps = 10_000_000_001 }` (one unit above the limit).
3. Embed this sub-transaction in a valid top-level `Tx TopTx DijkstraEra` (with valid fee, collateral, and inputs).
4. Submit the transaction. The ledger accepts it: `dijkstraSubUtxoTransition` never calls `validateExUnitsTooBigUTxO`, so the over-limit sub-transaction redeemer passes validation.
5. Repeat with N sub-transactions per top-level transaction to multiply the effect, each declaring execution units N × `maxTxExUnits`, bypassing the protocol-parameter resource cap entirely. [7](#0-6) [8](#0-7)

  let txBody = tx ^. bodyTxL

  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo
  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  let allSizedOutputs = txBody ^. allSizedOutputsTxBodyF
  let allOutputs = fmap sizedValue allSizedOutputs
  runTest $ Alonzo.validateOutputTooBigUTxO pp allOutputs

  runTest $ Shelley.validateInputSetEmptyUTxO txBody

  let inputs = txBody ^. inputsTxBodyL
  let refInputs = txBody ^. referenceInputsTxBodyL
