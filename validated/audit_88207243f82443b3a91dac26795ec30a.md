Audit Report

## Title
ETH Withdrawal Permanently Frozen When User's Smart Contract Address Becomes Unable to Receive ETH — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager` uses a two-phase withdrawal design where rsETH is burned and ETH is moved into the contract during `unlockQueue`, then delivered to the user via a raw `.call{value}` push-payment in `_processWithdrawalCompletion`. If the stored `user` address is a smart contract that permanently rejects ETH at claim time, the delivery reverts on every attempt, the ETH remains locked in `LRTWithdrawalManager`, and no on-chain recovery path exists because the rsETH was already destroyed in a prior transaction.

## Finding Description
**Phase 1 — `unlockQueue` (operator-triggered, separate transaction):**
rsETH held by the contract is burned and the corresponding ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`: [1](#0-0) 

After this call completes, the rsETH no longer exists. The ETH now sits in `LRTWithdrawalManager`.

**Phase 2 — `_processWithdrawalCompletion` (user or operator-triggered):**
The function pops the nonce, deletes the request, decrements `unlockedWithdrawalsCount`, and then attempts to push ETH to the stored `user` address: [2](#0-1) [3](#0-2) 

The delivery is performed by `_transferAsset` using a raw `.call`: [4](#0-3) 

If `user` is a smart contract whose `receive`/`fallback` reverts, the `.call` returns `false` and the function reverts with `EthTransferFailed`. Because the entire transaction reverts, all state mutations (the `popFront`, the `delete`, and the `unlockedWithdrawalsCount--`) are also reverted. The withdrawal record is fully restored, but the ETH transfer will fail identically on every subsequent attempt.

**No recovery path exists:**
- `sweepRemainingAssets` is gated by `hasUnlockedWithdrawals(asset)`, which checks `unlockedWithdrawalsCount[asset] > 0`. Since the stuck withdrawal keeps this counter elevated, the sweep is permanently blocked: [5](#0-4) [6](#0-5) 
- There is no admin function to redirect a stuck ETH withdrawal to an alternate address.
- There is no `cancelWithdrawal` function to return rsETH (it is already burned in Phase 1).
- `completeWithdrawalForUser` (operator path) calls the same `_processWithdrawalCompletion` with the same `user` address and suffers the same failure: [7](#0-6) 

## Impact Explanation
**Critical — Permanent freezing of funds.** Once `unlockQueue` burns the rsETH and moves ETH into `LRTWithdrawalManager`, the user has no rsETH left. If `completeWithdrawal` can never succeed because the destination contract permanently rejects ETH, the user loses both their rsETH (burned) and their ETH (frozen in the contract). There is no on-chain mechanism for the protocol to recover or redirect the frozen ETH. This matches the allowed Critical impact: *Permanent freezing of funds*.

## Likelihood Explanation
**Low.** The scenario requires the withdrawing address to be a smart contract (not an EOA) that, between the time `initiateWithdrawal` is called and `completeWithdrawal` is attempted, becomes unable to receive ETH. Realistic triggers include: an upgradeable smart contract wallet whose implementation is changed to remove `receive()`; a multisig or account-abstraction wallet whose guard logic begins reverting on ETH receipt; or a contract redeployed at the same address after `selfdestruct`. Smart contract wallets are increasingly common among large rsETH holders, making this a realistic edge case rather than a purely theoretical one.

## Recommendation
1. **Pull-payment pattern**: Instead of pushing ETH to `user` in `_processWithdrawalCompletion`, credit the amount to a per-user claimable balance mapping and let users pull it with a separate `claimETH(address recipient)` call that they can direct to any address they control.
2. **Alternate recipient at claim time**: Allow users to specify a `recipient` address at claim time (distinct from the initiating address), so a stuck smart contract wallet can redirect ETH to an EOA.
3. **Admin rescue**: Add a privileged function (e.g., `rescueStuckWithdrawal(address asset, address user, address alternateRecipient)`) callable only after a timeout and only by a high-privilege role (e.g., `TIMELOCK_ROLE`) that can redirect a permanently stuck ETH withdrawal to a different address without requiring `unlockedWithdrawalsCount` to be zero.

## Proof of Concept
```
1. Alice deploys an upgradeable proxy wallet W that has a receive() function.

2. Alice calls LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")
   from address W.
   → rsETH is transferred from W to LRTWithdrawalManager (line 166).
   → userAssociatedNonces[ETH_TOKEN][W] records the nonce.

3. Operator calls unlockQueue(ETH_TOKEN, ...).
   → rsETH held by LRTWithdrawalManager is burned (line 305).
   → ETH is pulled from LRTUnstakingVault into LRTWithdrawalManager (line 307).
   → Alice's rsETH is now gone.

4. Alice (or an attacker who compromised W) upgrades W's implementation to
   remove receive(), causing all ETH transfers to revert.

5. Alice calls LRTWithdrawalManager.completeWithdrawal(ETH_TOKEN, "").
   → _processWithdrawalCompletion is called with user = W.
   → _transferAsset calls payable(W).call{value: amount}("") (line 878).
   → W rejects ETH → sent == false → revert EthTransferFailed() (line 879).
   → Entire transaction reverts; all state mutations (popFront, delete, count--) are undone.

6. Operator calls completeWithdrawalForUser(ETH_TOKEN, W, "").
   → Same failure path.

7. unlockedWithdrawalsCount[ETH_TOKEN] > 0 forever.
   → sweepRemainingAssets reverts with PendingWithdrawalsExist() (line 403).

Result: Alice's ETH is permanently frozen in LRTWithdrawalManager.
        Alice's rsETH is permanently destroyed.
        No on-chain recovery path exists.
```
