### Title
IndexError in `get_block_challenge` when overflow block targets genesis-only chain — (`chia/consensus/get_block_challenge.py`)

### Summary
`get_block_challenge` breaks out of its backward-walk loop when it reaches the genesis block (height 0) without checking whether it has collected the required number of challenge hashes. When `challenges_to_look_for=2` but the genesis block supplies only 1 hash, the subsequent index access `reversed_challenge_hashes[1]` raises an unguarded `IndexError`, crashing block validation.

### Finding Description
In the `else` branch of `get_block_challenge` (lines 78–101), when `overflow=True` and `skip_overflow_last_ss_validation=False`, `challenges_to_look_for` is set to 2. The backward-walk loop appends hashes from `finished_challenge_slot_hashes` of each `first_in_sub_slot` block. When the genesis block is reached, the code asserts the list is non-empty and breaks — but does not verify that 2 hashes have been collected. The genesis block has exactly 1 `finished_challenge_slot_hash`, so `reversed_challenge_hashes` has length 1. Line 101 then accesses index 1, raising `IndexError`. [4](#0-3) 

### Impact Explanation
`pre_validate_block` calls `get_block_challenge` with `skip_overflow_last_ss_validation=False` unconditionally for every full block. An unhandled `IndexError` propagates out of `pre_validate_block`, crashing the validation coroutine. If not caught by the full node's block-processing loop, this crashes the node process. Even if caught, the attacker can repeat the submission indefinitely, keeping the node in a crash/restart cycle and preventing honest blocks from being processed. This is a sustained, low-cost chain-halt DoS.

### Likelihood Explanation
The attacker only needs a peer connection and the ability to send a `RespondBlock` message with a crafted block structure. No valid proof of space, VDF computation, or key material is required. The genesis-only chain state is transient but real (every node passes through it at startup or after a fresh sync).

### Recommendation
After the loop exits, guard the final index access:

```python
if len(reversed_challenge_hashes) < challenges_to_look_for:
    raise ValueError(
        f"Not enough challenge hashes: need {challenges_to_look_for}, "
        f"got {len(reversed_challenge_hashes)}"
    )
challenge = reversed_challenge_hashes[challenges_to_look_for - 1]
```

This converts the crash into a controlled `ValueError` that callers can catch and translate into a `ValidationError`, preserving deterministic, non-crashing behavior for all inputs.
