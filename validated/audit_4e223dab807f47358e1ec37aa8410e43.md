### Title
`tee_verifier_account_id` Is Voted In But Never Used to Dispatch DCAP Verification — (File: `crates/contract/src/lib.rs`)

### Summary
The `MpcContract` struct stores a `tee_verifier_account_id: Option<AccountId>` field that participants can vote to set via the `TeeVerifierVotes` mechanism. The field's own inline comment explicitly states it is **"Not yet used to dispatch verification."** As a result, the governance action of voting in a trusted DCAP verifier contract has zero effect on actual attestation verification: the voted-in account is stored but never read during `submit_participant_info` or any other attestation path.

### Finding Description

In `crates/contract/src/lib.rs`, the `MpcContract` struct declares:

```rust
/// The verifier contract account trusted for DCAP verification, or [`None`]
/// until participants vote one in. Not yet used to dispatch verification.
// TODO(#3639): once participants have voted a verifier in, make this
// non-optional via a migration that requires it be set.
tee_verifier_account_id: Option<AccountId>,
tee_verifier_votes: TeeVerifierVotes,
``` [1](#0-0) 

The `TeeVerifierVotes` type (in `crates/contract/src/tee/verifier_votes.rs`) implements a full threshold-voting mechanism: participants call a vote method, votes are tallied against the signing threshold, and when the threshold is crossed the winning `AccountId` is returned to the caller to be stored in `tee_verifier_account_id`. [2](#0-1) 

However, `submit_participant_info` — the sole entry point for adding a node's TEE attestation — calls only `self.tee_state.add_participant(...)` for internal verification and never reads `self.tee_verifier_account_id` to cross-check the attestation against the externally-audited verifier contract. [3](#0-2) 

The field is written (when votes cross threshold) but is never subsequently read anywhere in the contract's production code paths.

### Impact Explanation

Participants audit a specific verifier contract off-chain, commit to its `(account_id, code_hash)` pair, and vote it in expecting that future attestations will be validated against it. Because `tee_verifier_account_id` is never consulted, this governance action has no security effect. The TEE attestation verification relies entirely on the internal `TeeState` logic, bypassing the external verifier that the participant set explicitly chose and audited. This breaks the production safety invariant that the participant-voted verifier governs DCAP quote acceptance, and means any weakness in the internal verification path cannot be compensated by the external verifier contract. This maps to the **Medium** impact class: *contract execution-flow manipulation that breaks production safety/accounting invariants*.

### Likelihood Explanation

The condition is permanently active in the current codebase — the TODO comment and the field's own documentation confirm it is a known incomplete wiring. Any deployment where participants vote in a verifier account (the intended production flow per `docs/design/attestation-verifier-contract.md`) will silently ignore that vote for all verification decisions.

### Recommendation

1. Wire `tee_verifier_account_id` into `submit_participant_info`: once the field is `Some(verifier)`, dispatch a cross-contract call to `verifier` to validate the submitted DCAP quote before accepting the attestation.
2. Until the wiring is complete, either block `vote_change_tee_verifier` calls (to avoid misleading participants) or add a prominent runtime log/event indicating the verifier is stored but not yet enforced.
3. Track completion via the existing TODO `#3639` and gate production resharing on the field being non-`None` and actively used.

### Proof of Concept

1. Participant A calls `vote_change_tee_verifier` with `candidate_account_id = "audited-verifier.near"` and a specific `expected_code_hash`.
2. Threshold of participants vote the same proposal; `TeeVerifierVotes::vote` returns `Some("audited-verifier.near")` and the contract stores it in `self.tee_verifier_account_id`.
3. A new node calls `submit_participant_info` with any attestation blob.
4. `tee_state.add_participant` runs internal checks; `self.tee_verifier_account_id` is never read.
5. The node is accepted as a participant regardless of whether `"audited-verifier.near"` would have approved or rejected the quote — the governance vote had no effect. [1](#0-0) [4](#0-3) [5](#0-4)

```rust
impl TeeVerifierVotes {
    /// Records the participant's vote for the proposal. Returns the winning
    /// candidate account once it crosses the signing threshold (votes from
    /// dropped participants don't count); on a win, all pending votes are
    /// cleared and the caller must apply the new verifier account.
    pub fn vote(
        &mut self,
        proposal: VerifierChangeProposal,
        participant: AuthenticatedParticipantId,
        threshold_parameters: &ThresholdParameters,
    ) -> Result<Option<AccountId>, Error> {
        let protocol_threshold = threshold_parameters.threshold().value();
        let participants = threshold_parameters.participants();
        let proposal_hash: ProposalHash = proposal.clone().into();

        let count_usize = {
            let voter_set = self.pending.vote(participant, proposal_hash);
            voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
        };
        let count = u64::try_from(count_usize).map_err(|e| ConversionError::DataConversion {
            reason: format!("vote count {count_usize} does not fit in u64: {e}"),
        })?;

        if count >= protocol_threshold {
            self.pending.clear();
            Ok(Some(proposal.candidate_account_id))
        } else {
            Ok(None)
        }
    }
