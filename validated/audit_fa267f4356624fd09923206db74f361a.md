Let me now check the NEAR side handling of `LogMetadata` to understand if deduplication or identity checks exist there.

### Title
Token Identity Spoofing via Unverified MetadataPointer Cross-Mint Metadata Theft — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`LogMetadata::parse_metadata_account` verifies that the caller-supplied metadata account key matches the address stored in the mint's `MetadataPointer` extension, and that the account is owned by the Metaplex program — but it **never checks that the Metaplex metadata account's embedded `mint` field matches `self.mint.key()`**. An attacker can create a Token-2022 mint whose `MetadataPointer` points to the Metaplex PDA of a completely different, legitimate token, causing the bridge to post a Wormhole `LogMetadataPayload` that associates the attacker's mint with the legitimate token's name and symbol.

---

### Finding Description

**Root cause — missing `metadata.mint` binding check**

In `parse_metadata_account` the only two guards are:

```rust
require_keys_eq!(
    metadata.key(),
    address,                          // == metadata_pointer.metadata_address.0
    ErrorCode::InvalidTokenMetadataAddress,
);
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let metadata = MplMetadata::try_deserialize(&mut data.as_ref())?;
    Ok((metadata.name.clone(), metadata.symbol.clone()))   // ← returned verbatim
``` [1](#0-0) 

Neither guard checks `metadata.mint == self.mint.key()`. The Metaplex `MetadataAccount` struct carries a `mint` field that identifies which SPL mint the metadata belongs to, but it is never read here.

**How the MetadataPointer is consumed**

For a Token-2022 mint with a `MetadataPointer` that points to a third-party address (i.e., `metadata_pointer.metadata_address.0 != self.mint.key()` and `!= Pubkey::default()`), the code unconditionally delegates to `parse_metadata_account` with the pointer's target address:

```rust
} else if metadata_pointer.metadata_address.0 != Pubkey::default() {
    // Third-party metadata
    self.parse_metadata_account(metadata_pointer.metadata_address.0)?
``` [2](#0-1) 

**Concrete attack steps**

1. Attacker creates a Token-2022 mint (`attacker_mint`) with a `MetadataPointer` extension whose `metadata_address` is set to the Metaplex PDA of a legitimate token, e.g. USDC: `PDA([b"metadata", MetaplexID, usdc_mint_pubkey], MetaplexID)`.
2. Attacker calls `log_metadata`, passing the real USDC Metaplex account as the optional `metadata` account.
3. `require_keys_eq!(metadata.key(), address, ...)` passes — both sides equal the USDC Metaplex PDA.
4. `metadata.owner == &MetaplexID` passes — it is a genuine Metaplex account.
5. `MplMetadata::try_deserialize` succeeds and returns `name = "USD Coin"`, `symbol = "USDC"`.
6. The payload is assembled with `token = attacker_mint`, `name = "USD Coin"`, `symbol = "USDC"`, `decimals = attacker_mint.decimals`.

```rust
let payload = LogMetadataPayload {
    token: self.mint.key(),          // attacker_mint
    name: name.trim_end_matches('\0').to_string(),   // "USD Coin"
    symbol: symbol.trim_end_matches('\0').to_string(), // "USDC"
    decimals: self.mint.decimals,
}
.serialize_for_near(())?;
self.common.post_message(payload)?;
``` [3](#0-2) 

7. The Wormhole VAA is relayed to NEAR. `deploy_token_callback` verifies the emitter is the registered Solana factory and calls `deploy_token_internal` with `token_address = attacker_mint`, `name = "USD Coin"`, `symbol = "USDC"`. [4](#0-3) 

8. `deploy_token_internal` derives a NEAR account ID from `attacker_mint`'s pubkey prefix and deploys a new NEP-141 token with name "USD Coin" / symbol "USDC" backed by `attacker_mint`. [5](#0-4) 

9. NEAR now has a bridge-deployed token named "USD Coin (USDC)" whose canonical Solana address is `attacker_mint`. The attacker can mint arbitrary amounts of their worthless Solana token, bridge them to NEAR as "USDC", and sell them.

**The `mint_authority` guard does not prevent this.** It only blocks bridge-deployed (wrapped) mints from being re-registered; it does not prevent an attacker from creating a fresh native Token-2022 mint. [6](#0-5) 

---

### Impact Explanation

The attacker deploys a counterfeit "USD Coin / USDC" token on NEAR that is backed by a worthless attacker-controlled Solana mint. The attacker can:

- Mint unlimited attacker-side tokens on Solana, bridge them to NEAR as "USDC", and sell them to users who trust the name/symbol.
- Pre-empt legitimate USDC registration: NEAR deduplicates by `token_address` (the Solana mint pubkey), not by name/symbol, so the real USDC can still be registered later — but two tokens named "USDC" now exist in the bridge's registry, breaking ecosystem tooling and user trust.

This is asset-identity divergence: the bridge's backing guarantee (one NEAR token ↔ one authentic Solana mint) is violated. The NEAR-side "USDC" token has zero real backing.

---

### Likelihood Explanation

The attack requires no privileged access. Any unprivileged actor can:
- Create a Token-2022 mint (permissionless on Solana).
- Set `MetadataPointer` to any arbitrary address at mint creation time (no authority required).
- Call `log_metadata` (fully permissionless public instruction).

The only cost is SOL for rent and Wormhole fees. The attack is repeatable for any token whose Metaplex metadata exists on-chain.

---

### Recommendation

In `parse_metadata_account`, after deserializing the Metaplex account, assert that its `mint` field matches the mint being registered:

```rust
let mpl = MplMetadata::try_deserialize(&mut data.as_ref())?;
require_keys_eq!(
    mpl.mint,
    self.mint.key(),
    ErrorCode::InvalidTokenMetadataAddress,
);
Ok((mpl.name.clone(), mpl.symbol.clone()))
```

This single check closes the attack: an attacker can still point their `MetadataPointer` at any Metaplex PDA, but the deserialized `mint` field will not match `attacker_mint`, causing the instruction to revert.

    fn deploy_token_internal(
        &mut self,
        chain_kind: ChainKind,
        token_address: &OmniAddress,
        metadata: BasicMetadata,
        attached_deposit: NearToken,
    ) -> Promise {
        let deployer = self
            .token_deployer_accounts
            .get(&chain_kind)
            .unwrap_or_else(|| env::panic_str(BridgeError::DeployerNotSet.to_string().as_str()));
        let prefix = token_address.get_token_prefix();
        let token_id: AccountId = format!("{prefix}.{deployer}")
            .parse()
            .unwrap_or_else(|_| env::panic_str(BridgeError::ParseAccountId.to_string().as_str()));

        let storage_usage = env::storage_usage();
        self.add_token(
            &token_id,
            token_address,
            metadata.decimals,
            metadata.decimals,
        );

        require!(
            self.deployed_tokens.insert(&token_id),
            BridgeError::TokenExists.as_ref()
        );
        self.deployed_tokens_v2
            .insert(&token_id, &token_address.get_chain());

        let required_deposit = env::storage_byte_cost()
            .saturating_mul((env::storage_usage().saturating_sub(storage_usage)).into())
            .saturating_add(NEP141_DEPOSIT);

        require!(
            attached_deposit >= required_deposit,
            BridgeError::InsufficientStorageDeposit.as_ref()
        );

        env::log_str(
            &OmniBridgeEvent::DeployTokenEvent {
                token_id: token_id.clone(),
                token_address: token_address.clone(),
                metadata: metadata.clone(),
            }
            .to_log_string(),
        );

        ext_deployer::ext(deployer)
            .with_static_gas(DEPLOY_TOKEN_GAS)
            .with_attached_deposit(attached_deposit.saturating_sub(required_deposit))
            .deploy_token(token_id.clone(), metadata)
            .then(
                Self::ext(env::current_account_id())
                    .deploy_token_by_deployer_callback(token_address, token_id),
            )
    }
