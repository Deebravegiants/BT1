### Title
Unverified CKD Response Delivery for `AppPublicKey` Variant Allows Single Byzantine Participant to Compromise Confidential Key Derivation - (File: crates/contract/src/lib.rs)

### Summary

In `respond_ckd`, when a CKD request uses the `AppPublicKey` (privately-verifiable, legacy) variant, the contract performs **no cryptographic verification** of the response before delivering it to the user. A single Byzantine attested participant (strictly below the signing threshold) can submit a fabricated CKD response containing a key the attacker controls, causing the user to receive a compromised confidential key. This bypasses the threshold requirement that is the core security guarantee of the MPC network.

### Finding Description

The `respond_ckd` function in `crates/contract/src/lib.rs` contains a conditional verification block:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

When `app_public_key` is `AppPublicKeyPV`, the contract calls `ckd_output_check` to verify the response on-chain. When `app_public_key` is `AppPublicKey`, the arm is empty — the response is accepted and delivered unconditionally. The `resolve_yields_for` call that delivers the response to the waiting user runs in both cases.

This is the direct analog to the reported vulnerability: a flag/variant controls whether an actual verification step executes, but the state mutation (delivering the response) always occurs regardless. [1](#0-0) 

### Impact Explanation

The CKD protocol is designed so that the threshold of participants must cooperate to derive a key — no single participant should be able to unilaterally produce a valid CKD output. For `AppPublicKeyPV` requests, the contract enforces this on-chain via `ckd_output_check`. For `AppPublicKey` requests, it does not.

A single Byzantine attested participant can:
1. Observe a pending CKD request with `AppPublicKey` variant (the request is public on-chain).
2. Generate an arbitrary key pair `(sk_fake, pk_fake)`.
3. Encrypt `sk_fake` to the user's `app_public_key` (which is in the request).
4. Call `respond_ckd` with this fabricated `CKDResponse` before the honest threshold response lands.
5. The contract accepts the response without verification and delivers it to the user via `resolve_yields_for`.

The user decrypts the response with their app private key and receives `sk_fake`. The attacker knows `sk_fake`. The user's "confidential" derived key is now known to the attacker, breaking the confidentiality guarantee of the CKD protocol entirely.

This matches the allowed impact: **Critical — confidential key derivation output without the required participant authorization; bypass of threshold-signature requirements.** [2](#0-1) 

### Likelihood Explanation

- Only **one** attested participant is required — strictly below the signing threshold (which is ≥ 2 per the contract README).
- The `AppPublicKey` variant is the legacy/default path and is explicitly documented as supported.
- The pending CKD request is visible on-chain, so the attacker knows exactly when and what to respond to.
- The attacker must be an attested participant, but the entire threat model of threshold MPC is that individual participants may be Byzantine. [3](#0-2) 

### Recommendation

Apply the same on-chain verification to `AppPublicKey` requests, or reject `AppPublicKey` requests at the contract level if on-chain verification is not possible for that variant. At minimum, the contract should not deliver a CKD response for any variant without a cryptographic guarantee that the response was produced by the threshold protocol. If `AppPublicKey` cannot be verified on-chain, the contract should require the response to carry a threshold-signed attestation (e.g., a signature over the response bytes under the MPC key) so that a single participant cannot unilaterally forge a delivery. [4](#0-3) 

### Proof of Concept

1. User submits `request_app_private_key` with `app_public_key = AppPublicKey(user_g1_point)` and `domain_id` pointing to the BLS12-381 CKD domain.
2. The request enters `pending_ckd_requests` on-chain.
3. Attacker (a single attested participant) generates `sk_fake` and computes `enc = BLS_encrypt(sk_fake, user_g1_point)`.
4. Attacker calls `respond_ckd(request, CKDResponse { enc })`.
5. Contract executes the `AppPublicKey(_) => {}` arm — no check — then calls `resolve_yields_for`, resuming the user's yield with the fabricated response.
6. User's transaction completes successfully, returning the fabricated `CKDResponse`.
7. User decrypts with their app private key and obtains `sk_fake`. Attacker already knows `sk_fake`.

The user's confidential key is now known to the attacker. The threshold protocol was never consulted. [1](#0-0) [5](#0-4)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,

        )

### Submitting a confidential key derivation (ckd) request

Users can submit a ckd request to the MPC network via the
`request_app_private_key` endpoint of this contract. A **deposit of 1
yoctonear is required** (see [Deposit requirement](#deposit-requirement)).

The ckd request takes the following arguments:

- `derivation_path` (String): the derivation path (used to derive different keys from the same account).
- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
