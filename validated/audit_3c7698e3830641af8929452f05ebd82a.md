### Title
`conclude_node_migration` Removes Old Foreign Chain Config Without Transferring to New TLS Key, Breaking `verify_foreign_transaction` - (File: `crates/contract/src/lib.rs`)

---

### Summary

`conclude_node_migration` updates a participant's TLS key in the participant set and removes the old TLS key's entry from `foreign_chains_configs`, but does **not** transfer the foreign chain support registration to the new TLS key. After migration, the new node is counted as supporting zero foreign chains by `get_available_foreign_chains()`. If the migrated participant was needed to meet the signing threshold for a foreign chain, that chain drops out of the available set and `verify_foreign_transaction` is rejected for all callers until the new node explicitly re-registers.

---

### Finding Description

The contract maintains two separate foreign-chain support stores:

1. `foreign_chains.foreign_chains_configs` — keyed by **TLS public key** (`Ed25519PublicKey`), populated by `register_foreign_chains_config`. This is the store that drives `get_available_foreign_chains()` and gates `verify_foreign_transaction`.

2. `node_foreign_chain_support.foreign_chain_support_by_node` — keyed by **account ID**, populated by `register_foreign_chain_support`. This drives `get_supported_foreign_chains()`. [1](#0-0) 

When `conclude_node_migration` finalizes a migration, it:

- Updates the participant's `ParticipantInfo` (TLS key changes from old → new) in the participant set.
- Removes the **old** TLS key's entry from `foreign_chains_configs`.
- Calls `recompute_available_foreign_chains()`. [2](#0-1) 

It does **not** transfer the old TLS key's foreign chain config to the new TLS key. After migration, the new TLS key has no entry in `foreign_chains_configs` unless the new node pre-registered before calling `conclude_node_migration`. The contract does not enforce pre-registration, and `conclude_node_migration` does not check for it.

`recompute_available_foreign_chains` counts only participants whose current TLS key has an entry in `foreign_chains_configs`: [3](#0-2) 

Because the new TLS key has no entry, the migrated participant is counted as supporting zero chains. If this participant was needed to reach the signing threshold for a foreign chain, that chain leaves the available set.

`verify_foreign_transaction` gates on `get_available_foreign_chains()`, so it will reject requests for that chain until the new node re-registers.

The analog to the external report is exact:

| External Report | NEAR MPC |
|---|---|
| `setPlatform` updates platform address | `conclude_node_migration` updates participant TLS key |
| Old platform's allowance is revoked | Old TLS key's `foreign_chains_configs` entry is removed |
| New platform gets no allowance | New TLS key gets no `foreign_chains_configs` entry |
| `deposit` breaks | `verify_foreign_transaction` breaks |

---

### Impact Explanation

**Medium.** `verify_foreign_transaction` is gated on `get_available_foreign_chains()`. If the migrated participant was needed to meet the signing threshold for a foreign chain, that chain drops out of the available set immediately upon `conclude_node_migration`. All pending and future `verify_foreign_transaction` calls for that chain are rejected until the new node explicitly calls `register_foreign_chains_config`. This breaks the request lifecycle for foreign-chain verification without requiring any network-level DoS or threshold collusion — a single participant migration is sufficient.

---

### Likelihood Explanation

**Medium.** Any participant migration where the new node does not pre-register its foreign chain support before calling `conclude_node_migration` triggers this. The contract does not enforce pre-registration order. The test suite demonstrates pre-registration as a best practice but it is not a contract-enforced invariant. In production, a new node may call `conclude_node_migration` immediately after receiving keyshares, before registering foreign chain support, especially if the operator is unaware of the ordering requirement. [4](#0-3) 

---

### Recommendation

`conclude_node_migration` should transfer the old TLS key's foreign chain config to the new TLS key rather than simply deleting it. If the new node has already pre-registered (new TLS key already has an entry), the old entry should still be removed. If the new node has not pre-registered, the old entry should be moved to the new key:

```rust
// After updating participant info:
let new_tls_key = running_state
    .parameters
    .participants()
    .info(&account_id)
    .map(|info| info.tls_public_key.clone());

if let (Some(old_key), Some(new_key)) = (old_tls_key, new_tls_key) {
    let fc = self.foreign_chains.get_mut();
    // Only transfer if new node hasn't pre-registered
    if !fc.foreign_chains_configs.contains_key(&new_key) {
        if let Some(config) = fc.foreign_chains_configs.remove(&old_key) {
            fc.foreign_chains_configs.insert(new_key, config);
        }
    } else {
        fc.foreign_chains_configs.remove(&old_key);
    }
}
```

Alternatively, enforce that the new node must pre-register its foreign chain support before `conclude_node_migration` can succeed, by adding a check that the new TLS key has an entry in `foreign_chains_configs`.

### Proof of Concept

1. Network has 3 participants (threshold = 3 for foreign chain signing). All 3 support Bitcoin. Bitcoin is in the available set.
2. Participant 1 starts a node migration (`start_node_migration`) to a new node with TLS key Y.
3. New node receives keyshares and immediately calls `conclude_node_migration` **without** calling `register_foreign_chains_config` first.
4. `conclude_node_migration` executes:
   - Participant 1's TLS key is updated from X → Y in the participant set.
   - TLS key X's Bitcoin support entry is removed from `foreign_chains_configs`.
   - `recompute_available_foreign_chains()` is called.
   - TLS key Y has no entry → participant 1 counts as supporting zero chains.
   - Only 2 of 3 participants support Bitcoin → below threshold → Bitcoin leaves the available set.
5. Any user calling `verify_foreign_transaction` for a Bitcoin transaction receives a rejection.
6. This state persists until the new node calls `register_foreign_chains_config` with TLS key Y. [5](#0-4) [3](#0-2)

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
