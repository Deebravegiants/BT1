Audit Report

## Title
ETH Deposit Limit Bypass via Missing Deposit Amount in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison between ETH and ERC20 assets: the ERC20 branch correctly evaluates `totalAssetDeposits + amount > limit`, while the ETH branch evaluates only `totalAssetDeposits > limit`, omitting the incoming deposit amount. When the running ETH total equals the configured cap exactly, the ETH check returns `false` and the deposit proceeds, pushing the protocol's ETH holdings past the intended limit. The impact is the protocol failing to enforce its deposit-cap guarantee for ETH.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, `_checkIfDepositAmountExceedesCurrentLimit` branches on asset type:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount NOT included
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
``` [1](#0-0) 

The ETH branch uses a strict `>` against the pre-deposit total only. When `totalAssetDeposits == depositLimitByAsset`, the expression evaluates to `false`, signaling "limit not exceeded." `_beforeDeposit` (lines 648–670) then proceeds past the guard, computes `rsethAmountToMint`, and returns without reverting. [2](#0-1) 

`depositETH` (lines 76–93) calls `_beforeDeposit` and then `_mintRsETH`, accepting the ETH and minting rsETH for the caller with no further limit enforcement. [3](#0-2) 

After the call, `getTotalAssetDeposits(ETH_TOKEN)` — which aggregates ETH across the deposit pool, all NodeDelegators, EigenLayer pod shares, the unstaking vault, and the converter — reflects the new total exceeding the cap. [4](#0-3) 

## Impact Explanation
The `depositLimitByAsset` cap is the protocol's primary risk-management gate controlling how much ETH enters EigenLayer restaking. Bypassing it allows the protocol to absorb more ETH than its risk parameters permit and mint rsETH against the excess. No funds are directly stolen and the overage is bounded to a single deposit transaction. This matches the allowed impact: **Low — contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The bypass window opens when `getTotalAssetDeposits(ETH_TOKEN) == depositLimitByAsset(ETH_TOKEN)` exactly. This condition is reachable in normal operation (e.g., the last depositor fills the cap to the wei). The state is fully observable on-chain, `depositETH` is permissionless, and no privileged role or victim mistake is required. Likelihood is **Medium**.

## Recommendation
Include the incoming `amount` in the ETH branch, mirroring the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This unifies both paths and closes the one-deposit bypass window.

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Depositors fill the pool until `getTotalAssetDeposits(ETH_TOKEN) == 1000 ether`.
3. Attacker calls `depositETH{value: 100 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit`: `totalAssetDeposits = 1000 ether`; ETH branch evaluates `1000 ether > 1000 ether` → `false` → guard passes.
5. `_mintRsETH` mints rsETH for the attacker; 100 ETH is accepted.
6. `getTotalAssetDeposits(ETH_TOKEN)` is now `1100 ether`, 10% above the cap.
7. Subsequent depositors are correctly blocked (`1100 > 1000` → `true`), but the cap has already been breached.

**Foundry test plan:** Deploy `LRTDepositPool` with a mock `lrtConfig` returning `depositLimitByAsset = 1000 ether`. Pre-seed `getTotalAssetDeposits` to return exactly `1000 ether` (e.g., by sending ETH directly to the contract). Call `depositETH{value: 1 ether}(0, "")` and assert it does not revert and that `getTotalAssetDeposits` returns `1001 ether` post-call, confirming the cap was breached.
