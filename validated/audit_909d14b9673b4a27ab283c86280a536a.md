### Title
Unauthenticated P2P Peer Can Fabricate and Persist Arbitrary `ThinStateDiff` via Unverified `state_diff_length` and Missing State Diff Commitment Check — (`crates/apollo_p2p_sync/src/client/state_diff.rs`, `crates/apollo_p2p_sync/src/client/header.rs`)

### Summary
The P2P sync client accepts `SignedBlockHeader` messages from peers without ever verifying the block signature cryptographically, and after assembling a `ThinStateDiff` from peer-supplied chunks, never verifies the result against the `state_diff_commitment` stored in the header. Any unauthenticated peer can inject a header with an arbitrary `state_diff_length`, supply exactly that many units of fabricated `StateDiffChunk` data, and cause the node to persist a completely attacker-controlled `ThinStateDiff` to storage.

### Finding Description

**Step 1 — Header accepted without signature verification.**

`HeaderStreamBuilder::parse_data_for_block` receives a `SignedBlockHeader` from a peer and performs only two checks: block number ordering and that `signatures.len() == 1`. It never calls `verify_block_signature` (which exists in `starknet_api::block`) against a known sequencer public key. [1](#0-0) 

A grep across the entire `crates/apollo_p2p_sync` tree confirms zero calls to `verify_block_signature` or `SequencerPublicKey`. The signature is stored verbatim but never validated.

**Step 2 — `state_diff_length` from the untrusted header is used as the sole loop bound.**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` directly from the stored header's `state_diff_length` field — the same field the attacker controls. [2](#0-1) 

It then loops, consuming peer-supplied `StateDiffChunk` messages, until `current_state_diff_len == target_state_diff_len`. [3](#0-2) 

**Step 3 — No state diff commitment verification after assembly.**

After the loop exits, the assembled `ThinStateDiff` is returned directly. There is no call to `calculate_state_diff_hash` and no comparison against `header.state_diff_commitment`. The `state_diff_commitment` field is present in the stored header but is never used to validate the received data in this path. [4](#0-3) 

**Step 4 — Fabricated diff written unconditionally to storage.**

`write_to_storage` calls `append_state_diff` with no further validation. [5](#0-4) 

### Impact Explanation

A malicious peer (no privileges required — any node on the P2P network can serve the `SignedBlockHeader` and `StateDiff` SQMR protocols) can:

1. Send a `SignedBlockHeader` for block N with `state_diff_length = K` and any `state_diff_commitment` value.
2. Send K units of fabricated `StateDiffChunk` data (e.g., `ContractDiff` entries with attacker-chosen storage keys and values).
3. The sync client accepts and persists the fabricated `ThinStateDiff` for block N.

This corrupts the node's authoritative state: `starknet_getStorageAt` returns attacker-chosen values for any contract address modified in the fabricated diff. Deployed contract mappings, nonces, and declared class hashes can all be forged. If the affected node is the sequencer itself catching up via P2P sync, it will build subsequent blocks on top of corrupted state.

### Likelihood Explanation

The `SignedBlockHeader` SQMR protocol (`/starknet/headers/0.1.0-rc.0`) and `StateDiff` protocol (`/starknet/state_diffs/0.1.0-rc.0`) are unauthenticated at the application layer. Any peer that can establish a libp2p connection and respond to queries can execute this attack. No validator key, sequencer key, or operator privilege is required. [6](#0-5) 

### Recommendation

1. **Verify block signatures**: In `HeaderStreamBuilder::parse_data_for_block`, after receiving a `SignedBlockHeader`, call `verify_block_signature` with the known sequencer public key. Reject headers whose signature does not verify.
2. **Verify state diff commitment**: In `StateDiffStreamBuilder::parse_data_for_block`, after assembling the full `ThinStateDiff`, call `calculate_state_diff_hash(&result)` and compare against `header.state_diff_commitment`. Reject the diff if they do not match. [7](#0-6) [8](#0-7) 

### Proof of Concept

```
1. Attacker peer responds to a HeaderQuery for block 0 with:
   SignedBlockHeader {
     block_header: BlockHeader {
       block_number: 0,
       state_diff_length: Some(3),
       state_diff_commitment: Some(<arbitrary value>),
       ...
     },
     signatures: vec![BlockSignature::default()],  // passes the len==1 check
   }

2. Attacker peer responds to the StateDiffQuery with 3 ContractDiff chunks:
   ContractDiff { contract_address: 0xDEAD, storage_diffs: {0x1: 0xCAFE} }  // len=1
   ContractDiff { contract_address: 0xBEEF, storage_diffs: {0x2: 0xBABE} }  // len=1
   ContractDiff { contract_address: 0xF00D, storage_diffs: {0x3: 0xFACE} }  // len=1
   DataOrFin(None)

3. current_state_diff_len reaches 3 == target_state_diff_len=3, loop exits.
4. No commitment check. ThinStateDiff is written to storage.
5. storage_reader.get_state_diff(BlockNumber(0)) returns the fabricated diff.
6. starknet_getStorageAt(0xDEAD, 0x1) returns 0xCAFE.
```
