### Title
Ciphertext malleability via non-strict `convert<T>` in batch PVE parsing — (`src/cbmpc/api/pve_batch_single_recipient.cpp`)

### Summary
The `coinbase::convert<T>(T& dst, mem_t src)` template does not enforce that all input bytes are consumed during deserialization. This template is used at both the outer-blob and inner-blob parsing levels in the batch PVE API, allowing an attacker to produce multiple distinct ciphertext byte sequences that all pass `verify_batch` and `decrypt_batch` with identical output.

### Finding Description
`parse_batch_ciphertext` calls `coinbase::convert(out_blob, ciphertext)` [1](#0-0)  and `decrypt_batch`/`verify_batch` subsequently call `coinbase::convert(pve_ct, blob.ct)` [2](#0-1)  using the non-strict `convert<T>` template. [3](#0-2) 

This template only checks `converter.get_rv()` for a parse error; it does **not** check `converter.get_offset() != converter.get_size()`. [4](#0-3) 

By contrast, the `deser` template explicitly enforces strict consumption and rejects trailing bytes. [5](#0-4) 

The `ec_pve_batch_t::convert` method serializes a fixed structure of fields. [6](#0-5)  Any trailing bytes after the last field are silently ignored when deserializing via `convert<T>`.

### Impact Explanation
An attacker who can supply a ciphertext blob (e.g., a backup service consumer, a protocol peer, or any API caller) can produce a family of distinct byte sequences `{C, C', C'', ...}` that all:
- Pass `verify_batch` (line 117 also uses non-strict `convert<T>`) [7](#0-6) 
- Pass `decrypt_batch` and return the same scalar vector [8](#0-7) 

This breaks the invariant that a valid ciphertext has a unique canonical encoding. Concretely: systems using ciphertext bytes for deduplication, replay detection, audit logging, or commitment schemes will treat `C` and `C'` as distinct objects with identical cryptographic content, enabling confusion attacks. The `verify_batch` path accepting malleated ciphertexts as "verified" is the unsafe state acceptance.

### Likelihood Explanation
The attacker only needs to append arbitrary bytes to a ciphertext they already possess (or observe). No cryptographic knowledge is required. The path is directly reachable from the public C++ API (`decrypt_batch`, `verify_batch`) and the C API (`cbmpc_pve_batch_decrypt`, `cbmpc_pve_batch_verify`). [9](#0-8) 

### Recommendation
Replace `coinbase::convert(out_blob, ciphertext)` in `parse_batch_ciphertext` and `coinbase::convert(pve_ct, blob.ct)` in `decrypt_batch`, `verify_batch`, `get_public_keys_compressed_batch`, and `get_Label_batch` with the strict `coinbase::deser(...)` variant, which already exists and enforces that all bytes are consumed. [5](#0-4)
