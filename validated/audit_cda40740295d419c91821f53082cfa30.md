### Title
`compare_retrospective_block_hash` Disabled in Production Deployment Config Allows Divergent Block Hash to Be Silently Committed, Corrupting `get_block_hash` Syscall Results — (`crates/apollo_deployments/resources/app_configs/consensus_manager_config.json`)

---

### Summary

The production deployment configuration explicitly sets `compare_retrospective_block_hash: false`, disabling the cross-validation between the Batcher and State Sync for the retrospective block hash. This is the direct Sequencer analog to the `minBlockHeight = 0` issue: a configurable safety check is turned off in production, allowing a divergent (wrong) block hash to be silently embedded in the block's execution context and returned by the `get_block_hash` syscall to executing contracts.

---

### Finding Description

The function `retrospective_block_hash()` in `crates/apollo_consensus_orchestrator/src/utils.rs` is designed to cross-validate the retrospective block hash from two independent sources — the Batcher and State Sync — before embedding it in a block proposal or validation:

```rust
if compare_retrospective_block_hash && state_sync_block_hash != batcher_block_hash {
    // returns Err(HashMismatch)
}
Ok(Some(BlockHashAndNumber { number: block_number, hash: batcher_block_hash }))
```

When `compare_retrospective_block_hash = true`, a mismatch between the two sources causes an error, preventing the wrong hash from being used. When `false`, the function silently returns the Batcher's hash unconditionally, even if it diverges from State Sync. [1](#0-0) 

The production deployment configuration explicitly sets this flag to `false`: [2](#0-1) 

The same insecure setting appears in the replacer config: [3](#0-2) 

The code-level default is `true`: [4](#0-3) 

The returned hash flows directly into `ValidateBlockInput.retrospective_block_hash` (validator path via `initiate_validation`) and `ProposeBlockInput.retrospective_block_hash` (proposer path via `initiate_build`): [5](#0-4) [6](#0-5) 

Both paths pass the hash into `BlockMetadata.retrospective_block_hash`, which the blockifier uses to serve `get_block_hash` syscall responses for the block at `current_height - STORED_BLOCK_HASH_BUFFER`. [7](#0-6) 

---

### Impact Explanation

If the Batcher holds a wrong block hash for the retrospective block (due to a sync delay, chain reorganization, storage corruption, or a bug), and `compare_retrospective_block_hash = false`, the wrong hash is silently embedded in the block's execution context. Every contract that calls `get_block_hash(current_block - STORED_BLOCK_HASH_BUFFER)` during that block's execution will receive the wrong value. This constitutes a wrong result from blockifier/syscall execution logic for accepted input — a Critical-class impact per the defined scope.

Because both the proposer and validator run with the same flag value, and both independently query their own Batcher, they may agree on the same wrong hash, causing the proposal to be accepted and the wrong hash to be finalized on-chain.

---

### Likelihood Explanation

The trigger condition — Batcher diverging from State Sync on the retrospective block hash — is a realistic operational scenario during node startup, sync catch-up, or chain reorganization. The production config actively disables the guard that would detect and reject this divergence. This is not a theoretical misconfiguration: it is the shipped production deployment artifact.

---

### Recommendation

1. Set `compare_retrospective_block_hash: true` in all production and replacer deployment configs.
2. Add a startup validation that logs a prominent warning (or refuses to start) when `compare_retrospective_block_hash = false` in a non-test environment.
3. Treat `compare_retrospective_block_hash = false` as a debug/testing-only override, analogous to how the external report recommended refusing to run with `minBlockHeight = 0`.

---

### Proof of Concept

1. Deploy a node using the production config (`consensus_manager_config.json`) — `compare_retrospective_block_hash` is `false`.
2. Introduce a divergence between the Batcher's stored hash and State Sync's hash for block `N` (e.g., via a sync delay or a deliberate state corruption in the Batcher's storage).
3. When the node proposes or validates a block at height `N + STORED_BLOCK_HASH_BUFFER`, `retrospective_block_hash()` fetches both hashes, skips the comparison, and returns the Batcher's (wrong) hash.
4. The wrong hash is passed into `ProposeBlockInput` / `ValidateBlockInput` → `BlockMetadata.retrospective_block_hash`.
5. Any contract executing in that block that calls `get_block_hash(N)` receives the wrong value, producing wrong execution results that are finalized on-chain. [8](#0-7) [9](#0-8)

pub(crate) async fn retrospective_block_hash(
    batcher_client: Arc<dyn BatcherClient>,
    state_sync_client: SharedStateSyncClient,
    init: &ProposalInit,
    compare_retrospective_block_hash: bool,
) -> RetrospectiveBlockHashResult<Option<BlockHashAndNumber>> {
    if let Some(required_height) = (init.height.0 + 1).checked_sub(STORED_BLOCK_HASH_BUFFER) {
        // Just verify that the batcher has done calculating it.
        batcher_client.get_block_hash(BlockNumber(required_height)).await?;
    }

    let retrospective_block_number = init.height.0.checked_sub(STORED_BLOCK_HASH_BUFFER);

    let Some(block_number) = retrospective_block_number else {
        info!(
            "Retrospective block number is less than {STORED_BLOCK_HASH_BUFFER}, setting None as \
             expected."
        );
        return Ok(None);
    };

    let block_number = BlockNumber(block_number);

    // First try from state sync - assuming it takes longer to this one to be ready.
    let state_sync_block_hash = state_sync_client.get_block_hash(block_number).await?;

    // Then try from batcher.
    let batcher_block_hash = batcher_client.get_block_hash(block_number).await?;

    if compare_retrospective_block_hash && state_sync_block_hash != batcher_block_hash {
        warn!(
            "Retrospective block hashes mismatch for block {block_number}: state sync block hash: \
             {state_sync_block_hash:?}, batcher block hash: {batcher_block_hash:?}"
        );
        CONSENSUS_RETROSPECTIVE_BLOCK_HASH_MISMATCH.increment(1);
        return Err(RetrospectiveBlockHashError::HashMismatch {
            block_number,
            state_sync_block_hash,
            batcher_block_hash,
        });
    }
    Ok(Some(BlockHashAndNumber { number: block_number, hash: batcher_block_hash }))
}
