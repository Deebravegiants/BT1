### Title
Missing State Diff Hash Verification in P2P Sync Allows Malicious Peer to Inject Wrong ThinStateDiff — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

`parse_data_for_block` in the P2P sync client assembles a `ThinStateDiff` from peer-supplied `StateDiffChunk` messages and validates only the **total chunk length** against `state_diff_length` from the stored header. It never calls `calculate_state_diff_hash` to verify the assembled diff against the `state_diff_commitment` also stored in that header. A malicious peer can therefore supply N chunks whose lengths sum correctly but whose content differs from the committed diff, causing a wrong `ThinStateDiff` to be written to storage via `append_state_diff`.

### Finding Description

`parse_data_for_block` reads `target_state_diff_len` from the stored header and loops until `current_state_diff_len == target_state_diff_len`: [1](#0-0) 

After the loop it calls `validate_deprecated_declared_classes_non_conflicting` and returns `Ok(Some((result, block_number)))` — no hash check: [2](#0-1) 

The assembled diff is then written unconditionally to MDBX storage: [3](#0-2) 

The header that was stored earlier carries both `state_diff_length` (used) and `state_diff_commitment` (ignored): [4](#0-3) 

The `apollo_committer` service does have a `verify_state_diff_hash` guard (defaulting to `true`): [5](#0-4) 

However, the P2P sync client writes directly to `StorageWriter::append_state_diff`, completely bypassing the committer service. The committer guard is therefore irrelevant to this path.

### Impact Explanation

A syncing node (including a sequencer node catching up via P2P) stores a `ThinStateDiff` whose content does not match the `state_diff_commitment` in the header. Every subsequent RPC query for storage values, nonces, class hashes, or compiled class hashes for that block returns the attacker-chosen values. If the sequencer uses this corrupted state as the base for block production, the resulting blocks will have a wrong post-state root, breaking consensus.

### Likelihood Explanation

Any unauthenticated peer the node connects to for state diff sync can mount this attack. No privileged access is required. The attacker only needs to control one peer that the victim node queries for state diffs. The attack is silent: the node reports no error, the stored `state_diff_commitment` in the header appears valid, and the discrepancy is only detectable by re-hashing the stored diff.

### Recommendation

After the assembly loop completes, compute `calculate_state_diff_hash(&result)` and compare it against `storage_reader.get_block_header(block_number)?.state_diff_commitment`. Return `ParseDataError::BadPeer(...)` on mismatch, consistent with the existing length-mismatch handling.

### Proof of Concept

1. Seed a header for block N with `state_diff_commitment = calculate_state_diff_hash(honest_diff)` and `state_diff_length = honest_diff.len()`.
2. Construct a `crafted_diff` with the same `len()` but different storage/nonce/class values.
3. Split `crafted_diff` into `StateDiffChunk`s and feed them to `parse_data_for_block`.
4. Assert `storage_reader.get_state_diff(N) == crafted_diff` (not `honest_diff`).
5. Assert `calculate_state_diff_hash(stored_diff) != header.state_diff_commitment`.

Both assertions pass against the current code, confirming the wrong diff is silently committed.
