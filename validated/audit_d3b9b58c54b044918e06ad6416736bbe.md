### Title
`respond_verify_foreign_tx` Does Not Validate `response.payload_hash` Against the Pending Request - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never checks that `response.payload_hash` is the hash of a `ForeignTxSignPayload` that encodes the correct `request`. A single Byzantine attested participant can replay a legitimately-produced MPC signature (obtained for their own request) as the response to a different user's pending request, delivering a forged foreign-chain verification attestation.

### Finding Description

In `respond_verify_foreign_tx` the contract performs two independent checks:

1. The caller is an attested participant.
2. The signature in `response` is valid over `response.payload_hash` using the domain's root public key. [1](#0-0) 

What is **never** checked is whether `response.payload_hash` is the canonical hash of the pending `request`. The signed payload is defined as:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

The `values` (observed foreign-chain data) are not part of the on-chain `VerifyForeignTransactionRequest`, so the contract cannot recompute the expected hash. The `VerifyForeignTransactionResponse` carries only `payload_hash` and `signature` — both fully attacker-controlled when submitted by a Byzantine participant. [3](#0-2) 

The `request` parameter is used only as a lookup key into `pending_verify_foreign_tx_requests` to find which yield to resume. The `response.payload_hash` that is serialized and delivered to the waiting caller is never cross-checked against that key. [4](#0-3) 

Contrast this with the regular `respond` method, where the payload hash is taken from `request.payload` (contract-controlled), not from the response: [5](#0-4) 

### Impact Explanation

A bridge contract calling `verify_foreign_transaction` for transaction T2 receives a `VerifyForeignTransactionResponse` whose `payload_hash` actually attests to a completely different transaction T1 (with different extracted values). The signature is cryptographically valid (it was produced by the MPC network for T1), so any downstream verifier that trusts the returned `payload_hash` and signature will accept a fraudulent attestation. This enables invalid bridge execution or double-spend conditions — matching the **High** allowed impact of "forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."

### Likelihood Explanation

The attack requires exactly one Byzantine attested participant — well below the signing threshold. The attacker:

1. Submits their own `verify_foreign_transaction` for a transaction T1 they control.
2. The MPC network legitimately produces a threshold signature over `hash(T1, values1)`.
3. The attacker, as an attested participant, calls `respond_verify_foreign_tx` targeting a different pending request (for T2), supplying `request = T2` (to match the pending map entry) and `response = { payload_hash: hash(T1, values1), signature: <legitimate sig> }`.
4. The contract's signature check passes; the yield for T2 is resumed with the forged response.

No threshold collusion, no key leakage, and no network-level DoS is required.

### Recommendation

Include the `values` (extracted observations) in the `VerifyForeignTransactionResponse` so the contract can recompute and enforce:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),
}).compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Then verify the signature against `expected_hash` (not `response.payload_hash`). This mirrors the fix applied in the H-01 reference report: validate the user-supplied field against the contract's own authoritative value before accepting the response.

### Proof of Concept

```
1. Attacker (attested participant P_evil) submits verify_foreign_transaction(T1).
2. MPC threshold signs hash(T1, values1) → sig1.
3. Victim submits verify_foreign_transaction(T2); pending map now has entry for T2.
4. P_evil calls respond_verify_foreign_tx(
       request  = T2,                          // matches pending entry → lookup succeeds
       response = { payload_hash: hash(T1, values1), signature: sig1 }
   ).
5. Contract checks: verify_ecdsa(sig1, hash(T1, values1), root_pk) → OK.
6. Contract resolves yield for T2 with response containing hash(T1, values1).
7. Victim's bridge contract receives a valid MPC signature attesting to T1's
   observed values, not T2's — forged verification accepted.
```

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
