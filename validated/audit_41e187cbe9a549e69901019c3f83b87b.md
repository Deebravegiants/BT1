Audit Report

## Title
No Staleness Check on `CrossChainRateReceiver.getRate()` Allows Depositors to Mint Excess rsETH at Stale Rate — (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary

`CrossChainRateReceiver` stores both a `rate` and a `lastUpdated` timestamp, but `getRate()` returns `rate` unconditionally without verifying freshness. All L2 pool contracts use this oracle to compute rsETH minting amounts. When the L1 rsETH/ETH rate rises due to yield accrual but the L2 oracle is not refreshed, any depositor can exploit the stale (lower) rate to receive more rsETH than their ETH contribution is worth, extracting value from existing rsETH holders upon L1 redemption.

## Finding Description

`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness guard:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol:102-105
function getRate() external view returns (uint256) {
    return rate;
}
```

`lastUpdated` is written in `lzReceive()` at line 97 but is never read inside `getRate()`. The rate is only refreshed when someone calls `updateRate()` on the L1 provider and pays LayerZero messaging fees — there is no on-chain enforcement or incentive for timely updates.

`RSETHPoolV3.getRate()` delegates directly to this oracle:

```solidity
// contracts/pools/RSETHPoolV3.sol:234-237
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
```

The minting formula in `viewSwapRsETHAmountAndFee` divides by this rate:

```solidity
// contracts/pools/RSETHPoolV3.sol:307
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because division by a smaller (stale) rate yields a larger quotient, a depositor who acts while the oracle is stale receives more rsETH than their ETH is worth at the current true exchange rate. The `dailyMintLimit` modifier caps per-day exposure but does not prevent the attack — it only bounds the magnitude within a single day.

**Exploit path:**
1. L1 `rsETHPrice` rises from `1.05e18` to `1.10e18` due to staking yield accrual.
2. No one calls `updateRate()` (LayerZero fees are non-trivial, no reimbursement).
3. L2 `CrossChainRateReceiver.rate` remains `1.05e18`.
4. Attacker calls `RSETHPoolV3.deposit{value: 10 ether}("")`:
   - Minted: `10e18 * 1e18 / 1.05e18 ≈ 9.524e18` rsETH
   - Correct at true rate: `10e18 * 1e18 / 1.10e18 ≈ 9.091e18` rsETH
   - Excess: `≈ 0.433e18` rsETH
5. Attacker bridges wrsETH to L1 and redeems at `1.10e18`, receiving `≈ 10.476 ETH`.
6. Net profit: `≈ 0.476 ETH` per 10 ETH deposited, extracted from existing rsETH holders.

## Impact Explanation

**High — Theft of unclaimed yield.** The excess rsETH minted at the stale rate represents yield that has accrued on L1 but has not yet been reflected in the L2 oracle. When the attacker redeems on L1 at the true rate, they claim ETH that belongs to existing rsETH holders. The magnitude scales with both the staleness duration and deposit size. This is a direct, concrete extraction of value from protocol participants, not a hypothetical loss.

## Likelihood Explanation

**Medium.** No privileged access is required. The attacker only needs to monitor the divergence between the on-chain L1 `rsETHPrice` (publicly readable) and the L2 `CrossChainRateReceiver.rate` (also publicly readable). During periods of high gas costs or low protocol activity, the rate will naturally drift stale. The attack is repeatable up to the daily mint limit each day the oracle remains stale, and the attacker bears no risk beyond normal deposit/bridge/redeem costs.

## Recommendation

Add a configurable `MAX_STALENESS` constant and revert in `getRate()` if the rate is too old:

```solidity
uint256 public constant MAX_STALENESS = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate is stale");
    return rate;
}
```

This converts silent mis-pricing into a hard revert, preventing deposits when the oracle has not been refreshed. Pair this with an on-chain keeper incentive (e.g., a small ETH reward funded from protocol fees) to ensure `updateRate()` is called regularly.

## Proof of Concept

**Foundry fork test outline:**

1. Fork an L2 network where `CrossChainRateReceiver` is deployed.
2. Record `CrossChainRateReceiver.rate` (e.g., `1.05e18`) and `lastUpdated`.
3. `vm.warp(block.timestamp + 2 days)` to simulate staleness without an `lzReceive` call.
4. Call `RSETHPoolV3.deposit{value: 10 ether}("")` as an unprivileged attacker address.
5. Assert that `wrsETH.balanceOf(attacker) > 10e18 * 1e18 / trueRate` where `trueRate` is the current L1 `rsETHPrice`.
6. Simulate bridging wrsETH to L1 and redeeming; assert attacker receives more ETH than deposited.
7. Assert the shortfall is borne by existing rsETH holders (total ETH backing per rsETH decreases).
