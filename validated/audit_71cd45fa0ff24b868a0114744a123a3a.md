### Title
Pre-Funded rsETH Balance Exhaustion Renders `RSETHPoolNoWrapper` Deposit Pool Unavailable — (File: contracts/pools/RSETHPoolNoWrapper.sol)

---

### Summary

`RSETHPoolNoWrapper` distributes rsETH to depositors by transferring from its own pre-funded token balance rather than minting. Any unprivileged depositor can exhaust this balance in a single transaction, permanently blocking all subsequent deposits until an operator manually refills the pool.

---

### Finding Description

`RSETHPoolNoWrapper.deposit()` (ETH path) and `deposit(address token, ...)` (LST path) both fulfill user deposits by calling `rsETH.safeTransfer(msg.sender, rsETHAmount)` — transferring from the pool's own rsETH holdings:

```solidity
// ETH deposit path
rsETH.safeTransfer(msg.sender, rsETHAmount);   // line 241

// Token deposit path
rsETH.safeTransfer(msg.sender, rsETHAmount);   // line 268
```

The pool holds a finite, pre-funded supply of rsETH. There is no on-chain mechanism to replenish it; refilling requires an operator to manually send rsETH to the contract address. Once the balance reaches zero, every subsequent call to either `deposit` overload reverts on the `safeTransfer` call, making the pool completely unavailable to all depositors.

Unlike `RSETHPoolV3` and `RSETHPoolV3ExternalBridge`, which call `wrsETH.mint(msg.sender, rsETHAmount)` and are therefore not supply-constrained, `RSETHPoolNoWrapper` has no minting capability and is entirely dependent on its stored balance.

---

### Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

When the rsETH balance is exhausted:
- All calls to `deposit(string)` and `deposit(address, uint256, string)` revert.
- The pool is completely unavailable for new depositors until manually refilled.
- ETH and LST assets already held in the pool (from prior depositors) are unaffected and will be bridged normally; no existing user funds are frozen.
- The attacker receives rsETH at the oracle-determined fair rate — no value is extracted beyond what a normal depositor would receive.

---

### Likelihood Explanation

**High.**

- The entry path is the public, permissionless `deposit` function — no role, whitelist, or special condition required.
- The pool's rsETH balance is publicly readable on-chain (`rsETH.balanceOf(poolAddress)`), so an attacker can calculate the exact deposit amount needed to drain it in one transaction.
- The attack costs only gas plus the ETH/LST deposited (which is returned as rsETH at fair value), making it economically neutral for the attacker.
- The pool has no daily mint cap (unlike `RSETHPoolV3`), so there is no rate-limiting protection.

---

### Recommendation

Replace the `safeTransfer` distribution model with a mint-on-deposit model (as used in `RSETHPoolV3`), or introduce an on-chain rsETH reserve threshold that pauses deposits when the balance falls below a safe floor, triggering an automated refill alert. At minimum, add an explicit balance check before the transfer and emit a low-balance warning event so operators can react before the pool goes dark.

---

### Proof of Concept

1. Observer reads `rsETH.balanceOf(RSETHPoolNoWrapper)` = `X` rsETH.
2. Attacker calls `RSETHPoolNoWrapper.deposit{value: Y}("")` where `Y` is chosen so that `viewSwapRsETHAmountAndFee(Y).rsETHAmount ≥ X`.
3. The transfer succeeds; attacker receives `≥ X` rsETH at fair oracle rate.
4. Pool rsETH balance is now 0.
5. Any subsequent user calling `deposit{value: 1 ether}("")` receives a revert from `rsETH.safeTransfer` (ERC20 insufficient balance), with no error message distinguishing this from a pause or oracle failure.
6. Pool remains unavailable until an operator sends rsETH directly to the contract. [1](#0-0) [2](#0-1) [3](#0-2)
