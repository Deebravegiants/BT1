### Title
`unlockedWithdrawalsCount` Not Initialized in `initialize()`, Causing Withdrawal Completion Underflow After Upgrade - (File: contracts/LRTWithdrawalManager.sol)

### Summary

`LRTWithdrawalManager.initialize()` does not set `unlockedWithdrawalsCount` for any asset, leaving it at the Solidity default of 0. After a contract upgrade where withdrawals had already been unlocked, `_processWithdrawalCompletion()` unconditionally decrements this counter, causing an arithmetic underflow revert in Solidity 0.8+. Users whose rsETH was already transferred into the contract during `initiateWithdrawal()` cannot complete their withdrawals, and there is no cancel mechanism to recover their tokens.

### Finding Description

`LRTWithdrawalManager` tracks how many withdrawal requests have been unlocked but not yet completed via `unlockedWithdrawalsCount`: [1](#0-0) 

The `initialize()` function never sets this mapping: [2](#0-1) 

When `completeWithdrawal()` or `completeWithdrawalForUser()` is called, `_processWithdrawalCompletion()` unconditionally decrements the counter **after** verifying the request is unlocked (i.e., `usersFirstWithdrawalRequestNonce < nextLockedNonce[asset]`): [3](#0-2) 

If `unlockedWithdrawalsCount[asset]` is 0 while `nextLockedNonce[asset] > 0` (meaning requests have been unlocked), the decrement at line 717 panics with an arithmetic underflow, reverting the entire transaction.

The `initialize2()` reinitializer was added specifically to seed this counter for existing deployments: [4](#0-3) 

However, `initialize2()` is a `reinitializer(2)` — it can be called **exactly once**. If it is called with incorrect (e.g., zero) values, or if it is never called after an upgrade, there is no subsequent mechanism to correct `unlockedWithdrawalsCount`. The contract has no `cancelWithdrawal()` function, so rsETH already transferred in `initiateWithdrawal()` is permanently stuck.

The parallel to the ZetaChain bug is direct: a counter tracking live protocol state is not persisted through an initialization boundary, causing downstream state-dependent operations to fail irreversibly.

### Impact Explanation

Any user who called `initiateWithdrawal()` before the upgrade (transferring rsETH to the contract) and whose request was subsequently unlocked via `unlockQueue()` cannot call `completeWithdrawal()`. The call reverts at `unlockedWithdrawalsCount[asset]--` with an underflow panic. Because there is no cancel path, the rsETH is frozen in the contract.

**Impact: Medium — Temporary freezing of funds** (if `initialize2()` has not yet been called and can still be invoked with correct values) escalating to **Critical — Permanent freezing of funds** (if `initialize2()` was already called with incorrect values, exhausting the one-time reinitializer slot).

### Likelihood Explanation

The trigger is a contract upgrade to a new implementation without correctly seeding `unlockedWithdrawalsCount` via `initialize2()`. The protocol explicitly acknowledges this risk in the `initialize2()` NatSpec ("New deployments MUST NOT call this function") and created a separate `UnlockedWithdrawalsInitializer` utility contract to compute the correct seed values off-chain. The existence of this tooling confirms the scenario is realistic and was encountered in practice. Any future upgrade that introduces a new `initialize()` variant without carrying forward `unlockedWithdrawalsCount` would re-expose the same issue.

### Recommendation

1. In any future `initialize` or `reinitializer` function, explicitly read and persist `unlockedWithdrawalsCount` from on-chain state rather than relying on a one-time admin call with off-chain-computed values.
2. Add a guarded admin setter for `unlockedWithdrawalsCount` (e.g., callable only when the counter is demonstrably wrong relative to `nextLockedNonce - nextUnusedNonce`) so that misconfiguration after `initialize2()` can be corrected.
3. Add a `cancelWithdrawal()` function so users can reclaim their rsETH if the withdrawal system becomes inoperable.

### Proof of Concept

1. Protocol has `nextLockedNonce[ETH] = 5` and `nextUnusedNonce[ETH] = 7` (5 requests unlocked, 2 pending).
2. Contract is upgraded; `initialize2()` is not called (or is called with `unlockedWithdrawalsCountETH = 0`).
3. `unlockedWithdrawalsCount[ETH]` is 0.
4. User with nonce 3 (unlocked, since `3 < nextLockedNonce[ETH] = 5`) calls `completeWithdrawal(ETH, ...)`.
5. `_processWithdrawalCompletion()` passes the `WithdrawalLocked` check at line 707.
6. Execution reaches line 717: `unlockedWithdrawalsCount[ETH]--` → underflow panic → revert.
7. User's rsETH (transferred in step of `initiateWithdrawal`) remains locked in the contract with no recovery path. [5](#0-4)
