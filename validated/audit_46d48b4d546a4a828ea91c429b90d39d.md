Audit Report

## Title
Permissionless `updateRate()` with No Rate Validation Allows Any Caller to Broadcast Transient Oracle Dips to All Cross-Chain Receivers — (`contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary

`MultiChainRateProvider.updateRate()` carries no access control beyond a `nonReentrant` guard, and performs zero sanity-checking on the value returned by `getLatestRate()`. Any external caller who supplies enough ETH for LayerZero fees can invoke it at any block, including one where the underlying oracle is reporting a transient low value, permanently committing that depressed rate to every registered `AGETHRateReceiver` on every destination chain until the next `updateRate()` call.

## Finding Description

`AGETHMultiChainRateProvider.getLatestRate()` delegates unconditionally to the external oracle:

```solidity
// contracts/agETH/AGETHMultiChainRateProvider.sol L29-31
function getLatestRate() public view override returns (uint256) {
    return IAgEthRateProvider(agETHPriceOracle).getRate();
}
```

`MultiChainRateProvider.updateRate()` is `external payable` with only a `nonReentrant` guard — no `onlyOwner`, no role check, no rate-bounds check, no deviation limit, no cooldown:

```solidity
// contracts/cross-chain/MultiChainRateProvider.sol L108-113
function updateRate() external payable nonReentrant {
    uint256 latestRate = getLatestRate();
    rate = latestRate;
    lastUpdated = block.timestamp;
```

Whatever `getLatestRate()` returns is immediately written to `rate` and fanned out to every receiver via LayerZero:

```solidity
// contracts/cross-chain/MultiChainRateProvider.sol L115-134
for (uint256 i; i < rateReceiversLength;) {
    ...
    ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
        dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
    );
    ...
}
```

The exploit path is:
1. Attacker monitors `agETHPriceOracle.getRate()` off-chain.
2. During any block where the oracle reports a transient low value, attacker calls `updateRate{value: lzFee}()`.
3. The depressed rate is written to `rate` on the provider and broadcast via LayerZero to all registered `AGETHRateReceiver` contracts on destination chains.
4. Destination-chain consumers read the stale low rate until the next `updateRate()` call overwrites it.

No existing check prevents this: `nonReentrant` only guards against re-entrancy within a single call, not against permissionless invocation by an arbitrary caller.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

Cross-chain `AGETHRateReceiver` contracts on destination chains will store the attacker-supplied depressed rate and serve it to any consuming protocol until the next `updateRate()` call. During that window, any destination-chain protocol that prices agETH using the receiver's stored rate will underprice agETH, potentially allowing depositors to receive more agETH than the correct rate warrants. The provider contract itself does not lose funds. `AGETHPoolV3` on mainnet reads its own `agETHOracle` directly and is unaffected; the impact is confined to destination-chain consumers of `AGETHRateReceiver`.

## Likelihood Explanation

- No privileged role is required; any EOA or contract can call `updateRate()`.
- The attacker only needs ETH to cover LayerZero messaging fees.
- Oracle transient dips are a known occurrence during multi-block update windows for any on-chain rate provider; no oracle compromise is required.
- The attacker can monitor the oracle off-chain and submit the transaction in the same block as the dip.
- The attack is repeatable at low cost and can be sustained to keep the stale rate alive.

## Recommendation

1. Add an `onlyOwner` (or a dedicated `RATE_UPDATER_ROLE`) guard to `updateRate()` in `MultiChainRateProvider`, **or**
2. Enforce a maximum downward deviation limit: revert if `latestRate < rate * (BPS_BASE - MAX_DEVIATION_BPS) / BPS_BASE`, **or**
3. Enforce a minimum update interval (e.g., `require(block.timestamp >= lastUpdated + MIN_UPDATE_INTERVAL)`) so the rate cannot be pushed faster than the oracle's own update cadence.

Option 1 is the simplest and most robust fix.

The root cause is `contracts/cross-chain/MultiChainRateProvider.sol` L108: `function updateRate() external payable nonReentrant` — no caller restriction and no rate sanity check before committing and broadcasting.
