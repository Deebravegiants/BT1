### Title
Insufficient callback-output validation in `c_base_pke_adapter_t::encrypt` allows malicious base-PKE callback to produce and verify a PVE ciphertext with no actual encryption - (`src/cbmpc/c_api/pve_internal.h`)

### Summary
The ABI guard in `c_base_pke_adapter_t::encrypt` does not reject a callback return of `{nullptr, 0}`. This allows a malicious `cbmpc_pve_base_pke_t::encrypt` callback to silently produce empty base-PKE ciphertexts for all `kappa` rows. Because the PVE challenge hash `b` is computed over these empty ciphertexts, and the verify path re-runs the same callback and obtains the same empty values, both `cbmpc_pve_encrypt` and `cbmpc_pve_verify` return `SUCCESS` on a ciphertext that contains no actual encryption of the scalar shares.

### Finding Description
In `src/cbmpc/c_api/pve_internal.h`, `c_base_pke_adapter_t::encrypt` validates the callback output with:

```cpp
if (out.size < 0 || (out.size > 0 && !out.data)) {
    cbmpc_free(out.data);
    return E_FORMAT;
}
out_ct = coinbase::buf_t(out.data, out.size);
```

The condition `out.size > 0 && !out.data` evaluates to `false` when `out.size == 0`, so `{nullptr, 0}` passes the guard. `out_ct` is set to an empty `buf_t` and `CBMPC_SUCCESS` is returned to the caller.

In `ec_pve_t::encrypt` (`src/cbmpc/protocol/pve.cpp`), both `c0[i]` and `c1[i]` become empty for every row. The challenge hash `b` is computed over these empty ciphertexts and the EC points `X0[i]`, `X1[i]`. The stored `c[i]` fields are all empty.

In `ec_pve_t::verify`, the verifier re-encrypts using the same base PKE. With the same malicious callback, `c0[i]` is again empty, and `c1[i] = c[i]` is also empty. The EC points are reconstructed identically (same `r[i]` seeds, same `x_rows[i]` values). The recomputed hash `b_tag` equals `b`, so verify returns `SUCCESS`.

### Impact Explanation
A PVE ciphertext is produced and accepted by both encrypt and verify where all `kappa` base-PKE ciphertexts are empty buffers. The scalar shares `x0`, `x1` are not encrypted under the recipient's public key. The PVE scheme's core security property â€” that the ciphertext provably encrypts the scalar `x` â€” is violated. Any honest party using the same callback for verification will accept this invalid ciphertext.

### Likelihood Explanation
The attacker must supply a `cbmpc_pve_base_pke_t` struct with a malicious `encrypt` function pointer. This is a supported API entry point (`cbmpc_pve_encrypt` / `cbmpc_pve_verify` both accept a caller-supplied `base_pke`). The attack requires no special privileges, no threshold collusion, and no memory corruption â€” only a callback that returns `{nullptr, 0}` with `CBMPC_SUCCESS`.

### Recommendation
Strengthen the guard in `c_base_pke_adapter_t::encrypt` (and the symmetric guard in `c_base_pke_adapter_t::decrypt`) to also reject `{nullptr, 0}`:

```cpp
// Reject any output that is not a well-formed non-empty buffer.
if (out.size <= 0 || !out.data) {
    if (out.data) cbmpc_free(out.data);
    return E_FORMAT;
}
```

Apply the same fix to the analogous guard in `kem_encap_shim` / `kem_decap_shim` in `src/cbmpc/c_api/pve_base_pke.cpp`.
