### Title
Missing Minimum `BlockConfirmations` Enforcement Enables Double-Spend via Unconfirmed Bitcoin Transactions - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` contract method accepts a `BitcoinRpcRequest` with `confirmations: 0` — no minimum is enforced at the contract or node level. An unprivileged caller can obtain a valid threshold MPC signature attesting to a Bitcoin transaction that has only 1 confirmation (or even 0 actual confirmations if the RPC returns a block hash). If a bridge contract consuming this signature does not independently re-derive and check the confirmation count embedded in the signed payload hash, the attacker can claim bridge funds and then double-spend or wait for a reorg of the underlying Bitcoin transaction.

---

### Finding Description

**`BlockConfirmations` type — no minimum:** [1](#0-0) 

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI schema confirms `"minimum": 0.0` with no lower bound: [2](#0-1) 

**`verify_foreign_transaction` — no validation of the `confirmations` field:** [3](#0-2) 

The method stores the full `ForeignChainRpcRequest` (including `confirmations`) as the pending-request key and enqueues a yield, but performs zero validation on the numeric value of `confirmations`.

**Bitcoin inspector — threshold check passes trivially when threshold is 0:** [4](#0-3) 

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { … });
}
```

When `block_confirmations_threshold = 0`, the predicate `0 <= actual_confirmations` is always `true` for any transaction that has at least one confirmation (and therefore a `blockhash`). The MPC nodes proceed to extract the block hash and sign the payload.

**`respond_verify_foreign_tx` — only verifies the ECDSA signature, not the semantic content of `payload_hash`:** [5](#0-4) 

The contract verifies that the signature is valid over `response.payload_hash` under the MPC public key, then resolves all queued yields with the raw `response`. It does **not** re-derive the expected hash from the stored request to confirm that the nodes signed the correct payload. The `request` is used only as a lookup key.

**The signed payload commits to `confirmations: 0`:** [6](#0-5) 

`ForeignTxSignPayload::V1` borsh-serialises the full `ForeignChainRpcRequest` (including `confirmations`) together with the extracted values. So the hash does bind to `confirmations: 0` — but only a bridge contract that re-derives and checks this hash will notice.

---

### Impact Explanation

**Allowed impact category:** *High — forged foreign-chain verification / invalid bridge execution / double-spend conditions.*

Attack flow:

1. Attacker broadcasts a Bitcoin transaction; it receives 1 confirmation (enough to have a `blockhash`).
2. Attacker calls `verify_foreign_transaction` with `BitcoinRpcRequest { tx_id, confirmations: 0, extractors: [BlockHash] }`.
3. MPC nodes evaluate `0 <= 1` → pass; extract the block hash; threshold-sign `SHA-256(borsh(ForeignTxSignPayload { request: {…, confirmations: 0}, values: [block_hash] }))`.
4. Attacker receives a valid threshold signature over a payload that commits to `confirmations: 0`.
5. Attacker submits this signature to a bridge contract on NEAR.
6. If the bridge contract does not re-derive the expected payload hash with its own required confirmation count (e.g., 6), it accepts the signature and releases NEAR-side funds.
7. Attacker double-spends the Bitcoin transaction (or waits for a natural reorg at 1 confirmation depth).

The bridge contract receives a cryptographically valid MPC signature — the forgery is semantic, not cryptographic. The MPC network has faithfully signed exactly what it was asked to sign, but what it was asked to sign provides no meaningful finality guarantee.

---

### Likelihood Explanation

**Medium.** The primary use case of `verify_foreign_transaction` is the Omnibridge inbound flow (Bitcoin → NEAR). Any bridge contract that:

- calls `verify_foreign_transaction` for Bitcoin, and
- does not independently reconstruct the expected `ForeignTxSignPayload` with its own required `confirmations` value and compare hashes

is directly exploitable. The SDK's `ForeignChainSignatureVerifier` does perform this check if used correctly: [7](#0-6) 

However, the MPC contract itself provides no safety net. A bridge developer who trusts the MPC signature without re-deriving the hash — a natural mistake given that the MPC network is supposed to be the verifier — will be vulnerable. Bitcoin reorg probability at 1 confirmation is non-negligible (~0.1–1% historically), and an attacker can deliberately engineer a double-spend.

---

### Recommendation

Enforce a protocol-level minimum on `BlockConfirmations` inside `verify_foreign_transaction` before enqueuing the request:

```rust
// In verify_foreign_transaction, after parsing the request:
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    const MIN_BITCOIN_CONFIRMATIONS: u64 = 1; // or a configurable on-chain parameter
    if btc_req.confirmations.0 < MIN_BITCOIN_CONFIRMATIONS {
        env::panic_str("Bitcoin confirmations must be at least 1");
    }
}
```

A stronger fix would make the minimum configurable per-chain in the on-chain `ForeignChainsMetadata` so that governance can raise it (e.g., to 6 for Bitcoin) without a contract upgrade. This mirrors the recommendation in the reference report: enforce the minimum at the protocol layer so that downstream consumers cannot accidentally bypass it.

```
block_confirmations_threshold = BlockConfirmations(0)
transaction_block_confirmation = BlockConfirmations(1)   // from RPC
0 <= 1  →  true  →  no error raised
``` [8](#0-7) 

The nodes proceed to `verify_block_is_canonical`, extract the block hash, and sign. The returned `VerifyForeignTransactionResponse` carries a valid threshold ECDSA signature over a payload that encodes `confirmations: 0`. Any bridge contract that does not re-derive the expected hash with `confirmations ≥ 6` (or whatever its security policy requires) will accept this response and release funds — enabling a double-spend once the 1-confirmation block is reorganised.

            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
