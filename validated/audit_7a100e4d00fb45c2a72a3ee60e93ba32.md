### Title
`chain_gas_consumed` Hardcodes `Felt::ZERO` Instead of Actual `l2_gas` in Receipt Commitment ‚Äî (`crates/starknet_api/src/block_hash/receipt_commitment.rs`)

### Summary

`chain_gas_consumed` in `receipt_commitment.rs` always chains `Felt::ZERO` for the L2 gas consumed field instead of reading `gas_consumed.l2_gas` from the `GasVector` argument. This is the direct Sequencer analog of the external report's bug: a hardcoded `0` is passed to a hash function instead of the actual discriminating field, causing different execution outputs to produce the same receipt leaf hash and therefore the same `receipt_commitment`, which propagates into the `PartialBlockHash` / `ProposalCommitment`.

### Finding Description

`chain_gas_consumed` is the function that serializes a transaction's gas consumption into the Poseidon receipt hash:

```rust
// crates/starknet_api/src/block_hash/receipt_commitment.rs  lines 86-91
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO)               // ‚Üê hardcoded, ignores gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

`GasVector` carries three independent fields:

```rust
// crates/starknet_api/src/execution_resources.rs  lines 106-111
pub struct GasVector {
    pub l1_gas: GasAmount,
    pub l1_data_gas: GasAmount,
    #[serde(default)]
    pub l2_gas: GasAmount,
}
```

For V3 (`AllResources`) transactions the blockifier computes a non-zero `l2_gas` value. The regression test in `crates/blockifier/src/transaction/transactions_test.rs` shows values such as `l2_gas: GasAmount(653175)` and `l2_gas: GasAmount(602275)` for ordinary invoke transactions. `output_for_hashing()` on a `TransactionExecutionInfo` returns a `TransactionOutputForHash` whose `gas_consumed: GasVector` carries this real value. That struct is then passed through:

```
prepare_txs_hashing_data
  ‚Üí calculate_block_commitments (receipt_elements built via ReceiptElement::from)
    ‚Üí calculate_receipt_commitment
      ‚Üí calculate_receipt_hash
        ‚Üí chain_gas_consumed   ‚Üê always chains Felt::ZERO for l2_gas
```

Because `l2_gas` is silently dropped, two transactions that differ **only** in their L2 gas consumption produce identical receipt leaf hashes. The wrong `receipt_commitment` is then embedded in `BlockHeaderCommitments`, which feeds `PartialBlockHashComponents::new` and ultimately `PartialBlockHash::from_partial_block_hash_components` ‚Äî the value compared as `ProposalCommitment` during consensus.

### Impact Explanation

**Critical ‚Äî Wrong receipt commitment propagates into the block hash and ProposalCommitment.**

Any block that contains at least one V3 transaction (the current standard) will have a `receipt_commitment` that does not reflect the actual L2 gas consumed. The `PartialBlockHash` / `ProposalCommitment` derived from it is therefore incorrect. External verifiers (provers, light clients, sync peers) that compute the receipt commitment from the actual execution data will obtain a different value, causing proof verification failures and block hash mismatches.

### Likelihood Explanation

Every V3 invoke/declare/deploy-account transaction processed under `GasVectorComputationMode::All` produces a non-zero `l2_gas` in its `GasVector`. This is the default mode for all current Starknet V3 transactions. The bug is therefore triggered by every block that contains at least one V3 transaction, which is every block in normal operation.

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual `l2_gas` field:

```diff
 fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
     hash_chain
-        .chain(&Felt::ZERO) // L2 gas consumed
+        .chain(&gas_consumed.l2_gas.into())
         .chain(&gas_consumed.l1_gas.into())
         .chain(&gas_consumed.l1_data_gas.into())
 }
```

### Proof of Concept

1. Execute any V3 invoke transaction; the blockifier returns `gas_consumed.l2_gas > 0` (e.g. `GasAmount(653175)` as shown in the regression test).
2. `prepare_txs_hashing_data` wraps this in `TransactionHashingData`.
3. `calculate_block_commitments` calls `ReceiptElement::from` and then `calculate_receipt_hash` ‚Üí `chain_gas_consumed`.
4. `chain_gas_consumed` chains `Felt::ZERO` instead of `GasAmount(653175).into()`.
5. A second transaction with a different `l2_gas` value (e.g. `GasAmount(602275)`) produces the **same** receipt leaf hash as the first if all other fields are equal.
6. The resulting `receipt_commitment` differs from the value any external party would compute using the actual `l2_gas`, causing `PartialBlockHash` / `ProposalCommitment` mismatches. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

/// Hash of [`PartialBlockHashComponents`] only (no state root or parent hash).
#[derive(Clone, Copy, Debug, Default, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub struct PartialBlockHash(pub StarkHash);

impl PartialBlockHash {
    // TODO(Ariel): Use parent_partial_block_hash instead of zero.
    const GLOBAL_ROOT_FOR_PARTIAL_BLOCK_HASH: GlobalRoot = GlobalRoot(Felt::ZERO);
    const PARENT_HASH_FOR_PARTIAL_BLOCK_HASH: BlockHash = BlockHash(Felt::ZERO);

    /// Hash of [`PartialBlockHashComponents`].
    /// Uses the same formula as [`calculate_block_hash`] with the fixed constants above for the
    /// state root and parent hash.
    pub fn from_partial_block_hash_components(
        partial_block_hash_components: &PartialBlockHashComponents,
    ) -> StarknetApiResult<Self> {
        let block_hash = calculate_block_hash(
            partial_block_hash_components,
            Self::GLOBAL_ROOT_FOR_PARTIAL_BLOCK_HASH,
            Self::PARENT_HASH_FOR_PARTIAL_BLOCK_HASH,
        )?;
        Ok(Self(block_hash.0))
    }
