### Title
Unbounded Iteration Over DRep Delegators on `ConwayUnRegDRep` Exceeds Ledger Validation Limits — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

When a DRep unregisters via a `ConwayUnRegDRep` certificate, the `GOVCERT` rule iterates over the entire unbounded set of stake credentials delegated to that DRep (`drepDelegs`) to clear their delegation pointers. A second unbounded sweep over all active governance proposals also occurs in the `GOV` rule when the same transaction contains voting procedures. Both loops scale with attacker-controlled state and have no cap, causing ledger rule evaluation work to grow without bound.

---

### Finding Description

**Root cause 1 — `clearDRepDelegations` in `GOVCERT`** [1](#0-0) 

```haskell
let
  certState' =
    certState & certVStateL . vsDRepsL %~ Map.delete cred
  clearDRepDelegations delegs accountsMap =
    foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
pure $
  case mDRepState of
    Nothing -> certState'
    Just dRepState ->
      certState'
        & certDStateL . accountsL . accountsMapL
          %~ clearDRepDelegations (drepDelegs dRepState)
```

`drepDelegs` is typed as `Set (Credential Staking)` with no upper bound: [2](#0-1) 

The `foldr` iterates once per delegator, performing a `Map.adjust` (O(log M)) on the accounts map for each. Total cost is **O(N · log M)** where N = `|drepDelegs|` and M = total registered accounts. N is entirely attacker-controlled.

**Root cause 2 — `cleanupProposalVotes` in `GOV`** [3](#0-2) 

```haskell
cleanupProposalVotes
  | Set.null unregisteredDReps = id
  | otherwise =
      let cleanupVoters gas =
            gas & gasDRepVotesL %~ (`Map.withoutKeys` unregisteredDReps)
       in mapProposals cleanupVoters
```

`mapProposals` applies `cleanupVoters` to every entry in the `OMap` of active proposals: [4](#0-3) 

This fires whenever a transaction contains at least one `UnRegDRepTxCert` certificate alongside any voting procedures. The sweep is **O(P)** where P = total active proposals, which is also unbounded (no protocol parameter caps the total number of live proposals).

---

### Impact Explanation

Cardano Ledger has no per-transaction execution budget for ledger-rule evaluation (unlike Plutus scripts, which are bounded by `maxTxExUnits`). Both loops execute synchronously inside `applyTx` / block application on every validating node. An attacker who has pre-populated a DRep with N delegators can submit a single `ConwayUnRegDRep` transaction that forces every node to perform O(N · log M) map operations before the block can be accepted. With a sufficiently large N (tens of thousands of delegators, each costing only `ppKeyDeposit` ≈ 2 ADA), block validation time grows to the point where it exceeds the implicit wall-clock budget available to a slot leader, causing the leader to miss its slot or causing slower nodes to lag behind faster ones. This constitutes **attacker-controlled transactions exceeding intended validation limits** (Medium impact).

---

### Likelihood Explanation

The attack is economically feasible. The attacker:
1. Registers one DRep (deposit: `ppDRepDeposit`, refunded on unregistration).
2. Registers N stake credentials across many transactions (cost: N × `ppKeyDeposit`; these deposits remain locked but are not destroyed).
3. Delegates each credential to the DRep (no additional deposit).
4. Submits the `ConwayUnRegDRep` transaction, recovering the DRep deposit.

The net permanent cost is N × `ppKeyDeposit`. At current mainnet parameters (~2 ADA per key), 50,000 delegators costs ~100,000 ADA — significant but within reach of a motivated attacker. The `cleanupProposalVotes` vector is cheaper: the attacker need not own the proposals; any epoch with many live proposals (submitted by ordinary users) amplifies the sweep for free.

---

### Recommendation

1. **Cap `drepDelegs` size**: Introduce a protocol parameter `ppMaxDRepDelegators` and enforce it in the `DELEG` rule when adding a new DRep delegation. This mirrors the existing `ppPoolDeposit` / `ppEMax` pattern used to bound pool-related state growth.

2. **Lazy / incremental delegation cleanup**: Instead of eagerly clearing all delegator pointers on DRep unregistration, mark the DRep as unregistered and lazily clear individual delegator pointers the next time each delegator's account is touched (similar to the pulsing reward-update design already used in `PulsingReward.hs`).

3. **Decouple `cleanupProposalVotes` from per-transaction `GOV` processing**: Move the vote-cleanup sweep to the epoch-boundary `EPOCH` rule where it can be amortised, rather than executing it inline for every transaction that happens to contain an `UnRegDRepTxCert`.

### Proof of Concept

**Setup (spread across many transactions):**
```
-- Register attacker DRep
tx_reg_drep: [ConwayRegDRep attackerCred deposit SNothing]

-- Register 50,000 stake credentials and delegate each to attackerCred
for i in 1..50000:
  tx_i: [RegDepositDelegTxCert stakeCred_i
           (DelegVote (DRepCredential attackerCred))
           ppKeyDeposit]
```

**Trigger (single transaction):**
```
tx_attack:
  certificates: [ConwayUnRegDRep attackerCred ppDRepDeposit]
  -- optional: include any VotingProcedures to also trigger cleanupProposalVotes
```

When `tx_attack` is applied, `conwayGovCertTransition` reaches the `ConwayUnRegDRep` branch and calls `clearDRepDelegations (drepDelegs dRepState) accountsMap` where `|drepDelegs| = 50,000`. Every validating node must complete 50,000 `Map.adjust` calls before the transaction is accepted, with no execution-unit gate to reject it early. [5](#0-4) [2](#0-1) [6](#0-5) [4](#0-3)
