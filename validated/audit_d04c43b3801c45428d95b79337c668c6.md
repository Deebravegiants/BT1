### Title
`deployToken` Signed Payload Lacks Chain-ID Binding, Enabling Cross-Chain Signature Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `aptos/sources/bridge_types.move`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The NEAR MPC-signed `MetadataPayload` used by `deployToken` / `deploy_token` across all destination chains (EVM, Starknet, Aptos, Solana) does not include a destination chain identifier. Because the same NEAR MPC key (`nearBridgeDerivedAddress`) is used on every chain, a valid signature obtained from one chain's `deployToken` call can be replayed verbatim on any other chain, deploying the bridged token there without NEAR's knowledge or authorization for that chain.

---

### Finding Description

Every `finTransfer` / `fin_transfer` implementation correctly binds the signed message to a specific destination chain by embedding the chain ID in the Borsh-encoded payload:

- EVM embeds `omniBridgeChainId` twice (before `tokenAddress` and before `recipient`). [1](#0-0) 
- Starknet's `TransferMessagePayload::to_borsh` takes `chain_id` as a parameter and encodes it. [2](#0-1) 
- Aptos's `transfer_message_to_borsh` takes `chain_id` and encodes it. [3](#0-2) 
- Solana's `FinalizeTransferPayload::serialize_for_near` writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` before both the token and recipient fields. [4](#0-3) 

However, the `MetadataPayload` Borsh encoding used by `deployToken` on every chain is simply:

```
[PayloadType::Metadata (0x01)] | [token] | [name] | [symbol] | [decimals]
```

No chain ID is included anywhere in this encoding:

- EVM `deployToken`: [5](#0-4) 
- Starknet `MetadataPayloadImpl::to_borsh`: [6](#0-5) 
- Aptos `metadata_to_borsh`: [7](#0-6) 
- Solana `DeployTokenPayload::serialize_for_near`: [8](#0-7) 

Because the `nearBridgeDerivedAddress` / `derived_near_bridge_address` / `omni_bridge_derived_address` is the same NEAR MPC-derived Ethereum key across all chains, the signature check `ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress` passes on every chain for the same `(token, name, symbol, decimals)` tuple. [9](#0-8) 

---

### Impact Explanation

An attacker who observes a valid `deployToken` transaction on Chain A (e.g., EVM Ethereum) can extract the `signatureData` and `MetadataPayload` and immediately replay them on Chain B (e.g., Starknet, Aptos, or Solana) before the legitimate relayer does so.

Consequences:

1. **Permanently blocked legitimate deployment**: Once the token is deployed on Chain B via replay, the `ERR_TOKEN_ALREADY_DEPLOYED` / `"ERR_TOKEN_EXIST"` guard prevents any future deployment for that token on Chain B. [10](#0-9) [11](#0-10) 

2. **NEAR-side mapping never established**: The NEAR bridge only records a token's address on a destination chain after it initiates and confirms the deployment through its own flow. A replay-deployed token on Chain B is unknown to NEAR, so `fin_transfer` messages for that token on Chain B will never be signed by the MPC — all user funds bridging that token to Chain B are permanently unclaimable.

3. **Decimal normalization mismatch**: Aptos silently clamps decimals to 8, while EVM and Starknet clamp to 18. A signature for a 24-decimal NEAR token intended for EVM (normalized to 18) replayed on Aptos produces a token with 8 decimals, permanently corrupting the token's precision on that chain. [12](#0-11) 

---

### Likelihood Explanation

- `deployToken` is a fully permissionless, public entry point on all chains. [13](#0-12) 
- The signature is broadcast on-chain the moment the legitimate relayer submits it to Chain A; it is immediately visible to any observer.
- The attacker only needs to copy the signature and payload and submit them to Chain B before the legitimate relayer does — a straightforward front-run requiring no privileged access, no leaked keys, and no collusion.
- The Omni Bridge is deployed on multiple chains simultaneously (EVM, Starknet, Aptos, Solana), all sharing the same MPC signer, making every new `deployToken` event on any chain an opportunity for cross-chain replay on all others.

---

### Recommendation

Include the destination chain ID in the `MetadataPayload` Borsh encoding, mirroring the pattern already used by `TransferMessagePayload`. Concretely, add the chain ID byte immediately after the `PayloadType::Metadata` tag (or before the token field) in all four implementations:

```
[0x01] | [chain_id] | [token] | [name] | [symbol] | [decimals]
```

The NEAR MPC must sign a distinct payload per destination chain. This is consistent with how `finTransfer` already handles chain binding. [14](#0-13) 

---

### Proof of Concept

1. NEAR MPC signs `MetadataPayload { token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6 }` for EVM Ethereum (chain ID `2`). Relayer submits `deployToken(sig, payload)` to EVM — transaction is public.

2. Attacker copies `sig` and `payload` from the EVM transaction and calls `deploy_token(sig, payload)` on Starknet (chain ID `4`) before the legitimate relayer does.

3. Starknet's `_verify_borsh_signature` hashes `[0x01 | "usdc.near" | "USD Coin" | "USDC" | 0x06]` — identical bytes to what EVM hashed — and recovers the same `omni_bridge_derived_address`. Signature passes. [15](#0-14) 

4. USDC is now deployed on Starknet at an attacker-front-run address. NEAR has no record of this Starknet token address. The legitimate Starknet deployment fails with `ERR_TOKEN_ALREADY_DEPLOYED`. All future USDC bridge transfers to Starknet are permanently unclaimable.
