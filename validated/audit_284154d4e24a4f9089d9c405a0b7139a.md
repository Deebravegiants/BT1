### Title
Single Hot Credential Vote Counted Multiple Times via Multiple Cold Credential Authorizations — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

`committeeAcceptedRatio` in the Conway `RATIFY` rule iterates over committee cold credentials and looks up each one's authorized hot credential, then counts that hot credential's vote. Because there is no deduplication of hot credentials, if N cold credentials are all authorized to the same hot credential, a single vote cast by that hot credential is counted N times. This is the direct analog of the external report's "validator votes multiple times" pattern: one actor's attestation inflates the tally beyond its legitimate weight.

---

### Finding Description

`committeeAcceptedRatio` accumulates yes/total counts by folding over `members :: Map (Credential ColdCommitteeRole) EpochNo`:

```haskell
accumVotes (!yes, !tot) member expiry
  | currentEpoch > expiry = (yes, tot)
  | otherwise =
      case Map.lookup member (csCommitteeCreds committeeState) of
        Just (CommitteeHotCredential hotKey) ->
          case Map.lookup hotKey votes of
            Just VoteYes -> (yes + 1, tot + 1)
            ...
``` [1](#0-0) 

For each cold credential, the function resolves its hot credential and looks up that hot credential's vote in `gasCommitteeVotes :: Map (Credential HotCommitteeRole) Vote`. If two distinct cold credentials both resolve to the same hot credential, the same `VoteYes` entry is found twice and `(yes, tot)` is incremented twice.

The `AuthCommitteeHotKeyTxCert` certificate (`ConwayAuthCommitteeHotKey coldCred hotCred`) maps a cold credential to a hot credential. There is no ledger rule preventing multiple cold credentials from being authorized to the same hot credential. The `CommitteeState` is `Map (Credential ColdCommitteeRole) CommitteeAuthorization`, so each cold credential independently stores its authorization. [2](#0-1) 

The `VotingProcedures` structure is `Map Voter (Map GovActionId (VotingProcedure era))`, keyed on `Voter` (which includes `CommitteeVoter (Credential HotCommitteeRole)`). A hot credential can only appear once in this map per governance action, so the vote itself is not duplicated in storage — the duplication occurs purely in the counting loop of `committeeAcceptedRatio`. [3](#0-2) 

The existing test suite explicitly confirms this behavior:

```
it "Many CC Cold Credentials map to the same Hot Credential act as many votes"
```

The test registers all committee cold credentials to the same hot credential, casts one vote, and observes that the proposal is ratified — even though it would not have been ratified with a single vote counted once. [4](#0-3) 

---

### Impact Explanation

An attacker who controls N cold credentials all elected to the committee can:

1. Submit N `AuthCommitteeHotKeyTxCert` certificates, all pointing to the same hot credential.
2. Cast a single vote with that hot credential.
3. Have that vote counted N times in `committeeAcceptedRatio`, inflating the yes-vote numerator and the total denominator by N.

With a committee threshold of `t`, an attacker holding `k` cold credentials (all mapped to one hot key) out of a total of `m` cold credentials can achieve an accepted ratio of `k / m` from a single vote, rather than the legitimate `1 / m`. If `k / m >= t`, the committee threshold is crossed unilaterally.

This allows unauthorized enactment of any governance action requiring committee approval: hard-fork initiations, treasury withdrawals, protocol-parameter changes, constitution updates, and committee membership changes.

**Matched impact**: *Critical — Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.*

---

### Likelihood Explanation

The precondition is controlling multiple cold credentials that are simultaneously elected to the committee. This is achievable without requiring governance majority at the time of the attack:

- **Genesis committee**: The initial committee is configured at genesis. A single operator who controls multiple genesis cold credentials (e.g., a founding entity holding several seats) can immediately exploit this without any subsequent governance action.
- **Previously enacted UpdateCommittee**: Any past governance action that added multiple cold credentials controlled by the same entity creates the precondition. Once the cold credentials are in the committee, the attack requires only submitting `AuthCommitteeHotKeyTxCert` certificates — ordinary transactions signed by the cold credential keys — and then casting one vote.

The `AuthCommitteeHotKeyTxCert` certificate is an unprivileged transaction: any cold credential holder can submit it at any time. No special role, governance majority, or privileged access is needed after the cold credentials are in the committee. [5](#0-4) 

---

### Recommendation

In `committeeAcceptedRatio`, deduplicate hot credentials before counting votes. Instead of iterating over cold credentials and re-resolving the same hot credential multiple times, first build the set of distinct hot credentials (one per cold credential), then count votes over that deduplicated set:

```haskell
-- Collect the unique hot credentials for active, non-resigned, non-expired members
activeHotKeys :: Set (Credential HotCommitteeRole)
activeHotKeys = Set.fromList
  [ hotKey
  | (member, expiry) <- Map.toList members
  , currentEpoch <= expiry
  , Just (CommitteeHotCredential hotKey) <- [Map.lookup member (csCommitteeCreds committeeState)]
  ]
```

Then fold over `activeHotKeys` rather than over `members`. This ensures that a single hot credential's vote is counted at most once, regardless of how many cold credentials are authorized to it.

Alternatively, enforce at the certificate-validation level (`CERT`/`GOVCERT` rules) that a hot credential may not be authorized by more than one cold credential simultaneously.

---

### Proof of Concept

The existing test at line 52–68 of `RatifySpec.hs` is a direct proof of concept:

1. One committee member registers a hot credential and votes Yes on a parameter-change proposal.
2. The proposal does **not** pass (the single vote is insufficient given the committee size and threshold).
3. All remaining committee cold credentials are then authorized to the **same** hot credential via `registerCommitteeHotKeys (pure hotCred)`.
4. Two epochs later, the proposal **is** enacted — the single vote is now counted once per cold credential, crossing the threshold. [4](#0-3) 

The attacker-controlled entry path is:
- Submit `AuthCommitteeHotKeyTxCert coldCredN hotCredAttacker` for each cold credential `coldCredN` the attacker controls.
- Submit one `VotingProcedures` transaction with `CommitteeVoter hotCredAttacker` voting Yes on the target governance action.
- `committeeAcceptedRatio` counts the Yes vote once per cold credential, inflating the ratio. [6](#0-5)

committeeAcceptedRatio ::
  forall era.
  Map (Credential ColdCommitteeRole) EpochNo ->
  Map (Credential HotCommitteeRole) Vote ->
  CommitteeState era ->
  EpochNo ->
  Rational
committeeAcceptedRatio members votes committeeState currentEpoch =
  yesVotes %? totalExcludingAbstain
  where
    accumVotes ::
      (Integer, Integer) ->
      Credential ColdCommitteeRole ->
      EpochNo ->
      (Integer, Integer)
    accumVotes (!yes, !tot) member expiry
      | currentEpoch > expiry = (yes, tot) -- member is expired, vote "abstain" (don't count it)
      | otherwise =
          case Map.lookup member (csCommitteeCreds committeeState) of
            Nothing -> (yes, tot) -- member is not registered, vote "abstain"
            Just (CommitteeMemberResigned _) -> (yes, tot) -- member has resigned, vote "abstain"
            Just (CommitteeHotCredential hotKey) ->
              case Map.lookup hotKey votes of
                Nothing -> (yes, tot + 1) -- member hasn't voted, vote "no"
                Just Abstain -> (yes, tot) -- member voted "abstain"
                Just VoteNo -> (yes, tot + 1) -- member voted "no"
                Just VoteYes -> (yes + 1, tot + 1) -- member voted "yes"
    (yesVotes, totalExcludingAbstain) = Map.foldlWithKey' accumVotes (0, 0) members
```
