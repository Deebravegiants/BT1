### Title
`estimate_minimal_gas_vector` omits compiled-class-hash DA cost for Declare V2/V3, causing pre-validation to accept transactions with insufficient resource bounds — (File: `crates/blockifier/src/fee/gas_usage.rs`)

### Summary

`estimate_minimal_gas_vector` hard-codes `n_compiled_class_hash_updates: 0` for every `Declare` transaction, even though Declare V2/V3 transactions write one compiled class hash entry to state (`n_compiled_class_hash_updates: 1`). The function is used in `check_fee_bounds` (pre-validation) to enforce a minimum resource bound. Because the estimate is too low, the gateway and blockifier pre-validation accept Declare V2/V3 transactions whose resource bounds are below the actual execution cost, causing those transactions to revert after admission.

### Finding Description

In `estimate_minimal_gas_vector`, the `StateChangesCount` for a `Declare` transaction is:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,
    n_modified_contracts: 1,
},
``` [1](#0-0) 

The in-code TODO explicitly acknowledges the value should be `1`. The actual execution path for Declare V2/V3 calls `try_declare(self, state, class_hash, Some(*compiled_class_hash))`, which writes the compiled class hash to state, producing `n_compiled_class_hash_updates: 1` in the real `StateChanges`. This is confirmed by the test helper:

```rust
} else if version == TransactionVersion::TWO || version == TransactionVersion::THREE {
    StateChangesCount {
        n_storage_updates: 1,
        n_modified_contracts: 1,
        n_compiled_class_hash_updates: 1, // Also set compiled class hash.
        ..StateChangesCount::default()
    }
``` [2](#0-1) 

`get_onchain_data_segment_length` adds `n_compiled_class_hash_updates * 2` felts to the DA segment: [3](#0-2) 

With `n_compiled_class_hash_updates: 0` instead of `1`, the estimate is missing:
- **KZG DA mode**: 2 × `DATA_GAS_PER_FIELD_ELEMENT` blob gas units from `get_da_gas_cost`
- **non-KZG mode**: 2 × `SHARP_GAS_PER_DA_WORD` L1 gas units

The underestimated vector is then used in `check_fee_bounds` to enforce the minimum resource bound:

```rust
let minimal_gas_amount_vector = estimate_minimal_gas_vector(
    &tx_context.block_context,
    self,
    &tx_context.get_gas_vector_computation_mode(),
);
``` [4](#0-3) 

For `AllResources` (V3), the check compares `minimal_gas_amount_vector.l1_data_gas` against `l1_data_gas_resource_bounds.max_amount`: [5](#0-4) 

Because the minimum is underestimated, a Declare V3 transaction with `l1_data_gas.max_amount` set to the underestimated minimum passes pre-validation, but the actual execution charges 2 more blob gas units, causing the transaction to revert with `MaxGasAmountExceeded`.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid Declare V2/V3 transactions.**

Any Declare V2/V3 transaction whose `l1_data_gas` (KZG mode) or `l1_gas` (non-KZG mode) resource bound is set to exactly the underestimated minimum passes `check_fee_bounds` in `perform_pre_validation_stage`, is admitted by the gateway's stateful validator, enters the mempool, and is included in a block. During execution the actual DA cost exceeds the user's bound, the transaction reverts, the nonce is incremented, and a fee is charged. The gateway invariant "reject transactions that cannot succeed" is broken for this class of transactions.

### Likelihood Explanation

Any user or wallet that calls `estimate_minimal_gas_vector` (or the equivalent RPC `starknet_estimateFee` path that ultimately relies on the same undercount) to set their resource bounds for a Declare V2/V3 transaction will receive a value that is 2 blob-gas units too low. Wallets that set bounds tightly (e.g., exactly at the estimated minimum) will produce transactions that pass admission but revert on-chain. The trigger is unprivileged: any account can submit a Declare V2/V3 transaction.

### Recommendation

Change `n_compiled_class_hash_updates` from `0` to `1` for `Transaction::Declare(_)` in `estimate_minimal_gas_vector`, as the TODO comment already states:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    n_compiled_class_hash_updates: 1, // Declare V2/V3 writes compiled_class_hash.
    n_modified_contracts: 1,
},
```

Additionally, resolve the second TODO `// TODO(Yoni): BLOCKIFIER-RESET: reuse TransactionReceipt code.` by unifying the minimal-gas estimation with the actual receipt computation path to prevent future divergence. [6](#0-5) 

### Proof of Concept

1. Submit a Declare V3 transaction (Cairo 1 class) with KZG DA enabled.
2. Set `l1_data_gas.max_amount` to exactly `estimate_minimal_gas_vector(...).l1_data_gas` (the underestimated value).
3. Observe that `check_fee_bounds` passes (pre-validation succeeds, gateway admits the transaction).
4. Observe that during blockifier execution the actual `l1_data_gas` used is `estimate + 2` (the 2 felts for the compiled class hash entry), triggering `FeeCheckError::MaxGasAmountExceeded { resource: L1DataGas, ... }` and reverting the transaction.

The discrepancy is exactly `get_onchain_data_segment_length({n_compiled_class_hash_updates: 1}) - get_onchain_data_segment_length({n_compiled_class_hash_updates: 0}) = 2` felts, each costing `DATA_GAS_PER_FIELD_ELEMENT` blob gas. [7](#0-6) [8](#0-7)

pub fn get_onchain_data_segment_length(state_changes_count: &StateChangesCount) -> usize {
    // For each newly modified contract:
    // contract address (1 word).
    // + 1 word with the following info: A flag indicating whether the class hash was updated, the
    // number of entry updates, and the new nonce.
    let mut onchain_data_segment_length = state_changes_count.n_modified_contracts * 2;
    // For each class updated (through a deploy or a class replacement).
    onchain_data_segment_length +=
        state_changes_count.n_class_hash_updates * constants::CLASS_UPDATE_SIZE;
    // For each modified storage cell: key, new value.
    onchain_data_segment_length += state_changes_count.n_storage_updates * 2;
    // For each compiled class updated (through declare): class_hash, compiled_class_hash
    onchain_data_segment_length += state_changes_count.n_compiled_class_hash_updates * 2;

    onchain_data_segment_length

/// Returns the gas cost of data availability on L1.
pub fn get_da_gas_cost(state_changes_count: &StateChangesCount, use_kzg_da: bool) -> GasVector {
    let onchain_data_segment_length = get_onchain_data_segment_length(state_changes_count);

    let (l1_gas, blob_gas) = if use_kzg_da {
        (
            0_u8.into(),
            u64_from_usize(
                onchain_data_segment_length * eth_gas_constants::DATA_GAS_PER_FIELD_ELEMENT,
            )
            .into(),
        )
    } else {
        // TODO(Yoni, 1/5/2024): count the exact amount of nonzero bytes for each DA entry.
        let naive_cost = onchain_data_segment_length * eth_gas_constants::SHARP_GAS_PER_DA_WORD;

        // For each modified contract, the expected non-zeros bytes in the second word are:
        // 1 bytes for class hash flag; 2 for number of storage updates (up to 64K);
        // 3 for nonce update (up to 16M).
        let modified_contract_cost = eth_gas_constants::get_calldata_word_cost(1 + 2 + 3);
        let modified_contract_discount =
            eth_gas_constants::GAS_PER_MEMORY_WORD - modified_contract_cost;
        let mut discount = state_changes_count.n_modified_contracts * modified_contract_discount;

        // Up to balance of 8*(10**10) ETH.
        let fee_balance_value_cost = eth_gas_constants::get_calldata_word_cost(12);
        discount += eth_gas_constants::GAS_PER_MEMORY_WORD - fee_balance_value_cost;

        // Cost must be non-negative after discount.
        let gas = naive_cost.saturating_sub(discount);

        (u64_from_usize(gas).into(), 0_u8.into())
    };

    GasVector { l1_gas, l1_data_gas: blob_gas, ..Default::default() }
}
