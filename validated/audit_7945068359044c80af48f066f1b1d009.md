### Title
`respond_ckd` Skips Cryptographic Output Verification for `AppPublicKey` Variant, Allowing a Single Malicious Participant to Deliver a Fabricated Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` contract method contains an explicit, unconditional bypass of the CKD output check for the `CKDAppPublicKey::AppPublicKey` variant. While `AppPublicKeyPV` requests are protected by a pairing-based `ckd_output_check`, the `AppPublicKey` branch is an empty match arm with no verification whatsoever. Any single malicious attested participant can race the legitimate leader and call `respond_ckd` with an arbitrary fabricated `CKDResponse` for any pending `AppPublicKey` request. The contract accepts it unconditionally, resolves all queued yields with the fabricated output, and delivers it to every caller who submitted that request.

### Finding Description

In `respond_ckd` at lines 675–682 of `crates/contract/src/lib.rs`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` (defined in `crates/contract/src/primitives/ckd.rs` lines 80–102) enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, cryptographically binding the response to the network's master key and the app's public key. For `AppPublicKey`, the arm is empty — the contract performs zero verification on `big_y` and `big_c`.

The authorization guard `assert_caller_is_attested_participant_and_protocol_active` (line 666) only requires the caller to be any attested participant; it does not restrict `respond_ckd` to the elected leader. Therefore, a malicious attested participant can:

1. Observe a pending `AppPublicKey` CKD request in the contract's `pending_ckd_requests` map.
2. Immediately call `respond_ckd` with arbitrary `big_y` and `big_c` values — before the legitimate leader completes the threshold protocol.
3. `pending_requests::resolve_yields_for` (line 684) drains **all** queued yields for that request key, delivering the fabricated output to every caller who submitted the same request.

The legitimate leader's subsequent `respond_ckd` call then finds no pending yields and silently succeeds with no effect.

This is structurally identical to the reported `SharesCooldown` bypass: a safety invariant (output correctness) is enforced on one code path (`AppPublicKeyPV`) and explicitly absent on another (`AppPublicKey`), and the bypass is reachable by a single unprivileged-within-threshold actor.

### Impact Explanation

The caller receives a fabricated `(big_y, big_c)` pair. Using their private scalar `a`, they compute `derived_key = big_c' - a·big_y'`, which is an attacker-chosen value rather than the legitimate `hash_point · msk`. Any assets the caller subsequently deposits to the address derived from this wrong key are inaccessible via the expected derivation path. Because the attacker chose `big_c'` and `big_y'` without knowing `a`, neither party can recover those assets — constituting permanent freezing of funds. This satisfies the Critical impact criterion: "Unauthorized confidential key derivation output without the required participant authorization" and "permanent freezing of funds."

### Likelihood Explanation

The attacker must be an attested participant (a role reachable without threshold-level collusion — a single Byzantine node suffices). No leader election is required; `respond_ckd` imposes no leader-only restriction. The attacker simply monitors the NEAR chain for `AppPublicKey` CKD requests and races to submit a fabricated response. The race window is the time between request submission and the legitimate leader completing the threshold protocol (typically several seconds), making the attack practically feasible.

### Recommendation

Apply a cryptographic binding check for `AppPublicKey` responses analogous to the `AppPublicKeyPV` pairing check. Since `AppPublicKey` lacks a G2 component, one approach is to require threshold-many nodes to independently submit the same response (a quorum check on-chain) before resolving the yield. Alternatively, deprecate the `AppPublicKey` variant in favor of `AppPublicKeyPV`, which provides on-chain verifiability. At minimum, restrict `respond_ckd` to the elected leader for the request (mirroring the off-chain leader-election logic) so that a single Byzantine participant cannot unilaterally resolve a CKD request.

### Proof of Concept

1. User Alice calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)` and a deposit. The contract stores the pending yield under the `CKDRequest` key.
2. Malicious attested participant Eve (not the leader) observes the pending request on-chain.
3. Eve immediately calls `respond_ckd(ckd_request, CKDResponse { big_y: G1::generator(), big_c: G1::generator() })`.
4. The contract executes lines 675–682: `AppPublicKey` arm is empty — no check. Execution falls through to `resolve_yields_for`, which resolves Alice's yield with the fabricated response.
5. Alice's NEAR transaction callback fires with `big_y = G1`, `big_c = G1`. Alice computes `derived_key = G1 - a·G1 = (1-a)·G1`, an attacker-controlled value unrelated to the MPC master key.
6. The legitimate leader's `respond_ckd` call finds no pending yields and returns `Ok(())` silently.
7. Alice, trusting the MPC network, uses the wrong derived key to generate a foreign-chain address and deposits funds — which are permanently inaccessible.

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
