### Title
Direct Deposits Omitted from Preservation-of-Value Check Allows Arbitrary Account Balance Inflation - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces a new `DirectDeposits` field in the transaction body that allows a transaction to directly credit ADA to registered stake accounts. However, the sum of `DirectDeposits` is never included in either the `consumed` or `produced` side of the preservation-of-value (`consumed = produced`) check enforced by `validateValueNotConservedUTxO`. An unprivileged transaction sender can therefore include an arbitrarily large `DirectDeposits` map in a Dijkstra transaction, causing registered account balances to be inflated by any amount without providing the corresponding ADA from UTxO inputs. This creates ADA from nothing, directly violating the ledger's general accounting property.

---

### Finding Description

**The `DirectDeposits` feature (Dijkstra era)**

`DirectDeposits` is a new field (`dtbrDirectDeposits :: !DirectDeposits`) added to both top-level and sub-transaction bodies in the Dijkstra era. [1](#0-0) 

In the `ENTITIES` transition rule, after processing certificates, the direct deposits are applied unconditionally to account balances: [2](#0-1) 

The only guard before applying them is `directDepositsMissingAccounts` — which only checks that target accounts are registered, not that the ADA is funded: [3](#0-2) 

`applyDirectDeposits` adds the specified coin amounts to account balances with no further validation: [4](#0-3) 

**The preservation-of-value check does not include `DirectDeposits`**

In `dijkstraUtxoTransition`, the preservation-of-value check is: [5](#0-4) 

This calls `Shelley.validateValueNotConservedUTxO`, which computes:

```
consumed = UTxO inputs + withdrawals + cert refunds
produced = UTxO outputs + fees + cert deposits
``` [6](#0-5) 

The `consumedTxBody` and `producedTxBody` functions that back this check contain no reference to `DirectDeposits`: [7](#0-6) 

Similarly, `updateUTxOStateNoFees` — which updates the `utxosDeposited` pot — also does not account for `DirectDeposits`: [8](#0-7) 

**The structural gap**

For withdrawals, the `consumed` side includes `wbalance(txwdrls)`, ensuring withdrawn ADA must appear in UTxO outputs. For cert deposits, the `produced` side includes `certsTotalDepositsTxBody`, ensuring the deposited ADA must come from UTxO inputs. `DirectDeposits` have no corresponding entry on either side. The ADA credited to accounts via `DirectDeposits` is never deducted from any pot.

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker submits a Dijkstra transaction with:
- Minimal UTxO inputs (just enough to cover fees)
- A `DirectDeposits` map targeting any registered account(s) with an arbitrarily large `Coin` value

The `consumed = produced` check passes because `DirectDeposits` are invisible to it. The `ENTITIES` rule then credits the specified amounts to the target accounts. ADA is created from nothing, permanently inflating the total ADA supply in the ledger state. The attacker can:
- Inflate their own staking account balance to gain disproportionate staking rewards
- Inflate any registered account's balance (including governance participants) to manipulate governance weight
- Repeat indefinitely, creating unbounded ADA

This is a direct analog of the ERC20 `lockedBalances[msg.sender] += _amount` inflation without a successful `transferFrom`.

---

### Likelihood Explanation

**High.** The `DirectDeposits` field is freely settable by any transaction author in the Dijkstra era. No privileged role, key compromise, or consensus majority is required. The only prerequisite is that the target credential is a registered stake account, which is trivially satisfiable (the attacker can register their own). The Dijkstra era is the current development era and has not yet deployed to mainnet, but the rule is present in the production codebase.

---

### Recommendation

The sum of all `DirectDeposits` amounts in a transaction (and in each sub-transaction) must be included in the `produced` side of the preservation-of-value check, analogous to how cert deposits are handled:

```
produced = UTxO outputs + fees + cert_deposits + sum(DirectDeposits)
```

Concretely, the `getTotalDepositsTxBody` / `produced` computation for the Dijkstra era must be overridden to add `fold (unDirectDeposits (txBody ^. directDepositsTxBodyL))` to the produced value. The same fix must be applied for sub-transactions in `SUBENTITIES`.

---

### Proof of Concept

1. Register a stake credential `cred` and obtain its `AccountAddress addr`.
2. Construct a Dijkstra `TopTx` with:
   - One UTxO input covering only the transaction fee (e.g., 0.2 ADA)
   - One UTxO output returning the change
   - `directDepositsTxBodyL .~ DirectDeposits (Map.singleton addr (Coin 1_000_000_000_000))` (1 million ADA)
   - No certificates, no withdrawals
3. Submit the transaction.
4. The `UTXO` rule runs `validateValueNotConservedUTxO`: `consumed` = input value, `produced` = output value + fee. The check passes because `DirectDeposits` are absent from both sides.
5. The `ENTITIES` rule runs `applyDirectDeposits`, crediting 1,000,000 ADA to `addr`.
6. The attacker's account balance is now 1,000,000 ADA despite having provided only ~0.2 ADA in inputs. [2](#0-1) [9](#0-8) [10](#0-9)

validateValueNotConservedUTxO ::
  (EraUTxO era, EraCertState era) =>
  PParams era ->
  UTxO era ->
  CertState era ->
  TxBody TopTx era ->
  Test (ShelleyUtxoPredFailure era)
validateValueNotConservedUTxO pp utxo certState txBody =
  failureUnless (consumedValue == producedValue) $
    ValueNotConservedUTxO Mismatch {mismatchSupplied = consumedValue, mismatchExpected = producedValue}
  where
    consumedValue = consumed pp certState utxo txBody
    producedValue = produced pp certState txBody

-- | Compute the Coin part of what is consumed by a TxBody, itemized as a 'Consume'
consumedTxBody ::
  (EraTxBody era, EraCertState era) =>
  TxBody l era ->
  PParams era ->
  CertState era ->
  UTxO era ->
  Consumed
consumedTxBody txBody pp dpstate utxo =
  Consumed
    { conInputs =
        sumCoinUTxO (txInsFilter utxo (txBody ^. inputsTxBodyL))
    , conRefunds = certsTotalRefundsTxBody pp dpstate txBody
    , conWithdrawals = fold . unWithdrawals $ txBody ^. withdrawalsTxBodyL
    }

-- | Compute the Coin part of what is produced by a TxBody, itemized as a 'Produced'
producedTxBody ::
  (EraTxBody era, EraCertState era) =>
  TxBody TopTx era ->
  PParams era ->
  CertState era ->
  Produced
producedTxBody txBody pp dpstate =
  Produced
    { proOutputs = sumCoinUTxO (txouts txBody)
    , proFees = txBody ^. feeTxBodyL
    , proDeposits = certsTotalDepositsTxBody pp dpstate txBody
    }
