### Title
Hardcoded `paid_fee_on_l1 = Fee(1)` in Consensus L1 Handler Conversion Binds Wrong Executable Payload and Bypasses Fee Guard - (File: crates/apollo_transaction_converter/src/transaction_converter.rs)

### Summary

`TransactionConverter::convert_consensus_l1_handler_to_internal_l1_handler` unconditionally substitutes `Fee(1)` for the actual fee paid on L1 when converting every L1 handler transaction received through consensus. The only fee guard in the blockifier (`if paid_fee == Fee(0)`) is therefore always satisfied regardless of the real on-chain payment, and the wrong `paid_fee_on_l1` value is carried forward into the executable transaction, the execution receipt, and the central-objects blob committed to Aerospike.

### Finding Description

**Root cause — `TransactionConverter`:**

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs  lines 473-483
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),          // ← always Fee(1), never the real L1 payment
    )?)
}
``` [1](#0-0) 

The `ConsensusTransaction::L1Handler` wire type carries only the raw `transaction::L1HandlerTransaction` (no `paid_fee_on_l1` field). The converter is the single place where the fee is attached to the executable form. Instead of reading the value from the originating L1 event, it hard-codes `Fee(1)`.

**Fee guard in the blockifier — always satisfied:**

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs  lines 103-113
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
``` [2](#0-1) 

Because `Fee(1) ≠ Fee(0)`, the guard is unconditionally bypassed for every L1 handler transaction that arrives through consensus, regardless of what was actually paid on L1.

**Wrong value committed to the blob:**

`CentralL1HandlerTransaction` serialises `paid_fee_on_l1` verbatim into the Aerospike blob:

```rust
// crates/apollo_consensus_orchestrator/src/cende/central_objects.rs  lines 383-393
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,   // always Fee(1)
        }
    }
}
``` [3](#0-2) 

Every L1 handler transaction committed to the blob therefore carries `paid_fee_on_l1 = 1` instead of the real L1 payment amount.

### Impact Explanation

Two concrete effects:

1. **Fee guard bypass.** The blockifier's only enforcement that an L1 handler paid a non-zero fee is the `paid_fee == Fee(0)` check. With `Fee(1)` always injected, a proposer could include an L1 handler transaction whose actual on-chain payment was zero (or whose payment record was otherwise absent) and the validator would accept and execute it without error.

2. **Wrong executable payload committed.** `paid_fee_on_l1` is part of the `L1HandlerTransaction` executable struct and is serialised into the central-objects blob. The committed record permanently reflects `Fee(1)` rather than the true L1 payment, corrupting the authoritative off-chain view of every L1 handler transaction processed through consensus.

This matches the allowed impact: *"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

### Likelihood Explanation

Every L1 handler transaction that passes through the consensus path (validator side) triggers this code path unconditionally. No special attacker capability is required; the substitution happens as part of normal proposal validation. The TODO comment confirms the developers are aware the value is a placeholder, meaning the production system is currently operating with this invariant broken.

### Recommendation

Transmit the actual `paid_fee_on_l1` value through the consensus wire format (add it to the `ConsensusTransaction::L1Handler` variant or a wrapper), and use it in `convert_consensus_l1_handler_to_internal_l1_handler` instead of the hard-coded `Fee(1)`. Until the wire format is extended, at minimum document that the fee guard is non-functional for the consensus path and do not rely on it for security.

### Proof of Concept

1. A proposer builds a block containing an `L1HandlerTransaction` (sourced from an L1 event with any `paid_fee_on_l1` value, including `Fee(0)`).
2. The validator receives `ConsensusTransaction::L1Handler(tx)` and calls `convert_consensus_l1_handler_to_internal_l1_handler(tx)`.
3. The converter calls `L1HandlerTransaction::create(tx, &self.chain_id, Fee(1))` — the real fee is discarded.
4. The blockifier executes the transaction; `paid_fee == Fee(1) ≠ Fee(0)` so the guard passes unconditionally.
5. `CentralL1HandlerTransaction::from(tx)` serialises `paid_fee_on_l1: Fee(1)` into the Aerospike blob.
6. The committed record is wrong for every L1 handler transaction in every block produced by this sequencer.
