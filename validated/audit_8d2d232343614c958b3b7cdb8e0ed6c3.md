### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Malicious Participant — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` accepts a caller-supplied `payload_hash` inside `VerifyForeignTransactionResponse` and only checks that the MPC root key's signature is valid over that hash. It never verifies that `payload_hash` is actually the canonical hash of a `ForeignTxSignPayload` that encodes the specific `request` being resolved. A single malicious attested participant can replay a valid `(payload_hash, signature)` pair obtained from a prior legitimate signing round for `request_B` against a currently-pending `request_A`, causing the contract to deliver a fraudulent attestation to the user.

---

### Finding Description

The vulnerability class from the external report is **insufficient input validation before accepting a response**: the Vega `prettyPrint` function accepted any attribute without checking it against the original transaction object. The direct analog here is `respond_verify_foreign_tx`, which accepts any `payload_hash` without checking it against the `request` that is being resolved.

In `respond_verify_foreign_tx`:

```
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

The contract verifies only that `signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. It does **not** verify that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request: <the pending request>, values: <observed values> }))`. [1](#0-0) 

After the signature check passes, the contract resolves the pending yield for `request` with the full `response` (including the unvalidated `payload_hash`):

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The `ForeignTxSignPayload` is designed so that `payload_hash` encodes both the original `request` and the `values` observed from the foreign chain:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
``` [3](#0-2) 

The node-side code correctly computes `payload_hash` from the actual observed values before submitting:

```rust
let payload_hash = response.0.0.compute_msg_hash()?;
let response = contract_args::VerifyForeignTransactionRespondArgs::from_signature(
    verify_foreign_tx_attempt.request.clone(),
    payload_hash, ...
``` [4](#0-3) 

But the contract never re-derives or cross-checks this hash against the `request` it is resolving.

---

### Impact Explanation

**High — Forged foreign-chain verification causing invalid bridge execution.**

A malicious attested participant replays `(payload_hash_B, signature_B)` — a valid pair produced during a prior legitimate signing round for `request_B` — against a currently-pending `request_A`. The contract:

1. Finds the pending yield for `request_A` ✓
2. Verifies `signature_B` over `payload_hash_B` under the root key ✓ (the signature is genuinely valid)
3. Resolves the yield for `request_A` with `{payload_hash_B, signature_B}`

The user who submitted `request_A` (e.g., "verify Bitcoin tx X, extract BlockHash") receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes `request_B`'s data (e.g., a different tx_id and a different block hash). The user's bridge contract cannot detect this because it does not know the `values` the MPC nodes observed — that is the entire purpose of the MPC verification service. The bridge contract receives a cryptographically valid MPC attestation that does not correspond to the transaction it asked to verify, enabling invalid bridge execution (e.g., minting tokens on NEAR for a Bitcoin deposit that was never made, or for a different deposit amount).

---

### Likelihood Explanation

**Medium.** The attacker must be a single malicious attested MPC participant (below threshold — satisfies the allowed attacker model). The attack requires:

1. A prior legitimate `verify_foreign_transaction(request_B)` to have been processed, producing a reusable `(payload_hash_B, signature_B)` pair. This is a normal operational condition on a live bridge.
2. A concurrent pending `verify_foreign_transaction(request_A)` to target. On a live bridge, multiple requests are pending at any time.

No threshold collusion is required. The malicious node calls `respond_verify_foreign_tx` directly on-chain with the replayed response. The TEE/attestation requirement gates who can call this endpoint, but a single compromised or malicious attested node is sufficient.

---

### Recommendation

The contract must bind `payload_hash` to the `request` being resolved. The cleanest fix is to require the responding node to submit the full `ForeignTxSignPayload` (including `values`) alongside the signature, and have the contract:

1. Assert `payload.request == request` (the pending request matches).
2. Recompute `expected_hash = SHA-256(borsh(payload))`.
3. Assert `expected_hash == response.payload_hash`.
4. Verify the signature over `expected_hash`.

This mirrors the fix described in the external report: converting the untrusted object to a canonical format and strictly checking each field before accepting it.

Alternatively, if submitting `values` on-chain is too expensive, the contract should at minimum verify that `payload_hash` is a hash that encodes the correct `request` by requiring the node to submit a Borsh-serialized prefix of the payload that includes only the `request` portion, and checking it matches.

---

### Proof of Concept

**Setup:** Two pending foreign-tx requests exist: `request_A` (Bitcoin tx_id=`[0xAA;32]`) and `request_B` (Bitcoin tx_id=`[0xBB;32]`). Both are supported chains on the same domain.

**Step 1 — Obtain a valid signature for `request_B`:**
The MPC network legitimately processes `request_B`. The leader node computes `payload_hash_B = SHA256(borsh(ForeignTxSignPayload{request_B, [BlockHash([0x42;32])]}))` and the threshold protocol produces `signature_B` over `payload_hash_B` under the root key. The malicious node observes this `(payload_hash_B, signature_B)` pair (it participated in the signing round).

**Step 2 — Replay against `request_A`:**
The malicious node calls:
```
respond_verify_foreign_tx(
    request = VerifyForeignTransactionRequest { request: Bitcoin(tx_id=[0xAA;32]), ... },
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_B,   // hash of request_B's data
        signature: signature_B,          // valid sig over payload_hash_B
    }
)
```

**Step 3 — Contract accepts:**
- `assert_caller_is_attested_participant_and_protocol_active()` passes (malicious node is attested).
- `verify_ecdsa_signature(signature_B, payload_hash_B, root_pk)` returns `Ok` — the signature is genuinely valid.
- `resolve_yields_for(pending_verify_foreign_tx_requests, request_A, response)` resolves the yield for `request_A` with the fraudulent response.

**Step 4 — User receives fraudulent attestation:**
The user who submitted `request_A` receives `{payload_hash_B, signature_B}`. Their bridge contract verifies the signature (valid) and proceeds to execute a bridge action based on `payload_hash_B`, which encodes `request_B`'s block hash — not the block hash of the transaction they actually asked to verify.     #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
}
