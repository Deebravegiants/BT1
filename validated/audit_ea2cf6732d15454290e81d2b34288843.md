### Title
Irreversible Eviction of Gap-Account Transactions on Rejected Admission Enables Zero-Cost Mempool DoS — (`crates/apollo_mempool/src/mempool.rs`)

---

### Summary

When a new transaction overflows the mempool's byte-capacity, `handle_capacity_overflow` calls `try_make_space` to evict gap-account transactions. If the eviction still leaves insufficient room, the incoming transaction is rejected with `MempoolFull`. The evicted gap-account transactions are **never restored**. An attacker can exploit this by submitting a transaction that is large enough to trigger eviction but too large to fit even after eviction, permanently displacing legitimate transactions at zero fee cost.

---

### Finding Description

In `handle_capacity_overflow` (lines 1031–1052 of `crates/apollo_mempool/src/mempool.rs`):

```rust
fn handle_capacity_overflow(...) -> Result<(), MempoolError> {
    ...
    let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;
    let required_space = tx.total_bytes().saturating_sub(freed_bytes);
    if !creating_gap && self.try_make_space(required_space) {
        return Ok(());
    }
    Err(MempoolError::MempoolFull)   // ← eviction already committed, no rollback
}
```

`try_make_space` (lines 992–1029) permanently removes gap-account transactions from both `tx_pool` and `tx_queue` before the capacity check resolves. If the incoming transaction is ultimately rejected, those removals stand. The test `rejects_or_accepts_tx_based_on_freed_space` (lines 1894–1938) explicitly documents this:

> *"Transactions tx1 and tx2 are evicted regardless of whether large_tx is accepted or rejected. We do not revert the eviction attempt even if adding large_tx ultimately fails."* [1](#0-0) [2](#0-1) [3](#0-2) 

**Attack path:**

1. Victim submits transactions with a nonce gap (e.g., nonce 5 while account nonce is 0). These pass gateway stateful validation because `max_allowed_nonce_gap > 0`.
2. Attacker submits a transaction from a fresh account (nonce 0 = account nonce, so `creating_gap = false`). The transaction is crafted with a large signature, sized so that:
   - It exceeds current capacity → triggers `handle_capacity_overflow`.
   - Even after evicting all gap-account transactions, it still does not fit → `try_make_space` returns `false` → `MempoolFull` returned.
3. The gateway's stateful validator (`run_validate_entry_point`) runs against the current state and passes because the attacker holds enough balance to satisfy the fee bound check. The transaction is forwarded to the mempool.
4. The mempool evicts the victim's gap transactions, then rejects the attacker's transaction.
5. The attacker's transaction is never executed; **no fee is charged**. The victim's transactions are permanently gone.
6. The attacker reuses the same account (nonce unchanged, since the transaction was rejected) and repeats. [4](#0-3) [5](#0-4) [6](#0-5) 

The gateway's stateless validator enforces a maximum transaction size (`validate_tx_size`), but within that bound the attacker can still craft a transaction large enough to trigger eviction while remaining too large to fit after eviction, depending on the current pool occupancy. [7](#0-6) 

---

### Impact Explanation

Valid transactions that passed gateway admission are permanently removed from the mempool before sequencing. The victim must resubmit. Because the attacker's transaction is rejected at the mempool layer (never executed), **no fee is deducted**. The attacker can repeat the attack from the same account indefinitely (nonce never advances on rejection), or fan out across multiple accounts to evict a larger set of victim transactions simultaneously. This matches the "High — mempool/gateway/RPC admission rejects valid transactions before sequencing" impact class.

---

### Likelihood Explanation

- Gap-account transactions are common: any user who submits nonce N+1 before nonce N is committed creates a gap.
- The attacker only needs one account with sufficient balance to pass the gateway's fee-bound check (balance is not consumed on rejection).
- The maximum transaction size is bounded by `max_l2_gas_amount` / `validate_tx_size`, but within that bound the attack is feasible whenever the pool is near capacity.
- No privileged access is required; the attack is fully unprivileged via the public RPC endpoint.

---

### Recommendation

Restore evicted transactions if the incoming transaction is ultimately rejected. Concretely, `try_make_space` should be called only after confirming the incoming transaction will be admitted, or the eviction should be staged and rolled back on `MempoolFull`. A simpler guard: before calling `try_make_space`, verify that `required_space <= total_bytes_of_all_gap_account_txs`; if not, return `MempoolFull` immediately without mutating state.

---

### Proof of Concept

```
State: mempool capacity = C bytes, full with victim gap-account txs (nonce 5, account_nonce 0).

Attacker account A: nonce 0, account_nonce 0, balance ≥ max_fee threshold.
Attacker tx T: nonce 0, signature padded to size S where S > C (or S > remaining gap-tx bytes).

1. Gateway receives T.
2. stateless_tx_validator.validate(&T) → Ok  (size ≤ max_tx_size)
3. stateful_validator.extract_state_nonce_and_run_validations(&T) → Ok (nonce 0 == account_nonce 0, balance ok)
4. mempool_client.add_tx(T) called.
5. mempool.add_tx:
     add_tx_validations → exceeds_capacity(T, 0) = true
     handle_capacity_overflow:
       creating_gap = false  (T.nonce == account_nonce)
       try_make_space(S):
         evicts victim's gap txs → total_freed < S → returns false
     → Err(MempoolFull)
6. Gateway returns MempoolFull error to attacker. No fee charged.
7. Victim's gap txs are gone. Attacker repeats from step 1.
``` [8](#0-7) [1](#0-0)

    fn add_tx_validations(
        &mut self,
        tx_reference: TransactionReference,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
    ) -> MempoolResult<()> {
        self.validate_incoming_tx(tx_reference, account_nonce)?;
        let replaced_tx_reference = self.validate_fee_escalation(tx_reference)?;

        // The replaced transaction is still pooled, so its bytes still count toward
        // `size_in_bytes()`. Credit what its removal will free: a same-size bump nets to zero (no
        // overflow handling), and a larger replacement only needs room for the delta, consistent
        // with how a fresh next-nonce transaction is treated. The removal happens only after
        // capacity is confirmed below, so a rejected incoming transaction never strands the
        // account.
        let freed_bytes = replaced_tx_reference.map_or(0, |reference| {
            self.tx_pool
                .get_by_tx_hash(reference.tx_hash)
                .expect("Replacement target from pool must exist.")
                .total_bytes()
        });

        if self.exceeds_capacity(tx, freed_bytes) {
            self.handle_capacity_overflow(tx, account_nonce, freed_bytes)?;
        }

        // Capacity is confirmed: this is the final, infallible mutation before the incoming
        // transaction is inserted by the caller.
        if let Some(existing_tx_reference) = replaced_tx_reference {
            self.remove_replaced_tx(existing_tx_reference);
        }

        Ok(())
    }

    pub fn try_make_space(&mut self, required_space: u64) -> bool {
        let mut total_space_freed = 0;
        let mut evicted_txs = Vec::new();

        while total_space_freed < required_space {
            let Some(address) = self.get_evictable_account() else {
                break;
            };

            let txs: Vec<_> = self.tx_pool.account_txs_sorted_by_nonce(address).copied().collect();
            for tx_ref in txs.iter().rev() {
                let tx = self
                    .tx_pool
                    .remove(tx_ref.tx_hash)
                    .expect("Transaction must exist in the pool.");
                total_space_freed += tx.total_bytes();
                evicted_txs.push(*tx_ref);
                metric_count_evicted_txs(1);
                self.decrement_stuck_txs_if_gap_account(address, 1);
                if total_space_freed >= required_space {
                    break;
                }
            }

            // Clean up if account is now empty.
            if !self.tx_pool.contains_account(address) {
                self.accounts_with_gap.swap_remove(&address);
            }
        }

        // Keep the queue consistent with the pool: the evicted txs were removed from the pool, so
        // drop their (now-orphaned) queue references too. In fee mode gap accounts are never
        // queued, so this is a no-op there; in Echonet/FIFO mode they are, and skipping this leaves
        // a dangling reference that panics the next `get_txs`.
        self.tx_queue.remove_txs(&evicted_txs);

        total_space_freed >= required_space
    }

    fn handle_capacity_overflow(
        &mut self,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
        freed_bytes: u64,
    ) -> Result<(), MempoolError> {
        let address = tx.contract_address();

        let account_has_gap = self.accounts_with_gap.contains(&address);
        let account_has_txs = self.tx_pool.contains_account(address);
        let closing_gap = tx.nonce() == account_nonce;
        let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;

        // Only the net growth must be evicted: an accompanying replacement removal frees
        // `freed_bytes` (0 when there is no replacement).
        let required_space = tx.total_bytes().saturating_sub(freed_bytes);
        if !creating_gap && self.try_make_space(required_space) {
            return Ok(());
        }

        Err(MempoolError::MempoolFull)
    }
