### Title
`respond_ckd` Lacks Cryptographic Output Verification for `AppPublicKey` Variant, Enabling Single Byzantine Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function performs cryptographic output verification only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`, but silently skips all verification for the `AppPublicKey` variant. This is the direct analog of the `restartDca` pattern: a function that performs similar operations to a reference function (`respond`) but omits a critical safety check present in the reference. A single Byzantine participant — strictly below the signing threshold — can submit an arbitrary forged `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will accept and deliver it without any cryptographic check.

---

### Finding Description

`respond` (for signatures) verifies the submitted signature against the MPC root public key before resolving the yield: [1](#0-0) 

If the signature is invalid, the call is rejected. No single participant can forge a valid signature.

`respond_ckd` (for confidential key derivation) performs the analogous role but applies verification only conditionally: [2](#0-1) 

For `AppPublicKeyPV`, `ckd_output_check` is called and the response is verified against the BLS12-381 root public key. For `AppPublicKey`, the branch body is empty — **no verification whatsoever** — and `resolve_yields_for` is called immediately with whatever `CKDResponse` the caller supplied.

The only guards on `respond_ckd` are: [3](#0-2) 

These require the caller to be an attested participant, but impose **no threshold requirement** and perform **no cryptographic check on the response content** for the `AppPublicKey` path. A single attested-but-malicious participant satisfies all guards.

Once `resolve_yields_for` runs, the pending CKD request is removed from `pending_ckd_requests` and the yield is resolved with the forged payload. The legitimate MPC-computed response can never be submitted afterward — the request is gone. [4](#0-3) 

---

### Impact Explanation

**Critical — Unauthorized confidential key derivation output without the required participant authorization.**

A single Byzantine participant (below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request in the contract.
2. Craft an arbitrary `CKDResponse` — for example, one that encrypts a key the attacker already knows, or simply random bytes.
3. Call `respond_ckd` with the forged response. The contract accepts it unconditionally.
4. The yield is resolved; the user's callback receives the forged derived key.
5. The legitimate threshold-computed response can never be submitted — the request is permanently consumed.

If the attacker supplies a key they control, they gain access to the user's derived private key, enabling them to sign arbitrary transactions on foreign chains on the user's behalf. If they supply garbage, the user's CKD request is permanently destroyed with no recovery path. Either outcome constitutes unauthorized confidential key derivation output without threshold authorization.

---

### Likelihood Explanation

**Medium.** The attacker must be an attested MPC participant — a realistic threat in a threshold system where the security model explicitly tolerates up to `t-1` Byzantine participants. No collusion is required; a single compromised or malicious node suffices. The attack requires only one on-chain transaction and is immediately effective.

---

### Recommendation

Add cryptographic output verification for the `AppPublicKey` variant in `respond_ckd`, mirroring the check already present for `AppPublicKeyPV`. The BLS12-381 root public key is already retrieved at that point and can be used to verify the CKD output. Specifically:

- Define and call a verification function for the `AppPublicKey` path analogous to `ckd_output_check` for `AppPublicKeyPV`.
- Reject the call (panic or return error) if verification fails, consistent with how `respond` rejects invalid signatures.
- Add a unit test asserting that a forged `CKDResponse` submitted by a participant is rejected for both `AppPublicKey` and `AppPublicKeyPV` variants.

---

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key: AppPublicKey(pk)` and a valid `domain_id`. The request is stored in `pending_ckd_requests`.

2. Byzantine participant (single, attested) constructs a forged `CKDResponse { big_y: [attacker_controlled_bytes], big_c: [attacker_controlled_bytes] }`.

3. Byzantine participant calls `respond_ckd(request, forged_response)`.

4. Contract executes:
   - `assert_caller_is_signer()` — passes (participant calls directly).
   - `assert_caller_is_attested_participant_and_protocol_active()` — passes (participant is attested).
   - `match &request.app_public_key { AppPublicKey(_) => {} ... }` — **no verification, branch is empty**.
   - `resolve_yields_for(...)` — resolves the yield with the forged payload, removes the request.

5. User's callback receives the forged derived key. The legitimate threshold-computed response can never be submitted. The user's CKD request is permanently consumed with attacker-controlled output. [5](#0-4)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
