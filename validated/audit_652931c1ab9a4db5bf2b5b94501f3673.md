### Title
`completeWithdrawal` Can Revert and Temporarily Freeze User ETH Funds When Aave Liquidity Is Insufficient — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

When the Aave integration is enabled, `completeWithdrawal` for ETH assets relies on `_withdrawFromAave` to source liquidity. The function does not verify that Aave's underlying WETH pool has sufficient available liquidity before calling `aaveWETHGateway.withdrawETH`. If Aave's utilization is high and the pool cannot service the withdrawal, the call reverts, blocking the user's `completeWithdrawal` and temporarily freezing their ETH.

---

### Finding Description

`_processWithdrawalCompletion` (called by both `completeWithdrawal` and `completeWithdrawalForUser`) contains the following ETH-specific path when Aave integration is enabled:

```solidity
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
    uint256 contractBalance = address(this).balance;
    if (contractBalance < request.expectedAssetAmount) {
        uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
        _withdrawFromAave(amountNeeded);          // ← can revert or return short

        uint256 balanceAfter = address(this).balance;
        if (balanceAfter < request.expectedAssetAmount) {
            revert InsufficientLiquidityForWithdrawal();   // ← blocks user
        }
    }
}
``` [1](#0-0) 

`_withdrawFromAave` silently caps the amount it will withdraw at `withdrawablePrincipal = min(aaveBalance, totalETHDepositedToAave)`:

```solidity
withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
if (withdrawnAmount == 0) return 0;

aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
``` [2](#0-1) 

Two distinct failure modes exist:

1. **Silent cap path**: If `amountNeeded > withdrawablePrincipal`, `_withdrawFromAave` withdraws only `withdrawablePrincipal` and returns. `balanceAfter` remains below `request.expectedAssetAmount`, triggering `InsufficientLiquidityForWithdrawal`.

2. **Aave revert path**: Even when `withdrawnAmount <= withdrawablePrincipal`, if Aave's underlying WETH pool has high utilization, `aaveWETHGateway.withdrawETH` itself reverts, propagating the revert up through `_processWithdrawalCompletion`.

In both cases the entire transaction reverts (preserving the user's withdrawal request), but the user cannot complete their withdrawal until Aave liquidity recovers.

The design amplifies this risk because `unlockQueue` deposits all unlocked ETH directly into Aave via a try/catch that silently swallows deposit failures:

```solidity
try this.depositToAaveExternal(assetAmountUnlocked) { }
catch (bytes memory reason) {
    emit AaveDepositFailed(assetAmountUnlocked, reason);
    // Silently fail if Aave deposit fails
}
``` [3](#0-2) 

This means the protocol deliberately routes all unlocked ETH through Aave, making every ETH `completeWithdrawal` dependent on Aave's real-time liquidity.

---

### Impact Explanation

Any user with an unlocked ETH withdrawal request is blocked from calling `completeWithdrawal` for as long as Aave cannot service the withdrawal. Their rsETH has already been burned (during `unlockQueue`), and the ETH is locked in Aave. This constitutes **temporary freezing of funds** (Medium severity per the allowed impact scope).

---

### Likelihood Explanation

Aave WETH pool utilization regularly spikes during market stress events (e.g., liquidation cascades, depeg events), which are precisely the conditions under which users most urgently want to withdraw. The Aave integration is an opt-in feature that, once enabled, affects every ETH `completeWithdrawal`
