Audit Report

## Title
Unpermissioned `sendFunds()` Enables Persistent `DailyFeeMintLimitExceeded` DoS on `_updateRsETHPrice()` — (`contracts/FeeReceiver.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access-control modifier, allowing any caller to flush accumulated ETH into `LRTDepositPool` at will. If the resulting TVL delta produces a protocol-fee rsETH amount exceeding `maxFeeMintAmountPerDay`, every subsequent call to `updateRSETHPrice()` (and `updateRSETHPriceAsManager()`) reverts with `DailyFeeMintLimitExceeded`. Because `rsETHPrice` is written only after the fee check, the price stays stale, the full delta is re-evaluated on every future attempt, and the DoS is self-perpetuating until a manager manually raises the daily cap — permanently freezing the protocol's unclaimed fee yield.

## Finding Description

**Root cause — no access control on `sendFunds()`**

`FeeReceiver.sendFunds()` at line 53 carries no role modifier: [1](#0-0) 

`LRTDepositPool.receiveFromRewardReceiver()` is equally open: [2](#0-1) 

**TVL increases immediately; price write is deferred**

After `sendFunds()` executes, `getETHDistributionData()` reads `address(this).balance` directly, so `_getTotalEthInProtocol()` reflects the full dumped balance instantly: [3](#0-2) 

`previousTVL` is computed from the still-stale `rsETHPrice`: [4](#0-3) 

**Fee check reverts before the price write**

The fee is computed from the full TVL delta: [5](#0-4) 

Then checked against the daily cap: [6](#0-5) 

If the check reverts, execution never reaches the price write: [7](#0-6) 

**DoS is self-perpetuating**

Because `rsETHPrice` is never updated, `previousTVL` stays at the pre-dump value on every future call. The full accumulated reward delta is re-evaluated on every `updateRSETHPrice()` attempt. Even after the daily period resets (`currentPeriodMintedFeeAmount` zeroed at line 200), the single-call fee amount still exceeds `maxFeeMintAmountPerDay`, so the revert fires again. [8](#0-7) 

**No privileged bypass exists**

`updateRSETHPriceAsManager()` calls the same `_updateRsETHPrice()` and hits the identical revert path — the manager cannot bypass the DoS without first raising `maxFeeMintAmountPerDay`: [9](#0-8) 

## Impact Explanation

The protocol's fee share (rsETH minted to the treasury) is frozen indefinitely. The ETH is already inside `LRTDepositPool`, so rsETH holders accrue the full yield appreciation while the treasury receives nothing. The freeze is self-perpetuating and does not resolve without explicit manager intervention to raise the daily cap. This matches **Medium — Permanent freezing of unclaimed yield**. The claim's characterisation as "High — Theft of unclaimed yield" is overstated: the fee is recoverable once the manager raises `maxFeeMintAmountPerDay`, so it is frozen rather than permanently stolen.

## Likelihood Explanation

- `sendFunds()` requires zero privileges and zero capital.
- `FeeReceiver` accumulates ETH passively from MEV/execution-layer rewards; any multi-day accumulation can exceed `maxFeeMintAmountPerDay`.
- The attacker only needs to monitor the `FeeReceiver` balance and call one function at the right moment.
- The attack is repeatable: after the manager raises the cap and the price is updated, the attacker can wait for the next accumulation cycle and repeat.

## Recommendation

Add an access-control modifier to `sendFunds()` so only authorised callers can trigger the ETH transfer:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    ...
}
```

Additionally, consider capping the single-call TVL delta eligible for fee computation, or splitting large accumulated rewards across multiple `updateRSETHPrice()` calls, so the daily limit can never be permanently breached by a single dump.

## Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.27;
import "forge-std/Test.sol";

contract FeeBypassPoC is Test {
    IFeeReceiver feeReceiver = IFeeReceiver(FEE_RECEIVER_ADDR);
    ILRTOracle   oracle      = ILRTOracle(ORACLE_ADDR);

    function test_persistentDosViaSendFunds() public {
        // 1. Simulate multi-day MEV rewards accumulating in FeeReceiver
        uint256 accumulatedRewards = 100 ether; // >> maxFeeMintAmountPerDay equivalent
        vm.deal(address(feeReceiver), accumulatedRewards);

        uint256 priceBefore = oracle.rsETHPrice();

        // 2. Unprivileged attacker dumps all ETH into DepositPool
        vm.prank(address(0xdead));
        feeReceiver.sendFunds();

        // 3. Oracle update reverts — DailyFeeMintLimitExceeded
        vm.expectRevert();
        oracle.updateRSETHPrice();

        // 4. Price is stale; no fee was minted
        assertEq(oracle.rsETHPrice(), priceBefore);

        // 5. Daily period resets but delta is unchanged — still reverts
        vm.warp(block.timestamp + 1 days + 1);
        vm.expectRevert();
        oracle.updateRSETHPrice();

        // 6. Manager path hits the same revert
        vm.prank(MANAGER_ADDR);
        vm.expectRevert();
        oracle.updateRSETHPriceAsManager();
    }
}
```
