### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Pending `request` — Forged Foreign-Chain Verification by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the domain's root public key. It does **not** verify that `response.payload_hash` is the canonical hash of `(request, extracted_values)` for the specific `request` being resolved. A single malicious attested participant acting as signing leader can obtain a threshold signature over hash `H_A` (produced for a legitimate request `A`) and then submit it as the response to a completely different pending request `B`, delivering a `payload_hash` that encodes `A`'s foreign-chain data to `B`'s caller. This enables forged foreign-chain verification and invalid bridge execution without any threshold-level collusion.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The `payload_hash` is taken entirely from the caller-supplied `response`, not derived from the stored `request`. After the signature check passes, the contract resolves the yield for `request` with the full `response` blob:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

Contrast this with `respond` for ordinary signatures, where the payload is taken from the stored `request` — not from the response — so the contract itself enforces the binding:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [3](#0-2) 

The correct hash that should be verified is `SHA-256(borsh(ForeignTxSignPayload{request, values}))`:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
``` [4](#0-3) 

The SDK's `ForeignChainSignatureVerifier` does perform this binding check on the client side:

```rust
let expected_payload_hash = expected_payload.compute_msg_hash()...;
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [5](#0-4) 

But this is a client-side helper, not enforced by the contract. The contract itself accepts any `payload_hash` that has a valid root-key signature, regardless of whether it corresponds to the pending `request`.

---

### Impact Explanation

A malicious attested participant acting as the signing leader for request `A` can:

1. Initiate a legitimate signing session for `request_A` (e.g., verify Bitcoin tx `0xAA`).
2. Honest followers each query the foreign chain for `request_A`, compute `H_A = SHA-256(borsh({request_A, values_A}))`, and contribute partial signatures over `H_A`.
3. The malicious leader aggregates the partial signatures into a valid threshold signature `sig_A` over `H_A`.
4. Instead of calling `respond_verify_foreign_tx(request_A, {H_A, sig_A})`, the leader calls `respond_verify_foreign_tx(request_B, {H_A, sig_A})` for a different pending request `B` (e.g., verify Bitcoin tx `0xBB`).
5. The contract finds `request_B` in `pending_verify_foreign_tx_requests`, verifies that `sig_A` is a valid root-key signature over `H_A` (which it is), and resolves `request_B`'s yield with `{payload_hash: H_A, signature: sig_A}`.
6. The caller of `request_B` (e.g., a bridge contract) receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes `request_A`'s foreign-chain data (tx `0xAA`'s block hash, etc.), not `request_B`'s.

If the bridge contract does not independently recompute the expected `payload_hash` from its own request parameters and compare it to the returned value, it will accept the forged attestation and execute an invalid bridge action (e.g., releasing funds for a deposit that was never verified, or releasing funds for the wrong amount/chain).

This matches the allowed impact: **High — forged foreign-chain verification causing invalid bridge execution**.

---

### Likelihood Explanation

- The attacker must be a single attested MPC participant — a Byzantine node below the signing threshold. No collusion with other nodes is required.
- The attacker only needs to be elected leader for any one request (leadership rotates across requests, so every node will eventually lead).
- Any concurrently pending `request_B` is a valid target; in a live bridge with continuous traffic, multiple targets are always available.
- The attack requires no special timing, no network-level interference, and no key material beyond what the node legitimately holds.

---

### Recommendation

The contract must recompute the expected `payload_hash` from the stored `request` and the `extracted_values` supplied in the response, and reject any response whose `payload_hash` does not match. Because the contract does not currently receive `extracted_values` in the response, the fix requires one of:

1. **Include `extracted_values` in `VerifyForeignTransactionResponse`** and have the contract compute `expected_hash = SHA-256(borsh({request, values}))`, then assert `expected_hash == response.payload_hash` before accepting the response.
2. **Alternatively**, have the node sign a commitment that binds the `request` key (e.g., include the `request` hash as a domain separator in the signed payload), and verify that binding on-chain.

The client-side `ForeignChainSignatureVerifier` in `crates/near-mpc-sdk/src/foreign_chain.rs` already demonstrates the correct pattern; the same logic must be enforced inside `respond_verify_foreign_tx`.

### Proof of Concept

**Setup**: Two pending requests in the contract:
- `request_A`: verify Bitcoin tx `0xAA`, `confirmations=6`, `extractors=[BlockHash]`
- `request_B`: verify Bitcoin tx `0xBB`, `confirmations=6`, `extractors=[BlockHash]`

**Attack**:

```rust
// Malicious leader computes H_A for request_A with fabricated values
let payload_a = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request_A.request.clone(),
    values: vec![ExtractedValue::BitcoinExtractedValue(
        BitcoinExtractedValue::BlockHash([0xAA; 32].into()),
    )],
});
let H_A = payload_a.compute_msg_hash().unwrap();

// Honest followers sign H_A (they verified request_A legitimately)
// Malicious leader aggregates -> sig_A

// Leader submits sig_A as response to request_B (not request_A)
contract.respond_verify_foreign_tx(
    request_B,  // <-- different request
    VerifyForeignTransactionResponse {
        payload_hash: H_A,   // <-- hash of request_A's data
        signature: sig_A,    // <-- valid root-key signature over H_A
    },
).expect("contract accepts because sig_A is valid over H_A");

// request_B's caller receives {payload_hash: H_A, signature: sig_A}
// payload_hash encodes tx 0xAA's block hash, not tx 0xBB's
// A bridge that skips payload_hash verification releases funds for 0xBB
// based on attestation data from 0xAA
```

The contract's `respond_verify_foreign_tx` will accept this call because it only checks `verify_ecdsa_signature(sig_A, H_A, root_pk)`, which passes. The binding between `H_A` and `request_B` is never verified. [6](#0-5) [4](#0-3) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
