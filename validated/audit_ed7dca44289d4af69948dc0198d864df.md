### Title
Deposits Paid for Non-Winning Update Proposals Are Permanently Lost When `do_update` Clears All Entries - (File: crates/contract/src/update.rs)

### Summary

When a contract update reaches threshold and `do_update` is executed, it unconditionally clears **all** pending update proposals (`self.entries.clear()`). Because `UpdateEntry` stores neither the proposer's `AccountId` nor the deposit amount paid, the NEAR deposits attached by every non-winning proposer are permanently locked in the contract with no recovery path.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` is a `#[payable]` method that requires each proposer to attach a deposit calculated by `ProposedUpdates::required_deposit(&update)` (storage cost proportional to the update payload size). Only the excess above the required amount is refunded at proposal time; the exact required deposit is retained by the contract. [1](#0-0) 

The `UpdateEntry` struct that is stored per proposal contains only the update payload and a `bytes_used` field — it records neither the proposer's account ID nor the deposit amount: [2](#0-1) 

When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every other pending proposal: [3](#0-2) 

Because the proposer identity and deposit amount are never stored in `UpdateEntry`, there is no information available at clear-time to issue refunds. The NEAR tokens paid for all non-winning proposals are permanently locked in the contract.

### Impact Explanation

Every participant who proposed a competing update loses their deposit — which can be substantial for contract-code proposals (the deposit scales with the size of the WASM binary). The funds are not stolen by any party; they are simply irrecoverable, permanently inflating the contract's balance. This breaks the production accounting invariant that deposited funds must be returnable to their owners. This maps to the allowed Medium impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

The scenario is a routine governance event: multiple participants independently propose different contract upgrades (e.g., competing WASM binaries or config changes), then one proposal reaches threshold. The `test_propose_update_contract_many` sandbox test already exercises exactly this multi-proposal scenario and confirms that all proposals are cleared after one wins. [4](#0-3) 

No adversarial setup is required — the loss occurs in the normal governance flow whenever more than one proposal exists at the time an update is executed.

### Recommendation

Add `proposer: AccountId` and `deposit: NearToken` fields to `UpdateEntry`. In `do_update`, before calling `self.entries.clear()`, iterate over the remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the pattern already used in `propose_update` for refunding excess deposits.

### Proof of Concept

1. Participant A calls `propose_update` with a large WASM binary, attaching deposit D_A (e.g., several NEAR for a 1 MB binary).
2. Participant B calls `propose_update` with a different WASM binary, attaching deposit D_B.
3. Threshold participants vote for proposal B; `vote_update` calls `do_update(&id_B, gas)`.
4. Inside `do_update`:
   - `self.entries.remove(&id_B)` — removes B's entry.
   - `self.entries.clear()` — silently drops A's entry, including all information needed to refund D_A.
   - `self.vote_by_participant.clear()` — clears votes.
5. D_A is now permanently locked in the contract. Participant A has no recourse. [3](#0-2) [5](#0-4)

    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }
