### Title
Unchecked `payload_hash`-to-Request Binding in `respond_verify_foreign_tx` Enables Cross-Request Replay by a Single Attested Participant - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over `response.payload_hash`, but never verifies that `response.payload_hash` is actually derived from the `request` argument supplied in the same call. A single attested MPC participant (strictly below the signing threshold) can replay a threshold signature produced for one pending request as the response to a completely different pending request, causing the victim caller to receive a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to the foreign-chain transaction they requested.

### Finding Description

In `respond_verify_foreign_tx` the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The check confirms only that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. It does **not** confirm that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request: <the request argument>, values: <any values> }))`.

The canonical payload hash is defined as:

```rust
pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
    let mut hasher = sha2::Sha256::new();
    borsh::BorshSerialize::serialize(self, &mut hasher)?;
    Ok(Hash256(hasher.finalize().into()))
}
``` [2](#0-1) 

The `ForeignTxSignPayloadV1` struct embeds the original `ForeignChainRpcRequest` and the extracted values:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [3](#0-2) 

Because the contract never re-derives or bounds `response.payload_hash` to the `request` key used to look up the pending yield, any attested participant can substitute an arbitrary `payload_hash` (and its matching signature) into the response for any pending request.

After the signature check passes, the response is immediately serialised and used to resume all queued yields for `request`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [4](#0-3) 

### Impact Explanation

A single attested MPC participant (below the signing threshold) can execute the following cross-request replay:

1. A legitimate `respond_verify_foreign_tx(request_A, { payload_hash_A, sig_A })` is submitted on-chain (or observed in the mempool). `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload::V1 { request_A, values_A }))` and `sig_A` is a valid threshold signature over it.
2. The attacker, who is an attested participant, submits `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })` for a different pending request `request_B`.
3. The contract accepts the call: `sig_A` is a valid ECDSA signature over `payload_hash_A` under the root key, so `signature_is_valid = true`.
4. The yield for `request_B` is resumed with `{ payload_hash_A, sig_A }`.
5. The caller of `request_B` receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes `request_A`'s foreign-chain data, not `request_B`'s.

Any bridge or NEAR contract that trusts the contract's response without independently recomputing the expected hash (using `ForeignChainSignatureVerifier::verify_signature` from `near-mpc-sdk`) will accept a forged attestation that a different foreign-chain transaction was verified. This enables cross-chain replay and forged foreign-chain verification, which can cause invalid bridge execution or double-spend conditions.

This is a **High** impact finding matching: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

The attacker requires only:
- Membership as a single attested MPC participant (below threshold).
- Visibility of any on-chain `respond_verify_foreign_tx` call (public NEAR transaction data).
- A concurrently pending `verify_foreign_transaction` request from a victim caller.

No threshold collusion, no key material, and no privileged operator access are required. The threshold signature used in the replay was produced honestly by the MPC network for a different request; the attacker merely redirects it. This is a realistic single-Byzantine-node attack within the stated threat model.

### Recommendation

The contract must bind `response.payload_hash` to the `request` argument before accepting the response. Since the contract cannot know the extracted values, the recommended fix is to include the `request` as a domain-separation prefix in the signed payload and verify the prefix on-chain. Concretely:

1. Change `ForeignTxSignPayloadV1` so that the `request` hash is independently verifiable (e.g., store a commitment to the request in the pending-request map at submission time).
2. In `respond_verify_foreign_tx`, recompute `expected_request_hash = SHA-256(borsh(request))` and verify that `response.payload_hash` is a hash of a payload whose embedded request matches (e.g., by requiring nodes to also submit the full `ForeignTxSignPayload` and hashing it on-chain, or by storing the request hash in the pending map and checking a prefix).
3. At minimum, add prominent documentation that callers **must** call `ForeignChainSignatureVerifier::verify_signature` from `near-mpc-sdk` and treat the contract's response as untrusted without that check.

### Proof of Concept

```
// Step 1: observe on-chain
respond_verify_foreign_tx(
    request = request_A,          // Bitcoin tx 0xAAAA...
    response = { payload_hash: H_A, signature: sig_A }
)
// H_A = SHA-256(borsh(ForeignTxSignPayload::V1 { request_A, [BlockHash(0x1234...)] }))
// sig_A = valid threshold ECDSA signature over H_A

// Step 2: victim submits a different request
verify_foreign_transaction(request_B)   // Bitcoin tx 0xBBBB...
// → pending_verify_foreign_tx_requests[request_B] = [yield_id_B]

// Step 3: attacker (single attested participant) replays sig_A for request_B
respond_verify_foreign_tx(
    request = request_B,
    response = { payload_hash: H_A, signature: sig_A }   // ← H_A is for request_A
)
// Contract checks: verify_ecdsa_signature(sig_A, H_A, root_pk) → OK
// Contract resolves yield_id_B with { payload_hash: H_A, signature: sig_A }

// Step 4: victim's contract receives forged response
// payload_hash encodes request_A's data, not request_B's
// If victim does not call ForeignChainSignatureVerifier::verify_signature,
// it accepts the attestation that Bitcoin tx 0xBBBB... was verified,
// when in fact tx 0xAAAA... was verified.
```
