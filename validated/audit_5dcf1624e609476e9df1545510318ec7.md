### Title
Stale Cross-Chain Oracle Rate Allows Over-Minting of wrsETH, Causing Protocol Insolvency — (`contracts/pools/RSETHPoolV2.sol`)

---

### Summary

`RSETHPoolV2.deposit()` mints wrsETH using a rate sourced from `CrossChainRateReceiver.getRate()`. That function returns the stored `rate` with no staleness check. The `lastUpdated` timestamp is recorded on every update but is never validated anywhere in the mint path. When the L1 rsETH/ETH rate rises and the L2 oracle lags, any depositor (not just a targeted attacker) receives more wrsETH than the ETH deposited will back on L1, making the wrsETH supply unbacked.

---

### Finding Description

`CrossChainRateReceiver.getRate()` unconditionally returns the stored `rate`: [1](#0-0) 

`lastUpdated` is written on every `lzReceive` call: [2](#0-1) 

but is **never read** by `getRate()` or by any caller.

`RSETHPoolV2.viewSwapRsETHAmountAndFee()` derives the wrsETH amount directly from this unchecked rate: [3](#0-2) 

`deposit()` calls `viewSwapRsETHAmountAndFee` and mints without any additional oracle freshness guard: [4](#0-3) 

The formula is:

```
rsETHAmount = (amount - fee) * 1e18 / rsETHToETHrate
```

A **lower** stale rate produces a **higher** `rsETHAmount`. The ETH is later bridged to L1 via `bridgeAssets()`, where L1 mints rsETH at the **current** (higher) rate, producing fewer rsETH tokens than wrsETH already outstanding. [5](#0-4) 

---

### Impact Explanation

**Critical — Protocol insolvency.**

Every deposit made while the oracle is stale mints more wrsETH than the corresponding L1 rsETH that will be created. The shortfall is permanent: the ETH is bridged and the L1 mint is fixed by the L1 rate. wrsETH holders who redeem last receive less rsETH than their wrsETH represents, or redemptions fail entirely once the backing is exhausted.

---

### Likelihood Explanation

This does **not** require oracle operator compromise or any active malicious action. It requires only:

1. The L1 rsETH/ETH rate to increase (a routine, expected event as staking rewards accrue).
2. The LayerZero cross-chain message to be delayed or not yet triggered (normal operational latency — the contract itself has no mechanism to force an update).
3. Any user to call `deposit()` during that window.

The `dailyMintLimit` modifier partially bounds the total exposure per day, but it uses the same stale rate to compute `rsETHAmount`, so the limit is itself inflated by the stale rate and does not prevent the insolvency. [6](#0-5) 

---

### Recommendation

Add a maximum staleness threshold. In `CrossChainRateReceiver.getRate()` (or in `RSETHPoolV2.getRate()`), revert if `block.timestamp - lastUpdated` exceeds an acceptable bound (e.g., 24 hours):

```solidity
function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate stale");
    return rate;
}
```

`MAX_STALENESS` should be set conservatively relative to the expected LayerZero update frequency.

---

### Proof of Concept

```solidity
// Fork test (L2 fork, e.g. Arbitrum)
// 1. Deploy / point to live RSETHPoolV2 with CrossChainRateReceiver as rsETHOracle
// 2. Record current oracle rate (e.g. 1.05e18)
// 3. Advance time by 2 days without triggering lzReceive (warp + no LZ message)
// 4. Simulate L1 rate rising to 1.10e18 (query L1 LRTOracle.rsETHPrice())
// 5. Attacker deposits 10 ETH on L2:
//      rsETHMinted = 10e18 * 1e18 / 1.05e18 ≈ 9.523e18 wrsETH
// 6. L1 would mint for same 10 ETH:
//      rsETHOnL1 = 10e18 * 1e18 / 1.10e18 ≈ 9.090e18 rsETH
// 7. Shortfall = 9.523e18 - 9.090e18 = 0.433e18 unbacked wrsETH
// 8. Assert wrsETH.totalSupply() > rsETH that L1 will produce → insolvency confirmed
```

The `lastUpdated` field is observable on-chain, so the staleness window is publicly detectable and the deposit can be timed precisely.
