### Title
Lack of Slippage Protection in `LRTWithdrawalManager.initiateWithdrawal()` Allows Users to Receive Fewer Assets Than Expected - (File: contracts/LRTWithdrawalManager.sol)

### Summary
`LRTWithdrawalManager.initiateWithdrawal()` does not accept a `minOutputAmount` parameter. The `expectedAssetAmount` is computed at request time using live oracle prices, but the actual payout is recalculated at `unlockQueue()` time — up to 8 days later — using potentially different prices. If rsETH price falls or the asset price rises during the delay, the user receives fewer assets than they anticipated, with no on-chain mechanism to reject the outcome.

### Finding Description
When a user calls `initiateWithdrawal()`, their rsETH is immediately transferred to the contract and an `expectedAssetAmount` is recorded based on the current oracle prices: [1](#0-0) 

Later, when an operator calls `unlockQueue()`, `_calculatePayoutAmount()` resolves the final disbursement as the **minimum** of the originally recorded `expectedAssetAmount` and the freshly computed `currentReturn`: [2](#0-1) 

This means the user's payout is asymmetric: if prices improve the user is capped at the original estimate, but if prices worsen the user silently receives less. The function signature provides no `minOutputAmount` guard: [3](#0-2) 

No cancellation path exists in the contract (confirmed by absence of any `cancelWithdrawal` function), so once rsETH is transferred in, the user is committed regardless of how prices move during the delay window.

By contrast, `LRTDepositPool.depositETH()` and `depositAsset()` both accept a `minRSETHAmountExpected` parameter and revert if the minted amount falls short: [4](#0-3) 

The withdrawal path has no equivalent protection.

### Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who initiates a withdrawal expecting 1 ETH worth of stETH may receive materially less if rsETH depreciates during the 8-day default delay (`withdrawalDelayBlocks = 8 days / 12 seconds`): [5](#0-4) 

The user receives the current fair value of their rsETH at unlock time — not an inflated or stolen amount — but they had no ability to set a floor and cannot cancel once committed.

### Likelihood Explanation
**Medium likelihood.** The default withdrawal delay is 8 days, a window long enough for meaningful rsETH/asset price divergence. The oracle prices used (`lrtOracle.rsETHPrice()` and `lrtOracle.getAssetPrice(asset)`) reflect real EigenLayer restaking valuations that can shift due to slashing events, LST depegs, or protocol losses — exactly the scenario described in the reference report. Any unprivileged rsETH holder can trigger this path. [6](#0-5) 

### Recommendation
Add a `minExpectedAssetAmount` parameter to `initiateWithdrawal()` and revert if `expectedAssetAmount < minExpectedAssetAmount`, mirroring the pattern already used in `LRTDepositPool._beforeDeposit()`. Optionally, allow users to cancel a pending withdrawal request and reclaim their locked rsETH if the unlock-time payout would fall below their stated minimum.

### Proof of Concept
1. rsETH/ETH oracle price = 1.05 ETH per rsETH. Alice calls `initiateWithdrawal(stETH, 100e18, "")`. `expectedAssetAmount` is stored as `≈100 stETH`.
2. Over the next 8 days a slashing event reduces rsETH/ETH to 0.90. Alice cannot cancel.
3. Operator calls `unlockQueue(stETH, ...)`. `_calculatePayoutAmount` computes `currentReturn = 100e18 * 0.90e18 / stETHPrice ≈ 85.7 stETH`. Since `85.7 < 100`, Alice's `expectedAssetAmount` is overwritten to `85.7 stETH`.
4. Alice calls `completeWithdrawal(stETH, "")` and receives `85.7 stETH` — ~14.3% less than she expected — with no recourse. [7](#0-6)
