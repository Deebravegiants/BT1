Audit Report

## Title
`initiateWithdrawal` Availability Check Uses Total Protocol Assets While `unlockQueue` Uses Only Vault Balance, Causing Temporary rsETH Freeze — (`contracts/LRTWithdrawalManager.sol`)

## Summary

`getAvailableAssetAmount` computes availability using `getTotalAssetDeposits`, which aggregates all protocol assets including illiquid EigenLayer positions, NDC balances, and converter balances. However, `unlockQueue` can only service requests using `unstakingVault.balanceOf(asset)` — the liquid vault balance. This accounting mismatch allows `initiateWithdrawal` to accept requests that `unlockQueue` cannot service, locking the user's rsETH in `LRTWithdrawalManager` with no cancel path until operators manually complete an EigenLayer withdrawal (7-day mandatory delay).

## Finding Description

In `initiateWithdrawal`, the gate check is:

```solidity
// LRTWithdrawalManager.sol L168-173
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` (L599-603) calls `lrtDepositPool.getTotalAssetDeposits(asset)`, which (L385-397) sums `assetLyingInDepositPool + assetLyingInNDCs + assetStakedInEigenLayer + assetUnstakingFromEigenLayer + assetLyingInConverter + assetLyingUnstakingVault`. Under normal operation, the dominant term is `assetStakedInEigenLayer`, making `getAvailableAssetAmount` return a value close to the full protocol TVL.

When `unlockQueue` is called, `_createUnlockParams` (L846-851) sets `totalAvailableAssets = unstakingVault.balanceOf(asset)` — only the liquid vault balance. Inside `_unlockWithdrawalRequests` (L800):

```solidity
if (availableAssetAmount < payoutAmount) break;
```

If the vault is empty (the normal state), `availableAssetAmount = 0`, the loop breaks immediately, `nextLockedNonce` is not advanced, and the request remains locked. `completeWithdrawal` (L707) reverts with `WithdrawalLocked` for any nonce not past `nextLockedNonce[asset]`. A grep search confirms no `cancelWithdrawal` function exists anywhere in the codebase, leaving the user's rsETH with no recovery path until operators manually unstake from EigenLayer.

## Impact Explanation

**Temporary freezing of funds (Medium).** A user's rsETH is transferred to `LRTWithdrawalManager` at `initiateWithdrawal` time and cannot be recovered until operators queue an EigenLayer withdrawal (7-day mandatory delay), complete it, move assets to the unstaking vault, and call `unlockQueue`. During this period the user cannot transfer, use, or recover their rsETH. The freeze is temporary (not permanent) because the operator path exists, but it is entirely outside the user's control and imposes a mandatory multi-day delay with no user-initiated escape hatch.

## Likelihood Explanation

High. The normal operational state of the protocol is that the vast majority of assets are delegated to EigenLayer strategies via NodeDelegators. The unstaking vault holds assets only transiently — between EigenLayer withdrawal completion and queue unlock. Any user calling `initiateWithdrawal` when the vault is empty (the common case) triggers this condition. No attacker capability is required; any ordinary user performing a routine withdrawal is affected.

## Recommendation

Align the availability check in `initiateWithdrawal` with the actual liquid assets that `unlockQueue` can service. Replace the `getTotalAssetDeposits` call in `getAvailableAssetAmount` with `unstakingVault.balanceOf(asset)`, or introduce a separate "liquid available" check that gates `initiateWithdrawal` on vault liquidity rather than total protocol TVL. Additionally, implement a `cancelWithdrawal` function that returns the user's rsETH if their request has not been unlocked within a configurable timeout, providing a user-controlled escape hatch.

## Proof of Concept

1. Protocol state: 1,000 ETH staked in EigenLayer via NodeDelegators, 0 ETH in `LRTUnstakingVault`, `assetsCommitted[ETH] = 0`.
2. User calls `initiateWithdrawal(ETH, rsETHAmount)` where `getExpectedAssetAmount` returns 500 ETH.
3. `getAvailableAssetAmount(ETH)` returns `getTotalAssetDeposits(ETH) - 0 = 1000 ETH` → check passes.
4. `assetsCommitted[ETH] += 500 ETH`; user's rsETH is transferred to `LRTWithdrawalManager`.
5. Operator calls `unlockQueue(ETH, ...)`.
6. `_createUnlockParams` sets `totalAvailableAssets = unstakingVault.balanceOf(ETH) = 0`.
7. `_unlockWithdrawalRequests`: `availableAssetAmount (0) < payoutAmount (500 ETH)` → `break` at L800.
8. `nextLockedNonce` is not advanced; request remains locked.
9. User calls `completeWithdrawal(ETH, ...)` → reverts with `WithdrawalLocked` (L707).
10. No `cancelWithdrawal` exists; user's rsETH is frozen until operators complete a 7-day EigenLayer unstaking cycle.

**Foundry fork test plan:** Fork mainnet, deploy/configure the protocol, stake ETH via NodeDelegators to drain the unstaking vault, call `initiateWithdrawal` as an unprivileged user, assert `assetsCommitted` increased and rsETH transferred, call `unlockQueue` with vault balance = 0, assert `nextLockedNonce` unchanged, assert `completeWithdrawal` reverts with `WithdrawalLocked`.
