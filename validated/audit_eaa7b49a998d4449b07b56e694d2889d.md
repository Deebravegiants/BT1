### Title
`_fillBids` subtracts the wrong bin's `lengthE6` when walking downward, corrupting `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` for all bid bins below `curBinIdx` when bins have heterogeneous lengths — (`metric-periphery/contracts/lens/MetricOmmPoolDataProvider.sol`)

### Summary

`MetricOmmPoolDataProvider._fillBids` walks downward from `curBinIdx` to compute the bid depth ladder. For each bin `b < curBinIdx`, it subtracts `lengthE6` of bin `b+1` (the bin above) from `walkDistE6` to derive the lower boundary of bin `b`. The correct subtraction is `lengthE6` of bin `b` itself. When bins have different `lengthE6` values, every `lowerX64` and `upperX64` below the current bin is wrong, corrupting the `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` returned to integrators.

### Finding Description

In `_fillBids`, `walkDistE6` is initialized to `curBinDistFromProvidedPriceE6`, the lower boundary of `curBinIdx`. The loop walks downward:

```solidity
for (int256 b = int256(curBinIdx) - 1; b >= int256(lowCap); b--) {
    int8 binIdx = int8(b);
    (,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1)); // reads bin b+1
    walkDistE6 -= int256(uint256(lenAbove));                                 // subtracts b+1's length

    (, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);
    ...
    uint256 lowerX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6, ...);
    uint256 upperX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6 + int256(uint256(lengthE6)), ...);
``` [1](#0-0) 

The invariant for contiguous bins is:

```
lower_boundary(b) = lower_boundary(b+1) - lengthE6(b)
```

The code instead computes:

```
walkDistE6 = lower_boundary(b+1) - lengthE6(b+1)
```

which equals `lower_boundary(b)` only when `lengthE6(b) == lengthE6(b+1)`. When bins have different lengths, `walkDistE6` is wrong, and so are `lowerX64`, `upperX64`, `mStartX64`, `mEndX64`, and ultimately `binAvgExecPriceX64` / `cumulativeAvgExecPriceX64`.

**Contrast with `_fillAsks`**, which correctly advances `cumDistE6` by the current bin's own length:

```solidity
ctx.cumDistE6 += int256(uint256(lengthE6)); // lengthE6 of bin b, not b+1
``` [2](#0-1) 

**Concrete example:**

| Bin | `lengthE6` | Correct lower boundary | Code computes |
|-----|-----------|----------------------|---------------|
| `curBinIdx=0` | 10 000 | 0 | 0 (correct) |
| `-1` | 20 000 | −20 000 | −10 000 (off by 10 000) |
| `-2` | 5 000 | −25 000 | −30 000 (off by −5 000) |

The error accumulates and changes sign across bins, making the bid ladder non-monotone and internally inconsistent.

### Impact Explanation

`getLiquidityDepth` is the primary off-chain and on-chain integrator interface for reading bid depth. The corrupted `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` values are the direct inputs integrators use to choose `priceLimitX64` before calling `swap`. With wrong prices:

- If the reported bid price is **lower** than actual, the integrator sets `priceLimitX64` below the true execution price → the swap executes at a price worse than the integrator intended, causing direct loss of token1 output relative to the integrator's expectation.
- If the reported bid price is **higher** than actual, the integrator sets `priceLimitX64` above the true execution price → the swap reverts, breaking core swap functionality.

The direction of the error is pool-configuration-dependent (sign of `lengthE6(b+1) - lengthE6(b)`), so both outcomes are reachable in normal pool operation. The "bad-price execution" impact gate is met for the first case. [3](#0-2) 

### Likelihood Explanation

- Bins with heterogeneous `lengthE6` are a normal, supported pool configuration — the factory does not require uniform bin lengths.
- Any integrator (router, aggregator, UI) calling `getLiquidityDepth` on such a pool is affected.
- No privileged access or malicious setup is required; the bug triggers on any public `getLiquidityDepth` call.

### Recommendation

Replace the two-call pattern (read `b+1`, subtract its length) with a single read of the current bin `b`'s length and subtract that:

```solidity
// Before (wrong):
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1));
walkDistE6 -= int256(uint256(lenAbove));
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);

// After (correct):
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);
walkDistE6 -= int256(uint256(lengthE6));
```

This mirrors the correct pattern in `_fillAskRow` where `ctx.cumDistE6 += int256(uint256(lengthE6))` uses the current bin's own length.
