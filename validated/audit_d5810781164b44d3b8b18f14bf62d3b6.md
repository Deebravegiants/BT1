### Title
Unverified Cairo0 Class Declaration in `ExecutionStateReader::get_compiled_class` Allows Execution Against Undeclared Classes — (File: `crates/apollo_rpc_execution/src/state_reader.rs`)

### Summary

`ExecutionStateReader::get_compiled_class` enforces a state-number–aware `is_declared` guard for Cairo1 classes but explicitly skips it for Cairo0 (deprecated) classes, with a `TODO` comment acknowledging the gap. Because the class manager stores classes globally without state-number awareness, a Cairo0 class that is present in the class manager but was not declared at the queried `state_number` (e.g., declared in a later block or in a reverted block) is returned and used for execution without any validation. This produces authoritative-looking wrong values from RPC execution, fee estimation, and simulation endpoints.

### Finding Description

In `crates/apollo_rpc_execution/src/state_reader.rs`, the `get_compiled_class` function handles two branches when `class_manager_handle` is `Some`:

**Cairo1 path (guarded):**
```rust
ContractClass::V1(casm_contract_class) => {
    let is_declared = is_contract_class_declared(
        &self.storage_reader.begin_ro_txn()...,
        &class_hash,
        self.state_number,   // ← state-number check
    )?;
    if is_declared {
        Ok(RunnableCompiledClass::V1(casm_contract_class.try_into()?))
    } else {
        Err(StateError::UndeclaredClassHash(class_hash))
    }
}
```

**Cairo0 path (unguarded):**
```rust
// TODO(shahak): Verify cairo0 as well after get_class_definition_block_number is
// fixed.
ContractClass::V0(deprecated_contract_class) => {
    Ok(RunnableCompiledClass::V0(deprecated_contract_class.try_into()?))
}
``` [1](#0-0) 

The class manager (`apollo_class_manager`) is a persistent, state-number–agnostic store. It does not track at which block a class was declared. When an RPC query is made at a historical `state_number` (e.g., `starknet_call` at block N), the `ExecutionStateReader` is constructed with `state_number = right_after_block(N)`. For Cairo1 classes, `is_contract_class_declared` cross-checks the storage at that `state_number` to confirm the class was actually declared by block N. For Cairo0 classes, this cross-check is entirely absent.

The storage layer does have the correct function for deprecated classes (`get_deprecated_class_definition_block_number`), but it is not called here: [2](#0-1) 

The `get_compiled_class_hash` function in the same file also returns `CompiledClassHash::default()` (zero) for both Cairo0 classes and undeclared classes, collapsing two distinct states into one indistinguishable default: [3](#0-2) 

### Impact Explanation

**High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

The `ExecutionStateReader` is the state backend for `starknet_call`, `starknet_estimateFee`, and `starknet_simulateTransactions`. If a Cairo0 class is present in the class manager but was not declared at the queried block (e.g., declared in a later block, or declared in a block that was subsequently reverted during a chain reorg), the RPC will execute against that class as if it were declared, returning a wrong execution result, wrong fee estimate, or wrong simulation trace. The caller receives a well-formed, authoritative-looking response with no indication that the class was not actually declared at the queried state.

### Likelihood Explanation

**Low-Medium.** The most realistic trigger is a chain reorg: a Cairo0 class declared in a reverted block remains in the class manager (which is not rolled back atomically with storage), but the storage no longer shows it as declared. Any subsequent historical RPC query at a `state_number` before the original declaration block will incorrectly use the class. A second trigger is a node that has partially synced the class manager ahead of storage. Both scenarios are operational edge cases rather than adversarial inputs, but they are reachable without privileged access.

### Recommendation

Replace the unguarded Cairo0 branch with a declaration check analogous to the Cairo1 branch. Use `get_deprecated_class_definition_block_number` (already present in storage) to verify the class was declared at or before `self.state_number`:

```rust
ContractClass::V0(deprecated_contract_class) => {
    let block_number = self
        .storage_reader
        .begin_ro_txn()?
        .get_state_reader()?
        .get_deprecated_class_definition_block_number(&class_hash)?;
    match block_number {
        Some(bn) if StateNumber::right_after_block(bn) <= self.state_number => {
            Ok(RunnableCompiledClass::V0(deprecated_contract_class.try_into()?))
        }
        _ => Err(StateError::UndeclaredClassHash(class_hash)),
    }
}
```

Remove the TODO comment once the fix is in place.

### Proof of Concept

1. Sync a node to block N+2. At block N+1 a Cairo0 class `C` was declared. At block N+2 a reorg reverts block N+1, removing the declaration from storage. The class manager still holds `C`.
2. Issue `starknet_call` at `block_id = N+2` (post-reorg tip) targeting a contract whose class hash is `C` (set in storage before the reorg).
3. `get_class_hash_at` returns `C` from storage (the contract was deployed before the reorg).
4. `get_compiled_class(C)` calls `class_manager_client.get_executable(C)` → returns `ContractClass::V0(...)` because the class manager was not rolled back.
5. The Cairo0 branch returns `Ok(RunnableCompiledClass::V0(...))` **without** calling `get_deprecated_class_definition_block_number`, bypassing the undeclared-class guard.
6. The call executes against the reverted class and returns a result. The correct behavior is `StateError::UndeclaredClassHash(C)`, which would surface as a contract-not-found error to the caller.

The corrupted value is the RPC execution result: a successful call or fee estimate that should have been an error, because the class used for execution was not declared at the queried state.
