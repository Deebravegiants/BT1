### Title
Cross-Epoch Vote Replay in `vote_add_domains` Allows Stale Votes to Trigger Unauthorized State Transition - (File: `crates/contract/src/state/running.rs`)

---

### Summary

`vote_add_domains` does not bind votes to the current epoch. Unlike `vote_new_parameters`, which requires a `prospective_epoch_id` to prevent stale votes from counting across epoch boundaries, `vote_add_domains` stores votes indefinitely in `add_domains_votes` and never invalidates them on epoch change. After a resharing that reduces the participant set, stale votes from the previous epoch cast by participants who remain in the new set persist and count toward the new (smaller) quorum, allowing a single current participant to trigger an unauthorized `Running → Initializing` state transition by casting one new vote that combines with old votes to satisfy the new all-participant threshold.

---

### Finding Description

`vote_new_parameters` explicitly guards against cross-epoch vote reuse:

> "The epoch_id must be equal to 1 plus the current epoch ID… This is to prevent late transactions from accidentally voting on outdated proposals." [1](#0-0) 

`vote_add_domains` has no such guard. Its public entry point accepts only a `Vec<DomainConfig>` with no epoch identifier: [2](#0-1) 

Internally, the transition fires when the vote count equals the **current** participant count: [3](#0-2) 

Votes are stored in `add_domains_votes`, a `BTreeMap<AuthenticatedParticipantId, Vec<DomainConfig>>`. After resharing, only votes from accounts that lost participant status are pruned; votes from accounts that remain are kept intact: [4](#0-3) 

This is confirmed by the sandbox test `add_domain_votes_from_kicked_out_participants_are_cleared_after_resharing`, which explicitly asserts that one stale vote from a surviving participant persists after resharing: [5](#0-4) 

**Attack path:**

1. Epoch N — participants {A, B, C}, all-participant threshold = 3. Participant B votes for a domain addition proposal P. Vote count = 1 (need 3). No transition.
2. A resharing occurs (e.g., C's TEE attestation expires and `verify_tee` kicks C out). New participant set = {A, B}, all-participant threshold = 2.
3. After resharing, B's vote for P is still in `add_domains_votes` (B is still a participant).
4. A calls `vote_add_domains(P)`. A's vote is inserted; `n_votes` for P = 2 (A's new vote + B's stale epoch-N vote). The check `participants().len() == n_votes` → `2 == 2` → **transition to `Initializing` fires**.

B's vote from epoch N is effectively replayed in epoch N+1 without B re-consenting in the new epoch context.

---

### Impact Explanation

The unauthorized `Running → Initializing` transition:

- Forces the MPC network to begin distributed key generation for domains that participants did not explicitly approve in the current epoch.
- Disrupts the `Running` state, during which no new signing requests can be processed (the contract is in `Initializing`).
- If key generation completes, new cryptographic domains are permanently added to the network without current-epoch unanimous consent.
- Breaks the production safety invariant that all governance transitions require fresh, epoch-bound participant agreement.

This matches the **Medium** allowed impact: *"contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The preconditions are realistic in production:

- Participant removal via TEE attestation expiry is an expected operational event (`verify_tee` is designed to trigger resharing on expiry).
- Participants routinely cast `vote_add_domains` votes that do not immediately reach quorum (e.g., one participant votes early, others have not yet voted).
- After resharing reduces the participant count, any single remaining participant who previously voted can trigger the transition unilaterally by re-submitting the same proposal.
- The attacker need only be a legitimate current participant — no threshold collusion, no key leakage, no privileged access required.

---

### Recommendation

Bind `vote_add_domains` votes to the current epoch, mirroring the protection already present in `vote_new_parameters`. Add an `epoch_id` parameter and validate it against the current epoch before recording the vote. Clear `add_domains_votes` entirely on every epoch transition (resharing completion), not just for removed participants.

---

### Proof of Concept

```
State: Running, epoch_id=5, participants={A,B,C}, threshold=3

Step 1: B calls vote_add_domains([DomainConfig{id:3, protocol:Frost, ...}])
  → add_domains_votes = {B: [DomainConfig{id:3,...}]}
  → n_votes=1, 1 != 3, no transition

Step 2: verify_tee() detects C's expired attestation, triggers resharing
  → resharing completes, new epoch_id=6, participants={A,B}, threshold=2
  → add_domains_votes pruned: C had no vote, so map unchanged
  → add_domains_votes = {B: [DomainConfig{id:3,...}]}  ← B's epoch-5 vote persists

Step 3: A calls vote_add_domains([DomainConfig{id:3, protocol:Frost, ...}])
  → add_domains_votes = {A: [...], B: [...]}
  → n_votes=2, participants().len()=2, 2==2 → TRANSITION TO INITIALIZING
  → Key generation for domain 3 begins without B's explicit epoch-6 consent
```

The `vote_add_domains` entry point accepts the call with no epoch check: [6](#0-5) 

The transition condition uses the live participant count against the stale vote tally: [7](#0-6)
