### Title
Single-Provider Quorum Bypass in `FanOut::extract` — (`crates/foreign-chain-inspector/src/lib.rs`)

---

### Summary

`FanOut::extract` accepts a single `Ok` result from one RPC provider as authoritative when all remaining N-1 providers return transient errors (`NotFinalized`, `ClientError`, etc.). There is no minimum-quorum guard: one success is sufficient to return `Ok(values)` regardless of how many providers are configured. A compromised or malicious RPC provider that is shared across threshold-many MPC nodes can unilaterally determine the signed payload during any window of transient unavailability of the other providers.

---

### Finding Description

The aggregation logic in `FanOut::extract` partitions results into three buckets — successes, non-transient errors, and transient errors — and then applies the following decision tree: [1](#0-0) 

1. If `successes` is non-empty **and** `non_transient_errors` is empty, it checks that all successes agree with each other, then returns the first one.
2. There is **no check** on `successes.len()` against any quorum threshold.

`is_transient()` classifies `ClientError`, `RpcRequestFailed`, `NotFinalized`, and `NotEnoughBlockConfirmations` as transient: [2](#0-1) 

So with N=3 providers where provider-0 returns `Ok(fabricated_values)` and providers 1 and 2 return `Err(NotFinalized)`:
- `successes = [(0, fabricated_values)]`
- `non_transient_errors = []`
- `inspectors_split_between_success_and_failure = false`
- `all_successes_agree = true` (trivially, only one entry)
- **Returns `Ok(fabricated_values)`**

This behavior is not a bug in the test suite — it is explicitly tested and confirmed as the intended behavior: [3](#0-2) 

The design documentation acknowledges this gap. The `rpc_quorum` field (`ChainEntry.quorum`) is a **planned but unimplemented** feature: [4](#0-3) 

The doc explicitly states the quorum enforcement is an "Implementation requirement, not current behavior" (tracked in issue #3477): [5](#0-4) 

---

### Impact Explanation

Each MPC node independently runs `FanOut::extract` to verify a foreign-chain transaction before producing a signature share. If a compromised provider is shared across ≥ `signing_threshold` nodes (a common deployment pattern where all nodes use the same third-party provider, e.g., Alchemy or QuickNode), and all other providers of those nodes are transiently unavailable simultaneously, then every affected node independently accepts the fabricated values and produces a signature share. The threshold is met and an unauthorized MPC signature is issued over attacker-fabricated EVM observations (e.g., a fake log event, a fake block hash), enabling forgery of foreign-chain attestations and invalid bridge execution.

Impact category: **High** — forged foreign-chain verification / light-client-style verification bypass enabling invalid bridge execution.

---

### Likelihood Explanation

- **Compromised shared provider**: Realistic. Operators commonly configure the same well-known RPC providers (Alchemy, QuickNode, Infura). A single provider compromise affects all nodes using it.
- **Transient unavailability window**: Realistic. Network partitions, rate-limiting, provider maintenance, or a targeted slowdown of the non-compromised providers can create a window where only the attacker-controlled provider responds successfully.
- **No operator action required**: The attacker does not need any privileged access to the MPC network, NEAR contract, or TEE. The attack is entirely external, through the RPC layer.

---

### Recommendation

Enforce a minimum quorum on the number of agreeing successes before accepting a result. The planned `ChainEntry.quorum` field should be wired into `FanOut` so that `successes.len() >= quorum` is required before returning `Ok`. Until the on-chain whitelist and quorum field are implemented, a configurable `min_successes: usize` parameter should be added to `FanOut` and enforced at lines 130–142:

```rust
if successes.len() < self.min_successes {
    return Err(ForeignChainInspectionError::InspectorResponseMismatch);
}
```

### Proof of Concept

The existing test `fan_out__should_tolerate_transient_when_only_one_inspector_succeeds` already proves the bypass: [3](#0-2) 

A targeted PoC with fabricated values:

```rust
#[tokio::test]
async fn poc__single_provider_bypasses_quorum_with_fabricated_values() {
    let fabricated = vec![0xdeadbeef_u32]; // attacker-chosen extracted value
    let attacker = mock_returning(ok(fabricated.clone()));
    let transient = || mock_returning(err(|| ForeignChainInspectionError::NotFinalized));
    // N=3: attacker controls 1, other 2 return transient errors
    let fan_out = fan_out_of(vec![transient(), attacker, transient()]);

    let result = fan_out.extract((), (), vec![]).await;

    // Single provider unilaterally determines the signed payload
    assert_eq!(result.unwrap(), fabricated);
}
```

This test passes against the current production code, demonstrating that the RPC quorum requirement is not enforced.

## Verification behavior

Each node fans the query out to its whitelisted providers for `C` and accepts a
result only when ≥ `rpc_quorum(C)` return the same response. If fewer agree, the node
errors out and produces no signature share.

**This sub-quorum outcome must be terminal — the leader must not re-attempt the
request.** Implementation requirement, not current behavior: the generic queue
retries every request, so the foreign-tx path must special-case a sub-quorum
result as non-retryable. (Open: whether a sub-quorum from purely *transient*
failures — timeouts, finality not reached — should still retry, vs. only genuine
disagreement being terminal. Tracked in [#3477](https://github.com/near/mpc/issues/3477).)
