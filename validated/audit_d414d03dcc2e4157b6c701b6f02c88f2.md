### Title
`get_storage_at` Returns `CONTRACT_NOT_FOUND` Instead of `0` for Pending-Only Contracts with Zero Storage — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

When a contract is deployed only in the pending block and a caller queries `get_storage_at` for a key whose pending value is `Felt::default()` (zero), the RPC incorrectly returns `CONTRACT_NOT_FOUND` instead of `0`. The zero-value existence check consults only the accepted-state reader, ignoring pending deployed contracts.

### Finding Description

In `get_storage_at` (lines 339–384 of `api_impl.rs`), the flow is:

**Step 1 — Pending storage diffs are fetched correctly:**
When `BlockId::Tag(Tag::Pending)` is requested and the pending block is up to date (parent hash matches latest accepted block), `read_pending_data` returns the full pending data including `state_update.state_diff.storage_diffs`. [1](#0-0) [2](#0-1) 

**Step 2 — `execution_utils::get_storage_at` returns zero from pending diffs:**
If the contract address and key are found in the pending diffs with value `Felt::default()`, the function returns `Ok(Felt::default())` immediately, before ever touching the accepted state. [3](#0-2) 

**Step 3 — The zero-value existence check only consults the accepted state:**
When `res == Felt::default()`, the code checks whether the contract exists by calling `txn.get_state_reader().get_class_hash_at(state_number, &contract_address)`. This reader is bound to the accepted-block `StateNumber` and has no knowledge of pending deployed contracts. [4](#0-3) 

If the contract was deployed only in the pending block, `get_class_hash_at` returns `None`, and `CONTRACT_NOT_FOUND` is returned — even though the correct answer is `0`.

**Contrast with `maybe_get_class_hash_at`:**
The sibling helper `maybe_get_class_hash_at` correctly passes pending deployed contracts and replaced classes to `execution_utils::get_class_hash_at`, which checks the pending state first. The zero-value guard in `get_storage_at` does not use this helper. [5](#0-4) [6](#0-5) 

**Existing test gap:**
The existing test at lines 2121–2141 covers only the stale-pending-block case (where `CONTRACT_NOT_FOUND` is the correct result). There is no test for the up-to-date pending block case with a pending-only contract and a zero storage value. [7](#0-6) 

### Impact Explanation

Any RPC client calling `starknet_getStorageAt` with `block_id = "pending"` for a contract deployed in the current pending block will receive `CONTRACT_NOT_FOUND` for any storage key whose value is zero (the default for all keys). Since all storage keys of a freshly deployed contract are zero, this affects every newly deployed contract in the pending block. The RPC response is authoritative-looking and wrong — callers cannot distinguish this from a genuine missing contract.

**Impact category:** High — RPC pending view returns an authoritative-looking wrong value.

### Likelihood Explanation

The trigger condition is common: any contract deployed in the pending block (via a `deploy_account` or `deploy` syscall) will have all storage keys at zero by default. An unprivileged caller needs only to submit such a transaction and then call `get_storage_at` for any key on that contract before the block is accepted.

### Recommendation

Replace the raw `txn.get_state_reader().get_class_hash_at(state_number, &contract_address)` call in the zero-value guard with a call to `execution_utils::get_class_hash_at`, passing the pending deployed contracts and replaced classes from the pending state diff — exactly as `maybe_get_class_hash_at` already does. This ensures that a contract deployed only in the pending block is recognized as existing, and `0` is returned instead of `CONTRACT_NOT_FOUND`.

### Proof of Concept

1. Write a block with one accepted contract (so the pending block has a valid parent hash).
2. Populate `pending_data` with:
   - `state_update.state_diff.deployed_contracts`: `[{address: 0x1234, class_hash: 0xABCD}]`
   - `state_update.state_diff.storage_diffs`: `{0x1234: [{key: 0x1001, value: Felt::default()}]}`
   - `block.parent_block_hash`: set to the accepted block's hash (so the pending block is up to date).
3. Call `get_storage_at(0x1234, 0x1001, BlockId::Tag(Tag::Pending))`.
4. **Observed:** `CONTRACT_NOT_FOUND`.
5. **Expected:** `Felt::default()` (zero).

The existing test at line 2121 already demonstrates the mechanics but only for the stale-pending case. Running the same test with the parent hash set correctly (step 2 above) would reproduce the bug.

    txn: &StorageTxn<'_, Mode>,
    state_number: StateNumber,
    pending_deployed_contracts_and_replaced_classes: Option<(
        &Vec<DeployedContract>,
        &Vec<ReplacedClass>,
    )>,
    contract_address: ContractAddress,
) -> StorageResult<Option<ClassHash>> {
    if let Some((pending_deployed_contracts, pending_replaced_classes)) =
        pending_deployed_contracts_and_replaced_classes
    {
        // Searching first in the replaced classes because if the contract was deployed and
        // replaced, the replaced class is the contract's class.
        for ReplacedClass { address, class_hash } in pending_replaced_classes {
            if *address == contract_address {
                return Ok(Some(*class_hash));
            }

                return Ok(Some(*class_hash));
