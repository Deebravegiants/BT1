### Title
Missing RSA Modulus Structure Validation in `base_pke_rsa_ek_from_modulus` Allows Trivially Factorable Modulus to Be Accepted as Valid PVE Encryption Key — (`src/cbmpc/api/pve_base_pke.cpp`)

### Summary

`base_pke_rsa_ek_from_modulus` is a public API that builds a PVE RSA-OAEP encryption key blob from a caller-supplied modulus. The only validation performed on the modulus is a size check (exactly 256 bytes) and a zero check. No RSA structural validation is performed — specifically, there is no check that the modulus is composite, odd-semiprime, or has the correct bit-length distribution. An attacker who can supply the modulus bytes (e.g., as a malicious serialized-input provider or malicious HSM operator) can pass a 256-byte prime `p` instead of a valid `p*q`, obtain a successful return, and then decrypt any PVE ciphertext encrypted under the resulting key, because `phi(p) = p-1` is trivially computable.

### Finding Description

**Entrypoint**: `coinbase::api::pve::base_pke_rsa_ek_from_modulus` / `cbmpc_pve_base_pke_rsa_ek_from_modulus`

**Validation gap** in `src/cbmpc/api/pve_base_pke.cpp` lines 147–153:

```cpp

if (!n_bn) return coinbase::error(E_GENERAL, "BN_bin2bn(modulus) failed");

if (BN_is_zero(n_bn)) {
    BN_free(n_bn);
    return coinbase::error(E_BADARG, "modulus must not be zero");

```

Only zero is rejected. Any non-zero 256-byte value — including a 2048-bit prime — passes. [1](#0-0) 

The value is then passed directly to `rsa_pub_key_t::set(n_bn, e_bn)`, which calls `EVP_PKEY_fromdata` with `EVP_PKEY_PUBLIC_KEY`. OpenSSL's `EVP_PKEY_fromdata` for an RSA public key does not check whether `n` is a product of two primes — it only stores `(n, e)`. [2](#0-1) 

The resulting `rsa_pub_key_t` is stored in a `base_pke_ek_blob_v1_t` and returned as `out_ek` with `SUCCESS`. [3](#0-2) 

**Encryption path**: `kem_policy_rsa_oaep_t::encapsulate` generates a random 32-byte `kem_ss` and encrypts it via `pub_key.encrypt_oaep_with_seed` / `encrypt_oaep`. [4](#0-3) 

`encrypt_oaep_with_seed` applies OAEP padding then calls `encrypt_raw`, which performs `EVP_PKEY_encrypt` with `RSA_NO_PADDING` — i.e., raw modular exponentiation `m^e mod n`. This operation is mathematically valid for any modulus, including a prime. [5](#0-4) 

No check at any layer (`base_pke_rsa_ek_from_modulus`, `rsa_pub_key_t::set`, `kem_policy_rsa_oaep_t::encapsulate`, `encrypt_oaep`) validates that `n` is a valid RSA modulus.

### Impact Explanation

For a prime modulus `p`, `phi(p) = p - 1`. The attacker who chose `p` can compute `d = e^{-1} mod (p-1)` and recover `kem_ss = kem_ct^d mod p`. From `kem_ss`, the HKDF-derived AES-GCM key is fully determined, and the AES-GCM ciphertext (containing the private key scalar `x`) is decryptable. [6](#0-5) 

The PVE scheme is used to encrypt private key scalars (e.g., ECDSA/EdDSA key shares). Full recovery of `x` from a PVE ciphertext is a Critical-level outcome.

### Likelihood Explanation

The function is explicitly designed for HSM use cases where the modulus is supplied externally. A malicious HSM operator, a man-in-the-middle on the modulus export channel, or any API caller who can influence what modulus is passed to `base_pke_rsa_ek_from_modulus` can execute this attack. The `encrypt` API also accepts the ek blob as an opaque `mem_t`, so a malicious serialized-input provider who substitutes the ek blob achieves the same result without even calling `base_pke_rsa_ek_from_modulus` directly. [7](#0-6) 

### Recommendation

Add RSA modulus structural validation in `base_pke_rsa_ek_from_modulus` before calling `pk.set`:

1. **Minimum**: Verify the modulus is odd and has exactly 2048 bits (top bit set). A prime of exactly 2048 bits is odd and has the top bit set, so this alone is insufficient.
2. **Sufficient**: Run a Miller-Rabin primality test (`BN_check_prime` / `BN_is_prime_fasttest_ex`) and **reject** if the modulus is prime.
3. **Stronger**: Verify the modulus has exactly two large prime factors (e.g., by checking that it is not a perfect power and that its smallest prime factor is large), consistent with RSA-2048 key generation requirements.

The ECIES counterpart `base_pke_ecies_p256_ek_from_oct` correctly validates the public key with `curve_p256.check(pk)` after `pk.from_oct(...)`. The RSA path should apply equivalent rigor. [8](#0-7)
