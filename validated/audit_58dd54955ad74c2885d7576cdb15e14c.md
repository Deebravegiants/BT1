### Title
`authorized_declarer_accounts` / `block_declare` Restriction Enforced Only at Gateway, Bypassed via Consensus Proposal Path — (`crates/apollo_gateway/src/gateway.rs`)

---

### Summary

The gateway enforces a `check_declare_permissions` guard that rejects `Declare` transactions from senders not in `authorized_declarer_accounts`, or rejects all `Declare` transactions when `block_declare = true`. This guard exists exclusively in the gateway's RPC entry point. The consensus proposal-validation path — `validate_proposal` → `handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal` — contains no equivalent check. A proposer node can therefore include a `Declare` transaction from any sender in a proposal; every validator will accept and execute it, committing an unauthorized class declaration to the chain.

---

### Finding Description

**Entry-point guard (gateway only):**

`check_declare_permissions` is called inside `GenericGateway::add_tx` for every RPC-submitted `Declare` transaction:

```rust
// crates/apollo_gateway/src/gateway.rs  lines 407-433
fn check_declare_permissions(&self, declare_tx: &RpcDeclareTransaction) -> Result<(), StarknetError> {
    if self.config.static_config.block_declare {
        return Err(/* BLOCKED_TRANSACTION_TYPE */);
    }
    let RpcDeclareTransaction::V3(declare_v3_tx) = declare_tx;
    if !self.config.is_authorized_declarer(&declare_v3_tx.sender_address) {
        return Err(/* UnauthorizedDeclare */);
    }
    Ok(())
}
```

`is_authorized_declarer` returns `false` for any address not in the configured allow-list:

```rust
// crates/apollo_gateway_config/src/config.rs  lines 141-146
pub fn is_authorized_declarer(&self, declarer_address: &ContractAddress) -> bool {
    match &self.static_config.authorized_declarer_accounts {
        Some(allowed_accounts) => allowed_accounts.contains(declarer_address),
        None => true,
    }
}
```

**Consensus validation path — no equivalent check:**

When a validator processes a proposal, `handle_proposal_part` converts each `ConsensusTransaction` via `TransactionConverter::convert_consensus_tx_to_internal_consensus_tx`, which calls `convert_rpc_tx_to_internal`. For `Declare` transactions the only check performed is a `compiled_class_hash` mismatch; `is_authorized_declarer` is never consulted:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs  lines 347-376
RpcTransaction::Declare(RpcDeclareTransaction::V3(tx)) => {
    let ClassHashes { class_hash, executable_class_hash_v2 } =
        self.class_manager_client.add_class(tx.contract_class).await?;
    if tx.compiled_class_hash != executable_class_hash_v2 {
        return Err(TransactionConverterError::ValidateCompiledClassHashError(...));
    }
    // ← no is_authorized_declarer / block_declare check
    ...
}
```

The converted transaction is forwarded to the batcher via `send_txs_for_proposal` and executed by the blockifier without any further admission check. Confirmed by `grep` — neither `crates/apollo_batcher` nor `crates/blockifier` contain any reference to `authorized_declarer`, `is_authorized_declarer`, or `block_declare`.

---

### Impact Explanation

When an operator configures `authorized_declarer_accounts` (or sets `block_declare = true`), the intent is that only permitted accounts can declare new contract classes on the chain. Because the restriction is absent from the consensus execution path, a proposer can commit an unauthorized `Declare` transaction that:

1. Registers an arbitrary Sierra class hash in the global class registry.
2. Makes that class available for deployment via the `deploy` syscall by any account.
3. Permanently alters the chain state in a way the operator explicitly prohibited.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing**, and also touches **wrong class hash committed to state**.

---

### Likelihood Explanation

- The proposer is a consensus participant selected by the protocol; in a decentralized sequencer any validator can become proposer.
- The proposer does not need to go through the gateway at all — it constructs the proposal directly and streams it to peers.
- No signature or proof from the unauthorized sender is required beyond what the blockifier already validates (account `__validate__` entry point, nonce, fee).
- The trigger is therefore available to any node that wins a proposer slot, which is an unprivileged network-level event.

---

### Recommendation

Move the `authorized_declarer_accounts` / `block_declare` check out of the gateway and into the shared transaction-conversion layer so it is enforced on every path that produces an executable `Declare` transaction:

1. Add the restriction to `TransactionConverter::convert_rpc_tx_to_internal` (or a dedicated validation step called from both the gateway and `handle_proposal_part`), parameterised by the same config.
2. Alternatively, enforce it inside the batcher's `send_txs_for_proposal` before the transaction reaches the blockifier.
3. Ensure the check covers both `block_declare = true` (total block) and the `authorized_declarer_accounts` allow-list.

This mirrors the Evmos approach cited in the external report: the restriction must be enforced at the execution layer, not only at the RPC entry point.

---

### Proof of Concept

1. Operator deploys the sequencer with `authorized_declarer_accounts = ["0xAUTHORIZED"]` and `block_declare = false`.
2. Attacker controls a validator node that wins a proposer slot.
3. Attacker constructs a valid `RpcDeclareTransaction::V3` with `sender_address = 0xUNAUTHORIZED` (any funded account with a valid signature and nonce).
4. Attacker streams a `ProposalPart::Transactions` batch containing this transaction directly to peers, bypassing the gateway entirely.
5. Each validator calls `handle_proposal_part` → `convert_consensus_tx_to_internal_consensus_tx` → `convert_rpc_tx_to_internal`; the `compiled_class_hash` check passes; no `is_authorized_declarer` check fires.
6. The transaction reaches `send_txs_for_proposal`, is executed by the blockifier, and the class hash is written to `contract_class_changes`.
7. `ProposalFin` is accepted; the block is committed; the unauthorized class is now declared on-chain and available for deployment.
