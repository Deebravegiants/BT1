### Title
`ecc_prv_key_t::set()` and `generate()` Fail to Clear Stale `ed_bin` When Switching Curves, Causing Wrong Private Scalar to Be Used — (`File: src/cbmpc/crypto/base_ecc.cpp`)

---

### Summary

`ecc_prv_key_t` holds three fields: `curve`, `val` (scalar for non-EdDSA curves), and `ed_bin` (raw 32-byte seed for Ed25519). The `value()` method unconditionally prefers `ed_bin` over `val` whenever `ed_bin` is non-empty. Neither `set(curve, val)` nor `generate(curve)` clears `ed_bin` when the new curve is not Ed25519. If a key object previously held an Ed25519 key and is then "replaced" via `set()` or `generate()` for a different curve, the stale `ed_bin` silently overrides the new scalar in every downstream operation.

---

### Finding Description

`ecc_prv_key_t` is declared in `include-internal/cbmpc/internal/crypto/base_ecc.h`:

```cpp
class ecc_prv_key_t {
 private:
  ecurve_t curve;
  bn_t val;
  buf_t ed_bin;  // for EDDSA
};
``` [1](#0-0) 

The `value()` method in `src/cbmpc/crypto/base_ecc.cpp` always prefers `ed_bin`:

```cpp
bn_t ecc_prv_key_t::value() const {
  cb_assert(curve);
  bn_t x = val;
  if (!ed_bin.empty()) x = ed25519::prv_key_to_scalar(ed_bin);
  return x % curve.order();
}
``` [2](#0-1) 

`set()` replaces `curve` and `val` but leaves `ed_bin` untouched:

```cpp
void ecc_prv_key_t::set(ecurve_t curve, const bn_t& val) {
  this->curve = curve;
  this->val = val;
}
``` [3](#0-2) 

`generate()` has the same omission — when the new curve is not Ed25519, it writes a fresh `val` but never zeroes `ed_bin`:

```cpp
void ecc_prv_key_t::generate(ecurve_t curve) {
  this->curve = curve;
  if (curve == curve_ed25519)
    ed_bin = gen_random(32);
  else
    val = curve.get_random_value();
}
``` [4](#0-3) 

The direct analog to the Solidity bug is exact:

| Solidity (`setStaking`) | cb-mpc (`set` / `generate`) |
|---|---|
| Replaces staking contract pointer | Replaces `curve` + `val` |
| Does not revoke old approval | Does not clear old `ed_bin` |
| Does not approve new contract | New `val` is silently ignored by `value()` |
| New pool calls revert / use wrong allowance | New curve operations use old Ed25519 scalar |

The `convert()` serialization path also serializes all three fields independently, meaning a deserialized key can arrive with both `val` and `ed_bin` populated; `value()` will always pick `ed_bin`. [5](#0-4) 

---

### Impact Explanation

Every operation that calls `value()` — including `pub()` (public key derivation) and the scalar path of `sign()` — will silently use the old Ed25519 scalar on the new curve. This produces a public key that does not correspond to the intended private scalar. In MPC signing protocols (`ecdsa_mp`, `eddsa`, `schnorr_mp`) where `ecc_prv_key_t` carries a party's key share, the mismatch between the announced public share and the scalar actually used during signing causes the combined signature to be computed under the wrong key material. Depending on the protocol step, this can result in:

- A forged or invalid signature accepted by the protocol (if the wrong scalar happens to satisfy a verification step against a stale public key),
- Leakage of the old Ed25519 seed scalar into a non-Ed25519 signing transcript, or
- Silent divergence between a party's stored public share and the scalar it uses, breaking threshold consistency.

---

### Likelihood Explanation

The trigger requires a key object to be reused across curve types — e.g., a key first populated via `set_ed_bin()` or `generate(curve_ed25519)` and then repurposed via `set(other_curve, val)` or `generate(other_curve)`. This pattern arises naturally in:

- Key migration or refresh workflows where the same `ecc_prv_key_t` variable is reused,
- Any code path that calls `generate()` more than once on the same object (e.g., retry logic or test harnesses that feed into production paths),
- Deserialization via `convert()` of a peer-supplied blob that carries a non-empty `ed_bin` alongside a non-Ed25519 `curve`.

No attacker capability beyond sending a crafted serialized key blob (for the `convert()` path) or influencing the curve selection in a multi-curve API call is required.

---

### Recommendation

1. **In `set()`**: add `ed_bin.clear();` before assigning `curve` and `val`.
2. **In `generate()`**: in the `else` branch (non-Ed25519), add `ed_bin.clear();` before generating `val`.
3. **In `set_ed_bin()`**: add `val = bn_t{};` to symmetrically clear the scalar field.
4. Consider adding a private invariant-check helper that asserts `ed_bin.empty() || curve == curve_ed25519` and call it at the end of each setter and at the top of `value()`.

### Proof of Concept

```cpp
#include <cbmpc/internal/crypto/base_ecc.h>
using namespace coinbase::crypto;

// Step 1: create an Ed25519 key
ecc_prv_key_t key;
key.generate(curve_ed25519);
// ed_bin is now a 32-byte random seed; val is zero/unused.

// Step 2: "replace" it with a secp256k1 key
bn_t intended_scalar = curve_secp256k1.get_random_value();
key.set(curve_secp256k1, intended_scalar);
// BUG: ed_bin is still non-empty.

// Step 3: derive public key
ecc_pub_key_t Q = key.pub();
// value() returns ed25519::prv_key_to_scalar(ed_bin) % secp256k1.order()
// NOT intended_scalar.
// Q != intended_scalar * G  →  public/private key mismatch.

// Step 4: sign
buf_t sig = key.sign(some_hash);
// Signature is produced under the old Ed25519 seed's scalar on secp256k1,
// not under intended_scalar.  Verification against Q will fail or,
// if Q was also derived from the wrong scalar, the signature is valid
// under the wrong key — key material substitution.
``` [6](#0-5) [7](#0-6)
