### Title
Unverified Vote Signatures Allow Single Malicious Peer to Forge Quorum and Force Premature Consensus Decision - (File: `crates/apollo_consensus/src/single_height_consensus.rs`)

---

### Summary

`SingleHeightConsensus::handle_vote` accepts incoming consensus votes without verifying their cryptographic signatures. Because the `voter` field is a self-reported `ContractAddress` taken verbatim from the protobuf wire message, any connected peer can craft votes claiming to be any committee member. By sending one forged vote per committee member, a single malicious peer accumulates enough weight to pass the 2/3 Byzantine quorum threshold and force the local node to emit `DecisionReached` — committing a block without real network agreement.

---

### Finding Description

`handle_vote` in `SingleHeightConsensus` carries an explicit unimplemented guard:

```rust

``` [1](#0-0) 

The function performs two checks before accepting a vote:

1. **Committee membership** — `vote.voter` must be in the committee member list.
2. **Duplicate/conflict detection** — keyed on `(round, voter)`.

Neither check authenticates the actual sender. The `voter` field is deserialized directly from the protobuf `Vote` message:

```rust
let voter = value.voter.ok_or(missing("voter"))?.try_into()?;
let signature = value.signature.map(|hashes| hashes.try_into()).transpose()?.unwrap_or_default();
Ok(Vote { vote_type, height, round, proposal_commitment, voter, signature })
``` [2](#0-1) 

Votes are created with `RawSignature::default()` (an empty byte vector) even by the node itself:

```rust
// TODO(Asmaa): sign the vote
signature: RawSignature::default(),
``` [3](#0-2) 

The duplicate check uses `(round, voter)` as the HashMap key, so one forged vote per committee member per round is accepted without collision:

```rust
let key = (vote.round, vote.voter);
let map_entry = match vote.vote_type {
    VoteType::Prevote => self.prevotes.get(&key),
    VoteType::Precommit => self.precommits.get(&key),
};
``` [4](#0-3) 

Quorum is computed by summing the weights of all accepted votes:

```rust
let weight_sum = votes
    .iter()
    .filter_map(|(&(r, _voter), (v, w))| {
        (r == round && value.is_none_or(|val| val == &v.proposal_commitment)).then_some(*w)

    .sum();
threshold.is_met(weight_sum, self.total_weight)
``` [5](#0-4) 

The `upon_decision` path additionally requires the virtual proposer to have precommitted in favor — but since the virtual proposer's identity is a plain `ContractAddress` in the committee, the malicious peer can forge that vote too:

### Impact Explanation

Once `DecisionReached` is emitted, the manager calls `context.decision_reached`, which commits the block to the batcher and storage:

```rust
context.decision_reached(height, decided_round, decision.block, wait_for_last_commitment).await?;
``` [7](#0-6) 

A single malicious peer can force the local node to commit a block at a given height without any real validator having voted for it. The node diverges from the rest of the network, producing a wrong committed state root, wrong block hash, and wrong transaction receipts relative to the actual network consensus. This matches the **High** impact category: the consensus decision path accepts an unauthorized result and produces an authoritative-looking wrong committed state.

---

### Likelihood Explanation

Any peer reachable over the p2p network can send `Vote` protobuf messages on the consensus broadcast topic. No privileged access is required. For a committee of size N with Byzantine quorum (>2/3 weight), the attacker needs to send at most N forged votes — one per committee member — each with a different `voter` address and `RawSignature::default()`. For typical small committees (4–10 validators), this is trivially achievable in a single round-trip.

---

### Recommendation

Implement vote signature verification inside `handle_vote` before the committee-membership check. Each `Vote` must be signed over its canonical fields (`vote_type || height || round || proposal_commitment || voter`) using the validator's registered signing key. The verifier should look up the validator's public key from the committee by `vote.voter` and reject any vote whose signature does not verify. The `SignatureManager` infrastructure already exists in `crates/apollo_signature_manager/` and can be reused. [8](#0-7)

### Proof of Concept

Assume a committee `{P, V1, V2, V3}` with equal weights (total weight 4, Byzantine quorum > 8/3 ≈ 2.67, so weight 3 suffices). The local node is `V1`.

1. **Setup**: Malicious peer `M` connects to `V1`'s p2p endpoint.
2. **Proposal arrives**: A valid `ProposalInit` for `(height=H, round=0)` is received and validated by the local batcher, setting `proposals[0] = Some(commitment_X)`.
3. **Forge prevotes**: `M` sends three `Vote` messages over the gossipsub vote topic:
   - `Vote { vote_type: Prevote, height: H, round: 0, proposal_commitment: Some(commitment_X), voter: P, signature: [] }`
   - `Vote { vote_type: Prevote, height: H, round: 0, proposal_commitment: Some(commitment_X), voter: V2, signature: [] }`
   - `Vote { vote_type: Prevote, height: H, round: 0, proposal_commitment: Some(commitment_X), voter: V3, signature: [] }`
   Each passes the committee-membership check (P, V2, V3 are all valid members) and the duplicate check (distinct `voter` keys). Weight sum = 3 > 2.67 → prevote quorum reached. `V1` broadcasts its own precommit.
4. **Forge precommits**: `M` sends three more `Vote` messages:
   - `Vote { vote_type: Precommit, voter: P, ... }` (P is the virtual proposer)
   - `Vote { vote_type: Precommit, voter: V2, ... }`
   - `Vote { vote_type: Precommit, voter: V3, ... }`
   Weight sum = 3 > 2.67 → precommit quorum reached. Virtual proposer `P` is in favor.
5. **Decision**: `upon_decision` fires, emitting `DecisionReached { block: commitment_X, round: 0 }`. The manager calls `context.decision_reached(H, 0, commitment_X, ...)`, committing the block — without a single real validator having voted.
