### Title
Caller Identity Not Bound to `verify_foreign_transaction` Request or Signed Payload, Enabling Cross-Caller Signature Replay - (File: `crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` does not include the caller's account ID (`predecessor_account_id`) in either the pending-request map key or the MPC-signed payload. The resulting `VerifyForeignTransactionResponse` — a signature over the root public key — is caller-agnostic and can be replayed by any party to forge foreign-chain verification attestations in downstream bridge contracts.

### Finding Description

In `sign()` and `request_app_private_key()`, the caller's identity is cryptographically bound to the request by deriving a per-caller tweak via `derive_tweak(predecessor_id, path)`. The resulting `SignatureRequest` stores this tweak, so the MPC signature is only valid under the caller's derived public key. [1](#0-0) 

`verify_foreign_transaction()` takes a different path. It calls `args_into_verify_foreign_tx_request`, which copies only `domain_id`, `request`, and `payload_version` — the caller's `predecessor_account_id` is logged but never incorporated into the request: [2](#0-1) 

The mapping function confirms no caller field is added: [3](#0-2) 

The resulting `VerifyForeignTransactionRequest` struct has no `tweak`, no `predecessor`, and no caller-binding field: [4](#0-3) 

Because the pending-request map is keyed on this caller-agnostic struct, any two callers submitting the same foreign-chain query share a single map entry. The contract's own test explicitly documents this: [5](#0-4) 

When `respond_verify_foreign_tx` resolves the request, it verifies the signature against the **root public key** (no key derivation), and the `payload_hash` it signs over contains only the foreign-chain RPC data — no caller identity: [6](#0-5) 

The design document for this feature explicitly planned to include a `tweak` field (derived from `predecessor_id` and a `derivation_path`) in `VerifyForeignTransactionRequest`, but the shipped implementation dropped both fields entirely: [7](#0-6) 

### Impact Explanation

The `VerifyForeignTransactionResponse` — a `(payload_hash, root-key-signature)` pair — is not bound to any caller. Any party who obtains this pair (by submitting the same request, or by observing the on-chain response) can present it to a downstream bridge contract as proof that a foreign-chain transaction occurred. A bridge contract that uses this attestation to release funds (e.g., "Bitcoin tx X was confirmed, release NEAR to the requester") cannot distinguish Alice's legitimate claim from Bob's replay of the same signature. This directly enables double-spend and unauthorized fund release — forged foreign-chain verification causing invalid bridge execution.

**Impact: High** — Cross-chain replay / forged foreign-chain verification enabling double-spend conditions.

### Likelihood Explanation

The attack requires no special privilege. Any unprivileged NEAR account can call `verify_foreign_transaction` with the same arguments as a victim, receive the identical MPC-signed response, and use it in a bridge contract. The on-chain response data is public. No threshold collusion, key leakage, or TEE attack is needed.

**Likelihood: High**

### Recommendation

Bind the caller's identity to the request and the signed payload, mirroring the pattern used by `sign()`:

1. Add a `tweak` field to `VerifyForeignTransactionRequest`, derived from `(predecessor_id, derivation_path)` using a foreign-tx-specific prefix (as the design doc specifies).
2. Include the `tweak` in the `ForeignTxSignPayload` that MPC nodes hash and sign, so the resulting signature is only valid under the caller's derived public key.
3. Verify the response signature against the **derived** public key (not the root key) in `respond_verify_foreign_tx`.

This matches the already-documented design intent in `docs/foreign-chain-transactions.md` lines 254–286.

### Proof of Concept

1. Alice calls `verify_foreign_transaction({ domain_id, payload_version: V1, request: Bitcoin { tx_id: X, ... } })` with 1 yoctoNEAR deposit.
2. Bob calls `verify_foreign_transaction` with the **identical** arguments from a different account.
3. Both yields are queued under the same `VerifyForeignTransactionRequest` key (confirmed by the contract's own test at line 3255).
4. MPC nodes sign `hash(ForeignTxSignPayloadV1 { request, values })` — no caller field — against the root public key and call `respond_verify_foreign_tx`.
5. Both Alice and Bob receive the **identical** `VerifyForeignTransactionResponse { payload_hash, signature }`.
6. Bob presents Alice's `(payload_hash, signature)` to a bridge contract to claim Alice's Bitcoin deposit on NEAR, completing the double-spend. [8](#0-7) [9](#0-8)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    #[test]
    fn verify_foreign_transaction__should_queue_duplicates_from_different_callers() {
        // Given: two different callers will submit the same foreign-tx verification request.
        let mut rng = rand::rngs::StdRng::from_seed([42u8; 32]);
        let (context, mut contract, secret_key) =
            basic_setup_with_protocol(Protocol::CaitSith, DomainPurpose::ForeignTx, &mut rng);
        register_supported_chains(&mut contract, [dtos::ForeignChain::Bitcoin]);
        let SharedSecretKey::Secp256k1(secret_key) = secret_key else {
            unreachable!();
        };

        let request_args = VerifyForeignTransactionRequestArgs {
            domain_id: DomainId::default().0.into(),
            payload_version: ForeignTxPayloadVersion::V1,
            request: dtos::ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
                tx_id: [7u8; 32].into(),
                confirmations: 2.into(),
                extractors: vec![BitcoinExtractor::BlockHash],
            }),
        };
        let request = args_into_verify_foreign_tx_request(request_args.clone());

        // When: caller alice submits the request.
        let alice = AccountId::from_str("alice.near").unwrap();
        testing_env!(
            VMContextBuilder::new()
                .signer_account_id(alice.clone())
                .predecessor_account_id(alice)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()
        );
        contract.verify_foreign_transaction(request_args.clone());

        // And: caller bob submits the identical request — a different account would today
        // be blocked from receiving a response by alice's submission.
        let bob = AccountId::from_str("bob.near").unwrap();
        testing_env!(
            VMContextBuilder::new()
                .signer_account_id(bob.clone())
                .predecessor_account_id(bob)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()
        );
        contract.verify_foreign_transaction(request_args);

        // Then: both yields are queued under the single (caller-agnostic) request key.
        assert_eq!(
            contract
                .pending_verify_foreign_tx_requests
                .get(&request)
                .map(|q| q.len()),
            Some(2),
            "duplicate foreign-tx requests from different callers should fan out",
        );
