### Title
Caller-agnostic `VerifyForeignTransactionRequest` key allows fan-out queue saturation, permanently blocking legitimate `verify_foreign_transaction` callers - (File: `crates/contract/src/pending_requests.rs`)

### Summary

The `verify_foreign_transaction` request key does not include the caller's account ID. Any unprivileged account can saturate the 128-slot fan-out queue for a specific foreign transaction by submitting 128 identical requests, causing every subsequent legitimate caller's `verify_foreign_transaction` to panic with `PendingRequestQueueFull`. This breaks the foreign-chain bridge request lifecycle and can be sustained indefinitely at negligible cost.

### Finding Description

`push_pending_yield` in `pending_requests.rs` enforces a hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` per request key. When the queue is full it calls `env::panic_str`, reverting the entire enclosing transaction: [1](#0-0) 

The `verify_foreign_transaction` entry point converts caller arguments into a `VerifyForeignTransactionRequest` via `args_into_verify_foreign_tx_request`: [2](#0-1) 

The resulting key contains only `domain_id`, `request`, and `payload_version` — **no caller account ID**. This is in deliberate contrast to `SignatureRequest::new`, which folds the `predecessor` into the key, making each user's sign queue private. The caller-agnostic design for foreign-tx requests is confirmed by the existing test: [3](#0-2) 

The comment there even acknowledges the consequence: *"a different account would today be blocked from receiving a response by alice's submission."*

The full call chain that reaches the cap check is:

```
verify_foreign_transaction()          [lib.rs:519]
  └─ enqueue_yield_request()          [lib.rs:551]
       └─ add_verify_foreign_tx_request()
            └─ push_pending_yield()   [pending_requests.rs:43]
                 └─ env::panic_str()  ← revert if queue.len() >= 128
``` [4](#0-3) 

Once the queue is saturated, every new `verify_foreign_transaction` call for that specific foreign transaction panics. After MPC nodes drain the queue via `respond_verify_foreign_tx`, the attacker can immediately re-saturate it with another 128 calls, sustaining the block indefinitely.

### Impact Explanation

An unprivileged attacker can permanently prevent any specific foreign-chain transaction from being verified and signed by the MPC network. This breaks the bridge request lifecycle: a user whose bridge transaction is targeted cannot obtain the MPC-signed payload needed to finalize the cross-chain operation. The attack targets the `verify_foreign_transaction` flow specifically, which is the production path for foreign-chain bridge execution.

This matches: **Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

### Likelihood Explanation

The cost to saturate the queue is 128 NEAR transactions, each requiring only `MINIMUM_SIGN_REQUEST_DEPOSIT = 1 yoctoNEAR`: [5](#0-4) 

The deposit cost is negligible (128 yoctoNEAR total). Gas cost for 128 transactions is the only real barrier, and it is low relative to the value of blocking a bridge transaction. The attacker needs only to know the target `tx_id` and `domain_id`, both of which are observable on-chain. Re-saturation after each drain is straightforward to automate.

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, mirroring how `SignatureRequest::new` incorporates the predecessor. This makes each caller's queue slot private and prevents cross-caller saturation. Alternatively, enforce a per-account submission rate limit at the contract level before enqueuing.

### Proof of Concept

1. Attacker observes a target `VerifyForeignTransactionRequestArgs` (e.g., a specific Bitcoin `tx_id`) that a legitimate user intends to submit.
2. Attacker submits 128 identical `verify_foreign_transaction` calls with that request from any account(s), each with 1 yoctoNEAR deposit.
3. `push_pending_yield` fills the queue to `MAX_PENDING_REQUEST_FAN_OUT = 128`.
4. Legitimate user submits `verify_foreign_transaction` with the same request → `push_pending_yield` panics → transaction reverts with `PendingRequestQueueFull`.
5. MPC nodes eventually call `respond_verify_foreign_tx`, draining the queue.
6. Attacker immediately re-submits 128 calls → queue is saturated again.
7. The legitimate user's bridge transaction is permanently blocked as long as the attacker sustains the attack. [6](#0-5) [7](#0-6)

pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
