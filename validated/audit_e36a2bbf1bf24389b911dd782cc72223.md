Audit Report

## Title
Missing Staleness and Validity Checks on Chainlink `latestRoundData()` in Core Oracle Enables rsETH Price Manipulation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards all validity return values (`roundId`, `updatedAt`, `answeredInRound`), accepting any price — including stale or zero prices — without validation. A stale or inflated Chainlink price for any supported LST propagates into `rsETHPrice` via `LRTOracle._updateRsETHPrice()`, which any unprivileged caller can trigger via the public `updateRSETHPrice()`. An attacker can then redeem rsETH at the inflated rate through `instantWithdrawal()`, extracting more LST than their rsETH is worth and directly stealing from remaining depositors.

## Finding Description

**Root cause — missing validation in `ChainlinkPriceOracle.getAssetPrice()`:**

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol line 52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values `(roundId, answer, startedAt, updatedAt, answeredInRound)` are destructured; only `answer` is used. No check is performed for:
- `answeredInRound >= roundId` (stale round detection)
- `updatedAt > 0` (incomplete round detection)
- `price > 0` (valid price)
- `block.timestamp - updatedAt <= heartbeat` (freshness) [1](#0-0) 

**Contrast with the protocol's own L2 wrapper**, which performs all three structural checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol lines 30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

**Price propagation path:**

1. `LRTOracle._getTotalEthInProtocol()` calls `getAssetPrice(asset)` for every supported LST, multiplying the unvalidated price by `totalAssetDeposits`. [3](#0-2) 

2. `_updateRsETHPrice()` computes `newRsETHPrice = totalETHInProtocol / rsethSupply` and stores it in `rsETHPrice`. [4](#0-3) 

3. `updateRSETHPrice()` is `public` with only a `whenNotPaused` guard — any address can call it. [5](#0-4) 

**`pricePercentageLimit` guard is inactive by default:**

`initialize()` does not set `pricePercentageLimit`, so it defaults to `0`. The guard condition `pricePercentageLimit > 0 && ...` is always `false`, providing zero protection against price inflation in the default deployment state. [6](#0-5) [7](#0-6) 

**Withdrawal exploitation:**

`getExpectedAssetAmount()` computes `amount * rsETHPrice / assetPrice`. With an inflated `rsETHPrice`, the attacker receives more LST than their rsETH is worth. [8](#0-7) 

The queued withdrawal path via `_calculatePayoutAmount()` is equally affected. [9](#0-8) 

## Impact Explanation

**Critical — Direct theft of user funds.**

When a Chainlink feed for any supported LST (e.g., stETH/ETH, rETH/ETH) goes stale with an inflated last-stored price:
- `rsETHPrice` is updated upward by any caller.
- An attacker redeems rsETH via `instantWithdrawal()` and receives more LST than their rsETH is worth at the true price.
- The surplus LST is drawn from the pool, directly reducing the assets backing all other rsETH holders — constituting direct theft of depositor funds.

This matches the allowed impact: **Critical — Direct theft of any user funds.**

## Likelihood Explanation

Chainlink feeds go stale during network congestion, oracle node outages, or low market activity. The `latestRoundData()` call returns the last stored answer with a stale `updatedAt`; the contract has no mechanism to detect this. The public `updateRSETHPrice()` function requires no privileged access, so an attacker can trigger the price update at the exact moment a stale inflated price is present. No special privileges, no flash loan, no governance capture is required — only the ability to call two public functions. The SECURITY.md exclusion for "Incorrect data supplied by third-party oracles" does not apply here because the root cause is the **protocol's own missing validation code**, not the oracle itself being compromised; the SECURITY.md note explicitly states "This does not exclude oracle manipulation/flash-loan attacks."

## Recommendation

Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_HEARTBEAT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, ensure `pricePercentageLimit` is set to a non-zero value during deployment so the guard in `_updateRsETHPrice()` is active.

## Proof of Concept

**Minimal call sequence (local fork test):**

1. Deploy/fork with stETH as a supported LST, `ChainlinkPriceOracle` configured, `pricePercentageLimit = 0`.
2. Mock the stETH/ETH Chainlink feed to return a stale answer: `price = 1.05e18`, `updatedAt = block.timestamp - 48 hours`, `answeredInRound < roundId`.
3. Attacker (EOA) calls `LRTOracle.updateRSETHPrice()` — succeeds, stores inflated `rsETHPrice`.
4. Attacker calls `LRTWithdrawalManager.instantWithdrawal(stETH, X, "")`.
5. `getExpectedAssetAmount(stETH, X)` returns `X * rsETHPrice_inflated / 1.05e18` — more stETH than the attacker's rsETH is worth at the true price.
6. Assert: attacker's stETH received > fair value; pool stETH balance reduced by the surplus; remaining depositors' rsETH is underbacked.

**Foundry invariant test:** Assert that for any sequence of `updateRSETHPrice()` + `instantWithdrawal()` calls with a stale feed mock, `totalAssetDeposits * assetPrice >= rsethSupply * rsETHPrice` (solvency invariant) always holds. The invariant will break under the stale-price scenario with the current code.
