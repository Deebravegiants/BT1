### Title
`reset()` Called Before `last_sync_id` Validation Allows Malicious Harvester to Wipe Farmer's Plot View — (`chia/plot_sync/receiver.py`)

### Summary
In `Receiver._sync_started`, when `data.initial=True`, `reset()` is unconditionally called before the `last_sync_id` integrity check. A malicious harvester peer with an established connection can send a crafted `PlotSyncStart(initial=True, last_sync_id=0, ...)` to atomically clear the farmer's entire plot state for that harvester, causing farming to halt for that peer.

### Finding Description
`_sync_started` processes an initial sync start as follows:

```python
async def _sync_started(self, data: PlotSyncStart) -> None:
    if data.initial:
        self.reset()                                   # ← wipes _plots, _last_sync
    self._validate_identifier(data.identifier, True)   # ← only checks message_id==0
    if data.last_sync_id != self._last_sync.sync_id:   # ← _last_sync.sync_id is NOW 0
        raise InvalidLastSyncIdError(...)              # ← never raised
```

`reset()` sets `_last_sync = Sync()` (sync_id=0) and calls `_plots.clear()`. The subsequent guard `data.last_sync_id != self._last_sync.sync_id` compares the attacker-supplied `last_sync_id=0` against the now-zeroed `_last_sync.sync_id=0`, so it trivially passes. The `_validate_identifier` call with `start=True` only validates `message_id==0`, which the attacker also supplies. All guards pass and the sync proceeds from empty state.

### Impact Explanation
The farmer's `_plots` dict for the targeted harvester is cleared mid-session. The farmer immediately reports zero plots for that harvester. If the attacker follows up with a valid `PlotSyncDone`, the empty state is committed to `_last_sync`. Farming halts for that harvester for the duration of the attack. This matches the High impact criterion: "Permanent or long-lived inability for honest farmers or harvesters to process valid sync updates under normal network assumptions."

### Likelihood Explanation
The attacker must hold an established `HARVESTER`-type connection to the farmer (i.e., be a connected harvester peer). This is a non-trivial precondition but is reachable by any machine the farmer has accepted as a harvester. No key material or cryptographic break is required — only the ability to send a crafted protocol message over an existing connection.

### Recommendation
Move the `last_sync_id` validation (and optionally `_validate_identifier`) **before** the `reset()` call. Capture the pre-reset `_last_sync.sync_id`, validate `data.last_sync_id` against it, and only then call `reset()`:

```python
async def _sync_started(self, data: PlotSyncStart) -> None:
    if data.initial:
        # Validate last_sync_id BEFORE reset destroys _last_sync
        if data.last_sync_id != self._last_sync.sync_id:
            raise InvalidLastSyncIdError(data.last_sync_id, self._last_sync.sync_id)
        self.reset()
    self._validate_identifier(data.identifier, True)
    if not data.initial:
        if data.last_sync_id != self._last_sync.sync_id:
            raise InvalidLastSyncIdError(data.last_sync_id, self._last_sync.sync_id)
    ...
```
