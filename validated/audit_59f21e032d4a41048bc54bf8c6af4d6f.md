### Title
`action_transfer_to_gas_key` Bypasses `GasKeyInfo::MAX_BALANCE_TO_BURN` Limit, Permanently Blocking Gas Key and Account Deletion - (File: `runtime/runtime/src/access_keys.rs`)

### Summary

`action_transfer_to_gas_key` imposes no upper bound on a gas key's balance, while both `delete_gas_key` and `action_delete_account` enforce `GasKeyInfo::MAX_BALANCE_TO_BURN` (1 NEAR). Any unprivileged user can send a cross-account `TransferToGasKeyAction` to push a victim's gas key balance above that threshold, making the key and the account permanently undeletable until the victim manually withdraws the excess.

### Finding Description

`GasKeyInfo::MAX_BALANCE_TO_BURN` is defined as 1 NEAR and is the guard used in both deletion paths:

```rust
// core/primitives-core/src/account.rs
pub const MAX_BALANCE_TO_BURN: Balance = Balance::from_near(1);
```

`delete_gas_key` enforces it:

```rust
// runtime/runtime/src/access_keys.rs  lines 103-110
if gas_key_info.balance > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh { ... }.into());
    return Ok(());
}
```

`action_delete_account` enforces it for the aggregate balance:

```rust
// runtime/runtime/src/actions.rs  lines 339-348
let gas_key_balance_to_burn = compute_gas_key_balance_sum(state_update, account_id)?;
if gas_key_balance_to_burn > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh { ... }.into());
    return Ok(());
}
```

But `action_transfer_to_gas_key` has **no upper-bound check**:

```rust
// runtime/runtime/src/access_keys.rs  lines 281-286
gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit).ok_or_else(|| {
    RuntimeError::StorageError(...)
})?;
set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);
```

`TransferToGasKeyAction` is also reachable via the `promise_batch_action_transfer_to_gas_key` host function, meaning any deployed contract can trigger it cross-account.

`validate_transfer_to_gas_key_action` only gates on protocol version — no receiver restriction:

```rust
// runtime/runtime/src/action_validation.rs  lines 171-173
Action::TransferToGasKey(_) => {
    validate_transfer_to_gas_key_action(current_protocol_version)
}
```

### Impact Explanation

An attacker sends a transaction (or contract call) with `receiver_id = victim.near` and `TransferToGasKeyAction { public_key: victim_gas_key_pk, deposit: 2 NEAR }`. After execution:

- `gas_key_info.balance` in the trie for `victim.near` exceeds `MAX_BALANCE_TO_BURN`.
- Every subsequent `DeleteKey` on that gas key returns `GasKeyBalanceTooHigh` — the key is undeletable.
- Every subsequent `DeleteAccount` on `victim.near` returns `GasKeyBalanceTooHigh` — the account is undeletable.

The victim can recover by issuing `WithdrawFromGasKey` (which has no balance cap), but the attacker can immediately re-fund. This is a sustained, economically-costed griefing attack that breaks the protocol invariant that a gas key with balance ≤ `MAX_BALANCE_TO_BURN` is always deletable.

The concrete corrupted trie value is `gas_key_info.balance > GasKeyInfo::MAX_BALANCE_TO_BURN` stored under the `(account_id, public_key)` access-key trie key.

### Likelihood Explanation

Any unprivileged account holding >1 NEAR can execute this in a single signed transaction targeting any account that has a gas key. The `GasKeys` protocol feature is active on mainnet/testnet. No validator or operator privilege is required.

### Recommendation

Add a `MAX_BALANCE_TO_BURN` upper-bound check inside `action_transfer_to_gas_key` (and the equivalent host-function path) before updating the balance:

```rust
let new_balance = gas_key_info.balance.checked_add(action.deposit)...;
if new_balance > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh { ... }.into());
    return Ok(());
}
gas_key_info.balance = new_balance;
```

Alternatively, enforce the cap at the aggregate level (sum across all gas keys on the account) to mirror the `action_delete_account` check.

### Proof of Concept

1. Alice has account `alice.near` with a gas key `gas_key_pk` (balance = 0).
2. Attacker sends a transaction: `signer = attacker.near`, `receiver = alice.near`, `actions = [TransferToGasKeyAction { public_key: gas_key_pk, deposit: 2 NEAR }]`.
3. `action_transfer_to_gas_key` succeeds — no cap check — `gas_key_info.balance` is now 2 NEAR in the trie.
4. Alice sends `DeleteKey { public_key: gas_key_pk }` → `delete_gas_key` returns `GasKeyBalanceTooHigh { balance: 2 NEAR }` → action fails.
5. Alice sends `DeleteAccount { beneficiary_id: ... }` → `action_delete_account` computes aggregate balance = 2 NEAR > 1 NEAR → returns `GasKeyBalanceTooHigh` → action fails.
6. Alice can issue `WithdrawFromGasKey` to recover, but the attacker can immediately re-fund, sustaining the lock.

pub(crate) fn action_transfer_to_gas_key(
    state_update: &mut TrieUpdate,
    result: &mut ActionResult,
    account_id: &AccountId,
    action: &TransferToGasKeyAction,
) -> Result<(), RuntimeError> {
    let Some(mut access_key) = get_access_key(state_update, account_id, &action.public_key)? else {
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };
    let Some(gas_key_info) = access_key.gas_key_info_mut() else {
        // Key exists but is not a gas key
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };

    gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit).ok_or_else(|| {
        RuntimeError::StorageError(StorageError::StorageInconsistentState(
            "gas key balance integer overflow".to_string(),
        ))
    })?;
    set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);
    Ok(())
}
