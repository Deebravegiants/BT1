Audit Report

## Title
`_fillBids` subtracts the wrong bin's `lengthE6` when walking downward, corrupting `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` for all bid bins below `curBinIdx` when bins have heterogeneous lengths — (`metric-periphery/contracts/lens/MetricOmmPoolDataProvider.sol`)

## Summary

`_fillBids` initializes `walkDistE6` to the lower boundary of `curBinIdx` and then, for each bin `b < curBinIdx`, subtracts `lengthE6` of bin `b+1` (the bin above) instead of `lengthE6` of bin `b` (the current bin). This produces wrong `lowerX64` and `upperX64` values for every bid bin below `curBinIdx` whenever bins have different `lengthE6` values, corrupting `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` returned by `getLiquidityDepth`.

## Finding Description

In `_fillBids`, `walkDistE6` is initialized to `curBinDistFromProvidedPriceE6` (the lower boundary of `curBinIdx`). The loop at line 492 walks downward:

```solidity
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1)); // reads bin b+1
walkDistE6 -= int256(uint256(lenAbove));                                 // subtracts b+1's length

(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx); // reads bin b
uint256 lowerX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6, ...);
uint256 upperX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6 + int256(uint256(lengthE6)), ...);
``` [1](#0-0) 

The correct invariant for contiguous bins is:

```
lower_boundary(b) = lower_boundary(b+1) - lengthE6(b)
```

The code instead computes:

```
walkDistE6 = lower_boundary(b+1) - lengthE6(b+1)
```

which equals `lower_boundary(b)` only when `lengthE6(b) == lengthE6(b+1)`. When bins have different lengths, `walkDistE6` is wrong, and so are `lowerX64`, `upperX64`, `mStartX64`, `mEndX64`, and ultimately `binAvgExecPriceX64` / `cumulativeAvgExecPriceX64`.

By contrast, `_fillAsks` correctly advances `cumDistE6` by the current bin's own length:

```solidity
ctx.cumDistE6 += int256(uint256(lengthE6)); // lengthE6 of bin b, not b+1
``` [2](#0-1) 

## Impact Explanation

`getLiquidityDepth` is the primary interface for off-chain and on-chain integrators to read bid depth. The corrupted `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` values are the direct inputs integrators use to choose `priceLimitX64` before calling `swap`. With wrong prices, if the reported bid price is lower than actual, the integrator sets `priceLimitX64` below the true execution price and the swap executes at a worse price than intended (direct loss of token1 output). If the reported bid price is higher than actual, the integrator sets `priceLimitX64` above the true execution price and the swap reverts, breaking core swap functionality. Both outcomes are reachable depending on the sign of `lengthE6(b+1) - lengthE6(b)`, meeting the bad-price execution impact gate.

## Likelihood Explanation

Bins with heterogeneous `lengthE6` are a normal, supported pool configuration — the factory does not require uniform bin lengths. Any integrator (router, aggregator, UI) calling `getLiquidityDepth` on such a pool is affected. No privileged access or malicious setup is required; the bug triggers on any public `getLiquidityDepth` call to a pool where at least two adjacent bid bins have different `lengthE6` values.

## Recommendation

Replace the two-call pattern (read `b+1`, subtract its length) with a single read of the current bin `b`'s length and subtract that:

```solidity
// Before (wrong):
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1));
walkDistE6 -= int256(uint256(lenAbove));
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);

// After (correct):
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);
walkDistE6 -= int256(uint256(lengthE6));
```

This mirrors the correct pattern in `_fillAsks` where `ctx.cumDistE6 += int256(uint256(lengthE6))` uses the current bin's own length.

## Proof of Concept

```solidity
function test_fillBids_wrongBoundaries_heterogeneousLengths() public {
    // Deploy pool with bins: curBinIdx=0 (lengthE6=10_000), bin -1 (lengthE6=20_000)
    // Seed liquidity in both bins
    // Call getLiquidityDepth(pool, 2)
    LiquidityDepth memory depth = dataProvider.getLiquidityDepth(address(pool), 2);

    // Bin -1 upper boundary should equal lower boundary of bin 0 = curBinDistFromProvidedPriceE6
    // Code computes walkDistE6 = 0 - 10_000 = -10_000 (wrong; should be 0 - 20_000 = -20_000)
    // upperX64 = price at -10_000 + 20_000 = +10_000 (above current bin's lower boundary — wrong)

    uint256 expectedAvg = _priceFromMidAndDistE6(midPrice, -10_000); // midpoint of [-20_000, 0]
    assertEq(depth.bids[1].binAvgExecPriceX64, expectedAvg); // fails with current code
}
```
