Audit Report

## Title
Missing Caller Authentication in `MerkleDistributor.claim()` Enables Forced Fee Extraction from Any User - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` accepts an arbitrary `account` parameter but performs no `msg.sender == account` check, allowing any unprivileged caller with a valid (public) merkle proof to trigger a claim on behalf of any user. When `feeInBPS > 0`, this irrevocably transfers up to 10% of the victim's claimable yield to `protocolTreasury` without the victim's consent. The sibling contract `KernelMerkleDistributor` explicitly blocks this with `if (account != msg.sender) revert Unauthorized()`, confirming the omission is a defect.

## Finding Description
`MerkleDistributor.claim()` performs no identity check on the caller: [1](#0-0) 

After verifying the merkle proof, the contract computes and deducts a fee of up to 10% (`MAX_FEE_IN_BPS = 1000`) from the claimable amount and sends it to `protocolTreasury`: [2](#0-1) 

The remaining amount goes to `account`, not the attacker — so the attacker gains nothing financially, but the victim loses the fee portion. The victim's `userClaims[account]` state is also updated, preventing a re-claim until the next merkle root is published: [3](#0-2) 

By contrast, `KernelMerkleDistributor._processClaim()` explicitly enforces caller identity before any state changes: [4](#0-3) 

The only existing guard is `whenNotPaused`, which is owner-controlled and does not restrict which address may trigger a claim for which account.

## Impact Explanation
When `feeInBPS > 0` (owner-settable up to 1000 BPS / 10%), a forced claim extracts a non-recoverable fee from the victim's unclaimed yield and transfers it to `protocolTreasury`. The victim receives less than their full entitlement without ever interacting with the contract. This is a concrete instance of **High — Theft of unclaimed yield**: the attacker can sweep all eligible accounts at every root update, causing every user to permanently lose up to 10% of their yield per epoch.

## Likelihood Explanation
Merkle proofs are published off-chain for users to self-claim and are therefore fully public. No access control, rate limiting, or gas barrier prevents mass forced-claiming. An attacker needs only to iterate the published distribution list and submit one transaction per user per epoch. The precondition (`feeInBPS > 0`) is a normal operational state, not an edge case. Likelihood is **High**.

## Recommendation
Add the same caller identity check already present in `KernelMerkleDistributor`:

```solidity
function claim(...) external override whenNotPaused {
    if (account != msg.sender) revert Unauthorized();
    ...
}
```

If claiming on behalf of others is intentionally supported (e.g., for relayers), the fee should be waived when `msg.sender != account`, or explicit on-chain delegation must be required.

## Proof of Concept
1. Protocol publishes merkle root with leaf `(index=1, account=Alice, cumulativeAmount=1000e18)` and proof `P`. Owner has set `feeInBPS = 1000`.
2. Attacker (any EOA) calls:
   ```solidity
   merkleDistributor.claim(1, Alice, 1000e18, P);
   ```
3. Contract verifies proof (valid). Computes `claimableAmount = 1000e18`, `fee = 100e18`, `amountToSend = 900e18`.
4. `900e18` tokens transferred to Alice; `100e18` tokens transferred to `protocolTreasury`.
5. `userClaims[Alice].lastClaimedIndex = 1`; Alice cannot claim again until the next root.
6. Alice loses `100e18` tokens (10%) she never consented to pay as a fee.

**Foundry test plan**: Deploy `MerkleDistributor` with `feeInBPS = 1000`. Build a two-leaf merkle tree, set the root. Call `claim(index, alice, amount, proof)` from `address(attacker)` where `attacker != alice`. Assert `token.balanceOf(protocolTreasury) == fee` and `token.balanceOf(alice) == amount - fee`, confirming forced fee extraction with no attacker profit.
