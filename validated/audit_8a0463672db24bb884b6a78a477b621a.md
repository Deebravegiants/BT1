### Title
Incomplete ETH Deposit Limit Check Allows Bypassing the Configured Cap - (File: contracts/LRTDepositPool.sol)

### Summary
`LRTDepositPool.depositETH()` calls `_checkIfDepositAmountExceedesCurrentLimit()`, which omits the incoming ETH `amount` from the limit comparison. For LST assets the check correctly uses `totalAssetDeposits + amount > depositLimit`, but for ETH it only checks `totalAssetDeposits > depositLimit`. Any depositor can therefore push total ETH holdings arbitrarily above the configured cap in a single transaction.

### Finding Description
`_checkIfDepositAmountExceedesCurrentLimit()` contains an asymmetric guard:

```solidity
// contracts/LRTDepositPool.sol  L676-L682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));  // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct for LSTs
}
```

For ETH the function returns `true` (i.e., "limit exceeded") only when the *pre-deposit* total already exceeds the cap. The `amount` being deposited is never added to the comparison. Consequently:

- If `totalAssetDeposits == depositLimit`, the check returns `false` and the deposit proceeds, pushing the total to `depositLimit + amount`.
- If `totalAssetDeposits < depositLimit` by any margin, a depositor can send an arbitrarily large ETH value and the check still passes.

`depositETH()` is the sole public entry point for ETH deposits and calls `_beforeDeposit()`, which calls this function:

```solidity
// L76-L93
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    ...
}
```

### Impact Explanation
The deposit limit (`depositLimitByAsset[ETH]`) is the protocol's primary TVL cap for ETH. Bypassing it allows the protocol to accept and mint rsETH against more ETH than the risk parameters permit. This violates the protocol's stated invariant and can cause the protocol to take on more EigenLayer exposure than intended, constituting a failure to deliver the promised deposit-cap guarantee.

**Impact: Low** — Contract fails to deliver promised returns (deposit cap enforcement), but deposited ETH is not directly stolen or frozen.

### Likelihood Explanation
Any unprivileged user calling `depositETH()` with a sufficiently large `msg.value` can trigger this. No special role, front-running, or external dependency is required. The condition is reachable on every ETH deposit once `totalAssetDeposits` is within one transaction's worth of the cap.

### Recommendation
Include the incoming `amount` in the ETH branch of the limit check, consistent with the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

### Proof of Concept
Assume `depositLimitByAsset[ETH] = 1000 ether` and `totalAssetDeposits(ETH) = 999 ether`.

1. Attacker calls `depositETH{value: 500 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500 ether)` evaluates `999 ether > 1000 ether` → `false`.
3. Deposit proceeds; `totalAssetDeposits(ETH)` becomes `1499 ether`, 49.9 % above the cap.
4. rsETH is minted for the full 500 ETH with no revert.

The same call with an LST would correctly revert because `999 + 500 = 1499 > 1000`. [1](#0-0) [2](#0-1) [3](#0-2)
