Audit Report

## Title
`addLiquidity` Executes Deposits Into a Paused Pool Without Checking `pauseLevel` — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.addLiquidity` is missing the `whenNotPaused` modifier that `swap` correctly applies. When the pool is paused at level 1 or 2 (e.g., due to an oracle anomaly or active exploit), any caller can still deposit real tokens into the pool. The deposit path commits permanent state changes to `binTotals`, `_binStates`, `_binTotalShares`, and `_positionBinShares`, locking user principal into a pool that was halted for a safety reason.

## Finding Description
`MetricOmmPool` maintains a `pauseLevel` variable (0 = active, 1 = admin-paused, 2 = protocol-paused). The `whenNotPaused` modifier calls `_checkNotPaused()`, which reverts with `PoolPaused` whenever `pauseLevel != 0`: [1](#0-0) [2](#0-1) 

`swap` correctly applies this guard: [3](#0-2) 

`addLiquidity` does not — it carries only `nonReentrant`, with no pause check: [4](#0-3) 

The call to `LiquidityLib.addLiquidity` pulls tokens from the caller via callback and permanently updates `binTotals`, `_binStates`, `_binTotalShares`, and `_positionBinShares` — all real, persistent state changes that commit user principal to the pool. `removeLiquidity` intentionally omits the guard so LPs can always exit; omitting it on `addLiquidity` is not intentional and has no analogous justification.

`setPause` is callable only by the factory, confirming the pause mechanism is a real operational control: [5](#0-4) 

## Impact Explanation
A user who calls `addLiquidity` while `pauseLevel` is 1 or 2 transfers real token balances into a pool halted for a safety reason. If the pause was triggered by an oracle manipulation or active exploit of the bin-price math, the freshly deposited tokens are immediately at risk once the pool is unpaused or if the attacker reaches another non-paused entry point. This constitutes direct loss of user principal that bypasses the intended safety boundary of the pause mechanism — matching the "direct loss of user principal" and "broken core pool functionality causing loss of funds" allowed impacts.

## Likelihood Explanation
The pool's pause mechanism is explicitly designed for emergency use. Any time a protocol-level pause (level 2) is triggered, uninformed LPs or bots monitoring mempool liquidity opportunities can still call `addLiquidity`. No special privilege is required from the depositor; any address can trigger it. The missing guard is a single-transaction reachable path with no preconditions beyond the pool being paused.

## Recommendation
Add `whenNotPaused` to `addLiquidity`, mirroring the guard already present on `swap`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
```

`removeLiquidity` should remain unguarded so LPs can always exit a paused pool.

## Proof of Concept
1. Pool is deployed and operating normally (`pauseLevel == 0`).
2. Protocol detects an oracle anomaly and calls `factory.setPause(poolAddr, 2)` → `pool.setPause(2)`. `pauseLevel` is now 2.
3. `swap` is blocked: `whenNotPaused` → `_checkNotPaused()` → reverts with `PoolPaused`.
4. Alice calls `pool.addLiquidity(alice, salt, deltas, callbackData, extensionData)`. The call succeeds — no pause check exists at L182-196.
5. `LiquidityLib.addLiquidity` executes, the callback pulls Alice's tokens, and `binTotals.scaledToken0/1`, `_binStates`, `_binTotalShares`, and `_positionBinShares` are updated with her deposit.
6. Alice's tokens are now locked in a pool paused due to a safety issue. If the underlying issue (e.g., manipulated oracle price) is exploited before or after unpausing, Alice's principal is drained through a subsequent swap at the bad price.
