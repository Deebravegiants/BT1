### Title
Stale `drepDelegs` Reverse Mapping Causes Incorrect Delegation Clearing on DRep Unregistration During Bootstrap Phase — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version < 10), when a stake credential re-delegates from DRep A to DRep B, DRep A's `drepDelegs` reverse-index set is **not** updated to remove the stake credential (documented as issue #4772 and preserved intentionally). When DRep A subsequently unregisters via `ConwayUnRegDRep`, the handler unconditionally clears the `dRepDelegationAccountStateL` field for **every** credential in `drepDelegs`, including credentials that have already re-delegated to DRep B. This silently destroys those credentials' active DRep delegation, removing their governance participation without their knowledge or consent.

---

### Finding Description

**Root cause — stale reverse mapping not updated on re-delegation (bootstrap phase)**

In `processDelegationInternal` (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs`, lines 347–377), the `delegVote` branch handles re-delegation to a new DRep. During the bootstrap phase (`pvMajor pv < natVersion @10`), the `preserveIncorrectDelegation` flag is `True`, which causes the code to only **add** the stake credential to the new DRep's `drepDelegs` set without **removing** it from the old DRep's `drepDelegs` set:

```haskell
| isNothing mAccountState || preserveIncorrectDelegation ->
    certVStateL . vsDRepsL
      %~ Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
```

After this operation:
- Stake credential `S`'s forward mapping (`dRepDelegationAccountStateL`) correctly points to DRep B.
- DRep A's `drepDelegs` still contains `S` (stale entry).
- DRep B's `drepDelegs` now contains `S`.

**Exploitable consequence — DRep unregistration clears delegations unconditionally**

In `conwayGovCertTransition` (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`, lines 234–254), the `ConwayUnRegDRep` handler uses `drepDelegs` to clear delegations:

```haskell
clearDRepDelegations delegs accountsMap =
  foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
...
Just dRepState ->
  certState'
    & certDStateL . accountsL . accountsMapL
      %~ clearDRepDelegations (drepDelegs dRepState)
```

`Map.adjust (dRepDelegationAccountStateL .~ Nothing)` is applied **unconditionally** to every credential in `drepDelegs`. It does not verify whether the credential is currently delegated to the unregistering DRep or to a different one. When DRep A unregisters, `S`'s `dRepDelegationAccountStateL` is set to `Nothing`, even though `S` is legitimately delegated to DRep B.

**The PV10 fix does not restore lost delegations**

`updateDRepDelegations` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/HardFork.hs` (lines 82–105) rebuilds `drepDelegs` from the forward mapping at the PV10 hard fork. However, since `S`'s `dRepDelegationAccountStateL` was already set to `Nothing` by DRep A's unregistration, `S` will not be added to any DRep's delegation set at PV10. The damage is permanent until `S` manually re-delegates.

---

### Impact Explanation

**Classification: Medium** — Attacker-controlled certificate (`ConwayUnRegDRep`) modifies governance delegation state outside design parameters.

An attacker who registers as a DRep, attracts delegators, and then unregisters after those delegators have re-delegated to a legitimate DRep will cause those delegators to silently lose their DRep delegation. Concretely:

1. Attacker registers DRep A.
2. Victim delegates stake credential `S` to DRep A.
3. Victim re-delegates `S` to DRep B (a legitimate DRep).
4. Attacker unregisters DRep A.
5. `S`'s `dRepDelegationAccountStateL` is set to `Nothing` — `S` is no longer delegated to DRep B.
6. DRep B's voting power is reduced without DRep B or `S` being aware.

If many delegators are affected, this can materially reduce a targeted DRep's voting power and distort governance ratification outcomes. The affected stake credentials must manually re-delegate to restore participation.

---

### Likelihood Explanation

**Medium.** The bootstrap phase is the current live mainnet phase for Conway (PV9). DRep unregistration is a normal, permissionless operation requiring only the DRep's own credential witness. Any registered DRep can trigger this. The attacker does not need a majority, privileged access, or leaked keys — only their own DRep credential.

---

### Recommendation

In `clearDRepDelegations` (or at the call site in `ConwayUnRegDRep`), add a guard to only clear the delegation for a credential if its current `dRepDelegationAccountStateL` actually points to the unregistering DRep:

```haskell
clearDRepDelegations unregCred delegs accountsMap =
  foldr
    ( Map.adjust
        ( \as -> case as ^. dRepDelegationAccountStateL of
            Just (DRepCredential c) | c == unregCred ->
              as & dRepDelegationAccountStateL .~ Nothing
            _ -> as
        )
    )
    accountsMap
    delegs
```

This mirrors the correct behavior already implemented in `unDelegReDelegDRep` for the non-bootstrap path, which checks the current delegation before modifying it.

```haskell
updateDRepDelegations :: ConwayEraCertState era => CertState era -> CertState era
updateDRepDelegations certState =
  let accountsMap = certState ^. certDStateL . accountsL . accountsMapL
      dReps =
        -- Reset all delegations in order to remove any inconsistencies
        -- Delegations will be reset accordingly below.
        Map.map (\dRepState -> dRepState {drepDelegs = Set.empty}) $
          certState ^. certVStateL . vsDRepsL
      (dRepsWithDelegations, accountsWithoutUnknownDRepDelegations) =
        Map.mapAccumWithKey adjustDelegations dReps accountsMap
      adjustDelegations ds stakeCred accountState =
        case accountState ^. dRepDelegationAccountStateL of
          Just (DRepCredential dRep) ->
            let addDelegation _ dRepState =
                  Just $ dRepState {drepDelegs = Set.insert stakeCred (drepDelegs dRepState)}
             in case Map.updateLookupWithKey addDelegation dRep ds of
                  (Nothing, _) -> (ds, accountState & dRepDelegationAccountStateL .~ Nothing)
                  (Just _, ds') -> (ds', accountState)
          _ -> (ds, accountState)
   in certState
        -- Remove dangling delegations to non-existent DReps:
        & certDStateL . accountsL . accountsMapL .~ accountsWithoutUnknownDRepDelegations
        -- Populate DRep delegations with delegatees
        & certVStateL . vsDRepsL .~ dRepsWithDelegations
```
