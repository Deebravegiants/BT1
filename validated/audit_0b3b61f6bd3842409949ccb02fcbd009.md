### Title
Wrong Chain-Availability Set Used in `verify_foreign_transaction` Gate Allows Single Byzantine Participant to Block All Foreign-Chain Signing - (File: `crates/contract/src/lib.rs`)

---

### Summary
`verify_foreign_transaction` gates on `get_supported_foreign_chains()` — the strict all-participant intersection — instead of `get_available_foreign_chains()` — the threshold-based set. A single Byzantine participant strictly below the signing threshold can register an empty foreign-chain config, collapsing the intersection to the empty set and permanently blocking every `verify_foreign_transaction` request.

---

### Finding Description

In `verify_foreign_transaction`, the contract checks whether the requested chain is in the supported set:

```rust
// crates/contract/src/lib.rs  ~line 533-541
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported {
            requested: requested_chain,
        }
        .to_string(),
    );
}
``` [1](#0-0) 

`get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains. The design document explicitly identifies this as the wrong check:

> "Today, `get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains … A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down." [2](#0-1) 

The correct check, per the design, is `get_available_foreign_chains()`, which requires ≥ `signing_threshold` active participants to cover the chain:

> "`verify_foreign_transaction(C)` is **rejected unless `C` is available**" [3](#0-2) 

The `available_foreign_chains` cache is correctly computed in `update_available_chains_config_cache` using a `count >= threshold` filter:

```rust
// crates/contract/src/foreign_chains_metadata.rs  ~line 61-65
self.available_foreign_chains = chain_to_supporter_count
    .into_iter()
    .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
    .collect::<BTreeSet<_>>()
    .into();
``` [4](#0-3) 

But this correctly-computed set is never consulted by `verify_foreign_transaction`. The production gate still calls the wrong function.

**Analog to the external report:** The external report describes a check where the wrong values are placed in the comparison registers (`txCalldataLen + size < offset` instead of `offset + size < txCalldataLen`). Here, the wrong set is placed in the availability check: `get_supported_foreign_chains()` (all-participant intersection) is used where `get_available_foreign_chains()` (threshold-based set) is required. Both are "wrong operand in a critical guard" bugs.

---

### Impact Explanation

A single Byzantine participant strictly below the signing threshold calls `register_foreign_chain_config` with an empty `ForeignChainsConfig`. Because `get_supported_foreign_chains()` is the intersection across all active participants, one empty registration collapses the result to the empty set. Every subsequent `verify_foreign_transaction` call panics with `ForeignChainNotSupported`, regardless of which chain is requested. All pending and future foreign-chain signing requests are rejected at the contract gate — the MPC nodes never receive them. Bridge contracts waiting for MPC-signed foreign-chain attestations time out; depending on bridge design, user funds can be permanently frozen.

This matches: **Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

---

### Likelihood Explanation

The attack requires exactly one TEE-attested participant (below threshold) to submit a single `register_foreign_chain_config` call with an empty chain set. No collusion, no key material, no network access beyond normal participant operation is needed. The call is a standard participant-facing contract method. The attacker's cost is one NEAR transaction.

---

### Recommendation

Replace the gate in `verify_foreign_transaction` with the threshold-based check:

```rust
// crates/contract/src/lib.rs  ~line 533
// Before (wrong):
let supported_chains = self.get_supported_foreign_chains();

// After (correct):
let supported_chains = self.get_available_foreign_chains();
``` [5](#0-4) 

This is already the stated intent in the design document and the `get_available_foreign_chains` view method already exists and is correctly populated. [6](#0-5) 

---

### Proof of Concept

1. Deploy the contract in Running state with 4 participants, signing threshold 3. All 4 participants register Bitcoin as a covered chain. Bitcoin is whitelisted via `vote_update_foreign_chain_providers`.
2. Participant 1 (a Byzantine actor, below threshold) calls `register_foreign_chain_config` with an empty `ForeignChainsConfig` (no chains).
3. `get_supported_foreign_chains()` now returns `{}` (intersection of `{Bitcoin, …}` ∩ `{}` = `{}`).
4. Any caller submits `verify_foreign_transaction` for Bitcoin.
5. The contract panics: `ForeignChainNotSupported { requested: Bitcoin }`.
6. Confirm: `get_available_foreign_chains()` still returns `{Bitcoin}` (3 of 4 participants cover it, meeting threshold 3), proving the correct set is populated but never consulted. [7](#0-6) [8](#0-7)

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
