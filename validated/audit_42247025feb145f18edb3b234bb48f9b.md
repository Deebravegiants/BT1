Audit Report

## Title
Missing Time-Based Staleness Check in `getRate()` Enables Over-Minting via Stale Chainlink Price - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

## Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` validates only round completeness (`answeredInRound < roundID`) and never compares `updatedAt` against `block.timestamp`. When a Chainlink feed goes stale at an elevated price, any depositor of a supported collateral token receives rsETH computed against the inflated rate, extracting real value from the protocol and simultaneously exhausting the daily mint cap.

## Finding Description

`getRate()` at L26–37 of `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` reads `latestRoundData` and applies three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();   // round-based only
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [1](#0-0) 

The `updatedAt` field (aliased as `timestamp`) is captured but never compared against `block.timestamp`. When a Chainlink node stops publishing — because the price has not moved beyond the deviation threshold, or due to network congestion — the same round remains open (`answeredInRound == roundID`), so the round-based check passes and the stale price is returned as valid.

This price feeds directly into `viewSwapRsETHAmountAndFee`:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

The `limitDailyMint` modifier calls `viewSwapRsETHAmountAndFee` to compute `rsETHAmount`, checks it against `dailyMintLimit`, and increments `dailyMintAmount`: [3](#0-2) 

The `deposit(address token, ...)` function then calls `viewSwapRsETHAmountAndFee` a second time and mints the same inflated amount: [4](#0-3) 

Both the limit check and the actual mint use the same stale `tokenToETHRate`, so the attacker receives excess rsETH and simultaneously burns through the daily cap.

## Impact Explanation

**Critical — Direct theft of user funds / Protocol insolvency:** An attacker deposits collateral whose true ETH value is X but whose stale oracle value is 2X. The pool mints rsETH worth 2X ETH against collateral worth only X ETH. The attacker can redeem or sell the surplus rsETH, extracting real value from the protocol's backing. Existing rsETH holders are diluted and the protocol becomes undercollateralised by the difference.

**Medium — Temporary freezing of funds (secondary):** The inflated `rsETHAmount` counted against `dailyMintAmount` exhausts `dailyMintLimit` in a single deposit, blocking all other users from depositing for the remainder of the day.

## Likelihood Explanation

Chainlink feeds go stale without any oracle-operator compromise: a price that has not moved beyond the configured deviation threshold will not trigger a new round, leaving `updatedAt` hours behind `block.timestamp` while `answeredInRound == roundID`. This is a well-documented, passive failure mode. The attacker only needs to monitor the feed's `updatedAt` lag and submit a deposit when the stale price is sufficiently above the true market price. No privileged access, governance capture, or external protocol compromise is required.

## Recommendation

Add a configurable heartbeat/staleness check in `getRate()`:

```solidity
uint256 public constant MAX_STALENESS = 3600; // configure per feed

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 updatedAt, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
    if (ethPrice <= 0) revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

`MAX_STALENESS` should be set per feed based on its documented heartbeat (e.g., 3600 s for a 1-hour heartbeat feed).

## Proof of Concept

**Foundry test plan:**

1. Deploy a mock `AggregatorV3Interface` returning: `roundId=1`, `answer=2e18`, `updatedAt=block.timestamp - 7200`, `answeredInRound=1`.
2. Deploy `ChainlinkOracleForRSETHPoolCollateral` pointing at the mock.
3. Deploy `RSETHPoolV3` with `rsETHOracle` returning `1e18` (1:1 rsETH/ETH), collateral token oracle set to the above, and `dailyMintLimit = 1000e18`.
4. Attacker holds 500 collateral tokens (true value: 500 ETH; stale oracle value: 1000 ETH).
5. Call `pool.deposit(token, 500e18, "")`.
   - `viewSwapRsETHAmountAndFee` returns `rsETHAmount ≈ 1000e18` (2× true value).
   - `limitDailyMint`: `dailyMintAmount += 1000e18` → cap exhausted.
   - `wrsETH.mint(attacker, 1000e18)` — attacker receives 2× expected rsETH.
6. Assert:
   - `pool.dailyMintAmount() == 1000e18` (cap fully consumed).
   - `wrsETH.balanceOf(attacker) == 1000e18` (2× true value received).
   - Any subsequent deposit by another user reverts with `DailyMintLimitExceeded`.

The `answeredInRound < roundID` guard does not fire because the mock keeps `answeredInRound == roundID`. The missing `block.timestamp - updatedAt > heartbeat` check is the sole gate that would have rejected the stale price.
