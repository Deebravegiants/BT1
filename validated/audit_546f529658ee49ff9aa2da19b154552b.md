Audit Report

## Title
Unsafe `int256` to `uint256` Cast Without Negativity Check Freezes Protocol - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` casts the `int256 price` from Chainlink's `latestRoundData()` directly to `uint256` without validating it is positive. A zero or negative price causes the subsequent `uint256(price) * 1e18` multiplication to overflow under Solidity 0.8 checked arithmetic, reverting every protocol function that depends on asset pricing. The same codebase already applies the correct guard in `ChainlinkOracleForRSETHPoolCollateral`, confirming developer awareness of the pattern.

## Finding Description
In `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

There is no guard on `price`. If `price == -1`, then `uint256(-1) = 2**256 - 1`. Multiplying by `1e18` overflows and reverts under Solidity 0.8 checked arithmetic.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` applies the correct guard:

```solidity
if (ethPrice <= 0) revert InvalidPrice();
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / ...;
``` [2](#0-1) 

The revert propagates through four confirmed call chains:

- **Price update:** `updateRSETHPrice()` → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(asset)` → revert [3](#0-2) 
- **Deposits:** `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(asset)` → revert [4](#0-3) 
- **Withdrawals:** `getExpectedAssetAmount()` → `lrtOracle.getAssetPrice(asset)` → revert [5](#0-4) 
- **Unlock queue:** `_createUnlockParams()` → `lrtOracle.getAssetPrice(asset)` → revert [6](#0-5) 

No existing check in `ChainlinkPriceOracle` prevents this path. The `onlySupportedAsset` modifier only validates that a price feed is registered, not that the returned value is valid. [7](#0-6) 

## Impact Explanation
**Medium — Temporary freezing of funds.** Any single-block anomalous (zero or negative) price from a supported asset's Chainlink feed causes all user-facing protocol operations — deposits, withdrawal initiation, unlock queue processing, and rsETH price updates — to revert simultaneously. Funds committed to pending withdrawal requests are frozen until the feed recovers or governance replaces the oracle. This matches the allowed impact class of temporary freezing of funds.

## Likelihood Explanation
No privileged access is required. Any unprivileged caller invoking `depositETH()`, `depositAsset()`, `initiateWithdrawal()`, `unlockQueue()`, or the public `updateRSETHPrice()` triggers the revert automatically if any supported asset's feed returns a non-positive value. Chainlink feeds have historically returned zero or anomalous values during oracle incidents. The zero-validation state means a single anomalous round is sufficient to trigger the freeze.

## Recommendation
Add a positivity check before the cast, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
if (price <= 0) revert InvalidPrice();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Additionally, add staleness checks (`answeredInRound >= roundId`, `updatedAt != 0`) consistent with `ChainlinkOracleForRSETHPoolCollateral.getRate()`. [8](#0-7) 

## Proof of Concept
1. Deploy a mock Chainlink feed for a supported asset (e.g., stETH) that returns `price = -1`.
2. Register it via `ChainlinkPriceOracle.updatePriceFeedFor(stETH, mockFeed)`.
3. Call `LRTDepositPool.depositAsset(stETH, amount, minRSETH, "")` as any user.
4. Execution reaches `ChainlinkPriceOracle.getAssetPrice(stETH)`: `uint256(-1) = 2**256 - 1`; `(2**256 - 1) * 1e18` overflows → revert.
5. Repeat for `initiateWithdrawal`, `unlockQueue`, and `updateRSETHPrice` — all revert identically.

Foundry fork test: fork mainnet, use `vm.mockCall` on the registered stETH/ETH Chainlink feed to return `answer = -1`, then assert all four entry points revert with arithmetic overflow.
