### Title
`deployToken` Uses `CREATE` Opcode for Bridge Token Deployment, Enabling Reorg-Induced Permanent Fund Lock - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.deployToken()` deploys each bridged ERC-20 proxy via Solidity's `new` keyword, which compiles to the `CREATE` opcode. The resulting proxy address is derived solely from `OmniBridge`'s address and its current nonce. On reorg-susceptible chains where the bridge is deployed (Polygon, Arbitrum, Optimism), a reorg can change the nonce at the time `deployToken` re-executes, producing a different proxy address than the one the NEAR side already recorded. Any `finTransfer` signed by the NEAR MPC against the pre-reorg address will permanently fail, locking user funds.

### Finding Description

In `OmniBridge.deployToken()`, the bridge token proxy is deployed using the `CREATE` opcode:

```solidity
address bridgeTokenProxy = address(
    new ERC1967Proxy(
        tokenImplementationAddress,
        abi.encodeWithSelector(
            BridgeToken.initialize.selector,
            metadata.name,
            metadata.symbol,
            decimals
        )
    )
);
```

The resulting address is `keccak256(rlp(OmniBridge_address, nonce))[12:]`. No `CREATE2` salt binding the token identity (e.g., `metadata.token`) is used.

After deployment, the NEAR side observes the `DeployToken` event and permanently records the proxy address as the canonical EVM address for that NEAR token ID. The NEAR MPC then signs `finTransfer` payloads that hard-code this address in `payload.tokenAddress`.

If a reorg reverts the block containing `deployToken`, and the transaction is re-included in a block where `OmniBridge`'s nonce differs (because other `deployToken` calls were included or excluded by the reorg), the proxy lands at a different address. The NEAR side's recorded address is now stale.

The `finTransfer` function verifies the MPC signature over a payload that includes `payload.tokenAddress`:

```solidity
Borsh.encodeAddress(payload.tokenAddress),
```

The NEAR MPC cannot re-sign with the new address without consuming a new origin nonce — the original transfer's nonce is already spent on the NEAR side. The `finTransfer` call with the old address will always revert because `isBridgeToken[oldAddress]` is `false`, `multiTokens[oldAddress].tokenAddress` is `address(0)`, and `customMinters[oldAddress]` is `address(0)`, causing it to fall through to `IERC20(oldAddress).safeTransfer(...)` against a non-existent contract.

### Impact Explanation

**Critical — Irreversible fund lock.** A user who initiated a NEAR→EVM transfer after the pre-reorg `DeployToken` event was observed will have their NEAR-side tokens burned/locked and a signed `finTransfer` payload that permanently fails on the EVM side. There is no on-chain recovery path: the destination nonce has not been consumed (the `finTransfer` reverts), but the NEAR origin nonce has been spent and the MPC signature is bound to the stale token address. Funds become permanently unclaimable.

### Likelihood Explanation

**Medium.** The bridge is deployed on Polygon (confirmed reorgs of 1.5+ minutes documented), Arbitrum, and Optimism (both subject to fraud-proof-triggered rollbacks). `deployToken` is permissionless — any holder of a valid NEAR MPC signature can submit it. The window between the NEAR side observing a `DeployToken` event and a reorg reverting that block is realistic, especially for automated relayers that act on event confirmation without waiting for sufficient finality.

### Recommendation

Replace `new ERC1967Proxy(...)` with `CREATE2` using a salt that deterministically encodes the NEAR token identity:

```solidity
bytes32 salt = keccak256(abi.encodePacked(metadata.token));
address bridgeTokenProxy = address(
    new ERC1967Proxy{salt: salt}(
        tokenImplementationAddress,
        abi.encodeWithSelector(
            BridgeToken.initialize.selector,
            metadata.name,
            metadata.symbol,
            decimals
        )
    )
);
```

This ensures the same NEAR token ID always maps to the same EVM proxy address regardless of nonce state, making the address reorg-stable and pre-computable by the NEAR side.

### Proof of Concept

1. Relayer calls `deployToken(sig, {token: "wrap.near", ...})` on Polygon. `OmniBridge` nonce = 5. Proxy deployed at address `A`. `DeployToken(A, "wrap.near", ...)` emitted.
2. NEAR side observes event, records `nearToEthToken["wrap.near"] = A`.
3. User bridges 1000 wNEAR from NEAR to EVM. NEAR side burns 1000 wNEAR, signs `finTransfer` payload with `tokenAddress = A, amount = 1000, recipient = user`.
4. Polygon reorg reverts the block from step 1. In the new chain, a different `deployToken` call was included first, advancing `OmniBridge`'s nonce to 6 before `deployToken("wrap.near", ...)` re-executes. Proxy now deployed at address `B ≠ A`.
5. Relayer submits `finTransfer(sig, {tokenAddress: A, amount: 1000, recipient: user})`. `isBridgeToken[A]` is `false`. `multiTokens[A]` is empty. `customMinters[A]` is `address(0)`. Falls to `IERC20(A).safeTransfer(user, 1000)` — reverts (no code at `A`).
6. User's 1000 wNEAR is permanently locked. The NEAR origin nonce is spent; the MPC will not re-sign with address `B` for the same transfer. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
