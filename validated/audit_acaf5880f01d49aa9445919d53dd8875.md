### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Pending `request` — Enables Cross-Request Forged Foreign-Chain Verification - (File: crates/contract/src/lib.rs)

---

### Summary

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key. It never checks that `response.payload_hash` is the canonical hash of `ForeignTxSignPayload{ request = <the pending request>, values = … }`. A single malicious attested participant (strictly below the signing threshold) can take a legitimately-produced threshold signature for foreign-tx request **B** and submit it as the response to a different pending request **A**, causing the user who submitted request A to receive a valid MPC-signed attestation of the wrong foreign-chain data.

---

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs the following checks before resolving the pending yield:

1. Caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`). [1](#0-0) 
2. The signature in `response` is valid over `response.payload_hash` using the **root** public key (no tweak/derivation). [2](#0-1) 
3. The pending yield for `request` is resolved with the full `response`. [3](#0-2) 

What is **absent**: there is no check that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload{ request = request, values = … }))` for any set of `values`. The contract never reconstructs or constrains the hash; it simply trusts whatever hash the responding node supplies, as long as the signature over it is valid.

This is structurally identical to the LiFi finding: the connection between the on-chain request and the signed payload is enforced entirely off-chain (by MPC node behaviour), not by the contract itself.

For contrast, the regular `respond` function derives the expected public key from the request's tweak, cryptographically binding the signature to the specific `(predecessor, path)` pair of the original request. [4](#0-3)  `respond_verify_foreign_tx` uses the root key with no tweak, so every foreign-tx signature is interchangeable at the contract level. [5](#0-4) 

On the node side, `build_signature_request` confirms the tweak is always `[0u8; 32]` for foreign-tx signing: [6](#0-5) 

The `ForeignTxSignPayload` that nodes actually sign embeds the full `ForeignChainRpcRequest` and the extracted values: [7](#0-6)  but the contract never reconstructs this structure to verify the binding.

---

### Impact Explanation

**High — Forged foreign-chain verification causing invalid bridge execution.**

A user who calls `verify_foreign_transaction` for Bitcoin tx X receives `VerifyForeignTransactionResponse{ payload_hash_B, signature_B }` — a valid root-key signature that encodes Bitcoin tx Y's block hash and extracted values. Any downstream NEAR contract that uses this response to gate a bridge action (e.g., "release funds because tx X was confirmed") will be presented with a cryptographically valid MPC attestation for the wrong transaction. If the downstream contract does not independently reconstruct and compare the expected `payload_hash`, it will accept the forged attestation, enabling invalid bridge execution or double-spend conditions.

---

### Likelihood Explanation

**Medium.** The attack requires only a single malicious attested participant — strictly below the signing threshold. The attacker does not need to forge a signature; they only need to:

1. Participate honestly in the MPC signing round for request B (obtaining a legitimately-produced `(payload_hash_B, signature_B)`).
2. Race to call `respond_verify_foreign_tx(request = A, response = { payload_hash_B, signature_B })` before the honest nodes respond to request A.

Because `resolve_yields_for` drains all pending yields for request A on the first accepted response, the honest nodes' subsequent correct responses will find no yield to resolve. The attacker wins the race with a single transaction. Two concurrent pending requests for different transactions (a common scenario in a live bridge) are sufficient.

---

### Recommendation

The contract should verify that `response.payload_hash` is the canonical hash of the payload derived from the pending `request`. The simplest fix is to include the `ExtractedValue` list in the response and recompute the hash on-chain:

```rust
// In respond_verify_foreign_tx, after verifying the signature:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(), // add values to VerifyForeignTransactionResponse
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, include the `request` hash as a domain separator in the signed payload so that a signature for request B is cryptographically invalid for request A, mirroring how the regular `respond` path uses a per-request tweak.

---

### Proof of Concept

1. Two users submit concurrent requests:
   - Alice: `verify_foreign_transaction({ chain: Bitcoin, tx_id: X, confirmations: 6, extractors: [BlockHash] })`
   - Bob: `verify_foreign_transaction({ chain: Bitcoin, tx_id: Y, confirmations: 6, extractors: [BlockHash] })`

2. The MPC network signs Bob's request, producing `payload_hash_B = SHA-256(borsh({ request: Bob's, values: [block_hash_Y] }))` and `signature_B`.

3. The malicious attested participant calls:
   ```
   respond_verify_foreign_tx(
     request = Alice's request,
     response = { payload_hash: payload_hash_B, signature: signature_B }
   )
   ```

4. The contract at line 729 verifies `verify_ecdsa_signature(signature_B, payload_hash_B, root_key)` — this passes because `signature_B` is a valid root-key signature. [8](#0-7) 

5. The contract resolves Alice's yield with `response = { payload_hash_B, signature_B }`. [3](#0-2) 

6. Alice's NEAR contract receives a valid MPC-signed attestation claiming that Bitcoin tx Y (not X) was confirmed. Any bridge logic that does not independently reconstruct and compare `payload_hash` will accept this as proof that tx X occurred.
