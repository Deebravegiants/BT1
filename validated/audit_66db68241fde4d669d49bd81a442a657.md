### Title
Missing Vote Signature Verification Allows Any Peer to Forge Validator Votes and Manipulate Consensus Decisions - (`crates/apollo_consensus/src/single_height_consensus.rs`)

---

### Summary

`SingleHeightConsensus::handle_vote` accepts and counts prevote/precommit votes toward quorum without verifying the cryptographic signature on the vote. Any network peer can craft a `Vote` message claiming to be any high-weight committee validator and have it counted at full staking weight, enabling a single attacker to manufacture a 2/3 quorum for an arbitrary `ProposalCommitment` and force the wrong block to be decided.

---

### Finding Description

In `handle_vote`, the only admission checks performed are:

1. `vote.height == current_height`
2. `vote.voter` is a member of the current committee

The `vote.signature` field (`RawSignature`) is never verified against the voter's `public_key` stored in the `Staker` struct. The code contains an explicit unresolved TODO acknowledging this:

```rust
pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
    // TODO(Asmaa): verify the signature
    ...
    if !self.committee.members().iter().any(|s| s.address == vote.voter) {
        debug!("Ignoring vote from non validator: vote={:?}", vote);
        return VecDeque::new();
    }
    // vote is accepted and counted toward quorum
``` [1](#0-0) 

The `Staker` struct carries a `public_key: Felt` field specifically for this purpose: [2](#0-1) 

And the verification function `verify_precommit_vote_signature` is fully implemented and available: [3](#0-2) 

Yet neither is called from `handle_vote`. The `Vote` struct carries a `signature: RawSignature` field that is populated by the signing path (`sign_precommit_vote`) but is silently ignored on the receiving path. [4](#0-3) 

---

### Impact Explanation

The consensus state machine uses stake-weighted quorum thresholds. A 2/3 Byzantine quorum is required for both prevote and precommit decisions: [5](#0-4) 

Vote weight is looked up directly from the committee by `voter` address: [6](#0-5) 

Because the address-to-weight lookup is performed without signature verification, an attacker who forges a vote with `voter = <high_weight_staker_address>` and `signature = RawSignature::default()` (or any bytes) will have that vote counted at the full staking weight of the impersonated validator.

The `upon_decision` rule fires when precommit quorum is reached for a specific `ProposalCommitment`: [7](#0-6) 

An attacker who can reach 2/3 precommit weight for an arbitrary commitment forces `DecisionReached` for a block the honest proposer never built, causing the wrong `ProposalCommitment` (and thus wrong state root, block hash, receipts, and events) to be finalized.

---

### Likelihood Explanation

The attack requires only the ability to send `Vote` messages over the consensus broadcast channel. The manager's `handle_vote` receives votes from `broadcast_channels.broadcasted_messages_receiver` with no prior authentication of the vote content: [8](#0-7) 

The committee is public (fetched from the staking contract), so all validator addresses and their weights are known to any observer. The attacker needs only to know a target validator's `ContractAddress` (public) and send a vote with any `signature` bytes — no private key is required. The duplicate-vote guard only prevents the same `(round, voter)` pair from being counted twice, but does not prevent the first forged vote from being accepted. [9](#0-8) 

---

### Recommendation

Call `verify_precommit_vote_signature` (and an equivalent for prevotes) inside `handle_vote` before accepting the vote. The committee already stores each staker's `public_key`:

```rust
pub(crate) fn handle_vote(&mut self, vote: Vote) -> Requests {
    // ... height and committee membership checks ...

    // Retrieve the staker's public key from the committee.
    let Some(staker) = self.committee.members().iter().find(|s| s.address == vote.voter) else {
        return VecDeque::new();
    };
    let public_key = PublicKey(staker.public_key);

    // Verify the vote signature before counting it.
    let block_hash = BlockHash(vote.proposal_commitment.unwrap_or_default().0);
    match verify_precommit_vote_signature(block_hash, vote.signature, public_key) {
        Ok(true) => {}
        Ok(false) => {
            warn!("Invalid signature on vote: {:?}", vote);
            return VecDeque::new();
        }
        Err(e) => {
            warn!("Signature verification error on vote: {:?}: {e}", vote);
            return VecDeque::new();
        }
    }
    // ... proceed to state machine ...
}
```

A separate prevote message digest scheme should be defined analogously to `build_precommit_vote_message_digest`.

---

### Proof of Concept

1. Observe the current committee for height `H` via the public staking contract. Identify the highest-weight validator `V` with address `A` and weight `W`.
2. Construct a `Vote`:
   ```
   Vote {
       vote_type: VoteType::Precommit,
       height: H,
       round: 0,
       proposal_commitment: Some(ProposalCommitment(<attacker_chosen_felt>)),
       voter: A,                        // impersonated address
       signature: RawSignature::default(), // any bytes; never checked
   }
   ```
3. Broadcast this vote to the consensus network. `handle_vote` will:
   - Pass the height check (`vote.height == H`).
   - Pass the committee membership check (`A` is in the committee).
   - Skip signature verification (the TODO is unimplemented).
   - Insert the vote into `self.precommits` with weight `W`.
4. Repeat for additional validator addresses until the accumulated weight exceeds 2/3 of `total_weight`.
5. `upon_decision` fires, emitting `SMRequest::DecisionReached` for the attacker-chosen `ProposalCommitment`, causing the node to finalize a block that was never legitimately proposed or executed. [10](#0-9) [7](#0-6)

        match message.height.cmp(&height) {
