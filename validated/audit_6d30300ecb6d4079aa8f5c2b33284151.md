### Title
Dust-Amount Transfer Permanently Locks User Tokens With No Redemption Path â€” (`near/omni-bridge/src/lib.rs`)

### Summary
A user who initiates a NEARâ†’EVM bridge transfer with an amount that is non-zero on the NEAR side but normalizes to zero on the destination chain (due to decimal-precision floor division) will have their tokens permanently locked or burned. The transfer is stored in `pending_transfers`, but `sign_transfer` always reverts for it, and no cancellation or refund path exists.

### Finding Description
The NEAR `init_transfer` function validates only that `fee.fee < transfer_message.amount` (a fee-vs-amount ordering check). It does not validate that `normalize_amount(amount_without_fee, decimals) > 0`. [1](#0-0) 

The normalization check is deferred to `sign_transfer`, which is called later by a relayer: [2](#0-1) 

`normalize_amount` uses floor division: [3](#0-2) 

For a token where NEAR uses 24 decimals and the EVM destination uses 6 decimals (`diff_decimals = 18`), any NEAR-side amount below `10^18` normalizes to 0. The `require!(amount_to_transfer > 0, ...)` guard in `sign_transfer` then causes a permanent revert.

The critical problem is the ordering:

1. **`init_transfer`** succeeds: the transfer is inserted into `pending_transfers`, tokens are burned (bridged token) or locked (native token), and the nonce is incremented. [4](#0-3) 

2. **`sign_transfer`** always reverts for this transfer: `normalize_amount` returns 0, the `require!` fires, and the NEAR transaction is rolled back â€” but `pending_transfers` is not rolled back because it was written in a prior transaction. [5](#0-4) 

3. **No removal path exists.** The only ways `pending_transfers` entries are removed are:
   - `sign_transfer_callback` (only reached after a successful MPC call â€” never reached here) [6](#0-5) 
   - `claim_fee_callback` (requires a proof from the destination chain â€” impossible since the transfer was never signed) [7](#0-6) 
   - `submit_transfer_to_utxo_chain_connector` (UTXO chains only) [8](#0-7) 
   - `remove_transfer_message_without_refund` inside `init_transfer_internal` on storage failure (not applicable post-insertion) [9](#0-8) 

There is no `cancel_transfer` or DAO-accessible removal function.

`update_transfer_fee` cannot help: it can only raise the fee, and even with fee=amount-1, `normalize_amount(1, decimals)` still returns 0 for large decimal differences. [10](#0-9) 

### Impact Explanation
User tokens are permanently locked in the bridge contract (native tokens) or permanently burned (bridged tokens) with no redemption path. The `pending_transfers` entry is also permanently orphaned, consuming storage. This is an irreversible fund lock matching the Critical impact category.

### Likelihood Explanation
Any token registered with a large decimal difference between NEAR (e.g., 24) and the EVM destination (e.g., 6 or 8) is affected. A user sending fewer than `10^(origin_decimals - decimals)` raw NEAR-side units triggers the bug. For a 24-vs-6 token, any amount below `10^18` yocto-units (i.e., below 1 micro-unit on the EVM side) is affected. This is reachable by any unprivileged user via the public `ft_transfer_call` â†’ `ft_on_transfer` â†’ `init_transfer` path.

### Recommendation
Add a normalization check inside `init_transfer` (or `init_transfer_internal`) before storing the transfer and locking/burning tokens:

```rust
// In init_transfer, after building transfer_message:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee().unwrap_or(0),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

This mirrors the guard already present in `sign_transfer` but moves it to the entry point, before any state mutation or token movement occurs. [5](#0-4) 

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 6` (e.g., a NEAR-native token bridged to an EVM chain with 6-decimal representation).
2. Call `ft_transfer_call` on the token contract with `amount = 1` (1 yoctoNEAR-equivalent), `msg = InitTransfer { fee: 0, recipient: "eth:0x...", ... }`.
3. `ft_on_transfer` â†’ `init_transfer` succeeds: `fee(0) < amount(1)` passes. The 1-unit token is locked. A `TransferMessage` with `origin_nonce = N` is inserted into `pending_transfers`.
4. Relayer calls `sign_transfer({ origin_chain: Near, origin_nonce: N }, ...)`.
5. `normalize_amount(1, {decimals:6, origin_decimals:24}) = 1 / 10^18 = 0`. `require!(0 > 0, ...)` â†’ **REVERT**.
6. Repeat step 4 indefinitely â€” always reverts. The 1-unit token is permanently locked with no recovery path.
