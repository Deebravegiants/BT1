Audit Report

## Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Unchecked `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`MetricOmmPoolFactory` enforces `maxAdminSpreadFeeE6` on pool-level admin fees in `setPoolAdminFees`, but the sibling function `setPoolBinAdditionalFees` forwards per-bin fee values (`addFeeBuyE6`, `addFeeSellE6`) directly to the pool with no cap check. A pool admin can set per-bin fees up to 65,535 (≈6.55% in E6 units) regardless of the factory owner's configured ceiling, causing traders to pay fees far above the intended cap on every swap through the affected bin.

## Finding Description
`setPoolAdminFees` correctly enforces the cap before applying fees: [1](#0-0) 

However, `setPoolBinAdditionalFees` performs no such check and forwards values directly to the pool: [2](#0-1) 

`MetricOmmPool.setBinAdditionalFees` only validates the bin index, not the fee magnitudes: [3](#0-2) 

The stored per-bin fees are then added directly to `baseFeeX64` on every swap through that bin: [4](#0-3) 

Since `addFeeBuyE6` and `addFeeSellE6` are `uint16`, they can reach 65,535 (≈6.55%). The factory owner's `maxAdminSpreadFeeE6` cap — which can be set as low as 0 — is entirely bypassed for per-bin fees. The existing guard in `setPoolAdminFees` is insufficient because it covers only the pool-level spread fee, not the per-bin additional fees that are independently applied to every swap.

## Impact Explanation
Every swap routed through a bin with inflated `addFeeBuyE6`/`addFeeSellE6` pays fees above the factory owner's cap. The excess fee is extracted from the trader's swap output, constituting a direct loss of user principal. This is a confirmed admin-boundary break: the pool admin exceeds the factory owner's fee cap, which is the primary protection against pool-admin fee abuse.

## Likelihood Explanation
The pool admin is semi-trusted only within caps and timelocks. Any pool admin (malicious or compromised) can call `setPoolBinAdditionalFees` at any time with no timelock, no cap check, and no factory-owner approval. The call path is short, requires no special setup beyond holding the pool admin role, and is immediately effective on the next swap.

## Recommendation
Add a cap check in `setPoolBinAdditionalFees` mirroring the check in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinFeeE6` cap so the factory owner can tune per-bin and pool-level limits independently.

## Proof of Concept
1. Factory owner deploys factory and sets `maxAdminSpreadFeeE6 = 1_000` (0.1%) via `setFeeCaps`.
2. Pool is created with a pool admin.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — no revert occurs.
4. `addFeeBuyE6 = addFeeSellE6 = 65_535` is stored in `_binStates[0]`.
5. A trader swaps through bin 0. Effective fee becomes `baseFeeX64 + 65535/1e6 ≈ 6.55%` instead of the capped `0.1%`.
6. The trader receives ≈6.45% less output than the factory owner's cap permits, with the excess accruing as spread fees — a direct loss of trader principal.
