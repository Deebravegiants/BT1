### Title
`OmniBridgeWormhole` permanently undeployable due to double `initializer` modifier in parent `OmniBridge.initialize()` — (`evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

### Summary

`OmniBridgeWormhole` inherits from `OmniBridge`. Its entry-point `initializeWormhole()` carries the `initializer` modifier and internally calls `OmniBridge.initialize()`, which also carries the `initializer` modifier. OpenZeppelin's `initializer` sets `_initialized = 1` and `_initializing = true` on the first call; the nested call to `initialize()` then sees `initialized != 0` and `isTopLevelCall == false`, causing an unconditional `InvalidInitialization()` revert. `initializeWormhole()` can never succeed.

### Finding Description

`OmniBridge.initialize()` is declared `public initializer`: [1](#0-0) 

`OmniBridgeWormhole.initializeWormhole()` is also declared `external initializer` and calls the parent's `initialize()` from within its body: [2](#0-1) 

OpenZeppelin's `initializer` modifier (v5) sets `_initialized = 1` and `_initializing = true` at the start of the outermost call. When `initialize()` is reached from inside `initializeWormhole()`, the modifier evaluates:

- `isTopLevelCall = false` (because `_initializing` is already `true`)
- `initialized = 1` (already written by the outer call)
- `initialSetup = false`, `construction = false`

Both conditions are false → `revert InvalidInitialization()`.

`initializeWormhole()` therefore **always reverts**. The fix, per OZ best practice, is to change `OmniBridge.initialize()` to use `onlyInitializing` so it can be called safely from within a child's `initializer`-guarded function.

`SelectivePausableUpgradable.__Pausable_init()` already uses `onlyInitializing` correctly: [3](#0-2) 

`OmniBridge.initialize()` does not follow the same pattern.

### Impact Explanation

`OmniBridgeWormhole` cannot be initialized through its intended path. If a deployer deploys the proxy with empty init data (a common pattern when initialization is done in a separate transaction) and `initializeWormhole()` reverts, the proxy is left with `_initialized == 0`. Any unprivileged caller can then invoke `OmniBridge.initialize()` directly on the proxy, granting themselves `DEFAULT_ADMIN_ROLE`. From that position the attacker can:

- Set `nearBridgeDerivedAddress` to an address they control.
- Call `finTransfer()` with self-signed payloads, minting or releasing arbitrary amounts of bridge tokens to themselves.
- Drain all native ETH and ERC-20 assets held by the bridge.

This satisfies **Critical — unauthorized release of bridge assets through deployment/verification failure**.

### Likelihood Explanation

The likelihood is medium. The bug is deterministic: every call to `initializeWormhole()` reverts. A deployer who separates proxy deployment from initialization (a standard two-step pattern) will leave the proxy open. A front-running bot monitoring the mempool for the failing `initializeWormhole()` transaction can immediately call `initialize()` and seize control. No privileged access or key compromise is required.

### Recommendation

Change `OmniBridge.initialize()` from `initializer` to `onlyInitializing`, following the same pattern already used in `SelectivePausableUpgradable`:

```diff
// OmniBridge.sol
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
-) public initializer {
+) public onlyInitializing {
```

`OmniBridgeWormhole.initializeWormhole()` retains `initializer` as the single top-level entry point, which is correct.

### Proof of Concept

1. Deploy `OmniBridgeWormhole` implementation with `_disableInitializers()` in constructor (already present).
2. Deploy `ERC1967Proxy(implementation, "")` — empty init data, proxy `_initialized == 0`.
3. Deployer calls `initializeWormhole(...)` → reverts with `InvalidInitialization()` because `initialize()` inside also carries `initializer`.
4. Attacker calls `OmniBridge.initialize(attackerAddr, attackerAddr, 1)` on the proxy → succeeds, attacker holds `DEFAULT_ADMIN_ROLE`.
5. Attacker calls `setNearBridgeDerivedAddress(attackerEOA)`.
6. Attacker crafts a `TransferMessagePayload` and signs it with `attackerEOA`, then calls `finTransfer()` → bridge mints or releases tokens to attacker.
