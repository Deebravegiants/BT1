### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Per-Bin Additional Fees — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces a `maxAdminSpreadFeeE6` cap on the pool admin's global spread fee through `setPoolAdminFees`. However, `setPoolBinAdditionalFees` allows the pool admin to set per-bin additional fees (`addFeeBuyE6` / `addFeeSellE6`) without any cap validation. These per-bin fees are added directly to the base fee in swap math, allowing the pool admin to charge traders above the factory-enforced ceiling — an exact admin-boundary break analog to the Hats Protocol phantom-level bypass.

---

### Finding Description

**The guarded path — global admin spread fee:**

`setPoolAdminFees` enforces the cap before writing to the pool:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

**The unguarded path — per-bin additional fees:**

`setPoolBinAdditionalFees` passes `addFeeBuyE6` and `addFeeSellE6` directly to the pool with **no cap check**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap validation — it only checks the bin index is in range:

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

**How the per-bin fee reaches traders:**

In swap math and in `getSellAndBuyPrices`, the per-bin fee is added directly to `baseFeeX64` to compute the effective buy/sell price:

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

The ask price presented to traders is then:

```solidity
uint256 askBeforeNotional = Math.mulDiv(marginalPriceX64, ONE_X64 + buyFeeX64, ONE_X64, Math.Rounding.Ceil);
``` [5](#0-4) 

A `uint16` field allows values up to **65,535 E6 ≈ 6.55%** additional fee per bin, on top of whatever global spread fee is already set. The factory's hard cap (`HARD_MAX_SPREAD_FEE_E6 = 200,000`) and the owner-configurable `maxAdminSpreadFeeE6` are both completely bypassed for this code path. [6](#0-5) 

**Phantom-level analogy:**

| Hats Protocol | Metric OMM |
|---|---|
| Constraint: 65,536 direct children per hat | Constraint: `maxAdminSpreadFeeE6` cap on admin fees |
| Bypass: phantom intermediate hat IDs with `0000` segments | Bypass: per-bin `addFeeBuyE6`/`addFeeSellE6` — a parallel fee channel with no cap check |
| Result: unlimited children, violating the tree invariant | Result: pool admin charges traders above the factory-enforced ceiling |

---

### Impact Explanation

The factory owner uses `maxAdminSpreadFeeE6` to guarantee to traders that no pool admin can charge more than a defined spread. A pool admin can silently set `addFeeBuyE6 = 65535` on any active bin, adding ~6.55% to the effective ask price (or reducing the effective bid price by ~6.55%) beyond the capped global fee. If the factory owner sets `maxAdminSpreadFeeE6 = 0` to prohibit admin spread fees entirely, the pool admin can still extract up to ~6.55% per bin through this uncapped channel. Traders receive worse execution than the factory's published caps imply, constituting a direct loss of swap output relative to the advertised fee regime.

---

### Likelihood Explanation

The pool admin is a semi-trusted role with direct, permissioned access to `setPoolBinAdditionalFees`. No special conditions, flash loans, or oracle manipulation are required. The call is a single transaction. Any pool admin who wishes to extract additional value from traders can do so immediately after pool deployment.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` mirroring the check in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce a combined cap: `globalAdminSpreadFee + addFeeBuyE6 ≤ maxAdminSpreadFeeE6`.

---

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 0` — intending to prohibit all admin spread fees.
2. Pool admin calls `setPoolAdminFees(pool, 0, 0)` — passes the cap check, no fee set globally.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — **no cap check fires**.
4. `_binStates[0].addFeeBuyE6 = 65535` and `addFeeSellE6 = 65535` are written to storage.
5. On the next swap through bin 0, `buyFeeX64 = baseFeeX64 + 65535/1e6 * ONE_X64`, inflating the ask price by ~6.55% above the oracle mid-price plus base spread.
6. Traders pay ~6.55% more than the factory's zero-admin-fee cap permits; the excess accrues as LP spread revenue controlled by the pool admin's liquidity positions.
