### Title
Unvalidated `builder` Field in `ProposalInit` Allows Proposer to Redirect All Block Transaction Fees to an Arbitrary Address — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`ProposalInit` carries two distinct identity fields: `proposer` (the consensus validator) and `builder` (the sequencer/fee-recipient address). Validators rigorously check `proposer` against the committee-elected value, but `builder` is never validated. Because `builder` flows directly into `BlockInfo.sequencer_address`, a malicious-but-committee-elected proposer can set `builder` to any arbitrary address, redirecting all transaction fees for that block to an attacker-controlled account. Every validator will accept and execute the block with the wrong sequencer address, producing wrong fee-balance state and a wrong block hash.

---

### Finding Description

`ProposalInit` has two distinct address fields:

```
pub proposer: ContractAddress,   // consensus validator — validated
pub builder: ContractAddress,    // fee recipient / sequencer — NOT validated
``` [1](#0-0) 

The `proposer` field is validated at two layers of the consensus stack:

- `handle_proposal` in `manager.rs` compares the received `init.proposer` against the committee-derived expected proposer and drops the proposal on mismatch.
- `single_height_consensus.rs` performs the same check before forwarding to validation. [2](#0-1) [3](#0-2) 

`builder`, however, is never checked. `is_proposal_init_valid` validates `height`, `l1_da_mode`, `l2_gas_price_fri`, `timestamp`, `starknet_version`, `version_constant_commitment`, all four L1 gas-price fields, and `fee_proposal_fri` — but contains no check on `init.builder`. [4](#0-3) 

`ProposalInitValidation`, the struct that carries the validator's locally-trusted reference values, has no `builder` field at all: [5](#0-4) 

After `is_proposal_init_valid` passes, `initiate_validation` calls `convert_to_sn_api_block_info`, which maps `init.builder` directly to `sequencer_address` in the `BlockInfo` sent to the batcher:

```rust
sequencer_address: init.builder,
``` [6](#0-5) 

`sequencer_address` is the fee-transfer recipient for every transaction executed in the block. The batcher executes all transactions with this address as the fee sink, and the block hash commits to it.

On the build side, the proposer sets `builder` from its own static config:

```rust
builder_address: self.config.static_config.builder_address,
``` [7](#0-6) 

A malicious proposer simply configures `builder_address` to any address it controls. No validator checks this value.

---

### Impact Explanation

**Critical — Incorrect fee/balance with economic impact; wrong state committed.**

Every transaction fee in the block is transferred to `sequencer_address = init.builder`. If a malicious proposer sets `builder` to an attacker-controlled address, all fees for that block are stolen. The resulting state diff records the wrong fee-token balance changes, and the block hash commits to the wrong `sequencer_address`, producing a permanently incorrect on-chain state root.

---

### Likelihood Explanation

The proposer role rotates among committee validators. Any validator that reaches its proposer turn can exploit this by setting `builder_address` in its local config to an arbitrary address before proposing. No special network access or cryptographic capability is required beyond being a scheduled proposer. The attack is silent — validators log no warning and produce a `ProposalFinMismatch` only if the batcher-computed commitment diverges, which it will not, because the batcher itself uses the attacker-supplied `builder` address.

---

### Recommendation

Add `builder: ContractAddress` to `ProposalInitValidation` and check it in `is_proposal_init_valid`:

```rust
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!("builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder),
    ));
}
```

Populate `proposal_init_validation.builder` from the node's own `static_config.builder_address` (or, once available, from the committee). This mirrors the existing pattern for `proposer` validation and closes the gap. [8](#0-7) 

---

### Proof of Concept

1. A validator is scheduled as proposer for height H.
2. Before proposing, it sets `consensus_manager_config.context_config.static_config.builder_address = <attacker_address>` in its config.
3. It calls `build_proposal`. `initiate_build` constructs `ProposalInit { builder: <attacker_address>, ... }` and broadcasts it.
4. Every receiving validator calls `is_proposal_init_valid`. The function checks `height`, `l1_da_mode`, `l2_gas_price_fri`, gas prices, `starknet_version`, `version_constant_commitment`, and `fee_proposal_fri` — all of which are legitimate. `builder` is never examined. Validation passes.
5. `initiate_validation` calls `convert_to_sn_api_block_info`, producing `BlockInfo { sequencer_address: <attacker_address>, ... }`.
6. The batcher executes all transactions with `<attacker_address>` as the fee recipient. All fee-token balances are credited to the attacker.
7. `finish_proposal` returns a `partial_block_hash` that commits to `<attacker_address>` as the sequencer. The `ProposalFin` commitment matches on both sides (proposer and validator computed the same wrong hash), so `ProposalFinMismatch` is never triggered.
8. Consensus reaches decision; the block is finalized with the wrong sequencer address and wrong fee balances permanently committed to state. [9](#0-8) [10](#0-9)

```rust
#[instrument(level = "warn", skip_all, fields(?proposal_init_validation, ?init_proposed))]
async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is in the future: now={}, block_timestamp_window_seconds={}, \
                 proposed={}",
                now,
                proposal_init_validation.block_timestamp_window_seconds,
                init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.starknet_version != proposal_init_validation.starknet_version {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "starknet_version mismatch: expected={:?}, proposed={:?}",
                proposal_init_validation.starknet_version, init_proposed.starknet_version
            ),
        ));
    }
    // `version_constant_commitment` is proposer-supplied (network-derived). It is not yet a real
    // commitment (see `expected_version_constant_commitment`): the only valid value is the
    // sentinel, so reject anything else. Enforcing the same value the proposer emits keeps the two
    // sides in lockstep, so a real value cannot ship on one side without the other.
    let expected_commitment = expected_version_constant_commitment();
    if init_proposed.version_constant_commitment != expected_commitment {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "version_constant_commitment mismatch: expected={expected_commitment:?}, \
                 proposed={:?}",
                init_proposed.version_constant_commitment
            ),
        ));
    }
    if !(init_proposed.height == proposal_init_validation.height
        && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
        && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
    {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            "ProposalInit validation failed".to_string(),
        ));
    }
