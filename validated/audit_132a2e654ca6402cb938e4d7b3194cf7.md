Audit Report

## Title
Uninitialized `dailyMintLimit` Causes All Deposits to Revert Until Admin Configures the Limit - (File: `contracts/pools/RSETHPoolV3.sol`)

## Summary
`RSETHPoolV3.initialize()` does not set `dailyMintLimit`, leaving it at the Solidity default of `0`. The `limitDailyMint` modifier applied to both `deposit` functions unconditionally reverts with `DailyMintLimitExceeded` for any non-zero deposit when `dailyMintLimit == 0`. The contract is deployed in a fully non-functional state for its primary purpose until an admin separately calls `reinitialize()` or `setDailyMintLimit()`.

## Finding Description
`initialize()` (L207–232) sets roles, `wrsETH`, `feeBps`, `rsETHOracle`, and `isEthDepositEnabled`, but leaves `dailyMintLimit` and `startTimestamp` at their zero defaults.

The `limitDailyMint` modifier (L96–125) executes the following sequence on every deposit call:

1. **`startTimestamp` check (L97):** `block.timestamp < startTimestamp` evaluates to `block.timestamp < 0` in uint256 arithmetic — always `false` — so this check passes silently and does not protect the caller.
2. **Day reset (L113–116):** `getCurrentDay()` returns `block.timestamp / 1 days` (a large number), which is always `> lastMintDay == 0`, so `dailyMintAmount` is reset to `0`.
3. **Limit check (L119–121):** `0 + rsETHAmount > 0` is `true` for any non-zero deposit, causing an unconditional revert with `DailyMintLimitExceeded`.

Both `deposit(string)` (L246) and `deposit(address, uint256, string)` (L271) apply this modifier, so neither ETH nor ERC20 deposits can succeed. The revert occurs before any token transfer, so no funds are lost, but the contract's core function is entirely blocked.

The claim that `reinitialize()` is the *only* path to set `dailyMintLimit` is slightly overstated — `setDailyMintLimit()` (L605) is also available to `DEFAULT_ADMIN_ROLE`. However, both paths require explicit admin action after deployment, and `setDailyMintLimit()` itself rejects zero values (L606–608), confirming that `0` is an explicitly invalid operational state that the contract is nonetheless deployed into.

## Impact Explanation
Any unprivileged user calling either `deposit` function after a fresh deployment (before admin configuration) receives a `DailyMintLimitExceeded` revert. The contract fails to deliver its core promised function — minting wrsETH in exchange for deposited assets. No user funds are lost because the revert precedes any token transfer. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
This affects every fresh deployment of `RSETHPoolV3` before the admin calls `reinitialize()` or `setDailyMintLimit()`. The `reinitializer(2)` pattern signals an upgrade step rather than a mandatory post-deployment configuration, increasing the risk that operators treat it as optional. The failure is triggered by any unprivileged depositor with no special preconditions. The error message `DailyMintLimitExceeded` misleadingly implies a capacity issue rather than an uninitialized state, which may delay diagnosis.

## Recommendation
Pass `_dailyMintLimit` and `_startTimestamp` as parameters to `initialize()` and set them there, with the same non-zero validation currently present in `reinitialize()` and `setDailyMintLimit()`. Alternatively, add an explicit `require(dailyMintLimit > 0, "DailyMintLimitNotSet")` guard at the top of `limitDailyMint` to surface the uninitialized state with a clear error distinct from a genuine limit-exceeded condition.

## Proof of Concept
1. Deploy `RSETHPoolV3` behind a proxy and call `initialize(admin, bridger, wrsETH, feeBps, oracle, true)`.
2. Do **not** call `reinitialize()` or `setDailyMintLimit()`. `dailyMintLimit` remains `0`.
3. As any user, call `deposit{value: 1 ether}("")`.
4. Transaction reverts with `DailyMintLimitExceeded` because `limitDailyMint` evaluates `0 + rsETHAmount > 0 == true`.
5. As admin, call `setDailyMintLimit(1000 ether)` (or `reinitialize(1000 ether, block.timestamp + 1)`).
6. Repeat step 3 — deposit succeeds.
