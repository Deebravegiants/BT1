### Title
Unvalidated `builder` Address in `ProposalInit` Allows Malicious Proposer to Redirect All Block Transaction Fees — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates many `ProposalInit` fields (height, L1 DA mode, L2 gas price, starknet version, L1 gas prices, fee proposal) but never validates the `builder` field. The `builder` address is passed directly as `sequencer_address` into `BlockInfo` and becomes the recipient of every transaction fee in the block. A malicious proposer can set `builder` to an attacker-controlled address; all validators accept the proposal without checking it, and the entire block's fee revenue is redirected to the attacker.

---

### Finding Description

**Root cause — missing identity check on `builder`:**

`is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, `timestamp`, and all four L1 gas-price fields. The `builder` field is never compared against any locally-trusted reference. [1](#0-0) 

`ProposalInitValidation` — the struct that carries every locally-trusted reference value — has no `builder_address` field at all. [2](#0-1) 

**Propagation — `builder` becomes `sequencer_address`:**

`convert_to_sn_api_block_info` maps the proposer-supplied `init.builder` directly to `sequencer_address` in `BlockInfo` with no guard. [3](#0-2) 

**Fee routing — `sequencer_address` is the fee recipient:**

`execute_fee_transfer` sends every transaction's fee to `block_context.block_info.sequencer_address`. [4](#0-3) 

**Analogy to H-04:** In H-04, `collect(address dao)` trusted the caller-supplied `dao` address and called `OwnableUpgradeable(dao).owner()` to route fees — no ownership check. Here, `validate_proposal` trusts the network-supplied `init.builder` and routes all fees to it — no identity check. Both bugs share the same class: an unverified external identity is used to determine the fee recipient.

---

### Impact Explanation

A malicious proposer sets `builder` to an attacker-controlled address. Because all validators see the same `ProposalInit`, they all execute the block with the attacker's address as `sequencer_address`, compute the same `ProposalCommitment`, and reach consensus. The block is committed with the wrong fee recipient. Every transaction fee in that block — potentially thousands of STRK — is transferred to the attacker instead of the legitimate sequencer.

This matches the allowed impact: **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."**

---

### Likelihood Explanation

The attacker must be a consensus participant (a staked validator selected as proposer). This is a "malicious peer reachable without privileged assumptions" scenario per `RESEARCHER.md`. Once in the committee, the attack requires only setting one field in `ProposalInit` — no cryptographic break, no race condition, no special timing. The attack succeeds silently for every block the malicious node proposes. [5](#0-4) 

---

### Recommendation

Add `builder_address: ContractAddress` to `ProposalInitValidation` and check it in `is_proposal_init_valid`:

```rust
if init_proposed.builder != proposal_init_validation.builder_address {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder_address, init_proposed.builder
        ),
    ));
}
```

Populate `builder_address` from `ContextStaticConfig::builder_address` when constructing `ProposalInitValidation` in `validate_current_round_proposal`. [6](#0-5) 

---

### Proof of Concept

1. Attacker stakes and joins the validator committee.
2. When selected as proposer for height `H`, attacker constructs `ProposalInit` with `builder = ATTACKER_ADDRESS` (any address they control).
3. Attacker broadcasts the proposal. All validators call `validate_proposal` → `is_proposal_init_valid`.
4. `is_proposal_init_valid` checks height, DA mode, gas prices, starknet version — but **not** `builder`. Validation passes.
5. `initiate_validation` calls `convert_to_sn_api_block_info(init)` which sets `sequencer_address = ATTACKER_ADDRESS`. [7](#0-6) 
6. The batcher executes all transactions. For each transaction, `execute_fee_transfer` sends the fee to `ATTACKER_ADDRESS`.
7. All validators compute the same `ProposalCommitment` (they all used the same `builder`). Consensus succeeds.
8. The block is committed. All transaction fees for block `H` are in `ATTACKER_ADDRESS`. The legitimate sequencer receives nothing.
