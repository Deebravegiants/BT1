### Title
Missing shard_id binding in `set_state_header` allows a malicious peer to install shard X's state into shard Y's trie via state sync — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a peer-supplied `ShardStateSyncResponseHeader` for shard Y but never checks that the embedded chunk's `shard_id()` equals the requested `shard_id`. The `verify_path` call against `chunk_headers_root` only proves the chunk exists somewhere in the block's chunk Merkle tree — it does not bind the chunk to the correct shard position. A malicious peer can substitute shard X's chunk (with shard X's `prev_state_root`) into the header for shard Y, pass every validation step, and cause the node to write shard X's trie data into shard Y's `ShardUId` column, producing a permanently wrong `ChunkExtra.state_root` for shard Y.

---

### Finding Description

**Entry point**: `StateHeaderValidationRequest` handler in `client_actor.rs` calls `chain.state_sync_adapter.set_state_header(msg.shard_id, msg.sync_hash, msg.header)`. [1](#0-0) 

**Step 1 — `validate_chunk_proofs`** verifies the chunk's internal hash, tx root, and receipts root. It does not inspect `chunk.shard_id()`. [2](#0-1) 

**Step 2 — `verify_path` against `chunk_headers_root`** proves the chunk hash is a leaf in the block's chunk Merkle tree. The tree is built from all shards' chunks ordered by shard index: [3](#0-2) 

`verify_path` only reconstructs the root from the path and compares it — it does **not** verify which leaf index (shard position) the proof corresponds to: [4](#0-3) 

A position-aware variant `verify_path_with_index` exists but is **not used** in `set_state_header`: [5](#0-4) 

An attacker supplies shard X's chunk together with shard X's Merkle proof. Because shard X's chunk genuinely exists in the block, `verify_path(chunk_headers_root, proof_for_shard_X, ChunkHashHeight(shard_X_hash, height))` returns `true`. The check passes even though the syncing node requested shard Y. [6](#0-5) 

**Step 3 — Receipt proof check (step 4e)** hashes receipts as `ReceiptList(shard_id, receipts)` using the *requested* `shard_id` (Y), not the chunk's shard_id. The attacker supplies honest receipt proofs for shard Y (publicly available), so this check passes independently of which chunk is embedded. [7](#0-6) 

**Step 4 — `validate_state_root_node`** checks that the supplied `state_root_node` is consistent with `chunk_inner.prev_state_root()`. The attacker supplies shard X's `state_root_node` and shard X's `prev_state_root`, which are mutually consistent. This check passes. [8](#0-7) 

**Step 5 — Header is persisted** under key `StateHeaderKey(shard_id=Y, sync_hash)` but contains shard X's `prev_state_root`: [9](#0-8) 

**Step 6 — `set_state_part` / `apply_state_part`** reads the header for shard Y, extracts shard X's `prev_state_root`, validates state parts against it, then writes them into shard Y's `ShardUId` trie: [10](#0-9) 

**Step 7 — `set_state_finalize`** derives `shard_uid` from the requested `shard_id` (Y) but calls `apply_chunk` with `chunk_header.prev_state_root()` from shard X's chunk. The result is saved as shard Y's `ChunkExtra`: [11](#0-10) 

There is no `chunk.shard_id() != shard_id` guard anywhere in `set_state_header` or `set_state_finalize`.

---

### Impact Explanation

After the attack completes, the victim node's `DBCol::ChunkExtra` for shard Y contains a `state_root` derived from shard X's state. The node's trie for shard Y holds shard X's key-value data. Every subsequent block the node processes for shard Y will compute a state root that diverges from the honest chain's, causing the node to reject valid blocks or produce invalid ones. The node is permanently desynchronized for shard Y without any indication of tampering.

---

### Likelihood Explanation

Any peer that the syncing node contacts for state data can execute this attack. State sync peers are selected from the p2p network; no validator or operator privilege is required. The attacker only needs to:
1. Be a reachable peer of the syncing node.
2. Respond to the state header request for shard Y with a header containing shard X's chunk and shard X's Merkle proof.
3. Serve shard X's state parts (obtainable from honest nodes).
4. Supply honest receipt proofs for shard Y (also publicly available).

No re-signing is needed because historical chunks are not signature-checked in `set_state_header`.

---

### Recommendation

Add an explicit shard identity check at the top of `set_state_header`, before `validate_chunk_proofs`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the bare `verify_path` call with `verify_path_with_index`, passing the expected shard index derived from the epoch's shard layout, so the Merkle proof is bound to the correct leaf position.

### Proof of Concept

```
1. Node B enters state sync for shard Y (shard_id=1) at sync_hash H.
2. Attacker peer A intercepts the StateRequest::Header(shard_id=1, sync_hash=H).
3. A fetches the honest ShardStateSyncResponseHeader for shard X (shard_id=0) at H.
4. A fetches the honest receipt proofs for shard Y at H.
5. A constructs a malicious ShardStateSyncResponseHeader:
     - chunk = honest chunk for shard X (prev_state_root = root_X)
     - chunk_proof = Merkle proof for shard X's position in chunk_headers_root
     - incoming_receipts_proofs = honest proofs for shard Y
     - state_root_node = honest state_root_node for shard X
6. A sends this header to B as the response for shard Y.
7. B calls set_state_header(shard_id=1, sync_hash=H, header=malicious).
   - validate_chunk_proofs: passes (shard X's chunk is internally valid).
   - verify_path(chunk_headers_root, proof_for_shard_X, ChunkHashHeight(shard_X_hash, h)):
       passes (shard X IS in the tree).
   - Receipt proofs: pass (honest proofs for shard Y).
   - validate_state_root_node: passes (root_X node is consistent with root_X).
   - Header saved under StateHeaderKey(shard_id=1, sync_hash=H) with root_X.
8. B requests and receives shard X's state parts from A (or honest nodes).
9. apply_state_part writes shard X's trie nodes into shard Y's ShardUId column.
10. set_state_finalize applies shard X's chunk to shard Y's shard_uid,
    saving ChunkExtra{state_root: root_X} for shard Y.
11. B's DBCol::ChunkExtra for shard Y now contains root_X ≠ honest root_Y.
    B diverges from the canonical chain for shard Y.
```

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
