Audit Report

## Title
`RSETHPriceFeed.getRoundData` Returns Fabricated Historical Cross-Price by Using Current rsETH Rate Against Historical ETH/USD Price - (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary
`RSETHPriceFeed.getRoundData` fetches the historical ETH/USD price for a requested round from the underlying Chainlink feed but multiplies it by the **current** `RS_ETH_ORACLE.rsETHPrice()` rather than the rsETH/ETH rate that was valid at that round. `LRTOracle` stores only a single overwritten `rsETHPrice` value with no historical series, making it structurally impossible to retrieve the correct rate for any past round. Every call to `getRoundData` for a non-latest round returns a temporally mismatched, fabricated answer.

## Finding Description
In `contracts/oracles/RSETHPriceFeed.sol` lines 53–61:

```solidity
function getRoundData(uint80 _roundId)
    external view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.getRoundData(_roundId);
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;  // always current rate
}
```

`RS_ETH_ORACLE.rsETHPrice()` resolves to `LRTOracle.rsETHPrice`, a single storage slot declared at line 28 of `contracts/LRTOracle.sol`:

```solidity
uint256 public override rsETHPrice;
```

This value is unconditionally overwritten on every `_updateRsETHPrice` call (line 313 of `LRTOracle.sol`):

```solidity
rsETHPrice = newRsETHPrice;
```

No historical ring-buffer or mapping of `(timestamp → rate)` exists anywhere in `LRTOracle`. The root cause is that `getRoundData` correctly parameterizes the ETH/USD leg by round but has no mechanism to parameterize the rsETH/ETH leg by round, so it silently falls back to the global current value. The `updatedAt` field returned is the ETH/USD round timestamp, so any downstream staleness check on `updatedAt` passes as long as the ETH/USD feed is fresh, completely masking the staleness of the rsETH component.

By contrast, `latestRoundData` (lines 63–70) is internally consistent: both legs refer to the present moment.

## Impact Explanation
The contract implements `AggregatorV3Interface` and exposes `getRoundData` as a standard Chainlink method. Any caller invoking `getRoundData` for a historical round receives `current_rsETH_rate × historical_ETH_USD_price` instead of `historical_rsETH_rate × historical_ETH_USD_price`. The function fails to deliver the promised return (a correct historical rsETH/USD price) for every non-latest round query.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value within this repository.** The higher-severity escalation (TWAP-based collateral inflation in downstream lending protocols) depends on external protocol behavior not in scope here and is therefore not credited.

## Likelihood Explanation
No attacker action is required. The incorrect data is returned deterministically on every `getRoundData` call for any historical round by any caller. The feed is deployed as a drop-in Chainlink-compatible oracle, so any integrator calling the standard interface method is immediately affected. The magnitude of the error grows monotonically as rsETH accrues rewards over time.

## Recommendation
Two options:

1. **Revert on `getRoundData`** (minimal safe fix): Since no historical rsETH/ETH rate is stored, revert with a descriptive error (e.g., `HistoricalDataUnavailable()`) to prevent silent consumption of wrong data.
2. **Store a historical rate series in `LRTOracle`** (correct long-term fix): Record `(timestamp → rsETHPrice)` on every `_updateRsETHPrice` call, then binary-search for the rate closest to the queried round's `updatedAt` timestamp inside `getRoundData`.

## Proof of Concept
Call sequence (no privileged access required):

1. Note current `LRTOracle.rsETHPrice` = `R_current` (e.g., 1.05e18 after months of accrual).
2. Pick any historical Chainlink round ID `R1` from months ago where ETH/USD was `P_hist` and the correct rsETH/ETH rate was `R_hist` (e.g., 1.00e18).
3. Call `RSETHPriceFeed.getRoundData(R1)`.
4. Observe returned `answer = R_current × P_hist / 1e18` instead of the correct `R_hist × P_hist / 1e18`.
5. The error is `(R_current - R_hist) × P_hist / 1e18`, which is strictly positive and grows with time elapsed since round R1.
