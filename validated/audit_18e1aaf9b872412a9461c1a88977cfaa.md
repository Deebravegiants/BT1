### Title
Double-Refund via Multiple Subtransactions Claiming the Same Deposit - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`)

### Summary

In the Dijkstra era, `dijkstraTotalRefundsTxCerts` computes deposit refunds by summing the deposit amounts declared directly in `UnRegDepositTxCert` certificates, completely ignoring the `lookupStakingDeposit` lookup function. Because no batch-level guard prevents two subtransactions from each claiming the same credential's deposit, an unprivileged transaction author can craft a top-level transaction whose subtransactions each contain `UnRegDepositTxCert stakingCred D` for the same credential, receiving `2×D` in refunds while only `D` was ever deposited. The developers have already identified this gap and left a `TODO` marker in the disabled test `xit "Multiple subtransactions cannot get the same refund"`, confirming the predicate failure is not yet implemented.

### Finding Description

**Root cause — `dijkstraTotalRefundsTxCerts`** [1](#0-0) 

```haskell
-- Unlike previous eras, we no longer need to lookup refunds from the ledger state,
-- since all of the certificates specify the actual refund and ledger rules will
-- validate that they are accurate.
dijkstraTotalRefundsTxCerts ::
  ( Foldable f, ConwayEraTxCert era ) =>
  f (TxCert era) -> Coin
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert  _ deposit -> deposit
  _ -> zero
```

The comment claims "ledger rules will validate that they are accurate," but the batch-level validation for subtransactions is absent. The `getTotalRefundsTxCerts` instance for Dijkstra discards both lookup functions entirely: [2](#0-1) 

```haskell
getTotalRefundsTxCerts _ _ _ = dijkstraTotalRefundsTxCerts
```

**Contrast with Conway/Shelley** — in those eras `shelleyTotalRefundsTxCerts` uses `lookupDeposit cred` to retrieve the actual stored deposit and only credits a refund when the credential is found in the ledger state: [3](#0-2) 

**Missing batch-level guard** — `validateBatchWithdrawals` aggregates withdrawals across all subtransactions and checks them against account balances: [4](#0-3) 

No equivalent aggregation exists for deposit refunds across subtransactions.

**Developer acknowledgement** — the disabled test explicitly marks the missing check: [5](#0-4) 

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

`xit` skips the test; `error "TODO: predicate failure not yet implemented"` confirms no rejection rule exists yet, meaning the transaction currently succeeds.

### Impact Explanation

An attacker who registers a staking credential (paying deposit `D`) can submit a single top-level Dijkstra transaction containing two subtransactions, each bearing `UnRegDepositTxCert stakingCred D`. Each subtransaction's value-conservation check credits `D` as consumed (refund), so both balance independently. The deposit pot, however, only holds `D`. The net effect is that `D` lovelace is created from nothing — a direct violation of ADA preservation. This maps to the **Critical** impact class: *Direct creation of ADA through an invalid ledger state transition*.

### Likelihood Explanation

The attack requires only:
1. Registering a staking credential (permissionless, costs only the deposit itself).
2. Constructing a valid Dijkstra top-level transaction with two subtransactions sharing the same `UnRegDepositTxCert`.

No privileged access, governance majority, or external dependency is needed. The entry point is a serialized transaction submitted by any unprivileged user. The Dijkstra era is present in the production repository and is targeted for a future hard fork; once activated, this is immediately exploitable.

### Recommendation

1. Implement a batch-level refund aggregation check analogous to `validateBatchWithdrawals`: collect all `UnRegDepositTxCert` / `UnRegDRepTxCert` certificates across the top-level transaction and all subtransactions, deduplicate by credential, and verify that no credential's deposit is claimed more than once.
2. Restore the `lookupStakingDeposit` / `lookupDRepDeposit` validation inside `dijkstraTotalRefundsTxCerts` (or its call site) so that the refund amount is cross-checked against the actual stored deposit, mirroring the Conway/Shelley approach in `shelleyTotalRefundsTxCerts`.
3. Enable and fix the `xit "Multiple subtransactions cannot get the same refund"` test once the predicate failure is implemented.

### Proof of Concept

```
1. Alice registers stakingCred, paying keyDeposit D.
   Ledger state: depositPot += D, stakingCred registered.

2. Alice constructs:
     subTx1 = mkBasicTx mkBasicTxBody
               & bodyTxL . certsTxBodyL .~ [UnRegDepositTxCert stakingCred D]
     subTx2 = mkBasicTx mkBasicTxBody
               & bodyTxL . certsTxBodyL .~ [UnRegDepositTxCert stakingCred D]
     tx     = mkBasicTx mkBasicTxBody
               & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]

3. Alice submits tx.
   - subTx1 value conservation: consumed = inputs1 + D, produced = outputs1 + fees1  ✓
   - subTx2 value conservation: consumed = inputs2 + D, produced = outputs2 + fees2  ✓
   - No batch-level refund deduplication check exists (TODO marker in test).

4. Result: Alice's outputs include 2×D in refunds; depositPot decreases by 2×D.
   Net ADA created: D lovelace.
```

This directly mirrors the external report's pattern: a user-supplied value (the deposit amount embedded in the certificate) is processed without the protocol-level guard that should limit it to the actual stored deposit, allowing the attacker to extract more value than was ever locked.

shelleyTotalRefundsTxCerts pp lookupDeposit = snd . F.foldl' accum (mempty, Coin 0)
  where
    keyDeposit = pp ^. ppKeyDepositL
    accum (!regCreds, !totalRefunds) cert =
      case lookupRegStakeTxCert cert of
        Just k ->
          -- Need to track new delegations in case that the same key is later deregistered in
          -- the same transaction.
          (Set.insert k regCreds, totalRefunds)
        Nothing ->
          case lookupUnRegStakeTxCert cert of
            Just cred
              -- We first check if there was already a registration certificate in this
              -- transaction.
              | Set.member cred regCreds -> (Set.delete cred regCreds, totalRefunds <+> keyDeposit)
              -- Check for the deposit left during registration in some previous
              -- transaction. This de-registration check will be matched first, despite being
              -- the last case to match, because registration is not possible without
              -- de-registration.
              | Just deposit <- lookupDeposit cred -> (regCreds, totalRefunds <+> deposit)
            _ -> (regCreds, totalRefunds)

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL
