Audit Report

## Title
Missing Chainlink Staleness Checks Allow Stale Price to Corrupt rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `roundId`, `updatedAt`, and `answeredInRound`, performing zero staleness validation. A stale or incomplete Chainlink round feeds a corrupted asset price into the rsETH exchange rate calculation via `LRTOracle._updateRsETHPrice()`, enabling over-minting of rsETH and dilution of existing holders. The same codebase already implements all three required checks in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the omission is unintentional.

## Finding Description

**Root cause:** In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` reads only the `price` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

No check is made for `answeredInRound >= roundId` (stale carry-over), `updatedAt != 0` (incomplete round), or `price > 0` (invalid/negative price).

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` correctly validates all three:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

**Exploit path:**

1. `LRTOracle.updateRSETHPrice()` is `public whenNotPaused` — no access control beyond the pause flag, callable by any external account. [3](#0-2) 

2. It calls `_updateRsETHPrice()` → `_getTotalEthInProtocol()`, which iterates over all supported assets and calls `getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice(asset)` for each LST. [4](#0-3) 

3. A stale carry-over price (`answeredInRound < roundId`) that is lower than the true market price causes `totalETHInProtocol` to be undercounted. The resulting `newRsETHPrice` is artificially depressed and written to storage. [5](#0-4) 

4. `LRTDepositPool.getRsETHAmountToMint()` computes `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`. With a depressed stored `rsETHPrice`, the attacker (or any subsequent depositor) receives more rsETH than their deposit is worth at the true rate. [6](#0-5) 

**Why existing guards are insufficient:**

The `pricePercentageLimit` downside protection at lines 270–282 of `LRTOracle.sol` only triggers a pause when the price drop *exceeds* the configured threshold. If `pricePercentageLimit` is zero (unset) or the stale price is within the threshold, the corrupted price is accepted and stored without any revert or pause. [7](#0-6) 

## Impact Explanation

**Theft of unclaimed yield (High).** A stale low price artificially depresses `rsETHPrice`. Any depositor who calls `depositAsset()` immediately after the corrupted `updateRSETHPrice()` receives more rsETH than their deposit is worth at the true rate. This permanently dilutes all pre-existing rsETH holders' share of the underlying TVL — a direct theft of accrued yield from current holders. The impact is bounded by the magnitude of the stale price deviation and the deposit size, but is repeatable on every stale round.

## Likelihood Explanation

Chainlink feeds experience stale carry-overs during network congestion, sequencer downtime (L2s), or feed maintenance. The trigger function `updateRSETHPrice()` is public and permissionless — any external actor can call it at the exact moment a stale round is active. No privileged access, victim mistake, or governance action is required. The attacker can monitor the mempool for stale round conditions and atomically call `updateRSETHPrice()` followed by `depositAsset()` in the same block.

## Recommendation

Apply the same three staleness guards already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt != 0, "Incomplete round");
    require(price > 0, "Invalid price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept

**Foundry fork test outline:**

1. Fork mainnet at a block where a supported LST/ETH Chainlink feed has `answeredInRound < roundId` (or mock a `MockAggregatorV3` that returns a stale answer).
2. Deploy/use the existing `ChainlinkPriceOracle` pointing to the stale feed.
3. Record `rsETHPrice_before = lrtOracle.rsETHPrice()`.
4. Call `lrtOracle.updateRSETHPrice()` from an unprivileged EOA — confirm it succeeds (no revert).
5. Assert `rsETHPrice_after < rsETHPrice_before` (price was corrupted downward).
6. Call `lrtDepositPool.depositAsset(asset, amount, 0, "")` from the attacker EOA.
7. Assert `rsethReceived > expectedRsethAtTruePrice` — attacker received excess rsETH.
8. Assert existing holder's redemption value decreased — dilution confirmed.

The mock aggregator approach requires no live stale round: set `answeredInRound = roundId - 1` and `answer = truePrice * 95 / 100` to simulate a 5% stale deviation below `pricePercentageLimit`.
