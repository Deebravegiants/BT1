### Title
`init_transfer_resume` Bypasses Pause State, Freezing User Tokens During Bridge Pause — (`near/omni-bridge/src/lib.rs`)

### Summary

The NEAR bridge contract's `init_transfer_resume` callback does not check the pause state. When the bridge is paused (e.g., due to a security incident), any unprivileged caller can invoke `storage_deposit` for a pending virtual account, which triggers `init_transfer_resume` and commits the transfer — burning or locking tokens on NEAR — while `fin_transfer` on the destination chain remains blocked by the pause. This creates an irreversible frozen redemption path for the user's tokens.

### Finding Description

The NEAR bridge supports a yield/resume mechanism for `init_transfer`. When a user calls `ft_transfer_call` but lacks sufficient storage balance, `init_transfer` suspends execution via `env::promise_yield_create("init_transfer_resume", ...)` and stores the yield ID in `init_transfer_promises` keyed by a virtual account ID. [1](#0-0) 

The public `storage_deposit` function always calls `resume_promise` after updating the balance: [2](#0-1) 

`resume_promise` calls `env::promise_yield_resume`, which fires `init_transfer_resume` as a callback: [3](#0-2) 

The critical gap: `init_transfer_resume` is `#[private]` but contains **no pause check**. It unconditionally calls `init_transfer_internal`, which burns/locks tokens and emits `InitTransferEvent`: [4](#0-3) [5](#0-4) 

By contrast, `ft_on_transfer` — the normal entry point for new transfers — is correctly gated: [6](#0-5) 

And `fin_transfer` on the NEAR side is also gated: [7](#0-6) 

The destination-chain `fin_transfer` equivalents are similarly paused (EVM `PAUSED_FIN_TRANSFER`, Solana `FINALIZE_TRANSFER_PAUSED`, Starknet `PAUSE_FIN_TRANSFER`): [8](#0-7) [9](#0-8) [10](#0-9) 

### Impact Explanation

When the bridge is paused (e.g., `pauseAll()` triggered by `PAUSABLE_ADMIN_ROLE`):

1. `ft_on_transfer` is blocked — no new transfers can start.
2. But any pending yielded transfer can still be committed by anyone calling `storage_deposit` for the corresponding virtual account.
3. `init_transfer_resume` fires without a pause check, calling `init_transfer_internal`.
4. For deployed (wrapped) tokens, `burn_tokens_if_needed` is called — **the burn is irreversible**.
5. `InitTransferEvent` is emitted, signaling the NEAR side has committed the transfer.
6. `fin_transfer` on the destination chain is paused — the user cannot receive tokens.
7. The user's tokens are burned on NEAR and unclaimable on the destination chain.

If the bridge is paused indefinitely (e.g., during a critical security incident or deprecation), the tokens are permanently lost. This is a **Critical — irreversible fund lock / frozen redemption path**.

### Likelihood Explanation

- Requires the bridge to be paused (a `PauseManager` or `DAO` action — a normal operational event during incidents).
- Requires at least one pending yielded transfer (common: any user who initiated a transfer without sufficient storage balance).
- The trigger (`storage_deposit`) is a fully public, unprivileged call — any account can deposit storage for any virtual account ID, forcing the resume.
- The virtual account ID is deterministic and publicly computable from the transfer parameters. [11](#0-10) 

### Recommendation

Add a pause check at the top of `init_transfer_resume`. If the bridge is paused, return the full `transfer_message.amount` as a refund (returning a non-zero value from `ft_on_transfer` causes the NEP-141 token contract to refund the sender):

```rust
pub fn init_transfer_resume(
    &mut self,
    transfer_message: TransferMessage,
    message_storage_account_id: AccountId,
    storage_owner: AccountId,
    #[callback_result] response: Result<(), PromiseError>,
) -> U128 {
    self.remove_promise(&message_storage_account_id);

    // NEW: respect the pause state — refund if paused
    if self.pa_is_paused() {
        env::log_str("Init transfer resume: bridge is paused, refunding");
        return transfer_message.amount;
    }

    // ... existing logic ...
}
```

This mirrors the behavior of `ft_on_transfer` and ensures the pause mechanism is complete across all transfer entry paths.

### Proof of Concept

1. User calls `ft_transfer_call(receiver_id = bridge, amount = X, msg = InitTransfer{...})` while bridge is unpaused.
2. `ft_on_transfer` passes the pause check → `init_transfer` → insufficient storage → yield path taken; `init_transfer_promises[virtual_acct] = yield_id` stored.
3. `PauseManager` calls `pauseAll()` — bridge is now fully paused.
4. Attacker (any account) calls `storage_deposit(account_id = virtual_acct, deposit = required_balance)`.
5. `storage_deposit` → `resume_promise(virtual_acct)` → `env::promise_yield_resume(yield_id)` → `init_transfer_resume` fires as a callback.
6. `init_transfer_resume` has no pause check → `init_transfer_internal` → tokens burned/locked, `InitTransferEvent` emitted.
7. Relayer attempts `fin_transfer` on EVM/Solana/Starknet → reverts with `"Pausable: paused"` / `ErrorCode::Paused`.
8. User's tokens are burned on NEAR and permanently unclaimable on the destination chain while the bridge remains paused.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
