### Title
1:1 Token Swap Without Value Equivalence Check Enables Fund Theft - (`contracts/L2/RsETHTokenWrapper.sol`)

---

### Summary

`RsETHTokenWrapper` allows multiple "allowed" rsETH-variant tokens to be deposited and withdrawn at a strict 1:1 ratio against wrsETH. Because any allowed token can be deposited to mint wrsETH and any other allowed token can be withdrawn by burning the same wrsETH amount, a user who deposits a lower-valued rsETH variant can withdraw a higher-valued rsETH variant, stealing the difference from the wrapper's reserves.

---

### Finding Description

`RsETHTokenWrapper` is designed to wrap alternative rsETH tokens (bridged rsETH variants from different L2 bridges) into a canonical `wrsETH`. The contract supports multiple allowed tokens via `_addAllowedToken`, which is called both at initialization and via `reinitialize` (admin-gated) and `addAllowedToken` (TIMELOCK_ROLE-gated). [1](#0-0) 

The public `deposit` / `depositTo` and `withdraw` / `withdrawTo` functions are unrestricted — any caller can invoke them: [2](#0-1) 

Both operations are strictly 1:1 in token amount:

```solidity
// _deposit: transfers _amount of _asset in, mints _amount of wrsETH
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);
    ...
}

// _withdraw: burns _amount of wrsETH, transfers _amount of _asset out
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    ...
}
``` [3](#0-2) 

There is **no check** that the token being deposited and the token being withdrawn have the same market value. The only invariant enforced is that both tokens are in `allowedTokens`. If two allowed tokens trade at different prices (e.g., `rsETH_bridgeA` at $2,000 and `rsETH_bridgeB` at $2,050), an attacker can deposit the cheaper token and withdraw the more expensive one, extracting the price difference from the wrapper's reserves.

The `reinitialize` function explicitly adds a second allowed token: [4](#0-3) 

---

### Impact Explanation

**Critical — Direct theft of user funds.**

The wrapper holds rsETH-variant tokens deposited by users and bridgers. An attacker can drain the more valuable token from the wrapper by cycling through deposit (cheap token) → withdraw (expensive token). The remaining wrsETH supply is backed only by the cheaper token, causing a permanent loss for all wrsETH holders who attempt to redeem the more valuable variant.

---

### Likelihood Explanation

**Medium.** The protocol explicitly supports multiple allowed rsETH variants (canonical rsETH via LayerZero OFT, alternative bridge rsETH, etc.). Different bridge implementations carry different trust assumptions and liquidity profiles, meaning their secondary-market prices can and do diverge — even temporarily. Any price discrepancy between two allowed tokens, however small, is immediately exploitable by any unprivileged user with no capital at risk beyond gas.

---

### Recommendation

Enforce that all allowed tokens are treated as fungible only if they are provably equivalent in value. Options:

1. **Restrict to a single canonical token**: Remove the multi-token deposit/withdraw path and only allow one rsETH variant per wrapper instance.
2. **Oracle-gated swap**: Before executing a cross-token swap (deposit token A, withdraw token B), verify via an on-chain oracle that both tokens have the same price within an acceptable tolerance.
3. **Separate accounting per token**: Track wrsETH minted per deposited token type and only allow withdrawal of the same token type that was deposited (similar to how the original PoolTogether fix enforced `depositToken` equality).

---

### Proof of Concept

Assumptions:
- `allowedTokens[rsETH_A] = true` (worth $2,000 per token)
- `allowedTokens[rsETH_B] = true` (worth $2,050 per token)
- Wrapper holds 100 `rsETH_B` tokens (deposited by bridger)

Attack:
1. Attacker acquires 100 `rsETH_A` (cost: $200,000).
2. Attacker calls `deposit(rsETH_A, attacker, 100e18)` → wrapper receives 100 `rsETH_A`, mints 100 `wrsETH` to attacker.
3. Attacker calls `withdraw(rsETH_B, attacker, 100e18)` → wrapper burns 100 `wrsETH`, transfers 100 `rsETH_B` to attacker.
4. Attacker holds 100 `rsETH_B` worth $205,000, having spent $200,000 on `rsETH_A`.
5. **Profit: $5,000** (2.5%) extracted from the wrapper's reserves in a single atomic sequence.

The wrapper now holds 100 `rsETH_A` instead of 100 `rsETH_B`, and all remaining wrsETH holders are backed by the cheaper asset. [5](#0-4) [6](#0-5)
