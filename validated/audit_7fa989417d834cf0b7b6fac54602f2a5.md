Audit Report

## Title
Stale Cached `rsETHPrice` Allows Depositors to Mint Excess rsETH, Stealing Accrued Yield from Existing Holders - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getRsETHAmountToMint` divides by the stored `LRTOracle.rsETHPrice` state variable rather than computing the live exchange rate. Because `rsETHPrice` is only updated on explicit calls to `updateRSETHPrice()`, any depositor who deposits during the window between reward accrual and the next price update receives more rsETH than their deposit is worth at the true exchange rate, diluting existing holders' accrued yield.

## Finding Description
`getRsETHAmountToMint` at `contracts/LRTDepositPool.sol:520` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` resolves to the storage variable declared at `contracts/LRTOracle.sol:28`:

```solidity
uint256 public override rsETHPrice;
```

This variable is only written inside `_updateRsETHPrice()` at `contracts/LRTOracle.sol:313`:

```solidity
rsETHPrice = newRsETHPrice;
```

which is reached only when `updateRSETHPrice()` (line 87) or `updateRSETHPriceAsManager()` (line 94) is called. Neither is invoked atomically within `depositETH` or `depositAsset`.

When EigenLayer restaking rewards or LST appreciation cause `totalETHInProtocol` to grow, the true price `totalETHInProtocol / rsethSupply` exceeds the stored `rsETHPrice`. Any deposit made before the next `updateRSETHPrice()` call uses the stale (lower) denominator, minting more rsETH than the deposit is worth at the true rate. The excess rsETH represents yield that was already accrued by existing holders but is now captured by the depositor.

An additional aggravating factor: when the true price increase exceeds `pricePercentageLimit`, `updateRSETHPrice()` reverts for non-manager callers with `PriceAboveDailyThreshold` (`contracts/LRTOracle.sol:263-264`). This means after a large reward event, only a manager can close the staleness window, extending the exploitable period indefinitely for any depositor.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders continuously accrue yield reflected in the true `rsETHPrice`. A depositor minting at the stale lower price captures a fraction of that accrued yield proportional to `(truePrice - stalePrice) / truePrice × depositAmount`. This is a direct, concrete transfer of value from existing holders to the depositor, matching the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation
**Medium.** The exploit requires no special privileges — any caller of `depositETH` or `depositAsset` can trigger it. Rewards accrue continuously from EigenLayer strategies and LST rebasing, so a staleness window exists after every reward event. The `pricePercentageLimit` guard can extend this window to be manager-gated, making the attack window larger and more predictable. The attack is repeatable across every reward cycle.

## Recommendation
Replace the stored `rsETHPrice` lookup in `getRsETHAmountToMint` with a live computation of `totalETHInProtocol / rsethSupply` (matching the logic in `_updateRsETHPrice`), so the mint ratio always reflects the current state. Alternatively, call `updateRSETHPrice()` atomically at the start of `_beforeDeposit` before computing `getRsETHAmountToMint`, ensuring the stored price is always fresh at the time of minting.

## Proof of Concept
1. Protocol state: 1000 ETH TVL, 1000 rsETH supply → stored `rsETHPrice = 1.0e18`.
2. EigenLayer rewards of 10 ETH accrue → true price is `1010/1000 = 1.01e18`, but `rsETHPrice` remains `1.0e18`.
3. Attacker calls `depositETH{value: 100 ether}(0, "")`.
4. `_beforeDeposit` calls `getRsETHAmountToMint(ETH, 100e18)` → `100e18 * 1e18 / 1.0e18 = 100e18` rsETH minted.
5. Fair amount at true price: `100e18 * 1e18 / 1.01e18 ≈ 99.01e18` rsETH.
6. Attacker received `~0.99e18` excess rsETH worth `~1 ETH` of value stolen from existing holders.
7. When `updateRSETHPrice()` is subsequently called: TVL = 1110 ETH, supply = 1100 rsETH → new price = `1110/1100 ≈ 1.009e18` instead of `1.01e18` that existing holders would have had. The difference is permanently captured by the attacker.

**Foundry test plan:** Deploy protocol with mock EigenLayer strategy returning 1010 ETH for 1000 rsETH supply. Assert `rsETHPrice == 1.0e18` (not yet updated). Call `depositETH` with 100 ETH. Assert minted rsETH > `100e18 * 1e18 / 1.01e18`. Call `updateRSETHPrice()`. Assert new price < `1.01e18` and that attacker's rsETH balance × new price > 100 ETH (profit confirmed).
