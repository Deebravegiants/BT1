Audit Report

## Title
Lack of Slippage Protection in L2 Pool `deposit()` Functions Allows Users to Receive Fewer rsETH Tokens Than Expected - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPoolNoWrapper.sol)

## Summary
All L2 pool `deposit()` functions compute the rsETH/wrsETH output amount from the live oracle rate at execution time but accept no `minRSETHAmountExpected` parameter. The mainnet `LRTDepositPool` enforces such a minimum in `_beforeDeposit()`. Because the rsETH oracle rate monotonically increases with yield accrual, any oracle update that lands before a user's deposit transaction silently reduces the minted token quantity below the amount returned by `viewSwapRsETHAmountAndFee`, with no on-chain revert path for the user.

## Finding Description
`LRTDepositPool.depositETH()` and `depositAsset()` both accept `minRSETHAmountExpected` and enforce it:

```solidity
// contracts/LRTDepositPool.sol L667-668
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

Every L2 pool omits this guard. In `RSETHPoolV3.deposit(string)`:

```solidity
// contracts/pools/RSETHPoolV3.sol L258-262
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
```

`viewSwapRsETHAmountAndFee` derives output purely from the live oracle:

```solidity
// contracts/pools/RSETHPoolV3.sol L299-307
uint256 rsETHToETHrate = getRate();   // live oracle read
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The identical pattern is confirmed in:
- `RSETHPoolV3.deposit(address,uint256,string)` — lines 271–293
- `RSETHPoolV3ExternalBridge.deposit(string)` — lines 366–384
- `RSETHPoolV3ExternalBridge.deposit(address,uint256,string)` — lines 390–412
- `RSETHPoolV3WithNativeChainBridge.deposit(string)` — lines 282–301
- `RSETHPoolV3WithNativeChainBridge.deposit(address,uint256,string)` — lines 307–329
- `RSETHPoolNoWrapper.deposit(string)` — lines 231–244
- `RSETHPoolNoWrapper.deposit(address,uint256,string)` — lines 250–271

Existing guards (`nonReentrant`, `whenNotPaused`, `limitDailyMint`, `onlySupportedToken`, zero-amount check) are all orthogonal to the slippage issue and provide no protection against receiving fewer tokens than quoted.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

When the oracle rate rises between the time a user calls `viewSwapRsETHAmountAndFee` for a quote and the time their deposit transaction is included, the user receives strictly fewer wrsETH/rsETH tokens than the quote indicated. Because each token is worth proportionally more ETH, the deposited ETH value is preserved in aggregate. However, the contract does not honour the token quantity the user was shown. Users who rely on the quoted token amount for downstream operations — providing liquidity, collateralising a loan, meeting a minimum balance threshold — may find themselves short of the expected balance with no on-chain recourse.

## Likelihood Explanation
The rsETH oracle rate is updated on a regular cadence (at minimum daily, often more frequently) as EigenLayer staking rewards accrue. No attacker action is required; normal oracle maintenance is sufficient to trigger the discrepancy. Any user who previews a quote via `viewSwapRsETHAmountAndFee` and then submits a deposit is exposed. L2 chains with variable block times or sequencer congestion widen the window between submission and inclusion, increasing the probability of an intervening oracle update. The condition is repeatable on every deposit.

## Recommendation
Add a `minRSETHAmountExpected` parameter to every public `deposit()` overload in all L2 pool contracts, mirroring `LRTDepositPool._beforeDeposit()`:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same change to the token-deposit overloads in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`.

## Proof of Concept
1. User calls `RSETHPoolV3.viewSwapRsETHAmountAndFee(1 ether)` at oracle rate `R` and receives quote `X` wrsETH.
2. User submits `RSETHPoolV3.deposit{value: 1 ether}("ref")`.
3. Before inclusion, the rsETH oracle is updated to `R' > R` (normal reward accrual — no attacker required).
4. Deposit executes: `rsETHAmount = (1 ether - fee) * 1e18 / R'` which is strictly less than `X`.
5. User receives fewer wrsETH tokens than quoted; no revert occurs.

**Contrast:** `LRTDepositPool.depositETH{value: 1 ether}(X, "ref")` with the same quoted minimum would revert at step 4 via `MinimumAmountToReceiveNotMet`.

**Foundry fork test plan:**
```solidity
function testSlippageNoProtection() public {
    uint256 depositAmt = 1 ether;
    (uint256 quotedRsETH,) = pool.viewSwapRsETHAmountAndFee(depositAmt);

    // Simulate oracle rate increase (e.g., vm.mockCall or advance time to trigger update)
    _advanceOracleRate();

    uint256 balBefore = wrsETH.balanceOf(user);
    vm.prank(user);
    pool.deposit{value: depositAmt}("ref");
    uint256 received = wrsETH.balanceOf(user) - balBefore;

    assertLt(received, quotedRsETH, "received less than quoted with no revert");
}
```
