### Title
Sub-Transaction Execution Units Not Validated Against `maxTxExUnits` Protocol Parameter — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, the `dijkstraUtxoTransition` rule enforces the `maxTxExUnits` protocol parameter only against the top-level transaction's redeemers. Sub-transactions embedded in the top-level transaction body carry their own independent redeemer maps with their own declared `ExUnits` budgets, and these are never aggregated or checked against `maxTxExUnits`. An unprivileged transaction sender can therefore submit a batch whose sub-transactions collectively declare execution-unit budgets that far exceed the protocol-parameter limit, causing honest nodes to execute Plutus scripts beyond the intended per-transaction resource cap.

---

### Finding Description

**Vulnerability class (from the external report):** Insufficient validation of attacker-controlled size/count fields before use — the receiving component trusts a numeric field from an untrusted producer without bounding it, leading to resource exhaustion or invalid state.

**Root cause in this codebase:**

`dijkstraUtxoTransition` calls:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [1](#0-0) 

`validateExUnitsTooBigUTxO` delegates to `totExUnits tx`:

```haskell
validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) ...
  where
    totalExUnits = totExUnits tx
``` [2](#0-1) 

`totExUnits` is defined as:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
``` [3](#0-2) 

This folds only over the **top-level transaction's** witness redeemers. Sub-transactions are stored in the top-level body as `dtbrSubTransactions :: !(OMap TxId (Tx SubTx era))`: [4](#0-3) 

Each sub-transaction carries its own `dstWits` field (a full `TxWits era` including its own `rdmrsTxWitsL`): [5](#0-4) 

The `dijkstraUtxoTransition` rule never aggregates sub-transaction redeemer `ExUnits` into the check. The only batch-aware check in the rule is `validateBatchCollateral` (which correctly inspects sub-transactions for the presence of redeemers to decide whether collateral is required), but no analogous aggregation is performed for the `ExUnits` limit: [6](#0-5) 

The block-level ExUnits check in `alonzoBbodyTransition` also uses `foldMap totExUnits txs`, which suffers from the same omission — it sums only top-level redeemers across all transactions in the block: [7](#0-6) 

---

### Impact Explanation

An attacker can submit a top-level Dijkstra transaction whose own redeemer map is empty (so `totExUnits tx = ExUnits 0 0`, trivially satisfying `≤ maxTxExUnits`), while embedding sub-transactions whose redeemer maps declare `ExUnits` budgets that are arbitrarily large. Because the ledger executes Plutus scripts up to their declared budget, nodes will spend computation resources on sub-transaction script evaluation that is uncapped by the protocol parameter. This allows attacker-controlled transactions to exceed the intended per-transaction and per-block validation resource limits, matching the **Medium** impact category: *attacker-controlled transactions exceed intended validation limits*.

---

### Likelihood Explanation

The attack requires only the ability to submit a valid Dijkstra-era transaction — an unprivileged transaction sender. No privileged access, key compromise, or governance majority is needed. The Dijkstra era is the current development era, so this affects the production code path once deployed. The construction is straightforward: craft a top-level transaction with no top-level redeemers and one or more sub-transactions each carrying a Plutus script with a large `ExUnits` budget.

---

### Recommendation

In `dijkstraUtxoTransition`, replace the single-level `validateExUnitsTooBigUTxO pp tx` call with a batch-aware variant that aggregates `ExUnits` across the top-level transaction and all sub-transactions before comparing against `maxTxExUnits`. Concretely:

```haskell
let batchExUnits =
      totExUnits tx
        <> foldMap totExUnits (OMap.elems $ txBody ^. subTransactionsTxBodyL)
failureUnless (pointWiseExUnits (<=) batchExUnits (pp ^. ppMaxTxExUnitsL)) $
  ExUnitsTooBigUTxO Mismatch { ... }
```

Apply the same fix to the block-level check in the Dijkstra `BBODY` rule so that `foldMap totExUnits txs` also accounts for sub-transaction redeemers.

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction with:
   - No redeemers in `dtWits` (so `totExUnits tx = ExUnits 0 0`)
   - One sub-transaction whose `dstWits` contains a redeemer with `ExUnits { exUnitsMem = maxBound, exUnitsSteps = maxBound }`
2. Submit the transaction. The check at line 415 passes because `ExUnits 0 0 ≤ maxTxExUnits`.
3. The sub-transaction's Plutus script is executed with an unchecked budget, consuming node resources beyond the protocol-parameter limit.

The attacker entry path is a standard transaction submission — no special privileges required. The necessary code path is `dijkstraUtxoTransition` → `Alonzo.validateExUnitsTooBigUTxO pp tx` → `totExUnits tx` (top-level only), with sub-transaction redeemers never reaching the check. [8](#0-7)
