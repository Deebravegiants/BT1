### Title
Receipt Size Limit Bypass via `output_data_receivers` Appended After Validation — (File: `runtime/runtime/src/lib.rs`)

### Summary

The nearcore runtime checks a new receipt's serialized size against `max_receipt_size` (4 MB) at creation time, but then unconditionally appends `output_data_receivers` to that same receipt object **after** the check. An unprivileged user can craft a contract that sizes a receipt exactly at the limit, then uses `promise_return` to trigger the post-validation append, producing an oversized receipt that is accepted into the chain and forwarded cross-shard. The codebase explicitly acknowledges this as an open bug (issue #12606) and has added workarounds to prevent a halt, but the root cause is unfixed.

### Finding Description

**Broken invariant:** Every new receipt emitted by the runtime must satisfy `borsh_size(receipt) ≤ max_receipt_size` (4 MB). This invariant is enforced in `validate_receipt` only when `mode == ValidateReceiptMode::NewReceipt`.

**Root cause — two-step mutation after validation:**

Step 1 — size check in `validate_receipt`: [1](#0-0) 

Step 2 — `output_data_receivers` appended to the already-validated receipt in `apply_action_receipt`, **after** the check: [2](#0-1) 

The `output_data_receivers` field is populated from the parent receipt's `output_data_receivers()` when the child receipt is returned via `promise_return`. Each `DataReceiver` entry adds ~70 bytes (a `CryptoHash` + an `AccountId`). A contract that sizes receipt C to exactly `max_receipt_size` before the append will produce a receipt that exceeds the limit after the append.

**Acknowledged in the codebase:**

The `ValidateReceiptMode::ExistingReceipt` variant was introduced specifically because of this bug: [3](#0-2) 

The congestion-control forwarding path also clamps oversized receipts to avoid them getting stuck: [4](#0-3) 

The test suite explicitly verifies that oversized receipts reach the chain: [5](#0-4) 

### Impact Explanation

An unprivileged user can force receipts larger than `max_receipt_size` (4 MB) into the chain. The concrete corrupted protocol value is the receipt itself — its serialized size violates the hard limit that the state-witness size budget (`main_storage_proof_size_soft_limit`, `per_receipt_storage_proof_size_limit`, and the 17 MiB total witness target) is designed around. [6](#0-5) 

Because `ExistingReceipt` mode skips the size check uniformly on all nodes, there is no consensus divergence — all nodes accept the oversized receipt. However, the `max_receipt_size` invariant is broken, the state witness can exceed its intended budget, and the `per_receipt_storage_proof_size_limit` enforcement becomes unreliable for receipts that are already over the size limit.

### Likelihood Explanation

Any account can deploy a contract and call it with a single transaction. The `max_receipt_size_promise_return_method1` pattern in the test suite demonstrates the exact trigger. No validator, operator, or privileged role is required. The only prerequisite is a funded account and a deployed contract.

### Recommendation

Re-validate the receipt size **after** `output_data_receivers` are appended in `apply_action_receipt`, or enforce a size budget on `output_data_receivers` before appending. The fix should ensure `borsh_size(receipt) ≤ max_receipt_size` holds at the point the receipt is handed to the receipt sink, not only at the point it is first constructed.

### Proof of Concept

1. Deploy a contract containing a method that creates promise chain `A -then-> B`.
2. When `A` executes, it calls `promise_create` with `args` sized so that the resulting receipt C has `borsh_size(C) == max_receipt_size` (4,194,304 bytes).
3. `A` calls `promise_return(C)`.
4. The runtime appends `B`'s `output_data_receivers` to `C` at: [7](#0-6) 
5. `C`'s serialized size now exceeds 4,194,304 bytes.
6. `validate_receipt` is not re-run after the append; the receipt is forwarded cross-shard.
7. Receiving shards process it under `ValidateReceiptMode::ExistingReceipt`, which skips the size check: [8](#0-7) 

The test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` reproduces this end-to-end and asserts that an oversized receipt is present in the chain: [9](#0-8)

        receiver_id: account.clone(),

        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();

    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,

**File:** docs/misc/state_witness_size_limits.md (L23-28)
```markdown
* `per_receipt_storage_proof_size_limit - 4 MB`
  * Executing a receipt generates storage proof. A single receipt is allowed to generate at most 4MB of storage proof. This is a hard limit, receipts which generate more than that will fail.
* `main_storage_proof_size_soft_limit - 4 MB`
  * This is a limit on the total size of storage proof generated by receipts in one chunk. Once receipts generate more storage proof than this limit, the chunk producer stops processing receipts and moves the rest to the delayed queue.
  * It's a soft limit, which means that the total size of storage proof could reach 8 MB (3.99MB + one receipt which generates 4MB of storage proof)
  * Due to implementation details it's hard to find the exact amount of storage proof generated by a receipt, so an upper bound estimation is used instead. This upper bound assumes that every removal generates additional 2000 bytes of storage proof, so receipts which perform a lot of trie removals might be limited more than theoretically applicable.
```
