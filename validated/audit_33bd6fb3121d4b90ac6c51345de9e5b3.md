Audit Report

## Title
Pool Admin Can Set Per-Bin Additional Fees Without Cap Check, Bypassing Protocol Fee Governance — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`setPoolBinAdditionalFees` in `MetricOmmPoolFactory.sol` forwards caller-supplied `addFeeBuyE6` and `addFeeSellE6` values directly to `setBinAdditionalFees` on the pool with no upper-bound validation. Every other pool-admin fee setter enforces `maxAdminSpreadFeeE6`, but this path does not. The per-bin values are added directly to `baseFeeX64` in the swap hot-path, allowing a pool admin to impose fees above the protocol's hard-coded governance caps on every swap crossing the affected bin, constituting a direct loss of user principal.

## Finding Description
`setPoolAdminFees` enforces explicit caps before writing any fee: [1](#0-0) 

`setPoolBinAdditionalFees` performs no such check and forwards values directly: [2](#0-1) 

`setBinAdditionalFees` on the pool validates only the bin index, then writes the unchecked values: [3](#0-2) 

These values are consumed in the swap hot-path by adding them directly to `baseFeeX64`: [4](#0-3) [5](#0-4) 

`uint16` max is 65,535, so the maximum per-bin additional fee is 65,535 / 1,000,000 ≈ 6.55%. The hard cap for the base spread fee is `HARD_MAX_SPREAD_FEE_E6 = 200,000` (20%): [6](#0-5) 

`maxAdminSpreadFeeE6` is initialized to `HARD_MAX_SPREAD_FEE_E6`: [7](#0-6) 

The total base spread can reach 40% (both sides at 20%). Adding the uncapped per-bin fee pushes the effective per-bin fee to ≈46.55%, well above the 20% hard cap enforced everywhere else.

## Impact Explanation
Every swap crossing a bin with `addFeeBuyE6`/`addFeeSellE6` set to `type(uint16).max` pays an additional ≈6.55% fee on top of the already-capped base spread. This fee is taken from the swap input and credited to the bin's LP balance (minus the protocol's spread share), constituting a direct, quantifiable loss of user principal on every affected swap. The pool admin can apply this to every bin simultaneously, making the entire pool's swap surface more expensive than the protocol's governance caps permit. This is a clear admin-boundary break: pool admin exceeds caps — an explicitly allowed impact category.

## Likelihood Explanation
The pool admin is a semi-trusted role explicitly bounded by caps (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`). The protocol's own code demonstrates the intent to cap admin fee power. A compromised or malicious pool admin can exploit this gap without any on-chain resistance: no timelock, no protocol-owner approval, and no factory-level guard stands between the pool admin and the uncapped `setPoolBinAdditionalFees` call. The call is immediate and repeatable across all bins.

## Recommendation
Add an explicit cap check in `setPoolBinAdditionalFees` analogous to the checks already present in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, define a dedicated `maxAdminBinAdditionalFeeE6` constant so the per-bin cap can be tuned independently of the base spread cap.

## Proof of Concept
```solidity
// Pool admin sets addFeeBuyE6 to type(uint16).max on the active bin
vm.prank(poolAdmin);
factory.setPoolBinAdditionalFees(pool, activeBin, type(uint16).max, type(uint16).max);
// 65535 / 1e6 ≈ 6.55% additional fee — no revert, no cap check

// setPoolAdminFees would revert with AdminFeeTooHigh for any value > maxAdminSpreadFeeE6 (200_000)
// but setPoolBinAdditionalFees accepts 65_535 silently

// Any subsequent swap crossing that bin pays baseFee + 6.55% instead of baseFee
// baseFee can already be at HARD_MAX_SPREAD_FEE_E6 * 2 = 40%
// Effective per-bin fee: ~46.55% — far above the 20% hard cap
```
