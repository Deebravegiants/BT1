### Title
Dijkstra batch transaction fee undercharges reference script costs from sub-transactions — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the minimum fee for a batch (top-level) transaction is computed using only the reference scripts present in the top-level transaction body. Reference scripts used by sub-transactions are silently excluded from the fee calculation, even though a helper function `batchNonDistinctRefScriptsSize` that correctly aggregates all sub-transaction reference scripts exists in the same file but is never wired into the fee path. An attacker can therefore submit a batch transaction whose sub-transactions reference arbitrarily large scripts, paying only the fee for the top-level transaction's reference scripts and bypassing the tiered `tierRefScriptFee` pricing that was introduced specifically to deter DDoS via expensive-to-deserialize scripts.

---

### Finding Description

The Dijkstra era introduces batch transactions: a top-level `Tx TopTx` that embeds an ordered map of `Tx SubTx` sub-transactions. Sub-transactions have no `fee` field and do not pay fees independently; the single fee field in the top-level body is meant to cover the entire batch.

The minimum fee for the batch is computed via:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` delegates to `txNonDistinctRefScriptsSize`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
``` [2](#0-1) 

`txNonDistinctRefScriptsSize` only inspects the top-level transaction's own inputs and reference inputs:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

It never descends into `subTransactionsTxBodyL`. The correct batch-aware function already exists in the same Dijkstra UTxO module but is **never called from the fee path**:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [4](#0-3) 

Sub-transactions do carry reference inputs (`dstbrReferenceInputs`) and their scripts are aggregated into the batch's `ScriptsProvided` pool: [5](#0-4) [6](#0-5) 

Sub-transactions are validated (including reference input lookups) in `dijkstraSubUtxoTransition`, which validates `inputs ∪ refInputs` against the original UTxO: [7](#0-6) 

So the scripts are deserialized and validated, but their size is not charged.

The tiered reference-script fee formula (`tierRefScriptFee`) was introduced after a real DDoS attack on June 25 2024 precisely because large scripts are expensive to deserialize: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

An attacker can submit a Dijkstra batch transaction whose top-level body is minimal (small reference-script fee) while embedding sub-transactions that each reference large Plutus scripts via reference inputs. The node must deserialize every referenced script during validation, but the fee charged covers only the top-level transaction's reference scripts. The protocol therefore processes arbitrarily expensive-to-validate batches for a fee that is far below the intended cost, re-opening the DDoS vector that `tierRefScriptFee` was designed to close.

This matches the allowed Medium impact: *"Attacker-controlled transactions… exceed intended validation limits or modify fees… outside design parameters."*

---

### Likelihood Explanation

The Dijkstra era is production-bound code. The attack requires only the ability to submit a valid transaction — no privileged role, no governance majority, no key compromise. The attacker needs only to:
1. Create UTxO outputs containing large reference scripts (one-time cost).
2. Repeatedly submit batch transactions whose sub-transactions reference those outputs.

The existence of `batchNonDistinctRefScriptsSize` in the same file as the fee dispatch, combined with its non-use, strongly suggests an integration oversight rather than a deliberate design choice.

---

### Recommendation

Replace the `getMinFeeTxUtxo` dispatch for `DijkstraEra` with a version that uses `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
getDijkstraMinFeeTxUtxo ::
  ( EraTx era
  , BabbageEraTxBody era
  , DijkstraEraTxBody era
  ) =>
  PParams era -> Tx TopTx era -> UTxO era -> Coin
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx

instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo
```

This ensures the tiered reference-script fee covers all scripts deserialized during batch validation, consistent with the Conway-era DDoS mitigation.

---

### Proof of Concept

1. Create two UTxO outputs each containing a ~100 KiB Plutus script as a reference script.
2. Construct a Dijkstra `TopTx` with an empty top-level input set and no reference inputs (reference-script fee = 0).
3. Embed two `SubTx` sub-transactions, each referencing one of the large-script UTxO outputs via `referenceInputsTxBodyL`.
4. Compute `getMinFeeTxUtxo pp tx utxo` — it returns the base linear fee with zero reference-script surcharge.
5. Compute `batchNonDistinctRefScriptsSize utxo tx` — it returns ~200 KiB, which at current `minFeeRefScriptCostPerByte = 15` and the tiered multiplier would add thousands of lovelace to the required fee.
6. The transaction is accepted at the lower fee, demonstrating that ~200 KiB of script deserialization work is performed for free.

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
