Audit Report

## Title
Stale `rsETHPrice` Used in `instantWithdrawal()` Enables Over-Redemption Before Circuit Breaker Triggers - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTWithdrawalManager.instantWithdrawal()` computes the asset payout via `getExpectedAssetAmount()`, which reads `LRTOracle.rsETHPrice` — a cached storage variable updated only by explicit calls to `updateRSETHPrice()`. After a slashing event reduces the true backing of rsETH, any user can call `instantWithdrawal()` before the keeper refreshes the price and receive more underlying assets than their rsETH is currently worth. The circuit breaker that would pause the protocol on a significant price drop lives exclusively inside `_updateRsETHPrice()`, which is never invoked in the `instantWithdrawal()` execution path.

## Finding Description

`LRTOracle.rsETHPrice` is a plain storage variable, not a live computation: [1](#0-0) 

It is written only inside `_updateRsETHPrice()`, which is reachable only through `updateRSETHPrice()` (public, `whenNotPaused`) or `updateRSETHPriceAsManager()` (manager-only): [2](#0-1) 

`instantWithdrawal()` computes the payout amount before burning rsETH or touching the vault: [3](#0-2) 

`getExpectedAssetAmount()` reads the stale cached `rsETHPrice` directly, while `getAssetPrice(asset)` fetches a live Chainlink price: [4](#0-3) 

The circuit breaker that pauses `lrtDepositPool`, `withdrawalManager`, and `LRTOracle` on a price drop beyond `pricePercentageLimit` is evaluated only inside `_updateRsETHPrice()`: [5](#0-4) 

Because `instantWithdrawal()` never calls `updateRSETHPrice()`, the circuit breaker is never evaluated in this path. The only guard that could stop the withdrawal — `whenNotPaused` — is only effective if the protocol was already paused before the call. The `CantInstantWithdrawMoreThanAvailable` check only prevents withdrawing more than the vault's physical balance; it does not prevent over-redemption caused by a stale price: [6](#0-5) 

The complete execution path is: (1) read stale `rsETHPrice` → compute `assetAmountUnlocked`; (2) burn rsETH from caller; (3) redeem `assetAmountUnlocked` from the unstaking vault; (4) transfer assets to caller — with no price refresh at any step. [7](#0-6) 

## Impact Explanation

After a slashing event reduces the true per-share backing of rsETH, any caller who invokes `instantWithdrawal()` before `updateRSETHPrice()` is called receives more underlying assets than the current fair value of their rsETH. The excess is drawn from the unstaking vault, directly reducing the assets available to remaining rsETH holders. Because the burn and vault redemption are atomic and irreversible, the over-redeemed assets cannot be recovered once the circuit breaker eventually fires. This constitutes direct theft of other users' funds at rest, matching the **Critical** allowed impact (direct theft of user funds). The submitter classified it as Medium, but the PoC demonstrates concrete, irreversible extraction of assets belonging to other rsETH holders — not merely unclaimed yield.

## Likelihood Explanation

- `instantWithdrawal()` is callable by any unprivileged external user when `isInstantWithdrawalEnabled[asset]` is `true`.
- `rsETHPrice` is updated by off-chain keeper bots on a periodic basis, so a staleness window always exists between any real-world event and the next price update.
- EigenLayer slashing events are realistic and have occurred on mainnet.
- An attacker can monitor on-chain for slashing-related balance drops in `NodeDelegator` contracts and call `instantWithdrawal()` before the keeper bot calls `updateRSETHPrice()`, front-running the circuit breaker with no special privileges required.
- The attack is repeatable across multiple callers in the same staleness window.

## Recommendation

Call `updateRSETHPrice()` (or an equivalent fresh price computation) at the start of `instantWithdrawal()`, before `getExpectedAssetAmount()` is invoked. This ensures the circuit breaker is evaluated in the same transaction and the payout is based on the current rate:

```solidity
function instantWithdrawal(address asset, uint256 rsETHUnstaked, string calldata referralId)
    external nonReentrant whenNotPaused ...
{
    ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
    ...
}
```

If `updateRSETHPrice()` is too expensive to call on every instant withdrawal, a price-staleness check (e.g., a maximum age for `rsETHPrice`) should be enforced as a minimum mitigation.

## Proof of Concept

1. Protocol state: `rsETHPrice = 1.05e18` (stored, last updated 2 hours ago). `pricePercentageLimit = 0.05e18` (5%). Protocol is unpaused.
2. An EigenLayer slashing event reduces the true rsETH backing to `0.95e18` per token. `updateRSETHPrice()` has not been called; the circuit breaker has not fired.
3. Attacker holds `1000e18` rsETH. Calls `instantWithdrawal(ETH, 1000e18, "")`:
   - `getExpectedAssetAmount` computes: `1000e18 * 1.05e18 / 1e18 = 1050 ETH` (stale price).
   - rsETH is burned from attacker.
   - `1050 ETH` is redeemed from the unstaking vault.
   - Attacker receives `1050 ETH` (minus fee).
4. Fair value at current price: `1000e18 * 0.95e18 / 1e18 = 950 ETH`.
5. Attacker extracted ~100 ETH more than fair value, at the expense of remaining rsETH holders.
6. When `updateRSETHPrice()` is eventually called, the circuit breaker detects the `>5%` drop and pauses the protocol — but the over-redemption is already complete and irreversible.

**Foundry fork test plan:** Fork mainnet, set `rsETHPrice` to a value 10% above the current computed price (simulating a post-slash staleness window), call `instantWithdrawal()` as an unprivileged address, assert that the received asset amount exceeds `rsETHUnstaked * currentComputedPrice / assetPrice`, and assert that `updateRSETHPrice()` called immediately after triggers the circuit breaker pause.
