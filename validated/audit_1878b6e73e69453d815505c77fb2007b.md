### Title
Governance Proposal Deposit Permanently Lost to Treasury When Return Address Deregisters Before Proposal Removal - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Epoch.hs`)

---

### Summary

In Conway era, a governance proposal submitter pays a deposit (`ppGovActionDepositL`) and designates a `pProcReturnAddr` stake credential to receive the refund when the proposal is removed (expired, enacted, or removed due to enactment of a conflicting action). The `returnProposalDeposits` function checks registration of the return address **at removal time**, not at submission time. If the return address's stake credential is deregistered between proposal submission and proposal removal, the deposit is silently redirected to the treasury and permanently lost. Critically, the zero-balance check that protects staking rewards from being lost on deregistration does **not** cover governance proposal deposits, leaving no guard against this loss.

---

### Finding Description

`returnProposalDeposits` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Epoch.hs` processes all removed proposals at each epoch boundary:

```haskell
processProposal gas (!accounts, !unclaimed)
  | (Just _accountState, newAccounts) <- updateLookupAccountState addRefund cred accounts =
      (newAccounts, unclaimed)
  | otherwise = (accounts, Map.insert (gasId gas) (gasDeposit gas) unclaimed)
  where
    addRefund = balanceAccountStateL <>~ compactCoinOrError (gasDeposit gas)
    cred = gasReturnAddr gas ^. accountAddressCredentialL
```

If `cred` is not registered, the deposit is placed in `unclaimed` and then swept into the treasury:

```haskell
chainAccountState3 =
  chainAccountState2
    & casTreasuryL <>~ (utxoState0 ^. utxosDonationL <> fold unclaimed)
```

The `GOV` rule does check at proposal submission time (post-bootstrap) that the return address is registered:

```haskell
isAccountRegistered (refundAddress ^. accountAddressCredentialL) (certDState ^. accountsL)
  ?! (injectFailure . ProposalReturnAccountDoesNotExist) refundAddress
```

However, this check is **only at submission time**. After submission, the return address can be freely deregistered. The `ConwayUnRegCert` path in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs` only blocks deregistration when the account's **reward balance** (`balanceAccountStateL`) is non-zero:

```haskell
checkStakeKeyHasZeroRewardBalance = do
  accountState <- mAccountState
  let balanceCompact = accountState ^. balanceAccountStateL
  guard (balanceCompact /= mempty)
  Just $ fromCompact balanceCompact
```

Governance proposal deposits are tracked in `utxosDeposited` (the global deposit pot), **not** in the per-account reward balance. Therefore a user with a pending proposal and a zero reward balance can freely deregister, unknowingly forfeiting their proposal deposit.

The test `depositMovesToTreasuryWhenStakingAddressUnregisters` in `eras/conway/impl/testlib/Test/Cardano/Ledger/Conway/Imp/EpochSpec.hs` explicitly confirms this behavior: after the return address deregisters, the deposit moves to the treasury when the proposal expires.

---

### Impact Explanation

**High — Permanent freezing of funds/deposits where recovery requires a hard fork.**

The governance proposal deposit (`ppGovActionDepositL`) can be a substantial ADA amount. Once the return address deregisters and the proposal is later removed, the deposit is irrecoverably transferred to the treasury. There is no on-chain mechanism to reclaim it; recovery would require a hard fork or a treasury withdrawal governance action (which itself requires governance majority and another deposit).

---

### Likelihood Explanation

**Medium.** A user who:
1. Submits a governance proposal (paying the deposit),
2. Later deregisters their stake credential to reclaim their key deposit or migrate to a new credential,

will silently and permanently lose their governance proposal deposit. The loss is not surfaced as an error — the deregistration succeeds, and the deposit disappears at the next epoch boundary when the proposal is removed. This is a realistic user-error scenario with no in-protocol warning. The analog to the original report is exact: the zero-balance guard protects staking rewards but does not protect governance proposal deposits, leaving a gap identical to the "claim before burn" ordering flaw.

---

### Recommendation

Before allowing a stake credential deregistration (`ConwayUnRegCert`), the `DELEG` rule should additionally check whether the credential is referenced as the `pProcReturnAddr` of any live governance proposal. If so, deregistration should be rejected (or the deposit should be returned immediately to the UTxO before deregistration proceeds). Alternatively, `returnProposalDeposits` should fall back to returning the deposit to the UTxO via a new output rather than silently routing it to the treasury when the return address is unregistered.

---

### Proof of Concept

**Step-by-step:**

1. Alice registers stake credential `cred_A` and submits a governance proposal with `pProcReturnAddr = cred_A`, paying deposit `D` ADA. The `GOV` rule accepts it because `cred_A` is registered at submission time.

2. Alice deregisters `cred_A` (e.g., to reclaim her key deposit). The `DELEG` rule permits this because `cred_A`'s reward balance is zero — governance proposal deposits are not reflected in `balanceAccountStateL`.

3. The proposal eventually expires (or is removed due to enactment of a conflicting action). `returnProposalDeposits` is called at the epoch boundary.

4. `updateLookupAccountState` finds `cred_A` is not registered → the `otherwise` branch fires → `D` ADA is added to `unclaimed`.

5. `chainAccountState3 = chainAccountState2 & casTreasuryL <>~ fold unclaimed` — `D` ADA is permanently added to the treasury.

6. Alice has lost `D` ADA with no error, no warning, and no recourse.

depositMovesToTreasuryWhenStakingAddressUnregisters ::

depositMovesToTreasuryWhenStakingAddressUnregisters = do

  initialTreasury <- getsNES treasuryL
  modifyPParams $ \pp ->
    pp
      & ppGovActionLifetimeL .~ EpochInterval 8
      & ppGovActionDepositL .~ Coin 100
      & ppCommitteeMaxTermLengthL .~ EpochInterval 0
  returnAddr <- registerAccountAddress
  govActionDeposit <- getsNES $ nesEsL . curPParamsEpochStateL . ppGovActionDepositL
  keyDeposit <- getsNES $ nesEsL . curPParamsEpochStateL . ppKeyDepositL
  govPolicy <- getGovPolicy
  gaid <-
    mkProposalWithAccountAddress
      ( ParameterChange
          SNothing
          (emptyPParamsUpdate & ppuGovActionDepositL .~ SJust (Coin 1000000))
          govPolicy
      )
      returnAddr
      >>= submitProposal
  expectPresentGovActionId gaid
  replicateM_ 5 passEpoch
  expectTreasury initialTreasury
  expectRegisteredAccountAddress returnAddr
  submitTx_ $
    mkBasicTx mkBasicTxBody
      & bodyTxL . certsTxBodyL
        .~ SSeq.singleton
          (UnRegDepositTxCert (returnAddr ^. accountAddressCredentialL) keyDeposit)
  expectNotRegisteredRewardAddress returnAddr
  replicateM_ 5 passEpoch
  expectMissingGovActionId gaid
  expectTreasury $ initialTreasury <> govActionDeposit
