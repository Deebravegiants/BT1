### Title
User-Controlled Finality Parameters Allow MPC Signatures for Non-Final Foreign-Chain Transactions - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, `crates/foreign-chain-inspector/src/evm/inspector.rs`)

### Summary
The `verify_foreign_transaction` flow accepts user-supplied finality parameters (`BlockConfirmations` for Bitcoin, `EvmFinality` for EVM chains) with no protocol-enforced minimum floor. The MPC network will issue a valid threshold signature for any transaction that satisfies the caller-chosen threshold — including `confirmations: 1` on Bitcoin or `finality: Latest` on EVM chains — both of which are reversible by a chain reorganization. Because the signature is irrevocable once issued, an attacker can obtain a valid MPC attestation for a transaction that is subsequently reorganized away, enabling a double-spend against any bridge that relies on these signatures.

### Finding Description

The `VerifyForeignTransactionRequestArgs` struct exposes two user-controlled finality parameters:

- `BitcoinRpcRequest::confirmations: BlockConfirmations` — the minimum confirmation depth the caller requires before the MPC network signs.
- `EvmRpcRequest::finality: EvmFinality` — one of `Latest`, `Safe`, or `Finalized`.

Neither the on-chain contract (`verify_foreign_transaction`) nor the off-chain node enforces any minimum value for these fields.

In `crates/contract/src/lib.rs`, `verify_foreign_transaction` validates only that the domain exists, gas is attached, and the chain is whitelisted. It performs no check on the finality level: [1](#0-0) 

The node's Bitcoin inspector (`crates/foreign-chain-inspector/src/bitcoin/inspector.rs`) uses the caller-supplied `block_confirmations_threshold` directly as the acceptance criterion: [2](#0-1) 

The EVM inspector (`crates/foreign-chain-inspector/src/evm/inspector.rs`) maps the caller-supplied `EvmFinality::Latest` directly to `FinalityTag::Latest` when querying the chain head, meaning a transaction in the most recent (unfinalized) block satisfies the check: [3](#0-2) 

The signed payload (`ForeignTxSignPayloadV1`) embeds the original `ForeignChainRpcRequest` — including the caller-chosen `confirmations` or `finality` — so the signature is specifically bound to "this transaction at this (weak) finality level": [4](#0-3) 

The `EvmFinality` enum explicitly includes `Latest` as a valid, accepted variant with no contract-level restriction: [5](#0-4) 

### Impact Explanation

Once the MPC network issues a threshold signature, it is cryptographically irrevocable. A bridge contract that calls `verify_foreign_transaction` and receives a valid MPC signature has no way to distinguish a signature over a finalized transaction from one over a `Latest`-block or 1-confirmation transaction. If the underlying foreign-chain transaction is subsequently reorganized away, the MPC signature remains valid and can be presented to the bridge to claim NEAR-side funds for a deposit that no longer exists on the canonical foreign chain. This is a **double-spend via forged foreign-chain verification** — the MPC network has attested to a state that is no longer true.

This matches the allowed impact: **High — forged foreign-chain verification / light-client-style verification bypass that causes invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

Bitcoin 1-confirmation reorgs occur in practice (selfish mining, natural orphans). EVM `Latest`-block reorgs are routine on chains with fast block times. An attacker who controls mining power, or who simply races a natural reorg, can reliably exploit this. The attack requires only an unprivileged NEAR account and the ability to submit a `verify_foreign_transaction` call — no privileged access, no threshold collusion, no TEE compromise.

### Recommendation

1. **Enforce a protocol-level minimum confirmation/finality floor** in `verify_foreign_transaction` on the contract side. Reject requests where `BitcoinRpcRequest::confirmations < MIN_BITCOIN_CONFIRMATIONS` (e.g., 6) or where `EvmRpcRequest::finality == EvmFinality::Latest`.
2. **Remove `EvmFinality::Latest` from the accepted set** for `verify_foreign_transaction` requests, or gate it behind an explicit governance vote per chain.
3. **Document the security model** so bridge integrators understand that the MPC signature's security guarantee is only as strong as the finality level embedded in the signed payload.

### Proof of Concept

1. Attacker deposits 1 BTC into a bridge contract that calls `verify_foreign_transaction`.
2. Attacker immediately calls `verify_foreign_transaction` with:
   ```json
   { "Bitcoin": { "tx_id": "<deposit_txid>", "confirmations": 1, "extractors": ["BlockHash"] } }
   ```
3. The Bitcoin transaction has 1 confirmation; `block_confirmations_threshold <= transaction_block_confirmation` passes in `BitcoinInspector::extract`.
4. MPC nodes sign `ForeignTxSignPayloadV1 { request: BitcoinRpcRequest { confirmations: 1, ... }, values: [BlockHash(...)] }`.
5. Attacker receives the valid MPC threshold signature and presents it to the bridge to claim NEAR-side funds.
6. Attacker double-spends the Bitcoin deposit (via RBF or by mining a competing block), reorganizing the original transaction away.
7. The MPC signature remains valid; the attacker has claimed NEAR funds for a Bitcoin transaction that no longer exists on the canonical chain.

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }
