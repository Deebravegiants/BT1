### Title
CRCAT Wallet Exception Handler Permanently Deletes Valid Sibling Coin Records on Transient `get_timestamp_for_height` Failure — (`chia/wallet/vc_wallet/cr_cat_wallet.py`)

---

### Summary

The `except Exception` block in `CRCATWallet.add_crcat_coin` is intended to clean up child coin records when the parent spend is not a valid CRCAT. However, it fires on **any** exception — including a transient `PeerRequestException` raised by `get_timestamp_for_height` (the only network call inside the try block, at line 244). When this occurs while processing a pending-approval CRCAT child coin (C2), the handler queries all children of the same parent and permanently deletes every sibling already stored in the wallet DB that belongs to this wallet — including a previously-correctly-stored regular CRCAT sibling (C1). The result is a permanent CRCAT balance reduction until a full wallet resync.

---

### Finding Description

**Root cause — overly broad exception handler with incorrect assumption:**

The comment at line 292 reads:
> `# The parent is not a CAT which means we need to scrub all of its children from our DB`

But the `except Exception` at line 291 catches **all** exceptions, not just CRCAT parse failures. The only network call inside the try block is:

```python
# line 244
created_timestamp = await self.wallet_state_manager.wallet_node.get_timestamp_for_height(uint32(height))
```

This call is made only for pending-approval CRCATs (the `elif` branch, line 235). `get_timestamp_for_height` in `wallet_node.py` raises `PeerRequestException("Error fetching timestamp from all peers")` when no connected peer returns a timestamp: [1](#0-0) 

When that exception propagates to the `except` block, the cleanup code runs: [2](#0-1) 

`get_coin_records_by_parent_id(coin_spend.coin.name())` returns **all** children of the parent coin already in the DB: [3](#0-2) 

Every child whose `wallet_id == self.id()` is then permanently deleted: [4](#0-3) 

Note the `# pragma: no cover` on line 299 — this deletion path has **zero test coverage**, confirming the developers did not anticipate it being reached. [5](#0-4) 

**Why the outer `coin_added` try/except does not protect against this:**

`coin_added` has its own try/except (lines 211–216) that catches failures from `get_coin_state` and `fetch_coin_spend_for_coin_state`. If those fail, `add_crcat_coin` is never called. The transient failure must occur **inside** `add_crcat_coin` — specifically at `get_timestamp_for_height` — for the cleanup code to fire. [6](#0-5)

### Impact Explanation

A valid, correctly-stored CRCAT coin record (C1) is permanently deleted from the wallet DB. The wallet's CRCAT confirmed balance is reduced by C1's amount. The coin is not recoverable without a full wallet resync. This is **corruption of coin records with direct security impact** (High).

---

### Likelihood Explanation

**Required conditions:**

1. Two sibling CRCAT coins (C1 = regular CRCAT, C2 = pending-approval CRCAT) must both be tracked by the **same** CRCAT wallet (same `wallet_id`). The `wallet_id` guard at line 299 means only same-wallet siblings are deleted.
2. C1 must be processed and stored before C2.
3. `get_timestamp_for_height` must fail when processing C2.

**Concrete attack path (peer/sync path):**

- Attacker owns CRCAT coins and controls a full-node peer the victim's wallet connects to.
- Attacker crafts a CRCAT spend creating two outputs to the victim's wallet: C1 (regular CRCAT, inner puzzle hash matches a victim derivation record) and C2 (pending-approval CRCAT, inner puzzle hash matches `construct_pending_approval_state`).
- Victim's wallet processes C1 first → `add_crcat_coin` succeeds → C1 stored in DB.
- Victim's wallet processes C2 → `add_crcat_coin` enters the `elif` branch → calls `get_timestamp_for_height`.
- Attacker's peer returns `None` from `get_timestamp_for_height_from_peer` (e.g., returns an empty header block response), causing `get_timestamp_for_height` to raise `PeerRequestException`.
- Exception handler fires, finds C1 in DB (child of same parent, same `wallet_id`), deletes it permanently.

Non-attacker trigger: transient network disconnection during `get_timestamp_for_height` in a self-send scenario (wallet sends CRCAT to itself with a pending-approval output).

Likelihood is **low** due to the specific coin structure required, but the path is concrete and locally testable.

---

### Recommendation

1. **Narrow the exception handler**: Catch only CRCAT parse/validation exceptions (e.g., `ValueError`, `AssertionError` from `CRCAT.get_next_from_coin_spend`), not `PeerRequestException` or other transient errors.
2. **Move the network call outside the try block**: Call `get_timestamp_for_height` before entering the try block, or handle its failure separately without triggering the cleanup path.
3. **Add a guard**: Before deleting a coin record in the cleanup path, verify the parent spend is genuinely not a CRCAT (e.g., check `CRCAT.get_next_from_coin_spend` independently) rather than relying on any exception as proof.
