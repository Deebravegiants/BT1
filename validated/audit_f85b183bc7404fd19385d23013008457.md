### Title
Caller-Controlled Weak Finality Bypasses Foreign-Chain Verification Guarantees — (`crates/near-mpc-contract-interface/src/types/foreign_chain.rs`, `crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint accepts caller-specified finality levels including `EvmFinality::Latest` and `SolanaFinality::Processed`. These variants provide no meaningful finality guarantee — they represent the current, reorganizable chain tip. An unprivileged caller can obtain a valid MPC threshold signature attesting to a foreign-chain transaction that has not reached finality and may be reorganized away, enabling forged foreign-chain verification and double-spend conditions on any bridge or application consuming the signed payload.

---

### Finding Description

The `EvmFinality` enum exposes a `Latest` variant alongside `Safe` and `Finalized`: [1](#0-0) 

Similarly, `SolanaFinality` exposes `Processed` — the weakest Solana commitment level, not confirmed by a supermajority: [2](#0-1) 

The contract's `verify_foreign_transaction` entry point performs **no validation of the requested finality level** — it accepts any `EvmFinality` variant the caller supplies: [3](#0-2) 

On the node side, `EvmInspector::verify_finality_level` maps `EthereumFinality::Latest` directly to `FinalityTag::Latest` and then checks only that `latest_block_number >= receipt_block_number`. For a transaction that was just included, this check is trivially satisfied: [4](#0-3) 

`verify_block_is_canonical` then checks that the receipt's block hash matches the canonical block at that height — also trivially true for a freshly included transaction: [5](#0-4) 

After both checks pass, the MPC nodes proceed to threshold-sign the `ForeignTxSignPayloadV1` containing the request and extracted values. The signed payload is returned to the caller via `respond_verify_foreign_tx`: [6](#0-5) 

The root cause is structurally identical to the reference report: just as `getAmountsOut()` reads from manipulatable spot reserves and the slippage percentage provides no real protection, `EvmFinality::Latest` reads from the manipulatable chain tip and the finality check provides no real protection. In both cases, the "oracle" value is caller-controlled and can be made to reflect a transient, reversible state.

---

### Impact Explanation

**Impact: High** — forged foreign-chain verification causing invalid bridge execution or double-spend conditions.

An attacker can:
1. Submit a transaction on Ethereum (e.g., a deposit to a bridge contract).
2. Immediately call `verify_foreign_transaction` with `EvmFinality::Latest` before the transaction reaches `Safe` or `Finalized` status.
3. MPC nodes verify the transaction exists in the latest block and produce a threshold signature over `ForeignTxSignPayloadV1`.
4. The attacker receives a valid, irrevocable MPC signature attesting to the transaction.
5. The transaction is then reorganized out of the canonical chain (naturally or via attacker-induced reorg on a lower-hashrate EVM chain or L2).
6. The attacker presents the MPC signature to the destination bridge/application to claim funds, while the source transaction no longer exists on-chain.

The signed payload is a canonical Borsh-serialized structure that the destination contract treats as authoritative proof. There is no mechanism to revoke or invalidate an already-issued MPC signature.

---

### Likelihood Explanation

**Likelihood: Medium.**

- `EvmFinality::Latest` is explicitly available to any unprivileged caller with no contract-level restriction.
- The system supports multiple EVM-compatible chains beyond Ethereum mainnet: Base, Arbitrum, Polygon, BNB, HyperEvm, Abstract — all of which have different (and generally weaker) reorg resistance than Ethereum mainnet. [7](#0-6) 
- On Ethereum mainnet, reorgs are rare but not impossible (1–2 block reorgs occur periodically). On some supported L2s, sequencer-level reorgs are more feasible.
- The attacker does not need to cause the reorg themselves — they only need to race the MPC signing round against natural finality, which is a timing advantage they control by choosing when to submit the `verify_foreign_transaction` call.
- `SolanaFinality::Processed` carries the same risk on Solana, where `Processed` transactions are explicitly not guaranteed to be included in the ledger.

---

### Recommendation

1. **Remove weak finality variants from the public API**: Drop `EvmFinality::Latest` and `SolanaFinality::Processed` from the `ForeignChainRpcRequest` types, or mark them as explicitly unsafe and reject them at the contract level in `verify_foreign_transaction`.
2. **Enforce a minimum finality level in the contract**: Before enqueuing the yield request, validate that the supplied finality is at least `Safe` (EVM) or `Confirmed` (Solana). This mirrors the recommendation in the reference report to input a concrete expected amount rather than a manipulatable percentage.
3. **Per-chain minimum finality governance**: Allow the on-chain `ChainEntry` governance (already used for RPC provider whitelisting) to also encode a minimum acceptable finality level per chain, so the network can enforce stronger guarantees as chain-specific risk profiles evolve.

---

### Proof of Concept

```
1. Attacker deploys or uses an existing bridge on Ethereum that accepts
   VerifyForeignTransactionResponse signatures as proof of deposit.

2. Attacker submits a deposit tx on Ethereum (tx_hash = 0xABC...).
   The tx lands in block N (latest).

3. Attacker immediately calls verify_foreign_transaction on the NEAR MPC contract:
     request = EvmRpcRequest {
         tx_id: 0xABC...,
         finality: EvmFinality::Latest,   // <-- caller-controlled weak finality
         extractors: [EvmExtractor::Log { log_index: 0 }],
     }

4. MPC nodes query their RPC providers:
   - eth_getTransactionReceipt(0xABC...) → receipt in block N ✓
   - verify_finality_level: eth_getBlockByNumber("latest") → block N >= N ✓ (trivially)
   - verify_block_is_canonical: eth_getBlockByNumber(N) → hash matches ✓ (trivially)
   - Threshold sign ForeignTxSignPayloadV1 { request, extracted_log }

5. NEAR contract resolves the yield and returns VerifyForeignTransactionResponse
   { payload_hash, signature } to the attacker.

6. A 1-block reorg occurs (natural or induced) removing block N.
   The deposit tx 0xABC... no longer exists on the canonical Ethereum chain.

7. Attacker presents the MPC signature to the destination bridge contract,
   which verifies the NEAR MPC signature and releases funds — for a deposit
   that no longer exists on Ethereum.
```

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
