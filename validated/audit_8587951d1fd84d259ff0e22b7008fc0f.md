Audit Report

## Title
Protocol Spread-Fee and Notional-Fee Calculations Use Floor Division, Causing Systematic Undercollection of Protocol Revenue - (`metric-core/contracts/libraries/SwapMath.sol`, `metric-core/contracts/MetricOmmPool.sol`)

## Summary
Every swap computes the protocol's share of the spread fee and the notional fee using plain Solidity integer division (`/`), which truncates toward zero (floor division). This causes the protocol to collect strictly less than its entitled share on every swap that produces a non-zero remainder. The loss is one-directional, requires no special setup, and accumulates monotonically with swap volume.

## Finding Description
**Spread-fee path — three call sites in `SwapMath.sol`:**

`SwapMath.sol` line 409 correctly computes the total LP fee with `Math.ceilDiv` (rounds up, favours protocol), but the protocol's share of that fee is then split with plain `/` (floor):

```solidity
// SwapMath.sol L409-411
uint256 feeAmountScaled = Math.ceilDiv(amountInScaled * currBinBuyFeeX64, ONE_X64);
amountInScaled += feeAmountScaled;
uint256 protocolFeeAmountScaled = (feeAmountScaled * spreadFeeE6) / 1e6;  // ← floor
```

The bin (LP) receives `amountInScaled - protocolFeeAmountScaled` (line 415). Because `protocolFeeAmountScaled` is rounded down, the LP silently absorbs the remainder that should belong to the protocol. The identical pattern appears at lines 638 and 777 for the exact-output buy-token1 and sell-token0 paths.

**Notional-fee path — four call sites in `MetricOmmPool.sol`:**

```solidity
// MetricOmmPool.sol L756
uint256 notionalFeeScaled = uint256(-amount1DeltaScaled) * notionalFeeE8 / 1e8;  // ← floor
// MetricOmmPool.sol L766
uint256 notionalFeeScaled = uint256(-amount0DeltaScaled) * notionalFeeE8 / 1e8;  // ← floor
// MetricOmmPool.sol L777, L785
uint256 notionalFeeScaled = feeExclusiveInputScaled * notionalFeeE8 / 1e8;        // ← floor
```

`notionalFeeScaled` is the fee deducted from the trader's output (exact-in) or added to the trader's input (exact-out). Floor division makes `notionalFeeScaled` smaller than the exact value, so the trader receives more output or pays less input than the protocol is entitled to charge.

The codebase's own rounding discipline — `Math.ceilDiv` for LP-fee computation (line 409), `Math.Rounding.Ceil` for `baseFeeX64` (line 71), and `Math.ceilDiv` in `LiquidityLib` for token demand — is broken exclusively at these seven fee-split and notional-fee lines.

Existing guards (overflow checks, `toUint104`/`toUint128` casts, `notionalFeeScaled > 0` guard) do not address rounding direction and are therefore insufficient.

## Impact Explanation
On every swap with a non-zero remainder in either division, the protocol collects one fewer scaled unit of fee than it is owed. The loss accrues to `state.protocolFeeAmountScaled` (spread fee) and `notionalFeeToken{0,1}Scaled` (notional fee), both of which are the authoritative accounting variables for protocol revenue withdrawal. The shortfall is permanent and monotonically increasing with swap volume, constituting a direct, continuous loss of protocol fees — an allowed impact under the contest rules.

## Likelihood Explanation
Triggered unconditionally on every swap that produces a non-zero remainder in either division, which is the overwhelming majority of real swaps. No attacker action, privileged access, or special pool state is required; any unprivileged trader calling `swap` on any pool with `spreadFeeE6 > 0` or `notionalFeeE8 > 0` passively causes the loss.

## Recommendation
Replace all seven floor-division instances with `Math.ceilDiv`, consistent with the protocol's existing rounding policy:

```diff
// SwapMath.sol – spread fee split (all three call sites: L411, L638, L777)
- uint256 protocolFeeAmountScaled = (feeAmountScaled * spreadFeeE6) / 1e6;
+ uint256 protocolFeeAmountScaled = Math.ceilDiv(feeAmountScaled * spreadFeeE6, 1e6);

// MetricOmmPool.sol – notional fee (all four call sites: L756, L766, L777, L785)
- uint256 notionalFeeScaled = uint256(-amount1DeltaScaled) * notionalFeeE8 / 1e8;
+ uint256 notionalFeeScaled = Math.ceilDiv(uint256(-amount1DeltaScaled) * notionalFeeE8, 1e8);
```

## Proof of Concept
Concrete numeric example for the spread-fee path:

```
feeAmountScaled = 1_000_001
spreadFeeE6     = 500_000   (50%)

floor:  (1_000_001 * 500_000) / 1_000_000 = 500_000  ← protocol receives
ceil:   ceil(1_000_001 * 500_000 / 1_000_000) = 500_001 ← correct entitlement
Loss per swap = 1 scaled unit
```

For the notional-fee path with `notionalFeeE8 = 1_000_000` (1%):

```
-amount1DeltaScaled = 1_000_000_099

floor:  1_000_000_099 * 1_000_000 / 100_000_000 = 10_000_000  ← fee collected
ceil:   ceil(...)                                = 10_000_001  ← correct fee
Trader receives 1 extra scaled unit per swap.
```

A Foundry fuzz test parameterising `feeAmountScaled`, `spreadFeeE6`, `notionalFeeE8`, and swap amounts will reproduce the shortfall on any input where the remainder is non-zero, confirming the monotonic accumulation of protocol revenue loss.
