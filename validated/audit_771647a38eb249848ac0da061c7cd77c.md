### Title
Receipt Size Limit Not Enforced After Runtime Modification of `output_data_receivers` and Data Receipt Wrapping — (`runtime/runtime/src/lib.rs`, `runtime/near-vm-runner/src/logic/logic.rs`)

---

### Summary

The NEAR runtime enforces `max_receipt_size` (4 MiB) on newly created receipts via `validate_receipt(..., ValidateReceiptMode::NewReceipt)`. However, two code paths in `runtime/runtime/src/lib.rs` **mutate receipts after validation**, pushing their serialized size above `max_receipt_size` without re-checking. This is the direct analog of the ERC-4626 `maxDeposit()` bug: a limit is advertised and checked at one point, but the actual enforced boundary is different because additional data is appended afterward. The nearcore codebase itself acknowledges this as an open bug (issue #12606) and contains workarounds that confirm oversized receipts do enter the chain state.

---

### Finding Description

**Vulnerability class:** Runtime state invariant violation — receipt size limit check performed before receipt mutation, allowing oversized receipts to enter chain state.

**Two concrete trigger paths:**

**Path 1 — `promise_return` with near-max-size receipt:**

In `runtime/runtime/src/lib.rs` lines 1019–1037, when a contract executes `promise_return(receipt_index)` and the parent receipt has non-empty `output_data_receivers`, the runtime appends those receivers to the returned receipt's `output_data_receivers` field **after** the receipt was already validated:

```rust
// lib.rs:1028-1035
ReceiptEnum::Action(new_action_receipt)
| ReceiptEnum::PromiseYield(new_action_receipt) => new_action_receipt
    .output_data_receivers
    .extend_from_slice(&action_receipt.output_data_receivers()),
```

A contract can craft receipt C to be exactly `max_receipt_size` bytes. It passes `validate_receipt` with `NewReceipt` mode. Then the runtime appends `output_data_receivers` from the parent, making C exceed `max_receipt_size`. No re-validation occurs.

**Path 2 — `value_return` with near-max-size payload:**

In `runtime/near-vm-runner/src/logic/logic.rs` lines 3882–3887, `value_return` only checks the payload against `max_length_returned_data` (also 4 MiB):

```rust
if num_bytes > self.config.limit_config.max_length_returned_data {
    return Err(HostError::ReturnedValueLengthExceeded { ... });
}
```

When the runtime wraps this value in a `DataReceipt` (lib.rs lines 1044–1056), the resulting receipt includes additional Borsh-encoded fields (`predecessor_id`, `receiver_id`, `receipt_id`, `data_id`, enum discriminant, length prefix), making the total serialized receipt larger than `max_receipt_size`. The `validate_data_receipt` function in `verifier.rs` lines 619–631 only checks `data_len` against `max_length_returned_data`, not the total receipt size.

**The `validate_receipt` function** in `runtime/runtime/src/verifier.rs` lines 533–541 only checks receipt size in `NewReceipt` mode, and this check is performed on the receipt **before** the runtime mutates it. The `ExistingReceipt` mode (used for delayed receipts, line 2448) intentionally skips the size check to handle these already-oversized receipts gracefully.

The `congestion_control.rs` workaround at lines 413–427 explicitly acknowledges the bug and clamps oversized receipt sizes to `max_receipt_size` for bandwidth accounting, confirming that oversized receipts do reach the forwarding stage.

---

### Impact Explanation

The `max_receipt_size` limit was introduced (protocol version 69, `69.yaml` line 8) specifically to keep `ChunkStateWitness` under 17 MiB. Oversized receipts that bypass this limit can cause `ChunkStateWitness` to exceed its size budget. When chunk validators receive an oversized witness, they may produce a different chunk application result, failing to endorse the chunk and causing consensus disruption. The bandwidth scheduler's clamping workaround (treating oversized receipts as `max_receipt_size` for accounting) means the scheduler underestimates actual bandwidth usage, allowing more receipts than the budget permits.

The corrupted protocol value is: the `max_receipt_size` invariant in chain state, and transitively the `ChunkStateWitness` size bound that chunk validators enforce.

---

### Likelihood Explanation

Any unprivileged user can trigger this by:
1. Deploying a contract (standard operation)
2. Calling a method that creates a near-max-size receipt and uses `promise_return` through a promise chain, or returns a near-max-size value through a promise chain

The test contract `max_receipt_size_promise_return_method1` and `max_receipt_size_value_return_method` in `runtime/near-test-contracts/test-contract-rs/src/lib.rs` demonstrate the exact exploit pattern. The tests `test_max_receipt_size_promise_return` and `test_max_receipt_size_value_return` confirm that oversized receipts successfully enter the chain.

---

### Recommendation

After the runtime mutates a receipt (appending `output_data_receivers` at lib.rs lines 1028–1035, or creating a `DataReceipt` at lines 1044–1056), re-validate the final receipt size against `max_receipt_size` before forwarding. Alternatively, enforce a tighter limit in `value_return` and receipt creation that accounts for the Borsh serialization overhead of the wrapping structures, so the final receipt can never exceed `max_receipt_size`.

/// Validates given data receipt. Checks validity of the length of the returned data.
fn validate_data_receipt(
    limit_config: &LimitConfig,
    receipt: &DataReceipt,
) -> Result<(), ReceiptValidationError> {
    let data_len = receipt.data.as_ref().map(|data| data.len()).unwrap_or(0);
    if data_len as u64 > limit_config.max_length_returned_data {
        return Err(ReceiptValidationError::ReturnedValueLengthExceeded {
            length: data_len as u64,
            limit: limit_config.max_length_returned_data,
        });
    }
    Ok(())
}
