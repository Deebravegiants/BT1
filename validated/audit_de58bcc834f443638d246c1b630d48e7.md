Audit Report

## Title
Reward tokens permanently frozen when `totalKernelStaked` drops to zero mid-period — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool` guards `notifyRewardAmount` against zero stakers at period start but provides no on-chain enforcement preventing all stakers from exiting via `initiateWithdrawal` during an active reward window. When `totalKernelStaked` reaches zero mid-period, `rewardPerTokenStored` freezes while `updatedAt` advances, causing all reward tokens emitted during the zero-staker interval to become permanently irrecoverable with no admin rescue path. The contract's own NatSpec at lines 18–22 acknowledges this limitation and relies solely on an off-chain operational assumption that is not enforced on-chain.

## Finding Description
`initiateWithdrawal` immediately decrements `totalKernelStaked` at lines 325–326:

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
``` [1](#0-0) 

Once `totalKernelStaked == 0`, every subsequent call to the `updateReward` modifier (lines 232–234) executes `rewardPerTokenStored = rewardPerToken()`, which short-circuits and returns the unchanged stored value (lines 409–411), while `updatedAt` is advanced to `lastTimeRewardApplicable()`: [2](#0-1) [3](#0-2) 

The reward tokens that should have been distributed during the zero-staker interval (`rewardRate × elapsed_zero_staker_time`) are never credited to any account. When `notifyRewardAmount` is eventually called again, the `remaining` rollover at lines 582–583 only captures future undistributed rewards from the current rate; it does not recover the already-elapsed zero-staker gap: [4](#0-3) 

There is no `recoverUnallocatedRewards` or equivalent sweep function anywhere in the contract. The contract's own NatSpec (lines 18–22) explicitly acknowledges this behavior and states the mitigation is operational ("ensuring there are always some tokens staked… for the entire duration of the reward period"), which is not enforced on-chain: [5](#0-4) 

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens emitted during any interval where `totalKernelStaked == 0` are permanently locked in the contract with no recovery path. The magnitude of loss is `rewardRate × (duration of zero-staker period)`. This matches the allowed impact class "Permanent freezing of unclaimed yield."

## Likelihood Explanation
**Low.** Triggering the loss requires all stakers to exit simultaneously during an active reward window. `initiateWithdrawal` is fully permissionless and requires no admin action. Realistic triggers include coordinated exits, a depeg event for the staking token, or a migration scenario. The contract's own documentation confirms the protocol team is aware of this risk and relies on an off-chain operational assumption to prevent it.

## Recommendation
1. **Track unallocated rewards.** In the `updateReward` modifier, when `totalKernelStaked == 0`, accumulate `rewardRate * (lastTimeRewardApplicable() - updatedAt)` into a dedicated `unallocatedRewards` state variable instead of silently discarding it. Roll this surplus into the next `notifyRewardAmount` call.
2. **Add an admin rescue function.** Provide a `recoverUnallocatedRewards` function restricted to `DEFAULT_ADMIN_ROLE` that transfers any surplus reward tokens (`rewardsToken.balanceOf(address(this)) - totalTrackedRewards`) to a treasury address.

## Proof of Concept
```solidity
function test_rewardsLost_noStakers() public {
    // Setup: user1 stakes, admin notifies rewards
    kernelToken.mint(user1, 100 ether);
    vm.startPrank(user1);
    kernelToken.approve(address(pool), 100 ether);
    pool.stake(100 ether);
    vm.stopPrank();

    rewardToken.mint(admin, 1 ether);
    rewardToken.approve(address(pool), 1 ether);
    pool.notifyRewardAmount(1 ether);

    // user1 immediately initiates withdrawal → totalKernelStaked = 0
    vm.prank(user1);
    pool.initiateWithdrawal(100 ether);

    // Reward period elapses with zero stakers
    skip(7 days);

    vm.prank(user1);
    pool.claimWithdrawal(1);
    vm.prank(user1);
    pool.getReward();

    // Nearly all rewards are stuck in the contract
    assertGt(rewardToken.balanceOf(address(pool)), 0.99 ether);
}
```
The test demonstrates that rewards earned only for the instant before withdrawal are claimable; the remainder (~1 ether minus dust) is permanently stranded in the contract with no recovery path.
