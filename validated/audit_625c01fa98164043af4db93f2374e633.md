Audit Report

## Title
Uninitialized `rsETHPrice` in `LRTOracle.initialize()` Causes Division-by-Zero on All Deposits — (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.initialize()` never sets `rsETHPrice`, leaving it at the Solidity default of `0`. `LRTDepositPool.getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()` on every deposit with no zero-check guard. As a result, every call to `depositETH` and `depositAsset` reverts with an arithmetic panic (`0x12`) from the moment of deployment until `updateRSETHPrice()` is explicitly called, rendering the protocol's primary entry point non-functional.

## Finding Description
`LRTOracle.initialize()` (L64–68) only assigns `lrtConfig` and emits an event; `rsETHPrice` is never written, so it remains `0`. The sole setter is `_updateRsETHPrice()` (L214–316), which is invoked by the public `updateRSETHPrice()` or the manager-gated `updateRSETHPriceAsManager()`. When `rsethSupply == 0` (the state at fresh deployment), `_updateRsETHPrice()` sets `rsETHPrice = 1 ether` and returns early (L218–222), but this function is never called inside `initialize()`.

`LRTDepositPool.getRsETHAmountToMint` (L506–521) computes:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
with no guard against `rsETHPrice == 0`. Both `depositETH` (L76–93) and `depositAsset` (L99–118) call `_beforeDeposit` (L648–670), which calls `getRsETHAmountToMint` (L665). In Solidity 0.8.x, integer division by zero triggers an arithmetic panic revert (`0x12`), so every deposit attempt fails. For `depositETH`, the sent ETH is returned on revert. For `depositAsset`, the ERC20 `safeTransferFrom` (L114) executes only after `_beforeDeposit` returns, so no tokens are transferred before the revert.

No existing check prevents this: `_beforeDeposit` validates `depositAmount > 0` and deposit limits, but does not check `rsETHPrice`. The `whenNotPaused` modifier on `updateRSETHPrice()` is satisfied at deployment since `paused` defaults to `false`.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** All user deposits via `depositETH` and `depositAsset` are completely blocked from deployment until `updateRSETHPrice()` is called. No user funds are at risk of theft or permanent loss; transactions revert cleanly. The protocol's core deposit functionality is non-operational during this window.

## Likelihood Explanation
The condition is present on every fresh deployment or proxy upgrade that resets `rsETHPrice`. Because `updateRSETHPrice()` is `public` with no role restriction, any caller — including the depositor themselves — can unblock the protocol by calling it first. The window is therefore short in practice but structurally guaranteed to exist on every deployment. A depositor who does not know to call `updateRSETHPrice()` first will have their transaction revert.

## Recommendation
Call `_updateRsETHPrice()` at the end of `LRTOracle.initialize()`, or directly assign `rsETHPrice = 1 ether` (and `highestRsethPrice = 1 ether`) when `rsethSupply == 0` during initialization. This ensures `rsETHPrice` is valid before any dependent function can be called, eliminating the deployment window entirely.

## Proof of Concept
1. Deploy `LRTOracle` proxy and call `initialize(lrtConfigAddr)`. `rsETHPrice` is `0`.
2. Deploy `LRTDepositPool` proxy and call `initialize(lrtConfigAddr)`.
3. Any user calls `depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `_beforeDeposit` → `getRsETHAmountToMint(ETH_TOKEN, 1 ether)`.
5. `lrtOracle.rsETHPrice()` returns `0`.
6. Solidity executes `(1e18 * assetPrice) / 0` → arithmetic panic revert `0x12`.
7. Transaction reverts; ETH is returned to caller.
8. All subsequent deposits fail identically.
9. **Resolution:** Any caller invokes `updateRSETHPrice()` (public, no role required). `rsethSupply == 0` → `rsETHPrice = 1 ether`. Subsequent deposits succeed.
