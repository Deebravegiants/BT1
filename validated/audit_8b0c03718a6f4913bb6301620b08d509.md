### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Pending Request — Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but it never checks that `payload_hash` is the correct hash for the pending `request`. A single Byzantine attested participant can replay any previously observed valid `(payload_hash, signature)` pair to satisfy any pending `verify_foreign_transaction` request, delivering a fraudulent foreign-chain verification response without the MPC network ever inspecting the target transaction.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs two checks before resolving pending yields:

1. The ECDSA signature in `response.signature` is valid over `response.payload_hash` using the domain's root public key.
2. The `request` key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What it does **not** check is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1({ request: request.request, values: <actual_extracted_values> })))`. The `payload_hash` is a free parameter supplied by the caller; the contract only verifies the signature over it, not its semantic binding to the pending request.

The correct hash is computed by the MPC nodes in `build_signature_request` as `foreign_tx_payload.compute_msg_hash()`, where `foreign_tx_payload` encodes both the original `ForeignChainRpcRequest` and the extracted on-chain values: [2](#0-1) 

The `ForeignTxSignPayload` structure that produces this hash includes the full request and extracted values: [3](#0-2) 

The contract never reconstructs or validates this hash from the `request` argument. The SDK-side `ForeignChainSignatureVerifier::verify_signature` does perform this check, but it is a user-side helper, not enforced by the contract: [4](#0-3) 

### Impact Explanation

A Byzantine attested participant can:

1. Observe any previously submitted `respond_verify_foreign_tx` transaction on-chain, extracting a valid `(payload_hash_A, sig_A)` pair that the MPC network honestly produced for foreign transaction `tx_A`.
2. Submit (or wait for another user to submit) `verify_foreign_transaction(request_B)` for a different transaction `tx_B` — one that may not exist, may not be confirmed, or may carry different extracted values.
3. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
4. The contract accepts: `sig_A` is a valid MPC signature over `payload_hash_A`, and `request_B` is pending. It resolves all queued yields for `request_B` with the fraudulent response.

The waiting user receives a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to `tx_A`, not `tx_B`. Any downstream bridge contract that does not independently recompute and compare the expected `payload_hash` will treat this as a valid MPC attestation of `tx_B`'s on-chain state, enabling invalid bridge execution or double-spend conditions.

**Impact**: High — forged foreign-chain verification; invalid bridge execution.

### Likelihood Explanation

- Requires one Byzantine attested participant (below threshold). No threshold collusion needed.
- All past `(payload_hash, signature)` pairs are permanently visible on-chain; the attacker has an ever-growing pool of replayable pairs.
- The attacker can self-submit `verify_foreign_transaction` for any target request, requiring only 1 yoctonear deposit.
- Downstream contracts that rely on the MPC contract as a trust anchor (without re-running `ForeignChainSignatureVerifier::verify_signature`) are directly exploitable.

### Recommendation

In `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` argument and reject any response whose `payload_hash` does not match. Since the contract does not store the extracted `values` (they are off-chain), the binding must be enforced differently — for example, by including the `request` key's canonical hash in the signed payload (i.e., making the MPC nodes sign `SHA-256(borsh(ForeignTxSignPayload::V1({ request: request.request, values: ... })))` where `request.request` is the on-chain pending key), and then verifying on-chain that `payload_hash` starts with or is derived from the pending request's identity. Alternatively, store a commitment to the expected `payload_hash` at request submission time and verify it on response.

### Proof of Concept

```
// Setup: honest computation for tx_A already completed and is on-chain.
// payload_hash_A = SHA-256(borsh(ForeignTxSignPayload::V1({request: request_A, values: [block_hash_A]})))
// sig_A = MPC_sign(payload_hash_A)  -- visible on-chain from prior respond_verify_foreign_tx call

// Step 1: Byzantine participant submits a new request for a fake/unconfirmed tx_B
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest { tx_id: tx_B, ... }),
    domain_id: ...,
    payload_version: ForeignTxPayloadVersion::V1,
});

// Step 2: Byzantine participant replays (payload_hash_A, sig_A) for request_B
contract.respond_verify_foreign_tx(
    request_B,  // pending in contract
    VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,  // hash of tx_A's payload, not tx_B's
        signature: sig_A,              // valid MPC signature over payload_hash_A
    }
);
// Contract accepts: sig_A is valid over payload_hash_A, request_B is pending.
// User receives fraudulent response attesting to tx_A's state for their tx_B request.
```

The root cause is at: [5](#0-4) 

The contract uses `response.payload_hash` directly as the message to verify, without checking it equals the hash the MPC network should have computed for `request`. [6](#0-5)
