Audit Report

## Title
Stale `rsETHPrice` Enables Yield Extraction via Deposit-Before-Price-Update Attack - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.rsETHPrice` is a cached storage variable updated only when `updateRSETHPrice()` is called. Because deposits in `LRTDepositPool` and withdrawal initiations in `LRTWithdrawalManager` both read this cached value without any freshness enforcement, an attacker can deposit at a stale (lower) price to receive more rsETH than fair value, wait for the price to advance to its true value, and redeem at the higher price — extracting yield that belongs to existing rsETH holders.

## Finding Description

`LRTOracle.updateRSETHPrice()` is declared `public` with no role restriction beyond `whenNotPaused`:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`rsETHPrice` is a storage variable written only inside `_updateRsETHPrice()` at line 313:

```solidity
rsETHPrice = newRsETHPrice;
```

Between successive calls to `updateRSETHPrice()`, staking rewards and EigenLayer yield cause the true TVL to grow while `rsETHPrice` remains frozen. `LRTDepositPool.getRsETHAmountToMint()` divides by this stale price:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Because `rsETHPrice_stale < rsETHPrice_true`, the division yields a larger `rsethAmountToMint` than the depositor is entitled to.

After the price advances (either by the attacker calling `updateRSETHPrice()` or by the protocol's own keeper), the attacker initiates a withdrawal. `LRTWithdrawalManager.getExpectedAssetAmount()` uses the now-higher `rsETHPrice` to compute the payout:

```solidity
// contracts/LRTWithdrawalManager.sol L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

The `expectedAssetAmount` is locked in at withdrawal initiation time. At unlock time, `_calculatePayoutAmount()` returns `min(expectedAssetAmount, currentReturn)`, so if the price has not declined further, the attacker receives the full amount computed at the higher price.

**Regarding the `pricePercentageLimit` guard:** `_updateRsETHPrice()` contains a check at lines 252–266 that reverts an unprivileged caller if the price increase exceeds `pricePercentageLimit`. However, this guard does not defeat the attack because:
1. If `pricePercentageLimit == 0` (its default unset value), the guard is entirely bypassed (`pricePercentageLimit > 0 && ...` evaluates false).
2. Even when the guard is active, the attack does not require the attacker to call `updateRSETHPrice()` themselves — the attacker only needs to deposit *before* the protocol's own keeper/manager calls `updateRSETHPriceAsManager()` (which has no price-increase restriction). The attacker simply monitors the mempool or block timestamps and deposits ahead of the routine update.
3. For small, normal reward accrual (e.g., daily staking yield of ~0.01%), the price increase is unlikely to exceed any reasonable `pricePercentageLimit`, so the attacker can call `updateRSETHPrice()` themselves without triggering the revert.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders accumulate yield as the protocol's TVL grows between price updates. An attacker who deposits at the stale price receives more rsETH than their proportional share of TVL, then redeems at the true price, extracting a portion of the accrued yield that should have remained with pre-existing holders. The profit scales linearly with deposit size and with the magnitude of price staleness. For a protocol managing hundreds of millions in TVL with daily reward accrual, a single well-timed large deposit can extract material yield per execution. This matches the allowed impact: **High. Theft of unclaimed yield.**

## Likelihood Explanation

**Medium.**

- `updateRSETHPrice()` is permissionless and observable on-chain; the attacker knows exactly when the price was last updated.
- Reward accrual is continuous and predictable (Ethereum staking APY ~3–4%, EigenLayer rewards on top).
- The attacker does not need to front-run anyone — they deposit, then either call `updateRSETHPrice()` themselves (if within the price limit) or simply wait for the protocol's keeper to do so.
- The 8-day withdrawal delay adds friction but does not eliminate profit; if `isInstantWithdrawalEnabled` is active for the asset, the delay is bypassed entirely (minus the instant-withdrawal fee, which may be smaller than the yield captured).
- No special privileges are required; any EOA or contract can execute the full sequence.

## Recommendation

1. **Force a price refresh before minting.** Call `_updateRsETHPrice()` (or compute the live TVL-based price inline) inside `getRsETHAmountToMint()` so the mint calculation always uses the current price, not a cached one.
2. **Enforce a price-freshness window.** Record a `lastPriceUpdateTimestamp` and revert deposits if the price is older than a configurable threshold (e.g., 1 hour).
3. **Restrict `updateRSETHPrice()`.** If the price update is intended to be operator-driven, add a role check so external actors cannot trigger it on demand to sandwich their own deposits.

## Proof of Concept

```
Setup:
  TVL = 1,000 ETH, rsETH supply = 1,000, rsETHPrice_stored = 1.000 ETH (last updated 24h ago)
  True TVL after 24h of rewards = 1,001 ETH → true price = 1.001 ETH/rsETH

Step 1 — Attacker calls LRTDepositPool.depositETH{value: 100 ETH}(0, ""):
  rsethAmountToMint = (100e18 * 1e18) / 1.000e18 = 100.000 rsETH  ← uses stale price
  Fair amount                                     = 100 / 1.001 ≈ 99.900 rsETH
  Over-minted                                     ≈ 0.100 rsETH

Step 2 — Attacker (or protocol keeper) calls LRTOracle.updateRSETHPrice():
  New TVL    = 1,001 + 100 = 1,101 ETH
  New supply = 1,000 + 100 = 1,100 rsETH
  New price  = 1,101 / 1,100 ≈ 1.000909 ETH/rsETH

Step 3 — Attacker calls LRTWithdrawalManager.initiateWithdrawal(ETH, 100.000 rsETH):
  expectedAssetAmount = 100.000 * 1.000909 / 1.0 ≈ 100.091 ETH  ← locked at higher price

Step 4 — After 8-day withdrawal delay, attacker calls completeWithdrawal(ETH):
  ETH received ≈ 100.091 ETH

Profit ≈ 0.091 ETH extracted from existing holders' accrued yield.
(Scales to ~$910 per $1M deposit per 1% price staleness gap.)

Foundry fork test plan:
  1. Fork mainnet at a block where rsETHPrice is stale.
  2. Warp block.timestamp forward by 24h without calling updateRSETHPrice().
  3. Deposit 100 ETH as attacker; record rsethAmountToMint.
  4. Call updateRSETHPrice(); record new rsETHPrice.
  5. Initiate withdrawal for the full rsETH balance; record expectedAssetAmount.
  6. Warp forward by withdrawalDelayBlocks; call completeWithdrawal.
  7. Assert ETH received > 100 ETH (profit extracted from existing holders).
