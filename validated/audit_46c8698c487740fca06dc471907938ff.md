### Title
Relayer Deposit Stolen via Deliberately Failing Meta-Transaction Inner Action - (`runtime/runtime/src/actions.rs`)

### Summary

In NEAR's meta-transaction (delegate action) system, the relayer pays all attached deposits for inner actions upfront. However, when an inner action fails on the receiver's shard, the deposit is refunded to the **predecessor of the inner receipt** (the user/sender, Alice), not to the relayer who actually funded it. An unprivileged user can deliberately craft a `DelegateAction` with a large attached deposit targeting a call that will fail, causing the relayer to lose the deposit while the user receives it as a refund.

### Finding Description

When `apply_delegate_action` constructs the inner receipt from a `DelegateAction`, it sets `predecessor_id` to `sender_id` (Alice's account), not the relayer's account: [1](#0-0) 

The code comment at line 471–475 explicitly acknowledges this:

> "Note, Relayer prepaid all fees and all things required by actions: attached deposits and attached gas. If something goes wrong, deposit is refunded to the predecessor, this is sender_id/Sender in DelegateAction." [2](#0-1) 

When the inner receipt fails on the receiver's shard, `refund_unspent_gas_and_deposits` issues the deposit refund to `receipt.balance_refund_receiver()`, which resolves to the receipt's `predecessor_id` — Alice, not the relayer: [3](#0-2) 

The protocol documentation explicitly acknowledges the financial incentive this creates:

> "But what if the transaction fails execution on Bob's shard? At this point, the predecessor is Alice and therefore she receives the token balance refunded, not the relayer. This is something relayer implementations must be aware of since there is a **financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard**." [4](#0-3) 

### Impact Explanation

The corrupted protocol value is the **NEAR token balance** of both parties: the relayer's balance is drained by the deposit amount, and Alice's balance is increased by the same amount. This is a direct, on-chain balance transfer from relayer to user triggered by a deliberately failing inner action. The relayer cannot recover the deposit it paid.

### Likelihood Explanation

Any user (Alice) who interacts with a relayer can exploit this. The attack requires:
1. Alice creates a `DelegateAction` with a large `deposit` attached to a function call targeting a non-existent method or a contract that will revert.
2. Alice signs it and submits it to a relayer.
3. The relayer, after checking Alice's FT balance (or other off-chain checks), submits the transaction — paying the deposit from its own NEAR balance.
4. The inner action fails on the receiver's shard.
5. The deposit is refunded to Alice (predecessor), not the relayer.

The attack is repeatable and requires no special privileges. The only constraint is finding a relayer willing to submit the transaction, which is the relayer's normal function.

### Recommendation

The protocol should either:
1. Set the `predecessor_id` of the inner receipt to the relayer's account (the actual payer) so deposit refunds flow back to the relayer, or
2. Introduce a separate `balance_refund_receiver` field in the inner receipt that explicitly points to the relayer (the `signer_id` of the outer receipt), consistent with how gas refunds already go to the signer/relayer. [5](#0-4) 

Gas refunds already correctly go to the signer (relayer) via `action_receipt.signer_id()`. The same logic should apply to deposit refunds in the meta-transaction context.

### Proof of Concept

1. Relayer (Alice) has 100 NEAR. User (Bob) has 0 NEAR but a valid account and key.
2. Bob creates a `DelegateAction` targeting `receiver.near` with `FunctionCall { method: "nonexistent", deposit: 50 NEAR, gas: 30 TGas }`.
3. Bob signs it and sends it to the relayer.
4. Relayer submits the outer transaction, paying 50 NEAR deposit + gas from its own balance.
5. On `receiver.near`'s shard, the function call fails (`MethodNotFound`).
6. `refund_unspent_gas_and_deposits` issues `Receipt::new_balance_refund(receipt.balance_refund_receiver(), 50 NEAR)` where `balance_refund_receiver()` returns Bob's account (the `predecessor_id` of the inner receipt).
7. Bob receives 50 NEAR. Relayer loses 50 NEAR + gas costs. [6](#0-5) [7](#0-6)
