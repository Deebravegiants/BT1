### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Forged Key Material - (File: `crates/contract/src/lib.rs`)

### Summary
The `respond_ckd` function in the MPC contract performs a cryptographic output check only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`, but performs **no verification** for the `AppPublicKey` variant. A single Byzantine MPC participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary `CKDResponse` for any pending `AppPublicKey` CKD request, causing the contract to resolve the yield with forged key material before honest participants can respond.

### Finding Description

In `respond_ckd`, the contract applies an asymmetric verification gate:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ŌåÉ no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies that the CKD response is consistent with the app's public key and the domain's BLS12-381 root key. For `AppPublicKey`, the response is accepted and forwarded to the caller unconditionally ŌĆö no binding between the response and the stored request is enforced.

Compare this to `respond` (regular sign requests), which independently derives the expected public key from the stored tweak and verifies the signature against it: [2](#0-1) 

`resolve_yields_for` drains the **entire** pending queue on the first accepted call: [3](#0-2) 

This means the first `respond_ckd` call that passes the participant check permanently resolves the request. Honest participants' subsequent calls return `RequestNotFound`.

Both `AppPublicKey` and `AppPublicKeyPV` are accepted as valid production inputs in `request_app_private_key`: [4](#0-3) 

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) can deliver forged confidential key derivation output to any user who submits `request_app_private_key` with the `AppPublicKey` variant. The user receives incorrect private key material with no on-chain indication of the forgery. This constitutes **confidential key derivation output without the required participant authorization**: the threshold was never actually reached for the correct output, yet the contract treats the response as authoritative and resumes the caller's yield.

### Likelihood Explanation

The `AppPublicKey` variant is a reachable production code path. Any single attested MPC participant can exploit this by racing to call `respond_ckd` before the honest leader. The attack requires no special resources beyond holding a valid attestation, and the race window is the network propagation delay between the honest leader's submission and the contract's execution.

### Recommendation

Apply `ckd_output_check` (or an equivalent cryptographic binding) for the `AppPublicKey` variant as well. If the `AppPublicKey` variant is inherently unverifiable on-chain (because the caller did not supply a public key for binding), the contract should either:
1. Reject `AppPublicKey` requests entirely and require `AppPublicKeyPV`, or
2. Require threshold-many `respond_ckd` calls before resolving the yield, so a single Byzantine participant cannot win the race.

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key: AppPublicKey(some_bls_key)`. Contract stores the `CKDRequest` in `pending_ckd_requests` and yields.
2. Byzantine attested participant observes the pending request on-chain.
3. Byzantine participant immediately calls `respond_ckd(request, forged_ckd_response)` ŌĆö any `CKDResponse` value is accepted because the `AppPublicKey` branch is a no-op.
4. Contract calls `resolve_yields_for`, draining the queue and resuming the user's yield with `forged_ckd_response`.
5. Honest participants' subsequent `respond_ckd` calls return `Err(RequestNotFound)` ŌĆö the queue is already empty.
6. User receives incorrect private key material derived from the forged response, with no on-chain signal of the attack.

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
