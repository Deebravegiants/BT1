### Title
User-Controlled Zero-Confirmation Bitcoin Attestation Enables Invalid Bridge Execution — (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

### Summary

No minimum `BlockConfirmations` value is enforced anywhere in the MPC stack. An unprivileged caller can submit `confirmations: 0` in a `verify_foreign_transaction` request, causing the MPC network to produce a valid threshold signature attesting to a Bitcoin transaction with zero finality guarantee. This enables double-spend attacks against bridge contracts that rely on these MPC attestations.

### Finding Description

`BlockConfirmations` is defined as a plain `u64` wrapper with no minimum constraint:

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI schema confirms `"minimum": 0.0` — zero is explicitly allowed. [1](#0-0) 

The on-chain contract method `verify_foreign_transaction` performs no validation of the `confirmations` field — it only checks that the chain is supported and the domain exists: [2](#0-1) 

The node-side `BitcoinInspector::extract()` enforces the confirmation threshold with a single comparison:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

When `block_confirmations_threshold = 0`, the expression `0 <= any_value` is always `true`, so the check passes for any transaction that has been included in any block — even one with a single confirmation. [3](#0-2) 

After the trivially-passing confirmation check, `verify_block_is_canonical` is called, which verifies the block is on the canonical chain **at the moment of the RPC query**. [4](#0-3) 

The MPC network then signs the canonical payload `ForeignTxSignPayloadV1 { request, values }`, where `request` includes the user-supplied `confirmations: 0`. This produces a valid threshold signature over unfinalized state. [5](#0-4) 

The signed payload is committed to the on-chain contract via `respond_verify_foreign_tx`, which only verifies the cryptographic signature against the domain public key — it does not re-validate the confirmation count: [6](#0-5) 

This is the direct analog to the external report's root cause: just as the Usual protocol has no on-chain deviation check for LST prices (relying only on off-chain monitoring that can be front-run), the NEAR MPC protocol has no on-chain minimum finality check for Bitcoin attestations. The signed payload is the root of trust for downstream bridge contracts — once issued, it cannot be revoked.

### Impact Explanation

A bridge contract that uses MPC attestations to release NEAR-side funds is exploitable:

1. Attacker submits Bitcoin transaction T; waits for 1 confirmation in block B.
2. Attacker calls `verify_foreign_transaction` with `BitcoinRpcRequest { tx_id: T, confirmations: 0, extractors: [BlockHash] }`.
3. MPC nodes execute `BitcoinInspector::extract(T, BlockConfirmations(0), [BlockHash])`.
4. Check `0 <= 1` passes; `verify_block_is_canonical` passes (block B is canonical at query time).
5. MPC produces a valid threshold signature over `(request, [BlockHash(B)])`.
6. Attacker uses the signature to claim NEAR funds via the bridge contract.
7. A Bitcoin reorg removes block B; T is no longer confirmed on the canonical chain.
8. Attacker retains NEAR funds; Bitcoin double-spend is complete.

This constitutes **invalid bridge execution and a double-spend condition** — the allowed High impact category. The MPC threshold signature is the root of trust; once issued it is irrevocable.

### Likelihood Explanation

Medium. The attack requires: (a) a bridge contract that does not independently validate the `confirmations` field embedded in the signed payload — a reasonable assumption since the MPC network is positioned as a trusted oracle; (b) a Bitcoin reorg at 1–2 confirmations, which is rare but historically documented. The `confirmations: 0` parameter is freely settable by any unprivileged caller with a 1 yoctoNEAR deposit, with no rate limiting or governance gate.

### Recommendation

Enforce a protocol-level minimum confirmation count for Bitcoin at the contract layer in `verify_foreign_transaction` (e.g., reject `confirmations < 6`), or enforce a hardcoded floor inside `BitcoinInspector::extract()`. The minimum should be a protocol constant, not a caller-supplied value. Similarly, consider restricting `EvmFinality::Latest` for EVM chains, as it carries an analogous reorg risk with no canonical-check guarantee after query time.

### Proof of Concept

```
1. Attacker broadcasts Bitcoin tx T; waits for block B (1 confirmation).
2. Attacker calls verify_foreign_transaction({
       domain_id: <foreign_tx_domain>,
       payload_version: V1,
       request: Bitcoin({ tx_id: T, confirmations: 0, extractors: [BlockHash] })
   }) with 1 yoctoNEAR deposit.
3. Contract accepts (Bitcoin is a supported chain; confirmations field not validated).
4. MPC node: BitcoinInspector::extract(T, BlockConfirmations(0), [BlockHash])
   - getrawtransaction(T) -> { blockhash: B, confirmations: 1 }
   - Check: 0 <= 1  -> passes
   - verify_block_is_canonical(B) -> passes (B is canonical at this moment)
   - Returns [BlockHash(B)]
5. MPC signs ForeignTxSignPayloadV1 { request, values: [BlockHash(B)] }
6. respond_verify_foreign_tx submitted; contract resolves yield with valid signature.
7. Attacker presents signature to bridge contract -> NEAR funds released.
8. Bitcoin reorg: block B is orphaned; T is unconfirmed.
9. Attacker retains NEAR funds; double-spend complete.
```
