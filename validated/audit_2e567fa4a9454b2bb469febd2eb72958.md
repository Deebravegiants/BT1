### Title
No Incentive to Call `verify_tee` Allows Nodes with Expired TEE Attestations to Continue Signing — (`File: crates/contract/src/lib.rs`)

### Summary
The `verify_tee` function is the sole on-chain mechanism for enforcing TEE attestation validity across the active participant set. There is no automatic trigger, no reward, and in fact a negative incentive (it can initiate a disruptive resharing) for any participant to call it. As a result, nodes whose TEE attestations have expired continue to participate in threshold signing indefinitely, bypassing the attestation authorization requirement that is the security foundation of the MPC network.

### Finding Description

The NEAR MPC contract enforces TEE security through a two-phase model:

1. **Attestation submission** — a node calls `submit_participant_info`, which stores a `VerifiedAttestation` in `tee_state.stored_attestations` keyed by TLS public key. [1](#0-0) 

2. **Attestation re-verification** — `verify_tee` calls `reverify_and_cleanup_participants`, which re-checks every active participant's stored attestation against the current time and the current whitelist. Only this call can detect and act on expired attestations. [2](#0-1) 

The guard used on every sensitive node call (`respond`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`) is `assert_caller_is_attested_participant_and_protocol_active`, which calls `tee_state.is_caller_an_attested_participant`. [3](#0-2) 

This check verifies **presence** of a stored attestation, not its current validity. Re-verification of expiry only occurs inside `verify_tee` → `reverify_and_cleanup_participants`. Because `verify_tee` is never called automatically — it is a manually-invoked participant method with no reward and a potentially disruptive side-effect (triggering resharing) — expired attestations are never revoked in practice.

The access control on `verify_tee` itself is `voter_or_panic()`, meaning only current participants can call it. [4](#0-3) 

Participants have a rational disincentive to call `verify_tee`:
- It may trigger an automatic resharing (lines 1762–1764), which is operationally disruptive.
- It may set `accept_requests = false` (line 1737), halting all user-facing signing.
- There is no on-chain reward for calling it.

The result is that a node whose TEE attestation has expired — meaning the hardware-level security guarantee that its key share is protected has lapsed — continues to call `respond`, `vote_pk`, `vote_reshared`, and other privileged methods, and the contract accepts these calls as if the node were still attested.

### Impact Explanation

TEE attestation expiry is the contract's signal that a node's hardware isolation guarantee can no longer be trusted. A node operating past its attestation deadline may have had its enclave compromised, its key share extracted, or its signing behavior subverted. Because `verify_tee` is never called, the contract never learns of this state and continues to include the node's contributions in threshold signatures.

If even one node below the signing threshold has an expired attestation and a compromised enclave, its key share can be extracted. Combined with shares from other (honest) nodes obtained through normal protocol participation, an attacker can reconstruct the threshold key and forge arbitrary signatures for any foreign chain — constituting unauthorized threshold signature issuance and potential key-share recovery.

This maps to the allowed impact: **Critical — bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery.**

### Likelihood Explanation

TEE attestations have a finite validity period governed by `tee_upgrade_deadline_duration_seconds`. [5](#0-4) 

In a production network running continuously, attestations will eventually expire for at least some participants. Because there is no automatic enforcement and no incentive to call `verify_tee`, the window during which expired nodes participate in signing can be arbitrarily long. The likelihood is **Low** for active exploitation (requires a compromised TEE after expiry), but the precondition (expired attestations persisting) is **near-certain** over a long-running network.

### Recommendation

1. **Automatic enforcement at `respond` time**: Extend `is_caller_an_attested_participant` (or the `assert_caller_is_attested_participant_and_protocol_active` wrapper) to check attestation expiry inline, not just presence. This makes the security property self-enforcing without relying on `verify_tee` being called.

2. **Incentivize `verify_tee` calls**: Allow any account (not just participants) to call `verify_tee`, and optionally reward callers who successfully identify and trigger removal of expired participants (analogous to liquidation bounties).

3. **Scheduled enforcement**: Spawn a self-call to `verify_tee` automatically at the end of `vote_reshared` (alongside the existing cleanup promises) so that attestation validity is re-checked at every epoch boundary without relying on external actors. [6](#0-5) 

### Proof of Concept

1. Node A submits a valid TEE attestation via `submit_participant_info`. It is stored in `tee_state.stored_attestations`.
2. Time passes; Node A's attestation crosses its expiry deadline (`tee_upgrade_deadline_duration_seconds`).
3. No participant calls `verify_tee` (rational: doing so would trigger resharing and halt requests).
4. Node A calls `respond(request, signature)`. The contract executes `assert_caller_is_attested_participant_and_protocol_active` → `is_caller_an_attested_participant`, which finds Node A's entry in `stored_attestations` and passes. [3](#0-2) 
5. The contract accepts Node A's signature contribution. Node A's expired (potentially compromised) key share is used in threshold signing.
6. If Node A's enclave was compromised after attestation expiry, the attacker now has Node A's key share and can combine it with shares obtained through normal protocol participation to reconstruct the threshold key.
7. The contract's `accept_requests` flag remains `true` throughout; no safety mechanism fires. [7](#0-6)

    pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact — the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
                }

                // here, we set it to true, because at this point, we have at least `threshold`
                // number of participants with an accepted Tee status.
                self.accept_requests = true;

                // do we want to adjust the threshold?
                //let n_participants_new = new_participants.len();
                //let new_threshold = (3 * n_participants_new + 4) / 5; // minimum 60%
                //let new_threshold = new_threshold.max(2); // but also minimum 2
                let new_threshold = usize::try_from(current_params.threshold().value())
                    .expect("threshold value fits in usize");

                let threshold_parameters = ThresholdParameters::new(
                    participants_with_valid_attestation,
                    Threshold::new(new_threshold as u64),
                )
                .expect("Require valid threshold parameters"); // this should never happen.
                current_params.validate_incoming_proposal(&threshold_parameters)?;
                // This resharing only changes the participant set, so the
                // per-domain reconstruction-threshold updates map is empty.
                let proposed_parameters =
                    ProposedThresholdParameters::new(threshold_parameters, BTreeMap::new());
                let res = running_state.transition_to_resharing_no_checks(&proposed_parameters);
                if let Some(resharing) = res {
                    self.protocol_state = ProtocolContractState::Resharing(resharing);
                }

                Ok(true)
            }
        }
    }
