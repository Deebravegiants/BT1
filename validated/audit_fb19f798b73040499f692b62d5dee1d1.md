Audit Report

## Title
`protocolUnpausePool` Sets Pause Level to 1 Instead of 0, Leaving Pool Permanently Admin-Paused After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`MetricOmmPoolFactory.protocolUnpausePool` is intended to restore a protocol-paused pool (level 2) to active status (level 0), but it calls `setPause(1)` instead of `setPause(0)`. After the protocol owner calls `protocolUnpausePool`, the pool transitions from level 2 to level 1 (admin-paused) — still paused. All swaps continue to revert with `PoolPaused`, and the protocol owner has no further path to restore swap functionality without pool admin cooperation.

## Finding Description

`MetricOmmPool` defines three pause levels at [1](#0-0)  — 0 = active, 1 = admin-paused, 2 = protocol-paused.

`_checkNotPaused()` reverts on any non-zero level: [2](#0-1) 

`swap` applies `whenNotPaused` unconditionally: [3](#0-2) 

`protocolUnpausePool` correctly guards `cur != 2` but then calls `setPause(1)` instead of `setPause(0)`: [4](#0-3) 

After this call, `pauseLevel == 1 != 0`, so `_checkNotPaused()` still reverts. The admin-side `unpausePool` enforces `cur == 1 → 0` and is the only remaining path to level 0: [5](#0-4) 

The error in `protocolUnpausePool` also propagates to the revert message: `revert InvalidPauseTransition(cur, 1)` on line 401 signals the target is 1, consistent with the wrong `setPause(1)` call, confirming this is a systematic mistake rather than a typo in a single argument.

## Impact Explanation

Every invocation of `protocolUnpausePool` leaves the pool at level 1 (admin-paused). `swap` reverts with `PoolPaused` for all traders. If the pool was at level 0 before the protocol pause, the pool admin never issued a pause and may not know they must call `unpausePool`. If the admin is unresponsive or the admin key is unavailable, the pool is permanently stuck at level 1 with all swap volume and fee accrual lost. This constitutes broken core pool swap functionality — an allowed impact under the contest rules.

## Likelihood Explanation

The bug fires unconditionally on every call to `protocolUnpausePool`. No special preconditions, no attacker, no race condition. Every protocol-pause/unpause governance cycle triggers it. The protocol owner acting in good faith is sufficient.

## Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`, and update the revert argument to match:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

## Proof of Concept

1. Pool is active: `pauseLevel == 0`.
2. Protocol owner calls `protocolPausePool(pool)` → `setPause(2)` → `pauseLevel == 2`. Swaps revert.
3. Protocol owner calls `protocolUnpausePool(pool)`: reads `cur == 2`, passes guard, calls `setPause(1)` → `pauseLevel == 1`.
4. Any trader calls `swap(...)` → `whenNotPaused` → `_checkNotPaused()` → `pauseLevel != 0` → **`PoolPaused` revert**.
5. Pool admin must separately call `unpausePool(pool)` (checks `cur == 1`) → `setPause(0)` to restore swaps.
6. If admin is absent, pool remains permanently paused at level 1 despite the protocol owner having called "unpause."

Foundry test outline:
```solidity
factory.protocolPausePool(pool);
factory.protocolUnpausePool(pool);
// pauseLevel is now 1, not 0
vm.expectRevert(PoolPaused.selector);
pool.swap(recipient, true, 1e18, priceLimit, "", "");
```
