### Title
`verify_foreign_transaction` Accepts Zero-Confirmation Bitcoin Requests, Enabling Double-Spend Attestation — (File: `crates/contract/src/lib.rs`, `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` contract method accepts a `BitcoinRpcRequest` with a caller-controlled `confirmations: BlockConfirmations` field that has no on-chain minimum enforced. The MPC nodes' Bitcoin inspector treats a threshold of `0` as always satisfied, causing nodes to sign and attest unconfirmed (mempool) Bitcoin transactions. This is the direct analog of the TransmuterBuffer `depositUnderlying(amount, 0)` pattern: a critical operation accepts a "safety bound" parameter that the caller can set to zero, bypassing the intended protection entirely.

---

### Finding Description

`verify_foreign_transaction` in `crates/contract/src/lib.rs` performs four precondition checks — domain existence, gas, deposit, and `accept_requests` — but performs **no validation of the chain-specific request parameters** before enqueuing the request for MPC nodes to process. [1](#0-0) 

For Bitcoin, the request carries a `confirmations: BlockConfirmations` field, typed as a bare `u64` with `minimum: 0.0` in the ABI schema. [2](#0-1) 

The MPC node's Bitcoin inspector enforces finality with the following check:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
``` [3](#0-2) 

When `block_confirmations_threshold = 0`, the condition `0 <= transaction_block_confirmation` is trivially true for any `u64` value, including 0. The check is completely bypassed. The node then proceeds to extract values and sign the payload, producing a valid MPC attestation for a transaction that may be unconfirmed or in the mempool.

The `confirmations` value flows from the user's request directly into the inspector call without any transformation or floor enforcement: [4](#0-3) 

---

### Impact Explanation

The primary use case of `verify_foreign_transaction` is the Omnibridge inbound flow: a bridge contract on NEAR releases funds only after the MPC network attests that a Bitcoin transaction finalized. [5](#0-4) 

If an attacker submits a `verify_foreign_transaction` request with `confirmations: 0`, the MPC network will sign and attest a Bitcoin transaction that has not been included in any block (or has only 0 confirmations). The attacker can then:

1. Broadcast a Bitcoin transaction to the mempool.
2. Immediately call `verify_foreign_transaction` with `confirmations: 0`.
3. Receive a valid MPC signature attesting the transaction "finalized."
4. Use that attestation to claim bridge funds on NEAR.
5. Double-spend the Bitcoin transaction via RBF or by never broadcasting it to miners.

This constitutes a **forged foreign-chain verification** and **double-spend condition**, matching the High impact category.

---

### Likelihood Explanation

The entry path requires only a 1 yoctoNEAR deposit and is callable by any unprivileged NEAR account. No special role, key, or collusion is required. The `BitcoinRpcRequest` struct is fully user-controlled, and `confirmations: 0` is a valid serialized value accepted by the contract without rejection. Any user interacting with a bridge built on this system can exploit this. [6](#0-5) 

---

### Recommendation

Enforce a minimum confirmation threshold on-chain inside `verify_foreign_transaction` before enqueuing the request. The minimum should be a governance-controlled parameter stored in `Config` or in the per-chain foreign-chain configuration, similar to how `sign_call_gas_attachment_requirement_tera_gas` is stored.

```rust
// Example: enforce minimum confirmations for Bitcoin
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    let min_confirmations = self.config.bitcoin_min_confirmations; // e.g., 6
    if btc_req.confirmations.0 < min_confirmations {
        env::panic_str("Bitcoin confirmations below required minimum");
    }
}
```

This mirrors the pattern already used for gas and deposit bounds in `check_request_preconditions`. [7](#0-6) 

---

### Proof of Concept

1. Deploy or use the production MPC contract with a `ForeignTx` domain configured for Bitcoin.
2. Call `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "domain_id": <foreign_tx_domain_id>,
       "payload_version": 1,
       "request": {
         "Bitcoin": {
           "tx_id": "<mempool_tx_id_hex>",
           "confirmations": 0,
           "extractors": ["BlockHash"]
         }
       }
     }
   }
   ```
   with `attached_deposit: 1 yoctoNEAR`.
3. The contract enqueues the request without rejecting `confirmations: 0`.
4. MPC nodes query `getrawtransaction` for the mempool transaction. The inspector evaluates `0 <= rpc_response.confirmations`, which is always true.
5. Nodes sign the payload and call `respond_verify_foreign_tx`.
6. The caller receives a valid `VerifyForeignTransactionResponse` with a signature attesting the unconfirmed transaction.
7. The attacker presents this attestation to a bridge contract to claim funds, then double-spends the Bitcoin transaction. [8](#0-7)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    async fn extract(
        &self,
        transaction: BitcoinTransactionHash,
        block_confirmations_threshold: BlockConfirmations,
        extractors: Vec<BitcoinExtractor>,
    ) -> Result<Vec<BitcoinExtractedValue>, ForeignChainInspectionError> {
        let request_parameters = GetRawTransactionArgs {
            transaction_hash: TransportBitcoinTransactionHash::from(*transaction),
            verbose: VERBOSE_RESPONSE,
        };

        // TODO(#1978): add retry mechanism if the error from the request is transient
        let rpc_response: GetRawTransactionVerboseResponse = self
            .client
            .request(GET_RAW_TRANSACTION_METHOD, &request_parameters)
            .await?;

        let transaction_block_confirmation = rpc_response.confirmations.into();
        let enough_block_confirmations =
            block_confirmations_threshold <= transaction_block_confirmation;

        if !enough_block_confirmations {
            return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations {
                expected: block_confirmations_threshold,
                got: transaction_block_confirmation,
            });
        }

        self.verify_block_is_canonical(rpc_response.blockhash)
            .await?;

        let extracted_values = extractors
            .iter()
            .map(|extractor| extractor.extract_value(&rpc_response))
            .collect();

        Ok(extracted_values)
    }
