I need to trace the exact execution path through `conclude_node_migration` and `reverify_participants` to determine if the account-ownership check is actually enforced.

### Title
Missing Attestation Account-Ownership Check in `conclude_node_migration` Allows TLS Key Collision — (`crates/contract/src/lib.rs`)

### Summary

`conclude_node_migration` calls `reverify_participants` with a `NodeId` that contains the caller's `account_id` but the destination's `tls_public_key`. `reverify_participants` looks up the stored attestation **only by TLS key** and never verifies that the stored attestation's `account_id` matches the caller's `account_id`. This lets any existing participant P1 pass the TEE gate using P2's pre-existing attestation, then write P2's TLS key into P1's own `ParticipantInfo`, breaking the uniqueness invariant.

---

### Finding Description

**Entrypoint — `start_node_migration`**

P1 (an existing participant) calls `start_node_migration` with `destination_node_info.tls_public_key = K2` (P2's TLS key) and `signer_account_pk = P1_own_key`. No uniqueness check on the destination TLS key is performed. [1](#0-0) 

**TEE gate in `conclude_node_migration`**

The function constructs a `NodeId` with P1's `account_id` and K2 as the `tls_public_key`, then calls `reverify_participants`: [2](#0-1) 

**Root cause — `reverify_participants` ignores `account_id`**

The function looks up the stored attestation **only by `tls_public_key`**. It finds P2's valid attestation (stored under K2) and returns `TeeQuoteStatus::Valid` without ever checking that `participant_attestation.node_id.account_id == node_id.account_id`: [3](#0-2) 

Compare this with `add_participant`, which **does** enforce ownership: [4](#0-3) 

**`update_info` has no TLS key uniqueness check**

After the TEE gate passes, `update_info` only matches on `account_id` and blindly overwrites P1's `ParticipantInfo` with `{tls_public_key: K2, ...}`: [5](#0-4) 

The result: both P1 and P2 have `tls_public_key = K2` in the live participant list.

---

### Impact Explanation

- The `tls_public_key` uniqueness invariant across participants is broken.
- P1 does **not** control K2's private key (P2 does), so P1 becomes unreachable via TLS — effectively a self-inflicted participant-state corruption.
- P2P routing is ambiguous: two participants advertise K2, causing honest nodes to be unable to deterministically route protocol messages.
- Signing rounds requiring P1's participation will fail (liveness degradation).
- This does **not** directly enable unauthorized threshold signature issuance or key-share theft, because P1 gains no access to P2's private TLS key or key share.

**Impact: Medium** — participant-state manipulation that breaks a production safety invariant (TLS key uniqueness) without enabling unauthorized signing.

---

### Likelihood Explanation

Any existing participant can trigger this with two on-chain calls (`start_node_migration` + `conclude_node_migration`). No special privileges, no threshold collusion, and no TEE compromise are required. The only precondition is that the target participant (P2) already has a valid attestation registered — which is the normal operational state.

---

### Recommendation

In `conclude_node_migration`, after `reverify_participants` returns `Valid`, add an explicit ownership check:

```rust
// After reverify_participants returns Valid:
if let Some(stored) = self.tee_state.stored_attestations.get(&node_id.tls_public_key) {
    if stored.node_id.account_id != account_id {
        return Err(errors::InvalidParameters::InvalidTeeRemoteAttestation {
            reason: "destination TLS key attestation belongs to a different account".into(),
        }.into());
    }
}
```

Additionally, `update_info` (or its caller) should assert that the new `tls_public_key` is not already present in any other participant's `ParticipantInfo`.
