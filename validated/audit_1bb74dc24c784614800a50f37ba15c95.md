### Title
Pre-funded PDA Blocks Wrapped Token Deployment Permanently - (File: `solana/programs/bridge_token_factory/src/instructions/user/deploy_token.rs`)

### Summary
The `DeployToken` instruction uses Anchor's `init` constraint to create the wrapped mint PDA. Anchor's `init` internally calls `system_instruction::create_account`, which fails if the target account already has any lamports. An unprivileged attacker can pre-calculate the deterministic mint PDA and transfer as little as 1 lamport to it, permanently blocking deployment of that wrapped token on Solana and freezing the redemption path for all users who bridge that token from NEAR.

### Finding Description
In `deploy_token.rs`, the `mint` account is declared with the `init` constraint: [1](#0-0) 

The PDA seeds are `[WRAPPED_MINT_SEED, data.payload.token.to_hashed_bytes()]`, which are fully deterministic and publicly derivable from the token identifier in the cross-chain payload. Anchor's `init` constraint calls `system_instruction::create_account` under the hood. Per Solana system program rules, `create_account` fails with "account already in use" if the target address already holds any lamports — even 1 lamport — regardless of whether it has data or an assigned owner.

Because the lamports sent to the bare system-owned PDA have no authorized spender (the account has no program owner yet), they cannot be reclaimed. The DoS is permanent with a single one-time transaction costing the attacker a negligible amount of SOL.

The `used_nonces` accounts in `finalize_transfer.rs` and `finalize_transfer_sol.rs` use `init_if_needed` and are not affected: [2](#0-1) 

Only the `mint` (in `deploy_token.rs`) and `config` (in `initialize.rs`) accounts use plain `init` and are vulnerable. [3](#0-2) 

### Impact Explanation
If the wrapped mint PDA for a given token is pre-funded, `deploy_token` will always revert. Without the mint, `finalize_transfer` cannot mint tokens to recipients. Any user who has already locked assets on NEAR and is waiting for the Solana-side mint to be deployed will have their funds permanently unclaimable on Solana, with no on-chain refund path triggered by the Solana-side failure. This constitutes an irreversible frozen redemption path for bridge assets, matching the Critical impact tier.

### Likelihood Explanation
The attack requires no privilege. The PDA address is fully deterministic from public inputs (`WRAPPED_MINT_SEED` + hash of the token identifier, which appears in the cross-chain payload). Any observer of the NEAR→Solana message queue can front-run `deploy_token` by sending 1 lamport to the pre-computed address before the relayer submits the instruction. Cost to the attacker is negligible; the DoS is permanent.

### Recommendation
Replace the `init` constraint on `mint` with a manual three-step pattern that tolerates pre-funded accounts:
1. If the account's lamports are below the rent-exempt minimum, transfer the deficit.
2. Call `system_instruction::allocate` via `invoke_signed` to set the data size.
3. Call `system_instruction::assign` via `invoke_signed` to set the program owner.

Alternatively, add a guard that checks whether the account already has lamports and, if so, skips the `create_account` step and proceeds directly to initialization — the same pattern used by Anchor's `init_if_needed` but with an explicit discriminator check to prevent re-initialization of a legitimately deployed mint.

### Proof of Concept
```rust
// Attacker pre-computes the mint PDA for a known token being bridged
let token_id = "usdc.near"; // observed from NEAR→Solana message queue
let hashed = token_id.to_string().to_hashed_bytes(); // same hash as the program
let (mint_pda, _) = Pubkey::find_program_address(
    &[WRAPPED_MINT_SEED, hashed.as_ref()],
    &bridge_program_id,
);

// Attacker sends 1 lamport — permanently blocks deploy_token for this token
let ix = system_instruction::transfer(&attacker.pubkey(), &mint_pda, 1);
// ... sign and submit

// Subsequent deploy_token call by the relayer fails:
// "Create Account: account ... already in use"
// finalize_transfer for this token also fails — user funds are frozen on NEAR side
```
