Audit Report

## Title
Floor Division in `removeLiquidity` Burns LP Shares and Reduces Bin Balances While Returning Zero Native Tokens - (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`LiquidityLib.removeLiquidity` unconditionally burns LP shares and decrements both per-bin and global scaled balances before converting to native token amounts via floor division. For tokens with a large `token0ScaleMultiplier` (e.g., USDC with 6 decimals → multiplier = 10¹²), the scaled amount removed can be non-zero while the native transfer rounds to zero, leaving the LP with burned shares and no tokens. The stranded scaled balance is then swept to protocol/admin fee collectors on the next `collectFees` call.

## Finding Description
**Share burn is unconditional (lines 205–214):** [1](#0-0) 

`amount0Scaled` and `amount1Scaled` are computed via floor division, then `binState.token0BalanceScaled`, `binTotalShares`, and `positionBinShares` are all decremented before any transfer check.

**Global bin totals are decremented unconditionally (lines 230–232):** [2](#0-1) 

**Native transfer uses floor division and is skipped on zero (lines 239–247):** [3](#0-2) 

`_deltasScaledToExternal` with `Math.Rounding.Floor` computes `scaledDeltaAmount0 / ctx.token0ScaleMultiplier`. If `totalToken0ToRemoveScaled < token0ScaleMultiplier` (e.g., 1000 < 10¹²), `amount0Removed = 0` and the `safeTransfer` is skipped entirely. [4](#0-3) 

**Orphaned scaled units surface as surplus in `collectFees` (lines 385–388):** [5](#0-4) 

Because `binTotals.scaledToken0` was decremented but the ERC-20 balance was not reduced (no transfer occurred), `surplus0Scaled` increases by exactly `totalToken0ToRemoveScaled`. This surplus is then split between admin and protocol fee collectors (lines 391–414).

**Asymmetric rounding confirmed:** `addLiquidity` uses `Math.Rounding.Ceil` throughout (lines 87, 89, 109–110, 142), guaranteeing the LP always pays at least 1 native unit when adding but may receive 0 when removing. [6](#0-5) [7](#0-6) 

**`MINIMAL_MINTABLE_LIQUIDITY` guard is insufficient (lines 200–201):** [8](#0-7) 

This guard only prevents leaving a dust *remainder* after removal; it does not prevent removing a quantity of shares whose total scaled value is below `token0ScaleMultiplier`.

## Impact Explanation
Direct, unrecoverable loss of LP principal. The LP's shares are burned and the pool's internal accounting is permanently reduced, but no tokens are transferred. The lost value is silently redirected to fee collectors. For a USDC pool (`token0ScaleMultiplier = 10¹²`), any removal where `totalToken0ToRemoveScaled ∈ [1, 10¹² − 1]` produces `amount0Removed = 0`. Combined with the asymmetric rounding on `addLiquidity`, an LP who adds a small position pays at least 1 USDC but recovers 0 USDC on removal — a 100% loss of principal. This meets the Critical/High threshold for direct loss of LP assets.

## Likelihood Explanation
No privileged access is required. Any LP in a pool with a sub-18-decimal token (USDC, USDT, WBTC — all explicitly in scope) can trigger this by calling `removeLiquidity` with a small share count. The condition is reachable without any malicious setup: the asymmetric rounding in `addLiquidity` (ceil) vs. `removeLiquidity` (floor) guarantees the scenario is reachable for any LP who adds a minimal position. The attack is repeatable and requires no special permissions.

## Recommendation
1. **Revert on zero-output removal**: After computing `amount0Removed` and `amount1Removed`, add:
   ```solidity
   if (totalToken0ToRemoveScaled > 0 && amount0Removed == 0) revert ZeroTokenOutput();
   if (totalToken1ToRemoveScaled > 0 && amount1Removed == 0) revert ZeroTokenOutput();
   ```
2. **Alternatively**, expose `minAmount0Out` / `minAmount1Out` parameters so callers can enforce a minimum output.
3. **For the share-to-scaled step**, consider using `Math.ceilDiv` symmetrically with `addLiquidity` so the LP always recovers at least 1 scaled unit per share removed.

## Proof of Concept
**Setup**: USDC/ETH pool, `token0 = USDC` (6 decimals), `token0ScaleMultiplier = 10¹²`, `initialScaledToken0PerShareE18 = 1e18` (maps to 1 USDC per share), `MINIMAL_MINTABLE_LIQUIDITY = 1000`.

**Add liquidity** (1000 shares, empty bin):
```
amount0Scaled = ceilDiv(1e18 × 1000, 1e18) = 1000   // scaled units credited to bin
amount0Added  = ceilDiv(1000, 1e12)         = 1       // 1 USDC pulled from LP
```

**Remove liquidity** (all 1000 shares):
```
amount0Scaled          = 1000 × 1000 / 1000 = 1000   // bin balance decremented
binTotals.scaledToken0 -= 1000                        // global accounting decremented
amount0Removed         = 1000 / 1e12        = 0       // safeTransfer skipped
```

LP paid 1 USDC, received 0 USDC. The 1000 scaled units remain in the pool's ERC-20 balance but are no longer tracked in `binTotals.scaledToken0`, causing them to appear as `surplus0Scaled` in the next `collectFees` call, where they are distributed to admin and protocol fee collectors.
