### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Single Byzantine Node to Forge Foreign-Chain Verification Response - (File: crates/contract/src/lib.rs)

### Summary
`respond_verify_foreign_tx` accepts a `payload_hash` supplied entirely by the responding node and only checks that the provided signature is valid for that hash under the domain root key. It never verifies that `payload_hash` corresponds to the transaction committed in the original `VerifyForeignTransactionRequest`. A single attested participant (Byzantine node below the signing threshold) that has retained a valid signature from any prior `ForeignTx`-domain signing operation can replay it against a new, unrelated pending request, delivering a forged verification response to the waiting caller.

### Finding Description

`verify_foreign_transaction` stores a `VerifyForeignTransactionRequest` (containing the foreign-chain transaction details) as the key in `pending_verify_foreign_tx_requests` and suspends the caller via yield/resume. [1](#0-0) 

When a node calls `respond_verify_foreign_tx`, the contract:

1. Fetches the root public key for the domain.
2. Verifies that `response.signature` is valid for **`response.payload_hash`** — a value supplied entirely by the responding node.
3. Resolves every queued yield for the stored request with the full response. [2](#0-1) 

Crucially, there is no step that checks whether `response.payload_hash` equals the hash that should be derived from the transaction ID embedded in the stored `VerifyForeignTransactionRequest`. The contract only checks internal consistency of the response (signature ↔ hash), not external consistency (hash ↔ requested transaction).

Compare this with the regular `respond` path, which derives the expected public key from the request's own `tweak` field before verifying the signature, binding the response tightly to the original request parameters: [3](#0-2) 

No equivalent binding exists in `respond_verify_foreign_tx`.

### Impact Explanation

A single Byzantine attested participant that has retained a valid `(payload_hash_A, signature_A)` pair from any prior `ForeignTx`-domain operation can call `respond_verify_foreign_tx` for a completely different pending request (for transaction T_B) while supplying `payload_hash = H_A` and `signature = sig_A`. The contract accepts the call (the signature is valid for H_A under the domain key), drains the yield queue for request T_B, and delivers the forged response to the waiting caller. The caller receives a `VerifyForeignTransactionResponse` asserting that T_B was verified, but the embedded `payload_hash` actually corresponds to a different transaction T_A. Any downstream bridge logic that trusts this response to authorize a cross-chain action (e.g., releasing funds, minting tokens) will execute based on a forged verification — a direct forged foreign-chain verification enabling invalid bridge execution.

This matches the allowed High impact: **"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."**

### Likelihood Explanation

- Requires only **one** Byzantine attested participant — strictly below the signing threshold.
- The attacker needs a previously produced `(H, sig)` pair for the `ForeignTx` domain. Any node that has ever participated in a completed `verify_foreign_transaction` round retains the final threshold signature as part of normal protocol operation.
- No special privileges, no key leakage, no TEE break, and no network-level DoS are required.
- The attack is silent: the contract emits no event distinguishing a legitimate response from a replayed one.
- Likelihood is **Medium-High**: any single compromised or malicious node in the participant set can execute this after the first successful `verify_foreign_transaction` round completes.

### Recommendation

Before accepting the response, derive the expected `payload_hash` from the transaction identifier stored in the `VerifyForeignTransactionRequest` and assert equality with `response.payload_hash`:

```rust
let expected_hash = compute_payload_hash_from_request(&request);
if response.payload_hash.0 != expected_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

The derivation function must mirror exactly what the MPC nodes compute off-chain (e.g., `SHA-256(tx_id)`). This binds the response to the original request, eliminating the substitution vector.

### Proof of Concept

**Setup:**
- MPC network has a `ForeignTx` domain with root key `K`.
- Honest round: user submits `verify_foreign_transaction` for transaction T_A. The MPC network signs `H_A = hash(T_A.tx_id)`, producing `sig_A`. Malicious node M retains `(H_A, sig_A)`.

**Attack:**
1. User submits `verify_foreign_transaction` for transaction T_B (a high-value bridge unlock). The contract stores `request_B` in `pending_verify_foreign_tx_requests` and suspends the caller.
2. Malicious node M calls:
   ```
   respond_verify_foreign_tx(
       request  = request_B,          // correct stored key — passes map lookup
       response = { payload_hash: H_A, signature: sig_A }  // replayed from T_A
   )
   ```
3. Contract checks: `verify_ecdsa_signature(sig_A, H_A, K)` → **valid** (sig_A was honestly produced for H_A).
4. `resolve_yields_for` drains the yield queue for `request_B`, delivering `{ payload_hash: H_A, signature: sig_A }` to the caller.
5. Caller's bridge contract receives a response asserting T_B was verified, but `payload_hash = H_A ≠ hash(T_B.tx_id)`. If the bridge contract trusts the MPC signature without re-deriving the expected hash, it executes the unlock for T_B based on a forged verification. [4](#0-3)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
