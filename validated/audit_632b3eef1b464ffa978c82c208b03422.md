Audit Report

## Title
L2 Pool `deposit()` Functions Lack Minimum rsETH Output Protection — (File: `contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`)

## Summary

All five L2 pool contracts expose `deposit()` entry points that compute the rsETH output at execution time using a live oracle rate, but accept no caller-supplied minimum output floor. When the rsETH/ETH oracle rate rises between transaction submission and on-chain execution, the user silently receives fewer rsETH units than anticipated with no revert path. The L1 `LRTDepositPool` enforces an identical protection via `minRSETHAmountExpected`, confirming the design intent was present but omitted from all L2 variants.

## Finding Description

Every L2 pool `deposit()` function computes rsETH output via `viewSwapRsETHAmountAndFee`, which divides by the live oracle rate at execution time:

```solidity
// RSETHPool.sol L319, RSETHPoolV3.sol L307, RSETHPoolNoWrapper.sol L285
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

No floor check on `rsETHAmount` follows this computation in any of the five contracts:

- `RSETHPool.deposit` (L265, L284) — transfers `wrsETH` directly from pool balance
- `RSETHPoolNoWrapper.deposit` (L231, L250) — transfers `rsETH` directly
- `RSETHPoolV3.deposit` (L246, L271) — mints `wrsETH`
- `RSETHPoolV3ExternalBridge.deposit` (L366, L390) — mints `wrsETH`
- `RSETHPoolV3WithNativeChainBridge.deposit` (L282, L307) — mints `wrsETH`

The L1 counterpart enforces a caller-supplied floor in `_beforeDeposit`:

```solidity
// LRTDepositPool.sol L667-669
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

The rsETH/ETH oracle rate increases monotonically as EigenLayer restaking rewards accrue. Any gap between transaction submission and execution — widened by L2 block times, sequencer congestion, or mempool delays — allows the rate to drift upward, reducing the rsETH output below the user's off-chain estimate. The transaction succeeds and emits `SwapOccurred` regardless of the magnitude of the shortfall.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but does not lose value.**

The deposited ETH is not stolen; the ETH-equivalent value of the received rsETH equals the deposit at the execution-time rate. However, the user receives fewer rsETH units than they calculated off-chain, with no on-chain mechanism to reject the outcome. This is concretely harmful when the user requires a specific rsETH quantity — for example, to meet a collateral threshold in a downstream DeFi protocol — and silently receives less. This matches the allowed impact: *"Contract fails to deliver promised returns, but doesn't lose value."*

## Likelihood Explanation

No privileged action or attacker is required. The rsETH/ETH oracle rate drifts upward organically and continuously due to yield accrual. On L2 networks with block times differing from L1, or during sequencer congestion, the window between submission and execution is sufficient for a meaningful rate movement. Every deposit by any user on any of the five L2 pools is exposed on every call.

## Recommendation

Add a `minRsETHAmount` parameter to both `deposit` overloads in all five L2 pool contracts, mirroring the L1 pattern:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same pattern to the token-path overload `deposit(address token, uint256 amount, string memory referralId, uint256 minRsETHAmount)` in all five contracts.

## Proof of Concept

1. Oracle rate is `1.10 ETH/rsETH`. User previews `deposit{value: 1 ether}("ref")` on `RSETHPoolV3`, expecting `≈ 0.909 rsETH` (`1e18 * 1e18 / 1.1e18`).
2. Before the transaction is mined, the oracle rate updates to `1.20 ETH/rsETH` via organic yield accrual — no attacker required.
3. `viewSwapRsETHAmountAndFee(1 ether)` executes at the new rate: `rsETHAmount = 1e18 * 1e18 / 1.2e18 = 0.833e18`.
4. `wrsETH.mint(msg.sender, 0.833e18)` executes; `SwapOccurred` is emitted. The user receives `0.833 rsETH` — `0.076 rsETH` less than expected — with no revert.
5. The user has no on-chain mechanism to reject this outcome.

**Foundry fork test plan:** Deploy a fork of the target L2, call `getRate()` to record the current oracle rate, advance the block timestamp to simulate yield accrual triggering a rate update, then call `deposit{value: 1 ether}("ref")` and assert that the received rsETH balance is less than `1e18 * 1e18 / initialRate`, confirming the shortfall with no revert.
