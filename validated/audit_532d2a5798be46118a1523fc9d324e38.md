Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks LP Recipient (`owner`) Instead of Actual Payer (`sender`), Allowing Blocked Depositors to Bypass the Allowlist - (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual `msg.sender` of the pool call and token payer) and gates only on `owner` (the LP position recipient). Because `MetricOmmPool.addLiquidity` supports an operator pattern where `msg.sender != owner`, any address excluded from the allowlist can deposit tokens into the pool by specifying an allowlisted address as `owner`, fully circumventing the compliance gate.

## Finding Description
`MetricOmmPool.addLiquidity` passes `msg.sender` as `sender` and the caller-supplied `owner` as the position recipient to the extension hook: [1](#0-0) 

`DepositAllowlistExtension.beforeAddLiquidity` leaves the first parameter (`sender`) unnamed and unused, checking only `owner`: [2](#0-1) 

The `allowedDepositor` mapping is keyed by `depositor` and the admin setter is `setAllowedToDeposit(pool, depositor, allowed)`, making the intent clear — the depositor (payer) should be gated, not the recipient: [3](#0-2) 

A blocked address `A` calls `pool.addLiquidity(owner = B, ...)` where `B` is allowlisted. The extension receives `sender = A` (ignored) and `owner = B` (allowed), so `allowedDepositor[pool][B] == true` passes. `A` pays tokens via the `metricOmmModifyLiquidityCallback`; `B` receives LP shares. Since `removeLiquidity` enforces `msg.sender == owner`: [4](#0-3) 

`B` (controlled by `A`) calls `removeLiquidity` and returns tokens to `A`. The allowlist is fully circumvented.

By contrast, `SwapAllowlistExtension.beforeSwap` correctly checks `sender`: [5](#0-4) 

## Impact Explanation
The `DepositAllowlistExtension` is the primary on-chain mechanism for KYC/compliance gating on deposits. With this bug, any address excluded from the allowlist can deposit into the pool, earn LP fees, and withdraw — defeating the entire purpose of the extension. Pools relying on this for regulatory compliance are exposed to unrestricted participation by blocked parties, constituting a broken core access-control mechanism with direct fund-flow impact.

## Likelihood Explanation
Exploitation requires no special privileges, no flash loans, and no complex setup. Any blocked address that controls (or coordinates with) a single allowlisted address can execute the bypass in one transaction. The operator pattern (`msg.sender != owner`) is explicitly supported by the pool interface, making the attack surface wide and the exploit trivial and repeatable.

## Recommendation
Check `sender` (the actual caller/payer) instead of `owner` in `beforeAddLiquidity`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

## Proof of Concept
1. Deploy a pool with `DepositAllowlistExtension` configured for `beforeAddLiquidity`.
2. Pool admin calls `setAllowedToDeposit(pool, B, true)` — only `B` is allowed; `A` is not.
3. `A` (blocked) calls `pool.addLiquidity(owner = B, salt, deltas, callbackData, "")`.
4. Extension receives `sender = A` (ignored), `owner = B` (allowed) → `allowedDepositor[pool][B] == true` → passes.
5. `A` pays tokens via `metricOmmModifyLiquidityCallback`; `B` receives LP shares.
6. `B` (controlled by `A`) calls `pool.removeLiquidity(owner = B, ...)` → tokens returned to `B`, forwarded to `A`.
7. `A` has effectively deposited and withdrawn from the pool despite being blocked.
