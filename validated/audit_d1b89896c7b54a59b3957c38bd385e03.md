### Title
Missing Pre-Transfer Normalization Guard Causes Irreversible Fund Lock in `sign_transfer` - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer_internal` accepts and permanently locks/burns a user's tokens without validating that the decimal-normalized transfer amount is non-zero. When `sign_transfer` is later called, `normalize_amount` floors the amount to zero and the `require!(amount_to_transfer > 0)` guard panics unconditionally. Because no cancel or rescue path exists, the user's funds are permanently unclaimable.

### Finding Description

The NEAR bridge's outbound flow has two separate stages that apply decimal normalization inconsistently.

**Stage 1 — `init_transfer_internal` (entry, no normalization check):**

`init_transfer_internal` stores the transfer in `pending_transfers`, burns bridge tokens or locks native tokens, and returns `U128(0)` to the NEP-141 callback, signalling that all tokens are kept by the bridge. [1](#0-0) 

The only fee-related guard at entry is `fee < amount`; there is no check that `normalize_amount(amount - fee, decimals) > 0`. [2](#0-1) 

**Stage 2 — `sign_transfer` (completion, strict normalization check):**

`sign_transfer` calls `normalize_amount` on `amount_without_fee()` and then unconditionally panics if the result is zero:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [3](#0-2) 

**`normalize_amount` uses floor division:**

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [4](#0-3) 

For any token where `origin_decimals > decimals` (e.g., a NEAR token with 24 origin decimals bridged to an EVM chain with 6 decimals, giving a divisor of `10^18`), any transfer of fewer than `10^(origin_decimals - decimals)` base units produces `amount_to_transfer = 0`. The `require!` then panics on every subsequent `sign_transfer` call for that transfer ID.

**No recovery path exists.** There is no public `cancel_transfer`, admin rescue, or any other mechanism that removes a stuck entry from `pending_transfers` and returns the tokens. The only internal removal paths (`remove_transfer_message`, `remove_transfer_message_without_refund`) are called only from within the normal completion flow, which is permanently blocked. [5](#0-4) 

The `sign_transfer_callback` only removes the transfer when the fee is zero and the MPC signing succeeds — a branch that is never reached: [6](#0-5) 

### Impact Explanation

Any user who initiates an outbound NEAR→EVM transfer with an amount smaller than `10^(origin_decimals - destination_decimals)` base units will have their tokens permanently burned or locked inside the bridge contract with no redemption path. This satisfies the **Critical** impact category: *Irreversible fund lock / permanently unclaimable user value in bridge flows*.

The protocol's own comment acknowledges the floor-division behaviour but frames it as "dust" (sub-unit remainder): [7](#0-6) 

The case where the *entire* transfer amount normalizes to zero is categorically different from dust: the user's full deposited amount is lost, not a fractional remainder.

### Likelihood Explanation

**Low–Medium.** The condition requires a token registered with `origin_decimals > decimals` (common for NEAR-native tokens bridged to EVM chains with fewer decimals) and a user sending a sub-threshold amount. The threshold can be as large as `10^18` base units (e.g., 1 full token on the destination chain), making accidental triggering plausible for users sending small test or micro-payment amounts.

### Recommendation

Add a normalization guard at the entry point in `init_transfer` (or `init_transfer_internal`) before tokens are burned/locked:

```rust
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the existing guard in `sign_transfer` but fires before tokens are taken from the user, allowing the NEP-141 callback to return the full amount as a refund instead of locking it permanently.

### Proof of Concept

1. Register a NEAR token with `origin_decimals = 24`, `decimals = 6` (divisor = `10^18`).
2. User calls `ft_transfer_call` with `amount = 5 * 10^17` (0.5 tokens on NEAR, below the 1-token threshold on EVM), `fee = 0`.
3. `init_transfer_internal` stores the transfer, burns the tokens, returns `U128(0)` — tokens are gone.
4. Relayer calls `sign_transfer` for the stored `transfer_id`.
5. `normalize_amount(5 * 10^17, {origin: 24, dest: 6})` = `5 * 10^17 / 10^18` = `0`.
6. `require!(0 > 0, "ERR_INVALID_AMOUNT_TO_TRANSFER")` panics.
7. Step 4–6 repeats on every retry. The transfer stays in `pending_transfers` forever; the user's tokens are unrecoverable. [3](#0-2) [8](#0-7)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
