Audit Report

## Title
Stale `rsETHPrice` Cache Allows Depositors to Extract Accrued Yield Before Price Update - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.rsETHPrice` is a cached state variable updated only when `updateRSETHPrice()` is explicitly called. Because `depositAsset()`/`depositETH()` mint rsETH using this stale cached price as the denominator while `getAssetPrice()` reads live values, an attacker can deposit during the staleness window to receive excess rsETH, then call the public `updateRSETHPrice()` themselves, and initiate a withdrawal at the now-higher price — extracting yield that belongs to existing rsETH holders.

## Finding Description

**Root cause:** `LRTOracle.rsETHPrice` is a stored state variable updated only on explicit calls to `updateRSETHPrice()`. [1](#0-0) 

The deposit mint formula in `LRTDepositPool.getRsETHAmountToMint()` divides by this stale cached value: [2](#0-1) 

`getAssetPrice(asset)` reads a live Chainlink price, while `lrtOracle.rsETHPrice()` returns the stale cached value. As EigenLayer rewards accrue between `updateRSETHPrice()` calls, the true rsETH value rises above `rsETHPrice`. During this window the denominator is artificially low, so depositors receive more rsETH than fair value.

**Exploit path:**

1. **Deposit at stale price.** Attacker calls `depositAsset()` while `rsETHPrice` is stale (e.g., 1.000 ETH when true value is 1.010 ETH). They receive `amount * assetPrice / 1.000` rsETH instead of the fair `amount * assetPrice / 1.010`.

2. **Trigger price update.** Attacker calls the public `updateRSETHPrice()`. `_updateRsETHPrice()` computes the true price from live Chainlink data and updates `rsETHPrice` to 1.010 ETH. [3](#0-2) 

3. **Initiate withdrawal at updated price.** Attacker calls `initiateWithdrawal()`. `getExpectedAssetAmount()` now uses the updated `rsETHPrice = 1.010`, locking in a higher `expectedAssetAmount` in the withdrawal request struct. [4](#0-3) 

4. **Operator calls `unlockQueue()` (normal operations).** `_calculatePayoutAmount()` returns `min(expectedAssetAmount, currentReturn)`. Since the price is stable, both values are equal and the attacker receives the full inflated `expectedAssetAmount`. [5](#0-4) 

5. **Complete withdrawal.** After `withdrawalDelayBlocks`, attacker calls `completeWithdrawal()` and receives more assets than deposited. [6](#0-5) 

**Why existing guards fail:**

The `pricePercentageLimit` check in `_updateRsETHPrice()` only reverts if the price jump per call exceeds the configured threshold for non-managers. [7](#0-6) 

This does not prevent the attack — it only caps the profit per invocation. If rewards accrue slowly (within the limit), the attacker calls `updateRSETHPrice()` themselves. If rewards exceed the limit, the attacker waits for a manager to call `updateRSETHPriceAsManager()` and then immediately initiates withdrawal. Either way, the deposit-at-stale-price step has already occurred.

The `unlockQueue()` operator dependency is not a meaningful barrier: operators call this function routinely as part of normal protocol operations. The attacker does not need to collude with the operator — they simply wait.

## Impact Explanation

**High — Theft of unclaimed yield.** The excess rsETH minted to the attacker dilutes all existing holders' proportional claim on protocol TVL. The profit is extracted directly from yield that should have accrued to existing depositors. The attack is repeatable every reward cycle with no special permissions required beyond a standard deposit.

## Likelihood Explanation

**Medium.** EigenLayer rewards accrue continuously, so a staleness window exists between every pair of `updateRSETHPrice()` calls. `updateRSETHPrice()` is `public`, deposits are open, and the attacker controls the timing of both the deposit and the price update. The `pricePercentageLimit` guard limits per-call profit but does not eliminate the attack surface.

## Recommendation

Call `_updateRsETHPrice()` (or an equivalent internal price refresh) atomically at the start of `depositAsset()` and `depositETH()` before computing `rsethAmountToMint`. This ensures every deposit uses the current, reward-inclusive price and eliminates the staleness window entirely.

## Proof of Concept

**Initial state:**
- `rsETHPrice = 1.000 ETH` (stale; true value is `1.010 ETH` due to accrued EigenLayer rewards)
- stETH/ETH Chainlink price = `1.000`

**Step 1 — Attacker deposits 100 stETH at stale price:**
```
rsETH_minted = (100e18 * 1.000e18) / 1.000e18 = 100 rsETH
Fair amount  = (100e18 * 1.000e18) / 1.010e18 ≈  99.01 rsETH
Excess rsETH = ~0.99 rsETH
```

**Step 2 — Attacker calls `updateRSETHPrice()`:**
- `rsETHPrice` updates to `1.010 ETH` (within `pricePercentageLimit`)

**Step 3 — Attacker calls `initiateWithdrawal(stETH, 100 rsETH)`:**
```
expectedAssetAmount = 100e18 * 1.010e18 / 1.000e18 = 101 stETH  (stored in request)
```

**Step 4 — Operator calls `unlockQueue()` (normal operations):**
```
currentReturn       = 100e18 * 1.010e18 / 1.000e18 = 101 stETH
payoutAmount        = min(101, 101) = 101 stETH
```

**Step 5 — After `withdrawalDelayBlocks`, attacker calls `completeWithdrawal()`:**
- Attacker receives **101 stETH**
- Net profit: **~1 stETH** extracted from existing holders' accrued yield
