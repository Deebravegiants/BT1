### Title
`combine_ac` Accepts Fully Attacker-Crafted Ciphertext and Returns Attacker-Chosen Scalar Due to Unconditional `skip_verify=true` and Attacker-Controlled `Q[j]` in EC Point Check — (`src/cbmpc/api/pve_batch_ac.cpp`, `src/cbmpc/protocol/pve_ac.cpp`)

---

### Summary

`combine_ac` hardcodes `skip_verify=true` when calling `aggregate_to_restore_row`, bypassing the Fiat-Shamir consistency check (`verify()`). Because `Q[j]` is deserialized directly from the attacker-supplied ciphertext blob, the EC point check `x[j] * G != Q[j]` at line 285 of `pve_ac.cpp` is not a security guard — it is trivially satisfied by an attacker who sets `Q[j] = x_target * G`. The attacker can craft a fully consistent ciphertext blob that causes `combine_ac` to output any scalar of their choosing.

### Finding Description

**Serialization: all fields are attacker-controlled**

`ec_pve_ac_t::convert` deserializes `Q`, `L`, `b`, and every `rows[i]` field (`x_bin`, `r`, `c`, `quorum_c`) directly from the blob: [1](#0-0) 

The `ciphertext` argument to `combine_ac` is fully attacker-supplied. After `coinbase::convert(pve_ct, blob.ct)`, every field of `pve_ct` — including `Q[j]` and `b` — is attacker-controlled. [2](#0-1) 

**`combine_ac` unconditionally passes `skip_verify=true`** [3](#0-2) 

Inside `aggregate_to_restore_row`, this flag gates the only call to `verify()`, which is the Fiat-Shamir check that would detect any tampering with `Q`, `b`, `rows[i].r`, `rows[i].c`, or `rows[i].quorum_c`: [4](#0-3) 

**The `bit=0` path uses `row.r.take(16)` as the DRBG seed**

For a row where `b.get_bit(row_index) == 0` (attacker sets `b` freely since `verify` is skipped): [5](#0-4) 

`seed` is the first 16 bytes of the attacker-controlled `row.r`. The DRBG seeded with it generates `x0[j]`, and `x1[j]` comes from AES-GCM decryption of `row.c` using K reconstructed from the attacker-crafted `row.quorum_c`.

**The EC point check is circular when `Q[j]` is attacker-controlled** [6](#0-5) 

`Q[j]` is read from `this->Q`, which was deserialized from the attacker's blob. The check `x[j] * G != Q[j]` provides no security guarantee when the attacker sets `Q[j] = x_target * G`.

**Note on `L`:** `L` is recomputed inside `aggregate_to_restore_row` as `sha256(label, Q)` using the attacker's `Q`, so the attacker can compute the correct `L` for AES-GCM encryption of their crafted `row.c`. [7](#0-6)

### Impact Explanation

An attacker who supplies the ciphertext blob can make `combine_ac` output any scalar `x_target` of their choosing, regardless of what was originally encrypted. In a PVE-based key recovery workflow, this allows substitution of an arbitrary private scalar as the "recovered" value. Honest quorum parties are induced to participate in recovering the attacker's chosen value rather than the legitimate encrypted secret. This breaks the binding property of the PVE scheme: the recovered output is no longer tied to the original plaintext.

---

### Likelihood Explanation

The attack requires only that the attacker control the ciphertext blob passed to `combine_ac` and that honest quorum parties call `partial_decrypt_ac_attempt` on that blob without first calling `verify_ac`. The `combine_ac` API provides no internal verification step and no documentation requiring callers to pre-verify. The attack is fully constructive and requires no cryptographic breaks.

---

### Recommendation

`combine_ac` must not call `aggregate_to_restore_row` with `skip_verify=true` when the ciphertext originates from an untrusted source. Either:

1. Pass `skip_verify=false` and supply `all_ac_pks` in `combine_ac`, so `verify()` is called before any decryption; or
2. Explicitly call `verify_ac` inside `combine_ac` before proceeding to `aggregate_to_restore_row`.

The EC point check `x[j] * G != Q[j]` at line 285 is not a substitute for the Fiat-Shamir check, because `Q[j]` is deserialized from the same untrusted blob.

### Proof of Concept

```
Setup: 2-of-2 threshold AC with parties Alice and Bob, curve secp256k1.

1. Attacker picks x_target (any scalar in [1, q-1]).

2. Attacker picks r_fake (32 random bytes). Let seed = r_fake[0:16].

3. Attacker computes x0 = DRBG_AES_CTR(seed).gen_bn(q).

4. Attacker computes x1 = (x_target - x0) mod q.

5. Attacker picks any K. Computes K shares for the 2-of-2 AC.
   Encrypts each share under Alice's and Bob's public keys → row.quorum_c.

6. Attacker computes Q_fake = x_target * G.
   Computes L_fake = sha256(label, [Q_fake]).
   Encrypts x1_bin under AES-GCM(K, L_fake) → row.c.

7. Attacker sets b such that b.get_bit(attempt_index) == 0.

8. Attacker serializes ec_pve_ac_t with:
     Q = [Q_fake], L = L_fake, b = crafted,
     rows[attempt_index].r = r_fake,
     rows[attempt_index].c = crafted AES-GCM ciphertext,
     rows[attempt_index].quorum_c = crafted K-share ciphertexts.

9. Alice calls partial_decrypt_ac_attempt(malicious_ct, ...) → share_A.
   Bob  calls partial_decrypt_ac_attempt(malicious_ct, ...) → share_B.

10. Combiner calls combine_ac(malicious_ct, attempt_index, label,
        {Alice: share_A, Bob: share_B}, out_xs).

Result: out_xs[0] == x_target. The EC point check passes because
        x_target * G == Q_fake. The Fiat-Shamir check was skipped.
```
