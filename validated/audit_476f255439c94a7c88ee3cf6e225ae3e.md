## Verdict

The vulnerability is **real and exploitable**. The attack path is concrete:

1. Attacker runs a node and gets their IP gossiped into the crawler's peer list (trivial on the P2P network).
2. Crawler initiates an outbound connection to the attacker's node.
3. Attacker sends `NewPeak(height=minimum_height, header_hash=bytes32.zeros, weight=0, ...)`.
4. `Crawler.new_peak` passes the height check with no further validation.
5. `peer_connected_hostname(host, True)` is called → `reliability.update(True, age)` → `tries=1, successes=1`.
6. `is_reliable()` returns `True` on the `tries <= 3 and successes*2 >= tries` branch.
7. `load_reliable_peers_to_db()` writes the attacker's IP to `good_peers`.
8. DNS seed serves the attacker's IP to all bootstrapping nodes.

The invariant that only nodes with a verifiable chain tip at `minimum_height` are included in the DNS seed is violated. A sufficiently scaled attacker (multiple IPs) can dominate the `good_peers` list and eclipse bootstrapping nodes.

### Title
Unvalidated `NewPeak` height allows DNS seed poisoning via fake chain tip — (`chia/seeder/crawler.py`)

### Summary
`Crawler.new_peak` accepts any `NewPeak` message whose `height` field meets `minimum_height` without validating `header_hash`, `weight`, or any actual chain state. A single such message from a crawled peer causes `PeerReliability.is_reliable()` to return `True` (via the `tries=1, successes=1` fast path), resulting in the peer being written to the `good_peers` DNS table and served to all bootstrapping nodes.

### Finding Description
In `chia/seeder/crawler.py`, `Crawler.new_peak` (line 342) checks only `request.height >= self.minimum_height`. No fields of the `NewPeak` message are cryptographically verified. The `header_hash` and `weight` fields are completely ignored. After this single check passes, `crawl_store.peer_connected_hostname(host, True)` is called unconditionally, which invokes `PeerReliability.update(True, age)`, setting `tries=1` and `successes=1`. The `is_reliable()` method's first branch (`tries > 0 and tries <= 3 and successes * 2 >= tries`) then returns `True` immediately. On the next `save_to_db()` call, `load_reliable_peers_to_db()` writes the peer to the `good_peers` SQLite table, from which the DNS seed reads.

### Impact Explanation
Bootstrapping nodes query the DNS seed to find initial peers. If the `good_peers` table is poisoned with attacker-controlled IPs, bootstrapping nodes connect preferentially to attacker nodes. With enough attacker IPs in the list, a bootstrapping node can be fully eclipsed: it receives only attacker-crafted blocks, cannot sync the real chain, and is effectively halted or fed a divergent chain. This satisfies the High/Critical impact criterion of "long-lived inability for honest nodes to process valid blocks."

### Likelihood Explanation
The prerequisite (getting an IP into the crawler's peer list) is trivially met by running any Chia node that gossips the attacker's IP via `RespondPeers`. The crawler will eventually discover and connect to the attacker. The attacker needs only to respond to the inbound connection with a single crafted `NewPeak` message. No keys, admin access, or cryptographic breaks are required.

### Recommendation
In `Crawler.new_peak`, do not treat a `NewPeak` message as proof of reachability at `minimum_height`. The crawler should only call `peer_connected_hostname(host, True)` after independently verifying the peer's chain state — for example, by requesting and validating a block header or weight proof at the claimed height. At minimum, the `weight` field should be checked against a known minimum threshold derived from the expected chain weight at `minimum_height`.
