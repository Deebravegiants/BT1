### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Chain Verification Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the MPC signature is valid over `response.payload_hash`, but never checks that `response.payload_hash` was actually derived from the `request` supplied in the same call. A single Byzantine MPC node (strictly below the signing threshold) can replay any previously accepted `(payload_hash, signature)` pair against a different pending foreign-chain verification request, causing the contract to resolve that request with a forged attestation.

---

### Finding Description

In `respond_verify_foreign_tx`, the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // taken from the response

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← attacker-supplied hash
    &secp_pk,               // ← root MPC key (trusted)
)
.is_ok()
``` [1](#0-0) 

The `payload_hash` is taken verbatim from the caller-supplied `response` struct. The contract never recomputes the expected hash from the `request` argument and compares the two. After the signature check passes, the contract resolves all yields queued under `request` with the raw `response` bytes:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

**Contrast with `respond`** (the regular signing path): there, the message used for verification is taken from `request.payload` — the payload that was committed on-chain when the user submitted the request — not from the response:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(signature_response, payload_hash, &expected_public_key)
``` [3](#0-2) 

The correct binding is also demonstrated in the SDK-side verifier `ForeignChainSignatureVerifier::verify_signature`, which recomputes the expected hash from the request and extracted values and checks `expected_payload_hash == response.payload_hash` before accepting the signature:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [4](#0-3) 

That check exists only in the off-chain SDK helper used by callers; it is absent from the on-chain `respond_verify_foreign_tx` enforcement path.

The `VerifyForeignTransactionRequest` type that serves as the map key contains `request: ForeignChainRpcRequest`, `domain_id`, and `payload_version`, but no hash commitment: [5](#0-4) 

So the map lookup succeeds for any pending `request_B` regardless of what `payload_hash` is in the response.

---

### Impact Explanation

A single Byzantine MPC node (below the signing threshold) can resolve any pending `verify_foreign_transaction` request with a forged attestation. The `VerifyForeignTransactionResponse` delivered to the waiting caller will contain a `payload_hash` that corresponds to a completely different foreign-chain transaction (one that was legitimately signed in the past). Any smart contract that uses this response to authorize a bridge action — e.g., releasing funds after verifying a deposit transaction — will be operating on a fraudulent attestation. This enables invalid bridge execution and potential double-spend conditions.

Impact: **High** — forged foreign-chain verification / cross-chain replay causing invalid bridge execution.

---

### Likelihood Explanation

The attack requires only one compromised or malicious MPC node. The node must have previously participated in a legitimate `respond_verify_foreign_tx` call (to obtain a valid `(payload_hash, signature)` pair) and must observe a new pending request. Both conditions are trivially met in normal operation. No threshold collusion, key leakage, or privileged access is required. The attacker submits a single on-chain transaction.

---

### Recommendation

Inside `respond_verify_foreign_tx`, recompute the expected payload hash from the `request` argument and assert it equals `response.payload_hash` before accepting the signature:

```rust
// Recompute the canonical hash from the request (no extracted values are
// available on-chain, so the node must include them in the request or the
// hash must be committed at request-submission time).
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: /* extracted values committed at request time */,
}).compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::InvalidPayloadHash.into());
}
```

If extracted values are not available on-chain (they are currently only known to the nodes), the fix requires either (a) committing the expected `payload_hash` at `verify_foreign_transaction` submission time and storing it alongside the yield indices, or (b) including the extracted values in the `VerifyForeignTransactionRequest` so the contract can recompute and verify the hash itself.

---

### Proof of Concept

1. User Alice calls `verify_foreign_transaction` for Bitcoin tx `0xAAAA...` → `request_A` is queued.
2. Honest nodes respond; `respond_verify_foreign_tx(request_A, response_A)` is accepted. `response_A.payload_hash` = SHA-256(borsh(`ForeignTxSignPayload{request_A, [BlockHash(0xBBBB...)]}`)).
3. User Bob calls `verify_foreign_transaction` for Bitcoin tx `0xCCCC...` → `request_B` is queued.
4. Byzantine node calls `respond_verify_foreign_tx(request_B, response_A)` — supplying `request_B` as the key but `response_A` (the old, legitimately signed response) as the response.
5. The contract verifies `response_A.signature` over `response_A.payload_hash` against the root MPC key → **passes** (the signature is genuine).
6. `resolve_yields_for` resolves Bob's yield with `response_A` bytes.
7. Bob's contract receives a `VerifyForeignTransactionResponse` attesting that tx `0xAAAA...` (Alice's transaction) was verified — not `0xCCCC...`. If Bob's bridge contract releases funds based on this attestation, it does so for the wrong transaction. [6](#0-5)
