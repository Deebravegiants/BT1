### Title
`validateWithdrawalsDelegated` enforces DRep delegation existence at withdrawal time but not DRep activity, allowing reward withdrawals with an expired DRep — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

The Conway LEDGER rule's `validateWithdrawalsDelegated` check mirrors the structural flaw in the reported vulnerability: a condition (DRep registration) is enforced at **assignment time** (delegation certificate processing), but the analogous liveness condition (DRep non-expiry) is **not enforced at execution time** (reward withdrawal). Any unprivileged user whose delegated DRep has expired can still drain their reward account without re-delegating to an active DRep, bypassing the governance-participation gate that the Conway era introduced for withdrawals.

---

### Finding Description

**Assignment-time check (enforced):**

When a staking credential is delegated to a DRep via `ConwayDelegCert` or `ConwayRegDelegCert`, the `DELEG` rule calls `checkStakeDelegateeRegistered`, which verifies the target DRep is present in `vsDReps`. Only registered (and therefore currently non-expired) DReps can be delegated to at that moment. [1](#0-0) 

**Withdrawal-time check (incomplete):**

At withdrawal time, `validateWithdrawalsDelegated` is called from `conwayLedgerTransitionTRC`: [2](#0-1) 

The function itself only checks that `dRepDelegationAccountStateL` is not `Nothing` — i.e., that *some* DRep delegation record exists in the account state: [3](#0-2) 

It does **not** consult `VState.vsDReps` to verify that the referenced DRep is still active (i.e., `reCurrentEpoch <= drepExpiry drepState`). An expired DRep remains in `vsDReps` with a stale `drepExpiry`; the delegation record in the account state continues to point to it unchanged. Because `validateWithdrawalsDelegated` only tests for `Nothing`, the withdrawal passes.

**Contrast with ratification-time check (correct):**

`dRepAcceptedRatio` in the RATIFY rule correctly skips expired DReps: [4](#0-3) 

The expiry guard exists at ratification but is absent from the withdrawal gate — the same asymmetry as the reported Solidity bug.

**Confirmed by existing test infrastructure:**

A regular (non-disabled) integration test explicitly demonstrates that withdrawal succeeds when the delegated DRep is expired: [5](#0-4) 

A second scenario — where the DRep expires *after* delegation — is marked `disableInConformanceIt` with a reference to formal-ledger-specifications issue #923, explicitly acknowledging that the implementation diverges from the formal specification on this point: [6](#0-5) 

---

### Impact Explanation

The Conway era introduced the DRep-delegation requirement for withdrawals specifically to ensure that ADA holders participate in on-chain governance before accessing their rewards. An expired DRep's stake is excluded from the `dRepAcceptedRatio` denominator and numerator, meaning the delegator's stake no longer contributes to governance ratification. Allowing withdrawals while delegated to an expired DRep lets users extract rewards without their stake being counted in any governance decision — directly circumventing the design parameter that ties reward access to active governance participation. This constitutes **withdrawals outside design parameters** (Medium impact per the allowed scope).

---

### Likelihood Explanation

The path is fully unprivileged and requires no special access:

1. Register a staking credential and accumulate rewards normally.
2. Delegate to any DRep (active at delegation time — passes the assignment-time check).
3. Wait for `ppDRepActivityL` epochs without the DRep voting or receiving a dormant-epoch bump; the DRep expires.
4. Submit a withdrawal transaction. `validateWithdrawalsDelegated` passes because the delegation record is non-`Nothing`; the withdrawal drains the reward account.

No governance majority, no privileged key, no Sybil attack is required. The DRep expiry window (`ppDRepActivityL`, default 20 epochs on mainnet) is a normal protocol event that occurs organically.

---

### Recommendation

Extend `validateWithdrawalsDelegated` to also receive the `VState era` (or the full `CertState era`) and the current epoch number, then add an expiry check analogous to the one in `dRepAcceptedRatio`:

```haskell
validateWithdrawalsDelegated
  :: (EraTx era, ConwayEraCertState era)
  => EpochNo
  -> CertState era
  -> Tx l era
  -> Test (ConwayLedgerPredFailure era)
validateWithdrawalsDelegated curEpoch certState tx =
  let accounts  = certState ^. certDStateL . accountsL
      vsDReps_  = certState ^. certVStateL . vsDRepsL
      wdrls     = unWithdrawals $ tx ^. bodyTxL . withdrawalsTxBodyL
      wdrlsKeyHashes = [ kh | (ra, _) <- Map.toList wdrls
                            , Just kh <- [credKeyHash $ ra ^. accountAddressCredentialL] ]
      isNotActiveDRepDelegated keyHash = isNothing $ do
        accountState <- lookupAccountState (KeyHashObj keyHash) accounts
        drepDel      <- accountState ^. dRepDelegationAccountStateL
        case drepDel of
          DRepCredential cred -> do
            drepState <- Map.lookup cred vsDReps_
            guard (curEpoch <= drepExpiry drepState)
          _ -> Just ()   -- AlwaysAbstain / AlwaysNoConfidence are never "expired"
      nonActiveDelegations = filter isNotActiveDRepDelegated wdrlsKeyHashes
   in failureOnNonEmpty nonActiveDelegations ConwayWdrlNotDelegatedToDRep
```

The call site in `conwayLedgerTransitionTRC` already has `curEpochNo` and `certState` in scope, so the change is localised. [7](#0-6) 

---

### Proof of Concept

The existing test suite provides a direct proof of concept. The test `"Withdraw from a key delegated to an expired DRep"` at line 148–173 of `Test/Cardano/Ledger/Conway/Imp/LedgerSpec.hs`:

1. Sets `ppDRepActivityL = EpochInterval 1`.
2. Registers a DRep and lets it expire (`passNEpochs 4`; `isDRepExpired drep \`shouldReturn\` True`).
3. Delegates the staking credential to the now-expired DRep.
4. Calls `submitTx_` with a withdrawal — **the transaction succeeds**, confirming the bypass. [5](#0-4) 

The `disableInConformanceIt` test immediately following (line 177) further confirms that the formal specification disagrees with this behaviour and that the divergence is tracked but unresolved.
