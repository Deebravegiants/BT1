Audit Report

## Title
Missing Chainlink Price Feed Validation Allows Stale Prices to Corrupt rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and discards all validation fields (`roundId`, `updatedAt`, `answeredInRound`), accepting stale, incomplete, or zero/negative prices without revert. This unvalidated price flows directly into `LRTDepositPool.getRsETHAmountToMint()`, allowing a depositor to receive more rsETH than the true ETH value of their deposit when a Chainlink feed is stale and inflated, diluting all existing rsETH holders. The same codebase already implements the correct three-check pattern in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the omission is an inconsistency, not a design choice.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` silently discards all validation return values from `latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

No check on `answeredInRound >= roundId` (round completeness), `updatedAt != 0` (incomplete round), `price > 0` (valid price), or `block.timestamp - updatedAt <= heartbeat` (staleness).

In contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository performs all three checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated price propagates through the full public deposit path:

1. `LRTDepositPool.depositAsset()` (L99–118) calls `_beforeDeposit()` (L648–670)
2. `_beforeDeposit()` calls `getRsETHAmountToMint()` (L506–521)
3. `getRsETHAmountToMint()` computes `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`
4. `LRTOracle.getAssetPrice()` (L156–158) delegates to `ChainlinkPriceOracle.getAssetPrice()`

`lrtOracle.rsETHPrice()` is a stored value updated separately by `updateRSETHPrice()`. The live `assetPrice` fetch is the only oracle call at deposit time, and it has no guards. The `minRSETHAmountExpected` slippage parameter protects only the depositor, not existing holders.

## Impact Explanation

**High — Theft of unclaimed yield.**

When a Chainlink LST/ETH feed (e.g., stETH/ETH) is stale and its last reported price is higher than the true market price (e.g., feed shows 1.05 ETH/stETH, real price is 1.02 ETH/stETH after a depeg), the minting formula `rsethAmountToMint = (depositAmount × stalePriceHigh) / rsETHPrice` mints excess rsETH to the depositor. This excess dilutes the share of every existing rsETH holder: their tokens now represent a smaller fraction of the pool's true ETH value. The yield that had accrued to existing holders — embedded in the rsETH/ETH exchange rate — is partially transferred to the new depositor. This is a direct, quantifiable theft of unclaimed yield from existing holders, not merely a failure to deliver promised returns.

Additionally, if `price` is zero or negative (e.g., during an extreme feed failure), casting to `uint256` produces zero or an astronomically large value, either minting zero rsETH (DoS for the depositor) or overflowing the minting calculation.

## Likelihood Explanation

Any unprivileged depositor can trigger this by calling `depositAsset()` or `depositETH()` while a supported LST's Chainlink feed is stale. Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for stETH/ETH on mainnet). Staleness occurs naturally during L2 sequencer downtime, network congestion, or extreme market volatility — all documented, real-world events. The protocol supports multiple LST assets, each with its own feed, multiplying the attack surface. No on-chain guard prevents deposits from proceeding with a stale price. The attacker needs no special privileges, no flash loan, and no governance access.

## Recommendation

Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be configurable per asset feed based on the Chainlink heartbeat (e.g., 3600 seconds for a 1-hour heartbeat feed).

## Proof of Concept

**Setup (fork test):**
1. Fork mainnet at a block where the stETH/ETH Chainlink feed last updated > 1 hour ago (or mock the feed to return a stale `updatedAt`).
2. Set `assetPriceFeed[stETH]` to a mock aggregator returning `price = 1.05e18` with `updatedAt = block.timestamp - 7200` (2 hours stale) and `answeredInRound < roundId`.

**Attack sequence:**
1. Record `rsETHPrice` from `LRTOracle`.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `getRsETHAmountToMint` computes `(100e18 × 1.05e18) / rsETHPrice` — attacker receives ~3% more rsETH than the true 1.02 ETH/stETH value warrants.
4. No revert occurs anywhere in the path; `ChainlinkPriceOracle.getAssetPrice()` never inspects `updatedAt` or `answeredInRound`.
5. Existing rsETH holders' share of the pool is diluted by the excess minted rsETH.

**Verification:** Assert that `rsethMinted > (100e18 * 1.02e18) / rsETHPrice` and that the rsETH/ETH price (recomputed via `updateRSETHPrice()`) has decreased relative to pre-deposit, confirming dilution of existing holders.
