### Title
Unverified `compiled_class_hash` in P2P State Diff Sync Corrupts Class Hash Mapping — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

`StateDiffStreamBuilder::parse_data_for_block` accepts `StateDiffChunk::DeclaredClass` chunks from unauthenticated p2p peers and inserts the peer-supplied `compiled_class_hash` directly into `ThinStateDiff::class_hash_to_compiled_class_hash` without any verification against the actual CASM compilation of the Sierra class. The corrupted mapping is then committed to persistent storage via `append_state_diff`. No downstream check ever recomputes or validates the stored hash.

### Finding Description

In `unite_state_diffs`, the `DeclaredClass` arm performs only a duplicate-key check: [1](#0-0) 

The only guard against bad data is `ConflictingStateDiffParts` (duplicate key). A single, non-duplicate `DeclaredClass` chunk with an arbitrary `compiled_class_hash` passes all checks and is accepted.

`parse_data_for_block` validates only `state_diff_length` (entry count) against the stored header: [2](#0-1) 

It never calls `calculate_state_diff_hash` to verify the assembled diff against the `state_diff_commitment` field in the stored header. The header sync itself also never verifies the block hash or the signature cryptographically — it only checks that exactly one signature is present and that block numbers are ordered: [3](#0-2) 

The accepted `(ThinStateDiff, BlockNumber)` is written unconditionally: [4](#0-3) 

`append_state_diff` writes the peer-supplied `compiled_class_hash` directly to the `compiled_class_hash` table: [5](#0-4) [6](#0-5) 

The class sync (`class.rs`) separately syncs Sierra class content but has an explicit TODO acknowledging it does not verify the class hash returned by the class manager against the `compiled_class_hash` stored in the state diff: [7](#0-6) 

### Impact Explanation

The `class_hash_to_compiled_class_hash` mapping stored in the database is the authoritative source for `get_compiled_class_hash` during RPC execution, fee estimation, simulation, and tracing: [8](#0-7) 

A wrong value here means:
1. **Wrong class hash returned by `get_compiled_class_hash` syscall** — any contract that calls this syscall (e.g., for upgrade verification) receives the attacker-controlled value.
2. **State diff commitment mismatch** — `calculate_state_diff_hash` chains `compiled_class_hash` into the Poseidon commitment; the stored commitment diverges from the canonical chain value, breaking proof verification for the affected block.
3. **OS proof failure** — the OS validates `compiled_class_fact.hash == blake_compiled_class_hash(compiled_class)` and would reject the proof because the stored hash does not match the actual CASM. [9](#0-8) 

### Likelihood Explanation

Any unauthenticated p2p peer can trigger this. The p2p sync client connects to peers discovered via the network and accepts state diff chunks from them. No authentication, stake, or privilege is required. The attacker only needs to serve a single well-formed `DeclaredClass` chunk with the correct `class_hash` (observable from the chain) and an arbitrary `compiled_class_hash`.

### Recommendation

After assembling the full `ThinStateDiff` in `parse_data_for_block`, compute `calculate_state_diff_hash(&result)` and compare it against the `state_diff_commitment` stored in the block header. Reject the peer (via `BadPeerError`) if the hashes do not match. This single check covers all fields of the state diff, including `compiled_class_hash`, `deployed_contracts`, `storage_diffs`, and `nonces`.

Additionally, the class sync should verify that the `class_hash` returned by `class_manager_client.add_class()` matches the key in `class_hash_to_compiled_class_hash` (the TODO at line 39 of `class.rs`).

### Proof of Concept

1. Stand up a p2p peer that responds to state diff queries.
2. For a block that declares Sierra class `H` with correct `compiled_class_hash = CORRECT`, serve instead `StateDiffChunk::DeclaredClass { class_hash: H, compiled_class_hash: WRONG }` with the correct `state_diff_length` (1).
3. The syncing node calls `unite_state_diffs` → no error (single entry, no duplicate).
4. `parse_data_for_block` returns `Ok(Some((result, block_number)))`.
5. `write_to_storage` calls `append_state_diff` → `write_compiled_class_hashes` inserts `(H, WRONG)`.
6. Assert: `storage_reader.begin_ro_txn().get_state_diff(block_number).class_hash_to_compiled_class_hash[H] == WRONG`.
7. Assert: `exec_state_reader.get_compiled_class_hash(H) == WRONG`.
8. Assert: `calculate_state_diff_hash(&stored_diff) != header.state_diff_commitment`.

                &pending_data.declared_classes
            {
                if class_hash == *other_class_hash {
                    return Ok(*compiled_class_hash);
                }
            }
        }

        let maybe_block_number = self
            .storage_reader
            .begin_ro_txn()
            .map_err(storage_err_to_state_err)?
            .get_state_reader()
            .map_err(storage_err_to_state_err)?
            .get_class_definition_block_number(&class_hash)
            .map_err(storage_err_to_state_err)?;

        // Cairo 0 classes (and undeclared classes) do not have a compiled class hash.
        // According to the trait, return the default value.
        let Some(block_number) = maybe_block_number else {
            return Ok(CompiledClassHash::default());

            .begin_ro_txn()
            .map_err(storage_err_to_state_err)?
            .get_state_diff(block_number)

                "Inner storage error. Missing state diff at block {block_number}."
            )))?;

                 {class_hash}."
            )))?;

        Ok(*compiled_class_hash)
