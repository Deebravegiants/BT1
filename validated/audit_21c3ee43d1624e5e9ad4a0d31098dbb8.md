### Title
Double-Counting of Deposit Refunds Across Subtransactions Allows ADA Extraction from Deposit Pot - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, `getConsumedDijkstraValue` computes the total consumed value of a top-level transaction by summing the consumed values of the top-level body and all subtransactions, using the **same original-state deposit-lookup functions** for every subtransaction. If two or more subtransactions each include an `UnRegDepositTxCert` for the same staking credential, each subtransaction independently finds the deposit in the lookup and counts it as a refund. The preservation-of-value check (`consumed == produced`) then passes with N×D counted on the consumed side, while only D was ever deposited. The attacker receives N×D in outputs while the deposit pot loses N×D, a net gain of (N−1)×D ADA. The developers are aware of the scenario but have explicitly left the predicate failure unimplemented (`error "TODO: predicate failure not yet implemented"`).

---

### Finding Description

`getConsumedDijkstraValue` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` aggregates consumed value across the top-level body and all subtransactions:

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue :: forall m. TxBody m era -> Value era
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
``` [1](#0-0) 

The critical flaw is that `lookupStakingDeposit` and `lookupDRepDeposit` are **closure-captured from the original ledger state** and are passed unchanged into every subtransaction's consumed-value calculation. There is no mechanism to mark a credential as "already claimed" after the first subtransaction counts its refund. Each subtransaction independently calls `lookupStakingDeposit cred` and receives `Just D`, so each independently adds D to the consumed total.

The preservation-of-value check in the UTXO rule enforces `consumed == produced`. With N subtransactions each claiming refund D, the consumed side totals N×D, so the produced side (outputs + fees + deposits) is also allowed to total N×D. The attacker places N×D in outputs, the check passes, and N×D is removed from the deposit pot — but only D was ever deposited.

The developers have already identified this scenario. The test `"Multiple subtransactions cannot get the same refund"` in `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` is disabled with `xit` and its expected predicate failure is `error "TODO: predicate failure not yet implemented"`:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [2](#0-1) 

This confirms the protection is absent in the current production code path.

The `getConsumedMaryValue` function used per-subtransaction computes refunds via `getTotalRefundsTxBody`, which calls `lookupStakingDeposit` for each `UnRegDepositTxCert` in the subtransaction's certificate list: [3](#0-2) 

The `shelleyTotalRefundsTxCerts` function does track intra-transaction register/deregister pairs to avoid double-counting within a single certificate list, but it has no visibility into other subtransactions: [4](#0-3)

### Impact Explanation

**Critical. Direct loss of ADA through an invalid ledger state transition.**

An attacker registers one staking credential (cost: D ADA deposited). They then submit a single top-level transaction containing N subtransactions, each with `UnRegDepositTxCert cred D`. The consumed-value calculation counts N×D in refunds. The produced-value calculation allows N×D in outputs. The preservation-of-value check passes. The deposit pot loses N×D ADA while only D was ever deposited. The attacker nets (N−1)×D ADA from the deposit pot. With N subtransactions per batch and repeated batches, the deposit pot can be drained to zero, permanently destroying the ADA accounting invariant.

---

### Likelihood Explanation

The attack requires only an unprivileged transaction sender. No special keys, governance majority, or privileged access is needed. The attacker needs only to:
1. Register a staking credential (standard operation).
2. Construct a top-level Dijkstra transaction with multiple subtransactions sharing the same deregistration certificate.

The Dijkstra era is the next production era. The vulnerability is in the core consumed-value computation, which is exercised by every transaction containing subtransactions with deregistration certificates. The developers have already written the test case that demonstrates the attack, confirming it is a known gap.

---

### Recommendation

In `getConsumedDijkstraValue`, replace the static `lookupStakingDeposit` and `lookupDRepDeposit` closures with a stateful accumulator that tracks which credentials have already been claimed as refunds across subtransactions. Specifically:

1. Before iterating over subtransactions, collect the set of all credentials being deregistered across all subtransactions.
2. Enforce that no credential appears in more than one subtransaction's deregistration certificates (add a new predicate failure for this case, resolving the `TODO` in `CertSpec.hs`).
3. Alternatively, thread a "consumed credentials" set through `subTransactionsConsumedValue` and zero out `lookupStakingDeposit`/`lookupDRepDeposit` for any credential already counted by a prior subtransaction.

The fix must be applied before the Dijkstra era is activated on mainnet.

---

### Proof of Concept

**Setup:** Protocol parameter `keyDeposit = D`. Attacker registers `cred` paying D ADA.

**Attack transaction:**
```
TopTx {
  subTransactions = [
    SubTx1 { certs = [UnRegDepositTxCert cred D], inputs = [utxo1] },
    SubTx2 { certs = [UnRegDepositTxCert cred D], inputs = [utxo2] }
  ]
}
```

**Consumed-value computation (`getConsumedDijkstraValue`):**
- `txBodyConsumedValue topTxBody` = 0 (empty top-level body)
- `getConsumedValue ... SubTx1.body`:
  - `lookupStakingDeposit cred` → `Just D` (original state, cred still registered)
  - refunds = D; total for SubTx1 = value(utxo1) + D
- `getConsumedValue ... SubTx2.body`:
  - `lookupStakingDeposit cred` → `Just D` (same original state, unchanged)
  - refunds = D; total for SubTx2 = value(utxo2) + D
- **Total consumed = value(utxo1) + value(utxo2) + 2D**

**Produced-value computation:**
- Attacker sets outputs = value(utxo1) + value(utxo2) + 2D (minus fees)
- `consumed == produced` ✓ — check passes

**Ledger state after:**
- `cred` is deregistered (only one deregistration is actually applied)
- Deposit pot decreases by 2D
- Attacker's outputs contain 2D worth of refund
- **Net gain: D ADA extracted from deposit pot** [7](#0-6) [2](#0-1)

shelleyTotalRefundsTxCerts pp lookupDeposit = snd . F.foldl' accum (mempty, Coin 0)
  where
    keyDeposit = pp ^. ppKeyDepositL
    accum (!regCreds, !totalRefunds) cert =
      case lookupRegStakeTxCert cert of
        Just k ->
          -- Need to track new delegations in case that the same key is later deregistered in
          -- the same transaction.
          (Set.insert k regCreds, totalRefunds)
        Nothing ->
          case lookupUnRegStakeTxCert cert of
            Just cred
              -- We first check if there was already a registration certificate in this
              -- transaction.
              | Set.member cred regCreds -> (Set.delete cred regCreds, totalRefunds <+> keyDeposit)
              -- Check for the deposit left during registration in some previous
              -- transaction. This de-registration check will be matched first, despite being
              -- the last case to match, because registration is not possible without
              -- de-registration.
              | Just deposit <- lookupDeposit cred -> (regCreds, totalRefunds <+> deposit)
            _ -> (regCreds, totalRefunds)
