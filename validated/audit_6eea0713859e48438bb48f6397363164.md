### Title
Sub-Transaction Value Conservation Not Enforced — Direct ADA Creation/Destruction via Unbalanced Sub-Transactions - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

In the Dijkstra era's nested-transaction design, sub-transactions processed by the `SUBUTXO` rule have **no preservation-of-value check**. The top-level `UTXO` rule does perform a balance check, but it calls `consumed = conwayConsumed`, which only inspects the top-level transaction body and ignores sub-transactions entirely. Because neither layer enforces `consumed == produced` for sub-transactions, an unprivileged sender can include a sub-transaction whose outputs exceed its inputs, directly creating ADA (or destroying it) in a single valid batch.

---

### Finding Description

**Vulnerability class:** funds/accounting bug — a pre-processing step (sub-transaction execution) modifies the UTxO before the balance check, but the balance check does not account for those modifications.

**Step 1 — Sub-transactions have no balance check.**

`dijkstraSubUtxoTransition` validates inputs, output sizes, network IDs, and validity intervals, then calls `Shelley.updateUTxOStateNoFees` to apply the UTxO diff. There is no call to `validateValueNotConservedUTxO` anywhere in this rule. [1](#0-0) 

**Step 2 — The top-level balance check uses `conwayConsumed`, which ignores sub-transactions.**

`dijkstraUtxoTransition` does call `validateValueNotConservedUTxO`, but it passes `certState` and `originalUtxo` and relies on the `EraUTxO` instance's `consumed` method: [2](#0-1) 

For `DijkstraEra`, that instance binds `consumed = conwayConsumed`: [3](#0-2) 

`conwayConsumed` is imported from the Conway era and has no knowledge of sub-transactions. It only sums the top-level tx body's inputs, withdrawals, and deposit refunds.

**Step 3 — The Dijkstra-aware consumed function exists but is not wired into the ledger check.**

`getConsumedDijkstraValue` correctly aggregates both the top-level body and all sub-transaction bodies: [4](#0-3) 

Likewise, `dijkstraProducedValue` / `getProducedDijkstraValue` aggregate sub-transaction produced values: [5](#0-4) 

These are exposed through `getConsumedValue` / `getProducedValue` (used by the wallet API `evalBalanceTxBody`), but the ledger's own `validateValueNotConservedUTxO` call goes through `consumed`, which is bound to `conwayConsumed`. The two code paths diverge at the `EraUTxO` instance: [3](#0-2) 

**Step 4 — Sub-transactions are processed before the top-level balance check.**

The LEDGER rule runs `SUBLEDGERS` first (mutating the UTxO state), then calls `UTXOW` → `UTXO` for the top-level transaction: [6](#0-5) 

By the time `validateValueNotConservedUTxO` runs, sub-transaction UTxO diffs are already committed to `utxoStateAfterSubLedgers`, but the check only covers the top-level body.

---

### Impact Explanation

An unprivileged sender can craft a batch where:

- The top-level transaction body is perfectly balanced (passes `validateValueNotConservedUTxO`).
- One or more sub-transactions spend inputs worth X lovelace and produce outputs worth X + Y lovelace (Y > 0).

Because `dijkstraSubUtxoTransition` applies the UTxO diff via `updateUTxOStateNoFees` without any balance check, and because the top-level check uses `conwayConsumed` (which ignores sub-transactions), the Y lovelace surplus is silently written into the UTxO. This constitutes **direct creation of ADA through an invalid ledger state transition** — the highest-severity impact class.

The same asymmetry allows deliberate ADA destruction (outputs < inputs in a sub-transaction), permanently removing lovelace from circulation.

---

### Likelihood Explanation

The Dijkstra era is the only era with nested transactions; the vulnerability is reachable by any transaction sender who can submit a `TopTx` containing at least one `SubTx`. No privileged role, governance majority, or key compromise is required. The attacker only needs to construct a transaction with an unbalanced sub-transaction body, which is straightforward given the public ledger serialization format.

---

### Recommendation

**Short term:** Add `validateValueNotConservedUTxO` (or an equivalent sub-transaction-aware check) inside `dijkstraSubUtxoTransition`, analogous to the check already present in `dijkstraUtxoTransition`. Each sub-transaction must independently satisfy `consumed == produced` (with zero fees, as is already the design intent).

**Long term:** Reconcile the `consumed` / `getConsumedValue` split in the `EraUTxO DijkstraEra` instance. Either bind `consumed` to `getConsumedDijkstraValue` so that the ledger's own balance check covers the full batch, or document and enforce the invariant that each sub-transaction is individually balanced. Add fuzz tests that generate batches with deliberately unbalanced sub-transactions and assert that they are rejected.

---

### Proof of Concept

1. Construct a `TopTx` with a balanced body (e.g., spend input `I_top` worth 2 ADA, produce output worth 1 ADA + 1 ADA fee).
2. Embed a `SubTx` that spends input `I_sub` worth 1 ADA and produces a single output worth 1 000 000 ADA.
3. Submit the batch.

**Execution path:**

- `SUBLEDGERS` → `SUBLEDGER` → `SUBUTXOW` → `SUBUTXO`: validates that `I_sub` exists in `originalUtxo` ✓, checks output sizes ✓, calls `updateUTxOStateNoFees` — UTxO now contains a 1 000 000 ADA output. No balance check fires.
- `UTXOW` → `UTXO` (`dijkstraUtxoTransition`): `validateValueNotConservedUTxO pp originalUtxo certState topTxBody` calls `conwayConsumed` on the top-level body only — result is balanced ✓. The 999 999 ADA surplus from the sub-transaction is invisible to this check.
- The batch is accepted; the ledger state now contains 999 999 ADA that did not exist before.

getConsumedDijkstraValue ::
  forall era l.
  ( DijkstraEraTxBody era
  , EraUTxO era
  , Value era ~ MaryValue
  , STxLevel l era ~ STxBothLevels l era
  ) =>
  PParams era ->
  (Credential Staking -> Maybe Coin) ->
  (Credential DRepRole -> Maybe Coin) ->
  UTxO era ->
  TxBody l era ->
  Value era
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue :: forall m. TxBody m era -> Value era
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
