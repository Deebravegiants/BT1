### Title
`respond_ckd` Skips Cryptographic Verification for `AppPublicKey` CKD Requests, Allowing a Single Byzantine Participant to Deliver Fabricated Key Material — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` enforces a cryptographic output check only for `AppPublicKeyPV` requests, but performs **no verification** for `AppPublicKey` requests. This asymmetry is the direct analog of the Size `forSale` flag bypass: a status-gated check is enforced in one code path and silently skipped in another. A single attested Byzantine participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary fabricated `CKDResponse` for any pending `AppPublicKey` request, resolve it unilaterally, and deliver incorrect key material to the requester — bypassing the threshold authorization requirement entirely.

---

### Finding Description

In `respond_ckd`, after the caller-attestation and protocol-state guards pass, the response is conditionally verified:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKey` requests the arm is a no-op: any `CKDResponse` value is accepted and immediately used to drain the pending-request queue. Compare this to `respond`, which **always** verifies the ECDSA/EdDSA signature before resolving: [2](#0-1) 

The structural parallel to the Size bug is exact:

| Size | NEAR MPC |
|---|---|
| `buyCreditMarket()` checks `forSale` flag | `respond_ckd` with `AppPublicKeyPV` runs `ckd_output_check` |
| `sellCreditMarket()` **skips** `forSale` check | `respond_ckd` with `AppPublicKey` **skips** all verification |
| Owner force-sells a "not for sale" position | Byzantine participant force-resolves a CKD request with fabricated output |
| New owner stuck with unsellable position | Requester receives garbage key material; legitimate computation blocked |

The `respond_ckd` function's caller guards (`assert_caller_is_signer`, `is_running_or_resharing`, `accept_requests`, `assert_caller_is_attested_participant_and_protocol_active`) are all present: [3](#0-2) 

These guards confirm the caller is a legitimate attested participant — but they do **not** prevent a Byzantine participant from submitting a fabricated response for an `AppPublicKey` request.

---

### Impact Explanation

A single attested Byzantine participant (below threshold) can:

1. Observe any pending `AppPublicKey` CKD request in the contract's `pending_ckd_requests` map.
2. Call `respond_ckd` with an arbitrary `CKDResponse { big_y: <garbage>, big_c: <garbage> }`.
3. The contract accepts the response without any cryptographic check and calls `resolve_yields_for`, draining the entire fan-out queue. [4](#0-3) 

4. The requester's yield-resume promise fires with the fabricated bytes — the requester receives incorrect key material.
5. The legitimate threshold computation (which requires `t`-of-`n` participants) is permanently blocked for that request: the pending entry is gone.

This is **confidential key derivation output delivered without the required participant authorization** — one participant unilaterally resolves what should require a threshold quorum.

---

### Likelihood Explanation

The attacker must be an attested MPC participant (pass `assert_caller_is_attested_participant_and_protocol_active`). This is a single node below the signing threshold — exactly the Byzantine fault model threshold cryptography is designed to tolerate. No collusion, no key leakage, and no privileged operator access beyond normal participant status is required. The attack is executable on any `AppPublicKey` CKD request visible on-chain.

---

### Recommendation

1. **Require `AppPublicKeyPV` for all new requests** and deprecate `AppPublicKey`. The `AppPublicKeyPV` path already enforces `ckd_output_check`, which provides the missing on-chain verification.
2. If `AppPublicKey` must remain supported, require a threshold quorum of identical `CKDResponse` submissions before resolving the request (analogous to how `respond` requires a cryptographically valid signature from any single participant, but the signature itself encodes threshold agreement).
3. At minimum, document the known single-participant resolution risk for `AppPublicKey` requests so integrators can make an informed choice.

---

### Proof of Concept

```
1. User calls request_app_private_key({
       derivation_path: "my/path",
       app_public_key: AppPublicKey(<user_bls_g1_pk>),
       domain_id: <ckd_domain_id>
   })
   → pending_ckd_requests[ckd_request] = [yield_id]

2. Byzantine participant (attested, below threshold) calls respond_ckd(
       request = ckd_request,
       response = CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }
   )
   → AppPublicKey arm: no-op (no verification)
   → resolve_yields_for drains the queue, fires promise with fabricated bytes

3. User's sign callback receives { big_y: [0;48], big_c: [0;48] }
   → Decryption of big_c with user's private key yields garbage
   → Legitimate t-of-n computation never gets a chance to respond
     (pending entry already removed)
```

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
