### Title
Unchecked Return Value from Detached Refund Transfer Promises Permanently Locks User Deposits - (File: crates/contract/src/lib.rs)

### Summary
Multiple payable entry points in `mpc-contract` schedule NEAR token refunds via `Promise::new(account).transfer(amount).detach()`. The `.detach()` call explicitly discards the promise result, meaning a failed transfer is silently swallowed. If the recipient account is deleted between transaction submission and promise execution, the refunded deposit is permanently locked inside the contract with no recovery path.

### Finding Description
The NEAR analog of Solidity's unchecked low-level `call()` is a detached promise whose result is never inspected. In three distinct refund paths the contract uses exactly this pattern:

**1. `require_deposit` (excess-deposit refund, called from every payable entry point including `sign`):** [1](#0-0) 

**2. `propose_update` (excess-deposit refund for governance proposals):** [2](#0-1) 

**3. `submit_participant_info` (excess-deposit refund for attestation submissions):** [3](#0-2) 

In all three cases the pattern is identical: `Promise::new(recipient).transfer(amount).detach()`. The `.detach()` call tells the NEAR SDK to schedule the promise but never attach a callback or inspect its outcome. If the transfer receipt fails — for example because the recipient account was deleted in the same block or a subsequent block before the receipt executes — the NEAR runtime silently drops the failed receipt and the tokens remain in the contract balance forever. There is no fallback, no retry, and no administrative recovery function that could sweep stranded deposits back to their owners.

The `require_deposit` helper is the highest-value path because it is invoked by `sign`, `request_app_private_key`, and `request_verify_foreign_tx`. A user who attaches a deposit larger than the 1-yoctoNEAR minimum and whose account is deleted (or whose account ID is recycled) before the refund receipt executes loses the entire excess. [4](#0-3) 

### Impact Explanation
Permanently locks user-attached NEAR deposits inside the contract. The contract accumulates an ever-growing balance of stranded tokens with no on-chain mechanism to recover them. This breaks the production balance-accounting invariant: every deposit that exceeds the minimum must be returned to its sender, but the contract provides no guarantee of that when the transfer promise fails silently.

**Impact: Medium** — balance and request-lifecycle accounting invariant is broken; no reliance on network-level DoS or operator misconfiguration.

### Likelihood Explanation
NEAR account deletion is a first-class operation available to any account holder. A user can call `DeleteAccount` in the same block as (or immediately after) a `sign` call. Because NEAR processes receipts asynchronously, the refund transfer receipt executes one or more blocks after the original call receipt; if the account is gone by then, the transfer fails. This is a realistic, attacker-controllable condition requiring no privileged access — any unprivileged caller can trigger it against their own account to permanently strand their own excess deposit, or a malicious relayer could construct the sequence for a victim.

### Recommendation
Replace every `Promise::new(recipient).transfer(amount).detach()` refund with a checked pattern. Either:

1. **Attach a `.then` callback** that logs or re-credits the amount on failure, or
2. **Use `#[callback_result]`** in a dedicated refund callback that panics (rolling back the parent receipt) if the transfer fails, forcing the caller to retry.

At minimum, remove `.detach()` and let the promise result propagate so that a failed transfer causes the enclosing receipt to fail rather than silently succeeding with locked funds.

### Proof of Concept
1. Alice calls `sign(payload, path)` attaching 10 NEAR (far above the 1-yoctoNEAR minimum).
2. `require_deposit` computes `diff = 10 NEAR - 1 yoctoNEAR` and schedules `Promise::new(alice).transfer(diff).detach()`.
3. In the same block, Alice calls `DeleteAccount { beneficiary_id: bob }` — a valid, permissionless operation.
4. The refund transfer receipt executes one block later; Alice's account no longer exists; the NEAR runtime marks the receipt as failed.
5. Because the promise was detached, the failure is never observed by the contract.
6. `diff` (~10 NEAR) remains in the contract balance permanently with no recovery path. [1](#0-0) [2](#0-1)

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}
