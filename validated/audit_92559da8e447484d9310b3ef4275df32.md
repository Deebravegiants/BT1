Audit Report

## Title
SwapAllowlistExtension gates the router address instead of the real swapper, breaking allowlist enforcement for router-mediated swaps - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`MetricOmmPool.swap` passes `msg.sender` (the router contract) as the `sender` argument to `_beforeSwap`, which forwards it verbatim to `SwapAllowlistExtension.beforeSwap`. The extension then checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`. This produces two concrete failures: (1) allowlisted end-users are unconditionally blocked when they route through `MetricOmmSimpleRouter`, and (2) if the pool admin adds the router to the allowlist, every unprivileged user bypasses the gate entirely.

## Finding Description
**Root cause – wrong identity forwarded through the call chain:**

`MetricOmmPool.swap` (line 231) passes `msg.sender` as `sender` to `ExtensionCalling._beforeSwap`:
```solidity
_beforeSwap(msg.sender, recipient, zeroForOne, ...);
```
`ExtensionCalling._beforeSwap` (lines 160–176) encodes that value and calls the extension:
```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
```
`SwapAllowlistExtension.beforeSwap` (line 37) then checks:
```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```
Here `msg.sender` is the pool and `sender` is whoever called `pool.swap`. When `MetricOmmSimpleRouter.exactInputSingle` (lines 72–80) calls `pool.swap(...)`, the pool sees `msg.sender = router`. The router never forwards the original `msg.sender` (the end-user) to the pool; there is no such parameter in the `swap` signature.

**Exploit flow A – allowlisted user blocked:**
1. Pool admin deploys pool with `SwapAllowlistExtension`, allowlists `alice`.
2. `alice` calls `MetricOmmSimpleRouter.exactInputSingle`.
3. Router calls `pool.swap`; pool passes `sender = router` to the extension.
4. Extension checks `allowedSwapper[pool][router]` → `false` → reverts `NotAllowedToSwap`.
5. `alice` cannot use the router despite being explicitly allowlisted.

**Exploit flow B – allowlist bypass:**
1. Pool admin allowlists the router address (reasonable assumption: "the router is trusted").
2. Non-allowlisted `bob` calls `MetricOmmSimpleRouter.exactInputSingle`.
3. Router calls `pool.swap`; pool passes `sender = router`.
4. Extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
5. `bob` bypasses the allowlist entirely.

Existing guards are insufficient: `BaseMetricExtension.onlyPool` only verifies the caller is a registered pool; it does not verify the identity of the original swapper. The `extensionData` bytes are ignored by `SwapAllowlistExtension`, so there is no in-band channel to pass the real user address.

## Impact Explanation
Flow A breaks core pool functionality: allowlisted users cannot access the public router, which is the primary swap entry point for end-users. Flow B is a direct allowlist bypass reachable by any unprivileged user once the router is allowlisted. Both outcomes violate the "swap allowlist gate" invariant that the extension is designed to enforce. Flow A meets the Sherlock medium threshold (broken core swap functionality); Flow B meets the high threshold (security control bypass enabling unauthorized swaps).

## Likelihood Explanation
Flow A is triggered unconditionally whenever an allowlisted user routes through `MetricOmmSimpleRouter`—no attacker capability required, no special precondition beyond normal usage. Flow B requires the pool admin to add the router to the allowlist, which is a natural operational step when the router is considered a trusted intermediary. Both paths are repeatable and require no privileged attacker capability beyond calling a public function.

## Recommendation
The pool's `swap` function should accept an explicit `swapper` address parameter (distinct from `recipient`) that the router populates with `msg.sender` before calling the pool. `ExtensionCalling._beforeSwap` would then forward this verified swapper identity to extensions. Alternatively, `MetricOmmSimpleRouter` can encode the original `msg.sender` into `extensionData` and `SwapAllowlistExtension.beforeSwap` can decode and check it—but this requires a coordinated protocol-level convention. The cleanest fix is adding a `swapper` field to the `swap` call so the pool can pass the authenticated originator to all extensions.

## Proof of Concept
```solidity
// Foundry test sketch
function test_allowlistedUserBlockedViaRouter() public {
    // Setup: pool with SwapAllowlistExtension, alice allowlisted
    swapExtension.setAllowedToSwap(address(pool), alice, true);
    // alice swaps through the router (not directly)
    vm.prank(alice);
    // router.exactInputSingle calls pool.swap with msg.sender = router
    // extension checks allowedSwapper[pool][router] → false → revert
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    router.exactInputSingle(ExactInputSingleParams({..., pool: address(pool)}));
}

function test_nonAllowlistedUserBypassesViaRouter() public {
    // Pool admin adds router to allowlist
    swapExtension.setAllowedToSwap(address(pool), address(router), true);
    // bob (not allowlisted) swaps through router
    vm.prank(bob);
    // extension checks allowedSwapper[pool][router] → true → succeeds
    router.exactInputSingle(ExactInputSingleParams({..., pool: address(pool)}));
    // bob bypassed the allowlist
}
```
