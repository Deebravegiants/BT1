### Title
Sub-transaction Plutus Script ExUnits Excluded from `maxTxExUnits` Enforcement and Minimum Fee Calculation in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces nested sub-transactions (`dtbrSubTransactions`) embedded inside a top-level transaction body. The `maxTxExUnits` enforcement and the minimum fee's script-execution component both rely on `totExUnits`, which only sums redeemers from the **top-level** transaction's witness set. Sub-transactions carry their own witness sets with their own redeemers, but those ExUnits are never counted. An attacker can therefore include sub-transactions whose Plutus scripts declare arbitrarily large ExUnits budgets, causing validators to perform unbounded script execution that is neither bounded by the protocol parameter nor paid for in the transaction fee.

---

### Finding Description

**`totExUnits` only inspects the top-level transaction's redeemers.** [1](#0-0) 

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

Sub-transactions live in `tx ^. bodyTxL . subTransactionsTxBodyL` (an `OMap TxId (Tx SubTx era)`), not in the top-level witness set. They are invisible to `totExUnits`.

**`validateExUnitsTooBigUTxO` delegates directly to `totExUnits tx`.** [2](#0-1) 

**The Dijkstra UTXO transition rule calls this check unchanged.** [3](#0-2) 

```haskell
  {- totExunits tx ≤ maxTxExUnits pp -}
  runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

Only the top-level `tx` is passed; sub-transaction redeemers are never aggregated.

**The `SubUtxo` rule explicitly marks `ExUnitsTooBigUTxO` as impossible**, confirming no per-sub-transaction limit exists either. [4](#0-3) 

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
```

**The minimum fee's script-execution component also uses `totExUnits`.** [5](#0-4) 

```haskell
alonzoMinFeeTx pp tx =
  ...
    <+> txscriptfee (pp ^. ppPricesL) allExunits
  where
    allExunits = totExUnits tx   -- sub-tx redeemers absent
```

**Contrast: reference-script size already aggregates sub-transactions correctly.** [6](#0-5) 

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum (foldMap' (Sum . txNonDistinctRefScriptsSize utxo)
                       (tx ^. bodyTxL . subTransactionsTxBodyL))
```

The reference-script fee correctly folds over sub-transactions; the ExUnits check does not. This asymmetry is the root cause.

**Sub-transactions can carry Plutus scripts.** `DijkstraSubTxBodyRaw` contains `dstbrScriptIntegrityHash` and `dstbrGuards` (script credentials), so sub-transactions do execute phase-2 scripts. [7](#0-6) 

---

### Impact Explanation

An attacker submits a top-level transaction whose top-level redeemers declare minimal ExUnits (passing `validateExUnitsTooBigUTxO`) while embedding sub-transactions whose redeemers declare ExUnits far exceeding `maxTxExUnits`. Every validating node must execute those Plutus scripts up to the declared budget. Because the budget is uncapped and the fee does not cover the execution cost, the attacker obtains unbounded validator CPU/memory consumption at sub-minimum cost. This matches the **Medium** allowed impact: *attacker-controlled transactions exceed intended validation limits and modify fees outside design parameters*.

---

### Likelihood Explanation

The Dijkstra era is production-bound code. Any user who can submit a transaction (no privileged role required) can embed sub-transactions. The construction is straightforward: set top-level redeemers to `ExUnits 0 0` and sub-transaction redeemers to `ExUnits maxBound maxBound`. The `maxTxExUnits` check passes; the fee check passes; validators execute the scripts.

---

### Recommendation

Introduce a `batchTotExUnits` function analogous to `batchNonDistinctRefScriptsSize` that sums ExUnits across the top-level transaction and all sub-transactions:

```haskell
batchTotExUnits :: (AlonzoEraTxWits era, DijkstraEraTxBody era) => Tx TopTx era -> ExUnits
batchTotExUnits tx =
  totExUnits tx
    <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
```

Replace the call in `dijkstraUtxoTransition` with a Dijkstra-specific `validateBatchExUnitsTooBigUTxO` that uses `batchTotExUnits`, and update `alonzoMinFeeTx` (or its Dijkstra override) to use `batchTotExUnits` for the `txscriptfee` component. Apply the same fix to `validateExUnits` in the Dijkstra BBODY rule so the block-level `maxBlockExUnits` check also aggregates sub-transaction ExUnits.

---

### Proof of Concept

1. Construct a `DijkstraTx` with a top-level body whose redeemers map is empty (zero ExUnits at the top level).
2. Embed one or more sub-transactions (`dtbrSubTransactions`) each containing a Plutus script guard/spending script with redeemers declaring `ExUnits { exUnitsMem = maxBound, exUnitsSteps = maxBound }`.
3. Set the top-level fee to `minfee pp tx utxo` — this passes because `alonzoMinFeeTx` uses `totExUnits tx = mempty`.
4. Submit the transaction. `validateExUnitsTooBigUTxO pp tx` compares `mempty ≤ maxTxExUnits` — passes. Validators then execute the sub-transaction Plutus scripts up to `maxBound` ExUnits each, with no protocol-level cap and no fee collected for that execution.
