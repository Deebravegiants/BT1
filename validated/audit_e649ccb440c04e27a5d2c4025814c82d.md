### Title
Block Stuffing Delays `setQueuedWithdrawalsBuffer`, Enabling Instant Withdrawals to Drain Assets Reserved for Queued Users — (`contracts/LRTUnstakingVault.sol`)

---

### Summary

`queuedWithdrawalsBuffer` is the sole on-chain mechanism protecting vault assets from being consumed by instant withdrawals while queued-withdrawal users are waiting. Because it is updated by a manual operator call (`setQueuedWithdrawalsBuffer`), an attacker can perform block stuffing to delay that update, causing `getAssetsAvailableForInstantWithdrawal` to over-report available assets and allowing instant withdrawals to drain funds earmarked for queued users.

---

### Finding Description

`LRTUnstakingVault` maintains a per-asset `queuedWithdrawalsBuffer` that is subtracted from the vault balance to compute how much is available for instant withdrawal:

```solidity
// LRTUnstakingVault.sol:235-237
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
``` [1](#0-0) 

This buffer is set exclusively by the LRT Operator:

```solidity
// LRTUnstakingVault.sol:199-209
function setQueuedWithdrawalsBuffer(address asset, uint256 buffer)
    external
    onlyLRTOperator
    onlySupportedAsset(asset)
{
    queuedWithdrawalsBuffer[asset] = buffer;
    ...
}
``` [2](#0-1) 

There is **no automatic coupling** between `assetsCommitted[asset]` in `LRTWithdrawalManager` (which tracks how much asset is owed to queued users) and `queuedWithdrawalsBuffer` in `LRTUnstakingVault`. The operator must manually keep them in sync. [3](#0-2) 

`instantWithdrawal` enforces only the buffer-derived limit:

```solidity
// LRTWithdrawalManager.sol:231-233
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
``` [4](#0-3) 

If the buffer has not been raised to reflect newly queued commitments, `getAssetsAvailableForInstantWithdrawal` returns a value that includes assets that should be reserved for queued users, and instant withdrawals can consume them.

---

### Impact Explanation

When the vault is drained below the amount needed to service queued withdrawals, the subsequent `unlockQueue` call invokes `unstakingVault.redeem(asset, assetAmountUnlocked)`: [5](#0-4) 

If the vault balance is insufficient, this transfer reverts, temporarily freezing queued users' withdrawal claims until the vault is replenished. This constitutes **temporary freezing of funds** caused by the block stuffing attack.

---

### Likelihood Explanation

Block stuffing on Ethereum mainnet is expensive but not impossible, especially when the attacker stands to gain from delaying a single operator transaction for a short window. The attack window is bounded: the operator only needs to update the buffer once after large queued withdrawals are registered and EigenLayer unstaking completes. A targeted stuffing of even a few blocks during that window is sufficient. The scope explicitly recognises "Low. Block stuffing" as a valid impact category, confirming the protocol's threat model includes this vector.

---

### Recommendation

Eliminate the manual synchronisation requirement by deriving the reserved amount on-chain from `assetsCommitted` in `LRTWithdrawalManager` rather than from a separately maintained buffer. Alternatively, have `initiateWithdrawal` atomically update the buffer in `LRTUnstakingVault` whenever `assetsCommitted` increases, so no operator transaction is needed and block stuffing cannot create a gap. [6](#0-5) 

---

### Proof of Concept

```
State before attack:
  vault.balance[ETH]                  = 100 ETH
  queuedWithdrawalsBuffer[ETH]        = 10 ETH   (stale; not yet updated)
  assetsCommitted[ETH]                = 80 ETH   (large queued withdrawals just registered)

Step 1 – Large queued withdrawals registered:
  Users call initiateWithdrawal(ETH, ...) → assetsCommitted[ETH] = 80 ETH
  Operator prepares setQueuedWithdrawalsBuffer(ETH, 80 ETH) tx.

Step 2 – Attacker stuffs blocks:
  Attacker fills blocks with high-gas txs, preventing operator tx from landing.

Step 3 – getAssetsAvailableForInstantWithdrawal returns stale value:
  availableAmount = 100 - 10 = 90 ETH   ← should be 100 - 80 = 20 ETH

Step 4 – Instant withdrawals drain vault:
  Instant withdrawers consume up to 90 ETH.
  vault.balance[ETH] drops to ~10 ETH.

Step 5 – unlockQueue called:
  unstakingVault.redeem(ETH, 80 ETH) → reverts (only 10 ETH available).
  Queued users cannot complete withdrawals → temporary freeze.

Assert: vault.balance < assetsCommitted after instant withdrawals when buffer update is delayed.
```
