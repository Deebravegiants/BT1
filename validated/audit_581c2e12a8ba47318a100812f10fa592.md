### Title
`verify_foreign_transaction` Gates on Stale Intersection-Rule Instead of Threshold-Based Availability Cache — (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` checks chain availability using `get_supported_foreign_chains()` — the legacy strict-intersection rule that requires **every** active participant to have registered a chain. The contract simultaneously maintains a new threshold-based availability cache (`available_foreign_chains`) that is correctly computed but **never consulted** at the request gate. A single Byzantine participant strictly below the signing threshold can block all foreign-chain verification requests by simply not registering in the old system, even when ≥ `signing_threshold` other participants cover the chain.

---

### Finding Description

The contract implements two parallel foreign-chain support tracking systems:

**Old system (intersection rule):**
- Registration: `register_foreign_chain_support` → `node_foreign_chain_support`
- Query: `get_supported_foreign_chains()` — returns the strict intersection of every active participant's registered chains. One participant with an empty or absent registration drops every chain.

**New system (threshold-based):**
- Registration: `register_foreign_chains_config` → `foreign_chains.foreign_chains_configs`
- Recompute: `recompute_available_foreign_chains()` → `available_foreign_chains` cache
- Query: `get_available_foreign_chains()` — returns chains covered by ≥ max ForeignTx domain reconstruction threshold of active participants.

`verify_foreign_transaction` gates exclusively on the **old** system:

```rust
// crates/contract/src/lib.rs ~line 534
let supported_chains = self.get_supported_foreign_chains();   // ← old intersection rule
if !supported_chains.contains(&requested_chain) {
    env::panic_str(...)
}
```

The new threshold-based cache is computed and maintained (triggered on every `register_foreign_chains_config`, `vote_update_foreign_chain_providers`, resharing, and node migration) but the result is never used to gate `verify_foreign_transaction`. The design document explicitly states the old rule "is to be deprecated in favour of the two views above," and the new `recompute_available_foreign_chains` function was built to replace it — but the replacement was never wired into the request gate.

This is structurally identical to the reported Arbitrum bug: a correct accumulator is computed starting from the wrong baseline (the old intersection rule instead of the new threshold-based rule), so the security gate operates on an incomplete/incorrect value.

---

### Impact Explanation

**Medium — request-lifecycle manipulation that breaks production safety/accounting invariants.**

The production safety invariant is: *a foreign-chain verification request for chain C should be accepted whenever ≥ signing_threshold active participants cover C.* The old intersection rule breaks this invariant: one participant that has not called `register_foreign_chain_support` (or that calls it with an empty set) causes `get_supported_foreign_chains()` to return ∅, making every `verify_foreign_transaction` call panic with `ForeignChainNotSupported` — regardless of how many other participants have registered via the new system.

This permanently stalls the foreign-chain verification flow (Omnibridge inbound, chain-abstraction attestations) for all users until the non-registering participant registers. Because the new system's cache correctly shows the chain as available, operators and monitoring will see no anomaly in `get_available_foreign_chains()`, making the root cause hard to diagnose.

---

### Likelihood Explanation

**Medium.** Any single active participant can trigger this by omitting or clearing their `register_foreign_chain_support` call. No special privileges, key material, or collusion is required — the attacker only needs to be one of the ≤ n − threshold participants whose absence does not break the signing threshold. The attack is low-effort, persistent (the block lasts until the participant re-registers), and silent from the perspective of the new availability cache.

---

### Recommendation

Replace the old intersection-rule gate in `verify_foreign_transaction` with the new threshold-based cache:

```rust
// crates/contract/src/lib.rs ~line 534
// Replace:
let supported_chains = self.get_supported_foreign_chains();
// With:
let supported_chains = self.get_available_foreign_chains();
```

Ensure `recompute_available_foreign_chains` is called on every state-changing path that can affect availability (it already is for `register_foreign_chains_config`, `vote_update_foreign_chain_providers`, resharing, and node migration). Deprecate and remove `register_foreign_chain_support` / `get_supported_foreign_chains` once the migration is complete.

---

### Proof of Concept

1. Deploy contract with 4 participants, governance threshold = 3, ForeignTx domain reconstruction threshold = 3. Whitelist Bitcoin.
2. Participants P1, P2, P3 call `register_foreign_chains_config([Bitcoin])` (new system). `get_available_foreign_chains()` → `{Bitcoin}` (3 ≥ 3). ✓
3. Byzantine participant P4 (below threshold) never calls `register_foreign_chain_support` (old system). `get_supported_foreign_chains()` → `∅` (intersection with absent entry = ∅).
4. Any user calls `verify_foreign_transaction(Bitcoin, ...)`. Contract evaluates `get_supported_foreign_chains().contains(Bitcoin)` → `false` → `panic!("ForeignChainNotSupported")`.
5. All foreign-chain verification is blocked despite 3-of-4 participants covering Bitcoin in the new system.

The discrepancy is confirmed by the existing test `verify_foreign_tx__should_reject_chain_not_in_policy` which uses `register_supported_chains` (old method) to set up the gate, not `register_foreign_chains_config` (new method) — proving the gate has not been migrated.     pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
