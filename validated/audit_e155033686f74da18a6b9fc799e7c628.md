### Title
Zero `rsETHPrice` Allows Zero-Payout Withdrawal Request, Permanently Burning User rsETH — (File: `contracts/LRTWithdrawalManager.sol`)

---

### Summary

`LRTWithdrawalManager.initiateWithdrawal` does not validate that `expectedAssetAmount > 0`. When `rsETHPrice` is 0 in `LRTOracle`, `getExpectedAssetAmount` returns 0, and a withdrawal request is stored with `expectedAssetAmount = 0` while the user's rsETH is already held by the contract. When the operator later calls `unlockQueue`, `_unlockWithdrawalRequests` computes `payoutAmount = 0` and unconditionally adds the full `rsETHUnstaked` to the burn total — permanently destroying the user's rsETH — while unlocking zero assets. The user then calls `completeWithdrawal` and receives nothing.

---

### Finding Description

**Step 1 — Zero rsETHPrice is reachable.**

`rsETHPrice` is a stored state variable in `LRTOracle` that defaults to `0`. It is only updated when `updateRSETHPrice()` is called. Additionally, in `_updateRsETHPrice`, the downside-protection guard is:

```solidity
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
``` [1](#0-0) 

Because `pricePercentageLimit` defaults to `0`, the guard is permanently disabled unless the admin explicitly sets it. If `totalETHInProtocol` reaches 0 (complete slashing) while `pricePercentageLimit = 0`, `newRsETHPrice = 0` is written to storage without triggering a pause:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [2](#0-1) 

**Step 2 — `initiateWithdrawal` accepts a zero `expectedAssetAmount`.**

```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
// No check: expectedAssetAmount > 0
assetsCommitted[asset] += expectedAssetAmount;   // += 0
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);  // stores 0
``` [3](#0-2) 

`getExpectedAssetAmount` computes `rsETHUnstaked * rsETHPrice / assetPrice`. When `rsETHPrice = 0`, this returns 0 regardless of `rsETHUnstaked`. The user's rsETH is transferred to the contract at line 166, but the stored `expectedAssetAmount` is 0. [4](#0-3) 

**Step 3 — `_unlockWithdrawalRequests` burns rsETH unconditionally even when `payoutAmount = 0`.**

```solidity
uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);
if (availableAssetAmount < payoutAmount) break;   // 0 < 0 is false → never breaks
assetsCommitted[asset] -= request.expectedAssetAmount;
request.expectedAssetAmount = payoutAmount;        // set to 0
rsETHAmountToBurn += request.rsETHUnstaked;        // ← rsETH queued for burn
availableAssetAmount -= payoutAmount;              // -= 0
assetAmountToUnlock += payoutAmount;               // += 0
unlockedWithdrawalsCount[asset]++;
``` [5](#0-4) 

`_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`. Since `expectedAssetAmount = 0`, it returns 0 regardless of the current price. The guard `if (availableAssetAmount < payoutAmount)` evaluates to `if (x < 0)` for uint256, which is always false, so the loop never breaks on a zero-payout request. [6](#0-5) 

Back in `unlockQueue`, the rsETH is burned while zero assets are redeemed:

```solidity
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);  // redeem(asset, 0)
``` [7](#0-6) 

**Step 4 — `completeWithdrawal` transfers 0 assets.**

```solidity
_transferAsset(asset, user, request.expectedAssetAmount);  // amount = 0
``` [8](#0-7) 

`_transferAsset` with `amount = 0` succeeds silently for both ETH (`call{value:0}`) and ERC20 (`safeTransfer(..., 0)`). The user's rsETH is permanently destroyed and they receive nothing. [9](#0-8) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who initiates a withdrawal while `rsETHPrice = 0` has their rsETH transferred to the contract and later permanently burned by `unlockQueue`. `completeWithdrawal` transfers 0 assets. There is no recovery path: the withdrawal request is deleted, the rsETH is gone, and the user receives nothing. This is structurally identical to the referenced report's "NFT permanently stuck" scenario — the asset (rsETH) is consumed by the protocol with no corresponding output.

---

### Likelihood Explanation

**Low.**

Two conditions must coincide:

1. `rsETHPrice = 0` in `LRTOracle` storage. This occurs either (a) before `updateRSETHPrice()` has ever been called (initial deployment window), or (b) after a catastrophic full-slashing event when `pricePercentageLimit = 0` (its default value), which disables the automatic pause guard.
2. A user calls `initiateWithdrawal` while the price is in this state and the contract is not paused.

The default `pricePercentageLimit = 0` means the downside protection is off-by-default, making condition (b) more likely than it appears. The external report similarly rated its analogous condition (no treasury backing) as Low likelihood / High impact.

---

### Recommendation

1. **In `initiateWithdrawal`**: revert if `expectedAssetAmount == 0`:
   ```solidity
   if (expectedAssetAmount == 0) revert InvalidAmountToWithdraw();
   ```

2. **In `_unlockWithdrawalRequests`**: skip (do not burn rsETH for) requests where `payoutAmount == 0`, analogous to the external report's recommendation to re-queue the asset:
   ```solidity
   if (payoutAmount == 0) break; // or continue, depending on desired behavior
   ```

3. **Set a non-zero `pricePercentageLimit`** during deployment to ensure the automatic pause guard is active.

---

### Proof of Concept

```
1. Deploy protocol; do NOT call updateRSETHPrice() (rsETHPrice = 0 in storage).
   OR: deploy with pricePercentageLimit = 0 (default), slash all EigenLayer assets,
       call updateRSETHPrice() → rsETHPrice is set to 0.

2. Victim calls initiateWithdrawal(stETH, 1e18, ""):
   - 1e18 rsETH transferred from victim to LRTWithdrawalManager.
   - getExpectedAssetAmount returns 0 (1e18 * 0 / assetPrice = 0).
   - WithdrawalRequest stored: { rsETHUnstaked: 1e18, expectedAssetAmount: 0 }.

3. After withdrawalDelayBlocks, operator calls unlockQueue(stETH, ..., 0, type(uint256).max, 0, type(uint256).max):
   - _calculatePayoutAmount returns min(0, currentReturn) = 0.
   - payoutAmount = 0; guard `availableAssetAmount < 0` is false → loop continues.
   - rsETHAmountToBurn += 1e18; assetAmountToUnlock += 0.
   - burnFrom(withdrawalManager, 1e18) → victim's rsETH permanently destroyed.
   - unstakingVault.redeem(stETH, 0) → no assets moved.

4. Victim calls completeWithdrawal(stETH, ""):
   - _transferAsset(stETH, victim, 0) → victim receives 0 stETH.
   - Victim has lost 1e18 rsETH with no recovery path.
```
