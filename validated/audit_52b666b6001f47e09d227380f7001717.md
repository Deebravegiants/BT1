### Title
State Mutation Before Deposit Check in `submit_participant_info` Allows Free Attestation Registration - (File: crates/contract/src/lib.rs)

### Summary

`submit_participant_info` in `MpcContract` mutates `tee_state.stored_attestations` (via `add_participant`) **before** verifying that the caller attached sufficient deposit to cover storage costs. Because `#[handle_result]` in NEAR SDK does not roll back state on `Err` return (only a `panic` does), the attestation is permanently committed to contract storage even when the deposit check subsequently fails and returns `Err`. Any unprivileged caller with a valid attestation can register themselves in `stored_attestations` at zero cost, breaking the storage-accounting invariant and corrupting participant-state.

---

### Finding Description

In `crates/contract/src/lib.rs`, `submit_participant_info` follows this order:

1. Record `initial_storage = env::storage_usage()` (line 780).
2. Call `self.tee_state.add_participant(...)` which **inserts** the `NodeAttestation` into `stored_attestations` (lines 796–815).
3. Compute `storage_used` and `cost` (lines 831–832).
4. If `attached < cost`, **return `Err`** (lines 835–841). [1](#0-0) 

`add_participant` unconditionally writes to `stored_attestations`: [2](#0-1) 

In NEAR SDK, `#[handle_result]` serializes the `Err` value as the function's return data and completes the call **without panicking**. Because no panic occurs, the NEAR runtime writes the mutated contract state back to storage. The `Err` path at line 836 therefore does **not** undo the `stored_attestations` insert that happened at line 796. [3](#0-2) 

The analog to the external report is exact:

| External (GmxAdapter) | NEAR MPC (MpcContract) |
|---|---|
| Checks `balance > 0` then transfers `feeAmount` that may exceed balance → revert, state not cleaned up | Inserts attestation into state, then checks `attached < cost` → `Err` returned, state mutation committed |

---

### Impact Explanation

**Medium** — participant-state and contract execution-flow manipulation that breaks production safety/accounting invariants.

1. **Free storage consumption**: An attacker with a valid TEE attestation (or a `Mock` attestation in non-TEE deployments) calls `submit_participant_info` with 0 yoctoNEAR attached. The attestation is written to `stored_attestations` and the contract pays for the storage from its own balance, violating the invariant that callers must fund their own storage.

2. **Participant-state corruption**: `stored_attestations` is the authoritative source for TEE validation (`is_caller_an_attested_participant`, `reverify_participants`, `reverify_and_cleanup_participants`). An attacker who is already a governance participant can refresh or replace their attestation entry without paying, bypassing the economic guard that prevents spam re-registrations.

3. **`accept_requests` flag manipulation**: `reverify_and_cleanup_participants` drives the `accept_requests` flag. Injecting a free attestation entry can influence which participants are counted as "valid" during TEE validation sweeps, potentially affecting whether the contract accepts or rejects signing requests. [4](#0-3) 

---

### Likelihood Explanation

**Medium** — the entry path is a public, payable contract method callable by any NEAR account. The only prerequisite is a valid attestation (or `Mock` in non-TEE mode). No privileged role, threshold collusion, or network-level access is required. The attacker simply omits the deposit.

---

### Recommendation

Move the deposit check **before** any state mutation. Compute the expected storage cost using a pre-calculated upper bound (or a two-phase approach: check deposit first, then mutate), or use `env::panic_str` instead of `return Err(...)` so that NEAR's panic-based rollback undoes the `stored_attestations` insert.

```rust
// Correct order: check deposit BEFORE mutating state
let initial_storage = env::storage_usage();
// ... compute expected cost ...
let attached = env::attached_deposit();
if attached < expected_cost {
    env::panic_str("InsufficientDeposit");
}
// Only now mutate state:
self.tee_state.add_participant(...)?;
```

Alternatively, replace `return Err(...)` with `env::panic_str(...)` at the deposit-check site so the NEAR runtime rolls back the already-committed `stored_attestations` insert.

---

### Proof of Concept

1. Deploy the contract in non-TEE mode (Mock attestations accepted).
2. Call `submit_participant_info` with a valid `Attestation::Mock(MockAttestation::Valid)` and `attached_deposit = 0`.
3. Observe: the call returns `Err(InsufficientDeposit {...})` (the transaction is marked failed from the caller's perspective).
4. Query `get_attestation(tls_public_key)` — the attestation **is present** in `stored_attestations` despite the failed deposit check.
5. The contract's own NEAR balance has decreased by the storage cost, while the caller paid nothing. [5](#0-4) [6](#0-5)

    /// Adds a participant attestation for the given node iff the attestation succeeds verification.
    pub(crate) fn add_participant(
        &mut self,
        node_id: NodeId,
        attestation: Attestation,
        tee_upgrade_deadline_duration: Duration,
    ) -> Result<ParticipantInsertion, AttestationSubmissionError> {
        let expected_report_data: ReportData = ReportDataV1::new(
            *node_id.tls_public_key.as_bytes(),
            *node_id.account_public_key.as_bytes(),
        )
        .into();

        let accepted_measurements = self.get_accepted_measurements();
        // TODO(#3264): run DCAP in the verifier contract (Promise + callback) and
        // do the post-DCAP checks here, instead of verifying locally in-WASM.
        let AcceptedAttestation {
            attestation: verified_attestation,
            advisory_ids,
        } = attestation.verify_locally(
            expected_report_data.into(),
            Self::current_time_seconds(),
            &self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration),
            &self.get_allowed_launcher_compose_hashes(),
            &accepted_measurements,
        )?;

        log_informational_advisory_ids(&advisory_ids);

        let tls_pk = node_id.tls_public_key.clone();

        // Authorization: a TLS key registered to one account must not be
        // overwritten by a submission from a different account. Without this,
        // any caller could replace any participant's stored attestation, since
        // the entry is keyed only by `tls_public_key`.
        if let Some(existing) = self.stored_attestations.get(&tls_pk)
            && existing.node_id.account_id != node_id.account_id
        {
            return Err(AttestationSubmissionError::TlsKeyOwnedByOtherAccount);
        }

        let insertion = self.stored_attestations.insert(
            tls_pk,
            NodeAttestation {
                node_id,
                verified_attestation,
            },
        );

        Ok(match insertion {
            Some(_previous_attestation) => ParticipantInsertion::UpdatedExistingParticipant,
            None => ParticipantInsertion::NewlyInsertedParticipant,
        })
