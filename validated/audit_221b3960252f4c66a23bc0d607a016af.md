Audit Report

## Title
`PriceVelocityGuardExtension` Uses `block.number` Instead of `block.timestamp` to Measure Price Velocity, Making the Guard Systematically Miscalibrated on L2s - (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary

`PriceVelocityGuardExtension.beforeSwap` measures elapsed time using `block.number` and a per-block rate parameter (`maxChangePerBlockE18`). On fast-block L2s (Arbitrum ~0.25 s/block), the same parameter value produces a guard that is up to 48× weaker per real-time second than on Ethereum mainnet (~12 s/block). Swaps at oracle prices that moved far beyond the intended velocity envelope are accepted, exposing LPs to bad-price execution and direct loss of assets.

## Finding Description

`PriceVelocityGuardExtension.beforeSwap` computes the allowed price change as:

```solidity
uint256 blockDiff = block.number - prevBlock;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
```

The state struct stores `lastUpdateBlock` as a `uint64` block number, not a timestamp:

```solidity
struct PriceVelocityState {
    uint128 lastMidPriceX64;
    uint64 lastUpdateBlock;
    uint64 maxChangePerBlockE18;
}
```

Both `beforeSwap` and `setLastMidPrice` write `block.number` into `lastUpdateBlock`. The NatSpec and parameter name explicitly frame the guard as "per block." On Arbitrum (~0.25 s/block), 48 blocks elapse per 12 real seconds. With `maxChangePerBlockE18 = 1e16` (1% per block):

- **Mainnet** (1 block per 12 s): `allowedSq = (1e16)² × 2 = 2e32` → allows ~1.41% change
- **Arbitrum** (48 blocks per 12 s): `allowedSq = (1e16)² × 49 = 4.9e33` → allows ~7% change

A 6% oracle price move in 12 real seconds passes on Arbitrum (`3.6e33 < 4.9e33`) but reverts on mainnet (`3.6e33 > 2e32`). The guard is bypassed by any swap caller on an L2 pool — no privileged action required. The existence of `PriceProviderL2.sol` with L2-aware staleness logic confirms L2 deployment is in scope.

## Impact Explanation

The velocity guard is the sole on-chain rate-of-change check on the oracle mid-price in the swap path. When miscalibrated on L2, swaps execute at oracle prices that moved beyond the intended velocity envelope — matching the allowed impact gate of **bad-price execution** (unbounded bid/ask quote reaches a pool swap) and **direct loss of LP assets**. The loss is permanent once the swap settles tokens.

## Likelihood Explanation

Any pool admin who deploys `PriceVelocityGuardExtension` on an L2 and sets `maxChangePerBlockE18` using mainnet block-time intuition (or copies a mainnet value) silently deploys a guard that is orders of magnitude weaker than intended. No privileged attacker is required: any public `swap()` call triggers `beforeSwap`, and the miscalibrated guard simply does not revert when it should. The protocol explicitly targets multi-chain deployment.

## Recommendation

Replace `block.number` / `lastUpdateBlock` / `maxChangePerBlockE18` with `block.timestamp` / `lastUpdateTimestamp` / `maxChangePerSecondE18` throughout `PriceVelocityGuardExtension` and `IPriceVelocityGuardExtension`. The velocity formula becomes:

```solidity
uint256 secondsDiff = block.timestamp - prevTimestamp;
uint256 allowedSq = uint256(maxChangePerSecond) * uint256(maxChangePerSecond) * (1 + secondsDiff);
```

`block.timestamp` manipulation by validators is bounded to a few seconds and is non-cumulative, making it far safer than block-number arithmetic for cross-chain rate limiting.

## Proof of Concept

```
Scenario: pool deployed on Arbitrum (~0.25 s/block).
Admin sets maxChangePerBlockE18 = 1e16 (1% per block),
intending to allow ~0.083% per second (mainnet 12 s/block).

Step 1: Admin deploys pool on Arbitrum, attaches PriceVelocityGuardExtension,
        calls setMaxChangePerBlock(pool, 1e16).
Step 2: At block N, a swap executes; lastMidPriceX64 = P, lastUpdateBlock = N.
Step 3: 12 real seconds later (block N+48 on Arbitrum), oracle price has moved 6%.
        blockDiff = 48
        changeE18 = 6e16
        actualSq  = (6e16)^2 = 3.6e33
        allowedSq = (1e16)^2 * 49 = 4.9e33
        3.6e33 < 4.9e33 → guard PASSES, swap executes at manipulated price.
Step 4: On mainnet with the same 12-second gap (1 block):
        blockDiff = 1
        allowedSq = (1e16)^2 * 2 = 2e32
        3.6e33 > 2e32 → guard REVERTS as intended.

LPs on Arbitrum lose funds on every swap that exploits this 7× wider envelope.
```
