### Title
Missing RSA public exponent validation in `base_pke_ek_blob_v1_t` deserialization allows KEM shared-secret recovery — (`src/cbmpc/api/pve_internal.h`, `src/cbmpc/crypto/base_rsa.cpp`)

### Summary
An attacker who can supply a crafted `base_pke_ek_blob_v1_t` blob with `e=1` can cause the RSA-OAEP KEM encapsulation to produce a "ciphertext" that is simply the OAEP-padded shared secret. Because OAEP padding is reversible without a private key, the attacker recovers the KEM shared secret and decrypts the PVE AEAD ciphertext.

### Finding Description

`parse_ek_blob` validates only the blob version field: [1](#0-0) 

`rsa_pub_key_t::convert` deserializes `n` and `e` from attacker-controlled bytes and calls `set(n, e)` with no exponent check: [2](#0-1) 

`rsa_pub_key_t::set` passes `n` and `e` directly to `EVP_PKEY_fromdata` with no validation: [3](#0-2) 

OpenSSL 3.x's `EVP_PKEY_fromdata` explicitly does **not** validate key parameters — that requires a separate `EVP_PKEY_public_check()` call, which is never made anywhere in this code path. So `e=1` is accepted without error.

`kem_policy_rsa_oaep_t::encapsulate` then generates `kem_ss`, OAEP-pads it, and calls `encrypt_raw`: [4](#0-3) 

`encrypt_oaep_with_seed` pads then calls `encrypt_raw`: [5](#0-4) 

`encrypt_raw` computes `m^e mod n`. With `e=1`, this returns `m` (the OAEP-padded `kem_ss`): [6](#0-5) 

The OAEP-padded message is not a one-way transform — it is designed to be reversed by the private key holder. An attacker who sees the "ciphertext" (which is the OAEP-padded message) can reverse the MGF1 masking to recover `kem_ss`.

`kem_aead_ciphertext_t::seal` then uses `kem_ss` to derive the AES-GCM key via HKDF: [7](#0-6) 

With `kem_ss` recovered, the attacker derives the same AES-GCM key and decrypts the AEAD ciphertext.

### Impact Explanation
The KEM shared secret is exposed in the RSA ciphertext field of the PVE output. Any party who can observe the ciphertext (which is the normal case — PVE ciphertexts are transmitted) and who supplied the malicious `ek_blob` can recover the plaintext scalar `x` encrypted by the honest party. This falls under the **High** impact category: attacker-controlled blobs are accepted under a degenerate key, producing accepted invalid cryptographic output with direct secret-recovery impact.

### Likelihood Explanation
The `ek` blob is an externally-supplied opaque byte string in the public API (`cbmpc_pve_encrypt`, `coinbase::api::pve::encrypt`). Any caller who receives an `ek` from an untrusted source (e.g., a malicious peer in a protocol, a compromised key-distribution service) can be targeted. Crafting the blob requires only knowledge of the serialization format (which is internal but fixed), and the `e=1` value is trivially encodeable as a bignum.

### Recommendation
Add an explicit RSA public exponent validation step after deserialization in `rsa_pub_key_t::convert` (or in `rsa_pub_key_t::set`). At minimum, verify that `e > 1`, `e` is odd, and `e >= 65537` (or at least `e >= 3`). This can be done using `BN_cmp` and `BN_is_odd` on the deserialized `e` before calling `EVP_PKEY_fromdata`. Alternatively, call `EVP_PKEY_public_check` after key construction and propagate any failure as a converter error.

### Proof of Concept
1. Serialize a `base_pke_ek_blob_v1_t` with `version=1`, `key_type=1` (rsa_oaep_2048), and an `rsa_pub_key_t` containing any 2048-bit composite `n` and `e=1`.
2. Call `cbmpc_pve_encrypt(curve, malicious_ek, label, x, &out_ct)`.
3. Parse the resulting `out_ct` to extract the `kem_ct` field (the RSA-OAEP ciphertext).
4. Reverse the OAEP padding on `kem_ct`: compute `seed = maskedSeed XOR MGF1(maskedDB)`, then `DB = maskedDB XOR MGF1(seed)`, then extract `kem_ss` from `DB`.
5. Derive `aes_key = HKDF-Expand(HKDF-Extract(kem_ss), "CBMPC|KEM-AEAD|v1|KDF=HKDF-SHA256|AEAD=AES-GCM-256", 32)`.
6. Decrypt `aead_ciphertext` with `aes_key` and the `iv` from the ciphertext to recover `x`.
7. Assert `x` matches the original plaintext scalar.

error_t rsa_pub_key_t::encrypt_raw(mem_t in, buf_t& out) const {
  int n_size = size();
  if (n_size != in.size) return coinbase::error(E_CRYPTO);

  scoped_ptr_t<EVP_PKEY_CTX> ctx = EVP_PKEY_CTX_new(ptr, NULL);
  if (EVP_PKEY_encrypt_init(ctx) <= 0) return openssl_error("RSA encrypt RAW error");
  if (EVP_PKEY_CTX_set_rsa_padding(ctx, RSA_NO_PADDING) <= 0) return openssl_error("RSA encrypt RAW error");
  size_t outlen = n_size;
  if (EVP_PKEY_encrypt(ctx, out.alloc(n_size), &outlen, in.data, in.size) <= 0)
    return openssl_error("RSA encrypt RAW error");
  return SUCCESS;
}
