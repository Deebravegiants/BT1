### Title
Cross-Chain Replay: `verify_foreign_transaction` Does Not Prevent Re-Signing the Same Foreign Transaction — (`crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` entry point in the MPC contract maintains only a *pending* request map. Once a request is resolved and removed from that map, no completed-request record is kept. Any unprivileged caller can immediately re-submit the identical foreign-transaction verification request, causing the MPC network to produce a second (and third, etc.) valid threshold signature over the same foreign-chain event. Bridge contracts that rely on the MPC signature as the sole proof of a foreign-chain deposit can be double-spent.

---

### Finding Description

`verify_foreign_transaction` stores each in-flight request in `pending_verify_foreign_tx_requests`, keyed by `VerifyForeignTransactionRequest`:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,   // contains tx_id
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [1](#0-0) 

The struct carries no caller identity, no nonce, and no timestamp — it is fully determined by the foreign-chain transaction being verified.

When `respond_verify_foreign_tx` is called by an MPC node, `resolve_yields_for` **removes** the entry from the map and resumes all queued yields:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

After removal, the contract holds **no record** that this foreign transaction was ever verified. There is no `completed_verify_foreign_tx_requests` set, no nullifier map, and no per-`tx_id` flag anywhere in contract state. [3](#0-2) 

`verify_foreign_transaction` itself performs no prior-completion check before enqueuing a new yield:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(...);
    // chain support check only
    self.enqueue_yield_request(
        method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
        callback_args,
        callback_gas,
        move |this, id| this.add_verify_foreign_tx_request(request, id),
    );
}
``` [4](#0-3) 

The design explicitly allows different callers to queue under the same key (confirmed by the fan-out test): [5](#0-4) 

This means that after a legitimate verification completes, **any account** can re-submit the identical `(tx_id, domain_id, payload_version)` tuple. The contract accepts it, the MPC nodes observe it as a new on-chain request, verify the foreign transaction (which remains permanently valid on the foreign chain), and produce a fresh threshold signature over the same payload.

---

### Impact Explanation

The primary use case documented for `verify_foreign_transaction` is the Omnibridge inbound flow: a bridge service submits a foreign-chain deposit transaction, the MPC network verifies it and returns a threshold signature, and the bridge contract mints the corresponding tokens on NEAR. [6](#0-5) 

Because the MPC contract issues a second (and arbitrarily many) valid threshold signatures for the same foreign-chain `tx_id`, any bridge contract that:
1. accepts a `VerifyForeignTransactionResponse` as proof of a deposit, and
2. does not maintain its own nullifier set for already-processed foreign `tx_id`s

can be drained by replaying the same MPC-signed attestation. The attacker pays only 1 yoctonear per re-submission. The foreign transaction is permanently finalized on the foreign chain, so every re-submission succeeds. This constitutes a **cross-chain replay enabling double-spend** of bridge funds controlled by the MPC-attested flow.

---

### Likelihood Explanation

- **Attacker capability required:** none beyond a funded NEAR account (1 yoctonear deposit).
- **No privileged access needed:** `verify_foreign_transaction` is a public payable method.
- **Foreign transaction remains valid indefinitely:** Bitcoin/Ethereum/Starknet transactions do not expire; the MPC nodes will always confirm them.
- **No rate limit or cooldown** exists on re-submission of a completed request.
- **Bridge contracts are the intended consumers** of this API and are explicitly expected to act on the returned signature without a second source of truth.

Likelihood is **High**.

---

### Recommendation

Maintain a `LookupSet` (or equivalent) of completed foreign-transaction verifications, keyed by `VerifyForeignTransactionRequest` (or at minimum by `(ForeignChain, tx_id, domain_id)`). Before enqueuing a new yield in `verify_foreign_transaction`, reject the call if the key is already present in the completed set:

```rust
if self.completed_verify_foreign_tx_requests.contains(&request) {
    env::panic_str("ForeignTransactionAlreadyVerified");
}
```

Alternatively, document explicitly and prominently that the MPC contract provides **no replay protection** for foreign-transaction verifications, and that every consuming bridge contract **must** maintain its own nullifier set keyed on `tx_id`. Given that the contract is the trust anchor for the bridge flow, the former approach (contract-side protection) is strongly preferred.

---

### Proof of Concept

1. Alice calls `verify_foreign_transaction` with `BitcoinRpcRequest { tx_id: [0xAB; 32], confirmations: 2, extractors: [BlockHash] }`.
2. MPC nodes verify the Bitcoin transaction, call `respond_verify_foreign_tx`, and the entry is removed from `pending_verify_foreign_tx_requests`.
3. Alice receives `VerifyForeignTransactionResponse { payload_hash, signature }` and calls the bridge contract's `mint` function, receiving 1 BTC worth of wrapped tokens.
4. Attacker calls `verify_foreign_transaction` with the **identical** `BitcoinRpcRequest`.
5. The contract accepts it (no completed-request check), MPC nodes re-verify the same Bitcoin transaction (still finalized), and produce a second valid threshold signature.
6. Attacker receives a second `VerifyForeignTransactionResponse` with a valid MPC signature over the same `payload_hash`.
7. Attacker calls the bridge contract's `mint` function with the second response, receiving another 1 BTC worth of wrapped tokens — a double-spend.

The root cause is the absence of any completed-request tracking after `resolve_yields_for` removes the entry: [7](#0-6)

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
