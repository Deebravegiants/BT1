### Title
`directDeposits` Credited to Account Balances Without Value Conservation Coverage ‚Äî (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, a transaction body may include a `directDeposits` field that unconditionally adds ADA to registered account balances via `applyDirectDeposits`. However, the sum of those deposits is absent from both the `consumed` and `produced` sides of the value-conservation check (`consumed pp utxo‚ÇÄ txb = produced pp certState txb`). An unprivileged transaction author can therefore specify an arbitrarily large `directDeposits` map and have those coins credited to account balances without any corresponding deduction from UTxO inputs, creating ADA out of thin air.

---

### Finding Description

**Vulnerability class:** funds/accounting bug ‚Äî balance credited without backing (direct analog of the GasAccounting `reconcile()` issue).

**Root cause ‚Äî `directDeposits` absent from `produced`:**

`dijkstraProducedValue` delegates to `conwayProducedValue` for the top-level body and to `dijkstraSubTxProducedValue` for each sub-transaction body:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody          -- outputs + fee + deposits + donation
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)

dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody <> txBody ^. treasuryDonationTxBodyL)
    <> burnedMultiAssets txBody
```

Neither branch adds `txBody ^. directDepositsTxBodyL` to the produced value. [1](#0-0) [2](#0-1) 

**Root cause ‚Äî `directDeposits` absent from `consumed`:**

`getConsumedDijkstraValue` calls `getConsumedMaryValue` (UTxO inputs + withdrawals + deposit refunds) for every body level; `directDeposits` are not mentioned. [3](#0-2) 

**Value-conservation check in UTXO rule:**

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs  line 381
{- consumed pp utxo‚ÇÄ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
```

Because `directDeposits` appear in neither side, the check passes regardless of how large the deposit map is. [4](#0-3) 

**Application of `directDeposits` in ENTITIES rule:**

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts

pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

The only guard is that every target credential is a registered account. There is no check that the deposited amounts are covered by UTxO inputs. [5](#0-4) 

The same pattern is replicated verbatim in the `SUBENTITIES` rule for sub-transactions: [6](#0-5) 

`applyDirectDeposits` itself carries an explicit disclaimer that it performs no backing check:

```haskell
-- libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs
-- | Add each direct-deposit amount to the matching account balance.
--
-- /Note/ - There are no checks that direct deposits mention only registered accounts.
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [7](#0-6)

### Impact Explanation

**Critical ‚Äî Direct creation of ADA through an invalid ledger state transition.**

Account balances are part of the global ADA supply (the preservation-of-value invariant covers `Circulation + Deposits + Fees + Treasury + Rewards + Reserves`). Adding to account balances without a matching deduction from any other pot violates this invariant. An attacker can inflate their own (or any registered) account balance by an arbitrary amount, effectively minting unbacked ADA. This constitutes a direct, permanent loss of ledger integrity equivalent to counterfeiting.

---

### Likelihood Explanation

Any transaction author (no special role required) can craft a Dijkstra-era transaction body with a non-empty `directDeposits` map targeting any registered account. The UTXO rule accepts the transaction because the value-conservation check is blind to `directDeposits`. The ENTITIES rule then applies the deposits unconditionally. The attack requires only:

1. A registered target account (trivially obtained by registering one's own stake credential).
2. A valid Dijkstra-era transaction that otherwise satisfies all other checks (valid inputs, correct fee, etc.).

No governance majority, privileged key, or consensus threshold is needed.

---

### Recommendation

Add the sum of `directDeposits` to the **produced** side of the value-conservation equation so that UTxO inputs must cover the deposited amounts:

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject ( getTotalDepositsTxBody pp isRegPoolId txBody
             <> txBody ^. treasuryDonationTxBodyL
             <> fold (unDirectDeposits (txBody ^. directDepositsTxBodyL))  -- ADD THIS
              )
    <> burnedMultiAssets txBody
```

Apply the same correction to `conwayProducedValue` (or its Dijkstra override) for the top-level body. The `consumed` side does not need to change because `directDeposits` represent a new outflow from the UTxO, not an inflow.

---

### Proof of Concept

1. Register stake credential `C` and obtain its `AccountAddress` `A`.
2. Construct a Dijkstra top-level transaction `tx` with:
   - Any valid UTxO input covering only the transaction fee.
   - `directDeposits = { A ‚Üí 1_000_000_000_000 }` (one trillion lovelace).
   - No additional outputs beyond change.
3. Submit `tx`. The UTXO rule runs `validateValueNotConservedUTxO`: `consumed` = input value; `produced` = output value + fee. Both sides balance because `directDeposits` is invisible to both. The check passes.
4. The ENTITIES rule calls `directDepositsMissingAccounts`: `A` is registered, so no failure. `applyDirectDeposits` adds 1 trillion lovelace to `C`'s account balance.
5. The attacker now holds 1 trillion lovelace in their account balance that was never deducted from any UTxO or other pot ‚Äî ADA created out of thin air.
