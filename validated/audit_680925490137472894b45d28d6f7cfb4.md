### Title
Single Participant Can Permanently Block All `verify_foreign_transaction` Requests via Empty Chain-Support Registration - (`crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which computes the **strict intersection** of every active participant's registered chain set. A single Byzantine participant (below the signing threshold) can call `register_foreign_chain_support` with an empty set at any time, collapsing the intersection to empty and causing every `verify_foreign_transaction` call to panic with `ForeignChainNotSupported`. This is the direct analog of the Velodrome "wrong pool" manipulation: just as the attacker there manipulated an external pool to make the routing check always fail, here the attacker manipulates on-chain participant state to make the chain-support check always fail.

---

### Finding Description

`verify_foreign_transaction` in `crates/contract/src/lib.rs` calls `get_supported_foreign_chains()` and panics if the requested chain is absent:

```rust
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported { requested: requested_chain }.to_string(),
    );
}
``` [1](#0-0) 

`get_supported_foreign_chains()` computes the strict intersection: a chain is returned only if **every** active participant has registered it:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
if all_active_nodes_supports_chain {
    Some(foreign_chain)
} else {
    None
}
``` [2](#0-1) 

`register_foreign_chain_support` accepts an arbitrary (including empty) `SupportedForeignChains` set from any participant, with no validation or rate limiting:

```rust
pub fn register_foreign_chain_support(
    &mut self,
    foreign_chain_support: dtos::SupportedForeignChains,
) -> Result<(), Error> {
    let account_id = self.voter_or_panic();
    self.node_foreign_chain_support
        .foreign_chain_support_by_node
        .insert(account_id, foreign_chain_support);
    Ok(())
}
``` [3](#0-2) 

The design document explicitly acknowledges this as an open issue: "A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down."



The proposed fix (`get_available_foreign_chains()` using a threshold-based count) exists in the contract but `verify_foreign_transaction` still calls the legacy strict-intersection method `get_supported_foreign_chains()`. [4](#0-3) 

---

### Impact Explanation

A single Byzantine participant (strictly below the signing threshold) can call `register_foreign_chain_support({})` at any time. This immediately collapses `get_supported_foreign_chains()` to the empty set, causing every subsequent `verify_foreign_transaction` call to panic. The Omnibridge inbound flow (foreign chain → NEAR) is completely halted. Users who have already submitted foreign-chain transactions cannot obtain MPC attestations; their bridge flows are stalled for as long as the attacker sustains the registration. The attacker can front-run specific `verify_foreign_transaction` calls by monitoring the mempool and re-registering an empty set just before each attempt, selectively blocking individual bridge operations. This breaks the production safety invariant that a whitelisted chain remains serviceable as long as a signing-threshold of nodes cover it.

**Impact: Medium** — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

Any single active participant can execute this attack with one cheap contract call. No collusion, no key material, and no threshold is required. The attack is reversible (the attacker can re-register with chains), but it can be sustained indefinitely and targeted at specific transactions. The attacker's incentive could be to block a competitor's bridge transaction or to extract a ransom from bridge operators.

---

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which uses the threshold-based count already implemented in `ForeignChainsMetadata::update_available_chains_config_cache`. A chain is available only when ≥ `signing_threshold` active participants cover it, so no single participant below the threshold can collapse the set to empty.

```rust
// Before (strict intersection — one participant can zero it out):
let supported_chains = self.get_supported_foreign_chains();

// After (threshold-based — tolerates up to n − threshold non-registering nodes):
let supported_chains = self.get_available_foreign_chains();
``` [5](#0-4) 

This is the fix already proposed in `docs/design/calculating-supported-foreign-chains.md` (tracked as issue #3434) but not yet wired into `verify_foreign_transaction`.

---

### Proof of Concept

1. Network has `n` participants with signing threshold `t < n`. All participants have registered `{Bitcoin}` as their supported chain. `verify_foreign_transaction` for Bitcoin succeeds.
2. Attacker (participant `p`, where `p` is any single participant) calls:
   ```
   register_foreign_chain_support({ foreign_chain_support: [] })
   ```
3. `get_supported_foreign_chains()` now computes: `{Bitcoin} ∩ {} = {}` (empty intersection because participant `p` registered empty).
4. Any user calling `verify_foreign_transaction` for Bitcoin now receives:
   ```
   panicked: Requested foreign chain, Bitcoin, is not supported.
   ```
5. The Omnibridge inbound flow is completely halted. The attacker can sustain this indefinitely by re-registering `{}` after any honest re-registration by `p`. [1](#0-0) [2](#0-1)

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
