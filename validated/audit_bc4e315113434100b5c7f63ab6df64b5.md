### Title
Instantaneous Single-RPC-Provider Foreign Chain State Used Without Cross-Node Consensus Enables Forged Bridge Attestations - (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

Each MPC node independently queries a **single** RPC provider and uses the instantaneous response to compute the signing payload hash (`payload_hash = SHA-256(borsh(ForeignTxSignPayload{request, values}))`). There is no cross-node consensus on the observed `values`. The on-chain `respond_verify_foreign_tx` contract function accepts any `payload_hash` as long as the MPC signature over it is valid, without re-deriving or bounding the hash to the submitted `request`. An adversarial foreign-chain input producer who controls the RPC endpoints queried by a signing-threshold of nodes can cause the MPC network to produce and submit a valid threshold signature over a fabricated observation, enabling forged bridge attestations.

---

### Finding Description

**Node-side (instantaneous, single-source state):**

In `make_verify_foreign_tx_leader` and `make_verify_foreign_tx_follower`, every participating node independently calls `execute_foreign_chain_request`, which queries exactly one RPC provider and uses the raw, instantaneous response:

```rust
// make_verify_foreign_tx_leader (lines 73-80)
let response_payload = self
    .execute_foreign_chain_request(
        &foreign_tx_request.request,
        foreign_tx_request.payload_version,
    )
    .await?;
let sign_request = build_signature_request(&foreign_tx_request, &response_payload)?;
```

```rust
// make_verify_foreign_tx_follower (lines 103-110)
let response_payload = self
    .execute_foreign_chain_request(
        &foreign_tx_request.request,
        foreign_tx_request.payload_version,
    )
    .await?;
let sign_request = build_signature_request(&foreign_tx_request, &response_payload)?;
```

The design document explicitly confirms: *"each MPC node only queries 1 RPC provider per verification request"* (consistent-hash-based deterministic assignment). There is no per-node quorum across multiple providers, no time-averaging, and no cross-node agreement on the extracted `values` before signing begins.

`build_signature_request` hashes the full `ForeignTxSignPayload{request, values}` into the 32-byte `msg_hash` that drives the ECDSA signing protocol. The `values` field — the only part that encodes what was actually observed on the foreign chain — is derived entirely from the single instantaneous RPC call.

**Contract-side (no re-derivation of payload_hash from request):**

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` (lines 718–747) only checks that the submitted signature is cryptographically valid against the caller-supplied `response.payload_hash`:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

The contract does **not** re-derive or bound `payload_hash` to the pending `request`. It cannot know what `values` were observed, so it cannot verify that `payload_hash` encodes the correct `(request, values)` pair. Any `payload_hash` for which a valid threshold signature exists is accepted and used to resolve the pending yield.

**The combined gap (direct analog to the external report):**

Just as `AutoRedemption::checkUpkeep` and `calculateUSDsToTargetPrice` both consumed the same instantaneous, manipulable `sqrtPriceX96` — and the contract never re-checked the condition at fulfillment time — here:

- The **trigger** (whether the foreign transaction is considered verified) and the **payload** (what is signed and returned to the caller) both derive from the same instantaneous, single-provider RPC response.
- The contract never re-checks or bounds the `payload_hash` against the original `request` at response time.

If an adversary can cause all `threshold` participating nodes to receive the same false RPC response, the signing protocol converges on a valid threshold signature over a fabricated `payload_hash`, and the contract accepts it unconditionally.

---

### Impact Explanation

**Impact: High — Forged foreign-chain verification causing invalid bridge execution.**

The `verify_foreign_transaction` flow is the trust anchor for the Omnibridge inbound path: NEAR contracts are supposed to react to external chain events *only after* the MPC network attests that a specific foreign transaction finalized with specific observable values (e.g., `BlockHash`). A forged attestation breaks this guarantee:

- The Omnibridge (or any consumer of `VerifyForeignTransactionResponse`) receives a signed `payload_hash` that encodes fabricated `values` (e.g., a block hash that does not correspond to the real transaction, or a confirmation count that was not actually reached).
- The consumer has no way to distinguish this from a legitimate attestation — the signature is valid against the MPC public key.
- This enables double-spend or invalid-release conditions: funds can be released on NEAR for a foreign-chain deposit that never occurred, or for a deposit with different parameters than attested.

---

### Likelihood Explanation

**Likelihood: Medium.**

The attack requires causing `threshold` MPC nodes to receive the same false RPC response for the same request. Each node queries a deterministically assigned provider. Attack vectors include:

1. **Compromised or malicious RPC provider**: A single provider serving multiple nodes (e.g., a popular public endpoint) can return false data for a targeted `tx_id`. The design doc explicitly acknowledges: *"A malicious or faulty provider could return incorrect data for a subset of nodes."*
2. **Chain reorganization at query time**: On chains with probabilistic finality, a reorg between the time nodes query and the time the foreign chain finalizes can cause nodes to observe a block hash that is later orphaned — and the MPC signature attests to the orphaned state.
3. **Targeted provider compromise**: An attacker who can compromise `threshold` distinct providers (one per participating node, per the deterministic assignment) can fully control the attested values.

The design doc notes this is a known risk and defers the fix (per-node multi-provider quorum) to a future milestone.

---

### Recommendation

1. **Per-node RPC quorum (direct analog to TWAP):** Each node should query all its locally configured providers for the chain and accept a result only when ≥ `rpc_quorum(C)` return the same response. This is already described as a planned requirement in the design doc (*"A quorum of those RPC providers must agree on the verification; if fewer than the quorum agree, the node errors out"*) but is not yet implemented in `execute_foreign_chain_request`.

2. **Cross-node value agreement before signing:** Before the leader initiates the MPC signing round, nodes should exchange and agree on the extracted `values` (or their hash) via an authenticated broadcast, so that a single compromised provider cannot unilaterally determine the signed payload.

3. **Contract-side re-check:** While the contract cannot re-derive `values` from the foreign chain, it should at minimum verify that `response.payload_hash` is the hash of `ForeignTxSignPayload{request=<the pending request>, values=<something>}` by requiring the caller to submit the `values` alongside the response and recomputing the hash on-chain. This prevents a malicious leader from submitting a `payload_hash` that encodes a completely different `request`.

---

### Proof of Concept

**Setup:** MPC network with `n=5` nodes, `threshold=3`. Bitcoin `BlockHash` extractor. Attacker controls the RPC provider assigned (by consistent hash) to nodes 1, 2, and 3 for a specific `request_id`.

**Steps:**

1. Attacker calls `verify_foreign_transaction` with a `BitcoinRpcRequest { tx_id: T, confirmations: 6, extractors: [BlockHash] }` for a real transaction `T` that is in block `B_real`.

2. Attacker's controlled RPC provider returns `blockhash = B_fake` (a different block hash) for `tx_id = T` to nodes 1, 2, 3.

3. Nodes 1, 2, 3 each independently call `execute_foreign_chain_request` → `inspector.extract(T, 6, [BlockHash])` → receive `B_fake` → compute:
   ```
   payload_hash = SHA-256(borsh(ForeignTxSignPayloadV1 { request: BitcoinRpcRequest{T,6,[BlockHash]}, values: [BitcoinExtractedValue::BlockHash(B_fake)] }))
   ```

4. All three nodes sign `payload_hash` via the OT-based ECDSA threshold protocol. The coordinator (leader, node 1) collects shares and assembles a valid threshold signature `sig`.

5. Node 1 calls `respond_verify_foreign_tx(request, VerifyForeignTransactionResponse { payload_hash, signature: sig })`.

6. The contract verifies `verify_ecdsa_signature(sig, payload_hash, mpc_pubkey)` → valid. The pending yield is resolved and `VerifyForeignTransactionResponse { payload_hash: SHA-256(...B_fake...), signature: sig }` is returned to the caller.

7. The Omnibridge (or other consumer) calls `ForeignChainSignatureVerifier::verify_signature` with `expected_extracted_values = [BlockHash(B_fake)]` — which matches the attested hash — and accepts the forged attestation as proof that transaction `T` was included in block `B_fake`. pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
