### Title
Farmer Fee Threshold Unenforced: Malicious Harvester Can Unconditionally Redirect Farmer Block Reward to Arbitrary Address — (`chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

### Summary

A malicious third-party harvester (the explicit CHIP-22 threat model) can redirect the farmer's block reward to any attacker-controlled puzzle hash by sending a `NewProofOfSpace` with `farmer_reward_address_override` set to an attacker address. The CHIP-22 fee threshold check in `notify_farmer_reward_taken_by_harvester_as_fee` is **log-only** — it emits a warning but never blocks the flow. The farmer unconditionally accepts the override in `RespondSignatures` and builds a `DeclareProofOfSpace` with the attacker's puzzle hash as `farmer_puzzle_hash`, which the full node uses verbatim to assign the block reward.

### Finding Description

**Step 1 — Entrypoint: `new_proof_of_space` in `farmer_api.py`**

The farmer receives `NewProofOfSpace` from the harvester. It validates the proof and checks `required_iters`: [1](#0-0) 

If the proof is block-quality and `farmer_reward_address_override` is set, it calls `notify_farmer_reward_taken_by_harvester_as_fee`. This function is **purely advisory**: [2](#0-1) 

The function calculates `fee_quality` from the actual proof bytes and compares it to `fee_info.applied_fee_threshold`. When the threshold fails (`fee_quality > fee_threshold`), it logs two warnings — but **returns `None` in all cases**, never blocking the flow.

**Step 2 — Farmer proceeds unconditionally**

After `notify_farmer_reward_taken_by_harvester_as_fee` returns, the farmer sends `RequestSignatures` to the harvester regardless of whether the threshold passed or failed: [3](#0-2) 

**Step 3 — `_process_respond_signatures` accepts the override without any fee check**

When the harvester responds with `RespondSignatures` (which also carries `farmer_reward_address_override`), the farmer unconditionally substitutes the attacker's address: [4](#0-3) 

There is no fee threshold re-check here. The `DeclareProofOfSpace` is then built with `farmer_reward_address = attacker_ph`: [5](#0-4) 

**Step 4 — Full node uses the attacker's puzzle hash verbatim**

The full node's `declare_proof_of_space` handler reads `request.farmer_puzzle_hash` directly from the `DeclareProofOfSpace` message and uses it as the farmer reward address in the block: [6](#0-5) 

There is no validation at the full node that the `farmer_puzzle_hash` matches the farmer's configured address.

### Impact Explanation

A malicious third-party harvester with a valid block-quality proof can redirect 100% of the farmer's block reward (the `FEE_REWARD` coin, currently 0.25 XCH + transaction fees) to an arbitrary attacker-controlled puzzle hash. The farmer receives nothing. This is a direct, irreversible XCH loss per won block. The impact is **Critical** under "reward diversion affecting XCH."

### Likelihood Explanation

CHIP-22 explicitly describes the third-party harvester scenario as a supported use case. The code even acknowledges the attack in its warning message ("Harvester illegitimately took a fee reward") but fails to enforce the invariant. Any farmer using a third-party harvester service is exposed. The attacker only needs a valid proof of space — the `applied_fee_threshold` value is irrelevant since it is never enforced.

### Recommendation

In `new_proof_of_space` (`farmer_api.py`), after calling `notify_farmer_reward_taken_by_harvester_as_fee`, **return `None` (abort the block flow) if the fee threshold check fails**. Specifically:

- Compute `fee_quality = calculate_harvester_fee_quality(proof, challenge_hash)`
- If `fee_info is None` or `fee_quality > fee_info.applied_fee_threshold`, drop the proof and do not send `RequestSignatures`

Alternatively, in `_process_respond_signatures`, re-validate the fee threshold before accepting `farmer_reward_address_override`, and return `None` if it fails.

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
