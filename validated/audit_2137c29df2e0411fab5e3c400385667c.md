Audit Report

## Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Per-Bin Additional Fees — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`setPoolAdminFees` enforces the factory's `maxAdminSpreadFeeE6` cap before writing admin spread fees to the pool, but `setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` directly to the pool with no cap validation. These per-bin fees are added on top of `baseFeeX64` in every swap path, allowing a pool admin to charge traders above the factory-enforced ceiling even when `maxAdminSpreadFeeE6` is set to zero.

## Finding Description
**Guarded path:** `setPoolAdminFees` checks `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6` before updating pool fees. [1](#0-0) 

**Unguarded path:** `setPoolBinAdditionalFees` passes `addFeeBuyE6`/`addFeeSellE6` directly to the pool with no cap check: [2](#0-1) 

`setBinAdditionalFees` on the pool only validates the bin index range, not the fee magnitude: [3](#0-2) 

**Fee reaches traders in every swap direction.** In `getSellAndBuyPrices` and in all four swap execution paths (`buyToken0InBinSpecifiedOut`, `buyToken0InBinSpecifiedIn`, `buyToken1InBinSpecifiedOut`, `buyToken1InBinSpecifiedIn`), the per-bin fee is added directly to `baseFeeX64`: [4](#0-3) [5](#0-4) [6](#0-5) 

`uint16` allows values up to 65,535 E6 ≈ 6.55% additional fee per bin. The factory's `HARD_MAX_SPREAD_FEE_E6 = 200_000` and the owner-configurable `maxAdminSpreadFeeE6` are both completely bypassed for this code path. [7](#0-6) 

## Impact Explanation
This is a direct admin-boundary break: the factory owner uses `maxAdminSpreadFeeE6` to guarantee traders that no pool admin can charge above a defined spread. A pool admin can set `addFeeBuyE6 = 65535` on any active bin, adding ~6.55% to the effective ask price (or reducing the effective bid price by ~6.55%) beyond the capped global fee. If `maxAdminSpreadFeeE6 = 0` is set to prohibit all admin spread fees, the pool admin can still extract up to ~6.55% per bin through this uncapped channel. Traders receive worse execution than the factory's published caps imply, constituting a direct loss of swap output relative to the advertised fee regime. This meets the "admin-boundary break: pool admin exceeds caps" allowed impact.

## Likelihood Explanation
The pool admin is a semi-trusted role with direct, permissioned access to `setPoolBinAdditionalFees`. No special conditions, flash loans, or oracle manipulation are required. The call is a single transaction executable immediately after pool deployment. Any pool admin who wishes to extract additional value from traders can do so at any time.

## Recommendation
Add a cap check in `setPoolBinAdditionalFees` mirroring the check in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce a combined cap: `globalAdminSpreadFee + addFeeBuyE6 ≤ maxAdminSpreadFeeE6`.

## Proof of Concept
1. Factory owner sets `maxAdminSpreadFeeE6 = 0` — intending to prohibit all admin spread fees.
2. Pool admin calls `setPoolAdminFees(pool, 0, 0)` — passes the cap check, no global fee set.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — **no cap check fires**.
4. `_binStates[0].addFeeBuyE6 = 65535` and `addFeeSellE6 = 65535` are written to storage.
5. On the next swap through bin 0, `buyFeeX64 = baseFeeX64 + (65535 / 1e6) * ONE_X64`, inflating the ask price by ~6.55% above the oracle mid-price plus base spread.
6. Traders pay ~6.55% more than the factory's zero-admin-fee cap permits; the excess accrues as LP spread revenue controlled by the pool admin's liquidity positions.
