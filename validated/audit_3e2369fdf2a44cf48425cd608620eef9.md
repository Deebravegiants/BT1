### Title
Unbounded O(n\_DReps) Iteration Over All Registered DReps During Transaction Processing - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

### Summary
The `updateDormantDRepExpiry` function performs a full `Map.map` traversal over every registered DRep in `vsDReps` during the processing of any transaction that contains a governance proposal and when `numDormantEpochs > 0`. Because the number of registered DReps is unbounded (limited only by the deposit requirement), an attacker can register a large number of DReps and then submit a single governance-proposal transaction to force an O(n\_DReps) state-mutating computation inside the `CERTS` ledger rule, exceeding the intended per-transaction validation budget and potentially causing honest nodes to miss block-production slots or diverge on validation timing.

### Finding Description

`updateDormantDRepExpiry` is called from `updateDormantDRepExpiries` inside `conwayCertsTransition` whenever a transaction carries at least one `ProposalProcedure` and `numDormantEpochs > 0`:

```haskell
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry   -- iterates ALL registered DReps
``` [1](#0-0) 

The caller `updateDormantDRepExpiries` fires this traversal on every transaction that contains proposals:

```haskell
updateDormantDRepExpiries tx currentEpoch =
  let hasProposals = not . OSet.null $ tx ^. bodyTxL . proposalProceduresTxBodyL
   in if hasProposals
        then certVStateL %~ updateDormantDRepExpiry currentEpoch
        else id
``` [2](#0-1) 

It is invoked unconditionally (in the pre-`hardforkConwayMoveWithdrawalsAndDRepChecksToLedgerRule` path) from `conwayCertsTransition` when the certificate list is empty: [3](#0-2) 

The `vsDReps` map that is traversed is the full set of all registered DReps stored in `VState`: [4](#0-3) 

There is no protocol-parameter cap on the number of registered DReps; the only cost is the per-DRep deposit (`ppDRepDeposit`). The `DRepState` record itself is small, so the deposit cost per DRep is bounded and predictable.

A secondary, related unbounded traversal exists in `ConwayUnRegDRep` processing inside `GovCert.hs`, where `clearDRepDelegations` iterates over the entire `drepDelegs` set (all delegators of the unregistering DRep) with `foldr`:

```haskell
clearDRepDelegations delegs accountsMap =
  foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
``` [5](#0-4) 

For a DRep with millions of delegators, a single `ConwayUnRegDRep` certificate causes an O(n\_delegators) state-mutating traversal of the accounts map during transaction validation.

### Impact Explanation

Both traversals are state-mutating (they write updated expiry values or clear delegation pointers) and occur inside the synchronous ledger-rule evaluation path that every validating node must complete before accepting a block. If the computation takes longer than the node's validation time budget (which must fit within a single slot on mainnet, approximately one second), nodes may:

- Fail to validate the block in time, causing missed block production and temporary chain forks.
- Exhibit non-deterministic acceptance depending on hardware speed, producing **deterministic disagreement between honest nodes from ledger rule evaluation** — matching the High impact category.

At minimum, an attacker can craft a transaction that forces every node to perform an O(n\_DReps) or O(n\_delegators) computation, **exceeding intended validation limits** — matching the Medium impact category.

### Likelihood Explanation

- Registering DReps requires paying `ppDRepDeposit` per DRep, which is a real cost. However, the attacker recovers deposits upon unregistration, so the net cost is only the opportunity cost of locked ADA.
- `numDormantEpochs` accumulates naturally whenever no governance proposals are submitted for an epoch. After several dormant epochs, a single proposal transaction triggers the full traversal.
- The entry path requires only an unprivileged transaction sender: no governance majority, no privileged key, no consensus threshold.
- The number of registered DReps on mainnet is already in the thousands and growing, making the traversal cost non-trivial today and worsening over time.

### Recommendation

1. **Cap the number of registered DReps** via a protocol parameter (analogous to `maxPoolNum`) to bound the worst-case traversal cost.
2. **Amortize `updateDormantDRepExpiry` across multiple transactions** using a pulsing mechanism similar to the existing `DRepPulser` / `RewardPulser` pattern, rather than applying it atomically in a single transaction.
3. **Benchmark the `updateDormantDRepExpiry` path** under realistic DRep counts (e.g., 10 000+) to establish whether it already exceeds the per-slot validation budget.
4. For `clearDRepDelegations`, consider removing the `drepDelegs` set from `DRepState` entirely and instead computing delegations on demand, or capping the number of delegators per DRep.

### Proof of Concept

1. Register N DReps (e.g., N = 10 000), each paying `ppDRepDeposit`.
2. Submit no governance proposals for several epochs, allowing `numDormantEpochs` to accumulate to a large value.
3. Submit a single transaction containing one `ProposalProcedure` and no certificates.
4. The `CERTS` rule fires `updateDormantDRepExpiries` → `updateDormantDRepExpiry` → `Map.map updateExpiry` over all N DRep entries, performing N state-mutating map updates inside a single transaction's validation path.
5. Every validating node must complete this O(N) computation synchronously before the block can be accepted, potentially exceeding the one-second slot budget at large N. [6](#0-5) [7](#0-6)

-- | Update dormant expiry for all DReps that are active.
-- And also reset the `numDormantEpochs` counter.
updateDormantDRepExpiry ::
  -- | Current Epoch
  EpochNo ->
  VState era ->
  VState era
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry
  where
    numDormantEpochs = vState ^. vsNumDormantEpochsL
    updateExpiry =
      drepExpiryL
        %~ \currentExpiry ->
          let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
           in if actualExpiry < currentEpoch
                then currentExpiry
                else actualExpiry
```
