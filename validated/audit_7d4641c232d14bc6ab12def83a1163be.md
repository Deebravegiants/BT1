### Title
Resharing State Has No Timeout/Fallback — Contract Permanently Stuck If Previous Participants Become Unavailable During Transition - (`crates/contract/src/state/resharing.rs`)

### Summary

The `ResharingContractState` has no overall timeout or emergency escape mechanism. If the new proposed participants cannot complete resharing (e.g., one goes offline) AND the previous running-state participants cannot reach the cancellation threshold (e.g., they are the ones being replaced and have gone offline), the contract is permanently stuck in the `Resharing` state. All three exit paths from `Resharing` require either the new or the old participant set to be sufficiently online, with no time-bounded fallback.

### Finding Description

When `vote_new_parameters` reaches consensus, the contract transitions from `Running` to `Resharing`. The only ways to exit `Resharing` are:

1. **`vote_reshared`** — requires **all** new proposed participants to vote successfully for every domain.
2. **`vote_cancel_resharing`** — requires **≥ threshold** votes from the *previous* running-state participants.
3. **`vote_new_parameters`** (re-proposal) — also requires the previous running-state participants to vote.

`vote_reshared` requires unanimous participation from the new set:

```rust
// key_event.rs line 129
if count == self.parameters.participants().len() {
    Ok(true)
```

A single offline new participant permanently blocks this path.

`vote_cancel_resharing` requires the previous running-state threshold:

```rust
// resharing.rs lines 178-182
let cancellation_votes_count = self.cancellation_requests.len() as u64;
let previous_running_threshold = self.previous_running_state.parameters.threshold();
let threshold_cancellation_votes_reached: bool =
    cancellation_votes_count >= previous_running_threshold.value();
```

There is no time-bounded fallback. If fewer than `threshold` previous participants are reachable, this path is also permanently blocked.

The `KeyEventInstance` has a per-attempt timeout (`expires_on`) that only resets individual attempts, not the overall `ResharingContractState`. The resharing state itself has no expiry.

The realistic trigger scenario mirrors the oracle report exactly: resharing is initiated *because* some old participants are leaving/being replaced. Those departing participants go offline before the resharing completes. Meanwhile, one new participant also fails to complete. Now:
- New participants cannot complete `vote_reshared` (one is offline).
- Old participants cannot reach threshold for `vote_cancel_resharing` (they are the ones being replaced and are offline).
- `vote_new_parameters` re-proposal also requires old participants.

The contract is permanently stuck in `Resharing` with no escape.

### Impact Explanation

While stuck in `Resharing`, `respond()` still accepts calls because `is_running_or_resharing()` returns `true`:

```rust
// lib.rs line 575
if !self.protocol_state.is_running_or_resharing() {
    return Err(InvalidState::ProtocolStateNotRunning.into());
}
```

However, `respond()` also calls `assert_caller_is_attested_participant_and_protocol_active()`, which enforces valid TEE attestations. TEE attestations have a bounded validity window (`tee_upgrade_deadline_duration_seconds`). Once the previous participants' attestations expire — which is inevitable if they are permanently offline — `respond()` calls begin failing. All queued sign, CKD, and foreign-tx requests time out. The MPC network stops producing signatures, permanently freezing cross-chain operations for all users with pending requests. This constitutes a permanent liveness failure of the chain-signature contract, matching the "contract execution-flow manipulation that breaks production safety/accounting invariants" Medium impact class.

### Likelihood Explanation

The scenario is realistic and not contrived. Resharing is the standard mechanism for rotating out departing participants. The departing participants are, by definition, going offline. If any single new participant also fails during the resharing window (network partition, TEE attestation issue, software crash), the contract enters the stuck state. No attacker is required; this is a natural operational failure that the protocol provides no recovery path for.

### Recommendation

Add a block-height-based timeout to `ResharingContractState`. After `RESHARING_TIMEOUT_BLOCKS` have elapsed since the resharing began (tracked as a `started_at_block: BlockHeight` field on `ResharingContractState`), allow `vote_cancel_resharing` to succeed with any single vote from any participant (old or new), or introduce a separate `vote_emergency_cancel` callable by any attested participant once the timeout has passed. This mirrors the recommendation in the oracle report: "ignore the blocking condition if a certain timeout has passed after the switch."

### Proof of Concept

1. Contract is `Running` with participants `{A, B, C}`, threshold 2.
2. `vote_new_parameters` is called to reshare to `{B, C, D}` (replacing A). All of `{B, C, D}` vote. Contract enters `Resharing`.
3. Participant `A` goes fully offline (it was being replaced).
4. Participant `D` crashes mid-resharing before calling `vote_reshared`.
5. `vote_reshared` is blocked: `D` is offline, unanimous vote impossible.
6. `vote_cancel_resharing` is blocked: only `B` and `C` remain from the old set `{A, B, C}`, but threshold is 2 — wait, this works with B and C. Let me adjust: threshold 3, participants `{A, B, C, E}`, resharing to `{B, C, D}`. A and E go offline. Only B and C remain from old set, threshold is 3. Cancellation requires 3 votes from `{A, B, C, E}` but only 2 are reachable.
7. `vote_new_parameters` re-proposal also requires 3 votes from `{A, B, C, E}` — blocked.

    pub fn vote_reshared(
        &mut self,
        key_event_id: KeyEventId,
    ) -> Result<Option<RunningContractState>, Error> {
        let previous_key = self.previous_keyset().domains[self.reshared_keys.len()].clone();
        if self
            .resharing_key
            .vote_success(&key_event_id, previous_key.key.clone())?
        {
            let new_key = KeyForDomain {
                domain_id: key_event_id.domain_id,
                attempt: key_event_id.attempt_id,
                key: previous_key.key,
            };
            self.reshared_keys.push(new_key);
            if let Some(next_domain) = self
                .previous_running_state
                .domains
                .get_domain_by_index(self.reshared_keys.len())
            {
                self.resharing_key = KeyEvent::new(
                    self.prospective_epoch_id(),
                    next_domain.clone(),
                    self.resharing_key.proposed_parameters().clone(),
                );
            } else {
                // Resharing complete: fold the per-domain threshold updates into
                // the registry and store the proposed parameters. The updates live
                // only on this resharing state, so they are structurally dropped
                // here rather than scrubbed off the stored parameters.

                    .previous_running_state
                    .domains
                    .with_threshold_updates(&self.per_domain_thresholds)?;

                    self.resharing_key.proposed_parameters().clone(),
                    self.previous_running_state.add_domains_votes.clone(),
                )));
            }
        }
        Ok(None)
    }

    pub fn vote_cancel_resharing(&mut self) -> Result<Option<RunningContractState>, Error> {
        let previous_running_participants = self.previous_running_state.parameters.participants();
        let authenticated_candidate = AuthenticatedAccountId::new(previous_running_participants)?;
        self.cancellation_requests.insert(authenticated_candidate);

        let cancellation_votes_count = self.cancellation_requests.len() as u64;
        let previous_running_threshold = self.previous_running_state.parameters.threshold();

        let threshold_cancellation_votes_reached: bool =
            cancellation_votes_count >= previous_running_threshold.value();

        let running_state = if threshold_cancellation_votes_reached {
            let mut previous_running_state = self.previous_running_state.clone();
            let prospective_epoch_id = self.prospective_epoch_id();
            previous_running_state.previously_cancelled_resharing_epoch_id =
                Some(prospective_epoch_id);

            Some(previous_running_state)
        } else {
            None
        };

        Ok(running_state)
    }
