Audit Report

## Title
Stale `rsETHPrice` in `getRsETHAmountToMint` allows depositors to capture accrued yield from existing holders — (File: `contracts/LRTDepositPool.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint()` divides a live Chainlink asset price by a cached `rsETHPrice` state variable that is only refreshed via an explicit external call to `updateRSETHPrice()`. Because LST prices (e.g. stETH/ETH) increase continuously with staking rewards, any depositor who deposits between two keeper updates receives more rsETH than their proportional share of the protocol's assets, permanently diluting the yield that had accrued to existing rsETH holders.

## Finding Description

`getRsETHAmountToMint` at [1](#0-0)  computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

- `lrtOracle.getAssetPrice(asset)` delegates to a live `IPriceFetcher` and returns the **current** asset/ETH rate. [2](#0-1) 

- `lrtOracle.rsETHPrice` is a **stored state variable** updated only inside `_updateRsETHPrice()`, which is invoked only when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is called externally. [3](#0-2) [4](#0-3) 

The deposit entry point calls `_beforeDeposit`, which is declared `private view` — it cannot mutate state and therefore **cannot** refresh `rsETHPrice` before computing the mint amount. [5](#0-4) 

`depositAsset` itself also never calls `updateRSETHPrice()` before invoking `_beforeDeposit`. [6](#0-5) 

`rsETHPrice` is set as `totalETHInProtocol / rsethSupply` at the moment of the last keeper call, where `totalETHInProtocol` itself uses live asset prices at that moment. [7](#0-6)  Between keeper calls, as LST prices rise, the numerator (`getAssetPrice`) grows while the denominator (`rsETHPrice`) stays fixed, causing the formula to mint excess rsETH.

The `pricePercentageLimit` guard only applies inside `_updateRsETHPrice()` and does not constrain the deposit mint calculation. [8](#0-7) 

## Impact Explanation

Every deposit made while `rsETHPrice` is stale and the underlying LST price has risen mints more rsETH than the depositor's proportional contribution warrants. When `updateRSETHPrice()` is eventually called, the new price is computed over an inflated rsETH supply, permanently reducing the per-token value for all pre-existing holders. This constitutes **theft of unclaimed yield** — the staking rewards that had accrued to existing holders are partially redistributed to the new depositor.

**Impact: High — Theft of unclaimed yield.**

## Likelihood Explanation

- stETH and other LSTs accrue staking rewards continuously, so their ETH price increases every block.
- `updateRSETHPrice()` is called by an off-chain keeper on a periodic schedule, not on every deposit, creating a persistent window.
- `depositAsset` is fully permissionless — any user can exploit this passively by depositing at any time after the LST price has risen since the last keeper update.
- No special privileges, flash loans, oracle manipulation, or front-running are required.

**Likelihood: High.**

## Recommendation

Remove the `view` modifier from `_beforeDeposit` (or introduce a separate non-view pre-deposit hook) and call `_updateRsETHPrice()` before invoking `getRsETHAmountToMint`. This ensures the denominator reflects the same live asset prices as the numerator. Alternatively, compute the mint amount entirely from live prices (e.g., `amount * assetPrice / _getLiveTotalETHPerRsETH()`) without relying on the cached `rsETHPrice` state variable.

## Proof of Concept

**Setup:**
- Protocol holds 100 stETH; `rsethSupply = 100`; `rsETHPrice = 1.050e18` (set when stETH/ETH = 1.050)
- Chainlink stETH/ETH feed now returns `1.060e18` (staking rewards accrued); `updateRSETHPrice()` has not been called

**Exploit (passive deposit):**
1. Alice calls `depositAsset(stETH, 10e18, 0, "")`
2. `getRsETHAmountToMint` computes: `10e18 × 1.060e18 / 1.050e18 = 10.0952e18` rsETH
3. Alice receives **10.0952 rsETH** instead of the fair **~10.0 rsETH**

**After deposit:**
- Protocol holds 110 stETH; `rsethSupply = 110.0952`
- When keeper calls `updateRSETHPrice()`: `newRsETHPrice = (110 × 1.060e18) / 110.0952e18 ≈ 1.05908e18`
- Existing holders' 100 rsETH is worth `100 × 1.05908 = 105.908 ETH`
- Fair value should be `106.0 ETH` — **existing holders permanently lost ~0.092 ETH of yield**

**Foundry fork test plan:**
1. Fork mainnet; deploy/configure `LRTDepositPool` and `LRTOracle` with stETH support.
2. Call `updateRSETHPrice()` to set baseline `rsETHPrice`.
3. Warp forward (or mock Chainlink to return a higher stETH/ETH rate).
4. Record `rsETHPrice` (unchanged) and call `getRsETHAmountToMint(stETH, 10e18)`.
5. Assert returned amount exceeds `10e18 × currentStETHPrice / currentStETHPrice` (i.e., exceeds fair share).
6. Call `updateRSETHPrice()` after the deposit and assert existing holders' rsETH value is lower than it would have been without the deposit.
