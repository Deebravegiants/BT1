### Title
Consensus Vote Signature Never Verified — Any Peer Can Impersonate a Committee Member and Forge Quorum - (File: crates/apollo_consensus/src/single_height_consensus.rs)

### Summary
`SingleHeightConsensus::handle_vote` accepts incoming `Vote` messages and counts their weight toward prevote/precommit quorum thresholds using only the self-reported `vote.voter` address field. The cryptographic signature carried in `vote.signature` is never verified. A single malicious network peer can therefore forge votes that appear to originate from any committee member, inflate weighted vote totals to manufacture a false quorum for an attacker-chosen `ProposalCommitment`, or preemptively suppress a legitimate validator's vote by sending a conflicting vote before the real one arrives.

### Finding Description

The `Vote` struct carries both a `voter: ContractAddress` and a `signature: RawSignature` field. [1](#0-0) 

`handle_vote` in `SingleHeightConsensus` performs two checks before accepting a vote: height equality and committee membership of `vote.voter`. It then passes the vote directly to the state machine, which inserts it into the `prevotes` or `precommits` map keyed by `(round, voter)` and accumulates the weight returned by `vote_weight(voter)`. [2](#0-1)

The same omission exists when the state machine constructs its own self-votes: [4](#0-3) 

The `manager.rs` future-vote cache even documents the consequence explicitly:

> "Since vote signatures are not yet verified, a peer can forge votes with arbitrary voter addresses" [5](#0-4) 

The deduplication guard in `received_vote` uses `(round, voter)` as the key. [6](#0-5) 

This means:

**Attack 1 — False quorum.** An attacker sends `N` `Vote` messages, each with `voter` set to a different high-weight committee member and `signature = RawSignature::default()`. Each passes the committee-membership check, is treated as `VoteStatus::New`, and is inserted into the `prevotes`/`precommits` map with the real weight of the impersonated validator. Once the accumulated weight crosses the Byzantine quorum threshold and the virtual-proposer check passes, `upon_decision` fires and `SMRequest::DecisionReached` is emitted for the attacker's chosen `ProposalCommitment`. [7](#0-6) 

**Attack 2 — Vote suppression.** An attacker sends a forged vote for validator `V` with a `proposal_commitment` that differs from the one `V` will actually vote for. When `V`'s real vote arrives later, `received_vote` returns `VoteStatus::Conflict`, and the real vote is silently dropped. [8](#0-7) 

The `SignatureManager` crate already provides `verify_precommit_vote_signature` and the infrastructure to look up a validator's public key from the staking committee, but this is never called on the incoming vote path. [9](#0-8) 

### Impact Explanation

A successful false-quorum attack causes `decision_reached` to be called with an attacker-controlled `ProposalCommitment`. This propagates to `BlockExecutionArtifacts`, the committer, and ultimately to the state root, transaction receipts, event commitments, and L1 messages written to storage — all derived from the wrong block. This matches the **Critical** impact category: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result*.

A successful vote-suppression attack can prevent a legitimate quorum from forming, causing liveness failure or forcing the round to advance to a round where the attacker controls the proposer.

### Likelihood Explanation

The trigger requires only the ability to send a well-formed protobuf `Vote` message over the P2P gossip channel. No validator key material is needed. The `voter` field is a plain `ContractAddress` that any peer can set to any value in the committee. The `signature` field defaults to all-zero bytes and is never checked. The committee membership list is public (derived from the staking contract). The attack is therefore reachable by any peer that can connect to the consensus gossip topic.

### Recommendation

In `SingleHeightConsensus::handle_vote`, after confirming committee membership, retrieve the validator's `public_key` from the committee's `StakerSet` and call `verify_precommit_vote_signature` (or an equivalent prevote verifier) against `vote.signature`. Reject the vote if verification fails or if the public key is unavailable. The `SignatureManager` library and the `Staker.public_key` field are already present in the codebase and only need to be wired into the vote-handling path. [10](#0-9)

    pub(crate) fn received_vote(&self, vote: &Vote) -> VoteStatus {
        let determine_status = |old: &Vote, new: &Vote| {
            if old.proposal_commitment == new.proposal_commitment {
                VoteStatus::Duplicate
            } else {
                VoteStatus::Conflict(old.clone(), new.clone())
            }
        };

        // Check Map
        let key = (vote.round, vote.voter);
        let map_entry = match vote.vote_type {
            VoteType::Prevote => self.prevotes.get(&key),
            VoteType::Precommit => self.precommits.get(&key),
        };

        if let Some((old_vote, _)) = map_entry {
            return determine_status(old_vote, vote);
        }

        // Check Queue
        for event in &self.events_queue {
            let queued_vote = match (event, vote.vote_type) {
                (StateMachineEvent::Prevote(v), VoteType::Prevote) => v,
                (StateMachineEvent::Precommit(v), VoteType::Precommit) => v,
                _ => continue,
            };

            if queued_vote.round == vote.round && queued_vote.voter == vote.voter {
                return determine_status(queued_vote, vote);
            }
        }

        VoteStatus::New
    }
