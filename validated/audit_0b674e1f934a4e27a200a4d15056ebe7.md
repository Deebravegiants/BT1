### Title
Deposits for Non-Executed Proposals Are Permanently Locked When `do_update` Clears All Entries - (File: crates/contract/src/update.rs)

### Summary
When a contract upgrade or config update reaches threshold and `do_update` is executed, it clears **all** pending proposals via `self.entries.clear()`, but never refunds the storage deposits paid by proposers of the non-winning proposals. The `bytes_used` field stored in each `UpdateEntry` was presumably intended for refund accounting but is never used for that purpose. Deposits are permanently locked in the contract.

### Finding Description
In `propose_update` (`lib.rs`), every proposer must attach a deposit calculated by `ProposedUpdates::required_deposit(&update)`, which covers the storage cost of the proposal (including the full contract WASM blob): [1](#0-0) 

The deposit is accepted and only the *excess* is refunded at proposal time. The required portion is retained by the contract.

In `do_update` (`update.rs`), when a winning proposal is executed, the function removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()` to wipe all remaining proposals: [2](#0-1) 

No refund is issued to the proposers of the cleared entries. The `bytes_used` field stored in `UpdateEntry` is never read back for refund purposes: [3](#0-2) 

The `required_deposit` function shows these deposits can be substantial — for a full contract WASM upload, `bytes_used` includes the entire code length plus 128 × `size_of::<AccountId>()` overhead: [4](#0-3) 

### Impact Explanation
Any participant who proposes an update that is not ultimately executed loses their storage deposit permanently. For a large contract binary (e.g., 1–2 MB of WASM), the deposit can be on the order of 10–20 NEAR per proposal. Multiple concurrent proposals from different participants multiply the loss. The contract's NEAR balance grows by the sum of all non-winning deposits with no mechanism to recover them, breaking the accounting invariant that "deposits paid for proposals are refundable when those proposals are superseded."

**Impact: Medium** — balance/accounting invariant broken without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation
This is triggered under normal governance operation: two or more participants independently propose different updates (e.g., one proposes a code upgrade, another proposes a config change), threshold votes accumulate for one, and `vote_update` executes `do_update`. The losing proposer's deposit is silently forfeited. No adversarial coordination is required — this is a routine governance scenario.

**Likelihood: Medium** — requires only two concurrent proposals, which is a realistic governance scenario.

### Recommendation
Before calling `self.entries.clear()`, iterate over all remaining entries and issue a `Promise::new(proposer_account).transfer(refund_amount)` for each, using the stored `bytes_used` to compute the refund. The proposer's account ID must be tracked alongside the entry (currently it is not stored in `UpdateEntry`). Alternatively, store a `(proposer: AccountId, deposit: NearToken)` field in `UpdateEntry` and refund on clear.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB WASM blob, paying ~10 NEAR deposit. `UpdateId(0)` is created.
2. Participant B calls `propose_update` with a config change, paying a smaller deposit. `UpdateId(1)` is created.
3. Threshold participants call `vote_update(UpdateId(0))`. Threshold is reached.
4. `do_update(&UpdateId(0), gas)` is called:
   - `self.entries.remove(&UpdateId(0))` — winning entry removed.
   - `self.entries.clear()` — `UpdateId(1)` (Participant B's proposal) is wiped with no refund.
   - `self.vote_by_participant.clear()` — votes cleared.
5. Participant B's deposit is permanently locked in the contract. `bytes_used` stored in the now-deleted `UpdateEntry` is gone with no record. [5](#0-4)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
