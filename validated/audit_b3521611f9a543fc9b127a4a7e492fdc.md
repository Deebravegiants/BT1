Audit Report

## Title
`updateRSETHPrice()` Permanently Blocked When `maxFeeMintAmountPerDay` Is Zero and Protocol Fees Accrue - (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.initialize()` never sets `maxFeeMintAmountPerDay`, leaving it at `0`. When `protocolFeeInBPS > 0` and TVL grows, `_updateRsETHPrice()` computes a non-zero `rsethAmountToMintAsProtocolFee` and passes it to `_checkAndUpdateDailyFeeMintLimit()`, which unconditionally reverts because `feeAmount > 0 == maxFeeMintAmountPerDay`. The line `rsETHPrice = newRsETHPrice` is never reached, freezing the exchange rate and breaking yield accounting for all depositors and withdrawers.

## Finding Description
`initialize()` at lines 64–68 sets only `lrtConfig` and emits an event; `maxFeeMintAmountPerDay` is never assigned and defaults to `0`. [1](#0-0) 

`reinitialize()` at lines 72–79 also does not set `maxFeeMintAmountPerDay`. [2](#0-1) 

The only setter is `setMaxFeeMintAmountPerDay()`, which requires the manager role and is never called automatically. [3](#0-2) 

When `protocolFeeInBPS > 0` and `totalETHInProtocol > previousTVL`, a non-zero `protocolFeeInETH` is computed: [4](#0-3) 

This produces a non-zero `rsethAmountToMintAsProtocolFee`, which is passed to `_checkAndUpdateDailyFeeMintLimit()`: [5](#0-4) 

Inside `_checkAndUpdateDailyFeeMintLimit()`, the guard `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay` reduces to `feeAmount > 0` when `maxFeeMintAmountPerDay == 0`, which is always `true`, causing an unconditional revert: [6](#0-5) 

Consequently, `rsETHPrice = newRsETHPrice` at line 313 is never executed, and the stored price is frozen at its last value. [7](#0-6) 

The public entry point `updateRSETHPrice()` is callable by any EOA, so no privilege is required to trigger the revert. [8](#0-7) 

The stale (lower-than-actual) `rsETHPrice` is consumed directly by `LRTDepositPool.getRsETHAmountToMint()`, causing new depositors to receive more rsETH than they are entitled to, diluting existing holders' accrued yield. [9](#0-8) 

## Impact Explanation
**Medium. Permanent freezing of unclaimed yield.** The yield that has accrued in the protocol (reflected in the grown TVL) cannot be properly distributed via the price update mechanism. The freeze persists indefinitely — there is no on-chain automatic resolution; it requires the manager to call `setMaxFeeMintAmountPerDay()`. Until that call is made, all yield accrued since the last successful price update is inaccessible to rsETH holders, and new depositors are over-minted rsETH at the expense of existing holders.

## Likelihood Explanation
`maxFeeMintAmountPerDay` is `0` by default after every fresh deployment or upgrade that calls `initialize()`. Any deployment where the manager has not yet called `setMaxFeeMintAmountPerDay()` — the default post-deployment state — is immediately vulnerable as soon as `protocolFeeInBPS > 0` and any rewards accrue. No attacker action is required; any public caller of `updateRSETHPrice()` (including automated bots or any EOA) triggers the revert. The same condition recurs if the manager ever resets the limit to `0`.

## Recommendation
Decouple the fee-minting step from the price-update step. When `maxFeeMintAmountPerDay == 0` or the daily limit is exhausted, skip the fee mint but still commit the new price:

```solidity
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    if (maxFeeMintAmountPerDay > 0 &&
        currentPeriodMintedFeeAmount + rsethAmountToMintAsProtocolFee <= maxFeeMintAmountPerDay) {
        _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
        address treasury = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee);
        emit FeeMinted(treasury, rsethAmountToMintAsProtocolFee);
    }
} else {
    _checkAndUpdateDailyFeeMintLimit(0);
}
rsETHPrice = newRsETHPrice; // always update
```

Alternatively, require `maxFeeMintAmountPerDay` to be set to a non-zero value in `initialize()` or `reinitialize()` before fee collection is enabled.

## Proof of Concept
1. Deploy `LRTOracle` and call `initialize(lrtConfigAddr)`. `maxFeeMintAmountPerDay` is `0` (never set).
2. Configure `LRTConfig` with `protocolFeeInBPS = 500` (5%).
3. Allow EigenLayer rewards to accrue so `totalETHInProtocol > previousTVL`.
4. Call `LRTOracle.updateRSETHPrice()` from any EOA.
5. Execution enters `_updateRsETHPrice()`, computes `protocolFeeInETH > 0`, computes `rsethAmountToMintAsProtocolFee > 0`, calls `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)`.
6. Check: `0 + rsethAmountToMintAsProtocolFee > 0` → `true` → reverts with `DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)`.
7. `rsETHPrice` is never updated. All subsequent deposits use the stale price, over-minting rsETH for new depositors and diluting existing holders' yield indefinitely.

**Foundry test sketch:**
```solidity
function test_updateRSETHPrice_blockedWhenMaxFeeIsZero() public {
    // maxFeeMintAmountPerDay is 0 by default after initialize()
    // set protocolFeeInBPS > 0 in lrtConfig mock

    vm.expectRevert(abi.encodeWithSelector(
        LRTOracle.DailyFeeMintLimitExceeded.selector, feeAmount, 0
    ));

}
```
