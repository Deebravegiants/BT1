### Title
Unregistered ERC-20 Token Accepted by `initTransfer` Causes Permanent Fund Lock — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` accepts **any** ERC-20 token address without verifying that the token is registered in the bridge. Tokens are transferred from the user into the contract, but the corresponding NEAR-side finalization (`fin_transfer_callback`) unconditionally requires the token to be present in `token_decimals`, causing a hard panic and leaving the EVM-side tokens permanently locked with no on-chain recovery path.

---

### Finding Description

`initTransfer` in `OmniBridge.sol` handles three token categories in order:

1. `customMinters[tokenAddress] != address(0)` → burn via custom minter
2. `isBridgeToken[tokenAddress]` → burn bridge token
3. **else** → `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount)` [1](#0-0) 

The `else` branch executes for **any** ERC-20 that is neither a bridge token nor a custom-minter token — including tokens that have never been registered in the bridge at all. There is no guard such as `require(bytes(ethToNearToken[tokenAddress]).length > 0, "ERR_TOKEN_NOT_REGISTERED")`. [2](#0-1) 

After the tokens are locked, an `InitTransfer` event is emitted and a relayer submits the proof to the NEAR bridge. Inside `fin_transfer_callback`, the very first thing the NEAR contract does is look up the token's decimal configuration:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [3](#0-2) 

`token_decimals` is only populated when a token is explicitly registered via `bind_token` / `add_token`. For an unregistered token this lookup returns `None`, and `near_expect` panics with `ERR_TOKEN_DECIMALS_NOT_FOUND`, permanently aborting finalization. [4](#0-3) 

The EVM contract has no `cancelTransfer`, `rescueTokens`, or equivalent function. The only admin functions are `addCustomToken`, `removeCustomToken`, `setMetadata`, `upgradeToken`, and `setNearBridgeDerivedAddress` — none of which can recover arbitrary locked ERC-20 balances. [5](#0-4) 

The identical structural flaw exists in the Starknet bridge: `init_transfer` locks any ERC-20 via `transfer_from` without a registration check. [6](#0-5) 

---

### Impact Explanation

**Critical — Irreversible fund lock.**

A user who calls `initTransfer` with any ERC-20 token that is not yet registered in the bridge (not in `ethToNearToken`, not `isBridgeToken`, not `customMinters`) will have their tokens transferred into the OmniBridge contract. Every subsequent attempt to finalize the transfer on NEAR will panic. Because there is no on-chain refund or cancel path on the EVM side, the tokens are permanently unclaimable without an admin contract upgrade.

---

### Likelihood Explanation

**Medium.** The `initTransfer` function is a public, permissionless entry point. A user who attempts to bridge a token that has been announced but not yet fully registered, or who makes a simple address mistake, will trigger this path. The contract provides no feedback at deposit time that the token is unsupported.

---

### Recommendation

Add a registration guard at the top of `initTransfer` (and `initTransfer1155`) before any token transfer occurs:

```solidity
require(
    tokenAddress == address(0) ||
    isBridgeToken[tokenAddress] ||
    customMinters[tokenAddress] != address(0) ||
    bytes(ethToNearToken[tokenAddress]).length > 0,
    "ERR_TOKEN_NOT_REGISTERED"
);
```

Apply the same guard to the Starknet `init_transfer`. On the NEAR side, add a symmetric check in `ft_on_transfer` / `init_transfer` that the incoming token has a registered destination-chain address before accepting the `ft_transfer_call`.

---

### Proof of Concept

1. Deploy or identify any ERC-20 token `T` that is **not** in `ethToNearToken`, `isBridgeToken`, or `customMinters`.
2. Call `OmniBridge.initTransfer(T, 1000, 0, 0, "victim.near", "")` with sufficient ETH for `nativeFee = 0`.
3. The `else` branch executes: `T.safeTransferFrom(msg.sender, address(OmniBridge), 1000)`. Tokens are now held by the contract. `InitTransfer` event is emitted.
4. A relayer submits the proof to NEAR `fin_transfer`.
5. `fin_transfer_callback` calls `self.token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)` — panics because `T` was never registered.
6. The NEAR transaction reverts. The EVM tokens remain locked in `OmniBridge` with no recovery path. [7](#0-6) [8](#0-7)

        address tokenAddress,

        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,

    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );

                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }

            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,

            message,
