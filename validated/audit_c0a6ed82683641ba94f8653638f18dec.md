### Title
Hardcoded `Gas::ZERO` for `GlobalContractDistribution` Receipts in Congestion Gas Accounting Causes Underestimated Shard Load - (File: `runtime/runtime/src/congestion_control.rs`)

### Summary

`compute_receipt_congestion_gas` unconditionally returns `Gas::ZERO` for `GlobalContractDistribution` receipts, regardless of the actual compute cost of processing them (which scales linearly with contract code size). When these receipts accumulate in the delayed-receipts queue, the shard's `CongestionInfo.delayed_receipts_gas` is underestimated, causing other shards to forward more gas than the congested shard can handle.

### Finding Description

In `runtime/runtime/src/congestion_control.rs`, the function `compute_receipt_congestion_gas` is responsible for computing how much gas a receipt contributes to the congestion accounting when it is pushed into the delayed-receipts queue or an outgoing buffer. For `GlobalContractDistribution` receipts, it unconditionally returns `Gas::ZERO`: [1](#0-0) 

However, when a `GlobalContractDistribution` receipt is actually executed in `apply_distribution_current_shard`, it charges a non-trivial compute cost that scales with the contract code size: [2](#0-1) 

The test `test_deploy_global_contract_compute_cost_splits_chunks` explicitly confirms that in the production protocol config, this compute cost is non-zero and large enough to saturate a chunk's compute budget on its own: [3](#0-2) 

The congestion control system uses `delayed_receipts_gas` to compute `incoming_congestion`, which in turn drives `outgoing_gas_limit` — the amount of gas other shards are allowed to forward to this shard per chunk: [4](#0-3) 

When `GlobalContractDistribution` receipts are in the delayed queue with `Gas::ZERO` accounting, `incoming_congestion` is underestimated, and `outgoing_gas_limit` remains higher than it should be, allowing more receipts to be forwarded to the shard than it can process.

The `DelayedReceiptQueueWrapper::push` calls `compute_receipt_congestion_gas` to determine the gas contribution before storing the receipt: [5](#0-4) 

### Impact Explanation

The corrupted protocol value is `CongestionInfo.delayed_receipts_gas` (a `u128` field in the chunk header's `CongestionInfoV1`). When `GlobalContractDistribution` receipts are delayed, this field is underreported. Downstream, `incoming_congestion()` returns a lower fraction than the true workload, and `outgoing_gas_limit()` permits other shards to forward more gas than the shard can execute within its compute budget. The result is that the delayed queue grows faster than the congestion control system expects, degrading throughput on the affected shard. The `receipt_bytes` dimension is correctly tracked (providing partial protection), but the gas dimension — which is the primary signal for `incoming_congestion` — is wrong.

**Impact: Medium**

### Likelihood Explanation

Any unprivileged user can submit a `DeployGlobalContractAction` transaction. The resulting `GlobalContractDistributionReceipt` carries the full contract code and is forwarded to all shards. If the target shard is under load (e.g., during a traffic spike), the receipt is pushed to the delayed queue with `Gas::ZERO` gas contribution. A user deploying a large contract (near the maximum allowed size) to a busy shard can trigger this condition. The global contracts feature must be active (protocol version that enables it).

**Likelihood: Medium**

### Recommendation

In `compute_receipt_congestion_gas`, for `VersionedReceiptEnum::GlobalContractDistribution`, compute the actual gas cost using the contract code length and the configured fee parameters (`deploy_global_contract_execution_base + deploy_global_contract_execution_per_byte * code.len()`), mirroring the logic in `apply_distribution_current_shard`. This ensures the congestion accounting reflects the true compute cost of processing these receipts.

### Proof of Concept

1. Protocol version with global contracts and non-zero `deploy_global_contract_execution_base`/`deploy_global_contract_execution_per_byte` is active.
2. A shard is under moderate load (delayed queue is non-empty).
3. An unprivileged user submits `DeployGlobalContractAction` with a large contract (e.g., 1 MB).
4. The resulting `GlobalContractDistributionReceipt` arrives at the shard as an incoming receipt.
5. The shard's gas limit is hit; the receipt is pushed to the delayed queue via `DelayedReceiptQueueWrapper::push`.
6. `compute_receipt_congestion_gas` returns `Gas::ZERO` for this receipt.
7. `CongestionInfo.delayed_receipts_gas` is not incremented by the actual compute cost.
8. Other shards observe low `incoming_congestion` on this shard and continue forwarding at the full `max_outgoing_gas` rate.
9. The shard processes the `GlobalContractDistribution` receipt, burning `deploy_global_contract_execution_base + per_byte * code_len` compute — saturating the chunk — while the congestion signal remains artificially low. [6](#0-5) [7](#0-6)
