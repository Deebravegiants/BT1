### Title
Unrecoverable Proposal Deposits When `do_update` Clears All Pending Entries Without Refund - (File: `crates/contract/src/update.rs`)

### Summary
When a contract update reaches threshold and `do_update` executes, it unconditionally clears **all** pending update proposals and their associated votes. However, the NEAR deposits paid by proposers of the non-executed proposals are never refunded. Those funds are permanently locked in the contract account with no recovery path, breaking the balance/accounting invariant for every participant who proposed a competing update.

### Finding Description

`propose_update` requires a deposit proportional to the size of the proposed contract binary, calculated via `ProposedUpdates::required_deposit`: [1](#0-0) 

The exact required amount is charged and any excess is refunded to the proposer at submission time. The deposit is held in the contract's account balance as implicit storage compensation.

When `vote_update` collects enough votes, it calls `do_update`: [2](#0-1) 

`do_update` removes the winning entry, then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every other pending proposal from storage. No refund promise is issued for the deposits that funded those cleared proposals. The NEAR tokens remain in the contract account permanently, with no endpoint to recover them.

This is confirmed by the test that explicitly verifies all state is cleared on update execution: [3](#0-2) 

The test asserts entries and votes are empty after `do_update`, but never checks that deposits were returned.

### Impact Explanation

Every participant who called `propose_update` for a proposal that was not the one ultimately executed permanently loses their deposit. For a typical WASM contract binary (~500 KB), the NEAR storage cost is approximately 5 NEAR per proposal. With multiple competing proposals (a realistic scenario during governance disputes or upgrade races), several participants each lose 5+ NEAR with no recovery mechanism. The funds are not burned — they remain in the contract account — but there is no withdrawal, refund, or sweep function that can return them to the original proposers. This breaks the production accounting invariant that deposited funds are either consumed for their stated purpose or returned.

**Allowed impact match:** Medium — balance/accounting invariant broken without relying on network-level DoS or operator misconfiguration.

### Likelihood Explanation

This triggers on every successful contract upgrade where more than one proposal was active. The `propose_update` flow is a normal operational path used in every upgrade cycle. The sandbox tests confirm multiple simultaneous proposals are a supported and tested scenario: [4](#0-3) 

Any participant can propose an update; when a competing proposal wins, the proposer's deposit is silently forfeited. No special attacker capability is required beyond being a participant.

### Recommendation

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue a `Promise::new(original_proposer).transfer(deposit_amount)` for each. This requires storing the proposer's account ID alongside each `UpdateEntry` at proposal time. Alternatively, implement a separate `cancel_update(id)` function that allows a proposer to withdraw their own proposal and receive a refund before any update executes, mirroring the `require(balance == 0)` guard pattern from the referenced report.

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, paying ~5 NEAR deposit. Entry `id=0` is stored.
2. Participant B calls `propose_update` with a different 500 KB binary, paying ~5 NEAR deposit. Entry `id=1` is stored.
3. Threshold participants vote for `id=1` via `vote_update`.
4. `do_update(&id=1, gas)` executes:
   - Removes entry `id=1` (the winner).
   - Calls `self.entries.clear()` — entry `id=0` (Participant A's proposal) is deleted.
   - Calls `self.vote_by_participant.clear()`.
   - No refund is issued to Participant A.
5. Participant A's ~5 NEAR deposit is permanently locked in the contract account. There is no function in the contract ABI to recover it. [5](#0-4)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
