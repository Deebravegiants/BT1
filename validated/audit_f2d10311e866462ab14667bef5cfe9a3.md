Audit Report

## Title
Missing Chainlink Staleness Check Enables Block-Stuffing-Assisted Over-Withdrawal via `instantWithdrawal` — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice` silently discards the `updatedAt` return value from `latestRoundData()` and applies no heartbeat/staleness guard. An attacker who stuffs blocks to delay Chainlink keeper update transactions can hold a stale-low asset price on-chain, then call `instantWithdrawal` to receive more LST than the burned rsETH is worth at current market rates, draining excess LST from `LRTUnstakingVault`.

## Finding Description

**Root cause — no staleness check:**

`ChainlinkPriceOracle.getAssetPrice` (L49–55) calls `latestRoundData()` but only captures `answer`; `updatedAt` is silently discarded:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();   // updatedAt never checked
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

**Price propagation path:**

`LRTOracle.getAssetPrice` (L156–158) delegates directly to the registered `IPriceFetcher`, passing the stale price through unchanged. `LRTWithdrawalManager.getExpectedAssetAmount` (L590–593) then computes:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

If `getAssetPrice(asset)` returns a stale-low value `P_stale < P_real`, the division yields an inflated LST amount.

**Exploit path through `instantWithdrawal`:**

`instantWithdrawal` (L212–253) calls `getExpectedAssetAmount`, burns rsETH, and redeems from the vault. The only guard is a vault-liquidity cap (`CantInstantWithdrawMoreThanAvailable`); there is no price-sanity or staleness check before the burn-and-redeem executes.

Critically, `unlockQueue` (L268–295) does enforce caller-supplied price bounds via `_validatePrices`, but `instantWithdrawal` has no equivalent protection — this asymmetry is the direct enabler of the attack.

**Attack sequence:**

1. Attacker monitors a Chainlink LST/ETH feed for an impending price increase (e.g., stETH after a rebase).
2. Attacker stuffs blocks with high-fee transactions to prevent Chainlink keeper update transactions from landing, keeping the on-chain answer at the pre-rebase stale-low value.
3. Attacker calls `instantWithdrawal(asset, rsETHUnstaked, ...)` while the stale price persists.
4. `getExpectedAssetAmount` returns an inflated LST amount; rsETH is burned and excess LST is redeemed from `LRTUnstakingVault`.
5. Attacker sells the excess LST at the real market price, profiting the spread.

## Impact Explanation

The invariant `assetAmountUnlocked * realAssetPrice ≤ rsETHUnstaked * rsETHPrice` is violated. `LRTUnstakingVault` loses LST in excess of the rsETH value burned — a direct, quantifiable loss of protocol assets. This constitutes **Low — Block stuffing**, which is an explicitly listed allowed impact.

## Likelihood Explanation

Block stuffing on Ethereum mainnet requires the attacker to outbid normal gas prices across consecutive blocks, making it capital-intensive. Profitability requires a sufficiently wide price gap (e.g., a large stETH rebase) and adequate vault liquidity to cover the inflated redemption. This limits realistic exploitation to high-volatility events. The `instantWithdrawalFee` provides a small buffer but does not eliminate the attack surface. Likelihood is **Low**, but the code offers zero on-chain resistance once the price is stale.

## Recommendation

Add a per-asset staleness check in `ChainlinkPriceOracle.getAssetPrice`:

```solidity
(, int256 price,,uint256 updatedAt,) = priceFeed.latestRoundData();
require(block.timestamp - updatedAt <= stalenessThreshold[asset], "Stale price");
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`stalenessThreshold` should be set per-asset to slightly exceed the feed's documented heartbeat (e.g., 3 600 s for a 1-hour heartbeat feed). Additionally, add caller-supplied price-bounds parameters to `instantWithdrawal` analogous to the `minimumAssetPrice`/`maximumAssetPrice` guards already present in `unlockQueue` (L268–295).

## Proof of Concept

Fork test pinned to a block where the Chainlink LST/ETH feed has not yet updated after a rebase (simulating block stuffing):

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";

interface IChainlinkFeed {
    function latestRoundData() external view returns (uint80, int256, uint256, uint256, uint80);
}

contract StalePricePoC is Test {
    address constant WITHDRAWAL_MANAGER = 0x...;
    address constant RSETH             = 0x...;
    address constant STETH             = 0x...;
    address constant STETH_FEED        = 0x...;

    function testStalePrice_OverWithdrawal() external {
        // 1. Fork at a block where stETH feed is stale-low
        vm.createSelectFork("mainnet", STALE_BLOCK);

        // 2. Confirm feed is stale (updatedAt is old)
        (,, , uint256 updatedAt,) = IChainlinkFeed(STETH_FEED).latestRoundData();
        assertGt(block.timestamp - updatedAt, HEARTBEAT, "feed not stale");

        // 3. Attacker holds rsETH
        uint256 rsETHAmount = 1 ether;
        deal(RSETH, address(this), rsETHAmount);
        IERC20(RSETH).approve(WITHDRAWAL_MANAGER, rsETHAmount);

        uint256 stethBefore = IERC20(STETH).balanceOf(address(this));

        // 4. Instant withdrawal at stale-low price
        ILRTWithdrawalManager(WITHDRAWAL_MANAGER)
            .instantWithdrawal(STETH, rsETHAmount, "poc");

        uint256 stethReceived = IERC20(STETH).balanceOf(address(this)) - stethBefore;

        // 5. Assert received > fair value at real price
        uint256 fairAmount = rsETHAmount * getRsETHPrice() / getRealStETHPrice();
        assertGt(stethReceived, fairAmount, "attacker received more stETH than rsETH is worth");
    }
}
```

The test directly demonstrates the invariant violation: `stethReceived * realPrice > rsETHAmount * rsETHPrice`, without any admin compromise or privileged access.
