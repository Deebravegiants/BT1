Audit Report

## Title
Uninitialized `tokenFeeBps` in `addSupportedToken` Allows Fee-Free Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary
`addSupportedToken` registers a new ERC20 token but never initializes `tokenFeeBps[token]`, leaving it at the Solidity default of 0. Any unprivileged user who calls `deposit(token, amount, referralId)` before an admin separately calls `setTokenFeeBps` receives rsETH calculated on the full deposit amount with zero fee deducted, while ETH depositors always pay the configured `feeBps`. The protocol collects no fee revenue on all token deposits made during this window.

## Finding Description
`addSupportedToken` (L637–655) sets `supportedTokenOracle[token]` and `tokenBridge[token]` but never writes to `tokenFeeBps[token]`:

```solidity
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] is never set → remains 0
```

The token deposit path in `viewSwapRsETHAmountAndFee(uint256, address)` (L335–336) reads directly from this mapping:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];   // 0 until setTokenFeeBps is called
fee = amount * feeBpsForToken / 10_000;         // always 0
```

`deposit(address token, ...)` (L284–305) is a public, permissionless function callable by any user. It calls `viewSwapRsETHAmountAndFee(amount, token)` and transfers `rsETHAmount` (computed on the full `amount` with zero fee) to the caller. The ETH path (L311–312) uses the global `feeBps` which is always non-zero after initialization, creating a permanent asymmetry.

The only remediation path is a separate admin call to `setTokenFeeBps` (L583–593), which is not enforced or atomically coupled to `addSupportedToken`. There is no guard, default value, or require statement in `addSupportedToken` that prevents the zero-fee state.

## Impact Explanation
**High – Theft of unclaimed yield.**

The fee on deposits is the protocol's yield. When `tokenFeeBps[token] == 0`, `fee = 0` and `amountAfterFee = amount`, so the user receives rsETH minted against the full deposit. The protocol's share (the fee) is never deducted and never accrues to `feeEarnedInToken[token]`. Every token deposit during the window transfers what should be protocol fee revenue directly to the depositor as extra rsETH. Because `feeEarnedInToken` is never credited, the loss is permanent and unrecoverable.

## Likelihood Explanation
The two-step pattern (add token, then separately set fee) is not enforced by the contract. Any operator who adds a new token — whether by oversight or simply following the existing function signature — leaves the pool in the zero-fee state. The pool is live on Arbitrum with multiple supported tokens, and the pattern will recur with every future token listing. No special attacker capability is required: any user monitoring the mempool or chain for `AddSupportedToken` events can immediately deposit to capture fee-free rsETH.

## Recommendation
Add a `_feeBps` parameter to `addSupportedToken` and initialize `tokenFeeBps[token]` atomically:

```solidity
function addSupportedToken(
    address token,
    address oracle,
    address bridge,
    uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    tokenFeeBps[token] = _feeBps;
    ...
}
```

This eliminates the window during which token deposits are fee-free and mirrors how `feeBps` is set at initialization for ETH deposits.

## Proof of Concept
1. Admin calls `addSupportedToken(wstETH, oracle, bridge)`. `tokenFeeBps[wstETH]` is 0 (Solidity default). [1](#0-0) 
2. Before admin calls `setTokenFeeBps(wstETH, 30)`, a user calls `deposit(wstETH, 100e18, "")`. [2](#0-1) 
3. `viewSwapRsETHAmountAndFee(100e18, wstETH)` executes: `feeBpsForToken = 0`, `fee = 0`, `amountAfterFee = 100e18`. User receives rsETH on the full 100e18. [3](#0-2) 
4. An ETH depositor with the same value pays `feeBps` (e.g. 30 bps) and receives less rsETH. [4](#0-3) 
5. `feeEarnedInToken[wstETH]` remains 0; the protocol collects no fee revenue from the token deposit. [5](#0-4) 

**Foundry fork test plan:** Deploy a fork of the live Arbitrum pool. Call `addSupportedToken` with a new token. Immediately call `deposit(token, 1e18, "")` from an unprivileged address. Assert `feeEarnedInToken[token] == 0` and that the rsETH received equals the amount computed with `feeBps = 0`. Compare against a second deposit made after `setTokenFeeBps(token, 30)` to confirm the fee-free advantage.
