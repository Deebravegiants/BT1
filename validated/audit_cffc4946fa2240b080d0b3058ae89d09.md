Audit Report

## Title
ETH Deposit Limit Bypass via Missing Amount in Boundary Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` contains an asymmetric ETH branch that omits the incoming `amount` from its comparison and uses strict `>` instead of `>=`. As a result, the ETH deposit cap is entirely unenforced: any depositor can push the protocol's ETH holdings arbitrarily beyond the configured limit in a single transaction, as long as the current total has not yet crossed the limit at the moment of the call.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676â€“682), the LST path correctly evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. The ETH path, however, evaluates only `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)`, dropping `amount` entirely. Two compounding defects are present:

1. **Missing `amount`**: The function receives `amount` (equal to `msg.value` forwarded from `depositETH`) but never uses it in the ETH branch. The check only asks "has the limit already been exceeded?" â€” not "would this deposit exceed the limit?"
2. **Strict `>` instead of `>=`**: Even if `totalAssetDeposits` exactly equals `depositLimitByAsset`, the condition evaluates to `false`, permitting the deposit.

The caller `_beforeDeposit` (L648â€“670) reverts only when `_checkIfDepositAmountExceedesCurrentLimit` returns `true`. Since the ETH branch never accounts for the incoming deposit, it will return `false` for any deposit made while `totalAssetDeposits â‰¤ depositLimitByAsset`, regardless of how large `amount` is. `depositETH` (L87) passes `msg.value` as `depositAmount` into `_beforeDeposit`, which forwards it as `amount` to the limit check â€” where it is silently ignored.

## Impact Explanation
**Low â€” Contract fails to deliver promised returns, but doesn't lose value.**

The deposit cap is a protocol-level risk parameter set by the admin to bound exposure to ETH. With the ETH branch broken, any depositor can mint rsETH against an arbitrarily large ETH deposit in a single call, as long as `totalAssetDeposits â‰¤ depositLimitByAsset` at the moment of the call. The protocol accumulates ETH far beyond the intended ceiling, undermining the risk management guarantee the limit is meant to provide. No direct theft occurs, but the contract fails to enforce its own stated constraint, matching the allowed impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
High. The entry path is the public, permissionless `depositETH` function. No special role, flash loan, or oracle manipulation is required. Any depositor who observes that the current ETH total has not yet crossed the limit can immediately exploit this by sending a single large ETH deposit. The condition is trivially observable on-chain and repeatable across any block where the limit has not yet been crossed.

## Recommendation
Add `amount` to the ETH branch and use `>=` to match the semantics of the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount >= lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept

2. Current `totalAssetDeposits(ETH) = 50 ether`.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit(ETH, 10_000 ether)`:
   - `totalAssetDeposits = 50 ether`
   - ETH branch: `return (50 ether > 100 ether)` â†’ `false`
   - `amount = 10_000 ether` is never evaluated.
5. `_beforeDeposit` does **not** revert; rsETH is minted for 10,000 ETH.
6. After the call, `totalAssetDeposits(ETH) = 10,050 ether` â€” 100Ă— the intended cap.
