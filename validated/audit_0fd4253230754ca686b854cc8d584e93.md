### Title
Nested Unbounded Loop in `LRTOracle._getTotalEthInProtocol` Allows Permanent DoS on `updateRSETHPrice` - (File: contracts/LRTOracle.sol)

### Summary
`LRTOracle._getTotalEthInProtocol()` contains a nested loop — outer over all supported assets, inner over all node delegators — with multiple external calls per inner iteration. There is no cap on the number of supported assets. As the protocol legitimately grows, the gas cost of the public `updateRSETHPrice()` scales as O(assets × NDCs × external_calls) and can permanently exceed the block gas limit, freezing the rsETH price oracle and disabling the protocol's downside price-protection auto-pause mechanism.

### Finding Description

`LRTOracle._getTotalEthInProtocol()` is called by the public, permissionless `updateRSETHPrice()`. It iterates over every entry in `supportedAssetList` and, for each asset, calls `ILRTDepositPool.getTotalAssetDeposits(asset)`, which itself iterates over every entry in `nodeDelegatorQueue` with three external calls per NDC.

### Impact Explanation

If `updateRSETHPrice()` permanently reverts due to gas exhaustion:

1. **Price protection is permanently disabled.** `_updateRsETHPrice()` contains the only mechanism that auto-pauses `LRTDepositPool` and `LRTWithdrawalManager` when the rsETH price drops beyond `pricePercentageLimit`. If the function can never execute, a slashing event or oracle manipulation will not trigger the protective pause. [6](#0-5) 

2. **Protocol fee minting is permanently broken.** The fee-minting logic inside `_updateRsETHPrice()` can never run, permanently denying the treasury its yield. [7](#0-6) 

3. **rsETH price becomes permanently stale.** Deposits and withdrawals continue to use the last stored `rsETHPrice`, which diverges from true NAV over time, enabling value extraction by users who observe the divergence.

Impact classification: **Medium — Unbounded gas consumption** leading to **Medium — Permanent freezing of unclaimed yield** and **Medium — Temporary freezing of funds** (withdrawal amounts calculated from stale price).

### Likelihood Explanation

The protocol already supports stETH, ETHx, and native ETH, and is architecturally designed to onboard additional LSTs. Each new asset added via governance multiplies the gas cost of `updateRSETHPrice()`. With 10 supported assets and 10 NDCs, the function already performs approximately 310 external calls. At 20 assets and 15 NDCs, this reaches ~920 external calls — well within range of exceeding the 30M gas block limit on Ethereum mainnet given the SLOAD and external call costs involved. No attacker action is required; normal protocol growth triggers the condition.

### Recommendation

1. **Cap `supportedAssetList.length`** with an explicit maximum (e.g., 15–20 assets) enforced in `_addNewSupportedAsset`, analogous to the existing `maxNodeDelegatorLimit` guard on NDCs.
2. **Make the cap configurable** by admin so it can be adjusted as EVM gas limits evolve.
3. **Decouple price-update gas cost from asset/NDC count** by caching per-asset TVL snapshots that are updated incrementally (e.g., on deposit/withdrawal) rather than recomputing the full sum on every oracle update.
4. **Add a gas-cost estimate test** for `updateRSETHPrice()` at the maximum expected asset and NDC counts to validate the cap is safe.

### Proof of Concept

With `N` supported assets and `M` NDCs, each call to `updateRSETHPrice()` performs:
- `N` calls to `getAssetPrice` (external, ~2100 gas each for cold SLOAD + call overhead)
- `N × M` calls to `IERC20.balanceOf` (~2600 gas each)
- `N × M` calls to `INodeDelegator.getAssetBalance` (~5000+ gas each, involves EigenLayer strategy reads)
- `N × M` calls to `INodeDelegator.getAssetUnstaking` (~5000+ gas each)

At `N=15`, `M=10`: 15 + 150 + 150 + 150 = 465 external calls. At ~5000 gas average per call, that is ~2.3M gas for calls alone, before accounting for memory allocation, storage reads in `_updateRsETHPrice`, and the fee-minting logic. At `N=20`, `M=15` this exceeds 10M gas for external calls alone, and with full execution context approaches or exceeds the 30M block gas limit. [8](#0-7) [9](#0-8)

    /// @param asset Asset address

    function _addNewSupportedAsset(address asset, uint256 depositLimit) private {
        UtilLib.checkNonZeroAddress(asset);
        if (depositLimit == 0) {
            revert InvalidDepositLimit();
        }
        if (isSupportedAsset[asset]) {
            revert AssetAlreadySupported();
        }
        isSupportedAsset[asset] = true;
        supportedAssetList.push(asset);
        depositLimitByAsset[asset] = depositLimit;
