### Title
`respond_ckd` Skips Cryptographic Output Verification for `AppPublicKey` Variant, Allowing a Single Byzantine Participant to Forge CKD Responses - (File: `crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, the contract performs a BLS pairing-based output check (`ckd_output_check`) only when the request uses the `AppPublicKeyPV` variant. When the request uses the legacy `AppPublicKey` variant, the check arm is empty — no cryptographic verification of the response is performed. A single Byzantine MPC participant acting as the signing leader can submit an arbitrary `CKDResponse` (`big_y`, `big_c`) for any pending `AppPublicKey`-variant CKD request, and the contract will accept and deliver it to the waiting user.

---

### Finding Description

In `respond_ckd` at `crates/contract/src/lib.rs:675-682`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

The `AppPublicKeyPV` arm verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` via `ckd_output_check` (defined in `crates/contract/src/primitives/ckd.rs:80-102`). The `AppPublicKey` arm is entirely empty — the response `(big_y, big_c)` is accepted unconditionally and forwarded to the user via `resolve_yields_for`.

This is the direct analog of the reported `Funds.maxFundDur` bug: when one parameter variant is active (`AppPublicKey`), the safety check that would otherwise enforce correctness (`ckd_output_check`) is silently skipped.

The `AppPublicKey` variant is the default/legacy "privately verifiable" mode described in `crates/contract/README.md:119-120` and is the variant used in the non-PV e2e test (`crates/e2e-tests/tests/ckd_verification.rs:63-65`). It is the more commonly used path.

---

### Impact Explanation

**Critical — Unauthorized confidential key derivation output without required participant authorization.**

The CKD protocol is designed so that the user's derived secret is `big_c − a·big_y`, where `a` is the user's private scalar. If an attacker submits `big_y = G1_identity` (the identity point) and `big_c = attacker_chosen_point`, the user's derived key becomes `attacker_chosen_point − a·0 = attacker_chosen_point`. The attacker fully controls the user's derived key without knowing `a`.

No threshold of participants is required to mount this attack — only the single participant who acts as the signing leader for the targeted request needs to be Byzantine. The contract's `respond_ckd` function performs no check on the submitted `(big_y, big_c)` values for the `AppPublicKey` variant, so the forged response passes all contract-level validation and is delivered to the user.

---

### Likelihood Explanation

**Medium.** The attacker must be an attested MPC participant (satisfying `assert_caller_is_attested_participant_and_protocol_active`). However, only **one** such participant needs to be malicious — specifically, the one assigned as leader for the targeted request (leader assignment is deterministic per request ID, per the coordinator design). Any single Byzantine participant who is ever assigned as leader for an `AppPublicKey`-variant CKD request can exploit this. The `AppPublicKey` variant is the default mode and is used in production flows.

---

### Recommendation

Apply `ckd_output_check` (or an equivalent validity check) to the `AppPublicKey` arm as well. For the `AppPublicKey` variant, the contract holds only `pk1 = a·G1` (no G2 key), so the full pairing equation cannot be verified on-chain. At minimum, the contract should verify that `big_y` and `big_c` are valid, non-identity G1 points in the prime-order subgroup (analogous to the subgroup check already enforced in `app_public_key_check` at `crates/contract/src/primitives/ckd.rs:62-74`). Ideally, the `AppPublicKey` variant should be deprecated in favor of `AppPublicKeyPV`, which admits full on-chain verification.

---

### Proof of Concept

1. User submits `request_app_private_key` with `app_public_key = AppPublicKey(pk1)` where `pk1 = a·G1`.
2. The request is queued in `pending_ckd_requests`.
3. A Byzantine MPC participant (the assigned leader) calls `respond_ckd(request, CKDResponse { big_y: G1_identity, big_c: attacker_point })`.
4. The contract executes the match at line 675: the `AppPublicKey(_) => {}` arm fires — no check is performed.
5. `resolve_yields_for` delivers `CKDResponse { big_y: G1_identity, big_c: attacker_point }` to the user.
6. The user computes their derived key as `big_c − a·big_y = attacker_point − a·0 = attacker_point`.
7. The attacker, who chose `attacker_point`, knows the user's derived key. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
