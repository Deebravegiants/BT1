### Title
Wrong Shard Scanned in `GetReceiptToTx` Hint Mode for Cross-Shard Transactions — (File: chain/client/src/view_client_actor.rs)

---

### Summary

When `GetReceiptToTx` is called with a caller-supplied `shard_id` hint equal to the action receipt's **execution** shard (the receiver's shard), `handle_receipt_to_tx` seeds the scan with `ScanShards::Hint(receiver_shard)`. `resolve_scan_shards` then calls `hint_lineage`, which only walks **ancestor** shards of the hint — never sibling shards. In a cross-shard transaction the producing transaction outcome lives on the **sender's** shard, which is a sibling, not an ancestor. The scan therefore reads `OutcomeIds` rows from the wrong shard, finds nothing, and returns `GetReceiptToTxError::UnknownReceipt` instead of the correct transaction hash. This is the direct nearcore analog of M-25: a context-specific identifier (the receiver's shard) is used to look up data that belongs to a different context (the sender's shard), producing an incorrect result.

---

### Finding Description

`GetReceiptToTx` is a public JSON-RPC endpoint (wired through `chain/jsonrpc/src/api/receipts.rs`) that resolves a receipt back to its originating transaction. When the `ReceiptToTx` column is absent or disabled (`save_receipt_to_tx = false`), the handler falls back to a hint-driven scan of `DBCol::OutcomeIds`.

**Root cause — wrong shard seeded from caller hint:**

In `handle_receipt_to_tx` (`chain/client/src/view_client_actor.rs`, line 1347):

```rust
let mut scan_shards = msg.shard_id.map(ScanShards::Hint).unwrap_or(ScanShards::Enumerate);
```

When the caller passes `shard_id = Some(receiver_shard)`, the seed becomes `ScanShards::Hint(receiver_shard)`.

`resolve_scan_shards` (lines 1501–1507) resolves this seed via `hint_lineage`:

```rust
ScanShards::Hint(shard_id) => {
    for id in hint_lineage(layout, *shard_id) ... {
        push(id, &mut shards);
    }
}
```

`hint_lineage` (lines 1518–1535) walks only **upward** through the shard's split-parent chain:

```rust
fn hint_lineage(layout: &ShardLayout, hint: ShardId) -> Result<Vec<ShardId>, ShardLayoutError> {
    let mut lineage = vec![hint];
    let mut id = hint;
    loop {
        match layout.try_get_parent_shard_id(id) { ... }
    }
    Ok(lineage)
}
```

In a cross-shard transfer, the transaction executes on the **sender's** shard and emits the action receipt. The action receipt then executes on the **receiver's** shard. The sender's shard and receiver's shard are siblings — neither is an ancestor of the other. `hint_lineage` never reaches the sender's shard, so `resolve_receipt_via_hint` (`chain/chain/src/receipt_to_tx.rs`, line 143) queries `OutcomeIds` only for the receiver's shard:

```rust
let outcome_ids =
    chain_store.get_outcomes_by_block_hash_and_shard_id(&block_hash, shard_id);
```

The producing transaction outcome is absent from those rows, the window is exhausted, and the handler returns `UnknownReceipt`.

The codebase itself documents this as a "best-effort failure mode" in the regression test at `test-loop-tests/src/tests/receipt_to_tx/hint.rs` lines 434–444:

> "Action receipt executes on *receiver's* shard; tx executes on *sender's* shard. Hint at action receipt's execution shard can't find tx outcome on originating shard — center-out scan walks wrong shard's `OutcomeIds` rows."

---

### Impact Explanation

Any caller that supplies `shard_id = <receiver_shard>` (the natural hint derived from where the action receipt executed) receives `UnknownReceipt` for a receipt that genuinely exists and has a traceable origin. The corrupted value is the **RPC result** of `GetReceiptToTx`: it returns an error instead of the correct `(transaction_hash, sender_account_id)` pair. Downstream consumers — block explorers, indexers, wallets, and audit tools — that rely on this endpoint to trace cross-shard receipts back to their originating transactions will silently receive wrong data.

---

### Likelihood Explanation

Cross-shard transactions are the common case on a multi-shard NEAR network (any transfer between accounts on different shards). Any unprivileged user can:
1. Submit a cross-shard `send_money` or function-call transaction.
2. Observe the action receipt's execution block and shard from the transaction outcome.
3. Call `GetReceiptToTx` with `block_height = <action_height>` and `shard_id = <receiver_shard>`.

This is the most natural way to construct the hint (use the coordinates where the receipt was observed to execute), making the wrong-shard path the default failure mode for any caller that follows the obvious hint-construction strategy.

---

### Recommendation

When the hint scan resolves a `FromReceipt` hop and the receipt's `predecessor_id` is available, derive the producing shard from `account_id_to_shard_id(predecessor_id)` rather than carrying the receiver's shard forward. For the first hop (direct `FromTransaction` case), the producing shard is `account_id_to_shard_id(receipt.predecessor_id)` — the sender's shard — which should be added to the scan set alongside the hint's lineage. The test comment at line 448 already identifies this fix:

> "Future shard-aware hint derivation (e.g. via `parent.predecessor_id() -> account_id_to_shard_id`) → update test to reflect new contract."

Concretely, `resolve_scan_shards` should, for `ScanShards::Hint`, also resolve `account_id_to_shard_id` of the queried receipt's `predecessor_id` (readable from `DBCol::Receipts`) and union that shard into the scan set.

---

### Proof of Concept

The existing regression test at `test-loop-tests/src/tests/receipt_to_tx/hint.rs:451–525` (`test_hint_fallback_cross_shard_returns_unknown_receipt`) is a direct proof of concept:

1. Two-shard setup; sender on shard 0, receiver on shard 1.
2. `save_receipt_to_tx = false` forces the hint-scan path.
3. A cross-shard `send_money` is submitted; the action receipt executes on shard 1.
4. `GetReceiptToTx` is called with `block_height = action_height`, `shard_id = shard_1` (the receiver's shard).

fn hint_lineage(layout: &ShardLayout, hint: ShardId) -> Result<Vec<ShardId>, ShardLayoutError> {
    let mut lineage = vec![hint];
    let mut id = hint;
    loop {
        match layout.try_get_parent_shard_id(id) {
            Ok(Some(parent)) => {
                if parent == id {
                    break;
                }
                lineage.push(parent);
                id = parent;
            }
            Ok(None) => break,
            Err(ShardLayoutError::InvalidShardId { .. }) => break,
            Err(e) => return Err(e),
        }
    }
    Ok(lineage)

fn test_hint_fallback_cross_shard_returns_unknown_receipt() {
    init_test_logger();
    let sender_account = create_account_id("account0");
    let receiver_account: AccountId = "test1".parse().unwrap();

    let mut env = TestLoopBuilder::new()
        .num_shards(2)
        .add_user_account(&sender_account, Balance::from_near(1_000_000))
        .add_user_account(&receiver_account, Balance::from_near(1_000_000))
        .epoch_length(EPOCH_LENGTH)
        .track_all_shards()
        .config_modifier(|config, _| {
            config.save_receipt_to_tx = false;
        })
        .build();

    let signer = create_user_test_signer(&sender_account);
    let tx = SignedTransaction::send_money(
        1,
        sender_account,
        receiver_account,
        &signer,
        Balance::from_yoctonear(100),
        env.validator().head().last_block_hash,
    );
    let outcome = env.validator_runner().execute_tx(tx, Duration::seconds(10)).unwrap();

    let action_receipt_id = outcome.transaction_outcome.outcome.receipt_ids[0];
    let action_outcome = outcome
        .receipts_outcome
        .iter()
        .find(|r| r.id == action_receipt_id)
        .expect("action receipt outcome exists");
    let action_height = env
        .validator()
        .client()
        .chain
        .get_block_header(&action_outcome.block_hash)
        .unwrap()
        .height();

    // Identify which shard action receipt executed on. Don't hardcode
    // layout — multi_shard(2) puts boundary at "test1" but future layout
    // change still leaves test exercising cross-shard scenario.
    let action_shard = shard_containing_outcome(
        &env,
        action_height,
        action_receipt_id,
        &[ShardId::new(0), ShardId::new(1)],

    // outcome that produced this receipt lives on the *other* shard,
    // so hint scan walks wrong shard's outcomes + misses.

        receipt_id: action_receipt_id,
        block_height: Some(action_height),
        shard_id: Some(action_shard),
        window: None,
    });

    match result {
        Err(GetReceiptToTxError::UnknownReceipt(id)) => {
            assert_eq!(
                id, action_receipt_id,
                "expected UnknownReceipt for the action receipt; its producing tx is on the other shard"
            );
        }
        other => panic!(
            "expected UnknownReceipt at the cross-shard hop; got {other:?}. \
             If a future change adds shard-aware hint advancement, update this test."
        ),
    }
