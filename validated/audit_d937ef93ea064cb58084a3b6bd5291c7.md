Audit Report

## Title
Unvalidated Chainlink Price in `ChainlinkPriceOracle.getAssetPrice()` Enables rsETH Over-Minting via Stale Oracle Data - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validation fields (`roundId`, `updatedAt`, `answeredInRound`), accepting any returned price including stale, zero, or negative values. Because `LRTOracle.updateRSETHPrice()` is a public, permissionless function, an attacker can lock in a corrupted `rsETHPrice` during a period of stale Chainlink data and immediately deposit a different supported asset at the artificially deflated price, minting excess rsETH and diluting existing holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` performs no validation on the Chainlink response:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All three standard Chainlink guards are absent: staleness (`updatedAt` vs `block.timestamp`), round completeness (`answeredInRound >= roundId`), and non-zero/non-negative price. The sister contract `ChainlinkOracleForRSETHPoolCollateral.getRate()` enforces all three explicitly (L30-32), confirming the pattern is known to the codebase.

The unvalidated price flows into `LRTOracle._getTotalEthInProtocol()` (L339), which sums `totalAssetAmt.mulWad(assetER)` for every supported asset. A stale or zero price for any single LST silently underestimates `totalETHInProtocol`. `_updateRsETHPrice()` then computes and stores:

```solidity
// contracts/LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

The downside-protection circuit breaker at L270-281 only triggers when `pricePercentageLimit > 0` **and** the price drop exceeds that limit; if `pricePercentageLimit == 0` (unset) or the stale price causes a drop within the configured band, `rsETHPrice` is updated to the corrupted lower value and the deposit pool remains open. An attacker then calls `depositAsset` for a *different* (non-stale) asset:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Here `getAssetPrice(asset)` returns the correct live price for the deposited asset, while `rsETHPrice` is the artificially depressed stored value, so the division yields more rsETH than the deposit is worth.

## Impact Explanation
When `rsETHPrice` is set below its true value, every subsequent depositor receives more rsETH per unit of deposited asset than they are entitled to. When the oracle recovers and `rsETHPrice` is corrected upward, the attacker's rsETH is worth more than what was deposited — the excess value is extracted from the yield accrued by existing rsETH holders. This constitutes **theft of unclaimed yield** (High severity per the allowed impact scope).

## Likelihood Explanation
Chainlink feeds go stale during network congestion, oracle node downtime, or circuit-breaker events — these are documented, recurring occurrences. `updateRSETHPrice()` requires no role or privilege; any EOA can call it. The attacker needs only to monitor Chainlink `updatedAt` off-chain, wait for a feed to lag, call `updateRSETHPrice()`, and immediately deposit via a non-stale asset. No flash loan, governance action, or privileged key is required. The attack is repeatable across any future staleness event.

## Recommendation
Mirror the validation already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId)          revert StalePrice();
if (updatedAt == 0)                     revert IncompleteRound();
if (price <= 0)                         revert InvalidPrice();
if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
```

Additionally, consider adding a per-asset zero-price guard inside `_getTotalEthInProtocol()` so that a single bad oracle cannot silently corrupt the global TVL sum even if the oracle contract itself is replaced.

## Proof of Concept
1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale — `updatedAt` is hours old and `answeredInRound < roundId`.
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control). `pricePercentageLimit` is either 0 or the stale price is within the configured band, so the downside circuit breaker does not fire and `rsETHPrice` is written with the underestimated value.
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale (or near-zero) price without reverting, underestimating `totalETHInProtocol`.
4. `newRsETHPrice` is stored below its true value.
5. Attacker calls `LRTDepositPool.depositAsset(wstETH, largeAmount)`. The wstETH/ETH feed is live and returns the correct price; `rsETHPrice` is the corrupted low value. The division mints excess rsETH to the attacker.
6. When the stETH feed recovers and `rsETHPrice` is corrected upward, the attacker's rsETH is redeemable for more ETH than deposited — existing holders are diluted.

**Foundry fork test outline:**
- Fork mainnet; deploy protocol with `pricePercentageLimit = 0`.
- Mock the stETH/ETH Chainlink feed to return a price with `updatedAt = block.timestamp - 2 days`.
- Call `updateRSETHPrice()`; assert `rsETHPrice` decreased.
- Deposit wstETH as attacker; record rsETH minted.
- Restore the stETH feed to the correct price; call `updateRSETHPrice()` again.
- Assert attacker's rsETH is now worth more ETH than deposited, at the expense of pre-existing holders.
