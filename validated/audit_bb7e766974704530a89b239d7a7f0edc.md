### Title
`withdrawalDelay` Uninitialized in `initialize()` Allows Immediate Staked-Token Withdrawal Without Any Lockup - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.initialize()` never sets `withdrawalDelay`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal()` computes `unlockTime = block.timestamp + withdrawalDelay`, a zero delay means `unlockTime == block.timestamp`, and `claimWithdrawal()` passes its `block.timestamp < withdrawal.unlockTime` guard immediately. Any staker can therefore stake and reclaim KERNEL tokens in the same block, completely bypassing the intended lockup period of up to 30 days.

### Finding Description
`KernelDepositPool` is a Synthetix-style staking contract for KERNEL tokens. It enforces a withdrawal delay through a two-step flow: `initiateWithdrawal()` records a `Withdrawal` struct with an `unlockTime`, and `claimWithdrawal()` enforces that `block.timestamp >= unlockTime` before releasing tokens.

The `initialize()` function sets only `kernelToken`, `rewardsToken`, and the admin role: [1](#0-0) 

Neither `withdrawalDelay` nor `maxNumberOfWithdrawalsPerUser` is assigned, so both remain `0`. The only setter for `withdrawalDelay` is the admin-only `setWithdrawalDelay()`, which explicitly rejects `0` as a value: [2](#0-1) 

This means the contract can never be in a state where `withdrawalDelay` is `0` *after* an admin call — but it starts at `0` and remains there until the admin explicitly calls `setWithdrawalDelay`. During that window (or permanently if the admin omits the call), `initiateWithdrawal()` computes:

```
uint256 unlockTime = block.timestamp + withdrawalDelay; // = block.timestamp + 0
``` [3](#0-2) 

And `claimWithdrawal()` checks:

```
if (block.timestamp < withdrawal.unlockTime) revert WithdrawalNotReady();
// block.timestamp < block.timestamp → false → no revert
``` [4](#0-3) 

The withdrawal is claimable in the very same block it was initiated.

Note: `maxNumberOfWithdrawalsPerUser` also defaults to `0`, which causes `initiateWithdrawal` to revert with `WithdrawalLimitReached` until the admin calls `setMaxNumberOfWithdrawalsPerUser`. However, once the admin sets that value (a required operational step), the `withdrawalDelay = 0` vulnerability becomes immediately exploitable if `setWithdrawalDelay` was not also called.

### Impact Explanation
The staking lockup mechanism — the core security property of `KernelDepositPool` — is not enforced. Any user who calls `stake()` can call `initiateWithdrawal()` and `claimWithdrawal()` in the same block, recovering their KERNEL tokens with zero waiting time. The contract fails to deliver its promised lockup guarantee. No user funds are directly stolen, but the protocol's staking incentive design (locking tokens for up to 30 days to earn rewards) is rendered meaningless.

**Impact: Low** — Contract fails to deliver promised returns (lockup), but does not lose value.

### Likelihood Explanation
The likelihood is **medium**. The `initialize()` function is called exactly once at deployment. If the deployer does not immediately follow up with `setWithdrawalDelay()`, the contract operates with zero delay indefinitely. This is a realistic deployment oversight, especially since `maxNumberOfWithdrawalsPerUser` must also be set separately, creating two independent post-init configuration steps that could be missed or reordered.

### Recommendation
Set a non-zero default for `withdrawalDelay` directly inside `initialize()`, eliminating the dependency on a separate admin call:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... existing checks ...
    withdrawalDelay = 7 days; // enforce a safe default lockup
    maxNumberOfWithdrawalsPerUser = 10; // enforce a safe default limit
    // ...
}
```

Alternatively, add a `require(withdrawalDelay > 0)` guard inside `initiateWithdrawal()` so that withdrawals are blocked until the admin explicitly configures the delay.

### Proof of Concept
1. Admin deploys and calls `initialize(admin, kernelToken, rewardToken)`. `withdrawalDelay = 0`, `maxNumberOfWithdrawalsPerUser = 0`.
2. Admin calls `setMaxNumberOfWithdrawalsPerUser(100)` (required operational step). Forgets to call `setWithdrawalDelay`.
3. Alice calls `stake(1000e18)`. Her `balanceOf[Alice] = 1000e18`.
4. Alice calls `initiateWithdrawal(1000e18)`. `unlockTime = block.timestamp + 0 = block.timestamp`. Withdrawal record created.
5. Alice calls `claimWithdrawal(1)` in the same block. Check: `block.timestamp < block.timestamp` → `false` → passes. Alice receives `1000e18` KERNEL tokens immediately.
6. Alice has bypassed the entire lockup period with zero waiting time. [5](#0-4) [6](#0-5)

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }

    function claimWithdrawal(uint256 _withdrawalId) external nonReentrant {
        Withdrawal storage withdrawal = withdrawals[_withdrawalId];

        if (withdrawal.user == address(0)) {
            revert WithdrawalDoesNotExist();
        }

        if (withdrawal.user != msg.sender) {
            revert NotYourWithdrawal();
        }

        if (block.timestamp < withdrawal.unlockTime) {
            revert WithdrawalNotReady();
        }

        if (withdrawal.claimed) {
            revert WithdrawalAlreadyClaimed();
        }

        withdrawal.claimed = true;

        // Remove the withdrawal ID from the user's list of withdrawal IDs
        uint256[] storage userWithdrawalIdsArray = userWithdrawalIds[msg.sender];
        for (uint256 i = 0; i < userWithdrawalIdsArray.length; ++i) {
            if (userWithdrawalIdsArray[i] == _withdrawalId) {
                userWithdrawalIdsArray[i] = userWithdrawalIdsArray[userWithdrawalIdsArray.length - 1];
                userWithdrawalIdsArray.pop();
                break;
            }
        }

        // Transfer the KERNEL tokens from contract to user
        kernelToken.safeTransfer(msg.sender, withdrawal.amount);

        emit WithdrawalClaimed(msg.sender, withdrawal.amount, _withdrawalId);
    }
