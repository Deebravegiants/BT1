Audit Report

## Title
stETH 1-Wei Transfer Rounding Causes Last User's `completeWithdrawal` to Revert, Temporarily Freezing Funds - (File: contracts/LRTWithdrawalManager.sol)

## Summary
In `unlockQueue`, rsETH is burned and `unstakingVault.redeem(asset, assetAmountUnlocked)` transfers exactly `assetAmountUnlocked` stETH to the withdrawal manager. Because stETH's `transfer` rounds down by 1 wei due to its shares-based accounting, the withdrawal manager receives `assetAmountUnlocked - 1` stETH. When the last user in the batch calls `completeWithdrawal`, the `safeTransfer` of their full `expectedAssetAmount` reverts. The entire `completeWithdrawal` transaction rolls back (preserving the user's request), but the rsETH burn from the prior `unlockQueue` transaction is not reversed. The user is temporarily unable to complete their withdrawal until stETH rebasing covers the 1-wei deficit.

## Finding Description
**Step 1 ‚ÄĒ `unlockQueue`** (operator-triggered, separate transaction):

`_unlockWithdrawalRequests` accumulates `assetAmountToUnlock` as the exact sum of all payout amounts, then:
- Line 305: `IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned)` ‚ÄĒ irreversible in this transaction.
- Line 307: `unstakingVault.redeem(asset, assetAmountUnlocked)` ‚ÄĒ calls `IERC20(asset).safeTransfer(msg.sender, amount)` in `LRTUnstakingVault` (line 103).

stETH's `transfer(to, amount)` internally converts `amount` to shares via integer division (`amount * totalShares / totalPooledEth`, rounds down), then transfers those shares. The recipient receives `amount - 1` stETH in the common case where `amount * totalShares % totalPooledEth ‚Č† 0`. The withdrawal manager therefore holds `assetAmountUnlocked - 1` stETH after the redeem.

**Step 2 ‚ÄĒ `completeWithdrawal`** (user-triggered, separate transaction):

`_processWithdrawalCompletion` (line 734) calls `_transferAsset(asset, user, request.expectedAssetAmount)` with no balance check for non-ETH assets. For N users with payouts P1‚Ä¶PN summing to T, the contract holds T‚ąí1 stETH. Users 1 through N‚ąí1 succeed. User N's `safeTransfer(user, PN)` reverts because the balance is `PN - 1`. Since the entire `completeWithdrawal` transaction reverts, all state mutations in that call ‚ÄĒ `popFront()` (line 705), `delete withdrawalRequests[requestId]` (line 712), and `unlockedWithdrawalsCount[asset]--` (line 717) ‚ÄĒ are also rolled back. The user's withdrawal request is preserved and `unlockedWithdrawalsCount[asset]` remains > 0, blocking `sweepRemainingAssets` (line 403).

The user can retry `completeWithdrawal` after the next stETH rebase increases the contract's stETH balance above `PN`. stETH rebases approximately daily, so the freeze resolves within ~24 hours without any admin intervention.

## Impact Explanation
**Medium ‚ÄĒ Temporary freezing of funds.** The last user in any multi-user `unlockQueue` batch for stETH cannot complete their withdrawal until the next stETH rebase covers the 1-wei shortfall. Their rsETH is already burned (expected behavior of the withdrawal flow), but their withdrawal request remains intact and retryable. The freeze is bounded by the stETH rebase cadence (~24 hours), making this a temporary rather than permanent freeze.

## Likelihood Explanation
stETH's 1-wei rounding on `transfer` is deterministic and occurs whenever `amount * totalShares % totalPooledEth ‚Č† 0`. Given the large, non-round values of `totalShares` and `totalPooledEth` that change every block, this condition holds for the vast majority of transfer amounts. Any `unlockQueue` call processing two or more stETH withdrawal requests will expose the last user to this revert. No attacker capability is required; this triggers through normal user withdrawal flows.

## Recommendation
1. **Overshoot the redemption**: In `unlockQueue`, call `unstakingVault.redeem(asset, assetAmountUnlocked + 2)` to ensure the withdrawal manager always holds at least the full sum of pending payouts.
2. **Cap transfer at actual balance**: In `_processWithdrawalCompletion`, for rebasing tokens, use `min(request.expectedAssetAmount, IERC20(asset).balanceOf(address(this)))` as the transfer amount, absorbing the 1-wei deficit gracefully.
3. **Track actual received amount**: After `unstakingVault.redeem`, measure the actual balance increase and distribute proportionally rather than using the pre-computed `assetAmountUnlocked`.

## Proof of Concept
1. Alice and Bob each call `initiateWithdrawal(stETH, 1e18, "")`.
2. Operator calls `unlockQueue(stETH, ...)`. Alice's payout = P1, Bob's payout = P2; `assetAmountUnlocked = P1 + P2`.
3. `burnFrom(withdrawalManager, rsETHBurned)` ‚ÄĒ both users' rsETH burned (irreversible).
4. `unstakingVault.redeem(stETH, P1+P2)` ‚ÄĒ stETH rounding: withdrawal manager receives `P1+P2-1`.
5. Alice calls `completeWithdrawal` ‚ÄĒ receives P1. Manager holds `P2-1`.
6. Bob calls `completeWithdrawal` ‚ÄĒ `safeTransfer(Bob, P2)` reverts (balance is `P2-1`). Full transaction rolls back; Bob's request is preserved.
7. Bob retries after next stETH rebase (~24h later) ‚ÄĒ succeeds.

Foundry fork test: fork mainnet, deploy with real stETH, execute steps 1‚Äď6, assert Bob's `completeWithdrawal` reverts with the stETH `safeTransfer` failure, then `vm.roll` forward past the next rebase and assert step 7 succeeds.
