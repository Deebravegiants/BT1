### Title
EigenLayer `RewardsCoordinator` Authorized Claimer Can Bypass `NodeDelegator.processClaim()` and Redirect EigenLayer Rewards - (File: contracts/NodeDelegator.sol)

### Summary

`NodeDelegator` earns EigenLayer rewards and exposes `processClaim()` restricted to `onlyLRTOperator`, hardcoding the recipient as `lrtConfig.eigenLayerRewardReceiver()`. However, EigenLayer's `IRewardsCoordinator` provides a `setClaimerFor(address earner, address claimer)` function callable by operators or AVSs, which can set an authorized claimer for any earner — including `NodeDelegator`. A malicious EigenLayer operator (the one `NodeDelegator` delegates to) can set a rogue claimer for `NodeDelegator`, who can then call `IRewardsCoordinator.processClaim()` directly with an arbitrary recipient, completely bypassing `NodeDelegator.processClaim()` and its access controls.

---

### Finding Description

`NodeDelegator.processClaim()` is the protocol's intended entry point for claiming EigenLayer rewards: [1](#0-0) 

It enforces `onlyLRTOperator` and hardcodes the reward recipient to `lrtConfig.eigenLayerRewardReceiver()`. However, EigenLayer's `IRewardsCoordinator` exposes two overloads of `setClaimerFor`:

1. `setClaimerFor(address claimer)` — callable by the earner itself (msg.sender is the earner).
2. `setClaimerFor(address earner, address claimer)` — callable by **operators or AVSs**. [2](#0-1) 

The `processClaim` function on the `RewardsCoordinator` enforces: *"if `claimerFor[claim.earner]` is `address(0)` then only the earner can claim, otherwise only `claimerFor[claim.earner]` can claim the rewards."* [3](#0-2) 

`NodeDelegator` never calls `setClaimerFor` to lock in a trusted claimer for itself. This means the EigenLayer operator that `NodeDelegator` delegates to can call `setClaimerFor(address(nodeDelegator), maliciousAddress)`, making `maliciousAddress` the sole authorized claimer. That address can then call `IRewardsCoordinator.processClaim(claim, attackerAddress)` directly — with any recipient — bypassing `NodeDelegator.processClaim()` entirely.

The `eigenLayerRewardReceiver` is stored in `LRTConfig` and is the intended destination for all EigenLayer rewards: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

All EigenLayer rewards accrued by `NodeDelegator` (the earner) can be redirected to an attacker-controlled address. The `eigenLayerRewardReceiver` — the intended recipient — receives nothing. This is a direct theft of protocol yield that belongs to rsETH holders.

---

### Likelihood Explanation

`NodeDelegator` must delegate to an EigenLayer operator to participate in restaking. That operator, by EigenLayer's design, can call `setClaimerFor(address(nodeDelegator), maliciousAddress)`. This is a permissioned but externally accessible function — no private key compromise or governance capture is required. Any operator that `NodeDelegator` delegates to can execute this unilaterally. The protocol has no on-chain defense against it because `NodeDelegator` never pre-emptively calls `setClaimerFor` to lock in a trusted claimer.

---

### Recommendation

`NodeDelegator` should call `IRewardsCoordinator(lrtConfig.rewardsCoordinator()).setClaimerFor(trustedClaimerAddress)` during initialization (or via an admin function) to pre-emptively register a trusted claimer for itself. This prevents any external operator or AVS from overriding the claimer mapping. The trusted claimer should be a Kelp DAO-controlled address or the `NodeDelegator` itself. An admin-callable `setClaimerFor` wrapper should also be added to `NodeDelegator` to allow updating the claimer if needed.

---

### Proof of Concept

1. `NodeDelegator` is deployed and delegates to EigenLayer operator `OP`.
2. `NodeDelegator` never calls `IRewardsCoordinator.setClaimerFor(...)`, so `claimerFor[NodeDelegator] == address(0)`.
3. Operator `OP` calls `IRewardsCoordinator.setClaimerFor(address(nodeDelegator), attackerAddress)`.
4. Now `claimerFor[NodeDelegator] == attackerAddress`.
5. Attacker calls `IRewardsCoordinator.processClaim(validMerkleClaim, attackerAddress)` directly — this is valid because `attackerAddress` is the registered claimer.
6. EigenLayer rewards are transferred to `attackerAddress` instead of `lrtConfig.eigenLayerRewardReceiver()`.
7. `NodeDelegator.processClaim()` (restricted to `onlyLRTOperator`) is completely bypassed. [1](#0-0) [6](#0-5)
