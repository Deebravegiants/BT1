### Title
Missing `ppMaxCollateralInputs` Enforcement in Alonzo UTXO Transition Allows Unlimited Collateral Inputs — (`File: eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs`)

---

### Summary

The `validateTooManyCollateralInputs` function is defined in the Alonzo UTXO module and is correctly called in every later era (Babbage, Conway, Dijkstra), but is **never invoked** inside the Alonzo `utxoTransition`. A placeholder comment marks exactly where the call should appear, confirming the omission is unintentional. An unprivileged transaction author can therefore submit an Alonzo-era transaction with an arbitrarily large collateral-input set, bypassing the `ppMaxCollateralInputs` protocol parameter entirely.

---

### Finding Description

**Root cause — missing call in `utxoTransition`**

`eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxo.hs` defines the enforcement function:

```haskell
-- | Ensure that number of collaterals does not exceed the allowed @maxCollInputs@ parameter.
-- > ‖collateral tx‖  ≤  maxCollInputs pp
validateTooManyCollateralInputs :: ...
validateTooManyCollateralInputs pp txBody =
  failureUnless (numColl <= maxColl) $
    TooManyCollateralInputs Mismatch {mismatchSupplied = numColl, mismatchExpected = maxColl}
``` [1](#0-0) 

Inside `utxoTransition`, after enforcing `maxTxExUnits`, a comment marks the intended check location, but the actual `runTest` call is absent:

```haskell
  {-   totExunits tx ≤ maxTxExUnits pp    -}
  runTest $ validateExUnitsTooBigUTxO pp tx

  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  -- ← NO CALL HERE

  updatedGovState <-
    trans @(EraRule "UTXOS" era) $ ...
``` [2](#0-1) 

**Contrast with every later era**

Babbage explicitly calls the check:

```haskell
  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  runTest $ Alonzo.validateTooManyCollateralInputs pp txBody
``` [3](#0-2) 

Dijkstra also calls it: [4](#0-3) 

Conway inherits Babbage's `babbageUtxoValidation` which includes the same call. The Alonzo era is the sole era where the check is absent.

**Structural parallel to the external report**

| External report | Cardano Ledger analog |
|---|---|
| `purchase` has `notSoldOut` modifier | Babbage/Conway/Dijkstra call `validateTooManyCollateralInputs` |
| `shareKey` / `grantKey` skip the check | Alonzo `utxoTransition` skips the check |
| Root: `_assignNewTokenId` never checks the cap | Root: `utxoTransition` never calls the validator despite the comment |

---

### Impact Explanation

`ppMaxCollateralInputs` bounds the number of collateral inputs — and therefore the number of additional VKey signatures — that must be verified when a phase-2 script fails. The formal spec (Alonzo chain.tex / protocol-parameters.tex) states:

> *"The parameter `maxCollateralInputs` is used to limit, additionally, the total number of collateral inputs, and thus the total number of additional signatures that must be checked during validation."* [5](#0-4) 

Without the check, an attacker can craft an Alonzo-era transaction with an unbounded collateral-input set. Every honest node processing that block must verify all the extra signatures. This exceeds the intended per-transaction validation budget, matching the **Medium** allowed impact: *"Attacker-controlled transactions … exceed intended validation limits."*

---

### Likelihood Explanation

The Alonzo era is no longer the active era on Cardano mainnet (current era: Conway). Exploitation requires a network operating under Alonzo-era rules — e.g., a testnet, a private network, or any replay/conformance harness that exercises the Alonzo UTXO rule directly. The Alonzo STS instance is still compiled into the production binary and is exercised by conformance and era-translation tests. Likelihood is **low** for mainnet but **real** for any Alonzo-era environment.

---

### Recommendation

Add the missing enforcement call immediately after the comment in `utxoTransition`:

```haskell
  {-   ‖collateral tx‖  ≤  maxCollInputs pp   -}
  runTest $ validateTooManyCollateralInputs pp txBody
```

This mirrors the pattern already present in Babbage and Dijkstra and matches the intent expressed by the existing comment.

---

### Proof of Concept

1. Construct an Alonzo-era transaction with `collateralInputsTxBodyL` containing `N > ppMaxCollateralInputs` distinct UTxO inputs, each owned by a distinct VKey.
2. Submit the transaction to a node running in the Alonzo era.
3. Observe that the transaction is accepted: `validateTooManyCollateralInputs` is never called, so `TooManyCollateralInputs` is never raised.
4. The block producer includes the transaction; all validating nodes must verify all `N` collateral signatures, exceeding the protocol-parameter-bounded work budget.
5. Repeat with Babbage-era rules to confirm the same transaction is correctly rejected with `TooManyCollateralInputs`. [6](#0-5) [7](#0-6)

-- | Ensure that number of collaterals does not exceed the allowed @maxCollInputs@ parameter.
--
-- > ‖collateral tx‖  ≤  maxCollInputs pp
validateTooManyCollateralInputs ::
  AlonzoEraTxBody era =>
  PParams era ->
  TxBody TopTx era ->
  Test (AlonzoUtxoPredFailure era)
validateTooManyCollateralInputs pp txBody =
  failureUnless (numColl <= maxColl) $
    TooManyCollateralInputs Mismatch {mismatchSupplied = numColl, mismatchExpected = maxColl}
  where
    maxColl, numColl :: Word16
    maxColl = pp ^. ppMaxCollateralInputsL
    numColl = fromIntegral . Set.size $ txBody ^. collateralInputsTxBodyL
