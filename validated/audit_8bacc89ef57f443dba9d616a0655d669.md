### Title
Dust-Amount Deposit Permanently Locks User Tokens With No Recovery Path — (`near/omni-bridge/src/lib.rs`)

### Summary
`normalize_amount` performs integer floor division. When a user deposits a token amount smaller than `10^(origin_decimals − decimals)` via `ft_transfer_call`, the tokens are immediately burned or locked inside `init_transfer_internal`, but every subsequent `sign_transfer` call for that transfer will always panic with `InvalidAmountToTransfer` because the normalized amount is 0. No cancel or refund path exists, so the user's funds are permanently unclaimable.

### Finding Description

`normalize_amount` divides the raw NEAR-side amount by the decimal-scaling factor:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

For a token registered with `origin_decimals = 24` (NEAR) and `decimals = 6` (EVM), `diff_decimals = 18`, so any raw amount below `10^18` normalises to 0 via integer truncation.

`sign_transfer` calls `normalize_amount` on `amount_without_fee()` and then hard-panics if the result is 0:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

The panic happens **after** the tokens have already been consumed. `init_transfer_internal` burns deployed tokens and locks native tokens before returning `U128(0)` to the NEP-141 callback, confirming the full transfer:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [3](#0-2) 

The transfer message is stored in `pending_transfers` and is never removed when `sign_transfer` panics — the panic unwinds only the `sign_transfer` call, not the earlier `ft_transfer_call` receipt chain. There is no public `cancel_transfer` or user-callable refund function. The relayer will receive the same panic on every retry, making the lock permanent.

`init_transfer` itself has no minimum-amount guard; the only fee check is `fee < amount`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [4](#0-3) 

### Impact Explanation
Any user who deposits a sub-threshold amount of a token whose NEAR decimals exceed its destination-chain decimals loses those tokens permanently. The bridge holds them (burned or locked) with no settlement path and no refund mechanism. This matches **Critical — Irreversible fund lock / permanently unclaimable user value in bridge flows**.

### Likelihood Explanation
NEAR tokens commonly carry 24 decimals while their EVM mirrors carry 6 or 18. The threshold below which the loss occurs is `10^(24−6) = 10^18` raw units (≈ 1 full token for a 24-decimal asset). A user bridging a fractional amount — e.g., 0.5 of a 24-decimal token — triggers the bug. No special privilege is required; the entry point is the standard public `ft_transfer_call` → `ft_on_transfer` path.

### Recommendation
Add a minimum-amount guard inside `init_transfer` (before tokens are consumed) that rejects any deposit whose `normalize_amount(amount_without_fee(), decimals)` would be 0. Because `decimals` is known at registration time, the check can be performed eagerly:

```rust
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee()?,
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer);
```

This mirrors the existing guard in `sign_transfer` but places it at the deposit boundary so tokens are never consumed for an unsettleable transfer.

### Proof of Concept

1. Admin registers a token with `origin_decimals = 24`, `decimals = 6` (e.g., a NEAR-native token bridging to Ethereum).
2. User calls `ft_transfer_call` on the token contract with `amount = 5 × 10^17` (0.5 tokens in raw units, below the `10^18` threshold), `receiver_id = bridge`, `msg = InitTransfer{recipient: eth_address, fee: 0, ...}`.
3. `ft_on_transfer` → `init_transfer` → `init_transfer_internal`:
   - `burn_tokens_if_needed` burns the 5×10^17 tokens (deployed token case).
   - Transfer message stored in `pending_transfers`.
   - Returns `U128(0)` → NEP-141 confirms full transfer consumed.
4. Trusted relayer calls `sign_transfer` for the stored `transfer_id`.
5. `normalize_amount(5×10^17, {decimals:6, origin_decimals:24})` = `5×10^17 / 10^18` = **0**.
6. `require!(0 > 0, InvalidAmountToTransfer)` → **panic**.
7. Transfer message remains in `pending_transfers`; tokens are burned and unrecoverable. Every subsequent `sign_transfer` attempt produces the same panic.
