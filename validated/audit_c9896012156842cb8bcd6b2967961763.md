### Title
`setFeeBps` and `setTokenFeeBps` Allow 100% Fee (10,000 bps), Causing Depositors to Receive Zero rsETH - (File: contracts/pools/RSETHPool.sol)

---

### Summary

`RSETHPool.setFeeBps` and `RSETHPool.setTokenFeeBps` use a strict-greater-than check (`> 10_000`) that permits `feeBps = 10_000` (100%). When this boundary value is set, every depositor receives zero rsETH while their entire ETH or token deposit is silently captured as protocol fees. The root cause is structurally identical to the referenced report: an off-by-one in a parameter setter allows a boundary value that produces a degenerate downstream arithmetic result.

---

### Finding Description

`RSETHPool.setFeeBps` validates the new fee with:

```solidity
if (_feeBps > 10_000) revert InvalidFeeAmount();
```

This permits `feeBps = 10_000` (100 %). The same permissive check exists in `setTokenFeeBps`:

```solidity
if (_feeBps > 10_000) revert InvalidFeeAmount();
```

`viewSwapRsETHAmountAndFee` (ETH path) computes:

```solidity
fee = amount * feeBps / 10_000;          // = amount when feeBps == 10_000
uint256 amountAfterFee = amount - fee;   // = 0
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // = 0
```

The token path in `viewSwapRsETHAmountAndFee(amount, token)` uses `tokenFeeBps[token]` and produces the same zero result.

Both `deposit(string)` and `deposit(address,uint256,string)` call these view functions and then execute:

```solidity
feeEarnedInETH += fee;          // captures the full deposit amount
wrsETH.mint(msg.sender, 0);     // mints nothing to the user
```

No revert occurs. The transaction succeeds, the user's ETH or tokens are absorbed into `feeEarnedInETH` / `feeEarnedInToken[token]`, and the user receives zero wrsETH.

By contrast, `RSETHPoolV3.setFeeBps` caps the fee at 10 % (`> 1000`), demonstrating that the protocol's intended maximum is far below 100 %. The Arbitrum-specific `RSETHPool` contract is the outlier that allows the full 10,000 bps boundary.

---

### Impact Explanation

Any user who calls `deposit` after `feeBps` (or `tokenFeeBps[token]`) is set to `10_000` loses their entire deposit: ETH or tokens are transferred in, zero rsETH is minted out, and the deposited value accumulates in the fee accounting variables. The contract delivers none of its promised return. This maps to **Low — Contract fails to deliver promised returns, but doesn't lose value** from the protocol's perspective (fees remain withdrawable by the bridger), while from the depositor's perspective the deposit is entirely consumed.

---

### Likelihood Explanation

Setting `feeBps = 10_000` is a valid call under the current guard. An admin performing a routine fee update could supply this value believing it is the maximum allowed, without realising it collapses every subsequent deposit to zero output. The external report's judge accepted an analogous admin misconfiguration (wallSpread = 10000) as a realistic edge case warranting a Medium finding; the same reasoning applies here.

---

### Recommendation

Change both guards from strict-greater-than to greater-than-or-equal, mirroring the fix recommended in the referenced report:

```solidity
// RSETHPool.sol – setFeeBps
- if (_feeBps > 10_000) revert InvalidFeeAmount();
+ if (_feeBps >= 10_000) revert InvalidFeeAmount();

// RSETHPool.sol – setTokenFeeBps
- if (_feeBps > 10_000) revert InvalidFeeAmount();
+ if (_feeBps >= 10_000) revert InvalidFeeAmount();
```

Alternatively, align the cap with `RSETHPoolV3`'s stricter bound of `1_000` (10 %).

---

### Proof of Concept

1. Admin calls `RSETHPool.setFeeBps(10_000)` — passes the `> 10_000` guard.
2. User calls `RSETHPool.deposit{value: 1 ether}("ref")`.
3. Inside `viewSwapRsETHAmountAndFee(1 ether)`:
   - `fee = 1e18 * 10_000 / 10_000 = 1e18`
   - `amountAfterFee = 1e18 - 1e18 = 0`
   - `rsETHAmount = 0 * 1e18 / rsETHToETHrate = 0`
4. `feeEarnedInETH += 1e18` — the full ETH is captured as fees.
5. `wrsETH.mint(msg.sender, 0)` — user receives nothing.
6. Bridger later calls `withdrawFees` and recovers the 1 ETH.

The same sequence applies to `setTokenFeeBps` and the token `deposit` path.

        public

        onlySupportedToken(token)

        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount

    }
