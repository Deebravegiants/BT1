### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Byzantine Participant - (File: crates/contract/src/lib.rs)

### Summary
`respond_verify_foreign_tx` accepts a `VerifyForeignTransactionResponse` containing a caller-supplied `payload_hash` and `signature`, but only verifies that the signature is cryptographically valid over `payload_hash`. It never verifies that `payload_hash` is the correct hash for the provided `request`. A single malicious attested participant (Byzantine node below signing threshold) can replay a valid signature from any prior foreign-chain verification to resolve a completely different pending request, delivering a forged attestation to the waiting caller.

### Finding Description

In `respond_verify_foreign_tx`, the contract performs three checks:

1. Caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`)
2. Signature is valid over `response.payload_hash` against the domain root public key
3. `request` exists in `pending_verify_foreign_tx_requests` [1](#0-0) 

What is **absent** is any binding assertion that `response.payload_hash` is the correct hash for the given `request`. The correct hash would be `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request.clone(), values: <extracted_values> }).compute_msg_hash()`. The contract never recomputes or constrains this. [2](#0-1) 

Compare this to `respond`, where the payload is taken directly from the stored `request` (not from the response), making it impossible to substitute: [3](#0-2) 

The `ForeignTxSignPayload` encodes both the `ForeignChainRpcRequest` (including `tx_id`) and the extracted values: [4](#0-3) 

Because `payload_hash` is an opaque 32-byte hash, the caller of `verify_foreign_transaction` cannot independently verify that the returned `payload_hash` encodes their submitted `tx_id` — they have no access to the extracted values needed to recompute it.

This is the direct analog of the external report: just as the Solana contract accepted an `IdentityAccount` without verifying it was linked to the `WalletIdentity`, this contract accepts a `payload_hash` without verifying it is linked to the `request`.

### Impact Explanation

A single malicious attested participant can execute a **cross-request signature replay**:

1. Observe any prior on-chain `respond_verify_foreign_tx` call for transaction Y, extracting `(payload_hash_Y, sig_Y)` — both are public on-chain data.
2. Wait for a victim to submit `verify_foreign_transaction` for transaction X (or submit one on the victim's behalf).
3. Race-call `respond_verify_foreign_tx` with `request = X`, `response.payload_hash = payload_hash_Y`, `response.signature = sig_Y`.
4. The contract accepts: signature is valid over `payload_hash_Y` ✓, request X is pending ✓, no binding check ✗.
5. The victim's yield resolves with `(payload_hash_Y, sig_Y)` — a valid MPC signature that attests to transaction Y, not X.

Any bridge or downstream contract consuming this response receives a cryptographically valid attestation for the wrong transaction, enabling invalid bridge execution (e.g., crediting a deposit that never occurred on the foreign chain, or double-spending by replaying a single foreign-chain event across multiple NEAR-side claims).

**Impact class**: High — forged foreign-chain verification causing invalid bridge execution.

### Likelihood Explanation

- The attacker only needs to be a single attested participant, a role any node operator can obtain via `submit_participant_info`.
- No threshold collusion is required; the attacker reuses an existing valid signature produced by the honest network.
- All prior `respond_verify_foreign_tx` calls and their `payload_hash`/`signature` values are permanently visible on-chain.
- The attacker must win a race against honest nodes, but as the first valid `respond_verify_foreign_tx` call drains the queue, a well-positioned attacker (e.g., one monitoring the mempool or with low-latency access) can reliably win.

### Recommendation

The contract must bind `response.payload_hash` to `request` on-chain. Two approaches:

1. **Require the MPC node to provide extracted values**: Change `VerifyForeignTransactionResponse` to include `values: Vec<ExtractedValue>`. In `respond_verify_foreign_tx`, recompute `expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request.clone(), values }).compute_msg_hash()` and assert `expected_hash == response.payload_hash`.

2. **Bind the request into the signed payload at submission time**: Store a commitment to the expected `payload_hash` structure in the pending request entry so the contract can verify the binding without knowing the extracted values.

### Proof of Concept

```
// Setup: honest network previously signed tx_id = Y
// On-chain: payload_hash_Y, sig_Y are visible from a prior respond_verify_foreign_tx call

// Step 1: victim submits request for tx_id = X
victim.call(contract, "verify_foreign_transaction", { request: { tx_id: X, ... } })

// Step 2: attacker (single attested participant) replays Y's signature against X's pending request
attacker.call(contract, "respond_verify_foreign_tx", {
    request: { tx_id: X, ... },          // matches the pending queue entry
    response: {
        payload_hash: payload_hash_Y,    // hash of (tx_id=Y, extracted_values_Y)
        signature: sig_Y,                // valid MPC signature over payload_hash_Y
    }
})

// Result: contract accepts (sig_Y valid over payload_hash_Y ✓, request X pending ✓)
// Victim receives (payload_hash_Y, sig_Y) — a valid attestation for tx_id=Y, not X
// Bridge contract is deceived into executing for the wrong foreign transaction

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
