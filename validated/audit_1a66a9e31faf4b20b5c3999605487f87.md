### Title
Anyone Can Arbitrarily Call `FeeReceiver.sendFunds()` to Front-Run MEV Reward Distribution - (File: contracts/FeeReceiver.sol)

---

### Summary

`FeeReceiver.sendFunds()` has no access control modifier. Any external caller can trigger the transfer of the entire accumulated MEV/execution-layer ETH balance from `FeeReceiver` to `LRTDepositPool` at any time, allowing an attacker to control the exact moment accumulated rewards are added to the protocol TVL and thus manipulate the rsETH price update to steal yield from existing holders.

---

### Finding Description

`FeeReceiver` is the protocol's MEV and execution-layer reward accumulator. Its `sendFunds()` function is intended to be called by a privileged manager to flush accumulated ETH rewards into the deposit pool at a controlled time. However, the function carries no access control:

```solidity
// contracts/FeeReceiver.sol, line 53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

The `receiveFromRewardReceiver()` target in `LRTDepositPool` is also unguarded — it is a bare `external payable {}` stub that simply accepts ETH and increases the deposit pool's balance:

```solidity
// contracts/LRTDepositPool.sol, line 61
function receiveFromRewardReceiver() external payable { }
``` [2](#0-1) 

The deposit pool's ETH balance is included in `_getTotalEthInProtocol()`, which drives the rsETH price calculation inside `_updateRsETHPrice()`. `updateRSETHPrice()` is itself a public, permissionless function:

```solidity
// contracts/LRTOracle.sol, line 87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [3](#0-2) 

The price update mints protocol fee rsETH and sets the new exchange rate used for all subsequent deposits and withdrawals: [4](#0-3) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

An attacker who observes accumulated ETH in `FeeReceiver` can execute the following sandwich:

1. **Deposit ETH** into `LRTDepositPool` at the current (pre-reward) rsETH price, receiving rsETH that represents a proportional share of the entire pool including the not-yet-distributed MEV rewards.
2. **Call `FeeReceiver.sendFunds()`** — this immediately moves all accumulated MEV rewards into the deposit pool, inflating TVL.
3. **Call `LRTOracle.updateRSETHPrice()`** — the new price reflects the inflated TVL; the attacker's rsETH is now worth more.
4. **Initiate a withdrawal** at the higher price and claim after the delay.

Concretely: if existing holders have 100 rsETH backed by 100 ETH (price = 1.0), and `FeeReceiver` holds 10 ETH in accumulated rewards, an attacker who deposits 100 ETH (receiving 100 rsETH) and then calls `sendFunds()` + `updateRSETHPrice()` causes the price to become 210/200 = 1.05. The attacker withdraws 105 ETH (net +5 ETH). Existing holders' share of the 10 ETH reward drops from 10 ETH to 5 ETH — the other 5 ETH is captured by the attacker. This is direct theft of unclaimed yield from legitimate rsETH holders.

A secondary impact: if accumulated rewards are large enough to push the new price above `pricePercentageLimit`, calling `sendFunds()` followed by `updateRSETHPrice()` will revert for non-managers (line 263–265 of `LRTOracle.sol`), temporarily blocking price updates and potentially triggering the downside-protection pause path. [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The attack is fully permissionless and requires no special role. The only friction is the `withdrawalDelayBlocks` (~8 days) before the attacker can claim. The accumulated MEV balance in `FeeReceiver` is publicly visible on-chain, making it straightforward to monitor and time the attack. Any holder of rsETH (or anyone willing to deposit ETH) can execute this.

---

### Recommendation

Add an access control modifier to `sendFunds()` so that only a trusted role (e.g., `LRTConstants.MANAGER`) can trigger the reward flush:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

This mirrors the fix recommended in H-01: ensure the function is only callable from a trusted, whitelisted source rather than being open to any external caller.

---

### Proof of Concept

1. Observe `FeeReceiver` has accumulated `R` ETH in MEV rewards (publicly visible).
2. Call `LRTDepositPool.depositETH{value: D}(0, "")` — receive `D / currentPrice` rsETH.
3. Call `FeeReceiver.sendFunds()` — `R` ETH moves to `LRTDepositPool`.
4. Call `LRTOracle.updateRSETHPrice()` — new price = `(TVL_before + D + R) / totalRsETH`.
5. Call `LRTWithdrawalManager.initiateWithdrawal(...)` at the new higher price.
6. After `withdrawalDelayBlocks`, claim withdrawal — receive `D + (R * D / (TVL_before + D))` ETH.
7. Net profit = `R * D / (TVL_before + D)` ETH, stolen from existing rsETH holders' unclaimed yield.
