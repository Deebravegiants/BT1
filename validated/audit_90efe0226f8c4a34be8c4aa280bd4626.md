### Title
Missing Zero-Check Before Division by `totalStake` in Epoch Reward Calculation Causes Runtime Exception Freezing All Rewards - (File: `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rewards.hs`)

---

### Summary

`rewardOnePoolMember` and `mkPoolRewardInfo` in `Rewards.hs` both divide by `totalStake` (total circulating ADA) using Haskell's raw `%` operator from `Data.Ratio`, without first checking whether `totalStake` is zero. The code itself acknowledges this with explicit `-- warning: totalStake could be zero!` comments. If `totalStake` is zero at epoch boundary, both functions throw a `Ratio has zero denominator` runtime exception, aborting the pulsing reward update computation on every honest node and permanently freezing all staking rewards until a hard fork.

---

### Finding Description

In `rewardOnePoolMember`:

```haskell
-- warning: totalStake could be zero!
stakeShare = StakeShare $ c % unCoin totalStake   -- line 320
```

`%` is `Data.Ratio.(%)`, which unconditionally throws `Ratio has zero denominator` when the second argument is `0`. There is no guard.

In `mkPoolRewardInfo`:

```haskell
-- warning: In theory `totalStake` and `totalActiveStake` could be zero...
poolRelativePledge = pledge % totalStake          -- line 390
```

Same raw `%` operator, same missing guard. Note the contrast with the two lines immediately below it:

```haskell
poolOwnerRelativeStake = selfDelegatedOwnersStake %? totalStake   -- line 391
sigma = poolTotalStake %? totalStake                              -- line 392
```

Those two use the safe `%?` operator (defined in `NonZero.hs` as returning `0` when the denominator is `0`), but `poolRelativePledge` and the `stakeShare` in `rewardOnePoolMember` do not.

The `totalStake` value is computed as `circulation es maxSupply` in `startStep` (line 144 of `PulsingReward.hs`), which equals `maxLovelaceSupply − reserves`. It is then stored in `FreeVars` (line 202) and passed unchanged into every call to `rewardOnePoolMember` via the pulser.

The call chain that reaches the unsafe division is:

```
startStep → (builds Pulser with totalStake)
  pulseStep / completeStep
    → rewardStakePoolMember (RewardUpdate.hs)
      → rewardOnePoolMember (Rewards.hs line 320)
        → c % unCoin totalStake   ← THROWS if totalStake == 0
```

and separately:

```
startStep
  → mkPoolRewardInfo (Rewards.hs line 390)
    → pledge % totalStake         ← THROWS if totalStake == 0
```

The `%?` operator is already available in scope (imported from `BaseTypes`) and is used correctly on lines 391–392 of the same `where` block, confirming the fix is known and partially applied but not consistently applied.

---

### Impact Explanation

If `totalStake` is zero at the epoch boundary when `startStep` fires, every call to `rewardOnePoolMember` for every registered stake credential throws a `Ratio has zero denominator` Haskell runtime exception. This aborts the reward update computation (`PulsingRewUpdate`) on all honest nodes simultaneously. Because the epoch transition cannot complete, no rewards are ever distributed and the ledger is permanently stuck at that epoch boundary. Recovery requires a hard fork to patch the division and replay the epoch transition. This matches:

> **High. Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork.**

---

### Likelihood Explanation

`totalStake = maxLovelaceSupply − reserves`. It is zero when all ADA remains in reserves and none has entered circulation — a condition that holds at genesis and could theoretically recur if a governance action drained all circulating ADA back to reserves. The developers themselves flag this with two separate `-- warning: totalStake could be zero!` comments in the same file, confirming they consider the condition reachable. The likelihood is low on mainnet today but non-negligible across the full protocol lifetime, and the code provides no mitigation.

---

### Recommendation

Replace the two unsafe `%` divisions with the safe `%?` operator already used on the adjacent lines:

In `rewardOnePoolMember` (line 320):
```haskell
-- Before
stakeShare = StakeShare $ c % unCoin totalStake
-- After
stakeShare = StakeShare $ c %? unCoin totalStake
```

In `mkPoolRewardInfo` (line 390):
```haskell
-- Before
poolRelativePledge = pledge % totalStake
-- After
poolRelativePledge = pledge %? totalStake
```

`%?` is already imported and used correctly on lines 391–392 of the same `where` block.
