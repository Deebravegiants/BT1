### Title
Proposal Deposit Double-Counted in Both DRep and SPO Voting Distributions for `DelegStakeVote` Credentials - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Governance/DRepPulser.hs`)

### Summary

In the Conway governance system, a staking credential that delegates to **both** a DRep and a stake pool via `DelegStakeVote` will have its governance proposal deposit counted in **both** the DRep voting distribution and the SPO pool voting distribution simultaneously. An attacker who controls or delegates to both a DRep and an SPO can submit multiple governance proposals to inflate their voting weight in both governance bodies beyond what their actual locked ADA supports.

### Finding Description

The `computeDRepDistr` function in `eras/conway/impl/src/Cardano/Ledger/Conway/Governance/DRepPulser.hs` iterates over all staking accounts at epoch boundary to build the DRep and SPO stake distributions used for ratification. For each account it calls both `addToDRepDistr` and `addToPoolDistr`:

```haskell

  let ...

      stakeAndDeposits = fold $ mInstantStake <> mProposalDeposit
   in ( addToDRepDistr accountState stakeAndDeposits drepAccum   -- deposit counted here
      , addToPoolDistr accountState mProposalDeposit poolAccum   -- deposit counted here too
      )
``` [1](#0-0) 

`addToDRepDistr` adds `stakeAndDeposits <> balance` (which includes `mProposalDeposit`) to the DRep's entry in `reDRepDistr`: [2](#0-1) 

`addToPoolDistr` adds `mProposalDeposit` to the SPO's entry in `reStakePoolDistr`: [3](#0-2) 

The code comment explicitly acknowledges that regular stake and rewards are already in the SNAP snapshot for SPOs, so only proposal deposits are added there. However, for a credential using `DelegStakeVote`, the same proposal deposit ADA is added to **both** distributions in the same pulser pass. The `dRepAcceptedRatio` and `spoAcceptedRatio` functions then each independently use these inflated distributions: [4](#0-3) [5](#0-4) 

The `DelegStakeVote` delegatee type, which enables simultaneous delegation to both a pool and a DRep, is a first-class supported certificate type: [6](#0-5) 

The pulser is initialized at epoch boundary with a snapshot of `proposalDeposits` (a map from staking credential to total deposit amount across all live proposals for that credential): [7](#0-6) 

### Impact Explanation

**Medium.** An attacker who controls or delegates to both a registered DRep and a stake pool can submit N governance proposals (each costing `ppGovActionDeposit` ADA) to inflate their voting weight in **both** the DRep ratification check and the SPO ratification check by `N × ppGovActionDeposit` ADA simultaneously. A single unit of locked ADA (the deposit) is counted once toward `dRepAcceptedRatio` and once toward `spoAcceptedRatio`, exceeding the intended one-ADA-one-vote accounting. This allows governance actions to be ratified with a lower true stake backing than the thresholds require, modifying governance outcomes outside design parameters. This matches the Medium impact class: *"Attacker-controlled transactions… exceed intended validation limits or modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters."*

### Likelihood Explanation

**Low-Medium.** The attacker must:
1. Control or delegate to both a registered DRep and a registered stake pool.
2. Submit multiple proposals (each requiring `ppGovActionDeposit` ADA, currently 100,000 ADA on mainnet) and keep them alive until the target proposal is ratified.
3. Have the DRep and SPO vote Yes on the target proposal.

The capital cost of submitting many proposals is significant, but the attack is permissionless and requires no privileged access — any proposal author with a `DelegStakeVote` credential can trigger it. The effect scales linearly with the number of live proposals.

### Recommendation

For a staking credential that has delegated to both a DRep and an SPO (`DelegStakeVote`), the proposal deposit should be counted in **at most one** of the two distributions, not both. The simplest fix is to exclude `mProposalDeposit` from `addToDRepDistr` when the same credential also has an SPO delegation (i.e., when `addToPoolDistr` will also count it), or alternatively to exclude it from `addToPoolDistr` when a DRep delegation is present. A more principled fix would track proposal deposits as a separate, non-duplicated component and apply them to only one voting body per credential.

### Proof of Concept

1. Register staking credential `C` and delegate it to DRep `D` and pool `P` via `DelegStakeVote P (DRepCredential D)`.
2. Submit `N` governance proposals using `C` as the return address. Each proposal locks `ppGovActionDeposit` ADA attributed to `C` in `proposalsDeposits`.
3. At the next epoch boundary, `setFreshDRepPulsingState` snapshots `proposalDeposits` containing `C → N × ppGovActionDeposit`.
4. `computeDRepDistr` processes `C`:
   - `addToDRepDistr` adds `N × ppGovActionDeposit` to `D`'s entry in `reDRepDistr`.
   - `addToPoolDistr` adds `N × ppGovActionDeposit` to `P`'s entry in `reStakePoolDistr`.
5. When `dRepAcceptedRatio` and `spoAcceptedRatio` run, both see the inflated stake. The same `N × ppGovActionDeposit` ADA has been counted twice — once per voting body — allowing both ratification thresholds to be met with less true backing stake than required. [8](#0-7)

    { reStakePoolDistr = PoolDistr individualPoolStake totalActiveStake
    , reAccounts
    , reStakePools
    }
  GovActionState
    { gasStakePoolVotes
    , gasProposalProcedure = ProposalProcedure {pProcGovAction}
    }
  pv =
    toInteger yesStake %? (unCoin (unNonZero totalActiveStake) - toInteger abstainStake)
    where

            vote = Map.lookup poolId gasStakePoolVotes
         in case vote of

                | hardforkConwayBootstrapPhase pv -> (yes, abstain + stake)

              Just Abstain -> (yes, abstain + stake)
              Just VoteNo -> (yes, abstain)

dRepAcceptedRatio ::
  forall era.
  RatifyEnv era ->
  Map (Credential DRepRole) Vote ->
  GovAction era ->
  Rational
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
