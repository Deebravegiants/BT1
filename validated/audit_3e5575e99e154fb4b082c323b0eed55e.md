### Title
`respond_verify_foreign_tx` Does Not Verify `payload_hash` Binds to the Submitted `request` - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over the caller-supplied `response.payload_hash`, but **never checks that `response.payload_hash` was actually derived from the `request` argument**. The contract's name and purpose imply it verifies a foreign-chain transaction, but the critical binding between the on-chain request and the signed hash is absent — a direct analog to the ERC20 `safeTransferFrom` that claims safety but omits the essential check.

### Finding Description

When a user calls `verify_foreign_transaction`, the contract stores the `VerifyForeignTransactionRequest` (containing `tx_id`, chain, extractors, `payload_version`) and parks a yield. MPC nodes are expected to:

1. Query the foreign chain for the transaction.
2. Run the requested extractors to produce `ExtractedValue`s.
3. Compute `payload_hash = SHA-256(borsh(ForeignTxSignPayload { request, values }))`.
4. Threshold-sign `payload_hash` and call `respond_verify_foreign_tx`.

The contract's `respond_verify_foreign_tx` performs only one cryptographic check:

```rust
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← taken directly from response, never re-derived
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

It verifies `sig` over `response.payload_hash`, but **never recomputes or constrains `payload_hash`** from the stored `request`. There is no assertion of the form:

```
assert!(response.payload_hash == SHA-256(borsh(ForeignTxSignPayload { request, any_values })))
```

Compare with `respond()` for plain sign requests, where the payload is embedded in the stored `SignatureRequest` key itself and cannot be substituted:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(signature_response, payload_hash, ...)
``` [2](#0-1) 

For `respond_verify_foreign_tx`, the `payload_hash` is a free field in the response DTO, not derived from stored state:

```rust
pub struct VerifyForeignTransactionResponse {
    pub payload_hash: Hash256,   // ← attacker-controlled
    pub signature: SignatureResponse,
}
``` [3](#0-2) 

The correct `payload_hash` must commit to both the original `request` and the `ExtractedValue`s:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
// msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [4](#0-3) 

The contract never enforces that `response.payload_hash` encodes the correct `request` field.

### Impact Explanation

The `verify_foreign_transaction` / `respond_verify_foreign_tx` flow is the on-chain attestation primitive for the Omnibridge inbound path: a bridge contract calls `verify_foreign_transaction` and uses the returned `(payload_hash, signature)` to prove a foreign-chain event occurred. If `payload_hash` can be substituted for one encoding fake `ExtractedValue`s (e.g., a different amount, a different recipient, a non-existent block hash), the bridge contract receives a valid MPC-signed attestation for a fabricated event and releases funds accordingly.

This maps to the **High** impact category: *forged foreign-chain verification that causes invalid bridge execution or double-spend conditions*.

### Likelihood Explanation

The attacker must be a Byzantine leader node that is an attested participant (passed TEE attestation). The leader node controls the final step of the Cait-Sith signing computation where the message hash is consumed. Follower nodes independently execute the foreign chain request and compute their own `sign_request`:

```rust
let response_payload = self.execute_foreign_chain_request(...).await?;
let sign_request = build_signature_request(&foreign_tx_request, &response_payload)?;
self.ecdsa_signature_provider
    .make_signature_follower_given_request(channel, presignature_id, sign_request)
    .await
``` [5](#0-4) 

However, in Cait-Sith the message hash enters only in the leader's final aggregation step; follower contributions are presignature-based and do not bind to the message. A Byzantine leader can therefore use honest followers' presignature shares to produce a valid threshold signature over a chosen `payload_hash_fake`, then submit it to the contract. The contract accepts it because it only checks `verify_ecdsa_signature(sig, payload_hash_fake, root_pk)` — which passes — without checking that `payload_hash_fake` encodes the original `request`.

The TEE attestation requirement raises the bar (the attacker must be an attested participant), but a single compromised or malicious TEE node below the signing threshold is sufficient to be the leader and execute this substitution.

### Recommendation

In `respond_verify_foreign_tx`, re-derive the expected `payload_hash` prefix from the stored `request` and reject any response whose `payload_hash` does not commit to the correct `request` field. Concretely, the contract should verify:

```rust
// The payload_hash must encode the original request (any valid extracted values are fine,
// but the request portion must match).
// One approach: store the request's canonical prefix hash at enqueue time and check it here.
```

Alternatively, store the `ForeignTxSignPayload` (or at least a commitment to the `request` portion) in the pending-request map at `verify_foreign_transaction` time, and verify that `response.payload_hash` is consistent with it before accepting the response. This mirrors how `respond()` embeds the payload directly in the stored `SignatureRequest` key, making substitution impossible. [6](#0-5) 

### Proof of Concept

1. User calls `verify_foreign_transaction` for Bitcoin tx `[0xAA; 32]` with extractor `BlockHash`. Contract stores the request and parks a yield.
2. Byzantine leader node sees the pending request. It queries Bitcoin and gets the real block hash `0xREAL`.
3. Instead of signing `SHA-256(borsh(ForeignTxSignPayload { request: Bitcoin([0xAA;32]), values: [BlockHash(0xREAL)] }))`, the leader constructs `payload_hash_fake = SHA-256(borsh(ForeignTxSignPayload { request: Bitcoin([0xAA;32]), values: [BlockHash(0xFAKE)] }))`.
4. The leader initiates threshold signing with `payload_hash_fake` as the message. Honest followers contribute presignature shares (they do not bind their contribution to the message in Cait-Sith's offline phase). The leader aggregates a valid threshold signature `sig_fake` over `payload_hash_fake`.
5. Leader calls:
   ```
   respond_verify_foreign_tx(
     request = { tx_id: [0xAA;32], chain: Bitcoin, ... },
     response = { payload_hash: payload_hash_fake, signature: sig_fake }
   )
   ```
6. Contract checks: `verify_ecdsa_signature(sig_fake, payload_hash_fake, root_pk)` → **passes**.
7. Contract does **not** check: `payload_hash_fake` encodes `request = Bitcoin([0xAA;32])` → **no check**.
8. Contract resolves the yield and returns `{ payload_hash: payload_hash_fake, signature: sig_fake }` to the caller.
9. The Omnibridge contract receives a valid MPC-signed attestation claiming Bitcoin tx `[0xAA;32]` produced block hash `0xFAKE`, and releases funds based on this fabricated observation. [7](#0-6) [8](#0-7)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
