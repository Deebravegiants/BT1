### Title
Missing `numDormantEpochs` Adjustment in DRep Expiry Check Causes Incorrect Exclusion from Ratification - (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

### Summary

`dRepAcceptedRatio` in the `RATIFY` rule checks DRep expiry using only the raw stored `drepExpiry` field, without adding `vsNumDormantEpochs`. The Cardano ledger tracks a separate "actual" expiry = `drepExpiry + numDormantEpochs` to account for dormant periods when no governance proposals exist. During a dormant period, a DRep whose raw expiry has passed but whose actual expiry has not is incorrectly treated as expired in the ratification calculation, silently discarding its votes and stake from the denominator.

### Finding Description

The Conway governance system tracks DRep activity through two related values:

1. `drepExpiry` â€” the raw epoch stored in `DRepState`, bumped when a new proposal is submitted (via `updateDormantDRepExpiries`).
2. `vsNumDormantEpochs` â€” a counter incremented each epoch in which there are zero active governance proposals, reset to zero when a new proposal is submitted.

The "actual" expiry of a DRep is `drepExpiry + numDormantEpochs`, as implemented in `vsActualDRepExpiry`: [1](#0-0) 

The test helper `isDRepExpired` also uses this two-field formula: [2](#0-1) 

However, `dRepAcceptedRatio` in the `RATIFY` rule checks only the raw `drepExpiry`: [3](#0-2) 

The `RatifyEnv` type does not carry `numDormantEpochs` at all â€” it has `reDRepState` (raw expiry) and `reCurrentEpoch`, but no dormant-epoch counter â€” so the ratification rule structurally cannot perform the correct check.

**Concrete scenario:**

| Epoch | Event | `drepExpiry` | `numDormantEpochs` | Actual expiry |
|-------|-------|-------------|-------------------|---------------|
| 0 | DRep registers (`drepActivity=2`) | 2 | 0 | 2 |
| 1 | Proposal P submitted; DRep votes **No** | 2 | 0 | 2 |
| 2 | No proposals | 2 | 1 | 3 |
| 3 | No proposals | 2 | 2 | 4 |
| 4 | Ratification of P runs (`reCurrentEpoch=3`) | 2 | 2 | 4 |

At epoch 4, the ratification check is `3 > 2` â†’ **true** â†’ DRep treated as expired. But the actual expiry is 4, and `3 â‰¤ 4`, so the DRep is still active. Its **No** vote and stake are silently dropped from the denominator, making the proposal easier to ratify.

The `CERTS` rule correctly bumps expiries when a new proposal is submitted: [4](#0-3) 

But the pulser snapshot used by `RATIFY` is taken at the epoch boundary, before any transactions in the new epoch. If the dormant period spans multiple epochs after the last proposal, the snapshot carries the old raw expiry while `numDormantEpochs > 0`, and the ratification check is wrong.

The `GOV` rule does enforce a correct expiry check for vote submission: [5](#0-4) 

But this only prevents submitting new votes on expired actions; it does not fix the ratification-time expiry check for DReps.

### Impact Explanation

**Medium â†’ potential Critical.** A governance proposer can benefit from this bug by submitting a proposal and then waiting for a natural dormant period (no new proposals for several epochs) to elapse. DReps with significant **No** stake whose raw expiry falls within the dormant window are silently excluded from the ratification denominator. Their **No** votes are discarded, and the proposal's acceptance ratio is artificially inflated. This can cause treasury withdrawals, hard-fork initiations, constitutional changes, or committee updates to be ratified and enacted despite having sufficient opposing DRep stake â€” matching the "unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted" impact category.

### Likelihood Explanation

**Medium.** The condition arises naturally without any attacker action: it requires only that (a) a governance proposal remains unratified across multiple epochs with no new proposals, and (b) at least one DRep's raw expiry falls in the resulting dormant window. Both conditions are plausible in normal network operation, especially for DReps with short `drepActivity` values. No privileged access, key compromise, or consensus majority is required.

### Recommendation

Add `numDormantEpochs` to `RatifyEnv` and use it in `dRepAcceptedRatio`:

```haskell
-- In RatifyEnv, add:
reNumDormantEpochs :: EpochNo

-- In dRepAcceptedRatio, change:
| reCurrentEpoch > drepExpiry drepState -> (yes, tot)
-- to:
| reCurrentEpoch > binOpEpochNo (+) (drepExpiry drepState) reNumDormantEpochs -> (yes, tot)
```

Alternatively, pre-compute the actual expiry for each DRep before passing `reDRepState` to the pulser, so the stored value already reflects the dormant-epoch adjustment.

### Proof of Concept

1. Set `ppDRepActivityL = EpochInterval 2`.
2. Register a DRep; it gets `drepExpiry = currentEpoch + 2`.
3. Submit proposal P; DRep votes **No**.
4. Pass 3 epochs with no new proposals (`numDormantEpochs` reaches 3).
5. Observe: `isDRepExpired drep` returns `False` (actual expiry = raw + 3 > current epoch).
6. Observe: `dRepAcceptedRatio` in the pulser treats the DRep as expired (raw expiry < current epoch), dropping its **No** vote and stake from the denominator.
7. Proposal P reaches the required acceptance threshold and is ratified, despite the DRep's **No** vote.

The discrepancy between `isDRepExpired` (which uses `drepExpiry + numDormantEpochs`) and `dRepAcceptedRatio` (which uses only `drepExpiry`) is the root cause. [6](#0-5) [7](#0-6)

```haskell
dRepAcceptedRatio ::
  forall era.
  RatifyEnv era ->
  Map (Credential DRepRole) Vote ->
  GovAction era ->
  Rational
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
```
