Based on my analysis of the code, I can now provide a definitive answer.

**Tracing the full attack path:**

1. A malicious peer connects and advertises a fake `highest_block_height` (e.g., `u64::MAX`) in their chain info.

2. In `peer_manager_actor.rs:highest_height_peers()`, the max height across all peers is computed. The malicious peer's fake height becomes the max, and legitimate peers (at real heights far below) are **excluded** from `highest_height_peers` because their height + `highest_peer_horizon` < fake max. [1](#0-0) 

3. In `client_actor.rs:syncing_info()`, `eligible_peers` is filtered from `highest_height_peers` by `is_block_invalid`. The malicious peer's `highest_block_hash` is an unknown hash — not in the invalid set — so it passes. The malicious peer is the only eligible peer, and their fake height becomes `highest_height`. [2](#0-1) 

4. In `StateSync::run()`, the check fires immediately: [3](#0-2) 

5. `StaleSyncHash` → `EpochSyncDataReset` → `ShutdownReason::EpochSyncDataReset` → marker file written → process re-execs → **entire data directory wiped on next startup**. [4](#0-3) [5](#0-4) [6](#0-5) 

**No guard prevents this.** The `is_block_invalid` filter only catches hashes explicitly marked invalid — an unknown fake hash passes. There is no cryptographic proof required for a peer's advertised height. The `STALE_SYNC_HASH_THRESHOLD = 100` in production is trivially bypassed by claiming any height > `sync_hash_height + epoch_length + 100`. [7](#0-6) 

**However**, the impact classification matters here. The consequence — data wipe and restart — is a **persistent denial of service** against a syncing node. The node can never complete state sync as long as the malicious peer maintains a connection. This is explicitly within the stated scope ("fail to recover canonical state").

But the rejection criteria states: *"Malicious peer behavior is only a trigger, not an impact: reject reports where the whole issue is that a peer can send malformed, stale, duplicated, conflicting, oversized, or unsolicited data and nearcore rejects, ignores, rate-limits, disconnects, retries, or only wastes resources."*

In this case, nearcore does **not** reject, ignore, or rate-limit — it takes a destructive irreversible action (data wipe). This is beyond "only wastes resources." The attack is concrete, locally testable (the existing `test_far_horizon_stale_sync_hash_detection` test under `test_features` demonstrates the exact mechanism), and repeatable indefinitely. [8](#0-7)

### Title
Unverified peer-supplied `highest_height` triggers unconditional data-directory wipe during state sync — (`chain/client/src/sync/state/mod.rs`)

### Summary
A single malicious peer can advertise a fake `highest_block_height` that exceeds `sync_hash_height + epoch_length + STALE_SYNC_HASH_THRESHOLD`, causing a victim node mid-state-sync to immediately trigger `StateSyncResult::StaleSyncHash`, write the `.EPOCH_SYNC_DATA_RESET` marker, wipe its entire data directory on restart, and loop indefinitely — permanently preventing the node from recovering canonical state.

### Finding Description
`StateSync::run()` checks whether `highest_height > block_header.height() + chain.epoch_length + STALE_SYNC_HASH_THRESHOLD` to detect a stale sync hash. The `highest_height` value is taken directly from a randomly chosen peer's self-reported `highest_block_height` field with no cryptographic verification. A malicious peer advertising a sufficiently large fake height will:

1. Become the sole entry in `highest_height_peers` (legitimate peers are filtered out because their real height + `highest_peer_horizon` < fake max).
2. Pass the `is_block_invalid` filter (the fake `highest_block_hash` is unknown, not marked invalid).
3. Supply a `highest_height` that unconditionally triggers `StaleSyncHash`.

The resulting `EpochSyncDataReset` flow writes a marker file, re-execs the process, and on the next startup deletes the entire hot store data directory. The malicious peer can repeat this on every subsequent sync attempt.

### Impact Explanation
The victim node can never complete state sync. Every time it reaches `SyncStatus::StateSync`, the malicious peer triggers a data wipe and restart, sending the node back to square one. This permanently prevents the node from recovering canonical state — a targeted, repeatable, zero-cost DoS against any syncing non-archival node.

### Likelihood Explanation
Any unprivileged actor who can establish a TCP connection to the victim node's P2P port can execute this attack. No validator keys, stake, or special privileges are required. The attack requires only a single connection and a single fake chain-info message. The `STALE_SYNC_HASH_THRESHOLD = 100` in production provides no meaningful protection since the attacker can claim any height.

### Recommendation
Do not use peer-supplied `highest_height` as the sole input to the stale-sync-hash check. Instead, verify staleness against the node's own locally-verified header head (i.e., `chain.header_head().height`), which reflects headers the node has actually downloaded and validated. A peer's self-reported height should only be used for requesting data, never for triggering destructive local state transitions.

### Proof of Concept
The existing `test_far_horizon_stale_sync_hash_detection` test (under `test_features`) demonstrates the exact mechanism: it drops state part responses to keep the node in `StateSync`, then advances the chain past the threshold. Replacing the chain advancement with a single fake `highest_height` injection into `NetworkInfo.highest_height_peers` would trigger the same `EpochSyncDataReset` without any real chain advancement, confirming the peer-controlled trigger. [9](#0-8)
