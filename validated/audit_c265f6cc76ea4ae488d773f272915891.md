Audit Report

## Title
Unconditional Zero-Value Fee Transfer in `claim()` Permanently Freezes All User Claims - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `safeTransfer(protocolTreasury, fee)` at line 144 regardless of whether `fee` is zero. When `feeInBPS` is set to `0` — a valid configuration explicitly permitted by the contract — and the distributed token reverts on zero-value transfers, every invocation of `claim()` reverts, permanently preventing all users from claiming their allocated tokens.

## Finding Description
In `contracts/utils/MerkleDistributor/MerkleDistributor.sol`, the `claim()` function at lines 138–144 computes the fee and unconditionally executes two `safeTransfer` calls:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;   // line 138
uint256 amountToSend = claimableAmount - fee;           // line 139

IERC20(token).safeTransfer(account, amountToSend);     // line 141

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);     // line 144 — always called
``` [1](#0-0) 

When `feeInBPS == 0`, `fee` evaluates to `0`, and `safeTransfer(protocolTreasury, 0)` is called unconditionally. Tokens that implement `require(amount > 0)` (e.g., `LEND`, `cUSDCv3`) will revert on this call, causing the entire transaction to revert.

`feeInBPS = 0` is explicitly permitted by both `initialize()` and `setFeeInBPS()`, which only reject values strictly greater than `MAX_FEE_IN_BPS` (1000):

```solidity
if (_feeInBPS > MAX_FEE_IN_BPS) {
    revert InvalidFeeInBPS();
}
``` [2](#0-1) [3](#0-2) 

The sibling contract `KernelMerkleDistributor` already applies the correct guard in its `_processClaim()` internal function, confirming the protocol is aware of this pattern:

```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
``` [4](#0-3) 

This guard is absent in `MerkleDistributor`, making it inconsistent and vulnerable.

## Impact Explanation
When triggered, no user can successfully call `claim()`. All unclaimed tokens remain locked in the contract with no alternative withdrawal path for users. This constitutes **permanent freezing of unclaimed yield**, which is a valid Medium impact in the allowed scope.

## Likelihood Explanation
Two conditions must hold simultaneously:
1. `feeInBPS` is set to `0` — a normal operational choice (zero-fee distribution) explicitly permitted by the contract's validation logic, requiring no privileged compromise beyond routine admin configuration.
2. The configured `token` reverts on zero-value transfers — a known, documented property of several deployed ERC-20 tokens.

Neither condition is adversarial. The combination is realistic whenever the protocol deploys a zero-fee distributor for a token with this transfer restriction. Once triggered, the freeze is permanent for all claimants with no recovery path.

## Recommendation
Guard the fee transfer with a zero-check, mirroring the pattern already used in `KernelMerkleDistributor`:

```solidity
// Before (vulnerable):
IERC20(token).safeTransfer(protocolTreasury, fee);

// After (fixed):
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
``` [5](#0-4) 

## Proof of Concept
1. Deploy a mock ERC-20 token that reverts on zero-value transfers (`require(amount > 0, "zero transfer")`).
2. Deploy `MerkleDistributor` with `feeInBPS = 0`, the mock token, and a valid `protocolTreasury`.
3. Owner calls `setMerkleRoot(root)` with a valid distribution root; `currentIndex` becomes `1`.
4. Fund the distributor with tokens.
5. Any user calls `claim(1, account, cumulativeAmount, merkleProof)` with a valid proof and non-zero `cumulativeAmount`.
6. Execution reaches line 138: `fee = (claimableAmount * 0) / 10_000 = 0`.
7. Line 141 succeeds: `safeTransfer(account, claimableAmount)`.
8. Line 144 reverts: `safeTransfer(protocolTreasury, 0)` — the mock token rejects the zero-value transfer.
9. The entire transaction reverts. Every subsequent `claim()` call hits the same revert path, permanently freezing all unclaimed yield.
