### Title
`vote_tee_verifier_change` Discards `expected_code_hash` After Threshold — No On-Chain Code-Immutability Enforcement for Trusted Verifier — (`File: crates/contract/src/lib.rs`)

---

### Summary

`mpc-contract` implements a threshold-vote mechanism (`vote_tee_verifier_change`) to elect a trusted `tee-verifier` contract account. Voters commit to a `(candidate_account_id, expected_code_hash)` pair, but once the vote crosses threshold, **only the `account_id` is stored** — the `expected_code_hash` is silently discarded. The contract then calls `verify_quote` on that account with no runtime check that the deployed code still matches the hash voters audited. Because NEAR contracts cannot read another account's code hash or access-key list, there is no on-chain enforcement that the verifier account is locked. If the verifier account retains full-access keys, its owner can silently replace the WASM after the vote passes, causing `submit_participant_info` to call a malicious verifier that returns `VerificationResult::Verified` for any input — bypassing all TEE attestation checks and allowing non-genuine nodes to join the MPC participant set.

---

### Finding Description

**Root cause — `vote_tee_verifier_change` drops the code hash on threshold crossing:**

In `crates/contract/src/lib.rs` lines 1642–1647, when the vote crosses threshold, `TeeVerifierVotes::vote()` returns only `proposal.candidate_account_id` (see `crates/contract/src/tee/verifier_votes.rs` line 84), and the contract stores it:

```rust
if let Some(new_verifier) =
    self.tee_verifier_votes
        .vote(proposal, participant, threshold_parameters)?
{
    self.tee_verifier_account_id = Some(new_verifier);  // account_id only; code_hash gone
}
```

The `expected_code_hash` field of `VerifierChangeProposal` is used exclusively to differentiate vote buckets (preventing two voters who disagree on the hash from combining into a winning quorum). It is never stored in contract state and never re-checked at call time.

**No runtime code-hash verification:**

The design doc (`docs/design/attestation-verifier-contract.md` lines 231–236) explicitly acknowledges:

> `mpc-contract` cannot verify either claim itself — a NEAR contract has no way to read another account's code hash or access-key list — so the audit is the voter's responsibility, not the contract's.

The two security requirements — (1) the right code is deployed, (2) the account has no full-access keys — are entirely off-chain obligations with no on-chain enforcement.

**The `tee-verifier` integration is already partially deployed:**

- `crates/tee-verifier/src/lib.rs` implements the `verify_quote` WASM contract.
- `vote_tee_verifier_change` and `tee_verifier_account_id` are live in the current contract state.
- The design doc (`docs/design/attestation-verifier-contract.md` lines 386–461) describes `submit_participant_info` calling `Promise::new(self.tee_verifier_account_id.clone()).function_call("verify_quote", ...)` — the integration is marked "Ready for development."

Once that integration lands, every `submit_participant_info` call routes through whatever WASM is currently deployed at `tee_verifier_account_id`, with no check that it still matches the hash the voters audited.

---

### Impact Explanation

**Impact: High — participant/attestation authorization bypass.**

A malicious `tee-verifier` that unconditionally returns `VerificationResult::Verified` causes `resolve_verification` to accept any submitted attestation as genuine. The accepted node is stored in `stored_attestations`, passes `assert_caller_is_attested_participant_and_protocol_active`, and can call `respond`, `respond_ckd`, and `respond_verify_foreign_tx`. A non-genuine TEE node that gains participant status can:

- Participate in threshold signing rounds, potentially learning partial key-share material.
- Submit fraudulent `respond_verify_foreign_tx` responses, causing the contract to issue signatures over attacker-chosen foreign-chain payloads — a forged foreign-chain verification / bridge execution bypass.
- Participate in resharing, potentially corrupting the next key epoch.

This matches the allowed impact: *"participant/attestation authorization bypass that causes invalid bridge execution"* and *"unauthorized transaction execution … without the required participant authorization."*

---

### Likelihood Explanation

**Likelihood: Medium.**

The attack requires the verifier account to retain full-access keys after the vote — an operational failure the contract cannot detect. The design doc calls this a voter responsibility, and the common deployment path ("trust the published hash") means operators may not independently verify account locking. A single entity who deployed the verifier contract and did not remove their keys can execute the upgrade unilaterally, with no threshold collusion required. The window is permanent: once the verifier account is voted in, the contract never re-checks the code hash.

---

### Recommendation

1. **Store and re-check the code hash at call time.** After the vote crosses threshold, store both `account_id` and `expected_code_hash` in `tee_verifier_account_id`. Before each cross-contract call to `verify_quote`, use NEAR's `env::account_code_hash(account_id)` (if/when available) or a dedicated view-call promise to assert the deployed hash still matches. If it does not match, treat the call as if the verifier is unreachable (let the yield timeout fire) rather than proceeding.

2. **Enforce account locking on-chain.** Require that `vote_tee_verifier_change` only accepts accounts that are provably locked. Until NEAR exposes a synchronous `view_access_key_list` host function, this can be approximated by requiring the candidate to be a NEP-591 global contract (whose `CodeHash` is protocol-level immutable) rather than a plain account.

3. **Fail closed, not open.** If the verifier account's code hash cannot be verified at call time, reject the attestation rather than proceeding. The OrchidLottery fix (`extcodehash` check with fail-open fallback) is the wrong direction for a security gate; fail-closed is the correct default.

4. **Short expiration window.** As the design doc notes, lower `DEFAULT_EXPIRATION_DURATION_SECONDS` so that any attestations accepted by a compromised verifier age out quickly after a rotation.

### Proof of Concept

```

   always returns `VerificationResult::Verified(fake_report)`.
   The attacker does NOT remove their full-access key from the account.

2. The attacker presents the legitimate tee-verifier WASM (matching the real
   code hash H) to MPC participants for audit. Participants vote:
     vote_tee_verifier_change("evil-verifier.near", H)
   Threshold is reached; contract stores:
     tee_verifier_account_id = Some("evil-verifier.near")
   The expected_code_hash H is discarded.

   `DeployContract` on `evil-verifier.near`, replacing the WASM with the
   always-Verified stub. The account's code hash is now H' ≠ H.
   The contract has no record of H and makes no check.

4. Attacker's non-genuine node calls:
     submit_participant_info(Attestation::Dstack(fake_dstack), fake_tls_pk)

   Because the stub can craft a report whose RTMR3 / image-hash fields match
   the current allowlist, all checks pass and the node is inserted into
   stored_attestations.

   respond_verify_foreign_tx() with attacker-chosen payloads, issuing

```markdown
### Requirements on the verifier account

A trusted verifier account must not be replaceable by a malicious stub that returns `VerificationResult::Verified` for any input. Two checkable conditions prevent it:

1. **The right code is deployed.** The hash of the contract code currently deployed at the account matches the expected hash for that verifier release.
2. **That code can never be replaced.** The account has no full-access keys.

The verifier can be a regular contract or a NEP-591 global contract — both satisfy the requirements once locked. Globals have one small audit win — `view_account(account_id).contract` returns the protocol's own `CodeHash` directly, no client-side hashing — and enable cross-shard WASM dedup if other teams adopt the same hash.

### Auditing a candidate verifier

Before any operator votes yes on a candidate `account_id`, they need to confirm that the right code is deployed there and that the code can never be replaced. `mpc-contract` cannot verify either claim itself — a NEAR contract has no way to read another account's code hash or access-key list — so the audit is the voter's responsibility, not the contract's. The same four checks apply whether the candidate is a regular contract or a NEP-591 global:

1. Reproducibly build the verifier source → `H_source`.
2. Fetch `H_deployed`: `view_account(account_id).contract` for a global (returns `CodeHash` directly), or `view_code(account_id)` + local hash for a plain contract.
3. `view_access_key_list(account_id)` → empty.
4. `H_source == H_deployed`.
```

    #[result_serializer(borsh)]
    pub fn verify_quote(
        &self,
        #[serializer(borsh)] quote: QuoteBytes,
        #[serializer(borsh)] collateral: Collateral,
    ) -> VerificationResult {
        let now_seconds = env::block_timestamp_ms() / 1000;
        let quote_bytes: Vec<u8> = quote.into_dcap_type();
        let collateral = collateral.into_dcap_type();
        match dcap_qvl::verify::verify(&quote_bytes, &collateral, now_seconds) {
            Ok(report) => VerificationResult::Verified(report.into_interface_type()),
            Err(err) => {
                VerificationResult::Rejected(VerifierError::DcapVerification(err.to_string()))
            }
        }
    }
