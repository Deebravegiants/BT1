Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Blocks All Withdrawals Until Admin Intervenes — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool.initialize()` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal` evaluates `0 >= 0` as `true` and reverts with `WithdrawalLimitReached` for every caller. Tokens staked via `stake`, `stakeFor`, or `claimAndStake` before an admin calls `setMaxNumberOfWithdrawalsPerUser` are temporarily frozen with no user-accessible exit path.

## Finding Description
`initialize()` sets only `kernelToken`, `rewardsToken`, and the admin role; `maxNumberOfWithdrawalsPerUser` is never written. [1](#0-0) 

`initiateWithdrawal` unconditionally checks:
```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
``` [2](#0-1) 

With `maxNumberOfWithdrawalsPerUser == 0`, any user with zero pending withdrawals satisfies `0 >= 0`, causing an immediate revert. `stake` and `stakeFor` carry no equivalent guard, so staking succeeds while withdrawal is blocked. [3](#0-2) 

`claimAndStake` in `KernelMerkleDistributor` calls `stakeFor` directly, meaning merkle claimants who use the combined claim-and-stake path are also affected. [4](#0-3) 

The only resolution is an admin call to `setMaxNumberOfWithdrawalsPerUser`, which validates that the new value is non-zero and at most `MAX_WITHDRAWALS_PER_USER`. [5](#0-4) 

## Impact Explanation
**Medium — Temporary freezing of funds.** All staked KERNEL tokens are inaccessible via `initiateWithdrawal` until the admin executes `setMaxNumberOfWithdrawalsPerUser`. Because that setter is callable at any time and immediately unblocks withdrawals, the freeze is temporary, not permanent. The original claim of Critical/permanent is overstated: no code path prevents the admin from calling the setter, and the setter itself rejects `0`, so once set it cannot be reset to re-freeze.

## Likelihood Explanation
Moderate. The deployment sequence requires the admin to remember a post-`initialize` configuration call with no on-chain enforcement. Any user who stakes (directly or via `claimAndStake`) during the window between deployment and the admin's setter call is affected. The window duration is entirely dependent on operator diligence.

## Recommendation
Set a safe non-zero default inside `initialize()`:
```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
```
Alternatively, add a guard in `stake` and `stakeFor` that reverts if `maxNumberOfWithdrawalsPerUser == 0`, preventing any staking before the parameter is configured.

    function setMaxNumberOfWithdrawalsPerUser(uint256 _maxNumberOfWithdrawalsPerUser)
        external
        onlyRole(DEFAULT_ADMIN_ROLE)
    {
        if (_maxNumberOfWithdrawalsPerUser == 0 || _maxNumberOfWithdrawalsPerUser > MAX_WITHDRAWALS_PER_USER) {
            revert InvalidMaxNumberOfWithdrawalsPerUser();
        }

        maxNumberOfWithdrawalsPerUser = _maxNumberOfWithdrawalsPerUser;
        emit MaxNumberOfWithdrawalsPerUserUpdated(_maxNumberOfWithdrawalsPerUser);
    }
