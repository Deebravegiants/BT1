Audit Report

## Title
`addLiquidity` Accepts `address(0)` as `owner`, Permanently Locking Deposited Tokens — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address with no zero-address validation. Shares minted to `address(0)` are keyed under `keccak256(abi.encode(address(0), salt, bin))` and real tokens are pulled from the payer via callback. Because `removeLiquidity` enforces `msg.sender == owner`, no EOA or contract can ever satisfy `msg.sender == address(0)`, making the deposited tokens permanently irrecoverable. The only guard against this exists in the periphery, not in the core pool.

## Finding Description
`MetricOmmPool.addLiquidity` (L182–196) accepts `owner` as a caller-supplied parameter with no zero-address check:

```solidity
// metric-core/contracts/MetricOmmPool.sol L182-196
function addLiquidity(
    address owner,   // ← no zero-address check
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) ...
```

Inside `LiquidityLib.addLiquidity`, shares are minted to `posKey = keccak256(abi.encode(owner, salt, bin))` (L72) and `binTotalShares`/`positionBinShares` are updated (L120–121). The callback then pulls real tokens from `msg.sender` into the pool (L147–154). If `owner == address(0)`, shares are minted to `keccak256(abi.encode(address(0), salt, bin))` and real tokens are transferred into the pool.

`removeLiquidity` then enforces `msg.sender == owner` (L206):

```solidity
// metric-core/contracts/MetricOmmPool.sol L206
if (msg.sender != owner) revert NotPositionOwner();
```

No EOA or contract can satisfy `msg.sender == address(0)`. Even if this check were bypassed, `LiquidityLib.removeLiquidity` would `safeTransfer` tokens to `address(0)` (L242–246), burning them.

A grep across all `metric-core/**/*.sol` confirms there is no `owner == address(0)` check, no `InvalidOwner` revert, and no `InvalidPositionOwner` revert anywhere in the core. The only guard is in the periphery:

```solidity
// metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol L247-249
function _validateOwner(address owner) internal pure {
    if (owner == address(0)) revert InvalidPositionOwner();
}
```

This guard is absent from the core pool, creating a gap for any direct integrator.

## Impact Explanation
Any caller that invokes `pool.addLiquidity(address(0), ...)` directly — bypassing the periphery — permanently loses their full token deposit. The tokens are accounted in `binTotals.scaledToken0`/`scaledToken1` as LP-owned liquidity and are not accessible as fee surplus. This is a direct, unrecoverable loss of user principal with no admin recovery path. Severity: High.

## Likelihood Explanation
`addLiquidity` is a public external function explicitly designed for the operator pattern where `msg.sender` (payer) differs from `owner` (position holder). The NatSpec documents this pattern, inviting direct integration by third-party routers and contracts. Any integrator that passes `address(0)` as `owner` — by mistake or by a bug in their own validation — will permanently lose the deposited tokens. The periphery validates this, but the core pool does not, and this gap is invisible to integrators reading only the pool interface.

## Recommendation
Add a zero-address check for `owner` at the top of `MetricOmmPool.addLiquidity`, mirroring the guard already present in the periphery:

```solidity
function addLiquidity(
    address owner,
    ...
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
+   if (owner == address(0)) revert InvalidOwner();
    if (deltas.binIdxs.length == 0) return (0, 0);
    ...
```

Alternatively, enforce it inside `LiquidityLib.addLiquidity` so the invariant holds regardless of call path.

## Proof of Concept
```solidity
// Direct pool call, bypassing periphery
pool.addLiquidity(
    address(0),   // owner = zero address — no revert
    0,            // salt
    deltas,       // valid bin deltas with nonzero shares
    callbackData, // callback pays tokens into pool
    ""
);
// Tokens are now in the pool under posKey = keccak256(abi.encode(address(0), 0, bin))
// removeLiquidity(address(0), ...) requires msg.sender == address(0) → impossible
// Tokens are permanently locked; binTotals.scaledToken0/1 include them as LP liquidity
