### Title
`accept_requests` Flag Simultaneously Blocks New Requests and Prevents Resolution of Already-Pending Yield Requests When TEE Validation Fails - (File: crates/contract/src/lib.rs)

### Summary
When `verify_tee` sets `accept_requests = false` due to a TEE validation failure, the `respond`, `respond_ckd`, and `respond_verify_foreign_tx` functions also gate on this same flag and immediately return `TeeValidationFailed`. This means any sign/CKD/verify_foreign_tx requests that were already queued as pending yields before the flag was cleared cannot be resolved by MPC nodes, and will time out permanently — breaking the request-lifecycle invariant that a validly submitted, in-flight request should always be resolvable.

### Finding Description
`verify_tee` is callable by any participant and sets `accept_requests = false` when TEE validation is partial and kicking out invalid participants would break the threshold relation:

```rust
// crates/contract/src/lib.rs ~L1737
self.accept_requests = false;
return Ok(false);
```

The three node-facing response methods each independently gate on this flag:

```rust
// respond (L579-581)
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}

// respond_ckd (L662-664)
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}

// respond_verify_foreign_tx (L711-713)
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
```

A user's `sign`, `request_app_private_key`, or `verify_foreign_transaction` call creates a NEAR yield-resume promise with a fixed ~200-block timeout window. If `accept_requests` is set to `false` during that window, every MPC node's attempt to call `respond*` is rejected. The yield then fires its timeout callback (`return_signature_and_clean_state_on_success`, etc.), which pops the pending entry and calls `fail_on_timeout` — permanently discarding the request. The user's deposit (1 yoctoNEAR) is consumed and the cross-chain operation fails with no recovery path for that specific yield.

The design intent of `accept_requests` is to gate **new** user requests (enforced in `check_request_preconditions`). Extending the same gate to the node-side `respond*` path conflates two distinct lifecycle phases: admission of new work vs. completion of already-admitted work. This is the direct analog of `pauseAndWithdrawToPurple` combining a pause with an ETH drain — here, the "drain" is the irrecoverable expiry of in-flight yields.

### Impact Explanation
Any sign/CKD/verify_foreign_tx request that is in-flight when `accept_requests` transitions to `false` is permanently unresolvable. MPC nodes hold a valid computed signature but cannot deliver it. The yield times out, the pending-request map entry is removed, and the user's transaction fails. For `verify_foreign_transaction` flows, the user has already broadcast a foreign-chain transaction whose completion depends on receiving the MPC signature; that transaction is permanently stuck. The user must re-submit a new request (and potentially a new foreign-chain transaction) after `accept_requests` is restored to `true`. This breaks the production safety invariant that a validly submitted, in-flight request is always completable by honest nodes.

**Impact: Medium** — request-lifecycle invariant break causing permanent failure of in-flight requests without direct fund theft, matching the allowed Medium scope.

### Likelihood Explanation
`verify_tee` is callable by any current participant (enforced by `voter_or_panic()`), not just a privileged operator. The `accept_requests = false` branch fires when `TeeValidationResult::Partial` is returned and the surviving valid-attestation set is too small to satisfy the threshold relation. This is a realistic production scenario: attestations expire on a fixed schedule, and if enough participants fail to renew before expiry and the cluster is near-minimum size, the condition triggers. A single participant call to `verify_tee` at the right moment is sufficient to freeze all in-flight requests.

**Likelihood: Low** — requires a specific combination of expired attestations and near-threshold cluster size, but is a realistic operational condition, not a theoretical one.

### Recommendation
Remove the `accept_requests` check from `respond`, `respond_ckd`, and `respond_verify_foreign_tx`. The flag should gate only the admission of **new** requests (already enforced in `check_request_preconditions`). Requests that have already been admitted and are awaiting a yield-resume should always be completable by attested participants, regardless of whether the contract is currently accepting new work. Separating admission-time gating from completion-time gating eliminates the invariant break without weakening the TEE safety property.

### Proof of Concept

1. Contract is Running with 3 participants, threshold 2. Two participants have attestations expiring soon.
2. User calls `sign(...)` with 1 yoctoNEAR deposit. A yield-resume promise is created with a ~200-block timeout. The request is stored in `pending_signature_requests`.
3. MPC nodes compute the threshold signature off-chain.
4. Before any node calls `respond`, participant A calls `verify_tee`. The two expiring attestations are now past their deadline. `TeeValidationResult::Partial` fires with only 1 valid participant remaining — below threshold 2. The contract sets `accept_requests = false` and stays Running.
5. Every MPC node attempts `respond(request, signature)`. Each call hits the check at line 579–581 and returns `Err(TeeValidationFailed)`. No yield is resumed.
6. After ~200 blocks the NEAR runtime fires `return_signature_and_clean_state_on_success` with `Err(PromiseError::Failed)`. `pop_oldest_pending_yield` removes the entry; `fail_on_timeout` is called. The user's transaction permanently fails.
7. Even after `accept_requests` is restored to `true` (e.g., participants renew attestations and `verify_tee` is called again), the specific yield handle is gone and cannot be recovered. The user must submit a new `sign` request.

pub(crate) fn pop_oldest_pending_yield<K>(requests: &mut LookupMap<K, Vec<YieldIndex>>, request: &K)
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let Some(queue) = requests.get_mut(request) else {
        return;
    };
    if queue.is_empty() {
        requests.remove(request);
        return;
    }
    queue.remove(0);
    if queue.is_empty() {
        requests.remove(request);
    }
}
