### Title
Missing CKD Response Verification for `AppPublicKey` Variant Allows Single Attested Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_ckd` in the MPC contract performs **no cryptographic verification** of the CKD response when the request uses the `AppPublicKey` (privately-verifiable) variant. A single attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `(big_y, big_c)` pair, and the contract will accept and deliver it to the user. The user then unmasks `big_y` with their secret scalar `a` and derives a key that the attacker fully controls.

### Finding Description

`respond_ckd` branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract runs `ckd_output_check`, a BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` that cryptographically binds the response to the MPC master key and the user's public key pair. [2](#0-1) 

For `AppPublicKey`, the arm is **empty** (`{}`). No pairing check, no signature check, no binding to the MPC master key. Any `(big_y, big_c)` bytes pass.

The `AppPublicKey` variant is the legacy default. It is the variant used in the `ckd-example-cli`, in the gas-regression sandbox test, and in the e2e `ckd_response__passes_cryptographic_verification` test. [3](#0-2) 

After `respond_ckd` succeeds, `resolve_yields_for` immediately resumes the pending yield and delivers the raw `response` bytes to the waiting user callback. [4](#0-3) 

### Impact Explanation

The CKD scheme is designed so that the user submits `pk1 = a·G1` (their ephemeral public key) and the MPC network returns `big_y = a·(derived_key_point)`. The user recovers the derived key by computing `big_y.unmask(a) = derived_key_point`.

An attacker who submits `big_y = t·pk1 = (t·a)·G1` for any attacker-chosen scalar `t` causes the user to compute `unmask(a) = t·G1`. Because `t` is chosen by the attacker, the attacker knows the user's derived key. For example, submitting `big_y = pk1` forces the derived key to `G1` (the generator); submitting `big_y = identity` forces it to the identity element.

This is **unauthorized confidential key derivation output**: a single Byzantine participant below the signing threshold can replace the honest MPC output with attacker-controlled material, giving the attacker full knowledge of the key the user believes is secret. Any asset, wallet, or secret protected by that derived key is compromised.

This matches the allowed impact: *"Critical. Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization."*

### Likelihood Explanation

- The `AppPublicKey` variant is the legacy default and is the variant used in the primary user-facing documentation and CLI example.
- Only **one** attested participant needs to be malicious — well below the signing threshold.
- The attacker simply races to call `respond_ckd` before the honest coordinator submits the real response. Because `resolve_yields_for` resolves on the first call, the first valid `respond_ckd` transaction wins.
- No special knowledge is required beyond the pending `CKDRequest` (which is observable on-chain via `get_pending_ckd_request`).

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` arm. The check requires `pk2` (the G2 component), which is absent in the `AppPublicKey` variant by design. The fix is one of:

1. **Preferred**: Deprecate `AppPublicKey` and require all new requests to use `AppPublicKeyPV`, which carries both `pk1` and `pk2` and enables the on-chain pairing check.
2. **Alternative**: Require the node to include a BLS proof-of-knowledge or a Schnorr proof over `big_y` relative to the MPC master key, verifiable on-chain without `pk2`.

The `AppPublicKeyPV` path already demonstrates the correct pattern: [5](#0-4) 

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk1)` where `pk1 = a·G1`.
2. The pending request is now visible on-chain via `get_pending_ckd_request`.
3. Attacker (any single attested participant) constructs `forged_big_y = pk1` (i.e., `a·G1`) and any `forged_big_c`.
4. Attacker calls `respond_ckd(ckd_request, CKDResponse { big_y: forged_big_y, big_c: forged_big_c })`.
5. The contract executes the empty `AppPublicKey(_) => {}` arm — no check — and calls `resolve_yields_for`, delivering the forged response. [6](#0-5) 

6. User receives `big_y = a·G1` and computes `unmask(a) = G1` — the generator point, a value the attacker predicted exactly.
7. The user's "confidential" derived key is now known to the attacker.

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` confirms the contract accepts `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (invalid BLS points) without any error for the `AppPublicKey` variant, demonstrating the absence of any output check. [7](#0-6)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
