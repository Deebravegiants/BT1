### Title
Missing Cryptographic Output Validation for `AppPublicKey` CKD Responses Allows Single Byzantine Participant to Forge Confidential Key Material - (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` in the MPC contract applies a cryptographic pairing-check to validate the CKD output only when the request uses the `AppPublicKeyPV` (publicly verifiable) variant. When the request uses the `AppPublicKey` (privately verifiable) variant — the simpler, default variant — **no validation of the response is performed at all**. A single attested MPC participant, strictly below the signing threshold, can call `respond_ckd` with an arbitrary forged `CKDResponse` and the contract will accept it, delivering attacker-controlled key material to the user.

---

### Finding Description

In `respond_ckd()`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically proves the response was computed correctly from the MPC master secret key and the user's app public key. [2](#0-1) 

For `AppPublicKey`, the empty arm `{}` means **any** `CKDResponse` — with arbitrary `big_y` and `big_c` byte values — is accepted and forwarded to the user via `resolve_yields_for`. [3](#0-2) 

The only gate before this branch is `assert_caller_is_attested_participant_and_protocol_active()`, which requires the caller to be a single attested participant — not a threshold quorum. [4](#0-3) 

The structural parallel to the external report is exact:
- `_mint()` skips `onERC721Received()` entirely → `AppPublicKey` arm skips `ckd_output_check` entirely.
- `safeTransferFrom()` calls `onERC721Received()` with an incomplete check → `AppPublicKeyPV` arm calls `ckd_output_check` correctly.

---

### Impact Explanation

The CKD protocol delivers `(big_y, big_c)` to the user, who decrypts their secret as `big_c − a·big_y` (where `a` is the user's private scalar). If an attacker submits:

- `big_y = 0` (the G1 identity point, a valid compressed encoding)
- `big_c = s·G1` for an attacker-chosen scalar `s`

then the user computes `s·G1 − a·0 = s·G1`. The attacker knows `s`, so they know the user's derived confidential key. This is a complete key disclosure. The attacker can also deliver a random garbage response, causing the user to derive a wrong key and lose access to any assets or secrets protected by it.

This matches the allowed impact: **Critical — confidential key derivation output without the required participant authorization** (a single Byzantine participant below threshold forges the output without threshold cooperation).

---

### Likelihood Explanation

- The `AppPublicKey` variant is the default, simpler variant (no `--publicly-verifiable` flag). It is the variant used in the basic CKD example CLI and in the majority of documented use cases.
- The attacker only needs to be one attested MPC participant — strictly below the signing threshold — to call `respond_ckd` successfully.
- The attack requires no special knowledge: the attacker simply calls `respond_ckd` with crafted `big_y`/`big_c` values for any pending `AppPublicKey` request they observe on-chain.
- The existing test at line 3424 confirms the contract accepts completely arbitrary byte arrays (`[1u8; 48]`, `[2u8; 48]`) as a valid `AppPublicKey` response with no error. [5](#0-4) 

---

### Recommendation

Apply the same cryptographic output validation to the `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. For `AppPublicKey`, the contract has access to the domain's BLS12-381 public key and the `app_id`. A verification equation analogous to `ckd_output_check` — but using the user's `app_public_key` (G1 point) in place of `app_pk2` — should be enforced before `resolve_yields_for` is called. Alternatively, require all CKD requests to use the `AppPublicKeyPV` variant so the on-chain pairing check is always enforced.

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(user_pk)` and a deposit.
2. Contract stores the pending request and yields.
3. Attacker (a single attested participant) observes the pending request on-chain.
4. Attacker calls `respond_ckd(ckd_request, CKDResponse { big_y: identity_point, big_c: s_times_G1 })` where `s` is attacker-chosen.
5. Contract executes the `AppPublicKey` arm — empty, no check — and calls `resolve_yields_for`, delivering the forged response to the user.
6. User receives `(big_y=0, big_c=s·G1)` and computes their "secret" as `s·G1 − a·0 = s·G1`.
7. Attacker knows `s` and therefore knows the user's derived confidential key.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
