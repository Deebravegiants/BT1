### Title
Missing Value-Conservation Check in Sub-Transaction UTxO Rule Allows Arbitrary ADA Creation — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The Dijkstra era introduces nested (sub-)transactions. The top-level `UTXO` rule enforces value conservation (`validateValueNotConservedUTxO`) only for the top-level transaction body. The `SUBUTXO` rule that processes each sub-transaction applies UTxO changes via `updateUTxOStateNoFees` with **no value-conservation check**. An unprivileged transaction author can therefore include sub-transactions whose outputs exceed their inputs, minting arbitrary ADA from thin air in a single valid-looking Dijkstra-era transaction.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — missing value-conservation invariant in a deferred-validation context (direct analog of the Mango HealthRegion access-control bypass).

**Top-level conservation check (only covers the top-level body):**

In `dijkstraUtxoTransition`, the sole conservation check is:

```haskell
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [1](#0-0) 

Here `txBody` is the **top-level** transaction body. Sub-transaction bodies are never passed to this check.

**Sub-transaction UTxO rule — no conservation check:**

`dijkstraSubUtxoTransition` validates inputs, output sizes, network IDs, and validity intervals, then unconditionally applies UTxO changes:

```haskell
newState <-
  Shelley.updateUTxOStateNoFees
    pp utxoState txBody certState
    (utxosGovState utxoState)
    (tellEvent . TotalDeposits (hashAnnotated txBody))
    (\a b -> tellEvent $ TxUTxODiff a b)
``` [2](#0-1) 

There is no call to `validateValueNotConservedUTxO` anywhere in `dijkstraSubUtxoTransition`.

**`updateUTxOStateNoFees` applies changes without conservation:**

```haskell
updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  let ...
      !utxoAdd = txouts txBody   -- outputs inserted into UTxO
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
``` [3](#0-2) 

The function removes inputs and inserts outputs with no check that `Σ outputs ≤ Σ inputs + deposit_refunds`. The comment at line 602 explicitly states this is "used for sub-transactions where fees are not applicable" — but the omission of the conservation invariant is not intentional. [4](#0-3) 

**Sub-transaction input checks do not prevent value inflation:**

The two input-existence checks in `dijkstraSubUtxoTransition` only verify that inputs exist in the UTxO; they say nothing about the relationship between input value and output value:

```haskell
runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxoState) inputs
``` [5](#0-4) 

**`validateBatchWithdrawals` does not close the gap:**

The only batch-level check in the top-level rule is `validateBatchWithdrawals`, which guards against over-withdrawal from account balances. It does not check UTxO value conservation across sub-transactions. [6](#0-5) 

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker can include one or more sub-transactions that spend UTxO inputs worth `X` lovelace and produce outputs worth `X + N` lovelace. Because `updateUTxOStateNoFees` blindly applies the delta, the UTxO set gains `N` lovelace that was never in the system. The top-level conservation check passes because it only accounts for the top-level body. Every honest node applies the same (broken) rule, so the inflated UTxO becomes the canonical ledger state — a permanent, consensus-level ADA creation event.

---

### Likelihood Explanation

The Dijkstra era is described as experimental, but it is present in the production codebase and its `SUBLEDGER`/`SUBLEDGERS`/`SUBUTXO` rules are fully wired into the STS hierarchy. Any transaction author who can submit a Dijkstra-era transaction can trigger this path. No privileged key, governance majority, or external dependency is required. The attack requires only the ability to construct a well-formed Dijkstra transaction with a sub-transaction whose outputs exceed its inputs — a straightforward operation for anyone with access to the transaction-building API.

---

### Recommendation

Add a value-conservation check inside `dijkstraSubUtxoTransition`, mirroring the check already present in the top-level rule:

```haskell
-- After input-existence checks, before updateUTxOStateNoFees:
runTest $ Shelley.validateValueNotConservedUTxO pp (utxosUtxo utxoState) certState txBody
```

Alternatively, add a batch-level conservation check in `dijkstraUtxoTransition` that accounts for all sub-transaction inputs and outputs alongside the top-level body, similar to how `validateBatchWithdrawals` aggregates withdrawals across the batch. [7](#0-6)

### Proof of Concept

Assume the attacker controls UTxO entry `U1` (100 ADA) and `U2` (10 ADA).

**Step 1 — Construct top-level transaction `T`:**
- Inputs: `{U1}` (100 ADA)
- Outputs: `{O_top: 99 ADA to attacker}`
- Fee: 1 ADA
- Sub-transactions: `[S1]`

Top-level conservation check: `consumed(U1=100) == produced(O_top=99 + fee=1)` ✓

**Step 2 — Sub-transaction `S1`:**
- Inputs: `{U2}` (10 ADA) — must exist in `originalUtxo` ✓
- Outputs: `{O_sub: 1,000,000 ADA to attacker}`
- No fee field (sub-transactions are fee-free)

`dijkstraSubUtxoTransition` checks:
- `validateBadInputsUTxO originalUtxo {U2}` ✓ (U2 exists)
- `validateBadInputsUTxO (utxosUtxo utxoState) {U2}` ✓ (U2 not yet spent)
- No `validateValueNotConservedUTxO` call

`updateUTxOStateNoFees` removes `U2` (10 ADA) and inserts `O_sub` (1,000,000 ADA). No error.

**Result:** The ledger accepts the transaction. The attacker's UTxO now contains `O_top` (99 ADA) + `O_sub` (1,000,000 ADA). Net gain: **999,990 ADA created from thin air**, accepted by all honest nodes as a valid state transition.

updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  let UTxOState {utxosUtxo, utxosDeposited, utxosFees, utxosDonation} = utxos
      UTxO utxo = utxosUtxo
      !utxoAdd = txouts txBody -- These will be inserted into the UTxO
      {- utxoDel  = txins txb ◁ utxo -}
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      {- newUTxO = (txins txb ⋪ utxo) ∪ outs txb -}
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
      deletedUTxO = UTxO utxoDel
      totalRefunds = certsTotalRefundsTxBody pp certState txBody
      totalDeposits = certsTotalDepositsTxBody pp certState txBody
      depositChange = totalDeposits <-> totalRefunds
  depositChangeEvent depositChange
  txUtxODiffEvent deletedUTxO utxoAdd
  pure $!
    UTxOState
      { utxosUtxo = UTxO newUTxO
      , utxosDeposited = utxosDeposited <> depositChange
      , utxosFees = utxosFees
      , utxosGovState = govState
      , utxosInstantStake =
          deleteInstantStake deletedUTxO (addInstantStake utxoAdd (utxos ^. instantStakeL))
      , utxosDonation = utxosDonation
      }
