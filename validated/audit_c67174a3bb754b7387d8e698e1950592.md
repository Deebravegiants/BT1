### Title
Stale Votes from Removed Participants Counted in `MeasurementVotes::count_votes`, Enabling Threshold Bypass for OS Measurement Approval — (File: `crates/contract/src/tee/measurements.rs`)

---

### Summary

`MeasurementVotes::count_votes` tallies all stored votes without filtering by the current participant set. After a resharing that removes participants, their pre-resharing votes persist in `vote_by_account` and are counted toward the threshold for `vote_add_os_measurement`. A single post-resharing vote from a current participant can combine with stale votes from removed participants to satisfy the threshold, allowing a malicious OS measurement to be whitelisted with fewer current-participant approvals than the protocol requires.

---

### Finding Description

In `crates/contract/src/tee/measurements.rs`, `MeasurementVotes::count_votes` counts every entry in `vote_by_account` that matches the action, with no filter for whether the voter is still a current participant: [1](#0-0) 

```rust
fn count_votes(&self, action: &MeasurementVoteAction) -> u64 {
    u64::try_from(
        self.vote_by_account
            .values()
            .filter(|a| *a == action)
            .count(),
    )
    .expect("participant count should not overflow u64")
}
```

This is inconsistent with every other vote-counting function in the codebase:

- `ThresholdParametersVotes::n_votes` explicitly cross-filters against the live `participants` set: [2](#0-1) 
- `ProviderVotes::vote` uses `count_for(|(p, c)| *c == chain && participants.is_participant_given_participant_id(&p.get()))`: [3](#0-2) 
- `TeeVerifierVotes::vote` uses `count_for(|p| participants.is_participant_given_participant_id(&p.get()))`: [4](#0-3) 

The `vote_add_os_measurement` entry point calls `self.tee_state.vote_measurement(action, &participant)`, which internally calls `MeasurementVotes::vote` → `count_votes`. The returned count is compared directly against `self.threshold()?.value()`: [5](#0-4) 

```rust
let votes = self.tee_state.vote_measurement(action, &participant);
if votes >= self.threshold()?.value() {
    let added = self.tee_state.add_measurement(measurement);
    log!("OS measurement add result: {}", added);
}
```

The post-resharing cleanup path (`clean_non_participant_votes`) is tested only for `CodeHashesVotes` (`tee_state.votes`), not for `tee_state.measurement_votes`: [6](#0-5) 

This leaves stale measurement votes unremoved after participant-set changes, while the analogous `get_remaining_votes` helper that *does* filter by participants is never called in the hot path: [7](#0-6) 

---

### Impact Explanation

An OS measurement added to the allowed list determines which TEE environments are accepted as valid by the contract's attestation verification. If a malicious OS measurement is whitelisted, nodes running that environment can submit passing attestations and be voted in as participants. With enough such participants, the attacker can compromise threshold signing, issue unauthorized signatures, or expose key shares. This is a **participant/attestation authorization bypass** that breaks the safety invariant requiring threshold votes from *current* participants — matching the "participant/attestation authorization bypass" category in the allowed impact scope.

---

### Likelihood Explanation

The attack requires: (1) an attacker controlling below-threshold participants who pre-vote for a malicious measurement before being removed via resharing; (2) at least one post-resharing participant voting for the same measurement (either controlled by the attacker or deceived into voting for a measurement that appears legitimate). Resharing events are a normal part of the protocol lifecycle (participant rotation, threshold changes), making the precondition realistic. The attacker does not need to control threshold-or-above participants at any single point in time — the stale votes bridge the gap across the resharing boundary.

---

### Recommendation

Pass the current `Participants` into `count_votes` and filter votes to only those from current participants, mirroring the pattern used by `ThresholdParametersVotes::n_votes`:

```rust
fn count_votes(&self, action: &MeasurementVoteAction, participants: &Participants) -> u64 {
    u64::try_from(
        self.vote_by_account
            .iter()
            .filter(|(participant_id, a)| {
                *a == action
                    && participants
                        .is_participant_given_participant_id(&participant_id.get())
            })
            .count(),
    )
    .expect("participant count should not overflow u64")
}
```

Additionally, ensure the post-resharing cleanup (`clean_non_participant_votes` or equivalent) also purges stale entries from `measurement_votes`, consistent with how `CodeHashesVotes` is already cleaned.

---

### Proof of Concept

1. Deploy contract with N=5 participants {P1, P2, P3, P4, P5}, governance threshold=3.
2. Attacker controls P1 and P2. Both call `vote_add_os_measurement(malicious_measurement)`. `count_votes` returns 2 — below threshold=3. Measurement not added.
3. Honest participants vote to reshard, removing P1 and P2. New set: {P3, P4, P5}, threshold=2.
4. Post-resharing cleanup runs but does **not** purge `measurement_votes`; P1's and P2's entries remain in `vote_by_account`.
5. Attacker controls P3 (or deceives P3 into voting for the same measurement). P3 calls `vote_add_os_measurement(malicious_measurement)`.
6. `MeasurementVotes::vote` inserts P3's entry, then calls `count_votes`. The filter `|a| *a == action` matches P1, P2, and P3 — returning **3**.
7. Current threshold = 2. Condition `3 >= 2` is true.
8. `add_measurement(malicious_measurement)` executes — the malicious OS measurement is whitelisted with only **1 current-participant vote** instead of the required 2.
9. Nodes running the malicious TEE environment now pass attestation checks and can be voted in as participants, enabling downstream threshold-signature compromise.
