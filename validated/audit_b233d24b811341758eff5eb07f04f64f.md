### Title
Propose-Update Deposits Are Permanently Locked in the MPC Contract With No Withdrawal Path - (File: crates/contract/src/update.rs, crates/contract/src/lib.rs)

### Summary
`propose_update` requires participants to attach a NEAR deposit proportional to the storage cost of the update entry. When `do_update` executes after threshold votes, it clears **all** pending update entries and votes without refunding any of the deposits paid by proposers. No withdrawal function exists for these accumulated deposits, causing them to be permanently frozen in the contract.

### Finding Description
In `propose_update` (`lib.rs`), the contract collects a deposit from the proposer equal to `ProposedUpdates::required_deposit(&update)`, which is computed as `env::storage_byte_cost() * bytes_used` — a non-trivial amount for a contract binary upload. Only the excess above the required amount is refunded: [1](#0-0) 

The required deposit is retained by the contract and stored alongside the `UpdateEntry`: [2](#0-1) [3](#0-2) 

When `do_update` is called after threshold votes are reached, it removes the executed entry and then **clears all remaining entries and all votes** with no refund logic: [4](#0-3) 

Specifically, lines 198–200 wipe every competing proposal's entry: [5](#0-4) 

There is no function in the contract to withdraw or reclaim update proposal deposits. The only `withdraw`-named function is `withdraw_tee_verifier_vote`, which removes a governance vote — it transfers no NEAR: [6](#0-5) 

`remove_update_vote` similarly only removes the vote record, not the deposit: [7](#0-6) 

### Impact Explanation
Every completed contract upgrade cycle permanently locks the deposits of all proposers — both the proposer of the executed update and any proposers of competing updates that were cleared. The deposits are held by the MPC contract account with no code path to recover them. This breaks the balance/accounting invariant that participants who pay storage deposits should be able to reclaim them when the storage is freed. **Impact: Medium** — balance and accounting invariant violation that permanently freezes participant funds without requiring any privileged access or collusion.

### Likelihood Explanation
Contract upgrades are a routine operational event in the MPC network lifecycle. Every upgrade cycle that involves more than one competing proposal (or even a single proposal that is executed and cleared) results in locked deposits. The deposit size scales with the contract binary size, making the locked amount non-trivial (a full WASM binary can be hundreds of kilobytes, costing several NEAR at mainnet storage prices). **Likelihood: Medium** — occurs on every upgrade cycle.

### Recommendation
1. Track the proposer's account ID and the exact deposit amount in `UpdateEntry`.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit_paid)` for each.
3. Alternatively, expose a `cancel_update(id: UpdateId)` method that removes the entry and refunds the proposer, callable by the proposer or by governance.

### Proof of Concept
1. Participant A calls `propose_update` with a 500 KB contract binary. Required deposit ≈ `500_000 * storage_byte_cost` NEAR. Excess is refunded; the required amount stays in the contract.
2. Participant B calls `propose_update` with a different binary, paying a similar deposit.
3. Threshold participants vote for A's update via `vote_update`.
4. `do_update` is triggered: it executes A's update, then calls `self.entries.clear()` and `self.vote_by_participant.clear()`.
5. Both A's and B's deposits remain in the contract. Neither participant has any code path to recover them. The funds are permanently frozen. [4](#0-3) [8](#0-7)

    /// Propose update to either code or config, but not both of them at the same time.
    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
