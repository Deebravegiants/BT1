Audit Report

## Title
stETH Negative Rebase Between `unlockQueue` and `completeWithdrawal` Temporarily Freezes Tail-Queue User Funds - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`unlockQueue` transfers a fixed quantity of stETH from `LRTUnstakingVault` into `LRTWithdrawalManager` and records each user's `expectedAssetAmount` at that moment. Because stETH is a rebasing token, a subsequent negative rebase (validator slashing) silently reduces the contract's stETH balance below the sum of all pending `expectedAssetAmount` values. Users at the tail of the queue whose cumulative payout exceeds the remaining balance receive a revert on `completeWithdrawal`, and because rsETH was already burned at `unlockQueue` time, they have no self-service recovery path.

## Finding Description
`_unlockWithdrawalRequests` sets each request's final payout and accumulates the total:

```solidity
request.expectedAssetAmount = payoutAmount;   // L804
assetAmountToUnlock += payoutAmount;           // L807
```

`unlockQueue` then redeems exactly that total from the vault, so the withdrawal manager holds precisely `assetAmountUnlocked` stETH and the sum of all unlocked `expectedAssetAmount` values equals `assetAmountUnlocked`:

```solidity
unstakingVault.redeem(asset, assetAmountUnlocked);   // L307
```

rsETH is burned at this same call:

```solidity
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);   // L305
```

`_processWithdrawalCompletion` later transfers the stored amount unconditionally for non-ETH assets:

```solidity
_transferAsset(asset, user, request.expectedAssetAmount);   // L734
```

The Aave balance top-up logic at L720–732 is gated on `asset == LRTConstants.ETH_TOKEN`, so it never executes for stETH. `_transferAsset` for a non-ETH asset calls `IERC20(asset).safeTransfer(to, amount)` (L881), which reverts if the contract balance is insufficient.

stETH is explicitly a supported withdrawal asset: `initialize2` seeds `unlockedWithdrawalsCount[stETH]` at L118. The developers themselves acknowledge the slashing edge case in the `initiateWithdrawal` NatSpec (L149): *"There is an edge case were the user withdraws last underlying asset and that asset gets slashed."* — yet no mitigation is implemented in the completion path.

After a negative rebase the accounting invariant `balance(stETH) == Σ expectedAssetAmount` is broken. FIFO ordering means early callers succeed; the last callers whose cumulative `expectedAssetAmount` exceeds the remaining balance revert. Their rsETH is already gone (burned at L305) and no user-callable function exists to recover the shortfall.

## Impact Explanation
**Medium — Temporary freezing of funds.** Affected users cannot complete their withdrawal and cannot re-initiate because their rsETH has been burned. Their stETH remains locked in the contract until an operator manually tops up the balance or a subsequent positive rebase restores it. This matches the allowed impact class "Temporary freezing of funds."

## Likelihood Explanation
**Low-Medium.** Ethereum validator slashing events are infrequent but historically documented (e.g., Lido slashing incidents). The vulnerability window spans the entire period between `unlockQueue` and the last `completeWithdrawal` in a batch — up to 8+ days given the default `withdrawalDelayBlocks` of `8 days / 12 seconds` (L94). A single slashing event during a large batch unlock is sufficient to freeze the tail of the queue. No attacker action is required; the stETH rebase mechanism is the trigger.

## Recommendation
- In `_processWithdrawalCompletion`, before calling `_transferAsset` for non-ETH rebasing assets, read the live contract balance and transfer `min(request.expectedAssetAmount, contractBalance)`, emitting a shortfall event so operators can monitor and replenish.
- Alternatively, pro-rate all pending `expectedAssetAmount` values at `completeWithdrawal` time if the live balance has decreased below the recorded sum.
- At minimum, add an operator-callable `replenishAsset(address asset, uint256 amount)` function so that a slashing shortfall can be covered without requiring a full positive rebase.

## Proof of Concept
1. Operator calls `unlockQueue(stETH, ...)` when the vault holds 100 stETH. Ten requests are unlocked, each with `expectedAssetAmount = 10 stETH`. `unstakingVault.redeem` transfers 100 stETH to the withdrawal manager (L307). rsETH for all ten users is burned (L305).
2. A Lido slashing event causes a 6% negative stETH rebase. The withdrawal manager's balance drops to ~94 stETH. No storage variable is updated.
3. Users 1–9 call `completeWithdrawal(stETH, ...)`. Each receives 10 stETH. After 9 transfers, ~4 stETH remain.
4. User 10 calls `completeWithdrawal(stETH, ...)`. `_transferAsset` attempts `safeTransfer(user10, 10 stETH)` (L881). The contract holds only ~4 stETH → `safeTransfer` reverts.
5. User 10's rsETH was burned at step 1. Their ~4 stETH is locked in the contract with no self-service recovery path.
