### Title
ETH Deposit Limit Check Excludes New Deposit Amount, Allowing Limit Bypass — (File: `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` applies an asymmetric limit check: for LST tokens it correctly adds the incoming `amount` to the running total before comparing against the cap, but for ETH it omits `amount` entirely. Any unprivileged depositor can therefore push total ETH deposits above the configured cap whenever the current total sits exactly at the limit.

---

### Finding Description

The vulnerability class in the reference report is **missing input validation on data received from an untrusted source**: an unvalidated field (`sz`) is used directly in a critical operation (`fd_memcpy`), bypassing the intended safety boundary. The direct Solidity analog is a user-supplied value (`depositAmount`) that is silently excluded from the bounds check that is supposed to gate the operation.

In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(
    address asset,
    uint256 amount
) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        // ← amount is NOT added; only the pre-deposit total is checked
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
    }
    // ← amount IS added for every LST
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

For LST tokens the guard is `totalAssetDeposits + amount > limit`, which correctly blocks a deposit that would breach the cap. For ETH the guard is `totalAssetDeposits > limit`, which only fires when the cap has **already** been exceeded. When `totalAssetDeposits == limit` the check returns `false`, the deposit is accepted, and the total rises to `limit + depositAmount`.

This check is the sole enforcement point called from `_beforeDeposit`, which is invoked by both `depositETH` and `depositAsset`: [2](#0-1) 

The entry path is fully permissionless — `depositETH` is `external payable` with no role restriction: [3](#0-2) 

---

### Impact Explanation

The deposit limit is the protocol's primary risk-management cap on ETH exposure. Bypassing it allows any depositor to push total ETH holdings above the configured ceiling, violating the invariant the protocol promises to enforce. This constitutes **"contract fails to deliver promised returns"** (Low severity per the allowed scope). No funds are directly stolen, but the risk-management boundary is silently broken.

---

### Likelihood Explanation

The condition is triggered whenever `getTotalAssetDeposits(ETH_TOKEN) == depositLimitByAsset(ETH_TOKEN)`. This is a realistic steady-state: the limit is actively managed, deposits approach the cap regularly, and any depositor who observes the on-chain state can exploit the window. No special privilege is required.

---

### Recommendation

Add `amount` to the ETH branch, mirroring the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

---

### Proof of Concept

1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Organic deposits bring `getTotalAssetDeposits(ETH_TOKEN)` to exactly `1000 ether`.
3. Attacker calls `depositETH{value: 100 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the attacker; total ETH in protocol becomes `1100 ether`, 10 % above the configured cap.
6. Subsequent legitimate depositors are blocked (`1100 > 1000 = true`), but the cap has already been breached.
