### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Allows Unprivileged Attacker to Exhaust Fan-Out Queue and Block Legitimate Foreign-Tx Verification - (File: crates/contract/src/lib.rs, crates/contract/src/pending_requests.rs)

### Summary

The `verify_foreign_transaction` endpoint stores pending requests under a caller-agnostic key (keyed only on `domain_id`, `payload_version`, and the foreign-chain RPC parameters — never on the submitting account). All callers share a single 128-slot fan-out queue per request key. An unprivileged attacker can submit 128 identical `verify_foreign_transaction` calls for any target foreign transaction, filling the queue to `MAX_PENDING_REQUEST_FAN_OUT`, and causing every subsequent legitimate submission for that transaction to be rejected with `PendingRequestQueueFull`. The attacker can sustain the block by re-filling the queue after each ~200-block timeout window.

### Finding Description

`MpcContract` stores three pending-request maps:

```
pending_signature_requests: LookupMap<SignatureRequest, Vec<YieldIndex>>
pending_ckd_requests:       LookupMap<CKDRequest, Vec<YieldIndex>>
pending_verify_foreign_tx_requests: LookupMap<VerifyForeignTransactionRequest, Vec<YieldIndex>>
``` [1](#0-0) 

For `sign`, the map key is `SignatureRequest`, which is constructed as:

```rust
pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
    let tweak = crate::kdf::derive_tweak(predecessor_id, path);
    SignatureRequest { domain_id: domain, tweak, payload }
}
``` [2](#0-1) 

The `tweak` encodes the caller's `predecessor_id`, so each distinct caller gets a distinct map key and their own independent queue. An attacker cannot fill another user's queue.

For `verify_foreign_transaction`, the map key is `VerifyForeignTransactionRequest`, which is constructed from `args_into_verify_foreign_tx_request(request)` — using only `domain_id`, `payload_version`, and the foreign-chain RPC parameters. The caller's account ID is **never included**. The contract's own test explicitly documents this:

> "both yields are queued under the single (caller-agnostic) request key" [3](#0-2) 

The fan-out queue is capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
// ...
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
}
``` [4](#0-3) 

Because the queue is shared across all callers for the same foreign-tx parameters, an attacker who submits 128 calls for a specific `(chain, tx_id, domain_id)` tuple fills the queue entirely. Any subsequent legitimate user submitting the same foreign-tx verification receives `PendingRequestQueueFull` and their transaction is rejected.

The `verify_foreign_transaction` entry point only requires a 1-yoctonear deposit and standard gas: [5](#0-4) 

Queued yields time out after `REQUEST_EXPIRATION_BLOCKS = 200` blocks (~4 minutes on NEAR). The attacker can re-fill the queue immediately after each timeout cycle, sustaining the block indefinitely at a cost of only gas fees per cycle.

### Impact Explanation

A targeted attacker can permanently prevent any specific foreign-chain transaction from being verified through the MPC contract. In the context of cross-chain bridge flows, this means:

- A user's bridge withdrawal (e.g., a Bitcoin or Ethereum transaction that must be verified before NEAR-side funds are released) can be blocked indefinitely.
- The user's yield-resume promise times out, their transaction fails, and they must retry — only to be blocked again.
- This breaks the request-lifecycle safety invariant: a valid, on-chain foreign transaction that satisfies all policy checks cannot be processed because the queue for its key is permanently saturated by an adversary.

This maps to the allowed **Medium** impact: *"request-lifecycle or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

### Likelihood Explanation

- **Entry path**: Any NEAR account. No special role, no key material, no threshold collusion required.
- **Cost**: 128 × 1 yoctonear + gas per ~4-minute blocking window. Economically trivial.
- **Targeting**: The attacker only needs to know the target `tx_id` and chain — both are observable on-chain or in the mempool before the victim's call lands.
- **Sustainability**: The attacker can front-run the victim's retry attempts indefinitely.

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, mirroring the `SignatureRequest` design. This gives each caller their own independent queue slot, so an attacker filling their own 128 slots cannot affect any other user's queue.

Alternatively, enforce a per-account submission rate limit or a per-account cap on concurrent pending foreign-tx requests, so a single account cannot consume more than a bounded fraction of the shared queue.

### Proof of Concept

1. Alice submits `verify_foreign_transaction` for Bitcoin tx `[0xAA; 32]` on domain 0.
2. Attacker (Bob) observes Alice's pending request on-chain.
3. Bob submits the identical `verify_foreign_transaction` request 127 more times (from one or many accounts — the key is caller-agnostic).
4. The queue for `(domain=0, chain=Bitcoin, tx_id=[0xAA;32])` now holds 128 entries.
5. Alice's retry after her first yield times out is rejected: `PendingRequestQueueFull { limit: 128 }`.
6. Bob re-fills the queue every ~200 blocks. Alice's foreign-tx verification is permanently blocked.

The existing unit test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` already demonstrates that alice and bob share the same queue slot — the attacker simply needs to repeat this 128 times: [6](#0-5)

    #[payable]
    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
