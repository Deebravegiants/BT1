Audit Report

## Title
Pool Admin Can Permanently Lock Protocol Fees via Reverting `adminFeeDestination` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`setPoolAdminFeeDestination` accepts any non-zero address without verifying it can receive ERC-20 tokens. Because `collectFees` in `MetricOmmPool.sol` transfers to `adminFeeDestination_` before `FACTORY`, a reverting destination permanently DoS's every factory path that collects fees, locking all accrued protocol and admin fees inside the pool with no recovery mechanism available to the factory owner.

## Finding Description
`setPoolAdminFeeDestination` performs only a zero-address check: [1](#0-0) 

`collectFees` is `onlyFactory` and transfers to `adminFeeDestination_` first, before the `FACTORY` leg: [2](#0-1) 

`transferToken0/1` use `safeTransfer`, which propagates any revert from the destination: [3](#0-2) 

Because `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled` are zeroed only after the transfers complete, a revert leaves them intact but permanently unclaimable. Every factory entry-point that collects fees passes `poolAdminFeeDestination[pool]` unchanged with no override:

- `collectPoolFees` (permissionless): [4](#0-3) 
- `setPoolProtocolFee` (`onlyOwner`): [5](#0-4) 
- `setPoolAdminFees` (`onlyPoolAdmin`): [6](#0-5) 

Since `collectFees` is `onlyFactory`, the factory owner has no direct bypass path to the pool — the factory always forwards `poolAdminFeeDestination[pool]` and there is no overload accepting an alternative destination.

## Impact Explanation
All accrued spread fees (pool balance surplus) and notional fees (`notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`) are permanently locked inside the pool. The factory owner additionally loses the ability to adjust protocol fee rates because `setPoolProtocolFee` calls `collectFees` as its first step and will always revert. This is a direct, irrecoverable loss of protocol fee revenue and a broken core governance function, matching the allowed impact gate for "Critical/High direct loss of protocol fees" and "broken core pool functionality causing loss of funds."

## Likelihood Explanation
The pool admin is a distinct, semi-trusted role. Any party that deploys a pool controls the initial `admin` address and can call `setPoolAdminFeeDestination` at any time after deployment — including after significant fees have accumulated — with a single permissioned transaction and no preconditions beyond holding the pool admin role. The only on-chain signal is the `PoolAdminFeeDestinationUpdated` event. This exceeds the pool admin's trust boundary: the pool admin should control only their own fee destination, not the protocol's ability to collect its share.

## Recommendation
Separate the admin and protocol fee transfers so a failure in the admin leg cannot block the protocol leg. Options include: (1) wrapping the admin transfer in a `try/catch` and emitting an event on failure, leaving admin fees claimable via a pull pattern; (2) splitting `collectFees` into two independent functions — one for the protocol share and one for the admin share; or (3) at minimum, validating in `setPoolAdminFeeDestination` that the new address can receive both pool tokens (e.g., via a small dry-run transfer or explicit interface check).

## Proof of Concept
1. Pool admin deploys a `RevertOnReceive` contract whose ERC-20 receipt hook reverts unconditionally.
2. Pool admin calls `factory.setPoolAdminFeeDestination(pool, address(revertOnReceive))`.
3. Swaps accumulate spread surplus and increment `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`.
4. Anyone calls `factory.collectPoolFees(pool)` → `pool.collectFees(...)` → `transferToken0(revertOnReceive, ...)` → **revert**. Protocol receives nothing.
5. Factory owner calls `factory.setPoolProtocolFee(pool, newFee, 0)` → same revert path → **cannot update fee rates**.
6. Fees remain locked in the pool indefinitely with no recovery mechanism available to the protocol.
