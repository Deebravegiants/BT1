Audit Report

## Title
Unbounded `assetsCommitted` Consumption Allows Temporary DOS of Withdrawal Initiation - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.initiateWithdrawal` increments a shared `assetsCommitted[asset]` counter with no per-user cap. A sufficiently capitalised rsETH holder can saturate this counter up to the total protocol TVL for a given asset, causing `getAvailableAssetAmount` to return zero and reverting every subsequent `initiateWithdrawal` call from other users. The attacker recovers capital after each operator `unlockQueue` cycle and can immediately re-saturate, sustaining the DOS.

## Finding Description
`initiateWithdrawal` (L170–173) gates new requests on `getAvailableAssetAmount` and then increments the shared counter:

```solidity
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` (L599–603) computes:

```solidity
uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
```

`assetsCommitted[asset]` is a single global counter with no per-user ceiling, no rate limit, and no maximum open-request count per user. An attacker calls `initiateWithdrawal` in a loop — each call committing `minRsEthAmountToWithdraw[asset]` worth of the target asset — until `assetsCommitted[asset] >= totalAssets`. At that point `getAvailableAssetAmount` returns 0 and every honest user's call reverts with `ExceedAmountToWithdraw`.

`assetsCommitted` is only decremented inside `unlockQueue` (L802): `assetsCommitted[asset] -= request.expectedAssetAmount;`. After the operator unlocks the queue, the attacker calls `completeWithdrawal` to recover the underlying asset, re-acquires rsETH, and re-saturates the counter. The existing `nonReentrant` and `whenNotPaused` guards do not prevent this multi-transaction pattern.

By contrast, `KernelDepositPool.initiateWithdrawal` (L323) enforces `if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();` — an analogous guard that is absent from `LRTWithdrawalManager`.

## Impact Explanation
When `assetsCommitted[asset] >= totalAssets`, `initiateWithdrawal` — the sole standard entry point for the withdrawal lifecycle — is completely blocked for all users. rsETH holders cannot begin redeeming their tokens for the underlying LST or ETH. This constitutes **temporary freezing of funds** (Medium severity per the allowed impact scope): users' rsETH is economically stranded for as long as the attacker sustains the saturated state across operator unlock cycles.

## Likelihood Explanation
The attacker must hold rsETH whose redemption value equals the remaining `getAvailableAssetAmount` for the targeted asset. In the worst case (no prior pending withdrawals) this equals the full protocol TVL for that asset — a high but not impossible capital bar. The net cost per cycle is gas plus the opportunity cost of locking funds for the withdrawal delay (default ~8 days, L94). The attack is most feasible at low TVL (early deployment or a lower-TVL supported LST). No privileged access is required; the entry path is fully open to any rsETH holder.

## Recommendation
Introduce a per-user cap on open withdrawal requests in `LRTWithdrawalManager`, mirroring the `maxNumberOfWithdrawalsPerUser` pattern already present in `KernelDepositPool` (L107–108, L323). Alternatively, track each user's committed amount in a separate mapping and reject requests that would push any single user's committed share above a configurable fraction of `totalAssets`. Either approach bounds the maximum `assetsCommitted` attributable to a single address and prevents saturation by a single actor.

## Proof of Concept
1. Attacker acquires rsETH via `LRTDepositPool.depositETH` sufficient to cover `getAvailableAssetAmount(ETH_TOKEN)`.
2. Attacker calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, minRsEthAmountToWithdraw[ETH_TOKEN], "")` in a loop until `assetsCommitted[ETH_TOKEN] == totalAssets`.
3. Any honest user calling `initiateWithdrawal` now reverts with `ExceedAmountToWithdraw` (L170).
4. Operator calls `unlockQueue` → `assetsCommitted` decreases for each unlocked request (L802).
5. Attacker calls `completeWithdrawal` to recover ETH, re-deposits to obtain rsETH, and repeats from step 2.
6. The withdrawal initiation path remains blocked for honest users across cycles.
