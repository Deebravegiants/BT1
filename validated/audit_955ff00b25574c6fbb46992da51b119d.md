### Title
Fee Transfer L2 Gas Excluded from `l2_gas_used` Causes EIP-1559 Fee Market Underestimation — (File: `crates/apollo_batcher/src/block_builder.rs`)

---

### Summary

The `l2_gas_used` metric fed into the EIP-1559 (SNIP-35) fee market adjustment is computed exclusively from `receipt.gas.l2_gas`, which is derived from `execution_summary_without_fee_transfer`. The L2 Sierra gas consumed by the fee transfer call itself is never added back to the receipt or to the block-level gas total. This systematically understates actual L2 gas consumption, biasing the next block's gas price downward and leaving fee-transfer computation gas uncharged to users.

---

### Finding Description

**Step 1 — Receipt is frozen before fee transfer executes.**

In `execute_raw`, `run_or_revert` returns a `TransactionReceipt` whose `gas: GasVector` is computed solely from validate + execute call infos:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  L919-L930
let ValidateExecuteCallInfo {
    ...,
    final_cost: TransactionReceipt { fee: final_fee, da_gas: final_da_gas,
                                     resources: final_resources, gas: total_gas },
} = self.run_or_revert(state, &mut GasCounter::new(initial_gas), tx_context.clone())?;
```

`run_or_revert` calls `TransactionReceipt::from_account_tx` with the parameter explicitly named `execution_summary_without_fee_transfer`:

```rust
// crates/blockifier/src/fee/receipt.rs  L169-L192

    ...
    execution_summary_without_fee_transfer: ExecutionSummary,
    ...
) -> Self { ... }
``` [1](#0-0) 

**Step 2 — Fee transfer executes after the receipt is frozen.**

```rust

The fee transfer is a live Cairo 1 (Sierra) ERC-20 `transfer` call. Test data confirms it consumes real Sierra gas (e.g., `gas_consumed: 11690` in `central_transaction_execution_info_reverted.json`). That gas is stored in `fee_transfer_call_info.execution.gas_consumed` but is never folded back into `receipt.gas.l2_gas`. [3](#0-2) 

**Step 3 — Block-level `l2_gas_used` sums only the frozen receipts.**

```rust

    let mut res = GasAmount::ZERO;
    for (execution_info, _) in self.execution_infos_and_signatures.values() {
        res = res.checked_add(execution_info.receipt.gas.l2_gas)
                 .expect("Total L2 gas overflow.");
    }
    res
}

**Step 4 — Understated `l2_gas_used` drives the EIP-1559 price adjustment.**

`l2_gas_used` is passed directly into `calculate_next_l2_gas_price_for_fin` (the EIP-1559 base-fee update) and broadcast in `ProposalFin.fin_payload.l2_gas_info.l2_gas_used`:

```rust
// crates/apollo_consensus_orchestrator/src/build_proposal.rs  L326-L339
let next_l2_gas_price = calculate_next_l2_gas_price_for_fin(
    args.l2_gas_price,
    args.build_param.height,
    info.l2_gas_used,          // ← understated
    ...
);
let fin_payload = ProposalFinPayload {
    l2_gas_info: L2GasInfo {
        next_l2_gas_price_fri: next_l2_gas_price,
        l2_gas_used: info.l2_gas_used,   // ← understated, broadcast to validators
    },
    ...
};
``` [5](#0-4) 

`calculate_next_base_gas_price` compares `gas_used` to `gas_target`; a systematically low `gas_used` biases the price downward every block:

```rust
// crates/apollo_consensus_orchestrator/src/fee_market/mod.rs  L118-L129

### Impact Explanation

| Dimension | Effect |
|---|---|
| **User fee undercharge** | Every account transaction avoids paying for ~11,690 L2 gas (fee transfer Sierra gas). At a 1 Gwei L2 gas price this is ~11.69 × 10⁻⁶ STRK per tx; at 1000 TPS it accumulates to ~1 STRK/day of uncharged gas. |
| **EIP-1559 fee market distortion** | For a block with N transactions, `l2_gas_used` is understated by N × ~11,690 gas. At 1000 tx/block and a 5 B gas target, this is a ~0.23% systematic downward bias per block, compounding over time. |
| **Block header `l2_gas_consumed` incorrect** | The on-chain field and the value broadcast in `ProposalFin` both carry the understated figure, making the authoritative record of block gas consumption wrong. |

This matches the "Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact" criterion.

---

### Likelihood Explanation

The condition is triggered by every account transaction where `charge_fee = true` and the fee token is a Cairo 1 ERC-20 contract (the standard case on mainnet). No special crafting is required; the undercharge is automatic and universal.

---

### Recommendation

After `handle_fee` returns the `fee_transfer_call_info`, extract its Sierra gas consumption and add it to the receipt's gas vector before assembling `TransactionExecutionInfo`:

```rust
// After handle_fee returns:
if let Some(ref ft_info) = fee_transfer_call_info {
    total_gas.l2_gas = total_gas.l2_gas
        .checked_add(GasAmount(ft_info.execution.gas_consumed))
        .expect("L2 gas overflow adding fee transfer gas");
}
```

Alternatively, define a versioned-constants entry for the expected fee-transfer Sierra gas cost (analogous to the existing OS VM resource constants) and add it to the receipt at receipt-construction time, keeping the receipt computation self-contained.

---

### Proof of Concept

1. Execute any Cairo 1 invoke transaction with `charge_fee = true`.
2. Inspect `tx_execution_info.fee_transfer_call_info.execution.gas_consumed` — it will be non-zero (e.g., 11,690).
3. Inspect `tx_execution_info.receipt.gas.l2_gas` — it will not include the fee-transfer gas.
4. Observe that `BlockTransactionExecutionData::l2_gas_used()` sums only `receipt.gas.l2_gas`, so the fee-transfer gas is absent from the block total.
5. Trace `l2_gas_used` into `calculate_next_l2_gas_price_for_fin`: the next block's price is computed against a gas-used figure that is lower than actual consumption by `N × fee_transfer_gas`, causing the EIP-1559 adjustment to undershoot.
