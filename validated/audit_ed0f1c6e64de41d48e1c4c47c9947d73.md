### Title
`respond_ckd` Accepts Arbitrary CKD Response for `AppPublicKey` Variant Without Cryptographic Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` contract method performs no cryptographic validity check on the submitted `CKDResponse` when the pending request uses the `CKDAppPublicKey::AppPublicKey` variant. A single attested participant — strictly below the signing threshold — can call `respond_ckd` with an attacker-fabricated derived-key response, and the contract will accept it and deliver it to the waiting user, bypassing the threshold authorization requirement for confidential key derivation.

---

### Finding Description

The vulnerability class from the reference report is **missing existence/validity check before processing a request**. In `Vault721.tokenURI`, the missing check was a token-existence guard. The direct analog here is a missing **cryptographic output validity check** in `respond_ckd` for the `AppPublicKey` variant.

In `respond_ckd`, the contract branches on the request's `app_public_key` field:

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies the derived-key response against the app public key and the MPC root key. For `AppPublicKey`, the arm is a no-op — any `CKDResponse` is unconditionally forwarded to `resolve_yields_for`, which resumes the user's yield promise with whatever bytes were supplied. [2](#0-1) 

Contrast this with `respond` (for ECDSA/EdDSA signatures), which **always** verifies the submitted signature against the cryptographically derived public key before touching the pending-request map:

```rust
// crates/contract/src/lib.rs  lines 596-608
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,   // derived key — not root
)
.is_ok()
``` [3](#0-2) 

That verification is what prevents a single malicious attested participant from injecting a wrong signature. No equivalent guard exists for `AppPublicKey` CKD responses.

The `resolve_yields_for` helper itself does not validate content — it simply resumes every queued yield with the raw bytes it receives: [4](#0-3) 

---

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) can:

1. Observe a pending `CKDRequest` with `AppPublicKey` in `pending_ckd_requests`.
2. Construct a `CKDResponse` containing an attacker-chosen derived key.
3. Call `respond_ckd` directly — the contract checks only that the caller is an attested participant and the protocol is running, then skips the output check and delivers the fabricated key to the user.

The user receives a derived key that was **not** computed by the threshold MPC protocol. The attacker knows the key they injected and can use it to decrypt any data the user subsequently encrypts with it, or to impersonate the user on the target application.

This is "unauthorized confidential key derivation output without the required participant authorization" — **Critical** per the allowed impact scope.

---

### Likelihood Explanation

- The attacker must be an attested participant, which is a realistic Byzantine-node scenario explicitly in scope.
- The `CKDRequest` key is deterministic from the user's request parameters (domain, app public key, derivation path, predecessor), so the attacker can reconstruct it from on-chain data.
- The attacker simply races the legitimate leader: whichever call to `respond_ckd` arrives first drains the yield queue; the second call returns `RequestNotFound`.
- No collusion above threshold is required; a single compromised node suffices.

---

### Recommendation

Add a cryptographic output check for the `AppPublicKey` variant in `respond_ckd`, analogous to the check already present for `AppPublicKeyPV`. If the `AppPublicKey` variant's output is not verifiable on-chain (e.g., because the necessary cryptographic material is absent), the contract should either:

- Require callers to use `AppPublicKeyPV` exclusively (so the check is always enforceable), or
- Enforce a threshold-vote mechanism (multiple attested participants must agree on the same response) before resolving the yield.

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(app_pk) => {
+       if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
+           env::panic_str("CKD output check failed");
+       }
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

```
1. User calls request_app_private_key({ app_public_key: AppPublicKey(legit_pk), domain_id, derivation_path })
   → contract stores CKDRequest in pending_ckd_requests and suspends the user's promise.

2. Attacker (single attested participant) reconstructs the CKDRequest key from on-chain state.

3. Attacker calls respond_ckd(request, CKDResponse { derived_key: attacker_controlled_key }).
   → contract checks: is_running_or_resharing ✓, accept_requests ✓, is_attested_participant ✓
   → AppPublicKey arm: no-op (no ckd_output_check)
   → resolve_yields_for resumes user's promise with attacker_controlled_key.

4. User's promise resolves with the attacker-controlled derived key.
   Legitimate leader's subsequent respond_ckd call returns RequestNotFound.

5. User uses the attacker-controlled key, believing it was threshold-computed.
   Attacker can decrypt or forge anything protected by that key.
```

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
