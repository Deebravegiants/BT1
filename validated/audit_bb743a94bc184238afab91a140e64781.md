Audit Report

## Title
Lack of Slippage Protection in `instantWithdrawal` Allows Users to Receive Fewer Assets Than Expected - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`LRTWithdrawalManager.instantWithdrawal` burns a user-specified amount of rsETH and computes the asset payout entirely from the live oracle price at execution time, with no `minAssetAmountExpected` parameter. A user who previews the exchange rate off-chain via `getExpectedAssetAmount` and then submits the transaction has no on-chain guarantee that the oracle price will not have moved unfavorably by the time the transaction is mined, causing them to receive fewer assets than anticipated for the rsETH they burned. The deposit path in `LRTDepositPool._beforeDeposit` already enforces an equivalent `minRSETHAmountExpected` guard, making this an explicit asymmetry in the protocol's slippage protection.

## Finding Description

`instantWithdrawal` at [1](#0-0)  accepts only `asset`, `rsETHUnstaked`, and `referralId`. The payout is computed at [2](#0-1)  via `getExpectedAssetAmount`, which resolves to:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [3](#0-2) 

The rsETH is burned unconditionally at [4](#0-3)  before any check on the received asset amount. The only guards present check for a minimum rsETH input amount and sufficient vault balance — neither protects against oracle price movement between query time and execution time. [5](#0-4) 

By contrast, `LRTDepositPool._beforeDeposit` explicitly enforces a `minRSETHAmountExpected` slippage guard and reverts with `MinimumAmountToReceiveNotMet` if the minted rsETH falls below it: [6](#0-5) 

The same missing guard applies to `initiateWithdrawal`, which also computes `expectedAssetAmount` from the live oracle and locks it in without any user-specified minimum. [7](#0-6) 

## Impact Explanation

**Low — Contract fails to deliver promised returns, but does not lose value.**

A user who queries `getExpectedAssetAmount` off-chain and submits `instantWithdrawal` has their rsETH burned atomically. If the oracle price for rsETH or the target asset moves between query time and execution time, the user receives fewer assets than anticipated. The rsETH is already burned and cannot be recovered. The protocol itself does not lose funds; only the user's expected return is silently reduced.

## Likelihood Explanation

Oracle prices for rsETH and LSTs (stETH, ETHx) update on a regular heartbeat (typically every hour or on a 0.5% deviation threshold). Any transaction that sits in the mempool across an oracle update, or that is submitted during a period of price movement, will silently deliver a worse-than-expected exchange rate. No privileged actor is required; this is triggered by any unprivileged user calling `instantWithdrawal` under normal market conditions.

## Recommendation

Add a `minAssetAmountExpected` parameter to `instantWithdrawal` (and `initiateWithdrawal`) and revert if the computed payout falls below it, mirroring the existing guard in `LRTDepositPool._beforeDeposit`:

```solidity
function instantWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
    uint256 minAssetAmountExpected,   // <-- add this
    string calldata referralId
) external ... {
    uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
    if (assetAmountUnlocked < minAssetAmountExpected) revert SlippageExceeded();
    IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
    ...
}
```

## Proof of Concept

1. Alice queries `getExpectedAssetAmount(stETH, 1 ether)` off-chain and sees she will receive `1.05 stETH`.
2. Alice submits `instantWithdrawal(stETH, 1 ether, "ref")`.
3. Before Alice's transaction is mined, a Chainlink oracle heartbeat update raises the stETH/ETH price, reducing the rsETH-to-stETH ratio.
4. `getExpectedAssetAmount` now returns `1.02 stETH` at execution time.
5. Alice's 1 rsETH is burned at [4](#0-3)  and she receives only `1.02 stETH` (minus fee) — 0.03 stETH less than expected — with no on-chain protection to revert the transaction.

**Foundry fork test plan:** Deploy a fork, call `instantWithdrawal` with a previewed amount, warp time or mock the oracle to simulate a heartbeat update mid-block, and assert that `userAmount` is less than the off-chain preview with no revert.
