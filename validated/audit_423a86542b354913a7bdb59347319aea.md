### Title
DamgardEtAl `ReconstructionThreshold` / `MaxMalicious` Semantic Mismatch Allows Effective Signing Threshold to Diverge from Contract-Stored Value — (File: `crates/node/src/providers/robust_ecdsa.rs`)

---

### Summary

The on-chain contract stores a `ReconstructionThreshold` (`t`) for every domain, including DamgardEtAl (Robust ECDSA) domains. The DamgardEtAl cryptographic protocol, however, is parameterized by `MaxMalicious` (`f`), where the required participant count is exactly `n = 2f + 1`. The node bridges this gap with a `translate_threshold()` function that the codebase's own design documentation explicitly labels a "hack" and a "mismatch." This is a direct structural analog to the external report's "Tree Level incompatible" class: one component (the contract) assumes one semantic for a structural parameter, while the other component (the DamgardEtAl signing provider) is built for a different semantic.

---

### Finding Description

**Component 1 — Contract (`crates/contract/src/primitives/domain.rs`)**

The contract validates DamgardEtAl domains with the constraint `2t − 1 ≤ n`: [1](#0-0) 

This means the contract stores a `ReconstructionThreshold` `t` and enforces the honest-majority bound, but it does **not** require `2t − 1 = n` (equality). The contract therefore permits, for example, `t = 2` with `n = 5` (since `2·2−1 = 3 ≤ 5`), or `t = 3` with `n = 7` (since `2·3−1 = 5 ≤ 7`).

**Component 2 — DamgardEtAl Presign Protocol (`crates/threshold-signatures/src/ecdsa/robust_ecdsa/presign.rs`)**

The cryptographic presign protocol enforces that the participant count must be **exactly** `2 · max_malicious + 1`: [2](#0-1) 

This is a hard equality check, not a bound. The protocol is structurally built around `n = 2f + 1`, not around a `t`-of-`n` reconstruction threshold.

**Component 3 — The Bridge (`crates/node/src/providers/robust_ecdsa.rs`)**

The node's DamgardEtAl provider contains a `translate_threshold()` function that converts the contract's `ReconstructionThreshold` into the `MaxMalicious` value the protocol expects. The project's own design document explicitly acknowledges this as a semantic mismatch and a workaround: [3](#0-2) 

The two semantics are not equivalent:

| Concept | Contract stores | Protocol uses |
|---|---|---|
| Parameter | `ReconstructionThreshold t` | `MaxMalicious f` |
| Participant count needed | `t` shares to reconstruct | Exactly `2f + 1` to sign |
| Honest-majority bound | `2t − 1 ≤ n` (inequality) | `n = 2f + 1` (equality) |

The correct translation from `t` to `f` is `f = t − 1`, giving `2(t−1)+1 = 2t−1` required participants. However, if `translate_threshold()` uses integer division — `f = (t−1)/2` — then for any even `t`, the actual cryptographic threshold is strictly lower than the contract-stored value. For example:

- Contract stores `t = 4`, `n = 7` (valid: `2·4−1 = 7 ≤ 7`)
- Node computes `f = (4−1)/2 = 1` (integer division)
- Protocol requires exactly `2·1+1 = 3` participants
- **Effective signing threshold: 3, not 4**

Even if the formula is `f = t − 1` (the correct mapping), the contract's inequality `2t−1 ≤ n` permits configurations where the actual number of participants used in signing (`2t−1`) is less than `n`, meaning the contract's governance threshold (which is `≥ t`) does not bound the actual signing participant count.

---

### Impact Explanation

The contract's `ReconstructionThreshold` for DamgardEtAl domains does not accurately represent the actual cryptographic threshold enforced by the signing protocol. Depending on the translation formula in `translate_threshold()`:

1. **If integer division is used (`f = (t−1)/2`)**: For even values of `t`, the actual signing threshold is strictly lower than the contract-stored value. A Byzantine participant set of size `f+1` (below the contract's intended threshold) can prevent honest completion, and a colluding set of size `f+1` can produce a valid signature with fewer participants than the contract's governance rules require. This constitutes a threshold bypass.

2. **If the correct formula is used (`f = t−1`)**: The contract's inequality `2t−1 ≤ n` still permits configurations where the signing participant count (`2t−1`) is less than `n`, meaning the contract's stored `t` does not uniquely determine the signing set size. The contract's governance threshold (which bounds `t`) does not bound the actual signing participant count.

In either case, the contract's stored `ReconstructionThreshold` for DamgardEtAl is semantically incompatible with the protocol's `MaxMalicious` parameter, breaking the invariant that the contract's threshold governs the minimum number of participants required to produce a valid signature.

---

### Likelihood Explanation

This is a production code path. DamgardEtAl (Robust ECDSA) is listed as an active protocol in the domain registry: [4](#0-3) 

The mismatch is triggered whenever a DamgardEtAl domain is active and a signing request is processed. The design document confirms the mismatch is known and present in the current codebase. Any unprivileged user who submits a `sign` request routed to a DamgardEtAl domain exercises this code path.

---

### Recommendation

1. Define a single canonical mapping between `ReconstructionThreshold` and `MaxMalicious` and enforce it in both the contract's validation logic and the node's `translate_threshold()` function.
2. The contract's `validate_domain_threshold` for DamgardEtAl should enforce `2t − 1 = n` (equality, not inequality) if the intent is that all `n` participants must sign, or should store `MaxMalicious` directly as the per-domain parameter for DamgardEtAl domains.
3. Remove `translate_threshold()` in favor of reading the correct parameter directly from the contract's domain configuration, as proposed in the domain-separation design: [5](#0-4)

### Proof of Concept

1. Deploy the MPC contract with a DamgardEtAl domain: `t = 4`, `n = 7`. The contract accepts this (since `2·4−1 = 7 ≤ 7`).
2. The node's `translate_threshold()` in `crates/node/src/providers/robust_ecdsa.rs` converts `t = 4` to `MaxMalicious`.
3. If the formula uses integer division: `f = (4−1)/2 = 1`, requiring only `2·1+1 = 3` participants.
4. The presign protocol at `crates/threshold-signatures/src/ecdsa/robust_ecdsa/presign.rs` line 196 enforces `participants.len() == 3`, not 7.
