### Title
Storage Deposits for Superseded Update Proposals Are Permanently Locked — (`File: crates/contract/src/update.rs`)

---

### Summary

When `do_update` executes a winning governance proposal, it unconditionally clears **all** other pending proposals and their storage entries without refunding the NEAR deposits paid by their proposers. Because `UpdateEntry` does not record the proposer's account ID or the deposited amount, there is no mechanism — sweep, admin withdrawal, or otherwise — to recover those funds. They are permanently locked in the MPC contract.

---

### Finding Description

`propose_update` is a `#[payable]` function that requires a deposit proportional to the size of the proposed update (contract binary or config blob):

```rust
// crates/contract/src/lib.rs  ~L1308-1331

let required = ProposedUpdates::required_deposit(&update);
if attached < required { return Err(...); }
let id = self.proposed_updates.propose(update);
// Refund only the *excess* above required; the required portion stays in the contract.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
``` [1](#0-0) 

The required deposit is computed as:

    env::storage_byte_cost().saturating_mul(bytes_used)

The stored `UpdateEntry` contains only the update payload and `bytes_used`; **the proposer's account ID and the deposited amount are never recorded**:

```rust

    pub(super) update: Update,
    pub(super) bytes_used: u128,

When `vote_update` reaches threshold it calls `do_update`, which clears every pending proposal with no refund logic:

```rust
// crates/contract/src/update.rs  ~L195-200
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();           // ← all other proposals deleted
    self.vote_by_participant.clear();
    // ← no refund to any proposer
``` [4](#0-3) 

There is no sweep function, admin withdrawal, or any other path to recover these deposits from the contract.

---

### Impact Explanation

Every participant who proposed a competing (non-winning) update loses their storage deposit permanently. For a contract binary update, the deposit scales with the WASM size. NEAR contracts can be up to 4 MB; at the protocol's storage byte cost (~10 yoctoNEAR/byte), a 1 MB binary requires roughly **10 NEAR** in deposit. With multiple participants each proposing their own binary, the total locked amount grows proportionally. The funds are irrecoverable: the `UpdateEntry` does not record who paid, so even a future contract upgrade cannot reconstruct the refund list.

This breaks the production accounting invariant that storage deposits are returned when the storage they cover is freed.

---

### Likelihood Explanation

This is a routine governance scenario, not an edge case. The README explicitly states that all pending proposals are cleared on every executed update:

> "When an update receives sufficient votes and is executed … all pending update proposals and votes are cleared as they are no longer valid after the contract migration." [5](#0-4) 

In any real governance round where more than one participant independently proposes an update (a normal occurrence when participants disagree on the binary or config), the non-winning proposers lose their deposits. No privileged access, collusion, or adversarial action is required — ordinary governance use triggers the loss.

---

### Recommendation

1. **Record the proposer and deposit in `UpdateEntry`**: Add `proposer: AccountId` and `deposit: NearToken` fields so the information needed for a refund is available at cleanup time.
2. **Refund on clear**: In `do_update`, iterate over the remaining entries before calling `self.entries.clear()` and issue a `Promise::new(entry.proposer).transfer(entry.deposit)` for each.
3. **Alternatively**, store a separate `BTreeMap<UpdateId, (AccountId, NearToken)>` alongside `entries` and drain it with refunds in `do_update`.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, attaching ~5 NEAR deposit. The deposit is accepted and stored in the contract balance.
2. Participant B calls `propose_update` with a different 500 KB binary, attaching ~5 NEAR deposit. Same result.
3. Participants vote for B's proposal until threshold is reached; `vote_update` calls `do_update`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — deleting A's entry — and `self.vote_by_participant.clear()`. No refund is issued to A.
5. A's ~5 NEAR is now permanently locked in the contract. `UpdateEntry` contains no record of A's account ID or deposit amount, so no future code path can reconstruct the refund. [6](#0-5) [7](#0-6)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}

Participants can propose and vote on contract updates (code or configuration changes). When an update receives sufficient votes and is executed (via the `vote_update` endpoint which calls `do_update` internally), all pending update proposals and votes are cleared as they are no longer be valid after the contract migration.
