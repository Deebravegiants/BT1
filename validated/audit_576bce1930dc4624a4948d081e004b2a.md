### Title
Tautological Height Check in `is_proposal_init_valid` — `ProposalInitValidation.height` Seeded from Proposer-Supplied `init.height` - (File: `crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

### Summary

In both the direct and queued validation paths, `ProposalInitValidation.height` is populated from the proposer-supplied `init.height` rather than from the validator's own authoritative `self.current_height`. The downstream check `init_proposed.height == proposal_init_validation.height` in `is_proposal_init_valid` therefore always evaluates to `true`, making the height guard a dead-code tautology. The only real height enforcement is an `assert_eq!` that panics instead of gracefully rejecting, creating a DoS vector.

### Finding Description

**Direct validation path** (`validate_proposal`, lines 880–900):

```rust
// sequencer_consensus_context.rs:880-900
let proposal_init_validation = ProposalInitValidation {
    height: init.height,   // ← seeded from the proposer-supplied init
    ...
};
self.validate_current_round_proposal(init, proposal_init_validation, ...).await;
```

**Queued proposal path** (`set_height_and_round`, lines 1178–1198):

```rust
// sequencer_consensus_context.rs:1178-1198
let proposal_init_validation = ProposalInitValidation {
    height: init.height,   // ← same pattern: seeded from the queued init
    ...
};
```

Both paths then call `is_proposal_init_valid` with `init_proposed = &args.init` (the same `init` object) and `proposal_init_validation` (whose `.height` was copied from that same `init`):

```rust
// validate_proposal.rs:312-314
if !(init_proposed.height == proposal_init_validation.height   // always true
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
```

`init_proposed.height` and `proposal_init_validation.height` are the same value, so the height sub-expression is unconditionally `true`. The only actual height enforcement is the `assert_eq!` at line 866:

```rust
// sequencer_consensus_context.rs:866
assert_eq!(Some(init.height), self.current_height);
```

This panics on mismatch instead of returning a graceful `InvalidProposalInit` error.

### Impact Explanation

The tautological height check means `is_proposal_init_valid` never rejects a proposal on the basis of a wrong height. The sole guard is a panic. Any code path that reaches `validate_proposal` with a mismatched height (e.g., a future refactor that removes or bypasses the assert, or a consensus-layer bug that passes the wrong height) would silently accept the proposal. A proposal accepted at the wrong height would produce a block committed at an incorrect block number, corrupting the chain state, block hash, and all downstream commitments.

Additionally, the panic-based guard is a DoS vector: any peer that can cause the consensus layer to invoke `validate_proposal` with a wrong height will crash the validator node.

### Likelihood Explanation

Currently the consensus layer (Tendermint) enforces the correct height before calling `validate_proposal`, so the assert rarely fires in practice. However, the tautology is a latent defect: the intended defense-in-depth check inside `is_proposal_init_valid` provides zero protection, and the panic-based guard is fragile. The pattern is directly analogous to the reported CVI bug.

### Recommendation

Populate `ProposalInitValidation.height` from the validator's own authoritative state (`self.current_height.unwrap()`) rather than from the proposer-supplied `init.height`:

```rust
// Both construction sites in sequencer_consensus_context.rs
let proposal_init_validation = ProposalInitValidation {
    height: self.current_height.expect("height must be set before validating"),
    // ...
};
```

Replace the `assert_eq!` at line 866 with a graceful rejection that returns an error instead of panicking, so that a wrong-height proposal is rejected cleanly rather than crashing the node.

### Proof of Concept

1. `validate_proposal` is called with `init.height = H`.
2. `ProposalInitValidation { height: init.height, ... }` → `proposal_init_validation.height = H`.
3. `is_proposal_init_valid(&proposal_init_validation, &init, ...)` is called.
4. Check: `init.height (H) == proposal_init_validation.height (H)` → always `true`.
5. The height guard is never exercised; any height value passes.
