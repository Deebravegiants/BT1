### Title
`updateFutureVRFKeyHash` Uses `Map.delete` Instead of Decrement, Allowing Duplicate VRF Registration After v11 ŌĆö (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Pool.hs`)

---

### Summary

`updateFutureVRFKeyHash` in the re-registration branch of `poolTransition` uses `Map.insert ŌĆ” (knownNonZeroBounded @1)` (overwrite) and `Map.delete` (full removal) to manage `psVRFKeyHashes`. Both operations are reference-countŌĆōunaware. When a VRF key hash has a count > 1 (possible for pools that shared a VRF before the v11 hard fork), these operations corrupt the count, ultimately removing the entry entirely. A third pool can then register with the "deleted" VRF, bypassing `VRFKeyHashAlreadyRegistered`.

---

### Finding Description

`psVRFKeyHashes` is a `Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)` ŌĆö a reference-counted multiset. At the v11 hard fork, `populateVRFKeyHashes` correctly initialises it by counting every occurrence across `psStakePools` and `psFutureStakePoolParams`, so two pools sharing `vrfX` before v11 produce `psVRFKeyHashes[vrfX] = 2`. [1](#0-0) 

In the re-registration branch of `poolTransition`, `updateFutureVRFKeyHash` is defined as:

```
Nothing -> Map.insert sppVrf (knownNonZeroBounded @1)          -- overwrites count
Just futureStakePoolParams
  | futureStakePoolParams ^. sppVrfL /= sppVrf ->
      Map.insert sppVrf (knownNonZeroBounded @1)
        . Map.delete (futureStakePoolParams ^. sppVrfL)         -- full removal
  | otherwise -> id
``` [2](#0-1) 

**Bug 1 ŌĆö `Nothing` branch overwrites the count.** When pool A re-registers with its own current VRF (`sppVrf == stakePoolState ^. spsVrfL`), the guard at lines 280ŌĆō282 passes unconditionally. The `Nothing` branch then calls `Map.insert vrfX 1`, silently overwriting `psVRFKeyHashes[vrfX]` from 2 to 1. [3](#0-2) 

**Bug 2 ŌĆö `Map.delete` removes the entire entry.** When pool A subsequently re-registers with a fresh `vrfY` in the same epoch, `psFutureStakePoolParams[A]` now holds `{vrf: vrfX}`. The `Just` branch fires: `Map.insert vrfY 1 . Map.delete vrfX`. `Map.delete vrfX` removes the entry entirely, even though pool B still holds `vrfX`. [4](#0-3) 

By contrast, `poolReapTransition` correctly uses `Map.update (mapNonZero (\n -> n - 1))` ŌĆö decrementing and removing only when the count reaches zero. [5](#0-4) 

---

### Impact Explanation

After the two-step re-registration, `vrfX` is absent from `psVRFKeyHashes` even though pool B still uses it. A third pool C can now submit a `RegPool` certificate with `vrfX`; the new-registration guard `Map.notMember sppVrf psVRFKeyHashes` evaluates to `True` and the certificate is accepted. [6](#0-5) 

After the epoch boundary, pool B and pool C both hold `vrfX` as their active VRF key, with `psVRFKeyHashes[vrfX] = 1` ŌĆö a permanent under-count. The `VRFKeyHashAlreadyRegistered` invariant is violated: the ledger now contains two live pools with identical VRF keys, which the Conway uniqueness rule was specifically designed to prevent. [7](#0-6) 

---

### Likelihood Explanation

The prerequisite is that two pools share a VRF key before the v11 hard fork, which was explicitly permitted in earlier eras. `populateVRFKeyHashes` was written to handle exactly this case (counts > 1). On any chain that upgraded through v11 with such pools present, the precondition is already satisfied. The attacker needs only to control one of those pools and submit two sequential re-registration certificates within the same epoch ŌĆö both of which are valid, standard pool operations.

---

### Recommendation

Replace both `Map.insert ŌĆ” (knownNonZeroBounded @1)` (overwrite) and `Map.delete` in `updateFutureVRFKeyHash` with reference-countŌĆōaware operations, mirroring the pattern already used in `poolReapTransition`:

- **Removal**: use `Map.update (mapNonZero (\n -> n - 1))` instead of `Map.delete`.
- **Insertion of a new VRF**: use `Map.insertWith` with the same saturating-increment combinator used in `populateVRFKeyHashes` / `addVRFKeyHashOccurrence`, rather than a hard-coded `knownNonZeroBounded @1`.
- **Same-VRF re-registration** (`Nothing` branch, `sppVrf == stakePoolState ^. spsVrfL`): no change to `psVRFKeyHashes` is needed; the branch should be a no-op for the map.

populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
