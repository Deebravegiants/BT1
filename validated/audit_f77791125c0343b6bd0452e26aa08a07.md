Audit Report

## Title
Permissionless `sendFunds()` + `updateRSETHPrice()` Enables Flash-Deposit Yield Extraction via `instantWithdrawal` — (`contracts/FeeReceiver.sol`, `contracts/LRTOracle.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary

`FeeReceiver.sendFunds()` carries no access control, allowing any caller to push accumulated MEV/EL rewards into `LRTDepositPool` and immediately inflate the TVL. Because `LRTOracle.updateRSETHPrice()` is also permissionless, an attacker can atomically deposit ETH at the stale rsETH price, trigger both permissionless calls to raise the price, and then call `instantWithdrawal()` which prices the exit at the newly inflated rsETH price — extracting yield earned by existing stakers. The ETH is drained from `LRTUnstakingVault` while the reward ETH remains in `LRTDepositPool`.

## Finding Description

**Root cause:** Two permissionless state-changing functions can be chained with a deposit and instant withdrawal in a single transaction to extract yield.

**Step 1 — Permissionless reward injection**

`FeeReceiver.sendFunds()` has no access control modifier:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`receiveFromRewardReceiver()` in `LRTDepositPool` is a plain `external payable` function with no access control, so the ETH lands directly in the deposit pool's balance. `getETHDistributionData()` counts `ethLyingInDepositPool = address(this).balance`, so the reward ETH is immediately reflected in `getTotalAssetDeposits(ETH_TOKEN)`.

**Step 2 — Permissionless price update**

`LRTOracle.updateRSETHPrice()` is gated only by `whenNotPaused`:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Inside `_updateRsETHPrice()`, the new price is:

```solidity
// contracts/LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

`totalETHInProtocol` now includes the reward ETH just pushed by `sendFunds()`. The protocol fee is deducted and minted as rsETH to the treasury, but the remaining yield is fully reflected in the stored `rsETHPrice`.

**Step 3 — `instantWithdrawal` uses the live price**

`getExpectedAssetAmount` reads the freshly updated `rsETHPrice`:

```solidity
// contracts/LRTWithdrawalManager.sol L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

`instantWithdrawal` calls this and redeems from `LRTUnstakingVault`:

```solidity
// contracts/LRTWithdrawalManager.sol L228-235
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
unstakingVault.redeem(asset, assetAmountUnlocked);
```

**The math:**

Let T = TVL before rewards, R = reward amount, S = rsETH supply, P₀ = T/S (old price), f = protocolFeeInBPS/10000, w = instantWithdrawalFee/10000.

- Attacker deposits D ETH at P₀, receiving `rsETH = D·S/T`
- After `sendFunds()` + `updateRSETHPrice()`: `P₁ ≈ (T + R·(1−f)) / S`
- `assetAmountUnlocked = (D·S/T) · P₁ = D · (T + R·(1−f)) / T`
- `userAmount = assetAmountUnlocked · (1 − w)`

**Net gain = userAmount − D > 0** whenever `R·(1−f)/T > w`. For typical MEV rewards and low instant-withdrawal fees, this condition is satisfied.

**Why existing guards fail:**

The `pricePercentageLimit` guard at `LRTOracle.sol` L256-265 only triggers when `newRsETHPrice > highestRsethPrice` AND `pricePercentageLimit > 0`. The default value of `pricePercentageLimit` is 0 (no limit set at initialization), bypassing the guard entirely. Even when set, typical MEV rewards are small relative to TVL and stay within any reasonable daily threshold.

The `queuedWithdrawalsBuffer` in `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal()` (L235-237) is a partial mitigation but does not prevent the attack if there is ETH above the buffer, which is the normal operating condition.

## Impact Explanation

This is a **High** severity finding: **Theft of unclaimed yield**. MEV/EL rewards accumulated in `FeeReceiver` and intended for existing rsETH holders are partially captured by a flash depositor who contributes nothing to the protocol's staking activity. The ETH is drained from `LRTUnstakingVault`, reducing the pool available for legitimate queued withdrawals. The attacker's profit comes directly at the expense of existing stakers' yield.

## Likelihood Explanation

- Both `sendFunds()` and `updateRSETHPrice()` are permissionless public functions callable by any EOA or contract.
- The attack is executable atomically in a single transaction with no special privileges.
- `pricePercentageLimit` defaults to 0 (no protection), and even when set, small MEV rewards relative to TVL stay within any reasonable threshold.
- `instantWithdrawalFee` reduces but does not eliminate profit unless it exceeds the per-unit yield fraction.
- ETH must be present in `LRTUnstakingVault` above the `queuedWithdrawalsBuffer` — a realistic precondition given normal protocol operation.
- The attack is repeatable every time rewards accumulate in `FeeReceiver`.

Likelihood: **Medium-High**.

## Recommendation

1. **Add access control to `sendFunds()`** (e.g., `onlyRole(LRTConstants.MANAGER)`) so rewards can only be injected by trusted operators who coordinate with the oracle update.
2. **Alternatively, enforce a minimum holding period** before `instantWithdrawal` is available to a given rsETH balance, preventing atomic deposit→price-update→withdraw sequences.
3. **Snapshot the rsETH price at deposit time** and use the lesser of the deposit-time price and the current price when computing instant-withdrawal amounts — preventing a depositor from benefiting from a price increase that occurred after their deposit.
4. **Ensure `pricePercentageLimit` is always set** to a non-zero value in production to limit the magnitude of any single price update.

## Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

// Foundry fork test — run against a local fork with FeeReceiver holding accumulated rewards
// and LRTUnstakingVault holding ETH above queuedWithdrawalsBuffer.

function testYieldExtractionViaInstantWithdrawal() public {
    address attacker = makeAddr("attacker");
    uint256 depositAmount = 10 ether;
    vm.deal(attacker, depositAmount);

    // 1. Record attacker's starting balance
    uint256 startBalance = attacker.balance; // 10 ETH

    // 2. Attacker deposits ETH at current (pre-reward) rsETH price P0
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: depositAmount}(0, "");
    uint256 rsETHBalance = rsETH.balanceOf(attacker);

    // 3. Anyone calls sendFunds() — pushes MEV rewards into LRTDepositPool
    feeReceiver.sendFunds(); // permissionless — no access control

    // 4. Anyone calls updateRSETHPrice() — price rises to P1
    lrtOracle.updateRSETHPrice(); // permissionless — only whenNotPaused

    // 5. Attacker instantly withdraws at P1
    vm.startPrank(attacker);
    rsETH.approve(address(lrtWithdrawalManager), rsETHBalance);
    lrtWithdrawalManager.instantWithdrawal(LRTConstants.ETH_TOKEN, rsETHBalance, "");
    vm.stopPrank();

    // 6. Assert net gain
    uint256 endBalance = attacker.balance;
    assertGt(endBalance, startBalance, "Attacker extracted yield from existing stakers");
}
```
