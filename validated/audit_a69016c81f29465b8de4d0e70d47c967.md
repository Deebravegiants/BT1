### Title
Production Deployment Config Enables Unsafe `assume_no_malicious_validators: true`, Reducing Consensus Quorum from Byzantine (>2/3) to Honest (>1/2) — (`File: crates/apollo_deployments/resources/app_configs/consensus_manager_config.json`)

---

### Summary

The production deployment configuration file `consensus_manager_config.json` explicitly sets `assume_no_malicious_validators: true`. This overrides the safe Rust-code default of `false` and causes the Tendermint consensus engine to use `HONEST_QUORUM` (>1/2 of stake) instead of `BYZANTINE_QUORUM` (>2/3 of stake). The safety threshold of the BFT protocol is thereby reduced: a coalition controlling just over 50% of validator stake — rather than the standard 2/3 — can finalize any block, including one carrying wrong state, wrong receipts, or wrong events.

---

### Finding Description

**Config struct default (safe):**

`ConsensusManagerConfig::assume_no_malicious_validators` defaults to `false` in the Rust code. [1](#0-0) 

The field carries an explicit "Use with caution!" warning in its serialization description. [2](#0-1) 

**Production deployment config (unsafe override):**

The shipped deployment config overrides the safe default to `true` on line 2. [3](#0-2) 

The replacer deployment config does the same. [4](#0-3) 

**How the flag is consumed:**

`create_run_consensus_args` reads the flag and selects `QuorumType::Honest` when it is `true`. [5](#0-4) 

**Quorum constants:**

`BYZANTINE_QUORUM` requires strictly more than 2/3 of total weight; `HONEST_QUORUM` requires strictly more than 1/2. [6](#0-5) 

`VotesThreshold::from_quorum_type` maps the enum to the threshold used by the state machine for every prevote and precommit decision. [7](#0-6) 

The state machine wires the chosen threshold into every quorum check at construction time. [8](#0-7) 

**No validation guard:**

`ConsensusManagerConfig` derives `Validate` but the `assume_no_malicious_validators` field carries no `#[validate(...)]` constraint, so the unsafe value passes all config-load checks silently. [9](#0-8) 

**Compounding unsafe setting in the same file:**

The same production config also sets `compare_retrospective_block_hash: false`, disabling the cross-check between the batcher's and state-sync's view of the retrospective block hash. [10](#0-9) 

The guard that would catch a batcher/state-sync divergence is conditioned on this flag. [11](#0-10) 

---

### Impact Explanation

With `HONEST_QUORUM` active, any coalition of validators holding strictly more than 50% of total stake can reach both prevote and precommit quorum for an arbitrary `ProposalCommitment`. The batcher then executes the transactions in that commitment, producing state diffs, receipts, and events that are committed as canonical. This maps to **wrong state, receipt, event, and storage values** being accepted by the sequencer — the Critical impact class in the allowed scope.

---

### Likelihood Explanation

Every node operator who deploys using the provided `consensus_manager_config.json` (the canonical deployment artifact) runs with the weakened quorum. No code change is required; the unsafe value ships in the repository. The exploitation requires a coalition of validators controlling >50% of stake, which is a higher bar than the standard 1/3 Byzantine threshold, but the configuration removes the standard safety margin entirely and is the direct analog of the router's `confirmations: 1` unsafe default.

---

### Recommendation

1. Change `assume_no_malicious_validators` to `false` in both `consensus_manager_config.json` and `replacer_consensus_manager_config.json`.
2. Add a `#[validate(custom = "reject_if_true")]` (or equivalent) constraint on the `assume_no_malicious_validators` field in `ConsensusManagerConfig` so that any deployment config that sets it to `true` fails config validation at startup with an explicit error.
3. Change `compare_retrospective_block_hash` back to `true` in the production deployment config to restore the batcher/state-sync divergence guard.
4. Document in the deployment guide that `assume_no_malicious_validators: true` is a development/testing knob only and must never appear in a production deployment.

---

### Proof of Concept

1. Deploy the sequencer using the shipped `crates/apollo_deployments/resources/app_configs/consensus_manager_config.json`.
2. `ConsensusManager::create_run_consensus_args` reads `self.config.assume_no_malicious_validators == true` and passes `QuorumType::Honest` to `RunConsensusArguments`.
3. `StateMachine::new` sets `self.quorum = VotesThreshold::new(1, 2)` (>50%).
4. A coalition of validators controlling 51% of stake broadcasts prevotes and precommits for a crafted `ProposalCommitment` containing wrong transactions.
5. `VotesThreshold::is_met(51, 100)` → `51 * 2 > 100 * 1` → `true`; quorum is declared.
6. `decision_reached` is called; the batcher executes the wrong transactions; wrong state, receipts, and events are committed as canonical chain state.
