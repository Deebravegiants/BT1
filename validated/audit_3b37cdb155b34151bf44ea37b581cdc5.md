### Title
Caller-Agnostic `verify_foreign_transaction` Queue Key Enables Queue Saturation, Blocking Legitimate Bridge Requests - (File: `crates/contract/src/pending_requests.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `pending_verify_foreign_tx_requests` map uses a caller-agnostic key (`VerifyForeignTransactionRequest` without the caller's account ID). Any unprivileged caller can saturate the fan-out queue for a specific foreign transaction to its hard cap of 128 entries at negligible cost (128 yoctoNEAR), permanently blocking legitimate users from enqueuing their own verification requests until the queue drains.

---

### Finding Description

`verify_foreign_transaction` stores pending requests in:

```rust
pending_verify_foreign_tx_requests: LookupMap<VerifyForeignTransactionRequest, Vec<YieldIndex>>
```

The map key type is `VerifyForeignTransactionRequest`, which contains only `(domain_id, payload_version, request)` — **no caller identity**:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

This is explicitly acknowledged in the test comment: *"both yields are queued under the single (caller-agnostic) request key."*

By contrast, `sign()` requests derive a `tweak = derive_tweak(predecessor_id, path)` that is embedded in `SignatureRequest`, making every user's sign-request key unique:

```rust
pub struct SignatureRequest {
    pub tweak: Tweak,   // derived from (predecessor_id, path)
    pub payload: Payload,
    pub domain_id: DomainId,
}
```

Because the foreign-tx key is caller-agnostic, `push_pending_yield` will append any caller's yield to the **same** queue entry. The queue cap is:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
```

Once the queue reaches 128 entries, `push_pending_yield` panics with `PendingRequestQueueFull`, reverting the caller's transaction:

```rust
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string(),
    );
}
```

An attacker who knows a foreign transaction ID (publicly visible on-chain) can submit 128 identical `verify_foreign_transaction` calls for that tx, saturating the queue. Any subsequent legitimate user call for the same tx reverts immediately.

---

### Impact Explanation

**Request-lifecycle manipulation breaking a production safety invariant.** The invariant is: *any user who attaches the required 1 yoctoNEAR deposit and gas can enqueue a foreign-tx verification request.* The attacker breaks this invariant for a targeted tx at negligible cost.

Concrete consequences:
- The legitimate user's `verify_foreign_transaction` transaction reverts with `PendingRequestQueueFull`.
- The NEAR yield-resume timeout is 200 blocks (`REQUEST_EXPIRATION_BLOCKS = 200`). If the attacker refills the queue each time it drains (128 yoctoNEAR per refill, essentially free), the legitimate user's yield will time out and their transaction will permanently fail.
- Bridge contracts that depend on `verify_foreign_transaction` to release funds locked on a foreign chain will be unable to complete, potentially freezing those funds for the duration of the attack.

This maps to the allowed Medium impact: *"request-lifecycle … manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

- **Entry path is fully unprivileged**: any NEAR account with 128 yoctoNEAR and gas can execute the attack.
- **Target is observable**: foreign-chain tx IDs are public; the attacker can watch the foreign chain mempool or block explorer and front-run the victim's NEAR submission.
- **Cost is negligible**: 128 yoctoNEAR ≈ $0.000000000000128 at current prices; gas cost is the only real expense.
- **Sustaining the attack is cheap**: the queue drains only when MPC nodes respond (a few blocks); the attacker can immediately re-saturate it.

---

### Recommendation

1. **Include caller identity in the queue key** — derive a per-caller component (e.g., `predecessor_id` hashed into the key) analogous to the `tweak` used in `SignatureRequest`. This ensures each user's slot is independent and an attacker cannot fill another user's queue.
2. **Per-account submission cap** — track how many pending yields a single account has queued for a given request key and reject submissions that exceed a per-account limit (e.g., 1 or 2).
3. **Minimum deposit scaled to queue depth** — require a deposit proportional to the current queue length, making saturation economically costly.

---

### Proof of Concept

```
1. Attacker observes Bitcoin tx_id = 0xABCD... on the Bitcoin mempool.

2. Attacker submits 128 calls to verify_foreign_transaction({
       domain_id: <ForeignTx domain>,
       payload_version: V1,
       request: Bitcoin { tx_id: 0xABCD..., confirmations: 1, extractors: [BlockHash] }
   })
   each with 1 yoctoNEAR deposit.
   Total cost: 128 yoctoNEAR + gas.

3. pending_verify_foreign_tx_requests[key].len() == 128  (queue full).

4. Legitimate user submits the identical verify_foreign_transaction call.
   → push_pending_yield panics: "Pending-request queue is full (limit: 128)"
   → User's transaction reverts; their yield is never created.

5. MPC nodes eventually respond; queue drains.
   Attacker immediately re-submits 128 calls (step 2).

6. User's yield-resume window (200 blocks) expires before they can enqueue.
   Bridge contract that awaited the verification cannot release the user's funds.
```

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```

```rust
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",

            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,

            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
```
