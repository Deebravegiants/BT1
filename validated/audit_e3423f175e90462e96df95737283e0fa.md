Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` Enables Front-Running of MEV Reward Distribution to Steal Yield from Existing rsETH Holders - (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` has no access control, allowing any external caller to control the exact timing of MEV/execution-layer reward distribution into the protocol TVL. Because the `FeeReceiver` balance is explicitly excluded from TVL until `sendFunds()` is called, an attacker can deposit at a stale (lower) rsETH price, flush the rewards, and then trigger a price update — capturing a portion of yield that rightfully belongs to existing holders.

## Finding Description
**Root cause:** `FeeReceiver.sendFunds()` carries no role restriction:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

The `LRTDepositPool.getETHDistributionData()` function explicitly excludes the `FeeReceiver` balance from TVL (the comment at L465-466 confirms this is by design):

```solidity
// contracts/LRTDepositPool.sol L464-500
/// @dev rewards are not accounted here
/// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
function getETHDistributionData() ...
    ethLyingInDepositPool = address(this).balance;  // only DepositPool balance, not FeeReceiver
```

`LRTOracle._getTotalEthInProtocol()` calls `getTotalAssetDeposits()` → `getETHDistributionData()`, so the stored `rsETHPrice` does not reflect accumulated FeeReceiver rewards. The mint price in `getRsETHAmountToMint()` divides by this stale stored price:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`LRTOracle.updateRSETHPrice()` is permissionless (`public whenNotPaused`, no role check):

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

**Exploit path:**
1. Rewards R accumulate in `FeeReceiver` (excluded from TVL, so rsETHPrice is stale/low).
2. Attacker calls `LRTDepositPool.depositETH()` with amount A, receiving `A / rsETHPrice` rsETH at the stale price.
3. Attacker calls `FeeReceiver.sendFunds()` → R ETH moves into `LRTDepositPool.balance`, increasing TVL.
4. Attacker calls `LRTOracle.updateRSETHPrice()` → new price = `(TVL + R - fee) / totalSupply > rsETHPrice`.
5. Attacker's rsETH is now worth more than A; the gain comes directly from existing holders' unclaimed yield.

**Partial mitigation analysis:** `_updateRsETHPrice()` contains a `pricePercentageLimit` guard (L256-265) that reverts non-manager callers if the price increase exceeds the configured threshold. However, this does not eliminate the vulnerability because: (a) `pricePercentageLimit` can be set to 0, disabling the guard entirely; (b) even when set, the attacker can wait for rewards to accumulate just below the threshold and execute the attack within the allowed band; (c) the guard only blocks the `updateRSETHPrice()` call — the attacker can still call `sendFunds()` to move rewards into TVL and wait for a legitimate price update.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders accumulate MEV/execution-layer rewards over time. Because the `FeeReceiver` balance is excluded from TVL until `sendFunds()` is called, a new depositor who enters just before that call receives rsETH priced as if those rewards do not exist, then immediately benefits from the price increase when rewards are flushed. The yield that accrued to existing holders is permanently diluted proportional to the attacker's deposit size relative to total TVL. The protocol fee mechanism (L244-247 in `LRTOracle.sol`) partially offsets the gain but does not eliminate it.

## Likelihood Explanation
Low/Medium. The attack requires rewards to accumulate in `FeeReceiver` between calls to `sendFunds()`. Because `sendFunds()` is permissionless, the attacker fully controls the timing of the flush. The attacker can monitor the `FeeReceiver` balance on-chain, deposit when it reaches a profitable threshold, then call `sendFunds()` and `updateRSETHPrice()` atomically (or across two blocks). No privileged access is required. The `pricePercentageLimit` guard constrains the per-attack profit but does not prevent repeated small-scale extractions.

## Recommendation
Restrict `FeeReceiver.sendFunds()` to a trusted role so that reward distribution timing cannot be weaponized:

```solidity
// contracts/FeeReceiver.sol
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Alternatively, include the `FeeReceiver` balance in `LRTDepositPool.getETHDistributionData()` so that accumulated rewards are always reflected in the rsETH price, eliminating the pricing gap entirely.

## Proof of Concept
**Setup:** TVL = 1000 ETH, rsETH totalSupply = 1000, rsETHPrice = 1.0 ETH/rsETH. FeeReceiver holds 10 ETH in accumulated MEV rewards (not in TVL).

**Attack sequence (single block or two consecutive blocks):**
1. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")` → receives `100 / 1.0 = 100 rsETH`. TVL = 1100 ETH, totalSupply = 1100 rsETH, rsETHPrice still = 1.0.
2. Attacker calls `FeeReceiver.sendFunds()` → 10 ETH moves to DepositPool. DepositPool balance increases by 10 ETH. TVL (as seen by oracle) = 1110 ETH.
3. Attacker calls `LRTOracle.updateRSETHPrice()` → new price ≈ `(1110 - fee) / 1100`. Assuming 10% protocol fee on 10 ETH reward = 1 ETH fee → new price = `1109 / 1100 ≈ 1.00818`.
4. Attacker's 100 rsETH is now worth ≈ 100.818 ETH — a gain of ~0.818 ETH at the expense of existing holders.

**Without the attack:** Existing 1000 rsETH holders would have received the full reward (minus protocol fee) → price ≈ `1009 / 1000 = 1.009`. With the attack, they receive ≈ `1000 × 1.00818 = 1008.18 ETH` — losing ~0.818 ETH of yield to the attacker.

**Foundry test plan:**
```solidity
function testFrontRunSendFunds() public fork {
    // 1. Record attacker ETH balance and current rsETHPrice
    uint256 priceBefore = lrtOracle.rsETHPrice();
    // 2. Deposit 100 ETH as attacker
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 100 ether}(0, "");
    // 3. Call sendFunds (no access control)
    feeReceiver.sendFunds();
    // 4. Update price
    lrtOracle.updateRSETHPrice();
    uint256 priceAfter = lrtOracle.rsETHPrice();
    // 5. Assert attacker's rsETH value > deposited ETH
    assertGt(priceAfter, priceBefore);
    uint256 attackerRsETH = rsETH.balanceOf(attacker);
    assertGt(attackerRsETH * priceAfter / 1e18, 100 ether);
}
```
