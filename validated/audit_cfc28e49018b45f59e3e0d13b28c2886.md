Audit Report

## Title
Free LP Share Minting in Drained Bins via Missing `msg.sender == owner` Guard in `addLiquidity` — (File: `metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no `msg.sender == owner` check, unlike `removeLiquidity` which enforces this invariant. When a bin has been fully drained by swaps (`token0BalanceScaled == 0` and `token1BalanceScaled == 0`) but retains non-zero `binTotalShares`, the proportional share-price formula yields zero token amounts, the payment callback is skipped entirely, yet shares are unconditionally credited to the attacker-controlled `owner` address. The attacker can later call `removeLiquidity` as that owner to extract a proportional share of any tokens that subsequently enter the bin, directly stealing LP principal.

## Finding Description
`removeLiquidity` enforces caller identity at `MetricOmmPool.sol` line 206:
```solidity
if (msg.sender != owner) revert NotPositionOwner();
```
`addLiquidity` has no equivalent guard (`MetricOmmPool.sol` lines 182–196). [1](#0-0) 

Inside `LiquidityLib.addLiquidity`, when `binTotalSharesVal > 0` the proportional branch executes:
```solidity
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
```
If both `token0BalanceScaled` and `token1BalanceScaled` are zero (a fully drained bin), `Math.ceilDiv(0, binTotalSharesVal) == 0` for any non-zero denominator, so both scaled amounts are zero. [2](#0-1) 

The payment callback is gated on non-zero amounts and is therefore skipped entirely:
```solidity
if (amount0Added > 0 || amount1Added > 0) { ... callback ... }
``` [3](#0-2) 

Despite zero payment, shares are written unconditionally:
```solidity
binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd;
positionBinShares[posKey] = newUserShares;
``` [4](#0-3) 

The position key is `keccak256(abi.encode(owner, salt, binIdx))`, so the attacker sets `owner = address(this)` and receives the shares. [5](#0-4) 

The `_beforeAddLiquidity` extension hook passes `sender` and `owner` separately to extensions but imposes no core-protocol ownership check. The optional `DepositAllowlistExtension` checks `owner` against an allowlist, but this extension is not mandatory and does not prevent the attack on pools without it. [6](#0-5) 

## Impact Explanation
Direct loss of LP principal — High severity. After the attacker mints an arbitrarily large share count at zero cost, every token that enters the drained bin (via a reverse swap or a new LP deposit) is split proportionally across all shares including the attacker's. Existing LPs receive a fraction of what they are owed; the remainder is extractable by the attacker via `removeLiquidity`. The loss scales with `attacker_shares / (original_shares + attacker_shares)` and is bounded only by the value that subsequently flows through the bin. This meets the Sherlock direct-loss-of-LP-principal threshold.

## Likelihood Explanation
Medium-High. Bins are routinely drained to zero during normal swap activity (a bin above the current price holds only token0; a full `!zeroForOne` traversal sets `token0BalanceScaled = 0` while `token1BalanceScaled` was already 0). The attack requires no special privilege, no malicious token, and no trusted role — only the ability to call `addLiquidity` directly on the pool, bypassing the periphery. The attacker can monitor on-chain state for drained bins and act in the same block.

## Recommendation
**Short term:** Add the same ownership check to `addLiquidity` that already exists in `removeLiquidity`:
```solidity
// MetricOmmPool.addLiquidity — add before LiquidityLib call
if (msg.sender != owner) revert NotPositionOwner();
```
If the protocol intentionally allows third-party deposits on behalf of another address (e.g., for `MetricOmmPoolLiquidityAdder`), introduce an explicit allowance or approval mechanism rather than leaving the `owner` field unconstrained.

**Long term:** Add an invariant test asserting that `addLiquidity` with `owner != msg.sender` reverts when called directly on the pool, and add a fuzz test verifying no shares can be minted without a corresponding token transfer whenever `sharesToAdd > 0` and the bin is non-empty.

## Proof of Concept
1. Deploy a pool and seed bin `B` with liquidity so `binTotalShares[B] > 0`.
2. Execute a swap that fully drains bin `B` so `token0BalanceScaled[B] == 0` and `token1BalanceScaled[B] == 0`.
3. Call `pool.addLiquidity(address(attacker), 0, delta, "", "")` where `delta` specifies bin `B` and a large `sharesToAdd` (e.g., `1e30`).
4. Observe: `amount0Added == 0`, `amount1Added == 0`, callback never invoked, no tokens transferred, yet `binTotalShares[B]` and `positionBinShares[keccak256(attacker, 0, B)]` are both incremented.
5. Execute a reverse swap or new LP deposit that refills bin `B` with real tokens.
6. Call `pool.removeLiquidity(address(attacker), 0, delta, "")` — passes the `msg.sender == owner` check — and receive a proportional share of the refilled tokens, leaving existing LPs with less than their entitled share.
