### Title
Expired TEE Attestation Bypasses `respond` Authorization Due to Inconsistent Validity Checks - (File: `crates/contract/src/tee/tee_state.rs`)

### Summary

The contract uses two structurally different mechanisms to determine whether a participant's TEE attestation is valid. The check used to gate `respond`, `respond_ckd`, and `respond_verify_foreign_tx` (`is_caller_an_attested_participant`) only verifies identity/key binding and **never checks attestation expiry**. The check used in `vote_new_parameters` and `verify_tee` (`reverify_participants`) **does** check expiry via `re_verify`. A node whose attestation has expired is simultaneously considered invalid by the governance path and valid by the signing path, allowing it to continue submitting threshold signatures after its TEE certificate has lapsed.

### Finding Description

`is_caller_an_attested_participant` in `crates/contract/src/tee/tee_state.rs` performs three checks:

1. The caller's `account_id` is in the active participant list.
2. A stored attestation exists for the participant's TLS key.
3. The stored `account_public_key` matches the transaction signer key. [1](#0-0) 

It does **not** call `re_verify` and does **not** inspect the attestation's expiry timestamp.

`reverify_participants`, by contrast, calls `re_verify` with the current block timestamp and rejects any attestation whose `expiry_timestamp_seconds` has passed: [2](#0-1) 

`assert_caller_is_attested_participant_and_protocol_active` — the guard used by `respond`, `respond_ckd`, and `respond_verify_foreign_tx` — delegates exclusively to `is_caller_an_attested_participant`: [3](#0-2) 

`vote_new_parameters` and `verify_tee` delegate to `reverify_and_cleanup_participants`, which calls `reverify_participants` and therefore enforces expiry: [4](#0-3) 

The two paths therefore disagree on whether a participant with an expired attestation is valid:

| Operation | Check used | Expiry enforced? |
|---|---|---|
| `respond` / `respond_ckd` / `respond_verify_foreign_tx` | `is_caller_an_attested_participant` | **No** |
| `vote_new_parameters` / `verify_tee` | `reverify_participants` | **Yes** |

### Impact Explanation

TEE attestations expire after `DEFAULT_EXPIRATION_DURATION_SECONDS` (7 days). After expiry, a node may no longer be running inside a trusted execution environment — its measurements may have changed, its image may have been replaced, or the hardware isolation may have been removed. The contract's TEE security model is predicated on the invariant that only nodes with currently-valid attestations can contribute to threshold signatures.

Because `respond` does not enforce expiry, a node whose attestation has lapsed can continue calling `respond` and having its signature contributions accepted. This persists until `verify_tee` is explicitly invoked (a separate, periodic, externally-triggered call). During the gap — which can span the full 7-day attestation validity window if `verify_tee` is not called promptly — the expired node participates in signing as if its TEE status were still valid.

A Byzantine participant below the signing threshold can exploit this by intentionally not refreshing its attestation, then continuing to sign after expiry while the governance path would reject it.

### Likelihood Explanation

Attestations expire on a fixed 7-day cycle. `verify_tee` is a separate, manually or periodically triggered call; it is not invoked on every `respond`. Any window between attestation expiry and the next `verify_tee` call is an exploitation window. The design documentation confirms nodes are expected to call `verify_tee` every 7 days, meaning the window can be as long as the full attestation validity period. [5](#0-4) 

### Recommendation

`assert_caller_is_attested_participant_and_protocol_active` should call `reverify_participants` (or an equivalent expiry check) in addition to `is_caller_an_attested_participant`. Both the identity check and the liveness/expiry check must pass before a node is permitted to submit a response. This mirrors the fix recommended in M-12: both paths must use the same validity criterion.

Alternatively, `is_caller_an_attested_participant` itself should be extended to call `re_verify` so that a single function enforces the complete validity invariant regardless of call site.

### Proof of Concept

1. Node A submits a valid TEE attestation at time T=0. Attestation expires at T=7d.
2. At T=7d+1s, `reverify_participants` returns `TeeQuoteStatus::Invalid` for Node A — `vote_new_parameters` would reject any proposal including Node A.
3. `verify_tee` has not yet been called; `accept_requests` remains `true`.
4. Node A calls `respond` with a valid signature. `assert_caller_is_attested_participant_and_protocol_active` calls `is_caller_an_attested_participant`, which finds Node A's stored attestation, verifies the key binding, and returns `Ok(())` — **no expiry check occurs**.
5. The `respond` call succeeds. Node A's contribution is accepted into the threshold signature despite its TEE attestation being expired.
6. This continues until an operator explicitly calls `verify_tee`, which may not happen for up to another 7 days. [1](#0-0) [2](#0-1) [6](#0-5)

    pub(crate) fn reverify_participants(
        &self,
        node_id: &NodeId,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeQuoteStatus {
        let allowed_mpc_docker_image_hashes =
            self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
        let allowed_launcher_compose_hashes = self.get_allowed_launcher_compose_hashes();
        let allowed_measurements = self.get_accepted_measurements();

        let participant_attestation = self.stored_attestations.get(&node_id.tls_public_key);
        let Some(participant_attestation) = participant_attestation else {
            return TeeQuoteStatus::Invalid("participant has no attestation".to_string());
        };

        // Verify the attestation quote
        let time_stamp_seconds = Self::current_time_seconds();
        match participant_attestation.verified_attestation.re_verify(
            time_stamp_seconds,
            &allowed_mpc_docker_image_hashes,
            &allowed_launcher_compose_hashes,
            &allowed_measurements,
        ) {
            Ok(()) => TeeQuoteStatus::Valid,
            Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
        }
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
