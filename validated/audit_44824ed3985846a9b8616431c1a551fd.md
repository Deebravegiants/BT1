### Title
Fee Exemption Bypass via Intra-Block Redeem-Address Chaining — (`eras/byron/ledger/impl/src/Cardano/Chain/UTxO/Validation.hs`)

---

### Summary

The Byron UTxO validation logic in `updateUTxO` processes transactions sequentially via `foldM`, threading the updated UTxO state through each transaction. Because `validateTxAux` grants a zero-minimum-fee exemption to any transaction whose inputs are all redeem-type addresses, and because an attacker can generate their own redeem key pair and create a redeem address freely, an attacker who controls a block slot can craft a two-transaction block where tx1 creates a self-controlled redeem-address output and tx2 spends it with zero fee — bypassing the fee requirement entirely for tx2.

---

### Finding Description

**`updateUTxO`** (line 413) folds over the transaction list, passing the post-tx UTxO forward:

```haskell
updateUTxO env as = foldM (updateUTxOTxWitness env) as
``` [1](#0-0) 

When tx2 is validated, `utxo` is already the post-tx1 state, so tx2's inputs (outputs of tx1) are present and resolvable.

**`validateTxAux`** computes `inputUTxO` as the UTxO restricted to tx2's inputs, then checks:

```haskell
minFee <-
  if isRedeemUTxO inputUTxO
    then pure $ mkKnownLovelace @0
    else calculateMinimumFee feePolicy
``` [2](#0-1) 

**`isRedeemUTxO`** returns `True` if every output in the restricted UTxO has a redeem-type address:

```haskell
isRedeemUTxO =
  all (isRedeemAddress . txOutAddress . fromCompactTxOut)
    . M.elems . unUTxO
``` [3](#0-2) 

**`isRedeemAddress`** is purely a type-tag check — it does not verify the output came from genesis:

```haskell
isRedeemAddress addr = case addrType addr of
  ATRedeem -> True
  _ -> False
``` [4](#0-3) 

**Anyone can generate a redeem key pair and derive a redeem address.** `makeRedeemAddress` takes any `RedeemVerificationKey` and produces a valid `ATRedeem`-typed address: [5](#0-4) 

The witness check in `updateUTxOTxWitness` only verifies that the `RedeemWitness` key matches the address — it does not require the address to originate from the genesis UTxO: [6](#0-5) 

**Attack steps:**
1. Attacker generates redeem key pair `(rk_priv, rk_pub)` and derives `addr_redeem = makeRedeemAddress nm rk_pub`.
2. **tx1**: Spends a normal input, creates an output to `addr_redeem`. Pays normal fee.
3. **tx2**: Spends the `addr_redeem` output from tx1 with a valid `RedeemWitness` signed by `rk_priv`. Sets fee = 0 (all input value goes to output).
4. Attacker (as block producer) includes both in the same block in order.
5. `updateUTxO` processes tx1 first, adds the redeem output to UTxO, then processes tx2 against the updated UTxO. `isRedeemUTxO inputUTxO` is `True`, so `minFee = 0`. Fee check `0 <= 0` passes.

**Amplification**: tx2 can itself create another redeem output (to a second attacker-controlled redeem address), and tx3 can spend that with zero fee, and so on — all within the same block. The attacker pays one fee (tx1) and can chain arbitrarily many zero-fee transactions within the block size limit.

---

### Impact Explanation

This falls under the **Medium** impact category: attacker-controlled transactions modify fees outside design parameters. The fee exemption for redeem addresses was designed for one-time genesis redemptions, not for attacker-generated redeem addresses. The attacker can evade fees on an unbounded number of intra-block transactions, paying only the cost of tx1.

---

### Likelihood Explanation

The attacker must control a block-producing slot (Byron slot leader). This is a realistic condition for any party with sufficient stake. The exploit requires no external dependencies, no privileged keys beyond slot leadership, and is locally constructable and testable. However, Byron era is historical on mainnet — new blocks are no longer validated under Byron rules — which significantly limits real-world exploitability to chain-replay scenarios or Byron-compatible testnets.

---

### Recommendation

1. In `validateTxAux`, restrict the zero-fee exemption to UTxO entries that can be traced to the genesis UTxO (e.g., by checking against a genesis redeem set), rather than any output with `ATRedeem` address type.
2. Alternatively, remove the `isRedeemUTxO` exemption entirely for outputs not present at chain genesis.
3. At minimum, document that the exemption is only safe when the genesis UTxO is the sole source of redeem-type outputs.

---

### Proof of Concept

```haskell
-- Pseudocode sketch
let (rk_priv, rk_pub) = redeemDeterministicKeyGen seed
    addr_redeem        = makeRedeemAddress nm rk_pub

-- tx1: normal input -> redeem output (pays normal fee)
tx1 = Tx { txInputs  = [normalInput]
          , txOutputs = [TxOut addr_redeem 1000000] }

-- tx2: redeem output -> normal output, fee = 0
tx2 = Tx { txInputs  = [TxInUtxo (hash tx1) 0]
          , txOutputs = [TxOut addr_normal 1000000] }  -- no fee deducted

-- Block contains [tx1Aux, tx2Aux] in order
result <- runExceptT $ runReaderT
            (updateUTxO env genesisUTxO [tx1Aux, tx2Aux])
            TxValidation

-- Expected: Right utxo  (tx2 passes with minFee = 0)
-- Demonstrates: fee evasion via self-controlled redeem address chaining
```

The key assertion is that after `updateUTxO`, tx2 passes `validateTxAux` with `minFee = mkKnownLovelace @0` because `isRedeemUTxO (inputsOf tx2 <| post_tx1_utxo) = True`.
