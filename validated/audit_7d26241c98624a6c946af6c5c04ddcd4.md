### Title
MPC Node Docker Containers Run as Root, Enabling Privilege Escalation to Key Share and Signing Key Exposure — (File: deployment/Dockerfile-node, deployment/Dockerfile-node-gcp, deployment/Dockerfile-rust-launcher)

---

### Summary
All three production Dockerfiles in this repository omit a `USER` directive, causing the `mpc-node` binary and the `tee-launcher` binary to execute as root (UID 0) inside their containers. Because the MPC node processes attacker-reachable inputs (public `sign` requests relayed from the NEAR blockchain, P2P messages from network peers), any code-execution vulnerability reachable through those surfaces immediately grants the attacker full root access inside the container. At that point the attacker can read the local encryption key and the NEAR account signing keys from the process environment, decrypt the on-disk keyshares, and obtain the secret material needed to forge threshold signatures or recover key shares.

---

### Finding Description

`deployment/Dockerfile-node`, `deployment/Dockerfile-node-gcp`, and `deployment/Dockerfile-rust-launcher` each build a final image that copies the production binary and immediately sets a `CMD` — with no `USER` instruction anywhere in the file. [1](#0-0) [2](#0-1) [3](#0-2) 

Docker's default when no `USER` is specified is root (UID 0). The `start.sh` entrypoint fetches and exports three categories of secret material into the process environment before exec-ing `mpc-node`:

* `MPC_P2P_PRIVATE_KEY` — the node's P2P identity key
* `MPC_ACCOUNT_SK` — the NEAR account signing key used to submit `respond` transactions
* `MPC_SECRET_STORE_KEY` — the AES-128 key used to encrypt keyshares on disk [4](#0-3) [5](#0-4) 

The keyshares themselves are stored locally as AES-128-GCM ciphertext, but the decryption key lives in the same process environment: [6](#0-5) [7](#0-6) 

Secret files are written with `0o600` permissions, but those permissions are meaningless when the process that created them — and any attacker who achieves code execution inside the container — is root and can bypass DAC entirely: [8](#0-7) 

The `Dockerfile-rust-launcher` image additionally installs `docker-cli` and `docker-compose`, meaning the launcher process runs as root with Docker CLI access — a well-known path to full host escape if the Docker socket is mounted: [9](#0-8) 

---

### Impact Explanation

An attacker who achieves code execution inside the `mpc-node` container (see Likelihood below) immediately has root access. From there they can:

1. Read `MPC_SECRET_STORE_KEY` from `/proc/1/environ` or the live process environment.
2. Decrypt the on-disk permanent and temporary keyshares using that AES-128 key.
3. Read `MPC_ACCOUNT_SK` and use it to call `respond` on the MPC contract, submitting attacker-chosen signatures.

A single compromised node's keyshare, combined with threshold-minus-one honest shares obtained through the normal signing protocol, is sufficient to reconstruct the full private key. This maps directly to the Critical allowed impact: **unauthorized access to MPC key shares and signing capability that materially enables forgery or secret recovery**.

---

### Likelihood Explanation

The `mpc-node` binary exposes multiple attacker-reachable surfaces:

* **Public `sign` requests** — any NEAR account can submit a `sign` call; the node parses the resulting receipt and payload without authentication.
* **P2P message handling** — while TLS-authenticated between participants, the message parsing and protocol logic inside the node is a large attack surface for memory-safety or logic bugs in Rust async code and third-party cryptographic libraries.

Running as root does not introduce a new vulnerability by itself, but it eliminates the OS-level containment that would otherwise limit the blast radius of any such bug to the unprivileged user's files. The principle-of-least-privilege fix (adding `USER`) is the standard mitigation for exactly this amplification.

---

### Recommendation

Add a dedicated non-root user to each Dockerfile, transfer ownership of all required files to that user, and switch to it before the `CMD`:

```dockerfile
RUN addgroup --system mpc && adduser --system --ingroup mpc mpc
COPY --chown=mpc:mpc target/reproducible/mpc-node mpc-node
COPY --chown=mpc:mpc --chmod=0755 deployment/start.sh /app/start.sh
USER mpc
CMD ["/app/start.sh"]
```

Apply the same pattern to `Dockerfile-node-gcp` and `Dockerfile-rust-launcher`. For the launcher image, also ensure the Docker socket is not mounted as root-owned, or use rootless Docker.

---

### Proof of Concept

```
$ docker build -f deployment/Dockerfile-node -t mpc-node-test .
$ docker run --rm --entrypoint id mpc-node-test
uid=0(root) gid=0(root) groups=0(root)
```

No `USER` instruction appears anywhere in any of the three Dockerfiles. Once inside the container as root, an attacker with code execution can trivially read the decryption key and all secret material:

```bash
# inside container as root (uid=0)
cat /proc/1/environ | tr '\0' '\n' | grep MPC_SECRET_STORE_KEY
# → MPC_SECRET_STORE_KEY=<hex key>
# decrypt keyshare files with recovered key → obtain raw keyshare scalar
``` [10](#0-9) [11](#0-10)

/// Writes data to a file with owner-only permissions (0o600).
/// Permissions are set explicitly after writing to ensure correctness
/// even if the file already exists with different permissions.
fn write_secret_file(path: &Path, data: &[u8]) -> anyhow::Result<()> {
    let mut file = fs::OpenOptions::new()
        .write(true)
        .create(true)
        .truncate(true)
        .mode(0o600)
        .open(path)
        .with_context(|| format!("failed to create secret file {}", path.display()))?;
    file.write_all(data)
        .with_context(|| format!("failed to write secret file {}", path.display()))?;
    fs::set_permissions(path, fs::Permissions::from_mode(0o600)).with_context(|| {
        format!(
            "failed to set permissions on secret file {}",
            path.display()
        )
    })?;
    Ok(())
