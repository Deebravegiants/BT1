### Title
Silent Cross-Session Q Inconsistency in 2-Party Schnorr Refresh — (`src/cbmpc/protocol/ec_dkg.cpp`)

### Summary

`key_share_2p_t::refresh` copies `new_key.Q` exclusively from the calling party's local blob and never cross-checks it against the peer's Q. A Byzantine P2 supplying a blob from a different DKG session causes both parties to return `SUCCESS` from refresh while holding new key blobs that encode different global public keys.

### Finding Description

The full call chain is:

```
cbmpc_schnorr_2p_refresh          (src/cbmpc/c_api/schnorr2pc.cpp:46-72)
  → coinbase::api::schnorr_2p::refresh  (src/cbmpc/api/schnorr2pc.cpp:79-99)
    → key_share_2p_t::refresh           (src/cbmpc/protocol/ec_dkg.cpp:91-111)
```

At the API layer, the only validation performed on the incoming blob is a role check: [1](#0-0) 

Inside `key_share_2p_t::refresh`, `new_key.Q` is set unconditionally from the local blob: [2](#0-1) 

The only inter-party interaction is `agree_random`, which exchanges random bits and XORs them to produce a shared scalar `r`: [3](#0-2) 

`agree_random` transmits no Q values and performs no consistency check on them. After it returns, P1 sets `new_key.x_share = key.x_share + r` and P2 sets `new_key.x_share = key.x_share - r`, each using their own local `key.Q` unchanged. There is no broadcast, commitment, or comparison of Q between the two parties anywhere in the 2P refresh path.

Contrast this with `key_share_mp_t::refresh`, which explicitly broadcasts a consistency hash over `(sid, current_key.Q, current_key.Qis)` and verifies it against every peer before proceeding: [4](#0-3) 

And at the end of the MP refresh, it asserts `SUM(new_key.Qis) == current_key.Q`: [5](#0-4) 

No equivalent guard exists in the 2P path.

### Impact Explanation

Given:
- P1 holds `(x1_A, Q_A)` from session A
- Byzantine P2 holds `(x2_B, Q_B)` from session B (a different DKG)

After refresh both parties receive `SUCCESS`. The resulting blobs are:
- P1: `(x1_A + r mod q, Q_A)`
- P2: `(x2_B − r mod q, Q_B)`

Both blobs pass all deserialization and role checks. The two parties now hold refreshed key blobs that encode different global public keys (`Q_A ≠ Q_B`). This is accepted invalid cryptographic output: the refresh protocol's postcondition — that both parties share the same public key — is silently violated. Any subsequent signing attempt will produce a signature that is invalid under both `Q_A` and `Q_B`, and P1 has no way to detect that its "refreshed" key is now unusable without attempting a sign.

This falls squarely within the **High** impact scope: attacker-controlled blob data is accepted under the wrong key, producing accepted invalid cryptographic output.

### Likelihood Explanation

Exploitation requires only that P2 possess any second valid key blob (from a prior DKG with any counterparty, or from a DKG with P1 itself). No cryptographic break is needed. The attacker simply calls the refresh API with the wrong blob. The API accepts it without error.

### Recommendation

Add a cross-party Q consistency check at the start of `key_share_2p_t::refresh`, mirroring the MP pattern. Both parties should commit-and-reveal their local `key.Q` (or a hash of it) and abort if the values differ. Concretely:

1. Before calling `agree_random`, have each party broadcast a commitment to `hash(key.Q)`.
2. After opening, verify that both parties' committed Q values are equal.
3. Return an error if they differ.

Alternatively, include `key.Q` as a session-binding input to `agree_random` (e.g., bind it into the commitment or the hash), so that a Q mismatch causes the protocol to produce divergent outputs that are detected downstream.

### Proof of Concept

```

// Session B DKG: P1 gets (x1_B, Q_B), P2 gets (x2_B, Q_B)
run dkg(job_B) → blob_p1_B, blob_p2_B

// Refresh: P1 uses blob from session A, Byzantine P2 uses blob from session B

    P1 input: blob_p1_A   (Q = Q_A)
    P2 input: blob_p2_B   (Q = Q_B, different session)

// Both parties return SUCCESS
assert rv_p1 == SUCCESS
assert rv_p2 == SUCCESS

// Extract Q from each new blob
Q_new_p1 = get_public_key(new_blob_p1)   // == Q_A
Q_new_p2 = get_public_key(new_blob_p2)   // == Q_B

// Assertion: the two refreshed blobs encode different public keys
assert Q_new_p1 != Q_new_p2   // passes — vulnerability confirmed
```

Both `cbmpc_schnorr_2p_refresh` calls return `CBMPC_SUCCESS` and both output blobs pass `cbmpc_schnorr_2p_get_public_key_compressed` without error, yet they encode different public keys.

```cpp
error_t key_share_2p_t::refresh(job_2p_t& job, const key_share_2p_t& key, key_share_2p_t& new_key) {
  error_t rv = UNINITIALIZED_ERROR;
  new_key.role = key.role;
  new_key.curve = key.curve;
  new_key.Q = key.Q;

  const mod_t& q = key.curve.order();
  buf_t rand_bits;
  if (rv = agree_random(job, q.get_bits_count() + SEC_P_STAT, rand_bits)) return rv;
  bn_t r = bn_t::from_bin(rand_bits) % q;

  if (job.is_p1()) {
    MODULO(q) { new_key.x_share = key.x_share + r; }
  }

  if (job.is_p2()) {
    MODULO(q) { new_key.x_share = key.x_share - r; }
  }

  return SUCCESS;
}
```

```cpp
error_t agree_random(job_2p_t& job, int bitlen, buf_t& out) {
  error_t rv = UNINITIALIZED_ERROR;
  buf_t r1, r2;
  const crypto::mpc_pid_t& sender_pid = job.get_pid(party_t::p1);
  coinbase::crypto::commitment_t com(sender_pid);

  if (job.is_p1()) {
    r1 = crypto::gen_random_bitlen(bitlen);
    com.gen(r1);
  }

  if (rv = job.p1_to_p2(com.msg)) return rv;

  if (job.is_p2()) {
    r2 = crypto::gen_random_bitlen(bitlen);
  }

  if (rv = job.p2_to_p1(r2)) return rv;
  if (rv = job.p1_to_p2(r1, com.rand)) return rv;

  if (job.is_p2()) {
    if (rv = com.open(r1)) return rv;
  }

  if (r1.size() != coinbase::bits_to_bytes(bitlen)) return coinbase::error(E_CRYPTO);
  if (r2.size() != coinbase::bits_to_bytes(bitlen)) return coinbase::error(E_CRYPTO);

  out = mem_t(r1) ^ mem_t(r2);
  return SUCCESS;
}
```
