Audit Report

## Title
Permissionless `sendFunds()` + `updateRSETHPrice()` Allows Any Caller to Exhaust Daily Fee Minting Limit — (`contracts/FeeReceiver.sol` / `contracts/LRTOracle.sol`)

## Summary

`FeeReceiver.sendFunds()` carries no access control and can be called by any address to push accumulated MEV/EL rewards into the deposit pool. Because `getETHDistributionData()` counts `address(this).balance` of the deposit pool, this immediately inflates `totalETHInProtocol`. A subsequent permissionless call to `LRTOracle.updateRSETHPrice()` triggers fee minting that consumes some or all of `maxFeeMintAmountPerDay`, permanently blocking legitimate protocol fee collection for the remainder of that 24-hour period.

## Finding Description

**Root cause — no access control on `sendFunds()`:**

`FeeReceiver.sendFunds()` is declared `external` with no role modifier: [1](#0-0) 

Any EOA or contract can call it at any time, forwarding the entire ETH balance of `FeeReceiver` to the deposit pool via `receiveFromRewardReceiver()`: [2](#0-1) 

**TVL inflation path:**

`getETHDistributionData()` measures `ethLyingInDepositPool` as `address(this).balance`: [3](#0-2) 

This raw balance is returned through `getTotalAssetDeposits(ETH_TOKEN)` → `getAssetDistributionData(ETH_TOKEN)` → `getETHDistributionData()`, and is summed into `totalETHInProtocol` inside `_getTotalEthInProtocol()`: [4](#0-3) 

**Fee computation triggered by TVL spike:**

`updateRSETHPrice()` is `public whenNotPaused` — no role restriction: [5](#0-4) 

Inside `_updateRsETHPrice()`, whenever `totalETHInProtocol > previousTVL` and the protocol is not paused, a fee is computed: [6](#0-5) 

**Daily limit consumption:**

The computed fee is passed unconditionally to `_checkAndUpdateDailyFeeMintLimit`: [7](#0-6) 

`_checkAndUpdateDailyFeeMintLimit` increments `currentPeriodMintedFeeAmount` and reverts if the daily cap is exceeded: [8](#0-7) 

**Period-reset does not protect against this:**

The reset fires only when `block.timestamp >= feePeriodStartTime + 1 days`. An attacker who calls at the very start of a new period resets the counter to 0 and then immediately fills it with the fee derived from the pushed ETH: [9](#0-8) 

**Why existing checks are insufficient:**

The only guard on `updateRSETHPrice()` for unprivileged callers is the `pricePercentageLimit` check, which reverts if the price increase exceeds the configured threshold: [10](#0-9) 

This is not a reliable protection: (a) `pricePercentageLimit` may be 0 (disabled), (b) if the FeeReceiver balance is small relative to total TVL the price increase stays within the threshold, and (c) the attacker can observe both values off-chain and only execute when the condition is satisfied.

## Impact Explanation

The daily fee minting limit is a protocol-level cap on rsETH minted as protocol fees per 24-hour window. By exhausting it at period start, the attacker prevents the protocol from collecting fees on any additional yield accruing later in the day (staking rewards, further MEV). Those fees are **permanently lost** for that period — they are not deferred, because the `updateRSETHPrice()` call that would have minted them reverts with `DailyFeeMintLimitExceeded`. The fees minted during the attack do go to the treasury, so the attacker gains nothing financially; the harm is entirely to the protocol's fee revenue on subsequently accrued yield.

This matches the allowed impact: **High — Theft of unclaimed yield.**

## Likelihood Explanation

- Both `sendFunds()` and `updateRSETHPrice()` are callable by any address with no preconditions beyond the contract not being paused.
- `FeeReceiver` accumulates ETH passively from validator MEV/EL rewards; over a multi-day window it holds a predictable balance observable on-chain.
- The attacker only needs to monitor the FeeReceiver balance and `maxFeeMintAmountPerDay` off-chain, then call at a period boundary when the resulting `feeAmount ≤ maxFeeMintAmountPerDay` and the price increase is within `pricePercentageLimit`. This is straightforward to script.
- No privileged key, front-running dependency, or external protocol compromise is required. The attack is repeatable every 24 hours.

## Recommendation

1. **Add access control to `sendFunds()`** — restrict it to `MANAGER` or a dedicated keeper role, consistent with how other sensitive state-changing functions in the protocol are guarded (e.g., `setDepositPool` at line 66 of `FeeReceiver.sol`).
2. **Alternatively**, decouple TVL accounting from fee-minting: accumulate reward ETH in a staging variable inside the deposit pool and only recognize it during a permissioned price-update call, so an unprivileged caller cannot force fee recognition.
3. Consider whether `updateRSETHPrice()` itself should be restricted to a keeper/manager role, given that it has material economic side-effects (fee minting, price updates, potential auto-pause). The existing `updateRSETHPriceAsManager()` function already provides a permissioned variant.

## Proof of Concept

```solidity
// Foundry fork test — illustrative
function test_exhaustDailyFeeMintLimit() public {
    // 1. Simulate MEV accumulation in FeeReceiver
    vm.deal(address(feeReceiver), 10 ether);

    // 2. Warp to the start of a new 24-hour period
    vm.warp(oracle.getCurrentPeriodStartTime() + 1 days);

    // 3. Attacker (unprivileged) pushes ETH into deposit pool
    //    getETHDistributionData() will now return a higher ethLyingInDepositPool
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // 4. Attacker triggers price update — fee is minted, daily limit consumed
    //    _updateRsETHPrice: totalETHInProtocol > previousTVL => protocolFeeInETH > 0
    //    => _checkAndUpdateDailyFeeMintLimit increments currentPeriodMintedFeeAmount
    vm.prank(attacker);
    oracle.updateRSETHPrice();

    // 5. Legitimate price update later in the same period reverts
    //    (e.g., after staking rewards accrue and a keeper tries to collect fees)
    vm.expectRevert(
        abi.encodeWithSelector(ILRTOracle.DailyFeeMintLimitExceeded.selector, ...)
    );
    oracle.updateRSETHPrice();
}
```

The minimal reproducible sequence is: `vm.deal(feeReceiver, X)` → `vm.warp(periodStart + 1 days)` → `feeReceiver.sendFunds()` → `oracle.updateRSETHPrice()` (attacker) → `oracle.updateRSETHPrice()` (keeper, reverts). The value `X` must be chosen so that the resulting fee amount does not exceed `maxFeeMintAmountPerDay` and the price increase does not exceed `pricePercentageLimit`.

            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
