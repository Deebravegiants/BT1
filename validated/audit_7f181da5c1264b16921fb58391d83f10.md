### Title
Single-Step Ownership Transfer Across Multiple Ownable Contracts Removes Critical Safeguard - (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

### Summary
Multiple production contracts in the LRT-rsETH codebase inherit from `OwnableUpgradeable` (or `Ownable`) instead of `Ownable2StepUpgradeable` (or `Ownable2Step`). This means ownership can be transferred to an inoperable address in a single transaction with no confirmation step. The most severe instance is `KernelTop100MerkleDistributor`, which holds KERNEL tokens and exposes an owner-only `withdrawTokens()` recovery function — if ownership is accidentally transferred to an inoperable address, all KERNEL tokens held in the contract beyond what users can claim via the vesting schedule are permanently frozen.

### Finding Description
`KernelTop100MerkleDistributor` inherits `OwnableUpgradeable`, which implements a single-step `transferOwnership()`. There is no pending-owner confirmation step. The same pattern is repeated in `KernelMerkleDistributor`, `MerkleDistributor`, `MerkleBlastPointsDistributor`, `CrossChainRateProvider`, `CrossChainRateReceiver`, and `MultiChainRateProvider` — none of which use `Ownable2StepUpgradeable` or `Ownable2Step`. A grep across all production contracts confirms zero usage of any two-step ownership mechanism in the scope.

The critical path in `KernelTop100MerkleDistributor`:

- The contract holds KERNEL tokens that are distributed to users over a 30-day vesting schedule.
- The owner-only `withdrawTokens()` function is the sole mechanism to recover any tokens held in the contract (e.g., excess tokens, tokens sent by mistake, or tokens remaining after the vesting period ends).
- The owner-only `pause()` / `unpause()` functions are the sole emergency circuit breakers.
- The owner-only `setKernelDepositPool()`, `setProtocolTreasury()`, `setFeeInBPS()`, and `setVestingStartTimestamp()` functions control all critical protocol parameters.

If the owner calls `transferOwnership(wrongAddress)` — due to a typo, an incompatible multisig, or a contract that cannot call `acceptOwnership` — ownership is immediately and irrevocably transferred. There is no pending-owner state, no confirmation, and no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

### Impact Explanation
**Critical — Permanent freezing of funds.**

If ownership of `KernelTop100MerkleDistributor` is accidentally transferred to an inoperable address:

1. `withdrawTokens()` becomes permanently inaccessible. Any KERNEL tokens held in the contract that are not claimable by users (e.g., excess funding, tokens remaining after vesting ends, or tokens sent directly to the contract) are permanently frozen with no recovery path.
2. `pause()` / `unpause()` become permanently inaccessible, removing the emergency circuit breaker.
3. All admin configuration functions (`setKernelDepositPool`, `setProtocolTreasury`, `setFeeInBPS`, `setVestingStartTimestamp`) become permanently inaccessible, preventing any protocol parameter updates.

For `KernelMerkleDistributor` and `MerkleDistributor`, loss of ownership additionally freezes `setMerkleRoot()`, permanently preventing new reward distributions from being configured — freezing unclaimed yield for all future distribution rounds.

### Likelihood Explanation
Low-to-medium. Ownership transfers are infrequent administrative operations, but they are high-stakes. The risk is elevated because:
- The contracts are upgradeable proxies managed by a multisig or EOA, and multisig address changes are common during protocol evolution.
- A typo in a 42-character hex address, or transferring to a new multisig that has not yet been deployed or configured, would silently succeed with no recovery.
- The absence of a two-step mechanism means there is zero on-chain protection against this class of mistake.

### Recommendation
Replace `OwnableUpgradeable` with `Ownable2StepUpgradeable` in all affected upgradeable contracts, and replace `Ownable` with `Ownable2Step` in all non-upgradeable contracts:

```solidity
// KernelTop100MerkleDistributor.sol
- import { OwnableUpgradeable } from "@openzeppelin/contracts-upgradeable/access/OwnableUpgradeable.sol";
+ import { Ownable2StepUpgradeable } from "@openzeppelin/contracts-upgradeable/access/Ownable2StepUpgradeable.sol";

- contract KernelTop100MerkleDistributor is IMerkleDistributor, Initializable, OwnableUpgradeable, ...
+ contract KernelTop100MerkleDistributor is IMerkleDistributor, Initializable, Ownable2StepUpgradeable, ...

// In initialize():
- __Ownable_init();
+ __Ownable2Step_init();
```

Apply the same fix to `KernelMerkleDistributor`, `MerkleDistributor`, `MerkleBlastPointsDistributor`, `CrossChainRateProvider`, `CrossChainRateReceiver`, and `MultiChainRateProvider`.

### Proof of Concept
1. Deploy `KernelTop100MerkleDistributor` and fund it with 1,000,000 KERNEL tokens.
2. Owner calls `transferOwnership(address(0xDEAD))` — a typo or an address that cannot call `acceptOwnership`.
3. `OwnableUpgradeable.transferOwnership` immediately sets `_owner = 0xDEAD` with no confirmation step.
4. All `onlyOwner` functions (`withdrawTokens`, `pause`, `unpause`, `setKernelDepositPool`, `setProtocolTreasury`, `setFeeInBPS`, `setVestingStartTimestamp`) are now permanently inaccessible.
5. Any KERNEL tokens in the contract beyond what users can claim via the vesting schedule are permanently frozen with no recovery path. [8](#0-7) [9](#0-8)
