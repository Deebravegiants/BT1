Audit Report

## Title
Stale `rsETHPrice` in `getRsETHAmountToMint` Allows Depositors to Dilute Existing Holders' Unclaimed Yield — (`contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle` stores `rsETHPrice` as a persistent state variable that is only updated when `updateRSETHPrice()` is explicitly called. `LRTDepositPool.getRsETHAmountToMint()` divides by this stored value to compute how many rsETH tokens to mint. Between price updates, staking yield causes the true rsETH/ETH rate to rise above the stored value, so any depositor during this window receives excess rsETH not backed by ETH, permanently diluting the unclaimed yield owed to pre-existing holders.

## Finding Description

`LRTOracle` stores `rsETHPrice` as a mutable state variable: [1](#0-0) 

It is updated only when `_updateRsETHPrice()` is called via the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`: [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` uses this stored value directly as the denominator: [3](#0-2) 

Neither `depositETH` nor `depositAsset` nor `_beforeDeposit` calls `updateRSETHPrice()` before computing the mint amount: [4](#0-3) [5](#0-4) 

When yield `Y` has accrued since the last update (true price = `(S·P + Y)/S`, where `S` = rsETH supply, `P` = stale stored price), a depositor of `D` ETH receives `D/P` rsETH instead of the correct `D·S/(S·P + Y)`. The excess is `D·Y / (P·(S·P + Y))`.

When `updateRSETHPrice()` is eventually called, `previousTVL` is computed as: [6](#0-5) 

With the inflated supply from the stale-price deposit, `previousTVL = (S + D/P)·P = S·P + D`, so `rewardAmount = totalETHInProtocol − previousTVL = Y` (the full yield). However, the new price is set as `(S·P + Y + D − fee) / (S + D/P)`, which is strictly lower than the price `(S·P + Y − fee) / S` that pre-existing holders would have received without the stale deposit. The difference is `D·Y / (S·(S·P + D))`, representing yield permanently transferred from pre-existing holders to the late depositor. [7](#0-6) 

The `pricePercentageLimit` guard in `_updateRsETHPrice()` does not prevent this — it only throttles how fast the price can increase for non-manager callers, which can delay the price update and worsen the staleness window. [8](#0-7) 

## Impact Explanation

Pre-existing rsETH holders continuously accumulate yield as EigenLayer staking rewards increase protocol TVL. That yield is only crystallised into the price when `updateRSETHPrice()` is called. Any deposit made while the price is stale mints excess rsETH not backed by ETH, permanently diluting the yield owed to pre-existing holders. This is a concrete, quantifiable **theft of unclaimed yield** (High impact per the allowed scope). The dilution is irreversible once the price update occurs with the inflated supply.

## Likelihood Explanation

`updateRSETHPrice()` is not called atomically with deposits; it is an independent transaction callable by anyone. In any realistic deployment the price is updated on a schedule (e.g., daily or per-epoch). Every deposit made in the interval between two updates exploits the stale price. No special permissions, front-running, or external compromise are required — any unprivileged depositor calling `depositETH` or `depositAsset` during a stale-price window is affected. The condition is structurally guaranteed to recur with every update cycle. Likelihood is **Low** (bounded by update cadence and yield rate) but the structural recurrence makes it a reliable, repeatable condition.

## Recommendation

Call `_updateRsETHPrice()` (or an equivalent live-price computation) atomically at the start of `_beforeDeposit` before computing `rsethAmountToMint`. Alternatively, compute the mint amount on-the-fly using `_getTotalEthInProtocol() / rsethSupply` rather than the stored `rsETHPrice`, so the minting rate always reflects the current protocol state. The withdrawal path in `LRTWithdrawalManager.getExpectedAssetAmount()` should receive the same fix. [9](#0-8) 

## Proof of Concept

**Setup:** rsETH supply `S = 1000e18`, TVL = 1100 ETH, stored `rsETHPrice = 1.0e18` (true price = 1.1e18; 100 ETH yield has accrued since last update).

**Step 1:** Attacker calls `depositETH{value: 100 ether}(0, "")`.

**Step 2:** `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.0e18 = 100e18` rsETH. Correct amount at true price: `100e18 * 1e18 / 1.1e18 ≈ 90.9e18` rsETH. Attacker receives **≈ 9.1e18 excess rsETH**.

**Step 3:** Anyone calls `updateRSETHPrice()`. New supply = 1100e18, TVL = 1200 ETH. `previousTVL = 1100e18 * 1.0e18 / 1e18 = 1100 ETH`. `rewardAmount = 100 ETH`. New price = `1200e18 / 1100e18 ≈ 1.0909e18` — lower than the 1.1e18 the original holders deserved.

**Step 4:** Attacker holds 100e18 rsETH worth ≈ 109.09 ETH, having deposited 100 ETH — a risk-free gain of ≈ 9.09 ETH extracted from existing holders' unclaimed yield.

**Foundry test plan:** Deploy `LRTOracle` and `LRTDepositPool` on a local fork. Set `rsETHPrice = 1e18` with supply = 1000e18 and TVL = 1100 ETH (simulate accrued yield by crediting ETH to the deposit pool without minting rsETH). Call `depositETH{value: 100 ether}` from an unprivileged address. Assert rsETH minted > `100e18 * 1e18 / 1.1e18`. Call `updateRSETHPrice()`. Assert new price < `1100e18 / 1000e18 = 1.1e18`, confirming yield dilution.
