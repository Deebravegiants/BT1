Audit Report

## Title
Chainlink Price Feed Used Without Staleness or Validity Checks in `ChainlinkPriceOracle` - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `priceFeed.latestRoundData()` and silently discards all round metadata, accepting zero, negative, or stale prices without any validation. A degraded Chainlink feed propagates directly into rsETH minting calculations and the public `updateRSETHPrice()` function, which can corrupt the stored `rsETHPrice` for all subsequent protocol interactions.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at `contracts/oracles/ChainlinkPriceOracle.sol` L52 discards `roundId`, `updatedAt`, and `answeredInRound`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

There are no checks for `price > 0`, `updatedAt != 0`, or `answeredInRound >= roundId`. This price is consumed by two critical paths:

**Path 1 — rsETH minting:** `LRTDepositPool.getRsETHAmountToMint()` at L520 computes `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`. If `getAssetPrice` returns 0, this yields 0 rsETH minted. The `_beforeDeposit` guard at L667 only reverts if `rsethAmountToMint < minRSETHAmountExpected`; a caller passing `minRSETHAmountExpected = 0` (the common default) will have their tokens transferred in at L114 with 0 rsETH minted — permanently locking the deposited assets. [2](#0-1) [3](#0-2) 

**Path 2 — rsETHPrice corruption:** `updateRSETHPrice()` is a public, permissionless function. It calls `_getTotalEthInProtocol()`, which iterates all supported assets and calls `getAssetPrice(asset)` for each. A stale or zero price for any asset corrupts `totalETHInProtocol`, which then permanently overwrites the stored `rsETHPrice` at L313. [4](#0-3) [5](#0-4) [6](#0-5) 

The `pricePercentageLimit` guard provides partial protection only when the deviation exceeds the configured threshold and only for non-manager callers; it does not protect against stale prices within the threshold, and it defaults to 0 (disabled) if never set. [7](#0-6) 

The sibling contract `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` already implements all three required checks, confirming the protocol is aware of the pattern and intentionally applied it elsewhere: [8](#0-7) 

## Impact Explanation
**Temporary freezing of funds / Contract fails to deliver promised returns (Medium):** During any Chainlink oracle degradation event (stale round, incomplete round, zero price), every deposit processed through `depositAsset()` or `depositETH()` receives an incorrect rsETH amount. In the zero-price case, callers who do not set a non-zero `minRSETHAmountExpected` receive 0 rsETH while their deposited tokens are transferred into the pool — a permanent loss of those assets. Concurrently, any call to the public `updateRSETHPrice()` during the degraded window writes a corrupted `rsETHPrice` to storage, causing systematic mis-accounting for all subsequent deposits and withdrawals until manually corrected by an admin.

## Likelihood Explanation
No attacker action is required. The vulnerability is passively triggered whenever the Chainlink feed enters a degraded state: sequencer downtime on Arbitrum and other L2s where the protocol is deployed, node abandonment, or chain congestion. These are documented, recurring events. Any depositor transacting during the window is affected, and any caller of the public `updateRSETHPrice()` (including keeper bots) can permanently corrupt the stored price.

## Recommendation
Apply the same three checks already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    require(price > 0, "ChainlinkPriceOracle: Invalid price");
    require(updatedAt != 0, "ChainlinkPriceOracle: Incomplete round");
    require(answeredInRound >= roundId, "ChainlinkPriceOracle: Stale price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, consider adding a `heartbeat`-based staleness check (`block.timestamp - updatedAt <= heartbeat`) per feed, as `answeredInRound >= roundId` alone does not catch feeds that have simply stopped updating.

## Proof of Concept

**Scenario — Zero price corrupts deposit (permanent fund lock):**

1. Chainlink feed for a supported LST (e.g., stETH/ETH) enters an incomplete round: `latestRoundData()` returns `price = 0`, `updatedAt = 0`.
2. User calls `LRTDepositPool.depositAsset(stETH, 1e18, 0, "")` with `minRSETHAmountExpected = 0`.
3. `_beforeDeposit` → `getRsETHAmountToMint(stETH, 1e18)` → `LRTOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `0`.
4. `rsethAmountToMint = (1e18 * 0) / rsETHPrice = 0`.
5. Guard: `0 < 0` is false — no revert.
6. `safeTransferFrom` moves 1e18 stETH into the pool; `_mintRsETH(0)` mints nothing.
7. User holds 0 rsETH and cannot redeem. Funds are permanently locked.

**Scenario — Stale price corrupts rsETHPrice (systematic mis-accounting):**

1. Chainlink feed returns a stale price (e.g., 10% below current market, `answeredInRound < roundId`).
2. Any account calls `LRTOracle.updateRSETHPrice()`.
3. `_getTotalEthInProtocol()` undervalues all stETH holdings by 10%.
4. `newRsETHPrice` is computed from the deflated TVL and written to `rsETHPrice` storage.
5. All subsequent deposits use the corrupted rate, causing depositors to receive more rsETH than deserved (diluting existing holders) or fewer (depending on direction of staleness).
