### Title
`RequireGuard` Native Script with `KeyHashObj` Credential Enforces No Actual Authorization — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`)

### Summary

In the Dijkstra era, the new `RequireGuard` native script type is evaluated by checking whether a credential appears in the transaction body's `guards` field (`guardsTxBodyL`). For `KeyHashObj kh` credentials, the ledger never requires a corresponding signature from `kh`; the `guards` field is freely populated by any transaction author. Any unprivileged sender can therefore satisfy a `RequireGuard (KeyHashObj kh)` script — and spend any UTxO locked by it — without possessing or using the private key for `kh`.

### Finding Description

**Root cause — script evaluation without authorization enforcement**

`evalDijkstraNativeScript` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs` evaluates the new `RequireGuard` constructor as:

```haskell
RequireGuard cred -> cred `OSet.member` guards
```

where `guards` is the `OSet (Credential Guard)` taken directly from the transaction body field `guardsTxBodyL`. [1](#0-0) 

The `guards` field is an ordinary transaction-body field; any transaction author can place any `Credential Guard` value into it at will. [2](#0-1) 

**No witness requirement for `KeyHashObj` guards**

For `ScriptHashObj sh` entries in `guards`, `getDijkstraScriptsNeeded` adds the script hash to the set of scripts that must be provided and validated (`GuardingPurpose`), giving real authorization:

```haskell
(\sh -> (GuardingPurpose (AsIxItem idx sh), sh)) <$> credScriptHash cred
```

`credScriptHash` returns `Nothing` for `KeyHashObj`, so key-hash guard entries are **never** added to the required-scripts set and no Plutus/native-script execution is triggered for them. [3](#0-2) 

The Dijkstra UTXOW transition rule delegates witness-key computation to `getConwayWitsVKeyNeeded`, which does not include `guards` credentials:

```haskell
getWitsVKeyNeeded _ = getConwayWitsVKeyNeeded
``` [4](#0-3) 

The `dijkstraUtxowTransition` therefore never demands a signature from any key hash appearing in `guards`:

```haskell
runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody
``` [5](#0-4) 

**Contrast with `RequireSignature`**

`RequireSignature hash -> hash `Set.member` keyHashes` checks the *cryptographically verified* witness set. `RequireGuard (KeyHashObj kh)` checks only the self-declared `guards` field — an attacker-controlled input with no cryptographic binding. [6](#0-5) 

**The `allowAll`-analog: the `guards` field**

The `guards` field is the direct analog of the `allowAll` flag in the external report. The test suite explicitly confirms that arbitrary, unowned key-hash credentials may be inserted into `guards` with no rejection:

```haskell
it "A redundant guard is ignored" $ do
  guardKeyHash <- KeyHashObj <$> freshKeyHash
  let tx = mkBasicTx mkBasicTxBody & bodyTxL . guardsTxBodyL .~ [guardKeyHash]
  submitTx_ tx
``` [7](#0-6) 

And the positive test for spending a `RequireGuard (KeyHashObj kh)`-locked UTxO requires only that the key hash be declared in `guards` — no signature from `kh` is needed:

```haskell
submitTx_ $ tx & bodyTxL . guardsTxBodyL .~ [guardKeyHash]
``` [8](#0-7) 

### Impact Explanation

**Critical — Direct loss of ADA or native assets through an invalid ledger state transition.**

Any UTxO whose payment credential is a `RequireGuard (KeyHashObj kh)` native script can be spent by any unprivileged transaction sender. The attacker:

1. Observes the on-chain script (the key hash `kh` is public).
2. Constructs a transaction spending the victim's UTxO.
3. Inserts `KeyHashObj kh` into their own transaction's `guards` field.
4. Signs only with their own keys (for their own fee inputs).

The transaction passes all UTXOW checks and the victim's funds are transferred to the attacker's outputs. No key material belonging to `kh` is required. This is a direct, unconditional theft of ADA and/or native assets.

### Likelihood Explanation

**High.** The entry path requires only the ability to submit a transaction — the lowest possible privilege level. The target key hash is visible in the on-chain script. No special timing, governance majority, or privileged access is needed. The attack is deterministic and repeatable for every UTxO locked by such a script.

### Recommendation

For `KeyHashObj kh` entries in `guardsTxBodyL`, the ledger must require a corresponding VKey witness, exactly as `RequireSignature` does. Concretely, `getWitsVKeyNeeded` (or a Dijkstra-specific override) should collect all `KeyHashObj kh` values from `guardsTxBodyL` and add them to the required-witness set, so that `validateNeededWitnesses` enforces a valid signature from `kh`. Alternatively, `RequireGuard (KeyHashObj kh)` should be removed and users directed to use `RequireSignature kh` instead, since the two are semantically identical only when the signature requirement is enforced.

### Proof of Concept

```
-- Setup: Alice locks 100 ADA at a script address
guardKeyHash = KeyHashObj aliceKeyHash          -- Alice's key hash
script       = RequireGuard guardKeyHash        -- "only Alice can authorize"
scriptHash   = hashScript script
aliceUTxO    = TxIn (produce scriptHash) 0      -- 100 ADA locked here

-- Attack: Bob (no Alice key) steals the UTxO
attackTx =
  mkBasicTx mkBasicTxBody
    & bodyTxL . inputsTxBodyL    .~ [aliceUTxO]
    & bodyTxL . outputsTxBodyL   .~ [bobOutput 100_ADA]
    & bodyTxL . guardsTxBodyL    .~ [guardKeyHash]   -- declare Alice's hash; no sig needed
    -- Bob signs only for his fee input; no Alice witness present

-- Result: attackTx passes UTXOW validation
--   evalDijkstraNativeScript: RequireGuard aliceKeyHash -> aliceKeyHash ∈ {aliceKeyHash} = True
--   validateNeededWitnesses:  aliceKeyHash NOT in required set (getConwayWitsVKeyNeeded ignores guards)
--   => Transaction accepted; Alice's 100 ADA transferred to Bob
```

evalDijkstraNativeScript ::
  (DijkstraEraScript era, NativeScript era ~ DijkstraNativeScript era) =>
  Set.Set (KeyHash Witness) ->
  ValidityInterval ->
  OSet (Credential Guard) ->
  NativeScript era ->
  Bool
evalDijkstraNativeScript keyHashes (ValidityInterval txStart txExp) guards = go
  where
    -- the evaluation will stop as soon as it reaches the required number of valid scripts
    isValidMOf n SSeq.Empty = n <= 0
    isValidMOf n (ts SSeq.:<| tss) =
      n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
    go = \case
      RequireTimeStart lockStart -> lockStart `lteNegInfty` txStart
      RequireTimeExpire lockExp -> txExp `ltePosInfty` lockExp
      RequireSignature hash -> hash `Set.member` keyHashes
      RequireAllOf xs -> all go xs
      RequireAnyOf xs -> any go xs
      RequireMOf m xs -> isValidMOf m xs
      RequireGuard cred -> cred `OSet.member` guards
      _ -> error "Impossible: All NativeScripts should have been accounted for"
```
