### Title
Malicious P2P Peer Writes Empty Transaction Body for Non-Empty Block via Forged `n_transactions=0` Header — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

The transaction sync loop uses the stored header's `n_transactions` field as the sole gate for how many transactions to fetch from the network. A malicious P2P peer that supplies a `SignedBlockHeader` with `n_transactions = 0` causes the syncing node to immediately write an empty `BlockBody` for a block that actually contains transactions, with no network messages consumed and no error raised.

---

### Finding Description

**Step 1 — Header accepted without validating `n_transactions`.**

`HeaderStreamBuilder::parse_data_for_block` performs only two checks on the peer-supplied header: [1](#0-0) 

- Block number must match the expected value.
- `signatures.len()` must equal `ALLOWED_SIGNATURES_LENGTH` (1).

There is no cryptographic verification of the signature against the block hash, no check that `block_hash` is correctly derived from the header fields, and — critically — no check that `n_transactions` is consistent with `transaction_commitment` or `concatenated_counts`. The codebase itself acknowledges this gap with an explicit TODO: [2](#0-1) 

The header (including the attacker-controlled `n_transactions = 0`) is written verbatim to storage: [3](#0-2) 

**Step 2 — Transaction sync reads `n_transactions` from the stored header.**

`TransactionStreamFactory::parse_data_for_block` sets `target_transaction_len` directly from the stored header: [4](#0-3) 

**Step 3 — Loop body never executes; empty `BlockBody` is returned.**

The while loop condition is `current_transaction_len < target_transaction_len`. With `target_transaction_len = 0`, the loop is skipped entirely and the function returns `Ok(Some((block_body, block_number)))` with a default (empty) `BlockBody`: [5](#0-4) 

**Step 4 — Empty body is committed to storage.**

`write_to_storage` calls `append_body` with the empty `BlockBody` and advances the body marker: [6](#0-5) 

No error is raised, no peer is reported, and the body marker advances to `block_number + 1` as if the block were fully synced.

---

### Impact Explanation

Any RPC method that reads transaction data from storage for the affected block will return authoritative-looking wrong values:

- `starknet_getBlockTransactionCount` → returns `0` instead of the real count.
- `starknet_getTransactionByBlockIdAndIndex` → returns `INVALID_TRANSACTION_INDEX` for every index.
- `starknet_getBlockWithTxs` / `starknet_getBlockWithReceipts` → returns an empty transaction list.

This is a persistent, silent corruption: the body marker advances, so the node believes it has fully synced the block and will never re-fetch the transactions.

**Impact class**: High — RPC returns an authoritative-looking wrong value for synced block data.

---

### Likelihood Explanation

Any node reachable on the P2P network can act as a peer. The syncing node connects to peers and issues queries; a malicious peer only needs to respond to a `BlockHeadersRequest` with a crafted `SignedBlockHeader` containing `n_transactions = 0`. No validator, operator, or privileged role is required. The attack is silent (no error, no peer report) and persistent (body marker advances).

---

### Recommendation

1. **Verify `n_transactions` against `concatenated_counts`**: After receiving a header, decode `concatenated_counts` and assert that the embedded transaction count matches `n_transactions`. The `concat_counts` / `extract_*` helpers already exist in `block_hash_calculator.rs`.
2. **Verify the block hash**: Recompute the block hash from the received header fields and compare it to the peer-supplied `block_hash`. Reject headers where these differ.
3. **Cryptographically verify the block signature**: Verify the sequencer's signature over the block hash before storing the header.

---

### Proof of Concept

```
1. Spin up a mock P2P peer that responds to BlockHeadersRequest for block N
   with a SignedBlockHeader where n_transactions = 0 (and a plausible but
   unchecked block_hash / signature).

2. Connect a syncing node to this peer only.

3. Observe: parse_data_for_block in transaction.rs returns
   Ok(Some((BlockBody::default(), N))) without consuming any network message.

4. Storage now has an empty body for block N and body_marker = N+1.

5. Call starknet_getBlockTransactionCount(block_id=N) → 0.
   Call starknet_getTransactionByBlockIdAndIndex(block_id=N, index=0)
   → INVALID_TRANSACTION_INDEX.
```
