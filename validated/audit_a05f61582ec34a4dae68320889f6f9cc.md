### Title
Block-Level Reference Script Size Limit Bypassed via Sub-Transaction Reference Inputs — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

### Summary

`dijkstraBbodyTransition` delegates the block-level reference script size check to the Conway-era `validateBodyRefScriptsSizeTooBig`, which internally calls `totalRefScriptSizeInBlock` → `txNonDistinctRefScriptsSize`. That function only inspects the top-level transaction's `referenceInputsTxBodyL` and `inputsTxBodyL`. It never recurses into the sub-transactions embedded in a Dijkstra `TxBody TopTx`. The Dijkstra-specific per-transaction check (`validateAllRefScriptSize`) correctly uses `batchNonDistinctRefScriptsSize`, which sums both levels. The per-block check does not. A block producer can therefore include an arbitrarily large aggregate of sub-transaction reference scripts in a single block while the block-level guard reports zero bytes consumed.

### Finding Description

**Root cause — `totalRefScriptSizeInBlock` is sub-tx-blind:**

`totalRefScriptSizeInBlock` iterates over `StrictSeq (Tx TopTx era)` and calls `txNonDistinctRefScriptsSize` on each top-level tx:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs:357-370
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 = ...
  | otherwise = snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      ...
      in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
```

`txNonDistinctRefScriptsSize` only reads the single-level inputs of the tx it receives:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs:183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

Sub-transactions are stored in `dtbrSubTransactions :: OMap TxId (Tx SubTx era)` inside `DijkstraTxBodyRaw TopTx era`. Each `Tx SubTx era` has its own `dstbrReferenceInputs` and `dstbrSpendInputs`. Neither field is visited by `txNonDistinctRefScriptsSize`.

**The Dijkstra-aware function exists but is not used here:**

`batchNonDistinctRefScriptsSize` was introduced precisely to cover both levels:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs:263-277
-- | Total size of reference scripts across a top-level transaction and all its subtransactions.
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

The per-transaction guard in `dijkstraLedgerTransition` uses it correctly:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs:321-322
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
```

But `dijkstraBbodyTransition` calls the Conway function unchanged:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs:363
Conway.validateBodyRefScriptsSizeTooBig @era pp blockBody (ls ^. utxoL)
```

which routes through `totalRefScriptSizeInBlock` → `txNonDistinctRefScriptsSize`, never touching sub-tx inputs.

**Asymmetry table:**

| Check | Function used | Counts sub-tx ref scripts? |
|---|---|---|
| Per-tx (`LEDGER`) | `batchNonDistinctRefScriptsSize` | Yes |
| Per-block (`BBODY`) | `txNonDistinctRefScriptsSize` | **No** |

### Impact Explanation

The block-level cap (`ppMaxRefScriptSizePerBlock`, currently 1 MiB) was introduced specifically to bound the total script-loading and validation work per block and prevent resource exhaustion. Because the cap is evaluated only over top-level reference inputs, a block producer can fill a block with top-level txs whose top-level reference inputs are empty (or minimal) while each tx embeds sub-txs that collectively reference hundreds of KiB of scripts. The per-tx cap (`ppMaxRefScriptSizePerTx`, 200 KiB) still applies per top-level tx (including its sub-txs), but with N txs in a block the aggregate sub-tx reference script bytes can reach N × 200 KiB — far above the 1 MiB block limit — without triggering `BodyRefScriptsSizeTooBig`. Validators must still load and validate every referenced script, so the intended DoS protection of the block-level limit is nullified for the sub-transaction dimension.

This matches the **Medium** impact category: attacker-controlled blocks exceed intended validation limits.

### Likelihood Explanation

The attacker must be a block producer (stake pool operator), which requires staking ADA and winning a slot. This is a legitimate, accessible role. The exploit requires no special knowledge beyond constructing Dijkstra transactions with sub-transactions, which is a documented feature. The discrepancy is mechanical and deterministic — any block with the described structure will pass the block-level check regardless of the aggregate sub-tx reference script size.

### Recommendation

Replace the call to `Conway.validateBodyRefScriptsSizeTooBig` in `dijkstraBbodyTransition` with a Dijkstra-specific variant that uses `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize` when accumulating the per-block total. Concretely, `totalRefScriptSizeInBlock` (or a Dijkstra override of it) should fold over each top-level tx using `batchNonDistinctRefScriptsSize` so that sub-transaction reference inputs are included in the block-level accounting, consistent with how `validateAllRefScriptSize` already handles the per-tx limit.

### Proof of Concept

1. Pre-create UTxOs whose outputs carry large Plutus reference scripts (e.g., 100 KiB each).
2. Construct a Dijkstra `TxBody TopTx` with zero top-level reference inputs but with one or more `SubTx` entries, each referencing several of those UTxOs. Keep the total per-tx (top + sub) below `ppMaxRefScriptSizePerTx` (200 KiB).
3. Repeat for enough top-level txs so that the aggregate sub-tx reference script bytes exceed `ppMaxRefScriptSizePerBlock` (1 MiB).
4. Package all top-level txs into a single block and submit it.
5. Observe: `validateAllRefScriptSize` passes for each individual tx (sub-tx scripts counted). `validateBodyRefScriptsSizeTooBig` also passes because `totalRefScriptSizeInBlock` sums only top-level reference inputs (≈ 0 bytes). The block is accepted despite the aggregate sub-tx reference script size exceeding the block limit.
6. Assert: `sum (map batchNonDistinctRefScriptsSize txs) > ppMaxRefScriptSizePerBlock` while `totalRefScriptSizeInBlock txs utxo <= ppMaxRefScriptSizePerBlock`.
