The key facts are confirmed:

1. `swap` uses `nonReentrant(PoolActions.SWAP)` (line 224), which sets a transient storage flag during execution.
2. `_executeSwap` writes updated `binTotals` to persistent storage (lines 735–739) **before** the callback at line 258.
3. `Extsload` has **no** `nonReentrantView` modifier on any of its three `extsload` functions.
4. `nonReentrantView` is defined in `MetricReentrancyGuardTransient` but has **zero usages** anywhere in the codebase — it was never applied to `Extsload`.
5. Same pattern holds for `addLiquidity` via `LiquidityLib` (lines 134–148).

Audit Report

## Title
Read-Only Reentrancy via Unguarded `Extsload` During Swap and `addLiquidity` Callback Windows Exposes Inflated `binTotals` State — (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/Extsload.sol`)

## Summary
`MetricOmmPool` writes updated `binTotals` to persistent storage inside `_executeSwap` and `LiquidityLib.addLiquidity` before issuing the caller's callback and before receiving input tokens. The `Extsload` contract, inherited by the pool and used as the documented external state-read path, carries no reentrancy guard. During the callback window, any external call to `extsload` reads `binTotals.scaledToken0` (or `scaledToken1`) that is inflated by the full pending input amount, enabling exploitation of any external protocol that uses this data for financial decisions.

## Finding Description
**Swap path (`zeroForOne = true`):**

`_executeSwap` is called at line 248 and immediately writes post-trade accounting to persistent storage:

```solidity
// MetricOmmPool.sol L735-739
binTotals.scaledToken0 =
  (uint256(binTotals.scaledToken0) + uint256(amount0DeltaScaled) - protocolFeeScaled).toUint128();
binTotals.scaledToken1 = uint128(uint256(binTotals.scaledToken1) - uint256(-amount1DeltaScaled));
```

After `_executeSwap` returns, the pool sends token1 out and snapshots `balance0Before`, then fires the callback at line 258. At this exact moment `binTotals.scaledToken0` is inflated by the expected input that has not yet arrived, while `balance0()` is unchanged.

**`addLiquidity` path:**

`LiquidityLib.addLiquidity` updates `binTotals.scaledToken0` and `binTotals.scaledToken1` at lines 135–138, then fires `metricOmmModifyLiquidityCallback` at line 148 before verifying token receipt at lines 149–154. The same window exists.

**`Extsload` is unguarded:**

`MetricReentrancyGuardTransient` defines `nonReentrantView` precisely to block view-path reads while any guarded action is active:

```solidity
// MetricReentrancyGuardTransient.sol L24-33
modifier nonReentrantView() {
    _nonReentrantBeforeView();
    _;
}
function _nonReentrantBeforeView() private view {
    if (_currentAction() != 0) { revert ReentrancyGuardReentrantCall(); }
}
```

However, `nonReentrantView` has **zero usages** in the entire codebase — it is never applied to any of the three `extsload` overloads in `Extsload.sol`. The `nonReentrant(PoolActions.SWAP)` modifier on `swap` sets the transient guard flag, but `Extsload` never checks it. An attacker calling `extsload` during the callback window reads stale-inflated storage with no revert.

The pool's own documentation confirms `PoolStateLibrary` mirrors the storage layout for EXTSLOAD-based reads used by periphery quoters, making this the intended external integration path.

## Impact Explanation
An attacker controlling the swap callback can read inflated `binTotals.scaledToken0` via `extsload`, present it to an external lending protocol that uses Metric OMM pool state as a collateral oracle, borrow against the phantom collateral, and use the borrowed funds to satisfy the callback payment. The pool's own `IncorrectDelta` check passes normally; the loss falls entirely on the external protocol's depositors. This constitutes a direct loss of user principal enabled by a broken invariant in the pool's external state-read interface — Medium severity.

## Likelihood Explanation
- Trigger is unprivileged: any caller can initiate a swap or `addLiquidity`.
- The window opens on every swap and every `addLiquidity` call; no special pool configuration is required.
- `Extsload` / `PoolStateLibrary` is the documented and intended external read path for periphery quoters and integrators, making it a natural target for co-deployed lending protocols.
- Exploitation requires a co-deployed external protocol that reads Metric OMM pool state for financial decisions — the expected production integration pattern — placing likelihood at Medium.

## Recommendation
1. Apply `nonReentrantView` to all three `extsload` overloads in `Extsload.sol` so that storage reads revert while any guarded action (`swap`, `addLiquidity`, etc.) is active. The modifier is already implemented and only needs to be used.
2. Alternatively, restructure swap and liquidity flows to receive input tokens **before** updating `binTotals` (receive → update accounting → verify), eliminating the inconsistency window entirely.
3. Document in `PoolStateLibrary` and all periphery integrations that pool state read via EXTSLOAD must not be consumed inside a callback context.
