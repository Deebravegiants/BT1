### Title
No Fund Recovery Mechanism in KernelMerkleDistributor Allows Permanent Token Freeze on Misconfiguration - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

---

### Summary

`KernelMerkleDistributor.sol` holds KERNEL tokens for distribution to users via Merkle proofs but contains **no fund recovery, clawback, or emergency withdrawal function**. If the Merkle root is misconfigured (e.g., wrong hash set via `setMerkleRoot`), all KERNEL tokens deposited into the contract become permanently unrecoverable, as there is no expiration mechanism and no owner-callable withdrawal path.

---

### Finding Description

`KernelMerkleDistributor` is an upgradeable contract that receives KERNEL tokens and distributes them to eligible users who submit valid Merkle proofs. The owner can update the Merkle root via `setMerkleRoot`, pause/unpause the contract, and update administrative parameters — but the contract exposes **zero fund recovery capability**.

The complete set of admin functions is:
- `setKernelDepositPool` [1](#0-0) 
- `setProtocolTreasury` [2](#0-1) 
- `setFeeInBPS` [3](#0-2) 
- `setMerkleRoot` [4](#0-3) 
- `pause` / `unpause` [5](#0-4) 

None of these allow the owner to withdraw or recover KERNEL tokens held by the contract. There is no `withdrawTokens`, no `clawback`, no `emergencyWithdraw`, and no expiration timestamp after which recovery becomes possible. [6](#0-5) 

The `_processClaim` internal function is the only path through which tokens leave the contract — exclusively to valid Merkle proof claimants. [7](#0-6) 

This stands in direct contrast to the sibling contract `KernelTop100MerkleDistributor.sol`, which explicitly implements a `withdrawTokens` admin function for exactly this recovery scenario. [8](#0-7) 

---

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

If a wrong Merkle root is set (e.g., a hash computed from an incorrect dataset, a typo, or a stale snapshot), no user can produce a valid proof against it. All KERNEL tokens held in the contract at that point are permanently frozen: users cannot claim them, and the owner has no mechanism to recover them. Even correcting the Merkle root via `setMerkleRoot` does not help — the tokens remain locked because the only egress path is through `claim`/`claimAndStake`, which require a valid proof against the *current* root. [9](#0-8) 

---

### Likelihood Explanation

Merkle root misconfiguration is a realistic operational risk: off-chain snapshot scripts can include wrong addresses, wrong amounts, or be run against a stale state. The protocol already acknowledges this risk class by implementing `withdrawTokens` in `KernelTop100MerkleDistributor`. The omission in `KernelMerkleDistributor` is an inconsistency that makes the misconfiguration scenario irreversible. [8](#0-7) 

---

### Recommendation

Add an owner-callable fund recovery function to `KernelMerkleDistributor`, mirroring the pattern already used in `KernelTop100MerkleDistributor`:

```solidity
function withdrawTokens(address _token, uint256 _amount, address _recipient) external onlyOwner {
    UtilLib.checkNonZeroAddress(_token);
    UtilLib.checkNonZeroAddress(_recipient);
    if (_amount == 0) revert ZeroValueProvided();
    IERC20(_token).safeTransfer(_recipient, _amount);
    emit TokensWithdrawn(_token, _amount, _recipient);
}
```

Optionally, consider adding a grace-period guard (e.g., only callable before the first claim or within N days of deployment) to limit the owner's ability to rug claimants while still allowing recovery from misconfiguration. [8](#0-7) 

---

### Proof of Concept

1. Owner deploys `KernelMerkleDistributor` and funds it with 1,000,000 KERNEL tokens.
2. Owner calls `setMerkleRoot(badRoot)` where `badRoot` was computed from an incorrect off-chain snapshot. [4](#0-3) 
3. All users attempt `claim(...)` — every call reverts with `InvalidMerkleProof` because no valid proof exists for `badRoot`. [9](#0-8) 
4. Owner calls `setMerkleRoot(correctRoot)` to fix the root — but the 1,000,000 KERNEL tokens are still in the contract and can only exit via `claim`/`claimAndStake`.
5. Owner searches for any admin withdrawal function — none exists. [6](#0-5) 
6. If the distribution campaign is abandoned (e.g., replaced by a new contract), all remaining KERNEL tokens are permanently frozen with no recovery path.

```text
    function _processClaim(
        uint256 index,
        address account,
        uint256 cumulativeAmount,
        bytes32[] calldata merkleProof
    )
        internal
        returns (uint256)
    {
        UtilLib.checkNonZeroAddress(account);

        if (currentMerkleRoot == bytes32(0)) {
            revert ZeroValueProvided();
        }

        if (index == 0 || index > currentIndex) {
            revert InvalidIndex();
        }

        if (account != msg.sender) {
            revert Unauthorized();
        }

        if (isClaimed(index, account)) {
            revert AlreadyClaimed();
        }

        // Verify the merkle proof
        bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
        if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
            revert InvalidMerkleProof();
        }

        // Calculate the claimable amount
        uint256 claimableAmount = cumulativeAmount - userClaims[account].cumulativeAmount;

        // Ensure there is something to claim
        if (claimableAmount == 0) {
            revert NoTokensToClaim();
        }

        // Update user claim info
        userClaims[account].lastClaimedIndex = index;
        userClaims[account].cumulativeAmount = cumulativeAmount;

        // Calculate the fee and the amount to send
        uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;
        uint256 amountToSend = claimableAmount - fee;

        if (fee > 0) {
            kernel.safeTransfer(protocolTreasury, fee);
        }

        return amountToSend;
    }
```

```text
    /**
     * @notice Sets the KernelDepositPool contract address
     * @param _kernelDepositPool The address of the new KernelDepositPool contract
     */
    function setKernelDepositPool(address _kernelDepositPool) external onlyOwner {
        UtilLib.checkNonZeroAddress(_kernelDepositPool);

        address oldKernelDepositPool = address(kernelDepositPool);
        kernelDepositPool = IKernelDepositPool(_kernelDepositPool);

        // Revoke the approval of the old KernelDepositPool contract to spend KERNEL tokens on behalf of this contract
        kernel.forceApprove(oldKernelDepositPool, 0);

        // Approve the KernelDepositPool contract to spend an unlimited amount of KERNEL tokens on behalf of this
        // contract
        kernel.forceApprove(_kernelDepositPool, type(uint256).max);

        emit KernelDepositPoolUpdated(_kernelDepositPool);
    }

    /**
     * @notice Sets the protocol treasury address
     * @param _protocolTreasury The address of the new protocol treasury
     */
    function setProtocolTreasury(address _protocolTreasury) external onlyOwner {
        UtilLib.checkNonZeroAddress(_protocolTreasury);

        protocolTreasury = _protocolTreasury;

        emit ProtocolTreasuryUpdated(protocolTreasury);
    }

    /**
     * @notice Sets the fee in basis points
     * @param _feeInBPS The new fee in basis points
     */
    function setFeeInBPS(uint256 _feeInBPS) external onlyOwner {
        if (_feeInBPS > MAX_FEE_IN_BPS) {
            revert InvalidFeeInBPS();
        }

        feeInBPS = _feeInBPS;

        emit FeeInBPSUpdated(feeInBPS);
    }

    /**
     * @notice Sets the new merkle root
     * @param _merkleRootToSet The new merkle root to be set
     */
    function setMerkleRoot(bytes32 _merkleRootToSet) external onlyOwner {
        if (_merkleRootToSet == bytes32(0)) {
            revert ZeroValueProvided();
        }

        currentMerkleRoot = _merkleRootToSet;

        currentMerkleRootIndex++;
        currentIndex++;

        emit MerkleRootSet(currentMerkleRootIndex, currentMerkleRoot);
    }
```
