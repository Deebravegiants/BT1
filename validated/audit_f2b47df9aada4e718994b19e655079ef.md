### Title
Adversary Can Saturate `verify_foreign_transaction` Fan-Out Queue to Permanently Block Legitimate Bridge Verifications - (File: `crates/contract/src/pending_requests.rs`)

### Summary

The `verify_foreign_transaction` fan-out queue uses a caller-agnostic request key (`VerifyForeignTransactionRequest` = `{ForeignChainRpcRequest, domain_id, payload_version}`). Any unprivileged account can submit 128 identical requests for the same foreign transaction, filling the bounded queue to its cap. Once full, every subsequent legitimate caller (e.g., a bridge service) receives `PendingRequestQueueFull` and is rejected. The adversary can continuously re-saturate the queue after each drain cycle, permanently blocking bridge inbound flows for targeted transactions at negligible cost (128 yoctonear ≈ $0 per cycle).

### Finding Description

The `pending_requests::push_pending_yield` function enforces a hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` on the number of yield-resume promises queued per request key:

```rust
// crates/contract/src/pending_requests.rs
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where ...
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
}
```

For `sign()`, the queue key is `SignatureRequest` which is constructed with the caller's `predecessor` account ID, so different callers cannot collide on the same key:

```rust
// crates/contract/src/lib.rs:379-384
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // caller-specific
    &request.path,
);
```

For `verify_foreign_transaction()`, the queue key is `VerifyForeignTransactionRequest`, which contains only `{ForeignChainRpcRequest, domain_id, payload_version}` — **no caller account ID**:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

The contract itself confirms this is intentional — the test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` explicitly shows alice and bob sharing the same queue slot for the same foreign transaction. However, this design means **any unprivileged account can occupy all 128 slots** for any specific foreign transaction.

Attack path:
1. Adversary observes a pending or predictable foreign-chain transaction (e.g., a large Bitcoin inbound transfer to a bridge).
2. Adversary submits 128 identical `verify_foreign_transaction` calls for that transaction (cost: 128 yoctonear ≈ $0).
3. Queue for that `VerifyForeignTransactionRequest` key is saturated.
4. Bridge service submits its own `verify_foreign_transaction` for the same transaction → `PendingRequestQueueFull` panic → transaction fails.
5. MPC nodes eventually respond, draining the queue (adversary's 128 yields all receive the signature).
6. Adversary immediately re-saturates the queue before the bridge can retry.
7. Bridge inbound flow for that transaction is indefinitely blocked.

### Impact Explanation

The `verify_foreign_transaction` flow is the sole on-chain mechanism for the MPC network to attest that a foreign-chain transaction finalized. Bridge services depend on it to release NEAR-side funds. An adversary who continuously saturates the queue for a targeted transaction prevents the bridge from ever obtaining the MPC attestation, causing the user's foreign-chain funds to remain locked indefinitely. This breaks the bridge's core safety invariant: "a finalized foreign transaction must be processable." The adversary can profit by short-selling the bridged asset, extorting the user, or blocking a competitor's bridge flow.

### Likelihood Explanation

The attack requires no privileged access, no threshold collusion, and no key material. The cost is 128 yoctonear per drain cycle (effectively free). Foreign transaction IDs are public on-chain, so the adversary can predict or observe any pending bridge verification. The attack is trivially scriptable and can be maintained indefinitely.

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key (analogous to how `SignatureRequest` includes `predecessor`). Alternatively, enforce a per-account submission limit per request key so a single account cannot occupy more than a small fraction of the 128 slots.

### Proof of Concept

The existing test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` already demonstrates that alice and bob share the same queue slot. Extending it to 128 identical submissions from a single adversary account would trigger `PendingRequestQueueFull` on the 129th call, blocking any subsequent legitimate caller.

```
// Adversary fills queue:
for _ in 0..128 {
    adversary.call("verify_foreign_transaction")
        .args_json({"request": target_bitcoin_tx_request})
        .deposit(1)
        .transact();
}

// Bridge service is now blocked:
bridge.call("verify_foreign_transaction")
    .args_json({"request": target_bitcoin_tx_request})
    .deposit(1)
    .transact();
// → panics: "Pending-request queue is full for this request key (limit: 128)"
```
