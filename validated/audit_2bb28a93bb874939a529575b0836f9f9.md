### Title
Caller-Controlled `confirmations: 0` Bypasses Bitcoin Finality Check, Enabling Double-Spend via Forged Foreign-Chain Attestation - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

An unprivileged caller can submit a `verify_foreign_transaction` request for a Bitcoin transaction with `confirmations: 0`. Neither the on-chain contract nor the node-side inspector enforces a minimum confirmation threshold. The MPC network will sign a payload attesting to an unconfirmed (mempool) transaction, which can then be used to claim bridge funds on NEAR before the Bitcoin transaction is confirmed — or after it is reorganized away — enabling a double-spend.

---

### Finding Description

The `verify_foreign_transaction` entry point accepts a `BitcoinRpcRequest` whose `confirmations` field is a plain `u64` with no enforced minimum:

```rust
pub struct BitcoinRpcRequest {
    pub tx_id: BitcoinTxId,
    pub confirmations: BlockConfirmations, // required confirmations before considering final
    pub extractors: Vec<BitcoinExtractor>,
}
``` [1](#0-0) 

The ABI schema explicitly allows `minimum: 0.0` for `BlockConfirmations`: [2](#0-1) 

The contract's `verify_foreign_transaction` method performs no validation of the `confirmations` value — it only checks domain purpose, gas, deposit, and chain support: [3](#0-2) 

On the node side, `BitcoinInspector::extract` enforces the threshold with a simple `<=` comparison:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [4](#0-3) 

When `block_confirmations_threshold` is `0`, the condition `0 <= any_value` is always `true`. Bitcoin's `getrawtransaction` RPC returns mempool transactions with `confirmations: 0`, so the check passes unconditionally for any transaction that exists in the mempool. The MPC nodes then proceed to sign the `ForeignTxSignPayload` containing `(request, observed_values, observed_at)` for this unconfirmed transaction.

---

### Impact Explanation

The signed MPC attestation is indistinguishable from one produced for a fully-confirmed transaction. Any NEAR bridge contract (e.g., Omnibridge inbound flow) that consumes `VerifyForeignTransactionResponse` to release funds will accept this signature. The attacker can:

1. Broadcast a Bitcoin transaction sending funds to the bridge deposit address.
2. Immediately call `verify_foreign_transaction` with `confirmations: 0` before any block confirmation.
3. Receive the MPC-signed attestation and redeem NEAR-side bridge funds.
4. Double-spend the Bitcoin transaction via RBF or a miner-assisted reorg.

The bridge has released funds for a Bitcoin transaction that no longer exists on the canonical chain. This matches the allowed impact: **"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."**

---

### Likelihood Explanation

The attack requires only a 1 yoctoNEAR deposit and a valid NEAR account — no privileged access, no collusion, no TEE bypass. The `confirmations` field is fully caller-controlled and the contract imposes no floor. Any user of the `verify_foreign_transaction` API can exploit this. Bitcoin mempool transactions are routinely accessible via public RPC nodes, making the precondition trivially satisfiable.

---

### Recommendation

1. **Enforce a minimum confirmation threshold at the contract level.** Reject any `BitcoinRpcRequest` where `confirmations == 0` inside `verify_foreign_transaction`, or define a protocol-wide minimum (e.g., `>= 1`, with bridge-specific guidance recommending 6).
2. **Enforce a minimum at the inspector level as a defense-in-depth measure.** `BitcoinInspector::extract` should return `ForeignChainInspectionError` if `block_confirmations_threshold == 0`.
3. **Consider a governance-controlled per-chain minimum confirmation floor** stored in the on-chain foreign-chain configuration, analogous to how the external report's mitigation used a Chainlink-backed slippage floor that operators cannot bypass.

---

### Proof of Concept

**Attacker call (NEAR CLI):**
```json
{
  "request": {
    "domain_id": 1,
    "payload_version": "V1",
    "request": {
      "Bitcoin": {
        "tx_id": "<mempool_tx_id_hex>",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    }
  }
}
```

**Node-side execution path:**

1. `verify_foreign_transaction` accepts the request — no `confirmations` floor check. [5](#0-4) 

2. MPC node calls `BitcoinInspector::extract(tx_id, BlockConfirmations(0), extractors)`. [6](#0-5) 

3. Inspector fetches the mempool transaction (confirmations = 0 from RPC), evaluates `0 <= 0 → true`, proceeds to sign. [7](#0-6) 

4. MPC returns a valid `VerifyForeignTransactionResponse` with a threshold signature over the unconfirmed transaction payload. The attacker redeems bridge funds, then double-spends the Bitcoin transaction.

        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config

        );

        let callback_args = serde_json::to_vec(&(&request,)).unwrap();

            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
