### Title
`MetadataPayload::to_borsh()` Omits `chain_id`, Enabling Cross-Deployment Replay of `deploy_token` Signatures — (`starknet/src/bridge_types.cairo`)

---

### Summary

`MetadataPayload::to_borsh()` produces a chain-agnostic byte sequence. A valid MPC-signed `deploy_token` signature obtained for one Omni Bridge StarkNet deployment can be submitted verbatim to any other deployment that shares the same `omni_bridge_derived_address` (i.e., the same MPC key), causing an unauthorized wrapped token to be registered on the second chain.

---

### Finding Description

`MetadataPayload::to_borsh()` encodes only five fields:

```
[PayloadType::Metadata (1 byte)] [token] [name] [symbol] [decimals]
``` [1](#0-0) 

No `chain_id` is included. Compare this to `TransferMessagePayload::to_borsh(chain_id: u8)`, which explicitly receives and embeds `chain_id` **twice** in its encoding: [2](#0-1) 

The protocol's own design for `fin_transfer` demonstrates that chain-specific binding is intentional — it is simply absent from the metadata path.

`deploy_token` calls `payload.to_borsh()` with no chain argument and passes the result directly to `_verify_borsh_signature`: [3](#0-2) 

`_verify_borsh_signature` computes `keccak(borsh_bytes)` and verifies it against `self.omni_bridge_derived_address`: [4](#0-3) 

Because `omni_bridge_derived_address` is the Ethereum address derived from the global MPC key — shared across all Omni Bridge deployments — any two deployments with different `omni_bridge_chain_id` values but the same MPC key will accept each other's `deploy_token` signatures without modification.

---

### Impact Explanation

An unprivileged relayer who observes a valid `deploy_token(sig, payload)` on chain A can submit the identical `(sig, payload)` to chain B's `deploy_token`. The borsh bytes are identical (no chain binding), the keccak hash is identical, and the MPC address is identical, so `_verify_borsh_signature` passes. The token is then registered in `near_to_starknet_token` on chain B: [5](#0-4) 

Consequences:
1. **Preemptive slot occupation**: Once registered, any future legitimate `deploy_token` for the same NEAR token ID on chain B will revert with `ERR_TOKEN_ALREADY_DEPLOYED` (line 209), permanently blocking the protocol-authorized deployment path for that token on chain B.
2. **Unauthorized wrapped asset existence**: The wrapped token contract exists on chain B without the NEAR side having authorized it for chain B. If the NEAR side later routes `fin_transfer` messages to chain B for this token, minting flows against this contract — which was deployed without chain-specific MPC consent.

---

### Likelihood Explanation

The precondition — two deployments sharing the same `omni_bridge_derived_address` — is the normal operating state of the Omni Bridge protocol. The MPC key is global; every chain deployment uses the same derived Ethereum address. Any `deploy_token` event on any chain is publicly observable on-chain. No privileged access, key compromise, or MPC collusion is required. The attacker only needs to copy a transaction.

---

### Recommendation

Add `chain_id` to `MetadataPayload::to_borsh()`, mirroring the pattern already used by `TransferMessagePayload::to_borsh`:

```rust
fn to_borsh(self: @MetadataPayload, chain_id: u8) -> ByteArray {
    let mut borsh_bytes: ByteArray = Default::default();
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append_byte(chain_id);          // <-- add this
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
```

Update `deploy_token` to pass `self.omni_bridge_chain_id.read()` to `to_borsh`, exactly as `fin_transfer` does at line 253. Ensure the NEAR-side MPC signing logic also includes `chain_id` in the metadata payload it signs.

---

### Proof of Concept

1. Deploy bridge instance A with `omni_bridge_chain_id = 10`, bridge instance B with `omni_bridge_chain_id = 11`; both use the same MPC key → same `omni_bridge_derived_address`.
2. Obtain a valid MPC-signed `deploy_token` for token `"usdc.near"` on chain A. Record `(sig, payload)`.
3. Submit the identical `deploy_token(sig, payload)` to chain B.
4. `payload.to_borsh()` produces the same bytes on both chains (no `chain_id` field).
5. `_verify_borsh_signature` passes on chain B.
6. Assert `near_to_starknet_token[keccak("usdc.near")]` is non-zero on chain B — token is deployed without chain-B-specific MPC authorization.
7. Any subsequent legitimate `deploy_token` for `"usdc.near"` on chain B reverts with `ERR_TOKEN_ALREADY_DEPLOYED`.
