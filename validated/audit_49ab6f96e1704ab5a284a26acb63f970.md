Audit Report

## Title
`LRTConverter.ethValueInWithdrawal` Uses Stale Entry-Time Price, Under-Reporting Total ETH in Protocol - (File: contracts/LRTConverter.sol)

## Summary
`LRTConverter` stamps the ETH value of transferred LST assets at the oracle price at the moment of transfer and stores it in `ethValueInWithdrawal`, which is never refreshed during the Lido withdrawal queue period. Because yield-bearing LSTs such as stETH continuously appreciate in ETH terms, `ethValueInWithdrawal` chronically under-reports the true ETH value of assets sitting in the converter. This causes `_getTotalEthInProtocol()` in `LRTOracle` to return a deflated figure, setting `rsETHPrice` lower than it should be and allowing new depositors to mint more rsETH than warranted, diluting existing rsETH holders.

## Finding Description
When an operator calls `transferAssetFromDepositPool`, the converter records the ETH value of the transferred LST at the current oracle price:

```solidity
// LRTConverter.sol L140
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
```

This value is never refreshed. Critically, `getAssetDistributionData` in `LRTDepositPool` explicitly zeroes out the converter's LST contribution for per-asset accounting:

```solidity
// LRTDepositPool.sol L460
assetLyingInConverter = 0; // assets in converter are accounted in their eth value => getETHDistributionData()
```

The converter's entire contribution to total protocol ETH therefore flows exclusively through `ethValueInWithdrawal` via `getETHDistributionData`:

```solidity
// LRTDepositPool.sol L498-499
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
```

`_getTotalEthInProtocol()` in `LRTOracle` iterates over all supported assets and calls `getTotalAssetDeposits(asset)`. For the ETH asset, this resolves to `getETHDistributionData()`, which includes the stale `ethValueInWithdrawal`. For LST assets, `assetLyingInConverter` is always 0. The stale ETH value then feeds directly into `rsETHPrice`:

```solidity
// LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

`_sendEthToDepositPool` reduces `ethValueInWithdrawal` by the actual ETH amount received from Lido, not by the original stamped value. Since stETH appreciates, the actual ETH received exceeds the stamped value, and `ethValueInWithdrawal` is floored to 0. However, this correction only occurs at claim time — during the entire Lido withdrawal queue window (typically 1–7 days), the growing ETH value of the stETH held in the converter is invisible to the oracle.

## Impact Explanation
`rsETHPrice` is computed from a deflated `totalETHInProtocol`. A lower `rsETHPrice` causes `getRsETHAmountToMint` to mint more rsETH per unit of deposited asset than is warranted:

```solidity
// LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Depositors who enter while stETH is sitting in the converter receive a slightly inflated rsETH allocation, diluting the claims of all existing rsETH holders for the duration of the unstaking window. This matches the allowed impact: **Low — contract fails to deliver promised returns to existing rsETH holders, but no direct fund theft occurs.**

## Likelihood Explanation
stETH accrues rewards continuously (~3–4% APR). Any time the operator routes stETH through the converter for Lido withdrawal — a routine, expected operational step — the discrepancy is active for the full withdrawal queue duration. No attacker action is required; the condition arises from normal protocol operations. Any depositor calling `depositAsset` during this window benefits at the expense of existing holders. This is a recurring, high-likelihood condition.

## Recommendation
Replace the static entry-time ETH value with a live calculation of the current ETH value of LST tokens held in the converter. Instead of storing `ethValueInWithdrawal` as a fixed ETH amount, track raw LST token balances and compute their current ETH value on-the-fly using the oracle:

```solidity
// Track raw token balances
mapping(address => uint256) public tokenAmountsInWithdrawal;

// On transfer in:
tokenAmountsInWithdrawal[_asset] += _amount;

// ethValueInWithdrawal() view function:
for each asset: total += tokenAmountsInWithdrawal[asset] * oracle.getAssetPrice(asset) / 1e18;
```

This ensures the converter's contribution to `totalETHInProtocol` always reflects the current market value of the LSTs held, eliminating the stale-price window.

## Proof of Concept
1. Operator calls `transferAssetFromDepositPool(stETH, 1000e18)` when stETH price = 1.00 ETH.
   - `ethValueInWithdrawal` = 1000 ETH (stamped at L140).
2. Seven days pass; stETH accrues ~0.077% yield; price = 1.00077 ETH.
   - Actual ETH value of 1000 stETH in converter = 1000.77 ETH.
   - `ethValueInWithdrawal` still = 1000 ETH (stale, never refreshed).
3. `_getTotalEthInProtocol()` under-counts by 0.77 ETH because `assetLyingInConverter = 0` for stETH (L460) and `ethLyingInConverter` reads the stale value (L499).
4. `updateRSETHPrice()` is called (publicly callable); `rsETHPrice` is set ~0.077% lower than correct.
5. A depositor calling `depositAsset` during this window receives ~0.077% more rsETH than warranted via L519-520, diluting all existing rsETH holders by that margin for the duration of the unstaking window.
