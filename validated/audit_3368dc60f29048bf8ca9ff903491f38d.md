### Title
Missing `state_diff_commitment` Verification in P2P State Diff Sync Allows Malicious Peer to Corrupt Stored State Diffs — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

A malicious P2P peer can serve a `SignedBlockHeader` with an arbitrary `state_diff_length` value, then serve exactly that many `StateDiffChunk` units containing attacker-chosen storage diffs, deployed contracts, and nonces. Because `parse_data_for_block` never verifies the assembled `ThinStateDiff` against the `state_diff_commitment` stored in the header, the corrupted state diff is written directly to storage and subsequently served by RPC as authoritative.

### Finding Description

**Phase 1 — Header injection:**

`HeaderStreamBuilder::parse_data_for_block` performs only two checks on a received `SignedBlockHeader`:
- `block_number` matches the expected value
- `signatures.len() == ALLOWED_SIGNATURES_LENGTH` (i.e., exactly 1 element)

There is no cryptographic verification of the signature, and no verification of `state_diff_commitment` or `state_diff_length` against any trusted source (L1, consensus, or otherwise). The header — including its attacker-controlled `state_diff_length` — is written directly to storage via `append_header`. [1](#0-0) 

**Phase 2 — State diff injection:**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` from the already-stored header: [2](#0-1) 

It then consumes exactly `target_state_diff_len` units of `StateDiffChunk` from the peer, assembles them into a `ThinStateDiff`, and — after only checking for duplicate deprecated declared classes — returns the result for storage: [3](#0-2) 

**No commitment check exists anywhere in this path.** The `state_diff_commitment` field is stored in the header but is never retrieved and compared against `calculate_state_diff_hash(&result)` before or after writing. The `calculate_state_diff_hash` function exists and is used in `apollo_committer`, but it is entirely absent from the P2P sync client: [4](#0-3) 

**Write to storage is unconditional:** [5](#0-4) 

### Impact Explanation

Any node running P2P sync that connects to a malicious peer will store an attacker-chosen `ThinStateDiff` for block N. This corrupts the state diff column so that `starknet_getStateUpdate` returns wrong storage values, class hashes, nonces, and deployed contracts as authoritative. This matches the **High** impact category: *RPC returns an authoritative-looking wrong value*.

### Likelihood Explanation

The P2P network is open — any node can act as a peer and serve headers and state diff chunks. The attacker needs only to:
1. Be selected as a sync peer (standard P2P behavior).
2. Send a well-formed `SignedBlockHeader` with `block_number` matching the syncing node's current position and `signatures.len() == 1`.
3. Send exactly `state_diff_length` non-empty, non-conflicting `StateDiffChunk` messages with arbitrary content.

All existing format checks (`EmptyStateDiffPart`, `WrongStateDiffLength`, `ConflictingStateDiffParts`) are trivially satisfied by a cooperative attacker. No privileged access is required.

### Recommendation

After assembling `result` and before returning `Ok(Some((result, block_number)))`, verify the assembled state diff against the commitment stored in the header:

```rust
let stored_commitment = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .state_diff_commitment
    .ok_or(...)?;
let calculated = calculate_state_diff_hash(&result);
if calculated != stored_commitment {
    return Err(ParseDataError::BadPeer(BadPeerError::InvalidStateDiffCommitment { ... }));
}
```

This mirrors the pattern already used in `apollo_committer`: [6](#0-5) 

### Proof of Concept

Two-phase Rust integration test (mirrors the existing test harness in `state_diff_test.rs`):

1. **Phase 1**: Send a `SignedBlockHeader` for `BlockNumber(0)` with `state_diff_length = 3` and `signatures.len() == 1`. Verify it is stored.
2. **Phase 2**: Send 3 `StateDiffChunk::ContractDiff` entries with arbitrary storage keys (e.g., key `0x1337` → value `0xdead`). Send `DataOrFin(None)` to close the session.
3. **Assert**: `reader.begin_ro_txn().get_state_diff(BlockNumber(0))` returns the injected storage diffs, not the real on-chain state diff.

The existing `validate_state_diff_fails` helper in `state_diff_test.rs` already demonstrates that the header's `state_diff_length` fully controls how many chunks are consumed — the only missing assertion is that a *correctly-sized but wrong-content* diff is accepted and stored. [7](#0-6)

        HashMap::from([

    )
