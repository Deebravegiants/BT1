### Title
`respond_verify_foreign_tx` Accepts Arbitrary `payload_hash` Without Binding It to the Pending Request — (`File: crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` contract method verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the root MPC public key, but it never checks that `response.payload_hash` is the canonical hash of the `ForeignTxSignPayload` derived from the `request` argument that is being resolved. A single malicious attested participant who has legitimately obtained a threshold signature for any foreign-tx request can substitute that signature — along with its associated `payload_hash` — as the response to a completely different pending request, delivering a forged foreign-chain attestation to the caller.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

It then resolves the yield for the caller of `request`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

The contract confirms only that `(payload_hash, signature)` is a self-consistent pair under the root key. It never enforces that `payload_hash == SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: <observed> }))`. The `payload_hash` field is entirely attacker-supplied and is never bound to the `request` being resolved. [1](#0-0) 

By contrast, the off-chain SDK helper `ForeignChainSignatureVerifier::verify_signature` does perform this binding check — it recomputes the expected hash from `(request, expected_extracted_values)` and compares it to `response.payload_hash` before accepting the signature. The contract itself provides no equivalent guarantee. [2](#0-1) 

The `ForeignTxSignPayload` that nodes are supposed to sign encodes both the original `ForeignChainRpcRequest` and the extracted values observed on the foreign chain:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
// msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [3](#0-2) 

The node-side `build_signature_request` correctly derives the payload hash from the actual foreign-chain observation and uses a zero tweak, so the threshold signature is over `SHA-256(borsh(ForeignTxSignPayload{request_A, values_A}))` for request A. [4](#0-3) 

### Impact Explanation

A single malicious attested participant who is the leader for any foreign-tx request A obtains the legitimately-produced threshold signature `(payload_hash_A, signature_A)` as part of normal protocol execution. They can then call `respond_verify_foreign_tx(request_B, {payload_hash_A, signature_A})` for any other pending request B. The contract accepts the call because:

1. The caller is an attested participant — `assert_caller_is_attested_participant_and_protocol_active()` passes.
2. `signature_A` is a valid ECDSA signature over `payload_hash_A` under the root key — `verify_ecdsa_signature` passes.
3. `request_B` is present in `pending_verify_foreign_tx_requests` — `resolve_yields_for` succeeds.

The caller of request B receives `{payload_hash_A, signature_A}` — a valid MPC-signed attestation that encodes the foreign-chain state observed for transaction A, not transaction B. A bridge contract consuming this response cannot distinguish it from a legitimate response for B, because `payload_hash_A` is an opaque 32-byte hash; the bridge cannot recompute the expected hash without knowing the extracted values (which are the very thing being attested). This enables forged bridge execution: the attacker can make the bridge contract believe that transaction B was finalized with the block-hash or other extracted values that were actually observed for transaction A, enabling double-spend or invalid inbound bridge claims.

This is the direct analog of the external report's vulnerability: just as the Foundation market contract accepted an attacker-controlled royalty-registry entry that caused `isCreator = true` and redirected all revenue to the creator, `respond_verify_foreign_tx` accepts an attacker-controlled `payload_hash` that redirects the MPC attestation to a different foreign-chain observation, delivering a forged result to the legitimate requester.

### Likelihood Explanation

Any single attested participant who is elected leader for at least one foreign-tx request obtains a valid threshold signature as a normal byproduct of protocol execution. Leader election is deterministic from the request ID, so over time every participant will lead some requests. No threshold collusion is required: the attacker uses a legitimately-produced signature, merely submitting it against the wrong pending request. The `respond_verify_foreign_tx` entry point is open to any attested participant with no additional access control beyond attestation. [5](#0-4) 

### Recommendation

Inside `respond_verify_foreign_tx`, after verifying the signature, recompute the expected payload hash from the `request` and the extracted values embedded in the response, and assert equality with `response.payload_hash`. Concretely, the response should carry the full `ForeignTxSignPayload` (not just its hash), or the contract should recompute `SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: response.values }))` and compare it to `response.payload_hash` before resolving the yield. This mirrors the binding check already present in the off-chain SDK verifier and ensures that the attested hash is always derived from the specific request being resolved, not from an arbitrary attacker-supplied value.

### Proof of Concept

1. Two foreign-tx requests are pending:
   - **Request A**: `{tx_id: [0xAA; 32], chain: Bitcoin, extractors: [BlockHash]}`
   - **Request B**: `{tx_id: [0xBB; 32], chain: Bitcoin, extractors: [BlockHash]}`

2. Malicious participant M is elected leader for request A. M queries Bitcoin, observes `block_hash_A`, and the threshold MPC protocol produces:
   - `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload{request_A, [BlockHash(block_hash_A)]}))` 
   - `signature_A` = valid ECDSA signature over `payload_hash_A` under the root key.

3. Instead of submitting the correct response for request A, M calls:
   ```
   respond_verify_foreign_tx(
       request = request_B,          // resolves request B's yield
       response = {
           payload_hash: payload_hash_A,   // hash of request A's payload
           signature: signature_A,          // valid sig over payload_hash_A
       }
   )
   ```

4. The contract at lines 726–734 checks only that `signature_A` is valid over `payload_hash_A` — it passes. `resolve_yields_for` resolves request B's yield with `{payload_hash_A, signature_A}`.

5. The bridge contract waiting on request B receives `{payload_hash_A, signature_A}`. It verifies the signature (valid), trusts `payload_hash_A` as the MPC attestation for transaction B, and processes an inbound bridge claim for `tx_id = [0xBB; 32]` using the block-hash that was actually observed for `tx_id = [0xAA; 32]`. ```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
