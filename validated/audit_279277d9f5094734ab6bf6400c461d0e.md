### Title
Attacker Can Steal All Remaining Staking Rewards by Staking 1 Wei When `totalKernelStaked` Drops to Zero - (`contracts/KERNEL/KernelDepositPool.sol`)

---

### Summary

`KernelDepositPool` is a Synthetix-style staking rewards contract. When `totalKernelStaked` drops to zero during an active reward period, `rewardPerToken()` freezes at `rewardPerTokenStored`. Any attacker who stakes as little as 1 wei immediately after this condition is met will capture all remaining rewards for the period, because `updatedAt` is reset to the current timestamp on their stake and the entire remaining reward stream is then divided by a `totalKernelStaked` of 1.

---

### Finding Description

The `rewardPerToken()` function in `KernelDepositPool` is:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

When `totalKernelStaked == 0`, the function returns the frozen `rewardPerTokenStored`. The `updateReward` modifier, called on every state-changing function, sets `updatedAt = lastTimeRewardApplicable()`. So when the last staker calls `initiateWithdrawal`, `updatedAt` is stamped to the current time and `totalKernelStaked` becomes 0.

When an attacker then calls `stake(1)`, the `updateReward(attacker)` modifier fires **before** `balanceOf[attacker]` or `totalKernelStaked` are incremented:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // still frozen (totalKernelStaked == 0)
    updatedAt = lastTimeRewardApplicable();    // reset to NOW
    if (_account != address(0)) {
        rewards[_account] = earned(_account); // 0 (balance still 0)
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

After the modifier, `balanceOf[attacker] += 1` and `totalKernelStaked += 1`. From this point forward, `rewardPerToken()` advances at rate `rewardRate * Δt * DECIMAL_PRECISION / 1`, i.e., the entire reward stream flows to the single 1-wei staker. The attacker earns:

```
earned ≈ rewardRate × (finishAt − stakeTime)
```

which equals all remaining rewards for the period.

The protocol's own NatSpec comment explicitly acknowledges this:

> *"Otherwise, staking just 1 wei by any address will ensure that the contract never has any unallocated rewards."*

The only code-level guard is in `notifyRewardAmount`:

```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
```

This prevents **starting** a new reward period with zero stakers, but does nothing to prevent stakers from leaving **after** the period has begun, which is the actual attack surface.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

An attacker who monitors the contract for `totalKernelStaked == 0` and immediately stakes 1 wei captures the entire remaining reward budget for the active period. The rewards were funded by the admin/protocol for the benefit of legitimate stakers. If legitimate stakers re-stake after the attacker, they find `rewardPerTokenStored` has already advanced to near `finishAt`, leaving them with negligible earnings.

---

### Likelihood Explanation

**Medium.**

The condition requires `totalKernelStaked` to reach zero during an active reward window. This can occur naturally if all stakers call `initiateWithdrawal` (which immediately decrements `totalKernelStaked`). The protocol relies entirely on an off-chain operational guarantee — "ensuring there are always some tokens staked for the entire duration of the reward period" — rather than any on-chain enforcement. A sophisticated attacker can monitor the chain and front-run any re-staking attempt once the condition is met.

---

### Recommendation

Add an on-chain guard in `stake` / `stakeFor` that prevents a single depositor from capturing a disproportionate share of accumulated rewards when `totalKernelStaked` was zero. One approach is to send the "frozen" rewards (those accrued while `totalKernelStaked == 0`) to the treasury or roll them into the next reward period rather than allowing them to be captured by the next staker. Concretely, track the timestamp at which `totalKernelStaked` last became zero and, in `updateReward`, skip the time interval during which no tokens were staked when computing `rewardPerTokenStored`.

---

### Proof of Concept

```
1. Admin calls notifyRewardAmount(1_000_000e18) with rewardRate = 1_000_000e18 / duration.
   totalKernelStaked > 0 at this point (check passes).

2. All legitimate stakers call initiateWithdrawal().
   → totalKernelStaked = 0.
   → updateReward stamps updatedAt = block.timestamp.
   → rewardPerTokenStored is frozen.

3. Attacker calls stake(1).
   → updateReward fires: rewardPerTokenStored unchanged (totalKernelStaked still 0 in view),
     updatedAt = block.timestamp (NOW), userRewardPerTokenPaid[attacker] = rewardPerTokenStored.
   → totalKernelStaked = 1.

4. Time passes until finishAt.
   → rewardPerToken() = rewardPerTokenStored + rewardRate * (finishAt - NOW) * 1e18 / 1
   → earned(attacker) ≈ rewardRate * (finishAt - NOW) ≈ all remaining rewards.

5. Attacker calls getReward() and receives the entire remaining reward budget.
   Legitimate stakers who re-stake after step 3 find rewardPerTokenStored already maxed out.
