### Title
`DepositAllowlistExtension` Allowlist Bypassed via Arbitrary `owner` in `addLiquidity` ‚Äî (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

---

### Summary

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address without requiring `msg.sender == owner`. The `DepositAllowlistExtension.beforeAddLiquidity` hook checks the `owner` parameter against the allowlist while silently ignoring the actual caller (`msg.sender`). Any address not on the allowlist can bypass the gate by specifying an allowlisted address as `owner`, paying tokens themselves while the allowlisted address receives LP shares.

---

### Finding Description

`MetricOmmPool.addLiquidity` passes `msg.sender` as the first argument and the caller-supplied `owner` as the second argument to the extension hook: [1](#0-0) 

There is no `msg.sender == owner` check in `addLiquidity` (unlike `removeLiquidity` which enforces it at line 206): [2](#0-1) 

Inside `LiquidityLib.addLiquidity`, the token-pull callback is invoked on `msg.sender` (the actual payer), while shares are credited to `owner`: [3](#0-2) [4](#0-3) 

`DepositAllowlistExtension.beforeAddLiquidity` receives the actual caller as its first (unnamed, discarded) parameter and checks only `owner`: [5](#0-4) 

The extension's own NatSpec states it "Gates `addLiquidity` by depositor address" ‚Äî but the depositor is `msg.sender`, not `owner`. Because the first parameter is unnamed and unused, the gate is evaluated against the wrong identity. An attacker who is not on the allowlist can call `pool.addLiquidity(allowlisted_victim, salt, deltas, ...)`, pass the extension check (since `allowlisted_victim` is approved), pay the tokens themselves, and credit LP shares to the victim.

The same bypass is reachable through the periphery `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)`, where `_validateOwner` only checks `owner != address(0)` and does not enforce `owner == msg.sender`: [6](#0-5) 

---

### Impact Explanation

The `DepositAllowlistExtension` is the pool admin's mechanism to restrict which addresses may deposit into the pool (e.g., for permissioned or regulated pools). The bypass allows any unprivileged address to inject tokens into the pool and credit LP shares to an allowlisted address, violating the admin-configured access boundary. This matches the allowed impact gate: **admin-boundary break ‚Äî factory/pool-admin role checks bypassed by an unprivileged path**.

Secondary effects:
- The allowlisted victim receives unwanted LP shares in an arbitrary `(owner, salt, bin)` position they did not initiate.
- The attacker can manipulate specific bin balances (e.g., inflate a bin's `token0BalanceScaled` or `token1BalanceScaled`) to affect proportional share redemption for other LPs in that bin, since per-bin token amounts are shared pro-rata across all position holders. [7](#0-6) 

---

### Likelihood Explanation

Medium. The precondition is that the pool has `DepositAllowlistExtension` registered as a `beforeAddLiquidity` hook and at least one allowlisted address is publicly known (e.g., from on-chain events). Both conditions are observable on-chain. The attacker must hold enough tokens to fund the callback, but gains nothing financially ‚Äî the attack is a griefing/compliance bypass rather than a theft.

---

### Recommendation

`DepositAllowlistExtension.beforeAddLiquidity` should check the actual caller (first parameter) rather than `owner`:

```solidity
function beforeAddLiquidity(address caller, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][caller]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Alternatively, if the intent is to gate position ownership (not payment), the extension's NatSpec and name should be corrected, and `addLiquidity` should enforce `msg.sender == owner` when a restrictive extension is active.

---

### Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is not allowlisted.
3. Bob (not allowlisted) calls:
   ```solidity
   pool.addLiquidity(alice, salt, deltas, callbackData, extensionData);
   ```
4. Pool calls `extension.beforeAddLiquidity(bob, alice, salt, deltas, extensionData)`.
5. Extension checks `allowedDepositor[pool][alice]` ‚Üí `true` ‚Üí passes. Bob's identity is never checked.
6. `LiquidityLib.addLiquidity` calls `IMetricOmmModifyLiquidityCallback(bob).metricOmmModifyLiquidityCallback(...)` ‚Äî Bob pays tokens.
7. `positionBinShares[keccak256(alice, salt, bin)]` is incremented ‚Äî Alice receives LP shares.
8. Bob (not allowlisted) has successfully deposited into the restricted pool, violating the admin's access control invariant. [8](#0-7)

contract DepositAllowlistExtension is BaseMetricExtension, IDepositAllowlistExtension {
  mapping(address pool => mapping(address depositor => bool)) public allowedDepositor;
  mapping(address pool => bool) public allowAllDepositors;

  constructor(address factory_) BaseMetricExtension(factory_) {}

  function setAllowedToDeposit(address pool_, address depositor, bool allowed) external onlyPoolAdmin(pool_) {
    allowedDepositor[pool_][depositor] = allowed;
    emit AllowedToDepositSet(pool_, depositor, allowed);
  }

  function setAllowAllDepositors(address pool_, bool allowed) external onlyPoolAdmin(pool_) {
    allowAllDepositors[pool_] = allowed;
    emit AllowAllDepositorsSet(pool_, allowed);
  }

  function isAllowedToDeposit(address pool_, address depositor) external view returns (bool) {
    return allowAllDepositors[pool_] || allowedDepositor[pool_][depositor];
  }

  function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override

      revert IMetricOmmPoolActions.NotAllowedToDeposit();

    return IMetricOmmExtensions.beforeAddLiquidity.selector;
  }
