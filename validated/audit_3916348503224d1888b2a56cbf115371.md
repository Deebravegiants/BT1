Audit Report

## Title
`checkDailyMintLimit` Blocks All Minting When `maxMintAmountPerDay` Is Zero — (`contracts/RSETH.sol`)

## Summary
`RSETH.initialize()` and `reinitialize()` never set `maxMintAmountPerDay`, leaving it at the default `uint256` value of `0`. The `checkDailyMintLimit` modifier treats `0` as a hard limit rather than a sentinel for "unlimited," causing every call to `mint()` — and therefore every call to `LRTDepositPool.depositETH` and `depositAsset` — to revert with `DailyMintLimitExceeded` until a manager explicitly calls `setMaxMintAmountPerDay` with a non-zero value.

## Finding Description
`maxMintAmountPerDay` is declared as a plain `uint256` state variable and defaults to `0`. Neither `initialize(address,address)` nor `reinitialize(uint256,address)` assigns it a value. The only setter, `setMaxMintAmountPerDay(uint256)`, accepts `0` without validation.

The enforcement modifier is:

```solidity
// contracts/RSETH.sol L42-56
modifier checkDailyMintLimit(uint256 amount) {
    if (block.timestamp >= periodStartTime + 1 days) {
        currentPeriodMintedAmount = 0;
        periodStartTime = getCurrentPeriodStartTime();
    }
    if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
        revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
    }
    currentPeriodMintedAmount += amount;
    _;
}
```

When `maxMintAmountPerDay == 0`, the condition `currentPeriodMintedAmount + amount > 0` reduces to `amount > 0`, which is always true for any non-zero deposit. `mint()` applies this modifier unconditionally, and `LRTDepositPool._mintRsETH` calls `IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint)` for both `depositETH` and `depositAsset`. The result is that all deposits revert in the default post-deployment state and whenever a manager resets the value to `0`.

`remainingDailyMintLimit()` returns `0` when `maxMintAmountPerDay == 0` rather than `type(uint256).max`, confirming there is no sentinel bypass anywhere in the contract.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

No depositor can mint rsETH while `maxMintAmountPerDay == 0`. The protocol's core function — issuing liquid restaking tokens in exchange for ETH/LSTs — is completely unavailable. No funds already held are at risk of loss.

## Likelihood Explanation
**Low.** The most realistic path is the default post-deployment state: after both `initialize()` and `reinitialize()` are called, the contract appears fully initialized but `maxMintAmountPerDay` is still `0`. Any depositor who attempts `depositETH` or `depositAsset` during the window before `setMaxMintAmountPerDay` is called with a non-zero value will have their transaction revert. A second path exists if a manager later calls `setMaxMintAmountPerDay(0)`, which is accepted without revert and silently re-enters the broken state. Neither path requires an unprivileged attacker — the harm to depositors is automatic.

## Recommendation
Add a sentinel bypass in `checkDailyMintLimit` so that `maxMintAmountPerDay == 0` means "no limit":

```solidity
modifier checkDailyMintLimit(uint256 amount) {
    if (maxMintAmountPerDay != 0) {
        if (block.timestamp >= periodStartTime + 1 days) {
            currentPeriodMintedAmount = 0;
            periodStartTime = getCurrentPeriodStartTime();
        }
        if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
            revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
        }
        currentPeriodMintedAmount += amount;
    }
    _;
}
```

Additionally, update `remainingDailyMintLimit()` to return `type(uint256).max` when `maxMintAmountPerDay == 0` to reflect the "unlimited" semantic consistently. Optionally, add `require(_maxMintAmountPerDay > 0)` in `setMaxMintAmountPerDay` and set a non-zero default in `initialize()` or `reinitialize()`.

## Proof of Concept
1. Deploy `RSETH` proxy and call `initialize(admin, lrtConfig)`. `maxMintAmountPerDay` is `0`.
2. Call `reinitialize(periodStartTime, custodyAddress)` as manager. `maxMintAmountPerDay` remains `0`.
3. A depositor calls `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. `_beforeDeposit` succeeds; `_mintRsETH` calls `IRSETH(rsethToken).mint(depositor, rsethAmount)`.
5. `checkDailyMintLimit` evaluates `0 + rsethAmount > 0` → `true` → reverts with `DailyMintLimitExceeded(rsethAmount, 0)`.
6. All deposits revert until `setMaxMintAmountPerDay` is called with a non-zero value.
