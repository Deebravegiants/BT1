### Title
Unconditional CKD Response Delivery for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC contract unconditionally resolves CKD (Confidential Key Derivation) requests when the `AppPublicKey` variant is used, without verifying the correctness of the derived key output. A single Byzantine MPC participant (strictly below the signing threshold) can call `respond_ckd` with a fabricated `CKDResponse`, and the contract will accept it, permanently delivering an attacker-controlled derived key to the user.

---

### Finding Description

The vulnerability class from the report is **unconditional processing in a loop where conditional processing was intended** — items are appended/resolved regardless of whether they pass the required eligibility check, producing zero-value or incorrect entries.

The direct analog in this codebase is in `respond_ckd` at `crates/contract/src/lib.rs` lines 653–689:

```rust
pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
    // ...
    self.assert_caller_is_attested_participant_and_protocol_active();

    let PublicKeyExtended::Bls12381 { public_key: dtos::PublicKey::Bls12381(public_key) }
        = self.public_key_extended(request.domain_id)? else { ... };

    match &request.app_public_key {
        dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
        dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
            if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                env::panic_str("CKD output check failed");
            }
        }
    }

    pending_requests::resolve_yields_for(          // ← ALWAYS CALLED
        &mut self.pending_ckd_requests,
        &request,
        serde_json::to_vec(&response).unwrap(),
    )
}

When `AppPublicKey` is used, `resolve_yields_for` is called unconditionally — whatever `CKDResponse` the caller provides is serialized and delivered to the waiting user. There is no cryptographic check that the response is the correct MPC-derived output.

Contrast this with `respond` (regular signatures), which always verifies the signature against the derived public key before resolving:

The structural parallel to the beacon-kit bug is exact:

| beacon-kit | NEAR MPC |
|---|---|
| `if is_fully_withdrawable { amount = balance } else if is_partially_withdrawable { amount = balance - max }` then **unconditionally** `withdrawals.append(withdrawal.New(..., amount))` | `if AppPublicKeyPV { ckd_output_check(...) }` then **unconditionally** `resolve_yields_for(...)` |

Additionally, `respond_ckd` has **no leader restriction** — any attested participant may call it, not only the designated leader. `resolve_yields_for` removes the request from `pending_ckd_requests` on first call, so the first caller wins and the legitimate leader is permanently locked out. ### Impact Explanation

**Critical — Unauthorized confidential key derivation output without the required participant authorization.**

A single Byzantine MPC participant (below the signing threshold) can:

1. Observe a `request_app_private_key` call with `AppPublicKey` on-chain (the full request is public).
2. Immediately call `respond_ckd(request, fabricated_response)` where `fabricated_response` contains attacker-chosen key material.
3. The contract accepts the fabricated response with no verification and calls `resolve_yields_for`, delivering the wrong key to the user and removing the request from the pending map.
4. The legitimate leader's subsequent `respond_ckd` call returns `RequestNotFound` — the user cannot recover the correct key.

The user's application receives a derived key controlled by the attacker. Any assets or identities secured by that derived key are compromised.

---

### Likelihood Explanation

**Medium.** The attack requires exactly one Byzantine attested MPC participant — strictly below the signing threshold. No collusion is needed. The on-chain request is fully visible, so the attacker can craft the fabricated response immediately after the request lands. The window is the block-processing latency between the user's `request_app_private_key` transaction and the legitimate leader's `respond_ckd` transaction.

---

### Recommendation

Apply the same pattern used for `AppPublicKeyPV` and for `respond` (regular signatures): **reject the response if it cannot be verified**. Options:

1. **Reject `AppPublicKey` in `respond_ckd`** — require callers to use `AppPublicKeyPV` so the output can always be verified on-chain.
2. **Restrict `respond_ckd` to the designated leader** — add a leader-election check analogous to the off-chain `get_requests_to_attempt` logic, so only the elected leader can submit a response.
3. **Add a cryptographic commitment** — require the response to include a proof that the output was produced by the threshold protocol (e.g., a BLS proof of correct derivation), and verify it for both variants.

---

### Proof of Concept

```
1. User calls request_app_private_key({
       domain_id: <CKD domain>,
       derivation_path: "m/0/1",
       app_public_key: AppPublicKey(<user_app_pk>)
   })
   → contract stores CKDRequest in pending_ckd_requests, issues yield

2. Byzantine participant P (one node, below threshold) observes the
   CKDRequest on-chain.

3. P calls respond_ckd(
       request = <observed CKDRequest>,
       response = CKDResponse { <attacker-chosen key material> }
   )
   → assert_caller_is_attested_participant_and_protocol_active() passes (P is attested)
   → AppPublicKey arm: no check, falls through
   → resolve_yields_for() delivers attacker key to user, removes request

4. Legitimate leader calls respond_ckd(...) → returns RequestNotFound.

5. User's yield resumes with attacker-controlled key material.
   Attacker knows the corresponding private key → can steal user assets.
```

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
