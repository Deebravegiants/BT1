### Title
Missing Zero Address Check for `admin` in `initialize()` Permanently Locks Admin Functions - (File: contracts/pools/RSETHPoolV2NBA.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/agETH/AGETHPoolV3.sol)

### Summary
Multiple pool `initialize()` functions grant `DEFAULT_ADMIN_ROLE` to the `admin` parameter without first checking that it is non-zero. If `address(0)` is passed as `admin`, the role is permanently assigned to the zero address, making all `onlyRole(DEFAULT_ADMIN_ROLE)` functions permanently uncallable and locking the contract's administrative controls forever.

### Finding Description
In `RSETHPoolV2NBA.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV2ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `AGETHPoolV3.sol`, the `initialize()` function validates the token and oracle addresses via `UtilLib.checkNonZeroAddress`, but does **not** validate the `admin` (or `bridger`) parameter before granting roles.

For example, in `RSETHPoolV2NBA.sol`:

```solidity
function initialize(
    address admin,
    address bridger,
    address _wrsETH,
    uint256 _feeBps,
    address _rsETHOracle
) external initializer {
    UtilLib.checkNonZeroAddress(_wrsETH);      // checked
    UtilLib.checkNonZeroAddress(_rsETHOracle); // checked
    // admin and bridger are NOT checked
    _grantRole(DEFAULT_ADMIN_ROLE, admin);     // admin could be address(0)
    _setupRole(BRIDGER_ROLE, admin);
    _setupRole(BRIDGER_ROLE, bridger);         // bridger could be address(0)
    ...
}
```

The same pattern appears in `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV2ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `AGETHPoolV3.sol`.

`UtilLib.checkNonZeroAddress` is available and used elsewhere in the same functions, but is simply not applied to `admin` or `bridger`.

### Impact Explanation
If `admin` is `address(0)` at initialization:
- `DEFAULT_ADMIN_ROLE` is permanently assigned to `address(0)`.
- All `onlyRole(DEFAULT_ADMIN_ROLE)` functions — including `setFeeBps`, `unpause`, `setDailyMintLimit`, and role management — become permanently inaccessible.
- The contract cannot be reconfigured or unpaused, permanently failing to deliver its promised operational behavior.
- If `bridger` is also `address(0)`, `withdrawFees` and `moveAssetsForBridging` become permanently inaccessible, trapping accumulated protocol fees in the contract.

This matches the **Low** impact class: *Contract fails to deliver promised returns, but doesn't lose value.*

### Likelihood Explanation
This is a deployment-time misconfiguration. The deployer must accidentally pass `address(0)` as `admin`. While unlikely in a careful deployment, the absence of a guard means there is no on-chain safety net. The same class of bug was already identified and fixed in other contracts in this repository (e.g., `KernelVaultETH.sol`, `RSETHPoolNoWrapper.sol`, `L1Vault.sol` all call `UtilLib.checkNonZeroAddress(_admin)`), confirming the pattern is known and the omission in these contracts is an oversight.

### Recommendation
Add `UtilLib.checkNonZeroAddress(admin)` and `UtilLib.checkNonZeroAddress(bridger)` at the top of each affected `initialize()` function, consistent with the pattern already used in sibling contracts:

```solidity
UtilLib.checkNonZeroAddress(admin);
UtilLib.checkNonZeroAddress(bridger);
UtilLib.checkNonZeroAddress(_wrsETH);
UtilLib.checkNonZeroAddress(_rsETHOracle);
```
