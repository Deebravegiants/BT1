Audit Report

## Title
Missing Token Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Near-Zero wrsETH Minted for Non-18-Decimal Token Deposits - (File: `contracts/pools/RSETHPoolV3.sol`)

## Summary

`RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18 decimals. Because `ChainlinkOracleForRSETHPoolCollateral.getRate()` returns the price of one full human-readable token unit in ETH (1e18-scaled), the formula is only dimensionally consistent when the deposited token has 18 decimals. For a 6-decimal token such as USDC, the result is `10^12` times smaller than correct, so a depositor surrenders their full token balance and receives economically worthless wrsETH.

## Finding Description

**Root cause — `RSETHPoolV3.sol` L324–334:**

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;

uint256 rsETHToETHrate = getRate();                                       // 1e18-scaled
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // 1e18-scaled

rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;          // ← no decimal normalisation
``` [1](#0-0) 

**Oracle semantics — `ChainlinkOracleForRSETHPoolCollateral.sol` L34:**

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [2](#0-1) 

`decimals()` here is the Chainlink feed's answer precision (typically 8), **not** the ERC-20 token's own decimal count. The returned value is therefore the ETH price of exactly one full token unit (e.g., 1 USDC), scaled to 1e18. It carries no information about how many raw token units constitute one full unit.

**Dimensional analysis for USDC (6 decimals, ETH ≈ $3 000):**

| Variable | Value |
|---|---|
| `amountAfterFee` (1 000 USDC) | `1e9` (micro-USDC) |
| `tokenToETHRate` (1 USDC → ETH, 1e18-scaled) | `≈ 3.33 × 10^14` |
| `rsETHToETHrate` (1e18-scaled) | `≈ 1.05 × 10^18` |
| **Computed `rsETHAmount`** | `1e9 × 3.33e14 / 1.05e18 ≈ 317 000` raw units |
| **Correct `rsETHAmount`** | `≈ 3.17 × 10^17` raw units (0.317 wrsETH) |

The depositor receives `≈ 3.17 × 10^−13` wrsETH — a factor of `10^12` below the correct value.

**Contrast with the ETH deposit path** (L307), which is correct because ETH is always 18 decimals:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

The token path omits the equivalent `* 1e18` scaling and instead relies on `tokenToETHRate`, which does not encode the token's own decimal count.

**Exploit flow:**

1. Admin calls `addSupportedToken(USDC, chainlinkUSDCOracle)` — permitted by the generic `addSupportedToken` which imposes no 18-decimal restriction. [4](#0-3) 
2. Any user approves the pool and calls `deposit(USDC, 1_000e6, "ref")`.
3. `safeTransferFrom` moves 1 000 USDC from the caller to the pool. [5](#0-4) 
4. `viewSwapRsETHAmountAndFee(1_000e6, USDC)` returns `≈ 317 000` raw wrsETH units.
5. `wrsETH.mint(msg.sender, 317_000)` mints negligible wrsETH. [6](#0-5) 
6. The depositor has permanently lost 1 000 USDC with no recovery path.

No existing guard checks token decimals or validates the computed `rsETHAmount` against a minimum threshold.

## Impact Explanation

**Critical — Direct theft of user funds.** Any depositor calling the public `deposit(address token, uint256 amount, string referralId)` function with a supported non-18-decimal token loses their entire deposit immediately and irrecoverably. The token is transferred to the pool and will eventually be moved by `BRIDGER_ROLE` via `moveAssetsForBridging`, permanently beyond the depositor's reach. This matches the allowed impact class: *Direct theft of any user funds, whether at-rest or in-motion*.

## Likelihood Explanation

**High.** The `addSupportedToken` function is generic and imposes no decimal restriction, making it straightforward for the protocol to onboard USDC or any other sub-18-decimal token. `ChainlinkOracleForRSETHPoolCollateral` is explicitly designed to handle arbitrary Chainlink feed decimal counts, confirming the protocol's intent to support diverse collateral. USDC is the most widely deployed non-18-decimal collateral on every chain where these pools operate. No special attacker capability is required; any ordinary depositor triggers the loss with a standard `deposit` call.

## Recommendation

Normalize `amountAfterFee` to 18 decimals before applying the rate formula in `viewSwapRsETHAmountAndFee`:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the symmetric inverse normalization in `viewSwapAssetToPremintedRsETH` (L400), which has the same dimensional error in the reverse direction:

```solidity
tokenAmount = (rsETHAmount * rsETHToETHrate / tokenToETHRate) / 10 ** (18 - tokenDecimals);
```

The same fix must be applied to all pool variants that share this formula: `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, `RSETHPool.sol`, and `RSETHPoolNoWrapper.sol`.

## Proof of Concept

**Foundry fork test outline:**

```solidity
function test_usdc_deposit_loss() public {
    // 1. Deploy RSETHPoolV3 with a mock ChainlinkOracleForRSETHPoolCollateral
    //    configured for a USDC/ETH feed (8-decimal Chainlink answer).
    // 2. Grant TIMELOCK_ROLE to test contract; call addSupportedToken(USDC, oracle).
    // 3. Deal 1_000e6 USDC to `alice`; alice approves pool.
    // 4. alice calls pool.deposit(USDC, 1_000e6, "ref").
    // 5. Assert: wrsETH.balanceOf(alice) < 1e6   // near-zero, not ~0.317e18
    // 6. Assert: IERC20(USDC).balanceOf(pool) == 1_000e6  // full deposit captured
    // 7. Compute expected = 1_000e6 * 1e12 * tokenToETHRate / rsETHToETHrate
    //    and assert actual is off by exactly 1e12.
}
```

The test requires no privileged attacker — only a standard depositor interacting with a pool that has a non-18-decimal supported token, which is a configuration the protocol's own `addSupportedToken` API is designed to enable.
