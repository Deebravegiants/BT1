### Title
User-Controlled Finality Parameters in `verify_foreign_transaction` Enable MPC Signing of Non-Finalized Foreign Chain State — (`crates/node/src/providers/verify_foreign_tx/sign.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint accepts user-controlled finality parameters — including `EvmFinality::Latest` for EVM chains and an arbitrary `BlockConfirmations` count for Bitcoin — without any on-chain or node-side restriction. MPC nodes sign payloads derived from non-finalized (reorg-able or double-spendable) foreign chain state. An unprivileged caller can exploit this to obtain a valid threshold signature attesting to a foreign transaction that is subsequently reorganized or double-spent, enabling forged bridge verification and invalid fund release.

---

### Finding Description

**Root cause — contract layer (`crates/contract/src/lib.rs`)**

`verify_foreign_transaction` validates only domain existence, gas, deposit, and chain-policy membership. It performs **no validation of the `finality` or `confirmations` fields** inside the `ForeignChainRpcRequest`:

```rust

The `EvmFinality` enum explicitly includes `Latest` (non-finalized, reorg-able), and `BitcoinRpcRequest` accepts any `BlockConfirmations` value including `0`: [2](#0-1) 

**Root cause — node layer (`crates/node/src/providers/verify_foreign_tx/sign.rs`)**

`execute_foreign_chain_request` passes the user-supplied `finality` / `confirmations` directly to the foreign chain inspector with no floor check:

```rust

Both the leader and follower independently call `execute_foreign_chain_request` and derive their signing payload from the result: [4](#0-3) 

The signed payload is `SHA-256(borsh(ForeignTxSignPayload{request, values}))`, where `values` are the extracted values from the non-finalized state: [5](#0-4)

### Impact Explanation

An attacker who obtains a valid threshold signature over a non-canonical transaction can present it to a bridge contract as proof that the foreign transaction finalized. The bridge releases funds based on a forged attestation. This constitutes:

- **Forged foreign-chain verification** — the MPC signature attests to a transaction that was never canonical.
- **Invalid bridge execution / double-spend** — funds are released on NEAR for a foreign transaction that was reorganized or replaced.

This maps to the allowed High impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- **Bitcoin (`confirmations: 0`)**: Zero-confirmation double-spends are a well-documented, low-cost attack. An attacker broadcasts TX_A, requests MPC signing with `confirmations: 0`, then RBF-replaces TX_A with TX_B at a higher fee. TX_A is dropped; the attacker holds a valid MPC signature for it.
- **EVM chains with `EvmFinality::Latest`**: Polygon and BNB Smart Chain have documented reorganization histories. An attacker submits a transaction, requests signing with `Latest`, then causes or waits for a reorg.
- Entry path requires only 1 yoctonear deposit — accessible to any unprivileged caller.

---

### Recommendation

1. **Contract layer**: Reject `EvmFinality::Latest` in `verify_foreign_transaction`; enforce `EvmFinality::Finalized` (or at minimum `Safe`) for all EVM requests.
2. **Contract layer**: Enforce a minimum `BlockConfirmations` floor (e.g., ≥ 6 for Bitcoin) and reject requests below it.
3. **Node layer**: Add a defensive check in `execute_foreign_chain_request` that refuses to proceed if the effective finality is below the protocol minimum, even if the contract somehow passes a non-finalized request through.

---

### Proof of Concept

**Bitcoin 0-confirmation double-spend path:**

1. Attacker broadcasts Bitcoin TX_A (e.g., sending 1 BTC to a bridge deposit address).
2. Attacker calls `verify_foreign_transaction` with `confirmations: 0` and TX_A's txid (costs 1 yoctonear).
3. MPC nodes observe TX_A in the mempool, extract `BlockHash` (or another extractor), compute `payload_hash = SHA-256(borsh({request, values}))`, and produce a threshold signature.
4. Attacker submits `respond_verify_foreign_tx` with the valid signature; the contract resolves the pending yield and returns the signed response to the attacker.
5. Attacker broadcasts TX_B (RBF replacement of TX_A, sending the same UTXO to themselves). TX_A is evicted from the mempool and never confirms.
6. Attacker presents the MPC-signed response to the bridge contract as proof that TX_A finalized, claiming the bridged funds.

The contract's `respond_verify_foreign_tx` only checks that the signature is valid for the provided `payload_hash` against the root public key — it does not re-verify that the foreign transaction actually reached the required confirmation depth: [6](#0-5)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
}
