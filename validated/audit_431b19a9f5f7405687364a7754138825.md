### Title
Missing Zero-Hash Guard in `sign()` for `DamgardEtAl` Protocol Causes Silent Request Lifecycle Failure — (File: `crates/contract/src/lib.rs`)

---

### Summary

The contract's `sign()` function validates ECDSA payloads for both `Protocol::CaitSith` and `Protocol::DamgardEtAl` by checking only that the hash is a valid field element, but does **not** check for zero. The MPC nodes' robust ECDSA signing function (`robust_ecdsa::sign()`) explicitly rejects zero-hash inputs as a documented security requirement. This mismatch means a zero-hash signing request for a `DamgardEtAl` domain is accepted by the contract but silently fails at the MPC node level, causing the request to time out after ~200 blocks instead of failing immediately — breaking the contract's stated production safety invariant.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `sign()` function validates ECDSA payloads for `Protocol::CaitSith | Protocol::DamgardEtAl` as follows:

```rust
Protocol::CaitSith | Protocol::DamgardEtAl => {
    let hash = *request.payload.as_ecdsa().expect("Payload is not Ecdsa");
    k256::Scalar::from_repr(hash.into())
        .into_option()
        .expect("Ecdsa payload cannot be converted to Scalar");
}
``` [1](#0-0) 

This check only verifies that the hash is a valid scalar (not an overflow of the field order). A zero hash (`[0u8; 32]`) is a valid scalar and passes this check silently.

However, in `crates/threshold-signatures/src/ecdsa/robust_ecdsa/sign.rs`, the `sign()` function explicitly rejects zero-hash inputs:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
``` [2](#0-1) 

This is a documented security requirement. The `robust_ecdsa` README states:

> "Additionally, `msg_hash == 0` is rejected to prevent a related-key split-view attack." [3](#0-2) 

And the signing specification elaborates:

> "Do not sign with h = 0 (the zero message hash). This input enables a related algebraic split-view attack in the modified scheme when N1 > 2t + 1." [4](#0-3) 

The contract's own comment at the validation site states the invariant that is broken:

> "It's important we fail here because the MPC nodes will fail in an identical way. This allows users to get the error message." [5](#0-4) 

**Attack flow for a `DamgardEtAl` domain:**

1. Attacker calls `sign()` with `payload = [0u8; 32]` (zero hash) targeting a `DamgardEtAl` domain.
2. The contract's `from_repr` check passes (zero is a valid scalar).
3. The contract enqueues the request via `push_pending_yield` and yields.
4. MPC nodes pick up the request and invoke `robust_ecdsa::sign()` with `msg_hash = Scalar::ZERO`.
5. `robust_ecdsa::sign()` returns `Err(InitializationError::BadParameters(...))` — the signing protocol never starts.
6. No `respond()` call is made by any node.
7. The yield times out after ~200 blocks, returning `Err(PromiseError::Failed)` to the caller.
8. A presignature is consumed without producing a valid signature.

This is the direct analog of the external report: a zero value passes the contract-level guard, reaches a downstream function that rejects it, and causes a silent failure instead of an immediate, informative rejection.

Note: for `Protocol::CaitSith` (OT-based ECDSA), the MPC nodes do **not** reject zero hash — `ot_based_ecdsa::sign()` has no `is_zero()` check — so the mismatch is specific to `DamgardEtAl`. [6](#0-5) 

---

### Impact Explanation

**Medium.** This breaks the production safety/request-lifecycle invariant that the contract should mirror the MPC nodes' input validation and surface errors immediately to callers. Concretely:

- A signing request that the MPC network will unconditionally reject is accepted by the contract and enqueued.
- A presignature (a limited, expensive offline resource) is consumed without producing a valid signature.
- The caller waits ~200 blocks for a timeout error rather than receiving an immediate, actionable rejection.
- The contract's stated design invariant ("fail here because the MPC nodes will fail in an identical way") is violated for the `DamgardEtAl` protocol.

This does not require network-level DoS or operator misconfiguration — it is triggered by a single unprivileged `sign()` call.

---

### Likelihood Explanation

**Low-Medium.** The attacker needs only to:

1. Identify that a `DamgardEtAl` domain exists in the contract's domain registry (observable via `state()` or `public_key()`).
2. Submit `sign()` with a zero-byte payload and the target domain ID.
3. Attach the minimum deposit of 1 yoctoNEAR.

No special privileges, key material, or threshold collusion are required. The entry path is fully unprivileged and reachable from any NEAR account.

---

### Recommendation

Add an explicit zero-hash guard in `sign()` for `Protocol::DamgardEtAl`, mirroring the check already present in `robust_ecdsa::sign()`:

```rust
Protocol::CaitSith | Protocol::DamgardEtAl => {
    let hash = *request.payload.as_ecdsa().expect("Payload is not Ecdsa");
    let scalar = k256::Scalar::from_repr(hash.into())
        .into_option()
        .expect("Ecdsa payload cannot be converted to Scalar");
    // DamgardEtAl (robust ECDSA) rejects zero msg_hash at the node level
    // to prevent split-view attacks; mirror that check here for immediate feedback.
    if matches!(domain_config.protocol, Protocol::DamgardEtAl) {
        if bool::from(scalar.is_zero()) {
            env::panic_str("msg_hash cannot be 0 for DamgardEtAl protocol");
        }
    }
}
```

This ensures the contract surfaces the rejection immediately, preserves the stated invariant, and prevents presignature waste.

### Proof of Concept

```
1. Query contract state to confirm a DamgardEtAl domain exists (domain_id = X).
2. Call sign({
       domain_id: X,
       payload: { ecdsa: [0u8; 32] },   // zero hash
       path: "test"
   }) with deposit=1 yoctoNEAR, max_gas.
3. Observe: contract returns a pending yield (no immediate error).
4. Wait ~200 blocks.
5. Observe: the yield callback fires with PromiseError::Failed (timeout).
6. Confirm: no respond() was ever called; a presignature was consumed.
7. Compare: calling sign() with a non-zero invalid scalar (e.g., field-order overflow)
   produces an immediate panic — demonstrating the inconsistency.
```
