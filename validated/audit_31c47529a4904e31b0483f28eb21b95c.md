Audit Report

## Title
ETH Withdrawal Permanently Frozen for Contract Recipients That Revert on ETH Receive - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager` uses a push-payment model to deliver ETH to withdrawing users. When a user's address is a contract that cannot receive ETH, `completeWithdrawal` permanently reverts for that user. Because rsETH is burned in a prior, separate transaction (`unlockQueue`), the user's rsETH is permanently destroyed while their ETH is frozen inside `LRTWithdrawalManager` with no on-chain recovery path.

## Finding Description
`_transferAsset` at L876–879 delivers ETH via a low-level call and reverts on failure:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

This is the terminal step of `_processWithdrawalCompletion` (L734), called by both `completeWithdrawal` (L183) and `completeWithdrawalForUser` (L192).

The critical ordering spans two separate, already-finalized transactions:

**Transaction 1 – `unlockQueue`** (L301–307): rsETH is permanently burned via `burnFrom` (L305) and ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager` (L307). This transaction is finalized on-chain.

**Transaction 2 – `completeWithdrawal`** (L183 → L734): `_transferAsset` pushes ETH to `user`. If `user` is a contract with no `receive()` or a reverting fallback, the call returns `success = false`, `EthTransferFailed` is thrown, and the entire transaction reverts. The in-transaction state changes (nonce pop at L705, request deletion at L712, counter decrement at L717) are all rolled back, restoring the withdrawal request to the queue. However, the rsETH burn from Transaction 1 is not undone.

Recovery via `sweepRemainingAssets` is also permanently blocked at L403:
```solidity
if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();
```
Because `unlockedWithdrawalsCount[asset]` remains > 0 (the stuck withdrawal is still counted as unlocked after each revert of Transaction 2), the manager can never sweep the ETH to treasury either.

## Impact Explanation
This constitutes **Critical – Permanent freezing of funds**. The user's rsETH is irreversibly burned (Transaction 1 is finalized) while the corresponding ETH is permanently locked inside `LRTWithdrawalManager`. There is no on-chain function that can recover the ETH: `completeWithdrawal` always reverts for this user, and `sweepRemainingAssets` is gated behind `hasUnlockedWithdrawals` which remains true indefinitely.

## Likelihood Explanation
Smart-contract wallets (e.g., Gnosis Safe), multisigs, and protocol-owned treasuries routinely hold rsETH and initiate withdrawals. Many such contracts do not implement a `receive()` function or implement one that conditionally reverts. No privileged access is required; any such contract can trigger this state by calling `initiateWithdrawal` as an unprivileged user. The protocol's own NatDoc on `completeWithdrawalForUser` (L191: *"Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH"*) confirms developer awareness of the fragility of ETH push-delivery to arbitrary addresses, further supporting realistic likelihood.

## Recommendation
Replace the push-payment model for ETH withdrawals with a pull-payment (claimable) model:
- In `_processWithdrawalCompletion`, instead of calling `_transferAsset` immediately, record the owed amount in a `mapping(address => uint256) public claimableETH`.
- Add a separate `claimETH()` function that allows the user to pull their ETH at any time.
- This eliminates the dependency on the recipient's ability to receive ETH and removes the permanent-freeze risk entirely.

## Proof of Concept
1. A smart-contract wallet with no `receive()` holds rsETH and calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is **permanently burned** (`burnFrom` at L305) and ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager` (L307). Transaction finalized.
3. The smart-contract wallet calls `completeWithdrawal(ETH_TOKEN, "")`. Inside `_processWithdrawalCompletion`, `_transferAsset` executes `payable(wallet).call{ value: amount }("")`. The wallet has no `receive()`, so the call returns `success = false`. `revert EthTransferFailed()` is triggered; the entire transaction reverts.
4. In-transaction state (nonce pop, request deletion, counter decrement) is rolled back. rsETH burn (prior tx) is not undone. ETH remains in `LRTWithdrawalManager`.
5. Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` for this user repeats step 3 and reverts indefinitely.
6. `sweepRemainingAssets` is permanently blocked because `unlockedWithdrawalsCount[ETH_TOKEN] > 0` (L403).
7. The user's ETH is permanently frozen with no recovery path.
