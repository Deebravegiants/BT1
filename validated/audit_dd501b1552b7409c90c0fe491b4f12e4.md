### Title
Unbounded Sub-Transaction Plutus Execution Units Bypass `maxTxExUnits` Limit in Dijkstra Era — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces nested ("sub") transactions. The `SUBLEDGERS` rule iterates over every sub-transaction in a batch using an unbounded `foldM` loop. The sub-transaction UTxO validation rule (`SubUtxo.hs`) omits the `validateExUnitsTooBigUTxO` check that is present in every prior era's UTxO rule. The top-level `dijkstraUtxoTransition` applies `totExUnits tx ≤ maxTxExUnits pp` only to the top-level transaction's own witness set; sub-transactions carry independent witness sets whose redeemers are never summed into that check. As a result, an attacker can embed arbitrarily large Plutus execution-unit budgets across sub-transactions, forcing every validating node to execute those scripts well beyond the protocol-intended `maxTxExUnits` ceiling.

---

### Finding Description

**Root cause — missing execution-unit check in `SubUtxo.hs`**

Every era from Alonzo onward enforces:

```
totExunits tx ≤ maxTxExUnits pp
```

In Dijkstra's top-level UTXO rule this check is present:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [1](#0-0) 

`Alonzo.validateExUnitsTooBigUTxO` calls `totExUnits tx`, which sums redeemers from the **top-level** transaction's witness set (`tx ^. witsTxL . rdmrsTxWitsL`). [2](#0-1) 

Sub-transactions are stored in `dtbrSubTransactions :: OMap TxId (Tx SubTx era)` inside the top-level body and each carries its **own** `TxWits` (including its own redeemers). [3](#0-2) 

The sub-transaction UTxO validation rule (`subUtxoTransition` in `SubUtxo.hs`) performs output-size, network-ID, input-set, and min-value checks, but **never** calls `validateExUnitsTooBigUTxO`: [4](#0-3) 

**Root cause — unbounded `foldM` in `SUBLEDGERS`**

`dijkstraSubLedgersTransition` iterates over every sub-transaction with no explicit count limit:

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
``` [5](#0-4) 

Each `SUBLEDGER` invocation runs `SUBUTXOW` (script validation), `SUBENTITIES` (certificate processing), and `SUBGOV` (governance). The only implicit bound on the number of sub-transactions is `maxTxSize`, which limits serialized bytes — not Plutus execution cost. [6](#0-5) 

**No `dppMaxSubTransactions` protocol parameter exists** in `DijkstraPParams`: [7](#0-6) 

---

### Impact Explanation

An attacker submits a single Dijkstra top-level transaction whose serialized size is within `maxTxSize` but whose sub-transactions collectively declare Plutus redeemers with execution-unit budgets far exceeding `maxTxExUnits`. Because:

1. `totExUnits tx` only sums the top-level witness set — sub-transaction redeemers are invisible to it.
2. `SubUtxo.hs` never calls `validateExUnitsTooBigUTxO`.
3. `SUBLEDGERS` processes every sub-transaction unconditionally.

Every honest node must execute those Plutus scripts up to the declared budget before it can reject or accept the transaction. This forces computation well beyond the protocol-intended ceiling, matching the **Medium** allowed impact: *attacker-controlled transactions exceed intended validation limits*.

---

### Likelihood Explanation

The Dijkstra era is the newest era and sub-transactions are a novel feature. Any transaction author (unprivileged) can craft a top-level transaction with sub-transactions. No special role, key, or governance majority is required. The attacker only needs to construct a valid-sized transaction body containing sub-transactions with large redeemer `ExUnits` fields. This is a straightforward serialization exercise.

---

### Recommendation

1. **Add `validateExUnitsTooBigUTxO` to `subUtxoTransition`** in `SubUtxo.hs`, checking each sub-transaction's own execution units against `maxTxExUnits pp`.
2. **Extend `totExUnits`** for Dijkstra (or add a dedicated `totBatchExUnits`) to aggregate execution units across the top-level transaction **and** all sub-transactions, and apply the combined check in `dijkstraUtxoTransition`.
3. **Introduce a `dppMaxSubTransactions` protocol parameter** (analogous to `maxCollateralInputs`) to cap the number of sub-transactions per top-level transaction, providing a direct bound on the `foldM` iteration in `SUBLEDGERS`.
4. Consider applying `maxBlockExUnits` accounting to sub-transaction execution units in the `BBODY` rule as well.

---

### Proof of Concept

```
Construct a DijkstraTx with:
  - top-level body: 1 spend input, 1 output, fee, N sub-transactions
  - each sub-transaction body: 1 spend input, 1 output, 1 Plutus-locked output
  - each sub-transaction witness set: 1 redeemer with ExUnits { mem = maxBound, steps = maxBound }
  - top-level witness set: no redeemers (totExUnits tx == 0, passes maxTxExUnits check)
  - total serialized size: within maxTxSize

Result:
  - dijkstraUtxoTransition: totExUnits tx = 0 ≤ maxTxExUnits pp  → PASSES
  - SUBLEDGERS foldM: iterates over all N sub-transactions
  - Each SUBLEDGER → SUBUTXOW → SUBUTXO: no ExUnits check → PASSES
  - Each sub-transaction's Plutus script is executed with declared maxBound budget
  - Nodes spend O(N × maxBound) execution steps on a single transaction
  - maxTxExUnits is effectively bypassed for the sub-transaction batch
```

validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) $
    ExUnitsTooBigUTxO Mismatch {mismatchSupplied = totalExUnits, mismatchExpected = maxTxExUnits}
  where
    maxTxExUnits = pp ^. ppMaxTxExUnitsL
    -- This sums up the ExUnits for all embedded Plutus Scripts anywhere in the transaction:
    totalExUnits = totExUnits tx
