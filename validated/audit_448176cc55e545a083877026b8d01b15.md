### Title
Proposal Deposits Permanently Locked When `do_update` Clears Non-Executed Proposals - (File: `crates/contract/src/update.rs`)

---

### Summary

`propose_update` requires a storage-staking deposit proportional to the size of the proposed contract binary or config. When `do_update` executes a winning proposal, it bulk-clears **all** other pending proposals via `self.entries.clear()` — but issues no refund to the proposers of the discarded entries. Those deposits are permanently absorbed into the contract's balance with no recovery path.

---

### Finding Description

`propose_update` collects a deposit from each proposer: [1](#0-0) 

The required amount is computed from the serialized byte-size of the update payload (a full WASM binary can be hundreds of kilobytes): [2](#0-1) 

The `UpdateEntry` struct stores only `update` and `bytes_used` — **not** the proposer's `AccountId` or the deposit amount: [3](#0-2) 

When threshold votes are reached, `do_update` removes the winning entry and then bulk-clears every other entry and all votes — with no `Promise::transfer` call for any of the cleared proposals: [4](#0-3) 

Because the proposer identity and deposit amount are never stored in `UpdateEntry`, there is no structural way to issue refunds at clear-time. The freed storage reduces the contract's on-chain storage obligation, but the NEAR tokens that funded it remain in the contract's balance permanently.

---

### Impact Explanation

Every participant whose proposal is discarded by `do_update` loses their full storage deposit. For a contract binary update, NEAR's storage pricing (~1 NEAR per 100 KB) means a 500 KB WASM binary costs ~5 NEAR per proposal. In a governance round where multiple participants each propose a competing update (as demonstrated by `test_propose_update_contract_many`), all but one proposer permanently forfeit their deposits. [5](#0-4) 

This breaks the production accounting invariant that storage-staking deposits are returned when the associated storage is freed. The locked funds accumulate in the contract's balance with no admin withdrawal function and no expiry-based refund path.

---

### Likelihood Explanation

This is triggered by normal governance operation, not by any adversarial action. Any time two or more participants independently propose updates — a routine occurrence during upgrades — all but the winning proposer lose their deposit. The `only_one_vote_from_participant` sandbox test explicitly exercises the multi-proposal scenario: [6](#0-5) 

No collusion, no privileged escalation, and no external dependency failure is required. The loss is deterministic and silent.

---

### Recommendation

Store the proposer `AccountId` and the exact deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, iterate over the entries being cleared and issue a `Promise::new(entry.proposer).transfer(entry.deposit)` for each discarded proposal before calling `self.entries.clear()`. This mirrors the pattern already used for excess-deposit refunds in `propose_update` itself.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM binary, attaching ~5 NEAR.
2. Participant B calls `propose_update` with a different binary, attaching ~5 NEAR.
3. Threshold participants vote for B's proposal; `vote_update` calls `do_update`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry is dropped with no refund.
5. A's ~5 NEAR is permanently locked in the contract balance. No function exists to recover it. [7](#0-6)

async fn test_propose_update_contract_many() {
    let SandboxTestSetup {
        contract,
        mpc_signer_accounts,
        ..
    } = SandboxTestSetup::builder()
        .with_protocols(ALL_PROTOCOLS)
        .build()
        .await;
    dbg!(contract.id());

    const PROPOSAL_COUNT: usize = 2;
    let mut proposals = Vec::with_capacity(PROPOSAL_COUNT);
    // Try to propose multiple updates to check if they are being proposed correctly
    // and that we can have many at once living in the contract state.
    for i in 0..PROPOSAL_COUNT {
        let execution = mpc_signer_accounts[i % mpc_signer_accounts.len()]
            .call(contract.id(), method_names::PROPOSE_UPDATE)
            .args_borsh(current_contract_proposal())
            .max_gas()
            .deposit(CURRENT_CONTRACT_DEPLOY_DEPOSIT)
            .transact()
            .await
            .unwrap();

        assert!(
            execution.is_success(),
            "failed to propose update [i={i}]; {execution:#?}"
        );
        let proposal_id = execution.json().expect("unable to convert into UpdateId");
        proposals.push(proposal_id);
    }

    // Vote for the last proposal
    vote_update_till_completion(&contract, &mpc_signer_accounts, proposals.last().unwrap()).await;

    // Ensure all proposals are removed after update
    for proposal in proposals {
        let voter = mpc_signer_accounts.first().unwrap();
        let execution = voter
            .call(contract.id(), method_names::VOTE_UPDATE)
            .args_json(serde_json::json!({
                "id": proposal,
            }))
            .gas(GAS_FOR_VOTE_UPDATE)
            .transact()
            .await
            .unwrap();
        dbg!(&execution);

        assert!(execution.is_failure());
    }

    // Let's check that we can call into the state and see all the proposals.
    let state: ProtocolContractState = get_state(&contract).await;
    dbg!(state);
}
