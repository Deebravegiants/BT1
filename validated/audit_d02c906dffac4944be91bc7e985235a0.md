Audit Report

## Title
`maxFeeMintAmountPerDay` Uninitialized Causes `updateRSETHPrice()` to Permanently Revert When Protocol Fees Are Due - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle` declares `maxFeeMintAmountPerDay` but never initializes it in `initialize()` or `reinitialize()`, leaving it at `0`. When `protocolFeeInBPS > 0` is set in `LRTConfig` and TVL grows, `_updateRsETHPrice()` computes a non-zero fee amount and passes it to `_checkAndUpdateDailyFeeMintLimit()`, which unconditionally reverts because `0 + feeAmount > 0` is always true. This permanently blocks protocol fee collection and freezes the rsETH price oracle for all callers.

## Finding Description
`maxFeeMintAmountPerDay` is declared at [1](#0-0)  but is never assigned in `initialize()` [2](#0-1)  nor in `reinitialize()` [3](#0-2) , so it defaults to `0`.

When TVL increases and `protocolFeeInBPS > 0`, a non-zero `protocolFeeInETH` is computed: [4](#0-3) 

This leads to a non-zero `rsethAmountToMintAsProtocolFee` being passed to `_checkAndUpdateDailyFeeMintLimit`: [5](#0-4) 

Inside that function, the guard `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay` evaluates to `0 + feeAmount > 0`, which is always `true` for any non-zero fee, causing an unconditional revert: [6](#0-5) 

The only setter requires an explicit manager call post-deployment: [7](#0-6) 

`protocolFeeInBPS` lives in a separate contract and is set independently with no cross-check: [8](#0-7) 

Both `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` call `_updateRsETHPrice()` internally, so neither provides a bypass. [9](#0-8) 

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield + temporary freezing of funds.**

1. Protocol fees (rsETH minted to treasury) can never be collected while `maxFeeMintAmountPerDay == 0` and `protocolFeeInBPS > 0` — this is permanent freezing of unclaimed yield, a valid Medium impact.
2. `updateRSETHPrice()` reverts for every caller whenever TVL has grown, leaving the rsETH price stale. A stale price distorts deposit and withdrawal accounting for all users — temporary freezing of funds, a valid Medium impact.

## Likelihood Explanation
**Medium.** The manager enabling fees via `setProtocolFeeBps()` is a normal operational action. `maxFeeMintAmountPerDay` has no default and no initialization path, so any deployment that enables fees without an explicit `setMaxFeeMintAmountPerDay()` call triggers the revert. The two settings live in different contracts with no enforcement of ordering or coordination, making the oversight realistic in a live deployment.

## Recommendation
Modify `_checkAndUpdateDailyFeeMintLimit` to treat `maxFeeMintAmountPerDay == 0` as "no cap" rather than "zero cap":

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }
    if (maxFeeMintAmountPerDay != 0 &&
        currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
    }
    currentPeriodMintedFeeAmount += feeAmount;
}
```

Alternatively, initialize `maxFeeMintAmountPerDay` to a safe non-zero default in `reinitialize()`.

## Proof of Concept
1. Deploy `LRTConfig` and `LRTOracle`; call `initialize()` on both.
2. Call `lrtConfig.setProtocolFeeBps(100)` — 1% fee enabled.
3. Do **not** call `lrtOracle.setMaxFeeMintAmountPerDay(...)` — `maxFeeMintAmountPerDay` remains `0`.
4. Simulate TVL growth: asset prices rise so `totalETHInProtocol > previousTVL`.
5. Call `lrtOracle.updateRSETHPrice()` (public, callable by anyone).
6. Transaction reverts: `DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)`.
7. rsETH price is permanently stale; protocol fees are never minted to treasury.
