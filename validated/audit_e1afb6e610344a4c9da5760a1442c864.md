### Title
`VerifyForeignTxRequest.tweak` Set by Contract but Ignored in Node Signing — Zero Tweak Hardcoded in `build_signature_request` - (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`)

### Summary

The `build_signature_request` function in the MPC node's `verify_foreign_tx` provider hardcodes `Tweak::new([0u8; 32])` when constructing the `SignatureRequest` for foreign-chain verification, completely ignoring the `tweak` field that the contract derives from the caller's `derivation_path`. This is the direct analog of the external report: a parameter is configured/set at one layer but silently not applied at the execution layer.

### Finding Description

The design specification in `docs/foreign-chain-transactions.md` explicitly requires that `verify_foreign_transaction()` derive a **distinct tweak** from the caller's `derivation_path` using a foreign-tx-specific prefix (`"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`), ensuring that the same `(predecessor_id, derivation_path)` pair never yields the same derived key across `sign()` and `verify_foreign_transaction()` calls.

The contract converts `VerifyForeignTransactionRequestArgs.derivation_path` into a `Tweak` stored in `VerifyForeignTransactionRequest.tweak` (via `args_into_verify_foreign_tx_request` in `crates/contract/src/dto_mapping.rs`). This tweak is persisted in the pending request and delivered to the node.

However, in the node's `build_signature_request`:

```rust
fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),   // ← request.tweak is silently discarded
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
}
```

`request.tweak` is never read. The zero tweak is passed to the threshold-signing protocol, so every `verify_foreign_transaction` request — regardless of `derivation_path` — is signed by the **base key** of the ForeignTx domain.

Confirming that `derive_foreign_tx_tweak` does not exist anywhere in the production source tree (only in `docs/foreign-chain-transactions.md`), the intended derivation function was designed but never implemented.

### Impact Explanation

**Medium.** This breaks the production safety invariant that different `derivation_path` values produce different signing keys for foreign-tx requests:

1. **All callers share the same signing key.** Every `verify_foreign_transaction` call, regardless of `derivation_path`, produces a signature from the ForeignTx domain's base key (zero-tweaked). A bridge contract at path `"bridge-v1"` and one at `"bridge-v2"` receive signatures from the identical key.

2. **Public-key mismatch.** A caller that derives the expected public key using their `derivation_path` (as the design intends) will compute a key that does not match the key that actually signed. Any on-chain verifier performing this check will reject all foreign-tx signatures, breaking the entire foreign-chain verification flow.

3. **Cross-path key reuse.** Conversely, a verifier that simply checks against the ForeignTx domain base key will accept signatures for any path, eliminating the path-based access control that the tweak derivation was designed to enforce.

This does not require threshold collusion or privileged access — it is triggered by any unprivileged caller submitting a `verify_foreign_transaction` request with any `derivation_path`.

### Likelihood Explanation

**High.** The code path is exercised on every `verify_foreign_transaction` call. The zero tweak is unconditional — there is no fallback or flag. Any deployment of the foreign-chain verification feature will exhibit this behavior immediately.

### Recommendation

Replace the hardcoded zero tweak in `build_signature_request` with the tweak stored in the request, which the contract already derives from `derivation_path`:

```diff
 Ok(SignatureRequest {
     id: request.id,
     receipt_id: request.receipt_id,
     payload: Payload::Ecdsa(payload_bytes),
-    tweak: Tweak::new([0u8; 32]),
+    tweak: request.tweak,
     entropy: request.entropy,
     timestamp_nanosec: request.timestamp_nanosec,
     domain: request.domain_id,
 })
```

Additionally, implement `derive_foreign_tx_tweak` in `crates/near-mpc-crypto-types/src/kdf.rs` using the foreign-tx-specific prefix (`"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`) and use it in `args_into_verify_foreign_tx_request` in `crates/contract/src/dto_mapping.rs` to ensure the contract-side tweak derivation is also domain-separated from the regular `sign()` tweak.

### Proof of Concept

1. Alice calls `verify_foreign_transaction` with `derivation_path = "alice-bridge"`.
2. The contract derives `tweak_alice = derive_tweak("alice.near", "alice-bridge")` and stores it in `VerifyForeignTransactionRequest.tweak`.
3. The node's `build_signature_request` discards `tweak_alice` and uses `Tweak::new([0u8; 32])`.
4. The threshold signature is produced over the ForeignTx domain base key (zero tweak).
5. Bob calls `verify_foreign_transaction` with `derivation_path = "bob-bridge"`.
6. Steps 2–4 repeat; Bob also receives a signature from the same base key.
7. Alice's bridge contract, expecting a signature from `base_key + tweak_alice`, rejects the signature. Bob's bridge contract, expecting `base_key + tweak_bob`, also rejects it.
8. Alternatively, if either bridge contract was written to accept the base key directly, it accepts signatures intended for the other bridge — cross-path key reuse. [1](#0-0) [2](#0-1) [3](#0-2)

```markdown
## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.
```
