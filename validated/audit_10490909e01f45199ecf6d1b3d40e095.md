### Title
`LRTUnstakingVault.transferAssetToNodeDelegator()` Ignores Pending Withdrawal Obligations, Enabling Temporary Freeze of User Withdrawals - (File: contracts/LRTUnstakingVault.sol)

---

### Summary

The `LRTUnstakingVault` contract exposes `transferAssetToNodeDelegator()` and `transferETHToNodeDelegator()` to the `ASSET_TRANSFER_ROLE`, which move assets out of the vault to NodeDelegator contracts for restaking. Neither function checks whether the vault must retain assets to service pending user withdrawal requests. This is a direct structural analog to M-04: a legitimate operational action can drain the vault, causing `unlockQueue()` to revert when it calls `redeem()`, temporarily freezing all queued user withdrawals.

---

### Finding Description

The withdrawal lifecycle in LRT-rsETH is:

1. User calls `LRTWithdrawalManager.initiateWithdrawal()`, which increments `assetsCommitted[asset]` and records the request.
2. An operator calls `unlockQueue()`, which reads `unstakingVault.balanceOf(asset)` as `totalAvailableAssets` and then calls `unstakingVault.redeem(asset, assetAmountUnlocked)` to pull funds into the withdrawal manager.
3. User calls `completeWithdrawal()` to receive the asset.

The critical step is `unlockQueue()` reading the vault's live balance:

```solidity
// LRTWithdrawalManager.sol:846-850
return UnlockParams({
    rsETHPrice: lrtOracle.rsETHPrice(),
    assetPrice: lrtOracle.getAssetPrice(asset),
    totalAvailableAssets: unstakingVault.balanceOf(asset)  // live vault balance
});
```

If `totalAvailableAssets == 0`, `unlockQueue()` reverts immediately:

```solidity
// LRTWithdrawalManager.sol:297
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
```

The vault's balance can be drained to zero by the `ASSET_TRANSFER_ROLE` via:

```solidity
// LRTUnstakingVault.sol:112-126
function transferAssetToNodeDelegator(uint256 ndcIndex, address asset, uint256 amount)
    external nonReentrant onlyAssetTransferRole onlySupportedAsset(asset)
{
    // ... no check against pending withdrawal amounts
    IERC20(asset).safeTransfer(nodeDelegator, amount);
}
```

```solidity
// LRTUnstakingVault.sol:132-146
function transferETHToNodeDelegator(uint256 ndcIndex, uint256 amount)
    external nonReentrant onlyAssetTransferRole onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    // ... no check against pending withdrawal amounts
    INodeDelegator(nodeDelegator).sendETHFromUnstakingVaultToNDC{ value: amount }();
}
```

The contract does define a `queuedWithdrawalsBuffer` intended to reserve vault funds for queued withdrawals, but this buffer is **only enforced in `getAssetsAvailableForInstantWithdrawal()`** — it is completely ignored in `transferAssetToNodeDelegator()` and `transferETHToNodeDelegator()`:

```solidity
// LRTUnstakingVault.sol:229-238
function getAssetsAvailableForInstantWithdrawal(address asset) external view ... {
    uint256 vaultBalance = balanceOf(asset);
    uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
    availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
}
```

The buffer protects instant withdrawals from the vault but does not protect queued withdrawals from being drained by the asset transfer role. There is no analogous guard in `transferAssetToNodeDelegator()`.

---

### Impact Explanation

When the `ASSET_TRANSFER_ROLE` moves assets from `LRTUnstakingVault` to a NodeDelegator (a routine restaking operation), the vault balance drops. If it drops below the amount needed for pending withdrawal requests, `unlockQueue()` either reverts with `AmountMustBeGreaterThanZero` (if fully drained) or processes fewer requests than expected. Users who have already burned/locked their rsETH via `initiateWithdrawal()` cannot progress to `completeWithdrawal()` until the vault is replenished — which requires completing an EigenLayer withdrawal cycle (typically 7+ days). This constitutes a **temporary freeze of user funds**.

**Impact: Medium — Temporary freezing of funds.**

---

### Likelihood Explanation

The `ASSET_TRANSFER_ROLE` is an operational role that routinely moves assets between protocol components (deposit pool → NodeDelegator → EigenLayer, and back). Moving assets from the unstaking vault to a NodeDelegator for restaking is a legitimate and expected operational action. Without an explicit guard, an operator performing routine asset management can inadvertently drain the vault while users have pending withdrawal requests. This is not a malicious scenario — it is a design gap identical in nature to M-04, where the lock manager legitimately withdraws profits and accidentally breaks refunds.

**Likelihood: Medium** — routine operational actions can trigger this without any malicious intent.

---

### Recommendation

Add a check in `transferAssetToNodeDelegator()` and `transferETHToNodeDelegator()` in `LRTUnstakingVault` that enforces the `queuedWithdrawalsBuffer` (or a computed pending-withdrawal reserve) before allowing the transfer. For example:

```solidity
function transferAssetToNodeDelegator(uint256 ndcIndex, address asset, uint256 amount)
    external nonReentrant onlyAssetTransferRole onlySupportedAsset(asset)
{
    uint256 vaultBalance = IERC20(asset).balanceOf(address(this));
    uint256 reserved = queuedWithdrawalsBuffer[asset];
    if (vaultBalance < reserved + amount) revert InsufficientVaultBalanceForWithdrawals();
    // ...
    IERC20(asset).safeTransfer(nodeDelegator, amount);
}
```

The same pattern should be applied to `transferETHToNodeDelegator()`.

---

### Proof of Concept

1. Alice calls `LRTWithdrawalManager.initiateWithdrawal(stETH, 10e18)`. Her rsETH is locked; `assetsCommitted[stETH] += X`.
2. The protocol operator (with `ASSET_TRANSFER_ROLE`) calls `LRTUnstakingVault.transferAssetToNodeDelegator(0, stETH, vaultBalance)`, moving all stETH out of the vault to a NodeDelegator for restaking. No revert occurs.
3. The unlock operator calls `LRTWithdrawalManager.unlockQueue(stETH, ...)`. `_createUnlockParams` reads `unstakingVault.balanceOf(stETH) == 0`. The function reverts at:
   ```solidity
   if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
   ```
4. Alice's withdrawal is frozen. She cannot call `completeWithdrawal()` because her request was never unlocked. Her rsETH remains locked in the withdrawal manager. The vault can only be replenished after a full EigenLayer withdrawal cycle completes (7+ days).
