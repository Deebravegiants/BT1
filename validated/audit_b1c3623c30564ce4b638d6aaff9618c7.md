### Title
Hard `assert` in Pool Wallet Singleton Sync Loop Permanently Drops Coin State, Corrupting Pool Membership Tracking - (File: `chia/wallet/wallet_state_manager.py`)

### Summary

Inside `_add_coin_states`, when processing a spent pool-wallet singleton coin, a `while` loop walks the chain of singleton state transitions. At each iteration it calls `get_coin_state` on the connected peer and then unconditionally asserts the result has exactly one entry. If the peer returns an empty list (adversarial or transient), an `AssertionError` is raised. The outer exception handler classifies this as a *permanent* failure (not a `PeerRequestException` or `aiosqlite.Error`), so it calls `retry_store.remove_state` and discards the coin state forever. The pool wallet's local state tracking is left permanently behind the on-chain state.

### Finding Description

In `chia/wallet/wallet_state_manager.py`, `_add_coin_states` processes pool-wallet singleton coins with this loop:

```python
while curr_coin_state.spent_height is not None:
    cs = await fetch_coin_spend_for_coin_state(curr_coin_state, peer)
    ...
    new_coin_state = await self.wallet_node.get_coin_state(
        [new_singleton_name], peer=peer, fork_height=fork_height
    )
    assert len(new_coin_state) == 1          # ← hard assertion
    curr_coin_state = new_coin_state[0]
``` [1](#0-0) 

The entire loop runs inside a single `async with self.db_wrapper.writer()` transaction. If the assertion fires, the transaction rolls back and the exception propagates to the outer handler:

```python
except Exception as e:
    ...
    if isinstance(e, (PeerRequestException, aiosqlite.Error)):
        await self.retry_store.add_state(coin_state, peer.peer_node_id, fork_height)
    else:
        await self.retry_store.remove_state(coin_state)   # ← permanent drop
    continue
``` [2](#0-1) 

`AssertionError` is not a `PeerRequestException` or `aiosqlite.Error`, so the coin state is permanently removed from `WalletRetryStore` and the background retry task (`_retry_failed_states`) will never re-queue it. [3](#0-2) 

The `get_coin_state` call that feeds the assertion is a network call to the connected peer. An adversarial (but otherwise unprivileged) full-node peer can return an empty list for any `RegisterForCoinUpdates` response, making the assertion fail deterministically on every attempt while that peer is connected. [4](#0-3) 

### Impact Explanation

The pool wallet's local state machine (`PoolWallet.apply_state_transition`) is never advanced past the point of failure. The wallet's view of the singleton tip diverges from the on-chain tip. Concretely:

- The wallet cannot generate valid `join_pool` / `self_pool` travel transactions because `get_current_state` returns a stale tip coin that has already been spent on-chain.
- Pool reward coins accumulate at the `p2_singleton` puzzle hash but the wallet, not knowing the correct current singleton state, cannot construct the spend bundle needed to claim them.
- The corruption is persistent across restarts because `retry_store` is a SQLite-backed persistent store. [5](#0-4) 

This matches the allowed High impact: *"Corruption of … pool membership state … with direct security impact"* and *"Permanent or long-lived inability for … farmers … to process … pool actions."*

### Likelihood Explanation

The attacker only needs to operate a full node that the victim wallet connects to (no keys, no admin access required). The wallet connects to full nodes by default and trusts their coin-state responses. A single malicious peer that consistently returns an empty `CoinState` list for the new singleton coin name is sufficient to trigger the permanent drop. The condition is also reachable non-maliciously during a race where the peer has not yet indexed the new singleton coin, though in that case the wallet would eventually reconnect to a correct peer and re-receive the state — unless the state was already removed from `retry_store` on the first failure.

### Recommendation

1. Replace the hard `assert` with a conditional that raises `PeerRequestException` (so the state is re-queued for retry, not permanently dropped):

```python
if len(new_coin_state) != 1:
    raise PeerRequestException(
        f"Expected 1 coin state for {new_singleton_name.hex()}, got {len(new_coin_state)}"
    )
```

2. Alternatively, break out of the `while` loop gracefully when the peer cannot supply the next state, leaving the partial progress committed and allowing the next sync cycle to continue from the last known tip.

### Proof of Concept

1. Wallet has a `PoolWallet` singleton in `LEAVING_POOL` state; the singleton coin is spent on-chain, creating a new singleton coin.
2. The wallet receives the spent `CoinState` for the old singleton coin from an adversarial peer.
3. `_add_coin_states` enters the `while` loop, fetches the coin spend, calls `apply_state_transition` successfully, then calls `get_coin_state([new_singleton_name])`.
4. The adversarial peer returns `[]`.
5. `assert len(new_coin_state) == 1` raises `AssertionError`.
6. The DB transaction rolls back; `retry_store.remove_state(coin_state)` is called.
7. `_retry_failed_states` never re-queues the state.
8. The wallet's pool wallet remains stuck at the old tip; `pw_status` reports the stale state indefinitely; `pw_join_pool` / `pw_self_pool` generate spends for an already-spent coin, which are rejected by the mempool. [6](#0-5) [7](#0-6)

class WalletRetryStore:
    """
    Persistent coin states that we have received but failed to add
    """

    db_wrapper: DBWrapper2

    @classmethod
    async def create(cls, db_wrapper: DBWrapper2) -> WalletRetryStore:
        self = cls()
        self.db_wrapper = db_wrapper
        async with self.db_wrapper.writer_maybe_transaction() as conn:
            await conn.execute(
                "CREATE TABLE IF NOT EXISTS retry_store( coin_state blob PRIMARY KEY, peer blob, fork_height int)"
            )

        return self

    async def get_all_states_to_retry(self) -> list[tuple[CoinState, bytes32, uint32]]:
        """
        Return all states that were failed to sync
        """

        async with self.db_wrapper.reader_no_transaction() as conn:
            rows = await conn.execute_fetchall("SELECT * from retry_store")

        return [(CoinState.from_bytes(row[0]), bytes32(row[1]), uint32(row[2])) for row in rows]

    async def add_state(self, state: CoinState, peer_id: bytes32, fork_height: uint32 | None) -> None:
        """
        Adds object to key val store. Obj MUST support __bytes__ and bytes() methods.
        """
        async with self.db_wrapper.writer_maybe_transaction() as conn:
            cursor = await conn.execute(
                "INSERT OR IGNORE INTO retry_store VALUES(?, ?, ?)",
                (bytes(state), peer_id, 0 if fork_height is None else fork_height),
            )
            await cursor.close()

    async def remove_state(self, state: CoinState) -> None:
        async with self.db_wrapper.writer_maybe_transaction() as conn:
            cursor = await conn.execute("DELETE FROM retry_store where coin_state=?", (bytes(state),))
            await cursor.close()

    async def apply_state_transition(
        self, new_state: CoinSpend, block_height: uint32, action_scope: WalletActionScope
    ) -> bool:
        """
        Updates the Pool state (including DB) with new singleton spends.
        The DB must be committed after calling this method. All validation should be done here. Returns True iff
        the spend is a valid transition spend for the singleton, False otherwise.
        """
        tip: tuple[uint32, CoinSpend] = await self.get_tip()
        tip_spend = tip[1]

        tip_coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(tip_spend)
        assert tip_coin is not None
        spent_coin_name: bytes32 = tip_coin.name()

        if spent_coin_name != new_state.coin.name():
            history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
            if new_state.coin.name() in [sp.coin.name() for _, sp in history]:
                self.log.info(f"Already have state transition: {new_state.coin.name().hex()}")
            else:
                self.log.warning(
                    f"Failed to apply state transition. tip: {tip_coin} new_state: {new_state} height {block_height}"
                )
            return False
