### Title
Atomicity Failure Between Resharing Completion and Foreign Chain Registration Causes Complete `verify_foreign_transaction` Denial - (File: `crates/contract/src/lib.rs`)

### Summary

The `get_supported_foreign_chains()` function enforces a strict intersection rule: every active participant must have registered their supported foreign chains for any chain to be considered supported. When resharing completes and a new participant is added to the active set, there is an unavoidable window before that participant calls `register_available_foreign_chain_config`. During this window, the intersection is empty and every `verify_foreign_transaction` call is rejected. A single Byzantine participant below the signing threshold can exploit this indefinitely by deliberately never registering.

### Finding Description

`verify_foreign_transaction()` gates on `get_supported_foreign_chains()`: [1](#0-0) 

`get_supported_foreign_chains()` computes the strict intersection — a chain is returned only if **every** active participant's account ID appears in the set of nodes that registered support for it: [2](#0-1) 

The two-step lifecycle is:

1. **Tx-set A** — Threshold participants call `vote_new_parameters`, resharing completes via `vote_reshared`, and the new participant's account ID enters `active_participants()`.
2. **Tx B** — The new participant calls `register_available_foreign_chain_config` (a separate, independent transaction).

Between A and B, `active_participant_account_ids` contains the new participant, but `nodes_supporting_chain` does not (no registration exists yet). The `is_superset` check therefore fails for every chain, and `get_supported_foreign_chains()` returns the empty set. Every concurrent or subsequent `verify_foreign_transaction` call panics with `ForeignChainNotSupported`.

A Byzantine participant can exploit this structurally:
1. Gets voted in by threshold-many honest participants acting in good faith.
2. Deliberately never calls `register_available_foreign_chain_config`.
3. The intersection remains empty for as long as they remain a participant.

The design documentation explicitly acknowledges this root cause: [3](#0-2) 

The proposed fix (whitelisted vs. available two-set model) is tracked as issue #3434 and is **not yet implemented** in the production contract code.

### Impact Explanation

All `verify_foreign_transaction` requests are rejected at the contract level with `ForeignChainNotSupported`. The Omnibridge inbound flow (foreign chain → NEAR) and any bridge service relying on this endpoint is completely halted for the duration of the attack. Users who have already committed funds on the foreign chain cannot obtain the MPC-signed attestation needed to claim them on NEAR, effectively freezing those assets until the participant registers or is removed via a subsequent resharing.

This matches **Medium: request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants** — the foreign-chain verification accounting invariant (accepted requests can reach signing threshold) is violated without any network-level DoS.

### Likelihood Explanation

Being voted in as a new participant requires threshold votes from existing participants, which is a meaningful barrier. However, once admitted, a single participant below the signing threshold can trigger the condition unilaterally and sustain it indefinitely at zero cost. The window also opens non-maliciously on every resharing if any new participant's node is slow to start and register — making the condition reachable in normal operations as well as adversarially.

### Recommendation

Implement the proposed two-set model described in `docs/design/calculating-supported-foreign-chains.md`: [4](#0-3) 

`verify_foreign_transaction` should gate on the **available** set (≥ `signing_threshold` active participants cover the chain) rather than the strict intersection of all participants. This eliminates the atomicity window: a single unregistered participant reduces availability only if it drops coverage below the signing threshold, and the rejection becomes temporary rather than total.

### Proof of Concept

```
Setup: n=10 participants, threshold=7, all supporting Ethereum.
       Resharing adds participant P_new (threshold honest votes).

Block B:   vote_reshared() completes → P_new enters active_participants()
Block B+1: User calls verify_foreign_transaction(Ethereum, ...)
           → get_supported_foreign_chains() evaluates:
               active_participant_account_ids = {P_0,...,P_9, P_new}  (11 accounts)
               nodes_supporting_chain[Ethereum] = {P_0,...,P_9}       (10 accounts)
               is_superset({P_0..P_9}, {P_0..P_9,P_new}) = FALSE
           → supported_chains = {}
           → panic: ForeignChainNotSupported { requested: Ethereum }

P_new never calls register_available_foreign_chain_config.
Every verify_foreign_transaction call continues to fail until
P_new registers or a new resharing removes them.
```

The root cause is in `get_supported_foreign_chains()` at `crates/contract/src/lib.rs` lines 2176–2218, and the entry point is `verify_foreign_transaction()` at lines 519–557. The enabling step (resharing) and the parameter step (chain registration) are separate transactions with no atomicity guarantee between them.

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }

Today, `get_supported_foreign_chains()` returns the **strict intersection** of every
active participant's registered chains, and `verify_foreign_transaction` rejects any

feature down. That is what this proposal fixes.

**File:** docs/design/calculating-supported-foreign-chains.md (L24-34)
```markdown
The network distinguishes the **whitelisted** set (vote-driven policy, `get_whitelisted_foreign_chains()`)
from the **available** set (servable right now, `get_available_foreign_chains()`):

- **Whitelisted** is derived purely from the on-chain RPC whitelist — **no per-node input can add or
  remove a chain**, so no single operator can change it.
- **Available** is computed dynamically from the per-node config reports: `C` is available iff
  ≥ `signing_threshold` active participants cover `C`. `available ⊆ whitelisted` always.

`verify_foreign_transaction(C)` is **rejected unless `C` is available**: the contract fails fast
instead of accepting a request that can't reach the signing threshold and letting it time out. The
rejection is temporary — `C` becomes serviceable again as soon as enough nodes report coverage.
```
