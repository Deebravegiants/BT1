### Title
Pending Sign Requests Cannot Be Invalidated, Enabling Byzantine Leader Node to Trigger Duplicate Callback Execution — (`crates/contract/src/pending_requests.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The NEAR MPC contract stores pending signature requests keyed by `(tweak, payload, domain_id)` — a deterministic value derived from the caller's account, derivation path, payload, and domain. There is no user-controlled nonce and no mechanism to invalidate a pending request. The contract intentionally queues duplicate submissions under the same key and drains all queued yields when a single `respond()` call arrives. A Byzantine leader node below the signing threshold can exploit this by deliberately delaying its response, inducing the user to re-submit the same request while the first yield is still live, and then calling `respond()` — causing every queued yield to fire simultaneously and delivering the same signature to the user's callback contract multiple times.

---

### Finding Description

**Root cause — contract-side `SignatureRequest` has no nonce:**

The contract-side `SignatureRequest` struct used as the `pending_signature_requests` map key contains only `tweak`, `payload`, and `domain_id`:

```rust
pub struct SignatureRequest {
    pub tweak: Tweak,
    pub payload: Payload,
    pub domain_id: DomainId,
}
``` [1](#0-0) 

The `tweak` is deterministically derived from `(predecessor_id, path)`:

```rust
pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
    let tweak = crate::kdf::derive_tweak(predecessor_id, path);
    SignatureRequest { domain_id: domain, tweak, payload }
}
``` [2](#0-1) 

There is no user-controlled nonce, timestamp, or unique submission ID in this key. Two calls to `sign()` with identical `(caller, path, payload, domain_id)` produce the exact same map key.

**Fan-out queue drains all yields on a single `respond()`:**

`push_pending_yield` appends a new `YieldIndex` to the queue for the key on every `sign()` call: [3](#0-2) 

`resolve_yields_for` — called from `respond()` — removes the entire queue and calls `promise_yield_resume` for every entry: [4](#0-3) 

This is confirmed by the contract test:

> "When: same caller submits the same request twice in close succession … Then: the entire queue is drained — both queued yields received the response." [5](#0-4) 

**No invalidation mechanism:**

There is no contract method that allows a user to cancel or invalidate a pending sign request. The only way a queued yield is removed without receiving the signature is via the NEAR yield-resume timeout (~200 blocks, ~200 seconds), handled by `pop_oldest_pending_yield`: [6](#0-5) 

**Attack path — Byzantine leader node:**

The node-side queue assigns a deterministic leader per request ID and retries up to `MAX_ATTEMPTS_PER_REQUEST_AS_LEADER = 10` times: [7](#0-6) 

A Byzantine node that is the elected leader for a request can:
1. Observe the sign request on-chain.
2. Deliberately not call `respond()` — consuming its retry budget without producing a result.
3. The user, seeing no response after a short wait (well within the ~200-block window), re-submits the identical `sign()` call.
4. Both `YieldIndex` entries are now live under the same `SignatureRequest` key.
5. The Byzantine node (or any honest node that eventually takes over as leader) calls `respond()` with a valid signature.
6. `resolve_yields_for` resumes **both** yields with the same signature bytes.
7. `return_signature_and_clean_state_on_success` fires twice for the user's callback contract. [8](#0-7) [9](#0-8) 

---

### Impact Explanation

The user's callback contract receives the same threshold signature twice. If the callback contract uses the signature to authorize and submit a foreign-chain transaction (the primary use case for chain signatures), it may submit the same transaction twice — causing a double-spend or double-execution on the foreign chain. This maps to the **High** allowed impact: *"Cross-chain replay … that causes invalid bridge execution or double-spend conditions."*

The MPC contract itself does not lose funds, but the duplicate callback delivery breaks the exactly-once execution invariant that integrators must rely on, and the contract provides no mechanism to prevent or recover from it.

---

### Likelihood Explanation

**Medium.** The attack requires:
- A Byzantine node that is the elected leader for the target request (probabilistic, but achievable by targeting requests where the attacker is leader).
- The user to re-submit the same request within the ~200-block (~200-second) yield window — a realistic behavior for any integrator that polls for a response and retries on apparent failure.
- No threshold collusion: a single node below the signing threshold suffices.

The window is wide enough (200 seconds) that any integrator with a retry timeout shorter than the NEAR yield timeout is vulnerable.

---

### Recommendation

1. **Add a user-controlled nonce to `SignRequestArgs`** and include it in the `SignatureRequest` map key. This makes every submission unique, preventing duplicate queuing for re-submissions.
2. **Alternatively, add a cancellation method** that allows the original caller to remove their pending yield from the queue (e.g., `cancel_sign_request(request: SignatureRequest)`), analogous to the external report's recommendation to allow payers to set the `processedTransferIntents` mapping.
3. **Document the re-submission risk** prominently in the contract README and SDK, advising integrators to wait for the full yield timeout before re-submitting and to implement idempotency guards in their callback handlers.

---

### Proof of Concept

```
1. Alice's contract calls sign(payload=P, path="eth/1", domain_id=0)
   → YieldIndex Y1 stored under key K=(tweak(Alice,"eth/1"), P, 0)

2. Byzantine node N is elected leader for this request.
   N deliberately does not call respond() for 30 seconds.

3. Alice's contract, seeing no response, calls sign(payload=P, path="eth/1", domain_id=0) again.
   → YieldIndex Y2 appended to queue under same key K.
   → pending_signature_requests[K] = [Y1, Y2]

4. N (or an honest fallback leader) calls respond(K, valid_signature).
   → resolve_yields_for drains [Y1, Y2], calling promise_yield_resume for both.

5. return_signature_and_clean_state_on_success fires twice for Alice's contract.
   → Alice's callback executes twice with the same valid signature.
   → Alice's contract submits the same Ethereum transaction twice → double-spend.
```

    request: K,

    K: BorshSerialize + BorshDeserialize + Clone + Ord,

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

pub(crate) fn pop_oldest_pending_yield<K>(requests: &mut LookupMap<K, Vec<YieldIndex>>, request: &K)
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let Some(queue) = requests.get_mut(request) else {
        return;
    };
    if queue.is_empty() {
        requests.remove(request);
        return;
    }
    queue.remove(0);
    if queue.is_empty() {
        requests.remove(request);
    }
}
