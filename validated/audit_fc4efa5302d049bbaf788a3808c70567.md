### Title
Fee-proposal bounds enforcement silently bypassed during V0_14_3 protocol-upgrade transition because `fee_proposals_window` returns `None` — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` enforces that a proposer's `fee_proposal_fri` lies within a tight margin of `fee_actual`. However, the entire bounds check is gated on `fee_actual` being `Some`. `fee_actual` is computed by `compute_fee_actual` as the median of the last `window_size` (10) entries in `fee_proposals_window`. If **any** entry in that window is `None` (i.e., a pre-V0_14_3 block), `compute_fee_actual` returns `None`, and the bounds check is skipped. During the V0_14_3 protocol upgrade, the window is filled with `None` entries from pre-V0_14_3 blocks for the first `window_size` blocks after the upgrade. A malicious proposer can exploit this to publish an arbitrary `fee_proposal_fri` — including `u128::MAX` — for every block in the transition window, permanently anchoring the fee market at an extreme value once the window fills with V0_14_3+ entries.

---

### Finding Description

**The guard that is bypassed:**

In `is_proposal_init_valid`, the fee-proposal bounds check is:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
``` [1](#0-0) 

When `fee_actual` is `None`, the entire block is skipped — no bounds are enforced.

**How `fee_actual` becomes `None` during the upgrade:**

`compute_fee_actual` iterates over heights `[height - window_size, height - 1]` in `fee_proposals_window`. If **any** height maps to `None` (pre-V0_14_3 block) or is absent, it returns `None`:

```rust
Some(None) | None => {
    warn!("Cannot compute fee_actual for height {height}: ...");
    return None;
}
``` [2](#0-1) 

Pre-V0_14_3 blocks have `fee_proposal_fri = None` by definition. The window is populated from `BlockHeaderWithoutHash.fee_proposal_fri` via `initialize_fee_proposals_window` and `try_sync`. At the V0_14_3 upgrade boundary (block N), heights `[N - window_size, N - 1]` all carry `None`. Therefore `compute_fee_actual` returns `None` for every block from N through N + window_size − 1 (10 blocks).

**The `fee_proposals_window` starts empty and is populated from storage:**

```rust
fee_proposals_window: BTreeMap::new(),
``` [3](#0-2) 

`initialize_fee_proposals_window` reads `fee_proposal_fri` from state-sync for the bootstrap range. Pre-V0_14_3 blocks return `None` for this field, so the window is populated with `None` entries for the entire pre-upgrade range. [4](#0-3) 

**`fee_actual` is passed to validation without any fallback:**

Both `validate_proposal` and `set_height_and_round` (for queued proposals) compute `fee_actual` from the window and pass it directly into `ProposalInitValidation`:

```rust
fee_actual: compute_fee_actual(
    &self.fee_proposals_window,
    init.height,
    VersionedConstants::latest_constants().fee_proposal_window_size,
),
``` [5](#0-4) 

When this returns `None`, the `if let (Some(fee_actual), ...)` pattern in `is_proposal_init_valid` silently skips the bounds check.

**`fee_proposal_fri` is bound into the `ProposalCommitment`:**

```rust
pub(crate) fn proposal_commitment_from(
    partial: PartialBlockHash,
    fee_proposal: Option<GasPrice>,
) -> ProposalCommitment {
    let Some(fee_proposal) = fee_proposal else {
        return ProposalCommitment(partial.0);
    };
    ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
}
``` [6](#0-5) 

Validators compute the commitment using the proposer's `fee_proposal_fri` from `ProposalInit`. Since no bounds are enforced during the transition window, any value the proposer supplies is accepted and voted on by all validators.

**The extreme value enters the window and permanently shifts `fee_actual`:**

After the transition window, `fee_actual` is the median of the 10 entries in the window — all of which were set by proposers during the unchecked period. If a malicious proposer controlled even 5 of those 10 slots and set `fee_proposal_fri = u128::MAX`, the median becomes `u128::MAX`. All subsequent blocks then have bounds `[u128::MAX / 1.002, u128::MAX]`, permanently anchoring the fee market at the maximum value.

The extreme `fee_proposal_fri` also flows into `calculate_next_l2_gas_price_for_fin` via `fee_actual`, affecting the `l2_gas_price` stored in every subsequent block header. [7](#0-6) 

---

### Impact Explanation

A malicious validator who is elected proposer for any block in the 10-block V0_14_3 transition window can publish an arbitrary `fee_proposal_fri`. Because the bounds check is completely skipped when `fee_actual` is `None`, all other validators accept the block and vote on the resulting `ProposalCommitment`. The extreme `fee_proposal_fri` value enters the sliding window, and once the window is fully populated with V0_14_3+ entries, `fee_actual` is computed from those extreme values. All subsequent blocks inherit the manipulated fee market, causing incorrect L2 gas price accounting for all users. This matches the "Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact" impact category.

---

### Likelihood Explanation

The V0_14_3 protocol upgrade is a one-time, scheduled network event. The transition window is exactly `fee_proposal_window_size` = 10 blocks. In a validator set of size N, the expected number of transition blocks where a given validator is proposer is 10/N. For a small validator set (e.g., N = 10), a malicious validator is expected to be proposer for exactly 1 transition block on average, which is sufficient to shift the median if combined with other manipulated entries. The attack requires no special privileges beyond being a registered validator.

---

### Recommendation

1. **Use `l2_gas_price` as a fallback reference when `fee_actual` is `None`**: Instead of skipping bounds enforcement entirely, enforce bounds relative to `self.l2_gas_price` when `fee_actual` is `None`. This preserves the startup/genesis behavior while preventing unbounded manipulation during the upgrade transition.

2. **Enforce an absolute cap on `fee_proposal_fri`**: Even when `fee_actual` is `None`, reject `fee_proposal_fri` values that exceed a configurable multiple of `min_gas_price` or the current `l2_gas_price`.

3. **Document the security implication explicitly**: The comment "During initiation (fee_actual is None, <window_size blocks), bounds are not enforced" does not distinguish between the genesis case (safe) and the protocol-upgrade transition case (exploitable). The two cases should be handled separately.

---

### Proof of Concept

1. Network runs pre-V0_14_3. All blocks have `fee_proposal_fri = None`.
2. Protocol upgrades to V0_14_3 at block N. `initialize_fee_proposals_window` populates heights `[N-10, N-1]` with `None`.
3. For blocks N through N+9, `compute_fee_actual` returns `None` (window contains `None` entries).
4. Malicious validator M is elected proposer for block N. M sets `fee_proposal_fri = u128::MAX` in `ProposalInit`.
5. `is_proposal_init_valid` reaches the bounds check: `fee_actual = None` → `if let (Some(fee_actual), Some(fee_proposal))` does not match → check is skipped. Block N is accepted.
6. M repeats for any subsequent transition blocks where M is elected proposer.
7. At block N+10, `compute_fee_actual` computes the median of `[u128::MAX, ..., u128::MAX, ...]` = `u128::MAX` (or a very large value if M controlled ≥5 slots).
8. `fee_proposal_bounds(u128::MAX, 2)` returns `(upper ≈ u128::MAX, lower ≈ u128::MAX)`.
9. All subsequent proposers must set `fee_proposal_fri ≈ u128::MAX` or their proposals are rejected.
10. `calculate_next_l2_gas_price_for_fin` receives `fee_actual ≈ u128::MAX`, setting `l2_gas_price ≈ u128::MAX` for all future blocks.
11. All user transactions are charged at the maximum possible L2 gas price.

    pub async fn initialize_fee_proposals_window(
        &mut self,
        start_height: BlockNumber,
    ) -> StateSyncClientResult<()> {
        const STATE_SYNC_RETRY_INTERVAL: Duration = Duration::from_millis(500);
        let window_size = VersionedConstants::latest_constants().fee_proposal_window_size;
        let window_end_height = start_height.0;
        let window_start_height = window_end_height.saturating_sub(window_size);
        let mut pending_heights: VecDeque<BlockNumber> =
            (window_start_height..window_end_height).map(BlockNumber).collect();
        while let Some(block_number) = pending_heights.pop_front() {
            match self.deps.state_sync_client.get_block(block_number).await {
                Ok(block) => self.record_fee_proposal(
                    block_number,
                    block.block_header_without_hash.fee_proposal_fri,
                ),
                Err(StateSyncClientError::StateSyncError(StateSyncError::BlockNotFound(_))) => {
                    warn!(
                        "State sync not ready for height {block_number}; re-queueing after \
                         {STATE_SYNC_RETRY_INTERVAL:?}"
                    );
                    pending_heights.push_back(block_number);
                    tokio::time::sleep(STATE_SYNC_RETRY_INTERVAL).await;
                }
                Err(e) => return Err(e),
            }
        }
        Ok(())
    }
