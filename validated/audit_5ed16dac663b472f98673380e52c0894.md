Audit Report

## Title
Missing Chainlink Staleness Checks in `ChainlinkPriceOracle.getAssetPrice()` Allows Stale Prices to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-related return values, accepting any price the feed returns without validation. A stale inflated price flows into `LRTDepositPool.getRsETHAmountToMint()` as a live numerator against a stored `rsETHPrice` denominator, causing depositors to receive excess rsETH and diluting the ETH-per-rsETH ratio for all existing holders.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` captures only the `price` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No checks are performed on `roundId`, `updatedAt`, or `answeredInRound`. By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 30–32 in the same repository performs all three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The exploit path is:

1. Chainlink feed goes stale with an inflated price (e.g., last heartbeat price before congestion).
2. `rsETHPrice` in `LRTOracle` is a stored value last set by `updateRSETHPrice()` at a prior block when the price was normal.
3. Any depositor calls `LRTDepositPool.depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()`.
4. `getRsETHAmountToMint()` computes: `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` — the numerator uses the live stale inflated price; the denominator uses the stored pre-staleness `rsETHPrice`.
5. The depositor receives more rsETH than the true ETH value of their deposit.

The asymmetry is structural: `getAssetPrice()` is a live call while `rsETHPrice` is a storage variable updated separately via `updateRSETHPrice()`. The SECURITY.md exclusion for "incorrect data supplied by third-party oracles" does not apply here — this is a code-level validation omission in the contract itself, as demonstrated by the same repository implementing the correct pattern in `ChainlinkOracleForRSETHPoolCollateral`.

## Impact Explanation
Each deposit made while the feed is stale mints excess rsETH unbacked by real ETH value. This excess rsETH represents a claim on protocol TVL that dilutes the ETH-per-rsETH ratio for all existing holders — constituting theft of unclaimed yield. This matches the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation
Chainlink feeds go stale during network congestion, node outages, or circuit-breaker events. No privileged access is required; any unprivileged depositor can call `depositAsset()` or `depositETH()` during the stale window. The window persists until either the feed updates or `updateRSETHPrice()` is called with the stale price (which would then also inflate `rsETHPrice`, closing the window — but any deposits made before that call are already exploited). **Likelihood: Medium**.

## Recommendation
Apply the same staleness checks already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt != 0, "Incomplete round");
    require(price > 0, "Invalid price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept

**Root cause** — `ChainlinkPriceOracle.sol` line 52 discards all staleness fields: [1](#0-0) 

**Contrast** — `ChainlinkOracleForRSETHPoolCollateral.sol` performs all three checks: [2](#0-1) 

**Propagation** — `LRTOracle.getAssetPrice()` passes the stale price through without any additional validation: [3](#0-2) 

**Exploit trigger** — `getRsETHAmountToMint()` uses the live stale price against the stored `rsETHPrice`: [4](#0-3) 

**Entry point** — any unprivileged depositor triggers this via `depositAsset()`: [5](#0-4) 

**Foundry fork test plan:**
1. Fork mainnet at a block where a supported LST Chainlink feed is live.
2. Record `rsETHPrice` (stored value).
3. `vm.mockCall` the feed's `latestRoundData()` to return an inflated `price` with `answeredInRound < roundId` (stale condition).
4. Call `depositAsset()` as an unprivileged address with a known `depositAmount`.
5. Assert that `rsethAmountToMint` exceeds `(depositAmount * truePrice) / rsETHPrice`, confirming excess rsETH is minted.
6. Assert that calling the equivalent `ChainlinkOracleForRSETHPoolCollateral.getRate()` with the same mock reverts with `StalePrice`, confirming the fix works.
