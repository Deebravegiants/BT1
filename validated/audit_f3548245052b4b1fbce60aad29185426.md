### Title
Predictable Bridge Withdrawal UID Generation Enables Collision-Based DoS and Potential Permanent Fund Freeze - (File: `contracts/bridges/SonicChainNativeTokenBridge.sol`)

---

### Summary

`SonicChainNativeTokenBridge.bridgeTokenToL1` generates a `uint96` withdrawal UID using only fully predictable on-chain values. Two calls sharing the same `msg.sender`, `amount`, `tx.gasprice`, `block.timestamp`, and `block.number` produce an identical UID. When the UID is passed to `sonicBridge.withdraw(uid, ...)`, a collision either causes the second bridge operation to revert (temporary DoS) or, if the Sonic bridge accepts duplicate UIDs, permanently freezes the second withdrawal's tokens on L1 because `SonicBridgeReceiver.claimedWithdrawals[uid]` will already be marked `true` after the first claim.

---

### Finding Description

In `bridgeTokenToL1`, the UID is computed as:

```solidity
uint96 uid = uint96(
    uint256(
        keccak256(
            abi.encodePacked(
                block.timestamp, block.number, msg.sender, recipient, amount, tx.gasprice, bridgeReceiver
            )
        )
    ) % type(uint96).max
);
``` [1](#0-0) 

Every input is either a fixed constant (`bridgeReceiver = address(this)`, `recipient` is always `l1VaultETHForL2Chain` when called through the pool) or a block-level value shared by all transactions in the same block (`block.timestamp`, `block.number`). The only per-call differentiators are `msg.sender`, `amount`, and `tx.gasprice`. There is no monotonic counter, no `tx.origin`, no `blockhash`, and no other source of per-call entropy.

The fallback path (triggered when the primary UID is zero) is even weaker — it drops `block.timestamp`, `recipient`, and `amount` entirely:

```solidity
uid = uint96(uint256(keccak256(abi.encodePacked(block.number, msg.sender, tx.gasprice, bridgeReceiver))));
``` [2](#0-1) 

`bridgeTokenToL1` has **no access control** — any address that holds the bridged token and approves the bridge can call it directly, independent of the pool's `BRIDGER_ROLE` gate. [3](#0-2) 

The UID is then submitted to the Sonic bridge as the canonical withdrawal identifier:

```solidity
sonicBridge.withdraw(uid, originalToken, amount);
``` [4](#0-3) 

On L1, `SonicBridgeReceiver` tracks claimed withdrawals by this same UID:

```solidity
mapping(uint256 withdrawalId => bool claimed) public claimedWithdrawals;
...
if (claimedWithdrawals[withdrawalId]) revert WithdrawalAlreadyClaimed();
claimedWithdrawals[withdrawalId] = true;
``` [5](#0-4) [6](#0-5)

### Impact Explanation

**Scenario A — Sonic bridge enforces UID uniqueness (rejects duplicate):**
The second `sonicBridge.withdraw(uid, ...)` reverts. The `BridgeFailed()` check at line 119 propagates the revert, rolling back the entire transaction including the `safeTransferFrom`. The bridge operation silently fails; the bridger must retry. This is a **temporary freeze / DoS** of the bridge operation.

**Scenario B — Sonic bridge accepts duplicate UIDs:**
Both withdrawals are registered under the same UID. On L1, the first `claimAndTransferToVault(uid, ...)` succeeds and sets `claimedWithdrawals[uid] = true`. The second claim reverts with `WithdrawalAlreadyClaimed`. The tokens from the second withdrawal are permanently locked inside the Sonic bridge on L1 with no recovery path in `SonicBridgeReceiver` (the `emergencyRecover` function only recovers tokens already held by `SonicBridgeReceiver`, not tokens stuck in the Sonic bridge itself). This is a **permanent freeze of funds**. [7](#0-6) 

---

### Likelihood Explanation

The collision requires identical `msg.sender`, `amount`, `tx.gasprice`, `block.timestamp`, and `block.number`. When `bridgeTokenToL1` is called through `RSETHPoolV3WithNativeChainBridge.bridgeTokens`, `msg.sender` is always the fixed pool contract address and `recipient` is always `l1VaultETHForL2Chain`. [8](#0-7) 

This means two pool-initiated bridge calls in the same block with the same `amount` and `tx.gasprice` produce an identical UID. An automated bridger bot processing multiple bridge requests in a single block, or a direct caller submitting two transactions with identical parameters in the same block, can trigger this. Likelihood is **low** under normal operation but non-negligible for automated bridging infrastructure.

---

### Recommendation

Replace the hash-based UID with a monotonically incrementing per-contract nonce that is stored in contract state:

```solidity
uint96 private _withdrawalNonce;

// Inside bridgeTokenToL1:
uint96 uid = ++_withdrawalNonce;
```

Alternatively, incorporate `tx.origin`, the contract's own nonce, or a `blockhash` of a recent block to ensure per-call uniqueness. The nonce approach is the simplest and most robust, as it guarantees uniqueness regardless of block context or caller parameters.

### Proof of Concept

1. Deploy `SonicChainNativeTokenBridge` on Sonic chain (deployed at `0x7C39d591005b580df6CB63EFfCd0872AF0f48c8D`).
2. Obtain two units of the bridged token and approve the bridge for both.
3. In a single block, submit two transactions calling `bridgeTokenToL1(recipient, X)` with identical `amount = X` and identical `tx.gasprice`.
4. Both transactions share `block.timestamp`, `block.number`, `msg.sender` (same caller), `recipient`, `amount`, `tx.gasprice`, and `bridgeReceiver`.
5. `keccak256(abi.encodePacked(...))` produces the same hash → same `uid` for both calls.
6. If the Sonic bridge accepts both: two withdrawals are registered under the same UID. On L1, `claimAndTransferToVault(uid, ...)` succeeds for the first and reverts with `WithdrawalAlreadyClaimed` for the second, permanently freezing the second withdrawal's tokens.

    function claimAndTransferToVault(
        uint256 withdrawalId,
        address token,
        uint256 amount,
        bytes calldata proof
    )
        external
        nonReentrant
        onlyRole(CLAIMER_ROLE)
    {
        if (withdrawalId == 0) revert InvalidWithdrawalId();
        if (claimedWithdrawals[withdrawalId]) revert WithdrawalAlreadyClaimed();

        // Mark as claimed before external call to prevent reentrancy
        claimedWithdrawals[withdrawalId] = true;

        // Get balance before claim
        uint256 balanceBefore = IERC20(token).balanceOf(address(this));

        // Claim from Sonic bridge
        ETHEREUM_TOKEN_DEPOSIT.claim(withdrawalId, token, amount, proof);
        emit WithdrawalClaimed(withdrawalId, token, amount);

        // Verify we received the tokens and transfer to L1Vault
        uint256 balanceAfter = IERC20(token).balanceOf(address(this));
        uint256 received = balanceAfter - balanceBefore;
        if (received == 0) revert InsufficientBalance();

        // Transfer tokens to L1Vault
        IERC20(token).safeTransfer(l1Vault, received);
        emit TokensTransferredToVault(token, received, l1Vault);
    }
```
