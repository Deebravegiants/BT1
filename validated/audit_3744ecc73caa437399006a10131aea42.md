Audit Report

## Title
Free Share Minting in Drained Bins Dilutes Existing LP Claims — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
In `LiquidityLib.addLiquidity`, when a bin has been fully drained by swaps (`token0BalanceScaled == 0` and `token1BalanceScaled == 0`) but still carries positive `binTotalShares`, the proportional share-price calculation yields zero for both token amounts. Because the token-payment callback is gated on `amount0Added > 0 || amount1Added > 0`, an attacker can mint an arbitrary number of shares in such a bin at zero cost. When the price reverses and swaps refill the bin, the attacker's free shares entitle them to a proportional fraction of the incoming tokens, directly stealing from existing LPs.

## Finding Description
In the `else` branch of the share-price calculation (lines 108–111), when `binTotalSharesVal > 0` and both `binState.token0BalanceScaled` and `binState.token1BalanceScaled` are zero:

```solidity
amount0Scaled = Math.ceilDiv(_checkedMul(0, sharesToAdd), binTotalSharesVal); // = 0
amount1Scaled = Math.ceilDiv(_checkedMul(0, sharesToAdd), binTotalSharesVal); // = 0
```

The share and balance state updates at lines 120–121 are unconditional:

```solidity
binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd;
positionBinShares[posKey] = newUserShares;
```

The token-payment callback at lines 144–148 is only invoked when `amount0Added > 0 || amount1Added > 0`, which is false here, so no tokens are transferred. The drained-bin state is reachable because swap logic updates `binState.token0BalanceScaled`/`token1BalanceScaled` but never touches `_binTotalShares`. No existing guard checks for the combination of `binTotalSharesVal > 0` with both balances at zero before writing share state.

## Impact Explanation
When the price reverses and `zeroForOne` swaps refill the drained bin, `removeLiquidity` computes each LP's entitlement proportionally to their share of `binTotalShares`. The attacker's free shares dilute all existing LP positions: legitimate LPs receive fewer tokens than they deposited, constituting a direct loss of user principal. The loss scales with the ratio of free shares minted to pre-existing shares. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation
Bins being fully drained is a routine market event in any active pool — it occurs whenever the oracle price moves far enough that all liquidity in a bin is consumed by swaps. The attacker requires no privileged access: `addLiquidity` is open to any caller with any `owner` address. The attacker only needs to monitor `BinSwapped` events, detect when a bin's scaled balance reaches zero while `_binTotalShares` remains positive, and call `addLiquidity` before the price reverses. The attack is repeatable across any pool and any bin.

## Recommendation
Add a guard in `LiquidityLib.addLiquidity` inside the `else` branch (after computing `amount0Scaled` and `amount1Scaled`) that reverts when `binTotalSharesVal > 0` and both computed amounts are zero:

```solidity
if (amount0Scaled == 0 && amount1Scaled == 0) {
    revert DrainedBin(binIdx);
}
```

This enforces the invariant that shares must always represent a non-zero token claim, mirroring the behavior of the `binTotalSharesVal == 0` branch which always computes a non-zero amount via `initialScaledToken0PerShareE18`/`initialScaledToken1PerShareE18`.

## Proof of Concept
1. Pool deployed with bins `[-1, 0, 1]`. Alice calls `addLiquidity` for bin `1` (above current price), depositing token0 and receiving 10,000 shares. `_binTotalShares[1] = 10_000`, `_binStates[1].token0BalanceScaled > 0`.
2. A series of `!zeroForOne` swaps (token1 in, token0 out) fully consume bin `1`: `_binStates[1].token0BalanceScaled = 0`, `_binStates[1].token1BalanceScaled = 0`. `_binTotalShares[1]` remains `10_000` (swaps never modify share accounting).
3. Bob calls `addLiquidity(owner=Bob, salt=0, deltas={binIdxs:[1], shares:[10_000]}, ...)`.
   - `binTotalSharesVal = 10_000 > 0` → proportional branch taken.
   - `amount0Scaled = ceilDiv(0 * 10_000, 10_000) = 0`.
   - `amount1Scaled = ceilDiv(0 * 10_000, 10_000) = 0`.
   - `totalToken0ToAddScaled = 0`, `totalToken1ToAddScaled = 0` → `amount0Added = 0`, `amount1Added = 0`.
   - Callback guard is false; no tokens transferred.
   - `_binTotalShares[1]` updated to `20_000`; `_positionBinShares[key(Bob,0,1)]` set to `10_000`. **No tokens paid.**
4. Price reverses; `zeroForOne` swaps refill bin `1` with 1,000 units of token0 (scaled). `_binStates[1].token0BalanceScaled = 1_000`.
5. Bob calls `removeLiquidity` for his 10,000 shares:
   - `amount0Scaled = 1_000 * 10_000 / 20_000 = 500` → Bob receives 500 units of token0 he never paid for.
6. Alice calls `removeLiquidity` for her 10,000 shares:
   - `amount0Scaled = 500 * 10_000 / 10_000 = 500` → Alice receives only 500 units instead of 1,000.
   - Alice loses 500 units of token0 she legitimately deposited.
