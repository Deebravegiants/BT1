### Title
`toPlutusScriptPurpose` for `PlutusV4` in `DijkstraEra` is a stub that always throws a runtime exception, permanently breaking PlutusV4 script validation — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs`)

---

### Summary

The `EraPlutusTxInfo 'PlutusV4 DijkstraEra` instance declares full support for PlutusV4 scripts (including registering PlutusV4 as a supported language via `mkSupportedLanguage`), but its `toPlutusScriptPurpose` method is an unconditional stub: `error "stub: PlutusV4 not yet implemented"`. This is called from `toPlutusV4Args` (the `toPlutusArgs` implementation for PlutusV4) every time the ledger attempts to build a script context for a PlutusV4 script. Any unprivileged user can trigger this by submitting a Dijkstra-era transaction that uses a PlutusV4 script.

---

### Finding Description

In `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs`, the `EraPlutusContext DijkstraEra` instance advertises PlutusV4 as a supported language:

```haskell
-- line 228-232
mkSupportedLanguage = \case
  PlutusV1 -> Just $ SupportedLanguage SPlutusV1
  PlutusV2 -> Just $ SupportedLanguage SPlutusV2
  PlutusV3 -> Just $ SupportedLanguage SPlutusV3
  PlutusV4 -> Just $ SupportedLanguage SPlutusV4   -- ← PlutusV4 is "supported"
```

The `EraPlutusTxInfo 'PlutusV4 DijkstraEra` instance then provides `toPlutusArgs = toPlutusV4Args`. That function calls `toPlutusScriptPurpose`:

```haskell
-- lines 589-592
toPlutusV4Args proxy pv txInfo plutusPurpose maybeSpendingData redeemerData = do
  scriptPurpose <- toPlutusScriptPurpose proxy pv plutusPurpose   -- ← called here
  ...
```

But `toPlutusScriptPurpose` for PlutusV4 is:

```haskell
-- line 498
toPlutusScriptPurpose _ = error "stub: PlutusV4 not yet implemented"
```

This is a Haskell `error` call — not a graceful `Left` in the `Either` monad — so it throws an `ErrorCall` exception that escapes the `Either (ContextError era)` return type entirely. Unlike `transFailUnsupportedScriptInSubTx` (which correctly returns `Left (UnsupportedScriptInSubTx ...)` for PlutusV1–V3 sub-transactions), this stub bypasses the ledger's structured error-handling path.

The call chain for any Dijkstra transaction containing a PlutusV4 script is:

```
collectTwoPhaseScriptInputs
  → toPlutusArgs (= toPlutusV4Args)
    → toPlutusScriptPurpose
      → error "stub: PlutusV4 not yet implemented"   ← uncaught exception
```

---

### Impact Explanation

Two outcomes are possible depending on whether the node's outer exception handler catches `ErrorCall`:

1. **If uncaught**: The node process crashes while applying a block containing a PlutusV4 transaction. Honest nodes that crash diverge from nodes that do not, causing a permanent ledger split that requires a hard fork to resolve. *(Critical — ledger divergence)*

2. **If caught and treated as a block/tx rejection**: Every Dijkstra transaction that uses a PlutusV4 script is permanently rejected. Any ADA or native assets locked at a PlutusV4 script address are permanently frozen, because no spending transaction can ever pass validation. Recovery requires a hard fork. *(High — permanent freezing of funds)*

In either case the impact falls within the allowed scope. The more certain outcome is the High-severity permanent fund freeze, because the ledger's outer `try`/`catch` machinery typically converts unexpected exceptions into block rejections rather than crashing the process.

---

### Likelihood Explanation

- The Dijkstra era registers PlutusV4 as a fully supported language (`mkSupportedLanguage` returns `Just`), so the ledger will accept transactions that reference PlutusV4 scripts through phase-1 checks.
- Any unprivileged user can craft and submit such a transaction; no special role, key, or governance action is required.
- The stub is in a mandatory code path (`toPlutusArgs`) that is exercised for every PlutusV4 script in every transaction.

---

### Recommendation

Implement `toPlutusScriptPurpose` for `EraPlutusTxInfo 'PlutusV4 DijkstraEra` to correctly translate each `PlutusPurpose AsIxItem DijkstraEra` value into the corresponding `PV3.ScriptInfo` (reusing the existing `Conway.transPlutusPurposeV3` logic or a Dijkstra-specific variant). Until a correct implementation is ready, either:

- Remove `PlutusV4 -> Just $ SupportedLanguage SPlutusV4` from `mkSupportedLanguage` so the ledger rejects PlutusV4 transactions at phase-1 with a proper predicate failure, **or**
- Replace the `error` stub with `Left $ inject $ SomeContextError ...` so that failures are handled gracefully within the `Either` monad.
