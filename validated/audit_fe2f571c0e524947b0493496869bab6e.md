Audit Report

## Title
Composite `RSETHPriceFeed.latestRoundData()` Returns `updatedAt` Solely from the ETH/USD Feed, Masking Staleness of the rsETH/ETH Component - (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary
`RSETHPriceFeed.latestRoundData()` computes a composite rsETH/USD price by multiplying the ETH/USD Chainlink answer by `LRTOracle.rsETHPrice`, but propagates `updatedAt` exclusively from the ETH/USD feed. Because `LRTOracle.rsETHPrice` is a manually-triggered storage variable with no on-chain timestamp, the contract structurally cannot reflect the true staleness of the composite answer, causing it to fail its Chainlink-compatible interface contract.

## Finding Description
`RSETHPriceFeed.latestRoundData()` (lines 63–70) fetches all five return values from `ETH_TO_USD.latestRoundData()` and then overwrites only `answer` with the composite product:

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

`updatedAt` is left as the ETH/USD heartbeat timestamp. `LRTOracle.rsETHPrice` (line 28) is a plain `uint256` storage variable written only inside `_updateRsETHPrice()` (line 313), which is invoked by the public `updateRSETHPrice()` (line 87) or the manager-gated `updateRSETHPriceAsManager()` (line 94). No timestamp of the last write is stored or exposed anywhere in `LRTOracle`. The `PriceAboveDailyThreshold` guard (lines 260–265) can additionally block public callers from updating `rsETHPrice` when the price rises above the configured limit, extending the potential staleness window further. As a result, `RSETHPriceFeed` has no mechanism to incorporate the rsETH/ETH component's age into `updatedAt`, making the returned staleness metadata structurally incomplete for any composite-price consumer.

## Impact Explanation
The contract fails to deliver the accurate, timely staleness metadata it promises as a Chainlink `AggregatorV3Interface` implementation. Any consumer applying a standard staleness guard (`block.timestamp - updatedAt <= MAX_STALENESS`) will pass the check as long as the ETH/USD feed is live, regardless of how old `rsETHPrice` is. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value (within the LRT-rsETH protocol itself).**

## Likelihood Explanation
`updateRSETHPrice()` is public but requires an explicit external call; it is not self-triggering. Keeper downtime, network congestion, or a `PriceAboveDailyThreshold` revert can all delay updates to `rsETHPrice` for hours or days while the ETH/USD Chainlink feed continues to update on its normal ~1-hour heartbeat. This is a realistic operational scenario requiring no privileged access or attacker action — the staleness condition arises passively.

## Recommendation
Store the timestamp of the last `rsETHPrice` write in `LRTOracle` and expose it via the `IRSETHOracle` interface. In `latestRoundData()`, return the minimum of the two timestamps:

```solidity
uint256 rsETHUpdatedAt = RS_ETH_ORACLE.rsETHUpdatedAt();
if (rsETHUpdatedAt < updatedAt) updatedAt = rsETHUpdatedAt;
```

This ensures consumers see the true staleness of the composite price. The same fix should be applied to `getRoundData()`.

## Proof of Concept
1. At `T=0`, `updateRSETHPrice()` is called; `LRTOracle.rsETHPrice` is set to `1.05e18`.
2. At `T=2h`, the ETH/USD Chainlink feed updates normally; `ETH_TO_USD.latestRoundData()` returns `updatedAt = T+2h`.
3. At `T=25h`, the keeper is down; `rsETHPrice` has not been updated. A slashing event has reduced the true rsETH/ETH rate to `1.00e18`.
4. A consumer calls `RSETHPriceFeed.latestRoundData()` and receives `updatedAt = T+24h` (fresh-looking, from the most recent ETH/USD heartbeat) paired with `answer = 1.05e18 * ETH_USD_price / 1e18` (stale rsETH component).
5. The consumer's staleness guard (e.g., `MAX_STALENESS = 2h`, `block.timestamp - updatedAt = 1h`) passes.
6. The consumer prices rsETH ~5% above its true value.
