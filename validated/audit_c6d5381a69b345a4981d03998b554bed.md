### Title
Unconstrained `fee_proposal_fri` During Startup Window Poisons `fee_actual` and Future L2 Gas Prices — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

When `fee_actual` is `None` — which occurs for the first `fee_proposal_window_size` blocks after genesis or restart — the `fee_proposal_fri` bounds check inside `is_proposal_init_valid` is completely skipped via an `if let (Some(...), Some(...))` guard. Any proposer during this window can embed an arbitrary `fee_proposal_fri` value in `ProposalInit`, have it accepted by all validators, and have it stored in `fee_proposals_window`. Once the window fills, `compute_fee_actual` returns the median of these poisoned values, permanently skewing the L2 gas price for all subsequent blocks.

This is the direct sequencer analog of the Uniswap V3 zero-liquidity price manipulation: a zero/null critical value (`liquidity = 0` / `fee_actual = None`) causes a guard condition to be trivially bypassed, allowing arbitrary state manipulation (pool price / L2 gas price).

---

### Impact Explanation

**Impact: High — Incorrect fee/gas accounting with economic impact.**

A proposer during the startup window (heights `0` to `window_size - 1`) can set `fee_proposal_fri` to any value — e.g., `u128::MAX` to inflate fees, or `0` to deflate them. All validators accept the proposal because the bounds check is gated on `fee_actual` being `Some`. The poisoned values fill `fee_proposals_window`. Once `height >= window_size`, `compute_fee_actual` returns the median of the poisoned window, which is then used as the floor/reference for the EIP-1559 L2 gas price adjustment for the next `window_size` blocks. This causes:

- **Inflation attack:** `fee_proposal_fri = u128::MAX` → `fee_actual = u128::MAX` → `l2_gas_price` for future blocks inflated → all users pay inflated fees or transactions are priced out entirely.
- **Deflation attack:** `fee_proposal_fri = 0` → `fee_actual = 0` → `l2_gas_price` collapses → network can be spammed with near-zero-cost transactions.

The effect persists for `window_size` blocks after the startup window ends (until the poisoned entries are pruned from the window).

---

### Likelihood Explanation

The startup window is a deterministic, predictable period. In a multi-validator BFT system (Tendermint), the proposer role rotates among validators. Any validator that holds the proposer role during heights `0` to `window_size - 1` can trigger this. The condition is reachable on every fresh chain deployment or after a node restart that resets the window. No special privileges beyond being a scheduled proposer are required.

---

### Recommendation

**Short term:** When `fee_actual` is `None`, bound `fee_proposal_fri` against the current `l2_gas_price` using the same `fee_proposal_margin_ppt` margin. This prevents arbitrary values during the startup window while still allowing the system to bootstrap:

```rust
let reference = match proposal_init_validation.fee_actual {
    Some(fa) => fa,
    None => proposal_init_validation.l2_gas_price_fri, // fallback reference
};
let (lower_bound, upper_bound) = fee_proposal_bounds(
    reference,
    VersionedConstants::latest_constants().fee_proposal_margin_ppt,
);
// enforce bounds unconditionally
```

**Long term:** Ensure the `fee_proposals_window` is pre-populated from persisted state on restart so that `fee_actual` is never `None` after the very first genesis window. This eliminates the attack surface entirely.

---

### Proof of Concept

1. Chain starts at height 0. `fee_proposal_window_size = W` (e.g., 10).
2. For heights `0` to `W-1`, the attacker holds the proposer role (or is one of the rotating proposers).
3. Attacker crafts `ProposalInit` with `fee_proposal_fri = Some(GasPrice(u128::MAX))`.
4. Validator calls `is_proposal_init_valid`. `proposal_init_validation.fee_actual = None` (because `height < W`). The `if let (Some(fee_actual), Some(fee_proposal))` guard short-circuits — no bounds check is performed.
5. Proposal is accepted. `record_fee_proposal(height, Some(GasPrice(u128::MAX)))` stores the poisoned value.
6. At height `W`, `compute_fee_actual` computes `median([u128::MAX, ..., u128::MAX]) = u128::MAX`.
7. `calculate_next_l2_gas_price_for_fin(..., fee_actual = Some(u128::MAX))` returns `u128::MAX` as the next L2 gas price.
8. All blocks from height `W` onward carry `l2_gas_price_fri = u128::MAX` in their `ProposalInit`, making every transaction prohibitively expensive.
9. The poisoned window persists until height `2W`, after which normal values re-enter the window.

The existing test `validate_empty_proposal` confirms that `fee_actual: None` is the default in test setup, and `fee_proposal_within_margin_of_fee_actual` confirms that bounds are only enforced when `fee_actual` is `Some` — the `None` path has no corresponding rejection test. pub fn compute_fee_actual(
    fee_proposals_window: &BTreeMap<BlockNumber, Option<GasPrice>>,
    height: BlockNumber,
    window_size: u64,
) -> Option<GasPrice> {
    let Some(start) = height.0.checked_sub(window_size) else {
        warn!(
            "Cannot compute fee_actual for height {height}: height is below window_size \
             ({window_size})"
        );
        return None;
    };
    let window_size_usize = usize::try_from(window_size).expect("window_size fits in usize");
    let mut window = Vec::with_capacity(window_size_usize);
    for source_height in (start..height.0).map(BlockNumber) {
        match fee_proposals_window.get(&source_height) {
            Some(Some(price)) => window.push(*price),
            Some(None) | None => {
                warn!(
                    "Cannot compute fee_actual for height {height}: fee_proposals_window has no \
                     recorded fee_proposal for height {source_height}"
                );
                return None;
            }
        }
    }
    window.sort();
    let mid = window_size_usize / 2;
    let median = if window_size_usize.is_multiple_of(2) {
        // Even: average of the two middle values, rounded down.
        // Overflow-safe averaging: a + (b - a) / 2 (safe because sorted, so b >= a).
        GasPrice(window[mid - 1].0 + (window[mid].0 - window[mid - 1].0) / 2)
    } else {
        window[mid]
    };
    Some(median)
