### Title
Out-of-Order LayerZero Message Delivery Permanently Overwrites L2 Rate with Stale Value — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

### Summary
`CrossChainRateReceiver.lzReceive()` accepts any message from the trusted provider without enforcing message ordering or checking that the incoming rate is newer than the currently stored rate. LayerZero v1 does not guarantee ordered delivery. Two rapid `updateRate()` calls can result in the older rate permanently overwriting the newer one on L2.

### Finding Description

`lzReceive()` receives the `_nonce` parameter but explicitly discards it (unnamed `uint64,`): [1](#0-0) 

After validating the endpoint, source chain, and source address, it unconditionally overwrites `rate` with whatever value is in the payload: [2](#0-1) 

The payload sent by `CrossChainRateProvider.updateRate()` encodes only the rate value — no timestamp, no nonce, no sequence number is embedded: [3](#0-2) 

The same flaw exists in `MultiChainRateProvider.updateRate()`: [4](#0-3) 

This affects all deployed `RSETHRateReceiver` and `AGETHRateReceiver` contracts, which inherit `CrossChainRateReceiver` without adding any ordering protection: [5](#0-4) [6](#0-5) 

### Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Attack sequence:
1. `updateRate()` called at T1 → LZ message M1 carrying `rate_old` is in flight.
2. `updateRate()` called at T2 → LZ message M2 carrying `rate_new > rate_old` is in flight.
3. M2 arrives first (possible under LZ v1, which does not guarantee ordering) → `rate = rate_new`.
4. M1 arrives second → `rate = rate_old`, overwriting the correct value.
5. All L2 pools/integrations now compute yield using `rate_old` until the next manual `updateRate()` call.
6. The yield that should have accrued during the window `[T_M1_delivery, next_updateRate]` at `rate_new` is permanently lost — it cannot be retroactively corrected because yield accrual already occurred at the wrong rate.

### Likelihood Explanation

LayerZero v1 does not guarantee in-order delivery. Out-of-order delivery is a documented, known behavior of the protocol. It can occur due to differing gas prices, relayer behavior, or network conditions. The `updateRate()` function is callable by anyone (no access control), so two rapid successive calls are realistic. The README confirms `RSETHRateReceiver` is deployed on 15+ chains, widening the attack surface. [7](#0-6) 

### Recommendation

Add a monotonicity guard in `lzReceive()`:

```solidity
uint256 _rate = abi.decode(_payload, (uint256));
require(_rate >= rate, "Stale rate: must be non-decreasing");
rate = _rate;
```

Alternatively, embed the L1 `block.timestamp` or the provider-side nonce in the payload and reject messages with a timestamp/nonce older than the last accepted one. The `_nonce` parameter already passed to `lzReceive` by the LZ endpoint can also be used directly:

```solidity
function lzReceive(uint16 _srcChainId, bytes memory _srcAddress, uint64 _nonce, bytes calldata _payload) external {
    // existing auth checks ...
    require(_nonce > lastNonce, "Out-of-order message");
    lastNonce = _nonce;
    rate = abi.decode(_payload, (uint256));
    ...
}
```

### Proof of Concept

```solidity
// Mock LZ endpoint delivers M2 (rate_new=1.05e18) then M1 (rate_old=1.00e18)
receiver.lzReceive(srcChainId, providerBytes, 2, abi.encode(1.05e18)); // nonce=2 first
receiver.lzReceive(srcChainId, providerBytes, 1, abi.encode(1.00e18)); // nonce=1 second
assertEq(receiver.rate(), 1.05e18); // FAILS: actual is 1.00e18
```

Both calls pass all existing guards (correct endpoint, correct chain, correct provider). The second call silently overwrites the correct rate with the stale one.
