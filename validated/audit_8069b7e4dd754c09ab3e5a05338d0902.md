### Title
Byzantine Participant Can Submit Mismatched `payload_hash` to Forge Foreign-Chain Verification Response - (File: crates/contract/src/lib.rs)

### Summary
`respond_verify_foreign_tx` verifies that `response.signature` is valid over `response.payload_hash`, but never verifies that `response.payload_hash` was actually derived from the `request` parameter. A single Byzantine attested participant (below signing threshold) can replay a threshold signature produced for one pending request as the response for a completely different pending request, delivering a forged `VerifyForeignTransactionResponse` to the victim caller.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs two checks before resolving a pending yield:

1. The caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. [1](#0-0) 

What it **does not** check is that `response.payload_hash` was computed from the `request` argument. The canonical hash is defined as:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload { request, values }))
``` [2](#0-1) 

Because `values` (the extracted chain data) are unknown to the contract, it cannot recompute the hash. The contract therefore has no way to bind `response.payload_hash` to the `request` it is resolving.

On the node side, both leader and follower independently query the foreign chain and compute the payload hash: [3](#0-2) [4](#0-3) 

The threshold signing protocol enforces that all participating nodes agree on the hash. However, once a valid `(hash_A, sig_A)` pair exists on-chain (submitted by the honest leader for request A), any single attested participant can call `respond_verify_foreign_tx(request_B, {hash_A, sig_A})` for a different pending request B. The contract will:

- Accept the call (caller is attested) ✓
- Accept the signature (`sig_A` is valid over `hash_A` under the root key) ✓
- Resolve request B's yield with `{hash_A, sig_A}` ✓ [5](#0-4) 

The victim caller of request B receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes the data of transaction A, not transaction B.

The SDK's `ForeignChainSignatureVerifier::verify_signature` does check `expected_payload_hash == response.payload_hash`: [6](#0-5) 

However, this check is performed by the **caller's NEAR contract**, not by the MPC contract itself. Any bridge or application contract that does not invoke the SDK verifier — or that only checks whether the `verify_foreign_transaction` promise succeeded — will accept the forged attestation.

### Impact Explanation

A Byzantine attested participant below the signing threshold can cause any pending `verify_foreign_transaction` request to be resolved with a `payload_hash` that corresponds to a different transaction. Application contracts that rely on the MPC contract's response to gate cross-chain asset releases (e.g., Omnibridge inbound flows) and that do not independently re-verify `payload_hash` will accept the forged attestation, enabling invalid bridge execution or double-spend conditions. The root public key used for foreign-tx signing carries no per-request tweak: [7](#0-6) 

This means every foreign-tx signature is issued under the same root key, making cross-request replay structurally possible.

### Likelihood Explanation

The attack requires:
1. A single Byzantine attested participant (below threshold) — within the stated scope.
2. Two or more concurrent pending `verify_foreign_transaction` requests — routine under any bridge load.
3. The Byzantine participant observes `(hash_A, sig_A)` from the NEAR blockchain after the honest leader submits it, then immediately calls `respond_verify_foreign_tx(request_B, {hash_A, sig_A})`.

No threshold collusion is needed. The attacker does not need to forge a signature; they only replay one that the honest network already produced. The window for the attack is any period when multiple requests are simultaneously pending.

### Recommendation

Bind the signed payload to the specific pending request at the contract level. Two concrete approaches:

1. **Include the full `ForeignTxSignPayload` in the response** (not just the hash) and have the contract verify that `payload.request == request`. This requires relaxing the promise-data size constraint or splitting the response
