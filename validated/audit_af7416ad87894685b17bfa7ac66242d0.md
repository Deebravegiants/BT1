### Title
`updateFutureVRFKeyHash` incorrectly deletes current pool's active VRF key from `psVRFKeyHashes` when future params share the same VRF as current params — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Pool.hs`)

### Summary
When a pool re-registers with its own current VRF key (a valid operation to change other parameters), then re-registers again with a new VRF key in the same epoch, `updateFutureVRFKeyHash` deletes the current pool's active VRF key from `psVRFKeyHashes`. This allows a second pool to register with that VRF key, violating the uniqueness invariant introduced by `hardforkConwayDisallowDuplicatedVRFKeys`.

### Finding Description
In `poolTransition` (`Pool.hs` lines 283–294), the `updateFutureVRFKeyHash` function handles the case where a pool re-registers and `psFutureStakePoolParams` already has an entry from an earlier same-epoch re-registration. It deletes the previous future VRF key and inserts the new one. The missing guard: it does not check whether `futureStakePoolParams ^. sppVrfL` equals `stakePoolState ^. spsVrfL` (the current active VRF key). When they are equal — which happens when the first re-registration used the same VRF as the current params — the `Map.delete` removes the active pool's VRF key from `psVRFKeyHashes`, leaving `psStakePools[A].spsVrf` referencing a key no longer tracked.

### Impact Explanation
After the deletion, `Map.notMember vrf1 psVRFKeyHashes` returns `True`, so a second pool can register with vrf1 and pass the uniqueness check at line 266. Two pools now share the same VRF key. In Ouroboros Praos, the VRF key determines slot leadership; two pools with the same VRF key are elected for the same slots, enabling one pool to produce blocks in slots belonging to another, distorting rewards outside design parameters. This matches: **Medium — attacker-controlled certificates modify rewards outside design parameters.**

### Likelihood Explanation
The trigger requires three sequential, individually valid pool certificates from a single pool operator: register, re-register with same VRF (legitimate to change cost/margin/metadata), re-register with new VRF. All three are standard pool operations. No privileged access, governance majority, or key leakage is required.

### Recommendation
In the `Just futureStakePoolParams` branch of `updateFutureVRFKeyHash`, guard the `Map.delete` with a check that the future VRF key differs from the current active VRF key:

```haskell
Just futureStakePoolParams
  | futureStakePoolParams ^. sppVrfL /= sppVrf ->
      Map.insert sppVrf (knownNonZeroBounded @1)
        . ( if futureStakePoolParams ^. sppVrfL /= stakePoolState ^. spsVrfL
              then Map.delete (futureStakePoolParams ^. sppVrfL)
              else id
          )
  | otherwise -> id
```

### Proof of Concept
```
1. Register pool A with vrf1
   → psVRFKeyHashes = {vrf1:1}, psStakePools[A].spsVrf = vrf1

2. Re-register pool A with vrf1 (change cost only)
   → psFutureStakePoolParams[A].sppVrf = vrf1
   → psVRFKeyHashes = {vrf1:1}  (unchanged)

3. Re-register pool A with vrf2
   → psFutureStakePoolParams lookup = Just {sppVrf: vrf1}
   → vrf1 /= vrf2 → executes Map.insert vrf2 1 . Map.delete vrf1
   → psVRFKeyHashes = {vrf2:1}   ← BUG: vrf1 deleted despite psStakePools[A].spsVrf = vrf1

4. Register pool B with vrf1
   → Map.notMember vrf1 psVRFKeyHashes = True → succeeds
   → psStakePools[A].spsVrf = vrf1, psStakePools[B].spsVrf = vrf1  ← invariant violated

Assert: psVRFKeyHashes does not contain vrf1, yet two active pools reference it.
```
