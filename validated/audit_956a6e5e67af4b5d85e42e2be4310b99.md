Audit Report

## Title
Missing `forceResumeReceive` Enables Permanent Channel Block, Freezing Rate Updates — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver` implements only `ILayerZeroReceiver` and omits `ILayerZeroUserApplicationConfig`, meaning it never exposes `forceResumeReceive`. Under LayerZero v1's blocking delivery model, any revert inside `lzReceive` causes the endpoint to store the failed payload and block all subsequent messages on that channel. With no on-chain path to clear the stored payload, the `rate` value is permanently frozen, preventing yield accrual from propagating to destination chains.

## Finding Description

`CrossChainRateReceiver` is declared as:

```solidity
abstract contract CrossChainRateReceiver is ILayerZeroReceiver, Ownable {
``` [1](#0-0) 

It does not implement `ILayerZeroUserApplicationConfig`. A repository-wide search confirms `forceResumeReceive` appears only in the interface definition and is never implemented anywhere in the contract hierarchy. [2](#0-1) 

The `lzReceive` function contains three `require` guards that can revert:

```solidity
require(msg.sender == layerZeroEndpoint, "Sender should be lz endpoint");
require(_srcChainId == srcChainId, "Src chainId must be correct");
require(srcAddress == rateProvider, "Src address must be provider");
``` [3](#0-2) 

Additionally, `abi.decode(_payload, (uint256))` at line 93 reverts on a malformed payload, and an out-of-gas condition during delivery also causes a revert. [4](#0-3) 

Under LZ v1's blocking model, any such revert causes the endpoint to store the payload and block all future deliveries on that channel. The only recovery mechanism is for the UA to call `ILayerZeroEndpoint.forceResumeReceive(srcChainId, srcAddress)`. Neither `RSETHRateReceiver` nor `AGETHRateReceiver` adds this function. [5](#0-4) [6](#0-5) 

## Impact Explanation

The `rate` stored in `CrossChainRateReceiver` is consumed by pool contracts to compute swap amounts:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [7](#0-6) 

A permanently blocked channel means `rate` and `lastUpdated` are never refreshed. All staking yield accruing on mainnet (reflected in an increasing rsETH/ETH rate) is never propagated to destination chains. This matches **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation

Two realistic triggers exist:

1. **Out-of-gas**: If the relayer forwards insufficient gas (gas estimation error or deliberate griefing by a permissionless LZ v1 relayer), `lzReceive` reverts with OOG. This is a well-documented LZ v1 production risk requiring no privileged access.
2. **Misconfiguration during upgrade**: If `srcChainId` or `rateProvider` is updated on one side but not the other during a provider migration, the `require` checks at lines 90–91 revert on the next delivery, blocking the channel. The owner functions `updateSrcChainId` and `updateRateProvider` make this a realistic operational scenario. [8](#0-7) 

## Recommendation

Implement `ILayerZeroUserApplicationConfig` in `CrossChainRateReceiver` and add a `forceResumeReceive` function restricted to `onlyOwner` that proxies to the LZ endpoint:

```solidity
function forceResumeReceive(
    uint16 _srcChainId,
    bytes calldata _srcAddress
) external override onlyOwner {
    ILayerZeroEndpoint(layerZeroEndpoint).forceResumeReceive(
        _srcChainId,
        _srcAddress
    );
}
```

This matches the pattern used by `NonblockingLzApp` in the official LZ SDK and the recommendation in LZ v1 documentation.
