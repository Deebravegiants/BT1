### Title
Fee Threshold Check Is Log-Only: Harvester Can Always Redirect Farmer Reward — (`File: chia/farmer/farmer_api.py`)

---

### Summary

CHIP-22 introduces a fee convention allowing a third-party harvester to redirect the farmer's block reward (XCH) to itself as a fee, gated by a proof-quality threshold (`fee_info.applied_fee_threshold`). The farmer's implementation accepts this threshold parameter and evaluates it, but the evaluation result is **only logged** — it never blocks the override from being applied. A malicious harvester can unconditionally redirect the farmer's XCH block reward to any address it chooses, regardless of whether its proof quality meets the declared threshold.

---

### Finding Description

The CHIP-22 fee convention works as follows:

1. A harvester sends `NewProofOfSpace` with `farmer_reward_address_override` (its own address) and `fee_info.applied_fee_threshold` (the quality bar it claims to have applied).
2. The farmer is supposed to verify that the proof quality actually meets the threshold before honoring the override.
3. If the threshold is met, the farmer reward goes to the harvester; otherwise it should stay with the farmer.

**Step 1 — The check is log-only.**

In `new_proof_of_space()`, when `farmer_reward_address_override` is set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [1](#0-0) 

Inside that function, the fee quality is computed and compared against the threshold, but the outcome is **only logged** — the function has no return value and raises no exception: [2](#0-1) 

The docstring even confirms this is intentional: *"This will be tested against the fee threshold reported by the harvester (if any), and logged."* [3](#0-2) 

After the log-only call, execution continues unconditionally — `RequestSignatures` is sent to the harvester regardless of whether the threshold was met.

**Step 2 — The override is applied unconditionally in `_process_respond_signatures()`.**

When the harvester returns `RespondSignatures` (also carrying `farmer_reward_address_override`), the farmer applies it with no fee-threshold check whatsoever: [4](#0-3) 

The overridden address is then embedded directly into `DeclareProofOfSpace.farmer_reward_address`, which the full node uses to assign the XCH block reward: [5](#0-4) 

There is no cross-check between the threshold evaluated at `NewProofOfSpace` time and the address applied at `RespondSignatures` time.

---

### Impact Explanation

A malicious third-party harvester can redirect the farmer's entire XCH block reward to an address it controls on every winning block, without satisfying the declared fee quality threshold. The farmer's software accepts the `fee_info.applied_fee_threshold` parameter, evaluates it, and then ignores the result — making the threshold control void. This is an unauthorized payout redirection of XCH, matching the **High** impact category: *"Bypass of … authorization that enables … payout redirection."*

---

### Likelihood Explanation

Any third-party harvester connected to a farmer can trigger this. No special privileges, leaked keys, or cryptographic breaks are required. The harvester simply sets `farmer_reward_address_override` to its own address in both `NewProofOfSpace` and `RespondSignatures`. The farmer will log a warning but proceed to create a block that pays the harvester. Farmers using third-party harvesters (a common configuration) are exposed on every block they win.

---

### Recommendation

`notify_farmer_reward_taken_by_harvester_as_fee()` must return a boolean (or raise) and the caller must act on it. If the fee threshold is not met, the farmer should either:

1. **Reject the override** — drop the `farmer_reward_address_override` and use `self.farmer.farmer_target` instead, or
2. **Abort the block** — return `None` from `new_proof_of_space()` to refuse to sign for this harvester.

Additionally, `_process_respond_signatures()` should verify that the `farmer_reward_address_override` in `RespondSignatures` was previously validated (e.g., by checking a stored flag set only when the threshold passed).

---

### Proof of Concept

1. Malicious harvester connects to a farmer node.
2. On receiving a `NewSignagePointHarvester`, the harvester finds a valid proof of space.
3. Harvester sends `NewProofOfSpace` with:
   - `farmer_reward_address_override = <attacker_address>`
   - `fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=0)` — threshold of 0 means `fee_quality <= 0` is almost certainly false, so the check logs a warning.
4. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()` → logs `"Invalid fee threshold"` warning → **returns None, execution continues**.
5. Farmer sends `RequestSignatures` to the harvester.
6. Harvester responds with `RespondSignatures` carrying `farmer_reward_address_override = <attacker_address>`.
7. `_process_respond_signatures()` sets `farmer_reward_address = <attacker_address>` with no threshold check.
8. `DeclareProofOfSpace` is sent to the full node with `farmer_reward_puzzle_hash = <attacker_address>`.
9. The block is farmed; the XCH block reward is paid to the attacker's address instead of the farmer's.
