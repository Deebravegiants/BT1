### Title
Unbounded Iteration Over DRep Delegators During `ConwayUnRegDRep` Causes Disproportionate Block Validation Work â€” (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

When a DRep unregisters via a `ConwayUnRegDRep` certificate, the ledger iterates over the entire `drepDelegs` set stored in `DRepState` to clear each delegator's DRep delegation field. This set is unbounded â€” any staking credential holder can grow it by delegating to the DRep â€” and there is no protocol parameter capping the number of delegators per DRep. A DRep with a very large delegator set causes the block containing the unregistration certificate to require O(n log n) ledger work, where n is the number of delegators, exceeding the intended per-certificate validation cost.

---

### Finding Description

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`, the `ConwayUnRegDRep` branch of `govCertTransition` executes the following cleanup:

```haskell
let
  certState' =
    certState & certVStateL . vsDRepsL %~ Map.delete cred
  clearDRepDelegations delegs accountsMap =
    foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
pure $
  case mDRepState of
    Nothing -> certState'
    Just dRepState ->
      certState'
        & certDStateL . accountsL . accountsMapL
          %~ clearDRepDelegations (drepDelegs dRepState)
``` [1](#0-0) 

`clearDRepDelegations` performs a `foldr` over `drepDelegs dRepState`, calling `Map.adjust` once per delegator credential. The `drepDelegs` field is defined as an unbounded `Set (Credential Staking)` inside `DRepState`:

```haskell
data DRepState = DRepState
  { drepExpiry :: !EpochNo
  , drepAnchor :: !(StrictMaybe Anchor)
  , drepDeposit :: !(CompactForm Coin)
  , drepDelegs :: !(Set (Credential Staking))
  }
``` [2](#0-1) 

This set is grown by any staking credential holder who submits a `DelegVote` or `DelegStakeVote` certificate targeting the DRep. The insertion path is in `processDelegationInternal`:

```haskell
certVStateL . vsDRepsL
  %~ Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
``` [3](#0-2) 

There is no protocol parameter in `PParams` that limits the number of delegators per DRep. The transaction size limit (`maxTxSize`) bounds the serialized transaction, but the ledger work performed to process the certificate scales with the size of `drepDelegs`, which is ledger state accumulated over many prior transactions.

The analogous pattern exists for `DRepState.drepDelegs` cleanup in `unDelegReDelegDRep` in `eras/conway/impl/src/Cardano/Ledger/Conway/State/VState.hs` (single-credential path, O(log n)), but the bulk-clear path in `GovCert.hs` is the unbounded one. [4](#0-3)

### Impact Explanation

**Medium â€” Attacker-controlled certificates exceed intended validation limits.**

The Cardano ledger is designed so that each certificate in a transaction requires O(1) or O(log n) ledger work (where n is the size of the relevant state map). The `ConwayUnRegDRep` certificate violates this assumption: its processing cost is O(|drepDelegs| Ă— log(|accountsMap|)). With a DRep that has accumulated hundreds of thousands or millions of delegators, a single `UnRegDRepTxCert` certificate causes the block containing it to require disproportionate validation work. This can cause honest nodes to spend excessive CPU time validating a single block, degrading throughput and potentially causing slot leaders to miss their slot if block validation overlaps with their leadership window.

---

### Likelihood Explanation

Each staking credential registration requires a `keyDeposit` (currently 2 ADA on mainnet). Growing `drepDelegs` to N entries requires N Ă— 2 ADA in deposits, which is a real economic barrier for a pure attacker. However, the scenario does not require a malicious actor: a genuinely popular DRep that accumulates hundreds of thousands of legitimate delegators (plausible at mainnet scale, where millions of staking credentials exist) will trigger the same O(n) cleanup on unregistration. The attacker-controlled path (register many credentials, delegate all to a target DRep, then unregister the DRep) is reachable by any unprivileged transaction sender with sufficient ADA.

---

### Recommendation

1. **Add a `ppMaxDRepDelegators` protocol parameter** capping the number of delegators per DRep, enforced at delegation time in `processDelegationInternal`. This mirrors the existing `maxCollateralInputs` pattern.
2. **Alternatively, remove `drepDelegs` from `DRepState` entirely** and replace it with a delegator count (`Word64`). The full set is only needed for the bulk-clear on unregistration; the count suffices for all other uses (`spssNumDelegators` pattern in `StakePoolSnapShot` shows this is viable). Cleanup of individual account DRep delegation fields can then be deferred to a lazy scan or handled at the account level when the account is next touched.
3. As a short-term mitigation, enforce that `ConwayUnRegDRep` is only accepted when `drepDelegs` is empty (requiring delegators to redelegate before the DRep can unregister), analogous to the `StakeKeyNonZeroAccountBalanceDELEG` check for stake key deregistration.

### Proof of Concept

1. Register a DRep credential `D` via `RegDRepTxCert`.
2. Register N staking credentials `c_1 â€¦ c_N` (each costs `keyDeposit` = 2 ADA).
3. For each `c_i`, submit a `DelegTxCert c_i (DelegVote (DRepCredential D))`. Each accepted transaction inserts `c_i` into `drepDelegs` of `D`'s `DRepState` via `Set.insert`.
4. Submit `UnRegDRepTxCert D deposit`. The ledger executes `clearDRepDelegations (drepDelegs dRepState)`, which calls `Map.adjust` N times against the accounts map. For N = 500,000 (a realistic mainnet delegator count), this is ~500,000 Ă— O(log(accountsMap)) operations within a single certificate's processing, all within one block.

The entry path is entirely unprivileged: steps 1â€“4 require only ADA for deposits and fees, no governance majority or privileged key. [5](#0-4) [2](#0-1) [6](#0-5)
