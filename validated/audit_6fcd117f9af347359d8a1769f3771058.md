### Title
Stale Cross-Chain rsETH/ETH Rate Used for wrsETH Minting Allows Over-Issuance and Pool Insolvency - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

### Summary
The `CrossChainRateReceiver.getRate()` function returns the last stored `rate` with no staleness check. All L2 RSETHPool variants (`RSETHPool`, `RSETHPoolV2`, `RSETHPoolV2NBA`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) call `getRate()` on this receiver to price every `deposit()`. Because the rate is only updated when someone explicitly calls `updateRate()` on the L1 provider (paying LayerZero fees with no on-chain incentive), the L2 rate can lag arbitrarily behind the true rsETH/ETH rate. Since rsETH only appreciates over time, a stale (lower) rate causes the pool to mint more wrsETH per ETH deposited than the true rate warrants, draining the pool's ETH reserves.

### Finding Description
The cross-chain rate pipeline is:

1. **L1 provider** (`RSETHRateProvider` / `RSETHMultiChainRateProvider`) reads `LRTOracle.rsETHPrice()` and sends it via LayerZero when `updateRate()` is called.
2. **L2 receiver** (`CrossChainRateReceiver`) stores the received value in `rate` and records `lastUpdated = block.timestamp`.
3. **L2 pool** calls `IOracle(rsETHOracle).getRate()` → `CrossChainRateReceiver.getRate()` → returns `rate` unconditionally.

`CrossChainRateReceiver.getRate()` is:

```solidity
function getRate() external view returns (uint256) {
    return rate;   // no check on lastUpdated
}
```

There is no guard of the form `require(block.timestamp - lastUpdated <= MAX_STALENESS)`. The `lastUpdated` field is stored but never consulted at the point of use.

`updateRate()` on the provider is permissionless but requires the caller to supply LayerZero messaging fees out-of-pocket with no protocol-side incentive. If no one calls it, the L2 rate freezes at its last value indefinitely.

Every L2 pool `deposit()` path computes the wrsETH amount as:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

A stale (lower) `rsETHToETHrate` produces a larger `rsETHAmount`. The pool mints or transfers this inflated amount of wrsETH to the depositor while receiving only the deposited ETH/token at face value.

### Impact Explanation
rsETH/ETH rate is monotonically non-decreasing (staking rewards accumulate). Any period without a `updateRate()` call leaves the L2 rate below the true rate. A depositor who deposits `X` ETH at a stale rate receives `wrsETH` worth more than `X` ETH at the true rate. When that wrsETH is bridged to L1 and redeemed, the depositor extracts more ETH than they contributed. The pool's ETH reserves are insufficient to back all outstanding wrsETH, resulting in **direct theft of ETH from the pool** and eventual **pool insolvency**. The magnitude scales with the staleness duration and the total deposit volume during that window.

**Impact: Critical — direct theft of user funds / protocol insolvency.**

### Likelihood Explanation
`updateRate()` requires the caller to pay LayerZero cross-chain messaging fees with no reimbursement or on-chain reward. During periods of high gas prices or low protocol activity, no one has an economic incentive to call it. The rate can realistically go stale for hours or days. An attacker can passively monitor `CrossChainRateReceiver.lastUpdated` and `LRTOracle.rsETHPrice()` on-chain, wait for a meaningful divergence, then deposit a large amount in a single transaction. No special role or privileged access is required.

### Recommendation
1. Add a `MAX_STALENESS` constant (e.g., 24 hours) and revert in `getRate()` if `block.timestamp - lastUpdated > MAX_STALENESS`.
2. Alternatively, have the pool's `deposit()` function revert if the oracle rate is stale, using the `lastUpdated` field already stored in `CrossChainRateReceiver`.
3. Introduce an on-chain incentive (e.g., a small ETH bounty from protocol fees) for calling `updateRate()` to ensure liveness.

### Proof of Concept

**Setup:**
- L2 pool: `RSETHPoolV2` (or any variant) with `rsETHOracle` pointing to a `CrossChainRateReceiver`.
- True rsETH/ETH rate on L1: `1.05e18` (has grown due to staking rewards).
- Stale rate stored in `CrossChainRateReceiver.rate`: `1.00e18` (last updated 48 hours ago, no one paid to call `updateRate()`).

**Attack:**
1. Attacker observes `CrossChainRateReceiver.lastUpdated` is 48 hours old and `CrossChainRateReceiver.rate == 1.00e18` while L1 `LRTOracle.rsETHPrice() == 1.05e18`.
2. Attacker calls `RSETHPoolV2.deposit{value: 100 ether}("ref")`.
3. Pool computes: `rsETHAmount = 100e18 * 1e18 / 1.00e18 = 100e18` wrsETH.
4. True fair amount at current rate: `100e18 * 1e18 / 1.05e18 ≈ 95.24e18` wrsETH.
5. Attacker receives `~4.76e18` excess wrsETH (≈ 5 ETH of value at true rate).
6. Attacker bridges wrsETH to L1 and redeems for `~105 ETH`, having deposited only `100 ETH`.
7. Pool is left with `100 ETH` but has issued wrsETH worth `105 ETH` — insolvent by `5 ETH` per `100 ETH` deposited.

The attack is repeatable until the pool's ETH reserves are exhausted or the rate is updated.
