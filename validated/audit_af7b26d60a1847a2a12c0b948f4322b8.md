### Title
Missing Duplicate-`rid` Guard in `combine_additive` Allows Chosen-Plaintext Substitution — (`src/cbmpc/crypto/tdh2.cpp`)

### Summary

`combine_additive` accepts a `partial_decryptions_t` vector and uses each entry's embedded `rid` field to select which public share `Qi[rid-1]` to validate against. There is no check that each `rid` value appears exactly once. An attacker who controls the partial-decryption vector and the ciphertext can duplicate one party's partial decryption, craft a ciphertext that encrypts a chosen plaintext under the resulting wrong `V`, and cause `combine_additive` to output that chosen plaintext as a valid decryption.

### Finding Description

In `combine_additive` (lines 141–148), the loop iterates over all `n` partial decryptions, reads `rid`, bounds-checks it, validates the ZK proof against `Qi[rid-1]`, and accumulates `Xi` into `V`: [1](#0-0) 

There is no set or bitmap tracking which `rid` values have already been seen. If two entries carry `rid=1`, both pass `check_partial_decryption_helper(Qi[0], ...)` independently (the ZK proof is valid for each), and `V` accumulates `Xi_0 + Xi_0` instead of `Xi_0 + Xi_1`.

The `combine_ac` path is immune because it uses a `party_map_t` (keyed by party name), and the API wrapper additionally enforces uniqueness explicitly: [2](#0-1) 

No equivalent guard exists in the `combine_additive` path at either the crypto layer or the API wrapper layer: [3](#0-2) 

### Impact Explanation

**Scenario — chosen-plaintext substitution (full impact):**

In an n-of-n additive scheme the public shares `Qi[i] = xi·G` are public. An attacker who controls both the ciphertext submitted for decryption and the partial-decryption vector passed to `combine_additive` proceeds as follows:

1. Choose random `r`; compute `R1 = r·G`, `R2 = r·Gamma`.
2. Compute `V_wrong = r·(2·Q0 + Q2 + … + Q_{n-1})` using only public shares.
3. Encrypt a chosen plaintext `m*` under `V_wrong` using AES-GCM; build a valid ZK proof (the attacker knows `r`). The resulting ciphertext passes `ciphertext_t::verify` because that check only validates the Schnorr proof over `(R1, R2)` and does not bind `c` to the correct decryption key.
4. Submit the crafted ciphertext to honest parties; each calls `private_share_t::decrypt`, which verifies the ciphertext and produces a valid `Xi_i = xi·R1`.
5. Build the partial-decryption vector with `partial_decryptions[0].rid = 1` (honest) and `partial_decryptions[1].rid = 1` (duplicate of slot 0, replacing party 1's share).
6. Call `combine_additive`. The function computes `V = Xi_0 + Xi_0 + Xi_2 + … = V_wrong`. AES-GCM decryption succeeds and outputs `m*`.

The caller receives `m*` with `SUCCESS`, believing it is the legitimate decryption of the ciphertext. [4](#0-3) 

**Scenario — decryption failure (DoS) for honest ciphertexts:**

If the ciphertext is honestly generated (not attacker-controlled), duplicating a partial decryption produces a wrong `V` and AES-GCM authentication fails. This is a denial-of-service and is out of scope per the rejection criteria, but it confirms the guard is absent.

### Likelihood Explanation

The public API `coinbase::api::tdh2::combine_additive` (in `src/cbmpc/api/tdh2.cpp`) deserializes the partial-decryption blobs from caller-supplied byte buffers and passes them directly to the crypto-layer function with no `rid`-uniqueness check. Any caller who assembles the partial-decryption vector — including a Byzantine participant in the n-of-n scheme who collects honest parties' partial decryptions before calling combine — can mount this attack. The attacker needs no secret material beyond what is already public (`Qi` values).

### Recommendation

Add a duplicate-`rid` guard in `combine_additive` before the accumulation loop, mirroring the protection already present in `combine_ac`:

```cpp
std::vector<bool> seen_rid(n, false);
for (int i = 0; i < n; i++) {
    const int rid = partial_decryptions[i].rid;
    if (rid < 1 || rid > n) return coinbase::error(E_CRYPTO);
    if (seen_rid[rid - 1]) return coinbase::error(E_CRYPTO, "duplicate rid");
    seen_rid[rid - 1] = true;
    // ... existing check_partial_decryption_helper and V accumulation
}
```

Additionally, add a corresponding check in the API wrapper `coinbase::api::tdh2::combine_additive` after deserializing the `partials` vector, consistent with the duplicate-name guard already present in `combine_ac`.

### Proof of Concept

```cpp
// 3-of-3 additive setup
int n = 3;
public_key_t enc_key;
pub_shares_t pub_shares;
std::vector<private_share_t> dec_shares;
generate_additive_shares(n, enc_key, pub_shares, dec_shares, curve_p256);

buf_t label = buf_t("test-label");

// Attacker computes V_wrong = r*(2*Q0 + Q2) using only public shares
bn_t r = curve_p256.get_random_value();
ecc_point_t V_wrong = r * (2 * pub_shares[0] + pub_shares[2]);

// Attacker encrypts chosen plaintext under V_wrong
buf_t chosen_plain = buf_t("attacker-chosen-plaintext!!");
buf_t key_wrong = ro::hash_string(V_wrong).bitlen(256);
// ... build ciphertext with valid ZK proof (attacker knows r)
ciphertext_t crafted_ct = enc_key.encrypt_with_r(chosen_plain, label, r, ...);
// crafted_ct passes ciphertext_t::verify because ZK proof is valid

// Honest parties produce partial decryptions of crafted_ct
partial_decryptions_t partials(n);
for (int i = 0; i < n; i++)
    dec_shares[i].decrypt(crafted_ct, label, partials[i]);
// partials[i].rid == i+1, partials[i].Xi == dec_shares[i].x * R1

// Attacker duplicates party 0's partial decryption into slot 1
partials[1] = partials[0];  // rid=1 appears twice; rid=2 never appears

// combine_additive: no duplicate-rid check, V = Xi_0 + Xi_0 + Xi_2 = V_wrong
buf_t result;
error_t rv = combine_additive(enc_key, pub_shares, label, partials, crafted_ct, result);
// rv == SUCCESS, result == chosen_plain  ← wrong plaintext accepted as valid decryption
assert(result == chosen_plain);
assert(result != honest_plain);
```
