Audit Report

## Title
Pool Admin Bypasses Fee Caps via Uncapped `setPoolBinAdditionalFees()`, Enabling Front-Running of Swaps — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`setPoolAdminFees()` enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps before updating global fees, but `setPoolBinAdditionalFees()` applies no equivalent cap, allowing the pool admin to set per-bin `addFeeBuyE6`/`addFeeSellE6` to any `uint16` value (up to 65535, ≈6.55% in E6 units) instantly and without delay. Because the per-bin fee is read from storage at the moment each bin is processed during `swap()`, a pool admin who observes a pending swap in the mempool can front-run it by spiking the bin fee, forcing the swapper to pay materially more input tokens than expected. The `priceLimitX64` parameter constrains only the marginal price, not the fee, and provides no protection.

## Finding Description

**Cap asymmetry — global admin fees are bounded, per-bin fees are not.**

`setPoolAdminFees()` enforces caps before writing: [1](#0-0) 

`setPoolBinAdditionalFees()` performs no cap check and directly delegates to the pool: [2](#0-1) 

`setBinAdditionalFees()` on the pool only validates the bin index range, then writes the caller-supplied values verbatim: [3](#0-2) 

**Per-bin fee is read from live storage at swap time.**

During `swap()`, the combined fee passed to `SwapMath.buyToken0InBinSpecifiedOut` is `baseFeeX64 + mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)`, where `binState` is the current on-chain state: [4](#0-3) 

The fee is then applied to the gross input amount: [5](#0-4) 

A fee increase committed in a front-running transaction is therefore fully reflected in the victim's swap.

**`priceLimitX64` does not protect against fee manipulation.**

`priceLimitX64` is compared against the interpolated bin-position price to cap output quantity; it has no relationship to the fee multiplier: [6](#0-5) 

A swapper can set a tight price limit and still be charged an arbitrarily higher fee on the same output quantity.

**Effective cap bypass.** The contest rules classify the pool admin as "semi-trusted, bounded" and state findings are valid when "the Pool admin can bypass the caps." Because `maxAdminSpreadFeeE6` caps the global spread fee but no analogous cap exists for per-bin additional fees, the pool admin can impose a total effective fee far exceeding `maxAdminSpreadFeeE6` by routing through `setPoolBinAdditionalFees()`. The `BinState` struct stores `addFeeBuyE6`/`addFeeSellE6` as `uint16`, so the maximum injectable additional fee is 65535 (≈6.55%): [7](#0-6) 

## Impact Explanation

For an exact-output swap (token1 → token0), the swapper's required token1 input is proportional to `baseFeeX64 + addFeeBuyE6/1e6`. If the pool admin raises `addFeeBuyE6` from 0 to 65535 immediately before the swap executes, the swapper pays up to ≈6.55% more token1 than they observed when constructing the transaction. The excess fee accrues to LPs in the bin; if the pool admin is also an LP, they capture it directly. This is a direct, quantifiable loss of user principal with no on-chain recourse, meeting the "admin-boundary break: pool admin exceeds caps" and "direct loss of user principal" criteria at High/Medium severity.

## Likelihood Explanation

The attack requires: (1) a pool with a mutable pool admin (not an immutable oracle pool), (2) the admin monitoring the mempool — trivial on any public EVM chain (Ethereum, Base, HyperEVM per contest scope), and (3) a single `setPoolBinAdditionalFees` call to spike the fee, followed by a reset after the victim's transaction confirms. No timelock, no cap, and no on-chain slippage guard on the fee dimension stand in the way. The attack is repeatable and low-cost.

## Recommendation

1. **Add a cap on per-bin fees** inside `setPoolBinAdditionalFees()` (e.g., bounded by `maxAdminSpreadFeeE6` or a dedicated `maxBinAdditionalFeeE6` constant), mirroring the cap already enforced for global admin fees in `setPoolAdminFees()`.
2. **Add a timelock** for per-bin fee increases (a short delay is sufficient to eliminate mempool front-running), consistent with the timelock already applied to oracle rotation via `proposePoolPriceProvider`.
3. Alternatively, expose a `maxTotalFeeE6` parameter on `swap()` so callers can bound the combined fee they accept, providing an on-chain slippage guard on the fee dimension.

## Proof of Concept

```
Setup:
  - Pool deployed with addFeeBuyE6 = 0 on bin 0 (active bin).
  - Alice quotes a swap: 1000 token1 in → 990 token0 out (fee ≈ 1%).

Attack:
  1. Alice submits swap(recipient, false, -990e18, 0, ...) to the mempool.
  2. Bob (pool admin) sees Alice's tx.
  3. Bob front-runs with:
       factory.setPoolBinAdditionalFees(pool, 0, 65535, 0)
     This sets addFeeBuyE6 = 65535 (≈ 6.55% extra fee on bin 0).
  4. Alice's swap executes. The fee applied is now baseFeeX64 + 6.55%.
     Alice pays ≈ 1066 token1 instead of 1000 — a ~6.6% overcharge.
  5. Bob resets: factory.setPoolBinAdditionalFees(pool, 0, 0, 0).

Result:
  Alice loses ~66 token1 (6.6% of principal) with no on-chain protection.
  The excess accrues to LPs in bin 0; if Bob is an LP, he captures it directly.

Foundry test plan:
  - Deploy pool with addFeeBuyE6 = 0.
  - Record Alice's expected amountIn via simulateSwapAndRevert.
  - vm.prank(admin); factory.setPoolBinAdditionalFees(pool, 0, 65535, 0);
  - Execute Alice's swap; assert actual amountIn > expected amountIn by ~6.55%.
```
