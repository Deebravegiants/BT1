### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Replay of Foreign-Chain Verification — (File: `crates/contract/src/lib.rs`)

### Summary

`MpcContract::respond_verify_foreign_tx` accepts a caller-supplied `response.payload_hash` alongside a `request` struct but never validates that `payload_hash` is actually the canonical hash of `(request.request, extracted_values)`. Any single attested MPC participant can replay a legitimately-produced `(payload_hash, signature)` pair from one pending request to resolve a completely different pending request, delivering a valid threshold signature over the wrong foreign-chain data.

### Finding Description

`respond_verify_foreign_tx` (lines 692–754 of `crates/contract/src/lib.rs`) performs two checks:

1. The caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`, line 705).
2. The `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key (lines 718–743). [1](#0-0) 

What it does **not** check is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: <actual extracted values> }))`. The `ForeignTxSignPayloadV1` structure that defines the canonical hash is: [2](#0-1) 

It contains only `(ForeignChainRpcRequest, Vec<ExtractedValue>)` — it does **not** include `domain_id`, `payload_version`, or any binding to the outer `VerifyForeignTransactionRequest`. The contract never recomputes this hash from the `request` it holds in state; it blindly trusts whatever 32-byte value the caller places in `response.payload_hash`.

The pending-request map is keyed on the full `VerifyForeignTransactionRequest` (including `domain_id`, `payload_version`, and `ForeignChainRpcRequest`): [3](#0-2) 

So `resolve_yields_for` correctly finds and drains the yield for `request_B`, but it delivers `response_A`'s `payload_hash` and signature to the waiting caller.

### Impact Explanation

A single malicious attested MPC node (strictly below the signing threshold) can:

1. Observe or participate in a legitimate signing session for `request_A`, obtaining `(payload_hash_A, sig_A)` — either as the signing leader or by reading the on-chain `respond_verify_foreign_tx` call.
2. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })` for any other pending request `request_B` that shares the same domain.
3. The contract accepts the call: `sig_A` is a valid threshold signature over `payload_hash_A` under the domain key, so the signature check passes. The yield for `request_B` is resolved and the pending entry is permanently consumed.
4. The original submitter of `request_B` receives `{ payload_hash_A, sig_A }` — a valid MPC-network signature, but over the hash of a **different** foreign transaction's data.

If the receiving contract does not independently reconstruct and compare the expected `payload_hash` (e.g., using `ForeignChainSignatureVerifier` from the SDK), it will accept the forged attestation and act on it — enabling invalid bridge execution or double-spend conditions. Even when the receiver does verify the hash, the pending request is permanently consumed: the user cannot obtain a correct response, constituting an irreversible denial of the verification service.

The impact matches: **High — forged foreign-chain verification that causes invalid bridge execution**.

### Likelihood Explanation

- Requires only one malicious attested MPC participant (below threshold).
- No cryptographic forgery is needed; the attacker reuses a legitimately-produced threshold signature.
- The on-chain `respond_verify_foreign_tx` call is public, so `(payload_hash, signature)` pairs are observable by all participants.
- Any participant who is the signing leader for any request already holds the pair before it is published on-chain.
- The window for the attack is the lifetime of any concurrent pending `verify_foreign_tx` request.

### Recommendation

The contract must bind `response.payload_hash` to the `request` it is resolving. Because the contract cannot independently fetch foreign-chain data, the node must also supply the `extracted_values`, and the contract must recompute and compare the hash on-chain:

```rust
// In respond_verify_foreign_tx, after signature verification:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // new field in response
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, include `domain_id` and `payload_version` inside `ForeignTxSignPayloadV1` so that the signed hash is fully bound to the outer request, and then enforce the recomputation above.

### Proof of Concept

**Setup**: Two pending requests exist in the contract:
- `request_A`: Bitcoin tx_id = `[0xAA; 32]`, domain 1
- `request_B`: Bitcoin tx_id = `[0xBB; 32]`, domain 1

**Step 1** — Attacker (attested participant) acts as signing leader for `request_A`. The MPC network produces:
```
payload_hash_A = SHA-256(borsh(ForeignTxSignPayload::V1 {
    request: Bitcoin { tx_id: [0xAA;32], ... },
    values: [BlockHash([0x11;32])],
}))
sig_A = threshold_sign(payload_hash_A, domain_1_key)
```

**Step 2** — Before calling `respond_verify_foreign_tx(request_A, ...)`, the attacker calls:
```
respond_verify_foreign_tx(
    request_B,                          // different pending request
    { payload_hash: payload_hash_A,     // hash of request_A's data
      signature: sig_A }                // valid threshold sig
)
```

**Step 3** — Contract execution (lines 715–752):
- `domain = request_B.domain_id` → same domain 1 → same root public key
- `verify_ecdsa_signature(sig_A, payload_hash_A, domain_1_pk)` → **passes**
- `resolve_yields_for(&request_B, serialize({payload_hash_A, sig_A}))` → **drains `request_B`'s yield**

**Step 4** — The submitter of `request_B` receives `{ payload_hash_A, sig_A }`. Their contract sees a valid MPC signature but `payload_hash_A` encodes Bitcoin tx `[0xAA;32]`, not their tx `[0xBB;32]`. If they skip the hash-binding check, they accept a forged attestation for the wrong transaction. [4](#0-3) [2](#0-1) [5](#0-4)
