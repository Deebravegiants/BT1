### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Confidential Key Derivation Output — (`File: crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, the contract performs a cryptographic output check (`ckd_output_check`) only when the request used the `AppPublicKeyPV` (publicly verifiable) variant. When the request used the legacy `AppPublicKey` (privately verifiable) variant, the response `(big_y, big_c)` is accepted and delivered to the user with **zero cryptographic verification**. A single attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse`, causing the user to receive a confidential key that the attacker fully controls rather than one derived from the MPC network's master key.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function dispatches on the request's `app_public_key` variant to decide whether to verify the response:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKey` arm is an empty no-op. Immediately after this match, the unverified response is serialized and delivered to every queued yield for that request:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

By contrast, `respond` (ECDSA/EdDSA) always verifies the signature against the derived public key before resolving yields: [3](#0-2) 

And `respond_verify_foreign_tx` similarly always verifies the signature: [4](#0-3) 

The `ckd_output_check` function that is skipped verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which proves the output was computed using the MPC network's master key: [5](#0-4) 

For `AppPublicKey` requests, no equivalent check exists. The `AppPublicKey` variant carries only `pk1 = a·G1` (the user's G1 point), which is insufficient to verify the response on-chain without the corresponding G2 point `pk2 = a·G2` — but the contract does not even attempt a partial check or reject the call.

**Attack path:**

1. A user submits `request_app_private_key` with `AppPublicKey(pk1)` and a valid 1 yoctoNEAR deposit. The request enters `pending_ckd_requests`.
2. An attested participant (one Byzantine node, below threshold) calls `respond_ckd(request, CKDResponse { big_y: attacker_big_y, big_c: attacker_big_c })` where `big_c = r' · pk1` for an attacker-chosen scalar `r'`.
3. The contract passes `assert_caller_is_attested_participant_and_protocol_active()`, finds the pending request, skips the empty `AppPublicKey` arm, and calls `resolve_yields_for`, delivering the forged `(big_y, big_c)` to the user.
4. The user receives a confidential key encrypted under `r'`, which the attacker knows. The attacker can now compute the same derived key the user believes is secret. [6](#0-5) 

---

### Impact Explanation

**Critical — Unauthorized confidential key derivation output without the required participant authorization.**

The CKD protocol's security guarantee is that the derived confidential key is bound to the MPC network's master key and is unknown to any individual participant. By forging `(big_y, big_c)` with a self-chosen scalar `r'`, the attacker delivers a key that:

- Is not derived from the MPC network's master key (breaks the derivation integrity invariant).
- Is fully known to the attacker (the attacker set `r'`, so `big_c = r' · pk1` decrypts to a value the attacker controls).

Any application secret the user derives from this forged CKD output — wallet keys, encryption keys, authentication credentials — is compromised. This constitutes **theft or permanent compromise of confidential key material** controlled by the MPC network's chain-signature contract.

---

### Likelihood Explanation

The `AppPublicKey` (privately verifiable) variant is the **legacy and default** CKD mode documented in the contract README and used in all non-PV tests and integrations: [7](#0-6) 

Any attested participant — a single node that has passed TEE attestation — can execute this attack. The threshold is not enforced at the contract level for CKD responses; `respond_ckd` requires only that the caller is an attested participant, not that a threshold of participants agreed on the output. A single Byzantine node below the signing threshold is sufficient. [8](#0-7) 

---

### Recommendation

Apply the same pattern used for `AppPublicKeyPV`: require a verifiable response format for `AppPublicKey` requests, or reject `AppPublicKey` CKD requests at the contract level and require all new requests to use `AppPublicKeyPV`.

The minimal fix is to require callers to always supply `AppPublicKeyPV` so the on-chain pairing check can be enforced. If backwards compatibility with `AppPublicKey` must be preserved, the contract should at minimum verify that `big_c` and `big_y` are valid G1 points in the prime-order subgroup (using `bls12381_p1_decompress` and a subgroup check), and document clearly that the privately verifiable variant provides no on-chain integrity guarantee — which is a weaker security model that should be explicitly acknowledged.

---

### Proof of Concept

1. Deploy the contract with a BLS12-381 CKD domain.
2. User Alice calls `request_app_private_key` with `AppPublicKey(pk1)` and 1 yoctoNEAR deposit. The request is stored in `pending_ckd_requests`.
3. Attacker (attested participant) calls:
   ```json
   respond_ckd(
     request = <Alice's CKDRequest>,
     response = { "big_y": "<attacker_chosen_G1>", "big_c": "<r_prime * pk1>" }
   )
   ```
4. The contract executes lines 675–682 of `lib.rs`, hits the empty `AppPublicKey(_) => {}` arm, and proceeds directly to `resolve_yields_for`, delivering the forged response to Alice.
5. Alice receives `(big_y, big_c)` and decrypts `big_c` with her private key `a` to obtain a "derived secret" that equals `r'` — a value the attacker chose and knows.
6. The attacker has full knowledge of Alice's supposedly confidential derived key. [9](#0-8)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

**File:** crates/contract/README.md (L118-121)
```markdown
- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.
```
