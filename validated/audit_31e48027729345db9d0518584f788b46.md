### Title
`assetsCommitted` Is Only Decremented via Operator-Gated `unlockQueue`, Causing Stale Accounting That Temporarily Freezes User rsETH - (File: contracts/LRTWithdrawalManager.sol)

---

### Summary

`assetsCommitted[asset]` tracks the total asset amount reserved for pending withdrawal requests. It is incremented on every `initiateWithdrawal` call but is **only decremented** inside `_unlockWithdrawalRequests`, which is exclusively reachable through `unlockQueue` — a function restricted to `onlyAssetTransferOrOperatorRole`. When the withdrawal delay naturally passes, `assetsCommitted` is never automatically reduced. This mirrors the original `totalCoverTokens` bug: a collateral-tracking variable that should reflect expired/matured obligations is only updated when a privileged action occurs, not when the time-based condition is satisfied.

---

### Finding Description

The withdrawal lifecycle in `LRTWithdrawalManager` has three stages:

1. **`initiateWithdrawal`** (any user): rsETH is transferred from the user to the contract, and `assetsCommitted[asset]` is incremented by `expectedAssetAmount`.
2. **`unlockQueue`** (operator only, `onlyAssetTransferOrOperatorRole`): iterates pending requests, decrements `assetsCommitted[asset]`, burns rsETH, redeems assets from the vault, and advances `nextLockedNonce[asset]`.
3. **`completeWithdrawal`** (any user): transfers the underlying asset to the user, but **only if** `usersFirstWithdrawalRequestNonce < nextLockedNonce[asset]` — i.e., only after the operator has already called `unlockQueue`.

The critical accounting variable `assetsCommitted[asset]` is decremented **only** in `_unlockWithdrawalRequests`:

```solidity
// LRTWithdrawalManager.sol:802
assetsCommitted[asset] -= request.expectedAssetAmount;
```

which is called exclusively from `unlockQueue`:

```solidity
// LRTWithdrawalManager.sol:268-281
function unlockQueue(...) external nonReentrant onlySupportedAsset(asset) whenNotPaused onlyAssetTransferOrOperatorRole ...
```

Meanwhile, `getAvailableAssetAmount` computes available capacity as:

```solidity
// LRTWithdrawalManager.sol:602
availableAssetAmount = totalAssets > assetsCommitted[asset] ? totalAssets - assetsCommitted[asset] : 0;
```

And `initiateWithdrawal` enforces:

```solidity
// LRTWithdrawalManager.sol:170
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
```

**Consequence 1 — New withdrawals blocked**: As long as the operator has not called `unlockQueue`, `assetsCommitted` remains inflated with already-matured (delay-passed) requests. New users calling `initiateWithdrawal` receive `ExceedAmountToWithdraw` even when the protocol holds sufficient total assets, because the stale `assetsCommitted` consumes the apparent available capacity.

**Consequence 2 — Existing rsETH locked**: Users who have already called `initiateWithdrawal` have their rsETH held in the contract. `completeWithdrawal` enforces:

```solidity
// LRTWithdrawalManager.sol:707
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```

`nextLockedNonce[asset]` is only advanced inside `_unlockWithdrawalRequests` (line 815). Even after the block-based delay has fully elapsed, users cannot reclaim their rsETH without the operator first calling `unlockQueue`. There is no cancellation path in the contract.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

User rsETH is transferred into `LRTWithdrawalManager` at `initiateWithdrawal` time and cannot be recovered until the operator calls `unlockQueue`. If the operator is delayed, paused, or simply slow, all rsETH sitting in the withdrawal queue is frozen for an indefinite period. Additionally, the inflated `assetsCommitted` prevents any new withdrawal initiations, compounding the freeze for the entire user base of a given asset.

---

### Likelihood Explanation

**Medium.** Operators are expected to call `unlockQueue` regularly as part of normal protocol operations. However, any operational delay — network congestion, price volatility causing `_validatePrices` to revert, operator key rotation, or a temporary pause — directly translates into user funds being locked with no permissionless escape hatch. The longer the operator delay, the larger the set of affected users and the greater the total rsETH frozen.

---

### Recommendation

Introduce a permissionless function that advances `nextLockedNonce[asset]` and decrements `assetsCommitted[asset]` for requests whose withdrawal delay has already passed, without requiring the full `unlockQueue` flow (which burns rsETH and redeems vault assets). Alternatively, allow `completeWithdrawal` to self-unlock a request if the delay has passed and the request has not yet been processed by the operator, so that users always have a permissionless path to recover their rsETH once the time condition is satisfied.

---

### Proof of Concept

1. Alice calls `initiateWithdrawal(ETH, 10e18, ...)`. Her rsETH is transferred to `LRTWithdrawalManager`; `assetsCommitted[ETH] += X`.
2. `withdrawalDelayBlocks` (≈8 days) elapses. Alice's request is now time-mature.
3. Bob calls `initiateWithdrawal(ETH, 1e18, ...)`. `getAvailableAssetAmount` returns `totalAssets - assetsCommitted[ETH]`, which is still reduced by Alice's uncommitted amount. If total assets equal the committed amount, Bob receives `ExceedAmountToWithdraw`.
4. Alice calls `completeWithdrawal(ETH, ...)`. The check `usersFirstWithdrawalRequestNonce >= nextLockedNonce[ETH]` reverts with `WithdrawalLocked` because the operator has not yet called `unlockQueue`.
5. Alice's rsETH remains locked in `LRTWithdrawalManager` with no permissionless recovery path until the operator calls `unlockQueue`.
