### Title
Fee-on-Transfer Token Accounting Divergence Breaks Bridge Backing Guarantee — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.initTransfer` records the caller-supplied `amount` as the bridged quantity and emits it cross-chain, but never verifies how many tokens the contract actually received. For fee-on-transfer ERC20 tokens the bridge is credited with more than it holds, permanently breaking the 1-to-1 backing guarantee and eventually making redemptions impossible for later users.

### Finding Description
In `initTransfer`, when the token is neither a bridge-deployed token nor a custom-minter token, the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-controlled
);
``` [1](#0-0) 

Immediately after, the unverified caller-supplied `amount` is forwarded to `initTransferExtension` and emitted in `InitTransfer`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
``` [2](#0-1) 

No pre/post balance snapshot is taken. For a fee-on-transfer token (e.g. STA, PAXG, or any token with an active fee switch), the bridge receives `amount − fee_amount` but the NEAR side is told `amount` was deposited and mints the full `amount` of wrapped tokens to the user.

On the redemption path (`finTransfer`), the bridge releases the full recorded `amount`:

```solidity
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
``` [3](#0-2) 

Because the bridge holds less than it owes, each round-trip drains the shortfall from the pool of other depositors' funds. Once the deficit accumulates, `safeTransfer` reverts and all remaining holders of that wrapped token are permanently locked out.

The contract imports and uses `SafeERC20` correctly for the non-bool-returning / false-returning token classes: [4](#0-3) [5](#0-4) 

Those two categories from the external report are therefore mitigated. The fee-on-transfer gap is the remaining unmitigated analog.

### Impact Explanation
Every deposit of a fee-on-transfer token over-credits the NEAR side by the fee amount. The bridge's EVM custody balance falls below the total outstanding wrapped supply. When the deficit is large enough, `safeTransfer` in `finTransfer` reverts, permanently locking all remaining depositors' funds in the bridge — matching the **Critical** impact class: *irreversible fund lock / permanently unclaimable user value in bridge flows*, and the **High** class: *balance-accounting divergence that breaks backing guarantees*.

### Likelihood Explanation
Medium. Fee-on-transfer tokens are a known, deployed token class (STA, PAXG, tokens with activatable fee switches such as USDT on some chains). `initTransfer` is a fully public, permissionless entry point — no privileged role is required. Any user can call it with any ERC20 address.

### Recommendation
Record the actual received amount by snapshotting the contract's balance before and after the transfer, and use the delta as the canonical bridged amount:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use `received` (cast to uint128 with overflow check) instead of `amount`
```

Alternatively, explicitly document and enforce (via a token allowlist or a revert guard) that fee-on-transfer tokens are not accepted, so the protocol cannot be put into an insolvent state.

### Proof of Concept
1. Deploy or use any ERC20 that deducts a 1 % transfer fee.
2. Call `OmniBridge.initTransfer(feeToken, 1000e18, 0, 0, "alice.near", "")`.
3. Bridge receives 990e18 tokens; NEAR side mints 1000e18 wrapped tokens to the caller.
4. Repeat N times until the bridge's custody balance is exhausted.
5. Any subsequent `finTransfer` for that token reverts — all remaining wrapped-token holders are permanently unable to redeem.
