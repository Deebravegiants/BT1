### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Any Attested Participant to Deliver Forged Confidential Key - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` performs no cryptographic verification of the CKD response when the request uses the `AppPublicKey` variant. Any single attested participant can front-run the honest leader and submit an arbitrary forged CKD payload; the contract accepts it unconditionally and delivers it to the waiting user.

### Finding Description

`respond_ckd` dispatches on the `app_public_key` field of the request:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant the contract calls `ckd_output_check` and panics on failure. For the `AppPublicKey` variant the arm is completely empty — the response bytes are passed directly to `resolve_yields_for` and delivered to the user with zero on-chain verification. [2](#0-1) 

`resolve_yields_for` removes the pending entry on the first successful call and resumes all queued yields with whatever bytes were supplied: [3](#0-2) 

Because the entry is removed on the first call, whichever participant calls `respond_ckd` first wins the race. The honest leader's subsequent call returns `RequestNotFound` and is silently discarded.

The only guards on `respond_ckd` are `assert_caller_is_signer` and `assert_caller_is_attested_participant_and_protocol_active`: [4](#0-3) 

Neither guard restricts which participant may respond, nor do they require threshold consensus. Any single attested participant satisfies both checks.

### Impact Explanation

A single Byzantine participant below the signing threshold can submit an arbitrary `CKDResponse` for any pending `AppPublicKey` CKD request. The contract delivers the forged payload to the user's yield-resume callback without any cryptographic check. The user receives a confidential key that was never produced by the threshold BLS protocol. If the user subsequently uses that key to control on-chain assets (the primary purpose of CKD), those assets are at risk of theft or permanent loss. This matches the Critical impact class: *confidential key derivation output without the required participant authorization*.

### Likelihood Explanation

The `AppPublicKey` variant is documented as the current "legacy" path and remains fully supported. All pending CKD request IDs and their `app_public_key` fields are visible on-chain the moment `request_app_private_key` is called. A malicious participant needs only to observe the chain, construct any `CKDResponse` (the format is public), and call `respond_ckd` before the honest leader does. No special knowledge, no threshold collusion, and no TEE bypass is required — a single compromised or malicious participant suffices.

### Recommendation

Apply the same `ckd_output_check` guard unconditionally, regardless of which `CKDAppPublicKey` variant was used. If on-chain verification is structurally impossible for the `AppPublicKey` variant (because it lacks the second G1 point needed for public verifiability), the `AppPublicKey` variant should be deprecated and rejected at the `request_app_private_key` entry point so that only the verifiable `AppPublicKeyPV` path remains reachable. Concretely:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {
        // Cannot verify on-chain; reject to prevent forged delivery.
        env::panic_str("AppPublicKey variant is deprecated; use AppPublicKeyPV");
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

1. Alice submits `request_app_private_key` with `AppPublicKey` variant. The request appears on-chain with a known `CKDRequest` key.
2. Mallory (any attested participant) observes the pending request and immediately calls `respond_ckd` with a `CKDResponse` containing an arbitrary, attacker-chosen encrypted blob.
3. `resolve_yields_for` removes the pending entry and resumes Alice's yield with Mallory's payload.
4. The honest leader's subsequent `respond_ckd` call returns `Err(RequestNotFound)` and is dropped.
5. Alice's callback receives Mallory's forged confidential key. Any assets Alice subsequently controls with that key are under Mallory's control.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
