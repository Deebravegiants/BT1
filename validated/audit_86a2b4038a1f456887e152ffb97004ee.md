Audit Report

## Title
Permissionless `updateRate()` Enables Profitable Deposits Against Stale Cross-Chain Rate — (`contracts/cross-chain/CrossChainRateProvider.sol`, `contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary
`CrossChainRateProvider.updateRate()` and `MultiChainRateProvider.updateRate()` carry no access control, allowing any caller to broadcast the current L1 `rsETHPrice` to L2 receiver contracts at will. L2 pool contracts (`RSETHPoolV3`, `RSETHPoolNoWrapper`) price deposits using the stale cached rate in `CrossChainRateReceiver`, so an attacker can deposit at the lower stale rate to receive excess rsETH/wrsETH, then immediately trigger the rate update themselves. For `RSETHPoolV3`, the excess is freshly minted wrsETH, directly diluting existing holders' unclaimed yield.

## Finding Description
**Permissionless rate broadcast.** `CrossChainRateProvider.updateRate()` is declared `external payable nonReentrant` with no `onlyOwner` or role guard. [1](#0-0) 
`MultiChainRateProvider.updateRate()` is identical in access posture. [2](#0-1) 

**Stale cached rate on L2.** `CrossChainRateReceiver` stores the last received rate in `rate` storage, updated only when `lzReceive` is called by the LayerZero endpoint. Between updates, `getRate()` returns the stale value. [3](#0-2) 

**L2 pools use the stale rate for minting.** `RSETHPoolV3.getRate()` and `RSETHPoolNoWrapper.getRate()` both delegate to `IOracle(rsETHOracle).getRate()`, which resolves to `CrossChainRateReceiver.rate`. The deposit path computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` using this stale value. [4](#0-3) [5](#0-4) 

**`RSETHPoolV3` mints new supply.** The deposit function calls `wrsETH.mint(msg.sender, rsETHAmount)`, creating new wrsETH tokens. Excess minting directly dilutes all existing wrsETH holders. [6](#0-5) 

**`RSETHPoolNoWrapper` transfers from pool reserves.** Unlike `RSETHPoolV3`, this pool calls `rsETH.safeTransfer(msg.sender, rsETHAmount)` from its pre-funded balance. Excess transfers drain the pool's rsETH reserves faster than intended, representing theft from the pool's yield-bearing holdings. [7](#0-6) 

**Existing guards are insufficient.** The `dailyMintLimit` modifier in `RSETHPoolV3` caps total daily minting volume but does not prevent minting at a stale rate — it only bounds the scale of each attack window. [8](#0-7) 
The `pricePercentageLimit` guard in `LRTOracle._updateRsETHPrice()` applies only to L1 oracle updates and has no effect on the L2 cached rate. [9](#0-8) 

## Impact Explanation
**High — Theft of unclaimed yield.**

When `rsETHPrice` on L1 has risen above the stale `CrossChainRateReceiver.rate` on L2, the formula `rsETHAmount = amountAfterFee * 1e18 / staleRate` produces more wrsETH than the deposited ETH is worth at the true rate. For `RSETHPoolV3`, this excess is freshly minted, diluting all existing wrsETH holders. The surplus wrsETH represents staking yield that belongs to existing holders but is instead captured by the attacker. The profit scales linearly with deposit size and the magnitude of the rate lag, and is repeatable after every reward accrual event on L1. [10](#0-9) 

## Likelihood Explanation
**High.** Both the L1 oracle price (`LRTOracle.rsETHPrice`, a public storage variable) and the L2 cached rate (`CrossChainRateReceiver.rate`, also public) are readable on-chain at any time without mempool monitoring. Because `updateRate()` is permissionless, the attacker controls the exact timing of the rate push — they can deposit at the stale rate and immediately trigger the update in the same or next block, making the attack deterministic and repeatable. No privileged access, victim cooperation, or external protocol compromise is required. [11](#0-10) [12](#0-11) 

## Recommendation
1. **Restrict `updateRate()`** in both `CrossChainRateProvider` and `MultiChainRateProvider` to a trusted role (e.g., `onlyOwner` or a dedicated `RATE_UPDATER_ROLE`), preventing an attacker from controlling the timing of rate propagation.
2. **Add a minimum deposit lock-up or redemption delay** on L2 pools so that wrsETH minted cannot be immediately redeemed or sold against a freshly updated rate in the same block.
3. **Bound the acceptable rate delta** in `CrossChainRateReceiver.lzReceive()` — reject updates that increase the rate by more than a configurable threshold per period, consistent with the `pricePercentageLimit` guard already present in `LRTOracle._updateRsETHPrice()` on L1. [9](#0-8) 

## Proof of Concept
**Minimal call sequence (fork test against deployed contracts):**

1. Read `LRTOracle.rsETHPrice()` on L1 → `1.05e18`.
2. Read `CrossChainRateReceiver.rate` on L2 → `1.04e18` (stale).
3. Call `RSETHPoolV3.deposit{value: 100 ether}("")` on L2.
   - `viewSwapRsETHAmountAndFee(100e18)` computes: `rsETHAmount = 100e18 * 1e18 / 1.04e18 ≈ 96.15 wrsETH`
   - At the true rate: `100e18 * 1e18 / 1.05e18 ≈ 95.24 wrsETH`
   - **Excess minted: ~0.91 wrsETH** (≈ 0.96 ETH of value at current rate).
4. Call `RSETHRateProvider.updateRate{value: lzFee}()` on L1 — permissionless, no revert.
5. LayerZero delivers the message; `CrossChainRateReceiver.rate` is updated to `1.05e18`.
6. Attacker redeems or sells the 96.15 wrsETH at the new `1.05e18` rate, recovering ≈ 100.96 ETH — a net profit of ≈ 0.96 ETH on a 100 ETH deposit, minus gas and LayerZero fees.

**Foundry fork test plan:**
- Fork L2 (e.g., Optimism/Arbitrum) at a block where `CrossChainRateReceiver.rate < LRTOracle.rsETHPrice` (or mock the receiver to set a stale rate).
- Assert `wrsETH.balanceOf(attacker)` after deposit exceeds `100e18 * 1e18 / trueRate`.
- Assert total wrsETH supply increased by more than the fair share.
- Confirm `updateRate()` succeeds from an unprivileged EOA with sufficient ETH for LayerZero fees. [12](#0-11) [13](#0-12)
