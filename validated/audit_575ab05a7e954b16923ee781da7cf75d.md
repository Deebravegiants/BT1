### Title
Zero-Confirmation Bitcoin Transaction Accepted and Signed Without Minimum Finality Enforcement - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` flow accepts a caller-supplied `BlockConfirmations` value of `0` for Bitcoin requests. Because the inspector's finality check is `block_confirmations_threshold <= transaction_block_confirmation`, a threshold of zero always passes regardless of the transaction's actual confirmation count. The MPC network will therefore sign a payload attesting to a Bitcoin transaction that is still unconfirmed (mempool-only), enabling a bridge contract to release funds for a transaction that may never finalize or may be double-spent.

---

### Finding Description

**Root cause — no minimum confirmation floor anywhere in the stack.**

`BlockConfirmations` is a plain `u64` newtype with no invariant: [1](#0-0) 

The ABI schema explicitly allows `0`: [2](#0-1) 

`verify_foreign_transaction` on the contract performs no minimum-confirmation check before enqueuing the request: [3](#0-2) 

Inside `BitcoinInspector::extract`, the finality guard is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [4](#0-3) 

When `block_confirmations_threshold == 0`, the inequality `0 <= transaction_block_confirmation` is trivially true for any `u64`, including a mempool transaction that has `confirmations = 0` from the RPC. The canonical-chain check that follows (`verify_block_is_canonical`) is also skipped for unconfirmed transactions because `getrawtransaction` on an unconfirmed tx returns no `blockhash`, which would surface as a deserialization/client error rather than a security rejection — but the confirmation check fires first and already passes.

**Exploit path:**

1. Attacker broadcasts a Bitcoin transaction `T` to the mempool (0 confirmations).
2. Attacker calls `verify_foreign_transaction` with `BitcoinRpcRequest { tx_id: T, confirmations: 0, extractors: [BlockHash] }`.
3. MPC nodes query `getrawtransaction` — the tx exists in the mempool; `confirmations` returned is `0`.
4. `0 <= 0` → `enough_block_confirmations = true`; the inspector proceeds and signs.
5. The MPC network produces a valid threshold signature over `SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))`.
6. Attacker presents this signature to a bridge contract (e.g., Omnibridge inbound flow) as proof that `T` finalized.
7. Bridge releases funds. Attacker then double-spends `T` on Bitcoin (RBF or simply never mines it).

The signed payload is cryptographically valid — the contract's `respond_verify_foreign_tx` verifies only that the signature matches the domain public key, not that the underlying finality claim is sound: [5](#0-4) 

---

### Impact Explanation

**High — forged foreign-chain verification enabling double-spend / invalid bridge execution.**

The MPC network's core promise for the foreign-chain feature is that it signs only after verifying a transaction has reached the requested finality level. With `confirmations: 0` accepted, that promise is broken: the signature attests to a transaction that has not finalized and may never finalize. Any bridge contract that relies on the MPC signature as a finality proof (the primary stated use-case — Omnibridge inbound flow) can be drained by presenting a signature over an unconfirmed transaction and then double-spending it on Bitcoin.

---

### Likelihood Explanation

**Low-to-Medium.** The attack requires:
- An unprivileged NEAR account (no special role needed).
- A bridge contract that trusts the MPC signature without independently enforcing a minimum confirmation count.
- A Bitcoin transaction that can be double-spent (RBF-flagged or simply dropped).

The first condition is trivially satisfied. The second is the intended production use-case (Omnibridge). The third is routine on Bitcoin. No threshold collusion, no TEE break, and no privileged operator access is required.

---

### Recommendation

Enforce a protocol-level minimum confirmation floor. The fix has two layers:

1. **Contract-side (`verify_foreign_transaction`)**: Reject `BitcoinRpcRequest` with `confirmations == 0` before enqueuing:

```rust
if let ForeignChainRpcRequest::Bitcoin(ref btc) = request.request {
    if btc.confirmations.0 == 0 {
        env::panic_str("Bitcoin confirmations must be at least 1");
    }
}
```

2. **Inspector-side (`BitcoinInspector::extract`)**: Add a hard floor (e.g., `MIN_BITCOIN_CONFIRMATIONS = 1`) independent of the caller-supplied threshold, so that even if the contract check is bypassed or the threshold is set to 0 by a future code path, the node refuses to sign:

```rust
const MIN_BITCOIN_CONFIRMATIONS: BlockConfirmations = BlockConfirmations(1);
let effective_threshold = block_confirmations_threshold.max(MIN_BITCOIN_CONFIRMATIONS);
``` [4](#0-3)

### Proof of Concept

```
1. Broadcast Bitcoin tx T (RBF-flagged) to mempool.
2. Call verify_foreign_transaction({
       domain_id: <foreign_tx_domain>,
       payload_version: V1,
       request: Bitcoin({ tx_id: T, confirmations: 0, extractors: [BlockHash] })
   }) with 1 yoctoNEAR deposit.
3. MPC nodes call getrawtransaction(T, verbose=true).
   → Response: { blockhash: null, confirmations: 0 }
   → block_confirmations_threshold (0) <= transaction_block_confirmation (0) → true
   → Inspector proceeds; BlockHash extractor returns null/zero hash.
4. MPC threshold-signs SHA-256(borsh(ForeignTxSignPayloadV1{request, values})).
5. respond_verify_foreign_tx delivers valid Secp256k1 signature to contract.
6. Contract resolves the yield; caller receives VerifyForeignTransactionResponse.
7. Attacker submits response to bridge contract → bridge releases funds.
8. Attacker RBF-replaces T with a conflicting transaction → double-spend complete.
```
