## Core Issue: `has_promise_yield_status` checks key existence, not value

`has_promise_yield_status` in `core/store/src/utils/mod.rs` only checks whether the `PromiseYieldStatus` trie key **exists**, regardless of its value: [1](#0-0) 

After the first `submit_promise_resume_data` call succeeds, it sets the status to `ResumeInitiated` — but the key **still exists** in the `TrieUpdate`. The second call therefore passes the same guard.

## Double-Resume Path in `submit_promise_resume_data` [2](#0-1) 

**First call** with `data_id`:
- `has_yield_status_in_state` = `true` (status is `Yielded`)
- `create_promise_resume_receipt` is called → resume receipt #1 appended
- Status overwritten to `ResumeInitiated` in `TrieUpdate`

**Second call** with the same `data_id`:
- `has_yield_status_in_state` = `true` again — `ResumeInitiated` is still present in the `TrieUpdate` under the same key
- `create_promise_resume_receipt` is called again → resume receipt #2 appended
- No deduplication guard exists

`create_promise_resume_receipt` is a blind `Vec::push` with no uniqueness check on `data_id`: [3](#0-2) 

## Two Attack Scenarios

**Scenario A — yield created in a prior chunk:**
The `PromiseYieldReceipt` is stored in the trie. `has_promise_yield_receipt` returns `true` on both calls because the receipt is never removed from the `TrieUpdate` during the same function call execution. [4](#0-3) 

**Scenario B — yield created in the same chunk:**
`create_promise_yield_receipt` writes `PromiseYieldStatus::Yielded` into the `TrieUpdate`: [5](#0-4) 

The first resume overwrites it to `ResumeInitiated` but the key persists, so the second call's `has_promise_yield_status` check still returns `true`.

## Impact

Two `DataReceiptMetadata` entries with identical `data_id` and `is_promise_resume: true` appear in `ReceiptManager::data_receipts`. These are converted to two outgoing `PromiseResume` receipts for the same `data_id`, producing a **different receipt set and state root** than a single-resume execution. Any node replaying the same input will produce the same two receipts (the behavior is deterministic), but it violates the protocol invariant that at most one `PromiseResume` receipt is emitted per `data_id` per chunk. Downstream processing of two resumes for the same yield is undefined and can corrupt yield state.

## Root Cause

The guard condition `has_yield_receipt_in_state || has_yield_status_in_state` is not invalidated after the first resume within the same `TrieUpdate` scope. The correct fix is to either remove the `PromiseYieldStatus` key (via `remove_promise_yield_status`) after emitting the resume receipt, or check the actual status value and reject if it is already `ResumeInitiated`.

### Title
Double `PromiseResume` receipt emission via repeated `promise_yield_resume` with same `data_id` — (`runtime/runtime/src/ext.rs`)

### Summary
`submit_promise_resume_data` guards against duplicate resumes using `has_promise_yield_status`, which only checks key existence. After the first call sets the status to `ResumeInitiated`, the key remains in the `TrieUpdate`, so a second call with the same `data_id` passes the guard and appends a second `PromiseResume` receipt, violating the one-resume-per-data_id invariant.

### Finding Description
In `ext.rs::submit_promise_resume_data`, after emitting a `PromiseResume` receipt, the code calls `set_promise_yield_status(..., ResumeInitiated)` but does not remove the status key. `has_promise_yield_status` uses `trie.contains_key(...)`, which returns `true` for any value including `ResumeInitiated`. A contract calling `promise_yield_resume` twice with the same `data_id` in one execution therefore triggers `create_promise_resume_receipt` twice, appending two entries to `data_receipts` with the same `data_id`.

### Impact Explanation
Two `PromiseResume` receipts with the same `data_id` appear in the outgoing receipt set, producing a different receipt set and state root than a correct single-resume execution. This is a consensus-critical state divergence reachable by any unprivileged contract deployer.

### Likelihood Explanation
Any account can deploy a contract and call `promise_yield_resume` twice with the same `data_id` in a single function call. No special privileges are required.

### Recommendation
In `submit_promise_resume_data`, after emitting the resume receipt, call `remove_promise_yield_status` instead of (or in addition to) setting `ResumeInitiated`, so that subsequent calls within the same `TrieUpdate` scope find no status key and return `false`. Alternatively, read the actual status value and reject if it is already `ResumeInitiated`.

### Proof of Concept
Deploy a contract with a method that:
1. Calls `promise_yield_create` to obtain a `data_id`
2. Immediately calls `promise_yield_resume(data_id, b"payload1")`
3. Immediately calls `promise_yield_resume(data_id, b"payload2")`

Assert that `ReceiptManager::data_receipts` contains two entries with the same `data_id` and `is_promise_resume: true`, and verify the runtime emits both as outgoing receipts.

    fn submit_promise_resume_data(
        &mut self,
        data_id: CryptoHash,
        data: Vec<u8>,
    ) -> Result<bool, VMLogicError> {
        let has_yield_receipt_in_state =
            has_promise_yield_receipt(self.trie_update, self.account_id.clone(), data_id)
                .map_err(wrap_storage_error)?;
        let has_yield_status_in_state =
            has_promise_yield_status(self.trie_update, &self.account_id, data_id)
                .map_err(wrap_storage_error)?;

        if has_yield_receipt_in_state || has_yield_status_in_state {
            self.receipt_manager.create_promise_resume_receipt(data_id, data);
            set_promise_yield_status(
                &mut self.trie_update,
                &self.account_id,
                data_id,
                PromiseYieldStatus::ResumeInitiated,
            );
            return Ok(true);
        }

        Ok(false)
    }
