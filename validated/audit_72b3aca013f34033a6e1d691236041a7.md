### Title
Single-Participant Chain Deregistration Permanently Freezes Already-Queued `verify_foreign_transaction` Requests - (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` flow performs a supported-chain check at request creation time (on-chain, in `verify_foreign_transaction()`) and again at processing time (off-chain, in `execute_foreign_chain_request()`). Because `get_supported_foreign_chains()` computes the **strict intersection** of every active participant's registered chains, a single Byzantine participant below the signing threshold can re-register with an empty chain set after a request is already queued. This causes the node-side `chain_is_supported()` check to fail for every processing attempt, exhausting all retries and permanently freezing the pending yield until it times out.

---

### Finding Description

**At request creation time**, `verify_foreign_transaction()` in `crates/contract/src/lib.rs` checks whether the requested chain is currently supported:

```rust
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(...);
}
// ... enqueue_yield_request(...)
```

`get_supported_foreign_chains()` computes the strict intersection of all active participants' registered chains:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
if all_active_nodes_supports_chain { Some(foreign_chain) } else { None }
```

If the check passes, the request is stored as a pending yield in `pending_verify_foreign_tx_requests`.

**At processing time**, every MPC node leader calls `execute_foreign_chain_request()` in `crates/node/src/providers/verify_foreign_tx/sign.rs`, which re-evaluates the same condition by reading the current on-chain state:

```rust
async fn execute_foreign_chain_request(...) -> anyhow::Result<...> {
    chain_is_supported(&self.foreign_chain_policy_reader, request).await?;
    ...
}
```

`chain_is_supported()` calls `ReadSupportedForeignChain::get_supported_chains()`, which issues a live view call to `GET_SUPPORTED_FOREIGN_CHAINS` on the contract — the same `get_supported_foreign_chains()` function. If the chain is no longer in the intersection, the function returns `Err(ForeignChainSupportError::ChainNotSupported)`, causing the leader to abort without producing a signature share.

**The gap**: Between the moment the yield is parked and the moment nodes attempt to process it, any single active participant can call `register_foreign_chain_config` (or `register_available_foreign_chain_config`) with an empty set or a set that excludes the requested chain. Because the intersection rule requires **all** participants to support a chain, one participant's deregistration immediately removes the chain from `get_supported_foreign_chains()`. Every subsequent leader attempt fails the node-side check. After `MAX_ATTEMPTS_PER_REQUEST_AS_LEADER` failures per node, the request is dropped with `DropReason::MaxAttemptsExceeded`. The on-chain yield then times out (~200 blocks), and the user receives a timeout error instead of a signature.

The design document explicitly acknowledges this property of the intersection rule: *"A single node that registers an empty list (or hasn't registered yet) drops every chain — one operator can take the whole feature down."* The proposed fix (tracked in issue #3434) is not yet implemented in production code.

---

### Impact Explanation

**Medium.** A single Byzantine participant (strictly below the signing threshold) can permanently freeze any already-queued `verify_foreign_transaction` request for a targeted chain. The user's yield times out, the bridge or application that submitted the request receives no signature, and the cross-chain operation it depended on cannot proceed. This breaks the request-lifecycle safety invariant: a request that passed all creation-time checks and was accepted into the pending queue is never fulfilled. The 1 yoctoNEAR deposit is lost (trivial), but the inability to obtain the foreign-chain signature is the real harm for bridge users.

---

### Likelihood Explanation

Any single active participant can trigger this by calling `register_foreign_chain_config` (or `register_available_foreign_chain_config`) with a chain set that excludes the target chain. The call is permissioned to participants only, but no threshold vote is required — one participant suffices. The attack is cheap, requires no special capability beyond being an active participant, and can be timed precisely to target specific in-flight requests. The window between `verify_foreign_transaction` acceptance and node processing is at least one block and can be several blocks in practice.

---

### Recommendation

1. **Decouple the creation-time and processing-time checks.** Snapshot the supported-chain set at request creation time and store it alongside the pending request, so the node-side check uses the same set that was valid when the request was accepted. Alternatively, skip the node-side `chain_is_supported` re-check entirely: if the contract accepted the request, nodes should attempt to fulfill it regardless of subsequent chain-set changes.

2. **Implement the proposed "available set" fix** (issue #3434): replace the strict-intersection rule with a threshold-based availability check (`≥ signing_threshold` participants cover the chain). This prevents a single participant from removing a chain from the supported set and eliminates the root cause of both the creation-time and processing-time divergence.

3. **Treat `ChainNotSupported` at processing time as a non-retryable terminal error** (as noted in the design doc for sub-quorum outcomes) so that the request is dropped immediately rather than exhausting all retry slots, giving the user a faster and clearer failure signal.

---

### Proof of Concept

**Setup**: Network with participants P1, P2, P3 (threshold = 2). All three have registered Bitcoin as supported. `get_supported_foreign_chains()` = `{Bitcoin}`.

**Step 1 — User submits request**: User calls `verify_foreign_transaction(Bitcoin)`. The contract checks `get_supported_foreign_chains().contains(Bitcoin)` → `true`. The yield is parked in `pending_verify_foreign_tx_requests`.

**Step 2 — Byzantine participant deregisters**: P3 (a single participant, below the signing threshold of 2) calls `register_foreign_chain_config({})` (empty set). The contract updates `node_foreign_chain_support` for P3. `get_supported_foreign_chains()` now computes the intersection: P1 supports Bitcoin, P2 supports Bitcoin, P3 supports nothing → intersection = `{}`. Bitcoin is no longer in the supported set.

**Step 3 — Node processing fails**: The leader node (say P1) picks up the pending request and calls `execute_foreign_chain_request()`. `chain_is_supported()` calls `get_supported_chains()` → reads current on-chain state → `{}`. Bitcoin is not in `{}`. Returns `Err(ChainNotSupported { requested: Bitcoin })`. The leader aborts without producing a signature share.

**Step 4 — All retries fail**: Every subsequent leader (P1, P2, P3 in rotation) hits the same `chain_is_supported` failure. After `MAX_ATTEMPTS_PER_REQUEST_AS_LEADER` attempts, the request is dropped with `DropReason::MaxAttemptsExceeded`.

**Step 5 — Yield times out**: The on-chain yield for the user's `verify_foreign_transaction` call times out after ~200 blocks. The user receives a timeout error. The bridge operation that depended on the signature cannot proceed.

```rust
async fn chain_is_supported(
    policy_reader: &impl ReadSupportedForeignChain,
    request: &dtos::ForeignChainRpcRequest,
) -> Result<(), ForeignChainSupportError> {
    let on_chain_foreign_chains_support = policy_reader
        .get_supported_chains()
        .await
        .map_err(ForeignChainSupportError::FailedToReadContract)?;

    let requested_chain = request.chain();

    if on_chain_foreign_chains_support.contains(&requested_chain) {
        Ok(())
    } else {
        Err(ForeignChainSupportError::ChainNotSupported {
            requested: requested_chain,
        })
    }
}
```
