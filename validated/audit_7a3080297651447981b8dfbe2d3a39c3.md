Audit Report

## Title
Uninitialized `tokenFeeBps` Mapping Allows Zero-Fee Token Deposits After `addSupportedToken()` - (File: contracts/pools/RSETHPool.sol)

## Summary

`RSETHPool.sol` declares a per-token fee mapping `tokenFeeBps` that is never initialized when a new token is added via `addSupportedToken()`. Because Solidity mappings default to zero, any depositor who calls `deposit(token, amount, referralId)` before the admin separately calls `setTokenFeeBps(token, feeBps)` pays a 0% fee, causing the protocol to permanently lose fee revenue on those deposits.

## Finding Description

`RSETHPool.sol` declares the per-token fee mapping at line 88:

```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
```

`addSupportedToken()` (lines 637–656) is gated by `TIMELOCK_ROLE` and sets `supportedTokenOracle[token]` and `tokenBridge[token]`, but never touches `tokenFeeBps[token]`. After this call, the `onlySupportedToken` modifier (line 100–103) passes for the new token because it only checks `supportedTokenOracle[token] != address(0)`, which is now set. The token is immediately depositable.

`setTokenFeeBps()` (lines 583–594) is a completely separate privileged call gated by `DEFAULT_ADMIN_ROLE` — a different role from `TIMELOCK_ROLE` — making atomic initialization in a single transaction structurally impossible.

`viewSwapRsETHAmountAndFee(amount, token)` (lines 335–336) reads `tokenFeeBps[token]` directly:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```

With `feeBpsForToken == 0`, `fee == 0` and `amountAfterFee == amount`. The `deposit(address token, ...)` function (lines 298–300) accumulates this zero fee:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;  // += 0
```

The depositor receives the full token value converted to rsETH with no fee deducted, and `feeEarnedInToken[token]` remains zero for those deposits permanently.

## Impact Explanation

Every token deposit made in the window between `addSupportedToken()` and `setTokenFeeBps()` pays zero protocol fee. The protocol's `feeEarnedInToken[token]` accumulates nothing for those deposits, permanently losing that fee revenue. This constitutes **theft of unclaimed yield** (protocol fee revenue) — a High severity impact under the allowed scope.

## Likelihood Explanation

The window exists every time a new token is listed. The two calls are made by different roles (`TIMELOCK_ROLE` vs `DEFAULT_ADMIN_ROLE`), making atomic initialization impossible without a multi-sig or deployment script that batches them. Any depositor — without any special privileges — can deposit in this window. Even without active front-running, the window may span multiple blocks or longer if the admin delays calling `setTokenFeeBps`. The condition is structurally guaranteed by the two-step design, making this Medium likelihood.

## Recommendation

Initialize `tokenFeeBps[token]` inside `addSupportedToken()` by accepting a `_feeBps` parameter:

```diff
function addSupportedToken(
    address token,
    address oracle,
-   address bridge
+   address bridge,
+   uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    // ...
+   if (_feeBps > 10_000) revert InvalidFeeAmount();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
+   tokenFeeBps[token] = _feeBps;
+   emit TokenFeeBpsSet(token, _feeBps);
    emit AddSupportedToken(token, oracle, bridge);
}
```

This eliminates the uninitialized window entirely, mirroring how `feeBps` is set atomically in `initialize()` for ETH deposits.

## Proof of Concept

**Setup:** `RSETHPool` is live on Arbitrum. Admin calls `addSupportedToken(rETH, rETHOracle, rETHBridge)`.

**Block N:** `addSupportedToken` executes. `tokenFeeBps[rETH] == 0` (Solidity default). `onlySupportedToken(rETH)` now passes.

**Block N (same or next):** Attacker calls `deposit(rETH, 100e18, "")`.

Inside `viewSwapRsETHAmountAndFee(100e18, rETH)`:
```
feeBpsForToken = tokenFeeBps[rETH] = 0
fee = 100e18 * 0 / 10_000 = 0
amountAfterFee = 100e18
rsETHAmount = 100e18 * rETHToETHRate / rsETHToETHrate  // full value, no fee
```
`feeEarnedInToken[rETH] += 0` — protocol earns nothing.

**Block N+1:** Admin calls `setTokenFeeBps(rETH, 50)`. The attacker's deposit already settled at 0% fee; the protocol permanently lost the fee revenue on that deposit.

A Foundry fork test can reproduce this by: (1) deploying/forking the contract, (2) calling `addSupportedToken`, (3) calling `deposit` from an unprivileged address before `setTokenFeeBps`, and (4) asserting `feeEarnedInToken[rETH] == 0` while the depositor received full rsETH value.
