### Title
Missing Chain ID in `SignedDelegateAction` Signed Payload Enables Cross-Network Replay - (File: core/primitives/src/signable_message.rs)

### Summary
The `SignableMessage` scheme used to sign `DelegateAction` (NEP-366 meta transactions) and `DelegateActionV2` (NEP-611 gas keys) does not include a chain ID or network identifier in the signed payload. A signed `DelegateAction` produced on one NEAR network (e.g., testnet) is cryptographically valid on any other NEAR network (e.g., mainnet) where the same account and key exist with a compatible nonce and `max_block_height`. An attacker acting as a relayer — an explicitly untrusted role in the meta-transaction model — can replay the signed action on a different network than the user intended, causing unauthorized state changes on that network.

### Finding Description

`SignableMessage` is the signing envelope for meta transactions:

```rust
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,   // only a NEP number, e.g. 366
    pub msg: &'a T,
}
``` [1](#0-0) 

The `discriminant` is derived solely from a NEP number constant:

```rust
const NEP_366_META_TRANSACTIONS: u32 = 366;
const NEP_611_GAS_KEYS: u32 = 611;
``` [2](#0-1) 

No chain ID, genesis hash, or network identifier is mixed into the signed bytes. `DelegateAction::get_nep461_hash()` hashes exactly this envelope:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
``` [3](#0-2) 

`SignedDelegateAction::verify()` checks only this hash against the public key:

```rust
pub fn verify(&self) -> bool {
    let hash = delegate_action.get_nep461_hash();
    self.signature.verify(hash.as_ref(), public_key)
}
``` [4](#0-3) 

The `DelegateAction` body contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — none of which are network-specific. The same is true for `DelegateActionV2` / `VersionedDelegateActionPayload::get_nep461_hash()`: [5](#0-4) 

The `SignableMessage::sign()` path confirms no additional context is added: [6](#0-5) 

### Impact Explanation

A user who signs a `DelegateAction` on testnet (or any NEAR network) produces a signature that is equally valid on mainnet if:
- The same `sender_id` account exists on mainnet,
- The same `public_key` is registered as an access key on mainnet,
- The `nonce` in the action is ≥ the current nonce of that key on mainnet, and
- The current mainnet block height is ≤ `max_block_height`.

All four conditions are routinely satisfied: developers frequently use the same account IDs and key pairs across testnet and mainnet; mainnet and testnet block heights are in the same order of magnitude; and `max_block_height` is typically set hundreds of blocks in the future.

The attacker submits the replayed `SignedDelegateAction` inside a normal `Transaction` via public RPC. The runtime's `apply_delegate_action` path verifies only the signature and nonce, then executes the inner actions with `predecessor_id = sender_id`. Corrupted protocol values include: account balance (token transfers), access key set (AddKey / DeleteKey), and contract storage (FunctionCall). [7](#0-6) 

### Likelihood Explanation

The NEAR meta-transaction model explicitly designates the relayer as an **untrusted** third party. Any account can act as a relayer by wrapping a `SignedDelegateAction` in a transaction and submitting it via RPC — no special protocol privilege is required. An attacker who operates a relayer service, or who intercepts the off-chain transmission of a signed action, can trivially replay it on a different network. The conditions for a successful replay (shared account ID, shared key, compatible nonce) are common in practice, especially for developers and dApp users who operate on both testnet and mainnet.

### Recommendation

Include the chain ID (or genesis block hash, which is already used as a unique network identifier in NEAR) in the `SignableMessage` signed payload, analogous to EIP-712's `domainSeparator`. Concretely, add a `chain_id` or `genesis_hash` field to `MessageDiscriminant` or to the `SignableMessage` wrapper, and verify it matches the executing network inside `apply_delegate_action`. This is the same fix recommended by the external report: bind the signature to the specific network context in which it is intended to be used.

### Proof of Concept

1. On testnet, user Alice creates and signs:
   ```
   DelegateAction {
     sender_id: "alice.near",
     receiver_id: "token.near",
     actions: [FunctionCall("ft_transfer", {"receiver_id":"attacker","amount":"1000000"})],
     nonce: 5,
     max_block_height: testnet_height + 500,
     public_key: alice_key,
   }
   ```
2. Alice sends the `SignedDelegateAction` to what she believes is a legitimate relayer.
3. The attacker (relayer) instead constructs a mainnet `Transaction` with `signer = attacker_account` and `actions = [Action::Delegate(signed_delegate_action)]` and submits it to the mainnet RPC.
4. Mainnet runtime calls `apply_delegate_action`, which calls `SignedDelegateAction::verify()`. The hash is `sha256(borsh(discriminant=NEP_366) || borsh(DelegateAction))` — identical on both networks because no chain ID is included.
5. Verification passes. The `ft_transfer` executes on mainnet `token.near` with `predecessor_id = "alice.near"`, draining Alice's mainnet token balance to the attacker. [8](#0-7) [9](#0-8)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}
