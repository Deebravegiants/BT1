### Title
P2P Sync Peer Can Inject Mismatched Transaction/Receipt Type Pairs Into Storage, Causing `starknet_getTransactionReceipt` to Return Wrong Receipt Type — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

`TryFrom<protobuf::TransactionWithReceipt> for FullTransaction` independently deserializes the `transaction` and `receipt` fields without cross-checking their types. A malicious sync peer can send a `TransactionWithReceipt` where the `txn` oneof is `InvokeV3` but the `receipt.type` is `Declare`. The conversion succeeds, producing a `FullTransaction` with `Transaction::Invoke` paired with `TransactionOutput::Declare`. This mismatched pair is stored verbatim in `block_body` and later served by `starknet_getTransactionReceipt` as an authoritative `"DECLARE"` receipt for what is stored as an `INVOKE` transaction.

---

### Finding Description

**Deserialization without cross-type validation** (`crates/apollo_protobuf/src/converters/transaction.rs`, lines 109–121):

```rust
impl TryFrom<protobuf::TransactionWithReceipt> for FullTransaction {
    fn try_from(value: protobuf::TransactionWithReceipt) -> Result<Self, Self::Error> {
        let (transaction, transaction_hash) = <(Transaction, TransactionHash)>::try_from(
            value.transaction.ok_or(missing("TransactionWithReceipt::transaction"))?,
        )?;
        let transaction_output = TransactionOutput::try_from(
            value.receipt.ok_or(missing("TransactionWithReceipt::output"))?,
        )?;
        Ok(FullTransaction { transaction, transaction_output, transaction_hash })
    }
}
``` [1](#0-0) 

Each field is deserialized independently. There is no check that `transaction` and `transaction_output` carry the same variant (e.g., both `Invoke`, both `Declare`).

**Unvalidated storage** (`crates/apollo_p2p_sync/src/client/transaction.rs`, lines 86–89):

```rust
block_body.transactions.push(transaction);
block_body.transaction_outputs.push(transaction_output);
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [2](#0-1) 

The TODO comment itself acknowledges that peer-supplied data is not validated. No type-consistency check is performed before the `BlockBody` is committed to storage via `append_body`. [3](#0-2) 

**Wrong RPC output** (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`, lines 548–569 and 1733–1746):

`get_transaction_receipt` reads `tx_version` from the stored `Transaction` (e.g., `Invoke`) and then independently reads `TransactionOutput` from storage (e.g., `Declare`). The two are passed to `TransactionOutput::from((output, tx_version, msg_hash))`, which matches on the `starknet_api::transaction::TransactionOutput` variant — producing a receipt with `type: "DECLARE"` for what is stored as an Invoke transaction. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

`starknet_getTransactionReceipt` returns an authoritative-looking receipt with the wrong `type` field (e.g., `"DECLARE"` instead of `"INVOKE"`). The stored `block_body` violates the invariant that `transactions[i]` and `transaction_outputs[i]` carry the same variant. Any downstream consumer (wallet, explorer, bridge, relayer) that branches on receipt type receives incorrect data. The stored data is also permanently inconsistent with the block header's `receipt_commitment` and `transaction_commitment`, since those were computed from the real block data during header sync.

The claim that `calculate_receipt_commitment` itself is corrupted is **partially inaccurate**: the header's stored `receipt_commitment` comes from the header sync and is not recomputed from the injected transaction data. However, the RPC serving wrong receipt type is a concrete, authoritative wrong value, which meets the "High" impact threshold.

---

### Likelihood Explanation

Any node that the victim connects to for P2P sync can act as the malicious peer. No special privilege is required — the attacker only needs to be reachable as a sync peer. The attack requires crafting a valid protobuf `TransactionWithReceipt` with mismatched `txn`/`receipt` variants, which is straightforward. There is no rate-limiting, disconnection, or rejection of this data anywhere in the pipeline.

---

### Recommendation

Add a type-consistency check inside `TryFrom<protobuf::TransactionWithReceipt> for FullTransaction` after both fields are deserialized:

```rust
let type_ok = matches!(
    (&transaction, &transaction_output),
    (Transaction::Invoke(_), TransactionOutput::Invoke(_))
    | (Transaction::Declare(_), TransactionOutput::Declare(_))
    | (Transaction::Deploy(_), TransactionOutput::Deploy(_))
    | (Transaction::DeployAccount(_), TransactionOutput::DeployAccount(_))
    | (Transaction::L1Handler(_), TransactionOutput::L1Handler(_))
);
if !type_ok {
    return Err(ProtobufConversionError::WrongVariant { ... });
}
```

Additionally, validate transaction data against the header's stored `transaction_commitment` and `receipt_commitment` before committing to storage, resolving the broader TODO at line 88 of `transaction.rs`.

### Proof of Concept

```rust
#[test]
fn mismatched_tx_receipt_type_accepted() {
    use crate::protobuf;
    use starknet_api::transaction::FullTransaction;

    // Build a minimal InvokeV3 transaction body
    let invoke_txn = protobuf::TransactionInBlock {
        txn: Some(protobuf::transaction_in_block::Txn::InvokeV3(
            protobuf::InvokeV3 { /* minimal valid fields */ ..Default::default() }
        )),
        transaction_hash: Some(/* any felt */),
    };

    // Build a Declare receipt body
    let declare_receipt = protobuf::Receipt {
        r#type: Some(protobuf::receipt::Type::Declare(
            protobuf::receipt::Declare { common: Some(/* minimal common */), ..Default::default() }
        )),
    };

    let tw = protobuf::TransactionWithReceipt {
        transaction: Some(invoke_txn),
        receipt: Some(declare_receipt),
    };

    let result = FullTransaction::try_from(tw);
    assert!(result.is_ok(), "Expected Ok but got {:?}", result);

    let ft = result.unwrap();
    assert!(matches!(ft.transaction, starknet_api::transaction::Transaction::Invoke(_)));
    assert!(matches!(ft.transaction_output, starknet_api::transaction::TransactionOutput::Declare(_)));
    // Invariant violated: Invoke tx paired with Declare output
}
```
