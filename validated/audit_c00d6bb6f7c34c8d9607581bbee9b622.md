Audit Report

## Title
Stale Chainlink Price Accepted Without Validation in `ChainlinkPriceOracle` Enables Over-Minting of rsETH - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `roundId`, and `answeredInRound`, performing no staleness, round-completeness, or positivity check on the returned price. When a Chainlink feed stops updating while the real LST price has fallen, the stale inflated price is used to compute rsETH minting amounts in `LRTDepositPool.getRsETHAmountToMint()`, allowing any depositor to receive more rsETH than the true ETH value of their deposit warrants, diluting all existing rsETH holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` is the sole price source for LST assets using Chainlink feeds:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol  lines 49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();   // updatedAt, answeredInRound silently discarded
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

The `updatedAt` timestamp, `roundId`, and `answeredInRound` are all discarded. No check of the form `block.timestamp - updatedAt <= heartbeat`, `answeredInRound >= roundId`, or `price > 0` exists anywhere in the function.

This price feeds directly into rsETH minting:

```solidity
// contracts/LRTDepositPool.sol  line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

`lrtOracle.getAssetPrice(asset)` delegates to `ChainlinkPriceOracle.getAssetPrice()` (live oracle call), while `lrtOracle.rsETHPrice` is a **stored** value updated only when `updateRSETHPrice()` is called. [3](#0-2) 

**Exploit mechanics:**

1. At time T=0, stETH/ETH feed reports `1.00 ETH`. `updateRSETHPrice()` is called; `rsETHPrice` is stored reflecting `1.00 ETH` per stETH.
2. At T=12h, the real stETH price drops to `0.95 ETH`. The Chainlink feed has not yet updated (within its 24h heartbeat window), so it still returns `1.00 ETH`.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `(1000e18 * 1.00e18) / rsETHPrice` — using the stale `1.00 ETH` price. The attacker's 1000 stETH (real value: 950 ETH) is valued at 1000 ETH, yielding ~5.26% excess rsETH.
5. When the feed eventually updates and `updateRSETHPrice()` is called, `rsETHPrice` drops to reflect the real 0.95 ETH/stETH TVL. The attacker's over-minted rsETH remains outstanding; the loss is borne by all pre-existing rsETH holders.

The `pricePercentageLimit` downside-protection mechanism in `_updateRsETHPrice()` only triggers at the time `updateRSETHPrice()` is called — it provides no protection during the deposit itself. [4](#0-3) 

The same stale price also propagates into `_getTotalEthInProtocol()` via `getAssetPrice(asset)`, meaning `rsETHPrice` itself will be inflated when next updated if the feed is still stale at that point, compounding the accounting error. [5](#0-4) 

## Impact Explanation

**High — Theft of unclaimed yield.**

Every rsETH token represents a proportional claim on the protocol's total ETH TVL. When a depositor mints rsETH at an inflated stale price, they acquire a larger share of the TVL than they contributed. All pre-existing rsETH holders are diluted: their tokens now represent a smaller fraction of the same TVL. The attacker can later redeem via `LRTWithdrawalManager` for more ETH than they deposited. The yield that would have accrued to existing holders is instead captured by the attacker. In a severe depeg scenario (e.g., stETH depegs significantly while the feed is stale), the magnitude could approach Critical — direct theft of principal from existing holders.

## Likelihood Explanation

**Medium.** Chainlink feeds have defined heartbeat intervals (e.g., 24 hours for stETH/ETH on mainnet). During low-volatility periods the feed may not update for the full heartbeat window, creating a natural staleness window. No special privilege is required: any address can call `LRTDepositPool.depositAsset()`. The attack is repeatable for as long as the feed remains stale. A block-stuffing attack can additionally prevent Chainlink keepers from submitting updates before the heartbeat expires, making the stale-price window attacker-controllable.

## Recommendation

Add staleness and validity checks inside `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= stalenessThreshold[asset], "Stale price");
```

Store a per-asset `stalenessThreshold` (e.g., 25 hours for a 24-hour heartbeat feed) in a mapping alongside `assetPriceFeed`, settable by `onlyLRTManager`. This ensures that if a feed stops updating, deposits are blocked rather than processed at a stale price.

## Proof of Concept

**Foundry fork test outline:**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed is within its heartbeat window.
2. Deploy or reference the existing `ChainlinkPriceOracle` pointed at the stETH/ETH feed.
3. Warp `block.timestamp` forward by 13 hours (within the 24h heartbeat, feed not yet stale by Chainlink's own definition, but real price has moved).
4. Mock the Chainlink aggregator to return the old `price` with an `updatedAt` 13 hours in the past (simulating a feed that has not updated).
5. Record `rsETHPrice` (stored from last `updateRSETHPrice()` call at the old price).
6. Call `depositAsset(stETH, 1000e18, 0, "")` as an unprivileged attacker.
7. Assert that `rsethAmountToMint` is greater than `(1000e18 * 0.95e18) / rsETHPrice` — i.e., the attacker received more rsETH than the real price warrants.
8. Call `updateRSETHPrice()` after the feed updates to the real price.
9. Assert that `rsETHPrice` has decreased, and that the attacker's rsETH balance now represents more ETH value than they deposited, at the expense of pre-existing holders.
