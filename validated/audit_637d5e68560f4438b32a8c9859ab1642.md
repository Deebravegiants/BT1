### Title
Missing `removed` Flag Validation in EVM Log Extractor Enables Forged Foreign-Chain Verification — (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

The `EvmExtractor::Log` branch in the EVM inspector extracts logs from `eth_getTransactionReceipt` responses by matching `log_index`, but never checks the `removed` field that signals a log was invalidated by a chain reorganization. A Byzantine participant (below signing threshold) who registers a malicious RPC provider can return receipts with `removed: true` logs while still passing the canonical-block check, causing all MPC nodes to sign a `ForeignTxSignPayload` that attests to a foreign-chain event that was actually reverted.

---

### Finding Description

The `EvmExtractor::extract_value` function selects a log solely by its block-level `log_index`:

```rust
// crates/foreign-chain-inspector/src/evm/inspector.rs  L183-L193
EvmExtractor::Log { log_index } => {
    let target_index = ethereum_types::U64::from(*log_index);
    let log = rpc_response
        .logs
        .iter()
        .find(|log| log.log_index == target_index)
        .cloned()
        .ok_or(ForeignChainInspectionError::LogIndexOutOfBounds)?;

    Ok(EvmExtractedValue::Log(log))
}
```

The `Log` struct (defined in `crates/foreign-chain-rpc-interfaces/src/evm.rs`) carries a `removed: bool` field that the Ethereum JSON-RPC specification uses to mark logs invalidated by a reorganization:

```rust
// crates/foreign-chain-rpc-interfaces/src/evm.rs  L88-L100
pub struct Log {
    pub removed: bool,   // <-- never checked by the extractor
    pub log_index: U64,
    ...
}
```

The upstream `extract` method does call `verify_block_is_canonical`, which cross-checks the receipt's `block_hash` against `eth_getBlockByNumber`. However, a malicious provider can satisfy that check while still returning logs with `removed: true` — it only needs to return a consistent `block_hash` value across both RPC calls. The canonical-block check does not inspect individual log fields.

The signed payload is `SHA-256(borsh(ForeignTxSignPayload { request, values }))`. Because `values` is derived directly from the unchecked log, the MPC network's threshold signature attests to a log that was never finalized on the canonical chain.

---

### Impact Explanation

The `ForeignTxSignPayload` signed by the MPC network is consumed by bridge contracts (e.g., Omnibridge inbound flow) to authorize fund releases or state transitions on NEAR. If the signed payload contains a removed log — representing a deposit or transfer that was reverted on the foreign chain — the bridge contract has no way to distinguish it from a legitimate attestation. The result is an invalid bridge execution: funds could be released on NEAR for a foreign-chain transaction that never finalized, enabling a double-spend or theft of bridged assets.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

Provider registration is gated behind `register_foreign_chain_config`, which requires the caller to be a participant (voter). A single Byzantine participant — strictly below the signing threshold — can register a malicious RPC endpoint for any supported EVM chain. Provider selection is deterministic (nodes do not specify an RPC URL; they derive it from the on-chain configuration), so all MPC nodes will query the same provider for a given request. A single malicious provider therefore controls the RPC responses seen by every node simultaneously, making threshold-level collusion unnecessary. The attacker does not need to compromise any signing key or any other participant.

---

### Recommendation

Add a `!log.removed` guard in the `EvmExtractor::Log` branch before accepting the log:

```rust
EvmExtractor::Log { log_index } => {
    let target_index = ethereum_types::U64::from(*log_index);
    let log = rpc_response
        .logs
        .iter()
        .find(|log| log.log_index == target_index && !log.removed)
        .cloned()
        .ok_or(ForeignChainInspectionError::LogIndexOutOfBounds)?;
    Ok(EvmExtractedValue::Log(log))
}
```

This is directly analogous to the ERC20 recommendation in the source report: just as `safeTransfer` ensures the transfer actually succeeded before updating accounting, checking `!log.removed` ensures the extracted log actually exists on the canonical chain before it is committed to the signed payload.

---

### Proof of Concept

1. Byzantine participant P (one of N, below threshold T) calls `register_foreign_chain_config` on the MPC contract, registering a malicious HTTPS endpoint as the sole RPC provider for an EVM chain (e.g., Ethereum).
2. Victim user U calls `verify_foreign_transaction` with an `EvmRpcRequest` containing `EvmExtractor::Log { log_index: K }` for a real transaction `tx_id` that previously emitted a high-value bridge deposit event at log index K, but was later reorganized out.
3. All MPC nodes deterministically select P's malicious endpoint. P returns:
   - `eth_getTransactionReceipt(tx_id)` → receipt with `block_hash = H`, `block_number = N`, and the reorged log at index K with **`removed: true`**.
   - `eth_getBlockByNumber(N)` (finality check) → block with `number ≥ N` (passes finality).
   - `eth_getBlockByNumber(N)` (canonical check) → block with `hash = H, number = N` (passes canonical check).
4. Each node's `extract` call returns `EvmExtractedValue::Log(log)` where `log.removed == true`, without error.
5. All nodes independently compute the same `ForeignTxSignPayload { request, values: [Log{removed:true, ...}] }` and cooperate to produce a threshold signature over its SHA-256 hash.
6. The node submits `respond_verify_foreign_tx` with the signed `payload_hash`.
7. The MPC contract resolves the yield and returns `VerifyForeignTransactionResponse { payload_hash, signature }` to U.
