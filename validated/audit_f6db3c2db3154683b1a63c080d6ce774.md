Audit Report

## Title
Missing Time-Based Staleness Check in `ChainlinkOracleForRSETHPoolCollateral.getRate()` Enables Excess rsETH Minting and Daily Limit Exhaustion — (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

## Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` validates only round completeness (`answeredInRound < roundID`) but performs no time-based freshness check. A Chainlink price that was answered in its own round but last updated many hours ago passes all three guards and is returned as current. Any depositor can exploit a stale, inflated collateral price to receive excess wrsETH proportional to the price inflation, while simultaneously exhausting the daily mint cap.

## Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` fetches the Chainlink price and applies three guards:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0)            revert IncompleteRound();
if (ethPrice <= 0)             revert InvalidPrice();
``` [1](#0-0) 

There is no check of the form `block.timestamp - timestamp > MAX_DELAY`. A price where `answeredInRound == roundID` (self-answered round) but `timestamp` is arbitrarily old passes all three guards.

This oracle is registered as the collateral-token oracle via `supportedTokenOracle[token]`. When a token deposit is processed, `viewSwapRsETHAmountAndFee(amount, token)` calls `IOracle(supportedTokenOracle[token]).getRate()` to obtain the (potentially stale) collateral price: [2](#0-1) 

The `limitDailyMint` modifier computes `rsETHAmount` using this same stale oracle and charges it against the daily cap: [3](#0-2) 

The deposit body then calls `viewSwapRsETHAmountAndFee` a second time and mints the same inflated `rsETHAmount` to the caller: [4](#0-3) 

Because both the modifier and the body read the same stale oracle in the same block, the inflated amount is both minted to the depositor and charged against the daily limit. The identical pattern is present in `RSETHPoolV3ExternalBridge.deposit()` and `RSETHPoolV3WithNativeChainBridge.deposit()`: [5](#0-4) [6](#0-5) 

## Impact Explanation

**Critical — Direct theft of user funds.** The attacker deposits real collateral tokens (e.g., wstETH) and receives wrsETH in excess of the true collateral value. wrsETH is redeemable 1:1 for rsETH on L1, which is backed by the protocol's L1 collateral pool. The attacker bridges the excess wrsETH to L1 and redeems it for more collateral than deposited, extracting value from other protocol participants.

**Secondary — Medium: Temporary freezing of funds.** The inflated `rsETHAmount` is charged against `dailyMintAmount`. A single deposit with a 2× stale price exhausts the daily cap at half the intended deposit volume, blocking all other users from depositing for the remainder of the day.

## Likelihood Explanation

Chainlink price feeds can become stale due to network congestion, L2 sequencer downtime, or feed-specific heartbeat windows. The `answeredInRound == roundID` condition is satisfied for any price answered in its own round, regardless of age. No attacker action is required to cause staleness; the attacker only needs to observe the oracle state and submit a deposit while the stale high price is active. This is a passive, permissionless exploit path requiring no privileges, no admin collusion, and no oracle compromise.

## Recommendation

Add a configurable maximum age check in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
uint256 public constant MAX_PRICE_AGE = 3600; // tune per feed heartbeat

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID)                    revert StalePrice();
    if (timestamp == 0)                               revert IncompleteRound();
    if (ethPrice <= 0)                                revert InvalidPrice();
    if (block.timestamp - timestamp > MAX_PRICE_AGE)  revert StalePrice(); // ADD THIS

    uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
    return normalizedPrice;
}
```

`MAX_PRICE_AGE` should be set per-feed based on the Chainlink heartbeat for that price pair (e.g., 3600 s for a 1-hour heartbeat feed). Consider making it a constructor parameter to support multiple feeds with different heartbeats.

## Proof of Concept

```
Setup:
  dailyMintLimit       = 1000e18 rsETH
  true wstETH/ETH rate = 1.1e18
  stale oracle rate    = 2.2e18  (answeredInRound == roundID, timestamp = 12 hours ago)
  rsETH/ETH rate       = 1.05e18 (current)
  feeBps               = 0

Attacker deposits 450e18 wstETH:
  tokenToETHRate = 2.2e18  (stale, passes all three guards)
  rsETHToETHrate = 1.05e18
  rsETHAmount    = 450e18 * 2.2e18 / 1.05e18 ≈ 942e18

limitDailyMint: 0 + 942e18 ≤ 1000e18 → passes; dailyMintAmount = 942e18
deposit body:   mints 942e18 wrsETH to attacker

True rsETH owed = 450e18 * 1.1e18 / 1.05e18 ≈ 471e18
Attacker profit ≈ 471e18 rsETH stolen from protocol collateral pool
Remaining daily cap: 58e18 rsETH — all other users effectively locked out

Foundry fork test outline:
1. Fork the target L2 at a block where the Chainlink wstETH/ETH feed is within heartbeat.
2. vm.warp() forward past the feed's heartbeat without triggering a new round.
3. Confirm answeredInRound == roundID still holds (round not invalidated).
4. Call RSETHPoolV3.deposit(wstETH, 450e18, "") as an unprivileged address.
5. Assert wrsETH.balanceOf(attacker) ≈ 942e18 (inflated).
6. Assert dailyMintAmount ≈ 942e18 (cap nearly exhausted).
7. Assert a second depositor's deposit() reverts with DailyMintLimitExceeded.
```
