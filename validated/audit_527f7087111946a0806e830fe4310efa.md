### Title
Peer-Injected Fake `FinalExecutionOutcomeView` Returned to RPC Clients via Unverified `TxStatusResponse` Cache — (`chain/client/src/view_client_actor.rs`)

---

### Summary

`Handler<TxStatusResponse>::handle` accepts a `FinalExecutionOutcomeView` from any peer and stores it in the `tx_status_response` LRU cache with no verification against the local chain store. `get_tx_status` checks this cache **before** it checks the local chain, so a peer-injected fake outcome is returned directly to the RPC caller.

---

### Finding Description

**`Handler<TxStatusResponse>::handle`** (lines 1713–1731) accepts any `TxStatusResponse` from any peer. The only guard is whether the tx hash was previously placed in `tx_status_requests`:

```rust
if request_manager.tx_status_requests.pop(&tx_hash).is_some() {
    request_manager.tx_status_response.put(tx_hash, *tx_result);
}
```

No field of `tx_result` — not `status`, `gas_burnt`, `tokens_burnt`, `receipt_ids`, nor the transaction body — is verified against `DBCol::TransactionResult`, `DBCol::OutcomeIds`, or any other chain-store column. [1](#0-0) 

**`get_tx_status`** checks the cache as its **first** action, before the shard-tracking branch that would consult the local chain store:

```rust
if let Some(res) = request_manager.tx_status_response.pop(&tx_hash) {
    request_manager.tx_status_requests.pop(&tx_hash);
    let status = self.get_tx_execution_status(&res)?;
    return Ok(TxStatusView { execution_outcome: Some(...(res)), status });
}
``` [2](#0-1) 

The only downstream check, `get_tx_execution_status`, validates that the `block_hash` fields embedded in the outcome exist in the local chain and are canonical/final. It does **not** verify that the outcome's `status`, `gas_burnt`, `tokens_burnt`, or `receipt_ids` match anything stored in the chain: [3](#0-2) 

An attacker who supplies a valid, canonical `block_hash` (trivially obtained from any public RPC call) passes this check while keeping all other fields fabricated.

The `tx_status_requests` cache is populated whenever the node does not track the target shard and issues a `NetworkRequests::TxStatus`: [4](#0-3) 

Any peer — not only the specific validator that was queried — can send a `TxStatusResponse` for that hash and have it accepted.

---

### Impact Explanation

An RPC client querying `tx_status` on a node that does not track the relevant shard receives a `FinalExecutionOutcomeView` whose `status` (e.g., `SuccessValue`), `gas_burnt`, `tokens_burnt`, and `receipt_ids` are entirely attacker-controlled. The node presents this as authoritative chain data. Downstream consumers (wallets, bridges, relayers) that trust the RPC response will act on fabricated execution results.

---

### Likelihood Explanation

- In a sharded NEAR network, most nodes do not track all shards, making the non-tracking branch the common case for cross-shard transactions.
- Any node can connect as a peer; no validator or operator privilege is required.
- The tx hash is public the moment a transaction is broadcast.
- The attacker only needs to race the legitimate `TxStatusResponse` from the real chunk producer, or simply be the first peer to respond.

---

### Recommendation

In `Handler<TxStatusResponse>::handle`, before caching the peer-supplied outcome, verify it against the local chain store: confirm the transaction hash exists in `DBCol::Transactions`, that the outcome id matches `DBCol::OutcomeIds` for the claimed block, and that `gas_burnt`/`tokens_burnt` match the stored `ExecutionOutcomeWithProof`. If the node does not have the data locally (because it does not track the shard), the response should be forwarded to the RPC waiter without being cached as authoritative, or the cache entry should be marked as unverified and re-checked once the node can confirm it.

---

### Proof of Concept

1. Node A does not track shard S. RPC client calls `tx_status(tx_hash, signer)` for a transaction on shard S.
2. Node A places `tx_hash` in `tx_status_requests` and sends `NetworkRequests::TxStatus` to the chunk producer.
3. Attacker peer B (connected to A) sends a `TxStatusResponse` containing a `FinalExecutionOutcomeView` with `status: SuccessValue(b"fake")`, `gas_burnt: 0`, `tokens_burnt: 0`, `receipt_ids: []`, and `block_hash` set to any canonical block hash obtained from A's public RPC.
4. `Handler<TxStatusResponse>::handle` on A finds `tx_hash` in `tx_status_requests`, pops it, and stores the fake outcome in `tx_status_response`.
5. The next `get_tx_status` call on A hits the cache at line 645, calls `get_tx_execution_status` (which passes because the block hash is valid), and returns the fabricated `FinalExecutionOutcomeView` to the RPC client.
