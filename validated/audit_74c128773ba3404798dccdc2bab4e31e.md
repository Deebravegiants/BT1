Audit Report

## Title
Unbounded `merkleProof` Array Enables Arbitrary Gas Consumption via Linear `keccak256` Iteration — (`contracts/KERNEL/KernelMerkleDistributor.sol`)

## Summary
`KernelMerkleDistributor._processClaim()` imposes no upper bound on the caller-supplied `merkleProof` array before passing it to `MerkleProofUpgradeable.verify()`. Because `processProof` iterates unconditionally over every element, any unprivileged caller who passes the cheap O(1) pre-checks can submit a proof of arbitrary length L and force the EVM to execute L `keccak256` operations before the transaction reverts with `InvalidMerkleProof`, consuming gas proportional to L with no state change.

## Finding Description
In `_processClaim` (lines 292–346 of `KernelMerkleDistributor.sol`), the execution path is:

1. O(1) guards: non-zero address check, non-zero root check, `1 ≤ index ≤ currentIndex`, `account == msg.sender`, `!isClaimed(index, account)` — all passable by any fresh address with a valid index.
2. `bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));`
3. `MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)` — delegates to `processProof`, which runs `for (uint256 i = 0; i < proof.length; i++) { computedHash = _hashPair(computedHash, proof[i]); }` over all L elements unconditionally.
4. Returns `false` → `revert InvalidMerkleProof()`.

There is no `require(merkleProof.length <= MAX_PROOF_LENGTH)` or equivalent guard anywhere in the call chain. [1](#0-0) [2](#0-1) 

## Impact Explanation
Gas cost scales linearly with L. Calldata alone for L = 5,000 elements costs approximately 5,000 × 32 × 16 ≈ 2.56 M gas; computation adds ~200 K gas. A single transaction with L ≈ 9,000 can approach Ethereum's ~30 M block gas limit. An attacker can repeatedly submit such transactions to stuff blocks and delay legitimate claimants. This concretely matches **Medium. Unbounded gas consumption** and **Low. Block stuffing** from the allowed impact scope.

## Likelihood Explanation
Preconditions are minimal: the attacker needs any `index` in the range `[1, currentIndex]` (readable from the public state variable) and must not have previously claimed under that index with their own address. Because `account == msg.sender` is enforced, the attacker uses their own address, which trivially satisfies `!isClaimed(index, account)` for any fresh address. No privileged role, no front-running, and no external dependency is required. The attack is repeatable with different fresh addresses. [3](#0-2) 

## Recommendation
Add a proof-length cap before the `verify` call. A Merkle tree over N leaves requires at most `ceil(log2(N))` proof elements. For any realistic distribution (e.g., ≤ 2²⁰ ≈ 1 M recipients), 20 elements is sufficient:

```solidity
uint256 public constant MAX_PROOF_LENGTH = 20;

// inside _processClaim, before the verify call:
if (merkleProof.length > MAX_PROOF_LENGTH) revert InvalidMerkleProof();
```

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "../contracts/KERNEL/KernelMerkleDistributor.sol";

contract UnboundedProofGasTest is Test {
    KernelMerkleDistributor distributor;

    function setUp() public {
        // Deploy and initialize distributor with a non-zero merkle root,
        // currentIndex = 1, attacker address registered as index 1 (unclaimed).
    }

    function test_unboundedGasOnInvalidProof() public {
        uint256 L = 5_000;
        bytes32[] memory bigProof = new bytes32[](L);
        for (uint256 i = 0; i < L; i++) {
            bigProof[i] = bytes32(uint256(i + 1));
        }

        uint256 gasBefore = gasleft();
        vm.expectRevert(IMerkleDistributor.InvalidMerkleProof.selector);
        distributor.claim(1, address(this), 1 ether, bigProof);
        uint256 gasUsed = gasBefore - gasleft();

        assertGt(gasUsed, 100_000, "Expected significant gas consumption");
    }
}
```

Running with increasing values of L (1, 10, 100, 1,000, 5,000) and plotting `gasUsed` vs L confirms linear growth, demonstrating that a single transaction with L ≈ 9,000 can approach the 30 M block gas limit.

    function _processClaim(
        uint256 index,
        address account,
        uint256 cumulativeAmount,
        bytes32[] calldata merkleProof
    )
        internal
        returns (uint256)
    {
        UtilLib.checkNonZeroAddress(account);

        if (currentMerkleRoot == bytes32(0)) {
            revert ZeroValueProvided();
        }

        if (index == 0 || index > currentIndex) {
            revert InvalidIndex();
        }

        if (account != msg.sender) {
            revert Unauthorized();
        }

        if (isClaimed(index, account)) {
            revert AlreadyClaimed();
        }

        // Verify the merkle proof
        bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
        if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
            revert InvalidMerkleProof();
        }
