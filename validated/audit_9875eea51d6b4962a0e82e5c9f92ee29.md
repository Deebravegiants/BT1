Audit Report

## Title
Silent Zero-Return in `_withdrawFromAave` When `totalETHDepositedToAave == 0` Blocks User Withdrawals and Permanently Diverts Accrued Yield to Treasury — (`contracts/LRTWithdrawalManager.sol`)

## Summary

When all principal has been withdrawn from Aave and `totalETHDepositedToAave` reaches `0`, but Aave interest has caused `aaveAWETH.balanceOf(address(this)) > 0`, `_withdrawFromAave` silently returns `0` without reverting. Any subsequent `completeWithdrawal` call that requires ETH from Aave then reverts with `InsufficientLiquidityForWithdrawal`, blocking affected users. The stranded interest can only be recovered via `collectInterestToTreasury()` — permanently routing it to the treasury rather than the withdrawal pool.

## Finding Description

The root cause is in `_withdrawFromAave` (L905–921). When `totalETHDepositedToAave == 0` but `aaveBalance > 0`:

- L909: `if (aaveBalance == 0) revert InsufficientAaveBalance()` does **not** fire because `aaveBalance > 0`.
- L912: `withdrawablePrincipal = min(aaveBalance, totalETHDepositedToAave) = min(aaveBalance, 0) = 0`.
- L914: `withdrawnAmount = min(amount, 0) = 0`.
- L915: `if (withdrawnAmount == 0) return 0` — silent return, no ETH moved, no revert, no event.

The caller at L724–730 then checks `balanceAfter < request.expectedAssetAmount` and always reverts with `InsufficientLiquidityForWithdrawal`.

The guard at L909 is insufficient because it only catches the case where `aaveBalance == 0`, not the case where `aaveBalance > 0` but `totalETHDepositedToAave == 0`.

The code comment at L911 confirms this is intentional design: *"Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)"* — but the design creates no fallback path when only interest remains and user withdrawals are pending.

`_collectInterestToTreasury` (L950–952) correctly handles this state: when `principal == 0`, `aaveBalance > principal` passes, and the full balance is withdrawn and sent to treasury. This confirms the interest is not permanently frozen in Aave — it is permanently diverted away from the user withdrawal pool and into the treasury.

**State progression:**
1. ETH deposited to Aave: `totalETHDepositedToAave = P`, `aaveBalance = P`.
2. Interest accrues: `aaveBalance = P + I`.
3. Users complete withdrawals, each decrementing `totalETHDepositedToAave` by the withdrawn amount.
4. After sufficient completions: `totalETHDepositedToAave = 0`, `aaveBalance = I > 0`.
5. Any subsequent `completeWithdrawal` needing ETH from Aave silently gets `0` and reverts.
6. `collectInterestToTreasury()` recovers `I` — to treasury, not to the withdrawal pool.

## Impact Explanation

**Medium. Permanent freezing of unclaimed yield.**

The accrued Aave interest (yield generated on user-deposited ETH) is permanently inaccessible for user withdrawals once `totalETHDepositedToAave == 0`. It can only be routed to treasury via `collectInterestToTreasury()`. Users with unlocked, pending withdrawal requests are blocked from completing them until new principal ETH enters the contract through other means. The yield itself is permanently diverted from the withdrawal pool.

## Likelihood Explanation

No privileged role, oracle manipulation, or attacker action is required. Interest accrues automatically in Aave over time. Users calling `completeWithdrawal()` is the normal, intended flow. The longer ETH sits in Aave and the more interest accrues relative to principal, the more likely the final principal withdrawal leaves a non-zero `aaveBalance`. This is reachable by any unprivileged user through the standard withdrawal flow.

## Recommendation

In `_withdrawFromAave`, add an explicit check for the case where `totalETHDepositedToAave == 0` but `aaveBalance > 0`, and either:

1. Revert with a descriptive error (e.g., `OnlyInterestRemains`) so callers can handle it explicitly, or
2. Allow withdrawal of interest up to `aaveBalance` when `totalETHDepositedToAave == 0`, updating accounting accordingly (treating it as a zero-principal withdrawal).

Additionally, `completeWithdrawal` should detect this state and either call `_collectInterestToTreasury` first to clear the Aave balance (allowing the contract to receive new ETH deposits), or route interest to fund user withdrawals before falling back to `InsufficientLiquidityForWithdrawal`.
