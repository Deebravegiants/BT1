### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that a submitted signature is valid for `response.payload_hash`, but never verifies that `response.payload_hash` is the correct hash derived from the accompanying `request`. Any attested participant can replay a valid `{payload_hash, signature}` pair from a prior on-chain response to resolve a completely different pending foreign-tx yield, delivering a forged verification result to the waiting caller.

---

### Finding Description

The money-flow analog from the external report is: a required prior step is missing before a downstream operation that depends on it. Here, the required step is binding the response's `payload_hash` to the pending `request`; without it, the signature check passes on an unrelated hash.

**Root cause — `respond_verify_foreign_tx`:**

```
crates/contract/src/lib.rs  lines 718–734
```

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← taken verbatim from caller

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← verified only against itself
    &secp_pk,
)
.is_ok()
```

The contract checks `sig(payload_hash)` is valid under the root key, then immediately resolves all pending yields for `request`:

```
crates/contract/src/lib.rs  lines 749–753
```

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),   // ← response with unverified payload_hash
)
```

There is no step that computes the expected hash from `request` and compares it to `response.payload_hash`. The correct binding logic exists only in the off-chain SDK helper:

```
crates/near-mpc-sdk/src/foreign_chain.rs  lines 48–64
```

```rust
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: self.request,
    values: self.expected_extracted_values,
});
let expected_payload_hash = expected_payload.compute_msg_hash()?;
let payload_is_correct = expected_payload_hash == response.payload_hash;  // ← missing in contract
```

The contract never performs this check.

**`VerifyForeignTransactionRequest` contains no `payload_hash` field** — it holds only `{request, domain_id, payload_version}`:

```
crates/near-mpc-contract-interface/src/types/foreign_chain.rs  lines 124–128
```

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

So the contract has no stored expected hash to compare against; the caller-supplied `response.payload_hash` is accepted unconditionally as long as the signature over it is valid.

---

### Impact Explanation

**High — forged foreign-chain verification / cross-request signature replay.**

A Byzantine attested participant executes the following replay:

1. Observe any prior on-chain `respond_verify_foreign_tx(R′, {H′, sig(H′)})` call (all NEAR transactions are public).
2. Wait for a new pending `verify_foreign_transaction(R)` yield to appear.
3. Call `respond_verify_foreign_tx(R, {H′, sig(H′)})` — same `{payload_hash, signature}` pair, different `request`.
4. The contract passes the attestation check, verifies `sig(H′)` under the root key (valid), finds the pending yield for `R`, and resolves it with `{payload_hash: H′, signature: sig(H′)}`.

The original caller of `verify_foreign_transaction(R)` receives a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to their request `R`. Any downstream bridge contract that trusts the on-chain response without independently recomputing the expected hash (a reasonable assumption given the contract is the authoritative verifier) will accept a forged attestation — enabling invalid bridge execution or double-spend conditions.

---

### Likelihood Explanation

**High.**

- The attacker only needs to be an attested participant — a realistic role for a Byzantine node below the signing threshold.
- No new cryptographic material is required: valid `{payload_hash, signature}` pairs are permanently visible in NEAR transaction history from the moment the first `respond_verify_foreign_tx` call is made.
- The attack is fully deterministic and requires no timing luck beyond observing a pending yield.
- The `respond_verify_foreign_tx` function imposes no nonce, no per-request signature commitment, and no check that the signer was the one who computed the payload.

---

### Recommendation

The contract must verify that `response.payload_hash` is the canonical hash of the pending `request` before accepting the response. Because the contract does not store `extracted_values`, the recommended fix is to require the responding node to supply the `extracted_values` alongside the response, then recompute and compare the hash on-chain:

```rust
// In respond_verify_foreign_tx, before signature verification:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),   // new field in response
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, bind the expected hash into the `VerifyForeignTransactionRequest` at submission time (computed from the request parameters alone, without extracted values), and verify the response hash falls within the committed scope.

### Proof of Concept

**Setup:** Contract is Running, domain is `ForeignTx`/Secp256k1. Two Bitcoin requests are submitted:

```
Alice → verify_foreign_transaction(R_alice)   // pending yield Y_alice
Bob   → verify_foreign_transaction(R_bob)     // pending yield Y_bob
```

**Step 1 — Legitimate response for Bob's request:**

```
honest_node → respond_verify_foreign_tx(R_bob, {H_bob, sig(H_bob)})
```

`H_bob = SHA-256(borsh(ForeignTxSignPayload{R_bob, extracted_values_bob}))`. Contract accepts; `Y_bob` resolved correctly. This call is now visible on-chain.

**Step 2 — Byzantine replay against Alice's pending yield:**

```
byzantine_node → respond_verify_foreign_tx(R_alice, {H_bob, sig(H_bob)})
```

The contract at lines 718–734:
- Reads `payload_hash = H_bob` from the response.
- Verifies `sig(H_bob)` under the root key → **valid** (reused from Step 1).
- Performs no check that `H_bob` corresponds to `R_alice`.
- Resolves `Y_alice` with `{payload_hash: H_bob, signature: sig(H_bob)}`.

Alice's caller receives a `VerifyForeignTransactionResponse` asserting that `H_bob` was verified — a hash derived from Bob's transaction, not Alice's. Any bridge contract consuming this response without re-deriving the expected hash from `R_alice` will process a forged attestation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
