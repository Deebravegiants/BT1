Audit Report

## Title
L2 Price Providers Lack Sequencer Uptime Check, Enabling Stale-Price Swaps After Restart - (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are explicitly designed for L2 deployment (both carry `FUTURE_TOLERANCE` to handle sequencer clock skew) but neither checks a Chainlink sequencer uptime feed before quoting bid/ask prices. When the L2 sequencer goes offline, oracle prices freeze on-chain. If the stored oracle `refTime` is younger than `MAX_TIME_DELTA` at the moment of sequencer restart, the staleness check passes and an attacker can immediately execute a swap against the stale pre-outage price, causing direct LP loss.

## Finding Description

Both L2 providers implement `_isStale` that compares `refTime` against `block.timestamp` with a configurable `MAX_TIME_DELTA` (bounded only at `> 0` and `<= 7 days`):

```solidity
// PriceProviderL2.sol L135-L150
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool)
{
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;
}
```

`_getBidAndAskPrice()` in `PriceProviderL2` performs no sequencer liveness check before accepting the oracle price — it reads `refTime` from the on-chain oracle and passes it directly to `_isStale`:

```solidity
// PriceProviderL2.sol L208-L217
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spread, , uint256 refTime) =
        IPricedOracle(address(offchainOracle)).price(offchainFeedId, msg.sender);
    if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
        return (0, type(uint128).max);
    }
    ...
}
```

`ProtectedPriceProviderL2._computeBidAsk` has the identical gap — staleness and price guards but no sequencer uptime gate.

By contrast, `ChainlinkVerifierL2` (confirmed in registry ABI) carries both `sequencerUptimeFeed` and `GRACE_PERIOD`, and the Chainlink-hybrid L2 provider constructor explicitly accepts `_sequencerUptimeFeed` (registry.json L631-636, L2002-2005). `PriceProviderL2` and `ProtectedPriceProviderL2` accept no such parameter and perform no such check.

**Exploit path:**
1. `MAX_TIME_DELTA` is configured to 1 day (valid per constructor bounds).
2. L2 sequencer goes offline. The on-chain oracle's stored price has `refTime = T`.
3. Market price moves significantly during the outage (e.g., 4 hours).
4. Sequencer restarts. `block.timestamp` is now `T + 4h`. `_isStale(T, T+4h, 1 day, ...)` → `false` (4h < 1 day).
5. Attacker calls `pool.swap()` before any oracle keeper pushes a fresh price.
6. Pool quotes the stale pre-outage bid/ask. Attacker buys the underpriced token and immediately sells on a DEX at the true market price.
7. LPs absorb the loss; pool balances no longer cover fair-value LP claims.

## Impact Explanation

Direct LP principal loss via bad-price execution. The pool quotes a stale bid/ask that does not reflect the true market price after the sequencer outage. The attacker extracts the price difference (proportional to market movement during the outage) from LP reserves. This satisfies the "bad-price execution" and "pool insolvency" impact gates: pool token balances no longer cover fair-value LP claims at current market prices. Severity is High given the magnitude of potential price movement during documented multi-hour outages.

## Likelihood Explanation

- `PriceProviderFactoryL2` is deployed on Arbitrum, Base, Avalanche, BSC, Berachain, Optimism, Polygon, and others (registry.json L5800-5836), all of which have sequencers.
- L2 sequencer outages are documented historical events (Arbitrum ~7-hour outage in 2022; Base has had brief outages).
- The attack requires no special privilege — any EOA can call `swap()` immediately after sequencer restart.
- The vulnerability window is bounded only by how quickly oracle keepers push a fresh price post-restart, which can be minutes.
- `MAX_TIME_DELTA` up to 7 days means even long outages leave the price appearing fresh.

## Recommendation

Add a Chainlink sequencer uptime check with a grace period to both `PriceProviderL2._getBidAndAskPrice()` and `ProtectedPriceProviderL2._getBidAndAskPrice()`, mirroring the pattern already present in `ChainlinkVerifierL2`:

```solidity
// Add immutable:
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

// Add to _getBidAndAskPrice() before the staleness check:
if (address(sequencerUptimeFeed) != address(0)) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer != 0 || block.timestamp - startedAt < GRACE_PERIOD) {
        return (0, type(uint128).max); // fail closed
    }
}
```

## Proof of Concept

```
Setup:
  - Deploy PriceProviderL2 on Arbitrum fork with MAX_TIME_DELTA = 1 days
  - Oracle last updated at T=0 with price = 100 (refTime = T=0)

Attack:
  T=0:    Sequencer goes offline. Oracle price frozen at 100.
  T=4h:   True market price moves to 110 (10% up).
  T=4h:   Sequencer restarts.
  T=4h+ε: Attacker calls pool.swap(zeroForOne=true, ...) before any oracle update.
           PriceProviderL2._isStale(T=0, T=4h, 1 day, FUTURE_TOLERANCE) → false (4h < 1 day).
           Pool quotes bid/ask around 100 (stale).
           Attacker buys token1 at price 100 when market is 110.
           Attacker immediately sells on a DEX at 110 → 10% profit.
           LPs receive token0 worth 100 but gave up token1 worth 110.

Result: LP loss of ~10% on the swapped amount. Pool balances no longer
        cover fair-value LP claims at current market prices.

Foundry fork test:
  1. Fork Arbitrum at a block just after a sequencer restart.
  2. Deploy PriceProviderL2 with MAX_TIME_DELTA = 1 days.
  3. Set oracle refTime to (block.timestamp - 4 hours).
  4. Call getBidAndAskPrice() — assert it returns valid (non-stale) bid/ask.
  5. Confirm no sequencer uptime check blocks the call.
```
