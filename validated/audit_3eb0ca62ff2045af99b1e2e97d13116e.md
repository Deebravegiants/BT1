### Title
Unchecked `RAND_bytes` Return Value Produces Uninitialized EdDSA Nonce, Enabling Private-Key Recovery — (`File: src/cbmpc/crypto/ec25519_core.cpp`)

### Summary

`ED25519_sign_with_scalar` in `ec25519_core.cpp` calls `RAND_bytes(nonce, 64)` without checking its return value. If `RAND_bytes` returns 0 (failure), the 64-byte `nonce` buffer holds uninitialized stack data. That buffer is passed directly into `sign_with_nonce`, which uses it as the EdDSA signing nonce. A predictable or zero nonce in EdDSA allows full private-key recovery from a single signature. The same function also ignores all return values from `EVP_DigestInit`, `EVP_DigestUpdate`, and `EVP_DigestFinal` inside `hash_hram`, which can produce a garbage transcript hash under the same failure conditions.

### Finding Description

`ED25519_sign_with_scalar` is the low-level EdDSA signing primitive used by the MPC EdDSA protocol stack:

```
coinbase::api (public API)
  → mpc::eddsa2pc::sign / mpc::eddsampc::sign   (eddsa.cpp)
  → schnorr2p::sign / schnorrmp::sign  (variant_e::EdDSA)
  → base_eddsa.cpp  (calls ED25519_sign_with_scalar)
  → ec25519_core.cpp  ED25519_sign_with_scalar
```

extern "C" int ED25519_sign_with_scalar(...) {
  uint8_t nonce[64];

``` [1](#0-0) 

`RAND_bytes` returns 1 on success, 0 on failure, and −1 when the PRNG is not supported. On failure the 64-byte stack buffer is never written and retains whatever bytes happened to be on the stack. The library's own `gen_random` wrapper (used everywhere else) correctly asserts the return value:

```c
void gen_random(byte_ptr output, int size) {
  int res = RAND_bytes(output, size);
  cb_assert(res > 0);             // ← checked here
}
``` [2](#0-1) 

`ED25519_sign_with_scalar` bypasses this wrapper and calls `RAND_bytes` directly, omitting the check.

The secondary issue is in `hash_hram` (lines 956–967), where `EVP_MD_CTX_new()`, `EVP_DigestInit`, `EVP_DigestUpdate`, and `EVP_DigestFinal` are all called without checking return values. If `EVP_MD_CTX_new()` returns NULL (OOM), the subsequent EVP calls silently fail and `hram[64]` remains uninitialized stack data, producing a garbage transcript hash `hram_bn`. [3](#0-2) 

### Impact Explanation

In EdDSA the signature is:

```
R  = nonce · G
s  = nonce + H(R ‖ pk ‖ msg) · key   (mod q)
```

If `nonce = 0` (e.g., stack is zeroed by the OS after a fresh allocation), then `R = identity` and `s = H(identity ‖ pk ‖ msg) · key`. The private scalar is immediately recoverable as `key = s · H⁻¹ (mod q)`. Even a non-zero but predictable nonce (e.g., repeated stack pattern) allows recovery from two signatures via the standard lattice or linear-algebra attack. This directly satisfies the **Critical** impact criterion: a shipped API path lets an attacker recover the private scalar.

### Likelihood Explanation

`RAND_bytes` failure is uncommon in normal operation but is not theoretical:

- Entropy exhaustion on embedded or containerized systems during early boot.
- FIPS-mode OpenSSL with a failed self-test puts the PRNG into an error state that causes `RAND_bytes` to return 0 on every subsequent call.
- A custom `RAND_METHOD` installed by the integrating application that can return 0.

Because the function returns `1` unconditionally regardless of the `RAND_bytes` outcome, the caller has no way to detect the failure. The nonce is consumed immediately, so there is no observable side-effect until a second signature is collected and the key is reconstructed.

### Recommendation

**Short term:** Replace the bare `RAND_bytes` call with the library's own checked wrapper, or add an explicit check:

```c
// Option A – use the existing checked wrapper
coinbase::crypto::gen_random(nonce, 64);

// Option B – explicit check matching OpenSSL convention
if (RAND_bytes(nonce, 64) != 1) {
  OPENSSL_cleanse(nonce, sizeof(nonce));
  return 0;  // propagate failure to caller
}
```

Apply the same fix to `hash_hram`: check `EVP_MD_CTX_new()` for NULL and check each `EVP_Digest*` return value, returning an error on failure rather than proceeding with an uninitialized `hram` buffer.

**Long term:** Enforce a static-analysis rule (e.g., via Clang's `warn_unused_result` attribute or a custom Clang-Tidy check) that flags any direct call to `RAND_bytes` outside the `gen_random` wrapper, and similarly for `EVP_MD_CTX_new`.

### Proof of Concept

1. Configure OpenSSL in FIPS mode with a deliberately failed PRNG self-test, or install a custom `RAND_METHOD` whose `bytes` callback always returns 0.
2. Call the public EdDSA 2-party signing API (`mpc::eddsa2pc::sign`) twice on different messages with the same key.
3. Both calls reach `ED25519_sign_with_scalar`; `RAND_bytes` returns 0 each time; `nonce[64]` holds the same uninitialized stack pattern (or zeros) on both calls.
4. Collect both signatures `(R, s₁)` and `(R, s₂)`. Because `R` is identical (same nonce), solve:
   ```
   s₁ − s₂ = (H₁ − H₂) · key  (mod q)
   key = (s₁ − s₂) · (H₁ − H₂)⁻¹  (mod q)
   ```
5. The full private scalar is recovered without any threshold collusion. [1](#0-0) [2](#0-1)

static bn_t hash_hram(const uint8_t sig[32], mem_t message, const uint8_t public_key[32]) {
  uint8_t hram[64];
  unsigned int hash_len = 0;
  EVP_MD_CTX* ctx = EVP_MD_CTX_new();
  EVP_DigestInit(ctx, EVP_sha512());
  EVP_DigestUpdate(ctx, sig, 32);
  EVP_DigestUpdate(ctx, public_key, 32);
  EVP_DigestUpdate(ctx, message.data, message.size);
  EVP_DigestFinal(ctx, hram, &hash_len);
  EVP_MD_CTX_free(ctx);
  return from_le_mod_q(mem_t(hram, 64));
}
