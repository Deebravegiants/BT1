### Title
Stale Cross-Chain rsETH/ETH Rate Used for L2 Minting Without Staleness Check — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

### Summary
The `CrossChainRateReceiver` caches the rsETH/ETH exchange rate received via LayerZero from L1. The `getRate()` function returns this cached value with no staleness validation. All L2 pool contracts (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`) use this rate to compute how many rsETH/wrsETH tokens to mint per deposited ETH. Because rsETH is a yield-bearing token whose price monotonically increases over time, a stale (lower-than-actual) cached rate causes depositors to receive more rsETH than they are entitled to, diluting existing holders.

### Finding Description

`CrossChainRateReceiver.lzReceive()` updates `rate` and `lastUpdated` only when a LayerZero message arrives from L1: [1](#0-0) 

`getRate()` returns the cached `rate` unconditionally — `lastUpdated` is stored but never compared against any maximum staleness threshold: [2](#0-1) 

Every L2 pool's `getRate()` delegates directly to this oracle: [3](#0-2) 

The minting formula divides by this rate: [4](#0-3) 

The same pattern is present in `RSETHPoolV3ExternalBridge`: [5](#0-4) 

And in `RSETHPoolNoWrapper`: [6](#0-5) 

The rate is pushed from L1 by calling `updateRate()` on `RSETHMultiChainRateProvider` or `RSETHRateProvider`, which reads `LRTOracle.rsETHPrice()`: [7](#0-6) 

`rsETHPrice` on L1 is itself only updated when `updateRsETHPrice()` is called by an off-chain keeper. Between keeper calls and LayerZero message delivery, the L2 cached rate can lag significantly behind the true on-chain rate.

### Impact Explanation

rsETH is a yield-bearing token: its ETH price strictly increases over time as staking rewards accrue. When the L2 cached rate is stale (lower than the current L1 rate), the minting formula `rsETHAmount = amountAfterFee * 1e18 / staleRate` produces a larger rsETH amount than the depositor is entitled to at the true rate. The excess rsETH minted is backed by no additional ETH, diluting the value held by all existing rsETH holders. This constitutes theft of unclaimed yield from existing holders.

**Impact: High** — theft of unclaimed yield from existing rsETH/wrsETH holders.

### Likelihood Explanation

The rate update path requires: (1) an off-chain keeper to call `updateRsETHPrice()` on L1, (2) a separate call to `updateRate()` on the provider, and (3) LayerZero message delivery to the L2 receiver. Each step introduces latency. During periods of high staking reward accrual, network congestion, or keeper inactivity, the L2 rate can be materially stale. Any unprivileged depositor can call `deposit()` at any time, including when the rate is stale. No special access is required.

### Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver`. In `getRate()`, revert if `block.timestamp - lastUpdated > maxStaleness`:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This mirrors the staleness check already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`: [8](#0-7) 

### Proof of Concept

1. L1 `LRTOracle.rsETHPrice` = 1.05e18 (current true rate, updated by keeper).
2. L2 `CrossChainRateReceiver.rate` = 1.02e18 (stale, last updated 48 hours ago; LayerZero message not yet delivered or keeper not called `updateRate()`).
3. Attacker calls `RSETHPoolV3.deposit{value: 1 ether}("ref")` on L2.
4. `viewSwapRsETHAmountAndFee(1e18)` computes: `rsETHAmount = 1e18 * 1e18 / 1.02e18 ≈ 0.9804 rsETH`.
5. Correct amount at true rate: `1e18 * 1e18 / 1.05e18 ≈ 0.9524 rsETH`.
6. Attacker receives `≈ 0.028 rsETH` excess — approximately 2.8% more than entitled — at the expense of all existing rsETH holders whose share value is diluted.
7. No admin action, no special role, no front-running required. The attacker simply deposits while the rate is stale.
