### Title
Queued `ProposalInit` Timestamp Lacks Lower-Bound Staleness Check, Allowing Stale Block Timestamps and L1 Gas Prices to Be Committed - (File: `crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

---

### Summary

When a `ProposalInit` for a future consensus round is stored in `queued_proposals`, its `timestamp` field is accepted without any validation. When the proposal is eventually dequeued after one or more round timeouts, `is_proposal_init_valid` only checks that the timestamp is not before the previous block's timestamp and not more than 1 second in the future. There is no lower-bound check against the current wall-clock time. A proposal with a timestamp from minutes ago passes all checks, causing the block to be committed with a stale `block_timestamp` and stale L1 gas prices.

---

### Finding Description

In `SequencerConsensusContext::validate_proposal`, when `init.round > self.current_round`, the proposal is stored in `queued_proposals` with no timestamp validation: [1](#0-0) 

When `set_height_and_round` later advances to that round, the queued proposal is dequeued and `validate_current_round_proposal` is called with the original `init`: [2](#0-1) 

Inside `validate_proposal`, `is_proposal_init_valid` is called, which enforces only two timestamp constraints: [3](#0-2) 

1. `init_proposed.timestamp >= last_block_timestamp` (monotonicity)
2. `init_proposed.timestamp <= now + block_timestamp_window_seconds` (not too far in future; configured as **1 second** in production) [4](#0-3) 

There is **no check** that `init_proposed.timestamp >= now - some_threshold`. A proposal with a timestamp from minutes ago passes both checks because:
- The stale timestamp is after the previous block's timestamp (monotonicity passes)
- The stale timestamp is not in the future (upper bound passes)

The stale `init.timestamp` is then used in two critical ways:

**1.** It becomes the `block_timestamp` in `BlockInfo` via `convert_to_sn_api_block_info(init)`, which is returned by the `get_block_timestamp()` syscall during execution.

**2.** It is passed directly to `get_l1_prices_in_fri_and_wei` to compute the L1 gas prices for the block: [5](#0-4) 

Since the validator also uses the same stale timestamp for L1 price lookup, both sides agree on the stale prices, and the block is committed with stale L1 gas prices.

---

### Impact Explanation

**Impact 1 — Wrong `get_block_timestamp()` syscall result (Critical):** The committed block carries a stale `block_timestamp`. Every contract that calls `get_block_timestamp()` during execution sees a wrong value. Time-sensitive contracts (options, auctions, time-locks, oracle freshness checks) can behave incorrectly or be exploited.

**Impact 2 — Stale L1 gas prices with economic impact (Critical):** The L1 gas prices for the block are computed from the stale timestamp. All transactions in the block are charged based on L1 gas prices from the past. If L1 gas prices have risen since the stale timestamp, users underpay; if they have fallen, users overpay. This is a direct, measurable economic impact on fee calculations for every transaction in the block.

---

### Likelihood Explanation

This occurs naturally in Tendermint consensus without any malicious actor:

1. A proposal for a future round arrives (normal behavior — validators queue proposals for rounds they have not yet reached).
2. Multiple rounds time out before the proposal is validated (realistic during network congestion, slow proposers, or Byzantine behavior).

Each round timeout is typically 30–60 seconds. If 3–5 rounds time out, the timestamp can be 90–300 seconds stale. The likelihood is **Low**, as it requires multiple consecutive round timeouts, but it is a realistic production scenario.

---

### Recommendation

Add a lower-bound staleness check in `is_proposal_init_valid` to reject proposals whose timestamp is too far in the past relative to the current wall-clock time:

```rust
// Reject timestamps that are too stale (e.g., older than block_timestamp_window_seconds in the past).
if now > init_proposed.timestamp + proposal_init_validation.block_timestamp_window_seconds {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "Timestamp is too stale: now={}, proposed={}, max_staleness={}",
            now,
            init_proposed.timestamp,
            proposal_init_validation.block_timestamp_window_seconds
        ),
    ));
}
```

Alternatively, when dequeuing a proposal from `queued_proposals` in `set_height_and_round`, check whether the stored timestamp is still within the acceptable window and drop the proposal if it is too stale.

---

### Proof of Concept

1. Validator is at round 0 for height H; previous block timestamp = `T_prev`.
2. A proposal for round 5 arrives with `init.timestamp = T_now` (current time). It is stored in `queued_proposals` — **no timestamp check is performed**. [1](#0-0) 
3. Rounds 0–4 all time out (~30 s each → ~150 s total).
4. `set_height_and_round(H, 5)` is called; the proposal is dequeued. [6](#0-5) 
5. `is_proposal_init_valid` is called with `init.timestamp = T_now` (150 s stale):
   - `T_now < T_prev` → **False** (passes)
   - `T_now > (T_now + 150) + 1` → **False** (passes, `T_now` is in the past) [3](#0-2) 
6. `get_l1_prices_in_fri_and_wei(provider, T_now, ...)` returns L1 prices from 150 seconds ago. [5](#0-4) 
7. The block is committed with `block_timestamp = T_now` (150 s stale) and stale L1 gas prices.
8. All transactions in the block are charged based on 150-second-old L1 gas prices.
9. Contracts calling `get_block_timestamp()` see `T_now` instead of the actual current time — a wrong syscall result baked into the committed state.
