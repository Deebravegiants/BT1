Audit Report

## Title
Missing Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Allows Stale-Price Deposits to Dilute rsETH Holders - (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-related return values, performing no `updatedAt` heartbeat check and no `answeredInRound >= roundId` check. Because this oracle is the sole price source for `LRTDepositPool.getRsETHAmountToMint()`, an attacker can deposit an LST when its Chainlink price is stale-high relative to true market value and receive rsETH representing more ETH than was actually contributed, diluting all existing rsETH holders when `updateRSETHPrice()` next corrects the stored price.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values of `latestRoundData()` — `(roundId, answer, startedAt, updatedAt, answeredInRound)` — are available via the interface declared in the same file, but only `price` is bound; `updatedAt` and `answeredInRound` are silently discarded with no validation. [2](#0-1) 

This oracle feeds directly into `LRTOracle.getAssetPrice()`: [3](#0-2) 

Which is consumed by `LRTDepositPool.getRsETHAmountToMint()`: [4](#0-3) 

Called unconditionally in `_beforeDeposit()`, the gate for all public deposits: [5](#0-4) 

**Exploit path:**
1. Chainlink rETH/ETH feed last updated at `P_stale = 1.09149e18`. True market price has since dropped to `P_true = 1.070e18` (within the 2% deviation threshold — no on-chain update triggered).
2. `rsETHPrice` was computed using `P_stale` via `_getTotalEthInProtocol()` → `getAssetPrice()`. [6](#0-5) 
3. Attacker calls `depositAsset(rETH, amount, 0, "")`. The protocol values the deposited rETH at `P_stale`, minting rsETH representing `amount × P_stale` ETH of claims.
4. The attacker contributed rETH worth only `amount × P_true` in true market terms — a ~2% over-issuance.
5. When `updateRSETHPrice()` is next called with fresh prices, `rsETHPrice` drops, diluting all prior holders proportionally.

**Existing checks are insufficient:**
- `minRSETHAmountExpected` is a user-set slippage parameter; the attacker sets it to 0.
- `pricePercentageLimit` guards against large rsETH price swings on `updateRSETHPrice()`, but does not prevent stale-price deposits from occurring before the update.
- The newer `ChainlinkOracleForRSETHPoolCollateral` performs partial checks (`answeredInRound < roundID`, `timestamp == 0`) but the primary L1 oracle has **none**. [7](#0-6) 

## Impact Explanation
Existing rsETH holders suffer dilution of their holdings: the attacker's over-minted rsETH claims a larger share of the pool's ETH than was contributed. This constitutes **theft of unclaimed yield** (High) from all rsETH holders. During an extended feed outage (price frozen for hours/days), the stale premium can far exceed the deviation threshold, escalating toward **protocol insolvency** (Critical) if large deposits are made at a severely inflated stale price before `updateRSETHPrice()` corrects it.

## Likelihood Explanation
Chainlink LST/ETH feeds have 24-hour heartbeats and deviation thresholds of 0.5%–2%. Within any 24-hour window the on-chain price can legitimately lag the true price by up to the deviation threshold without triggering an update. This is a **constant, always-present** condition. The attack requires no privileged access; `depositAsset()` is a public function. An attacker needs only to monitor the on-chain Chainlink price vs. spot price and deposit when the spread exceeds transaction costs. The attack is repeatable every time the spread reopens.

## Recommendation
Add per-feed staleness validation in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (block.timestamp - updatedAt > heartbeat[asset]) revert StalePrice();
if (price <= 0) revert InvalidPrice();
```

Store a `heartbeat` mapping (e.g., `86400` for 24-hour feeds) set alongside `assetPriceFeed` in `updatePriceFeedFor()`. Apply the same time-based heartbeat check to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which already has the `answeredInRound` check but lacks the `updatedAt` window check. [8](#0-7) 

## Proof of Concept

**Foundry fork test outline:**

```solidity
// Fork mainnet at a block where rETH/ETH Chainlink price is P_stale
// Warp block.timestamp forward by 23 hours (within 24h heartbeat, no update triggered)
// True rETH market price has dropped ~2% but Chainlink still reports P_stale

uint256 attackerRETH = 10_000e18;
deal(rETH, attacker, attackerRETH);

// Record existing holder's rsETH balance and total supply
uint256 supplyBefore = rsETH.totalSupply();
uint256 holderShareBefore = holderRsETH * 1e18 / supplyBefore;

vm.prank(attacker);
depositPool.depositAsset(rETH, attackerRETH, 0, "");

// Trigger price update with fresh oracle data
lrtOracle.updateRSETHPrice();

// Assert: existing holder's share of pool ETH has decreased
uint256 holderShareAfter = holderRsETH * 1e18 / rsETH.totalSupply();
assertLt(holderShareAfter, holderShareBefore); // holder diluted

// Assert: attacker's rsETH redeemable value > true ETH contributed
uint256 attackerRedeemableETH = attackerRsETH * lrtOracle.rsETHPrice() / 1e18;
assertGt(attackerRedeemableETH, attackerRETH * P_true / 1e18);
```

With 10,000 rETH at a 2% stale premium, the attacker extracts ~200 ETH of value from existing holders in a single transaction.
