Audit Report

## Title
Stale Cross-Chain rsETH/ETH Rate in `CrossChainRateReceiver.getRate()` Enables Over-Minting of wrsETH at Depositor's Benefit — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver` caches the rsETH/ETH rate received via LayerZero and exposes it through `getRate()` with no staleness check. Because rsETH is a yield-bearing token whose ETH value increases monotonically, any gap between rate updates leaves the L2 oracle below the true L1 price. Every L2 pool contract uses this stale rate to compute wrsETH mint/transfer amounts, causing depositors to receive more wrsETH than their ETH deposit justifies and diluting the yield owed to existing wrsETH holders.

## Finding Description

`CrossChainRateReceiver` stores the last received rate and its timestamp in `rate` and `lastUpdated` (L13–16). The `lzReceive()` function (L82–99) is the sole update path; it is triggered only when an off-chain actor calls `updateRate()` on the L1 provider and pays LayerZero fees. Between updates the L2 oracle is frozen.

`getRate()` (L102–105) returns `rate` unconditionally:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
```

No comparison against `lastUpdated` or any configurable `maxStaleness` threshold exists anywhere in `CrossChainRateReceiver` or its concrete subclass `RSETHRateReceiver` (L9–15 of `RSETHRateReceiver.sol`).

Every L2 pool contract calls `IOracle(rsETHOracle).getRate()` and divides by it:

```solidity
// RSETHPool.sol L316–319 / RSETHPoolV2.sol L230–233
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because `rsETHAmount` is inversely proportional to the rate, a stale (lower-than-true) rate produces a larger `rsETHAmount`. For `RSETHPoolV2` and all later variants the pool **mints** new wrsETH (`wrsETH.mint(msg.sender, rsETHAmount)`), directly inflating total supply beyond what the deposited ETH backs. For `RSETHPool` (Arbitrum) the pool **transfers** pre-held wrsETH, depleting reserves faster than the collected ETH justifies. In both cases the shortfall is borne by existing wrsETH holders whose accrued yield is diluted.

The L1 provider `RSETHMultiChainRateProvider.getLatestRate()` (L26–28) reads the live `rsETHPrice()` from `LRTOracle`, but this value is only pushed cross-chain on demand. No on-chain mechanism forces a push before a deposit is processed on L2.

Existing guards (`nonReentrant`, `whenNotPaused`, `limitDailyMint`) do not address oracle freshness. The `dailyMintLimit` in `RSETHPoolV2` caps total daily volume but does not prevent the per-deposit over-issuance caused by a stale rate.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing wrsETH holders hold a proportional claim on the ETH backing the total wrsETH supply. When new wrsETH is minted at a stale (below-true) rate, the new depositor's tokens represent more ETH value than was deposited. The ETH collected is eventually bridged to L1 and used to acquire rsETH; because too much wrsETH was issued, there is insufficient rsETH to back all outstanding wrsETH. The deficit is absorbed by existing holders, whose accrued yield is silently transferred to the new depositor. This matches the allowed impact "Theft of unclaimed yield" (High).

## Likelihood Explanation

**Medium.** Rate updates are not automatic; they require an off-chain actor to call `updateRate()` and pay LayerZero fees, making periodic (not per-block) updates the practical norm. rsETH accrues EigenLayer rewards continuously, so the L2 rate drifts below the true L1 rate between every pair of updates. Any unprivileged user can read `CrossChainRateReceiver.lastUpdated` on L2 and `LRTOracle.rsETHPrice()` on L1 to identify stale windows, then call the public `deposit()` function with no special role or capital lockup. The attack is repeatable every update cycle and requires no flash loan or governance action.

## Recommendation

1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**: introduce a configurable `maxStaleness` (e.g., 24 hours) and revert if `block.timestamp - lastUpdated > maxStaleness`.
2. **Enforce freshness in pool contracts**: before computing `rsETHAmount`, assert that the oracle rate is within the acceptable staleness window.
3. **Consider a push-on-deposit mechanism or a heartbeat keeper** that ensures the L2 rate is refreshed at a cadence shorter than the minimum economically meaningful drift.

## Proof of Concept

1. At T₀, `lzReceive()` sets `CrossChainRateReceiver.rate = 1.03e18` and `lastUpdated = T₀`.
2. EigenLayer rewards accrue. At T₁ = T₀ + 48 h, `LRTOracle.rsETHPrice()` on L1 returns `1.05e18`. No `updateRate()` has been called; the L2 oracle still returns `1.03e18`.
3. Attacker calls `RSETHPoolV2.deposit{value: 100 ether}("")`.
4. `viewSwapRsETHAmountAndFee(100e18)` computes (assuming `feeBps = 0` for clarity):
   - `rsETHAmount = 100e18 * 1e18 / 1.03e18 ≈ 97.087 wrsETH`
5. Correct amount at true rate: `100e18 * 1e18 / 1.05e18 ≈ 95.238 wrsETH`.
6. Attacker receives ≈ **1.849 excess wrsETH per 100 ETH** — yield stolen from existing holders — with no admin involvement and no special role.
