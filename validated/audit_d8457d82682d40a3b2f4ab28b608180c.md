### Title
`deploy_token` Metadata Signature Lacks Chain Binding, Enabling Cross-Chain Replay — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `aptos/sources/bridge_types.move`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The NEAR MPC signature used to authorize `deploy_token` across all destination chains (EVM, Starknet, Aptos, Solana) is computed over a Borsh-encoded `MetadataPayload` that contains only `(prefix, token_id, name, symbol, decimals)` — with no chain identifier. Because the same NEAR MPC key signs for all chains and the hash is chain-agnostic, any valid `deploy_token` signature observed on one chain can be replayed verbatim on every other chain to deploy the same bridged token without authorization.

---

### Finding Description

**Root cause — missing chain binding in `MetadataPayload` Borsh encoding:**

On EVM, `OmniBridge.deployToken` constructs the signed hash as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
```

No `omniBridgeChainId` is included. [1](#0-0) 

On Starknet, `MetadataPayloadImpl::to_borsh` encodes identically — `(PayloadType::Metadata, token, name, symbol, decimals)` — with no `omni_bridge_chain_id`:

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
``` [2](#0-1) 

On Aptos, `bridge_types::metadata_to_borsh` is byte-identical — no `chain_id` field:

```move
public fun metadata_to_borsh(self: &MetadataPayload): vector<u8> {
    buf.push_back(PAYLOAD_TYPE_METADATA);
    buf.append(borsh::encode_string(&self.token));
    buf.append(borsh::encode_string(&self.name));
    buf.append(borsh::encode_string(&self.symbol));
    buf.push_back(self.decimals);
    buf
}
``` [3](#0-2) 

On Solana, `DeployTokenPayload::serialize_for_near` serializes `(IncomingMessageType::Metadata, token, name, symbol, decimals)` — again no chain ID:

```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    IncomingMessageType::Metadata.serialize(&mut writer)?;
    self.serialize(&mut writer)?;
    ...
}
``` [4](#0-3) 

**Contrast with `fin_transfer`:** The `TransferMessagePayload` Borsh encoding correctly interleaves `chain_id` before both `token_address` and `recipient`, binding the signature to the executing chain. [5](#0-4) [6](#0-5)  The `deploy_token` path has no equivalent protection.

**Per-chain duplicate guard does not prevent cross-chain replay:** Each chain only checks whether the token is already deployed on *that* chain:

- EVM: `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` [7](#0-6) 
- Starknet: `assert(existing_token.is_zero(), 'ERR_TOKEN_ALREADY_DEPLOYED')` [8](#0-7) 
- Aptos: `assert!(!state.near_to_aptos_token.contains(token_id), E_TOKEN_ALREADY_DEPLOYED)` [9](#0-8) 

None of these guards prevent the same signature from being accepted on a *different* chain.

**Signature verification is chain-agnostic:** All chains verify against the same NEAR MPC-derived address (`nearBridgeDerivedAddress` / `omni_bridge_derived_address` / `near_bridge_derived_address`). [10](#0-9) [11](#0-10)  Because the payload hash is identical across chains for the same token metadata, the ECDSA signature recovers to the same signer on every chain.

---

### Impact Explanation

An attacker who observes a legitimate `deploy_token` transaction on chain A (e.g., Ethereum) can extract the `(signatureData, metadata)` from the calldata and submit it to `deploy_token` on chain B (e.g., Starknet, Aptos, Solana). This:

1. **Deploys an unauthorized bridged token** on chain B — a token the protocol never intended to exist there.
2. **Enables `bind_token` on NEAR** — the attacker can submit a proof of the chain B `DeployToken` event to NEAR's `bind_token`, registering the token for chain B in the NEAR bridge's token mappings. [12](#0-11) 
3. **Opens unauthorized transfer routes** — once registered, the NEAR bridge will sign `fin_transfer` payloads for chain B for this token, enabling cross-chain transfers to a chain the token issuer never authorized.
4. **Bypasses the intended authorization gate** — the MPC signature was issued in response to a `log_metadata` event on a specific source chain for a specific destination; the missing chain binding allows it to authorize deployment on every chain simultaneously.

This fits the allowed impact: **High — acceptance of insufficiently-bound signatures that bypass execution gates**, and potentially **asset-identity/token-mapping divergence** that breaks backing guarantees if the unauthorized chain has different security properties.

---

### Likelihood Explanation

- The `deploy_token` calldata (signature + metadata) is fully public on-chain once submitted on any chain.
- Any unprivileged actor can copy it and submit it to `deploy_token` on any other chain.
- No privileged access, leaked keys, or colluding validators are required.
- The only prerequisite is that a legitimate `deploy_token` has been executed on at least one chain — a routine operational event.

---

### Recommendation

Include the destination chain identifier in the `MetadataPayload` Borsh encoding, mirroring the existing `chain_id` binding in `TransferMessagePayload`. Concretely:

- Add `chain_id` as the first field after the payload type prefix in `metadata_to_borsh` / `to_borsh` / `serialize_for_near` on all chains.
- Update the NEAR MPC signing path (`log_metadata_callback`) to include the destination chain in the signed payload. [13](#0-12) 
- Update all SDK/tooling that constructs `deploy_token` calls to pass the chain-bound payload.

---

### Proof of Concept

**Step 1 — Observe a legitimate `deploy_token` on Ethereum:**

```
tx = eth.getTransaction(deploy_token_tx_hash)
signatureData = tx.input[4:69]   # first 65 bytes of calldata after selector
metadata      = abi.decode(tx.input[69:], MetadataPayload)
# metadata = { token: "near:token.near", name: "Foo", symbol: "FOO", decimals: 18 }
```

**Step 2 — Replay on Starknet:**

```python
starknet_bridge.deploy_token(
    signature=signatureData,   # identical bytes
    payload=metadata           # identical struct
)
# Succeeds: keccak256(borsh(metadata)) is chain-agnostic; same MPC key verifies
```

**Step 3 — Register on NEAR via `bind_token`:**

```bash
near call omni-bridge.near bind_token \
  '{"chain_kind": "strk", "prover_args": <proof_of_starknet_DeployToken_event>}' \
  --deposit 1
# Succeeds: Starknet bridge is a registered factory; token now mapped for Starknet
```

**Result:** Token `near:token.near` is now deployed and registered on Starknet without the token issuer or NEAR protocol ever authorizing a Starknet deployment. The NEAR bridge will subsequently sign `fin_transfer` payloads for Starknet for this token, opening an unauthorized transfer route.
