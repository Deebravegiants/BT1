### Title
`skip_verify=true` Hardcoded in `api::pve::decrypt` Enables Attacker-Controlled Scalar Substitution — (`src/cbmpc/api/pve_base_pke.cpp`)

### Summary

The public API function `coinbase::api::pve::decrypt` unconditionally passes `skip_verify=true` to `ec_pve_t::decrypt`, bypassing the cut-and-choose proof that binds the ciphertext's `Q` field to the encrypted scalar. Because `restore_from_decrypted` only checks `x_value * G == Q` using the `Q` deserialized from the attacker-supplied blob, an attacker who controls the ciphertext can substitute any scalar `x_evil` of their choice as the output of `decrypt`.

### Finding Description

**Hardcoded bypass.** `api::pve::decrypt` calls `ec_pve_t::decrypt` with `skip_verify=true`: [1](#0-0) 

Inside `ec_pve_t::decrypt`, this flag gates the only call to `verify`: [2](#0-1) 

**What `verify` would have caught.** `ec_pve_t::verify` recomputes `b_tag = hash(Q, label, c0, c1, X0, X1)` and rejects any ciphertext where `b_tag != b`. This is the cut-and-choose binding that ties `b`, `x_rows`, `r`, and `c` to a specific `Q`. With `skip_verify=true`, this check never runs. [3](#0-2) 

**Attacker-controlled `Q` flows into the only remaining guard.** After skipping `verify`, `ec_pve_t::decrypt` derives `inner_label` from the ciphertext's `Q` (attacker-controlled) and the caller-supplied `label`: [4](#0-3) 

`restore_from_decrypted` then reconstructs `x_value = x_bi_bar + x_bi (mod q)` and accepts it if and only if `x_value * G == Q`: [5](#0-4) 

Since `Q` is read directly from the attacker-supplied blob, the attacker can set `Q = x_evil * G` and craft `c[i]` so that decryption yields `x_bi_bar = x_evil - x_bi (mod q)`, making the check trivially pass for their chosen scalar.

**`Q` is fully attacker-controlled.** The `ec_pve_t` struct is deserialized from the blob's `ct` field with no integrity protection before `decrypt` is called: [6](#0-5) 

The `convert` method deserializes `Q`, `L`, `b`, `x_rows`, `r`, and `c` directly from attacker bytes: [7](#0-6) 

### Impact Explanation

An attacker who can supply the ciphertext blob to an honest party holding `dk` can force `api::pve::decrypt` to output any scalar `x_evil` of the attacker's choice. Because the attacker chose `x_evil`, they know the private scalar the honest party will use for all subsequent operations (signing, key derivation, etc.). This is a full key-substitution attack: the honest party believes it has recovered a legitimate private key, but the attacker already knows it.

### Likelihood Explanation

The attack requires only:
1. Knowledge of `ek` (public) and `label` (public or observable).
2. The ability to supply a crafted ciphertext blob to the honest party's `decrypt` call — a standard attacker capability at any API or protocol boundary where ciphertexts are received from peers or untrusted storage.

No cryptographic breaks, threshold collusion, or memory compromise are needed. The construction is fully explicit (see PoC below).

### Recommendation

Remove the `skip_verify=true` override in `api::pve::decrypt`. The `ek` parameter is already present in the function signature, so `verify` can be called unconditionally before proceeding to decryption. If performance is a concern for callers who have already verified externally, expose a separate internal API rather than disabling the guard in the public path. [8](#0-7)
