### Title
Missing Vote Signature Verification Allows Unprivileged Peer to Forge Validator Votes and Corrupt Consensus — (`crates/apollo_consensus/src/single_height_consensus.rs`)

### Summary
`SingleHeightConsensus::handle_vote` accepts any `Vote` whose `voter` field names a known committee member, without verifying that `Vote.signature` was produced by the private key corresponding to that validator's address. An unprivileged gossipsub peer can broadcast forged votes for any validator address with an empty/default signature and have them counted toward quorum.

### Finding Description
The vote processing pipeline is:

```
gossipsub peer → protobuf decode → manager::handle_vote → shc::handle_vote → state_machine
```

At no point in this pipeline is `Vote.signature` verified against `Vote.voter`. The only guard in `shc::handle_vote` is:

```rust
// TODO(Asmaa): verify the signature          // line 242
if !self.committee.members().iter().any(|s| s.address == vote.voter) {
    return VecDeque::new();  // only rejects non-validators
}
```

This check confirms `vote.voter` is a committee member address, but does not confirm the sender controls the corresponding private key. The `verify_precommit_vote_signature` function in `apollo_signature_manager` exists and is correct, but is never invoked in this path. The protobuf converter also explicitly defaults to an empty signature when none is provided (`unwrap_or_default()`).

### Impact Explanation
An attacker who knows the validator committee (public information) can:
1. Construct `Vote` structs with `voter` set to any known validator `ContractAddress` and `signature = RawSignature::default()`.
2. Broadcast them on the consensus gossipsub topic before legitimate votes arrive.
3. The duplicate/conflict check (`received_vote`) only blocks a second vote from the same `(round, voter)` pair — it does not block the first forged vote.
4. With enough forged votes (≥ 2/3+1 for Byzantine quorum), the target node reaches a false `DecisionReached` for an attacker-chosen `ProposalCommitment`, diverging from the honest network and producing wrong committed state.

This maps to: **Critical — Wrong state, receipt, event, and storage values from executing a block decided via forged consensus.**

### Likelihood Explanation
The gossipsub network is public; any peer can connect and publish on the vote topic. The validator committee is public. No cryptographic material is required. The attack requires only knowledge of the committee addresses and the ability to send protobuf-encoded messages, making it trivially executable.

### Recommendation
Call `verify_precommit_vote_signature` (already implemented in `apollo_signature_manager`) inside `shc::handle_vote` immediately after the committee membership check, using the public key associated with `vote.voter` from the committee roster. Reject and report the peer for any vote whose signature does not verify.

### Proof of Concept
Construct a `Vote` with an arbitrary committee validator address and `RawSignature::default()`, inject it via a mock `broadcasted_messages_receiver`, and assert `shc.handle_vote` returns a non-empty `Requests` (i.e., the vote is accepted and counted). The existing test utilities already do exactly this — `prevote()` and `precommit()` in `test_utils.rs` construct votes with `signature: RawSignature::default()` and the tests confirm they are accepted: [7](#0-6)
