Audit Report

## Title
Stale rsETH Rate Masked by Fresh ETH/USD Metadata in `RSETHPriceFeed.latestRoundData()` â€” (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary
`RSETHPriceFeed.latestRoundData()` computes `answer` as `rsETHPrice Ă— ETH/USD` but returns `updatedAt`, `startedAt`, `roundId`, and `answeredInRound` sourced entirely from the ETH/USD Chainlink feed. Because `LRTOracle.rsETHPrice` is a stored value updated only via on-chain transactions, the returned metadata is structurally incapable of reflecting rsETH price staleness. Any downstream consumer applying the standard Chainlink staleness check will receive a false freshness guarantee when the rsETH component is stale.

## Finding Description
`RSETHPriceFeed.latestRoundData()` at lines 63â€“70 copies all metadata fields from `ETH_TO_USD.latestRoundData()` and only replaces `answer`:

```solidity
// contracts/oracles/RSETHPriceFeed.sol, lines 68-69
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

`RS_ETH_ORACLE.rsETHPrice()` resolves to `LRTOracle.rsETHPrice` (line 28), a plain storage variable that is only written at line 313 inside `_updateRsETHPrice()`, which is only triggered by calling `updateRSETHPrice()` (line 87) or `updateRSETHPriceAsManager()` (line 94).

The mismatch is unconditional: even under normal keeper delays, high gas periods, or block stuffing, `updatedAt` and `answeredInRound` will reflect ETH/USD freshness while `answer` embeds an arbitrarily old rsETH/ETH rate. There is no code path in `latestRoundData()` that checks or signals rsETH price staleness. The `AggregatorV3Interface` implicit contract â€” that `updatedAt` reflects when the returned `answer` was last valid â€” is violated structurally.

**Exploit path (block stuffing variant):**
1. Attacker identifies a downstream consumer of `RSETHPriceFeed` that applies `require(block.timestamp - updatedAt < heartbeat)` or `require(answeredInRound >= roundId)`.
2. Attacker stuffs blocks to prevent `updateRSETHPrice()` from landing, freezing `rsETHPrice`.
3. ETH/USD Chainlink feed continues updating normally; `updatedAt` from `latestRoundData()` remains fresh.
4. Consumer's staleness guard passes; it acts on a composite price embedding a frozen rsETH rate.

**Exploit path (keeper failure variant, no attacker required):**
Same outcome if the keeper simply fails to call `updateRSETHPrice()` for any reason â€” the metadata mismatch is unconditional.

## Impact Explanation
**Low â€” Contract fails to deliver promised returns, but does not directly lose value.**

`RSETHPriceFeed` implements `AggregatorV3Interface` and is designed to be consumed by downstream protocols. The returned `updatedAt` and `answeredInRound` fields are structurally incapable of reflecting rsETH price staleness, violating the interface's freshness semantics. Consumers relying on standard Chainlink staleness patterns will silently accept a stale composite price. No funds are lost within this repository directly, but the contract fails to deliver the accurate freshness metadata it promises.

## Likelihood Explanation
The metadata mismatch is unconditional and persistent â€” it exists whenever `rsETHPrice` has not been updated as recently as the ETH/USD feed. Keeper delays, high gas periods, or deliberate block stuffing all trigger it. Block stuffing on Ethereum mainnet is expensive but has been executed in practice. The structural flaw requires no attacker at all for the keeper-failure variant, making it a latent risk present at all times.

## Recommendation
Store a `rsETHPriceUpdatedAt` timestamp in `LRTOracle` alongside `rsETHPrice`, updated atomically at line 313 when `rsETHPrice` is written. In `RSETHPriceFeed.latestRoundData()`, return `min(ethUsdUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt`. This ensures any staleness check on `updatedAt` reflects the freshness of the weakest component of the composite price. Apply the same fix to `getRoundData()` at lines 53â€“61, which has the identical structural flaw.
