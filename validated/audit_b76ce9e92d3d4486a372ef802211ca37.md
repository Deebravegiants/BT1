### Title
`compare_retrospective_block_hash` Disabled in Production Silently Accepts Wrong Chain-Anchor Hash, Corrupting Block Hash Contract State - (File: `crates/apollo_deployments/resources/app_configs/consensus_manager_config.json`)

### Summary

The production deployment configuration explicitly sets `compare_retrospective_block_hash` to `false`, disabling the only cross-check that detects divergence between the batcher's and state sync's view of historical block hashes. When disabled, both the proposer and validator silently use the batcher's `retrospective_block_hash` without verifying it against state sync. If the batcher's view diverges from state sync (e.g., due to a chain reorganization or storage inconsistency), the wrong hash is embedded into the block hash contract state and committed to the chain without any detection or rejection.

### Finding Description

The function `retrospective_block_hash()` in `crates/apollo_consensus_orchestrator/src/utils.rs` fetches the block hash for block `height - STORED_BLOCK_HASH_BUFFER` from two independent sources: the batcher client and the state sync client. [1](#0-0) 

The critical guard is the conditional at line 384:

```rust
if compare_retrospective_block_hash && state_sync_block_hash != batcher_block_hash {
    // ... returns Err(HashMismatch)
}
Ok(Some(BlockHashAndNumber { number: block_number, hash: batcher_block_hash }))
```

When `compare_retrospective_block_hash` is `false`, the function **skips the comparison entirely** and returns `batcher_block_hash` unconditionally — with no warning, no metric, and no error — even if `state_sync_block_hash != batcher_block_hash`.

The production deployment config overrides the default (`true` in `config_schema.json`) to `false`: [2](#0-1) 

```json
"consensus_manager_config.context_config.dynamic_config.compare_retrospective_block_hash": false,
```

The default in `config_schema.json` is `true`: [3](#0-2) 

This flag flows into both the **proposer** path (`build_proposal.rs` → `wait_for_retrospective_block_hash`) and the **validator** path (`validate_proposal.rs` → `initiate_validation` → `retrospective_block_hash`): [4](#0-3) [5](#0-4) 

The resulting `retrospective_block_hash` is passed directly into `ValidateBlockInput` / `ProposeBlockInput` and forwarded to the batcher, which stores it in the block hash contract at the start of block execution: [6](#0-5) 

The block hash contract stores `retrospective_block_hash` at storage key `block_number`, making it accessible to all smart contracts via the `get_block_hash` syscall. This value is part of the state trie, so it flows into the state root and ultimately into the block hash.

### Impact Explanation

If the batcher's stored block hash for block `N - STORED_BLOCK_HASH_BUFFER` diverges from state sync's view (e.g., due to a chain reorganization, a storage bug, or a race condition during sync), the following chain of corruption occurs:

1. Both proposer and validator compute the same wrong `retrospective_block_hash` (since both read from their respective batchers, which may share the same diverged state).
2. The wrong hash is written into the block hash contract at `BLOCK_HASH_CONTRACT_ADDRESS[block_number]`.
3. Any smart contract calling `get_block_hash(N - STORED_BLOCK_HASH_BUFFER)` receives the wrong value — a corrupted authoritative storage read.
4. The state root is computed over the corrupted state, and the block hash is computed over the corrupted state root.
5. The block is committed with a wrong state, wrong block hash, and wrong `get_block_hash` syscall results.

This matches the impact category: **Wrong state, storage value, or revert result from blockifier/syscall/execution logic for accepted input** (Critical), and **RPC execution returns an authoritative-looking wrong value** (High).

### Likelihood Explanation

The trigger requires the batcher's block hash storage to diverge from state sync. This can occur during:
- A Starknet chain reorganization (the direct analog of the Ethereum bridge bug)
- A batcher restart or storage inconsistency during sync
- A race condition between `add_sync_block` and `get_block_hash` calls

With `compare_retrospective_block_hash: false`, **no detection occurs** — not even a log warning. The divergence is invisible until downstream effects (wrong `get_block_hash` results) are observed. The flag being explicitly set to `false` in the production app config (not a test or overlay) means this is the live operational state.

### Recommendation

1. **Short term**: Re-enable `compare_retrospective_block_hash: true` in `crates/apollo_deployments/resources/app_configs/consensus_manager_config.json`. The default in `config_schema.json` is already `true`; the production override should be removed.
2. **Short term**: Even when the flag is `false` (e.g., during a migration window), add an unconditional `warn!` log when `state_sync_block_hash != batcher_block_hash` so divergence is observable.
3. **Long term**: Design and document a reorganization-handling strategy for the batcher analogous to the recommendation in the external report: detect divergence, calculate which committed blocks need to be unapplied, and handle the resulting state inconsistencies.
