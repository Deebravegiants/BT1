### Title
Proposal Deposits for Non-Executed Updates Are Permanently Lost When `do_update` Clears All Entries — (`crates/contract/src/update.rs`)

---

### Summary

`propose_update` requires each proposer to attach a NEAR deposit covering estimated storage costs. However, `ProposedUpdates` never records which account paid which deposit. When any update reaches threshold and `do_update` is called, it unconditionally clears **all** pending entries — including those whose proposers paid deposits — without issuing any refunds. Those deposits are permanently locked in the contract with no recovery path.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` collects a deposit from the caller, computes the required amount via `ProposedUpdates::required_deposit`, refunds any excess, and then stores the update via `ProposedUpdates::propose`:

```rust
// lib.rs:1308-1331
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
if attached < required { return Err(...); }
let id = self.proposed_updates.propose(update);
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
``` [1](#0-0) 

`ProposedUpdates::propose` stores only the update payload and a `bytes_used` estimate — no depositor account, no deposit amount:

```rust
// update.rs:167-173
pub fn propose(&mut self, update: Update) -> UpdateId {
    let bytes_used = bytes_used(&update);
    let id = self.id.generate();
    self.entries.insert(id, UpdateEntry { update, bytes_used });
    id
}
``` [2](#0-1) [3](#0-2) 

When `do_update` is triggered by a successful `vote_update`, it removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()` — wiping every other pending proposal without any refund logic:

```rust
// update.rs:195-200
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
``` [4](#0-3) 

There is no `withdraw_proposal` or `cancel_proposal` method, and no other code path that refunds proposal deposits. The only other entry-removal path is `remove_update_vote`, which removes a vote but leaves the entry (and its deposit) in place. [5](#0-4) 

The `bytes_used` function deliberately over-estimates storage (adding a fixed 128-account buffer) to ensure the deposit always covers actual storage, meaning the required deposit is always larger than the actual storage consumed:

```rust
// update.rs:278-295
fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;
    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;
    match update {
        Update::Contract(code) => { bytes_used += code.len() as u128; }
        ...
    }
    bytes_used
}
``` [6](#0-5) 

For a 1 MB contract binary, the required deposit is approximately `(1_000_000 + 8_192 + overhead) * storage_byte_cost`. At NEAR's current storage cost (~10^19 yoctoNEAR/byte), this is on the order of **10 NEAR per proposal**. When multiple participants each propose a different update and one reaches threshold, all other proposers permanently lose their deposits.

---

### Impact Explanation

**Medium.** The contract breaks the accounting invariant that storage deposits must be refundable when the associated storage is freed. When `do_update` clears all non-winning entries, the freed storage reduces the contract's on-chain storage footprint, but the NEAR tokens paid by the non-winning proposers remain locked in the contract with no recovery mechanism. In a governance scenario where several participants each propose a competing update (e.g., different config values or different contract binaries), all but one proposer permanently lose their deposits. The amount per proposal scales with the size of the contract binary, making this a meaningful financial loss.

---

### Likelihood Explanation

**Low.** The scenario requires multiple participants to have active, distinct proposals simultaneously. In practice, governance operations are coordinated off-chain, so concurrent competing proposals are uncommon. However, the condition is entirely reachable in normal operation (no adversarial setup required), and the loss is automatic and irreversible once any update reaches threshold.

---

### Recommendation

Store the depositor's `AccountId` and the exact deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) depositor: AccountId,   // add
    pub(super) deposit: NearToken,     // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.depositor).transfer(entry.deposit)` for each non-winning entry. This mirrors the refund pattern already used in `propose_update` itself and in `submit_participant_info`.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR.
2. Participant B calls `propose_update` with a different 1 MB contract binary, attaching ~10 NEAR.
3. Threshold participants vote for A's proposal via `vote_update`.
4. `vote_update` calls `do_update(&id_A, gas)`.
5. `do_update` removes entry A, then calls `self.entries.clear()` — entry B is deleted.
6. No refund is issued to Participant B. Their ~10 NEAR is permanently locked in the contract.
7. There is no `withdraw_proposal` or any other method that can recover B's deposit. [7](#0-6) [8](#0-7)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

            bytes_used += bytes.len() as u128;
