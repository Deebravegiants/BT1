### Title
Stale Votes from Removed Participants Count Toward OS Measurement Threshold After Resharing — (`crates/contract/src/tee/measurements.rs`)

---

### Summary

`MeasurementVotes::count_votes()` tallies every entry in `vote_by_account` without filtering by the current participant set. After a resharing removes a participant, their vote persists in `measurement_votes` until the asynchronously-spawned `clean_tee_status` promise executes. In the window between resharing completion and that cleanup, a single colluding current participant can cast the deciding vote for a malicious OS measurement, reaching the governance threshold with stale votes from removed participants supplying the remainder.

---

### Finding Description

`MeasurementVotes::count_votes()` counts all stored votes unconditionally: [1](#0-0) 

`vote_add_os_measurement` uses this raw count directly against the governance threshold: [2](#0-1) 

After resharing completes in `vote_reshared`, the contract transitions to Running state and then spawns `clean_tee_status` as a **detached** (fire-and-forget) promise: [3](#0-2) 

`clean_tee_status` calls `clean_non_participant_votes`, which filters `measurement_votes` to only retain current participants: [4](#0-3) 

Because the cleanup is a separate receipt processed after the resharing receipt, there is a window — at minimum one receipt-processing round — during which stale votes from removed participants remain in `measurement_votes` and are counted by `count_votes()`.

The same flaw affects `vote_code_hash` (via `CodeHashesVotes`) and `vote_add_launcher_hash` (via `LauncherHashVotes`), both of which are also cleaned by `clean_non_participant_votes` but share the same asynchronous cleanup gap. [5](#0-4) 

---

### Impact Explanation

OS measurements (`MRTD`, `RTMR0-2`, `key_provider_event_digest`) are the on-chain whitelist that gates TEE attestation acceptance. Adding a malicious measurement allows a node running in a compromised CVM environment to pass `submit_participant_info` attestation verification. Once attested, that node can be proposed and voted into the participant set in a subsequent resharing, gaining access to key-share material and signing capability.

This is a **participant/attestation authorization bypass**: the governance threshold for adding a trusted measurement is circumvented, enabling unauthorized TEE environments to be whitelisted.

---

### Likelihood Explanation

The attack requires:
1. One participant (P1) who is about to be removed casts a vote for a malicious measurement M before resharing.
2. Resharing completes, removing P1. The new participant set has a lower total count and potentially a lower threshold.
3. One colluding current participant (P2) submits `vote_add_os_measurement(M)` in the receipt window before `clean_tee_status` executes.

With a 3-participant set (threshold = 2), P1's stale vote + P2's fresh vote = 2 ≥ threshold, so M is added with only **one** current participant voting — below the required threshold of two. The attack requires one removed and one current compromised participant, meaning the effective current-participant collusion is strictly below threshold.

The timing window is narrow (one or a few receipt rounds on NEAR), but it is deterministic and does not require network-level interference.

---

### Recommendation

Apply the same participant-membership filter inside `count_votes()` that `get_remaining_votes()` already implements, passing the current `Participants` set at vote-counting time:

```rust
fn count_votes(&self, action: &MeasurementVoteAction, participants: &Participants) -> u64 {
    u64::try_from(
        self.vote_by_account
            .iter()
            .filter(|(pid, a)| {
                *a == action
                    && participants.is_participant_given_participant_id(&pid.get())
            })
            .count(),
    )
    .expect("count fits in u64")
}
```

This mirrors the pattern already used correctly in `TeeVerifierVotes::vote()`: [6](#0-5) 

and in `ThresholdParametersVotes::n_votes()`: [7](#0-6) 

Apply the same fix to `CodeHashesVotes` and `LauncherHashVotes` for consistency.

---

### Proof of Concept

**Setup**: 3 participants `{P1, P2, P3}`, governance threshold = 2.

1. **P1 votes** for malicious measurement `M` via `vote_add_os_measurement(M)`. Count = 1 (below threshold 2). Vote stored in `measurement_votes[P1] = Add(M)`.

2. **Resharing** removes P1. New set: `{P2, P3}`, threshold = 2. `vote_reshared` transitions to Running and spawns `clean_tee_status` as a detached promise.

3. **Before `clean_tee_status` receipt executes**, P2 calls `vote_add_os_measurement(M)`.

4. Inside `MeasurementVotes::vote()`, `count_votes(Add(M))` iterates `vote_by_account` and finds **two** entries: P1's stale entry and P2's new entry → returns 2.

5. `2 >= threshold (2)` → `add_measurement(M)` is called → malicious measurement `M` is whitelisted.

6. `clean_tee_status` eventually runs and removes P1's stale entry, but the measurement is already committed to `allowed_measurements`.

The existing unit test `test_clean_non_participant_votes_removes_stale_votes` confirms that stale votes persist until the cleanup runs: [8](#0-7) 

and the comment in `vote_reshared` acknowledges that the detached cleanup can fail or be delayed: [9](#0-8)

    /// 1. P1 and P2 vote for malicious hash before resharing.

    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
    }
