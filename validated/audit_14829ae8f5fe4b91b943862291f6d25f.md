### Title
Ethereum-Specific Finality Tags Applied Uniformly to All EVM Chains in `EvmInspector` May Cause Finality Verification Bypass on Non-Ethereum Chains — (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

The `EvmInspector` uses a single implementation for all EVM-compatible chains (BNB, Base, Arbitrum, HyperEvm, Polygon, Abstract). Its `verify_finality_level` function blindly maps the user-supplied `EvmFinality` enum to Ethereum-specific block tags (`"safe"`, `"finalized"`, `"latest"`) and calls `eth_getBlockByNumber` with those tags on every supported EVM chain. Several supported chains do not implement these tags with Ethereum semantics, or do not implement them at all. When a chain's RPC silently returns the latest block for an unsupported tag (rather than an error), the finality check passes for any confirmed transaction regardless of actual finality, allowing the MPC network to issue a signed attestation for a transaction that has not reached the requested finality level.

---

### Finding Description

`EvmInspector<Client, Chain>` is the single inspector implementation shared by every EVM-family chain in the system. [1](#0-0) 

Its `verify_finality_level` method converts the caller-supplied `EthereumFinality` value directly to a `FinalityTag` and issues `eth_getBlockByNumber` with that tag: [2](#0-1) 

The `FinalityTag` enum serialises to the JSON strings `"safe"`, `"finalized"`, and `"latest"`: [3](#0-2) 

This same inspector is instantiated for every EVM chain variant — BNB, Base, Arbitrum, HyperEvm, Polygon, Abstract — and the `EvmFinality` enum (including `Safe` and `Finalized`) is exposed directly to callers in the public `EvmRpcRequest` DTO: [4](#0-3) 

The node-side dispatch in `execute_foreign_chain_request` converts the DTO finality value to `EthereumFinality` and passes it to the shared inspector for every EVM chain without any chain-specific guard: [5](#0-4) 

The `verify_finality_level` logic only checks `head.number >= receipt_block_number`. If the RPC returns the latest block for an unsupported tag (a common fallback in non-Ethereum EVM nodes), this comparison always succeeds for any confirmed transaction, bypassing the finality requirement entirely. [6](#0-5) 

The contract's `respond_verify_foreign_tx` does not independently re-verify finality; it only checks that the submitted signature is valid over the `payload_hash`. If the MPC nodes produced a signature over a payload that includes an unfinalized block hash, the contract accepts it: [7](#0-6) 

---

### Impact Explanation

The MPC network's `verify_foreign_transaction` feature is the trust anchor for the Omnibridge inbound flow: a NEAR contract releases funds only after the MPC network attests that a foreign-chain transaction has reached a specified finality level. If the finality check is bypassed, the MPC network issues a valid ECDSA attestation over a payload that claims `EvmFinality::Safe` (or `Finalized`) for a transaction that has not actually reached that level. An attacker can use this premature attestation to claim bridge funds before the foreign-chain transaction is safe from reorg, constituting a forged foreign-chain verification that enables invalid bridge execution or double-spend.

---

### Likelihood Explanation

Several chains in the supported set are not Ethereum mainnet and do not implement `"safe"` / `"finalized"` block tags with Ethereum semantics:

- **Abstract** is a ZK-rollup whose RPC may not implement these tags at all, or may alias them to `"latest"`.
- **HyperEvm** is a new chain whose finality model differs from Ethereum's; its RPC behaviour for these tags is not standardised.
- **Arbitrum** implements `"finalized"` as L1-finalized (≈15 min), making `"safe"` and `"finalized"` semantically very different from Ethereum's usage.

An unprivileged caller can submit a `verify_foreign_transaction` request for any supported chain with any `EvmFinality` value. No privileged access, collusion, or key material is required to trigger the path.

---

### Recommendation

1. **Per-chain finality validation**: Introduce a chain-specific trait method (e.g., `supported_finality_levels`) that each `EvmChain` implementor declares. Reject requests at the inspector level if the requested finality level is not supported by the target chain.
2. **Allowlist finality levels per chain**: In the contract or node configuration, restrict which `EvmFinality` values are accepted for each `ForeignChain` variant, mirroring the 1inch resolution of restricting the feature to chains where it is known to work correctly.
3. **Treat RPC null/error for finality tags as `NotFinalized`**: Ensure that any non-success response to `eth_getBlockByNumber` with a finality tag is treated as a hard `NotFinalized` error rather than propagated as a generic client error that might be retried or silently ignored.

---

### Proof of Concept

1. Operator registers **Abstract** (or HyperEvm) as a supported foreign chain.
2. Attacker submits `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "Abstract": {
         "tx_id": "<tx just included in block N>",
         "finality": "Safe",
         "extractors": [{"BlockHash": null}]
       }
     }
   }
   ```
3. Each MPC node calls `eth_getTransactionReceipt` → gets `block_number = N`.
4. Each MPC node calls `eth_getBlockByNumber("safe", false)` on the Abstract RPC. The Abstract RPC, not implementing `"safe"`, returns the latest block (block M ≥ N).
5. `verify_finality_level` checks `M >= N` → passes.
6. `verify_block_is_canonical` passes (block N is on the canonical chain).
7. MPC nodes collectively sign `SHA-256(borsh(ForeignTxSignPayloadV1 { request, values: [BlockHash(hash_N)] }))`.
8. `respond_verify_foreign_tx` is called; the contract verifies the ECDSA signature and resolves the yield with the signed payload.
9. Attacker presents the signature to the Omnibridge contract to claim funds.
10. Block N is subsequently reorged; the attacker has already claimed funds against a transaction that no longer exists on the canonical chain. [8](#0-7) [9](#0-8) [10](#0-9)

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
