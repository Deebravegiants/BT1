### Title
User-Controlled `confirmations` Field Bypasses Bitcoin Finality Verification — (`crates/node/src/providers/verify_foreign_tx/sign.rs`)

### Summary

The `BitcoinRpcRequest.confirmations` field is fully user-controlled and is forwarded verbatim to `BitcoinInspector.extract()` as the block-confirmation threshold. A caller can set `confirmations: 0` (or any value below the bridge's intended safety margin) to obtain a valid MPC threshold signature for a Bitcoin transaction that has not yet reached finality, enabling a double-spend or invalid bridge execution.

### Finding Description

`verify_foreign_transaction` accepts a `VerifyForeignTransactionRequestArgs` whose inner `BitcoinRpcRequest` carries a user-supplied `confirmations` field. The contract performs no lower-bound validation on this value before storing the request: [1](#0-0) 

The stored `VerifyForeignTransactionRequest` (including the raw `confirmations` value) is then indexed by MPC nodes. When the leader node processes the request, it extracts the user-supplied value and passes it directly to the inspector as the finality threshold: [2](#0-1) 

Inside `BitcoinInspector::extract`, the only finality gate is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { … });
}
``` [3](#0-2) 

When `block_confirmations_threshold` is `0`, the inequality `0 ≤ N` is trivially true for any `N ≥ 0`, so the check is a no-op. The inspector proceeds to extract values and the MPC nodes produce a threshold signature for a transaction that may have zero or only one confirmation — well below any safe finality margin for Bitcoin.

This is the direct analog of the SpinLottery `_prizeCount` bug: in both cases a user-controlled numeric parameter is used verbatim as a safety threshold that is supposed to be protocol-enforced, allowing the caller to reduce the threshold to zero and bypass the intended invariant entirely.

### Impact Explanation

An attacker who controls a bridge-facing NEAR account can:

1. Broadcast a Bitcoin transaction `tx_A` (e.g., depositing funds into a bridge escrow).
2. Immediately call `verify_foreign_transaction` with `confirmations: 0` before `tx_A` accumulates any safe depth.
3. MPC nodes sign `tx_A` (the `0 ≤ actual_confirmations` check always passes).
4. The attacker receives a valid threshold signature over the extracted payload hash.
5. The attacker then RBF-replaces or double-spends `tx_A` on Bitcoin.
6. The attacker redeems the NEAR-side funds using the already-issued signature.

This constitutes a forged foreign-chain verification / double-spend condition, matching the **High** impact tier: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass … that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

The entry path requires only a standard NEAR account and a 1 yoctonear deposit — no privileged role, no threshold collusion, no TEE bypass. Setting `confirmations: 0` is a single field in the public `verify_foreign_transaction` call. Bitcoin shallow-reorg probability is non-trivial at 1–2 confirmations and the attacker can time the attack to maximise reorg risk (e.g., during periods of high fee volatility). Likelihood is **Medium**.

### Recommendation

Enforce a protocol-level minimum confirmation count on-chain or in the node before the request is accepted. Options:

- **Contract-side**: reject `BitcoinRpcRequest` with `confirmations` below a governance-set floor (stored in `Config` or `ForeignChainsMetadata`).
- **Node-side**: clamp `block_confirmations` to `max(request.confirmations, MIN_BITCOIN_CONFIRMATIONS)` before calling `inspector.extract`, so a user-supplied zero cannot reduce the effective threshold.

The `confirmations` field should be treated as a *hint* (or removed from the user-facing API entirely), with the authoritative minimum enforced by the protocol.
