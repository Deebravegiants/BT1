Audit Report

## Title
Missing Chainlink Oracle Staleness and Validity Checks Enable Incorrect rsETH Minting — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all four validity return values (`roundId`, `startedAt`, `updatedAt`, `answeredInRound`), using only the raw `answer`. When a Chainlink feed goes stale at an inflated price, any unprivileged depositor can call `depositAsset()` and receive excess rsETH proportional to the price inflation, extracting value from honest depositors. The same repository already implements the correct validation pattern in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the omission is unintentional.

## Finding Description

**Root cause — `contracts/oracles/ChainlinkPriceOracle.sol` L52:**

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All four validity fields are silently dropped. No check is performed for:
- **Stale round**: `answeredInRound < roundId` — answer is from a prior round
- **Incomplete round**: `updatedAt == 0` — round has not completed
- **Zero price**: `price == 0` — feed returned zero during an anomaly

Note: the claim that a negative `price` causes wrap-around over-minting is incorrect in Solidity 0.8+; `uint256(negativeInt)` reverts due to checked arithmetic. The stale and zero-price paths remain fully exploitable.

**Contrast with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` L30–32**, which correctly validates all three conditions before using the price.

**Exploit call chain:**

1. `LRTDepositPool.depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()` (`LRTDepositPool.sol` L520)
2. `getRsETHAmountToMint()` calls `ILRTOracle(lrtOracle).getAssetPrice(asset)` (`LRTOracle.sol` L157)
3. `LRTOracle.getAssetPrice()` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)` — which resolves to `ChainlinkPriceOracle.getAssetPrice()` with no staleness guard
4. The stale inflated price is multiplied by `amount` and divided by `rsETHPrice`, minting excess rsETH to the attacker

The same stale price also flows into `_updateRsETHPrice()` via `_getTotalEthInProtocol()` (`LRTOracle.sol` L339), inflating the global rsETH/ETH exchange rate used by all protocol participants.

**Zero-price path:** If `price == 0`, `rsethAmountToMint = 0`. If the depositor passes `minRSETHAmountExpected = 0`, `_beforeDeposit` does not revert (`0 < 0` is false), the depositor's tokens are transferred in, and 0 rsETH is minted — a permanent loss of deposited funds.

**Existing guards are insufficient:** The `pricePercentageLimit` circuit-breaker in `_updateRsETHPrice()` only fires when `updateRSETHPrice()` is called separately; it does not gate individual `depositAsset()` calls. The `minRSETHAmountExpected` slippage parameter protects the depositor from low prices, not from inflated stale prices.

## Impact Explanation

**Critical — Direct theft of user funds / Permanent freezing of funds.**

- **Stale inflated price path:** Attacker deposits LST at a stale inflated price, receives excess rsETH, and redeems via the unstaking vault for more ETH than deposited, extracting value from the pool at the expense of honest depositors.
- **Zero-price path:** Depositor's tokens are transferred to the protocol and 0 rsETH is minted, constituting a permanent freeze of deposited funds with no recovery mechanism visible in the deposit flow.

Both impacts match allowed Critical impact classes.

## Likelihood Explanation

Chainlink feeds go stale during network congestion, sequencer downtime (on L2s), or oracle node failures. The vulnerability window is the entire duration of the stale period. No privileges are required — any external caller can invoke `depositAsset()` or `depositETH()` with no preconditions. The attack is repeatable for the full duration of staleness and requires no victim interaction.

## Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept

**Stale-price theft (Foundry fork test outline):**

1. Fork mainnet at a block where the `stETH/ETH` Chainlink feed is live.
2. Warp `block.timestamp` forward past the feed's heartbeat interval without a new round being posted (simulating staleness); the last reported price is `1.05e18`.
3. Call `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")` as an unprivileged attacker.
4. Assert `getRsETHAmountToMint` returns `~1050 rsETH` (vs. the fair `~1000 rsETH` at true price `1.00e18`).
5. Initiate withdrawal of the excess rsETH via the unstaking vault; assert the attacker receives more ETH than deposited.
6. Assert `ChainlinkPriceOracle.getAssetPrice()` never reverts throughout, confirming no staleness guard fires.

**Zero-price freeze (unit test outline):**

1. Deploy a mock `AggregatorV3Interface` that returns `(1, 0, block.timestamp, block.timestamp, 1)` (price = 0, round complete).
2. Register it as the price feed for a supported asset.
3. Call `depositAsset(asset, 1e18, 0, "")`.
4. Assert the call succeeds, the depositor's tokens are transferred in, and the depositor's rsETH balance remains 0.
