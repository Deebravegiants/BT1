Audit Report

## Title
Missing `msg.sender == account` Authorization Allows Force-Claiming at Higher Fee - (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary
`MerkleDistributor.claim()` accepts a caller-supplied `account` parameter with no check that `msg.sender == account`, allowing any unprivileged caller to trigger a claim on behalf of any user. When `feeInBPS` is non-zero, an attacker can force-claim for a victim immediately before a fee reduction executes, permanently redirecting a portion of the victim's unclaimed tokens to `protocolTreasury` at the higher fee rate. The sibling contract `KernelMerkleDistributor` explicitly guards against this with `if (account != msg.sender) { revert Unauthorized(); }`, confirming the protocol's own recognition that this check is required.

## Finding Description
`MerkleDistributor.claim()` at L97–147 of `contracts/utils/MerkleDistributor/MerkleDistributor.sol` accepts `index`, `account`, `cumulativeAmount`, and `merkleProof` from the caller. There is no check that `msg.sender == account`. The merkle proof is verified against `(index, account, cumulativeAmount)`, but all of these values are derivable from the publicly posted off-chain merkle tree. After proof verification, the contract computes `claimableAmount = cumulativeAmount - userClaims[account].cumulativeAmount`, deducts `fee = claimableAmount * feeInBPS / 10_000`, transfers `amountToSend` to `account`, and transfers `fee` to `protocolTreasury`. The user's `userClaims[account]` state is permanently updated, consuming the claim slot.

`KernelMerkleDistributor._processClaim()` at L311–313 of `contracts/KERNEL/KernelMerkleDistributor.sol` contains `if (account != msg.sender) { revert Unauthorized(); }`, demonstrating the protocol's own standard for this pattern. `MerkleDistributor` omits it entirely.

Exploit flow:
1. `feeInBPS = 1000` (10%). Owner broadcasts a transaction to call `setFeeInBPS(0)`.
2. Attacker reads victim's `(index, account, cumulativeAmount, merkleProof)` from the public merkle tree.
3. Attacker calls `MerkleDistributor.claim(index, victim, cumulativeAmount, proof)` before the owner's transaction confirms.
4. Victim receives `claimableAmount * 90%`; `claimableAmount * 10%` is permanently sent to `protocolTreasury`.
5. `userClaims[victim].lastClaimedIndex` and `cumulativeAmount` are updated; the claim slot is consumed.
6. The owner's fee reduction executes. Victim cannot reclaim the lost 10% — they would need to wait for the next merkle root update with a higher `cumulativeAmount`.

No special privileges, no funds, and no private information are required. The attack is repeatable against any unclaimed account.

## Impact Explanation
**Low. Contract fails to deliver promised returns, but doesn't lose value.** The victim receives their tokens, but receives fewer tokens than they would have received by claiming after the fee reduction. The difference (`claimableAmount * feeInBPS / 10_000`) is permanently redirected to `protocolTreasury` and is unrecoverable for the victim within the current merkle epoch. The contract fails to deliver the full claimable amount the user was entitled to receive under the lower fee.

## Likelihood Explanation
High. The attack requires no privileges, no capital, and no private information. All parameters needed to call `claim()` on behalf of any user are derivable from the public merkle tree. The attack is executable by any EOA or contract whenever the contract is unpaused and `feeInBPS > 0`. The only precondition for financial harm is a pending or announced fee reduction, which is a realistic and observable on-chain event.

## Recommendation
Add the same authorization guard present in `KernelMerkleDistributor._processClaim()` to `MerkleDistributor.claim()`:

```solidity
if (account != msg.sender) revert Unauthorized();
```

Insert this check at the top of `MerkleDistributor.claim()` (after the `whenNotPaused` modifier), consistent with the pattern at `KernelMerkleDistributor.sol` L311–313. The `Unauthorized` error should also be added to the `IMerkleDistributor` interface. The `MerkleBlastPointsDistributor.claim()` function has the same structural omission but transfers no tokens, so its impact is limited to event-based griefing; the same fix should be applied for consistency.

## Proof of Concept
Foundry fork test outline:

```solidity
function testForceClaimAtHighFee() public {
    // Setup: deploy MerkleDistributor with feeInBPS = 1000 (10%)
    // Fund distributor with 1000e18 tokens
    // Build merkle tree: leaf = (index=1, alice, cumulativeAmount=1000e18)
    // Call setMerkleRoot(root)

    // Attacker (Bob) reads alice's proof from the public merkle tree
    // Bob calls claim before owner reduces fee
    vm.prank(bob);
    distributor.claim(1, alice, 1000e18, aliceProof);

    // Alice receives 900e18; 100e18 goes to protocolTreasury
    assertEq(token.balanceOf(alice), 900e18);
    assertEq(token.balanceOf(protocolTreasury), 100e18);

    // Owner reduces fee to 0
    vm.prank(owner);
    distributor.setFeeInBPS(0);

    // Alice's claim slot is consumed; she cannot reclaim the 100e18
    vm.prank(alice);
    vm.expectRevert(IMerkleDistributor.AlreadyClaimed.selector);
    distributor.claim(1, alice, 1000e18, aliceProof);
}
```
