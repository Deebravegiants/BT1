### Title
Harvester Can Unconditionally Redirect Farmer Block Reward to Arbitrary Address — (`chia/farmer/farmer_api.py`)

### Summary
A third-party harvester (an unprivileged peer in the CHIP-22 model) can set `farmer_reward_address_override` to any arbitrary puzzle hash in its `RespondSignatures` message. The farmer accepts this override without validating it against any allowlist, fee-quality threshold enforcement, or ownership check, and directly uses it as the `farmer_reward_address` in `DeclareProofOfSpace`. This causes the farmer's 1/8 block reward to be paid to the attacker's address instead of the farmer's configured address.

### Finding Description

The `farmer_reward_address_override` field exists in both `NewProofOfSpace` and `RespondSignatures` protocol messages to support the CHIP-22 third-party harvester fee convention, where a harvester may claim the farmer reward as a fee if a quality threshold is met.

In `new_proof_of_space`, when the override is present, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, which **only logs** a warning if the fee quality threshold is not met — it does not block the override or return early: [1](#0-0) [2](#0-1) 

Then in `_process_respond_signatures`, the farmer unconditionally replaces the farmer reward address with whatever the harvester sends in `RespondSignatures.farmer_reward_address_override`, with no validation: [3](#0-2) 

This override is then placed directly into `DeclareProofOfSpace.farmer_reward_puzzle_hash`: [4](#0-3) 

The `RespondSignatures` protocol message definition confirms the field is an unconstrained optional `bytes32`: [5](#0-4) 

There is no check that:
- The override address belongs to the harvester
- The fee quality threshold (`fee_quality <= fee_threshold`) was actually satisfied
- The override in `RespondSignatures` matches the one declared in `NewProofOfSpace`
- The address is in any allowlist

### Impact Explanation

**Critical — Unauthorized reward diversion of XCH.** A malicious third-party harvester can redirect the farmer's 1/8 block reward (currently ~0.25 XCH per block won) to any attacker-controlled puzzle hash. The farmer loses their reward on every block won by plots managed by the malicious harvester. This is a direct, on-chain, irreversible theft of XCH.

### Likelihood Explanation

Third-party harvesters are an explicitly supported and documented use case (CHIP-22). Farmers routinely connect to external harvesters they do not fully control. Any such harvester can exploit this with zero additional privileges — it only needs a valid proof of space, which it legitimately possesses.

### Recommendation

In `_process_respond_signatures`, before accepting `farmer_reward_address_override`:

1. **Enforce fee quality**: Recompute `calculate_harvester_fee_quality` and verify `fee_quality <= applied_fee_threshold` from the original `NewProofOfSpace.fee_info`. If the threshold is not met, reject the override and use `self.farmer.farmer_target`.
2. **Cross-check consistency**: Verify that `RespondSignatures.farmer_reward_address_override` matches the value declared in the corresponding `NewProofOfSpace.farmer_reward_address_override` (stored alongside the proof in `self.farmer.proofs_of_space`).
3. **Optionally allowlist**: Allow farmers to configure a set of trusted harvester reward addresses.

### Proof of Concept

1. A malicious harvester connects to a farmer's harvester port.
2. It finds a valid proof of space for a block-winning quality.
3. It sends `NewProofOfSpace` with `farmer_reward_address_override = attacker_puzzle_hash` and `fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=0xFFFFFFFF)` (maximum threshold, which will never be met).
4. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, computes `fee_quality`, sees `fee_quality > fee_threshold`, and **logs a warning only** — no abort.
5. The farmer sends `RequestSignatures` to the harvester.
6. The harvester responds with `RespondSignatures(farmer_reward_address_override=attacker_puzzle_hash, ...)`.
7. `_process_respond_signatures` sets `farmer_reward_address = attacker_puzzle_hash` at line 918 with no further checks.
8. `DeclareProofOfSpace` is sent to the full node with `farmer_reward_puzzle_hash = attacker_puzzle_hash`.
9. The block is created; the farmer reward coin is created at `attacker_puzzle_hash`. The farmer receives nothing. [6](#0-5) [7](#0-6)

```python
    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
```
