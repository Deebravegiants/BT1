Let me check the `OmniAddress` and `ChainKind` types to understand whether different EVM chains are distinguished.

### Title
Missing `emitter_chain` Validation in Wormhole VAA Allows Cross-Chain Factory Spoofing — (`near/omni-prover/wormhole-omni-prover-proxy/src/parsed_vaa.rs`)

---

### Summary

The NEAR `wormhole-omni-prover-proxy` parses the `emitter_chain` field from a VAA header but **never validates it** against any expected Wormhole chain ID. The `emitter_address` placed into `InitTransferMessage` is constructed using the chain tag from the attacker-controlled **payload**, not from the VAA header. An attacker who controls a contract at the same address as a registered EVM factory on a different chain can emit a guardian-signed VAA from that chain, craft the payload to claim a different chain's identity, and have NEAR credit tokens as if the transfer originated from the legitimate factory.

---

### Finding Description

**Step 1 — VAA parsing: `emitter_chain` is read but never used for validation.**

In `parsed_vaa.rs`, `ParsedVAA::parse` extracts `emitter_chain` (u16) from the VAA body: [1](#0-0) 

The field is stored in the struct but is never referenced again in any validation path.

**Step 2 — `verify_vaa_callback` only checks `proof_kind`, not `emitter_chain`.** [2](#0-1) 

The only post-guardian-verification check is that `proof_kind` matches `payload[0]`. `emitter_chain` is silently ignored.

**Step 3 — `emitter_address` chain identity is derived from the payload, not the VAA header.**

In the `TryInto<InitTransferMessage>` conversion: [3](#0-2) 

`transfer.token_address.get_chain()` comes from the Borsh-decoded payload field — fully attacker-controlled. `self.emitter_address` is the raw 32-byte emitter address from the VAA header. The resulting `OmniAddress` therefore carries the **payload's chain tag** combined with the **VAA's emitter bytes**, with no cross-check between the two.

**Step 4 — The factory check in `fin_transfer_callback` uses the chain derived in Step 3.** [4](#0-3) 

`emitter_address.get_chain()` is the chain from the payload. If the attacker sets `token_address` to claim `ChainKind::Eth`, the lookup is `factories[Eth]`. If the raw emitter bytes in the VAA match the registered Ethereum factory address, the check passes — regardless of which Wormhole chain actually emitted the VAA.

**Step 5 — `ChainKind` has distinct per-chain variants, each with its own factory.** [5](#0-4) 

`Eth`, `Bnb`, `Arb`, `Base`, `Pol`, `HyperEvm`, `Abs` are all separate keys in the `factories` map. Each chain's factory is registered independently. The attack exploits the fact that the same 20-byte address can be deployed on multiple chains (common with CREATE2 or deterministic deployers), while NEAR only checks the address bytes, not which Wormhole chain signed the emission.

---

### Impact Explanation

An attacker who controls a contract at the same address as the registered Ethereum factory on BNB (or any other supported EVM chain) can:

1. Emit a Wormhole message from BNB — guardians sign it legitimately.
2. Craft the payload with `omniBridgeChainId = Eth` and a valid Ethereum token address.
3. Submit the VAA to NEAR `fin_transfer`.
4. NEAR constructs `emitter_address = OmniAddress::Eth(factory_bytes)`, which matches `factories[Eth]`.
5. NEAR credits the attacker with Ethereum-backed wrapped tokens without any actual lock on Ethereum.

This is unauthorized creation of wrapped assets — **Critical** under the allowed impact scope.

---

### Likelihood Explanation

The precondition — a contract at the same address as the Ethereum factory on another supported EVM chain — is realistic because:

- Bridge factories are routinely deployed at identical addresses across chains using CREATE2 with the same salt and deployer.
- An attacker can front-run factory deployment on a newly supported chain.
- The attacker does not need any privileged role; Wormhole guardian signing is permissionless for any on-chain emitter.

---

### Recommendation

In `verify_vaa_callback`, after parsing the VAA, validate `parsed_vaa.emitter_chain` against a stored mapping of `ChainKind → expected Wormhole chain ID`. Reject any VAA whose `emitter_chain` does not match the Wormhole chain ID corresponding to the chain derived from the payload. Concretely:

```rust
// pseudocode
let expected_wormhole_chain_id = self.chain_to_wormhole_id
    .get(&payload_chain)
    .ok_or("UnknownChain")?;
require!(
    parsed_vaa.emitter_chain == expected_wormhole_chain_id,
    "emitter_chain mismatch"
);
```

The proxy contract must store a `ChainKind → u16` mapping populated at initialization.

---

### Proof of Concept

1. Ethereum factory deployed at `0xF00...` (registered in NEAR as `factories[Eth] = OmniAddress::Eth(0xF00...)`).
2. Attacker deploys a contract at `0xF00...` on BNB (same address, e.g., via CREATE2).
3. Attacker calls `publishMessage` from `0xF00...` on BNB with payload:
   - `payload[0]` = `0x00` (InitTransfer)
   - `sender` = `OmniAddress::Bnb(attacker)`
   - `token_address` = `OmniAddress::Eth(some_registered_eth_token)` ← chain tag = Eth
   - `amount`, `recipient`, etc. = attacker-chosen values
4. Wormhole guardians sign the VAA; `emitter_chain = 4` (BNB), `emitter_address = 0xF00...`.
5. Attacker submits VAA to NEAR `fin_transfer` with `chain_kind = Eth`.
6. `verify_vaa_callback` passes (guardian sigs valid, `payload[0]` matches `ProofKind::InitTransfer`).
7. `emitter_address` is constructed as `OmniAddress::Eth(0xF00...)` (chain from payload, bytes from VAA header).
8. `factories[Eth] == OmniAddress::Eth(0xF00...)` → check passes.
9. NEAR mints/releases Ethereum-backed tokens to the attacker with no Ethereum lock.
