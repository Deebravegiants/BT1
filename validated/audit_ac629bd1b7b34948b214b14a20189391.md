### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Creates Unbacked Wrapped Supply — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer` emits `InitTransfer` with the caller-supplied `amount` without verifying the actual ERC20 balance received. For fee-on-transfer tokens, the bridge holds `amount − transferFee` but the NEAR side mints/releases `amount`, permanently breaking the 1:1 backing guarantee.

### Finding Description
In `initTransfer` (lines 373–437 of `OmniBridge.sol`), when `tokenAddress` is neither a bridge token nor a custom-minter token, the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
```

followed immediately by:

```solidity
emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, currentOriginNonce, amount, fee, nativeFee, recipient, message);
``` [1](#0-0) [2](#0-1) 

No balance-before/balance-after check is performed. For a fee-on-transfer ERC20 (e.g., a 1% transfer-fee token), the bridge receives `amount × 0.99` but the event records `amount`. The NEAR bridge's `fin_transfer_callback` reads the `InitTransfer` proof and uses the `amount` field from the event to determine how many tokens to mint or release to the recipient on NEAR. [3](#0-2) 

The `logMetadata` function is publicly callable by any address with no access restriction, so any ERC20 token — including fee-on-transfer tokens — can be registered with the bridge. [4](#0-3) 

### Impact Explanation
Every `initTransfer` call with a fee-on-transfer token causes the EVM bridge to hold `amount − fee` tokens while the NEAR side mints `amount` tokens. The wrapped supply on NEAR exceeds the locked collateral on EVM by `fee` per transaction. An attacker (or any user) repeatedly bridging a fee-on-transfer token gradually drains the EVM bridge's reserves for that token. When other users attempt to bridge back their NEAR-side tokens, the EVM bridge cannot release the full amount, resulting in irreversible fund lock or failed redemptions for legitimate users. This is unauthorized creation of wrapped bridge assets and a direct backing-guarantee violation — **Critical** under the allowed impact scope.

### Likelihood Explanation
`logMetadata` is permissionless. Any fee-on-transfer ERC20 token can be registered. Once registered and the NEAR-side token is deployed (via the normal relayer flow), any unprivileged user calling `initTransfer` with that token triggers the discrepancy. No special privileges, leaked keys, or colluding parties are required. Likelihood is **Medium-High** given the prevalence of fee-on-transfer tokens in the wild.

### Recommendation
Use a balance-before/balance-after pattern to determine the actual amount received, and use that value in the event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "Zero received");
// use actualReceived (cast to uint128) in the event and downstream logic
```

### Proof of Concept
1. Deploy a fee-on-transfer ERC20 token `FeeToken` with a 1% transfer fee.
2. Call `OmniBridge.logMetadata(feeTokenAddress)` — succeeds with no restriction.
3. Relayer processes the `LogMetadata` event; NEAR side deploys the wrapped token.
4. Attacker calls `OmniBridge.initTransfer(feeTokenAddress, 1000e18, 0, 0, nearRecipient, "")`.
   - Bridge receives `990e18` tokens (`safeTransferFrom` deducts 1% fee).
   - Event emits `amount = 1000e18`.
5. NEAR bridge processes the proof and mints `1000e18` wrapped tokens to `nearRecipient`.
6. `nearRecipient` bridges back `1000e18` tokens to EVM via `fin_transfer`.
7. EVM bridge attempts to release `1000e18` but only holds `990e18` → shortfall of `10e18`.
8. Repeated iterations drain the bridge's `FeeToken` reserves entirely, locking all other users' funds.

    pub fn fin_transfer_callback(
        &mut self,
        #[serializer(borsh)] storage_deposit_actions: &Vec<StorageDepositAction>,
        #[serializer(borsh)] predecessor_account_id: AccountId,
    ) -> PromiseOrValue<Nonce> {
        let Ok(ProverResult::InitTransfer(init_transfer)) = Self::decode_prover_result(0) else {
            env::panic_str(BridgeError::InvalidProofMessage.to_string().as_str())
        };
        require!(
            self.factories
                .get(&init_transfer.emitter_address.get_chain())
                == Some(init_transfer.emitter_address),
            BridgeError::UnknownFactory.as_ref()
        );

        let decimals = self
            .token_decimals
            .get(&init_transfer.token)
            .near_expect(BridgeError::TokenDecimalsNotFound);

        let destination_nonce =
            self.get_next_destination_nonce(init_transfer.recipient.get_chain());
        let transfer_message = TransferMessage {
            origin_nonce: init_transfer.origin_nonce,
            token: init_transfer.token,
            amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
            recipient: init_transfer.recipient,
            fee: Self::denormalize_fee(&init_transfer.fee, decimals),
            sender: init_transfer.sender,
            msg: init_transfer.msg,
            destination_nonce,
            origin_transfer_id: None,
        };

        if let OmniAddress::Near(recipient) = transfer_message.recipient.clone() {
            self.process_fin_transfer_to_near(
                recipient,
                &predecessor_account_id,
                transfer_message,
                storage_deposit_actions,
            )
            .into()
        } else {
            self.process_fin_transfer_to_other_chain(predecessor_account_id, transfer_message);
            PromiseOrValue::Value(destination_nonce)
        }
    }
