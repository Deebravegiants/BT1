### Title
Fee-on-Transfer Token Backing Divergence in `initTransfer` Inflates NEAR Mint Amount Beyond Actual Locked Collateral - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.initTransfer` performs a `safeTransferFrom` for the caller-supplied `amount` but emits the `InitTransfer` event with that same nominal `amount` regardless of how many tokens the bridge actually received. For fee-on-transfer (FoT) tokens, the bridge receives `amount × (1 − fee_rate)` while the event records `amount`. The NEAR side reads the event and mints the full nominal `amount` to the recipient, creating unbacked wrapped supply. The same pattern exists in the Starknet bridge.

### Finding Description

In `OmniBridge.initTransfer`, the non-bridge-token path locks tokens via:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // nominal amount requested
);
```

Immediately after, the event is emitted with the same nominal value:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,         // ← nominal, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [1](#0-0) [2](#0-1) 

The NEAR bridge's `fin_transfer` path reads the `amount` field from this event (via the prover) and mints that full amount to the recipient. No balance-before/after check is performed on the EVM side to measure actual receipt. [3](#0-2) 

The NEAR `fin_transfer_send_tokens_callback` distributes `amount_without_fee()` to the recipient and `fee.fee` to the relayer — both derived from the event's nominal `amount`, not from what the EVM bridge actually holds. [4](#0-3) 

The identical pattern exists in the Starknet bridge:

```cairo
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
// ...
self.emit(Event::InitTransfer(InitTransfer { ..., amount, ... }))
``` [5](#0-4) 

### Impact Explanation

Every `initTransfer` with a FoT token creates a shortfall: the EVM bridge holds `amount × (1 − fee_rate)` tokens but NEAR mints `amount` wrapped tokens. The shortfall compounds with each transfer. When wrapped-token holders later bridge back (NEAR → EVM `finTransfer`), the EVM bridge attempts `safeTransfer(recipient, amount)` from its locked balance. The last redeemers cannot withdraw because the bridge's actual balance is insufficient — their redemption path is permanently frozen for the deficit amount. This breaks the 1:1 backing guarantee and constitutes an irreversible fund lock for the final claimants. [6](#0-5) 

### Likelihood Explanation

Any unprivileged user can call `initTransfer` with any ERC-20 address that is not a bridge token and not a custom minter — there is no whitelist on the token address for the locked-token path. FoT tokens (e.g. tokens with a built-in transfer tax) are a well-known token class. A single transfer with such a token is sufficient to begin the undercollateralization. No privileged role, leaked key, or external dependency compromise is required. [7](#0-6) 

### Recommendation

Measure the actual received amount using a balance-before/after pattern in the locked-token path:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "FEE_ON_TRANSFER_NOT_SUPPORTED");
// or: use actualReceived as the amount in the event
```

Either reject FoT tokens outright (simplest, matches the bridge's 1:1 backing invariant) or propagate `actualReceived` into the event and all downstream accounting. Apply the same fix to the Starknet `init_transfer`.

### Proof of Concept

1. Deploy a standard ERC-20 with a 1% transfer tax (FoT token). It is not registered as a bridge token or custom minter.
2. Call `OmniBridge.initTransfer(fotToken, 10_000, 0, 0, "alice.near", "")`.
3. `safeTransferFrom` deducts 1% tax; bridge receives **9,900** tokens. Event emits `amount = 10_000`.
4. NEAR prover parses the event; NEAR bridge mints **10,000** wrapped tokens to Alice.
5. Alice bridges back: NEAR burns 10,000 wrapped tokens; EVM `finTransfer` attempts `safeTransfer(alice, 10_000)`.
6. If the bridge's balance is exactly 9,900 (only one such transfer ever occurred), the `safeTransfer` reverts — Alice's 10,000 wrapped tokens are permanently unclaimable.
7. Repeated transfers compound the deficit, locking out an increasing fraction of wrapped-token holders. [8](#0-7)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
        );
    }
