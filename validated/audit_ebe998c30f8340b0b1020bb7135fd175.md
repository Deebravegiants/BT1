### Title
Unbounded `feeBps` in `RSETHPool`, `RSETHPoolNoWrapper`, and `AGETHPoolV3` allows 100% deposit fee, causing depositors to receive zero rsETH/agETH — (File: contracts/pools/RSETHPool.sol)

---

### Summary

`RSETHPool.setFeeBps`, `RSETHPool.setTokenFeeBps`, `RSETHPoolNoWrapper.setFeeBps`, and `AGETHPoolV3.setFeeBps` all permit fees up to 10,000 BPS (100%). The protocol's own newer contract, `RSETHPoolV3`, already corrected this by capping `feeBps` at 1,000 BPS (10%), demonstrating that 100% is not an intended upper bound. If the admin sets `feeBps = 10_000` — even by a simple typo (e.g., entering `10000` instead of `100` for 1%) — every subsequent depositor receives 0 rsETH/agETH while their entire ETH or token deposit is silently accumulated as protocol fees and later extracted by the BRIDGER_ROLE.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

Any depositor calling `deposit()` after `feeBps` is set to 10,000 BPS loses 100% of their deposited ETH or tokens. They receive 0 rsETH/agETH in return. The entire deposited value accumulates in `feeEarnedInETH` / `feeEarnedInToken` and is withdrawable by the BRIDGER_ROLE to an arbitrary receiver. There is no mechanism for the depositor to recover their funds once the transaction is confirmed.

---

### Likelihood Explanation

**Low.** The admin must set `feeBps` to an excessively high value. The realistic scenario is a typo: entering `10000` (100%) instead of `100` (1%) or `1000` (10%). The protocol itself already recognized this risk and applied a 1,000 BPS cap in `RSETHPoolV3`, but failed to backport the fix to `RSETHPool`, `RSETHPoolNoWrapper`, and `AGETHPoolV3`. The absence of an on-chain guard means no transaction-level protection exists against this class of misconfiguration.

---

### Recommendation

Apply the same 1,000 BPS (10%) upper bound already used in `RSETHPoolV3.setFeeBps` to all affected setters:

- `RSETHPool.setFeeBps` and `RSETHPool.setTokenFeeBps` — `contracts/pools/RSETHPool.sol`
- `RSETHPoolNoWrapper.setFeeBps` — `contracts/pools/RSETHPoolNoWrapper.sol`
- `AGETHPoolV3.setFeeBps` — `contracts/agETH/AGETHPoolV3.sol`

Additionally, consider a timelock on fee changes so existing depositors can observe and react before a new fee takes effect.

---

### Proof of Concept

1. Admin calls `RSETHPool.setFeeBps(10_000)` — intending 1% (100 BPS) but entering the raw denominator by mistake.
2. User calls `RSETHPool.deposit{value: 1 ether}("ref")`.
3. `viewSwapRsETHAmountAndFee(1 ether)` computes:
   - `fee = 1e18 * 10_000 / 10_000 = 1e18`
   - `amountAfterFee = 1e18 - 1e18 = 0`
   - `rsETHAmount = 0 * 1e18 / rsETHToETHrate = 0`
4. `IERC20(wrsETH).safeTransfer(user, 0)` — user receives nothing.
5. `feeEarnedInETH += 1e18` — the full 1 ETH is recorded as fees.
6. BRIDGER_ROLE calls `withdrawFees(receiver)` — 1 ETH is transferred out of the pool to an arbitrary address.

The depositor has permanently lost 1 ETH with no recourse. The same path applies to token deposits via `setTokenFeeBps` and to `AGETHPoolV3` / `RSETHPoolNoWrapper` via their respective `setFeeBps`.
