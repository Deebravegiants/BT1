### Title
Unbounded `fee_proposal_fri` During SNIP-35 Startup Window Permanently Anchors Fee Floor at Attacker-Chosen Value - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

During the first `fee_proposal_window_size` (= 10) blocks, `is_proposal_init_valid` skips the `fee_proposal_fri` bounds check entirely because `fee_actual` is `None`. A malicious proposer can set `fee_proposal_fri = 1` (or any arbitrarily low value) during this window; validators accept it unconditionally. Once the window fills, `fee_actual = 1 FRI`, and the SNIP-35 fee floor mechanism is permanently disabled: future proposals are constrained to `fee_proposal_fri ∈ {0, 1}` (within 0.2% of 1), making the SNIP-35 floor permanently inoperative. This is the direct Sequencer analog of the English auction's "first bid can be any small amount > 0" invariant — the startup window is the unconstrained first-bid phase, and the `margin_ppt` ratchet is the 1.05× bid-increment rule that locks in whatever value was set during that phase.

---

### Finding Description

In `is_proposal_init_valid`, the bounds check for `fee_proposal_fri` is gated on `fee_actual` being `Some`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
``` [1](#0-0) 

`fee_actual` is `None` for the first `fee_proposal_window_size` (= 10) blocks because `compute_fee_actual` returns `None` when `height < window_size`:

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ({window_size})");
    return None;
};
``` [2](#0-1) 

The versioned constants set `fee_proposal_window_size = 10` and `fee_proposal_margin_ppt = 2` (parts per thousand): [3](#0-2) 

During the startup window, any `fee_proposal_fri` value (including `GasPrice(1)` or `GasPrice(0)`) passes `is_proposal_init_valid` without rejection. The accepted value is then recorded into `fee_proposals_window` at decision time:

```rust
self.record_fee_proposal(height, init.fee_proposal_fri);
``` [4](#0-3) 

Once the window contains 10 entries of `GasPrice(1)`, `compute_fee_actual` returns `Some(GasPrice(1))`. The `fee_proposal_bounds` function then computes:

- `upper = 1 * (1000 + 2) / 1000 = 1` (integer truncation)
- `lower = 1 * 1000 / (1000 + 2) = 0` (integer truncation) [5](#0-4) 

Every subsequent proposal must have `fee_proposal_fri ∈ {0, 1}`. The SNIP-35 ratchet is permanently stuck. The manipulated `fee_proposal_fri` is also bound into the proposal commitment hash:

```rust
ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
``` [6](#0-5) 

This means every committed block header carries `fee_proposal_fri = 1` and the consensus commitment is `Poseidon(partial_block_hash, 1)` — a permanently wrong value that all validators sign over.

---

### Impact Explanation

**Broken invariant:** `fee_proposal_fri` is supposed to be bounded by the market rate via the SNIP-35 mechanism. During the startup window this invariant is not enforced.

**Exact corrupted value:** `fee_proposal_fri = 1 FRI` (or any attacker-chosen value) in every block header and proposal commitment for the lifetime of the chain.

**Fee market effect:** `calculate_next_l2_gas_price_for_fin` uses `effective_min = max(config_min, fee_actual)`:

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),
    None => config_min,
};
``` [7](#0-6) 

With `fee_actual = 1` and `config_min = 8_000_000_000 FRI`, `effective_min = config_min`. The EIP-1559 price floor is still protected by `config_min`. However, the SNIP-35 mechanism that was designed to raise the floor above `config_min` based on STRK/USD market conditions is permanently disabled. The gas price can go as low as `config_min` even when market conditions indicate it should be significantly higher, allowing users to underpay for transactions and enabling sustained spam at below-market rates.

Additionally, the `fee_proposal_fri` field stored in `BlockHeaderWithoutHash` and committed into the proposal commitment hash is permanently wrong — every block from height 10 onward carries a manipulated value that consensus has signed over.

**Impact category:** High — incorrect fee/gas accounting with economic impact; authoritative-looking wrong value in block headers and proposal commitments.

---

### Likelihood Explanation

The attack window is the first 10 blocks of the network (genesis). The attacker must be selected as proposer for a majority of those blocks. In a permissionless validator set, any party with sufficient stake can achieve this. The attack is a one-time action at genesis with permanent effect; there is no recovery path without a hard fork or versioned-constants upgrade.

---

### Recommendation

Enforce a minimum value for `fee_proposal_fri` even during the startup window. The simplest fix is to require `fee_proposal_fri >= l2_gas_price` (the current EIP-1559 price, which is already bounded by `config_min`) when `fee_actual` is `None`:

```rust
// During initiation (fee_actual is None), enforce a floor of l2_gas_price.
if let Some(fee_proposal) = init_proposed.fee_proposal_fri {
    let floor = proposal_init_validation.l2_gas_price_fri;
    if fee_proposal < floor {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!("fee_proposal {fee_proposal:?} below startup floor {floor:?}"),
        ));
    }
}
```

Alternatively, seed `fee_proposals_window` with `window_size` copies of `l2_gas_price` at genesis so `fee_actual` is never `None` and the bounds check always applies.

---

### Proof of Concept

1. Deploy a new Starknet network (`fee_proposal_window_size = 10`, `fee_proposal_margin_ppt = 2`, `min_gas_price = 8_000_000_000 FRI`).
2. A malicious validator is selected as proposer for blocks 0–9.
3. For each block, the proposer emits `ProposalInit { fee_proposal_fri: Some(GasPrice(1)), ... }`.
4. Each validator node calls `is_proposal_init_valid`. Because `compute_fee_actual` returns `None` for heights 0–9, the `if let (Some(fee_actual), Some(fee_proposal))` guard is false and the bounds check is skipped. The proposal is accepted.
5. At decision time, `record_fee_proposal(height, Some(GasPrice(1)))` is called for each of the 10 blocks.
6. At height 10, `compute_fee_actual` returns `Some(GasPrice(1))` (median of ten `1`s).
7. `fee_proposal_bounds(GasPrice(1), 2)` returns `(lower=0, upper=1)`.
8. Any honest proposer at height 10 who tries to set `fee_proposal_fri = 8_000_000_000` (the market rate) is rejected by validators with "Fee proposal out of bounds: fee_actual=1, fee_proposal=8000000000, allowed range=[0, 1]".
9. The SNIP-35 floor is permanently stuck at 1 FRI. Every block header carries `fee_proposal_fri = 1` and the proposal commitment is `Poseidon(partial_block_hash, 1)` — a permanently wrong value signed by all validators.

pub(crate) fn fee_proposal_bounds(fee_actual: GasPrice, margin_ppt: u128) -> (u128, u128) {
    let denom = U256::from(PPT_DENOMINATOR);
    let scaled = denom + U256::from(margin_ppt);
    let fee_actual_u256 = U256::from(fee_actual.0);
    let upper = u128::try_from(fee_actual_u256 * scaled / denom).unwrap_or(u128::MAX);
    let lower = u128::try_from(fee_actual_u256 * denom / scaled).unwrap_or(0);
    (lower, upper)
}
