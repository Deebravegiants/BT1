### Title
Missing RSA Modulus Primality Validation in `base_pke_rsa_ek_from_modulus` Allows PVE Ciphertext Decryption by Attacker-Supplied Weak Key — (`src/cbmpc/api/pve_base_pke.cpp`)

### Summary

`base_pke_rsa_ek_from_modulus` accepts any 256-byte non-zero value as a valid RSA-2048 modulus. An attacker who can supply the modulus (e.g., as a malicious serialized-input provider or via a compromised/substituted HSM export) can provide a smooth composite that OpenSSL accepts for public-key construction and encryption, then factor it offline to recover the RSA-OAEP KEM shared secret and decrypt the PVE backup ciphertext.

### Finding Description

The function `base_pke_rsa_ek_from_modulus` (C++ API) and its C wrapper `cbmpc_pve_base_pke_rsa_ek_from_modulus` are public APIs designed for HSM integration, where only the raw modulus is exported. The validation is:

```cpp
if (modulus.size != kExpectedModulusBytes)
    return coinbase::error(E_BADARG, "modulus must be exactly 256 bytes (RSA-2048)");
// ...
if (BN_is_zero(n_bn)) {
    BN_free(n_bn);
    return coinbase::error(E_BADARG, "modulus must not be zero");
}
```

No further checks are performed. The modulus is passed directly to `rsa_pub_key_t::set` → `EVP_PKEY_fromdata(..., EVP_PKEY_PUBLIC_KEY, ...)`. OpenSSL does not perform primality checks when constructing a public key from raw parameters. The resulting `rsa_pub_key_t` is wrapped in an `ek_blob` and returned as a valid encryption key.

When `encrypt(curve, ek, label, x, out_ciphertext)` is subsequently called with this blob, the PVE layer calls `pve_base_pke_rsa().encrypt(...)` which calls `rsa_pub_key_t::encrypt_oaep` → `encrypt_raw` → `EVP_PKEY_encrypt` with `RSA_NO_PADDING`. This performs `m^65537 mod n` over the attacker-supplied composite, producing a ciphertext that is structurally valid but cryptographically weak.

### Impact Explanation

The PVE ciphertext's RSA-OAEP layer is encrypted under a modulus the attacker chose. If the attacker chose a smooth composite (e.g., a product of primes each ≤ 2^20), they can:
1. Factor `n` in milliseconds using trial division or Pollard's rho.
2. Compute `phi(n)` and then `d = 65537^{-1} mod phi(n)`.
3. Decrypt the RSA-OAEP ciphertext to recover the KEM shared secret.
4. Use the shared secret to decrypt the DEM layer and recover the scalar `x` (the backed-up key share).

This fully breaks the confidentiality guarantee of the PVE backup ciphertext.

### Likelihood Explanation

The function is explicitly designed for HSM integration where the modulus comes from an external system. Any path where the modulus is received from an untrusted or attacker-influenced source (network, user input, compromised HSM firmware, MITM on the HSM export channel) is exploitable. The API is public and documented, making it a natural integration point for attacker-controlled input.

### Recommendation

Add RSA modulus validity checks before constructing the public key:
- Reject even moduli (`BN_is_odd` check).
- Reject moduli with small prime factors (trial division up to at least 2^16 or use `BN_check_prime` / Miller-Rabin with sufficient rounds — noting that for a public key, a probabilistic primality test on `n` itself is not meaningful, but checking that `n` has no small factors is).
- Verify the modulus bit-length is exactly 2048 bits (not just byte-length, to catch leading-zero padding).
- Optionally, require the caller to provide a proof of knowledge of the factorization (i.e., require the private key or a ZK proof that the modulus is a valid RSA modulus), which is the strongest mitigation.

At minimum, reject moduli divisible by any prime below a threshold (e.g., 65537) and reject even moduli.

### Proof of Concept

```cpp
// Build a smooth 256-byte composite: n = 3 * 5 * 7 * 11 * ... (product of small primes, padded to 256 bytes)
// 1. Construct n as a product of small primes with total bit-length ~2048.
// 2. Call cbmpc_pve_base_pke_rsa_ek_from_modulus(n_bytes, &ek) -> SUCCESS
// 3. Call cbmpc_pve_encrypt(NULL, CURVE_SECP256K1, ek, label, x_bytes, &ct) -> SUCCESS
// 4. Factor n offline (trivial for smooth composite).
// 5. Compute phi(n), d = 65537^{-1} mod phi(n).
// 6. Decrypt the RSA-OAEP layer of ct using d to recover the KEM shared secret.
// 7. Use the shared secret to decrypt the DEM layer and recover x.
// Assert: recovered x == original x.
```

The `base_pke_rsa_ek_from_modulus` call at step 2 returns `SUCCESS` with no error, and the `encrypt` call at step 3 produces a well-formed ciphertext, so honest code has no indication that the key is weak.

error_t rsa_pub_key_t::encrypt_raw(mem_t in, buf_t& out) const {
  int n_size = size();
  if (n_size != in.size) return coinbase::error(E_CRYPTO);

  scoped_ptr_t<EVP_PKEY_CTX> ctx = EVP_PKEY_CTX_new(ptr, NULL);
  if (EVP_PKEY_encrypt_init(ctx) <= 0) return openssl_error("RSA encrypt RAW error");
  if (EVP_PKEY_CTX_set_rsa_padding(ctx, RSA_NO_PADDING) <= 0) return openssl_error("RSA encrypt RAW error");
  size_t outlen = n_size;
  if (EVP_PKEY_encrypt(ctx, out.alloc(n_size), &outlen, in.data, in.size) <= 0)
    return openssl_error("RSA encrypt RAW error");
  return SUCCESS;
}
