### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Enables Stale-Response Replay by a Single Malicious Attested Participant — (`crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `respond_verify_foreign_tx` endpoint accepts any `(request, response)` pair where the signature is valid against the root public key and the `request` key matches a pending queue entry. Because `VerifyForeignTransactionRequest` contains no nonce, receipt-ID, or timestamp, a previously valid on-chain response can be replayed verbatim by a single malicious attested participant for any subsequent submission of the same foreign-transaction request. The victim receives a stale signed attestation — one that may reflect a since-reorged chain state — without the MPC network re-verifying the current foreign-chain state.

---

### Finding Description

**Root cause — caller-agnostic, instance-free request key**

`VerifyForeignTransactionRequest`, the map key used by `pending_verify_foreign_tx_requests`, contains only the chain-level transaction content:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,   // tx_id, confirmations, extractors
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [1](#0-0) 

No caller account ID, receipt ID, block height, or nonce is included. This is explicitly confirmed by the test comment: *"both yields are queued under the single (caller-agnostic) request key."* [2](#0-1) 

**Root cause — `respond_verify_foreign_tx` does not bind response to request instance**

The function verifies only that the signature is valid over `response.payload_hash` against the root public key. It does **not** verify that `payload_hash` was freshly computed for this specific queue entry, nor that it corresponds to the current foreign-chain state:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
``` [3](#0-2) 

After this check passes, `resolve_yields_for` drains the queue and delivers `response` to all waiting callers: [4](#0-3) 

**Attack path**

1. Alice calls `verify_foreign_transaction(bitcoin_tx_X, confirmations=2)` at block T₁.
2. MPC nodes verify the transaction (confirmed in block B₁) and call `respond_verify_foreign_tx(request={bitcoin_tx_X, …}, response={payload_hash₁, sig₁})` where `payload_hash₁ = hash(bitcoin_tx_X, block_hash_B₁, …)`.
3. Alice's queue entry is drained; the signed attestation is publicly visible on-chain.
4. A Bitcoin reorg occurs: `bitcoin_tx_X` is now in a different block B₂ (or absent).
5. Bob calls `verify_foreign_transaction(bitcoin_tx_X, confirmations=2)` at block T₂, creating a new queue entry under the **same** key.
6. A single malicious attested participant replays the old `respond_verify_foreign_tx(request, response₁)` call before the honest MPC nodes produce a fresh response.
7. The contract accepts it: signature is valid against the root key, request key matches the new queue entry.
8. Bob receives `(payload_hash₁, sig₁)` — a stale attestation asserting `bitcoin_tx_X` was confirmed in block B₁, which no longer exists.

The node-side `SignRequestStorage.add()` deduplicates by receipt ID: [5](#0-4) 

But this only prevents the honest nodes from processing the same receipt twice; it does not prevent a malicious participant from replaying a previously submitted `respond_verify_foreign_tx` transaction.

---

### Impact Explanation

A single Byzantine attested participant (below signing threshold) can deliver a stale MPC-signed attestation to any caller of `verify_foreign_transaction`. The attestation carries a valid threshold signature over a `payload_hash` that encodes chain state from a prior block. Downstream bridge or settlement contracts that rely on this attestation to authorize fund releases or cross-chain actions may execute against a reorged or otherwise invalid foreign-chain state, enabling double-spend or invalid bridge execution.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

Likelihood is low. Exploitation requires: (a) a reorg on the foreign chain after a prior verification, (b) the same transaction being re-submitted for verification, and (c) a malicious attested participant willing to replay the old response before honest nodes respond. Bitcoin shallow-reorg risk is real but infrequent; the window between Bob's submission and the honest nodes' fresh response is narrow. However, the attack requires only one compromised TEE node and no threshold collusion, making it structurally reachable.

---

### Recommendation

Bind each `respond_verify_foreign_tx` response to a specific request instance by including a unique identifier (e.g., the NEAR receipt ID of the originating `verify_foreign_transaction` call) in both the pending-queue key and the signed payload:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    pub receipt_id: CryptoHash,   // <-- add this
}
```

The MPC nodes already have access to the receipt ID via `VerifyForeignTxRequestFromChain.receipt_id`: [6](#0-5) 

Including `receipt_id` in the signed payload and the map key ensures that a response produced for one submission cannot be replayed for any other submission, even if the foreign-transaction content is identical.

---

### Proof of Concept

The existing unit test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` demonstrates that two different callers (Alice and Bob) are queued under the **same** key: [7](#0-6) 

A replay attack exploits this: after Alice's queue entry is drained by a legitimate `respond_verify_foreign_tx`, Bob's new submission creates a fresh entry under the identical key. The old `(request, response)` pair — publicly visible on-chain — passes all checks in `respond_verify_foreign_tx` (valid signature, matching key) and drains Bob's entry with stale data, without the MPC network re-verifying the current foreign-chain state. [8](#0-7)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
