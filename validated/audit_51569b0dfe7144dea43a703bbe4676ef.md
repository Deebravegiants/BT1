### Title
Single Byzantine MPC Participant Can Persistently Disrupt All Signing Requests Without Being Identified — (`File: crates/threshold-signatures/src/frost/eddsa/sign.rs`, `crates/threshold-signatures/src/frost/redjubjub/sign.rs`)

---

### Summary

The FROST/EdDSA and FROST/RedJubJub signing coordinators collect signature shares from all participants but explicitly disable per-share verification (`CheaterDetection::Disabled`) and pass an empty verifying-shares map. A single Byzantine participant strictly below the signing threshold can send a garbage signature share, causing every signing request that includes them to fail permanently. Because cheater detection is disabled, the malicious participant is never identified and cannot be excluded from future rounds, making the disruption persistent and undetectable at the protocol level.

---

### Finding Description

In both `do_sign_coordinator_v1` and `do_sign_coordinator_v2` in `crates/threshold-signatures/src/frost/eddsa/sign.rs`, and in `do_sign_coordinator` in `crates/threshold-signatures/src/frost/redjubjub/sign.rs`, the coordinator:

1. Calls `recv_from_others` to collect a signature share from **every** participant in the signing set (not just a threshold-sized subset).
2. Constructs a `PublicKeyPackage` with an **empty** `BTreeMap` for verifying shares: `PublicKeyPackage::new(BTreeMap::new(), vk_package, None)`.
3. Calls `aggregate_custom(..., CheaterDetection::Disabled)`, which skips per-share verification entirely.
4. Only verifies the **final aggregated** signature. [1](#0-0) [2](#0-1) [3](#0-2) 

The `recv_from_others` helper waits for exactly one message from **every** participant before proceeding — it does not stop at a threshold-sized quorum: [4](#0-3) 

Because the verifying-shares map is empty and `CheaterDetection::Disabled` is set, the `aggregate_custom` call cannot attribute a bad share to any specific participant. The code comment itself acknowledges this design choice:

> "We supply empty map as `verifying_shares` because we have disabled 'cheater-detection' feature flag. Feature 'cheater-detection' only points to a malicious participant, if there's such." [5](#0-4) 

The same pattern is confirmed in the protocol specification:

> *"We do not make use of the cheater detection feature which requires additional computation and potentially an extra round to communicate the cheater to the rest of the participants."* [6](#0-5) 

---

### Impact Explanation

A single malicious MPC participant (strictly below the signing threshold) can send an invalid/garbage `SignatureShare` to the coordinator. The coordinator sums all shares and then verifies the final signature — which fails. The signing request returns `ProtocolError::AssertionFailed`. Because no per-share verification is performed, the malicious participant is never identified. The system has no protocol-level mechanism to exclude them from future signing rounds. Every subsequent signing request that includes this participant will fail in the same way.

This breaks the request-lifecycle safety invariant of the MPC network: a threshold-signature system is supposed to tolerate up to `t` malicious participants, but the implementation requires **all N** participants in the signing set to provide valid shares (due to `recv_from_others` waiting for all), reducing the effective fault tolerance to zero for the signing phase. Users whose cross-chain transactions depend on MPC signatures cannot complete them, which can result in funds being frozen on foreign chains.

**Impact class:** Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

- Requires only **one** malicious participant in the signing set, strictly below the threshold.
- The malicious participant incurs no cryptographic cost: sending a random scalar as a `SignatureShare` is trivially cheap.
- The malicious participant is never identified or penalized (no cheater detection, no slashing).
- The attack is repeatable for every signing request that includes the malicious participant.
- No privileged access, TEE compromise, or collusion is required.

---

### Recommendation

Enable per-share verification before aggregation. The FROST library supports this natively: populate the `verifying_shares` map in `PublicKeyPackage` with each participant's public verifying share (derived from the keygen output), and use `CheaterDetection::FirstCheater` (or equivalent). This allows the coordinator to:

1. Verify each `SignatureShare` against the corresponding participant's public key share before summing.
2. Identify and log the first malicious participant.
3. Exclude the identified participant from future signing rounds without operator intervention.

The keygen output already contains the per-participant verifying shares needed to populate this map. The additional cost is one scalar multiplication per participant per signing round.

---

### Proof of Concept

**Setup:** N=5 participants, threshold t=3. Participant P4 is malicious.

1. A signing request is initiated. The coordinator selects all 5 participants.
2. P1, P2, P3, P5 compute and send valid `SignatureShare` values.
3. P4 sends `SignatureShare(random_scalar)` — a garbage value computed in O(1) time.
4. The coordinator receives all 5 shares via `recv_from_others` (waits for all N).
5. `aggregate_custom(..., CheaterDetection::Disabled)` sums all shares including P4's garbage.
6. The final signature fails verification: `ProtocolError::AssertionFailed("signature failed to verify")` (or `ErrorFrostAggregation`).
7. The signing request fails. No participant is identified as malicious.
8. P4 repeats step 3 for every subsequent signing request. All fail. [7](#0-6) [8](#0-7)

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
