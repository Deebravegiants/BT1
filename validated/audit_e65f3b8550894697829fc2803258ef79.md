Audit Report

## Title
Linea Bridge `minimumFee` Silently Reduces Bridged ETH, Causing rsETH Undercollateralization — (File: contracts/bridges/LineaMessenger.sol)

## Summary
`LineaMessenger.sendETHToL1ViaBridge` forwards the full depositor ETH balance to Linea's `sendMessage` but passes `minimumFeeInWei()` as the fee argument, causing the L1 vault to receive `value − minimumFee` rather than `value`. Because `minimumFee` is never deducted from the amount used to mint rsETH, every bridge cycle leaves the L1 vault short by `minimumFee` wei relative to the outstanding rsETH supply, progressively undercollateralizing the token.

## Finding Description
In `LineaMessenger.sendETHToL1ViaBridge`, the Linea bridge fee is fetched at call time and passed as the `_fee` argument to `sendMessage`:

```solidity
// contracts/bridges/LineaMessenger.sol L39-43
uint256 minimumFee = ILineaMessageService(l2bridge).minimumFeeInWei();
if (value <= minimumFee) revert InsufficientAmountForBridge();
ILineaMessageService(l2bridge).sendMessage{ value: value }(target, minimumFee, bytes(""));
```

Per Linea's bridge design (`ILineaMessageService.sendMessage(address _to, uint256 _fee, bytes _calldata)`), the `_fee` is deducted from `msg.value` before crediting `_to` on L1. The L1 vault therefore receives `value − minimumFee`, not `value`.

The caller — `RSETHPoolV2.bridgeAssets()` and `RSETHPoolV3ExternalBridge.bridgeAssetsViaNativeBridge()` — passes `ethBalanceMinusFees` (the full depositor ETH balance net of protocol fees) as `value`:

```solidity
// contracts/pools/RSETHPoolV2.sol L292-296
uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
    l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees
);
```

```solidity
// contracts/pools/RSETHPoolV3ExternalBridge.sol L657-661
uint256 ethBalanceMinusFees = getETHBalanceMinusFees();
IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
    l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees
);
```

At deposit time, rsETH is minted against `amount − feeBps` with no deduction for `minimumFee`:

```solidity
// contracts/pools/RSETHPoolV2.sol L225-233
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

The `InsufficientAmountForBridge` guard only ensures `value > minimumFee`; it does not correct the accounting gap. The `minimumFee` is silently absorbed from depositor ETH, reducing the ETH that backs already-minted rsETH.

## Impact Explanation
Every invocation of `bridgeAssets()` / `bridgeAssetsViaNativeBridge()` on a Linea pool using `LineaMessenger` leaves the L1 vault short by `minimumFee` wei. Over many bridge cycles, or if Linea governance raises `minimumFeeInWei()`, the cumulative shortfall grows. rsETH holders redeeming on L1 receive less ETH than the rate they were quoted at deposit time.

**Impact class: Low — Contract fails to deliver promised returns.**

## Likelihood Explanation
The gap fires automatically on every bridge operation performed by the `BRIDGER_ROLE` on any Linea pool configured with `LineaMessenger`. No malicious actor is required; the structural accounting error is triggered by normal, expected protocol operation. `minimumFeeInWei()` is a dynamic, governance-adjustable value on Linea, so the per-cycle shortfall is not bounded to a negligible constant.

## Recommendation
The Linea bridge fee must be excluded from the ETH used to mint rsETH. Two sound approaches:

1. **Caller-supplied fee**: Require `BRIDGER_ROLE` to supply `minimumFee` as additional `msg.value` on top of `ethBalanceMinusFees`, so depositor ETH is not consumed by the bridge fee.
2. **Pre-deduction at mint time**: Query `minimumFeeInWei()` (or a conservative upper bound) and subtract a pro-rata share from each deposit's `amountAfterFee` before computing `rsETHAmount`, analogous to how Gelato explicitly accounts for L1 data fees in its fee oracle.

## Proof of Concept
1. Alice deposits 1 ETH into the Linea `RSETHPoolV2` (`feeBps = 10`, i.e. 0.1%).
2. `viewSwapRsETHAmountAndFee(1e18)` → `fee = 1e15`, `amountAfterFee = 0.999 ETH`. rsETH minted to Alice at the current rate.
3. `feeEarnedInETH += 1e15`; pool holds `0.999 ETH` of depositor ETH.
4. `BRIDGER_ROLE` calls `bridgeAssets()`. `getETHBalanceMinusFees()` returns `0.999 ETH`.
5. `LineaMessenger` fetches `minimumFee = 5e14` (0.0005 ETH, a realistic Linea value).
6. `sendMessage{ value: 0.999 ETH }(l1Vault, 5e14, "")` — Linea bridge credits L1 vault with `0.999 ETH − 0.0005 ETH = 0.9985 ETH`.
7. Alice's rsETH was minted against `0.999 ETH`; only `0.9985 ETH` backs it. The `0.0005 ETH` gap accumulates with every bridge cycle.

**Foundry fork test plan**: Fork Linea mainnet, deploy `RSETHPoolV2` + `LineaMessenger` pointing at the live Linea message service, call `deposit{value: 1 ether}`, then call `bridgeAssets()` as `BRIDGER_ROLE`, and assert that the L1 vault's received ETH equals `ethBalanceMinusFees − minimumFeeInWei()` rather than `ethBalanceMinusFees`.
