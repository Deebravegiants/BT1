### Title
`priceLimitX64 = 0` Silently Zeroes All `!zeroForOne` Swap Output Due to Incorrect Sentinel Comparison — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

The early-return guard in both `_swapToken1ForToken0SpecifiedInput` and `_swapToken1ForToken0SpecifiedOutput` uses the comparison `params.priceLimitX64 <= initialPriceX64`. When a caller passes `priceLimitX64 = 0` to mean "no upper price constraint", this condition evaluates to `0 <= initialPriceX64`, which is always `true` for any valid price. The swap silently returns zero deltas and no tokens are exchanged. The `zeroForOne` direction does not share this defect — its guard is `priceLimitX64 >= initialPriceX64`, so `0 >= initialPriceX64` is always `false` and the swap proceeds normally. The asymmetry is the root cause.

---

### Finding Description

In `MetricOmmPool._swapToken1ForToken0SpecifiedInput`:

```solidity
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}
```

And in `MetricOmmPool._swapToken1ForToken0SpecifiedOutput`:

```solidity
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0, 0);
}
```

`initialPriceX64` is always a positive Q64.64 price derived from the oracle bid/ask. When `priceLimitX64 = 0`, the condition `0 <= initialPriceX64` is unconditionally `true`, so both functions return immediately with all-zero outputs.

The `zeroForOne` counterparts (`_swapToken0ForToken1SpecifiedInput`, `_swapToken0ForToken1SpecifiedOutput`) use the opposite comparison:

```solidity
if (params.priceLimitX64 >= initialPriceX64) {
    return (0, 0, 0);
}
```

Here `0 >= initialPriceX64` is always `false`, so `priceLimitX64 = 0` correctly acts as "no lower-bound constraint" and the swap proceeds.

The outer swap loop in `_swapToken1ForToken0SpecifiedInput` already handles `priceLimitX64 = 0` correctly at the bin-crossing level:

```solidity
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= upperPriceX64) {
    break;
}
```

This `!= 0` guard is present in the loop body but absent from the function-level early-return guard, creating the inconsistency.

The interface NatSpec acknowledges the asymmetry without resolving it: *"Some legs treat `priceLimitX64 == 0` as 'no extra crossing constraint'; see `MetricOmmPool` / `SwapMath` for exact comparisons."*

---

### Impact Explanation

Any caller — including the periphery router — that passes `priceLimitX64 = 0` for a `!zeroForOne` swap receives `(0, 0)` deltas. The pool emits a `Swap` event with zero amounts, no ERC-20 transfers occur, and the transaction succeeds silently. The entire `!zeroForOne` swap path is rendered unusable for the common "no price limit" convention. This breaks core pool swap functionality for one of the two swap directions.

---

### Likelihood Explanation

`priceLimitX64 = 0` is the natural "unconstrained" sentinel. The `zeroForOne` direction accepts it correctly, so callers who test only that direction will not discover the defect. Any router, aggregator, or user that passes `0` for `!zeroForOne` swaps — a reasonable expectation given the `zeroForOne` behavior — will silently receive no output.

---

### Recommendation

Add the same `!= 0` guard to the `!zeroForOne` early-return checks, mirroring the bin-crossing loop:

```solidity
// _swapToken1ForToken0SpecifiedInput and _swapToken1ForToken0SpecifiedOutput
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}
```

This makes `priceLimitX64 = 0` mean "no upper-bound constraint" symmetrically with the `zeroForOne` direction, matching the documented intent.

---

### Proof of Concept

1. Deploy `MetricOmmPool` with a valid oracle returning `bid = 0.99 × Q64`, `ask = 1.01 × Q64`.
2. Add liquidity across several bins.
3. Call `swap(recipient, false, 10_000, 0, callbackData, "")` — `!zeroForOne`, exact-in, `priceLimitX64 = 0`.
4. Observe: `amount0Delta = 0`, `amount1Delta = 0`. No tokens transferred. Transaction succeeds.
5. Call `swap(recipient, true, 10_000, 0, callbackData, "")` — `zeroForOne`, exact-in, `priceLimitX64 = 0`.
6. Observe: swap executes normally, tokens are exchanged.
