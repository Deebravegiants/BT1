### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Allows a Single Malicious MPC Node to Deliver a Mismatched Foreign-Chain Verification Response — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the root public key, but it never verifies that `response.payload_hash` was actually derived from the `request` argument supplied in the same call. A single attested MPC participant (below threshold) who has already participated in a legitimate signing round for request A can reuse that signature to resolve the pending yields of a completely different request B, delivering the wrong payload hash to every caller waiting on request B and permanently removing request B from the pending map.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 691–754), the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

This only confirms that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root key. It does **not** confirm that `response.payload_hash` was computed from the `request` parameter passed in the same call.

The `payload_hash` for a foreign-tx response is defined as `hash(ForeignTxSignPayload::V1 { request, extracted_values })`. Because the contract never recomputes this hash from the submitted `request`, a caller can supply any `request` key (pointing to a different pending entry) together with a `payload_hash` and signature that were legitimately produced for a completely different request.

After the signature check passes, `resolve_yields_for` atomically removes the entry for the supplied `request` key and resumes all queued yields with the mismatched response bytes:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The `resolve_yields_for` helper removes the entry before resuming yields (analogous to the "mark as processed before sending" fix recommended in the external report), but the binding check between the response and the request is entirely absent: [3](#0-2) 

---

### Impact Explanation

A single malicious attested MPC participant (below threshold) who has participated in a legitimate signing round for request A obtains a valid `(payload_hash_A, sig_A)` pair. They then call:

```
respond_verify_foreign_tx(request = B, response = { payload_hash_A, sig_A })
```

The contract accepts the call because `sig_A` is a valid signature over `payload_hash_A`. It then:

1. Removes request B's entry from `pending_verify_foreign_tx_requests` — request B can never be legitimately responded to again (permanent DoS of that request slot).
2. Resumes all yields queued under request B with the serialized `{ payload_hash_A, sig_A }` — every caller waiting on request B receives a response whose `payload_hash` encodes request A's transaction details, not request B's.

If a downstream bridge contract does not independently recompute the expected `payload_hash` from its own request parameters (using `ForeignChainSignatureVerifier`), it will accept the mismatched response as proof that request B's foreign transaction was verified, enabling unauthorized bridge execution or double-spend conditions. Even when the SDK verifier is used correctly, the request B slot is permanently poisoned and its callers receive an unusable response, breaking the request-lifecycle accounting invariant.

**Allowed impact matched**: *Medium — request-lifecycle manipulation that breaks production safety/accounting invariants.* Potentially *High* for bridge contracts that do not re-verify the payload hash.

---

### Likelihood Explanation

- Requires a single attested MPC participant to be malicious — this is the standard Byzantine adversary model for threshold systems and is explicitly within scope.
- No threshold collusion is needed; one node suffices.
- The attacker only needs to have participated in any prior legitimate signing round to obtain a reusable `(payload_hash, sig)` pair.
- The attack is silent: the contract emits no event distinguishing a legitimate response from a substituted one.

---

### Recommendation

Before calling `resolve_yields_for`, the contract should verify that `response.payload_hash` encodes the submitted `request`. The most direct fix is to require the MPC node to also submit the extracted `values` in the response, allowing the contract to recompute `hash(ForeignTxSignPayload::V1 { request, values })` and assert it equals `response.payload_hash`. Alternatively, the contract can require the node to include the raw `ForeignTxSignPayload` in the response and verify both the hash preimage and the embedded `request` field match the pending request key before accepting the response.

---

### Proof of Concept

**Setup:**
- Request A: `verify_foreign_transaction({ tx_id: [0xAA; 32], ... })` — submitted and legitimately processed by the MPC network, producing `payload_hash_A` and `sig_A`.
- Request B: `verify_foreign_transaction({ tx_id: [0xBB; 32], ... })` — submitted and pending.

**Attack (single malicious attested participant):**

```rust
// Malicious node calls respond_verify_foreign_tx with request=B but response for A
contract.respond_verify_foreign_tx(
    request_B,                                    // key pointing to pending entry for tx_id_BB
    VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,             // hash encodes tx_id_AA
        signature: sig_A,                         // valid sig over payload_hash_A
    },
);
```

**Result:**
1. Signature check passes — `sig_A` is valid over `payload_hash_A` under the root key. [4](#0-3) 
2. `resolve_yields_for` removes request B's entry and resumes all its yields with `{ payload_hash_A, sig_A }`. [2](#0-1) 
3. Every caller waiting on request B receives a response whose `payload_hash` encodes `tx_id_AA`, not `tx_id_BB`.
4. Request B's pending slot is permanently gone — a legitimate `respond_verify_foreign_tx` for request B would now return `RequestNotFound`. [5](#0-4) 
5. A bridge contract that trusts the MPC response without re-verifying the payload hash would accept this as proof that `tx_id_BB` was verified on the foreign chain, enabling unauthorized bridge execution.
