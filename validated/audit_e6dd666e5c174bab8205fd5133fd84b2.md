### Title
Missing `root_Q` Consistency Check in `blob_to_keyset` Allows `derive_ecdsa_2p_keys` to Accept and Output Derived Keys with an Arbitrary Public Key — (`src/cbmpc/api/hd_keyset_ecdsa_2p.cpp`)

### Summary

`blob_to_keyset` deserializes a caller-supplied keyset blob and populates `key_share_ecdsa_hdmpc_2p_t` without verifying that `root_Q_compressed` is consistent with the stored `x_share`. An attacker who controls the blob can substitute any valid curve point for `root_Q`. The downstream `derive_keys` VRF-Compute-2P protocol never touches `root_Q` in its ZK proof (which is exclusively about `K`/`k_share`), so the protocol completes successfully and emits derived ECDSA-2PC key blobs whose stored public key `Q` is the attacker-chosen point rather than the true joint public key `(x_share_1 + x_share_2 + delta + non_hard_delta)*G`. The same flaw applies to `refresh`, which blindly propagates the tampered `root_Q` into the refreshed keyset.

### Finding Description

**Deserialization path — missing guard:**

`blob_to_keyset` in `src/cbmpc/api/hd_keyset_ecdsa_2p.cpp` performs only these checks on `root_Q_compressed`:

```cpp
error_t rv = keyset.root.Q.from_bin(icurve, blob.root_Q_compressed);
if (rv) return rv;
``` [1](#0-0) 

It verifies only that the bytes decode to a valid curve point. There is no check that `root_Q == x_share * G` (for the party's own share) or that `root_Q` is the sum of both parties' public shares.

**Contrast with `ecdsa_mp`'s `deserialize_key_blob`**, which explicitly enforces both invariants:

```cpp
if (Q != Q_sum) return coinbase::error(E_FORMAT, "invalid key blob");
// ...
if (blob.x_share * G != it_self->second) return coinbase::error(E_FORMAT, "invalid key blob");
``` [2](#0-1) 

**Protocol path — ZK proof does not cover `root_Q`:**

In `derive_keys`, the VRF-Compute-2P protocol computes `Q = key.root.Q` and uses it only to compute `Q_derived = Q + delta*G`. The ZK DH proof (`zk_dh1`, `zk_dh2`) proves only the DH relation for `k_share` and `K`:

```cpp
zk_dh1.prove(P, K_share, Z1, k_share, sid, 1);
// ...
if (rv = zk_dh1.verify(P, other_K_share, Z1, sid, 1)) return rv;
``` [3](#0-2) 

`root_Q` is never included in any ZK statement, commitment, or cross-party check. The protocol completes successfully regardless of what `root_Q` contains.

**Derived key output with wrong public key:**

```cpp
ecc_point_t Q_derived = Q + delta_G;   // Q is the attacker-chosen point
// ...
derived_keys[i].Q = Q_derived + non_hard_delta_G;  // wrong public key stored
``` [4](#0-3) 

The actual secret shares are `x_share_1` (P1, unchanged) and `x_share_2 + delta + non_hard_delta mod q` (P2). Their true joint public key is `(x_share_1 + x_share_2 + delta + non_hard_delta)*G`, which differs from the stored `Q_derived` whenever `root_Q` was tampered.

**`refresh` propagates the tampered `root_Q` silently:**

```cpp
ecdsa2pc::key_t ecdsa_key = ecdsa2pc::key_t{..., key.root.Q, ...};
// ...
if (rv = ecdsa2pc::refresh(job, ecdsa_key, new_ecdsa_key)) return rv;
new_key.root.Q = new_ecdsa_key.Q;  // tampered Q carried forward
``` [5](#0-4) 

`ecdsa2pc::refresh` sets `new_key.Q = key.Q` without any consistency check, so the tampered `root_Q` survives across refreshes.

### Impact Explanation

An attacker who controls the keyset blob (the API caller, or any party that can write to the blob storage) can:

1. Take a legitimately produced keyset blob.
2. Replace `root_Q_compressed` with any valid curve point `Q_fake`.
3. Call `coinbase::api::hd_keyset_ecdsa_2p::derive_ecdsa_2p_keys` — the call returns `SUCCESS`.
4. Receive derived ECDSA-2PC key blobs whose `Q` field is `Q_fake + delta*G + non_hard_delta*G` — a point unrelated to the actual secret shares.

The derived key blobs are accepted as valid output. Any subsequent signing attempt using these blobs will produce signatures that fail verification against the stored public key, or — if the attacker chose `Q_fake` such that they know its discrete log — the attacker knows the full private key of the derived key's public key without needing the honest party's cooperation.

Additionally, `refresh` silently propagates the tampered `root_Q`, so the invalid state persists indefinitely.

### Likelihood Explanation

The keyset blob is an opaque byte string that the API documentation explicitly says "can be persisted by the caller." Any party that controls blob storage (including a malicious API caller, a compromised storage backend, or a Byzantine participant who manages their own blob) can perform this substitution. No network access or threshold collusion is required — it is a purely local blob manipulation.

### Recommendation

Add a consistency check in `blob_to_keyset` that verifies the Paillier ciphertext `c_key` decrypts to `x_share` (available to P1 who holds the Paillier private key), and store each party's individual public share `x_share * G` in the blob so that each party can locally verify `x_share * G == Q_self_share`. Additionally, enforce that `root_Q == Q_self_share_1 + Q_self_share_2` during the first interactive protocol step (e.g., at the start of `derive_keys` or `refresh`), analogous to the check already present in `ecdsa_mp`'s `deserialize_key_blob` and `key_share_mp_t::refresh`:

```cpp
if (current_key.x_share * G != current_key.Qis.at(job.get_name(i)))
    return coinbase::error(E_BADARG, "x_share does not match Qi");
if (SUM(current_key.Qis) != current_key.Q)
    return coinbase::error(E_BADARG, "Q does not match the sum of Qis");
``` [6](#0-5) 

### Proof of Concept

```
1. Run dkg(job1, secp256k1, blob1) / dkg(job2, secp256k1, blob2) to get valid blobs.
2. Deserialize blob1 as keyset_blob_v1_t.
3. Replace root_Q_compressed with the compressed encoding of a random valid curve point Q_fake
   (e.g., a freshly generated random scalar r_fake times G).
4. Re-serialize to produce tampered_blob1.
5. Call derive_ecdsa_2p_keys(job1, tampered_blob1, hard, non_hard, sid, derived1)
         / derive_ecdsa_2p_keys(job2, blob2, hard, non_hard, sid, derived2).
   Both calls return SUCCESS.
6. Extract the public key from derived1[0] → pub_tampered.
   Extract the public key from derived2[0] → pub_correct.
7. Assert pub_tampered != pub_correct.
8. Assert pub_tampered == Q_fake + delta*G + non_hard_delta*G
   (where delta is deterministically derived from the VRF output Z = Z1+Z2).
9. Assert pub_correct == (x_share_1 + x_share_2 + delta + non_hard_delta)*G.
```

This demonstrates that `derive_ecdsa_2p_keys` returns `SUCCESS` and emits a derived key blob whose public key is the attacker-chosen `Q_fake`-derived point, not the true joint public key of the secret shares.
