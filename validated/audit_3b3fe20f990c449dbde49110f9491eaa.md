### Title
Propose-Update Deposits Permanently Stuck When `do_update` Clears All Entries - (`crates/contract/src/update.rs`)

### Summary

When `propose_update` is called, a participant must attach a NEAR deposit to cover storage costs for the `UpdateEntry`. When any update reaches threshold and `do_update` executes, it calls `entries.clear()`, which removes **all** pending proposals — including those from other proposers — without refunding their deposits. Because `UpdateEntry` does not record the depositor's account ID or the deposit amount, there is no mechanism to recover these funds. They are permanently absorbed into the contract's balance.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires a deposit at least equal to `ProposedUpdates::required_deposit(&update)` to cover storage for the new `UpdateEntry`. Only the excess above `required` is refunded to the proposer; the `required` portion is retained by the contract. [1](#0-0) 

The `UpdateEntry` struct stored in `entries` records only the update payload and `bytes_used` — **not** the depositor's account ID or the deposit amount: [2](#0-1) 

When threshold votes are reached, `do_update` is called. It removes the winning entry, then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every other pending proposal: [3](#0-2) 

The storage freed by `entries.clear()` is returned to the contract's own balance (NEAR storage staking), but the original depositors of the cleared proposals receive nothing. No refund path exists because the depositor identity and deposit amount were never recorded.

### Impact Explanation

Every participant whose `propose_update` call is cleared by a competing update execution permanently loses their NEAR deposit. For a contract-code update, `required_deposit` is proportional to the WASM binary size (potentially megabytes), making the per-proposal loss significant. This breaks the accounting invariant that a participant's deposit is returned when their proposal is no longer needed. The funds are irrecoverably absorbed into the contract balance with no withdrawal mechanism.

This matches the allowed Medium impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

This is triggered whenever two or more participants propose different updates concurrently — a realistic scenario during governance disputes or when participants independently propose the same upgrade. The `do_update` path is the normal execution path for every successful upgrade, so the loss occurs on every upgrade cycle where more than one proposal exists.

### Recommendation

Store the depositor's `AccountId` and the deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) depositor: AccountId,   // add
    pub(super) deposit: NearToken,     // add
}
```

In `do_update`, before calling `entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.depositor).transfer(entry.deposit)` for each one. This mirrors the refund pattern already used in `propose_update` for excess deposits. [4](#0-3) 

### Proof of Concept

1. Participant A calls `propose_update` with a large WASM binary, attaching deposit `D_A` (e.g., 8 NEAR as used in devnet tooling). The `required` portion is kept; only excess is refunded.
2. Participant B calls `propose_update` with a different WASM binary, attaching deposit `D_B`.
3. Participants vote for B's proposal until threshold is reached.
4. `vote_update` calls `do_update(&id_B, gas)`.
5. Inside `do_update`: `self.entries.remove(&id_B)` removes B's entry, then `self.entries.clear()` removes A's entry. `self.vote_by_participant.clear()` removes all votes.
6. A's deposit `D_A` is now permanently in the contract's balance. A has no method to reclaim it — `remove_update_vote` only removes the vote record, not the entry, and issues no refund. [3](#0-2) [5](#0-4)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
