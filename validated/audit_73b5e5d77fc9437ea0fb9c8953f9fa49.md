Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Non-Allowlisted Callers to Bypass the Deposit Gate — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

## Summary
`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` parameter (the actual token payer and caller) and gates only on `owner` (the position beneficiary). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender ≠ owner`, any non-allowlisted address can call `addLiquidity(owner = <any allowlisted address>, ...)`, pass the allowlist check, pay tokens from its own balance, and mint LP shares — completely defeating the deposit gate.

## Finding Description
`MetricOmmPool.addLiquidity` passes both `msg.sender` (as `sender`) and the caller-supplied `owner` to the extension hook via `_beforeAddLiquidity`:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`ExtensionCalling._beforeAddLiquidity` encodes both arguments and forwards them to the extension:

```solidity
// ExtensionCalling.sol L96-98
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
```

`DepositAllowlistExtension.beforeAddLiquidity` receives both but discards `sender` (unnamed first parameter) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

The interface NatSpec at `IMetricOmmPoolActions.sol L147` explicitly documents the operator pattern: "`msg.sender` pays but need not equal `owner`." The pool's `removeLiquidity` enforces `msg.sender == owner`, but `addLiquidity` has no such restriction. The callback in `LiquidityLib.addLiquidity` calls `IMetricOmmModifyLiquidityCallback(msg.sender).metricOmmModifyLiquidityCallback(...)` — i.e., it pulls tokens from the caller (`sender`), not from `owner`. The allowlist check on `owner` therefore gates the wrong party.

`SwapAllowlistExtension.beforeSwap` demonstrates the correct pattern — it checks `sender` (the actual caller) and discards the second argument (`recipient`):

```solidity
// SwapAllowlistExtension.sol L31-37
function beforeSwap(address sender, address, ...)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
```

`DepositAllowlistExtension` has the binding inverted relative to this correct pattern.

## Impact Explanation
A non-allowlisted address can inject tokens into a pool protected by `DepositAllowlistExtension` with a single direct call, requiring no special privileges or setup. The deposit allowlist — a core pool access-control mechanism — provides zero protection against the actual token payer. Additionally, the allowlisted `owner` receives LP shares they never requested; if `owner` is a smart contract without a `removeLiquidity` path, the underlying tokens become permanently trapped under that position key. Pool depth and fee accrual for all LPs are affected by the unauthorized injection. This constitutes broken core pool functionality (access control gate fully bypassed) causing potential loss of funds and unusable pool invariants.

## Likelihood Explanation
Exploitation requires only: (1) a pool configured with `DepositAllowlistExtension`, (2) knowledge of one allowlisted address (any existing LP's address is publicly visible on-chain via `LiquidityAdded` events), and (3) a single direct call to `pool.addLiquidity`. No flash loan, privileged access, or special setup is needed. The `MetricOmmPoolLiquidityAdder.addLiquidityExactShares(pool, owner, ...)` overload widens the attack surface further, as it also routes through the same broken hook with a caller-supplied `owner`.

## Recommendation
Check `sender` (the actual payer/caller) instead of `owner` in `beforeAddLiquidity`, mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intent is to gate both the payer and the position beneficiary, check both `sender` and `owner`.

## Proof of Concept
```
Setup:
  - Pool deployed with DepositAllowlistExtension
  - allowedDepositor[pool][alice] = true   (alice is allowlisted)
  - bob is NOT in allowedDepositor and allowAllDepositors[pool] = false

Attack:
  1. bob calls pool.addLiquidity(
         owner        = alice,   // allowlisted address — hook checks this
         salt         = 99,
         deltas       = { binIdxs: [0], shares: [minimalMintableLiquidity] },
         callbackData = abi.encode(KIND_PAY),  // or direct callback
         extensionData = ""
     )
  2. beforeAddLiquidity hook: allowedDepositor[pool][alice] == true → no revert
  3. LiquidityLib calls IMetricOmmModifyLiquidityCallback(bob).metricOmmModifyLiquidityCallback(amount0, amount1, ...)
     bob transfers tokens to the pool from his own balance
  4. Pool mints shares under positionBinKey(alice, salt=99, bin=0)

Result:
  - bob (non-allowlisted) successfully deposited tokens
  - alice holds an LP position she did not create
  - The deposit allowlist provided zero protection
```

A Foundry integration test can confirm this by deploying a pool with `DepositAllowlistExtension`, allowlisting `alice`, then calling `addLiquidity` from `bob` with `owner = alice` and asserting no revert occurs and `bob`'s token balance decreases.
