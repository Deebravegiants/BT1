Audit Report

## Title
Missing Chainlink Price Feed Staleness Check Allows Stale Asset Prices to Corrupt rsETH Minting and Price Updates - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness validation on `updatedAt` or `answeredInRound`. A stale asset price flows directly into rsETH minting calculations and the rsETH price update, enabling depositors to receive incorrect rsETH amounts and diluting existing holders' yield.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` silently discards all staleness indicators:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

Neither `updatedAt` nor `answeredInRound` is checked. If a Chainlink feed stops updating, the last recorded price is returned indefinitely without revert.

This stale price propagates through two critical paths:

**Path 1 — Deposit minting**: `LRTDepositPool.getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(asset)`, which delegates to `ChainlinkPriceOracle.getAssetPrice()`. The stale price directly determines how many rsETH tokens a depositor receives. [2](#0-1) 

**Path 2 — rsETH price update**: `LRTOracle._getTotalEthInProtocol()` calls `getAssetPrice(asset)` for each supported asset to compute total protocol TVL, which then sets `rsETHPrice`. [3](#0-2) 

The sibling oracle `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` does include partial staleness guards (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`), confirming the team is aware of the pattern but did not apply it to `ChainlinkPriceOracle`. [4](#0-3) 

No existing check in the deposit or oracle update flow compensates for this omission.

## Impact Explanation
**High — Theft of unclaimed yield.**

When a supported LST feed (e.g., stETH/ETH) goes stale at a price *above* the true current price:
- `getRsETHAmountToMint()` over-values the deposited asset → the depositor receives more rsETH than their deposit is worth.
- This dilutes the rsETH/ETH rate for all existing holders, effectively transferring accrued yield from existing rsETH holders to the new depositor.

When the stale price is *below* the true price:
- `_updateRsETHPrice()` computes a lower-than-actual TVL → `newRsETHPrice` drops → if the drop exceeds `pricePercentageLimit`, the protocol auto-pauses all deposits and withdrawals (temporary freezing of funds, Medium). [5](#0-4) 

## Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (e.g., 24 hours for stETH/ETH on mainnet). A feed can go stale during L2 sequencer downtime, Chainlink node outages, feed deprecation, or extreme market volatility triggering circuit-breaker behavior. No privileged access is required — any depositor calling `depositAsset()` or any caller of `updateRSETHPrice()` triggers the vulnerable path. Real-world Chainlink feed outages have occurred historically, making likelihood **medium**.

## Recommendation
Add staleness checks in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(answeredInRound >= roundId, "Stale price: answeredInRound < roundId");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= MAX_PRICE_AGE, "Stale price: updatedAt too old");
require(price > 0, "Invalid price");
```

Where `MAX_PRICE_AGE` is a configurable per-feed heartbeat (e.g., 25 hours for a 24-hour heartbeat feed). This mirrors the pattern already present in `ChainlinkOracleForRSETHPoolCollateral.sol`. [1](#0-0) 

## Proof of Concept
1. Chainlink's stETH/ETH feed stops updating (node outage). Last recorded price: `1.05e18`. True current price: `0.98e18` (stETH has depegged).
2. Attacker calls `depositAsset(stETH, 100e18, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint(stETH, 100e18)` → `LRTOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.05e18`.
4. `rsethAmountToMint = (100e18 * 1.05e18) / rsETHPrice` → attacker receives ~7% more rsETH than their deposit is worth.
5. Existing rsETH holders' redemption value is diluted by the over-minted rsETH.
6. No revert occurs anywhere in the call chain because `updatedAt` is never checked.

**Foundry fork test plan**: Fork mainnet, mock the stETH/ETH Chainlink feed to return a stale `updatedAt` (e.g., `block.timestamp - 2 days`) with an inflated price. Call `depositAsset` and assert that `rsethAmountToMint` exceeds the fair value. Then call `updateRSETHPrice()` and assert that `rsETHPrice` is lower than the pre-deposit value, confirming dilution of existing holders.
