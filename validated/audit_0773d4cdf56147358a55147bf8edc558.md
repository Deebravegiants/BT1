The key vulnerability is in `OmniBridge.sol`'s `initTransfer` function. When a native ERC20 token (not a bridge token, not a custom minter) is transferred in, the contract calls `safeTransferFrom` with `amount` but then uses the caller-supplied `amount` — not the actual received balance — for all downstream accounting and cross-chain message encoding.

### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Causes Unbacked Cross-Chain Releases - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer` calls `safeTransferFrom` to lock native ERC20 tokens but unconditionally uses the caller-supplied `amount` in the emitted event and the cross-chain message payload. For fee-on-transfer tokens, the bridge receives fewer tokens than `amount`, yet the destination chain is instructed to release the full `amount`, draining the bridge's reserves.

### Finding Description

In `initTransfer`, the native ERC20 lock path is:

```solidity
// OmniBridge.sol lines 407-412
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // actual received = amount - transfer_fee
);
```

Immediately after, the unmodified `amount` is passed to `initTransferExtension` and emitted:

```solidity
// OmniBridge.sol lines 415-436
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // <-- caller-supplied, not balance-checked
    fee, nativeFee, recipient, message, extensionValue
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // <-- same inflated value
    fee, nativeFee, recipient, message
);
```

In `OmniBridgeWormhole.initTransferExtension`, this `amount` is Borsh-encoded into the Wormhole VAA:

```solidity
// OmniBridgeWormhole.sol lines 135-136
Borsh.encodeUint128(amount),
```

On the destination chain, `finTransfer` decodes `payload.amount` from that VAA and releases it:

```solidity
// OmniBridge.sol lines 351-354
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount   // inflated amount, not what was actually locked
);
```

No balance snapshot is taken before/after the `safeTransferFrom` call to compute the true received amount.

### Impact Explanation

For any fee-on-transfer ERC20 token bridged through the native lock path, the bridge locks `amount - fee_deducted` tokens on the source chain but instructs the destination chain to release `amount` tokens. Each such transfer creates an unbacked surplus on the destination side. Repeated transfers drain the bridge's token reserves, eventually causing legitimate redemptions to fail (frozen redemption path) or allowing the attacker to extract more value than was deposited (unauthorized withdrawal of locked assets).

This falls squarely under:
- **Critical**: Unauthorized release/withdrawal of locked bridge assets through accounting failure.
- **Critical**: Irreversible fund lock for later users whose redemptions fail once reserves are exhausted.

### Likelihood Explanation

USDT's transfer fee is currently 0% but is a configurable parameter in its contract. Other fee-on-transfer tokens (e.g., STA, tokens with reflection mechanics) already charge non-zero fees. Any such token that is not a `isBridgeToken` and has no `customMinters` entry will hit this code path. An unprivileged user only needs to call `initTransfer` with such a token — no special role or key is required.

### Recommendation

Capture the contract's token balance before and after `safeTransferFrom`, and use the delta as the canonical locked amount for all downstream calculations:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived (cast to uint128) instead of amount going forward
```

Pass `actualReceived` (not `amount`) to `initTransferExtension` and the `InitTransfer` event. Apply the same pattern to the `customMinters` path where `safeTransferFrom` is called before `burn`.

### Proof of Concept

1. Deploy or use any ERC20 token that deducts a 1% fee on every transfer (e.g., a reflection token or a future USDT with fee enabled).
2. Call `OmniBridge.initTransfer(tokenAddress, 1000, 0, 0, "recipient.near", "")` with `amount = 1000`.
3. The bridge receives 990 tokens (`safeTransferFrom` deducts 10 as fee).
4. The Wormhole VAA encodes `amount = 1000`.
5. On the destination chain, `finTransfer` is called with `payload.amount = 1000` and releases 1000 tokens to the recipient.
6. The bridge's reserve is now 10 tokens short per transfer.
7. Repeat until the bridge's reserve of that token is exhausted; subsequent legitimate `finTransfer` calls revert, permanently locking users' cross-chain transfers.
