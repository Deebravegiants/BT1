### Title
`OmniBridgeWormhole` Initialization Always Reverts Due to Double `initializer` Modifier Conflict, Enabling Admin Takeover - (File: `evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

### Summary

`OmniBridgeWormhole.initializeWormhole()` carries the `initializer` modifier and internally calls `OmniBridge.initialize()`, which also carries the `initializer` modifier. Under OpenZeppelin v5 `Initializable`, a second `initializer`-guarded call within an already-initializing context always reverts with `InvalidInitialization()` (outside a constructor). This makes `OmniBridgeWormhole` impossible to initialize through a proxy. Because `OmniBridge.initialize()` is `public`, an unprivileged attacker can call it directly on the uninitialized proxy and seize `DEFAULT_ADMIN_ROLE`.

### Finding Description

`OmniBridgeWormhole` inherits from `OmniBridge`. Its entry-point initializer is:

```solidity
// OmniBridgeWormhole.sol L32-46
function initializeWormhole(...) external initializer {
    initialize(                    // ← calls OmniBridge.initialize()
        tokenImplementationAddress,
        nearBridgeDerivedAddress,
        omniBridgeChainId
    );
    _wormhole = IWormhole(wormholeAddress);
    _consistencyLevel = consistencyLevel;
}
``` [1](#0-0) 

The parent function it calls is:

```solidity
// OmniBridge.sol L72-86
function initialize(...) public initializer {   // ← also initializer
    ...
    _grantRole(DEFAULT_ADMIN_ROLE, _msgSender());
    _grantRole(PAUSABLE_ADMIN_ROLE, _msgSender());
}
``` [2](#0-1) 

OpenZeppelin v5 `Initializable.initializer` logic (simplified):

```
bool initialSetup = initialized == 0 && isTopLevelCall;
bool construction = initialized == 1 && address(this).code.length == 0;
if (!initialSetup && !construction) revert InvalidInitialization();
```

Execution trace when `initializeWormhole` is called on a proxy:

| Step | `_initialized` | `_initializing` | `isTopLevelCall` | Result |
|------|---------------|-----------------|------------------|--------|
| Enter `initializeWormhole`'s `initializer` | 0 | false | true | `initialSetup=true` → **passes**, sets `_initialized=1`, `_initializing=true` |
| Enter `initialize`'s `initializer` | 1 | true | **false** | `initialSetup=false`, `construction=false` (proxy has code) → **REVERTS** |

The entire transaction reverts, leaving the proxy with `_initialized = 0` (uninitialized). Because `OmniBridge.initialize()` is `public` and inherited by `OmniBridgeWormhole`, any caller can invoke it directly on the proxy.

### Impact Explanation

An attacker who front-runs or races the deployer after proxy creation (before a successful initialization) calls `OmniBridge.initialize()` directly on the proxy. This grants the attacker `DEFAULT_ADMIN_ROLE` and `PAUSABLE_ADMIN_ROLE`. With `DEFAULT_ADMIN_ROLE` the attacker can:

- Call `upgradeToken` to replace bridge token implementations with malicious ones, enabling arbitrary minting or burning of all wrapped assets.
- Call `addCustomToken` / `removeCustomToken` to reroute token mappings, redirecting user funds.
- Call `setNearBridgeDerivedAddress` to accept forged settlement proofs from an attacker-controlled address.
- Pause or permanently freeze the bridge via `PAUSABLE_ADMIN_ROLE`.

This satisfies **Critical** impact: unauthorized custody escape of locked/wrapped bridge assets and irreversible fund lock.

### Likelihood Explanation

The revert is deterministic and 100% reproducible — `initializeWormhole` will always fail on a proxy. Any deployment attempt that does not use a single atomic constructor call (i.e., standard UUPS proxy deployment via `ERC1967Proxy` + separate `initializeWormhole` call) leaves the proxy open. An attacker monitoring the mempool can front-run the initialization call with a direct call to `initialize()`.

### Recommendation

`OmniBridge.initialize()` is a base-contract initializer that is called by a child. It must use `onlyInitializing` instead of `initializer`, following the OpenZeppelin pattern for upgradeable base contracts. Rename it to an internal `__OmniBridge_init` function:

```solidity
// OmniBridge.sol
function __OmniBridge_init(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) internal onlyInitializing {
    ...
    _grantRole(DEFAULT_ADMIN_ROLE, _msgSender());
    _grantRole(PAUSABLE_ADMIN_ROLE, _msgSender());
}
```

```solidity
// OmniBridgeWormhole.sol
function initializeWormhole(...) external initializer {
    __OmniBridge_init(tokenImplementationAddress, nearBridgeDerivedAddress, omniBridgeChainId);
    _wormhole = IWormhole(wormholeAddress);
    _consistencyLevel = consistencyLevel;
}
```

If `OmniBridge` must also be deployable standalone, add a separate `public initializer` wrapper that calls `__OmniBridge_init`.

### Proof of Concept

1. Deploy `OmniBridgeWormhole` implementation contract.
2. Deploy `ERC1967Proxy` pointing to the implementation (proxy `_initialized = 0`).
3. Call `proxy.initializeWormhole(...)` → reverts with `InvalidInitialization()`. Proxy remains uninitialized.
4. Attacker calls `proxy.initialize(attackerAddr, attackerAddr, 0)` → succeeds. Attacker holds `DEFAULT_ADMIN_ROLE`.
5. Attacker calls `proxy.upgradeToken(anyBridgeToken, maliciousImpl)` to replace token logic, or `proxy.setNearBridgeDerivedAddress(attackerControlledVerifier)` to accept forged cross-chain proofs. [3](#0-2) [4](#0-3)
