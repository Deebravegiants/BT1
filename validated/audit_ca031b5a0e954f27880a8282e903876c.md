Audit Report

## Title
Excess native ETH sent directly to payable swap entry-points is permanently stranded and claimable by any caller — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
When a user calls any payable swap function (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) directly with `tokenIn = WETH` and `msg.value` exceeding the actual swap input, `pay()` wraps only the required amount and leaves the surplus native ETH on the router. Because `refundETH()` is an unrestricted public function that forwards the router's entire ETH balance to `msg.sender`, any third party can immediately claim the stranded ETH, causing a direct loss of user principal.

## Finding Description
`PeripheryPayments.pay()` handles WETH settlement by consuming exactly `value` wei from the router's native balance: [1](#0-0) 

When `address(this).balance > value`, only `value` is wrapped and forwarded; the remainder is silently left on the router. None of the four swap entry-points perform a post-swap refund: [2](#0-1) [3](#0-2) 

`refundETH()` has no access control and sends the full contract ETH balance to whoever calls it: [4](#0-3) 

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) does not protect against this because ETH sent via a payable function call bypasses `receive()` entirely. [5](#0-4) 

The protocol's own test suite confirms the design intent: surplus ETH recovery requires the user to bundle `refundETH()` as a second call inside `multicall`. When a user calls a swap function directly (not via `multicall`), no automatic refund occurs and the surplus is permanently stranded until claimed by an arbitrary caller. [6](#0-5) 

## Impact Explanation
Direct loss of user principal. Any surplus ETH sent with a direct swap call is immediately claimable by any third party via `refundETH()`. The loss is bounded only by the excess ETH the user sent; there is no protocol-level cap or recovery path once the ETH is claimed. This meets the Sherlock threshold for a direct loss of user funds.

## Likelihood Explanation
The `exactOutputSingle` path is especially prone: the user knows only their `amountInMaximum` at call time, not the actual `amountIn`, making over-sending native ETH a natural usage pattern. MEV bots already monitor router balances on mainnet for exactly this class of stranded-value opportunity and can front-run or back-run the swap transaction in the same block.

## Recommendation
1. **Auto-refund in each payable entry-point:** After the swap completes, call `_transferETH(msg.sender, address(this).balance)` (guarded by `> 0`) inside `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`.
2. **Alternatively, enforce exact ETH:** Revert if `msg.value` exceeds the amount actually consumed by `pay()`, preventing over-sending at the entry-point level.

## Proof of Concept
1. Pool has WETH/Token1. User calls `router.exactInputSingle{value: 1.5 ether}(params)` where `params.amountIn = 1 ether` and `params.tokenIn = WETH`.
2. Inside `_justPayCallback`, `pay(WETH, user, pool, 1 ether)` is called. `nativeBalance = 1.5 ETH >= value = 1 ETH`, so only `1 ETH` is wrapped and forwarded; `0.5 ETH` remains on the router.
3. The swap completes successfully; the user receives their Token1 output.
4. Attacker calls `router.refundETH()` in the same or next block and receives `0.5 ETH`.
5. User has lost `0.5 ETH` with no recourse.

A minimal Foundry test can reproduce this by calling `exactInputSingle{value: 2 * amountIn}(...)` directly (not via `multicall`), then calling `refundETH()` from a different address and asserting the attacker's balance increased by `amountIn`.

  function exactOutputSingle(ExactOutputSingleParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
    int128 amountOut = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = MetricOmmSwapInputs.int128ToUint128(
      MetricOmmSwapResults.extractAmountIn(params.zeroForOne, amount0Delta, amount1Delta)
    );

    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
  }
