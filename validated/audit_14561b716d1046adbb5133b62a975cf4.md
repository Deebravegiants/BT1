### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Verification Forgery - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that `response.payload_hash` carries a valid MPC signature, but never checks that `payload_hash` is actually derived from the `request.request` stored in `pending_verify_foreign_tx_requests`. A single Byzantine attested participant (below signing threshold) who holds a legitimately produced signature for any other pending request can substitute that signature and hash into an unrelated pending request, causing the contract to resolve it with fabricated foreign-chain data.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs two checks:

1. The ECDSA signature in `response` is valid over `response.payload_hash` using the domain's root public key.
2. The `request` key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What it does **not** check is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: <actual_values> }))`. The `ForeignTxSignPayload` is defined as: [2](#0-1) 

The `payload_hash` is computed by the nodes as `SHA-256(borsh(ForeignTxSignPayload))`, which commits to both the `ForeignChainRpcRequest` and the extracted values: [3](#0-2) 

Because the contract only verifies the signature over the caller-supplied `payload_hash` without binding it to the stored `request.request`, any valid `(payload_hash, signature)` pair produced for **any** pending request can be submitted as the response to **any other** pending request.

The leader node is the sole submitter of `respond_verify_foreign_tx` on-chain: [4](#0-3) 

In the CaitSith threshold protocol the leader broadcasts the final signature to all participants. Any attested participant who received a final signature for request B therefore possesses a valid `(hash_B, sig_B)` pair and can call `respond_verify_foreign_tx(request = request_A, response = {payload_hash: hash_B, signature: sig_B})`. The contract accepts it because `sig_B` is a valid signature over `hash_B` and `request_A` exists in the pending map.

Compare with `respond` for regular signatures, which correctly binds the signature to the stored payload by verifying against the derived key and `request.payload`: [5](#0-4) 

No equivalent binding exists in `respond_verify_foreign_tx`.

### Impact Explanation

A bridge contract that calls `verify_foreign_transaction` for Bitcoin tx X (a 1 BTC deposit) receives a `VerifyForeignTransactionResponse` whose `payload_hash` is actually `SHA-256(borsh({request: Bitcoin(tx_Y), values: [BlockHash(hash_Y)]}))` — the commitment for a completely different transaction. The response signature is valid, so the bridge contract's signature check passes. Because the `VerifyForeignTransactionResponse` contains only `payload_hash` and `signature` (not the raw extracted values), the bridge contract cannot independently reconstruct the expected hash without knowing the extracted values in advance — which is precisely the information it was asking the MPC to provide. The bridge contract is therefore forced to trust the returned `payload_hash`, and will proceed as if tx X was legitimately verified. This constitutes a forged foreign-chain verification enabling invalid bridge execution (e.g., minting tokens for a deposit that was never made, or accepting a finality proof for a transaction that was not confirmed).

This matches the allowed High impact: **"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."**

### Likelihood Explanation

The attacker is a single Byzantine attested MPC participant — strictly below the signing threshold. No collusion with other participants is required. The attacker only needs to:

1. Participate honestly in the threshold signing of two or more concurrent requests (which is normal node behavior).
2. Retain the final signature produced for one request.
3. Submit `respond_verify_foreign_tx` pairing that signature with a different pending request key.

The `respond_verify_foreign_tx` endpoint is callable by any attested participant, not only the designated leader: [6](#0-5) 

Concurrent `verify_foreign_transaction` requests are a normal production condition (the contract explicitly supports fan-out queuing of duplicate requests): [7](#0-6) 

### Recommendation

Before resolving the pending yield, recompute the expected `payload_hash` from the stored `request.request` and the extracted values supplied in the response, and assert equality. Concretely, require the node to include the full `ForeignTxSignPayload` (not just its hash) in the response, then verify:

```rust
let expected_hash = payload.compute_msg_hash()?;
assert!(expected_hash == response.payload_hash, "payload_hash mismatch");
assert!(payload.request() == request.request, "request mismatch");
```

This mirrors how `respond` binds the signature to the stored `request.payload` via the derived key, and closes the analogous gap in `respond_verify_foreign_tx`.

### Proof of Concept

**Setup**: Two concurrent pending requests exist:
- `request_A = { request: Bitcoin(tx_X, confirmations=6, extractors=[BlockHash]), domain_id: 0, payload_version: V1 }`
- `request_B = { request: Bitcoin(tx_Y, confirmations=6, extractors=[BlockHash]), domain_id: 0, payload_version: V1 }`

**Step 1** — Byzantine participant P joins the threshold signing for both requests as a follower. The CaitSith leader broadcasts the final signatures; P retains:
- `(hash_A, sig_A)` for request A
- `(hash_B, sig_B)` for request B

where `hash_B = SHA-256(borsh(ForeignTxSignPayload::V1 { request: Bitcoin(tx_Y), values: [BlockHash(block_hash_Y)] }))`.

**Step 2** — P calls:
```
respond_verify_foreign_tx(
    request = request_A,          // key for tx_X's pending yield
    response = {
        payload_hash: hash_B,     // hash of tx_Y's payload
        signature:    sig_B,      // valid sig over hash_B
    }
)
```

**Step 3** — Contract validation (lines 718–747):
- `verify_ecdsa_signature(sig_B, hash_B, root_pk)` → **passes** (sig_B is a genuine threshold signature)
- `pending_verify_foreign_tx_requests.get(request_A)` → **found**
- `resolve_yields_for(request_A, serialize(response))` → **resolves** the yield for tx_X's caller with `{payload_hash: hash_B, signature: sig_B}` [7](#0-6) 

**Step 4** — The bridge contract for User A receives `{payload_hash: hash_B, signature: sig_B}`. It verifies the signature is valid (it is), and proceeds as if tx_X was verified. The actual commitment is to tx_Y's block hash, not tx_X's. The bridge contract has been deceived into accepting a forged foreign-chain verification.

        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
