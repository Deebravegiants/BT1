Audit Report

## Title
ETH Push-Transfer to Reverting Contract Recipient Permanently Freezes User Funds - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`_processWithdrawalCompletion` uses a FIFO deque to process ETH withdrawal requests and pushes ETH directly to the recipient via an uncapped `call`. If the recipient is a smart-contract address whose `receive()` reverts, the entire transaction reverts — undoing `popFront()` and `unlockedWithdrawalsCount[asset]--` — leaving the queue permanently stuck at that entry. Because rsETH is burned during `unlockQueue`, the user loses their rsETH and cannot recover the ETH, and `sweepRemainingAssets` is also permanently blocked for the ETH asset.

## Finding Description
`_processWithdrawalCompletion` (L699–738) is the sole code path for both `completeWithdrawal` (L183–185) and `completeWithdrawalForUser` (L192–204). It executes in this order:

1. **L705**: `userAssociatedNonces[asset][user].popFront()` — removes the front of the FIFO deque.
2. **L717**: `unlockedWithdrawalsCount[asset]--` — decrements the unlocked counter.
3. **L734**: `_transferAsset(asset, user, request.expectedAssetAmount)` — pushes ETH to `user`.

`_transferAsset` (L876–883) for ETH:
```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```
No gas cap is applied. If `user` is a contract whose `receive()` reverts (or consumes all forwarded gas), the entire call reverts. Solidity unwinds all state changes: `popFront()` is undone, `unlockedWithdrawalsCount[asset]--` is undone, and the queue pointer stays at the same stuck entry. Every future call to complete this user's withdrawals hits the same stuck entry first (FIFO), permanently blocking all of the user's queued withdrawals.

`completeWithdrawalForUser` provides no relief — it calls the identical internal function and still pushes ETH to `user`. The developer comment at L191 ("Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH") acknowledges ETH transfer risk but incorrectly dismisses it.

`sweepRemainingAssets` (L395–414) is gated on `!hasUnlockedWithdrawals(asset)` (L403), which checks `unlockedWithdrawalsCount[asset] > 0` (L630). Because the stuck request was already counted as unlocked during `unlockQueue` → `_unlockWithdrawalRequests` → `unlockedWithdrawalsCount[asset]++`, and the decrement at L717 is always reverted, this counter never reaches zero. `sweepRemainingAssets` is permanently blocked for the ETH asset across the entire protocol.

## Impact Explanation
**Critical — Permanent freezing of funds.** The rsETH is burned during `unlockQueue` (before `completeWithdrawal` is called), so the user has already surrendered their rsETH. The ETH sits in `LRTWithdrawalManager` with no admin rescue path: `sweepRemainingAssets` is blocked by the permanently inflated `unlockedWithdrawalsCount`, and both `completeWithdrawal` and `completeWithdrawalForUser` always revert for this user. All subsequent ETH withdrawal requests by the same user are also frozen because the FIFO queue always presents the stuck entry first.

## Likelihood Explanation
No privileged role, oracle manipulation, or external compromise is required. Any depositor who used a smart-contract address (Gnosis Safe, Argent, account-abstraction wallet) to call `initiateWithdrawal` is a potential victim. A contract wallet loses the ability to receive ETH if: all signers lose access (key loss, death, legal seizure), the wallet is upgraded to a version lacking a `receive()` function, or the wallet's `receive()` is intentionally or accidentally made to revert. This is a realistic, non-exotic scenario for DeFi users.

## Recommendation
Replace the push-payment pattern with a pull-payment (credit) pattern: instead of calling `_transferAsset` inside `_processWithdrawalCompletion`, record the owed amount in a `mapping(address => uint256) pendingETH` and expose a separate `claimETH()` function. Alternatively, wrap the `_transferAsset` call in a `try/catch` and, on failure, credit the amount to a claimable balance rather than reverting.

## Proof of Concept
1. Deploy `VictimWallet` — a proxy contract whose `receive()` initially succeeds.
2. From `VictimWallet`, call `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`; a withdrawal request is queued.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. `_unlockWithdrawalRequests` advances `nextLockedNonce`, sets `request.expectedAssetAmount`, increments `unlockedWithdrawalsCount[ETH]`, and burns the rsETH. ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`.
4. Upgrade `VictimWallet`'s implementation so `receive()` reverts (or brick the wallet).
5. Call `completeWithdrawal(ETH_TOKEN, "")` from `VictimWallet` — reverts with `EthTransferFailed`.
6. Operator calls `completeWithdrawalForUser(ETH_TOKEN, VictimWallet, "")` — also reverts with `EthTransferFailed`.
7. ETH sits in `LRTWithdrawalManager` permanently. `unlockedWithdrawalsCount[ETH]` remains > 0, blocking `sweepRemainingAssets`. The user's rsETH is already burned and unrecoverable.

**Foundry test plan:** Deploy a mock `VictimWallet` with an upgradeable `receive()`. Run the full withdrawal lifecycle (initiate → unlock → brick wallet → attempt complete). Assert that `completeWithdrawal` and `completeWithdrawalForUser` both revert, `unlockedWithdrawalsCount[ETH_TOKEN]` remains 1, and `sweepRemainingAssets` reverts with `PendingWithdrawalsExist`.
