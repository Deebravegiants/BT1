### Title
Post-Restart `sent_to` Accumulation Causes Permanent Suppression of Valid Unconfirmed Transactions — (`chia/wallet/wallet_transaction_store.py`)

### Summary

`tx_submitted` is an in-memory dict that does not survive wallet restarts. The DB-persisted `sent_to` list is never trimmed of FAILED entries on restart. After a restart, `get_not_sent` re-queues any unconfirmed tx (since `tx_submitted` is empty), and `_messages_to_resend` sends it to peers not already in `sent_to`. A single additional FAILED response from any new peer pushes `len(sent_to)` to `minimum_send_attempts` (6), causing `is_valid()` to return `False`, and `increment_sent` to permanently mark the tx as `confirmed=True, confirmed_at_height=0`.

### Finding Description

**Step 1 — `tx_submitted` reset on restart:** [1](#0-0) 

`tx_submitted` is set to `{}` unconditionally on every `create()` call. The DB-persisted `sent_to` list is not cleared.

**Step 2 — `get_not_sent` re-queues the tx after restart:** [2](#0-1) 

Because `tx_submitted` is empty, the tx falls into the `else` branch and is returned for re-sending, regardless of how many FAILED entries already exist in `sent_to`.

**Step 3 — `_messages_to_resend` sends only to new peers:** [3](#0-2) 

Peers already in `sent_to` with FAILED status are excluded. So the tx is sent to a new peer (peer6).

**Step 4 — `increment_sent` appends unconditionally and checks `is_valid()`:** [4](#0-3) 

The new FAILED entry is appended. `sent_to` now has 6 entries.

**Step 5 — `is_valid()` threshold is a raw count of `sent_to` length:** [5](#0-4) 

`minimum_send_attempts = 6`. With 6 FAILED entries and no SUCCESS or fee-error entries, `is_valid()` returns `False`. The tx is written back as `confirmed=True, confirmed_at_height=0`. [6](#0-5) 

### Impact Explanation

The tx is permanently removed from the unconfirmed queue and will never be retried. Any wallet flow that depends on this tx — pool wallet state transitions, clawback sends/receives, offer settlement — is silently broken. The user sees the tx as "confirmed" at height 0 with no on-chain confirmation. The only recovery is manually reconstructing and re-submitting the transaction, which is non-obvious and may not be possible for singleton-controlled flows (pool wallets, DIDs).

### Likelihood Explanation

The pre-condition of 5 FAILED entries is reachable in two ways:

1. **Natural/no attacker**: During a network upgrade or mass node restart, `MEMPOOL_NOT_INITIALIZED` is returned by honest peers. This error is not in the fee-error exemption list in `is_valid()`, so it counts toward the suppression threshold. Five such responses across five connected peers before a wallet restart is plausible.

2. **Malicious peer**: An attacker running a full node that the wallet connects to can respond with `FAILED` to any `send_transaction` message. The wallet connects to untrusted full nodes on the public network. The attacker needs to be one of the peers the wallet sends to across two sessions (pre- and post-restart). Controlling or operating as a single malicious peer is sufficient to contribute the final FAILED entry that crosses the threshold.

### Recommendation

- On `WalletTransactionStore.create()`, trim FAILED-only `sent_to` entries (or reset `sent_to` entirely) for unconfirmed transactions that have not yet reached a SUCCESS, so the restart does not carry over stale failure counts.
- Alternatively, `is_valid()` should count only FAILED entries from the **current session** (i.e., entries added after the last restart), not the cumulative historical list.
- Extend the fee-error exemption in `is_valid()` to cover other known-transient errors such as `MEMPOOL_NOT_INITIALIZED`.

### Proof of Concept

```python
# Pre-populate tx with 5 FAILED sent_to entries in DB
tx = dataclasses.replace(
    tr1,
    sent_to=[
        (f"peer{i}", uint8(MempoolInclusionStatus.FAILED.value), "UNKNOWN")
        for i in range(1, 6)
    ],
    sent=uint32(5),
)
await store.add_transaction_record(tx)

# Simulate restart: re-create the store (tx_submitted = {})
store2 = await WalletTransactionStore.create(db_wrapper, MINIMUM_CONFIG)
assert store2.tx_submitted == {}

# One more FAILED response from a new peer
await store2.increment_sent(tx.name, "peer6", MempoolInclusionStatus.FAILED, Err.UNKNOWN)

# Assert the tx is now permanently suppressed
result = await store2.get_transaction_record(tx.name)
assert result.confirmed is True
assert result.confirmed_at_height == 0
# The tx was never confirmed on-chain — it is silently dead
```

    def is_valid(self) -> bool:
        if len(self.sent_to) < minimum_send_attempts:
            # we haven't tried enough peers yet
            return True
        if any(x[1] == MempoolInclusionStatus.SUCCESS.value for x in self.sent_to):
            # we managed to push it to mempool at least once
            return True
        if any(x[2] in {Err.INVALID_FEE_LOW_FEE.name, Err.INVALID_FEE_TOO_CLOSE_TO_ZERO.name} for x in self.sent_to):
            # we tried to push it to mempool and got a fee error so it's a temporary error
            return True
        return False
