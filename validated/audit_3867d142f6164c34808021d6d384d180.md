Audit Report

## Title
ETH Withdrawal Permanently Freezes Funds When Recipient Is a Non-Payable Contract — (`contracts/LRTWithdrawalManager.sol`)

## Summary
When a non-payable smart contract initiates an ETH withdrawal via `initiateWithdrawal`, the operator's subsequent `unlockQueue` call irreversibly burns the rsETH and moves ETH into `LRTWithdrawalManager`. Every attempt to complete the withdrawal then permanently reverts because `_transferAsset` unconditionally pushes ETH to the original requesting address, which cannot receive it. No on-chain recovery path exists: the completion functions always revert, and `sweepRemainingAssets` is blocked while the stuck withdrawal remains pending.

## Finding Description
**Root cause:** The withdrawal recipient is hardcoded to the original `msg.sender` at request time with no mechanism to redirect funds.

**Exploit flow:**

1. **Initiation** — A non-payable contract (no `receive()`/`fallback()`) calls `initiateWithdrawal(ETH_TOKEN, amount, ...)`. rsETH is pulled from `msg.sender` and the request is stored with `msg.sender` as the implicit owner via `userAssociatedNonces[asset][msg.sender]`. [1](#0-0) [2](#0-1) 

2. **Unlock (point of no return)** — An operator calls `unlockQueue`. rsETH held by the manager is burned and the corresponding ETH is redeemed from the unstaking vault into `LRTWithdrawalManager`. Both actions are irreversible. [3](#0-2) 

3. **Completion always reverts** — `_processWithdrawalCompletion` unconditionally calls `_transferAsset(asset, user, request.expectedAssetAmount)` where `user` is always the original requesting address. [4](#0-3) 

4. **`_transferAsset` reverts for non-payable contracts** — The low-level call returns `sent = false` and the function reverts with `EthTransferFailed`. Because this revert unwinds the entire transaction, the `popFront()`, `delete withdrawalRequests[requestId]`, and `unlockedWithdrawalsCount[asset]--` at lines 705, 712, and 717 are all rolled back, leaving the withdrawal permanently stuck in the queue. [5](#0-4) 

5. **`completeWithdrawalForUser` provides no escape** — The operator-callable variant passes the same `user` address directly into `_processWithdrawalCompletion`; it cannot redirect ETH to a different recipient. [6](#0-5) 

6. **`sweepRemainingAssets` is permanently blocked** — Because the withdrawal transaction always reverts and state is rolled back, `unlockedWithdrawalsCount[asset]` remains nonzero, so `hasUnlockedWithdrawals(asset)` returns `true` and `sweepRemainingAssets` reverts with `PendingWithdrawalsExist`. There is no cancel, rescue, or recipient-update function anywhere in the contract. [7](#0-6) 

## Impact Explanation
**Critical — Permanent freezing of funds.**

- The rsETH is burned during `unlockQueue` and is unrecoverable.
- The corresponding ETH is held in `LRTWithdrawalManager` and can never be transferred out: every completion call reverts, sweep is blocked, and no rescue function exists.
- Both assets are permanently lost with zero on-chain recourse.

## Likelihood Explanation
Smart contracts routinely hold and manage liquid staking tokens — multisigs, DAOs, DeFi vaults, and yield aggregators are common examples. Many such contracts intentionally omit `receive()`/`fallback()` to prevent accidental ETH acceptance. Any such contract that calls `initiateWithdrawal` for ETH triggers this path deterministically. No special permissions, front-running, or external compromise is required — only a standard withdrawal initiation from a non-payable contract address.

## Recommendation
1. **Preferred fix:** Add a `recipient` parameter to `initiateWithdrawal` (defaulting to `msg.sender`) and store it in `WithdrawalRequest`. `_processWithdrawalCompletion` then sends ETH to the stored recipient rather than the requesting address.
2. **Minimal fix:** Add an admin/operator function that allows updating the recipient address of an unlocked-but-uncompleted withdrawal request, enabling recovery when the original address is non-payable.
3. **Complementary:** Remove or correct the misleading NatDoc on `completeWithdrawalForUser` (L191) that states it is "not expected to be used for ETH," since this implies ETH cases are safe to ignore when in fact they are the only path that can cause permanent fund loss. [8](#0-7) 

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// NonPayableWithdrawer: no receive() / fallback()
contract NonPayableWithdrawer {
    ILRTWithdrawalManager wm;
    IERC20 rsETH;

    constructor(address _wm, address _rsETH) {
        wm = ILRTWithdrawalManager(_wm);
        rsETH = IERC20(_rsETH);
    }

    function doInitiate(uint256 amount) external {
        rsETH.approve(address(wm), amount);
        wm.initiateWithdrawal(ETH_TOKEN, amount, "");
    }
}

function testPermanentFreeze() public {
    // 1. Deploy non-payable contract and fund it with rsETH
    NonPayableWithdrawer victim = new NonPayableWithdrawer(address(wm), address(rsETH));
    deal(address(rsETH), address(victim), 1 ether);

    // 2. Initiate withdrawal from non-payable contract
    vm.prank(address(victim));
    victim.doInitiate(1 ether);

    // 3. Operator unlocks the queue — rsETH burned, ETH moved into LRTWithdrawalManager
    vm.prank(operator);
    wm.unlockQueue(ETH_TOKEN, type(uint256).max, 0, 0, type(uint256).max, type(uint256).max);

    // 4. completeWithdrawal always reverts
    vm.prank(address(victim));
    vm.expectRevert(LRTWithdrawalManager.EthTransferFailed.selector);
    wm.completeWithdrawal(ETH_TOKEN, "");

    // 5. completeWithdrawalForUser also always reverts
    vm.prank(operator);
    vm.expectRevert(LRTWithdrawalManager.EthTransferFailed.selector);
    wm.completeWithdrawalForUser(ETH_TOKEN, address(victim), "");

    // 6. sweepRemainingAssets is blocked
    vm.prank(manager);
    vm.expectRevert(LRTWithdrawalManager.PendingWithdrawalsExist.selector);
    wm.sweepRemainingAssets(ETH_TOKEN);

    // 7. ETH is permanently stuck — rsETH already burned
    assertGt(address(wm).balance, 0); // ETH locked forever
    assertEq(rsETH.balanceOf(address(victim)), 0); // rsETH gone
}
```
