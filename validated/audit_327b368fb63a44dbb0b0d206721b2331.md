### Title
`addNewSupportedAsset` without oracle desynchronizes `LRTConfig.supportedAssetList` and `LRTOracle.assetPriceOracle`, permanently breaking `updateRSETHPrice()` - (File: contracts/LRTOracle.sol / contracts/LRTConfig.sol)

---

### Summary

`LRTConfig.addNewSupportedAsset()` appends an asset to `supportedAssetList` without atomically registering a price oracle in `LRTOracle`. `LRTOracle._getTotalEthInProtocol()` iterates over every entry in `supportedAssetList` and calls `getAssetPrice(asset)`, which hard-reverts via the `onlySupportedOracle` modifier when `assetPriceOracle[asset] == address(0)`. Once users deposit the new asset (which only requires `isSupportedAsset`), `removeSupportedAsset` is also blocked by the deposit-balance guard, making the desynchronization irrecoverable without a valid oracle.

---

### Finding Description

**`LRTConfig.addNewSupportedAsset`** pushes a new token into `supportedAssetList` and sets `isSupportedAsset[asset] = true`, but does not touch `LRTOracle.assetPriceOracle`:

```solidity
// LRTConfig.sol
function _addNewSupportedAsset(address asset, uint256 depositLimit) private {
    ...
    isSupportedAsset[asset] = true;
    supportedAssetList.push(asset);          // registry grows
    depositLimitByAsset[asset] = depositLimit;
    emit AddedNewSupportedAsset(asset, depositLimit);
}
``` [1](#0-0) 

**`LRTOracle._getTotalEthInProtocol`** then iterates over the full `supportedAssetList` and calls `getAssetPrice(asset)` for each entry:

```solidity
// LRTOracle.sol
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    uint256 assetER = getAssetPrice(asset);   // reverts if oracle == address(0)
    ...
}
``` [2](#0-1) 

**`getAssetPrice`** enforces the `onlySupportedOracle` modifier, which reverts unconditionally when `assetPriceOracle[asset] == address(0)`:

```solidity
modifier onlySupportedOracle(address asset) {
    if (assetPriceOracle[asset] == address(0)) {
        revert AssetOracleNotSupported();
    }
    _;
}
``` [3](#0-2) 

The two registries are managed independently with no atomic coupling. `updatePriceOracleFor` can only set a non-zero oracle for a currently-supported asset, so the admin cannot zero-out the oracle for a supported asset to "undo" the addition:

```solidity
function updatePriceOracleFor(address asset, address priceOracle) public onlyLRTAdmin {
    if (lrtConfig.isSupportedAsset(asset)) {
        UtilLib.checkNonZeroAddress(priceOracle);   // blocks zero-out while supported
    }
    assetPriceOracle[asset] = priceOracle;
}
``` [4](#0-3) 

The escape hatch — `removeSupportedAsset` — is blocked once any user deposits the new asset, because it guards on `getTotalAssetDeposits(asset) > maxNegligibleAmount`:

```solidity
if (ILRTDepositPool(depositPool).getTotalAssetDeposits(asset) > maxNegligibleAmount) {
    revert CannotRemoveAssetWithDeposits(asset);
}
``` [5](#0-4) 

`depositAsset` only checks `isSupportedAsset`, not whether an oracle exists, so users can deposit immediately after the asset is added and before the oracle is set.

---

### Impact Explanation

Once the desynchronization is locked in (asset added, users deposited, no valid oracle available), `updateRSETHPrice()` — a public function — permanently reverts. This freezes:

- **Protocol fee minting**: `_updateRsETHPrice` is the only path that mints protocol fees as rsETH; it is permanently blocked.
- **Price staleness**: The stored `rsETHPrice` cannot be updated, causing all subsequent deposits and withdrawals to use a stale exchange rate.

Impact: **Medium — permanent freezing of unclaimed yield** (protocol fees cannot be minted) and **Low — contract fails to deliver promised returns** (stale price for all users).

---

### Likelihood Explanation

The scenario requires two sequential steps that are both plausible in normal operations:

1. Admin adds a new LST asset via `addNewSupportedAsset` before a valid oracle is deployed or configured (a realistic operational ordering, especially for newly listed assets).
2. At least one user deposits the new asset before the oracle is set (likely given public deposit access and no oracle check in `depositAsset`).

Once both steps occur, the state is irrecoverable without a contract upgrade. The M-12 reference finding was accepted at Medium under the same reasoning (admin operational mistake + no recovery path).

---

### Recommendation

Enforce oracle registration atomically with asset addition. Either:

1. Require `LRTOracle.assetPriceOracle[asset] != address(0)` inside `_addNewSupportedAsset`, or
2. Add a combined admin function that calls both `_addNewSupportedAsset` and `updatePriceOracleFor` atomically, or
3. Make `_getTotalEthInProtocol` skip assets with no oracle rather than reverting, so a missing oracle degrades gracefully instead of bricking the price update.

---

### Proof of Concept

1. Admin calls `LRTConfig.addNewSupportedAsset(newAsset, depositLimit)` — `newAsset` enters `supportedAssetList`; `LRTOracle.assetPriceOracle[newAsset]` remains `address(0)`.
2. Any user calls `LRTDepositPool.depositAsset(newAsset, amount)` — succeeds because only `isSupportedAsset` is checked.
3. Anyone calls `LRTOracle.updateRSETHPrice()` (public, no access control).
4. Execution reaches `_getTotalEthInProtocol()` → `getAssetPrice(newAsset)` → `onlySupportedOracle` modifier fires → `revert AssetOracleNotSupported()`.
5. Admin attempts `LRTConfig.removeSupportedAsset(newAsset, idx)` → reverts with `CannotRemoveAssetWithDeposits` because user deposited in step 2.
6. Admin attempts `LRTOracle.updatePriceOracleFor(newAsset, address(0))` → reverts because `isSupportedAsset(newAsset) == true` enforces non-zero oracle.
7. `updateRSETHPrice()` is permanently broken; protocol fee minting halts; rsETH price is frozen.
