I need to trace the full call chain for `depositETH`, check how `rsETHPrice` is computed and stored, and determine whether a timing attack around `updateRSETHPrice()` can let an unprivileged caller capture accumulated rewards.

### Title
Stale `rsETHPrice` Allows Depositor to Capture Unclaimed Yield Before Price Update — (`contracts/LRTDepositPool.sol::depositETH` / `contracts/LRTOracle.sol::updateRSETHPrice`)

---

### Summary

`depositETH` mints rsETH using a stored, potentially stale `rsETHPrice`. Because `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` are both permissionless, an unprivileged EOA can atomically: (1) flush accumulated rewards into the deposit pool, (2) deposit ETH at the stale (pre-reward) price, and (3) trigger the price update — capturing a share of yield that belonged to existing rsETH holders.

---

### Finding Description

**Mint-rate formula (no freshness check):**

`getRsETHAmountToMint` computes:

```
rsethAmountToMint = (msg.value × getAssetPrice(ETH)) / rsETHPrice
```

`getAssetPrice(ETH_TOKEN)` always returns `1e18` (from `OneETHPriceOracle`). `rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` is explicitly called. [1](#0-0) 

**`updateRSETHPrice()` is public with no access control:** [2](#0-1) 

**`FeeReceiver.sendFunds()` is also public with no access control** — anyone can push accumulated MEV/EL rewards into the deposit pool at will: [3](#0-2) 

**`getETHDistributionData()` uses live `address(this).balance`**, so rewards pushed by `sendFunds()` immediately inflate TVL: [4](#0-3) 

**`depositETH` never refreshes `rsETHPrice` before minting:** [5](#0-4) 

**ETH deposit-limit check omits the incoming amount** (separate bug, but confirms the deposit is not blocked): [6](#0-5) 

---

### Impact Explanation

Existing rsETH holders accumulate yield as ETH rewards raise TVL without increasing rsETH supply. That yield is "unclaimed" until `updateRSETHPrice()` is called. An attacker who deposits at the stale price and then triggers the update dilutes existing holders' yield proportionally to their deposit size.

**Numerical example:**

| State | TVL | rsETH supply | Stored rsETHPrice |
|---|---|---|---|
| Before rewards | 100 ETH | 100 | 1.000 |
| After `sendFunds()` (5 ETH rewards) | 105 ETH | 100 | 1.000 (stale) |
| Attacker deposits 100 ETH at stale price | 205 ETH | 200 | 1.000 (stale) |
| After `updateRSETHPrice()` | 205 ETH | 200 | 1.025 |

- Attacker deposited 100 ETH, now holds 100 rsETH × 1.025 = **102.5 ETH** (+2.5 ETH profit)
- Original holders' 100 rsETH = 102.5 ETH instead of the 105 ETH they were owed (**−2.5 ETH stolen yield**)

Impact: **High. Theft of unclaimed yield.** (The question's claimed Critical/direct-theft-of-principal does not apply; the attacker captures yield, not depositor principal.)

---

### Likelihood Explanation

- No privileged role required
- No front-running required — all three steps (`sendFunds`, `depositETH`, `updateRSETHPrice`) are callable by the same EOA in consecutive transactions within the same block
- Rewards accumulate continuously (MEV, EL rewards); the attack window is always open
- Profit scales with deposit size and accumulated reward amount

---

### Recommendation

1. **Call `_updateRsETHPrice()` (or at minimum read a freshly computed price) inside `_beforeDeposit`** before computing `rsethAmountToMint`, so deposits always use a price that reflects the current TVL.
2. Alternatively, restrict `FeeReceiver.sendFunds()` to a privileged role so reward flushing cannot be weaponized by an attacker.
3. As a defense-in-depth measure, fix the ETH deposit-limit check to include the incoming amount: `totalAssetDeposits + amount > depositLimit` (matching the ERC-20 path).

contract YieldDiversionTest is Test {
    LRTDepositPool depositPool;
    LRTOracle oracle;
    FeeReceiver feeReceiver;
    RSETH rseth;
    // ... (setup omitted for brevity)

    function test_stalePrice_yieldTheft() public {
        // 1. Simulate 5 ETH of MEV rewards sitting in FeeReceiver
        vm.deal(address(feeReceiver), 5 ether);

        // Record existing holder's rsETH balance (100 rsETH minted at genesis)
        uint256 holderBalanceBefore = rseth.balanceOf(existingHolder);

        // 2. Attacker: flush rewards into deposit pool (public, no auth)
        vm.prank(attacker);
        feeReceiver.sendFunds();
        // rsETHPrice is now STALE — still reflects pre-reward TVL

        // 3. Attacker: deposit at stale price
        vm.deal(attacker, 100 ether);
        vm.prank(attacker);
        depositPool.depositETH{value: 100 ether}(0, "");

        // 4. Attacker: trigger price update (public, no auth)
        vm.prank(attacker);
        oracle.updateRSETHPrice();

        // 5. Assert attacker captured yield
        uint256 attackerRsETH = rseth.balanceOf(attacker);
        uint256 attackerETHValue = attackerRsETH * oracle.rsETHPrice() / 1e18;
        assertGt(attackerETHValue, 100 ether, "Attacker profited from stale price");

        // 6. Assert existing holders lost yield
        // Their 100 rsETH should be worth 105 ETH but is now worth ~102.5 ETH
        uint256 holderETHValue = holderBalanceBefore * oracle.rsETHPrice() / 1e18;
        assertLt(holderETHValue, 105 ether, "Existing holders lost unclaimed yield");
    }
}
