### Title
Fee-on-Transfer Token Undercollateralization Causes Frozen Redemption Path for EVM Depositors - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer` records and broadcasts the caller-supplied `amount` for plain ERC20 tokens without verifying the actual amount received. For fee-on-transfer tokens (e.g., stETH, which delivers 1–2 wei less than requested), the bridge becomes undercollateralized. The NEAR side credits the full `amount`, but when users redeem back to EVM, `finTransfer` attempts to release the full signed amount from a balance that is short by the transfer tax, causing a permanent revert and frozen redemption.

### Finding Description
In `OmniBridge.initTransfer`, the plain ERC20 branch (the `else` at line 406) calls:

```solidity
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
``` [1](#0-0) 

The contract then immediately emits `BridgeTypes.InitTransfer` with the caller-supplied `amount` — not the actual balance delta received:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
``` [2](#0-1) 

The NEAR bridge observes this event and mints `amount` wrapped tokens to the user. No balance-before/after check is performed; the contract blindly trusts the caller-supplied `amount`.

On the return path, `finTransfer` releases tokens from the bridge's own ERC20 balance:

```solidity
IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
``` [3](#0-2) 

`payload.amount` is the NEAR-signed amount (`amount - bridge_fee`). Because the bridge only holds `amount - transfer_tax` (where `transfer_tax` is the fee-on-transfer token's deduction), if `transfer_tax > bridge_fee` the `safeTransfer` reverts. With zero bridge fee and a 1-wei stETH rounding loss, the bridge holds exactly 1 wei less than it must release, causing every redemption to revert.

The discrepancy accumulates with each deposit: N deposits of `amount` each leave the bridge short by `N × transfer_tax` tokens while NEAR has minted `N × amount` wrapped tokens. The last redeemers are permanently locked out.

### Impact Explanation
This is a **Critical** frozen redemption path. Wrapped tokens on NEAR are fully backed by the event-recorded `amount`, but the EVM bridge holds less. Users who deposited fee-on-transfer tokens cannot redeem their NEAR-side wrapped tokens back to EVM. Their NEAR-side tokens are burned during the redemption attempt, but the EVM `finTransfer` reverts, leaving them with neither the EVM token nor the NEAR wrapped token (the nonce is marked used at line 287 before the transfer executes, so the redemption cannot be retried). [4](#0-3) 

### Likelihood Explanation
Medium. The `initTransfer` plain ERC20 path has no token whitelist; any ERC20 can be deposited. stETH is a widely held, high-value token with a documented 1–2 wei rounding behavior on every transfer. A user depositing stETH triggers the bug without any privileged access or special setup. The attacker is strictly unprivileged and enters through the public `initTransfer` entrypoint.

### Recommendation
Measure the actual received amount by comparing the contract's token balance before and after the `safeTransferFrom` call, and use that delta — not the caller-supplied `amount` — for all subsequent accounting and event emission:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint128 actualReceived = uint128(IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore);
// use actualReceived instead of amount going forward
```

### Proof of Concept
1. Alice calls `initTransfer(stETH, 1e18, 0, 0, "near:alice", "")` on EVM.
2. `safeTransferFrom` delivers `1e18 - 1` wei of stETH to the bridge (1-wei rounding loss).
3. `InitTransfer` event emits `amount = 1e18`.
4. NEAR bridge mints `1e18` wrapped stETH to Alice on NEAR.
5. Alice calls the NEAR bridge to redeem; NEAR burns `1e18` wrapped stETH and signs a `finTransfer` payload for `amount = 1e18` (bridge fee = 0).
6. Relayer calls `finTransfer` on EVM with `payload.amount = 1e18`.
7. `completedTransfers[nonce] = true` is set (line 287).
8. `safeTransfer(alice, 1e18)` reverts because the bridge holds only `1e18 - 1` wei.
9. The entire transaction reverts, the nonce is not persisted (state reverts too), so the relayer retries — but every retry reverts for the same reason. Alice's NEAR-side tokens are burned and her EVM tokens are permanently locked in the bridge.
