### Title
Floor-Truncated Refresh Randomness Silently Discards Entropy in EdDSA HD-MPC Key Refresh — (File: `src/cbmpc/protocol/hd_keyset_eddsa_2p.cpp`)

---

### Summary

In `key_share_eddsa_hdmpc_2p_t::refresh`, the byte count for the agreed-upon refresh randomness is computed with `bits_to_bytes_floor` (floor division `bits >> 3`) instead of `bits_to_bytes` (ceiling). When the curve order bit-length is not a multiple of 8 — as with Ed25519 (255-bit order) — the protocol generates the full `2 × rand_bitlen` bits via `agree_random`, but then silently discards the trailing partial-byte worth of bits from each half. The consumed randomness is shorter than intended, reducing the effective statistical security parameter for the refresh blinding values.

---

### Finding Description [1](#0-0) 

```cpp
int rand_bitlen = q.get_bits_count() + SEC_P_STAT;   // e.g. 255 + 64 = 319 for Ed25519
int rand_size = bits_to_bytes_floor(rand_bitlen);     // 319 >> 3 = 39 bytes (312 bits)
if (rv = agree_random(job, 2 * rand_bitlen, rand)) return rv;
                                                      // generates bits_to_bytes(638) = 80 bytes

bn_t r_x = bn_t::from_bin(rand.take(rand_size)) % q;          // consumes 39 bytes
bn_t r_k = bn_t::from_bin(rand.skip(rand_size).take(rand_size)) % q; // consumes next 39 bytes
```

`bits_to_bytes_floor` is defined as a pure floor shift: [2](#0-1) 

```cpp
inline int bits_to_bytes_floor(int bits) {
  cb_assert(bits >= 0);
  return bits >> 3;   // truncates, no rounding up
}
```

Every other sampling site in the library uses the ceiling variant `bits_to_bytes` (`(bits + 7) >> 3`): [3](#0-2) [4](#0-3) 

For Ed25519 (`q` = 255 bits, `SEC_P_STAT` = 64):

| Value | Ceiling (`bits_to_bytes`) | Floor (`bits_to_bytes_floor`) |
|---|---|---|
| `rand_bitlen` | 319 bits | 319 bits |
| `rand_size` | 40 bytes (320 bits) | **39 bytes (312 bits)** |
| `agree_random` output | 80 bytes | 80 bytes |
| bytes actually used | 80 bytes | **78 bytes** |
| bits discarded | 0 | **16 bits (2 bytes)** |
| statistical security of `r_x`, `r_k` | 320 − 255 = **64 bits** | 312 − 255 = **57 bits** |

The two trailing bytes produced by `agree_random` are never read; they are silently dropped. This is the direct analog of the Gumball swap bug: the full agreed-upon randomness is generated (both parties execute the commit-and-reveal protocol), but only the floor-truncated slice is consumed, and the remainder is wasted.

---

### Impact Explanation

The refresh blinding values `r_x` and `r_k` are used to re-randomize the EdDSA HD-MPC key shares: [5](#0-4) 

Their purpose is to provide forward secrecy: after a refresh, a party whose old share is later compromised should not be able to reconstruct the new share. The statistical security of this guarantee depends on `r_x` and `r_k` being drawn from a range at least `2^(|q| + SEC_P_STAT)`. With the floor truncation, the effective range is `2^312` instead of `2^319` for Ed25519, reducing the statistical security margin from 64 bits to 57 bits — a 7-bit shortfall against the library's own `SEC_P_STAT` target.

---

### Likelihood Explanation

The bug is deterministic and affects every call to `key_share_eddsa_hdmpc_2p_t::refresh` on any curve whose order bit-length is not a multiple of 8. Ed25519 (255-bit order) is the primary affected curve. The function is part of the shipped protocol layer and is reachable through the internal API used by integrators.

---

### Recommendation

Replace `bits_to_bytes_floor` with `bits_to_bytes` (ceiling) on line 67:

```cpp
// Before (wrong):
int rand_size = bits_to_bytes_floor(rand_bitlen);

// After (correct):
int rand_size = coinbase::bits_to_bytes(rand_bitlen);
```

This matches the pattern used everywhere else in the codebase for sampling randomness of a given bit length.

---

### Proof of Concept

For Ed25519 (`q.get_bits_count() == 255`, `SEC_P_STAT == 64`):

1. `rand_bitlen = 319`
2. `rand_size = bits_to_bytes_floor(319) = 39` (floor of 39.875)
3. `agree_random(job, 638, rand)` → `rand` is `bits_to_bytes(638) = 80` bytes
4. `r_x` is derived from `rand[0..38]` (39 bytes = 312 bits)
5. `r_k` is derived from `rand[39..77]` (39 bytes = 312 bits)
6. `rand[78..79]` (2 bytes) are never read
7. Both `r_x` and `r_k` are in `[0, 2^312)` instead of `[0, 2^319)`, giving `312 − 255 = 57` bits of statistical security instead of the intended 64 bits — a shortfall of 7 bits against `SEC_P_STAT`
