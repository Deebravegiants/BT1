### Title
Cross-Network Signature Replay of `DelegateAction` (Meta-Transaction) - (`File: core/primitives/src/action/delegate.rs`)

### Summary
The `DelegateAction` (NEP-366 meta-transaction) signature hash does not include any chain-specific identifier. A `SignedDelegateAction` created and signed on one NEAR network (e.g., testnet) can be replayed verbatim on a different NEAR network (e.g., mainnet) by any relayer who possesses the signed payload, provided the sender account and key exist on both networks and the nonce is valid on the target network.

### Finding Description

`DelegateAction.get_nep461_hash()` computes the signed hash as:

```
SHA-256( borsh( MessageDiscriminant(NEP_366) || DelegateAction ) )
``` [1](#0-0) 

`SignableMessage` contains only a `MessageDiscriminant` (a constant derived from the NEP number `366`) and the message body. Neither field carries any chain-specific value. [2](#0-1) [3](#0-2) 

The `DelegateAction` struct itself contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — but **no chain ID, genesis hash, or network name**. [4](#0-3) 

By contrast, normal NEAR transactions (`TransactionV0`, `TransactionV1`) include a `block_hash` field that ties the signature to a specific chain's block history, preventing cross-network replay. [5](#0-4) 

`DelegateAction` uses `max_block_height` (a plain integer) instead of a block hash. Block heights are not chain-specific — height 1,000,000 exists on both mainnet and testnet. The `MessageDiscriminant` is a constant (`MIN_ON_CHAIN_DISCRIMINANT + 366 = 0x4000016E`) that is identical across all NEAR networks. [6](#0-5) 

Signature verification in `SignedDelegateAction::verify()` checks only the signature against the hash of the action body — no chain context is consulted. [7](#0-6) 

### Impact Explanation

A user who signs a `DelegateAction` on testnet (e.g., `AddKey`, `Transfer`, `FunctionCall`, `DeleteKey`) and hands it to a relayer can have that exact signed payload submitted on mainnet by a malicious relayer. The runtime's `apply_delegate_action` will verify the signature, find it valid (same key, same hash), advance the nonce, and execute the inner actions — draining tokens, adding attacker-controlled keys, or deploying contracts — all without the user's mainnet consent.

The corrupted protocol values are: the sender's **balance** (for `Transfer`), the sender's **access key set** (for `AddKey`/`DeleteKey`), and the sender's **nonce** (advanced on the wrong network).

### Likelihood Explanation

- Many NEAR users maintain the same named account and the same key pair on both mainnet and testnet.
- Nonces on the two networks are independent; a testnet nonce of N is valid on mainnet if the mainnet access-key nonce is less than N.
- `max_block_height` is a block height integer that is easily satisfiable on either network.
- The relayer role is unprivileged: any party that receives a `SignedDelegateAction` off-chain (the normal meta-transaction flow) can submit it to any NEAR network.
- The meta-transaction documentation explicitly describes the relayer as an untrusted third party that receives the signed payload off-chain. [8](#0-7) 

### Recommendation

Include a chain-specific identifier in the `DelegateAction` signed payload. The cleanest fix is to add a `chain_id` (the NEAR chain's string identifier, e.g., `"mainnet"`, `"testnet"`) or the genesis block hash to `DelegateAction` and incorporate it into `get_nep461_hash()`. Alternatively, `SignableMessage` could be extended with a chain-binding field so all future signable message types inherit the protection automatically. [9](#0-8) 

### Proof of Concept

1. Alice has account `alice.near` on both mainnet and testnet, with the same ED25519 key pair. Her mainnet access-key nonce is 4; her testnet nonce is 10.
2. Alice signs a `DelegateAction` on testnet: `{ sender_id: "alice.near", receiver_id: "attacker.near", actions: [Transfer(100 NEAR)], nonce: 11, max_block_height: 200_000_000, public_key: <alice_pk> }`.
3. She sends the `SignedDelegateAction` to a testnet relayer.
4. The relayer (or any interceptor) instead wraps it in a **mainnet** transaction: `SignedTransaction { signer: relayer, receiver: alice.near, block_hash: <mainnet_block_hash>, actions: [Delegate(signed_delegate_action)] }`.
5. The mainnet node receives the transaction. `apply_delegate_action` calls `signed_delegate_action.verify()`, which recomputes `SHA-256(borsh(0x4000016E || DelegateAction_body))` — identical on both networks — and the signature checks out.
6. Nonce 11 > 4 (mainnet nonce), so the nonce check passes. `max_block_height` 200,000,000 is in the future on mainnet.
7. The transfer of 100 NEAR from Alice to `attacker.near` executes on mainnet, which Alice never authorized for mainnet. [10](#0-9) [11](#0-10)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }

**File:** docs/architecture/how/meta-tx.md (L40-52)
```markdown
With meta transactions, Alice can create a `DelegateAction`, which is very
similar to a transaction. It also contains a list of actions to execute and a
single receiver for those actions. She signs the `DelegateAction` and forwards
it (off-chain) to a relayer. The relayer wraps it in a transaction, of which the
relayer is the signer and therefore pays the gas costs. If the inner actions
have an attached token balance, this is also paid for by the relayer.

On chain, the `SignedDelegateAction` inside the transaction is converted to an
action receipt with the same `SignedDelegateAction` on the relayer's shard. The
receipt is forwarded to the account from `Alice`, which will unpacked the
`SignedDelegateAction` and verify that it is signed by Alice with a valid Nonce
etc. If all checks are successful, a new action receipt with the inner actions
as body is sent to `FT`. There, the `ft_transfer` call finally executes.
```

/// Validate access key which was used for signing DelegateAction:
///
/// - Checks whether the access key is present fo given public_key and sender_id.
/// - Validates nonce and updates it if it's ok.
/// - Validates access key permissions.
fn validate_delegate_action_key(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    delegate_action: VersionedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let sender_id = delegate_action.sender_id();
    let public_key = delegate_action.public_key();
    // 'sender_id' account existence must be checked by a caller
    let mut access_key = match get_access_key(state_update, sender_id, public_key)? {
        Some(access_key) => access_key,
        None => {
            result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
                InvalidAccessKeyError::AccessKeyNotFound {
                    account_id: sender_id.clone(),
                    public_key: public_key.clone().into(),
                },
            )
            .into());
            return Ok(());
        }
    };
