Audit Report

## Title
Missing Chainlink price validation (zero/negative price and staleness) in `ChainlinkPriceOracle` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and discards all return values except `price`, with no check for zero, negative, or stale values. A corrupted price propagates directly into `LRTOracle._getTotalEthInProtocol()` and then into `_updateRsETHPrice()`, which can trigger an automatic protocol-wide pause (temporary freezing of all deposits and withdrawals) or write a deflated rsETH price that dilutes existing holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at [1](#0-0)  performs no validation:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Three values are silently discarded: `roundId`, `updatedAt`, and `answeredInRound`. Consequently:

- **Zero price**: returns `0`, understating `totalETHInProtocol`.
- **Negative price**: `uint256(negativeInt256)` wraps to near `2^256` via two's complement, overstating `totalETHInProtocol` astronomically.
- **Stale price**: `updatedAt` is never compared to `block.timestamp`, so an arbitrarily old cached answer is accepted.

The sibling contract `ChainlinkOracleForRSETHPoolCollateral.getRate()` at [2](#0-1)  already applies all three checks (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`), confirming the pattern is known and intentionally applied elsewhere.

The corrupted price flows through `_getTotalEthInProtocol()` at [3](#0-2) , which accumulates `assetER * totalAssetAmt` for every supported asset, and into `_updateRsETHPrice()` at [4](#0-3) , which computes `newRsETHPrice`.

The entry point `updateRSETHPrice()` at [5](#0-4)  is `public` with no access control beyond `whenNotPaused`, so any unprivileged address can trigger the mispricing.

The downside-protection logic at [6](#0-5)  pauses `lrtDepositPool`, `withdrawalManager`, and the oracle itself when `newRsETHPrice` drops more than `pricePercentageLimit` below `highestRsethPrice`. A zero or sufficiently stale price will produce exactly this drop, triggering the auto-pause.

## Impact Explanation
**Primary — Medium: Temporary freezing of funds.**
A zero price (or a stale price sufficiently below the current `highestRsethPrice`) causes `newRsETHPrice` to fall beyond `pricePercentageLimit`. The auto-pause fires, freezing all user deposits and withdrawals across `lrtDepositPool` and `withdrawalManager` until an admin manually unpauses. No privileged actor is required to trigger this; any address calling the public `updateRSETHPrice()` after the feed goes stale/zero suffices.

**Secondary — Low: Contract fails to deliver promised returns.**
If the stale/zero price drop falls within `pricePercentageLimit` (or `pricePercentageLimit` is unset/zero), `rsETHPrice` is written at the deflated value. Subsequent depositors mint rsETH at the artificially low price, receiving more rsETH than fair value and diluting all existing rsETH holders.

## Likelihood Explanation
Chainlink feeds have historically returned zero during circuit-breaker events (minAnswer/maxAnswer clamp) and stale prices during extreme network congestion or L2 sequencer outages. The exploit requires no special access: `updateRSETHPrice()` is public and callable by any EOA or contract. The condition (a feed anomaly) is rare but documented and has occurred on mainnet. Likelihood: **Low**.

## Recommendation
Replace the bare `latestRoundData` call in `ChainlinkPriceOracle.getAssetPrice()` with full validation, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId)  revert StalePrice();
if (updatedAt == 0)             revert IncompleteRound();
if (price <= 0)                 revert InvalidPrice();
if (block.timestamp - updatedAt > HEARTBEAT + BUFFER) revert StalePrice();
```

Per-feed heartbeat constants (e.g., 3600 s for 1-hour feeds, 86 400 s for 24-hour feeds) should be stored alongside `assetPriceFeed`. For L2 deployments, additionally consult the Chainlink sequencer uptime feed and enforce the `GRACE_PERIOD` before consuming any price.

## Proof of Concept
1. A Chainlink ETH/stETH feed goes stale (no update for >24 h during congestion) and returns a price 10% below the true market price.
2. Any EOA calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale low price without reverting.
4. `totalETHInProtocol` is understated; `newRsETHPrice` is computed ~10% below `highestRsethPrice`.
5. If `pricePercentageLimit` is set to ≤10% (e.g., 5%), the drop triggers `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()` — all user deposits and withdrawals are frozen until an admin manually unpauses.
6. If `pricePercentageLimit` is unset (0) or the drop is within the limit, `rsETHPrice` is written at the deflated value; a depositor immediately minting rsETH at this price receives proportionally more rsETH than fair value, diluting all existing holders.

**Foundry fork test outline:**
```solidity
// Fork mainnet, mock the stETH/ETH Chainlink feed to return a stale answer
// (block.timestamp - updatedAt > heartbeat) with price 10% below current.
// Call lrtOracle.updateRSETHPrice() from address(0xBEEF).
// Assert lrtDepositPool.paused() == true.
```
