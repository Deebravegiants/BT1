### Title
Zero rsETH/wrsETH Minted on Small Deposits Due to Floor-Division Rounding in `viewSwapRsETHAmountAndFee` — (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

The L2 pool deposit functions compute the rsETH amount to mint using floor (truncating) integer division. When the deposited amount is small relative to the current rsETH/ETH rate, the computed `rsETHAmount` rounds down to zero. No zero-check is performed before minting, so a depositor can send ETH (or a supported token) and receive zero wrsETH in return, permanently losing their deposited assets to the pool.

---

### Finding Description

In every L2 pool variant (`RSETHPoolV3`, `RSETHPoolV2`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`), the rsETH amount is computed as:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0) 

This is a plain Solidity floor division. When `amountAfterFee * 1e18 < rsETHToETHrate`, the result truncates to `0`. The `deposit` function then calls `wrsETH.mint(msg.sender, 0)` without any guard:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // rsETHAmount may be 0
``` [2](#0-1) 

The only input guard is `if (amount == 0) revert InvalidAmount()` — there is no minimum-rsETH-out parameter and no post-computation zero check.

The same pattern is present in the token-deposit overload:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [3](#0-2) 

The same root cause exists in `LRTDepositPool.getRsETHAmountToMint`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

There, `_beforeDeposit` checks `rsethAmountToMint < minRSETHAmountExpected`, but if the caller passes `minRSETHAmountExpected = 0` (the default), a zero-mint deposit is accepted silently. [5](#0-4) 

---

### Impact Explanation

A depositor who sends a non-zero ETH (or token) amount that is smaller than `rsETHToETHrate / 1e18` wei receives zero wrsETH. Their ETH is retained by the pool contract with no corresponding share minted, constituting a direct loss of deposited funds. The ETH is not returned; it becomes part of the pool's balance available for bridging or fee accounting, effectively transferred away from the depositor.

**Impact:** Low — Contract fails to deliver promised returns (depositor loses deposited ETH/tokens without receiving any wrsETH).

---

### Likelihood Explanation

The rsETH/ETH rate (`rsETHToETHrate`) starts at `1e18` and grows monotonically as restaking yield accrues. Once the rate exceeds `1e18` (which occurs naturally over time), any deposit of exactly 1 wei of ETH produces `rsETHAmount = 0`. The condition is always reachable by any unprivileged caller after the protocol has been live for any meaningful period. The affected deposit amount per call is tiny (1–2 wei), but there is no on-chain protection preventing repeated calls.

**Likelihood:** Low — Only affects deposits smaller than `rsETHToETHrate / 1e18` wei; normal user deposits are far larger and unaffected.

---

### Recommendation

Add a zero-check on the computed rsETH amount before minting in all `deposit` functions:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, expose a `minRsETHAmountExpected` parameter (as `LRTDepositPool` does) so callers can enforce a slippage floor. The same fix should be applied to the token-deposit overload and to `LRTDepositPool._beforeDeposit` (enforce `minRSETHAmountExpected > 0` or revert when `rsethAmountToMint == 0`).

---

### Proof of Concept

1. The rsETH/ETH oracle rate is `1.05e18` (rsETH worth 1.05 ETH, a realistic post-yield value).
2. A user calls `RSETHPoolV3.deposit{value: 1}("")` (1 wei of ETH, `feeBps = 0` for simplicity).
3. `viewSwapRsETHAmountAndFee(1)` computes: `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (floor division).
4. `wrsETH.mint(msg.sender, 0)` executes — zero tokens minted.
5. The user's 1 wei ETH is retained by the pool; the user holds no wrsETH and has no claim to the deposited ETH.

The same outcome occurs in `RSETHPoolV2`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge` via their respective `viewSwapRsETHAmountAndFee` implementations. [6](#0-5) [7](#0-6) [8](#0-7)
