Audit Report

## Title
Missing L2 Sequencer Uptime Check Enables Stale Price Exploitation on Arbitrum - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

## Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls `AggregatorV3Interface(oracle).latestRoundData()` without verifying the Arbitrum sequencer is live or that a grace period has elapsed after it restarts. When the sequencer goes down and prices move, the stale price passes all three existing guards and flows directly into `RSETHPool.viewSwapRsETHAmountAndFee()`, causing `deposit()` to over-mint wrsETH relative to the real value of the deposited collateral, draining the pool's wrsETH reserve.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs three checks after calling `latestRoundData()`:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [1](#0-0) 

None of these detect sequencer downtime. When the Arbitrum sequencer is offline, Chainlink oracle updates are not processed on-chain. The last committed answer satisfies `answeredInRound >= roundID`, `timestamp != 0`, and `ethPrice > 0`, so all three guards pass and the stale price is returned.

The most realistic attack window is immediately after the sequencer restarts: the sequencer comes back online before the Chainlink oracle has been updated, leaving a window where the stale (pre-downtime) price is still returned. No force-inclusion delay is required — the attacker simply calls `deposit(token, amount, referralId)` through the normal sequencer path the moment it resumes.

`RSETHPool` is explicitly the Arbitrum pool: [2](#0-1) 

The stale rate flows directly into the minting calculation via `supportedTokenOracle[token].getRate()`: [3](#0-2) 

`rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` — if `tokenToETHRate` is stale-high, the attacker receives more wrsETH than the deposited collateral is worth.

`ChainlinkPriceOracle.getAssetPrice()` has the same gap, discarding all return values except `price` with no sequencer check: [4](#0-3) 

## Impact Explanation
**Critical — Direct theft of any user funds.** The attacker deposits collateral tokens at a stale inflated price and receives more wrsETH than the deposited collateral is worth. The excess wrsETH is extracted from the pool's reserve at the expense of other depositors and liquidity providers. This is a direct, immediate transfer of value out of the pool with no recovery mechanism.

## Likelihood Explanation
Arbitrum sequencer outages have occurred (e.g., December 2022). The primary attack window — the period immediately after the sequencer restarts before the Chainlink oracle is updated — requires no special privilege, no force-inclusion delay, and no victim mistake. Any unprivileged user can call `deposit(token, amount, referralId)` on `RSETHPool`. The only precondition is that the collateral token's price dropped during the outage, which is a realistic market condition during any significant downtime event. The attack is repeatable across multiple deposits within the same window.

## Recommendation
Add a sequencer uptime check with a grace period in `ChainlinkOracleForRSETHPoolCollateral.getRate()` following Chainlink's documented pattern:

```solidity
AggregatorV3Interface internal sequencerUptimeFeed;
uint256 private constant GRACE_PERIOD = 3600; // 1 hour

function getRate() public view returns (uint256) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();

    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();
    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();

    uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
    return normalizedPrice;
}
```

Apply the same fix to `ChainlinkPriceOracle.getAssetPrice()`, adding staleness checks (`answeredInRound`, `timestamp`) alongside the sequencer check.

## Proof of Concept
1. Arbitrum sequencer goes offline. Last committed wstETH/ETH Chainlink price: `1.15e18`.
2. Real market price of wstETH drops to `1.05e18` during the outage.
3. Sequencer restarts. Chainlink oracle has not yet been updated on-chain.
4. Attacker immediately calls `RSETHPool.deposit(wstETH, 100e18, "")`.
5. `viewSwapRsETHAmountAndFee(100e18, wstETH)` calls `ChainlinkOracleForRSETHPoolCollateral.getRate()`.
6. `latestRoundData()` returns stale `1.15e18`; all three guards pass.
7. `rsETHAmount = 100e18 * 1.15e18 / rsETHToETHrate` — attacker receives wrsETH valued at ~1.15 ETH per wstETH, while each wstETH is worth ~1.05 ETH.
8. Attacker gains ~9.5% excess wrsETH per deposit, redeemable for real ETH value, draining the pool reserve.

**Foundry fork test plan:** Fork Arbitrum mainnet at a block just after a historical sequencer restart. Mock the Chainlink feed to return the pre-downtime price. Call `RSETHPool.deposit(wstETH, largeAmount, "")` and assert that `rsETHAmount` exceeds the fair value computed using the post-downtime spot price, confirming the over-mint.
