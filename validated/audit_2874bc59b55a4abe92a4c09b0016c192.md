### Title
Bootstrap-Phase DRep Reverse-Delegation Staleness Allows Governance Vote-Weight Manipulation — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version < 10), `processDelegationInternal` intentionally preserves a known inconsistency (issue #4772) where redelegating a stake credential to a new DRep does **not** remove the credential from the old DRep's reverse-delegation set (`drepDelegs`). When the old DRep subsequently unregisters via `ConwayUnRegDRep`, the `clearDRepDelegations` helper unconditionally nullifies the forward delegation of every credential in the stale `drepDelegs` set — including credentials whose current forward delegation already points to a **different, still-registered DRep**. This silently zeroes out those credentials' contributions to the new DRep's stake in the governance distribution, enabling an attacker-controlled DRep operator to reduce a competing DRep's ratification weight.

---

### Finding Description

**Two independent tracking mechanisms** represent the same DRep-delegation relationship:

| Mechanism | Location | Direction |
|---|---|---|
| Forward delegation | `certDStateL . accountsL . accountsMapL` (per-credential `dRepDelegationAccountStateL`) | credential → DRep |
| Reverse delegation | `certVStateL . vsDRepsL` (`DRepState.drepDelegs`) | DRep → Set Credential |

**Root cause — `processDelegationInternal` (Deleg.hs lines 347–377):**

When `preserveIncorrectDelegation = True` (i.e., `pvMajor pv < natVersion @10`), the `delegVote` branch only **inserts** the credential into the new DRep's `drepDelegs` without **removing** it from the old DRep's `drepDelegs`:

```haskell
| isNothing mAccountState || preserveIncorrectDelegation ->
    certVStateL . vsDRepsL
      %~ Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
```

The correct path (post-bootstrap) calls `unDelegReDelegDRep`, which atomically removes from the old DRep and inserts into the new one (`VState.hs` lines 129–143). The bootstrap path skips the removal entirely.

**Consequence — `ConwayUnRegDRep` (GovCert.hs lines 244–254):**

```haskell
clearDRepDelegations delegs accountsMap =
  foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
```

`clearDRepDelegations` iterates over the unregistering DRep's `drepDelegs` and **unconditionally** sets each credential's forward delegation to `Nothing`, regardless of whether that credential's current forward delegation still points to the unregistering DRep. Because `drepDelegs` contains stale entries from the bootstrap-phase redelegation, credentials that have already moved to a new DRep have their current (valid) forward delegation wiped.

This is confirmed by the test at `eras/conway/impl/testlib/Test/Cardano/Ledger/Conway/Imp/DelegSpec.hs` lines 313–323, which explicitly asserts that in bootstrap phase, `lookupDRepDelegation cred accounts` returns `Nothing` after the old DRep unregisters — even though the credential had already redelegated to a second DRep.

---

### Impact Explanation

The DRep stake distribution used in ratification is computed by `computeDRepDistr` in `DRepPulser.hs` (lines 391–397), which iterates over the accounts map (forward delegations). When a credential's forward delegation is incorrectly cleared to `Nothing`, its stake is excluded from every DRep's total in `reDRepDistr`. This directly feeds `dRepAcceptedRatio` in `Ratify.hs` (lines 252–281):

```haskell
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
```

Reducing the stake attributed to a target DRep lowers both the numerator (if that DRep voted Yes) and the denominator, with the net effect depending on the vote. An attacker can selectively suppress a Yes-voting DRep's weight, causing a governance action (hard-fork initiation, committee update, constitution change, treasury withdrawal, parameter change) to fail to meet its ratification threshold — or conversely inflate the ratio for a No-voting DRep to block a legitimate action.

**Allowed impact matched:** *Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted (or blocked).*

---

### Likelihood Explanation

The attack requires:
1. The network to be in the Conway bootstrap phase (protocol version 9, before the v10 hard fork).
2. At least one stake credential to have redelegated from the attacker's DRep to a target DRep.
3. The attacker to unregister their DRep.

Steps 1 and 3 are fully attacker-controlled. Step 2 is achievable by registering an attractive DRep (e.g., with a compelling anchor/metadata), accumulating delegations, then waiting for natural redelegation churn. The attacker recovers their DRep deposit upon unregistration, so the cost is only opportunity cost. The bootstrap phase is a live, production-active window on mainnet until the v10 hard fork is enacted.

---

### Recommendation

The `clearDRepDelegations` function in `GovCert.hs` should guard each adjustment with a check that the credential's current forward delegation still points to the unregistering DRep before nullifying it:

```haskell
clearDRepDelegations cred delegs accountsMap =
  foldr
    ( Map.adjust
        ( \as ->
            if as ^. dRepDelegationAccountStateL == Just (DRepCredential cred)
              then as & dRepDelegationAccountStateL .~ Nothing
              else as
        )
    )
    accountsMap
    delegs
```

This mirrors the fix applied to `unDelegReDelegDRep` (VState.hs lines 129–143), which already checks `if Just dRep == mNewDRep then id` before modifying `drepDelegs`. The same defensive check should be applied symmetrically in the unregistration path.

---

### Proof of Concept

**Setup (bootstrap phase, `pvMajor pv == 9`):**

1. Attacker registers **DRep X** (`RegDRepTxCert`).
2. Victim's stake credential **A** delegates to DRep X (`DelegTxCert A (DelegVote (DRepCredential X))`).
   - `accountsMap[A].dRepDelegation = Just (DRepCredential X)`
   - `vsDReps[X].drepDelegs = {A}`
3. Victim redelegates **A** to **DRep Y** (`DelegTxCert A (DelegVote (DRepCredential Y))`).
   - `processDelegationInternal` with `preserveIncorrectDelegation = True`:
     - `accountsMap[A].dRepDelegation = Just (DRepCredential Y)` ✓
     - `vsDReps[Y].drepDelegs = {A}` ✓
     - `vsDReps[X].drepDelegs = {A}` ← **stale, not removed**

**Attack:**

4. Attacker unregisters **DRep X** (`UnRegDRepTxCert X`).
   - `clearDRepDelegations {A} accountsMap` runs unconditionally.
   - `accountsMap[A].dRepDelegation = Nothing` ← **incorrectly cleared**
   - `vsDReps[Y].drepDelegs = {A}` ← stale (not cleaned up either)

**Result:**

5. `computeDRepDistr` iterates accounts; credential A has `dRepDelegation = Nothing`, so A's stake is **not attributed to DRep Y**.
6. DRep Y's weight in `reDRepDistr` is reduced by A's stake.
7. `dRepAcceptedRatio` computes a lower (or higher) ratio than the true delegation state warrants, potentially causing a governance action to be incorrectly ratified or rejected.

```haskell
processDelegationInternal ::
  ConwayEraCertState era =>
  -- | Preserve the buggy behavior where DRep delegations are not updated correctly (See #4772)
  Bool ->
  -- | Delegator
  Credential Staking ->
  -- | Account state for the above stake credential
  Maybe (AccountState era) ->
  -- | New delegatee
  Delegatee ->
  CertState era ->
  CertState era
processDelegationInternal preserveIncorrectDelegation stakeCred mAccountState newDelegatee =
  case newDelegatee of
    DelegStake sPool -> delegStake sPool
    DelegVote dRep -> delegVote dRep
    DelegStakeVote sPool dRep -> delegVote dRep . delegStake sPool
  where
    delegStake stakePool cState =
      cState
        & certDStateL . accountsL
          %~ adjustAccountState (stakePoolDelegationAccountStateL ?~ stakePool) stakeCred
        & maybe
          (certPStateL . psStakePoolsL %~ Map.adjust (spsDelegatorsL %~ Set.insert stakeCred) stakePool)
          (\accountState -> certPStateL %~ unDelegReDelegStakePool stakeCred accountState (Just stakePool))
          mAccountState

    delegVote dRep cState =
      let handleReverseDelegation =
            case dRepToCred dRep of
              Just dRepCred
                -- This is the case where we only add the new reverse delegation and do not remove
                -- the old one, which is the behavior that we want:
                --
                -- 1) for new accounts, since there is no old reverse delegation to remove
                --
                -- 2) in the bootstrap phase, in order to preserve the incorrect behavior, where old reverse
                --   delegation for the prior DRep was wrongfully retained. It is important to note
                --   that in case when the new delegation was to a predefined DRep, the reverse
                --   delegations where handled correctly even in the boostrap phase
                --
                -- For reference here is the original bug report:
                --   https://github.com/IntersectMBO/cardano-ledger/issues/4772
                | isNothing mAccountState || preserveIncorrectDelegation ->
                    certVStateL . vsDRepsL
                      %~ Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
              _
                -- AccountState existed before this delegation, therefore we need to properly handle
                -- potential undelegation of the old DRep
                | Just accountState <- mAccountState ->
                    certVStateL %~ unDelegReDelegDRep stakeCred accountState (Just dRep)
                -- If this is a fresh registration with delegation to a predefined DRep, there are
                -- no extra steps that need to be done
                | otherwise -> id
       in cState
            & certDStateL . accountsL
              %~ adjustAccountState (dRepDelegationAccountStateL ?~ dRep) stakeCred
            & handleReverseDelegation
```

```haskell
unDelegReDelegDRep stakeCred accountState mNewDRep =
  fromMaybe (vsDRepsL %~ addNewDelegation) $ do
    dRep@(DRepCredential dRepCred) <- accountState ^. dRepDelegationAccountStateL
    pure $
      -- There is no need to update set of delegations if delegation is unchanged
      if Just dRep == mNewDRep
        then id
        else
          vsDRepsL %~ addNewDelegation . Map.adjust (drepDelegsL %~ Set.delete stakeCred) dRepCred
  where
    addNewDelegation =
      case mNewDRep of
        Just (DRepCredential dRepCred) ->
          Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
        _ -> id
```

```haskell
dRepAcceptedRatio ::
  forall era.
  RatifyEnv era ->
  Map (Credential DRepRole) Vote ->
  GovAction era ->
  Rational
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
```
