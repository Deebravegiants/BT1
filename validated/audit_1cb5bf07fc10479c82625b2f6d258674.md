### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Allows Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over `response.payload_hash`, but never checks that `response.payload_hash` is actually the correct hash for the pending `request`. A single Byzantine MPC participant (below the signing threshold) can replay any previously observed valid threshold signature — which is public on-chain — against a different pending `verify_foreign_transaction` request, causing the contract to resolve that request with a forged `VerifyForeignTransactionResponse`.

### Finding Description

The `respond_verify_foreign_tx` function performs the following checks:

1. Caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key.
3. `request` matches an entry in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What it does **not** check is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: <observed_values> }))` for the specific pending `request`. The `payload_hash` field in the response is completely unconstrained relative to the pending request key. [2](#0-1) 

The canonical payload that nodes are supposed to sign is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }))
``` [3](#0-2) 

Only the hash is returned in the response (not the `values`), so the contract cannot reconstruct the expected hash. However, the contract also makes no attempt to verify that the `ForeignChainRpcRequest` embedded in `payload_hash` matches the pending `request.request`. This is the missing invariant.

Once the signature check passes, `resolve_yields_for` drains all queued yields for `request` and delivers the attacker-controlled `response` (including the forged `payload_hash`) to every waiting caller. [4](#0-3) 

### Impact Explanation

A bridge contract (e.g., Omnibridge inbound flow) that calls `verify_foreign_transaction` and uses the returned `VerifyForeignTransactionResponse` to authorize fund releases will receive a response whose `payload_hash` commits to a **different** foreign transaction than the one it submitted. If the bridge contract does not independently reconstruct and compare the expected hash, it will accept the forged attestation and release funds for a transaction that was never actually verified by the MPC network. This constitutes **forged foreign-chain verification enabling invalid bridge execution**.

### Likelihood Explanation

The attacker must be a single attested MPC participant — no threshold collusion is required. The previously observed valid threshold signature (needed for replay) is publicly available on-chain from any prior successful `respond_verify_foreign_tx` call. Any user can submit a `verify_foreign_transaction` request to create a target pending entry. The attacker simply races to call `respond_verify_foreign_tx` with the replayed `{payload_hash_old, sig_old}` before the honest nodes respond. This is a realistic race condition in a live bridge deployment.

### Recommendation

The contract must bind `response.payload_hash` to the pending `request`. Since the `values` are not included in the response, the contract cannot fully reconstruct the hash. The fix is to include the `ForeignChainRpcRequest` (and `payload_version`) as a prefix in the signed payload and verify on-chain that the hash's committed request matches the pending request. Alternatively, require nodes to include the `values` in the response so the contract can recompute and verify the hash directly. At minimum, the contract should verify that `payload_hash` is not a hash previously used to resolve a different request (replay prevention via a spent-hash set).

### Proof of Concept

1. Alice submits `verify_foreign_transaction` for Bitcoin tx `[0xAA; 32]` (request A). The honest MPC nodes respond with `{payload_hash_A, sig_A}` where `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload { request: Bitcoin([0xAA;32]), values: [BlockHash([0x11;32])] }))`. This call is recorded on-chain.

2. Bob submits `verify_foreign_transaction` for Bitcoin tx `[0xBB; 32]` (request B, a fraudulent or non-existent transaction). Request B is now pending in `pending_verify_foreign_tx_requests`.

3. The Byzantine participant calls:
   ```
   respond_verify_foreign_tx(
     request = VerifyForeignTransactionRequest { request: Bitcoin([0xBB;32]), ... },
     response = VerifyForeignTransactionResponse { payload_hash: payload_hash_A, signature: sig_A }
   )
   ```

4. The contract checks: (a) caller is attested ✓, (b) `sig_A` is valid over `payload_hash_A` under root key ✓, (c) request B is pending ✓. No check that `payload_hash_A` corresponds to `Bitcoin([0xBB;32])`.

5. `resolve_yields_for` drains request B's queue and delivers `{payload_hash_A, sig_A}` to Bob's bridge contract.

6. Bob's bridge contract receives a valid MPC-signed response for its request and, if it does not independently verify `payload_hash_A` against the expected hash for `Bitcoin([0xBB;32])`, releases funds — completing an invalid bridge execution for a transaction that was never verified. [5](#0-4)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
