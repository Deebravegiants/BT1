### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Submitted `request` — Cross-Request Signature Replay - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over `response.payload_hash`, but never checks that `response.payload_hash` was actually computed from the `request` argument supplied in the same call. A single Byzantine MPC participant acting as leader can replay a legitimately produced MPC signature for one pending request as the response to a different pending request, delivering a forged attestation to the bridge contract that submitted the second request.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs the following checks:

```
1. assert_caller_is_signer()                          // caller is a registered participant
2. assert_caller_is_attested_participant_and_protocol_active()
3. verify_ecdsa_signature(signature, payload_hash, root_pk)  // signature valid over payload_hash
4. resolve_yields_for(pending_verify_foreign_tx_requests, &request, response_bytes)
``` [1](#0-0) 

Step 3 only confirms that `response.payload_hash` was signed by the MPC network's root key. It does **not** confirm that `response.payload_hash` encodes the `request` argument. The `ForeignTxSignPayloadV1` struct that produces `payload_hash` contains both the `ForeignChainRpcRequest` and the `ExtractedValue` list: [2](#0-1) 

Because the contract never reconstructs or checks the preimage, any valid MPC signature over *any* `payload_hash` passes the check, regardless of which request it was originally produced for.

The `near-mpc-sdk` client helper `ForeignChainSignatureVerifier::verify_signature` does perform the binding check (`expected_payload_hash == response.payload_hash`), but this is a **client-side** library used by bridge contracts — it is not enforced by the on-chain contract itself: [3](#0-2) 

### Impact Explanation

A Byzantine MPC participant (a single registered signer, below threshold) who is elected leader for `request_B` can:

1. Observe a legitimately produced `VerifyForeignTransactionResponse` for a different pending `request_A` (the response is broadcast on-chain when the leader for `request_A` calls `respond_verify_foreign_tx`).
2. Call `respond_verify_foreign_tx(request_B, response_for_A)`. The contract accepts this because `response_for_A.signature` is a valid MPC signature over `response_for_A.payload_hash`.
3. `resolve_yields_for` drains all yields queued under `request_B` and delivers `response_for_A` to every caller that submitted `request_B`. [4](#0-3) 

The bridge contract that submitted `request_B` receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes `request_A`'s transaction data and extracted values, not `request_B`'s. Any bridge contract that does not independently verify the payload binding (i.e., does not use `ForeignChainSignatureVerifier::verify_signature` correctly) will treat this as a valid attestation of `request_B`'s foreign transaction and may release funds or execute logic based on a fraudulent observation. This constitutes forged foreign-chain verification enabling invalid bridge execution.

### Likelihood Explanation

**Low.** The attack requires: (a) a Byzantine registered MPC participant, (b) two concurrently pending `verify_foreign_transaction` requests, (c) the Byzantine node being elected leader for the target request, and (d) a bridge contract that omits or incorrectly implements the client-side payload-hash binding check. Conditions (a)–(c) are operationally constrained; condition (d) is a realistic implementation gap for new integrators who rely on the contract as the sole trust anchor.

### Recommendation

Inside `respond_verify_foreign_tx`, after verifying the signature, recompute the expected `payload_hash` prefix from the submitted `request` and assert it matches `response.payload_hash`. Because the contract does not know the `ExtractedValue` list (only the hash is stored), the simplest fix is to require the responder to also supply the extracted values, then compute `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` on-chain and compare it to `response.payload_hash` before accepting the response. Alternatively, store a commitment to the expected `request` alongside each pending yield and verify the preimage at response time.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
