### Title
Sub-Transaction ExUnits Bypass `maxTxExUnits` and `maxBlockExUnits` Limits, Enabling Underpaid Script Execution - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may embed an arbitrary number of sub-transactions (`subTransactionsTxBodyL`), each carrying its own Plutus redeemers with independent `ExUnits` budgets. The `totExUnits` function — used for both the per-transaction limit check (`validateExUnitsTooBigUTxO`) and the per-block limit check (`validateExUnits`) — only sums redeemers from the **top-level** transaction's witnesses. Sub-transaction ExUnits are never counted. An unprivileged submitter can therefore craft a top-level transaction with near-zero top-level ExUnits but embed N sub-transactions each declaring up to `maxTxExUnits` ExUnits, bypassing both protocol-parameter limits and the minimum-fee script-execution charge.

---

### Finding Description

`totExUnits` is defined as:

```haskell
-- eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs:390-394
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It folds only over the redeemers stored in the top-level transaction's witness set. [1](#0-0) 

The Dijkstra UTXO transition rule calls this function unchanged for the per-transaction limit:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs:414-415
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [2](#0-1) 

The Dijkstra BBODY transition rule calls the same Alonzo helper for the per-block limit:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs:361
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
```

which internally does `foldMap totExUnits txs` — again only top-level ExUnits. [3](#0-2) 

The `SUBUTXO` rule (`dijkstraSubUtxoTransition`) performs no ExUnits check at all. The injection mapping explicitly marks `ExUnitsTooBigUTxO` as impossible for sub-transactions:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs:340
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [4](#0-3) 

The minimum-fee calculation also uses `totExUnits`, so sub-transaction script fees are never charged:

```haskell
-- eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs:383-388
alonzoMinFeeTx pp tx =
  (tx ^. sizeTxF <×> ...) <+> (pp ^. ppTxFeeFixedL)
    <+> txscriptfee (pp ^. ppPricesL) allExunits
  where allExunits = totExUnits tx
``` [5](#0-4)

### Impact Explanation

**Medium — Attacker-controlled transactions exceed intended validation limits and modify fees outside design parameters.**

1. **`maxTxExUnits` bypass**: A top-level transaction with 0 top-level redeemers but N sub-transactions each declaring `maxTxExUnits` ExUnits passes `validateExUnitsTooBigUTxO` (sees 0 ≤ max), yet forces nodes to execute N × `maxTxExUnits` worth of Plutus computation.

2. **`maxBlockExUnits` bypass**: The block-level `validateExUnits` sums only top-level ExUnits across all transactions in the block. Sub-transaction ExUnits are invisible to this check, so a single block can carry unbounded total Plutus execution work.

3. **Fee undercharge**: `alonzoMinFeeTx` omits sub-transaction ExUnits from the script-fee component, so the attacker pays zero script-execution fees for all Plutus scripts run inside sub-transactions. This modifies the effective fee outside design parameters.

---

### Likelihood Explanation

Any unprivileged user who can submit a Dijkstra-era transaction can exploit this. No special keys, governance majority, or privileged access is required. The Dijkstra era explicitly supports sub-transactions as a first-class feature, so the attack surface is reachable on every block. The attacker's only cost is the base transaction fee (size-based), not the script-execution fee.

---

### Recommendation

1. Redefine `totExUnits` (or introduce a Dijkstra-specific override) to recursively include sub-transaction ExUnits:

```haskell
totExUnitsDijkstra tx =
  totExUnits tx
    <> foldMap totExUnits (OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL)
```

2. Apply this aggregate in `validateExUnitsTooBigUTxO` and `validateExUnits` for the Dijkstra era.

3. Apply the same aggregate in `alonzoMinFeeTx` (or its Dijkstra override) so that sub-transaction script fees are included in the minimum fee.

4. Add a `SubExUnitsTooBigUTxO` predicate failure to `DijkstraSubUtxoPredFailure` and enforce `totExUnits subTx ≤ maxTxExUnits pp` inside `dijkstraSubUtxoTransition`.

---

### Proof of Concept

**Attacker constructs:**

```
TopTx {
  inputs = [someUTxO],
  fee    = minBaseFee,          -- only size-based, no script fee
  wits   = { redeemers = {} },  -- zero top-level ExUnits
  subTransactions = [
    SubTx { wits = { redeemers = { ptr0 -> (datum, maxTxExUnits) } } },
    SubTx { wits = { redeemers = { ptr0 -> (datum, maxTxExUnits) } } },
    ...  -- repeated K times
  ]
}
```

**Ledger checks that pass:**

- `totExUnits topTx = 0 ≤ maxTxExUnits` → `validateExUnitsTooBigUTxO` passes. [2](#0-1) 
- `foldMap totExUnits [topTx] = 0 ≤ maxBlockExUnits` → `validateExUnits` passes. [3](#0-2) 
- `minfee` charges 0 script fee → `validateFeeTooSmallUTxO` passes. [5](#0-4) 
- `SUBUTXO` rule runs each sub-transaction with no ExUnits check. [6](#0-5) 

**Actual execution cost:** K × `maxTxExUnits` Plutus steps/memory, with K unbounded by any protocol rule, and zero script-execution fee paid.
