Audit Report

## Title
Missing Chainlink Staleness Validation Enables Over-Minting of rsETH, Extracting Value from Existing Depositors — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt` and `answeredInRound`, returning a potentially stale price with no freshness check. Because `LRTDepositPool.getRsETHAmountToMint()` uses this live oracle price as the numerator against the stored `rsETHPrice` as the denominator, a stale-high price causes the protocol to mint more rsETH than the deposited collateral is worth, creating an undercollateralisation gap that dilutes all existing rsETH holders.

## Finding Description

**Root cause — `ChainlinkPriceOracle.getAssetPrice()` (line 52):**

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

All five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. Only `answer` is captured; `updatedAt` and `answeredInRound` are discarded with no staleness check. [1](#0-0) 

**Minting formula — `LRTDepositPool.getRsETHAmountToMint()` (line 520):**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` is fetched live from the Chainlink feed on every call. `lrtOracle.rsETHPrice()` is a stored value last written by `updateRSETHPrice()`. If the live feed is stale-high, the numerator is inflated while the denominator reflects a prior correct state, so the minted rsETH exceeds the true ETH value deposited. [2](#0-1) 

**TVL calculation — `LRTOracle._getTotalEthInProtocol()` (line 339):**

`_getTotalEthInProtocol()` also calls `getAssetPrice()` for every supported asset, so it too uses the stale price. When `updateRSETHPrice()` is called while the stale price is active, it bakes the inflated TVL into `rsETHPrice`, keeping it near 1.0e18. When the price corrects, the next `updateRSETHPrice()` call sees a lower TVL against the now-larger rsETH supply, driving `rsETHPrice` down. [3](#0-2) 

**Existing guard is insufficient — downside protection (lines 270–282):**

`_updateRsETHPrice()` contains a downside protection mechanism that pauses the protocol if `newRsETHPrice` drops more than `pricePercentageLimit` below `highestRsethPrice`. This guard is insufficient for three reasons: (1) it only fires when `updateRSETHPrice()` is called *after* the price corrects — the over-minting has already occurred; (2) if `pricePercentageLimit` is 0 (unset), there is no protection at all; (3) even when triggered, it pauses the protocol but does not reverse the already-minted rsETH. [4](#0-3) 

**Exploit flow:**
1. Wait for a Chainlink LST/ETH feed to go stale with a price above true value (e.g., stETH/ETH = 1.05e18, true = 0.99e18).
2. Call `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")` — no privilege required.
3. `getRsETHAmountToMint` computes `(100e18 * 1.05e18) / 1.00e18 = 105 rsETH` instead of the correct `~99 rsETH`.
4. Price corrects; anyone calls `updateRSETHPrice()`.
5. `_getTotalEthInProtocol()` now returns true TVL (1099 ETH) against 1105 rsETH supply → `rsETHPrice ≈ 0.9946e18`.
6. Attacker's 105 rsETH redeems for ≈104.4 ETH on 99 ETH deposited; original holders' 1000 rsETH is now worth ≈994.6 ETH.

## Impact Explanation

**Critical — Protocol insolvency.** The protocol accepts 99 ETH of real collateral but issues rsETH claims totalling 104.4 ETH, creating a permanent undercollateralisation gap. Value is directly transferred from existing depositors to the attacker. The gap is not recoverable without an external capital injection; repeated exploitation compounds the insolvency. This matches the allowed impact: *Protocol insolvency* and *Direct theft of any user funds*. [5](#0-4) 

## Likelihood Explanation

No special privilege is required — any address can call `depositAsset()`. Chainlink feeds go stale during network congestion, gas price spikes, or node outages, all of which are observed on mainnet. The stETH/ETH feed has a 24-hour heartbeat and a 0.5% deviation threshold; a period of low volatility combined with network congestion can leave the feed stale for hours. The `pricePercentageLimit` guard operates on rsETH price movement, not on raw asset price, and only after minting has already occurred. The attack is repeatable across every stale window and every supported LST asset. [1](#0-0) 

## Recommendation

Add staleness and sanity checks inside `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(price > 0,                                        "non-positive price");
require(answeredInRound >= roundId,                       "stale round");
require(updatedAt + stalenessThreshold[asset] > block.timestamp, "stale price");
```

Store a per-asset `stalenessThreshold` (e.g., heartbeat + buffer) alongside `assetPriceFeed` in the mapping and enforce it in `updatePriceFeedFor()`. This prevents any stale price from reaching the minting formula. [6](#0-5) 

## Proof of Concept

Minimal Foundry fork test plan:

1. Fork mainnet at a block where the stETH/ETH Chainlink feed is fresh.
2. Deploy or reference the live `ChainlinkPriceOracle`, `LRTDepositPool`, and `LRTOracle`.
3. Use `vm.mockCall` to make `priceFeed.latestRoundData()` return `updatedAt = block.timestamp - 2 days` (stale) with `answer = 1.05e18`.
4. Assert `getAssetPrice(stETH)` returns `1.05e18` (no revert — staleness not checked).
5. Call `pool.getRsETHAmountToMint(stETH, 100e18)` and assert it returns `105e18`.
6. `vm.prank(attacker); pool.depositAsset(stETH, 100e18, 0, "")`.
7. Restore the mock to return `answer = 0.99e18` with fresh `updatedAt`.
8. Call `lrtOracle.updateRSETHPrice()`.
9. Assert `lrtOracle.rsETHPrice() < 1e18` (≈0.9946e18).
10. Assert `(105e18 * lrtOracle.rsETHPrice()) / 1e18 > 99e18` — attacker profit confirmed.
11. Assert `(1000e18 * lrtOracle.rsETHPrice()) / 1e18 < 1000e18` — holder loss confirmed. [1](#0-0)
