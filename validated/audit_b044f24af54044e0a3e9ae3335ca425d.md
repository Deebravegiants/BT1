Audit Report

## Title
Zero-wrsETH Mint on Dust Deposits Due to Missing `rsETHAmount == 0` Guard — (`contracts/pools/RSETHPoolV2NBA.sol`)

## Summary
`RSETHPoolV2NBA.deposit()` accepts ETH and mints `wrsETH` without verifying that the computed `rsETHAmount` is non-zero. Integer division in `viewSwapRsETHAmountAndFee` truncates `rsETHAmount` to zero for sufficiently small deposits when the oracle rate exceeds `1e18`, which is the normal operating state. The transaction succeeds silently, the depositor's ETH is locked in the contract, and the user receives zero `wrsETH` with no on-chain recovery path.

## Finding Description
In `viewSwapRsETHAmountAndFee`, the output is computed via plain integer division:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0) 

When `amountAfterFee * 1e18 < rsETHToETHrate`, this truncates to `0`. `deposit()` then proceeds unconditionally — the only guard checks `msg.value`, not the computed output:

```solidity
if (amount == 0) revert InvalidAmount();
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);  // mint(msg.sender, 0) — no revert
``` [2](#0-1) 

`InterimRSETHOracle._setRate` enforces `rate >= 1e18`, meaning the rate is always at or above `1e18` and increases monotonically as yield accrues: [3](#0-2) 

At any rate strictly above `1e18` (e.g., `1.05e18`), a deposit of `1 wei` produces `1 * 1e18 / 1.05e18 = 0`. The ETH is accepted into the contract balance, `feeEarnedInETH` is unchanged (fee also truncates to 0), and the depositor has no function to reclaim it. `moveAssetsForBridging()` and `withdrawFees()` are both `BRIDGER_ROLE`-only and route funds to the bridger/receiver, not back to the original depositor: [4](#0-3) 

## Impact Explanation
A depositor who sends a dust ETH amount (any value where `amountAfterFee < rsETHToETHrate / 1e18`) receives zero `wrsETH` while their ETH is permanently inaccessible to them. The invariant "every accepted non-zero ETH deposit produces a non-zero wrsETH mint" is broken. This matches **Medium — Temporary freezing of funds**: the ETH is accepted by the contract but the depositor has no recovery path; only a privileged bridger can sweep it (to themselves, not the depositor).

## Likelihood Explanation
No privileged access, oracle manipulation, or unusual configuration is required. Any external user can call `deposit()` with a dust `msg.value` under the normal post-genesis oracle rate (any rate `> 1e18`). The precondition is the protocol's normal operating state. The affected deposit range grows as the oracle rate increases over time. Likelihood: **Medium**.

## Recommendation
Add a zero-output guard immediately after computing `rsETHAmount` in `deposit()`:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

This mirrors the existing `amount == 0` guard at line 109 and ensures the invariant holds for all accepted deposits. [5](#0-4) 

## Proof of Concept
Concrete trigger (no oracle manipulation, no privileged access):

1. Deploy `InterimRSETHOracle` with `initRate = 1.05e18` (5% yield accrued — routine operating state).
2. Deploy `RSETHPoolV2NBA` pointing to the oracle, with `feeBps = 0`.
3. Deploy a mock `wrsETH` that records `mint` call arguments.
4. Call `pool.deposit{value: 1}("")` from any EOA.
5. Assert `mockWrsETH.lastMintAmount == 0` — **passes**, user received nothing.
6. Assert `address(pool).balance == 1` — **passes**, ETH is locked.
7. Assert depositor ETH balance decreased by 1 — **passes**, ETH is gone with no recovery path.

The `deposit()` call at step 4 does not revert, confirming the vulnerability on unmodified code. A Foundry fuzz test varying `msg.value` from `1` to `rsETHToETHrate / 1e18 - 1` and asserting `rsETHAmount > 0` would also demonstrate the full affected range.
