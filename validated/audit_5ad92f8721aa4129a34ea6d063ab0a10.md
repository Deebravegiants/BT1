The checks are: role valid, curve is secp256k1, `x_share ∈ [0, q)`, `Q_compressed` decodes as a valid curve point. **There is no check that `x_share * G` relates to `Q_compressed` in any way.**

**Contrast with `schnorr_mp::deserialize_key_blob`** (`src/cbmpc/api/schnorr_mp.cpp`, line 155):

```cpp

```

This check exists because the schnorr_mp blob stores individual `Qis_compressed` per party. The schnorr_2p blob (`key_blob_v1_t`) stores only `Q_compressed` (the global key) and `x_share` — **no per-party `Qi` is stored** — so there is no stored value to check `x_share * G` against.

## Why the Scenario Succeeds

A bit-flipped `x_share'` that remains in `[0, q)` passes `blob_to_key` cleanly. `get_public_share_compressed` then computes and returns `x_share' * G`, which is a valid curve point but is **not** the actual share point `Qi` that satisfies `Qi + Qi_other = Q`. The function returns `SUCCESS` with the wrong point.

## Does `attach_private_scalar` Save It?

No. `attach_private_scalar` (`src/cbmpc/api/schnorr2pc.cpp`, lines 186–232) does check `x * G != Qi_self`, but `Qi_self` is the caller-supplied `public_share_compressed` argument — which was itself produced by the same `get_public_share_compressed` call on the tampered blob. So `flipped_x * G == flipped_Qi` trivially passes. The self-consistency check is circular when both inputs come from the same tampered blob.

## Impact Scope

The concrete bad output path is:

1. Attacker provides tampered blob (bit-flipped `x_share'`, still in range) to `get_public_share_compressed`
2. API returns `Qi' = x_share' * G` with `SUCCESS`
3. Caller uses `Qi'` as the expected public point in a PVE backup
4. PVE encrypts `x_share'` against `Qi'`; the ZK proof `x_share' * G == Qi'` verifies correctly
5. The backup is internally consistent but **encodes the wrong share** — it does not correspond to the global key `Q`
6. Recovery from this backup yields a share that cannot participate in valid signing

This fits the stated High scope: *"Public API reachable validation bypass … creates accepted invalid cryptographic output."*

### Title
Missing self-consistency check in `blob_to_key` allows `get_public_share_compressed` to return a share point inconsistent with the global key — (`src/cbmpc/api/schnorr2pc.cpp`)

### Summary

`blob_to_key` for the schnorr-2p key blob validates only that `x_share ∈ [0, q)` and that `Q_compressed` decodes as a valid point. It does not verify any relationship between `x_share` and `Q_compressed`. `get_public_share_compressed` therefore computes and returns `x_share * G` from whatever scalar is in the blob, with no check that the result is consistent with the stored global key `Q`.

### Finding Description

The schnorr_2p `key_blob_v1_t` stores two independent fields: `Q_compressed` (the global public key) and `x_share` (the private scalar share). [1](#0-0) 

`blob_to_key` validates `x_share` is in range and `Q_compressed` is a valid point, but performs no cross-field consistency check. [2](#0-1) 

`get_public_share_compressed` then computes `Qi = (x_share % q) * G` and returns it as the party's share point. [3](#0-2) 

By contrast, `schnorr_mp::deserialize_key_blob` explicitly checks `blob.x_share * G != it_self->second` (the stored per-party `Qi`) before accepting the blob. [4](#0-3) 

The schnorr_2p blob has no stored `Qi` field, so the equivalent check is structurally absent. [1](#0-0) 

### Impact Explanation

A caller (or storage-layer attacker) who provides a blob with a bit-flipped `x_share'` (still in `[0, q)`) receives a wrong `Qi' = x_share' * G` from `get_public_share_compressed` with no error. If this `Qi'` is passed to a PVE backup, the backup encrypts `x_share'` against `Qi'`; the internal ZK proof `x_share' * G == Qi'` verifies, so the backup is accepted as valid. The backup does not correspond to the actual key share, making recovery produce a share that cannot sign under `Q`. [5](#0-4) 

### Likelihood Explanation

The attacker must control the key blob bytes passed to the API — either as the blob owner, a compromised storage layer, or a malicious serialized-input provider. The bit-flip must keep `x_share'` in `[0, q)`, which is trivially achievable by flipping any bit below the most-significant bit of `q`. No protocol interaction is required; the attack is a single local API call.

### Recommendation

Store the per-party share point `Qi` in the schnorr_2p blob (analogous to `Qis_compressed` in the schnorr_mp blob) and add the check `x_share * G == Qi_self` in `blob_to_key`, mirroring the guard already present in `schnorr_mp::deserialize_key_blob` at line 155. [4](#0-3) 

Alternatively, if storing `Qi` is undesirable, `get_public_share_compressed` should be documented as computing `Qi` from the stored scalar with no consistency guarantee, and callers must independently verify `Qi + Qi_other == Q` before using the result in a PVE backup.

### Proof of Concept

```
1. Run DKG to obtain a valid key_blob for party P1.
2. Deserialize the blob bytes; locate the x_share field.
3. Flip one bit of x_share (e.g., bit 0), keeping the result in [0, q).
4. Re-serialize the tampered blob.
5. Call get_public_share_compressed(tampered_blob) → returns Qi' = x_share' * G, SUCCESS.
6. Call get_public_share_compressed(original_blob) → returns Qi = x_share * G.
7. Assert Qi' != Qi  (trivially true).
8. Assert Qi' + Qi_other != Q  (where Qi_other is P2's share point from a separate call).
   → The returned Qi' does not satisfy the additive share relation for Q.
9. Feed Qi' into a PVE backup call; observe the backup is accepted with no error.
```
