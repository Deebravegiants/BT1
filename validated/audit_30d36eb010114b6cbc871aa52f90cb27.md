Audit Report

## Title
ETH Deposit Limit Check Omits Incoming `amount`, Allowing Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ETH it evaluates `totalAssetDeposits > limit` (pre-deposit total only), while for ERC20 tokens it correctly evaluates `totalAssetDeposits + amount > limit`. A single unprivileged depositor can therefore push total ETH holdings arbitrarily beyond the configured cap in one transaction, breaking the protocol's deposit-limit invariant.

## Finding Description
In `contracts/LRTDepositPool.sol` at L676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The ETH branch never adds `amount` to `totalAssetDeposits`. Because `getTotalAssetDeposits` reflects the state *before* the incoming deposit is recorded, the check will return `false` (i.e., "limit not exceeded") for any deposit made while `totalAssetDeposits <= limit`, regardless of how large `amount` is.

The full call chain (all confirmed in the repository):
- `depositETH` (L76–93) is public and payable.
- It calls `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected)` (L87).
- `_beforeDeposit` (L648–670) calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` (L661) and reverts only if it returns `true`.
- With the broken ETH branch, the revert never fires as long as the pre-deposit total is at or below the cap.

No other guard in `depositETH` or `_beforeDeposit` enforces the deposit cap for ETH.

## Impact Explanation
The deposit cap is the protocol's primary mechanism for bounding ETH exposure to EigenLayer strategies. With the bug, a single depositor can mint rsETH for an amount many multiples of the cap. All excess ETH is eventually forwarded to node delegators and deployed into EigenLayer, exposing rsETH holders to slashing risk far beyond what governance intended. This concretely maps to the allowed impact: **Low — Contract fails to deliver promised returns** (the deposit-cap invariant is broken). If slashing materialises on the over-deposited ETH, losses to rsETH holders escalate the impact further.

## Likelihood Explanation
`depositETH` is public, payable, and requires no role, whitelist, or special precondition. Any depositor who observes `totalAssetDeposits < depositLimit` (trivially readable on-chain) can exploit this in a single transaction. The condition is easily and repeatably reachable. Likelihood is **High**.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Deploy with `depositLimitByAsset[ETH_TOKEN] = 1_000 ether`.
2. Seed the pool so `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.
3. Call `depositETH{value: 10_000 ether}(0, "")` from any EOA.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999e18 > 1_000e18` → `false`; no revert.
5. `_mintRsETH` mints rsETH for the full `10_000 ETH`.
6. `getTotalAssetDeposits(ETH_TOKEN)` now returns `10_999 ether` — 10× the configured cap.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    // set limit to 1_000 ether, seed 999 ether
    vm.deal(attacker, 10_000 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 10_000 ether}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 1_000 ether);
}
