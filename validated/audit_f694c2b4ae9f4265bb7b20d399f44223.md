### Title
`respond_ckd()` Skips CKD Output Verification for `AppPublicKey` Variant, Allowing a Single Byzantine Participant to Deliver Forged Key Derivation Output — (File: `crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd()`, the contract enforces a cryptographic output check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`. The `AppPublicKey` variant silently skips this check entirely. Because `respond_ckd()` requires only a single attested participant to call it (not a threshold), a Byzantine MPC participant below the signing threshold can call `respond_ckd()` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will accept and deliver it to the waiting user without any cryptographic validation.

---

### Finding Description

The `respond_ckd()` function in `crates/contract/src/lib.rs` handles MPC node responses to confidential key derivation (CKD) requests. After verifying the caller is an attested participant, it performs a conditional output check:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies that the submitted `CKDResponse` is consistent with the master public key and the app's public key commitment — preventing any single node from forging the output. For `AppPublicKey`, the arm is a no-op: the response is accepted unconditionally and immediately delivered to the user via `resolve_yields_for`. [2](#0-1) 

The caller authorization check (`assert_caller_is_attested_participant_and_protocol_active`) only requires the caller to be **one** attested participant — not a threshold-many quorum. [3](#0-2) 

The pending CKD request map is keyed by `CKDRequest`, which is derived from public on-chain data (predecessor account, derivation path, domain, app public key). Any attested participant can observe a pending `AppPublicKey` CKD request and race to call `respond_ckd()` with a forged `CKDResponse` before the legitimate threshold protocol completes. [4](#0-3) 

---

### Impact Explanation

A single Byzantine MPC participant (strictly below the signing threshold) can deliver an arbitrary forged confidential key derivation output to any user who submitted an `AppPublicKey` CKD request. The user's waiting yield promise is resolved with the attacker-chosen `CKDResponse`. The user's application then uses this forged derived key — which the attacker knows — for all subsequent operations (encryption, signing, etc.), enabling the attacker to decrypt user data or forge signatures on behalf of the user. This constitutes **unauthorized confidential key derivation output without the required participant authorization**, matching the Critical allowed impact.

---

### Likelihood Explanation

Any single attested MPC participant can exploit this. The `CKDRequest` key is deterministically derived from public inputs and is stored in the on-chain `pending_ckd_requests` map, making it observable by all participants. A Byzantine node only needs to submit `respond_ckd()` before the legitimate nodes do. Since `AppPublicKey` is a supported and documented variant of the CKD API, real users will submit such requests. The attacker does not need threshold collusion — one compromised or malicious TEE node suffices.

---

### Recommendation

Apply `ckd_output_check` (or an equivalent cryptographic binding) to the `AppPublicKey` variant as well, or remove the `AppPublicKey` variant's ability to bypass the check. At minimum, the contract should verify that the submitted `CKDResponse` is cryptographically consistent with the domain's master public key and the request parameters before resolving the yield, regardless of which `CKDAppPublicKey` variant was used.

---

### Proof of Concept

1. User calls `request_app_private_key()` with `CKDAppPublicKey::AppPublicKey(some_pk)`. The contract stores the `CKDRequest` in `pending_ckd_requests` and yields.
2. Byzantine attested participant observes the pending request on-chain (the map key is deterministic from public inputs).
3. Byzantine participant calls `respond_ckd(request, forged_response)` where `forged_response` contains an attacker-chosen derived key.
4. In `respond_ckd()`, the `AppPublicKey` arm executes as a no-op — no `ckd_output_check` is performed.
5. `resolve_yields_for` resumes the user's yield with `forged_response`.
6. The user's contract receives the attacker-controlled derived key and uses it, while the attacker already knows the corresponding secret. [5](#0-4)
