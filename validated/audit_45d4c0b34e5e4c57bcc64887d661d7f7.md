Audit Report

## Title
ETH Permanently Frozen When Withdrawal Initiator Is a Contract Without `receive()` - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`initiateWithdrawal` imposes no check that `msg.sender` can receive ETH. When a smart contract lacking a `receive`/`fallback` function initiates an ETH withdrawal, `unlockQueue` irreversibly burns the rsETH and transfers the corresponding ETH into `LRTWithdrawalManager`. Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` reverts because the low-level ETH push to the contract fails, and `sweepRemainingAssets` is permanently gated by the stuck unlocked withdrawal. The ETH is frozen with no on-chain recovery path.

## Finding Description
`initiateWithdrawal` (L150–178) accepts any `msg.sender` without verifying the caller can receive ETH:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
// ...
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
```

The operator later calls `unlockQueue` (L301–307), which atomically burns the rsETH held by the manager and pulls the corresponding ETH from `LRTUnstakingVault`:

```solidity
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);
```

After this point the rsETH is gone and the ETH sits in `LRTWithdrawalManager`. When the user calls `completeWithdrawal`, `_processWithdrawalCompletion` (L699–738) pops the nonce, deletes the request, decrements `unlockedWithdrawalsCount`, and then calls:

```solidity
_transferAsset(asset, user, request.expectedAssetAmount);
```

`_transferAsset` (L876–883) executes:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` is a contract without a `receive` function, `sent` is `false` and the entire transaction reverts — unwinding the `popFront`, the `delete`, and the `unlockedWithdrawalsCount--`. The withdrawal request is restored to its pre-call state, so the same revert repeats on every future attempt.

`sweepRemainingAssets` (L395–413) is blocked as long as `hasUnlockedWithdrawals` returns `true`:

```solidity
if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist();
```

Because `unlockedWithdrawalsCount[asset]` never decrements (every completion attempt reverts), the sweep is permanently gated. `completeWithdrawalForUser` (L192–204) calls the identical `_processWithdrawalCompletion` and fails in the same way. There is no administrative function to forcibly remove a stuck withdrawal request or redirect the ETH to a different address.

## Impact Explanation
After `unlockQueue` executes, the user's rsETH is irreversibly burned and the corresponding ETH is held in `LRTWithdrawalManager`. No on-chain path exists to move that ETH out: completion always reverts, and the sweep is permanently blocked. This constitutes **Critical — Permanent freezing of funds**, an explicitly listed allowed impact.

## Likelihood Explanation
`initiateWithdrawal` is a public, permissionless function. Smart contracts that hold rsETH — vaults, DAOs, multisigs, DeFi integrations — routinely lack a plain ETH `receive` function. No special privilege is required; the caller simply needs to be a contract that cannot accept ETH. The condition is realistic and repeatable. Likelihood is **Medium**.

## Recommendation
Add an EOA-only guard to `initiateWithdrawal` when `asset == LRTConstants.ETH_TOKEN`:

```solidity
if (asset == LRTConstants.ETH_TOKEN && tx.origin != msg.sender) {
    revert ContractsNotAllowed();
}
```

Alternatively, record a caller-supplied `recipient` address at request time and deliver ETH to that address in `_processWithdrawalCompletion`, allowing the initiating contract to designate a payable EOA as the beneficiary. Either fix must be applied before `unlockQueue` can process the request; there is no safe mitigation after rsETH has been burned.

## Proof of Concept
1. Deploy `Victim`: a contract that holds rsETH but declares no `receive()` or `fallback()`.
2. `Victim` approves `LRTWithdrawalManager` and calls `initiateWithdrawal(ETH_TOKEN, amount, "")`. rsETH is transferred to the manager.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned via `burnFrom`; ETH is moved from `LRTUnstakingVault` into `LRTWithdrawalManager`. State is now irreversible.
4. `Victim` calls `completeWithdrawal(ETH_TOKEN, "")`. `_transferAsset` executes `payable(Victim).call{value: amount}("")` → returns `(false, "")` → reverts with `EthTransferFailed`. All state changes in the call unwind.
5. Step 4 can be repeated indefinitely; it always reverts.
6. `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] > 0`.
7. The ETH is permanently locked in `LRTWithdrawalManager`; the rsETH is permanently burned.
