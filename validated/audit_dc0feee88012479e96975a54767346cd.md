### Title
Missing Aggregated Output Check in Non-PV CKD Protocol Allows Byzantine Participant to Deliver Cryptographically Invalid Confidential Key - (`crates/threshold-signatures/src/confidential_key_derivation/protocol.rs`)

### Summary

The non-PV CKD coordinator (`do_ckd_coordinator` in `protocol.rs`) aggregates participant contributions and returns the result without any cryptographic validity check. The PV variant (`protocol_pv.rs`) performs `aggregated_output_check` before returning. The contract's `respond_ckd` explicitly skips output verification for the `AppPublicKey` (non-PV) variant. A single Byzantine MPC participant (below threshold) can send a malformed contribution that corrupts the aggregate, causing the coordinator to return and the contract to accept a cryptographically invalid `(big_y, big_c)` pair, silently delivering a wrong confidential key to the app.

### Finding Description

**Non-PV coordinator — no output check:**

In `protocol.rs`, `do_ckd_coordinator` aggregates all participant shares and immediately returns `Some(ckd_output)` with no verification:

```rust
// protocol.rs lines 54-61
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))   // ← no aggregated_output_check
``` [1](#0-0) 

**PV coordinator — has the check:**

`protocol_pv.rs` performs the pairing check before returning:

```rust
// protocol_pv.rs lines 66-70
if !aggregated_output_check(&ckd_output, app_pk, &key_pair.public_key, &hash_point) {
    return Err(ProtocolError::AssertionFailed(
        "CKD output failed to verify".to_string(),
    ));
}
``` [2](#0-1) 

**Contract skips output check for non-PV:**

`respond_ckd` in `lib.rs` explicitly does nothing for the `AppPublicKey` arm:

```rust
// lib.rs lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [3](#0-2) 

**Attack path:**

1. A contract caller submits a CKD request with `CKDAppPublicKey::AppPublicKey(pk)`.
2. The MPC network selects `threshold` participants. One of them is Byzantine.
3. The Byzantine participant sends a random `CKDOutput { big_y: rand_G1, big_c: rand_G1 }` to the coordinator instead of the correct Lagrange-weighted share.
4. The coordinator aggregates without verification and returns `Some(corrupted_output)`. [4](#0-3) 
5. `CKDComputation::compute` maps this to `Ok(Some((big_y, big_c)))` and the node calls `respond_ckd` on the contract. [5](#0-4) 
6. The contract clears the pending request and yields the corrupted response to the caller — no pairing check is performed. [6](#0-5) 

### Impact Explanation

The app receives `(big_y, big_c)` and computes `confidential_key = big_c − app_sk · big_y`. With a corrupted aggregate this value is a random group element, not `msk · H(pk, app_id)`. The pending request is permanently resolved; the app has no on-chain mechanism to detect the corruption. The non-PV variant provides no G2 component for a pairing check, so the only detection path is the app running `verify_signature` itself — which is not enforced anywhere in the protocol or contract. This breaks the core safety invariant: a completed CKD request must yield a valid confidential key.

### Likelihood Explanation

A single Byzantine MPC node operator (strictly below the signing threshold) is sufficient. The non-PV path is the default/legacy path (`AppPublicKey` is the original format per the deserialization tests). Any participant selected for a CKD round can mount this attack unilaterally.

### Recommendation

Add `aggregated_output_check` to `do_ckd_coordinator` in `protocol.rs`, mirroring the PV variant. Because the non-PV variant uses only `app_pk: PublicKey` (a G1 point, not a `PublicVerificationKey`), the check must be adapted: verify that `big_c − app_sk_contribution · big_y` lies on the expected BLS signature curve, or alternatively require callers to always supply the PV key pair and deprecate the non-PV path. The contract's `respond_ckd` should also enforce an equivalent on-chain check for the `AppPublicKey` arm.
