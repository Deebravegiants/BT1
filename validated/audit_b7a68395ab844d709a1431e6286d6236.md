### Title
Hardcoded `paid_fee_on_l1 = Fee(1)` in `stored_txn_to_executable_txn` Causes `traceTransaction`/`traceBlockTransactions` to Return Wrong Results for L1Handler Transactions â€” (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

---

### Summary

`stored_txn_to_executable_txn` unconditionally substitutes `Fee(1)` for the actual L1 fee when re-executing any stored `L1Handler` transaction. Any unprivileged RPC caller invoking `starknet_traceTransaction` or `starknet_traceBlockTransactions` on a block containing an L1Handler transaction receives an authoritative-looking trace that was produced with the wrong fee, potentially showing a revert where the original execution succeeded (or vice versa).

---

### Finding Description

In `stored_txn_to_executable_txn`, the `L1Handler` arm is:

```rust
starknet_api::transaction::Transaction::L1Handler(value) => {
    // todo(yair): This is a temporary solution until we have a better way to get the l1 fee.
    let paid_fee_on_l1 = Fee(1);
    Ok(ExecutableTransactionInput::L1Handler(value, paid_fee_on_l1, false))
}
``` [1](#0-0) 

The actual `paid_fee_on_l1` that was recorded when the transaction was originally sequenced is stored in `L1HandlerTransaction.paid_fee_on_l1`: [2](#0-1) 

But `stored_txn_to_executable_txn` receives a `starknet_api::transaction::Transaction` (the raw stored form), not the executable form, so the original fee is not available at that call site â€” and instead of fetching it from storage, the code substitutes `Fee(1)` unconditionally.

This function is called from `api_impl.rs` for both `traceTransaction` and `traceBlockTransactions`: [3](#0-2) 

---

### Impact Explanation

The blockifier's L1Handler execution path uses `paid_fee_on_l1` to validate that the fee paid on L1 is sufficient to cover the execution cost. When `Fee(1)` (1 wei) is substituted:

- If the actual execution cost exceeds 1 wei (which is true for virtually every non-trivial L1Handler), the re-execution **reverts** with an insufficient-fee error, even though the original on-chain execution **succeeded**. The RPC trace reports a failed transaction for a transaction that actually succeeded.
- Even when execution does not revert, the fee accounting in the trace is wrong (showing 1 wei paid instead of the real amount).

This satisfies the **High** impact criterion: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

- The attack surface is fully public: any RPC client can call `starknet_traceTransaction` with the hash of any historical L1Handler transaction.
- L1Handler transactions appear on mainnet regularly (every L1â†’L2 message produces one).
- No authentication, signing, or privileged access is required.
- The bug is deterministic and reproducible for every L1Handler transaction in every block.

Note: the claim about `starknet_simulateTransactions` is **incorrect** â€” that endpoint accepts only `BroadcastedTransaction` (Invoke/Declare/DeployAccount), which cannot be an L1Handler, so `stored_txn_to_executable_txn` is not reached from that path. The real affected endpoints are `traceTransaction` and `traceBlockTransactions`.

---

### Recommendation

When re-executing a stored L1Handler transaction, retrieve the actual `paid_fee_on_l1` from storage rather than hardcoding `Fee(1)`. The storage layer already persists the full transaction receipt, which includes the original fee. Alternatively, store the `paid_fee_on_l1` alongside the raw transaction in the storage schema so it can be recovered at re-execution time.

---
