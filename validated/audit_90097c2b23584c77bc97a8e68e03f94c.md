Audit Report

## Title
`RSETHPriceFeed` Returns ETH/USD Round Metadata Masking rsETH Price Staleness, and `getRoundData()` Returns Structurally Incorrect Historical Prices - (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary
`RSETHPriceFeed` implements `AggregatorV3Interface` as the rsETH/USD price feed for external consumption. In `latestRoundData()`, all freshness metadata (`updatedAt`, `answeredInRound`, `roundId`) is sourced from the ETH/USD Chainlink feed while `answer` uses the current `LRTOracle.rsETHPrice`, meaning standard heartbeat staleness checks validate ETH/USD freshness rather than rsETH oracle freshness. In `getRoundData(_roundId)`, a historical ETH/USD price is multiplied by the **current** rsETH/ETH rate, making every historical round query return a structurally incorrect price. Both issues cause the contract to fail to deliver its promised function as an accurate rsETH/USD price feed.

## Finding Description
In `RSETHPriceFeed.latestRoundData()` (lines 63–70), all five return values are populated from `ETH_TO_USD.latestRoundData()`, then only `answer` is overwritten with `int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18`. The `updatedAt` field therefore reflects when the ETH/USD Chainlink feed last updated, not when `LRTOracle.rsETHPrice` was last written. `LRTOracle` stores only `rsETHPrice` (line 28) with no accompanying timestamp variable — confirmed by the absence of any `rsETHPriceUpdatedAt` or equivalent field. `updateRSETHPrice()` (line 87) is public but has no on-chain enforcement of a call frequency. If it is not called for an extended period, `rsETHPrice` is stale while `updatedAt` from the ETH/USD feed remains recent, causing any consumer performing `block.timestamp - updatedAt < heartbeat` to incorrectly conclude the rsETH price is fresh.

In `getRoundData(_roundId)` (lines 53–61), the ETH/USD price at historical round `_roundId` is fetched and then multiplied by `RS_ETH_ORACLE.rsETHPrice()` — the **current** rsETH/ETH rate. This is unconditionally incorrect for any historical round: the returned `answer` is not the rsETH/USD price at that round but a synthetic value mixing a past ETH/USD price with a present rsETH/ETH rate.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

`RSETHPriceFeed` is deployed as an `AggregatorV3Interface`-compliant rsETH/USD price feed. The contract structurally fails to deliver its core promise on two axes: (1) `latestRoundData()` can return a stale rsETH component with metadata indicating freshness, and (2) `getRoundData()` always returns a price that is not the actual rsETH/USD price at the queried historical round. The Medium impact (temporary freezing of funds) asserted in the submission is not supported within the LRT-rsETH protocol itself — it depends on the behavior of unspecified external lending protocols, which constitutes missing external context and is excluded per the validation rules.

## Likelihood Explanation
The `getRoundData` historical price corruption is unconditional and present on every call regardless of oracle state. The `latestRoundData` staleness masking requires a gap in keeper activity (no call to `updateRSETHPrice()` for a meaningful period), which is a realistic but not guaranteed condition. Both issues are reachable by any unprivileged caller of these public view functions.

## Recommendation
1. **`latestRoundData()`**: Store a `rsETHPriceUpdatedAt` timestamp in `LRTOracle` that is written alongside `rsETHPrice` in `_updateRsETHPrice()`. In `latestRoundData()`, return `min(ethToUsdUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt` so staleness checks reflect the least-recently-updated component.
2. **`getRoundData()`**: Revert with a `NotSupported()` error, since accurate historical rsETH/USD prices cannot be reconstructed without a historical rsETH/ETH rate index. Document this limitation clearly.
3. Optionally, maintain a monotonically increasing `rsETHRoundId` in `LRTOracle` incremented on each `updateRSETHPrice()` call and return it as `answeredInRound` in `latestRoundData()`.

## Proof of Concept
**`getRoundData` — always incorrect (no special conditions required):**
```solidity
// At any point in time:
// ETH/USD at round R (6 months ago) = 2000e8
// rsETH/ETH today = 1.08e18
// True rsETH/USD at round R = 2040e8 (1.02 * 2000)
// Returned answer = 1.08e18 * 2000e8 / 1e18 = 2160e8 — always wrong
feed.getRoundData(R); // returns 2160e8, not 2040e8
```

**`latestRoundData` — staleness masking (requires keeper gap):**
```solidity
// T=0: updateRSETHPrice() called, rsETHPrice = 1.05e18
// T+25h: ETH/USD feed updates normally, rsETHPrice NOT updated (keeper offline)
// T+25h: latestRoundData() called
// Returns: updatedAt = T+25h (ETH/USD), answer uses stale 1.05e18 rsETH rate
// Consumer check: block.timestamp - updatedAt ≈ 0 < heartbeat → passes incorrectly
```

A Foundry fork test can demonstrate both: (1) call `getRoundData` with any historical `_roundId` and verify the returned `answer` differs from the true historical rsETH/USD price; (2) mock `LRTOracle.rsETHPrice` to return a value set 25+ hours ago while the ETH/USD feed updates normally, then confirm `latestRoundData().updatedAt` reflects the recent ETH/USD update rather than the stale rsETH update.
