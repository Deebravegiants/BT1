### Title
State Update Before Integrity Check in `validate_proposal` Stores Unverified Proposals in `valid_proposals` — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

In `validate_proposal`, the `valid_proposals.insert_proposal` state mutation is executed **before** the `built_block != received_fin.proposal_commitment` integrity check. If the check fails and `ProposalFinMismatch` is returned, the proposal is already durably stored in `valid_proposals` under the batcher's commitment, but the validator never sends a prevote. This "ghost" entry violates the Checks-Effects-Interactions invariant: the effect (state write) precedes the final check, and the effect is never rolled back on failure.

---

### Finding Description

In `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, the `validate_proposal` function performs the following sequence after the proposal loop completes:

```rust
// 1. STATE UPDATE — happens unconditionally
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

// 2. INTEGRITY CHECK — happens after the state update
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}
``` [1](#0-0) 

`built_block` is the `ProposalCommitment` computed locally by the batcher from `finished_info.proposal_commitment.partial_block_hash` + `init.fee_proposal_fri` via `proposal_commitment_from`. `received_fin.proposal_commitment` is the commitment claimed by the proposer over the wire. [2](#0-1) 

`insert_proposal` computes and stores the batcher-derived commitment keyed by `(height, round)`:

```rust
pub(crate) fn insert_proposal(...) {
    let proposal_commitment = proposal_commitment_from(
        finished_info.proposal_commitment.partial_block_hash,
        init.fee_proposal_fri,
    );
    ...
    let previous = by_round.insert(round, (proposal_commitment, ...));
    assert!(previous.is_none(), "Overwriting existing proposal ...");

``` [3](#0-2) 

When `ProposalFinMismatch` fires, `validate_and_send` propagates the error, `fin_sender` is dropped without sending, and the consensus layer receives `FinishedValidation(None, round, ...)` — treating the proposal as invalid. However, `valid_proposals` already holds the entry under the batcher's commitment `B`. [4](#0-3) 

`get_proposal`, called during `decision_reached` and `update_for_reproposal`, asserts that the stored commitment equals the requested commitment:

```rust
assert_eq!(
    stored_commitment, commitment,
    "Proposal commitment mismatch for height {height} round {round}: ..."
);
``` [5](#0-4) 

---

### Impact Explanation

**Concrete attack path — validator node crash via `decision_reached` panic:**

1. A malicious proposer streams different transaction batches to different validator subsets (equivocation at the content level). Majority validators receive batch `T_A` and compute batcher commitment `A`; this validator receives batch `T_B` and computes batcher commitment `B`.
2. The proposer sends `ProposalFin { proposal_commitment: A, ... }` to all validators.
3. Majority validators: `built_block == A == received_fin.proposal_commitment` → proposal accepted, prevote for `A` sent.
4. This validator: `built_block == B != A` → `ProposalFinMismatch` returned. But `valid_proposals` already holds `(height, round) → B`. No prevote is sent.
5. Consensus collects 2/3+ prevotes and precommits for `A` and calls `decision_reached(height, round, A)`.
6. `decision_reached` calls `get_proposal(height, round, A)`. The map finds the entry for `(height, round)` with stored commitment `B`. The `assert_eq!(B, A)` fires → **node crash (panic)**.

This is a targeted denial-of-service against individual validators. The attacker only needs to be the proposer for one round and deliver divergent content to a minority of validators.

**Secondary impact — ghost entry persists across the height:**

Even without the crash scenario, the ghost entry occupies the `(height, round)` slot. The `assert!(previous.is_none())` guard in `insert_proposal` means any subsequent attempt to insert a proposal for the same `(height, round)` — e.g., via a legitimate reproposal path — also panics. [6](#0-5) 

---

### Likelihood Explanation

The `ProposalFinMismatch` path is reachable whenever the proposer-supplied `ProposalFin.proposal_commitment` diverges from the batcher's locally computed commitment. This can occur:

- **Malicious proposer** equivocating on transaction content across validators (the crash scenario above).
- **Buggy proposer** emitting a wrong commitment in `ProposalFin` (ghost entry created, but crash only if consensus still decides on a different commitment).

The proposer role rotates per round, so any validator that becomes proposer can trigger this against peers. The attack requires no special privilege beyond being the scheduled proposer.

---

### Recommendation

Move `insert_proposal` to **after** the `ProposalFinMismatch` check, so the state is only mutated when the proposal is confirmed valid:

```rust
// CHECKS first
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}

// EFFECTS only after all checks pass
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

Ok(built_block)
```

The existing comment notes the lock must be held before `fin_sender.send` (in `validate_and_send`) to prevent a `repropose` race. Since `fin_sender.send` is called in `validate_and_send` **after** `validate_proposal` returns `Ok`, the lock can be acquired after the fin check and before returning `Ok(built_block)` — the race condition is still prevented because the lock is held before the caller sends the commitment to consensus. [7](#0-6)

### Proof of Concept

```
1. Proposer P is scheduled for height H, round R.
2. P streams transactions T_A to validators V1, V2, V3 (majority).
3. P streams transactions T_B to validator V4 (minority).
4. All validators' batchers execute their respective transactions:
   - V1/V2/V3: batcher computes partial_block_hash_A → commitment A
   - V4:        batcher computes partial_block_hash_B → commitment B  (B ≠ A)
5. P sends ProposalFin { proposal_commitment: A } to ALL validators.
6. V1/V2/V3: built_block(A) == received_fin(A) → insert_proposal(A) → prevote(A).
7. V4: insert_proposal(B) [STATE WRITTEN] → built_block(B) != received_fin(A)
       → ProposalFinMismatch returned → no prevote sent.
8. Consensus collects 2/3+ precommits for A → decision_reached(H, R, A) called on V4.
9. V4: get_proposal(H, R, A) → finds entry (H,R) with stored commitment B
       → assert_eq!(B, A) → PANIC → V4 crashes.
``` [8](#0-7) [5](#0-4)

pub(crate) async fn validate_proposal(
    mut args: ProposalValidateArguments,
) -> ValidateProposalResult<ProposalCommitment> {
    let mut content = Vec::new();
    let mut verify_and_store_proof_tasks: Vec<VerifyAndStoreProofTask> = Vec::new();
    let now = args.deps.clock.now();

    let Some(deadline) = now.checked_add_signed(chrono::TimeDelta::from_std(args.timeout).unwrap())
    else {
        return Err(ValidateProposalError::CannotCalculateDeadline { timeout: args.timeout, now });
    };

    is_proposal_init_valid(
        &args.proposal_init_validation,
        &args.init,
        args.deps.clock.as_ref(),
        args.deps.l1_gas_price_provider,
        &args.gas_price_params,
    )
    .await?;

    initiate_validation(
        args.deps.batcher.clone(),
        args.deps.state_sync_client,
        &args.init,
        args.proposal_id,
        args.timeout + args.batcher_timeout_margin,
        args.deps.clock.as_ref(),
        args.compare_retrospective_block_hash,
    )
    .await?;

    let deadline_params = ProposalDeadlineParams {
        clock: args.deps.clock.clone(),
        deadline,
        cancel_token: args.cancel_token.clone(),
    };

    // Validating the rest of the proposal parts.
    let (built_block, received_fin, finished_info) = loop {
        tokio::select! {
            _ = args.cancel_token.cancelled() => {
                // Ignoring batcher errors, to better reflect the proposal interruption.
                batcher_abort_proposal(args.deps.batcher.as_ref(), args.proposal_id).await.ok();
                return Err(ValidateProposalError::ProposalInterrupted(
                    "validating proposal parts".to_string(),
                ));
            }
            _ = args.deps.clock.sleep_until(deadline) => {
                // Ignoring batcher errors, to better reflect the proposal deadline timeout.
                batcher_abort_proposal(args.deps.batcher.as_ref(), args.proposal_id).await.ok();
                return Err(ValidateProposalError::ValidationTimeout(
                    "validating proposal parts".to_string(),
                ));
            }
            proposal_part = args.content_receiver.next() => {
                match handle_proposal_part(
                    args.proposal_id,
                    args.deps.batcher.as_ref(),
                    proposal_part.clone(),
                    &mut content,
                    &mut verify_and_store_proof_tasks,
                    args.deps.transaction_converter.clone(),
                    &deadline_params,
                    args.init.fee_proposal_fri,
                ).await {
                    HandledProposalPart::Finished(built_block, received_fin, finished_info) => {
                        break (built_block, received_fin, finished_info);
                    }
                    HandledProposalPart::Continue => {continue;}
                    HandledProposalPart::Invalid(err) => {
                        // No need to abort since the Batcher is the source of this info.
                        return Err(ValidateProposalError::InvalidProposal(err));
                    }
                    HandledProposalPart::Failed(fail_reason) => {
                        batcher_abort_proposal(args.deps.batcher.as_ref(), args.proposal_id).await?;
                        return Err(ValidateProposalError::ProposalPartFailed(fail_reason,proposal_part));
                    }
                    HandledProposalPart::Timeout(msg) => {
                        // Ignoring batcher errors, to better reflect the validation timeout.
                        batcher_abort_proposal(args.deps.batcher.as_ref(), args.proposal_id).await.ok();
                        return Err(ValidateProposalError::ValidationTimeout(msg));
                    }
                    HandledProposalPart::Interrupted(msg) => {
                        // Ignoring batcher errors, to better reflect the proposal interruption.
                        batcher_abort_proposal(args.deps.batcher.as_ref(), args.proposal_id).await.ok();
                        return Err(ValidateProposalError::ProposalInterrupted(msg));
                    }
                }
            }
        }
    };

    let n_executed_txs = content.iter().map(|batch| batch.len()).sum::<usize>();
    CONSENSUS_NUM_BATCHES_IN_PROPOSAL.set_lossy(content.len());
    CONSENSUS_NUM_TXS_IN_PROPOSAL.set_lossy(n_executed_txs);

    // Update valid_proposals before sending fin to avoid a race condition
    // with `repropose` being called before `valid_proposals` is updated.
    let mut valid_proposals = args.valid_proposals.lock().unwrap();
    valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

    // TODO(matan): Switch to signature validation.
    if built_block != received_fin.proposal_commitment {
        CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
        return Err(ValidateProposalError::ProposalFinMismatch);
    }

    Ok(built_block)
}
