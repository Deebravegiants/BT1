### Title
Pre-Populated Invalid Proof Bypasses Cryptographic Verification in `run_proof_verification` — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

`ProofManager::set_proof` stores a proof without any cryptographic verification. `run_proof_verification` skips verification entirely when `contains_proof` returns `true`. Because the `ProofManager` exposes a `SetProof` RPC endpoint bound to `0.0.0.0` in the distributed deployment with no authentication, an attacker who can reach that port can pre-populate the proof store with an invalid proof for any `proof_facts` key. Every subsequent transaction carrying those same `proof_facts` will have its proof silently accepted without ever being verified, allowing transactions backed by invalid proofs to be sequenced and committed.

---

### Finding Description

**Step 1 — `ProofManager::set_proof` stores without verifying.**

`ProofManager::set_proof` accepts any `(ProofFacts, Proof)` pair and writes it to persistent storage and the LRU cache. There is no call to `starknet_proof_verifier::verify_proof` inside this function. [1](#0-0) 

**Step 2 — `run_proof_verification` treats "already stored" as "already verified".**

The shared verification helper used by both the gateway flow (`spawn_proof_verification`) and the consensus flow (`spawn_verify_and_store_proof`) calls `contains_proof` first. If the result is `true` it returns `Ok(false)` — meaning "no verification was performed" — and the `verify_proof` call is never reached. [2](#0-1) 

**Step 3 — The `SetProof` endpoint is exposed unauthenticated on all interfaces.**

The `ProofManager` component handler dispatches `ProofManagerRequest::SetProof` directly to `set_proof` with no authentication check. [3](#0-2) 

In the distributed deployment the `RemoteComponentServer` for the proof manager is bound to `0.0.0.0`, making it reachable from any network interface.



**Step 4 — Consensus path awaits the (now-skipped) task and proceeds.**

During proposal validation, `handle_proposal_part` collects `VerifyAndStoreProofTask` handles and awaits them at `ProposalFin`. Because `run_proof_verification` returned `Ok(false)` (skipped), the task resolves immediately with `Ok(())`, and the proposal is accepted. [4](#0-3) 

**Step 5 — The blockifier's `validate_proof_facts` only checks the public inputs, not the proof.**

`validate_proof_facts` verifies `program_hash`, `block_hash`, `block_number`, and `config_hash` — all fields drawn from `proof_facts` (the public inputs). It never touches the `Proof` bytes. So a transaction whose proof bytes are garbage but whose `proof_facts` are structurally valid passes blockifier pre-validation. [5](#0-4) 

---

### Impact Explanation

An attacker pre-populates the proof store with an invalid `Proof` for a chosen `proof_facts` key (valid `block_hash`/`block_number`/`program_hash`/`config_hash`, garbage proof bytes). Any Invoke V3 transaction that subsequently carries those `proof_facts` will be accepted by the gateway and by every validating consensus node without the cryptographic proof ever being checked. The transaction is executed by the blockifier and its state changes are committed. Because the proof was never verified, the claimed execution result has no cryptographic backing — wrong state, wrong receipts, and wrong storage values can be committed to the canonical chain.

This matches the **Critical** impact category: wrong state/receipt/execution result from accepted input.

---

### Likelihood Explanation

In the distributed deployment the `ProofManager` `RemoteComponentServer` binds to `0.0.0.0`. No authentication layer is visible in the component server infrastructure. Any party that can reach the proof-manager port — including other nodes in the same data-centre network, or an attacker who has compromised any co-located service — can issue a `SetProof` RPC call. The attacker only needs to know (or predict) a valid `(block_number, block_hash, program_hash, config_hash)` tuple, all of which are public on-chain data. The `proof` payload can be arbitrary bytes.

---

### Recommendation

1. **Verify before storing.** `ProofManager::set_proof` should call `starknet_proof_verifier::verify_proof` before persisting any proof. The "already stored implies already verified" invariant must be enforced at the storage boundary, not assumed by callers.

2. **Authenticate the `SetProof` endpoint.** The `RemoteComponentServer` for the proof manager should require mutual TLS or a shared secret so that only authorised sequencer components can write proofs.

3. **Remove the early-exit optimisation, or make it safe.** If the skip is kept for performance, `run_proof_verification` must only skip when the stored proof was written by a path that already verified it — this requires a separate "verified" flag or a dedicated verified-proof store distinct from the raw storage written by `set_proof`.

        &self,
        proof_facts: ProofFacts,
        proof: Proof,

        if self.contains_proof(proof_facts.clone()).await? {

        let facts_hash = proof_facts.hash();
        self.proof_storage.set_proof(facts_hash, proof.clone()).await?;
        self.cache.insert(facts_hash, proof);
        Ok(())

    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
        {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof version {} is not allowed under this protocol version.",
                snos_proof_facts.proof_version
            )));
        }

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let virtual_os_config_hash = block_context.virtual_os_config_hash();
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
