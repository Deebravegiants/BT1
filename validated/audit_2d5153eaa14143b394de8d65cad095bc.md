### Title
Attacker Can Suppress DRep Dormant-Epoch Counting via Dust Governance Proposals — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Epoch.hs`)

---

### Summary

`updateNumDormantEpochs` increments the dormant-epoch counter only when the active-proposal set is **exactly empty** (`null`). An unprivileged actor can keep at least one live proposal in the queue at all times by repeatedly submitting minimal `InfoAction` governance proposals, paying only the refundable `ppGovActionDepositL`. This permanently suppresses `vsNumDormantEpochs`, causing every DRep that has not voted recently to expire at its raw stored-expiry epoch rather than at the extended actual-expiry epoch (`storedExpiry + numDormantEpochs`). The result is that legitimate DReps are silently removed from the active voting set earlier than the protocol intends, skewing DRep-stake ratios and potentially allowing governance actions to be ratified that would otherwise fail.

---

### Finding Description

The Conway epoch-boundary rule contains:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Epoch.hs
updateNumDormantEpochs currentEpoch ps vState =
  if null $ OMap.filter ((currentEpoch <=) . gasExpiresAfter) $ ps ^. pPropsL
    then vState & vsNumDormantEpochsL %~ succ
    else vState
```

The counter increments **only** when the filtered set of non-expired proposals is empty. [1](#0-0) 

When a transaction carries a new proposal, `updateDormantDRepExpiries` fires first:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs
updateDormantDRepExpiries tx currentEpoch =
  let hasProposals = not . OSet.null $ tx ^. bodyTxL . proposalProceduresTxBodyL
   in if hasProposals
        then certVStateL %~ updateDormantDRepExpiry currentEpoch
        else id
``` [2](#0-1) 

`updateDormantDRepExpiry` adds the accumulated `numDormantEpochs` to every DRep's stored expiry and resets the counter to zero. After that reset, if the attacker's proposal remains live, `updateNumDormantEpochs` never fires again until the proposal expires.

DRep actual expiry is computed as:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs
computeDRepExpiry ppDRepActivity currentEpoch =
  binOpEpochNo (-)
    (addEpochInterval currentEpoch ppDRepActivity)
-- third argument is numDormantEpochs:
-- result = (currentEpoch + activity) - numDormantEpochs
``` [3](#0-2) 

The **actual** expiry visible to the ratification rule is `storedExpiry + numDormantEpochs`. With `numDormantEpochs` pinned at zero by the attacker, the actual expiry equals the stored expiry, so DReps expire `N` epochs sooner than they would in a governance-inactive period of `N` epochs.

The `InfoAction` governance action requires no committee or DRep votes to pass and is explicitly allowed at any time:

```haskell
InfoAction {} -> NoVotingThreshold
``` [4](#0-3) 

The attacker submits one `InfoAction` per `ppGovActionLifetimeL` epochs. The deposit is fully refunded on expiry, so the only cost is the opportunity cost of locking the deposit.

---

### Impact Explanation

With `numDormantEpochs` suppressed, DReps that have not voted since epoch `E` expire at epoch `E + ppDRepActivity` instead of `E + ppDRepActivity + N` (where `N` is the number of genuinely dormant epochs). Expired DReps are excluded from `reDRepDistr` in `dRepAcceptedRatio`:

```haskell
Just drepState
  | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
``` [5](#0-4) 

Removing DReps that would otherwise be counted as "no" or "abstain" raises the accepted ratio, potentially crossing the ratification threshold for `ParameterChange`, `TreasuryWithdrawals`, `HardForkInitiation`, or `UpdateCommittee` actions that would otherwise fail. This maps to **unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action being enacted**.

---

### Likelihood Explanation

Any holder of `ppGovActionDepositL` ADA can execute this attack. `InfoAction` proposals are unconditionally accepted by the `GOV` rule, require no witnesses beyond the submitter's payment key, and are refunded on expiry. The attacker needs to submit one proposal every `ppGovActionLifetimeL` epochs (a small, bounded, recurring cost). No privileged role, key compromise, or majority is required. The attack is fully permissionless and can be sustained indefinitely.

---

### Recommendation

Replace the exact-empty (`null`) guard with a minimum-activity threshold, or track governance activity through a separate, manipulation-resistant signal (e.g., counting only proposals that received at least one vote, or proposals submitted by registered DReps). A minimal fix is to require that the active-proposal set has been empty for at least `k` consecutive epochs before incrementing `numDormantEpochs`, making a single dust proposal insufficient to suppress the counter indefinitely.

---

### Proof of Concept

1. Governance is quiet; `numDormantEpochs` begins incrementing each epoch.
2. Attacker submits `InfoAction` with deposit `ppGovActionDepositL`. `updateDormantDRepExpiry` fires, crystallising the accumulated dormant epochs into stored expiries and resetting `numDormantEpochs = 0`.
3. The attacker's proposal remains live for `ppGovActionLifetimeL` epochs. During this window `updateNumDormantEpochs` sees `null … = False` every epoch and never increments the counter.
4. Before the proposal expires, the attacker submits a fresh `InfoAction`. Step 2 repeats; `numDormantEpochs` is reset to 0 again.
5. DReps that last voted at epoch `E` expire at epoch `E + ppDRepActivity` (stored expiry) rather than `E + ppDRepActivity + N`. After `N` suppressed epochs, `N` DReps that should still be active are treated as expired by `dRepAcceptedRatio`.
6. The attacker submits a governance action (e.g., `TreasuryWithdrawals`) that benefits from the reduced "no"/"abstain" DRep stake. The action ratifies because the denominator of the DRep acceptance ratio has been artificially reduced.
7. The deposit from each expired `InfoAction` is refunded; the attacker's net cost approaches zero.
