### Title
`respond_ckd` Skips Output Validation for Legacy `AppPublicKey` CKD Requests, Enabling Single-Participant Key Substitution - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` validates the CKD output cryptographically for the `AppPublicKeyPV` (publicly verifiable) variant via `ckd_output_check`, but performs **no output validation** for the `AppPublicKey` (privately verifiable, legacy) variant. A single Byzantine attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary fabricated `CKDResponse` for any pending legacy CKD request. The contract accepts it unconditionally, resolves the yield, and delivers the attacker-controlled key material to the user.

---

### Finding Description

In `respond_ckd` at `crates/contract/src/lib.rs` lines 675–682, the output check is gated on the request variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no output check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` on-chain, binding the response to the MPC master key and the user's ephemeral key pair. For `AppPublicKey`, the branch is an empty no-op — the contract accepts any `big_y` and `big_c` values without any cryptographic check. [2](#0-1) 

The `respond` function for threshold signatures, by contrast, always verifies the signature against the derived public key before resolving the yield: [3](#0-2) 

The asymmetry is confirmed by the existing test suite: `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` passes `big_y = [1u8; 48]` and `big_c = [2u8; 48]` — cryptographically invalid values — and the contract accepts them without complaint: [4](#0-3) 

The `AppPublicKey` variant is the legacy default, documented as the "privately verifiable" path and used in the primary e2e CKD test: [5](#0-4) 

---

### Impact Explanation

**Critical.** A single Byzantine attested participant can:

1. Observe a pending `AppPublicKey` CKD request on-chain.
2. Construct a `CKDResponse` where they know the discrete log of `big_y` (i.e., choose `y` freely, set `big_y = y·G1`, `big_c = y·app_pk1`). This produces a response that, when decrypted by the user with their ephemeral private key `a`, yields a secret `s` that the attacker also knows.
3. Call `respond_ckd` before the honest nodes do (front-running the mempool). The contract resolves the yield immediately with the attacker's response.
4. The user receives attacker-controlled key material. Any assets or secrets protected by the derived key are now accessible to the attacker.

This constitutes direct theft of funds or secrets controlled by the MPC network's CKD output, matching the Critical impact tier.

---

### Likelihood Explanation

The attacker is a single Byzantine participant — strictly below the signing threshold — who is already an attested member of the MPC network. The `assert_caller_is_attested_participant_and_protocol_active` check confirms only that the caller is in the active participant set with a valid TEE attestation: [6](#0-5) 

No threshold agreement is required to call `respond_ckd`. The first call to `resolve_yields_for` wins and drains all queued yields for the request. A malicious participant can monitor pending CKD requests and submit a fabricated response before honest nodes respond. The `AppPublicKey` variant is the legacy default path, so it is the most commonly used variant in production.

---

### Recommendation

Apply the same cryptographic output check to the `AppPublicKey` variant. For the privately verifiable case, the contract can verify the weaker condition that `big_c` is a valid G1 point and that `big_y` is a valid G1 point, and additionally verify the BLS signature embedded in the response using the MPC master public key and the `app_id` (analogous to the pairing check done for `AppPublicKeyPV`). Alternatively, require all new CKD requests to use `AppPublicKeyPV` and deprecate the unverifiable `AppPublicKey` path.

---

### Proof of Concept

```rust
#[test]
fn test_respond_ckd_legacy_accepts_arbitrary_response() {
    // Setup: contract with BLS domain, one pending AppPublicKey CKD request
    let (context, mut contract, _secret_key) = basic_setup(Curve::Bls12381, &mut OsRng);
    let app_public_key: dtos::Bls12381G1PublicKey =
        "bls12381g1:6KtVVcAAGacrjNGePN8bp3KV6fYGrw1rFsyc7cVJCqR16Zc2ZFg3HX3hSZxSfv1oH6"
            .parse().unwrap();
    let request = CKDRequestArgs {
        derivation_path: "victim_key".to_string(),
        app_public_key: CKDAppPublicKey::AppPublicKey(app_public_key.clone()),
        domain_id: dtos::DomainId::default(),
    };
    let ckd_request = CKDRequest::new(
        CKDAppPublicKey::AppPublicKey(app_public_key),
        request.domain_id,
        &context.predecessor_account_id,
        &request.derivation_path,
    );
    contract.request_app_private_key(request);
    assert!(contract.get_pending_ckd_request(&ckd_request).is_some());

    // Attacker: single Byzantine attested participant submits fabricated response
    // big_y and big_c are attacker-chosen; attacker knows the corresponding secret
    with_active_participant_and_attested_context(&contract);
    let attacker_response = CKDResponse {
        big_y: dtos::Bls12381G1PublicKey([0xAA; 48]), // attacker-controlled
        big_c: dtos::Bls12381G1PublicKey([0xBB; 48]), // attacker-controlled
    };

    // Contract accepts without any output validation
    contract
        .respond_ckd(ckd_request.clone(), attacker_response)
        .expect("Contract accepted fabricated CKD response for AppPublicKey variant");

    // Yield is resolved; user receives attacker-controlled key material
    assert!(contract.get_pending_ckd_request(&ckd_request).is_none());
}
