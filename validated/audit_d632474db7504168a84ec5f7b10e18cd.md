Audit Report

## Title
Excess ETH sent to `updateRate()` is permanently locked with no recovery path - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary
`MultiChainRateProvider.updateRate()` is `payable` and accepts ETH for LayerZero cross-chain fees, but forwards only the on-chain `estimatedFee` per receiver — not `msg.value`. Any ETH exceeding the sum of per-receiver estimated fees is retained by the contract permanently, as the contract has no `receive()` fallback, no withdrawal function, and no owner-callable ETH recovery path.

## Finding Description
In `updateRate()` (L108–137), for each entry in `rateReceivers`, the contract calls `ILayerZeroEndpoint.estimateFees()` to obtain `estimatedFee`, then calls `ILayerZeroEndpoint.send{ value: estimatedFee }(...)`. The `_refundAddress` argument passed to `send()` is `payable(msg.sender)`, which only triggers a LayerZero-side refund if the actual protocol fee is less than `estimatedFee` — it does not refund ETH that was never forwarded to LayerZero in the first place. After the loop completes, any ETH in `address(this).balance` beyond what was consumed by the loop remains in the contract. The contract inherits only `Ownable` and `ReentrancyGuard` and defines no `receive()`, `fallback()`, or ETH sweep function. There is no mechanism — owner-callable or otherwise — to recover this ETH.

## Impact Explanation
Any ETH overpaid to `updateRate()` is permanently frozen in the contract. This constitutes **permanent freezing of funds** (Critical per the allowed impact scope), as the caller's ETH is irrecoverably locked with zero possibility of retrieval.

## Likelihood Explanation
`updateRate()` has no access control — any external account can call it. Callers must supply ETH for LayerZero fees and routinely overpay to avoid reverts due to fee estimation variance. The contract itself provides `estimateTotalFee()` as a helper, but on-chain estimates at view-call time can differ from estimates at execution time (due to block-to-block gas price changes), making exact matching unreliable in practice. Every call that overpays by even 1 wei results in permanently stuck ETH. The condition is reachable by any unprivileged caller on every invocation.

## Recommendation
After the loop, refund any remaining contract balance to `msg.sender`:

```solidity
uint256 remaining = address(this).balance;
if (remaining > 0) {
    (bool ok,) = payable(msg.sender).call{value: remaining}("");
    require(ok, "ETH refund failed");
}
```

Alternatively, add an owner-callable `recoverETH()` function, or compute the total estimated fee before the loop and `require(msg.value == totalEstimatedFee)` to enforce exact payment.

## Proof of Concept
1. Deploy `MultiChainRateProvider` with 3 rate receivers, each with `estimatedFee = 0.01 ETH` (total needed: `0.03 ETH`).
2. Call `updateRate{value: 0.1 ETH}()` from any EOA.
3. The loop sends `0.01 ETH × 3 = 0.03 ETH` to the LayerZero endpoint.
4. `address(MultiChainRateProvider).balance` is now `0.07 ETH`.
5. Attempt any call to recover the ETH — no such function exists. The `0.07 ETH` is permanently frozen.

Foundry fork test: fork mainnet, deploy a mock `ILayerZeroEndpoint` returning fixed `estimatedFee = 0.01 ether`, add 3 receivers, call `updateRate{value: 0.1 ether}()`, assert `address(provider).balance == 0.07 ether` and that no function can reduce it.
