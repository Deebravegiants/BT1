### Title
ERC1155 `safeTransferFrom` in `finTransfer` permanently locks bridged assets when recipient cannot receive ERC1155 tokens - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.finTransfer` uses `IERC1155.safeTransferFrom` to deliver ERC1155 tokens to the bridge recipient. Because the recipient address is cryptographically bound into the signed payload, any contract recipient that does not implement `IERC1155Receiver` causes every finalization attempt to revert, permanently stranding the user's assets cross-chain with no recovery path.

### Finding Description

In `finTransfer`, when the incoming transfer maps to an ERC1155 token, the contract calls:

```solidity
// OmniBridge.sol L323-330
} else if (multiToken.tokenAddress != address(0)) {
    IERC1155(multiToken.tokenAddress).safeTransferFrom(
        address(this),
        payload.recipient,
        multiToken.tokenId,
        payload.amount,
        ""
    );
}
``` [1](#0-0) 

`IERC1155.safeTransferFrom` mandates that if the recipient is a contract, it must return `IERC1155Receiver.onERC1155Received.selector`; otherwise the call reverts. The `payload.recipient` is part of the Borsh-encoded, MPC-signed message:

```solidity
// OmniBridge.sol L298
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

Because the recipient is embedded in the signed payload, no relayer can substitute a different address without obtaining a fresh MPC signature. The protocol provides no refund or re-routing mechanism for a permanently-failing `finTransfer`. The source chain has already burned or locked the tokens when the outbound transfer was initiated via `initTransfer1155`:

```solidity
// OmniBridge.sol L458-464
IERC1155(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    tokenId,
    amount,
    ""
);
``` [3](#0-2) 

### Impact Explanation

The user's ERC1155 tokens are locked in the source-side bridge contract and can never be delivered on the EVM destination side. Although the nonce is not permanently consumed (the state write at line 287 is reverted along with the failed transaction), the signed payload is immutable, so `finTransfer` will revert on every retry. There is no admin rescue function, no pull-claim fallback, and no refund path. This satisfies **Critical – Irreversible fund lock / permanently unclaimable user value**. [4](#0-3) 

### Likelihood Explanation

Any user who bridges ERC1155 tokens to a contract recipient that does not implement `IERC1155Receiver` (e.g., a Gnosis Safe without the ERC1155 fallback handler, a DAO treasury, a custom vault, or any legacy multisig) triggers this path. The user controls the recipient string at `initTransfer1155` call time; no privileged role is required. The scenario is realistic and reachable by an ordinary unprivileged user.

### Recommendation

ERC1155 does not expose a non-safe `transferFrom`, so the mitigation differs from the ERC721 analog. Two viable approaches:

1. **Try/catch with a claimable escrow**: Wrap the `safeTransferFrom` in a `try/catch`. On failure, record `(recipient → tokenAddress, tokenId, amount)` in a mapping and emit an event. Add a separate `claimERC1155` function that the recipient can call later (or that an admin can redirect after governance).

2. **Low-level bypass**: Use a raw `call` to invoke `safeTransferFrom` and, on failure, fall back to storing the tokens for pull-claim. This avoids the revert-propagation while preserving the ERC1155 standard's intent for cooperative recipients.

Either approach breaks the atomicity that causes the permanent lock.

### Proof of Concept

1. User calls `initTransfer1155(tokenAddress, tokenId, amount, fee, nativeFee, recipientContract, "")` on EVM, where `recipientContract` is a deployed contract without `onERC1155Received` (e.g., a plain Gnosis Safe 1.3.0 with no ERC1155 fallback handler). The ERC1155 tokens are transferred into `OmniBridge` and an `InitTransfer` event is emitted.
2. The MPC observes the event and signs a `TransferMessagePayload` with `recipient = recipientContract` (the address is fixed in the signature).
3. A relayer calls `finTransfer(signatureData, payload)`. The call reaches line 324 and executes `IERC1155(...).safeTransferFrom(address(this), recipientContract, ...)`. Because `recipientContract` does not return the correct selector, the ERC1155 contract reverts.
4. The entire `finTransfer` transaction reverts. The nonce is not consumed, but the signed payload is immutable — no alternative payload with a different recipient can be produced without a new MPC signature.
5. The ERC1155 tokens remain locked in `OmniBridge` on EVM forever. The source-chain tokens are already burned/locked. The user has no recovery path. [5](#0-4)
