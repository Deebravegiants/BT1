Audit Report

## Title
Missing zero-output guard in L2 pool `deposit()` functions allows dust deposits to permanently lose user funds - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All L2 pool `deposit()` variants (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolNoWrapper`) accept ETH or tokens and compute the wrsETH/rsETH output via integer division. When the deposit amount is small enough that the division truncates to zero, the user's funds are permanently absorbed by the pool while zero tokens are minted or transferred in return. The only existing guard (`if (amount == 0) revert InvalidAmount()`) does not catch the case where a non-zero input produces a zero output.

## Finding Description
In `RSETHPoolV3.deposit()`, the ETH path is:

```solidity
// RSETHPoolV3.sol L256-262
if (amount == 0) revert InvalidAmount();
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // executes with rsETHAmount == 0
```

`viewSwapRsETHAmountAndFee` at L299–308 computes:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // integer division
```

When `amountAfterFee * 1e18 < rsETHToETHrate` (e.g., `amountAfterFee = 1 wei`, `rsETHToETHrate ≈ 1.05e18`), `rsETHAmount` truncates to `0`. The `wrsETH.mint(msg.sender, 0)` call succeeds silently (standard ERC-20 mint allows zero amounts), and the user's ETH is permanently retained by the pool with no wrsETH issued.

The same truncation applies to the ERC-20 token path at L334 (`rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate`) and is identically present in `RSETHPoolV3ExternalBridge` (L418–427), `RSETHPoolV3WithNativeChainBridge` (L335–344), and `RSETHPoolNoWrapper` (L277–286, L301–312), where `rsETH.safeTransfer(msg.sender, 0)` also succeeds without revert.

By contrast, `LRTDepositPool._beforeDeposit()` at L667 enforces `if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet()`, providing slippage protection that is entirely absent from all L2 pool variants.

## Impact Explanation
A depositor who sends a dust amount of ETH (e.g., 1 wei) receives 0 wrsETH while their ETH is permanently absorbed into the pool with no recovery path for the user. This matches **Low — Contract fails to deliver promised returns**: the per-call loss is bounded by one unit of the deposited asset (≤ 1 wei of ETH under current rate conditions), making the per-transaction loss negligible, but the contract structurally fails to deliver the promised exchange output without reverting.

## Likelihood Explanation
Under current deployment parameters (`rsETHToETHrate ≈ 1.05e18`, `feeBps` in the low single digits), only a deposit of exactly 1 wei triggers the zero-output path for ETH. Any user or integration that passes a dust amount without a minimum-output check will silently lose it. The condition is reachable by any unprivileged external caller via the public `deposit()` function with no special preconditions. Likelihood is low but non-zero, and no on-chain protection prevents it.

## Recommendation
Add an explicit zero-output guard immediately after computing `rsETHAmount` in every `deposit()` function across all L2 pool variants:

```solidity
if (rsETHAmount == 0) revert ZeroOutputAmount();
```

This should be applied in:
- `RSETHPoolV3.sol` — both ETH and token `deposit()` paths
- `RSETHPoolV3ExternalBridge.sol` — both ETH and token `deposit()` paths
- `RSETHPoolV3WithNativeChainBridge.sol` — both ETH and token `deposit()` paths
- `RSETHPoolNoWrapper.sol` — both ETH and token `deposit()` paths

## Proof of Concept
1. Deploy `RSETHPoolV3` with `feeBps = 0` and configure the oracle to return `rsETHToETHrate = 1.05e18`.
2. Call `deposit{value: 1}("")` (1 wei ETH) from any EOA.
3. `viewSwapRsETHAmountAndFee(1)` returns `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation).
4. `wrsETH.mint(msg.sender, 0)` executes without revert.
5. Caller's 1 wei is in the pool balance; caller holds 0 wrsETH. The ETH is unrecoverable by the caller.

Foundry fuzz test plan: fuzz `amount` over `[1, rsETHToETHrate / 1e18]` with `feeBps = 0`; assert that for any `amount` where `viewSwapRsETHAmountAndFee(amount).rsETHAmount == 0`, the `deposit()` call reverts. Currently this assertion fails, confirming the bug.
