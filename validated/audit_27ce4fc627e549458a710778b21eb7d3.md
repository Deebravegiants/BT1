Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` Enables Front-Running to Steal Yield from rsETH Holders — (File: `contracts/FeeReceiver.sol`)

## Summary

`FeeReceiver.sendFunds()` has no access-control modifier, allowing any caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at will. Because `LRTDepositPool.depositETH()` mints rsETH at the **stored** (stale) oracle price and `LRTOracle.updateRSETHPrice()` is also callable by any unprivileged account (when `pricePercentageLimit` is zero or the reward is within the configured threshold), an attacker can deposit before flushing rewards, then force a price update — capturing a disproportionate share of yield that should accrue to pre-existing rsETH holders.

## Finding Description

**Root cause — three permissionless entry points compose into an exploit:**

1. `FeeReceiver.sendFunds()` carries no role modifier: [1](#0-0) 

2. `LRTDepositPool.receiveFromRewardReceiver()` is an unrestricted payable stub: [2](#0-1) 

3. `LRTOracle.updateRSETHPrice()` is `public whenNotPaused` with no role check: [3](#0-2) 

**Minting uses the stored (stale) price:**

`depositETH()` calls `_beforeDeposit()` → `getRsETHAmountToMint()`, which divides by `lrtOracle.rsETHPrice()` — the last *stored* value, not a freshly computed one: [4](#0-3) 

**Rewards immediately enter TVL once flushed:**

`getETHDistributionData()` counts `address(this).balance` directly, so the moment `sendFunds()` transfers rewards into the pool they are part of the TVL used by the next `updateRSETHPrice()` call: [5](#0-4) 

**Exploit flow:**

1. Attacker deposits A ETH → receives `A·S/T` rsETH at stored price `T/S` (pre-reward).
2. Attacker calls `FeeReceiver.sendFunds()` → R ETH moves to deposit pool; TVL becomes `T + A + R`.
3. Attacker calls `LRTOracle.updateRSETHPrice()` → new price = `(T+A+R−fee) / (S + A·S/T)`.
4. Attacker's rsETH is now worth `A + A·R/(T+A) − fee_share` ETH.
5. **Net profit ≈ `A·R/(T+A)`** — a direct transfer of yield from pre-existing holders.

**Why existing guards are insufficient:**

`_updateRsETHPrice()` contains a `pricePercentageLimit` check that reverts for non-managers if the price increase exceeds the configured threshold: [6](#0-5) 

This is not a complete mitigation because: (a) `pricePercentageLimit` defaults to `0`, disabling the check entirely; (b) even when set, an attacker can exploit smaller reward accumulations that stay within the threshold; (c) if the attacker cannot force step 3 themselves, they still benefit when a manager eventually calls `updateRSETHPriceAsManager()` — the deposit-before-flush advantage is already locked in.

The protocol fee taken during `_updateRsETHPrice()` reduces profit marginally but does not prevent the attack: [7](#0-6) 

## Impact Explanation

**High — Theft of unclaimed yield.** Pre-existing rsETH holders receive only `R·T/(T+A)` of the R ETH reward instead of the full R. With a flash loan amplifying A toward the total TVL T, the attacker can capture nearly the entire reward in a single block. The stolen amount is concrete and measurable: `A·R/(T+A)` ETH per attack, repeatable every time rewards accumulate in the FeeReceiver.

## Likelihood Explanation

**High.** No privileged role, leaked key, or governance action is required. Both `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` are reachable by any EOA or contract. The FeeReceiver accumulates ETH continuously from validator tips and MEV; MEV searchers routinely monitor on-chain balances. The attack is profitable whenever pending rewards exceed gas cost, which is true after any meaningful accumulation period. Flash loans make the capital requirement negligible.

## Recommendation

1. **Restrict `FeeReceiver.sendFunds()`** to a trusted role so reward-flush timing cannot be controlled by an adversary:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

2. **Alternatively**, snapshot pending rewards at deposit time and exclude them from the minting price until a permissioned price update is performed, preventing a depositor from benefiting from rewards that were already accruing before their deposit.

3. As defence-in-depth, ensure `pricePercentageLimit` is always set to a non-zero value on deployment and document it as a required invariant, so the threshold check in `_updateRsETHPrice()` is always active.

## Proof of Concept

**Minimal call sequence (single block with flash loan):**

```
State:  FeeReceiver holds R = 50 ETH (accumulated MEV).
        Protocol TVL T = 1 000 ETH, rsETH supply S = 1 000, stored price = 1.00 ETH/rsETH.
        pricePercentageLimit = 0 (default).

1. Attacker flash-borrows 1 000 ETH.
2. LRTDepositPool.depositETH{value: 1000 ETH}()
       → minted 1 000 rsETH at stored price 1.00.
       → pool balance = 2 000 ETH, supply = 2 000, stored price still 1.00.
3. FeeReceiver.sendFunds()                    ← no modifier, anyone can call
       → 50 ETH transferred to deposit pool.
       → pool balance = 2 050 ETH.
4. LRTOracle.updateRSETHPrice()               ← public whenNotPaused, no role
       → totalETHInProtocol = 2 050 ETH (via address(this).balance).
       → previousTVL = 2 000 * 1.00 = 2 000 ETH.
       → rewardAmount = 50 ETH; fee taken (e.g. 5% → 2.5 ETH).
       → newPrice = (2 050 − 2.5) / 2 000 ≈ 1.02375 ETH/rsETH.
5. Attacker's 1 000 rsETH ≈ 1 023.75 ETH.
   Repay flash loan (1 000 ETH) → net profit ≈ 23.75 ETH.

Pre-existing holders: their 1 000 rsETH is worth ≈ 1 023.75 ETH instead of ≈ 1 047.5 ETH.
Yield stolen: ≈ 23.75 ETH out of 47.5 ETH net reward.
```

**Foundry fork test plan:** Deploy against a mainnet fork; seed `FeeReceiver` with 50 ETH; record pre-existing holder rsETH value; execute the four-step sequence above in a single `vm.prank` block; assert that the attacker's ending ETH balance exceeds starting balance and that pre-existing holder value is less than `T + R − fee`.
