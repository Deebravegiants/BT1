### Title
Stale Participant-State Not Cleaned Up After `vote_cancel_resharing` Breaks Quorum Accounting Invariants - (File: `crates/contract/src/lib.rs`)

### Summary

`vote_reshared` spawns six post-transition cleanup promises to purge stale per-participant state accumulated during resharing. `vote_cancel_resharing` performs no equivalent cleanup. When resharing is cancelled and the contract reverts to the previous running state, TEE image-hash votes, TEE verifier-change votes, node-migration records, and foreign-chain support entries from prospective (non-current) participants remain in contract storage and continue to count toward quorum thresholds in the reverted running state.

### Finding Description

`vote_reshared` in `crates/contract/src/lib.rs` spawns six detached cleanup promises upon a successful resharing transition: [1](#0-0) 

These promises invoke:
- `REMOVE_NON_PARTICIPANT_UPDATE_VOTES`
- `CLEAN_TEE_STATUS` → `tee_state.clean_non_participant_votes(participants)`
- `CLEAN_INVALID_ATTESTATIONS`
- `CLEANUP_ORPHANED_NODE_MIGRATIONS`
- `CLEAN_FOREIGN_CHAIN_DATA`
- `REMOVE_NON_PARTICIPANT_TEE_VERIFIER_VOTES`

`vote_cancel_resharing`, by contrast, simply reassigns `self.protocol_state` and returns: [2](#0-1) 

The underlying state transition in `ResharingContractState::vote_cancel_resharing` clones and returns the `previous_running_state` unchanged: [3](#0-2) 

After cancellation, `tee_state.votes` (TEE image-hash votes), `tee_verifier_votes`, `node_migrations`, and `node_foreign_chain_support` all retain entries written by prospective participants during the resharing window. The `clean_tee_status` endpoint, which calls `tee_state.clean_non_participant_votes`, is never invoked: [4](#0-3) 

The unit test `test_clean_non_participant_votes_removes_stale_votes` explicitly demonstrates that without this cleanup, stale votes from removed participants are counted toward quorum: [5](#0-4) 

The same pattern applies to `tee_verifier_votes`; `remove_non_participant_tee_verifier_votes` retains only current-participant votes: [6](#0-5) 

### Impact Explanation

After resharing cancellation, prospective participants' TEE image-hash votes and TEE verifier-change votes remain in `tee_state.votes` and `tee_verifier_votes` respectively. Because vote counting iterates all entries in `proposal_by_account` without filtering by the current participant set, these stale entries reduce the number of genuine current-participant votes required to reach the governance threshold.

In the worst case — where enough prospective participants voted for a target proposal during resharing — the threshold can be reached with zero additional votes from current participants after cancellation. This allows:

- **Unauthorized TEE image-hash approval**: A malicious node image hash could be approved, permitting nodes running adversarial software to submit valid attestations and join the MPC network.
- **Unauthorized TEE verifier substitution**: A malicious verifier account could be installed, enabling it to approve forged attestations for adversarial nodes.

Either outcome breaks the invariant that only current participants' votes govern TEE configuration, and could ultimately enable unauthorized participation in threshold signing or key derivation.

### Likelihood Explanation

The attack path is reachable by an unprivileged adversary who:
1. Becomes a prospective participant in a resharing proposal (no special privilege required beyond being nominated).
2. Votes for a target TEE image hash or verifier during the resharing window (a normal participant action).
3. Coordinates with threshold-many previous-running-state participants to cancel the resharing (or waits for an organic cancellation due to a stalled node).
4. After cancellation, the stale votes persist with no on-chain mechanism to remove them.

Resharing cancellation is a documented, tested production flow. The adversary does not need to compromise any existing participant or perform any off-chain attack.

### Recommendation

Add the same set of detached cleanup promises to `vote_cancel_resharing` that `vote_reshared` already spawns. At minimum, `CLEAN_TEE_STATUS` and `REMOVE_NON_PARTICIPANT_TEE_VERIFIER_VOTES` must be invoked after the state reverts to the previous running state, so that stale votes from prospective participants are purged before they can influence future quorum checks.

### Proof of Concept

```
Setup: N=5 current participants, governance threshold T=3.

1. Propose resharing to a new set that includes 3 fresh prospective participants
   (P_new_1, P_new_2, P_new_3) alongside 2 retained participants.

2. During resharing, P_new_1, P_new_2, P_new_3 each call vote_tee_image_hash
   with a malicious_hash. Three entries are written to tee_state.votes.

3. Three previous-running-state participants call vote_cancel_resharing.
   Threshold reached; contract reverts to previous RunningContractState.
   No cleanup promises are spawned.

4. tee_state.votes now contains 3 stale entries for malicious_hash from
   prospective participants who are NOT in the reverted participant set.

5. Any single current participant calls vote_tee_image_hash(malicious_hash).
   Vote count = 3 (stale) + 1 (current) = 4 >= T=3.
   malicious_hash is approved as the authorised node image hash.

6. Nodes running the malicious image can now submit valid-looking attestations,
   pass is_caller_an_attested_participant checks, and participate in threshold
   signing or CKD without holding a legitimate key share.
```

    pub fn vote_cancel_resharing(&mut self) -> Result<Option<RunningContractState>, Error> {
        let previous_running_participants = self.previous_running_state.parameters.participants();
        let authenticated_candidate = AuthenticatedAccountId::new(previous_running_participants)?;
        self.cancellation_requests.insert(authenticated_candidate);

        let cancellation_votes_count = self.cancellation_requests.len() as u64;
        let previous_running_threshold = self.previous_running_state.parameters.threshold();

        let threshold_cancellation_votes_reached: bool =
            cancellation_votes_count >= previous_running_threshold.value();

        let running_state = if threshold_cancellation_votes_reached {
            let mut previous_running_state = self.previous_running_state.clone();
            let prospective_epoch_id = self.prospective_epoch_id();
            previous_running_state.previously_cancelled_resharing_epoch_id =
                Some(prospective_epoch_id);

            Some(previous_running_state)
        } else {
            None
        };

        Ok(running_state)
    }

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
    }
