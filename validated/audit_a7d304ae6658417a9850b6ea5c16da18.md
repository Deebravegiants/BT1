### Title
Absent CKD Output Validation for `AppPublicKey` Variant Allows Single Byzantine Participant to Corrupt Confidential Key Derivation Responses - (File: `crates/contract/src/lib.rs`)

### Summary

In `respond_ckd`, the contract performs a cryptographic pairing check (`ckd_output_check`) on the CKD response only when the request uses the `AppPublicKeyPV` (publicly verifiable) variant. For the `AppPublicKey` (privately verifiable) variant, the match arm is an empty no-op, and any arbitrary `CKDResponse` is accepted and delivered to the waiting caller without any on-chain verification. A single Byzantine attested participant (strictly below the signing threshold) can exploit this to front-run any pending `AppPublicKey` CKD request and resolve it with a forged response, permanently corrupting the user's derived key and blocking the legitimate MPC response.

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function contains the following branching logic:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` via `ckd_output_check`: [2](#0-1) 

For `AppPublicKey`, the arm is `{}` — no cryptographic check whatsoever. The `response` (containing arbitrary `big_y` and `big_c` BLS12-381 G1 points) is immediately serialized and delivered to the yield-resume callback.

The only caller restrictions are:
1. `assert_caller_is_signer()` — NEAR account-level check
2. `assert_caller_is_attested_participant_and_protocol_active()` — must be an attested participant [3](#0-2) 

There is no check that the responding node was part of the signing group elected for this specific request. Any attested participant can call `respond_ckd` for any pending `AppPublicKey` CKD request.

The analog to the original Roll Token ID bug is structural: the `AppPublicKeyPV` branch enforces correctness (like the "valid roll" path), while the `AppPublicKey` branch is an unrestricted fallthrough (like the "else block that assigns without restriction"). The attacker exploits the absent validation path to substitute arbitrary output parameters.

### Impact Explanation

A single Byzantine attested participant (below signing threshold) can:

1. Monitor the NEAR blockchain for pending `CKDRequest` entries with `AppPublicKey` variant.
2. Call `respond_ckd` with the correct `request` key (to match the pending queue entry) but with arbitrary `big_y` and `big_c` values.
3. The contract accepts the forged response without any cryptographic check and resolves all queued yields for that request key.
4. The user's NEAR transaction completes and returns the forged `CKDResponse`.
5. The legitimate MPC response can no longer be submitted — the pending request has been removed from the queue.

The user receives a wrong key: they compute `big_c' − app_sk · big_y'`, which is an arbitrary BLS12-381 G1 point unrelated to the MPC master secret. If the user's application has already committed assets to an address derived from the expected CKD key, those assets become permanently inaccessible. The attacker can repeat this for every subsequent retry, making the denial persistent.

This breaks the production safety invariant that CKD responses must be cryptographically bound to the MPC network's master secret and the user's `app_id`.

**Impact class**: Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants. Escalates toward Critical if the application has pre-committed funds to an address derived from the expected CKD output.

### Likelihood Explanation

The attacker must be an attested participant — a role that requires being voted into the participant set and maintaining a valid TEE attestation. However, the attack requires only a single Byzantine participant strictly below the signing threshold. Once in the participant set, the attack is:

- **Permissionless within the role**: no additional privilege beyond being an attested participant is needed.
- **Deterministic**: the attacker can observe pending requests on-chain and submit the forged response in the same or next block.
- **Repeatable**: the attacker can target every new `AppPublicKey` CKD request indefinitely.

The `AppPublicKey` variant is described as the "legacy" and default variant in the README, making it the most commonly used path.

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` variant. Since `AppPublicKey` provides only a G1 point (no G2 component for the pairing equation), the check must be adapted. The simplest fix is to require callers to use `AppPublicKeyPV` for all new requests and deprecate `AppPublicKey`, or to add an equivalent binding check using the G1 public key and the network's G2 public key. At minimum, add a comment explaining why no check is performed and document the trust assumption explicitly.

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_app_pk) => {
        // TODO: add binding check for AppPublicKey variant
        // Currently no on-chain verification is possible without a G2 component.
        // Consider requiring AppPublicKeyPV for all new requests.
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates the vulnerability: it submits a response with `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (invalid BLS12-381 points) for an `AppPublicKey` request, and the contract accepts it without error: [4](#0-3) 

**Attack steps**:

1. User submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk)`.
2. Attacker (Byzantine attested participant) observes the pending request on-chain.
3. Attacker calls `respond_ckd(ckd_request, CKDResponse { big_y: [0u8;48], big_c: [0u8;48] })`.
4. Contract executes `AppPublicKey(_) => {}` — no check — and calls `resolve_yields_for`.
5. User's transaction resolves with the forged response; legitimate MPC response is now rejected as `RequestNotFound`. [1](#0-0)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
