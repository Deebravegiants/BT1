### Title
Insufficient Chainlink Price Verification in `ChainlinkPriceOracle` Allows Stale Prices to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validity fields (`updatedAt`, `answeredInRound`, `roundId`), accepting stale or incomplete round data without reverting. This is the oracle used by the L1 core protocol (`LRTOracle` → `LRTDepositPool`) to price LST assets when minting rsETH. The L2 pool oracle wrapper (`ChainlinkOracleForRSETHPoolCollateral`) correctly validates all three conditions; the L1 core oracle does not.

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` reads only the `price` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

It ignores `roundId`, `updatedAt`, and `answeredInRound`. No check is performed for:
- `answeredInRound < roundId` (stale round)
- `updatedAt == 0` (incomplete round)
- `price <= 0` (invalid price)

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` — used by L2 pools for collateral token pricing — performs all three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The stale price flows through `LRTOracle.getAssetPrice()`: [3](#0-2) 

into `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

and is used directly in `depositAsset()` and `depositETH()` to determine how many rsETH tokens to mint for a depositor. [5](#0-4) 

### Impact Explanation

If a Chainlink LST/ETH feed goes stale (e.g., during network congestion or a rapid market move), the last reported price remains in the feed. If that stale price is **higher** than the true current price (e.g., the LST has depegged but Chainlink has not yet updated), a depositor can call `depositAsset()` with the depegged LST and receive rsETH calculated at the inflated stale rate. This mints excess rsETH backed by less real value, diluting all existing rsETH holders and constituting theft of their yield. Repeated exploitation during a prolonged stale period can cause protocol insolvency.

**Impact: High — Theft of unclaimed yield / potential protocol insolvency.**

### Likelihood Explanation

Chainlink feeds have documented heartbeat intervals (e.g., 24 hours for some LST feeds) and deviation thresholds. During extreme market events or L1 congestion, feeds can remain stale for hours. The entry path requires no special permissions — any unprivileged depositor can call `depositAsset()` at any time. The condition (stale feed + depegged LST) is realistic and has occurred historically with LSTs.

**Likelihood: Medium.**

### Recommendation

Apply the same validation pattern used in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt != 0, "Incomplete round");
    require(price > 0, "Invalid price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, consider adding a configurable `maxStaleness` duration check (`block.timestamp - updatedAt <= maxStaleness`).

### Proof of Concept

1. Assume stETH/ETH Chainlink feed last updated 25 hours ago at `1.05e18` (stETH at a premium). The true current rate has dropped to `0.95e18` due to a depeg event, but the feed has not updated.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint` computes: `(1000e18 * 1.05e18) / rsETHPrice` — using the stale `1.05e18` rate.
4. Attacker receives rsETH worth ~10.5% more than the actual value of the deposited stETH.
5. Existing rsETH holders are diluted by the excess minted supply. [1](#0-0) [6](#0-5)
