### Title
P2P Sync Accepts Peer-Supplied Transaction Hashes Without Validation, Corrupting the `transaction_hash_to_idx` Index and Causing Wrong RPC Responses - (File: `crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

The P2P sync client accepts `transaction_hash` values from untrusted peers verbatim and writes them directly into the authoritative `transaction_hash_to_idx` storage table. Because no validation is performed against the transaction content or the block header's `transaction_commitment`, a malicious peer can supply a fabricated hash for any transaction. This corrupts the hash-to-index mapping: the real transaction hash is never indexed, making the transaction invisible to all hash-based RPC lookups, while the fabricated hash maps to the wrong transaction data.

---

### Finding Description

In `crates/apollo_p2p_sync/src/client/transaction.rs`, `TransactionStreamFactory::parse_data_for_block` collects `FullTransaction` objects from a peer and pushes the peer-supplied `transaction_hash` directly into the `BlockBody` without any verification:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [1](#0-0) 

The resulting `BlockBody` is then written to storage via `append_body` → `write_transactions`. Inside `write_transactions`, the peer-supplied hash is inserted into two tables:

1. `transaction_hash_to_idx_table.insert(txn, tx_hash, &transaction_index)` — the authoritative reverse-lookup index used by every hash-based RPC query.
2. `transaction_metadata_table.append(txn, &transaction_index, &TransactionMetadata { tx_hash: *tx_hash, ... })` — the per-index metadata record. [2](#0-1) 

Neither call validates that `tx_hash` equals the hash computed from the transaction's actual content, nor that it matches the `transaction_commitment` Merkle root stored in the already-synced block header.

**Attack scenario (unique fabricated hash):** A malicious peer sends a `FullTransaction` for transaction `T` with `transaction_hash = H_fake` (any value not already in the DB). The write succeeds. The real hash `H_real = hash(T)` is never inserted into `transaction_hash_to_idx_table`. Downstream:

- `get_transaction_idx_by_hash(H_real)` returns `None` — the transaction is invisible.
- `get_transaction_idx_by_hash(H_fake)` returns the index and the transaction body `T`, but with the wrong hash attached. [3](#0-2) 

**Attack scenario (collision with existing hash):** A malicious peer sends `transaction_hash = H_existing` where `H_existing` is already in the DB from a prior block. The `insert` call returns `KeyAlreadyExists`, causing the entire block-body write to fail and the sync node to be unable to advance past that block. [4](#0-3) 

The block header's `transaction_commitment` (a Poseidon Merkle root over `H(tx_hash, tx_signature)` for each transaction) is synced via a separate protocol and is already stored before the body arrives. The body-write path never cross-checks the stored hashes against this commitment. [5](#0-4) 

---

### Impact Explanation

Every hash-based RPC endpoint reads through `get_transaction_idx_by_hash`:

- `starknet_getTransactionByHash(H_real)` → `TRANSACTION_HASH_NOT_FOUND` for a transaction that exists on-chain.
- `starknet_getTransactionReceipt(H_real)` → not found.
- `starknet_getTransactionStatus(H_real)` → not found.
- `starknet_simulateTransactions` / `starknet_traceTransaction` → wrong or missing execution context when the transaction index is resolved via the corrupted table. [6](#0-5) 

The node presents these wrong answers with full authority (no "pending" or "unverified" qualification), matching the impact class: **High — RPC returns an authoritative-looking wrong value**.

---

### Likelihood Explanation

Any participant in the Starknet P2P network can act as a sync peer. The P2P sync client connects to peers to download block bodies and accepts their responses without authentication of individual field values. The `TODO(eitan)` comment in the production code explicitly acknowledges the missing validation. No privilege is required beyond being a reachable peer.

---

### Recommendation

After receiving each `FullTransaction` from a peer, recompute the transaction hash from the transaction content and compare it to the peer-supplied value:

```rust
let computed_hash = transaction.calculate_transaction_hash(&chain_id)?;
if computed_hash != transaction_hash {
    return Err(ParseDataError::BadPeer(BadPeerError::InvalidTransactionHash { ... }));
}
block_body.transaction_hashes.push(computed_hash); // use the locally computed hash
```

Additionally, after assembling the full `BlockBody`, verify that the Merkle commitment over the transaction leaf elements matches the `transaction_commitment` already stored in the block header before calling `append_body`. This mirrors the Eigenlayer recommendation in the original report: derive the authoritative identifier from the cryptographic path rather than trusting the caller-supplied value.

---

### Proof of Concept

1. Run a sync node that connects to a controlled malicious peer.
2. The malicious peer responds to a transaction query for block `N` with `FullTransaction { transaction: T, transaction_output: O, transaction_hash: H_fake }` where `H_fake ≠ hash(T)`.
3. `parse_data_for_block` pushes `H_fake` into `block_body.transaction_hashes` without checking.
4. `write_to_storage` calls `append_body(N, block_body)`.
5. `write_transactions` inserts `H_fake → TransactionIndex(N, 0)` into `transaction_hash_to_idx_table`.
6. The real hash `H_real` has no entry in `transaction_hash_to_idx_table`.
7. Call `starknet_getTransactionByHash(H_real)` on the node's RPC → returns `TRANSACTION_HASH_NOT_FOUND`.
8. Call `starknet_getTransactionByHash(H_fake)` → returns transaction `T` with hash `H_fake` (wrong hash attached to wrong transaction identity). [7](#0-6) [8](#0-7) [9](#0-8)

    fn parse_data_for_block<'a>(
        transactions_response_manager: &'a mut ClientResponsesManager<DataOrFin<FullTransaction>>,
        block_number: BlockNumber,
        storage_reader: &'a StorageReader,
    ) -> BoxFuture<'a, Result<Option<Self::Output>, ParseDataError>> {
        async move {
            let mut block_body = BlockBody::default();
            let mut current_transaction_len = 0;
            let target_transaction_len = storage_reader
                .begin_ro_txn()?
                .get_block_header(block_number)?
                .expect("A header with number lower than the header marker is missing")
                .n_transactions;
            while current_transaction_len < target_transaction_len {
                let maybe_transaction = transactions_response_manager.next().await.ok_or(
                    ParseDataError::BadPeer(BadPeerError::SessionEndedWithoutFin {
                        type_description: Self::TYPE_DESCRIPTION,
                    }),
                )?;
                let Some(FullTransaction { transaction, transaction_output, transaction_hash }) =
                    maybe_transaction?.0
                else {
                    if current_transaction_len == 0 {
                        return Ok(None);
                    } else {
                        return Err(ParseDataError::BadPeer(BadPeerError::NotEnoughTransactions {
                            expected: target_transaction_len,
                            actual: current_transaction_len,
                            block_number: block_number.0,
                        }));
                    }
                };
                block_body.transactions.push(transaction);
                block_body.transaction_outputs.push(transaction_output);
                // TODO(eitan): Validate transaction hash from untrusted sources
                block_body.transaction_hashes.push(transaction_hash);
                current_transaction_len += 1;
            }
            Ok(Some((block_body, block_number)))
        }
        .boxed()
    }
