### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Pending `request`, Enabling Cross-Request Signature Replay for Forged Foreign-Chain Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never checks that `response.payload_hash` was actually derived from the `request` argument. A single malicious attested MPC node (Byzantine participant below threshold) can replay any previously-issued, publicly-visible `payload_hash`/`signature` pair from an earlier signing session to resolve an unrelated pending request, delivering a fabricated foreign-chain verification result to the caller.

---

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` (lines 691–754) accepts two independent arguments:

- `request: VerifyForeignTransactionRequest` — the pending on-chain request (tx_id, extractors, domain, payload version)
- `response: VerifyForeignTransactionResponse` — contains `payload_hash: Hash256` and `signature: SignatureResponse`

The only cryptographic check performed is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract confirms that `response.signature` is a valid root-key signature over `response.payload_hash`. It does **not** confirm that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the specific `request` being resolved.

After the signature check passes, the full `response` (including the unchecked `payload_hash`) is serialised and delivered to every queued yield under `request`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The correct binding — computing the expected hash from `(request, extracted_values)` and comparing it to `response.payload_hash` — is only performed in the **SDK helper** used by downstream user contracts, not in the MPC contract itself:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [3](#0-2) 

This is the direct analog of the PortalFacet bug: just as `repayAavePortalFor` accepted any valid token to clear a debt identified by `_transferId` without checking the token matched the borrowed asset, `respond_verify_foreign_tx` accepts any validly-signed `payload_hash` to resolve a request identified by `request` without checking the hash encodes that request.

---

### Impact Explanation

A single malicious attested MPC node can execute the following cross-request signature replay:

1. **Collect a prior valid response.** Any previously completed `verify_foreign_transaction` for tx_id X produces a publicly-visible `{payload_hash: H_X, signature: sig_X}` where `H_X = SHA-256(borsh(ForeignTxSignPayload { request_X, values_X }))`. The signature `sig_X` is a valid root-key ECDSA signature over `H_X`.

2. **Observe a new pending request.** A new `verify_foreign_transaction` for tx_id Y is submitted and stored as a pending yield under `request_Y`.
