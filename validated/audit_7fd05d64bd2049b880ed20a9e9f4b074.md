Looking at the exact code path for `vote_code_hash`, `count_votes`, and `clean_tee_status`, and how they interact across the resharing boundary.

### Title
Stale Pre-Resharing Vote Counted in `vote_code_hash` Before `clean_tee_status` Executes, Enabling Below-Threshold TEE Image Whitelisting — (`crates/contract/src/lib.rs`, `crates/contract/src/tee/proposal.rs`)

---

### Summary

`vote_code_hash` compares the raw `count_votes` result (which counts every entry in `proposal_by_account` without filtering by current participants) against the post-resharing threshold. Because `clean_tee_status` is spawned as a **detached** promise and executes in a later block, a removed participant's stale vote persists in the map during the inter-block window. A single Byzantine participant in the new set can combine their fresh vote with the stale vote to reach threshold and whitelist an attacker-controlled image hash.

---

### Finding Description

**Root cause — `count_votes` is unfiltered:** [1](#0-0) 

`count_votes` iterates all values in `proposal_by_account` and counts matches. It has no knowledge of who is currently a participant.

**`vote_code_hash` uses this raw count directly against the current threshold:** [2](#0-1) 

`self.tee_state.vote(code_hash, &participant)` returns the unfiltered count. `self.threshold()?.value()` returns the **new** (post-resharing) threshold. There is no filtering step equivalent to what `vote_update` does.

**Contrast: `vote_update` explicitly filters by current participants before checking threshold:** [3](#0-2) 

The comment at line 1362 reads: *"Filter votes to only count current participants voting for this specific update. This ensures correctness even if the cleanup promise in `MpcContract::vote_reshared()` fails."* `vote_code_hash` has no such guard.

**`clean_tee_status` is detached — it runs in a later block:** [4](#0-3) 

`.detach()` creates a new action receipt that is queued for a subsequent block. Between the block where `vote_reshared` completes and the block where `clean_tee_status` executes, the stale vote remains in `proposal_by_account`.

**`clean_tee_status` itself only filters votes, not the whitelist:** [5](#0-4) [6](#0-5) 

Once `whitelist_tee_proposal` fires before `clean_tee_status` runs, the hash is already in `allowed_docker_image_hashes` — cleanup of the vote map is irrelevant.

---

### Impact Explanation

**Concrete attack (3-of-3 → 2-of-2 resharing):**

1. Old set `{P_evil, P2, P3}`, threshold = 3.
2. P_evil calls `vote_code_hash(H_evil)` → 1 entry in `proposal_by_account`. Count = 1 < 3, no whitelist.
3. Resharing to `{P2, P3}`, threshold = 2 completes via `vote_reshared`. State is now Running with new participants. `clean_tee_status` receipt is queued but **not yet executed**.
4. P2 (Byzantine, colluding with P_evil) calls `vote_code_hash(H_evil)` in the same or next block.
5. `count_votes` returns **2** (P_evil stale + P2 fresh). `self.threshold()?.value()` = 2. Condition `2 >= 2` is true.
6. `whitelist_tee_proposal(H_evil)` executes → H_evil is inserted into `allowed_docker_image_hashes`.

Only **1 of 2** new participants needs to be Byzantine (below the new threshold of 2). P_evil's stale vote bridges the gap.

**Effect:** H_evil is now an allowed MPC Docker image hash. Any node running attacker-controlled code that produces a quote referencing H_evil will pass `submit_participant_info` attestation verification, enabling it to be added as a participant and participate in threshold signing.

This is an **attestation authorization bypass**: the TEE image whitelist, which is the on-chain root of trust for which code may touch key shares, is modified without genuine threshold agreement from the current participant set.

---

### Likelihood Explanation

- Requires P_evil to have been a legitimate participant in the old set (not an external attacker).
- Requires P2 to be Byzantine in the new set — 1 of 2 participants in the minimal scenario.
- Requires P2 to submit `vote_code_hash` before `clean_tee_status` executes. In NEAR, detached receipts are processed in the next block (~1 second). P2 can submit the transaction in the same block as `vote_reshared` completes, guaranteeing it is processed before the cleanup receipt.
- The timing window is deterministic and reliable, not a race condition in the probabilistic sense.

---

### Recommendation

Apply the same participant-filtering pattern used in `vote_update` to `vote_code_hash`. Before comparing against threshold, filter `proposal_by_account` to only count votes from accounts that are in the current participant set:

```rust
// In vote_code_hash, replace:
let votes = self.tee_state.vote(code_hash, &participant);
if votes >= self.threshold()?.value() { ... }

// With:
self.tee_state.vote(code_hash, &participant);
let current_participants = threshold_parameters.participants();
let filtered_votes = self.tee_state.count_votes_for_current_participants(
    code_hash, current_participants
);
if filtered_votes >= self.threshold()?.value() { ... }
```

This makes `vote_code_hash` safe regardless of whether `clean_tee_status` has executed, matching the defensive pattern already applied to `vote_update`. [7](#0-6) 

`get_remaining_votes` already implements the filtering logic needed — it can be reused or adapted for an inline count.

---

### Proof of Concept

The existing unit test at `tee_state.rs:1504-1545` demonstrates that `clean_non_participant_votes` correctly removes stale votes **after** it is called. [8](#0-7) 

The vulnerability is the **window before** that call. A deterministic unit test proving the exploit:

1. Create a 3-of-3 contract (`setup_tee_test_contract(3, 3)`).
2. Have P_evil (`participants[0]`) call `vote_code_hash(H_evil)` → 1 vote, no whitelist.
3. Simulate resharing: directly mutate `contract.protocol_state` to a Running state with `{P2, P3}` and threshold=2 (as done in the existing test at `lib.rs:3958-3964`). Do **not** call `clean_tee_status`.
4. Have P2 call `vote_code_hash(H_evil)`.
5. Assert `contract.tee_state.allowed_docker_image_hashes` contains H_evil.
6. Assert `contract.tee_state.votes.proposal_by_account` was cleared (confirming `whitelist_tee_proposal` fired).

This proves H_evil is whitelisted with only 1 current-participant vote (below the threshold of 2) by leveraging P_evil's stale entry.

    /// quorum after resharing.

    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
    }
