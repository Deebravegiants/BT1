Audit Report

## Title
Stale `rsETHPrice` Allows Depositors to Mint Excess rsETH, Stealing Unclaimed Yield from Existing Holders - (File: contracts/LRTDepositPool.sol, contracts/LRTOracle.sol)

## Summary

`LRTOracle` stores the rsETH/ETH exchange rate in the state variable `rsETHPrice`, which is only refreshed when `updateRSETHPrice()` is explicitly called. Neither `LRTDepositPool.depositAsset()` nor `depositETH()` invokes `updateRSETHPrice()` before computing the rsETH mint amount, meaning any depositor who acts during a staleness window receives more rsETH than their deposit warrants. The excess rsETH is funded by diluting the accumulated, unclaimed yield of all existing rsETH holders.

## Finding Description

`LRTOracle.rsETHPrice` is a plain storage variable updated only inside `_updateRsETHPrice()`, which is reached via the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`.

```solidity
// LRTOracle.sol:28
uint256 public override rsETHPrice;

// LRTOracle.sol:87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`LRTDepositPool.depositAsset()` calls `_beforeDeposit()` → `getRsETHAmountToMint()`, which reads the stored value directly:

```solidity
// LRTDepositPool.sol:519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

No call to `updateRSETHPrice()` appears anywhere in `depositAsset()`, `depositETH()`, or their internal helpers.

As staking rewards accrue (e.g., stETH rebases, EigenLayer yield), `_getTotalEthInProtocol()` grows continuously, but `rsETHPrice` only reflects this growth after an explicit update. During the gap, the stored price is lower than the true backing ratio.

**Exploit path:**

1. Staking rewards accrue between oracle updates. True rsETH/ETH rate rises from `P_old` to `P_true`, but `rsETHPrice` still holds `P_old < P_true`.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, D, 0, "")`.
3. `getRsETHAmountToMint` computes `rsethMinted = D × assetPrice / P_old`. Because `P_old < P_true`, `rsethMinted > D × assetPrice / P_true` — the attacker receives more rsETH than their deposit is worth at the true rate.
4. The bot (or the attacker themselves, since `updateRSETHPrice()` is public) calls `updateRSETHPrice()`. The new price is `totalETHInProtocol / rsethSupply`. Because the attacker's deposit increased `rsethSupply` by more than it increased `totalETHInProtocol` (relative to the true rate), the updated price is slightly lower than `P_true` — existing holders' rsETH is worth less.
5. Attacker initiates withdrawal and recovers more ETH-equivalent than deposited. The difference is yield extracted from existing holders.

**Why the `pricePercentageLimit` guard does not prevent this:**

When the price drift exceeds `pricePercentageLimit`, `updateRSETHPrice()` reverts for non-managers (`PriceAboveDailyThreshold`). This does not protect depositors — it extends the staleness window, giving the attacker more time to deposit at the stale price. The attacker does not need to call `updateRSETHPrice()` themselves; the bot will eventually do so.

## Impact Explanation

**High — Theft of unclaimed yield.**

Every deposit made while `rsETHPrice` is stale mints rsETH at a discount relative to the true backing ratio. When the price is subsequently updated, the attacker's rsETH is worth more than what they deposited, and the shortfall is borne by all existing rsETH holders whose share of accumulated staking rewards is diluted. The attack is repeatable across every oracle update interval and scales with deposit size and staleness duration.

## Likelihood Explanation

**Medium.** `updateRSETHPrice()` is driven by off-chain bots on a periodic schedule (typically every few hours to once per day). Any of the following realistic conditions creates an exploitable window:

- Normal bot scheduling gap (hours of staleness during which stETH/ETH rebases or EigenLayer rewards accrue).
- Bot downtime or network congestion.
- `pricePercentageLimit` causing `updateRSETHPrice()` to revert for non-managers, extending the staleness window.

No privileged role is required. `depositAsset()` is fully permissionless.

## Recommendation

Call `updateRSETHPrice()` (or an internal equivalent that skips the `pricePercentageLimit` revert for the protocol itself) at the beginning of `depositAsset()`, `depositETH()`, `initiateWithdrawal()`, and `instantWithdrawal()` before any share/asset calculation. This ensures the price is always fresh at the moment of minting or redemption, eliminating the staleness window entirely.

## Proof of Concept

**Setup (fork mainnet or local simulation):**

- Protocol has `rsethSupply = 1,000,000e18`, `rsETHPrice = 1.010e18` (last updated 24 h ago).
- stETH has rebased; true backing is now `1.011e18` per rsETH.
- `updateRSETHPrice()` has not been called yet.

**Steps:**

1. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
2. `getRsETHAmountToMint` returns `100e18 × 1e18 / 1.010e18 = 99.0099e18` rsETH (stale price).
3. Fair amount at true price: `100e18 × 1e18 / 1.011e18 = 98.9120e18` rsETH.
4. Attacker receives `≈ 0.0979e18` excess rsETH, funded by diluting existing holders.
5. Bot (or attacker) calls `updateRSETHPrice()`. New price: `(1,011,000e18 + 100e18) / (1,000,000e18 + 99.0099e18) ≈ 1.011e18` (existing holders' share of the 1 ETH of accrued rewards is now diluted by the attacker's extra rsETH).
6. Attacker calls `initiateWithdrawal(stETH, 99.0099e18, "")`. `getExpectedAssetAmount` returns `99.0099e18 × 1.011e18 / 1e18 ≈ 100.099e18` stETH — more than the 100 stETH deposited.
7. Net profit: `≈ 0.099 stETH` per 100 stETH deposited, sourced from existing holders' unclaimed yield.

**Foundry test plan:** Fork mainnet, warp time forward by 24 h without calling `updateRSETHPrice()`, execute the deposit, call `updateRSETHPrice()`, initiate withdrawal, assert attacker's recovered asset amount exceeds deposit amount and that existing holder rsETH value decreased relative to a baseline run where `updateRSETHPrice()` was called before the deposit.
