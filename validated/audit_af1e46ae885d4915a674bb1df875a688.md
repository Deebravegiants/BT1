### Title
Dijkstra Era Minimum Fee Calculation Omits Subtransaction Reference Scripts, Enabling Underpriced Deserialization Work - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the minimum fee enforced on a top-level transaction is computed using only the reference scripts reachable from the top-level transaction's own inputs. Reference scripts attached to subtransaction inputs are silently excluded from the fee calculation. Because the fee field exists only on `TopTx` bodies and subtransactions carry no independent fee, an unprivileged sender can embed arbitrarily large reference scripts inside subtransactions and pay only the base size-linear fee, imposing deserialization work on every validating node without the corresponding economic cost.

---

### Finding Description

The Cardano Ledger introduced tiered reference-script pricing in Conway to prevent the DDoS attack that occurred on 25 June 2024 (documented in `docs/adr/2024-08-14_009-refscripts-fee-change.md`). The fix charges a fee proportional to the total serialised size of all reference scripts touched by a transaction, computed by `txNonDistinctRefScriptsSize`.

The Dijkstra era adds *nested transactions* (`subTransactionsTxBodyL`). Each subtransaction is a `DijkstraSubTx` whose body type is `TxBody SubTx era`; the `feeTxBodyL` lens is restricted to `TxBody TopTx era`, so subtransactions carry no fee field and pay no fee independently.

The fee validation path for a Dijkstra top-level transaction is:

```
dijkstraUtxoTransition
  → Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
    → getMinFeeTxUtxo pp tx originalUtxo          -- EraUTxO DijkstraEra
      → getConwayMinFeeTxUtxo pparams tx utxo      -- line 141, Dijkstra/UTxO.hs
        → getMinFeeTx pparams tx
            (txNonDistinctRefScriptsSize utxo tx)  -- line 175, Conway/UTxO.hs
```

`txNonDistinctRefScriptsSize` unions only the top-level transaction's `referenceInputsTxBodyL` and `inputsTxBodyL`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs  lines 183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

Subtransaction inputs are never consulted. The codebase already contains the correct aggregate function:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs  lines 264-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

but `batchNonDistinctRefScriptsSize` is **not** used in the fee enforcement path. The `EraUTxO DijkstraEra` instance wires `getMinFeeTxUtxo = getConwayMinFeeTxUtxo`, which calls the single-transaction variant.

---

### Impact Explanation

Every validating node must deserialise every reference script reachable from every input of every subtransaction before it can evaluate phase-2 scripts. The deserialization cost is the exact overhead that motivated the June 2024 emergency patch. By placing large Plutus reference scripts exclusively in subtransaction inputs, an attacker pays only the base `a + b·txSize` fee while forcing nodes to deserialise up to `maxRefScriptSizePerTx` bytes of scripts per subtransaction, multiplied by the number of subtransactions permitted per top-level transaction.

This matches the allowed Medium impact: *"Attacker-controlled transactions… modify fees… outside design parameters."* The fee actually charged is strictly less than the fee the protocol intends to charge for the deserialization work performed, because the reference-script fee component is computed over an incomplete set of inputs.

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a `DijkstraTx TopTx` containing one or more `DijkstraSubTx` subtransactions whose `referenceInputsTxBodyL` point to UTxO entries holding large Plutus scripts. No special role, key, or governance action is required. The attack is reachable from the public mempool as soon as the Dijkstra era is active on mainnet.

---

### Recommendation

Replace the call to `txNonDistinctRefScriptsSize` inside `getConwayMinFeeTxUtxo` (or override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance) with `batchNonDistinctRefScriptsSize` so that the fee covers reference scripts from all subtransactions. Concretely, the `EraUTxO DijkstraEra` instance in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` should define:

```haskell
getMinFeeTxUtxo pp tx utxo =
  getMinFeeTx pp tx (batchNonDistinctRefScriptsSize utxo tx)
```

The `validateAllRefScriptSize` size-limit check should similarly be verified to use `batchNonDistinctRefScriptsSize` rather than `txNonDistinctRefScriptsSize`.

---

### Proof of Concept

1. Attacker creates a UTxO entry whose output contains a large Plutus V3 reference script (e.g., 50 KiB, well within per-tx limits).
2. Attacker constructs a `DijkstraTx TopTx` with N subtransactions, each referencing that UTxO via `referenceInputsTxBodyL`. The top-level transaction body has no reference inputs.
3. `getConwayMinFeeTxUtxo` computes `txNonDistinctRefScriptsSize` over the top-level inputs only → result is 0 bytes → `tierRefScriptFee` returns `Coin 0`.
4. The minimum fee enforced is `alonzoMinFeeTx pp tx` (base size-linear fee only), regardless of how many subtransaction reference scripts must be deserialised.
5. Every honest node deserialises N × 50 KiB of Plutus scripts while the attacker pays only the base fee, achieving the same economic asymmetry as the June 2024 attack.

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
