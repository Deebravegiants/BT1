### Title
Immutable `FACTORY` as Protocol Fee Recipient Causes Permanent Fee Lock on USDC/USDT Pools — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary
`FACTORY` is stored as an `immutable` address in `MetricOmmPool` and is the hardcoded recipient of all protocol fees in `collectFees`. If the factory address is blacklisted by USDC or USDT, every call to `collectFees` reverts, permanently locking accrued protocol fees inside the pool with no recovery path.

### Finding Description
In `MetricOmmPool`, the factory address is set once at construction and stored as an EVM immutable: [1](#0-0) [2](#0-1) 

Inside `collectFees`, after computing the protocol share of both spread and notional fees, the function unconditionally transfers them to `FACTORY`: [3](#0-2) 

The state reset (`notionalFeeToken0Scaled = 0`) happens **after** both transfers: [4](#0-3) 

If USDC's `transfer` reverts because `FACTORY` is blacklisted, the entire `collectFees` call reverts atomically — the fee accounting is never cleared, and the tokens remain in the pool with no alternative withdrawal path.

There is no mechanism to change `FACTORY` on a deployed pool. The factory's own `collectTokens` function only rescues tokens **held by the factory itself**, not tokens held by the pool: [5](#0-4) 

### Impact Explanation
Protocol fees denominated in USDC or USDT accumulate in the pool as `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` and as spread surplus. Once `FACTORY` is blacklisted:

1. `collectPoolFees` (permissionless) reverts — fees cannot be swept.
2. `setPoolProtocolFee` (owner) reverts — it calls `collectFees` before updating rates: [6](#0-5) 
3. `setPoolAdminFees` (pool admin) reverts for the same reason: [7](#0-6) 

All three paths are permanently bricked. Protocol fees are stuck in the pool forever, and fee configuration for the affected pool cannot be updated.

### Likelihood Explanation
USDC and USDT both implement address-level blacklists. The `MetricOmmPoolFactory` is a smart contract that could be blacklisted if it is associated with sanctioned activity (e.g., if the factory owner's key is compromised and used for illicit transfers, or if a regulatory action targets the protocol's treasury contract). The contest scope explicitly includes USDC/USDT blacklisting behavior as in-scope. The pattern is identical to the Sparkn `STADIUM_ADDRESS` finding: a single immutable fee-recipient address with no update path.

### Recommendation
Introduce a mutable `protocolFeeRecipient` address on the factory (updatable by the owner), and pass it into `collectFees` as a parameter (analogous to how `adminFeeDestination_` is already passed). This mirrors the existing design for the admin leg and eliminates the single point of failure:

```solidity
// In MetricOmmPoolFactory
address public protocolFeeRecipient;

function setProtocolFeeRecipient(address newRecipient) external onlyOwner {
    require(newRecipient != address(0));
    protocolFeeRecipient = newRecipient;
}

// Pass to collectFees alongside adminFeeDestination
pool.collectFees(..., poolAdminFeeDestination[pool], protocolFeeRecipient);
```

On the pool side, replace the hardcoded `FACTORY` transfer with the passed-in recipient parameter.

### Proof of Concept

1. A USDC/USDT pool is deployed; `FACTORY` is baked in as immutable.
2. Swaps accrue protocol fees: `notionalFeeToken0Scaled > 0` and spread surplus grows.
3. USDC blacklists the `MetricOmmPoolFactory` address (e.g., due to a regulatory action or key compromise).
4. Anyone calls `collectPoolFees(pool)` → factory calls `pool.collectFees(...)` → pool executes `transferToken0(FACTORY, totalFee0ToProtocol)` → USDC's `transfer` reverts with `Blacklistable: account is blacklisted` → entire tx reverts.
5. `notionalFeeToken0Scaled` is never cleared (the reset at lines 429–430 is never reached).
6. Owner attempts `setPoolProtocolFee(pool, 0, 0)` to zero out fees and bypass collection — same revert, because `collectFees` is called first.
7. Pool admin attempts `setPoolAdminFees(pool, 0, 0)` — same revert.
8. Protocol fees remain permanently locked in the pool. No rescue function exists on the pool, and `collectTokens` on the factory cannot reach pool balances. [8](#0-7)
