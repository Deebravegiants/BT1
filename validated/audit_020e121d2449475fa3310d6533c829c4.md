Audit Report

## Title
Stale rsETH/ETH Rate in `CrossChainRateReceiver.getRate()` Allows Depositors to Mint Excess wrsETH on L2 — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver` stores both `rate` and `lastUpdated` when a LayerZero message arrives from L1, but `getRate()` returns the stored rate with no staleness check. All L2 pool variants compute the wrsETH mint amount by dividing the deposit by this unchecked rate. When the L1 rsETH/ETH rate has risen since the last LayerZero update, any depositor can exploit the gap to receive more wrsETH than their deposit entitles them to, extracting value from existing holders' unclaimed staking yield.

## Finding Description

`CrossChainRateReceiver` records both the rate and its update timestamp:

```solidity
uint256 public rate;        // L13
uint256 public lastUpdated; // L16
```

`lzReceive` sets both fields on arrival of a LayerZero message from the L1 rate provider (`contracts/cross-chain/CrossChainRateReceiver.sol`, lines 82–100). However, `getRate()` ignores `lastUpdated` entirely:

```solidity
function getRate() external view returns (uint256) {
    return rate; // L103-105 — no staleness check
}
```

`RSETHPoolV3.getRate()` delegates directly to this oracle (`contracts/pools/RSETHPoolV3.sol`, lines 234–237), and `viewSwapRsETHAmountAndFee` uses it to compute the mint amount:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate; // L304-307
```

A lower (stale) `rsETHToETHrate` produces a proportionally larger `rsETHAmount`. The `updateRate()` function on `CrossChainRateProvider` is permissionless (`contracts/cross-chain/CrossChainRateProvider.sol`, lines 85–101) — anyone can push the current L1 rate to L2 — but there is no requirement to do so before a deposit executes. The deposit path (`RSETHPoolV3.deposit`, lines 246–265) has no staleness guard; the only checks are `nonReentrant`, `whenNotPaused`, and `limitDailyMint`.

## Impact Explanation

**High — Theft of unclaimed yield.** When staking rewards accrue on L1 and the L2 rate has not been refreshed, a depositor receives wrsETH in excess of their fair entitlement. The surplus wrsETH represents the yield that should belong to existing holders: their proportional claim on the underlying ETH pool is diluted. This is a direct, repeatable extraction of unclaimed yield from existing rsETH/wrsETH holders by any unprivileged depositor.

## Likelihood Explanation

- No special role or privilege is required; any depositor can trigger this.
- LayerZero message delivery is not instantaneous and is not enforced before deposits; the rate update window is a normal operational condition, not an edge case.
- The attacker only needs to observe the L1 rsETH price (publicly available on-chain) and compare it to the stale L2 rate before depositing.
- The attack is repeatable every time staking rewards accrue and before the next `updateRate()` call.
- The `dailyMintLimit` modifier caps per-day exposure but does not prevent the attack; it only bounds the magnitude per day.

## Recommendation

Introduce a configurable `maxAge` parameter in `CrossChainRateReceiver` and enforce it in `getRate()`:

```solidity
uint256 public maxAge; // e.g. 24 hours, set by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxAge, "Rate is stale");
    return rate;
}
```

This ensures deposits revert when the cross-chain rate has not been refreshed within an acceptable window, preventing exploitation of the staleness gap.

## Proof of Concept

1. Deploy a fork of the L2 chain with `CrossChainRateReceiver.rate = 1e18` and `lastUpdated = T`.
2. Advance time by 24+ hours without calling `updateRate()` (simulating delayed LayerZero delivery or no one triggering the update).
3. On L1, rsETH/ETH rate has risen to `1.05e18` due to staking rewards.
4. Call `RSETHPoolV3.deposit{value: 1 ether}("")` from an attacker EOA.
5. `getRate()` returns the stale `1e18`; attacker receives `1e18 * 1e18 / 1e18 = 1e18` wrsETH (1 wrsETH) instead of the correct `~0.952e18`.
6. Call `CrossChainRateProvider.updateRate{value: fee}()` to push the updated rate to L2.
7. `CrossChainRateReceiver.rate` is now `1.05e18`; attacker's 1 wrsETH is redeemable for 1.05 ETH worth of assets, having paid only 1 ETH — a 5% gain extracted from existing holders' unclaimed yield.

A Foundry fork test can reproduce this by: (a) forking L2 at a block where `lastUpdated` is stale, (b) recording the attacker's wrsETH balance after deposit, (c) simulating the `lzReceive` call with the updated rate, and (d) asserting the attacker's wrsETH value exceeds their ETH deposit.
