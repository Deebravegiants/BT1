### Title
`respond_verify_foreign_tx()` Does Not Bind `response.payload_hash` to the Submitted `request`, Enabling a Byzantine Participant to Forge Foreign-Chain Verification Responses - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx()` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the domain's root public key, but it never checks that `response.payload_hash` is the canonical hash of the `request` being answered. A single attested-but-Byzantine participant can replay a valid `(payload_hash, signature)` pair from any previously completed `respond_verify_foreign_tx` call to drain the yield queue of any currently-pending `verify_foreign_transaction` request, delivering a forged response to every waiting caller.

---

### Finding Description

`respond_verify_foreign_tx()` performs two checks before draining the pending yield queue:

1. The caller is an attested participant and the protocol is active.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. [1](#0-0) 

What it does **not** check is that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))` for the specific `request` argument passed to the call. The `ForeignTxSignPayload` structure binds the hash to the original `ForeignChainRpcRequest` (tx_id, extractors, finality, etc.): [2](#0-1) 

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) does enforce this binding by recomputing the expected hash and comparing it to `response.payload_hash`: [3](#0-2) 

But the on-chain contract does not. Once `signature_is_valid` is `true`, `resolve_yields_for` is called unconditionally, draining every queued yield for the supplied `request` with whatever `response` bytes were provided: [4](#0-3) 

`resolve_yields_for` removes the map entry and resumes all parked yields in one pass; there is no second chance to deliver the correct response: [5](#0-4) 

---

### Impact Explanation

A Byzantine participant (a single attested node, strictly below the signing threshold) can:

1. Read any previously-confirmed `respond_verify_foreign_tx` transaction from the NEAR blockchain to obtain a valid `(payload_hash_A, sig_A)` pair for a completed request A.
2. Wait for a new `verify_foreign_transaction(request_B)` to appear in the pending map.
3. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
4. The contract accepts the call because `sig_A` is a valid root-key signature over `payload_hash_A`, regardless of whether `payload_hash_A` has any relation to `request_B`.
5. All yields queued under `request_B` are drained with the forged `(payload_hash_A, sig_A)` response; the honest nodes can no longer deliver the correct response.

Every caller of `request_B` receives a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to the transaction they asked about. Callers that skip the SDK-level binding check (i.e., do not call `ForeignChainSignatureVerifier::verify_signature`) will accept the forged attestation as proof that a foreign-chain transaction occurred, enabling invalid bridge execution or double-spend conditions. Even callers that do verify will find their request permanently poisoned and must resubmit — only to have the Byzantine participant repeat the attack.

**Impact category**: High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.

---

### Likelihood Explanation

- The attacker needs only one attested participant role (below threshold), which is a realistic Byzantine assumption explicitly in scope.
- The required `(payload_hash, signature)` material is fully public on-chain from any prior `respond_verify_foreign_tx` call.
- No cryptographic capability beyond reading chain history and submitting a transaction is required.
- The attack can be repeated for every new submission of the targeted request, making it persistent.

---

### Recommendation

Inside `respond_verify_foreign_tx`, after verifying the signature, recompute the expected payload hash from the `request` argument and assert it matches `response.payload_hash`:

```rust
// After signature_is_valid check, before resolve_yields_for:
let expected_payload_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    // The contract cannot know extracted_values, so the hash must be
    // included in the request or the nodes must commit to it on-chain.
    ...
});
```

Because the contract does not store the `extracted_values` that the nodes observed, the cleanest fix is to include the `payload_hash` commitment in the `VerifyForeignTransactionRequest` stored on-chain (set by the MPC nodes when they call `respond_verify_foreign_tx`), or to require the responding node to supply the full `ForeignTxSignPayload` so the contract can recompute and verify the hash itself. At minimum, the contract should reject any `respond_verify_foreign_tx` call where `response.payload_hash` does not commit to the `request.request` field, mirroring the binding check already present in the SDK verifier. [3](#0-2)

### Proof of Concept

1. Honest nodes process `verify_foreign_transaction(request_A)` and call `respond_verify_foreign_tx(request_A, { payload_hash_A, sig_A })`. This is recorded on-chain.
2. A user submits `verify_foreign_transaction(request_B)` (different `tx_id`). The contract queues a yield under `request_B`.
3. Byzantine participant reads `(payload_hash_A, sig_A)` from the chain and immediately calls:
   ```
   respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })
   ```
4. The contract at lines 718–734 verifies `sig_A` over `payload_hash_A` against the root key → `signature_is_valid = true`.
5. `resolve_yields_for` at line 749 removes `request_B` from `pending_verify_foreign_tx_requests` and resumes all yields with the forged response bytes.
6. The user's `verify_foreign_transaction(request_B)` call returns `{ payload_hash: payload_hash_A, signature: sig_A }` — a response that attests to transaction A, not transaction B.
7. Any downstream contract that does not call `ForeignChainSignatureVerifier::verify_signature` will treat this as valid proof that transaction B occurred. [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
