### Title
Negative `RequireMOf` Threshold Bypasses Script Authorization in Shelley Native Scripts — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Scripts.hs`)

---

### Summary

The `MultiSigMOf` constructor stores its threshold as `!Int`, the CBOR decoder for tag `3` calls `decCBOR :: Decoder s Int` with no non-negativity guard, and the CDDL spec at `shelley.cddl:284` explicitly types the field as `int32` (range `−2147483648 .. 2147483647`). Any negative threshold causes `isValidMOf` to return `True` immediately regardless of the witness set, allowing a transaction with zero vkey witnesses to satisfy a script-locked credential.

---

### Finding Description

**Root cause — CBOR decoder accepts negative threshold:** [1](#0-0) 

`MultiSigMOf !Int` stores a signed integer. The decoder: [2](#0-1) 

calls `m <- decCBOR` which resolves to `decodeInt` — a plain CBOR integer decoder that accepts CBOR major-type-1 (negative) values with no bounds check. The CDDL spec confirms this is within the accepted wire format: [3](#0-2) 

**Root cause — evaluator short-circuits on any `n ≤ 0`:** [4](#0-3) 

`isValidMOf n StrictSeq.Empty = n <= 0` returns `True` for any `n < 0`. The non-empty branch `n <= 0 || ...` also short-circuits immediately. A script `RequireMOf (-1) []` (or any negative threshold with any sub-script list) therefore evaluates to `True` with an empty witness set.

**Validation path — no guard exists in Shelley UTXOW:**

`validateFailedNativeScripts` only checks whether `validateNativeScript` returns `True` or `False`; it does not inspect the threshold value: [5](#0-4) 

`validateScriptsWellFormed` (which calls `validScript`) was introduced only in Babbage and is absent from the Shelley UTXOW transition: [6](#0-5) 

The same `isValidMOf` pattern is replicated verbatim in `evalTimelock` for Allegra and later eras: [7](#0-6) 

**Formal spec mismatch:** The formal spec types the threshold as `N` (natural numbers), not integers: [8](#0-7) 

The CDDL diverges from the formal spec by using `int32`.

---

### Impact Explanation

An attacker who controls a script-locked stake credential (registered with a `RequireMOf (negative) [...]` script hash) can:

1. **Withdraw rewards** from that credential with zero vkey witnesses — `validateMultiSig` calls `evalMultiSig` with the empty witness set extracted from `addrTxWitsL`, which returns `True`.
2. **Spend UTxOs** locked by such a script address with zero witnesses.
3. **Submit delegation certificates** (`ShelleyDelegCert`) for such a credential with zero witnesses.

If a victim's tooling generates or accepts a script with a negative threshold (e.g., due to a serialization bug or attacker-supplied script), the attacker can drain rewards or spend locked funds without any cryptographic authorization. The impact maps to **unauthorized withdrawal from a script-locked credential** and **unauthorized delegation**, satisfying the High/Medium impact tiers.

---

### Likelihood Explanation

- The CDDL spec explicitly permits `int32`, so any conforming CBOR encoder can produce a negative threshold without violating the wire format.
- No node-side rejection exists at deserialization or at UTXOW validation time in Shelley (and Allegra/Mary eras).
- The attack is self-contained: the attacker crafts a CBOR blob, derives the script hash, registers a stake credential, and withdraws rewards — no privileged access required.
- Likelihood is **medium**: requires the attacker to either use the script themselves (limited self-impact) or trick a victim into using a negative-threshold script as their credential.

---

### Recommendation

1. **Add a non-negativity guard in the CBOR decoder** for tag `3`:
   ```haskell
   3 -> do
     m <- decCBOR
     when (m < 0) $ fail "RequireMOf threshold must be non-negative"
     ...
   ``` [2](#0-1) 

2. **Change the CDDL type** from `int32` to `uint` for the `script_n_of_k` threshold field to align with the formal spec. [3](#0-2) 

3. **Apply the same fix to `evalTimelock`** in Allegra and all subsequent eras. [7](#0-6)
