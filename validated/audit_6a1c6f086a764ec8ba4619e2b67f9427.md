### Title
Sub-Transaction Witness Check Uses Stale UTxO, Bypassing Key Authorization for Outputs Created by the Top-Level Transaction — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxow.hs`)

---

### Summary

In the Dijkstra era's `SUBUTXOW` rule, the required-witness check (`validateNeededWitnesses`) is called with `originalUtxo` — the UTxO snapshot from *before* the enclosing top-level transaction executes — rather than the live UTxO state that already contains outputs created by that top-level transaction. Because `getWitsVKeyNeeded` only demands a key witness for inputs it can find in the supplied UTxO, any sub-transaction input that was created by the top-level transaction is invisible to the check and requires **no witness at all**, regardless of the payment credential locking it.

---

### Finding Description

The Dijkstra era introduces nested transactions (`TopTx` / `SubTx`). The `SUBLEDGER` rule processes a sub-transaction in two phases:

1. **`SUBENTITIES` / `SUBGOV`** — certificate and governance processing, using the live `certState`.
2. **`SUBUTXOW`** — UTxO witness validation, then **`SUBUTXO`** — UTxO state update.

The environment passed to `SUBUTXOW` is constructed in `dijkstraSubLedgersTransition`:

```haskell
-- SubLedger.hs lines 287-293
utxoStateAfterSubUtxow <-
  trans @(EraRule "SUBUTXOW" era) $
    TRC
      ( SubUtxoEnv slot pp certState originalUtxo topIsValid
      , utxoStateBeforeSubUtxow   -- ← live UTxO (contains top-tx outputs)
      , stAnnTx
      )
```

`utxoStateBeforeSubUtxow` is the live UTxO state (which already reflects outputs created by the top-level transaction), but `originalUtxo` is the UTxO *before* the top-level transaction ran. [1](#0-0) 

Inside `dijkstraSubUtxowTransition`, the witness check is:

```haskell
-- SubUtxow.hs lines 269-270
{- vKeyHashesNeeded ⊆ vKeyHashesProvided -}
runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody
``` [2](#0-1) 

`validateNeededWitnesses` calls `getWitsVKeyNeeded`, which looks up each input in the supplied UTxO to find the payment key hash that must be witnessed. [3](#0-2) 

If an input is absent from the supplied UTxO, `getWitsVKeyNeeded` simply does not add its payment key to the "needed" set. Because `originalUtxo` does not contain outputs created by the top-level transaction, any sub-transaction input that was created by the top-level transaction is silently excluded from the witness requirement. The `SUBUTXO` rule (which uses the live UTxO state) will still confirm the input exists and consume it — but no key witness is ever demanded.

---

### Impact Explanation

**Critical — Direct loss of ADA or native assets through an invalid ledger state transition.**

A top-level transaction author can:

1. Create one or more UTxO outputs locked by arbitrary payment key credentials (e.g., a victim's key hash) within the top-level transaction body.
2. Include a sub-transaction that spends those outputs.
3. Because those outputs are absent from `originalUtxo`, `validateNeededWitnesses` requires no witness for them.
4. The sub-transaction is accepted and the outputs are consumed without the credential owner's authorization.

Any ADA or native assets placed in those outputs are permanently transferred to the attacker-controlled destination in the sub-transaction, with no recourse.

---

### Likelihood Explanation

**Medium-to-High.** The Dijkstra era is the only era with nested transactions; the attack surface is new and the rule is production code. The attacker needs only to author a valid top-level transaction — an unprivileged capability available to any network participant. No privileged role, key leak, or consensus majority is required. The only constraint is that the attacker must fund the top-level transaction fees, which is trivially achievable.

---

### Recommendation

Replace `originalUtxo` with the live UTxO state in the `validateNeededWitnesses` call inside `dijkstraSubUtxowTransition`. The live UTxO is already available as the UTxO component of `utxoState` (the `State` of `SUBUTXOW`):

```haskell
-- Replace:
runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody

-- With:
let liveUtxo = utxoState ^. utxosUtxoL
runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState liveUtxo txBody
```

This ensures that outputs created by the top-level transaction are visible to the witness check and their payment credentials are properly enforced.

### Proof of Concept

**Setup:**
- `originalUtxo` contains one input `i₀` locked by attacker key `Kₐ`.
- Top-level transaction spends `i₀`, creates output `o₁` locked by victim key `Kᵥ` (value = 100 ADA), and creates output `o₂` to attacker (change).
- Sub-transaction spends `o₁` and sends 100 ADA to attacker address.

**Witness check trace in `SUBUTXOW`:**
1. `witsKeyHashes` = `{Kₐ}` (only the top-level input witness is provided).
2. `getWitsVKeyNeeded originalUtxo subTxBody`:
   - Looks up `o₁` in `originalUtxo` → **not found** (it was created by the top-level tx).
   - Returns `{}` (empty needed set).
3. `missingWitnesses = {} \ {Kₐ} = {}` → **check passes**.
4. `SUBUTXO` looks up `o₁` in live UTxO → **found** → consumes it.
5. 100 ADA transferred to attacker without `Kᵥ` ever signing anything.

The root cause is the mismatch between the UTxO used for existence checks (`utxoState`, live) and the UTxO used for witness checks (`originalUtxo`, stale) in `dijkstraSubUtxowTransition`. [4](#0-3)

dijkstraSubUtxowTransition = do
  TRC (env@(SubUtxoEnv _ pp certState originalUtxo _), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG
      txBody = tx ^. bodyTxL
      witsKeyHashes = keyHashWitnessesTxWits (tx ^. witsTxL)
      scriptsProvided = scriptsProvidedStAnnTx stAnnTx

  {- ∀[ (vk , σ) ∈ vKeySigs ] isSigned vk (txidBytes txId) σ -}
  runTestOnSignal $ Shelley.validateVerifiedWits tx

  let scriptsNeeded = scriptsNeededStAnnTx stAnnTx
      scriptHashesNeeded = getScriptsHashesNeeded scriptsNeeded

  {- ∀[ s ∈ p1ScriptsNeeded ] validP1Script vKeyHashesProvided txVldt s -}
  runTest $ Babbage.validateFailedBabbageScripts tx scriptsProvided scriptHashesNeeded

  {- vKeyHashesNeeded ⊆ vKeyHashesProvided -}
  runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody

  {- dataHashesNeeded ⊆ mapˢ hash dataProvided -}
  runTest $ Alonzo.missingRequiredDatums scriptsProvided originalUtxo tx

  {- txADhash ≡ map hash txAuxData -}
  runTestOnSignal $ Shelley.validateMetadata pp tx

  let scriptIntegrity = mkScriptIntegrity pp tx (plutusLanguagesUsedStAnnTx stAnnTx)
  runTest $ Alonzo.checkScriptIntegrityHash tx pp scriptIntegrity

  runTest $ Alonzo.hasExactSetOfRedeemers tx scriptsProvided scriptsNeeded

  runTest $
    Babbage.validateScriptsWellFormedTxOuts
      pp
      (tx ^. witsTxL . scriptTxWitsL)
      (tx ^. bodyTxL . outputsTxBodyL)

  runTest $ validateGuardDatums scriptsProvided txBody

  trans @(EraRule "SUBUTXO" era) $ TRC (env, utxoState, stAnnTx)

validateNeededWitnesses ::
  EraUTxO era =>
  -- | Provided witness
  Set (KeyHash Witness) ->
  CertState era ->
  UTxO era ->
  TxBody t era ->
  Test (ShelleyUtxowPredFailure era)
validateNeededWitnesses witsKeyHashes certState utxo txBody =
  let needed = getWitsVKeyNeeded certState utxo txBody
      missingWitnesses = Set.difference needed witsKeyHashes
   in failureOnNonEmptySet missingWitnesses MissingVKeyWitnessesUTXOW
