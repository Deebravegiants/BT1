### Title
Attached NEAR Deposit Permanently Locked on Existing-Participant Re-Submission in `submit_participant_info` - (File: `crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is a `#[payable]` function that conditionally checks and refunds the attached deposit. When an **existing participant** re-submits their attestation (the normal periodic renewal path), the deposit-handling block is entirely skipped, causing any attached NEAR tokens to be permanently locked in the contract with no withdrawal mechanism.

---

### Finding Description

`submit_participant_info` is marked `#[payable]` and accepts arbitrary NEAR deposits. After inserting the attestation, the function computes two boolean flags:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... deposit check and refund logic
}
``` [1](#0-0) 

When the caller **is** a current participant (`caller_is_not_participant = false`) **and** the attestation is a re-submission rather than a first-time insertion (`is_new_attestation = false`), the entire `if` block is skipped. No deposit check is performed and no refund is issued. Any NEAR tokens attached to the call are silently absorbed by the contract and permanently locked, because the contract has no withdrawal mechanism.

This is the normal operational path: MPC nodes periodically re-submit their TEE attestations on a 1-hour cadence. [2](#0-1) 

By contrast, every other payable entry point in the contract (`sign`, `request_app_private_key`, `verify_foreign_transaction`, `propose_update`) unconditionally refunds excess deposits. [3](#0-2) 

---

### Impact Explanation

Any NEAR tokens attached to a re-submission call by an existing participant are permanently locked in the MPC contract. The contract has no admin withdrawal function. This breaks the balance/accounting safety invariant: funds that should be returned to the caller are instead silently retained. This matches the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The periodic re-submission path (`periodic_attestation_submission`, running every hour per node) is the primary production caller of `submit_participant_info`. [4](#0-3) 

Any node operator who attaches a non-zero deposit to a re-submission call — whether by misconfiguration, a tooling default, or a front-end that mirrors the first-submission deposit — will have those funds locked. Because the function is `#[payable]` with no upper-bound enforcement on the silent path, the loss scales with whatever is attached. No privileged access or collusion is required; the caller is the participant themselves.

---

### Recommendation

1. **Unconditionally refund excess deposits** regardless of whether the submission is new or a re-submission. The simplest fix is to move the refund logic outside the `if attestation_storage_must_be_paid_by_caller` block, or to call `require_deposit(NearToken::from_yoctonear(0), &account_id)` (which already handles refunds) unconditionally.

2. **Reject non-zero deposits on the re-submission path** by adding an explicit check that reverts if `attached_deposit > 0` when `!attestation_storage_must_be_paid_by_caller`, mirroring the pattern used in `require_deposit`. [5](#0-4) 

---

### Proof of Concept

1. Participant `alice.near` is already a current protocol participant with a stored attestation.
2. `alice.near` calls `submit_participant_info` with a valid re-submission attestation and attaches `1 NEAR` (e.g., copied from the first-submission call).
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` → `is_new_attestation = false`.
4. `self.voter_account()` succeeds → `caller_is_not_participant = false`.
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The `if` block is skipped entirely; no refund promise is created.
7. The call succeeds with `Ok(())`. `alice.near`'s balance is reduced by `1 NEAR`. The contract balance increases by `1 NEAR`. No mechanism exists to recover these funds. [6](#0-5)
