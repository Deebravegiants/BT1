### Title
Wrong Chain-Availability Function Used in `verify_foreign_transaction` Gate — (`File: crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` gates incoming requests with `get_supported_foreign_chains()` (the legacy all-participants intersection rule, no RPC-whitelist check) instead of `get_available_foreign_chains()` (the new threshold-based rule that also requires the chain to be in the on-chain RPC whitelist). This is a direct analog of the Vault.sol copy-paste bug: the wrong function is used in the conditional check, producing incorrect request-lifecycle outcomes for every caller.

### Finding Description

`verify_foreign_transaction` at line 534 calls `self.get_supported_foreign_chains()`:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();   // ← wrong function
if !supported_chains.contains(&requested_chain) {
    env::panic_str(&InvalidParameters::ForeignChainNotSupported { … }.to_string());
}
```

`get_supported_foreign_chains()` (lines 2176–2218) computes the **strict intersection** of every active participant's old-style registration (`node_foreign_chain_support.foreign_chain_support_by_node`) and performs **no RPC-whitelist check**:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
```

The design document (`docs/design/calculating-supported-foreign-chains.md`) and the contract's own doc-comment on `get_available_foreign_chains` (line 2224–2228) both state that `verify_foreign_transaction` **must gate on `get_available_foreign_chains()`** — the cached set that requires (a) the chain to be voted into the on-chain RPC whitelist and (b) at least `signing_threshold` active participants to cover it:

```rust
pub fn get_available_foreign_chains(&self) -> dtos::AvailableForeignChains {
    self.foreign_chains.get().available_foreign_chains.clone()
}
```

The two functions are semantically different in two critical ways:

| Property | `get_supported_foreign_chains()` | `get_available_foreign_chains()` |
|---|---|---|
| Coverage requirement | **All** active participants | **≥ signing_threshold** participants |
| RPC whitelist check | **No** | **Yes** |

### Impact Explanation

**Path A — Non-whitelisted chain accepted (safety invariant broken):** A chain registered by all participants via the old `register_foreign_chain_config` path but **not** voted into the RPC whitelist passes the contract gate. The contract enqueues the yield and charges the user's deposit. MPC nodes attempt to process the request; because no whitelisted providers exist for the chain, every node fails at the inspector level. The request times out, the user's deposit is consumed, and no signature is produced. The contract's own stated invariant — "only accept requests the network can fulfill" — is violated.

**Path B — Whitelisted, threshold-covered chain rejected (liveness broken):** A chain that is whitelisted and covered by `signing_threshold` participants (i.e., `get_available_foreign_chains()` would return it) is rejected if even one active participant has not yet registered via the old system. Any user calling `verify_foreign_transaction` for that chain receives `ForeignChainNotSupported` even though the network is fully capable of servicing the request.

Both paths break the request-lifecycle and accounting invariants described in the design document without requiring any privileged access.

### Likelihood Explanation

The entry point is the public, payable `verify_foreign_transaction` method callable by any NEAR account. No special role or threshold collusion is required. The discrepancy is present in the current production code path and is triggered on every call to `verify_foreign_transaction` for any chain that is in one set but not the other — a condition that is explicitly expected during the documented migration period.

### Recommendation

Replace the call to `get_supported_foreign_chains()` with `get_available_foreign_chains()` in `verify_foreign_transaction`:

```rust
// Before (incorrect):
let supported_chains = self.get_supported_foreign_chains();

// After (correct):
let supported_chains = self.get_available_foreign_chains();
```

This aligns the contract gate with the design specification and ensures that only chains that are both RPC-whitelisted and covered by the signing threshold are accepted.

### Proof of Concept

1. Operator registers chain `X` for all participants via the old `register_foreign_chain_config` path.
2. Chain `X` is **not** voted into the RPC whitelist (`vote_update_foreign_chain_providers` never called for `X`).
3. `get_available_foreign_chains()` returns an empty set for `X`; `get_supported_foreign_chains()` returns `X`.
4. Any user calls `verify_foreign_transaction` with chain `X` and attaches the required deposit.
5. The contract passes the gate check (line 535) and enqueues the yield.
6. MPC nodes call `execute_foreign_chain_request` → `chain_is_supported` → reads the contract's supported chains → chain `X` is present → nodes proceed to the inspector.
7. No inspector is configured for `X` (no whitelist entry, no provider URLs) → every node returns an error → the request times out.

```rust
    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }
```

```rust
async fn chain_is_supported(
    policy_reader: &impl ReadSupportedForeignChain,
    request: &dtos::ForeignChainRpcRequest,
) -> Result<(), ForeignChainSupportError> {
    let on_chain_foreign_chains_support = policy_reader
        .get_supported_chains()
        .await
        .map_err(ForeignChainSupportError::FailedToReadContract)?;

    let requested_chain = request.chain();

    if on_chain_foreign_chains_support.contains(&requested_chain) {
        Ok(())
    } else {
        Err(ForeignChainSupportError::ChainNotSupported {
            requested: requested_chain,
        })
    }
}
```
