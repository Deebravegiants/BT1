### Title
`ProposalFinPayload.l2_gas_info` declared `optional` in proto but required in Rust converter — (`crates/apollo_protobuf/src/converters/consensus.rs`)

### Summary

The protobuf wire definition marks `l2_gas_info` as `optional` inside `ProposalFinPayload`, but the Rust `TryFrom` converter unconditionally requires it with `.ok_or(missing("l2_gas_info"))?`. Any peer that sends a `ProposalFinPayload` with `l2_gas_info` absent — which is valid per the proto contract — causes the validator's deserialization to fail inside `handle_proposal_part`, aborting the proposal before the `ProposalCommitment` comparison is ever reached.

### Finding Description

**Proto definition** (`crates/apollo_protobuf/src/proto/p2p/proto/consensus/consensus.proto`, line 87):

```proto
message ProposalFinPayload {
    CommitmentParts commitment_parts = 1;
    optional L2GasInfo l2_gas_info   = 2;   // ← optional per wire spec
}
```

**Generated Rust struct** (`crates/apollo_protobuf/src/protobuf/protoc_output.rs`, lines 518–523):

```rust
pub struct ProposalFinPayload {
    pub commitment_parts: Option<CommitmentParts>,
    pub l2_gas_info:      Option<L2GasInfo>,      // ← correctly Option<>
}
```

**Rust converter** (`crates/apollo_protobuf/src/converters/consensus.rs`, lines 365–372):

```rust
impl TryFrom<protobuf::ProposalFinPayload> for ProposalFinPayload {
    fn try_from(value: protobuf::ProposalFinPayload) -> Result<Self, Self::Error> {
        let commitment_parts =
            value.commitment_parts.ok_or(missing("commitment_parts"))?.try_into()?;
        let l2_gas_info =
            value.l2_gas_info.ok_or(missing("l2_gas_info"))?.try_into()?;  // ← REQUIRED
        Ok(ProposalFinPayload { commitment_parts, l2_gas_info })
    }
}
```

The proto marks `l2_gas_info` as `optional`; the generated struct correctly represents it as `Option<L2GasInfo>`; but the converter then calls `.ok_or(missing("l2_gas_info"))?`, treating it as mandatory. This is a direct mismatch between the interface definition (proto) and the implementation (converter) — the exact class of bug described in the external report.

The execution path that reaches this code is:

`validate_proposal` → `handle_proposal_part` (line 511, `Some(ProposalPart::Fin(fin))`) → `ProposalFin` deserialization via `TryFrom<protobuf::ProposalFin>` (line 390: `value.fin_payload.map(TryInto::try_into).transpose()?`) → `TryFrom<protobuf::ProposalFinPayload>` → error on absent `l2_gas_info`.

The error surfaces as `HandledProposalPart::Failed(...)` (line 217 of `validate_proposal.rs`), which causes `validate_proposal` to return `Err(ValidateProposalError::ProposalPartFailed(...))` and the validator to abort the proposal — before the `built_block != received_fin.proposal_commitment` check at line 244 is ever evaluated.

### Impact Explanation

A peer that sends a `ProposalFinPayload` with `l2_gas_info = None` — which is valid per the proto wire contract — causes every validator that receives it to reject the proposal with a deserialization error. Because the rejection happens before the `ProposalCommitment` comparison, the validator cannot distinguish a legitimate proto-compliant peer from a malformed message. This breaks the proposal validation path and can cause consensus to stall or force unnecessary round increments, effectively preventing valid transactions from being sequenced.

### Likelihood Explanation

The current Apollo proposer always populates `l2_gas_info: Some(...)` in `build_proposal.rs` (line 334–340) and `send_reproposal` (line 1328–1334). However, the proto contract is the authoritative wire interface. Any alternative implementation (a different sequencer client, a future version, or a fuzzer) that follows the proto spec and omits `l2_gas_info` will trigger the mismatch. The trigger requires no privilege — any consensus peer can send a `ProposalFin` message.

### Recommendation

Align the converter with the proto contract. Either:

1. **Make `l2_gas_info` required in the proto** (remove `optional`) if it is always expected, so the wire spec matches the converter:
   ```proto
   L2GasInfo l2_gas_info = 2;   // required
   ```

2. **Or handle `None` gracefully in the converter** if the field is genuinely optional, using a default value:
   ```rust
   let l2_gas_info = value.l2_gas_info
       .map(TryInto::try_into)
       .transpose()?
       .unwrap_or_default();
   ```
