### Title
Unverified Node-Supplied `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Replay — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` accepts a caller-controlled `response.payload_hash` and only checks that the supplied signature is valid for that hash. It never verifies that `payload_hash` was correctly derived from the `request` argument. A single malicious attested node can replay a valid MPC signature obtained from one signing session to resolve a completely different pending foreign-tx request, delivering a forged `payload_hash` to the waiting user contract.

### Finding Description

In `respond_verify_foreign_tx`, the contract performs the following signature check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← node-supplied, unchecked

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← verified only against itself
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract confirms that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root MPC public key. It does **not** recompute the expected hash from the `request` argument and compare it. The correct hash would be:

```
SHA-256(borsh(ForeignTxSignPayload::V1 { request, values: <actual_extracted_values> }))
```

as defined in `ForeignTxSignPayload::compute_msg_hash`. [2](#0-1) 

By contrast, the client-side SDK (`near-mpc-sdk`) **does** perform this check:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [3](#0-2) 

The on-chain contract enforces no equivalent invariant.

### Impact Explanation

A single Byzantine attested node that participates in a legitimate signing session for request R2 obtains a valid threshold signature `sig` over hash `H2`. It then calls `respond_verify_foreign_tx(request = R1, response = { payload_hash: H2, signature: sig })` for a different pending request R1. The contract:

1. Confirms the caller is an attested participant. ✓
2. Verifies `sig` is valid for `H2` under the root key. ✓ (it is — from R2's session)
3. Calls `resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &request, ...)`, permanently draining R1's yield queue with the forged response. [4](#0-3) 

Two concrete harms result:

- **Request lifecycle corruption (Medium):** R1's pending slot is permanently consumed. The user cannot obtain a valid response; their deposit is lost and the yield cannot be retried.
- **Forged foreign-chain verification (High):** User contracts that do not independently recompute and compare `payload_hash` (i.e., do not use `ForeignChainSignatureVerifier::verify_signature`) receive a response asserting that a different set of extracted values was observed on the foreign chain. This can cause invalid bridge execution or double-spend conditions — the exact impact class listed in the allowed scope.

### Likelihood Explanation

The attacker must be a single attested MPC participant (below the signing threshold). No collusion with other nodes is required. The attacker only needs to:

1. Be an active, attested participant (already a precondition for calling `respond_verify_foreign_tx`).
2. Participate in any legitimate foreign-tx signing session to obtain a valid `(payload_hash, signature)` pair.
3. Submit that pair against a different pending request.

This is a realistic, low-barrier attack for any single compromised or malicious node.

### Recommendation

Before resolving the yield, recompute the expected payload hash on-chain from the `request` argument and assert it equals `response.payload_hash`:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    // values cannot be recomputed on-chain; instead, verify the hash
    // is consistent with what was signed by checking it is the hash
    // the signature covers — and bind it to the request key.
}).compute_msg_hash();
```

Since the contract cannot independently re-execute the foreign chain query, the minimal fix is to store the `payload_hash` alongside the pending request at enqueue time (once the leader broadcasts it) and reject any `respond_verify_foreign_tx` call whose `response.payload_hash` does not match the stored value. Alternatively, include the `request` data in the signed payload in a way that the contract can verify deterministically without re-executing the RPC.

### Proof of Concept

1. Attacker node A is an attested participant.
2. User U1 calls `verify_foreign_transaction` for Bitcoin tx `tx_id = X`. A pending entry is created under key `R1`.
3. User U2 calls `verify_foreign_transaction` for Bitcoin tx `tx_id = Y`. A pending entry is created under key `R2`.
4. The MPC network (including A) runs the signing protocol for R2, producing `(H2, sig2)` where `H2 = SHA-256(borsh(ForeignTxSignPayload { request: R2, values: [block_hash_Y] }))`.
5. Node A, instead of calling `respond_verify_foreign_tx(R2, {H2, sig2})`, calls `respond_verify_foreign_tx(R1, {H2, sig2})`.
6. The contract at line 729 verifies `verify_ecdsa_signature(sig2, H2, root_pk)` → passes.
7. R1's yield queue is drained with response `{payload_hash: H2, signature: sig2}`.
8. U1's contract receives a response claiming the Bitcoin tx `X` produced block hash `block_hash_Y` — a forged attestation. If U1's contract does not call `ForeignChainSignatureVerifier::verify_signature`, it accepts the forged result and executes a bridge action based on incorrect foreign chain state. impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
