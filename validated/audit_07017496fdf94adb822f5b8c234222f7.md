### Title
Non-Winning Proposal Deposits Permanently Lost on `do_update` — (`crates/contract/src/update.rs`)

### Summary

When `do_update` executes a winning proposal, it unconditionally clears **all** pending proposal entries via `self.entries.clear()`. Because the contract never records the proposer's account ID in `UpdateEntry`, and issues no refund `Promise` for the cleared entries, every deposit paid by non-winning proposers is permanently absorbed into the contract balance with no recovery path.

### Finding Description

`propose_update` enforces a mandatory storage deposit proportional to the size of the proposed update: [1](#0-0) 

Only the *excess* above the required amount is refunded at proposal time: [2](#0-1) 

The proposer's `AccountId` is never stored. `UpdateEntry` only holds the update payload and its byte size: [3](#0-2) 

When threshold votes are reached for any proposal, `do_update` is called. It removes the winning entry, then immediately clears the entire entries map and all votes: [4](#0-3) 

No `Promise::transfer` is emitted for any of the cleared entries. The NEAR that backed their storage is freed back to the contract's own balance — not to the original proposers.

### Impact Explanation

Any participant who paid a deposit for a proposal that is not the one that reaches threshold loses that deposit permanently. For a contract binary update, `bytes_used` includes the full code length plus a 128-account-ID overhead estimate: [5](#0-4) 

At NEAR's storage cost of ~1 NEAR per 10 KB, a 1 MB contract binary proposal costs ~100 NEAR. With multiple competing proposals, each non-winning proposer loses their full required deposit. There is no admin function, no expiry/withdrawal path, and no stored proposer identity to enable any future recovery.

### Likelihood Explanation

The existing sandbox test `test_propose_update_contract_many` already demonstrates two simultaneous proposals where one wins and all are cleared: [6](#0-5) 

The test asserts that all proposals are removed after the update but does **not** assert that non-winning proposers receive refunds — confirming the loss is the current behavior. Any voter (participant) can trigger this by proposing a competing update and gathering threshold votes for it.

### Recommendation

1. Add a `proposer: AccountId` field to `UpdateEntry` so the contract knows who to refund.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and emit a `Promise::new(entry.proposer).transfer(required_deposit(entry.bytes_used))` for each one.
3. Alternatively, implement a withdrawal pattern: store the proposer and allow them to reclaim their deposit once their proposal is no longer active.

//   -> entries.clear()        [A's entry silently dropped, deposit_a absorbed]
//   -> vote_by_participant.clear()

The `do_update` call at line 199 (`self.entries.clear()`) is the exact root cause: it discards all non-winning entries without iterating over them to issue refunds, and `UpdateEntry` contains no `proposer` field to make such refunds possible. [7](#0-6)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
