### Title
Storage-Payment Accounting Bypass via State-Before-Check Ordering in `submit_participant_info` - (File: crates/contract/src/lib.rs)

### Summary

`submit_participant_info` writes the caller's attestation into persistent contract storage via `tee_state.add_participant` **before** verifying that the caller attached sufficient deposit to cover the storage cost. Because the function is decorated with `#[handle_result]`, returning `Err` does not trigger a NEAR state rollback — state changes are committed even when the function returns an error value. A Byzantine participant with valid TEE hardware can therefore register their attestation entry at zero cost, forcing the contract to absorb the storage-staking charge from its own balance.

### Finding Description

In `crates/contract/src/lib.rs`, `submit_participant_info` follows this order:

1. **Capture baseline storage** (`initial_storage = env::storage_usage()`, line 780).
2. **Write to storage** — `self.tee_state.add_participant(...)` inserts the `NodeAttestation` entry into the `stored_attestations` `IterableMap` (lines 796–815). This is a committed state mutation.
3. **Compute cost and check deposit** (lines 826–841). If `attached < cost`, the function executes `return Err(InvalidParameters::InsufficientDeposit { … }.into())`. [1](#0-0) 

Because `#[handle_result]` serialises the `Err` value and returns it to the caller without panicking, the NEAR runtime **commits** the state written in step 2. The attestation entry persists in `tee_state.stored_attestations` even though the deposit check failed.

Contrast this with `propose_update`, which correctly performs the deposit check **before** calling `self.proposed_updates.propose(update)`: [2](#0-1) 

The inconsistency is the root cause. The `tee_state.add_participant` path has no equivalent pre-check guard.

### Impact Explanation

**Medium.** The production safety/accounting invariant — *callers who create new attestation entries must pay for the resulting storage staking* — is violated. The contract's NEAR balance is reduced by the storage-staking cost of each inserted entry without receiving the corresponding deposit from the caller. A Byzantine participant (one with valid TEE hardware but acting adversarially) can register their attestation for free, and because attestation entries are keyed per `tls_public_key`, a single operator controlling multiple TEE nodes can repeat this across all of them. The stored attestations are subsequently read by `vote_new_parameters` to validate proposed participants, so free insertion also bypasses the economic friction that storage payment was intended to impose on participant-state changes. [3](#0-2) 

### Likelihood Explanation

Low-to-medium. In production the attacker must possess valid TEE hardware capable of producing an attestation that passes `add_participant`'s verification step; the `Mock` path is test-only. However, every legitimate MPC node operator already satisfies this precondition, so any one of them can exploit the bug without any additional privilege. The exploit requires only a single crafted transaction with zero attached deposit.

### Recommendation

Move the deposit check **before** the `add_participant` call, mirroring the pattern used in `propose_update`. Alternatively, replace the `return Err(…)` branch with `env::panic_str(…)` so that the NEAR runtime rolls back the storage write on insufficient deposit. A pre-check is preferable because it avoids writing and then discarding storage on every under-funded call.

### Proof of Concept

1. Attacker (a legitimate TEE node operator) calls `submit_participant_info` with a valid `Attestation` and their `tls_public_key`, attaching **0 yoctoNEAR** as deposit.
2. `add_participant` succeeds (attestation is valid) and writes the `NodeAttestation` entry to `tee_state.stored_attestations`.
3. `env::storage_usage().saturating_sub(initial_storage)` returns a non-zero byte count; `cost` is computed as a positive `NearToken`.
4. `attached (0) < cost` → the function returns `Err(InsufficientDeposit)`.
5. Because `#[handle_result]` does not panic, NEAR commits the state. The attestation entry is now permanently stored.
6. The contract's NEAR balance is reduced by `cost` (storage staking) with no offsetting deposit received.
7. The attacker's node can now be proposed as a participant in `vote_new_parameters` with a valid stored attestation, having paid nothing for the storage that entry occupies.
