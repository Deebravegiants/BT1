### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Attested Node to Forge Confidential Key Derivation Output — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract applies a cryptographic output check (`ckd_output_check`) only when the request uses the `AppPublicKeyPV` variant. When the legacy `AppPublicKey` variant is used, the contract performs **no verification** of the returned `CKDResponse`. A single attested MPC participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary `big_y`/`big_c` response and the contract will accept and deliver it to the waiting user.

---

### Finding Description

The `respond_ckd` method contains a flag-controlled branch that conditionally skips the only on-chain cryptographic integrity check for the CKD output:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `ckd_output_check` function verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the network's master secret key and the user's app identity. This check is the only on-chain guard that ensures the CKD output was honestly computed by a threshold of participants. [2](#0-1) 

For `AppPublicKey` requests, the contract skips this check entirely and proceeds directly to resolving the yield with whatever `CKDResponse` the caller supplied: [3](#0-2) 

The `AppPublicKey` variant is the legacy/non-PV form and is still accepted by both `request_app_private_key` and `respond_ckd`. The node-side `CKDComputation` runs a different protocol branch (`ckd` vs `ckd_pv`) depending on the variant, but the contract-side check is entirely absent for the `AppPublicKey` branch. [4](#0-3) 

The only access control on `respond_ckd` is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a single attested participant — not a threshold of them. [5](#0-4) 

---

### Impact Explanation

**Critical.** A single attested MPC participant can call `respond_ckd` with a `CKDRequest` that uses the `AppPublicKey` variant and supply an arbitrary `CKDResponse` (`big_y`, `big_c`). The contract resolves the user's yield with this forged output. The user then uses `big_y` and `big_c` to reconstruct their derived private key — a key that the malicious node chose and therefore knows. This constitutes unauthorized confidential key derivation output without the required threshold participant authorization, and gives the attacker full knowledge of the victim's derived private key, enabling decryption of all data encrypted to the corresponding public key and impersonation of the user on any system relying on that key.

---

### Likelihood Explanation

**Medium.** The `AppPublicKey` variant is the legacy format and is still accepted by the contract. Any single attested participant — not a threshold — can exploit this. The attacker does not need to collude with other nodes; they only need to be an active, attested participant in the current epoch. The attack is executable in a single transaction and requires no special tooling beyond constructing a valid `respond_ckd` call.

---

### Recommendation

Apply `ckd_output_check` unconditionally for all CKD responses, regardless of the `app_public_key` variant. If the `AppPublicKey` variant cannot provide the G2 component (`pk2`) needed for the pairing check, deprecate and remove it from the contract's accepted input set. The `AppPublicKeyPV` variant already provides the necessary public verification material; migrating all callers to it eliminates the unverified branch entirely. Until migration is complete, the contract should reject `AppPublicKey` requests at the `request_app_private_key` entry point rather than accepting them without the ability to verify the response.

---

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key: CKDAppPublicKey::AppPublicKey(some_g1_point)` and a valid `domain_id`. The contract stores the `CKDRequest` and yields.

2. Attacker (a single attested MPC participant) observes the pending request on-chain and calls `respond_ckd` with:
   - `request`: the exact `CKDRequest` stored by the contract (reconstructible from the on-chain event)
   - `response`: `CKDResponse { big_y: attacker_chosen_g1, big_c: attacker_chosen_g1 }` — completely arbitrary values

3. The contract executes the match arm `AppPublicKey(_) => {}` — no check — and calls `resolve_yields_for`, delivering the forged response to the user's waiting callback.

4. The user receives `(big_y, big_c)` chosen by the attacker. The user computes their derived private key from these values. The attacker, having chosen `big_y` and `big_c`, knows the resulting derived key. [6](#0-5) [2](#0-1)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
