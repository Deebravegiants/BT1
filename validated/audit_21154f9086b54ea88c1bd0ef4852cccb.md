### Title
Wrong `TransactionOffsetInBlock` in Pre-confirmed Receipts Due to Index from Successful-Only Map — (`crates/apollo_batcher/src/block_builder.rs`)

### Summary

In `collect_execution_results_and_stream_txs`, the `TransactionOffsetInBlock` embedded in each pre-confirmed `StarknetClientTransactionReceipt` is derived from the insertion index into `execution_infos_and_signatures` — an `IndexMap` that holds **only successfully executed transactions**. When rejected transactions precede successful ones in the same execution chunk, the map-derived index is smaller than the transaction's true position in the block, producing a wrong `transaction_index` in every pre-confirmed receipt that follows a rejection.

### Finding Description

`handle_executed_txs` calls `collect_execution_results_and_stream_txs` with a slice of `block_txs` (`&self.block_txs[old_n_executed_txs..self.n_executed_txs]`) and the corresponding executor results. [1](#0-0) 

Inside that function, for each successfully executed transaction the code does:

```rust
let (tx_index, duplicate_tx_hash) =
    execution_data.execution_infos_and_signatures.insert_full(
        tx_hash,
        (tx_execution_info, input_tx.tx_signature_for_commitment()),
    );
// ...
let tx_receipt = StarknetClientTransactionReceipt::from((
    tx_hash,
    TransactionOffsetInBlock(tx_index),   // ← wrong index
    &execution_data.execution_infos_and_signatures[&tx_hash].0,
    optional_l1_handler_tx,
));
``` [2](#0-1) 

`insert_full` returns the index of the newly inserted entry **within `execution_infos_and_signatures`**, which only contains successful transactions. Rejected transactions are routed to `rejected_tx_hashes` and never appear in this map. [3](#0-2) 

`block_txs`, by contrast, holds **all** transactions in block order (both successful and rejected). The true block offset of the `i`-th transaction in `tx_chunk` is `old_n_executed_txs + i`, regardless of how many of the preceding transactions were rejected.

**Concrete example:**

| Block position | tx | Result | `execution_infos_and_signatures` index | Emitted `TransactionOffsetInBlock` |
|---|---|---|---|---|
| 0 | tx0 | success | 0 | 0 ✓ |
| 1 | tx1 | success | 1 | 1 ✓ |
| 2 | tx2 | **rejected** | — | — |
| 3 | tx3 | success | **2** | **2 ✗** (should be 3) |

The receipt for tx3 carries `transaction_index = 2` instead of `3`.

The receipt is then forwarded to the `PreconfirmedBlockWriter`, which batches it and writes it to the Cende recorder (Aerospike) as part of the pre-confirmed block: [4](#0-3) 

The `StarknetClientTransactionReceipt.transaction_index` field is the authoritative `transaction_index` field in the pre-confirmed block's receipt array: [5](#0-4) 

### Impact Explanation

The pre-confirmed block is the live, authoritative view of the block being built. Any consumer reading pre-confirmed receipts (e.g., via the Cende recorder or a pending-block RPC endpoint) will see a `transaction_index` that is off by the number of rejected transactions that preceded the successful one. This is a wrong value in an authoritative-looking pending view, matching the **High** impact criterion: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

The `transaction_index` is also used by the `apollo_starknet_client` reader to validate receipt-to-transaction alignment: [6](#0-5) 

A downstream consumer that applies the same validation to pre-confirmed data would reject the block or misroute receipts to the wrong transactions.

### Likelihood Explanation

Rejected transactions (insufficient fee, invalid nonce, resource-bound violations, etc.) are a normal, unprivileged occurrence in every production block. Any block that contains at least one rejected transaction followed by at least one successful transaction in the same execution chunk triggers the bug. No special attacker capability is required; ordinary user traffic is sufficient.

### Recommendation

Pass the starting block offset (`old_n_executed_txs`) into `collect_execution_results_and_stream_txs` and compute the receipt index from the chunk position rather than from the map:

```rust
async fn collect_execution_results_and_stream_txs(
    tx_chunk: &[InternalConsensusTransaction],
    chunk_start_offset: usize,          // ← add this
    results: Vec<TransactionExecutorResult<TransactionExecutionOutput>>,
    execution_data: &mut BlockTransactionExecutionData,
    pre_confirmed_tx_sender: &mut Option<PreconfirmedTxSender>,
) -> BlockBuilderResult<()> {
    for (i, (input_tx, result)) in tx_chunk.iter().zip(results).enumerate() {
        let block_offset = chunk_start_offset + i;   // true block position
        // ...
        let tx_receipt = StarknetClientTransactionReceipt::from((
            tx_hash,
            TransactionOffsetInBlock(block_offset),  // ← correct
            ...
        ));
    }
}
```

The call site already has `old_n_executed_txs` available immediately before the call: [1](#0-0) 

### Proof of Concept

1. Submit a block with at least two transactions: `tx_A` (will be rejected — e.g., nonce too low) and `tx_B` (valid, submitted after `tx_A`).
2. Both land in the same execution chunk returned by `get_new_results`.
3. `collect_execution_results_and_stream_txs` processes them in order:
   - `tx_A` → `Err` → inserted into `rejected_tx_hashes`; no receipt emitted.
   - `tx_B` → `Ok` → `insert_full` returns index `0` (first entry in `execution_infos_and_signatures`); receipt emitted with `TransactionOffsetInBlock(0)`.
4. `tx_B`'s true block offset is `1` (it is the second transaction in the block).
5. The pre-confirmed block written to Aerospike contains a receipt for `tx_B` with `transaction_index = 0`, which is the index of `tx_A` — a different transaction. [7](#0-6)

async fn collect_execution_results_and_stream_txs(
    tx_chunk: &[InternalConsensusTransaction],
    results: Vec<TransactionExecutorResult<TransactionExecutionOutput>>,
    execution_data: &mut BlockTransactionExecutionData,
    pre_confirmed_tx_sender: &mut Option<PreconfirmedTxSender>,
) -> BlockBuilderResult<()> {
    assert!(
        results.len() == tx_chunk.len(),
        "The number of results match the number of transactions."
    );

    for (input_tx, result) in tx_chunk.iter().zip(results) {
        let optional_l1_handler_tx =
            if let InternalConsensusTransaction::L1Handler(l1_handler_tx) = input_tx {
                Some(l1_handler_tx.tx.clone())
            } else {
                None
            };
        let tx_hash = input_tx.tx_hash();

        // Insert the tx_hash into the appropriate collection if it's an L1_Handler transaction.
        if let InternalConsensusTransaction::L1Handler(_) = input_tx {
            let is_new_entry = execution_data.consumed_l1_handler_tx_hashes.insert(tx_hash);
            // Even though this doesn't get past the set insertion, this indicates a major, possibly
            // reorg-producing bug, either in some batcher cache or the l1 provider.
            assert!(is_new_entry, "Duplicate L1 handler transaction hash: {tx_hash}.");
        }

        match result {
            Ok((tx_execution_info, state_maps)) => {
                if let Some(ref revert_error) = tx_execution_info.revert_error {
                    warn!(
                        "Transaction {} is reverted during execution while still accepted. Revert \
                         Error: {}",
                        input_tx.tx_hash(),
                        revert_error,
                    );
                }
                let (tx_index, duplicate_tx_hash) =
                    execution_data.execution_infos_and_signatures.insert_full(
                        tx_hash,
                        (tx_execution_info, input_tx.tx_signature_for_commitment()),
                    );
                assert_eq!(duplicate_tx_hash, None, "Duplicate transaction: {tx_hash}.");

                if let Some(block_number) = proof_facts_block_number(input_tx) {
                    execution_data.proof_facts_block_numbers.insert(tx_hash, block_number);
                }

                // Skip sending the pre confirmed executed transactions, receipts and state diffs
                // during validation flow or if the channel was closed. In validate flow
                // pre_confirmed_tx_sender is None.
                if let Some(pre_confirmed_sender) = pre_confirmed_tx_sender {
                    let tx_receipt = StarknetClientTransactionReceipt::from((
                        tx_hash,
                        TransactionOffsetInBlock(tx_index),
                        // TODO(noamsp): Consider using tx_execution_info and moving the line that
                        // consumes it below this (if it doesn't change functionality).
                        &execution_data.execution_infos_and_signatures[&tx_hash].0,
                        optional_l1_handler_tx,
                    ));

                    let tx_state_diff = StarknetClientStateDiff::from(state_maps).0;

                    let result = pre_confirmed_sender.try_send((
                        input_tx.clone(),
                        tx_receipt,
                        tx_state_diff,
                    ));

                    match result {
                        Ok(_) => {}
                        Err(TrySendError::Closed(_)) => {
                            warn!(
                                "Preconfirmed block writer channel was closed. Skipping to send \
                                 further preconfirmed transactions."
                            );
                            *pre_confirmed_tx_sender = None;
                        }
                        Err(err) => {
                            warn!("Sending data to preconfirmed block writer failed: {:?}", err);
                        }
                    }
                }
            }
            Err(err) => {
                info!(
                    "Transaction {} failed to execute with error: {}.",
                    tx_hash,
                    err.log_compatible_to_string()
                );
                let is_new_entry = execution_data.rejected_tx_hashes.insert(tx_hash);
                assert!(is_new_entry, "Duplicate rejected transaction hash: {tx_hash}.");
            }
        }
    }

    Ok(())
}
