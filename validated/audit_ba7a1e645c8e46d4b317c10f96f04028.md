### Title
Single Attested Participant Can Submit Arbitrary CKD Response for `AppPublicKey` Variant, Bypassing Threshold Authorization — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs no cryptographic verification for the `AppPublicKey` (legacy) CKD variant. Unlike the `AppPublicKeyPV` variant (which has on-chain `ckd_output_check` verification) and the `respond` function (which verifies signatures against the derived key), the `AppPublicKey` branch is entirely empty. A single Byzantine attested participant below the signing threshold can race legitimate nodes and deliver an arbitrary `CKDResponse` to any waiting user, bypassing the threshold authorization model entirely.

---

### Finding Description

In `respond_ckd`, the function branches on the request's `app_public_key` variant: [1](#0-0) 

- For `AppPublicKeyPV`: `ckd_output_check` is called, cryptographically verifying the response against the BLS12-381 root public key.
- For `AppPublicKey` (legacy): the match arm is **empty** — no verification whatsoever is performed on the response.

The function's only gate is `assert_caller_is_attested_participant_and_protocol_active`, which is a single-participant check: [2](#0-1) 

There is no threshold requirement enforced on-chain for the response content. A single Byzantine attested participant can call `respond_ckd` first with any arbitrary `CKDResponse`, and `resolve_yields_for` will accept it and deliver it to the waiting user.

This contrasts sharply with `respond` (the signature path), where the response is cryptographically verified against the **derived** public key before the yield is resolved — making forgery computationally infeasible for a single participant: [3](#0-2) 

The `AppPublicKey` variant is described in the README as "privately verifiable (legacy)" — meaning verification is expected off-chain by the app using its secret key. However, the contract still accepts and resolves the yield based on the unverified on-chain response, so a forged response is indistinguishable from a legitimate one at the contract level.

---

### Impact Explanation

A single Byzantine participant below the signing threshold can deliver an arbitrary `CKDResponse` to any user who submitted a CKD request using the `AppPublicKey` variant. The user receives a forged confidential key derivation output without the required threshold participant authorization. This directly matches the critical allowed impact: **"confidential key derivation output without the required participant authorization."**

The attacker can craft `big_y` and `big_c` values (BLS12-381 G1 points) that, when decrypted by the user's app private key, yield a key the attacker controls — enabling theft of any assets the user subsequently places under that derived key.

---

### Likelihood Explanation

The `AppPublicKey` variant is still live and accepted by the contract (not deprecated or gated). Any single attested participant in the MPC network can exploit this. The attacker only needs to observe a pending CKD request in `pending_ckd_requests` and submit a forged response before legitimate nodes. Since `resolve_yields_for` resolves on the first valid call, the race is winnable by any participant who acts promptly. [4](#0-3) 

---

### Recommendation

- **Short term**: Add cryptographic verification for the `AppPublicKey` variant in `respond_ckd`. Even a basic consistency check (e.g., verifying that `big_c` is a valid point on BLS12-381 G1 and that the response is consistent with the domain's root public key) would raise the bar significantly.
- **Long term**: Deprecate and remove the `AppPublicKey` variant entirely, requiring all CKD requests to use `AppPublicKeyPV`, which supports full on-chain verification via `ckd_output_check`.

---

### Proof of Concept

1. User submits a CKD request with the `AppPublicKey` variant via `request_app_private_key`.
2. Byzantine attested participant observes the pending request in `pending_ckd_requests`.
3. Byzantine participant calls `respond_ckd` with a crafted `CKDResponse { big_y: <attacker_controlled_point>, big_c: <attacker_controlled_point> }` before legitimate nodes respond.
4. `respond_ckd` passes all checks (`assert_caller_is_signer`, `assert_caller_is_attested_participant_and_protocol_active`, protocol state) and skips verification for `AppPublicKey` (empty match arm at line 676).
5. `resolve_yields_for` resolves the yield with the forged response.
6. User's `request_app_private_key` call returns the attacker-controlled `CKDResponse`. The user decrypts it with their app private key and receives a key the attacker knows, enabling theft of any assets placed under that key.

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
