### Title
Stale Gateway Snapshot Allows `skip_stateful_validations` to Bypass `__validate__` for Already-Deployed Accounts — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is designed to skip `__validate__` for an Invoke with nonce=1 when the gateway state shows nonce=0 and the mempool confirms a deploy_account is in-flight. Due to a race condition between the gateway's fixed-block snapshot and the mempool's committed-block state, an already-deployed account (on-chain nonce=1) can appear as nonce=0 to the gateway while `account_tx_in_pool_or_recent_block` simultaneously returns `true`. This causes the gateway to skip `__validate__` for an Invoke with an invalid signature, admitting it to the mempool.

---

### Finding Description

**Step 1 — Nonce read from a fixed, potentially stale snapshot.**

`get_nonce_from_state` reads the account nonce from `gateway_fixed_block_state_reader`, which is a snapshot pinned to `latest_block_number` at the moment `instantiate_validator` was called: [1](#0-0) 

That snapshot is created once per transaction via `get_blockifier_state_reader_and_gateway_fixed_block_from_latest_block`, which calls `get_latest_block_number()` and pins both readers to that block: [2](#0-1) 

**Step 2 — `skip_stateful_validations` triggers on nonce=0 + mempool hit.**

If the snapshot returns nonce=0 and the incoming Invoke has nonce=1, the function delegates to `account_tx_in_pool_or_recent_block` and returns its result directly as the skip decision: [3](#0-2) 

**Step 3 — `account_tx_in_pool_or_recent_block` returns `true` for accounts seen in any committed block.**

The mempool's state is updated via `commit_block`, which records every address whose nonce advanced. After `commit_block` is called for the block containing the deploy_account tx, `state.contains_account(address)` is permanently `true`: [4](#0-3) 

The mempool test explicitly documents that this flag persists after commit: [5](#0-4) 

**Step 4 — Race window.**

The batcher calls `commit_block` on the mempool as soon as a block is finalized. The gateway's state sync is updated independently and may lag by one or more blocks. During this window:
- Mempool: `account_tx_in_pool_or_recent_block(addr)` → `true` (account seen in committed block)
- Gateway snapshot: `get_nonce(addr)` → `Nonce(0)` (stale, pre-deploy block)

**Step 5 — `__validate__` is skipped; invalid signature admitted.**

`run_validate_entry_point` sets `execution_flags.validate = !skip_validate`, so when `skip_validate=true`, the blockifier's `perform_validations` returns immediately after `perform_pre_validation_stage` without calling `validate_tx`: [6](#0-5) 

`perform_pre_validation_stage` uses `strict_nonce_check = false`, so nonce=1 against state nonce=0 passes (0 ≤ 1): [7](#0-6) [8](#0-7) 

The transaction with an invalid signature is admitted to the mempool.

---

### Impact Explanation

An unprivileged user can submit an Invoke transaction with nonce=1 and an arbitrary (invalid) signature during the race window after a deploy_account is committed to a block but before the gateway's state sync advances. The gateway skips `__validate__`, so the signature is never verified at admission time. The transaction enters the mempool and is forwarded to the batcher, where it will fail `__validate__` during execution and be included as a reverted transaction (with fee charged to the sender). The primary confirmed impact is **gateway admission of an invalid transaction** — a High-severity finding per the defined scope.

---

### Likelihood Explanation

The race window exists in normal operation whenever the batcher commits a block faster than the gateway's state sync propagates it. This is an architectural certainty, not a theoretical edge case. The window duration depends on sync latency (typically seconds). An attacker who monitors the mempool for deploy_account transactions and times their Invoke submission to the window can reliably trigger this. No privileged access is required.

---

### Recommendation

Replace the binary `account_nonce == Nonce(Felt::ZERO)` check with a check that also consults the mempool for a **pending** deploy_account transaction specifically (not just any historical presence). Alternatively, only skip `__validate__` when `account_tx_in_pool_or_recent_block` returns `true` **and** the mempool confirms a deploy_account with nonce=0 is currently in the pending pool (not merely in a committed block). A committed-block hit should not be treated as evidence that the account is undeployed.

---

### Proof of Concept

1. Account `A` sends deploy_account (nonce=0). Batcher includes it in block N+1 and calls `mempool.commit_block({A: nonce=1})`. Now `account_tx_in_pool_or_recent_block(A)` → `true`.
2. Gateway state sync is still at block N. `gateway_fixed_block_state_reader.get_nonce(A)` → `Nonce(0)`.
3. Attacker submits `Invoke { sender: A, nonce: 1, signature: [0xdeadbeef] }` to the gateway RPC.
4. `extract_state_nonce_and_run_validations`: `account_nonce = 0`.
5. `validate_nonce`: `0 ≤ 1 ≤ 0 + max_gap` → passes.
6. `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0` → calls `account_tx_in_pool_or_recent_block(A)` → `true` → returns `true`.
7. `run_validate_entry_point(skip_validate=true)`: `execution_flags.validate = false` → `__validate__` not called.
8. Transaction with invalid signature is admitted to the mempool.
