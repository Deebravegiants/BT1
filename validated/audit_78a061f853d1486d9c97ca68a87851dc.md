Audit Report

## Title
`collectFees` push-transfer DoS when `adminFeeDestination` is USDC-blacklisted, freezing fee collection and fee-rate update flows — (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool.collectFees` unconditionally pushes accrued fees to `adminFeeDestination_` via `transferToken0`/`transferToken1`. If that address is USDC-blacklisted, every call reverts before clearing `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled`. Because `MetricOmmPoolFactory.collectPoolFees`, `setPoolProtocolFee`, and `setPoolAdminFees` all invoke `collectFees` as a mandatory first step, a blacklisted destination simultaneously locks all accrued protocol fees inside the pool and DoS-es the two factory functions that govern fee configuration.

## Finding Description
`MetricOmmPool.collectFees` (lines 416–427) pushes tokens to `adminFeeDestination_` before clearing the notional-fee accumulators at lines 429–430:

```solidity
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // reverts if blacklisted
}
...
notionalFeeToken0Scaled = 0;   // never reached
notionalFeeToken1Scaled = 0;
```

`MetricOmmPoolFactory.collectPoolFees` (lines 379–389) is permissionless and passes `poolAdminFeeDestination[pool]` directly into `collectFees` — any caller triggers the revert. `setPoolProtocolFee` (lines 328–335) and `setPoolAdminFees` (lines 418–425) both call `collectFees` before writing new fee parameters; a revert rolls back the entire transaction, leaving fee rates unchanged. The escape hatch — `setPoolAdminFeeDestination` (lines 438–447) — does not call `collectFees` and is available to the pool admin, so the DoS is temporary rather than permanent. However, until the admin acts, all three functions are fully blocked and accrued fees are inaccessible.

## Impact Explanation
Accrued spread surplus and notional fee accumulators cannot be extracted from the pool while the blacklisting persists. `setPoolAdminFees` (pool-admin path) and `collectPoolFees` (public path) are completely DoS-ed. Protocol fees remain in the pool's balance but are inaccessible. This meets the Medium threshold for temporary but material loss of access to protocol fees and broken core admin fee-management functionality; USDC/USDT non-standard behavior is explicitly in scope.

## Likelihood Explanation
USDC blacklisting of an `adminFeeDestination` is an uncommon external event, but the destination is an arbitrary address set by the pool admin (multisig, DAO treasury, or any contract). The scenario is realistic for pools whose admin fee destination is a regulated or sanctioned entity. No attacker action is required beyond the external blacklisting event.

## Recommendation
Replace the push pattern in `collectFees` with a pull (claim) pattern: accumulate admin and protocol fee amounts in per-token storage mappings keyed by destination address, and expose a separate `claimFees(address token)` function. `setPoolProtocolFee` and `setPoolAdminFees` can then checkpoint accrued amounts without executing any token transfer, eliminating the revert risk entirely.

## Proof of Concept
1. Deploy pool with `token0 = USDC`, `token1 = WETH`; set `adminFeeDestination` to address `D`.
2. Execute swaps so `notionalFeeToken0Scaled > 0` and spread surplus accumulates.
3. USDC issuer blacklists `D`.
4. Call `MetricOmmPoolFactory.collectPoolFees(pool)` → `collectFees(..., D)` → `transferToken0(D, amount)` → USDC `transfer` reverts → `collectPoolFees` reverts. Fee accumulators unchanged.
5. Owner calls `setPoolProtocolFee(pool, newFee, ...)` → same `collectFees` revert path → fee rate not updated.
6. Pool admin calls `setPoolAdminFees(pool, newFee, ...)` → same revert path → fee rate not updated.
7. All accrued fees remain locked; fee configuration frozen until pool admin calls `setPoolAdminFeeDestination` to a non-blacklisted address and retries.
