### Title
Unrestricted `deploy()` Allows Permanent Salt Griefing in CREATE3Factory - (File: `contracts/utils/CREATE3Factory.sol`)

### Summary
`CREATE3Factory.deploy()` has no access control. Any caller can consume any salt by deploying a minimal contract to it, permanently preventing the protocol from deploying its intended contract at the deterministic address derived from that salt.

### Finding Description

`CREATE3Factory.deploy()` is `external payable` with no ownership, role, or caller restriction: [1](#0-0) 

Internally, `CREATE3.deploy()` first deploys a proxy via CREATE2 using the caller-supplied salt: [2](#0-1) 

Once the proxy is deployed at `keccak256(0xFF || factory || salt || PROXY_BYTECODE_HASH)`, that CREATE2 slot is permanently occupied. Any subsequent call with the same salt causes `create2(...)` to return `address(0)`, triggering the `"DEPLOYMENT_FAILED"` revert: [3](#0-2) 

The final child address is also permanently consumed because the proxy already used its nonce=1 CREATE to deploy the attacker's bytecode: [4](#0-3) 

### Impact Explanation

The factory's documented purpose is cross-chain deterministic deployment at a known address: [5](#0-4) 

Once a salt is griefed, the protocol cannot deploy its intended contract at the expected address. Off-chain systems, bridge configurations, or pool setups that reference the pre-computed address will point to attacker-controlled bytecode. No funds are lost directly, but the factory fails to deliver its promised deterministic deployment — matching **Low: Contract fails to deliver promised returns, but doesn't lose value**.

### Likelihood Explanation

The factory's salts are likely derived from predictable inputs (contract names, chain IDs) and may be published in deployment scripts. The attacker does not need to front-run a mempool transaction — they only need to call `deploy()` with the target salt at any point before the protocol's deployment transaction. The cost is a single low-gas transaction deploying a 1-byte contract (`STOP` opcode).

### Recommendation

Add an access control modifier to `deploy()`, restricting it to a trusted deployer role or the contract owner:

```solidity
// Example fix
address public owner;
modifier onlyOwner() { require(msg.sender == owner, "NOT_OWNER"); _; }

function deploy(bytes32 salt, bytes memory creationCode)
    external payable onlyOwner returns (address deployed) { ... }
```

Alternatively, incorporate `msg.sender` into the salt derivation so each caller's salt-space is isolated:

```solidity
bytes32 effectiveSalt = keccak256(abi.encodePacked(msg.sender, salt));
```

### Proof of Concept

```solidity
// 1. Attacker deploys a minimal contract (STOP opcode) to the target salt
factory.deploy(targetSalt, hex"00"); // succeeds

// 2. Protocol attempts to deploy its intended contract
factory.deploy(targetSalt, protocolCreationCode); // reverts: "DEPLOYMENT_FAILED"

// 3. The expected address holds attacker bytecode
address expected = factory.getDeployed(targetSalt);
assert(expected.code.length > 0); // attacker's contract is there
``` [1](#0-0)
