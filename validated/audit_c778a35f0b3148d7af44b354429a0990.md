Audit Report

## Title
Uninitialized `minRsEthAmountToWithdraw` Allows Dust-Request Queue Flooding, Enabling Unbounded Gas Consumption and Temporary Freezing of Funds - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`minRsEthAmountToWithdraw` is a `mapping(address => uint256)` that defaults to `0` for every asset and is never initialized in `initialize()`. Because the guard in `initiateWithdrawal` compares `rsETHUnstaked < 0` (a vacuously false `uint256` comparison) when the mapping is unset, any rsETH holder can submit arbitrarily many 1-wei withdrawal requests. The FIFO `_unlockWithdrawalRequests` loop must process every preceding entry before reaching legitimate requests, causing unbounded gas consumption and temporarily freezing legitimate withdrawers' funds.

## Finding Description
`minRsEthAmountToWithdraw` is declared at line 35 but never assigned in `initialize()` (lines 90–98), so it is `0` for every asset by default. The guard at lines 162–164:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

only rejects a zero amount; when the mapping value is `0`, the second condition `rsETHUnstaked < 0` is always false for `uint256`, so any amount ≥ 1 wei is accepted.

Each accepted request is appended to the global FIFO queue via `_addUserWithdrawalRequest` (lines 744–759), incrementing `nextUnusedNonce[asset]`. The operator's unlock path, `_unlockWithdrawalRequests` (lines 790–815), iterates from `nextLockedNonce[asset]` to `firstExcludedIndex` in a `while` loop, performing multiple storage reads and writes per iteration. The only early-exit conditions are:
- withdrawal delay not yet elapsed (`block.number < request.withdrawalStartBlock + withdrawalDelayBlocks`)
- insufficient available assets (`availableAssetAmount < payoutAmount`)

For 1-wei dust requests, the payout amount is negligible, so neither condition triggers early. Legitimate requests queued after the dust flood cannot be unlocked until every preceding dust entry is processed.

## Impact Explanation
**Medium — Unbounded gas consumption / Temporary freezing of funds.**

If the operator passes `firstExcludedIndex = nextUnusedNonce[asset]` after a large dust flood, the while loop iterates all entries and reverts out-of-gas. Even with a bounded `firstExcludedIndex`, the operator must drain all dust entries across many transactions before reaching legitimate requests, temporarily preventing those users from completing their withdrawals. Both "Unbounded gas consumption" and "Temporary freezing of funds" are explicitly listed as Medium impacts in the allowed scope.

## Likelihood Explanation
Any rsETH holder — an unprivileged, externally reachable role — can call `initiateWithdrawal` with 1-wei amounts. The only cost is gas per call and a negligible rsETH balance that is recovered after the withdrawal delay. No special role, coordination, or privileged access is required. The attack is cheap, repeatable, and requires no victim mistakes.

## Recommendation
1. Initialize `minRsEthAmountToWithdraw` to a sensible non-zero default (e.g., `1e15` wei) for each supported asset in `initialize()`, or require it to be set before an asset is usable for withdrawals.
2. Enforce the minimum strictly so that a zero value in the mapping is treated as "no minimum set" and rejects all withdrawals until configured:
   ```solidity
   if (rsETHUnstaked == 0 || minRsEthAmountToWithdraw[asset] == 0
       || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
       revert InvalidAmountToWithdraw();
   }
   ```
3. Consider adding a per-user cap on pending withdrawal requests to further limit queue flooding.

## Proof of Concept
1. Admin deploys `LRTWithdrawalManager` and supports asset `ETH`. `minRsEthAmountToWithdraw[ETH]` is `0` (never set in `initialize()`).
2. Attacker acquires 10,000 wei of rsETH (negligible cost).
3. Attacker calls `initiateWithdrawal(ETH, 1, "")` 10,000 times. Each call passes the check at line 162 (`1 != 0` and `1 < 0` is false). Each call appends a new entry to the FIFO queue via `_addUserWithdrawalRequest`.
4. Legitimate user calls `initiateWithdrawal(ETH, 1 ether, "")`. Their request lands at nonce 10,000.
5. Operator calls the unlock function with `firstExcludedIndex = nextUnusedNonce[ETH]` (10,001). The while loop at lines 790–815 iterates all 10,001 entries, each requiring multiple storage reads/writes, consuming gas proportional to the flood size and reverting out-of-gas on mainnet.
6. Even if the operator batches in chunks, the legitimate user's withdrawal is delayed until all 10,000 dust entries are drained, temporarily freezing their funds.

**Foundry test plan:** Deploy `LRTWithdrawalManager` on a local fork, call `initiateWithdrawal` with 1-wei amounts in a loop (e.g., 5,000 iterations), then call the unlock function with `firstExcludedIndex = nextUnusedNonce[asset]` and assert the transaction reverts with out-of-gas, or measure gas scaling linearly with the number of dust entries.
