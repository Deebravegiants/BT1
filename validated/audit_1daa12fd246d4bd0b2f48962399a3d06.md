### Title
`respond_verify_foreign_tx` Accepts Replayed `payload_hash` From Any Prior Response, Enabling Forged Foreign-Chain Verification - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash` using the MPC root public key. It never checks that `response.payload_hash` is the correct hash for the specific `request` being resolved. A Byzantine attested participant (below signing threshold) can replay a `(payload_hash, signature)` pair obtained from any prior legitimate `verify_foreign_transaction` response to resolve an entirely different pending request, delivering fabricated foreign-chain attestation data to the waiting caller.

### Finding Description

`respond_verify_foreign_tx` performs two checks before resolving all queued yields:

1. The ECDSA signature is valid over `response.payload_hash` under the MPC **root** public key (no tweak, no caller binding).
2. The `request` key exists in `pending_verify_foreign_tx_requests`.

It does **not** verify that `response.payload_hash` is the canonical hash of `ForeignTxSignPayload{request, extracted_values}` for the specific `request` being resolved.

```rust
// crates/contract/src/lib.rs ~726-733
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,          // root key — no tweak, no caller binding
)
.is_ok()
```

The node-side signing uses a **zero tweak** (`[0u8; 32]`), so every `verify_foreign_transaction` signature is produced under the same root key regardless of which transaction or caller originated the request:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs ~43
tweak: Tweak::new([0u8; 32]),
```

Because the root key is shared across all foreign-tx requests, any `(payload_hash, signature)` pair produced for request A is cryptographically indistinguishable from one produced for request B at the contract level. The contract has no way to tell them apart — it only checks "is this a valid root-key signature over *some* hash?"

Additionally, `VerifyForeignTransactionRequest` contains no caller binding (no `predecessor_id`, no derivation path, no tweak):

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs ~124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

This is confirmed by the test that explicitly queues two different callers (alice, bob) under the same map key.

### Impact Explanation

A Byzantine attested participant (one TEE node acting maliciously, below the signing threshold) can:

1. Call `verify_foreign_transaction(tx_X)` legitimately and receive `{payload_hash_X, sig_X}` — a valid root-key signature over the hash of `{tx_X, extracted_values_X}`.
2. Observe a victim's pending `verify_foreign_transaction(tx_Y)` on-chain.
3. Front-run the honest MPC leader by calling `respond_verify_foreign_tx(tx_Y, {payload_hash_X, sig_X})`.
4. The contract accepts: `sig_X` is a valid root-key signature over `payload_hash_X`, and `tx_Y` exists in the pending map. All queued yields for `tx_Y` are drained with the fabricated response.
5. The victim's bridge contract receives `{payload_hash_X, sig_X}` as the attestation for `tx_Y`.

If the bridge contract verifies the signature (which passes — it is a genuine MPC root-key signature) but does not independently recompute the expected `payload_hash` from `tx_Y`'s on-chain data, it will accept the forged attestation. This enables:

- **Cross-chain replay**: attestation data from one transaction is accepted as proof of a different transaction.
- **Invalid bridge execution / double-spend**: a bridge that gates fund release on a valid MPC attestation can be tricked into releasing funds for a transaction that was never confirmed, or for a transaction whose extracted values (e.g., block hash, log data) differ from what the bridge expects.

This matches the allowed High impact: *"forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

- **Entry path is reachable**: `respond_verify_foreign_tx` is a public contract method gated only on `assert_caller_is_attested_participant_and_protocol_active`. Any single Byzantine TEE participant satisfies this gate.
- **No threshold collusion required**: the attacker does not forge a new threshold signature; they replay an existing one. A single Byzantine participant below the signing threshold is sufficient.
- **Replay material is freely available**: any prior `verify_foreign_transaction` response (including one the attacker themselves requested) provides a valid `(payload_hash, signature)` pair. Responses are returned on-chain and are publicly observable.
- **Timing window is exploitable**: the attacker only needs to submit their `respond_verify_foreign_tx` call before the honest leader's transaction finalizes. NEAR's ~1-second block time and the node's `MAX_LATENCY_BEFORE_EXPECTING_TRANSACTION_TO_FINALIZE` (10 seconds) give the attacker a comfortable window.

### Recommendation

The contract must bind the accepted `payload_hash` to the specific `request` being resolved. Since the contract cannot independently query the foreign chain, the binding should be enforced structurally:

1. **Include the `request` hash inside the signed payload**: change `ForeignTxSignPayload` so that `compute_msg_hash()` commits to the `ForeignChainRpcRequest` (it already does via `ForeignTxSignPayloadV1.request`). Then, in `respond_verify_foreign_tx`, recompute the minimum expected prefix of `payload_hash` from `request` and reject any response whose `payload_hash` does not commit to the correct `request`. Because the contract cannot know `extracted_values`, it cannot recompute the full hash — but it can at minimum verify that the `request` portion of the Borsh-serialized payload matches.

2. **Alternatively, include a nonce / yield-resume `data_id` in the signed payload**: bind each signature to the specific yield it resolves, making cross-request replay impossible even if the attacker holds a valid prior signature.

3. **Short-term mitigation**: document clearly that bridge contracts **must** independently recompute the expected `payload_hash` from the transaction data and reject responses where it does not match, since the MPC contract cannot enforce this invariant itself.

### Proof of Concept

```
// Setup: attacker is an attested MPC participant.

// Step 1: attacker obtains a valid (payload_hash_X, sig_X) for Bitcoin tx X.
attacker.call(mpc_contract, "verify_foreign_transaction", {
    request: { Bitcoin: { tx_id: TX_X, confirmations: 6, extractors: [BlockHash] } },
    domain_id: FOREIGN_TX_DOMAIN,
    payload_version: V1,
}, deposit=1)
// → MPC nodes respond; attacker receives {payload_hash_X, sig_X}

// Step 2: victim submits a request for a different Bitcoin tx Y.
victim.call(mpc_contract, "verify_foreign_transaction", {
    request: { Bitcoin: { tx_id: TX_Y, confirmations: 6, extractors: [BlockHash] } },
    domain_id: FOREIGN_TX_DOMAIN,
    payload_version: V1,
}, deposit=1)
// → pending request for TX_Y is now in the contract map

// Step 3: attacker front-runs the honest MPC leader.
attacker.call(mpc_contract, "respond_verify_foreign_tx", {
    request: { Bitcoin: { tx_id: TX_Y, ... }, domain_id: ..., payload_version: V1 },
    response: { payload_hash: payload_hash_X, signature: sig_X }
})
// Contract checks:
//   ✓ attacker is attested participant
//   ✓ sig_X is valid over payload_hash_X under root key
