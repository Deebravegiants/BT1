### Title
MPC Nodes Sign Foreign-Chain Transactions at Reorg-Vulnerable Finality Levels, Enabling Bridge Double-Spend - (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

### Summary

The `verify_foreign_transaction` flow accepts `EvmFinality::Latest` (and analogously `StarknetFinality::AcceptedOnL2`, `BlockConfirmations(1)` for Bitcoin) as valid finality levels. MPC nodes perform a point-in-time canonicality check and then produce a threshold signature attesting to the extracted values. If a reorg occurs after signing, the signature remains cryptographically valid and on-chain accepted, but it attests to a foreign-chain transaction that no longer exists on the canonical chain. A bridge contract consuming this attestation would release funds for a deposit that was never finalized — a direct double-spend.

### Finding Description

The `EvmFinality` enum in `crates/near-mpc-contract-interface/src/types/foreign_chain.rs` exposes `Latest` as a first-class finality option:

```rust
pub enum EvmFinality {
    Latest,
    Safe,
    Finalized,
}
``` [1](#0-0) 

The `EvmInspector::verify_finality_level` in `crates/foreign-chain-inspector/src/evm/inspector.rs` maps `Latest` directly to `FinalityTag::Latest` and passes the check as long as `head.number >= receipt_block_number` — trivially true for the latest block:

```rust
let finality_tag = match finality {
    EthereumFinality::Finalized => FinalityTag::Finalized,
    EthereumFinality::Safe => FinalityTag::Safe,
    EthereumFinality::Latest => FinalityTag::Latest,
};
``` [2](#0-1) 

The `verify_block_is_canonical` call that follows is also a point-in-time check — it re-fetches the canonical block at the receipt's height and compares hashes: [3](#0-2) 

Both checks pass at the moment of the RPC call. The MPC nodes then sign `SHA-256(borsh(ForeignTxSignPayload{request, values}))` and submit `respond_verify_foreign_tx` to the contract.

The contract's `respond_verify_foreign_tx` only verifies that the ECDSA signature is valid against the MPC public key and that a pending yield exists for the request key. It does **not** re-verify foreign-chain state and has no mechanism to invalidate a signature after a reorg: [4](#0-3) 

The signed `ForeignTxSignPayload` is constructed from the request and the extracted values (e.g., block hash, logs): [5](#0-4) 

Once signed, this payload is permanently valid. No on-chain mechanism revokes it if the attested block is later reorged out.

The same structural issue applies to:
- `StarknetFinality::AcceptedOnL2` (Starknet L2 blocks before L1 acceptance can be reorged)
- `BlockConfirmations(1)` for Bitcoin (user-controlled, no contract-enforced minimum) [6](#0-5) 

### Impact Explanation

A bridge contract using `verify_foreign_transaction` with `EvmFinality::Latest` to gate inbound deposits (foreign chain → NEAR) receives a valid MPC threshold signature attesting to a deposit transaction. If that transaction is subsequently reorged out of the canonical chain, the signature remains valid and the bridge has no way to detect the reorg. The bridge releases NEAR-side assets for a deposit that never finalized on the foreign chain — a direct theft of bridge funds. This matches the allowed impact: **"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass … that causes invalid bridge execution or double-spend conditions."**

### Likelihood Explanation

- `EvmFinality::Latest` is explicitly supported in the production contract interface and is a valid user-supplied parameter — no privileged access required.
- 1–2 block reorgs on EVM chains (BNB, Polygon, Arbitrum, Base, Abstract, HyperEVM) occur naturally and regularly; the attacker does not need to cause the reorg.
- The attacker only needs to time the `verify_foreign_transaction` call to land while their transaction is in the latest block, then wait for a natural reorg or use their own validator influence on chains with weaker finality guarantees.
- The contract enforces no minimum finality level, so any caller can submit with `Latest`.

### Recommendation

1. **Remove `EvmFinality::Latest` from the accepted finality options** in `verify_foreign_transaction`. Only `Safe` and `Finalized` should be accepted for EVM chains, as these correspond to checkpointed or finalized states that cannot be reorged under normal conditions.
2. **Enforce a contract-level minimum finality policy** per chain in the on-chain foreign-chain configuration (e.g., `min_finality: EvmFinality::Safe`), so operators cannot accidentally configure a bridge with insufficient finality.
3. **For Bitcoin, enforce a minimum confirmation count** (e.g., ≥ 6) in the on-chain chain configuration rather than trusting the user-supplied `confirmations` field.
4. **For Starknet, reject `AcceptedOnL2`** and require `AcceptedOnL1` for bridge-critical use cases.

### Proof of Concept

1. Operator deploys a bridge contract that calls `verify_foreign_transaction` with `EvmFinality::Latest` to gate inbound BNB deposits.
2. Attacker sends a transaction on BNB depositing 10,000 USDT to the bridge address.
3. Attacker immediately calls `verify_foreign_transaction` with `finality: EvmFinality::Latest` while the transaction is in the latest block.
4. MPC nodes call `eth_getTransactionReceipt` → receipt found; `eth_getBlockByNumber("latest")` → head ≥ receipt block; `eth_getBlockByNumber(receipt_block_number)` → hashes match. All checks pass.
5. MPC nodes sign `SHA-256(borsh(ForeignTxSignPayload{request, [BlockHash(0xABCD...)]}))` and submit `respond_verify_foreign_tx`.
6. A 1-block reorg occurs (naturally, as is common on BNB), removing the attacker's transaction. The canonical block at that height now has a different hash.
7. The attacker presents the valid MPC signature to the bridge contract. The bridge verifies the signature (valid), checks the extracted log (shows 10,000 USDT deposit), and releases 10,000 USDT equivalent on NEAR.
8. The attacker has received 10,000 USDT on NEAR for a deposit that no longer exists on BNB.

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
