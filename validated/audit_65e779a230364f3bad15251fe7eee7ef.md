### Title
Single Byzantine Node Can Deliver Unverified CKD Response for `AppPublicKey` Requests - (File: `crates/contract/src/lib.rs`)

### Summary
The `respond_ckd` function conditionally skips cryptographic output verification based on the `CKDAppPublicKey` variant. When a request uses `AppPublicKey`, the `ckd_output_check` gate is bypassed entirely while `resolve_yields_for` executes unconditionally. A single Byzantine participant (strictly below the signing threshold) can submit an arbitrary `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will accept and deliver it to all waiting callers without any cryptographic validation.

### Finding Description
In `respond_ckd`, the output check is conditional on the request variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // no check â€” empty arm
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(                     // always executes
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies that the response is cryptographically consistent with the master public key and the app's public key, ensuring threshold participation produced the result. For `AppPublicKey`, this gate is absent. The `resolve_yields_for` call that delivers the response to all queued callers executes unconditionally in both branches.

The structural analog to the external report is exact: just as `shrine.melt` (the killed-state guard) is skipped when `can_absorb_any` is false while `shrine.redistribute` (the harmful operation) still executes, here `ckd_output_check` (the threshold-authorization guard) is skipped when `AppPublicKey` is used while `resolve_yields_for` (the delivery operation) still executes.

The `AppPublicKey` variant is a fully supported, documented code path. The `request_app_private_key` entry point accepts it without restriction:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}
    dtos::CKDAppPublicKey::AppPublicKeyPV(pk) => {
        if !app_public_key_check(pk) { env::panic_str("app public key check failed") }
    }
}
``` [2](#0-1)
