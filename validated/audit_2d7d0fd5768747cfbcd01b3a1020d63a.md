Audit Report

## Title
`_checkAndUpdateDailyFeeMintLimit()` Reverts When `maxFeeMintAmountPerDay` Is Zero and Protocol Fees Are Due — (File: `contracts/LRTOracle.sol`)

## Summary

`LRTOracle._checkAndUpdateDailyFeeMintLimit` lacks the zero-guard present in `remainingDailyFeeMintLimit`, causing every call to `updateRSETHPrice()` to revert with `DailyFeeMintLimitExceeded` whenever `maxFeeMintAmountPerDay == 0` and TVL has grown. Since `maxFeeMintAmountPerDay` is never set in `initialize` or `reinitialize`, any deployment or upgrade that omits a subsequent `setMaxFeeMintAmountPerDay` call enters this broken state. Protocol fees accrued during this window are permanently unrecoverable.

## Finding Description

`maxFeeMintAmountPerDay` is declared as a storage variable with Solidity's default value of `0`: [1](#0-0) 

Neither `initialize` nor `reinitialize` sets it: [2](#0-1) [3](#0-2) 

`_checkAndUpdateDailyFeeMintLimit` performs a strict greater-than check with no zero-guard: [4](#0-3) 

When `maxFeeMintAmountPerDay == 0` and `feeAmount > 0`, the condition `feeAmount > 0` is trivially true, so the function always reverts. By contrast, `remainingDailyFeeMintLimit` correctly short-circuits: [5](#0-4) 

`feeAmount > 0` occurs whenever TVL has grown and the protocol is not paused: [6](#0-5) 

The full call chain is:

```
updateRSETHPrice() [public, permissionless]
  → _updateRsETHPrice()
      → protocolFeeInETH > 0 branch (L299)
          → _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)  ← reverts
``` [7](#0-6) 

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Every call to the public `updateRSETHPrice()` reverts while `maxFeeMintAmountPerDay == 0` and TVL is growing. The protocol fee (`protocolFeeInBPS` of all yield) is never minted to the treasury. Once `setMaxFeeMintAmountPerDay` is eventually called, the fee accrued during the broken window is not retroactively minted — it is permanently lost. This matches the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation

`maxFeeMintAmountPerDay` defaults to `0` and is absent from both `initialize` and `reinitialize`. Any deployment or upgrade that does not immediately call `setMaxFeeMintAmountPerDay` is in this state. The trigger requires no privilege: any external caller invoking the public `updateRSETHPrice()` after TVL has increased will hit the revert. No malicious intent is required.

## Recommendation

Add a zero-guard in `_checkAndUpdateDailyFeeMintLimit` consistent with the existing guard in `remainingDailyFeeMintLimit`:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
+   if (maxFeeMintAmountPerDay == 0) return; // limit not configured; skip enforcement
    ...
}
```

Alternatively, require `maxFeeMintAmountPerDay` to be set to a non-zero value inside `reinitialize` so the invariant is enforced at upgrade time.

## Proof of Concept

1. Deploy `LRTOracle` and call `reinitialize` without subsequently calling `setMaxFeeMintAmountPerDay`. `maxFeeMintAmountPerDay` remains `0`.
2. Users deposit assets into `LRTDepositPool`, increasing TVL above `rsETHPrice × rsETHSupply`.
3. Any caller invokes `updateRSETHPrice()`.
4. Inside `_updateRsETHPrice`, `totalETHInProtocol > previousTVL` → `protocolFeeInETH > 0` (L244–247).
5. `rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice) > 0`.
6. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` evaluates `0 + rsethAmountToMintAsProtocolFee > 0` → reverts with `DailyFeeMintLimitExceeded(rsethAmountToMintAsProtocolFee, 0)`.
7. `updateRSETHPrice()` reverts on every call. Protocol fee is never minted. Yield accrued during this window is permanently unrecoverable once `maxFeeMintAmountPerDay` is later set to a non-zero value.

**Foundry test sketch:**
```solidity
