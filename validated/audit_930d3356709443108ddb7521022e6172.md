Audit Report

## Title
Stale `rsETHPrice` During `pricePercentageLimit` Guard Revert Enables Excess wrsETH Minting on L2 — (File: `contracts/LRTOracle.sol`)

## Summary

When `_updateRsETHPrice()` computes a `newRsETHPrice` that exceeds `highestRsethPrice` by more than `pricePercentageLimit`, non-manager callers of `updateRSETHPrice()` receive a `PriceAboveDailyThreshold` revert and `rsETHPrice` is never updated. The stale (lower) price is then broadcast via `RSETHRateProvider` to L2, where `RSETHPoolV3.deposit()` uses it to compute wrsETH amounts, minting excess wrsETH to new depositors and permanently diluting existing holders' accrued yield.

## Finding Description

In `LRTOracle._updateRsETHPrice()`, lines 252–266, after computing `newRsETHPrice`, the following guard runs:

```solidity
if (newRsETHPrice > highestRsethPrice) {
    uint256 priceDifference = newRsETHPrice - highestRsethPrice;
    bool isPriceIncreaseOffLimit =
        pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
    if (isPriceIncreaseOffLimit) {
        if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
            revert PriceAboveDailyThreshold();
        }
    }
}
``` [1](#0-0) 

The revert causes the function to exit before reaching `rsETHPrice = newRsETHPrice` at line 313, leaving `rsETHPrice` at its previous (lower) value. [2](#0-1) 

The public entry point `updateRSETHPrice()` is gated only by `whenNotPaused`; any keeper or EOA can call it but will revert whenever the price appreciation exceeds the limit. [3](#0-2) 

The only escape hatch is `updateRSETHPriceAsManager()`, restricted to `MANAGER`. [4](#0-3) 

The stale `rsETHPrice` is then read directly by the cross-chain rate provider: [5](#0-4) 

On L2, `RSETHPoolV3.viewSwapRsETHAmountAndFee()` uses this stale rate:

```solidity
uint256 rsETHToETHrate = getRate();          // stale, lower than actual
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [6](#0-5) 

A stale (lower) `rsETHToETHrate` inflates `rsETHAmount`. The excess wrsETH minted to new depositors permanently dilutes the share of underlying ETH belonging to existing holders.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders have accrued yield reflected in the true (higher) `newRsETHPrice`. When new depositors on L2 mint wrsETH at the stale (lower) rate, they receive more wrsETH per ETH than the true exchange rate warrants. Once the oracle is corrected by the manager, the inflated wrsETH supply is permanent: each existing holder's wrsETH now represents a smaller fraction of the underlying ETH pool. The excess wrsETH already minted cannot be clawed back, making the yield dilution irreversible.

## Likelihood Explanation

No attacker action is required. The condition is triggered by normal protocol operation: a sufficiently large EigenLayer reward distribution or TVL increase that causes `newRsETHPrice` to exceed `highestRsethPrice` by more than `pricePercentageLimit`. A conservatively small `pricePercentageLimit` (e.g., 0.5%–1%) makes this scenario plausible during routine reward accrual cycles. The staleness window persists until the manager manually calls `updateRSETHPriceAsManager()`. Any L2 deposits processed during this window are affected. The condition is repeatable every reward cycle.

## Recommendation

1. **Cap instead of revert**: When `isPriceIncreaseOffLimit` is true for a non-manager caller, update `rsETHPrice` to `highestRsethPrice + pricePercentageLimit.mulWad(highestRsethPrice)` (the maximum allowed increment) rather than reverting. Emit an event signaling that the full update is pending manager confirmation.
2. **Add a staleness heartbeat check** in `RSETHRateProvider.getLatestRate()` that reverts or returns a sentinel value if `rsETHPrice` has not been updated within a configurable window, preventing L2 pools from consuming a stale rate.
3. **Emit an anomaly event** on the revert path so off-chain monitoring can alert the manager immediately.

## Proof of Concept

1. Protocol state: `highestRsethPrice = 1.05e18`, `pricePercentageLimit = 1e16` (1%).
2. EigenLayer distributes rewards; `_getTotalEthInProtocol()` yields `newRsETHPrice = 1.062e18` (1.14% increase, above the 1% limit).
3. Keeper calls `updateRSETHPrice()`. At line 257, `isPriceIncreaseOffLimit = true`. Keeper is not manager → revert `PriceAboveDailyThreshold`. `rsETHPrice` stays at `1.05e18`.
4. `RSETHRateProvider.getLatestRate()` returns stale `1.05e18`; this rate is broadcast to L2 via LayerZero.
5. Depositor on L2 sends 1 ETH. `viewSwapRsETHAmountAndFee(1e18)` computes `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.9524 wrsETH` instead of the correct `1e18 / 1.062e18 ≈ 0.9416 wrsETH`. Excess: ~0.0108 wrsETH per ETH deposited.
6. Manager eventually calls `updateRSETHPriceAsManager()`; oracle corrects to `1.062e18`. The excess wrsETH already minted is permanent — existing holders' share of underlying ETH is permanently diluted by the amount of the excess.

**Foundry fork test outline:**
- Fork mainnet; deploy/configure `LRTOracle` with `pricePercentageLimit = 1e16`.
- Simulate reward accrual by increasing `getTotalAssetDeposits` return value to push `newRsETHPrice` above the 1% threshold.
- Call `updateRSETHPrice()` from a non-manager EOA; assert revert `PriceAboveDailyThreshold`.
- Assert `rsETHPrice` unchanged.
- Simulate L2 deposit via `RSETHPoolV3.deposit{value: 1 ether}(...)` using the stale rate.
- Assert minted wrsETH amount exceeds `1e18 * 1e18 / 1.062e18`.
- Call `updateRSETHPriceAsManager()`; assert oracle corrects.
- Assert the previously minted excess wrsETH remains, confirming permanent dilution.
