Audit Report

## Title
`MerkleDistributor.claim()` Allows Third-Party Fee-Bearing Claims Without Caller Authorization - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` accepts an arbitrary `account` parameter with no check that `msg.sender == account`. Any caller with access to a victim's public merkle proof can trigger the claim on their behalf, causing the victim to permanently receive `claimableAmount - fee` tokens instead of their full entitlement. The fee (up to 10%) is irrecoverably diverted to `protocolTreasury` because the claim state is finalized at `cumulativeAmount`.

## Finding Description
`MerkleDistributor.claim()` at L97–106 accepts `account` as a caller-supplied parameter and performs no ownership check before executing the claim. The merkle proof is verified against the supplied `account` value, not `msg.sender`, so any party who obtains the victim's `(index, account, cumulativeAmount, merkleProof)` tuple — necessarily published off-chain for a merkle distributor to function — can submit the transaction. [1](#0-0) 

After proof verification, the contract computes `fee = (claimableAmount * feeInBPS) / 10_000` and transfers only `amountToSend = claimableAmount - fee` to `account`, routing the fee to `protocolTreasury`. [2](#0-1) 

The victim's `userClaims` state is then updated to `cumulativeAmount`, permanently marking the claim as consumed and blocking any future re-claim. [3](#0-2) 

`KernelMerkleDistributor._processClaim()` demonstrates the protocol is aware of this requirement and explicitly enforces it in the newer distributor: [4](#0-3) 

`MerkleDistributor` has no equivalent guard, no `Unauthorized` error defined in its interface, and no delegation/allowance mechanism that would make third-party claims a legitimate design choice.

## Impact Explanation
**High. Theft of unclaimed yield.**

When `feeInBPS > 0` (configurable up to `MAX_FEE_IN_BPS = 1000`, i.e., 10%), an attacker forces any victim's claim, causing the victim to permanently lose up to 10% of their entitled token allocation. The loss is irrecoverable: `userClaims[account].cumulativeAmount` is set to the full `cumulativeAmount`, so the victim cannot re-claim the deducted fee. The stolen yield is redirected to the treasury. [5](#0-4) 

## Likelihood Explanation
**Medium.** The only preconditions are that `feeInBPS > 0` (a normal operating state within the contract's configured range) and that the victim has not yet claimed. Merkle proof data for all eligible accounts is necessarily published publicly so users can self-claim. Any unprivileged external caller can read this data and replay it. The attack is repeatable across all unclaimed accounts in a single block.

## Recommendation
Add an ownership check at the start of `claim()`, mirroring `KernelMerkleDistributor._processClaim()`:

```solidity
if (account != msg.sender) revert Unauthorized();
```

The `Unauthorized` error must also be added to the `IMerkleDistributor` interface in `MerkleDistributor.sol`.

## Proof of Concept
1. Deploy `MerkleDistributor` with `feeInBPS = 500` (5%).
2. Owner calls `setMerkleRoot(root)` where the tree includes Alice at `index=1` with `cumulativeAmount = 1000e18`.
3. Merkle proof data is published off-chain; attacker reads Alice's `(1, alice, 1000e18, aliceProof)`.
4. Attacker calls `MerkleDistributor.claim(1, alice, 1000e18, aliceProof)` from their own EOA.
5. Contract computes `fee = 50e18`, transfers `950e18` to Alice and `50e18` to treasury.
6. `userClaims[alice].cumulativeAmount = 1000e18` — Alice can never reclaim the 50e18.
7. Alice permanently loses 50e18 tokens she was entitled to.

**Foundry test sketch:**
```solidity
function test_forcedClaimDeductsFee() public {
    vm.prank(attacker);
    distributor.claim(1, alice, 1000e18, aliceProof);
    assertEq(token.balanceOf(alice), 950e18);       // victim receives less
    assertEq(token.balanceOf(treasury), 50e18);     // fee stolen
    assertTrue(distributor.isClaimed(1, alice));    // permanently consumed
}
```
