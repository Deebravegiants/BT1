### Title
Caller-Agnostic Request Key in `verify_foreign_transaction` Enables Queue-Filling DoS and Signature Theft - (File: `crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint stores pending requests under a key (`VerifyForeignTransactionRequest`) that does **not** include the caller's identity (`predecessor_id`). Any unprivileged caller can submit `MAX_PENDING_REQUEST_FAN_OUT` identical requests for a target foreign transaction, filling the bounded fan-out queue before the legitimate user. The legitimate user's call is then rejected with `PendingRequestQueueFull`, while the attacker's queued yields receive the MPC-produced signature — which can be used to claim bridged funds on NEAR.

---

### Finding Description

`SignatureRequest` binds the caller's identity into its map key via tweak derivation from `predecessor_id` and `path`:

```rust
// crates/near-mpc-contract-interface/src/types/sign.rs
impl SignatureRequest {
    pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
        let tweak = crate::kdf::derive_tweak(predecessor_id, path);
        SignatureRequest { domain_id: domain, tweak, payload }
    }
}
``` [1](#0-0) 

`VerifyForeignTransactionRequest`, by contrast, contains only the raw RPC request, domain ID, and payload version — **no caller identity**:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [2](#0-1) 

The contract stores pending yields under this caller-agnostic key:

```rust
pending_verify_foreign_tx_requests: LookupMap<VerifyForeignTransactionRequest, Vec<YieldIndex>>,
``` [3](#0-2) 

The `verify_foreign_transaction` handler converts the user-supplied args to this key without injecting `predecessor_id`:

```rust
let request = args_into_verify_foreign_tx_request(request);
...
move |this, id| this.add_verify_foreign_tx_request(request, id),
``` [4](#0-3) 

The fan-out queue is bounded by `MAX_PENDING_REQUEST_FAN_OUT`. Once full, any further submission for the same key panics with `PendingRequestQueueFull`: [5](#0-4) 

The developers themselves acknowledge the caller-agnostic nature in a test comment:

> "caller bob submits the identical request — a different account would today be blocked from receiving a response by alice's submission" [6](#0-5) 

---

### Impact Explanation

**Critical — verified foreign-chain flow theft.**

In a bridge scenario (the primary use case for `verify_foreign_transaction`):
1. A user deposits funds on a foreign chain (e.g., Bitcoin tx_id = X).
2. The user intends to call `verify_foreign_transaction` to obtain an MPC signature proving the deposit, then use that signature to claim equivalent tokens on NEAR.
3. An attacker who observes the foreign-chain deposit submits `MAX_PENDING_REQUEST_FAN_OUT` identical `verify_foreign_transaction` calls for the same tx_id before the legitimate user.
4. The attacker's yields fill the queue. The legitimate user's call is rejected with `PendingRequestQueueFull`.
5. The MPC network processes the request and fans the signature out to all queued yields — all of which belong to the attacker.
6. The attacker uses the received signature to claim the bridged funds on NEAR that were intended for the legitimate user.

The cost of the attack is `MAX_PENDING_REQUEST_FAN_OUT × 1 yoctonear` — effectively zero.

---

### Likelihood Explanation

**High.** Foreign-chain deposits are publicly observable on-chain. An attacker running a bot that monitors the foreign chain and immediately submits the maximum number of `verify_foreign_transaction` calls for any new deposit can reliably frontrun legitimate users. The 1-yoctonear deposit per call makes the attack economically free. No privileged access, TEE bypass, or threshold collusion is required.

---

### Recommendation

Include the caller's identity in the `VerifyForeignTransactionRequest` key, mirroring the `sign()` design. The simplest approach is to add `predecessor_id: AccountId` to `VerifyForeignTransactionRequest` and populate it from `env::predecessor_account_id()` inside `verify_foreign_transaction`, exactly as `SignatureRequest` binds the caller via tweak derivation. This ensures each caller's pending slot is isolated and cannot be pre-filled by an adversary.

---

### Proof of Concept

```
1. Attacker observes Bitcoin tx_id = [7u8; 32] depositing 1 BTC intended for alice.near.

2. Attacker submits MAX_PENDING_REQUEST_FAN_OUT calls to verify_foreign_transaction
   with { request: Bitcoin { tx_id: [7u8;32], ... }, domain_id: X, payload_version: V1 }
   from attacker-controlled accounts, each paying 1 yoctonear.

3. alice.near submits the same verify_foreign_transaction call.
   → Contract panics: "Pending-request queue is full for this request key"

4. MPC nodes process the request; the signature is fanned out to all
   MAX_PENDING_REQUEST_FAN_OUT attacker yields.

5. Attacker's contract receives the MPC signature and calls the bridge contract
   to claim the 1 BTC equivalent on NEAR, stealing alice's deposit.
```

The unit test at `crates/contract/src/lib.rs:3208` already demonstrates that two different callers (alice, bob) share the same queue slot under the caller-agnostic key, confirming the root cause is present in production code. [7](#0-6)

    #[test]
    fn add_signature_request__should_panic_when_pending_queue_is_full() {
        // Given: a contract with a queue already at the fan-out cap for some request key.
        let (context, mut contract, _) = basic_setup(Curve::Secp256k1, &mut OsRng);
        let signature_request = SignatureRequest::new(
            DomainId::default(),
            Payload::from_legacy_ecdsa([3u8; 32]),
            &context.predecessor_account_id,
            "m/44'\''/60'\''/0'\''/0/0",
        );
        for i in 0..MAX_PENDING_REQUEST_FAN_OUT {
            contract.add_signature_request(signature_request.clone(), [i; 32]);
        }
        assert_eq!(
            contract
                .pending_signature_requests
                .get(&signature_request)
                .map(|q| q.len()),
            Some(usize::from(MAX_PENDING_REQUEST_FAN_OUT)),
        );

        // When: one more append is attempted.
        let result = panic::catch_unwind(panic::AssertUnwindSafe(|| {
            contract.add_signature_request(signature_request.clone(), [0xff; 32]);
        }));

        // Then: it panics with the typed cap-exceeded error and leaves the queue untouched.
        let err = result.expect_err("appending past the cap should panic");
        let msg = err
            .downcast_ref::<String>()
            .map(String::as_str)
            .or_else(|| err.downcast_ref::<&str>().copied())
            .unwrap_or_default();
        assert!(
            msg.contains("Pending-request queue is full"),
            "unexpected panic message: {msg}",
        );
        assert_eq!(
            contract
                .pending_signature_requests
                .get(&signature_request)
                .map(|q| q.len()),
            Some(usize::from(MAX_PENDING_REQUEST_FAN_OUT)),
            "queue should not have grown past the cap",
        );
