Looking at the `set_state_header` function in detail, I need to trace whether `chunk.shard_id()` is ever compared to the `shard_id` parameter, and whether the Merkle proof check is positional.

### Title
Missing `chunk.shard_id()` vs `shard_id` Parameter Check in `set_state_header` Allows Peer to Store Wrong Chunk for a Shard — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer and validates it before storing it. The function never compares `chunk.shard_id()` (the shard encoded inside the chunk) to the `shard_id` function parameter (the shard being synced). The Merkle proof check at step 3a only proves the chunk is *somewhere* in the block's `chunk_headers_root` tree — it does not prove the chunk is at the correct shard index. A malicious peer can supply chunk_A (shard A's chunk) with chunk_A's valid positional proof while claiming `shard_id=B`, pass all validation steps, and cause the node to store shard A's chunk as shard B's state sync header.

---

### Finding Description

**Confirmed missing check:** A `grep` for `chunk.shard_id()` in `adapter.rs` returns zero matches inside `set_state_header`. The function parameter `shard_id` and the chunk's internal `shard_id()` are never compared.

**Step 3a — `verify_path` is not a positional shard check:** [1](#0-0) 

`verify_path` (merkle.rs lines 113–118) computes `compute_root_from_path(path, hash(item)) == root`. The `MerklePath` encodes position implicitly through sibling hashes and `Direction::Left/Right` values, but the code does not verify that the path corresponds to shard B's index. Providing chunk_A with chunk_A's own valid proof path (index 0) against the same `chunk_headers_root` passes this check, even when `shard_id=B` (index 1). [2](#0-1) 

**Step 4e — receipt hash uses the parameter `shard_id`, not `chunk.shard_id()`:** [3](#0-2) 

`CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` uses the attacker-supplied `shard_id=B`. The attacker provides the actual on-chain receipts destined for shard B, so this hash matches the real leaf in the outgoing receipts Merkle tree for shard B. Steps 4e and 4f pass with legitimate chain data.

**Header stored under wrong key:** [4](#0-3) 

The header (containing chunk_A) is stored under `StateHeaderKey(shard_id=B, sync_hash)`.

**`set_state_finalize` applies chunk_A's data to shard B's trie:** [5](#0-4) [6](#0-5) 

`shard_uid` is derived from `shard_id=B`, but `chunk` is chunk_A. The runtime applies chunk_A's transactions and receipts to shard B's state trie, producing a wrong state root for shard B.

**Server-side construction for comparison:** The legitimate server uses `prev_shard_index` (derived from `shard_id`) to select the correct chunk from the block: [7](#0-6) 

The client-side validation in `set_state_header` has no equivalent index-to-shard binding check.

---

### Impact Explanation

A malicious peer responding to state sync requests can corrupt a syncing node's state for any shard it is syncing. The node stores shard A's chunk as shard B's state sync header, then applies shard A's transactions and state root to shard B's trie. After `set_state_finalize`, the node holds an incorrect state root for shard B. Subsequent block processing for shard B will produce wrong state roots, causing the node to reject valid blocks or produce invalid chunk state witnesses, effectively disabling the node for that shard.

---

### Likelihood Explanation

Any peer on the network can respond to state sync header requests. The attack requires only public chain data: chunk_A's hash, its valid Merkle proof (obtainable from any full node), and the legitimate receipts for shard B. No validator privileges, key material, or cryptographic forgery is required. The attack is triggered whenever a node enters state sync for shard B and a malicious peer responds first.

---

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, verify that the `chunk_proof` corresponds to the correct shard index by checking the proof's positional encoding against `shard_layout.get_shard_index(shard_id)`.

---

### Proof of Concept

```
1. Victim node enters state sync for shard B (shard_id=1), sync_hash=H.
2. Attacker constructs ShardStateSyncResponseHeader:
   - chunk = chunk_A (shard_id()=0, genuinely included in sync_prev_block at index 0)
   - chunk_proof = chunk_A's valid Merkle proof (index 0 path against chunk_headers_root)
   - incoming_receipts_proofs = actual on-chain receipts for shard B
   - root_proofs = actual on-chain outgoing receipt proofs for shard B
3. Victim calls set_state_header(shard_id=1, sync_hash=H, header).
4. validate_chunk_proofs(chunk_A) → passes (chunk_A is internally consistent).
5. verify_path(chunk_headers_root, chunk_A_proof, ChunkHashHeight(chunk_A.hash, ...)) → passes
   (chunk_A IS in the tree at index 0; no check that index 0 == shard_id=1's index).
6. Receipt proofs verified with shard_id=1 → passes (legitimate receipts for shard 1).
7. Header stored under StateHeaderKey(shard_id=1, H) containing chunk_A.
8. set_state_finalize(shard_id=1, H):
   - shard_uid = uid_for(shard_id=1)
   - chunk = chunk_A (shard 0's chunk)
   - apply_chunk(shard_uid=shard_1_uid, chunk_A's txs/receipts) → wrong state root for shard 1.
```
