### Title
Single Byzantine Participant Can Block All Foreign-Chain Verification via Empty Chain Registration — (File: `crates/contract/src/lib.rs`)

### Summary
`get_supported_foreign_chains()` computes the **strict intersection** of every active participant's registered foreign chains. A single Byzantine participant (strictly below the signing threshold) can register an empty chain list, collapsing the intersection to the empty set and permanently blocking every `verify_foreign_transaction` request until governance removes or re-registers that participant.

### Finding Description

The production contract's `get_supported_foreign_chains()` uses an all-participant intersection rule: a foreign chain is "supported" only if **every** active participant has registered it. The `verify_foreign_transaction` entry point rejects any request whose target chain is absent from this set.

This is structurally identical to the H-12 Astaria finding. In Astaria, each lien's `liquidationInitialAsk` is checked against cumulative debt for every stack position, but only position 0's value actually governs the auction — so any lien can set its value to block future borrows. Here, each participant's chain registration is applied as a universal gate, but only a **threshold-many** participants need to cover a chain for it to be serviceable. The check is over-constrained relative to the actual security requirement, and any single participant can exploit the gap.

The project's own design document acknowledges this directly:

> "Today, `get_supported_foreign_chains()` returns the strict intersection of every active participant's registered chains, and `verify_foreign_transaction` rejects any request whose target chain is not in it. **A single node that registers an empty list (or hasn't registered yet) drops every chain — one operator can take the whole feature down.** That is what this proposal fixes." ### Impact Explanation

All `verify_foreign_transaction` calls are rejected for every foreign chain. Users cannot execute any foreign-chain transaction through the MPC network. This is a request-lifecycle manipulation that breaks the production safety invariant — foreign-chain verification must remain available whenever a signing threshold of honest participants are online — without requiring network-level DoS or operator misconfiguration. The attacker needs only to register an empty chain list; no collusion, no threshold-level cooperation, and no privileged access beyond being a registered participant is required.

**Impact: Medium** — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants.

### Likelihood Explanation

Any single registered participant (one of N, strictly below the signing threshold) can trigger this unilaterally by submitting an empty foreign-chain registration. The action is cheap, reversible by the attacker at will (re-register to restore, then empty again), and requires no coordination. Removing the attacker requires a threshold-level governance vote, giving the attacker a persistent lever.

### Recommendation

Replace the strict all-participant intersection with a threshold-based availability check: a chain is "available" if at least `signing_threshold` active participants have registered it (`available ⊆ whitelisted` always). This is exactly the fix proposed in `docs/design/calculating-supported-foreign-chains.md`: [5](#0-4) 

Concretely:
- Introduce `get_whitelisted_foreign_chains()` (governance-voted, immutable by any single node).
- Introduce `get_available_foreign_chains()` (dynamic, requires ≥ `signing_threshold` participants to cover a chain).
- Gate `verify_foreign_transaction` on `available`, not on the all-participant intersection.

### Proof of Concept

1. Participant **P** (one of N registered participants, below signing threshold) calls `register_available_foreign_chain_config` with an empty chain list.
2. `get_supported_foreign_chains()` computes the intersection of all participants' registered chains; P's empty set makes the result empty.
3. Any user calling `verify_foreign_transaction` for any chain receives a rejection — no chain passes the supported-set check.
4. The entire foreign-chain verification feature is blocked.
5. Restoring service requires either P to voluntarily re-register (attacker controls this) or a threshold-level governance vote to remove P — a slow, expensive operation that the attacker can race against by re-emptying their registration.

```markdown
## Proposal: two sets of chains

> **Terms** (whitelisted, available, RPC quorum, signing threshold, *covers*) are defined in
> [Foreign Chain Transaction Verification Design — Terminology](../foreign-chain-transactions.md#terminology).

The network distinguishes the **whitelisted** set (vote-driven policy, `get_whitelisted_foreign_chains()`)
from the **available** set (servable right now, `get_available_foreign_chains()`):

- **Whitelisted** is derived purely from the on-chain RPC whitelist — **no per-node input can add or
  remove a chain**, so no single operator can change it.
- **Available** is computed dynamically from the per-node config reports: `C` is available iff
  ≥ `signing_threshold` active participants cover `C`. `available ⊆ whitelisted` always.

`verify_foreign_transaction(C)` is **rejected unless `C` is available**: the contract fails fast
instead of accepting a request that can't reach the signing threshold and letting it time out. The
rejection is temporary — `C` becomes serviceable again as soon as enough nodes report coverage.

The legacy `get_supported_foreign_chains()` (the intersection rule) is **to be deprecated** in favour
of the two views above.
```
