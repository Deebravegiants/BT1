### Title
Hardcoded Reference-Script Size Limits and Fee Parameters in Conway Era Cannot Be Updated via Governance, Enabling Permanent Fund Freezing - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/PParams.hs`)

---

### Summary

In the Conway era, four critical reference-script parameters — the per-transaction size cap, the per-block size cap, the fee-tier multiplier, and the fee-tier stride — are implemented as **compile-time constants** that are completely invisible to the governance `ParameterChange` mechanism. Any UTxO whose spending path requires referencing more than 200 KiB of scripts is permanently unspendable in Conway era; recovery requires a hard fork (era transition to Dijkstra).

---

### Finding Description

The `ConwayEraPParams` type class exposes four getters that the fee and size-limit rules consume at validation time:

```
ppMaxRefScriptSizePerTxG
ppMaxRefScriptSizePerBlockG
ppRefScriptCostMultiplierG
ppRefScriptCostStrideG
```

The Conway-era instance of this class implements all four as `L.to . const <hardcoded_value>` — a Haskell lens idiom that **ignores the `PParams` record entirely** and always returns the baked-in literal:

```haskell
ppMaxRefScriptSizePerTxG    = L.to . const $ 200 * 1024          -- 204 800 bytes
ppMaxRefScriptSizePerBlockG = L.to . const $ 1024 * 1024         -- 1 048 576 bytes
ppRefScriptCostMultiplierG  = L.to . const . fromJust $ boundRational 1.2
ppRefScriptCostStrideG      = L.to . const $ knownNonZeroBounded @25_600
``` [1](#0-0) 

Because these are constant getters, no on-chain governance `ParameterChange` proposal can ever alter them in Conway era. The ADR that introduced this design explicitly acknowledges the constraint: *"we had to hard code some values, which will be turned into proper protocol parameters in the next era."* [2](#0-1) 

The per-transaction limit is enforced in `validateRefScriptSize` inside the `LEDGER` rule:

```haskell
maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $
    ConwayTxRefScriptsSizeTooBig ...
``` [3](#0-2) 

The per-block limit is enforced in `validateBodyRefScriptsSizeTooBig` inside the `BBODY` rule:

```haskell
maxSize = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerBlockG
totalSize <= maxSize ?! injectFailure (BodyRefScriptsSizeTooBig ...)
``` [4](#0-3) 

The fee multiplier and stride are consumed by `getConwayMinFeeTx` → `tierRefScriptFee`:

```haskell
tierRefScriptFee
    (unboundRational $ pp ^. ppRefScriptCostMultiplierG)
    (fromIntegral @Word32 @Int . unNonZero $ pp ^. ppRefScriptCostStrideG)
    refScriptCostPerByte
    refScriptsSize
``` [5](#0-4) 

In Dijkstra era these four values are promoted to proper `DijkstraPParams` fields (`dppMaxRefScriptSizePerBlock`, `dppMaxRefScriptSizePerTx`, `dppRefScriptCostStride`, `dppRefScriptCostMultiplier`) and are fully updatable via governance. [6](#0-5) 

---

### Impact Explanation

**High — Permanent freezing of funds where recovery requires a hard fork.**

A DApp that deploys Plutus scripts as reference scripts and creates UTxOs whose spending path requires referencing a combined script payload exceeding 204 800 bytes will produce **permanently unspendable outputs** in Conway era. Because `ppMaxRefScriptSizePerTxG` is a compile-time constant, no governance action can raise the ceiling. The only remediation path is an era transition to Dijkstra (a hard fork), which is exactly the criterion for this impact tier.

Secondary impact: the hardcoded fee-tier multiplier (1.2) and stride (25 600) cannot be tightened if the `minFeeRefScriptCostPerByte` protocol parameter is set to a low value by governance, potentially re-enabling the reference-script DDoS vector that was exploited on 25 June 2024.

---

### Likelihood Explanation

**Medium.** The 200 KiB per-transaction ceiling is large relative to a single script, but complex DApps routinely compose many scripts. A transaction that references 8–10 large Plutus V2/V3 scripts (each ~25 KiB) already exceeds the limit. Any DApp developer who deploys such a composition and locks ADA into outputs guarded by those scripts will find those outputs permanently frozen with no on-chain recourse. The scenario requires no privileged access and is reachable by any unprivileged transaction sender.

---

### Recommendation

1. **Short-term (Conway era):** Document the 200 KiB per-transaction and 1 MiB per-block hard caps prominently so DApp developers do not inadvertently create unspendable UTxOs.
2. **Long-term (already partially addressed in Dijkstra):** The Dijkstra era correctly promotes all four values to proper protocol parameters. Ensure the Dijkstra genesis file initialises them to the same values currently hardcoded in Conway so there is no discontinuity at the era boundary.
3. Consider adding a ledger-level warning or predicate failure at UTxO creation time if the output's script hash corresponds to a reference script that, when combined with other known reference scripts, would exceed the per-transaction limit.

---

### Proof of Concept

1. Deploy two Plutus V3 scripts of ~110 KiB each as reference scripts (stored in UTxO outputs).
2. Create a UTxO locked by a multi-validator that requires both scripts to be present as reference inputs.
3. Attempt to spend the locked UTxO by constructing a transaction with both reference inputs (total reference script size ≈ 220 KiB > 204 800 bytes).
4. The `LEDGER` rule calls `validateRefScriptSize`, which reads `ppMaxRefScriptSizePerTxG` → constant `204800`, computes `totalRefScriptSize = 220 * 1024`, and emits `ConwayTxRefScriptsSizeTooBig`.
5. No governance `ParameterChange` proposal can raise the limit in Conway era because `ppMaxRefScriptSizePerTxG = L.to . const $ 200 * 1024` ignores the `PParams` record.
6. The locked ADA is permanently frozen until a Dijkstra hard fork. [1](#0-0) [7](#0-6) [8](#0-7)
