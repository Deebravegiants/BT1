### Title
TEE Verifier Dispatch Not Implemented Despite Voting Mechanism Being Live - (`crates/contract/src/lib.rs`)

---

### Summary

The `MpcContract` stores a `tee_verifier_account_id` field and exposes a complete participant-voting mechanism (`vote_tee_verifier_change`) to elect a trusted external DCAP verifier contract. However, `submit_participant_info` never dispatches to this verifier — the field is explicitly annotated **"Not yet used to dispatch verification."** Attestations are accepted based on local/inline checks only, meaning the intended DCAP cryptographic verification (X.509-chain walking, ECDSA quote verification via `dcap_qvl`) is never enforced in production.

---

### Finding Description

`MpcContract` declares the field and voting infrastructure for an external TEE verifier: [1](#0-0) 

The voting endpoint `vote_tee_verifier_change` is fully implemented and will write a non-`None` value into `tee_verifier_account_id` once threshold votes are reached: [2](#0-1) 

Yet `submit_participant_info` — the only path by which a node registers its TEE attestation — calls only `self.tee_state.add_participant(...)` with the caller-supplied attestation and never cross-calls the elected verifier: [3](#0-2) 

The design document (`docs/design/attestation-verifier-contract.md`) explicitly describes the intended future flow: `submit_participant_info` should dispatch to the verifier contract, receive a `VerificationResult`, and only then insert into `stored_attestations`. That dispatch is absent from the production contract. The mock-attestation path (`VerifiedAttestation::Mock`) used in tests further illustrates that the local path accepts attestations without full DCAP proof: [4](#0-3) 

---

### Impact Explanation

The MPC network's security model requires that every participant runs inside a verified TEE enclave. The external DCAP verifier is the only component that performs full cryptographic verification of an Intel TDX quote (ECDSA signature over the quote body, X.509 PCK chain, TCB status). Without dispatching to it, a node can submit a structurally valid but cryptographically unverified attestation, pass `add_participant`'s local checks, and be accepted as a legitimate MPC participant. A non-TEE node admitted this way holds a real key share and participates in threshold signing — breaking the confidentiality guarantee of the key material and enabling unauthorized signing capability.

This maps to the allowed impact: **Medium — participant-state manipulation that breaks production safety invariants** (the invariant that every stored attestation has been DCAP-verified by the trusted external verifier).

---

### Likelihood Explanation

The voting mechanism is live and callable by any current participant. Once `tee_verifier_account_id` is set (threshold votes reached), operators and users would reasonably expect that subsequent `submit_participant_info` calls are routed through the verifier. Because the dispatch is silently absent, a malicious or compromised node operator can submit an attestation that would be rejected by the real DCAP verifier but is accepted by the contract's local path. No special privilege beyond being a prospective participant is required.

---

### Recommendation

Implement the cross-contract dispatch described in `docs/design/attestation-verifier-contract.md`: when `tee_verifier_account_id` is `Some`, `submit_participant_info` must call `verify_quote` on the verifier contract, park the submission under a `pending_attestations` yield entry, and only insert into `stored_attestations` inside the `resolve_verification` callback on a `VerificationResult::Verified` outcome. Until the dispatch is live, the `tee_verifier_account_id` field provides a false sense of security — participants vote in a verifier that is never consulted.

---

### Proof of Concept

1. Participants vote via `vote_tee_verifier_change` until threshold is reached; `tee_verifier_account_id` is now `Some(verifier)`.
2. A malicious node calls `submit_participant_info` with a structurally valid but DCAP-invalid attestation (e.g., a quote whose PCK signature does not verify against Intel's CA, or whose TCB status is `OutOfDate`).
3. `submit_participant_info` calls `self.tee_state.add_participant(...)` — no cross-contract call to `tee_verifier_account_id` is made.
4. The attestation is stored in `stored_attestations`; the node is treated as a valid TEE participant.
5. `verify_tee` subsequently reads `stored_attestations` and finds the node's entry valid, so `accept_requests` remains `true` and the node participates in threshold signing with a real key share. [5](#0-4)

    pub fn vote_tee_verifier_change(
        &mut self,
        candidate_account_id: AccountId,
        expected_code_hash: TeeVerifierCodeHash,
    ) -> Result<(), Error> {
        log!(
            "vote_tee_verifier_change: signer={}, candidate={}, expected_code_hash={}",
            env::signer_account_id(),
            candidate_account_id,
            expected_code_hash,
        );
        self.voter_or_panic();

        // Voting in the already-current verifier is a no-op
        if self.tee_verifier_account_id.as_ref() == Some(&candidate_account_id) {
            return Ok(());
        }

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();
        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;

        let proposal = VerifierChangeProposal {
            candidate_account_id,
            expected_code_hash,
        };
        if let Some(new_verifier) =
            self.tee_verifier_votes
                .vote(proposal, participant, threshold_parameters)?
        {
            log!("vote_tee_verifier_change: new verifier = {}", new_verifier);
            self.tee_verifier_account_id = Some(new_verifier);
        }
        Ok(())
    }
