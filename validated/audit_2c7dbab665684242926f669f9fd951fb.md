Audit Report

## Title
Arithmetic Underflow in `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` Causes Uninformative Revert Before `startTimestamp` - (File: contracts/pools/RSETHPoolV2.sol)

## Summary
`reinitialize()` enforces that `startTimestamp >= block.timestamp`, creating a guaranteed pre-activation window. During this window, `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` call `getCurrentDay()`, which performs `block.timestamp - startTimestamp` without a pre-activation guard, causing a Solidity 0.8 arithmetic underflow panic revert. The `limitDailyMint` modifier already contains the correct guard but it is absent from these view functions across all five pool variants.

## Finding Description
`reinitialize()` enforces `block.timestamp > _startTimestamp` reverts, so `startTimestamp` is always set to the current block or a future time: [1](#0-0) 

`getCurrentDay()` performs an unchecked subtraction: [2](#0-1) 

The `limitDailyMint` modifier correctly guards this with a pre-activation check before calling `getCurrentDay()`: [3](#0-2) 

However, `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` call `getCurrentDay()` directly with no such guard: [4](#0-3) 

The identical unguarded pattern is confirmed in all other pool variants: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

Any external caller invoking either view function while `block.timestamp < startTimestamp` receives a generic arithmetic panic (0x11) instead of a meaningful error or a valid return value.

## Impact Explanation
The contract fails to deliver its promised view return values during the guaranteed pre-activation window. Off-chain integrations, front-ends, and on-chain contracts relying on `remainingDailyMintLimit()` or `getNextDailyLimitResetTimestamp()` receive an uninformative revert. No funds are at risk and no state is modified.

**Impact: Low** — Contract fails to deliver promised returns, but doesn't lose value.

## Likelihood Explanation
The underflow is deterministically reachable by any unprivileged external caller during the window between `reinitialize()` and `startTimestamp`. Since `reinitialize()` requires `startTimestamp >= block.timestamp`, this window always exists when `startTimestamp` is set to a future time (the intended use case). No special privileges, economic conditions, or victim mistakes are required — a simple view call suffices.

## Recommendation
Add a pre-activation guard in `remainingDailyMintLimit()` and `getNextDailyLimitResetTimestamp()` mirroring the one in `limitDailyMint`, applied consistently across all five pool contracts:

```solidity
function remainingDailyMintLimit() external view returns (uint256) {
    if (block.timestamp < startTimestamp) return dailyMintLimit;
    uint256 effectiveDailyMintAmount = (getCurrentDay() > lastMintDay) ? 0 : dailyMintAmount;
    return dailyMintLimit - effectiveDailyMintAmount;
}

function getNextDailyLimitResetTimestamp() external view returns (uint256) {
    if (block.timestamp < startTimestamp) return startTimestamp;
    return startTimestamp + (getCurrentDay() + 1) * 1 days;
}
```

Apply to: `RSETHPoolV2.sol`, `RSETHPoolV3.sol`, `RSETHPoolV2ExternalBridge.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`.

## Proof of Concept
1. Admin calls `reinitialize(1000 ether, block.timestamp + 1 days)` — `startTimestamp` is set to 24 hours from now.
2. Any account calls `remainingDailyMintLimit()` (or `getNextDailyLimitResetTimestamp()`).
3. Execution reaches `getCurrentDay()`, which computes `block.timestamp - startTimestamp`; since `block.timestamp < startTimestamp` and both are `uint256`, Solidity 0.8 reverts with arithmetic panic `0x11`.
4. The caller receives a generic revert with no actionable message, while the `limitDailyMint` modifier would have emitted `MintBeforeStartTimestamp`.

**Foundry test sketch:**
```solidity
function test_remainingDailyMintLimit_underflow() public {
    uint256 futureStart = block.timestamp + 1 days;
    vm.prank(admin);
    pool.reinitialize(1000 ether, futureStart);
    // Still before startTimestamp
    vm.expectRevert(stdError.arithmeticError);
    pool.remainingDailyMintLimit();
}
```
