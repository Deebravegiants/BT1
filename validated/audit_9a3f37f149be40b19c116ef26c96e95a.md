### Title
Peer-Supplied NFT Spend Solution Trusted Without On-Chain Verification in `puzzle_solution_received` — (`chia/wallet/nft_wallet/nft_wallet.py`)

### Summary
`NFTWallet.puzzle_solution_received` uses the solution from a peer-supplied `NFTCoinData.parent_coin_spend` to reconstruct the child NFT coin's puzzle hash and metadata without verifying the solution against the actual confirmed on-chain spend. A malicious peer can supply a crafted but CLVM-valid NFT state layer solution, causing the wallet to track a phantom child coin and lose track of the real on-chain NFT.

### Finding Description
In `puzzle_solution_received` (lines 194, 217), both `get_metadata_and_phs` and `recurry_nft_puzzle` consume `data.parent_coin_spend.solution` verbatim. [5](#0-4) [2](#0-1) 

The `compute_additions` check at line 233 is self-referential — it runs the puzzle against the attacker's solution, not the actual on-chain solution, so it cannot distinguish a crafted solution from the real one. [3](#0-2) 

The only filtering guard is the `derivation_record` lookup (line 199), which checks that the extracted `p2_puzzle_hash` belongs to the wallet. Since the victim's puzzle hashes are publicly observable on-chain, an attacker can embed one in the crafted solution. [4](#0-3) 

The phantom child coin is then persisted via `add_coin` (line 256). [6](#0-5) 

### Impact Explanation
The wallet's NFT store is corrupted with a phantom coin that does not exist on-chain. The real child NFT (from the actual on-chain spend) is never registered. The victim loses wallet-level access to the NFT until a full trusted resync. This is wallet sync state corruption with direct NFT loss impact.

### Likelihood Explanation
Any peer the wallet syncs from can execute this attack. Wallets connect to multiple peers and trust them for puzzle/solution data. No special privilege is required beyond being a connected peer.

### Recommendation
After fetching `parent_coin_spend` from a peer, verify the solution by re-deriving the coin's puzzle hash from the spend's puzzle and confirming it matches the actual on-chain coin's `puzzle_hash`, then independently fetch and compare the spend solution from a trusted full node or via the confirmed block. Do not use the peer-supplied solution as the authoritative source for child puzzle reconstruction.

4. Observe that `puzzle_solution_received` completes without error and `nft_store` contains a coin with the phantom puzzle hash.
