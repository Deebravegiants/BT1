### Title
`skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions When Any Pending Transaction Exists — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is intended to skip the gateway-level `__validate__` call only when a `deploy_account` transaction is pending in the mempool (UX improvement for simultaneous deploy+invoke). However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** pending transaction from the account, not exclusively a `deploy_account`. An attacker who observes a pending transaction for an account with on-chain nonce=0 can inject an invoke(nonce=1) carrying an invalid signature directly into the mempool, bypassing the `__validate__` entry-point check entirely.

---

### Finding Description

`skip_stateful_validations` fires when three conditions are simultaneously true:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain state). [1](#0-0) 

When all three hold, the function calls `account_tx_in_pool_or_recent_block` and returns its result as the `skip_validate` flag: [2](#0-1) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

It returns `true` if the account has **any** transaction in the pool — an `Invoke`, a `Declare`, or a `DeployAccount`. The code comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of that reasoning is incorrect: an `Invoke(nonce=0)` for a pre-deployed account (on-chain nonce=0) passes `__validate__` and lands in the pool, but its presence says nothing about whether a subsequent `Invoke(nonce=1)` from a different sender is legitimate.

When `skip_validate == true`, `run_validate_entry_point` sets `execution_flags.validate = false` and calls the blockifier validator without running `__validate__`: [4](#0-3) 

The full call chain is:

```
add_tx_inner (gateway.rs:253-254)
  → extract_state_nonce_and_run_validations
      → run_pre_validation_checks
          → skip_stateful_validations  ← returns true (wrong)
      → run_validate_entry_point(skip_validate=true)  ← __validate__ skipped
  → mempool_client.add_tx(...)  ← invalid tx admitted
``` [5](#0-4) 

---

### Impact Explanation

An attacker can inject an `Invoke(nonce=1)` with an **invalid signature** into the mempool for any account that satisfies the three conditions above. Concrete consequences:

- **Invalid transaction admitted**: The mempool accepts a transaction that will fail `__validate__` during blockifier execution. This directly matches the High impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*
- **Nonce slot squatting**: The attacker's invalid `Invoke(nonce=1)` occupies the nonce=1 slot. The legitimate owner's `Invoke(nonce=1)` is rejected as a duplicate nonce by the mempool's `validate_incoming_tx` / `validate_fee_escalation` path.
- **Forced fee escalation**: The owner must submit a replacement transaction with a strictly higher tip and max-L2-gas-price (fee escalation) to evict the attacker's invalid transaction, or wait for it to be executed and rejected by the batcher.
- **Batcher resource waste**: The invalid transaction is eventually dequeued, executed, and rejected by the blockifier, consuming execution resources for no legitimate purpose.

---

### Likelihood Explanation

- The attack requires only that the target account has on-chain nonce=0 and a pending transaction visible in the mempool — a common state during the deploy_account + invoke UX flow that this feature was designed to support.
- No privileged access is required; any unprivileged actor can submit transactions to the gateway.
- The attacker only needs to observe the mempool (via `mempool_snapshot` or P2P gossip) to identify eligible accounts.
- The attack is cheap: submitting a single malformed invoke costs only the gateway's stateless validation overhead (no fee is charged at admission time).

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction is pending for the account. For example, expose a `deploy_account_in_pool(address)` query on the mempool that inspects the transaction type, and use that in `skip_stateful_validations`:

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await

// Use:
mempool_client.deploy_account_in_pool(tx.sender_address()).await
```

This preserves the intended UX improvement (deploy + invoke in one shot) while closing the bypass for accounts that have non-deploy_account transactions pending.

---

### Proof of Concept

**Setup**: Account `X` is a pre-deployed contract with on-chain nonce=0 (e.g., a system contract, or an account deployed via L1 message before the sequencer started).

1. **Owner** submits `Invoke(sender=X, nonce=0, sig=valid_sig)` to the gateway.
   - `validate_nonce`: `0 <= 0 <= max_gap` ✓
   - `run_validate_entry_point`: `__validate__` runs, passes ✓
   - Transaction admitted to mempool.

2. **Attacker** submits `Invoke(sender=X, nonce=1, sig=garbage)` to the gateway.
   - `validate_nonce`: `0 <= 1 <= max_gap` ✓
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true` → returns `true`
   - `run_validate_entry_point`: called with `skip_validate=true` → `execution_flags.validate=false` → `__validate__` **not run** ✓
   - Attacker's invalid transaction admitted to mempool.

3. **Owner** submits `Invoke(sender=X, nonce=1, sig=valid_sig)`.
   - Mempool's `validate_incoming_tx` → `state.validate_incoming_tx` detects duplicate nonce=1 for address X → **rejected** with `MempoolError::DuplicateNonce`.

4. Owner must either wait for the attacker's invalid tx to be executed and rejected by the batcher (one block delay), or pay a higher fee via fee escalation to replace it. [6](#0-5) [3](#0-2) [4](#0-3)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
