### Title
Expired TEE Attestation Treated as Valid in `respond` Authorization — (`crates/contract/src/tee/tee_state.rs`)

### Summary

`is_caller_an_attested_participant` — the function used to gate every `respond`, `respond_ckd`, and `respond_verify_foreign_tx` call — checks only that a stored attestation entry *exists* and that the key fields match. It never calls `re_verify` to check whether the attestation has expired or whether the node's image hash is still on the approved list. A node whose attestation has lapsed (e.g., because its Docker image hash was rotated out) can therefore continue submitting accepted signatures to the contract until `verify_tee()` is explicitly called and triggers a resharing — a window of up to two days in normal operation.

### Finding Description

`assert_caller_is_attested_participant_and_protocol_active` is called at the top of every node-facing respond method:

```rust
fn assert_caller_is_attested_participant_and_protocol_active(&self) {
    let participants = self.protocol_state.active_participants();
    Self::assert_caller_is_signer();
    let attestation_check = self
        .tee_state
        .is_caller_an_attested_participant(participants);
    assert_matches::assert_matches!(attestation_check, Ok(()), "Caller must be an attested participant");
}
``` [1](#0-0) 

`is_caller_an_attested_participant` performs four checks: (1) caller is in the participant list, (2) a stored attestation entry exists for the caller's TLS key, (3) the stored `account_id` matches, (4) the stored `account_public_key` matches. It returns `Ok(())` on all four passing — **without ever calling `re_verify`**:

```rust
pub(crate) fn is_caller_an_attested_participant(
    &self,
    participants: &Participants,
) -> Result<(), AttestationCheckError> {
    // ...
    let attestation = self
        .stored_attestations
        .get(&info.tls_public_key)
        .ok_or(AttestationCheckError::AttestationNotFound)?;
    // account_id and account_public_key checks only — no expiry check
    Ok(())
}
``` [2](#0-1) 

By contrast, `reverify_participants` — used by `verify_tee()` and `reverify_and_cleanup_participants` — calls `re_verify`, which checks expiry, image-hash allowlist, launcher-hash allowlist, and measurements:

```rust
match participant_attestation.verified_attestation.re_verify(
    time_stamp_seconds,
    &allowed_mpc_docker_image_hashes,
    &allowed_launcher_compose_hashes,
    &allowed_measurements,
) {
    Ok(()) => TeeQuoteStatus::Valid,
    Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
}
``` [3](#0-2) 

`re_verify` on a `Dstack` attestation explicitly rejects entries whose `expiry_timestamp_seconds` has passed:

```rust
let attestation_has_expired = *expiration_timestamp_seconds < timestamp_seconds;
if attestation_has_expired {
    return Err(VerificationError::Custom(format!(
        "The attestation expired at t = {:?}, time_now = {:?}",
        expiration_timestamp_seconds, timestamp_seconds
    )));
}
``` [4](#0-3) 

The `respond` function does check `accept_requests`, but that flag is only set to `false` when `verify_tee()` determines that a kickout would drop the participant count below threshold. In the normal case (enough valid participants remain), `accept_requests` stays `true` and the expired node's `respond` call proceeds:

```rust
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
``` [5](#0-4) 

### Impact Explanation

The TEE attestation system's core invariant is that only nodes running an approved image inside a genuine enclave may submit signatures. `is_caller_an_attested_participant` breaks this invariant: a node whose attestation has expired — because its Docker image hash was rotated out, its certificate chain lapsed, or its measurements are no longer in the allowed set — can still call `respond` and have its signature accepted by the contract. This is a participant/attestation authorization bypass: the contract accepts signing contributions from a node that the TEE governance system has implicitly revoked, allowing potentially non-genuine or non-approved TEE code to participate in threshold signature generation.

### Likelihood Explanation

Attestations expire after 7 days (`DEFAULT_EXPIRATION_DURATION_SECONDS`). `verify_tee()` is called by participants roughly every 2 days. The gap between expiry and the next `verify_tee()` call is the exploitation window — up to 2 days in normal operation. The trigger is passive: any image-hash rotation or certificate expiry opens the window automatically, with no attacker action required beyond the node continuing to operate. The node retains its NEAR account key and key share throughout, so it can produce cryptographically valid signatures that pass the contract's signature-verification step.

### Recommendation

`is_caller_an_attested_participant` should call `re_verify` (or delegate to `reverify_participants`) before returning `Ok(())`, so that an expired or image-hash-revoked attestation is rejected at the `respond` call site rather than only at the next `verify_tee()` sweep. Concretely, after the key-match checks pass, add:

```rust
match self.reverify_participants(node_id, tee_upgrade_deadline_duration) {
    TeeQuoteStatus::Valid => Ok(()),
    TeeQuoteStatus::Invalid(reason) => Err(AttestationCheckError::AttestationExpiredOrInvalid(reason)),
}
```

This mirrors the check already performed by `reverify_and_cleanup_participants` and closes the window between attestation expiry and the next `verify_tee()` call.

### Proof of Concept

1. Contract is `Running` with nodes A, B, C; `accept_requests = true`.
2. The governance threshold votes in a new Docker image hash; node A's old hash is now expired.
3. Node A's stored `VerifiedAttestation` now fails `re_verify` (image hash no longer allowed), but the entry remains in `stored_attestations` — `clean_invalid_attestations` has not yet been called.
4. `verify_tee()` has not been called since the rotation (within the 2-day window).
5. A user submits a `sign` request; it enters the pending queue.
6. Node A calls `respond(request, signature)`.
7. `assert_caller_is_attested_participant_and_protocol_active` → `is_caller_an_attested_participant` → finds node A in participants list, finds stored attestation, keys match → returns `Ok(())`. **No expiry or image-hash check is performed.**
8. `accept_requests` is `true` → passes.
9. Signature is cryptographically valid (node A still holds its key share) → `resolve_yields_for` resumes the user's yield with the signature.
10. The contract has accepted a signing contribution from a node running a non-approved image, bypassing the TEE security guarantee. [2](#0-1) [6](#0-5) [7](#0-6)

    pub(crate) fn reverify_participants(
        &self,
        node_id: &NodeId,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeQuoteStatus {
        let allowed_mpc_docker_image_hashes =
            self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
        let allowed_launcher_compose_hashes = self.get_allowed_launcher_compose_hashes();

            &allowed_measurements,

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
