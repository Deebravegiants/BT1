### Title
Stale TEE-vote counts bypass image-hash whitelisting threshold after resharing — (`File: crates/contract/src/lib.rs`)

### Summary

`vote_code_hash`, `vote_add_launcher_hash`, and `vote_add_os_measurement` compare the raw stored-vote count (which includes votes from participants removed during resharing) against the current threshold. The cleanup of stale votes (`clean_tee_status`) is a detached, asynchronous promise that fires after resharing completes, leaving a window of at least one NEAR block during which stale votes from removed participants are still counted. An attacker who controls `threshold − 1` removed participants (who pre-voted for a malicious image hash) and a single new participant can reach the threshold with only one current-participant vote, whitelisting an unauthorized Docker image hash.

---

### Finding Description

`vote_reshared` in `crates/contract/src/lib.rs` transitions the contract to the new running state and then spawns several cleanup receipts as detached promises: [1](#0-0) 

`clean_tee_status` is one of those detached receipts: [2](#0-1) 

It is marked `#[private]` and calls `tee_state.clean_non_participant_votes()`: [3](#0-2) 

Until that receipt is processed (at minimum one block later), the `tee_state.votes.proposal_by_account` map still contains entries for removed participants.

`vote_code_hash` authenticates the caller against the **current** participant set, then calls `tee_state.vote()` which inserts the new vote and returns the count of **all** stored votes for that hash — including stale ones: [4](#0-3) 

The threshold check at line 1425 therefore operates on an inflated count. The same pattern exists in `vote_add_launcher_hash` and `vote_add_os_measurement`: [5](#0-4) [6](#0-5) 

The codebase's own unit test explicitly documents that without cleanup, stale votes **are** counted: [7](#0-6) 

The comment at line 1496–1503 describes exactly the scenario: P0 and P1 vote for a malicious hash, resharing removes them, and without `clean_non_participant_votes` a single new participant's vote would be counted as 3.

---

### Impact Explanation

An attacker who controls `threshold − 1` participants that are subsequently removed via resharing, plus a single new participant, can whitelist an arbitrary Docker image hash with only **one** current-participant vote instead of the required `threshold` votes. A whitelisted hash allows a node running a compromised image to submit a valid TEE attestation (`submit_participant_info`), be admitted into the next resharing, receive a key share, and participate in threshold signing — breaking the TEE security invariant that gates MPC key-share access.

This maps to: **Medium** — participant-state manipulation that breaks a production safety invariant (TEE image-hash gating) without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

- Requires `threshold − 1` colluding participants to pre-vote for a target hash before being removed. This is below the signing threshold, so they cannot harm the network on their own.
- Requires one new (colluding) participant to submit `vote_code_hash` in the one-block window between resharing completion and `clean_tee_status` execution.
- NEAR block time is ~1 second; the window is deterministic and predictable from on-chain events.
- The attack is fully on-chain with no privileged access beyond being a (new) participant.

---

### Recommendation

Before counting votes toward the threshold in `vote_code_hash`, `vote_add_launcher_hash`, and `vote_add_os_measurement`, filter the stored vote map to only count votes from **current** participants — analogous to how `vote_update` was fixed (see the regression test at line 5434): [8](#0-7) 

Apply the same pattern to the TEE voting functions: after calling `tee_state.vote(...)`, recount only votes whose keys are present in the current `threshold_parameters.participants()` set before comparing against `self.threshold()?.value()`. This eliminates the race window entirely without relying on the asynchronous cleanup promise.

---

### Proof of Concept

**Setup:** N = 5, threshold = 3. Participants: {P1, P2, P3, P4, P5}.

1. P1 and P2 (malicious) call `vote_code_hash(malicious_hash)`. Stored votes: {P1→H, P2→H}. Count = 2 < 3 → not whitelisted.
2. Governance votes to reshard: P1 and P2 are removed. New set: {P3, P4, P5}, threshold = 3.
3. `vote_reshared` completes. `clean_tee_status` receipt is queued but not yet processed.
4. P3 (colluding new participant) immediately calls `vote_code_hash(malicious_hash)` in the same or next block.
5. `tee_state.vote()` inserts P3's vote and counts: P1 + P2 + P3 = **3 ≥ threshold(3)** → `whitelist_tee_proposal(malicious_hash)` executes.
6. `malicious_hash` is now in the allowed list. A node running the malicious image can submit a valid attestation and be admitted into the next resharing.

Only **1 current participant** voted, but the threshold was satisfied via 2 stale votes from removed participants.

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
