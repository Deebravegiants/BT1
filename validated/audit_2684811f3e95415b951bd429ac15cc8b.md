### Title
Token Decimal Mismatch in `viewSwapRsETHAmountAndFee` Causes Severe Under-Minting of wrsETH for Non-18 Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

---

### Summary

The `viewSwapRsETHAmountAndFee(uint256 amount, address token)` overload in all L2 pool contracts computes the wrsETH mint amount using the raw token amount without normalizing it to 18 decimals. The formula silently assumes the deposited token has 18 decimals. If any ERC-20 token with fewer decimals (e.g., USDC at 6) is added via `addSupportedToken`, every depositor of that token will receive a wrsETH amount that is `10^(18 − tokenDecimals)` times smaller than correct — effectively destroying their deposit value.

---

### Finding Description

All five L2 pool contracts share the same token-deposit pricing function:

```solidity
// RSETHPoolV3.sol lines 315–335
function viewSwapRsETHAmountAndFee(
    uint256 amount,
    address token
) public view onlySupportedToken(token) returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;

    uint256 rsETHToETHrate = getRate();                                    // 18-decimal rate
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // 18-decimal rate

    rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;        // ← no decimal normalisation
}
``` [1](#0-0) 

Both `rsETHToETHrate` and `tokenToETHRate` are 18-decimal rates (price of **one full token** in ETH). The formula is dimensionally correct only when `amount` is also expressed in 18-decimal units. For a token with `d < 18` decimals, `amount` carries `d`-decimal units, so the product `amountAfterFee * tokenToETHRate` is `10^(18−d)` times too small, and the minted wrsETH is correspondingly tiny.

The same pattern is replicated verbatim in:

- `RSETHPoolV3ExternalBridge.sol` lines 433–453 [2](#0-1) 
- `RSETHPoolV3WithNativeChainBridge.sol` lines 351–371 [3](#0-2) 
- `RSETHPool.sol` lines 326–347 [4](#0-3) 
- `RSETHPoolNoWrapper.sol` lines 292–312 [5](#0-4) 

The `addSupportedToken` / `_addSupportedToken` functions impose no decimal constraint on the token being added — any ERC-20 with a valid oracle passes:

```solidity

``` [6](#0-5)

...

``` [7](#0-6)

### Impact Explanation

**Impact: Low — Contract fails to deliver promised returns.**

A user depositing a non-18-decimal token receives a wrsETH balance that is `10^(18−d)` times smaller than the economically correct amount. For USDC (`d = 6`) the shortfall is a factor of `10^12`. The deposited tokens are transferred to the pool and eventually bridged to L1; the user retains only a dust wrsETH balance with negligible redemption value. No funds are directly stolen by a third party, but the user's deposit is rendered economically worthless — the contract fails to deliver the promised exchange.

---

### Likelihood Explanation

**Likelihood: Low.**

The vulnerability is latent: it activates only when a TIMELOCK_ROLE holder adds a token whose `decimals() < 18`. All currently supported assets (native ETH, wstETH) have 18 decimals, so no user is harmed today. However, the protocol's architecture explicitly supports adding new tokens, and the code contains no guard against non-18-decimal tokens. A future governance decision to add a stablecoin or other low-decimal asset would silently activate the bug for every subsequent depositor.

---

### Recommendation

Normalize the deposited amount to 18 decimals before applying the rate formula:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

function viewSwapRsETHAmountAndFee(
    uint256 amount,
    address token
) public view onlySupportedToken(token) returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;

    // Normalise to 18 decimals
    uint8 tokenDecimals = IERC20Metadata(token).decimals();
    uint256 amountIn18 = amountAfterFee * 1e18 / (10 ** uint256(tokenDecimals));

    uint256 rsETHToETHrate = getRate();
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

    rsETHAmount = amountIn18 * tokenToETHRate / rsETHToETHrate;
}
```

Apply the same fix to the reverse-swap helper `viewSwapAssetToPremintedRsETH` (which has the symmetric over-payment bug for non-18-decimal tokens).

---

### Proof of Concept

Assume USDC (6 decimals) is added with an oracle returning `4e14` (≈ 0.0004 ETH per USDC) and `rsETHToETHrate = 1.05e18`.

**User deposits 1 000 USDC → `amount = 1e9` (6-decimal units):**

```
fee            = 1e9 * feeBps / 10_000          ≈ 0 (negligible)
amountAfterFee ≈ 1e9
rsETHAmount    = 1e9 * 4e14 / 1.05e18
               = 4e23 / 1.05e18
               ≈ 380 wei of wrsETH
```

**Correct amount:**

```
1 000 USDC = 0.4 ETH → 0.4 / 1.05 rsETH ≈ 3.8e17 wrsETH units
```

The user receives **380 wei** instead of **3.8 × 10¹⁷ wei** — a shortfall of `10^12×`. The 1 000 USDC is transferred to the pool and bridged to L1; the user's wrsETH balance is economically worthless.
