### Title
`respond_ckd` accepts arbitrary CKD response for `AppPublicKey` requests without output verification - (File: crates/contract/src/lib.rs)

### Summary
The `respond_ckd` function in the MPC contract unconditionally skips output verification for `CKDAppPublicKey::AppPublicKey` requests. Any single attested participant (strictly below the signing threshold) can call `respond_ckd` with a forged `CKDResponse`, bypassing the threshold requirement and delivering an attacker-controlled private key to the requesting user.

### Finding Description
In `crates/contract/src/lib.rs`, the `respond_ckd` function handles two variants of `CKDAppPublicKey`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the response against the app public key and the MPC root key. For `AppPublicKey`, the match arm is an empty no-op — the contract accepts whatever `CKDResponse` the caller provides without any cryptographic verification.

The function's only access control is `assert_caller_is_attested_participant_and_protocol_active()` — there is no leader check and no threshold enforcement on who may submit the response. [2](#0-1) 

After the (absent) check, the contract immediately resolves all pending yields for the matching request with the unverified response:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [3](#0-2) 

This is structurally identical to the reported `RewardOracle` pattern: a critical output parameter (`response`) is accepted from an authorized caller without being validated against the expected protocol state, allowing that caller to inject an arbitrary value.

### Impact Explanation
A single Byzantine attested participant (below the signing threshold) can:
1. Observe any pending `AppPublicKey` CKD request stored in `pending_ckd_requests` (on-chain, publicly readable).
2. Construct a `CKDRequest` that exactly matches the pending entry.
3. Call `respond_ckd` with an arbitrary `CKDResponse` — for example, a key the attacker controls.
4. The contract accepts the forged response with no verification and resumes the user's yield with the attacker-controlled key.
5. The user receives an attacker-controlled private key instead of the legitimately threshold-derived one.

This is unauthorized confidential key derivation output without the required participant authorization — a critical impact under the allowed scope.

### Likelihood Explanation
- Any node that has passed TEE attestation qualifies as an attested participant; this is a standard operational role, not a special privilege.
- Pending CKD requests are stored in contract state and are publicly observable on-chain.
- The attacker must front-run the legitimate leader's `respond_ckd` submission. On NEAR, a participant who monitors the chain can submit a transaction in the same or next block, making this practically feasible.
- No collusion above threshold is required; a single Byzantine participant suffices.

### Recommendation
Add output verification for `AppPublicKey` requests analogous to the `AppPublicKeyPV` branch. If on-chain verification is not possible for this variant (because no verifiable public key is available), enforce that only the designated leader (participant with the lowest ID, consistent with `verify_leader` used in `KeyEvent::start`) may submit the response, or require threshold-many matching responses before the yield is resolved. [4](#0-3) 

### Proof of Concept
```
1. User calls request_app_private_key with AppPublicKey variant
   → pending_ckd_requests[CKDRequest{..., app_public_key: AppPublicKey(pk)}] = [yield_id]

2. Malicious attested participant reads the pending CKDRequest from on-chain state.

3. Malicious participant calls:
     respond_ckd(
       request  = CKDRequest{..., app_public_key: AppPublicKey(pk)},  // matches pending key
       response = CKDResponse{ /* attacker-chosen key material */ }
     )

4. Contract executes:
     match &request.app_public_key {
         AppPublicKey(_) => {}   // ← empty arm, no verification
         ...
     }
     resolve_yields_for(&mut self.pending_ckd_requests, &request, forged_bytes)
     // user's yield is resumed with the forged key

5. User's callback receives the attacker-controlled private key.
   The legitimate leader's subsequent respond_ckd call returns RequestNotFound
   (the entry was already drained), so the correct key is never delivered.
```

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
