Audit Report

## Title
Stale Cached Rate in `CrossChainRateReceiver` Allows Deposit at Below-Market Rate, Draining Pool rsETH Reserves — (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver` stores a single cached `rate` that is only written when a LayerZero message arrives via `lzReceive`. Both `RSETHPoolNoWrapper` and `RSETHPoolV3` price every deposit against this cached value with no staleness check. Because the rsETH/ETH rate monotonically increases on L1 as staking rewards accrue, the L2 oracle is structurally stale between updates. An unprivileged depositor can deposit ETH during the staleness window, receive excess rsETH/wrsETH at the outdated rate, and redeem at the true rate on L1, extracting value from the pool's reserves.

## Finding Description

**Root cause — `CrossChainRateReceiver.sol`**

`rate` is written only inside `lzReceive` and returned raw by `getRate()` with no staleness guard:

```solidity
// CrossChainRateReceiver.sol L95-97
rate = _rate;
lastUpdated = block.timestamp;
emit RateUpdated(_rate);

// CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

`lastUpdated` is stored but never checked by any consumer.

**Consumption in deposit pools — no freshness check**

`RSETHPoolNoWrapper.getRate()` and `RSETHPoolV3.getRate()` both forward directly to the oracle with no staleness validation:

```solidity
// RSETHPoolNoWrapper.sol L220-222
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
```

`viewSwapRsETHAmountAndFee` in both pools divides by this cached rate:

```solidity
// RSETHPoolNoWrapper.sol L282-285
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

A stale (lower) `rate` produces a larger `rsETHAmount` for the same ETH input.

**Rate update mechanism — permissionless trigger**

`MultiChainRateProvider.updateRate()` has no access control; anyone can call it:

```solidity
// MultiChainRateProvider.sol L108
function updateRate() external payable nonReentrant {
```

This means the exact moment a rate update is dispatched from L1 is publicly observable on-chain (via the `RateUpdated` event on L1), and the LayerZero delivery window (minutes) is the exploitable gap.

**Exploit path for `RSETHPoolNoWrapper`**

The pool holds a finite pre-minted rsETH OFT reserve and transfers it directly to depositors:

```solidity
// RSETHPoolNoWrapper.sol L237-241
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
rsETH.safeTransfer(msg.sender, rsETHAmount);
```

1. L1 rsETH rate increases from `R_old` to `R_new` (reward accrual or rebalancing).
2. Anyone calls `updateRate()` on L1; LayerZero message is in-flight.
3. Before `lzReceive` executes on L2, attacker calls `deposit()` with ETH.
   - Receives `rsETHAmount = ETH * 1e18 / R_old` — more rsETH than `ETH * 1e18 / R_new`.
4. `lzReceive` lands; `rate = R_new`.
5. Attacker bridges rsETH OFT to L1 and redeems at `R_new`.
   - ETH out = `rsETHAmount * R_new / 1e18 = ETH_in * R_new / R_old > ETH_in`.

The excess rsETH transferred out of the pool is never replenished by the deposited ETH (which is bridged to L1 separately at face value), leaving the pool short.

**`RSETHPoolV3` path**

`deposit()` calls `wrsETH.mint(msg.sender, rsETHAmount)`, minting excess wrsETH at the stale rate. The attacker bridges wrsETH to L1, unwraps, and redeems at `R_new`. The `dailyMintLimit` caps per-day scale but does not prevent the attack.

**No existing mitigations**

- `CrossChainRateReceiver` stores `lastUpdated` but no consumer checks it.
- Neither pool enforces a maximum staleness window.
- Neither pool has a rate-change circuit-breaker.
- `RSETHPoolNoWrapper` has no daily mint cap at all.

## Impact Explanation

**Critical — Direct theft of protocol/user funds.**

In `RSETHPoolNoWrapper`, the pool's pre-minted rsETH OFT reserve is a finite asset. Every deposit at a stale low rate transfers more rsETH than the deposited ETH is worth at the true rate. The attacker redeems the excess rsETH on L1 for ETH, leaving the pool with a deficit that cannot be recovered from the deposited ETH alone. This is direct, irreversible extraction of value from the pool's reserves, matching the "Direct theft of any user funds" Critical impact class.

In `RSETHPoolV3`, excess wrsETH minting over-claims on the underlying ETH collateral, diluting all existing wrsETH holders and risking protocol insolvency at scale.

## Likelihood Explanation

**Medium-High.**

- The rsETH rate increases continuously; every update cycle creates a window.
- `updateRate()` is permissionless — the attacker can observe the L1 `RateUpdated` event and know the exact moment the in-flight message was sent.
- LayerZero delivery latency (typically 1–10 minutes) is publicly visible; the window is predictable.
- No special role, governance access, or external compromise is required — only a standard ETH deposit.
- The attack is repeatable on every rate update cycle.
- Profitability depends on the rate gap exceeding bridge costs; gaps accumulate if updates are infrequent or if large one-time rate increases occur (e.g., rebalancing events).

## Recommendation

1. **Staleness check in pools**: In `getRate()` (or at the top of `deposit()`), revert if `block.timestamp - CrossChainRateReceiver(rsETHOracle).lastUpdated() > MAX_STALENESS` (e.g., 1–2 hours).
2. **Rate-change circuit-breaker in `lzReceive`**: Revert or pause deposits if the incoming rate deviates from the previous rate by more than a threshold (e.g., 0.5%) to prevent large single-step jumps from being exploited.
3. **Rate smoothing**: Linearly interpolate from `rate_old` to `rate_new` over a short window after each `lzReceive`, analogous to ERC-4626 share-price smoothing, to eliminate the instantaneous jump that creates the arbitrage.
4. **Increase update frequency**: Trigger `updateRate()` automatically on every L1 rsETH rate change to minimize the staleness window.

## Proof of Concept

```
Assumptions:
  R_old = 1.050e18  (stale L2 rate)
  R_new = 1.055e18  (true L1 rate after reward accrual)
  ETH deposited = 100 ETH
  feeBps = 0

Step 1 — attacker calls RSETHPoolNoWrapper.deposit() before lzReceive:
  rsETHAmount = 100e18 * 1e18 / 1.050e18 = 95.238095... rsETH

Step 2 — lzReceive lands: rate = 1.055e18

Step 3 — attacker bridges rsETH OFT to L1 and redeems:
  ETH received = 95.238095 * 1.055e18 / 1e18 = 100.476... ETH

Profit = ~0.476 ETH (~0.476%) on 100 ETH.

Foundry fork test outline:
1. Fork Arbitrum mainnet; deploy/use live RSETHPoolNoWrapper and CrossChainRateReceiver.
2. Record current rate R_old from CrossChainRateReceiver.getRate().
3. Impersonate LayerZero endpoint; call lzReceive with R_new = R_old * 1.005 (0.5% increase)
   but do NOT execute it yet (store calldata).
4. Call RSETHPoolNoWrapper.deposit{value: 100 ether}("") as attacker.
5. Record rsETHAmount received.
6. Now execute the stored lzReceive call to update rate to R_new.
7. Assert rsETHAmount > 100 ether * 1e18 / R_new (attacker received excess rsETH).
8. Compute ETH value of rsETHAmount at R_new; assert it exceeds 100 ether.
```
