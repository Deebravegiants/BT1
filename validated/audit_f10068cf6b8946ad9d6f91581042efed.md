### Title
Missing Receiver Existence Validation in `promise_set_refund_to` Allows Permanent Burning of User Deposits - (File: runtime/near-vm-runner/src/logic/logic.rs)

### Summary
The `promise_set_refund_to` host function validates only the syntactic format of the `refund_to` account ID, not whether the account actually exists on-chain. A malicious or buggy contract can redirect deposit refunds to a non-existent named account. When the associated receipt fails, the runtime generates a balance-refund receipt to that non-existent account; that refund receipt itself then fails, and the deposit is permanently burnt per the documented protocol rule: *"If the execution of a refund fails, the refund amount is burnt."*

### Finding Description

`promise_set_refund_to` in `runtime/near-vm-runner/src/logic/logic.rs` reads and parses the caller-supplied account ID, then unconditionally stores it as the `refund_to` field on the outgoing receipt:

```rust
let refund_to = self.read_and_parse_account_id(account_id_ptr, account_id_len)?;
// ...
self.ext.set_refund_to(receipt_idx, refund_to);
``` [1](#0-0) 

`read_and_parse_account_id` enforces only syntactic validity (the string must be a well-formed NEAR account ID). No check is made that the account actually exists in the trie.

The `refund_to` field is stored in `ActionReceiptV2`: [2](#0-1) 

Receipt validation in `validate_action_receipt` also only validates format:

```rust
if let Some(account_id) = receipt.refund_to() {
    AccountId::validate(account_id.as_ref()).map_err(|_| {
        ReceiptValidationError::InvalidRefundTo { account_id: account_id.to_string() }
    })?;
}
``` [3](#0-2) 

When the receipt fails, the runtime calls `receipt.balance_refund_receiver()` to determine where to send the deposit refund, which returns the `refund_to` account if set:

```rust
if deposit_refund > Balance::ZERO {
    result.new_receipts.push(Receipt::new_balance_refund(
        receipt.balance_refund_receiver(),
        deposit_refund,
    ));
}
``` [4](#0-3) 

`Receipt::new_balance_refund` creates a system-predecessor Transfer receipt to the specified account: [5](#0-4) 

When this refund receipt is processed, `check_account_existence` for a Transfer action calls `check_transfer_to_nonexisting_account`. For a non-existent **named** account (not a 64-char hex implicit account), this returns `AccountDoesNotExist`: [6](#0-5) 

The refund receipt fails. Per protocol design, a failed refund receipt burns the amount. The deposit is gone permanently.

### Impact Explanation

The corrupted protocol value is the **user's NEAR balance**: the deposit attached to the cross-contract call is permanently burnt rather than returned. This is an irreversible loss of funds. The state root after the block reflects a lower total supply (tokens burnt) without the user receiving their deposit back.

### Likelihood Explanation

Any unprivileged user can deploy a contract on NEAR mainnet. A malicious contract that:
1. Accepts a deposit from a victim caller,
2. Forwards it in a cross-contract call to a non-existent or always-failing target,
3. Calls `promise_set_refund_to` with a non-existent named account (e.g., `"burn.near"` if unregistered),

will cause the victim's deposit to be permanently burnt when the inner receipt fails. The victim has no way to detect this from the contract interface alone. The attack is fully reachable via a public RPC transaction calling the malicious contract.

### Recommendation

Before storing the `refund_to` account, validate that it either:
- Is an implicit account (64-char hex or eth-implicit), which can be created by a Transfer, **or**
- Actually exists in the current trie state.

Alternatively, at refund-generation time in `refund_unspent_gas_and_deposits`, fall back to `receipt.predecessor_id()` if the `refund_to` account does not exist, mirroring the safe-fallback pattern already used for gas-key allowance refunds.

### Proof of Concept

1. Deploy a contract `malicious.near` with a method `trap_deposit`:
   - Creates a promise batch targeting `nonexistent-named-account.near` (a named account that does not exist).
   - Attaches the full caller deposit to the promise.
   - Calls `promise_set_refund_to(promise_idx, "burn-sink.near")` where `"burn-sink.near"` also does not exist.
   - The inner promise action is a `FunctionCall` to a method that will fail (or the target simply has no contract).
2. Victim calls `malicious.near::trap_deposit` with 10 NEAR attached.
3. The inner receipt to `nonexistent-named-account.near` fails with `AccountDoesNotExist`.
4. Runtime generates `Receipt::new_balance_refund("burn-sink.near", 10 NEAR)`.
5. That refund receipt fails because `"burn-sink.near"` does not exist.
6. 10 NEAR is permanently burnt; victim's balance is reduced with no recovery path.

The root cause is the absence of an existence check in `promise_set_refund_to` at: [7](#0-6)

    pub fn promise_set_refund_to(
        &mut self,
        promise_idx: u64,
        account_id_len: u64,
        account_id_ptr: u64,
    ) -> Result<()> {
        self.result_state.gas_counter.pay_base(base)?;
        if self.context.is_view() {
            return Err(HostError::ProhibitedInView {
                method_name: "promise_set_refund_to".to_string(),
            }
            .into());
        }
        let refund_to = self.read_and_parse_account_id(account_id_ptr, account_id_len)?;
        let promise = self
            .promises
            .get(promise_idx as usize)
            .ok_or(HostError::InvalidPromiseIndex { promise_idx })?;

        let receipt_idx = match &promise {
            Promise::Receipt(receipt_idx) => Ok(*receipt_idx),
            Promise::NotReceipt(_) => Err(HostError::CannotSetRefundToOnJointPromise),
        }?;

        self.ext.set_refund_to(receipt_idx, refund_to);
        Ok(())
    }

fn check_transfer_to_nonexisting_account(
    config: &RuntimeConfig,
    account_id: &AccountId,
    implicit_account_creation_eligible: bool,
) -> Result<(), ActionError> {
    if implicit_account_creation_eligible
        && account_is_implicit(account_id, config.wasm_config.eth_implicit_accounts)
    {
        // OK. It's implicit account creation.
        // Notes:
        // - Transfer action has to be the only action in the transaction to avoid
        // abuse by hijacking this account with other public keys or contracts.
        // - Refunds don't automatically create accounts, because refunds are free and
        // we don't want some type of abuse.
        // - Account deletion with beneficiary creates a refund, so it'll not create a
        // new account.
        Ok(())
    } else {
        Err(ActionErrorKind::AccountDoesNotExist { account_id: account_id.clone() }.into())
    }

    ctx: &mut Ctx,
    memory: &mut [u8],
    promise_idx: u64,
    account_id_len: u64,
    account_id_ptr: u64,

    ctx.result_state.gas_counter.pay_base(base)?;

        .into());
    }
    let refund_to = read_and_parse_account_id(
        &mut ctx.result_state.gas_counter,

        account_id_ptr,
        account_id_len,

    let promise = ctx
        .promises
        .get(promise_idx as usize)
        .ok_or(HostError::InvalidPromiseIndex { promise_idx })?;

    let receipt_idx = match &promise {
        Promise::Receipt(receipt_idx) => Ok(*receipt_idx),
        Promise::NotReceipt(_) => Err(HostError::CannotSetRefundToOnJointPromise),
    }?;

    ctx.ext.set_refund_to(receipt_idx, refund_to);
    Ok(())
}
