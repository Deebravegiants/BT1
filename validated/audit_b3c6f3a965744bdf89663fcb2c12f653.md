Audit Report

## Title
`protocolUnpausePool` Restores Pool to Admin-Paused State Instead of Active State — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`protocolUnpausePool` calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). Any pool that was fully active (`pauseLevel == 0`) before a protocol pause is left permanently in admin-paused state after the protocol "unpauses" it. The protocol owner has no unilateral path to restore the pool to active; only the pool admin can do so via `unpausePool`.

## Finding Description

The pause state machine has three levels: 0 = active, 1 = admin-paused, 2 = protocol-paused.

`protocolPausePool` accepts both level 0 and level 1 as valid source states before transitioning to level 2: [1](#0-0) 

`protocolUnpausePool` checks `cur == 2` and then unconditionally calls `setPause(1)`, landing in admin-paused regardless of what the pre-pause level was: [2](#0-1) 

`_checkNotPaused` rejects every non-zero pause level, so a pool at level 1 is fully blocked for swaps: [3](#0-2) 

The only path from level 1 back to level 0 is `unpausePool`, which is restricted to `onlyPoolAdmin`: [4](#0-3) 

The factory exposes no function for the protocol owner to set level 0 directly. The result is that a pool that was never admin-paused is driven through the cycle `0 → 2 → 1` and left in a state it was never in before, with no protocol-owner remedy.

## Impact Explanation

After a single protocol pause/unpause cycle on any live (level 0) pool, all swaps revert with `PoolPaused`. LP fee accrual stops. If the pool admin key is lost, rotated to an unresponsive address, or the admin refuses to cooperate, the pool is permanently stuck in admin-paused state. This constitutes broken core pool functionality causing unusable swap flows, matching the allowed impact gate.

## Likelihood Explanation

The condition triggers on every protocol pause/unpause cycle applied to a pool in its normal operational state (level 0), which is the default state for any live pool. No special setup, attacker, or privileged collusion is required — the protocol owner exercising their documented authority is sufficient to reproduce the bug deterministically.

## Recommendation

Preserve the pre-pause level when `protocolPausePool` is called and restore it on `protocolUnpausePool`. The simplest fix is to have `protocolUnpausePool` call `setPause(0)` and update the guard:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0); // restore to active
}
```

Alternatively, store the pre-pause level in factory state inside `protocolPausePool` and restore it here.

## Proof of Concept

1. Pool is deployed and active: `pauseLevel = 0`.
2. Protocol owner calls `protocolPausePool(pool)` — guard passes (`cur == 0`). Pool: `pauseLevel = 2`.
3. Protocol owner calls `protocolUnpausePool(pool)` — guard passes (`cur == 2`), sets `pauseLevel = 1`.
4. Any user calls `swap(...)` → `_checkNotPaused` reverts with `PoolPaused` because `pauseLevel == 1 != 0`.
5. Protocol owner has no further lever; `unpausePool` is `onlyPoolAdmin` and transitions only `1 → 0`.
6. If `poolAdmin[pool]` is unresponsive or lost, the pool is permanently paused despite the protocol owner having called "unpause".
