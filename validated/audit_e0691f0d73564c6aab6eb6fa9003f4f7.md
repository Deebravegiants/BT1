Audit Report

## Title
Missing Staleness Check on Chainlink `updatedAt` Timestamp Allows Stale Price Acceptance - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

## Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` retrieves the Chainlink `updatedAt` timestamp but never compares it to `block.timestamp`. The two existing guards (`answeredInRound < roundID` and `timestamp == 0`) do not detect a round that was answered but is now hours or days old. Any supported-token deposit into the RSETHPool family of contracts that uses this oracle as its collateral pricer will accept the stale price, allowing an attacker to receive more rsETH than the deposited value warrants and diluting existing rsETH holders.

## Finding Description
In `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 26–37, `getRate()` calls `latestRoundData()` and applies three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();   // only catches unanswered rounds
if (timestamp == 0) revert IncompleteRound();          // only catches zero timestamp
if (ethPrice <= 0) revert InvalidPrice();              // only catches non-positive price
```

None of these guards check `block.timestamp - timestamp > MAX_STALENESS`. A Chainlink feed can stop updating (e.g., during an L2 sequencer outage) while still satisfying `answeredInRound == roundID` and `timestamp != 0`, so all three guards pass and the stale price is returned.

This oracle is wired as the collateral pricer for supported tokens across the RSETHPool family. For example, `RSETHPoolV3.viewSwapRsETHAmountAndFee(amount, token)` calls `IOracle(supportedTokenOracle[token]).getRate()` to convert the deposited token amount into rsETH. If that rate is stale and inflated, the minted rsETH amount is proportionally overstated.

Exploit flow:
1. Chainlink collateral feed (e.g., wstETH/ETH) last reported `price = P_old` at `updatedAt = T`.
2. L2 sequencer goes offline; the collateral's market price drops to `P_new < P_old`.
3. Sequencer resumes. The feed has not yet posted a new round; `answeredInRound == roundID`, `timestamp == T` (stale). All three guards in `getRate()` pass; the function returns `P_old`.
4. Attacker calls `deposit(token, amount, referralId)` on any RSETHPool contract. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`, where `rsETHToETHrate` is derived from the stale `P_old`.
5. Attacker receives rsETH minted at the inflated rate. They immediately redeem or sell, extracting the difference `(P_old - P_new) / P_old` per unit deposited at the expense of existing rsETH holders.

## Impact Explanation
The overmint dilutes the backing pool shared by all rsETH holders: each existing holder's pro-rata claim on the pool's collateral is reduced. This constitutes **theft of unclaimed yield** (High severity) from existing rsETH holders. At sufficient scale or with a large price deviation, it can escalate to **protocol insolvency** (Critical severity) if the pool's collateral value no longer covers outstanding rsETH.

## Likelihood Explanation
L2 sequencer outages on Arbitrum, Optimism, and Base are documented historical events. During an outage, Chainlink feeds cannot post new rounds. An attacker monitoring sequencer status can submit the deposit transaction immediately upon sequencer resumption, before the oracle posts a new round. No privileged access is required; the `deposit` functions are fully public and permissionless. The window is bounded by the Chainlink heartbeat (typically 1 hour for ETH/USD on L2), making the attack repeatable whenever a sequencer outage coincides with a significant price move.

## Recommendation
Add a configurable maximum staleness threshold and validate `updatedAt` against `block.timestamp` in `getRate()`:

```solidity
uint256 public constant MAX_STALENESS = 3600; // 1 hour; tune per feed heartbeat

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 updatedAt, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice(); // ← add this
    if (ethPrice <= 0) revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

`MAX_STALENESS` should be set per-feed based on the Chainlink heartbeat (e.g., 3600 s for a 1-hour heartbeat feed, with a small buffer).

```solidity
// 1. Fork Arbitrum at a block where the wstETH/ETH Chainlink feed is live.

// 3. vm.warp(block.timestamp + 7200); // simulate 2-hour sequencer outage
//    (feed's updatedAt is now 2 hours in the past; no new round posted)
// 4. Call oracle.getRate() — it returns the pre-outage price without reverting.
// 5. Call RSETHPoolV3.deposit{value: 1 ether}("ref") — rsETHAmount is computed
//    using the stale rate; assert rsETHAmount > fair_value_at_current_price.
// 6. Confirm the delta represents extraction from existing rsETH holders.
```

The test requires no privileged setup: `deposit` is public, and `vm.warp` simulates the passage of time without a new oracle round being posted.
