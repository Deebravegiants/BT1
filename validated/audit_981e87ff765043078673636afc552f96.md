Audit Report

## Title
Stale Cross-Chain Rate in L2 Deposit Pools Allows Over-Minting of wrsETH, Causing Protocol Insolvency - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver.getRate()` returns a cached L2 rate with no staleness check. All L2 pool contracts (`RSETHPoolV2`, `RSETHPoolV3`, `RSETHPool`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) use this rate to compute `wrsETH` mint amounts. When the L1 rsETH price has increased but the L2 cached rate has not yet been updated via LayerZero, depositors receive more `wrsETH` than the deposited ETH can back on L1, creating a structural insolvency gap.

## Finding Description
The cross-chain rate system is push-based. On L1, `RSETHMultiChainRateProvider.updateRate()` reads `LRTOracle.rsETHPrice` and sends it via LayerZero to `CrossChainRateReceiver` on each L2.

`CrossChainRateReceiver` stores the rate and a `lastUpdated` timestamp but `getRate()` performs no staleness check:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L102-105
function getRate() external view returns (uint256) {
    return rate;
}
```

Every L2 pool calls `IOracle(rsETHOracle).getRate()` to price deposits, e.g. in `RSETHPoolV3.viewSwapRsETHAmountAndFee()`:

```solidity
// contracts/pools/RSETHPoolV3.sol L299-307
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The same pattern is confirmed in `RSETHPoolV2` (L225-233), `RSETHPool` (L311-319), `RSETHPoolV3ExternalBridge` (L418-426), and `RSETHPoolV3WithNativeChainBridge` (L335-343).

When `rsETHToETHrate` is stale (lower than the true L1 rate), the division yields a larger `rsETHAmount`, minting more `wrsETH` than the deposited ETH can back. The ETH is later bridged to L1 and deposited into `LRTDepositPool`, which mints rsETH at the **current** L1 rate:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If the L2 rate was stale (lower), the bridged ETH produces fewer rsETH on L1 than the `wrsETH` already minted on L2. The `RsETHTokenWrapper` backing is insufficient — `wrsETH` holders' redemption claims are diluted permanently.

There is no on-chain mechanism that atomically updates both the L1 oracle and the L2 receiver. The `lastUpdated` field exists in `CrossChainRateReceiver` but is never checked in `getRate()`. The daily mint limit (`dailyMintLimit`) in the pool contracts does not prevent over-minting per unit of ETH — it only caps total volume, not the per-unit rate accuracy.

## Impact Explanation
**Critical — Protocol insolvency.** Every deposit during a stale window mints more `wrsETH` than the deposited ETH can back in rsETH terms on L1. The shortfall is permanent: the `RsETHTokenWrapper` holds fewer rsETH than the outstanding `wrsETH` supply, diluting all existing `wrsETH` holders' redemption claims. Repeated exploitation accumulates an unbounded insolvency gap. This matches the allowed impact: **Critical — Protocol insolvency**.

## Likelihood Explanation
**Medium.** The rsETH/ETH rate increases continuously as EigenLayer staking rewards accrue. The rate update is entirely off-chain and push-based — `updateRate()` on `MultiChainRateProvider` must be called by an off-chain keeper, and the LayerZero message must be delivered and processed by `lzReceive()` on L2. Any gap between `updateRSETHPrice()` on L1 and LayerZero message delivery on L2 is a live exploitation window. This window is routine (it exists after every L1 price update) and requires no special permissions — any address can call `deposit()`. The daily mint limit provides partial mitigation on total volume but does not prevent the per-unit over-minting.

## Recommendation
1. **Short term:** Add a staleness threshold check in `CrossChainRateReceiver.getRate()`. Revert if `block.timestamp - lastUpdated` exceeds a configured maximum (e.g., 24 hours). This prevents deposits when the rate is known to be stale.
2. **Long term:** Consider a pull-based oracle model where the L2 pool fetches the rate on-demand from a trusted source, or implement a circuit-breaker that pauses deposits when the rate has not been updated within the expected window.

## Proof of Concept
1. L1 `LRTOracle.rsETHPrice` is updated from `1.05e18` to `1.10e18` via `updateRSETHPrice()`.
2. The off-chain keeper has not yet called `updateRate()`, so `CrossChainRateReceiver.rate` on L2 is still `1.05e18`.
3. Attacker calls `RSETHPoolV3.deposit{value: 1 ether}("ref")` on L2.
4. Pool computes: `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.952 wrsETH` (correct amount at true rate would be `1e18 * 1e18 / 1.10e18 ≈ 0.909 wrsETH`).
5. Attacker receives `~0.952 wrsETH` — approximately 4.7% more than the deposited ETH can back.
6. The 1 ETH is later bridged to L1 and deposited into `LRTDepositPool`, minting only `~0.909 rsETH` at the current L1 rate.
7. The `RsETHTokenWrapper` is short `~0.043 rsETH` per 1 ETH deposited during the stale window.
8. Existing `wrsETH` holders bear the loss through dilution of their redemption claims.

**Foundry fork test plan:**
- Fork mainnet + target L2 at a block where L1 price has just been updated but L2 `CrossChainRateReceiver.rate` has not yet been updated.
- Call `deposit()` on the L2 pool and record `wrsETH` minted.
- Simulate bridging the ETH to L1 and calling `LRTDepositPool.depositETH()`.
- Assert that rsETH minted on L1 < `wrsETH` minted on L2, confirming the shortfall.
