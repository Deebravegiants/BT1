### Title
Unverified Caller-Supplied `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Replay of Foreign-Chain Verification — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx()` verifies that a threshold signature is valid for a caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` was actually computed from the on-chain `request` data. A single malicious attested MPC participant (below the signing threshold) can replay a legitimately-generated `(payload_hash, signature)` pair from a prior request to resolve a completely different pending request, causing the contract to accept a forged foreign-chain verification.

---

### Finding Description

In `respond()` (the standard signing path), the payload hash is derived exclusively from the on-chain stored request — a trusted, contract-controlled source:

```rust
// crates/contract/src/lib.rs, line 600
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
``` [1](#0-0) 

In `respond_verify_foreign_tx()`, however, the payload hash is taken directly from the caller-supplied response — an untrusted source — and the contract only checks that the signature is valid for that hash, without ever verifying the hash was derived from `request.request`:

```rust
// crates/contract/src/lib.rs, lines 726-734
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [2](#0-1) 

The `ForeignTxSignPayload` that nodes are supposed to sign binds the request data to the extracted values:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
// msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [3](#0-2) 

The contract never recomputes this hash from `request.request` to confirm the response is bound to the correct request. After the signature check passes, `resolve_yields_for` drains the pending queue for `request` and delivers the response — including the attacker-supplied `payload_hash` — to all waiting callers: [4](#0-3) 

The `near-mpc-sdk`'s `ForeignChainSignatureVerifier::verify_signature` does perform this binding check on the client side:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [5](#0-4) 

But this is an optional SDK helper for callers — the on-chain contract itself provides no such guarantee. The contract is the authoritative verification oracle; its acceptance of a response is the security boundary that downstream bridge contracts rely on.

---

### Impact Explanation

**High — Forged foreign-chain verification / invalid bridge execution.**

A malicious attested participant can:

1. Observe a legitimately completed `respond_verify_foreign_tx` for request **R1** (Bitcoin tx T1), obtaining the valid pair `(H1, S1)` where `H1 = SHA256(borsh(ForeignTxSignPayload{request: R1, values: V1}))`.
2. Wait for a different pending request **R2** (e.g., a different Bitcoin tx T2) to appear in `pending_verify_foreign_tx_requests`.
3. Submit `respond_verify_foreign_tx(R2, {payload_hash: H1, signature: S1})`.
4. The contract checks: is `S1` a valid ECDSA signature over `H1` under the root key? **Yes** → `signature_is_valid = true`.
5. `resolve_yields_for` drains R2's queue and delivers `{payload_hash: H1, signature: S1}` to all callers waiting on R2.
6. R2 is permanently consumed from the pending map.

Callers of R2 receive a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to T1, not T2. Any bridge contract that does not independently re-verify the payload hash (using the SDK helper or equivalent logic) will treat T2 as verified when it was not. This enables double-spend or invalid bridge execution. Even callers that do re-verify are denied their legitimate response — R2 is consumed and cannot be re-answered.

---

### Likelihood Explanation

**Medium.** The attacker must be a single attested MPC participant (a Byzantine node below the signing threshold). This is explicitly within the allowed threat model. The attacker needs only to observe one completed `respond_verify_foreign_tx` call (publicly visible on-chain) and have a second request pending simultaneously — both routine conditions in a live bridge. No threshold collusion, key compromise, or TEE attack is required.

---

### Recommendation

The contract must independently compute the expected `payload_hash` from `request.request` and the extracted values provided in the response, then assert equality before accepting the response. Concretely, mirror the pattern used in `respond()` where the hash is derived from the on-chain request, not from the response:

```rust
// Reconstruct the expected payload hash from the request and the
// extracted values the node claims to have observed.
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // values must be in the response
});
let expected_hash = expected_payload.compute_msg_hash()
    .map_err(|_| RespondError::PayloadHashComputationFailed)?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This requires `VerifyForeignTransactionResponse` to also carry the `extracted_values` so the contract can recompute the hash. Alternatively, the `extracted_values` can be included in the `VerifyForeignTransactionRequest` at submission time (if the caller knows them in advance), allowing the contract to store and reuse them.

---

### Proof of Concept

**Setup:** Two pending requests, R1 (Bitcoin tx `[0x01;32]`) and R2 (Bitcoin tx `[0x02;32]`), both in `pending_verify_foreign_tx_requests`. The MPC network has already produced a valid response `(H1, S1)` for R1.

**Attack call (single malicious attested node):**

```rust
// Replay R1's response against R2
contract.respond_verify_foreign_tx(
    VerifyForeignTransactionRequest { /* R2 data */ tx_id: [0x02;32], ... },
    VerifyForeignTransactionResponse {
        payload_hash: H1,   // hash of R1's payload — NOT R2's
        signature: S1,      // valid threshold signature over H1
    },
);
```

**Contract execution path:**

1. `assert_caller_is_attested_participant_and_protocol_active()` — passes (attested node).
2. `verify_ecdsa_signature(S1, H1, root_pk)` — passes (S1 is a valid signature over H1).
3. `resolve_yields_for(&mut pending_verify_foreign_tx_requests, &R2, serialize({H1, S1}))` — R2's queue is drained; callers receive `{payload_hash: H1, signature: S1}`.

**Result:** R2 is permanently consumed. Callers of R2 receive a response whose `payload_hash` corresponds to T1 (Bitcoin tx `[0x01;32]`), not T2. The contract accepted a forged foreign-chain verification. [6](#0-5) [3](#0-2) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
