Audit Report

## Title
Token Deposits Default to Zero Fees, Allowing Bypass of ETH Deposit Fees - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains two separate fee variables: `feeBps` for ETH deposits and a per-token `tokenFeeBps[token]` mapping for ERC20 deposits. Because `addSupportedToken()` never initializes `tokenFeeBps[token]`, every newly added token has a zero fee by default. Any user can deposit a supported ERC20 token (e.g., wstETH) instead of ETH to receive the same `wrsETH` output while paying zero protocol fees, permanently depriving the protocol of intended fee revenue.

## Finding Description
`RSETHPool.sol` declares two distinct fee storage variables:

- `feeBps` (L43): applied to native ETH deposits via `viewSwapRsETHAmountAndFee(uint256)` at L312.
- `tokenFeeBps` (L88): a per-token mapping applied to ERC20 deposits via `viewSwapRsETHAmountAndFee(uint256, address)` at L335–336.

The token deposit path at L284–305 calls `viewSwapRsETHAmountAndFee(amount, token)`, which reads `tokenFeeBps[token]` directly:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token]; // L335
fee = amount * feeBpsForToken / 10_000;       // L336
```

However, `addSupportedToken()` at L637–656 only sets `supportedTokenOracle[token]` and `tokenBridge[token]`; it never writes to `tokenFeeBps[token]`:

```solidity
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] is never set
```

Because Solidity mappings default to `0`, every newly added token has `tokenFeeBps[token] == 0`, making `fee = amount * 0 / 10_000 = 0`. The only remediation path is a separate admin call to `setTokenFeeBps()` at L583–594, which is not enforced or required by `addSupportedToken()`.

The exploit path requires no special privileges: a user simply calls `deposit(token, amount, referralId)` (L284) instead of the ETH overload `deposit(referralId)` (L265), receiving the full `rsETHAmount` with zero fee deducted.

## Impact Explanation
This is a **High** severity finding: **Theft of unclaimed yield**. The protocol permanently loses fee revenue it intended to collect on ERC20 token deposits. Any depositor holding a supported token (e.g., wstETH) can route their deposit through the token path to receive more `wrsETH` than the ETH path would yield, with zero protocol fee paid. The fee revenue is not deferred — it is simply never collected, constituting a permanent loss of protocol yield.

## Likelihood Explanation
The condition is always present for any token added without a subsequent `setTokenFeeBps()` call. No flash loans, timing windows, or special privileges are required. The attacker only needs to hold a supported ERC20 token and call the public `deposit(token, amount, referralId)` function. The attack is repeatable indefinitely and is economically rational for any depositor who would otherwise pay a non-zero ETH deposit fee.

## Recommendation
Initialize `tokenFeeBps[token]` inside `addSupportedToken()` to prevent silent zero-fee defaults:

```solidity
function addSupportedToken(address token, address oracle, address bridge, uint256 _feeBps)
    external onlyRole(TIMELOCK_ROLE)
{
    // ... existing checks ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    tokenFeeBps[token] = _feeBps;
    // ...
}
```

Alternatively, audit all currently supported tokens and call `setTokenFeeBps()` for any that have not had their fee explicitly set.

## Proof of Concept
1. Protocol sets `feeBps = 30` (0.30% ETH deposit fee) via `setFeeBps(30)`.
2. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0`.
3. Alice holds wstETH equivalent to 10 ETH of value.
   - **ETH path**: `deposit{value: 10 ETH}("")` → `fee = 10e18 * 30 / 10000 = 0.03 ETH`; Alice receives `rsETHAmount` based on `9.97 ETH`.
   - **Token path**: `deposit(wstETH, wstETHAmount, "")` → `feeBpsForToken = tokenFeeBps[wstETH] = 0`; `fee = 0`; Alice receives `rsETHAmount` based on the full `10 ETH` equivalent.
4. Alice uses the token path, receiving more `wrsETH` and paying zero protocol fee.

**Foundry test sketch:**
```solidity
function test_zeroFeeTokenDeposit() public {
    // Setup: feeBps = 30, wstETH added via addSupportedToken (no setTokenFeeBps called)
    assertEq(pool.tokenFeeBps(wstETH), 0);
    
    uint256 amount = 1e18;
    deal(wstETH, alice, amount);
    vm.startPrank(alice);
    IERC20(wstETH).approve(address(pool), amount);
    pool.deposit(wstETH, amount, "");
    vm.stopPrank();
    
    // Fee earned in token should be 0 despite feeBps = 30
    assertEq(pool.feeEarnedInToken(wstETH), 0);
}
```
