Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison: the ERC20 branch correctly adds the incoming `amount` to `totalAssetDeposits` before comparing against the limit, but the ETH branch omits `amount` entirely. When total ETH deposits exactly equal the configured cap, the ETH check evaluates `limit > limit` → `false`, so `_beforeDeposit` does not revert and the deposit is accepted, pushing total ETH deposits past the cap.

## Finding Description
The root cause is at [1](#0-0)  — the ETH branch on line 679 uses `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` while the ERC20 branch on line 681 uses `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. The incoming `amount` is absent from the ETH comparison.

The call path is: `depositETH` [2](#0-1)  → `_beforeDeposit` [3](#0-2)  → `_checkIfDepositAmountExceedesCurrentLimit`. When `totalAssetDeposits == depositLimit`, the guard at line 661-663 does not revert, rsETH is minted, and `totalAssetDeposits` exceeds the cap by `msg.value`.

## Impact Explanation
The deposit limit is the protocol's primary on-chain cap for ETH inflows. The bypass allows `totalAssetDeposits` to exceed `depositLimitByAsset(ETH_TOKEN)` by up to one `msg.value`. No funds are lost or stolen; the protocol simply fails to enforce its promised deposit ceiling for ETH. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The condition `totalAssetDeposits == depositLimit` is a natural boundary reached as deposits accumulate. Any depositor monitoring on-chain state can observe this moment and call `depositETH` with any non-zero `msg.value`. No privileged access is required; the path is fully public and repeatable each time the limit is exactly met.

## Recommendation
Mirror the ERC20 branch in the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
1. Set `depositLimitByAsset(ETH_TOKEN)` to `100 ether`.
2. Accumulate prior deposits until `getTotalAssetDeposits(ETH_TOKEN) == 100 ether`.
3. Call `depositETH(0, "ref")` with `msg.value = 1 ether`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether)` evaluates `100 ether > 100 ether` → `false`.
5. `_beforeDeposit` does not revert; rsETH is minted to the caller.
6. `getTotalAssetDeposits(ETH_TOKEN)` is now `101 ether`, exceeding the configured limit.

**Foundry test sketch:**
```solidity
function test_ethLimitBypass() public {
    // set limit to 100 ether, fill to exactly 100 ether via prior deposits
    // then deposit 1 more ether — expect success (no revert)
    vm.deal(attacker, 1 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1 ether}(0, "ref");
    assertGt(lrtDepositPool.getTotalAssetDeposits(ETH_TOKEN), depositLimit);
}
```

    function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
