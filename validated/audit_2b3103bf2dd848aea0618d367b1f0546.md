### Title
Zero-Confirmation Bitcoin Transaction Signing Bypass via Unconstrained `BlockConfirmations` - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

### Summary

The `verify_foreign_transaction()` flow accepts a caller-supplied `BlockConfirmations(0)` for Bitcoin requests. Because neither the on-chain contract nor the MPC node enforces a minimum confirmation threshold, the MPC network will produce a valid threshold signature attesting to a Bitcoin transaction that has zero confirmations — i.e., a transaction that is still in the mempool and can be double-spent or reorganized out.

### Finding Description

`BlockConfirmations` is defined as a plain `u64` newtype with no minimum bound:

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI schema confirms `minimum: 0.0` is accepted. The on-chain `verify_foreign_transaction()` entry point performs no validation of the `confirmations` field — it only checks that the chain is supported and the domain purpose is `ForeignTx`.

On the node side, `execute_foreign_chain_request()` passes the caller-supplied value directly to the inspector:

```rust
let block_confirmations = request.confirmations.0.into();
let extracted_values = inspector
    .extract(transaction_id, block_confirmations, extractors)
    ...
```

Inside `BitcoinInspector::extract()`, the finality check is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

When `block_confirmations_threshold = 0`, the condition `0 <= any_u64` is always `true`. The check is trivially bypassed for every transaction, including mempool-only (0-confirmation) transactions. The node then proceeds to extract the block hash and sign the payload.

The signed payload (`ForeignTxSignPayloadV1`) includes the original request — which contains `confirmations: 0` — so the signature is cryptographically bound to the zero-confirmation request. However, bridge contracts consuming MPC attestations are designed to trust MPC signatures as the authoritative finality oracle; they are not expected to re-verify confirmation counts independently.

### Impact Explanation

An attacker can obtain a valid MPC threshold signature attesting that a Bitcoin transaction is "finalized" with zero confirmations. This signature is indistinguishable in structure from one produced for a 6-confirmation transaction. A bridge contract (e.g., an Omnibridge inbound flow) that relies on MPC attestations to release NEAR-side funds would accept this signature and release funds for a Bitcoin transaction that was never confirmed and can be double-spent. This is the exact double-spend scenario described in the Venus/LUNA incident referenced in the original report.

**Impact class:** High — forged foreign-chain verification enabling invalid bridge execution and double-spend conditions.

### Likelihood Explanation

The attack path is fully unprivileged: any account can call `verify_foreign_transaction()` with `confirmations: 0`. No special role, no threshold collusion, and no TEE bypass is required. The MPC nodes honestly execute the protocol — they are simply given a threshold of 0, which is always satisfied. The attack is deterministic and repeatable.

### Recommendation

1. **Contract-level enforcement**: In `verify_foreign_transaction()`, validate that `BitcoinRpcRequest.confirmations >= MINIMUM_BITCOIN_CONFIRMATIONS` (e.g., 1 or 6) and `env::panic_str` if the check fails.
2. **Type-level enforcement**: Replace `BlockConfirmations(pub u64)` with a validated newtype whose constructor rejects zero, so the invariant is enforced at deserialization time.
3. **Node-level defense-in-depth**: In `execute_foreign_chain_request()`, assert `block_confirmations >= 1` before calling the inspector, and return an error if the caller-supplied threshold is zero.

### Proof of Concept

1. Attacker broadcasts a Bitcoin transaction `TX` to the mempool (0 confirmations).
2. Attacker calls `verify_foreign_transaction()` on the MPC contract with:
   ```json
   {
     "request": {
       "Bitcoin": {
         "tx_id": "<TX hash>",
         "confirmations": 0,
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": <foreign_tx_domain>,
     "payload_version": 1
   }
   ```
3. MPC nodes query the Bitcoin RPC. `getrawtransaction` returns the mempool transaction with `confirmations = 0`. The check `0 <= 0` passes.
4. `verify_block_is_canonical` is called on the `blockhash` field. For a mempool transaction, `blockhash` may be absent or zero — depending on the RPC implementation this either errors or returns a zero hash. If the RPC returns a valid (pending) block hash, the canonical check also passes.
5. MPC nodes sign `SHA-256(borsh(ForeignTxSignPayload { request: Bitcoin{confirmations:0,...}, values: [BlockHash(...)] }))` and submit `respond_verify_foreign_tx()`.
6. Attacker receives a valid MPC signature and presents it to a bridge contract to claim NEAR-side funds.
7. Attacker double-spends `TX` on Bitcoin (e.g., via RBF or a reorg), keeping both the Bitcoin funds and the NEAR-side funds.
