### Title
Unvalidated CKD Response Accepted for `AppPublicKey` Variant in `respond_ckd` - (File: `crates/contract/src/lib.rs`)

### Summary
`respond_ckd` applies a cryptographic pairing check (`ckd_output_check`) only when the request carries `CKDAppPublicKey::AppPublicKeyPV`. When the request carries `CKDAppPublicKey::AppPublicKey`, the branch is an explicit no-op and any arbitrary `CKDResponse` is accepted and delivered to the caller. A single malicious leader node (strictly below the signing threshold) can therefore submit a fraudulent key-derivation output that the contract will accept and forward to the user.

### Finding Description

In `respond_ckd` the output-validation logic is:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant the contract verifies the pairing equation `e(C, G2) = e(Y, pk2) · e(H(pk, app_id), network_pk)` before resolving the yield. For the `AppPublicKey` variant no such check exists; the contract immediately calls `resolve_yields_for` with whatever `(big_y, big_c)` the responding node supplied. [2](#0-1) 

The existing unit test for the non-PV path confirms this: it passes `big_y = [1u8; 48]` and `big_c = [2u8; 48]` — values that are not valid BLS12-381 G1 points and certainly do not satisfy any CKD relation — and the call succeeds. [3](#0-2) 

The `ckd_output_check` function that is correctly applied for the PV variant verifies the full pairing relation using the network public key and the app's `pk2`. [4](#0-3) 

### Impact Explanation

The CKD protocol is designed so that the derived key `sig = msk · H(pk, app_id)` is a function of the MPC network's master secret and the caller's identity. A malicious leader node can submit any `(big_y, big_c)` pair; the contract resolves the yield with that pair, and the user's application computes `sig = big_c − a · big_y` from it. Because the leader does not know the user's secret scalar `a`, the leader cannot predict the exact key the user will derive, but the leader can ensure the key is **not** the legitimate `msk · H(pk, app_id)`. The user's application silently operates with a wrong key — one that was never produced by the threshold MPC computation. This constitutes unauthorized/incorrect confidential key derivation output delivered without the required threshold-participant authorization, matching the Critical impact class.

### Likelihood Explanation

Every CKD request that uses the `AppPublicKey` variant (the default, non-PV path used in all non-PV e2e tests and the SDK) is vulnerable. The attack requires only that the elected leader for that request be a single malicious participant — strictly below the signing threshold. No collusion with other nodes is needed. The leader role rotates, so any participant can eventually be elected leader for a target request.

### Recommendation

Apply the same style of on-chain output validation to the `AppPublicKey` branch. For the non-PV variant the contract holds `pk1 = a · G1` but not `pk2 = a · G2`, so the full pairing check used in `ckd_output_check` cannot be reproduced on-chain. Two mitigations are possible:

1. **Require `AppPublicKeyPV` for all CKD requests** so that on-chain verification is always possible.
2. **Collect threshold-many independent `respond_ckd` calls** (one per participant) and accept the response only when a quorum of participants submit the same `(big_y, big_c)` pair, analogous to how `vote_pk` works for key generation.

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)`.
2. The MPC network elects a leader. The malicious leader skips the real CKD computation.
3. The malicious leader calls `respond_ckd(ckd_request, CKDResponse { big_y: [0u8;48], big_c: [0u8;48] })`.
4. The contract executes line 676 (`AppPublicKey(_) => {}`), skips all validation, and calls `resolve_yields_for` — delivering the fraudulent response to the user.
5. The user's application computes `sig = big_c − a · big_y` from the attacker-supplied values, obtaining a key that has no relation to `msk · H(pk, app_id)`.

The existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates step 3–4 with `big_y = [1u8;48]` and `big_c = [2u8;48]` and asserts success. [5](#0-4)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
