### Title
Stale Cert State in Dijkstra LEDGER Rule Allows ADA Creation via Sub-Transaction Deposit Manipulation - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`)

---

### Summary

In the Dijkstra era `dijkstraLedgerTransition`, sub-transactions are processed first (updating both UTxO state and cert state), then the main transaction's certificates are processed by `ENTITIES` using the **updated** cert state (`certStateAfterSubLedgers`). However, the subsequent `UTXOW`/`UTXO` call — which performs the preservation-of-value balance check and the deposit-pot accounting update — is passed the **original** cert state (`lsCertState ledgerState`). This ordering mismatch allows an attacker to cause the deposit pot to be inflated while the corresponding deposit is simultaneously returned to a reward account, creating ADA out of thin air.

---

### Finding Description

In `dijkstraLedgerTransition`, the execution order is:

1. **SUBLEDGERS** processes sub-transactions, producing `utxoStateAfterSubLedgers` and `certStateAfterSubLedgers`.
2. **ENTITIES** processes the main transaction's certificates using `certStateAfterSubLedgers`.
3. **UTXOW → UTXO** is called with the **original** cert state:

```haskell
-- Call UTXOW with DijkstraUtxoEnv, passing the original UTxO and original certState
utxoStateFinal <-
  trans @(EraRule "UTXOW" era) $
    TRC
      ( DijkstraUtxoEnv slot pp (lsCertState ledgerState) originalUtxo
      , utxoStateBeforeUtxow
      , stAnnTx
      )
``` [1](#0-0) 

Inside `dijkstraUtxoTransition`, both the preservation-of-value check and the deposit-pot update use this stale cert state:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
...
Babbage.updateUTxOStateByTxValidity
    pp
    certState          -- original cert state, not certStateAfterENTITIES
    (utxosGovState utxos)
    tx
    (Conway.updateTreasuryDonation tx utxos)
``` [2](#0-1) [3](#0-2) 

`updateUTxOStateNoFees` computes the deposit-pot delta using the cert state from the environment:

```haskell
totalRefunds = certsTotalRefundsTxBody pp certState txBody
totalDeposits = certsTotalDepositsTxBody pp certState txBody
depositChange = totalDeposits <-> totalRefunds
...
utxosDeposited = utxosDeposited <> depositChange
``` [4](#0-3) 

Because `certState` is the **original** cert state (before sub-transactions), any credential registered by a sub-transaction is invisible to this calculation. `certsTotalRefundsTxBody` calls `lookupDepositDState` per credential; if the credential is absent from the original cert state, the refund is zero and `depositChange = 0`, so `utxosDeposited` is never decreased.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

Concrete accounting trace (deposit amount D, fees omitted for clarity):

| Step | Action | UTxO | Deposit Pot | Reward Acct C | Total |
|------|--------|------|-------------|---------------|-------|
| Initial | — | 1100 | 0 | 0 | 1100 |
| Sub-tx registers C, pays D | SUBLEDGERS | 1100−D | D | 0 | 1100 |
| Main tx ENTITIES: UnRegCert C D | deposit returned to reward acct | 1100−D | D | +D | 1100+D |
| Main tx UTXO update (original cert state) | depositChange=0 | 1100−D | D (unchanged) | D | 1100+D |

The deposit pot retains D while the reward account also holds D — D ADA is created from nothing. The attacker can subsequently withdraw D from the reward account in a normal withdrawal transaction, permanently extracting the created ADA.

---

### Likelihood Explanation

**High.** The Dijkstra era is the current head era (protocol version 12). The attack requires only:
- Constructing a top-level transaction with an embedded sub-transaction (a standard Dijkstra feature).
- The sub-transaction registers a stake credential.
- The main transaction includes `UnRegCert` for the same credential.

No privileged access, governance majority, or leaked keys are required. The attacker controls all inputs and can repeat the attack for arbitrarily large D (bounded only by the deposit parameter and available UTxO).

---

### Recommendation

Pass `certStateAfterENTITIES` (or `certStateFinal`) — not `lsCertState ledgerState` — to the `UTXOW`/`UTXO` call, so that the preservation-of-value check and the deposit-pot update reflect the cert state that was actually used when processing the main transaction's certificates. Specifically, change line 439 of `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`:

```haskell
-- Before (buggy):
DijkstraUtxoEnv slot pp (lsCertState ledgerState) originalUtxo

-- After (fixed):
DijkstraUtxoEnv slot pp certStateFinal originalUtxo
``` [5](#0-4) 

This ensures that `certsTotalRefundsTxBody` sees the credential as registered (because the sub-transaction registered it), correctly computes `depositChange = 0 − D = −D`, and decrements `utxosDeposited` by D to match the deposit returned to the reward account by `ENTITIES`.

---

### Proof of Concept

1. Attacker holds UTxO entry `A` (≥ 2·D + fees).
2. Attacker constructs a Dijkstra top-level transaction `T` with:
   - **Sub-transaction** `S`: inputs `A[0..D+fee_s]`, outputs `A'[0..A-D-fee_s]`, certificate `RegCert C D` (registers fresh credential `C` with deposit `D`).
   - **Main transaction body**: inputs `A[D+fee_s..]`, outputs `A''`, certificate `UnRegCert C D`.
3. Submit `T`. The ledger:
   - Runs `SUBLEDGERS(S)`: `utxosDeposited += D`, `C` registered in `certStateAfterSubLedgers`.
   - Runs `ENTITIES` on main tx certs using `certStateAfterSubLedgers`: `C` deregistered, reward account of `C` credited `D`.
   - Runs `UTXO` with original cert state: `depositChange = 0`, `utxosDeposited` stays at `D`.
4. Attacker submits a second transaction withdrawing `D` from `C`'s reward account.
5. Net gain: `D` ADA extracted; `utxosDeposited` permanently inflated by `D`, violating preservation of value.
