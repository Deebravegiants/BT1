Audit Report

## Title
Precision Loss in `rewardPerToken()` Permanently Destroys Unclaimed Yield — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
Integer division in `rewardPerToken()` truncates the per-token increment to zero whenever `rewardRate * elapsed * DECIMAL_PRECISION < totalKernelStaked`. The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when the computed increment is zero, permanently consuming those elapsed seconds. The affected reward tokens remain locked in the contract with no rescue or recovery mechanism.

## Finding Description
`rewardPerToken()` computes:

```solidity
rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
``` [1](#0-0) 

When `rewardRate * elapsed * 1e18 < totalKernelStaked`, Solidity integer division truncates the result to `0`, so `rewardPerToken()` returns `rewardPerTokenStored` unchanged.

The `updateReward` modifier then unconditionally writes:

```solidity
rewardPerTokenStored = rewardPerToken();   // unchanged (still 0 increment)
updatedAt = lastTimeRewardApplicable();    // advances past the zero-increment window
``` [2](#0-1) 

Because `updatedAt` advances regardless of whether any increment was recorded, the elapsed seconds that produced a zero increment are permanently consumed. Every subsequent call to any `updateReward`-gated function (`stake`, `stakeFor`, `initiateWithdrawal`, `getReward`) repeats this cycle.

`rewardRate` is set as:

```solidity
rewardRate = receivedAmount / duration;
``` [3](#0-2) 

The only guard is `rewardRate == 0`; a `rewardRate` of 1 or any small value passes. There is no sweep, rescue, or emergency-withdrawal function for reward tokens in the contract. [4](#0-3) 

**Concrete example:**
- `rewardRate = 1`, `totalKernelStaked = 2e18`, `elapsed = 1s`
- Increment = `(1 * 1 * 1e18) / 2e18 = 0` → truncated
- `updatedAt` advances 1 second; that second's reward is permanently lost
- Repeated every block for the entire reward duration

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens deposited via `notifyRewardAmount` that fall into truncated windows are permanently locked in the contract. No user can ever claim them, and no admin function exists to recover them. The staked KERNEL principal is unaffected; only the yield is destroyed.

## Likelihood Explanation
The condition is reachable without any privileged compromise. The admin sets `rewardRate` via `notifyRewardAmount` (a routine operational action, not an attack). Any unprivileged user can increase `totalKernelStaked` by staking, worsening the truncation condition. As protocol TVL grows naturally, the ratio `totalKernelStaked / (rewardRate * 1e18)` increases, making per-second truncation more likely over the protocol's lifetime. The condition is not a one-time edge case; it worsens monotonically with adoption.

## Recommendation
Scale `rewardRate` by `DECIMAL_PRECISION` at the point it is set in `notifyRewardAmount`, deferring the division to `rewardPerToken()`:

```solidity
// In notifyRewardAmount:
rewardRate = (receivedAmount * DECIMAL_PRECISION) / duration;

// In rewardPerToken():
return rewardPerTokenStored + (rewardRate * elapsed) / totalKernelStaked;
```

This shifts precision loss to a magnitude 1e18× smaller. Alternatively, accumulate a `rewardPerTokenRemainder` and only advance `updatedAt` when the accumulated numerator meets or exceeds `totalKernelStaked`.

## Proof of Concept
Deploy `KernelDepositPool`, stake `2e18` KERNEL (Alice), call `notifyRewardAmount(2_592_000)` with `duration = 30 days` → `rewardRate = 1`. Warp forward 1 second and call `pool.getReward()` (triggers `updateReward`). Repeat 100 times. Assert `pool.earned(alice) == 0` and `rewards[alice] == 0` after 100 seconds. The 100 reward-seconds (`rewardRate=1`, 100s) that should have produced `~100 wei` are permanently destroyed because each 1-second window truncates to zero and `updatedAt` advances past it. The same result holds at `rewardRate = 1000` with `totalKernelStaked = 2e21`.
