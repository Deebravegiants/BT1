### Title
Hardcoded `Felt::ZERO` for L2 Gas in Receipt Commitment Produces Wrong Block Hash - (File: `crates/starknet_api/src/block_hash/receipt_commitment.rs`)

### Summary

`chain_gas_consumed` in `receipt_commitment.rs` always chains a constant `Felt::ZERO` for the L2 gas consumed field instead of reading the actual `gas_consumed.l2_gas` value from the `GasVector`. For any block containing transactions that consume non-zero L2 gas (present in Starknet ≥ v0.14.x), this produces a wrong `receipt_commitment`, which propagates into `BlockHeaderCommitments`, `PartialBlockHashComponents`, `PartialBlockHash`, and ultimately the `ProposalCommitment` that consensus validators compare.

### Finding Description

In `crates/starknet_api/src/block_hash/receipt_commitment.rs`, the private function `chain_gas_consumed` is responsible for appending the three gas-consumed fields to the Poseidon hash chain that forms each transaction's receipt leaf:

```rust
// crates/starknet_api/src/block_hash/receipt_commitment.rs  lines 86-91
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO) // L2 gas consumed   ← always zero, ignores gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
``` [1](#0-0) 

The `GasVector` struct passed in carries a live `l2_gas` field:

```rust
pub struct GasVector {
    pub l1_gas: GasAmount,
    pub l1_data_gas: GasAmount,
    pub l2_gas: GasAmount,   // ← populated from actual execution
}
``` [2](#0-1) 

The `gas_consumed` value fed into `chain_gas_consumed` originates from `TransactionOutputForHash.gas_consumed`, which is populated by `output_for_hashing()` on the blockifier's `TransactionExecutionInfo`. The batcher's `prepare_txs_hashing_data` calls this directly:

```rust
transaction_output: info.output_for_hashing(),
``` [3](#0-2) 

The `l2_gas_used()` helper on the same execution data confirms that `l2_gas` is non-zero in practice:

```rust
res = res.checked_add(execution_info.receipt.gas.l2_gas).expect("Total L2 gas overflow.");
``` [4](#0-3) 

Real network data in `block_post_0_14_3.json` shows `total_gas_consumed.l2_gas: 76014735` for transactions in Starknet v0.14.3 blocks, confirming the field is non-zero in production. The comment "In the current RPC: always 0" is therefore stale.

The wrong receipt commitment propagates through the full commitment pipeline:

1. `calculate_receipt_commitment` → wrong `ReceiptCommitment`
2. `calculate_block_commitments` → wrong `BlockHeaderCommitments.receipt_commitment`
3. `PartialBlockHashComponents::new` → wrong `header_commitments`
4. `PartialBlockHash::from_partial_block_hash_components` → wrong `PartialBlockHash`
5. `BlockExecutionArtifacts::commitment()` → wrong `ProposalCommitment` [5](#0-4) [6](#0-5) 

The `ProposalCommitment` is the value that consensus validators compare in `ProposalFin`:

```rust
pub struct ProposalFin {
    pub proposal_commitment: ProposalCommitment,
    ...
}
``` [7](#0-6) 

Because both proposer and validator execute the same buggy `chain_gas_consumed`, they agree on the wrong commitment. The final block hash written to storage and used by the Starknet OS/prover is therefore wrong for every block with non-zero L2 gas.

### Impact Explanation

**Critical — Wrong receipt commitment and block hash for accepted blocks.**

For every block produced on Starknet ≥ v0.14.x (where L2 gas is non-zero), the `receipt_commitment` stored in the block header is computed with `l2_gas = 0` instead of the actual consumed value. This wrong commitment is chained into the `PartialBlockHash` / `ProposalCommitment` and into the final `BlockHash` computed by the committer. The Starknet OS and prover, which use the actual L2 gas values, will compute a different block hash, causing proof verification failures and an incorrect authoritative block hash visible through RPC.

### Likelihood Explanation

**High.** Any transaction that consumes L2 gas (all v3 transactions on Starknet ≥ v0.13.4 with `GasVectorComputationMode::All`) triggers the wrong path. Real mainnet blocks (e.g., `block_post_0_14_3.json`) already show `l2_gas: 76014735`. No version guard exists in `chain_gas_consumed` or `calculate_receipt_commitment` to skip the wrong zero for older blocks. No privileged access is required; any ordinary user transaction suffices.

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual `l2_gas` field from the `GasVector`:

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&gas_consumed.l2_gas.into())   // actual L2 gas consumed
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

If the Starknet protocol intentionally specifies zero for L2 gas in the receipt hash for a specific version range, add an explicit version-gated branch (matching the pattern already used for the transaction commitment in `calculate_block_commitments`) rather than silently discarding the field.

### Proof of Concept

1. Submit any v3 invoke transaction on Starknet ≥ v0.14.x. The transaction will consume non-zero L2 gas (e.g., `l2_gas: 76014735` as seen in `block_post_0_14_3.json`).
2. The batcher calls `prepare_txs_hashing_data` → `output_for_hashing()`, producing a `TransactionOutputForHash` with `gas_consumed.l2_gas = 76014735`.
3. `calculate_block_commitments` calls `calculate_receipt_commitment`, which calls `calculate_receipt_hash` → `chain_gas_consumed`.
4. `chain_gas_consumed` chains `Felt::ZERO` instead of `76014735`, producing a wrong receipt leaf hash.
5. The wrong leaf propagates to a wrong `ReceiptCommitment`, wrong `PartialBlockHashComponents`, wrong `PartialBlockHash`, and wrong `ProposalCommitment`.
6. The Starknet OS, computing the same receipt hash with the actual `76014735`, arrives at a different block hash, causing proof verification to fail and the on-chain block hash to be incorrect.
