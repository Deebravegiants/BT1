## Code Trace

**`get_peak_block`** (wallet_blockchain.py:188-195) returns `self._peak` directly from memory if it is not `None`, only falling back to a DB read when `_peak` is `None`: [1](#0-0) 

**`new_valid_weight_proof`** (wallet_blockchain.py:81-103) reads the peak, performs the weight check, and only then enters the writer lock: [2](#0-1) 

**`set_peak_block`** updates `self._peak` only inside the writer block: [3](#0-2) 

**`DBWrapper2.writer`** acquires `self._lock` (an `asyncio.Lock`) at line 392, which is the first point of mutual exclusion — but this is *after* the weight check has already been evaluated: [4](#0-3) 

## Race Condition Analysis

The weight check (line 84) is **outside** the writer lock (line 88). The sequence that produces the bug:

1. Task A (`weight=200`) calls `new_valid_weight_proof` → `get_peak_block()` → `self._peak` is `None` → DB read → **yields**
2. Task B (`weight=150`) calls `new_valid_weight_proof` → `get_peak_block()` → `self._peak` is still `None` → DB read → **yields**
3. Task A resumes: `peak=None`, passes weight check (200 > None)
4. Task A enters `async with db_wrapper.writer()` → acquires `_lock` (uncontested, no yield) → enters `_savepoint_ctx` → `await self._write_connection.execute("SAVEPOINT s0")` → **yields**
5. Task B resumes: `peak=None` (its local variable from step 2), passes weight check (150 > None)
6. Task B tries `async with db_wrapper.writer()` → `_lock` held by Task A → **yields/waits**
7. Task A completes all writes, calls `set_peak_block` → sets `self._peak = heavier_block`, releases `_lock`
8. Task B acquires `_lock`, writes lighter proof, calls `set_peak_block` → **sets `self._peak = lighter_block`** → **BUG**

The invariant is violated: `PEAK_BLOCK`, `SYNCED_WEIGHT_PROOF`, `SUB_SLOT_ITERS`, and `DIFFICULTY` all now correspond to the lighter chain.

## Reachability

`long_sync_from_untrusted` is called per-peer with no outer lock: [5](#0-4) 

The guard against a second secondary sync (`_secondary_peer_sync_task`) is only checked when `syncing=False`. When both peers trigger `far_behind=True`, both proceed with `syncing=True` and both call `fetch_and_update_weight_proof` → `new_valid_weight_proof` concurrently: [6](#0-5) 

An attacker controlling two peers presenting valid (but different-weight) proofs simultaneously can trigger this. The weight proofs must be cryptographically valid, but the attacker does not need to forge anything — they can present two real chain tips.

## Verdict

### Title
TOCTOU Race in `new_valid_weight_proof` Allows Lighter Weight Proof to Overwrite Heavier Peak — (`chia/wallet/wallet_blockchain.py`)

### Summary
A time-of-check/time-of-use race in `WalletBlockchain.new_valid_weight_proof` allows two concurrent weight proof tasks to both pass the weight guard before either commits, enabling a lighter proof to overwrite the heavier one as the wallet's canonical peak.

### Finding Description
The weight comparison at line 84 reads `self._peak` via `get_peak_block()` **before** the `db_wrapper.writer()` lock is acquired at line 88. Because `asyncio` yields at the `SAVEPOINT` execute inside `_savepoint_ctx`, a second concurrent task can read the same stale `_peak`, pass the same weight check, and then queue behind the first task's lock. When the first (heavier) task releases the lock and updates `self._peak`, the second (lighter) task acquires the lock and overwrites `PEAK_BLOCK`, `SYNCED_WEIGHT_PROOF`, `SUB_SLOT_ITERS`, and `DIFFICULTY` with the lighter chain's values.

### Impact Explanation
The wallet's `_peak` and all four persisted keys are left pointing to a lighter fork. Subsequent coin-record subscriptions and sync operations use this corrupted state as their baseline, causing the wallet to miss coin records that exist only on the heavier chain. This is a direct corruption of wallet sync state with security impact (incorrect balance, missed received coins, incorrect spend history).

### Likelihood Explanation
Requires an attacker to control two peers that simultaneously deliver valid weight proofs to the same wallet node while it is in long-sync mode (`far_behind=True`). No cryptographic forgery is needed. The timing window is wide because `validate_weight_proof_inner` runs for multiple seconds (spawning a `ProcessPoolExecutor`), making it straightforward to arrange concurrent arrival.

### Recommendation
Move the peak read and weight comparison **inside** the `db_wrapper.writer()` lock so the check and the commit are atomic:

```python
async def new_valid_weight_proof(self, weight_proof: WeightProof, records: list[BlockRecord]) -> None:
    async with self._basic_store.db_wrapper.writer():
        peak: HeaderBlock | None = await self.get_peak_block()
        if peak is not None and weight_proof.recent_chain_data[-1].weight <= peak.weight:
            return None
        # ... rest of writes ...
```

### Proof of Concept
Spawn two asyncio tasks against the same `WalletBlockchain` instance, each calling `new_valid_weight_proof` with a different valid weight proof (A heavier, B lighter). Use `asyncio.gather`. After both complete, assert `(await chain.get_peak_block()).weight` equals the heavier proof's weight. The assertion will fail, demonstrating that the lighter proof won the last write.
