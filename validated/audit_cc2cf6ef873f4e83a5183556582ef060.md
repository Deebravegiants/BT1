Audit Report

## Title
Uninitialized `rsETHPrice` Causes Division-by-Zero, Freezing All Deposits — (`contracts/LRTDepositPool.sol`)

## Summary
`LRTOracle.initialize()` does not set `rsETHPrice`, leaving it at its Solidity default of `0`. `LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` without a zero-guard, causing every call to `depositETH()` and `depositAsset()` to revert with a division-by-zero error until `updateRSETHPrice()` is explicitly called. This temporarily freezes all user deposits on every fresh deployment or upgrade.

## Finding Description
`LRTOracle` declares `rsETHPrice` as a plain storage variable with no initialization:

```solidity
uint256 public override rsETHPrice; // LRTOracle.sol:28
```

`LRTOracle.initialize()` only sets `lrtConfig` and emits an event; it never calls `_updateRsETHPrice()` or assigns a non-zero value to `rsETHPrice`. [1](#0-0)  The variable is only written inside `_updateRsETHPrice()`, which is invoked by the public `updateRSETHPrice()`. When `rsethSupply == 0` (the state at deployment), `_updateRsETHPrice()` would set `rsETHPrice = 1 ether`, but this only happens if the function is explicitly called. [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` unconditionally divides by this value:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

`getRsETHAmountToMint()` is called by `_beforeDeposit()`, which is invoked by both `depositETH()` (line 87) and `depositAsset()` (line 111) — the two primary user-facing deposit entry points. [4](#0-3)  When `rsETHPrice == 0`, Solidity 0.8's checked arithmetic causes the division to revert, blocking all deposits. No existing check prevents this: neither `_beforeDeposit()` nor `getRsETHAmountToMint()` guards against a zero `rsETHPrice`.

The secondary path claimed (withdrawal fund loss via `getExpectedAssetAmount()` returning `0`) requires `rsETHPrice == 0` while rsETH is already in circulation. Since deposits require `rsETHPrice != 0`, the only way to reach that state post-mint is through complete slashing of all protocol ETH — an external protocol event outside the scope of this codebase. That path is therefore not validated here.

## Impact Explanation
**Medium — Temporary freezing of funds.** All calls to `depositETH()` and `depositAsset()` revert with division-by-zero whenever `rsETHPrice == 0`. This blocks the entire deposit surface until an operator calls `updateRSETHPrice()`. No funds are permanently lost, but the protocol is non-functional for deposits during this window, which is a concrete, reproducible denial of service on the primary user-facing entry points.

## Likelihood Explanation
This is a near-certain transient condition on every fresh deployment or upgrade: `rsETHPrice` starts at `0` and remains there until `updateRSETHPrice()` is explicitly called as a post-deployment step. Any user who attempts a deposit in this window will have their transaction revert. No attacker action is required; the condition arises purely from the deployment sequence. The window closes only when an operator manually calls `updateRSETHPrice()`.

## Recommendation
Initialize `rsETHPrice` in `LRTOracle.initialize()` by setting `rsETHPrice = 1 ether` (matching the value `_updateRsETHPrice()` assigns when `rsethSupply == 0`), or by calling `_updateRsETHPrice()` directly. Additionally, add an explicit zero-guard in `getRsETHAmountToMint()`:

```solidity
uint256 currentRsETHPrice = lrtOracle.rsETHPrice();
if (currentRsETHPrice == 0) revert InvalidRsETHPrice();
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / currentRsETHPrice;
```

## Proof of Concept
1. Deploy `LRTOracle` and `LRTDepositPool` without calling `updateRSETHPrice()`.
2. `rsETHPrice` is `0` (Solidity default for uninitialized `uint256`). [5](#0-4) 
3. Any user calls `depositETH{value: 1 ether}(0, "")`.
4. Execution flows: `depositETH` → `_beforeDeposit` → `getRsETHAmountToMint` → `(1e18 * assetPrice) / 0` → Solidity 0.8 reverts with division-by-zero. [3](#0-2) 
5. All deposits are blocked until an operator calls `updateRSETHPrice()`.

**Foundry test sketch:**
```solidity
function test_depositRevertsWhenRsETHPriceIsZero() public {
    // Deploy LRTOracle and LRTDepositPool without calling updateRSETHPrice()
    // assertEq(lrtOracle.rsETHPrice(), 0);
    vm.expectRevert(); // division by zero
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
}
```
