### Title
Single Byzantine Participant Can Permanently DoS All `verify_foreign_transaction` Requests via Empty Chain Re-registration — (`File: crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which uses a strict all-participant intersection rule. A single participant (well below the signing threshold) can call `register_foreign_chain_support` with an empty set at any time, immediately resetting the supported-chain set to empty and causing every subsequent `verify_foreign_transaction` call to revert. This is the direct analog of the bonding-token DoS: a state condition that should be stable after setup can be reset by a single low-privilege actor.

### Finding Description

`verify_foreign_transaction` checks whether the requested chain is in the supported set before enqueuing the yield request:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(...)
}
``` [1](#0-0) 

`get_supported_foreign_chains()` computes the **strict intersection** of every active participant's registered chains — a chain is included only if **all** active participants have registered it:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
if all_active_nodes_supports_chain { Some(foreign_chain) } else { None }
``` [2](#0-1) 

Any participant can re-register their chain support at any time with an arbitrary (including empty) set via `register_foreign_chain_support`, which simply overwrites the stored entry with no lower bound:

```rust
pub fn register_foreign_chain_support(
    &mut self,
    foreign_chain_support: dtos::SupportedForeignChains,
) -> Result<(), Error> {
    let account_id = self.voter_or_panic();
    self.node_foreign_chain_support
        .foreign_chain_support_by_node
        .insert(account_id, foreign_chain_support);
    Ok(())
}
``` [3](#0-2) 

Because the intersection rule requires unanimity, a single participant submitting `SupportedForeignChains(BTreeSet::new())` immediately collapses `get_supported_foreign_chains()` to the empty set, causing every `verify_foreign_transaction` call to panic with `ForeignChainNotSupported` regardless of which chain is requested.

The project's own design documentation explicitly identifies this as the known flaw of the current implementation:

> "Today, `get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains, and `verify_foreign_transaction` rejects any request whose target chain is not in it. A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down."



The proposed fix (`get_available_foreign_chains()` with a threshold-based count) exists only as a design document and has **not** been applied to `verify_foreign_transaction` in the production contract code.

### Impact Explanation

Every `verify_foreign_transaction` request for every foreign chain is permanently rejected until the attacker re-registers with a non-empty set. No pending requests can be enqueued; any already-enqueued yield requests that time out are lost. This breaks the entire foreign-chain transaction verification flow — a core production feature — without requiring any threshold collusion, key compromise, or network-level attack.

**Impact: Medium** — request-lifecycle and contract execution-flow manipulation that breaks production safety invariants for the foreign-chain bridge path.

### Likelihood Explanation

Requires exactly one participant to act maliciously or be compromised. The participant set is typically small (e.g., 7–10 nodes on mainnet), and a single compromised node key is sufficient. The call is cheap, requires no deposit, and can be repeated after any honest re-registration. The attacker does not need to be above the signing threshold.

**Likelihood: Medium**

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which uses the threshold-based count already implemented in `ForeignChainsMetadata::update_available_chains_config_cache`. A chain is available only when ≥ `signing_threshold` active participants cover it, so no single participant can collapse the set to empty.

```diff
- let supported_chains = self.get_supported_foreign_chains();
- if !supported_chains.contains(&requested_chain) {
+ let available_chains = self.get_available_foreign_chains();
+ if !available_chains.contains(&requested_chain) {
``` [1](#0-0) [4](#0-3) 

### Proof of Concept

Assume a running contract with participants `[A, B, C]`, threshold 2, and Bitcoin registered by all three (so `get_supported_foreign_chains()` returns `{Bitcoin}`).

1. Participant `A` calls `register_foreign_chain_support` with `SupportedForeignChains(BTreeSet::new())`.
2. `get_supported_foreign_chains()` now computes the intersection of `{}` (A), `{Bitcoin}` (B), `{Bitcoin}` (C) → `{}` (empty, because A's set is not a superset of all active participants for Bitcoin).
3. Any user calling `verify_foreign_transaction` for Bitcoin now receives `ForeignChainNotSupported`.
4. Participant `A` can repeat this after any honest re-registration by B or C, maintaining the DoS indefinitely.

The attack requires only one participant-level NEAR account key and costs only gas.

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
