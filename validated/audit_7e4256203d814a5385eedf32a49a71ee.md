### Title
`RSETHPoolV3ExternalBridge` Missing `isEthDepositEnabled` Guard Present in Sibling Pool Variants - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol)

---

### Summary
`RSETHPoolV3ExternalBridge.sol` is missing the `isEthDepositEnabled` storage variable, the `setIsEthDepositEnabled()` setter, and the corresponding guard in its `deposit(string)` function — all of which exist in the sibling contracts `RSETHPoolV3.sol` and `RSETHPoolV3WithNativeChainBridge.sol`. This is the direct analog of M-12: a feature was added to the base/sibling pool variants but was not propagated to the external-bridge variant.

---

### Finding Description

`RSETHPoolV3.sol` declares `isEthDepositEnabled` and guards its ETH deposit path:

```solidity
// RSETHPoolV3.sol line 39
bool public isEthDepositEnabled;

// RSETHPoolV3.sol line 253
if (!isEthDepositEnabled) revert EthDepositDisabled();
```

`RSETHPoolV3WithNativeChainBridge.sol` does the same:

```solidity
// RSETHPoolV3WithNativeChainBridge.sol line 45
bool public isEthDepositEnabled;

// RSETHPoolV3WithNativeChainBridge.sol line 289
if (!isEthDepositEnabled) revert EthDepositDisabled();
```

`RSETHPoolV3ExternalBridge.sol` has **neither** the storage variable nor the guard. Its `deposit(string)` function proceeds directly to minting after only checking `whenNotPaused` and `limitDailyMint`:

```solidity
// RSETHPoolV3ExternalBridge.sol lines 366–384
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

There is no `isEthDepositEnabled` field anywhere in `RSETHPoolV3ExternalBridge`'s storage layout, and no `setIsEthDepositEnabled()` function exists in the contract.

---

### Impact Explanation

The protocol's intended design includes a granular ETH-deposit kill-switch (`isEthDepositEnabled`) that allows operators to halt ETH deposits independently of a full contract pause. On `RSETHPoolV3ExternalBridge` this control is entirely absent. If the protocol needs to stop ETH deposits on this pool variant (e.g., in response to an oracle anomaly, a bridge issue, or a rate manipulation attempt that only affects the ETH path), the only available lever is a full `pause()`, which simultaneously disables token deposits, fee withdrawals, and bridging operations. The contract therefore fails to deliver the selective-disable capability it promises across the pool family.

**Impact: Low — Contract fails to deliver promised returns / missing safety control.**

---

### Likelihood Explanation

The feature was clearly intended to be uniform across pool variants: both `RSETHPoolV3` and `RSETHPoolV3WithNativeChainBridge` implement it identically. The omission in `RSETHPoolV3ExternalBridge` is consistent with the M-12 pattern: the feature was added to the base/sibling contracts but the external-bridge variant was not updated. Any scenario where the protocol needs to selectively disable ETH deposits on the external-bridge pool (e.g., Arbitrum/Base/Optimism deployments using Stargate) will expose this gap.

---

### Recommendation

Add `isEthDepositEnabled` to `RSETHPoolV3ExternalBridge`'s storage, add the guard to `deposit(string)`, and expose a `setIsEthDepositEnabled(bool)` setter restricted to `TIMELOCK_ROLE`, matching the pattern in `RSETHPoolV3.sol` lines 39, 253, and 526–529 and `RSETHPoolV3WithNativeChainBridge.sol` lines 45, 289, and 660–663.
