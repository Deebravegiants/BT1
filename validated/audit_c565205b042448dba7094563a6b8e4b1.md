### Title
Pending Signature Requests Cannot Be Completed When `accept_requests` Is False — (`File: crates/contract/src/lib.rs`)

### Summary

When `verify_tee()` sets `accept_requests = false` due to a TEE validation failure, the `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` functions are all blocked by the same flag. This means MPC nodes with valid attestations cannot fulfill **already-pending** yield-resume requests that were submitted before the flag was cleared. Those in-flight requests will time out and fail, permanently breaking the request lifecycle for all users who submitted requests before the pause.

### Finding Description

The `MpcContract` uses an `accept_requests: bool` field as a circuit-breaker. It is set to `false` inside `verify_tee()` when TEE re-verification finds that kicking out invalid participants would violate the governance-vs-reconstruction threshold relation: [1](#0-0) 

When this happens, three node-facing response methods each contain an identical guard:

```rust
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
```

- `respond()` — lines 579–581 [2](#0-1) 
- `respond_ckd()` — lines 662–664 [3](#0-2) 
- `respond_verify_foreign_tx()` — lines 711–713 [4](#0-3) 

The user-facing `sign()`, `request_app_private_key()`, and `verify_foreign_transaction()` methods also check `accept_requests` via `check_request_preconditions()`: [5](#0-4) 

Blocking new requests when TEE validation fails is intentional and correct. However, blocking `respond()` is not: requests that were already enqueued via `promise_yield_create` before the flag was cleared are now orphaned. The MPC nodes that still hold valid attestations cannot call `respond()` to resolve those yields. NEAR's yield-resume mechanism has a finite timeout; once it expires the callback fires with a `PromiseError` and the request is permanently lost.

### Impact Explanation

Every in-flight `sign`, `request_app_private_key`, or `verify_foreign_transaction` request that was submitted before `accept_requests` was set to `false` will time out and fail. Users whose cross-chain transactions depended on those signatures receive no output and cannot retry until the contract is unpaused (i.e., until `verify_tee()` returns `true` again). This breaks the request-lifecycle accounting invariant: a request that was accepted and paid for by the user is silently dropped rather than completed or explicitly rejected. This matches the **Medium** allowed impact: *"request-lifecycle … manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

`verify_tee()` is callable by any current participant at any time. TEE attestations expire naturally (certificate lifetimes, image-hash rotation). If one or more participants' attestations expire and the remaining valid set cannot satisfy the governance-vs-reconstruction threshold relation, any participant can call `verify_tee()` to flip `accept_requests = false`. This is a realistic operational scenario, not a contrived attack. The window during which in-flight requests are orphaned lasts until the invalid participants refresh their attestations and `verify_tee()` is called again to restore the flag.

### Recommendation

Remove the `accept_requests` guard from `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()`. These functions already require the caller to be an attested participant (`assert_caller_is_attested_participant_and_protocol_active`) and the protocol to be in a running-or-resharing state. Blocking new requests via `check_request_preconditions` is sufficient to prevent new work from entering the queue; nodes must still be allowed to drain the existing queue regardless of the flag.

```rust
// respond(), respond_ckd(), respond_verify_foreign_tx():
- if !self.accept_requests {
-     return Err(TeeError::TeeValidationFailed.into());
- }
```

### Proof of Concept

1. Users call `sign()` while the contract is healthy → yield promises are created and stored in `pending_signature_requests`.
2. One or more participants' TEE attestations expire.
3. A participant calls `verify_tee()`. The surviving valid set is too small to satisfy the threshold relation → `accept_requests = false`.
4. MPC nodes (even those with valid attestations) attempt to call `respond()` with the computed signature.
5. `respond()` returns `Err(TeeError::TeeValidationFailed)` at line 579–581 — the yield is never resolved.
6. NEAR's yield-resume timeout fires; the callback receives a `PromiseError`; the pending request is cleaned up with no signature delivered.
7. Users' cross-chain transactions are permanently lost until the contract is unpaused. [6](#0-5) [7](#0-6) [8](#0-7)
