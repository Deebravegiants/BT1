### Title
Unauthorized `TransferToGasKey` Action Allows Any Account to Inflate Victim's Gas Key Balance, Permanently Blocking `DeleteKey` and `DeleteAccount` - (File: `runtime/runtime/src/actions.rs`)

---

### Summary

`Action::TransferToGasKey` is explicitly excluded from the `check_actor_permissions` authorization gate, allowing any unprivileged account to inflate the gas key balance of any other account's gas key. Because `DeleteKey` and `DeleteAccount` both hard-fail when a gas key's balance exceeds `GasKeyInfo::MAX_BALANCE_TO_BURN` (1 NEAR), an attacker can permanently block a victim from deleting their gas key or their account by sending a single transaction that overfunds the victim's gas key.

---

### Finding Description

`check_actor_permissions` in `runtime/runtime/src/actions.rs` enforces that sensitive actions (`AddKey`, `DeleteKey`, `WithdrawFromGasKey`, `DeployContract`, `Stake`, etc.) can only be executed when `actor_id == account_id`. However, `Action::TransferToGasKey` is explicitly placed in the unchecked arm alongside `Transfer` and `FunctionCall`:

```rust
Action::CreateAccount(_)
| Action::FunctionCall(_)
| Action::Transfer(_)
| Action::TransferToGasKey(_) => (),   // ← no actor check
``` [1](#0-0) 

This means any account can submit a transaction with `receiver_id = victim` and `Action::TransferToGasKey { public_key: victim_gas_key, deposit: X }`, and the runtime will execute it without verifying that the sender owns the target account.

`action_transfer_to_gas_key` then unconditionally increments the gas key balance on the receiver's account:

```rust
gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit)...;
set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);
``` [2](#0-1) 

`delete_gas_key` (called by `action_delete_key`) hard-fails if the gas key balance exceeds `MAX_BALANCE_TO_BURN`:

```rust
if gas_key_info.balance > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh { ... }.into());
    return Ok(());
}
``` [3](#0-2) 

`action_delete_account` applies the same check across all gas keys on the account:

```rust
let gas_key_balance_to_burn = compute_gas_key_balance_sum(state_update, account_id)?;
if gas_key_balance_to_burn > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh { ... }.into());
    return Ok(());
}
``` [4](#0-3) 

The `TransferToGasKey` action is also reachable via the `promise_batch_action_transfer_to_gas_key` host function, which any deployed contract can call targeting any account: [5](#0-4) 

---

### Impact Explanation

An attacker who knows a victim's gas key public key (public information, queryable via RPC) can:

1. Submit a transaction: `signer_id = attacker, receiver_id = victim, actions = [TransferToGasKey { public_key: victim_gas_key, deposit: 1 NEAR + 1 yoctoNEAR }]`
2. The runtime executes this without any authorization check, inflating `gas_key_info.balance` on the victim's gas key above `MAX_BALANCE_TO_BURN`
3. The victim's subsequent `DeleteKey` action fails with `GasKeyBalanceTooHigh`
4. The victim's subsequent `DeleteAccount` action also fails with `GasKeyBalanceTooHigh`

The corrupted trie state is the `gas_key_info.balance` field stored under `(victim_account_id, victim_gas_key_public_key)`. The broken invariant is that `DeleteKey` and `DeleteAccount` actions on the victim's account now unconditionally fail at the runtime level, not just at the RPC level.

The victim can recover only by first calling `WithdrawFromGasKey` to drain the excess balance below the threshold — an extra forced transaction the victim did not consent to and may not be able to afford if their account balance is near zero (gas key transactions pay gas from the gas key, but `WithdrawFromGasKey` requires a full-access key transaction).

---

### Likelihood Explanation

The attack requires only:
- Knowledge of the victim's gas key public key (publicly visible via `view_access_key` RPC)
- Sufficient NEAR balance to fund the attack (just over 1 NEAR)
- A single signed transaction

No special privileges, no contract deployment, no validator access. Any unprivileged user can execute this against any account that has a gas key.

---

### Recommendation

Add `Action::TransferToGasKey` to the `actor_id == account_id` check in `check_actor_permissions`, or introduce a separate check that restricts cross-account `TransferToGasKey` to amounts that cannot push the gas key balance above `MAX_BALANCE_TO_BURN`. Alternatively, enforce a per-key balance cap inside `action_transfer_to_gas_key` itself:

```rust
if gas_key_info.balance > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh { ... }.into());
    return Ok(());
}
```

If cross-account funding is intentional (e.g., for relayers), the cap enforcement inside `action_transfer_to_gas_key` is the minimal fix that preserves the intended use case while closing the griefing vector.

### Proof of Concept

**Attacker-controlled entry path (direct transaction):**

```
Transaction {
    signer_id:   "attacker.near",
    receiver_id: "victim.near",       // victim's account
    actions: [
        TransferToGasKey {
            public_key: <victim_gas_key_pubkey>,  // from RPC view_access_key
            deposit:    1_000_000_000_000_000_000_000_001,  // 1 NEAR + 1 yoctoNEAR
        }
    ]
}
```

**Runtime execution path:**

1. `Runtime::apply_action` is called with `account_id = "victim.near"`, `actor_id = "victim.near"` (set from `predecessor_id` of the receipt)
2. `check_actor_permissions` is called — `TransferToGasKey` matches the unchecked arm, returns `Ok(())` [1](#0-0) 
3. `action_transfer_to_gas_key` is called, inflating `gas_key_info.balance` above `MAX_BALANCE_TO_BURN` [6](#0-5) 
4. Victim subsequently submits `DeleteKey { public_key: victim_gas_key }` — runtime calls `delete_gas_key`, which returns `GasKeyBalanceTooHigh` error [3](#0-2) 
5. Victim subsequently submits `DeleteAccount { beneficiary_id: ... }` — runtime calls `action_delete_account`, which also returns `GasKeyBalanceTooHigh` error [4](#0-3)

pub(crate) fn action_transfer_to_gas_key(
    state_update: &mut TrieUpdate,
    result: &mut ActionResult,
    account_id: &AccountId,
    action: &TransferToGasKeyAction,
) -> Result<(), RuntimeError> {
    let Some(mut access_key) = get_access_key(state_update, account_id, &action.public_key)? else {
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };
    let Some(gas_key_info) = access_key.gas_key_info_mut() else {
        // Key exists but is not a gas key
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };

    gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit).ok_or_else(|| {
        RuntimeError::StorageError(StorageError::StorageInconsistentState(
            "gas key balance integer overflow".to_string(),
        ))
    })?;
    set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);
    Ok(())
```
