### Title
DID Recovery Attestation Uses `AGG_SIG_UNSAFE` Without Coin Binding, Enabling Cross-DID Signature Replay — (File: `chia/wallet/did_wallet/did_wallet_puzzles.py`)

### Summary

The DID recovery attestation puzzle in `create_recovery_message_puzzle` uses `AGG_SIG_UNSAFE` to sign only the new puzzle hash (`newpuz`), with no binding to the specific DID launcher ID, the recovering coin ID, or any nonce. A previously observed on-chain attestation signature can be replayed by an unprivileged attacker to forge valid attestations for a different DID's recovery, enabling unauthorized takeover of any DID that shares a backup holder with a previously recovered DID.

### Finding Description

`create_recovery_message_puzzle` in `chia/wallet/did_wallet/did_wallet_puzzles.py` constructs the attestation coin puzzle used in the DID social-recovery flow:

```python
puzzle = Program.to(
    (
        1,
        [
            [ConditionOpcode.CREATE_COIN_ANNOUNCEMENT, recovering_coin_id],
            [ConditionOpcode.AGG_SIG_UNSAFE, bytes(pubkey), newpuz],
        ],
    )
)
``` [1](#0-0) 

The `AGG_SIG_UNSAFE` condition requires a BLS signature over `newpuz` (the new inner puzzle hash) from `pubkey`. Critically, `AGG_SIG_UNSAFE` does **not** append any coin-specific domain separation data (no coin ID, no parent, no puzzle hash, no launcher ID). The signed message is exactly `newpuz` — a 32-byte value that is identical for any recovery targeting the same destination address.

The attestation coin's puzzle hash is deterministic from `(recovering_coin_id, newpuz, pubkey)`, and the coin ID is `sha256(parent_of_message, puzzle_hash, 0)`. The coin is created by `create_spend_for_message`: [2](#0-1) 

When a backup DID holder spends their attestation coin on-chain, their BLS signature over `newpuz` is permanently visible in the blockchain. Because `AGG_SIG_UNSAFE` carries no coin-specific binding, this signature is valid for **any** attestation coin that uses the same `(pubkey, newpuz)` pair, regardless of which DID is being recovered or which coin ID is embedded in the puzzle.

The `pkm_pairs_for_conditions_dict` validation only rejects messages that end with one of the AGG_SIG additional-data suffixes — it does not prevent reuse of the same signature across different coins: [3](#0-2) 

### Impact Explanation

An attacker can steal any DID whose recovery list includes a backup DID holder whose attestation signature for the attacker's target `newpuz` has previously appeared on-chain. The attacker:

1. Creates DID-A with victim backup holder Alice listed as a backup.
2. Initiates recovery of DID-A to `attacker_newpuz` (attacker-controlled address).
3. Alice creates and spends an attestation coin, publishing her BLS signature over `attacker_newpuz` on-chain.
4. Attacker extracts Alice's signature from the blockchain.
5. Attacker identifies DID-B (a different DID) that also lists Alice as a backup.
6. Attacker creates a new attestation coin: `Coin(any_parent, create_recovery_message_puzzle(DID-B-coin-id, attacker_newpuz, alice_pubkey).get_tree_hash(), 0)`.
7. Attacker spends this forged attestation coin using Alice's previously observed signature — the spend is valid because `AGG_SIG_UNSAFE` only checks the signature over `newpuz`.
8. Attacker submits the DID-B recovery spend asserting the announcement from the forged attestation coin.
9. If the threshold of backup attestations is met, DID-B is transferred to `attacker_newpuz`.

This constitutes unauthorized control of a DID singleton and all assets it governs (NFTs, VCs, etc.), matching the Critical/High impact scope.

### Likelihood Explanation

- Recovery lists and backup DID pubkeys are fully public (curried into the on-chain DID puzzle).
- All attestation signatures are permanently visible on-chain once spent.
- Creating coins with arbitrary puzzle hashes is permissionless in Chia.
- The attacker only needs to have previously triggered one legitimate recovery involving the target backup holder signing for the attacker's address.
- The DID recovery mechanism, while partially deprecated for new DIDs, remains live code and is reachable for any DID with a non-nil `recovery_list_hash`. [4](#0-3) 

### Recommendation

Replace `AGG_SIG_UNSAFE` in `create_recovery_message_puzzle` with a coin-binding signature condition. The signed message should commit to at minimum the launcher ID of the DID being recovered and the recovering coin ID, preventing cross-DID replay. For example, use `AGG_SIG_ME` or `AGG_SIG_PUZZLE` on the attestation coin itself, or construct the signed message as `sha256(launcher_id || recovering_coin_id || newpuz)` and verify it with `AGG_SIG_UNSAFE` only after confirming the message includes these non-reusable fields. Additionally, add a nonce or height-lock to attestation coins to bound their validity window.

def create_recovery_message_puzzle(recovering_coin_id: bytes32, newpuz: bytes32, pubkey: G1Element) -> Program:
    """
    Create attestment message puzzle
    :param recovering_coin_id: ID of the DID coin needs to recover
    :param newpuz: New wallet puzzle hash
    :param pubkey: New wallet pubkey
    :return: Message puzzle
    """
    puzzle = Program.to(
        (
            1,
            [
                [ConditionOpcode.CREATE_COIN_ANNOUNCEMENT, recovering_coin_id],
                [ConditionOpcode.AGG_SIG_UNSAFE, bytes(pubkey), newpuz],
            ],
        )
    )
    return puzzle

def create_spend_for_message(
    parent_of_message: bytes32, recovering_coin: bytes32, newpuz: bytes32, pubkey: G1Element
) -> CoinSpend:
    """
    Create a CoinSpend for a atestment
    :param parent_of_message: Parent coin ID
    :param recovering_coin: ID of the DID coin needs to recover
    :param newpuz: New wallet puzzle hash
    :param pubkey: New wallet pubkey
    :return: CoinSpend
    """
    puzzle = create_recovery_message_puzzle(recovering_coin, newpuz, pubkey)
