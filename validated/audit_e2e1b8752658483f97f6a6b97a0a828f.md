Audit Report

## Title
rsETH Inflation Attack via ETH Donation to `LRTDepositPool` Enables Zero-Share Minting - (File: `contracts/LRTDepositPool.sol`)

## Summary
An attacker who is the first depositor can donate ETH directly to `LRTDepositPool` and call the permissionless `updateRSETHPrice()` to inflate `rsETHPrice` by an unbounded factor, because `pricePercentageLimit` defaults to `0`. Any subsequent depositor who passes `minRSETHAmountExpected = 0` will have their ETH accepted while receiving 0 rsETH in return, permanently freezing their funds with no withdrawal path.

## Finding Description
Three confirmed weaknesses cooperate:

**1. Direct ETH donation is counted in TVL.**
`LRTDepositPool` exposes a bare `receive()` at line 58, and `getETHDistributionData()` reads `ethLyingInDepositPool = address(this).balance` at line 480. Any ETH sent directly to the contract is immediately included in `_getTotalEthInProtocol()`.

**2. `updateRSETHPrice()` is public and uncapped at default state.**
`updateRSETHPrice()` at lines 87–89 of `LRTOracle.sol` is callable by anyone when not paused. The only guard against a large price jump is:
```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```
`pricePercentageLimit` is a storage variable that defaults to `0`. When `0`, the condition short-circuits to `false` and the price can be inflated by any arbitrary multiple in a single call.

**3. `_beforeDeposit` does not reject a zero-rsETH mint.**
Lines 665–669 of `LRTDepositPool.sol` only check `rsethAmountToMint < minRSETHAmountExpected`. If the caller passes `minRSETHAmountExpected = 0` and `rsethAmountToMint` rounds to `0` due to integer division, the check `0 < 0` is `false` and the deposit proceeds. `RSETH.mint(victim, 0)` does not revert: the `checkDailyMintLimit(0)` modifier evaluates `0 > maxMintAmountPerDay` as `false`, and OpenZeppelin `_mint` with amount `0` is a no-op.

**Mint formula:**
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
After inflation with donation `X` wei, `rsETHPrice ≈ X * 1e18`. For any victim deposit `Y < X`, integer division yields `rsethAmountToMint = 0`.

## Impact Explanation
**Critical — Permanent freezing of funds.** The victim's ETH is transferred into `LRTDepositPool` as `msg.value` before any check can revert it, but 0 rsETH is minted. With no rsETH balance, the victim has no mechanism to initiate a withdrawal. Their ETH is permanently locked in the protocol. This exactly matches the allowed impact: *"Critical. Permanent freezing of funds."*

## Likelihood Explanation
**Medium.** The attacker must be the first depositor (or act when rsETH supply is negligibly small). On Ethereum mainnet, front-running the first legitimate deposit is straightforward. The victim must pass `minRSETHAmountExpected = 0`, which is the default for many integrations and scripts that omit slippage protection. The critical enabler — `pricePercentageLimit == 0` — is the contract's default state at deployment and requires no special conditions.

## Recommendation
1. **Enforce a non-zero mint amount.** Add `require(rsethAmountToMint > 0, "ZeroMint")` inside `_beforeDeposit` before the slippage check (after line 665 of `LRTDepositPool.sol`).
2. **Initialize `pricePercentageLimit` to a safe non-zero value** (e.g., 1% = `1e16`) during deployment so that a single `updateRSETHPrice()` call cannot inflate the price by an unbounded factor.
3. **Seed the pool at deployment.** Mint a small amount of rsETH to a dead address (e.g., `address(0xdead)`) during initialization so that `rsethSupply` is never `0` for an attacker-controlled first deposit.

## Proof of Concept
```
State: rsETHPrice = 1e18, pricePercentageLimit = 0, rsETH totalSupply = 0

Step 1 — Attacker deposits 1 wei ETH:
  depositETH{value: 1}(minRSETHAmountExpected=0, "")
  rsethAmountToMint = (1 * 1e18) / 1e18 = 1
  → Attacker receives 1 wei rsETH. rsETH totalSupply = 1.

Step 2 — Attacker donates X wei ETH directly to LRTDepositPool:
  (bool ok,) = address(lrtDepositPool).call{value: X}("");
  → LRTDepositPool.balance = 1 + X

Step 3 — Attacker calls LRTOracle.updateRSETHPrice():
  totalETHInProtocol = (1 + X) * 1e18 / 1e18 = 1 + X  [via mulWad with assetER=1e18]
  previousTVL = 1 * 1e18 / 1e18 = 1
  newRsETHPrice = (1 + X) * 1e18 / 1 = (1 + X) * 1e18  [divWad]
  pricePercentageLimit == 0 → isPriceIncreaseOffLimit = false → no revert
  → rsETHPrice = (1 + X) * 1e18

Step 4 — Victim deposits Y wei ETH (Y < X) with minRSETHAmountExpected = 0:
  rsethAmountToMint = (Y * 1e18) / ((1 + X) * 1e18) = Y / (1 + X) = 0  [integer division]
  0 < 0 → false → no revert
  RSETH.mint(victim, 0) → checkDailyMintLimit(0): 0 > maxMintAmountPerDay = false → no revert
  _mint(victim, 0) → no-op
  → Victim's Y wei ETH is in the pool; victim has 0 rsETH and no withdrawal path.

Step 5 — Attacker redeems 1 wei rsETH:
  Attacker holds 100% of rsETH supply.
  Attacker recovers 1 + X + Y wei ETH.
  Net profit: Y wei (victim's entire deposit).
```
