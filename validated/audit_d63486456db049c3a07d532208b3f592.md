### Title
Meta-Transaction Relayer Loses Attached Deposit to Sender When Inner Action Fails - (`runtime/runtime/src/actions.rs`)

### Summary

In NEAR's meta-transaction (NEP-366 `DelegateAction`) flow, the relayer pays all attached deposits for inner actions. However, when the inner action receipt fails on the receiver's shard, the deposit refund is sent to `Alice` (the `predecessor_id` of the inner receipt, i.e., the `DelegateAction` sender), not to the relayer who actually funded it. An unprivileged user (Alice) can deliberately craft a `DelegateAction` with a large attached deposit targeting an action that will fail, causing the relayer to lose the deposit while Alice receives it as a refund.

### Finding Description

In `apply_delegate_action` (`runtime/runtime/src/actions.rs`), when the inner receipt is constructed from a `DelegateAction`, the `predecessor_id` of the new receipt is set to `sender_id` (Alice's account), not the relayer: [1](#0-0) 

The code comment at lines 471–475 explicitly acknowledges this behavior: [2](#0-1) 

When the inner action receipt fails, `refund_unspent_gas_and_deposits` in `runtime/runtime/src/lib.rs` generates a deposit refund receipt directed to `receipt.balance_refund_receiver()`: [3](#0-2) 

`balance_refund_receiver()` returns `predecessor_id()` for `ReceiptV0` (the default receipt type used for inner delegate receipts): [4](#0-3) 

Since the inner receipt's `predecessor_id` is Alice (not the relayer), the deposit refund goes to Alice. The relayer's `signer_id` is preserved only for **gas** refunds, not deposit refunds: [5](#0-4) 

The architecture documentation explicitly describes this as a known financial risk to relayers: [6](#0-5) 

### Impact Explanation

The relayer loses the full attached deposit (NEAR tokens) when an inner action fails. Alice receives the deposit refund instead. The corrupted protocol value is the **NEAR token balance**: the relayer's balance decreases by the deposit amount, and Alice's balance increases by the same amount, with no legitimate basis for this transfer. This is a direct, concrete financial loss for the relayer.

### Likelihood Explanation

Alice can deliberately trigger this by crafting a `DelegateAction` with a large `deposit` attached to a function call that is guaranteed to fail (e.g., calling a non-existent method, a method that panics, or a contract that rejects the call). The relayer cannot fully prevent this: even if the relayer checks Alice's `DelegateAction` before submission, Alice can manipulate contract state between the check and execution to cause the failure. The attack is cheap for Alice (she only needs to sign a `DelegateAction`) and profitable (she receives the deposit refund). Any general-purpose relayer accepting arbitrary `DelegateAction`s is vulnerable.

### Recommendation

When constructing the inner receipt from a `DelegateAction`, use `ActionReceiptV2` (which has a `refund_to` field) and set `refund_to` to the relayer's account ID (the `signer_id` of the outer receipt). This ensures deposit refunds on failure are returned to the relayer who funded them, not to Alice. The `balance_refund_receiver()` method already supports this via the `refund_to` field: [7](#0-6) [8](#0-7) 

### Proof of Concept

1. Alice creates a `DelegateAction` with `receiver_id = some_contract`, actions = `[FunctionCall("method_that_panics", deposit=1000 NEAR)]`, and signs it.
2. Alice sends the signed `DelegateAction` to a relayer.
3. The relayer wraps it in a transaction (relayer is `signer_id`), paying 1000 NEAR deposit.
4. On Alice's shard, `apply_delegate_action` creates an inner receipt with `predecessor_id = Alice`, `signer_id = relayer`.
5. On `some_contract`'s shard, the function call panics. `refund_unspent_gas_and_deposits` fires:
   - Deposit refund → `balance_refund_receiver()` = `predecessor_id` = **Alice** ✓ (Alice gains 1000 NEAR)
   - Gas refund → `signer_id` = **relayer** (relayer gets unspent gas back, but not the deposit)
6. Net result: relayer loses 1000 NEAR deposit; Alice gains 1000 NEAR.

    pub fn refund_to(&self) -> &Option<AccountId> {
        match self.receipt() {
            ReceiptEnum::Action(_)
            | ReceiptEnum::Data(_)
            | ReceiptEnum::PromiseYield(_)
            | ReceiptEnum::PromiseResume(_)
            | ReceiptEnum::GlobalContractDistribution(_) => &None,
            ReceiptEnum::ActionV2(action_receipt_v2)
            | ReceiptEnum::PromiseYieldV2(action_receipt_v2) => &action_receipt_v2.refund_to,
        }
    }

    pub fn balance_refund_receiver(&self) -> &AccountId {
        self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
    }
