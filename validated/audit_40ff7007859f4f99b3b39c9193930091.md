### Title
Stale `rsETHPrice` After `updateRSETHPrice()` Revert Due to Daily Fee Mint Limit Allows New Depositors to Mint rsETH at Below-Market Price, Diluting Existing Holders — (`contracts/LRTOracle.sol`)

---

### Summary

`rsETHPrice` is written **after** the protocol-fee mint block in `_updateRsETHPrice()`. If either the LRTOracle-level daily fee limit (`_checkAndUpdateDailyFeeMintLimit`) or the RSETH-level daily mint limit (`RSETH.checkDailyMintLimit`) causes a revert, the price update is permanently skipped for that call. Because `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` are both permissionless, an attacker can trigger this revert at will, leaving `rsETHPrice` stale while the ETH that caused the TVL increase is already settled in the deposit pool. Subsequent depositors then mint rsETH at the artificially low stale price, extracting value from existing holders.

---

### Finding Description

**Step 1 — `sendFunds()` is permissionless and immediately settles ETH.** [1](#0-0) 

`sendFunds()` has no access control. It calls `receiveFromRewardReceiver()` on the deposit pool, which is a plain `payable` receiver: [2](#0-1) 

The ETH is now permanently in the deposit pool and is immediately counted by `_getTotalEthInProtocol()` via `address(this).balance`.

**Step 2 — `updateRSETHPrice()` is permissionless.** [3](#0-2) 

**Step 3 — `rsETHPrice` is written only after the fee mint block.**

Inside `_updateRsETHPrice()`, the sequence is:

1. Compute `newRsETHPrice` (line 250).
2. Call `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` — **reverts** with `DailyFeeMintLimitExceeded` if the LRTOracle fee cap is breached.
3. Call `IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee)` — **reverts** with `DailyMintLimitExceeded` (RSETH's own `checkDailyMintLimit` modifier) if the RSETH daily cap is breached.
4. `rsETHPrice = newRsETHPrice` — **only reached if neither revert fires**. [4](#0-3)

**Step 4 — Deposit pricing uses the stale `rsETHPrice`.** [7](#0-6) 

`rsethAmountToMint = (amount × assetPrice) / rsETHPrice`. A stale (lower) `rsETHPrice` yields a higher rsETH amount per ETH deposited. There is no oracle-freshness check in `_beforeDeposit`.

### Impact Explanation

After the revert, `rsETHPrice` is frozen at the pre-reward value `P`, while the true price is `P' > P` (TVL increased but supply did not). Any depositor who calls `depositETH` during the window receives `Y / P` rsETH instead of the correct `Y / P'`. The excess `Y × (P' − P) / (P × P')` rsETH per deposited ETH is backed by no additional ETH, diluting every existing holder proportionally. The attack window lasts up to 24 hours (until the daily limit resets and `updateRSETHPrice()` can succeed again). This constitutes direct, quantifiable theft of value from existing rsETH holders.

---

### Likelihood Explanation

Both trigger conditions are realistic:

- **LRTOracle fee cap** (`maxFeeMintAmountPerDay`): A conservatively set cap combined with a large MEV reward block (e.g., a high-value validator slot) can produce a fee rsETH amount that exceeds the cap in a single call.
- **RSETH daily mint cap** (`maxMintAmountPerDay`): Heavy user deposit activity during the day can exhaust most of the RSETH daily mint budget, leaving insufficient headroom for the fee mint.

Neither `sendFunds()` nor `updateRSETHPrice()` requires any role. An attacker needs only gas to trigger the revert and then deposit at the stale price.

---

### Recommendation

Move `rsETHPrice = newRsETHPrice` **before** the fee mint block, or use a try/catch around the fee mint so that a fee-mint failure skips the mint but still commits the price update. Alternatively, add a last-updated timestamp to `rsETHPrice` and reject deposits (or apply a penalty rate) when the price is stale beyond a configurable threshold.
