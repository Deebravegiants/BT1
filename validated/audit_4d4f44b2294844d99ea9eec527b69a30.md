### Title
Missing Observation Timestamp in `ForeignTxSignPayloadV1` Enables Stale-Attestation Replay After Foreign-Chain Reorganization - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary

The `ForeignTxSignPayloadV1` struct — the data structure whose hash the MPC network signs for every `verify_foreign_transaction` request — does not include any timestamp or block-height field recording *when* the observation was made. The design document explicitly states the payload should commit to `(request, observed_values, observed_at)`, but the implementation omits `observed_at`. As a result, a signature produced for a non-finalized foreign-chain transaction (e.g., `EvmFinality::Latest`, `SolanaFinality::Processed`, or a Bitcoin request with low `confirmations`) remains cryptographically valid forever, even after the underlying transaction has been reorganized out of the foreign chain. A bridge contract consuming this signature has no way to distinguish a fresh attestation from a stale one, enabling invalid bridge execution.

### Finding Description

The design document at `docs/foreign-chain-transactions.md` line 24 states:

> "If extraction succeeds, MPC signs a canonical encoding of `(request, observed_values, observed_at)` and returns the signature on-chain."

    pub request: ForeignChainRpcRequest,

    // observed_at is absent

### Impact Explanation

A bridge contract (e.g., Omnibridge) that calls `verify_foreign_transaction` with `EvmFinality::Latest` or `SolanaFinality::Processed` receives a threshold signature over `SHA-256(borsh(request, values))`. Because no `observed_at` field is committed to, this signature:

1. Is valid forever — no bridge contract can enforce a freshness window.
2. Cannot be distinguished from a signature produced at a different point in time.
3. Remains valid even if the foreign-chain transaction is reorganized out of the canonical chain after signing.

An attacker who obtains a valid MPC attestation for a non-finalized transaction, then causes or waits for a reorganization that removes or alters that transaction, can present the stale attestation to a NEAR bridge contract to claim funds for a transaction that no longer exists on the foreign chain. This constitutes **forged foreign-chain verification causing invalid bridge execution**.

### Likelihood Explanation

- Any unprivileged caller can submit `verify_foreign_transaction` with `EvmFinality::Latest` or `SolanaFinality::Processed` — no special access is required.
- Natural short-range reorganizations occur on Ethereum (1-2 blocks) and Solana regularly.
- Bitcoin requests with `confirmations: 1` or `confirmations: 2` are similarly vulnerable.
- The attacker does not need to collude with any MPC participant; the vulnerability is in the payload structure, not the signing protocol.
- Bridge contracts built on the `near-mpc-sdk` `ForeignChainSignatureVerifier` have no API surface to enforce freshness, so the vulnerability propagates to all downstream consumers.

### Recommendation

1. Add an `observed_at` field (NEAR block height or timestamp at which the MPC nodes performed the RPC query) to `ForeignTxSignPayloadV1`, matching the design document's stated intent.
2. Expose `observed_at` in `VerifyForeignTransactionResponse` so bridge contracts can enforce a maximum attestation age.
3. Consider rejecting `EvmFinality::Latest` and `SolanaFinality::Processed` at the contract level for bridge-critical domains, or document clearly that callers using non-finalized finality levels accept reorganization risk.
4. Update `ForeignChainSignatureVerifier::verify_signature` in the SDK to accept and validate an `observed_at` bound.

### Proof of Concept

1. Attacker submits an Ethereum transaction `T` that deposits funds into a bridge.
2. Attacker calls `verify_foreign_transaction` with `EvmFinality::Latest` before `T` is finalized.
3. MPC nodes observe `T` in the latest block, extract `BlockHash: 0xabc`, and sign `SHA-256(borsh(V1 { request: {tx_id: T, finality: Latest, extractors: [BlockHash]}, values: [BlockHash(0xabc)] }))`.
4. Attacker receives `VerifyForeignTransactionResponse { payload_hash, signature }`.
5. A short-range reorganization (or attacker-induced reorg) removes `T` from the canonical chain.
6. Attacker submits the stale `(payload_hash, signature)` to the NEAR bridge contract.
7. The bridge contract calls `ForeignChainSignatureVerifier::verify_signature`, which checks only cryptographic validity — it passes.
8. The bridge contract releases funds on NEAR for a deposit that no longer exists on Ethereum.

The root cause is the absence of `observed_at` in `ForeignTxSignPayloadV1` at `crates/near-mpc-contract-interface/src/types/foreign_chain.rs` lines 1499–1502, contradicting the design specification at `docs/foreign-chain-transactions.md` line 24. [6](#0-5) [7](#0-6)

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
