### Title
`ProposalInit.builder` (sequencer address) is not validated in `is_proposal_init_valid`, allowing a malicious proposer to redirect transaction fees and corrupt `get_sequencer_address` syscall results - (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`ProposalInit` carries a `builder: ContractAddress` field that is mapped directly to `sequencer_address` in `BlockInfo` and passed to the blockifier for every transaction in the block. The validator's `is_proposal_init_valid` function checks `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices, `starknet_version`, `version_constant_commitment`, `timestamp`, and `fee_proposal_fri` — but never checks `builder`. The local node has the correct expected value in `ContextStaticConfig.builder_address`, yet that value is never placed in `ProposalInitValidation` and never compared against the wire value. A malicious proposer can set `builder` to any address; every validator will accept the proposal, execute the block with the spoofed sequencer address, credit all transaction fees to the attacker-controlled address, and return the wrong value from the `get_sequencer_address` syscall.

### Finding Description

`ProposalInit.builder` is defined as the address of the entity that builds/sequences the block: [1](#0-0) 

In `convert_to_sn_api_block_info`, `init.builder` is assigned verbatim to `sequencer_address` in the `starknet_api::block::BlockInfo` that is handed to the batcher for both proposal building and validation: [2](#0-1) 

This `sequencer_address` is the address to which transaction fees are credited (`get_sequencer_balance_keys`) and the value returned by the `get_sequencer_address` / `GetExecutionInfo` syscalls during contract execution.

The proposer sets `builder` from its own static config: [3](#0-2) 

The validator's `ProposalInitValidation` struct, however, has no `builder` field: [4](#0-3) 

And `is_proposal_init_valid` never reads or checks `init_proposed.builder` at any point in its body (lines 259–418). The combined check at line 312 only covers `height`, `l1_da_mode`, and `l2_gas_price_fri`: [5](#0-4) 

The expected value is available in `ContextStaticConfig.builder_address`: [6](#0-5) 

but it is never forwarded into `ProposalInitValidation` when the validator constructs it: [7](#0-6) 

Because `initiate_validation` calls `convert_to_sn_api_block_info(init)` with the unvalidated `init`, the batcher executes every transaction in the block using the attacker-supplied `sequencer_address`: [8](#0-7) 

The proposer's batcher and the validator's batcher both execute with the same spoofed address, so their partial block hashes agree and the `ProposalFinMismatch` check at line 244 passes silently: [9](#0-8) 

### Impact Explanation

**Fee theft (Critical — Incorrect fee/balance with economic impact):** `get_sequencer_balance_keys` derives the storage keys for the fee token balance from `block_context.block_info.sequencer_address`: [10](#0-9) 

With a spoofed `builder`, every transaction fee in the block is credited to the attacker-controlled address instead of the legitimate sequencer. The committed state diff reflects this wrong balance update.

**Wrong syscall result (Critical — Wrong state/execution result):** The `get_sequencer_address` syscall and `GetExecutionInfo` return `block_info.sequencer_address` to executing contracts: [11](#0-10) 

Contracts that gate logic on the sequencer address (e.g., access-control checks, fee-routing contracts) will observe the wrong value, potentially allowing or denying operations incorrectly. The wrong value is also committed to storage and propagated to L1.

### Likelihood Explanation

The trigger requires a validator whose turn it is to propose a block (a legitimate committee member acting maliciously). No external unprivileged user can trigger this directly. However, in a multi-validator Tendermint network, any validator in the committee can exploit this on their proposal round. The attack is silent: no error is logged, no metric fires, and the `ProposalFinMismatch` guard does not catch it because both sides execute with the same spoofed address.

### Recommendation

Add `builder_address: ContractAddress` to `ProposalInitValidation` and check it in `is_proposal_init_valid`:

```rust
// In ProposalInitValidation:
pub builder_address: ContractAddress,

// In is_proposal_init_valid, alongside the existing combined check:
if init_proposed.builder != proposal_init_validation.builder_address {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder_address, init_proposed.builder
        ),
    ));
}
```

Populate it when constructing `ProposalInitValidation` in `validate_proposal` (in `sequencer_consensus_context.rs`):

```rust
builder_address: self.config.static_config.builder_address,
```

### Proof of Concept

1. Attacker controls validator `V` whose turn it is to propose at height `H`.
2. `V` constructs `ProposalInit` with `builder = attacker_wallet_address` (any address).
3. `V` streams the proposal to all other validators.
4. Each validator calls `is_proposal_init_valid` — the function checks timestamp, gas prices, starknet version, etc., but never reads `init_proposed.builder`. Validation passes.
5. `initiate_validation` calls `convert_to_sn_api_block_info(init)`, producing a `BlockInfo` with `sequencer_address = attacker_wallet_address`.
6. The batcher executes all transactions with this `BlockInfo`. Every fee payment credits `attacker_wallet_address`.
7. The batcher returns a `partial_block_hash` computed over the execution with the spoofed address. The proposer's batcher produced the same hash. `ProposalFinMismatch` check passes.
8. Consensus reaches decision; the block is committed with `sequencer_address = attacker_wallet_address` in the block header and state diff.
9. All fees from block `H` are in `attacker_wallet_address`; the legitimate sequencer receives nothing. Any contract that called `get_sequencer_address` during block `H` received `attacker_wallet_address`.
