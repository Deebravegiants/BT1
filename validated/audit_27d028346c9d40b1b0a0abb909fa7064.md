### Title
Unprotected `init` Function Allows Any Caller to Seize the MPC Participant Set Before Legitimate Deployment - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `MpcContract::init` function lacks the `#[private]` access-control modifier present on every other privileged initializer in the same file. Because NEAR's `#[init]` macro only prevents *re-initialization* (panics when state already exists) but does **not** restrict *who* may call the function, any NEAR account can race the legitimate deployer and install an attacker-controlled participant set. The first caller wins permanently; all subsequent `init` calls revert.

---

### Finding Description

`MpcContract::init` is decorated with `#[handle_result]` and `#[init]` only: [1](#0-0) 

Its sibling `init_running` — which performs the same bootstrapping role for contract migrations — correctly carries `#[private]` in addition to `#[init]`: [2](#0-1) 

The ABI snapshot makes the asymmetry explicit: `init` exposes modifiers `["init"]` while `init_running` exposes `["init", "private"]`: [3](#0-2) 

In NEAR, `#[private]` enforces `predecessor_account_id == current_account_id`, i.e. only a self-call from the contract account is accepted. Without it, **any external account** may call `init` as the direct predecessor.

The README explicitly documents that deployment and initialization are separate steps:

> "After deploying the contract, it will first be in an uninitialized state. The owner will need to initialize it via `init`."

The deployment scripts confirm the two-transaction pattern — `init` is submitted as a standalone call after the `DeployContract` action: [4](#0-3) 

This creates a window — at minimum one NEAR block — during which an attacker can submit their own `init` call.

Inside `init`, the initial participants receive **mocked TEE attestations** via `TeeState::with_mocked_participant_attestations`, removing the TEE barrier that would otherwise gate participation: [5](#0-4) 

An attacker-supplied participant set therefore passes all validation and receives valid-enough attestation state to proceed through key generation.

---

### Impact Explanation

An attacker who wins the initialization race:

1. Installs themselves as the sole participant(s) with a threshold they choose.
2. Calls `vote_add_domains` to trigger distributed key generation across attacker-controlled nodes.
3. Once key generation completes, the attacker's nodes hold all key shares for every domain.
4. Every subsequent `sign()`, `request_app_private_key()`, or `verify_foreign_transaction()` call submitted by legitimate users is processed by the attacker's nodes, which can produce valid threshold signatures for arbitrary payloads.
5. The legitimate deployer's `init` call fails permanently ("state already exists"), locking them out with no recovery path short of redeploying the contract account.

This is **unauthorized threshold signature issuance** — a Critical impact matching the allowed scope: *"Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization."*

---

### Likelihood Explanation

- Requires no special privilege — any NEAR account suffices.
- The deployment-then-init two-transaction pattern is the documented and scripted workflow; the window is structural, not accidental.
- An attacker passively monitoring the NEAR chain for `DeployContract` actions targeting the known contract account ID can react within the next block.
- No TEE hardware, key material, or threshold collusion is needed; the mocked-attestation path in `init` removes that barrier.

---

### Recommendation

Add `#[private]` to `init`, consistent with `init_running`:

```rust
#[handle_result]
#[private]   // ← add this
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

Alternatively, perform deployment and initialization atomically as a single NEAR batch transaction (a `DeployContract` action followed immediately by a `FunctionCall` action in the same transaction), so no inter-block window exists.

---

### Proof of Concept

1. The legitimate deployer submits transaction **T1**: `DeployContract` on `mpc-contract.near`. T1 is confirmed in block N.
2. The attacker observes block N and submits transaction **T2**: `mpc-contract.near::init({ parameters: { participants: [attacker_node], threshold: 1 }, init_config: null })` signed by `attacker.near`.
3. The legitimate deployer submits transaction **T3**: `mpc-contract.near::init({ parameters: { participants: [legitimate_nodes], threshold: k }, ... })`.
4. T2 is included in block N+1 before T3. `init` succeeds; contract state is written with `attacker_node` as the sole participant.
5. T3 is included in block N+1 or N+2. NEAR SDK's `#[init]` guard detects existing state and panics: *"The contract has already been initialized"*. T3 fails.
6. Attacker calls `vote_add_domains` from `attacker_node` to add a Secp256k1 signing domain.
7. Attacker's MPC node completes key generation and calls `vote_pk` to finalize the domain key.
8. Any user calling `sign()` receives a signature produced entirely by the attacker's key share — constituting unauthorized threshold signature issuance over the entire MPC network.
