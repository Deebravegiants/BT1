### Title
Missing Predicate Failure Allows Multiple Sub-Transactions to Double-Claim the Same Deposit Refund - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs)

### Summary

In the Dijkstra era, the `SUBLEDGERS` rule processes sub-transactions sequentially via `foldM`. There is no implemented predicate failure preventing multiple sub-transactions within the same top-level transaction from each submitting an `UnRegDepositTxCert` for the same staking credential. An unprivileged transaction author can craft a top-level transaction whose sub-transactions each claim the same deposit refund, creating ADA from nothing and breaking the deposit-pot invariant.

### Finding Description

The `SUBLEDGERS` rule in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs` folds over sub-transactions, threading `LedgerState` through each step:

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
``` [1](#0-0) 

Each sub-transaction's refund amount is taken directly from the certificate field, not from the current protocol parameter:

```haskell
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert _ deposit -> deposit
  _ -> zero
``` [2](#0-1) 

The `updateUTxOStateNoFees` function uses this refund to decrement `utxosDeposited`:

```haskell
totalRefunds = certsTotalRefundsTxBody pp certState txBody
...
utxosDeposited = utxosDeposited <> depositChange  -- depositChange = deposits - refunds
``` [3](#0-2) 

The developers themselves acknowledge the missing protection in a disabled test:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
    stakingCred <- KeyHashObj <$> freshKeyHash
    _ <- registerStakeCredential stakingCred
    keyDeposit <- getsPParams ppKeyDepositL
    ...
    let
      subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
      subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
      tx = mkBasicTx mkBasicTxBody
             & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
    submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [4](#0-3) 

The `xit` marker disables the test, and the `error "TODO: predicate failure not yet implemented"` confirms that no predicate failure exists in the SUBDELEG/SUBCERT rules to reject a second `UnRegDepositTxCert` for a credential already unregistered by a prior sub-transaction in the same fold. Without this guard, both sub-transactions succeed, each decrementing `utxosDeposited` by `keyDeposit`, while only one deposit was ever paid.

The `Obligations` type tracks what the system owes:

```haskell
data Obligations = Obligations
  { oblStake :: !Coin
  , oblPool  :: !Coin
  , oblDRep  :: !Coin
  , oblProposal :: !Coin
  }
``` [5](#0-4) 

The invariant `potEqualsObligation` requires `utxosDeposited == totalObligation certState govState`. A double-refund breaks this invariant within the epoch, and the epoch-boundary recalculation in the Conway/Dijkstra `EPOCH` rule (`utxosDepositedL .~ totalObligation certState2 govState1`) would then set `utxosDeposited` to a value lower than the actual ADA that left the deposit pot, permanently destroying ADA from the deposit pot. [6](#0-5) 

### Impact Explanation

**Critical. Direct loss/creation of ADA through an invalid ledger state transition.**

An attacker registers one staking credential (paying deposit `D`), then submits a top-level transaction containing `N` sub-transactions each with `UnRegDepositTxCert stakingCred D`. If the missing predicate failure is absent, all `N` sub-transactions succeed, returning `N × D` lovelace while only `D` was deposited. The surplus `(N-1) × D` lovelace is created from nothing, violating preservation of value. The deposit pot (`utxosDeposited`) is driven below its true obligations, permanently corrupting the ledger's accounting.

### Likelihood Explanation

Any unprivileged user who can submit a Dijkstra-era transaction can trigger this. No special role, key, or governance majority is required. The attacker only needs to register a staking credential and construct a top-level transaction with multiple sub-transactions. The attack is deterministic and repeatable.

### Recommendation

Implement a predicate failure in the `SUBDELEG` (or `SUBCERT`) rule that rejects `UnRegDepositTxCert` when the credential is not present in the current `CertState`. Because `SUBLEDGERS` threads state sequentially, the first sub-transaction's unregistration removes the credential; the second sub-transaction must then fail with a `StakeKeyNotRegisteredDELEG`-equivalent failure. Enable and complete the disabled test `"Multiple subtransactions cannot get the same refund"` in `CertSpec.hs` once the predicate failure is in place.

### Proof of Concept

1. Register staking credential `C`, paying deposit `D`.
2. Construct top-level transaction `T` with two sub-transactions:
   - `subTx1`: `UnRegDepositTxCert C D`
   - `subTx2`: `UnRegDepositTxCert C D`
3. Submit `T`.
4. Without the missing predicate failure, `SUBLEDGERS` folds over `[subTx1, subTx2]`. `subTx1` unregisters `C` and decrements `utxosDeposited` by `D`. `subTx2` also decrements `utxosDeposited` by `D` (total `2D` removed), while only `D` was ever deposited.
5. The deposit pot is now `D` short of its true obligations, and the attacker has received `2D` in refunds for a single `D` deposit.
