Audit Report

## Title
Stale Cached `rsETHPrice` Used in Deposit Minting Without Prior Update - (`contracts/LRTDepositPool.sol` / `contracts/LRTOracle.sol`)

## Summary

`LRTOracle` stores the rsETH/ETH exchange rate in a persistent state variable `rsETHPrice` that is only updated when `updateRSETHPrice()` is called explicitly. Neither `depositETH()` nor `depositAsset()` triggers a price refresh before computing the rsETH amount to mint. An unprivileged depositor can exploit the window between price updates — when staking rewards or LST price appreciation have grown TVL — to receive excess rsETH minted at a stale, below-market rate, extracting value from existing rsETH holders' accrued yield.

## Finding Description

`LRTOracle` declares `rsETHPrice` as a mutable state variable:

```solidity
// contracts/LRTOracle.sol:28
uint256 public override rsETHPrice;
```

It is only written inside `_updateRsETHPrice()`, which is invoked by the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`. Neither is called anywhere in the deposit path.

`depositETH()` and `depositAsset()` both delegate to `_beforeDeposit()`, which calls `getRsETHAmountToMint()`:

```solidity
// contracts/LRTDepositPool.sol:520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The numerator uses a **live** oracle call (`getAssetPrice`), while the denominator reads the **cached** `rsETHPrice` state variable. When TVL has grown since the last explicit price update (e.g., staking rewards accrued, underlying LST prices rose), the cached `rsETHPrice` is lower than the real current price, causing the division to yield a larger rsETH amount than the deposit warrants.

`_beforeDeposit()` is declared `private view` — it cannot mutate state and therefore cannot call `updateRSETHPrice()` even if desired. The only guard it applies is a `minRSETHAmountExpected` slippage check, which protects the depositor from receiving *too little*, not existing holders from the depositor receiving *too much*.

## Impact Explanation

**High — Theft of unclaimed yield.**

When `rsETHPrice` is stale and below the real current price, a depositor receives more rsETH than their deposit is worth at the current exchange rate. The excess rsETH represents a proportional claim on the protocol's TVL that was not backed by the depositor's contribution. After the price is eventually updated, the attacker's rsETH is correctly priced at the new higher rate, but they acquired it at a discount. The shortfall is borne by existing rsETH holders whose accrued yield is diluted. This matches the "Theft of unclaimed yield" impact class exactly.

## Likelihood Explanation

Staking rewards accrue continuously on EigenLayer and underlying LSTs. `updateRSETHPrice()` is called on a schedule (not on every deposit), so there is always a window — potentially hours — where `rsETHPrice` is stale. No privileged access is required. An attacker only needs to:

1. Monitor `rsETHPrice` on-chain and compare it to the live TVL computable from `getTotalAssetDeposits` and `getAssetPrice`.
2. Observe that TVL has grown since the last price update.
3. Call `depositETH()` or `depositAsset()` before `updateRSETHPrice()` is called.

The entire sequence is permissionless and repeatable. Likelihood is **Medium**.

## Recommendation

Remove the `view` modifier from `_beforeDeposit()` and call `ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice()` at the start of `_beforeDeposit()`, before `getRsETHAmountToMint()` is invoked. This ensures the denominator always reflects the current TVL at the time of deposit. Alternatively, replace the cached `rsETHPrice` read in `getRsETHAmountToMint()` with an inline computation of the live price (mirroring `_getTotalEthInProtocol() / rsethSupply`), though this is more invasive.

## Proof of Concept

**State at time T:**
- `rsETHPrice = 1.05e18` (TVL = 105 ETH, supply = 100 rsETH, set by last `updateRSETHPrice()` call)

**Staking rewards accrue:**
- TVL grows to 106 ETH. Real price = `106e18 / 100 = 1.06e18`. `rsETHPrice` is still `1.05e18`.

**Attacker calls `depositETH{value: 1 ether}("")`:**
- `getRsETHAmountToMint(ETH, 1e18)` = `(1e18 * 1e18) / 1.05e18 ≈ 0.9524 rsETH`
- Correct amount at real price: `(1e18 * 1e18) / 1.06e18 ≈ 0.9434 rsETH`
- Excess minted: `≈ 0.009 rsETH`

**After deposit:**
- TVL = 107 ETH, supply = 100.9524 rsETH
- When `updateRSETHPrice()` is eventually called: `newPrice = 107 / 100.9524 ≈ 1.05993e18`
- Correct post-deposit price (had attacker received fair amount): `107 / 100.9434 ≈ 1.06001e18`
- Existing holders' rsETH is worth ~0.008% less per unit — the attacker extracted their accrued yield.

**Foundry fork test plan:**
1. Fork mainnet at a block where `rsETHPrice` is known.
2. Advance time / simulate reward accrual so live TVL exceeds `rsETHPrice * supply`.
3. Call `depositETH` as an unprivileged address.
4. Assert `rsethAmountToMint > (depositAmount * 1e18) / liveComputedPrice`.
5. Call `updateRSETHPrice()` and assert the attacker's rsETH value exceeds their deposit.
