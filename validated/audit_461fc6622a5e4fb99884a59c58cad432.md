Audit Report

## Title
Fee Reduction Delay via Block Stuffing Forces Users to Pay Old Fee — (`contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

## Summary

`setFeeInBPS` applies the new fee immediately with no timelock. When the owner broadcasts a fee reduction, an attacker can stuff blocks with high-gas transactions to delay that transaction, causing users who submit `claim()` or `claimAndStake()` during the delay window to execute against the old (higher) fee. Neither claim function accepts a `maxFeeInBPS` guard, so users have no on-chain protection.

## Finding Description

`setFeeInBPS` writes directly to `feeInBPS` with no queuing or delay:

```solidity
// L426-432
function setFeeInBPS(uint256 _feeInBPS) external onlyOwner {
    if (_feeInBPS > MAX_FEE_IN_BPS) { revert InvalidFeeInBPS(); }
    feeInBPS = _feeInBPS;
    emit FeeInBPSUpdated(feeInBPS);
}
```

Both `claim` (L328) and `claimAndStake` (L363) read `feeInBPS` at execution time:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / FEE_DENOMINATOR;
```

Neither function accepts a caller-supplied `maxFeeInBPS` parameter, so there is no on-chain slippage guard.

**Attack path:**
1. Owner broadcasts `setFeeInBPS(0)` (reducing from 1000 BPS / 10%).
2. Users observe the pending transaction and submit `claim()` expecting 0% fee.
3. Attacker fills blocks with high-gas dummy transactions, keeping the owner's transaction out of the chain for N blocks.
4. Users' `claim()` transactions land in stuffed blocks and execute against `feeInBPS = 1000`.
5. Owner's `setFeeInBPS(0)` lands in block N+1 — after all user claims have settled.

No existing guard prevents this: there is no timelock on `setFeeInBPS`, no `maxFeeInBPS` parameter in `claim`/`claimAndStake`, and no pause-then-update pattern.

## Impact Explanation

Users receive `claimableAmount * 0.90` instead of the full `claimableAmount`. The 10% excess is transferred to `protocolTreasury` rather than to the claimants. This is a direct, concrete instance of **Low — Block stuffing**, which is an explicitly listed allowed impact. The contract fails to deliver the promised return (the announced 0% fee) due to a block-stuffing-enabled ordering attack.

## Likelihood Explanation

Block stuffing is expensive on Ethereum mainnet but is a well-documented, locally reproducible technique requiring no admin compromise or private-key leak. The precondition — owner publicly signals a fee reduction, users rush to claim — is realistic for a token distribution event. The attacker only needs capital to outbid normal gas prices for a small number of blocks. The contract provides zero on-chain defence.

## Recommendation

1. **Add a timelock to `setFeeInBPS`**: queue the new fee and enforce a minimum delay (e.g., 24–48 h) before it takes effect, giving users time to react.
2. **Add a `maxFeeInBPS` parameter to `claim` / `claimAndStake`**: revert if `feeInBPS > maxFeeInBPS` at execution time, giving users a slippage guard analogous to DEX `minAmountOut`.
3. **Alternatively, use a two-step commit-reveal or a scheduled fee change** that is announced on-chain before it activates.

## Proof of Concept

Deploy `KernelTop100MerkleDistributor` locally with `feeInBPS = 1000`. Submit `setFeeInBPS(0)` from the owner but do not mine it. Simulate block stuffing by advancing blocks (e.g., `vm.roll`) while submitting `claim()` calls from a valid merkle-proof holder. Each `claim()` executes against `feeInBPS = 1000`, deducting 10% to `protocolTreasury`. After mining the owner's transaction, `feeInBPS` becomes 0 — but all prior claims already settled at 10%.
