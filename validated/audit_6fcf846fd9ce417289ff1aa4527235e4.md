### Title
Single Attested Participant Can Deliver Forged CKD Output via Unverified `respond_ckd` for Legacy `AppPublicKey` Requests - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_ckd` skips all cryptographic output verification when the pending CKD request uses the legacy `CKDAppPublicKey::AppPublicKey` variant. A single Byzantine attested participant — strictly below the signing threshold — can race to call `respond_ckd` with an arbitrary, attacker-crafted `CKDResponse` and have it accepted and delivered to the waiting user. The MPC threshold is never enforced on the content of the response.

### Finding Description

`respond_ckd` enforces that the caller is an attested participant, but then branches on the request's `app_public_key` field:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For the `AppPublicKeyPV` (publicly-verifiable) variant the contract calls `ckd_output_check`, which verifies that the encrypted response is consistent with the MPC public key and the user's ephemeral key pair. For the legacy `AppPublicKey` variant the arm is an empty block — the response bytes are passed directly to `resolve_yields_for` and delivered to the user with zero verification. [1](#0-0) 

The `respond` (ECDSA/EdDSA) and `respond_verify_foreign_tx` endpoints both verify the submitted signature against the on-chain public key before resolving yields. [2](#0-1) 

`respond_ckd` is the only respond endpoint that omits this check for one of its input variants.

### Impact Explanation

A single Byzantine attested participant can:

1. Monitor the chain for a pending `CKDRequest` whose `app_public_key` is the legacy `AppPublicKey` variant.
2. Construct an arbitrary `CKDResponse` — for example, a ciphertext that encrypts an attacker-chosen plaintext under the user's `app_public_key`.
3. Call `respond_ckd` before the honest MPC network submits its response. Because `resolve_yields_for` removes the entry on first resolution, the attacker's response wins the race.
4. The user's yield resumes with the forged ciphertext. The user decrypts it and obtains a derived key whose plaintext the attacker already knows.

The confidentiality guarantee of the CKD protocol — that the derived key is known only to the user — is completely broken by a single participant acting alone, without any threshold cooperation. This matches the allowed critical impact: **confidential key derivation output without the required participant authorization**. [3](#0-2) 

### Likelihood Explanation

- The `AppPublicKey` (legacy, privately-verifiable) variant is the older format and is still accepted by the contract.
- Any single attested participant — a role reachable without threshold collusion — can exploit this.
- The attack requires only monitoring the NEAR chain for pending CKD requests (public data) and submitting a transaction before the honest nodes do. No leaked keys, no TEE bypass, no network-level access is needed.
- The attacker's transaction competes with the honest MPC response; on a lightly-loaded network the race is straightforward to win.

### Recommendation

Apply the same output verification to the `AppPublicKey` branch that is already applied to `AppPublicKeyPV`. If a publicly-verifiable check is not possible for the legacy format (because only one G1 point is available), the contract should either:

1. Reject new `AppPublicKey` CKD requests entirely (deprecate the variant), or
2. Require that `respond_ckd` callers supply a zero-knowledge proof of correct encryption even for the legacy variant, or
3. Require threshold-many identical responses before resolving the yield (analogous to how the MPC signing threshold works for signatures).

### Proof of Concept

```
// Attacker is one attested participant (account: attacker.near).
// User submitted: request_app_private_key({ app_public_key: AppPublicKey(user_g1_pk), ... })
// The pending CKDRequest is now visible on-chain.

// Attacker crafts a fake CKDResponse:
//   - Encrypts attacker-chosen plaintext P under user_g1_pk
//   - Produces a valid-looking CKDResponse bytes

// Attacker calls:
contract.respond_ckd(
    pending_ckd_request,   // copied from chain
    fake_ckd_response,     // attacker-controlled ciphertext
)
// → assert_caller_is_attested_participant passes (attacker is a participant)
// → AppPublicKey branch: empty, no check
// → resolve_yields_for delivers fake_ckd_response to the user's yield
// → User decrypts and obtains plaintext P, which attacker already knows
```

The contract accepts the call and the user receives a derived key whose value is fully controlled by the attacker, without any threshold of honest participants having participated in the derivation. [4](#0-3)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
