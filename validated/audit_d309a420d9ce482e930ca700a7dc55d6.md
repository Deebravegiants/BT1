Audit Report

## Title
`_checkAndUpdateDailyFeeMintLimit` Missing Zero-Guard Causes DoS on `updateRSETHPrice` When `maxFeeMintAmountPerDay` Is Uninitialized — (`contracts/LRTOracle.sol`)

## Summary

`maxFeeMintAmountPerDay` defaults to zero and is never set during `initialize` or `reinitialize`. When yield accrues and `protocolFeeInBPS > 0`, `_checkAndUpdateDailyFeeMintLimit` is called with a positive `feeAmount`, and the check `0 + feeAmount > 0` always evaluates to true, causing an unconditional revert. As a result, `rsETHPrice` is never updated past the stale value, freezing the yield that accruing users are entitled to claim.

## Finding Description

`maxFeeMintAmountPerDay` is declared at [1](#0-0)  with no initialization. `reinitialize` only sets `feePeriodStartTime` and does not touch `maxFeeMintAmountPerDay`: [2](#0-1) 

`setMaxFeeMintAmountPerDay` is an optional, privileged call: [3](#0-2) 

When `totalETHInProtocol > previousTVL` and `protocolFeeInBPS > 0`, a positive `rsethAmountToMintAsProtocolFee` is computed and passed to `_checkAndUpdateDailyFeeMintLimit`: [4](#0-3) 

Inside `_checkAndUpdateDailyFeeMintLimit`, there is no zero-guard on `maxFeeMintAmountPerDay`. The only check is: [5](#0-4) 

With `maxFeeMintAmountPerDay == 0` and any `feeAmount > 0`, this reduces to `feeAmount > 0` — always true — so every call reverts unconditionally. The `rsETHPrice = newRsETHPrice` assignment is never reached: [6](#0-5) 

By contrast, `remainingDailyFeeMintLimit()` already handles this case with an early return, but `_checkAndUpdateDailyFeeMintLimit` does not: [7](#0-6) 

`updateRSETHPriceAsManager()` calls the same `_updateRsETHPrice()` and is equally blocked: [8](#0-7) 

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

`rsETHPrice` is the exchange rate used to compute ETH owed to withdrawing users. While it is frozen at a stale (lower) value, the actual ETH backing per rsETH continues to grow. Users who withdraw during this window receive fewer ETH than the protocol holds on their behalf. The accrued yield is inaccessible — it cannot be realized by withdrawing users and accumulates silently in the protocol. No attacker captures the yield; it is frozen rather than stolen, placing this squarely in **Medium — Permanent freezing of unclaimed yield** rather than the High theft claim made in the submission.

## Likelihood Explanation

The preconditions are the default deployment state: `maxFeeMintAmountPerDay` is never required to be set during `initialize` or `reinitialize`; `protocolFeeInBPS > 0` is the normal operating configuration; yield accrues automatically. No attacker action is required — the broken state is reached as soon as the first yield-bearing `updateRSETHPrice()` call is made after deployment without the manager having called `setMaxFeeMintAmountPerDay`. The public `updateRSETHPrice()` function is callable by any address.

## Recommendation

Add a zero-limit bypass at the top of `_checkAndUpdateDailyFeeMintLimit`, consistent with the logic already present in `remainingDailyFeeMintLimit()`:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
+   // 0 means the limit has not been configured; treat as unlimited / fee disabled
+   if (maxFeeMintAmountPerDay == 0) return;

    if (block.timestamp >= feePeriodStartTime + 1 days) { ... }
    if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
        revert DailyFeeMintLimitExceeded(...);
    }
    currentPeriodMintedFeeAmount += feeAmount;
}
```

Alternatively, require `maxFeeMintAmountPerDay` to be set to a non-zero value inside `reinitialize` so the uninitialized state is impossible in production.

## Proof of Concept

```solidity
// Foundry-style pseudocode
function test_updateRSETHPrice_revertsWhenMaxFeeMintAmountPerDayIsZero() public {
    // Preconditions (default deployment state):
    assertEq(lrtOracle.maxFeeMintAmountPerDay(), 0);
    // protocolFeeInBPS > 0 (normal config)
    // rsETH totalSupply > 0

    // Simulate yield: increase totalAssetDeposits so totalETHInProtocol > previousTVL
    _simulateYield(1 ether);

    // updateRSETHPrice must revert with DailyFeeMintLimitExceeded
    vm.expectRevert(
        abi.encodeWithSelector(ILRTOracle.DailyFeeMintLimitExceeded.selector)
    );
    lrtOracle.updateRSETHPrice();

    // rsETHPrice is unchanged (stale)
    assertEq(lrtOracle.rsETHPrice(), previousPrice);
}
```

Revert path: `updateRSETHPrice` → `_updateRsETHPrice` → `protocolFeeInETH > 0` → `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` → `0 + rsethAmountToMintAsProtocolFee > 0` → `revert DailyFeeMintLimitExceeded`. [9](#0-8)
