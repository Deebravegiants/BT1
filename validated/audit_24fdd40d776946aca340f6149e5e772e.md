### Title
Excess Deposit Silently Absorbed in `submit_participant_info` When Existing Participant Re-Submits Attestation — (File: crates/contract/src/lib.rs)

---

### Summary

`submit_participant_info` is marked `#[payable]` but contains a conditional deposit-handling block that is entirely skipped when an existing participant re-submits their attestation. Any NEAR tokens attached to such a call are silently absorbed by the contract with no refund, permanently locking the caller's funds in the contract balance.

---

### Finding Description

In `submit_participant_info` (lines 758–852 of `crates/contract/src/lib.rs`), the deposit-handling and refund logic is gated behind a single condition:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... compute storage cost, check deposit, refund excess
}
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — which happens whenever the caller is an **existing participant** (`caller_is_not_participant = false`) **and** the submission is a **re-submission** (`is_new_attestation = false`) — the entire block is bypassed. No deposit check, no refund, and no rejection of a non-zero deposit occurs. Any NEAR tokens attached to the call are silently transferred to the contract's balance and cannot be recovered by the caller. [2](#0-1) 

This is the **normal periodic re-submission path** for every active MPC node. The node's `periodic_attestation_submission` task and `monitor_attestation_removal` task both call `submit_participant_info` on a recurring basis. [3](#0-2) 

The function is declared `#[payable]` with no guard that rejects or refunds a non-zero deposit on the re-submission branch: [4](#0-3) 

By contrast, every other payable user-facing function in the contract (`sign`, `request_app_private_key`, `verify_foreign_transaction`, `propose_update`) either calls `require_deposit` (which explicitly refunds any excess) or manually computes and refunds the difference: [5](#0-4) [6](#0-5) 

The README explicitly states "Any excess deposit is automatically refunded" as a general contract property, but this invariant is violated for the re-submission path of `submit_participant_info`. [7](#0-6) 

---

### Impact Explanation

**Medium.** Any NEAR tokens attached to a `submit_participant_info` re-submission call by an existing participant are permanently absorbed into the contract's balance. The caller has no mechanism to recover them: unlike the Ethereum analog where `refundETH()` at least makes the funds claimable (by anyone), in NEAR the tokens simply accumulate in the contract's storage-staking balance with no withdrawal path for the original depositor. This breaks the production accounting invariant that deposits are refunded when not consumed, and constitutes a direct, irreversible loss of funds for the caller.

---

### Likelihood Explanation

**Low-to-Medium.** The node software (`remote_attestation.rs`) does not currently attach a deposit when calling `submit_participant_info`, so the bug is not triggered in normal automated operation. However:

1. The function is public and `#[payable]`, so any NEAR account (including an existing participant calling manually, a script, or a future integration) can attach a deposit.
2. The re-submission path is the **most frequently executed** path for active nodes (hourly cadence plus on-removal events), making it the most likely to be hit if deposit attachment is ever introduced or accidentally included.
3. There is no on-chain guard that rejects a non-zero deposit on this branch, so the failure is silent — the caller receives `Ok(())` with no indication that their funds were not returned.

---

### Recommendation

Add an explicit refund for any attached deposit when `attestation_storage_must_be_paid_by_caller` is `false`. The simplest fix mirrors the pattern already used in `propose_update`:

```rust
} else {
    // Existing participant re-submitting: no storage cost, but refund any deposit.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, add a `#[deposit(0)]` guard (or equivalent panic) to reject non-zero deposits on the re-submission branch, making the invariant explicit.

---

### Proof of Concept

1. An existing MPC participant (`alice.near`) is already registered; `voter_account()` returns `Ok(...)`.
2. Alice calls `submit_participant_info` with a fresh attestation (same account, updated quote) and attaches `1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR).
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant`, so `is_new_attestation = false`.
4. `caller_is_not_participant = false` (Alice is a voter).
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The `if` block is skipped entirely — no cost check, no refund.
7. The call returns `Ok(())`. Alice's 1 NEAR is now in the contract's balance with no recovery path. [8](#0-7)

            }
        }
    }
}
