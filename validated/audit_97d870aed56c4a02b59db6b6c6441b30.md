### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` function applies cryptographic output verification only for the `AppPublicKeyPV` (publicly verifiable) variant of CKD requests. For the `AppPublicKey` (legacy/privately verifiable) variant, no on-chain verification is performed. This is the direct analog to the external report's pattern: one path (direct deposit to staker) applies the critical mechanism, while the other path (deposit + transfer) bypasses it. Here, `respond` always verifies signatures cryptographically, but `respond_ckd` skips verification for the `AppPublicKey` variant. Any single attested participant can submit an arbitrary fabricated `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept and deliver it to the requesting user without threshold authorization.

### Finding Description

In `respond_ckd`, the output check is gated on the variant of `app_public_key`:

```rust
// crates/contract/src/lib.rs:675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the response against the BLS master public key on-chain. For `AppPublicKey`, the contract performs **zero cryptographic verification** of the response values (`big_y`, `big_c`).

Contrast this with `respond` for threshold signatures, which **always** verifies the signature regardless of scheme (ECDSA or EdDSA):

```rust
// crates/contract/src/lib.rs:586-644
let signature_is_valid = match (&response, public_key) {
    (Secp256k1(...), ...) => { verify_ecdsa_signature(...).is_ok() }
    (Ed25519 {...}, ...) => { verify_eddsa_signature(...).is_ok() }
    ...
};
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
``` [2](#0-1) 

The `respond_ckd` function's only caller-authorization check is `assert_caller_is_attested_participant_and_protocol_active()`: [3](#0-2) 

There is no leader-only restriction and no threshold-vote requirement. Any single attested participant can call `respond_ckd` with arbitrary `big_y` and `big_c` values for any pending `AppPublicKey` CKD request. The first call to `resolve_yields_for` wins and delivers the response to the user: [4](#0-3) 

**Exploit path:**
1. User submits `request_app_private_key` with `AppPublicKey` variant (the legacy/default path).
2. Byzantine attested participant monitors the NEAR chain for the pending CKD request.
3. Byzantine participant immediately calls `respond_ckd` with attacker-chosen `big_y` and `big_c` values before the legitimate leader can respond.
4. The contract accepts the forged response (no verification), resolves all queued yields, and delivers the attacker-controlled key material to the user.
5. The legitimate leader's subsequent `respond_ckd` call finds no pending request and silently fails.

### Impact Explanation

**Critical.** A single Byzantine attested participant — strictly below the signing threshold — can forge a CKD response for any pending `AppPublicKey` request. The user receives an attacker-controlled confidential key derivation output instead of the legitimate threshold-computed one. Because the attacker chose `big_y` and `big_c`, they can compute the confidential key the user will derive, enabling full disclosure of the user's confidential key material. This constitutes **unauthorized confidential key derivation output without the required participant authorization**, matching the Critical impact category.

### Likelihood Explanation

**Low.** Requires a Byzantine attested participant — one that has passed TEE attestation but acts maliciously (e.g., compromised TEE, insider threat, or a node operator who has passed attestation but later turns adversarial). The attacker must also win the race against the legitimate leader. However, no collusion above one participant is required, and the `AppPublicKey` variant is the legacy default path, making it the most commonly used variant in practice.

### Recommendation

1. **Require `AppPublicKeyPV` for all new requests.** Deprecate and eventually remove the `AppPublicKey` variant, since it cannot be verified on-chain.
2. **If `AppPublicKey` must be retained**, implement a threshold-vote mechanism for `respond_ckd` analogous to how signing requires threshold computation: require that at least `t` participants independently submit the same response before it is accepted, rather than accepting the first response from any single participant.
3. **Apply the same consistency principle as `respond`**: every response path must have a cryptographic validity check before `resolve_yields_for` is called.

### Proof of Concept

```rust
// Attacker is an attested participant (account: "byzantine.near")
// User has submitted: request_app_private_key({ app_public_key: AppPublicKey(pk), ... })

// Byzantine participant calls respond_ckd with fabricated values:
contract.respond_ckd(
    CKDRequest { app_public_key: AppPublicKey(pk), domain_id, app_id, ... },
    CKDResponse {
        big_y: attacker_chosen_g1_point,  // arbitrary, not derived from MPC key
        big_c: attacker_chosen_g1_point,  // arbitrary, not derived from MPC key
    }
)
// → passes all checks (attested participant, running state, accept_requests=true)
// → AppPublicKey arm: no verification
// → resolve_yields_for delivers forged response to user
// → user receives attacker-controlled key material
// → attacker knows the "confidential key" the user will compute
```

The `AppPublicKeyPV` path would reject this immediately via `ckd_output_check`, but the `AppPublicKey` path has no equivalent guard. [1](#0-0) [5](#0-4)
