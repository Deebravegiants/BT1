### Title
Unbounded Pool Wallet Singleton Chain-Following Loop Allows Peer-Driven Wallet Sync Denial-of-Service and State Corruption — (`chia/wallet/wallet_state_manager.py`)

### Summary

The `while curr_coin_state.spent_height is not None` loop in `WalletStateManager._add_coin_states` that follows the pool wallet singleton chain has no iteration bound. A malicious peer can serve an infinite sequence of crafted `CoinSpend` objects and `CoinState` responses, causing the wallet sync task to block indefinitely and the pool store / coin store to be populated with fabricated state.

---

### Finding Description

The loop in question is at lines 2089–2123 of `chia/wallet/wallet_state_manager.py`:

```python
while curr_coin_state.spent_height is not None:
    cs: CoinSpend = await fetch_coin_spend_for_coin_state(curr_coin_state, peer)
    ...
    success = await singleton_wallet.apply_state_transition(cs, ...)
    if not success:
        break
    new_singleton_coin = get_most_recent_singleton_coin_from_coin_spend(cs)
    if new_singleton_coin is None:
        break
    ...
    new_coin_state = await self.wallet_node.get_coin_state([new_singleton_name], peer=peer, ...)
    assert len(new_coin_state) == 1
    curr_coin_state = new_coin_state[0]   # ← loop continues if spent_height is not None
``` [1](#0-0) 

**Termination conditions** — the loop only exits when:
1. `apply_state_transition` returns `False`
2. `get_most_recent_singleton_coin_from_coin_spend` returns `None`
3. The peer returns a coin state with `spent_height is None`

All three are fully under the peer's control.

**`get_most_recent_singleton_coin_from_coin_spend` executes the CLVM locally** — it calls `compute_additions`, which runs `run_with_cost(cs.puzzle_reveal, max_cost, cs.solution)` on the peer-supplied `CoinSpend`: [2](#0-1) [3](#0-2) 

The peer can supply a trivial puzzle such as `(q . ((51 <any_puzzle_hash> 1)))` that produces a `CREATE_COIN` condition with an odd amount, which is all that is needed for the function to return a non-`None` coin.

**`apply_state_transition` does not validate CLVM correctness or signatures** — it only checks that the spend's coin name matches the current tip and that the parent chain is consistent: [4](#0-3) 

The `add_spend` parent-chain check (`spend.coin.parent_coin_info != prev.coin.name()`) is also fully satisfiable by the attacker because the peer controls the `CoinSpend.coin` fields and the CLVM output that determines the next coin: [5](#0-4) 

**Concrete attack chain:**

| Iteration | Peer action | Guard outcome |
|---|---|---|
| 1 | Returns `CoinSpend` for real singleton A with trivial puzzle producing coin B (odd amount, parent=A) | `apply_state_transition` passes (coin A matches tip); `get_most_recent_singleton_coin_from_coin_spend` returns B |
| 1 | Returns `CoinState(B, created_height=H, spent_height=H+1)` | Loop continues |
| 2 | Returns `CoinSpend` for coin B with trivial puzzle producing coin C (parent=B) | Same guards pass |
| … | … | Loop never exits |

---

### Impact Explanation

- **Wallet sync task blocked indefinitely**: The asyncio task running `_add_coin_states` is stuck in the loop. No other coin state updates (XCH, CATs, NFTs, offers) can be processed while the loop runs.
- **Pool store and coin store corrupted**: Each iteration calls `apply_state_transition` → `pool_store.add_spend` and `coin_store.coin_added` / `set_spent`, writing fabricated spends and coin records into the DB within the outer `db_wrapper.writer()` transaction.
- **Sync state invariant violated**: The loop is supposed to terminate at the current chain tip; a peer can prevent this indefinitely.

This satisfies **High: Corruption of wallet sync state** and **High: Long-lived inability for honest wallets to process valid sync updates**.

---

### Likelihood Explanation

- Any peer the wallet connects to (including an untrusted full node) can trigger this.
- The only precondition is that the victim wallet has a `POOLING_WALLET` (a pool NFT / plot NFT), which is common among farmers.
- No keys, admin access, or broken cryptography are required.
- The attack is trivially scriptable: serve a fabricated `CoinState` with `spent_height is not None` for the singleton, then respond to every `RequestPuzzleSolution` and `get_coin_state` call with crafted data.

---

### Recommendation

Add an explicit iteration cap to the singleton chain-following loop, e.g.:

```python
MAX_SINGLETON_CHAIN_DEPTH = 100  # or a consensus-derived bound
depth = 0
while curr_coin_state.spent_height is not None:
    if depth >= MAX_SINGLETON_CHAIN_DEPTH:
        self.log.warning("Singleton chain depth exceeded limit; aborting loop")
        break
    depth += 1
    ...
```

Additionally, validate that the `CoinSpend` returned by the peer actually corresponds to a valid pool wallet puzzle (check puzzle hash against the known singleton structure) before calling `apply_state_transition`.

def compute_additions_with_cost(
    cs: CoinSpend,
    *,
    max_cost: int = DEFAULT_CONSTANTS.MAX_BLOCK_COST_CLVM,
) -> tuple[list[Coin], int]:
    """
    Run the puzzle in the specified CoinSpend and return the cost and list of
    coins created by the puzzle, i.e. additions. If the cost (CLVM- and
    condition cost) exceeds the specified max_cost, the function fails with a
    ValidationError exception. Byte cost is not included since at this point the
    puzzle and solution may have been decompressed, the true byte-cost can only be
    measured at the block generator level.
    """
    parent_id = cs.coin.name()
    ret: list[Coin] = []
    cost, r = run_with_cost(cs.puzzle_reveal, max_cost, cs.solution)
    for cond in Program.to(r).as_iter():
        if cost > max_cost:
            raise ValidationError(Err.BLOCK_COST_EXCEEDS_MAX, "compute_additions() for CoinSpend")
        atoms = cond.as_iter()
        op = next(atoms).atom
        if op in {
            ConditionOpcode.AGG_SIG_PARENT,
            ConditionOpcode.AGG_SIG_PUZZLE,
            ConditionOpcode.AGG_SIG_AMOUNT,
            ConditionOpcode.AGG_SIG_PUZZLE_AMOUNT,
            ConditionOpcode.AGG_SIG_PARENT_AMOUNT,
            ConditionOpcode.AGG_SIG_PARENT_PUZZLE,
            ConditionOpcode.AGG_SIG_UNSAFE,
            ConditionOpcode.AGG_SIG_ME,
        }:
            cost += ConditionCost.AGG_SIG.value
            continue
        if op != ConditionOpcode.CREATE_COIN.value:
            continue
        cost += ConditionCost.CREATE_COIN.value
        puzzle_hash = next(atoms).as_atom()
        amount = uint64(next(atoms).as_int())
        ret.append(Coin(parent_id, puzzle_hash, uint64(amount)))

    return ret, cost
```
