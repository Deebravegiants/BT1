### Title
NFT Offer Royalty Dilution via Multi-NFT Offer Bundling — (`File: chia/wallet/nft_wallet/nft_wallet.py`)

### Summary
The Chia NFT offer settlement code in `make_nft1_offer` divides the fungible trade price evenly across all offered NFTs to compute per-NFT royalties. A seller can bundle additional zero- or low-royalty NFTs into the same offer to dilute the trade price per NFT, causing the high-royalty NFT's creator to receive a fraction of the royalty they are owed. This is enforced at the CLVM puzzle level via the `trade_prices_list` embedded in the NFT spend.

### Finding Description

In `make_nft1_offer`, the `offer_side_royalty_split` counter is incremented once for every royalty-enabled NFT the maker is offering: [1](#0-0) 

The trade price passed to each offered NFT's transfer program is then: [2](#0-1) 

`amount // offer_side_royalty_split` — the total fungible amount divided equally by the number of offered NFTs. This per-NFT trade price is what the NFT's on-chain transfer program uses to compute the royalty it enforces. The same dilution exists in the display-layer `royalty_calculation` static method: [3](#0-2) 

Line 719 divides `abs(amount)` by `len(royalty_assets_dict)` before applying the royalty percentage. Both paths — the on-chain spend construction and the off-chain summary — share the same flaw.

The `trade_prices_list` is passed directly into the NFT spend at: [4](#0-3) 

This means the diluted trade price is embedded in the actual CLVM spend bundle, not merely a display artifact.

### Impact Explanation

A seller offering NFT_A (10% royalty, 1000 basis points) alongside N zero-royalty NFTs in a single offer reduces the royalty paid to NFT_A's creator from:

```
royalty = total_xch * 1000 // 10000
```

to:

```
royalty = (total_xch // (N+1)) * 1000 // 10000
```

With N=9 zero-royalty NFTs, the royalty drops to ~10% of what it should be. With large N and small `total_xch`, integer truncation in `abs(offered_amount) // royalty_split` drives the royalty to zero entirely. This is an on-chain accounting change affecting NFT royalty settlement — the royalty creator's coin receives fewer mojos than the NFT's embedded royalty percentage entitles them to.

### Likelihood Explanation

Any NFT holder can create an offer. Minting zero-royalty NFTs is permissionless and cheap. The attacker only needs to own (or mint) additional NFTs to bundle into the offer. No privileged access, leaked keys, or cryptographic breaks are required. The offer is accepted by the taker in the normal flow via `respond_to_offer`, and the diluted royalty is settled on-chain.

### Recommendation

Royalty for each NFT should be computed against the full fungible amount, not the amount divided by the number of NFTs. The `offer_side_royalty_split` divisor should be removed from the trade price calculation, and each NFT's royalty should be computed independently as `total_fungible_amount * royalty_percentage // 10000`. The `royalty_calculation` static method should similarly remove the `// len(royalty_assets_dict)` divisor.

### Proof of Concept

**Setup:** Maker owns NFT_A (royalty: 1000 bps = 10%) and NFT_B (royalty: 0 bps).

**Offer:** Maker creates an offer via `make_nft1_offer`:
- Offering: NFT_A + NFT_B (both negative amounts in `offer_dict`)
- Requesting: 1,000,000 mojos XCH

**Execution path:**

1. `offer_side_royalty_split` = 2 (NFT_A + NFT_B both have `amount < 0`)
2. Trade price per NFT = `1,000,000 // 2 = 500,000` mojos
3. NFT_A royalty enforced by transfer program = `500,000 * 1000 // 10000 = 50,000` mojos
4. NFT_B royalty = `500,000 * 0 // 10000 = 0` mojos
5. Total royalty paid = 50,000 mojos

**Without NFT_B bundled:**
- Trade price = `1,000,000 // 1 = 1,000,000` mojos
- NFT_A royalty = `1,000,000 * 1000 // 10000 = 100,000` mojos

**Result:** By bundling NFT_B, the maker halves the royalty owed to NFT_A's creator (100,000 → 50,000 mojos). With N zero-royalty NFTs, the royalty approaches zero. The diluted `trade_prices_list` is embedded in the spend bundle and enforced on-chain by the NFT transfer program puzzle. [5](#0-4) [6](#0-5)

```python
def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
