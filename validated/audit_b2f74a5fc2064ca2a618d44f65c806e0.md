### Title
Caller-Agnostic `verify_foreign_transaction` Fan-Out Queue Can Be Exhausted for Negligible Cost, Permanently Blocking Specific Foreign-Transaction Verifications - (File: `crates/contract/src/pending_requests.rs`)

### Summary

The `verify_foreign_transaction` endpoint uses a caller-agnostic request key, meaning all callers submitting the same foreign-chain transaction share a single bounded fan-out queue capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`. An unprivileged attacker can fill this queue for any targeted foreign transaction with 128 spam submissions costing only 128 yoctonear + gas, permanently blocking legitimate callers from queuing their verification of that transaction.

### Finding Description

The `pending_requests` module caps each request key's fan-out queue at `MAX_PENDING_REQUEST_FAN_OUT = 128`: [1](#0-0) 

When the queue is full, `push_pending_yield` panics with `PendingRequestQueueFull`, reverting the caller's transaction: [2](#0-1) 

For `sign()`, the stored key is a `SignatureRequest` whose `tweak` field is derived from `(predecessor_id, path)`, so each caller's queue is isolated: [3](#0-2) 

For `verify_foreign_transaction()`, however, the stored key is a `VerifyForeignTransactionRequest` that contains only `domain_id`, `payload_version`, and the chain-specific `request` — **the caller's account ID is not part of the key**. The contract itself acknowledges this in a test comment: [4](#0-3) 

This means every caller submitting the same foreign-chain transaction (same `tx_id`, `chain`, `extractors`, `domain_id`) shares one 128-slot queue. An attacker who submits 128 identical `verify_foreign_transaction` calls for a targeted transaction fills the queue entirely. Any subsequent legitimate submission for that transaction is rejected with `PendingRequestQueueFull`.

The `verify_foreign_transaction` function enforces only a 1-yoctonear deposit: [5](#0-4) 

The minimum deposit constant is 1 yoctonear: [6](#0-5) 

Total attacker cost to saturate the queue: **128 yoctonear + gas** (negligible). The NEAR yield-resume timeout is approximately 200 blocks. After the queue drains (via MPC response or timeout), the attacker can immediately refill it, sustaining the DoS indefinitely.

### Impact Explanation

A bridge or dApp that relies on `verify_foreign_transaction` to confirm a foreign-chain deposit or trigger an on-chain action can have a specific transaction's verification permanently blocked. The victim's `verify_foreign_transaction` call reverts with `PendingRequestQueueFull` every time they retry, as long as the attacker continuously refills the queue. This breaks the request-lifecycle safety invariant: a valid, confirmed foreign-chain transaction that should be verifiable cannot be verified, potentially locking user funds in a bridge indefinitely.

This maps to: **Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

### Likelihood Explanation

The attack requires the attacker to know the target transaction's parameters (`tx_id`, `chain`, `extractors`, `domain_id`), which are observable on-chain or in the mempool. The cost is 128 yoctonear + gas per ~200-block cycle — effectively free. No privileged access, threshold collusion, or TEE compromise is required. Any unprivileged NEAR account can execute this.

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key (analogous to how `SignatureRequest` includes the `tweak` derived from the caller). This gives each caller an isolated queue slot, preventing cross-caller queue saturation. Alternatively, enforce a per-account rate limit or a meaningful deposit that scales with the number of concurrent pending requests per account.

### Proof of Concept

1. Alice submits `verify_foreign_transaction` for Bitcoin `tx_id = 0xABC` (1 yoctonear deposit).
2. Attacker submits 128 identical `verify_foreign_transaction` calls for the same `tx_id = 0xABC` (128 yoctonear + gas total), filling the queue.
3. Alice's transaction reverts: `PendingRequestQueueFull` (limit: 128).
4. MPC nodes respond to the attacker's 128 queued yields; queue drains after ~200 blocks if no response.
5. Attacker immediately resubmits 128 calls. Alice is permanently blocked.

The fan-out cap and the caller-agnostic key are both confirmed by the existing unit test: [7](#0-6)

```rust
/// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```

```rust
    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
```

```rust
            domain_id: DomainId::default().0.into(),
            payload_version: ForeignTxPayloadVersion::V1,
            request: dtos::ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
                tx_id: [7u8; 32].into(),

            }),

        let alice = AccountId::from_str("alice.near").unwrap();

                .signer_account_id(alice.clone())
                .predecessor_account_id(alice)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()
        );

                .predecessor_account_id(bob)
```
