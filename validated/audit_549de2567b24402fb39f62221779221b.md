### Title
Empty-Metadata Token Registration via Uninitialized Metaplex PDA in `log_metadata` — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

### Summary
An unprivileged attacker can call `log_metadata` for any classic SPL token whose Metaplex metadata PDA is uninitialized (system-owned). The `parse_metadata_account` function passes the `require_keys_eq` address check but returns empty `name`/`symbol` strings because the account owner is not `MetaplexID`. The vault is created and a Wormhole VAA with empty name/symbol is emitted. The NEAR `deploy_token_callback` has no non-empty metadata guard, so the token is permanently registered on NEAR with empty name/symbol, blocking any future legitimate registration.

### Finding Description

In `parse_metadata_account`:

```rust
require_keys_eq!(
    metadata.key(),
    address,
    ErrorCode::InvalidTokenMetadataAddress,
);
if metadata.owner == &MetaplexID {
    // ... deserialize and return real name/symbol
} else {
    Ok((String::default(), String::default()))  // ← empty strings, no error
}
``` [1](#0-0) 

The `require_keys_eq!` guard only validates the account *address* matches the deterministically derived Metaplex PDA — it does not validate the account is owned by `MetaplexID` or is initialized. For a mint with no Metaplex metadata, the PDA exists as a system-owned zero-data account. Passing it satisfies the key check, but the owner branch falls to the `else` arm returning `("", "")`.

The vault is unconditionally created via `init_if_needed`: [2](#0-1) 

And the payload is posted to Wormhole with empty strings: [3](#0-2) 

On the NEAR side, `deploy_token_callback` passes the name/symbol directly to `deploy_token_internal` with no validation: [4](#0-3) 

This is inconsistent with the NEAR-native `log_metadata_callback`, which explicitly enforces non-empty metadata: [5](#0-4) 

`deploy_token_internal` then inserts the token into `deployed_tokens` with a uniqueness guard: [6](#0-5) 

Once inserted with empty metadata, any subsequent legitimate `deploy_token` call for the same mint will fail with `TokenExists`, permanently blocking the correct registration path.

The test harness itself documents this exact scenario as the default case: [7](#0-6) 

### Impact Explanation

- Any classic SPL token without a Metaplex metadata account can be registered on NEAR with empty `name`/`symbol` by an unprivileged attacker.
- The legitimate token owner is permanently locked out of the normal `deploy_token` registration path for their token.
- Downstream chains (EVM, Starknet, Aptos) that subsequently call `deploy_token` using the NEAR-signed payload would deploy wrapped tokens with empty name/symbol, breaking asset-identity guarantees across the entire bridge.
- Recovery requires privileged admin intervention via `set_token_metadata`.

This is an asset-identity divergence: the on-chain token identity (name/symbol) does not match the real token, violating the invariant that native token registration requires valid metadata — an invariant explicitly enforced on the NEAR-native path but absent on the Solana→NEAR path.

### Likelihood Explanation

High. The attack requires no privileges, no special setup beyond creating (or finding) a classic SPL mint without Metaplex metadata, and no external dependencies. The attacker only needs to submit a valid Solana transaction and then relay the resulting Wormhole VAA to NEAR. The test suite confirms the code path succeeds.

### Recommendation

Add a non-empty guard in `parse_metadata_account` (or in `process`) that returns an error if `name` or `symbol` is empty after resolving metadata:

```rust
if name.trim().is_empty() || symbol.trim().is_empty() {
    return err!(ErrorCode::InvalidTokenMetadata);
}
```

Alternatively, require that the metadata account is owned by `MetaplexID` for classic SPL tokens — i.e., treat a system-owned Metaplex PDA as a missing metadata error rather than silently returning empty strings.

### Proof of Concept

1. Create a classic SPL mint (`spl-token create-token`) — do **not** create Metaplex metadata for it.
2. Derive the Metaplex PDA: `find_program_address([b"metadata", MetaplexID, mint_pubkey], MetaplexID)`. This PDA will be system-owned with zero data.
3. Call `log_metadata` on the Solana bridge program, passing the system-owned PDA as the `metadata` account.
4. `require_keys_eq` passes (correct PDA address). `metadata.owner != MetaplexID` → returns `("", "")`.
5. Vault is created. Wormhole VAA is emitted with `name=""`, `symbol=""`, correct `decimals` and `token` pubkey.
6. Relay the VAA to NEAR's `deploy_token`. `deploy_token_callback` has no empty-string guard; `deploy_token_internal` registers the token with empty name/symbol.
7. Assert: `get_token_id` returns a valid token ID; `ft_metadata` on the deployed token returns `name=""`, `symbol=""`.
8. Assert: a second `deploy_token` call for the same mint fails with `TokenExists`, confirming the legitimate registration path is permanently blocked.
