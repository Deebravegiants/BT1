Audit Report

## Title
Third-party `addLiquidity` on behalf of victim creates dust remainder that permanently blocks `removeLiquidity` — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no requirement that `msg.sender == owner`, allowing any caller to credit shares to a victim's position. `LiquidityLib.removeLiquidity` enforces that any non-zero post-burn remainder must be at or above `minimalMintableLiquidity`. An attacker can front-run a victim's full-withdrawal transaction by adding exactly 1 share to the victim's position, causing the victim's transaction to leave a 1-share remainder that falls below the dust floor and reverts. This can be repeated indefinitely, permanently locking the victim's principal.

## Finding Description

`MetricOmmPool.addLiquidity` accepts `owner` independently of `msg.sender` with no ownership check:

```solidity
// metric-core/contracts/MetricOmmPool.sol L182-196
function addLiquidity(
  address owner,   // ← arbitrary, not validated against msg.sender
  uint80 salt,
  LiquidityDelta calldata deltas,
  ...
```

The callback is invoked on `msg.sender` (the attacker pays tokens), but shares are credited to the victim's `(owner, salt, binIdx)` key via `LiquidityLib.addLiquidity`. The add-side dust check at `LiquidityLib.sol` L76-79 only reverts if `newUserShares < minimalMintableLiquidity`; when the victim already holds `X >= minimalMintableLiquidity` shares, adding 1 yields `X+1 >= minimalMintableLiquidity`, so the check passes silently.

`removeLiquidity` enforces `msg.sender == owner` (L206), so only the victim can call it, but then enforces the dust floor on the remainder:

```solidity
// metric-core/contracts/libraries/LiquidityLib.sol L199-202
uint256 newUserShares = userShares - sharesToRemove;
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
  revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

Attack sequence:
1. Victim holds `X` shares (`X >= minimalMintableLiquidity`, e.g. 10,000) and submits `removeLiquidity(victim, salt, {X shares}, "")`.
2. Attacker front-runs with `addLiquidity(victim, salt, {1 share}, callbackData, "")`. Victim now holds `X+1` shares.
3. Victim's tx executes: `newUserShares = (X+1) - X = 1`. Check: `1 > 0 && 1 < 1000` → reverts with `MinimalLiquidity(1, 1000)`.
4. Attacker repeats on every retry.

The position key `keccak256(abi.encode(owner, salt, bin))` is fully reconstructible from on-chain `LiquidityAdded` events, so the attacker can always target the correct position. The periphery test `test_exactShares_canAddOnBehalfOfAnotherOwner` explicitly confirms the third-party add path works. The optional `DepositAllowlistExtension` (checked against `sender`, not `owner`) could block the attacker only if deployed and configured — it is not present by default.

## Impact Explanation

An EOA victim cannot withdraw their liquidity as long as the attacker keeps front-running. The victim's principal is locked in the pool indefinitely. This constitutes broken core pool functionality causing loss of funds / unusable withdraw flow, which is an explicitly allowed impact. The victim's only escape is a contract-based atomic read-and-remove, which the protocol's periphery does not provide.

## Likelihood Explanation

The attack requires MEV front-running capability (available on most EVM chains) and knowledge of the victim's `(owner, salt, binIdx)` tuple, which is fully public from on-chain `LiquidityAdded` events. The per-attack token cost is the value of 1 share — negligible — making sustained griefing economically viable. Any pool deployed without a `DepositAllowlistExtension` is vulnerable to any unprivileged caller.

## Recommendation

1. **Restrict third-party adds**: In `addLiquidity`, require `msg.sender == owner` unless the owner has explicitly granted operator approval via an on-chain allowance mapping or signed permit.
2. **Alternatively**, add a `removeAllLiquidity` variant that burns the caller's entire share balance in a bin without applying the dust floor check on the remainder (since the remainder is exactly 0).
3. **Alternatively**, remove the `MinimalLiquidity` check from `removeLiquidity` entirely (apply it only on `addLiquidity`), so users can always exit to zero regardless of intermediate share counts.

## Proof of Concept

```solidity
contract Attacker is IMetricOmmModifyLiquidityCallback {
    IERC20 token0; IERC20 token1;
    constructor(address t0, address t1) { token0 = IERC20(t0); token1 = IERC20(t1); }

    function grief(address pool, address victim, uint80 salt, int256 bin) external {
        int256[] memory bins = new int256[](1);
        bins[0] = bin;
        uint256[] memory shares = new uint256[](1);
        shares[0] = 1; // passes add-side check because victim already has >= 1000 shares
        LiquidityDelta memory d = LiquidityDelta({binIdxs: bins, shares: shares});
        IMetricOmmPool(pool).addLiquidity(victim, salt, d, "", "");
    }

    function metricOmmModifyLiquidityCallback(uint256 a0, uint256 a1, bytes calldata) external {
        if (a0 > 0) token0.transfer(msg.sender, a0);
        if (a1 > 0) token1.transfer(msg.sender, a1);
    }
}
// 1. victim has 10_000 shares in bin 4
// 2. victim submits removeLiquidity(victim, salt, {10_000 shares}, "")
// 3. attacker.grief(pool, victim, salt, 4)  ← front-runs; victim now has 10_001 shares
// 4. victim's tx: newUserShares = 10_001 - 10_000 = 1 < 1000 → MinimalLiquidity revert
// 5. repeat indefinitely
```
