### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the domain's root public key, but it never verifies that `response.payload_hash` is the hash of a `ForeignTxSignPayload` that encodes the `request` argument supplied to the same call. A single Byzantine attested MPC node can therefore replay a legitimately-produced MPC signature (obtained from a completed signing round for Request A) as the response to a different pending request (Request B), causing every caller waiting on Request B to receive a `VerifyForeignTransactionResponse` whose `payload_hash` encodes Request A's foreign-chain observations rather than Request B's.

---

### Finding Description

The `respond_verify_foreign_tx` entry point in `crates/contract/src/lib.rs` accepts two arguments:

- `request: VerifyForeignTransactionRequest` — used as the map key to locate and drain the pending yield queue.
- `response: VerifyForeignTransactionResponse` — contains `payload_hash` (a `Hash256`) and `signature`.

The only cryptographic check performed is:

```rust
// crates/contract/src/lib.rs  lines 718–734
let signature_is_valid = match (&response.signature, public_key) {
    (
        dtos::SignatureResponse::Secp256k1(signature_response),
        PublicKeyExtended::Secp256k1 { near_public_key },
    ) => {
        let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
            .expect("Secp256k1 variant always has a secp256k1 key");

        let payload_hash: [u8; 32] = response.payload_hash.0;

        // Check the signature is correct against the root public key
        near_mpc_signature_verifier::verify_ecdsa_signature(
            signature_response,
            &payload_hash,
            &secp_pk,
        )
        .is_ok()
    }
``` [1](#0-0) 

The contract confirms only that `signature` is a valid ECDSA signature over `payload_hash`. It does **not** confirm that `payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request: <the request argument>, values: <any values> }))`.

Compare this with the regular `respond` function, where the payload is taken directly from the `request` argument (the user-submitted hash), so the binding is automatic:

```rust
// crates/contract/src/lib.rs  lines 600–608
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
.is_ok()
``` [2](#0-1) 

For `verify_foreign_transaction`, the `payload_hash` is MPC-node-supplied because the extracted values are determined off-chain. The contract must therefore explicitly verify the binding — but it does not.

The `ForeignTxSignPayload` that nodes sign encodes both the original request and the observed values:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [3](#0-2) 

Because `request` is embedded in the signed payload, a `payload_hash` produced for Request A (`tx_id = [1u8;32]`) is cryptographically distinct from one produced for Request B (`tx_id = [2u8;32]`). The contract never checks this distinction.

After the signature check passes, the contract unconditionally drains the pending yield queue for the supplied `request` key and delivers the unchecked `response` to every waiting caller:

```rust
// crates/contract/src/lib.rs  lines 749–753
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [4](#0-3) 

---

### Impact Explanation

A single Byzantine attested MPC node (strictly below the signing threshold) can execute the following replay:

1. Participate honestly in the signing round for Request A (`tx_id_A`). After the round completes, the node holds the threshold signature `sig_A` over `hash_A = SHA-256(borsh(ForeignTxSignPayload{request=Request_A, values=[block_hash_A]}))`.
2. While Request B (`tx_id_B`) is pending in the contract, call `respond_verify_foreign_tx(request = Request_B, response = { payload_hash: hash_A, signature: sig_A })`.
3. The contract verifies `sig_A` over `hash_A` using the root key — this check passes because `sig_A` is a genuine MPC signature.
4. The contract looks up Request B in `pending_verify_foreign_tx_requests` — this check passes because Request B is pending.
5. Every caller waiting on Request B receives `{ payload_hash: hash_A, signature: sig_A }` — a response that encodes Request A's foreign-chain observations, not Request B's.

Any downstream NEAR bridge contract that does not independently recompute the expected `payload_hash` from its own `request` and `extracted_values` (using `ForeignChainSignatureVerifier`) will accept this response as proof that Request B's foreign transaction was verified, enabling invalid bridge execution or double-spend conditions. The SDK helper does perform this check, but the on-chain contract — the authoritative security boundary — does not enforce it.

This matches the allowed impact: **High — forged foreign-chain verification and invalid bridge execution**.

---

### Likelihood Explanation

- The attacker is a single attested MPC participant, strictly below the signing threshold. No collusion is required.
- The attacker obtains `sig_A` by participating in a legitimate signing round; no key material is leaked.
- The attacker needs only one pending request (Request B) to be in the queue simultaneously with a completed signing round for any other request (Request A) on the same domain.
- The call to `respond_verify_foreign_tx` is permissioned only to attested participants; no privileged operator access beyond that is required.
- The attack is replayable: any previously produced signature for the same domain can be reused against any future pending request on that domain.

---

### Recommendation

Inside `respond_verify_foreign_tx`, require the caller to also supply the `extracted_values` that were observed, then recompute and compare the expected hash before accepting the response:

```rust
// Pseudocode
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: extracted_values,   // caller must now supply these
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, embed the `request` field into the signature verification path so the contract can at minimum confirm that `payload_hash` encodes the correct `request` without needing the `extracted_values` (e.g., by signing `(request_hash || extracted_values_hash)` as two separate commitments).

---

### Proof of Concept

1. Deploy the contract with a ForeignTx domain (Secp256k1).
2. Submit `verify_foreign_transaction(request = Request_A)` and let the MPC network produce `sig_A` over `hash_A`. Record `(hash_A, sig_A)` from the on-chain event.
3. Submit `verify_foreign_transaction(request = Request_B)` (different `tx_id`).
4. Call `respond_verify_foreign_tx(request = Request_B, response = { payload_hash: hash_A, signature: sig_A })` from any attested participant account.
5. Observe that the call succeeds (`Ok(())`), Request B's pending queue is drained, and the yield callback `return_verify_foreign_tx_and_clean_state_on_success` delivers `{ payload_hash: hash_A, signature: sig_A }` to Request B's caller — a response that encodes Request A's foreign-chain state, not Request B's.

The contract-level unit test infrastructure in `crates/contract/src/lib.rs` already demonstrates the full `respond_verify_foreign_tx` → `return_verify_foreign_tx_and_clean_state_on_success` flow and can be extended to reproduce this scenario directly.
