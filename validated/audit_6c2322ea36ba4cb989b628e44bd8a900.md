### Title
`get_supported_foreign_chains` check is enforced at request submission but not at response time, allowing nodes to fulfill pending `verify_foreign_transaction` requests for chains that have since been removed from the supported list - (File: crates/contract/src/lib.rs)

### Summary
The `verify_foreign_transaction` entry-point enforces that the requested foreign chain is in the contract's supported-chain set. However, the corresponding node response handler `respond_verify_foreign_tx` never re-checks this condition. If participants vote to remove a foreign chain from the supported list after a request has already been queued, MPC nodes can still call `respond_verify_foreign_tx` and the contract will accept the response, producing a valid threshold signature for a chain that was explicitly disabled.

### Finding Description
`verify_foreign_transaction` (the user-facing entry point) performs two key checks before enqueuing a request:

1. `check_request_preconditions` — verifies `accept_requests` is `true` and the domain exists.
2. An explicit supported-chain guard:

```rust
// crates/contract/src/lib.rs ~L533-541
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported { requested: requested_chain }.to_string(),
    );
}
```

The request is then stored in `pending_verify_foreign_tx_requests` via `enqueue_yield_request`.

`respond_verify_foreign_tx` (the node-facing response handler) checks:
- `assert_caller_is_signer()`
- `assert_caller_is_attested_participant_and_protocol_active()`
- `protocol_state.is_running_or_resharing()`
- `accept_requests`

It then verifies the cryptographic signature and calls `pending_requests::resolve_yields_for`. **At no point does it re-check whether the foreign chain is still in the supported-chain set.**

```rust
// crates/contract/src/lib.rs ~L692-753
pub fn respond_verify_foreign_tx(
    &mut self,
    request: VerifyForeignTransactionRequest,
    response: VerifyForeignTransactionResponse,
) -> Result<(), Error> {
    let signer = Self::assert_caller_is_signer();
    self.assert_caller_is_attested_participant_and_protocol_active();
    if !self.protocol_state.is_running_or_resharing() { ... }
    if !self.accept_requests { ... }
    // ← no get_supported_foreign_chains() check here
    ...
    pending_requests::resolve_yields_for(
        &mut self.pending_verify_foreign_tx_requests,
        &request,
        serde_json::to_vec(&response).unwrap(),
    )
}
```

This is structurally identical to the reported pattern: the guard is applied at the assignment/submission path but is absent from the execution/resolution path.

### Impact Explanation
When participants vote to remove a foreign chain from the supported set (e.g., because a verification flaw or chain-level compromise is discovered), the intent is to halt all MPC-backed activity for that chain. However, any `verify_foreign_transaction` request that was already enqueued before the removal will still be processed and signed by the MPC network. The contract will accept the `respond_verify_foreign_tx` call and deliver a valid threshold signature to the original requester. This breaks the production safety invariant that only supported chains receive MPC-signed attestations, and constitutes a request-lifecycle/contract execution-flow manipulation that bypasses the per-chain circuit breaker.

This maps to the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

### Likelihood Explanation
The scenario is realistic in any incident-response situation: a chain is found to have a verification weakness or is compromised, participants vote to remove it from `node_foreign_chain_support`, but one or more `verify_foreign_transaction` requests submitted just before the removal are still in the yield queue. Nodes will observe those pending requests via the indexer and call `respond_verify_foreign_tx`, which the contract accepts without complaint. No collusion above the signing threshold is required — this is normal honest-node behavior.

### Recommendation
Add a supported-chain re-check inside `respond_verify_foreign_tx`, mirroring the guard in `verify_foreign_transaction`:

```diff
 pub fn respond_verify_foreign_tx(
     &mut self,
     request: VerifyForeignTransactionRequest,
     response: VerifyForeignTransactionResponse,
 ) -> Result<(), Error> {
     let signer = Self::assert_caller_is_signer();
     self.assert_caller_is_attested_participant_and_protocol_active();
     if !self.protocol_state.is_running_or_resharing() {
         return Err(InvalidState::ProtocolStateNotRunning.into());
     }
     if !self.accept_requests {
         return Err(TeeError::TeeValidationFailed.into());
     }
+    let requested_chain = request.request.chain();
+    let supported_chains = self.get_supported_foreign_chains();
+    if !supported_chains.contains(&requested_chain) {
+        return Err(InvalidParameters::ForeignChainNotSupported {
+            requested: requested_chain,
+        }.into());
+    }
     ...
 }
```

This ensures the per-chain circuit breaker is honored consistently across both the submission and resolution paths, matching the behavior of `accept_requests` which is already enforced in both.

### Proof of Concept

1. Participants vote to add Bitcoin to the supported foreign-chain set; `get_supported_foreign_chains()` now returns `{Bitcoin}`.
2. User calls `verify_foreign_transaction` with a Bitcoin request. `supported_chains.contains(&Bitcoin)` passes; the request is stored in `pending_verify_foreign_tx_requests` under a yield ID.
3. A Bitcoin verification flaw is discovered. Participants vote to remove Bitcoin from `node_foreign_chain_support`; `get_supported_foreign_chains()` now returns `{}`.
4. An honest MPC node, having already picked up the pending request from the indexer, calls `respond_verify_foreign_tx` with a valid threshold signature.
5. The contract executes lines 705–753: checks `accept_requests` (true), verifies the ECDSA signature (valid), and calls `resolve_yields_for` — delivering the signature to the user.
6. The user receives a valid MPC-signed attestation for a Bitcoin transaction on a chain the network has explicitly disabled, bypassing the intended circuit breaker. [1](#0-0) [2](#0-1) [3](#0-2)
