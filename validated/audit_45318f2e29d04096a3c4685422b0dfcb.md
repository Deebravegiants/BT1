### Title
Unchecked Chainlink `latestRoundData()` Return Values Enable Stale/Invalid Price Acceptance - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but only consumes the `price` field, silently discarding `roundId`, `updatedAt`, and `answeredInRound`. This is the direct analog of the reported bug: the call executes successfully, but the validity of the returned result is never verified. A stale, incomplete, or zero-price round passes through unchecked and propagates into the rsETH exchange rate.

### Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`latestRoundData()` returns five values: `(roundId, answer, startedAt, updatedAt, answeredInRound)`. The contract discards all fields except `answer`. No check is performed for:

- **Stale round**: `answeredInRound < roundId` — the answer was computed in a prior round and has not been refreshed.
- **Incomplete round**: `updatedAt == 0` — the round has not been finalised.
- **Non-positive price**: `price <= 0` — a zero or negative answer is cast directly to `uint256`, producing a wildly incorrect value.

Contrast this with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 27–32, which is present in the same repository and correctly validates all three conditions before returning a price.

The stale/invalid price then propagates through `LRTOracle.getAssetPrice()` → `LRTOracle._getTotalEthInProtocol()` → `LRTOracle._updateRsETHPrice()`, setting the on-chain `rsETHPrice` used to mint rsETH tokens for every depositor.

### Impact Explanation
Any depositor calling `LRTDepositPool.depositAsset()` or `LRTDepositPool.depositETH()` has their rsETH mint amount computed from `rsETHPrice`, which is derived from the stale/invalid Chainlink answer. If an LST's real market price has fallen significantly but the feed is stale (e.g., during a sequencer outage, extreme volatility, or a Chainlink node failure), depositors receive more rsETH than the underlying collateral warrants, diluting all existing rsETH holders and causing protocol insolvency over time. Conversely, a zero price (if `price` returns 0) causes a division-by-zero or zero-valued rsETH mint, freezing deposits. Impact: **share/asset mis-accounting → protocol insolvency or temporary fund freeze**.

### Likelihood Explanation
Chainlink feeds do go stale during network congestion, sequencer downtime (on L2s), or when a feed is deprecated. The condition requires no attacker action — a passive market event is sufficient. Any unprivileged depositor can call `depositAsset()` immediately after `updateRSETHPrice()` is called with a stale price, receiving excess rsETH at the expense of the protocol.

### Recommendation
Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();
    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

### Proof of Concept

1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale — `answeredInRound` lags behind `roundId` — while the real stETH price has dropped 5%.
2. Anyone calls `LRTOracle.updateRSETHPrice()`. Inside `_getTotalEthInProtocol()`, `ChainlinkPriceOracle.getAssetPrice(stETH)` is called.
3. `latestRoundData()` returns the stale (inflated) price; no staleness check exists, so it is accepted.
4. `rsETHPrice` is set 5% higher than it should be.
5. An attacker calls `LRTDepositPool.depositAsset(stETH, amount, 0, "")`. The mint calculation uses the inflated `rsETHPrice`, issuing ~5% more rsETH than the deposited collateral is worth.
6. The attacker immediately redeems or holds rsETH, extracting value from existing holders.
