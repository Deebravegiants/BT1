Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice()` Accepts Stale/Zero Chainlink Prices, Enabling rsETH Price Deflation and Fund Theft - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validity fields, using only the raw `price` value with no checks for staleness, round completeness, or positivity. A stale or zero Chainlink price deflates `rsETHPrice` via `LRTOracle._updateRsETHPrice()`, allowing an attacker to mint rsETH at below-fair-value and extract value from all existing rsETH holders when the oracle recovers. The same codebase already applies all three required checks in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the omission is a code defect.

## Finding Description

**Root cause:** `ChainlinkPriceOracle.getAssetPrice()` at lines 52–54 silently discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check that `price > 0`, `answeredInRound >= roundId`, or `updatedAt != 0`. By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` at lines 30–32 applies all three:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

**Exploit path (Critical — theft):**

1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale, returning `price = 0` or a stale low value.
2. Anyone calls `LRTOracle.updateRSETHPrice()` (public, no access control). This calls `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `0`.
3. `totalETHInProtocol` is computed excluding all stETH TVL. `newRsETHPrice` is set to a deflated value.
4. The downside-protection check at lines 270–282 only pauses the protocol if `pricePercentageLimit > 0` AND the drop exceeds the limit. When `pricePercentageLimit == 0` (the default uninitialized value), the condition `pricePercentageLimit > 0 && ...` is always `false` — no pause occurs regardless of how large the price drop is.
5. With `rsETHPrice` deflated and deposits still open, the attacker calls `LRTDepositPool.depositAsset()` with a different, validly-priced asset (e.g., rETH). `getRsETHAmountToMint` computes `(amount * getAssetPrice(rETH)) / rsETHPrice` — the deflated denominator yields far more rsETH than the deposit is worth.
6. When the Chainlink feed recovers and `rsETHPrice` is updated to its true value, the attacker's rsETH is worth more than what they paid. They withdraw at profit, diluting all other rsETH holders.

**Existing guards and why they fail:**

- `pricePercentageLimit` downside protection: ineffective when `pricePercentageLimit == 0` (default). Even when set, a stale price drop within the configured limit passes through without pause.
- `_validatePrices` in `unlockQueue` (lines 853–870): only applies to the withdrawal unlock path and only if the operator passes a non-zero `minimumAssetPrice`. It does not protect `updateRSETHPrice()` or `depositAsset()`.
- `updatePriceOracleForValidated` sanity check (lines 102–107): only runs at oracle registration time, not on every price fetch.

## Impact Explanation

**Critical — Direct theft of user funds (rsETH holder dilution):** A stale Chainlink feed for any supported LST causes `rsETHPrice` to be set below its true value. An attacker mints rsETH at the deflated price using a different asset, then redeems at the recovered true price, extracting value from all existing rsETH holders. This is a direct, concrete theft of at-rest funds, not hypothetical. The default `pricePercentageLimit == 0` means no automatic circuit-breaker prevents the deflated price from being committed.

**Medium — Temporary freezing of funds:** If `assetPrice = 0` reaches `_calculatePayoutAmount` (when `minimumAssetPrice = 0` is passed to `unlockQueue`), division by zero reverts, blocking withdrawal unlocks for that asset until the oracle recovers.

## Likelihood Explanation

Chainlink feeds can return stale data during network congestion, sequencer downtime (on L2), or feed deprecation — all externally observable, requiring no privileged access. `updateRSETHPrice()` is callable by any unprivileged address. The default `pricePercentageLimit == 0` means no deployment configuration is required to trigger the worst-case scenario. The attacker needs only to observe a stale feed event and call two public functions (`updateRSETHPrice` then `depositAsset`).

## Recommendation

Apply the same validity checks already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, add a per-feed heartbeat staleness check (`block.timestamp - updatedAt > MAX_DELAY`) and ensure `pricePercentageLimit` is set to a non-zero value at deployment.
