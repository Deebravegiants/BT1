### Title
Stale Participant Votes Not Filtered in TEE Code-Hash / Launcher / OS-Measurement Voting After Resharing — (`crates/contract/src/lib.rs`)

### Summary

After a resharing completes, the contract spawns a detached (fire-and-forget) cleanup promise to remove non-participant votes from `tee_state`. Unlike `vote_update`, which explicitly re-filters votes against the current participant set at evaluation time, the TEE governance voting functions (`vote_code_hash`, `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, `vote_remove_os_measurement`) count raw stored votes without filtering out stale entries from removed participants. If the cleanup promise fails silently, a removed adversarial participant's pre-resharing vote persists and can combine with a single new participant's vote to cross the threshold, whitelisting an unauthorized MPC node image.

---

### Finding Description

After `vote_reshared` transitions the contract to `Running`, it spawns several detached cleanup promises: [1](#0-0) 

One of these is `CLEAN_TEE_STATUS`, which calls `self.tee_state.clean_non_participant_votes(participants)`: [2](#0-1) 

All cleanup promises are `.detach()`-ed — they are fire-and-forget. If any fail (e.g., due to insufficient gas in `self.config.clean_tee_status_tera_gas`), stale votes from removed participants silently persist in `tee_state`.

The TEE voting functions then count raw stored votes without filtering:

```rust
// vote_code_hash (line ~1418)
let votes = self.tee_state.vote(code_hash, &participant);
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
``` [3](#0-2) 

The same pattern applies to `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, and `vote_remove_os_measurement`: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

By contrast, `vote_update` explicitly defends against exactly this failure mode with a comment acknowledging the risk: [8](#0-7) 

The comment reads: *"Filter votes to only count current participants voting for this specific update. This ensures correctness even if the cleanup promise in `MpcContract::vote_reshared()` fails."* This defense was applied to `vote_update` but **not** to any of the TEE governance voting functions.

---

### Impact Explanation

If a removed participant's stale vote for a malicious node image hash (`code_hash`) persists after resharing, a single new participant's vote can push the tally over the threshold. `whitelist_tee_proposal` then marks that image as an allowed TEE binary. Nodes running the malicious image are subsequently accepted as valid attested participants via `submit_participant_info` and `vote_new_parameters`. A malicious MPC node inside the threshold set can exfiltrate key shares or produce unauthorized signatures, constituting **critical** impact: unauthorized access to MPC key shares and unauthorized threshold signature issuance.

---

### Likelihood Explanation

- The cleanup promise is detached and gas-bounded by `self.config.clean_tee_status_tera_gas`. Misconfiguration or a large participant set can cause it to fail silently.
- The developers themselves acknowledged this exact failure mode for `vote_update` (see the inline comment), confirming it is a realistic operational condition, not a theoretical one.
- An adversarial participant being removed via resharing (e.g., after TEE attestation expiry triggers `verify_tee`) is a normal protocol event, not a privileged action.
- The attacker only needs to cast one vote before being removed; no threshold-level collusion is required.

---

### Recommendation

Apply the same at-evaluation-time participant filtering used in `vote_update` to all TEE governance voting functions. Before comparing `votes >= threshold`, recount only votes cast by accounts in the current participant set:

```rust
// Example fix for vote_code_hash
let valid_votes = threshold_parameters
    .participants()
    .participants()
    .iter()
    .filter(|(account_id, _, _)| {
        self.tee_state.has_voted_for_code_hash(code_hash, account_id)
    })
    .count() as u64;

if valid_votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
```

Apply the same pattern to `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, and `vote_remove_os_measurement`.

---

### Proof of Concept

1. Network has 3 participants (A, B, C), threshold = 2.
2. Participant A calls `vote_code_hash(malicious_hash)`. Vote stored in `tee_state`.
3. A's TEE attestation expires; `verify_tee` triggers resharing that removes A. New set: {B, C}.
4. `vote_reshared` completes; `CLEAN_TEE_STATUS` detached promise is dispatched but fails due to gas exhaustion. A's vote for `malicious_hash` remains in `tee_state`.
5. Participant B (colluding or deceived) calls `vote_code_hash(malicious_hash)`. `tee_state.vote()` returns 2 (A's stale + B's fresh). `2 >= threshold(2)` → `whitelist_tee_proposal(malicious_hash)` executes.
6. Attacker deploys a node running `malicious_hash`. It passes `submit_participant_info` attestation checks and is voted into the next epoch via `vote_new_parameters`.
7. The malicious node participates in signing rounds and exfiltrates key shares. [9](#0-8)

    pub fn clean_tee_status(&mut self) -> Result<(), Error> {
        log!("clean_tee_status: signer={}", env::signer_account_id());

        let participants = match &self.protocol_state {
            ProtocolContractState::Running(state) => state.parameters.participants(),
            _ => {
                return Err(InvalidState::ProtocolStateNotRunning.into());
            }
        };

        self.tee_state.clean_non_participant_votes(participants);
        Ok(())
    }
