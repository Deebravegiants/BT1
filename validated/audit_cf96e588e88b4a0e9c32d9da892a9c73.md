### Title
Post-Reshard Stale ShardUId Insertion Strands Transactions in Dead Pools — (`chain/chunks/src/client.rs`, `chain/client/src/client.rs`, `chain/client/src/rpc_handler.rs`)

---

### Summary

A TOCTOU race between the RPC handler's shard-layout read and the pool mutex acquisition allows a transaction to be inserted under an old `ShardUId` after `reshard` has already migrated all transactions to the new shard layout. The transaction is accepted with `ValidTx` but is permanently stranded in a dead pool and never included in any chunk. The codebase itself contains a TODO comment acknowledging this exact gap.

---

### Finding Description

**Step 1 — `reshard` leaves empty HashMap entries.**

`ShardedTransactionPool::reshard` drains each old-shard pool by iterating through `get_pool_iterator`, which is a consuming/draining iterator. After the loop the old `ShardUId` keys remain in `tx_pools` as empty `TransactionPool` entries; they are never removed. [1](#0-0) 

**Step 2 — `pool_for_shard` reuses existing (empty) entries.**

`pool_for_shard` uses `HashMap::entry().or_insert_with(...)`. If the old `ShardUId` key already exists (even as an empty pool), `or_insert_with` is a no-op and the existing empty pool is returned. A subsequent `insert_transaction(old_shard_uid, tx)` therefore silently inserts into the dead pool. [2](#0-1) 

**Step 3 — The TOCTOU window in `process_tx_internal`.**

`RpcHandlerActor::process_tx_internal` reads the current epoch's `shard_layout` (and derives `shard_uid` from it) **before** it acquires the pool mutex:

- Line 175–177: epoch_id and shard_layout are read from `epoch_manager`.
- Line 216: `shard_uid` is computed from that (potentially stale) layout.
- Line 279: the pool mutex is acquired.
- Line 280: `insert_transaction(shard_uid, ...)` is called. [3](#0-2) [4](#0-3) [5](#0-4) 

Between lines 217 and 279, the Client actor (on its own thread) can process the epoch-boundary block, acquire the same mutex, call `reshard`, and release the mutex. When the RPC handler then acquires the mutex, it inserts the transaction under the now-dead old `shard_uid`.

**Step 4 — The codebase acknowledges this gap.**

The call site for `reshard` in `client.rs` contains an explicit TODO:

```
// TODO make sure transactions don't get added for the old shard
// layout after the pool resharding
``` [6](#0-5) 

---

### Impact Explanation

A transaction inserted into a dead old-shard pool is never pulled by the chunk producer, because chunk production iterates pools keyed by **new** `ShardUId`s. The transaction expires after the validity period. The submitter receives `ProcessTxResponse::ValidTx` (or `Success` from `insert_transaction`) but the transaction is never included in any chunk — a silent, confirmed-but-unexecuted transaction loss. [7](#0-6) 

---

### Likelihood Explanation

The race window is narrow (between the shard-layout read and the pool-lock acquisition, spanning only a few microseconds of CPU time), but:

- It is triggered at every epoch boundary where a shard-layout change occurs.
- `RpcHandlerActor` is explicitly spawned as a **multi-threaded** actor running concurrently with the Client actor.
- An unprivileged user only needs to submit a transaction at the right moment near an epoch boundary — no special privileges required.
- The TODO comment confirms the developers are aware the window exists and have not yet closed it. [8](#0-7) 

---

### Recommendation

1. **Remove stale entries after reshard**: after draining each old pool, call `tx_pools.remove(&old_shard_uid)` so `pool_for_shard` cannot resurrect a dead pool.
2. **Re-derive shard_uid under the lock**: inside the pool-lock critical section, re-read the current shard layout from `epoch_manager` and recompute `shard_uid` before calling `insert_transaction`, so the layout used for insertion is always consistent with the post-reshard state.
3. **Resolve the TODO**: the comment at `client.rs:1890–1891` should be converted into one of the above fixes.

---

### Proof of Concept

```
Thread A (RpcHandlerActor):
  epoch_id  = epoch_manager.get_epoch_id(...)          // returns OLD epoch
  shard_layout = epoch_manager.get_shard_layout(epoch_id) // OLD layout
  shard_uid = old_layout.account_id_to_shard_uid(signer) // OLD shard uid
  // --- preempted here ---

Thread B (Client actor):
  guarded_pool = sharded_tx_pool.lock()
  guarded_pool.reshard(&old_layout, &new_layout)
    // drains old pools, leaves empty HashMap entries for old ShardUIds
    // re-inserts all txs under new ShardUIds
  drop(guarded_pool)

Thread A (resumes):
  pool = sharded_tx_pool.lock()
  pool.insert_transaction(OLD_shard_uid, validated_tx)
    // pool_for_shard finds existing empty entry for OLD_shard_uid
    // inserts tx into dead pool
  // returns InsertTransactionResult::Success → caller sees ValidTx
  // tx is never pulled by chunk producer → permanently stranded
```
