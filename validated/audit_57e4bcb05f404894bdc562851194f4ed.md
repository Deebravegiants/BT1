### Title
`respond_verify_foreign_tx` Does Not Validate `payload_hash` Against the Original Request — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function in the MPC contract verifies that the submitted signature is valid over the caller-supplied `response.payload_hash`, but never checks that `payload_hash` is actually derived from the `request` field in the same call. A single Byzantine attested participant (below threshold) can execute a **cross-request replay**: take a valid MPC signature produced for one foreign-chain verification request and use it to resolve a completely different pending request, delivering an incorrect `payload_hash` to the waiting user.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 691–754), the contract performs the following checks:

1. Caller is an attested participant.
2. The ECDSA signature in `response.signature` is valid over `response.payload_hash` using the domain's root public key.
3. The pending request keyed by `request` exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What the contract does **not** check is that `response.payload_hash` was actually computed from the `request` argument supplied in the same call. The expected hash is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))
```

Because `extracted_values` are determined off-chain by the MPC nodes, the contract cannot recompute this hash independently. However, it also makes no attempt to bind the `payload_hash` to the `request` in any other way.

The `VerifyForeignTransactionResponse` struct simply carries an opaque `payload_hash: Hash256` alongside the signature: [2](#0-1) 

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) **does** perform this check — it recomputes the expected hash from the request and the caller-supplied expected extracted values and asserts equality: [3](#0-2) 

But this check lives entirely in the off-chain SDK, not in the on-chain contract. The contract resolves the yield and delivers the response to the user without enforcing it.

**Cross-request replay attack path:**

1. User submits `verify_foreign_transaction` for request A (`tx_id = X`). The pending request is stored under key `request_A`.
2. The Byzantine participant happens to be the elected leader for a different request B (`tx_id = Y`). After the threshold signing protocol completes, the leader holds the full signature `sig_B` over `hash_B = SHA256(borsh({request_B, values_B}))`.
3. Instead of (or in addition to) submitting the honest response for request B, the attacker calls:
   ```
   respond_verify_foreign_tx(
       request  = request_A,          // matches the pending entry → lookup succeeds
       response = { payload_hash: hash_B, signature: sig_B }
   )
   ```
4. The contract verifies `sig_B` over `hash_B` → valid. It resolves request A's yield with the serialized response.
5. The user who submitted request A receives `{ payload_hash: hash_B, signature: sig_B }` — a response that commits to `tx_id = Y`'s block hash, not `tx_id = X`'s. [4](#0-3) 

---

### Impact Explanation

A bridge or application contract that calls `verify_foreign_transaction` and then acts on the returned `VerifyForeignTransactionResponse` without independently recomputing and comparing `payload_hash` (i.e., without using the SDK verifier) will accept a forged attestation. Concretely:

- A bridge waiting for confirmation that deposit transaction X occurred on a foreign chain receives a response whose `payload_hash` actually commits to transaction Y's block hash.
- If the bridge trusts the MPC contract's response directly (a natural assumption given the contract already verified the signature), it may credit the deposit, enabling a double-spend or theft of bridged funds.

This maps to the allowed High impact category: **forged foreign-chain verification that causes invalid bridge execution or double-spend conditions**.

---

### Likelihood Explanation

- Requires only a **single** Byzantine attested participant — well below the signing threshold.
- The attacker does not need to forge any cryptographic material; they reuse a legitimately produced MPC signature from a different session.
- The attacker must be elected leader for at least one request (probabilistic, but guaranteed over time for any long-running participant).
- The target pending request (request A) is publicly visible on-chain, so the attacker can choose a high-value target.
- No network-level DoS, TEE break, or key leak is required.

---

### Recommendation

The contract should bind the submitted `payload_hash` to the `request` it is resolving. Two complementary approaches:

1. **Require extracted values in the response**: Extend `VerifyForeignTransactionResponse` to include the `Vec<ExtractedValue>` alongside `payload_hash`. The contract then recomputes `SHA256(borsh(ForeignTxSignPayload { request, extracted_values }))` and asserts it equals `payload_hash` before accepting the response. This mirrors the SDK check and moves it on-chain.

2. **Short-term mitigation**: At minimum, add a prominent on-chain assertion that rejects any `respond_verify_foreign_tx` call where `response.payload_hash` cannot be verified against the request. Until extracted values are included in the response, document clearly that every consumer **must** call `ForeignChainSignatureVerifier::verify_signature` from the SDK before acting on the response.

### Proof of Concept

```rust
// Setup: two pending requests, A and B, both for Bitcoin.
// Byzantine node is the leader for request B and obtains sig_B over hash_B.

// Attacker calls respond_verify_foreign_tx with request=A but response from B:
contract.respond_verify_foreign_tx(
    request_A,   // pending key → lookup succeeds
    VerifyForeignTransactionResponse {
        payload_hash: hash_B,   // hash of (request_B, values_B), NOT request_A
        signature: sig_B,       // valid MPC signature over hash_B
    },
);
// Contract checks: verify_ecdsa_signature(sig_B, hash_B, domain_pk) → OK
// Contract resolves request_A's yield with the wrong payload_hash.
// User receives hash_B, which commits to tx_id=Y's block hash, not tx_id=X's.
// A bridge contract that does not call ForeignChainSignatureVerifier::verify_signature
// will accept this as a valid attestation for tx_id=X.
```

The root cause is at `crates/contract/src/lib.rs` lines 718–743, where `response.payload_hash` is used directly as the message to verify without any check that it was derived from the `request` argument in the same call. [5](#0-4)
