Audit Report

## Title
Stale `rsETHPrice` Combined with Live Asset Price in `instantWithdrawal` Enables Over-Redemption — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager.instantWithdrawal()` computes the payout via `getExpectedAssetAmount`, which divides the stored (cached) `rsETHPrice` by the live `getAssetPrice(asset)`. When an LST depegs upward, `updateRSETHPrice()` inflates `rsETHPrice` to reflect the higher TVL. If the LST price then normalises before the next `updateRSETHPrice()` call, the stale inflated `rsETHPrice` is divided by the now-lower live asset price, causing `instantWithdrawal` to pay out more LST than the burned rsETH is worth. The excess is drawn from the vault, permanently reducing the backing of all remaining rsETH holders.

## Finding Description

`LRTOracle` stores `rsETHPrice` as a state variable updated only when `_updateRsETHPrice()` is explicitly called: [1](#0-0) [2](#0-1) 

`_getTotalEthInProtocol()` reads **live** oracle prices at call time: [3](#0-2) 

`updateRSETHPrice()` is a public, permissionless function (no role restriction): [4](#0-3) 

When called during an LST upward depeg, `newRsETHPrice` is computed from the inflated TVL and written to storage: [5](#0-4) 

`instantWithdrawal` calls `getExpectedAssetAmount`, which uses the **stored** `rsETHPrice` against the **live** asset price: [6](#0-5) 

The full `instantWithdrawal` execution path applies no cap — it pays out the full computed amount: [7](#0-6) 

By contrast, the queued-withdrawal path (`_unlockWithdrawalRequests` → `_calculatePayoutAmount`) caps the payout at `min(expectedAssetAmount, currentReturn)`: [8](#0-7) 

This cap is entirely absent from `instantWithdrawal`. The `CantInstantWithdrawMoreThanAvailable` check (line 231–233) only limits the total vault liquidity consumed, not the per-rsETH exchange rate.

The `pricePercentageLimit` guard (lines 252–266 of `LRTOracle.sol`) only blocks non-manager callers when the price increase exceeds the configured threshold. When `pricePercentageLimit == 0` (unset), or when the depeg is within the limit, any caller can write the inflated price. Even with a limit set, a manager calling `updateRSETHPriceAsManager()` can write it unconditionally.

The downside-protection auto-pause (lines 270–282 of `LRTOracle.sol`) only triggers on a subsequent `updateRSETHPrice()` call after the price drops. The attacker exploits the window **before** that call, while `rsETHPrice` is still stale-high and the live asset price has already normalised.

## Impact Explanation

**Critical — Direct theft of funds from other rsETH holders.**

The attacker burns rsETH worth `X` ETH at fair value but receives LST worth `X + δ` ETH. The excess `δ` is drawn from the vault's LST reserves. When `updateRSETHPrice()` is next called, the reduced TVL is divided across the remaining rsETH supply, permanently lowering `rsETHPrice` for all holders. This is a reduction in principal backing, not merely unclaimed yield — qualifying as direct theft of user funds (Critical).

## Likelihood Explanation

- LST upward depegs are historically observed (stETH has traded above 1 ETH on secondary markets).
- `updateRSETHPrice()` is public and permissionless; any actor can call it to lock in the inflated price.
- LST oracle prices can normalise within the same block or within a few blocks, creating an exploitable window before the next `updateRSETHPrice()` call.
- `isInstantWithdrawalEnabled[asset]` must be set by the manager, but this is a realistic and documented operational configuration.
- No special privilege is required to call `instantWithdrawal`. [9](#0-8) 

## Recommendation

In `getExpectedAssetAmount` (or directly in `instantWithdrawal`), replace the stored `lrtOracle.rsETHPrice()` with a freshly computed rsETH price derived from `_getTotalEthInProtocol()` at the time of the call. Alternatively, apply the same `min(expectedAmount, currentReturn)` cap already present in `_calculatePayoutAmount` — computing `currentReturn` using a fresh TVL snapshot rather than the cached price. A simpler short-term mitigation is to require that `rsETHPrice` was updated within the current block before any `instantWithdrawal` is processed.

## Proof of Concept

1. Protocol holds 100 stETH. stETH oracle = 1.000 ETH. `rsETHPrice` = 1.000 ETH. 100 rsETH outstanding.
2. stETH depegs upward: Chainlink reports stETH = 1.050 ETH.
3. Attacker (or anyone) calls `updateRSETHPrice()`. `_getTotalEthInProtocol()` returns `100 × 1.050 = 105 ETH`. `newRsETHPrice = 105 / 100 = 1.050 ETH`. Written to `rsETHPrice`.
4. stETH normalises: Chainlink reports stETH = 1.000 ETH. `rsETHPrice` is still `1.050` (stale).
5. Attacker holds 10 rsETH (fair value = 10 ETH at current stETH price).
6. Attacker calls `instantWithdrawal(stETH, 10e18, "")`:
   - `getExpectedAssetAmount`: `10 × 1.050 / 1.000 = 10.5 stETH`
   - Attacker receives 10.5 stETH (worth 10.5 ETH). Burned rsETH worth 10 ETH. **0.5 ETH stolen.**
7. Vault now holds 89.5 stETH for 90 rsETH. Next `updateRSETHPrice()` sets `rsETHPrice = 89.5 / 90 ≈ 0.9944 ETH` — all remaining holders are diluted.

        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;
