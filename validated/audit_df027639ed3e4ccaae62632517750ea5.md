Audit Report

## Title
Depositor Mints rsETH at Stale Cached Price, Stealing Accrued Yield from Existing Holders - (File: contracts/LRTDepositPool.sol, contracts/LRTOracle.sol)

## Summary

`LRTOracle` stores a cached `rsETHPrice` that is only updated on explicit calls to the public, permissionless `updateRSETHPrice()`. `LRTDepositPool.getRsETHAmountToMint()` divides by this cached value to determine how many rsETH tokens to mint. When EigenLayer rewards increase the true TVL between price updates, a depositor can mint rsETH at the stale lower price, receive more rsETH than their deposit warrants, and then trigger the price update — diluting all pre-existing rsETH holders by the over-minted amount.

## Finding Description

`LRTOracle.rsETHPrice` is a storage variable updated only inside `_updateRsETHPrice()`: [1](#0-0) [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` reads this cached value directly: [3](#0-2) 

`_beforeDeposit()` calls `getRsETHAmountToMint()` with no price-refresh step: [4](#0-3) 

Both public deposit entry points (`depositETH`, `depositAsset`) are unrestricted and flow through `_beforeDeposit`: [5](#0-4) 

**Exploit path:**

1. EigenLayer rewards accrue. True TVL = `T_true`, rsETH supply = `S`, true price `P_true = T_true/S`. Stored `rsETHPrice = P_old < P_true`.
2. Attacker calls `depositETH{value: Y}(0, "")`. Minted rsETH = `Y / P_old` (fair amount = `Y / P_true`). Attacker receives `Y*(P_true - P_old)/(P_old * P_true)` excess rsETH.
3. Attacker (or any keeper) calls `updateRSETHPrice()`. New price = `(T_true + Y) / (S + Y/P_old)`.
4. Attacker's excess rsETH now represents a real claim on protocol ETH, diluting all pre-existing holders.

**Existing guards are insufficient:**

- `minRSETHAmountExpected` slippage parameter protects the depositor from receiving *less* than expected; it does not prevent receiving *more* at a stale price.
- `pricePercentageLimit` in `_updateRsETHPrice()` limits how much the price can increase per update call, but does not prevent the deposit at the stale price. If the reward accrual is within the limit, the attack proceeds unimpeded. If above the limit, a non-manager attacker cannot call `updateRSETHPrice()` themselves, but the deposit at stale price has already occurred and the manager will eventually update the price. [6](#0-5) 

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders earn yield as EigenLayer rewards increase the protocol's TVL, reflected in a rising `rsETHPrice`. By depositing at the stale lower price, the attacker captures a portion of that accrued yield before it is priced in, diluting every existing holder proportionally. The stolen amount per attack is approximately `deposit_amount × (P_true − P_old) / P_old`. The attack is repeatable every time rewards accrue and the price has not yet been updated.

## Likelihood Explanation

**Medium.** No special role or permission is required — any EOA with capital can execute this. The attacker does not need to frontrun a specific mempool transaction; they only need to observe on-chain state (e.g., EigenLayer strategy balances or the deposit pool ETH balance) to confirm that rewards have accrued and `rsETHPrice` is stale. The attacker can also frontrun a pending `updateRSETHPrice()` transaction in the mempool to guarantee ordering. The staleness window exists by design (price is not updated on every block), making the precondition routinely satisfied.

## Recommendation

1. **Atomic price refresh before minting**: Call `_updateRsETHPrice()` (or compute the live price inline) inside `_beforeDeposit()` before invoking `getRsETHAmountToMint()`, so every deposit uses the current TVL-derived price.
2. **Staleness guard**: Revert deposits if `rsETHPrice` was last updated more than a configurable maximum staleness window ago (e.g., 1 hour), requiring a price update before deposits can proceed.
3. **Commit-reveal / deposit delay**: Introduce a delay between deposit initiation and rsETH minting (analogous to `withdrawalDelayBlocks`), so the minting price reflects state after the deposit is committed and the price has been refreshed.

## Proof of Concept

```
Initial state:
  rsETHPrice (stored) = 1.00e18   (stale)
  True rsETH price    = 1.01e18   (1% rewards accrued, TVL increased)
  rsETH supply        = 100_000e18

Step 1: Attacker calls depositETH{value: 10_000 ether}(0, "")
  rsethAmountToMint = 10_000e18 * 1e18 / 1.00e18 = 10_000e18 rsETH
  Fair amount       = 10_000e18 * 1e18 / 1.01e18 ≈  9_901e18 rsETH
  Over-minted       ≈ 99e18 rsETH

Step 2: Attacker calls LRTOracle.updateRSETHPrice()
  New TVL    = 101_000e18 (old) + 10_000e18 (deposit) = 111_000e18 ETH
  New supply = 100_000e18 + 10_000e18 = 110_000e18 rsETH
  New price  = 111_000e18 / 110_000e18 ≈ 1.00909e18

Step 3: Attacker redeems 10_000 rsETH
  Value = 10_000e18 * 1.00909e18 / 1e18 ≈ 10_090.9 ETH
  Profit ≈ 90.9 ETH — stolen from existing holders' accrued yield
```

**Foundry fork test plan**: Fork mainnet, set `rsETHPrice` to a value 1% below the live TVL-derived price (simulating staleness), call `depositETH` with a large value, then call `updateRSETHPrice()`, and assert that the attacker's rsETH redemption value exceeds their deposit amount while existing holders' per-rsETH redemption value has decreased.
