### Title
Presignature Rerandomization Entropy Derived from Predictable Block Random Seed Enables Precomputation of Rerandomization Scalar — (File: `crates/node/src/types.rs`)

---

### Summary

The `entropy` field used to rerandomize ECDSA presignatures is sourced directly from the NEAR block's random seed (`block.entropy`). NEAR block producers know their own VRF output — and therefore the block's `random_seed` — before they publish the block. This makes the rerandomization scalar `delta = HKDF(entropy, …)` precomputable by any adversary who is also a NEAR block producer, violating the explicit security requirement of the [GS21] rerandomization scheme that entropy must be "public, freshly generated, and **unpredictable**."

---

### Finding Description

The `SignatureRequest` and `CKDRequest` structs both receive their `entropy` field from `block.entropy` at the moment the on-chain event is indexed:

```rust
// crates/node/src/types.rs  line 138
entropy: block.entropy.clone().into(),
```

This entropy is then passed directly into `RerandomizationArguments` and used as the IKM for HKDF to derive the rerandomization scalar `delta`:

```rust
// crates/threshold-signatures/src/ecdsa/rerandomization.rs  line 88
let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);
```

The `RerandomizationArguments` struct's own documentation states the requirement:

```rust
// crates/threshold-signatures/src/ecdsa/rerandomization.rs  lines 16-17
/// *** Warning ***
/// Following [GS21], the entropy should be public, freshly generated, and unpredictable.
```

In NEAR Protocol, the `random_seed` of a block is the VRF output of the block producer. The block producer computes this value locally before publishing the block, meaning they know `block.entropy` for any block they produce **before** that block is finalized and before the signing protocol runs. An adversary who is a NEAR block producer can therefore:

1. Receive a signing request with a chosen payload.
2. Compute the entropy value that will be assigned to that request (their own VRF output).
3. Precompute `delta = HKDF(entropy, pk, msg_hash, big_r, participants)` before the MPC signing protocol begins.
4. Know the rerandomized presignature nonce `R' = delta * R` in advance.

This is the direct analog of the VDF precomputation issue: just as the VDF report found that an attacker could precompute the VDF solution for a predictable input `x`, here a malicious block producer can precompute the rerandomization scalar for a predictable entropy value.

---

### Impact Explanation

The [GS21] rerandomization scheme is the defense against the **ROS problem** / Wagner attack on threshold ECDSA. Its security proof requires that `delta` be unpredictable to any adversary who is choosing messages to sign. If `delta` is precomputable, the rerandomization no longer provides its intended security guarantee. An adversary who can predict `delta` for multiple signing sessions can potentially mount a Wagner-style attack: by submitting signing requests with carefully chosen payloads and knowing the rerandomization scalar in advance, they can accumulate information that allows forging a signature on a new message without the cooperation of the threshold participants. This maps to the allowed impact: **unauthorized threshold signature issuance without required participant authorization** (Critical), or at minimum a **production safety invariant break** in the signing flow (Medium), depending on the practical feasibility of the Wagner attack under the specific parameters.

---

### Likelihood Explanation

A NEAR block producer is not required to be an MPC participant. Any entity running a NEAR validator node can produce blocks and thus know the block's `random_seed` before it is published. The attacker does not need to collude with any MPC participant — they only need to be a NEAR block producer (or collude with one) and submit signing requests. The Wagner attack itself requires many signing queries, but the precomputation of `delta` is a necessary precondition that is already satisfied by any block producer. The likelihood is **Medium**: NEAR has a permissioned validator set, but the set is large enough that a motivated adversary could participate.

---

### Recommendation

Replace the block-derived entropy with a value that is not known to any single party before the signing protocol commits to the presignature. Options include:

1. **Node-contributed entropy**: Each MPC node contributes a fresh random value at the start of each signing session; the combined entropy is the hash of all contributions. This ensures no single party can predict the final value.
2. **Receipt-ID-based entropy**: Use the NEAR receipt ID (which is determined by the transaction hash and is not controllable by the block producer) combined with a node-local random value.
3. **Document the assumption**: At minimum, explicitly document that the security of presignature rerandomization relies on the NEAR block producer not being adversarial, and assess whether this assumption holds in the deployment threat model.

---

### Proof of Concept

**Step 1.** Attacker operates a NEAR validator node (block producer).

**Step 2.** Attacker submits a signing request with a chosen ECDSA payload to the MPC contract.

**Step 3.** When the attacker's node produces a block containing that request, it knows `random_seed = VRF_output` before publishing.

**Step 4.** Attacker computes `entropy = random_seed` and precomputes:
```
delta = HKDF-SHA3-256(salt, entropy; info = pk || tweak || msg_hash || big_r || participants)
```
using the known presignature `big_r` (which is stored in the presignature pool and is public within the MPC network once selected).

**Step 5.** Attacker knows `R' = delta * R` before the signing protocol runs, violating the unpredictability requirement of [GS21] and enabling precomputation-based attacks on the rerandomization.

```rust
    pub fn derive_randomness(&self) -> Result<Scalar, ProtocolError> {
        // create a string containing (pk, msg_hash, big_r, sorted(participants))
        let pk_encoded_point = self.pk.to_encoded_point(true);
        let encoded_pk: &[u8] = pk_encoded_point.as_bytes();
        let encoded_tweak: &[u8] = &<Secp256K1ScalarField as Field>::serialize(&self.tweak.value());
        let encoded_msg_hash: &[u8] = &self.msg_hash;
        let big_r_encoded_point = self.big_r.to_encoded_point(true);
        let encoded_big_r: &[u8] = big_r_encoded_point.as_bytes();

        // concatenate all the bytes
        let mut concatenation = Vec::new();
        // First byte is a counter (for the unlikely case the derived scalar is 0); the second byte is a fixed tag.
        concatenation.extend_from_slice(&[0u8, 1]);
        concatenation.extend_from_slice(encoded_pk);
        concatenation.extend_from_slice(encoded_tweak);
        concatenation.extend_from_slice(encoded_msg_hash);
        concatenation.extend_from_slice(encoded_big_r);
        // Append each ParticipantId
        for participant in self.participants.participants() {
            concatenation.extend_from_slice(&participant.bytes());
        }

        // initiate hkdf with the salt and with some `good' entropy
        let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);

        let mut delta = Scalar::ZERO;
        // If the randomness created is 0 then we want to generate a new randomness
        while bool::from(delta.is_zero()) {
            // Generate randomization out of HKDF(counter, entropy, pk, msg_hash, big_r, participants, )
            // where entropy is a public but unpredictable random string.
            // The counter depends on the number of times we enter into this loop
            let mut okm = [0u8; 32];

            hk.expand(&concatenation, &mut okm)
                .map_err(|_| ProtocolError::HashingError)?;

            // derive the randomness delta
            delta = Scalar::from_repr(okm.into()).unwrap_or(
                // if delta falls outside the field
                // probability is negligible: in the order of 1/2^224
                Scalar::ZERO,
            );
            // Increment the counter, the probability that this overflows is astronomically low
            let concatenation_0 = concatenation
                .first_mut()
                .ok_or(ProtocolError::InvalidIndex)?;
            *concatenation_0 += 1;
        }
        Ok(delta)
    }
```
