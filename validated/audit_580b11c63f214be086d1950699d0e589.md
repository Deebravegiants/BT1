### Title
Proposal Deposits Permanently Locked in Contract When Non-Winning Proposals Are Cleared — (`File: crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

---

### Summary

When a participant calls `propose_update`, they must attach a NEAR deposit sized to cover storage for the proposal. When any proposal reaches threshold votes and `do_update` executes, it clears **all** proposals and votes — including those from other proposers — but never refunds the deposits those other proposers paid. There is no withdrawal function for proposal deposits. The NEAR tokens are permanently locked in the contract.

---

### Finding Description

`propose_update` in `lib.rs` enforces a deposit equal to `ProposedUpdates::required_deposit(&update)`: [1](#0-0) 

Only the **excess** above `required` is refunded to the proposer. The exact `required` amount stays in the contract: [2](#0-1) 

The deposit is computed as `storage_byte_cost × bytes_used(update)`, where `bytes_used` accounts for the contract binary or config blob plus a fixed overhead for 128 participant votes: [3](#0-2) 

When `do_update` is triggered by a successful `vote_update`, it removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every other pending proposal: [4](#0-3) 

The deposits paid by all non-winning proposers are never returned. There is no `withdraw_proposal` or equivalent function anywhere in the contract. `remove_update_vote` only removes a vote record, not the proposal entry or its deposit: [5](#0-4) 

---

### Impact Explanation

Every participant who proposes an update that is not the one ultimately executed permanently loses their deposit. For a contract-code proposal, `bytes_used` includes the full WASM binary (up to ~1.5 MB) plus a 128-account overhead, making the required deposit on the order of **tens of NEAR** per proposal. When multiple participants propose competing updates (a normal governance scenario), all but one lose their deposits with no recourse. The NEAR tokens accumulate in the contract balance with no withdrawal path, constituting a permanent, irreversible loss of funds for participants.

This matches the allowed Medium impact: *"Balance… manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

This triggers under normal protocol operation. Any governance round where more than one participant proposes a different update (e.g., one proposes a code upgrade, another proposes a config change) results in the non-winning proposers losing their deposits. The sandbox tests already demonstrate multiple concurrent proposals being proposed and then cleared: [6](#0-5) 

No adversarial action is required; the loss occurs as a side-effect of the normal threshold-vote lifecycle.

---

### Recommendation

Track the depositor and deposit amount per `UpdateEntry`. When `do_update` clears non-winning entries, issue a `Promise::new(depositor).transfer(deposit)` for each cleared entry. Alternatively, add a `withdraw_proposal(id: UpdateId)` function that lets a proposer retract their proposal and recover their deposit before a vote concludes.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1.5 MB WASM blob, attaching ~30 NEAR (the required deposit).
2. Participant B calls `propose_update` with a config change, attaching ~0.1 NEAR.
3. Threshold participants vote for B's proposal via `vote_update`.
4. `do_update` executes: it removes B's entry, then calls `self.entries.clear()` — deleting A's entry — and `self.vote_by_participant.clear()`.
5. A's ~30 NEAR deposit remains in the contract balance. No function exists to recover it. The funds are permanently locked. [7](#0-6)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}

    for i in 0..PROPOSAL_COUNT {
        let execution = mpc_signer_accounts[i % mpc_signer_accounts.len()]
            .call(contract.id(), method_names::PROPOSE_UPDATE)
            .args_borsh(current_contract_proposal())

            .deposit(CURRENT_CONTRACT_DEPLOY_DEPOSIT)
            .transact()
            .await
            .unwrap();

        assert!(
            execution.is_success(),
            "failed to propose update [i={i}]; {execution:#?}"

        let proposal_id = execution.json().expect("unable to convert into UpdateId");

    }

    // Vote for the last proposal
    vote_update_till_completion(&contract, &mpc_signer_accounts, proposals.last().unwrap()).await;
