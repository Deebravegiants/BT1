### Title
Rosetta RPC Silently Drops NEP-141 `ft_mint`/`ft_burn` Events, Corrupting Block-Level Fungible Token Operation Data - (File: chain/rosetta-rpc/src/adapters/nep141.rs)

### Summary

The Rosetta RPC's NEP-141 event model only recognizes `ft_transfer` events. The `ft_mint` and `ft_burn` event kinds defined by the NEP-141 standard are silently discarded during deserialization. Any unprivileged user who deploys a standards-compliant NEP-141 token contract and calls a minting or burning function causes the Rosetta `/block` endpoint to return a block response with those operations entirely absent, corrupting the RPC result that exchanges and wallets rely on for balance reconciliation.

### Finding Description

`Nep141EventKind` is defined with a single variant:

```rust
pub(crate) enum Nep141EventKind {
    FtTransfer(Vec<FtTransferData>),
}
``` [1](#0-0) 

The NEP-141 standard specifies three distinct events: `ft_transfer`, `ft_mint`, and `ft_burn`. When `extract_events` parses execution outcome logs, it attempts to deserialize every `EVENT_JSON:` log as `Nep141Event`. Because `Nep141EventKind` has no `FtMint` or `FtBurn` variant, `serde_json` fails to match those event tags and the error arm silently returns `None`:

```rust
match serde_json::from_str::<'_, crate::models::Nep141Event>(log[prefix.len()..].trim())
{
    Ok(result) => Some(result),
    Err(_err) => None,   // ft_mint / ft_burn silently dropped here
}
``` [2](#0-1) 

`collect_nep141_events` then feeds only the surviving `FtTransfer` events into `compose_rosetta_nep141_events`, which builds the `FungibleTokenEvent` list that is ultimately embedded in the Rosetta `/block` response via `convert_fungible_token_balance_change_to_operations`. [3](#0-2) [4](#0-3) 

The result is that any block containing `ft_mint` or `ft_burn` log entries has those operations entirely absent from the Rosetta block response, while the underlying on-chain state (actual token balances) is correct.

### Impact Explanation

The corrupted value is the RPC result returned by the Rosetta `/block` endpoint. Exchanges and wallets that use the Rosetta Data API to reconstruct fungible-token balance histories will observe no credit for minting operations and no debit for burning operations. This produces a persistent, growing divergence between the Rosetta-derived balance and the true on-chain balance for any NEP-141 token that correctly emits `ft_mint`/`ft_burn` events. An exchange crediting deposits based on Rosetta block data would fail to credit users whose tokens were minted directly to their accounts.

### Likelihood Explanation

Any unprivileged user can deploy a standards-compliant NEP-141 contract (e.g., the reference implementation) and call its mint function. The resulting `EVENT_JSON:{"standard":"nep141","version":"1.0.0","event":"ft_mint","data":[...]}` log is emitted on every successful mint call. Every such block processed by a Rosetta node silently omits those operations. The trigger requires no special privileges and is reachable through ordinary public RPC transaction submission.

### Recommendation

Add `FtMint` and `FtBurn` variants to `Nep141EventKind` in `chain/rosetta-rpc/src/models.rs`, with corresponding data structs (`owner_id`, `amount`, `memo`). Extend `compose_rosetta_nep141_events` to handle these variants, emitting `FungibleTokenEvent` entries with `cause = "MINT"` / `cause = "BURN"` and appropriate `delta_amount` signs (positive for mint on the recipient, negative for burn on the owner). This mirrors the fix applied to the BAT crowdsale contract: use a dedicated, semantically correct event rather than overloading or ignoring the transfer event.

### Proof of Concept

1. Deploy a NEP-141 token contract that emits the standard `ft_mint` event on token creation.
2. Call the mint function from any account; the contract logs:
   `EVENT_JSON:{"standard":"nep141","version":"1.0.0","event":"ft_mint","data":[{"owner_id":"alice.near","amount":"1000"}]}`
3. Query the Rosetta `/block` endpoint for the block containing that receipt.
4. Observe that the returned `transactions` array contains no `FungibleTokenEvent` operation for the mint — the `delta_amount` credit to `alice.near` is absent — even though `alice.near`'s on-chain NEP-141 balance increased by 1000.
5. Repeat with `ft_burn`; the debit is likewise absent from the Rosetta response.

    receipt_execution_outcomes: &Vec<ExecutionOutcomeWithIdView>,
    block_header: &near_primitives::views::BlockHeaderView,
    currencies: &Option<Vec<Currency>>,
) -> crate::errors::Result<Vec<FungibleTokenEvent>> {
    let mut res = Vec::new();
    for outcome in receipt_execution_outcomes {
        let events = extract_events(outcome);
        for event in events {
            res.extend(compose_rosetta_nep141_events(&event, outcome, block_header, currencies)?);
        }
    }
    Ok(res)
}
