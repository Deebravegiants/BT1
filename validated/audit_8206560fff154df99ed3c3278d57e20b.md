Audit Report

## Title
`RSETH._transfer` Override Omits `whenNotPaused` Guard, Allowing rsETH Transfers During Emergency Pause â€” (File: `contracts/RSETH.sol`)

## Summary
`RSETH` inherits `ERC20Upgradeable` and `PausableUpgradeable` separately, providing no automatic pause hook on the ERC-20 transfer path. The `_transfer` override at L287â€“291 enforces the per-address block list but omits `whenNotPaused`, while `mint` (L235) and `burnFrom` (L245) both correctly gate on the pause state. Any rsETH holder can call `transfer`/`transferFrom` freely while the contract is paused, undermining the emergency stop guarantee.

## Finding Description
`RSETH` does not inherit `ERC20PausableUpgradeable`, so no automatic pause hook is wired into `_transfer`. The override at L287â€“291 only calls `_enforceNotBlocked` for both parties before delegating to `super._transfer`:

```solidity
// contracts/RSETH.sol L287-291
function _transfer(address from, address to, uint256 amount) internal override {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);   // no whenNotPaused check
}
```

`mint` and `burnFrom` both carry `whenNotPaused`:

```solidity
// L235
function mint(...) external onlyRole(LRTConstants.MINTER_ROLE) whenNotPaused ...

// L245
function burnFrom(...) external onlyRole(LRTConstants.BURNER_ROLE) whenNotPaused { ... }
```

There is no `_beforeTokenTransfer` override to compensate. The result is that `pause()` halts minting and burning but leaves the full ERC-20 transfer surface open. The `recoverFrozenFunds` mechanism (L206â€“219) depends on a blocked address retaining its balance; a holder can transfer their rsETH to a fresh address during the pause window before `blockUserTransfers` is applied, leaving the original address with zero balance and making recovery impossible.

## Impact Explanation
**Low â€” Contract fails to deliver promised returns, but doesn't lose value.**

The pause mechanism's security guarantee â€” halt all token movements during an incident â€” is not upheld for the `transfer`/`transferFrom` path. No direct fund theft or permanent freeze results, but the protocol's incident-response tooling (`recoverFrozenFunds`) can be rendered ineffective during the exact window it is most needed.

## Likelihood Explanation
Any rsETH holder can trigger this with a standard ERC-20 `transfer` call. No special role, no oracle manipulation, no external dependency. The window is the entire duration of any pause event. The front-run of `blockUserTransfers` requires only that the holder monitors the mempool and submits a transfer before the block is applied â€” a trivial operation for any motivated actor.

## Recommendation
Add `whenNotPaused` to the `_transfer` override so pause enforcement is uniform across all token-movement paths:

```solidity
function _transfer(address from, address to, uint256 amount) internal override whenNotPaused {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
```

Alternatively, override `_beforeTokenTransfer` with `require(!paused(), "Pausable: paused")`, which is the standard OpenZeppelin pattern used by `ERC20PausableUpgradeable`.

## Proof of Concept

```solidity
function testTransferBypassesPause() public {
    // Setup: mint rsETH to user
    vm.prank(minter);
    rsETH.mint(user, 1e18);

    // Admin pauses the contract
    vm.prank(pauser);
    rsETH.pause();
    assertTrue(rsETH.paused());

    // mint is correctly blocked
    vm.expectRevert("Pausable: paused");
    vm.prank(minter);
    rsETH.mint(user, 1e18);

    // burnFrom is correctly blocked
    vm.expectRevert("Pausable: paused");
    vm.prank(burner);
    rsETH.burnFrom(user, 1e18);

    // transfer is NOT blocked â€” succeeds while paused
    vm.prank(user);
    rsETH.transfer(user2, 1e18);           // should revert, but doesn't
    assertEq(rsETH.balanceOf(user2), 1e18); // funds moved during pause
}
```
