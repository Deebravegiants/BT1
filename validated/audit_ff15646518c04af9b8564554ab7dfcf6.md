Audit Report

## Title
Mistakenly Sent ETH Is Silently Ignored and Drainable by Any Caller When Swapping or Adding Liquidity With Non-WETH ERC20 Tokens - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`PeripheryPayments.pay()` only consumes native ETH from `address(this).balance` when `token == WETH`. For every other ERC20, the function falls through to `safeTransferFrom` and leaves any attached ETH stranded in the router. Because all external entry points on `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` are `payable`, a caller who attaches `msg.value` to a non-WETH ERC20 swap or liquidity call loses that ETH to the first address that calls the public `refundETH()`, which unconditionally forwards the full contract balance to `msg.sender`.

## Finding Description
`PeripheryPayments.pay()` (lines 69–88) has three branches. The `token == WETH` branch (lines 73–84) is the only one that touches `address(this).balance`; the non-WETH ERC20 branch (lines 85–87) calls `safeTransferFrom` and ignores any native balance entirely.

The `receive()` fallback (lines 32–34) guards against direct ETH transfers by reverting unless `msg.sender == WETH`, but this guard is irrelevant when ETH is attached to a named function call — in that case the EVM routes execution to the called function, not to `receive()`. Every external entry point is declared `payable`:

- `MetricOmmSimpleRouter`: `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`
- `MetricOmmPoolLiquidityAdder`: both `addLiquidityExactShares` overloads, both `addLiquidityWeighted` overloads, `multicall`

So a call such as `exactInputSingle{value: 1 ether}(params)` where `params.tokenIn = USDC` succeeds: USDC is pulled via `safeTransferFrom` inside `_justPayCallback` → `pay(USDC, victim, pool, amount)`, while 1 ETH remains in `address(router).balance`. `refundETH()` (lines 58–63) is `external`, performs no access control, and sends the entire balance to `msg.sender`. Any address — including an MEV bot watching the mempool — can call it and receive the stranded ETH before the victim does.

The same path applies to `addLiquidityExactShares{value: X}(...)` and `addLiquidityWeighted{value: X}(...)` when both pool tokens are non-WETH ERC20, because `metricOmmModifyLiquidityCallback` calls `pay(token0, ...)` and `pay(token1, ...)` (lines 172–177), neither of which touches ETH when the tokens are not WETH.

Existing guards are insufficient: the `receive()` guard does not apply to named-function calls with `msg.value`, and there is no `msg.value == 0` check at any entry point for non-WETH paths.

## Impact Explanation
A user who attaches ETH to a non-WETH ERC20 swap or liquidity call suffers a direct, immediate loss of principal equal to `msg.value`. The loss is permanent once any third party calls `refundETH()`. No privileged role is required. This constitutes a direct loss of user principal meeting the Sherlock Medium/High threshold.

## Likelihood Explanation
The `payable` modifier on all entry points is intentional (to support native-ETH-as-WETH flows), making it natural for users, frontend bugs, or integrating contracts to attach ETH. The griefing step — calling `refundETH()` — requires no special access and is executable by any MEV bot monitoring the mempool. The attack is repeatable and requires no setup beyond observing a transaction with `msg.value > 0` on a non-WETH path.

## Recommendation
In the non-WETH ERC20 branch of `pay()`, revert if the contract holds unexpected native ETH:

```solidity
} else {
    if (address(this).balance > 0) revert UnexpectedETH();
    IERC20(token).safeTransferFrom(payer, recipient, value);
}
```

Alternatively, add a `require(msg.value == 0)` guard at the top of each external entry point that does not explicitly support native ETH input (i.e., those whose `tokenIn`/pool tokens are not WETH), or validate at the start of `pay()` that `msg.value` is zero when `token != WETH`.

## Proof of Concept
1. Pool is `token0 = USDC`, `token1 = DAI` (neither is WETH).
2. Victim calls:
   ```solidity
   router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
       tokenIn: address(USDC), tokenOut: address(DAI), ...
   }));
   ```
3. Inside the swap callback, `_justPayCallback` calls `pay(USDC, victim, pool, amount)` (lines 192–199).
4. `USDC != WETH` → branch at line 85 executes `safeTransferFrom(victim, pool, amount)`; 1 ETH remains in `address(router).balance`.
5. Attacker calls `router.refundETH()` → `_transferETH(msg.sender, 1 ether)` executes (lines 58–63); attacker receives 1 ETH.
6. Victim's USDC was spent correctly; their 1 ETH is permanently lost.

The same path applies to `addLiquidityExactShares{value: X}(...)` when both pool tokens are non-WETH ERC20, via `metricOmmModifyLiquidityCallback` → `pay(token0, ...)` / `pay(token1, ...)` (lines 172–177).
