Audit Report

## Title
`notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` Unconditionally Zeroed Without Corresponding Transfer, Permanently Losing Accumulated Notional Fees - (File: `metric-core/contracts/MetricOmmPool.sol`)

## Summary

In `MetricOmmPool.collectFees`, the notional fee accumulators `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled` are unconditionally reset to zero at lines 429–430 regardless of whether any tokens were actually transferred. When accumulated scaled fees are below `TOKEN_X_SCALE_MULTIPLIER` (e.g., `10^12` for USDC), floor division yields zero external units, no transfer occurs, but the accumulator is still cleared. The orphaned scaled units remain in the pool's ERC-20 balance untracked, and in pools with `spreadSumE6 == 0` they can never be recovered.

## Finding Description

`collectFees` (lines 411–430) converts scaled fee amounts to external units using `Math.Rounding.Floor` via `deltasScaledToExternal` (lines 626–627: `deltaAmount0 = scaledDeltaAmount0 / TOKEN_0_SCALE_MULTIPLIER`). Transfers are gated on `> 0` (lines 416–427), but the accumulators are cleared unconditionally (lines 429–430):

```solidity
notionalFeeToken0Scaled = 0;   // always cleared
notionalFeeToken1Scaled = 0;   // always cleared
```

For a USDC pool, `TOKEN_0_SCALE_MULTIPLIER = 10^12`. Any `notionalFeeToken0Scaled < 10^12` floors to 0 external units — no transfer, but the accumulator is zeroed. On the next `collectFees` call, the surplus formula (lines 385–388) is:

```solidity
surplus0Scaled = balance0() * TOKEN_0_SCALE_MULTIPLIER
               - uint256(binTotals.scaledToken0)
               - notionalFee0AmountScaled;  // now 0
```

The orphaned units appear in `surplus0Scaled`, but when `spreadSumE6 == 0` (lines 391–395 set all spread fee shares to 0), neither admin nor protocol receives them. The tokens are permanently stuck. The trigger is fully unprivileged: `collectPoolFees` (factory line 379) has no access control beyond `nonReentrant`.

## Impact Explanation

Direct loss of protocol and admin fee revenue. Accumulated notional fees paid by traders are permanently destroyed — neither the protocol nor the admin receives them. In notional-only fee pools (`protocolSpreadFeeE6 = 0`, `adminSpreadFeeE6 = 0`), orphaned tokens can never be recovered. An adversary can call `collectPoolFees` repeatedly after small swaps to drain the notional fee accumulator in dust increments, causing unbounded cumulative loss. Maximum loss per call is up to `TOKEN_0_SCALE_MULTIPLIER − 1` scaled units per token per recipient leg (~1 USDC per call for a USDC pool). This meets the Sherlock threshold for direct loss of protocol fees.

## Likelihood Explanation

`collectPoolFees` is callable by any address with no restriction. Pools pairing 18-decimal tokens with USDC/USDT (6 decimals) are the primary deployment target and are explicitly in scope. Small swaps generating sub-unit notional fees are routine in any active pool. No privileged access or special setup is required. The attack is repeatable indefinitely.

## Recommendation

Only subtract from `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` the scaled equivalent of what was actually paid out, rather than unconditionally zeroing:

```diff
-     notionalFeeToken0Scaled = 0;
-     notionalFeeToken1Scaled = 0;
+     uint256 paid0Scaled = (totalFee0ToAdmin + totalFee0ToProtocol) * TOKEN_0_SCALE_MULTIPLIER;
+     uint256 paid1Scaled = (totalFee1ToAdmin + totalFee1ToProtocol) * TOKEN_1_SCALE_MULTIPLIER;
+     notionalFeeToken0Scaled = uint128(notionalFee0AmountScaled > paid0Scaled
+         ? notionalFee0AmountScaled - paid0Scaled : 0);
+     notionalFeeToken1Scaled = uint128(notionalFee1AmountScaled > paid1Scaled
+         ? notionalFee1AmountScaled - paid1Scaled : 0);
```

Alternatively, enforce a minimum accumulation threshold before allowing `collectFees` to proceed.

## Proof of Concept

1. Deploy a pool with USDC (6 decimals) as token0 and an 18-decimal token as token1. Set `protocolNotionalFeeE8 = 1_000_000` (1%), `protocolSpreadFeeE6 = 0`, `adminSpreadFeeE6 = 0`.
2. `TOKEN_0_SCALE_MULTIPLIER = 10^12`.
3. Execute a swap that produces `notionalFeeToken0Scaled = 5 * 10^11` (< `10^12`), requiring ~0.5 USDC output — a realistic small swap.
4. Any address calls `factory.collectPoolFees(pool)` (no access control at factory line 379).
5. Inside `collectFees`: `totalFee0ToProtocol = (5 * 10^11) / 10^12 = 0` (floor). No transfer occurs.
6. `notionalFeeToken0Scaled` is set to 0 (line 429). The 5 * 10^11 scaled units remain in the pool's ERC-20 balance but are now untracked.
7. On the next `collectFees` call, `surplus0Scaled` includes the orphaned units, but since `spreadSumE6 = 0`, `spreadFee0ToProtocolScaled = 0` and `spreadFee0ToAdminScaled = 0`. Tokens are never distributed.
8. Repeat steps 3–7 to accumulate unbounded losses.
