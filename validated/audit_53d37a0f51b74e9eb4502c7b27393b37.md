Audit Report

## Title
Wrong Variable Used in ETH Deposit Limit Check Allows Deposit Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch comparison, checking only `totalAssetDeposits > limit` instead of `totalAssetDeposits + amount > limit`. Any unprivileged caller can invoke `depositETH` and deposit an amount that pushes total ETH holdings above the governance-set cap, because the guard never accounts for the new deposit. The contract fails to enforce its promised deposit ceiling.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit` discards the `amount` parameter:

```solidity
// L678-681
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
```

`_beforeDeposit` (L661–663) calls this function and reverts only when it returns `true`. Because the ETH branch never adds `amount`, the check only fires if the pre-existing total already exceeds the cap — the incoming deposit is never counted. `depositETH` (L76–93) is `external payable` with no role restriction, so any caller can reach this path. The only guard between the caller and the rsETH mint is `_beforeDeposit`, which is insufficient for ETH deposits.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**
The deposit cap is a governance-controlled risk parameter that the protocol explicitly promises to enforce. The ETH branch silently fails to enforce it, allowing total ETH deposits to exceed the cap by an arbitrary amount in a single transaction. No direct theft or freezing of funds occurs from the bypass alone; the contract simply does not honour its stated ceiling.

## Likelihood Explanation
`depositETH` is public and payable with no role restriction. Any depositor who observes that `getTotalAssetDeposits(ETH_TOKEN)` is below the limit can immediately exploit the gap. No special conditions, timing windows, or privileged access are required. The condition is trivially observable on-chain and repeatable.

## Recommendation
Add `amount` to the ETH branch to match the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Governance sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999 ether > 1000 ether` → `false`.
5. `_beforeDeposit` does not revert; `10_000 ether` worth of rsETH is minted.
6. Protocol now holds `10_999 ether`, far above the governance-set cap, with no revert or event indicating the breach. [1](#0-0) [2](#0-1) [3](#0-2)
