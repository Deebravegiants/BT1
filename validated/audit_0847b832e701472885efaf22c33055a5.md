Audit Report

## Title
Missing `priceLimitX64 != 0` Guard in Initial Price-Limit Check Silently Blocks All `!zeroForOne` Swaps With No Price Limit — (File: `metric-core/contracts/MetricOmmPool.sol`)

## Summary
In `_swapToken1ForToken0SpecifiedInput` and `_swapToken1ForToken0SpecifiedOutput`, the initial early-exit check `params.priceLimitX64 <= initialPriceX64` is missing the `priceLimitX64 != 0` guard that the inner swap loop explicitly applies. When a caller passes `priceLimitX64 == 0` (the "no upper price limit" sentinel), the condition `0 <= initialPriceX64` is unconditionally true because `initialPriceX64` is always positive, causing both functions to immediately return zero amounts. The swap transaction succeeds without reverting but delivers nothing to the recipient and debits nothing from the caller.

## Finding Description

The inner loop of `_swapToken1ForToken0SpecifiedInput` correctly guards against `priceLimitX64 == 0`:

```solidity
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= upperPriceX64) {
    break;
}
``` [1](#0-0) 

But the initial guard in both `!zeroForOne` functions lacks this protection:

```solidity
// _swapToken1ForToken0SpecifiedOutput (L888)
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0, 0);
}
``` [2](#0-1) 

```solidity
// _swapToken1ForToken0SpecifiedInput (L970)
if (params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}
``` [3](#0-2) 

`initialPriceX64` is derived from `SwapMath.calculatePriceAtBinPosition` applied to bin bounds computed from `midPriceX64`. The oracle enforces `bid > 0` and `bid < ask`, so `midPriceX64 > 0` and therefore `initialPriceX64 > 0` always holds. [4](#0-3) 

When `priceLimitX64 == 0`, the comparison `0 <= initialPriceX64` is always true, so both functions return zero immediately before any bin logic executes. The `swap` entry point imposes no non-zero requirement on `priceLimitX64`. [5](#0-4) 

Back in `swap`, `amount0Delta = 0` and `amount1Delta = 0`. The `!zeroForOne` branch checks `if (amount0Delta < 0)` — false, so no token0 is sent to the recipient. The callback is invoked with `(0, 0)`. The `IncorrectDelta` guard `if (amount1Delta > 0 && ...)` evaluates false, so no revert occurs. The transaction completes silently with zero output. [6](#0-5) 

Note: the symmetric `zeroForOne` functions (`_swapToken0ForToken1SpecifiedInput`, `_swapToken0ForToken1SpecifiedOutput`) use `priceLimitX64 >= initialPriceX64` as their guard. When `priceLimitX64 == 0`, this evaluates `0 >= initialPriceX64` which is always false (since `initialPriceX64 > 0`), so `zeroForOne` swaps are unaffected. [7](#0-6) 

## Impact Explanation
Every `!zeroForOne` swap (token1 → token0) submitted with `priceLimitX64 == 0` silently returns `(0, 0)`. The recipient receives zero token0, the caller pays zero token1, and pool state is unchanged. This constitutes broken core pool swap functionality: the entire `!zeroForOne` swap path is unusable for any router or integrator following the convention of passing `0` for "no price limit," including the periphery `MetricOmmSimpleRouter`. The impact is loss of core swap functionality for a primary pool operation.

## Likelihood Explanation
Medium-High. The `swap` function accepts `priceLimitX64` as a raw `uint128` with no non-zero validation. The inner loop's explicit `!= 0` guard is strong evidence the protocol intended `0` to mean "no limit." Any router built on this convention will silently fail on all `!zeroForOne` swaps. The condition is triggered by a standard, unprivileged `swap` call with no special setup required.

## Recommendation
Add the `priceLimitX64 != 0` guard to the initial check in both functions, matching the inner loop pattern:

```solidity
// _swapToken1ForToken0SpecifiedOutput
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0, 0);
}

// _swapToken1ForToken0SpecifiedInput
if (params.priceLimitX64 != 0 && params.priceLimitX64 <= initialPriceX64) {
    return (0, 0, 0);
}
```

## Proof of Concept
1. Deploy a pool with token0/token1 and add liquidity across bins.
2. Call `pool.swap(recipient, false, 1e18, 0, callbackData, "")` — `zeroForOne = false`, `priceLimitX64 = 0`.
3. `_executeSwap` routes to `_swapToken1ForToken0SpecifiedInput` with `params.priceLimitX64 = 0`.
4. `_getInitialStateForSwap` computes `initialPriceX64 > 0` from current bin bounds and `midPriceX64 > 0`.
5. `0 <= initialPriceX64` → `true` → function returns `(0, 0, 0)` immediately.
6. Back in `_executeSwap`: `amount0DeltaScaled = 0`, `amount1DeltaScaled = 0`.
7. `deltasScaledToExternal(0, 0)` → `amount0Delta = 0`, `amount1Delta = 0`.
8. In `swap`: `if (amount0Delta < 0)` → false, no token0 sent to recipient.
9. Callback invoked with `(0, 0)` — caller pays nothing.
10. `if (amount1Delta > 0 && ...)` → false, no `IncorrectDelta` revert.
11. `swap` returns `(0, 0)`. Recipient receives 0 token0. Swap silently fails.
