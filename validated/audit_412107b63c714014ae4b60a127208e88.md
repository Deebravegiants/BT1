### Title
Sub-Transaction ExUnits Budget Completely Unchecked Against `maxTxExUnits` and `maxBlockExUnits` - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

In the Dijkstra era, the `maxTxExUnits` and `maxBlockExUnits` protocol-parameter limits are enforced only against the top-level transaction's redeemers. Sub-transactions (`Tx SubTx era`) carry their own independent redeemer maps with their own `ExUnits` declarations, and no validation rule ever checks those sub-transaction ExUnits against either limit. An unprivileged transaction submitter can therefore declare arbitrarily large ExUnits budgets inside sub-transactions, bypassing the intended per-transaction and per-block execution-unit caps.

---

### Finding Description

**Vulnerability class:** Resource-limit bypass (unbounded sub-budget).

**Root cause — `totExUnits` only covers the top-level transaction:**

`totExUnits` is defined as:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
``` [1](#0-0) 

For a `Tx TopTx era`, `witsTxL . rdmrsTxWitsL` accesses only the top-level transaction's witness set. Sub-transactions (`Tx SubTx era`) are stored in `dtbrSubTransactions :: OMap TxId (Tx SubTx era)` inside the top-level body and carry their own independent witnesses and redeemers. [2](#0-1) 

**Root cause — the top-level UTXO rule only checks `totExUnits tx`:**

`dijkstraUtxoTransition` calls:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [3](#0-2) 

`validateExUnitsTooBigUTxO` computes `totalExUnits = totExUnits tx`, which, as shown above, sums only the top-level redeemers. [4](#0-3) 

**Root cause — `dijkstraSubUtxoTransition` has no ExUnits check:**

The sub-transaction validation rule `dijkstraSubUtxoTransition` performs output-size, input-set, network-ID, and collateral checks, but contains no call to `validateExUnitsTooBigUTxO` or any equivalent. [5](#0-4) 

The `DijkstraSubUtxoPredFailure` type has no `ExUnitsTooBigUTxO` constructor, and the conversion function `dijkstraUtxoToDijkstraSubUtxoPredFailure` explicitly marks the case as impossible:

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [6](#0-5) 

**Root cause — the block-level check is also bypassed:**

`validateExUnits` in the BBODY rule uses `foldMap totExUnits txs`, which again only accumulates top-level redeemers across all transactions in the block:

```haskell
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
  in pointWiseExUnits (<=) txTotal ppMax
``` [7](#0-6) 

Sub-transaction ExUnits are therefore invisible to both the per-transaction and per-block enforcement paths.

---

### Impact Explanation

The `maxTxExUnits` and `maxBlockExUnits` parameters exist to bound the CPU and memory consumed by the Plutus interpreter during block validation, preventing a single transaction or block from monopolising validator resources. By embedding arbitrarily large ExUnits declarations inside sub-transactions, an attacker can cause the Plutus interpreter to run with an uncapped budget for sub-transaction scripts, exceeding the intended validation limits for both the transaction and the block. This matches the **Medium** allowed impact: *"Attacker-controlled transactions… exceed intended validation limits."*

---

### Likelihood Explanation

The Dijkstra era is protocol version 12. Any unprivileged transaction submitter who can construct a valid Dijkstra-era top-level transaction with embedded sub-transactions can exploit this. No special keys, governance roles, or majority are required. The attacker simply sets the top-level redeemers to `ExUnits 0 0` (passing the `maxTxExUnits` check) and places large ExUnits values in sub-transaction redeemers.

---

### Recommendation

1. Extend `totExUnits` (or introduce a new `batchTotExUnits`) to recursively sum ExUnits across all sub-transactions:

   ```haskell
   batchTotExUnits tx =
     totExUnits tx
       <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
   ```

2. Replace the call in `dijkstraUtxoTransition` with `batchTotExUnits`:

   ```haskell
   runTest $ validateExUnitsTooBigUTxO' pp (batchTotExUnits tx) maxTxExUnits
   ```

3. Ensure the BBODY-level `validateExUnits` also uses `batchTotExUnits` so that `maxBlockExUnits` is correctly enforced across all sub-transactions in a block.

4. Optionally add a per-sub-transaction ExUnits check in `dijkstraSubUtxoTransition` as a defence-in-depth measure.

---

### Proof of Concept

1. Construct a Dijkstra-era top-level transaction `txTop` with:
   - One or more sub-transactions `txSub`, each containing a Plutus script redeemer with `ExUnits { exUnitsMem = maxBound, exUnitsSteps = maxBound }`.
   - The top-level redeemer map is empty (or contains only `ExUnits 0 0` entries).

2. Submit `txTop`. The ledger evaluates:
   - `totExUnits txTop = ExUnits 0 0 ≤ maxTxExUnits` → **passes**.
   - `dijkstraSubUtxoTransition` for each `txSub` → **no ExUnits check, passes**.
   - Block-level `foldMap totExUnits [txTop] = ExUnits 0 0 ≤ maxBlockExUnits` → **passes**.

3. The Plutus interpreter executes sub-transaction scripts with an uncapped budget, consuming resources far beyond `maxTxExUnits` and `maxBlockExUnits`. [3](#0-2) [1](#0-0) [5](#0-4) [6](#0-5) [7](#0-6)

validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) $
    ExUnitsTooBigUTxO Mismatch {mismatchSupplied = totalExUnits, mismatchExpected = maxTxExUnits}
  where
