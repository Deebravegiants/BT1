### Title
Missing Cryptographic Output Verification in `respond_ckd` for Legacy `AppPublicKey` Variant Allows Single Byzantine Participant to Forge CKD Output - (File: crates/contract/src/lib.rs)

### Summary

The `respond_ckd` contract method performs no cryptographic verification of the CKD response when the request uses the legacy `CKDAppPublicKey::AppPublicKey` variant. A single Byzantine attested participant (strictly below the signing threshold) can submit an arbitrary forged `CKDResponse` for any pending legacy CKD request, and the contract will accept and deliver it to the user without any check. This is the direct analog of the `ecrecover` null-return bug: just as the Ethereum bridge accepted invalid signatures because it skipped a return-value check, the NEAR MPC contract accepts invalid CKD output because it skips the output-validity check for the legacy key variant.

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` type:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty arm, no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKeyPV` arm calls `ckd_output_check`, which verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, cryptographically binding the response to the MPC master key and the user's app public key. [2](#0-1) 

The legacy `AppPublicKey` arm is completely empty. No pairing check, no point validation, no binding to the MPC public key. Any `big_y` and `big_c` values pass through.

By contrast, `respond` (for threshold signatures) always verifies the signature against the derived MPC public key before resolving the yield: [3](#0-2) 

The `respond_ckd` function requires only that the caller is an attested participant — a single node, not a threshold quorum: [4](#0-3) 

The first node to call `respond_ckd` for a given request resolves the yield and delivers the response to the user. There is no quorum or aggregation step.

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request in the contract's `pending_ckd_requests` map.
2. Craft a `CKDResponse` with attacker-chosen `big_y` and `big_c` values. Specifically, the attacker can pick a scalar `k` and set `big_y = k · pk1` (where `pk1` is the user's G1 app public key) and `big_c = k · G1`. This makes the user decrypt a key that the attacker knows (`k`), because the CKD decryption computes `big_y - k_user · big_c` — but since `k_user` is the user's private scalar and the attacker set `big_y = k · pk1 = k · k_user · G1` and `big_c = k · G1`, the decrypted value is `k · k_user · G1 - k_user · k · G1 = 0`, or more precisely the attacker can engineer any output they want.
3. Call `respond_ckd` before the honest nodes do. The contract accepts the forged response unconditionally and delivers it to the user via `resolve_yields_for`.

The user receives forged CKD output — key material that does not correspond to the MPC network's actual threshold key derivation. This constitutes unauthorized confidential key derivation output without the required threshold participant authorization, matching the Critical impact class: *"Unauthorized … confidential key derivation output without the required participant authorization."* [5](#0-4) 

### Likelihood Explanation

The `AppPublicKey` variant is the legacy default used by existing integrations (described as "privately verifiable, legacy" in the contract README). Any single attested participant — one that has submitted a valid TEE attestation — can exploit this. The attacker does not need to collude with other nodes, does not need to break any cryptography, and does not need privileged access beyond being an active participant. The attack requires only racing the honest nodes to submit `respond_ckd` first, which is straightforward since the attacker controls their own NEAR transaction submission timing.

### Recommendation

**Short term:** Add a cryptographic output check for the `AppPublicKey` variant. Since `AppPublicKey` provides only `pk1` (a G1 point) without `pk2` (a G2 point), the full pairing check used by `AppPublicKeyPV` cannot be applied. The contract should either:
- Reject `AppPublicKey` CKD requests entirely (deprecate the legacy variant), or
- Require the responding node to include a proof of correct derivation that can be verified on-chain with only `pk1`.

**Long term:** Migrate all CKD users to `AppPublicKeyPV`, which provides the `pk2` G2 point needed for the on-chain pairing check. Once migration is complete, remove the `AppPublicKey` arm entirely so the empty match arm cannot recur.

### Proof of Concept

1. User submits `request_app_private_key` with `app_public_key: AppPublicKey(pk1)` where `pk1 = k_user · G1`.
2. Attacker (a single attested participant) observes the pending request.
3. Attacker picks scalar `r` and calls `respond_ckd` with:
   - `big_y = r · G1` (arbitrary G1 point)
   - `big_c = r · G1` (arbitrary G1 point)
4. The contract executes the empty `AppPublicKey` match arm, skips all verification, and calls `resolve_yields_for` — delivering the forged response to the user.
5. The user receives `(big_y, big_c)` that bear no relation to the MPC master key or the actual threshold derivation. The attacker has successfully injected unauthorized CKD output without any threshold participation.

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` confirms this path: it submits `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (clearly invalid BLS points) and the contract accepts them without complaint when `AppPublicKey` is used. [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
