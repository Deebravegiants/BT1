Audit Report

## Title
Push-Pattern ETH Transfer in `_processWithdrawalCompletion` Permanently Freezes Funds for Non-Payable Contract Withdrawers - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._processWithdrawalCompletion` unconditionally pushes ETH to the withdrawer via `payable(to).call{value: amount}("")`. If the withdrawer is a smart contract that cannot receive ETH, every call to `completeWithdrawal` or `completeWithdrawalForUser` reverts permanently. Because rsETH is already burned and ETH already transferred into `LRTWithdrawalManager` during the prior `unlockQueue` step, the ETH is irretrievably frozen with no admin recovery path.

## Finding Description
The withdrawal lifecycle is split across two separate transactions:

**Step 1 — `unlockQueue`** (operator-only, lines 305–307): rsETH is burned via `IRSETH.burnFrom` and the corresponding ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. These state changes are final and cannot be undone.

**Step 2 — `completeWithdrawal` / `completeWithdrawalForUser`** (lines 183–204): both call `_processWithdrawalCompletion`, which at line 734 calls `_transferAsset(asset, user, request.expectedAssetAmount)`.

`_transferAsset` for ETH (lines 876–883):
```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` is a contract without a `receive()` function (or one that reverts), `sent == false` and the entire transaction reverts. Because Solidity reverts unwind all state changes in that transaction, the `popFront()` (line 705), `delete withdrawalRequests[requestId]` (line 712), and `unlockedWithdrawalsCount[asset]--` (line 717) are all rolled back. The request remains intact and every subsequent retry fails identically.

There is no admin escape hatch: no redirect, cancel, or rescue function exists in the contract. `sweepRemainingAssets` (lines 395–413) is gated by `if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist()` (line 403), so it is permanently blocked for the affected asset as long as the stuck request exists. The operator-facing `completeWithdrawalForUser` (line 202) calls the same `_processWithdrawalCompletion` and hits the same revert; the NatDoc comment at line 191 even acknowledges it is "not expected to be used for ETH."

## Impact Explanation
**Critical — Permanent freezing of funds.**

Once `unlockQueue` executes, the user's rsETH is burned and the ETH is held in `LRTWithdrawalManager`. If the user's address cannot receive ETH, the ETH is permanently stranded: the user loses both their rsETH (burned, unrecoverable) and their ETH (undeliverable). Additionally, `unlockedWithdrawalsCount[ETH_TOKEN]` can never reach zero for the affected asset, permanently blocking `sweepRemainingAssets` for all users of that asset.

## Likelihood Explanation
**Medium.** Smart contract wallets (Gnosis Safe, multisigs, DAO treasuries) are primary holders of large rsETH positions and are the most likely to initiate large ETH withdrawals. Many such contracts do not implement a `receive()` function or implement one that conditionally reverts. The protocol performs no on-chain check at `initiateWithdrawal` time to verify the caller can receive ETH. No attacker is required — the scenario is triggered by ordinary protocol usage from a contract address.

## Recommendation
Replace the push pattern for ETH with a pull pattern. In `_processWithdrawalCompletion`, instead of calling `_transferAsset` directly, record the owed amount in a mapping (e.g., `mapping(address => uint256) public pendingETHWithdrawals`) and emit an event. Provide a separate `claimETH()` function that the user calls to pull their ETH. A failed pull only affects the caller's own transaction and cannot freeze funds or block `sweepRemainingAssets` for other users.

## Proof of Concept
1. Deploy a contract `NoReceive` (no `receive()` or `fallback()`) holding rsETH.
2. `NoReceive` calls `initiateWithdrawal(ETH_TOKEN, amount, "")` — succeeds; rsETH transferred to `LRTWithdrawalManager`.
3. Operator calls `unlockQueue` — rsETH burned (line 305), ETH moved into `LRTWithdrawalManager` (line 307). These are final.
4. Any call to `completeWithdrawal(ETH_TOKEN, "")` or `completeWithdrawalForUser(ETH_TOKEN, address(NoReceive), "")` reverts at `_transferAsset` (line 878–879) with `EthTransferFailed`.
5. All state changes in step 4 are rolled back; the request persists. Every retry fails identically.
6. ETH is permanently locked in `LRTWithdrawalManager`; `unlockedWithdrawalsCount[ETH_TOKEN]` never decrements; `sweepRemainingAssets` is permanently blocked for ETH.

**Foundry test sketch:**
```solidity
function test_permanentFreeze() public {
    NoReceive victim = new NoReceive();
    // fund victim with rsETH, initiate withdrawal
    vm.prank(address(victim));
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
    // operator unlocks — rsETH burned, ETH in contract
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH_TOKEN, ...);
    // completeWithdrawal always reverts
    vm.expectRevert(LRTWithdrawalManager.EthTransferFailed.selector);
    vm.prank(address(victim));
    withdrawalManager.completeWithdrawal(ETH_TOKEN, "");
    // ETH balance of withdrawalManager unchanged — funds frozen
    assertGt(address(withdrawalManager).balance, 0);
}
```
