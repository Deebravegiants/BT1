### Title
Stale `isSupportedAsset` and Related Mappings After `setToken` Update — (File: `contracts/LRTConfig.sol`)

---

### Summary

`LRTConfig.setToken()` updates only `tokenMap[key]` to a new token address, but leaves `isSupportedAsset`, `supportedAssetList`, `depositLimitByAsset`, and `assetStrategy` keyed on the **old** token address. After the update, the new token address is unrecognized as supported across the entire protocol, while the old (potentially deprecated) address remains fully active. This is a direct structural analog to M-12: a key address is updated in one mapping while all dependent mappings remain stale.

---

### Finding Description

`LRTConfig` maintains two independent address-keyed systems for supported assets:

1. `tokenMap[bytes32 key] → address` — a named lookup updated by `setToken` / `_setToken`.
2. `isSupportedAsset[address] → bool`, `supportedAssetList[]`, `depositLimitByAsset[address]`, `assetStrategy[address]` — all set during `_addNewSupportedAsset`, keyed by the **literal token address**. [1](#0-0) 

When the admin calls `setToken(ST_ETH_TOKEN, newStETH)`: [2](#0-1) 

Only `tokenMap[ST_ETH_TOKEN]` is updated to `newStETH`. The function does **not** touch:

- `isSupportedAsset[newStETH]` → remains `false`
- `isSupportedAsset[oldStETH]` → remains `true`
- `supportedAssetList` → still contains `oldStETH`
- `depositLimitByAsset[oldStETH]` → still set
- `assetStrategy[oldStETH]` → still set [3](#0-2) 

Downstream consumers of `getLSTToken` now receive `newStETH`, but every guard that checks `isSupportedAsset` or iterates `supportedAssetList` still operates on `oldStETH`:

- `LRTDepositPool.depositAsset(newStETH, ...)` reverts at `onlySupportedERC20Token` because `isSupportedAsset[newStETH] == false`.
- `LRTWithdrawalManager.initiateWithdrawal(newStETH, ...)` reverts at `onlySupportedAsset`.
- `LRTDepositPool.stakeEthForStETH()` calls `getLSTToken(ST_ETH_TOKEN)` and gets `newStETH`, but `LRTOracle.assetPriceOracle[newStETH]` is unset (zero), causing `getAssetPrice` to return 0 and breaking rsETH mint calculations. [4](#0-3) 

Meanwhile, `depositAsset(oldStETH, ...)` continues to succeed against a token the admin intended to retire, and `getTotalAssetDeposits` still accounts for `oldStETH` balances, not `newStETH`.

---

### Impact Explanation

**Medium — Temporary freezing of funds.**

After a legitimate `setToken` call, all user-facing deposit and withdrawal paths for the new token address are bricked. Users holding the new token cannot enter or exit the protocol through the intended asset. The old token address remains active, creating a split-state where the protocol accepts a deprecated asset and rejects the intended one. The freeze persists until the admin manually calls `addNewSupportedAsset(newStETH, ...)` and `removeSupportedAsset(oldStETH, ...)` — a non-obvious remediation step not enforced or documented by `setToken`.

---

### Likelihood Explanation

**Low.** The trigger is a legitimate admin call to `setToken` with a new address for an already-registered key (e.g., updating stETH or ETHx to a new contract). This is a realistic operational scenario (token contract upgrade, migration). The `_setToken` function explicitly permits overwriting an existing key: [5](#0-4) 

The admin has no on-chain signal that additional steps are required, making the incomplete update easy to miss.

---

### Recommendation

`setToken` should detect when a key already has a registered address and, if so, atomically migrate all dependent mappings (`isSupportedAsset`, `supportedAssetList`, `depositLimitByAsset`, `assetStrategy`) from the old address to the new one. Alternatively, `setToken` should be restricted to first-time initialization only, and a dedicated `migrateToken(bytes32 key, address newAddr)` function should be introduced that performs the full migration atomically.

---

### Proof of Concept

1. Protocol is initialized: `stETH = 0xAAA...`, `tokenMap[ST_ETH_TOKEN] = 0xAAA`, `isSupportedAsset[0xAAA] = true`, `supportedAssetList = [0xAAA]`.
2. Admin calls `setToken(ST_ETH_TOKEN, 0xBBB)` (new stETH contract).
3. State after: `tokenMap[ST_ETH_TOKEN] = 0xBBB`, but `isSupportedAsset[0xBBB] = false`, `isSupportedAsset[0xAAA] = true`, `supportedAssetList = [0xAAA]`.
4. User calls `LRTDepositPool.depositAsset(0xBBB, amount, ...)` → reverts `AssetNotSupported` at `onlySupportedERC20Token`.
5. User calls `LRTWithdrawalManager.initiateWithdrawal(0xBBB, rsETHAmount, ...)` → reverts `AssetNotSupported`.
6. `LRTDepositPool.stakeEthForStETH()` calls `getLSTToken(ST_ETH_TOKEN)` → gets `0xBBB` → calls `ILido(0xBBB).submit(...)` → reverts or misbehaves if `0xBBB` is not yet a valid Lido contract.
7. All new-token deposit/withdrawal paths are frozen until the admin performs the additional (undocumented) remediation steps.

```text
    function _addNewSupportedAsset(address asset, uint256 depositLimit) private {
        UtilLib.checkNonZeroAddress(asset);
        if (depositLimit == 0) {
            revert InvalidDepositLimit();
        }
        if (isSupportedAsset[asset]) {
            revert AssetAlreadySupported();
        }
        isSupportedAsset[asset] = true;
        supportedAssetList.push(asset);
        depositLimitByAsset[asset] = depositLimit;
        emit AddedNewSupportedAsset(asset, depositLimit);
    }
```

```text
    /// @dev private function to set a token
    /// @param key Token key
    /// @param val Token address
    function _setToken(bytes32 key, address val) private {
        UtilLib.checkNonZeroAddress(val);
        if (tokenMap[key] == val) {
            revert ValueAlreadyInUse();
        }
        tokenMap[key] = val;
        emit SetToken(key, val);
    }
```
