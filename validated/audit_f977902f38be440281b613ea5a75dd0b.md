### Title
Per-Block Reference Script Size Limit Bypassed via Sub-Transaction Reference Scripts in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces nested sub-transactions. The per-transaction reference script size limit (`ppMaxRefScriptSizePerTx`) is correctly enforced for the entire batch (top-level + sub-transactions) via `batchNonDistinctRefScriptsSize`. However, the per-block reference script size limit (`ppMaxRefScriptSizePerBlock`) is enforced by the inherited Conway BBODY rule using `totalRefScriptSizeInBlock`, which calls `txNonDistinctRefScriptsSize` — a function that only counts reference scripts from the **top-level** transaction body, not from sub-transactions. An unprivileged transaction author can craft top-level transactions whose sub-transactions collectively use large reference scripts, causing the actual per-block reference script total to exceed `ppMaxRefScriptSizePerBlock` while the block-level check passes.

---

### Finding Description

**Vulnerability class**: Resource-limit bypass — missing enforcement of accumulated total against a maximum limit (direct analog to the `investFIAT`/`maxRaiseWei` pattern).

**Root cause — the asymmetry between per-TX and per-block checks:**

The Dijkstra era defines `batchNonDistinctRefScriptsSize` specifically to account for sub-transactions:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs:263-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

This function is used in `validateAllRefScriptSize` (the per-TX check in the Dijkstra LEDGER rule):

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs:313-329
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $
        DijkstraTxRefScriptsSizeTooBig ...
```

However, the per-block limit is enforced in the Conway BBODY rule (inherited by Dijkstra), which calls `totalRefScriptSizeInBlock`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs:342-355
validateBodyRefScriptsSizeTooBig pp blockBody utxo =
  let totalSize = totalRefScriptSizeInBlock protVer txs utxo
      maxSize = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerBlockG
   in totalSize <= maxSize ?! ...
```

`totalRefScriptSizeInBlock` accumulates sizes using `txNonDistinctRefScriptsSize` per transaction:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs:357-370
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
```

`txNonDistinctRefScriptsSize` inspects only the top-level transaction's inputs and reference inputs. It has no awareness of `subTransactionsTxBodyL`. The Dijkstra-specific `batchNonDistinctRefScriptsSize` is **never called** in the block-level check.

**The gap**: Reference scripts used exclusively inside sub-transactions contribute 0 bytes to `totalRefScriptSizeInBlock`, even though they are fully deserialized and validated by every node processing the block.

---

### Impact Explanation

This is a **Medium** impact finding: attacker-controlled transactions exceed intended validation limits.

The `ppMaxRefScriptSizePerBlock` parameter was introduced after the June 2024 DDoS attack to cap the total deserialization cost of reference scripts per block. By placing all reference script usage inside sub-transactions, an attacker can submit a block where:

- Each top-level transaction passes `validateAllRefScriptSize` (batch total ≤ `ppMaxRefScriptSizePerTx`)
- `totalRefScriptSizeInBlock` counts near-zero bytes for each transaction (no top-level reference inputs)
- The actual total reference script bytes deserialized by nodes is `N × ppMaxRefScriptSizePerTx`, where N is the number of such transactions in the block

With default parameters (`ppMaxRefScriptSizePerBlock = 1 MiB`, `ppMaxRefScriptSizePerTx = 200 KiB`), a block with 5+ such transactions would cause nodes to deserialize 1+ MiB of reference scripts while the block-level check reports 0 bytes — a complete bypass of the per-block cap. The only remaining bound is the raw block body size (`ppMaxBBSize`), which is a much weaker constraint.

---

### Likelihood Explanation

The Dijkstra era targets protocol version 13+ and is not yet active on mainnet (current mainnet is protocol version 10/11). However, the code is present in the production codebase and will be activated via a hard fork. Once active, any unprivileged transaction author can exploit this — no special keys, governance majority, or privileged access is required. The attack requires only crafting a valid Dijkstra top-level transaction with sub-transactions that reference large scripts, which is a standard transaction-building operation.

---

### Recommendation

Override `validateBodyRefScriptsSizeTooBig` (or `totalRefScriptSizeInBlock`) in the Dijkstra BBODY rule to use `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize` when computing the per-block total. Specifically, the accumulation step in `totalRefScriptSizeInBlock` should be replaced with a Dijkstra-aware variant:

```haskell
-- Dijkstra-aware accumulation:
accum (!accUtxo, !accSum) tx =
  (updatedUtxo, accSum + batchNonDistinctRefScriptsSize (UTxO accUtxo) tx)
```

This mirrors the fix applied at the per-transaction level and closes the asymmetry between the two limit checks.

---

### Proof of Concept

1. Set `ppMaxRefScriptSizePerBlock = 1 MiB`, `ppMaxRefScriptSizePerTx = 200 KiB`.
2. Produce a UTxO entry containing a large Plutus reference script (e.g., 100 KiB).
3. Craft a Dijkstra top-level transaction with **zero top-level reference inputs** and 2 sub-transactions, each referencing the 100 KiB script (batch total = 200 KiB ≤ `ppMaxRefScriptSizePerTx` ✓).
4. `validateAllRefScriptSize` passes (200 KiB ≤ 200 KiB).
5. `totalRefScriptSizeInBlock` counts **0 bytes** for this transaction (no top-level reference inputs).
6. Repeat step 3–5 for 10 such transactions in the same block.
7. Block is accepted. Actual reference script bytes deserialized by nodes: 10 × 200 KiB = **2 MiB**, which is double the `ppMaxRefScriptSizePerBlock` limit of 1 MiB.
