### Title
Unelected Committee Members Can Cast Binding Governance Votes by Bypassing MEMPOOL Sanitizer on Protocol Versions 9–10 — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

---

### Summary

On Conway protocol versions 9 and 10, the `GOV` consensus rule contains no check preventing unelected Constitutional Committee (CC) members from casting votes on governance actions. The only guard against this is a MEMPOOL-layer sanitizer, which a block producer can trivially bypass by including the transaction directly in a block. A proposed-but-not-yet-enacted CC member can register a hot key (permitted by `GOVCERT`), pre-cast a vote through a colluding block producer, and have that vote counted toward ratification the moment their `UpdateCommittee` proposal is enacted — without ever having been legitimately elected.

---

### Finding Description

**Root cause — missing check in the GOV consensus rule for pv 9–10**

`conwayGovTransition` in `Gov.hs` gates the unelected-committee-voter check behind `hardforkConwayDisallowUnelectedCommitteeFromVoting`:

```haskell
when (hardforkConwayDisallowUnelectedCommitteeFromVoting $ pp ^. ppProtocolVersionL) $
    failOnNonEmpty
      (unelectedCommitteeVoters committee committeeState gsVotingProcedures)
      (injectFailure . UnelectedCommitteeVoters)
``` [1](#0-0) 

The predicate is defined as:

```haskell
hardforkConwayDisallowUnelectedCommitteeFromVoting :: ProtVer -> Bool
hardforkConwayDisallowUnelectedCommitteeFromVoting pv = pvMajor pv > natVersion @10
``` [2](#0-1) 

For protocol versions 9 and 10 this evaluates to `False`, so the `when` block is never entered and the GOV rule accepts votes from unelected committee members unconditionally.

**The MEMPOOL check is not a consensus rule**

The only active guard for pv ≤ 10 lives in the MEMPOOL sanitizer:

```haskell
unless (hardforkConwayDisallowUnelectedCommitteeFromVoting protVer) $
  -- This check can completely be removed once mainnet switches to protocol
  -- version 11, since the same check has been implemented in the GOV rule.
  failOnNonEmpty
    ( unelectedCommitteeVoters ... )
    (ConwayMempoolFailure . ...)
``` [3](#0-2) 

The MEMPOOL rule is invoked only when a transaction enters the mempool. A block producer building a block applies the LEDGER rule directly, bypassing MEMPOOL entirely. The CHANGELOG entry for protocol version 11 explicitly acknowledges this gap: *"Made Constitutional Committee voting restriction into a proper ledger predicate failure, instead of a mempool sanitizer."* [4](#0-3) 

**Hot-key registration for proposed-but-unelected members is permitted**

`conwayGovCertTransition` in `GovCert.hs` allows `ConwayAuthCommitteeHotKey` for any cold credential that is either a current committee member **or** appears in a pending `UpdateCommittee` proposal (`isPotentialFutureMember`):

```haskell
let isCurrentMember =
      strictMaybe False (Map.member coldCred . committeeMembers) cgceCurrentCommittee
    committeeUpdateContainsColdCred GovActionState {gasProposalProcedure} =
      case pProcGovAction gasProposalProcedure of
        UpdateCommittee _ _ newMembers _ -> Map.member coldCred newMembers
        _ -> False
    isPotentialFutureMember =
      any committeeUpdateContainsColdCred cgceCommitteeProposals
isCurrentMember || isPotentialFutureMember ?! (injectFailure . ConwayCommitteeIsUnknown) coldCred
``` [5](#0-4) 

This means the hot credential of a proposed-but-unelected member appears in `authorizedHotCommitteeCredentials committeeState`, which is used to populate `knownCommitteeMembers` in `conwayGovTransition`:

```haskell
knownCommitteeMembers = authorizedHotCommitteeCredentials committeeState
``` [6](#0-5) 

Because the hot credential is "known," the `VotersDoNotExist` check passes and the vote is stored in `gasCommitteeVotes`.

**Pre-cast vote is counted at ratification once the member is elected**

`committeeAccepted` in `Ratify.hs` iterates over the current committee's `members` map and looks up each member's hot key in `csCommitteeCreds`, then looks up that hot key in `gasCommitteeVotes`:

```haskell
Just (CommitteeHotCredential hotKey) ->
  case Map.lookup hotKey votes of
    Nothing -> (yes, tot + 1) -- member hasn't voted, vote "no"
    Just VoteYes -> (yes + 1, tot + 1) -- member voted "yes"
``` [7](#0-6) 

Once the `UpdateCommittee` proposal is enacted and the attacker's member enters `committeeMembers`, the pre-cast vote stored in `gasCommitteeVotes` is immediately counted. This is confirmed by the existing test:

```
it "at protocol version 10, the vote is counted once the committee update is enacted"
``` [8](#0-7) 

---

### Impact Explanation

A proposed CC member whose cold key is controlled by an attacker can, with the cooperation of a single block producer, pre-cast a `VoteYes` on any in-flight governance action (treasury withdrawal, parameter change, hard fork initiation, constitution update). The moment the `UpdateCommittee` proposal is enacted, the pre-cast vote is retroactively counted toward the committee acceptance threshold. This can cause a governance action to be ratified that would not otherwise meet the required threshold, constituting an **unauthorized governance, treasury, protocol-parameter, or hard-fork action being enacted** — a Critical/High impact per the allowed scope.

---

### Likelihood Explanation

The attack requires two conditions:
1. The attacker controls the cold key of a credential appearing in a pending `UpdateCommittee` proposal (realistic: the attacker can propose their own member or compromise a proposed member's key).
2. A block producer includes the attacker's vote transaction directly in a block, bypassing MEMPOOL (realistic: the attacker can be a stake pool operator, or bribe/collude with one; a single block producer suffices — no supermajority is needed).

Both conditions are achievable by an unprivileged actor on protocol versions 9–10. The CHANGELOG confirms this was a known architectural gap that was only closed at protocol version 11.

---

### Recommendation

The fix — already applied at protocol version 11 — is to move the `unelectedCommitteeVoters` check unconditionally into the `GOV` consensus rule, removing the `when (hardforkConwayDisallowUnelectedCommitteeFromVoting ...)` guard, so that the check applies at all protocol versions and cannot be bypassed by block producers who skip the MEMPOOL rule. [1](#0-0)

### Proof of Concept

**Attack chain on protocol version 10:**

1. Attacker controls cold key `ccCold` and submits (or observes) an `UpdateCommittee` proposal adding `ccCold` to the committee.
2. Attacker calls `ConwayAuthCommitteeHotKey ccCold ccHot` — accepted by `GOVCERT` because `isPotentialFutureMember` is `True`.
3. Attacker constructs a transaction containing `VotingProcedures { CommitteeVoter ccHot → VoteYes on govActionId }`.
4. Attacker (as a block producer, or colluding with one) includes this transaction directly in a block, bypassing MEMPOOL.
5. `conwayGovTransition` processes the transaction: `hardforkConwayDisallowUnelectedCommitteeFromVoting pv10 = False`, so the `UnelectedCommitteeVoters` check is skipped; `ccHot ∈ knownCommitteeMembers` so `VotersDoNotExist` passes; vote is stored in `gasCommitteeVotes`.
6. The `UpdateCommittee` proposal is ratified and enacted; `ccCold` enters `committeeMembers`.
7. At the next `RATIFY` invocation, `committeeAccepted` finds `ccHot` in `csCommitteeCreds` and the pre-cast `VoteYes` in `gasCommitteeVotes`, counting it toward the threshold.
8. The target governance action (e.g., `TreasuryWithdrawals`) is ratified and enacted without the attacker's committee member having gone through a legitimate election cycle. [1](#0-0) [9](#0-8) [10](#0-9) [11](#0-10) [12](#0-11)
