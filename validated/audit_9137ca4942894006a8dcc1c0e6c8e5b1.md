Audit Report

## Title
Missing Chainlink Staleness Validation Enables Over-Minting of rsETH, Extracting Value from Existing Depositors — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt` and `answeredInRound`, returning a raw price with no freshness validation. Because `LRTDepositPool.getRsETHAmountToMint()` divides the live (potentially stale) asset price by the stored `rsETHPrice`, a stale-high feed causes the protocol to mint more rsETH than the deposited collateral is worth. When the feed corrects and `updateRSETHPrice()` is called, the rsETH price drops, diluting all existing holders and transferring value to the attacker — constituting protocol insolvency.

## Finding Description

**Root cause — `ChainlinkPriceOracle.getAssetPrice()` (line 52):**

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

All five return values of `latestRoundData()` — `(roundId, answer, startedAt, updatedAt, answeredInRound)` — are available, but only `answer` is used. `updatedAt` and `answeredInRound` are discarded with no staleness or round-completeness check. The interface at lines 14–17 correctly declares all five fields, confirming this is a deliberate omission rather than an interface limitation.

Notably, the same codebase implements proper staleness checks in `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` (lines 27–32):
```solidity
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
```
This demonstrates the developers know how to validate Chainlink data but failed to apply it in `ChainlinkPriceOracle`.

**Minting formula — `LRTDepositPool.getRsETHAmountToMint()` (line 520):**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` is fetched live from the Chainlink feed on every call. `lrtOracle.rsETHPrice()` is a stored value last written by `updateRSETHPrice()`. If the live feed is stale-high, the numerator is inflated while the denominator reflects a prior correct state, so the minted rsETH exceeds the true ETH value deposited.

**Price update — `LRTOracle._updateRsETHPrice()` (lines 249–250, 339):**

`_getTotalEthInProtocol()` also calls `getAssetPrice()` for every supported asset. When the stale price is active, `updateRSETHPrice()` bakes the inflated TVL into `rsETHPrice`, keeping it near 1.0e18. When the price corrects, the next `updateRSETHPrice()` call sees a lower TVL against the now-larger rsETH supply, driving `rsETHPrice` down and diluting all holders.

**Existing guards are insufficient:**

The downside protection at lines 270–282 of `LRTOracle.sol` pauses the protocol only if `pricePercentageLimit > 0` AND the price drop exceeds the configured threshold. In the described scenario (~0.54% drop), a threshold of 1% or higher would not trigger a pause. If `pricePercentageLimit == 0` (unset), no protection fires at all. Critically, even if the pause triggers, the attacker has already received the over-minted rsETH.

## Impact Explanation

**Critical — Protocol insolvency.** The protocol accepts collateral worth X ETH but issues rsETH claims totalling more than X ETH. After price correction, total rsETH claims exceed total collateral, creating a permanent undercollateralisation gap. This is a direct, concrete instance of protocol insolvency — an allowed critical impact. The value is not merely "at risk"; it is irreversibly transferred from existing holders to the attacker at the moment of the stale-price deposit.

## Likelihood Explanation

No special privilege is required: any user can call `depositAsset()`. Chainlink feeds go stale during network congestion, gas price spikes, or node outages — all observed on mainnet. The stETH/ETH feed has a 24-hour heartbeat and a 0.5% deviation threshold; a period of low volatility combined with network congestion can leave the feed stale for hours. The attack window is the entire duration of the stale period, and it is repeatable across multiple blocks. The attacker needs only to monitor the feed's `updatedAt` timestamp off-chain and submit a deposit transaction when staleness is detected.

## Recommendation

Add staleness and sanity checks inside `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(price > 0,                          "non-positive price");
require(answeredInRound >= roundId,         "stale round");
require(updatedAt + STALENESS_THRESHOLD > block.timestamp, "stale price");
```

`STALENESS_THRESHOLD` should be set per feed (e.g., heartbeat + buffer). Store it alongside `assetPriceFeed` in the mapping and enforce it in `updatePriceFeedFor()`. This pattern is already implemented correctly in `ChainlinkOracleForRSETHPoolCollateral.sol` and should be applied consistently.

## Proof of Concept

**Minimal reproducible call sequence (Foundry fork test):**

1. Deploy protocol with 1 000 stETH deposited, 1 000 rsETH minted, `rsETHPrice = 1.0e18`.
2. Mock the stETH/ETH Chainlink feed to return `price = 1.05e18` with `updatedAt = block.timestamp - 2 days` (stale, beyond 24h heartbeat). `answeredInRound < roundId` to simulate an incomplete round.
3. Call `pool.getRsETHAmountToMint(stETH, 100e18)` — returns `105e18` (over-minted by 6 rsETH) because `ChainlinkPriceOracle` does not revert on the stale data.
4. Attacker calls `pool.depositAsset(stETH, 100e18, 0, "")` — receives 105 rsETH for 100 stETH (true value 99 ETH).
5. Restore mock feed to `price = 0.99e18`, `updatedAt = block.timestamp`. Call `lrtOracle.updateRSETHPrice()`.
6. Assert `lrtOracle.rsETHPrice() < 1.0e18` (≈ 0.9946e18).
7. Assert attacker's 105 rsETH redeems for > 99 ETH (≈ 104.4 ETH) — profit ≈ 5.4 ETH.
8. Assert original 1 000 rsETH holders' claim < 1 000 ETH (≈ 994.6 ETH) — loss ≈ 5.4 ETH.
9. Assert total claims (≈ 1 099 ETH) == total collateral (1 099 ETH) but holders are underpaid by 5.4 ETH — insolvency confirmed.

The `ChainlinkOracleForRSETHPoolCollateral` contract in the same repo demonstrates the correct pattern and can serve as a reference implementation for the fix.
