### Title
`RSETHPriceFeed.getRoundData()` Returns Corrupted Historical Prices and `latestRoundData()` Allows Same `roundId` to Have Multiple Answers - (File: contracts/oracles/RSETHPriceFeed.sol)

### Summary
`RSETHPriceFeed` is deployed as a Chainlink `AggregatorV3Interface`-compatible price feed for rsETH/USD. It computes its answer by multiplying the ETH/USD Chainlink feed's answer by the current `rsETHPrice()`. However, it inherits the `roundId` system entirely from the ETH/USD base feed, while the rsETH price component can change independently. This breaks two Chainlink invariants: (1) `latestRoundData()` can return different answers for the same `roundId`, and (2) `getRoundData(_roundId)` returns corrupted historical data by mixing a historical ETH/USD price with the **current** rsETH/ETH ratio.

### Finding Description
`RSETHPriceFeed` implements `AggregatorV3Interface` and is intended to be used as a drop-in Chainlink price feed for rsETH/USD by external integrating protocols (lending markets, DEXes, etc.).

**Issue 1 — `latestRoundData()` allows the same `roundId` to have multiple answers:** [1](#0-0) 

The `roundId` returned is sourced entirely from `ETH_TO_USD.latestRoundData()`. The ETH/USD feed's `roundId` only increments when the ETH/USD Chainlink oracle posts a new round. However, `rsETHPrice()` (from `LRTOracle`) changes continuously as assets are deposited, rewards accrue, or EigenLayer positions change — all without any ETH/USD round update. Therefore, the same `roundId` can be observed returning different `answer` values at different block timestamps, which is impossible in a standard Chainlink feed.

**Issue 2 — `getRoundData(_roundId)` returns factually incorrect historical data:** [2](#0-1) 

When a protocol queries a historical round via `getRoundData(oldRoundId)`, the function fetches the historical ETH/USD price for that round from `ETH_TO_USD.getRoundData(_roundId)`, but then multiplies it by `RS_ETH_ORACLE.rsETHPrice()` — the **current** rsETH/ETH ratio, not the ratio that existed at the time of that historical round. The result is a fabricated price that never actually existed on-chain.

For example:
- Round 100: ETH/USD = $2000, rsETH/ETH = 1.05 → actual rsETH/USD at round 100 = $2100
- Later: rsETH/ETH changes to 1.10 (no new ETH/USD round)
- `getRoundData(100)` now returns: $2000 × 1.10 = $2200 — a price that never existed at round 100

### Impact Explanation
Any protocol integrating `RSETHPriceFeed` as a Chainlink-compatible oracle and relying on the `roundId` system for staleness detection, price consistency checks, or historical price verification will receive incorrect data. Specifically:
- Staleness checks of the form `answeredInRound == roundId` will not detect rsETH price changes that occur between ETH/USD rounds.
- Circuit breakers or TWAP-style checks using `getRoundData()` for historical comparison will operate on corrupted prices.
- This can cause integrating lending protocols to misprice rsETH collateral — either accepting under-collateralized positions or incorrectly liquidating healthy ones.

Impact: **Low — Contract fails to deliver promised returns** (the contract promises full `AggregatorV3Interface` compatibility but delivers broken roundId semantics and corrupted historical data).

### Likelihood Explanation
`RSETHPriceFeed` is a production contract explicitly designed for external protocol integration. Any protocol that integrates it and uses `getRoundData()` or relies on `roundId` monotonicity for staleness/consistency checks will be affected. The rsETH price changes on every deposit, reward accrual, or EigenLayer update, making the divergence between `roundId` and `answer` a routine occurrence, not an edge case.

### Recommendation
1. **For `latestRoundData()`**: Maintain an internal `roundId` counter that increments every time `rsETHPrice()` changes materially (i.e., whenever the answer would change), decoupled from the ETH/USD feed's round system.
2. **For `getRoundData()`**: Either revert with `"No data present"` (as the Chainlink spec recommends when historical data is unavailable), or store historical rsETH/USD snapshots keyed by an internal round counter so that historical queries return data that actually existed at that round.
3. Add documentation clearly stating that the `roundId` returned reflects only ETH/USD feed rounds and does not capture rsETH price updates, so integrating protocols are not misled.

### Proof of Concept
1. At block B1: ETH/USD round = 100, ETH/USD price = $2000, rsETH/ETH = 1.05 → `latestRoundData()` returns `(roundId=100, answer=2100e8, ...)`.
2. A large deposit occurs, increasing rsETH/ETH to 1.10. No new ETH/USD round is posted (roundId stays 100).
3. At block B2: `latestRoundData()` returns `(roundId=100, answer=2200e8, ...)` — same `roundId`, different `answer`.
4. A protocol queries `getRoundData(100)` expecting the price at round 100 ($2100): it receives $2200 — the current rsETH price applied to the historical ETH/USD price.
5. The protocol's staleness check (`answeredInRound == roundId` → `100 == 100`) passes, falsely indicating the price is fresh and consistent with round 100. [3](#0-2)

contract RSETHPriceFeed is AggregatorV3Interface {
    /// @notice Price feed for (ETH / USD) pair
    AggregatorV3Interface public immutable ETH_TO_USD;

    /// @notice rsETH oracle contract
    IRSETHOracle public immutable RS_ETH_ORACLE;

    string public description;

    /// @param ethToUSDAggregatorAddress the address of ETH / USD feed
    /// @param rsETHOracle the address of rsETHOracle contract
    /// @param description_ priceFeed description (RSETH / USD)
    constructor(address ethToUSDAggregatorAddress, address rsETHOracle, string memory description_) {
        ETH_TO_USD = AggregatorV3Interface(ethToUSDAggregatorAddress);
        RS_ETH_ORACLE = IRSETHOracle(rsETHOracle);

        description = description_;
    }

    function decimals() external view returns (uint8) {
        return ETH_TO_USD.decimals();
    }

    function version() external view returns (uint256) {
        return ETH_TO_USD.version();
    }

    function getRoundData(uint80 _roundId)
        external
        view
        returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
    {
        (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.getRoundData(_roundId);

        answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
    }

    function latestRoundData()
        external
        view
        returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
    {
        (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
        answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
    }
}
