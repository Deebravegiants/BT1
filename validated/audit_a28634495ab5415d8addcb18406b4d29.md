### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check via Overly Broad `account_tx_in_pool_or_recent_block` Return Assumption — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` in the gateway assumes that `account_tx_in_pool_or_recent_block` returning `true` proves a `deploy_account` transaction is pending for the sender. In reality the function returns `true` for **any** transaction type. An unprivileged attacker who observes a valid nonce-0 invoke for an account in the mempool can submit a forged nonce-1 invoke (invalid signature) for the same account and have it admitted to the mempool without ever running `__validate__`.

---

### Finding Description

`skip_stateful_validations` is the gateway's UX shortcut: when a user sends `deploy_account` + `invoke(nonce=1)` simultaneously, the account does not yet exist on-chain, so `__validate__` would fail. The function is supposed to skip `__validate__` only when a `deploy_account` is pending.

The condition that triggers the skip is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    ...
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The mempool implementation of `account_tx_in_pool_or_recent_block` is:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` for **any** transaction type — invoke, declare, or deploy_account. The code comment in `skip_stateful_validations` acknowledges this indirectly:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second branch of that reasoning is the flaw. A nonce-0 invoke for an already-deployed account (account_nonce == 0 on-chain) is **not** a future-nonce transaction and is **not** a deploy_account, yet it causes `account_tx_in_pool_or_recent_block` to return `true`. The gateway then skips `__validate__` for any subsequent nonce-1 invoke submitted for that account, regardless of whether its signature is valid.

The result of `skip_validate = true` is passed directly to `run_validate_entry_point`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 175-177
let skip_validate =
    self.run_pre_validation_checks(executable_tx, account_nonce, mempool_client).await?;
self.run_validate_entry_point(executable_tx, skip_validate).await?;
``` [3](#0-2) 

Inside `run_validate_entry_point`, `skip_validate = true` sets `execution_flags.validate = false`, so the blockifier's `StatefulValidator` never calls `__validate__`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

The `StatefulTransactionValidatorConfig` even defines a `max_nonce_for_validation_skip` field (default `Nonce(Felt::ONE)`) that is never consulted by `skip_stateful_validations`; the nonce-1 threshold is hardcoded, making the config field dead code in this path. [5](#0-4) 

---

### Impact Explanation

An attacker who observes that account `A` (nonce 0 on-chain, already deployed) has a valid nonce-0 invoke in the mempool can submit a crafted nonce-1 invoke for `A` with a completely invalid signature. The gateway admits it to the mempool without running `__validate__`. The mempool's `validate_tx` only checks for duplicate hashes and nonce range, not signatures. [6](#0-5) 

The admitted invalid transaction pollutes the mempool, occupies account `A`'s nonce-1 slot, and can block Alice's legitimate nonce-1 transaction (which would need to fee-escalate to displace it). The batcher will eventually reject the invalid transaction when the blockifier re-runs `__validate__`, but the gateway invariant — "only transactions that pass account validation are admitted" — is broken.

This matches the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

---

### Likelihood Explanation

The precondition is an account with on-chain nonce 0 that has a pending nonce-0 transaction in the mempool. This is a common state: any account that has just been deployed and is sending its first transaction satisfies it. The attacker needs no privileged access — only the ability to observe the public mempool and submit a transaction.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that is specific to `deploy_account` transactions, or add a transaction-type filter inside `skip_stateful_validations`. For example, the mempool could expose `deploy_account_in_pool_or_recent_block(address)` that only returns `true` when a `DeployAccount` transaction is present. Alternatively, mirror the `native_blockifier` path (`PyValidator::should_run_stateful_validations`) which requires an explicit `deploy_account_tx_hash` parameter rather than a generic account-presence check. [7](#0-6) 

---

### Proof of Concept

1. Account `A` is deployed on-chain with nonce 0 (has contract code, valid `__validate__`).
2. Alice (the legitimate owner) submits `invoke(nonce=0, valid_sig)` for `A`. It passes `__validate__` and is added to the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Attacker submits `invoke(nonce=1, garbage_sig)` for `A`.
4. Gateway calls `skip_stateful_validations`: `tx.nonce() == 1` ✓, `account_nonce == 0` ✓ → calls `account_tx_in_pool_or_recent_block(A)` → `true`.
5. `skip_validate = true` → `run_validate_entry_point` sets `validate = false` → `__validate__` is **never called**.
6. `validate_by_mempool` checks only for duplicate hash and nonce range — both pass.
7. The forged `invoke(nonce=1, garbage_sig)` is admitted to the mempool, occupying Alice's nonce-1 slot without any signature verification. [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
