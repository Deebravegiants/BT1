Audit Report

## Title
Missing Chainlink Staleness and Round-Completeness Checks Allow Over-Minting of rsETH — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt` and `answeredInRound`, accepting stale or incomplete-round prices without any freshness gate. Because `rsETHPrice` is a stored value updated only on explicit `updateRSETHPrice()` calls, a stale high LST price inflates the numerator of the mint ratio while the denominator remains at a prior correct value, causing any depositor to receive more rsETH than their collateral is worth. When the price corrects, all existing rsETH holders bear the shortfall, driving the protocol toward insolvency.

## Finding Description

**Root cause — `ChainlinkPriceOracle.getAssetPrice()` (`contracts/oracles/ChainlinkPriceOracle.sol`, L52–54):**

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`) are available via the `AggregatorV3Interface` declared at lines 14–17, but only `answer` is consumed. Neither `block.timestamp - updatedAt > threshold` (staleness) nor `answeredInRound < roundId` (incomplete round) is checked.

**Mint calculation — `LRTDepositPool.getRsETHAmountToMint()` (`contracts/LRTDepositPool.sol`, L519–520):**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` delegates directly to `ChainlinkPriceOracle.getAssetPrice()` with no additional freshness gate (`LRTOracle.sol` L156–158). `lrtOracle.rsETHPrice()` is a storage variable updated only when `updateRSETHPrice()` is explicitly called (`LRTOracle.sol` L87–89) — it is **not** refreshed on every deposit.

**Exploit flow:**
1. Chainlink stETH/ETH feed becomes stale at 1.05e18 (true price: 0.99e18); `rsETHPrice` was last set at 1.00e18.
2. Attacker calls the public `depositAsset()` (`LRTDepositPool.sol` L99–118).
3. `getRsETHAmountToMint()` computes `100e18 * 1.05e18 / 1.00e18 = 105 rsETH` for 100 stETH worth only 99 ETH.
4. `updateRSETHPrice()` is later called; `_getTotalEthInProtocol()` uses the corrected price, `rsETHPrice` drops, and the attacker's 105 rsETH represents a claim on ~6 ETH more than deposited.

**Why existing guards fail:**
The downside-protection pause in `_updateRsETHPrice()` (`LRTOracle.sol` L270–281) triggers only *after* `updateRSETHPrice()` is called with the corrected price. By that point the over-minting has already occurred and the attacker holds the excess rsETH. The pause does not prevent the initial over-mint.

## Impact Explanation
**Critical — Protocol insolvency.** The attacker extracts ETH value (via over-minted rsETH) that exceeds their deposit. The shortfall is socialized across all rsETH holders through a reduced `rsETHPrice` after correction. This is a direct, concrete path to protocol insolvency, matching the allowed critical impact class.

## Likelihood Explanation
No privileged access is required. The attacker only needs to call the public `depositAsset()` function while a Chainlink feed is stale. The stETH/ETH feed on mainnet has a 24-hour heartbeat; during rapid LST depeg events (slashing, withdrawal queue freeze, secondary-market sell-off), the on-chain price can lag the true price for minutes to hours. The attack is repeatable across every deposit made during the staleness window and requires no governance capture or oracle operator compromise.

## Recommendation
Add staleness and round-completeness checks in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
uint256 public constant STALENESS_THRESHOLD = 25 hours; // tune per feed heartbeat

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (price <= 0) revert InvalidPrice();
    if (answeredInRound < roundId) revert StalePrice();
    if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Per-feed thresholds stored in a mapping are preferable because different assets have different heartbeat intervals.

## Proof of Concept
The following Foundry fork/mock test demonstrates the invariant breach:

```solidity
// forge test --match-test testStalePriceOverMint -vvv
function testStalePriceOverMint() external {
    // Mock stETH/ETH feed: stale answer 1.05e8, updatedAt = 26 hours ago, answeredInRound < roundId
    vm.mockCall(
        CHAINLINK_FEED,
        abi.encodeWithSignature("latestRoundData()"),
        abi.encode(uint80(10), int256(1.05e8), uint256(0), block.timestamp - 26 hours, uint80(9))
    );

    uint256 rsethMinted = ILRTDepositPool(DEPOSIT_POOL).getRsETHAmountToMint(STETH, 100e18);
    // Returns 105e18 — 6 rsETH over-minted vs. 99 ETH true value
    assertApproxEqRel(rsethMinted, 105e18, 0.001e18);

    // After updateRSETHPrice() with corrected price, attacker's claim > deposit value
    // Shortfall borne by all other rsETH holders → insolvency
}
```

A fuzz variant over `stalePriceHigh ∈ [1.01e18, 1.10e18]` will fail for every value, confirming the invariant `minted_rsETH_value ≤ deposited_ETH_value` is broken whenever the stale price exceeds the true price and `rsETHPrice` has not been updated.
