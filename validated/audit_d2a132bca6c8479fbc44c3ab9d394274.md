### Title
Stale Votes from Removed Participants Inflate TEE Code-Hash Governance Count, Enabling Below-Threshold Allowlist Manipulation - (File: `crates/contract/src/tee/proposal.rs`)

---

### Summary

`CodeHashesVotes::count_votes` (and the analogous `MeasurementVotes::count_votes` / `LauncherHashVotes`) tallies **all** entries in the vote map without filtering to current participants. The `vote_update` path explicitly guards against this with a per-participant filter and a comment acknowledging that the post-resharing cleanup promise can fail. No equivalent guard exists for `vote_code_hash`, `vote_add_launcher_hash`, or `vote_add_measurement`. If the cleanup self-Promise scheduled by `vote_reshared()` fails, stale votes from removed participants persist and are counted toward the governance threshold, allowing a single new participant's vote to cross the threshold and add an unauthorized MPC image hash to the allowlist.

---

### Finding Description

**Root cause — unfiltered vote count in `CodeHashesVotes`**

`crates/contract/src/tee/proposal.rs` `CodeHashesVotes::count_votes`:

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
```

This iterates every entry in `proposal_by_account` regardless of whether the voter is still a participant. The returned count is what `vote_code_hash` (and the launcher / measurement equivalents) uses to decide whether the governance threshold has been crossed.

**Contrast with the defensive pattern in `vote_update`**

`crates/contract/src/lib.rs` `vote_update` (lines 1361–1374):

```rust
// Filter votes to only count current participants voting for this specific update.
// This ensures correctness even if the cleanup promise in MpcContract::vote_reshared() fails.
let valid_votes_count = running_state
    .parameters
    .participants()
    .participants()
    .iter()
    .filter(|(account_id, _, _)| {
        self.proposed_updates
            .vote_by_participant
            .get(account_id)
            .is_some_and(|voted_id| *voted_id == id)
    })
    .count();
```

The comment is explicit: the cleanup promise **can** fail, so `vote_update` re-derives the valid count from the live participant set. `CodeHashesVotes`, `MeasurementVotes`, and `LauncherHashVotes` have no such guard.

**Cleanup path that can fail**

`crates/contract/src/tee/tee_state.rs` `clean_non_participant_votes` (lines 396–400):

```rust
pub fn clean_non_participant_votes(&mut self, participants: &Participants) {
    self.votes = self.votes.get_remaining_votes(participants);
    self.launcher_votes = self.launcher_votes.get_remaining_votes(participants);
    self.measurement_votes = self.measurement_votes.get_remaining_votes(participants);
}
```

This is invoked via a self-Promise after resharing completes. If that Promise runs out of gas, panics, or is otherwise dropped, the stale entries remain in all three vote maps and are counted by the unfiltered `count_votes` on every subsequent governance call.

**Analogous pattern to the external report**

The `RevenueHandler` bug: the entire token balance (including unclaimed prior-epoch revenue) is treated as new revenue for the current epoch, inflating `epochRevenues`.

The NEAR MPC analog: the entire vote map (including votes from participants removed in the last resharing) is treated as the current-epoch vote count, inflating the apparent support for a proposal.

---

### Impact Explanation

An unauthorized `NodeImageHash` added to `allowed_docker_image_hashes` passes the `reverify_participants` check in `crates/contract/src/tee/tee_state.rs` (lines 211–231). A node running that unauthorized image can then:

1. Submit a valid `submit_participant_info` attestation and be accepted as an attested participant.
2. Participate in threshold signing protocols, gaining access to key-share material.
3. Produce or co-produce threshold signatures without the genuine authorization of the required participant set.

This satisfies the **Critical** impact: *Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery.*

---

### Likelihood Explanation

- The `vote_update` comment is an in-code acknowledgment that the cleanup Promise **does** fail in practice (otherwise the guard would be unnecessary).
- A resharing that removes two or more participants who had already voted for a hash leaves those votes in the map.
- A single new participant (or any remaining participant) casting one vote for the same hash can then cross the governance threshold with the stale votes providing the remainder.
- No privileged access, threshold collusion, or TEE attack is required — only a participant-level call to `vote_code_hash` after a resharing whose cleanup Promise failed.

---

### Recommendation

Apply the same defensive pattern used in `vote_update` to every governance vote that uses `CodeHashesVotes`, `MeasurementVotes`, and `LauncherHashVotes`. Specifically, derive the valid count by intersecting the stored vote map with the current participant set at the point of threshold evaluation, rather than relying solely on the post-resharing cleanup Promise to keep the map clean.

---

### Proof of Concept

**Setup:** 5 participants `{P0…P4}`, governance threshold = 3.

1. `P0` and `P1` call `vote_code_hash(malicious_hash)` — count = 2, threshold not reached.
2. A resharing removes `P0` and `P1`; new set = `{P2, P3, P4}`, threshold still = 3.
3. The post-resharing cleanup self-Promise fails (e.g., out of gas). `P0` and `P1`'s votes remain in `CodeHashesVotes::proposal_by_account`.
4. `P2` calls `vote_code_hash(malicious_hash)`. `count_votes` iterates all three entries (`P0`, `P1`, `P2`) and returns 3.
5. 3 ≥ threshold (3) → `malicious_hash` is inserted into `allowed_docker_image_hashes`.
6. A node running the image identified by `malicious_hash` submits a valid attestation, is accepted as an attested participant, and participates in threshold signing.

The effective governance threshold was bypassed with only **one** vote from a current participant, because two stale votes from removed participants were counted as new — the direct analog of the `RevenueHandler` epoch-revenue inflation bug. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
