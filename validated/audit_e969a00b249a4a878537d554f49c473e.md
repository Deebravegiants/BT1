### Title
`accept_requests` Guard on `respond()` Blocks In-Flight Signature Completions During TEE Validation Failure - (`File: crates/contract/src/lib.rs`)

### Summary

The `accept_requests` boolean flag is checked inside `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` — the node-facing callbacks that complete already-queued yield-resume promises. When `verify_tee()` sets `accept_requests = false` (because valid participants drop below the signing threshold), every MPC node's attempt to submit a computed signature is rejected. The yield promises for all in-flight requests then expire, permanently failing those requests. This is the direct analog of the Reserve Protocol finding: a lifecycle-completing function carries a state guard that should only apply to new request intake, not to the resolution of already-accepted work.

---

### Finding Description

`MpcContract` uses a single `accept_requests: bool` field to gate both the acceptance of new signing requests and the submission of responses to existing ones.

**New-request path** (`sign`, `request_app_private_key`, `verify_foreign_transaction`) — all call `check_request_preconditions`, which panics when `accept_requests == false`: [1](#0-0) 

**Response path** (`respond`, `respond_ckd`, `respond_verify_foreign_tx`) — each independently re-checks the same flag and returns an error before doing any work: [2](#0-1) [3](#0-2) [4](#0-3) 

`accept_requests` is set to `false` inside `verify_tee()` when the number of participants with valid TEE attestations falls below the signing threshold: [5](#0-4) 

`verify_tee()` is callable by any current participant (`voter_or_panic()`): [6](#0-5) 

When `accept_requests` is `false`, MPC nodes cannot call `respond()` to deliver signatures they have already computed off-chain. The yield-resume promises created by `sign()` carry a finite block-based timeout. Once that timeout elapses, `pop_oldest_pending_yield` removes the entry and the user's callback receives a failure: [7](#0-6) 

The signature verification inside `respond()` is independent of `accept_requests` — a valid, already-computed signature would pass the cryptographic check if the flag were not present: [8](#0-7) 

---

### Impact Explanation

Every signature request that was enqueued before `accept_requests` was set to `false` will time out and fail. The user's cross-chain operation (e.g., a Bitcoin or EVM transaction that was waiting for the MPC signature) is permanently aborted. The deposit attached to the `sign()` call is consumed. Because `resolve_yields_for` is never called, the pending-request map entry is cleaned up only by the timeout path, not by a successful response. This breaks the request-lifecycle safety invariant: a request that was validly accepted and for which a threshold signature was computed off-chain is never delivered on-chain.

---

### Likelihood Explanation

`verify_tee()` is callable by any participant. The condition that sets `accept_requests = false` — enough participants having invalid/expired attestations to drop below the signing threshold — is realistic during coordinated TEE software upgrades, certificate-expiry windows, or when a Byzantine participant deliberately allows their own attestation to lapse while other participants are also in a grace period. A single Byzantine participant below the signing threshold can time the `verify_tee()` call to coincide with a period of high pending-request volume, maximising the number of yields that expire before `accept_requests` is restored to `true`.

---

### Recommendation

Remove the `accept_requests` check from `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()`. The flag should gate only the creation of new yield-resume promises (i.e., `check_request_preconditions`). Responses to already-accepted requests carry their own cryptographic validity check and should be allowed to complete regardless of the current TEE validation state. This mirrors the Reserve Protocol mitigation: remove the pause guard from the settlement function while keeping it on the trade-initiation function.

---

### Proof of Concept

1. The contract is `Running` with N participants and threshold T. All attestations are valid; `accept_requests = true`.
2. A user calls `sign(...)`, paying the minimum deposit. A yield-resume promise is created and stored in `pending_signature_requests`. MPC nodes begin computing the threshold signature off-chain.
3. Concurrently, enough participants' TEE attestations expire (or a Byzantine participant engineers this) so that only N-k < T participants remain valid.
4. A participant calls `verify_tee()`. `reverify_and_cleanup_participants` returns `TeeValidationResult::Partial`. `validate_governance_against_reconstruction` fails because N-k < T. `accept_requests` is set to `false`.
5. MPC nodes finish computing the signature and call `respond(request, response)`. The check at line 579–581 returns `TeeError::TeeValidationFailed` before `resolve_yields_for` is reached. The yield is not resumed.
6. The yield timeout elapses. `pop_oldest_pending_yield` removes the entry. The user's callback receives a failure. The cross-chain operation is permanently lost.
7. Participants renew their attestations; a subsequent `verify_tee()` call restores `accept_requests = true` — but the in-flight requests have already expired.
