### Title
`LRTOracle.initialize()` Does Not Seed `rsETHPrice` When rsETH Supply Already Exists, Causing Entire TVL to Be Treated as Yield on First Price Update — (File: `contracts/LRTOracle.sol`)

---

### Summary

`LRTOracle.initialize()` leaves `rsETHPrice` and `highestRsethPrice` at zero. If a new oracle proxy is deployed and registered in `LRTConfig` while rsETH already has outstanding supply, the first call to `updateRSETHPrice()` computes `previousTVL = rsethSupply × 0 = 0`, treats the entire protocol TVL as newly earned yield, and mints an outsized protocol fee to the treasury — permanently diluting every existing rsETH holder.

---

### Finding Description

`LRTOracle.initialize()` only stores the config address; it never seeds `rsETHPrice` or `highestRsethPrice`:

```solidity
// contracts/LRTOracle.sol L64-68
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

Both storage variables therefore start at `0`.

`_updateRsETHPrice()` has a guard for the zero-supply case but not for the zero-price / non-zero-supply case:

```solidity
// contracts/LRTOracle.sol L218-226
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}

if (highestRsethPrice == 0) {
    highestRsethPrice = rsETHPrice;   // rsETHPrice is still 0 → highestRsethPrice = 0
}
```

With both prices at zero and `rsethSupply > 0`, the fee calculation becomes:

```solidity
// contracts/LRTOracle.sol L234, L244-246, L250
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);   // = 0

if (!protocolPaused && totalETHInProtocol > previousTVL) {   // always true
    uint256 rewardAmount = totalETHInProtocol - previousTVL; // = entire TVL
    protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
}

uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

The entire TVL is classified as "reward", so the fee is taken on the full TVL rather than on incremental yield. The resulting `newRsETHPrice` is permanently depressed, diluting all existing holders.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

If `protocolFeeInBPS = 1000` (10 %) and TVL = 10 000 ETH, the treasury receives rsETH worth ~1 000 ETH on the very first price update after the new oracle is registered. Every existing rsETH holder's position is diluted by that amount. The loss is permanent: `rsETHPrice` is written at the deflated value and `highestRsethPrice` is set accordingly, so the stolen yield can never be recovered through normal price appreciation.

---

### Likelihood Explanation

The trigger condition — a freshly deployed `LRTOracle` proxy registered against a live protocol — is a standard upgrade pattern. The admin deploys a new proxy (e.g., to add the fee-minting logic introduced in `reinitialize`), calls `setContract(LRT_ORACLE, newOracle)` in `LRTConfig`, and the new oracle's `rsETHPrice` is 0. The public `updateRSETHPrice()` function (no access control beyond `whenNotPaused`) can then be called by any external account to trigger the fee theft. No privileged compromise is required after the oracle is registered.

---

### Recommendation

`initialize()` should accept and store an initial `rsETHPrice` (and `highestRsethPrice`) that reflects the actual current exchange rate, or `_updateRsETHPrice()` should revert when `rsETHPrice == 0` and `rsethSupply > 0`, forcing the admin to seed the price before the first public update.

---

### Proof of Concept

1. Protocol is live: `rsethSupply = 10 000e18`, actual price ≈ `1.05e18`.
2. Admin deploys a new `LRTOracle` proxy and calls `LRTConfig.setContract(LRT_ORACLE, newOracle)`.
3. New oracle state: `rsETHPrice = 0`, `highestRsethPrice = 0`, `pricePercentageLimit = 0`.
4. Attacker (any EOA) calls `LRTOracle.updateRSETHPrice()`.
5. Inside `_updateRsETHPrice()`:
   - `rsethSupply = 10 000e18` → does **not** hit the `rsethSupply == 0` early-return.
   - `highestRsethPrice == 0` → `highestRsethPrice = rsETHPrice = 0`.
   - `previousTVL = 10 000e18 × 0 = 0`.
   - `totalETHInProtocol ≈ 10 500e18` (actual TVL).
   - `rewardAmount = 10 500e18 − 0 = 10 500e18`.
   - `protocolFeeInETH = 10 500e18 × 1000 / 10 000 = 1 050e18` (10 % of full TVL).
   - `newRsETHPrice = (10 500e18 − 1 050e18) / 10 000e18 = 0.945e18`.
6. Treasury receives rsETH worth 1 050 ETH; every holder's rsETH is now worth `0.945 ETH` instead of `1.05 ETH` — a ~10 % loss.
