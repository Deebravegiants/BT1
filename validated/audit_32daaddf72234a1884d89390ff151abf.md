### Title
Missing Manual Emergency Pause/Unpause Governance for `accept_requests` Flag — (`File: crates/contract/src/lib.rs`)

---

### Summary

`MpcContract` gates every critical request-processing method behind an `accept_requests: bool` flag, but exposes no dedicated governance function to manually set that flag to `false` (pause) or `true` (unpause). The flag is exclusively controlled by the automated `verify_tee()` path. When `verify_tee()` freezes the contract by setting `accept_requests = false`, there is no participant-callable method to re-enable it without TEE validation first passing again — leaving the contract in a permanently frozen state if attestation recovery is impossible.

---

### Finding Description

`MpcContract` declares `accept_requests: bool` as a first-class contract field: [1](#0-0) 

This flag is checked as a hard gate in every user-facing and node-facing request path:

- `check_request_preconditions` (called by `sign`, `request_app_private_key`, `verify_foreign_transaction`): [2](#0-1) 

- `respond`: [3](#0-2) 

- `respond_ckd`: [4](#0-3) 

- `respond_verify_foreign_tx`: [5](#0-4) 

The **only** code paths that mutate `accept_requests` are inside `verify_tee()`:

```
TeeValidationResult::Full          → accept_requests = true   (line 1710)
TeeValidationResult::Partial (ok)  → accept_requests = true   (line 1743)
TeeValidationResult::Partial (bad) → accept_requests = false  (line 1737)
``` [6](#0-5) 

The contract's own log message at line 1733 explicitly acknowledges the gap:

> *"This requires manual intervention. We will not accept new signature requests as a safety precaution."*

Yet no such manual-intervention function exists in the public API. There is no `pause()`, no `unpause()`, and no threshold-voted governance call that writes to `accept_requests`. [7](#0-6) 

---

### Impact Explanation

When `verify_tee()` sets `accept_requests = false` (the threshold-breaking partial-attestation branch), the contract enters a state where:

1. All new `sign` / `request_app_private_key` / `verify_foreign_transaction` calls panic with `TeeValidationFailed`.
2. All in-flight yield-resume promises waiting for `respond` / `respond_ckd` / `respond_verify_foreign_tx` are permanently stuck — nodes cannot deliver responses.
3. The only recovery path is for `verify_tee()` to subsequently return `Full` or a safe `Partial` — which requires the same attestation conditions that caused the freeze to be resolved first.

If attestation recovery is not possible (e.g., a TEE firmware update invalidates all stored attestations, or a bug in the attestation-verification logic causes persistent rejection), the contract is frozen indefinitely. A contract upgrade via `propose_update` / `vote_update` is the only escape hatch, but that is a heavyweight governance operation with its own latency and coordination cost, not an emergency stop.

**Impact class**: Medium — request-lifecycle and contract execution-flow freeze that breaks the production safety invariant that the MPC network remains available to process pending and new signing requests.

---

### Likelihood Explanation

The freeze path is reachable without any attacker involvement:

- A TEE firmware/BIOS update can invalidate stored RTMR measurements for all nodes simultaneously.
- An expired or revoked attestation for enough participants can push the valid-participant count below the threshold relation check.
- Any participant calling `verify_tee()` (a permissioned but routine operation) triggers the evaluation; if the bad branch is hit, `accept_requests` flips to `false` immediately.

Once frozen, re-enabling requires all (or enough) participants to obtain fresh, valid attestations and re-submit them via `submit_participant_info`, then have `verify_tee()` called again — a multi-step, time-sensitive coordination that may not be achievable under operational pressure.

---

### Recommendation

Add two threshold-voted governance functions analogous to the `pause()` / `unpause()` pattern:

```rust
/// Emergency pause: requires threshold votes from current participants.
pub fn vote_pause_requests(&mut self) -> Result<(), Error> { ... }

/// Emergency unpause: requires threshold votes from current participants.
pub fn vote_resume_requests(&mut self) -> Result<(), Error> { ... }
```

Alternatively, expose a single `set_accept_requests(value: bool)` that is callable only by the contract itself (i.e., `#[private]`) so it can be triggered via a threshold-voted `propose_update` / `vote_update` config-change path without requiring a full WASM upgrade.

---

### Proof of Concept

1. Contract is `Running`, `accept_requests = true`, N participants, threshold T.
2. A TEE firmware update invalidates attestations for `N - T + 1` participants (just enough to break the threshold relation).
3. Any participant calls `verify_tee()`.
4. `reverify_and_cleanup_participants` returns `TeeValidationResult::Partial` with only `T - 1` valid participants.
5. `validate_governance_against_reconstruction` fails → `accept_requests = false` is written (line 1737).
6. All subsequent `sign(...)` calls panic: `TeeError::TeeValidationFailed`.
7. All in-flight `respond(...)` calls return `Err(TeeError::TeeValidationFailed)` — yield promises never resolve.
8. No public contract method exists to set `accept_requests = true` without first passing TEE validation — the contract is frozen. [8](#0-7) [9](#0-8)
