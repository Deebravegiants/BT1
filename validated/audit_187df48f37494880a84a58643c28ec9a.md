### Title
Sub-Transaction Reference Script Fee Undercharging in Dijkstra Era — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the minimum fee calculation for a top-level transaction does not account for reference scripts used by its embedded sub-transactions. The `getMinFeeTxUtxo` method for `DijkstraEra` delegates to `getConwayMinFeeTxUtxo`, which only measures reference script sizes from the top-level transaction's own inputs. A helper function `batchNonDistinctRefScriptsSize` that correctly aggregates reference script sizes across the entire batch exists but is never wired into the fee validation path. An unprivileged transaction author can therefore submit a Dijkstra batch whose sub-transactions reference arbitrarily large Plutus scripts while paying a fee that does not reflect the deserialization cost of those scripts.

---

### Finding Description

**Vulnerability class:** Resource-limit / fee accounting bug.

**Root cause — step 1: `getMinFeeTxUtxo` is not overridden for Dijkstra.**

The `EraUTxO DijkstraEra` instance sets:

```haskell
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` computes the reference-script component of the minimum fee using only the top-level transaction's inputs:

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
``` [2](#0-1) 

`txNonDistinctRefScriptsSize` only unions the top-level transaction's `referenceInputsTxBodyL` and `inputsTxBodyL`:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

**Root cause — step 2: the batch-aware helper is never used in fee validation.**

`batchNonDistinctRefScriptsSize` correctly aggregates reference script sizes across the top-level transaction *and all sub-transactions*:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [4](#0-3) 

This function is imported in the LEDGER rule but is used only for the `validateAllRefScriptSize` *size-limit* check, not for the minimum-fee computation. [5](#0-4) 

**Root cause — step 3: sub-transactions carry no fee field and pay no fees.**

`DijkstraSubTxBodyRaw` has no fee field; the required fields for `SSubTx` are only `inputs` and `outputs`:

```haskell
requiredFields = \case
  STopTx -> [(0, "inputs"), (1, "outputs"), (2, "fee")]
  SSubTx -> [(0, "inputs"), (1, "outputs")]
``` [6](#0-5) 

Sub-transactions are processed via `updateUTxOStateNoFees`, which explicitly skips fee collection:

```haskell
-- | Like 'updateUTxOState', but does not collect fees. This is used for sub-transactions
-- where fees are not applicable.
updateUTxOStateNoFees :: ...
``` [7](#0-6) 

The design intent is that the single top-level fee covers the entire batch. However, the minimum-fee formula used to validate that fee does not include the reference-script deserialization cost of sub-transactions.

**Root cause — step 4: `dijkstraSubTxProducedValue` omits a fee term.**

The produced-value function for a sub-transaction body contains outputs, deposits, treasury donation, and burned assets — but no fee term — confirming that sub-transactions contribute zero to the fee pot:

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody <> txBody ^. treasuryDonationTxBodyL)
    <> burnedMultiAssets txBody
``` [8](#0-7)

### Impact Explanation

The Conway era introduced a tiered reference-script fee (`minFeeRefScriptCostPerByte`) specifically to prevent DDoS attacks where an attacker submits transactions that are cheap to submit but expensive to validate due to large script deserialization costs (ADR 009). In Dijkstra, this protection does not extend to sub-transactions. An attacker can embed sub-transactions whose reference inputs point to UTxOs containing many large Plutus scripts. The top-level fee is validated against a minimum that ignores those scripts entirely, so the attacker pays only the size-based fee component for the sub-transaction bytes while forcing validators to deserialize and process arbitrarily large reference scripts. This allows attacker-controlled transactions to consume validation resources beyond what the fee mechanism is designed to permit, matching the **Medium** impact category: *attacker-controlled transactions exceed intended validation limits or modify fees outside design parameters*.

---

### Likelihood Explanation

The Dijkstra era is the current development frontier (protocol version 12). Any user who can submit a valid Dijkstra top-level transaction can exploit this gap. No privileged access, governance majority, or key compromise is required. The attacker only needs to know the addresses of UTxOs containing large reference scripts (which are public on-chain) and to construct a top-level transaction embedding sub-transactions that reference those UTxOs. The attack is directly analogous to the June 2024 reference-script DDoS that affected mainnet Conway transactions, making it a known and realistic threat model.

---

### Recommendation

Override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance to use `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
getMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx
```

This ensures the reference-script fee component of the minimum fee reflects the total deserialization cost across the top-level transaction and all its sub-transactions, consistent with the protection introduced in Conway.

---

### Proof of Concept

1. Deploy several UTxOs on Dijkstra testnet, each containing a large Plutus V3 reference script (e.g., near the per-script size limit).
2. Construct a Dijkstra top-level transaction whose body contains N sub-transactions. Each sub-transaction's `referenceInputsTxBodyL` points to one of the large-script UTxOs.
3. Compute the minimum fee using `getMinFeeTxUtxo` (which calls `getConwayMinFeeTxUtxo` → `txNonDistinctRefScriptsSize`). Observe that the reference-script fee component is zero or minimal because the top-level transaction itself has no reference inputs.
4. Compute the fee using `batchNonDistinctRefScriptsSize`. Observe that it is significantly larger (proportional to N × script size × `minFeeRefScriptCostPerByte` tier).
5. Submit the transaction with the fee from step 3. The ledger accepts it because `feesOK` uses `getMinFeeTxUtxo`. Validators must deserialize all N large scripts to process the sub-transactions, consuming resources far in excess of what the fee covers.
