Audit Report

## Title
Token Decimal Mis-Accounting in `viewSwapRsETHAmountAndFee` Causes Severe wrsETH Under-Minting for Non-18-Decimal Collateral - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The token overload of `viewSwapRsETHAmountAndFee` multiplies a raw token-unit amount directly against WAD-scaled (1e18) oracle rates without first normalising to 18 decimals. For any supported token with fewer than 18 decimals, the minted wrsETH amount is off by a factor of `10**(18 − tokenDecimals)`. The full token value is transferred to the pool while the depositor receives a negligible fraction of the wrsETH they are owed.

## Finding Description
The ETH overload compensates for the WAD-scaled rate by multiplying by `1e18` in the numerator:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0) 

This works because ETH has 18 decimals — `amountAfterFee` is already 1e18-scaled, and the extra `1e18` in the numerator cancels the WAD denominator, yielding a result in 1e18 units.

The token overload omits this normalisation:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

Here `amountAfterFee` is in the token's native decimal scale (e.g. 1e6 for USDC), while both `tokenToETHRate` and `rsETHToETHrate` are 1e18-scaled. The two WAD factors cancel each other, leaving the result in the token's native decimal scale rather than 1e18. For a 6-decimal token the output is `10**12` times smaller than correct.

`ChainlinkOracleForRSETHPoolCollateral.getRate()` always returns a 1e18-normalised price regardless of feed decimals:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [3](#0-2) 

So `tokenToETHRate` is always 1e18-scaled, confirming the bug is entirely in the pool arithmetic.

The `deposit(address token, uint256 amount, string referralId)` function calls `viewSwapRsETHAmountAndFee(amount, token)` and mints the returned (incorrect) amount directly to the caller: [4](#0-3) 

No existing check in `addSupportedToken` or `deposit` validates or normalises token decimals: [5](#0-4) 

## Impact Explanation
**Critical — direct theft of depositor funds.**

A depositor sending 1,000 USDC (6 decimals, `tokenToETHRate = 1e18`, `rsETHToETHrate = 1e18`, zero fee) receives:

```
rsETHAmount = 1_000e6 * 1e18 / 1e18 = 1_000e6   // ≈ 1e-12 wrsETH in human units
```

The correct amount is `1_000e18`. The depositor loses `10**12 − 1` times their entitled wrsETH while the full 1,000 USDC is retained by the pool, permanently enriching existing wrsETH holders. This is direct, irreversible theft of depositor funds matching the Critical impact class.

## Likelihood Explanation
The precondition is a single `TIMELOCK_ROLE` admin call to `addSupportedToken` with any non-18-decimal ERC-20 (USDC, USDT, WBTC). This is a normal operational step — the contract is explicitly designed to accept arbitrary ERC-20 collateral. The admin is not acting maliciously; the bug is in the accounting logic. Once such a token is live, every unprivileged depositor calling `deposit(token, amount, referralId)` triggers the loss with no further preconditions and no way to recover funds. **Likelihood: Medium.**

## Recommendation
Normalise `amountAfterFee` to 18 decimals before the rate division, mirroring the ETH path:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

function viewSwapRsETHAmountAndFee(
    uint256 amount,
    address token
) public view onlySupportedToken(token) returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;

    uint8 tokenDecimals = IERC20Metadata(token).decimals();
    uint256 normalizedAmount = amountAfterFee * 1e18 / (10 ** uint256(tokenDecimals));

    uint256 rsETHToETHrate = getRate();
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

    rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
}
```

## Proof of Concept
**Setup:**
- Deploy `RSETHPoolV3` with `feeBps = 0`.
- Add USDC (6 decimals) as a supported token with a mock oracle returning `tokenToETHRate = 1e18`.
- Set `rsETHToETHrate = 1e18`.

**Call:** `deposit(USDC, 1_000e6, "")`

**Observed:** `wrsETH.balanceOf(depositor) == 1_000e6` (≈ 0.000001 wrsETH)

**Expected:** `wrsETH.balanceOf(depositor) == 1_000e18` (1,000 wrsETH)

**Foundry test skeleton:**
```solidity
function test_decimal_mismatch() public {
    // mint 1000 USDC to alice
    usdc.mint(alice, 1_000e6);
    vm.prank(alice);
    usdc.approve(address(pool), 1_000e6);
    vm.prank(alice);
    pool.deposit(address(usdc), 1_000e6, "");
    // assert under-mint
    assertEq(wrsETH.balanceOf(alice), 1_000e6);   // bug: should be 1_000e18
    assertLt(wrsETH.balanceOf(alice), 1_000e18);
}
```
