Audit Report

## Title
Uninitialized `tokenFeeBps` in `addSupportedToken` Allows Zero-Fee Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` introduces a per-token fee mapping `tokenFeeBps` for ERC-20 deposits, but `addSupportedToken` never initializes it. Because Solidity defaults unset mapping values to `0`, every supported token deposit immediately incurs zero protocol fees until an admin separately calls `setTokenFeeBps`. Any user can deposit supported tokens (e.g., wstETH) and receive the full rsETH amount at the oracle rate with no fee deducted, while ETH depositors pay the configured `feeBps`.

## Finding Description
`RSETHPool.sol` maintains two distinct fee variables:

- `feeBps` (L43) — used for native ETH deposits via `deposit(string)`, set at initialization.
- `tokenFeeBps[token]` (L88) — used for ERC-20 deposits via `deposit(address,uint256,string)`, never initialized.

The token deposit path at L298 calls `viewSwapRsETHAmountAndFee(amount, token)`, which at L335–336 reads `tokenFeeBps[token]` and computes `fee = amount * 0 / 10_000 = 0`. The `feeEarnedInToken[token]` accumulator at L300 therefore never increases.

`addSupportedToken` (L637–656) registers a token with an oracle and bridge but contains no assignment to `tokenFeeBps[token]`. The only way to set a non-zero fee is via the separate `setTokenFeeBps` call (L583–594), which requires `DEFAULT_ADMIN_ROLE` and must be invoked explicitly after token registration. There is no guard in `deposit` or `viewSwapRsETHAmountAndFee` that enforces a non-zero fee before allowing deposits.

Exploit flow:
1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0`.
2. Any user immediately calls `deposit(wstETH, amount, "ref")`.
3. `viewSwapRsETHAmountAndFee(amount, wstETH)` returns `fee = 0`, `rsETHAmount = amount * tokenToETHRate / rsETHToETHrate` (full amount, no deduction).
4. User receives full rsETH; `feeEarnedInToken[wstETH]` stays at `0`.
5. This repeats for every token deposit until admin calls `setTokenFeeBps`.

## Impact Explanation
Protocol fee revenue on token deposits is permanently zero from the moment a token is added until `setTokenFeeBps` is explicitly called. Depositors receive more rsETH than the protocol intends to issue (the fee-equivalent rsETH that should be withheld for the treasury is instead transferred to the user). This constitutes **High — Theft of unclaimed yield**: yield (fee revenue) that should accrue to the protocol treasury is silently diverted to depositors on every token deposit.

## Likelihood Explanation
The entry path is fully permissionless — any user can call `deposit(token, amount, referralId)` on any supported token at any time the contract is unpaused. No special role, flash loan, or external condition is required. The fee bypass is structural and applies from the instant `addSupportedToken` is called. Because `addSupportedToken` gives no indication that a follow-up `setTokenFeeBps` call is required, the window of zero-fee deposits is likely to persist across multiple transactions or longer.

## Recommendation
Initialize `tokenFeeBps[token]` inside `addSupportedToken`, either by accepting a `_feeBps` parameter or by falling back to the global `feeBps` when `tokenFeeBps[token]` is zero:

```solidity
// Option A: require fee at registration time
function addSupportedToken(address token, address oracle, address bridge, uint256 _feeBps)
    external onlyRole(TIMELOCK_ROLE)
{
    // ... existing checks ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    tokenFeeBps[token] = _feeBps;
    // ...
}

// Option B: fallback in viewSwapRsETHAmountAndFee
uint256 feeBpsForToken = tokenFeeBps[token] != 0 ? tokenFeeBps[token] : feeBps;
```

## Proof of Concept
Foundry fork test sequence against Arbitrum mainnet:

```solidity
// 1. Impersonate TIMELOCK_ROLE holder
vm.prank(timelockAddress);
pool.addSupportedToken(wstETH, wstETHOracle, wstETHBridge);

// 2. Confirm tokenFeeBps is 0
assertEq(pool.tokenFeeBps(wstETH), 0);

// 3. User deposits wstETH
deal(wstETH, user, 10 ether);
vm.startPrank(user);
IERC20(wstETH).approve(address(pool), 10 ether);
pool.deposit(wstETH, 10 ether, "ref");
vm.stopPrank();

// 4. Assert zero fee collected
assertEq(pool.feeEarnedInToken(wstETH), 0);

// 5. Assert user received full rsETH (no fee deducted)
(uint256 expectedRsETH, uint256 expectedFee) = pool.viewSwapRsETHAmountAndFee(10 ether, wstETH);
assertEq(expectedFee, 0);
assertGt(expectedRsETH, 0);
assertEq(IERC20(address(pool.wrsETH())).balanceOf(user), expectedRsETH);
```
