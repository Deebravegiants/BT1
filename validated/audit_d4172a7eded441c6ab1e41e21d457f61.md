Audit Report

## Title
FIFO Withdrawal Queue Blocking Causes Temporary Freeze of User rsETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._unlockWithdrawalRequests` processes withdrawal requests in strict FIFO order and halts entirely when the head-of-queue request cannot be covered by the vault's available assets. Because no `cancelWithdrawal` function exists and no skip mechanism is implemented, all subsequent users' rsETH is temporarily frozen in the contract with no user-accessible recovery path until the blocking request is eventually serviced.

## Finding Description
**Root cause — admission vs. unlock asset source mismatch:**

`initiateWithdrawal` checks capacity using `getAvailableAssetAmount`, which calls `lrtDepositPool.getTotalAssetDeposits(asset)` — the protocol-wide total including assets still deployed in EigenLayer strategies. [1](#0-0) 

When the operator calls `unlockQueue`, the available amount passed to `_unlockWithdrawalRequests` is only `unstakingVault.balanceOf(asset)` — what has physically been returned from EigenLayer. The unlock loop iterates from `nextLockedNonce[asset]` and **breaks** on the first request it cannot cover: [2](#0-1) 

`nextLockedNonce[asset]` is only advanced when a request is successfully unlocked (L811–813). A request that cannot be covered keeps `nextLockedNonce` frozen at its nonce, blocking every subsequent nonce.

When any later user calls `completeWithdrawal`, `_processWithdrawalCompletion` pops their nonce and checks:
```
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
``` [3](#0-2) 

Since `nextLockedNonce` is stuck at the blocking nonce (e.g., 0), any later user's nonce (e.g., 1) satisfies `1 >= 0` and the call reverts. No `cancelWithdrawal` function exists anywhere in the contract (confirmed by search), so the user has no path to reclaim their rsETH. [4](#0-3) 

**Concrete exploit path:**
1. Protocol holds 100 ETH total: 80 ETH in EigenLayer, 20 ETH in `LRTUnstakingVault`.
2. User A calls `initiateWithdrawal` for rsETH worth 80 ETH. Admission check: `80 < 100` passes. `assetsCommitted = 80`. Nonce = 0.
3. User B calls `initiateWithdrawal` for rsETH worth 10 ETH. Admission check: `10 < 20` passes. `assetsCommitted = 90`. Nonce = 1.
4. Operator calls `unlockQueue`; vault has 20 ETH.
5. Loop evaluates nonce 0: `20 < 80` → `break`. `nextLockedNonce` stays at 0.
6. User B calls `completeWithdrawal`: `popFront()` returns nonce 1; `1 >= 0` → `WithdrawalLocked` revert.
7. User B's rsETH is locked in the contract with no exit path.

## Impact Explanation
User B's rsETH is locked inside `LRTWithdrawalManager` with no user-accessible exit. The funds are not permanently lost — they are released once the operator unstakes sufficient assets from EigenLayer to cover User A's request and calls `unlockQueue` again — but the freeze duration is unbounded from the user's perspective and entirely outside their control. This constitutes **temporary freezing of funds**, a Medium-severity impact within the allowed scope.

## Likelihood Explanation
Medium. The gap between `getTotalAssetDeposits` (admission) and `unstakingVault.balanceOf` (unlock) is a normal, persistent operational state: the protocol routinely has the majority of assets deployed in EigenLayer. Any large withdrawal request queued ahead of smaller ones, when the vault holds less than the large request's payout, triggers this condition. No adversarial intent is required; it arises from ordinary concurrent usage by unprivileged users.

## Recommendation
1. **Add a `cancelWithdrawal` function** that allows a user to remove their own still-locked (nonce ≥ `nextLockedNonce`) withdrawal request and receive their rsETH back, decrementing `assetsCommitted` accordingly.
2. **Alternatively**, modify `_unlockWithdrawalRequests` to `continue` (skip) rather than `break` when a request cannot be covered, so smaller subsequent requests are not blocked by a single large head-of-queue entry. Note this changes FIFO semantics and may require additional design consideration.

function test_FIFOQueueBlocking() public {
    // Setup: 100 ETH total assets, 20 ETH in unstaking vault, 80 ETH in EigenLayer

    // Step 1: User A initiates withdrawal for rsETH worth 80 ETH
    vm.prank(userA);

    // Step 2: User B initiates withdrawal for rsETH worth 10 ETH
    vm.prank(userB);
    withdrawalManager.initiateWithdrawal(ETH, rsETH_10, "");
    // assetsCommitted[ETH] = 90 ETH; nonce 1 assigned to userB

    // Step 3: Operator calls unlockQueue; vault has only 20 ETH
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH, 2, minAsset, minRsETH, maxAsset, maxRsETH);
    // _unlockWithdrawalRequests: nonce 0 → 20 < 80 → break
    // nextLockedNonce[ETH] remains 0

    // Step 4: User B attempts to complete withdrawal
    vm.prank(userB);
    vm.expectRevert(WithdrawalLocked.selector);
    withdrawalManager.completeWithdrawal(ETH, "");
    // nonce 1 >= nextLockedNonce(0) → reverts; userB's rsETH is frozen
    // No cancelWithdrawal exists; userB has no recovery path
}
