### Title
Expired Governance Proposals Can Still Be Ratified and Enacted Due to Missing Expiry Guard in RATIFY Rule - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

In `ratifyTransition`, the expiry check (`gasExpiresAfter < reCurrentEpoch`) is placed exclusively in the `else` (non-ratification) branch. If a proposal has accumulated sufficient votes and meets all other ratification conditions, it is enacted **without any check that it has not expired**. A proposal that has passed its `gasExpiresAfter` epoch can therefore be ratified and enacted, violating the protocol guarantee that expired proposals are removed and their deposits refunded.

---

### Finding Description

The `ratifyTransition` function in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs` processes each governance action in the `RatifySignal` sequence:

```haskell
gas@GovActionState {gasId, gasExpiresAfter} SSeq.:<| sigs -> do
  let govAction = gasAction gas
  if prevActionAsExpected gas ensPrevGovActionIds
    && validCommitteeTerm govAction ensCurPParams reCurrentEpoch
    && not rsDelayed
    && withdrawalCanWithdraw govAction ensTreasury
    && acceptedByEveryone env st gas
    then do
      -- proposal is enacted — no expiry check here
      newEnactState <- trans @(EraRule "ENACT" era) $ ...
      ...
    else do
      st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
      -- expiry check is ONLY here, in the non-ratification branch
      if gasExpiresAfter < reCurrentEpoch
        then pure $ st' & rsExpiredL %~ Set.insert gasId
        else pure st'
``` [1](#0-0) 

The condition `gasExpiresAfter < reCurrentEpoch` is evaluated **only** when the proposal fails the ratification conditions. If `acceptedByEveryone env st gas` is `True` and all other guards pass, the proposal is immediately enacted via the `ENACT` sub-rule — regardless of whether `gasExpiresAfter < reCurrentEpoch`.

The `gasExpiresAfter` field is set at proposal submission time as `addEpochInterval curEpoch expiryInterval`: [2](#0-1) 

The `GovActionState` type stores this as an `EpochNo`: [3](#0-2) 

The scenario that triggers this is the `rsDelayed` mechanism. When a delaying action (e.g., `NewConstitution`, `UpdateCommittee`, `HardForkInitiation`, `NoConfidence`) is enacted, `rsDelayed` is set to `True`, preventing all other proposals from being ratified in that epoch. If a proposal is delayed across enough epoch boundaries that `gasExpiresAfter < reCurrentEpoch` becomes true, but then in a subsequent epoch `rsDelayed = False` and the proposal still has sufficient votes, the proposal is enacted despite being expired.

The codebase itself acknowledges this as a known conformance failure. Multiple tests in `eras/conway/impl/testlib/Test/Cardano/Ledger/Conway/Imp/RatifySpec.hs` are explicitly disabled with `disableInConformanceIt` and reference formal-spec issue #923:

```haskell
-- https://github.com/IntersectMBO/formal-ledger-specifications/issues/923
-- TODO: Re-enable after issue is resolved, by removing this override
disableInConformanceIt "Same lineage" $ whenPostBootstrap $ do
  ...
  passEpoch
  getLastEnactedConstitution `shouldReturn` SJust (GovPurposeId gai2)
  passEpoch
  -- gai3 should NOT be enacted (expired), but the implementation enacts it
  getLastEnactedConstitution `shouldReturn` SJust (GovPurposeId gai2)
``` [4](#0-3) 

The same pattern is confirmed for the "Other lineage" case: [5](#0-4) 

---

### Impact Explanation

Any governance action type can be affected: `NewConstitution`, `UpdateCommittee`, `HardForkInitiation`, `ParameterChange`, `TreasuryWithdrawals`, or `NoConfidence`. An expired `TreasuryWithdrawals` proposal being enacted causes ADA to be transferred out of the treasury after the protocol-guaranteed expiry window — a direct, invalid loss of ADA from the treasury. An expired `HardForkInitiation`, `UpdateCommittee`, or `NewConstitution` being enacted constitutes an unauthorized governance state transition. Both match the Critical and High allowed impacts:

- **Critical**: Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.
- **Critical**: Direct loss of ADA through an invalid ledger state transition (for `TreasuryWithdrawals`).

The `EPOCH` rule's `returnProposalDeposits` function is also affected: expired proposals are supposed to have their deposits returned to the submitter, but if an expired proposal is instead enacted, the deposit is consumed rather than refunded: [6](#0-5) 

---

### Likelihood Explanation

The trigger requires a delaying action to be enacted in the same epoch window as a proposal that is close to expiry. Delaying actions (`NewConstitution`, `UpdateCommittee`, `HardForkInitiation`, `NoConfidence`) are normal governance operations. Any DRep, SPO, or CC member can vote on proposals; no privileged access is required. The scenario is realistic in any active governance period where multiple proposals are live simultaneously. The codebase's own disabled conformance tests confirm the bug is reachable and reproducible.

---

### Recommendation

Add an explicit expiry guard as a precondition for ratification in `ratifyTransition`. The `then` branch should only be entered when `gasExpiresAfter >= reCurrentEpoch`:

```haskell
if gasExpiresAfter >= reCurrentEpoch   -- <-- add this guard
  && prevActionAsExpected gas ensPrevGovActionIds
  && validCommitteeTerm govAction ensCurPParams reCurrentEpoch
  && not rsDelayed
  && withdrawalCanWithdraw govAction ensTreasury
  && acceptedByEveryone env st gas
  then do
    -- enact
  else do
    st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
    if gasExpiresAfter < reCurrentEpoch
      then pure $ st' & rsExpiredL %~ Set.insert gasId
      else pure st'
```

This ensures that an expired proposal is always routed to the expiry path, regardless of its vote tally.

---

### Proof of Concept

The existing disabled conformance test in `RatifySpec.hs` directly demonstrates the issue. With `ppGovActionLifetimeL = EpochInterval 2`, four chained constitution proposals (`gai0`–`gai3`) are submitted and all voted `Yes`. Each enacted constitution is a delaying action. After `gai0` is enacted at epoch boundary +2, `gai1` is enacted at +3, `gai2` at +4. At this point `gai3` has `gasExpiresAfter = submissionEpoch + 2`, which is now `< reCurrentEpoch`. The test expects `gai3` to be expired and not enacted, but the current implementation enacts it because `acceptedByEveryone` is `True` and the expiry check is never reached in the `then` branch: [7](#0-6) 

The same pattern applies to `TreasuryWithdrawals`: if a treasury withdrawal proposal is delayed past its `gasExpiresAfter` epoch by a sequence of delaying actions but retains sufficient votes, it will be enacted and ADA will be transferred from the treasury after the intended expiry window.

returnProposalDeposits ::
  (Foldable f, EraAccounts era) =>
  f (GovActionState era) ->
  Accounts era ->

returnProposalDeposits removedProposals oldAccounts =
  foldr' processProposal (oldAccounts, mempty) removedProposals
  where
    processProposal gas (!accounts, !unclaimed)
      | (Just _accountState, newAccounts) <- updateLookupAccountState addRefund cred accounts =
          (newAccounts, unclaimed)
      | otherwise = (accounts, Map.insert (gasId gas) (gasDeposit gas) unclaimed)
      where
        addRefund = balanceAccountStateL <>~ compactCoinOrError (gasDeposit gas)
        cred = gasReturnAddr gas ^. accountAddressCredentialL
