### Title
Forged Foreign-Chain Verification via Missing Payload-Hash Integrity Binding in `respond_verify_foreign_tx` - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that a submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but never checks that `response.payload_hash` is actually derived from the pending `request`. A Byzantine participant below the signing threshold can replay any previously published `(payload_hash, signature)` pair from an old foreign-tx response to resolve a completely different pending request, delivering a forged verification result to the waiting caller.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs two checks:

1. The ECDSA signature in `response` is valid over `response.payload_hash` against the root public key.
2. The `request` argument exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

Critically, it never recomputes the expected `payload_hash` from the pending `request` and compares it to `response.payload_hash`. The `payload_hash` is defined as:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

It encodes both the original `ForeignChainRpcRequest` **and** the extracted on-chain values (e.g., the Bitcoin block hash). The contract never verifies that `response.payload_hash` is consistent with the `request` being resolved.

By contrast, the off-chain SDK helper `ForeignChainSignatureVerifier::verify_signature` does perform this binding check: [3](#0-2) 

But this check is only available to callers who use the SDK; the on-chain contract itself does not enforce it.

**Attack path:**

1. A user submits `verify_foreign_transaction(request_A)` — e.g., verify Bitcoin `tx_id_X` with `BlockHash` extractor. The request is stored in `pending_verify_foreign_tx_requests`.
2. At some earlier point, the MPC network legitimately processed `request_B` (a different transaction). The resulting `(payload_hash_B, sig_over_B)` was published on-chain in a prior `respond_verify_foreign_tx` call and is publicly visible.
3. A Byzantine participant (a registered, attested MPC node) calls `respond_verify_foreign_tx(request_A, { payload_hash: payload_hash_B, signature: sig_over_B })`.
4. The contract checks: is `sig_over_B` valid over `payload_hash_B` using the root key? **Yes.** Does `request_A` exist in `pending_verify_foreign_tx_requests`? **Yes.** Both checks pass.
5. The contract resolves `request_A` with the response containing `payload_hash_B` — data that corresponds to `request_B`'s foreign chain state, not `request_A`'s.
6. The caller of `request_A` (e.g., a bridge contract) receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes the wrong transaction's extracted values.

The `assert_caller_is_attested_participant_and_protocol_active` guard requires the attacker to be a registered participant, but does not require threshold cooperation — a single Byzantine node can execute this unilaterally. [4](#0-3) 

### Impact Explanation

A bridge contract that calls `verify_foreign_transaction` and trusts the contract's resolution to have validated the response (rather than independently re-running `ForeignChainSignatureVerifier::verify_signature`) will receive a `VerifyForeignTransactionResponse` whose `payload_hash` encodes a different transaction's extracted values. If the bridge uses the returned `payload_hash` to gate fund releases or cross-chain state transitions, it will act on forged foreign-chain data — enabling invalid bridge execution or double-spend conditions.

This matches the allowed High impact: **forged foreign-chain verification that causes invalid bridge execution**.

### Likelihood Explanation

The attacker must be a registered, attested MPC participant. Once that condition is met, the attack requires only:
- Observing any previously published `(payload_hash, signature)` pair from on-chain history (fully public).
- Submitting a single `respond_verify_foreign_tx` transaction with the replayed pair against a different pending request.

No threshold cooperation, no key material, and no off-chain coordination are required. The attack is executable by a single Byzantine node.

### Recommendation

Inside `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` argument and verify that `response.payload_hash` matches it. Because the `values` (extracted on-chain data) are not stored in the contract, the contract cannot recompute the full hash itself. The correct fix is to bind the `payload_hash` to the request at the time of signing and verify the binding on-chain. Concretely:

- Store a commitment to the expected `payload_hash` in `pending_verify_foreign_tx_requests` at request submission time (once the MPC nodes have agreed on the extracted values), **or**
- Require the responding node to include the full `ForeignTxSignPayload` (request + values) in the response, recompute the hash on-chain, and verify it matches `response.payload_hash` before accepting the signature.

This mirrors the fix the external report describes: adding authenticated integrity verification so that the signed data cannot be substituted without detection.
