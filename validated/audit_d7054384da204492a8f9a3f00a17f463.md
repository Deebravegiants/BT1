### Title
Caller-Agnostic Request Key in `verify_foreign_transaction` Allows Any Unprivileged Caller to Saturate the Fan-Out Queue and Block Legitimate Bridge Verification Requests — (File: `crates/contract/src/lib.rs`, `crates/contract/src/pending_requests.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint uses a request key (`VerifyForeignTransactionRequest`) that does **not** include the caller's account ID, unlike `sign()` and `request_app_private_key()` which both bind the request key to the caller's `predecessor_account_id`. Because the fan-out queue is shared across all callers for the same foreign-chain request, any unprivileged attacker can saturate the 128-slot queue for a targeted foreign-tx request at a cost of 128 yoctonear per MPC response cycle (effectively free), permanently blocking legitimate callers from queuing their verification requests for that transaction.

---

### Finding Description

**Root cause — asymmetric caller binding across request types:**

`SignatureRequest::new()` explicitly binds the request key to the caller:

```rust
// crates/contract/src/lib.rs ~line 379
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,        // ← caller's account ID is part of the key
    &request.path,
);
```

`VerifyForeignTransactionRequest` has no such binding:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs ~line 124
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,   // chain + tx_id
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    // ← no caller field
}
```

The contract itself confirms this is intentional fan-out design — the test comment at line 3242 reads: *"a different account would today be blocked from receiving a response by alice's submission"* — but the fan-out cap creates the exploitable surface.

**Fan-out cap enforcement in `push_pending_yield`:**

```rust
// crates/contract/src/pending_requests.rs lines 50-59
let queue = requests.entry(request).or_default();
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }
            .to_string(),
    );
}
queue.push(YieldIndex { data_id });
```

`MAX_PENDING_REQUEST_FAN_OUT = 128`. Once the queue is full, every subsequent `verify_foreign_transaction` call for the same request panics with `PendingRequestQueueFull`.

**Attack path:**

1. Attacker observes a target Bitcoin/Ethereum/Solana tx_id that a bridge user needs to verify (all foreign-chain transactions are public).
2. Attacker submits 128 `verify_foreign_transaction` calls for that exact `(tx_id, domain_id, payload_version)` tuple, each with 1 yoctonear deposit. Total cost: 128 yoctonear ≈ $0.
3. Queue is saturated. Any subsequent call from the legitimate user for the same request is rejected with `PendingRequestQueueFull`.
4. MPC nodes respond → `resolve_yields_for` drains the queue → attacker immediately resubmits 128 calls. The cycle repeats indefinitely.

The attacker's 128 slots also receive the MPC response (they are not wasted from the protocol's perspective), but the legitimate user is permanently excluded from the queue as long as the attacker maintains the attack.

---

### Impact Explanation

In the `verify_foreign_transaction` bridge flow, a user sends funds to a bridge address on a foreign chain (controlled by the MPC-derived key) and then calls `verify_foreign_transaction` to obtain the MPC signature needed to unlock equivalent funds on NEAR. If the attacker continuously saturates the queue for the user's specific tx_id:

- The user's `verify_foreign_transaction` calls are permanently rejected.
- The user cannot obtain the MPC signature.
- The user's funds remain locked in the MPC-controlled bridge address on the foreign chain with no self-service recovery path.

This constitutes a **request-lifecycle manipulation that breaks the bridge's safety accounting invariant** (a verified foreign transaction must eventually be resolvable by its submitter) without requiring any privileged access, collusion, or network-level DoS. The attacker's cost is ~128 yoctonear per MPC response cycle — effectively zero.

**Severity: Medium** (request-lifecycle/balance manipulation breaking production safety invariants, reachable by an unprivileged caller).

---

### Likelihood Explanation

- Foreign-chain transactions are public; the attacker can observe any target tx_id.
- The attack requires no privileged access — only a NEAR account with 128 yoctonear.
- The attack is self-sustaining: the attacker's queued yields receive the MPC response, so the attacker can immediately requeue after each drain.
- The only friction is the 1 yoctonear deposit per call, which is refunded if excess is attached, making the net cost negligible.

---

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, mirroring the design of `SignatureRequest`. This gives each caller an independent queue slot so no external party can fill another user's queue:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    pub caller: AccountId,   // ← bind to predecessor_account_id
}
```

Alternatively, if caller-agnostic fan-out is intentional (so multiple bridge contracts sharing the same tx_id all receive one MPC response), add a per-caller sub-queue or a per-caller slot reservation so a single external account cannot exhaust the entire 128-slot budget for a given request.

### Proof of Concept

```
// Attacker script (pseudocode, cost: 128 yoctonear per cycle)
let target_request = VerifyForeignTransactionRequestArgs {
    domain_id: <victim's domain>,
    payload_version: V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: <victim's observed tx_id>,
        confirmations: 2,
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};

loop {
    // Submit 128 identical requests from attacker account
    for _ in 0..128 {
        contract.verify_foreign_transaction(target_request.clone())
            .deposit(1 yoctonear)
            .call();
    }
    // Queue is now full (PendingRequestQueueFull for any new caller)
    // Wait for MPC to respond and drain the queue, then repeat
    wait_for_mpc_response();
}

// Victim's call is permanently rejected:
// victim.verify_foreign_transaction(target_request) → PendingRequestQueueFull
```

The test at `crates/contract/src/lib.rs` lines 3209–3263 confirms that `verify_foreign_transaction` queues callers under a single caller-agnostic key, and `crates/contract/src/lib.rs` lines 3300–3344 confirms that exceeding `MAX_PENDING_REQUEST_FAN_OUT` panics with `PendingRequestQueueFull`.
