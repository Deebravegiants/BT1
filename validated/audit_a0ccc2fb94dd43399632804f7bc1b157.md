### Title
Epoch-boundary shard-layout mismatch in `process_tx_internal` silently drops congestion check, admitting transactions to fully-congested receiver shards — (`chain/client/src/rpc_handler.rs`)

### Summary

When the chain head is the **last block of epoch N** and a resharding takes effect at the N→N+1 boundary, `process_tx_internal` resolves `receiver_shard` using epoch N+1's shard layout (child shard IDs) but looks up congestion info in the current block's `block_congestion_info`, which is keyed by epoch N's shard IDs (parent shard IDs). The child shard ID is absent from the epoch-N map, so the lookup returns `None`. `validate_tx` treats `None` as "no congestion data available" and skips the congestion gate entirely, admitting the transaction to the mempool unconditionally.

### Finding Description

**Step 1 — epoch ID resolution.**
`process_tx_internal` calls `get_epoch_id_from_prev_block(&head.last_block_hash)`. When `head.last_block_hash` is the last block of epoch N, this returns epoch N+1. [1](#0-0) 

**Step 2 — shard layout for epoch N+1.**
The code then fetches the shard layout for epoch N+1 (which contains the child shards produced by resharding) and maps `receiver_id` to a child shard ID that does not exist in epoch N. [2](#0-1) 

**Step 3 — `block_congestion_info` is keyed by epoch N shard IDs.**
`cur_block.block_congestion_info()` iterates the block's chunk headers and inserts entries keyed by each chunk's `shard_id()`. Because `cur_block` is in epoch N, every key is an epoch-N (parent) shard ID. [3](#0-2) 

**Step 4 — lookup returns `None`.**
The child shard ID (epoch N+1) is not present in the epoch-N map, so `.get(&receiver_shard)` returns `None`. [4](#0-3) 

**Step 5 — `validate_tx` skips the congestion gate when `receiver_congestion_info` is `None`.**
The entire `if let Some(congestion_info) = receiver_congestion_info { … }` block is bypassed. The function falls through directly to `validate_transaction`, which performs only signature/nonce/balance checks — no congestion check. [5](#0-4) 

### Impact Explanation

A transaction whose `receiver_id` maps to a child shard in epoch N+1 is admitted to the mempool (and returned as `ProcessTxResponse::ValidTx`) even when the parent shard — and therefore the child shard — is fully congested (`congestion_level == 1.0`). The invariant that every RPC-submitted transaction must pass the receiver-shard congestion gate before mempool admission is violated for the duration of the last block of epoch N whenever a resharding is active.

### Likelihood Explanation

The trigger window is narrow — exactly one block per resharding event — but the condition is deterministic and publicly observable (epoch boundaries are predictable from block height and epoch length). An attacker monitoring the chain can time a burst of transactions to the congested child-shard account during that single block. The resharding itself is a normal protocol event (static V1→V2 or dynamic V3), not an adversarial precondition.

### Recommendation

In `process_tx_internal`, after computing `receiver_shard` from the epoch N+1 layout, check whether the current block is at a resharding boundary. If it is, resolve the parent shard ID in epoch N's layout and use the parent's congestion entry for the admission check. Alternatively, always look up congestion info using the **current block's epoch** shard layout (epoch N) for the congestion map key, and separately use the next-epoch layout only for routing/forwarding decisions.

### Proof of Concept

1. Configure a test-loop environment with a static V1→V2 resharding scheduled at epoch boundary E.
2. Congest the parent shard during epoch N (e.g., by dropping chunks or flooding receipts until `missed_chunks_count >= max_congestion_missed_chunks`).
3. Advance the chain to the last block of epoch N (i.e., `is_next_block_epoch_start(head.last_block_hash) == true`).
4. Submit a `SignedTransaction` whose `receiver_id` maps to a child shard in epoch N+1's layout.
5. Assert `process_tx_internal` returns `ProcessTxResponse::ValidTx` instead of `ProcessTxResponse::InvalidTx(InvalidTxError::ShardCongested { … })` or `ShardStuck { … }`.

The assertion passes because at line 200 of `rpc_handler.rs`, `cur_block.block_congestion_info().get(&child_shard_id)` returns `None`, and `validate_tx` at lines 724–747 of `runtime/mod.rs` skips the congestion branch entirely. [6](#0-5) [7](#0-6)
