Based on my research, I found a concrete analog in nearcore to the MarmoStork vulnerability class (size/type validation that happens too late or after the data is already used to construct something critical).

### Title
Receipt Size Validation Bypass via `promise_return` Appending `output_data_receivers` After Size Check — (`runtime/runtime/src/verifier.rs`)

### Summary

The `validate_receipt` function enforces `max_receipt_size` only at receipt creation time (`NewReceipt` mode). However, when a contract calls `promise_return`, the runtime appends `DataReceiver` entries to the `output_data_receivers` field of the returned receipt **after** the size check has already passed. A receipt crafted at exactly `max_receipt_size` grows beyond the limit once `output_data_receivers` are appended, and is never re-validated. The oversized receipt enters protocol state and is forwarded between shards.

### Finding Description

In `runtime/runtime/src/verifier.rs`, `validate_receipt` checks the serialized receipt size only when `mode == ValidateReceiptMode::NewReceipt`:

```rust
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 = borsh::object_length(receipt)...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [1](#0-0) 

The `ValidateReceiptMode::ExistingReceipt` variant explicitly skips this check, with the comment:

> "There is a bug which allows to create receipts that are above the size limit. Runtime has to handle them gracefully until the receipt size limit bug is fixed. See https://github.com/near/nearcore/issues/12606" [2](#0-1) 

The attack path is:

1. A contract creates promise A whose receipt is sized at exactly `max_receipt_size` (4,194,304 bytes). The receipt passes `NewReceipt` validation.
2. The contract creates promise B (a callback) and calls `promise_return(A)`.
3. The runtime resolves the promise DAG and appends B's `DataReceiver` to A's `output_data_receivers` field — **after** the size check.
4. A's receipt now exceeds `max_receipt_size` but is never re-validated.
5. The oversized receipt is stored in the `OutgoingReceipts` DB column and forwarded to the target shard.

This is confirmed by the test `test_max_receipt_size_promise_return`:

> "A function call will generate a new receipt. Size of this receipt will be equal to `max_receipt_size`, it'll pass validation, but then `output_data_receivers` will be modified and the receipt's size will go above max_receipt_size. The receipt should be rejected, but currently isn't because of a bug." [3](#0-2) 

A second path exists via `value_return`: returning a value of size `max_receipt_size` produces a `DataReceipt` whose total serialized size (value + envelope overhead) exceeds `max_receipt_size`, also bypassing the limit. [4](#0-3) 

The congestion control layer has a workaround that caps the reported size at `max_receipt_size` to prevent receipts from getting stuck in the outgoing buffer:

```rust
// There is a bug which allows to create receipts that are above the size limit.
// Let's pretend that all receipts are at most `max_receipt_size`...
if size > max_receipt_size {
    size = max_receipt_size;
}
``` [5](#0-4) 

This workaround means `own_congestion_info.remove_receipt_bytes(size)` uses the capped value, not the actual receipt size, corrupting the congestion accounting.

### Impact Explanation

**Corrupted protocol value**: Receipts stored in the `OutgoingReceipts` DB column that exceed `max_receipt_size` — a direct violation of the protocol invariant enforced by `validate_receipt`. The congestion control's `own_congestion_info` byte-tracking is also corrupted (underestimates actual bytes forwarded), allowing more data to flow through shards than the congestion control intends to permit.

### Likelihood Explanation

Any unprivileged user who can deploy a contract (standard NEAR account with sufficient balance) can trigger this. The contract only needs to use `promise_return` with a near-maximum-size receipt. The pattern is straightforward and the test contract (`near_test_contracts::rs_contract`) already implements the exact method (`max_receipt_size_promise_return_method1`).

### Recommendation

Re-validate the receipt size **after** `output_data_receivers` are appended during promise DAG resolution, before the receipt is committed to the outgoing buffer. Alternatively, enforce the size limit at the point where `output_data_receivers` are added, rejecting the modification if it would push the receipt over `max_receipt_size`. The analog fix from MarmoStork applies directly: validate the complete, final form of the data structure rather than an intermediate form.

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete, runnable proof of concept. It:
1. Deploys a contract.
2. Calls `max_receipt_size_promise_return_method1` which creates a receipt at exactly `max_receipt_size` and uses `promise_return`.
3. Asserts via `assert_oversized_receipt_occurred` that a receipt exceeding `max_receipt_size` was observed in the chain state. [6](#0-5)

    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]

    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {

        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),

            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();

    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),

        "".into(),
