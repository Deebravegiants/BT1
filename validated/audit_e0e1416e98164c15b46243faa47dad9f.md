### Title
Dual Unrestricted Mock Attestation Bypass Allows Participants to Circumvent TEE Verification in Production — (`crates/mpc-attestation/src/attestation.rs`, `crates/contract/src/lib.rs`, `crates/contract/src/tee/tee_state.rs`)

### Summary

The NEAR MPC contract contains two independent, production-accessible mechanisms that bypass the TEE attestation requirement: (1) any participant can submit `Attestation::Mock(MockAttestation::Valid)` to `submit_participant_info` at any time, which unconditionally passes all verification checks without hardware evidence; and (2) the `init()` function pre-populates every initial participant with a `VerifiedAttestation::Mock(MockAttestation::Valid)` entry via `TeeState::with_mocked_participant_attestations()`, meaning initial participants are considered "attested" from genesis without ever submitting real Dstack attestations. Neither mechanism is gated by a feature flag, environment check, or allowlist. Together they make the TEE attestation requirement effectively optional for any participant, undermining the security model that protects key shares from host OS extraction.

### Finding Description

**Mechanism 1 — Runtime parameter bypass:**

`submit_participant_info` in `crates/contract/src/lib.rs` accepts a caller-supplied `dtos::Attestation` that includes the `Mock(MockAttestation::Valid)` variant:

```rust
pub fn submit_participant_info(
    &mut self,
    proposed_participant_attestation: dtos::Attestation,
    tls_public_key: dtos::Ed25519PublicKey,
) -> Result<(), Error> {
``` [1](#0-0) 

`MockAttestation::Valid` is defined to unconditionally return `Ok(())` from `verify_constraints`:

```rust
MockAttestation::Valid => Ok(()),
``` [2](#0-1) [3](#0-2) 

There is no `#[cfg(test)]` guard, no feature flag, no allowlist check, and no environment restriction on this variant. Any NEAR account can call `submit_participant_info` with `{ "Mock": "Valid" }` and be stored in `tee_state.stored_attestations` as a fully attested node — confirmed by the e2e cluster initialization code that uses exactly this JSON in what is intended to be a test-only path but is structurally identical to the production call: [4](#0-3) 

**Mechanism 2 — Initialization-time bypass:**

`init()` calls `TeeState::with_mocked_participant_attestations()` for every initial participant:

```rust
let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);
``` [5](#0-4) 

This inserts `VerifiedAttestation::Mock(MockAttestation::Valid)` for every participant without any real attestation submission: [6](#0-5) 

The contract also initializes `accept_requests: true`, meaning signing is open from genesis with only mock attestations backing the TEE guarantee: [7](#0-6) 

**The dual-mechanism inconsistency:**

Mechanism 2 means initial participants never need to submit real attestations — they already hold mock ones. Mechanism 1 means any participant can replace a real attestation with a mock one at any time. The `verify_tee()` function, which controls `accept_requests`, calls `reverify_and_cleanup_participants()`, which re-verifies stored `VerifiedAttestation::Mock(MockAttestation::Valid)` entries and always passes them — so mock-attested participants are never kicked out: [8](#0-7) 

The result: the TEE attestation requirement is structurally optional. A Byzantine participant below the signing threshold can operate without TEE hardware, expose key shares to the host OS, and remain in the active participant set indefinitely.

### Impact Explanation

The TEE attestation system is the contract's mechanism for ensuring that MPC key shares are held inside hardware-isolated enclaves (Intel TDX CVMs). A participant who bypasses attestation via `MockAttestation::Valid` can run the MPC node software outside a TEE, where the host OS has full memory visibility. This exposes the participant's key share to extraction by the host. If a Byzantine participant below the signing threshold exploits this, they can extract their key share without triggering threshold-level collusion detection. Combined with other participants' shares (obtained through separate compromise), this enables unauthorized key reconstruction and forgery of threshold signatures.

This maps to the allowed impact: **Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery.**

### Likelihood Explanation

The attack requires no privileged access beyond being a governance-approved participant. The `MockAttestation::Valid` variant is a first-class, serializable, production-reachable enum variant with no runtime restriction. The JSON encoding `{ "Mock": "Valid" }` is documented in e2e test code and trivially discoverable from the ABI. A Byzantine participant can submit it on every attestation renewal cycle, permanently maintaining mock-attested status. Likelihood is **High** for any participant who wishes to avoid running in a TEE.

### Recommendation

1. **Restrict `MockAttestation` to non-production builds.** Gate the `Mock` variant behind a compile-time feature flag (e.g., `#[cfg(feature = "mock-attestation")]`) that is excluded from production WASM builds. Fail the build if the feature is enabled in a release profile.

2. **Remove `with_mocked_participant_attestations()` from the production `init()` path.** Require participants to submit real Dstack attestations before the contract transitions to `Running` state and sets `accept_requests = true`. Track this under the existing TODO(#1087).

3. **Add a mutual-exclusivity assertion.** If both mechanisms must coexist during a transition period, add a runtime assertion in `submit_participant_info` that rejects `MockAttestation` when `allowed_docker_image_hashes` is non-empty (i.e., when the contract is in production TEE mode).

4. **Unify the bypass control.** Mirror the external report's recommendation: centralize the bypass decision behind a single, auditable flag rather than two independent code paths.

### Proof of Concept

1. Deploy the contract with `init()` — all participants receive `VerifiedAttestation::Mock(MockAttestation::Valid)` automatically. `accept_requests = true`. No real TEE hardware required.

2. A Byzantine participant (below threshold) calls:
   ```json
   submit_participant_info({
     "proposed_participant_attestation": { "Mock": "Valid" },
     "tls_public_key": "<their_tls_key>"
   })
   ```
   This succeeds unconditionally. The participant is stored as attested.

3. The participant runs the MPC node binary outside a TEE. The host OS can read process memory and extract the key share.

4. `verify_tee()` is called by any participant. `reverify_and_cleanup_participants()` re-verifies the stored `Mock(Valid)` entry, which always passes. The Byzantine participant is never kicked out. `accept_requests` remains `true`. Signing continues.

5. The two bypass mechanisms — init-time mock population (Mechanism 2) and runtime mock submission (Mechanism 1) — operate independently and reinforce each other, making the TEE requirement permanently bypassable with no on-chain evidence of the bypass. [2](#0-1) [9](#0-8) [10](#0-9) [11](#0-10)

impl TeeState {
    /// Creates a [`TeeState`] with an initial set of participants that will receive a valid mocked attestation.
    pub(crate) fn with_mocked_participant_attestations(participants: &Participants) -> Self {
        let mut tee_state = Self::default();

        for (account_id, _, participant_info) in participants.participants() {
            let tls_public_key = participant_info.tls_public_key.clone();
            // TODO(#1087): replace account_public_key with a real account public
            // key passed in by the caller. `Participants` does not currently
            // carry the operator's account public key, so a mocked entry
            // cannot record the real one and we use the TLS key as a unique
            // per-participant placeholder. The mock keeps the
            // participant from being kicked out of an empty `TeeState` until
            // a real `submit_participant_info` call replaces it (keyed by
            // TLS), but any caller-facing check that compares
            // `signer_account_pk` against the stored key will fail until
            // then. #1087 tracks threading real attestations through
            // initialization so this sentinel can go away.
            let node_id = NodeId {
                account_id: account_id.clone(),
                tls_public_key: tls_public_key.clone(),
                // Use tls_public_key as account_public_key instead of hardcoded
                // Ed25519PublicKey::from([0u8; 32]) so that same account public
                // key isn't associated with different tls keys.
                // This is not a fix for above issue: #1087, which should be
                // addressed outside this PR.
                account_public_key: tls_public_key.clone(),
            };

            tee_state.stored_attestations.insert(
                tls_public_key,
                NodeAttestation {
                    node_id,
                    verified_attestation: VerifiedAttestation::Mock(
                        attestation::MockAttestation::Valid,
                    ),
                },
            );
        }

        tee_state
    }
