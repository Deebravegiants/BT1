### Title
Caller-Controlled `confirmations=0` Bypasses Bitcoin Finality Check in `verify_foreign_transaction` ‚ÄĒ (`crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint accepts a `BitcoinRpcRequest` with `confirmations: 0`. Because `BlockConfirmations` has no minimum enforced at the contract level, and the `BitcoinInspector` approves any transaction when the threshold is 0 (`0 ‚Č§ actual_confirmations` is always true for a `u64`), an unprivileged caller can obtain a valid MPC threshold signature over an unconfirmed or minimally-confirmed Bitcoin transaction. A downstream bridge contract that trusts the MPC network to enforce finality ‚ÄĒ without independently re-checking the `confirmations` field ‚ÄĒ will accept this signature as proof of settlement, enabling a double-spend.

---

### Finding Description

`BlockConfirmations` is a plain `u64` newtype with no lower bound:

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI schema confirms `"minimum": 0.0` ‚ÄĒ zero is explicitly valid. [1](#0-0) [2](#0-1) 

The contract's `verify_foreign_transaction` performs no minimum-confirmation validation before enqueuing the request:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(...);   // only checks domain, gas, deposit
    // no check on request.request.confirmations
    ...
    self.enqueue_yield_request(...);
}
``` [3](#0-2) 

Inside the MPC node, `BitcoinInspector::extract` enforces the threshold with a `<=` comparison:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
``` [4](#0-3) 

When `block_confirmations_threshold = 0`, the condition `0 ‚Č§ actual_confirmations` is trivially true for every `u64`, so the guard is completely bypassed. The node proceeds to sign the transaction regardless of its actual confirmation depth.

The signed payload (`ForeignTxSignPayloadV1`) Borsh-serializes the full `ForeignChainRpcRequest` ‚ÄĒ including the caller-supplied `confirmations: 0` ‚ÄĒ into the hash:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,   // contains confirmations=0
    pub values: Vec<ExtractedValue>,
}
impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
``` [5](#0-4) 

The resulting signature is cryptographically valid and indistinguishable from one produced over a properly-confirmed transaction. A bridge contract that reconstructs the payload hash from the caller-supplied request parameters and verifies the MPC signature ‚ÄĒ without separately asserting `confirmations ‚Č• minimum` ‚ÄĒ will accept it as proof of finality.

---

### Impact Explanation

An attacker can:

1. Broadcast a Bitcoin transaction.
2. Immediately call `verify_foreign_transaction` with `confirmations: 0` (cost: 1 yoctoNEAR).
3. Receive a valid MPC threshold signature over the unconfirmed transaction.
4. Submit the signature to a bridge contract to claim bridged assets.
5. Double-spend the original Bitcoin transaction via a reorg or RBF replacement.

This is a **light-client-style verification bypass** that directly enables **double-spend conditions** on any bridge that delegates finality enforcement to the MPC network ‚ÄĒ matching the High impact category.

---

### Likelihood Explanation

The attack requires no special privileges. Any NEAR account can call `verify_foreign_transaction` with an arbitrary `confirmations` value. The only barrier is a 1 yoctoNEAR deposit. The `BlockConfirmations` type accepts 0 at both the JSON and Borsh layers, and no contract-level guard exists. The attack is deterministic and repeatable.

---

### Recommendation

Enforce a per-chain minimum confirmation floor at the contract level. Two complementary approaches:

1. **Contract-level guard**: Reject `BitcoinRpcRequest` with `confirmations < MINIMUM_BITCOIN_CONFIRMATIONS` (e.g., 1 or 6) inside `verify_foreign_transaction` before enqueuing.
2. **Chain metadata**: Store a `min_confirmations` field in `ForeignChainsMetadata` per chain and validate the caller-supplied value against it during request acceptance.

The fix is analogous to the Symmio fix: verify that the caller-supplied safety parameter meets a minimum acceptable value before the request is processed.

---

### Proof of Concept

```json
{
  "request": {
    "domain_id": 1,
    "payload_version": 1,
    "request": {
      "Bitcoin": {
        "tx_id": "bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    }
  }
}
```

1. Submit the above to `verify_foreign_transaction` with 1 yoctoNEAR attached.
2. The contract enqueues the request ‚ÄĒ no minimum-confirmation check fires. [6](#0-5) 
3. Each MPC node calls `BitcoinInspector::extract(tx_id, threshold=0, extractors)`.
4. `0 ‚Č§ actual_confirmations` passes unconditionally; the node extracts the block hash and signs. [7](#0-6) 
5. The caller receives a valid MPC signature over `SHA256(borsh(ForeignTxSignPayloadV1 { request: {tx_id, confirmations=0, ...}, values: [block_hash] }))`.
6. A bridge contract that verifies this signature without asserting `confirmations ‚Č• minimum` accepts the unconfirmed transaction as final, enabling a double-spend.
