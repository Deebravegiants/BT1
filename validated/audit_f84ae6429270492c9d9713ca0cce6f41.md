Audit Report

## Title
FIFO Withdrawal Queue Permanently Stalled by Large Front-of-Queue Request — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`initiateWithdrawal` gates admission using `getAvailableAssetAmount`, which counts all protocol assets including EigenLayer-staked funds, while `unlockQueue` supplies only `unstakingVault.balanceOf(asset)` as available liquidity. Because `_unlockWithdrawalRequests` unconditionally `break`s on the first request it cannot cover, any user can place a large request at the head of the FIFO queue that the vault cannot immediately satisfy, freezing every subsequent user's withdrawal until the vault accumulates sufficient assets.

## Finding Description
`getAvailableAssetAmount` at line 599–603 computes capacity against `lrtDepositPool.getTotalAssetDeposits(asset)`, which aggregates assets across the deposit pool, all NodeDelegators, EigenLayer strategies (staked + unstaking), the converter, and the unstaking vault:

```solidity
// LRTDepositPool.sol L394-396
uint256 effectiveAssetWithEigenLayer = assetStakedInEigenLayer + assetUnstakingFromEigenLayer;
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer + assetLyingInConverter
        + assetLyingUnstakingVault);
```

This allows `initiateWithdrawal` (line 170) to accept a request sized against the full protocol TVL.

`_createUnlockParams` (line 846–850) then supplies only `unstakingVault.balanceOf(asset)` as `totalAvailableAssets`:

```solidity
totalAvailableAssets: unstakingVault.balanceOf(asset)
```

`_unlockWithdrawalRequests` (line 800) hard-breaks on the first request it cannot cover:

```solidity
if (availableAssetAmount < payoutAmount) break;
```

Because `nextLockedNonce[asset]` only advances when a request is successfully unlocked (line 815), any request that cannot be covered freezes every request behind it. `_processWithdrawalCompletion` (line 707) then reverts with `WithdrawalLocked` for all users whose nonce is `>= nextLockedNonce[asset]`.

## Impact Explanation
**Medium — Temporary freezing of funds.** Victims' rsETH is transferred into the contract at `initiateWithdrawal` time and cannot be reclaimed. Their withdrawal nonces remain `>= nextLockedNonce[asset]` until the vault accumulates enough assets to cover the blocker, which may require multiple EigenLayer unstaking cycles (days to weeks per cycle). The attacker can repeat the attack each time the queue clears, making the freeze effectively continuous at low cost.

## Likelihood Explanation
Any rsETH holder can call `initiateWithdrawal`. The gap between `getTotalAssetDeposits` and `unstakingVault.balanceOf` is structural and large in normal operation — the vast majority of protocol assets reside in EigenLayer strategies. The attacker does not lose funds; their rsETH is locked but eventually claimable. The only cost is opportunity cost and gas. No privileged access, oracle manipulation, or external protocol compromise is required.

## Recommendation
Replace the hard `break` at line 800 with a `continue` so that smaller requests behind an underfunded one can still be processed. Alternatively, cap `expectedAssetAmount` at admission time against `unstakingVault.balanceOf(asset)` rather than `getTotalAssetDeposits`, so only requests the vault can immediately service are admitted to the queue.

## Proof of Concept
**Setup:** Protocol holds 1,000 ETH total: 950 ETH in EigenLayer, 50 ETH in `LRTUnstakingVault`. `assetsCommitted[ETH] = 0`.

1. **Attacker calls `initiateWithdrawal`** with rsETH worth 500 ETH. `getAvailableAssetAmount` returns 1,000 ETH → check passes. `assetsCommitted[ETH]` becomes 500 ETH. Request enqueued at nonce 0.

2. **Victims call `initiateWithdrawal`** with rsETH worth 10 ETH each. `getAvailableAssetAmount` = 500 ETH → checks pass. Requests enqueued at nonces 1, 2, 3, …

3. **Operator calls `unlockQueue`.** `totalAvailableAssets = unstakingVault.balanceOf(ETH) = 50 ETH`. Loop iteration 0: attacker's `payoutAmount = 500 ETH`. `50 < 500` → `break`. `nextLockedNonce[ETH]` stays at 0.

4. **Victims call `completeWithdrawal`.** Their nonces (1, 2, 3, …) are all `>= nextLockedNonce[ETH]` (= 0) → `revert WithdrawalLocked()`. Funds are frozen until the vault accumulates 500 ETH.
