### Title
Unrestricted ADA Creation via Unaccounted `directDeposits` in Dijkstra Era Value Conservation Check - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

The Dijkstra era introduces a new `directDeposits` transaction body field that credits ADA directly to registered account balances. However, `directDeposits` is excluded from both sides of the preservation-of-value check (`consumed == produced`). Any unprivileged transaction submitter can include arbitrary `directDeposits` amounts in a transaction body, causing the ledger to add ADA to targeted account balances with no corresponding deduction from the transaction's UTxO inputs — effectively creating ADA from nothing.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — missing value-conservation accounting for a new ledger operation, analogous to the unrestricted asset-transfer access-control bypass in the reference report.

**New Dijkstra feature — `DirectDeposits`:**

`DirectDeposits` is a new Dijkstra-era transaction body field (CBOR key 25) that maps `AccountAddress → Coin`. When a transaction is processed, the `dijkstraEntitiesTransition` rule applies these deposits directly to account balances: [1](#0-0) 

The only validation performed on `directDeposits` is that the target accounts exist (`directDepositsMissingAccounts`) and that network IDs match (`validateWrongNetworkInDirectDeposit`). There is no authorization check and no requirement that the transaction pays for the deposited amounts.

**Root cause — missing from value conservation:**

The preservation-of-value check in `dijkstraUtxoTransition` calls `Shelley.validateValueNotConservedUTxO`, which compares `consumed` and `produced` values using the `EraUTxO DijkstraEra` instance: [2](#0-1) 

The `consumed` side is computed by `getConsumedDijkstraValue`, which delegates to `getConsumedMaryValue` (UTxO inputs + withdrawals + refunds) — `directDeposits` is absent: [3](#0-2) 

The `produced` side is computed by `dijkstraProducedValue`, which delegates to `conwayProducedValue` (outputs + fee + deposits + treasury donation) — `directDeposits` is also absent: [4](#0-3) 

Neither `getConsumedDijkstraValue` nor `dijkstraProducedValue` includes `directDeposits`. The `EraUTxO DijkstraEra` instance confirms these are the functions used: [5](#0-4) 

**No witness required either:**

`getWitsVKeyNeeded` for `DijkstraEra` delegates to `getConwayWitsVKeyNeeded`, which has no knowledge of `directDeposits`. No key witness or script witness is required to authorize a direct deposit to any account: [6](#0-5) 

**`applyDirectDeposits` unconditionally adds ADA to account balances:** [7](#0-6) 

The comment on this function explicitly states: *"There are no checks that direct deposits mention only registered accounts."* The caller (`dijkstraEntitiesTransition`) adds only the existence check, not a payment check.

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

A concrete example:

| Field | Value |
|---|---|
| UTxO inputs | 100 ADA |
| UTxO outputs | 99 ADA |
| Fee | 1 ADA |
| `directDeposits` | `{attacker_account → 1,000,000 ADA}` |

The value conservation check evaluates: `consumed (100) == produced (99 + 1)` → **passes**. The `ENTITIES` rule then calls `applyDirectDeposits`, crediting 1,000,000 ADA to the attacker's account balance. The attacker paid only 1 ADA in fees and received 1,000,000 ADA in their staking account — ADA created from nothing.

The attacker can then withdraw the inflated balance via a standard withdrawal transaction (which does require a witness for the staking credential, but the attacker controls that credential since they registered it). This results in direct, permanent loss of ledger integrity and constitutes an invalid state transition that all honest nodes would accept.

---

### Likelihood Explanation

**High.** The entry path requires only:
1. A registered staking account (trivially obtained by any user)
2. A valid Dijkstra-era transaction with a non-empty `directDeposits` field targeting that account
3. Sufficient ADA to pay the transaction fee

No privileged access, no governance majority, no leaked keys, and no external dependency is required. The attacker controls the entire attack path.

---

### Recommendation

Include `directDeposits` in the `produced` side of the value conservation calculation in `dijkstraProducedValue` (and `dijkstraSubTxProducedValue` for sub-transactions), so that the transaction must provide UTxO inputs sufficient to fund the deposits:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> inject (fold . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
    <> foldMap' (getProducedValue pp isRegPoolId . view bodyTxL)
                (txBody ^. subTransactionsTxBodyL)
```

Similarly update `getConsumedDijkstraValue` if the design intent is for direct deposits to be funded from a different source (e.g., treasury), and add the corresponding authorization check. Either way, `directDeposits` must appear on exactly one side of the `consumed == produced` equation.

### Proof of Concept

**Attacker-controlled entry path:**

1. Register a staking credential `cred` and obtain `accountAddress`.
2. Construct a Dijkstra-era transaction:
   - `inputs`: one UTxO output worth `fee` ADA (e.g., 1 ADA)
   - `outputs`: empty (or change output of 0)
   - `fee`: 1 ADA
   - `directDeposits`: `{ accountAddress → N }` for arbitrarily large `N`
3. Submit the transaction. `dijkstraUtxoTransition` runs `validateValueNotConservedUTxO`: `consumed = 1 ADA = produced = 0 + 1 ADA` → passes.
4. `dijkstraEntitiesTransition` runs `directDepositsMissingAccounts` → passes (account exists). Then calls `applyDirectDeposits`, crediting `N` ADA to `accountAddress`.
5. Submit a withdrawal transaction signed by `cred` to drain `N` ADA from the account into a UTxO output.

**Relevant code path:**

```
LEDGER → UTXOW → UTXO (validateValueNotConservedUTxO passes, directDeposits absent)
       → ENTITIES (applyDirectDeposits credits N ADA, no payment check)
