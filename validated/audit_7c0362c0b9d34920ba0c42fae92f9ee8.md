Audit Report

## Title
Pool Admin Can Instantly Set Uncapped Bin Additional Fees to Frontrun Swaps — (`metric-core/contracts/MetricOmmPoolFactory.sol` / `metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees()` forwards caller-supplied `addFeeBuyE6`/`addFeeSellE6` values to the pool with no upper-bound cap and no timelock, while the analogous `setPoolAdminFees()` enforces explicit caps. A pool admin can atomically spike per-bin fees to `type(uint16).max` (65 535 ≈ 6.55% in E6 units) and frontrun a pending user swap, causing the trader to pay up to 6.55% more input than expected, with the excess retained as LP balance inside the bin.

## Finding Description
`MetricOmmPoolFactory.setPoolBinAdditionalFees()` passes values directly to the pool with no validation: [1](#0-0) 

`MetricOmmPool.setBinAdditionalFees()` performs only a bin-index range check before writing: [2](#0-1) 

These values are added directly to `params.baseFeeX64` at swap execution time for all four swap directions: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

By contrast, `setPoolAdminFees()` enforces explicit caps before writing: [7](#0-6) 

No `maxAdminBinFeeE6` or equivalent constant exists anywhere in the codebase. A timelock (`proposePoolPriceProvider`/`executePoolPriceProvider`) exists for price-provider changes but not for bin additional fee changes, confirming the asymmetric treatment.

## Impact Explanation
Direct loss of user principal on every swap crossing a manipulated bin. The pool admin can extract up to `65535/1e6 ≈ 6.55%` of swap notional per bin traversed, retained as LP balance inside the bin. This is an admin-boundary break: the pool admin exceeds what the protocol's own cap pattern (enforced for spread and notional fees) implies should be bounded behavior, meeting the "Admin-boundary break: pool admin exceeds caps" criterion in the allowed impact gate. Severity: **High**.

## Likelihood Explanation
The pool admin is a semi-trusted role that acts unilaterally and atomically. No timelock, governance delay, or cap guard stands between the admin and the fee change. On any public mempool chain, the admin can observe a large pending swap and frontrun it in the same block. Even without malicious intent, a legitimate fee increase can accidentally frontrun a user whose transaction was already in flight. The attack is repeatable on every swap.

## Recommendation
Add a hard cap in `setPoolBinAdditionalFees()` analogous to the caps already enforced for pool-level admin fees:

```solidity
// MetricOmmPoolFactory.sol — setPoolBinAdditionalFees()
if (addFeeBuyE6  > maxAdminBinFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
```

Additionally, apply a two-step propose/execute timelock for bin additional fee changes, consistent with the pattern already used for `proposePoolPriceProvider`/`executePoolPriceProviderUpdate`.

## Proof of Concept
1. Pool is live; `curBinIdx = 0`; `addFeeBuyE6 = 0`.
2. User submits `swap(recipient, false, 1_000e18, priceLimitX64, ...)` (exact-input, buying token0 with token1).
3. Pool admin observes the pending transaction and frontruns with:
   ```solidity
   factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
   ```
4. User's swap executes with effective fee = `baseFee + 65535/1e6` (≈ baseFee + 6.55%), paying ~6.55% more token1 than expected for the same token0 output.
5. The extra token1 is retained in the bin as LP balance; the user has no recourse.

A Foundry integration test can confirm this by: (a) recording `token0` received before the fee spike, (b) replaying the same swap after `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`, and (c) asserting the output difference equals `~6.55%` of the input notional.
