### Title
Unelected Committee Member Votes Bypass Block-Level Enforcement Before Protocol Version 11 — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Mempool.hs`)

---

### Summary

Before protocol version 11, the restriction preventing unelected committee members from casting governance votes is enforced **only** in the `MEMPOOL` rule, not in the `GOV` rule invoked during block validation. A block producer can include a transaction containing votes from unelected committee members directly in a block, bypassing the mempool check entirely. These pre-votes are stored in `gasCommitteeVotes` and will count toward governance ratification once the member's election is enacted.

---

### Finding Description

The `hardforkConwayDisallowUnelectedCommitteeFromVoting` flag gates where the unelected-committee-member check is enforced:

**In `MEMPOOL` (`Mempool.hs` lines 123–135):**
```haskell
unless (hardforkConwayDisallowUnelectedCommitteeFromVoting protVer) $
  -- Disallow votes by unelected committee members
  failOnNonEmpty
    ( unelectedCommitteeVoters ... )
    (ConwayMempoolFailure . ...)
```
When PV ≤ 10, `unless False` = the check **IS** performed in `MEMPOOL`.

**In `GOV` (`Gov.hs` lines 478–481):**
```haskell
when (hardforkConwayDisallowUnelectedCommitteeFromVoting $ pp ^. ppProtocolVersionL) $
  failOnNonEmpty
    (unelectedCommitteeVoters committee committeeState gsVotingProcedures)
    (injectFailure . UnelectedCommitteeVoters)
```
When PV ≤ 10, `when False` = the check is **NOT** performed in `GOV`.

The flag is defined in `Era.hs` as:
```haskell
hardforkConwayDisallowUnelectedCommitteeFromVoting pv = pvMajor pv > natVersion @10
```

Block validation follows the path `BBODY → LEDGERS → LEDGER → GOV`, **not** through `MEMPOOL`. The `applyBlock` function in `Cardano.Ledger.Shelley.API.Validation` invokes `EraRule "BBODY"` directly:
```haskell
applySTSOptsResult @(EraRule "BBODY" era) opts $
  TRC (mkBbodyEnv newEpochState, bBodyState, wrapBlockSignal block)
```
`BBODY` calls `LEDGERS`, which calls `LEDGER`, which calls `GOV` — `MEMPOOL` is never invoked. Therefore, before PV 11, the unelected-committee-member check is entirely absent from the block validation path.

The `committeeAcceptedRatio` function in `Ratify.hs` counts votes by iterating over elected members (cold keys in `committeeMembers`), looking up their authorized hot key in `CommitteeState`, and then looking up that hot key's vote in `gasCommitteeVotes`. A pre-vote stored in `gasCommitteeVotes` before enactment will be counted once the member's cold key appears in the enacted committee.

---

### Impact Explanation

An unelected committee member — one whose election via `UpdateCommittee` has not yet been enacted (enactment takes two epochs) — can pre-vote on a live governance action by colluding with a block producer. Once their election is enacted, their cold key enters `committeeMembers`, and `committeeAcceptedRatio` counts their previously stored vote. This can tip the ratification threshold for `NewConstitution`, `HardForkInitiation`, `ParameterChange`, or `TreasuryWithdrawals` actions, constituting an unauthorized governance action enacted without the member being a legitimate active committee member at the time of voting.

This matches the allowed impact: **Critical — Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.**

---

### Likelihood Explanation

The attack requires only a single cooperating block producer (not a 51% attack or supermajority). Any stake pool operator selected to produce a block can include such a transaction. The window of opportunity is the two-epoch gap between a committee election being approved and being enacted — a predictable, recurring window during any committee transition. The test at `RatifySpec.hs` lines 1883–1906 explicitly documents that before PV 11, unelected CC members can vote and their votes count once elected, confirming this is a reachable code path.

---

### Recommendation

The unelected-committee-member check should be enforced unconditionally in the `GOV` rule regardless of protocol version, not gated behind `hardforkConwayDisallowUnelectedCommitteeFromVoting`. Alternatively, the check should be present in **both** `MEMPOOL` and `GOV` before PV 11 to close the block-producer bypass path. The current design creates a gap where the mempool-only restriction can be circumvented by any block producer.

---

### Proof of Concept

1. Submit an `UpdateCommittee` governance action adding a new CC member with cold key `ccCold`.
2. Authorize a hot key `ccHot` for `ccCold` via `AuthCommitteeHotKeyTxCert`.
3. Before the election is enacted (within the 2-epoch window), construct a transaction containing `VoteYes` from `CommitteeVoter ccHot` on a target governance action (e.g., `HardForkInitiation`).
4. Have a block producer include this transaction directly in a block, bypassing the mempool.
5. The `GOV` rule accepts the vote — `when (hardforkConwayDisallowUnelectedCommitteeFromVoting ...)` is `False` at PV ≤ 10, so no check is performed.
6. Wait two epochs for the `UpdateCommittee` election to be enacted.
7. `ccCold` is now in `committeeMembers`; `committeeAcceptedRatio` finds `ccHot` in `gasCommitteeVotes` and counts the pre-vote as `VoteYes`, potentially ratifying the governance action. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

```haskell
committeeAcceptedRatio members votes committeeState currentEpoch =
  yesVotes %? totalExcludingAbstain
  where
    accumVotes ::
      (Integer, Integer) ->
      Credential ColdCommitteeRole ->
      EpochNo ->
      (Integer, Integer)
    accumVotes (!yes, !tot) member expiry
      | currentEpoch > expiry = (yes, tot) -- member is expired, vote "abstain" (don't count it)
      | otherwise =
          case Map.lookup member (csCommitteeCreds committeeState) of
            Nothing -> (yes, tot) -- member is not registered, vote "abstain"
            Just (CommitteeMemberResigned _) -> (yes, tot) -- member has resigned, vote "abstain"
            Just (CommitteeHotCredential hotKey) ->
              case Map.lookup hotKey votes of
                Nothing -> (yes, tot + 1) -- member hasn't voted, vote "no"
                Just Abstain -> (yes, tot) -- member voted "abstain"
                Just VoteNo -> (yes, tot + 1) -- member voted "no"
                Just VoteYes -> (yes + 1, tot + 1) -- member voted "yes"
    (yesVotes, totalExcludingAbstain) = Map.foldlWithKey' accumVotes (0, 0) members
```
