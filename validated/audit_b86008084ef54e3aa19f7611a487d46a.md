### Title
Unbounded `fee_proposal_fri` Accepted When `fee_actual` Is `None` (Window-Incomplete Period) â€” (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

When the `fee_proposals_window` sliding window is incomplete (`fee_actual` is `None`), the validator in `is_proposal_init_valid` skips all bounds enforcement on the proposer-supplied `fee_proposal_fri`. A malicious proposer can set `fee_proposal_fri` to any arbitrary value (e.g., `u128::MAX` or `1`) during this period, and every validator node will accept the proposal. The arbitrary value is then committed into the consensus-signed `ProposalCommitment` and stored in `fee_proposals_window`, poisoning future `fee_actual` and `l2_gas_price` computations for all subsequent blocks.

---

### Finding Description

The `fee_proposal_fri` field in `ProposalInit` is the proposer's oracle-derived recommended L2 gas fee. Validators are supposed to enforce that it lies within a narrow margin of `fee_actual` (the median of the last `window_size` = 10 accepted `fee_proposal` values). This is the authoritative reference â€” the sequencer's analog to the LP oracle price.

In `is_proposal_init_valid` (`validate_proposal.rs` lines 396â€“416), the bounds check is guarded by an `if let` that requires **both** `fee_actual` and `fee_proposal_fri` to be `Some`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    // bounds check
}
``` [1](#0-0) 

When `fee_actual` is `None`, the entire block is skipped. There is no fallback check that `fee_proposal_fri` equals `l2_gas_price` (the honest proposer's own fallback defined in `compute_proposer_fee_proposal`). [2](#0-1) 

`fee_actual` is `None` in two reachable production conditions:

1. **Near-genesis / V0_14_3 activation**: `compute_fee_actual` returns `None` for any height below `window_size` (= 10), because `height.checked_sub(window_size)` underflows.
2. **Transition from pre-V0_14_3 blocks**: Any `None` entry in the window (a pre-V0_14_3 block) causes `compute_fee_actual` to return `None` for the entire window. [3](#0-2) 

`window_size` = 10 across all deployed versioned-constants files: [4](#0-3) 

The `fee_proposal_fri` value accepted without bounds is then:

**1. Committed into `ProposalCommitment`** via `proposal_commitment_from`:
```rust
ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
``` [5](#0-4) 

**2. Stored in `fee_proposals_window`** at decision time via `record_fee_proposal(height, init.fee_proposal_fri)`: [6](#0-5) 

**3. Forwarded to the centralized cende pipeline** as `FeeProposalInfo { fee_proposal_fri: init.fee_proposal_fri }`: [7](#0-6) 

---

### Impact Explanation

**Wrong `ProposalCommitment` (block hash)**. The consensus-signed commitment is `Poseidon(partial_block_hash, fee_proposal_fri)`. An arbitrary `fee_proposal_fri` produces an arbitrary commitment that all validators accept and sign. This commitment is what gets anchored to L1 and used as the parent commitment for subsequent blocks.

**Poisoned fee window â†’ wrong `l2_gas_price`**. The arbitrary `fee_proposal_fri` is stored in `fee_proposals_window`. Once the window fills (10 blocks), `fee_actual` becomes the median of the poisoned window. `l2_gas_price` for all subsequent blocks is derived from this poisoned `fee_actual`, causing incorrect fee accounting for every transaction in every future block.

**Wrong cende blob**. The `fee_proposal_info` forwarded to the centralized proving pipeline carries the arbitrary value, corrupting the off-chain record used for proof generation.

This matches: **Critical â€” Incorrect fee, gas, resource accounting with economic impact** and **Critical â€” Wrong block commitment / receipt / state value**.

---

### Likelihood Explanation

The window-incomplete period is predictable and permanent at V0_14_3 activation: the first 10 blocks after the protocol upgrade always have `fee_actual = None`. Any validator that is the Tendermint proposer during those 10 blocks can exploit this. The transition from pre-V0_14_3 blocks (where `fee_proposal_fri = None` is stored in the window) also triggers the same bypass for up to `window_size` additional blocks. The attack window is narrow but deterministic and known in advance.

---

### Recommendation

When `fee_actual` is `None`, the validator should enforce that `fee_proposal_fri` equals the locally-computed fallback (`l2_gas_price`), mirroring the honest proposer's own behavior in `compute_proposer_fee_proposal`. Specifically, add a check in `is_proposal_init_valid`:

```rust
if let Some(fee_proposal) = init_proposed.fee_proposal_fri {
    match proposal_init_validation.fee_actual {
        Some(fee_actual) => {
            // existing bounds check
        }
        None => {
            // Enforce the fallback: proposer must use l2_gas_price
            if fee_proposal != proposal_init_validation.l2_gas_price_fri {
                return Err(...);
            }
        }
    }
}
```

This is the direct analog to the external bug's fix: "consider the oracle price if the supply is zero" â€” here, "enforce the authoritative fallback (`l2_gas_price`) if the window is incomplete."

### Proof of Concept

1. Network activates V0_14_3. For blocks 0â€“9, `fee_actual` is `None` (window incomplete).
2. A validator is the Tendermint proposer for block 5.
3. The malicious proposer sets `fee_proposal_fri = u128::MAX` in `ProposalInit`.
4. All validators call `is_proposal_init_valid`. Since `proposal_init_validation.fee_actual` is `None`, the `if let (Some(fee_actual), Some(fee_proposal))` guard is false and the bounds check is skipped entirely.
5. The proposal is accepted. `ProposalCommitment = Poseidon(partial_block_hash, u128::MAX)` â€” a wrong commitment â€” is signed by consensus and anchored to L1.
6. `record_fee_proposal(BlockNumber(5), Some(GasPrice(u128::MAX)))` stores the extreme value in `fee_proposals_window`.
7. At block 10, `fee_actual = median([..., u128::MAX, ...])` is dominated by the extreme value. `l2_gas_price` for block 10 and all subsequent blocks is wrong, causing all transactions to be charged incorrect fees. [1](#0-0) [8](#0-7) [6](#0-5) [5](#0-4)

pub fn compute_fee_actual(
    fee_proposals_window: &BTreeMap<BlockNumber, Option<GasPrice>>,
    height: BlockNumber,
    window_size: u64,
) -> Option<GasPrice> {
    let Some(start) = height.0.checked_sub(window_size) else {
        warn!(
            "Cannot compute fee_actual for height {height}: height is below window_size \
             ({window_size})"
        );
        return None;
    };
    let window_size_usize = usize::try_from(window_size).expect("window_size fits in usize");
    let mut window = Vec::with_capacity(window_size_usize);
    for source_height in (start..height.0).map(BlockNumber) {
        match fee_proposals_window.get(&source_height) {
            Some(Some(price)) => window.push(*price),
            Some(None) | None => {
                warn!(
                    "Cannot compute fee_actual for height {height}: fee_proposals_window has no \
                     recorded fee_proposal for height {source_height}"
                );
                return None;
            }
        }
    }
    window.sort();
    let mid = window_size_usize / 2;
    let median = if window_size_usize.is_multiple_of(2) {
        // Even: average of the two middle values, rounded down.
        // Overflow-safe averaging: a + (b - a) / 2 (safe because sorted, so b >= a).
        GasPrice(window[mid - 1].0 + (window[mid].0 - window[mid - 1].0) / 2)
    } else {
        window[mid]
    };
    Some(median)
