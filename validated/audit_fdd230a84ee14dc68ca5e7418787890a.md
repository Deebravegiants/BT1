### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Forged Derived Key - (File: crates/contract/src/lib.rs)

### Summary

In `respond_ckd`, the contract enforces a cryptographic correctness check (`ckd_output_check`) on the CKD response only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`. For the `AppPublicKey` variant, the check is silently skipped. This means any single attested participant ŌĆö strictly below the signing threshold ŌĆö can call `respond_ckd` with an arbitrary, attacker-chosen `CKDResponse` for an `AppPublicKey` request, and the contract will accept and deliver it to the waiting user without any cryptographic verification.

### Finding Description

**Root cause ŌĆö asymmetric verification in `respond_ckd`:**

In `crates/contract/src/lib.rs`, the `respond_ckd` method contains the following match:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ŌåÉ no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies that the submitted `CKDResponse` is the correct output of the threshold CKD protocol given the app's public key and the domain's master public key. For `AppPublicKey`, the arm is a no-op ŌĆö the response is accepted unconditionally.

The same asymmetry exists at request time in `request_app_private_key`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ŌåÉ no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(pk) => {
        if !app_public_key_check(pk) { env::panic_str("app public key check failed") }
    }
}
``` [2](#0-1) 

**Exploit path:**

1. A user calls `request_app_private_key` with an `AppPublicKey` variant request. The contract enqueues a yield and stores the request in `pending_ckd_requests`.
2. A single malicious attested participant (one node, well below the signing threshold) calls `respond_ckd` with a `CKDResponse` containing a key pair they control ŌĆö before any honest node responds.
3. `respond_ckd` passes all checks (caller is attested, protocol is running, domain is BLS12-381) and reaches the match. The `AppPublicKey` arm is a no-op, so the forged response is accepted.
4. `resolve_yields_for` drains the pending queue and resumes the user's yield with the attacker-chosen key material. [3](#0-2) [4](#0-3) 

The first call to `respond_ckd` for a given request wins ŌĆö `resolve_yields_for` removes the entry and resumes all queued yields in one pass. A racing honest node's subsequent call returns `RequestNotFound`.

**Contrast with `respond` (regular signatures):**

In `respond`, the payload hash comes from the **on-chain request** (not the response), and the signature is verified against the derived public key before the yield is resolved:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(signature_response, payload_hash, &expected_public_key).is_ok()
``` [5](#0-4) 

No equivalent check exists for `AppPublicKey` CKD responses.

### Impact Explanation

A single Byzantine attested participant (one node, below the signing threshold) can deliver an arbitrary `CKDResponse` to any user who submits an `AppPublicKey` CKD request. The user receives a derived key that was not produced by the threshold protocol and may correspond to a key pair the attacker controls. If the user subsequently uses this key to receive assets on a foreign chain (the primary use case for CKD), the attacker ŌĆö who holds the corresponding private key ŌĆö can drain those assets. This breaks the core safety invariant that CKD outputs are threshold-computed and unforgeable by any strict subset of participants.

**Impact class:** Medium ŌĆö request-lifecycle and contract execution-flow manipulation that breaks a production safety/accounting invariant (correct key derivation) without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation

Any single attested participant who monitors the NEAR chain for `request_app_private_key` calls with the `AppPublicKey` variant can race to call `respond_ckd` with a forged response. No threshold collusion is required. The attacker only needs to win the race against honest nodes, which is feasible given that NEAR block times are ~1 second and the attacker can submit a transaction immediately upon seeing the request event.

### Recommendation

Apply `ckd_output_check` (or an equivalent cryptographic correctness check) for `AppPublicKey` requests as well. If the `AppPublicKey` variant genuinely cannot be verified against a known app public key, the contract should at minimum verify that the response is a valid point on the BLS12-381 curve and is consistent with the domain's master public key and the request's derivation path. The `AppPublicKeyPV` branch demonstrates the correct pattern:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(pk) => {
        // Add equivalent output check here
        if !ckd_output_check_for_app_public_key(&request.app_id, &response, pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
