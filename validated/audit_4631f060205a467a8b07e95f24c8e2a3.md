### Title
Propose-Update Deposit Permanently Locked in MPC Contract — No Withdrawal Mechanism - (File: `crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

### Summary
The MPC contract's `propose_update` function collects a NEAR deposit from the proposer to cover on-chain storage costs for the uploaded contract binary or config. When the update is executed (or superseded by another update), `do_update` calls `entries.clear()` and `vote_by_participant.clear()`, freeing the storage. However, the deposited NEAR is never refunded to the proposer and there is no `withdraw`/`sweep` function anywhere in the contract to recover these accumulated funds. The `UpdateEntry` struct does not even record the proposer's account ID, making a future refund path structurally impossible without a state migration.

### Finding Description

`propose_update` in `lib.rs` collects a deposit sized to cover storage:

```rust
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// only the excess above `required` is refunded
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
``` [1](#0-0) 

The `required` deposit is computed from the byte size of the stored update:

```rust
pub fn required_deposit(update: &Update) -> NearToken {
    required_deposit(bytes_used(update))
}
fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
``` [2](#0-1) [3](#0-2) 

For a 1 MB contract binary, `bytes_used` includes the code length plus a 128-account-vote overhead, yielding a deposit on the order of ~10 NEAR at NEAR's storage byte cost.

When `do_update` executes (triggered by `vote_update` reaching threshold), it clears **all** pending entries — including proposals from other participants — and never issues any refund:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ... no refund Promise anywhere
``` [4](#0-3) 

The `UpdateEntry` struct stores only `update` and `bytes_used` — the proposer's `AccountId` is never recorded, so there is no way to know who to refund even if a refund path were added later:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
``` [5](#0-4) 

There is no `withdraw`, `sweep`, or `claim` function anywhere in the contract's public API. [6](#0-5) 

### Impact Explanation

Every time a contract upgrade proposal is submitted and then executed (or cleared by a competing proposal reaching threshold), the proposer's storage deposit — potentially ~10 NEAR for a full contract binary — is permanently locked in the contract's balance with no on-chain path to recover it. Over the protocol's lifetime, multiple upgrade cycles will compound this loss. This breaks the production accounting invariant that deposited funds are recoverable once the storage they paid for is freed.

**Allowed impact match:** Medium — balance/accounting invariant broken without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation

Every contract upgrade cycle triggers this. The MPC contract README explicitly describes the upgrade flow as a normal governance operation. Each upgrade proposal from a participant costs them a non-trivial deposit (~10 NEAR for a 1 MB binary). When `do_update` fires, all competing proposals are bulk-cleared with no refunds. This is a deterministic, unconditional loss on every upgrade cycle, reachable by any authorized participant.

### Recommendation

1. Add a `proposer: AccountId` field to `UpdateEntry` so the depositor is recorded.
2. In `do_update`, before calling `entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(required_deposit(entry.bytes_used))` for each cleared proposal.
3. Alternatively, add a guardian-controlled `sweep(recipient: AccountId, amount: NearToken)` function (analogous to the mitigation suggested in the reference report) to drain accumulated NEAR from the contract.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR deposit. The exact amount is kept (excess refunded).
2. Participant B calls `propose_update` with a different binary, also paying ~10 NEAR.
3. Participants vote for B's proposal. `vote_update` reaches threshold and calls `do_update`.
4. `do_update` calls `self.entries.clear()` — A's entry is deleted, freeing its storage. A's ~10 NEAR deposit is now part of the contract's liquid balance.
5. No refund is issued to A. No function exists to withdraw the accumulated NEAR. A's ~10 NEAR is permanently locked in the contract. [7](#0-6) [8](#0-7)

        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(

                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );

            Update::Config(config) => {
                // If we vote for a new config, we should use

                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
