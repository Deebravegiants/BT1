Audit Report

## Title
Permissionless `updateRSETHPrice()` with Ratcheting `highestRsethPrice` Enables Stale-Transaction-Triggered Protocol Pause - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.updateRSETHPrice()` is a public, permissionless function. Its downside-protection logic pauses the entire protocol when the computed price falls more than `pricePercentageLimit` below the persistent all-time-high `highestRsethPrice`. Because `highestRsethPrice` ratchets upward with every new peak, a transaction submitted when the price was within tolerance can become out-of-tolerance by the time it executes if an intervening call raised the baseline. This causes an unexpected, admin-recovery-required protocol-wide pause, temporarily freezing all deposits and withdrawals.

## Finding Description
`updateRSETHPrice()` has no access control and is callable by any address:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Inside `_updateRsETHPrice()`, the downside check compares the freshly computed price against `highestRsethPrice`:

```solidity
// contracts/LRTOracle.sol L270-282
if (newRsETHPrice < highestRsethPrice) {
    uint256 diff = highestRsethPrice - newRsETHPrice;
    bool isPriceDecreaseOffLimit =
        pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
    if (isPriceDecreaseOffLimit) {
        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!withdrawalManager.paused()) withdrawalManager.pause();
        _pause();
        return;
    }
```

`highestRsethPrice` is a persistent all-time-high that only moves upward:

```solidity
// contracts/LRTOracle.sol L293-296
if (newRsETHPrice > highestRsethPrice) {
    highestRsethPrice = newRsETHPrice;
}
```

**Exploit path:**

1. `highestRsethPrice = 1.00e18`, `pricePercentageLimit = 1e16` (1%). Current computed price = `1.005e18` (within threshold). An unprivileged caller submits `updateRSETHPrice()` with low gas; the transaction enters the mempool but does not land.
2. Before the delayed tx lands, the price rises to `1.02e18`. A second `updateRSETHPrice()` executes successfully, setting `highestRsethPrice = 1.02e18`.
3. The price drifts back to `1.005e18` (normal market movement).
4. The delayed transaction from step 1 now executes. It computes `newRsETHPrice = 1.005e18`. `diff = 1.02e18 - 1.005e18 = 0.015e18`. `pricePercentageLimit.mulWad(highestRsethPrice) = 0.01 * 1.02e18 = 0.0102e18`. Since `0.015e18 > 0.0102e18`, `isPriceDecreaseOffLimit = true`.
5. `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()` are all called. The protocol is frozen.

No adversarial intent is required; a legitimate keeper bot submitting with insufficient gas is sufficient.

## Impact Explanation
When the protocol is paused, all user-facing operations revert. `LRTDepositPool.depositETH()` and `depositAsset()` are gated by `whenNotPaused`. `LRTWithdrawalManager.initiateWithdrawal()`, `completeWithdrawal()`, and `instantWithdrawal()` are likewise gated. Funds already in the withdrawal queue are frozen until an admin manually calls `unpause()` on each contract. This constitutes **temporary freezing of funds**, a valid Medium-severity impact under the allowed scope.

## Likelihood Explanation
- `updateRSETHPrice()` is permissionless; any EOA or contract can call it.
- Ethereum mempool congestion routinely delays low-gas transactions by minutes to hours.
- The scenario requires only normal price volatility (a small rise followed by a return to prior levels) combined with a delayed transaction â€” no adversarial action is required.
- The condition is more likely when `pricePercentageLimit` is set tightly (e.g., 0.5â€“1%), which is the intended operational configuration for downside protection.
- The scenario is repeatable: after an admin unpauses, the same sequence can recur.

## Recommendation
Add a `deadline` (or `maxTimestamp`) parameter to `updateRSETHPrice()`. If `block.timestamp > deadline`, revert without pausing the protocol:

```solidity
function updateRSETHPrice(uint256 deadline) public whenNotPaused {
    if (block.timestamp > deadline) revert TransactionExpired();
    _updateRsETHPrice();
}
```

Alternatively, accept a `maxHighestRsethPrice` parameter and revert if `highestRsethPrice` has moved beyond it since the transaction was constructed, ensuring stale transactions fail cleanly rather than triggering a pause.
