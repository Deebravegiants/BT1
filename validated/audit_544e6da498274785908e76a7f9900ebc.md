### Title
Stale `ethValueInWithdrawal` Understates Protocol TVL When stETH Oracle Price Rises While Assets Sit in Converter, Allowing Depositors to Mint Excess rsETH at Depressed Price — (`contracts/LRTConverter.sol` / `contracts/LRTDepositPool.sol`)

---

### Summary

`LRTConverter.ethValueInWithdrawal` is written once at transfer time using the oracle price at that moment and is never refreshed. Because `getAssetDistributionData()` returns `assetLyingInConverter = 0` for non-ETH assets and `getETHDistributionData()` reads the stale `ethValueInWithdrawal` for the converter's ETH contribution, any yield that accrues on stETH (or ETHx) while it sits in the converter is invisible to `_getTotalEthInProtocol()`. When `updateRSETHPrice()` is called, `rsETHPrice` is set below its true value. Any depositor who calls `depositAsset()` before the price is corrected receives more rsETH than they are entitled to, permanently diluting existing holders.

---

### Finding Description

**Root cause — `LRTConverter.transferAssetFromDepositPool`** [1](#0-0) 

`ethValueInWithdrawal` is incremented by `(_amount × oracle_price_at_transfer) / 1e18` and never updated again. The actual stETH tokens sit in the converter and continue to accrue yield, but the stored ETH value does not move.

**Root cause — `LRTDepositPool.getAssetDistributionData`** [2](#0-1) 

For every non-ETH asset the converter balance is explicitly zeroed out with the comment "assets in converter are accounted in their eth value => getETHDistributionData()". This means the 100 stETH sitting in the converter contributes **zero** to the stETH total.

**Root cause — `LRTDepositPool.getETHDistributionData`** [3](#0-2) 

The ETH contribution from the converter is read directly from `ethValueInWithdrawal` — the stale, transfer-time snapshot.

**Consequence in `LRTOracle._getTotalEthInProtocol`** [4](#0-3) 

For each supported asset, `getTotalAssetDeposits(asset)` is multiplied by the **current** oracle price. For stETH the converter balance is 0 (excluded), so the current oracle price is irrelevant for those tokens. For ETH the converter contributes only the stale `ethValueInWithdrawal`. The yield gap — `(current_price − transfer_price) × amount` — is never counted.

**Consequence in `LRTOracle._updateRsETHPrice`** [5](#0-4) 

`newRsETHPrice` is computed from the understated `totalETHInProtocol`, so it is set below the true value.

**Attacker entry point — `LRTDepositPool.depositAsset` / `getRsETHAmountToMint`** [6](#0-5) 

`rsethAmountToMint = (amount × assetPrice) / rsETHPrice`. With `rsETHPrice` depressed, the attacker receives more rsETH per unit of stETH deposited than the fair exchange rate warrants.

---

### Impact Explanation

The yield that accrues on stETH while it resides in the converter is not reflected in `ethValueInWithdrawal`. Every call to `updateRSETHPrice()` (which is `public`) during this window sets `rsETHPrice` below its true value. Any depositor who deposits stETH (or ETH) between that price update and the eventual `claimStEth` call receives excess rsETH. When the claim is processed, `_sendEthToDepositPool` sends the full ETH amount (including yield) to the deposit pool, `ethValueInWithdrawal` is reduced, and the next `updateRSETHPrice()` call corrects the price upward — but the attacker's excess rsETH is already minted and cannot be clawed back. Existing holders are permanently diluted by the stolen yield.

---

### Likelihood Explanation

- `transferAssetFromDepositPool` is a routine operational step; stETH will regularly sit in the converter for hours-to-days while Lido withdrawal requests are queued.
- stETH accrues yield continuously (~3–4 % APY); even a 1-day window on 1 000 stETH produces ~0.1 ETH of invisible yield.
- `updateRSETHPrice()` is `public whenNotPaused` — the attacker can trigger it themselves immediately before depositing.
- `depositAsset()` is permissionless for any supported asset.
- No privileged role, no oracle manipulation, and no front-running dependency are required.

---

### Recommendation

Replace the static `ethValueInWithdrawal` snapshot with a live valuation. In `getETHDistributionData()` (or in a dedicated converter view), compute the converter's ETH contribution as:

```solidity
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
address stETH = lrtConfig.getLSTToken(LRTConstants.ST_ETH_TOKEN);
uint256 stETHInConverter = IERC20(stETH).balanceOf(lrtConverter);
ethLyingInConverter = (stETHInConverter * lrtOracle.getAssetPrice(stETH)) / 1e18;
// repeat for each LST held by the converter
```

This ensures the converter's contribution to TVL always reflects the current oracle price, eliminating the yield gap. Alternatively, `ethValueInWithdrawal` could be refreshed on every `updateRSETHPrice()` call, but the live-balance approach is simpler and more robust.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Fork test (Foundry) — run against a mainnet fork with the deployed contracts
// Demonstrates that rsETHPrice is depressed after stETH yield accrues in converter

contract PoC_StaleEthValueInWithdrawal is Test {
    // Assume addresses are set up via fork
    LRTDepositPool  depositPool;
    LRTConverter    converter;
    LRTOracle       oracle;
    IERC20          stETH;

    function test_staleEthValueDepressesRsETHPrice() external {
        // 1. Operator moves 100 stETH to converter at oracle price 1.0e18
        //    ethValueInWithdrawal becomes 100e18
        vm.prank(assetTransferRole);
        converter.transferAssetFromDepositPool(address(stETH), 100e18);

        uint256 ethValueBefore = converter.ethValueInWithdrawal(); // 100e18

        // 2. Advance time so stETH oracle price rises to ~1.05e18 (5% yield)
        vm.warp(block.timestamp + 365 days);
        // (In a real fork the Lido oracle would have updated; mock it here)
        uint256 newStETHPrice = oracle.getAssetPrice(address(stETH)); // 1.05e18

        // 3. ethValueInWithdrawal is still stale
        assertEq(converter.ethValueInWithdrawal(), ethValueBefore); // still 100e18

        // 4. Anyone can call updateRSETHPrice — attacker does so
        oracle.updateRSETHPrice();
        uint256 depressedPrice = oracle.rsETHPrice();

        // 5. Fair price would account for 105 ETH worth of stETH in converter
        //    Compute fair price manually: totalETH_correct = totalETH_actual + 5e18
        uint256 rsethSupply = IRSETH(lrtConfig.rsETH()).totalSupply();
        uint256 fairPrice = (depressedPrice * rsethSupply + 5e18) / rsethSupply;

        assertLt(depressedPrice, fairPrice, "rsETHPrice is depressed");

        // 6. Attacker deposits 1 stETH and receives excess rsETH
        uint256 fairMint   = (1e18 * newStETHPrice) / fairPrice;
        uint256 actualMint = depositPool.getRsETHAmountToMint(address(stETH), 1e18);

        assertGt(actualMint, fairMint, "attacker mints excess rsETH");
    }
}
```
