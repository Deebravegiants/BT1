Audit Report

## Title
Stale Cross-Chain rsETH/ETH Rate Used for L2 Minting Without Staleness Check — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver` caches the rsETH/ETH exchange rate received via LayerZero and stores a `lastUpdated` timestamp, but `getRate()` returns the cached value unconditionally with no staleness validation. All three L2 pool contracts (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`) use this rate to compute rsETH minting amounts. When the cached rate is stale and lower than the true L1 rate, depositors receive more rsETH than they are entitled to, diluting the unclaimed yield of all existing rsETH holders.

## Finding Description
`CrossChainRateReceiver.lzReceive()` updates `rate` and `lastUpdated` only upon receipt of a LayerZero message from L1: [1](#0-0) 

`getRate()` returns the cached `rate` with no comparison against `lastUpdated` or any maximum staleness threshold: [2](#0-1) 

Every L2 pool's `getRate()` delegates directly to this oracle. For example, `RSETHPoolV3`: [3](#0-2) 

The minting formula in all three pools divides by this rate: [4](#0-3) [5](#0-4) [6](#0-5) 

The rate originates from `RSETHMultiChainRateProvider.getLatestRate()`, which reads `LRTOracle.rsETHPrice()` on L1: [7](#0-6) 

The update path requires: (1) an off-chain keeper to call `updateRsETHPrice()` on L1, (2) a separate call to `updateRate()` on the provider, and (3) LayerZero message delivery to the L2 receiver. Each step introduces latency during which the L2 cached rate can lag behind the true L1 rate. The `dailyMintLimit` guard caps total daily minting volume but does not prevent minting at a stale rate within that cap.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` — used for collateral token pricing in the same pools — already enforces staleness checks: [8](#0-7) 

The absence of an equivalent check in `CrossChainRateReceiver.getRate()` is the root cause.

## Impact Explanation
rsETH is a yield-bearing token whose ETH price strictly increases over time as staking rewards accrue. When the L2 cached rate is stale (lower than the current L1 rate), the formula `rsETHAmount = amountAfterFee * 1e18 / staleRate` produces a larger rsETH amount than the depositor is entitled to at the true rate. The excess rsETH minted is backed by no additional ETH, diluting the per-share value held by all existing rsETH/wrsETH holders. This constitutes **theft of unclaimed yield** from existing holders — a High-severity impact within the allowed scope.

## Likelihood Explanation
No special role or privilege is required. Any unprivileged external caller can invoke `deposit()` on any of the three pool contracts at any time. The stale-rate window arises during normal protocol operation whenever the keeper has not recently called `updateRsETHPrice()` on L1, or when LayerZero message delivery is delayed. The attack is repeatable up to the daily mint limit on each affected chain, and the attacker bears no cost beyond the deposited ETH (which they receive rsETH for, just at an inflated ratio).

## Recommendation
Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver`. In `getRate()`, revert if `block.timestamp - lastUpdated > maxStaleness`:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This mirrors the staleness check already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()` and should be set to a value consistent with the expected keeper update frequency.

## Proof of Concept
1. L1 `LRTOracle.rsETHPrice` = 1.05e18 (current true rate, updated by keeper).
2. L2 `CrossChainRateReceiver.rate` = 1.02e18 (stale; `lastUpdated` is 48 hours ago; LayerZero message not yet delivered).
3. Attacker calls `RSETHPoolV3.deposit{value: 1 ether}("ref")` on L2.
4. `viewSwapRsETHAmountAndFee(1e18)` computes: `rsETHAmount = 1e18 * 1e18 / 1.02e18 ≈ 0.9804 rsETH`.
5. Correct amount at true rate: `1e18 * 1e18 / 1.05e18 ≈ 0.9524 rsETH`.
6. Attacker receives ≈ 0.028 rsETH excess — approximately 2.8% more than entitled — at the expense of all existing rsETH holders whose share value is diluted.
7. No admin action, no special role, no front-running required.

**Foundry fork test plan**: Fork the target L2 chain. Deploy or reference the live `RSETHPoolV3` and its `rsETHOracle` (pointing to `CrossChainRateReceiver`). Warp `block.timestamp` forward by 48 hours without triggering `lzReceive`. Call `deposit{value: 1 ether}`. Assert that the minted wrsETH amount exceeds `1e18 * 1e18 / currentL1Rate`, confirming over-minting at the stale rate.
