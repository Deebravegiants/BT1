Audit Report

## Title
Missing Minimum Output Slippage Protection in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPool.sol)

## Summary
All L2 pool `deposit()` functions compute the wrsETH/rsETH output amount using a live oracle rate at execution time but accept no `minRsETHAmountExpected` parameter. A user who observes a quoted rate off-chain and submits a deposit transaction has no on-chain mechanism to enforce a minimum acceptable output, leaving them exposed to oracle rate updates that occur between transaction submission and execution. This is in direct contrast to the L1 `LRTDepositPool`, which enforces a caller-supplied minimum.

## Finding Description
Every L2 pool `deposit()` overload calls `viewSwapRsETHAmountAndFee()`, which reads `getRate()` from the oracle at execution time and computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The result is immediately minted/transferred to the caller with no floor check:

- `RSETHPoolV3.sol` lines 246–265 (ETH deposit) and 271–293 (token deposit): `wrsETH.mint(msg.sender, rsETHAmount)` with no minimum check.
- `RSETHPoolNoWrapper.sol` lines 231–244 and 250–271: `rsETH.safeTransfer(msg.sender, rsETHAmount)` with no minimum check.
- `RSETHPool.sol` lines 265–278 and 284–305: `IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount)` with no minimum check.
- `RSETHPoolV3ExternalBridge.sol` and `RSETHPoolV3WithNativeChainBridge.sol` follow the identical pattern.

A grep across all pool contracts confirms zero occurrences of `minRsETH`, `minRSETH`, `MinimumAmount`, or `slippage`.

By contrast, `LRTDepositPool._beforeDeposit()` (lines 667–669) explicitly enforces:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

The rsETH/ETH oracle rate is propagated from L1 to each L2 via cross-chain messaging (LayerZero/CCIP). These rate updates are permissionless to relay once the cross-chain message is in flight and occur regularly as EigenLayer rewards accrue. On any L2 with a public mempool, a rate-update relay transaction and a user deposit transaction can land in the same block with the rate update ordered first. No privileged access is required; any block builder or sequencer can produce this ordering.

Exploit path:
1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and observes `rsETHAmount ≈ 0.952e18` at rate `1.05e18`.
2. Alice submits `deposit{value: 1 ether}("ref")`.
3. Before Alice's transaction is included, a cross-chain rate relay updates the oracle to `1.10e18`.
4. Alice's transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909e18`.
5. Alice receives `0.909 wrsETH` instead of `0.952 wrsETH` — a ~4.5% shortfall — with no revert and no recourse.

Existing guards (`nonReentrant`, `whenNotPaused`, `limitDailyMint`, `onlySupportedToken`) do not address output-amount slippage.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

Because wrsETH/rsETH is redeemable at the current oracle rate, the user's position value in ETH terms is preserved. However, the number of shares received is lower than the user intended to accept, and the user has no on-chain mechanism to enforce their own acceptable exchange rate. This matches the allowed impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
Oracle rate updates are a routine, permissionless protocol operation triggered by cross-chain message relays. On L2s with public mempools, transaction ordering that places a rate update before a pending deposit requires no privileged access and no attacker profit motive — it can occur naturally or be induced by any block builder. The condition is repeatable on every oracle update cycle.

## Recommendation
Add a `minRsETHAmountExpected` parameter to every public `deposit()` overload in all L2 pool contracts and revert if the computed output falls below it:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused ...
{
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(msg.value);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

This mirrors the protection already present in `LRTDepositPool._beforeDeposit()` on L1.

## Proof of Concept
Foundry fork test plan:
1. Fork the target L2 at a block where the oracle rate is `R0`.
2. Deploy or reference the live pool contract.
3. Call `viewSwapRsETHAmountAndFee(1 ether)` → record `expected`.
4. Simulate a cross-chain rate relay by calling the oracle's rate-update function to set rate to `R1 > R0` (or warp/prank to trigger the update).
5. Call `deposit{value: 1 ether}("ref")` in the same block.
6. Assert `wrsETH.balanceOf(alice) < expected` — the shortfall is `(1/R0 - 1/R1) * 1e18 * amountAfterFee`, with no revert.
