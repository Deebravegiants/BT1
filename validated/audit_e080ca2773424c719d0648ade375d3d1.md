### Title
Staking Functions Record Nominal Amount Without Verifying Actual Tokens Received, Causing Insolvency for Last Withdrawers - (File: contracts/KERNEL/KernelDepositPool.sol)

### Summary
`KernelDepositPool.stake()` and `stakeFor()` credit `balanceOf` and `totalKernelStaked` with the caller-supplied `_amount` without measuring the actual token balance change. If `kernelToken` carries a transfer fee, the contract's internal accounting permanently overstates its real holdings, making it impossible for the last withdrawers to reclaim their tokens.

### Finding Description
In `KernelDepositPool.sol`, both staking entry points update accounting state with the nominal `_amount` before (or without) verifying what was actually received:

```solidity
// stake() – lines 284-286
balanceOf[msg.sender] += _amount;
totalKernelStaked += _amount;
kernelToken.safeTransferFrom(msg.sender, address(this), _amount);
```

```solidity
// stakeFor() – lines 309-311
balanceOf[_account] += _amount;
totalKernelStaked += _amount;
kernelToken.safeTransferFrom(msg.sender, address(this), _amount);
```

Neither function snapshots the contract's `kernelToken` balance before and after the transfer to derive the true received amount. The same contract's `notifyRewardAmount()` already demonstrates the correct pattern for `rewardsToken`:

```solidity
// notifyRewardAmount() – lines 573-577
uint256 balanceBefore = rewardsToken.balanceOf(address(this));
rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
uint256 balanceAfter = rewardsToken.balanceOf(address(this));
uint256 receivedAmount = balanceAfter - balanceBefore;
```

This defensive pattern is applied to `rewardsToken` but is absent for `kernelToken` in `stake`/`stakeFor`. The divergence means that if `kernelToken` ever charges a transfer fee, every stake call inflates `balanceOf` and `totalKernelStaked` beyond the contract's real token balance.

When a user later calls `initiateWithdrawal(_amount)`, the withdrawal record stores the full nominal `_amount`. `claimWithdrawal()` then attempts `kernelToken.safeTransfer(msg.sender, withdrawal.amount)` using that inflated figure. Because the contract holds fewer tokens than the sum of all recorded withdrawal amounts, the last claimants will find the transfer reverts.

### Impact Explanation
Each fee-bearing stake call widens the gap between `totalKernelStaked` (accounting) and the contract's actual `kernelToken` balance. As users withdraw in FIFO order, the shortfall accumulates until the contract is drained. The final cohort of withdrawers cannot reclaim their principal — a permanent freeze of their staked funds and effective protocol insolvency proportional to the cumulative fee leakage.

**Impact: Critical — Permanent freezing of funds / Protocol insolvency.**

### Likelihood Explanation
The `kernelToken` address is fixed at initialization and is expected to be the KERNEL governance token, which is not currently a fee-on-transfer token. However:
- The token contract could be upgraded or replaced.
- The same `KernelDepositPool` code could be redeployed with a different staking token.
- The inconsistency with `notifyRewardAmount` (which already guards against this) shows the risk was recognized for one token but not the other.

**Likelihood: Low** — requires `kernelToken` to exhibit transfer-fee behavior, which is not the current deployment configuration.

### Recommendation
Apply the same balance-before/after pattern used in `notifyRewardAmount` to both `stake` and `stakeFor`:

```solidity
function stake(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    if (_amount == 0) revert AmountZero();

    uint256 balanceBefore = kernelToken.balanceOf(address(this));
    kernelToken.safeTransferFrom(msg.sender, address(this), _amount);
    uint256 received = kernelToken.balanceOf(address(this)) - balanceBefore;

    balanceOf[msg.sender] += received;
    totalKernelStaked += received;

    emit Staked(msg.sender, received);
}
```

Apply the same fix to `stakeFor`. This ensures `balanceOf` and `totalKernelStaked` always reflect actual holdings regardless of token behavior.

### Proof of Concept

1. Deploy `KernelDepositPool` with a mock `kernelToken` that deducts a 1% transfer fee on every `transferFrom`.
2. Alice calls `stake(100e18)`.
   - Contract receives `99e18` tokens.
   - `balanceOf[Alice]` is set to `100e18`. `totalKernelStaked` = `100e18`.
3. Alice calls `initiateWithdrawal(100e18)` — succeeds because `balanceOf[Alice] >= 100e18`.
   - `withdrawal.amount` = `100e18`.
4. After the delay, Alice calls `claimWithdrawal(withdrawalId)`.
   - `kernelToken.safeTransfer(Alice, 100e18)` is attempted.
   - Contract only holds `99e18` → transfer reverts.
   - Alice's funds are permanently frozen in the contract.

The gap grows with every additional staker, eventually making the contract insolvent for all but the earliest withdrawers.
