### Title
Pigeonhole receipt proof check in `set_state_header` allows duplicate `root` reuse across distinct `from_shard_id` labels, causing wrong state root on syncing node — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

The uniqueness check in `set_state_header` enforces that `from_shard_id` values are distinct, but never verifies that each `RootProof.root` corresponds to the chunk at the position implied by `from_shard_id`. A malicious peer can supply two receipt proofs with different `from_shard_id` labels but identical `(root, block_proof, receipts, proof)` tuples, both of which pass all merkle checks. The syncing node then applies one shard's receipts twice and omits another shard's receipts entirely, producing a wrong state root.

### Finding Description

In `set_state_header` (`chain/chain/src/state_sync/adapter.rs`, lines 463–503), for each block in the incoming receipts range, the code:

1. Checks `receipt_proofs.len() == block_header.chunks_included()` [1](#0-0) 
2. Checks uniqueness of `from_shard_id` via a `HashSet` [2](#0-1) 
3. Verifies `verify_path(*root, proof, &receipts_hash)` where `root` comes from the attacker-supplied `root_proofs[i][j]` [3](#0-2) 
4. Verifies `verify_path(*block_header.prev_chunk_outgoing_receipts_root(), block_proof, root)` [4](#0-3) 

The code never checks that `root` is the outgoing receipts root of the chunk *at the shard index corresponding to `from_shard_id`*. The `from_shard_id` field is only checked for label uniqueness. An attacker can reuse `root_0` (shard 0's outgoing receipts root) for two different `from_shard_id` values (0 and 1), with the same `block_proof` path. Both pass step 4f because `verify_path` only checks that `root` is *a* valid leaf in the block's merkle tree, not *which* leaf position it occupies.

The legitimate construction (in `get_state_header`, lines 202–206) ties `root_proof` to `from_shard_index` via `block.chunks()[from_shard_index].prev_outgoing_receipts_root()`, but this binding is not re-verified on the receiving side. [5](#0-4) 

### Impact Explanation

`set_state_finalize` calls `collect_receipts_from_response`, which flattens all `ReceiptProof` vectors without checking `from_shard_id`: [6](#0-5) 

The resulting `receipts` vector is passed directly to `apply_chunk`: [7](#0-6) 

Applying duplicate receipts produces a wrong `new_root` (state root). The syncing node stores this wrong state root and subsequently fails to validate any block built on top of it, causing it to permanently diverge from the canonical chain. This is a consensus-level impact: a syncing node can be forced into an invalid state by any malicious peer.

### Likelihood Explanation

State sync responses are accepted from any peer (`StateResponseReceived` handler in `client_actor.rs`). [8](#0-7) 

All block data needed to construct valid merkle proofs is public. A malicious peer only needs to know the actual `root_0` and its `block_proof` (both derivable from public block data) to craft the attack. No validator or privileged access is required.

### Recommendation

In the verification loop, after step 4f, additionally verify that `root` equals the outgoing receipts root of the chunk at the shard index corresponding to `from_shard_id`:

```rust
let from_shard_index = shard_layout.get_shard_index(*from_shard_id)?;
let expected_root = *block_header_full.chunks()[from_shard_index].prev_outgoing_receipts_root();
if *root != expected_root {
    return Err(Error::Other("set_shard_state failed: root does not match from_shard_id".into()));
}
```

Alternatively, track a `HashSet` of used `root` values (in addition to `from_shard_id`) to prevent the same outgoing receipts root from being counted twice.

### Proof of Concept

1. Obtain a block B with `chunks_included = 2` (shard 0 has receipts [R1, R2] for target shard T; shard 1 has no receipts for T).
2. Compute `root_0` = shard 0's `prev_outgoing_receipts_root`, and `block_path_0` = its merkle path in the block's `prev_chunk_outgoing_receipts_root` tree.
3. Compute `proof_0` = the merkle path within `root_0`'s tree proving `hash(ReceiptList(T, [R1, R2]))`.
4. Craft a `ShardStateSyncResponseHeader` where for block B:
   - `incoming_receipts_proofs[i]` = `[ReceiptProof([R1,R2], ShardProof{from_shard_id:0, ...proof_0}), ReceiptProof([R1,R2], ShardProof{from_shard_id:1, ...proof_0})]`
   - `root_proofs[i]` = `[RootProof(root_0, block_path_0), RootProof(root_0, block_path_0)]`
5. Call `set_state_header` with this crafted header — it returns `Ok(())`.
6. Call `set_state_finalize` — the node applies [R1, R2, R1, R2] and stores a wrong state root.
