### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Inject Arbitrary Key Material — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_ckd` in the MPC contract performs on-chain cryptographic verification of the CKD output **only** when the request uses the `AppPublicKeyPV` variant. When the request uses the `AppPublicKey` (privately verifiable / legacy) variant, the response branch is an empty no-op, and any `(big_y, big_c)` pair is accepted and forwarded to the waiting TEE app without any on-chain check. A single Byzantine attested participant can exploit this to substitute an attacker-controlled key for the real MPC-derived secret, bypassing the threshold requirement entirely.

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 675–682), the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty: no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the BLS pairing equation `e(big_c, G2) = e(big_y, pk2) · e(H(app_id, pk), pk)`, confirming the response is cryptographically bound to the MPC master key and the app's public key. [2](#0-1) 

For `AppPublicKey`, the branch body is **empty** — no pairing check, no hash check, no binding to the MPC public key. The contract immediately proceeds to resolve the pending yield with whatever `(big_y, big_c)` the caller supplied: [3](#0-2) 

By contrast, `respond` (for ECDSA/EdDSA signatures) **always** verifies the signature against the derived public key before resolving: [4](#0-3) 

The `AppPublicKey` variant is not deprecated — it is the "legacy" format still accepted by the contract, documented as the default non-PV path, and exercised in production e2e tests: [5](#0-4) 

The `AppPublicKey` variant is also the default path in the example CLI: [6](#0-5) 

### Impact Explanation

A single Byzantine attested participant can call `respond_ckd` for any pending `AppPublicKey` CKD request and supply an arbitrary `(big_y, big_c)` pair. Because the contract performs no on-chain verification for this variant, the arbitrary response is accepted and forwarded to the TEE app.

Concretely, the attacker sets `big_y = identity (0)` and `big_c = K` for any key `K` they choose. The TEE app decrypts as `big_c − a · big_y = K − a · 0 = K`, receiving the attacker-controlled value `K` instead of the real `msk · H(app_id)`. The attacker knows `K`, so they can impersonate the TEE app, decrypt its communications, or forge any operation that depends on the derived key.

This bypasses the threshold requirement: the threshold of participants never computed the correct CKD output, yet the contract accepted a single participant's arbitrary response as authoritative. This is **unauthorized confidential key derivation output without the required participant authorization**.

### Likelihood Explanation

- The `AppPublicKey` variant is the legacy default and is actively used in production.
- Only **one** attested participant is required — well below the signing threshold.
- The attacker is already an active participant and observes pending CKD requests via the indexer.
- There is no race condition to win: the attacker can submit their forged response before the honest coordinator does, or race it.
- No special tooling is needed beyond calling `respond_ckd` with crafted arguments.

### Recommendation

Apply the same unconditional on-chain verification to `AppPublicKey` requests that is already applied to `AppPublicKeyPV` requests. For the `AppPublicKey` variant, the contract holds `pk1 = a · G1` but not `pk2 = a · G2`, so the full `ckd_output_check` pairing cannot be run. The options are:

1. **Deprecate and remove `AppPublicKey`** — require all callers to use `AppPublicKeyPV`, which supports full on-chain verification. This is the cleanest fix.
2. **Add a weaker binding check for `AppPublicKey`** — at minimum, verify that `big_c` and `big_y` are valid, non-identity points on the G1 curve and that `big_y` is not the identity (preventing the trivial `big_y=0` attack). This does not fully prevent substitution but raises the bar.

The root fix is option 1: the `AppPublicKey` variant structurally cannot be verified on-chain without `pk2`, so it should not be accepted in `respond_ckd` without a corresponding on-chain check.

### Proof of Concept

1. Attacker is an active, TEE-attested MPC participant.
2. A TEE app submits `request_app_private_key` with `AppPublicKey(pk1 = a·G1)`.
3. The contract stores the pending CKD request; the attacker observes it via the indexer.
4. The attacker calls `respond_ckd(request, CKDResponse { big_y: identity, big_c: K })` where `K` is any G1 point the attacker controls.
5. The contract executes the `AppPublicKey(_) => {}` branch — no check — and calls `resolve_yields_for`, resolving the yield with the attacker's `(identity, K)`.
6. The TEE app receives `(big_y=0, big_c=K)` and decrypts: `K − a·0 = K`.
7. The TEE app uses `K` (attacker-controlled) as its derived secret instead of `msk · H(app_id)`.

The unit test at line 3403 confirms the contract accepts arbitrary `[1u8; 48]` / `[2u8; 48]` byte arrays as a valid `AppPublicKey` CKD response with no cryptographic check: [7](#0-6)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Verify that a CKD response (AppPublicKey variant) is mathematically correct.
#[tokio::test]
#[expect(non_snake_case)]
async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),
        "CKD response failed cryptographic verification"
    );
}
