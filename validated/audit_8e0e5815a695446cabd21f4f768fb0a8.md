### Title
`DirectDeposits` Omitted from Preservation-of-Value Equation Allows Unbounded ADA Creation — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces `DirectDeposits`, a transaction field that credits arbitrary coin amounts directly to registered account balances. These amounts are applied to accounts in the `ENTITIES` rule but are **never included in either side of the preservation-of-value equation**. An unprivileged transaction author can therefore include arbitrarily large `DirectDeposits` in a balanced transaction, creating ADA out of thin air.

---

### Finding Description

**Vulnerability class:** funds/accounting bug — an amount is applied to account balances without being reflected in the `consumed = produced` invariant, analogous to the external report's `registerTradeAndBorrow` crediting the full `outAmount` instead of the net amount.

**Root cause — `DirectDeposits` applied but not accounted for:**

In `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`, the `ENTITIES` rule applies `DirectDeposits` unconditionally to account balances:

```haskell
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

The only validation performed is that target accounts exist:

```haskell
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts
```

No check is made that the deposited amounts are covered by the transaction's inputs.

**Root cause — preservation-of-value equation omits `DirectDeposits`:**

The Dijkstra UTXO rule (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`) enforces:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
```

`validateValueNotConservedUTxO` calls `consumed` and `produced`. For `DijkstraEra`:

```haskell
instance EraUTxO DijkstraEra where
  consumed = conwayConsumed          -- Conway function; knows nothing of DirectDeposits
  getProducedValue = getProducedDijkstraValue
```

`getProducedDijkstraValue` is:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)
```

Neither `conwayProducedValue` (a Conway-era function) nor `dijkstraProducedValue` adds the sum of `DirectDeposits` to the produced side. `DirectDeposits` is a Dijkstra-specific field absent from all prior-era accounting functions.

The equation that is actually checked is therefore:

```
UTxO inputs + withdrawals + refunds
  = UTxO outputs + fees + cert deposits + proposal deposits + treasury donation
```

`DirectDeposits` appear on neither side. A transaction that balances without them will still balance after including arbitrarily large `DirectDeposits`, while those amounts are silently credited to account balances.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker can craft a Dijkstra `TopTx` whose inputs and outputs balance normally, while embedding `DirectDeposits` of any magnitude targeting their own registered account. After the transaction is accepted:

1. The preservation-of-value check passes (the `DirectDeposits` field is invisible to it).
2. The `ENTITIES` rule credits the attacker's account with the specified amount.
3. The attacker issues a withdrawal transaction to move the newly created ADA into UTxO outputs.
4. The process can be repeated indefinitely, draining the system's total ADA supply invariant.

This is a direct, repeatable, and unbounded ADA-creation exploit.

---

### Likelihood Explanation

Any unprivileged transaction author can submit a Dijkstra-era transaction. No special keys, governance majority, or privileged role is required. The attacker only needs:

- A registered stake credential (trivially obtained).
- A valid Dijkstra transaction with balanced inputs/outputs and a non-empty `DirectDeposits` map targeting their own account.

The attack is deterministic and requires no brute-force or probabilistic element.

---

### Recommendation

Add the sum of all `DirectDeposits` coin amounts to the **produced** side of the preservation-of-value equation in `dijkstraProducedValue` (and analogously in `dijkstraSubTxProducedValue` for sub-transactions that also carry `DirectDeposits`):

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> inject (foldMap id . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)
```

This mirrors how `treasuryDonationTxBodyL` is already included in `dijkstraSubTxProducedValue`:

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody <> txBody ^. treasuryDonationTxBodyL)
    <> burnedMultiAssets txBody
```

`DirectDeposits` must be treated identically to treasury donations and cert deposits: amounts that leave the transaction's value and must be covered by its inputs.

---

### Proof of Concept

1. Attacker registers stake credential `C` and obtains account address `A`.
2. Attacker constructs a `DijkstraTxBody` with:
   - `dtbSpendInputs`: one UTxO worth 2 ADA
   - `dtbOutputs`: one output worth 1.8 ADA
   - `dtbTxfee`: 0.2 ADA
   - `dtbDirectDeposits`: `{ A → 1_000_000_000_000 }` (1 million ADA)
3. `validateValueNotConservedUTxO` checks: `2 ADA = 1.8 ADA + 0.2 ADA` ✓ (passes).
4. `ENTITIES` rule calls `applyDirectDeposits`, crediting 1 million ADA to account `A`.
5. Attacker submits a withdrawal transaction draining account `A` into UTxO.
