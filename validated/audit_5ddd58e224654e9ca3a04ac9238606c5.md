### Title
Missing Cryptographic Output Verification in `respond_ckd` for `AppPublicKey` Variant Allows a Single Malicious Attested Participant to Forge CKD Responses - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in `MpcContract` performs no cryptographic output check when the pending CKD request used the `AppPublicKey` (privately verifiable) variant. Any single attested participant can call `respond_ckd` with an arbitrary `CKDResponse` and the contract will accept it, resolving the user's yield with a forged key-derivation output. This is the direct analog of H-05: just as `takeMultipleOneOrders` skipped the per-order "Complication" check that enforced order-type rules, `respond_ckd` skips the per-request-type output check that enforces the correctness of the CKD result.

---

### Finding Description

`respond_ckd` branches on the `app_public_key` variant stored in the pending `CKDRequest`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the network's master public key and the app's ephemeral key. [2](#0-1) 

For `AppPublicKey`, the arm is empty — the contract accepts any `(big_y, big_c)` pair unconditionally. The existing unit test confirms this by passing `[1u8; 48]` / `[2u8; 48]` (not valid BLS12-381 points) and expecting success: [3](#0-2) 

By contrast, `respond` (ECDSA/EdDSA) always verifies the signature against the derived public key before resolving the yield: [4](#0-3) 

`respond_ckd` requires only one attested participant to call it — there is no threshold voting or quorum before the yield is resolved: [5](#0-4) 

---

### Impact Explanation

A single malicious attested participant (strictly below the signing threshold) can race the honest leader and call `respond_ckd` with a fabricated `CKDResponse`. The contract resolves the user's yield immediately with the forged `(big_y, big_c)` values. The user's TEE application then attempts to decrypt the response with its private key `a` and obtains a wrong or random secret. Because the yield is already resolved, the user cannot retry — the request lifecycle is permanently corrupted for that submission.

The CKD feature is specifically designed to give TEE applications deterministic, confidential secrets. Corrupting the output breaks the core safety invariant: the derived secret is no longer the one the threshold network computed, and the malicious node chose it. This matches the **Medium** allowed impact: *"request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The attacker must be an attested participant (has passed `submit_participant_info` and TEE attestation). However, the threshold for *responding* is just one — any single attested participant can call `respond_ckd`. A Byzantine node below the signing threshold can monitor the NEAR chain for `AppPublicKey` CKD requests and race the honest leader's `respond_ckd` transaction. Because NEAR transaction ordering within a block is not guaranteed, this race is realistic on a live network, especially if the attacker has low-latency access to the RPC.

---

### Recommendation

Apply the same pattern used for `AppPublicKeyPV`: require a verifiable proof that the response is consistent with the network's BLS12-381 master public key. Two concrete options:

1. **Require `AppPublicKeyPV` for all on-chain CKD requests.** Deprecate the unverifiable `AppPublicKey` variant. The `AppPublicKeyPV` variant already supports the identity point as `pk1`/`pk2` for use-cases where the derived key is intentionally public, so no functionality is lost.

2. **Add a threshold-voting layer for `AppPublicKey` responses.** Collect `respond_ckd` calls from at least `t` distinct attested participants and only resolve the yield when a quorum agrees on the same `(big_y, big_c)` pair, mirroring how threshold signing works off-chain.

---

### Proof of Concept

1. Alice (a TEE application) calls `request_app_private_key` with `AppPublicKey(pk)` on a CKD domain.
2. The contract stores the pending request and issues a yield.
3. Mallory, a single malicious attested participant, observes the pending request on-chain.
4. Mallory calls `respond_ckd(ckd_request, CKDResponse { big_y: [0u8;48], big_c: [0u8;48] })` before the honest leader.
5. `respond_ckd` passes all checks: Mallory is an attested participant, the protocol is running, `accept_requests` is true, the domain is BLS12-381, and the `AppPublicKey` arm performs no output check.
6. `resolve_yields_for` resolves Alice's yield with the forged response.
7. Alice's TEE application receives `(big_y=[0;48], big_c=[0;48])`, decrypts with her private key `a`, and obtains a garbage secret — not the one the MPC network would have computed.
8. Alice's request lifecycle is permanently consumed; she must pay a new deposit and resubmit, and Mallory can repeat the attack indefinitely. [6](#0-5) [7](#0-6)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {

            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
