Audit Report

## Title
wrsETH Minted Without Backing rsETH Locked in Wrapper, Causing Structural Undercollateralization and Temporary Fund Freeze - (File: contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV2NBA.sol, contracts/L2/RsETHTokenWrapper.sol)

## Summary
Every L2 pool contract calls `wrsETH.mint(msg.sender, rsETHAmount)` at deposit time, which mints `wrsETH` with zero rsETH locked in the `RsETHTokenWrapper`. The wrapper's `_withdraw()` path unconditionally attempts `safeTransfer` of rsETH from its own balance, which reverts if the balance is insufficient. Because rsETH only enters the wrapper asynchronously via the privileged `depositBridgerAssets()` call after a full L2→L1→L2 bridge round-trip, any user who attempts to redeem `wrsETH` for rsETH before that round-trip completes will have their transaction revert, freezing their funds temporarily.

## Finding Description
**Root cause:** `RsETHTokenWrapper.mint()` (line 190–192) calls `_mint(_to, _amount)` with no rsETH deposited into the contract. This is the entry point used by all pool contracts:

- `RSETHPoolV2ExternalBridge.deposit()` line 298: `wrsETH.mint(msg.sender, rsETHAmount)`
- `RSETHPoolV3ExternalBridge.deposit()` line 381: `wrsETH.mint(msg.sender, rsETHAmount)`
- `RSETHPoolV2NBA.deposit()` line 115: `wrsETH.mint(msg.sender, rsETHAmount)`

The only way rsETH enters the wrapper is via `depositBridgerAssets()` (lines 162–170), which is restricted to `BRIDGER_ROLE` and is called only after the bridger completes the full off-chain round-trip (bridge ETH to L1 → deposit into LRTDepositPool → receive rsETH → bridge rsETH back to L2).

The `_withdraw()` path (lines 120–128) burns `wrsETH` and then calls `ERC20Upgradeable(_asset).safeTransfer(_to, _amount)` directly against the wrapper's rsETH balance. There is no guard checking that `rsETH.balanceOf(address(this)) >= _amount` before the transfer; the revert comes from the ERC20 transfer itself when balance is insufficient.

The protocol itself tracks the deficit via `maxAmountToDepositBridgerAsset()` (lines 99–110): `return wrsETHSupply - balanceOfAssetInWrapper`, confirming the undercollateralization is structural and persistent between deposit and bridger settlement.

**Exploit path:**
1. Alice calls `RSETHPoolV2ExternalBridge.deposit{value: X ether}("")` → `wrsETH.mint(Alice, rsETHAmount)` executes; wrapper rsETH balance = 0, wrsETH totalSupply = rsETHAmount.
2. Alice immediately calls `RsETHTokenWrapper.withdraw(rsETH, rsETHAmount)` → `_burn(Alice, rsETHAmount)` succeeds, then `safeTransfer(Alice, rsETHAmount)` **reverts** because wrapper holds 0 rsETH.
3. Alice's wrsETH is burned in the failed transaction (revert unwinds the burn), but her ETH is already held in the pool contract awaiting bridging. She cannot redeem until the bridger completes the round-trip.

**Existing checks are insufficient:** The `limitDailyMint` modifier only caps total daily minting volume; it does not ensure collateralization. The `depositBridgerAssets()` guard (`maxAmountToDepositBridgerAsset(_asset) < _amount`) only prevents over-collateralization, not under-collateralization. No check in `_withdraw()` gates redemption on available rsETH balance.

## Impact Explanation
**Medium — Temporary freezing of funds.** Any depositor who attempts to redeem `wrsETH` for rsETH via `withdraw()` or `withdrawTo()` before the bridger completes the round-trip will have their redemption revert. The freeze duration equals the bridger's round-trip latency (bridge finality + L1 deposit + bridge back). In a failure scenario (LayerZero message drop, L1Vault misconfiguration, bridger key compromise), the freeze escalates to **Critical — Permanent freezing of funds**, but the baseline demonstrable impact is Medium.

## Likelihood Explanation
**High.** The condition is triggered by every single ETH deposit through any of the four pool contracts. No special attacker capability, flash loan, or privileged access is required — any ordinary depositor who calls `deposit()` and then `withdraw()` before bridger settlement is affected. The undercollateralization window is a permanent structural feature, not an edge case.

## Recommendation
1. **Do not call `wrsETH.mint()` at deposit time.** Issue a receipt/IOU to the depositor instead, and mint `wrsETH` only after `depositBridgerAssets()` is called with the corresponding rsETH, using the `_deposit()` internal path (lines 134–141) which locks rsETH 1:1 before minting.
2. Alternatively, add a balance check in `_withdraw()`: require `ERC20Upgradeable(_asset).balanceOf(address(this)) >= _amount` before proceeding, and revert with a descriptive error (e.g., `InsufficientCollateral`) so users understand the redemption queue, and document the undercollateralization window prominently.
3. Consider a queued-withdrawal mechanism that processes redemptions in FIFO order as rsETH arrives via `depositBridgerAssets()`.

// Foundry fork test outline
function test_temporaryFreeze() public {
    // 1. Alice deposits 10 ETH
    vm.deal(alice, 10 ether);
    vm.prank(alice);
    pool.deposit{value: 10 ether}("");

    // 2. Verify wrapper is undercollateralized
    uint256 supply = wrsETH.totalSupply();
    uint256 wrapperBal = rsETH.balanceOf(address(wrsETH));
    assertGt(supply, wrapperBal); // supply > 0, wrapperBal == 0

    // 3. Alice attempts to withdraw — reverts
    uint256 aliceBal = wrsETH.balanceOf(alice);
    vm.prank(alice);
    vm.expectRevert(); // ERC20 transfer fails: insufficient balance
    wrsETH.withdraw(address(rsETH), aliceBal);

    // 4. After bridger settles, withdrawal succeeds
    vm.prank(bridger);
    rsETH.approve(address(wrsETH), aliceBal);
    vm.prank(bridger);
    wrsETH.depositBridgerAssets(address(rsETH), aliceBal);

    vm.prank(alice);
    wrsETH.withdraw(address(rsETH), aliceBal); // now succeeds
}
