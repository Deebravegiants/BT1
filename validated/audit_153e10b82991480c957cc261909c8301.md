### Title
Double Deposit Refund via Multiple Sub-Transactions Claiming the Same Credential Deregistration - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may embed multiple sub-transactions via `subTransactionsTxBodyL`. Each sub-transaction is processed sequentially by the `SUBLEDGERS` rule. An attacker can place an `UnRegDepositTxCert` for the **same staking credential** in two or more sub-transactions. The `dijkstraTotalRefundsTxCerts` function blindly sums the deposit amounts declared in certificate bodies without verifying whether the credential is still registered. No cross-sub-transaction deduplication check exists. The developers have explicitly acknowledged this gap with a skipped (`xit`) test and the comment `"TODO: predicate failure not yet implemented"`.

---

### Finding Description

**Vulnerability class**: Funds/accounting bug — deposit refund claimed multiple times by batching sub-transactions.

**Root cause 1 — `dijkstraTotalRefundsTxCerts` does not consult ledger state:**

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs:229-238
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert _ deposit -> deposit
  _ -> zero
```

This function is a pure fold over certificate bodies. It returns the sum of all claimed refund amounts without checking whether the credential is actually registered in the current `CertState`. It is used by `getTotalRefundsTxCerts` for `DijkstraEra` (line 285 of the same file), which is called by `certsTotalRefundsTxBody` inside `updateUTxOStateNoFees` to reduce `utxosDeposited`.

**Root cause 2 — `SUBLEDGERS` applies sub-transactions sequentially with no cross-sub-tx deduplication:**

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs:128-135
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
```

Each `SUBLEDGER` transition independently processes its sub-transaction's certificates through `SUBENTITIES → SUBCERTS → SUBCERT → SUBDELEG`. After `subTx1` deregisters the credential and credits the refund, `subTx2` attempts the same deregistration. The missing predicate failure means the second deregistration is not properly rejected, allowing the refund to be credited again.

**Root cause 3 — Developers acknowledge the missing check:**

```haskell
-- eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53-75
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  let
    subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = ... & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

The `xit` prefix skips this test entirely. The `error "TODO: predicate failure not yet implemented"` confirms that the rejection logic has not been implemented. The test was written to document the intended behavior but the enforcement is absent.

**Accounting consequence:**

`updateUTxOStateNoFees` computes:
```haskell
-- eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Utxo.hs:627-635
totalRefunds = certsTotalRefundsTxBody pp certState txBody
depositChange = totalDeposits <-> totalRefunds
...
utxosDeposited = utxosDeposited <> depositChange
```

For each sub-transaction containing `UnRegDepositTxCert stakingCred D`, `utxosDeposited` is reduced by `D`. With two such sub-transactions, `utxosDeposited` is reduced by `2*D` while only one deposit of `D` was ever paid, violating the deposit-pot invariant.

---

### Impact Explanation

**Critical. Direct loss of ADA through an invalid ledger state transition.**

An attacker who registered a staking credential paying deposit `D` can construct a top-level transaction with `N` sub-transactions each containing `UnRegDepositTxCert stakingCred D`. The attacker receives `N * D` ADA in refunds while having paid only `D`. The surplus `(N-1) * D` is extracted from the shared deposit pot (`utxosDeposited`), which tracks all outstanding obligations. Draining this pot below its true obligations constitutes a direct, irreversible loss of ADA from the protocol's accounting state. Recovery would require a hard fork to reconcile the deposit pot.

---

### Likelihood Explanation

The Dijkstra era is the current development frontier and sub-transactions are a new feature. Any user who can submit a Dijkstra-era transaction (an unprivileged transaction sender) can exploit this. The attacker needs only one registered staking credential and the ability to construct a transaction with multiple sub-transactions — both are freely available to any participant. The developers have already identified the scenario (the `xit` test exists), meaning the attack vector is known and the window of exposure is open until the predicate failure is implemented.

---

### Recommendation

1. **Implement the missing predicate failure** in the `SUBDELEG` rule (or `SUBCERTS`/`SUBENTITIES`) to reject an `UnRegDepositTxCert` for a credential that was already deregistered by an earlier sub-transaction in the same top-level transaction. The `CertState` is threaded through `foldM` in `SUBLEDGERS`, so after `subTx1` removes the credential, `subTx2`'s `SUBDELEG` transition should see the credential as absent and fail with a proper `StakeKeyNotRegisteredDELEG`-equivalent failure.

2. **Enable the skipped test** in `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` (change `xit` to `it`) once the predicate failure is defined, to prevent regression.

3. **Apply the same deduplication logic to `UnRegDRepTxCert`** across sub-transactions, as `dijkstraTotalRefundsTxCerts` also sums DRep refunds without cross-sub-tx checks.

---

### Proof of Concept

Conceptual transaction structure (mirrors the existing skipped test):

```
Top-level Tx {
  subTransactions: [
    SubTx1 {
      inputs: [utxo1],
      certs: [UnRegDepositTxCert stakingCred D]
      -- credits D to attacker, removes stakingCred from DState
    },
    SubTx2 {
      inputs: [utxo2],
      certs: [UnRegDepositTxCert stakingCred D]
      -- no predicate failure implemented; credits D again
    }
  ]
}
```

**Expected (correct) behavior**: `SubTx2` fails because `stakingCred` is no longer registered after `SubTx1`.

**Actual (buggy) behavior**: Both sub-transactions succeed; `utxosDeposited` is reduced by `2*D`; attacker receives `2*D` having paid only `D`.

The developers' own test skeleton at [1](#0-0)  confirms this exact scenario is known and unguarded. The `dijkstraTotalRefundsTxCerts` function that performs the unchecked summation is at [2](#0-1) . The sequential `foldM` over sub-transactions with no cross-sub-tx deduplication is at [3](#0-2) . The deposit accounting that reduces `utxosDeposited` per sub-transaction is at [4](#0-3) .
