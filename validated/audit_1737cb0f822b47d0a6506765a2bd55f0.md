Looking at the `withinUnstakeLimits` modifier in `LRTConverter.sol` and the `assetsCommitted` mapping in `LRTWithdrawalManager.sol`.

### Title
`withinUnstakeLimits` Modifier Unconditionally Drains `whitelistedUnstakeAllowance` Even When Fully Covered by Active ETH Withdrawals — (`contracts/LRTConverter.sol`)

---

### Summary

The `withinUnstakeLimits` modifier in `LRTConverter` always consumes from `whitelistedUnstakeAllowance` whenever it is non-zero, even when `amountToUnstake` is entirely covered by `availableActiveETHWithdrawals`. Over repeated normal operator calls to `unstakeStEth`, this silently depletes `whitelistedUnstakeAllowance` to zero. Once both `whitelistedUnstakeAllowance == 0` and `assetsCommitted[ETH_TOKEN] == 0`, every subsequent `unstakeStEth` call reverts with `UnstakeLimitExceeded`, temporarily freezing any stETH held in `LRTConverter`.

---

### Finding Description

The modifier logic is:

```solidity
// LRTConverter.sol lines 58–77
modifier withinUnstakeLimits(uint256 amountToUnstake) {
    uint256 availableActiveETHWithdrawals = _getActiveETHUserWithdrawals();

    if (amountToUnstake > whitelistedUnstakeAllowance + availableActiveETHWithdrawals) {
        revert UnstakeLimitExceeded();
    }

    // Consume intended withdrawal limit
    if (whitelistedUnstakeAllowance > 0) {
        uint256 whitelistedAmountConsumed =
            amountToUnstake > whitelistedUnstakeAllowance ? whitelistedUnstakeAllowance : amountToUnstake;
        whitelistedUnstakeAllowance -= whitelistedAmountConsumed;
    }
    _;
}
``` [1](#0-0) 

The gate check correctly allows the call when `amountToUnstake <= whitelistedUnstakeAllowance + availableActiveETHWithdrawals`. However, the consumption block does not first exhaust `availableActiveETHWithdrawals` before touching `whitelistedUnstakeAllowance`. It unconditionally deducts `min(amountToUnstake, whitelistedUnstakeAllowance)` from `whitelistedUnstakeAllowance` regardless of how much of `amountToUnstake` was already covered by active ETH withdrawals.

`_getActiveETHUserWithdrawals()` reads `assetsCommitted[ETH_TOKEN]` from `LRTWithdrawalManager`:

```solidity
// LRTConverter.sol lines 266–270
function _getActiveETHUserWithdrawals() internal view returns (uint256 activeETHWithdrawals) {
    ILRTWithdrawalManager lrtWithdrawalManager = ...;
    activeETHWithdrawals = lrtWithdrawalManager.assetsCommitted(LRTConstants.ETH_TOKEN);
}
``` [2](#0-1) 

`assetsCommitted[ETH_TOKEN]` is incremented by `initiateWithdrawal` and decremented by `_unlockWithdrawalRequests` (inside `unlockQueue`): [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**Temporary freezing of stETH funds in `LRTConverter`.**

stETH is transferred into `LRTConverter` via `transferAssetFromDepositPool` and must be converted to ETH via `unstakeStEth` → `claimStEth`. If `whitelistedUnstakeAllowance` is silently drained to zero and no active ETH withdrawal requests exist, every `unstakeStEth` call reverts:

```
amountToUnstake > 0 + 0  →  revert UnstakeLimitExceeded
```

The stETH balance in `LRTConverter` cannot be unstaked until a whitelisted user manually calls `declareWithdrawalIntent` again, which is an unplanned manual intervention. Until that happens, the stETH is frozen in the contract.

---

### Likelihood Explanation

This occurs during **normal operation**, not an adversarial edge case. The sequence is:

1. Whitelisted user calls `declareWithdrawalIntent(W)` → `whitelistedUnstakeAllowance = W` (normal setup).
2. Any user calls `initiateWithdrawal(ETH_TOKEN, rsETH)` → `assetsCommitted[ETH_TOKEN] = A` (normal user withdrawal).
3. Operator calls `unstakeStEth(X)` where `X ≤ A` (normal operation, fully justified by active ETH withdrawals).
   - Modifier: `X ≤ W + A` ✓ — passes.
   - Bug: consumes `min(X, W)` from `whitelistedUnstakeAllowance` even though `X` was entirely covered by `A`.
4. Operator calls `unlockQueue(ETH_TOKEN, ...)` → `assetsCommitted[ETH_TOKEN] -= X` → back toward 0.
5. After enough such cycles: `whitelistedUnstakeAllowance = 0`, `assetsCommitted[ETH_TOKEN] = 0`.
6. `unstakeStEth` now reverts for any non-zero amount.

No malicious actor is required. This is a logic error that manifests whenever both `whitelistedUnstakeAllowance > 0` and `assetsCommitted[ETH_TOKEN] > 0` at the time of an `unstakeStEth` call — a routine operational state.

---

### Recommendation

The modifier should consume from `whitelistedUnstakeAllowance` only the portion of `amountToUnstake` that exceeds `availableActiveETHWithdrawals`:

```solidity
modifier withinUnstakeLimits(uint256 amountToUnstake) {
    if (amountToUnstake == 0) revert InvalidAmount();

    uint256 availableActiveETHWithdrawals = _getActiveETHUserWithdrawals();

    if (amountToUnstake > whitelistedUnstakeAllowance + availableActiveETHWithdrawals) {
        revert UnstakeLimitExceeded();
    }

    // Only consume whitelistedUnstakeAllowance for the portion NOT covered by active ETH withdrawals
    uint256 coveredByActive = amountToUnstake < availableActiveETHWithdrawals
        ? amountToUnstake
        : availableActiveETHWithdrawals;
    uint256 neededFromWhitelisted = amountToUnstake - coveredByActive;

    if (neededFromWhitelisted > 0) {
        whitelistedUnstakeAllowance -= neededFromWhitelisted;
    }
    _;
}
```

This ensures `whitelistedUnstakeAllowance` is only consumed when active ETH withdrawals are insufficient to cover the requested unstake amount.

### Proof of Concept

**Setup:**
- `whitelistedUnstakeAllowance = 100 ether` (set via `declareWithdrawalIntent`)
- `assetsCommitted[ETH_TOKEN] = 100 ether` (set via `initiateWithdrawal`)
- stETH balance in `LRTConverter` = 100 ether

**Step 1:** Operator calls `unstakeStEth(100 ether)`.
- `availableActiveETHWithdrawals = 100 ether`
- Gate check: `100 ≤ 100 + 100` ✓
- Bug: consumes `min(100, 100) = 100 ether` from `whitelistedUnstakeAllowance`
- After: `whitelistedUnstakeAllowance = 0`

**Step 2:** Operator calls `unlockQueue(ETH_TOKEN, ...)`.
- `assetsCommitted[ETH_TOKEN] -= 100 ether` → `assetsCommitted[ETH_TOKEN] = 0`

**Step 3:** More stETH arrives in `LRTConverter` via `transferAssetFromDepositPool`. Operator tries `unstakeStEth(1 ether)`.
- `availableActiveETHWithdrawals = 0`
- Gate check: `1 > 0 + 0` → **reverts `UnstakeLimitExceeded`**

stETH is now frozen in `LRTConverter` until a whitelisted user manually calls `declareWithdrawalIntent` again.

**Fuzz assertion** (as described in the question): for any `(amountToUnstake, W, A)` with `amountToUnstake ≤ A` and `W > 0`, after `unstakeStEth(amountToUnstake)`, assert `whitelistedUnstakeAllowance == W` (unchanged). The current code fails this assertion, consuming `min(amountToUnstake, W)` instead.
