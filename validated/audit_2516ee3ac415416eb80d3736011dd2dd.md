Audit Report

## Title
Unconditional Zero-Value Fee Transfer in `MerkleDistributor.claim()` Freezes User Claims When `feeInBPS == 0` - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` at line 144 even when `fee == 0`. For ERC20 tokens that revert on zero-value transfers, this permanently blocks all users from claiming their allocated tokens whenever `feeInBPS` is set to zero — a valid and reachable operational state with no on-chain restriction preventing it.

## Finding Description
In `MerkleDistributor.claim()` (lines 138–144), the fee is computed and unconditionally transferred:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);  // ← no zero-check
```

When `feeInBPS == 0`, `fee` evaluates to `0`, and `safeTransfer(protocolTreasury, 0)` is executed. ERC20 tokens that revert on zero-value transfers (e.g., LEND, cUSDCv3) will cause every `claim()` call to revert. Both `initialize()` (line 77) and `setFeeInBPS()` (line 199) only enforce an upper-bound check (`> MAX_FEE_IN_BPS`), so `feeInBPS = 0` is a valid and settable state. The `token` address is also freely configurable via `setToken()` (line 185), allowing any ERC20 to be used.

The sibling contract `KernelMerkleDistributor._processClaim()` (lines 341–343) correctly guards this transfer:

```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
```

`MerkleDistributor` is missing this guard entirely.

## Impact Explanation
When `feeInBPS == 0` and the distributed token reverts on zero-value transfers, `claim()` reverts for every user. No user can receive their allocated tokens. This constitutes **permanent freezing of unclaimed yield** (Medium) — matching the allowed impact scope — until the owner intervenes by setting `feeInBPS` to a non-zero value, which itself imposes an unintended fee on all subsequent claimants.

## Likelihood Explanation
`feeInBPS` can be set to `0` by the owner at any time with no on-chain restriction. The `MerkleDistributor` is explicitly a generic contract whose `token` can be set to any ERC20 via `setToken()`. Running with zero fees is a legitimate operational choice. Any user calling `claim()` under these conditions triggers the revert with no special preconditions beyond holding a valid merkle proof.

## Recommendation
Add a zero-check before the fee transfer, consistent with `KernelMerkleDistributor`:

```solidity
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
1. Deploy `MerkleDistributor` with a token that reverts on zero-value transfers (e.g., a cToken variant or LEND).
2. Owner calls `setFeeInBPS(0)`.
3. Owner sets a valid merkle root via `setMerkleRoot()`.
4. User calls `claim(index, account, cumulativeAmount, merkleProof)`.
5. `fee = (claimableAmount * 0) / 10_000 = 0`.
6. `IERC20(token).safeTransfer(account, claimableAmount)` succeeds.
7. `IERC20(token).safeTransfer(protocolTreasury, 0)` reverts due to the token's zero-value transfer restriction.
8. Entire transaction reverts; user state is rolled back; user cannot claim.
9. All users are permanently blocked from claiming until owner sets `feeInBPS > 0`.

**Foundry test plan:** Deploy a mock ERC20 that reverts on `transfer(_, 0)`. Initialize `MerkleDistributor` with this token and `feeInBPS = 0`. Construct a valid merkle tree with one leaf, set the root, then call `claim()` — assert it reverts. Then call `setFeeInBPS(1)` and assert `claim()` succeeds, confirming the root cause.
