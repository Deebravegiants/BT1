### Title
Missing On-Chain Output Validation for `AppPublicKey` CKD Variant Allows Single Malicious Participant to Inject Arbitrary Key Derivation Output - (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_ckd` contract method performs cryptographic output verification only for the `AppPublicKeyPV` variant of a CKD request. For the `AppPublicKey` variant, the match arm is intentionally empty — no check is performed on the returned `CKDResponse`. This means a single malicious attested participant (below the signing threshold) can call `respond_ckd` with an arbitrary, attacker-chosen `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will accept and deliver it to the caller without any cryptographic validation.

---

### Finding Description

In `respond_ckd`, after authenticating the caller as an attested participant, the contract performs a conditional output check:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies that the response is consistent with the app public key and the domain's root public key — ensuring the output was actually produced by the threshold MPC network. For `AppPublicKey`, the arm is empty: the response is accepted unconditionally and immediately passed to `resolve_yields_for`, which delivers it to the waiting caller. [1](#0-0) 

Contrast this with `respond` (for threshold signatures), where the signature is always verified against the derived public key before resolving the yield:

```rust
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
``` [2](#0-1) 

The `respond_ckd` path for `AppPublicKey` has no equivalent gate. Any attested participant can race to call `respond_ckd` first with a fabricated `CKDResponse`, and the contract will resolve the pending yield with that fabricated output.

---

### Impact Explanation

**Impact: Critical** — Confidential key derivation output without the required participant authorization.

The CKD flow is designed to derive app-specific secrets under the MPC network's root key, such that no single node ever learns the derived secret. The `AppPublicKey` variant is used when the caller provides a public key for the derivation. If a single malicious attested participant can substitute an arbitrary `CKDResponse`, the caller receives a derived key that the attacker chose — not one produced by the threshold network. The attacker knows the corresponding private key. Any funds, encrypted data, or authentication material protected by that derived key are immediately compromised.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attacker must be an attested TEE participant (below the signing threshold), which is a realistic adversary model explicitly in scope.
- NEAR's transaction ordering is deterministic and publicly observable; a malicious node can monitor the mempool for `request_app_private_key` calls using `AppPublicKey` and front-run honest nodes' `respond_ckd` calls.
- No threshold cooperation is required — a single rogue node suffices.
- The `resolve_yields_for` call resolves all pending yields for the request on the first successful `respond_ckd` call, so the first caller wins. [3](#0-2) 

---

### Recommendation

Apply the same "always verify before resolving" discipline used in `respond`. For `AppPublicKey` requests, either:

1. **Require `AppPublicKeyPV` for all on-chain CKD requests** so that `ckd_output_check` can always be applied, or
2. **Add a cryptographic commitment** to the `AppPublicKey` variant that allows the contract to verify the response is consistent with the MPC root key and the request parameters before resolving the yield.

The empty match arm for `AppPublicKey` should be treated as a missing `default-src` fallback — without it, any attested participant can inject an unverified output.

---

### Proof of Concept

1. Caller invokes `request_app_private_key` with `CKDRequestArgs { app_public_key: AppPublicKey(pk), domain_id, derivation_path }`. A yield is created and stored in `pending_ckd_requests`.
2. Malicious attested participant observes the pending request on-chain.
3. Malicious participant calls `respond_ckd(request, fabricated_response)` where `fabricated_response` contains an attacker-chosen derived key.
4. The contract executes:
   - `assert_caller_is_signer()` — passes (attested participant).
   - `is_running_or_resharing()` — passes.
   - `accept_requests` — passes.
   - `assert_caller_is_attested_participant_and_protocol_active()` — passes.
   - `match &request.app_public_key { AppPublicKey(_) => {} ... }` — **no check performed**.
   - `resolve_yields_for(...)` — yield resolved with `fabricated_response`.
5. The original caller receives the attacker-controlled derived key and uses it, believing it was produced by the threshold MPC network. [4](#0-3)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
