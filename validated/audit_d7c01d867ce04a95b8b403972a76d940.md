### Title
Single Byzantine Participant Can Corrupt Non-PV CKD Output via Unverified Contribution Aggregation — (`crates/threshold-signatures/src/confidential_key_derivation/protocol.rs`)

---

### Summary

The non-publicly-verifiable CKD protocol (`protocol.rs`) aggregates participant contributions with no cryptographic verification. A single Byzantine participant (strictly below any threshold) can send a crafted `CKDOutput` to the coordinator, shifting the final derived confidential key by an arbitrary attacker-controlled offset. The PV variant (`protocol_pv.rs`) is protected; the non-PV variant is not.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator receives `CKDOutput` from every other participant and unconditionally adds each contribution to its running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

There is no pairing check, commitment check, or any other guard on the received `(big_y, big_c)` pair. Any participant in the list can substitute arbitrary G1 points.

The honest per-participant contribution is:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk‖app_id) + y_i · app_pk)
``` [2](#0-1) 

After honest aggregation, `unmask(app_sk)` recovers `msk · H(pk‖app_id)` because the `y_i` blinding terms cancel:

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
``` [3](#0-2) 

If a Byzantine participant sends `(ΔY, ΔC)` instead of their honest share, the coordinator computes:

```
big_Y_final = honest_sum_Y + ΔY
big_C_final = honest_sum_C + ΔC
```

`unmask(app_sk)` then returns:

```
(honest_sum_C + ΔC) − app_sk · (honest_sum_Y + ΔY)
= msk · H(pk‖app_id) + (ΔC − app_sk · ΔY)
```

By setting `ΔY = G1::identity()` and `ΔC = arbitrary_point`, the attacker injects a fully controlled additive offset into the derived key, independent of `app_sk`.

**Contrast with `protocol_pv.rs`:** the PV coordinator calls `aggregated_output_check` immediately after aggregation:

```rust
if !aggregated_output_check(&ckd_output, app_pk, &key_pair.public_key, &hash_point) {
    return Err(ProtocolError::AssertionFailed(
        "CKD output failed to verify".to_string(),
    ));
}
``` [4](#0-3) 

This pairing check `e(C, g2) = e(Y, pk2) · e(H, msk_pk)` cryptographically binds the aggregated output to the public key and hash point, making the same attack impossible. [5](#0-4) 

The non-PV protocol is a production-accessible code path: `protocol` is a public module and `ckd()` is a public function. [6](#0-5) 

The contract also exposes a non-PV CKD path (accepting `CKDAppPublicKey::AppPublicKey`) that does not perform the pairing check, meaning a corrupted output from the MPC node would be accepted on-chain.

---

### Impact Explanation

A single Byzantine MPC participant (no threshold collusion required) can cause the coordinator to output a wrong confidential key. The derived key is shifted by an attacker-chosen G1 point. Any application relying on the derived key receiving the correct value `msk · H(pk‖app_id)` will silently receive a corrupted value. The attacker cannot recover the correct key or forge a valid BLS signature, but they can permanently deny correct key derivation for any CKD request routed through the non-PV protocol.

---

### Likelihood Explanation

Any participant in the CKD participant list can mount this attack unilaterally. No threshold collusion is needed. The attack requires only that the non-PV `ckd()` path is invoked (i.e., the caller provides a bare G1 `app_pk` rather than a `PublicVerificationKey`). The attack is a single-round message substitution with no observable side-channel.

---

### Recommendation

1. **Preferred fix:** Deprecate or remove the non-PV `protocol.rs` path and route all production CKD requests through `protocol_pv.rs` (`ckd_pv`), which already has the correct pairing-based aggregated output check.
2. **If the non-PV path must be retained:** Add an equivalent `aggregated_output_check` to `do_ckd_coordinator` in `protocol.rs`. This requires the coordinator to hold the network public key in G2 and the app public key in G2 as well, which is exactly what `PublicVerificationKey` provides.
3. **Contract-side:** Ensure the on-chain CKD handler always invokes `ckd_output_check` regardless of which app-key variant is submitted.
