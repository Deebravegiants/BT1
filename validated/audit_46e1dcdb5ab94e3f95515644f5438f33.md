### Title
Missing CKD Response Validation for `AppPublicKey` Variant Allows Byzantine Participant to Deliver Forged Key Material - (File: crates/contract/src/lib.rs)

### Summary
`respond_ckd` performs no on-chain validation of the `CKDResponse` when the request uses the legacy `AppPublicKey` variant. A single Byzantine MPC participant (below the signing threshold) can submit an arbitrary `CKDResponse` that the contract unconditionally accepts and fans out to every waiting caller, delivering forged confidential key-derivation output without the required threshold authorization.

### Finding Description
The `respond_ckd` function in `crates/contract/src/lib.rs` contains a branch that explicitly skips all response validation for the `AppPublicKey` (legacy, privately-verifiable) variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no validation whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` verifies that the encrypted output is cryptographically consistent with the publicly-verifiable key pair `(pk1, pk2) = (a·G1, a·G2)`. For the `AppPublicKey` variant, no equivalent check exists — the contract proceeds directly to `resolve_yields_for`, which drains the entire fan-out queue and delivers the response to every caller who submitted the same request. [2](#0-1) 

`resolve_yields_for` calls `env::promise_yield_resume` for every queued yield, returning `Ok(())` as long as at least one yield was resumed — regardless of whether the response data is correct. [3](#0-2) 

The `AppPublicKey` variant is explicitly documented as the production-supported legacy format: [4](#0-3) 

### Impact Explanation
A single Byzantine MPC participant (the leader for a given CKD request) can call `respond_ckd` with an arbitrary `CKDResponse { big_y, big_c }` for any pending `AppPublicKey` request. The contract accepts it unconditionally and delivers it to all waiting callers. Because the `AppPublicKey` variant is privately verifiable only, callers have no on-chain mechanism to detect the forgery. Depending on the BLS12-381 ElGamal-style encryption scheme used by the CKD protocol, the attacker may be able to craft a ciphertext that decrypts to a specific key they control, giving them knowledge of the user's derived secret — an unauthorized confidential key derivation output. At minimum, the attacker can deliver garbage, silently corrupting the user's key-derivation result with no on-chain indication of failure.

This matches the Critical impact: *"Unauthorized … confidential key derivation output without the required participant authorization."*

### Likelihood Explanation
The attack requires only a single Byzantine MPC participant — strictly below the signing threshold — who is already an attested participant and therefore authorized to call `respond_ckd`. No threshold collusion, no leaked keys, and no network-level attack are needed. The attacker simply submits a `respond_ckd` transaction with fabricated `big_y`/`big_c` values before the honest leader does. Because the contract performs no validation for the `AppPublicKey` branch, the forged response is accepted on the first attempt.

### Recommendation
**Short term:** Add the same cryptographic output check for the `AppPublicKey` variant that already exists for `AppPublicKeyPV`. If a publicly-verifiable check is not possible for the legacy format, reject `AppPublicKey` requests at the `respond_ckd` entry point and require callers to migrate to `AppPublicKeyPV`.

**Long term:** Deprecate the `AppPublicKey` (privately-verifiable) variant entirely. The `AppPublicKeyPV` variant provides on-chain verifiability that closes this class of forgery attack. Requiring it for all new CKD requests eliminates the unvalidated branch.

### Proof of Concept
1. Alice calls `request_app_private_key` with `AppPublicKey: bls12381g1:<alice_pk>` and `domain_id` pointing to the BLS12-381 CKD domain. The request is queued in `pending_ckd_requests`.
2. A Byzantine MPC participant (Mallory, the elected leader) observes the pending request on-chain.
3. Mallory calls `respond_ckd(request, CKDResponse { big_y: <arbitrary_G1_point>, big_c: <arbitrary_G1_point> })`.
4. The contract checks: attested participant ✓, protocol running ✓, domain is BLS12-381 ✓, `AppPublicKey` branch → **no validation** ✓.
5. `resolve_yields_for` drains Alice's queued yield, delivering the forged `CKDResponse` to her transaction.
6. Alice's transaction resolves with the forged key material. She has no on-chain way to detect the forgery. If Mallory crafted `big_y`/`big_c` as a valid ElGamal encryption of a key Mallory knows, Mallory now shares knowledge of Alice's derived secret. [5](#0-4)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
