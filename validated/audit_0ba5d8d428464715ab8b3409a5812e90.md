### Title
Sub-Transaction Value Conservation Not Enforced in Dijkstra Era SUBUTXO Rule — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The Dijkstra era's `SUBUTXO` rule applies UTxO state changes for sub-transactions (consuming inputs, producing outputs) without ever checking that value is conserved. The top-level `UTXO` rule's value-conservation check covers only the top-level transaction body. An unprivileged transaction author can therefore craft a sub-transaction whose outputs exceed its inputs, minting ADA or native assets from nothing.

---

### Finding Description

**Vulnerability class:** Accounting bug / invalid state transition — analogous to the PuttyV2 flash-loan-without-fee bug, where a mid-execution callback allowed assets to be extracted and returned while bypassing the fee/accounting invariant. Here, the "callback" is the sub-transaction execution path, which modifies ledger state without the conservation invariant being applied to it.

**Root cause — `dijkstraSubUtxoTransition` has no value-conservation check:**

`dijkstraSubUtxoTransition` (SubUtxo.hs lines 231–278) validates inputs, outputs, network IDs, and validity intervals for each sub-transaction, then calls `Shelley.updateUTxOStateNoFees` to apply the UTxO delta. It never calls `validateValueNotConservedUTxO`. [1](#0-0) 

The mapping function `dijkstraUtxoToDijkstraSubUtxoPredFailure` makes this explicit — both `FeeTooSmallUTxO` and `ValueNotConservedUTxO` are marked `error "Impossible"` for the `SUBUTXO` rule, confirming these checks are structurally absent: [2](#0-1) 

**Root cause — top-level check covers only the top-level tx body:**

`dijkstraUtxoTransition` calls `Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody` where `txBody` is `tx ^. bodyTxL` — the top-level transaction body only. Sub-transaction inputs and outputs are not part of `txBody.inputs` or `txBody.outputs`, so they are invisible to this check: [3](#0-2) 

`validateValueNotConservedUTxO` computes `consumed` from `txBody ^. inputsTxBodyL` and `produced` from `txBody ^. outputsTxBodyL`. Neither field aggregates sub-transaction inputs or outputs: [4](#0-3) 

**Contrast with `validateBatchWithdrawals`:**

The Dijkstra era does implement a batch-level check for withdrawals — it aggregates withdrawals from the top-level tx and all sub-transactions before comparing against original account balances: [5](#0-4) 

No analogous `validateBatchValueConservation` exists. The asymmetry confirms this is a design gap, not an intentional omission.

**State application path:**

`updateUTxOStateNoFees` (used by `SUBUTXO`) removes sub-tx inputs from the live UTxO and inserts sub-tx outputs. If outputs carry more value than inputs, the surplus is permanently added to the ledger: [6](#0-5) 

---

### Impact Explanation

**Critical — Direct creation of ADA or native assets through an invalid ledger state transition.**

A sub-transaction that consumes N lovelace from `originalUtxo` and produces N + Δ lovelace in outputs causes the total ADA supply tracked in the UTxO to increase by Δ. This is an invalid state transition that honest nodes would accept (no predicate failure is raised), permanently inflating the supply. Recovery would require a hard fork.

---

### Likelihood Explanation

Any unprivileged user who can submit a Dijkstra-era transaction can include sub-transactions. No special role, key, or governance threshold is required. The attacker only needs to:
1. Own UTxOs to reference as sub-transaction inputs.
2. Declare sub-transaction outputs with inflated values.
3. Satisfy the top-level value-conservation check independently (trivially done by keeping the top-level tx body balanced).

---

### Recommendation

Add a per-sub-transaction value-conservation check inside `dijkstraSubUtxoTransition`, analogous to the top-level check but scoped to the sub-transaction body and the current `utxoState` (not `originalUtxo`, since earlier sub-transactions may have already consumed some UTxOs):

```haskell
-- In dijkstraSubUtxoTransition, before updateUTxOStateNoFees:
runTest $ Shelley.validateValueNotConservedUTxO pp (utxosUtxo utxoState) certState txBody
```

Alternatively, replace the top-level `validateValueNotConservedUTxO` call with a batch-level check (similar to `validateBatchWithdrawals`) that sums consumed and produced values across the top-level tx and all sub-transactions against `originalUtxo`.

---

### Proof of Concept

**Setup:** Ledger contains two UTxOs:
- `utxo_A`: 10 ADA (owned by attacker, used for top-level tx)
- `utxo_B`: 100 ADA (owned by attacker, used for sub-transaction)

**Attack transaction:**

```
TopTx:
  inputs:  { utxo_A }
  outputs: { addr_attacker: 10 ADA − fee }
  fee:     <fee>
  sub-transactions:
    SubTx_1:
      inputs:  { utxo_B }          -- consumes 100 ADA
      outputs: { addr_attacker: 200 ADA }  -- produces 200 ADA
```

**Validation path:**

1. `validateValueNotConservedUTxO pp originalUtxo certState topTxBody` — checks `consumed(topTxBody) == produced(topTxBody)`, i.e., `10 ADA == (10 ADA − fee) + fee`. **Passes.**
2. `dijkstraSubUtxoTransition` for `SubTx_1` — validates inputs exist in `originalUtxo` and `utxoState`, checks output sizes, network IDs, etc. **No value-conservation check. Passes.**
3. `updateUTxOStateNoFees` removes `utxo_B` (100 ADA) and inserts the 200 ADA output. **100 ADA created from nothing.**

**Result:** The attacker's address now holds 200 ADA where 100 ADA existed before. The ADA supply in the UTxO has increased by 100 ADA. All honest nodes accept this block, causing permanent ledger divergence from the intended supply invariant.

validateValueNotConservedUTxO pp utxo certState txBody =
  failureUnless (consumedValue == producedValue) $
    ValueNotConservedUTxO Mismatch {mismatchSupplied = consumedValue, mismatchExpected = producedValue}
  where
    consumedValue = consumed pp certState utxo txBody
    producedValue = produced pp certState txBody

updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  let UTxOState {utxosUtxo, utxosDeposited, utxosFees, utxosDonation} = utxos
      UTxO utxo = utxosUtxo
      !utxoAdd = txouts txBody -- These will be inserted into the UTxO
      {- utxoDel  = txins txb ◁ utxo -}
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      {- newUTxO = (txins txb ⋪ utxo) ∪ outs txb -}
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
      deletedUTxO = UTxO utxoDel
      totalRefunds = certsTotalRefundsTxBody pp certState txBody
      totalDeposits = certsTotalDepositsTxBody pp certState txBody
      depositChange = totalDeposits <-> totalRefunds
  depositChangeEvent depositChange
  txUtxODiffEvent deletedUTxO utxoAdd
  pure $!
    UTxOState
      { utxosUtxo = UTxO newUTxO
      , utxosDeposited = utxosDeposited <> depositChange
      , utxosFees = utxosFees
      , utxosGovState = govState
      , utxosInstantStake =
          deleteInstantStake deletedUTxO (addInstantStake utxoAdd (utxos ^. instantStakeL))
      , utxosDonation = utxosDonation
      }
