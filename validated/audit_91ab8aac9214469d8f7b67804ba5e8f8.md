### Title
`eraMaxLanguage = PlutusV3` Incorrectly Excludes PlutusV4 from Supported Languages in `DijkstraEra` - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`)

---

### Summary

The `AlonzoEraScript DijkstraEra` instance declares `eraMaxLanguage = PlutusV3`, but the era's own data type, `mkPlutusScript`, `mkSupportedLanguage`, and `TxInfoResult` all fully support PlutusV4. This is the direct Cardano Ledger analog of the EIP-1155 `supportsInterface` bug: a capability-reporting function returns an incorrect (too-low) value while the underlying implementation actually supports the higher capability. As a result, `eraLanguages @DijkstraEra` and `supportedLanguages @DijkstraEra` silently exclude PlutusV4, causing inconsistent behavior across validation paths that rely on these enumerations.

---

### Finding Description

In `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`, the `AlonzoEraScript DijkstraEra` instance sets:

```haskell
eraMaxLanguage = PlutusV3
``` [1](#0-0) 

Yet the same instance defines a `PlutusScript DijkstraEra` data type with four constructors including `DijkstraPlutusV4`, and `mkPlutusScript` accepts all four language versions without restriction:

```haskell
mkPlutusScript plutus =
  case plutusSLanguage plutus of
    SPlutusV1 -> pure $ DijkstraPlutusV1 plutus
    SPlutusV2 -> pure $ DijkstraPlutusV2 plutus
    SPlutusV3 -> pure $ DijkstraPlutusV3 plutus
    SPlutusV4 -> pure $ DijkstraPlutusV4 plutus
``` [2](#0-1) 

The `EraPlutusContext DijkstraEra` instance in `TxInfo.hs` also explicitly returns `Just` for PlutusV4 in `mkSupportedLanguage`, and `TxInfoResult DijkstraEra` carries a `PlutusTxInfoResult 'PlutusV4 DijkstraEra` field: [3](#0-2) 

The CDDL specification for Dijkstra also explicitly defines `plutus_v4_script` as a valid script type at tag `4`: [4](#0-3) 

The `eraMaxLanguage` value feeds directly into `eraLanguages`:

```haskell
eraLanguages :: forall era. AlonzoEraScript era => [Language]
eraLanguages = [minBound .. eraMaxLanguage @era]
``` [5](#0-4) 

With `eraMaxLanguage = PlutusV3`, `eraLanguages @DijkstraEra` evaluates to `[PlutusV1, PlutusV2, PlutusV3]`, silently omitting PlutusV4. Similarly, `supportedLanguages` iterates `[minBound .. eraMaxLanguage @era]`, so PlutusV4 is excluded from the enumeration of supported languages: [6](#0-5) 

The `AlonzoTxAuxData` pattern construction uses `eraMaxLanguage` to gate which Plutus script versions are permitted in auxiliary data:

```haskell
let unsupportedScripts =
      Map.filterWithKey (\lang _ -> lang > eraMaxLanguage @era) atadrPlutusScripts
``` [7](#0-6) 

Because `PlutusV4 > PlutusV3 = True`, any attempt to construct `AlonzoTxAuxData` containing PlutusV4 scripts triggers a runtime `error`, crashing the node process. This is reachable whenever node software constructs auxiliary data for a Dijkstra-era transaction that includes PlutusV4 scripts.

Additionally, any validation logic that iterates `eraLanguages` to check cost-model coverage will skip PlutusV4. If the protocol parameters are validated to contain cost models only for languages in `eraLanguages`, no PlutusV4 cost model would be required. When a transaction with a PlutusV4 script is later submitted, `collectTwoPhaseScriptInputs` would fail with `NoCostModel PlutusV4`, causing the transaction to be rejected as a phase-1 failure and permanently preventing execution of PlutusV4 scripts.

---

### Impact Explanation

**High — Permanent freezing of funds locked by PlutusV4 scripts, where recovery requires a hard fork.**

Any UTxO locked by a PlutusV4 script in the Dijkstra era cannot be spent: the ledger's `eraLanguages` excludes PlutusV4, so cost-model validation for PlutusV4 is skipped during protocol-parameter updates, leaving no PlutusV4 cost model in the parameters. When a spending transaction is submitted, `collectTwoPhaseScriptInputs` returns `NoCostModel PlutusV4`, causing a phase-1 failure. The funds are permanently unspendable without a hard fork to correct `eraMaxLanguage`.

Additionally, any node code path that constructs `AlonzoTxAuxData` with PlutusV4 scripts will crash at runtime due to the `error` call in the pattern synonym.

---

### Likelihood Explanation

The Dijkstra era is the current development frontier and PlutusV4 is its newly introduced script language. Any user or protocol participant who deploys a PlutusV4 script to lock funds will immediately encounter this issue upon attempting to spend those funds. The attacker-controlled entry path is simply submitting a transaction that spends a PlutusV4-locked UTxO, which is a standard, unprivileged ledger operation.

---

### Recommendation

Change `eraMaxLanguage` in the `AlonzoEraScript DijkstraEra` instance from `PlutusV3` to `PlutusV4`:

```diff
- eraMaxLanguage = PlutusV3
+ eraMaxLanguage = PlutusV4
```

This aligns the capability-reporting value with the era's actual implementation, ensuring `eraLanguages @DijkstraEra`, `supportedLanguages @DijkstraEra`, and `AlonzoTxAuxData` construction all correctly include PlutusV4.

### Proof of Concept

The inconsistency can be demonstrated by evaluating the following in the context of the Dijkstra era:

```haskell
-- eraMaxLanguage reports PlutusV3
eraMaxLanguage @DijkstraEra
-- => PlutusV3

-- eraLanguages excludes PlutusV4
eraLanguages @DijkstraEra
-- => [PlutusV1, PlutusV2, PlutusV3]

-- But mkPlutusScript accepts PlutusV4 without failure
mkPlutusScript (Plutus @'PlutusV4 someBytes) :: Maybe (PlutusScript DijkstraEra)
-- => Just (DijkstraPlutusV4 ...)

-- And mkSupportedLanguage returns Just for PlutusV4

-- AlonzoTxAuxData mempty mempty (Map.singleton PlutusV4 someScripts)
