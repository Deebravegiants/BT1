Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Blocks All Withdrawals Post-Initialization - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool.initialize()` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal` evaluates `userWithdrawalIds[msg.sender].length >= 0`, which is always `true` for a `uint256` array length, causing every withdrawal attempt to revert with `WithdrawalLimitReached`. Because `setMaxNumberOfWithdrawalsPerUser` explicitly rejects `0`, the only exit from this broken state is an explicit admin call. Meanwhile, `stake()` accepts deposits freely, so users can lock tokens into a contract that cannot return them.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared as a plain `uint256` state variable: [1](#0-0) 

`initialize()` sets `kernelToken`, `rewardsToken`, and roles, but never assigns `maxNumberOfWithdrawalsPerUser`: [2](#0-1) 

`initiateWithdrawal` enforces the limit at line 323: [3](#0-2) 

With `maxNumberOfWithdrawalsPerUser == 0`, the comparison becomes `uint256_length >= 0`, which is unconditionally `true` in Solidity (unsigned integers cannot be negative), so the function always reverts with `WithdrawalLimitReached`.

The only setter explicitly rejects `0`: [4](#0-3) 

This means there is no on-chain path out of the broken state except an admin call to `setMaxNumberOfWithdrawalsPerUser` with a value in `[1, 100]`. `stake()` has no such guard and accepts deposits freely: [5](#0-4) 

## Impact Explanation
Any user who calls `stake()` after `initialize()` but before the admin calls `setMaxNumberOfWithdrawalsPerUser` cannot initiate a withdrawal. Their principal is held correctly by the contract but is inaccessible via the normal exit path. No funds are permanently lost, but the contract fails to deliver its core promised function (withdrawals). This matches **Low: Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
This is the deterministic post-`initialize` state of every fresh deployment. No attacker is required — the condition is triggered by any user calling `initiateWithdrawal()` before the admin has called `setMaxNumberOfWithdrawalsPerUser`. Since `stake()` is open immediately after `initialize()`, the window where users can deposit into a non-withdrawable contract exists on every deployment that does not atomically set the limit in the same transaction as initialization.

## Recommendation
Set a safe default inside `initialize()`, either inline at declaration:

```solidity
uint256 public maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // = 100
```

Or explicitly in `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER;
```

Alternatively, add a `_maxWithdrawalsPerUser` parameter to `initialize()` with the same validation logic used in `setMaxNumberOfWithdrawalsPerUser`.

## Proof of Concept
Deploy `KernelDepositPool`, call `initialize()` without subsequently calling `setMaxNumberOfWithdrawalsPerUser`. Have a user call `stake(1e18)` followed by `initiateWithdrawal(1e18)`. The withdrawal reverts with `WithdrawalLimitReached`. Confirm with `assertEq(pool.maxNumberOfWithdrawalsPerUser(), 0)`. The submitted Foundry test in the report reproduces this exactly against unmodified contract code.
