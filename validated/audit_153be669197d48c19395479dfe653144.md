Audit Report

## Title
Stale Cached Rate in `CrossChainRateReceiver.getRate()` Enables Over-Minting of wrsETH — (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver` stores both a cached `rate` and a `lastUpdated` timestamp, but `getRate()` returns the cached rate unconditionally without any staleness check. All L2 pool contracts use this rate to compute wrsETH minting amounts. When the LayerZero relayer is delayed, the stale (lower) rate causes depositors to receive more wrsETH than their ETH backs, extracting accrued yield from the protocol.

## Finding Description

`CrossChainRateReceiver` maintains `uint256 public rate` and `uint256 public lastUpdated` as state variables (lines 13–16). The `lastUpdated` field is written inside `lzReceive()` at line 97 alongside `rate`, but is never read anywhere in the contract. The public accessor at lines 102–105 returns `rate` with no staleness guard:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
```

`RSETHPoolV3.getRate()` (line 235–237) delegates directly to `IOracle(rsETHOracle).getRate()`, and `viewSwapRsETHAmountAndFee()` (lines 299–308 and 315–335) uses the returned value as the denominator:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The public `deposit()` functions (lines 246–265 and 271–293) call `viewSwapRsETHAmountAndFee` and then call `wrsETH.mint(msg.sender, rsETHAmount)` directly. Because rsETH is yield-bearing, its true ETH value grows monotonically. A stale (lower) cached rate makes the denominator smaller, inflating `rsETHAmount` beyond what the depositor's ETH actually backs. The same pattern is present in `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`.

Existing guards do not prevent this:
- The `whenNotPaused` modifier only blocks deposits when the contract is manually paused.
- The `limitDailyMint` modifier caps total daily wrsETH issuance but uses the same stale `getRate()` call to compute the cap, so it limits scale but does not prevent over-issuance within the daily window.
- No check compares `block.timestamp` against `lastUpdated` anywhere in the call path.

The SECURITY.md exclusion for "Incorrect data supplied by third-party oracles" does not apply here: the contract itself stores `lastUpdated` and has all the information needed to detect staleness, but the code omits the check. The root cause is in the contract's own logic, not in LayerZero's behavior.

## Impact Explanation

**High — Theft of unclaimed yield.**

rsETH accrues yield continuously on L1. The delta between the stale cached rate and the true current rate represents yield that has accrued but not yet been reflected on L2. Any depositor during a stale window receives wrsETH backed by that unaccounted yield. When they later unwrap wrsETH → rsETH and redeem on L1, they extract more ETH than they deposited, consuming yield that belongs to all other rsETH holders. This matches the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation

**Medium.** LayerZero relayer downtime, network congestion, or a temporary L1/L2 connectivity gap can leave `lastUpdated` hours or days behind. The `lastUpdated` value is publicly readable on-chain, so an attacker can passively monitor it and deposit only when the rate is stale. No privileged access, no front-running, and no victim mistakes are required — the entry path is the standard public `deposit()` function.

## Recommendation

Add a configurable maximum staleness period and revert in `getRate()` if the cached rate is too old:

```solidity
uint256 public constant MAX_RATE_STALENESS = 24 hours;

function getRate() external view returns (uint256) {
    require(
        block.timestamp - lastUpdated <= MAX_RATE_STALENESS,
        "Rate is stale"
    );
    return rate;
}
```

Alternatively, enforce the staleness check at the pool level inside `viewSwapRsETHAmountAndFee()` before computing `rsETHAmount`, using the publicly available `lastUpdated` value from the oracle contract.

## Proof of Concept

1. At `T=0`, `lzReceive` sets `rate = 1.010e18` and `lastUpdated = T` (`CrossChainRateReceiver.sol` lines 95–97).
2. LayerZero relayer goes offline. True rsETH rate on L1 grows to `1.015e18` by `T+48h`.
3. Attacker calls `RSETHPoolV3.deposit{value: 10 ether}("")` at `T+48h`.
4. Pool calls `CrossChainRateReceiver.getRate()` → returns stale `1.010e18` (line 103–105).
5. `rsETHAmount = 10e18 * 1e18 / 1.010e18 ≈ 9.901 wrsETH`; correct value would be `≈ 9.852 wrsETH` (line 307).
6. Attacker receives `≈ 0.049 wrsETH` excess, redeemable for `≈ 0.05 ETH` profit per 10 ETH deposited.
7. Repeated up to the daily mint limit across the stale window, the protocol's wrsETH supply becomes partially unbacked by the accrued-but-uncaptured yield.

**Foundry fork test plan:** Fork the target L2 at a block where `lastUpdated` is recent. Warp `block.timestamp` forward by 48 hours without triggering `lzReceive`. Call `deposit{value: 10 ether}` and assert that the minted `rsETHAmount` exceeds `10e18 * 1e18 / trueRate`, confirming over-issuance proportional to the yield accrued during the stale window.
