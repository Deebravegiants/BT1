Audit Report

## Title
Withdrawers Forfeit Yield Accrued During Withdrawal Delay Due to `min()` Cap in `_calculatePayoutAmount` - (`contracts/LRTWithdrawalManager.sol`)

## Summary
When a user initiates a withdrawal, `expectedAssetAmount` is computed and locked at the current rsETH oracle price. During the mandatory withdrawal delay (default 8 days), rsETH price rises as staking yield accrues. The `_calculatePayoutAmount` function enforces a hard cap at the originally locked amount via a `min()` comparison, so withdrawers receive no benefit from yield that accrues while their rsETH is held in the contract. The accrued yield is redistributed to remaining rsETH holders.

## Finding Description
In `initiateWithdrawal` (L168–175), `expectedAssetAmount` is computed at the current oracle price and stored in the `WithdrawalRequest` struct (L751–753):

```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
assetsCommitted[asset] += expectedAssetAmount;
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
```

`getExpectedAssetAmount` (L592–593) computes:
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

When `unlockQueue` is called by the operator after the delay, `_calculatePayoutAmount` (L833–834) computes:
```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

This is a `min(lockedAmount, currentValue)`. If rsETH price has risen during the delay (the normal operating condition), `currentReturn > expectedAssetAmount`, and the user is capped at the lower locked amount. The result is written back at L804 and transferred at L734:
```solidity
request.expectedAssetAmount = payoutAmount;  // L804
_transferAsset(asset, user, request.expectedAssetAmount);  // L734
```

The rsETH transferred to the contract at initiation continues to contribute to protocol TVL and earns yield reflected in the rising rsETH price, but the withdrawer receives none of it. The `min()` guard is asymmetric: it correctly protects against price decreases (user gets less if rsETH price drops), but incorrectly penalizes withdrawers when price rises.

## Impact Explanation
Every withdrawal through the standard queue path loses yield proportional to the rsETH price appreciation during the delay window. The yield is not destroyed — it remains in the protocol and accrues to remaining rsETH holders, constituting a concrete transfer of value away from withdrawers. This matches the allowed impact: **High — Theft of unclaimed yield**. No special conditions are required; the effect occurs under normal protocol operation whenever any staking yield accrues during the delay.

## Likelihood Explanation
Every user who calls `initiateWithdrawal` is affected. The withdrawal delay is initialized to `8 days / 12 seconds` blocks (L94). As long as any staking yield accrues during those 8 days — the normal operating condition for an ETH liquid restaking protocol — the cap is triggered. No attacker action is required; the loss is structural and automatic for every withdrawal.

## Recommendation
Remove the cap at `expectedAssetAmount` in `_calculatePayoutAmount`. The `expectedAssetAmount` locked at initiation should only serve as the upper bound for `assetsCommitted` accounting (over-withdrawal prevention), not as a ceiling on the final payout. At unlock time, compute the payout purely from current oracle prices:

```solidity
function _calculatePayoutAmount(...) private view returns (uint256) {
    return (request.rsETHUnstaked * rsETHPrice) / assetPrice;
}
```

Alternatively, track `assetsCommitted` in rsETH units rather than asset units, and convert to asset amounts only at unlock time using then-current prices.

## Proof of Concept
1. rsETH price = 1.05 ETH/rsETH. User calls `initiateWithdrawal(stETH, 100e18)`.
   - `expectedAssetAmount = 100e18 * 1.05e18 / 1e18 = 105e18` stETH.
   - `assetsCommitted[stETH] += 105e18`. rsETH transferred to contract.
2. 8 days pass. Staking rewards accrue. rsETH price updates to 1.06 ETH/rsETH.
3. Operator calls `unlockQueue(stETH, ...)`.
   - `currentReturn = 100e18 * 1.06e18 / 1e18 = 106e18` stETH.
   - `_calculatePayoutAmount` returns `min(105e18, 106e18) = 105e18`.
   - `request.expectedAssetAmount` set to `105e18`.
4. User calls `completeWithdrawal(stETH)` and receives `105e18` stETH.
   - Lost yield: `1e18` stETH (≈ 1 ETH) that accrued on their rsETH during the 8-day delay.

A Foundry fork test can confirm this by: (1) deploying against a mainnet fork, (2) calling `initiateWithdrawal`, (3) warping 8 days and mocking an rsETH price increase via the oracle, (4) calling `unlockQueue` and `completeWithdrawal`, and (5) asserting the received amount equals the initiation-time value rather than the higher unlock-time value.
