### Title
Multiple Sub-Transactions Can Claim the Same Deposit Refund via Intermediate CertState Reuse - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

In the Dijkstra era, the `SUBLEDGERS`/`SUBUTXO` rules process each sub-transaction's deposit refunds using a `CertState` snapshot that is **not updated between sub-transactions**. An attacker can craft a top-level transaction containing multiple sub-transactions that each include an `UnRegDepositTxCert` for the **same** staking credential, causing the deposit pot to be debited multiple times for a single registered deposit. The codebase itself contains a disabled test (`xit`) that explicitly acknowledges this protection is not yet implemented.

---

### Finding Description

The Dijkstra era introduces nested (sub-)transactions. Each sub-transaction is processed by the `SUBUTXO` rule, whose environment is `SubUtxoEnv`:

```haskell
data SubUtxoEnv era = SubUtxoEnv
  { sueSlot        :: SlotNo
  , suePParams     :: PParams era
  , sueCertState   :: CertState era   -- ← snapshot, not updated between sub-txs
  , sueOriginalUtxo :: UTxO era
  , sueTopTxIsValid :: IsValid
  }
``` [1](#0-0) 

Inside the `SUBUTXO` transition, when the sub-transaction is valid, `updateUTxOStateNoFees` is called with `certState` taken from the environment:

```haskell
newState <-
  Shelley.updateUTxOStateNoFees
    pp
    utxoState
    txBody
    certState          -- ← from SubUtxoEnv, same for every sub-tx
    (utxosGovState utxoState)
    ...
``` [2](#0-1) 

Inside `updateUTxOStateNoFees`, the refund is computed as:

```haskell
totalRefunds = certsTotalRefundsTxBody pp certState txBody
``` [3](#0-2) 

Because `certState` is a **fixed environment value** (not the evolving state output of each sub-transaction), the deposit entry for a staking credential remains present and non-zero for every sub-transaction in the sequence. Two sub-transactions that both carry `UnRegDepositTxCert stakingCred keyDeposit` for the same credential will each independently pass the balance check and each independently reduce `utxosDeposited` by `keyDeposit`, even though only one deposit was ever paid.

The codebase explicitly acknowledges this gap with a disabled test:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  tx = ... & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [4](#0-3) 

The `xit` marker and the `error "TODO: predicate failure not yet implemented"` confirm that no guard exists to prevent this double-refund.

---

### Impact Explanation

**Critical. Direct loss of ADA through an invalid ledger state transition.**

An attacker who has registered one staking credential (paying one `keyDeposit`) can construct a top-level transaction with N sub-transactions each deregistering the same credential. The `utxosDeposited` pot is decremented N times while only one deposit was ever paid, draining N − 1 extra ADA from the deposit pot. The deposit pot is part of the global ADA supply accounting; over-refunding it constitutes a direct, permanent creation of ADA out of thin air at the expense of the deposit reserve, violating the preservation-of-value invariant.

---

### Likelihood Explanation

The Dijkstra era is not yet deployed on mainnet, but the vulnerability is present in the production implementation code and is reachable by any unprivileged transaction author. The attacker only needs to:
1. Register a staking credential (a normal, permissionless operation).
2. Submit a single top-level transaction containing multiple sub-transactions that each deregister the same credential.

No special privilege, governance majority, or key compromise is required. The attack is deterministic and repeatable.

---

### Recommendation

The `SUBLEDGERS` rule (or the `SUBUTXO` rule) must thread the **updated** `CertState` — reflecting deregistrations performed by earlier sub-transactions — into the environment of each subsequent sub-transaction, rather than passing the same pre-transaction snapshot to all of them. Concretely:

- After each sub-transaction's `SUBENTITIES`/`SUBCERTS` step removes a staking credential from `certDState`, the resulting `CertState` must be used as `sueCertState` for the next sub-transaction.
- Alternatively, a pre-validation pass over all sub-transaction certificates can detect duplicate `UnRegDepositTxCert` entries for the same credential and reject the top-level transaction before any state mutation occurs.

The disabled test at `Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53` should be re-enabled once the fix is in place.

---

### Proof of Concept

Using the existing test scaffold (re-enabling the `xit` as `it` after the fix is absent):

```haskell
-- Currently xit (disabled) because "TODO: predicate failure not yet implemented"
it "Multiple subtransactions CAN get the same refund (demonstrates the bug)" $ do
  stakingCred <- KeyHashObj <$> freshKeyHash
  _ <- registerStakeCredential stakingCred
  keyDeposit <- getsPParams ppKeyDepositL
  (_, addr1) <- freshKeyAddr
  input1 <- sendCoinTo addr1 (Coin 10_000_000)
  (_, addr2) <- freshKeyAddr
  input2 <- sendCoinTo addr2 (Coin 10_000_000)
  let
    subTx1 = mkBasicTx mkBasicTxBody
      & bodyTxL . inputsTxBodyL .~ Set.singleton input1
      & bodyTxL . certsTxBodyL  .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = mkBasicTx mkBasicTxBody
      & bodyTxL . inputsTxBodyL .~ Set.singleton input2
      & bodyTxL . certsTxBodyL  .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = mkBasicTx mkBasicTxBody
      & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  -- Without the fix this succeeds, draining keyDeposit twice from utxosDeposited
  submitTx_ tx
  -- utxosDeposited has been decremented by 2 * keyDeposit despite only 1 deposit paid
