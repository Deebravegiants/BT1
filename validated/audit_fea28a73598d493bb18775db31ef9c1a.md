### Title
Deposit Refund in Meta-Transactions Sent to Sender Instead of Relayer on Inner Receipt Failure - (File: runtime/runtime/src/actions.rs)

### Summary
When a meta-transaction (`DelegateAction`) inner receipt fails on the receiver's shard, the attached deposit is refunded to the **sender** (Alice) rather than the **relayer** who actually paid for it. An unprivileged user can deliberately craft a `DelegateAction` with a large deposit whose inner action is guaranteed to fail, causing the relayer to permanently lose the deposit while the user gains it.

### Finding Description

In `apply_delegate_action`, the inner receipt spawned from a `DelegateAction` is constructed with `predecessor_id` set to `sender_id` (Alice):

```rust
let new_receipt = Receipt::V0(ReceiptV0 {
    predecessor_id: sender_id.clone(),   // ← Alice, not the relayer
    receiver_id: delegate_action.receiver_id().clone(),
    ...
});
``` [1](#0-0) 

When this inner receipt fails on the receiver's shard, `refund_unspent_gas_and_deposits` issues the deposit refund to `receipt.balance_refund_receiver()`:

```rust
if deposit_refund > Balance::ZERO {
    result.new_receipts.push(Receipt::new_balance_refund(
        receipt.balance_refund_receiver(),
        deposit_refund,
    ));
}
``` [2](#0-1) 

`balance_refund_receiver()` returns `predecessor_id()` of the receipt — which is Alice, not the relayer:

```rust
pub fn balance_refund_receiver(&self) -> &AccountId {
    self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
}
``` [3](#0-2) 

Yet the deposit was charged from the **relayer** at transaction-validation time. The `total_deposit` function even contains an inline comment acknowledging this mismatch:

```rust
// Note, here Relayer pays the deposit but if actions fail, the deposit is
// refunded to Sender of DelegateAction
``` [4](#0-3) 

The architecture documentation also explicitly acknowledges the financial incentive this creates:

> "there is a financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard." [5](#0-4) 

### Impact Explanation

The corrupted protocol value is the **account balance** of both parties:

- The relayer's `amount` field in state is reduced by the full deposit (they paid it at transaction time and never receive it back).
- Alice's `amount` field in state is increased by the same deposit (she receives the refund receipt from `system`).

This is a direct, permanent balance transfer from the relayer to Alice, triggered by a valid on-chain transaction that executes successfully at the protocol level. No consensus divergence occurs — the state transition is correct by the protocol's own rules — but the economic invariant that "whoever pays the deposit receives the refund on failure" is broken for meta-transactions.

### Likelihood Explanation

The attack requires no special privileges. Any NEAR account holder (Alice) can:

1. Construct a `DelegateAction` whose inner action targets a non-existent method or a contract that is guaranteed to reject the call, with a large `deposit` attached.
2. Submit the signed `DelegateAction` to any public relayer service.
3. The relayer wraps it in a transaction, pays the deposit, and submits it.
4. The inner receipt fails on the receiver's shard; the deposit is refunded to Alice.

The attack is repeatable, requires no validator or node-operator access, and is executable through the standard public RPC transaction path.

### Recommendation

When `apply_delegate_action` constructs the inner receipt, it should either:

1. Set `predecessor_id` to the relayer's account ID (the outer receipt's `predecessor_id`) so that deposit refunds flow back to the relayer on failure, **or**
2. Use the `ActionReceiptV2.refund_to` field (already present in the codebase) to explicitly redirect deposit refunds to the relayer (`action_receipt.signer_id()` or the outer `predecessor_id`), overriding the default `predecessor_id`-based routing.

The `refund_to` field on `ActionReceiptV2` was introduced precisely for this kind of redirection: [6](#0-5) 

### Proof of Concept

**Attacker-controlled entry path:**

1. Alice creates a `DelegateAction` with `receiver_id = bob.near`, inner action = `FunctionCall { method_name: "nonexistent", deposit: 10 NEAR, gas: 30 TGas }`.
2. Alice signs it and sends it off-chain to a relayer.
3. Relayer wraps it: `Transaction { signer: relayer, receiver: alice, actions: [Delegate(signed_delegate)] }`. The relayer's account is charged `10 NEAR` deposit + gas at transaction validation time (`total_deposit` recurses into the inner actions).
4. On Alice's shard, `apply_delegate_action` fires, creating an inner receipt with `predecessor_id = alice`, `receiver_id = bob`.
5. On Bob's shard, the function call fails (`MethodNotFound`). `refund_unspent_gas_and_deposits` issues `Receipt::new_balance_refund(alice, 10 NEAR)`.
6. Alice's balance increases by 10 NEAR. The relayer's balance is permanently reduced by 10 NEAR.

The exact corrupted DB entries are the `Account.amount` fields for both `alice.near` and the relayer account in the NEAR state trie, producing a state root that reflects an incorrect balance distribution relative to who funded the deposit.

```markdown
## Balance refunds in meta transactions

Unlike gas refunds, the protocol sends balance refunds to the predecessor
(a.k.a. sender) of the receipt. This makes sense, as we deposit the attached
balance to the receiver, who has to explicitly reattach a new balance to new
receipts they might spawn.

In the world of meta transactions, this assumption is also challenged. If an
inner action requires an attached balance (for example a transfer action) then
this balance is taken from the relayer.

The relayer can see what the cost will be before submitting the meta transaction
and agrees to pay for it, so nothing wrong so far. But what if the transaction
fails execution on Bob's shard? At this point, the predecessor is `Alice` and
therefore she receives the token balance refunded, not the relayer. This is
something relayer implementations must be aware of since there is a financial
incentive for Alice to submit meta transactions that have high balances attached
but will fail on Bob's shard.
```
