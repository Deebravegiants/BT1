Audit Report

## Title
Missing Staleness Validation on Chainlink `latestRoundData()` Enables Over-Minting of rsETH — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` but silently discards `updatedAt`, `answeredInRound`, and `roundId`, retaining only the raw `answer`. A stale LST/ETH Chainlink feed will return the last recorded price without any revert, causing `LRTDepositPool.getRsETHAmountToMint()` to compute an inflated rsETH output. Any unprivileged depositor can exploit this window to receive more rsETH than the deposited collateral is worth, diluting all existing rsETH holders and moving the protocol toward insolvency.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at line 52 destructures the five return values of `latestRoundData()` but binds only `price`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol  L52
(, int256 price,,,) = priceFeed.latestRoundData();
```

The discarded values are `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`. None of the following guards are present:
- `require(answeredInRound >= roundId)` — round-completeness check
- `require(updatedAt > 0)` — non-zero timestamp check
- `require(block.timestamp - updatedAt <= maxStaleness)` — time-based freshness check
- `require(price > 0)` — positive price check

This price flows directly into `LRTDepositPool.getRsETHAmountToMint()` at line 520:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Which is called by the public `depositAsset()` (L99–118) and `depositETH()` (L76–93) functions, both accessible by any address with no role restriction.

The same codebase's `ChainlinkOracleForRSETHPoolCollateral` (L27–32) demonstrates the team is aware of this requirement — it explicitly checks `answeredInRound < roundID` and `timestamp == 0` — yet the mainnet `ChainlinkPriceOracle` has none of these guards.

## Impact Explanation

**Critical — Direct theft of user funds / Protocol insolvency.**

When a Chainlink LST/ETH feed goes stale while the real market price of the LST drops (e.g., a depeg event), the contract continues to use the last recorded (higher) price. `getRsETHAmountToMint` returns a value larger than the deposited collateral justifies. The attacker receives excess rsETH backed by insufficient collateral. Selling or redeeming this excess rsETH extracts value from all existing rsETH holders and, at scale, drives the protocol toward insolvency. The impact is concrete, not hypothetical: the attacker receives more rsETH than the deposited assets are worth, constituting direct theft of funds from existing holders.

## Likelihood Explanation

Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for stETH/ETH on mainnet). Staleness occurs during Ethereum network congestion, Chainlink node outages, or extreme market volatility — precisely the conditions under which LST prices move most sharply. An attacker only needs to monitor the on-chain `updatedAt` field of the relevant Chainlink feed. No privileged access, no governance capture, and no oracle operator compromise is required. `depositAsset` is callable by any EOA or contract. The attack is repeatable for every staleness window.

## Recommendation

Add staleness validation inside `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
uint256 public maxStaleness; // configurable per-asset by LRTManager

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(answeredInRound >= roundId, "Stale round");
    require(updatedAt > 0, "Incomplete round");
    require(block.timestamp - updatedAt <= maxStaleness, "Stale price");
    require(price > 0, "Invalid price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`maxStaleness` should be set per-asset to match each feed's documented heartbeat and be configurable by the LRT manager role, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`.

## Proof of Concept

1. Chainlink stETH/ETH feed on mainnet has a 1-hour heartbeat. The feed has not updated for 90 minutes (stale); last recorded price is `1.00 ETH` per stETH.
2. During this window, stETH depegs to `0.90 ETH` on the open market.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint` computes: `rsethAmountToMint = (1000e18 * 1.00e18) / rsETHPrice` — using the stale `1.00` price instead of the real `0.90`.
5. Attacker receives ~11% more rsETH than the deposited stETH is currently worth.
6. Attacker sells or redeems the excess rsETH, extracting value from existing holders.
7. No admin action, no privileged key, and no oracle operator compromise is required.

**Foundry fork test outline:**
- Fork mainnet at a block where the stETH/ETH Chainlink feed is stale (or mock `latestRoundData` to return an `updatedAt` > 1 hour in the past with a price above current market).
- Call `depositAsset(stETH, amount, 0, "")` as an unprivileged address.
- Assert that `rsethAmountToMint` exceeds the fair value computed using the current market price, confirming over-minting with no revert. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
