### Title
Harvester `farmer_reward_address_override` Accepted Without Enforcement in `_process_respond_signatures`, Enabling Unconditional Block Reward Diversion - (`chia/farmer/farmer_api.py`)

---

### Summary

Any harvester connected to a farmer can set `farmer_reward_address_override` to an arbitrary puzzle hash in its `RespondSignatures` message. The farmer's `_process_respond_signatures` unconditionally substitutes this address for the farmer's configured reward address with no enforcement of the CHIP-22 fee quality convention and no authorization check. This allows a malicious third-party harvester to redirect 100% of the farmer's XCH block rewards to an attacker-controlled address.

---

### Finding Description

The vulnerability class from the external report is **auth bypass via a missing "only-once" or access-control guard on a privileged state-setting function**. In Chia, the analog is a **missing enforcement guard on a privileged reward-address override** in the farmer-harvester protocol.

**The two-step protocol flow:**

**Step 1 — `new_proof_of_space` (farmer_api.py lines 128–129):**
When a harvester sends `NewProofOfSpace` with `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`. This function computes a fee quality score and logs a warning if the threshold is not met — but it does **not** block the flow or record a rejection decision. [1](#0-0) 

**Step 2 — `_process_respond_signatures` (farmer_api.py lines 916–919):**
When the harvester later sends `RespondSignatures`, the farmer unconditionally applies `farmer_reward_address_override` if it is non-`None`, with **no fee quality check, no authorization check, and no reference to any prior decision from Step 1**: [2](#0-1) 

The `farmer_reward_address_override` field in `RespondSignatures` is a separate field from the one in `NewProofOfSpace`: [3](#0-2) 

**The bypass path:** A harvester can set `farmer_reward_address_override=None` in `NewProofOfSpace` (skipping the fee quality logging entirely) and then set `farmer_reward_address_override=<attacker_address>` in `RespondSignatures`. The farmer applies the override with no check whatsoever.

Even if the harvester sets the override in `NewProofOfSpace`, the fee quality check in `notify_farmer_reward_taken_by_harvester_as_fee` is purely advisory — it only logs a warning and never returns a value that gates the override: [4](#0-3) 

The resulting `DeclareProofOfSpace` message sent to the full node carries the attacker's address as `farmer_reward_address`: [5](#0-4) 

---

### Impact Explanation

**High — Unauthorized payout redirection of XCH block rewards.**

Any harvester connected to the farmer (including third-party harvesters explicitly supported by CHIP-22) can redirect the farmer's block reward coin — the 0.25 XCH farmer reward per block — to an arbitrary puzzle hash. Since the override is applied unconditionally, a malicious harvester can redirect 100% of rewards, not just the agreed fee percentage. This is a direct, permanent loss of XCH for every block won while the malicious harvester is connected.

---

### Likelihood Explanation

**Medium-High.** Third-party harvesters are an explicitly supported and common deployment pattern (CHIP-22, DrPlotter, etc.). Any operator running a third-party harvester is exposed. The attacker only needs a valid SSL connection to the farmer — no private keys, no admin access. The attack is silent (the farmer logs a warning only if `farmer_reward_address_override` was set in `NewProofOfSpace`, and nothing at all if it was only set in `RespondSignatures`).

---

### Recommendation

1. **Enforce the fee quality check as a gate, not just a log.** In `new_proof_of_space`, if `farmer_reward_address_override` is set and the fee quality does not satisfy the threshold, reject the proof (return early) rather than logging a warning and continuing.

2. **Validate `RespondSignatures.farmer_reward_address_override` against the override declared in `NewProofOfSpace`.** If the harvester did not declare an override in `NewProofOfSpace`, reject any override in `RespondSignatures`. Store the validated override (or `None`) in `proofs_of_space` alongside the proof, and use only that stored value in `_process_respond_signatures`.

3. **Add a configurable farmer-side allowlist** for which harvesters are permitted to use the fee override mechanism at all.

---

### Proof of Concept

**Attacker:** A third-party harvester with a valid SSL connection to the farmer.

**Steps:**

1. Harvester sends `NewProofOfSpace` with `farmer_reward_address_override=None` and a valid proof. No fee quality check is triggered. [1](#0-0) 

2. Farmer sends `RequestSignatures` to the harvester (normal flow).

3. Harvester responds with `RespondSignatures` where `farmer_reward_address_override=<attacker_puzzle_hash>` (any 32-byte value). [3](#0-2) 

4. Farmer's `_process_respond_signatures` applies the override unconditionally:

   ```python
   farmer_reward_address = self.farmer.farmer_target
   if response.farmer_reward_address_override is not None:
       farmer_reward_address = response.farmer_reward_address_override  # attacker address
   ``` [2](#0-1) 

5. Farmer broadcasts `DeclareProofOfSpace` with `farmer_reward_address=<attacker_puzzle_hash>` to the full node. The block is created with the attacker's address as the farmer reward recipient. The 0.25 XCH farmer reward coin is created at the attacker's puzzle hash. [5](#0-4) 

No warning is logged. No rejection occurs. The farmer's XCH reward is permanently diverted.

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
