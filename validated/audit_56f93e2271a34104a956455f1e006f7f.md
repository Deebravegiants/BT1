### Title
Missing Value-Conservation Check in Sub-Transaction UTXO Rule Enables ADA Creation — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The Dijkstra era's `SUBUTXO` rule (`dijkstraSubUtxoTransition`) omits the `validateValueNotConservedUTxO` predicate that is mandatory in every top-level UTXO rule. Because `updateUTxOStateNoFees` mechanically removes inputs and adds outputs without verifying that their ADA values balance, a sub-transaction can produce outputs whose total value exceeds its consumed inputs, creating ADA from nothing. The inflated UTxO state is then handed to the top-level `UTXOW` rule, whose own value-conservation check only covers the top-level transaction body, leaving the sub-transaction imbalance permanently undetected.

---

### Finding Description

**Vulnerability class:** Missing validation / incomplete ledger rule — direct analog of the "empty/absent unit test" pattern from the reference report, where a critical invariant is simply never checked.

**Affected rule chain:**

```
LEDGER (dijkstraLedgerTransition)
  └─ SUBLEDGERS (dijkstraSubLedgersTransition)
       └─ SUBLEDGER (dijkstraSubLedgersTransition)
            └─ SUBUTXOW
                 └─ SUBUTXO (dijkstraSubUtxoTransition)   ← missing check
```

**Why the top-level check does not compensate:**

After all sub-transactions are processed, the Dijkstra LEDGER rule passes the *post-sub-transaction* UTxO state as the `State` argument to `UTXOW`: [5](#0-4) 

The top-level UTXO rule's value-conservation check therefore operates on the already-inflated UTxO. It verifies only that the top-level transaction body is self-consistent (`consumed_top = produced_top`); it has no visibility into whether the sub-transactions that produced the current UTxO state were themselves balanced.

The `SUBLEDGERS` rule iterates sub-transactions sequentially, each one feeding its output UTxO state into the next: [6](#0-5)

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker who can author a Dijkstra-era transaction with embedded sub-transactions can craft a sub-transaction whose outputs exceed its inputs by an arbitrary amount of Lovelace. `updateUTxOStateNoFees` inserts those outputs into the live UTxO without complaint. The top-level transaction then spends those outputs; its own value-conservation check passes because it is self-balanced. The net effect is that ADA is permanently created in the ledger state with no corresponding destruction, violating the global preservation-of-value invariant that is the foundational security property of the Cardano ledger.

---

### Likelihood Explanation

The Dijkstra era introduces sub-transactions as a new, unprivileged feature available to any transaction author. No special role, key, or governance threshold is required. The attacker only needs to submit a well-formed Dijkstra-era transaction containing a sub-transaction with unbalanced outputs. The missing check is structural (absent from the rule, not gated behind a flag), so it is reachable on every Dijkstra-era block.

---

### Recommendation

Add a value-conservation predicate to `dijkstraSubUtxoTransition` analogous to `validateValueNotConservedUTxO` used in the top-level UTXO rule. The check must verify:

```
Σ(value of inputs consumed from UTxO)
  + withdrawals + refunds
= Σ(value of outputs produced)
  + deposit_change
```

(Fees are correctly excluded for sub-transactions via `updateUTxOStateNoFees`, but value conservation must still hold.) Remove the `error "Impossible: ValueNotConservedUTxO for SUBUTXO"` branch and replace it with a proper `SubValueNotConservedUTxO` failure constructor, mirroring the top-level pattern.

Additionally, add a `PostCondition` assertion to the Dijkstra LEDGER rule that verifies total ADA in the UTxO + deposits + fees + treasury equals the pre-transaction total, catching any future regressions of this class.

---

### Proof of Concept

1. Construct a Dijkstra-era top-level transaction `T` that contains one sub-transaction `S`.
2. In `S`, include one input `i` worth 10 ADA and two outputs worth 15 ADA total (5 ADA excess).
3. Submit `T`. `dijkstraSubUtxoTransition` runs for `S`:
   - All present checks pass (valid interval, non-empty inputs, inputs exist in UTxO, etc.).
   - `updateUTxOStateNoFees` removes `i` (10 ADA) and inserts the two outputs (15 ADA) — no balance check.
   - UTxO state now contains 5 ADA more than before.
4. The top-level transaction `T` spends one of `S`'s outputs (5 ADA) and produces a 5 ADA output to the attacker's address. Its own value-conservation check passes.
5. Result: the ledger has permanently gained 5 ADA. Repeating this across many transactions allows unbounded ADA inflation. [8](#0-7)
