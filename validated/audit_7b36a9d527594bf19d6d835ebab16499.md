Audit Report

## Title
`LRTWithdrawalManager._unlockWithdrawalRequests` Burns Full rsETH Amount When Payout Is Capped at `expectedAssetAmount` - (File: contracts/LRTWithdrawalManager.sol)

## Summary
In `_unlockWithdrawalRequests`, when the rsETH/asset exchange rate moves favorably for rsETH between withdrawal initiation and unlock, `_calculatePayoutAmount` caps the asset payout at `expectedAssetAmount`. However, the full `request.rsETHUnstaked` is unconditionally added to `rsETHAmountToBurn` and burned, causing the user to burn more rsETH value than the assets they receive.

## Finding Description
When a user calls `initiateWithdrawal`, their rsETH is transferred to the contract and `expectedAssetAmount` is recorded at current oracle prices. [1](#0-0) 

During `unlockQueue`, `_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`: [2](#0-1) 

When `currentReturn > expectedAssetAmount` (i.e., rsETH has appreciated relative to the asset), `payoutAmount` is capped. Despite this cap, line 805 unconditionally accumulates the full original rsETH amount: [3](#0-2) 

The full accumulated amount is then burned: [4](#0-3) 

The fair rsETH cost of `expectedAssetAmount` assets at unlock time is `expectedAssetAmount * assetPrice / rsETHPrice`, which is strictly less than `request.rsETHUnstaked` when `currentReturn > expectedAssetAmount`. The excess rsETH (`rsETHUnstaked - expectedAssetAmount * assetPrice / rsETHPrice`) is burned rather than returned to the user.

## Impact Explanation
The withdrawing user loses the rsETH value equivalent to `currentReturn - expectedAssetAmount` assets. The burned excess rsETH reduces total supply, benefiting remaining rsETH holders at the direct expense of the withdrawing user. The user receives exactly the asset amount they expected, but their rsETH — which has appreciated — is burned in full rather than proportionally. This matches **Low**: the contract fails to deliver the promised fair rsETH-to-asset exchange return, but does not result in direct theft or protocol insolvency.

## Likelihood Explanation
This condition triggers whenever the rsETH/asset exchange rate moves favorably for rsETH between `initiateWithdrawal` and `unlockQueue`. This is a normal market scenario (stETH depeg, ETH price movement, rsETH yield accrual) requiring no attacker action. Any ordinary user who initiates a withdrawal before a favorable rate shift is affected. The operator calling `unlockQueue` need not be malicious; the loss occurs mechanically for every affected request in the queue.

## Recommendation
When `payoutAmount < currentReturn`, burn only the rsETH proportional to the actual payout:

```solidity
uint256 rsETHToBurnForRequest = (payoutAmount < currentReturn)
    ? (payoutAmount * 1e18 / rsETHPrice) * assetPrice / 1e18  // proportional burn
    : request.rsETHUnstaked;
rsETHAmountToBurn += rsETHToBurnForRequest;
```

The remainder (`request.rsETHUnstaked - rsETHToBurnForRequest`) should be transferred back to the original requester or credited for future use, rather than burned.

## Proof of Concept
1. Alice calls `initiateWithdrawal` with `rsETHUnstaked = 1000e18`. At initiation: `rsETHPrice = 1.0e18`, `assetPrice = 1.0e18`, so `expectedAssetAmount = 1000e18` stETH. Alice's 1000 rsETH is transferred to the contract.
2. Before `unlockQueue` is called, stETH depegs: `assetPrice` drops to `0.8e18`, `rsETHPrice` stays at `1.0e18`.
3. Operator calls `unlockQueue`. `currentReturn = 1000e18 * 1.0e18 / 0.8e18 = 1250e18` stETH.
4. `payoutAmount = min(1000e18, 1250e18) = 1000e18`. Alice receives 1000 stETH.
5. `rsETHAmountToBurn += 1000e18` — Alice's full 1000 rsETH is burned.
6. Fair rsETH cost of 1000 stETH at current prices: `1000e18 * 0.8e18 / 1.0e18 = 800e18` rsETH.
7. Alice loses 200 rsETH of value (burned without compensation). A Foundry fuzz test varying `assetPrice` downward after initiation and asserting `rsETHToBurn * rsETHPrice >= payoutAmount * assetPrice` would reproduce this invariant violation for any `assetPrice < initialAssetPrice`.
