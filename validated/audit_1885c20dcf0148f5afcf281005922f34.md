### Title
Sub-Transaction Reference Script Sizes Excluded from Minimum Fee Calculation in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces nested (sub-)transactions embedded within a top-level transaction. The minimum fee validation for a Dijkstra top-level transaction uses `getConwayMinFeeTxUtxo`, which only measures reference script sizes from the **top-level** transaction's inputs. Reference scripts referenced by **sub-transactions** are silently excluded from the fee calculation. A helper function `batchNonDistinctRefScriptsSize` that correctly aggregates reference script sizes across the entire batch (top-level + all sub-transactions) exists in the same file but is never wired into the fee enforcement path.

---

### Finding Description

The vulnerability class from the reference report is **incorrect size pre-allocation for a batch of items**: a limit is computed for a single item but applied to a collection, allowing the collection to exceed the intended bound.

The analog in Cardano Ledger is the minimum fee check for reference scripts in the Dijkstra era.

**Step 1 — Fee enforcement path.**

`dijkstraUtxoTransition` enforces the minimum fee at line 373:

```haskell
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
```

`validateFeeTooSmallUTxO` calls `getMinFeeTxUtxo pp tx utxo`. For `DijkstraEra`, this is bound to `getConwayMinFeeTxUtxo`: [1](#0-0) 

**Step 2 — `getConwayMinFeeTxUtxo` only measures top-level inputs.**

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

`txNonDistinctRefScriptsSize` unions only the top-level transaction's `referenceInputsTxBodyL` and `inputsTxBodyL`: [2](#0-1) 

Sub-transactions' inputs and reference inputs are never consulted.

**Step 3 — The correct batch function exists but is unused.**

`batchNonDistinctRefScriptsSize` in the same Dijkstra UTxO module correctly sums reference script sizes across the top-level transaction **and** all sub-transactions:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [3](#0-2) 

This function is exported from the module but is **not** called from `getMinFeeTxUtxo` or from `validateFeeTooSmallUTxO`. The `EraUTxO` instance for `DijkstraEra` binds `getMinFeeTxUtxo` to the Conway implementation, not to a Dijkstra-aware one: [4](#0-3) 

**Step 4 — The BBODY-level ref-script size limit is also affected.**

`dijkstraBbodyTransition` calls `Conway.validateBodyRefScriptsSizeTooBig`, which is a Conway-era function that predates sub-transactions and therefore also uses per-top-level-tx accounting, not batch accounting: [5](#0-4) 

---

### Impact Explanation

The reference script fee mechanism was introduced in Conway specifically to prevent DDoS attacks where large scripts are cheap to include but expensive for nodes to deserialize and validate (see ADR 009). The tiered pricing model in `tierRefScriptFee` is designed to make large reference script usage prohibitively expensive: [6](#0-5) 

By placing large reference scripts exclusively in sub-transactions, an attacker pays only the fee for the top-level transaction's (potentially zero) reference scripts, while the node must still deserialize and validate all sub-transaction reference scripts. This allows an attacker to submit transactions whose actual validation cost to nodes is arbitrarily higher than the fee paid, bypassing the fee parameter `minFeeRefScriptCostPerByte` and the hard cap `maxRefScriptSizePerTx`/`maxRefScriptSizePerBlock` for the sub-transaction portion of the batch.

This maps to the allowed impact: **Medium — attacker-controlled transactions modify fees outside design parameters.**

---

### Likelihood Explanation

The Dijkstra era is the current development era and sub-transactions are a new feature. The discrepancy is introduced by reusing `getConwayMinFeeTxUtxo` unchanged for `DijkstraEra` despite the new batch semantics. The existence of `batchNonDistinctRefScriptsSize` in the same file confirms the developers are aware of the batch accounting requirement; the function simply was not connected to the enforcement path. Any unprivileged transaction sender can exploit this by constructing a top-level transaction with sub-transactions that carry large reference inputs.

---

### Recommendation

Replace the `getMinFeeTxUtxo` binding in the `EraUTxO DijkstraEra` instance with a Dijkstra-specific implementation that calls `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
-- In EraUTxO DijkstraEra instance:
getMinFeeTxUtxo pp tx utxo =
  getMinFeeTx pp tx (batchNonDistinctRefScriptsSize utxo tx)
```

Similarly, `validateBodyRefScriptsSizeTooBig` in `dijkstraBbodyTransition` should be replaced or extended with a Dijkstra-aware variant that sums reference script sizes across all sub-transactions in each top-level transaction before comparing against `maxRefScriptSizePerBlock`.

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction with **no** reference inputs at the top level (so `txNonDistinctRefScriptsSize` returns 0).
2. Embed one or more sub-transactions, each with `referenceInputsTxBodyL` pointing to UTxO entries that carry large Plutus reference scripts (e.g., near the 200 KiB per-tx limit each).
3. Set the top-level transaction fee to the minimum computed by `getConwayMinFeeTxUtxo` (which ignores sub-tx ref scripts).
4. Submit the transaction. `validateFeeTooSmallUTxO` passes because it only measures the top-level ref script size (0 bytes). The node must still deserialize all sub-transaction reference scripts, incurring validation cost far exceeding the fee paid.

getConwayMinFeeTxUtxo ::
  ( EraTx era
  , BabbageEraTxBody era
  ) =>
  PParams era ->
  Tx l era ->
  UTxO era ->
  Coin
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx

-- | Calculate the total size of reference scripts used by the transactions. Duplicate
-- scripts will be counted as many times as they occur, since there is never a reason to
-- include an input with the same reference script.
--
-- Any input that appears in both regular inputs and reference inputs of a transaction is
-- only used once in this computation.
txNonDistinctRefScriptsSize :: (EraTx era, BabbageEraTxBody era) => UTxO era -> Tx l era -> Int
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```
