### Title
Deposit Funds Permanently Locked in Contract When Competing Update Proposals Are Cleared by `do_update` - (File: `crates/contract/src/update.rs`)

### Summary

When `ProposedUpdates::do_update` executes a winning update proposal, it unconditionally clears **all** pending proposals from storage. However, the NEAR token storage deposits paid by proposers of the cleared (non-executed) proposals are never tracked or refunded, permanently locking those funds inside the MPC contract. This is the direct analog of the `AgentPolice.distributeLiquidatedFunds()` residual-funds issue: a settlement/distribution operation consumes only what it needs and silently discards the rest.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires each proposer to attach a deposit covering the storage cost of their update entry:

```rust
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// ...
let id = self.proposed_updates.propose(update);
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

The deposit is sized by `bytes_used`, which for a contract-code update includes the full WASM blob:

```rust
fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;
    match update {
        Update::Contract(code) => { bytes_used += code.len() as u128; }
        // ...
    }
    bytes_used
}
```

At NEAR's storage rate (~10^19 yoctoNEAR/byte), a 500 KB WASM proposal requires roughly **5 NEAR** in deposit.

When `vote_update` reaches threshold it calls `do_update`, which clears every pending entry:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ...
}
```

The storage is freed, but **no deposit is stored per entry and no refund is issued** to the proposers of the cleared proposals. The NEAR tokens are permanently absorbed by the contract with no recovery path.

---

### Impact Explanation

Every participant whose proposal is cleared by a competing `do_update` permanently loses their storage deposit. For contract-code proposals this is proportional to the WASM size — potentially several NEAR per proposal. The contract has no `cancel_update`, `withdraw_deposit`, or equivalent method, and `UpdateEntry` does not record the depositor's account ID, making post-hoc recovery impossible without a contract migration.

This breaks the production accounting invariant that storage deposits must be returned when the corresponding storage is freed.

---

### Likelihood Explanation

Multiple concurrent update proposals are a normal governance scenario: different participants may propose different contract versions or config changes simultaneously. Whenever any one proposal reaches threshold, all others are silently cleared and their deposits are lost. No adversarial coordination is required — the loss occurs as a side-effect of ordinary threshold governance. A Byzantine participant at or above threshold can also deliberately trigger this to cause honest proposers to lose their deposits.

---

### Recommendation

Store the depositor's `AccountId` and the deposit amount inside `UpdateEntry`. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(depositor).transfer(deposit)` for each one. This mirrors the pattern already used in `propose_update` itself for excess-deposit refunds.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM, attaching ~5 NEAR deposit. Entry `id=0` is stored.
2. Participant B calls `propose_update` with a different WASM, attaching ~5 NEAR deposit. Entry `id=1` is stored.
3. Threshold participants vote for `id=1` via `vote_update`.
4. `do_update(&id=1, gas)` executes: removes entry `id=1`, then calls `self.entries.clear()` which removes entry `id=0`.
5. Participant A's ~5 NEAR deposit is now permanently locked in the contract. No refund is issued. No recovery path exists.

fn bytes_used(update: &Update) -> u128 {

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

            bytes_used += bytes.len() as u128;
        }

    bytes_used

}
