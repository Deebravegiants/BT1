### Title
Stale TEE Governance Votes from Removed Participants Counted Toward Threshold in Post-Resharing Window — (`crates/contract/src/lib.rs`, `crates/contract/src/tee/proposal.rs`)

---

### Summary

After resharing completes in `vote_reshared()`, the contract immediately transitions to the new `Running` state with the updated participant set, but defers cleanup of stale TEE governance votes (from removed participants) to a **detached promise** (`clean_tee_status`) that executes in a later block. In the window between resharing completion and cleanup, `vote_code_hash()`, `vote_add_launcher_hash()`, and `vote_add_os_measurement()` count **all** votes in their respective maps — including stale votes from removed participants — without filtering by the current participant set. This allows the governance threshold to be satisfied with fewer legitimate current-participant votes than required.

---

### Finding Description

**Root cause — stale vote maps used before cleanup:**

In `vote_reshared()`, once all domains have been reshared, the contract transitions to the new Running state and spawns `clean_tee_status` as a **detached** promise:

```rust
// lib.rs:1170-1193
if let Some(new_state) = self.protocol_state.vote_reshared(key_event_id)? {
    self.protocol_state = new_state;          // ← new participant set is live
    self.recompute_available_foreign_chains();

    Promise::new(env::current_account_id())
        .function_call(
            method_names::CLEAN_TEE_STATUS.to_string(), ...
        )
        .detach();                            // ← cleanup runs in a LATER block
``` [1](#0-0) 

The detached promise calls `clean_tee_status()`, which invokes `tee_state.clean_non_participant_votes(participants)` to purge stale entries:

```rust
// lib.rs:1807-1818
pub fn clean_tee_status(&mut self) -> Result<(), Error> {
    let participants = match &self.protocol_state {
        ProtocolContractState::Running(state) => state.parameters.participants(),
        ...
    };
    self.tee_state.clean_non_participant_votes(participants);
    Ok(())
}
``` [2](#0-1) 

**Stale votes counted without filtering:**

`vote_code_hash()` authenticates the *caller* against the new participant set, but the vote count returned by `CodeHashesVotes::count_votes()` iterates **all** entries in `proposal_by_account` — including stale entries from removed participants — without any participant-set filter:

```rust
// lib.rs:1407-1431
pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
    let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
    let votes = self.tee_state.vote(code_hash, &participant);  // counts ALL entries
    if votes >= self.threshold()?.value() {
        self.tee_state.whitelist_tee_proposal(code_hash, ...);
    }
    Ok(())
}
``` [3](#0-2) 

```rust
// proposal.rs:47-52
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64   // no participant-set filter

``` [4](#0-3) 

The same pattern applies to `vote_add_launcher_hash()` and `vote_add_os_measurement()`.

**The developers are aware of this class of bug for `vote_update` but not for TEE governance votes:**

The comment in `vote_reshared` explicitly notes that `vote_update` uses filtering to handle the case where cleanup fails:

> "Note: MpcContract::vote_update uses filtering to ensure correctness even if this cleanup fails." [5](#0-4) 

No equivalent filtering exists in `vote_code_hash`, `vote_add_launcher_hash`, or `vote_add_os_measurement`.

**The codebase's own test documents the exact stale-vote counting behavior:**

```
// tee_state.rs:1496-1544
// Scenario (N=5, T=3):
// 1. P1 and P2 vote for malicious hash before resharing.
// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
// 3. clean_non_participant_votes removes stale votes.
// 4. P3 votes for the same hash — only 1 vote, not 3.
``` [6](#0-5) 

Without step 3, P3's single vote would be counted as 3 (2 stale + 1 new), meeting the threshold of 3.

---

### Impact Explanation

An adversary who controls participants that are subsequently removed via resharing can pre-vote for a malicious TEE Docker image hash (or launcher hash / OS measurement) before resharing. After resharing completes — but before `clean_tee_status` executes in the next block — a single legitimate new participant voting for the same hash causes `count_votes()` to return a total that includes the stale votes, satisfying the threshold. The malicious hash is then whitelisted.

A whitelisted malicious image hash allows a node running adversary-controlled code to submit a valid TEE attestation and be admitted to the participant set in a future resharing. Once admitted, the adversary-controlled node participates in threshold signing, enabling unauthorized signature issuance or key-share exposure.

This breaks the production safety invariant that whitelisting a TEE image hash requires threshold votes exclusively from **current** participants.

---

### Likelihood Explanation

- The attacker must have controlled participants that were removed in a resharing — a realistic scenario (e.g., a participant is removed for misbehavior or rotation).
- The attacker must have pre-voted for a malicious hash before resharing — trivially achievable by any participant.
- A single new participant must vote for the same hash in the 1–2 block window before `clean_tee_status` runs — this window is short but deterministic and observable on-chain; an attacker who also controls or can influence a new participant can time this.
- No flash-loan or network-level attack is required; all actions are normal contract calls.

---

### Recommendation

Apply the same filtering approach used by `vote_update` directly inside `vote_code_hash`, `vote_add_launcher_hash`, and `vote_add_os_measurement`: count only votes from accounts that are in the **current** participant set at the time of the call, rather than counting all entries in the vote map. Alternatively, move the `clean_non_participant_votes` call to execute **synchronously** within `vote_reshared` before the new Running state is committed, mirroring the fix applied to the Zivoe `distributeYield` bug (update state before using it in calculations).

---

### Proof of Concept

**Setup:** N=5, T=3. Attacker controls P1 and P2.

1. P1 calls `vote_code_hash(malicious_hash)` → 1 vote stored.
2. P2 calls `vote_code_hash(malicious_hash)` → 2 votes stored. Threshold not met (2 < 3).
3. Governance votes to reshard, removing P1 and P2. New set: {P3, P4, P5}, T=3.
4. The final `vote_reshared` call transitions `protocol_state` to the new Running state and spawns `clean_tee_status` as a detached promise (executes next block).
5. **In the same block**, P3 calls `vote_code_hash(malicious_hash)`.
6. `count_votes(malicious_hash)` returns 3 (P1 stale + P2 stale + P3 new).
7. `3 >= 3` (threshold) → `whitelist_tee_proposal(malicious_hash)` executes.
8. `clean_tee_status` runs in the next block and removes P1/P2 votes — but the hash is already whitelisted.

The malicious hash is now in `allowed_docker_image_hashes` with only 1 legitimate vote from the current participant set.

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
