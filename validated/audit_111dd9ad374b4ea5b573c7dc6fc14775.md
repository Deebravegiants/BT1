Audit Report

## Title
Missing Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Stale-Price Exploitation - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt` and `answeredInRound`, accepting stale Chainlink answers as current. Because the stale per-asset price feeds directly into both the deposit mint formula and the withdrawal redemption formula, an unprivileged attacker can extract value from the protocol whenever any supported LST feed goes stale. The same codebase already implements the correct staleness checks in `ChainlinkOracleForRSETHPoolCollateral`, confirming this is a known requirement that was omitted from the primary oracle.

## Finding Description

**Root cause — `contracts/oracles/ChainlinkPriceOracle.sol` L52:**

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

All five return values (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`) are available but only `answer` is consumed. Neither `block.timestamp - updatedAt > heartbeat` nor `answeredInRound < roundId` is checked.

**Contrast with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` L27–32**, which correctly validates:

```solidity
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
```

**Exploit path (deposit):**

1. Chainlink stETH/ETH feed goes stale; last reported price is `1.05e18` (true market: `1.00e18`).
2. `rsETHPrice` in storage still reflects the last honest update (e.g., `1.00e18`).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18)`.
4. `getRsETHAmountToMint` computes: `(100e18 × 1.05e18) / 1.00e18 = 105 rsETH` — 5 rsETH more than fair value.
5. Attacker redeems or sells the excess rsETH, extracting value from all existing holders.

**Exploit path (withdrawal):**

If the stale price is *deflated* (e.g., `0.95e18`), `getExpectedAssetAmount` computes `amount × rsETHPrice / 0.95e18`, yielding more LST per rsETH than is fair, directly draining the deposit pool.

**Why the `pricePercentageLimit` guard does not mitigate this:**

`_updateRsETHPrice()` checks `newRsETHPrice > highestRsethPrice` against a percentage threshold, but:
- The deposit path calls `lrtOracle.getAssetPrice(asset)` *live* at deposit time — it does not go through `_updateRsETHPrice()` and is not subject to the threshold guard.
- `pricePercentageLimit` may be `0` (unset), disabling the check entirely.
- Even when set, a stale price within the configured limit passes unchecked.

## Impact Explanation

**Critical — Direct theft of user funds.**

An attacker can mint rsETH at an inflated rate (or redeem LST at a deflated rate) whenever any supported LST Chainlink feed is stale. The excess rsETH minted represents a direct transfer of value from existing rsETH holders to the attacker. Both entry points (`depositAsset` and the withdrawal claim flow) are permissionless and reachable by any external user.

## Likelihood Explanation

Chainlink feeds on Ethereum mainnet operate on heartbeat intervals (typically 1 hour for LST/ETH feeds). Feeds can fail to update due to network congestion, RPC node issues, or oracle node failure — all historically observed events. No privileged access, governance capture, or oracle operator compromise is required; the attacker only needs to observe a stale feed and submit a standard deposit or withdrawal transaction. The affected feeds cover all production LSTs (stETH, ETHx, rETH, swETH, sfrxETH).

## Recommendation

Add staleness and round-completeness checks in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`STALENESS_THRESHOLD` should be set per feed (e.g., `3600` for 1-hour heartbeat feeds), ideally as a per-asset configurable value.
