Audit Report

## Title
Missing Try/Catch on Aave Withdrawal Causes Temporary Fund Freeze in `_processWithdrawalCompletion` - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`LRTWithdrawalManager._processWithdrawalCompletion()` calls `_withdrawFromAave()` at line 724 without any error handling. If the Aave v3 pool is paused, the `aaveWETHGateway.withdrawETH()` call at line 917 reverts, causing `completeWithdrawal()` to revert entirely. Because rsETH is already burned during `unlockQueue()` (line 305), affected users cannot recover their rsETH and cannot claim their ETH for the duration of the pause — a temporary freeze of funds.

## Finding Description

The ETH withdrawal lifecycle has two distinct phases. In `unlockQueue()`, rsETH is burned from the contract at line 305 and ETH is deposited to Aave via a `try/catch` block at lines 311–316, which silently absorbs failures. In `_processWithdrawalCompletion()` (lines 719–732), when the contract's native ETH balance is insufficient, `_withdrawFromAave(amountNeeded)` is called at line 724 with no error handling. `_withdrawFromAave()` unconditionally calls `aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this))` at line 917. If the Aave v3 pool is paused, this external call reverts, propagating the revert up through `_processWithdrawalCompletion()` and `completeWithdrawal()`.

The asymmetry is explicit: the deposit path at line 311 uses `try this.depositToAaveExternal(...) {} catch (bytes memory reason) { emit AaveDepositFailed(...); }`, while the withdrawal path at line 724 has no equivalent guard. The `emergencyWithdrawFromAave()` escape hatch at line 560 also calls `_withdrawFromAave()` directly and would equally revert during a pause, eliminating the only operator-side workaround.

## Impact Explanation

**Temporary freezing of funds (Medium).** Users whose ETH withdrawal requests were unlocked (rsETH burned) while ETH was held in Aave cannot call `completeWithdrawal()` while the Aave pool is paused. Their rsETH is permanently burned and cannot be re-minted. The ETH remains inaccessible until the pool is unpaused. This matches the allowed impact of "Temporary freezing of funds."

## Likelihood Explanation

The Aave v3 guardian has paused pools on Ethereum mainnet in response to real security incidents. The Aave integration is explicitly opt-in via `isAaveIntegrationEnabled`, and ETH is the primary asset routed through it. Any user whose withdrawal was unlocked while ETH was deposited in Aave is affected for the full duration of the pause. No attacker action is required — the condition is triggered by a legitimate, documented Aave guardian action.

## Recommendation

Apply the same defensive pattern used in `unlockQueue()` to the `_withdrawFromAave` call in `_processWithdrawalCompletion()`. If the Aave withdrawal fails, catch the error and check whether the contract balance is still sufficient; revert with `InsufficientLiquidityForWithdrawal` only if it is not:

```diff
 if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
     uint256 contractBalance = address(this).balance;
     if (contractBalance < request.expectedAssetAmount) {
         uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
-        _withdrawFromAave(amountNeeded);
+        try this.withdrawFromAaveExternal(amountNeeded) { } catch { }
         uint256 balanceAfter = address(this).balance;
         if (balanceAfter < request.expectedAssetAmount) {
             revert InsufficientLiquidityForWithdrawal();
         }
     }
 }
```

This mirrors the `depositToAaveExternal` pattern and ensures a paused Aave pool does not block user withdrawals when the contract holds sufficient native ETH.

## Proof of Concept

1. Deploy with `isAaveIntegrationEnabled = true`; ETH is deposited to Aave (`totalETHDepositedToAave > 0`).
2. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH is burned at line 305; ETH is deposited to Aave at line 311.
3. Aave guardian pauses the Aave v3 pool.
4. User calls `completeWithdrawal(ETH_TOKEN, ...)`.
5. `_processWithdrawalCompletion` evaluates `address(this).balance < request.expectedAssetAmount` → true.
6. `_withdrawFromAave(amountNeeded)` is called at line 724; internally `aaveWETHGateway.withdrawETH(aavePool, ...)` at line 917 reverts due to the paused pool.
7. The entire transaction reverts. The user's rsETH is already burned; they cannot withdraw ETH for the duration of the pause.

Foundry fork test: fork Ethereum mainnet at a block where the Aave v3 pool is paused (or simulate via `vm.mockCallRevert` on `aaveWETHGateway.withdrawETH`), complete steps 1–4 above, and assert that `completeWithdrawal` reverts while `address(this).balance` remains zero and the user's rsETH balance is zero.
