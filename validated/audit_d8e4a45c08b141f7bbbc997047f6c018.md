### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Single Byzantine Participant to Corrupt Confidential Key Derivation - (File: `crates/contract/src/lib.rs`)

### Summary
`respond_ckd` performs on-chain output verification only for the `AppPublicKeyPV` variant of CKD requests. For the `AppPublicKey` variant, no output check is performed. A single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with a fabricated `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept it unconditionally, delivering a corrupted confidential key derivation result to the requester.

### Finding Description
The `request_app_private_key` (start phase) stores a `CKDRequest` in `pending_ckd_requests` without committing to what the response should contain. The `respond_ckd` (end phase) is supposed to complete the operation with a correctly-computed CKD output. However, the output check is asymmetric:

```rust
// crates/contract/src/lib.rs lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the encrypted BLS output against the MPC root public key on-chain. For `AppPublicKey`, the arm is a no-op. After this block, `resolve_yields_for` immediately drains the pending queue and delivers whatever `response` was passed in: [2](#0-1) 

The caller must be an attested participant (`assert_caller_is_attested_participant_and_protocol_active`), but that is a single-node check — any one of the MPC participants qualifies: [3](#0-2) 

This is the direct analog of the external report: `request_app_private_key` (start phase) does not commit to what `respond_ckd` (end phase) must verify for the `AppPublicKey` variant, leaving the end phase without a correctness guard.

### Impact Explanation
A single Byzantine attested participant can:
1. Observe a pending `AppPublicKey` CKD request on-chain (the `CKDRequest` key is derivable from the public `request_app_private_key` call arguments).
2. Race-call `respond_ckd` with an arbitrary fabricated `CKDResponse` before the honest leader submits the real result.
3. The contract accepts it — `resolve_yields_for` drains the queue and delivers the fabricated response to the requester.
4. Honest nodes' subsequent `respond_ckd` calls fail with `RequestNotFound` because the queue was already drained.

The requester receives a fabricated confidential key. If the application uses this key to derive wallet addresses or encrypt assets, those assets become inaccessible (permanent freeze) or are sent to an address no one controls (direct loss). This matches the Critical allowed impact: *confidential key derivation output without the required participant authorization* — a single participant delivers a CKD result that bypasses the threshold requirement entirely.

### Likelihood Explanation
Any one of the current MPC participants who turns Byzantine can execute this attack. The attacker needs only:
- An active attested-participant status (already held by every MPC node).
- The ability to read the NEAR chain for pending `CKDRequest` keys (public information).
- The ability to submit a NEAR transaction faster than the honest leader — straightforward with a high-gas-price transaction or by exploiting the leader-election window.

No threshold collusion, no TEE break, and no privileged operator access is required.

### Recommendation
Add output verification for `AppPublicKey` requests in `respond_ckd`. Since `AppPublicKey` is privately verifiable (only the app can decrypt), full on-chain verification is not possible. However, the contract should at minimum:
- Require the response to carry a zero-knowledge proof of correct encryption, or
- Deprecate `AppPublicKey` in favour of `AppPublicKeyPV`, which already has `ckd_output_check` enforced on-chain.

At a minimum, document clearly that `AppPublicKey` responses are unverified on-chain and that callers must independently verify the decrypted result before using the derived key for any value-bearing operation.

### Proof of Concept
1. User calls `request_app_private_key` with `CKDRequestArgs { app_public_key: AppPublicKey(pk), ... }`.
2. Contract stores the `CKDRequest` in `pending_ckd_requests`.
3. Byzantine participant reads the `CKDRequest` key from chain state.
4. Byzantine participant calls `respond_ckd(request, fabricated_response)` — passes `assert_caller_is_attested_participant_and_protocol_active`, hits the `AppPublicKey(_) => {}` arm (no check), and `resolve_yields_for` succeeds.
5. Honest nodes' `respond_ckd` calls return `Err(RequestNotFound)`.
6. User's promise resolves with `fabricated_response`; any assets derived from or encrypted to the resulting key are lost. [4](#0-3) [5](#0-4)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
