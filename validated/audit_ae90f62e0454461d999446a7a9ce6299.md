Audit Report

## Title
`priceProviderTimelock = 0` Bypasses Oracle-Rotation Timelock, Enabling Atomic Malicious Price-Provider Swap — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`MetricOmmPoolFactory._validatePoolParameters` enforces no minimum value on `priceProviderTimelock`. A pool deployed with `priceProviderTimelock = 0` lets the pool admin atomically propose and execute a price-provider replacement in a single block, completely nullifying the M01 fix. LPs have zero blocks to react before a manipulated oracle is live and draining their funds via swaps.

## Finding Description
`_validatePoolParameters` (L548–563) validates fee caps and token config but has no lower-bound check on `params.priceProviderTimelock`. The only sentinel is `type(uint256).max` (immutable mode, L164); any value below — including `0` — is stored verbatim as the mutable-mode delay at L213.

In `proposePoolPriceProvider` (L487):
```solidity
uint256 executeAfter = block.timestamp + timelock; // timelock=0 → executeAfter == block.timestamp
```
In `executePoolPriceProviderUpdate` (L499):
```solidity
if (block.timestamp < execAfter) revert ...; // block.timestamp < block.timestamp → false → no revert
```
Both calls succeed in the same block. `_validatePriceProvider` (L541–546) only checks that the new provider's `token0()`/`token1()` match — a malicious oracle returning an extreme bid/ask passes this check. Once installed via `setPriceProvider`, every subsequent `swap` reads the attacker-controlled price, settling at a manipulated rate and draining LP balances.

## Impact Explanation
Direct loss of LP principal. A malicious oracle returning a near-zero ask price causes the pool to pay out tokens at an extreme rate on every swap. LPs have no reaction window — the oracle is live in the same block as the proposal. This is a broken admin-boundary invariant: the M01 fix (timelock) is entirely negated, and pool insolvency follows immediately after the first swap against the manipulated oracle.

## Likelihood Explanation
`createPool` is permissionless once `poolDeployer` is set. Any external caller can deploy a pool with `priceProviderTimelock = 0`, act as pool admin, attract LP deposits (the pool appears normal), then atomically rotate to a malicious oracle and drain funds in a single transaction via a wrapper contract. The parameter is immutable post-deployment, so all LPs who deposit inherit the risk permanently. No special privilege beyond being the pool creator is required.

## Recommendation
Add a minimum timelock guard in `_validatePoolParameters`, mirroring the existing fee-cap pattern:

```solidity
uint256 constant MIN_PRICE_PROVIDER_TIMELOCK = 1 days;

// inside _validatePoolParameters:
if (params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
    revert PriceProviderTimelockTooShort();
}
```
