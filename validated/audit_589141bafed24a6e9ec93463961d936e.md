Audit Report

## Title
ETH Deposit Limit Bypass via Missing `amount` in Limit Check — (`contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison for ETH versus ERC20 assets: the ETH branch checks only `totalAssetDeposits > limit` while the ERC20 branch correctly checks `totalAssetDeposits + amount > limit`. Any unprivileged caller can invoke `depositETH` with an arbitrarily large `msg.value` while the pre-call total is at or below the limit, silently violating the protocol's configured deposit ceiling and minting rsETH for the full over-limit amount.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, `_checkIfDepositAmountExceedesCurrentLimit` reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

The `depositETH` function (lines 76–93) is `external payable`, callable by any EOA, and passes `msg.value` as `amount` to `_beforeDeposit`, which calls this guard. [2](#0-1) 

Because `totalAssetDeposits` for ETH is computed from `address(this).balance`, NDC balances, EigenLayer pod shares, unstaking amounts, and `LRTConverter.ethValueInWithdrawal()` — none of which include the incoming `msg.value` before the check — the guard will return `false` (not exceeded) for any `msg.value` as long as the pre-call total is ≤ limit. [3](#0-2) 

## Impact Explanation
The deposit limit is a governance-approved risk-management cap. Because the incoming deposit amount is never factored into the ETH branch comparison, the limit is entirely unenforceable for ETH deposits. The protocol fails to deliver its promised risk ceiling — a concrete instance of **Low — Contract fails to deliver promised returns**. The deposited ETH does back the minted rsETH (no direct theft in isolation), but the protocol operates in a state that governance explicitly prohibited.

## Likelihood Explanation
No special role, oracle access, or front-running is required. Any EOA can trigger this by calling `depositETH` with any `msg.value` while `totalAssetDeposits ≤ depositLimit`. The condition is trivially reachable at any point before the limit is naturally reached, and the bypass is repeatable across multiple transactions.

## Recommendation
Mirror the ERC20 branch in the ETH branch by adding `amount` to `totalAssetDeposits` before comparing:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
```solidity
function testETHDepositLimitBypass() public {
    uint256 limit = lrtConfig.depositLimitByAsset(LRTConstants.ETH_TOKEN);

    // Bring total to exactly limit - 1 wei
    vm.deal(alice, limit - 1);
    vm.prank(alice);
    depositPool.depositETH{value: limit - 1}(0, "");

    // Attacker deposits 50_000 ETH — guard returns false because:
    // totalAssetDeposits (limit - 1) > limit → false → deposit proceeds
    vm.deal(attacker, 50_000 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 50_000 ether}(0, "");

    // Total now ≈ limit + 50_000 ether; limit was never enforced
    assertGt(depositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), limit);
}
```

The guard at line 679 evaluates `(limit - 1 wei) > limit` → `false`, so `_beforeDeposit` does not revert, rsETH is minted for the full 50,000 ETH, and `totalAssetDeposits` ends up 1.5× the configured limit. [5](#0-4)
