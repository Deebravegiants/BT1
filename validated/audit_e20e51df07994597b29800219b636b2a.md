### Title
Detached NEAR Transfer Promises Leave Excess Deposits Permanently Stuck in Contract - (File: crates/contract/src/lib.rs)

### Summary
The MPC contract uses `Promise::transfer(...).detach()` in three refund paths. Because the promise is detached, its failure is never observed. If a refund transfer fails, the excess NEAR deposit is permanently locked in the contract with no recovery path — a direct analog to the unchecked ERC20 `transfer()` return value pattern in the original report.

### Finding Description

Three locations in `crates/contract/src/lib.rs` issue a NEAR token transfer via a detached promise:

**Location 1 — `require_deposit` (called from every user-facing request method):** [1](#0-0) 

**Location 2 — `submit_participant_info` (node attestation submission):** [2](#0-1) 

**Location 3 — `propose_update` (governance update proposal):** [3](#0-2) 

In NEAR's execution model, `.detach()` schedules the transfer as an independent receipt in the next block and discards the result. If that receipt fails — for example because the recipient account was deleted in the one-block window between the original call and the transfer receipt, or because the contract's own balance fell below its storage-staking floor — the excess deposit silently remains in the contract. The original call's state changes are already committed (NEAR atomicity is per-receipt), so there is no rollback and no error surfaced to the caller.

The affected entry points are `sign`, `request_app_private_key`, `verify_foreign_transaction` (all via `check_request_preconditions` → `require_deposit`), `submit_participant_info`, and `propose_update`. [4](#0-3) 

### Impact Explanation

Excess NEAR deposits attached by users are permanently frozen in the contract. There is no administrative sweep, no retry mechanism, and no way for the user to reclaim the funds once the transfer receipt has failed silently. This breaks the accounting invariant that every deposit in excess of the minimum is returned to the sender. Matches: **Medium — balance/accounting invariant broken without network-level DoS or operator misconfiguration.**

### Likelihood Explanation

A NEAR account can be deleted by its owner at any time. An attacker (or a user who self-inflicts the condition) can:
1. Call `sign` (or any other payable entry point) with a large excess deposit.
2. In the same block or the immediately following block, submit a `DeleteAccount` transaction for the predecessor account.
3. The transfer receipt executes in the next block, finds the account absent, and fails silently.

The window is narrow (one block, ~1 second on NEAR mainnet), but the operation is cheap and requires no privileged access — any unprivileged caller can trigger it against their own account. The minimum required deposit is only 1 yoctoNEAR, so a user who accidentally attaches 1 NEAR loses ~1 NEAR permanently. [5](#0-4) 

### Recommendation

Replace every `Promise::transfer(...).detach()` refund with a chained callback that verifies the transfer succeeded, or — simpler and idiomatic in NEAR — use `#[callback_result]` on a dedicated `on_refund` method. At minimum, remove `.detach()` and return the `Promise` so the caller's receipt fails atomically if the refund fails, rather than silently swallowing the error.

Alternatively, adopt a pull-payment pattern: record the owed refund amount in contract storage and expose a `claim_refund` method the user calls explicitly, eliminating the push-transfer failure mode entirely.

### Proof of Concept

```
1. Alice calls `sign(request)` attaching 2 NEAR (minimum is 1 yoctoNEAR).
2. `require_deposit` computes diff ≈ 2 NEAR and fires:
       Promise::new(alice.near).transfer(2_NEAR).detach();
3. Alice immediately submits DeleteAccount { beneficiary: bob.near } in the same block.
4. NEAR runtime processes DeleteAccount before the transfer receipt (or in the same block).
5. Transfer receipt executes: alice.near does not exist → receipt fails.
6. Contract state: request enqueued correctly, but 2 NEAR is permanently locked in the MPC contract.
7. No error is returned to Alice; no log is emitted; no recovery path exists.
``` [6](#0-5) [7](#0-6)

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}

        // 1. Look up the domain and check its purpose.

            Err(err) => env::panic_str(&err.to_string()),

        let Some(domain_config) = domains.get_domain_by_domain_id(domain_id) else {
            env::panic_str(
                &InvalidParameters::DomainNotFound {
                    provided: domain_id,
                }
                .to_string(),
            );
        };
        if domain_config.purpose != expected_purpose {
            env::panic_str(
                &InvalidParameters::WrongDomainPurpose {
                    domain_id: domain_config.id,
                    expected: expected_purpose,
                    actual: domain_config.purpose,
                }
                .to_string(),
            );
        }
        let domain_config = domain_config.clone();

        // 2. Make sure the call will not run out of gas doing yield/resume logic.
        let prepaid_gas = env::prepaid_gas();
        if prepaid_gas < minimum_gas {
            env::panic_str(
                &InvalidParameters::InsufficientGas {
                    provided: prepaid_gas.as_gas(),
                    required: minimum_gas.as_gas(),
                }
                .to_string(),
            );
        }

        // 3. Require the minimum deposit and refund any excess.
        let predecessor = env::predecessor_account_id();
        require_deposit(minimum_deposit, &predecessor);

        // 4. Refuse the request if the contract is not currently accepting requests
        //    (e.g. because TEE validation has failed).
        if !self.accept_requests {
            env::panic_str(&TeeError::TeeValidationFailed.to_string())
        }

        (domain_config, predecessor)
    }
