### Title
Missing Payload-to-Request Binding Verification in `respond_verify_foreign_tx()` Enables Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx()` function in the MPC contract verifies that the submitted `response.signature` is cryptographically valid over `response.payload_hash`, but **never verifies that `response.payload_hash` actually commits to the content of the pending `request`**. This allows any single attested participant to replay a previously-issued, legitimately-produced `(payload_hash, signature)` pair from one foreign-chain verification request to satisfy a completely different pending request, delivering a forged attestation to the waiting caller.

---

### Finding Description

The `respond_verify_foreign_tx()` function performs two operations:

1. It verifies the ECDSA signature in `response.signature` over `response.payload_hash` using the domain's root public key.
2. It resolves all pending yields queued under the `request` key. [1](#0-0) 

The critical gap is that `response.payload_hash` is a caller-supplied value that is **never checked against the pending `request`**. The `payload_hash` is supposed to be `SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))` — a commitment to both the original chain-specific request and the extracted values observed on the foreign chain. The contract never recomputes or validates this binding. [2](#0-1) 

The `ForeignTxSignPayloadV1` struct binds the `ForeignChainRpcRequest` (chain type, tx_id, extractors) to the `values` extracted from the foreign chain. The hash of this struct is what the MPC network signs. But the contract's `respond_verify_foreign_tx()` accepts any `payload_hash` that has a valid signature — it does not verify that the hash encodes the specific `request` that is pending. [3](#0-2) 

The SDK-side `ForeignChainSignatureVerifier::verify_signature()` does perform this binding check on the caller's side, but this is an off-chain, optional client library — it is not enforced by the contract itself. [4](#0-3) 

---

### Impact Explanation

**Impact: High — Forged foreign-chain verification enabling invalid bridge execution.**

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow: a NEAR contract releases funds only after the MPC network attests that a specific foreign-chain transaction finalized. If a Byzantine node replays a valid `(payload_hash, signature)` from a previous request (e.g., for Bitcoin tx_id A) to satisfy a new pending request (for Bitcoin tx_id B), the caller's NEAR contract receives a signed attestation that does not correspond to their transaction. If the caller's contract does not independently recompute the expected `payload_hash` from known extracted values (which it often cannot, since the extracted values are determined by the MPC nodes), it will accept the forged attestation and release funds for a transaction that was never actually verified.

---

### Likelihood Explanation

**Likelihood: Medium.**

The attack requires a single Byzantine attested MPC participant. All `respond_verify_foreign_tx` calls are submitted on-chain and are publicly visible, so any attested participant can observe and record valid `(payload_hash, signature)` pairs from past responses. The attacker only needs to wait for a new pending request and replay a previously-seen valid response against it. No threshold collusion is required — a single node below the signing threshold can execute this attack. [5](#0-4) 

---

### Recommendation

In `respond_verify_foreign_tx()`, after verifying the signature, recompute the minimum expected `payload_hash` from the `request` and verify that `response.payload_hash` is consistent with it. Concretely, the contract should verify that `response.payload_hash` is a valid hash of a `ForeignTxSignPayloadV1` whose `request` field matches the pending `request`. At minimum, the contract should verify that the `ForeignChainRpcRequest` embedded in the signed payload matches the pending request by deserializing or re-hashing with the known request fields.

Alternatively, store the expected `payload_hash` (computed by the leader node and committed on-chain before signing) alongside the pending request, and verify the response's `payload_hash` against it.

---

### Proof of Concept

**Setup**: Two pending `verify_foreign_transaction` requests exist:
- **R1**: Bitcoin tx_id `[0xAA; 32]`, already responded to with valid `(payload_hash_A, sig_A)` (visible on-chain)
- **R2**: Bitcoin tx_id `[0xBB; 32]`, currently pending in `pending_verify_foreign_tx_requests`

**Attack**:

```
// Attacker (any single attested participant) calls:
respond_verify_foreign_tx(
    request = R2,   // correct pending request key — found in the map
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,  // replayed from R1's response
        signature: sig_A,              // replayed from R1's response
    }
)
```

**Contract execution** (`crates/contract/src/lib.rs:718-753`):
1. `verify_ecdsa_signature(sig_A, payload_hash_A, root_pk)` → **valid** (sig_A was legitimately produced)
2. `resolve_yields_for(&pending_verify_foreign_tx_requests, &R2, response)` → **resolves R2's yield** with the forged response

**Result**: The caller of R2 receives `{payload_hash: payload_hash_A, signature: sig_A}` — an attestation that commits to Bitcoin tx_id A's data, not tx_id B's. If the caller's NEAR bridge contract uses this to release funds for tx_id B, a double-spend or invalid bridge execution occurs. [6](#0-5) [7](#0-6) [8](#0-7)
