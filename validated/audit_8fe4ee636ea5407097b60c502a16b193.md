Audit Report

## Title
`LRTOracle._getTotalEthInProtocol()` Excludes FeeReceiver/RewardReceiver ETH from TVL, Understating rsETH Price and Enabling Theft of Unclaimed Yield - (`contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary

`getETHDistributionData()` in `LRTDepositPool` explicitly omits ETH rewards sitting in `FeeReceiver`/reward receiver contracts from the TVL calculation, as documented in its own NatSpec. Because `_getTotalEthInProtocol()` in `LRTOracle` sources all ETH accounting from this function, the rsETH price is persistently understated between reward-sweep cycles. Any depositor who deposits during this window receives excess rsETH, permanently diluting the unclaimed yield owed to existing holders.

## Finding Description

**Code path:**

1. `LRTOracle._updateRsETHPrice()` calls `_getTotalEthInProtocol()` [1](#0-0) 

2. `_getTotalEthInProtocol()` iterates over supported assets and calls `ILRTDepositPool.getTotalAssetDeposits(asset)` for each. [2](#0-1) 

3. `getTotalAssetDeposits()` calls `getAssetDistributionData(asset)`, which for `ETH_TOKEN` delegates entirely to `getETHDistributionData()`. [3](#0-2) 

4. `getETHDistributionData()` explicitly documents the omission in its NatSpec and counts only: deposit pool balance, NDC balances, EigenLayer pod shares, queued withdrawals, unstaking vault balance, and converter ETH. It does **not** include the balance of `FeeReceiver` or any reward receiver contract. [4](#0-3) 

5. The rsETH price is then computed as: [5](#0-4) 

6. New deposits use this price to determine rsETH minted: [6](#0-5) 

**Root cause:** The TVL used for pricing excludes accrued-but-unswept ETH rewards. When rewards accumulate in `FeeReceiver`, the denominator (`rsETHPrice`) in the minting formula is lower than the true per-share value, so `rsethAmountToMint` is inflated. Once rewards are swept and `updateRSETHPrice()` is called, the price jumps — but the excess rsETH already minted to the new depositor is permanent, diluting all prior holders' share of those rewards.

**Existing guards are insufficient:** The `pricePercentageLimit` check only fires when the price *increases* beyond a threshold after a sweep; it does not prevent the dilution that already occurred during the underpriced deposit window. The `minRSETHAmountExpected` slippage parameter protects the depositor, not existing holders.

## Impact Explanation

This is a **High — Theft of unclaimed yield**. Existing rsETH holders have an accrued but not-yet-distributed claim on ETH sitting in `FeeReceiver`. When a new depositor enters at the understated price, they capture a portion of that yield. The loss to existing holders is proportional to the size of the accumulated rewards and the size of the deposit. The attacker retains the excess rsETH permanently; the prior holders' rsETH is worth less than it should be after the sweep.

## Likelihood Explanation

Validator tips, MEV, and EigenLayer AVS rewards accrue continuously. The protocol relies on an off-chain keeper to periodically sweep rewards from `FeeReceiver` to the deposit pool. Between sweeps, the gap between true TVL and reported TVL grows monotonically. Any unprivileged user can exploit this by calling `depositETH()` or `depositAsset()` — no special role, no governance action, and no victim mistake is required. A sophisticated attacker can monitor the `FeeReceiver` balance on-chain and time deposits to maximize the captured yield, but even passive depositors benefit at existing holders' expense. Likelihood: **Medium** (occurs naturally every reward cycle).

## Recommendation

Include the ETH balance of `FeeReceiver` (and any other reward receiver contracts registered in `LRTConfig`) in `getETHDistributionData()` so that accrued-but-unswept rewards are always reflected in the rsETH price. Concretely, add a loop over all registered reward receivers and sum their `.balance` into the returned TVL. Alternatively, enforce that `updateRSETHPrice()` atomically sweeps rewards before computing the price, eliminating the gap entirely.

## Proof of Concept

**Minimal call sequence (no fork required):**

1. Deploy protocol with 1000 ETH TVL, 1000 rsETH supply → `rsETHPrice = 1.0e18`.
2. 10 ETH of validator rewards accumulate in `FeeReceiver` (not swept).
3. Call `updateRSETHPrice()` → `_getTotalEthInProtocol()` returns 1000 ETH (FeeReceiver excluded) → `rsETHPrice` remains `1.0e18`.
4. Attacker calls `depositETH{value: 100 ether}(0, "")` → `getRsETHAmountToMint` computes `(100e18 * 1e18) / 1e18 = 100e18` rsETH minted. Correct amount at true price `(1010/1000)` would be `≈99.01e18`.
5. Keeper sweeps 10 ETH from `FeeReceiver` to deposit pool. Call `updateRSETHPrice()` → TVL = 1110 ETH, supply = 1100 rsETH → `rsETHPrice ≈ 1.009e18`.
6. Attacker's 100 rsETH is worth ≈100.9 ETH. Original 1000 rsETH holders receive ≈1009 ETH instead of the ≈1010 ETH they were entitled to. The attacker captured ≈0.9 ETH of yield belonging to prior holders.

**Foundry test plan:** Deploy mock `LRTConfig`, `LRTDepositPool`, `LRTOracle`, and a `FeeReceiver` stub. Fund `FeeReceiver` with 10 ETH without calling `receiveFromRewardReceiver`. Assert `_getTotalEthInProtocol()` returns 1000 ETH (not 1010). Deposit 100 ETH as attacker, assert minted amount > 99.01e18. Sweep rewards, call `updateRSETHPrice()`, assert attacker's rsETH value exceeds 100 ETH while original holders' aggregate value is less than 1010 ETH.

    /// @dev rewards are not accounted here
    /// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
    function getETHDistributionData()
        public
        view
        override
        returns (
            uint256 ethLyingInDepositPool,
            uint256 ethLyingInNDCs,
            uint256 ethStakedInEigenLayer,
            uint256 ethUnstakingFromEigenLayer,
            uint256 ethLyingInConverter,
            uint256 ethLyingInUnstakingVault
        )
    {
        ethLyingInDepositPool = address(this).balance;

        uint256 ndcsCount = nodeDelegatorQueue.length;

        for (uint256 i; i < ndcsCount;) {
            ethLyingInNDCs += nodeDelegatorQueue[i].balance;

            ethStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getEffectivePodShares();
            ethUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i])
                .getAssetUnstaking(LRTConstants.ETH_TOKEN);

                ++i;
            }
        }

        address lrtUnstakingVault = lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT);

        address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
        ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
    }
