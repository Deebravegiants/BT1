### Title
`DEFAULT_KEY_EVENT_TIMEOUT_BLOCKS = 30` Too Tight for Resharing Round-Trip, Causing Persistent Resharing State Lock — (File: `crates/contract/src/config.rs`)

---

### Summary

The default `key_event_timeout_blocks` of **30 blocks (~18 seconds on mainnet)** is too tight for the full resharing round-trip. Under normal mainnet conditions — especially with multiple domains and any RPC or indexer latency — every resharing attempt expires before threshold votes can be collected. The contract becomes permanently stuck in the `Resharing` state, breaking the participant-state and contract execution-flow invariants. The codebase itself acknowledges this in a comment that explicitly overrides the default to 240 blocks in e2e tests.

---

### Finding Description

`DEFAULT_KEY_EVENT_TIMEOUT_BLOCKS` is set to `30` and is used as the expiry window for both initial keygen and resharing attempts:

```rust
// crates/contract/src/config.rs
const DEFAULT_KEY_EVENT_TIMEOUT_BLOCKS: u64 = 30;
```

This value is passed directly into `start_keygen_instance` and `start_reshare_instance`:

```rust
// crates/contract/src/lib.rs
self.protocol_state
    .start_reshare_instance(key_event_id, self.config.key_event_timeout_blocks)
```

Inside `KeyEventInstance::new`, the expiry is set as:

```rust
expires_on: env::block_height() + 1 + timeout_blocks,
```

And `active()` returns `false` once `env::block_height() >= expires_on`. When `cleanup_if_timed_out()` fires on the next call, the instance is silently cleared and the leader must restart from scratch.

The resharing round-trip requires:
1. Leader submits `start_reshare_instance` on-chain
2. All N nodes index the block containing that transaction
3. All N nodes run the MPC resharing computation (cryptographically intensive)
4. Threshold nodes submit `vote_reshared` transactions
5. The contract collects threshold votes and transitions to `Running`

On mainnet with ~600 ms blocks, 30 blocks = **~18 seconds**. This window must cover steps 1–5 end-to-end. With multiple domains (each requiring its own `KeyEvent` round-trip sequentially), the total time multiplies. The e2e cluster code explicitly documents this:

```rust
// crates/e2e-tests/src/cluster.rs
// The contract's default `key_event_timeout_blocks = 30` is ~18 s on
// mainnet (~600 ms blocks). The e2e sandbox runs ~8 blocks/s, so the
// same 30 collapses to ~3.7 s — too tight for the resharing
// round-trip (start_reshare → indexer × N → MPC compute →
// vote_reshared) once concurrent e2e tests put the runner under
// load. Override to 240 blocks (~30 s in sandbox) as a comfortable
// budget over mainnet's effective headroom.
const KEY_EVENT_TIMEOUT_BLOCKS: u64 = 240;
```

The e2e resharing test further confirms this by explicitly guarding against repeated timeout failures:

```rust
// crates/e2e-tests/tests/key_resharing.rs
// Verify resharing didn't repeatedly fail. Allow a small number of
// retries for transient sandbox contention (key_event_timeout_blocks
// expiring under RPC load).
const MAX_TOLERATED_ATTEMPTS: u64 = 5;
```

When every attempt times out, the contract remains in `Resharing` state indefinitely. There is no automatic recovery path: the leader must keep restarting attempts, each of which expires before votes arrive.

---

### Impact Explanation

**Medium.** The contract execution-flow and participant-state invariants are broken:

- The contract cannot transition from `Resharing` → `Running`, so key rotation never completes.
- The participant set cannot be updated (no new participants can be added, no compromised participants removed).
- Signing continues to work using the *old* keyset from `previous_running_state`, but only as long as the old participants remain online. If any old participant goes offline, the signing threshold may no longer be reachable.
- The MPC network is effectively frozen in its pre-resharing configuration indefinitely.

This does not require network-level DoS or operator misconfiguration — it is triggered by the default configuration under normal mainnet load.

---

### Likelihood Explanation

**High.** The codebase itself treats the default as insufficient: the e2e test harness overrides it to 240 blocks (8× the default) and the resharing test explicitly tolerates repeated timeout-driven retries. On mainnet with multiple domains and any indexer or RPC latency, the 18-second window is routinely insufficient for the full round-trip. The issue is not theoretical — it is acknowledged in production test infrastructure.

---

### Recommendation

Increase `DEFAULT_KEY_EVENT_TIMEOUT_BLOCKS` to a value that provides realistic headroom for the full resharing round-trip on mainnet. The e2e tests use 240 blocks as a "comfortable budget." A value in the range of 200–300 blocks (2–3 minutes on mainnet) is appropriate given the multi-step, multi-domain nature of the resharing protocol. The constant should also be documented with an explicit derivation (e.g., `N_domains × round_trip_latency_blocks`).

---

### Proof of Concept

1. Deploy the contract with the default config (`key_event_timeout_blocks = 30`).
2. Initiate a resharing operation with multiple domains and ≥3 participants.
3. Observe that `start_reshare_instance` succeeds, but `vote_reshared` transactions arrive after block `started_in + 31`, causing `active()` to return `false`.
4. `cleanup_if_timed_out()` clears the instance on the next call; the leader restarts.
5. Every subsequent attempt also expires. The contract remains in `Resharing` state indefinitely.
