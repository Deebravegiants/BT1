### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Reward via Unenforceable `farmer_reward_address_override` - (`chia/farmer/farmer_api.py`)

### Summary

Under the CHIP-22 third-party harvester protocol, a harvester that finds a valid proof of space can set `farmer_reward_address_override` in its `RespondSignatures` message to redirect the farmer's block reward (1/8 of block reward, ~0.25 XCH) to any address it controls. The farmer accepts this override unconditionally: the only guard is an advisory log warning that the harvester itself can trivially bypass by self-reporting a permissive fee threshold. No cryptographic enforcement prevents a malicious harvester from always diverting the farmer reward.

### Finding Description

The CHIP-22 fee convention allows a third-party harvester to redirect the farmer's block reward as a service fee, gated by a "fee quality" check. The harvester self-reports an `applied_fee_threshold` in `ProofOfSpaceFeeInfo`. The farmer computes `fee_quality = calculate_harvester_fee_quality(proof, challenge_hash)` and checks `fee_quality <= fee_threshold`. If the check fails, the farmer only emits a log warning — it does **not** reject the override.

**Attack path:**

1. A malicious harvester connects to a farmer (standard harvester protocol, no special privilege required).
2. When it finds a valid proof of space, it sends `NewProofOfSpace` with `farmer_reward_address_override` set to the attacker's puzzle hash and `fee_info.applied_fee_threshold = 0xFFFFFFFF` (the maximum `uint32` value).
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`. Since `fee_quality` (a `uint32`) is always `<= 0xFFFFFFFF`, the check `fee_quality <= fee_threshold` always passes and the farmer logs "Fee threshold passed" — no rejection occurs.
4. The farmer requests signatures from the harvester via `RequestSignatures`.
5. The harvester responds with `RespondSignatures` containing `farmer_reward_address_override` set to the attacker's address.
6. In `_process_respond_signatures()`, the farmer unconditionally accepts the override with no re-validation of the fee convention:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override  # no check
    include_source_signature_data = True
```

7. The farmer sends `DeclareProofOfSpace` to the full node with the attacker's address as `farmer_puzzle_hash`.
8. The full node calls `create_unfinished_block()` → `create_foliage()` using the attacker-supplied `farmer_reward_puzzlehash`, embedding it in `FoliageBlockData`.
9. The resulting block pays the farmer reward coin to the attacker's address.

The fee threshold is entirely self-reported by the harvester. Even without the `0xFFFFFFFF` trick, the farmer still accepts the override when `fee_info` is absent — it only logs a different warning message and proceeds.

### Impact Explanation

**Critical — Reward Diversion of XCH.** A malicious harvester can redirect the farmer's block reward (currently 0.25 XCH per block) to any address it controls on every block the harvester wins. The farmer receives no reward for blocks found by the malicious harvester. This is a direct, on-chain, irreversible theft of XCH from the farmer.

### Likelihood Explanation

Any entity that can convince a farmer to connect to their harvester (e.g., by advertising a third-party harvesting service) can execute this attack silently. The farmer's logs will show "Fee threshold passed" — indistinguishable from a legitimate fee. No leaked keys, admin access, or cryptographic break is required.

### Recommendation

The farmer must enforce the fee convention, not merely log it. Specifically:

1. In `_process_respond_signatures()`, before accepting `response.farmer_reward_address_override`, re-derive `fee_quality` from the proof and challenge hash and verify that `fee_quality <= applied_fee_threshold` where `applied_fee_threshold` was advertised in the original `NewProofOfSpace.fee_info`. If the check fails, reject the override and use `self.farmer.farmer_target`.
2. Reject `farmer_reward_address_override` entirely if `fee_info` is absent in the original `NewProofOfSpace`.
3. Cap the maximum acceptable `applied_fee_threshold` to a protocol-defined value to prevent the `0xFFFFFFFF` bypass.

def calculate_harvester_fee_quality(proof: bytes, challenge: bytes32) -> uint32:
    """
    This calculates the 'fee quality' given a convention between farmers and third party harvesters.
    See CHIP-22: https://github.com/Chia-Network/chips/pull/88
    """
    return uint32(int.from_bytes(std_hash(proof + challenge)[32 - 4 :], byteorder="big", signed=False))
