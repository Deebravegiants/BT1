### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Forged Foreign-Chain Verification — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted `response.payload_hash` has a valid threshold ECDSA signature, but never checks that `payload_hash` is the canonical hash of the original pending request. A single Byzantine attested participant (the signing leader) who legitimately obtains a threshold signature for request B can submit it as the response for a different pending request A. The contract accepts it, and the caller of request A receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes a completely different foreign-chain transaction.

---

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs the following validation:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
``` [1](#0-0) 

It checks only that `signature` is a valid ECDSA signature over the caller-supplied `payload_hash`. It does **not** recompute the expected hash from the stored `VerifyForeignTransactionRequest` (which contains `tx_id`, `extractors`, `domain_id`, `payload_version`) and verify that `payload_hash` matches.

The canonical hash is defined as `SHA-256(borsh(ForeignTxSignPayload{request, values}))`: [2](#0-1) 

The contract stores the original request in `pending_verify_foreign_tx_requests` and uses it only as a lookup key to resolve the yield — it never uses the stored request to recompute and cross-check `payload_hash`: [3](#0-2) 

The SDK's `ForeignChainSignatureVerifier` does perform this cross-check (line 57: `expected_payload_hash == response.payload_hash`), but it is a client-side library, not enforced by the contract: [4](#0-3) 

---

### Impact Explanation

A Byzantine leader node that has legitimately received a threshold signature `sig_B` over `payload_hash_B` (for request B, e.g., Bitcoin tx `0xBB`) can call `respond_verify_foreign_tx(request_A, {payload_hash: payload_hash_B, sig: sig_B})` for a different pending request A (e.g., Bitcoin tx `0xAA`). The contract accepts this because:

1. The caller is an attested participant ✓
2. `verify_ecdsa_signature(sig_B, payload_hash_B, root_pk)` passes ✓
3. `request_A` is found in `pending_verify_foreign_tx_requests` ✓

The yield for request A is resolved with a response whose `payload_hash` encodes a completely different transaction. Any NEAR bridge contract that does not independently recompute and verify `payload_hash` (i.e., does not use `ForeignChainSignatureVerifier`) will accept this as proof that tx `0xAA` was verified, when in fact it was not. This enables forged foreign-chain verification and invalid bridge execution (e.g., releasing bridged funds for a deposit that never occurred).

---

### Likelihood Explanation

The attacker must be a single attested MPC participant who acts as the signing leader for some request B. This is strictly below the signing threshold. The leader role is assigned deterministically per request, so any single Byzantine participant who is scheduled as leader for any request can obtain a valid threshold signature and replay it against any other pending request in the same domain. No threshold collusion is required — the honest threshold nodes cooperate in good faith for request B, and the Byzantine leader misuses the resulting signature.

---

### Recommendation

The contract should recompute the expected `payload_hash` from the stored request and the submitted extracted values, or alternatively require the responding node to include the full `ForeignTxSignPayload` (not just the hash) so the contract can independently verify the hash. At minimum, the contract should enforce that `payload_hash` is structurally bound to the original `VerifyForeignTransactionRequest` stored in `pending_verify_foreign_tx_requests`, analogous to how `respond` for regular signing derives and checks the expected public key from the stored request before accepting the signature.

---

### Proof of Concept

1. User A calls `verify_foreign_transaction({tx_id: 0xAA, extractors: [BlockHash], domain_id: D})` — request A is queued.
2. User B calls `verify_foreign_transaction({tx_id: 0xBB, extractors: [BlockHash], domain_id: D})` — request B is queued.
3. The MPC network processes request B. The Byzantine leader node receives the final threshold signature `sig_B` over `payload_hash_B = SHA-256(borsh(ForeignTxSignPayload{request_B, values_B}))`.
4. Instead of submitting `respond_verify_foreign_tx(request_B, {payload_hash_B, sig_B})`, the Byzantine leader submits `respond_verify_foreign_tx(request_A, {payload_hash_B, sig_B})`.
5. The contract at lines 718–734 verifies `verify_ecdsa_signature(sig_B, payload_hash_B, root_pk)` — this passes because `sig_B` is a valid threshold signature over `payload_hash_B`.
6. The contract at lines 749–753 resolves the yield for request A with `{payload_hash: payload_hash_B, sig: sig_B}`.
7. User A's NEAR contract receives a `VerifyForeignTransactionResponse` claiming tx `0xAA` was verified, but `payload_hash` encodes tx `0xBB`'s data. Any bridge logic that does not call `ForeignChainSignatureVerifier::verify_signature` will accept this as valid proof of tx `0xAA`. [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {

            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });

            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),

                &expected_payload_hash,

        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
