### Title
Byzantine Participant Can Deliver Forged CKD Output via Missing Output Verification for `AppPublicKey` Variant - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_ckd` in the MPC contract performs a cryptographic pairing check (`ckd_output_check`) on the CKD response **only** when the request uses the `AppPublicKeyPV` variant. When the request uses the `AppPublicKey` variant (the standard, non-publicly-verifiable path), no verification of the response is performed. A single Byzantine participant below the signing threshold can call `respond_ckd` with an arbitrary forged `CKDResponse`, bypassing the threshold requirement and delivering a corrupted or attacker-controlled key to the user.

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 675–682), the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO output check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `ckd_output_check` function verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically proves the response is the correct threshold-computed output: [2](#0-1) 

For `AppPublicKey`, this check is entirely absent. The contract accepts any `CKDResponse` (`big_c`, `big_y`) without verification and resolves the pending yield with it: [3](#0-2) 

**Exploit path:**

1. A legitimate user calls `request_app_private_key` with `AppPublicKey(A)` where `A = a·G1`.
2. A Byzantine participant (attested, below signing threshold) races to call `respond_ckd` before the honest leader, supplying:
   - `big_y = G1_identity` (the identity/zero point)
   - `big_c = X` (any attacker-chosen G1 point)
3. The contract performs no output check and resolves the yield with the forged response.
4. The user's client computes `secret = big_c − a·big_y = X − a·0 = X`.
5. The attacker knows `X`, so they now know the user's derived confidential key.

The identity point is a valid G1 encoding accepted by the NEAR host (confirmed by the existing test `app_public_key_check__should_accept_identity_key_pair`): [4](#0-3) 

The `assert_caller_is_signer` and `assert_caller_is_attested_participant_and_protocol_active` checks only require the caller to be any attested participant — not specifically the elected leader: [5](#0-4) 

This is structurally identical to the reference vulnerability: the second-leg check (`ckd_output_check`) is the only guard, but it is conditionally skipped when `AppPublicKey` is used, leaving the first leg (accepting the response) completely unprotected.

### Impact Explanation

**Critical — Confidential key derivation output delivered without required participant authorization.**

A single Byzantine participant can cause the user to derive a key that the attacker fully controls and knows. The user's application (e.g., a bridge, a wallet, a TEE app) will use this attacker-known key as its private key, enabling the attacker to impersonate the user, forge signatures, or steal funds associated with that derived key. The threshold guarantee of the CKD protocol is entirely bypassed for the `AppPublicKey` variant.

### Likelihood Explanation

Any single attested participant who is Byzantine can exploit this. The attacker only needs to:
- Be an active, attested participant (no threshold collusion required).
- Observe a pending `AppPublicKey` CKD request on-chain (publicly visible).
- Submit `respond_ckd` before the honest leader does.

The `AppPublicKey` variant is the default/basic path used by callers who do not need on-chain verifiability (e.g., the `ckd-example-cli` uses it by default): [6](#0-5) 

### Recommendation

Add a minimum validity check for the `AppPublicKey` branch. Since `app_pk2` (G2) is unavailable, the full pairing check cannot be applied directly, but the protocol should at minimum:

1. **Require `AppPublicKeyPV` for all CKD requests** so the on-chain pairing check is always enforced, or
2. **Add a partial check** for `AppPublicKey`: verify that `big_y` is not the identity point and that both `big_c` and `big_y` are valid, non-identity G1 points, preventing the trivial `big_y = 0` key-disclosure attack.

```diff
match &request.app_public_key {
-   dtos::CKDAppPublicKey::AppPublicKey(_) => {}
+   dtos::CKDAppPublicKey::AppPublicKey(_) => {
+       // Reject identity points to prevent key-disclosure via big_y=0 trick
+       let big_c = env::bls12381_p1_decompress(&response.big_c);
+       let big_y = env::bls12381_p1_decompress(&response.big_y);
+       require!(!is_identity_g1(&big_y), "big_y must not be the identity point");
+   }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

The stronger fix is to deprecate `AppPublicKey` and require `AppPublicKeyPV` for all production CKD requests.

### Proof of Concept

1. Deploy the MPC contract with a BLS12-381 CKD domain.
2. User calls `request_app_private_key` with `AppPublicKey(A)` where `A = a·G1` for some private scalar `a`.
3. Byzantine participant (attested) calls `respond_ckd` with:
   ```json
   {
     "request": { /* matching pending request */ },
     "response": {
       "big_y": "<compressed G1 identity point>",
       "big_c": "<attacker-chosen G1 point X>"
     }
   }
   ```
4. Contract accepts (no `ckd_output_check` for `AppPublicKey`), resolves the yield.
5. User receives `{ big_y: identity, big_c: X }` and computes `secret = X − a·0 = X`.
6. Attacker knows `X` → attacker knows user's derived confidential key. [7](#0-6) [8](#0-7)

/// Point validation is fully delegated to the host, as in

    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

        big_c.as_slice(),

        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
