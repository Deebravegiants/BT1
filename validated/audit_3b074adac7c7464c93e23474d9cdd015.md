### Title
Cross-Request Response Replay in `respond_verify_foreign_tx` Allows Forged Foreign-Chain Verification — (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies only that the submitted ECDSA signature is valid over `response.payload_hash` under the **root** public key. It does not verify that `payload_hash` is cryptographically bound to the specific `VerifyForeignTransactionRequest` being resolved. A single Byzantine attested participant (below threshold) can replay a legitimately-produced response from one pending request to resolve a completely different pending request, delivering a fabricated `payload_hash` to the waiting caller.

### Finding Description

In `respond_verify_foreign_tx` the contract performs the following check: [1](#0-0) 

The verification is:
```
sig_valid = ECDSA_verify(response.signature, response.payload_hash, root_public_key)
```

If `sig_valid == true`, the contract immediately resolves all yields queued under `request` with `response` as the data: [2](#0-1) 

There is no check that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request: <the pending request>, values: <observed values> }))`. The `payload_hash` is accepted as-is from the caller.

Compare this to the regular `respond` function, which correctly derives the expected public key from the request's tweak before verifying: [3](#0-2) 

On the node side, `build_signature_request` uses a zero tweak (root key) and signs `compute_msg_hash()` of the payload, which includes the full `ForeignChainRpcRequest`: [4](#0-3) 

The `ForeignTxSignPayload::V1` binds the hash to `{request, values}`: [5](#0-4) 

Because the contract never checks this binding, a response produced for `request_B` (with `payload_hash_B = SHA-256(borsh({request_B, values_B}))`) is cryptographically valid and will pass the contract's check when submitted against `request_A`.

### Impact Explanation

A Byzantine attested participant (one node, below threshold) can:

1. Observe a legitimately-produced `respond_verify_foreign_tx` call for `request_B` (e.g., a high-value bridge deposit of 100 BTC), capturing `{payload_hash_B, sig_B}`.
2. Call `respond_verify_foreign_tx(request_A, {payload_hash_B, sig_B})` where `request_A` is a different pending request (e.g., a 1 BTC deposit from a different user).
3. The contract accepts the call: `sig_B` is a valid MPC-produced signature over `payload_hash_B` under the root key.
4. The yield for `request_A` is resolved with `payload_hash_B` — the hash of a completely different foreign-chain observation.
5. The caller's bridge contract receives `payload_hash_B` and a valid MPC signature over it. If the bridge contract mints tokens proportional to the value encoded in the payload (without independently re-deriving the expected hash from `request_A`), it will mint 100 BTC worth of tokens for a 1 BTC deposit.

This is a **forged foreign-chain verification** enabling **invalid bridge execution / double-spend conditions**, matching the High impact tier.

### Likelihood Explanation

- Requires one Byzantine attested MPC participant (below threshold). The attacker does not need to compromise the threshold or collude with other nodes.
- The attacker only needs to observe a legitimate `respond_verify_foreign_tx` call on-chain (all NEAR transactions are public) and submit a replay transaction before the target request times out.
- Bridge contracts that rely on `verify_foreign_transaction` for inbound flows (the primary stated use case) are the natural target.
- Likelihood: **Medium** — the attacker must be an active attested participant, but no cryptographic break is required.

### Recommendation

The contract must verify that `response.payload_hash` is bound to the specific `request` being resolved. Since the contract cannot independently compute `observed_values`, the recommended fix is to require the MPC nodes to include the full `ForeignTxSignPayload` (not just its hash) in the response, and have the contract:

1. Verify `payload.request == request.request` (the embedded request matches the pending request).
2. Compute `expected_hash = SHA-256(borsh(payload))` and verify `expected_hash == response.payload_hash`.
3. Verify the signature over `expected_hash`.

This ensures the `payload_hash` delivered to the caller is always the hash of data that includes the exact `ForeignChainRpcRequest` the caller submitted, making cross-request replay impossible.

### Proof of Concept

```
Setup:
  - pending_verify_foreign_tx_requests contains:
      request_A (Bitcoin tx_A, 1 BTC deposit) → [yield_id_alice]
      request_B (Bitcoin tx_B, 100 BTC deposit) → [yield_id_bob]

Step 1: MPC network legitimately processes request_B.
  - Nodes compute payload_hash_B = SHA-256(borsh({request_B, [BlockHash_B]}))
  - Nodes produce sig_B = ECDSA_sign(payload_hash_B, root_secret_key)
  - Honest node calls: respond_verify_foreign_tx(request_B, {payload_hash_B, sig_B})
  - yield_id_bob resolved with {payload_hash_B, sig_B} ✓

Step 2: Byzantine node replays response_B against request_A.
  - Attacker calls: respond_verify_foreign_tx(request_A, {payload_hash_B, sig_B})
  - Contract check: verify_ecdsa(sig_B, payload_hash_B, root_pk) → TRUE ✓
  - Contract resolves yield_id_alice with {payload_hash_B, sig_B}

Step 3: Alice's bridge contract receives {payload_hash_B, sig_B}.
  - verify_ecdsa(sig_B, payload_hash_B, root_pk) → TRUE (valid MPC signature)
  - payload_hash_B encodes a 100 BTC observation
  - Bridge mints 100 BTC worth of tokens for Alice's 1 BTC deposit → double-spend
```

Root cause line: [6](#0-5) 

The `payload_hash` taken directly from `response` is never checked against the `request` key used to look up the pending yield.

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
