### Title
Unguarded `init` Entry Point Allows Any Account to Hijack MPC Network During Deployment Race Window — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `MpcContract::init` function carries no access-control guard (`#[private]` is absent), while every documented deployment procedure deploys the contract binary in one transaction and calls `init` in a separate, later transaction — sometimes hours later. Any account that submits an `init` call during that window becomes the sole authority over the participant set, threshold, and all future signing operations.

---

### Finding Description

`MpcContract::init` is decorated with `#[init]` (prevents re-initialization) but **not** `#[private]` (which would restrict the caller to the contract account itself). [1](#0-0) 

By contrast, the migration-time initializer `init_running` is explicitly marked `#[private]`, proving the developers know the attribute and intentionally apply it where they want caller restriction. [2](#0-1) 

Every documented deployment flow — localnet, testnet, and the automated TEE cluster script — deploys the WASM binary with `without-init-call` and then calls `init` in a completely separate, later transaction: [3](#0-2) 

The automated script `deploy-tee-cluster.sh` makes the gap explicit: the `near_contract` phase (deploy) and the `near_init` phase (initialize) are separated by at least three additional phases — CVM deployment, key collection, and init-args generation — which together can span hours: [4](#0-3) 

During that window, `init` accepts arbitrary `ThresholdParameters` from any caller. The function performs no check on `env::predecessor_account_id()` or `env::signer_account_id()` before writing the participant set and transitioning the contract into `Running` state. [5](#0-4) 

---

### Impact Explanation

An attacker who calls `init` first supplies their own `ThresholdParameters`, installing themselves (or colluding accounts) as the sole participants. From that point:

1. The contract is in `Running` state; the legitimate deployer's subsequent `init` call fails with "already initialized."
2. The attacker's nodes perform DKG, producing key shares only they hold.
3. Every `sign`, `respond`, `vote_pk`, `start_keygen_instance`, and `respond_ckd` call is gated on `assert_caller_is_attested_participant_and_protocol_active`, which now passes only for the attacker's accounts. [6](#0-5) 
4. The attacker can issue threshold signatures for arbitrary payloads on behalf of the MPC network's derived addresses, draining any funds controlled by those addresses.

This matches **Critical — Unauthorized threshold signature issuance without required participant authorization**.

---

### Likelihood Explanation

- The deployment gap is not accidental; it is the documented, required procedure across all environments (localnet, testnet, mainnet TEE cluster).
- The gap is long (CVM boot, key collection, init-args generation) and involves publicly observable on-chain state: once the WASM is deployed but `state()` returns `NotInitialized`, the window is open.
- No special privilege, key leak, or TEE access is required — any NEAR account with enough gas can call `init`.
- NEAR's public mempool means a monitoring attacker can detect the deploy transaction and submit a front-running `init` in the next block.

---

### Recommendation

Add `#[private]` to `init`, matching the pattern already used on `init_running`:

```rust
#[private]
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> { ... }
```

`#[private]` in NEAR SDK restricts the caller to `env::current_account_id()` (the contract account itself). The deployer already controls that account's keys and can call `init` from it — the restriction adds no operational burden while closing the race window entirely.

Alternatively, deploy and initialize atomically using `--initFunction init --initArgs ...` in a single `near deploy` invocation, eliminating the uninitialized window regardless of the access-control annotation.

---

### Proof of Concept

1. Observe the NEAR chain. When a transaction deploys WASM to `v1.signer` (or any MPC contract account) but the contract's `state()` view still returns `NotInitialized`, the window is open.
2. Submit from any funded account:
   ```json
   near contract call-function as-transaction <MPC_CONTRACT_ACCOUNT> init \
     json-args '{
       "parameters": {
         "participants": { "next_id": 1, "participants": [["attacker.near", 0, {"url":"http://attacker.example","tls_public_key":"<attacker_tls_pk>"}]] },
         "threshold": 1
       }
     }' \
     prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
     sign-as attacker.near network-config testnet sign-with-keychain send
   ```
3. The contract transitions to `Running` with `attacker.near` as the sole participant.
4. The legitimate deployer's subsequent `init` call panics: contract is already initialized.
5. The attacker's node calls `vote_add_domains`, performs DKG, and begins issuing signatures — `assert_caller_is_attested_participant_and_protocol_active` passes because the attacker submitted their own attestation via `submit_participant_info`. [6](#0-5)
