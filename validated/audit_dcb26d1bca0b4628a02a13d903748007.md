### Title
Incomplete Receipt Size Validation: `output_data_receivers` Appended After Size Check Allows Oversized Receipts to Bypass `max_receipt_size` Limit - (File: runtime/runtime/src/lib.rs)

### Summary

The NEAR runtime validates a newly-created receipt's size against `max_receipt_size` at creation time, but then unconditionally appends `output_data_receivers` entries to that receipt when `promise_return` is used. The size check is never re-run after this mutation, allowing receipts that exceed `max_receipt_size` to be forwarded into the state and across shards. This is a direct structural analog to the LongShortPair bug: a field is validated in isolation, but additional data appended later causes the total to exceed the enforced limit.

### Finding Description

`validate_receipt()` in `runtime/runtime/src/verifier.rs` enforces `max_receipt_size` only when called with `ValidateReceiptMode::NewReceipt`: [1](#0-0) 

This check is performed when a contract creates a new receipt (e.g., via `promise_create`). However, in `apply_action_receipt()` in `runtime/runtime/src/lib.rs`, after execution, if the parent receipt had `output_data_receivers` and the contract called `promise_return(C)`, the runtime appends those receivers directly to receipt C's `output_data_receivers` field: [2](#0-1) 

No size re-validation occurs after this mutation. Receipt C is then forwarded via `receipt_sink.forward_or_buffer_receipt()` without any size check: [3](#0-2) 

The codebase explicitly acknowledges this bug in the `ValidateReceiptMode` documentation: [4](#0-3) 

A second variant exists: when a contract returns a value of size `max_receipt_size`, the runtime wraps it in a `DataReceipt` without checking the total receipt size (the `DataReceipt` wrapper adds overhead), producing an oversized data receipt. Both variants are confirmed by integration tests: [5](#0-4) [6](#0-5) 

### Impact Explanation

An unprivileged user can craft a transaction that causes an oversized receipt to be stored in the state and forwarded to other shards, bypassing the `max_receipt_size` hard limit. This corrupts the `ChunkStateWitness` size invariant: the witness size limits (`max_receipt_size = 4 MiB`, `main_storage_proof_size_soft_limit`, etc.) are designed to keep the total uncompressed witness under 17 MiB. An oversized receipt can push the witness beyond this bound. Chunk validators independently enforce these limits; if the chunk producer includes an oversized receipt, validators may compute a different chunk application result, causing them to withhold endorsements and stalling finality for the affected shard.

### Likelihood Explanation

The trigger is straightforward and requires only a deployed smart contract that uses `promise_return` with a near-maximum-size receipt, or returns a value of size `max_receipt_size`. Any unprivileged account can deploy such a contract and submit the triggering transaction. The nearcore test suite already contains working proof-of-concept code demonstrating the bug is reachable.

### Recommendation

After appending `output_data_receivers` to a receipt in `apply_action_receipt`, re-validate the receipt size before forwarding it. Similarly, when constructing `DataReceipt` from a return value, validate the total receipt size (including wrapper overhead) against `max_receipt_size`. The validation should use `ValidateReceiptMode::NewReceipt` to enforce the hard limit.

### Proof of Concept

The existing integration test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` demonstrates the bug end-to-end:

1. A contract creates receipt C with args sized to exactly `max_receipt_size - base_receipt_size` (so C is exactly `max_receipt_size` bytes).
2. C passes `validate_receipt(..., NewReceipt)` at creation time.
3. The parent receipt calls `promise_return(C)`, causing the runtime to append `output_data_receivers` to C.
4. C is forwarded without re-validation; its size now exceeds `max_receipt_size`.
5. `assert_oversized_receipt_occurred` confirms the oversized receipt appears in the chain. [7](#0-6)
