### Title
Propose-Update Deposits Permanently Locked When Competing Proposals Are Cleared - (File: `crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

### Summary

When `do_update` executes a winning update proposal, it unconditionally clears **all** pending update entries and votes. Because `UpdateEntry` never records the proposer's account ID or the deposit amount, the NEAR storage deposits paid by proposers of non-executed proposals are permanently locked in the contract with no refund path.

### Finding Description

`propose_update` requires a caller-paid deposit proportional to the storage consumed by the update payload: [1](#0-0) 

The exact deposit required is computed from the WASM or config size plus a fixed overhead for up to 128 participant votes: [2](#0-1) 

Only the excess above `required` is refunded; the `required` portion is retained by the contract: [3](#0-2) 

The stored `UpdateEntry` records only the update payload and `bytes_used` — **not** the proposer's `AccountId` or the deposit amount: [4](#0-3) 

When threshold votes are reached for any update, `do_update` removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()` on all remaining proposals: [5](#0-4) 

Because no proposer identity or deposit amount is stored in `UpdateEntry`, there is no information available to issue refunds to the proposers of the cleared entries. The contract has no `remove_update_proposal` endpoint; `remove_update_vote` only removes the vote record, not the entry or the deposit: [6](#0-5) 

### Impact Explanation

NEAR tokens paid as storage deposits by proposers of non-executed update proposals are permanently frozen inside the contract. For a contract-code update, `bytes_used` includes the full WASM binary size plus a 128-participant-vote overhead. A typical MPC node WASM binary is hundreds of kilobytes; at NEAR's storage cost of ~10 yoctoNEAR per byte, the required deposit can reach tens of NEAR per proposal. Every time a winning update is executed, all competing proposers lose their full required deposit with no recovery mechanism.

This breaks the production accounting invariant that deposited funds are either consumed for their stated purpose or returned to the depositor. The funds are not consumed (the storage is freed when entries are cleared) yet they are never returned.

**Allowed impact matched**: Medium — balance/accounting invariant broken without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation

This occurs in normal governance operation whenever two or more participants independently propose different updates (e.g., different WASM versions or config changes) before one reaches threshold. This is a realistic scenario during any contested or concurrent upgrade cycle. No adversarial setup is required; the loss is a deterministic consequence of the current clearing logic.

### Recommendation

Store the proposer's `AccountId` and the deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. Alternatively, expose a `remove_update_proposal` endpoint that refunds the deposit to the original proposer when they withdraw their own proposal.

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM binary, attaching ~50 NEAR as the required deposit. `UpdateEntry { update: ..., bytes_used: 500_000 + overhead }` is stored; no proposer or deposit is recorded.
2. Participant B calls `propose_update` with a different WASM binary, attaching ~50 NEAR.
3. Threshold participants vote for A's update via `vote_update(id_A)`.
4. `do_update` is triggered: it removes entry A, then calls `self.entries.clear()` (removing entry B) and `self.vote_by_participant.clear()`.
5. Participant B's ~50 NEAR deposit remains in the contract permanently. There is no function to recover it. [5](#0-4) [7](#0-6)

    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
