### Title
Stale Cross-Chain Oracle Rate Causes Structural Pool Insolvency in L2 Deposit Pools - (`contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`)

---

### Summary

The L2 deposit pools issue rsETH/wrsETH to depositors using a cross-chain oracle rate that is always ≤ the actual current L1 rate (because rsETH appreciates continuously and the oracle is updated periodically, not in real-time). When the deposited ETH is later bridged to L1 and deposited into `LRTDepositPool`, the L1 mints fewer rsETH than the L2 pool already gave out. This structural rate mismatch creates a cumulative deficit — an exact analog to the NonCustodialPSM insolvency where CPI growth outpaces pcvDeposit yield.

---

### Finding Description

The L2 pools use `rsETHOracle` (either `CrossChainRateReceiver`/`RSETHRateReceiver` updated via LayerZero, or `InterimRSETHOracle` updated manually) to compute how much rsETH/wrsETH a depositor receives:

**`RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee`:**
```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0) 

The pool then immediately transfers rsETH from its own pre-loaded balance to the user:
```solidity
rsETH.safeTransfer(msg.sender, rsETHAmount);
``` [2](#0-1) 

The deposited ETH is later bridged to L1 via `bridgeAssets()` / `bridgeAssetsViaNativeBridge()`, where `L1Vault.depositETHForL1VaultETH()` deposits it into `LRTDepositPool` at the **current L1 rate** to mint rsETH, which is then bridged back to replenish the L2 pool. [3](#0-2) 

The oracle on L2 stores only the last received rate with no staleness enforcement:
```solidity
rate = _rate;
lastUpdated = block.timestamp;
``` [4](#0-3) 

The pool contracts never check `lastUpdated` before using `getRate()`. Since rsETH appreciates monotonically (staking + EigenLayer rewards), the L2 oracle rate `R_L2` is always ≤ the actual L1 rate `R_L1` at the time of L1 deposit.

**Deficit per deposit:**
- L2 gives user: `ETH * 1e18 / R_L2` rsETH
- L1 mints back: `ETH * 1e18 / R_L1` rsETH (where `R_L1 ≥ R_L2`)
- Deficit: `ETH * 1e18 * (R_L1 - R_L2) / (R_L2 * R_L1)` rsETH per deposit

The same structural deficit applies to `RSETHPoolV3` and `RSETHPoolV3ExternalBridge`, which mint wrsETH to users via `wrsETH.mint(msg.sender, rsETHAmount)` before the backing rsETH arrives from L1. [5](#0-4) [6](#0-5) 

The `InterimRSETHOracle` is manually set by a `MANAGER_ROLE` and has no on-chain staleness bound, making the lag potentially larger: [7](#0-6) 

---

### Impact Explanation

**Protocol insolvency (Critical class).** The L2 pool's rsETH reserve (for `RSETHPoolNoWrapper`) or the wrsETH wrapper's rsETH backing (for `RSETHPoolV3`/`RSETHPoolV3ExternalBridge`) is systematically underfunded on every deposit. The cumulative deficit grows with:
- Oracle update interval (hours to days)
- rsETH appreciation rate (~4–6% APR from staking + EigenLayer)
- Total deposit volume

At scale (e.g., $100M TVL, 5% APR, 24-hour oracle lag), the daily deficit is approximately $13,700. Over months, the pool cannot fully replenish its rsETH balance. For `RSETHPoolNoWrapper`, the pool eventually runs out of rsETH and cannot serve deposits. For the wrapper-based pools, wrsETH becomes undercollateralized — last redeemers receive less rsETH than their wrsETH entitles them to, or redemptions fail entirely.

There is no stability fund, backstop, or deficit-coverage mechanism in any of the pool contracts.

---

### Likelihood Explanation

**Medium.** The deficit is structural and continuous — it occurs on every single deposit, not just under adversarial conditions. No attacker action is required; ordinary depositors trigger it. The rate difference is small per block but the cumulative effect is material at protocol scale. The `InterimRSETHOracle` path (manual updates) creates larger windows of staleness. The `CrossChainRateReceiver` path depends on LayerZero message frequency, which can be delayed by network congestion or operator inaction.

---

### Recommendation

1. **Enforce oracle freshness**: Add a `maxStaleness` check in the pool's `getRate()` call — revert or cap the rate if `block.timestamp - lastUpdated > maxStaleness`.
2. **Rate-cap on deposit**: When computing `rsETHAmount`, use `max(oracleRate, lastKnownL1Rate)` to avoid issuing more rsETH than the ETH can buy on L1.
3. **Deficit accounting**: Track the cumulative rsETH deficit and limit new deposits when the deficit exceeds a threshold, similar to the report's recommendation to limit redemptions to the user's proportional share of the backing pool.
4. **Stability reserve**: Allocate a portion of collected fees to a reserve fund that covers the rate-lag deficit, analogous to the FEI backstop referenced in the original report.

### Proof of Concept

**Setup**: `RSETHPoolNoWrapper` on Arbitrum. Oracle (`InterimRSETHOracle`) last set at `R_L2 = 1.050e18` (1 rsETH = 1.050 ETH). Actual L1 rate at time of L1 deposit: `R_L1 = 1.055e18` (24 hours of ~4.7% APR accrual).

**Step 1 — User deposits 100 ETH on L2:**
```
rsETHAmount = 100e18 * 1e18 / 1.050e18 = 95.238 rsETH
```
Pool transfers 95.238 rsETH to user from its balance. [8](#0-7) 

**Step 2 — Bridger bridges 100 ETH to L1 (next day):**
`L1Vault.depositETHForL1VaultETH()` calls `lrtDepositPool.depositETH(100 ETH)` at current L1 rate `1.055e18`:
```
rsETHMinted = 100e18 * 1e18 / 1.055e18 = 94.787 rsETH
``` [3](#0-2) 

**Step 3 — rsETH bridged back to L2 pool:**
Pool receives 94.787 rsETH but gave out 95.238 rsETH.

**Deficit per cycle: 0.451 rsETH (~$1,350 at $3,000/ETH) on a single 100 ETH deposit.**

At $100M TVL with daily bridging, the annualized deficit is ~$4.9M. No contract mechanism covers this shortfall. The pool's rsETH reserve depletes monotonically until it cannot fulfill new deposits.
