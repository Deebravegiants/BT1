### Title
`LRTWithdrawalManager._unlockWithdrawalRequests` Burns Full rsETH Amount When Payout Is Capped at `expectedAssetAmount` - (File: contracts/LRTWithdrawalManager.sol)

### Summary
In `LRTWithdrawalManager._unlockWithdrawalRequests`, when the asset price has fallen (or rsETH price has risen) since a user initiated their withdrawal, the payout is capped at `expectedAssetAmount` (the amount locked in at initiation time). However, the full `request.rsETHUnstaked` is always added to `rsETHAmountToBurn`, regardless of whether the payout was capped. This causes the user to burn more rsETH than the fair value of the assets they receive.

### Finding Description

When a user calls `initiateWithdrawal`, their rsETH is transferred to the contract and `expectedAssetAmount` is recorded at the current oracle prices: [1](#0-0) 

Later, when an operator calls `unlockQueue`, `_unlockWithdrawalRequests` is invoked. Inside, `_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`: [2](#0-1) 

When `currentReturn > expectedAssetAmount` (i.e., the asset price has fallen or rsETH price has risen since initiation), `payoutAmount` is capped at `expectedAssetAmount`. However, the rsETH burn accumulator always uses the full original amount: [3](#0-2) 

The full `rsETHAmountToBurn` is then burned unconditionally: [4](#0-3) 

The fair value of `expectedAssetAmount` assets in rsETH at unlock time is `expectedAssetAmount * assetPrice / rsETHPrice`, which is strictly less than `rsETHUnstaked` when `currentReturn > expectedAssetAmount`. The user burns the full `rsETHUnstaked` but receives only `expectedAssetAmount` assets — the excess rsETH value is destroyed rather than returned.

### Impact Explanation

The user loses the rsETH value equivalent to `currentReturn - expectedAssetAmount` assets. This excess rsETH is burned (reducing total supply), benefiting remaining rsETH holders at the direct expense of the withdrawing user. The user receives exactly what they expected in asset terms, but their rsETH — which has appreciated — is burned in full rather than proportionally. This matches the **Low** impact tier: the contract fails to deliver the promised return (fair rsETH-to-asset exchange), but does not result in direct theft.

### Likelihood Explanation

This condition triggers whenever the rsETH/asset exchange rate moves favorably for rsETH between `initiateWithdrawal` and `unlockQueue`. This is a normal market scenario (e.g., stETH depeg, ETH price movement, or rsETH yield accrual) and requires no attacker action — any ordinary user who initiates a withdrawal before a favorable rate shift is affected. The operator calling `unlockQueue` need not be malicious; the loss occurs mechanically.

### Recommendation

When `payoutAmount < currentReturn`, burn only the rsETH proportional to the actual payout:

```solidity
// Instead of always using request.rsETHUnstaked:
uint256 rsETHToBurnForRequest = (payoutAmount < currentReturn)
    ? (payoutAmount * assetPrice / rsETHPrice)
    : request.rsETHUnstaked;
rsETHAmountToBurn += rsETHToBurnForRequest;
// Return the remainder to the user or credit it back
```

The remaining rsETH (`request.rsETHUnstaked - rsETHToBurnForRequest`) should be returned to the original requester.

### Proof of Concept

1. Alice calls `initiateWithdrawal` with `rsETHUnstaked = 1000e18` rsETH. At initiation, `rsETHPrice = 1.0e18`, `assetPrice = 1.0e18`, so `expectedAssetAmount = 1000e18` stETH. Alice's 1000 rsETH is transferred to the contract.
2. Before `unlockQueue` is called, stETH depegs: `assetPrice` drops to `0.8e18` while `rsETHPrice` stays at `1.0e18`.
3. Operator calls `unlockQueue`. `currentReturn = 1000e18 * 1.0e18 / 0.8e18 = 1250e18` stETH.
4. `payoutAmount = min(1000e18, 1250e18) = 1000e18` stETH. Alice receives 1000 stETH.
5. `rsETHAmountToBurn += 1000e18` — Alice's full 1000 rsETH is burned.
6. The fair rsETH cost of 1000 stETH at current prices is `1000e18 * 0.8e18 / 1.0e18 = 800e18` rsETH.
7. Alice loses 200 rsETH of value (burned but not compensated in assets). [5](#0-4)

    function _calculatePayoutAmount(
        WithdrawalRequest storage request,
        uint256 rsETHPrice,
        uint256 assetPrice
    )
        private
        view
        returns (uint256)
    {
        uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
        return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
    }
