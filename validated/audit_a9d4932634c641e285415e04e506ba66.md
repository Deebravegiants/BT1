### Title
Farmer Blindly Signs Attacker-Controlled `foliage_block_data_hash` Without Verifying `farmer_reward_puzzle_hash` — (`chia/farmer/farmer_api.py`)

### Summary

`FarmerAPI.request_signed_values` accepts a `RequestSignedValues` message from any connected full node and signs whatever `foliage_block_data_hash` is supplied, as long as the `quality_string` is recognized. It never verifies that the hash corresponds to a `FoliageBlockData` whose `farmer_reward_puzzle_hash` equals `farmer.farmer_target`. A malicious full node that has obtained a valid `quality_string` (by receiving a legitimate `DeclareProofOfSpace`) can substitute an attacker-controlled puzzle hash, obtain a valid plot-key signature over it, and broadcast a block that pays the farmer reward to the attacker.

### Finding Description

**Entrypoint — `FarmerAPI.request_signed_values`**

The only guard in the handler is a lookup of `quality_string`: [1](#0-0) 

After that check passes, the handler unconditionally forwards `full_node_request.foliage_block_data_hash` as the message to sign, and `full_node_request.foliage_block_data` as CHIP-22 source data, with no validation of either: [2](#0-1) 

There is no assertion that:
- `foliage_block_data.get_hash() == foliage_block_data_hash`
- `foliage_block_data.farmer_reward_puzzle_hash == self.farmer.farmer_target`

**Harvester signs blindly**

`HarvesterAPI.request_signatures` signs every hash in `request.messages` without inspecting the source data: [3](#0-2) 

**Farmer co-signs blindly**

`_process_respond_signatures` then co-signs the same `foliage_block_data_hash` returned by the harvester: [4](#0-3) 

**The check exists only in test code**

The test intercept `intercept_farmer_request_signed_values` in `test_third_party_harvesters.py` explicitly performs the missing check: [5](#0-4) 

This check is absent from the production handler.

**`quality_str_to_identifiers` stores no expected hash**

When a proof of space is found, the farmer stores only `(plot_identifier, challenge_hash, sp_hash, node_id)`: [6](#0-5) 

There is no stored expected `foliage_block_data_hash` to compare against.

**Consensus does not protect the farmer**

Block header validation only enforces `farmer_reward_puzzle_hash` for the genesis pre-farm block: [7](#0-6) 

For all other blocks, the only check is that the foliage block data signature is valid over `foliage_block_data.get_hash()` — which the attacker has obtained.

**Full node `signed_values` handler does not protect the farmer either**

The honest full node verifies the returned signature against the stored candidate's foliage block data hash: [8](#0-7) 

But the attacker IS the full node. They never call this handler; they use the obtained signature directly to construct and broadcast a block with their own `foliage_block_data`.

### Impact Explanation

A malicious full node that a farmer connects to can redirect 100% of the farmer's block rewards to an attacker-controlled address for every block the farmer wins while connected. The farmer's plot key produces a cryptographically valid signature over the attacker's foliage block data; the resulting block passes all consensus validation. This is a direct, irreversible financial loss.

### Likelihood Explanation

Farmers routinely connect to external full nodes. The attacker only needs to operate a full node that the farmer connects to, then wait for the farmer to find a proof of space. No leaked keys, no broken cryptography, and no social engineering are required. The attack is fully automated.

### Recommendation

In `FarmerAPI.request_signed_values`, before forwarding to the harvester, add:

1. If `foliage_block_data` is present, assert `foliage_block_data.get_hash() == full_node_request.foliage_block_data_hash`.
2. If `foliage_block_data` is present, assert `foliage_block_data.farmer_reward_puzzle_hash == self.farmer.farmer_target` (allowing the genesis pre-farm constant as the test code does).
3. For the case where `foliage_block_data` is absent (no CHIP-22 source data), the farmer has no way to verify the puzzle hash from the hash alone. The farmer should store the expected `foliage_block_data_hash` at the time the candidate block is created (i.e., when `DeclareProofOfSpace` is processed and the full node sends back `RequestSignedValues` in the legitimate flow), or require `foliage_block_data` to always be present so the check in point 2 can be applied.

async def test_malicious_full_node_redirects_farmer_reward():
    attacker_ph = bytes32(b"\xaa" * 32)

    # 1. Farmer finds a proof of space; quality_string is registered in
    #    farmer.quality_str_to_identifiers by new_proof_of_space handler.
    quality_string = <valid quality string from farmer>

    # 2. Construct attacker foliage block data with attacker's puzzle hash.
    attacker_foliage_block_data = FoliageBlockData(
        unfinished_reward_block_hash=<valid hash>,
        pool_target=<valid pool target>,
        pool_signature=<valid or None>,
        farmer_reward_puzzle_hash=attacker_ph,  # <-- attacker's address
        extension_data=bytes32.zeros,
    )

    # 3. Malicious full node sends RequestSignedValues with attacker's hash.
    malicious_request = farmer_protocol.RequestSignedValues(
        quality_string=quality_string,
        foliage_block_data_hash=attacker_foliage_block_data.get_hash(),
        foliage_transaction_block_hash=bytes32.zeros,
        foliage_block_data=attacker_foliage_block_data,  # CHIP-22 source data
    )

    # 4. Farmer signs without rejection.
    result = await farmer_api.request_signed_values(malicious_request)
    assert result is not None  # farmer returned SignedValues

    signed_values = farmer_protocol.SignedValues.from_bytes(result.data)

    # 5. Verify the signature is over the attacker's foliage block data hash.
    assert AugSchemeMPL.verify(
        plot_public_key,
        attacker_foliage_block_data.get_hash(),
        signed_values.foliage_block_data_signature,
    )
    # Attacker now has a valid signature to build a block paying rewards to attacker_ph.
