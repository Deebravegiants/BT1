### Title
Unvalidated HTTP Redirect in `_pool_get_pool_info` Permanently Overwrites Farmer `pool_url` Without Domain Validation — (`File: chia/farmer/farmer.py`)

### Summary
The `Farmer._pool_get_pool_info` method in `chia/farmer/farmer.py` follows HTTP 301/308 redirects from the pool server and, without any domain validation, permanently overwrites the farmer's stored `pool_url` in `PoolingShareState` with the redirect destination. A malicious or compromised pool operator can redirect the farmer to an attacker-controlled server, causing all subsequent pool protocol traffic (partial submissions, authentication signatures, payout instructions) to be sent to the attacker's server instead of the legitimate pool.

### Finding Description

In `_pool_get_pool_info`, the farmer makes a GET request to `{pool_config.pool_url}/pool_info`. If the response was reached via a chain of only 301 or 308 redirects, the final response URL is extracted and stored as the new `pool_url`:

```python
new_pool_url: str | None = None
response_url_str = f"{resp.url}"
if (
    response_url_str != url
    and len(resp.history) > 0
    and all(r.status in {301, 308} for r in resp.history)
):
    new_pool_url = response_url_str.replace("/pool_info", "")
``` [1](#0-0) 

This `new_pool_url` is then unconditionally written back to the persistent `PoolingShareState` config:

```python
if pool_info_result is not None and pool_info_result.new_pool_url is not None:
    with PoolingShareState.acquire(...) as editable_pool_config:
        editable_pool_config.pool_url = pool_info_result.new_pool_url
``` [2](#0-1) 

There is **no validation** that the redirect destination belongs to the same domain, a trusted domain, or even uses HTTPS. The only guard in `update_pool_state` is a `startswith("https://")` check on the *original* `pool_config.pool_url` before the request is made — not on the redirect destination. [3](#0-2) 

After the `pool_url` is overwritten, all subsequent pool protocol calls — `GET /farmer`, `POST /farmer`, `PUT /farmer`, and `POST /partial` — are sent to the attacker-controlled URL: [4](#0-3) [5](#0-4) 

### Impact Explanation

Once the `pool_url` is redirected to an attacker-controlled server:

1. **Payout redirection**: All proof-of-space partials (`POST /partial`) are submitted to the attacker's server. The attacker never credits the farmer's points, causing permanent loss of pool rewards.
2. **Authentication key exposure**: The farmer sends signed `AuthenticationPayload` messages (via `GET /farmer`) to the attacker's server, leaking authentication signatures.
3. **Persistent state corruption**: The overwrite is written to the on-disk `PoolingShareState` YAML file, so the redirection persists across farmer restarts.

This matches the allowed impact: **High — Corruption of pool membership state with direct security impact** and **High — Payout redirection**.

### Likelihood Explanation

The attacker must control or compromise the pool's HTTP server to issue a 301/308 redirect. This is a realistic threat: pool operators can turn malicious, pool infrastructure can be compromised, or a MITM attacker on the network path can inject a redirect response. The farmer has no mechanism to detect or reject a cross-domain redirect.

### Recommendation

After following a redirect, validate that the final URL's scheme and host match the original `pool_config.pool_url`. For example:

```python
from urllib.parse import urlparse
original = urlparse(pool_config.pool_url)
redirected = urlparse(response_url_str)
if original.netloc != redirected.netloc or original.scheme != redirected.scheme:
    # reject redirect to different domain
    new_pool_url = None
```

Additionally, apply the HTTPS enforcement check to `new_pool_url` before persisting it.

### Proof of Concept

1. Farmer joins a pool at `https://legitimate-pool.com`.
2. Pool operator configures `https://legitimate-pool.com/pool_info` to return HTTP 301 → `https://attacker.com/pool_info`.
3. `aiohttp` follows the redirect; `resp.url` becomes `https://attacker.com/pool_info`; `resp.history[0].status == 301`.
4. The condition at lines 374–378 is satisfied; `new_pool_url = "https://attacker.com"`.
5. `PoolingShareState.pool_url` is overwritten to `https://attacker.com` on disk.
6. On the next `update_pool_state` cycle, all `GET /farmer`, `POST /farmer`, `PUT /farmer`, and `POST /partial` calls go to `https://attacker.com`.
7. The attacker's server receives the farmer's authentication signatures and all partial proofs, and never credits the farmer's points with the legitimate pool — permanently diverting pool rewards.
