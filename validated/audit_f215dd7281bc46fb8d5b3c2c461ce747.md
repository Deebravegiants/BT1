### Title
Single Attested Participant Can Submit Mismatched `payload_hash` in `respond_verify_foreign_tx`, Enabling Forged Foreign-Chain Verification — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function accepts a `payload_hash` from the MPC node's response without verifying it against the hash expected from the original request. A single malicious attested participant can submit a cryptographically valid signature (produced by the threshold protocol for a different transaction) paired with a mismatched `payload_hash`, causing the contract to resolve a victim's pending foreign-transaction verification request with a forged response.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract verifies the signature against `response.payload_hash`, which is entirely controlled by the submitting MPC node: [1](#0-0) 

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

Unlike the `respond` function, which derives the payload hash directly from the stored user request: [2](#0-1) 

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
.is_ok()
```

`respond_verify_foreign_tx` trusts the node-supplied hash without cross-checking it against the transaction details in `request`. The contract then resolves the yield keyed on `request` (the victim's original request) while delivering the forged `response`: [3](#0-2) 

**Attack path:**

1. Malicious attested participant submits `verify_foreign_transaction` for transaction **Y** (as a normal user).
2. The threshold MPC protocol runs normally; the malicious node, acting as signing leader for Y's session, obtains the full signature over Y's `payload_hash`.
3. The malicious node calls `respond_verify_foreign_tx` supplying a **different victim's** pending `request` (for transaction **X**) but Y's `payload_hash` and Y's signature.
4. The contract verifies the signature against Y's hash (valid), finds the pending yield for X, and resolves it with Y's response.
5. The victim receives a `VerifyForeignTransactionResponse` asserting their transaction X was verified, but the embedded signature is for Y.

The authorization check only confirms the caller is an attested participant: [4](#0-3) 

It does not enforce that `response.payload_hash` corresponds to the transaction described in `request`.

---

### Impact Explanation

**High.** The victim's application receives a `VerifyForeignTransactionResponse` whose `payload_hash` and `signature` correspond to a completely different foreign transaction. Any downstream bridge logic that trusts the contract's response without independently re-deriving the expected hash will authorize an action based on a forged verification, enabling invalid bridge execution or double-spend conditions. This matches the allowed impact: *"forged foreign-chain verification… that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

A **single** attested participant (strictly below threshold) can execute this attack. The attacker must be the signing leader for at least one foreign-transaction session — leadership rotates among participants, making the attack opportunistic but not requiring threshold collusion. The TEE attestation requirement raises the bar but does not eliminate the risk from a compromised or malicious node operator. The on-chain pending request map is public, so the attacker can freely choose which victim request to target.

---

### Recommendation

Derive the expected `payload_hash` inside the contract from the original `request` fields (chain, transaction ID, etc.) and assert that `response.payload_hash` equals this derived value before accepting the response. This mirrors the pattern already used in `respond`, where the payload is taken from the stored request rather than from the node's response: [5](#0-4) 

Concretely, add a binding check before the signature verification in `respond_verify_foreign_tx`:

```rust
// Derive expected hash from the original request, not from the response
let expected_hash = derive_payload_hash_from_request(&request);
if response.payload_hash.0 != expected_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
// Then verify signature against expected_hash
```

---

### Proof of Concept

```
// Malicious leader node for session Y calls:
respond_verify_foreign_tx(
    request  = <victim X's pending VerifyForeignTransactionRequest>,
    response = VerifyForeignTransactionResponse {
        payload_hash: <hash of Y's tx_id>,   // mismatched — Y, not X
        signature:    <valid MPC threshold signature over Y's hash>,
    }
)
```

The contract:
1. Confirms caller is an attested participant ✓
2. Verifies signature against `response.payload_hash` (Y's hash) — **passes** because the threshold protocol legitimately signed Y.
3. Resolves the yield keyed on X's `request` — **delivers Y's forged response to X's caller**.

The victim's bridge contract receives a valid-looking `VerifyForeignTransactionResponse` for a transaction it never actually verified.
