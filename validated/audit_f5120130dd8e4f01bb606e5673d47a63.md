Looking at the two settlement paths and their deduplication mechanisms carefully.

### Title
Double-Mint of eNear via Disjoint Receipt-ID Namespaces Between OmniBridge MPC Path and Legacy NEAR Proof Path — (`evm/src/eNear/contracts/ENearProxy.sol`)

---

### Summary

`ENearProxy` exposes two independent settlement paths for the same NEAR→ETH eNear transfer. The OmniBridge MPC path (`OmniBridge.finTransfer` → `ENearProxy.mint`) injects a **synthetic, sequential receipt ID** (`currentReceiptId`) into a fake proof before calling `eNear.finaliseNearToEthTransfer`. The legacy NEAR-proof path (`ENearProxy.finaliseNearToEthTransfer`) accepts the **real NEAR receipt ID** from an on-chain proof. Because the two receipt IDs are always different, eNear's internal deduplication (keyed on receipt ID) cannot detect that both calls settle the same underlying NEAR lock event. An unprivileged attacker who is also the transfer recipient can collect eNear twice for a single NEAR-side lock.

---

### Finding Description

**OmniBridge MPC path — fake receipt ID**

`ENearProxy.mint` is called by `OmniBridge.finTransfer` when eNear is the custom-minter token:

```
OmniBridge.finTransfer
  └─ ICustomMinter(customMinters[eNear]).mint(eNear, recipient, amount)
       └─ ENearProxy.mint(token, to, amount)
```

Inside `ENearProxy.mint`, a synthetic proof is assembled using `currentReceiptId` — a monotonically incrementing counter that has **no relationship** to any real NEAR receipt: [1](#0-0) 

`currentReceiptId` is incremented after each call, so the fake proof always carries a fresh, never-before-seen receipt ID (call it **N**). `eNear.finaliseNearToEthTransfer` records N as consumed and mints.

**Legacy NEAR-proof path — real receipt ID**

`ENearProxy.finaliseNearToEthTransfer` is a public, permissionless function gated only by a runtime pause flag: [2](#0-1) 

It validates the proof via `prover.proveOutcome` and forwards it directly to `eNear.finaliseNearToEthTransfer`. The real NEAR proof carries the actual receipt ID **M** (M ≠ N). There is **no check** against `OmniBridge.completedTransfers` here.

**The invariant break**

`OmniBridge.finTransfer` deduplicates by `destinationNonce`: [3](#0-2) 

`eNear` deduplicates by receipt ID embedded in the proof. Because the MPC path injects receipt ID N (synthetic) and the legacy path presents receipt ID M (real), the two deduplication domains are **disjoint**. Neither check covers the other path.

---

### Impact Explanation

An attacker who is the intended recipient of a NEAR→ETH eNear transfer can:

1. Receive eNear via the OmniBridge MPC path (receipt N consumed by eNear).
2. Immediately call `ENearProxy.finaliseNearToEthTransfer` with the real NEAR proof (receipt M, never consumed) and receive the same amount again.

Result: 2× eNear minted for 1× NEAR locked. The excess eNear is unbacked, violating the 1:1 backing invariant and constituting unauthorized creation of wrapped bridge assets.

---

### Likelihood Explanation

- `PAUSED_LEGACY_FIN_TRANSFER` is a runtime flag with no on-chain enforcement that it must be set when the OmniBridge MPC path is active. If both paths are live simultaneously, the attack is immediately executable.
- The attacker needs only a valid NEAR proof for their own transfer — a standard output of any NEAR light-client prover, requiring no privileged access.
- The attacker is the transfer recipient, so this is not self-harm: they profit directly.

---

### Recommendation

1. **Pause the legacy path permanently** once the OmniBridge MPC path is the canonical settlement mechanism. Enforce this at the contract level, not just via a runtime flag.
2. If both paths must coexist, `ENearProxy.finaliseNearToEthTransfer` must record the real NEAR receipt ID in a shared set and cross-check it before calling `eNear.finaliseNearToEthTransfer`. Alternatively, `ENearProxy.mint` must extract and use the **actual NEAR receipt ID** from the MPC payload so that eNear's own deduplication covers both paths.
3. Do not use a synthetic sequential counter (`currentReceiptId`) as a stand-in for the real NEAR receipt ID; it severs the deduplication link between the two settlement paths.

---

### Proof of Concept

```
// Setup: eNear is registered in OmniBridge with ENearProxy as customMinter.
//        PAUSED_LEGACY_FIN_TRANSFER is NOT set.
//        Attacker is the recipient of a NEAR→ETH transfer of 100 eNear.

// Step 1: MPC relayer settles via OmniBridge.
//   OmniBridge.finTransfer(sig, {destinationNonce: 42, tokenAddress: eNear,
//                                 amount: 100, recipient: attacker, ...})
//   → completedTransfers[42] = true
//   → ENearProxy.mint(eNear, attacker, 100)
//       → fakeProof uses currentReceiptId = N
//       → eNear.finaliseNearToEthTransfer(fakeProof, 0)  // eNear marks N used
//   attacker.balance(eNear) == 100  ✓

// Step 2: Attacker calls legacy path with real NEAR proof (receipt ID M ≠ N).
//   ENearProxy.finaliseNearToEthTransfer(realProof_M, blockHeight)
//   → prover.proveOutcome(realProof_M, blockHeight) == true  (valid proof)
//   → eNear.finaliseNearToEthTransfer(realProof_M, blockHeight)
//       // eNear checks: is M used? NO → mints 100 to attacker
//   attacker.balance(eNear) == 200  ✓

// assert eNear.totalSupply() == 200   // 2× expected; 100 unbacked
```
