### Title
Incomplete CKD Response Verification: `AppPublicKey` Variant Accepts Unverified Output in `respond_ckd` - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` applies a cryptographic output check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`, while the `AppPublicKey` variant receives **no cryptographic verification** of the returned `CKDResponse`. Any single attested participant can race to call `respond_ckd` with an arbitrary forged response for a pending `AppPublicKey`-type CKD request and have it accepted and delivered to the user, bypassing the threshold-computation requirement for that output.

---

### Finding Description

The contract exposes three node-facing respond endpoints: `respond`, `respond_ckd`, and `respond_verify_foreign_tx`. Both `respond` and `respond_verify_foreign_tx` always perform cryptographic verification of the submitted response before resolving the pending yield. `respond_ckd` does so only conditionally:

```
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKey`, the branch is a no-op. The function then immediately calls `resolve_yields_for`, which delivers whatever `CKDResponse` was supplied to the waiting caller.

The access guard (`assert_caller_is_attested_participant_and_protocol_active`) requires only that the caller is **any** single attested participant — not the designated leader, not a threshold quorum. There is no leader-binding check on `respond_ckd`.

By contrast, `respond` always verifies the ECDSA/EdDSA signature against the derived public key before resolving:

```
// crates/contract/src/lib.rs  lines 602-608
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
.is_ok()
```

And `respond_verify_foreign_tx` always verifies the ECDSA signature against the root public key before resolving. The `AppPublicKey` branch of `respond_ckd` is the only respond path with no cryptographic output check.

---

### Impact Explanation

A Byzantine attested participant (strictly below threshold) can submit an arbitrary `CKDResponse{big_y, big_c}` for any pending `AppPublicKey`-type CKD request. The contract accepts it without verification and delivers it to the requesting user via `resolve_yields_for`. The user's application then uses the attacker-supplied values as its derived private key material. This constitutes **unauthorized confidential key derivation output without the required participant authorization** — the threshold MPC computation is bypassed entirely for this request type.

---

### Likelihood Explanation

- The attacker needs only to be a single attested participant, which is a realistic adversary model (Byzantine participant below threshold).
- No threshold collusion is required.
- The attacker simply monitors the contract for pending `CKDRequest` entries with `AppPublicKey` variant and races to call `respond_ckd` before the legitimate leader.
- `resolve_yields_for` delivers the first accepted response; there is no deduplication or quorum requirement at the contract level for this call.

---

### Recommendation

**Short term:** Add a cryptographic output check for the `AppPublicKey` variant in `respond_ckd`, analogous to the existing `ckd_output_check` for `AppPublicKeyPV`. The check should verify that the returned `(big_y, big_c)` is consistent with the master BLS12-381 public key and the request parameters, so that a forged response is rejected on-chain.

**Long term:** Unify the verification logic across all three `respond*` functions so that every path performs a mandatory cryptographic check before resolving a yield. Consider binding `respond_ckd` to the designated leader's account ID to prevent non-leader participants from racing to submit responses.

---

### Proof of Concept

1. Alice calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk)` and attaches the required deposit. A `CKDRequest` is enqueued in `pending_ckd_requests`.

2. Attacker (an attested participant, not the leader) observes the pending request on-chain.

3. Attacker calls `respond_ckd(ckd_request, CKDResponse { big_y: [1u8;48].into(), big_c: [2u8;48].into() })`.

4. The contract executes:
   - `assert_caller_is_signer()` — passes (attacker calls directly) [1](#0-0) 
   - `is_running_or_resharing()` — passes [2](#0-1) 
   - `accept_requests` — passes [3](#0-2) 
   - `assert_caller_is_attested_participant_and_protocol_active()` — passes (attacker is attested) [4](#0-3) 
   - `AppPublicKey` branch: **no-op** [5](#0-4) 
   - `resolve_yields_for` delivers the forged response to Alice [6](#0-5) 

5. Alice's application receives `big_y` and `big_c` values chosen by the attacker and uses them as her derived key material.

The same attacker calling `respond` with a forged ECDSA signature would be rejected at the `verify_ecdsa_signature` step. [7](#0-6)  The `AppPublicKey` branch of `respond_ckd` is the only respond path that lacks this protection. [8](#0-7)
