### Title
Delayed `verify_tee` Transaction Can Unexpectedly Halt All Signing Operations — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_tee` function accepts no deadline or block-height argument and evaluates TEE attestation validity using `block_timestamp_ms()` at execution time. A single participant (below threshold) can submit a `verify_tee` call when all attestations are valid, have it delayed in the NEAR transaction pool, and have it execute after some attestations have expired — causing the contract to set `accept_requests = false` and permanently halt all signing, CKD, and foreign-chain-verification operations until manual intervention.

---

### Finding Description

`verify_tee` is a public, participant-callable method that re-evaluates every current participant's TEE attestation against the current block timestamp and the current whitelist of allowed image hashes. [1](#0-0) 

When `reverify_and_cleanup_participants` returns `TeeValidationResult::Partial` and the surviving participant count is too small to satisfy the threshold relation, the function unconditionally writes: [2](#0-1) 

This flag is the global gate for every user-facing and node-facing operation: [3](#0-2) [4](#0-3) 

The function carries **no deadline, no block-height bound, and no epoch guard**. The codebase itself documents why such a guard is necessary — `vote_new_parameters` explicitly requires a `prospective_epoch_id` argument for exactly this reason: [5](#0-4) 

`verify_tee` has no equivalent protection.

---

### Impact Explanation

When `accept_requests` is set to `false`:

- Every call to `sign()`, `request_app_private_key()`, and `verify_foreign_transaction()` panics with `TeeError::TeeValidationFailed`.
- Every call to `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` returns an error, so MPC nodes cannot resolve any pending yield-resume promise.
- All in-flight signature requests time out without resolution, permanently losing the attached deposit and blocking the requesting contract.

The MPC network is effectively frozen for all cross-chain signing until a governance action restores `accept_requests = true` — which itself requires a quorum of participants to coordinate a new `verify_tee` call that succeeds, or a contract upgrade.

This matches the allowed Medium impact: **contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration**.

---

### Likelihood Explanation

`verify_tee` requires only `voter_or_panic()` — a single participant, strictly below the signing threshold, can call it at any time. [6](#0-5) 

NEAR transactions are not guaranteed to execute in the same block they are submitted. A transaction submitted during a period of high network load, or one that is re-broadcast after a transient failure, can execute seconds to minutes later. TEE attestation certificates have finite validity windows; the `tee_upgrade_deadline_duration` grace period for image-hash transitions is also finite.

A targeted attacker who is a participant can:
1. Observe that one or more participants' attestations are within seconds of expiry.
2. Submit `verify_tee` just before expiry.
3. Allow the transaction to execute after expiry.
4. The contract finds fewer valid participants than the threshold requires and sets `accept_requests = false`.

No collusion, no key material, and no network-level interference is required — a single Byzantine participant below the signing threshold suffices.

---

### Recommendation

Add a `deadline_block_height: u64` (or `deadline_timestamp_ms: u64`) parameter to `verify_tee`. At the start of the function, before any state mutation, check:

```rust
if env::block_height() > deadline_block_height {
    // Revert without touching accept_requests
    return Err(InvalidParameters::DeadlineExceeded.into());
}
```

This mirrors the protection already present in `vote_new_parameters` via `prospective_epoch_id` and ensures that a stale or delayed `verify_tee` transaction reverts harmlessly rather than pausing the contract.

---

### Proof of Concept

**Setup**: A running MPC network with N=5 participants, threshold=3. Participant attestations expire at block timestamp T.

**Step 1 — Attacker submits `verify_tee` at T−1 s.**
The transaction is valid at submission time: all 5 attestations pass `re_verify`.

**Step 2 — Transaction is delayed.**
Due to NEAR network congestion or deliberate re-broadcast timing, the transaction executes at T+1 s.

**Step 3 — `reverify_and_cleanup_participants` runs at T+1 s.**
`TeeState::reverify_participants` calls `VerifiedAttestation::re_verify` with `current_time_seconds = T+1`. [7](#0-6) 

Suppose 3 of the 5 attestations have expired. `TeeValidationResult::Partial` is returned with only 2 valid participants.

**Step 4 — Threshold check fails.**
`validate_governance_against_reconstruction` finds that 2 < threshold (3), so: [8](#0-7) 

**Step 5 — All signing halts.**
Every subsequent `sign()` call panics; every `respond()` call returns `TeeError::TeeValidationFailed`. Pending yield-resume promises time out. The MPC network is frozen until a governance quorum intervenes.
