### Title
Chainlink Price Feed Staleness and Round Completeness Not Validated in `ChainlinkPriceOracle` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards all validation fields (`roundId`, `updatedAt`, `answeredInRound`). A stale or incomplete Chainlink round will propagate an incorrect price into `LRTOracle._updateRsETHPrice()`, corrupting the rsETH exchange rate used for all deposits and withdrawals.

### Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, the call to `latestRoundData()` discards every field except `price`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is made for:
- **Stale price**: `answeredInRound >= roundId` is never verified.
- **Round completeness**: `updatedAt > 0` is never verified.
- **Non-positive price**: `price > 0` is never verified. A negative `price` cast to `uint256` wraps to an astronomically large value. [1](#0-0) 

By contrast, the sibling contract `ChainlinkOracleForRSETHPoolCollateral.sol` in the same repository correctly validates all three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unvalidated price flows through `LRTOracle.getAssetPrice()` → `LRTOracle._getTotalEthInProtocol()` → `LRTOracle._updateRsETHPrice()`, which computes and stores the global `rsETHPrice`: [3](#0-2) [4](#0-3) 

### Impact Explanation

`rsETHPrice` is the exchange rate used to mint rsETH on deposit and to determine ETH owed on withdrawal. A stale or zero price for any supported collateral asset (e.g., stETH, cbETH) will:

- **Artificially deflate** `totalETHInProtocol` → lower `rsETHPrice` → depositors receive more rsETH than the true value of their deposit, diluting all existing rsETH holders.
- **Artificially inflate** `totalETHInProtocol` (e.g., via negative-price wrap-around) → higher `rsETHPrice` → depositors receive less rsETH, or the price-increase guard triggers incorrectly.

This constitutes incorrect share accounting that directly harms rsETH holders and depositors. Impact: **Low to Medium** — contract fails to deliver correct exchange rates; in the negative-price wrap-around edge case the impact escalates toward protocol insolvency.

### Likelihood Explanation

`updateRSETHPrice()` is a public function callable by any unprivileged address. [5](#0-4) 

Chainlink feeds can go stale during network congestion, oracle node downtime, or when a feed is deprecated. The probability is low-to-medium but the trigger requires no special privilege.

### Recommendation

Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral.sol`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    require(price > 0, "ChainlinkPriceOracle: invalid price");
    require(answeredInRound >= roundId, "ChainlinkPriceOracle: stale price");
    require(updatedAt > 0, "ChainlinkPriceOracle: incomplete round");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Optionally add a `heartbeat`-based staleness check (e.g., `block.timestamp - updatedAt <= MAX_DELAY`) per feed.

### Proof of Concept

1. A Chainlink feed for a supported collateral (e.g., stETH/ETH) goes stale — `answeredInRound < roundId` — returning a price from a previous round that is significantly lower than the current market price.
2. Any unprivileged caller invokes `LRTOracle.updateRSETHPrice()`.
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale (lower) price with no revert.
4. `newRsETHPrice` is computed lower than the true value.
5. A depositor who observes the stale update immediately deposits ETH-equivalent assets and receives more rsETH than the true exchange rate warrants, at the expense of existing rsETH holders. [1](#0-0) [5](#0-4) [6](#0-5)
