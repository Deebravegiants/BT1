### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Resolved `request` — Forged Foreign-Chain Verification Attestation - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash` using the root public key, but it never checks that `response.payload_hash` is actually derived from the `request` being resolved. A single malicious MPC participant who acts as leader for any signing session can reuse a legitimately-produced threshold signature — computed for request R1 — to resolve a completely different pending request R2, delivering a fabricated foreign-chain attestation to R2's caller.

---

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs two independent checks before resolving a pending yield:

1. The submitted `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key.
2. The `request` key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What it does **not** check is that `response.payload_hash` is actually the canonical hash of the foreign-chain payload for the specific `request` being resolved. The `payload_hash` is a caller-supplied field inside `VerifyForeignTransactionResponse`: [2](#0-1) 

The correct hash is `SHA-256(borsh(ForeignTxSignPayload { request, values }))`, where `values` are the extracted on-chain values the MPC nodes observed: [3](#0-2) 

Because the contract never recomputes or validates this binding, a malicious leader who has obtained a valid threshold signature `S1` over hash `H1` (produced for request R1 with extracted values V1) can submit:

```
respond_verify_foreign_tx(request = R2, response = { payload_hash: H1, signature: S1 })
```

The contract accepts this call because:
- `S1` is a valid signature over `H1` under the root key → `signature_is_valid = true`
- R2 is present in `pending_verify_foreign_tx_requests` → `resolve_yields_for` succeeds [4](#0-3) 

R2's yield is then resumed with `{ payload_hash: H1, signature: S1 }` — an attestation that encodes R1's foreign-chain data, not R2's. R2's caller receives a cryptographically valid but semantically forged verification response.

The node-side signing code confirms that `payload_hash` is computed by the leader from the actual observed foreign-chain data and is never independently verified by the contract: [5](#0-4) 

The SDK-level verifier (`ForeignChainSignatureVerifier::verify_signature`) does check that `response.payload_hash` matches the expected hash for the request — but this check lives entirely off-chain in the caller's code, not in the contract: [6](#0-5) 

Any bridge or dApp that does not perform this off-chain check (or that trusts the contract's acceptance as sufficient) is directly exploitable.

---

### Impact Explanation

**Critical/High — Forged foreign-chain verification enabling invalid bridge execution.**

A bridge contract that calls `verify_foreign_transaction` to gate fund releases receives a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to the transaction it submitted for verification. If the bridge checks only that the signature is valid over the returned `payload_hash` (rather than recomputing the expected hash from the original request), it will release funds for a transaction that was never actually verified on the foreign chain. This directly enables double-spend or invalid bridge execution.

Even when the off-chain SDK check is performed, the pending request R2 has been permanently consumed (its yield drained from `pending_verify_foreign_tx_requests`). R2's caller cannot retry; their request is gone.

---

### Likelihood Explanation

**High.** The attack requires only a single malicious MPC participant to be the leader for any one signing session. Leader selection is deterministic and rotates across participants, so any participant will eventually lead at least one session. The attacker needs two simultaneously pending foreign-tx requests (R1 and R2) — a normal condition in any active bridge. No threshold collusion is required: the attacker participates honestly in R1's signing session as leader, obtains the final threshold signature, and then submits it against R2 instead of R1. The `respond_verify_foreign_tx` entry point does not restrict which pending request a participant may respond to. [7](#0-6) 

---

### Recommendation

The contract must verify that `response.payload_hash` is consistent with the `request` being resolved. Since the contract cannot independently query the foreign chain to obtain the extracted `values`, the binding must be enforced structurally. Two complementary approaches:

**Option A — Include the request hash in the signed payload at the node level, and verify it on-chain.**
Require that `payload_hash` is always `SHA-256(borsh(ForeignTxSignPayload { request, values }))` and that the `request` field inside the borsh-serialized payload matches the `request` argument to `respond_verify_foreign_tx`. This can be enforced by having the contract recompute `SHA-256(borsh(request))` and checking that it is a prefix/component of `payload_hash` — or by changing the payload format so the request hash is a separately verifiable component.

**Option B — Bind the pending-request map key to the expected payload hash at submission time.**
When `verify_foreign_transaction` is called, store the expected `payload_hash` prefix (i.e., `SHA-256(borsh(request))`) alongside the yield index. In `respond_verify_foreign_tx`, verify that `response.payload_hash` starts with (or is derived from) the stored prefix.

At minimum, add a guard analogous to the `_swap()` fix in the reference report:

```rust
// Recompute the hash of just the request portion and assert it is
// a prefix of the submitted payload_hash, preventing cross-request replay.
let request_hash = sha256(borsh::to_vec(&request.request)?);
require!(
    payload_hash[..32] == request_hash,
    "payload_hash does not correspond to the submitted request"
);
```

### Proof of Concept

```
Setup:
  - Two pending foreign-tx requests exist:
      R1 = VerifyForeignTransactionRequest { request: Bitcoin(tx_id: T1, ...), domain_id: D, ... }
      R2 = VerifyForeignTransactionRequest { request: Bitcoin(tx_id: T2, ...), domain_id: D, ... }
  - Malicious participant M is the leader for R1's signing session.

Attack:
  1. M participates honestly in the threshold signing session for R1.
     The session produces signature S1 over H1 = SHA256(borsh(ForeignTxSignPayload{R1, V1})).
  2. M calls respond_verify_foreign_tx(request = R2, response = { payload_hash: H1, signature: S1 }).
  3. Contract checks:
       - assert_caller_is_attested_participant_and_protocol_active() → passes (M is a participant)
       - verify_ecdsa_signature(S1, H1, root_pk) → passes (S1 is a valid threshold sig over H1)
       - resolve_yields_for(pending_verify_foreign_tx_requests, R2, serialize({H1, S1})) → passes (R2 is pending)
  4. R2's yield is resumed with { payload_hash: H1, signature: S1 }.
     R2's caller receives an attestation that encodes T1's block hash, not T2's.
  5. R1 remains pending and eventually times out; its caller receives no response.

Result:
  - R2's caller holds a cryptographically valid signature over H1, which encodes T1's data.
  - Any bridge that does not recompute the expected hash for T2 will accept this as proof that T2 was verified.
  - R2's pending slot is permanently consumed; the caller cannot retry.
```
