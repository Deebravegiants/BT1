### Title
`RequireGuard` Native Script Checks Self-Declared `guards` Field Instead of Verified Witness Set, Allowing Unauthorized Script Satisfaction - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`)

### Summary
In the Dijkstra era, the new `RequireGuard` native script type evaluates by checking whether a credential is present in the transaction body's `guards` field. For `KeyHashObj` credentials, no signature from the corresponding key is required — any transaction author can include any key hash in the `guards` set and satisfy the script. This allows an attacker to spend UTxOs locked by `RequireGuard (KeyHashObj kh)` scripts without possessing the private key for `kh`.

### Finding Description

The Dijkstra era introduces `RequireGuard` as a new native script constructor. Its evaluation is defined in `evalDijkstraNativeScript`:

```haskell
RequireGuard cred -> cred `OSet.member` guards
``` [1](#0-0) 

The `guards` parameter is the `OSet (Credential Guard)` drawn directly from the transaction body's `guardsTxBodyL` field: [2](#0-1) 

Contrast this with `RequireSignature`, which checks against `keyHashes` — the set of key hashes derived from the **verified** witness set (`keyHashWitnessesTxWits (tx ^. witsTxL)`):

```haskell
RequireSignature hash -> hash `Set.member` keyHashes
``` [3](#0-2) 

The `dijkstraUtxowTransition` rule never adds `KeyHashObj` entries from `guardsTxBodyL` to the required witnesses. The only guard-related check in the UTXOW rule is:

```haskell
{- concatMapˡ (λ txSub → mapˢ proj₁ (TopLevelGuardsOf txSub)) (SubTransactionsOf txTop) ⊆ GuardsOf txTop -}
let requiredGuardsBySubTxs = foldMap (Map.keysSet . (^. bodyTxL . requiredTopLevelGuardsL)) subTxs
    topLevelGuards = OSet.toSet (txBody ^. guardsTxBodyL)
    missingGuards = requiredGuardsBySubTxs `Set.difference` topLevelGuards
runTestOnSignal $ failureOnNonEmptySet missingGuards MissingRequiredGuards
``` [4](#0-3) 

This check only verifies that guards required by sub-transactions are **declared** in the top-level `guards` set. It does not verify that any `KeyHashObj` guard credential has actually signed the transaction. The `getConwayWitsVKeyNeeded` function used by Dijkstra's `getWitsVKeyNeeded` does not include `guardsTxBodyL` entries: [5](#0-4) 

The existing test suite confirms this behavior — a UTxO locked by `RequireGuard guardKeyHash` is successfully spent by simply placing `guardKeyHash` in the `guards` set, with no signature from `guardKeyHash` provided:

```haskell
submitTx_ $ tx & bodyTxL . guardsTxBodyL .~ [guardKeyHash]
``` [6](#0-5) 

### Impact Explanation

**Critical.** A UTxO locked by a native script containing `RequireGuard (KeyHashObj kh)` can be spent by any transaction that includes `kh` in its `guards` field, without the spender possessing the private key for `kh`. The script author's intent — that only the holder of `kh` can authorize the script — is completely bypassed. This constitutes a direct, unauthorized loss of ADA or native assets through an invalid ledger state transition: the ledger accepts a spend that the script's authorization model prohibits.

### Likelihood Explanation

**High.** The attack requires no special privileges. Any transaction author can include an arbitrary key hash in the `guards` field of a transaction body. The `guards` field is a plain CBOR-encoded set with no cryptographic binding to the transaction's signers. Any user who knows the key hash embedded in a `RequireGuard` script (which is visible on-chain) can construct a valid-looking transaction that satisfies the script without the corresponding signing key.

### Recommendation

For `KeyHashObj` credentials in the `guards` set, the ledger must require a corresponding VKey witness, exactly as `reqSignerHashes` does. The `getWitsVKeyNeeded` implementation for the Dijkstra era (currently delegating to `getConwayWitsVKeyNeeded`) should be extended to include key hash credentials from `guardsTxBodyL`:

```haskell
getDijkstraWitsVKeyNeeded utxo txBody =
  getConwayWitsVKeyNeeded utxo txBody
    `Set.union` guardKeyHashWitnesses txBody

guardKeyHashWitnesses :: DijkstraEraTxBody era => TxBody l era -> Set (KeyHash Witness)
guardKeyHashWitnesses txBody =
  Set.fromList
    [ asWitness kh
    | KeyHashObj kh <- OSet.toList (txBody ^. guardsTxBodyL)
    ]
```

This mirrors the existing pattern for `reqSignerHashes`: [7](#0-6) 

### Proof of Concept

1. Alice creates a native script `s = RequireGuard (KeyHashObj kh_alice)` where `kh_alice` is her key hash. She locks a UTxO `u` with `s`, expecting only she can spend it.
2. Attacker Bob, who does not possess the private key for `kh_alice`, constructs a transaction:
   - `inputs = {u}`
   - `guards = {KeyHashObj kh_alice}` (self-declared, no signature)
   - `witnesses = {Bob's own signature}` (only needed for fee payment)
3. The ledger evaluates `RequireGuard (KeyHashObj kh_alice)` as `KeyHashObj kh_alice \`OSet.member\` {KeyHashObj kh_alice}` → `True`.
4. `validateNeededWitnesses` does not require `kh_alice` to be in the witness set (guards are not in `getWitsVKeyNeeded`).
5. The transaction is accepted. Bob has stolen Alice's UTxO without her key.

The test at line 38 of `UtxowSpec.hs` demonstrates step 3–5 passing without any signature from `guardKeyHash`: [6](#0-5)
