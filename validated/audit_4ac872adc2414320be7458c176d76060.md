Audit Report

## Title
Missing Staleness Check Enables Stale-Price Ratchet of `highestRsethPrice`, Triggering Spurious Auto-Pause ‚Äî (`contracts/oracles/ChainlinkPriceOracle.sol`, `contracts/LRTOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice` discards the `updatedAt` return value from `latestRoundData()`, accepting any price regardless of age. An attacker can use block stuffing to prevent Chainlink heartbeat updates, call the permissionless `updateRSETHPrice()` while a stale-high price is live, ratchet `highestRsethPrice` to an inflated value within the upside tolerance window, and then ‚Äî once the oracle corrects ‚Äî trigger the downside-protection auto-pause on the next legitimate price update, freezing all user deposits and withdrawals despite no real loss of TVL.

## Finding Description

**Root cause 1 ‚Äî No staleness check in `ChainlinkPriceOracle.getAssetPrice`:**

`getAssetPrice` calls `latestRoundData()` but silently discards `updatedAt` and `answeredInRound`, accepting any price regardless of age:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

**Root cause 2 ‚Äî `updateRSETHPrice` is permissionless:**

Any address can call it at any time while the contract is not paused: [2](#0-1) 

**Root cause 3 ‚Äî `highestRsethPrice` ratchets up monotonically:**

When `newRsETHPrice > highestRsethPrice`, the peak is updated with no validation that the underlying oracle data is fresh: [3](#0-2) 

**Root cause 4 ‚Äî Downside protection compares against the ratcheted peak:**

If the next computed price falls more than `pricePercentageLimit` below `highestRsethPrice`, the protocol auto-pauses: [4](#0-3) 

**Why the upside guard does not prevent the attack:**

Lines 252‚Äì267 revert non-manager callers when `newRsETHPrice` exceeds `highestRsethPrice` by more than `pricePercentageLimit`. However, this guard is only triggered when the stale-price increase is *above* the limit. The attacker only needs the stale price to be slightly above `highestRsethPrice` ‚Äî within the `pricePercentageLimit` band ‚Äî to successfully ratchet the peak without reverting. Once the peak is ratcheted even slightly, the true (lower) price can fall more than `pricePercentageLimit` below the new peak, triggering the downside pause. [5](#0-4) 

**Attack sequence:**

1. Attacker stuffs every block for the Chainlink heartbeat window, preventing the feed from updating. The feed retains a stale-high price.
2. Attacker calls `updateRSETHPrice()`. `_getTotalEthInProtocol()` reads the stale-high price, inflating `totalETHInProtocol` and therefore `newRsETHPrice`.
3. The stale-inflated `newRsETHPrice` is within `pricePercentageLimit` above `highestRsethPrice`, so the upside guard does not revert. Line 295 sets `highestRsethPrice` to the inflated value.
4. Block stuffing ends; Chainlink updates to the true (lower) price.
5. Any caller invokes `updateRSETHPrice()` again. `newRsETHPrice` now reflects the true TVL ‚Äî lower than the artificially ratcheted `highestRsethPrice`.
6. If the gap exceeds `pricePercentageLimit * highestRsethPrice`, lines 278‚Äì281 pause `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle`, freezing all user deposits and withdrawals.

## Impact Explanation

**Medium ‚Äî Temporary freezing of funds.** The auto-pause freezes deposits and withdrawals for all users until an admin manually unpauses. The `highestRsethPrice` ratchet is permanent and does not decay, so the inflated peak remains as the comparison baseline for all future price updates, permanently lowering the threshold at which a future legitimate price movement triggers another spurious pause. This constitutes temporary (and potentially repeated) freezing of user funds with no financial gain required by the attacker.

## Likelihood Explanation

Block stuffing on Ethereum mainnet is expensive but not infeasible for a motivated attacker targeting a high-TVL protocol. Chainlink ETH-denominated feeds typically have a 1-hour heartbeat and a 0.5‚Äì1% deviation threshold. The attacker does not need to stuff for the full heartbeat ‚Äî only long enough to lock in a stale price that is slightly above `highestRsethPrice` (within `pricePercentageLimit`). The permissionless `updateRSETHPrice` means no privileged role is required. The `SECURITY.md` explicitly notes that oracle manipulation attacks are not excluded from scope.

## Recommendation

1. **Add a staleness check** in `ChainlinkPriceOracle.getAssetPrice`: revert if `block.timestamp - updatedAt` exceeds a configured `maxStaleness` threshold per feed.

```solidity
(, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
if (block.timestamp - updatedAt > maxStaleness[asset]) revert StalePrice();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

2. **Guard `highestRsethPrice` updates** so the ratchet only occurs when oracle data is confirmed fresh (i.e., after the staleness check passes).

3. Consider requiring a **MANAGER role** to call `updateRSETHPrice` directly, or at minimum rate-limiting public calls to once per block or per heartbeat window.
