### Title
Incorrect `eraMaxLanguage` in `DijkstraEra` Excludes `PlutusV4` from Supported Language Range — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`)

### Summary

`DijkstraEra` introduces `PlutusV4` as a new Plutus language version and fully implements it (`DijkstraPlutusV4` constructor, `mkPlutusScript`, `withPlutusScript`, `mkSupportedLanguage` all handle `PlutusV4`), but the `eraMaxLanguage` boundary constant is left at `PlutusV3`. Every range-based check of the form `[minBound .. eraMaxLanguage @era]` silently excludes `PlutusV4`, causing incorrect behavior in auxiliary-data script reconstruction and a runtime panic in the `AlonzoTxAuxData` smart constructor.

### Finding Description

In `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Scripts.hs`, the `AlonzoEraScript DijkstraEra` instance declares four Plutus constructors (`DijkstraPlutusV1` through `DijkstraPlutusV4`) and handles all four in `mkPlutusScript` and `withPlutusScript`, yet sets:

```haskell
eraMaxLanguage = PlutusV3   -- line 446
``` [1](#0-0) 

`PlutusV4` is also fully wired into `EraPlutusContext DijkstraEra` — `mkSupportedLanguage` returns `Just` for it and `TxInfoResult DijkstraEra` carries a `PlutusV4` slot: [2](#0-1) 

The stale `eraMaxLanguage = PlutusV3` propagates into three production code sites:

**1. `getAlonzoTxAuxDataScripts` silently drops `PlutusV4` scripts**

```haskell
[ ... | lang <- [PlutusV1 .. eraMaxLanguage @era], ... ]
``` [3](#0-2) 

Because `PlutusV4 > eraMaxLanguage @DijkstraEra = PlutusV3`, any `PlutusV4` scripts stored in a transaction's auxiliary data are silently omitted when the script set is reconstructed from `AlonzoTxAuxData`.

**2. `AlonzoTxAuxData` smart-constructor panics on `PlutusV4` scripts**

```haskell
let unsupportedScripts =
      Map.filterWithKey (\lang _ -> lang > eraMaxLanguage @era) atadrPlutusScripts
in if Map.null unsupportedScripts
     then mkMemoizedEra @era ...
     else error $ prefix ++ " not supported in " ++ eraName @era
``` [4](#0-3) 

`PlutusV4 > PlutusV3`, so any code path that constructs `AlonzoTxAuxData` with `PlutusV4` scripts via this pattern synonym hits an unchecked `error` call.

**3. `supportedLanguages` excludes `PlutusV4`**

```haskell
supportedLanguages =
  let langs = [ errorFail (mkSupportedLanguageM lang)
              | lang <- [minBound .. eraMaxLanguage @era] ]
``` [5](#0-4) 

`PlutusV4` is absent from the resulting `NonEmpty (SupportedLanguage DijkstraEra)`, which is used for language-level validation and test generation.

The `Language` enum derives `Ord` and `Enum`, so `PlutusV4 > PlutusV3` is a concrete ordering fact: [6](#0-5) 

### Impact Explanation

**Medium — attacker-controlled serialized inputs cause incorrect ledger behavior outside design parameters.**

A transaction author submitting a `DijkstraEra` transaction that includes `PlutusV4` scripts in auxiliary data triggers two concrete misbehaviors:

- `getAlonzoTxAuxDataScripts` silently discards those scripts, so any downstream consumer of the reconstructed script set (e.g., auxiliary-data integrity checks, script-hash lookups) operates on an incomplete set. This is a silent data-loss path reachable from any submitted transaction.
- Any node code path that constructs `AlonzoTxAuxData` with `PlutusV4` scripts via the pattern synonym hits an unguarded `error`, which in Haskell throws an asynchronous exception. Depending on exception-handling discipline in the calling code, this can cause a node crash or inconsistent transaction rejection across nodes, matching the "deterministic disagreement" impact class.

### Likelihood Explanation

`DijkstraEra` is the era that introduces `PlutusV4`. Any user who legitimately attaches a `PlutusV4` script to auxiliary data in a `DijkstraEra` transaction will trigger the bug. No special privilege is required — only the ability to submit a transaction. The bug is latent until `DijkstraEra` activates on-chain.

### Recommendation

Update `eraMaxLanguage` for `DijkstraEra` to reflect the highest actually-supported language:

```diff
-  eraMaxLanguage = PlutusV3
+  eraMaxLanguage = PlutusV4
``` [7](#0-6) 

This mirrors the fix recommended in the ZkSync report (`CURRENT_MAX_PRECOMPILE_ADDRESS` updated to the highest new precompile address) and ensures that all range-based checks (`[minBound .. eraMaxLanguage @era]`, `lang > eraMaxLanguage @era`) correctly include `PlutusV4`.

### Proof of Concept

The inconsistency is directly observable from the source:

1. `mkPlutusScript` for `DijkstraEra` accepts `SPlutusV4` and returns `pure $ DijkstraPlutusV4 plutus` — `PlutusV4` is a supported script type.
2. `mkSupportedLanguage` for `DijkstraEra` returns `Just $ SupportedLanguage SPlutusV4` — `PlutusV4` is a supported execution language.
3. `eraMaxLanguage = PlutusV3` — the boundary constant is one version behind.
4. `getAlonzoTxAuxDataScripts` iterates `[PlutusV1 .. PlutusV3]`, omitting `PlutusV4`.
5. `AlonzoTxAuxData` pattern evaluates `PlutusV4 > PlutusV3 = True`, classifying `PlutusV4` as unsupported and calling `error`. [8](#0-7) [2](#0-1) [9](#0-8) [10](#0-9)
