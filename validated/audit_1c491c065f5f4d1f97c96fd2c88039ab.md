### Title
Sub-Transaction Plutus Script Execution Units Excluded from Fee and ExUnits Limit Checks - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may embed multiple sub-transactions (`subTransactionsTxBodyL`), each carrying its own witness set including Plutus redeemers. The per-transaction ExUnits limit check (`validateExUnitsTooBigUTxO`) and the minimum-fee script-execution component (`txscriptfee`) both operate exclusively on the top-level transaction's redeemer map. Sub-transaction redeemers are never summed into either check. An attacker can therefore submit a single on-chain transaction whose top-level ExUnits budget is zero (or minimal), while embedding arbitrarily large Plutus ExUnits budgets across its sub-transactions, bypassing the `maxTxExUnits` protocol parameter and paying no script-execution fee for those sub-transactions.

---

### Finding Description

**Sub-transaction structure.** The Dijkstra era introduces a `DijkstraSubTxBodyRaw SubTx era` type that has no fee field; required fields are only `inputs` and `outputs`. Sub-transactions carry a full `transaction_witness_set` including redeemers (field 5 in the CDDL spec). [1](#0-0) [2](#0-1) 

**ExUnits limit check is top-level only.** In `dijkstraUtxoTransition`, the ExUnits check calls `Alonzo.validateExUnitsTooBigUTxO pp tx`, where `tx` is the top-level transaction: [3](#0-2) 

`validateExUnitsTooBigUTxO` calls `totExUnits tx`, which folds only over the top-level transaction's own redeemer map: [4](#0-3) 

Sub-transactions are stored in the tx body (`dtbrSubTransactions`), not in the top-level witness set. Their redeemers are therefore never included in `totExUnits`.

**Minimum fee excludes sub-transaction script costs.** The Dijkstra minimum fee is computed as: [5](#0-4) 

`alonzoMinFeeTx pp tx` uses `totExUnits tx` for the `txscriptfee` component: [6](#0-5) 

Because `totExUnits tx` only covers the top-level redeemers, sub-transaction Plutus execution costs contribute zero to the minimum fee.

**Sub-transaction scripts are executed.** `validateBatchCollateral` explicitly checks whether any sub-transaction carries redeemers and, if so, enforces collateral on the top-level transaction. This confirms sub-transaction Plutus scripts are evaluated by the ledger: [7](#0-6) 

**SUBUTXO rule has no fee or ExUnits check.** The `dijkstraSubUtxoTransition` performs no `validateFeeTooSmallUTxO` and no `validateExUnitsTooBigUTxO`. It calls `updateUTxOStateNoFees`, explicitly bypassing fee accounting: [8](#0-7) [9](#0-8) 

**Block-level ExUnits check is also blind to sub-transactions.** The BBODY-level `validateExUnits` also uses `foldMap totExUnits txs`, so the block-level cap is equally bypassed: [10](#0-9) 

---

### Impact Explanation

An attacker submits a top-level Dijkstra transaction with zero or minimal top-level redeemers (staying well under `maxTxExUnits`) and embeds N sub-transactions each declaring a Plutus ExUnits budget approaching `maxTxExUnits`. The ledger:

1. Accepts the transaction because the top-level ExUnits check passes.
2. Charges a fee that does not include sub-transaction script execution costs.
3. Executes all sub-transaction Plutus scripts, consuming up to N × `maxTxExUnits` of CPU/memory per block slot.

This allows attacker-controlled transactions to exceed the intended per-transaction and per-block validation resource limits, and to pay fees outside the design parameters (script execution fee component is zero for sub-transaction scripts). This matches the **Medium** allowed impact: *"Attacker-controlled transactions... exceed intended validation limits or modify fees... outside design parameters."*

---

### Likelihood Explanation

The Dijkstra era is implemented in the production codebase and is the next scheduled era after Conway. Any unprivileged transaction sender can craft such a transaction. No special access, governance majority, or key compromise is required. The attacker only needs to construct a valid top-level transaction body with embedded sub-transactions carrying large redeemer ExUnits budgets. The collateral requirement (enforced by `validateBatchCollateral`) is the only cost barrier, but collateral is returned on success, so the net cost to the attacker is only the top-level transaction fee — which does not include sub-transaction script execution costs.

---

### Recommendation

1. **Extend `totExUnits` (or introduce a new function) to recursively sum ExUnits across all sub-transactions** and use this aggregate in both `validateExUnitsTooBigUTxO` and `alonzoMinFeeTx` for Dijkstra-era transactions.
2. **Update the minimum fee calculation** (`getConwayMinFeeTx` / `getConwayMinFeeTxUtxo`) to include sub-transaction script execution costs in the `txscriptfee` component.
3. **Update the BBODY-level `validateExUnits`** to use the same aggregate ExUnits function so the block-level cap is also enforced correctly.

---

### Proof of Concept

```
Top-level tx:
  inputs:  [utxo_A]
  outputs: [change_output]
  fee:     <minimum fee based on tx size + 0 script fee>
  witnesses: { redeemers: {} }   -- zero top-level ExUnits
  sub_transactions:
    subTx_1:
      inputs: [utxo_B]
      outputs: [...]
      witnesses: { redeemers: { spend_0: (data, ExUnits { mem=maxMem, steps=maxSteps }) } }
    subTx_2: ... (same pattern, up to block limit)
    ...
    subTx_N: ...
```

The top-level `validateExUnitsTooBigUTxO` check sees `totExUnits tx = ExUnits 0 0 ≤ maxTxExUnits` — passes. The fee check sees `txscriptfee prices (ExUnits 0 0) = 0` — passes. The ledger then executes all N sub-transaction Plutus scripts, each consuming up to `maxTxExUnits` of resources, for a total of N × `maxTxExUnits` per transaction, at a fee that covers none of that execution cost. dijkstraSubUtxoTransition = do
  TRC (SubUtxoEnv slot pp certState originalUtxo (IsValid isValid), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG

  let txBody = tx ^. bodyTxL

  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo
  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  let allSizedOutputs = txBody ^. allSizedOutputsTxBodyF
  let allOutputs = fmap sizedValue allSizedOutputs
  runTest $ Alonzo.validateOutputTooBigUTxO pp allOutputs

  runTest $ Shelley.validateInputSetEmptyUTxO txBody

  let inputs = txBody ^. inputsTxBodyL
  let refInputs = txBody ^. referenceInputsTxBodyL
  runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxoState) inputs

  runTestOnSignal $ Shelley.validateOutputBootAddrAttrsTooBig allOutputs

  runTestOnSignal $ Babbage.validateOutputTooSmallUTxO pp allSizedOutputs

  netId <- liftSTS $ asks networkId
  runTestOnSignal $ Shelley.validateWrongNetwork netId allOutputs

  runTestOnSignal $ validateWrongNetworkInDirectDeposit netId txBody
  runTestOnSignal $ Alonzo.validateWrongNetworkInTxBody netId txBody

  if isValid
    then do

        Shelley.updateUTxOStateNoFees
          pp

          certState

-- | Like 'updateUTxOState', but does not collect fees. This is used for sub-transactions
-- where fees are not applicable.
updateUTxOStateNoFees ::
  ( EraTxBody era
  , EraStake era
  , EraCertState era
  , Monad m
  ) =>
  PParams era ->
  UTxOState era ->
  TxBody l era ->
  CertState era ->
  GovState era ->
  (Coin -> m ()) ->
  (UTxO era -> UTxO era -> m ()) ->
  m (UTxOState era)
updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  let UTxOState {utxosUtxo, utxosDeposited, utxosFees, utxosDonation} = utxos
      UTxO utxo = utxosUtxo
      !utxoAdd = txouts txBody -- These will be inserted into the UTxO
      {- utxoDel  = txins txb ◁ utxo -}
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      {- newUTxO = (txins txb ⋪ utxo) ∪ outs txb -}
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
      deletedUTxO = UTxO utxoDel
      totalRefunds = certsTotalRefundsTxBody pp certState txBody
      totalDeposits = certsTotalDepositsTxBody pp certState txBody
      depositChange = totalDeposits <-> totalRefunds
  depositChangeEvent depositChange
  txUtxODiffEvent deletedUTxO utxoAdd
  pure $!
    UTxOState
      { utxosUtxo = UTxO newUTxO
      , utxosDeposited = utxosDeposited <> depositChange
      , utxosFees = utxosFees
      , utxosGovState = govState
      , utxosInstantStake =
          deleteInstantStake deletedUTxO (addInstantStake utxoAdd (utxos ^. instantStakeL))
      , utxosDonation = utxosDonation
      }
