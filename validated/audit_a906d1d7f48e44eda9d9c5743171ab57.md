### Title
Unvalidated `builder` Field in `ProposalInit` Allows Proposer to Commit Arbitrary Sequencer Address into Block Hash and Redirect Transaction Fees — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary
The `is_proposal_init_valid` function validates many fields of the network-received `ProposalInit` against locally-derived reference values, but it never checks the `builder` field. The `builder` field is passed directly to `convert_to_sn_api_block_info` as `sequencer_address` in `BlockInfo`, which is then used in transaction execution (fee collection) and block hash computation. A malicious proposer can set `builder` to any arbitrary `ContractAddress`, and the validator will accept the proposal, execute transactions with the wrong sequencer address, and commit a block whose hash encodes that wrong address.

### Finding Description

`is_proposal_init_valid` enforces the following fields from `ProposalInit`:

- `height`, `l1_da_mode`, `l2_gas_price_fri` (exact match)
- `starknet_version`, `version_constant_commitment` (exact match)
- `l1_gas_price_fri/wei`, `l1_data_gas_price_fri/wei` (within margin)
- `fee_proposal_fri` (within bounds of `fee_actual`)
- `timestamp` (range check only) [1](#0-0) 

The `builder` field is **never checked** against any locally-known expected value. After `is_proposal_init_valid` returns `Ok`, `initiate_validation` calls `convert_to_sn_api_block_info(init)`, which maps `init.builder` directly to `sequencer_address`: [2](#0-1) 

This `BlockInfo` (with the proposer-supplied `builder` as `sequencer_address`) is forwarded to the batcher for block execution: [3](#0-2) 

The `sequencer_address` flows into `PartialBlockHashComponents.sequencer`, which is hashed into the `PartialBlockHash` and ultimately into the `ProposalCommitment`: [4](#0-3) [5](#0-4) 

Because both the proposer and validator use the same proposer-supplied `builder` value, the `ProposalCommitment` comparison at line 244 passes: [6](#0-5) 

The block is then committed with the arbitrary `sequencer_address` baked into the block hash and the execution context. The same unvalidated `builder` is also forwarded to the cende pipeline and state sync: [7](#0-6) 

### Impact Explanation

The `sequencer_address` in `BlockInfo` is the address to which transaction fees are paid during execution. By setting `builder` to an arbitrary address, a malicious proposer can:

1. **Redirect all transaction fees** from the block to any address they choose, including one they control or one that is unspendable (burning fees).
2. **Corrupt the block hash**: `sequencer_address` is a direct input to `calculate_block_hash`. The committed block hash will encode the attacker-chosen address, diverging from what any honest observer would compute from the expected sequencer address.
3. **Corrupt `get_execution_info` syscall results**: Contracts that read the sequencer address during execution will observe the attacker-supplied value, potentially breaking fee-related or access-control logic in contracts that rely on it.

This matches: *Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.*

### Likelihood Explanation

Triggering this requires being the designated proposer for a consensus round, which is a normal operational role for any staked validator — not a special administrative privilege. In a multi-validator deployment, any validator will eventually be selected as proposer. The attack requires no external oracle manipulation, no race condition, and no special tooling beyond modifying the `builder_address` field in the outgoing `ProposalInit` message before it is streamed to peers.

### Recommendation

Add a check for `builder` in `is_proposal_init_valid` (or in `ProposalInitValidation`) that compares `init_proposed.builder` against the locally-configured expected sequencer address for the current height. Concretely:

1. Add an `expected_builder: ContractAddress` field to `ProposalInitValidation`.
2. Populate it from the node's own `builder_address` configuration when constructing `ProposalInitValidation` in `validate_proposal` and `validate_current_round_proposal`.
3. In `is_proposal_init_valid`, reject any `init_proposed.builder != expected_builder`.

This mirrors the pattern already used for `height`, `l1_da_mode`, and `l2_gas_price_fri`.

### Proof of Concept

1. A validator node is selected as proposer for round R at height H.
2. The node modifies `initiate_build` to set `builder: ContractAddress::from(ATTACKER_ADDRESS)` instead of `args.builder_address`.
3. The modified `ProposalInit` is streamed to all validators.
4. Each validator calls `is_proposal_init_valid`: all checked fields pass; `builder` is not checked.
5. `initiate_validation` calls `convert_to_sn_api_block_info(&init)`, producing `BlockInfo { sequencer_address: ATTACKER_ADDRESS, ... }`.
6. The batcher executes all transactions with `sequencer_address = ATTACKER_ADDRESS`; all fees are credited to `ATTACKER_ADDRESS`.
7. `PartialBlockHashComponents::new(&block_info, ...)` records `sequencer: ATTACKER_ADDRESS`.
8. Both proposer and validator compute the same `PartialBlockHash` (both used `ATTACKER_ADDRESS`), so `built_block == received_fin.proposal_commitment` passes.
9. The block is committed. The on-chain block hash encodes `ATTACKER_ADDRESS` as sequencer, and all fees from the block reside at `ATTACKER_ADDRESS`.
