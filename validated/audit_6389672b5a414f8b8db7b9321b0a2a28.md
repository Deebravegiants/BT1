Audit Report

## Title
Griefing DOS on `removeLiquidity` via Unrestricted `addLiquidity` Owner Parameter and `MinimalLiquidity` Dust Guard — (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`addLiquidity` allows any `msg.sender` to credit shares to an arbitrary `owner` address with no caller restriction. An attacker can frontrun a victim's `removeLiquidity` call by depositing a minimal share amount into the victim's position, leaving the victim's remaining balance in the forbidden dust range `(0, minimalMintableLiquidity)`, causing a permanent `MinimalLiquidity` revert. The attacker can repeat this indefinitely at near-zero cost, permanently blocking the victim from withdrawing their LP assets.

## Finding Description
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` parameter with no restriction on the caller identity:

```solidity
// MetricOmmPool.sol L182-196
function addLiquidity(
  address owner, // any caller can specify any owner
  ...
) external nonReentrant(PoolActions.ADD_LIQUIDITY) ...
```

`removeLiquidity` enforces `msg.sender == owner` (L206) but has a dust guard in `LiquidityLib.removeLiquidity` (L200-202):

```solidity
uint256 newUserShares = userShares - sharesToRemove;
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
  revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

The `addLiquidity` path only checks that the *resulting* position is `>= minimalMintableLiquidity` (LiquidityLib.sol L76-79):

```solidity
uint256 newUserShares = userShares + sharesToAdd;
if (newUserShares < ctx.minimalMintableLiquidity) {
  revert IMetricOmmPoolActions.MinimalLiquidity(...);
}
```

Because the victim's existing position already satisfies the floor, an attacker can add as few as 1 share to the victim's position (since `victimShares + 1 >= minimalMintableLiquidity` is trivially true for any valid position). After the frontrun, the victim's attempt to remove their original share count leaves exactly 1 share remaining — which is `> 0` and `< minimalMintableLiquidity` — triggering the revert.

The `DepositAllowlistExtension` does not mitigate this: it checks `allowedDepositor[msg.sender][owner]` where `msg.sender` is the pool and `owner` is the position owner (DepositAllowlistExtension.sol L38). The actual caller (`msg.sender` of `addLiquidity`) is never checked, so any attacker can add to any allowlisted victim's position.

The victim cannot escape by submitting a "remove all" transaction because the interface requires specifying exact share amounts. The attacker can always frontrun with another 1-share deposit, ensuring the victim's specified `sharesToRemove` always leaves exactly 1 share remaining.

## Impact Explanation
The victim's LP principal is permanently locked in the pool — they cannot call `removeLiquidity` successfully as long as the attacker continues to frontrun. This constitutes broken core pool functionality causing loss of access to LP assets, matching the allowed impact criterion of "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows." Severity is High.

## Likelihood Explanation
The attack requires only a standard `addLiquidity` call with `owner = victim` from any unprivileged address that implements `IMetricOmmModifyLiquidityCallback`. No special role or privileged access is needed. Frontrunning is straightforward on any public mempool chain. The cost per frontrun is a dust token amount proportional to `initialScaledToken*PerShareE18` for 1 share. The attack is repeatable indefinitely, making it a permanent griefing vector.

## Recommendation
Two independent fixes, either sufficient:
1. **Symmetric caller restriction**: Require `msg.sender == owner` in `addLiquidity` (mirroring the existing check in `removeLiquidity`), or implement an on-chain approval mapping (similar to ERC-20 `allowance`) so only addresses explicitly authorized by `owner` can add to their position.
2. **Skip dust guard on full exit**: In `LiquidityLib.removeLiquidity`, skip the `MinimalLiquidity` check when `newUserShares == 0` (already done) — but also allow the check to be bypassed when the caller is removing their *entire current position* (i.e., `sharesToRemove == userShares`). The dust guard is only meaningful for partial removals; a full exit to zero is always safe.

## Proof of Concept
```
Setup:
  - Pool with minimalMintableLiquidity = 1000
  - Victim holds 10_000 shares in bin 4 (salt = 1)

Step 1 (victim submits):
  pool.removeLiquidity(victim, salt=1, [{bin:4, shares:10_000}], "")
  → newUserShares = 0 → passes MinimalLiquidity check → would succeed

Step 2 (attacker frontruns, same block):
  pool.addLiquidity(owner=victim, salt=1, [{bin:4, shares:1}], callbackData, "")
  → newUserShares = 10_001 >= 1000 → passes add check
  → attacker pays ~1 share worth of tokens via callback

Step 3 (victim's original tx executes):
  removeLiquidity removes 10_000 shares
  → newUserShares = 10_001 - 10_000 = 1
  → 1 > 0 && 1 < 1000 → revert MinimalLiquidity(1, 1000)

Step 4 (victim re-queries balance = 10_001, resubmits remove 10_001):
  Attacker frontruns again with 1 share → victim has 10_002
  → remove 10_001 → newUserShares = 1 → revert again

Victim's funds remain locked indefinitely.
```
