Audit Report

## Title
Division Rounding to Zero Allows ETH Deposits to Yield Zero rsETH — (File: `contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`)

## Summary

Every L2 pool deposit function computes rsETH output via integer division `amountAfterFee * 1e18 / rsETHToETHrate`. When `amountAfterFee` is 1 wei and `rsETHToETHrate` is any value above `1e18` (which is always the case post-rewards), the division truncates to zero. The pool accepts the ETH but transfers or mints zero rsETH to the caller, with no guard against a zero output. The depositor's ETH is permanently absorbed by the pool.

## Finding Description

All five pool contracts share the same `viewSwapRsETHAmountAndFee` pattern:

```solidity
// RSETHPool.sol:319, RSETHPoolV3.sol:307, RSETHPoolNoWrapper.sol:285,
// RSETHPoolV3ExternalBridge.sol:426, RSETHPoolV3WithNativeChainBridge.sol:343
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

`rsETHToETHrate` is always `>= 1e18` (e.g., `1.05e18`). For `amountAfterFee = 1 wei`: `1 * 1e18 / 1.05e18 = 0` (Solidity integer truncation). The deposit functions only guard against `amount == 0`:

```solidity
// RSETHPool.sol:269, RSETHPoolV3.sol:256, RSETHPoolNoWrapper.sol:235,
// RSETHPoolV3ExternalBridge.sol:375, RSETHPoolV3WithNativeChainBridge.sol:292
if (amount == 0) revert InvalidAmount();
```

No guard against `rsETHAmount == 0` exists in any of these contracts. After computing `rsETHAmount = 0`, each contract proceeds to transfer or mint zero tokens:

- `RSETHPool.sol:275`: `IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount)` — transfers 0
- `RSETHPoolV3.sol:262`: `wrsETH.mint(msg.sender, rsETHAmount)` — mints 0
- `RSETHPoolNoWrapper.sol:241`: `rsETH.safeTransfer(msg.sender, rsETHAmount)` — transfers 0
- `RSETHPoolV3ExternalBridge.sol:381`: `wrsETH.mint(msg.sender, rsETHAmount)` — mints 0
- `RSETHPoolV3WithNativeChainBridge.sol:298`: `wrsETH.mint(msg.sender, rsETHAmount)` — mints 0

The token-path variant (`amountAfterFee * tokenToETHRate / rsETHToETHrate`) is also affected when `tokenToETHRate` is small relative to `rsETHToETHrate`, widening the zero-output range.

`LRTDepositPool` has the same root cause at `LRTDepositPool.sol:520` but is partially mitigated by the caller-controlled `minRSETHAmountExpected` slippage parameter checked at line 667.

## Impact Explanation

A depositor who sends 1 wei of ETH to any L2 pool passes the `amount == 0` guard, receives `rsETHAmount = 0` from `viewSwapRsETHAmountAndFee`, has their ETH accepted by the pool, and receives zero rsETH/wrsETH in return. The ETH is not returned; it remains in the pool and will eventually be bridged to L1. The depositor permanently loses their deposit without receiving the promised rsETH. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation

The condition is reachable by any unprivileged caller with no preconditions beyond the pool being unpaused and ETH deposits being enabled. The rsETH rate is always above `1e18` (it only increases as staking rewards accrue), so the 1-wei threshold is permanently active. A user could trigger this accidentally (e.g., a script sending a dust amount) or a griefing actor could trigger it repeatedly to pollute pool accounting. No special role, oracle manipulation, or external dependency is required.

## Recommendation

Add a zero-output guard in every deposit function (or in `viewSwapRsETHAmountAndFee` itself) across all five pool contracts:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

Apply the same guard to the token-path deposit variants. For `LRTDepositPool`, the existing `minRSETHAmountExpected` check in `_beforeDeposit` already provides a caller-controlled mitigation, but an explicit zero-output revert in `_beforeDeposit` would make the protection unconditional.

## Proof of Concept

Assume `rsETHToETHrate = 1.05e18` (realistic post-reward rate), `feeBps = 10` (0.1%):

```
amount         = 1 wei (msg.value)
fee            = 1 * 10 / 10_000 = 0   (rounds to zero)
amountAfterFee = 1 - 0 = 1 wei

rsETHAmount    = 1 * 1e18 / 1_050_000_000_000_000_000
               = 1_000_000_000_000_000_000 / 1_050_000_000_000_000_000
               = 0   (integer division truncates)
```

**Foundry test plan:**

```solidity
function test_depositOneWeiYieldsZeroRsETH() public {
    // Fork mainnet or use mock oracle returning 1.05e18
    uint256 balanceBefore = wrsETH.balanceOf(alice);
    vm.prank(alice);
    pool.deposit{value: 1}("ref");
    uint256 balanceAfter = wrsETH.balanceOf(alice);
    assertEq(balanceAfter - balanceBefore, 0); // alice received 0 rsETH
    assertGt(address(pool).balance, 0);        // pool kept the 1 wei
}
```

This test will pass against the current code, confirming the pool accepts 1 wei of ETH and issues 0 rsETH to the caller.
