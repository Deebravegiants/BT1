### Title
Missing DRep Expiry Check in Delegation Validation Allows Withdrawal Bypass via Expired DRep — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs`)

---

### Summary

The Conway `DELEG` rule's `checkDRepRegistered` predicate only verifies that a target DRep credential exists in the `vsDRepsL` map (i.e., is registered), but never checks whether that DRep has expired. An expired DRep remains in the map until it is reaped at an epoch boundary, so `Map.member` returns `True` for it. This allows any unprivileged transaction sender to delegate a stake credential to an already-expired DRep, and then immediately withdraw staking rewards — bypassing the Conway governance-participation requirement that mandates delegation to an active DRep before rewards can be claimed.

---

### Finding Description

In `conwayDelegTransition`, the helper `checkStakeDelegateeRegistered` calls `checkDRepRegistered`:

```haskell
checkDRepRegistered = \case
  DRepAlwaysAbstain -> pure ()
  DRepAlwaysNoConfidence -> pure ()
  DRepCredential targetDRep -> do
    let dReps = certState ^. certVStateL . vsDRepsL
    unless (hardforkConwayBootstrapPhase (pp ^. ppProtocolVersionL)) $
      targetDRep `Map.member` dReps ?! injectFailure (DelegateeDRepNotRegisteredDELEG targetDRep)
``` [1](#0-0) 

The check is purely a membership test. An expired DRep is **not** removed from `vsDRepsL` at the time of expiry; it is only reaped at the next epoch boundary. Therefore `Map.member` returns `True` for an expired DRep, and the delegation certificate is accepted without error.

The downstream withdrawal validator `validateWithdrawalsDelegated` in the `LEDGER` rule then only checks that `dRepDelegationAccountStateL` is not `Nothing`:

```haskell
isNotDRepDelegated keyHash = isNothing $ do
  accountState <- lookupAccountState (KeyHashObj keyHash) accounts
  accountState ^. dRepDelegationAccountStateL
``` [2](#0-1) 

Because the delegation to the expired DRep was accepted, this field is populated, and `ConwayWdrlNotDelegatedToDRep` is never raised. The reward withdrawal succeeds.

The existing test suite explicitly demonstrates this end-to-end path as a passing (non-failing) scenario:

```
it "Withdraw from a key delegated to an expired DRep" $ do
  ...
  isDRepExpired drep `shouldReturn` True
  _ <- delegateToDRep cred (Coin 1_000_000) (DRepCredential drep)
  submitTx_ $ mkBasicTx $ mkBasicTxBody & withdrawalsTxBodyL .~ Withdrawals [(ra, balance)]
``` [3](#0-2) 

By contrast, delegation to an **unregistered** DRep is correctly rejected with `DelegateeDRepNotRegisteredDELEG`, and withdrawal without any DRep delegation is rejected with `ConwayWdrlNotDelegatedToDRep`. The gap is specifically the missing expiry check.

---

### Impact Explanation

The Conway governance model requires that a stake credential be delegated to an active DRep before its owner can withdraw staking rewards. This is enforced by `validateWithdrawalsDelegated`. [4](#0-3) 

Because an expired DRep's votes are treated as abstain in ratification (`committeeAcceptedRatio` discounts expired members; the analogous DRep expiry logic discounts expired DRep stake from the active voting distribution), delegating to an expired DRep provides zero real governance participation. Yet the withdrawal gate is satisfied. An attacker can therefore withdraw accumulated rewards without contributing to governance, modifying reward withdrawals outside the design parameters of the governance-participation gate.

**Allowed impact matched**: *Medium — Attacker-controlled transactions modify withdrawals outside design parameters.*

---

### Likelihood Explanation

The attack requires no privileged access, no key compromise, and no coordination. Any stake credential holder can:
1. Identify an expired DRep (expiry is public ledger state).
2. Submit a `DelegVote` or `RegDepositDelegTxCert` certificate targeting that DRep.
3. Immediately submit a withdrawal transaction.

The only prerequisite is that a DRep has expired and not yet been reaped (i.e., within the same epoch as expiry or before the next epoch boundary). Given that DRep expiry is a normal, recurring event, this window is reliably available.

---

### Recommendation

Add an expiry check inside `checkDRepRegistered` after confirming registration. The `isDRepExpired` helper already exists in the test library and the underlying computation (`vsActualDRepExpiry`) is available in production state:

```haskell
checkDRepRegistered = \case
  DRepAlwaysAbstain -> pure ()
  DRepAlwaysNoConfidence -> pure ()
  DRepCredential targetDRep -> do
    let dReps = certState ^. certVStateL . vsDRepsL
    unless (hardforkConwayBootstrapPhase (pp ^. ppProtocolVersionL)) $ do
      targetDRep `Map.member` dReps
        ?! injectFailure (DelegateeDRepNotRegisteredDELEG targetDRep)
      -- Add: reject delegation to an expired DRep
      -- forM_ (Map.lookup targetDRep dReps) $ \drepState ->
      --   not (actuallyExpired drepState vState currentEpoch)
      --     ?! injectFailure (DelegateeDRepExpiredDELEG targetDRep)
``` [1](#0-0) 

Alternatively, `validateWithdrawalsDelegated` could be strengthened to additionally verify that the delegated DRep is not expired at the time of withdrawal. [4](#0-3)

### Proof of Concept

The existing test at `LedgerSpec.hs:148–173` already demonstrates the full attack path without any modification:

1. Set `ppDRepActivityL = EpochInterval 1` so a DRep expires quickly.
2. Register a DRep and let it expire (`isDRepExpired drep` returns `True`).
3. Submit a `DelegVote` certificate targeting the expired DRep — **accepted, no failure**.
4. Submit a withdrawal transaction — **accepted, no failure**, rewards drained. [3](#0-2)

isDRepExpired ::
  (HasCallStack, ConwayEraCertState era) =>
  Credential DRepRole ->
  ImpTestM era Bool
isDRepExpired drep = do
  vState <- getsNES $ nesEsL . esLStateL . lsCertStateL . certVStateL
  currentEpoch <- getsNES nesELL
  case Map.lookup drep $ vState ^. vsDRepsL of
    Nothing -> error $ unlines ["DRep not found", show drep]
    Just drep' ->
      pure $
        binOpEpochNo (+) (vState ^. vsNumDormantEpochsL) (drep' ^. drepExpiryL)
          < currentEpoch
