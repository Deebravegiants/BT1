### Title
Burn-on-Transfer Token Inflates Cross-Chain Credit, Breaking Backing Guarantee — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`initTransfer` records and broadcasts the caller-supplied `amount` rather than the actual tokens received by the bridge. For a burn-on-transfer ERC20, the bridge locks fewer tokens than it announces cross-chain, causing the destination chain to mint or release more wrapped assets than are backed, permanently breaking the 1:1 collateral guarantee.

### Finding Description
In `OmniBridge.initTransfer`, when the token is a plain ERC20 (neither a bridge token nor a custom-minter token), the contract pulls tokens from the caller:

```solidity
// OmniBridge.sol lines 406-412
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // nominal amount, not verified against actual receipt
);
```

No balance snapshot is taken before or after the transfer. The function then passes the caller-supplied `amount` directly to `initTransferExtension` and to the `InitTransfer` event:

```solidity
// OmniBridge.sol lines 415-436
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message, extensionValue
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
```

In `OmniBridgeWormhole.initTransferExtension`, this nominal `amount` is Borsh-encoded into the Wormhole message that is relayed cross-chain:

```solidity
// OmniBridgeWormhole.sol lines 135-136
Borsh.encodeUint128(amount),
```

For a burn-on-transfer token, `safeTransferFrom` delivers only `amount - burn_fee` to the bridge, but the cross-chain message carries the full `amount`. The NEAR side (or any destination chain) then mints or releases `amount` wrapped tokens to the recipient, while the EVM bridge holds only `amount - burn_fee` as collateral.

Any ERC20 token can be used in the `else` branch — there is no whitelist check. `logMetadata` is also permissionless, so an attacker can register any token on NEAR before initiating the transfer.

### Impact Explanation
Every `initTransfer` call with a burn-on-transfer token inflates the wrapped supply on the destination chain relative to the locked collateral on the source chain. Over repeated calls the deficit compounds. When legitimate holders attempt to redeem wrapped tokens back to EVM, `finTransfer` will attempt `safeTransfer` of the full recorded amount but the bridge vault is short, causing redemptions to fail for the last redeemers — a permanent, irreversible fund lock for those users. This matches **Critical: irreversible fund lock / permanently unclaimable user value** and **High: balance-accounting divergence that breaks backing guarantees**.

### Likelihood Explanation
The entry path is fully unprivileged: `initTransfer` is a public function with no token whitelist in the `else` branch, and `logMetadata` is permissionless. A single attacker can deploy or use an existing burn-on-transfer ERC20, register it via `logMetadata`, and call `initTransfer` repeatedly. No privileged role, leaked key, or external compromise is required.

### Recommendation
Measure the actual amount received by comparing the contract's token balance before and after the `safeTransferFrom`, and use that delta — not the caller-supplied `amount` — for all downstream accounting, event emission, and cross-chain message encoding:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived in place of amount going forward
```

If burn-on-transfer tokens are explicitly out of scope for the protocol, add a documented invariant check (e.g., `require(actualReceived == amount, "fee-on-transfer not supported")`) to fail loudly rather than silently under-collateralizing the bridge.

### Proof of Concept
1. Attacker deploys `BurnToken` — an ERC20 that burns 10% on every transfer.
2. Attacker calls `OmniBridge.logMetadata(burnTokenAddress)` — permissionless, registers the token on NEAR.
3. Attacker calls `OmniBridge.initTransfer(burnTokenAddress, 1000, 0, 0, "attacker.near", "")`.
4. `safeTransferFrom` moves 1000 tokens from attacker; bridge receives only 900 (100 burned).
5. `InitTransfer` event and Wormhole message both carry `amount = 1000`.
6. NEAR relayer processes the message and mints 1000 wrapped `BurnToken` to `attacker.near`.
7. Attacker now holds 1000 wrapped tokens backed by only 900 real tokens — a 100-token unbacked surplus per call.
8. Repeated calls drain the backing ratio; the last legitimate redeemers find the EVM bridge vault insufficient and their `finTransfer` calls revert permanently.
