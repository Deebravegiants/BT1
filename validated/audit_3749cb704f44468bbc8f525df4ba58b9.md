### Title
Single Byzantine Participant Bypasses Threshold Authorization to Deliver Arbitrary CKD Output via `AppPublicKey` Variant Logic Edge Case - (File: `crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, a logic branch on the user-controlled `app_public_key` variant causes the cryptographic output check to be **completely skipped** when the request uses `CKDAppPublicKey::AppPublicKey`. A single attested MPC participant (below threshold) can call `respond_ckd` with any arbitrary `CKDResponse` and the contract will accept and deliver it to the user without verifying that the threshold of participants actually authorized the derivation.

---

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` at lines 675–682 contains the following match:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKeyPV` arm calls `ckd_output_check`, which cryptographically verifies that the response `(big_y, big_c)` is a valid encryption of the MPC-derived key under the user's public key. The `AppPublicKey` arm is **empty** — no such check exists.

The `app_public_key` variant is set by the user at request time via `request_app_private_key`. When a user submits a request with `CKDAppPublicKey::AppPublicKey(...)`, the variant is stored in the pending `CKDRequest` and echoed back in the `respond_ckd` call. Because the contract only checks that the caller is an attested participant:

```rust
self.assert_caller_is_attested_participant_and_protocol_active();
``` [2](#0-1) 

...but performs **zero cryptographic verification** on the response body for the `AppPublicKey` variant, a single Byzantine attested participant can supply arbitrary `big_y` / `big_c` values and the contract will resolve the yield and deliver the fake output to the user.

This is structurally identical to the Trading.sol bug: a user-controlled boolean (`usePermit = false` / `AppPublicKey` variant) causes the authorization check to be skipped, and execution proceeds to the consequential action (minting tokens / delivering CKD output) without proper authorization.

---

### Impact Explanation

A single attested MPC participant (strictly below the signing threshold) can:

1. Wait for any pending `CKDRequest` that uses the `AppPublicKey` variant.
2. Call `respond_ckd` with fabricated `CKDResponse { big_y: [garbage], big_c: [garbage] }`.
3. The contract accepts the call (attested-participant check passes, no output check for this variant), resolves the yield, and returns the fake response to the user.

The user receives a cryptographically invalid CKD output — a garbage encrypted key. Any funds sent to addresses derived from this fake key are permanently unspendable. This constitutes **confidential key derivation output delivered without the required threshold-participant authorization**, matching the Critical impact tier: *"Unauthorized … confidential key derivation output without the required participant authorization."*

---

### Likelihood Explanation

- The `AppPublicKey` variant is the simpler, non-PV variant and is the default path for users who do not need public verifiability (e.g., the `ckd-example-cli` uses it by default).
- Any single attested MPC participant can execute the attack — no threshold collusion is required.
- The attacker only needs to observe a pending `CKDRequest` (visible on-chain) and submit a `respond_ckd` call before the honest nodes do.

---

### Recommendation

1. **Require `AppPublicKeyPV` for all on-chain CKD requests** so that `ckd_output_check` can always be enforced, or
2. **Add an equivalent cryptographic binding check for `AppPublicKey`** — e.g., verify that `big_c` is a valid ElGamal encryption of the expected key material under the user's `app_public_key`, or
3. At minimum, **document prominently** that `AppPublicKey` requests carry no on-chain threshold guarantee and that users must verify the response off-chain before using the derived key.

---

### Proof of Concept

1. User Alice calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(alice_pk)`.
2. Malicious attested participant Eve calls `respond_ckd(ckd_request, CKDResponse { big_y: [1u8;48], big_c: [2u8;48] })` before honest nodes respond.
3. The contract executes:
   - `assert_caller_is_attested_participant_and_protocol_active()` → passes (Eve is attested).
   - `match &request.app_public_key { AppPublicKey(_) => {} ... }` → **empty arm, no check**.
   - `resolve_yields_for(...)` → fake response delivered to Alice.
4. Alice receives `(big_y=[1;48], big_c=[2;48])` — a garbage CKD output — and cannot recover the correct derived key. [3](#0-2)

```rust
    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
```
