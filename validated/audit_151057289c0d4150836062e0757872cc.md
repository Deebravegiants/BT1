### Title
Unvalidated Zero `BlockConfirmations` Allows MPC Attestation of Unfinalized Bitcoin Transactions - (File: crates/near-mpc-contract-interface/src/types/foreign_chain.rs, crates/foreign-chain-inspector/src/bitcoin/inspector.rs)

### Summary
The `verify_foreign_transaction` contract entry point accepts a `BitcoinRpcRequest` with `confirmations: 0` without any validation. Because the Bitcoin inspector's finality check is `block_confirmations_threshold <= actual_confirmations`, a threshold of zero is trivially satisfied by any confirmed transaction (minimum 1 confirmation). The MPC network will therefore produce a valid threshold signature attesting to a Bitcoin transaction that has only 1 confirmation — far below the 6-confirmation standard for finality — enabling double-spend conditions on any bridge that relies on these attestations.

### Finding Description

**Root cause — no lower-bound validation on `BlockConfirmations`:**

`BlockConfirmations` is a plain newtype over `u64` with no minimum enforced:

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI JSON schema confirms `"minimum": 0.0`, meaning the wire format explicitly allows zero.

The contract's `verify_foreign_transaction` queues the request as-is, with no check that `confirmations > 0` (or any protocol-meaningful minimum). The full `BitcoinRpcRequest` — including the caller-supplied `confirmations` field — is stored verbatim and forwarded to MPC nodes.

**Node-side check is trivially bypassed at threshold = 0:**

In `BitcoinInspector::extract`, the only finality gate is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

When `block_confirmations_threshold = 0`, the condition `0 <= actual_confirmations` is always `true` for any transaction that has been mined (actual confirmations ≥ 1). The check never fires.

**Attacker-controlled entry path:**

1. Attacker broadcasts a Bitcoin transaction (e.g., a deposit to a bridge).
2. Attacker immediately calls `verify_foreign_transaction` on the MPC contract with `confirmations: 0` and `extractors: [BlockHash]`, attaching 1 yoctoNEAR.
3. The contract queues the request without validation.
4. MPC nodes query the Bitcoin RPC; the transaction has 1 confirmation; `0 <= 1` passes; nodes extract the block hash and produce a threshold signature over `ForeignTxSignPayload { request: { confirmations: 0, ... }, values: [BlockHash(...)] }`.
5. The attacker receives a valid MPC-signed attestation for a 1-confirmation transaction.
6. Attacker submits the attestation to the bridge contract to claim NEAR-side funds.
7. Attacker then mines a competing Bitcoin chain tip that reorgs the original transaction out.
8. The bridge has released funds for a transaction that no longer exists on the canonical chain — a double-spend.

### Impact Explanation

The MPC network's `verify_foreign_transaction` feature is explicitly designed for bridge use cases (Omnibridge inbound flow is cited in the design doc). Its security guarantee is that it only attests to finalized foreign-chain state. Accepting `confirmations: 0` breaks this guarantee: the MPC network issues a valid threshold signature over a transaction that is not final. Any bridge contract that gates fund releases on these attestations — without independently re-validating the `confirmations` field embedded in the signed payload — is exposed to double-spend attacks. This matches the "High" impact category: forged foreign-chain verification causing invalid bridge execution or double-spend conditions.

### Likelihood Explanation

The attack is reachable by any unprivileged NEAR account with 1 yoctoNEAR. No privileged access, key material, or threshold collusion is required. The attacker only needs to submit a crafted `verify_foreign_transaction` call. The double-spend step requires a Bitcoin reorg, which is probabilistic but well within reach for a well-resourced attacker targeting a 1-confirmation transaction (reorg probability is non-negligible at low confirmation counts). The vulnerability is structural and present in every production deployment of the foreign-chain verification feature.

### Recommendation

1. **Enforce a minimum `BlockConfirmations` at the contract level.** In `verify_foreign_transaction`, reject any `BitcoinRpcRequest` where `confirmations == 0` (or below a protocol-defined minimum, e.g., 1):

```rust
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    require!(btc_req.confirmations.0 >= MIN_BITCOIN_CONFIRMATIONS,
        "confirmations must be at least MIN_BITCOIN_CONFIRMATIONS");
}
```

2. **Alternatively, enforce a per-chain minimum in the `BitcoinInspector`.** Reject any `block_confirmations_threshold` of 0 before the RPC call, returning a `ForeignChainInspectionError::InvalidConfirmationThreshold`.

3. **Document the invariant** that bridge contracts must validate the `confirmations` field in the signed payload matches their expected minimum, as a defense-in-depth measure.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
