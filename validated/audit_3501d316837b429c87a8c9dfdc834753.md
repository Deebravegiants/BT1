### Title
Attacker-Controlled `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Replay by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but **never checks that `response.payload_hash` is the correct hash for the pending request**. A single Byzantine attested participant (below the signing threshold) can replay any previously observed valid `(payload_hash, signature)` pair to resolve any pending foreign-transaction verification request with forged data, bypassing the requirement that the signed hash actually corresponds to the transaction being verified.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 691–754), the contract performs the following steps:

1. Looks up the pending request by the `request` key in `pending_verify_foreign_tx_requests`.
2. Extracts `payload_hash` directly from the caller-supplied `response`:
   ```rust
   let payload_hash: [u8; 32] = response.payload_hash.0;
   ```
3. Verifies the signature against that caller-supplied hash:
   ```rust
   near_mpc_signature_verifier::verify_ecdsa_signature(
       signature_response,
       &payload_hash,
       &secp_pk,
   ).is_ok()
   ```
4. Resolves the pending yield with the full (unvalidated) response.

The contract **never recomputes or checks** that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the actual pending request. The `payload_hash` field is entirely attacker-controlled.

This is the direct analog of the FertilizerFacet bug: just as that contract checked `if (_amount > remaining) _amount = remaining` but then used the original uncapped `amount` downstream, this contract checks that the signature is valid over `response.payload_hash` but uses the attacker-supplied `response.payload_hash` as the authoritative hash — rather than computing the expected hash from the stored request.

The `ForeignTxSignPayload` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,  // the original request (tx_id, extractors, etc.)
    pub values: Vec<ExtractedValue>,       // the extracted on-chain data
}
```

and `msg_hash = SHA-256(borsh(ForeignTxSignPayload))`. The contract stores the `request` in `pending_verify_foreign_tx_requests` but does not store or recompute the `values`, so it cannot independently derive the expected hash. However, it should at minimum reject any response whose `payload_hash` was not produced for the specific pending request.

By contrast, the regular `respond` function for ECDSA signatures correctly derives the expected public key from the stored request's tweak and verifies the signature against the stored payload — it does not accept a caller-supplied payload hash.

---

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) can:

1. Observe any valid `(payload_hash_X, sig_X)` pair from any previous legitimate `respond_verify_foreign_tx` call (these are public on-chain).
2. Wait for a new pending `verify_foreign_transaction` request for transaction T.
3. Call `respond_verify_foreign_tx(request_T, { payload_hash: payload_hash_X, signature: sig_X })`.
4. The contract verifies `sig_X` over `payload_hash_X` against the root public key → **valid**.
5. The contract resolves request T's pending yield with `{ payload_hash_X, sig_X }`.
6. The user's contract receives a `VerifyForeignTransactionResponse` where `payload_hash` does not correspond to transaction T.

For downstream contracts that do not use the SDK's `ForeignChainSignatureVerifier::verify_signature` (which checks `expected_payload_hash == response.payload_hash`), this constitutes a **forged foreign-chain verification result** — the MPC network's root key appears to have attested to data it never actually verified. This enables invalid bridge execution or double-spend conditions.

Even for contracts that do use the SDK, the pending request is consumed and cannot be retried, causing a permanent denial of service on that verification slot.

**Impact: High** — Cross-chain replay / forged foreign-chain verification that causes invalid bridge execution.

---

### Likelihood Explanation

- The attacker must be a single attested MPC participant. No threshold collusion is required.
- The attacker only needs any previously observed valid `(payload_hash, signature)` pair, which is available from on-chain history.
- The attack is straightforward to execute: one transaction call to `respond_verify_foreign_tx` with a mismatched response.

**Likelihood: High** — Any single Byzantine attested participant can execute this with minimal effort using publicly available on-chain data.

---

### Recommendation

The contract must bind the accepted `payload_hash` to the specific pending request. Since the contract does not store the `values` (extracted data), it cannot fully recompute the hash independently. Two viable mitigations:

1. **Commit-then-reveal**: When the first MPC node calls `respond_verify_foreign_tx`, store the `payload_hash` in the pending request entry. Subsequent calls for the same request must provide the same `payload_hash`. This prevents cross-request replay while still allowing the MPC nodes to supply the `values`.

2. **Include the full payload in the response**: Require MPC nodes to submit the full `ForeignTxSignPayload` (not just the hash), allowing the contract to recompute and verify `SHA-256(borsh(payload))` and check that `payload.request == stored_request`.

The analogous fix in the FertilizerFacet report was to use the capped variable (`_amount`) downstream instead of the original (`amount`). Here, the fix is to use a hash derived from the stored request rather than the caller-supplied `response.payload_hash`.

### Proof of Concept

```
Setup:
- MPC network has previously processed a verify_foreign_transaction for Bitcoin tx_id=0xBB,
  producing a valid response: { payload_hash_B, sig_B } (visible on-chain).
- A new pending request exists for Bitcoin tx_id=0xAA (request_A stored in contract).

Attack:
1. Byzantine participant (single, below threshold) calls:
   respond_verify_foreign_tx(
       request = request_A,           // correct request key — resolves the pending yield
       response = {
           payload_hash: payload_hash_B,  // hash from tx_id=0xBB, not 0xAA
           signature: sig_B,              // valid sig over payload_hash_B
       }
   )

2. Contract check (lib.rs:729-734):
   verify_ecdsa_signature(sig_B, payload_hash_B, root_pk) → OK
   (signature is valid over the provided hash — check passes)

3. Contract resolves request_A's pending yield with { payload_hash_B, sig_B }.

4. User's contract receives VerifyForeignTransactionResponse for tx_id=0xAA
   but payload_hash corresponds to tx_id=0xBB.

5a. If user does NOT call ForeignChainSignatureVerifier::verify_signature:
    → Accepts forged verification result for tx_id=0xAA.
    → Invalid bridge execution / double-spend enabled.

5b. If user DOES call verify_signature:
    → expected_hash(tx_id=0xAA) ≠ payload_hash_B → verification fails.
    → Pending request consumed; user cannot retry → DoS.

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
