### Title
Missing Chainlink Oracle Staleness Validation Allows Stale Prices to Determine rsETH Minting Amounts - (File: contracts/oracles/ChainlinkPriceOracle.sol)

### Summary
`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` with **zero staleness validation** — no heartbeat check, no `answeredInRound < roundId` check, and no `timestamp == 0` guard. This stale price is directly used to compute how many rsETH tokens a depositor receives, meaning a price that has not been updated within the oracle's heartbeat window (e.g., 24 hours for stETH/ETH) silently propagates into the minting calculation, causing depositors to receive incorrect rsETH amounts.

### Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the latest Chainlink round data and immediately returns the price with no validation:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

All five return values from `latestRoundData()` — `roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound` — are available, but only `answer` is used. The `updatedAt` timestamp and `answeredInRound` are silently discarded.

This price flows directly into rsETH minting via `LRTOracle.getAssetPrice()` → `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Chainlink feeds for LSTs like stETH/ETH carry a **24-hour heartbeat** and a **2% deviation threshold**, meaning the on-chain price can legally lag the true market price by up to 2% for up to 24 hours before a new round is triggered. During this window, the stale price is accepted without any revert or warning.

By contrast, the protocol's own `ChainlinkOracleForRSETHPoolCollateral` (used for L2 pool collateral) at least checks `answeredInRound < roundID` and `timestamp == 0`, demonstrating that the team is aware of the need for staleness guards — yet `ChainlinkPriceOracle` has none.

### Impact Explanation

When the on-chain Chainlink price for a supported LST (e.g., stETH) is stale and lower than the true market price:

- `lrtOracle.getAssetPrice(stETH)` returns a deflated value
- `rsethAmountToMint = (amount * deflatedPrice) / rsETHPrice` yields fewer rsETH tokens than the depositor is entitled to
- The depositor permanently loses the difference in value — their LST is accepted at full face value but they receive under-priced rsETH

Conversely, if the stale price is inflated relative to the true price, the depositor receives excess rsETH, diluting existing rsETH holders and causing protocol insolvency over time.

The impact matches the external report's classification: **user funds are unfairly penalized due to the large variance between the on-chain price and the true asset price**.

### Likelihood Explanation

Chainlink's stETH/ETH feed has a documented 24-hour heartbeat and 2% deviation threshold. During periods of low volatility, the feed can go many hours without an update. Any depositor transacting during such a window is exposed. No attacker action is required — the condition arises from normal oracle operation. The entry path (`depositAsset`) is fully permissionless and reachable by any user.

### Recommendation

Add staleness validation in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (block.timestamp - updatedAt > MAX_HEARTBEAT) revert StalePrice();
if (price <= 0) revert InvalidPrice();
```

`MAX_HEARTBEAT` should be set per feed (e.g., 3600 seconds for a 1-hour heartbeat feed, 86400 for a 24-hour feed), stored as a configurable parameter alongside `assetPriceFeed`. Additionally, prefer feeds with tighter heartbeats and deviation thresholds where available (e.g., stETH/USD at 1-hour / 1% over stETH/ETH at 24-hour / 2%).

### Proof of Concept

1. Chainlink's stETH/ETH feed last updated 20 hours ago at a price of 0.98 ETH/stETH. True market price is now 0.999 ETH/stETH (within the 2% deviation band, so no new round triggered).
2. Alice calls `LRTDepositPool.depositAsset(stETH, 100e18, minRSETH, "")`.
3. `getRsETHAmountToMint` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `0.98e18` (stale).
4. `rsethAmountToMint = (100e18 * 0.98e18) / rsETHPrice` — Alice receives rsETH valued at 98 ETH worth of stETH instead of 99.9 ETH worth.
5. Alice loses ~1.9% of her deposit value with no recourse, as the contract accepted the stale price without revert.
