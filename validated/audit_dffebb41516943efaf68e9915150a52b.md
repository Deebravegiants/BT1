Audit Report

## Title
Unprivileged `updateRate()` Broadcasts Stale rsETH Price to L2, Enabling Excess `wrsETH` Minting at Existing Holders' Expense — (`contracts/cross-chain/RSETHRateProvider.sol`)

## Summary
`CrossChainRateProvider.updateRate()` carries no access control, allowing any caller to broadcast the current cached `LRTOracle.rsETHPrice` to L2 at any time. Because `rsETHPrice` is a stored state variable only refreshed when `updateRSETHPrice()` is explicitly called, an attacker can invoke `updateRate()` in the window after EigenLayer rewards accrue but before the oracle keeper refreshes the price, locking a below-true rate into `CrossChainRateReceiver`. L2 depositors then receive excess `wrsETH` relative to the true backing, diluting accrued yield owed to existing rsETH holders.

## Finding Description

**Root cause — no access control on `updateRate()`**

`CrossChainRateProvider.updateRate()` is declared `external payable nonReentrant` with no role modifier: [1](#0-0) 

Any EOA or contract can call it at any time by supplying only the LayerZero gas fee.

**Stale value source — `rsETHPrice` is a cached storage slot**

`RSETHRateProvider.getLatestRate()` does not compute a live price; it reads the `rsETHPrice` storage variable of `LRTOracle`: [2](#0-1) 

`rsETHPrice` is a plain `uint256` state variable: [3](#0-2) 

It is only updated inside `_updateRsETHPrice()`, which is triggered by the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`: [4](#0-3) 

Between EigenLayer reward epochs and the next oracle refresh, `rsETHPrice` reflects the pre-reward exchange rate.

**Stale rate forwarded verbatim to L2**

`updateRate()` encodes `latestRate` and sends it via LayerZero with no freshness or sanity check: [5](#0-4) 

`CrossChainRateReceiver.lzReceive()` stores whatever value arrives, also without any freshness check: [6](#0-5) 

**L2 pool mints excess `wrsETH` using the poisoned rate**

`RSETHPoolV3.viewSwapRsETHAmountAndFee()` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`: [7](#0-6) 

`getRate()` resolves to `CrossChainRateReceiver.getRate()` — the just-poisoned value: [8](#0-7) 

A stale rate lower than the true post-yield rate causes the division to produce a larger `rsETHAmount`, minting excess `wrsETH` that is not backed by real ETH at the true exchange rate.

**Existing guards are insufficient**

- The `nonReentrant` modifier on `updateRate()` prevents re-entrancy but not unprivileged calls.
- The `dailyMintLimit` in `RSETHPoolV3` caps total daily minting volume but does not prevent minting at a manipulated rate up to that cap.
- The `pricePercentageLimit` in `LRTOracle._updateRsETHPrice()` guards against large price jumps during oracle *updates*, but is entirely irrelevant to the staleness of the rate being broadcast.
- The `whenNotPaused` modifier on `updateRSETHPrice()` only prevents oracle updates when paused; it does not restrict `updateRate()`.

## Impact Explanation

**High — Theft of unclaimed yield.**

Suppose the true post-yield rsETH/ETH rate is 1.05 but `rsETHPrice` is still 1.00. An attacker calls `updateRate()` before the keeper refreshes the oracle, locking 1.00 on L2. A depositor of 1 ETH receives `1e18 / 1.00e18 = 1.0 wrsETH` instead of the correct `1e18 / 1.05e18 ≈ 0.952 wrsETH` — a ~5% excess. The extra `wrsETH` is unbacked at the true rate; when the rate is corrected to 1.05, the attacker's position is worth more than deposited, at the direct expense of existing rsETH holders whose accrued staking yield is diluted. This matches the allowed impact class "High — Theft of unclaimed yield."

## Likelihood Explanation

- EigenLayer distributes rewards on a predictable cadence, making the staleness window observable on-chain.
- `updateRate()` requires only ETH for LayerZero gas — no special role, no token approval, no privileged account.
- The attacker can monitor mempool or on-chain events for reward distribution and front-run the keeper's `updateRSETHPrice()` call by submitting `updateRate()` first.
- The attack is repeatable every reward epoch.
- No existing guard (reentrancy lock, pause, daily mint cap, price threshold) prevents this specific ordering attack.

## Recommendation

1. **Add access control to `updateRate()`** — restrict it to a trusted keeper/operator role (e.g., `onlyOwner` or a dedicated `KEEPER_ROLE`) so only authorized accounts can broadcast the rate.
2. **Enforce a freshness check before broadcasting** — record a `lastOracleUpdate` timestamp in `LRTOracle` when `rsETHPrice` is written, and require in `updateRate()` that `block.timestamp - lastOracleUpdate <= maxStaleness` before sending.
3. **Compute a live price in `getLatestRate()`** — have `RSETHRateProvider.getLatestRate()` call an equivalent view function that computes the current price from live TVL rather than reading the cached storage slot.
4. **Add a staleness guard in `RSETHPoolV3`** — reject deposits if the oracle's `lastUpdated` timestamp is older than an acceptable threshold.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Foundry fork test — run against a mainnet fork
// Precondition: EigenLayer rewards have accrued; LRTOracle.rsETHPrice has NOT been refreshed yet.

contract StaleRatePoC is Test {
    RSETHRateProvider provider = RSETHRateProvider(<deployed_address>);
    RSETHPoolV3       pool     = RSETHPoolV3(<l2_pool_address>);
    LRTOracle         oracle   = LRTOracle(<oracle_address>);

    function testStaleRateExploit() external {
        // 1. Confirm rsETHPrice is stale (below true TVL-derived price)
        uint256 staleRate = oracle.rsETHPrice();
        uint256 trueRate  = computeTrueRate(); // read TVL directly; e.g. 1.05e18
        assertLt(staleRate, trueRate, "no staleness — precondition not met");

        // 2. Attacker broadcasts stale rate to L2 — no role required
        provider.updateRate{value: 0.01 ether}();
        // LayerZero delivers stale rate to CrossChainRateReceiver on L2

        // 3. Attacker deposits 1 ETH on L2 at the stale rate
        uint256 wrsETHReceived = pool.deposit{value: 1 ether}("ref");

        // 4. Assert excess minting
        uint256 fairAmount  = 1e18 * 1e18 / trueRate;   // correct wrsETH
        uint256 staleAmount = 1e18 * 1e18 / staleRate;  // actual wrsETH minted
        assertGt(wrsETHReceived, fairAmount,
            "Excess wrsETH minted at stale rate — existing holder yield diluted");
    }
}
```

The test demonstrates that an unprivileged caller invoking `updateRate()` before `updateRSETHPrice()` is called results in L2 depositors receiving more `wrsETH` than the true L1 backing justifies, constituting theft of unclaimed yield from existing rsETH holders.
